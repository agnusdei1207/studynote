# CPU 스케줄링 (CPU Scheduling)

## 핵심 인사이트 (3줄 요약)
> **CPU를 어떤 프로세스에 얼마나 할당할지 결정**하는 운영체제의 핵심 기능. 선점(Preemptive)과 비선점(Non-preemptive)으로 크게 분류. 응답시간·처리량·공정성을 균형있게 최적화하는 것이 목표다.

### Ⅰ. 개요

#### 1. 개념
CPU 스케줄링이란 **준비(Ready) 상태의 프로세스 중 어떤 프로세스에 CPU를 할당할지 결정**하는 정책이다.

> 비유: "병원 접수 대기열 관리" - 응급도(우선순위), 대기시간, 처리 효율을 고려해 진료 순서 결정

---

### Ⅱ. 구성 요소 및 핵심 원리

#### 2. 스케줄링 기준 지표
| 지표 | 설명 | 최적화 방향 |
|------|------|------------|
| CPU 이용률 (CPU Utilization) | CPU가 실제 작업하는 비율 | 최대화 |
| 처리량 (Throughput) | 단위 시간당 완료 프로세스 수 | 최대화 |
| 총처리시간 (Turnaround Time) | 제출~완료까지 전체 시간 | 최소화 |
| 대기시간 (Waiting Time) | 준비 큐에서 기다린 시간 합계 | 최소화 |
| 응답시간 (Response Time) | 요청~첫 응답까지 시간 | 최소화 |

#### 4. 주요 스케줄링 알고리즘 비교
| 알고리즘 | 방식 | 선점 | 기아 | 특징 |
|---------|------|------|------|------|
| **FCFS** | 도착 순서 | X | X | 단순, Convoy Effect |
| **SJF** | 짧은 burst 먼저 | X | O | 평균 대기시간 최소 |
| **SRTF** | 남은 시간 짧은것 먼저 | O | O | SJF 선점 버전 |
| **우선순위** | 우선순위 높은 것 먼저 | 양쪽 | O | 에이징으로 기아 해결 |
| **Round Robin** | 시간 할당량(Time Quantum) | O | X | 응답시간 균등 |
| **MLQ** | 다단계 큐 | O | O | 큐 간 이동 불가 |
| **MLFQ** | 다단계 피드백 큐 | O | X | 큐 간 이동 가능 |

#### 5. 알고리즘 상세
### FCFS (First Come First Served)
```
프로세스: P1(24ms) → P2(3ms) → P3(3ms)

간트 차트:
| P1(0~24) | P2(24~27) | P3(27~30) |

대기시간: P1=0, P2=24, P3=27
평균 대기시간: (0+24+27)/3 = 17ms

문제: Convoy Effect (작은 프로세스들이 큰 프로세스 뒤에서 오래 기다림)
```

### Round Robin (RR)
```
Time Quantum = 4ms
프로세스: P1(24ms), P2(3ms), P3(3ms)

간트 차트:
|P1(0-4)|P2(4-7)|P3(7-10)|P1(10-14)|P1(14-18)|P1(18-22)|P1(22-26)|P1(26-30)|

특징:
- Time Quantum이 작을수록 응답시간 ↓, 문맥교환 오버헤드 ↑
- Time Quantum이 클수록 FCFS에 수렴
- 최적 Quantum: CPU burst의 80%를 Quantum 내 처리
```

### MLFQ (Multi-Level Feedback Queue)
```
Q0 (Time Quantum=8ms) ← 새 프로세스 진입
Q1 (Time Quantum=16ms) ← Q0에서 강등
Q2 (FCFS) ← Q1에서 강등

규칙:
- Q0에서 다 못끝내면 Q1으로 강등
- 낮은 큐에 있는 프로세스는 높은 큐가 비어야 실행
- 에이징으로 기아 방지: 오래 기다린 프로세스 승격

→ CPU burst 특성 자동 파악 (I/O 위주 ↔ CPU 위주)
```

#### 6. 기아(Starvation)와 에이징(Aging)
```
기아(Starvation):
- 낮은 우선순위 프로세스가 무한정 대기
- SJF, 우선순위 스케줄링에서 발생

에이징(Aging) 해결책:
- 오래 기다릴수록 우선순위 점진적 상향
- 예: 매 15분마다 우선순위 +1
```

#### 7. 실시간 스케줄링
| 유형 | 마감 시간 | 스케줄링 | 예 |
|------|---------|---------|-----|
| 하드 실시간 | 반드시 준수 | EDF, RMS | 항공 제어, 의료기기 |
| 소프트 실시간 | 가능한 준수 | Rate Monotonic | 스트리밍, 게임 |

```
EDF (Earliest Deadline First):
- 마감 시간이 가장 가까운 프로세스 우선
- 선점형, 최적 알고리즘

RMS (Rate Monotonic Scheduling):
- 실행 주기가 짧을수록 높은 우선순위
- 비선점형, 분석 가능한 알고리즘
```

### Ⅲ. 기술 비교 분석

#### 3. 선점 vs 비선점 비교
```
비선점 (Non-Preemptive):
┌────────────────────────────────────────────────────┐
│  CPU 할당 후 프로세스가 자발적으로 반납할 때까지   │
│  다른 프로세스가 CPU를 빼앗을 수 없음             │
│  → 응답시간 예측 쉬움, 오버헤드 낮음              │
│  → 실시간 처리 부적합                              │
│  예: FCFS, SJF(비선점), 우선순위(비선점)           │
└────────────────────────────────────────────────────┘

선점 (Preemptive):
┌────────────────────────────────────────────────────┐
│  우선순위 높은 프로세스가 현재 실행 중인           │
│  프로세스의 CPU를 강제로 빼앗을 수 있음           │
│  → 응답시간 짧음, 실시간 처리 적합               │
│  → 문맥 교환(Context Switch) 오버헤드 발생        │
│  예: Round Robin, SRTF, 선점 우선순위             │
└────────────────────────────────────────────────────┘
```

#### 8. 다른 것과 비교
| 항목 | FCFS | SJF | Round Robin | MLFQ |
|------|------|-----|-------------|------|
| 구현 복잡도 | 낮음 | 중간 | 낮음 | 높음 |
| 응답시간 | 나쁨 | 좋음 | 균등 | 좋음 |
| 평균 대기시간 | 나쁨 | 최소 | 중간 | 중간 |
| 기아 | 없음 | 있음 | 없음 | 없음 |
| 실시간 적합 | X | X | O | O |
| 범용 OS | X | X | O | O(리눅스 CFS) |

### Ⅳ. 실무 적용 방안

#### 9. 실무에선? (기술사적 판단)
- **리눅스 CFS**: Completely Fair Scheduler - 가중치 기반 공정 큐
- **Windows**: 다단계 우선순위 큐 (32단계)
- **임베디드/RTOS**: EDF 또는 Rate Monotonic
- **클라우드/컨테이너**: 쿠버네티스 스케줄러 (노드에 Pod 배치)
- **기술사 포인트**: 알고리즘별 trade-off와 선택 기준 명확히

### Ⅴ. 기대 효과 및 결론


| 효과 영역 | 내용 | 정량적 목표 |
|---------|-----|-----------|
| **시스템 안정성** | 자원 관리 최적화로 교착상태·기아 등 문제 방지 | 서비스 가용성 99.99% 이상 |
| **처리 성능** | 스케줄링·동기화 최적화로 CPU·메모리 효율 향상 | 처리량(Throughput) 20~40% 향상 |
| **개발 생산성** | 명확한 추상화로 애플리케이션 개발 단순화 | OS Level 버그 70% 감소 |

#### 결론
> **CPU 스케줄링 (CPU Scheduling)**은(는) 운영체제의 핵심 메커니즘들은 클라우드·컨테이너 시대에도 동일한 원리로 작동하며, eBPF 등 새로운 커널 확장 기술과 결합하여 더 유연하고 관측 가능한 시스템으로 진화하고 있다.

> **※ 참고 표준**: POSIX (IEEE 1003.1), Linux Kernel Documentation, Windows Internals
