### Ⅰ. 개요

#### 1. IPC 개념
프로세스는 기본적으로 독립된 메모리 공간을 가지므로, **직접 데이터를 공유할 수 없어 별도의 통신 메커니즘이 필요**하다.

> 비유: "건물 간 통신" - 건물이 다르면 내부 전화를 못 쓰고 외부 전화·우편·회의실을 이용

### Ⅱ. 구성 요소 및 핵심 원리

#### 3. 임계 영역(Critical Section)과 동기화
```
임계 영역 (Critical Section):
공유 자원에 접근하는 코드 영역
→ 동시에 하나의 프로세스/스레드만 실행해야 함

임계 영역 조건 (Peterson 해결책):
1. 상호 배제 (Mutual Exclusion): 한 번에 하나만
2. 진행 (Progress): 기다리는 프로세스만 선택 참여
3. 유한 대기 (Bounded Waiting): 무한정 기다리지 않음
```

#### 5. 생산자-소비자 문제 (Producer-Consumer)
```python
import threading
import time
from queue import Queue

buffer_size = 5
buffer = Queue(maxsize=buffer_size)

empty = threading.Semaphore(buffer_size)  # 빈 슬롯 수
full = threading.Semaphore(0)             # 채워진 슬롯 수
mutex = threading.Lock()                  # 버퍼 접근 보호

def producer(name, items):
    for item in items:
        empty.acquire()         # P(empty): 빈 슬롯 기다림
        mutex.acquire()         # P(mutex): 임계 영역 진입
        
        buffer.put(item)
        print(f"[{name}] 생산: {item} | 버퍼: {buffer.qsize()}")
        
        mutex.release()         # V(mutex)
        full.release()          # V(full): 채워진 슬롯 증가
        time.sleep(0.1)

def consumer(name, count):
    for _ in range(count):
        full.acquire()          # P(full): 아이템 기다림
        mutex.acquire()         # P(mutex)
        
        item = buffer.get()
        print(f"[{name}] 소비: {item} | 버퍼: {buffer.qsize()}")
        
        mutex.release()         # V(mutex)
        empty.release()         # V(empty): 빈 슬롯 증가
        time.sleep(0.2)

print("=== 생산자-소비자 패턴 ===\n")
t1 = threading.Thread(target=producer, args=("생산자A", ["item1","item2","item3","item4","item5"]))
t2 = threading.Thread(target=consumer, args=("소비자A", 3))
t3 = threading.Thread(target=consumer, args=("소비자B", 2))

t1.start(); t2.start(); t3.start()
t1.join(); t2.join(); t3.join()
print("\n완료!")
```

#### 6. 독자-저자 문제 (Readers-Writers)
```
규칙:
- 여러 독자(Reader)가 동시에 읽기 가능
- 저자(Writer)는 독점적 접근 필요
- 독자 우선 vs 저자 우선 두 가지 해결책

독자 우선:
- 독자 있으면 저자 대기
- 저자 기아(Starvation) 가능

저자 우선:
- 저자 대기 중이면 새 독자 진입 금지
- 독자 기아 가능
```

### Ⅲ. 기술 비교 분석

#### 2. IPC 방법 비교
| 방법 | 속도 | 용량 | 동기화 | 방향 | 용도 |
|------|------|------|--------|------|------|
| **파이프(Pipe)** | 보통 | 소 | 자동 | 단방향 | 부모↔자식 프로세스 |
| **Named Pipe(FIFO)** | 보통 | 소 | 자동 | 양방향 | 관련 없는 프로세스 |
| **메시지 큐** | 보통 | 중 | 자동 | 양방향 | 비동기 통신 |
| **공유 메모리** | 매우 빠름 | 대 | 필요 | 양방향 | 대량 데이터 |
| **소켓** | 느림 | 대 | 필요 | 양방향 | 네트워크/로컬 |
| **세마포어** | 빠름 | - | - | - | 동기화 신호 |
| **시그널** | 빠름 | 소 | - | 단방향 | 이벤트 알림 |

```
IPC 메커니즘 상세:

① 파이프 (Pipe):
  프로세스A ──[쓰기]──► [버퍼] ──[읽기]──► 프로세스B
  - 단방향, 익명 (부모-자식만)
  - $ ls | grep .txt  (쉘 파이프)

② 공유 메모리 (Shared Memory):
  물리 메모리의 동일 영역을 여러 프로세스의 가상 주소에 매핑
  
  프로세스A의 주소공간    공유 메모리 영역    프로세스B의 주소공간
  [... shm_ptr ...] ──► [데이터 블록] ◄── [... shm_ptr ...]
  
  가장 빠른 IPC (커널 개입 최소)
  But 동기화 별도 필요 (세마포어 사용)

③ 메시지 큐:
  프로세스A ──[enqueue]──► [큐] ──[dequeue]──► 프로세스B
  - 비동기: 보낸 후 기다리지 않아도 됨
  - 타입별 메시지 선택 수신 가능
  - Linux: System V msgq, POSIX mq

④ 소켓 (Socket):
  [프로세스A] ──IP:Port──► [커널] ──► [프로세스B]
  - 로컬(UNIX 도메인) 또는 네트워크
  - 가장 범용적 (TCP/UDP 모두 지원)
```

#### 4. 뮤텍스 vs 세마포어 vs 모니터 비교
```
┌────────────────────────────────────────────────────────┐
│           동기화 도구 비교                              │
├─────────────┬──────────────┬──────────────┬────────────┤
│ 항목        │ 뮤텍스       │ 세마포어     │ 모니터     │
├─────────────┼──────────────┼──────────────┼────────────┤
│ 소유권      │ 잠근 스레드만│ 없음         │ 없음       │
├─────────────┼──────────────┼──────────────┼────────────┤
│ 값 범위     │ 0/1 (이진)   │ 0~N (카운팅) │ -          │
├─────────────┼──────────────┼──────────────┼────────────┤
│ 해제        │ 잠근 측만    │ 누구든 가능  │ 자동       │
├─────────────┼──────────────┼──────────────┼────────────┤
│ 용도        │ 락(Lock)     │ 자원 개수 카운트│ 조건 동기화│
├─────────────┼──────────────┼──────────────┼────────────┤
│ 구현        │ OS 제공      │ OS 제공      │ 언어 수준  │
├─────────────┼──────────────┼──────────────┼────────────┤
│ 예시        │ Mutex, Lock  │ Semaphore    │ Java sync  │
└─────────────┴──────────────┴──────────────┴────────────┘

세마포어 종류:
- 이진 세마포어 (0 or 1): 뮤텍스처럼 사용
- 카운팅 세마포어: 자원 N개 관리 (생산자-소비자 패턴)

P(S) 연산: Wait. S가 0이면 대기, S > 0이면 S-1 후 진입
V(S) 연산: Signal. S+1 (대기 중인 프로세스 있으면 깨움)
```

### Ⅳ. 실무 적용 방안

#### 7. 실무에선? (기술사적 판단)
- **IPC 선택**: 속도 중시 → 공유 메모리, 쉬운 구현 → 소켓/메시지 큐
- **마이크로서비스**: 서비스 간 통신 → REST/gRPC/메시지 큐
- **Kafka**: 대규모 이벤트 스트리밍에서 IPC 역할
- **기술사 포인트**: 세마포어 P·V 연산, 생산자-소비자 패턴, 교착상태 조건
