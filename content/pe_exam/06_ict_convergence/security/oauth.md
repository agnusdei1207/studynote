### Ⅰ. 개요

#### 1. 개념
OAuth(Open Authorization)는 **사용자가 비밀번호를 공유하지 않고도 제3자 애플리케이션에 특정 리소스에 대한 제한된 접근 권한을 부여할 수 있게 하는 개방형 인증 프로토콜**이다.

> 비유: "호텔 카드키" - 방 열쇠(비밀번호) 대신 카드(토큰)로 입장

### Ⅱ. 구성 요소 및 핵심 원리

#### 2. OAuth 역할

│  ┌────────────────────────────────────────────────┐   │
│  │                                                │   │

│  │     • 권한 부여 승인                          │   │
│  │                                                │   │

│  │     • 예: "구글로 로그인" 하는 앱            │   │
│  │                                                │   │
│  │  🔐 Authorization Server (인증 서버)          │   │
│  │     • 사용자 인증                              │   │
│  │     • 토큰 발급                                │   │
│  │     • 예: Google, Facebook                    │   │
│  │                                                │   │
│  │  📦 Resource Server (리소스 서버)             │   │
│  │     • 보호된 리소스 보유                       │   │
│  │     • 토큰 검증 후 데이터 제공                │   │
│  │                                                │   │
│  └────────────────────────────────────────────────┘   │
│                                                        │
│  예시 - "Google로 로그인":                           │
│  ┌────────────────────────────────────────────────┐   │
│  │                                                │   │
│  │  👤 사용자                                    │   │
│  │     │                                          │   │
│  │     │ 1. 로그인 요청                          │   │
│  │     ↓                                          │   │
│  │  🖥️ 앱 (Client)                              │   │
│  │     │                                          │   │
│  │     │ 2. 인증 리다이렉트                      │   │
│  │     ↓                                          │   │
│  │  🔐 Google (Auth Server)                       │   │
│  │     │                                          │   │
│  │     │ 3. 승인 후 토큰 발급                    │   │
│  │     ↓                                          │   │
│  │  🖥️ 앱 → 📦 Google API (Resource Server)     │   │
│  │     │                                          │   │
│  │     │ 4. 토큰으로 데이터 요청                 │   │
│  │     ↓                                          │   │
│  │  👤 사용자: 프로필 정보 수신                  │   │
│  │                                                │   │
│  └────────────────────────────────────────────────┘   │
│                                                        │
└────────────────────────────────────────────────────────┘

#### 3. OAuth 2.0 흐름
```
┌────────────────────────────────────────────────────────┐
│           Authorization Code Flow                     │
├────────────────────────────────────────────────────────┤
│                                                        │
│  👤 사용자    🖥️ 앱    🔐 인증서버    📦 리소스서버  │
│     │          │          │              │            │
│     │ 1. 로그인 │          │              │            │
│     │─────────→│          │              │            │
│     │          │          │              │            │
│     │          │ 2. 인증 요청             │            │
│     │          │─────────→│              │            │
│     │          │ (client_id, redirect_uri)│            │
│     │          │          │              │            │
│     │ 3. 로그인/동의       │              │            │
│     │←──────────────────→│              │            │
│     │          │          │              │            │
│     │          │ 4. Auth Code            │            │
│     │          │←─────────│              │            │
│     │          │          │              │            │
│     │          │ 5. Code + Secret        │            │
│     │          │─────────→│              │            │
│     │          │          │              │            │
│     │          │ 6. Access Token         │            │
│     │          │←─────────│              │            │
│     │          │          │              │            │
│     │          │ 7. API 요청 (Token)     │            │
│     │          │─────────────────────────→│            │
│     │          │          │              │            │
│     │          │ 8. 보호된 리소스        │            │
│     │          │←─────────────────────────│            │
│     │          │          │              │            │
│                                                        │
└────────────────────────────────────────────────────────┘
```

#### 4. Grant Types
| 타입 | 설명 | 사용 시나리오 |
|-----|------|-------------|
| Authorization Code | 서버 사이드 앱 | 웹 애플리케이션 |
| Implicit | 클라이언트 사이드 | SPA (현재 권장 안 함) |
| Client Credentials | 앱 자체 인증 | 서비스 간 통신 |
| Resource Owner | 사용자 자격증명 | 신뢰 앱 (권장 안 함) |
| Refresh Token | 토큰 갱신 | 장기 세션 |

#### 5. 토큰 종류
```
┌────────────────────────────────────────────────────────┐
│           OAuth 토큰 종류                            │
├────────────────────────────────────────────────────────┤
│                                                        │
│  Access Token:                                        │
│  ┌────────────────────────────────────────────────┐   │
│  │                                                │   │
│  │  • 리소스 접근에 사용                         │   │
│  │  • 수명 제한 (보통 1시간)                     │   │
│  │  • 형식: JWT 또는 불투명 문자열               │   │
│  │                                                │   │
│  │  예: ya29.a0AfH6SMBx...                      │   │
│  │                                                │   │
│  └────────────────────────────────────────────────┘   │
│                                                        │
│  Refresh Token:                                       │
│  ┌────────────────────────────────────────────────┐   │
│  │                                                │   │
│  │  • 새 Access Token 발급에 사용                │   │
│  │  • 수명 김 (일~개월)                          │   │
│  │  • 안전하게 저장 필요                         │   │
│  │                                                │   │
│  └────────────────────────────────────────────────┘   │
│                                                        │
│  Scope (권한 범위):                                   │
│  ┌────────────────────────────────────────────────┐   │
│  │                                                │   │
│  │  • 요청하는 권한 지정                         │   │
│  │                                                │   │
│  │  예:                                          │   │
│  │  • profile: 기본 프로필                       │   │
│  │  • email: 이메일 주소                         │   │
│  │  • calendar: 캘린더 접근                      │   │
│  │  • drive.readonly: 드라이브 읽기              │   │
│  │                                                │   │
│  └────────────────────────────────────────────────┘   │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### Ⅲ. 기술 비교 분석

#### 6. OAuth vs OpenID Connect
| 특징 | OAuth 2.0 | OpenID Connect |
|-----|----------|----------------|
| 목적 | 인가 (Authorization) | 인증 (Authentication) |
| 토큰 | Access Token | ID Token |
| 사용자 정보 | API로 조회 | 토큰에 포함 |
| 용도 | API 접근 | 로그인 |

#### 7. 장단점
| 장점 | 단점 |
|-----|------|
| 비밀번호 노출 없음 | 구현 복잡성 |
| 범위 제한 가능 | 토큰 관리 |
| SSO 지원 | 의존성 |
| 표준 프로토콜 | 리다이렉트 필요 |
