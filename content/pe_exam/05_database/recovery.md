# 회복 기법 (Recovery)

### Ⅰ. 개요

#### 1. 개념
회복 기법은 **시스템 장애, 미디어 장애 등이 발생했을 때 데이터베이스를 일관된 상태로 복원**하는 기술이다.

> 비유: "게임 세이브" - 장애가 나면 마지막 저장점으로 되돌리기

---

### Ⅱ. 구성 요소 및 핵심 원리

#### 2. 장애 유형
```
1. 트랜잭션 장애 (Transaction Failure)
   - 논리적 오류
   - 잘못된 데이터
   - 사용자 취소
   → ROLLBACK으로 해결

2. 시스템 장애 (System Failure)
   - 정전, 하드웨어 오류
   - 소프트웨어 버그
   → 재시작 후 REDO/UNDO

3. 미디어 장애 (Media Failure)
   - 디스크 손상
   - 헤드 크래시
   → 백업본에서 복구

4. 자연재해
   - 화재, 홍수, 지진
   → 원격 백업 필요
```

#### 3. 로그 (Log)
### 3.1 로그 레코드 종류
```
┌────────────────────────────────────────────────┐
│                로그 파일 (Log File)             │
├────────────────────────────────────────────────┤
│ <T1, START>                                    │
│ <T1, A, 100, 150>    ← A를 100→150으로 변경    │
│ <T1, B, 200, 250>    ← B를 200→250으로 변경    │
│ <T1, COMMIT>                                   │
│ <T2, START>                                    │
│ <T2, C, 300, 350>                              │
│ ← 시스템 장애 발생!                            │
└────────────────────────────────────────────────┘

로그 레코드 형식:
- <T, START>: 트랜잭션 T 시작
- <T, X, old, new>: T가 X를 old→new로 변경
- <T, COMMIT>: 트랜잭션 T 커밋
- <T, ABORT>: 트랜잭션 T 중단
- <CHECKPOINT>: 체크포인트
```

### 3.2 로그 활용
```
UNDO (되돌리기):
- 커밋되지 않은 트랜잭션 취소
- old 값으로 복원
- T2는 COMMIT이 없으므로 UNDO

REDO (다시 하기):
- 커밋된 트랜잭션 재실행
- new 값으로 적용
- T1은 COMMIT이 있으므로 REDO
```

#### 4. 로그 기반 회복 기법
### 4.1 지연 갱신 (Deferred Update)
```
특징: 커밋 시점까지 디스크에 쓰지 않음

동작:
1. 트랜잭션 실행 중 → 로그에만 기록
2. COMMIT → 로그를 디스크에 기록
3. 그 후 실제 데이터 갱신

장점:
- UNDO 불필요 (디스크에 안 썼으니)
- REDO만 필요

단점:
- 큰 트랜잭션은 메모리 부족 가능
```

### 4.2 즉시 갱신 (Immediate Update)
```
특징: 실행 즉시 디스크에 반영 가능

동작:
1. 로그 기록 (old, new 모두)
2. 즉시 디스크에 반영 가능
3. 장애 시 UNDO/REDO 모두 가능

장점:
- 실시간 반영
- 유연성

단점:
- UNDO 필요
- 복잡함
```

#### 5. 체크포인트 (Checkpoint)
### 5.1 개념
```
정기적으로 시스템 상태를 디스크에 저장

체크포인트 수행:
1. 실행 중인 트랜잭션 일시 중단
2. 버퍼 내용을 디스크에 기록
3. <CHECKPOINT, T1, T2, ...> 로그 기록
   (실행 중인 트랜잭션 목록)

회복 시:
- 체크포인트 이전 로그는 검사 불필요
- 체크포인트 이후 로그만 검사
```

### 5.2 체크포인트 종류
```
1. 고정 체크포인트 (Consistent Checkpoint)
   - 모든 트랜잭션 중단
   - 버퍼 플러시
   - 일관된 상태 보장
   - 성능 저하

2. 퍼지 체크포인트 (Fuzzy Checkpoint)
   - 트랜잭션 계속 실행
   - 비동기 버퍼 플러시
   - 성능 향상
   - 실제 DBMS 사용 방식
```

### 5.3 회복 과정
```
로그: ─────────────────────────────────────>
       ↑                    ↑            ↑
    체크포인트            장애발생      복구시작
    (T1, T2실행중)

분류:
- T1: 체크포인트 전 시작, COMMIT 없음 → UNDO
- T2: 체크포인트 전 시작, COMMIT 있음 → REDO
- T3: 체크포인트 후 시작, COMMIT 있음 → REDO
- T4: 체크포인트 후 시작, COMMIT 없음 → UNDO
```

#### 6. ARIES 알고리즘
```
ARIES (Algorithm for Recovery and Isolation
       Exploiting Semantics)

현대 DBMS의 표준 회복 알고리즘

3단계:
┌─────────────────────────────────────┐
│ 1. Analysis (분석)                   │
│    - 로그 순차 스캔                  │
│    - RedoLSN 결정                    │
│    - Undo 리스트 구성                │
├─────────────────────────────────────┤
│ 2. Redo (재실행)                     │
│    - RedoLSN부터 로그 스캔           │
│    - 커밋된 변경 사항 재적용         │
│    - 데이터 일관성 복원              │
├─────────────────────────────────────┤
│ 3. Undo (취소)                       │
│    - 미완료 트랜잭션 취소            │
│    - 역순으로 로그 처리              │
│    - CLR (Compensation Log Record)   │
└─────────────────────────────────────┘

핵심 원칙:
- WAL (Write-Ahead Logging): 데이터보다 로그 먼저
- Repeating History: Redo로 정확한 상태 복원
- Logging Undos: Undo 작업도 로깅
```

#### 7. 백업 전략
### 7.1 백업 종류
```
1. 전체 백업 (Full Backup)
   - 전체 데이터베이스 복사
   - 시간 오래 걸림
   - 복구 간단

2. 차등 백업 (Differential Backup)
   - 마지막 전체 백업 이후 변경분
   - 전체 백업보다 빠름

3. 증분 백업 (Incremental Backup)
   - 마지막 백업 이후 변경분
   - 가장 빠름
   - 복구 복잡

4. 로그 백업 (Log Backup)
   - 트랜잭션 로그 백업
   - 특정 시점 복구 가능
```

### 7.2 백업 전략
```
일반적 전략:
┌─────────────────────────────────────┐
│ 일요일: 전체 백업                    │
│ 월~토: 증분 백업                     │
│ 매시간: 로그 백업                    │
└─────────────────────────────────────┘

복구 시나리오:
1. 일요일 전체 백업 복원
2. 월~금 증분 백업 순차 적용
3. 로그 백업으로 특정 시점 복구
```

#### 8. 데이터베이스 복제
### 8.1 복제 방식
```
동기식 복제 (Synchronous):
┌────────┐    COMMIT    ┌────────┐
│  주 DB  │ ──────────→ │  부 DB  │
└────────┘ ←─── ACK ─── └────────┘
- 데이터 손실 없음
- 성능 저하

비동기식 복제 (Asynchronous):
┌────────┐    COMMIT    ┌────────┐
│  주 DB  │ ──────────→ │  부 DB  │
└────────┘              └────────┘
        (나중에 복제)
- 성능 향상
- 일부 데이터 손실 가능
```

### 8.2 고가용성 구성
```
Active-Standby:
┌────────┐
│ Active │ ← 서비스
└────────┘
┌────────┐
│Standby │ ← 대기
└────────┘

Active-Active:
┌────────┐
│ Active │ ← 서비스
└────────┘
┌────────┐
│ Active │ ← 서비스
└────────┘
```

### Ⅳ. 실무 적용 방안

#### 11. 실무에선? (기술사적 판단)
- **로그 파일**: 별도 디스크에 저장 (I/O 분산)
- **체크포인트**: 5-10분 간격 권장
- **백업 전략**: RPO/RTO 기준 수립
- **HA 구성**: Active-Standby 기본, 중요 서비스 Active-Active
- **모니터링**: 로그 파일 크기, 체크포인트 빈도 모니터링
- **테스트**: 정기적 복구 훈련 필수
