# B-Tree와 B+ Tree - 효율적인 디스크 인덱스 자료구조

## 핵심 인사이트 (3줄 요약)
> **방대한 데이터를 디스크에서 효율적으로 탐색하기 위한 균형 트리**. B-Tree는 모든 노드에 데이터를 저장하고, B+ Tree는 리프 노드에만 데이터를 저장하며 노드 간 연결 리스트를 제공한다. 현대 데이터베이스 인덱스의 표준은 B+ Tree이다.

### Ⅰ. 개요

#### 1. 개념 및 특징
```
B-Tree (Balanced Tree):
- 모든 리프 노드가 같은 레벨에 있는 균형 다원 탐색 트리
- 노드 내 키(Key)가 오름차순 정렬
- M차 B-Tree: 노드가 최대 M개의 자식을 가짐

주요 특징:
- 대량의 데이터를 디스크에 저장할 때 I/O 최소화 (높이가 낮음)
- 자가 균형(Self-balancing)을 유지하며 최악의 경우에도 O(log n) 보장
- 각 노드가 여러 개의 키를 가져 한 번의 디스크 읽기로 많은 정보 획득
```

### Ⅱ. 구성 요소 및 핵심 원리

#### 2. B-Tree vs B+ Tree 비교 ★ (기술사 핵심)
| 항목 | B-Tree | B+ Tree |
|------|--------|---------|
| 데이터 저장 | 모든 노드(루트, 인터널, 리프)에 저장 | **리프 노드에만** 실제 데이터 저장 |
| 노드 키 저장 | 데이터와 함께 인덱스 키 저장 | 인덱스 노드에는 가이드용 키만 저장 |
| 리프 노드 연결 | 연결 없음 | **연결 리스트(Linked List)**로 연결됨 |
| 탐색 효율 | 운 좋으면 루트에서 바로 찾음 | 항상 리프 노드까지 내려가야 함 |
| 범위 스캔 | 트리를 여러 번 왕복해야 함 (비효율) | **매우 효율적** (연결 리스트 따라가기) |
| 메모리 활용 | 인덱스 노드에 데이터가 있어 무거움 | 인덱스 노드가 가벼워 더 많은 키 저장 가능 |
| 풀 스캔 | 트리 전체를 순회해야 함 | 리프 노드 연결 리스트만 훑으면 됨 |

#### 3. B+ Tree 아키텍처 (현대 DB의 표준)
```
        [ 20 | 40 ]           <-- 루트 노드 (가이드 키만 있음)
       /     |     \
  [10|15] [25|30] [45|50]     <-- 내부 인덱스 노드
   /  |    /  |    /  |
[5|7|8]-[10|12|14]-[20|24|28] <-- 리프 노드 (실제 데이터 + 연결 리스트)
```

**B+ Tree가 DB 인덱스로 선호되는 이유:**
1. **팬 아웃(Fan-out) 극대화**: 노드에 데이터가 없으므로 하나의 블록에 더 많은 키를 담을 수 있어 트리의 높이가 더 낮아짐.
2. **범위 검색(Range Search)**: `WHERE id BETWEEN 10 AND 100` 같은 쿼리를 처리할 때, 트리 탐색 후 리프 노드에서 옆으로만 이동하면 됨.
3. **안정성**: 모든 탐색이 동일한 깊이(리프트 노드)에서 끝내므로 응답 시간의 편차가 적음.

#### 4. B-Tree 연산 원리
### 삽입 (Insert)
1. 데이터를 넣을 적절한 리프 노드를 탐색
2. 해당 노드에 빈 자리가 있으면 정렬을 유지하며 삽입
3. 노드가 꽉 찼으면(Overflow), 중간값을 부모로 올리고 노드를 **분할(Split)**

### 삭제 (Delete)
1. 삭제할 키 탐색
2. 리프 노드에서 삭제 시 최소 키 개수(M/2) 미만이 되면(Underflow):
   - 형제 노드에서 키를 빌려옴 (Redistribution)
   - 형제 노드가 부족하면 형제와 **병합(Merge)**

#### 5. B-Tree 패밀리
| 자료구조 | 특징 |
|---------|------|
| **B-Tree** | 기본 다원 균형 트리 |
| **B+ Tree** | 리프 노드에만 데이터 저장, 리프 간 연결 리스트 추가 (DB 표준) |
| **B* Tree** | 노드 분할을 최소화하기 위해 형제 노드로 먼저 이동 (적재율 2/3 유지) |
| **LSM Tree** | 쓰기 성능 향상을 위해 메모리에 쓰다가 정기적으로 디스크 병합 (NoSQL 특화) |

#### 6. 성능 분석
| 연산 | 시간 복잡도 (B-Tree) | 사유 |
|------|-------------------|------|
| 탐색 | **O(log(M) N)** | 트리의 높이가 밑이 M인 로그에 비례 |
| 삽입 | **O(log(M) N)** | 탐색 후 분할 발생 시 부모로 전파될 수 있음 |
| 삭제 | **O(log(M) N)** | 탐색 후 병합 발생 시 부모로 전파될 수 있음 |

*M = 각 노드의 최대 자식 수 (Fan-out)*

### Ⅳ. 실무 적용 방안

#### 7. 실무 활용 사례
- **관계형 데이터베이스(RDBMS)**: MySQL(InnoDB), PostgreSQL, Oracle의 기본 인덱스 (대부분 B+ Tree)
- **파일 시스템**: NTFS, HFS+, EXT4 (디렉토리 및 파일 메타데이터 관리)
- **핵심 판단**: 랜덤 I/O를 줄여야 하는 디스크 기반 시스템에 최적
