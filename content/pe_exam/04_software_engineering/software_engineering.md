# 소프트웨어 공학 (Software Engineering)

## 핵심 인사이트 (3줄 요약)

### Ⅰ. 개요

#### 1. 개념
소프트웨어 공학은 **소프트웨어의 개발, 운영, 유지보수에 체계적인 공학적 원칙을 적용**하는 학문이다.

> 비유: "건축 공학" - 집을 짓는 데 체계적인 방법과 원칙을 적용하듯, 소프트웨어도 마찬가지

#### 2. 등장 배경: 소프트웨어 위기
```
1960년대 후반~1970년대 소프트웨어 위기:

문제점:
1. 개발 비용 증가
   - 하드웨어 비용 ↓
   - 소프트웨어 비용 ↑

2. 개발 기간 지연
   - 일정 초과 빈번

3. 품질 저하
   - 신뢰성 부족
   - 오류 다수

4. 유지보수 어려움
   - 문서 부족
   - 구조 파악 곤란

5. 사용자 요구 불만족
   - 요구사항 반영 실패

1968년 NATO 회의에서 "Software Engineering" 용어 처음 사용
```

### Ⅱ. 구성 요소 및 핵심 원리

#### 3. 소프트웨어 특성
```
1. 무형성 (Intangibility)
   - 눈에 보이지 않음
   - 물리적 속성 없음

2. 복잡성 (Complexity)
   - 하드웨어보다 복잡
   - 상태 조합 무한

3. 일치성 (Conformity)
   - 물리 법칙 따르지 않음
   - 인위적 제약

4. 변경성 (Changeability)
   - 끊임없는 수정
   - 버그 수정, 기능 추가

5. 비가시성 (Invisibility)
   - 구조 시각화 어려움
```

```
방법론 (Methodology):
- 어떻게 개발할 것인가
- 예: 폭포수, 애자일, 객체지향

도구 (Tool):
- 무엇을 사용할 것인가
- 예: IDE, 버전관리, 이슈트래커

프로세스 (Process):
- 어떤 순서로 할 것인가
- 예: 요구분석 → 설계 → 구현 → 테스트
```

#### 6. 소프트웨어 공학 원칙
```
1. 추상화 (Abstraction)
   - 불필요한 세부사항 숨김
   - 핵심 개념에 집중

2. 정보 은닉 (Information Hiding)
   - 모듈 내부 구조 숨김
   - 인터페이스만 공개

3. 모듈화 (Modularity)
   - 기능별 분리
   - 독립적 개발, 테스트

4. 단계적 분해 (Stepwise Refinement)
   - 상위에서 하위로
   - 점진적 상세화

5. 국소화 (Localization)
   - 관련 요소 모으기
   - 응집도 높이기

6. 균일성 (Uniformity)
   - 일관된 표기법, 스타일
   - 코딩 표준 준수

7. 완전성 (Completeness)
   - 모든 요구사항 충족
   - 누락 없음
```

#### 8. 소프트웨어 공학 지식 체계
```
SWEBOK (Software Engineering Body of Knowledge):

1. 요구사항 공학
2. 소프트웨어 설계
3. 소프트웨어 구축
4. 소프트웨어 테스트
5. 소프트웨어 유지보수
6. 소프트웨어 형상관리
7. 소프트웨어 공학 관리
8. 소프트웨어 공학 프로세스
9. 소프트웨어 공학 도구 및 방법
10. 소프트웨어 품질
11. 관련 학문 (수학, 컴퓨팅, 공학)
```

### Ⅲ. 기술 비교 분석

#### 9. 장단점
### 체계적 개발의 장점
| 장점 | 설명 |
|-----|------|
| 품질 향상 | 체계적인 프로세스 |
| 예측 가능성 | 일정, 비용 추정 |
| 유지보수성 | 문서화, 구조화 |
| 협업 용이 | 역할, 책임 명확 |

### 과도한 문서화의 단점
| 단점 | 설명 |
|-----|------|
| 시간 소요 | 문서 작성 부담 |
| 유연성 저하 | 변경 어려움 |
| 형식주의 | 내용보다 형식 |

### Ⅳ. 실무 적용 방안

#### 10. 실무에선? (기술사적 판단)
- **프로젝트 규모**: 소규모는 애자일, 대규모는 체계적 프로세스
- **안전 중요**: 의료, 항공 등은 엄격한 프로세스
- **하이브리드**: 워터폴 + 애자일 혼합
- **DevOps**: 개발과 운영의 통합
- **지속적 개선**: 프로세스 지속적 개선

### Ⅴ. 기대 효과 및 결론


| 효과 영역 | 내용 | 정량적 목표 |
|---------|-----|-----------|
| **개발 품질** | 체계적 방법론·테스트로 결함 조기 발견 및 수정 | 결함 밀도(Defect Density) 50% 감소 |
| **개발 생산성** | 자동화·표준화로 반복 작업 제거 및 협업 효율 향상 | 개발 속도 30~50% 향상 |
| **유지보수성** | 모듈화·문서화로 이후 변경·확장 비용 절감 | 유지보수 비용 40% 절감 |
