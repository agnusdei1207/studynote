# 폭포수 모델 (Waterfall Model)

## 핵심 인사이트 (3줄 요약)
> **순차적으로 진행하는 전통적 소프트웨어 개발 모델**. 요구분석→설계→구현→테스트→유지보수 순서. 단계가 끝나면 다음 단계로만 진행하며 되돌아가기 어렵다.

---

## 📝 기술사 모의답안 (2.5페이지 분량)

### 📌 예상 문제
> **"폭포수 모델 (Waterfall Model)의 개념과 구성 요소를 설명하고, 소프트웨어 품질 및 생산성 향상 측면에서의 적용 방안을 기술하시오."**

### Ⅰ. 개요

#### 1. 개념
폭포수 모델은 **1970년 Winston Royce가 제안한 순차적 소프트웨어 개발 방법론**으로, 각 단계가 폭포처럼 아래로만 흐르는 특징이 있다.

> 비유: "폭포" - 위에서 아래로만 떨어지고 거꾸로 올라갈 수 없음

---

### Ⅱ. 구성 요소 및 핵심 원리

#### 2. 폭포수 모델 단계
```
        ┌─────────────────┐
        │   요구사항 분석   │ ← 사용자 요구 수집, 명세화
        └────────┬────────┘
                 │ ↓
        ┌─────────────────┐
        │      설계       │ ← 시스템 구조, 상세 설계
        └────────┬────────┘
                 │ ↓
        ┌─────────────────┐
        │      구현       │ ← 코딩, 단위 테스트
        └────────┬────────┘
                 │ ↓
        ┌─────────────────┐
        │     테스트      │ ← 통합, 시스템, 인수 테스트
        └────────┬────────┘
                 │ ↓
        ┌─────────────────┐
        │    운영/유지보수  │ ← 배포, 유지보수
        └─────────────────┘

각 단계의 결과물:
요구분석 → 요구사항 명세서 (SRS)
설계 → 설계 문서 (SDD)
구현 → 소스 코드
테스트 → 테스트 보고서
운영 → 사용자 매뉴얼
```

#### 3. 각 단계 상세
```
1. 요구사항 분석 (Requirements Analysis)
   ┌─────────────────────────────────────┐
   │ - 사용자 요구 수집                  │
   │ - 요구사항 분석, 명세화             │
   │ - 요구사항 검토                     │
   │ - 결과물: 요구사항 명세서 (SRS)     │
   └─────────────────────────────────────┘

2. 설계 (Design)
   ┌─────────────────────────────────────┐
   │ - 시스템 아키텍처 설계              │
   │ - 상세 설계 (모듈, DB, UI)          │
   │ - 인터페이스 설계                   │
   │ - 결과물: 설계 문서 (SDD)           │
   └─────────────────────────────────────┘

3. 구현 (Implementation)
   ┌─────────────────────────────────────┐
   │ - 소스 코드 작성                    │
   │ - 단위 테스트                       │
   │ - 코드 리뷰                         │
   │ - 결과물: 실행 가능한 코드          │
   └─────────────────────────────────────┘

4. 테스트 (Testing)
   ┌─────────────────────────────────────┐
   │ - 통합 테스트                       │
   │ - 시스템 테스트                     │
   │ - 인수 테스트                       │
   │ - 결과물: 테스트 보고서             │
   └─────────────────────────────────────┘

5. 유지보수 (Maintenance)
   ┌─────────────────────────────────────┐
   │ - 운영 배포                         │
   │ - 오류 수정                         │
   │ - 기능 개선                         │
   │ - 결과물: 업데이트된 시스템         │
   └─────────────────────────────────────┘
```

#### 4. V-모델 (폭포수 변형)
```
폭포수 모델을 테스트 중심으로 확장한 것

        요구사항 분석 ─────────────── 인수 테스트
                 ↓                          ↑
             설계 ─────────────────── 시스템 테스트
                 ↓                          ↑
          상세 설계 ─────────────── 통합 테스트
                 ↓                          ↑
             구현 ──────────────── 단위 테스트
                 ↓                          ↑
                 └──────────────────────────┘
                           │
                    코딩 / 테스트

특징:
- 각 개발 단계에 대응하는 테스트 단계 존재
- 요구분석 → 인수 테스트 계획
- 설계 → 시스템 테스트 계획
- 상세 설계 → 통합 테스트 계획
- 구현 → 단위 테스트
```

### Ⅲ. 기술 비교 분석

#### 5. 장단점
### 장점
```
1. 명확한 단계 구분
   - 각 단계가 명확히 구분됨
   - 진행 상황 파악 용이

2. 문서화 중심
   - 모든 단계에서 문서 산출물
   - 유지보수에 유리

3. 관리 용이
   - 진척도 관리 쉬움
   - 예측 가능한 일정

4. 요구사항 고정
   - 초기에 요구사항 확정
   - 범위 관리 용이
```

### 단점
```
1. 유연성 부족
   - 요구사항 변경 어려움
   - 이전 단계로 복귀 곤란

2. 실제 개발과 불일치
   - 실제로는 순차적으로 진행 안 됨
   - 변경 불가피

3. 사용자 피드백 지연
   - 개발 후반에야 결과 확인
   - 잘못된 방향이면 낭비 큼

4. 위험 노출 지연
   - 문제를 나중에 발견
   - 수정 비용 증가
```

#### 8. 폭포수 모델 vs 애자일
| 항목 | 폭포수 모델 | 애자일 |
|------|------------|--------|
| 접근 방식 | 순차적 | 반복적 |
| 요구사항 | 초기 고정 | 지속적 변경 |
| 문서화 | 강조 | 최소화 |
| 고객 참여 | 시작/끝 | 지속적 |
| 변경 대응 | 어려움 | 유연함 |
| 위험 노출 | 후반 | 초기 |

---

### Ⅳ. 실무 적용 방안

#### 6. 적용 분야
```
적합한 경우:
✓ 요구사항이 명확하고 변경이 적음
✓ 안전 중요 시스템 (원자력, 의료)
✓ 대규모 프로젝트
✓ 계약 기반 개발
✓ 경험이 적은 팀

부적합한 경우:
✗ 요구사항이 불명확
✗ 빠른 변경이 필요
✗ 혁신적인 제품 개발
✗ 스타트업, 초기 단계
```

#### 9. 실무에선? (기술사적 판단)
- **안전 중요 시스템**: 의료, 항공, 원자력 → 폭포수 권장
- **공공사업**: 입찰, 계약 기반 → 폭포수 적합
- **SI 프로젝트**: 명확한 요구사항 → 폭포수 기반
- **하이브리드**: 단계는 폭포수, 내부는 애자일
- **V-모델**: 테스트 중심 필요 시 변형 모델 사용

### Ⅴ. 기대 효과 및 결론


| 효과 영역 | 내용 | 정량적 목표 |
|---------|-----|-----------|
| **개발 품질** | 체계적 방법론·테스트로 결함 조기 발견 및 수정 | 결함 밀도(Defect Density) 50% 감소 |
| **개발 생산성** | 자동화·표준화로 반복 작업 제거 및 협업 효율 향상 | 개발 속도 30~50% 향상 |
| **유지보수성** | 모듈화·문서화로 이후 변경·확장 비용 절감 | 유지보수 비용 40% 절감 |

#### 결론
> **폭포수 모델 (Waterfall Model)**은(는) 소프트웨어 공학 방법론은 AI 보조 코딩(GitHub Copilot), 로우코드 플랫폼, 플랫폼 엔지니어링의 부상으로 개발자의 인지 부하를 줄이면서 품질과 속도를 동시에 확보하는 방향으로 진화하고 있다.

> **※ 참고 표준**: ISO/IEC 25010(SQuaRE), IEEE 830, CMMI v2.0, OWASP
