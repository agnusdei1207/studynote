# 애자일 방법론 (Agile Methodology)

## 핵심 인사이트 (3줄 요약)
> **변화에 유연하게 대응하는 반복적 소프트웨어 개발 방식**. 짧은 주기로 개발하고 피드백을 반영. 협업, 고객 중심, 작동하는 소프트웨어를 중시한다.

### Ⅰ. 개요

#### 1. 개념
애자일(Agile)은 **2001년 애자일 선언문을 통해 등장한 유연한 소프트웨어 개발 방법론**으로, 변화를 수용하고 고객과 협업하며 작동하는 소프트웨어를 빠르게 전달하는 것을 목표로 한다.

> 비유: "요리의 맛보기" - 완성 전에 계속 맛을 보며 조리

---

### Ⅱ. 구성 요소 및 핵심 원리

#### 2. 애자일 선언문 (Agile Manifesto)
```
2001년, 17명의 소프트웨어 개발자가 모여 선언

┌─────────────────────────────────────────────────────────┐
│                애자일 소프트웨어 개발 선언               │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  우리는 소프트웨어를 개발하면서 더 나은 방법을 찾고     │
│  다른 사람들을 도와주면서 다음 가치를 더 중요하게       │
│  여깁니다:                                              │
│                                                         │
│  공정과 도구보다     [개인과 상호작용]을                 │
│  포괄적인 문서보다   [작동하는 소프트웨어]을             │
│  계약 협상보다       [고객과의 협력]을                   │
│  계획을 따르기보다   [변화에 대응]을                     │
│                                                         │
│  즉, 오른쪽의 가치도 중요하지만                         │
│  우리는 왼쪽의 가치에 더 높은 가치를 둡니다.            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 3. 애자일 12원칙
```
1. 고객 만족
   - 가치 있는 소프트웨어를 빠르고 지속적으로 전달

2. 변화 수용
   - 개발 후반라도 요구사항 변경을 환영

3. 짧은 릴리즈
   - 몇 주~몇 개월 단위로 작동하는 소프트웨어 전달

4. 협업
   - 비즈니스 담당자와 개발자는 매일 함께 일함

5. 동기 부여된 팀
   - 의지 있는 개인 중심, 환경과 지원 제공

6. 대면 커뮤니케이션
   - 효율적인 정보 전달 방법

7. 작동하는 소프트웨어
   - 진척도의 1차 척도

8. 지속 가능한 속도
   - 일정한 속도로 무기한 개발 가능

9. 기술적 우수성
   - 좋은 설계와 기술적 탁월성에 주목

10. 단순화
    - 안 하는 일의 양을 최대화

11. 자기 조직화 팀
    - 스스로 최선의 아키텍처, 요구사항, 설계 도출

12. 회고
    - 정기적으로 효율성을 반성하고 개선
```

#### 5. 주요 애자일 방법론
### 5.1 스크럼 (Scrum)
```
특징:
- 2-4주 스프린트
- 명확한 역할 (PO, SM, 팀)
- 정기 회의 (데일리, 리뷰, 회고)

구조:
┌─────────────────────────────────────┐
│            Product Backlog          │
│        (제품 백로그: 할 일 목록)    │
└────────────────┬────────────────────┘
                 │ 스프린트 계획
                 ▼
┌─────────────────────────────────────┐
│           Sprint Backlog            │
│      (스프린트 백로그: 이번 할 일)  │
└────────────────┬────────────────────┘
                 │ 2-4주 개발
                 ▼
┌─────────────────────────────────────┐
│           Potentially Shippable     │
│              Increment              │
│       (배포 가능한 제품 증분)       │
└─────────────────────────────────────┘
```

### 5.2 XP (Extreme Programming)
```
특징:
- 짧은 주기 (1-2주)
- 테스트 주도 개발 (TDD)
- 페어 프로그래밍
- 지속적 통합 (CI)

핵심 실천법:
┌─────────────────────────────────────┐
│ 1. Pair Programming (페어 프로그래밍)│
│ 2. TDD (테스트 주도 개발)           │
│ 3. CI (지속적 통합)                 │
│ 4. Simple Design (단순한 설계)      │
│ 5. Refactoring (리팩토링)           │
│ 6. Collective Ownership (공유 소유) │
│ 7. On-site Customer (현장 고객)     │
│ 8. Sustainable Pace (지속 가능 속도)│
└─────────────────────────────────────┘
```

### 5.3 칸반 (Kanban)
```
특징:
- 지속적 흐름
- WIP(Work In Progress) 제한
- 시각화 보드

칸반 보드:
┌───────────┬───────────┬───────────┬───────────┐
│   할 일   │   진행중   │   검토중   │   완료    │
│   (To Do) │ (Doing)   │ (Review)  │  (Done)   │
├───────────┼───────────┼───────────┼───────────┤
│   [WIP 3] │  [WIP 2]  │  [WIP 2]  │           │
├───────────┼───────────┼───────────┼───────────┤
│ □ 기능1   │ ■ 기능4   │ ■ 기능6   │ ✓ 기능8   │
│ □ 기능2   │ ■ 기능5   │ ■ 기능7   │ ✓ 기능9   │
│ □ 기능3   │           │           │ ✓ 기능10  │
└───────────┴───────────┴───────────┴───────────┘

WIP 제한:
- 동시 진행 작업 수 제한
- 병목 현상 방지
```

### Ⅲ. 기술 비교 분석

#### 4. 애자일 vs 전통적 방법론
```
┌────────────────┬─────────────────┬─────────────────┐
│      항목       │    전통적 방법   │     애자일      │
├────────────────┼─────────────────┼─────────────────┤
│ 접근 방식      │ 순차적          │ 반복적, 점진적  │
│ 요구사항      │ 초기에 고정     │ 지속적 변경     │
│ 문서화        │ 상세한 문서     │ 최소 문서       │
│ 고객 참여     │ 시작/끝        │ 지속적          │
│ 변경 비용     │ 높음            │ 낮음            │
│ 위험 노출     │ 후반            │ 초기            │
│ 팀 구성       │ 전문 분야별     │ 교차 기능 팀    │
│ 성공 기준     │ 계획 준수       │ 고객 가치       │
└────────────────┴─────────────────┴─────────────────┘
```

#### 8. 장단점
### 장점
| 장점 | 설명 |
|-----|------|
| 유연성 | 변경에 빠르게 대응 |
| 고객 만족 | 지속적 피드백 반영 |
| 품질 향상 | 빈번한 테스트, 리팩토링 |
| 위험 감소 | 초기에 문제 발견 |
| 협업 강화 | 팀 간 긴밀한 소통 |

### 단점
| 단점 | 설명 |

| 팀 역량 의존 | 숙련된 팀 필요 |

### Ⅳ. 실무 적용 방안

#### 9. 실무에선? (기술사적 판단)
- **스타트업**: 애자일 필수, 빠른 시장 적응
- **웹/모바일**: 2주 스프린트, 지속적 배포
- **SI/SM**: 하이브리드 (계약은 워터폴, 개발은 애자일)
- **대규모**: SAFe, LeSS 등 스케일드 애자일
- **도구**: Jira, Trello, Azure DevOps

| 효과 영역 | 내용 | 정량적 목표 |
