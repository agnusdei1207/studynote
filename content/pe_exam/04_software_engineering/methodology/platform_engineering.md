# 플랫폼 엔지니어링 (Platform Engineering) - DevOps의 진화

## 핵심 인사이트 (3줄 요약)
> **개발자가 인프라 복잡성에 신경 쓰지 않고 비즈니스 로직에만 집중할 수 있게 "내부 개발자 플랫폼(IDP)"을 구축하는 전략**. DevOps가 불러온 '인지적 과부하(Cognitive Overload)'를 해결하기 위한 대안이다. "Golden Path(표준 경로)"를 제공하여 셀프 서비스 기반의 운영을 실현한다.

### Ⅰ. 개요

#### 1. 등장 배경: DevOps의 피로도
```
DevOps의 문제 (You Build It, You Run It):
- 개발자가 코드만 짜는 게 아니라 K8s, 테라폼, 모니터링, 보안 설정까지 다 해야 함.
- 툴 체인의 파편화로 인해 개발자의 인지적 부하(Cognitive Load) 임계치 도달.
- 결과적으로 개발 속도가 저하되고 운영 리스크 증가.

플랫폼 엔지니어링:
- 운영 전문가가 "플랫폼(IDP)"이라는 제품을 만들어 개발자에게 제공.
- 개발자는 버튼 몇 번으로 인프라 배포 완료.
```

---

### Ⅱ. 구성 요소 및 핵심 원리

#### 2. IDP (Internal Developer Platform)의 핵심 기능 ★ (핵심)
| 기능 | 설명 |
|-----|------|
| **Self-Service** | 티켓 처리(Ticket-ops) 없이 개발자가 직접 자원 할당 및 배포 가능 |
| **Golden Path** | 회사의 표준 보안 및 운영 규정을 준수하는 '사전 정의된 경로' 제공 |
| **Infrastructure Orchestration** | IaC(Terraform, Crossplane) 기반으로 인프라 생성 및 관리 자동화 |
| **Observability & Ops** | 로그, 메트릭, 비용 관리 대시보드 통합 제공 |
| **Governance** | 플랫폼 단에서 보안 취약점 점검 및 감사 로그 자동 생성 |

#### 4. 플랫폼 엔지니어링의 핵심 원칙
1. **플랫폼을 제품으로 보기 (Platform as a Product)**: 사용자(개발자)의 피드백을 받아 지속적으로 개선.
2. **사고방식의 전환 (Enablement over Gatekeeping)**: 검사나 통제가 아니라 '가능하게(Enabling)' 돕는 도구로 접근.
3. **불필요한 수작업 제거 (Minimize Glue Code)**: 스크립트 땜질이 아닌 통합된 사용자 인터페이스 제공.

#### 5. 팀 토폴로지 (Team Topologies) 연계 ★ (중요)
- **Platform Team**: 다른 팀이 사용할 기반 플랫폼 구축.
- **Stream-aligned Team**: 실제 비즈니스 가치를 만드는 개발팀 (플랫폼 위에서 달림).
- **Enabling Team**: 기술적 격차를 줄여주는 도우미 팀.

#### 6. 주요 기술 스택
- **Portal**: Backstage (Spotify가 만든 오픈소스, 현재 사실상 표준), Port.
- **Delivery**: ArgoCD (GitOps 시스템), Helm.
- **Infrastructure**: Terraform, Crossplane, AWS/Azure/GCP.
- **Communication**: Slack/Teams 기반의 Chat-ops.

### Ⅲ. 기술 비교 분석

#### 3. 플랫폼 엔지니어링 vs DevOps 비교 ★ (기술사 필수)
| 항목 | DevOps (원칙) | 플랫폼 엔지니어링 (구현) |
|-----|---------------|-----------------------|
| **핵심 철학** | 개발과 운영의 벽 허물기 | **개발자의 인지 부하 감소** |
| **책임 범위** | 개발자가 운영까지 전체 책임 | 플랫폼 팀이 플랫폼 책임, 개발자는 사용 |
| **접근 방식** | 문화 및 절차의 협업 강조 | **플랫폼이라는 '제품' 제공** |
| **주체** | 서비스 개발팀 전체 | 별도의 **플랫폼 전담 팀** (Platform Team) |
| **관계** | DevOps 실현을 위한 방법론 | **DevOps를 대규모 조직에서 확장 가능하게 하는 도구** |

### Ⅳ. 실무 적용 방안

#### 7. 실무 및 기술사적 판단
- **효과**: 개발자 온보딩 시간 단축(Target: 하루 이내), 배포 주기 단축, 보안 가드레일 강화.
- **이슈**: "또 다른 운영팀 아냐?"라는 비판이 있을 수 있으나, '티켓 기반'이 아닌 'API/셀프서비스' 기반이라는 점이 핵심 차이임.
- **가트너 전망**: 2026년까지 소프트웨어 엔지니어링 조직의 80%가 플랫폼 엔지니어링 전담 팀을 구성할 것으로 예측.

> **플랫폼 엔지니어링 (Platform Engineering) - DevOps의 진화**은(는) 소프트웨어 공학 방법론은 AI 보조 코딩(GitHub Copilot), 로우코드 플랫폼, 플랫폼 엔지니어링의 부상으로 개발자의 인지 부하를 줄이면서 품질과 속도를 동시에 확보하는 방향으로 진화하고 있다.
