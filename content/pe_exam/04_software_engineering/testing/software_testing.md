# 소프트웨어 테스트 (Software Testing)

## 핵심 인사이트 (3줄 요약)
> **소프트웨어의 결함을 발견하고 품질을 검증**하는 활동. 단위→통합→시스템→인수 테스트로 진행. 블랙박스(명세 기반)와 화이트박스(구조 기반)로 분류.

---

## 📝 기술사 모의답안 (2.5페이지 분량)

### 📌 예상 문제
> **"소프트웨어 테스트 (Software Testing)의 개념과 구성 요소를 설명하고, 소프트웨어 품질 및 생산성 향상 측면에서의 적용 방안을 기술하시오."**

---

### Ⅰ. 개요

#### 1. 개념
소프트웨어 테스트는 **소프트웨어가 요구사항을 만족하는지 확인하고 결함을 발견**하기 위한 활동이다.

> 비유: "자동차 안전 검사" - 출고 전에 모든 기능을 테스트

---

### Ⅱ. 구성 요소 및 핵심 원리

#### 2. 테스트의 목적
```
1. 결함 발견 (Defect Detection)
   - 버그 찾기
   - 품질 문제 식별

2. 품질 검증 (Quality Verification)
   - 요구사항 충족 확인
   - 기능적/비기능적 요구 검증

3. 신뢰성 확보 (Reliability Assurance)
   - 오류 없는 실행 보장
   - 사용자 신뢰 구축

4. 정보 제공 (Information Providing)
   - 상태 파악
   - 의사결정 지원
```

#### 3. 테스트 레벨 (V-모델)
```
          요구사항 분석 ──────────────── 인수 테스트
                   ↓                          ↑
               설계 ────────────────── 시스템 테스트
                   ↓                          ↑
            상세 설계 ──────────────── 통합 테스트
                   ↓                          ↑
               구현 ────────────────── 단위 테스트
                   ↓                          ↑
                   └──────────────────────────┘

┌────────────────────────────────────────────────────────┐
│                   테스트 레벨                           │
├────────────────────────────────────────────────────────┤
│                                                        │
│  1. 단위 테스트 (Unit Test)                            │
│     - 대상: 모듈, 함수, 클래스                         │
│     - 수행자: 개발자                                   │
│     - 방법: 화이트박스                                 │
│                                                        │
│  2. 통합 테스트 (Integration Test)                     │
│     - 대상: 모듈 간 인터페이스                         │
│     - 수행자: 개발자/테스터                            │
│     - 방법: 화이트박스 + 블랙박스                      │
│                                                        │
│  3. 시스템 테스트 (System Test)                        │
│     - 대상: 전체 시스템                                │
│     - 수행자: 테스터                                   │
│     - 방법: 블랙박스                                   │
│                                                        │
│  4. 인수 테스트 (Acceptance Test)                      │
│     - 대상: 사용자 요구사항                            │
│     - 수행자: 사용자/고객                              │
│     - 방법: 블랙박스                                   │
│                                                        │
└────────────────────────────────────────────────────────┘
```

#### 4. 테스트 기법 분류
### 4.1 테스트 레벨별
```
┌────────────────────────────────────────────────────────┐
│                     테스트 기법                         │
├────────────────────────────────────────────────────────┤
│                                                        │
│  정적 테스트 (Static Testing)                          │
│  - 코드 실행 없이 검사                                 │
│  - 코드 리뷰, 정적 분석                                │
│                                                        │
│  동적 테스트 (Dynamic Testing)                         │
│  - 코드 실행하며 검사                                  │
│  - 블랙박스, 화이트박스                                │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 4.2 블랙박스 vs 화이트박스
```
┌────────────────┬─────────────────┬─────────────────┐
│      항목       │   블랙박스      │   화이트박스    │
├────────────────┼─────────────────┼─────────────────┤
│ 내부 구조      │ 모름            │ 앎              │
│ 기반          │ 명세, 요구사항  │ 코드, 구조      │
│ 관점          │ 사용자          │ 개발자          │
│ 기법          │ 동등분할, 경계값│ 구문, 조건 커버 │
│ 언제          │ 시스템, 인수    │ 단위, 통합      │
└────────────────┴─────────────────┴─────────────────┘
```

#### 5. 블랙박스 테스트 기법
### 5.1 동등 분할 (Equivalence Partitioning)
```
입력을 동등한 그룹으로 나누어 대표값만 테스트

예: 나이 입력 (0-150)
┌───────────┬───────────┬───────────┐
│  유효하지  │   유효함   │  유효하지  │
│   않음     │  0-150    │   않음     │
│   < 0     │           │   > 150   │
└───────────┴───────────┴───────────┘
     -1            25           200

테스트 케이스:
- -1 (무효)
- 25 (유효)
- 200 (무효)
```

### 5.2 경계값 분석 (Boundary Value Analysis)
```
경계값에서 오류 발생 확률이 높음

예: 1-100 점수 입력
┌────────────────────────────────────────┐
│ 0 │ 1 │ 2 │ ... │ 99 │ 100 │ 101 │    │
└───┴───┴───┴─────┴────┴─────┴─────┘
  ↑   ↑           ↑    ↑     ↑
  최소-1 최소   최대-1 최대  최대+1

테스트 케이스:
- 0 (경계 미만)
- 1 (경계)
- 100 (경계)
- 101 (경계 초과)
```

### 5.3 원인-결과 그래프 (Cause-Effect Graph)
```
입력 조건(원인)과 출력(결과)의 관계 분석

예: 로그인
원인:
C1: ID 올바름
C2: PW 올바름

결과:
E1: 로그인 성공
E2: 에러 메시지

      C1 ──┐
           ├── AND ── E1
      C2 ──┘

      NOT C1 ──┐
               ├── OR ── E2
      NOT C2 ──┘
```

#### 6. 화이트박스 테스트 기법
### 6.1 검증 기준
```
┌────────────────────────────────────────────────────────┐
│                  화이트박스 검증 기준                   │
├────────────────────────────────────────────────────────┤
│                                                        │
│  1. 구문 커버리지 (Statement Coverage)                 │
│     - 모든 문장이 최소 1회 실행                        │
│     - 기준: 100%                                       │
│                                                        │
│  2. 분기 커버리지 (Branch Coverage)                    │
│     - 모든 분기(T/F)가 최소 1회 실행                   │
│     - 기준: 100%                                       │
│                                                        │
│  3. 조건 커버리지 (Condition Coverage)                 │
│     - 모든 조건이 T/F가 최소 1회 실행                  │
│                                                        │
│  4. 분기/조건 커버리지                                 │
│     - 분기 + 조건 모두 만족                            │
│                                                        │
│  5. 변형 조건/분기 커버리지 (MC/DC)                    │
│     - 각 조건이 독립적으로 결과에 영향                 │
│                                                        │
│  6. 다중 조건 커버리지                                 │
│     - 모든 조건 조합 테스트                            │
│                                                        │
│  7. 경로 커버리지 (Path Coverage)                      │
│     - 모든 실행 경로 테스트                            │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 6.2 기본 경로 테스트
```
순환 복잡도(Cyclomatic Complexity) 기반

V(G) = E - N + 2
E: 엣지 수, N: 노드 수

또는
V(G) = P + 1
P: 조건문 수

예:
     ┌─────┐
     │ 시작│
     └──┬──┘
        │
     ┌──▼──┐
     │ if  │ ← 분기
     └──┬──┘
    ┌───┴───┐
    ▼       ▼
   ┌──┐   ┌──┐
   │T │   │F │
   └──┘   └──┘
    │       │
    └───┬───┘
        │
     ┌──▼──┐
     │ 끝  │
     └─────┘

V(G) = 2 (분기 1개)
→ 최소 2개 테스트 케이스 필요
```

#### 7. 테스트 종류
```
1. 회귀 테스트 (Regression Test)
   - 수정 후 기존 기능 영향 확인
   - 자동화 필수

2. 스모크 테스트 (Smoke Test)
   - 주요 기능 빠르게 확인
   - 빌드 후 즉시 수행

3. 알파/베타 테스트
   - 알파: 개발 환경, 내부 사용자
   - 베타: 실제 환경, 외부 사용자

4. 성능 테스트
   - 부하 테스트
   - 스트레스 테스트
   - 병목 테스트

5. 보안 테스트
   - 취약점 스캔
   - 침투 테스트

6. 사용성 테스트
   - UI/UX 평가
   - A/B 테스트
```

#### 9. 테스트 원칙
```
1. 살충제 패러독스 (Pesticide Paradox)
   - 같은 테스트 반복하면 새 버그 못 찾음
   - 테스트 케이스 지속적 갱신 필요

2. 테스트는 결함이 있음을 보여줌
   - 결함이 없음을 증명할 수 없음

3. 완전 테스트 불가능
   - 모든 경로, 입력 조합 불가능
   - 위험 기반으로 우선순위 결정

4. 테스트 일찍 시작
   - 비용 적게 듦
   - 조기 결함 발견

5. 결함 집중 (Defect Clustering)
   - 80% 결함이 20% 모듈에 집중
   - 파레토 법칙
```

### Ⅲ. 기술 비교 분석

#### 10. 장단점
### 테스트의 장점
| 장점 | 설명 |
|-----|------|
| 품질 향상 | 결함 조기 발견 |
| 비용 절감 | 나중에 고치는 비용 큼 |
| 신뢰성 | 사용자 신뢰 |
| 문서화 | 테스트가 명세 역할 |

### 테스트의 한계
| 한계 | 설명 |
|-----|------|
| 완전 불가 | 모든 경우 테스트 불가 |
| 시간 소요 | 충분한 테스트에 시간 필요 |
| 인력 필요 | 테스트 전문가 필요 |
| 자동화 비용 | 초기 투자 필요 |

### Ⅳ. 실무 적용 방안

#### 11. 실무에선? (기술사적 판단)
- **TDD**: 테스트 주도 개발로 품질 향상
- **자동화**: CI/CD 파이프라인 통합
- **커버리지**: 70-80% 목표, 100%는 비효율
- **위험 기반**: 중요 기능 우선 테스트
- **도구**: JUnit, pytest, Selenium, JMeter

### Ⅴ. 기대 효과 및 결론


| 효과 영역 | 내용 | 정량적 목표 |
|---------|-----|-----------|
| **개발 품질** | 체계적 방법론·테스트로 결함 조기 발견 및 수정 | 결함 밀도(Defect Density) 50% 감소 |
| **개발 생산성** | 자동화·표준화로 반복 작업 제거 및 협업 효율 향상 | 개발 속도 30~50% 향상 |
| **유지보수성** | 모듈화·문서화로 이후 변경·확장 비용 절감 | 유지보수 비용 40% 절감 |

#### 결론
> **소프트웨어 테스트 (Software Testing)**은(는) 소프트웨어 공학 방법론은 AI 보조 코딩(GitHub Copilot), 로우코드 플랫폼, 플랫폼 엔지니어링의 부상으로 개발자의 인지 부하를 줄이면서 품질과 속도를 동시에 확보하는 방향으로 진화하고 있다.

> **※ 참고 표준**: ISO/IEC 25010(SQuaRE), IEEE 830, CMMI v2.0, OWASP
