# 570. 데이터 관리 패턴 (CQRS, Event Sourcing, Database per Service)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 관리 패턴은 분산 아키텍처(MSA)에서 발생하는 데이터 파편화와 일관성 문제를 해결하기 위해, **데이터의 소유권(Isolation), 변경 이력(History), 조회 및 명령의 분리(Optimization)**를 설계하는 전략적 기법이다.
> 2. **가치**: 서비스별 독립 DB를 통해 결합도를 낮추고(Database per Service), 모든 상태 변경을 이벤트로 기록하여 데이터의 무결성과 복구 능력을 높이며(Event Sourcing), 조회 성능을 극대화(CQRS)하여 사용자 경험과 시스템 확장성을 동시에 잡는다.
> 3. **융합**: 메시지 큐(Kafka), NoSQL, 그리고 도메인 주도 설계(DDD)와 결합되어 복잡한 비즈니스 데이터 생태계를 구축하며, 특히 '최종 일관성(Eventual Consistency)' 모델을 통해 대규모 동시성 처리를 실현한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 마이크로서비스로 전환하면 가장 먼저 부딪히는 벽이 '데이터'다. 
  - **Database per Service**: 각 서비스가 자기만의 DB를 가짐. (간섭 배제)
  - **Event Sourcing**: 상태 값이 아닌, 상태를 변화시킨 '사건(Event)' 자체를 저장함. (이력 보존)
  - **CQRS**: 데이터를 쓰는 길과 읽는 길을 따로 만듦. (성능 최적화)

- **💡 비유**: 데이터 관리 패턴은 **"은행의 장부 관리 시스템"**과 같습니다. 1) 지점마다 자기 금고를 따로 관리하고(Database per Service), 2) 최종 잔액만 적는 게 아니라 입출금 내역 전체를 꼼꼼히 기록하며(Event Sourcing), 3) 입금 창구와 조회 창구를 분리하여 손님을 빨리 처리하는(CQRS) 것과 같습니다.

- **등장 배경**:
  1. **공유 DB의 한계**: 수십 개의 서비스가 하나의 DB를 공유하면 테이블 구조 하나 바꿀 때마다 모든 서비스가 멈추는 '강한 결합' 문제가 발생했다.
  2. **감사(Audit) 및 추적 요구**: 결과 데이터만 있으면 "왜 이 값이 되었는지" 알 수 없어, 사고 발생 시 원인 규명을 위한 상세 이력이 필요해졌다.
  3. **조회 성능의 병목**: 비즈니스 로직은 복잡한데 조회는 단순하고 빨라야 하는 서비스 불균형을 해결하기 위해 읽기/쓰기 모델의 분리가 요구되었다.

- **📢 섹션 요약 비유**: 찰흙 한 덩어리(공유 DB)로 모든 걸 만들려다 망치는 대신, 조각조각 부품을 나누고(분산) 제작 과정(이벤트)을 모두 기록하며 전시용 모델(조회 전용)을 따로 만드는 정교한 예술 작업과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 3대 핵심 데이터 관리 패턴 비교

| 패턴 명칭 | 핵심 원리 | 주요 목적 | 비유 |
|:---|:---|:---|:---|
| **Database per Service** | 서비스마다 전용 DB 구축 및 직접 접근 금지 | **결합도 제거**, 독립적 기술 선택 | 각자 지갑 쓰기 |
| **Event Sourcing** | 데이터 변경을 이벤트 스트림으로 저장 | **완벽한 이력 추적**, 상태 복구 | 회계 장부 기록 |
| **CQRS** | Command(CUD)와 Query(R) 모델 분리 | **조회 성능 극대화**, 모델 단순화 | 입구와 출구 분리 |

---

### 2. 통합 데이터 관리 아키텍처 (CQRS + Event Sourcing)

현대 분산 시스템에서 가장 강력하게 권장되는 융합 아키텍처를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      CQRS 및 Event Sourcing 통합 데이터 흐름                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. Command (Write)]                      [2. Query (Read)]                 │
 │      │                                           ▲                          │
 │      ▼                                           │ (Fast Read)              │
 │  ┌──────────────────┐                    ┌──────────────────┐               │
 │  │ Domain Model     │                    │ Read Model       │               │
 │  │ (Write-only DB)  │                    │ (Materialized)   │               │
 │  └──────┬───────────┘                    └──────▲───────────┘               │
 │         │                                       │                           │
 │         ▼ [3. Event Store]                      │                           │
 │  ┌──────────────────────────────────┐           │                           │
 │  │ Event A (Created)                │           │                           │
 │  │ Event B (Updated)                │ ──────────┘                           │
 │  │ Event C (Paid)                   │ (Asynchronous Sync)                   │
 │  └──────────────────────────────────┘                                       │
 │                                                                             │
 │  ※ 핵심: 데이터 상태는 이벤트들의 합으로 결정됨 (State = Sum of Events)           │
 │  ※ 가치: 읽기 모델은 검색에 최적화된 NoSQL 등을 자유롭게 선택 가능함               │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 중심에는 **Event Store**가 있다. 1) **Command Side**에서는 사용자의 요청을 받아 유효성을 검증하고 '사건(Event)'을 발생시켜 저장한다. 이때 현재 상태를 덮어쓰는 게 아니라 새로운 이벤트를 추가(Append-only)한다. 2) 저장된 이벤트는 비동기적으로 **Query Side**에 전달된다. 3) **Read Model**은 이 이벤트를 받아 조회하기 딱 좋은 형태(예: Elasticsearch 인덱스)로 미리 가공해둔다. 사용자가 정보를 찾을 때는 복잡한 조인(Join) 없이 이 읽기 전용 모델에서 즉시 데이터를 가져간다.

---

### 심층 동작 원리: 스냅샷 (Snapshot) 및 재생 (Replay)

Event Sourcing의 성능 문제를 해결하는 기술이다.
1. **문제**: 이벤트가 100만 개 쌓이면, 현재 잔액을 알기 위해 100만 번을 다 더해봐야 하므로 느림.
2. **해결 (Snapshot)**: 1,000번째 이벤트마다 현재 상태를 별도로 저장해둠.
3. **재생 (Replay)**: 상태가 꼬였을 때, 과거 특정 시점의 스냅샷부터 다시 이벤트를 적용하여 완벽하게 과거로 되돌리거나 시뮬레이션 가능.

- **📢 섹션 요약 비유**: 영화 전체를 처음부터 다시 보는(Replay) 게 아니라, 주요 장면(Snapshot)부터 빠르게 감기해서 원하는 장면을 찾는 스마트한 재생 기법과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Database per Service vs Shared Database

| 비교 항목 | Database per Service (MSA) | Shared Database (Monolith) |
|:---|:---|:---|
| **독립성** | 매우 높음 (영향 범위 0) | 낮음 (스키마 변경 시 전체 중단) |
| **데이터 정합성** | **최종 일관성 (보상 트랜잭션)** | **강력한 일관성 (ACID)** |
| **복잡도** | 높음 (분산 쿼리 필요) | 낮음 (조인만 하면 됨) |
| **기술 스택** | 서비스별 최적 DB 선택 가능 | 하나의 DB 엔진에 고정 |

---

### 기술 융합: Saga 패턴과의 시너지

데이터 관리 패턴은 트랜잭션 패턴인 **Saga**와 짝을 이룬다.
- **연결**: Database per Service로 인해 쪼개진 DB들의 정합성을 맞추기 위해, **Event Sourcing**으로 생성된 이벤트를 활용하여 Saga의 보상 로직을 실행한다.
- **결과**: 기술적 한계(분산 DB)를 아키텍처 패턴(Saga + Event)으로 극복하여 비즈니스의 신뢰성을 유지한다.

- **📢 섹션 요약 비유**: 각자 다른 장부를 쓰면서도(분산 DB), 소식을 실시간으로 주고받아(Event) 전 세계 모든 은행의 잔액을 오차 없이 맞추는 거대한 금융망과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 배달 앱의 '실시간 위치 및 상태' 시스템 설계

- **문제 상황**: 주문이 발생할 때마다 배달원의 위치, 조리 상태 등을 수천 번 조회하는데, 하나의 DB에서 주문 처리(Write)와 위치 조회(Read)를 같이 하니 DB가 뻗어버림.
- **의사결정 (데이터 패턴 적용)**:
  1. **CQRS 분리**: 주문 생성과 라이더 배정은 **RDBMS (Command DB)**에서 처리하고, 배달 현황 조회는 **Redis (Read DB)**로 분리한다.
  2. **Event Sourcing**: "라이더 출발", "치킨 튀기는 중" 등의 모든 상태 변화를 **Kafka (Event Store)**에 기록한다.
  3. **Projection**: Kafka의 이벤트를 소비하는 컨슈머가 Redis의 데이터를 실시간으로 업데이트하여 조회 속도를 0.01초 내로 단축한다.
  4. **결과**: 쓰기 작업이 몰려도 조회 시스템은 영향을 받지 않으며, 나중에 "라이더가 10분 전에 어디 있었나요?"라는 고객 문의에도 이벤트 로그를 통해 100% 답변 가능하다.

### 도입 체크리스트
- **Consistency Lag**: 읽기 DB와 쓰기 DB 사이의 데이터 시차(Lag)를 비즈니스에서 허용할 수 있는가?
- **Skill Curve**: 개발팀이 이벤트 소싱과 비동기 프로그래밍의 복잡성을 이해하고 운영할 수 있는가?
- **Tooling**: 이벤트 유실 방지를 위한 강력한 메시지 브로커(Kafka 등) 인프라가 준비되었는가?

### 안티패턴
- **Everything is Event Sourcing**: 모든 단순한 CRUD 기능에까지 이벤트 소싱을 적용하는 행위. 저장소 용량만 낭비되고 코드만 복잡해진다. (핵심 도메인에만 적용)
- **Shared DB in Disguise**: 서비스는 쪼갰는데 하나의 DB 안에서 스키마(Schema)만 나눠 쓰는 경우. 인프라 수준의 결함 전파를 막지 못한다.

- **📢 섹션 요약 비유**: 집을 지을 때 모든 벽을 금(이벤트 소싱)으로 바를 필요는 없습니다. 가장 중요한 금고(핵심 로직)에만 특별한 재료를 써야 경제적이고 안전합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Performance)** | 조회 성능 10배 향상 | 읽기 최적화 모델(CQRS) 구축을 통한 대규모 동시 조회 수용 |
| **정량적 (Reliability)** | 데이터 복구 성공률 100% | 모든 변경 이력이 보존(Event Sourcing)되어 어떠한 시점으로도 복구 가능 |
| **정성적 (Standard)** | 도메인 가시성 확보 | 비즈니스 사건 중심의 데이터 설계를 통해 기획-개발 간 언어 일치 |
| **정성적 (Agility)** | 기술 스택의 자유 | 서비스 성격에 따라 최적의 저장소(Graph, Document 등) 도입 가능 |

### 미래 전망
- **Serverless Event Sourcing**: 클라우드가 이벤트의 순서와 저장을 완벽히 보장해주는 'Managed Event Store' 서비스가 보편화되어 구현 난이도가 낮아질 것이다.
- **AI-Based Data Evolution**: AI가 이벤트 스트림을 분석하여, 미래의 데이터 조회 패턴을 예측하고 미리 읽기 전용 모델(View)을 자동 생성하는 자율형 데이터 아키텍처가 실현될 것이다.

### 참고 표준
- **Domain-Driven Design (DDD)**: 데이터 패턴의 철학적 배경
- **Reactive Streams**: 비동기 데이터 관리 프로토콜
- **ACID vs BASE**: 분산 데이터베이스의 이론적 기준

- **📢 섹션 요약 비유**: 정보를 '박제'하던 정적인 데이터 시대에서, 정보의 '흐름과 역사'를 관리하는 동적인 데이터 시대로, 소프트웨어의 심장은 더욱 뜨겁게 뛰고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **Database per Service**는 친구들끼리 장난감을 섞지 않고 **"각자 자기 보물 상자"**에 담아서 관리하는 규칙이에요.
2. **Event Sourcing**은 보물의 현재 모습만 보는 게 아니라, **보물을 얻게 된 날부터 지금까지 쓴 "그림 일기장"**을 보물과 함께 보관하는 거예요.
3. **CQRS**는 보물을 상자에 넣는 입구와, 보물을 구경하는 투명한 유리창을 따로 만들어서 **넣는 사람과 보는 사람 모두 편하게 해주는 똑똑한 방법**이랍니다!
