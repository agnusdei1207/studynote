# 506. 양자 내성 암호 (PQC) 전환 대비 SW 아키텍처 검토

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 양자 내성 암호 (Post-Quantum Cryptography, PQC) 전환 대비는 양자 컴퓨터의 '쇼어 알고리즘(Shor's Algorithm)'에 의해 무력화될 수 있는 현재의 비대칭키 암호(RSA, ECC) 체계를 **격자(Lattice) 기반 등 차세대 암호 체계로 교체하기 위한 선제적 아키텍처 수립 과정**이다.
> 2. **가치**: "지금 훔치고 나중에 해독(Harvest Now, Decrypt Later)"하는 공격 전략으로부터 장기 보관 데이터를 보호하며, 암호 알고리즘을 유연하게 교체할 수 있는 **암호 민첩성 (Crypto Agility)** 확보를 통해 국가 및 기업의 보안 레질리언스를 강화한다.
> 3. **융합**: NIST PQC 표준(ML-KEM, ML-DSA 등) 기반의 하이브리드 암호 체계 구축, KMS(키 관리 시스템)의 성능 확장성 검토, 그리고 애플리케이션 가용성을 고려한 점진적 마이그레이션 로드맵 수립이 핵심이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: PQC는 양자 컴퓨터가 실용화되어도 해독이 불가능하도록 설계된 고전 컴퓨터용 암호 알고리즘이다. 이를 소프트웨어 아키텍처에 적용한다는 것은, 단순히 함수를 바꾸는 수준을 넘어 대폭 커진 **키 크기와 서명 데이터**를 처리할 수 있도록 인프라와 로직을 재설계하는 것을 의미한다.

- **💡 비유**: PQC 전환 대비는 **"미래에 아주 강력한 드릴(양자 컴퓨터)이 발명될 것에 대비하여, 지금의 금고 자물쇠를 드릴 날이 아예 들어가지 않는 특수 합금(PQC 알고리즘)으로 미리 교체하고, 더 커진 자물쇠를 달기 위해 금고 문(SW 아키텍처) 자체를 넓히는 작업"**과 같습니다.

- **등장 배경**:
  1. **양자 컴퓨터의 가시화**: IBM, 구글 등의 양자 우위 달성으로 수천 큐비트급 양자 컴퓨터가 출현할 경우, 현재의 PKI(공개키 기반 구조)는 실시간으로 붕괴된다.
  2. **장기 기밀 유지 데이터의 위협**: 국가 기밀이나 의료 정보 등 수십 년간 보호되어야 할 데이터가 현재의 암호화 상태로 수집된 후, 나중에 양자 컴퓨터로 해독될 리스크가 상존한다.
  3. **글로벌 표준 확정**: 2024년 NIST(미국 표준기술연구소)가 주요 PQC 알고리즘의 표준을 확정함에 따라 전 세계적인 마이그레이션이 시작되었다.

- **📢 섹션 요약 비유**: 홍수가 나기 전에 댐의 높이를 높이고 배수관을 넓히듯, '양자 위협'이라는 거대한 해일이 닥치기 전 디지털 제방을 보수하는 전략적 인프라 개선 사업입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### PQC 전환을 위한 아키텍처적 도전 과제

PQC 알고리즘은 기존 ECC/RSA 대비 데이터 크기와 연산량이 크게 증가한다.

| 비교 항목 | 기존 암호 (ECC) | PQC (ML-KEM/Lattice) | 아키텍처적 영향 |
|:---|:---:|:---:|:---|
| **공개키 크기** | 약 32 Bytes | **약 1,000 ~ 2,000 Bytes** | DB 스키마 및 네트워크 패킷 증가 |
| **서명 크기** | 약 64 Bytes | **약 2,500 ~ 4,500 Bytes** | API 응답 지연 및 스토리지 부하 |
| **연산 속도** | 빠름 | 알고리즘별 상이 (복잡도 높음) | 서버 CPU 점유율 및 전력 소모 증가 |
| **암호화 원리** | 타원곡선 이산대수 | **격자(Lattice) 상의 최단벡터 문제** | 수학적 구조의 근본적 변화 |

---

### 하이브리드 PQC 전환 아키텍처 (PQC-Classical Hybrid)

갑작스러운 전환으로 인한 안정성 저하를 막기 위해 기존 암호와 PQC를 결합하여 사용한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      PQC 하이브리드 키 교환 및 데이터 암호화 구조                 │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [클라이언트]                                            [서버]               │
 │      │                                                  │                   │
 │  1. 하이브리드 헬로 ────────(ECC + PQC Info)─────────▶ │                   │
 │                                                           │                   │
 │  2. 이중 키 교환 (Key Exchange)                           │                   │
 │     - 기존 방식: ECDH Shared Secret (K1)                  │                   │
 │     - PQC 방식: ML-KEM Shared Secret (K2)                 │                   │
 │                                                           │                   │
 │  3. 마스터 세션키 생성 ────────( K = KDF(K1, K2) )────────▶ │                   │
 │     (두 암호 중 하나만 안전해도 세션키 보호됨)                   │                   │
 │                                                           │                   │
 │  4. 대칭키 암호화 통신 ──────────(AES-GCM)───────────▶ │                   │
 │                                                           │                   │
 │  ┌───────────────────────────────────────────────────────┐ │                   │
 │  │ ✅ 아키텍처 검토 포인트:                                 │ │                   │
 │  │ - MTU 사이즈 조정: 패킷 파편화(Fragmentation) 대응       │ │                   │
 │  │ - 메모리 할당: 커진 키/서명 처리를 위한 버퍼 확장         │ │                   │
 │  │ - KMS 연동: PQC 알고리즘을 지원하는 키 관리 시스템 도입    │ │                   │
 │  └───────────────────────────────────────────────────────┘                  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PQC로의 한 번에 가는 것이 아니라 **하이브리드 모드**를 거치는 것이 아키텍처 설계의 정석이다. 기존의 **ECDH**로 만든 비밀 값(K1)과 새로운 **PQC** 알고리즘(ML-KEM 등)으로 만든 비밀 값(K2)을 섞어서 최종 세션키를 만든다. 이렇게 설계하면 설령 새로운 PQC 알고리즘에서 예상치 못한 결함이 발견되더라도 기존 ECC 보안성이 버텨주고, 반대로 양자 컴퓨터가 ECC를 뚫더라도 PQC가 데이터를 방어해준다. 단, 이 과정에서 늘어난 데이터 크기로 인해 네트워크 장비의 MTU 설정을 넘어서는 패킷 파편화 문제가 발생할 수 있으므로 네트워크 아키텍처 전반의 검토가 수반되어야 한다.

---

### 심층 동작 원리: 암호 민첩성 (Crypto Agility) 설계

알고리즘을 코드 수정 없이 교체할 수 있는 유연한 아키텍처 설계 원칙이다.
1. **추상화 레이어**: 코드 내에 `RSA.encrypt()`를 직접 쓰는 대신, 암호화 인터페이스(`Cipher.encrypt()`)를 두고 설정 파일에 따라 알고리즘을 동적으로 로드한다.
2. **알고리즘 협상 (Negotiation)**: 통신 시작 시 양측이 지원하는 가장 강력한 암호를 선택하는 로직을 표준화한다.
3. **메타데이터 저장**: 데이터 저장 시 어떤 알고리즘으로 암호화되었는지 버전 정보와 식별자를 함께 저장하여 향후 일괄 변환(Migration)을 용이하게 한다.

- **📢 섹션 요약 비유**: 부품이 고장 날 때마다 차를 새로 사는 것이 아니라, 엔진(알고리즘)만 쏙 빼서 새것으로 갈아 끼울 수 있도록 '모듈형 자동차'를 만드는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### PQC 주요 표준 알고리즘 (NIST 2024 확정)

| 용도 | 알고리즘 명칭 | 기반 수학 이론 | 특징 |
|:---|:---|:---|:---|
| **공개키 암호 / KEM** | **ML-KEM (Kyber)** | 격자 (Lattice) 기반 | 효율성이 가장 우수, TLS 표준 유력 |
| **디지털 서명** | **ML-DSA (Dilithium)** | 격자 (Lattice) 기반 | 균형 잡힌 성능과 크기, 범용적 사용 |
| **디지털 서명 (보조)** | **SLH-DSA (Sphincs+)** | 해시 (Hash) 기반 | 서명 크기는 크나 안전성이 매우 검증됨 |
| **디지털 서명 (보조)** | **FN-DSA (Falcon)** | 격자 (Lattice) 기반 | 서명 크기가 매우 작으나 구현 복잡도 높음 |

---

### 융합 시너지: 클라우드 하드웨어 보안 (HSM)과 PQC

PQC 전환의 가장 큰 걸림돌은 하드웨어 가속이다.
- **PQC-aware HSM**: 기존 HSM은 RSA/ECC 연산 전용 칩으로 구성되어 PQC 처리가 매우 느리다. 따라서 FPGA나 가변 프로세서 기반의 차세대 HSM 도입이 소프트웨어 아키텍처의 필수 전제 조건이 된다.

- **📢 섹션 요약 비유**: 고성능 그래픽 게임(PQC)을 돌리기 위해 낡은 사무용 컴퓨터의 그래픽 카드(HSM)를 최신형으로 업그레이드하는 과정과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 국가 핵심 기간망 보안 마그네이션 로드맵

- **문제 상황**: 국가 중요 시설의 제어 시스템이 RSA 기반 인증을 사용 중이며, 10년 내 양자 위협에 노출될 우려가 있음.
- **의사결정 (PQC 전환 아키텍처 로드맵)**:
  1. **Phase 1 (재고 조사)**: 시스템 내에서 사용 중인 모든 암호 기술(알고리즘, 키 위치, 라이브러리)을 전수 조사하여 리스트업(Inventory)한다.
  2. **Phase 2 (하이브리드 적용)**: 내부 통신 구간부터 TLS 1.3 하이브리드(ML-KEM + ECDH) 모드를 적용하여 'Harvest Now' 공격을 방어한다.
  3. **Phase 3 (인프라 확장)**: 커진 키 크기를 수용할 수 있도록 DB 컬럼 길이를 확장하고, 암호화 키 관리 시스템(KMS)을 PQC 지원 버전으로 교체한다.
  4. **Phase 4 (Full PQC)**: 양자 컴퓨터가 실용화되는 시점에 맞추어 고전 암호를 완전히 제거하고 순수 PQC 체계로 전환한다.

### 도입 체크리스트
- **Latency Budget**: PQC 적용 시 늘어나는 연산 시간과 네트워크 전송 시간이 서비스 응답 속도 목표(SLA)를 만족하는가?
- **Schema Compatibility**: DB나 설정 파일의 고정된 필드 길이가 10배 이상 커진 PQC 키를 저장할 수 있는가?
- **Interoperability**: 연동되는 협력사나 클라이언트 환경이 PQC 알고리즘을 지원할 수 있는가?

### 안티패턴
- **조기 낙관주의**: "양자 컴퓨터는 아직 멀었으니 나중에 생각하자"는 태도. 이미 데이터는 수집되고 있으며 아키텍처 변경에는 수년이 걸린다.
- **자체 PQC 알고리즘 구현**: 표준화되지 않은 새로운 수학 이론을 바탕으로 독자 암호를 만드는 행위. 암호학적으로 매우 위험하다.

- **📢 섹션 요약 비유**: 지진이 오기 전에 내진 설계를 하는 것이 귀찮다고 미루다가, 지진이 일어난 뒤에 후회하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 양자 컴퓨터 공격 면역 확보 | 쇼어 알고리즘에 의한 비대칭키 해독 위협 원천 차단 |
| **정량적 (Compliance)** | 글로벌 표준 준거성 확보 | NIST 및 국내 외 PQC 전환 가이드 준수를 통한 대외 신뢰도 제고 |
| **정성적 (Resilience)** | 미래 지향적 보안 기반 마련 | 암호 민첩성 확보를 통해 어떤 새로운 위협에도 유연하게 대처 가능 |
| **정성적 (Trust)** | 장기 기밀성 보장 | "우리 서비스는 미래에도 안전하다"는 강력한 보안 브랜드 가치 확보 |

### 미래 전망
- **Integrated PQC Co-processors**: 모든 CPU와 모바일 칩셋에 PQC 전용 연산 유닛이 내장되어, 성능 저하 없는 암호화가 일상화될 것이다.
- **Quantum-Safe Internet**: 인터넷 프로토콜 전반(DNS, BGP 등)이 PQC 기반으로 재설계되어 인프라 수준의 보안이 완성될 것이다.

### 참고 표준
- **NIST FIPS 203, 204, 205**: 공식 PQC 알고리즘 표준 문서
- **KISA 양자 내성 암호 마이그레이션 가이드**: 국내 기업용 지침서
- **IETF RFC Hybrid Key Exchange**: 통신 표준안

- **📢 섹션 요약 비유**: 촛불에서 전구로, 이제는 LED(PQC)로 빛의 기술이 바뀌듯, 암호도 더 밝고 안전한 미래를 위해 근본적인 세대교체를 준비하고 있습니다.
