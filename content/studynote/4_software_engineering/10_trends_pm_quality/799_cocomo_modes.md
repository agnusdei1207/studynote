# 799. COCOMO 비용 산정 모드 (Organic, Semi, Embedded)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어의 규모(KDSI)와 프로젝트 환경의 복잡도에 따라 비용 산정 알고리즘의 계수(상수 $a$, 지수 $b$)를 차별화하여 적용하는 **경험적(Heuristic) 비용 산정 모델**의 핵심 분류 체계이다.
> 2. **메커니즘**: 단순 비즈니스용 **Organic**, 복잡도가 중간인 **Semi-Detached**, 시스템 제어 등 초고난도인 **Embedded** 모드로 나누어, 규모 증가에 따른 비선형적인 노력(Effort) 증가분을 수학적으로 모델링한다.
> 3. **가치**: 프로젝트의 성격에 따라 '규모가 커질수록 난이도가 상승하는 비율(지수 $b$)'을 다르게 설정함으로써, 단순 비례 산정이 가질 수 있는 '비용 과소 추정' 위험을 방지하고 현실적인 예산과 일정(TDEV)을 예측한다.

---

### Ⅰ. 개요 (Context & Background)

소프트웨어 비용 산정의 역사는 단순한 인월(Man-Month) 곱셈에서 시작해 복잡도를 반영한 비선형 모델로 진화해 왔습니다. 1981년 배리 보헴(Barry Boehm)이 제안한 COCOMO (Constructive Cost Model)는 소프트웨어 개발 비용을 산정하는 가장 대표적인 알고리즘 모델입니다. COCOMO의 핵심은 "모든 소프트웨어 1만 라인(Line of Code)이 동등하게 만들어지지 않는다"는 명제에서 출발합니다.

기본 COCOMO 모델은 프로젝트의 특성에 따라 세 가지 모드(Mode)를 정의합니다.
1.  **Organic Mode**: 소규모 팀, 낮은 복잡도, 유연한 환경
2.  **Semi-Detached Mode**: 중간 규모, 경험과 신규 기술 혼재, 제약 존재
3.  **Embedded Mode**: 대규모 복잡계, 엄격한 하드웨어/소프트웨어 제약, 실시간 처리 필수

이 모드는 단순한 분류기를 넘어, 프로젝트의 통신 비용(Communication Overhead), 관리 복잡도, 기술적 난이도를 수식의 지수(Exponent)로 치환하여 비용에 반영하는 정교한 장치입니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                          COCOMO 모드의 비유적 이해                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. Organic (유기적) "DIY 조립식 선반"                                       │
│     → 설계도가 이미 있고, 도구도 익숙하며, 혼자서도 충분히 빠르게 조립 가능.    │
│       부품이 2배가 되면 시간도 대략 2배 소모됨. (의사소통 비용 낮음)            │
│                                                                             │
│  2. Semi-Detached (반분리형) "맞춤형 주방 리모델링"                          │
│     → 목수, 전기 기사, 배관工이 협업해야 함.                                 │
│       부품이 늘어나면 협조 과정에서 오해가 생겨 시간이 2배보다 조금 더 걸림.    │
│                                                                             │
│  3. Embedded (내장형) "우주선 비행 제어 시스템"                              │
│     → 기계, 전자, 소프트웨어이 한 몸.                                        │
│       부품이 1개 추가될 때마다 전체 시스템의 안정성을 검증해야 하므로          │
│       작업 난이도와 시간이 **폭발적으로(지수적으로)** 증가함.                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유:**
> 마치 **'식당 난이도'**에 따라 견적 방식이 다른 것과 같습니다. 가정식 밀키트(Organic)는 재료 수에 비례해 요리 시간이 늘지만, 호텔 파티 케이터링(Embedded)은 음식 양이 늘어날수록 조리 시간뿐 아니라 인력 조율, 설비 운영 등의 오버헤드가 기하급수적으로 커지는 원리와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

COCOMO의 기본 공식은 다음과 같습니다.
$$E = a \times (KDSI)^b$$
여기서 $E$는 노력(Man-Month), $KDSI$는 납품된 소스 명령어 수(Delivered Source Instructions, 천 단위), $a$와 $b$는 모드별 상수입니다.

#### 1. 모드별 상세 분석 및 상수 테이블

| 모드 (Mode) | 시스템 특징 (Characteristics) | 대상 시스템 예시 | 규모 (Typical Size) | 상수 $a$ | **지수 $b$** | 스케줄 상수 $c$ | 스케줄 지수 $d$ |
|:---:|:---|:---|:---:|:---:|:---:|:---:|:---:|
| **Organic** | • 높은 도메인 지식<br>• 낮은 기술적 제약<br>• 소규모 팀 | 업무 자동화, 입출금 처리 | < 50 KDSI | 2.4 | **1.05** | 2.5 | 0.38 |
| **Semi-Detached** | • 중간 수준의 복잡도<br>• 경험자와 초보자 혼재<br>• 유연성과 제약의 균형 | 컴파일러, 데이터베이스 | < 300 KDSI | 3.0 | **1.12** | 2.5 | 0.35 |
| **Embedded** | • 엄격한 하드웨어 인터페이스<br>• 실시간 성능 필수<br>• 높은 결합도 | 항공기 제어, 미사일 유도 | No Limit | 3.6 | **1.20** | 2.5 | 0.32 |

*   **지수 $b$의 의미**: 복잡도 계수.
    *   Organic ($b \approx 1.05$): 규모가 늘어나도 난이도 상승이 선형에 가깝습니다.
    *   Embedded ($b \approx 1.20$): 규모가 커질수록 통신 오버헤드, 인터페이스 결합도로 인해 난이도가 급격히 상승합니다. (예: 10만 라인에서 20만 라인으로 늘어날 때, 2배의 노력이 아니라 약 $2^{1.2} \approx 2.3$배의 노력이 필요함)

#### 2. 산정 프로세스 아키텍처

비용 산정은 단순 공식 대입이 아니라 프로젝트의 속성을 분석하여 적절한 모드를 매핑하는 과정이 선행되어야 합니다.

```text
      [ Input: Project Requirements ]
                 │
                 ▼
      ┌─────────────────────────────┐
      │   1. Complexity Analysis     │
      │   (Interaction, Constraints) │
      └─────────────────────────────┘
                 │
                 ├──▶  Hard/Real-time ────▶ [ Embedded Mode ]
                 │                        (Strict, a=3.6, b=1.20)
                 │
                 ├──▶  Flexible/Routine ─▶ [ Organic Mode ]
                 │                        (Relaxed, a=2.4, b=1.05)
                 │
                 └──▶  Mixed/Utility ─────▶ [ Semi-Detached Mode ]
                                          (Balanced, a=3.0, b=1.12)
                 │
                 ▼
      ┌─────────────────────────────┐
      │  2. Size Estimation (KDSI)   │
      └─────────────────────────────┘
                 │
                 ▼
      ┌─────────────────────────────┐
      │  3. Calculate Effort (PM)    │
      │     Effort = a * (KDSI)^b    │
      └─────────────────────────────┘
                 │
                 ▼
      ┌─────────────────────────────┐
      │  4. Calculate Schedule (TDEV)│
      │     TDEV = c * (PM)^d        │
      └─────────────────────────────┘
```

**[다이어그램 해설]**
위 다이어그램은 COCOMO 적용의 의사결정 트리(Decision Tree)입니다. 가장 중요한 단계는 '1단계 Complexity Analysis'입니다. 여기서 시스템이 하드웨어에 종속되는지(Embedded), 단순 반복 업무인지(Organic) 판단합니다. 이 결정이 상수 $a$와 $b$를 결정짓고, 동일한 코드 라인수(KDSI)라도 최종 산출된 공수(PM)는 크게 달라지게 됩니다. 4단계의 TDEV(Time to Develop) 계산은 Brooks의 법칙(Brooks's Law)을 고려하여, 인력을 무조건 투입한다고 해서 일정이 단축되지 않음을 보여줍니다.

#### 3. 핵심 원리: 규모의 경제 vs 비경제

기본 COCOMO는 소프트웨어 개발에서 **'규모의 비경제(Diseconomies of Scale)'**를 수학적으로 증명합니다.
*   **Organic**: 팀원 간의 커뮤니케이션 비용이 적어, 규모가 커져도 생산성 저하가 미미함.
*   **Embedded**: 모듈 간 결합도가 높고 인터페이스가 복잡하여, 규모가 커질수록 단순 작업량을 넘어선 '조율(Coordination) 비용'이 폭발함.

**📢 섹션 요약 비유:**
> **'고속도로 톨게이트'**와 같습니다. Organic 모드는 하이패스 전용 차선처럼 통행량(코드량)이 늘어나도 속도(생산성)가 유지되지만, Embedded 모드는 통행량이 늘어날수록 교차로에서의 신호 대기 시간(통신 비용, 복잡도)이 기하급수적으로 늘어나 정체가 발생하는 구조와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

COCOMO 모드는 고립된 지표가 아니라 다른 소프트웨어 공학 지표들과 상호 보완적으로 사용될 때 가장 강력한 힘을 발휘합니다.

#### 1. 하향식(Top-Down) COCOMO vs 상향식(Bottom-Up) FP (Function Point)

| 비교 항목 | COCOMO (LOC 기반) | FP (기능점수 기반) |
|:---|:---|:---|
| **접근 방식** | **하향식(Top-Down)**: 전체 규모 추정 후 단계별 분배 | **상향식(Bottom-Up)**: 개별 기능 점수 합산 |
| **측정 단위** | **LOC** (Lines of Code): 개발자 관점, 기술 의존적 | **FP** (Function Point): 사용자 가치 관점, 기술 독립적 |
| **정확도 시점** | 상세 설계 후(LOC가 확정된 후) 정확도 급상승 | 요구사항 분석 단계에서도 비교적 정확한 예측 가능 |
| **COCOMO 모드와의 관계** | LOC 크기와 환경이 모드 결정의 주요 인자임 | FP 점수를 LOC로 변환(SW 저장소 등 활용)하여 COCOMO에 입력 가능 |
| **융합 전략** | **초기**: FP로 규모를 잡고 간이 COCOMO를 적용<br>**상세**: 상세 설계 후 예상 LOC로 모드별 COCOMO 재검증 |

*   **시너지**: 초기 프로젝트에는 사용자와 협의하기 쉬운 FP로 규모를 정하고, 이를历史 데이터(History Data)를 통해 LOC로 환산하여 COCOMO 모드를 적용하면 '사용자의 이해'와 '기술적 현실성'을 동시에 확보할 수 있습니다.

#### 2. COCOMO 81 vs COCOMO II (진화 과정)

| 특징 | COCOMO 81 (Basic/Intermediate) | COCOMO II |
|:---|:---|:---|
| **개발 모델** | 폭포수(Waterfall) 모델 기반 | 스파이럴(Spiral), 애자일(Agile) 고려 |
| **규모 산정** | KDSI (Source Instructions) | **FP**, Object Points, SLOC |
| **모드 구분** | Organic, Semi-Detached, Embedded (3개) | **Applications Composition**, Early Design, **Post-Architecture** (단계별) |
| **비용 절감因子** | 없음 (공식만 존재) | 17개의 **Cost Drivers** (RELY, DATA, CPLX 등 포함) |

*   **분석**: 799번에서 다루는 '3대 모드'는 고전 COCOMO의 개념입니다. 현대의 COCOMO II에서는 모드 개념이 후반(Post-Architecture) 단계의 '비용 견인인자(Cost Drivers)'로 세분화되어 더욱 정밀해졌습니다. 하지만 프로젝트의 '본질적 복잡도(Gnature of System)'를 구분한다는 철학은 여전히 유효합니다.

**📢 섹션 요약 비유:**
> **'주식 투자 방법'**과 유사합니다. COCOMO(I)는 과거 거래 패턴(LOC)을 보고 향후 주가를 예측하는 기술적 분석이라면, FP는 회사의 실질 가치(기능)를 보는 펀더멘털 분석입니다. 최고의 투자자(PE)는 이 두 가지를 결합하여 의사결정을 내립니다. 또한 COCOMO II는 단순 차트 보기를 넘어 뉴스, 정책, 시장 심리(Cost Drivers)까지 모두 반영하는 AI 트레이딩 시스템으로 진화한 것과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 COCOMO 모드를 선택하는 것은 단순한 계산이 아니라 **'리스크 관리(Risk Management)'**의 일환입니다.

#### 1. 실무 시나리오: 모드 선정 딜레마

**Case A: 금융 IT 신규 시스템 구축**
*   **상황**: Java 기반의 백엔드 시스템 구축. 코드량 예상 10만 라인.
*   **판단**: **Semi-Detached** 적용.
    *   **
