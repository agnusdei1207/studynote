# 643. 결함 밀도 측정 및 프로세스 통제

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어의 규모(LOC 또는 FP) 대비 잠재적 결함을 정량화하여, 모듈별 품질 수준과 테스트 효율성을 판단하는 **정량적 품질 척도(Metric)**다.
> 2. **통제**: 조직의 표준(Baseline)과 비교하여 **결함 집중(Defect Clustering)** 현상을 탐지하고, **SPC (Statistical Process Control)** 기법을 통해 개발 프로세스를 통제 및 개선한다.
> 3. **가치**: 주관적 직관을 데이터에 기반한 객관적 의사결정으로 대체하여, 테스트 리소스 배분 최적화 및 릴리즈(Release) 타당성을 입증하는 핵심 지표로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 및 정의
**DD (Defect Density, 결함 밀도)**란 소프트웨어의 특정 모듈 또는 시스템 전체에서 발견된 결함의 수를 소프트웨어의 규모(크기)로 나눈 값입니다. 단순히 "버그가 10개다"라는 정보는 소프트웨어의 크기가 고려되지 않아 품질 수준을 비교할 수 없습니다. 따라서 **KLOC (Kilo Lines of Code, 천 줄의 코드)** 또는 **FP (Function Point, 기능점수)** 당 결함 수로 환산하여, 표준화된 지표를 통해 품질 수준을 상대평가합니다.

### 2. 등장 배경 및 필요성
- **기존 한계**: 초기 개발 단계에서는 단순 결함 수(Defect Count)만으로 품질을 판단하였으나, 이는 모듈별 크기 차이를 반영하지 못해 '작은 모듈의 결함'이 간과되거나 '큰 모듈의 품질 낭비'가 발생함.
- **패러다임 변화**: 제조 분야의 **6-Sigma (Six Sigma)** 기법과 같은 통계적 품질 관리 도입 필요성이 대두됨. 이에 소프트웨어 공학 분야에서도 공정(Process)을 통계적으로 관리하려는 움직임이 시작됨.
- **현재 요구**: 대규모 시스템 및 클라우드 환경에서는 수백만 라인의 코드를 다루므로, 리소스를 효율적으로 배치하기 위해 **'위험 지역(Hotspot)'**을 정량적으로 식별하는 것이 필수적임.

### 💡 비유: 밭의 해충 밀도 조사
두 농부가 각자의 밭을 관리한다고 가정할 때, 단순히 "벌레가 10마리 발견되었다"고 보고하는 것만으로는 밭의 상태를 알 수 없습니다. 밭의 넓이(규모)를 고려하여 **면적당 해충 밀도**를 계산해야 어느 밭이 더 위험한지, 어디에 살충제(리소스)를 뿌려야 할지 판단할 수 있습니다.

### 📢 섹션 요약 비유
> **"테스트 리소스는 한정되어 있다. 결함 밀도는 100평짜리 밭과 1평짜리 밭의 해충 위험도를 단위 면적당(면적당)으로 환산하여, 어디에 살충제를 뿌려야 수확량을 가장 많이 지킬 수 있는지 판단하게 하는 '농업 전략 지표'와 같다."**

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 결함 밀도 산정 수식 및 파라미터
결함 밀도는 일반적으로 다음과 같은 수식을 통해 산출됩니다.

$$ \text{DD (Defects per Unit)} = \frac{\text{Total Identified Defects}}{\text{Total Size (KLOC or FP)}} $$

- **Total Identified Defects**: 특정 단계(통합 테스트, 시스템 테스트 등)에서 발견된 결함의 총합. (미발견 결함은 포함되지 않음에 유의)
- **Size (KLOC)**: 소스 코드의 라인 수(주석/공백 제외 여부는 표준에 따름).
- **Size (FP)**: 사용자 요구사항의 기능적 복잡도를 기반으로 한 기능점수.

### 2. 측정 아키텍처 및 데이터 흐름
결함 밀도 측정 시스템은 결함 추적 시스템과 형상 관리 시스템의 데이터를 연동하여 실시간(혹은 주기적)으로 계산합니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    결함 밀도 측정 및 통제 아키텍처                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 개발자/테스터                                                          │
│     │                                                                      │
│     ├───(결함 리포트)───>  ┌─────────────────────────────────────┐         │
│     │                      │  Bug Tracking System (Jira, Mantis) │         │
│     │                      │  - Defect ID, Severity, Module      │         │
│     │                      └──────────────┬──────────────────────┘         │
│     │                                     │                                 │
│     │                                     │ (Extract Data)                  │
│     │                                     ▼                                 │
│     │                      ┌─────────────────────────────────────┐         │
│     │                      │  Defect Density Analyzer Engine      │         │
│     │                      │  (Calculation Logic)                 │         │
│     │                      └──────────────┬──────────────────────┘         │
│     │                                     │                                 │
│     │                                     ▼                                 │
│     ├───(Code/FP Size)───>  ┌─────────────────────────────────────┐         │
│                            │  SCM (Git, SVN) / FP Counter         │         │
│                            └──────────────────────────────────────┘         │
│                                                                             │
│  2. 결과물 산출                                                             │
│     ┌──────────────────────────────────────────────────────────────────┐   │
│     │  [Dashboard] Module A: 5.2/KLOC (High) | Module B: 0.1/KLOC    │   │
│     │     - SPC Control Chart (UCL/LCL Check)                         │   │
│     │     - Trend Analysis                                             │   │
│     └──────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**해설**:
1.  **데이터 수집**: 테스트 과정에서 **BTS (Bug Tracking System, 결함 추적 시스템)**에 기록된 결함 데이터와 형상 관리 도구(**SCM, Source Configuration Management**)에서 측정된 소스 코드 라인 수가 수집됩니다.
2.  **분석**: 수집된 데이터를 바탕으로 모듈별, 개발자별, 기간별 결함 밀도를 자동 산출합니다.
3.  **시각화 및 통제**: 산출된 결과를 대시보드(Dashboard)에 표시하고, 사전에 정의된 **UCL (Upper Control Limit, 관리 상한선)**을 초과하는 모듈에 대해 알림(Notification)을 발생시킵니다.

### 3. 핵심 구성 요소 상세
| 구성 요소 | 역할 | 주요 파라미터/프로토콜 |
|:---|:---|:---|
| **Defect Counter** | 결함의 개수를 집계. 중복 결함 제거, 유효 결함만 필터링. | Status: Verified, Closed |
| **Size Counter** | 코드의 물리적/논리적 크기 측정. | LOC (Logical vs Physical), FP (UCP: Use Case Point) |
| **Baseline (기준선)** | 조직의 품질 목표치. 프로젝트 성격에 따라 다름. | 예: 2.0 Defects/KLOC (초기), 0.5 (안정기) |
| **SPC Engine** | 데이터의 분산과 추세를 분석하여 이상 징후 감지. | 3-Sigma Rule, Westgard Rules |
| **Control Chart** | 시계열 데이터를 시각화하여 프로세스 안정성 판단. | X-bar Chart, p-Chart |

### 4. 통계적 공정 관리 (SPC)와의 융합
결함 밀도는 시간의 흐름에 따라 그래프로 그렸을 때, **UCL (Upper Control Limit)**과 **LCL (Lower Control Limit)** 사이에서 분포해야 합니다. 이를 벗어나는 점이 발견되면 프로세스에 특별 원인(Special Cause)이 발생했다고 간주하여 원인 분석을 실시합니다.

```text
   ▲ Defect Density
   │
   │         ● (이상 발생: 프로세스 점검 요구)
   │         │
  UCL ───────┼─────────────────────────
   │     ●   │   ●       ●
   │   ●     │       ●       ●
   Mean ─────┼───────────●───────────
   │ ●       │   ●       ●   ●
   │         │
  LCL ───────┴───────────────────────────
   │
   └──────────────────────────────────────▶ Time (Build Iteration)
   (Normal Process Zone)
```

### 📢 섹션 요약 비유
> **"마치 공장에서 생산되는 자동차 부품의 치수를 오차 범위 내에서 관리하듯, 결함 밀도는 소프트웨어라는 '제품'의 품질 불량률을 미리 설정된 '오차 관리한계(UCL/LCL)' 내에서 통제하는 통계적 품질 관리의 핵심 공정도이다."**

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 심층 기술 비교: 결함 수 vs 결함 밀도 vs 회귀 오류율

| 비교 지표 | Defect Count (결함 수) | **Defect Density (결함 밀도)** | Regression Rate (회귀 오류율) |
|:---|:---|:---|:---|
| **정의** | 발견된 결함의 절대적 개수 | 규모(KLOC/FP) 대비 결함 비율 | 수정 후 재발생하거나 새로 발생한 결함 비율 |
| **목적** | 단순 작업량 및 남은 작업 파악 | 모듈 간 품질 비교 및 위험 지역 식별 | 수정의 품질 및 부작용(Side Effect) 파악 |
| **장점** | 직관적이고 측정이 쉬움 | 프로젝트 간, 모듈 간 공정한 비교 가능 | 테스트 안정성과 수정 완결성 확인 |
| **단점** | 규모를 고려하지 않아 비교 어려움 | LOC 측정 기준에 따라 편차 발생 가능 | 정밀한 테스트 케이스 관리가 필요 |
| **핵심 지표** | Total: 150 bugs | **2.5 Defects/KLOC** | 5% (Low), >15% (High) |

### 2. 과목 융합 분석
- **SW Engineering <-> 통계학 (Statistics)**:
    - **Pareto Principle (파레토 법칙)**: 결함 밀도 분석 시 "전체 결함의 80%는 20%의 모듈에서 발생한다"는 법칙이 적용됩니다. 즉, 결함 밀도가 높은 소수의 핵심 모듈(Hot Module)을 집중적으로 개선하면 전체 품질을 획기적으로 높일 수 있습니다.
    - **Standard Deviation (표준편차)**: 결함 밀도의 표준편차가 크다는 것은 모듈 간 품질 편차가 심하다는 뜻으로, 일부 개발자의 교육 부족이나 아키텍처 설계 문제를 시사합니다.
- **SW Engineering <-> 컴퓨터 구조 (Computer Architecture)**:
    - 결함 밀도가 높은 모듈은 **Complexity (복잡도)**가 높을 확률이 큽니다. McCabe의 **Cyclomatic Complexity (순환 복잡도)**가 높은 모듈(예: 15 이상)은 결함 밀도가 높게 나타나는 경향이 있으므로, 이를 병행하여 측정하면 더 정확한 위험 예측이 가능합니다.

### 3. 실무 시나리오: "킬러 모듈" 찾기
대규모 금융 프로젝트에서 50개의 모듈 중 '계산引擎(Calculation Engine)' 모듈만 결함이 15%를 차지하는 상황을 가정해 봅시다. 단순 결함수로는 보이지 않지만, 결함 밀도를 분석해보니 해당 모듈의 복잡도가 평균보다 3배 높은 것으로 확인됩니다. 이때 PM은 리팩토링(Restructuring)을 결정하고, 전체 품질을 20% 이상 향상시킬 수 있습니다.

### 📢 섹션 요약 비유
> **"단순한 체온계(결함 수)가 열이 났는지만 알려준다면, 결함 밀도 분석은 MRI(자기공명영상)와 같이 몸의 어느 부위(모듈)에 왜 결핵(결함)이 집중되었는지, 그것이 악성인지 양성인지까지 정밀 검진하는 '의료적 진단 도구'이다."**

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 적용 시나리오: 릴리즈 결정 (Go/No-Go)

**상황**:
- 프로젝트 말기, 통합 테스트 종료 1주일 전.
- 전체 시스템 결함 수는 500개로 저녀저녀해 보이나, 신규 추가된 '결제 모듈'의 결함 밀도가 **5.0 Defects/KLOC**을 기록 중임.
- 조직의 표준 릴리즈 기준(Baseline)은 **1.0 Defects/KLOC** 이하임.

**의사결정 프로세스 (Decision Matrix)**:
1.  **감안 (Assess)**: 단순 결함 수보다 결함 밀도가 중요함을 인지. '결제 모듈'은 시스템의 핵심 부품으로 신뢰성이 최우선임.
2.  **분석 (Analyze)**: 결함 밀도가 높은 원인을 분석.
    -   원인 1: 신규 개발자 투입 → 숙련도 부족.
    -   원인 2: 요구사항 변경 잦음 → 불안정한 명세서.
    -   원인 3: 모듈 간 인터페이스 결함 → 통합 오류.
3.  **조치 (Action)**:
    -   **Execute**: 결제 모듈에 대해서는 집중적인 **Pair Programming** 및 **Mutation Testing (변이 테스트)** 실시.
    -   **Hold**: 결
