# 671. UML 클래스, 시퀀스, 액티비티 다이어그램

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 복잡한 소프트웨어 시스템의 구조와 행위를 표준화된 기호로 시각화하여, 이해관계자 간의 명확한 의사소통과 정교한 설계를 가능하게 하는 **통합 모델링 언어(Unified Modeling Language)**이다.
> 2. **3대 다이어그램**: 정적 구조를 나타내는 **클래스 다이어그램**, 객체 간 상호작용의 시간적 흐름을 보여주는 **시퀀스 다이어그램**, 비즈니스 프로세스나 로직의 흐름을 표현하는 **액티비티 다이어그램**이 핵심이다.
> 3. **가치**: 아키텍처의 청사진을 제공하여 구현 단계의 오류를 방지하고, 유지보수 시 시스템의 로직을 빠르게 파악할 수 있는 시각적 명세서 역할을 한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "왜 UML이 필요한가?"

소프트웨어의 규모가 커지면 코드로만 시스템을 이해하는 것이 불가능해집니다. UML은 방법론의 난립을 종식하고 객체지향 모델링의 표준을 확립했습니다. 특히 '구조(Structural)'와 '행위(Behavioral)'라는 두 축을 중심으로 시스템을 입체적으로 조망할 수 있게 해 줍니다.

### 💡 비유: 건물의 설계 도면들

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        UML 3대 다이어그램 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 거대한 백화점 건물을 지으려고 함.                                       │
│                                                                             │
│  1. 클래스 다이어그램 (건물 단면도):                                           │
│     어떤 기둥(Class)이 있고, 방들이 어떻게 연결(Association)되어 있는지 보는 것.   │
│     움직이지 않는 정적인 뼈대를 확인!                                           │
│                                                                             │
│  2. 시퀀스 다이어그램 (손님 이용 시나리오):                                     │
│     손님이 들어와서(Actor) 안내데스크에 묻고, 에스컬레이터를 타는 시간 순서.       │
│     객체들이 메시지를 주고받는 흐름 확인!                                       │
│                                                                             │
│  3. 액티비티 다이어그램 (업무 매뉴얼):                                         │
│     물건이 들어오면 검수하고, 합격하면 창고로, 불합격하면 반품하는 처리 과정.      │
│     로직의 분기와 반복(Algorithm) 확인!                                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 다이어그램 상세 비교

| 항목 | 클래스 다이어그램 | 시퀀스 다이어그램 | 액티비티 다이어그램 |
|:---:|:---|:---|:---|
| **분류** | 구조 다이어그램 (Static) | 행위/상호작용 (Dynamic) | 행위 다이어그램 (Dynamic) |
| **관점** | 시스템의 정적 구성 요소 | 시간 순서에 따른 메시지 교환 | 업무 처리 흐름 및 제어 로직 |
| **핵심 기호** | 클래스, 연관, 상속, 인터페이스 | 라이프라인, 실행 오컬런스, 메시지 | 시작/종료점, 액션, 결정(Decision) |
| **용도** | 도메인 모델링, DB 설계 | 유스케이스 구현 상세 설계 | 비즈니스 프로세스, 알고리즘 표현 |

### 2. 다이어그램별 핵심 메커니즘 (ASCII)

#### (1) 클래스 다이어그램 (관계성)
```text
    ┌───────────┐          ┌──────────────┐
    │   Order   │ ◇──────▶ │  OrderItem   │ (Aggregation)
    ├───────────┤          ├──────────────┤
    │ - id: int │          │ - qty: int   │
    └─────▲─────┘          └──────────────┘
          │ (Inheritance)
    ┌─────┴─────┐
    │ CardOrder │
    └───────────┘
```

#### (2) 시퀀스 다이어그램 (상호작용)
```text
    User       Controller      Service
     │            │               │
     │── 요청 ──▶│               │
     │            │─── 작업() ──▶│
     │            │               │──┐ (자체 처리)
     │            │               │  │
     │◀─ 결과 ───│               │◀─┘
     │            │               │
```

#### (3) 액티비티 다이어그램 (로직 흐름)
```text
    (●) ──▶ [로그인] ──▶ <결정> ──(성공)──▶ [메인화면] ──▶ (◉)
                           │
                        (실패) ──▶ [경고창] ──▶ (●)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 기술적 시너지: 코드 자동 생성 (Forward/Reverse Engineering)
UML 도구(Enterprise Architect, StarUML 등)를 사용하면 클래스 다이어그램을 Java 코드로 변환하거나, 반대로 소스코드를 읽어 시퀀스 다이어그램을 자동으로 그려낼 수 있습니다. 이는 문서와 코드의 불일치를 막는 핵심 기술입니다.

### 2. 과목 융합: 디자인 패턴 (Pattern)
GoF 디자인 패턴은 모두 UML 클래스 다이어그램으로 정의되어 있습니다. 예를 들어, **전략 패턴(Strategy Pattern)**은 클래스 간의 위임(Delegation) 관계를 UML로 명확히 표현함으로써 알고리즘의 런타임 교체를 가능하게 하는 설계를 시각화합니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 복잡한 결제 도메인 설계
- **단계 1**: **클래스 다이어그램**으로 주문, 결제, 상품, 유저 간의 도메인 관계(Cardinality) 정립.
- **단계 2**: **시퀀스 다이어그램**으로 결제 요청이 들어왔을 때 외부 PG사와 통신하고 DB에 저장하는 순차적 메시지 흐름 설계 (에러 처리 포함).
- **단계 3**: **액티비티 다이어그램**으로 한도 초과, 잔액 부족 등 다양한 예외 비즈니스 로직(Swimlane 활용) 가시화.
- **판단**: 세 종류의 다이어그램을 병행 사용하여 정적인 뼈대와 동적인 시나리오를 모두 검증함으로써 설계 누락을 방지.

### 📢 기술사적 결언
> "UML은 그 자체가 목적이 아니라 **'생각의 도구'**다. 모든 다이어그램을 완벽하게 그릴 필요는 없다. 구조적 복잡성이 높을 때는 클래스 다이어그램을, 상호작용의 선후 관계가 중요할 때는 시퀀스 다이어그램을 선택하여 그리는 **'선택적 모델링'**이 실무 아키텍트의 지혜다. 가장 좋은 다이어그램은 화이트보드에 쓱싹 그려서 동료가 1분 안에 이해할 수 있는 그림이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적/정성적 기대효과
- **오류 감소**: 구현 전 설계 검증을 통해 로직 결함 40% 이상 사전 탐지.
- **커뮤니케이션**: 비전문가(고객)와 전문가 간의 기술적 간극 메우기.

### 미래 전망
미래의 UML은 **'실행 가능한 모델 (Executable UML)'**로 진화하고 있습니다. 그림을 그리면 바로 시스템이 동작하는 로우코드(Low-code) 플랫폼의 핵심 기술로 자리 잡고 있습니다. 또한 AI가 아키텍처 문서를 읽고 최적의 UML 다이어그램을 추천하거나, 다이어그램 간의 모순(Inconsistency)을 찾아내어 설계 품질을 자동 검수하는 스마트 모델링 시대로 나아가고 있습니다.
