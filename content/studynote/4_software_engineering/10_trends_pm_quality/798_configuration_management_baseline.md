# 798. 형상 통제 베이스라인 변경 심의 이력 추적

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 개발 생명주기 동안 발생하는 모든 산출물(형상 항목)의 변경을 체계적으로 승인, 기록, 검증하여 시스템의 **무결성(Integrity)과 가시성**을 확보하는 품질 관리 활동이다.
> 2. **통제 메커니즘**: 공식적으로 합의된 기준선인 **베이스라인(Baseline)**을 수립하고, 이후의 모든 변경 요청(CR)에 대해 **형상 통제 위원회(CCB)**의 심의를 거쳐 이력을 관리함으로써 무분별한 변경을 차단한다.
> 3. **가치**: "누가, 언제, 왜, 무엇을" 바꿨는지에 대한 완벽한 **추적성(Traceability)**을 제공하여, 장애 발생 시 신속한 원복과 원인 파악을 가능케 하며 프로젝트의 범위를 안정적으로 통제한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "변화의 파도 속에서 길을 잃지 않는 법"

개발 과정에서 소스코드는 수만 번 바뀌고, 설계서도 끊임없이 업데이트됩니다. 만약 이 변화를 기록하지 않는다면 "어제까지 잘 되던 게 왜 오늘 안 되지?"라는 질문에 대답할 수 없습니다. **형상 통제 (Configuration Control)**는 무질서한 변경을 방지하고, 시스템이 약속된 모습대로 유지되도록 만드는 '프로젝트의 기억장치'입니다.

### 💡 비유: 도서관의 희귀본 관리 시스템

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        형상 통제 및 베이스라인 비유                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 아주 소중한 백과사전(시스템)을 여러 명이 함께 고쳐 써야 함.              │
│                                                                             │
│  1. 베이스라인 (공식 판본): "자, 여기까지가 1판이다!"라고 도장을 찍고 금고에      │
│     넣어둔 공식 상태. 이후에는 누구도 마음대로 내용을 고칠 수 없음.              │
│                                                                             │
│  2. 변경 심의 (수정 요청): 내용을 고치고 싶은 사람은 '수정 제안서'를 써서         │
│     도서관 위원회(CCB)에 제출해야 함. 위원회가 "좋아!"라고 승인해야만 금고를      │
│     열어줌.                                                                    │
│                                                                             │
│  3. 이력 추적 (기록장): "홍길동이 3월 15일에 오타를 고침"이라고 꼼꼼히 적음.      │
│                                                                             │
│  → 함부로 낙서하지 못하게 하고(통제), **누가 바꿨는지 다 적어두는** 꼼꼼한 관리!   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 형상 통제의 4대 핵심 절차

| 단계 | 활동 명칭 | 상세 내용 |
|:---:|:---|:---|
| **1** | **변경 요청 (CR)** | 사용자나 개발자가 변경의 필요성을 문서로 제안 |
| **2** | **영향 분석** | 변경 시 일정, 비용, 타 모듈에 미치는 리스크 분석 |
| **3** | **CCB 심의** | **형상 통제 위원회**에서 승인/기각/보류 결정 |
| **4** | **실행 및 기록** | 변경 사항 반영 후 버전 번호 부여 및 이력 기록 (Baseline 갱신) |

### 2. 베이스라인 (Baseline)의 종류
- **기능적 베이스라인**: 요구사항 분석 완료 후.
- **설계 베이스라인**: 상세 설계 완료 후.
- **제품 베이스라인**: 개발 및 테스트 완료 후 최종 릴리즈 버전.

### 3. 형상 통제 워크플로우 (ASCII)

```text
    [ Current Baseline ] ──▶ [ Change Request ] ──▶ [ Impact Analysis ]
                                                         │
          ┌──────────────────────────────────────────────┘
          ▼ [ CCB Meeting ]
    ┌───────────┬───────────┐
    │  REJECT   │  APPROVE  │ ──▶ [ Implement Change ]
    └─────┬─────┴─────┬─────┘               │
          │           ▼                     ▼
    [ Notification ] [ Verify / Test ] ──▶ [ New Baseline ]
                                            (Audit Trail Saved)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 버전 관리(VCS) vs 형상 관리(SCM)
- **VCS (Git, SVN)**: 코드의 '차이'를 기록하는 **기술적 도구**.
- **SCM**: 도구뿐만 아니라 조직, 절차, 규정을 포함하는 **관리적 프레임워크**.
- **차이**: "Git commit을 했다"는 것은 버전 관리지만, "CCB 승인을 받고 Merge 했다"는 것은 형상 통제입니다.

### 2. 기술적 시너지: GitOps와 Immutable Infrastructure
현대의 형상 통제는 **GitOps**로 자동화됩니다. 코드 리뷰 승인(Approve) 행위 자체가 CCB의 역할을 대행하며, Merge와 동시에 베이스라인(Immutable Image)이 생성되어 운영 환경으로 배포되는 '실행 가능한 거버넌스'를 실현합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 마감 직전의 긴급 요구사항 변경
- **상황**: 오픈 3일 전, 고객이 로고 크기 변경을 요구. 개발자는 "금방 해요"라며 수정하려 함.
- **결단**: **긴급 CCB 소집 및 영향도 평가**. 
- **판단**: 로고 변경이 CSS 전체 구조를 흔들어 다른 페이지 레이아웃을 깨뜨릴 위험 포착. 
- **조치**: "오픈 후 첫 번째 패치로 반영"하기로 합의하고 현재 베이스라인 유지.
- **효과**: 무분별한 수정으로 인한 오픈 당일의 '레이아웃 깨짐' 장애를 미연에 방지.

### 📢 기술사적 결언
> "형상 통제는 프로젝트의 **'닻'**이다. 닻을 내리지 않은 배는 조류에 휩쓸려 어디로 갈지 모른다. 아키텍트는 개발자들에게 형상 통제가 '절차적 방해'가 아닌 **'자신의 코드를 보호하는 안전장치'**임을 인식시켜야 한다. 특히 베이스라인 사이의 **'추적성(Traceability)'**이 끊어지는 지점이 곧 기술 부채의 발원지임을 명심하고, 자동화된 감사(Audit) 체계를 아키텍처의 기본 요소로 포함해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **재작업 감소**: 버전 혼선으로 인한 롤백 및 재작업 시간 50% 이상 단축.
- **무결성 보장**: 요구사항과 최종 산출물 간의 일치도 100% 확보.

### 미래 전망
미래의 형상 통제는 **'AI 기반 영향도 분석'**으로 진화할 것입니다. 변경 요청이 들어오면 AI가 수백만 줄의 코드와 문서를 1초 만에 분석하여 "이 변경은 결제 모듈에 85% 확률로 영향을 줍니다"라고 CCB에게 사전 보고서를 제출할 것입니다. 또한 **블록체인** 기술과 결합하여, 수정 이력을 결코 조작할 수 없는 '절대적 신뢰의 형상 관리' 체계가 구축될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **형상 통제**는 우리가 만드는 로봇 설계도가 섞이지 않게 **"진짜 설계도"**를 금고에 넣어두는 거예요.
2. 설계도를 고치고 싶으면, 대장(CCB)에게 허락을 받고 **"누가, 왜 고쳤는지"** 공책에 꼭 적어야 하죠.
3. 이렇게 꼼꼼하게 기록해두면, 나중에 로봇이 고장 나도 **언제부터 잘못됐는지** 금방 찾아낼 수 있답니다!
