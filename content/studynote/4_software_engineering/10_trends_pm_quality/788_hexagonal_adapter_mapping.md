# 788. 헥사고날 아키텍처 어댑터 포트 매핑 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 알리스테어 코번(Alistair Cockburn)이 제안한 아키텍처로, 비즈니스 로직을 담은 **코어(Core)**를 중심에 두고 외부 환경(DB, UI, API)과의 연결을 **포트(Port)**와 **어댑터(Adapter)**로 추상화하여 물리적으로 격리하는 설계 방식이다.
> 2. **인터페이스 설계**: 코어가 외부로 나가는 길인 **Output Port**와 외부에서 코어로 들어오는 길인 **Input Port**를 정의하고, 실제 기술적 구현(SQL, REST 등)은 어댑터가 담당하여 코어의 순수성을 보장한다.
> 3. **가치**: 인프라 기술의 변화(예: MySQL -> MongoDB)에도 비즈니스 로직은 수정 없이 유지되는 높은 유연성을 제공하며, 실제 환경 없이도 테스트 대역(Mock)을 통해 완벽한 독립 테스트를 가능케 한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "기술이 도메인을 삼키지 않게 하라"

우리는 종종 '데이터베이스가 무엇인가'에 따라 코드를 짭니다. 이는 비즈니스 로직이 기술에 종속되는 결과를 낳습니다. **헥사고날 아키텍처 (Hexagonal Architecture)**는 시스템을 하나의 육각형(Hexagon)으로 보고, 모든 외부 장치들은 이 육각형의 면(Port)에 꽂히는 플러그인일 뿐이라는 철학을 제시합니다.

### 💡 비유: 범용 여행용 어댑터와 전자기기

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        헥사고날 아키텍처 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 코어 (전자기기 - 노트북):                                                │
│     노트북은 어느 나라에 가든 전기가 필요함. (비즈니스 로직은 어디서나 동일)       │
│                                                                             │
│  2. 포트 (충전 구멍 - Port):                                                 │
│     노트북에 뚫려 있는 표준 충전 단자(Interface). 어떤 콘센트 모양이든           │
│     이 구멍으로 전기만 들어오면 됨.                                            │
│                                                                             │
│  3. 어댑터 (국가별 돼지코 - Adapter):                                         │
│     한국의 220V, 미국의 110V 등 각기 다른 콘센트(Infra)를 노트북의 표준 구멍에     │
│     맞게 바꿔주는 변환기.                                                      │
│                                                                             │
│  → 나라(기술 환경)가 바뀌어도 **돼지코(Adapter)만 갈아 끼우면** 노트북은          │
│     그대로 쓸 수 있는 유연한 구조!                                             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 헥사고날 아키텍처의 구성 요소

| 요소 명칭 | 역할 | 구체적 사례 |
|:---:|:---|:---|
| **Domain Core** | 순수 비즈니스 로직 및 엔티티 | 주문 처리, 이자 계산 로직 |
| **Input Port** | 외부에서 내부로 들어오는 진입점 인터페이스 | `OrderUseCase` 인터페이스 |
| **Output Port** | 내부에서 외부로 나가는 호출 인터페이스 | `SaveOrderPort` 인터페이스 |
| **Input Adapter** | 외부 요청을 코어에 맞게 변환하여 전달 (Driving) | Web Controller, CLI, Message Consumer |
| **Output Adapter**| 코어의 요청을 외부 인프라에 맞게 구현 (Driven) | JPA Repository, Mail Client, API Client |

### 2. 제어 흐름과 의존성 역전 (DIP)
- **제어 흐름**: Input Adapter ──▶ Core ──▶ Output Adapter
- **의존성 방향**: Input Adapter ──▶ **[ Core ]** ◀── Output Adapter
- **핵심**: 외부(인프라)는 안쪽(코어)을 알지만, 코어는 밖을 전혀 모름.

### 3. 아키텍처 구조도 (ASCII)

```text
    [ Driving Side ]           [ Domain Core ]           [ Driven Side ]
    (Outer World)              (Inside Hexagon)          (Infrastructure)
    
    ┌─────────────┐            ┌─────────────┐           ┌─────────────┐
    │ Web Control │ ──(Call)─▶ │ [Input Port]│           │             │
    │ (Adapter)   │            ├─────────────┤           │             │
    └─────────────┘            │   Domain    │           │ DB / JPA    │
                               │   Logic     │           │ (Adapter)   │
    ┌─────────────┐            ├─────────────┤           │             │
    │ CLI / Script│ ──(Call)─▶ │[Output Port]│ ──(Impl)▶ │             │
    │ (Adapter)   │            └─────────────┘           └─────────────┘
    └─────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 계층형 아키텍처 vs 헥사고날 아키텍처
- **계층형**: 상위가 하위(주로 DB)에 직접 의존하여 DB 변경 시 상위가 파괴됨.
- **헥사고날**: 코어가 중심이며, DB는 단순한 '외부 서비스'로 취급되어 대칭적 구조를 가짐.

### 2. 기술적 시너지: 마이크로서비스 (MSA)
헥사고날은 마이크로서비스 하나하나를 설계하는 데 가장 이상적인 내부 구조입니다. 외부 통신 규약(gRPC, REST)이 바뀌어도 내부 로직은 보호받기 때문에, MSA의 핵심인 '독립적 진화'를 가능케 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 외부 연동이 잦은 결제 시스템
- **상황**: 처음엔 카카오페이만 연동했는데, 나중에 토스, 네이버페이가 계속 추가됨.
- **결단**: **포트 기반 추상화 및 멀티 어댑터 구현**. 
- **판단**: 코어 로직에는 `PaymentPort` 인터페이스만 둠. 
- **조치**: 각 결제사별로 전용 **어댑터**를 만듦. 
- **효과**: 신규 결제사 추가 시 코어 엔진은 단 한 줄의 코드 수정도 없이 설정만으로 연동 성공.

### 📢 기술사적 결언
> "헥사고날 아키텍처는 **'대칭의 미학'**이다. 사용자(User)와 데이터베이스(DB)를 동일한 '외부 주체'로 대우함으로써, 비즈니스 로직의 독존적 지위를 확립한다. 아키텍트는 포트를 설계할 때 특정 기술의 냄새(예: SQL 파라미터)가 배어 나오지 않도록 철저히 **'도메인 용어'**로만 정의해야 한다. 그래야만 기술의 홍수 속에서도 흔들리지 않는 영속적인 아키텍처를 가질 수 있다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **테스트 커버리지**: 인프라 종속성 제거로 비즈니스 로직 테스트 커버리지 100% 달성 용이.
- **마이그레이션 비용**: DB 또는 프레임워크 교체 시 리팩토링 비용 80% 이상 절감.

### 미래 전망
미래의 헥사고날 아키텍처는 **'자율 어댑터 생성'**으로 진화할 것입니다. 코어의 포트 명세만 정의하면, AI가 최신 클라우드 기술에 맞는 최적의 어댑터 코드(AWS SDK 연동 등)를 자동으로 생성해 주는 'Low-code Hexagon' 체계가 구축될 것입니다. 또한 서버리스 환경에서 각 어댑터가 독립적인 사이드카(Sidecar)로 기동되어 더욱 완벽한 격리를 실현하는 구조가 표준이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **헥사고날 아키텍처**는 복잡한 로봇 본체(코어)에 **"여러 개의 구멍(포트)"**을 뚫어두는 것과 같아요.
2. 어떤 구멍에는 바퀴를 끼우고, 어떤 구멍에는 드릴을 끼우는 식으로 **부품(어댑터)만 갈아 끼우면** 로봇이 변신하죠.
3. 부품을 바꿔도 로봇의 뇌는 그대로니까, **공부한 건 잊어버리지 않고** 새로운 능력만 가질 수 있답니다!
