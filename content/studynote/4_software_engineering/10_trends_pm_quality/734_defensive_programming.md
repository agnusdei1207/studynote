# 734. 방어적 프로그래밍 Assertion 계약 기반 설계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 예상치 못한 비정상적인 입력이나 실행 환경에서도 소프트웨어가 오작동하거나 중단되지 않고, 의도된 대로 안전하게 동작하도록 코드를 작성하는 **공격적 방어 설계 기법**이다.
> 2. **핵심 도구**: 개발 단계에서 논리적 모순을 즉시 식별하는 **단언문(Assertion)**과, 호출자와 수신자 간의 책임 관계를 명시하는 **계약 기반 설계(Design by Contract)**를 통해 결함의 조기 발견과 격리를 실현한다.
> 3. **가치**: "쓰레기를 넣어도 쓰레기가 나오지 않게(No GIGO)" 방어하여 시스템의 강건성(Robustness)을 높이고, 디버깅 효율을 극대화하여 고신뢰성 소프트웨어를 구축한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "아무도 믿지 마라"

소프트웨어 버그의 상당수는 "절대 일어나지 않을 것"이라고 믿었던 상황에서 발생합니다. 외부 API가 갑자기 Null을 반환하거나, 사용자가 나이 입력창에 이름을 적는 식입니다. **방어적 프로그래밍 (Defensive Programming)**은 "모든 입력은 잠재적 독이다"라는 의심에서 시작하여, 코드 곳곳에 안전장치를 설치하는 활동입니다.

### 💡 비유: 자동차의 에어백과 가이드레일

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        방어적 프로그래밍 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 가파른 산길을 운전해야 함.                                             │
│                                                                             │
│  1. 일반 프로그래밍: 지도를 믿고 앞만 보고 달림. (길이 끊기면 낭떠러지 추락)      │
│                                                                             │
│  2. 방어적 프로그래밍 (안전 장치):                                            │
│     - 가이드레일: 길 밖으로 나가지 못하게 물리적으로 막음 (입력값 제한).          │
│     - 에어백 (Assertion): 혹시 사고가 나더라도 운전자가 다치기 전에 즉시 팽창     │
│       해서 멈춤 (논리 오류 시 즉시 중단).                                       │
│     - 보조 브레이크: 메인 브레이크가 고장 나도 멈출 수 있는 수단 확보 (예외 처리). │
│                                                                             │
│  → 사고가 **안 날 것이라 믿는 게 아니라**, 사고가 **나도 살 수 있게** 만드는 것!   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 방어 기법 및 전략

| 기법 | 설명 | 적용 시점 |
|:---:|:---|:---|
| **입력 유효성 검사** | 모든 외부 데이터의 타입, 범위, 길이를 화이트리스트 기반으로 체크 | 함수 진입부 |
| **단언문 (Assertion)** | "이 변수는 반드시 양수여야 함"과 같이 개발자가 확신하는 조건을 명시 | 개발/테스트 단계 |
| **예외 처리 (Exception)** | 예측 가능한 오류 상황에 대한 복구 로직 정의 (Try-Catch) | 런타임/운영 단계 |
| **기본값(Default) 활용** | 오류 발생 시 시스템을 중단하는 대신 안전한 기본값 반환 | 서비스 가용성 중시 시 |

### 2. 단언문 (Assertion)의 특징
- **목적**: 프로그램의 논리적 모순을 잡아내기 위한 **디버깅용 도구**.
- **동작**: 조건이 거짓(False)이면 즉시 프로그램을 중단시키고 에러 메시지 출력.
- **주의**: 운영 환경(Production)에서는 성능을 위해 꺼두는 것이 일반적임 (예외 처리와 다른 점).

### 3. 계약 기반 설계 (Design by Contract, DbC)
함수 호출을 하나의 '비즈니스 계약'으로 봅니다.
- **선행 조건 (Pre-condition)**: 호출자가 지켜야 할 의무 (예: 인자는 0보다 커야 함).
- **후행 조건 (Post-condition)**: 수신자가 보장해야 할 결과 (예: 반환값은 항상 정렬되어 있음).
- **불변 조건 (Invariant)**: 실행 전후에 항상 변하지 않아야 할 상태 (예: 잔액은 0보다 큼).

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 예외 처리(Exception) vs 단언문(Assertion)

| 항목 | 예외 처리 (Exception) | 단언문 (Assertion) |
|:---:|:---|:---|
| **발생 원인** | 외부 환경, 유저 실수 (예측 가능) | **프로그램 논리 오류 (개발자 실수)** |
| **처리 목표** | 프로그램 가동 유지 (Graceful) | **즉시 중단 및 버그 수정 (Fail-fast)** |
| **사용 시점** | 런타임 전체 | 주로 개발 및 테스트 단계 |
| **비유** | "비가 오면 우산을 쓰자" | **"지구가 둥글지 않으면 멈춰라"** |

### 2. 기술적 시너지: 클린 코드 (Clean Code)
방어적 프로그래밍을 너무 과하게 하면 코드가 지저분해질 수 있습니다. 이를 방지하기 위해 **'가드 절(Guard Clause)'** 패턴을 사용하여 함수의 시작 부분에서 잘못된 입력을 먼저 걸러내고(Return Early), 본문에서는 순수한 비즈니스 로직에 집중하는 설계가 시너지를 냅니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 결제 시스템의 금액 연산
- **상황**: 복잡한 할인율 계산 후 최종 결제 금액 산출.
- **결단**: **중요 구간에 Assertion 및 DbC 적용**. 
- **판단**: 
    1. 선행 조건: `assert discountRate >= 0 && discountRate <= 1`.
    2. 후행 조건: `assert finalAmount <= originalAmount`.
- **효과**: 개발 과정에서 할인율이 마이너스가 되거나 결제 금액이 원금보다 커지는 논리 버그를 1초 만에 발견하여 대형 금융 사고 예방.

### 📢 기술사적 결언
> "방어적 프로그래밍은 **'겸손한 엔지니어'**의 자세다. 나는 언제든 실수할 수 있고, 사용자는 언제든 이상한 값을 넣을 수 있다는 사실을 인정하는 것이다. 아키텍트는 시스템의 **'실패 지점(Fail-fast vs Fail-safe)'**을 명확히 정의해야 한다. 내부 개발자 간의 약속은 Assertion으로 강제하고, 외부와의 접점은 철저한 유효성 검사로 방어하는 이원화 전략이 시스템의 강건성을 결정한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **디버깅 시간**: Assertion 활용 시 오류 원인 지점 특정 시간 70% 단축.
- **보안성 향상**: 주입 공격(SQLi 등)의 90% 이상을 입구 단계에서 차단.

### 미래 전망
미래의 방어적 프로그래밍은 **'형식 검증 (Formal Verification)'**과 결합할 것입니다. 코드를 돌려보지 않고도 수학적으로 모든 경로에서 계약(DbC)이 지켜지는지 AI가 증명해 주는 수준에 도달할 것입니다. 또한 **메모리 안전 언어(Rust 등)**의 보편화로 인해, 개발자가 직접 코딩하지 않아도 언어 차원에서 방어막이 쳐지는 'Secure by Default' 환경이 표준이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **방어적 프로그래밍**은 요리할 때 실수로 **"소금 대신 설탕을 넣지 않게"** 미리 통에 이름을 크게 써두는 거예요.
2. **단언문**은 "이 통엔 무조건 소금이 들어있어야 해!"라고 소리치는 마법 주문과 같아서, 다른 게 들어있으면 요리를 **즉시 멈추게** 하죠.
3. 이렇게 꼼꼼하게 준비하면, 요리 도중에 실수를 해도 **맛없는 음식이 만들어지는 걸** 미리 막을 수 있답니다!
