# 773. 3-Layer 아키텍처 (Presentation, Logic, Data)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템의 복잡도를 관리하기 위해 전체 구성을 사용자 인터페이스(**Presentation**), 비즈니스 규칙(**Logic**), 데이터 저장(**Data**)의 3가지 논리적 계층으로 분리한 **가장 전형적인 계층형 아키텍처**이다.
> 2. **관심사 분리**: 각 계층은 자신에게 주어진 역할에만 집중하고 인접한 계층과만 소통함으로써, 특정 계층의 기술 스택 변경이 다른 계층에 미치는 영향을 최소화한다.
> 3. **가치**: 모듈화된 설계를 통해 개발 생산성을 높이고, 계층별 독립적인 확장(Scaling)과 유지보수를 가능하게 하여 엔터프라이즈 시스템의 표준 구조로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "거대한 덩어리 코드의 한계"

초기의 소프트웨어는 화면을 그리는 코드와 DB를 조회하는 코드가 한 파일에 뒤섞여 있었습니다. 이는 작은 화면 수정 하나에도 DB 로직이 깨질 위험을 초래했습니다. **3-Layer (또는 3-Tier) 아키텍처**는 "UI는 UI답게, 로직은 로직답게, 데이터는 데이터답게" 분리하여 시스템의 유연성을 확보하려는 노력에서 시작되었습니다.

### 💡 비유: 레스토랑의 분업 시스템

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        3-Layer 아키텍처 비유                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. Presentation Layer (홀 직원):                                            │
│     손님을 맞이하고 메뉴판(UI)을 보여줌. 손님의 주문(Request)을 받아 주방에 전달함. │
│                                                                             │
│  2. Logic Layer (주방 요리사):                                               │
│     실제 요리(비즈니스 규칙)를 함. 재료를 볶고 찌는 복잡한 과정은 여기서 일어남.     │
│                                                                             │
│  3. Data Layer (식재료 창고):                                                │
│     고기, 채소(데이터)를 보관하는 곳. 요리사가 요청하면 재료를 꺼내줌.             │
│                                                                             │
│  → 손님이 바뀌어도 주방은 그대로고, 창고가 바뀌어도 홀 직원은 친절한 것과 같음!      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 계층별 역할 및 구성 요소

| 계층 | 명칭 | 주요 역할 | 주요 기술/구성 |
|:---:|:---|:---|:---|
| **Tier 1** | **Presentation (표현)** | 사용자 인터페이스 제공, 입력 유효성 검사 | HTML/CSS, React, Android UI, Controller |
| **Tier 2** | **Logic (비즈니스/서비스)** | 실제 업무 로직 처리, 트랜잭션 관리 | Service, Domain Model, Workflow |
| **Tier 3** | **Data (영속성/저장)** | 데이터의 영구 저장 및 검색 | Database(RDBMS), File System, DAO |

### 2. 계층 간 소통 규칙
- **하향식 의존**: 상위 계층은 하위 계층을 알지만, 하위 계층은 상위 계층을 몰라야 함.
- **인접 계층 간 소통**: 1계층에서 3계층으로 바로 가는 것은 금지 (Logic 계층을 반드시 거쳐야 함).

### 3. 아키텍처 구조도 (ASCII)

```text
    [ Client / User ]
           │
    ┌──────▼───────────────────┐
    │   Presentation Layer     │ (Web / Mobile App)
    └──────┬───────────────────┘
           │ (Request DTO)
    ┌──────▼───────────────────┐
    │      Logic Layer         │ (Business Logic & Service)
    └──────┬───────────────────┘
           │ (SQL / Query)
    ┌──────▼───────────────────┐
    │       Data Layer         │ (DBMS / Storage)
    └──────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 3-Layer vs MVC 패턴
- **MVC**: **Presentation 계층 내부**를 Model, View, Controller로 쪼개는 패턴. (미시적)
- **3-Layer**: **시스템 전체**를 물리적/논리적으로 3개로 쪼개는 아키텍처. (거시적)
- **시너지**: 보통 Presentation 계층은 MVC 패턴으로 구현되고, 그 모델(Model)이 Logic 계층과 대화하는 구조를 취합니다.

### 2. 기술적 시너지: 클라우드 오케스트레이션
3-Layer 아키텍처는 클라우드에서 각각 독립적인 노드군으로 배포됩니다. 부하가 몰릴 때 화면을 처리하는 '프론트엔드 서버'만 수평 확장(Auto-scaling)하거나, 데이터량이 늘어날 때 'DB 서버'만 고사양으로 업그레이드할 수 있는 유연성을 제공합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 모바일 전용 앱에서 웹 앱으로 전환
- **상황**: 기존 Android 전용 앱을 웹 브라우저에서도 쓸 수 있게 확장해야 함.
- **결단**: **Presentation Layer만 신규 개발**. 
- **판단**: 3-Layer 아키텍처가 잘 잡혀 있다면, 핵심 비즈니스 로직(Logic)과 데이터(Data)는 건드릴 필요가 없음. 
- **효과**: 전체 로직의 70% 이상을 재사용하며 최소 비용으로 멀티 채널 서비스 확장 성공.

### 📢 기술사적 결언
> "3-Layer 아키텍처는 **'엔지니어링의 질서'**를 잡는 가장 기본적이고 강력한 틀이다. 계층을 나누는 이유는 단순히 깔끔해보이기 위해서가 아니라, **'변화의 속도'**가 서로 다르기 때문이다. UI는 매주 변하고, 로직은 분기별로 변하며, 데이터는 수년간 유지된다. 아키텍트는 이 변화의 주기를 계층으로 격리하여, 사소한 변화가 시스템 전체의 재앙이 되지 않도록 방어해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **유지보수 효율**: 코드 수정 시 영향도 분석 범위 60% 이상 축소.
- **병렬 개발 가능**: 계층별 인터페이스 합의 후 각 팀이 독립적으로 동시 개발 가능.

### 미래 전망
미래의 3-Layer 아키텍처는 **'Micro-layered Architecture'**로 더욱 세분화될 것입니다. Logic 계층이 수백 개의 마이크로서비스로 쪼개지고, Data 계층이 분산 데이터 메시로 진화하더라도 '역할의 분리'라는 3-Layer의 근본 원칙은 변하지 않을 것입니다. 또한 **Serverless** 환경과 결합하여, 각 계층이 필요할 때만 기동되고 사라지는 '유동적 계층 아키텍처'가 표준이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **3-Layer**는 우리 몸의 **"입, 뇌, 위장"**이 하는 일이 다른 것과 같아요.
2. 입은 음식을 받고(표현), 뇌는 맛을 느끼고(로직), 위장은 소화해서 영양분을 저장(데이터)하죠.
3. 입이 아프다고 뇌나 위장까지 아픈 게 아닌 것처럼, **따로따로 맡은 일을 잘해야** 우리 몸(시스템)이 건강할 수 있답니다!
