# 731. 엔티티와 값 객체 (VO) 불변성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도메인 주도 설계(DDD)에서 비즈니스 객체를 모델링하는 두 가지 핵심 방식으로, 고유한 식별자를 가진 **엔티티(Entity)**와 속성값 자체로 정의되는 **값 객체(Value Object, VO)**로 구분된다.
> 2. **식별성 vs 불변성**: 엔티티는 시간이 흘러도 변하지 않는 **정체성(Identity)**을 추적하는 데 집중하고, VO는 상태를 변경하는 대신 새로운 객체로 교체하는 **불변성(Immutability)**을 통해 부수 효과(Side Effect)를 방지한다.
> 3. **가치**: 객체의 성격에 맞는 명확한 역할을 부여함으로써 도메인 모델의 복잡성을 낮추고, 데이터 무결성과 코드의 예측 가능성을 획기적으로 향상시킨다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "모든 것이 데이터베이스 테이블인가?"

전통적인 데이터 중심 설계에서는 모든 객체를 DB 테이블의 행(Row)으로만 생각했습니다. 하지만 비즈니스 로직에서는 '사람'처럼 추적해야 할 대상과 '주소'처럼 값 자체가 중요한 대상을 구분해야 합니다. **엔티티**와 **VO**는 객체지향의 본질을 살려 도메인의 의도를 코드로 명확히 표현하기 위해 고안되었습니다.

### 💡 비유: 사람과 지폐의 차이

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        엔티티와 값 객체(VO) 비유                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 엔티티 (사람 - 주민등록번호):                                             │
│     철수가 머리를 깎고 옷을 갈아입어도(상태 변경), 철수는 여전히 철수임.          │
│     이름이나 나이가 변해도 **주민등록번호(Identity)**가 같으면 동일 인물로 간주.   │
│                                                                             │
│  2. 값 객체 (만원권 지폐 - 일련번호 무시):                                     │
│     내 주머니에 있는 만 원과 네 주머니에 있는 만 원은 가치가 같음.                │
│     돈에 적힌 글자(값)가 중요하지, 특정 종이 조각(Identity)이 중요한 게 아님.      │
│     - 만약 돈이 찢어지면? 수선하는 게 아니라 **새 돈으로 바꿔(Replace)**야 함.    │
│                                                                             │
│  → 추적해야 할 '주인공'은 엔티티, 성질을 나타내는 '소품'은 VO!                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 엔티티 vs 값 객체 비교 분석

| 항목 | 엔티티 (Entity) | 값 객체 (Value Object) |
|:---:|:---|:---|
| **정의 기준** | 고유한 식별자 (ID) | 내부 속성값의 조합 |
| **동일성 판별** | ID가 같으면 동일 객체 | **모든 속성값이 같으면** 동일 객체 |
| **생명주기** | 연속성과 변화를 추적해야 함 | 한 번 생성되면 상태가 변하지 않음 |
| **가변성** | 가변 (Mutable) | **불변 (Immutable)** |
| **DB 매핑** | 별도의 테이블로 존재 | 주로 엔티티 테이블의 컬럼으로 포함 |

### 2. 값 객체(VO)의 불변성 구현 메커니즘
VO는 생성자(Constructor)를 통해서만 값을 설정하고, 수정자(Setter)를 두지 않습니다.
- **Side Effect 제거**: 여러 곳에서 참조해도 값이 변할 걱정이 없음.
- **교체(Replacement)**: 값을 바꾸고 싶으면 기존 객체를 버리고 새로운 값을 가진 객체를 새로 할당.

### 3. 도메인 모델 구조 (ASCII)

```text
    ┌───────────────────────────────────────────┐
    │             [ Entity: User ]              │
    ├───────────────────────────────────────────┤
    │ - id: Long (Identity)                     │
    │ - name: String                            │
    │ - address: Address (Value Object) ────────┼──▶ [ Value Object: Address ]
    └───────────────────┬───────────────────────┘    ├────────────────────────┤
                        │                            │ - city: String         │
                        ▼                            │ - zipCode: String      │
                [ Lifecycle Track ]                  ├────────────────────────┤
                (Created -> Updated)                 │ + equals(other): bool  │
                                                     └────────────────────────┘
                                                     (No ID, Equality by Value)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 데이터베이스(JPA) 관점의 융합
- **Entity**: `@Entity` 어노테이션으로 매핑되어 PK를 가짐.
- **VO**: `@Embeddable` 또는 `@Embedded`를 통해 엔티티의 일부로 저장되거나, 별도 테이블로 관리되더라도 식별자 없이 부모 엔티티에 종속됨.

### 2. 클린 코드와 함수형 프로그래밍
VO의 불변성은 함수형 프로그래밍의 핵심 원칙과 일치합니다. 상태 변화를 최소화함으로써 멀티스레드 환경에서 동기화 문제(Race Condition)를 원천적으로 방지하고, 단위 테스트를 매우 단순하게 만듭니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 배송 주소 변경 로직
- **나쁜 예**: `order.getAddress().setCity("Seoul")` (엔티티 내부의 VO를 직접 수정 - 사이드 이펙트 위험)
- **좋은 예**: `order.changeAddress(new Address("Seoul", "12345"))` (새로운 VO를 생성하여 통째로 교체)
- **판단**: 주소의 일부 필드만 바꾸는 행위는 실수로 다른 참조 지점의 주소까지 바꿔버릴 수 있음. VO를 불변으로 유지하고 통째로 갈아 끼우는 '교체 전략'을 통해 비즈니스 정합성 유지.

### 📢 기술사적 결언
> "모든 객체를 엔티티로 만들려는 욕심이 시스템을 무겁게 만든다. 식별자가 필요 없는 속성들을 VO로 적극적으로 떼어내는 것만으로도 시스템의 **'개념적 무게'**를 획기적으로 줄일 수 있다. 아키텍트는 객체의 **'정체성'**과 **'속성'**을 날카롭게 구분하여, 추적해야 할 가치가 있는 것에만 ID를 부여하는 절제의 미학을 발휘해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **버그 발생률**: 불변 객체 사용 시 상태 불일치 관련 결함 40% 이상 감소.
- **유지보수성**: 도메인 로직이 VO 내부로 캡슐화되어(예: 주소 형식 검증) 응집도 향상.

### 미래 전망
미래의 개발 언어(Java의 Record, Kotlin의 Data Class 등)는 VO와 같은 불변 객체를 더 쉽고 안전하게 정의할 수 있는 기능을 표준으로 강화하고 있습니다. 또한 분산 시스템에서 데이터 일관성을 위해 상태를 기록하는 **이벤트 소싱(Event Sourcing)** 기법이 확산됨에 따라, 한 번 발생하면 변하지 않는 '사실'을 기록하는 VO의 가치는 더욱 높아질 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **엔티티**는 이름표(ID)를 붙인 **"내 소중한 인형"**과 같아요. 옷을 갈아입혀도 이름표가 같으면 항상 내 인형이죠.
2. **값 객체**는 색종이 조각과 같아요. 빨간색 종이는 누가 가지고 있든 그냥 **"빨간색"**이라는 게 중요하죠.
3. 빨간 종이가 구겨지면(수정) 다리미로 펴는 대신, **새로운 빨간 종이**를 꺼내 쓰는 게 더 편하고 깔끔한 것과 같답니다!
