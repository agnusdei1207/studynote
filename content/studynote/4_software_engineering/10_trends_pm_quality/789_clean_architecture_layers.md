# 789. 클린 아키텍처 엔티티 유스케이스 프레젠테이션 계층 분리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어의 핵심인 비즈니스 로직(Entities, Use Cases)을 외부 환경(UI, DB, Framework)으로부터 완벽하게 격리하여, **기술적 세부 사항이 비즈니스 본질을 오염시키지 않게** 설계하는 계층 분리 전략이다.
> 2. **의존성 규칙**: 모든 의존성은 반드시 저수준(세부 사항)에서 **고수준(추상화/비즈니스)** 방향으로만 향해야 하며, 이를 통해 내부 코어 로직의 독립성과 테스트 가능성을 극대화한다.
> 3. **가치**: UI가 웹에서 모바일로 바뀌거나, 데이터베이스 기술이 바뀌어도 핵심 비즈니스 코드는 단 한 줄도 수정할 필요가 없는 **영속적인 아키텍처**를 실현한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "왜 계층을 나누는가?"

우리는 종종 데이터베이스 테이블 구조에 맞춰 비즈니스 로직을 짜거나, 프레임워크가 제공하는 라이브러리에 핵심 정책을 섞어버립니다. 이는 시스템을 특정 기술에 종속시켜 변화를 어렵게 만듭니다. **클린 아키텍처 (Clean Architecture)**는 이러한 '기술의 침투'를 막기 위해 성벽(Layer)을 쌓고 통행 방향(Dependency)을 엄격히 제한합니다.

### 💡 비유: 햄버거 매장의 역할 분담

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        클린 아키텍처 계층 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 엔티티 (레시피 본질): "고기를 굽고 빵 사이에 끼운다"는 햄버거의 본질.         │
│     - 주방장이나 손님이 바뀌어도 이 규칙은 변하지 않음.                         │
│                                                                             │
│  2. 유스케이스 (주문 처리 절차): "손님이 치즈버거를 주문하면 결제 후 조리한다."   │
│     - 구체적인 판매 시나리오를 지휘함.                                          │
│                                                                             │
│  3. 프레젠테이션 (키오스크/앱): 손님이 주문하는 화면.                          │
│     - 키오스크가 최신형으로 바뀌어도(UI 변경), 햄버거 레시피는 그대로임!          │
│                                                                             │
│  → 겉모양(UI)이나 장비(DB)는 수시로 바뀌어도, **햄버거의 맛(핵심 로직)**은        │
│     가장 안쪽에서 안전하게 지켜지는 것!                                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 클린 아키텍처 4대 주요 계층

| 계층 | 명칭 | 주요 내용 및 역할 | 의존성 관계 |
|:---:|:---|:---|:---:|
| **Entities** | **도메인 엔티티** | 전사적인 핵심 비즈니스 규칙 (Entity, VO) | **최상위 (의존성 0)** |
| **Use Cases** | **애플리케이션 로직** | 특정 앱의 흐름 제어, 인터랙터 (Interactor) | 엔티티로 향함 |
| **Presenters** | **인터페이스 어댑터** | 외부 데이터(DB, UI)를 코어에 맞게 변환 | 유스케이스로 향함 |
| **Frameworks** | **인프라/드라이버** | DB, 웹 프레임워크, 도구 등 기술적 세부 사항 | 어댑터로 향함 |

### 2. 의존성 규칙 (The Dependency Rule)
- **안쪽으로만**: 소스코드의 의존성은 항상 원의 안쪽으로만 향해야 합니다.
- **DIP 활용**: 하위 계층(예: DB)이 상위 계층(유스케이스)의 기능을 호출해야 할 때는, 상위 계층에 인터페이스를 두고 하위 계층이 이를 구현하게 하여 의존성 방향을 역전시킵니다.

### 3. 구조적 아키텍처 (ASCII)

```text
    [ Outer World ] ────▶ [ Adapters ] ────▶ [ Use Cases ] ────▶ [ Entities ]
    (Web, Device)         (Controller)       (Interactor)        (Business Core)
    
    ※ "Everything flows towards the center."
    
    ┌──────────────────────────────────────────────┐
    │  Presentation Layer (Controllers, Web)       │
    │  ┌────────────────────────────────────────┐  │
    │  │  Application Layer (Use Cases)         │  │
    │  │  ┌──────────────────────────────────┐  │  │
    │  │  │  Domain Layer (Entities)         │  │  │
    │  │  └──────────────────────────────────┘  │  │
    │  └────────────────────────────────────────┘  │
    └──────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 3-Layer vs Clean Architecture
- **3-Layer**: 계층을 나누지만, 보통 비즈니스 계층이 데이터 계층에 직접 의존함. (DB 중심)
- **Clean**: 의존성 역전을 통해 DB가 비즈니스 계층에 의존하게 함. (비즈니스 중심)

### 2. 기술적 시너지: 플러그인 아키텍처
클린 아키텍처를 적용하면 DB나 UI는 시스템의 본체에 꽂히는 **'플러그인'**이 됩니다. 테스트할 때는 진짜 DB 플러그인을 뽑고 '가짜(Mock) 플러그인'을 꽂아도 비즈니스 로직은 자신이 테스트당하고 있다는 사실조차 모른 채 완벽히 동작합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 프레임워크 마이그레이션 (Spring -> Quarkus)
- **상황**: 성능 최적화를 위해 기존 자바 프레임워크를 교체해야 함.
- **결단**: **코어 로직 보존 및 어댑터 교체**. 
- **판단**: 클린 아키텍처로 설계되었다면, 프레임워크 의존성은 가장 바깥쪽인 'Frameworks' 계층에만 존재함. 
- **조치**: 핵심 엔티티와 유스케이스 코드는 그대로 두고, 컨트롤러와 DB 연동 코드(Adapter)만 신규 프레임워크에 맞춰 재작성.
- **효과**: 전체 로직의 80% 이상을 수정 없이 이관하여 성공적인 기술 스택 전환 달성.

### 📢 기술사적 결언
> "클린 아키텍처는 **'비즈니스 로직의 결벽증'**이다. 외부의 지저분한 기술적 세부 사항이 안쪽의 고결한 비즈니스 로직을 건드리지 못하게 격리하는 것이다. 아키텍트는 '어떤 프레임워크가 유행인가'보다 **'우리의 비즈니스 본질이 무엇인가'**를 코드로 표현하는 데 더 많은 시간을 써야 한다. 계층의 경계(Boundary)를 넘나드는 데이터(DTO)조차 특정 프레임워크에 종속되지 않게 설계하는 것이 클린 아키텍처의 완성이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **테스트 용이성**: 인프라 없이 수행하는 비즈니스 테스트 속도 10배 향상.
- **유지보수 수명**: 기술 노후화와 관계없이 시스템의 핵심 가치 수명 2배 이상 연장.

### 미래 전망
미래의 클린 아키텍처는 **'Serverless Use Case'**로 파편화될 것입니다. 각 유스케이스 계층이 독립적인 클라우드 함수(FaaS)로 배포되면서도, 공통 도메인 엔티티를 공유하는 '분산형 클린 아키텍처'가 대세가 될 것입니다. 또한 AI가 유스케이스 정의서만 보고 인터페이스 어댑터(DB, UI 코드)를 자동 생성해주는 시대가 오면, 개발자는 오직 가장 안쪽 원인 **'Entities'**를 설계하는 지적 활동에만 집중하게 될 것입니다.
