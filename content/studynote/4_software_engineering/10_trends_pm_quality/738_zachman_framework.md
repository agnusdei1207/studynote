# 738. 잭맨 프레임워크 6x6 매트릭스

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 조직 전체의 아키텍처를 정의하기 위해 6가지 관점(Viewpoints)과 6가지 질문(Abstractions)을 교차시킨 **엔터프라이즈 아키텍처(EA)의 논리적 분류 체계**이다.
> 2. **포괄성**: 경영진부터 실무 개발자까지 각기 다른 시각에서 보아야 할 '데이터, 기능, 네트워크, 사람, 시간, 동기'를 36개의 셀(Cell)로 구조화하여 정보 시스템의 전체 청사진을 제시한다.
> 3. **가치**: 특정 방법론이나 기술에 종속되지 않는 독립적인 분류 스키마를 제공하여, 복잡한 기업용 정보 자산의 누락 없는 식별과 일관성 있는 의사소통을 지원한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "코끼리의 전체 모습을 보는 법"

거대 기업의 IT 시스템은 너무 복잡해서 한 사람이 다 이해할 수 없습니다. 장님들이 코끼리를 만지듯, 경영자는 '비용'만 보고 개발자는 '코드'만 봅니다. 존 잭맨(John Zachman)은 1987년, 건축의 설계 원리를 빌려와 기업 전체를 체계적으로 기술할 수 있는 **6x6 매트릭스**를 제안했습니다. 이것이 현대 모든 EA 프레임워크의 시초입니다.

### 💡 비유: 영화 제작용 큐시트와 기획안

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        잭맨 프레임워크 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 수천 명의 스태프가 참여하는 초대형 블록버스터 영화를 찍어야 함.           │
│                                                                             │
│  1. 관점 (누가 보는가?):                                                     │
│     - 제작자 (경영): 예산은 얼마고 목표 관객은 누구인가?                        │
│     - 시나리오 작가 (개념): 이야기의 줄거리는 무엇인가?                         │
│     - 감독 (논리): 장면을 어떻게 배치하고 배우는 누굴 쓸까?                     │
│     - 특수효과팀 (물리): 어떤 컴퓨터 장비와 CG 프로그램을 쓸까?                  │
│                                                                             │
│  2. 질문 (무엇을 보는가?):                                                   │
│     - What(소품), How(액션), Where(촬영지), Who(등장인물), When(상영일), Why(주제) │
│                                                                             │
│  → 36칸의 표(Matrix)에 이 내용들을 다 채우면, **누가 봐도 완벽한 영화 설계도** 탄생! │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 잭맨 프레임워크의 6x6 매트릭스 구조

| 구분 (How/Where/Who 등) | **What (Data)** | **How (Function)** | **Where (Network)** | **Who (People)** | **When (Time)** | **Why (Motivation)** |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| **Planner (경영)** | 비즈니스 객체 | 비즈니스 프로세스 | 사업 장소 | 조직 구조 | 이벤트 주기 | 사업 비전 |
| **Owner (개념)** | 개념 모델 (ERD) | 프로세스 모델 | 물류 네트워크 | 업무 역할 | 비즈니스 일정 | 비즈니스 규칙 |
| **Designer (논리)** | 논리 데이터 모델 | 시스템 설계 | 시스템 아키텍처 | 사용자 인터페이스 | 처리 순서 | 논리적 제약 |
| **Builder (물리)** | 물리 데이터 구조 | 기술 설계 | 네트워크 기술 | 보안 제어 | 실행 주기 | 기술적 규칙 |
| **Sub-contractor** | 데이터 정의 | 프로그램 모듈 | 통신 프로토콜 | 접근 권한 | 타이밍 로직 | 기능 제약 |
| **User (실제)** | 실제 데이터 | 실제 기능 | 실제 네트워크 | 실제 사용자 | 실제 시간 | 실제 비즈니스 |

### 2. 매트릭스의 특징
- **독립성**: 각 셀은 서로 독립적이며, 하나의 셀을 채운다고 다른 셀이 자동으로 채워지지 않음.
- **포괄성**: 36개의 셀을 모두 채우면 기업 정보화의 '완전한 기술'이 완료됨.
- **중립성**: 특정 도구나 방법론을 강요하지 않는 참조 모델(Reference Model).

### 3. 논리적 계층 구조 (ASCII)

```text
    (High Abstraction)
    [ Planner ] ──────────┐
    [ Owner   ] ──────▶ [ 6 Questions ] ──────▶ [ 36 Cells of Architecture ]
    [ Designer]           (What, How, Where,      (The Blueprint)
    [ Builder ]            Who, When, Why)
    (Low Detail)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 잭맨 프레임워크 vs TOGAF
- **Zachman**: "기업 아키텍처를 **어떻게 분류할 것인가(Classification)**"에 대한 정적인 스키마. (사전/도서관)
- **TOGAF**: "아키텍처를 **어떻게 구축할 것인가(Process)**"에 대한 동적인 방법론(ADM). (요리법)
- **시너지**: Zachman으로 분류 체계를 잡고, TOGAF의 ADM 프로세스를 따라 결과물을 만들어냅니다.

### 2. 기술적 시너지: 데이터 거버넌스
잭맨의 'What(Data)' 열은 엔터프라이즈 데이터 거버넌스의 기초가 됩니다. 경영진의 비즈니스 객체(Planner)부터 실제 DB의 테이블(Sub-contractor)까지의 추적성(Traceability)을 확보하는 기준이 됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 전사적 자원 관리(ERP) 도입 전 설계
- **상황**: 복잡한 사내 시스템을 ERP로 통합하려는데, 어디서부터 손을 댈지 모름.
- **결단**: **잭맨 매트릭스를 활용한 현황 분석(As-Is)**. 
- **판단**: 36개의 셀 중 비어있는 곳(문서화 안 된 곳) 식별. 
- **효과**: "우리 회사는 기술 아키텍처(Builder-Where)는 강하지만, 비즈니스 규칙(Owner-Why) 관리가 전혀 안 되고 있다"는 근본적인 문제점 도출 및 개선 로드맵 수립.

### 📢 기술사적 결언
> "잭맨 프레임워크는 EA의 **'주기율표'**다. 주기율표가 원소를 분류하듯, 잭맨은 기업의 유무형 자산을 분류한다. 아키텍트는 36칸을 모두 채우는 완벽주의에 빠지기보다, 현재 우리 조직에 가장 시급한 **'결정적 칸(Critical Cells)'**이 어디인지 파악하는 통찰력이 필요하다. 도구에 의존하기 전에 '전체 관점'을 잃지 않는 설계자의 눈을 기르는 것이 잭맨이 주는 진정한 교훈이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **정보 누락 방지**: 표준 매트릭스 준수를 통해 설계 누락률 40% 이상 감소.
- **의사소통 효율**: 계층별 맞춤형 뷰(View) 제공으로 의사결정 속도 향상.

### 미래 전망
미래의 잭맨 프레임워크는 **'AI 기반 자동 매핑'** 기술과 결합할 것입니다. 수천 개의 시스템 문서와 소스코드를 AI가 읽고, 36개의 셀 중 어디에 해당하는지 자동으로 분류하여 기업의 '아키텍처 가시성'을 실시간으로 유지해 주는 시스템이 보편화될 것입니다. 또한 클라우드 네이티브와 디지털 트윈 환경을 수용하기 위해 'Virtual Layer'가 추가된 확장형 잭맨 모델이 등장할 것입니다.
