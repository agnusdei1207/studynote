# 704. 피처 플래그 런타임 기능 토글

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 새로운 기능을 배포한 후에도 코드 수정 없이 **런타임에 동적으로 기능을 켜거나 끌 수 있게(Toggle)** 하는 조건부 제어 기술이다.
> 2. **배포와 출시의 분리**: 코드를 실제 서버에 배포하는 행위(Deployment)와 사용자에게 노출하는 행위(Release)를 분리하여, 배포 리스크를 줄이고 점진적 기능 노출을 가능케 한다.
> 3. **가치**: 운영 환경에서의 안전한 테스트(Testing in Production), 특정 사용자군 대상 카나리 배포, 장애 발생 시 즉각적인 기능 차단(Kill Switch)을 통해 시스템의 안정성과 비즈니스 민첩성을 확보한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "배포는 곧 장애다?"

전통적인 방식에서는 기능을 출시하기 위해 배포가 필수였습니다. 하지만 대규모 배포는 늘 장애의 위험이 따릅니다. "코드는 이미 서버에 가 있지만, 아직 아무도 못 보게 할 수는 없을까?"라는 질문에서 **피처 플래그 (Feature Flag)**가 탄생했습니다. 이는 배포의 단위를 쪼개는 기술이 아니라, 출시의 권한을 '개발'에서 '운영/기획'으로 옮기는 패러다임의 전환입니다.

### 💡 비유: 신축 건물의 조명 스위치

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        피처 플래그(Feature Flag) 비유                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 백화점에 새로운 화려한 전광판(신규 기능)을 설치함.                      │
│                                                                             │
│  1. 기존 방식 (공사 직후 점등):                                               │
│     전광판을 달자마자 불이 켜짐. 만약 전기 배선이 잘못되어 불이 나면(장애)        │
│     전광판을 다시 떼어내야 함(Rollback).                                       │
│                                                                             │
│  2. 피처 플래그 방식 (스위치 설치):                                           │
│     전광판 설치는 미리 끝내둠(Deployment). 하지만 전기는 꺼둠.                   │
│     손님들이 가장 많은 시간에 관리자가 스위치만 딸깍! 올림(Release).              │
│     - 문제가 생기면? 전광판을 뗄 필요 없이 스위치만 다시 내리면 끝(Off)!          │
│                                                                             │
│  → 배포는 미리 해두고, **보여주는 시점**은 내 마음대로 결정하는 마법의 스위치!      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 피처 플래그의 4가지 유형 (Martin Fowler)

| 유형 | 설명 | 수명 (Life span) |
|:---:|:---|:---:|
| **Release Toggles** | 미완성 기능을 감추고 점진적으로 배포할 때 사용 | 단기 |
| **Experiment Toggles** | A/B 테스팅처럼 유저 반응을 살필 때 사용 | 단기 |
| **Ops Toggles** | 장애 발생 시 특정 기능을 즉시 끄는 킬 스위치(Kill Switch) | 중장기 |
| **Permission Toggles** | 유료 회원이나 VIP에게만 기능을 열어줄 때 사용 | 장기 |

### 2. 작동 메커니즘 (ASCII)

```text
    [ User Request ] ──▶ [ Feature Flag Service ] ◀── [ Config Dashboard ]
           │             (Toggle Logic)              (Admin UI: On/Off)
           │                    │
           ▼                    ▼
    if (isEnabled("New_Payment_Logic", userID)) {
        runNewLogic();  // 런타임에 결정
    } else {
        runOldLogic();
    }
```

### 3. 주요 기술 구성
- **Flag Storage**: 플래그의 On/Off 상태를 저장하는 저장소 (Redis, Config File, LaunchDarkly 등).
- **Evaluation Engine**: 유저 ID, 지역, 기기 환경 등을 고려하여 플래그 활성화 여부를 계산하는 엔진.
- **SDK**: 애플리케이션 코드 내에서 플래그 값을 가져오는 라이브러리.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 롤백(Rollback) vs 피처 플래그(Feature Flag)
- **롤백**: 이전 버전의 실행 파일을 다시 배포하는 과정. 시간(5~30분)이 걸리고 서비스 전체에 영향.
- **피처 플래그**: 설정값만 바꿔서 기능을 끄는 과정. **밀리초(ms) 단위**로 즉시 반영되며 특정 기능만 차단.

### 2. 기술적 시너지: 트렁크 기반 개발 (Trunk-based Development)
피처 플래그는 롱리브드 브랜치(Long-lived Branch)를 없애줍니다. 미완성 코드라도 플래그로 감싸서 메인(Trunk) 브랜치에 매일 합칠 수 있기 때문입니다. 이는 머지 지옥(Merge Hell)을 방지하고 진정한 CI/CD를 가능하게 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 고위험 결제 모듈 엔진 교체
- **상황**: 결제 엔진을 통째로 갈아 끼워야 함. 한 번의 실수가 전사 매출 중단으로 이어짐.
- **결단**: **피처 플래그 기반 점진적 노출**. 
- **판단**: 새 엔진을 배포하되 플래그는 'Off'. 이후 사내 테스트 계정만 'On' 하여 검증. 문제없으면 일반 유저 1% -> 5% -> 50% 순으로 확대.
- **효과**: 배포 후 1% 시점에 발생한 미세한 버그를 발견하여 즉시 스위치 Off. 99%의 유저는 장애를 전혀 인지하지 못함.

### 📢 기술사적 결언
> "피처 플래그는 **'기술적 부채의 위험'**과 **'운영적 안전'** 사이의 줄타기다. 플래그가 많아질수록 코드의 복잡도는 급격히 상승하며, 배포가 끝난 뒤에도 제거되지 않은 플래그는 '좀비 코드'가 되어 시스템을 병들게 한다. 아키텍트는 플래그 도입만큼이나 **'플래그 제거 스케줄'**을 엄격히 관리하여 시스템의 청결도를 유지해야 한다. 플래그는 영구적인 설계가 아닌, 안전한 목적지까지 가기 위한 '임시 가교'임을 잊지 말아야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **평균 복구 시간 (MTTR)**: 장애 시 롤백 대비 99% 단축.
- **배포 빈도**: 리스크 감소로 인해 일일 배포 횟수 3~5배 증가.

### 미래 전망
미래의 피처 플래그는 **'AI 지능형 토글'**로 진화합니다. AI가 실시간 에러 로그와 비즈니스 지표를 감시하다가, 신기능 노출 후 구매 전환율이 떨어지거나 에러율이 0.1%만 올라도 자동으로 스위치를 내리고 개발자에게 원인을 브리핑하는 'Self-Governing Delivery' 체계가 구축될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **피처 플래그**는 집안의 **"전등 스위치"**와 같아요.
2. 새 전구를 달았어도(배포), 깜깜한 밤이 되기 전까지는 스위치를 꺼두었다가 **필요할 때만 켤 수** 있죠.
3. 전구가 펑 하고 터지면 전봇대를 고치러 나갈 필요 없이, 그냥 **집 안에서 스위치만 딸깍** 내리면 안전해진답니다!
