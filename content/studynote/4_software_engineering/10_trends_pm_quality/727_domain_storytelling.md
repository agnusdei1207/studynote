# 727. 도메인 스토리텔링 협업 도구

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도메인 전문가와 개발자가 함께 모여 비즈니스 프로세스를 그림과 이야기로 풀어내며, 소프트웨어 설계의 기반이 되는 지식을 공유하고 모델링하는 **시각적 협업 기법**이다.
> 2. **방법론**: "누가(Actor), 무엇을(Work Object) 가지고, 무엇을 하는가(Activity)"를 중심에 두고, 시간 순서에 따라 시각적인 문장을 나열하여 복잡한 비즈니스 로직을 한눈에 파악한다.
> 3. **가치**: 전문 용어(Jargon)의 장벽을 허물어 의사소통 오류를 줄이고, 도메인 주도 설계(DDD)의 바운디드 컨텍스트를 식별하거나 유스케이스를 정교화하는 핵심 도구로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "말이 안 통하는 전문가들"

기획자는 '비즈니스 언어'로 말하고, 개발자는 '코드 언어'로 생각합니다. 이 간극은 잘못된 요구사항 분석과 재작업의 근본 원인이 됩니다. **도메인 스토리텔링 (Domain Storytelling)**은 글이나 복잡한 UML 대신, 아이콘과 화살표라는 '공통의 시각 언어'를 사용하여 누구나 이해할 수 있는 비즈니스 지도를 그리는 작업입니다.

### 💡 비유: 영화 시나리오 보드

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        도메인 스토리텔링 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 복잡한 판타지 영화의 전투 장면(비즈니스 로직)을 찍으려 함.               │
│                                                                             │
│  1. 텍스트 시나리오 (전통적 명세):                                            │
│     "기사가 칼을 들고 용에게 달려가서 꼬리를 피한 뒤 목을 친다..."                │
│     (사람마다 상상하는 모습이 다름)                                             │
│                                                                             │
│  2. 스토리보드 (도메인 스토리텔링):                                            │
│     - 그림 1: [기사] --(칼)--> [용]                                          │
│     - 그림 2: [용] --(꼬리 공격)--> [기사] (기사가 피함)                        │
│     - 그림 3: [기사] --(공격)--> [용의 목]                                     │
│                                                                             │
│  → 그림과 순서 번호(1, 2, 3)를 통해, 감독(기획)과 배우(개발)가 **똑같은 장면**을    │
│     머릿속에 그리게 만드는 것!                                                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 스토리텔링의 3대 구성 요소 (Pictographic Language)
- **Actors (행위자)**: 사람, 부서, 혹은 외부 시스템 (아이콘으로 표시).
- **Work Objects (작업 객체)**: 문서, 데이터, 물건 등 행위의 대상.
- **Activities (활동)**: 행위자가 작업 객체에 가하는 구체적인 동작 (화살표와 동사).

### 2. 작성 규칙 및 프로세스
1. **Scope 정의**: 어떤 이야기를 나눌지 주제 선정.
2. **Step-by-step**: 비즈니스 흐름을 시간 순서대로 번호를 매겨 기록.
3. **Active Voice**: "누가 무엇을 한다"는 능동태 문장으로 표현.
4. **Validation**: 도메인 전문가에게 "이 그림이 당신의 업무와 일치합니까?"라고 즉시 검증.

### 3. 도메인 스토리 예시 구조 (ASCII)

```text
    (1) 주문서 작성          (2) 재고 확인           (3) 결제 요청
    [고객] ──────────▶ [판매 시스템] ──────────▶ [창고] ──────────▶ [PG사]
             주문서(Object)         재고 정보(Object)       결제 승인(Object)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 이벤트 스토밍 (Event Storming) vs 도메인 스토리텔링

| 항목 | 이벤트 스토밍 | 도메인 스토리텔링 |
|:---:|:---|:---|
| **중심 요소** | **이벤트 (사건 발생)** | **행위자 (사람과 행동)** |
| **적합한 단계** | Big Picture (광범위 탐색) | Process Modeling (상세 업무 분석) |
| **시각화** | 수많은 포스트잇 (다소 무질서) | 일러스트 기반의 문장 (질서 정연) |
| **결과물** | 시스템 설계 단위 도출 | 업무 매뉴얼 및 유스케이스 정의 |

### 2. 기술적 시너지: DDD (Domain-Driven Design)
도메인 스토리텔링 중에 특정 행위자와 물건들이 자주 뭉쳐서 움직이는 구간을 발견할 수 있습니다. 이 구간이 바로 DDD의 **바운디드 컨텍스트(Bounded Context)**를 결정하는 강력한 힌트가 됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 복잡한 물류 배송 시스템 분석
- **상황**: 택배가 입고되어 분류되고 배송되기까지의 로직이 너무 복잡해 개발자들이 이해를 못 함.
- **결단**: **화이트보드 기반 스토리텔링 세션**. 
- **판단**: 현장 반장님(도메인 전문가)을 모시고 "송장이 출력되면 그다음엔 누가 뭘 하죠?"라고 질문하며 그림을 그림. 
- **효과**: "송장이 나오기 전에 분류기 점검이 먼저 수행되어야 한다"는 누락된 비즈니스 규칙 발견. 명세서 100페이지보다 강력한 '한 장의 스토리보드' 완성.

### 📢 기술사적 결언
> "소프트웨어의 결함은 기술력 부족보다 **'업무 이해 부족'**에서 더 많이 발생한다. 도메인 스토리텔링은 개발자를 비즈니스의 세계로 안내하는 가장 친절한 가이드다. 아키텍트는 UML의 엄격한 규칙에 매몰되기보다, 이해관계자들이 **'함께 웃고 떠들며 정답을 찾아가는 과정'** 그 자체에 가치를 두어야 한다. 소통이 투명해지면 아키텍처는 저절로 단순하고 명확해진다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **요구사항 정제**: 초기 분석 단계에서의 오해로 인한 재작업 비용 50% 이상 감소.
- **지식 자산화**: 신규 입사자 교육 시간을 획기적으로 단축하는 시각적 자산 확보.

### 미래 전망
미래의 도메인 스토리텔링은 **'디지털 협업 캔버스'**와 **'AI 모델러'**의 결합으로 진화할 것입니다. 전문가들이 대화하는 내용을 AI가 실시간 리스닝하여 자동으로 스토리텔링 다이어그램을 그려주고, 이를 즉시 유스케이스 명세서나 코드 스텁(Stub)으로 변환해 주는 '지능형 지식 공학' 환경이 보편화될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **도메인 스토리텔링**은 복잡한 만화책을 만들기 전에, 작가와 그림 작가가 만나서 **"주인공이 어떻게 움직일지"** 미리 그려보는 거예요.
2. "철수가 영희에게 공을 던진다"라는 쉬운 그림을 그리면서, **서로 딴소리를 하지 않게** 마음을 맞추는 거죠.
3. 그림을 보면서 이야기하니까, 글자만 읽을 때보다 **내용이 훨씬 더 쏙쏙** 이해된답니다!
