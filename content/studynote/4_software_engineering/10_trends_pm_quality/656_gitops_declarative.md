# 656. GitOps 인프라 선언적 관리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 애플리케이션의 소스코드뿐만 아니라 인프라 및 설정 정보 전체를 Git 저장소에 저장하고, 이를 **단일 진실 공급원 (Single Source of Truth)**으로 삼아 운영하는 방식이다.
> 2. **메커니즘**: 원하는 상태(Desired State)를 Git에 선언하면, 자동화 도구가 실제 환경(Actual State)과 비교하여 항상 동일하게 맞추는 **'Pull 방식의 자동 동기화'**를 핵심으로 한다.
> 3. **가치**: 인프라 변경 이력을 완벽히 추적(Audit)할 수 있으며, 장애 발생 시 Git의 이전 커밋으로 되돌리는 것만으로 즉각적인 복구가 가능해져 운영 안정성을 극대화한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "인프라도 코드처럼 관리하자"

클라우드 네이티브 환경이 복잡해지면서 서버 설정을 수동으로 하거나 복잡한 스크립트로 관리하는 것이 불가능해졌습니다. Weaveworks에서 처음 제안한 **GitOps**는 "Git을 운영의 도구로 사용하자"는 아이디어에서 시작되었습니다. 이는 인프라를 하나의 '상태'로 정의하고, 그 상태를 Git에 커밋하는 행위 자체가 배포가 되는 패러다임의 전환입니다.

### 💡 비유: 레스토랑의 레시피북과 주방장

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                          GitOps 비유                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 기존 방식 (명령형):                                                      │
│     주인이 주방장에게 가서 "가스레인지 켜고, 냄비 올리고, 물 500ml 부으세요"      │
│     라고 하나하나 명령함. 주방장이 한 단계라도 실수하면 음식이 망함.              │
│                                                                             │
│  2. GitOps 방식 (선언적):                                                    │
│     주인이 '레시피북(Git)'에 "라면 1인분이 완성된 상태"라고 딱 써둠.              │
│     똑똑한 로봇 주방장(ArgoCD)은 레시피북을 보고, 지금 주방에 라면이 없으면       │
│     알아서 재료를 준비해 라면 1인분을 뚝딱 만들어냄.                             │
│                                                                             │
│  → 우리는 '결과'만 Git에 적고, '과정'은 기계가 책임지는 것!                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. GitOps의 4대 원칙
1. **선언적(Declarative)**: 시스템의 상태를 명령어가 아닌 '선언문(YAML 등)'으로 정의.
2. **버전 관리(Versioned)**: 원하는 상태를 Git에 저장하여 불변성 및 이력 확보.
3. **자동 당김(Pulled Automatically)**: 승인된 변경사항을 도구가 자동으로 감지하여 반영.
4. **지속적 조정(Continuously Reconciled)**: 실제 상태와 선언된 상태 사이의 차이(Drift)를 실시간으로 교정.

### 2. Push 방식 vs Pull 방식 배포

| 항목 | Push 방식 (CI/CD 중심) | Pull 방식 (GitOps 중심) |
|:---:|:---|:---|
| **동작** | Jenkins/GitLab이 서버에 직접 접속하여 실행 | 클러스터 내부의 에이전트가 Git을 감시 |
| **보안** | 배포 도구가 운영 서버의 권한(Token)을 가짐 | 운영 서버가 외부를 바라봄 (보안 유리) |
| **상태 관리** | 일회성 실행 (배포 후 상태 보장 불가) | **실시간 동기화** (수동 변경 시 자동 원복) |
| **대표 도구** | Jenkins, GitHub Actions | **ArgoCD, Flux** |

### 3. GitOps 아키텍처 (ASCII)

```text
    [ Developer ] ── 커밋 ──▶ [ Git Repository ] ◀── 감시 ──┐
                                (Desired State)           │
                                                          │
    ┌─────────────────────────────────────────────────────┘
    │  [ GitOps Controller (ArgoCD) ]
    │         │
    │         ▼ [ Sync / Reconcile ]
    │
    └────▶ [ Kubernetes Cluster ] (Actual State)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. IaC (Infrastructure as Code) vs GitOps
- **IaC**: 인프라를 코드로 작성하는 **기술** (Terraform, CloudFormation).
- **GitOps**: IaC를 Git을 통해 운영하는 **방법론**. (GitOps는 IaC를 전제로 함)

### 2. 기술적 시너지: 쿠버네티스 (K8s)
쿠버네티스는 자체적으로 '원하는 상태'를 유지하려는 컨트롤러 엔진을 가지고 있어, GitOps를 구현하기에 가장 완벽한 플랫폼입니다. YAML 파일이 곧 GitOps의 선언문이 됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 재해 복구(DR) 상황 대응
- **상황**: 실수로 운영 중인 쿠버네티스 네임스페이스 전체가 삭제됨.
- **결단**: **GitOps 복구**. 별도의 백업본을 찾을 필요 없이, ArgoCD의 'Sync' 버튼을 누르거나 새 클러스터에 Git 저장소 주소만 연결.
- **판단**: Git에 정의된 모든 리소스(Pod, Service, Ingress 등)가 수 분 내에 그대로 재현됨. 인적 오류로 인한 복구 지연을 원천 차단.

### 📢 기술사적 결언
> "GitOps는 운영의 주도권을 '사람의 손'에서 **'Git 저장소의 커밋'**으로 옮긴 혁명이다. 이는 단순히 배포를 편하게 하는 것을 넘어, 운영의 모든 행위를 투명하게 기록하고 검증 가능하게 만든다. 특히 다수의 클러스터를 운영해야 하는 엔터프라이즈 환경에서 GitOps는 선택이 아닌 **필수적인 거버넌스 체계**다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **배포 속도**: 수동 작업 대비 5배 이상 빠른 배포 및 롤백.
- **보안성 강화**: 운영 서버에 직접 접속하는 인원을 최소화하여 보안 사고 예방.

### 미래 전망
최근에는 GitOps가 애플리케이션을 넘어 클라우드 리소스 전체(DB, 스토리지 등)로 확장되는 **'Crossplane'**과 같은 기술과 결합하고 있습니다. 또한 기업의 컴플라이언스(Compliance) 정책까지 Git에 선언하여 위반 시 자동 교정하는 **'Policy as Code'**와 융합되며 진정한 'Self-healing Infrastructure'를 향해 진화하고 있습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **GitOps**는 요술 일기장(Git)에 **"내 방이 깨끗하게 청소된 모습"**을 그려 넣는 것과 같아요.
2. 그러면 청소 요정(ArgoCD)이 일기장을 보고, 내 방이 조금이라도 어질러져 있으면 **일기장 그림이랑 똑같이** 다시 치워준답니다.
3. 내가 직접 청소할 필요 없이, 일기장만 잘 써두면 방은 **항상 깨끗한 상태**로 유지되는 마법 같은 방법이에요!
