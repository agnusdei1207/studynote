# 761. 워크스루 비공식 기술 검토 회의

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 작성자가 산출물을 동료에게 직접 설명하는 방식을 통해, 공식적인 절차보다는 **상호 학습과 논리적 검증**에 중점을 둔 가볍고 유연한 정적 테스트 기법이다.
> 2. **가치**: 결함 조기 발견을 넘어 **집단 지성(Collective Intelligence)**을 활용한 기술 상향 평준화와 버스 지수(Bus Factor) 향상에 기여하며, 준비 비용 대비 높은 투자 대비 효과(ROI)를 제공한다.
> 3. **융합**: 애자일(Agile)의 페어 프로그래밍(Pair Programming) 및 현대적인 PR(Pull Request) 리뷰 문화의 근간이 되며, 빠르게 변하는 요구사항 대응을 위한 협업 필수 요소다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 정의**
**워크스루 (Walkthrough)**는 소프트웨어 개발 생명주기(SDLC)의 초기 단계나 구현 단계에서, 산출물(요구사항 명세서, 설계서, 코드 등)을 작성한 개발자가 동료 개발자나 관련 전문가들을 모아놓고 내용을 직접 설명하는 기법이다. 이는 엄격한 규칙이나 체크리스트를 따르는 인스펙션(Inspection)과 달리, 미리 준비된 시나리오(Scenario)에 따라 시스템의 동작 흐름을 '걸어서 through' 검토한다는 점이 특징이다. 주로 논리적 오류 발견, 기술적 이해도 공유, 교육(Education) 목적으로 활용된다.

**2. 등장 배경 및 필요성**
소프트웨어 복잡도가 급증하면서 개별 개발자의 인지적 한계로 인한 결함이 발생하기 쉬워졌다. 단순히 컴파일러가 문법 오류를 잡아주는 것을 넘어, 비즈니스 로직(Business Logic)의 모순이나 아키텍처적 설계 결함을 찾아내기 위해서는 타인의 시선이 필수적이다. 그러나 모든 코드를 격식 있는 인스펙션으로 검토하기에는 시간과 비용이 과다하게 소요된다. 이에 따라 '가볍고 자주' 수행할 수 있는 워크스루가 등장하였으며, 이는 폭포수(Waterfall) 모델의 무거운 리뷰 문화를 보완하고 애자일 환경의 빠른 피드백 문화를 선도하는 기반이 되었다.

**💡 핵심 비유: 영화 시나리오 리딩 (Table Read)**
> 영화 제작 전 배우와 스태프들이 모여 각본을 소리 내어 읽으며 자연스러운지 확인하는 과정과 같다. 감독(작성자)이 의도를 설명하고, 배우(동료)들이 "이 대사는 좀 어색해요"라고 피드백하며 자연스럽게 내용을 다듬어 가는 과정이다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        🔍 워크스루(Walkthrough) 개념도                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [작성자]           [참여자: Peer Reviewers]          [결과물]              │
│                                                                             │
│   👨‍💻 자신이 작성한                                    🧠 공통의 이해          │
│      코드/문서     ───────────────────────▶   (Shared Context)            │
│         │                 + Question & Discussion                           │
│         ▼                                                           ▲      │
│   🗣️ "이 부분은                                                  │      │
│      데이터가 들어오면                                            │      │
│      A -> B -> C 로                                             │      │
│      흐르도록 짰습니다."                                          │      │
│                                                                             │
│      ⇩                                                                │      │
│                                                                             │
│   👂 동료들의 반응:                                                      │      │
│      - "B에서 C로 갈 때 예외 처리가 안 된 것 같은데요?" (Defect Detection)  │      │
│      - "아, 저 부분은 제가 경험해 봤는데 비용이 비싸요." (Knowledge Share) │      │
│      - "좋은 아이디어인데 다른 방식도 있지 않을까요?" (Creative Solving)    │      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
> **도해 해설**: 작성자는 주도적인 발표자가 되어 로직의 흐름을 제시(Lead)하고, 참여자들은 수동적인 청자가 아니라 능동적인 토론자(Discussant)로 참여하여 실시간으로 의견을 조율한다. 이 과정에서 문서에 없는 암묵지(Tacit Knowledge)가 형성된다.

**📢 섹션 요약 비유**
마치 복잡한 길을 찾을 때 내비게이션만 보고 혼자 가는 것(단순 코딩)보다, 현지 친구를 옆에 세워두고 "이 길이 맞나요?"라고 확인하며 가는 것과 같습니다. 친구가 모르는 길을 알려줄 수도 있고(지식 공유), 내가 잘못 가고 있음을 바로잡아 줄 수도 있기(결함 발견) 때문입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. 구성 요소 및 상세 동작**

| 요소명 | 역할 | 상세 동작 메커니즘 | 주요 프로토콜/산출물 |
|:---:|:---|:---|:---|
| **작성자 (Author)** | **주도자 (Presenter)** | 자신이 작성한 산출물의 논리적 흐름을 설명한다. 방어적 태도보다 개방적 태도로 피드백을 수용한다. | 시나리오 목록, 발표 자료 |
| **중재자 (Moderator)** | **진행자 (Facilitator)** | (선택 사항) 논의가 흐르지 않도록 주제를 통제하고, 시간을 관리하며 분위기를 조성한다. | 회의록 (Meeting Minutes) |
| **참여자 (Reviewers)** | **검토자 (Inspectors)** | 개발자, 테스터, 설계자 등 관련 파트 스태프. 시나리오를 따라가며 논리적 모순, 표준 위반, 성능 이슈 등을 질문한다. | 체크리스트 (비공식) |
| **기록자 (Scribe)** | **비서 (Recorder)** | 논의 내용 중 중요한 결함(Action Item)이나 결정 사항을 메모한다. (보통 참여자가 겸임) | 이슈 로그 (Issue Log) |
| **산출물 (Artifact)** | **검토 대상** | 코드, 설계서, 테스트 케이스 등. 완성된 형태보다는 초안(Draft) 상태에서도 수행 가능하다. | SRS, SDD, Source Code |

**2. 심층 동작 원리 (The Mechanism)**
워크스루는 '정적 분석(Static Analysis)'의 일종으로, 프로그램을 실행시키지 않고 논리의 흐름을 추적(Mental Simulation)한다. 이 과정에서 인간의 두뇌가 수행하는 '실행 추적(Execution Tracing)'을 통해, 작성자가 놓쳤던 엣지 케이스(Edge Case)나 예외 상황(Exception Handling)을 발견한다. 특히, 워크스루는 코드의 '의도'를 공유하는 과정이므로, 향후 유지보수 시 코드를 이해하는 시간(Cognitive Load)을 획기적으로 단축시킨다.

**3. 워크스루 수행 절차 (Process Flow)**

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                     🔄 워크스루 표준 수행 프로세스                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ① Planning (계획)                                                         │
│     └─ "이 튜플 로직 검증 필요함. 내일 오후 2시에 잡자."                       │
│           │                                                                │
│           ▼                                                                │
│  ② Distribution (자료 배포)                                                │
│     └─ [팀 전체] 대상 코드/설계서 공유 (사전 독려는 약하게)                      │
│           │                                                                │
│           ▼                                                                │
│  ③ Walkthrough Meeting (핵심 회의) ──────────────────────┐                  │
│     │                                                     │                  │
│     ├─ STEP 1: 작성자 개요 설명 (Overview)                 │                  │
│     ├─ STEP 2: 시나리오 기반 스텝별 설명                   │                  │
│     │          (Ex. "사용자가 로그인을 누르면...")           │                  │
│     │                                                     │                  │
│     ├─ STEP 3: 동료 질의 및 응답 (Q&A)                     │                  │
│     │          (Ref. 이 부분은 Multi-threading 시          │                  │
│     │                Race Condition 발생 소지 있음)       │                  │
│     │                                                     │                  │
│     └─ STEP 4: 이슈 기록 (Note Taking)                    │                  │
│           │                                                │                  │
│           ▼                                                │                  │
│  ④ Rework (재작업)                                         │                  │
│     └─ [작성자] 발견된 오류 수정 및 개선안 반영               │                  │
│           │                                                │                  │
│           ▼                                                │                  │
│  ⑤ Follow-up (사후 확인)                                   │                  │
│     └─ (선택적) 수정 내용 간단 공지                         │                  │
│                                                             │                  │
└─────────────────────────────────────────────────────────────────────────────┘
```
> **도해 해설**: 일반적인 인스펙션과 달리 '사전 준비(Preparation)' 단계가 생략되거나 매우 가볍다. 회의 중심(③)이며, 이 회의는 '비난'이 아닌 '학습'의 장이다. ③ 단계의 STEP 2에서 작성자는 흐름을 제어하며, 동료들은 흐름 속에서 빠진 구멍을 찾는다.

**4. 핵심 알고리즘 및 코드 스니펫**
워크스루는 소프트웨어적 알고리즘보다는 **소통 프로토콜(Communication Protocol)**에 가깝다. 하지만 검토 대상이 코드일 때, 다음과 같은 '수학적 논리 검증'이 발생한다.

```python
# [시나리오] 계좌 이체 시스템의 잔액 차감 로직 검토

def transfer_balance(from_account, to_account, amount):
    # 🗣️ [작성자 설명]: "보내는 사람 잔액을 확인하고 차감합니다."
    if from_account.balance >= amount:
        # ⚠️ [동료 질문]: "잠금(Lock)은 어떻게 하나요? A가 송금하고 B가 동시에 인출하면?"
        # (이 시점에서 동시성 제어 문제 발견)
        from_account.balance -= amount
        to_account.balance += amount
        return True
    else:
        return False

# [결과] 작성자는 Lock 매커니즘(예: SELECT ... FOR UPDATE) 추가를 액션 아이템으로 등록함.
```

**5. 워크스루 vs 인스펙션 (심층 비교 다이어그램)**

```text
    [ Inspection (엄격한 검사) ]              [ Walkthrough (흐름 중심) ]
    
    🔴 FORMALITY (공식성)                     🟢 FLEXIBILITY (유연성)
    │                                         │
    ▼                                         ▼
[ Role Specific ]                       [ Role Flexible ]
Author은 발표 금지, Moderator 주도      Author가 발표 주도, 누구나 질문
    │                                         │
    ▼                                         ▼
[ Checklist Base ]                      [ Scenario Base ]
항목별 준수 여부 체크 (O/X)              흐름에 따른 자연스러운 발견
    │                                         │
    ▼                                         ▼
[ Output: Defect Log ]                  [ Output: Shared Model ]
결함 통계 및 메트릭스 보고                로직 이해 및 기술 전파
```

**📢 섹션 요약 비유**
자동차의 엔진을 다 조립한 후, 정비공이 체크리스트를 보며 볼트가 다 조여졌는지 일일이 점검하는 것(인스펙션)과 달리, 워크스루는 조립 중인 엔진을 시동을 걸어보지 않고 손으로 크랭크를 돌려보며 "어디서 걸리는지", "소리가 어떤지"를 팀원들이 함께 들어보며 기계의 성격을 파악하는 과정과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 인스펙션 vs 워크스루 vs 동료 검토 (Technical Review Matrix)**

| 비교 항목 | 인스펙션 (Inspection) | 워크스루 (Walkthrough) | 동료 검토 (Peer Review/Desk Check) |
|:---:|:---:|:---:|:---:|
| **주도자** | 중재자 (Moderator) | **작성자 (Author)** | 작성자 본인 |
| **준비 시간** | 높음 (High Overhead) | **낮음 (Low Cost)** | 매우 낮음 |
| **공식성** | 엄격함 (Formal) | **반공식 (Semi-formal)** | 비공식 (Informal) |
| **참여자 규모** | 3~8명 (정규직 위원) | **2~5명 (관련자)** | 1~2명 (Pair) |
| **주요 목표** | 결함 제거 (Defect Removal) | **학습 및 합의 (Consensus)** | 단순 오류 수정 |
| **문서화** | 상세한 로그 및 메트릭스 | **간단한 회의록** | 없음 또는 주석 |
| **RTO (복구 시간)** | 금지 (코드 수정 후 재검토) | 짧음 (즉시 수정 가능) | 즉시 |

**2. 기술 스택 융합 분석 (Cross-Domain Analysis)**

*   **⑴ SW 엔지니어링 + 인지 심리학 (Cognitive Psychology)**
    *   워크스루는 '생각의 소리 내어 말하기(Think-Aloud Protocol)' 기법을 활용한다. 작성자가 코드를 읽을 때 내면의 사고 과정을 밖으로 드러냄으로써, 숨어있는 오개념(Misconception)을 드러낸다. 이는 TDD(Test-Driven Development)의 'Red-Green-Refactor' 사이클이 코드를 통해 생각을 검증하는 것과 보완적인 관계에 있다.

*   **⑵ 프로젝트 관리 (PM) + 리스크 관리 (Risk Management
