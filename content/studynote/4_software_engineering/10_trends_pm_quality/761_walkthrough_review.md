# 761. 워크스루 비공식 기술 검토 회의

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 개발자가 자신의 산출물(코드, 설계서 등)을 동료들에게 설명하며 논리적 오류를 찾고 지식을 공유하는 **개발자 중심의 비공식 기술 검토 기법**이다.
> 2. **방법론**: 엄격한 절차나 체크리스트보다는 시나리오를 바탕으로 시스템의 흐름을 따라가는(Walk-through) 방식이며, 주로 교육, 로직의 타당성 확인, 팀원 간의 이해 일치를 목적으로 한다.
> 3. **가치**: 인스펙션 대비 준비 부담이 적어 수시로 수행 가능하며, 결함 발견뿐만 아니라 팀의 기술 상향 평준화와 '집단 지성'을 통한 창의적 해결책 도출에 기여한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "내 코드를 동료의 눈으로 읽다"

복잡한 알고리즘을 짰을 때, 혼자서는 보지 못하는 허점이 반드시 존재합니다. 그렇다고 매번 무거운 인스펙션을 하기엔 일정이 부족합니다. **워크스루 (Walkthrough)**는 "내가 짠 걸 한번 들어봐 줘"라는 가벼운 요청에서 시작하여, 팀원들과 함께 로직의 숲을 산책하듯 훑어보는 품질 활동입니다.

### 💡 비유: 영화 시나리오 낭독회

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        워크스루(Walkthrough) 비유                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 영화 감독이 시나리오를 다 썼음.                                         │
│                                                                             │
│  1. 인스펙션 (사후 검열): 법률 전문가, 역사학자가 모여 "이 장면은 법에 걸림",     │
│     "이 대사는 고증 오류임"이라고 딱딱하게 지적함. (엄격한 검사)                 │
│                                                                             │
│  2. 워크스루 (낭독회): 감독이 배우와 스태프를 모아놓고 시나리오를 직접 읽어줌.    │
│     - "이 장면에서 주인공이 이렇게 움직이면 이상하지 않을까요?" (로직 확인)       │
│     - "아! 저 부분은 제가 대신 연기할게요." (지식 공유/협업)                      │
│                                                                             │
│  → 작성자가 주인공이 되어 **직접 설명하고 함께 정답을 찾아가는** 따뜻한 회의!     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 워크스루의 주요 특징

| 항목 | 상세 내용 | 비고 |
|:---:|:---|:---|
| **주관자** | **작성자 (Author)** 가 직접 주관 | 인스펙션은 중재자가 주관 |
| **참석자** | 관련 개발자, 설계자, 테스터 (보통 2~5명) | 소규모 집중 논의 |
| **형식** | 비정형, 유연한 회의 방식 | 시나리오 기반 시뮬레이션 |
| **핵심 목적** | 로직 이해, 교육, 잠재적 결함 조기 식별 | 지식 전파 효과 큼 |

### 2. 워크스루 수행 절차
1. **준비**: 작성자가 검토 대상(코드, 설계서)과 시나리오 배포.
2. **발표**: 작성자가 산출물의 논리적 흐름을 한 단계씩 설명.
3. **토론**: 참석자들이 질문하고, 대안을 제시하거나 오류 가능성 제기.
4. **기록**: 발견된 문제점이나 개선 아이디어를 메모 (비공식적).
5. **반영**: 작성자가 회의 결과를 바탕으로 산출물 보완.

### 3. 검토 메커니즘 비교 (ASCII)

```text
    [ Inspection: 절차 중심 ]             [ Walkthrough: 흐름 중심 ]
    
    Rule Book ────────┐                  Scenario ─────────┐
           │          ▼                         │          ▼
    Checkpoint ──▶ [ Verify ]            Step-by-Step ──▶ [ Discuss ]
           │          ▲                         │          ▲
    Data Log  ────────┘                  Logic Flow ───────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 인스펙션 vs 워크스루 vs 동료 검토

| 구분 | 인스펙션 (Inspection) | 워크스루 (Walkthrough) | 동료 검토 (Peer Review) |
|:---:|:---|:---|:---|
| **공식성** | **최고 (정형)** | 중간 (비정형) | 최저 (비공식) |
| **목표** | 결함 발견, 통계 | **로직 이해, 지식 공유** | 가벼운 오류 수정 |
| **비용** | 높음 (사전 준비 필수) | **보통** | 낮음 |
| **효과** | 품질 보증(QA) | **팀 역량 강화** | 즉각적 피드백 |

### 2. 기술적 시너지: 코드 리뷰 문화
현대의 GitHub Pull Request(PR) 기반 리뷰는 비동기식 워크스루의 변형입니다. 하지만 아주 복잡한 아키텍처 변경이나 신규 도메인 도입 시에는, 화면을 띄워놓고 실시간으로 대화하는 **실시간 워크스루**가 PR의 텍스트 리뷰보다 수십 배 빠른 의사결정을 지원합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 신입 사원 온보딩 및 로직 검토
- **상황**: 주니어 개발자가 맡은 핵심 모듈의 첫 번째 배포를 앞두고 있음.
- **결단**: **세미나 형식의 워크스루 세션 개최**. 
- **판단**: 인스펙션으로 혼내기보다, 주니어가 직접 로직을 설명하게 하여 자신감을 북돋우고 선배들이 노하우를 전수하는 기회로 활용.
- **효과**: "예외 상황 처리가 빠졌다"는 기술적 피드백과 함께, 해당 모듈의 지식이 팀 전체로 전파되어 '버스 지수(Bus Factor)' 향상.

### 📢 기술사적 결언
> "워크스루는 **'기술적 대화'**의 장이다. 정형화된 인스펙션이 제품의 결함을 솎아내는 작업이라면, 워크스루는 그 제품을 만드는 **'사람들의 생각'**을 일치시키는 작업이다. 아키텍트는 프로젝트 초기나 기술적 난관에 부딪혔을 때 워크스루를 적극 활용하여, 팀이 하나의 유기체처럼 움직이도록 윤활유 역할을 수행해야 한다. 문서보다 강력한 것은 **'공유된 컨텍스트'**임을 잊지 말아야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적/정성적 기대효과
- **팀 응집도**: 소통 활성화로 인한 조직 문화 개선 및 협업 시너지 창출.
- **결함 예방**: 초기 설계 사상의 공유를 통해 구현 단계의 삽질 방지.

### 미래 전망
미래의 워크스루는 **'디지털 화이트보드'**와 **'AI 요약'** 기술이 결합될 것입니다. 원격지에 떨어진 팀원들이 VR 공간에 모여 3D 아키텍처 모델을 함께 거닐며 로직을 검토하고, AI가 대화 내용을 분석하여 실시간으로 '결정 사항'과 'ToDo 리스트'를 뽑아주는 스마트 워크스루 환경이 정착될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **워크스루**는 내가 만든 장난감을 친구들에게 보여주며 **"이건 이렇게 움직이는 거야"**라고 자랑하며 설명해주는 거예요.
2. 설명을 듣던 친구들이 **"어? 저기 바퀴가 헐렁한 것 같아!"**라고 알려주면 고맙게 생각하고 고치는 거죠.
3. 혼자 놀 때보다 친구들이랑 같이 장난감을 연구하니까, **훨씬 더 튼튼하고 멋진 장난감**을 만들 수 있답니다!
