# 650. CI/CD 지속적 통합, 배포 파이프라인

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 코드 변경사항을 자동으로 빌드, 테스트하고 운영 환경까지 안정적으로 전달하기 위해 구축된 **자동화된 소프트웨어 인도 체계**이다.
> 2. **구성**: 개발자의 빈번한 코드 병합과 검증을 의미하는 **지속적 통합(CI)**, 그리고 테스트를 통과한 코드를 실운영 또는 스테이징에 자동 반영하는 **지속적 제공/배포(CD)**로 나뉜다.
> 3. **가치**: 배포 주기(Lead Time)를 주 단위에서 분 단위로 단축하고, 수동 작업으로 인한 인적 오류를 제거하여 비즈니스 민첩성과 품질을 동시에 확보한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "배포는 이벤트가 아니라 일상이다"

전통적인 방식에서는 수개월간 개발한 코드를 한꺼번에 통합하고 배포했습니다. 이 과정에서 '통합 지옥(Integration Hell)'과 대규모 장애가 빈번했습니다. **CI/CD**는 코드를 조금씩, 자주 통합하고 자동으로 검증하여 배포의 리스크를 낮추는 현대 소프트웨어 공학의 정수입니다.

### 💡 비유: 수돗물 공급 시스템

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                          CI/CD 파이프라인 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. CI (지속적 통합):                                                        │
│     강물(코드)이 정수장으로 들어올 때마다 필터(자동 테스트)를 거쳐 깨끗한        │
│     물(빌드 산출물)로 만드는 과정. 오염된 물이 들어오면 즉시 차단함.             │
│                                                                             │
│  2. CD (지속적 배포):                                                        │
│     정수된 물을 파이프라인을 통해 각 가정(운영 환경)의 수도꼭지까지 자동으로      │
│     보내는 과정. 사람이 직접 물통을 배달할 필요가 없음.                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. CI/CD의 단계별 상세 정의

| 단계 | 명칭 | 핵심 활동 | 기술적 산출물 |
|:---:|:---|:---|:---|
| **CI** | **Continuous Integration** | 소스 병합, 정적 분석, 단위 테스트 수행 | Build Artifact (Jar, Docker Image 등) |
| **CD** | **Continuous Delivery** | 스테이징 환경 배포, 인수 테스트, 승인 대기 | 릴리즈 가능 상태의 제품 |
| **CD** | **Continuous Deployment** | **승인 절차 없이** 실운영 환경에 자동 배포 | 실제 고객 서비스 반영 |

### 2. CI/CD 파이프라인 아키텍처 (ASCII)

```text
    [ Code ] ──▶ [ Build ] ──▶ [ Test ] ──▶ [ Release ] ──▶ [ Deploy ]
       │           (Compile)     (Unit/ST)     (Artifact)     (Prod/Stage)
       │               ▲             │             │              │
       └────── CI ─────┘             └───── CD (Delivery) ────────┘
       │                                                          │
       └─────────────────────── CD (Deployment) ──────────────────┘
```

### 3. 파이프라인의 3대 핵심 요소
- **자동화 (Automation)**: 모든 단계에 인간의 개입을 최소화 (Scripted Pipeline).
- **피드백 (Feedback)**: 실패 시 개발자에게 즉시 알림 (Slack, Email 등).
- **멱등성 (Idempotency)**: 동일한 파이프라인 실행은 언제나 동일한 결과를 보장.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 지속적 제공(Delivery) vs 지속적 배포(Deployment)
- **제공(Delivery)**: 배포 직전 단계까지 자동화하되, 실제 실운영 반영은 **수동 승인**이 필요함 (금융, 공공 등 보수적 도메인).
- **배포(Deployment)**: 테스트 통과 시 즉시 **운영 환경까지 자동**으로 나감 (B2C 웹, SaaS 기업).

### 2. 기술적 시너지: IaC (Infrastructure as Code)
애플리케이션 코드뿐만 아니라, 서버 환경(Terraform, Ansible)도 파이프라인에 포함하여 배포함으로써 환경 불일치(Configuration Drift) 문제를 원천 차단합니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 배포 공포증(Release Phobia) 극복
- **상황**: 배포할 때마다 장애가 날까 봐 금요일에는 절대 배포하지 않는 문화.
- **결단**: 
    1. Jenkins/GitHub Actions를 이용한 **CI 강화**: 테스트 커버리지 80% 미만 시 빌드 실패 처리.
    2. **Blue-Green Deployment** 연계: 신규 버전을 미리 띄워보고 트래픽을 전환하는 안전장치 마련.
- **판단**: "작은 배포를 자주 하는 것이 큰 배포를 가끔 하는 것보다 훨씬 안전하다"는 데이터 기반의 신뢰 구축.

### 📢 기술사적 결언
> "CI/CD는 도구(Jenkins, GitLab)의 문제가 아니라 **문화의 문제**다. 테스트 코드를 짜지 않는 CI/CD는 '쓰레기를 더 빨리 배포하는 파이프라인'일 뿐이다. 진정한 파이프라인은 품질이라는 필터가 견고하게 작동할 때 비로소 비즈니스 가치를 창출한다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과 (DORA 메트릭스)
- **Deployment Frequency**: 배포 빈도 수십 배 증가.
- **Lead Time for Changes**: 코드 커밋부터 배포까지 시간 90% 단축.
- **Mean Time to Recover (MTTR)**: 장애 시 롤백 자동화로 복구 시간 단축.

### 미래 전망
미래의 CI/CD는 **'GitOps'**와 **'AIOps'**로 수렴하고 있습니다. Git의 상태가 곧 인프라의 상태가 되는 선언적 배포와, 파이프라인의 성공 여부를 AI가 실시간 모니터링하여 이상 징후를 미리 포착하고 자동 중단하는 '지능형 파이프라인'이 보편화될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **CI/CD**는 자동 장난감 공장과 같아요. 재료(코드)를 넣으면 로봇이 조립(빌드)하고, 튼튼한지 검사(테스트)까지 알아서 다 하죠.
2. 검사를 통과한 장난감은 컨베이어 벨트를 타고 **가게 진열대(운영 환경)까지 슝~** 하고 날아간답니다.
3. 아저씨들이 직접 배달할 필요가 없어서, 친구들은 **매일매일 새로운 장난감**을 만날 수 있어요!
