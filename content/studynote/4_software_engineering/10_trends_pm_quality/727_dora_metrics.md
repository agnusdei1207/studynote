# 727. DORA 메트릭스 4대 지표 (배포 빈도 등)

## # DORA (DevOps Research and Assessment) Metrics

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 구글의 DORA 팀이 10년 이상의 연구를 통해 정립한 **소프트웨어 전달 성과(Software Delivery Performance, SDP)**를 측정하는 글로벌 표준 지표입니다.
> 2. **가치**: 단순한 생산성 측정을 넘어, **배포 빈도(DF)**, **변경 리드 타임(LT)**, **변경 실패율(CFR)**, **서비스 복구 시간(MTTR)**의 4가지 변수를 통해 속도와 안정성의 상관관계를 정량화하여 **Elite 팀**으로의 진화 경로를 제시합니다.
> 3. **융합**: CI/CD (Continuous Integration/Continuous Deployment) 파이프라인의 효율성을 증명하고, SRE (Site Reliability Engineering)의 **에러 버짓(Error Budget)** 관리와 결합하여 비즈니스 임팩트를 극대화하는 의사결정 도구로 활용됩니다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 철학
과거의 소프트웨어 개발 성과 지표는 **Vanadium Metrics(바나나 지표)**라 불리는 허상에 가까웠습니다. 예를 들어, '코드 작성 라인 수(Line of Code)', '생성된 티켓 수', '작업 시간' 등은 개발자가 바쁘다는 것을 보여줄 뿐, 비즈니스에 실제로 가치를 전달했는지(Delivery)는 나타내지 못했습니다.

DORA 팀은 전 세계 31,000명 이상의 전문가를 대상으로 한 대규모 설문을 통해, **기술적 우수성(Technical Excellence)**이 비즈니스 성과(Profitability, Market Share, Productivity)와 어떤 상관관계를 맺는지 분석했습니다. 그 결과, 팀의 성과를 결정짓는 것은 특정 언어나 도구가 아니라, **'얼마나 빨리 그리고 안정적으로 가치를 고객에게 전달하는가'**임을 밝혀냈습니다. 이를 정량화한 것이 DORA 4대 지표입니다.

#### 2. 등장 배경 및 필요성
① **기존 한계 (Waterfall)**: 몇 달간의 개발 끝에 한꺼번에 배포하다 보니, 문제가 발생해도 원인을 찾기 어렵고 복구에 몇 주가 걸렸습니다. (블랙박스 부재)
② **패러다임 변화 (DevOps)**: 애자일(Agile)과 데브옵스(DevOps)의 확산으로 '작은 기능을 자주 출시'하는 방식으로 전환되었습니다. 이에 맞는 새로운 피드백 루프가 필요했습니다.
③ **비즈니스 요구 (Speed vs Stability)**: 경영진은 "빨리 내놓으라(Speed)"고 하면서도 "절대 죽지 말라(Stability)"고 요구합니다. DORA 지표는 이 두 마리 토끼를 모두 잡는 것이 가능함을 데이터로 증명합니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    [고전 지표 vs DORA 지표의 패러다임]                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  📉 고전적 지표 (Activity-based)          📈 DORA 지표 (Outcome-based)      │
│  ─────────────────────               ─────────────────────              │
│  • 코드 라인 수 (LOC)                   • 배포 빈도 (DF)                   │
│  • 생산된 버그 리포트 수                 • 변경 리드 타임 (LT)              │
│  • 야근 시간                             • 변경 실패율 (CFR)               │
│  (개인의 노력 강조)                      • 서비스 복구 시간 (MTTR)          │
│                                         (시스템의 성과 강조)               │
│                                                                             │
│       [Vanity] 헛된 성취의 늪              [Value] 진정한 가치 창출          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 📢 섹션 요약 비유
이는 자동차의 **'계기판(Instrument Cluster)'**을 도입하는 것과 같습니다. 과거에는 "운전을 많이 했나?"(야근 시간)로 성과를 뗏다면, 이제는 "얼마나 빨리 그리고 사고 없이 목적지에 도착했나?"(주행 기록)로 성과를 측정하는 것입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 구성 요소 상세 분석
DORA 지표는 크게 **속도(Throughput)**와 **안정성(Stability)**라는 두 가지 축으로 구성됩니다.

| 구분 | 핵심 지표 | 영문 명칭 (Abbreviation) | 정의 및 측정 기준 | 측정 주체 |
|:---:|:---|:---|:---|:---|
| **속도**<br>(Speed) | **배포 빈도** | **Deployment Frequency (DF)** | 성공적으로 프로덕션(Production) 환경에 배포된 횟수.<br>*(예: 배포/일, 배포/주)* | CI/CD 도구 |
| | **변경 리드 타임** | **Lead Time for Changes (LT)** | 코드가 커밋(Commit)되어 프로덕션에 **성공적으로** 배포될 때까지 걸리는 시간.<br>*(PR 리뷰, 테스트, 승인 시간 포함)* | SCM / VCS |
| **안정성**<br>(Stability) | **변경 실패율** | **Change Failure Rate (CFR)** | 배포(서비스 배포 또는 인프라 변경) 후 실패하여<br>핫픽스(Hotfix), 롤백(Rollback), 패치가 필요한 비율. | Incident Mgmt |
| | **서비스 복구 시간** | **Time to Restore Service (MTTR)** | 서비스 장애(Incident) 발생 시점부터<br>서비스가 정상화되어 고객에게 영향이 없는 상태로 복구될 때까지의 시간. | APM / Monitor |

#### 2. 성숙도 모델: High Performance vs Low Performance
DORA는 연구 결과를 바탕으로 팀을 4단계로 분류합니다. **Elite (엘리트)** 팀은 다른 팀보다 훨씬 높은 생산성과 안정성을 가집니다.

| 등급 | 배포 빈도 (DF) | 변경 리드 타임 (LT) | 변경 실패율 (CFR) | 서비스 복구 시간 (MTTR) |
|:---:|:---:|:---:|:---:|:---:|
| **Elite** | **On-demand**<br>(필요 시 수시) | **< 1 Hour** | **0-15%** | **< 1 Hour** |
| High | **Between once per week and once per month** | **Between one week and one month** | **0-15%** | **< 1 Day** |
| Medium | **Between once per month and once every 6 months** | **Between one month and 6 months** | **16-30%** | **< 1 Week** |
| Low | **Less than once per six months** | **More than 6 months** | **> 30%** | **> 6 Months** |

> **💡 해석**: Elite 팀은 저성과 팀(Low) 대비 배포 빈도는 **수천 배** 높으면서도, 실패율은 **7배** 낮습니다. 이는 **속도와 안정성이 트레이드오프(Trade-off) 관계가 아님**을 증명하는 핵심 데이터입니다.

#### 3. 지표 간 상관관계 및 데이터 흐름 (ASCII)

DORA 지표는 소프트웨어 인도 라이프사이클 전체를 관통합니다. 이를 시각화하면 다음과 같습니다.

```text
┌─────────────────── DATA FLOW & METRICS RELATIONSHIP ────────────────────┐
│                                                                          │
│   [Step 1] Development      [Step 2] Integration     [Step 3] Operation │
│                                                                          │
│    ┌─────────┐                 ┌──────────┐           ┌───────────┐     │
│    │  CODE   │───────────────> │  BUILD   │ ──────>   │ DEPLOY    │     │
│    └─────────┘   Commit       └──────────┘  Release  └───────────┘     │
│       ▲                                 │                              │
│       │                                 ▼                              │
│       │                           ┌──────────┐                         │
│       └──────────(Feedback)────── │  TEST    │ ───(Fail)──> 🔁 Rollback│
│                                   └──────────┘                         │
│                                                                        │
│   📊 Key Metrics Point Extraction                                       │
│   ────────────────────────────────────────────────────────────────────  │
│   1️⃣ Deployment Freq. :    [DEPLOY] 단계의 횟수 측정                    │
│   2️⃣ Lead Time:            [CODE] -> [DEPLOY]까지의 시간 간격           │
│   3️⃣ Change Failure Rate:   [DEPLOY] 이후 [Rollback/Hotfix] 발생 비율   │
│   4️⃣ MTTR:                 장애 발생 -> [Service Normal] 복구 시간      │
│                                                                        │
└──────────────────────────────────────────────────────────────────────────┘
```

#### 📢 섹션 요약 비유
이는 **F1 레이싱 팀의 피트 스톱(Pit Stop) 전략**과 같습니다.
- **배포 빈도**는 얼마나 자주 피트에 들어와 타이어를 갈아끼우는지입니다.
- **리드 타임**은 피트 인(Pit In)에서 피트 아웃(Pit Out)까지 걸리는 시간입니다.
- **실패율**은 타이어가 잘못 끼워져 레이스를 중도 포기하는 확률입니다.
- **복구 시간**은 사고가 났을 때 다시 트랙으로 복귀하는 시간입니다. 엘리트 팀은 더 자주 멈추고(빠른 업데이트), 더 빠르게 교체하고(자동화), 훨씬 적게 사고를 냅니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. DORA vs SPACE 프레임워크 (심층 기술 비교)

DORA 지표는 강력하지만, '개인의 성장'이나 '팀의 문화'는 직접적으로 측정하지 못한다는 한계가 있습니다. 이를 보완하기 위해 **SPACE (Satisfaction, Performance, Activity, Communication, Efficiency)** 프레임워크와 함께 사용하는 것이 권장됩니다.

| 비교 항목 | **DORA Metrics** | **SPACE Framework** | **결과적 융합 방향** |
|:---|:---|:---|:---|
| **관점** | 시스템 및 팀 결과 중심 (Output) | 개인 경험 및 팀 웰빙 중심 (Human) | DORA로 시스템 성능을 보고, SPACE로 그 과정의 건강도 확인 |
| **주요 지표** | 배포 빈도, 리드 타임 등 객관 수치 | 개발자 만족도, 코드 리뷰 참여도 등 주관/협력 수치 | 수치가 좋아도 만족도가 낮으면 번아웃 위험 경고 |
| **적용 대상** | DevOps, SRE 조직 | Agile, 엔지니어링 문화 조직 | DevOps 전사 도입 시 함께 측정하여 균형 맞춤 |

#### 2. 기술 스택 시너지: CI/CD 및 Observability (관찰 가능성)

DORA 지표를 단순히 '엑셀로 관리'하는 것은 무의미합니다. **DevOps 툴체인**과 자동으로 연결되어야 합니다.

1.  **CI/CD 파이프라인 (Jenkins, GitLab CI, Actions)**:
    - 배포 빈도(DF), 리드 타임(LT): 파이프라인 로그에서 자동 추출.
    - 실패율(CFR): 자동화 테스트 커버리지를 높여 CFR을 낮추는 핵심 기술.
2.  ** observability (Prometheus, Grafana, ELK)**:
    - 복구 시간(MTTR): 장애 알림 발생 시점부터 복구 완료 시점을 로그와 메트릭으로 연결하여 자동 계산.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│              [Tech Stack Synergy for DORA Metrics]                           │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                               │
│   🔧  Automation Layer (Infrastructure as Code)                               │
│   ─────────────────────────────────────────────────────────────────────────  │
│   [Terraform/Ansible]  --->  환경 구성 자동화  --->  실패율(CFR) 감소         │
│                                                                               │
│   🚀  Delivery Layer (CI/CD)                                                   │
│   ─────────────────────────────────────────────────────────────────────────  │
│   [Jenkins/GitLab]     --->  배포 파이프라인 구축 --->  빈도(DF) 증가         │
│                                                                     리드타임(LT) 감소 │
│                                                                               │
│   👁️  Observability Layer (Monitoring)                                         │
│   ─────────────────────────────────────────────────────────────────────────  │
│   [Datadog/NewRelic]   --->  실시간 장애 탐지    --->  복구시간(MTTR) 단축    │
│                                                                               │
│   💡  Convergence: 모든 레이어의 데이터가 집계되어 "단일의 진실 소스(SSOT)" 구축 │
│                                                                               │
└──────────────────────────────────────────────────────────────────────────────┘
```

#### 📢 섹션 요약 비유
이는 **자동차의 내비게이션(NAV)과 안전 시스템의 결합**과 같습니다. 단순히 얼마나 빨리 가는지(속도계)만 보는 것이 아니라, 도로 상황(로그/모니터링)을 실시간으로 파악하고, 운전자가 피로하지 않도록 컨디션을 체크(SPACE)하면서 안전하게 최단 경로를 찾는 것입니다.
