# 663. 스토리 포인트 플래닝 포커 합의

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 작업의 소요 시간(Hours/Days)이라는 절대적이고 주관적인 지표 대신, **복잡도(Complexity)`, `노력(Effort)`, `불확실성(Uncertainty)`을 종합하여 상대적 크기를 수치화하는 `스토리 포인트(Story Points)` 개념을 도입하여 추정의 객관화를 달성한다.
> 2. **메커니즘**: `플래닝 포커(Planning Poker)`라는 `와이드밴드 델파이(Wideband Delphi)` 기반의 합의 프로토콜을 통해, `앵커링 효과(Anchoring Effect)`를 차단하고 팀원 간의 암묵지를 공유하여 추정 오차범위를 최소화한다.
> 3. **가치**: 개별 역량 편차를 흡수하고 팀의 `속도(Velocity)`를 정량화함으로써, `MVP(Minimum Viable Product)` 출시일 등 장기적인 프로젝트 `ROI(Return on Investment)` 분석의 신뢰도를 획기적으로 높인다.

---

### Ⅰ. 개요 (Context & Background)

**개념 및 정의**
`스토리 포인트(Story Point)`는 `애자일(Agile)` 개발 방법론에서 `사용자 스토리(User Story)`의 규모를 측정하는 상대적 단위입니다. 이는 단순히 '얼마나 걸리는가(Time)'가 아니라 '얼마나 복잡한가(Complexity)'와 '얼마나 모르는가(Uncertainty)'를 척도화합니다. 전통적인 `절대 추정(Absolute Estimation)` 방식은 숙련도(Seniority)에 따라 편차가 심하여 프로젝트 리스크로 이어지는 반면, `상대 추정(Relative Estimation)`은 팀 전체의 공감대를 바탕으로 일관성을 확보합니다.

**등장 배경 및 필요성**
1940년대 `와이드밴드 델파이(Wideband Delphi)` 기법에서 시작된 아이디어는 현대 애자일로 발전하며 `시간 기반 추정(Time-based Estimation)`의 한계를 극복했습니다. "이 기능 구현에 3시간 걸려요"라는 발언은 그 사람이 누구냐에 따라 달라지지만, "이 기능은 저번 기능보다 2배 어렵다"는 발언은 상대적 비교이므로 팀원 간 합의가 가능합니다. 이를 통해 `스크럼(Scrum)` 팀은 `번다운 차트(Burndown Chart)`를 기반으로 한 투명한 일정 관리가 가능해집니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        [추정 패러다임의 변화]                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ● 과거 (절대 추정)                                                         │
│    "Login 기능: 8시간" / "결제 기능: 40시간"                                 │
│     └─▶ 문제: 시니어(1시간) vs 주니어(10시간) → '누구의 시간인가?' 모호함    │
│                                                                             │
│  ● 현재 (상대 추정)                                                         │
│    "Login 기능: 1pt (기준)" / "결제 기능: 5pt (상대적 복잡도 반영)"           │
│     └─▶ 해결: 팀 전체가 "결제이 로그인보다 5배 어렵다"는 점에 합의           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

> **📢 섹션 요약 비유**:
> 마치 여행객들이 "서울에서 부산까지 2시간 걸려요"라고 말하는 대신, "서울에서 부산 거리는 집에서 역까지 거리의 100배 정도 돼요"라고 상대적인 거리 단위를 약속하여 지도상의 위치를 파악하는 것과 같습니다. 누가 타든 비행기로 가는 시간은 다르지만, 두 지점 간의 상대적 거리 개념은 변하지 않기 때문입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 및 상세 분석**
스토리 포인트 시스템은 단순한 숫자가 아니라 다음의 3대 요소가 결합된 복합 지표입니다.

| 구성 요소 (Component) | 정의 및 내부 동작 | 산정 고려 사항 | 비유 (Analogy) |
|:---|:---|:---|:---|
| **복잡도 (Complexity)** | 로직의 난이도, 연계 시스템 수, 데이터 구조 | 코드 깊이(Depth), 알고리즘 난이도 | 도로의 굽이 정도와 신호등 개수 |
| **노력 (Effort)** | 수행에 필요한 총 작업량 (Amount of Work) | 라인의 수, 입력 필드 수, UI 개수 | 짐의 무게와 상자의 개수 |
| **불확실성 (Uncertainty)** | 기술적 모호성, 요구사항 변경 가능성 (Risk) | 신기술 도입 여부, 문서 부재 정도 | 안개가 낀 정도, 날씨 예보 없는 항해 |

**핵심 알고리즘: 피보나치 수열(Fibonacci Sequence)의 활용**
추정 단위로 `1, 2, 3, 5, 8, 13, 21, ...` (혹은 수정된 `0, ½, 1, 2, 3, 5, 8, 13, 20, 40, 100`)을 사용합니다. 이는 인간의 인지 능력이 선형적이지 않기 때문입니다. 8과 9의 차이를 구별하는 것보다, 8과 13(약 1.6배 차이)을 구별하는 것이 훨씬 수월하며, 숫자가 클수록 불확실성이 기하급수적으로 커짐을 반영합니다.

**플래닝 포커(Planning Poker) 프로토콜 동작 시퀀스**
이 과정은 `BDB(Bid Development Process)`의 형태를 띠며 다음과 같이 진행됩니다.

```text
     [Phase 1: Narration]              [Phase 2: Selection]          [Phase 3: Reveal]
  ┌──────────────────────┐        ┌───────────────────────┐      ┌───────────────────────┐
  │  Moderator (PO/SM)   │        │   Team Members        │      │   Synchronized Reveal  │
  │                      │        │                       │      │                       │
  │ "US-663 결제 모듈    │  ────▶  │ 1. Think Silence     │  ───▶│ 📢 Flip Cards!        │
  │  PG사 연동을 포함.   │        │ 2. Select Card        │      │                       │
  │  위험 요소는?"       │        │    (Hidden Number)    │      │   [3] [8] [3] [13]    │
  └──────────────────────┘        └───────────────────────┘      └───────────────────────┘
           │                                                              │
           ▼                                                              ▼
     [Phase 4: Consensus Building]                                  [Phase 5: Record]
  ┌───────────────────────────────────────────────────────────────────────────────┐
  │  IF |Max - Min| ≤ 1 (Consensus)     THEN    Record Point & Next Story         │
  │      │                                                               │        │
  │  ELSE (Outliers Detected)        ▶       Extreme Views Debate                │
  │       - Moderator: "Why 13?" (High/Low pivot discussion)                     │
  │       - Re-vote after discussion (usually converges)                         │
  └───────────────────────────────────────────────────────────────────────────────┘
```

**심층 해설**
위 다이어그램에서 가장 중요한 단계는 **Phase 3 (Reveal)**과 **Phase 4 (Debate)**입니다. 만약 3점(주니어)과 13점(시니어)이 나왔다면, 시니어는 13점을 제시한 이유(예: 보안 취약점 처리 리스크)를 설명해야 합니다. 이 토론 과정이 바로 스토리에 숨겨진 '알 수 없는 작업(Unknown Unknowns)'을 찾아내는 핵심 메커니즘입니다.

> **📢 섹션 요약 비유**:
> 고속도로 톨게이트에 하이패스 차로가 설치된 것과 같습니다. 모든 차량이 일일이 정지하고 요금을 내는 '절대 시간' 방식(각자 추정)은 병목을 일으키지만, 차종을 구분해 상대적 단위로 빠르게 통과시키는 '하이패스(플래닝 포커)' 시스템은 흐름을 원활하게 하고, 동시에 차량(스토리)의 무게(규모)를 자동으로 측정하는 저울처럼 작동합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**절대 추정 vs 상대 추정 (Quantitative Comparison)**

| 비교 항목 | 절대 추정 (Absolute Estimation) | 상대 추정 (Relative Estimation) |
|:---|:---|:---|
| **측정 단위** | Man-Hour, Man-Day, Calendar Days | Story Points, T-Shirt Sizes (S/M/L) |
| **의존성** | 개인의 숙련도(Expertise)에 심각하게 의존 | 팀의 합의된 기준(Team Baseline) 의존 |
| **예측 모델** | 선형적 회귀 (단순 누적) | `속도(Velocity)` 기반의 지수 평활 |
| **오차 범위** | 불확실성이 클수록 오차 폭발 (+100% 이상) | 오차가 범위 내에서 상쇄됨 (평균 회귀) |
| **심리학적 편향** | `낙관성 편향(Optimism Bias)` 취약 | `앵커링(Anchoring)` 효과 차단 가능 |

**융합 시너지: 프로젝트 관리(Project Management)와의 연계**
스토리 포인트는 단순한 개발 도구를 넘어 재무적 의사결정과 연결됩니다.

1.  **EVM(Earned Value Management) 대체재**: 
    -   `PV(Planned Value)` = (Sum of Story Points) × (Cost Per Point)
    -   `SPI(Schedule Performance Index)` = 현재까지 완료 포인트 / 계획된 포인트
    -   이를 통해 "예산 대비 몇 %가 진행되었는가"를 공학적으로 계산할 수 있습니다.

2.  **코드(Code)와 데이터(Data) 분석과의 시너지**:
    -   실제 구현 후 `코드 복잡도(Cyclomatic Complexity)`와 스토리 포인트 간의 상관관계를 분석하면, 팀이 과대/과소 추정하는 패턴을 발견하여 보정할 수 있습니다.
    -   `CI/CD(Continuous Integration/Continuous Deployment)` 파이프라인에서 빌드 실패 횟수가 높은 스토리의 포인트를 재조정하여 기술 부채를 반영합니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│               [Stakeholder 별 Story Points 활용 관점]                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  👔 경영진 (Management)                                                    │
│     └─▶ Cost Per Point = (팀 인건비 / Sprint별 총 포인트)                  │
│         → 단위당 원가를 알아 전체 예산 산정에 활용                           │
│                                                                             │
│  👨‍💻 개발팀 (Dev Team)                                                     │
│     └─▶ Velocity Tracking (속도 추적)                                       │
│         → "이번 Sprint은 평균보다 느리네? 장애가 있었나?"                     │
│         → 팀의 컨디션을 체크하는 체온계 역할                                 │
│                                                                             │
│  🤝 고객 (Customer)                                                         │
│     └─▶ Priority Ranking                                                   │
│         → "이 기능은 8점인데, 그 정도 가치가 있나?"                           │
│         → Backlog refinement(백로그 정제)의 우선순위 결정 도구               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

> **📢 섹션 요약 비유**:
> 마치 복잡한 국제 외환 시장에서 원화, 달러, 엔화라는 각기 다른 '시간' 화폐를 쓰는 대신, 모든 나라가 공감하는 '금(Gold)'의 무게(온스)를 기준으로 거래하는 것과 같습니다. 환율(역량 차이)이 변하더라도 1온스의 가치는 보편적이므로, 글로벌 시장(프로젝트)의 가치 흐름을 투명하게 파악할 수 있습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**실무 시나리오: 기술 부채(Debt)와 신규 기능의 딜레마**
- **상황**: 레거시 시스템에 신규 "실시간 알림" 기능 추가. 시니어 개발자는 인프라 변경이 필요해 13점을 제시하지만, 주니어는 단순 API 추가라고 생각해 2점을 제시함.
- **의사결정 프로세스**:
    1.  **토론 유도**: 시니어가 `메시지 큐(Message Queue)` 도입 필요성과 `레거시 DB 락(Lock)` 이슈를 설명함.
    2.  **재추정(Re-estimate)**: 주니어가 복잡도를 이해 후 8점으로 수정, 하지만 `기술 부채 상환(Refactoring)`이 선행되어야 함을 공감.
    3.  **전략적 분리**:
        -   Task A: 레거시 코드 리팩토링 (기술 부채, 13점)
        -   Task B: 알림 기능 구현 (신규 스토리, 5점)
    4.  **결론**: PO(Product Owner)와 협의하여 기술 부채를 별도 스토리로 분리하여 Sprint에 포함시킴.

**도입 체크리스트 (Guidelines)**
-   [ ] **기준점 정의(Anchor)**: "1점 짜리 스토리"를 구체적으로 정의했는가? (예: 로그인 페이지 수정, 단순 CRUD)
-   [ ] **숫자 세트 선택**: 팀의 성숙도에 따라 (1, 2, 3... 10)인지 피보나치인지 결정했는가?
-   [ ] **토론 시간 제한**: 스토리 하나당 합의까지 5~10분을 넘기지 않도록 타이머를 사용했는가? (시간
