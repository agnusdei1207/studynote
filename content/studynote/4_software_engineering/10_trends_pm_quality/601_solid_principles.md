# 601. 객체지향 5원칙 SOLID 완벽 매핑

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SOLID (Single Responsibility, Open-Closed, Liskov Substitution, Interface Segregation, Dependency Inversion) 원칙은 **변경에 유연하고 확장 가능한 OOD (Object-Oriented Design)**를 위한 5대 핵심 지침이다.
> 2. **가치**: 결합도(Coupling)를 낮추고 응집도(Cohesion)를 높여 유지보수 비용을 획기적으로 절감하며, 대규모 시스템에서 **기술 부채(Technical Debt)의 누적을 방지**하는 아키텍처적 기반이 된다.
> 3. **융합**: 클린 아키텍처(Clean Architecture), 헥사고날 아키텍처(Hexagonal Architecture), MSA (Microservices Architecture) 등 현대 아키텍처 패턴의 이론적 근간이며, 테스트 주도 개발(TDD, Test-Driven Development)과 결합하여 소프트웨어 생명주기 전반의 품질을 보장한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: SOLID 원칙은 2000년대 초반 Robert C. Martin (Uncle Bob)이 정립한 객체지향 설계 5대 원칙의 앞자를 딴 약어다. 각 문자는 SRP (Single Responsibility Principle), OCP (Open-Closed Principle), LSP (Liskov Substitution Principle), ISP (Interface Segregation Principle), DIP (Dependency Inversion Principle)를 의미하며, **"변경에는 닫히고 확장에는 열려 있는"** 소프트웨어 설계의 정수를 담고 있다.

- **💡 비유**: SOLID 원칙은 **"레고 블록 조립"**과 같습니다. 각 블록(클래스)은 하나의 명확한 역할만 수행하고(SRP), 기존 블록을 수정하지 않고도 새 블록을 추가하여 확장할 수 있으며(OCP), 호환되는 블록끼리는 자유롭게 교체 가능하다(LSP). 불필요한 연결부 없이 필요한 인터페이스만 노출되어(ISP), 상위 블록이 하위 블록의 구현이 아닌 추상에 의존하는(DIP) 구조입니다.

- **등장 배경**:
    1. **스파게티 코드의 폐해**: 1990년대 후반 절차적 프로그래밍 방식의 거대한 God Class(Monolith)와 순환 의존(Circular Dependency) 구조가 유지보수를 불가능하게 만들었다.
    2. **디자인 패턴(Design Pattern)의 정착**: GoF (Gang of Four) 23가지 디자인 패턴이 널리 쓰이면서, 이 패턴들이 내부적으로 따르는 공통 원칙을 체계화할 필요성이 대두되었다.
    3. **애자일(Agile) 선언과의 연계**: "변경에 대응하는 능력"이 핵심 가치로 부상하면서, 미래의 변경을 예측 가능하게 만드는 설계 원칙이 요구되었다.

- **📢 섹션 요약 비유**: 마치 건물을 지을 때 기둥·보·벽이 각자의 역할에 충실하면서도, 내부 인테리어를 바꿔도 구조가 흔들리지 않는 내진 설계(Seismic Design)와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### SOLID 원칙 5가지 구성 요소

| 원칙 | 핵심 정의 | 위배 시 증상 | 설계 목표 | 비유 |
|:---:|:---|:---|:---|:---|
| **SRP** | 클래스는 오직 하나의 변경 이유만 가져야 함 | God Class, Rigidity | 높은 응집도(Cohesion) | 멀티태스킹 금지 |
| **OCP** | 확장에는 열려 있고, 변경에는 닫혀 있어야 함 | 산탄대수 수정(Fragility) | 낮은 결합도(Coupling) | 플러그인 구조 |
| **LSP** | 자식 클래스는 부모 클래스를 대체 가능해야 함 | 서브클래싱 망가짐 | 치환 가능성(Substitutability) | 사각형은 직사각형 |
| **ISP** | 클라이언트는 사용하지 않는 인터페이스에 의존하면 안 됨 | Fat Interface, 불필요한 결합 | 인터페이스 분리 | 세분된 콘센트 |
| **DIP** | 상위 모듈은 하위 모듈이 아닌 추상화에 의존해야 함 | 구현체 결합, 테스트 불가 | 의존성 역전(Inversion) | 전구 소켓 표준 |

#### 심층 동작 원리: SRP와 OCP의 상호작용

SRP(단일 책임 원칙)는 클래스 내부의 결합도(Cohesion)를 높이고, OCP(개방-폐쇄 원칙)는 클래스 간의 결합도(Coupling)를 낮추는 상호 보완적 관계에 있다. SRP를 통해 각 모듈이 하나의 변경 이유만 갖도록 분리하면, 해당 변경이 발생했을 때 영향 범위가 해당 모듈로 한정된다. OCP를 통해 기존 모듈을 수정하지 않고 새 기능을 확장 가능하게 만들면, 변경으로 인한 Ripple Effect(파급 효과)를 차단할 수 있다.

```
변경 전이: Business Logic Change → SRP 위배(God Class) → 전체 코드 영역 수정 → 회귀 버그 발생
변경 후: Business Logic Change → SRP 준수(단일 책임) → 해당 클래스만 수정 + OCP 준수(확장) → 새 클래스 추가 → 격리 완료
```

- **📢 섹션 요약 비유**: 자동차의 부품(모듈) 간에 표준화된 볼트와 너트(인터페이스)가 있어, 어떤 회사의 부품이든 교체 가능하면서도, 엔진을 교체할 때 전체 차체를 뜯어고 고쳐야 하는 상황을 방지하는 모듈형 설계와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 절차적 프로그래밍 vs SOLID 기반 OOD (Object-Oriented Design)

| 비교 항목 | 절차적 프로그래밍 | SOLID 기반 OOD |
|:---:|:---|:---|
| **변경 대응성** | 낮음 (한 곳 수정 시 전체 영향) | 높음 (격리된 모듈만 수정) |
| **테스트 용이성** | 어려움 (높은 결합도) | 쉬움 (의존성 주입, Mock 활용) |
| **코드 재사용** | 제한적 (Cut & Paste 방식) | 우수 (인터페이스 기반 조립) |
| **확장성** | Switch 문 분기 증가 | 다형성(Polymorphism) 활용 |
| **비유** | 통나무 집 | 레고 블록 조립 |

#### 2. SOLID와 디자인 패턴(Design Pattern)의 관계

GoF 23가지 디자인 패턴의 대부분은 SOLID 원칙을 구체화한 재료(Recipe)다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│            SOLID Principles ↔ Design Patterns Mapping                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  SRP → Strategy Pattern, Command Pattern (책임 분리)                   │
│  OCP → Strategy Pattern, Decorator Pattern (확장 개방)                  │
│  LSP → Template Method Pattern (치환 가능성 보장)                       │
│  ISP → Facade Pattern, Proxy Pattern (인터페이스 분리)                  │
│  DIP → Factory Pattern, DI Container (의존성 역전)                     │
│                                                                         │
│  예: Strategy Pattern + OCP                                             │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │                                                                  │  │
│  │   Context (Client)                                              │  │
│  │      │                                                          │  │
│  │      │  uses (다형성)                                           │  │
│  │      ▼                                                          │  │
│  │   <<interface>> SortStrategy                                    │  │
│  │      ▲                                                          │  │
│  │      │ implements                                               │  │
│  │   ┌──┴──────────────────┬──────────────────┬─────────────┐      │  │
│  │   │                     │                  │             │      │  │
│  │ QuickSort          MergeSort        HeapSort      (NewSort)    │  │
│  │                                                                  │  │
│  │ ✅ OCP 준수: 정렬 알고리즘 추가 시 Context 코드 수정 불필요      │  │
│  └──────────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 3. 과목 융합 관점

- **데이터베이스(DB)**: ORM (Object-Relational Mapping) 매핑 시 LSP를 위배하면(예: 상속 구조의 자식 테이블 조회 시 JOIN 복잡도 폭발), 쿼리 성능 저하가 발생한다. ActiveRecord 패턴(Django ORM 등)은 SRP 위배 가능성이 있어 도메인 모델 패턴(Domain Model Pattern)으로 분리하는 전략이 요구된다.

- **운영체제(OS)**: LSP는 파일 시스템 인터페이스(File System Interface) 설계에 핵심적이다. 모든 파일 시스템 구현체(ext4, NTFS, FAT32)이 동일한 인터페이스를 준수해야 VFS (Virtual File System) 레이어에서 치환 가능하다. 만약 특정 파일 시스템이 `write()` 호출 시 기대와 다른 방식으로 동작하면(예: 쓰기 지연), 커널이 의도한 대로 교체되지 않는 버그가 발생한다.

- **📢 섹션 요약 비유**: 전구가 나사 규격(인터페이스)만 맞으면 어떤 브랜드든 소켓에 꽂히듯, SOLID 원칙은 소프트웨어 구성요소 간의 "표준화된 연결 규격"을 정의하여 플러그인 방식의 조립을 가능하게 합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오 1: 결제 모듈 리팩토링**
    - **문제**: 하나의 PaymentService 클래스가 신용카드, 간편결제, 계좌이체를 모두 처리하여(SRP 위배), 새로운 결제 수단 추가 시마다 기존 코드를 수정해야 했다(OCP 위배).
    - **의사결정**: 추상화 계층(IPaymentGateway) 도입 후 각 결제 수단을 독립 클래스로 분리.
    - **결과**: 신규 결제사 연동 시 기존 테스트 케이스 재실행 불필요, 배포 리스크 80% 감소.

- **실무 시나리오 2: LSP 위배로 인한 버그 발생**
    - **문제**: Rectangle을 상속받은 Square 클래스에서 width 설정 시 height도 함께 변경되도록 구현했으나, 클라이언트가 Rectangle- Square 치환 가능성을 가정하고 `setWidth(10); setHeight(20);` 호출 시 정사각형 성질이 깨져 버그 발생.
    - **의사결정**: Square와 Rectangle을 독립 클래스로 분리하거나, Shape 공통 인터페이스만 추출.
    - **결과**: 런타임 타입 캐스팅(Runtime Type Casting) 제거, 타입 안정성(Type Safety) 확보.

- **도입 체크리스트**:
    1. **SRP**: 각 클래스의 변경 이유(Reason to Change)가 단 하나인가? Git Log에서 특정 파일이 매 커밋마다 수정되는 패턴이 있다면 SRP 위배일 가능성이 높다.
    2. **OCP**: 새로운 기능 추가 시 기존 코드의 `if-else`나 `switch-case` 분기를 추가하는 대신, 상속/인터페이스 구현으로 해결했는가?
    3. **LSP**: 상위 타입의 인스턴스를 하위 타입으로 교체해도 프로그램의 정확성이 깨지지 않는가?
    4. **ISP**: 클라이언트가 사용하지 않는 메서드 때문에 빈 메서드(Stub Method)를 구현하게 만드는 Fat Interface가 존재하는가?
    5. **DIP**: 상위 모듈이 구체 클래스(Class)가 아닌 추상화(Interface/Abstract Class)를 통해 하위 모듈과 통신하는가?

- **안티패턴**:
    - **God Object**: 하나의 클래스가 시스템의 대부분의 기능을 수행(SRP 위배).
    - **Shotgun Surgery**: 하나의 요구사항 변경으로 여러 클래스를 동시에 수정해야 하는 상황(SRP/OCP 위배).
    - **Refused Bequest**: 상위 클래스의 메서드를 하위 클래스에서 빈 메서드로 오버라이딩(ISP 위배).

- **📢 섹션 요약 비유**: 의사가 수술을, 간호가 환자 간호를, 약제가 투약을 담당하듯(SRP), 각자의 영역을 명확히 분리하면서도, 서로 협력할 때는 표준화된 수술 절차(인터페이스)를 따르는 병원 시스템과 같습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. SOLID 원칙은 장난감 정리함처럼 **각 장난감(클래스)이 제자리에 있어서**, 필요할 때 금방 찾을 수 있고(단일 책임), 새 장난감을 사도 기존 정리 방식을 바꿀 필요가 없는(개방-폐쇄) 규칙이에요.
2. 모든 전구가 어떤 회사 제품이든 소켓에 꽂히듯(리스코프 치환), 불필요한 버튼이 없는 깔끔한 리모컨(인터페이스 분리)처럼, 서로 바꿔 끼울 수 있고(의존성 역전), 쓸모없는 기능 없이 딱 필요한 것만 있는 거죠.
3. 이 원칙을 지키면 컴퓨터 프로그램이 **부서마다 역할이 명확한 회사**처럼 잘 돌아가서, 나중에 기능을 추가하거나 고칠 때 모두가 행복해진답니다!
