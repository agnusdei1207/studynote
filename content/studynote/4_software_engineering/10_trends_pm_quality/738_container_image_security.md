# 738. 컨테이너 이미지 스캐닝 권한 통제

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컨테이너화된 애플리케이션의 보안을 위해, 베이스 이미지와 레이어 내의 취약점을 자동으로 탐지하는 **이미지 스캐닝**과, 신뢰할 수 있는 사용자/시스템만이 이미지를 생성·배포할 수 있도록 제한하는 **권한 통제** 기술이다.
> 2. **보안 게이트**: CI/CD 파이프라인에서 취약점이 발견된 이미지의 빌드를 자동 차단하고, 쿠버네티스의 **Admission Controller**를 통해 승인되지 않은(Untrusted) 이미지의 실행을 거부한다.
> 3. **가치**: "신뢰할 수 없는 이미지는 절대로 실행되지 않는다"는 원칙을 실현하여, 컨테이너 환경의 주된 위협인 악성코드 주입 및 취약점 악용 공격을 원천 봉쇄한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "보이지 않는 컨테이너의 속살"

컨테이너는 애플리케이션을 실행하기 위해 수많은 OS 패키지와 라이브러리를 포함합니다. 만약 베이스 이미지(예: Ubuntu, Alpine)에 취약점이 있다면, 그 위에 쌓인 우리 앱은 시작부터 해킹의 위협에 노출됩니다. 또한 누구나 이미지를 서버에 올릴 수 있다면, 악성코드가 심긴 가짜 이미지가 배포될 수도 있습니다. **이미지 스캐닝과 권한 통제**는 컨테이너 환경의 필수 방역 시스템입니다.

### 💡 비유: 공항의 수하물 검사와 신분증 확인

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        컨테이너 보안 비유                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 해외 여행을 위해 비행기(서버)에 짐(컨테이너 이미지)을 실어야 함.           │
│                                                                             │
│  1. 이미지 스캐닝 (X-레이 검사):                                              │
│     짐가방 속에 폭발물(취약점)이나 위험한 물건이 없는지 낱낱이 들여다봄.           │
│     - 위험한 게 나오면 가방을 뺏고 비행기에 못 싣게 함 (배포 차단).               │
│                                                                             │
│  2. 권한 통제 (신분증 및 비자 확인):                                          │
│     아무나 짐을 실을 수 없음. 정식 직원(승인된 개발자)만 짐을 옮길 수 있고,        │
│     비행기 근처(운영 환경)에는 허가된 사람만 접근 가능함 (RBAC).                │
│                                                                             │
│  → 내용물도 깨끗해야 하고, **믿을 수 있는 사람**이 가져온 짐만 싣는 것!           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 컨테이너 이미지 스캐닝 (Vulnerability Scanning)
- **정적 분석**: 이미지를 실행하지 않고 내부 파일 시스템과 패키지 매니저(apt, npm 등) 목록을 CVE 데이터베이스와 대조.
- **레이어 분석**: 도커의 각 레이어(Layer)별로 변경 사항을 추적하여 어느 단계에서 취약점이 유입되었는지 특정.
- **도구**: Trivy, Clair, Anchore, Snyk.

### 2. 권한 및 실행 통제 (Access & Admission Control)
- **RBAC (Role-Based Access Control)**: 이미지 저장소(Registry)에 푸시/풀 할 수 있는 권한을 역할별로 세분화.
- **Image Signing (이미지 서명)**: Notary나 Cosign을 사용하여 이미지의 제작자가 본인임을 디지털 서명으로 입증.
- **Admission Controller**: 쿠버네티스 엔진이 Pod를 실행하기 직전에 "이 이미지는 검증된 저장소에서 왔는가?", "스캔 결과가 통과인가?"를 최종 체크.

### 3. 보안 파이프라인 아키텍처 (ASCII)

```text
    [ Code ] ──▶ [ Build Image ] ──▶ [ Vulnerability Scan ] ──▶ [ Sign Image ]
                                            │ (If High Risk)       (Private Key)
                                            ▼                      │
                                       [ Fail Build ]              ▼
                                                            [ Push to Registry ]
                                                                   │
          ┌────────────────────────────────────────────────────────┘
          ▼
    [ K8s API Server ] ◀── [ Admission Controller ] ◀── [ Policy: Signed Only ]
           │                      (Final Check)
           ▼
    [ Run Container ]
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 일반 서버 보안 vs 컨테이너 보안

| 항목 | 일반 서버 (Host) | 컨테이너 (Image) |
|:---:|:---|:---|
| **검사 대상** | 설치된 패키지, 설정 파일 | **이미지 레이어, 라이브러리 의존성** |
| **보안 시점** | 운영 중 상시 감시 | **빌드 및 배포 시점 (Shift-left)** |
| **변경 방식** | 실행 중 패치 적용 (Mutable) | **이미지 새로 빌드 (Immutable)** |
| **통제 핵심** | SSH 접근 제어 | **레지스트리 및 Admission Control** |

### 2. 기술적 시너지: SBOM (Software Bill of Materials)
이미지 스캔 결과는 **SBOM**의 핵심 데이터가 됩니다. 반대로 SBOM이 있으면 이미지를 일일이 풀(Pull) 받지 않고도, 중앙에서 전사적인 취약점 영향도를 검색할 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: "Zero-Vulnerability" 배포 정책
- **상황**: 사내 서비스에서 보안 사고 발생 후 컨테이너 보안 강화 요구됨.
- **결단**: **Trivy와 OPA (Open Policy Agent) 연계**. 
- **판단**: 
    1. Jenkins 파이프라인에 Trivy 스캔 단계 추가. `Critical` 취약점 발생 시 빌드 강제 중단.
    2. 쿠버네티스에 Kyverno나 OPA를 설치하여, '신뢰할 수 있는 사내 레지스트리' 주소가 아닌 모든 이미지 실행 거부 정책 적용.
- **효과**: 개발자가 실수로 외부의 취약한 이미지를 가져다 쓰는 행위를 시스템적으로 차단.

### 📢 기술사적 결언
> "컨테이너 보안의 핵심은 **'가시성'**과 **'불변성'**이다. 이미지가 만들어지는 순간부터 그 안의 모든 요소를 낱낱이 알고 있어야 하며, 한 번 검증된 이미지는 운영 환경까지 절대 변하지 않음을 보장해야 한다. 아키텍트는 단순 스캐닝 도구 도입을 넘어, **'신뢰할 수 있는 이미지 공급망(Trusted Supply Chain)'**을 설계하고 이를 강제하는 정책 엔진을 구축하는 거버넌스 능력을 갖추어야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **취약점 노출 감소**: 운영 환경의 고위험 취약점 95% 이상 사전 제거.
- **컴플라이언스 준수**: 이미지 이력 및 스캔 결과 자동 보관으로 감사 대응력 향상.

### 미래 전망
미래의 컨테이너 보안은 **'AI 기반 동적 프로파일링'**으로 진화할 것입니다. 이미지가 실행될 때 필요한 최소한의 시스템 콜(Syscall)만 AI가 학습하여 정의하고, 그 외의 동작은 모두 차단하는 'Zero-trust Runtime'이 보편화될 것입니다. 또한 **하드웨어 기반 보안(TEE)**과 결합하여, 이미지가 암호화된 상태로 메모리에서 실행되어 탈취가 불가능한 'Confidential Container' 시대가 올 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **이미지 스캐닝**은 학교 급식 재료가 들어올 때, 상한 채소가 없는지 돋보기를 들고 **꼼꼼하게 검사하는 것**과 같아요.
2. **권한 통제**는 학교 영양사 선생님이 허락한 **믿을 수 있는 아저씨**만 재료를 배달할 수 있게 문을 지키는 거죠.
3. 이 두 가지를 잘하면, 우리 친구들이 **배 아프지 않고 건강하게** 맛있는 점심(서비스)을 먹을 수 있답니다!
