# 730. 객체지향 응집도 (우연, 논리, 시간, 절차, 통신, 순차, 기능)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모듈(클래스) 내부의 구성 요소들이 하나의 목적을 위해 얼마나 긴밀하게 협력하고 집중되어 있는지를 나타내는 지표로, **소프트웨어의 독립성과 품질**을 결정하는 핵심 척도이다.
> 2. **계단식 구조**: 응집도가 높을수록(기능적 응집도) 좋으며, 낮을수록(우연적 응집도) 나쁘다. **기능 > 순차 > 통신 > 절차 > 시간 > 논리 > 우연** 순으로 응집도가 낮아진다.
> 3. **가치**: "높은 응집도(High Cohesion)"를 달성함으로써 모듈의 의미를 명확히 하고, 코드의 이해도를 높이며 변경 발생 시 수정 범위를 국소화하여 유지보수성을 극대화한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "모듈은 하나의 일만 해야 한다"

로버트 C. 마틴의 SRP(단일 책임 원칙)의 모태가 되는 개념입니다. 한 클래스에 로그인, 결제, 로그 기록 로직이 다 들어있다면, 그 클래스는 '응집도가 낮다'고 합니다. 응집도가 낮은 모듈은 코드를 읽기 힘들게 만들고, 작은 수정에도 예상치 못한 오류를 발생시킵니다. 좋은 설계는 **'딱 필요한 것만 모아놓은'** 상태를 지향합니다.

### 💡 비유: 가방 속의 물건 정리

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        객체지향 응집도 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 기능적 응집도 (최상 - 필통):                                              │
│     필통 안에는 연필, 지우개, 자 등 "필기"라는 하나의 목적을 위한 도구만 있음.      │
│                                                                             │
│  2. 시간적 응집도 (주의 - 아침 준비 주머니):                                   │
│     양치 도구와 알람시계가 같이 들어있음. 관련은 없지만 "아침"이라는 시간에        │
│     같이 쓰기 때문임. (시간이 바뀌면 주머니 구성이 깨짐)                          │
│                                                                             │
│  3. 우연적 응집도 (최악 - 쓰레기통):                                          │
│     먹다 남은 빵, 찌그러진 캔, 헌 종이가 섞여 있음. 아무런 연관성 없이 그냥        │
│     한 공간에 던져진 상태. (무엇을 위한 가방인지 알 수 없음)                      │
│                                                                             │
│  → 하나의 가방이 하나의 역할만 명확히 수행할 때(기능), 가장 쓰기 편한 법!          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 응집도 7단계 상세 분석 (나쁨 → 좋음)

| 단계 | 유형 | 상세 내용 | 품질 |
|:---:|:---|:---|:---:|
| **1** | **우연적 (Coincidental)** | 모듈 내부 요소들이 서로 아무런 관련 없이 모여 있음 | **최악** |
| **2** | **논리적 (Logical)** | 유사한 성격의 기능을 수행하지만 서로 연관은 없음 (예: 모든 출력 로직 집합) | 나쁨 |
| **3** | **시간적 (Temporal)** | 특정 시간에 순차적으로 실행되는 활동들을 모아둠 (예: 초기화 함수) | 낮음 |
| **4** | **절차적 (Procedural)** | 정해진 순서에 따라 수행되지만 데이터 공유는 없음 | 보통 |
| **5** | **통신적 (Communicational)** | 동일한 입력/출력 데이터를 사용하는 다른 기능들을 모아둠 | 양호 |
| **6** | **순차적 (Sequential)** | 한 요소의 출력이 다음 요소의 입력으로 사용되는 관계 | 우수 |
| **7** | **기능적 (Functional)** | 모듈 내부의 모든 요소가 단일한 목적의 기능을 수행함 | **최상** |

### 2. 응집도와 결합도의 반비례 관계
일반적으로 **응집도가 높아지면**, 모듈이 독립적으로 변하므로 외부와의 **결합도는 낮아집니다.** 이 "High Cohesion, Loose Coupling"이 소프트웨어 공학의 성배(Holy Grail)입니다.

### 3. 응집도 시각화 (ASCII)

```text
    [ Low Cohesion ]                      [ High Cohesion ]
    (우연적/논리적)                         (기능적 응집도)
    
    ┌───────────────┐                     ┌───────────────┐
    │   Module A    │                     │   Module B    │
    │  - Login()    │                     │  - Pay()      │
    │  - CalcTax()  │                     │  - Validate() │
    │  - SaveLog()  │                     │  - UpdateDB() │
    └───────────────┘                     └───────────────┘
    (서로 다른 일들)                       (결제라는 하나의 목적)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 응집도 향상을 위한 리팩토링 기법
- **메서드 추출 (Extract Method)**: 하나의 메서드가 여러 일을 하면 잘게 쪼개어 기능적 응집도를 높입니다.
- **클래스 추출 (Extract Class)**: 거대 클래스 내에서 관련 있는 필드와 로직을 별도 클래스로 분리합니다.

### 2. 기술적 시너지: 마이크로서비스 (MSA)
MSA의 서비스 분할 원칙 중 하나는 '비즈니스 능력(Business Capability)'에 따른 분할입니다. 이는 아키텍처 수준에서의 **기능적 응집도**를 추구하는 행위이며, 서비스 내부의 응집도가 높을수록 장애 전파가 차단되고 독립적 배포가 쉬워집니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: "유틸리티 클래스"의 진화
- **상황**: `CommonUtils`라는 클래스에 날짜 변환, 문자열 처리, 암호화 로직이 모두 들어있음 (논리적 응집도).
- **결단**: **기능별 클래스 분리**. 
- **판단**: 개발자가 암호화 로직을 찾기 위해 수천 줄의 유틸리티 코드를 뒤져야 함. 또한 날짜 로직 수정 시 암호화 라이브러리 의존성까지 신경 써야 하는 문제 발생.
- **조치**: `DateConverter`, `StringHelper`, `CryptoManager`로 분리.
- **효과**: 각 클래스의 목적이 명확해지고(기능적 응집도), 코드 리뷰와 재사용이 훨씬 쉬워짐.

### 📢 기술사적 결언
> "응집도는 클래스의 **'정체성'**이다. 이름과 역할이 일치하지 않는 모듈은 잠재적인 버그의 온상이다. 아키텍트는 클래스 이름을 지을 때 'And'나 'Manager'와 같은 모호한 단어가 들어가는지 감시해야 한다. 하나의 단어로 그 클래스의 역할을 설명할 수 있을 때 비로소 기능적 응집도가 달성된 것이다. 높은 응집도는 단순한 코딩 스타일이 아니라, 시스템의 **'엔트로피 증가'**를 억제하는 유일한 방어 기제다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **가독성 향상**: 모듈 이해 시간 50% 이상 단축.
- **결함 수정 속도**: 로직 응집으로 인해 버그 발생 시 원인 지점 특정 속도 향상.

### 미래 전망
미래의 개발 환경에서는 **'AI 자동 모듈화'**가 활성화될 것입니다. AI가 코드 간의 데이터 흐름과 호출 빈도를 분석하여, 응집도가 낮은 클래스를 자동으로 식별하고 최적의 분리 방안을 추천하는 지능형 IDE 기능이 보편화될 것입니다. 또한 서버리스와 같은 '함수 단위 아키텍처'가 확산되면서, 응집도는 설계의 기본 단위를 넘어 **'과금과 성능의 단위'**가 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **응집도**는 내 방 서랍장이 얼마나 잘 정리되어 있느냐 하는 거예요. 
2. 첫 번째 칸엔 양말만, 두 번째 칸엔 티셔츠만 들어있는 게 **응집도가 높은(똑똑한)** 정리법이죠.
3. 양말과 과자가 섞여 있으면(응집도 낮음) 나중에 찾기도 힘들고 지저분해지는 것처럼, **끼리끼리 모아두는 것**이 제일 중요하답니다!
