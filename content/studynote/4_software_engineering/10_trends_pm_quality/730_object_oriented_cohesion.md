# 730. 객체지향 응집도 (우연, 논리, 시간, 절차, 통신, 순차, 기능)

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 모듈(Module) 내부 구성 요소들이 단일 목적 달성을 위해 얼마나 긴밀하게 상호작용하는지를 측정하는 **내부적 품질 지수**이며, 시스템의 복잡도를 통제하는 핵심 설계 원리이다.
> 2. **가치**: 기능적 응집도(Functional Cohesion)를 최우선으로 달성하여 **변경 파급효과(Side Effect)를 최소화**하고, 코드의 재사용성과 테스트 용이성을 비약적으로 향상시킨다.
> 3. **융합**: SRP(Single Responsibility Principle), MSA(Microservices Architecture), AI 기반 리팩토링 등 현대 소프트웨어 아키텍처의 근간을 이루는 철학이다.

## Ⅰ. 개요 (Context & Background)

### 1. 개념 및 정의
**응집도(Cohesion)**란 소프트웨어 공학에서 모듈(클래스, 함수, 컴포넌트) 내부의 구성 요소들이 서로 얼마나 밀접하게 관련되어 있는지를 나타내는 척도이다.
마치 원자의 결합력이 단단할수록 물질이 견고하듯, 모듈 내부의 요소들이 하나의 **'목적(Purpose)'**을 위해 강하게 뭉쳐질수록 시스템은 robust해진다. 
반대로, 서로 관련없는 기능들이 섞인 모듈은 응집도가 낮다고 하며, 이는 유지보수 시 '스파게티 코드'나 '신(God) 객체'를 양산하는 주범이 된다.

### 2. 등장 배경: 소프트웨어 위기 극복
1960년대 후반~1970년대 구조적 설계(Structured Design)의 시대에서 Yourdon, Constantine, Myers 등의 학자들은 복잡해지는 소프트웨어의 난이도를 관리하기 위해 **결합도(Coupling)**와 **응집도(Cohesion)**라는 두 가지 축을 제안했다.
단순히 코드를 줄이는 것을 넘어, **"변경이 발생했을 때 수정해야 할 코드의 범위를 최소화"**하는 것을 목표로 하며, 이는 현대의 **CI/CD (Continuous Integration/Continuous Deployment)** 및 **애자일(Agile)** 방법론의 이론적 기반이 되었다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                     소프트웨어 품질의 제1법칙: 응집도 vs 결합도                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   【Good Design】                                                           │
│    ┌──────────────────┐        High Cohesion  (강한 내부 결합)                │
│    │   High Cohesion  │◄───────+  ─┐                                      │
│    │   Low Coupling   │           │  │                                     │
│    └──────────────────┘           │  │  모듈 내부는 뭉치고(↑)                 │
│           │                       │  │  모듈 간은 느슨하고(↓)                 │
│           ▼                       ▼  ▼                                     │
│    * 독립적인 배포 가능                                                      │
│    * 유지보수 비용 최소화                                                      │
│    * 이해도 극대화                                                          │
│                                                                             │
│   【Bad Design】                                                            │
│    ┌──────────────────┐        Low Cohesion   (해적 내부 분리)                │
│    │   Low Cohesion   │◄───────+  ─┐                                     │
│    │   High Coupling  │           │  │                                    │
│    └──────────────────┘           │  │  모듈 내부는 흐트러지고(↓)             │
│           │                       │  │  모듈 간은 얽히고설키고(↑)             │
│           ▼                       ▼  ▼                                    │
│    * 삽질 유발 (한 줄 바꾸면 전체 멸망)                                          │
│    * 악몽 같은 버그 재현                                                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
**(해설)** 위 다이어그램은 이상적인 소프트웨어 구조와 그 반대의 상태를 대조한다. 좋은 설계는 모듈 내부의 응집도는 높여 스스로 완결성을 갖게 하고, 모듈 간의 결합도는 낮춰 독립성을 확보하는 것이다. 이는 건물로 치면 내력벽(응집도)은 튼튼하게, 연결부(결합도)는 유연하게 설계하는 것과 같다.

### 📢 섹션 요약 비유
마치 **'군대의 특수부대편'**과 같습니다. 특수부대원들은 모두 같은 임무(기능적 응집)를 수행하기 위해 훈련되어 있어, 한 명만 뺴도 작전에 차질이 생기죠. 반면 징병된 신병들(우연적 응집)을 섞어놓은 부대는 서로 눈치도 안 보고 혼란만 가중시키는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 응집도의 7단계 계층 구조 (The Ladder of Cohesion)
응집도는 최악인 '우연적 응집도'부터 최상인 '기능적 응집도'까지 7단계로 분류된다. 이는 단순히 나쁨/좋음의 차이를 넘어, **"왜 모였는가?"**에 대한 철학적 차이를 반영한다.

#### 1) 상세 구성 요소 분석표

| 등급 | 유형 (Type) | 내부 동작 메커니즘 | 모듈 예시 (Code Context) | 품질 |
|:---:|:---|:---|:---|:---:|
| **1** | **우연적 (Coincidental)** | 요소 간 관계 없음. 단순 편의상 묶임. `class Utils { a(), b(), c() }` | **최악** (Chaotic) |
| **2** | **논리적 (Logical)** | 카테고리만 유사 (예: 입력모음). 실제론 다른 데이터 처리. `class IO { readKeyboard(), readDisk() }` | 나쁨 (Disordered) |
| **3** | **시간적 (Temporal)** | 실행 시점이 같음. 순서 없음. `class Init { connectDB(), loadConfig() }` | 낮음 (Simple Bucket) |
| **4** | **절차적 (Procedural)** | 실행 순서에 의존. 데이터 공유 안 함. `void doJob() { a(); b(); c(); }` | 보통 (Ordered) |
| **5** | **통신적 (Communicational)** | 같은 데이터(입/출력)를 사용함. `class Customer { getName(), updateName() }` | 양호 (Data-focused) |
| **6** | **순차적 (Sequential)** | 출력 → 입력 연계 (Pipe 형태). `void process() { data = read(); parse(data); }` | 우수 (Data-flow) |
| **7** | **기능적 (Functional)** | 단일 목적 달성 (핵심 1개 + 보조 n개). `class Sin { calc(x) }` | **최상** (Clean) |

### 2. 응집도 계산 및 시각화 (수리적 모델)
**LCOM (Lack of Cohesion of Methods)** 메트릭을 통해 응집도를 수치화할 수 있다.
*   $LCOM = |P| - |Q|$ (단, $|P| > |Q|$인 경우)
*   $P$: 서로 다른 인스턴스 변수를 사용하는 메서드 쌍의 집합
*   $Q$: 같은 인스턴스 변수를 사용하는 메서드 쌍의 집합
값이 0에 가까울수록 응집도가 높음을 의미한다.

### 3. 응집도 레벨별 시각적 비교 (ASCII)

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    7단계 응집도의 내부 구조 (Inner Structure)                 │
└─────────────────────────────────────────────────────────────────────────────┘

1. 우연적 (Coincidental)         2. 논리적 (Logical)           3. 시간적 (Temporal)
   [ Random Box ]                  [ Type Container ]           [ Time Bucket ]
   +-------------+                 +-------------+              +-------------+
   │ Cat Sound   │                 │ Print A     │              │ Boot()      │
   │ Triangle    │                 │ Print B     │              │ Init()      │
   │ Rocket Calc │                 │ Print C     │              │ Start()     │
   +-------------+                 +-------------+              +-------------+
   (서로 전혀 다른 기능)             (비슷한 종류의 기능)           (시점이 같은 기능)


4. 절차적 (Procedural)          5. 통신적 (Communicational)   6. 순차적 (Sequential)
   [ Step Flow ]                  [ Data Holder ]              [ Assembly Line ]
   +-------------+                 +-------------+              +-------------+
   │ Step A: Cut │◄───────────┐    │ get Name()  │              │ Parse() ──► │
   │ Step B: Sew │───────┐    │    │ set Name()  │              ▼             │
   │ Step C: Pack │      │    │    │ get Addr()  │         Validate() ──► │
   +-------------+      │    │    +-------------+              ▼             │
       (순서는 있으나          (같은 데이터(Name)을           (출력이 다음의 입력으로)
        데이터 연계 없음)       공유하며 처리함)

                 7. 기능적 (Functional) [ Best Practice ]
                          [ Target Scope ]
                          +---------------------------------------+
                          │  >>> GOAL: Calculate Sin(x) <<<        │
                          │  +-----------------------------------+ │
                          │  | Core: calcSin(x)                  │ │
                          │  | Sub  : toRad(x)                   │ │
                          │  | Sub  : taylorSeries(x)            │ │
                          │  +-----------------------------------+ │
                          +---------------------------------------+
                   (모든 요소가 'Sin 계산'이라는 하나의 목적에 봉사)
```
**(해설)** 위 다이어그램은 각 응집도 단계가 모듈 내부적으로 어떻게 연결되어 있는지 도식화한 것이다. 우연적 응집도는 그저 쓰레기통에 물건을 던져넣은 것처럼 무질서하며, 순차적 응집도는 공장 라인처럼 데이터가 흘러간다. 최상위인 기능적 응집도는 레이저처럼 하나의 목표(Goal)를 향해 모든 내부 로직이 집중(Focus)되어 있음을 보여준다.

### 4. 핵심 알고리즘: 응집도 측정 로직 (Pseudo-code)

```python
# LCOM (Lack of Cohesion of Methods) 계산 알고리즘 예시
# Reference: Chidamber & Kemerer (1994)

def calculate_lcom(class_methods, instance_variables):
    """
    클래스의 응집도 부족도를 계산하는 함수.
    반환값이 높을수록 응집도가 낮다.
    """
    P = set()  # 서로 다른 변수를 사용하는 메서드 쌍 (Disconnected)
    Q = set()  # 같은 변수를 사용하는 메서드 쌍 (Connected)

    methods = list(class_methods)
    for i in range(len(methods)):
        for j in range(i + 1, len(methods)):
            vars_i = get_used_vars(methods[i]) # 메서드 i가 쓰는 변수 집합
            vars_j = get_used_vars(methods[j]) # 메서드 j가 쓰는 변수 집합

            if vars_i.isdisjoint(vars_j):
                P.add((methods[i], methods[j]))
            else:
                Q.add((methods[i], methods[j]))
    
    # LCOM = |P| - |Q| (단, 0보다 작으면 0으로 처리하는 경우도 많음)
    lcom_value = len(P) - len(Q)
    return max(0, lcom_value)
```

### 📢 섹션 요약 비유
마치 **'요리 레시피와 재료'**의 관계와 같습니다.
기능적 응집도는 **'스테이크를 굽는 집'**(불, 고기, 소스, 집게가 모두 스테이크를 위해 존재)이고, 
우연적 응집도는 **'할인마트'**(전구, 우유, 드라이버, 참치캔이 같이 있음)과 같습니다. 
스테이크 집이 스테이크 맛을 위해 모든 역량을 집중하는 것이 곧 높은 응집도입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 심층 기술 비교: 결합도(Coupling) vs 응집도(Cohesion)

| 비교 항목 | **응집도 (Cohesion)** | **결합도 (Coupling)** |
|:---|:---|:---|
| **관점** | **내부 지향 (Inward)**: 모듈 내부 요소 간의 결합력 | **외부 지향 (Outward)**: 모듈 간의 의존성 |
| **목표** | **최대화 (MAX)**: 높을수록 좋음 (초점화) | **최소화 (MIN)**: 낮을수록 좋음 (독립성) |
| **단위** | 메서드, 필드, 클래스 내부 | 클래스, 패키지, 라이브러리 간 |
| **주요 패턴** | SRP (단일 책임 원칙) | DIP (의존성 역전 원칙), DI (의존성 주입) |
| **파급 효과** | 낮은 응집도 → 이해도 저하, 내부 버그 증가 | 높은 결합도 → 연쇄 수정, 재사용성 저하 |

### 2. 과목 융합 관점

① **데이터베이스 (DB)와의 융합: 정규화(Normalization)**
- RDBMS의 **정규화 과정**은 이종적 응집도를 제거하고 데이터 기반의 높은 응집도를 달성하는 행위와 본질적으로 같다.
- 제1정규형(원자성), 제3정규형(이행적 함수 종속 제거)을 거치며 테이블 내부의 데이터 응집도를 높여 **이상 현상(Anomaly)**을 방지한다.

② **아키텍처 (Architecture)와의 융합: 마이크로서비스 (MSA)**
- **MSA (Microservices Architecture)**는 거시적 관점의 **서비스 단위 기능적 응집도** 분리 전략이다.
- Monolithic 구조에서는 시간적/절차적 응집도로 뭉쳐있던 로직들을, 비즈니스 Capability(역량) 단위로 쪼개어 **Bounded Context**로 만드는 과정이 곧 응집도 향상 작업
