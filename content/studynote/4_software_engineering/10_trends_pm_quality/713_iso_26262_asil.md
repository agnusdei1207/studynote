# 713. 기능 안전 ISO 26262 ASIL 등급

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 자동차 전기/전자(E/E) 시스템의 기능적 오류로 인한 위험을 관리하기 위한 **ISO 26262** 표준의 핵심인 **ASIL (Automotive Safety Integrity Level)**은 잠재 위험의 심각도(S), 노출 빈도(E), 통제 가능성(C)을 정량적으로 분석하여 시스템 무결성 수준을 결정하는 척도이다.
> 2. **메커니즘**: HARA (Hazard Analysis and Risk Assessment)를 통해 도출된 ASIL 등급(A~D)에 따라, 요구되는 SW/SW 안전 메커니즘의 엄격도(예: **MC/DC Coverage**, **Fault Tolerance** 등)가 계층적으로 결정되며, ASIL D는 가장 높은 무결성을 요구한다.
> 3. **실무 가치**: 단순한 품질 관리를 넘어, 시스템 고장 시 인명 피해 확률을 통계적으로 허용 불가능한 수준(예: < 10^-8 / hour)으로 낮추는 것을 목적으로 하며, 이는 글로벌 OEM (Original Equipment Manufacturer) 납품을 위한 필수 통과 관문(Ticket to Market)이다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 기능 안전의 개념과 필연성
**기능 안전(Functional Safety)**은 시스템이 고장 날 경우, 그 고장이 인명이나 환경에 해를 끼치지 않도록 예방 조치를 취하는 활동이다. 일반 소프트웨어의 버그가 서비스 중단을 유발한다면, 자동차 소프트웨어의 결함은 생명을 앗아갈 수 있다는 점에서 결정적인 차이가 있다. **ISO 26262**는 2011년 IEC 61508(산업용 기능 안전)을 자동차 환경에 맞게 적용한 표준으로, 전자 제어 장치(ECU)와 임베디드 소프트웨어의 안전성 생명주기를 규정한다.

#### 2. 등장 배경: 소프트웨어 정의 차량(SDV)의 도래
자동차는 기계적 연결(Mechanical Link)에서 전자적 제어(X-by-Wire)로 전환되었다. 과거 브레이크 시스템은 유압으로 직접 연결되었으나,現在는 브레이크 페달의 신호를 센서가 인식하고 ECU가 연산하여 모터를 제어한다. 이 과정에서 **SPoF (Single Point of Failure, 단일 고장점)**가 존재하면, 시스템은 제어권을 상실하여 대형 사고로 이어진다. 이를 방지하기 위해 유럽을 중심으로 법적/제도적 강제 사항이 되었다.

#### 3. 위험도 분석 및 ASIL 분류 매트릭스
**ASIL 등급**은 HARA (Hazard Analysis and Risk Assessment) 분석을 통해 결정된다.

| 척도 | 영문 명칭 | 설명 | 등급 |
|:---:|:---|:---|:---:|
| **심각도** | **Severity (S)** | 사고 발생 시 피해 규모 | S0 ~ S3 (치명상) |
| **노출률** | **Exposure (E)** | 운전 중 해당 상황 발생 빈도 | E0 ~ E4 (높음) |
| **제어성** | **Controllability (C)** | 운전자가 위험을 회피할 수 있는 정도 | C0 ~ C3 (불가) |

> **💡 일상 비유**
> 자동차를 '놀이공원의 롤러코스터'라고 생각해 보자.
> 기능 안전이란 "롤러코스터가 고장 나더라도 사람이 떨어지지 않도록, 안전 벨트(잠금 장치)와 비상 제동 장치를 미리 설계하는 규칙"이다.

#### 4. ASIL 결정 로직 다이어그램
아래는 세 가지 요소(S, E, C)가 어떻게 조합되어 ASIL 등급을 결정하는지 보여주는 흐름도이다.

```text
      [ Hazard Analysis (위험 분석) ]
                 │
      ───────────┼───────────────────────
      ▼          ▼          ▼
  Severity   Exposure  Controllability
  (심각도)   (노출도)     (제어성)
      │          │          │
      └──────────┴──────────┘
                 │
          [ 조합 (S, E, C) ]
                 │
       ┌─────────┴─────────┐
       ▼   ▼   ▼   ▼   ▼
      QM   A   B   C   D   ← 결정된 무결성 수준
       │   │   │   │   │
       ▼   ▼   ▼   ▼   ▼
   (품질관리) (최저) (보통) (높음) (최고)

   * 예: (S3, E4, C3) → ASIL D (치명상, 항상 발생, 제어 불가)
```
*(해설: 위 다이어그램은 ISO 26262 Part 3의 HARA 매트릭스를 개념화한 것이다. 가장 우측인 ASIL D는 '치명적인 상해'가 발생하고 '운전자가 아무것도 할 수 없는' 상황에 해당하며, 이에 대해서는 가장 엄격한 설계 기준을 적용한다.)*

> **📢 섹션 요약 비유**
> 마치 고속도로 톨게이트에서 차량의 위험도를 검사해, 위험한 차량일수록 전용 차선(ASIL D)으로 보내 아주 엄격한 검사(안전 메커니즘)를 수행하게 하는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. V-모델을 통한 안전 수명 주기 (Safety Lifecycle)
ISO 26262는 자동차 개발의 표준 프로세스인 **V-Model**을 기반으로 하며, 각 단계별로 안전 요구사항을 파생, 검증, 검증하는 순환 구조를 갖는다.

| 모듈 | 역할 | 내부 동작 및 프로토콜/기법 | 비고 |
|:---|:---|:---|:---|
| **Safety Goal** | 최상위 안전 목표 | "시스템 고장 시 차량이 멈추지 않는다"와 같은 추상적 목표 정의 | ASIL 결정 |
| **FSR (Functional Safety Req.)** | 기능적 안전 요구사항 | Safety Goal을 달성하기 위한 기능적 동작 정의 (예: 토크 모니터링) | 락스텝(Lock-step) 코어 |
| **TSR (Technical Safety Req.)** | 기술적 안전 요구사항 | HW/SW 구현 사양 (예: ECC 메모리, 주기적 셀프 테스트) | FIT (Failure In Time) |
| **SW Implementation** | 소프트웨어 구현 | MISRA C 준수, 형식 검증(Formal Verification) | MC/DC 달성 |

#### 2. ASIL 등급별 아키텍처 및 커버리지 요구사항
ASIL 등급이 높아질수록 요구되는 **고장 감지(Fault Detection)** 및 **중복성(Redundancy)**의 수준이 기하급수적으로 상승한다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                 ASIL 등급별 아키텍처 및 검증 엄격도                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  Level D (ASIL D)  [████████████████████████████] 100% (Near Absolute)      │
│  ─────────────────────────────────────────────────────────────────────────  │
│  [HW Architecture] Dual-Core Lockstep + ECC + Dedicated Safety Monitor      │
│  [SW Coverage]    MC/DC (Modified Condition/Decision Coverage) 100%         │
│  [Diagnostic]     > 99% Fault Detection Capability                          │
│                                                                             │
│  Level C (ASIL C)  [█████████████████████░░░░░░] 90% (High)                 │
│  ─────────────────────────────────────────────────────────────────────────  │
│  [HW Architecture] Partial Redundancy or High-Quality Component             │
│  [SW Coverage]    MC/DC (Partial) or Decision Coverage 100%                 │
│  [Diagnostic]     > 90% Fault Detection Capability                          │
│                                                                             │
│  Level B (ASIL B)  [█████████████████░░░░░░░░░░░] 60% (Medium)              │
│  ─────────────────────────────────────────────────────────────────────────  │
│  [HW Architecture] Random HW Failures consideration                          │
│  [SW Coverage]    Decision Coverage / Statement Coverage 100%               │
│  [Diagnostic]     > 60% Fault Detection Capability                          │
│                                                                             │
│  Level A (ASIL A)  [█████████░░░░░░░░░░░░░░░░░░] 30% (Low)                  │
│  ─────────────────────────────────────────────────────────────────────────  │
│  [SW Coverage]    Statement Coverage                                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
*(해설: 위 그래프와 설명은 ASIL 등급에 따른 기술적 부담의 차이를 시각화한 것이다. ASIL D는 우주 항공 수준의 무결성을 요구하며, 이를 위해 이중화된 코어가 서로의 계산을 실시간 비교하는 Lockstep 방식을 주로 사용한다.)*

#### 3. 핵심 알고리즘: MC/DC (Modified Condition/Decision Coverage)
ASIL D 소프트웨어 검증의 핵심은 **MC/DC**이다. 이는 단순한 코드 실행(Statement Coverage)을 넘어, 복합 조건문 내의 개별 조건이 결과에 독립적으로 영향을 미치는지 검증한다.

* **공식/원리**: 조건문 `(A and B) or C`가 있다면, A가 True일 때와 False일 때 전체 결과가 변하는지를 B, C의 상태를 고정하여 검증한다.
* **코드 예시 (C언어)**:
```c
// ASIL D 인증 대상 코드 예시
// 변수 A, B는 센서 입력값 (Safety-related)
bool check_sensor_failure(uint8_t A, uint8_t B) {
    /* 
     * Safety Mechanism: 각 조건이 독립적으로 영향을 미치는지 
     * 테스트 케이스로 증명해야 함 (MC/DC).
     * Case 1: A=Fail, B=Ok => Ret=True (Path 1)
     * Case 2: A=Ok, B=Fail => Ret=True (Path 2)
     * ...
     */
    if ((A > THRESHOLD_A) && (B > THRESHOLD_B)) {
        return SAFE_STATE; 
    }
    return FAULT_DETECTED;
}
```

#### 4. 안전 메커니즘: E-Gas 시스템 (3-Level Monitoring)
실무에서 엔진 제어나 모터 제어 시스템은 보통 **E-Gas 모니터링**이라는 3단계 안전 개념을 적용한다.

```text
   [ Level 1: Function Control ]
   (일반 제어 로직)
       │
       ▼ (Read Sensor / Calc Torque)
   ────┼─────────────────────────────────
       │
       ▼ (Watchdog / Plausibility Check)
   [ Level 2: Function Monitoring ]
   (모니터링 로직: 입력값/출력값 타당성 검사)
   - 예: 가속페달과 브레이크가 동시에 눌러짐?
       │
       ▼ (Error Detected -> Reset)
   ────┼─────────────────────────────────
       │
       ▼ (Hardware Logic / Comparator)
   [ Level 3: Controller Monitoring ]
   (HW적 오류 감지: 별도의 칩이 메인 칩 감시)
   - 예: Main CPU Freezing detection, Clock monitoring
       │
       ▼
   [ Safe State (Safe Torque Off) ]
```
*(해설: 레벨 1은 일반적인 기능 수행, 레벨 2는 ECU 내부에서 기능적 오류(논리 오류)를 감시, 레벨 3은 외부의 별도 하드웨어가 메인 컨트롤러 자체가 망가졌는지를 감시하는 방어 계층 구조이다.)*

> **📢 섹션 요약 비유**
> 성을 지키는 방어 시스템과 같습니다. 1층(기능)은 적을 공격하고, 2층(모니터링)은 1층 병사들이 제정신인지 감시하으며, 3층(컨트롤러)은 성 자체가 무너지지 않는지 확인하는 이중, 삼중의 방범 시스템을 갖추는 것입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 기능 안전 vs 제품 안전 (SOTIF) vs 사이버 보안

| 구분 | **기능 안전 (Functional Safety)** | **SOTIF (ISO 21448)** | **사이버 보안 (Cybersecurity)** |
|:---|:---|:---|:---|
| **대상 위험** | **시스템 오류(Hazard)**로 인한 위험 | 시스템은 정상이나 **AI 판단 오류** 등으로 인한 위험 | **악의적인 공격(Hacking)**으로 인한 위험 |
| **주요 표준** | **ISO 26262** (본 문서) | **ISO 21448 (SOTIF)** | **ISO/SAE 21434** |
| **접근 방식** | 고장(Fault)을 예방/관리 | 예측 불가능한 상황(Scenario)을 제한 | 공격 경로를 차단/탐지 |
| **융합 필요성** | 모든 기반 (기본) | AI 자율주행 시 필수 결합 | 해킹으로 인해 고장이 유발될 수 있으므로 통합 필수 |

#### 2. IEC 61508 (Industry) vs ISO 26262 (Automotive)

```text
     [ IEC 61508 (Functional Safety of Electrical/Electronic/Programmable Electronic Systems) ]
            │
            │  (Standard for all industries)
            ▼
     ┌──────┴──────┬──────────┬───────────┐
     ▼             ▼          ▼           ▼
  [ISO 26262]   [IEC 61511]  [IEC 62061]  [EN 50128]
    (Automotive)  (Process)   (Machine)    (Rail)
     
     [특징 비교]
     IEC 61508:  "범용 표준 (General)"
     ISO 26262:  "자동차 특화 (Automotive Specific)"
     - E/E 아키텍처 (Fault Tolerant Time Interval 등)
     - Part 1~10으로 구체화됨 (예: Part 11은 Semi-conductor 가이드)
```
*(해설: IEC 61508은 부모 표준이며, ISO 26262는 이를 자동차의 특수한 환경(배선, 배전, 고장 무시 시간 등)
