# 741. mTLS 상호 인증 서비스 간 보안

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기존 TLS (Transport Layer Security)가 서버의 신원만 검증하는 단방향 인증인 반면, mTLS (Mutual TLS)는 **클라이언트와 서버가 상호 인증서(Certificate)를 교차 검증**하는 양방향 인증 프로토콜로, '주체(Object)'와 '객체(Object)'의 신원을 모두 보증한다.
> 2. **가치**: 네트워크 엣지의 방화벽이 내부 트래픽에는 무력한 현실을 개선하여, 제로 트러스트(Zero Trust)의 핵심인 '신뢰할 수 있는 네트워크 구간'을 '신뢰할 수 있는 ID'로 대체함으로써 **내부자 위협 및 Lateral Movement(횡적 이동)를 효과적으로 차단**한다.
> 3. **융합**: 복잡한 인증서 발급 및 갱신(Rotation)의 운영 부담을 **Service Mesh(서비스 메시)**의 Control Plane(제어 영역)과 자동화하여, 마이크로서비스 아키텍처(MSA) 환경에서 보안과 가용성의 균형을 달성한다.

---

### Ⅰ. 개요 (Context & Background)

mTLS (Mutual Transport Layer Security)는 인터넷 상에서 정보를 암호화하여 전송하는 표준 프로토콜인 TLS의 확장된 형태로, 통신하려는 양쪽 당사자(클라이언트와 서버)가 각자의 신원을 증명하는 디지털 인증서(X.509 Certificate)를 서로 교환하고 검증하는 과정을 의미합니다. 기존의 TLS는 사용자가 웹사이트에 접속할 때, 웹사이트(서버)가 진짜인지만 확인했으나, mTLS는 서버가 클라이언트에게 "당신이 누구입니까?"라고 되묻고 그 신원을 암호학적으로 검증합니다.

이 기술의 등장 배경에는 클라우드 네이티브(Cloud Native) 환경의 보안 패러다임 변화가 자리 잡고 있습니다. 과거의 "경계 네트워크(Perimeter Network)" 내부는 안전하다는 가정(Trust Internal Network)은 더 이상 유효하지 않습니다. 해커가 외부 방화벽을 뚫거나 내부 사용자의 계정이 탈취될 경우, 내부 망에서는 서비스 간 통신에 대한 인증이 없어 무분별하게 데이터 유출이 발생할 수 있기 때문입니다. 이에 따라 **"신뢰하되 검증하며(Very Trust, But Always Verify)"**, 모든 서비스 간 호출마다 강력한 신원 검증을 수행하는 제로 트러스트(Zero Trust) 아키텍처의 핵심 요소로 mTLS가 자리 잡았습니다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                       [ 보안 패러다임의 변화 ]                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [과거: Castle & Moat 모델]          [현재: Zero Trust 모델]         │
│                                                                     │
│     🔒 Firewall                      🔒🔒🔒 End-to-End Encryption   │
│      ┌─────────┐                       ┌─────────┐                  │
│      │  SAFE   │  ← 내부는 신뢰         │  CHECK  │  ← 모든 구간 검증 │
│      │   ZONE  │     (암호화 없음)      │   EVERY │    (mTLS 적용)   │
│      └─────────┘                       └─────────┘                  │
│                                                                     │
│  💡 문제점: 방화벽 뚫리면           💡 해결책: 서비스 간          │
│             내부는 무법지대                 상호 인증으로              │
│             (Lateral Movement)             사칭 불가능               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

#### 📢 섹션 요약 비유:
마치 고급 보안 시설의 출입문에서, 방문객이 사원증을 찰칵 찍어 들어가는 것(일반 TLS)만으로는 부족하고, 경비원이 나와서 "신분증 좀 보겠습니다"라고 요청한 뒤, 방문객의 신원을 확인하고 경비원의 신분증도 방문객에게 보여주는 **'이중 출입 인증 절차'**와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

mTLS의 작동은 X.509 인증서 기반의 **PKI (Public Key Infrastructure)** 공개키 기반 구조에 의존하며, 크게 핸드셰이크(Handshake) 협상 과정과 데이터 평문 암호화 과정으로 나뉩니다. 실무적으로는 단순한 소켓 통신 레벨을 넘어, 마이크로서비스의 사이드카(Sidecar) 패턴을 통해 구현되는 경우가 대다수입니다.

#### 1. 핵심 구성 요소 (Component Analysis)
| 구성 요소 (Component) | 역할 (Role) | 상세 내부 동작 (Internal Operation) | 프로토콜/포맷 | 비유 (Analogy) |
|:---|:---|:---|:---|:---|
| **CA (Certificate Authority)** | **신뢰 기관** | 루트 인증서(Root Cert)를 관리하며, 하위 인증서 발급 및 폐지(CRL) 관리 | X.509, PKCS#10 | 여권 발행청 |
| **Server Certificate** | **서버 신원증** | 서버의 공개키(Public Key)와 FQDN 등 정보를 포함, CA의 개인키로 서명됨 | .pem, .crt | 편의점 영업증 |
| **Client Certificate** | **클라이언트 신원증** | 사용자나 서비스의 공개키 포함, Client Hello 시 서버로 전송됨 | .pem, .p12 | 고객 멤버십 카드 |
| **Session Key** | **대칭 키** | 핸드셰이크 시 생성되어 실제 데이터 암호화에 사용 (성능 보장) | AES-256-GCM | 일회용 대화 암호 |
| **Cipher Suite** | **암호화 제품군** | 키 교환(ECDHE), 인증(RSA/ECDSA), 암호화(AES), 무결성(SHA) 조합 | TLS_ECDHE_RSA_WITH_AES_128_GCM | 암호 규칙집 |

#### 2. mTLS Handshake 상세 메커니즘
mTLS의 핵심은 TCP 연결 성립 이후, 실제 데이터를 주고받기 전에 이루어지는 복잡한 협상 과정입니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    [ mTLS Handshake 상세 시퀀스 다이어그램 ]                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   [ Client ]                                          [ Server ]            │
│      │                                                    │                 │
│      │  1. Client Hello                                     │                 │
│      │  - 지원 가능한 Cipher Suite 목록 전송                   │                 │
│      │  - Random Bytes 생성                                   │                 │
│      ├──────────────────────────────────────────────────────▶│                 │
│      │                                                    │                 │
│      │  2. Server Hello + Certificate + Certificate Request   │                 │
│      │  ◀───────────────────────────────────────────────────┤                 │
│      │  - 서버 인증서 전송                                     │                 │
│      │  - "너도 인증서 내놔라" (CertificateRequest) 요청 ★핵심★ │                 │
│      │                                                    │                 │
│      │  3. Server Key Exchange (Optional) & ServerHelloDone   │                 │
│      │  ◀───────────────────────────────────────────────────┤                 │
│      │                                                    │                 │
│      │  4. Client Certificate + ClientKeyExchange             │                 │
│      ├──────────────────────────────────────────────────────▶│                 │
│      │  - 클라이언트 인증서 전송 (서버의 요청에 응답) ★핵심★     │                 │
│      │  - Pre-Master Secret 전송                              │                 │
│      │                                                    │                 │
│      │  5. Certificate Verify (Server 검증)                   │                 │
│      │  ◀───────────────────────────────────────────────────┤                 │
│      │  - CA 신뢰 체인 확인, CRL(폐지 목록) 확인, 서명 검증     │                 │
│      │  - 검증 실패 시 연결 종료 (Alert 메시지)                 │                 │
│      │                                                    │                 │
│      │  6. Change Cipher Spec & Finished (양방향)              │                 │
│      │  ◀───────────────────────────────────────────────────┤                 │
│      ├──────────────────────────────────────────────────────▶│                 │
│      │  - "이제부터 암호화 통신 시작"                          │                 │
│      │                                                    │                 │
└─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1.  **Certificate Request 단계의 결정적 차이**: 일반적인 TLS HTTPS 통신에서는 서버가 클라이언트에게 인증서를 요구하지 않지만(Step 2 생략), mTLS는 반드시 이 단계를 포함합니다.
2.  **검증(Verification)의 엄격함**: 서버는 클라이언트가 보낸 인증서가 신뢰할 수 있는 CA(Certificate Authority)에 의해 서명되었는지 확인하고, 해당 인증서가 폐지되지 않았는지 OCSP(Online Certificate Status Protocol)나 CRL(CRL)을 통해 확인합니다.
3.  **퍼포먼스 고려사항**: 이 핸드셰이크 과정은 비대칭 암호화(예: RSA 2048비트)를 사용하므로 CPU 연산 비용이 높습니다. 따라서 실무에서는 **Session Resumption(Session Ticket)**을 사용하여 연결을 재사용하거나, Connection Pooling을 통해 오버헤드를 줄이는 전략을 사용합니다.

#### 📢 섹션 요약 비유:
일반적인 화장실 입장 시, 사용자가 화장실 간판(서버 인증서)을 보고 남자/여자 화장실을 구분하는 것(TLS)과 달리, mTLS는 **화장실 입구에서 키를 관리하는 아저씨가 내 회원권(클라이언트 인증서)을 확인한 뒤, 나도 화장실 관리 아저씨(서버)가 진짜 직원인지 명찰을 확인하고 들어가는, 매우 꼬꼬마한 '상호 검색 절차'**와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

mTLS는 단순한 보안 프로토콜을 넘어 서비스 메시(Service Mesh) 및 API 게이트웨이와 결합하여 플랫폼 레벨의 보안을 제공합니다.

#### 1. 심층 기술 비교: 일반 TLS vs mTLS
| 비교 항목 (Metric) | 일반 TLS (One-way Authentication) | mTLS (Mutual Authentication) | 비고 (Note) |
|:---:|:---|:---|:---|
| **신뢰 모델 (Trust Model)** | 서버 신원만 신뢰 (Client는 익명) | **서버 ↔ 클라이언트 양방향 신뢰** | Zero Trust의 근간 |
| **핸드셰이크 복잡도** | 상대적으로 낮음 (RTT 2회) | **높음 (RTT 2~3회 + 검증 오버헤드)** | 초기 연결 지연 발생 가능 |
| **인증서 관리 (CA Ops)** | 서버 인증서만 관리 | **모든 서비스/기기 인증서 관리 필요** | 자동화 필수 (Istio CA 등) |
| **주요 공격 차단** | Phishing, MITM(일부) | **MITM(완전), Spoofing, Replay** | 내부 공격자 차단 가능 |
| **Latency (지연시간)** | 기준선 (Baseline) | **+5~10ms (초기 핸드셰이크 시)** | Keep-al� 사용 시 상쇄 가능 |

#### 2. 서비스 메시(Service Mesh)와의 융합
mTLS를 애플리케이션 소스 코드에 직접 구현하는 것은 '보안 로직의 산재'를 초래하여 유지보수를 어렵게 만듭니다. 이를 해결하기 위해 **Istio**나 **Linkerd** 같은 서비스 메시를 사용합니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│              [ 서비스 메시(Istio) 기반 mTLS 아키텍처 ]                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [ Control Plane (istiod) ]                                                 │
│  ┌──────────────────────┐                                                  │
│  │  Citadel (CA)        │ ──(1) Workload Bootstrap ──▶ 인증서 발급/배포    │
│  └──────────────────────┘      (gRPC Protocol)                             │
│           │                                                                │
│           │                                                                │
│  ─────────┼────────────────────────────────────────────────────────────────│
│           │                                                                │
│  [ Data Plane (Service Mesh) ]                                             │
│           │                                                                │
│   Pod A                               Pod B                                │
│  ┌─────────────┐     mTLS      ┌─────────────┐                            │
│  │ ┌─────────┐ │ ───────────▶  │ ┌─────────┐ │                            │
│  │ │  App    │ │   (Encrypted) │ │  App    │ │                            │
│  │ │ (Logic) │ │ ◀──────────── │ │ (Logic) │ │                            │
│  │ └────┬────┘ │               │ └────▲────┘ │                            │
│  │      │      │               │      │      │                            │
│  │ ┌────▼────┐ │  Local TCP    │ ┌────▼────┐ │                            │
│  │ │ Envoy   │ │ ============= │ │ Envoy   │ │                            │
│  │ │Sidecar  │ │   (Plaintext) │ │Sidecar  │ │                            │
│  │ └─────────┘ │               │ └─────────┘ │                            │
│  │             │               │             │                            │
│  └─────────────┘               └─────────────┘                            │
│                                                                             │
│   ① App는 로컬 Envoy(localhost)로 평문 전송 (암호화 로직 몰라도 됨)        │
│   ② Envoy 간에는 자동으로 mTLS 적용 (mTLS Permissive → Strict 모드 전환)  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**[융합 기술 효과]**
1.  **자동화된 인증서 관리**: 수천 개의 파드(Pod)가 뜰 때마다 수동으로 인증서를 넣어주는 것은 불가능합니다. Istio Citadel 같은 컴포넌트가 인증서의 수명 주기(Lifecycle)를 자동으로 관리
