# 787. 애그리게이트 루트 외부 접근 단일 진입점 설계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도메인 주도 설계(DDD)에서 연관된 객체들의 묶음인 애그리게이트의 데이터 정합성을 보장하기 위해, 오직 **애그리게이트 루트(Aggregate Root)**를 통해서만 내부의 다른 객체에 접근하고 상태를 변경하도록 강제하는 **캡슐화 설계 원칙**이다.
> 2. **일관성 보장**: 비즈니스 규칙(Invariants)을 루트 엔티티 내에 집중시켜, 외부에서 내부 객체를 직접 조작함으로써 발생할 수 있는 데이터 불일치 문제를 원천 차단한다.
> 3. **가치**: 트랜잭션의 범위를 애그리게이트 단위로 명확히 획정하여 동시성 제어를 단순화하고, 복잡한 도메인 모델 간의 결합도를 낮추어 유지보수성을 극대화한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "누구든 문을 통해서만 들어오세요"

객체들이 거미줄처럼 얽혀 있는 시스템에서는 어디서 누가 데이터를 바꿨는지 알기 어렵습니다. '주문' 내의 '주문 항목' 수량을 외부에서 마음대로 고치면, '주문'의 '총금액'은 틀린 정보가 됩니다. **애그리게이트 루트**는 이러한 무질서를 막기 위해 "이 묶음의 대장은 나다. 모든 요청은 나를 거쳐라"라고 선언하는 입구와 같습니다.

### 💡 비유: 아파트 단지와 보안 초소

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        애그리게이트 루트 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 한 아파트 단지(Aggregate) 안에 여러 집(Internal Objects)이 있음.        │
│                                                                             │
│  1. 루트가 없을 때: 외부인(Client)이 담을 넘어서 아무 집이나 들어감.             │
│     - 누가 들어왔는지, 집 안에서 무슨 일을 하는지 관리실이 모름. (정합성 파괴)      │
│                                                                             │
│  2. 루트가 있을 때 (보안 초소 - Root):                                        │
│     단지로 들어오는 길은 오직 하나, 정문 보안 초소뿐임.                          │
│     - 방문자는 초소에서 "누구세요? 왜 왔나요?"라고 검사를 받아야 함.              │
│     - 초소(Root)는 "지금은 청소 시간이라 못 들어갑니다"라고 규칙을 집행함.         │
│                                                                             │
│  → 모든 변화는 **정문(Root)**을 통해서만 일어나야 단지 전체가 안전함!             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 단일 진입점 설계의 3대 철칙
- **Direct Access Prohibition**: 외부 객체는 루트가 아닌 내부 객체의 참조를 가질 수 없음.
- **Reference by ID**: 다른 애그리게이트를 참조할 때는 객체가 아닌 **식별자(ID)**만 가짐.
- **Root-only Persistence**: 리포지토리는 오직 루트 엔티티 단위로만 존재함.

### 2. 애그리게이트 내부 구조 (ASCII)

```text
    [ Client / Application Service ]
           │
           ▼ (1) Call only Root methods
    ┌────────────────── [ Aggregate Boundary ] ──────────────────┐
    │                                                           │
    │      ┌─────────────────────────────┐                      │
    │      │   Order (Aggregate Root)    │                      │
    │      ├─────────────────────────────┤                      │
    │      │ - id: UUID                  │                      │
    │      │ + changeQuantity(itemId, n) │ ──┐                  │
    │      └──────────────┬──────────────┘   │ (2) Internal State│
    │                     │                  │     Update        │
    │      ┌──────────────▼──────────────┐   │                  │
    │      │      OrderItem (Member)     │ ◀─┘                  │
    │      └─────────────────────────────┘                      │
    │                                                           │
    └───────────────────────────────────────────────────────────┘
```

### 3. 루트의 비즈니스 규칙 집행 (Invariants)
루트는 내부 객체들의 상태를 변경하기 전, 항상 **비즈니스 제약 조건**을 검사합니다.
- 예: `Order.addItem()` 호출 시, 현재 주문 상태가 '배송 전'인지 확인하고, '최대 주문 수량'을 초과하지 않는지 계산 후 아이템 추가.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 캡슐화(Encapsulation) vs 애그리게이트 루트
- **캡슐화**: 클래스 내부의 변수를 숨기는 **미시적 보호**.
- **애그리게이트 루트**: 여러 클래스의 묶음을 외부로부터 통째로 보호하는 **거시적 보호**.

### 2. 기술적 시너지: 낙관적 잠금 (Optimistic Locking)
애그리게이트 루트에 **버전(Version)** 필드를 두어 관리하면, 내부의 어떤 객체가 바뀌어도 루트의 버전이 올라갑니다. 이를 통해 분산 환경에서 여러 사용자가 동시에 같은 묶음을 수정하려 할 때 데이터 충돌을 완벽하게 방지할 수 있습니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 게시판과 댓글 아키텍처
- **상황**: '게시글'과 '댓글'은 한 몸인가, 따로인가?
- **판단 1**: 댓글이 수만 개 달릴 수 있다면 별도의 애그리게이트로 분리 (성능 고려).
- **판단 2**: 게시글 삭제 시 댓글도 무조건 같이 삭제되어야 하고, 댓글 수가 게시글 테이블에 합산 관리되어야 한다면 하나의 애그리게이트로 묶음 (정합성 고려).
- **결단**: 비즈니스 중요도에 따라 '게시글'을 루트로 삼고, 모든 댓글 조작은 `Post.addComment()`를 통해 수행하도록 설계.

### 📢 기술사적 결언
> "애그리게이트 루트는 **'데이터 일관성의 최후 보루'**다. 이것을 무시하고 내부 객체를 직접 건드리는 것은 아키텍처에 '뒷문'을 열어주는 것과 같다. 아키텍트는 개발자들이 편리함을 이유로 루트의 권위를 흔들지 못하도록 리포지토리 설계와 인터페이스를 엄격히 제한해야 한다. 묶음의 크기가 커질수록 성능은 떨어지지만, 묶음을 쪼갤수록 정합성은 깨지기 쉽다는 **'트레이드오프'**를 이해하는 것이 설계의 핵심이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **데이터 무결성**: 로직 분산으로 인한 데이터 꼬임 사고 90% 이상 제거.
- **코드 추적성**: 모든 상태 변화가 루트의 메서드로 집중되어 디버깅 속도 향상.

### 미래 전망
미래의 애그리게이트 설계는 **'상태 머신(State Machine) 자동화'**와 결합할 것입니다. 루트 엔티티의 복잡한 상태 전이 규칙을 코드가 아닌 선언적 명세로 정의하고, 엔진이 이를 자동으로 검증하는 체계가 구축될 것입니다. 또한 마이크로서비스 환경에서 여러 애그리게이트 간의 일관성을 맞추기 위해 **사가(Saga) 패턴**과 루트의 이벤트 발행 기능이 더욱 긴밀하게 융합될 것입니다.
