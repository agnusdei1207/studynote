# 641. ISO 25010 소프트웨어 품질 모델

## # 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 제품 품질의 국제 측정 표준으로, **ISO/IEC 9126 (International Organization for Standardization/International Electrotechnical Commission 9126)**을 계승하여 **SQuaRE (System and Software Quality Requirements and Evaluation)** 시리즈의 핵심 모델로 자리 잡았습니다.
> 2. **진화**: 단순한 기능 위주의 품질에서 벗어나, 현대 소프트웨어의 생명인 **보안성(Security)**과 **호환성(Compatibility)**을 독립 주특성으로 격상시켜 클라우드 및 연결 환경에 최적화되었습니다.
> 3. **실무 가치**: 개발자의 주관적 판단을 배제하고, 고객의 요구사항을 정량적 지표(Metrics)로 변환하여 SDLC (Software Development Life Cycle) 전 단계에 걸쳐 품질 목표를 객체화하는 글로벌 공통 언어 역할을 수행합니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 및 철학
ISO 25010은 소프트웨어의 '품질'이라는 모호한 개념을 정량화하고 체계화한 국제 표준입니다. 이는 단순히 '버그가 없는 상태'를 넘어, 사용자 컨텍스트(Context), 비즈니스 목표, 그리고 기술적 제약 사항을 모두 고려하는 다차원적 모델입니다. 과거의 ISO/IEC 9126 모델을 기반으로 하되, 급변하는 기술 환경(모바일, 클라우드, IoT)에 대응하기 위해 품질 속성을 재정비하고 세분화했습니다.

### 2. 등장 배경 및 패러다임 변화
1.  **기존 한계**: 초기 품질 모델은 기능적 완결성에 집중하여, 시스템이 죽지 않고 돌아가는 수준의 신뢰성(Reliability)에만 초점을 맞춤.
2.  **혁신적 패러다임**: 인터넷 발전으로 데이터 유출 사고 등 보안 이슈가 대두되고, 타 시스템과의 연동이 필수가 되면서 '보안'과 '호환성'이 선택이 아닌 필수 생존 요소로 부상.
3.  **현재의 비즈니스 요구**: DevOps 및 MSA (Microservices Architecture) 환경에서 배포 속도와 품질을 동시에 확보하기 위해, 측정 가능한 품질 지표(Measurable Quality Metrics)에 대한 수요가 폭발적으로 증가함.

### 💡 비유: 초고층 빌딩의 종합 안전 진단
소프트웨어 품질 모델을 적용하는 것은 초고층 빌딩을 짓기 전에 내진 설계, 소방 시스템, 재료 강도, 편의 시설, 통신 설비 등을 종합적으로 점검하는 '건축 심사'와 같습니다. 단순히 무너지지 않는(신뢰성) 것을 넘어, 거주하기 편한(사용성)지, 화재 발생 시 확산을 막는지(보안성) 등을 기준에 따라 엄격히 따지는 과정입니다.

### 📢 섹션 요약 비유
> **마치 거대한 호텔을 짓기 위해 '건축 허가 기준'을 정하는 것과 같습니다. 단순히 방이 많은지(기능)만 볼 것이 아니라, 화재 경보시스템(보안)과 통신망(호환성)이 얼마나 튼튼한지를 미리 정의해야만 나중에 호텔이 무너지거나 손님을 잃지 않습니다.**

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 모델의 구조 (2계층 구조)
ISO 25010 모델은 크게 **사용 품질(Quality in Use)**과 **제품 품질(Product Quality)**로 구분됩니다.
- **사용 품질**: 특정 컨텍스트에서 특정 목표를 달성하기 위해 제품을 사용할 때, 사용자가 인지하는 효과성, 효율성, 만족도 등을 의미합니다.
- **제품 품질**: 소프트웨어 내부적인 정적/동적 특성으로, 8개의 주특성(Characteristics)과 31개의 부특성(Sub-characteristics)으로 세분화됩니다.

### 2. 아키텍처 다이어그램 (품질 특성 계층도)
아래 다이어그램은 ISO 25010의 핵심인 8대 제품 품질 특성과 그 하위 계층 구조를 시각화한 것입니다.

```text
+-----------------------------------------------------------------------+
|                    ISO/IEC 25010 Product Quality Model                |
+-----------------------------------------------------------------------+
|                                                                       |
|  1. Functional Suitability (기능 적합성)                              |
|     ├── Functional Completeness (기능 완전성): 필요 기능 커버 여부    |
|     ├── Functional Correctness (기능 정확성): 정확한 결과 도출 여부   |
|     └── Functional Appropriateness (기능 적절성): 명시된 목적 부합    |
|                                                                       |
|  2. Performance Efficiency (성능 효율성)                              |
|     ├── Time-behavior (시간 특성): 응답 시간, 처리 속도               |
|     ├── Resource utilization (자원 효율성): CPU/MEM/네트워크 사용량   |
|     └── Capacity (용량): 동시 사용자 수 및 데이터 처리량 한계         |
|                                                                       |
|  3. Compatibility (호환성) [9126에서 독립]                            |
|     ├── Co-existence (공존성): 동일 환경 타 소프트웨어 영향 없음      |
|     └── Interoperability (상호운용성): 타 시스템과 데이터/서비스 교환 |
|                                                                       |
|  4. Usability (사용성)                                                |
|     ├── Appropriateness Recognisability (인식 적절성)                  |
|     ├── Learnability (학습성): 사용법 습득 용이성                     |
|     ├── Operability (운영성): 조작 용이성 및 편의성                   |
|     └── User error protection (오류 방지): 실수 방지 UI               |
|                                                                       |
|  5. Reliability (신뢰성)                                              |
|     ├── Maturity (성숙도): 결함 빈도 (MTBF - Mean Time Between Failures)|
|     ├── Availability (가용성): 서비스 가동 가능 시간 비율             |
|     ├── Fault tolerance (결함 허용성): 장애 발생 시 연속 운영 능력    |
|     └── Recoverability (복구 가능성): 장애 후 데이터 복구 능력        |
|                                                                       |
|  6. Security (보안성) [9126 기능성에서 독립]                          |
|     ├── Confidentiality (기밀성): 무단 접근 방지                      |
|     ├── Integrity (무결성): 데이터 위변조 방지                         |
|     ├── Non-repudiation (부인방지): 행위 추적 및 증거                 |
|     └── Accountability (책임추적성): 주체 식별                         |
|                                                                       |
|  7. Maintainability (유지보수성)                                       |
|     ├── Modularity (모듈성): 구성 요소의 독립성                        |
|     ├── Reusability (재사용성): 타 프로젝트 활용 가능성               |
|     ├── Analyzability (분석성): 결함 원인 파악 용이성                  |
|     ├── Modifiability (수정성): 변경 영향 범위 최소화                  |
|     └── Testability (시험성): 테스트 수행 및 검증 용이성               |

|     ├── Adaptability (적응성): 다른 환경으로 변경 용이성               |

|     └── Replaceability (대체성): 동일 목적 타 소프트웨어 대체 가능성   |
```

### 3. 심층 해설 및 핵심 동작 메커니즘
위 다이어그램과 같이, ISO 25010은 소프트웨어를 하나의 유기체로 바라봅니다.
- **왜(WHY)**: '성능 효율성'은 단순히 '빠른 것'이 아니라, 제한된 자원(CPU, Memory) 내에서 최대의 처리량을 내는 '트레이드오프(Trade-off)'의 산물입니다.
- **어떻게(HOW)**: '신뢰성'의 **MTBF**와 '가용성'의 **MTTR** (Mean Time To Repair)은 반비례 관계에 있습니다. 시스템을 복구하기 쉽게 만들면(높은 유지보수성) 가용성이 높아지는 상관관계가 성립합니다.
- **연결성**: '보안성'이 강화될수록 '사용성'이 떨어질 수 있는 등, 품질 특성 간의 상충 관계(Trade-off)를 관리하는 것이 아키텍처 설계의 핵심입니다.

### 4. 핵심 지표 및 공식
- **가용성 계산 (Availability Formula)**
  $$ \text{Availability} (\%) = \frac{\text{MTBF}}{\text{MTBF} + \text{MTTR}} \times 100 $$
  - MTBF (Mean Time Between Failures): 평균 고장 간격 시간
  - MTTR (Mean Time To Repair): 평균 복구 시간

```c
// 예시: 시스템 가용성 모니터링 로직 (Pseudo-code)
double calculate_availability(double mtbf, double mttr) {
    // 핵심 지표: 시스템이 살아있는 시간의 비율
    if (mtbf <= 0 || mttr < 0) return 0.0;
    
    double uptime = mtbf;
    double total_time = mtbf + mttr;
    
    return (uptime / total_time) * 100.0; // 퍼센트 변환
}
```

### 📢 섹션 요약 비유
> **마치 자동차의 '설계 도면'을 보고 모든 부품의 명세를 확인하는 것과 같습니다. 엔진(성능), 브레이크(신뢰성), 네비게이션(기능), 연식(유지보수성), 혹은 다른 브랜드 부품 호환 여부(호환성) 등을 하나의 통합된 명세서에서 31가지 세부 항목으로 정밀 검사하는 과정입니다.**

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. ISO/IEC 9126 vs ISO/IEC 25010 심층 비교
과거 표준인 9126과 현대 표준인 25010의 차이는 단순한 업데이트가 아닌, 소프트웨어 환경의 패러다임 시프트(Paradigm Shift)를 반영합니다.

| 비교 항목 | ISO/IEC 9126 (구표준) | ISO/IEC 25010 (신표준) | 기술사적 판단 |
|:---:|:---|:---|:---|
| **보안(Security)** | 기능성(Functionality)의 **하위 특성** | **독립된 주특성**으로 격상 | 사이버 위협 증가로 보안은 선택이 아닌 필수 생존 요소가 됨. 이를 독립시켜 중요성을 부각함. |
| **호환(Compatibility)** | 상호운용성(Interoperability)만 존재 | **독립된 주특성**으로 신설 | API 경제 및 마이크로서비스(MSA) 환경에서 타 시스템과의 융합(Fusion)이 핵심이 되었음을 반영. |
| **품질 모델 구조** | 내부 품질에 집중 | 내부 품질 + **사용 품질(Quality in Use)** 강조 | 사용자 경험(UX)과 컨텍스트를 중시하는 현대 애자일(Agile) 방법론과의 정렬(Alignment). |
| **용어 정의** | 상대적으로 추상적 | 더 구체적이고 측정 가능한 정의 | CI/CD 파이프라인에서 자동화된 품질 측정을 지원하려는 의도. |

### 2. 타 과목 융합 분석 (Convergence)
ISO 25010은 소프트웨어 공학(SW Engineering)의 독자적 영역이 아니라, 타 분야와의 융합 지점입니다.

- **🔗 네트워크(Network) & 보안(Security) 융합**
  - **보안성(Security)의 기밀성(Confidentiality)**은 네트워크의 **암호화(Encryption)** 기술과 직결됩니다.
  - 예: TLS (Transport Layer Security) 프로토콜을 적용하여 전송 계층에서의 기밀성을 확보하는 것은 ISO 25010의 '기밀성'을 충족하는 네트워크적 구현 방식입니다.

- **🔗 데이터베이스(Database) 융합**
  - **무결성(Integrity)**은 DB의 **트랜잭션(Transaction) ACID 속성**, 특히 Atomicity와 Consistency와 깊이 연관됩니다.
  - **성능 효율성(Performance Efficiency)**의 자원 사용률은 DB의 **쿼리 튜닝(Query Tuning)** 및 **인덱스(Index) 전략**에 따라 결정됩니다.

- **🔗 프로젝트 관리(PM) 융합**
  - **유지보수성(Maintainability)**의 모듈성(Modularity)과 수정성(Modifiability)은 프로젝트 관리에서 **비용 산정(COCOMO Model)**에 직접적인 영향을 미치는 변수입니다. 좋은 모듈성은 LOC(Line of Code) 수정 시 발생하는 **Side Effect(부작용)**를 줄여 유지보수 비용을 절감합니다.

### 📢 섹션 요약 비유
> **마치 국가의 '법 체계'가 개헌을 통해 변화하는 것과 같습니다. 과거에는 '범죄(취약점)'를 일반적인 '사회 질서(기능성)'의 일부로 다루었지만, 현대에는 '치안(보안)'과 '외교(호환성)'를 독립된 부처(특성)로 격상하여 관리하는 만큼, 스마트폰과 인터넷이 된 세상을 반영해 규격을 대폭 개정한 셈입니다.**

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 대규모 공공기관 차세대 시스템 구축
금융 지주사의 핀테크 플랫폼 구축 시, ISO 25010을 활용하여 품질 요구사항 명세서(SRS - Software Requirements Specification)를 정의하는
