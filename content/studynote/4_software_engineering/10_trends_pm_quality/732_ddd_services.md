# 732. 도메인 서비스와 애플리케이션 서비스

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도메인 주도 설계(DDD)에서 비즈니스 로직의 위치를 정의하는 두 계층으로, 순수 비즈니스 규칙을 다루는 **도메인 서비스**와 시스템의 작업을 조정(Orchestration)하는 **애플리케이션 서비스**로 구분된다.
> 2. **역할 분담**: 특정 엔티티에 속하기 애매한 복합 로직은 **도메인 서비스**에 두고, 트랜잭션 관리, 보안, 외부 통신 등 인프라적 요소는 **애플리케이션 서비스**가 담당하여 관심사를 분리한다.
> 3. **가치**: 비즈니스 정책의 응집도를 높이고 핵심 도메인을 외부 환경으로부터 보호함으로써, 아키텍처의 유연성과 비즈니스 로직의 명확한 가시성을 확보한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "로직을 어디에 두어야 하는가?"

엔티티(Entity) 내부에 모든 로직을 넣으려다 보면, 여러 엔티티가 협력해야 하는 로직(예: 이체 처리)을 어디에 둘지 고민하게 됩니다. 그렇다고 컨트롤러에 넣으면 비즈니스 로직이 흩어져 유지보수가 힘들어집니다. DDD는 이 '로직의 거처'를 서비스라는 이름의 두 계층으로 명확히 구분하여 해결합니다.

### 💡 비유: 레스토랑의 주방장과 매니저

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        도메인 vs 애플리케이션 서비스 비유                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 도메인 서비스 (수석 주방장):                                              │
│     오직 "어떻게 맛있게 요리할 것인가(비즈니스 규칙)"만 고민함.                   │
│     - 재료들의 궁합(엔티티 간 협력)을 맞추고 최고의 맛(정책 집행)을 냄.           │
│     - 손님이 누구인지, 결제를 어떻게 했는지는 관심 없음.                         │
│                                                                             │
│  2. 애플리케이션 서비스 (홀 매니저):                                          │
│     시스템 전체의 흐름을 지휘(Orchestration)함.                               │
│     - 손님 안내(인증), 주문 접수(Input), 주방장에 전달(Logic Call),              │
│       계산(Transaction), 배달원 호출(Infrastructure)을 총괄함.                 │
│                                                                             │
│  → 주방장(도메인)은 요리만 하고, 매니저(애플리케이션)는 돌아가는 상황을 관리함!      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 서비스 계층별 특징 비교

| 항목 | 도메인 서비스 (Domain Service) | 애플리케이션 서비스 (App Service) |
|:---:|:---|:---|
| **핵심 성격** | **비즈니스 로직의 구현** | **시스템 작업의 조정 (Orchestration)** |
| **의존성** | 도메인 모델(Entity, VO)에만 의존 | 도메인, 외부 인프라, UI 계층 연결 |
| **포함 로직** | 복계 엔티티 계산, 도메인 정책 준수 확인 | 트랜잭션 처리, 보안, DTO 변환, 로깅 |
| **상태 보유** | 무상태 (Stateless) | 무상태 (Stateless) |
| **위치** | **Domain Layer (Inner)** | **Application Layer (Middle)** |

### 2. 로직의 흐름 (Data Flow)
애플리케이션 서비스는 '얇게(Thin)' 유지하고, 복잡한 지식은 도메인 서비스나 엔티티로 밀어넣는 것이 원칙입니다.

### 3. 아키텍처 구조도 (ASCII)

```text
    [ Presentation Layer ] (Controller)
           │ (Request DTO)
    ┌──────▼────────────────────────────────┐
    │      Application Service (Manager)    │ <── 트랜잭션 경계, 흐름 제어
    └──────┬────────────────┬───────────────┘
           │                │
    ┌──────▼──────┐  ┌──────▼─────────────┐
    │ Infrastructure│  │   Domain Service    │ <── 순수 비즈니스 규칙
    │ (Repository)  │  │ (Logic Specialist)  │     (여러 엔티티 조정)
    └─────────────┘  └──────┬──────┬──────┘
                            │      │
                     ┌──────▼──┐┌──▼──────┐
                     │ Entity A││ Entity B│ <── 핵심 상태와 행위
                     └─────────┘└─────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 안티 패턴: 빈약한 도메인 모델 (Anemic Domain Model)
모든 비즈니스 로직을 애플리케이션 서비스에 넣고 엔티티를 단순한 데이터 주머니(Getter/Setter)로만 쓰는 것을 말합니다. 이는 객체지향의 이점을 버리는 행위이며, 유지보수성을 극도로 떨어뜨립니다.

### 2. 기술적 시너지: 트랜잭션 관리
애플리케이션 서비스는 트랜잭션의 시작과 끝을 담당합니다. Spring의 `@Transactional` 어노테이션은 보통 애플리케이션 서비스 메서드에 붙어, 그 안에서 일어나는 여러 도메인 서비스와 리포지토리의 활동을 하나의 단위로 묶어줍니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 은행 이체(Transfer) 기능 개발
- **애플리케이션 서비스**: `transfer(from, to, amount)`
    1. 보낼/받을 사람을 DB에서 찾음.
    2. **도메인 서비스**의 `executeTransfer(fromAccount, toAccount, amount)` 호출.
    3. 성공 시 로그 남기고 알림 발송.
- **도메인 서비스**: `executeTransfer(...)`
    1. "출금 계좌 잔액 확인" 정책 체크.
    2. "두 계좌의 통화 단위 일치 여부" 체크.
    3. `fromAccount.withdraw()`와 `toAccount.deposit()` 실행.
- **판단**: DB에서 객체를 꺼내오는 행위는 애플리케이션 서비스가, 실제 '돈이 오가는 규칙'은 도메인 서비스가 담당하여 독립성 확보.

### 📢 기술사적 결언
> "도메인 서비스는 **'갈 곳 잃은 비즈니스 로직'**의 안식처다. 하지만 남용하면 다시 절차지향 코드로 돌아가게 된다. 아키텍트는 로직을 발견했을 때 가장 먼저 **'이것이 엔티티 내부에 들어갈 수 있는가?'**를 고민하고, 도저히 불가능할 때만 도메인 서비스를 만들어야 한다. 애플리케이션 서비스는 오직 전체 시스템의 **'워크플로우'**를 관리하는 지휘자 역할에 충실해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **응집도 향상**: 도메인 로직이 한곳에 모여 중복 코드 30% 이상 제거.
- **테스트 가시성**: 순수 도메인 서비스에 대한 고속 유닛 테스트 가능.

### 미래 전망
미래의 서비스 설계는 **'함수형 도메인 서비스'**와 결합할 것입니다. 부수 효과(Side-effect)가 없는 순수 함수로 도메인 로직을 정의하고, 이를 애플리케이션 서비스에서 조합하는 방식이 대세가 될 것입니다. 또한 서버리스 환경에서 각 서비스가 독립적인 실행 단위(Cloud Functions)로 배포되며, 계층 간의 통신이 더 명확해지는 '분산 서비스 아키텍처'로 진화할 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **도메인 서비스**는 주방에서 음식을 제일 맛있게 만드는 **"요리 비법"**을 아는 요리사예요.
2. **애플리케이션 서비스**는 손님의 주문을 받고, 음식을 배달하고, 계산을 도와주는 **"가게 지배인"**이죠.
3. 지배인이 요리사를 잘 도와주고, 요리사는 맛있는 요리에만 집중하면 **인기 만점 식당**이 될 수 있답니다!
