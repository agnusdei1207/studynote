# 604. 디자인 패턴(Design Patterns) 23가지 구조적 분류

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: GoF(Gang of Four) 디자인 패턴 23가지는 **"소프트웨어 설계에서 반복적으로 발생하는 문제에 대한 재사용 가능한 해결책"**으로, 범용적(Reusable)이고 검증된(Proven) Best Practice 집합이다.
> 2. **가치**: 설계 패턴을 적용하면 **코드 중복(Code Duplication)을 제거**하고, 변경에 유연한(Flexible) 구조를 확보하여 기술 부채(Technical Debt)를 선제적으로 방지할 수 있다.
> 3. **융합**: MVC(Model-View-Controller), MVVM(Model-View-ViewModel) 등 프레임워크 아키텍처의 기반이며, MSA(Microservices Architecture)의 서비스 디스커버리(Service Discovery), API Gateway 패턴 등으로 확장되고 있다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 디자인 패턴이란 무엇인가? 1977년 Christopher Alexander의 건축 패턴(Town Building Pattern) 이론을 소프트웨어에 적용한 것이다. 1994년 Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides(GoF)가 "Design Patterns: Elements of Reusable Object-Oriented Software"에서 23가지 패턴을 정립했다. 패턴은 **"Problem(문제) → Solution(해결책) → Consequence(결과)"**의 3요소로 구성되며, 특정 언어나 프레임워크에 종속되지 않는 범용적 설계 지침이다.

- **💡 비유**: 디자인 패턴은 **"요리 레시피(Recipe)"**와 같습니다. 수많은 요리사가 시행착오를 통해 겪은 문제(예: "마늘을 너무 일찍 넣으면 탄다")와 그 해결책("마늘은 나중에 넣어라")을 정리해둔 셈입니다. 이 레시피를 알면, 초보 요리사도 검증된 맛을 내고 실수를 피할 수 있죠. 패턴을 맹목적으로 적용하면 안 되듯, 재료와 상황에 따라 적절히 변형해야 한다.

- **등장 배경**:
    1. **Smalltalk 시대의 경험 정립**: 1980년대 후반 OOP 커뮤니티에서 반복되는 설계 문제와 해결책을 정리.
    2. **재사용성의 한계 돌파**: 클래스 라이브러리(Code Reuse)에서 설계(Design) 수준의 재사용으로 패러다임 전환.
    3. **커뮤니티 지식 공유**: 시행착오(Know-How)를 공유 패턴(Pattern Language)으로 체계화.

- **📢 섹션 요약 비유**: 도면(Plan)이 있는 건물은 무너지지 않듯, 디자인 패턴은 "시행착오로 얻은 도면"을 공유하여, 같은 실수를 반복하지 않고 검증된 설계를 빠르게 적용하게 하는 지식 공유의 도구입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 디자인 패턴 분류 체계

GoF 23가지 패턴은 **목적(Purpose)**과 **범위(Scope)**에 따라 분류된다. 목적은 생성(Creational), 구조(Structural), 행위(Behavioral)로 나뉘고, 범위는 클래스(Class)와 객체(Object)로 나뉜다.

| 분류 | 생성(Creational) | 구조(Structural) | 행위(Behavioral) |
|:---|:---|:---|:---|
| **클래스(Class)** | Factory Method | Adapter(Class) | Interpreter, Template Method |
| **객체(Object)** | Abstract Factory, Builder, Prototype, Singleton | Adapter(Object), Bridge, Composite, Decorator, Facade, Flyweight, Proxy | Chain of Responsibility, Command, Iterator, Mediator, Memento, Observer, State, Strategy, Visitor |

#### [대표적 패턴 3가지 심층 분석]

**1. Singleton Pattern (싱글톤 패턴)**

싱글톤은 애플리케이션 전체에서 인스턴스가 하나만 존재함을 보장한다. 데이터베이스 연결 풀(Connection Pool), 로거(Logger), 설정 관리자(Configuration Manager) 등에 사용된다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Singleton Pattern                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [문제]                                                                    │
│  - 데이터베이스 연결이 여러 곳에서 생성되면 리소스 누수 발생                 │
│  - 전역 설정을 여러 객체가 관리하면 일관성 깨짐                            │
│                                                                             │
│  [해결책]                                                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ public class DatabaseConnection {                                  │   │
│  │     private static DatabaseConnection instance;                   │   │
│  │     private DatabaseConnection() {}  // private 생성자              │   │
│  │                                                                  │   │
│  │     public static synchronized DatabaseConnection getInstance() { │   │
│  │         if (instance == null) {                                  │   │
│  │             instance = new DatabaseConnection();                  │   │
│  │         }                                                        │   │
│  │         return instance;                                          │   │
│  │     }                                                            │   │
│  │ }                                                                │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [문제점]                                                                  │
│  - 단위 테스트 시 Mock 대체 어려움 (Global State)                           │
│  - 멀티스레드 환경에서 DCL(Double-Checked Locking) 필요                      │
│  - 의존성 주입(DI)이 어려워 테스트 용이성 저하                              │
│                                                                             │
│  [대안]                                                                    │
│  - DI 컨테이너(Spring, Guice 등)의 싱글톤 스코프 활용                         │
│  - 필요시 상태 비저장(Stateless) 서비스로 설계                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

**2. Strategy Pattern (전략 패턴)**

전략 패턴은 알고리즘군을 정의하고, 각각을 캡슐화한 다음 교체 가능하게 만든다. 정렬 알고리즘, 결제 방식, 할인 정책 등 실행 시점에 알고리즘을 선택해야 할 때 유용하다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Strategy Pattern                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [문제]                                                                    │
│  - 결제 방식(카드, 간편, 계좌이체)마다 if-else 분기로 코드 복잡화          │
│  - 새로운 결제 수단 추가 시 기존 코드 수정 필요 (OCP 위배)                   │
│                                                                             │
│  [해결책]                                                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                 │   │
│  │     <<interface>> PaymentStrategy                                  │   │
│  │     ┌─────────────────────────────────────────────────────────┐    │   │
│  │     │ + pay(amount: int): void                                │    │   │
│  │     └───────────────────────────────▲───────────────────────────┘    │   │
│  │                                     │ implements                   │   │
│  │     ┌──────────────┬───────────────┼───────────────┬───────────┐   │   │
│  │     ▼              ▼               ▼               ▼           │   │   │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐        │   │   │
│  │  │CreditCard│  │KakaoPay  │  │NaverPay  │  │BankTransfer│      │   │   │
│  │  │Strategy  │  │Strategy  │  │Strategy  │  │Strategy    │      │   │   │
│  │  └──────────┘  └──────────┘  └──────────┘  └──────────┘        │   │   │
│  │                                                                 │   │
│  │     ┌─────────────────────────────────────────────────────────┐    │   │
│  │     │ ShoppingCart                                           │    │   │
│  │     │ ┌─────────────────────────────────────────────────┐   │    │   │
│  │     │ │ - paymentStrategy: PaymentStrategy                │   │    │   │
│  │     │ │ + checkout(amount: int) {                          │   │    │   │
│  │     │ │     paymentStrategy.pay(amount);  // 다형성 호출   │   │        │
│  │     │ │ }                                                   │   │    │   │
│  │     │ └─────────────────────────────────────────────────┘   │    │   │
│  │     └──────────────────────────────────────────────────────────┘    │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [장점]                                                                    │
│  - OCP 준수: 새 결제 수단 추가 시 기존 코드 수정 불필요                    │
│  - 단위 테스트 용이: Mock Strategy로 쉽게 대체                             │
│  - 런타임 알고리즘 교체 가능                                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

**3. Observer Pattern (옵저버 패턴)**

옵저버 패턴은 한 객체의 상태가 바뀌면 그 객체에 의존하는 다른 객체들에게 자동으로 알림을 보내고 내용을 갱신하는 일대다다(one-to-many) 의존성을 정의한다. 이벤트 처리(Event Handling), MVC 패턴의 Model-View 연동, PUB/SUB 시스템의 기반이다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Observer Pattern                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [문제]                                                                    │
│  - 주식 가격 변경 시 여러 화면(그래프, 리스트, 알림)에 동시 반영 필요        │
│  - 가격 객체가 화면들을 직접 호출하면 결합도(Coupling) 증가                   │
│                                                                             │
│  [해결책]                                                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │     ┌───────────────────┐    notifies        ┌───────────────────┐   │   │
│  │     │    Subject        │─────────────────>│   Observer        │   │   │
│  │     │   (StockPrice)    │                   │  (UI Components)  │   │   │
│  │     └───────────────────┘                    └───────────────────┘   │   │
│  │              △                                        △              │   │
│  │              │ implements                              │           │   │
│  │     ┌────────┴────────┐                       ┌────────┴────────┐   │   │
│  │     │ ConcreteSubject │                       │ConcreteObserver │   │   │
│  │     │                 │                       │                 │   │   │
│  │     │ - observers[]   │◀────register()────────│                 │   │   │
│  │     │ + attach()      │                       │ + update()      │   │   │
│  │     │ + detach()      │                       │                 │   │   │
│  │     │ + notify()      │───────────────────────│                 │   │   │
│  │     └─────────────────┘                       └─────────────────┘   │   │
│  │                                                                 │   │
│  │  [동작 순서]                                                             │   │
│  │  1. StockPrice.setPrice(100) → 주가 변경                              │   │
│  │  2. notify() 호출 → 등록된 모든 Observer에게 알림                      │   │
│  │  3. 각 Observer.update() 실행 → 화면 갱신                             │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [실무 적용]                                                               │
│  - UI 이벤트 리스너(ClickListener, KeyListener)                            │
│  - React/Vue의 상태 관리(Reactive State)                                   │
│  - Kafka, RabbitMQ의 PUB/SUB 메시징                                        │
└─────────────────────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 디자인 패턴은 목수(木手)들이 전수해 오는 대목장의 기술처럼, 시행착오를 통해 검증된 "최적의 설계 묘목"을 공유하는 지식 전달의 도구입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 생성 패턴 비교 (Factory vs Builder vs Abstract Factory)

| 비교 항목 | Factory Method | Builder | Abstract Factory |
|:---|:---|:---|:---|
| **목적** | 객체 생성을 서브클래스로 위임 | 복잡한 객체 단계적 생성 | 관련 객체군 생성 |
| **사용 시점** | 생성 타입이 미리 확정 | 생성 파라미터가 많고 복잡 | 제품군이 여러 family |
| **예시** | DAO, Repository 생성 | SQL Builder, HTML 생성 | DB Driver(GUI/Console) |
| **비유** | 공장장이 생산라인 지정 | 자동차 조립 라인 | 부품군 공급망 |

#### 2. 구조 패턴 비교 (Decorator vs Proxy vs Adapter)

```text

│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │    Coffee ────(decorate)───> Milk ───(decorate)───> Sugar           │   │
│  │       │                                │                            │   │
│  │       ▼                                ▼                            │   │
│  │   $2 + $0.5                         $2 + $0.5 + $0.3 = $2.8      │   │
│  │                                                                     │   │
│  │  ✅ 상속 없이 객체에 책임(기능) 추가                                │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [Proxy Pattern] - 접근 제어                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │    Client ── request ──> Proxy ── check access ──> RealSubject     │   │
│  │                           │                        △                   │   │
│  │                           │(lazy load, cache, access control)        │   │
│  │                           └────────────────────────────────           │   │
│  │                                                                     │   │
│  │  ✅ 대상 객체에 대한 간접 접근 제어                                  │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [Adapter Pattern] - 인터페이스 변환                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │    Client ── request() ──> Adapter ── specificRequest() ──> Adaptee│   │
│  │                                 │                            △        │   │
│  │                                 │(interface mismatch 변환)              │   │
│  │                                 └────────────────────────────────        │   │
│  │                                                                     │   │
│  │  ✅ 호환되지 않는 인터페이스를 변환                                  │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 3. 과목 융합 관점

- **컴퓨터 구조(CA)**: 컴포지트(Composite) 패턴은 파일 시스템(File System) 디렉토리 구조, GUI 위젯 트리(Window-Frame-Button) 등 계층적 구조를 표현하는 데 사용된다. 반복자(Iterator) 패턴은 컨테이너의 내부 구조를 노출하지 않고 순회하는 데 사용되며, CPU의 명령어 파이프라인 설계에도 영감을 주었다.

- **운영체제(OS)**: 옵저버(Observer) 패턴은 이벤트 드리븐(Event-Driven) OS의 핵심이다. 파일 시스템 감시(File Watcher), 프로세스 시그널(Signal Handling), 인터럽트 핸들러(Interrupt Handler) 등 이벤트 기반 시스템 전반에 적용된다.

- **📢 섹션 요약 비유**: 식탁(Proxy)가 음식을 대신 전달해주듯, 장식가(Decorator)가 케이크에 크림을 추가하듯, 어댑터(Adapter)가 110V 전기를 220V로 변환하듯, 패턴은 각각 "대리", "추가", "변환"이라는 명확한 역할을 수행합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오 1: 할인 정책 시스템에 전략 패턴 적용**
    - **문제**: VIP, 회원, 신규 회원별 할인 로직이 거대한 switch 문으로 구현되어, 새로운 등급 추가 시 기존 코드 수정 불가피.
    - **의사결정**: DiscountStrategy 인터페이스 도입 후 각 등급별 전략 클래스 분리.
    - **결과**: 새 등급 추가 시 기존 테스트 케이스 재실행 불필요, 신규 전략 클래스만 추가하여 OCP 준수.

- **실무 시나리오 2: 로깅 시스템에 옵저버 패턴 적용**
    - **문제**: 로그를 콘솔, 파일, DB, 외부 SIEM에 동시에 기록해야 하는데, 각 대상마다 직접 호출하여 결합도 증가.
    - **의사결정**: Logger Subject에 다수 LogObserver를 등록하고, 로그 발생 시 notify()로 자동 전파.
    - **결과**: 새로운 로그 대상(Slack, Datadog 등) 추가 시 Logger 코드 수정 불필요, Observer만 추가.

- **도입 체크리스트**:
    1. **패턴 적용 타당성**: 정말 이 패턴이 필요한가? (YAGNI: You Aren't Gonna Need It 원칙 준수)
    2. **과잉 설계(Over-Engineering)**: 3개 미만의 구체 클래스가 예상될 때도 패턴을 도입하는가?
    3. **팀 동료 학습 곡선**: 패턴이 팀의 보통 개발자에게 명확하게 이해되는가?
    4. **프레임워스 지원**: Spring, Guice, React 등 프레임워크가 이미 구현한 패턴을 활용하는가?

- **안티패턴**:
    - **Pattern Hatching**: 문제를 명확히 파악하지 않고 패턴을 먼저 적용하려는 시도.
    - **Golden Hammer**: 모든 문제에 싱글톤(Singleton) 패턴만 적용.
    - **죽은 추상화(Zombie Abstraction)**: 구체 클래스가 하나뿐데도 인터페이스를 만드는 오버 엔지니어링.

- **📢 섹션 요약 비유**: 비수가 망치질만 하면 못 만드는 가구가 없듯, 설계자는 패턴을 망치처럼 다룰 줄 알아야 합니다. 하지만 모든 것이 못으로 보이는 망치持有病(Hammer Syndrome)은 경계해야 합니다.
