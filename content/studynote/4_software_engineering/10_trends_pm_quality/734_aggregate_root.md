# 734. 팩토리와 애그리게이트 루트

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도메인 주도 설계(DDD)에서 연관된 객체들을 하나로 묶어 데이터 변경의 단위로 삼는 **애그리게이트(Aggregate)**와, 그 묶음의 유일한 진입점인 **애그리게이트 루트(Aggregate Root)**를 통해 데이터의 일관성(Invariants)을 유지한다.
> 2. **생성 전략**: 복잡한 애그리게이트를 생성하는 로직이 도메인 모델 내부에 섞이지 않도록 **팩토리(Factory)**를 사용하여 객체 생성의 책임을 분리하고 캡슐화를 강화한다.
> 3. **가치**: 외부 객체가 애그리게이트 내부 객체를 직접 수정하는 것을 금지함으로써 비즈니스 규칙을 강제하고, 트랜잭션의 범위를 명확히 하여 데이터 무결성을 보장한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "복잡한 객체망 속에서 정합성 지키기"

객체지향 설계에서 객체들은 서로 복잡하게 연결되어 있습니다. '주문' 객체 안에는 '주문 항목', '배송 정보' 등이 들어있죠. 만약 외부에서 '주문 항목'의 가격을 마음대로 바꿔버린다면, '주문' 전체의 총액 정보가 틀어지게 됩니다. **애그리게이트**는 이러한 '정합성이 깨지지 않아야 할 경계'를 긋는 기술입니다.

### 💡 비유: 여행용 캐리어와 팩토리

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        애그리게이트 및 팩토리 비유                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 애그리게이트 (캐리어 묶음):                                               │
│     캐리어(Root) 안에는 옷, 세면도구, 신발(Internal Objects)이 들어있음.         │
│     - 규칙: 짐을 넣거나 빼려면 반드시 캐리어 지퍼(Root)를 열어야 함.              │
│     - 효과: 가방만 챙기면(트랜잭션) 안의 물건들은 자동으로 다 따라옴.              │
│                                                                             │
│  2. 팩토리 (여행 준비 대행사):                                                │
│     내가 일일이 짐을 싸기 힘들 때, 대행사(Factory)에게 "여름휴가 짐 싸줘"라고      │
│     말하면 완벽하게 챙겨진 캐리어(Aggregate)를 통째로 만들어줌.                  │
│                                                                             │
│  → 복잡한 짐 싸기(생성)는 **팩토리**에게, 짐 관리(수정)는 **루트**에게!            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 애그리게이트 루트 (Aggregate Root)의 원칙
- **유일한 통로**: 외부 객체는 루트 엔티티만 참조할 수 있고, 내부 객체에는 직접 접근할 수 없음.
- **식별자 소유**: 루트 엔티티만 전역 식별자(ID)를 가지며, 내부 객체는 로컬 식별자만 가짐.
- **불변식(Invariant) 강제**: 데이터가 변경될 때 루트가 모든 비즈니스 규칙을 검사하여 일관성 유지.

### 2. 팩토리 (Factory)의 역할
- **복잡성 은닉**: 여러 엔티티와 VO를 조합하여 애그리게이트를 생성하는 복잡한 로직 캡슐화.
- **유효성 검사**: 객체가 생성되는 시점에 이미 완벽한 상태(Valid State)임을 보장.

### 3. 구조적 관계도 (ASCII)

```text
    [ Client ] ──▶ [ Order Factory ]
                        │ (Create)
                        ▼
    ┌────────────────── [ Order Aggregate Boundary ] ──────────────────┐
    │                                                                 │
    │      ┌─────────────────────────────┐                            │
    │      │   Order (Aggregate Root)    │ ◀─── [ 외부에서 참조 가능 ]   │
    │      ├─────────────────────────────┤                            │
    │      │ - id: UUID                  │                            │
    │      │ + addOrderItem(item)        │                            │
    │      └──────────────┬──────────────┘                            │
    │                     │ (Owner)                                   │
    │      ┌──────────────▼──────────────┐                            │
    │      │      OrderItem (Internal)   │ ◀─── [ 외부 참조 금지 ]       │
    │      └─────────────────────────────┘                            │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 애그리게이트 설계 규칙 (Vaughn Vernon)
- **작은 애그리게이트**: 메모리 효율과 동시성을 위해 가능한 한 작게 설계.
- **ID로 참조**: 애그리게이트 간에는 객체 직접 참조 대신 ID를 통해 참조하여 결합도 분리.
- **결과적 일관성**: 다른 애그리게이트와의 데이터 정합성은 도메인 이벤트를 통해 비동기로 맞춤.

### 2. 기술적 시너지: 리포지토리 (Repository)
리포지토리는 반드시 **애그리게이트 루트 단위**로 존재해야 합니다. `OrderItemRepository`는 존재해서는 안 되며, 오직 `OrderRepository`를 통해서만 전체 묶음을 저장하고 조회해야 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 쇼핑몰 주문 취소 로직
- **상황**: 주문 취소 시, '배송 시작' 전인지 확인하고 사용한 쿠폰을 복구해야 함.
- **결단**: **루트 엔티티(Order)에 응집된 로직 구현**. 
- **판단**: 외부 서비스에서 직접 `coupon.restore()`를 호출하지 않음. `order.cancel()` 메서드 하나만 호출하면, Order 루트가 상태를 확인한 뒤 내부의 쿠폰과 상태를 모두 스스로 업데이트.
- **효과**: 비즈니스 규칙이 흩어지지 않고 주문이라는 하나의 트랜잭션 안에서 완벽하게 처리됨.

### 📢 기술사적 결언
> "애그리게이트는 **'데이터 무결성의 방어벽'**이다. 시스템이 거대한 데이터 덩어리가 되지 않도록 칼로 자르듯 경계를 긋는 작업이다. 아키텍트는 '기술적으로 연결하기 쉬운가'가 아니라 **'비즈니스적으로 함께 변해야 하는가'**를 기준으로 경계를 정해야 한다. 팩토리를 통해 깨끗하게 탄생하고, 루트를 통해 올바르게 성장하는 객체 모델이 장수하는 시스템의 비결이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **데이터 정합성**: 동시 수정으로 인한 데이터 꼬임 현상 80% 이상 방지.
- **코드 가독성**: 시스템의 진입점이 명확해져 로직 탐색 시간 단축.

### 미래 전망
미래의 애그리게이트 설계는 **'분산 트랜잭션 자동화'**와 결합할 것입니다. 서비스가 쪼개진 환경에서도 여러 애그리게이트 간의 일관성을 AI 기반의 코디네이터가 자동으로 관리해 주는 기술이 등장할 것입니다. 또한 **이벤트 소싱**과 결합하여, 애그리게이트의 모든 상태 변화를 이벤트 스트림으로 관리하는 '불변 아키텍처'가 고신뢰성 시스템의 표준이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **애그리게이트**는 장난감을 잃어버리지 않게 **"세트로 묶어둔 주머니"**와 같아요.
2. 주머니 입구(루트)는 딱 하나뿐이라서, 주머니 속 물건을 만지려면 **무조건 입구로 손을 넣어야** 하죠.
3. **팩토리**는 처음부터 장난감 세트를 **빠진 것 없이 완벽하게 챙겨주는** 친절한 로봇 팔이랍니다!
