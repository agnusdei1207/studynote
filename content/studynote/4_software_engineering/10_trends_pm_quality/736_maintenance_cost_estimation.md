# 736. 소프트웨어 유지보수 비용 산정 기법

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 인도 후 발생하는 수정, 보완, 적응 및 예방 활동에 필요한 비용을 정량적으로 산출하는 기법으로, 초기 개발 비용보다 훨씬 큰 비중(전체 TCO의 60~80%)을 차지하는 장기적 경제 지표다.
> 2. **산정 모델**: 개발 당시의 규모(LOC, FP)를 기준으로 유지보수 난이도 계수를 곱하는 **경험적 모델**과, 실제 투입되는 인력의 등급별 단가를 적용하는 **투입 공수(Man-Month) 방식**이 병행된다.
> 3. **가치**: 지속 가능한 운영을 위한 예산을 확보하고, '기술 부채'로 인한 비용 증가를 가시화하여 합리적인 시스템 개편(Re-engineering) 시점을 결정하는 근거를 제공한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "배보다 배꼽이 더 큰 유지보수"

소프트웨어는 살아있는 생명체와 같아서 배포 후에도 환경 변화(OS 업데이트), 법규 개정, 사용자 요구에 따라 계속 변해야 합니다. 통계에 따르면 소프트웨어 생애 주기 전체 비용 중 개발비는 20~30%에 불과하며, 나머지 70% 이상이 유지보수에 투입됩니다. 따라서 정확한 유지보수 비용 산정은 기업 IT 거버넌스의 핵심입니다.

### 💡 비유: 자동차 구입비와 유지비

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        유지보수 비용 산정 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 새 차(소프트웨어)를 한 대 샀음.                                         │
│                                                                             │
│  1. 구입 비용 (개발비): 차를 살 때 낸 일시불 가격.                             │
│                                                                             │
│  2. 유지보수 비용:                                                           │
│     - 기름값 (운영): 매일 타고 다니는 데 드는 비용.                             │
│     - 소모품 교체 (수정/보완): 타이어가 닳거나 엔진오일을 갈 때 드는 비용.         │
│     - 튜닝 (적응): 새로운 환경(비포장도로)에 맞게 차를 고치는 비용.               │
│                                                                             │
│  → 차 값보다 수년간 들어가는 **기름값과 수리비**가 훨씬 클 수 있다는 사실!        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 유지보수 비용 산정의 주요 기법

| 기법 명칭 | 산정 원리 | 장단점 |
|:---:|:---|:---|
| **개발비 비율법** | 초기 개발비의 일정 비율(예: 10~15%)을 산정 | 가장 단순하지만 시스템 노후화 반영 불가 |
| **기능점수(FP)법** | 유지보수 대상 기능의 개수와 복잡도 기반 산정 | **국내 공공사업 표준**, 객관성 높음 |
| **벨라디-레만 모델** | $M = p + K \times e^{(c-d)}$ (경험적 수식) | 복잡한 요인 반영 가능, 계산 어려움 |
| **투입공수(MM)법** | 투입되는 인력의 등급과 기간으로 직접 계산 | 실제 지출액과 일치, 효율성 측정 불가 |

### 2. 비용 영향 요인 (Cost Drivers)
- **소프트웨어 특성**: 복잡도, 모듈화 정도, 문서화 수준.
- **인적 특성**: 유지보수 팀의 숙련도, 도메인 이해도.
- **환경 특성**: 하드웨어 가변성, 운영체제 안정성.

### 3. 유지보수 비용의 구성 (ASCII)

```text
    [ Total Maintenance Cost ]
           │
    ┌──────┴────────────────────────────────────────────────────┐
    │                                                           │
    ▼ [ 직접비 ]                                                ▼ [ 간접비 ]
    - 인건비 (Engineer Salary)                                  - 인프라 사용료

    - 기술료 (Technology Fee)                                   - 라이선스 갱신료
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 유지보수 4대 유형별 비용 비중
- **수정(Corrective)**: 버그 수정 (20%)
- **적응(Adaptive)**: 환경 변화 대응 (25%)
- **완전(Perfective)**: 신규 기능 추가, 성능 개선 (50% - **가장 큼**)
- **예방(Preventive)**: 리팩토링, 잠재 결함 제거 (5%)

### 2. 기술적 시너지: 클린 코드와 TDD
초기 개발 단계에서 클린 코드와 테스트 자동화를 적용하면 유지보수 단계의 **'분석 및 테스트 시간'**이 획기적으로 줄어듭니다. 이는 유지보수 단가를 낮추고 시스템의 수명을 늘리는 가장 경제적인 투자입니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 유지보수 계약 방식 변경
- **상황**: 매년 '정액제(개발비 %)'로 계약하다 보니, 신규 기능 요구가 늘어날수록 개발사가 적자를 보게 됨.
- **결단**: **기능점수(FP) 기반 사후 정산 도입**. 
- **판단**: 기본 운영비는 고정으로 하되, 고도화(Perfective) 작업은 실제 구현된 FP에 단가를 곱해 별도 청구하는 하이브리드 방식 제안. 
- **효과**: 발주처는 쓴 만큼 내고, 수주처는 일한 만큼 받는 '합리적 비용 상생 모델' 구축.

### 📢 기술사적 결언
> "유지보수 비용은 단순한 소모값이 아니라 **'시스템의 건강 진단서'**다. 시간이 지날수록 비용이 가파르게 상승한다면 그것은 아키텍처가 부패하고 있다는 강력한 증거다. 아키텍트는 유지보수 예산의 일부를 반드시 **'예방적 유지보수(리팩토링)'**에 할당하도록 설득하여, 장기적인 TCO(총 소유 비용)를 관리하는 경제적 안목을 가져야 한다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **예산 정확도**: 데이터 기반 산출로 예산 부족이나 낭비 30% 이상 방지.
- **ROI 최적화**: 유지보수 비용 임계치 초과 시 재개발(Modernization) 의사결정 기준 제공.

### 미래 전망
미래의 유지보수 비용 산정은 **'실시간 로그 분석'**과 결합될 것입니다. 개발자가 코드를 수정하는 패턴과 시간을 AI가 실시간 추적하여, 특정 모듈의 유지보수 단가를 자동으로 계산해 주는 '지능형 비용 모니터링'이 도입될 것입니다. 또한 서버리스 환경의 확산으로 인프라 운영비가 0에 수렴하면서, 비용의 100%가 '지적 자산(코드)의 진화'에만 집중되는 고효율 구조로 변모할 것입니다.
