# 759. QFD 품질 기능 전개 요구사항 변환 기법

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사용자의 주관적 요구(VOC, Voice of Customer)를 정량적 기술 특성(Quality Characteristics)으로 변환하여 설계 단계부터 품질을 보증하는 **"품질의 통역사"** 시스템이자 체계적인 의사결정 프레임워크이다.
> 2. **구조**: 핵심 도구인 **HOQ (House of Quality)** 매트릭스를 통해 무엇(What)을 어떻게(How)로 구현할지, 그리고 기술 요소 간의 상충(Trade-off) 관계를 시각화하여 설계 결함을 사전에 차단한다.
> 3. **파급**: 단순한 요구사항 분석을 넘어, 제품 기획에서 부품 설계, 공정 관리, 생산 지침까지 품질 정보를 연쇄적으로 전개(Cascading)하여 설계 변경 비용을 획기적으로 절감하고 고객 만족도(CSAT)를 극대화한다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 정의
**QFD (Quality Function Deployment)**는 고객의 요구사항을 제품 개발의 각 단계(기획, 설계, 생산, 서비스)에 반영하여 품질을 보증하는 방법론입니다. 1972년 일본 미쓰비시 중공업의 고바야시 신지가 처음 적용하였고, 이후 아카오 요지 교수에 의해 체계화되었습니다. 단순히 "고객이 원하는 것을 만든다"는 추상적인 목표를 달성하기 위해, 고객의 언어(감성적, 질적)를 엔지니어의 언어(물리적, 양적)로 정밀하게 변환하는 **"Translation Mechanism"**이 핵심 기술입니다.

#### 2. 등장 배경 및 철학
1970년대 이전의 품질 관리는 주로 제조 단계에서의 검사(Inspection)에 의존했습니다. 그러나 소프트웨어나 정밀 기계와 같이 복잡한 시스템에서는 설계 결함이 발생하면 이를 수정하는 비용이 천문학적으로 증가합니다(1:10:100 법칙). QFD는 이러한 문제를 해결하기 위해 **"설계가 결정되면 품질은 결정된다"**는 철학 아래, 설계 이전 단계에서 품질을 예방적으로 확보하는 것을 목표로 합니다.

#### 3. HOQ (House of Quality) 구조도
QFD의 핵심인 '품질의 집'은 고객 요구사항과 기술적 특성을 2차원 매트릭스로 연결합니다.

```text
       [ 왜? (WHY) ]           [ 얼마나? (HOW MUCH) ]
   ┌───────────────────┐      ┌───────────────────────┐
   │ 경쟁사 분석 (Competitor) │      │ 고객 만족도       │
   │ 평가 및 벤치마킹       │      │ (우리/경쟁사)      │
   └─────────────────────────────────────────────────┘
                     ▲
                     │   관계(Relationship) 강도  │
    ─────────────────┼─────────────────────────────│─────
    고 객 │ VOC       │   [ 기술적 특성 (How) ]      │
    요 구 │ (What)    │   T1      T2      T3      │  ──▶ 우선순위
    -------+-----------+---------------------------------
       A   │  ● 강     │  ○      ◎       △      │  목표치
       B   │  ○ 중     │          ●               │  (Target)
       C   │  △ 약     │  △               ○      │  (Spec)
    -------------------------------------------------
       ▲            ▲  ▲                ▲
       │            │  │                │
   [중요도]    [상관관계]         [난이도/기술적 벤치마킹]
            (CORRELATION)           (Tech Benchmark)

    ◎: 강한 양의 상관  ●: 양의 상관  ○: 약한 양의 상관
    △: 약한 음의 상관  ×: 강한 음의 상관
```

#### 4. 상세 해설
1.  **What (고객 요구사항)**: 고객이 바라는 "무엇"입니다. 예: "배터리가 오래가야 한다", "가벼워야 한다".
2.  **How (기술적 특성)**: 엔지니어가 "어떻게" 구현할 것인가입니다. 예: "배터리 용량(mAh)", "소비 전력(W)", "케이스 재료".
3.  **관계 매트릭스(Relationship Matrix)**: What과 How가 얼마나 밀접한지를 나타냅니다. 예: '배터리 용량'은 '배터리 수명'과 강한 양의 관계(◎)를 가집니다.
4.  **지붕(Correlation Matrix)**: 기술적 특성 간의 상충 관계를 보여줍니다. 예: '고해상도 디스플레이'와 '배터리 수명'은 음의 상관관계(Trade-off)를 가질 수 있습니다.

> **📢 섹션 요약 비유:**
> 마치 고급 레스토랑에서 **식당의 테마(분위기)**는 고객 요구사항(VOC)이고, **레시피의 각 재료 비율(소금, 설탕, 조리 온도)**은 기술적 특성(HOW)인 것과 같습니다. 요리사(아키텍트)는 이 HOQ 차트를 보고, "이 요리는 짭짤하다"는 고객의 말을 정확히 '소금 2g, 후추 1g'이라는 엔지니어링 규격으로 번역하여 매번 같은 맛을 내도록 보장합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 4단계 전개 모델 (The Four Phases)
QFD는 단순히 한 번의 매트릭스 작성으로 끝나지 않으며, 제품이 만들어지는 과정 전체를 따라 흐르는 폭포수(Waterfall) 형태의 정보 전개 과정을 거칩니다.

```text
┌───────────────────────────────────────────────────────────────────────────────┐
│                            QFD 전개 프로세스 (Cascading)                      │
├───────────────────────────────────────────────────────────────────────────────┤
│                                                                               │
│  1. 제품 기획 PHASE (Product Planning)                                        │
│     VOC (고객 요구사항)  ──▶  제품 기술적 특성 (Product Specs)                │
│     (e.g., "빨리 응답한다")    (e.g., "CPU 3GHz", "Memory 8GB")                │
│             │                                                                  │
│             ▼                                                                  │
│  2. 부품/구성 전개 PHASE (Component Deployment)                                │
│     제품 기술적 특성   ──▶  핵심 부품/서비스 특성 (Component Specs)             │
│     (e.g., "CPU 3GHz")     (e.g., "Core i5 설계", "Cache 16MB")                │
│             │                                                                  │
│             ▼                                                                  │
│  3. 공정/개발 프로세스 PHASE (Process Planning)                                │
│     핵심 부품 특성    ──▶  개발 공정/빌드 프로세스 (Process Key Params)         │
│     (e.g., "Cache")        (e.g., "단위 테스트 항목", "Compiler 최적화 옵션")   │
│             │                                                                  │
│             ▼                                                                  │
│  4. 생산/운영 관리 PHASE (Production/Operation Planning)                       │
│     개발 공정 파라미터 ──▶  운영 매뉴얼/모니터링 (Production Controls)          │
│     (e.g., "Compiler Opt")  (e.g., "배포 가이드라인", "모니터링 Agent 설정")     │
│                                                                               │
└───────────────────────────────────────────────────────────────────────────────┘
```

#### 2. HOQ 구성 요소 심층 분석 (상세 표)
품질의 집을 구성하는 주요 요소와 그 내부 동작 메커니즘은 다음과 같습니다.

| 구성 요소 | 영문 명칭 | 역할 및 내부 동작 | 주요 입력/출력 |
|:---:|:---|:---|:---|
| **① 고객 요구** | **Customer Requirements (Whats)** | 고객 관점에서의 요구사항. 설문조사나 인터뷰를 통해 도출하며 각 항목별 중요도(1~5점)를 부여함. | 입력: 설문조사 / 출력: 우선순위 목록 |
| **② 기술 특성** | **Engineering Specs (Hows)** | 엔지니어 관점에서 What을 충족시키기 위한 정량적 측정 가능한 특성치(단위 필수). | 입력: 기술 데이터 / 출력: 설계 목표치 |
| **③ 관계 행렬** | **Relationship Matrix** | What과 How의 관계성(强/中/弱)을 점수화(9/3/1)하여, How의 절대 중요도를 계산하는 영역. | 입력: 관계 점수 / 출력: 기술적 가중치 |
| **④ 상관 매트릭스** | **Correlation Roof** | 기술 특성(How) 간의 시너지(+) 또는 상충(-) 관계를 정의. 성능 향상 vs 비용 증가 등의 트레이드오프 분석에 활용. | 출력: 최적화 조건 |
| **⑤ 경쟁사 분석** | **Competitive Benchmarking** | 우리 기술과 경쟁사 수준을 비교(Salient Scale). 기술적 격차를 시각화하여 개선 방향을 설정. | 입력: 시장 조사 데이터 |
| **⑥ 목표치 설정** | **Target Values** | 계산된 기술적 가중치와 현실적 제약(기술력, 비용)을 고려하여 최종 설계 목표치를 결정. | 출력: 최종 설계 스펙 |

#### 3. 핵심 알고리즘: 기술적 우선순위 계산 (수식 포함)
QFD의 진정한 힘은 단순한 도식화가 아니라 **정량적 의사결정**에 있습니다. 기술적 중요도(Technical Importance, $T_j$)를 산출하는 알고리즘은 다음과 같습니다.

**[Step 1] 관계 행렬 점수화**
- **◎ (강함)**: 9점
- **○ (보통)**: 3점
- **△ (약함)**: 1점
- **(공백)**: 0점

**[Step 2] 중요도 가중 계산 (Weighted Importance)**
각 기술 특성($j$)의 가중 합($W_j$)을 계산합니다. $R_{ij}$는 $i$번째 고객 요구와 $j$번째 기술 특성 간의 관계 점수이며, $C_i$는 $i$번째 고객 요구의 중요도입니다.

$$ W_j = \sum_{i=1}^{n} (R_{ij} \times C_i) $$

**[Step 3] 정규화 및 우선순위 산출**
최종 기술적 우선순위($T_j$)는 $W_j$를 정규화하여 도출하며, 이를 바탕으로 리소스를 배분합니다.

> **📢 섹션 요약 비유:**
> 마치 **"자동차의 설계 도면을 제작하는 공정"**과 같습니다. 자동차가 "빨라야 한다"는 추상적인 요구(VOC)를 엔지니어는 엔진 출력(Horsepower), 무게(Weight), 공기 저항 계수(Cd)라는 숫자로 바꿉니다. HOQ는 "빨라야 한다"는 말을 0.26의 공기 저항 계수로 번역하여, 이를 충족시키기 위해 어떤 재료를 쓰고(부품 전개), 어떤 용접 기술을 써야 하는지(공정 전개)까지 연결시켜주는 거대한 **엔지니어링 사전**입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 요구사항 분석 도구 비교 (QFD vs 기타 기법)
소프트웨어 및 시스템 공학에서 사용되는 주요 요구사항 분석 기법과 QFD의 차이점을 분석합니다.

| 비교 항목 | **QFD (품질 기능 전개)** | **KANO 모델** | **MoSCoW 기법** | **FMEA (고장 모드 영향 분석)** |
|:---|:---|:---|:---|:---|
| **핵심 목적** | **요구사항의 구체적 Spec 변환** | 요구사항의 **만족 유형** 분류 | 개발 범위 우선순위 결정 | 잠재적 **고장 원인** 예방 |
| **입력 데이터** | VOC, 기술 데이터 | 기능적/비기능적 요구사항 | 이해관계자 요구 | 시스템 아키텍처 |
| **주요 산출물** | HOQ 매트릭스, 기술 중요도 | 매력적/일원적/당연적 속성 | Must, Should, Could, Won't | RPN(위험 우선 순위) |
| **시점** | 기획 및 설계 초기 | 아이디어 도출 및 기획 | 일정/범위 관리 시 | 설계 및 테스트 단계 |
| **QFD와의 관계** | **기반** | QFD의 가중치 산출에 활용 | QFD 후보군 축소에 활용 | QFD의 '상관관계(지붕)'에서 나온 위험을 분석 |
| **결과 지표** | **정량적 설계 변수(예: 100ms)** | 정성적 만족도(만족/불만족) | 진행 여부(Yes/No) | 위험 점수(1~1000) |

#### 2. 시스템 공학적 융합 (Systems Engineering Convergence)
QFD는 독립적으로 존재하지 않으며 다른 프로세스와 강력한 시너지를 냅니다.

```text
      ┌───────────────────────────────────────────────────────────────┐
      │            통합 품질 관리 시스템 (Integrated Quality System)    │
      ├───────────────────────────────────────────────────────────────┤
      │                                                               │
      │   [문제 정의 단계]                                              │
      │   ● 4M 1E 분석으로 잠재 원인 식별                                │
      │          │                                                     │
      │          ▼                                                     │
      │   ┌─────────────────┐                                          │
      │   │  QFD (HOQ)      │ ◀──────┐                                │
      │   │  - 우선순위 도출 │        │                                │
      │   └────────┬────────┘        │                                │
      │            │                 │                                │
      │            ▼                 │                                │
      │   [요구사항 성격 분석]        │                                │
      │   ● KANO 모델:                 │                                │
      │     "이 기능은 매력적 품질인가?" │                                │
      │            │                 │                                │
      │            ▼                 │                                │
      │   [위험 및 상충 분석]         │                                │
