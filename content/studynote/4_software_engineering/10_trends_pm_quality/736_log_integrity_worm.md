# 736. 로그 6하 원칙 WORM 스토리지 무결성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보안 사고 발생 시 정확한 사후 분석(Forensics)과 책임 추적성(Accountability)을 확보하기 위해, 시스템의 모든 활동을 **6하 원칙(5W1H)**에 따라 기록하고 위변조를 원천 차단하는 기술적 방어 체계이다.
> 2. **무결성 보장**: 한 번 기록하면 삭제나 수정이 불가능한 **WORM (Write Once Read Many)** 스토리지 기술을 적용하여, 공격자가 침입 흔적을 지우는 것을 방지한다.
> 3. **가치**: ISMS-P, SOC2 등 정보보호 규제 준수를 위한 핵심 증적 자료로 활용되며, 투명한 감사 추적(Audit Trail)을 통해 시스템의 신뢰성을 확보한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "지워진 로그는 사고의 공범이다"

해커가 시스템에 침입한 후 가장 먼저 하는 일은 자신의 흔적(Log)을 지우는 것입니다. 로그가 조작되거나 유실되면 사고의 원인 파악도, 범인 검거도 불가능해집니다. 따라서 로그는 기록하는 것만큼이나 **'안전하게 보관하는 것'**이 중요합니다. 이를 위해 데이터의 생성부터 보관까지 무결성을 유지하는 아키텍처가 필요합니다.

```text

│  1. 6하 원칙 (블랙박스 영상):                                                │

│                                                                             │
│  → 완벽하게 기록하고(6하 원칙), **절대로 못 지우게(WORM)** 가두는 것!             │
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 로그 기록의 6하 원칙 (5W1H)

| 항목 | 질문 | 로그에 포함될 내용 |
|:---:|:---|:---|
| **Who** | 누가? | 사용자 ID, 접근 IP 주소, 단말기 정보 |
| **When** | 언제? | 표준화된 타임스탬프 (UTC 등), NTP 동기화 필수 |
| **Where** | 어디서? | 접속한 서비스명, API 엔드포인트, DB 테이블명 |
| **What** | 무엇을? | 수행한 작업 (조회, 수정, 삭제), 접근한 데이터 |
| **How** | 어떻게? | 성공/실패 여부, HTTP 응답 코드, 사용된 도구 |
| **Why** | 왜? | 작업 사유 (민감 정보 접근 시 필수 기록) |

### 2. WORM (Write Once Read Many) 기술
- **하드웨어 WORM**: CD-R, 전용 스토리지 솔루션 (EMC Centera 등).
- **소프트웨어 WORM**: 클라우드 객체 스토리지의 **Object Lock** (AWS S3 Object Lock, Azure Immutable Blob).
- **작동 방식**: 보존 기간(Retention Period)을 설정하면, 해당 기간 동안 루트 관리자(Root)조차 데이터를 삭제할 수 없음.

### 3. 로그 무결성 아키텍처 (ASCII)

```text
    [ Application ] ──(1) Real-time Sync ──▶ [ Central Log Server ]
           │                                        │
           │ (2) Generate Hash                      │ (3) Multi-copy
           ▼                                        ▼
    [ Hash Chain ] ◀──────(Compare)────── [ WORM Storage ]
    (이전 로그 해시값 포함)                    (ReadOnly / Locked)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 일반 스토리지 vs WORM 스토리지

| 항목 | 일반 스토리지 (RW) | WORM 스토리지 |
|:---:|:---|:---|
| **데이터 수정** | 가능 | **불가능 (Append Only)** |
| **데이터 삭제** | 관리자 권한 시 가능 | **보존 기간 내 절대 불가능** |
| **주 용도** | 일반 데이터 저장 | **로그, 법적 증거, 백업본** |
| **무결성 수준** | 낮음 (변조 가능) | **매우 높음 (물리적/논리적 보호)** |

### 2. 기술적 시너지: 블록체인(Blockchain)
로그의 무결성을 보장하는 가장 현대적인 방법 중 하나는 **해시 체인(Hash Chain)**이나 블록체인에 로그 해시를 기록하는 것입니다. 각 로그가 이전 로그의 해시를 포함하게 설계하면, 중간에 로그 하나만 수정되어도 전체 체인이 깨지므로 즉각적인 변조 탐지가 가능합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 개인정보 처리 시스템 로그 관리
- **상황**: 개인정보 접속 기록을 1년 이상 보관해야 하는 법적 의무 발생.
- **결단**: **AWS S3 Object Lock (Compliance Mode) 적용**. 
- **판단**: 일반 서버 디스크에 두면 침해 사고 시 로그가 가장 먼저 삭제됨. 
- **조치**: 로그 수집기(Fluentd)를 통해 S3 버킷으로 로그를 실시간 전송하고, 'Compliance' 모드로 1년 락을 설정. 
- **효과**: 해커가 루트 권한을 탈취해도 S3에 저장된 로그는 지울 수 없어 완벽한 증거 보존 성공.

### 📢 기술사적 결언
> "로그는 시스템의 **'기억'**이다. 기억이 조작된 시스템은 자신의 정체성을 잃은 것과 같다. 아키텍트는 로그의 양(Quantity)보다 **'질(Quality)'**과 **'불변성(Immutability)'**에 더 집중해야 한다. 6하 원칙에 따라 상세히 기록되지 않은 로그는 소음(Noise)일 뿐이며, WORM으로 보호되지 않는 로그는 거짓말(Lie)의 대상이 될 뿐이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **컴플라이언스 통과**: ISMS 등 인증 심사 시 로그 무결성 항목 100% 만족.
- **포렌식 효율**: 사고 발생 시 원인 분석 및 책임 추적 시간 80% 단축.

### 미래 전망
미래의 로그 관리는 **'지능형 실시간 무결성 검증'**으로 발전할 것입니다. 로그가 기록되는 찰나에 AI가 기밀성 유출 여부를 판단하여 마스킹하고, 동시에 양자 암호화 기술을 적용하여 WORM 스토리지에 봉인하는 'Self-defending Log' 체계가 구축될 것입니다. 또한 법적 증거로서의 로그 신뢰성을 보장하기 위해 '국가 공인 디지털 타임스탬프 서비스'와의 연동이 모든 엔터프라이즈 시스템의 기본 사양이 될 것입니다.

2. **WORM**은 한 번 일기를 쓰면 절대로 지우개로 지울 수 없는 **"요술 펜"**으로 쓰는 거예요.
