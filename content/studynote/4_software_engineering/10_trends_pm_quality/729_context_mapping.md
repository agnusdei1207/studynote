# 729. 컨텍스트 매핑 팀 간 인터페이스 정의

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도메인 주도 설계(DDD)에서 여러 개의 **바운디드 컨텍스트(Bounded Context)** 사이의 관계와 데이터 흐름을 시각화하고, 팀 간의 협업 방식과 기술적 인터페이스를 정의하는 **전략적 설계 도구**이다.
> 2. **관계 유형**: 권력 구조와 협업 형태에 따라 공유 커널(Shared Kernel), 고객-공급자(Customer-Supplier), 준수자(Conformist), **안티-커럽션 레이어(ACL)** 등 9가지 표준 패턴으로 분류된다.
> 3. **가치**: 대규모 분산 시스템에서 팀 간의 의존성을 명확히 하여 소통 비용을 줄이고, 특정 컨텍스트의 변경이 다른 곳으로 오염(Corruption)되는 것을 방지하여 전체 아키텍처의 무결성을 유지한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "우리 팀의 언어가 저 팀에서는 외계어?"

하나의 거대한 도메인 안에서도 '회원'이라는 단어는 주문 팀에게는 '구매자'이고, 보안 팀에게는 '인증 주체'입니다. 이들이 섞이면 코드는 엉망이 됩니다. **컨텍스트 매핑 (Context Mapping)**은 서로 다른 언어를 쓰는 팀(Context)들이 만나는 접점(Boundary)에서 "어떻게 대화하고 데이터를 주고받을지"를 미리 정하는 **외교적 설계**입니다.

### 💡 비유: 다른 나라와의 무역 협정

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        컨텍스트 매핑 비유                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] '주문국'과 '결제국'이라는 두 나라가 무역을 하려 함.                      │
│                                                                             │
│  1. 준수자 (Conformist): '주문국'이 힘이 세서, '결제국'은 군말 없이 '주문국'의     │
│     언어와 화폐(데이터 규격)를 그대로 따라야 함.                                │
│                                                                             │
│  2. 안티-커럽션 레이어 (ACL - 통역관): '결제국'의 언어가 오염되는 게 싫어서,      │
│     중간에 유능한 통역관을 두어 '주문국'의 말을 자기네 언어로 완벽히 번역해서      │
│     들음. (독립성 유지)                                                        │
│                                                                             │
│  3. 공유 커널 (Shared Kernel): 두 나라가 국경 지대에 공동 시장을 만들고,         │
│     똑같은 규칙을 같이 쓰기로 합의함.                                           │
│                                                                             │
│  → 팀 간의 **서열과 소통 방식**을 미리 정해두어 싸움을 막는 평화 협정!            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 컨텍스트 매핑 패턴 (Patterns)

| 패턴 명칭 | 설명 | 권력 관계 |
|:---:|:---|:---:|
| **Shared Kernel** | 두 컨텍스트가 도메인 모델의 일부(코드, DB)를 공유 | 대등 / 협력 |
| **Customer-Supplier**| 공급자가 고객의 요구사항을 반영해주는 상생 관계 | 상하 (Supplier 위) |
| **Conformist** | 하류(Downstream) 팀이 상류의 모델을 그대로 사용 | 상하 (Upstream 주도) |
| **Anti-Corruption Layer** | 외부 모델의 변경이 내부로 침투하지 못하게 변환 계층(ACL) 구축 | 독립 지향 |
| **Open Host Service** | 다수에게 표준화된 프로토콜(API) 제공 (예: 전사 공통 서비스) | 1 : N 제공 |
| **Separate Ways** | 의존 관계를 완전히 끊고 각자 개발 | 완전 독립 |

### 2. Upstream(U)과 Downstream(D)
- **Upstream (상류)**: 데이터를 주거나 API를 제공하는 쪽. 상류가 변하면 하류가 영향을 받음.
- **Downstream (하류)**: 데이터를 받거나 API를 호출하는 쪽.

### 3. 컨텍스트 맵 구조도 (ASCII)

```text
    ┌──────────────────────┐                ┌──────────────────────┐
    │   Order Context      │                │   Billing Context    │
    │ (Bounded Context A)  │ [U] ────────▶ [D] (Bounded Context B)  │
    └──────────┬───────────┘                └───────────┬──────────┘
               │                                        │
               │          ┌──────────────────┐          │
               └─────────▶│ Anti-Corruption  │─────────┘
                          │ Layer (ACL)      │
                          └──────────────────┘
    (Order의 데이터 규격이 Billing 내부 모델을 오염시키지 않도록 중간에서 변환)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 기술적 시너지: 마이크로서비스 (MSA)
컨텍스트 매핑은 MSA의 **서비스 간 통신 규약(Contract)**을 정의하는 철학적 근거가 됩니다. "우리는 저 팀의 API를 그대로 쓸 것인가(Conformist), 아니면 우리만의 DTO로 변환할 것인가(ACL)?"라는 결정이 곧 마이크로서비스의 결합도를 결정합니다.

### 2. 조직 거버넌스와의 결합
컨텍스트 맵은 조직도와 일치해야 합니다. 기술적으로는 독립적이어도 두 팀의 팀장이 같다면 'Shared Kernel'이 발생하기 쉽고, 팀이 완전히 분리되어 있다면 'ACL'이나 'Open Host Service'를 지향해야 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 외부 레거시 시스템 연동
- **상황**: 우리 시스템(신규)에서 20년 된 인사 시스템(레거시)의 데이터를 가져와야 함. 레거시 데이터 구조가 매우 난해함.
- **결단**: **Anti-Corruption Layer (ACL) 적용**. 
- **판단**: 레거시의 지저분한 필드명과 데이터 형식이 우리 신규 시스템의 도메인 모델에 침투하면 향후 유지보수가 불가능해짐. 
- **조치**: 외부 데이터를 우리 도메인 객체로 변환해주는 '인사 연동 어댑터'를 두어, 내부 로직은 항상 깨끗한 모델만 보게 함.
- **효과**: 레거시가 바뀌거나 교체되어도 우리 시스템의 핵심 로직은 영향 0%.

### 📢 기술사적 결언
> "컨텍스트 매핑은 **'경계의 외교학'**이다. 모든 팀과 완벽하게 협력(Shared Kernel)하는 것은 이상적이지만 불가능에 가깝다. 아키텍트는 각 팀의 정치적 상황과 기술적 성숙도를 고려하여, 때로는 냉정하게 **'Separate Ways'**를 선택하거나, 때로는 자신을 보호하기 위해 **'ACL'**이라는 방패를 세워야 한다. 맵 상의 화살표 하나가 곧 팀 간의 소통 비용과 시스템의 유연성을 결정한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **소통 비용 절감**: 인터페이스 책임 소재 명확화로 불필요한 회의 40% 감소.
- **아키텍처 부패 방지**: 하위 시스템의 모델 오염을 차단하여 장기 유지보수성 확보.

### 미래 전망
미래의 컨텍스트 매핑은 **'계약 기반 테스트 (Consumer Driven Contract)'**와 결합하여 자동화될 것입니다. 컨텍스트 맵에서 정의된 관계가 코드로 선언되고, 상류 팀이 API를 바꿀 때 하류 팀의 테스트가 실시간으로 실패하여 즉각적인 피드백을 주는 '살아있는 맵' 체계가 구축될 것입니다. 또한 서비스 메시 아키텍처와 융합되어, 네트워크 레벨에서 컨텍스트 간의 데이터 변환을 AI가 보조하는 지능형 통합 환경이 도래할 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **컨텍스트 매핑**은 서로 다른 나라 친구들이 모여서 놀 때 **"우리는 어떤 언어로 대화할까?"**라고 미리 정하는 약속이에요.
2. 영어를 쓰는 친구와 한국어를 쓰는 친구 사이에 **통역사(ACL)**를 세울지, 아니면 서로 **새로운 몸짓 언어(Shared Kernel)**를 만들지 결정하는 거죠.
3. 이 약속을 잘 정해두어야, 서로 말이 안 통한다고 울거나 싸우지 않고 **즐겁게 성을 쌓을 수** 있답니다!
