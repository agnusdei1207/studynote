# 674. 델파이 기법 전문가 합의

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다수의 전문가가 물리적/심리적 상호작용을 차단하고 익명의 피드백 루프(Feedback Loop)를 통해 의견을 수렴하는 **구조적 통계 예측 기법**이다.
> 2. **가치**: '밴드왜건 효과(Bandwagon Effect)'나 '할로 효과(Halo Effect)' 같은 인지적 편향을 제거하여, 불확실성이 높은 프로젝트 초기(WBS 작성 단계)의 **비용 및 공기 산정 신뢰도를 획기적으로 향상**시킨다.
> 3. **융합**: **애자일(Agile)** 방법론의 **플래닝 포커(Planning Poker)** 및 **PPM(Portfolio Project Management)** 시스템의 의사결정 알고리즘으로 발전하여 현대 소프트웨어 공학의 합리적 의사결정 프로세스를 규정한다.

---

### Ⅰ. 개요 (Context & Background)

**델파이 기법(Delphi Method)**은 1950년대 냉전 시대, 미국의 RAND 연구소가 소련의 공격 전략 등 군사적 데이터가 부재한 상황에서 미래를 예측하기 위해 개발했습니다. 소프트웨어 공학(Software Engineering) 관점에서 이는 "과거 이력 데이터가 전무한 신규 프로젝트나 혁신적 R&D 과제"에서 **전문가의 경험_based 암묵지(Tacit Knowledge)**를 **정량화된 형식지(Explicit Knowledge)**로 변환하여 추정 오차를 최소화하는 절차입니다.

일반적인 회의에서 발생하는 "권위자의 의견에 기약 없이 따르는 현상"이나 "목소리 큰 소수에 의한 의사 결정"을 방지하기 위해, **중재자(Mediator/Facilitator)**를 통한 간접 소통을 강제합니다.

**💡 비유: 범인 찾기 추리 게임**
> 마치 여러 명의 탐정이 모였을 때, 가장 높은 직급의 탐정이 "범인은 이 사람이다"라고 말하면 다른 탐정들이 자신의 의견을 숨기고 따르는 것을 방지하기 위해, 모두가 별도의 방에서 자신의 추리 결과를 적어 낸 뒤, 이를 종합하여 다시 추리하는 과정을 반복하는 것과 같습니다.

**📢 섹션 요약 비유:**
> "마치 토론 시간에 목소리 큰 학생을 막기 위해, 친구들끼리 얼굴을 보지 말고 종이에 의견을 써서 돌려보며, 점점 답을 좁혀가는 '무음 토론' 방식과 같습니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

델파이 기법은 단순한 설문조사가 아닌, **익명성(Anonymity), 반복(Iteration), 통계적 피드백(Statistical Feedback)**의 3대 기둥으로 구성된 시스템입니다.

#### 1. 구성 요소 및 역할

| 구성 요소 | 역할 | 상세 동작 메커니즘 |
|:---:|:---|:---|
| **중재자 (Facilitator)** | 프로세스 조율 및 통제 | 설문지 배포, 결과 통계 집계(평균/중앙값/사분위수), 의견 요약 제공. **자신의 의견은 개입하지 않음.** |
| **전문가 패널 (Expert Panel)** | 예측 및 의사결정 주체 | 해당 분야의 지식을 가진 5~10명 내외. **서로의 신원을 모르는 상태**에서 독립적인 판단 내림. |
| **질문지 (Questionnaire)** | 합의 대상 구조화 | 개방형 질문(1라운드)에서 시작하여, 통계 결과와 함께 재검토가 필요한 항목을 묻는 폐쇄형 질문(후속 라운드)으로 진화. |

#### 2. 와이드밴드 델파이 (Wideband Delphi) 프로세스

#### 3. 핵심 알고리즘: 합의 도출 메커니즘
델파이 기법의 수학적/논리적 핵심은 **아웃라이어(Outlier) 탐색과 보정**입니다. 1차 추정치들을 분포(Distribution)로 나열한 후, 중앙값(Median)에서 벗어난 전문가(특히 사분위 범위 Interquartile Range 밖의 의견을 가진 사람)에게 **논리적 근거(Basis of Estimate)**를 요구합니다.

```text
[의사코드(Pseudo-Code)로 보는 합의 알고리즘]

Function Delphi_Consensus(Panel, Target_Issue):
    While (Standard_Deviation > Threshold):
        Estimates[] = Collect_Estimates(Panel)  // 익명 수집
        
        Median_Val = Calculate_Median(Estimates)
        Lower_Quartile = Calculate_Q1(Estimates)
        Upper_Quartile = Calculate_Q3(Estimates)
        
        // 중재자의 피드백 생성
        Feedback = Generate_Feedback(Median_Val, Lower_Quartile, Upper_Quartile)
        
        // 비동의자(Outlier)에게 근거 요구
        For Expert in Panel:
            If Expert.Estimate < Lower_Quartile OR Expert.Estimate > Upper_Quartile:
                Expert.Request_Justification("Why is your estimate an outlier?")
                
        Distribute(Panel, Feedback)  // 통계적 정보 공개
        
        // 전문가들은 타인의 근거를 보고 자신의 추정치 재검토
        Estimates[] = Re_Collect_Estimates(Panel)
        
    Return Final_Consensus(Estimates)
```
이 과정은 전문가가 단순히 '수치를 조절'하는 것이 아니라, **자신이 인지하지 못했던 리스크(Outlier의 근거)를 학습**하여 추정을 업데이트하는 **학습 루프(Learning Loop)** 역할을 합니다.

**📢 섹션 요약 비유:**
> "마치 길을 잃은 여행자들이 각자 지도를 그려보고, 중간 지점에서 모여 지도를 합치며, '산맥이 어디 있는지' 정보를 공유해 최종 정확한 지도를 완성해 나가는 과정과 같습니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

델파이 기법은 다른 의사결정 및 추정 기법들과 명확히 구별되며, 특히 **애자일(Agile)** 방법론과의 융합을 통해 현대적으로 재해석되고 있습니다.

#### 1. 델파이 vs. 타 추정 기법 비교

| 구분 | **델파이 기법 (Delphi)** | **전통 회의 (Traditional Meeting)** | **PERT (Program Evaluation Review Technique)** |
|:---|:---|:---|:---|
| **의견 표현** | **비대면, 익명** | 대면, 실명 | 개별 작성 후 집계 |
| **상호작용** | **통계적 피드백 (간접)** | 언어적 토론 (직접) | 수학적 3점 추정 |
| **편향성 위험** | **낮음 (Low)** | 높음 (Halo/Bandwagon) | 중간 (개별 작업 시) |
| **합의 속도** | 느림 (다소 시간 소요) | 빠름 (가결 가능) | 빠름 (공식 대입) |
| **주요 용도** | **복잡한 문제, 신기술 추정** | 단순 의사결정 | 일정 산정 (기대치 계산) |

#### 2. 기술적 융합: 플래닝 포커 (Planning Poker)
델파이 기법의 '익명성'과 '반복' 원칙을 애자일 스크럼(Agile Scrum) 환경에 맞춰 **게이미피케이션(Gamification)**화 한 것이 바로 **플래닝 포커**입니다.

```text
┌────────────────────────────────────────────────────────────────────────────┐
│             [ 델파이 기법의 현대적 진화: Planning Poker ]                  │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│   [ 원칙 매핑 ]                                                             │
│                                                                            │
│   델파이의 익명성      ──▶  카드를 동시에 오픈하여 타인의 영향력 차단        │
│   (Anonymity)           (Simultaneous Reveal)                              │
│                                                                            │
│   델파이의 피드백      ──▶  점수 차이(Outlier)가 난 사람의 토론 유도        │
│   (Feedback)            (Disscussion among outliers)                       │
│                                                                            │
│   델파이의 반복        ──▶  재투표하여 합의점 찾기                           │
│   (Iteration)           (Re-vote until consensus)                          │
│                                                                            │
│   ┌───────────────────────────────────────────────────────────────────┐    │
│   │  [ 시나리오 ]                                                     │    │
│   │                                                                   │    │
│   │  PO(Product Owner): "이 로그인 기능의 스토리 포인트는?"             │    │
│   │  Devs: 카드를 뒤집어 든 채 생각...                                  │    │
│   │                                                                   │    │
│   │  Action: 1, 2, 3... (동시에 카드 오픈)                              │    │
│   │                                                                   │    │
│   │  Result: [3, 3, 8, 3, 5]                                          │    │
│   │                                                                   │    │
│   │  PO: "8점을 든 지니(Genie), 왜 그렇게 어렵다고 보나?"                │    │
│   │  Dev 8: "레거시 DB 호환 작업이 필요해서요."                          │    │
│   │                                                                   │    │
│   │  Action: 재투표 후 [5, 5, 5, 5, 5]로 합의 도출                       │    │
│   └───────────────────────────────────────────────────────────────────┘    │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘
```
이는 델파이 기법이 단순한 설문이 아니라 **'팀의 합의 과정(Consensus Building)' 프로토콜**임을 보여줍니다.

**📢 섹션 요약 비유:**
> "델파이는 '기성 회의'이고, 플래닝 포커는 이를 '카드 게임'이라는 포장지로 감싼 '리마스터판'이라고 볼 수 있습니다. 핵심은 여전히 '서로의 눈치를 보지 말고 솔직하게 투표하라'는 것입니다."
