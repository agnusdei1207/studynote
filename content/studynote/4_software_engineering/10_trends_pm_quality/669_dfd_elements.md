# 669. DFD 자료 흐름도 4요소

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 내에서 데이터가 발생하여 가공되고 저장되는 **데이터의 수명 주기(Data Lifecycle)**를 시각화하는 **구조적 분석(Structured Analysis)**의 핵심 도구이다.
> 2. **가치**: 사용자와 개발자 간의 인지 차이를 해소하여, 개발 초기 단계에서 요구사항의 누락을 40% 이상 예측하고 비즈니스 로직을 정밀하게 정의할 수 있다.
> 3. **융합**: 현대의 **UML (Unified Modeling Language)** 활동도(Activity Diagram)이나 **MSA (Microservices Architecture)**의 데이터 파이프라인 설계에 적용되는 데이터 중심 아키텍처의 근간이다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 정의**
DFD (Data Flow Diagram, 자료 흐름도)는 시스템의 기능적 요구사항을 분석하기 위해, 데이터가 시스템의 외부로 유입되어 내부의 **프로세스(Process)**를 거쳐 가공되고, **자료 저장소(Data Store)**에 저장되거나 외부로 나가는 흐름을 모형화한 도구입니다.
이는 1970년대 래리 콘스탄틴(Larry Constantine), 에드워드 유어돈(Edvard Yourdon), 톰 드마코(Tom DeMarco) 등에 의해 정립된 구조적 설계(Structured Design) 방법론의 핵심이며, 제어 흐름(Control Flow)보다는 **데이터 흐름(Data Flow)**에 집중하여 시스템을 "어떻게(How)" 구현할 것인가보다 "무엇(What)"을 해야 하는지에 집중하게 합니다.

**2. 등장 배경 및 필요성**
소프트웨어 위기(Software Crisis)가 대두되던 시절, 난해한 코드나 복잡한 흐름도 대신 사용자가 이해하기 쉬운 그래픽 도구를 통해 요구사항을 명확히 하고자 등장했습니다. 특히, 하향식(Top-down) 접근법을 지원하여 시스템을 단순한 레벨(Level 0)부터 점차 세부적인 레벨(Level 1, 2...)으로 분해(Decomposition)하는 **계층 구조(Hierarchical Structure)** 분석을 가능하게 합니다.

**3. 하이브리드 다이어그램: 추상화 레벨 (Leveling) 구조**
DFD는 가장 추상화된 배경도(Context Diagram)에서 시작하여 구체적인 레벨 다이어그램으로 분할됩니다. 이 구조는 "분할하고 정복한다(Divide and Conquer)"는 소프트웨어 공학의 철학을 시각적으로 완벽하게 구현합니다.

```text
      [ Level 0: Context Diagram ] (최상위 추상화)
      ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
         +-------------------+
         │   System 0.0      │
         │ (Whole System)    │
         +-------------------+
              ▲      │
      ┌─────────┘      └─────────┐
      │                           │
 [Entity 1]                   [Entity 2]
 (Customer)                   (Bank)
      │                           │
      ▼                           ▼
 
      [ Level 1: Decomposition ] (하위 기능 분해)
      ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
      +-------------------------------------------------------+
      |  P1.0: Login      │  P2.0: Process │  P3.0: Logout   |
      │      +-------+    │      +-------+  │      +-------+  |
      |      | Auth  |    │      | Calc  |  │      | Save  |  |
      |      +-------+    │      +-------+  │      +-------+  |
      +-------------------------------------------------------+
           ▲    │            ▲    │          ▲    │
           │    ▼            │    ▼          │    ▼
      (Data Flow connects Processes & Stores)
```

**다이어그램 해설**
위 구조는 시스템 분석의 깊이(Depth)를 나타냅니다. **Level 0**에서는 시스템을 하나의 검은 상자(Black Box)로 보며, 시스템과 외부 엔티티 간의 상호작용만 정의합니다. 이어지는 **Level 1**에서는 검은 상자를 열어 주요 기능(프로세스)별로 분해하고, 프로세스 간의 데이터 교환과 저장소 연결을 상세히 명시합니다. 이를 통해 분석가는 복잡한 시스템을 계층별로 관리할 수 있습니다.

> 📢 **섹션 요약 비유**: 마치 거대한 지도를 그릴 때, 처음에는 '대한민국' 전체 지도(Context Diagram)를 그리고, 점차 확대해가며 특정 도시의 구획(Level 1)과 도로망(Data Flow)을 세부적으로 표시하는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

DFD는 기능적 분해(Functional Decomposition)를 수행하는 4가지의 필수 불가결한 구성 요소로 이루어집니다. 이 요소들은 **Gane-Sarson 표기법**이나 **Yourdon 표기법**에 따라 모양이 다를 수 있으나, 의미는 동일합니다.

**1. 상세 구성 요소 (Components)**

| 요소 (Component) | 표기 (Notation) | 역할 (Role) | 실무 예시 |
|:---:|:---:|:---|:---|
| **프로세스 (Process)** | ○ (원) 또는 ⬭ (둥근 사각형) | 입력 데이터를 받아 가공하여 출력하는 내부 연산 단위. 숫자(ID)로 식별(예: 1.0, 1.1). | '주문 계산', '비밀번호 검증' 로직 수행 |
| **자료 흐름 (Data Flow)** | → (화살표) | 데이터가 이동하는 통로. 화살표 위에 데이터 명칭 기재. | '주문서', '승인 결과' 전송 |
| **자료 저장소 (Data Store)**| ═ (이중선 또는 평행선) | 데이터가 정지해 있는 장소. DB, 파일, 큐 등을 의미. | '상품 마스터 파일', '로그 테이블' |
| **단말 (External Entity)**| □ (사각형) | 시스템 경계 밖에 있는 주체. 데이터의 발생지나 도착지. | '고객', '외부 결제 PG사', '은행 허브' |

**2. 데이터 흐름의 물리적/논리적 구성도**
DFD는 단순히 화살표를 연결하는 것이 아니라, **데이터가 어떤 경로를 통해 상태(State)가 변화하는지**를 정의하는 상태 전이 다이어그램의 역할을 수행합니다.

```text
   (Input Phase)           (Processing Phase)          (Output Phase)
 ┌──────────────┐      ┌──────────────────────────┐      ┌──────────────┐
 │ External    │      │    System Boundary       │      │ External    │
 │ Entity      │      │                          │      │ Entity      │
 │ [ Source ]  │      │                          │      │ [ Sink ]    │
 └──────┬───────┘      │                          │      └──────▲───────┘
        │              │                          │             │
        │ Data Flow    │                          │  Data Flow  │
        │  (Raw Data)  │                          │ (Result)    │
        ▼              │                          │             │
 ┌─────────────────────────────────────────────────────────────────────┐
 │  [ Process 1.0 ]      [ Data Store D1 ]      [ Process 2.0 ]        │
 │   ┌────────┐   Write   ┌─────────────┐  Read   ┌────────┐           │
 │   │ Verify│ ──────────►│  Customer   │────────►│ Update │           │
 │   │  Data  │           │    Master   │         │  Info  │           │
 │   └────────┘           └─────────────┘         └────────┘           │
 │      │                                                                      │
 │      │  Error Flow (Feedback)                                               │
 │      └──────────────────────────────────────────────────────────────────► │
 │                             (Exception Handling)                           │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**다이어그램 해설**
이 다이어그램은 DFD의 동적 메커니즘을 보여줍니다.
1. **프로세스(Process)**: 활성(Active)한 요소로서, 데이터 저장소(Data Store)에서 데이터를 읽거나(Read) 쓰기(Write)를 수행합니다. 저장소는 단순한 파일뿐만 아니라 메시지 큐(Message Queue)나 캐시(Cache)일 수 있습니다.
2. **흐름의 방향성**: 데이터는 반드시 프로세스를 거쳐야만 가공됩니다. 단말에서 단말로 곧바로 가는 것은 시스템이 처리했다는 것을 보장할 수 없으므로 금지됩니다.
3. **피드백 루프**: 오류 데이터나 재시도 요청과 같은 흐름은 프로세스에서 다시 단말이나 저장소로 돌아가는 흐름으로 표현됩니다.

**3. 핵심 제약 조건 및 심층 원리**
DFD를 작성할 때 반드시 준수해야 할 '보존의 법칙(Conservation Principle)'이 존재합니다.
*   **보존의 법칙**: 어떤 프로세스로 들어가는 입력 데이터(Input)는 반드시 그 프로세스에서 나오는 출력 데이터(Output)와 밸런스를 이루어야 합니다. 출력으로 나가지 않는 입력은 '블랙홀(Black Hole)'이라 하며, 입력 없이 출력되는 것은 '미라지(Miracle)'라 부르는 오류로 간주합니다.

> 📢 **섹션 요약 비유**: 공장의 **'제조 라인(Assembly Line)'**과 같습니다. 원료(Data)가 컨베이어 벨트(Flow)를 타고 기계(Process)에 들어가 가공된 후, 창고(Store)에 잠시 보관되거나 최종적으로 고객(Entity)에게 인도되는 과정을 설계도로 그린 것입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

DFD는 단순한 다이어그램을 넘어 다른 분석 기법과 상호 보완적이며, 타 IT 분야와의 융합 관계를 이해해야 합니다.

**1. DFD vs. Flowchart (순서도)**
가장 혼동하기 쉬운 두 도구는 시각적 유사성에도 불구하고 목적이 완전히 다릅니다.

| 구분 | **DFD (Data Flow Diagram)** | **Flowchart (순서도)** |
|:---|:---|:---|
| **핵심 관점** | **데이터(Data)**가 무엇인가, 어디서 와서 어디로 가는가? | **제어(Control)/순서**가 어떻게 진행되는가? |
| **시간성** | 순서(Time sequence)를 명확히 나타내지 않음 (비동기적 가능) | 시간의 흐름(Timeline)과 순서가 매우 중요 |
| **표시 대상** | 데이터의 변형과 저장, 시스템 간 인터페이스 | 루프(Loop), 조건 분기(If/Else), 연산 순서 |
| **주요 사용자** | 시스템 분석가, 사용자(비즈니스 이해도 높음) | 프로그래머, 알고리즘 설계자(구현 중심) |

**2. DFD vs. UML Activity Diagram (기술적 진화)**

```text
      [ Structured Analysis Era ]          [ Object Oriented Era ]
              DFD (Data Centric)   ────▶   UML Activity (Control + Data)
                   │                               │
                   ▼                               ▼
        "Data is the King"                  "Objects behave"
        
   ┌───────────────────────────────────────────────────────────────────┐
   │  Relationship: DFD의 요소가 UML에서 어떻게 변환되는가?            │
   ├───────────────────────────────────────────────────────────────────┤
   │  Process       ──▶   Activity (Action)                           │
   │  Data Store    ──▶   Object Node / Class Attribute               │
   │  Data Flow     ──▶   Object Flow (Control Flow + Data Flow)      │
   │  Terminator    ──▶   Actor / Swimlane (Partition)                │
   └───────────────────────────────────────────────────────────────────┘
```

**3. 데이터베이스 설계와의 시너지 (Data Modeling)**
DFD의 **자료 저장소(Data Store)**는 논리적 데이터 모델링(LDM)의 **Entity(实体)**와 1:1로 대응됩니다.
*   **연결고리**: DFD에서 식별된 저장소와 흐름은 추후 **ERD (Entity Relationship Diagram)** 작성 시 필수적인 엔터티와 관계(Relationship)를 도출하는 기초 자료가 됩니다.
*   **정규화(Normalization)**: DFD를 통해 데이터의 흐름을 파악하면, 중복되는 데이터 저장소를 발견하고 이를 통합하는 데이터 정규화 과정을 체계적으로 수행할 수 있습니다.

> 📢 **섹션 요약 비유**: **'내비게이션 지도 vs. 운전 매뉴얼'**의 차이와 같습니다. DFD는 "어느 도시에서 어느 도시로 가는 길(데이터 경로)"을 보여주는 내비게이션이라면, 순서도는 "신호를 밟고, 핸들을 꺾는 운전 조작(제어 논리)"을 설명하는 매뉴얼입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실제 현업에서 DFD는 레거시 시스템 재설계나 신규 시스템의 요구사항 분석 단계에서 필수적으로 사용됩니다. 특히 **이기종 시스템 간 연동**이나 **데이터 무결성**이 중요한 금융/공공 분야에서 강력한 효력을 발휘합니다.

**1. 실무 적용 시나리오: ATM 망(Data Network) 재설계**

*   **상황**: 기존 은행의 메인프레임(Mainframe) 기반 시스템을 MSA(Microservices Architecture)로 전환하는 프로젝트.
*   **적용**:
    1.  **범위 정의(Context)**: 외부 기관(망 지국, 카드사)을 단말로 정의하고, ATM망을 하나의 시스템으로 묶어 전체 트래픽 양을 파악.
    2.  **상세 분해(Decomposition)**: '인증', '잔액 조회', '출금'을 독립적인 프로세스(서비스)로 분리.
    3.  **데이터 무결성 확인**: '출금' 프로세스에서 '잔액 데이터'를 읽어(Read) 출금 후 '거래 내역'을 저장(Write)하는 흐름이 끊어지지 않는지
