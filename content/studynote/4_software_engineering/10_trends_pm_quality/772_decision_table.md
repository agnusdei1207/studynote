# 772. 결정 테이블 복잡한 논리 조합 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 복잡한 조건(Condition)과 그에 따른 행동(Action)의 조합을 표 형태로 정형화하여, 모든 가능한 논리적 케이스를 빠짐없이 분석하고 설계하는 **구조적 명세 기법**이다.
> 2. **구성**: 조건의 나열인 **조건부(Stub)**, 조건의 참/거짓 조합인 **조건항(Entry)**, 수행할 작업 목록인 **작업부**, 그리고 실행 여부를 표시하는 **작업항**의 4개 영역으로 이루어진다.
> 3. **가치**: 텍스트 명세에서 놓치기 쉬운 논리적 누락이나 중복을 시각적으로 식별하며, 구현 단계의 복잡한 `if-else`문을 체계화하고 테스트 케이스 도출의 근거로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "꼬여버린 조건문의 지옥"

"회원이면서 19세 이상이고, 첫 방문이면 쿠폰을 주는데, 만약 포인트가 있으면 중복 할인은 안 되고..."와 같은 비즈니스 로직을 글로만 적으면 반드시 구멍이 생깁니다. 개발자는 이를 코드로 옮기다 실수하고, 테스터는 어떤 상황을 테스트해야 할지 헷갈립니다. **결정 테이블 (Decision Table)**은 이 복잡한 실타래를 표 하나로 깔끔하게 정리하는 '논리 지도'입니다.

### 💡 비유: 식당 할인 룰북

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        결정 테이블(Decision Table) 비유                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 우리 식당의 할인 규칙을 아르바이트생에게 가르쳐야 함.                    │
│                                                                             │
│  [조건항]                                                                    │
│  - 조건 1: 오늘이 생일인가? (Y/N)                                            │
│  - 조건 2: 단골 손님인가? (Y/N)                                              │
│                                                                             │
│  [작업항]                                                                    │
│  - 결과 1: 50% 할인 (생일이고 단골일 때만!)                                   │
│  - 결과 2: 10% 할인 (둘 중 하나만 해당할 때)                                  │
│  - 결과 3: 할인 없음 (둘 다 아닐 때)                                          │
│                                                                             │
│  → 모든 경우의 수를 표로 그려두면, **어떤 손님이 와도** 당황하지 않고            │
│     정확한 가격을 계산할 수 있음!                                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 결정 테이블의 4대 구성 영역

| 영역 명칭 | 영문 명칭 | 상세 설명 |
|:---:|:---|:---|
| **조건부** | **Condition Stub** | 의사결정에 영향을 미치는 모든 조건들을 나열 (질문 리스트) |
| **조건항** | **Condition Entry** | 각 조건의 가능한 값 조합 (Y/N, T/F 등) |
| **작업부** | **Action Stub** | 조건 조합에 따라 취할 수 있는 모든 행동들의 나열 |
| **작업항** | **Action Entry** | 특정 조건 조합(Rule)에서 실행할 행동을 표시 (X 등) |

### 2. 작성 프로세스
1. **변수 식별**: 결과에 영향을 주는 입력 조건들을 모두 도출.
2. **경우의 수 계산**: $2^n$ (조건이 n개일 때) 또는 유효한 조합 계산.
3. **행동 정의**: 가능한 모든 시스템 반응 나열.
4. **테이블 병합**: 논리적으로 동일한 결과가 나오는 규칙들은 'Don't Care(-)' 기호를 써서 단순화.

### 3. 구조적 아키텍처 (ASCII)

```text
    ┌──────────────────────┬────────┬────────┬────────┬────────┐
    │      구분 (Rule)     │ Rule 1 │ Rule 2 │ Rule 3 │ Rule 4 │
    ├──────────────────────┼────────┼────────┼────────┼────────┤
    │ [조건부]             │        │        │        │        │
    │ - 유료 회원인가?      │   Y    │   Y    │   N    │   N    │
    │ - 쿠폰이 있는가?      │   Y    │   N    │   Y    │   N    │
    ├──────────────────────┼────────┼────────┼────────┼────────┤
    │ [작업부]             │        │        │        │        │
    │ - 20% 할인           │   X    │        │        │        │
    │ - 10% 할인           │        │   X    │   X    │        │
    │ - 정가 판매          │        │        │        │   X    │
    └──────────────────────┴────────┴────────┴────────┴────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 결정 테이블 vs 상태 전이도 (STD)
- **결정 테이블**: **현재 시점**의 복합 조건 조합에 집중. (정적 논리)
- **상태 전이도**: **시간 흐름**에 따른 과거 상태와 이벤트의 관계에 집중. (동적 흐름)
- **시너지**: 상태 전이도의 특정 상태 내에서 벌어지는 복잡한 판단 로직을 결정 테이블로 상세화합니다.

### 2. 기술적 시너지: 테스트 케이스 자동 생성
결정 테이블의 각 열(Rule)은 그대로 하나의 **테스트 케이스(Test Case)**가 됩니다. 이는 블랙박스 테스트 기법 중 '결정 테이블 테스팅'으로 불리며, 입력값 조합이 너무 많아 모두 테스트하기 힘들 때 논리적으로 최소한의 완벽한 케이스를 뽑아내는 근거가 됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 대출 승인 심사 엔진 설계
- **상황**: 신용 점수, 연 소득, 기존 대출 유무, 직업군 등 10가지 조건이 복합적으로 얽힘.
- **결단**: **결정 테이블 기반 로직 설계 및 단순화**. 
- **판단**: 1,024개($2^{10}$)의 모든 조합을 나열한 뒤, '기존 연체자'일 경우 다른 조건에 상관없이 무조건 '거절'되는 규칙들을 하나로 합쳐 테이블 크기를 대폭 축소.
- **효과**: 개발자는 복잡한 중첩 `if`문 대신 테이블 기반의 룰 엔진(Rule Engine)을 사용하여 가독성 확보.

### 📢 기술사적 결언
> "결정 테이블은 **'논리의 방탄복'**이다. 텍스트로 된 기획서는 행간의 의미를 추측하게 만들지만, 표로 된 명세는 오직 '예/아니오'만을 요구한다. 아키텍트는 설계 단계에서 '이럴 땐 어떻게 하죠?'라는 질문을 받기 전에, 결정 테이블을 통해 **'모든 경우의 수'**를 먼저 선점해야 한다. 특히 업무 규칙이 수시로 변하는 도메인일수록, 하드코딩된 로직보다 결정 테이블 구조의 룰 데이터를 관리하는 것이 아키텍처의 유연성을 결정한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **누락 발견**: 설계 단계에서 처리되지 않은 'Edge Case' 100% 식별.
- **테스트 효율**: 불필요한 중복 테스트 제거 및 핵심 경로 검증 보장.

### 미래 전망
미래의 결정 테이블은 **'AI 룰 제너레이터'**로 진화합니다. 자연어로 된 법령이나 복잡한 약관을 AI가 읽고 자동으로 결정 테이블을 생성하며, 이 테이블이 즉시 실행 가능한 코드(DMN, Decision Model and Notation)로 변환되는 '분석-구현 일체화' 환경이 구축될 것입니다. 또한 복잡한 금융 공학이나 법률 IT 분야에서 '설명 가능한 논리'를 제공하는 핵심 수단으로 각광받을 것입니다.
