# 690. 소프트웨어 자재 명세서 (SBOM) 공급망 보안

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어를 구성하는 모든 오픈소스 라이브러리, 모듈, 의존성 관계를 기록한 **표준화된 목록(Inventory)**으로, 소프트웨어 공급망의 투명성을 확보하기 위한 핵심 도구이다.
> 2. **표준**: SPDX, CycloneDX 등 기계 읽기 가능한 형식으로 작성되어, 특정 라이브러리의 취약점(예: Log4j) 발생 시 해당 부품이 사용된 소프트웨어를 수 초 내에 식별하고 대응하게 한다.
> 3. **가치**: 보안 사고 예방, 오픈소스 라이선스 컴플라이언스 준수, 그리고 '신뢰할 수 있는 소프트웨어 인도'를 보장하는 엔터프라이즈 보안의 필수 요건으로 자리 잡고 있다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "보이지 않는 위협, 소프트웨어 공급망"

현대 소프트웨어의 80~90%는 오픈소스로 이루어져 있습니다. 우리는 우리가 직접 짠 코드보다 남이 만든 라이브러리를 더 많이 사용합니다. 만약 그 라이브러리에 독(취약점)이 들어있다면? 2021년 전 세계를 공포에 떨게 한 **Log4j 사태**는 공급망 보안의 중요성을 일깨웠습니다. **SBOM (Software Bill of Materials)**은 "이 소프트웨어에 무엇이 들어있는가?"에 대한 정직한 답변입니다.

### 💡 비유: 가공식품의 성분 함량표

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        SBOM (소프트웨어 자재 명세서) 비유                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 마트에서 과자(소프트웨어)를 삼.                                        │
│                                                                             │
│  1. SBOM이 없다면:                                                           │
│     과자를 먹고 알레르기(취약점)가 생겼는데, 과자 속에 밀가루가 들었는지 땅콩이     │
│     들었는지 알 길이 없음. 모든 과자를 다 검사해야 함. (대응 지연)               │
│                                                                             │
│  2. SBOM이 있다면 (성분표):                                                  │
│     봉지 뒷면 성분표에 "밀가루(A사), 설탕(B사), 땅콩(C사)"이라고 정확히 적혀 있음. │
│     "C사 땅콩이 상했다"는 뉴스가 뜨자마자, 나는 이 과자를 먹지 않고 버릴 수 있음. │
│                                                                             │
│  → 우리 앱에 들어간 모든 '재료'를 투명하게 공개하여 사고를 막는 것!              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. SBOM의 핵심 구성 요소
- **Component Name**: 라이브러리 명칭.
- **Version**: 사용된 정확한 버전 번호.
- **Supplier**: 제작자 또는 조직.
- **Relationship**: 의존성 관계 (직접 참조 vs 간접 참조).
- **Hash Value**: 파일의 무결성을 증명하는 고유값.
- **License**: 오픈소스 라이선스 정보 (GPL, Apache 등).

### 2. 주요 기술 표준 (Formats)

| 표준 | 주관 | 특징 |
|:---:|:---|:---|
| **SPDX** | Linux Foundation | ISO 국제 표준, 라이선스 및 지식재산권 관리에 강점 |
| **CycloneDX** | OWASP | 보안 취약점 분석 및 자동화된 파이프라인에 최적화 |
| **SWID** | ISO/IEC | 설치된 소프트웨어 식별 태그 중심의 표준 |

### 3. SBOM 생성 및 관리 아키텍처 (ASCII)

```text
    [ Source Code / Build ] ──▶ [ SBOM Generator ] ──▶ [ Standard Output ]
    (Maven, NPM, Go, Docker)     (Syft, Tern, Grype)     (JSON / XML / YAML)
                                                                │
          ┌─────────────────────────────────────────────────────┘
          ▼
    [ SBOM Repository ] ◀─── 연동 ───▶ [ Vulnerability DB ]
    (Dependency Track)                 (NVD, OSV, Github Advisory)
          │                                     │
          └───────────▶ [ Risk Alert ] ◀────────┘
                        (Log4j 발견! 즉시 대응)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. SCA (Software Composition Analysis) vs SBOM
- **SCA**: 오픈소스 취약점을 **분석하는 행위나 도구**.
- **SBOM**: SCA 결과로 만들어진 **표준화된 산출물(문서)**.
- **시너지**: SCA 도구가 SBOM을 생성하고, 생성된 SBOM은 다른 보안 도구들이 읽어 전사적인 위험을 관리하는 양분이 됩니다.

### 2. 기술적 시너지: DevSecOps와 SLSA
SBOM은 배포 파이프라인의 필수 단계입니다. **SLSA (Supply-chain Levels for Software Artifacts)** 프레임워크와 결합하여, 소프트웨어가 빌드되는 전 과정의 무결성을 증명하는 핵심 증적 자료가 됩니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 미국 연방정부 공급망 보안 규제 대응
- **상황**: 미국 정부에 소프트웨어를 납품하기 위해 행정명령(EO 14028)에 따른 SBOM 제출 요구됨.
- **결단**: **CycloneDX 기반 자동화 파이프라인 구축**. 
- **판단**: CI/CD 단계에 'Syft' 도구를 추가하여 빌드 시마다 SBOM을 자동 생성. 이를 'Dependency-Track' 서버에 업로드하여 실시간 취약점 모니터링 수행.
- **효과**: 글로벌 규제 준수(Compliance)와 동시에, 내부적으로 사용하는 수만 개의 오픈소스 버전을 중앙에서 통제 가능해짐.

### 📢 기술사적 결언
> "SBOM은 소프트웨어의 **'신분증'**이자 **'족보'**다. 신뢰할 수 없는 재료로 만든 음식을 팔 수 없듯, 정체를 모르는 코드로 가득 찬 소프트웨어를 배포하는 시대는 끝났다. 아키텍트는 코드를 잘 짜는 것을 넘어, 우리 시스템이 의존하는 **'거대한 외부 생태계'**를 어떻게 가시화하고 통제할 것인지 SBOM이라는 렌즈를 통해 바라보아야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **취약점 대응 속도**: 보안 사고 발생 시 영향도 파악 시간 수 일에서 **수 초**로 단축.
- **컴플라이언스 리스크 제거**: 라이선스 위반으로 인한 법적 분쟁 리스크 90% 감소.

### 미래 전망
미래의 SBOM은 **'VEX (Vulnerability Exploitability eXchange)'**와 결합할 것입니다. 단순히 "취약한 라이브러리가 있다"는 정보를 넘어, "그 라이브러리가 우리 앱에서 실제로 실행되는 경로에 있어 위험하다/안전하다"라는 실행 맥락까지 포함하는 지능형 명세서로 진화할 것입니다. 이는 보안 담당자의 피로도를 획기적으로 줄여주는 결정적인 기술이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **SBOM**은 장난감 상자 겉면에 적힌 **"부품 목록표"**와 같아요.
2. "건전지 2개, 작은 나사 10개, 빨간 블록 5개"라고 정확히 적혀 있어서, 만약 **"파란 나사가 위험해요!"**라는 뉴스가 나오면 내 장난감에 그 나사가 있는지 금방 알 수 있죠.
3. 목록표만 잘 챙겨두면, 엄마 아빠가 **언제나 안전한 장난감**인지 한눈에 확인해 주실 수 있답니다!
