# 754. 테일러링 프로젝트 맞춤형 프로세스 재단

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 조직의 표준 소프트웨어 개발 프로세스를 그대로 따르는 것이 아니라, 특정 프로젝트의 규모, 성격, 기술적 제약, 위험도를 고려하여 불필요한 절차를 제거하거나 필요한 부분을 강화하는 **프로세스 최적화 활동**이다.
> 2. **방법론**: 조직 차원의 **표준 프로세스(OSP)**를 기반으로, 정의된 **테일러링 가이드라인**과 승인 절차를 거쳐 해당 프로젝트만을 위한 '정의된 프로세스'를 산출한다.
> 3. **가치**: 불필요한 산출물 제작에 따르는 행정적 낭비를 줄이고, 프로젝트의 핵심 리스크에 자원을 집중함으로써 개발 생산성과 품질을 동시에 확보한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "모두에게 맞는 옷은 없다 (No One-Size-Fits-All)"

대형 금융 시스템을 만드는 프로세스를 3명 규모의 간단한 사내 툴 개발에 그대로 적용한다면, 배보다 배꼽이 더 큰 상황이 벌어집니다. 반대로 간단한 웹 사이트 개발 프로세스로 원자력 발전소 제어 시스템을 만든다면 재앙이 될 것입니다. **테일러링 (Tailoring)**은 "공학적 엄격함"과 "실무적 효율성" 사이의 최적 지점을 찾는 **'재단(Sewing)'**의 기술입니다.

### 💡 비유: 기성복 수선과 맞춤 정장

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        테일러링(Tailoring) 비유                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 파티에 입고 갈 멋진 옷(프로세스)이 필요함.                                │
│                                                                             │
│  1. 표준 프로세스 (백화점 기성복):                                            │
│     누구에게나 맞도록 평균적인 크기로 만들어진 옷. (조직의 공통 표준)              │
│                                                                             │
│  2. 테일러링 (수선하기):                                                     │
│     - 바지가 너무 길면 줄임 (불필요한 단계 삭제).                                │
│     - 허리가 크면 줄이고 단추를 더 닮 (필요한 보안 검증 추가).                    │
│                                                                             │
│  → 내 몸(프로젝트의 특성)에 딱 맞춰서, **가장 편하고 멋지게 일할 수 있게**         │
│     옷(과정)을 고쳐 입는 것!                                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 테일러링 수행 프로세스 (Steps)
1. **표준 프로세스 검토**: 조직의 공통 개발 표준(OSP) 파악.
2. **프로젝트 특성 분석**: 규모, 난이도, 기간, 기술 스택, 리스크 식별.
3. **가이드라인 적용**: 사전에 정의된 '테일러링 매트릭스'를 보고 가감(Add/Drop) 결정.
4. **테일러링 승인**: 품질 관리 부서(QA)나 PMO의 공식적인 검토 및 승인.
5. **실행 및 환류**: 실제 적용 후 문제점을 다시 표준 가이드라인에 반영.

### 2. 테일러링 영향 요인 (Considerations)

| 분류 | 고려 사항 | 테일러링 예시 |
|:---:|:---|:---|
| **내적 요인** | 프로젝트 규모, 예산, 기간 | 소규모인 경우 중복된 검토 회의 통합 |
| | 기술적 난이도, 팀 숙련도 | 생소한 기술인 경우 R&D 및 기술 검토 단계 강화 |
| **외적 요인** | 법적 규제, 보안 요구사항 | 금융 사업인 경우 시큐어 코딩 및 감리 단계 의무화 |
| | 고객의 프로세스 수준 | 고객사 표준 양식에 맞춘 산출물 통합 |

### 3. 테일러링 아키텍처 구조도 (ASCII)

```text
    [ Organizational Standard Process ] (OSP)
           │
           ▼ [ Tailoring Guideline ] (Rule Book)
    ┌──────────────────────────────────────────────┐
    │           TAILORING DECISION LOGIC           │ <── Project Context Input
    │  - If Scale < Small: Skip Document A         │     (Risk, Size, Tech)
    │  - If Security == High: Add Step B           │
    └──────────────────────┬───────────────────────┘
                           │
                           ▼
    [ Defined Project Process ] ──▶ [ Quality Gate & Audit ]
    (The Customized Road)           (Verification)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 프로세스 성숙도(CMMI)와의 관계
CMMI 레벨 3(Defined) 이상을 달성하기 위한 필수 요건이 바로 **'조직 표준 프로세스의 테일러링'**입니다. 단순히 표준이 있는 것에 그치지 않고, 현장에 맞게 변형하여 쓸 수 있는 체계를 갖추었는지가 성숙도의 잣대가 됩니다.

### 2. 기술적 시너지: DevOps 파이프라인
현대의 테일러링은 Jenkins나 GitHub Actions의 **'워크플로우 템플릿'**으로 구현됩니다. 프로젝트 생성 시 옵션(예: `--security-high`)을 선택하면 해당 프로젝트만을 위한 자동화된 빌드/테스트 파이프라인이 즉시 재단되어 생성되는 'Automated Tailoring'이 보편화되고 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 사내 시스템의 클라우드 마이그레이션
- **상황**: 기존 워터폴 기반 표준 프로세스는 너무 무거워 클라우드 전환의 속도를 못 따라감.
- **결단**: **클라우드 특화 테일러링 실시**. 
- **판단**: 
    1. 문서 위주의 '설계 단계'를 'IaC 코드 리뷰'로 대체 (Drop & Add). 
    2. 2주 주기의 스프린트 회고를 공식적인 프로젝트 보고로 인정. 
    3. 대신 '보안 그룹 설정 검증' 절차를 기존보다 강화.
- **효과**: 행정 오버헤드 30% 감소 및 클라우드 네이티브 기술의 유연성 확보.

### 📢 기술사적 결언
> "테일러링은 **'책임 있는 자율성'**의 실현이다. 프로세스를 줄이는 것은 '귀찮아서'가 아니라 '중요한 것에 집중하기 위해서'여야 한다. 아키텍트는 테일러링 과정에서 발생할 수 있는 **'품질의 사각지대'**를 정확히 식별해야 하며, 특히 삭제된 절차가 나중에 법적/품질적 리스크로 돌아오지 않도록 **'대체 통제 수단'**을 마련하는 정교한 재단 능력을 보여주어야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **생산성**: 부적절한 절차 제거로 불필요한 인건비 20~40% 절감.
- **품질 만족도**: 현장 상황이 반영된 프로세스로 인해 실행력 및 구성원 만족도 향상.

### 미래 전망
미래의 테일러링은 **'AI 기반 동적 프로세스 최적화'**로 진화할 것입니다. 프로젝트의 실시간 진척도와 결함 발생 데이터를 AI가 모니터링하여, 현재 진행 상황에 맞춰 "테스트 단계를 보강하십시오" 혹은 "문서화를 간소화해도 안전합니다"라고 프로세스를 실시간으로 재조정하는 'Adaptive Process Management'가 표준이 될 것입니다.
