# 794. 지속적 배포 롤백 자동화 정책 파이프라인 구성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 배포 후 장애가 감지되었을 때, 인간의 개입 없이 시스템이 스스로 이전의 안정적인 버전으로 되돌리는 **자동화된 장애 복구 메커니즘**이다.
> 2. **트리거 설계**: 에러율, 응답 지연 시간, 자원 사용량 등 핵심 품질 지표(SLI)가 사전에 정의된 임계치를 벗어날 경우, 파이프라인이 즉시 **롤백 워크플로우**를 실행하도록 구성한다.
> 3. **가치**: 장애 복구 시간(MTTR)을 최소화하여 서비스 가용성을 극대화하고, 배포 실패에 대한 개발자의 심리적 부담을 줄여 실험적이고 빠른 릴리즈 문화를 정착시킨다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "배포는 순식간, 복구는 한참?"

자동 배포(CD) 기술로 코드가 운영 환경에 나가는 시간은 매우 짧아졌습니다. 하지만 배포 후 장애가 났을 때, 사람이 로그를 보고 원인을 파악한 뒤 수동으로 롤백 명령을 내리는 과정은 너무 느립니다. **롤백 자동화**는 "기계가 배포했으면, 문제 시 기계가 거둬들여야 한다"는 사상에서 출발했습니다.

### 💡 비유: 자동 브레이크가 달린 기차

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        롤백 자동화 정책 비유                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 기차(소프트웨어)가 새로운 선로(운영 환경)로 진입하려 함.                 │
│                                                                             │
│  1. 수동 방식:                                                               │
│     사고가 나면 관제사가 보고를 받고, 한참 고민한 뒤에 후진 명령을 내림.           │
│     (그동안 기차는 이미 탈선하여 큰 피해 발생)                                   │
│                                                                             │
│  2. 롤백 자동화 방식:                                                        │
│     기차 앞부분에 센서(Monitoring)를 닮.                                       │
│     - 만약 바퀴가 조금이라도 흔들리거나 연기가 나면(에러 발생),                  │
│     - 사람이 시키지 않아도 기차가 **스스로 즉시 멈추고** 안전한 이전 역으로        │
│       뒤로 돌아감 (Automatic Rollback).                                       │
│                                                                             │
│  → 사고가 커지기 전에 **시스템이 알아서 도망치는** 똑똑한 방어 기전!              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 자동 롤백 정책의 3대 요소
- **Health Check**: 앱의 생존 여부와 비즈니스 정상 동작 여부를 확인하는 지표.
- **Threshold (임계치)**: 롤백을 실행할 구체적인 기준 (예: 5xx 에러율 > 1% 지속).
- **Rollback Strategy**: 이전 버전으로 교체하는 기술적 방법 (Blue-Green, Canary 등).

### 2. 롤백 트리거 지표 (SLI) 예시

| 지표 | 임계치 예시 | 비고 |
|:---:|:---|:---|
| **HTTP 5xx Error** | 최근 5분간 1% 초과 | 즉각적인 로직 오류 의심 |
| **Response Latency** | P99 지연 시간 2초 초과 | 자원 고갈 또는 DB 병목 의심 |
| **Success Rate** | 99% 미만으로 하락 | 비즈니스 로직 결함 |
| **Custom Metric** | 주문 전환율 50% 하락 | 기획/UX 결함 탐지 |

### 3. 자동 롤백 파이프라인 구조 (ASCII)

```text
    [ CI/CD Pipeline ] ──▶ [ Deployment ] ──▶ [ Smoke Test ]
                                 │               │
                                 │               ▼ (Success)
                                 │        [ Active Monitoring ] (5-15 min)
                                 │               │
                                 │               ▼ [ Anomaly Detected! ]
                                 │        ┌────────────────────────────┐
                                 │        │    Rollback Controller     │
                                 │        │ (Switch traffic to OLD v.) │
                                 └────────┼────────────────────────────┘
                                          ▼
                                   [ Old Version Restore ] ──▶ [ Alert Team ]
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 수동 롤백 vs 자동 롤백

| 항목 | 수동 롤백 (Manual) | 자동 롤백 (Automated) |
|:---:|:---|:---|
| **판단 주체** | 사람 (Ops/Dev) | **시스템 (Controller)** |
| **복구 시간** | 분~시간 단위 | **초~분 단위** |
| **정확도** | 상황 판단에 따라 가변적 | **정해진 정책에 따라 일관적** |
| **심리적 장벽** | 배포를 두려워하게 됨 | **자신감 있는 빈번한 배포 가능** |

### 2. 기술적 시너지: Blue-Green Deployment
Blue-Green 방식은 롤백 자동화의 '치트키'입니다. 구 버전(Blue)을 끄지 않고 살려둔 채 신 버전(Green)으로 트래픽만 돌리기 때문에, 문제 발생 시 로드밸런서 설정만 다시 Blue로 바꾸는 것만으로 즉시 롤백이 완료됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 글로벌 핀테크 서비스의 야간 배포
- **상황**: 새벽 2시에 신규 기능을 배포했는데, 담당자가 잠든 사이 미세한 메모리 누수로 서버가 느려짐.
- **결단**: **ArgoCD의 자동 롤백 기능 활성화**. 
- **판단**: 쿠버네티스의 'Health Check'가 실패하거나, 프로메테우스(Prometheus)의 '에러 지표'가 임계치를 넘으면 즉시 이전 ReplicaSet으로 원복.
- **효과**: 담당자가 아침에 출근해서 로그를 보기 전에, 이미 새벽 2시 10분에 시스템이 스스로 롤백을 끝내고 평온을 되찾음. 고객 불만 제로.

### 📢 기술사적 결언
> "자동 롤백은 **'아키텍처의 겸손함'**을 기술로 표현한 것이다. 우리가 만든 코드가 완벽하지 않을 수 있음을 인정하고, 실패했을 때의 **'퇴로'**를 미리 닦아두는 것이다. 아키텍트는 롤백 트리거를 설계할 때 '가짜 경보(False Positive)'로 인한 불필요한 롤백을 방지하기 위해, 지표의 **'지속 시간(Duration)'**과 **'샘플링 크기'**를 정교하게 튜닝해야 한다. 복구 자동화는 이제 운영의 효율을 넘어 브랜드의 신뢰도를 결정하는 핵심 요소다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **복구 시간 (MTTR)**: 수동 대응 대비 90% 이상 단축.
- **가용성 증대**: 장애 노출 시간 최소화로 연간 가용성 99.99% 달성 기여.

### 미래 전망
미래의 롤백 정책은 **'AIOps 기반 문맥 인지형 롤백'**으로 진화할 것입니다. 고정된 숫자가 아니라, 현재 유저 트래픽의 특성과 시간대를 고려하여 AI가 "지금 이 에러율은 무시해도 되는 일시적 현상입니다" 혹은 "지금 당장 롤백해야 합니다"를 판단하는 수준에 도달할 것입니다. 또한 **GitOps**와 결합하여, 롤백이 발생하면 자동으로 Git 저장소의 상태(Revision)까지 이전으로 돌려놓는 '완전 무결한 상태 일치' 체계가 구축될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **롤백 자동화**는 자전거를 배울 때 뒤에서 아빠가 잡아주는 **"안전 손잡이"**와 같아요.
2. 내가 중심을 잃고 넘어지려고 하면, 아빠가 생각할 틈도 없이 **즉시 자전거를 똑바로** 세워주는 거죠.
