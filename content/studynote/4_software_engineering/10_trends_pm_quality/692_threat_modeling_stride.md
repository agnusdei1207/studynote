# 692. 위협 모델링 STRIDE

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 설계 단계에서 발생할 수 있는 잠재적인 보안 위협을 구조화된 방법론을 통해 사전에 식별하고, 이에 대한 대응 전략을 수립하는 **선제적 보안 설계 활동**이다.
> 2. **프레임워크**: 마이크로소프트가 고안한 **STRIDE (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege)** 모델을 사용하여 공격자의 관점에서 시스템의 약점을 분류한다.
> 3. **가치**: 보안 사고 발생 후 패치를 수행하는 사후 대응 방식에서 벗어나, 아키텍처 수준에서 근본적인 보안 결함을 제거함으로써 보안 비용을 최소화하고 시스템의 신뢰성을 확보한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "보안은 기능이 아니라 설계다"

대부분의 보안 사고는 구현상의 실수보다 '잘못된 설계'에서 기인합니다. 데이터가 암호화되지 않은 채 흐르거나, 적절한 권한 검증 없이 기능을 노출하는 등의 문제가 설계 단계에서 걸러지지 않으면 구현 단계의 보안 노력은 무의미해집니다. **위협 모델링(Threat Modeling)**은 시스템의 '지도(DFD)'를 펴놓고 공격자가 어디를 노릴지 미리 시뮬레이션하는 전략 회의입니다.

### 💡 비유: 은행 건물의 도면 검토

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        위협 모델링(STRIDE) 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 새로운 은행 건물을 지으려고 함.                                         │
│                                                                             │
│  1. DFD 작성: 돈이 정문으로 들어와서(Input), 창구를 거쳐(Process), 금고로        │
│     들어가는(Storage) 길을 도면에 그림.                                        │
│                                                                             │
│  2. STRIDE 분석 (공격자 시뮬레이션):                                          │
│     - "강도가 직원인 척 변장하면?" (Spoofing)                                  │
│     - "누가 금고 열쇠를 복사해가면?" (Tampering)                                │
│     - "돈을 훔치고도 안 훔쳤다고 발뺌하면?" (Repudiation)                       │
│     - "밖에서 금고 안이 다 보이면?" (Information Disclosure)                   │
│                                                                             │
│  → 건물을 짓기 전, 도면을 보고 **CCTV와 이중 잠금장치**를 어디에 둘지 정하는 것!    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. STRIDE 모델의 6대 위협 요소

| 약어 | 위협 유형 | 대응되는 보안 속성 | 핵심 내용 |
|:---:|:---|:---:|:---|
| **S** | **Spoofing (변조)** | 인증 (Authentication) | 타인의 아이디/암호나 시스템 권한을 도용 |
| **T** | **Tampering (변조)** | 무결성 (Integrity) | 전송 중이거나 저장된 데이터를 무단 수정 |
| **R** | **Repudiation (부인)** | 부인 방지 (Non-repudiation) | 자신의 행위를 증명할 수 없음을 악용하여 부정 |
| **I** | **Info Disclosure (유출)** | 기밀성 (Confidentiality) | 권한 없는 자에게 민감한 정보가 노출됨 |
| **D** | **DoS (서비스 거부)** | 가용성 (Availability) | 시스템 자원을 고갈시켜 정상 서비스를 방해 |
| **E** | **Elevation of Privilege** | 권한 부여 (Authorization) | 일반 사용자가 관리자 권한을 획득하여 접근 |

### 2. 위협 모델링 4단계 프로세스
1. **Model (시스템 모델링)**: 데이터 흐름도(DFD)를 통해 시스템 구조 파악.
2. **Identify (위협 식별)**: DFD의 각 요소(프로세스, 저장소, 흐름)에 대해 STRIDE 적용.
3. **Mitigate (위협 완화)**: 식별된 위협에 대한 보안 대책(Controls) 수립.
4. **Validate (검증)**: 위협이 적절히 해결되었는지 리뷰하고 문서화.

### 3. 위협 식별 아키텍처 (ASCII)

```text
    [ User ] ──(1) Credential ──▶ [ Web Server ] ──(2) Query ──▶ [ Database ]
       │                             │                             │
       ▼ 위협 분석                    ▼ 위협 분석                    ▼ 위협 분석
    (S) 세션 도용 가능성?          (T) 파라미터 변조?             (I) 평문 저장?
    (R) 로그인 기록 없음?          (D) 과부하 공격?               (E) DB 관리자 탈취?
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. STRIDE vs DREAD (위협 우선순위 산정)
- **STRIDE**: "어떤 종류의 위험이 있는가?"를 **식별**하는 모델.
- **DREAD**: 식별된 위험의 **심각도(Score)**를 측정하는 모델 (Damage, Reproducibility, Exploitability, Affected users, Discoverability).

### 2. 기술적 시너지: 시큐어 SDLC (Secure SDLC)
위협 모델링은 시큐어 SDLC의 **설계(Design)** 단계에서 수행되는 가장 중요한 활동입니다. 여기서 작성된 '위협 리포트'는 이후 구현 단계의 '시큐어 코딩' 기준이 되며, 테스트 단계의 '침투 테스트(Pentest)' 시나리오로 활용됩니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 모바일 뱅킹 송금 기능 설계
- **상황**: 스마트폰에서 돈을 보내는 기능을 새로 만듦.
- **결단**: **STRIDE 기반 설계 검토**. 
- **판단**: 
    - (T) 송금 금액을 가로채서 바꿀 수 있는가? → **메시지 무결성 체크(HMAC)** 도입.
    - (R) "나는 송금한 적 없다"고 우기면? → **디지털 서명(PKI)** 기반 로그 기록.
- **효과**: 개발 착수 전 핵심 보안 장치들을 설계에 반영하여, 나중에 아키텍처를 뒤엎는 대공사 방지.

### 📢 기술사적 결언
> "위협 모델링은 공격자의 머리로 생각하고 방어자의 손으로 설계하는 **'두뇌 싸움'**이다. 보안은 체크리스트를 지키는 행위가 아니라, 시스템의 모든 '경계(Trust Boundary)'를 의심하고 검증하는 과정이어야 한다. 아키텍트는 STRIDE를 통해 시스템의 약점을 투명하게 공개하고, 비즈니스 리스크를 수치화하여 경영진의 의사결정을 지원하는 전략가적 면모를 갖추어야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **설계 결함 감소**: 구조적 보안 취약점의 70% 이상을 초기 단계에서 차단.
- **보안 가시성**: 시스템 전체의 위협 지도를 확보하여 전사적 보안 거버넌스 강화.

### 미래 전망
미래의 위협 모델링은 **'자동화된 위협 감지 (Automated Threat Modeling)'**로 진화합니다. 클라우드 아키텍처 도면(IaC 코드 등)을 입력하면 AI가 STRIDE 데이터베이스와 대조하여 자동으로 취약 지점을 찍어주고 대응 코드를 제안하는 'Security-as-Code' 체계가 보편화될 것입니다. 또한 진화하는 공격 기법을 실시간 학습하여 위협 모델을 스스로 갱신하는 지능형 설계 도구들이 등장할 것입니다.
