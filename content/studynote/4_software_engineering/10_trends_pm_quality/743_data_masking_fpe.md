# 743. 데이터 마스킹 FPE 암호 유지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 민감 정보(Passenger Data)의 기밀성을 유지하면서도 데이터베이스(Database) 스키마 호환성을 보장하는 **데이터 마스킹(Data Masking)**과 **FPE (Format-Preserving Encryption, 형식 보존 암호화)** 기술은 근본적으로 '데이터의 가용성(Availability)과 보안성(Security)의 트레이드오프'를 해결하는 아키텍처적 패러다임입니다.
> 2. **가치**: 레거시 시스템(Legacy System)의 수정 비용을 최소화하면서 PCI-DSS 등 규정 준수(Compliance)를 만족시키며, 암호화된 상태에서도 인덱싱(Indexing)과 검색(Search)이 가능하도록 하여 운영 효율성을 극대화합니다.
> 3. **융합**: 토큰화(Tokenization), 차등 프라이버시(Differential Privacy), K-익명성(K-Anonymity) 등 프라이버시 공학 기술과 결합하여 데이터 생애 주기(Data Lifecycle) 전반에 걸쳐 'Self-Defending Data' 구현을 위한 핵심 인프라로 진화하고 있습니다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 철학**
**데이터 마스킹(Data Masking)**은 원본 데이터의 형식을 유지하되 특정 필드 값을 무의미한 값으로 교체하여 민감 정보를 노출시키지 않는 기술입니다. 여기에는 복구가 불가능한 파괴적 기법과 키를 통한 복구가 가능한 기술이 포함됩니다. 특히 **FPE (Format-Preserving Encryption)**는 암호화(Ciphertext)된 결과물이 평문(Plaintext)과 동일한 길이와 문자 집합(Alphabet)을 가지도록 설계된 대칭키 암호 시스템의 일종으로, 데이터베이스 제약 조건(Constraint)을 위배하지 않는 것이 핵심 철학입니다. 이는 NIST SP 800-38G 표준으로 규정되어 있습니다.

**2. 등장 배경: 암호화의 역설과 데이터 활용의 갈등**
기존의 블록 암호화 알고리즘인 AES (Advanced Encryption Standard)나 DES (Data Encryption Standard)는 데이터를 랜덤한 바이너리 스트림으로 변환합니다. 예를 들어 16자리 신용카드 번호(ASCII)는 AES-256 암호화 후 Base64 인코딩을 거치면 32자리 이상의 임의 문자열로 변형됩니다. 이는 데이터베이스 스키마 변경(VarChar(16) → VarChar(64)), 애플리케이션 로직 수정, 인덱스 깨짐 현상 등 막대한 비용을 유발합니다. 금융, 의료 등 레거시 시스템이 강력한 산업군은 "데이터는 안전하게 숨기되, 시스템은 바꾸고 싶지 않다"는 요구를 가지게 되었고, 여기서 탄생한 것이 형식 보존 기술입니다.

**3. 기술적 진화**
초기에는 단순 문자열 치환(Scrambling)이나 마스킹(Masking, 예: `1234-5678` → `1234-****`)이 주를 이루었습니다. 이는 보안성이 낮고 연산이 불가능하다는 한계가 있었습니다. 이후 수학적 난수 생성과 Feistel 네트워크 구조를 기반으로 한 FPE가 개발되었고, 현재는 클라우드(Cloud) 환경 및 실시간 데이터 스트리밍 처리를 위한 고성능 마스킹 솔루션으로 발전하고 있습니다.

**📢 섹션 요약 비유**
> 마치 영화 촬영용 가짜 지폐(Bill Prop)와 같습니다. 진짜 돈과 똑같은 크기와 무게(Format)를 가지고 있어서 지갑(구조/DB)을 바꿀 필요가 없지만, 실제 가치는 없어서 도둑이 훔쳐 가도 무용지물(Masking)이 되는 원리입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. 구성 요소 및 모듈 분석**
데이터 마스킹 및 FPE 시스템은 크게 정책 관리, 마스킹 엔진, 키 관리, 데이터 탐지 모듈로 구성됩니다.

| 요소 명칭 | 역할 (Role) | 내부 동작 메커니즘 | 주요 프로토콜/표준 | 비유 |
|:---:|:---|:---|:---|:---|
| **정책 관리자** | 마스킹 규칙 정의 | 데이터 타입(카드, 이메일)별 알고리즘 선택, 권한 부여 | RBAC (Role-Based Access Control) | 교통 정책 관제소 |
| **데이터 탐지기** | 민감 정보 식별 | Regex, NLP를 통해 데이터 내 필드 패턴 매칭 | Regex, XML Path | 금속 탐지기 |
| **마스킹 엔진 (FPE)** | 실시간 변환 처리 | Feistel 네트워크 기반 라운드 함수(Round Function) 수행 | NIST SP 800-38G (FF1, FF3-1) | 암호 변환 기계 |
| **키 관리자 (KMS)** | 암호화 키 생명주기 관리 | HSM(Hardware Security Module)과 연동하여 키 생성/폐기/저장 | KMIP (Key Management Interoperability Protocol) | 보안 금고 |
| **감사 로그 (Audit)** | 접근 기록 및 추적 | 누가, 언제, 어떤 필드를 평문으로 볼 수 있었는지 기록 | Syslog, SIEM 연동 | 블랙박스 |

**2. FPE (Format-Preserving Encryption) 심층 구조**
FPE의 핵심은 **Feistel 네트워크**를 사용하여 도메인 보존 암호화를 구현하는 것입니다. 평문 공간 $X$를 두 부분 $L_0$과 $R_0$으로 나누고, 라운드 함수 $F$를 반복하여 암호문을 생성합니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    FPE(FF1 Algorithm) 동작 구조도                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [입력] 평문 (Base10, 16자리 신용카드 번호)                                   │
│     │                                                                       │
│     ▼                                                                       │
│  ┌───────────────────┐                                                     │
│  │   1. 전처리        │  숫자 -> Radix 변환 (Base10 데이터 처리)              │
│  └───────────────────┘                                                     │
│     │                                                                       │
│     ▼                                                                       │
│  ┌───────────────────────────────────────────────────────┐                  │
│  │              2. Feistel Network (10 Rounds)            │                  │
│  │  ┌─────────┐    ┌───────────────────┐                  │                  │
│  │  │  Round 1│───>│ L(i) = R(i-1)     │                  │                  │
│  │  └─────────┘    │ R(i) = L(i-1) ⊕ F(K, R(i-1))        │                  │
│  │       ...       │      (AES 기반 F 함수)                │                  │
│  │  ┌─────────┐    └───────────────────────────────────┘                  │
│  │  │  Round N│                                            │                  │
│  │  └─────────┘                                            │                  │
│  └───────────────────────────────────────────────────────────┘                  │
│     │                                                    │                  │
│     ▼                                                    ▼                  │
│  [출력] 암호문 (여전히 Base10, 16자리, 검증 비포함)                            │
│                                                                             │
│  특징:                                                                   │
│  - Format(자릿수, 문자 집합)이 수학적으로 보장됨.                               │
│  - Tud(Tweaking) 사용: 카드 번호의 첫 6자리(BIN)는 그대로 유지 가능.              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
*(해설: 위 다이어그램은 Feistel 구조를 통해 데이터의 길이와 도메인을 보존하면서 암호 강도를 확보하는 과정을 도식화한 것입니다.)*

**3. 핵심 알고리즘 및 수식 (FF1 Mechanism)**
FPE의 대표적인 표준인 **FF1 (Feistel-based Encryption Mode 1)**은 AES 암호화를 내부 함수로 사용합니다. $n$자리 문자열 $X$를 암호화하는 과정은 다음과 같습니다.

1.  **분할**: $X$를 $u$비트와 $v$비트로 분할 ($u = \lfloor n/2 \rfloor$).
2.  **라운드 함수**: $i$번째 라운드에서의 반올림(Modulo) 연산.
    $$R_i = (L_{i-1} + F(K, R_{i-1} \oplus i)) \pmod{10^m}$$
    여기서 $F(K, \cdot)$은 AES-CBC 기반의 PRF (Pseudo-Random Function)입니다.
3.  **스왑(Swap)**: $L_i = R_{i-1}$.

**코드 스니펫 (Python-like Pseudo-code)**

```python
def encrypt_fpe(plaintext, key, radix=10):
    """
    FPE 암호화 의사 코드
    :param plaintext: 암호화할 숫자 문자열 (예: "123456")
    :param key: AES 암호화 키
    :param radix: 진법 (신용카드는 10진법)
    """
    # 1. 문자열을 정수 리스트로 변환
    x = [int(c) for c in plaintext]
    num_rounds = 10
    
    # 2. Feistel Network 라운드 수행
    for i in range(num_rounds):
        # 왼쪽, 오른쪽 분할
        left, right = split(x)
        
        # 라운드 함수 F: AES + Round Number를 이용한 의사 난수 생성
        round_function_val = aes_encrypt(key, right + i)
        
        # 반올림 연산 (Format Preserving의 핵심: 같은 자릿수 유지)
        # (left + round_function_val) % radix^len(left)
        new_right = (left + round_function_val) % (radix ** len(left))
        
        # 상태 업데이트
        x = right + new_right
        
    return "".join(map(str, x))
```

**4. 데이터 마스킹 유형 및 동작 원리**
*   **동적 마스킹 (Dynamic Data Masking)**: 데이터베이드 드라이버(JDBC/ODBC) 레벨이나 프록시(Proxy) 계층에서 작동. 실제 데이터는 저장소에 그대로 있으나, 사용자 조회 시 `SELECT` 결과값을 가로채어 마스킹하여 전달.
*   **정적 마스킹 (Static Data Masking)**: 개발/테스트 환경 구축 시, DDL(Data Definition Language) 스크립트와 함께 DML(Data Manipulation Language)을 통해 데이터 자체를 영구적으로 변환하여 복사.

**📢 섹션 요약 비유**
> 복잡한 스위치 보드에서 전선(데이터)의 색깔과 길이는 그대로 둔 채, 내부 연결만 빠르게 바꿔 다는 것과 같습니다. 겉보기에는 케이블의 형태(Format)가 변하지 않았지만, 실제 신호가 전달되는 경로(Logic)는 완전히 다른 값으로 치환되어 보안을 유지합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 심층 기술 비교 분석표**

| 구분 | AES (Advanced Encryption Standard) | 토큰화 (Tokenization) | FPE (Format-Preserving Encryption) |
|:---:|:---|:---|:---|
| **데이터 형식** | 변형됨 (Binary/Blob) | 변형 가능 (Random Token) | **완벽히 유지** (Length, Charset) |
| **복호화 가능성** | 가능 (Key 사용) | 가능 (Vault Lookup) | **가능 (Key 사용, 계산적)** |
| **검색(Index) 성능** | 낮음 (Decrypt 필요) | 중간 (Join 필요) | **높음 (직접 Index 사용)** |
| **보안 강도** | 매우 높음 (최적화됨) | 매우 높음 (Vault에 의존) | 높음 (AES 기반) |
| **구현 복잡도** | 낮음 (Library 지원) | 높음 (DB 관리/동기화) | 중간 (Key Rotation 주의) |
| **Latency** | 낮음 (<1ms) | 중간 (Disk I/O 발생) | 낮음 (<2ms) |

**2. 데이터 융합 분석: 보안 vs AI 분석**

*   **시나리오 A (통계 분석)**: 통계적 특성을 유지해야 하는 머신러닝 학습 데이터의 경우, FPE는 데이터 분포(Distribution)를 왜곡합니다. 이 때는 **데이터 마스킹**이나 **K-익명성(K-Anonymity)** 기반의 일반화가 더 유리합니다.
*   **시나리오 B (연산 처리)**: 암호화된 상태로 덧셈/곱셈이 필요한 경우 **동형암호(Homomorphic Encryption)**와 비교됩니다. FPE는 연산이 불가능하지만, 형식을 유지한다는 점에서 시스템 통합 관점에서 훨씬 실용적입니다.
