# 771. BDD Given-When-Then 행동 명세 테스트

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TDD에서 파생된 개발 방법론으로, 기술적인 구현보다는 사용자의 **행동(Behavior)**과 비즈니스 가치에 집중하여 "시스템이 어떻게 동작해야 하는가"를 자연어에 가까운 형식으로 정의하고 테스트한다.
> 2. **표준 형식**: **Given(전제 조건), When(발생 사건), Then(기대 결과)**으로 이어지는 Gherkin 구문을 사용하여, 비즈니스 이해관계자와 개발자 간의 언어적 장벽을 허문다.
> 3. **가치**: 요구사항 명세가 곧 실행 가능한 테스트 코드가 되는 '살아있는 문서'를 구축하며, 개발 초기 단계에서 비즈니스 로직의 오해를 원천 차단하여 협업의 효율성을 극대화한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "코드는 맞는데, 사용자가 원하는 게 아니에요"

전통적인 테스트는 개발자만의 언어였습니다. 기획자는 테스트 코드를 읽을 수 없었고, 개발자는 기획 의도를 잘못 해석하기 일쑤였습니다. 댄 노스(Dan North)가 제안한 **BDD (Behavior-Driven Development)**는 "테스트(Test)"라는 단어 대신 "행동(Behavior)"이라는 단어를 사용하여, 개발의 초점을 '검증'이 아닌 '명세와 소통'으로 옮겼습니다.

### 💡 비유: 레스토랑의 표준 주문서

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        BDD (행동 명세) 비유                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 손님이 요리사에게 복잡한 스테이크 주문을 함.                             │
│                                                                             │
│  1. 기존 방식: "고기를 잘 구워주세요." (요리사 맘대로 웰던으로 구움 - 버그 발생)    │
│                                                                             │
│  2. BDD 방식 (Given-When-Then 주문서):                                       │
│     - **Given (만약)**: 배고픈 손님이 테이블에 앉아 있고,                          │
│     - **When (언제)**: "안심 스테이크 미디움 레어"를 주문하면,                     │
│     - **Then (그러면)**: 겉은 바삭하고 속은 붉은 육즙이 흐르는 고기가 나와야 한다.   │
│                                                                             │
│  → 손님(기획)과 요리사(개발)가 **똑같은 기대치**를 가지고 요리를 시작하는 것!      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. BDD의 핵심 구조: Gherkin 구문

| 키워드 | 역할 | 상세 설명 |
|:---:|:---|:---|
| **Given** | **상황 설정** | 테스트가 시작되기 전의 시스템 상태나 전제 조건 정의 |
| **When** | **행동 수행** | 사용자가 시스템에 가하는 자극이나 핵심 동작 기술 |
| **Then** | **결과 확인** | 동작의 결과로 예상되는 시스템의 변화나 출력 확인 |
| **And / But**| **조건 추가** | Given, When, Then 단계에서 여러 조건을 덧붙일 때 사용 |

### 2. BDD 개발 라이프사이클
1. **Scenario 도출**: Three Amigos(기획, 개발, QA)가 모여 사용자 스토리 분석.
2. **Step Definition**: 자연어 시나리오를 실제 테스트 코드로 매핑 (Java, Python 등).
3. **Red-Green-Refactor**: 시나리오 통과를 위한 실제 비즈니스 로직 구현.

### 3. BDD 협업 아키텍처 (ASCII)

```text
    [ Business Requirement ]
           │
    ┌──────▼──────────────────────────────────┐
    │  User Story (자연어 시나리오 작성)         │ <── Given-When-Then
    └──────┬──────────────────────────────────┘
           │ (Mapping via Cucumber/SpecFlow)
    ┌──────▼──────────────────────────────────┐
    │  Test Automation (실행 가능한 명세서)      │
    └──────┬──────────────────────────────────┘
           │ (Logic Call)
    ┌──────▼────────────┐      ┌────────────────┐
    │  Production Code  │ ──▶  │  Business Value│
    └───────────────────┘      └────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. TDD vs BDD

| 항목 | TDD (Test-Driven) | BDD (Behavior-Driven) |
|:---:|:---|:---|
| **중심 질문** | "올바르게 구현했는가?" (How) | **"올바른 일을 하고 있는가?" (What)** |
| **언어** | 프로그래밍 언어 (기술적) | **자연어 (비즈니스적)** |
| **대상** | 개발자 | **기획자, 개발자, 테스터 전체** |
| **범위** | 메서드, 클래스 단위 | **기능, 사용자 경험 단위** |

### 2. 기술적 시너지: ATDD (인수 테스트 주도 개발)
BDD는 ATDD를 구현하는 가장 강력한 수단입니다. 고객의 인수 조건(Acceptance Criteria)을 BDD 시나리오로 작성하면, 별도의 문서 작업 없이도 자동화된 인수 테스트 체계가 완성됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 복잡한 할인 정책 엔진 개발
- **상황**: "주말이고, 첫 구매 고객이며, 5만 원 이상 구매 시 10% 할인"이라는 정책이 있음.
- **결단**: **Cucumber를 이용한 BDD 시나리오 작성**. 
- **판단**: 복잡한 조건문(`if-else`)은 오해를 낳기 쉬움. 
- **조치**: 시나리오 파일에 "Given 주말이고 첫 구매 고객임, When 6만 원 결제 시, Then 6,000원 할인됨"이라고 명시.
- **효과**: 기획자가 해당 파일을 읽고 "맞습니다"라고 컨펌한 뒤 개발에 착수하여 로직 오류 0건 달성.

### 📢 기술사적 결언
> "BDD는 **'번역의 고통'**을 해결하기 위한 공학적 응답이다. 비즈니스 언어를 기술 언어로 변환하는 과정에서 정보가 손실되는 것을 막아야 한다. 아키텍트는 BDD를 단순한 테스트 도구가 아닌, **'공통 언어(Ubiquitous Language)를 확립하는 수단'**으로 활용해야 한다. 시나리오가 곧 코드가 되고 코드가 곧 명세가 될 때, 소프트웨어는 비로소 비즈니스의 진정한 동반자가 된다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **재작업 감소**: 요구사항 오해로 인한 수정 비용 40% 이상 절감.
- **의사소통 효율**: 회의 시간 단축 및 명확한 완료 정의(DoD) 공유.

### 미래 전망
미래의 BDD는 **'AI 기반 시나리오 생성'**으로 진화할 것입니다. 비즈니스 미팅 음성을 듣고 AI가 즉시 Given-When-Then 시나리오를 도출하며, 개발자는 승인만 하면 코드 뼈대가 완성되는 'No-code BDD' 환경이 구축될 것입니다. 또한 사용자의 실시간 앱 사용 로그를 분석하여, 거꾸로 BDD 시나리오를 갱신해주는 '역방향 명세 기술'이 품질 관리의 표준이 될 것입니다.
