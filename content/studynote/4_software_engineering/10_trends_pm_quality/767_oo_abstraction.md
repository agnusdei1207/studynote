# 767. 객체지향 추상화 자료/제어/과정 분리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 복잡한 시스템의 불필요한 세부 사항은 숨기고 본질적인 특징만을 추출하여 모델링하는 기법으로, **자료, 제어, 과정**의 3가지 차원에서 시스템을 단순화한다.
> 2. **3대 차원**: 데이터의 형식을 정의하는 **자료 추상화**, 로직의 흐름을 구조화하는 **제어 추상화**, 복잡한 처리 단계를 하나의 이름으로 묶는 **과정(기능) 추상화**를 통해 결합도를 낮춘다.
> 3. **가치**: 현실 세계의 복잡한 문제를 컴퓨터가 처리 가능한 논리적 단위로 변환하며, 내부 구현이 바뀌어도 인터페이스가 유지되는 **유연한 확장성**을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "생각의 한계를 극복하는 법"

인간의 뇌는 한 번에 7개 내외의 정보만 처리할 수 있습니다. 수만 줄의 코드를 한꺼번에 이해하는 것은 불가능합니다. **추상화 (Abstraction)**는 복잡한 것을 '단순한 개념'으로 뭉뚱그려 인식하게 함으로써 우리가 거대한 시스템을 설계할 수 있게 해 줍니다. 객체지향은 이 추상화를 가장 체계적으로 실천하는 패러다임입니다.

### 💡 비유: 자동차 운전과 추상화

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        객체지향 3대 추상화 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 자동차를 운전하여 여행을 가려 함.                                       │
│                                                                             │
│  1. 자료 추상화 (계기판):                                                    │
│     엔진의 미세한 진동이나 기름의 화학 반응 대신, "속도: 80km/h", "연료: 50%"    │
│     라는 단순한 데이터(자료)만 보고 판단함.                                     │
│                                                                             │
│  2. 제어 추상화 (브레이크/액셀):                                              │
│     페달을 밟으면 내부적으로 어떤 유압 장치가 움직이는지는 몰라도 됨.             │
│     "밟으면 멈춘다/간다"는 논리적인 제어 규칙만 알면 됨.                        │
│                                                                             │
│  3. 과정 추상화 (내비게이션의 '길 찾기'):                                     │
│     수만 개의 교차로와 신호등 정보를 일일이 알 필요 없음. "서울역으로 가라"는     │
│     하나의 명령어(함수)로 복잡한 주행 과정 전체를 대표함.                       │
│                                                                             │
│  → 복잡한 기계 장치를 **단순한 조작법**으로 요약하는 것이 바로 추상화!            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 추상화의 3가지 범주 분석

| 범주 | 영문 명칭 | 핵심 내용 | 객체지향적 구현 |
|:---:|:---|:---|:---|
| **자료 추상화** | **Data Abstraction** | 데이터의 물리적 저장 방식 대신 논리적 구조 정의 | **클래스 (Class), 속성 (Property)** |
| **제어 추상화** | **Control Abstraction** | 프로그램의 실행 흐름을 구조화하여 복잡도 감소 | **if-else, Loop, 다형성 (Polymorphism)** |
| **과정 추상화** | **Process Abstraction** | 수행해야 할 상세 단계를 하나의 이름으로 캡슐화 | **메서드 (Method), 인터페이스 (Interface)** |

### 2. 추상화 수준 (Levels of Abstraction)
- **High Level**: 비즈니스 의도 표현 (예: `order.pay()`).
- **Low Level**: 구체적 기술 구현 (예: `db.execute("UPDATE...")`).
- **핵심**: 상위 수준은 하위 수준의 '어떻게(How)'를 몰라도 '무엇(What)'을 하는지 알 수 있어야 함.

### 3. 추상화 아키텍처 구조 (ASCII)

```text
    [ High Level: Concept ]  <── "무엇을 하는가?" (What)
           │
    ┌──────▼──────────────────────────────────┐
    │      Abstraction Layer (Interface)      │ <── 상세 구현 은닉
    └──────┬──────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────┐
    │      Implementation (Concrete Class)    │ <── "어떻게 하는가?" (How)
    │  - Data Mapping  - Step-by-step Logic   │
    └─────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 추상화(Abstraction) vs 캡슐화(Encapsulation)
- **추상화**: 외부에 **보여줄 특징**을 골라내는 '선택'의 과정. (디자인 타임)
- **캡슐화**: 선택된 특징을 하나로 묶고 **나머지를 감추는** '보호'의 과정. (구현 타임)
- **관계**: 추상화는 목적이고, 캡슐화는 그 목적을 달성하는 수단입니다.

### 2. 기술적 시너지: 디자인 패턴
디자인 패턴의 90%는 '추상화'를 극대화하는 기법입니다. 예를 들어 **팩토리 패턴**은 객체 생성의 '과정'을 추상화하고, **전략 패턴**은 알고리즘의 '제어'를 추상화하여 시스템의 결합도를 낮춥니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 다기종 프린터 드라이버 설계
- **자료 추상화**: 모든 프린터의 공통 속성(종이 잔량, 잉크 상태)을 `PrinterStatus` 객체로 정의.
- **과정 추상화**: 프린터마다 출력 방식이 다르지만 외부에는 `print(document)` 메서드 하나만 노출.
- **제어 추상화**: 다형성을 활용하여 `if(삼성) ... else if(HP) ...`를 제거하고 런타임에 적절한 드라이버가 호출되게 설계.
- **효과**: 신규 프린터 기종 추가 시 기존 인쇄 로직을 한 줄도 고치지 않고 확장 가능.

### 📢 기술사적 결언
> "추상화는 **'복잡성과의 전쟁'**에서 승리하기 위한 유일한 전략이다. 나쁜 코드는 항상 '어떻게'가 '무엇'을 압도한다. 아키텍트는 코드를 읽을 때 비즈니스 명세서처럼 읽히도록 추상화 수준을 높여야 한다. `List`라는 자료 추상화 하나가 수천 가지 데이터 구조의 세부 구현을 덮어주듯, 잘 설계된 추상화 계층은 시스템의 **'인지 부하'**를 줄여주는 최고의 아키텍처적 배려다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **가독성 향상**: 도메인 로직 파악 시간 50% 이상 단축.
- **변경 유연성**: 내부 구현 교체 시 클라이언트 코드 수정 0% 달성 가능.

### 미래 전망
미래의 추상화는 **'선언적 추상화'**로 진화할 것입니다. 코드를 직접 짜지 않고 '의도(Intent)'만 선언하면, AI가 최적의 자료와 과정을 스스로 선택하여 하위 수준의 구현을 채워 넣는 'Prompt-level Abstraction'이 대세가 될 것입니다. 또한 분산 환경에서 네트워크 지연을 추상화하는 **'서비스 메시'**와 같은 인프라 추상화 기술이 더욱 정교해질 것입니다.
