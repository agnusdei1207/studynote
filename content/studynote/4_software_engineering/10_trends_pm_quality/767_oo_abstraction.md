# 767. 객체지향 추상화 자료/제어/과정 분리

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어의 복잡성(Cognitive Complexity)을 통제하기 위해 시스템의 불필요한 세부 사항(Implementation Details)을 제거하고, **자료(Data)**, **제어(Control)**, **과정(Process)**의 3가지 차원에서 본질적인 구조만을 모델링하는 기법.
> 2. **구조**: **ADT(Abstract Data Type)**를 통한 자료 추상화, **다형성(Polymorphism)**을 통한 제어 추상화, **인터페이스(Interface)**를 통한 과정 추상화를 결합하여 시스템의 결합도(Coupling)를 최소화하고 응집도(Cohesion)를 극대화.
> 3. **가치**: 변경 관점의 유연성(Extensibility)을 제공하여, 내부 구현 로직의 변화가 외부 클라이언트에 파급되지 않도록 하는 **유지보수성(Maintainability)**의 핵심 설계 전략.

---

## Ⅰ. 개요 (Context & Background)

소프트웨어 공학에서 **추상화(Abstraction)**는 복잡한 현실 세계를 컴퓨터가 이해하고 처리할 수 있는 수학적 모델로 변환하는 핵심 메커니즘입니다. 인간의 인지 능력에는 한계가 있기에(밀러의 법칙: 7±2), 수만 개의 변수와 로직을 동시에 다룰 수 없습니다. 따라서 문제의 핵심적인 측면만을 강조하고 불필요한 세부 사항은 제거하는 '추상화'가 필수적입니다. 객체지향 패러다임(Object-Oriented Paradigm)은 이 추상화를 **'무엇(What)'**과 **'어떻게(How)'**로 분리하여 시스템을 계층화(Layering)하는 것을 철학으로 삼습니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                      추상화의 필요성: 인지 부하 감소                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [현실 세계의 복잡성]                                                        │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━         │
│  ▶ 불필요한 세부 사항(구현 방식, 물리적 배치, 메모리 주소 등)                 │
│                                                                             │
│           ▼  (Filtering: Abstraction)                                       │
│                                                                             │
│  [모델링 된 추상화 세계]                                                      │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━         │
│  ▶ 핵심 개념(속성, 기능, 관계)                                                │
│                                                                             │
│  → "사용자는 자동차의 내부 구조를 몰라도 운전(기능)할 수 있어야 한다."          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1. 정의 및 분류
추상화는 시스템을 바라보는 관점에 따라 크게 세 가지로 분류됩니다.
- **자료 추상화 (Data Abstraction)**: 데이터의 물리적 표현 방식이 아닌, 데이터가 제공하는 행위와 속성에 초점을 맞춤.
- **제어 추상화 (Control Abstraction)**: 프로그램의 실행 흐름(Control Flow)을 내부 로직의 복잡성 없이 단순화된 논리적 흐름으로 표현.
- **과정 추상화 (Process Abstraction)**: 특정 기능을 수행하는 일련의 절차를 하나의 명령어나 함수로 캡슐화.

### 2. 등장 배경 및 필요성
- **절차적 프로그래밍의 한계**: 데이터와 그 데이터를 처리하는 함수가 분리되어 있어, 데이터 구조가 변경되면 이를 사용하는 모든 로직(Procedure)을 수정해야 하는 '리스크(Ripple Effect)' 발생.
- **객체지향의 해결책**: 데이터와 데이터를 처리하는 과정을 하나의 객체(Object)로 묶음으로써, 외부에서는 내부 구현을 알지 못해도 상호작용할 수 있는 **블랙박스(Black Box)** 모델 구현.

> **📢 섹션 요약 비유**: 
> 마치 복잡한 고성능 카메라를 사용할 때, 사용자는 렌즈 내부의 광학 원리나 센서의 아날로그-디지털 변환 과정(어떻게)을 이해하지 못해도, '촬영' 버튼(무엇)만 누르면 사진이 찍히는 것과 같습니다. 추상화는 복잡한 내부 메커니즘을 숨기고 사용자에게 필요한 '인터페이스'만 단순화하여 제공하는 과정입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

객체지향에서 추상화는 단순한 개념적 도구를 넘어 클래스 설계(Class Design)의 기본 틀을 제공합니다. 이를 구체적으로 구현하기 위해 **ADT(Abstract Data Type)**, **인터페이스(Interface)**, 그리고 **폴리모피즘(Polymorphism)**이 활용됩니다.

### 1. 구성 요소 및 상세 동작

| 구성 요소 (Component) | 영문 명칭 | 역할 (Role) | 내부 동작 및 프로토콜 | 비유 |
|:---|:---|:---|:---|:---|
| **자료 추상화** | **Data Abstraction** | 데이터 구조의 은폐 | `private` 필드로 데이터 보호, `public` 메서드(Setter/Getter)를 통한 접근 제어 | 계기판 |
| **제어 추상화** | **Control Abstraction** | 실행 흐름의 단순화 | 조건문(`if`), 반복문(`loop`) 등의 제어 구조 캡슐화 및 다형성을 통한 런타임 로직 교체 | 자동변속기 |
| **과정 추상화** | **Process Abstraction** | 기능의 단위화 | 함수(Function)나 메서드(Method)로 로직을 그룹화하고 매개변수(Parameter)만으로 통제 | 명령 버튼 |

### 2. 추상화 계층 구조 (Layered Abstraction)

시스템은 일반적으로 하위 수준의 구체적인 구현(Concrete Layer)이 상위 수준의 추상적인 인터페이스(Abstract Layer)에 의해 덮이는 **온전-분리(Separation of Concerns)** 구조를 가집니다.

```text
    [ High Level: Business Logic ]  <-- 사용자가 보는 세계
           │
           │   (Client) Interface Abstraction
           │   └── "print(doc)" 호출
    ┌──────▼──────────────────────────────────┐
    │         Abstraction Layer               │ 
    │  ┌────────────────────────────────────┐ │
    │  │ <<interface>> Printer             │ │  <-- 과정 추상화
    │  │ + print(doc: Document): void      │ │
    │  └────────────────────────────────────┘ │
    └──────┬──────────────────────────────────┘
           │ Polymorphism (상위 타입 = 하위 객체)
           │
    ┌──────▼──────────────────────────────────┐
    │         Concrete Implementation Layer   │  <-- 개발자가 구현하는 세계
    │  ┌─────────────┐  ┌─────────────┐       │
    │  │HPDriver     │  │CanonDriver  │       │  <-- 제어 추상화
    │  │+print() {}  │  │+print() {}  │       │
    │  └─────────────┘  └─────────────┘       │
    │         ↓                  ↓             │
    │  ┌─────────────────────────────────────┐ │  <-- 자료 추상화
    │  │ private byte[] buffer;               │ │
    │  │ private int portStatus;              │ │
    │  └─────────────────────────────────────┘ │
    └───────────────────────────────────────────┘
```

### 3. 심층 동작 원리 및 코드

**자료 추상화의 실현: ADT (Abstract Data Type)**
자료 추상화는 데이터의 **표현(Representation)**과 **명세(Specification)**를 분리합니다. 예를 들어, '스택(Stack)'이라는 자료구조는 LIFO(Last In First Out)라는 명세만 중요하며, 내부적으로 배열(Array)로 구현했는지 연결 리스트(Linked List)로 구현했는지는 사용자에게 중요하지 않습니다.

**과정 및 제어 추상화의 실현: Strategy Pattern**
아래 코드는 다형성(Polymorphism)을 활용하여 제어 흐름을 추상화하는 전형적인 예입니다. `Order` 클래스는 `DiscountPolicy`라는 추상화된 인터페이스에만 의존하며, 실제 할인 로직(제어)이 런타임에 결정됩니다.

```java
// [과정 추상화] 할인 정책이라는 개념 정의
public interface DiscountPolicy {
    Money calculateDiscountAmount(Order order); 
}

// [자료 추상화] 금액 데이터 캡슐화
public class Money {
    private long value; // 내부 표현 은닉
    
    // ... 제어 로직 ...
}

// [구체적 구현] VIP 전략
public class VIPDiscountPolicy implements DiscountPolicy {
    @Override
    public Money calculateDiscountAmount(Order order) {
        // 복잡한 계산 로직...
        return discount;
    }
}

// [제어 추상화] 클라이언트 로직
public class OrderService {
    private DiscountPolicy discountPolicy; // 추상화된 타입에 의존

    public void createOrder(...) {
        // 내부 로직은 몰라도 됨. 단지 메시지만 전송.
        Money discount = discountPolicy.calculateDiscountAmount(order);
    }
}
```

> **📢 섹션 요약 비유**: 
> 자동차의 **계기판(자료 추상화)**은 엔진 내부의 복잡한 연료 압력이나 회전수를 단순한 '속도'와 'RPM'이라는 숫자로 변환하여 보여줍니다. **운전대와 페달(제어 추상화)**은 기어비 조절이나 유압 시스템과 같은 복잡한 기계적 연결을 '좌/우' 회전이나 '밟기'라는 단순한 동작으로 단순화합니다. 마지막으로 **내비게이션의 '길 안내(과정 추상화)'**는 수천 번의 방향 전환 과정을 '집으로'라는 하나의 버튼 누름으로 압축하여 제공합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

추상화는 독립적으로 존재하지 않으며, 캡슐화(Encapsulation)나 모듈화(Modularity)와 밀접하게 연관되어 있습니다. 또한 데이터베이스나 네트워크 등 타 영역의 기술과도 깊은 연관이 있습니다.

### 1. 기술적 비교: 추상화 vs 캡슐화 (Abstraction vs Encapsulation)

이 두 개념은 흔히 혼동되지만 명확히 구별되어야 합니다. 캡슐화는 추상화를 달성하기 위한 **구현 메커니즘** 중 하나입니다.

| 비교 항목 | **추상화 (Abstraction)** | **캡슐화 (Encapsulation)** |
|:---|:---|:---|
| **핵심 정의** | 불필요한 세부 사항을 숨기고 **필수적인 특징만 모델링** | 데이터와 데이터를 다루는 메서드를 **하나의 단위로 묶고** 외부 접근을 제한 |
| **관점** | "무엇(What)"을 보여줄 것인가에 관한 **설계 관점** | "어떻게(How)" 구현을 보호할 것인가에 관한 **구현 관점** |
| **주요 수단** | 인터페이스(Interface), 추상 클래스(Abstract Class) | 접근 제어자(private, protected), 패키지 |
| **목적** | 복잡성을 관리하고 시스템의 **유연성** 확보 | 데이터의 무결성 보장 및 내부 로직의 **은닉** |
| **융합 관점** | 캡슐화 없이는 불필요한 노출이 발생함 | 추상화 없이는 단순한 정보 은닉에 그침 |

### 2. 타 영역 융합 시너지

**[데이터베이스 관점] 뷰(View)와 자료 추상화**
RDBMS(관계형 데이터베이스 관리 시스템)에서 **뷰(View)**는 자료 추상화의 전형적인 예입니다. 복잡한 테이블 조인(JOIN)이나 데이터 필터링 로직을 뷰라는 가상 테이블로 정의하여, 사용자는 단순히 `SELECT * FROM v_user_summary`와 같이 복잡한 쿼리를 실행하지 않고도 필요한 데이터에 접근할 수 있습니다.

**[네트워크 관점] OSI 7계층과 제어 추상화**
네트워크 스택(Stack)의 각 계층은 상위 계층에게 **서비스 인터페이스(Service Interface)**를 제공함으로써 제어 추상화를 실현합니다. 예를 들어, 전송 계층(Transport Layer)은 패킷의 손실 여부를 확인하고 재전송(ARQ, Automatic Repeat reQuest)하는 복잡한 제어 과정을 은닉하고, 애플리케이션 계층에는 단순히 '데이터 전송 성공/실패' 결과만을 전달합니다.

> **📢 섹션 요약 비유**: 
> **추상화**는 건물을 설계할 때 "이 방은 거실이고, 저 방은 주방"이라는 **기능적 구분(설계도)**을 정의하는 것과 같습니다. 반면, **캡슐화**는 주방이라는 공간에 난로나 싱크대 같은 구체적인 설비를 배치하고, 외부인이 함부로 건드리지 못하게 문을 잠그는 **시공(벽과 문)** 과정에 비유할 수 있습니다. 즉, 추상화는 "보여줄 것"을, 캡슐화는 "감출 것"을 결정합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무 시스템 설계 시 추상화는 유지보수성과 확장성을 좌우하는 가장 중요한 척도입니다. 특히, **SPI(Service Provider Interface)** 설계나 **레거시 마이그레이션(Legacy Migration)**에서 핵심적인 역할을 수행합니다.

### 1. 실무 시나리오: 다기간 결제 시스템의 확장

**상황**: 온라인 쇼핑몰에 신용카드, 간편결제(카카오페이, 네이버페이), 무통장입금 등의 결제 수단이 지속적으로
