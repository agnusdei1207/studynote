# 660. 번다운 차트 작업 진척도

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스프린트(Sprint) 기간 동안 남은 작업량(Remaining Work)과 시간의 관계를 시각화하여, 팀이 정해진 기한 내에 목표를 달성할 수 있을지를 예측하는 **애자일(Agile) 가시화 도구**이다.
> 2. **가치**: '완료된 양'이 아닌 '해야 할 양'을 추적함으로써, 작업 속도(Velocity) 저하나 범위 증가(Scope Creep) 등의 위험을 일찍 감지하고, 실패 가능성을 수치로 예측하여 대응 시간을 확보한다.
> 3. **융합**: 단순한 진척률 표시를 넘어, 프로젝트 관리(PM)의 EVM(Earned Value Management) 개념을 접목하여 남은 노력량(Effort)을 정량화하고, CI(Continuous Integration) 빌드 성공률과 연계하여 기술적 부채를 관리하는 기반 데이터가 된다.

---

### Ⅰ. 개요 (Context & Background)

**번다운 차트 (Burndown Chart)**는 애자일 소프트웨어 개발, 특히 스크럼(Scrum) 프레임워크에서 스프린트의 진행 상황을 추적하는 핵심 도구입니다. 전통적인 '진척률(%)' 방식이 주관적이고 "90% 완료되었으나 끝내 완료되지 않는" 착시 현상을 유발하는 반면, 번다운 차트는 **'남은 일(Work Remaining)'**이라는 객관적 지표에 집중합니다.

이 도어는 '작업량'을 Y축으로, '시간'을 X축으로 하여, 시간이 지남에 따라 작업량이 0에 수렴하는 과정을 그래프로 나타냅니다. 즉, "우리가 가진 양초가 타서 없어지는 속도로 시간을 맞출 수 있는가?"를 시각적으로 검증하는 도구입니다.

#### 💡 비유: 산악 마라톤의 고도계
번다운 차트는 정상에 오르는 산악 마라톤에서 **'남은 고도(Altitude Loss)'**를 보여주는 고도계와 같습니다. 정상(완료)에 도달하려면 고도 0m가 되어야 합니다. 단순히 "우리가 얼마나 달렸는가(Distance)"가 중요한 것이 아니라, "정상까지 얼마나 높이를 더 내려가야 하는가(Remaining Work)"가 중요합니다.

```text
[개념의 등장 배경]
① Waterfall(폭포수) 모델의 한계: 계획 대비 실적(AP)을 보고하지만, 잔여 작업의 불확실성으로 인해 마감 기한 실패가 임박해서야 발견됨.
② Agile(애자일)의 반등: '작업 시간'을 실시간으로 추적하고, '남은 양'을 투명하게 공개하여 팀원 모두가 현실을 직시하도록 유도.
③ 현재의 비즈니스 요구: 날짜 준수(Date-line)와 범위(Scope)의 트레이드오프를 즉각적으로 결정해야 하는 VUCA 시대의 필수 관리 도구.
```

#### 📢 섹션 요약 비유
"마치 복잡한 고속도로 여행에서 '목적지까지 남은 거리'를 실시간으로 계산하여, 연료가 떨어지거나 날이 어두워지기 전에 속도를 조절하거나 경로를 변경하는 네비게이션과 같습니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

번다운 차트는 단순한 그래프가 아니라, 데이터 수집, 처리, 시각화의 3단계 아키텍처로 구성됩니다.

#### 1. 구성 요소 상세 (Component Analysis)

| 요소명 | 역할 | 내부 동작 및 데이터 | 비유 |
|:---|:---|:---|:---|
| **X축 (Time Axis)** | 시간 경과 표시 | 스프린트 기간(Day 1 ~ Day 14). 단위는 '일' 또는 '반나절' | 달력의 날짜 |
| **Y축 (Work Axis)** | 작업량 표시 | **SP (Story Point)** 또는 **Remaining Hours (잔여 시간)** | 배터리 잔량 |
| **이상적 선 (Ideal Burndown)** | 기준선 (Baseline) | `총 작업량 / 총 기간`의 기울기를 가진 직선. 시작점(총량)에서 종료점(0)을 잇음. | 예상 도착 시간 |
| **실제 선 (Actual Burndown)** | 추적선 (Tracking) | 매일 데일리 스크럼 후 남은 태스크(Task)의 양을 합산하여 좌표 찍기 | 실시간 위치 |
| **범위(Scope)** | 변동 변수 | 스프린트 중간에 태스크가 추가되면 그래프가 위로 솟구침 (Spike) | 갑작스런 우회 |

#### 2. 수학적 원리 및 데이터 흐름
번다운 차트의 기본 원리는 $y = -ax + b$ (여기서 $a$는 속도, $b$는 초기 작업량)의 선형 방정식에 기반합니다. 하지만 실제 소프트웨어 개발은 비선형적이므로, 실제 선은 잔여 작업량의 합계 함수로 정의됩니다.

**Algorithm: Daily Update Routine**
```python
# Pseudo-code for Burndown Calculation
def calculate_burndown(sprint_backlog, completed_tasks):
    remaining_effort = 0
    
    for task in sprint_backlog:
        if task.status != "DONE":
            # 잔여 작업 시간 합산 (Estimate 가 아닌 Remaining 기반)
            remaining_effort += task.remaining_hours
            
    return remaining_effort
```

#### 3. ASCII 다이어그램: 번다운 차트 구조

```text
   Amount of Work (SP or Hours)
   ^
   |(Start) 
 60|  ● 
   |    \   [Scope Creep 발생 시나리오: 작업이 추가됨]
   |     \      _____
   |      \    /     \   (이상선보다 위쪽 위치 = 위험 신호)
 40|       \  /       \
   |        \/         \______
   |        /\                \______
   |       /  \                       \
 20|      /    \                       \___
   |     /      \                          \___
   |    /                                     \
  0|___/____________________________________________\____> Time (Days)
     Day 1    Day 5    Day 10   Day 14   Day 15
 
  [Key]:
  ●  : Sprint Start (초기 작업량)
  ---: Ideal Burndown Line (기울기 = -Velocity)
  ───: Actual Burndown Line (실제 잔여량의 궤적)
```

**다이어그램 해설**:
위 다이어그램은 스프린트 초기에는 정상적으로 진행되다가, Day 5 경위에 새로운 이슈(Scope Creep)가 발생하여 그래프가 위로 솟구치는(Assume Up) 전형적인 리스크 상황을 보여줍니다. 이후 팀이 파이어파이팅(Overwork)을 통해 하강 폭을 키우려 하지만, 이상적인 선(직선)을 따르지 못하고 종료 시점이 지연(Day 15 이상)되거나, 작업량이 0이 되지 않은 상태로 스프린트가 끝날 위험을 시사합니다.

#### 4. 패턴 분석 (Pattern Recognition)
- **Parabolic (포물선형)**: 초반에 진행이 느리다가 후반부에 가파르게 떨어짐. → 설계 및 초기 개발 시간이 부족했을 가능성.
- **Flat-lined (수평선형)**: 그래프가 Y축에서 내려오지 않음. → 장애물(Blocker) 발생 혹은 스토리 포인트 과대 평가.

#### 📢 섹션 요약 비유
"양초(Burnup)가 꺼지기까지 남은 시간을 예측하기 위해, 초의 길이가 줄어드는 속도를 눈으로 확인하는 과정과 같습니다. 초가 너무 빨리 줄어들면 불을 끄고, 너무 느리면 바람을 불어 태워야 합니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 번다운 vs 번업 (Burndown vs Burnup)
번다운 차트는 주로 스프린트 내부 관리에 쓰이지만, 프로젝트 전체를 볼 때는 번업 차트와의 비교가 필수적입니다.

| 비교 항목 | **Burndown Chart (번다운)** | **Burnup Chart (번업)** |
|:---|:---|:---|
| **핵심 메트릭** | **남은 작업 (Work Remaining)** | **완료된 작업 (Work Completed)** |
| **주요 용도** | "마감 기한 내에 끝낼 수 있는가?" (속도 감시) | "우리가 얼마나 달성했는가?" (성과 감시) |
| **Scope Change 표현** | Y축이 갑자기 증가 (혼란 유발 가능) | **Scope Line(범위선)**이 상승하여 가시화 명확 |
| **심리적 영향** | 팀에게 '해야 할 일'의 압박감 부여 | 팀에게 '해낸 일'의 성취감 부여 |
| **적합 상황** | 단기 스프린트 (Sprint) | 장기 프로젝트 (Release) |

#### 2. 타 영역과의 융합 (Convergence)

**A. EVM (Earned Value Management)과의 연계**
전통적인 PMBOK 영역의 EVM에서 `CPI (Cost Performance Index)`는 수행 대비 비용 효율을 나타냅니다. 번다운 차트의 기울기는 애자일 방식의 **`Velocity (속도)`**와 직결됩니다.
- **관계식**: $EVM_{CPI} \approx Agile_{ActualVelocity} / Agile_{ExpectedVelocity}$
- 차트의 기울기가 이상선보다 완만하다면 `CPI < 1`로 해석할 수 있으며, 예산 초과 리스크로 해석 가능합니다.

**B. CI/CD (Continuous Integration)와의 연계**
번다운 차트가 내려가지 않고 멈춰 있다면? 이는 단순히 개발자가 게을러서가 아니라, **CI 파이프라인 빌드 실패**나 **QA 테스트 실패**와 같은 기술적 장애물(Technical Blocker)이 있음을 의미할 수 있습니다. 따라서 번다운 차트는 빌드 서버의 건강 상태를 간접적으로 모니터링하는 지표가 됩니다.

#### 3. 심층 비교표 (Quantitative Metrics)

| 구분 | 정량적 지표 (Metric) | 관리 포인트 (Control Point) |
|:---|:---|:---|
| **이상적 기울기** | `-(Total Points / Total Days)` | 일정한 리듬 유지 |
| **현재 속도** | `(Points Remaining Today) / (Days Remaining)` | 목표 달성 가능성 검증 |
| **예측 완료일** | `Current Day + (Points Remaining / Avg Velocity)` | 기한 미준수 시 스테이크홀더 고지 |

#### 📢 섹션 요약 비유
"번다운 차트는 자동차의 **'속도계와 연료 게이지'**이고, 번업 차트는 **'주행 거리계'**입니다. 둘을 함께 봐야만 내가 목적지까지 제 시간에 갈 수 있는지, 혹은 연료가 떨어지는 건 아닌지 정확히 판단할 수 있습니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

번다운 차트를 단순한 보고용 자료가 아닌, 의사결정을 위한 '트리거(Trigger)'로 활용하는 것이 중요합니다.

#### 1. 실무 시나리오 및 의사결정 매트릭스

**Scenario A: "Sprint Mid-point Crisis" (중반 위기)**
- **관찰**: 스프린트 기간의 50%가 경과했으나, 번다운 선이 이상적 선보다 높게 위치하고 기울기가 평평함.
- **분석**: 팀의 속도(Velocity)가 기대치에 미달함. 원인은 기술적 난관(Low Technical Skill)이나 잦은 회의(Interrupt Context Switching)일 수 있음.
- **의사결정**:
    1. **Descope (범위 축소)**: PO(Product Owner)에게 현재 상황 공유 후, 우선순위가 낮은 아이템을 백로그에서 제거(Scope cutting)하여 0에 도달 가능성 확보.
    2. **Resource Swap (인력 교체)**: 해당 태스크에 특화된 인력 투입.
    3. **Crash Mode (집중 모드)**: 스크럼 마스터(Scrum Master)가 팀원의 외부 간섭을 차단하고 집중 시간 확보.

**Scenario B: "Scope Creep Spike" (범위 돌발 상승)**
- **관찰**: 차트가 하향하다가 갑자기 위로 솟구침 (Spike 발생).
- **분석**: 개발 중 발견된 숨겨진 요구사항이나 버그가 추가됨.
- **의사결정**:
    1. **Trade-off**: 새로운 작업을 추가하려면, 기존 작업 중 하나를 제거하여 '총 작업량'을 일정하게 유지하는 **'Zero-Sum Game'** 전략 적용.

#### 2. 도입 체크리스트 (Checklist)

| 구분 | 체크항목 | 설명 |
|:---|:---|:---|
| **기술적** | **단위의 일관성** | 스토리 포인트(SP)와 시간(Hours)을 혼용하여 추적하지 말 것. SP가 추천됨. |
| **운영적** | **자동화 도구** | Jira, Azure DevOps 등과 연동하여 자동으로 그려지게 하고, 수기 업데이트 지양. |
| **보안/품질** | **Done 정의 준수** | 'Done' 상태를 단순히 '개발 완료'로 두지 말고, 'QA 완료 및 배포 가능' 상태로 정의하여 잔여 작업량의 품질 보장. |

#### 3. 안티패턴 (Anti-Pattern)
- **"Burndown Theater" (번다운 연극)**: 차트가 예쁘게 나오라고 잔여 작업 시간을 억지로 줄여서 적는 행위. 이는 팀의 위험을 은폐하여 최악의 실패를 초래함.
- **"Micro-management" (미시 관리)**: 차트의 하루하루 변동에 일희일비하며 팀원을 닦달하는 도구로 전락시키는 것.

#### 📢 섹션 요약
