# 661. 칸반 WIP (Work In Progress) 제한

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 업무의 흐름을 시각화하고 현재 진행 중인 작업의 수(**WIP, Work In Progress**)를 제한함으로써, 병목 현상을 해결하고 팀의 처리량(Throughput)을 극대화하는 **린(Lean) 기반 워크플로우 관리 기법**이다.
> 2. **제한의 미학**: "일을 시작하는 것보다 끝내는 것이 중요하다"는 원칙하에, 동시 작업 수를 제한하여 문맥 전환(Context Switching) 비용을 줄이고 집중도를 높인다.
> 3. **가치**: 정해진 반복 주기(Iteration) 없이 지속적인 흐름(Continuous Flow)을 중시하며, 실제 역량에 맞춰 일을 당겨오는 **풀(Pull) 시스템**을 구현한다.

---

## Ⅰ. 개요 (Context & Background)

### 칸반(Kanban)의 유래 및 정의

일본어로 '간판' 또는 '신호 카드'를 뜻하며, 도요타의 JIT(Just-In-Time) 생산 방식에서 유래되었습니다. 소프트웨어 개발에서는 화이트보드나 Trello 같은 도구에 포스트잇을 붙여 작업 상태를 관리하는 방식으로 발전했습니다. 스크럼이 '기간(Timebox)' 중심이라면, 칸반은 **'흐름(Flow)'** 중심입니다.

### 💡 비유: 고속도로의 적정 차량 수

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        칸반 WIP 제한 비유                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 고속도로(우리 팀의 개발 프로세스)에 차가 너무 많음.                      │
│                                                                             │
│  1. WIP 제한 없음:                                                           │
│     일단 차를 다 들여보냄. 고속도로가 꽉 차서 정체가 발생하고, 아무도 빨리        │
│     나가지 못함. (멀티태스킹의 함정)                                            │
│                                                                             │
│  2. WIP 제한 적용:                                                           │
│     고속도로에 동시에 올라갈 수 있는 차를 5대로 제한함. 차 한 대가 빠져나가면     │
│     그때야 대기하던 다음 차가 들어옴.                                           │
│                                                                             │
│  → 길은 텅 빈 것 같아 보이지만, 실제로는 차들이 쌩쌩 달려 목적지에 훨씬         │
│     빨리 도착함(리드 타임 단축)!                                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 칸반의 4대 기본 원칙
1. **시각화 (Visualize)**: 모든 작업을 보드에 게시하여 누가 무엇을 하는지 한눈에 파악.
2. **WIP 제한 (Limit WIP)**: 각 단계(Column)별로 동시에 가질 수 있는 작업 카드 수 상한 설정.
3. **흐름 관리 (Manage Flow)**: 작업이 막히는 구간(Bottleneck)을 찾아내고 흐름을 매끄럽게 조정.
4. **명시적 정책 (Make Policies Explicit)**: '완료 정의(DoD)' 등을 팀원 모두가 알 수 있게 명문화.

### 2. WIP 제한의 메커니즘 (Pull System)
- **Push 방식**: 할 일을 계속 밀어 넣음 (과부하 발생).
- **Pull 방식**: 다음 단계에 빈자리(WIP 여유)가 생겼을 때만 앞 단계에서 작업을 가져옴.

### 3. 칸반 보드 아키텍처 (ASCII)

```text
    ┌──────────┬──────────┬──────────┬──────────┐
    │  Backlog │  To-Do   │ In-Prog  │   Done   │
    │          │ (WIP: 3) │ (WIP: 2) │          │
    ├──────────┼──────────┼──────────┼──────────┤
    │ [Story A]│ [Task 1] │ [Task 3] │ [Task 5] │
    │ [Story B]│ [Task 2] │ [Task 4] │          │
    │          │          │          │          │
    │          │ (Empty)  │ (FULL!)  │          │ <── [Task 4]가 끝나야
    └──────────┴──────────┴──────────┴──────────┘      [Task 2] 진입 가능
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 스크럼 (Scrum) vs 칸반 (Kanban)

| 항목 | 스크럼 (Scrum) | 칸반 (Kanban) |
|:---:|:---|:---|
| **핵심 지표** | 속도 (Velocity) | 리드 타임 (Lead Time), 사이클 타임 |
| **반복 주기** | 고정된 스프린트 (1-4주) | 지속적 흐름 (No Iteration) |
| **변경 통제** | 스프린트 중 변경 지양 | 언제든지 백로그 우선순위 변경 가능 |
| **역할 분담** | PO, SM, 팀원 명확 | 정해진 역할 없음 (유연함) |

### 2. 기술적 시너지: 리틀의 법칙 (Little's Law)
- **공식**: $L = \lambda \times W$ (평균 재고 = 도착률 x 평균 체류 시간)
- **의미**: 진행 중인 작업(WIP)이 줄어들면, 작업을 완료하는 데 걸리는 시간(Lead Time)도 정비례하여 줄어든다는 수학적 근거를 제공합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 유지보수 운영 팀의 칸반 도입
- **상황**: 수시로 들어오는 자잘한 수정 요청 때문에 스크럼 스프린트 계획이 매번 깨짐.
- **결단**: **칸반으로 전환 및 WIP 설정**.
    1. 개발 중(In-Progress) WIP를 팀원 수와 동일하게 설정하여 한 명당 한 업무만 집중.
    2. 테스트(QA) 대기열의 WIP를 작게 잡아 개발팀이 QA 병목을 돕도록 유도 (Swarming).
- **판단**: 계획 수립에 드는 오버헤드를 줄이고, 요청 접수부터 해결까지의 '평균 리드 타임'을 30% 단축.

### 📢 기술사적 결언
> "칸반의 핵심은 보드가 아니라 **'제한'**에 있다. WIP를 제한하지 않는 칸반은 단순히 포스트잇이 붙은 벽일 뿐이다. 제한을 통해 발생하는 '불편함'이 곧 시스템의 병목을 드러내는 신호이며, 이 신호를 따라 프로세스를 개선하는 것이 진정한 린(Lean) 싱킹의 완성이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **리드 타임 단축**: 고객의 요청이 실제 가치로 변하는 시간 최소화.
- **품질 향상**: 멀티태스킹 방지로 인한 집중도 향상 및 결함률 감소.

### 미래 전망
미래의 칸반은 **'디지털 트윈 기반의 자동 최적화'**로 진화할 것입니다. AI가 팀의 과거 사이클 타임을 분석하여 최적의 WIP 수치를 실시간으로 제안하고, 특정 단계에서 병목이 예상되면 자동으로 인력 재배치(Dynamic Staffing)를 추천하는 지능형 워크플로우 관리 환경이 보편화될 것입니다.

1. **칸반**은 맛있는 붕어빵을 만드는 기계와 같아요. 붕어빵 틀(WIP)이 5개면 딱 5개만 구워야 하죠.
