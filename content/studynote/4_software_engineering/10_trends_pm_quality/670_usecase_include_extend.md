# 670. 유스케이스 포함(Include) 확장(Extend)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템이 사용자에게 제공하는 기능적 요구사항을 사용자의 관점에서 정의하는 **UML 행위 다이어그램**으로, 시스템의 범위와 상호작용을 시각화한다.
> 2. **포함 (Include)**: 여러 유스케이스에서 공통적으로 발생하는 기능을 별도로 분리하여 **반드시 실행**되어야 하는 필수 종속 관계를 나타낸다.
> 3. **확장 (Extend)**: 특정 조건이 만족될 때만 **선택적으로 실행**되는 부가 기능을 정의하며, 기본 흐름을 방해하지 않고 기능을 유연하게 추가할 때 사용한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "누가 시스템으로 무엇을 하는가?"

유스케이스(Use Case) 모델링은 이바 야콥슨(Ivar Jacobson)에 의해 제안되었습니다. 시스템의 내부 구현이 아닌, 시스템 외부의 행위자(Actor)가 시스템을 통해 얻고자 하는 '목표'에 집중합니다. 이때 유스케이스 간의 관계를 효율적으로 정리하기 위해 포함(Include)과 확장(Extend)이라는 개념이 필수적으로 사용됩니다.

### 💡 비유: 마트 계산 시스템

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        유스케이스 관계 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [기본 유스케이스] : 물건 결제하기                                             │
│                                                                             │
│  1. 포함 (Include - 필수):                                                   │
│     결제를 하려면 "바코드 스캔하기"를 **무조건** 해야 함.                        │
│     (스캔 없이는 결제 자체가 불가능함)                                          │
│                                                                             │
│  2. 확장 (Extend - 선택):                                                   │
│     결제 도중 "포인트 적립하기"는 **할 수도 있고 안 할 수도** 있음.               │
│     (적립을 안 해도 결제라는 기본 목적은 달성 가능함)                             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 유스케이스 관계 상세 비교

| 관계 유형 | 키워드 | 실행 여부 | 방향성 | 주요 목적 |
|:---:|:---:|:---:|:---:|:---|
| **포함 (Include)** | `<<include>>` | **필수** (Always) | 기본 → 피포함 | 중복 제거, 공통 로직 분리 |
| **확장 (Extend)** | `<<extend>>` | **선택** (Optional) | 확장 → 기본 | 예외 처리, 부가 기능 추가 |
| **일반화 (Generalization)** | - | 계층적 상속 | 하위 → 상위 | 추상화 및 그룹화 |

### 2. UML 표기법 및 방향 (ASCII)

```text
    [ 유스케이스 A ] ─── <<include>> ──▶ [ 유스케이스 B ]
    (A가 실행되면 B는 무조건 실행됨)
    
    [ 유스케이스 C ] ◀── <<extend>> ─── [ 유스케이스 D ]
    (C가 실행될 때 특정 조건(Extension Point)에서만 D가 실행됨)
```

### 3. 유스케이스 다이어그램의 4대 구성 요소
- **액터 (Actor)**: 시스템과 상호작용하는 사람 또는 외부 시스템 (졸라맨 표기).
- **유스케이스 (Use Case)**: 시스템이 제공하는 서비스 단위 (타원 표기).
- **시스템 경계 (System Boundary)**: 개발 범위 (사각형 표기).
- **관계 (Relationship)**: 연관, 포함, 확장, 일반화.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 포함 vs 확장의 결정적 차이
- **제어권**: 포함은 기본 유스케이스가 피포함 유스케이스를 **호출**하는 느낌이고, 확장은 확장 유스케이스가 기본 유스케이스에 **끼어드는** 느낌입니다.
- **의존성**: 포함 관계에서 피포함 유스케이스가 없으면 기본 기능이 불완전해지지만, 확장 관계에서는 확장 기능이 없어도 기본 기능은 완벽하게 동작합니다.

### 2. 기술적 시너지: 객체지향 설계 (SOLID)
- **포함**: 코드 레벨의 **메서드 추출(Extract Method)**이나 공통 모듈화와 맥을 같이 합니다.
- **확장**: **개방-폐쇄 원칙(OCP)**을 준수하며 기존 코드를 고치지 않고 기능을 추가하는 전략 패턴이나 데코레이터 패턴과 유사한 철학을 가집니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 뱅킹 시스템 설계
- **기본 유스케이스**: 계좌 이체.
- **포함 적용**: "본인 인증" (이체 시 반드시 거쳐야 하므로 모든 이체/조회 유스케이스에서 포함).
- **확장 적용**: "SMS 통보" (이체 완료 후 사용자가 신청한 경우에만 실행되므로 확장).
- **판단**: 공통되는 보안 로직은 '포함'으로 묶어 재사용성을 높이고, 마케팅성 알림 기능은 '확장'으로 정의하여 핵심 비즈니스 로직의 복잡도를 낮춤.

### 📢 기술사적 결언
> "유스케이스 다이어그램은 단순히 그림을 그리는 것이 아니라 **'시스템의 책임 분계선'**을 긋는 작업이다. 포함과 확장을 남발하면 다이어그램이 복잡해져 가독성을 해치므로, 사용자에게 가치를 주는 '목표' 수준에서 관계를 설정해야 한다. 특히 확장 관계의 **'확장 지점(Extension Point)'**을 명확히 정의하는 것이 설계의 정교함을 결정한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **요구사항 명확화**: 텍스트 중심 명세의 모호성 제거로 의사소통 오류 30% 감소.
- **설계 재사용성**: 공통 유스케이스 식별을 통해 초기 설계 공수 절감.

### 미래 전망
최근에는 유스케이스 모델링이 **'사용자 스토리 매핑 (User Story Mapping)'**과 결합하여 더 동적인 형태로 진화하고 있습니다. 또한 AI가 요구사항 텍스트를 읽고 자동으로 UML 유스케이스 다이어그램 초안을 생성해 주는 도구들이 등장하며, 분석 모델링의 자동화(Model-Driven Development) 속도가 빨라지고 있습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **포함**은 신발을 신을 때 **"양말 신기"**처럼 꼭 세트로 해야 하는 일이에요.
2. **확장**은 아이스크림을 먹을 때 **"초코 시럽 뿌리기"**처럼 먹고 싶을 때만 골라서 하는 일이죠.
3. 이 두 가지를 잘 구별해두면, 나중에 시스템을 만들 때 **어떤 게 제일 중요한지** 헷갈리지 않는답니다!
