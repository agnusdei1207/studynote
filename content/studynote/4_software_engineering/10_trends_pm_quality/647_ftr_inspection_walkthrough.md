# 647. FTR (정형 기술 검토) 인스펙션/워크스루

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 결함의 **'선제적 발견과 제거(Defect Prevention)'**를 목표로, 산출물을 실행하지 않고(Static) 논리적, 구조적 오류를 탐색하는 가장 효율적인 품질 보증(QA, Quality Assurance) 활동이다.
> 2. **메커니즘**: 인간의 인지 능력과 집단 지성을 활용하여 **Fagan의 Inspection 모델**과 같은 정형화된 프로세스를 통해 결함을 식별하며, 테스트 단계에서의 수정 비용을 기하급수적으로(약 1:100 비율) 절감한다.
> 3. **가치**: 단순한 오류 수정을 넘어 팀 내 **암묵지의 형성(Shared Knowledge)**, 요구사항에 대한 **기술적 합의(Consensus)**, 그리고 프로젝트 리스크의 조기 해소를 통해 소프트웨어 엔지니어링의 성숙도를 높인다.

---

### Ⅰ. 개요 (Context & Background)

소프트웨어 위기(Software Crisis) 이후, "결함은 생애 주기가 진행될수록 수정 비용이 기하급수적으로 증가한다"는 대원칙이 정립되었습니다. 요구사항 분석 단계의 버그 수정 비용을 1이라고 가정할 때, 운영 단계에서의 수정 비용은 약 100~1,000배에 달합니다.

**FTR (Formal Technical Review)**은 이러한 비폭발적인 비용 증가를 억제하는 브레이크 역할을 합니다. 코드를 실행하여 증상(Symptom)을 관찰하는 동적 테스트(Dynamic Testing)와 달리, FTR은 명세서, 설계서, 코드 등의 산출물을 분석하여 결함의 **원인(Cause)** 자체를 밝혀냅니다. 현대적인 애자일(Agile) 환경에서도 페어 프로그래밍(Pair Programming)이나 코드 리뷰(Code Review)의 형태로 진화하여 계속되고 있으며, 이는 FTR이 소프트웨어 품질의 근간(Root)임을 증명합니다.

#### 💡 비유: 건물의 설계도 검토와 안전 진단
```text
[Scenario] 건축 현장에서의 공사 중 수정 vs 설계 단계 수정

1. 설계도 검토 (FTR/Inspection):
   - 설계사, 구조 엔지니어가 모여 청사진을 펼놓고
   - "이 기둥의 하중이 부담스럽습니다", "이 창문은 법정 환기 면적에 부합하지 않습니다"라며
   - 실제 시공(코딩) 전에 문제를 찾아내고 수정함.
   -> 비용: 연필과 지우개 개비 (매우 저렴)

2. 시공 후 테스트 (Dynamic Testing):
   - 건물을 다 지은 뒤 내진 테스트를 수행하거나
   - 사람이 입주한 뒤 비가 새는 것을 발견함.
   -> 보수 공사를 위해 벽을 뜯어야 하며, 입주민 피해 보상이 발생함.
   -> 비용: 수천만 원 ~ 수억 원 (막대함)
```

**📢 섹션 요약 비유**: FTR은 건물을 짓기 전에 청사진(설계서)을 놓고 전문가들이 모여 벽돌 한 장이라도 잘못된 곳은 없는지 돋보기로 들여다보는 **'설계도 면밀 검사'**와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

FTR은 단순한 회의가 아닌, 입력(산출물)이 프로세스(검토)를 거쳐 출력(수정된 산출물, 리포트)으로 나가는 **정형화된 시스템**입니다. 가장 대표적인 방법인 **인스펙션(Inspection)**과 개발자 주도의 **워크스루(Walkthrough)**는 목적과 절차에서 명확한 차이를 보입니다.

#### 1. FTR 핵심 구성 요소 및 역할 (Role Definition)

FTR이 성공하기 위해서는 명확한 역할 분담이 필수적입니다. 특히 인스펙션에서는 '중재자(Moderator)'의 핵심적인 리더십이 요구됩니다.

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 및 책임 (Responsibility) |
|:---|:---|:---|
| **중재자 (Moderator)** | 진행자 (Facilitator) | 검토 회의를 주관하며 분위기 조성, 시간 엄수, 논쟁 중재. 가장 숙련된 엔지니어가 맡음. |
| **작성자 (Author)** | 생산자 (Producer) | 산출물을 작성한 사람. 오류 설명 요청 시 응답하며, 방어적 태도를 지양하고 피드백 수용. |
| **검토자 (Reviewer)** | 검열관 (Inspector) | 사전 준비 시간을 통해 결함을 찾아냄. 각자 다른 관점(사용자, 성능, 보안 등)을 가짐. |
| **기록자 (Scribe)** | 서기 (Recorder) | 회의 중 발견된 결함의 유형, 위치, 심각도를 실시간으로 **오류 리포트(Error Log)**에 기록. |

#### 2. Fagan's Inspection 라이프사이클 (Process Flow)

IBM의 Michael Fagan이 개발한 인스펙션 모델은 소프트웨어 검증의 표준 프로세스로 자리 잡았습니다.

```text
┌───────────────────────────────────────────────────────────────────────────────┐
│                  Fagan's Inspection Lifecycle (Fagan 모델)                    │
├───────────────────────────────────────────────────────────────────────────────┤
│                                                                               │
│   [Input]  ──► 1. Planning ──► 2. Overview ──► 3. Preparation ──► 4. Meeting │
│  (Docs)         (계획)        (사전 교육)       (준비)            (검토 회의)   │
│                     │             │              │    │            │          │
│                     ▼             ▼              ▼    ▼            ▼          │
│                 (Kick-off)   (Author       (Individual   (Defect     (Logging) │
│                 (팀 구성)      Briefing)     Study)       Listing)    (기록)    │
│                                                                       │        │
│                                                                       ▼        │
│   [Output] ◄─── 6. Follow-up ◄─── 5. Rework ◄────────────────────────┘        │
│ (Verified)        (추적/확인)      (재작성/수정)                                   │
│                                                                               │
│   * Key Entry Criteria: 검토 가능 상태의 문서                                 │
│   * Key Exit Criteria:  모든 Critical 결함 수정 완료                           │
└───────────────────────────────────────────────────────────────────────────────┘
```

**[도해 상세 설명]**
1.  **Planning (계획)**: 중재자가 검토자 3~5명을 선정하고, 산출물이 '입구 기준(Entry Criteria)'을 만족하는지 확인 후 배포.
2.  **Overview (개요)**: 작성자가 배경 지식, 알고리즘, 참조 문서 등을 설명하여 검토자들의 이해를 돕음.
3.  **Preparation (준비)**: **가장 중요한 단계**. 검토자들이 각자 체크리스트를 들고 산출물을 정독하며 의심 스러운 부분을 표시함.
4.  **Meeting (회의)**: 중재자 주도 하에 진행. 결함만을 나열하고 해결책 논의는 금지(시간 낭비 방지). 보통 2시간 내외로 제한.
5.  **Rework (재작성)**: 작성자가 보고받은 결함을 수정.
6.  **Follow-up (추적)**: 중재자가 수정 결과를 확인하고, '출구 기준(Exit Criteria)'을 충족 시킬지 최종 판단.

#### 3. 핵심 파라미터 및 지표 (Metrics & Formulas)
FTR의 효과를 측정하기 위해 아래와 같은 정량적 지표가 활용됩니다.

*   **결함 밀도 (Defect Density)**:
    $$ \text{Defect Density} = \frac{\text{Total Defects Found}}{\text{Size (e.g., KLOC or Function Points)}} $$
    > *해석*: 천 줄의 코드(1 KLOC)당 몇 개의 버그가 발견되었는지를 나타내며, 이 값이 높을수록 품질이 낮거나 검토가 효과적으로 진행됨을 의미합니다.

*   **검토 속도 (Inspection Rate)**:
    $$ \text{Rate} = \frac{\text{Pages Reviewed}}{\text{Hours Spent}} $$
    > *해석*: 너무 빠른 검토 속도(예: 10페이지/시간)는 훑어보기(Skimming)에 그칠 확률이 높아 경계해야 합니다.

**📢 섹션 요약 비유**: 인스펙션 프로세스는 마치 **'의료진의 협진(Clinical Conference)'**와 같습니다. 주치의(작성자)가 병력(산출물)을 설명하면, 전문의들(검토자)이 각자의 전공 분야(안과, 내과 등)별로 조사를 준비하고(Preparation), 회의(Meeting)를 통해 질병(결함)을 확진한 뒤 수술(재작성)을 결정하는 체계적인 시스템입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

FTR은 소프트웨어 개발 수명 주기(SDLC, Software Development Life Cycle) 전반에 걸쳐 다른 기술들과 시너지를 발휘합니다. 특히 정적 분석(Static Analysis) 도구와의 결합이 중요합니다.

#### 1. FTR 유형 비교: 인스펙션 vs 워크스루

| 비교 항목 (Criteria) | 인스펙션 (Inspection) | 워크스루 (Walkthrough) |
|:---|:---|:---|
| **목적 (Goal)** | 결함 발견 (Defect Detection) 최우선 | 교육 및 정보 공유 (Education) |
| **진행자 (Leader)** | 중재자 (Moderator) - 제3자 | 작성자 (Author) |
| **준비 단계** | 필수 (체크리스트 배포 및 개별 사전 검토) | 선택 사항 (즉석 진행 가능) |
| **문서화 (Doc)** | 엄격한 오류 리포트 작성 및 통계 관리 | 간단한 메모나 회의록 수준 |
| **공식성 (Formality)** | 매우 높음 (High Ceremony) | 상대적으로 낮음 (Low Ceremony) |
| **산출물 (Output)** | 수정된 결과물 + 품질 보고서 | 수정된 결과물 (통계 미흡) |

#### 2. 정적 vs 동적 검증 (Static vs Dynamic Verification)

```text
┌───────────────────────────────────────────────────────────────────────────────┐
│                     V&V (Verification & Validation) Matrix                    │
├───────────────────────────────────────────────────────────────────────────────┤
│                                                                               │
│  ┌────────────────┐         ┌───────────────────────────────────────┐         │
│  │  Dynamic Test  │         │          Static Analysis Tools         │         │
│  │  (Unit Test)   │         │    (Lint, SonarQube, Checkstyle)      │         │
│  └────────────────┘         └───────────────────────────────────────┘         │
│           ▲                            ▲                               ▲      │
│           │                            │                               │      │
│           ▼                            ▼                               │      │
│  ┌───────────────────────────────────────────────────────────────────────┐│      │
│  │                      SQA (Software Quality Assurance)                 ││      │
│  │                  FTR (Human Cognitive Inspection)                     ││      │
│  │   ── "도구가 발견하지 못하는 로직 오류, 비즈니스 정합성 검증" ──      ││      │
│  └───────────────────────────────────────────────────────────────────────┘│      │
│                                                                               │
└───────────────────────────────────────────────────────────────────────────────┘
```
*   **Tool vs Human**: 정적 분석 도구는 코딩 표준 위반, Null Pointer 참조 등의 **구문적 오류**를 잘 잡아내지만, "이 요구사항이 비즈니스 로직에 맞는가?"와 같은 **의미적/논리적 오류**는 FTR(인간)이 검토해야 합니다.

#### 3. 타 과목 융합 관점
*   **프로젝트 관리 (PM)**: FTR은 일정(Schedule) 지연의 원인으로 비치될 수 있으나, 장기적으로는 리스크(Risk)를 조기에 제거하여 프로젝트 실패 확률을 낮춥니다.
*   **소프트웨어 아키텍처**: 설계 단계의 FTR(ATAM: Architecture Tradeoff Analysis Method 등)은 시스템의 품질 속성(Quality Attributes: 성능, 보안 등)을 검증하는 핵심 활동입니다.

**📢 섹션 요약 비유**: 워크스루는 친구들끼리 모여서 "내 이번 과제 어때?"라고 보여주는 **'간단한 발표 회'**라면, 인스펙션은 변호사와 검사가 법정에 서서 증거를 하나하나 대조하며 사건의 진실을 규명하는 **'공식 재판'**과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 FTR을 도입할 때는 '비용 대비 효과(Cost-Benefit)'를 따져야 합니다. 모든 코드를 인스펙션할 수 없으므로, **리스크 기반(Risk-Based)** 전략이 필요합니다.

#### 1. 실무 시나리오: 결정적 결함 탐지

*   **상황**: 금융 결제 모듈 개발. 복잡한 다중 트랜잭션 처리 로직 설계 완료 상태.
*   **문제**: 단위 테스트(Unit Test)를 작성 중인데, "네트워크 두절 시 일부 결제가 누락될 수 있다"는 우려가 제기됨. 로직이 복잡하여 테스트 케이스만으로는 모든 경로를 커버하기 어려움.
*   **전략**:
    1.  **대상**: 핵심 클래스 설계서 및 시퀀스 다이어그램.
    2.  **구성**: 수석 아키텍처(Moderator), 보안 전문가(Reviewer), DBA(Reviewer).
    3.  **수행**: 워크스루 형태로 로직 흐름을 따라가며, 'Commit' 및 'Rollback' 시점의 원자성(Atomicity)을 집중 검토.
*   **결과**: 특
