# 636. 탐색적 테스트 차터 기반 휴리스틱

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 테스트 설계와 수행을 동시에 진행하며, 테스터의 지적 능력과 경험을 극대화하여 예기치 못한 결함을 찾아내는 **경험 기반 테스팅(Experience-based Testing)** 기법이다.
> 2. **체계**: 무계획적 수행이 아닌, **테스트 차터(Test Charter)**를 통해 범위를 설정하고 **타임박싱(Time-boxing)**과 **휴리스틱(Heuristic)**을 활용하여 체계적으로 결함을 사냥한다.
> 3. **가치**: 문서화된 명세가 부족하거나 개발 속도가 빠른 애자일 환경에서, 정형화된 테스트 케이스가 놓치기 쉬운 '살충제 패러독스' 구간의 결함을 효과적으로 발견한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: 스크립트 테스팅의 한계

미리 작성된 테스트 케이스(Test Case)에 의존하는 **스크립트 테스팅(Scripted Testing)**은 안정적이지만, 정해진 경로 외의 결함을 발견하기 어렵습니다. 또한 요구사항이 수시로 변하는 환경에서는 테스트 케이스를 매번 업데이트하는 비용이 큽니다. **탐색적 테스팅(Exploratory Testing)**은 이러한 경직성을 극복하고 테스터의 직관을 도구로 활용합니다.

### 💡 비유: 가이드 투어 vs 자유 배낭여행

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                       탐색적 테스팅 비유                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 스크립트 테스팅 (가이드 투어):                                             │
│     깃발을 든 가이드를 따라 정해진 명소만 방문. 안전하지만 숨은 맛집은 못 찾음.      │
│                                                                             │
│  2. 탐색적 테스팅 (자유 배낭여행):                                             │
│     "어느 동네의 뒷골목을 탐험하겠다"는 계획(차터)만 세우고, 현장에서 느낌이         │
│     오는 길(휴리스틱)을 따라 걷다가 숨겨진 맛집(결함)을 발견함.                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 탐색적 테스팅의 3대 핵심 요소

| 요소 | 설명 | 목적 |
|:---:|:---|:---|
| **테스트 차터 (Charter)** | 테스트의 목표, 범위, 도구를 정의한 짧은 지침서 | "무엇을 탐색할 것인가"에 대한 방향성 제시 |
| **타임박싱 (Time-boxing)** | 짧은 시간(예: 60~120분) 동안 집중하여 테스트 수행 | 테스터의 집중력 유지 및 테스트 누수 방지 |
| **디브리핑 (Debriefing)** | 테스트 종료 후 결과를 관리자/개발자와 공유 | 발견된 결함 및 새로운 지식 전파 |

### 2. 차터(Charter)의 구조 (예시)
- **대상**: "결제 모듈의 예외 처리"
- **방법**: "다양한 결제 수단과 네트워크 차단을 조합"
- **도구**: "Fiddler, 브라우저 개발자 도구"
- **목표**: "데이터 불일치 및 런타임 에러 탐지"

### 3. 세션 기반 테스트 관리 (SBTM)
탐색적 테스팅을 정량적으로 관리하기 위해 세션(Session) 단위로 기록하고, 다음 지표를 추적합니다.
- **T (Test)**: 실제 테스트 수행 시간
- **B (Bug)**: 결함 조사 및 보고 시간
- **Setup**: 환경 설정 시간

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 휴리스틱 (Heuristic) 기법
테스터가 결함을 찾기 위해 사용하는 **지적 지름길**입니다.
- **Goldilocks**: 너무 큰 값, 너무 작은 값, 딱 맞는 값을 넣어봄.
- **Zero-One-Many**: 데이터가 없을 때, 하나일 때, 아주 많을 때의 반응 확인.
- **CRUD**: 생성, 조회, 수정, 삭제를 불규칙한 순서로 반복.

### 2. 블랙박스 vs 탐색적 테스팅
| 항목 | 블랙박스 (명세 기반) | 탐색적 테스팅 (경험 기반) |
|:---:|:---|:---|
| **지식원** | 요구사항 명세서, 설계서 | 테스터의 지식, 제품 사용 경험 |
| **산출물** | 상세 테스트 케이스 | 테스트 차터, 세션 리포트 |
| **장점** | 결과 재현성, 정량적 커버리지 | 결함 발견 효율, 조기 테스트 가능 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 신규 모바일 게임 런칭 전 테스트
- **상황**: 정규 테스트 케이스는 모두 통과했으나, 실제 유저들이 만질 때 '느낌'이 이상하다는 제보가 있음.
- **결단**: 숙련된 QA 엔지니어 3명을 투입하여 90분간 3회의 세션으로 탐색적 테스팅 실시. 차터는 "멀티터치 및 비정상 앱 종료 시 데이터 저장 로직"으로 설정.
- **결과**: 빠르게 연타할 때 발생하는 데드락(Deadlock) 결함 2건 발견 및 수정.

### 📢 기술사적 결언
> "탐색적 테스팅은 '아무렇게나 하는 테스트'가 아니라, **'고도로 훈련된 직관을 체계적으로 쏟아붓는 활동'**이다. 문서화에 드는 비용을 줄이고 실제 사용자가 겪을 법한 결함을 먼저 찾아냄으로써, 품질의 마지막 2%를 채우는 화룡점정의 역할을 수행한다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적/정성적 기대효과
- **결함 발견율**: 정형화된 테스트 대비 치명적 결함 발견 확률 약 30% 이상 높음.
- **학습 효과**: 테스터가 제품을 깊이 이해하게 되어 차후 테스트 설계 품질 향상.

### 미래 전망
최근에는 **AI 기반 탐색적 테스팅**이 주목받고 있습니다. AI 에이전트가 서비스의 UI 구조를 스스로 탐색하고, 과거의 결함 패턴을 기반으로 '버그가 날 만한 곳'을 스스로 추론하여 차터를 제안하는 기술이 발전하고 있습니다. 이는 인간 테스터가 더 창의적인 탐색에 집중할 수 있도록 돕는 강력한 도구가 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **탐색적 테스팅**은 보물찾기를 할 때 지도를 보고 따라가는 게 아니라, 보물이 숨겨져 있을 법한 **나무 밑이나 돌 틈을 직접 찾아보는 것**과 같아요.
2. 하지만 그냥 헤매는 게 아니라, "오늘은 놀이터 주변을 1시간 동안만 찾아보자!"라는 **계획(차터)**을 세우고 똑똑하게 움직이는 거예요.
3. 이렇게 하면 미리 정해진 길에는 없던 **진짜 숨겨진 보물(버그)**을 훨씬 더 많이 찾을 수 있답니다!
