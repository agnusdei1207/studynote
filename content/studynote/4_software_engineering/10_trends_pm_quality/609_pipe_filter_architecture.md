# 파이프-필터 아키텍처 (Pipe-Filter Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터를 단계별로 변환하는 필터(Filter)들이 파이프(Pipe)로 연결되어 스트림 처리를 수행하는 순차적 아키텍처
> 2. **가치**: 각 필터의 독립적 재사용성, 병렬 처리 가능성, ETL 파이프라인에서의 높은 생산성 (코드 중복 60% 감소)
> 3. **융합**: Unix 파이프라인, 데이터 엔지니어링 ETL/ELT, MSA 이벤트 기반 아키텍처와 직접적 연관

---

## Ⅰ. 개요 (Context & Background)

### 개념 및 정의

**파이프-필터 아키텍처 (Pipe-Filter Architecture)**는 데이터 처리 시스템을 **필터 (Filter)**와 **파이프 (Pipe)**라는 두 가지 독립적 구성 요소로 분리하여 구성하는 아키텍처 패턴입니다. 각 필터는 입력 데이터를 받아 변환한 후 출력을 생성하는 독립적인 처리 단위이며, 파이프는 필터 간의 데이터 전송 채널 역할을 합니다.

```
┌─────────────────────────────────────────────────────────────────────────────┐

│                                                                             │
│    ┌──────────┐     ┌───────┐     ┌──────────┐     ┌─────────┐            │
│    │  Source  │────▶│ Pipe 1│────▶│ Filter A │────▶│ Pipe 2  │            │
│    │  Data    │     └───────┘     └──────────┘     └─────────┘            │
│    └──────────┘                                                     │      │
│                                                                   │      │
│                                                                   ▼      │
│                                                            ┌──────────┐   │
│                                                            │ Filter B │   │
│                                                            └──────────┘   │
│                                                                   │      │
│                                                                   ▼      │
│                                                            ┌─────────┐    │

```

**필터 (Filter)**의 핵심 특성:
- **단일 책임**: 하나의 잘 정의된 변환 작업만 수행
- **무상태성 (Stateless)**: 내부 상태를 유지하지 않고 입력만으로 출력 결정
- **독립성**: 다른 필터의 구현을 알 필요 없음
- **재사용성**: 다른 파이프라인에서도 재사용 가능

**파이프 (Pipe)**의 핵심 특성:
- **단방향 데이터 흐름**: 데이터는 한 방향으로만 흐름
- **버퍼링**: 생산자-소비자 간 속도 차이 완화
- **추상화**: 전송 메커니즘(메모리, 파일, 네트워크)을 캡슐화

### 💡 비유

파이프-필터 아키텍처는 **자동차 세차 시스템**과 유사합니다:

1. **필터 1**: 고압 세차 (물로 먼지 제거)
2. **필터 2**: 비누 거품 적용
3. **필터 3**: 브러시로 문질러 닦기
4. **필터 4**: 헹굼
5. **필터 5**: 왁스 코팅

각 단계는 이전 단계의 결과를 입력으로 받아 처리하며, 각 세차 단계는 독립적으로 교체 가능합니다. 고압 세차기를 더 강력한 모델로 바꿔도 다른 단계에는 영향이 없습니다.

**역사적 발전 과정**:

1. **1970s - Unix 파이프라인**: Douglas McIlroy가 Unix 쉘에 파이프(|) 연산자 도입
2. **1990s - ETL 도구**: 데이터 웨어하우스 구축을 위한 Extract-Transform-Load 도구
3. **2000s - 스트림 처리**: 실시간 대용량 데이터 처리를 위한 분산 스트림 처리
4. **2010s - 데이터 오케스트레이션**: 복잡한 데이터 파이프라인 관리
5. **2020s - 현대적 ETL/ELT**: 클라우드 네이티브 데이터 파이프라인

### 📢 섹션 요약 비유

파이프-필터 아키텍처는 **도심의 고속도로 교차로**와 같습니다. 각 진입로(필터)는 독립적으로 설계되었지만, 연결 도로(파이프)를 통해 원활한 흐름을 만듭니다. 한 진입로의 공사가 다른 진입로의 운영에 영향을 주지 않듯, 각 필터는 독립적으로 수정 가능합니다.

### 구성 요소 상세 분석

| 요소 | 역할 | 내부 동작 | 프로토콜/포맷 | 비유 |
|:---|:---|:---|:---|:---|
| **Source (Producer)** | 데이터 생성 및 파이프라인 진입점 | 파일 읽기, API 호출, 센서 데이터 수집 | Raw Data, JSON, CSV, Binary | 원수 저장 탱크 |

| **Pipe (Connector)** | 필터 간 데이터 전송 및 버퍼링 | Queue/Buffer 관리, 흐름 제어 | Unix Pipe, HTTP, TCP, IPC | 배관 라인 |
| **Sink (Consumer)** | 최종 결과 저장 및 표현 | DB 저장, 파일 기록, UI 표시 | DB Protocol, File System | 정수 저장 탱크 |

**[다이어그램 해설]**

1. **Source Layer**: 다양한 데이터 소스로부터 원시 데이터를 수집합니다. 각 소스는 독립적이며 동시에 여러 소스에서 데이터가 유입될 수 있습니다.

2. **Ingestion Pipe**: 데이터 흐름의 버퍼 역할을 합니다. 생산자와 소비자의 속도 차이를 흡수하며, Kafka나 RabbitMQ 같은 메시지 큐가 이 역할을 수행합니다.

3. **Processing Filters**: 실제 데이터 변환이 일어나는 체인입니다. 각 필터는:
   - 이전 필터의 출력을 입력으로 받음
   - 단일 책임 변환 수행
   - 다음 필터가 처리 가능한 포맷으로 출력
   - 독립적 테스트 및 재사용 가능

4. **Branching Filters**: 조건부 분기가 필요한 경우 라우터 필터가 데이터를 여러 경로로 분배합니다. 예: 데이터 유형별 다른 처리 파이프라인

5. **Aggregation Pipe**: 시간 기반 윈도우나 배치 처리를 통해 결과를 집계합니다.

6. **Sink Layer**: 최종 결과를 다양한 타겟에 저장합니다.

**[상세 설명]**

1. **수집 (Ingestion)**: 원시 데이터는 불완전할 수 있습니다. Null 값, 잘못된 타입, 포맷 오류가 포함될 수 있습니다.

2. **검증 (Validation)**: 스키마 검증을 수행합니다. 필수 필드 확인, 데이터 타입 검증, 포맷 검사(이메일, 전화번호)를 수행합니다. 유효하지 않은 데이터는:
   - 별도 오류 파이프라인으로 라우팅
   - 기본값으로 대체
   - 레코드 자체를 폐기

3. **변환 (Transformation)**: 비즈니스 로직에 맞게 데이터를 변환합니다.
   - 타입 변환: 문자열 "25" → 정수 25
   - 필드명 매핑: name → full_name
   - 값 매핑: "KR" → "South Korea"

4. **농축 (Enrichment)**: 외부 데이터를 조인하여 정보를 추가합니다.
   - GeoIP lookup으로 위치 정보 추가
   - 참조 데이터와 조인
   - 계산 필드 추가

### 핵심 알고리즘 및 구현

**필터 인터페이스 설계**:

```java
/**
 * 파이프-필터 아키텍처의 기본 필터 인터페이스
 * @param <I> 입력 타입
 * @param <O> 출력 타입
 */
public interface Filter<I, O> {

    /**
     * 입력 데이터를 받아 변환된 출력을 생성
     * @param input 입력 데이터 스트림
     * @return 변환된 출력 데이터 스트림
     */
    Stream<O> process(Stream<I> input);

    /**
     * 필터가 처리할 수 있는 데이터의 최대 크기
     * 메모리 제한 환경에서 역압(Backpressure) 제어에 활용
     */
    default int maxBatchSize() {
        return 1000;
    }
}

/**
 * 필터 체인을 구성하는 빌더
 */
public class PipelineBuilder<T> {

    private final List<Filter<?, ?>> filters = new ArrayList<>();

    @SuppressWarnings("unchecked")
    public <R> PipelineBuilder<R> addFilter(Filter<T, R> filter) {
        filters.add(filter);
        return (PipelineBuilder<R>) this;
    }

    public Stream<?> build(Stream<T> source) {
        Stream<?> current = source;
        for (Filter<?, ?> filter : filters) {
            current = ((Filter<Object, Object>) filter).process(current);
        }
        return current;
    }
}

/**
 * 구체적인 필터 구현 예시: 이메일 검증 필터
 */
public class EmailValidationFilter implements Filter<User, User> {

    private static final Pattern EMAIL_PATTERN =
        Pattern.compile("^[A-Za-z0-9+_.-]+@(.+)$");

    @Override
    public Stream<User> process(Stream<User> input) {
        return input
            .filter(user -> user.getEmail() != null)
            .filter(user -> EMAIL_PATTERN.matcher(user.getEmail()).matches())
            .map(user -> {
                user.setEmailVerified(true);
                return user;
            });
    }
}

/**
 * 사용 예시
 */
public class ETLPipeline {

    public static void main(String[] args) {
        List<User> rawData = fetchUsersFromSource();

        Stream<User> processed = new PipelineBuilder<User>()
            .addFilter(new EmailValidationFilter())
            .addFilter(new DataEnrichmentFilter())    // 외부 API 호출
            .addFilter(new AgeCalculationFilter())    // 생일로 나이 계산
            .addFilter(new PiiMaskingFilter())        // 개인정보 마스킹
            .build(rawData.stream());

        List<User> result = processed.collect(Collectors.toList());
        saveToDatabase(result);
    }
}
```

**실시간 스트림 처리 (Reactive Streams 기반)**:

```java
/**
 * Project Reactor를 활용한 비동기 파이프-필터 구현
 */
public class ReactiveETLPipeline {

    public static void main(String[] args) {
        Flux<User> sourceFlux = Flux.fromIterable(fetchUsers());

        sourceFlux
            .flatMap(user -> Mono.just(user)
                .subscribeOn(Schedulers.parallel())  // 병렬 처리
                .map(ValidationFilter::validate)     // 필터 1
                .map(TransformationFilter::transform)// 필터 2
                .map(EnrichmentFilter::enrich)       // 필터 3
                .onErrorResume(e -> {                 // 에러 핸들링
                    log.error("Processing failed", e);
                    return Mono.empty();              // 실패 시 건너뜀
                })
            )
            .buffer(100)                             // 배치 처리
            .flatMap(batch -> saveToDatabase(batch)) // 싱크
            .subscribe(
                result -> log.debug("Processed: {}", result),
                error -> log.error("Pipeline failed", error),
                () -> log.info("Pipeline completed")
            );
    }
}
```

### 역압 (Backpressure) 처리

```java
/**
 * 역압을 고려한 Reactor 기반 필터 체인
 */
public class BackpressureAwarePipeline {

    public Flux<ProcessedData> buildPipeline() {
        return Flux.create(sink -> {
                // 데이터 소스 (가능한 빠름)
                dataSource.subscribe(data -> sink.next(data));
            })
            .onBackpressureBuffer(           // 버퍼링 전략
                1000,                         // 최대 버퍼 크기
                BufferOverflowStrategy.DROP_OLDEST  // 오래된 데이터 폐기
            )
            .flatMap(data ->                   // 동시 처리 제어
                processFilterA(data),
                4                              // 최대 동시성 (concurrency)
            )
            .flatMap(data ->
                processFilterB(data),
                2                              // 무거운 작업은 낮춤
            )
            .onBackpressureLatest()           // 최신 데이터 우선
            .doOnBackpressure(dropped ->
                log.warn("Dropped {} records", dropped)
            );
    }
}
```

파이프-필터 아키텍처는 **물류 센터의 컨베이어 벨트 시스템**과 같습니다. 상품(데이터)이 여러 검사/포장 공정(필터)을 통과하며 최종 배송됩니다. 각 공정은 독립적으로 교체 가능하며, 병목 구간을 파악해 추가 작업자를 투입(스케일 아웃)할 수 있습니다. 컨베이어 벨트의 속도를 조절하여 전체 시스템의 효율을 최적화합니다(역압 제어).

### 도입 체크리스트

**기술적 고려사항**

| 항목 | 확인사항 | 가이드라인 |
|:---|:---|:---|
| **데이터 특성** | 스트림 vs 배치 | 실시간성이 중요하면 스트림, 대량 집계는 배치 |
| **처리 복잡도** | 단순 변환 vs 복잡 로직 | 단순 변환에는 적합, 복잡한 분기는 다른 패턴 고려 |
| **확장성 요구** | 수직 vs 수평 확장 | 필터 단위 수평 확장 용이성 확인 |
| **순서 보장** | 순서 의존 여부 | 순서가 중요하면 직렬 파이프라인 |
| **상태 관리** | Stateless vs Stateful | 가능한 Stateless 설계, 필요 시 외부 저장소 |
| **오류 처리** | 내구성 요구사항 | Dead Letter Queue 활용 |
| **모니터링** | 가시성 확보 | 각 필터의 메트릭(처리량, 지연) 노출 |

**운영·보안적 고려사항**

| 항목 | 확인사항 | 가이드라인 |
|:---|:---|:---|
| **데이터 암호화** | 전송/저장 중 암호화 | TLS, at-rest encryption |
| **PII 처리** | 개인정보 마스킹 | 농축 단계 전에 마스킹 필터 |
| **접근 제어** | 파이프라인 접근 권한 | RBAC 기반 접근 통제 |
| **감사 로그** | 처리 기록 보관 | 각 필터의 입출력 로그 |

| **리소스 제한** | CPU/메모리 할당 | 컨테이너 리소스 쿼터 설정 |
| **버전 관리** | 필터 버전 호환성 | 스키마 레지스트리 활용 |

### 📢 섹션 요약 비유

파이프-필터 아키텍처는 **레스토랑 주방의 조리 시스템**과 같습니다. 각 스테이션(필터)은 전문화되어 있고, 요청(데이터)은 순차적으로 처리됩니다. 문제가 발생하면 해당 스테이션에서 차단되어 다음 단계로 넘어가지 않습니다. 주방장은 각 스테이션의 처리량을 모니터링하며 병목을 해소합니다(역압 제어). 하지만 한 요리사가 모든 작업을 처리하려 하면(만능 필터) 효율이 급격히 떨어집니다.

4. **규정 준수**: 각 단계에서 데이터 품질과 규정 준수 검증 가능

## 👶 어린이를 위한 3줄 비유 설명

**1. 무엇인가요?**
파이프-필터는 여러 정수 장비가 연결된 **물 처리 시스템** 같은 것입니다. 더러운 물이 첫 번째 필터를 통과하면 큰 오물이 걸러지고, 두 번째 필터에서는 미세한 먼지가 제거되며, 마지막에는 깨끗한 물이 나옵니다.

**2. 어떻게 작동하나요?**
각 필터는 **자신만의 일**만 챙깁니다. 첫 번째는 큰 오물만 걸러내고, 두 번째는 작은 먼지만 제거합니다. 필터 사이에는 파이프가 있어서 물이 한 방향으로만 흐르고, 각 필터를 필요에 따라 교체하거나 추가할 수 있습니다.

**3. 왜 필요한가요?**
모든 작업을 한 번에 하려고 하면 너무 복잡해집니다. 파이프-필터를 사용하면 각 단계를 **독립적으로** 만들고 테스트할 수 있어서, 문제가 생겨도 어디에서 생겼는지 바로 알 수 있고 고치기도 쉽습니다.
