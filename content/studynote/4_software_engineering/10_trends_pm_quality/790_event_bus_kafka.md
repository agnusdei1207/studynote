# 790. 이벤트 버스 카프카(Kafka) 비동기 내결함성 설계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서비스 간의 직접적인 호출(RPC) 대신, 상태 변화를 나타내는 이벤트를 중앙의 메시지 통로인 **이벤트 버스(Event Bus)**를 통해 비동기적으로 주고받는 **이벤트 기반 아키텍처(EDA)**의 핵심 인프라이다.
> 2. **기술적 우위**: 아파치 카프카(Kafka)는 고성능 분산 로그 저장소 구조를 활용하여, 메시지의 순서 보장, 영속성(Persistence), 그리고 대규모 트래픽 처리를 위한 **수평적 확장성**을 제공한다.
> 3. **가치**: 서비스 간의 결합도를 낮추어 독립적인 배포와 확장을 가능하게 하며, 특정 서비스 장애 시에도 메시지를 안전하게 보관했다가 재처리하는 강력한 **내결함성(Fault Tolerance)**을 실현한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "강하게 연결된 시스템의 붕괴"

전통적인 HTTP/REST 기반의 동기 호출 방식은 서비스 A가 서비스 B를 부를 때 B가 죽어있으면 A도 함께 망가집니다. 서비스가 늘어날수록 이 연쇄 장애는 걷잡을 수 없습니다. **이벤트 버스 (Event Bus)**는 "나는 할 말만 던져두고 내 할 일을 하겠다"는 비동기 사상을 통해 시스템의 생존력을 높입니다. 특히 **카프카**는 이 분야의 '표준 고속도로'로 자리 잡았습니다.

### 💡 비유: 동네 소식통과 게시판

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        이벤트 버스(Kafka) 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 기존 방식 (동기 호출 - 전화):                                             │
│     친구에게 소식을 전하려고 전화를 함. 친구가 전화를 안 받으면 나도 소식을        │
│     못 전하고 계속 전화기만 붙들고 있어야 함. (Blocking)                         │
│                                                                             │
│  2. 이벤트 버스 방식 (카프카 - 마을 게시판):                                   │
│     나는 게시판(Kafka)에 "내일 잔치함!"이라고 포스트잇(Event)을 붙이고             │
│     집에 가서 잠. (Asynchronous)                                               │
│     - 소식이 궁금한 사람(Consumer)은 각자 편한 시간에 게시판에 와서 읽어감.        │
│     - 게시판은 튼튼해서 비가 와도 포스트잇이 안 젖음 (Persistence).               │
│                                                                             │
│  → 보낸 사람과 받는 사람이 **동시에 깨어있지 않아도** 소통이 완벽한 시스템!        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 카프카의 핵심 설계 아키텍처
- **Topic & Partition**: 메시지를 분류하는 단위(Topic)와 병렬 처리를 위해 이를 쪼갠 단위(Partition).
- **Producer / Consumer**: 이벤트를 발행하는 주체와 구독하는 주체.
- **Offset Management**: Consumer가 어디까지 읽었는지 기록하여 장애 복구 시 이어서 처리 가능하게 함.
- **Replication**: 여러 노드에 메시지를 복제하여 서버 한두 대가 죽어도 데이터 유실 방지.

### 2. 내결함성(Fault Tolerance) 보장 기법
- **At-least-once Delivery**: 메시지가 최소 한 번은 전달됨을 보장.
- **Consumer Group**: 동일한 그룹 내 Consumer 중 하나가 죽으면 다른 Consumer가 즉시 업무 대행 (Rebalancing).
- **Log Compaction**: 동일 키의 메시지는 최신 상태만 남겨 저장 공간 효율화 및 빠른 복구 지원.

### 3. 이벤트 버스 통신 구조도 (ASCII)

```text
    [ Producer A ] ──▶ [ Topic: Orders ] ──▶ [ Partition 1 ] ──▶ [ Consumer Group 1 ]
    (주문 서비스)           (Kafka Cluster)    [ Partition 2 ]       (결제 서비스)
                                               [ Partition 3 ]
                                                      │
    [ Producer B ] ──▶ [ Topic: Stocks ] ─────────────┘ ──────▶ [ Consumer Group 2 ]
    (재고 서비스)                                                    (알림 서비스)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. Message Queue(RabbitMQ) vs Event Bus(Kafka)

| 항목 | 일반 MQ (RabbitMQ) | 이벤트 버스 (Kafka) |
|:---:|:---|:---|
| **메시지 보관** | 소비 후 즉시 삭제 (휘발성) | **디스크에 영구 보관 (Persistence)** |
| **처리량** | 수만 건/초 (중간) | **수백만 건/초 (초고속)** |
| **소비 모델** | 스마트 브로커 (Push) | **덤 브로커, 스마트 컨슈머 (Pull)** |
| **주 용도** | 단순 작업 전달 | **이벤트 스트리밍, 로그 통합, 분석** |

### 2. 기술적 시너지: 데이터 정합성 (Saga Pattern)
이벤트 버스는 분산 트랜잭션의 대안인 **Saga 패턴**의 척추 역할을 합니다. 각 서비스가 트랜잭션 완료 후 이벤트를 버스에 실어 보내면, 다음 서비스가 이를 이어받아 처리함으로써 거대한 비즈니스 흐름의 '최종 일관성'을 달성합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 고부하 트래픽 대응 및 장애 격리
- **상황**: 블랙 프라이데이 주문 폭주로 인해 '알림 메일' 발송 서버가 과부하로 다운됨.
- **결단**: **Kafka를 활용한 트래픽 버퍼링**. 
- **판단**: 주문 서비스가 직접 알림 서버를 호출하지 않고 Kafka에 '주문 완료' 이벤트를 쌓아둠. 
- **조치**: 알림 서버가 복구된 후, Kafka에 쌓여있던 수만 건의 이벤트를 자신의 처리 속도에 맞춰 차례대로 읽어 처리.
- **효과**: 알림 서버 장애가 주문 시스템으로 전이되지 않았으며, 데이터 유실 없이 모든 고객에게 지연 발송 성공.

### 📢 기술사적 결언
> "이벤트 버스는 시스템의 **'완충 장치'**이자 **'기록 장치'**다. 동기식 통신이 주는 즉각적인 안도감에 속지 말아야 한다. 분산 시스템이 커질수록 '실패'는 일상이 되며, 이때 카프카와 같은 견고한 이벤트 버스는 시스템을 다시 일으켜 세우는 **'생명선'**이 된다. 아키텍트는 단순한 전달자가 아닌, 메시지의 **'순서 보장'**과 **'중복 처리(Idempotency)'**에 대한 엄격한 기준을 세워 아키텍처의 정합성을 수호해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **가용성 향상**: 결합도 제거로 인한 장애 전파 차단 (Uptime 99.99% 달성 용이).
- **데이터 분석 활용**: 모든 이벤트를 재생(Replay)하여 과거 시점의 상태를 분석하거나 AI 학습에 활용.

### 미래 전망
미래의 이벤트 버스는 **'Serverless Streaming'**으로 진화하고 있습니다. 파티션 개수나 서버 관리를 신경 쓰지 않아도 트래픽에 따라 자동 확장되는 Cloud-native Kafka(예: Confluent Cloud)가 표준이 될 것입니다. 또한 **데이터 메시(Data Mesh)** 아키텍처와 결합하여, 기업의 모든 실시간 데이터가 흐르는 '전사적 신경망'으로서 비즈니스 인텔리전스의 핵심 인프라가 될 것입니다.
