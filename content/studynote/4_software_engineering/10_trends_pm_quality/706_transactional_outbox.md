# 706. 트랜잭셔널 아웃박스 이벤트 유실 방지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스(MSA)에서 데이터베이스 업데이트와 메시지 발행을 **하나의 로컬 트랜잭션**으로 묶어 처리함으로써, 시스템 장애 시에도 이벤트 유실이나 불일치를 방지하는 신뢰성 보장 패턴이다.
> 2. **메커니즘**: 비즈니스 데이터를 변경할 때 알림 메시지를 외부 큐로 직접 보내지 않고, 동일 DB 내의 **아웃박스(Outbox) 테이블**에 먼저 저장한 뒤 별도의 프로세스(Relay)가 이를 읽어 발행한다.
> 3. **가치**: 2PC(2-Phase Commit)와 같은 무거운 분산 트랜잭션 없이도 **최종 일관성(Eventual Consistency)**을 안전하게 달성하며, 'At-least-once' 전달을 보장한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "DB는 성공했는데 메시지 전송에 실패한다면?"

주문 서비스에서 주문 정보를 DB에 저장한 직후, 결제 서비스로 "결제 요청" 이벤트를 보내야 한다고 가정해봅시다. 만약 DB 저장은 성공했는데 네트워크 장애로 이벤트 전송에 실패하면 어떻게 될까요? 고객은 돈을 안 냈는데 주문은 완료된 상태가 됩니다. 반대로 전송 후 DB가 롤백되면 더 큰 혼란이 생깁니다. **트랜잭셔널 아웃박스 (Transactional Outbox)**는 이 '원자성(Atomicity)' 문제를 해결하기 위해 탄생했습니다.

### 💡 비유: 배달 주문과 영수증 보관함

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        트랜잭셔널 아웃박스 비유                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 손님이 피자를 주문함. 주방장은 요리를 하고 배달원에게 알려야 함.          │
│                                                                             │
│  1. 일반 방식 (구두 전달):                                                   │
│     주방장이 요리 완료 후 배달원에게 "가져가세요!"라고 소리침. 배달원이 마침       │
│     화장실에 갔거나 못 들으면 피자는 식어버림. (이벤트 유실)                      │
│                                                                             │
│  2. 아웃박스 방식 (영수증 함):                                                │
│     주방장은 요리를 끝냄과 동시에 **영수증(이벤트)**을 뽑아서 주방 벽에 있는       │
│     **'배달 대기함(Outbox Table)'**에 꽂아둠.                                  │
│     - 주방 일과 영수증 꽂기는 동시에 이루어짐 (로컬 트랜잭션).                    │
│     - 배달원(Message Relay)은 수시로 함을 확인하여 영수증이 있으면 배달을 감.      │
│                                                                             │
│  → 배달원이 잠깐 졸아도 영수증은 남아있으니, 나중에라도 **반드시 배달됨!**        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 작동 프로세스
1. **Service**: 비즈니스 로직을 수행하고, 결과 데이터를 메인 테이블에 저장함과 동시에 이벤트 메시지를 `Outbox` 테이블에 삽입 (동일 DB 트랜잭션 내).
2. **Message Relay**: 아웃박스 테이블을 주기적으로 폴링(Polling)하거나 DB 로그를 읽어(CDC) 새로운 이벤트를 감지.
3. **Broker**: Relay가 읽어온 메시지를 메시지 브로커(Kafka, RabbitMQ)로 발행.
4. **Cleanup**: 발행이 확인된 메시지는 아웃박스 테이블에서 삭제하거나 처리 완료 표시.

### 2. 메시지 릴레이(Relay) 구현 방식

| 방식 | 설명 | 장점 | 단점 |
|:---:|:---|:---|:---|
| **Transaction Log Tailining** | DB의 커밋 로그(Binary Log 등)를 추적하여 변경 감지 | DB 부하 없음, 실시간성 | 구현 복잡, DB 종속적 |
| **Polling Publisher** | 주기적으로 아웃박스 테이블을 `SELECT` 하여 발행 | 구현이 매우 쉬움 | DB 부하 발생 가능, 지연 시간 |

### 3. 아키텍처 구조도 (ASCII)

```text
    [ Microservice ]
           │
    ┌──────▼───────────────────────────────────┐
    │  Local Transaction (ACID)                │
    │  ┌───────────────┐      ┌─────────────┐  │
    │  │ Business Data │      │ Outbox Table│  │
    │  └───────────────┘      └──────┬──────┘  │
    └────────────────────────────────┼─────────┘
                                     │ (Watch)
    ┌────────────────────────────────▼─────────┐
    │         Message Relay (CDC / Polling)    │
    └────────────────────────────────┬─────────┘
                                     │ (Publish)
    ┌────────────────────────────────▼─────────┐
    │         Message Broker (Kafka)           │
    └──────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 2PC (2-Phase Commit) vs Transactional Outbox
- **2PC**: 여러 DB/메시지 큐를 하나의 트랜잭션으로 묶음. 정합성은 높으나 성능이 매우 낮고 장애에 취약 (Blocking 리스크).
- **Outbox**: 로컬 트랜잭션만 사용. 성능이 우수하고 서비스 간 독립성이 유지됨. 대신 **최종 일관성**을 수용해야 함.

### 2. 기술적 시너지: 멱등성 (Idempotency)
아웃박스 패턴은 '최소 한 번(At-least-once)' 전달을 보장합니다. 즉, 동일한 메시지가 중복으로 갈 수 있습니다. 따라서 수신 측 서비스는 동일한 메시지를 여러 번 받아도 결과가 같도록 **멱등성 컨슈머(Idempotent Consumer)**로 설계되어야 완벽한 시너지를 냅니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 이커머스 주문-재고 시스템
- **상황**: 주문이 완료되면 재고 서비스에서 수량을 차감해야 함.
- **결단**: **Debezium을 활용한 CDC 기반 아웃박스 패턴 적용**. 
- **판단**: 주문 서비스 DB에 `OUTBOX` 테이블 생성. 주문 저장 시 `STOCK_REDUCED` 이벤트를 기록. Debezium이 DB 로그를 읽어 Kafka로 전송. 
- **효과**: 주문 서비스의 응답 속도에 영향을 주지 않으면서도, 재고 차단 이벤트가 누락 없이 전달됨을 보장.

### 📢 기술사적 결언
> "분산 시스템에서 **'신뢰할 수 있는 소통'**은 아키텍처의 생명이다. 아웃박스 패턴은 분산 트랜잭션의 복잡성을 회피하면서도 데이터 무결성을 지켜내는 가장 현실적이고 우아한 해법이다. 아키텍트는 단순히 패턴을 도입하는 것을 넘어, 메시지 릴레이의 지연 시간과 중복 발행에 대한 **'예외 처리 전략'**을 함께 수립해야 진정한 시스템 회복탄력성을 확보할 수 있다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **데이터 일관성**: 이벤트 유실율 0% 달성 가능.
- **시스템 가용성**: 외부 브로커 장애 시에도 내부 비즈니스 로직은 정상 수행 (느슨한 결합).

### 미래 전망
미래의 데이터베이스는 아예 **'내장형 아웃박스'** 기능을 제공하는 방향으로 진화할 것입니다. 별도의 릴레이 프로세스 없이 DB 엔진 자체가 메시지 브로커의 역할을 일부 수행하거나, 표준화된 이벤트 스트리밍 인터페이스를 내장하여 '트랜잭션-이벤트 일체화'를 실현할 것입니다. 또한 클라우드 서비스(AWS EventBridge 등)와의 통합을 통해 서버리스 환경에서도 이 패턴이 기본 제공되는 'Managed Outbox' 시대가 도래할 것입니다.
