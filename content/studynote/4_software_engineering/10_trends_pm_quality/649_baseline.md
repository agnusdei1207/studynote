# 649. 기준선 (Baseline) 수립 변경 통제

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 개발 생명주기(SDLC)의 특정 시점에서 공식적으로 검토되고 합의된 **산출물의 집합**으로, 이후 모든 변경의 기준이 되는 "공식적인 상태"다.
> 2. **역할**: 무분별한 변경(Scope Creep)을 차단하고, 프로젝트 구성원 간의 합의된 목표를 공유하며, 장애 발생 시 신속하게 돌아갈 수 있는 **복구 지점(Recovery Point)** 역할을 한다.
> 3. **통제**: 베이스라인 수립 이후의 모든 변경은 반드시 형상 통제 위원회(CCB)의 공식적인 승인 절차를 거쳐야 하는 **엄격한 통제 상태**로 전이된다.

---

## Ⅰ. 개요 (Context & Background)

### 기준선 (Baseline)의 정의

IEEE에서는 베이스라인을 "형상 항목의 공식적인 검토 및 동의를 거쳐 확정된 상태"로 정의합니다. 프로젝트 초기에는 요구사항이 베이스라인이 되고, 개발 중에는 설계서가, 마지막에는 소스코드와 실행 파일이 베이스라인이 됩니다. 베이스라인이 수립되지 않은 프로젝트는 '달리는 기차 위에서 선로를 까는 것'과 같이 불안정합니다.

### 💡 비유: 게임의 '체크포인트'와 '세이브 파일'

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        베이스라인 비유                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 아주 어려운 게임 스테이지(프로젝트)를 깨야 함.                           │
│                                                                             │
│  1. 베이스라인 (체크포인트):                                                  │
│     보스몹을 잡기 전, 지금까지의 아이템과 레벨을 '공식적으로 저장'함.             │
│     이후에 게임 오버가 되어도 이 저장된 시점(기준선)에서 다시 시작할 수 있음.      │
│                                                                             │
│  2. 변경 통제:                                                               │
│     저장된 세이브 파일을 마음대로 수정할 수 없음. 수정하려면 '수정 요청서'를       │
│     쓰고 관리자의 허락(CCB 승인)을 받아야 함.                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. SDLC 단계별 주요 기준선 유형

| 기준선 명칭 | 수립 시점 | 포함되는 주요 형상 항목 (CI) |
|:---:|:---|:---|
| **기능적 기준선** | 요구사항 분석 완료 | 사용자 요구사항 명세서, 시스템 명세서 |
| **분배적 기준선** | 요구사항 분석 후반 | 요구사항 정의서, 인터페이스 정의서 |
| **설계 기준선** | 기본/상세 설계 완료 | 아키텍처 설계서, DB 설계서, 클래스 다이어그램 |
| **시험 기준선** | 테스트 계획 완료 | 테스트 계획서, 테스트 케이스, 시나리오 |
| **제품 기준선** | 개발 및 테스트 완료 | 최종 소스코드, 실행 파일, 사용자 매뉴얼 |
| **운영 기준선** | 운영 이관 완료 | 설치 가이드, 패치 이력, 운영 환경 설정값 |

### 2. 베이스라인 전이 및 통제 아키텍처 (ASCII)

```text
    [ 비공식 상태 ] ───── 공식 검토(FTR) ────▶ [ 공식 기준선 수립 ]
    (개발자 PC 내부)       (인스펙션 등)         (형상 저장소 보관)
                                                   │
                                                   ▼
    [ 변경 요청 발생 ] ◀──── CCB 심의 ◀───── [ 엄격한 통제 단계 ]
      (CR 처리)           (영향도 분석)        (임의 수정 불가)
```

### 3. 기준선 관리의 3대 핵심 규칙
- **공식성**: 반드시 권한 있는 자(PM, 고객 등)의 승인이 있어야 함.
- **불변성**: 승인 없이 일절 수정 불가 (Read-only 권한 부여).
- **식별성**: 버전 번호(예: v1.0, v2.0)를 통해 명확히 구분되어야 함.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 버전(Version) vs 기준선(Baseline)
- **버전**: 개별 파일이나 모듈의 변화를 나타내는 **미시적 단위** (예: index.js v1.1).
- **기준선**: 특정 시점의 모든 관련 파일들을 묶은 **거시적 단위** (예: 1차 릴리즈 버전 전체 패키지).

### 2. 기술적 시너지: Git Tag 및 Release
현대적인 개발 환경에서 베이스라인 수립은 Git의 **'Tag'** 기능을 통해 수행됩니다. 특정 커밋 해시(Commit Hash)에 `v1.0.0-baseline`과 같은 태그를 붙임으로써, 전 세계 어디서든 동일한 상태의 산출물을 내려받을 수 있게 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 설계 변경으로 인한 개발 혼선 방지
- **상황**: 개발자들이 각자 최신이라고 생각하는 설계서를 보고 코딩하여 모듈 간 인터페이스가 맞지 않음.
- **결단**: 
    1. 현재 설계서를 모두 취합하여 **'설계 기준선 v1.0'**으로 긴급 확정 및 공표.
    2. 모든 개발자에게 해당 기준선 이하의 설계서 사용 금지 명령.
- **판단**: 이후의 변경은 '변경 요청서'를 통해서만 받도록 절차를 강제하여 인터페이스 일관성 회복.

### 📢 기술사적 결언
> "기준선은 프로젝트의 **'말뚝'**이다. 말뚝을 단단히 박지 않으면 프로젝트는 조류에 휩쓸려 어디로 갈지 모르게 된다. 아키텍트는 각 단계가 끝날 때마다 명확한 말뚝(Baseline)을 박고, 이를 넘어가는 변경에 대해서는 CCB라는 파수꾼을 통해 철저히 필터링해야 한다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **재작업 감소**: 기준 미비로 인한 중복/오류 작업 40% 이상 감소.
- **추적성 확보**: 요구사항 변경이 코드와 테스트에 어떻게 반영되었는지 완벽한 이력 추적 가능.

### 미래 전망
최근에는 **'Infrastructure as Code (IaC)'**의 확산으로 인해 서버 설정과 클라우드 인프라 자체도 베이스라인 관리의 대상이 되고 있습니다. 테라폼(Terraform) 상태 파일 등을 베이스라인으로 관리하여, 소프트웨어뿐만 아니라 인프라 전체를 특정 시점으로 되돌리는 **'Full-stack Baseling'** 기술이 표준으로 자리 잡고 있습니다.
