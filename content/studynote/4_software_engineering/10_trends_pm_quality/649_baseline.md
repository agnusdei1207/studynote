# 649. 기준선 (Baseline) 수립 변경 통제

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SDLC (Software Development Life Cycle) 전체의 안정성을 보장하는 **형상 관리 (Configuration Management)**의 핵심 메커니즘으로, 특정 시점의 산출물을 논리적/물리적으로 고정하는 '기준점'이다.
> 2. **가치**: 개발 생산성을 30% 이상 향상시키고, 잦은 요구사항 변경으로 인한 **Scope Creep (범위 확산)** 현상을 원천 차단하며, 결함 발생 시 RTO (Recovery Time Objective)를 획기적으로 단축한다.
> 3. **융합**: 최근에는 소스코드뿐만 아니라 Docker 이미지, Kubernetes 매니페스트, Terraform 스크립트 등 인프라 자산까지 포함하는 **IaC (Infrastructure as Code)** 기반의 확장된 베이스라인으로 진화하고 있다.

---

## Ⅰ. 개요 (Context & Background)

### 기준선 (Baseline)의 정의와 철학

**기준선 (Baseline)**이란 단순한 파일 저장 단계가 아니라, **형상 항목 (Configuration Item, CI)**이 공식적인 검토(Review)와 승인(Approval)을 거쳐 이후 변경 통제의 기준이 되는 **"불변의 상태(Immutable State)"**로 격상되는 과정을 의미합니다.

IEEE 828 표준에서는 이를 "향후의 변경은 공식적인 변경 통제 절차를 통해서만 수행될 수 있도록 승인된 작업 산출물 집합"으로 정의합니다. 즉, 베이스라인이 수립되기 전까지는 '유동적인 상태(Working State)'에 있지만, 수립되는 순간 '통제된 상태(Controlled State)'로 전이됩니다.

### 💡 비유: 건축의 '설계 확정 (Freeze)'과 '기공식'

```text
[유동적인 상태]                [통제된 상태 (Baseline)]
(설계 변경 자유롭음)            (설계 확정 및 변경 불가/승인제)
      │                            │
      │  ▶  형상관리자(CO) 승인    │
      ▼                            ▼
┌─────────────┐              ┌─────────────┐
│  연필 스케치  │      ────▶   │  청사진 인장  │
│  (초안/와이어프레임) │              │  (공식 도면)   │
└─────────────┘              └─────────────┘
```

### 등장 배경 및 필요성
1.  **기존 한계**: 애자일(Agile) 등 변화에 유연한 개발 방법론론조차, 핵심 아키텍처나 인터페이스가 매번 덮어쓰기되면 **Regression (회귀)**이 발생하여 시스템 붕괴로 이어짐.
2.  **혁신적 패러다임**: "변경을 막는 것이 아니라, **변경을 관리 가능한 단위로 쪼개어 통제**"함으로써 안정성과 유연성을 동시에 확보.
3.  **현재 요구**: CI/CD (Continuous Integration/Continuous Deployment) 파이프라인 내에서 **어떤 버전을 프로덕션(Production)으로 배포할지 결정**하는 절대적인 기준점(Gate) 역할.

> **📢 섹션 요약 비유**: 프로젝트 진행은 마치 영화 촬영과 같습니다. 아무렇게나 찍은 컷(Cut)들 중에서 감독이 "OK, 이 컷은 완성이다!"라고 마크(Mark)하는 순간이 베이스라인 수립입니다. 이후에 이 컷을 수정하려면 제작사와 배급사의 회의(CCB)가 필요합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. SDLC 단계별 기준선 계층 구조 (Baseline Hierarchy)

소프트웨어 개발 수명 주기(SDLC)에서 베이스라인은 계층적으로 구축됩니다. 하위 베이스라인이 변경되면 상위 베이스라인에 영향을 미치는 **Ripple Effect (리플 효과)**를 고려해야 합니다.

| 단계 | 기준선 명칭 | 주요 구성 요소 (CI) | 통제 포인트 (Control Point) |
|:---:|:---|:---|:---|
| **요구사항** | **기능적 기준선 (Functional Baseline)** | SRS (Software Requirements Specification), 사용자 스토리, UI/UX 프로토타입 | **What** (무엇을 만들 것인가?) |
| **설계** | **할당된 기준선 (Allocated Baseline)** | SDD (Software Design Document), 인터페이스 명세서(ID), DB 스키마, API契约 | **How** (어떻게 구현할 것인가?) |
| **개발** | **개발 기준선 (Development Baseline)** | 소스 코드, 유닛 테스트 코드, 빌드 스크립트 | **Code** (구현 물리적 단위) |
| **제품** | **제품 기준선 (Product Baseline)** | 실행 파일(바이너리), 배포 이미지(Docker/ISO), 사용자 매뉴얼, 설치 가이드 | **Release** (고객에게 전달되는 것) |

### 2. 기준선 수립 및 변경 통제 절차 (ASCII)

다음은 개발자가 변경 요청을 제기하여 베이스라인을 갱신하는 전체 수명 주기(Lifecycle)입니다.

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                        [SCM 흐름도]                                   │
  └───────────────────────────────────────────────────────────────────────┘

  (1) 개발 환경 (Development)          (2) 통제 환경 (Repository)
  ┌─────────────────┐                 ┌───────────────────────────────┐
  │  ─────┐         │                 │                               │
  │  Edit │         │                 │   ┌─────────────────────┐     │
  │  (작업 중)       │                 │   │  Master (Trunk)     │     │
  │  ─────┘         │                 │   │  ──────┐            │     │
  └────┬───────────┘                 │   │  BL v1.0 (Immutable)│     │
       │                             │   └───────┬──────────────┘     │
       │ Check-in                    │           │                    │
       ▼                             │   (4) CCB 승인 시 Merge        │
  ┌─────────────────┐                 │           ▼                    │
  │  [잠정 Zone]    │                 │   ┌─────────────────────┐     │
  │  ─────┐         │                 │   │  BL v1.1 (New)      │     │
  │  Review/Test    │                 │   │  ──────┐            │     │
  │  ─────┘         │                 │   └─────────────────────┘     │
  └────┬───────────┘                 │                               │
       │ (3) CR (Change Request)     └───────────────────────────────┘
       │
       ▼
  ┌───────────────────────────────────────────────────────────────────┐
  │                     CCB (Change Control Board)                    │
  │  ──────────────────────────────────────────────────────────       │
  │  [심의 사항]                                                        │
  │  ① 영향도 분석 (Impact Analysis): "베이스라인 v1.0 코드를 건드는가?"  │
  │  ② 보안/성능 테스트 결과 확인                                        │
  │  ③ 승인/반려 결정 (Approval/Rejection)                             │
  └───────────────────────────────────────────────────────────────────┘
```

**상세 해설:**
1.  **Uncontrolled (비통제 영역)**: 개발자는 자유롭게 코드를 수정(Checkout)하고 커밋합니다. 이 시점의 코드는 'Working Copy'입니다.
2.  **Transition (전이)**: 개발자가 기능 완료 후 Pull Request(MR)를 생성하면, 자동화된 파이프라인(CI/CD)에서 빌드 및 테스트가 수행됩니다.
3.  **Controlled (통제 영역)**: 테스트 통과 후, **CR (Change Request)**가 발생합니다. 이때 해당 변경 사항이 기존 베이스라인(Baseline v1.0)을 수정하는 행위인지 확인합니다.
4.  **Baseline Update (기준선 갱신)**: CCB(CCB: Configuration Control Board)가 승인하면, 마스터 브랜치(Master Branch)에 병합(Merge)되고, **Tag**나 **Release**를 생성하여 **BL v1.1**이라는 새로운 불변의 지점을 확정합니다.

### 3. 핵심 기술 매커니즘 (Git Flow)

현대적인 **Git Flow 전략**에서 베이스라인은 아래와 같이 구현됩니다.

```text
*   (Tag: v1.0)  <--- [제품 기준선 Production Baseline]
| 
| * (dev)        <--- [개발 기준선 Development Baseline]
|/
*   (Tag: v0.9)  <--- [이전 기준선]
```

- **Object Store (객체 저장소)**: Git의 경우 모든 파일의 내용을 압축하여 **Blob Object**로 저장하며, 파일 시스템의 스냅샷(Snapshot)을 **Tree Object**로 보관합니다. 베이스라인은 특정 시점의 **Commit Hash (SHA-1)** 가리킵니다.
- **Immutability (불변성)**: 한번 생성된 커밋(베이스라인)은 내용을 절대 변경할 수 없습니다. 변경을 위해서는 반드시 새로운 커밋을 생성해야 하므로, **Append-Only (추가 전용)** 로그가 자동으로 생성되어 훼손이 불가능합니다.

> **📢 섹션 요약 비유**: 베이스라인은 암호화폐의 '블록체인'과 유사합니다. 이전 데이터(상태)를 해시값으로 고정해두고, 새로운 변경(거래)이 발생하면 이전 블록을 참조하여 새로운 블록(버전)을 체인형으로 연결하는 구조입니다. 이로 인해 "언제 누가 무엇을 변경했는가"가 위변조 불가능한 형태로 보존됩니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 버전 관리 vs 기준선 관리 (정량적 비교)

| 구분 | **Version (버전)** | **Baseline (기준선)** | **비고** |
|:---|:---|:---|:---|
| **정의** | 개별 파일의 단순 변화 횟수 | **산출물 집합(Set)의 논리적 묶음** | 1:N 관계 |
| **예시** | `main.c v1.1` (수정 1회) | `Release 1.0` (v1.0 파일 200개 묶음) | |
| **관리 대상** | 파일(File) 단위 | 시스템(System) 단위 | |
| **변경 권한** | 개발자 자유롭게 생성 | **CCB** 승인 필수 | |
| **비용(Cost)** | 저장소 디스크 소모 증가 | 변경 통제 프로세스에 따른 **오버헤드 발생** | |

### 2. 타 영역과의 융합 (Synergy)

#### A) 보안(Security)과의 융합: 취약점 기준선 (Vulnerability Baseline)
- **개념**: 소스 코드를 정적으로 분석(SAST)하여 취약점이 없는 상태를 **"Clean Baseline"**으로 설정.
- **효과**: 이후 배포되는 모든 버전은 이 Clean Baseline을 기준으로 **Diff(차분) 검사**를 수행하여, 새로 주입된 악성 코드나 암호화 알고리즘 오타를 즉각 탐지할 수 있습니다.

#### B) 인프라(Infrastructure)와의 융합: IaC (Infrastructure as Code)
- **개념**: 테라폼(Terraform)의 `.tfstate` 파일이나 쿠버네티스(K8s)의 매니페스트 YAML을 베이스라인화.
- **사례**: AWS EC2 인스턴스 설정이 변경된 경우, 코드 리뷰(PR)를 통해 **"인프라 기준선 변경 요청"**을 처리하고, `terraform plan` 결과를 CCB에게 승인 받음으로써, **서킷 브레이커(Circuit Breaker)** 사고를 예방합니다.

> **📢 섹션 요약 비유**: 버전은 사람의 '나이(생년월일)'와 같습니다. 매년 자동으로 하나씩 늘어나는 자연스러운 변화입니다. 하지만 베이스라인은 '성인식'이나 '졸업장'과 같습니다. 사회적(시스템적)으로 특정 자격(상태)을 공인하고, 이후부터는 다른 대우(통제)를 받게 되는 **사회적 계약(Social Contract)**의 관점입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 치명적인 장애 복구 (Disaster Recovery)

**상황**: 금융권 핵심 결제 시스템에서 배포 이후 장애가 발생하여 RTO 10분 이내 서비스 복구가 필요함.
**문제점**: 개발자가 로컬에서 수정한 소스코드가 서버에 반영되어 있어, 형상 저장소(Git)의 최신 코드와 실제 운영 서버의 상태가 불일치하는 상태(Drift)임.

**의사결정 과정 (Engineer's Judgment):**
1.  **안티패턴 배제**: 현재 운영 중인 서버의 데이터베이스 덤프를 뜨거나 실행 파일을 그대로 복사하는 것은 환경 설정(Environment Variable) 누락 등 **구성 Drift**를 유발할 수 있어 위험함.
2.  **베이스라인 활용**: 가장 최근에 승인된 **제품 기준선(Product Baseline)** 태그(예: `release-v2.3-stable`)를 확인.
3.  **재배포 (Redeploy)**: CI/CD 파이프라인을 통해 해당 베이스라인을 재빌드하여 자동으로 서버에 롤백(Rollback) 진행.
4.  **결과**: 소스코드 + 설정파일 + 라이브러리 의존성이 모두 정확히 일치하는 상태로 복구됨. **RTO 목표 달성.**

### 2. 도입 체크리스트 (Adoption Checklist)

| 분류 | 점검 항목 | 세부 내용 |
|:---|:---|:---|
| **기술적** | **식별 가능성** | 베이스라인에 고유 ID(Tag/Version)와 **메타데이터(빌드 번호, 빌더, 날짜)**가 포함되어 있는가? |
| **기술적** | **불변성 보장** | 베이스라인 저장소(Repository)
