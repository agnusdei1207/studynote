# 659. 스프린트 백로그 / 프로덕트 백로그

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 제품의 전체 비전을 담은 **프로덕트 백로그**와, 당장 이번 스프린트에서 수행할 할 일을 정의한 **스프린트 백로그**는 스크럼의 계획과 실행을 지탱하는 핵심 산출물이다.
> 2. **역동성**: 프로덕트 백로그는 시장 변화에 따라 끊임없이 우선순위가 조정(Grooming)되는 살아있는 문서이며, 스프린트 백로그는 해당 기간 동안 팀이 완수할 것을 약속한 확정된 목표다.
> 3. **가치**: 무엇을 먼저 해야 할지(Prioritization)와 현재 우리 팀이 무엇을 하고 있는지(Visibility)를 명확히 하여, 자원 낭비를 방지하고 비즈니스 가치 전달을 최적화한다.

---

## Ⅰ. 개요 (Context & Background)

### 백로그(Backlog)의 정의

백로그는 직역하면 '미뤄둔 일'이지만, 애자일에서는 '앞으로 해야 할 일의 목록'을 의미합니다. 폭포수 모델의 '요구사항 정의서'가 고정된 문서라면, 스크럼의 백로그는 사용자 스토리(User Story)를 기반으로 계속해서 진화하는 유연한 목록입니다.

### 💡 비유: 전체 메뉴판과 오늘의 추천 요리

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        백로그 체계 비유                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 맛집(프로젝트)을 운영하며 손님들에게 요리(기능)를 대접함.                 │
│                                                                             │
│  1. 프로덕트 백로그 (전체 메뉴판):                                            │
│     손님이 원할 수 있는 모든 요리가 적혀 있음. 유행에 따라 새로운 요리를          │
│     넣기도 하고, 맛없는 요리는 빼기도 함. (순서가 중요!)                         │
│                                                                             │
│  2. 스프린트 백로그 (오늘의 주문서):                                          │
│     전체 메뉴 중 "오늘 1시간(스프린트) 동안 이 테이블에 나갈 요리"들만 골라냄.     │
│     요리사들은 이 주문서에 적힌 것만 집중해서 최고로 맛있게 만듦.                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 백로그 유형 상세 비교

| 항목 | 프로덕트 백로그 (Product Backlog) | 스프린트 백로그 (Sprint Backlog) |
|:---:|:---|:---|
| **범위** | 제품 전체 (전체 라이프사이클) | 개별 스프린트 (1~4주 분량) |
| **관리 주체** | 제품 책임자 (Product Owner) | 개발 팀 (Development Team) |
| **구성 요소** | 에픽(Epic), 사용자 스토리, 결함, 기술 부채 | 구체적인 태스크(Task), 시간 단위 작업 |
| **순서 조정** | 비즈니스 가치에 따라 수시로 변경 가능 | 스프린트 진행 중에는 원칙적으로 변경 불가 |
| **목표** | 제품의 비전 달성 (Long-term) | 스프린트 목표 달성 (Short-term) |

### 2. 백로그 관리 (Refinement / Grooming)
백로그는 수시로 정제되어야 합니다.
- **DEEP 원칙**: **D**etailed appropriately(적절히 상세한), **E**stimated(추정된), **E**mergent(진화하는), **P**rioritized(우선순위가 정해진).
- **INVEST 원칙**: 사용자 스토리 작성 기준 (**I**ndependent, **N**egotiable, **V**aluable, **E**stimatable, **S**mall, **T**estable).

### 3. 백로그 흐름도 (ASCII)

```text
    [ Product Vision ]
           │
    ┌──────▼────────────────────────┐
    │      Product Backlog          │ (PO 관리)
    │  1. 핵심 결제 기능 (High)      │
    │  2. 회원 가입 UI   (Mid)       │ <── Grooming (정제)
    │  3. 통계 리포트    (Low)       │
    └───────────┬───────────────────┘
                │
         [ Sprint Planning ] ──▶ 이번 스프린트 목표 수립
                │
    ┌───────────▼───────────────────┐
    │      Sprint Backlog           │ (Team 관리)
    │  - 결제 API 설계 (2h)          │
    │  - PG사 연동 모듈 개발 (8h)     │ <── Daily Update
    │  - 결제 결과 DB 저장 (4h)      │
    └───────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 우선순위 결정 기법
PO는 프로덕트 백로그의 순서를 정할 때 다음 기법들을 활용합니다.
- **MoSCoW**: **M**ust have, **S**hould have, **C**ould have, **W**on't have.
- **Kano 모델**: 만족도와 기능 수행도를 기준으로 매력적/당연적/일원적 요소를 구분.

### 2. 기술적 시너지: 이슈 트래커 (Jira, GitHub Issues)
현대적인 팀은 백로그를 도구로 관리합니다. 프로덕트 백로그의 사용자 스토리가 스프린트 백로그의 'Issue'나 'Ticket'으로 변환되며, 이 과정에서 코드 커밋(Git)과 연결되어 작업의 추적성을 확보합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 스프린트 도중 추가 요구사항 발생
- **상황**: 스프린트가 시작된 지 3일째, PO가 "이거 아주 간단한 건데 지금 넣어주면 안 돼?"라고 요청함.
- **결단**: **스크럼 가이드 준수**.
    1. 스프린트 백로그는 개발팀의 약속이므로 원칙적으로 중간 삽입은 거절.
    2. 해당 요청을 프로덕트 백로그 상단에 위치시키고, 다음 스프린트 계획 회의에서 다루기로 합의.
- **판단**: 팀의 집중력을 흐트러뜨리는 'Context Switching' 비용을 방지하여 현재 약속한 기능의 품질 보장.

### 📢 기술사적 결언
> "백로그는 단순히 할 일의 목록이 아니라 **'팀의 의사소통 창구'**다. 프로덕트 백로그가 PO의 시장 통찰력을 보여준다면, 스프린트 백로그는 팀의 실행 의지를 보여준다. 특히 백로그 정제(Grooming)가 제대로 되지 않으면 스프린트 계획 회의가 무의미한 토론장으로 변질되므로, 지속적인 정제 활동이 애자일 성공의 80%를 결정한다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **가치 전달 효율**: 상위 20%의 핵심 기능(Backlog High items) 개발로 비즈니스 가치의 80% 조기 달성.
- **예측 가능성**: 스프린트 백로그 완수율을 통해 팀의 속도(Velocity)를 파악하고 향후 일정 정교화.

### 미래 전망
미래의 백로그 관리는 **'AI 기반 자동 우선순위 추천'** 기술과 결합할 것입니다. 사용자 피드백 데이터와 로그 분석 결과를 AI가 학습하여, 현재 제품에서 가장 개선이 시급한 기능이 무엇인지 백로그 순서를 제안해 주는 'Smart Backlog Management' 시스템이 도입될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **프로덕트 백로그**는 "내가 나중에 어른이 되면 하고 싶은 일"들을 다 적어놓은 **꿈의 리스트**예요.
2. **스프린트 백로그**는 그 꿈 중에서 "이번 주말에 당장 해볼 일"만 골라서 적은 **이번 주의 계획표**죠.
3. 리스트를 잘 만들어두면, 뭘 해야 할지 몰라 멍하니 있지 않고 **매일매일 꿈을 향해** 한 걸음씩 나갈 수 있답니다!
