# 769. 구조적 분석 도구 데이터 사전(DD) 표기법

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DFD (Data Flow Diagram)에서 시각화할 수 없는 **데이터의 미시적 구조(Microscopic Structure)**를 정의하는 메타데이터 저장소로, 시스템의 '어휘(Vocabulary)'를 표준화하는 핵심 명세서이다.
> 2. **가치**: 개발자와 분석가 간의 **데이터 정의 모호성(Data Ambiguity)**을 해소하여, 설계 및 구현 단계에서의 불일치를 원천적으로 차단하는 단일 진실 공급원(SSoT) 역할을 수행한다.
> 3. **융합**: 현대의 **DBMS (Database Management System)** 스키마 정의 및 데이터 거버넌스(Data Governance)의 시초가 되는 개념으로, 정형 데이터 분석의 기초를 다진다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 철학
**데이터 사전 (Data Dictionary, DD)** 또는 **데이터 정의서 (Data Definition)**는 구조적 분석 방법론(Structured Analysis)에서 시스템의 논리적 설계를 완성하기 위해 작성하는 산출물이다. DFD가 시스템의 기능(Function)과 데이터의 흐름(Flow)을 '동적(Dynamic)'인 그림으로 표현한다면, 데이터 사전은 그 흐름 속에 존재하는 데이터 요소들의 '정적(Static)'인 구조, 타입, 길이, 값의 범위 등을 문자와 기호로 엄격하게 정의한다. 이는 단순한 참고 문서가 아니라, 코딩 단계에서 변수를 선언하고 DB 스키마를 설계하는 절대적인 기준이 된다.

#### 2. 등장 배경 및 필요성
1970년대 소프트웨어 위기(Software Crisis) 당시, 소프트웨어 복잡도가 급증하면서 "데이터의 구조가 변경되었을 때 이를 반영하지 못해 시스템이 오동작하거나, 데이터 항목에 대해 개발자마다 제각각 다른 해석을 하는 문제"가 대두되었다. 이를 해결하기 위해 **Tom DeMarco**와 **Chris Gane** 등의 학자들이 DFD를 보완하는 형식으로 데이터 사전을 도입하였다. 이를 통해 데이터의 중앙집중식 관리와 무결성(Integrity) 확보가 가능해졌다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                      [DFD] VS [Data Dictionary] 관계도                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [DFD: Data Flow Diagram]           [DD: Data Dictionary]                   │
│  ────────────────────               ─────────────────────                   │
│  "무엇을(What) 처리하는가?"          "데이터가(What) 무엇인가?"               │
│  "어디서 어디로 흐르는가?"           "어떻게 구성되었는가?"                    │
│                                                                             │
│       ┌──────┐        △                                                         │
│       │고객  │───────▶│ 주문정보 ───────────────────────────────────▶│  │
│       └──────┘        △   (Data Flow Name)                                  │
│                      │      │                                               │
│                      │      └──▶ "주문정보"란 무엇인가?                       │
│                      │             ▼                                         │
│                      │    ┌──────────────────────────────────────────┐    │
│                      │    │  주문정보 = 주문번호 + 고객ID + {상품목록}     │    │
│                      │    │            + 배송지 + (결제일)               │    │
│                      │    └──────────────────────────────────────────┘    │
│                      ▼                                                      │
│               [Macroscopic View]         [Microscopic View]                  │
│               (거시적 관점)               (미시적 관점)                      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
*(도해 설명: DFD는 시스템의 '동맥과 정맥'을 보여주는 지도라면, 데이터 사전은 그 지도 속에 흐르는 '혈액의 성분(세포, 단백질 등)'을 분석한 실험 보고서와 같다.)*

#### 3. 💡 비유: 건축 설계도와 시방서 (Specification)
건축가가 집을 지을 때 **평면도(DFD)**를 그리면 "거실에서 주방으로 이동한다"는 흐름을 알 수 있다. 하지만 벽체를 실제로 시공하기 위해서는 벽돌의 종류, 시멘트의 배합 비율, 단열재의 두께를 적은 **시방서(DD)**가 반드시 필요하다. 시방서가 없으면 건물은 무너지거나 기능에 문제가 생기듯, 데이터 사전이 없는 소프트웨어는 데이터 일관성을 유지할 수 없다.

> **📢 섹션 요약 비유**: 데이터 사전은 복잡한 레고(Lego) 조립 설명서 옆에 있는 **'부품 목록 및 분류표'**와 같습니다. 조립 순서(DFD)만 보고는 이 기어와 핀이 정확히 어떤 역할을 하는지 알 수 없지만, 목록(DD)을 보면 모든 부품의 용도와 규격이 명확해집니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 데이터 사전의 4대 구성 요소
데이터 사전은 DFD에 등장하는 모든 데이터 항목을 정의하며, 크게 네 가지 범주로 분류된다.

| 요소 명칭 | 역할 | 내부 동작 | DFD 표현 |
|:---|:---|:---|:---|
| **데이터 원소**<br>(Data Element) | 데이터의 최소 단위 | 더 이상 쪼개지지 않는 원자적 값(Atomic Value) | 화살표 위의 레이블 |
| **데이터 구조**<br>(Data Structure) | 원소들의 논리적 집합 | 원소를 조합하여 복합적인 데이터 레코드 형성 | 복합적인 흐름 |
| **데이터 흐름**<br>(Data Flow) | 프로세스 간의 이동 데이터 | DFD의 화살표가 나르는 데이터 패킷 정의 | DFD의 화살표 |
| **데이터 저장소**<br>(Data Store) | 데이터의 정착 장소 | 파일(File)이나 데이터베이스(DB)에 저장된 구조 정의 | DFD의 양쪽 막대 |

#### 2. 표준 표기법 (Notation) 및 문법 (Syntax)
DD는 자연어 대신 수학적 기호를 사용하여 데이터 구조를 정의한다. 이를 통해 정확성과 간결성을 확보한다.

| 기호 | 명칭 | 의미 | 문법 예시 | 설명 |
|:---:|:---|:---|:---|:---|
| **=** | Is Defined As | **정의** | `A = B + C` | A는 B와 C로 구성됨 |
| **+** | And | **연결(Sequence)** | `Name = Last + First` | 순차적으로 결합 (합집합) |
| **{ }** | Iteration | **반복** | `List = {Item}` | Item이 0번 이상 반복됨 (List/Array 구조) |
| **[ \| ]** | Selection | **선택** | `Sex = [M \| F]` | OR 관계, 선택지 중 하나만 선택 |
| **( )** | Optional | **생략** | `Phone = (Num)` | 해당 항목이 있을 수도, 없을 수도 있음 |
| **\* \*** | Comment | **주석** | `*주민번호*` | 데이터에 대한 부연 설명 |

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    [Complex Data Structure Example]                         │
│                    실무 데이터 구조 파싱 및 정의 예시                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 단순 정의 (Simple Definition)                                           │
│  ─────────────────────────────────────────────────────────────────          │
│     성명 = 성 + 이름                                                         │
│                                                                             │
│  2. 복잡한 반복 및 선택 구조 (Complex E-commerce Order)                     │
│  ─────────────────────────────────────────────────────────────────          │
│                                                                             │
│     주문서 = 주문헤더 + {주문상품내역} + (결제정보)                           │
│                                                                             │
│     주문헤더   = 주문번호 + 주문일자 + 고객ID + *배송지주소*                  │
│     주문상품내역 = 상품코드 + 수량 + 단가 + [개별옵션A | 개별옵션B]           │
│     결제정보   = 카드사명 + 카드번호 + 할부개월수                             │
│                                                                             │
│     [해석]:                                                                  │
│     1) 주문서는 반드시 헤더가 있어야 하고(+),                                 │
│     2) 상품 내역은 1개 이상 반복되며({ }),                                   │
│     3) 결제정보는 카드 결제 시에만 존재하므로 생략 가능하다(( )).             │
│     4) 개별옵션은 A 혹은 B 둘 중 하나만 선택한다([ | ]).                     │
│                                                                             │
│  3. 계층적 구조 확장 (Hierarchical Decomposition)                            │
│  ─────────────────────────────────────────────────────────────────          │
│     시스템로그 = {로그레코드}                                                │
│     로그레코드 = 타임스탬프 + 사용자ID + [이벤트로그 | 에러로그]             │
│     이벤트로그 = 액션명 + (파라미터)                                         │
│     에러로그   = 에러코드 + 스택트레이스                                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
*(도해 설명: 위 예시는 실제 전자상거래 시스템의 데이터 구조를 모델링한 것이다. 데이터 사전 표기법을 통해 개발자는 복잡한 중첩 구조(Nested Structure)와 Null 값 허용 여부를 코드를 보지 않고도 직관적으로 이해할 수 있다.)*

#### 3. 실무 코드 구현 예시 (Pseudo-code)
데이터 사전에 정의된 구조는 곧바로 코드로 구현된다.

```python
# [Pseudo-code] Data Dictionary Implementation

# Definition: Order = Header + {Item} + (Payment)
class Order:
    def __init__(self):
        self.header = Header()          # Mandatory Sequence
        self.items = []                 # Iteration (List)
        self.payment = None             # Optional (Nullable)

    def add_item(self, item_code, qty, option_type):
        # Definition: Item = Code + Qty + [OptA | OptB]
        new_item = {
            "code": item_code,
            "qty": qty,
            "option": option_type       # Selection logic applied
        }
        self.items.append(new_item)

# Data Dictionary acts as a Schema Contract
# DB Schema(Column 'items' is NOT NULL if array is initialized)
```

> **📢 섹션 요약 비유**: 데이터 사전의 표기법은 **'DNA 염기서열 표기'**와 같습니다. A, C, G, T 네 가지 문자(기호)만으로 생명체라는 거대한 구조를 설명하듯, `=`, `+`, `{}`, `[]` 기호만으로 복잡한 비즈니스 데이터의 모든 구조를 수학적으로 완벽하게 정의해낼 수 있습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 심층 기술 비교: DD vs ERD (Entity-Relationship Diagram)

| 구분 | **데이터 사전 (DD)** | **개체-관계 다이어그램 (ERD)** |
|:---|:---|:---|
| **관점** | **데이터 중심 (Data-Centric)** | **정보 중심 (Info-Centric)** |
| **상세도** | 매우 상세함 (Element Level) | 개념적/논리적 (Schema Level) |
| **표현 대상** | 스칼라 값, 구조체의 내부 필드 | 테이블(Table), 관계(Relationship) |
| **주 사용자** | 개발자, 시스템 설계자 | DBA, 데이터 아키텍트 |
| **시기** | 구조적 분석 (요구분석/설계 초반) | DB 설계 (논리적/물리적 설계) |
| **예시** | `주소 = 번지 + (Building)` | `Customer 1──N Order` |

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                 [Mapping] DD to Database Schema                             │
│                 데이터 사전과 스키마의 상관관계                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [DD Definition]                                                           │
│  ─────────────────                                                      │
│  고객 = 고객ID + 성명 + {연락처}                                             │
│  연락처 = [유선전화 | 휴대전화]                                             │
│                                                                             │
│           ▼ (Mapping Logic)                                                │
│                                                                             │
│  [DB Schema (DDL)]                                                         │
│  ────────────────────                                                      │
│  CREATE TABLE Customer (                                                    │
│      Customer_ID   VARCHAR(20) PRIMARY KEY,   -- (=) Definition             │
│      Customer_Name VARCHAR(50) NOT NULL,       -- (+) Sequence              │
│      Contact_Type  CHAR(1),                    -- [|] Selection (Type)      │
│      Contact_Num   VARCHAR(15)                 -- (+) Sequence              │
│  );                                                                         │
│  -- { } Iteration은 별도 Table Contact_List로 1:N 정규화 반영                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
*(도해 설명: 데이터 사전의 '반복' 기호는 정규화 과정에서 1:N 관계의 테이블로 변환되며, '선택' 기호는 ENUM 타입이나 타입 구분 컬럼으로 변환된다. 즉, DD는 물리적 스키마의 논리적 청사진이다.)*

#### 2. 타 과목 융합: 소프트웨어 공학 및 데이터베이스
- **[SW Engineering]** 데이터 사전은 요구사항 분석(Requirements Analysis) 단계에서 작성되며, 나중에 **코드 생성기(Code Generator)**의 입력값이 되어 소스코드의 `struct`, `class`, `interface`를 자동으로 생성할 수 있다.
- **[Database]** DD에 정의된 **도메인 제약 조건(Domain Constraint)**은 데이터베이스의 `CHECK` 제약 조건이나 `TRIGGER`로 변환되어 데이터 무결성을 강제한다.

> **📢 섹션 요약 비유**: 데이터 사전(DD)은 **'설계도면(ERD)'**에 있는 건물의 재질과 강도를 계산한 **'구조 계산서'**와 같습니다. 건물이 예쁘게 보이는 것(ERD)도 중요하지만, 바람이 불어도 무너지지 않게 각 부재의 강도를 계산해두는 것(DD)이 더 중요하
