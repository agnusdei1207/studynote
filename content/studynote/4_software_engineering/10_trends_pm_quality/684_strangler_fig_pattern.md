# 684. 스트랭글러 패턴 레거시 분할

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 거대한 레거시 시스템을 한꺼번에 바꾸는 대신, 특정 기능을 새로운 서비스(마이크로서비스 등)로 하나씩 구현하고 점진적으로 대체하여 최종적으로 기존 시스템을 제거하는 **점진적 현대화 전략**이다.
> 2. **메커니즘**: 클라이언트와 레거시 사이에 **파사드(Facade) 또는 프록시**를 두어, 이관이 완료된 기능은 새 서비스로, 아직 남은 기능은 레거시로 라우팅하며 위험을 분산한다.
> 3. **가치**: 대규모 '빅뱅(Big-bang)' 방식의 전환 실패 리스크를 최소화하고, 서비스 중단 없이 비즈니스 연속성을 유지하면서 아키텍처를 안전하게 진화시킨다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "빅뱅 방식의 위험성"

오래된 시스템을 한꺼번에 갈아엎는 '빅뱅 마이그레이션'은 성공 확률이 매우 낮습니다. 개발 기간이 길어질수록 요구사항은 변하고, 오픈 당일 예상치 못한 연쇄 장애가 발생할 확률이 높기 때문입니다. **스트랭글러 피그 (Strangler Fig)** 패턴은 나무를 휘감아 자라며 결국 원래의 나무를 대체하는 '교살자 무화과' 식물의 생태에서 착안한 안전한 전환 모델입니다.

### 💡 비유: 낡은 아파트 단지의 재건축

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        스트랭글러 패턴 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 빅뱅 방식:                                                               │
│     단지 전체 사람들을 다 내보내고(서비스 중단), 아파트를 다 허문 뒤(고위험)      │
│     한꺼번에 새로 지음. 공사가 늦어지면 주민들은 갈 곳이 없음.                   │
│                                                                             │
│  2. 스트랭글러 방식:                                                         │
│     - 1동을 먼저 허물고 멋진 새 건물을 지음. 주민 일부가 이사함.                 │
│     - 그 옆에 2동을 짓고 또 이사함.                                            │
│     - 마지막 동까지 다 지으면 예전의 낡은 단지(레거시)는 자연스럽게 사라짐!       │
│                                                                             │
│  → 사는 데 불편함 없이(서비스 유지), 하나씩 새것으로 바꿔나가는 지혜!             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 스트랭글러 패턴의 3단계 프로세스
1. **Transform (변환)**: 레거시의 특정 기능을 최신 기술 스택으로 별도 개발.
2. **Coexist (공존)**: 기존 시스템과 새 시스템을 동시에 운영하며 프록시를 통해 요청 분기.
3. **Eliminate (제거)**: 레거시의 모든 기능이 새 시스템으로 옮겨지면 기존 시스템을 안전하게 셧다운.

### 2. 핵심 컴포넌트: 스트랭글러 파사드 (Facade)
클라이언트의 요청을 가로채서 어디로 보낼지 결정하는 관문입니다.
- **기능 A 요청**: (이관 완료) -> **New Service**로 라우팅.
- **기능 B 요청**: (이관 미완료) -> **Legacy System**으로 라우팅.

### 3. 전환 아키텍처 변화 (ASCII)

```text
    [ Phase 1: Start ]        [ Phase 2: Coexist ]        [ Phase 3: Final ]
    
      Client                    Client                      Client
        │                         │                           │
        ▼                         ▼                           ▼
    ┌────────┐              ┌───────────┐               ┌──────────┐
    │ Legacy │              │  Facade   │               │ New App  │
    │ System │              └─┬───────┬─┘               │ (MSA)    │
    └────────┘                │       │                 └──────────┘
                              ▼       ▼
                          ┌────────┐┌────────┐
                          │ Legacy ││ New SVC│
                          └────────┘└────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 빅뱅(Big-bang) vs 스트랭글러(Strangler)

| 항목 | 빅뱅 전환 | 스트랭글러 전환 |
|:---:|:---|:---|
| **리스크** | 매우 높음 (실패 시 전사 마비) | **낮음 (기능별 리스크 분산)** |
| **비용** | 일시적 대규모 투입 | 점진적/지속적 투입 |
| **가치 전달** | 프로젝트 종료 후 한꺼번에 | **개발 즉시 부분적 가치 창출** |
| **복구(Rollback)** | 어려움 (전체 원복 필요) | 용이 (해당 기능만 원복) |

### 2. 기술적 시너지: 데이터 동기화 (CDC)
레거시와 새 서비스가 공존하는 동안 데이터 불일치가 발생할 수 있습니다. 이를 위해 **CDC (Change Data Capture)** 기술을 활용하여 레거시 DB의 변경사항을 실시간으로 새 서비스의 DB로 복제하는 전략이 필수적으로 병합됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 메인프레임 기반 커머스의 클라우드 전환
- **상황**: 30년 된 메인프레임을 한 번에 끄는 것은 자살 행위와 같음.
- **결단**: **스트랭글러 패턴 적용**. 
- **판단**: API 게이트웨이를 파사드로 설정. 부하가 가장 많고 독립적인 '상품 조회' 기능부터 마이크로서비스로 개발하여 트래픽 전환. 이후 '장바구니', '주문', '결제' 순으로 하나씩 쪼개 나감.
- **효과**: 2년에 걸친 대장정 동안 단 한 번의 전체 장애 없이 성공적으로 클라우드 MSA 전환 완료.

### 📢 기술사적 결언
> "스트랭글러 패턴은 기술적 기법을 넘어선 **'심리적 안전장치'**다. 이해관계자들은 눈에 보이지 않는 전체 전환보다, 당장 다음 달에 하나씩 완성되어 나오는 작은 성공(Small Wins)에 더 큰 신뢰를 보낸다. 아키텍트는 전환 대상을 고를 때, **'영향도가 작으면서 가시적 효과가 큰'** 기능부터 골라내는 비즈니스적 안목이 필요하다. 레거시는 부수는 대상이 아니라, 새 생명을 위해 영양분을 공급하고 조용히 퇴장하는 '부모 나무'와 같다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **가용성**: 전환 기간 중 서비스 중단 시간 0시간 달성 가능.
- **품질**: 부분적 배포와 테스트를 통해 전체 시스템의 결함 밀도 30% 이상 감소.

### 미래 전망
미래의 스트랭글러 패턴은 **'자율형 코드 마이그레이션'**과 결합할 것입니다. AI가 레거시 코드의 의존성을 자동으로 분석하여 최적의 분할 지점을 추천하고, 파사드의 라우팅 룰을 실시간 트래픽 상황에 맞춰 자동으로 조절하는 'AI-Driven Modernization'이 상용화될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **스트랭글러 패턴**은 낡은 신발(레거시)을 버리고 새 신발(신규)을 신을 때, 한꺼번에 바꾸지 않고 **끈부터 하나씩 새 걸로** 갈아 끼우는 거예요.
2. 새 끈으로 바꿨는데 불편하면 다시 옛날 끈으로 금방 바꿀 수 있어서 **넘어질 걱정이 없죠.**
3. 그렇게 끈, 밑창, 가죽을 하나씩 다 바꾸다 보면 어느새 **반짝반짝한 새 신발**이 되어 있을 거랍니다!
