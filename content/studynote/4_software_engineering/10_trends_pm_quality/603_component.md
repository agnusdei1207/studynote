# 603. 컴포넌트(Component) 및 독립 배포 단위

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컴포넌트(Component)는 **"독립적으로 교체 가능하고, 배포할 수 있는 소프트웨어 단위"**로, 잘 정의된 인터페이스(Interface)를 통해 다른 컴포넌트와 통신하는 재사용 가능한 모듈이다.
> 2. **가치**: 시스템을 컴포넌트로 분리함으로써 **병렬 개발(Parallel Development)**이 가능해지고, 특정 컴포넌트의 수정이 전체 시스템 재배포를 강요하지 않아 **배포 리드 타임(Deployment Lead Time)**을 획기적으로 단축한다.
> 3. **융합**: MSA(Microservices Architecture)의 서비스(Service), React/Vue의 컴포넌트, OSGi 번들(Bundle), Docker 컨테이너 등 현대 소프트웨어 아키텍처의 기본 구성 단위이며, CI/CD 파이프라인과 결합하여 지속적 배포(Continuous Deployment)의 기반이 된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 컴포넌트란 무엇인가? 1968년 Doug McIlroy가 "Mass-Produced Software Components" 논문에서 소프트웨어도 하드웨어처럼 부품(Component)으로 조립 가능해야 한다고 주장했다. 오늘날 컴포넌트는 **"런타임에 독립적으로 로딩, 교체, 배포 가능한 단위"**로 정의된다. 클래스(Class)가 컴파일 타임 단위라면, 컴포넌트는 배포(Deployment) 단위의 차이가 있다. JAR(Java Archive), DLL(Dynamic Link Library), npm 패키지, Docker 이미지가 모두 컴포넌트의 실체화다.

- **💡 비유**: 컴포넌트는 **"LEGO 블록"**과 같습니다. 각 블록은 독립적으로 생산되고, 표준화된 연결부(인터페이스)를 통해 다른 블록과 조립 가능합니다. 하나의 블록을 디자인이 개선된 새 버전으로 교체해도, 이미 완성된 조립물(시스템) 전체를 다시 만들 필요가 없죠. 블록 하나만 뽑아 꽂으면 됩니다(핫 스왑, Hot Swap).

- **등장 배경**:
    1. **소프트웨어 위기(Software Crisis)**: 1960~70년대 난해한 스파게티 코드를 유지보수하는 비용이 개발 비용을 초과.
    2. **객체지향(OOP) 등장**: 클래스 단위 재사용이 가능해졌으나, 여전히 소스 코드 수준에서의 재사용에 그침.
    3. **컴포넌트 기반 개발(CBD, Component-Based Development)**: 1990년대 COM(Component Object Model), EJB(Enterprise JavaBean) 등 런타임에 바이너리를 플러그인 방식으로 탑재하는 기술이 등장.
    4. **클라우드 네이티브(Cloud Native)**: Docker 쿠버네티스(Kubernetes)가 컴포넌트 개념을 프로세스 단위로 확장.

- **📢 섹션 요약 비유**: 자동차에서 타이어, 엔진, 브레이크가 각각 별도로 생산되고 교체 가능하듯, 소프트웨어도 시스템을 "부품" 단위로 분해하여 조립, 교체, 업그레이드 가능하게 만드는 설계 철학입니다.

#### 심층 동작 원리: 컴포넌트 의존성 관리

컴포넌트 기반 아키텍처의 성공 열쇠는 **의존성 방향(Dependency Direction)**을 제어하는 데 있다. 상위 수준 컴포넌트(Business Logic)가 하위 수준 컴포넌트(Infrastructure)에 의존하면, 하위 기술 교체 시 상위 로직까지 수정해야 하는 재악(Fragility)이 발생한다. DIP(Dependency Inversion Principle)를 적용하여 상위 컴포넌트가 추상화(Interface)에 의존하도록 설계해야 한다.

```
[의존성 방향 비교]

❌ 안티패턴: 구체적 의존 (Concrete Dependency)
PaymentService → KakaoPayGateway → Kakaocorp SDK
                    ↓
              NaverPayGateway로 교체 시 PaymentService 재컴파일 필요

✅ 베스트 프랙티스: 추상화 의존 (Abstract Dependency)
PaymentService → IPaymentGateway (interface)
         △                    △
         │                    │
KakaoPayGateway    NaverPayGateway
(implements)       (implements)
```

- **📢 섹션 요약 비유**: 건물을 지을 때 내부 인테리어를 바꿔도 구조를 무너뜨리지 않으려면, 내장 가구를 건물 구조와 분리해서 모듈화해야 하듯, 소프트웨어도 "변경될 것"과 "변경되지 않을 것"을 분리하여 의존성을 제어해야 합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 모놀리식(Monolith) vs 컴포넌트 기반(Component-Based)

| 비교 항목 | 모놀리식 아키텍처 | 컴포넌트 기반 아키텍처 |
|:---|:---|:---|
| **배포 단위** | 전체 시스템을 하나로 배포 | 독립적인 컴포넌트별 배포 |
| **확장성(Scalability)** | 전체 시스템을 확장 | 필요한 컴포넌트만 확장 |
| **기술 다양성** | 단일 기술 스택 강제 | Polyglot(다기술) 가능 |
| **장애 격리(Failure Isolation)** | 한 곳 장애 시 전체 중단 | 장애가 컴포넌트로 국한 |
| **복잡도(Complexity)** | 배포 관리는 단순 | 분산 트랜잭션, 데이터 일관성 복잡 |
| **비유** | 하나의 거대한 공장 | 전문화된 부품 공장망 |

#### 2. 계층(Layer)형 vs 컴포넌트(Component)형 비교

#### 3. 과목 융합 관점

- **배포옵스(DevOps)**: 컴포넌트는 CI/CD 파이프라인의 기본 단위다. GitHub Actions, Jenkins, GitLab CI에서 각 컴포넌트별로 독립적인 파이프라인을 구성하면, 코드 커밋에서 배포까지의 리드 타임(Lead Time)을 획기적으로 단축할 수 있다. Docker 이미지는 컴포넌트의 포맷(Packaging Format)이며, 쿠버네티스의 Deployment, Service, Ingress는 컴포넌트의 런타임 배포를 자동화한다.

- **데이터베이스(DB)**: 데이터베이스는 컴포넌트의 경계를 넘어 공유되기 쉬운 리스크 영역이다. 서로 다른 컴포넌트가 동일한 테이블을 접근하면(Data Ownership 위배), 스키마 변경 시 모든 컴포넌트가 동시에 수정되어야 하는 재악이 발생한다. Database per Service 패턴으로 각 컴포넌트가 독립적인 스키마를 소유하고, API나 이벤트(Event)를 통해 데이터를 동기화해야 한다.

- **📢 섹션 요약 비유**: 스마트폰에서 앱(App)이 독립적으로 개발되고 배포되듯, 컴포넌트 기반 아키텍처는 시스템의 각 부분을 "앱"처럼 자율적으로 운영 가능하게 만드는 조직 분업(Decoupled Organization)의 기술적 표현입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오 1: 결제 모듈의 컴포넌트 분리**
    - **문제**: 결제 로직이 모놀리식 OrderService에 포함되어, 새로운 PG사(Payment Gateway) 연동 시마다 전체 시스템 재배포 강요.
    - **의사결정**: PaymentService를 독립 컴포넌트로 추출하고, 인터페이스(`IPaymentGateway`)를 통해 통신.
    - **결과**: PG사 추가 시 OrderService는 전혀 수정 불필요, 배포 시간 3시간에서 15분으로 단축.

- **실무 시나리오 2: 레거시 시스템의 Strangler Fig 패턴**
    - **문제**: 20년된 코어 뱅킹 시스템을 한 번에 MSA로 전환 불가.
    - **의사결정**: Strangler Fig 패턴으로 점진적 마이그레이션. 새로운 컴포넌트를 조립하고, 역방향 프록시(Reverse Proxy)를 통해 트래픽을 새 컴포넌트로 점진적으로 전환.
    - **결과**: 2년간 10번의 컴포넌트 분리로 무중단 전환 완료.

- **도입 체크리스트**:
    1. **인터페이스 안정성**: 컴포넌트 간 인터페이스를 얼마나 엄격하게 관리하는가? (Semantic Versioning 준수 여부)
    2. **독립 배포 가능성**: 단일 컴포넌트만 독립적으로 배포 가능한가? (의존하는 다른 컴포넌트의 재배포 강요 여부)
    3. **팀 소유권(Team Ownership)**: 컴포넌트별로 책임 있는 팀(2-Pizza Team)이 존재하는가?
    4. **데이터 독립성**: 컴포넌트별로 독립적인 데이터 스토어(Data Store)를 소유하는가?

- **안티패턴**:
    - **Distributed Monolith**: 컴포넌트로 분리했지만, 여전히 순차적(Sequential) 동기 호출에 의존하여 한 곳 장애 시 전체가 멈추는 구조.
    - **Shared Kernel**: 여러 컴포넌트가 공유 라이브러리(Shared Library)에 의존하여, 라이브러리 수정 시 모두 재배포 필요.
    - **Database per Service 위배**: 여러 서비스가 동일한 DB 스키마를 직접 접근하여 Data Coupling 발생.

- **📢 섹션 요약 비유**: 아파트 건물을 지을 때 각 호수(컴포넌트)가 독립적인 구조를 가져서, 한 집에서 리모델링을 해도 이웃집에 영향이 없어야 합니다. 만약 벽을 공유하면(Shared Database), 이웬 집의 허락 없이는 드릴 수도 없는 것이죠.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량/정성 기대효과**:

| 구분 | 모놀리식 배포 시 | 컴포넌트 배포 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 전체 배포 시간 3시간 | 단일 컴포넌트 15분 | **배포 속도 92% 단축** |
| **정량** | 배포 실패 시 롤백 시간 2시간 | 영향 받은 컴포넌트만 롤백 10분 | **복구 시간 95% 단축** |
| **정량** | 신규 기능 추가 시 전체 재테스트 | 변경 컴포넌트만 테스트 | **테스트 범위 70% 축소** |
| **정성** | 팀 간 배포 대기 시간 발생 | 독립 배포로 병렬화 | **조정 오버헤드 제거** |

- **미래 전망**:
    1. **서비스 메시(Service Mesh)**: Istio, Linkerd 같은 서비스 메시가 컴포넌트 간 통신(인증, 추적, 회로 차단)을 인프라 계층으로 이관하여, 비즈니스 로직에서 통신 관심사(Cross-cutting Concern)를 제거하고 있다.
    2. **마이크로 프론트엔드(Micro Frontends)**: 웹 프론트엔드도 컴포넌트 단위로 분리하여, 각 팀이 독립적으로 프레임워크(React, Vue, Angular)를 선택하고 빌드/배포 가능한 아키텍처가 확산되고 있다.

- **참고 표준**:
    - **ISO/IEC 19500**: CORBA(Component Object Request Broker Architecture) 표준
    - **OMG (Object Management Group)**: 컴포넌트 모델링 표준
    - **RFC 1122**: 인터넷 프로토콜 계층 설계 원칙

- **📢 섹션 요약 비유**: 자동차가 모듈화되어 부품별로 교체 가능해진 것처럼, 소프트웨어도 "플러그 앤 플레이(Plug and Play)" 수준의 컴포넌트화가 진화되고 있습니다. 미래에는 아이폰에서 앱을 다운로드하듯, AI 모델, 데이터 파이프라인, UI 컴포넌트를 런타임에 조립/교체하는 시대가 올 것입니다.

3. 그래서 많은 사람들이 동시에 각자 맡은 부품을 만들 수 있어서, 더 크고 복잡한 것도 빨리 만들 수 있답니다!
