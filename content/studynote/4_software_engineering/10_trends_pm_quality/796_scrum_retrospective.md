# 796. 스크럼 스프린트 회고(Retrospective) 개선 액션 도출

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 단순한 회고를 넘어, 팀의 **프로세스, 도구, 상호작용**을 검토하여 지속적 개선 (**Kaizen, 지속적 개선**)을 도모하는 **Scrum (스크럼)** 프레임워크의 핵심 엔진이자 자가 치유 메커니즘이다.
> 2. **가치**: "일하는 방식"을 개선함으로써 기술적 부채를 줄이고, 팀원들의 **Psychological Safety (심리적 안전감)**을 확보하여 장기적으로 **Velocity (속도, 스프린트 당 처리 가능한 업무량)**와 품질을 비약적으로 향상시킨다.
> 3. **융합**: **PDCA (Plan-Do-Check-Act)** 사이클을 Check 단계에서 구현하며, **Action Item (실행 항목)**은 다음 스프린트의 **Product Backlog (제품 백로그)** 최상단으로 연결되어 실질적인 프로세스 리팩토링을 수행한다.

---

### Ⅰ. 개요 (Context & Background)

**스프린트 회고(Sprint Retrospective)**는 스크럼 팀이 스프린트가 종료된 후, 다음 스프린트를 계획하기 전에 수행하는 필수 회의입니다. 이는 단순한 평가 회의가 아닌, 팀의 효율성과 품질을 높이기 위해 '어떻게 일했는가(How)'를 점검하고 개선안을 도출하는 **Inspect and Adapt (검토와 적응)**의 구체적인 행위입니다.

**💡 비유: F1 레이싱 팀의 피트 스탑 전략 회의**
F1 그랑프리에서 레이싱 카가 피트로 들어오면, 정비사들은 단순히 기름만 채우는 것이 아닙니다. 타이어 마모 상태, 서스펜션 반응, 드라이버의 피드백을 종합하여 "다음 랩을 위해 세팅을 어떻게 미세 조정할 것인가?"를 논의합니다. 결과(점수)는 이미 정해져 있지만, 다음 레이스(스프린트)의 승리를 위해 프로세스를 최적화하는 과정이 바로 회고입니다.

**등장 배경**
1.  **기존 한계**: 전통적인 **Waterfall (폭포수 모델)**에서는 프로젝트 종료 후에야 회고가 이루어지므로, 해당 프로젝트에서 발생한 문제를 해결할 기회가 사라집니다.
2.  **혁신적 패러다임**: 애자일(Agile)은 **Time-boxed (시간 제한된)** 반복 주기 내에서 즉각적인 피드백과 수정을 가능하게 하여, 실패 비용을 최소화합니다.
3.  **비즈니스 요구**: 변화하는 시장 요구에 대응하기 위해 팀 스스로 학습하고 진화하는 자율적인 조직 문화가 필요해졌습니다.

> **📢 섹션 요약 비유**: "경기가 끝난 뒤 선수들이 모여 앉아 경기 영상을 돌려보며, 다음 경기를 위해 '수비 패턴을 바꿔보자'고 전술을 수정하는 팀 타임 전략 회의와 같습니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

스프린트 회고의 성공 여부는 구조화된 프로세스와 심리적 안전감, 그리고 실행 가능한 액션 아이템 도출에 달려 있습니다. 이를 위해 **Esther Derby와 Diana Larsen**이 제안한 5단계 모델이 표준으로 사용됩니다.

#### 1. 회고의 5단계 아키텍처 (5 Stages of Retrospective)
회고는 단순히 문제점을 나열하는 것에서 끝나지 않고, 데이터 수집부터 실행 계획 수립까지 이어지는 순환 구조를 가집니다.

```text
+-----------------------------------------------------------------------+
|           Sprint Retrospective Architecture (5 Stages)                |
+-----------------------------------------------------------------------+
|                                                                       |
|  1. Set the Stage (기본 설정)                                         |
|     ├── Goal: Psychological Safety 확보                               |
|     └── Activity: Check-in (예: ESVP, 1-2-4-All)                      |
|           ↓                                                           |
|  2. Gather Data (데이터 수집)                                         |
|     ├── Goal: 객관적 사실과 주관적 감정 공유                          |
|     └── Activity: Timeline, Mad/Sad/Glad                              |
|           ↓                                                           |
|  3. Generate Insights (통찰 생성) [핵심]                              |
|     ├── Goal: 원인 분석 (Root Cause Analysis)                         |
|     └── Activity: 5 Whys, Fishbone, Brainstorming                     |
|           ↓                                                           |
|  4. Decide What to Do (실행 계획 수립)                                 |
|     ├── Goal: Action Item 도출 (SMART)                                |
|     └── Activity: Dot Voting, ROI Ranking                             |
|           ↓                                                           |
|  5. Close (마무리)                                                     |
|     ├── Goal: 다음 스프린트에 대한 기대감 형성                         |
|     └── Activity: Appreciation, Retrospective of Retrospective        |
|                                                                       |
+-----------------------------------------------------------------------+
```

#### 2. 핵심 기법: KPT (Keep, Problem, Try)
팀이 객관적으로 현황을 파악하고 개선안을 도출하기 위해 가장 널리 사용되는 프레임워크입니다.

| 구성 요소 | 정의 | 내부 동작 및 예시 |
|:---:|:---|:---|
| **Keep** | 잘하고 있어서 **유지**해야 할 것 | "데일리 스큠(Scrum)을 15분 내에 끝내서 호흡이 좋았음" |
| **Problem** | 문제가 되어 **개선**이 필요한 것 | "코드 리뷰(Code Review)가 스프린트 후반에 몰려 병목 발생" |
| **Try** | 다음 스프린트에 **시도**해 볼 것 | "코드 리뷰를 Pull Request 생성 시 4시간 내에 완료하는 규칙 시도" |

#### 3. 액션 아이템 도출 알고리즘 (SMART & Experiment)
회고를 통해 도출된 액션 아이템은 **SMART (Specific, Measurable, Achievable, Relevant, Time-bound)** 원칙을 준수해야 하며, 실험적 가설로서 접근해야 합니다.

**의사결정 매트릭스 예시**
```text
+----------------+---------------------------+------------------+----------------+
| 발생 문제      | 제안된 액션 아이템        | 구현 난이도      | 기대 효과(ROI) |
+----------------+---------------------------+------------------+----------------+
| 배포 실패 빈도 | CI 파이프라인 강화        | High (Hard)      | Very High      |
| 높음           |                           |                  |                |
+----------------+---------------------------+------------------+----------------+
| 회의 시간 초과  | 타이머(Timer) 사용 강제   | Low (Easy)       | Medium         |
+----------------+---------------------------+------------------+----------------+
| -> 선택 (Quick Win: 빠르게 실현 가능하고 효과가 확실한 것 우선)           |
```

**[코드 스니펫] 회고 액션 아이템 템플릿 (Markdown/JSON 구조)**
회고 결과는 문서화되어 추적 가능해야 합니다.

```markdown
## Action Item #796-01
**Title**: 파이썬 Linter(Linting) 도입 및 자동화
**Type**: Process Improvement
**Owner**: @Alice (QA Lead)
**Due Date**: Sprint 24 (End of)
**Acceptance Criteria**:
1. `flake8` CI 파이프라인에 통합 완료
2. 전체 팀원 로컬 환경에 Pre-commit Hook 설치
3. 새로운 커밋의 Linter 에러 0건 유지
**Hypothesis**: 커밋 시점에 코드 스타일을 교정하면 리뷰 시간이 20% 단축될 것이다.
```

> **📢 섹션 요약 비유**: "자동차 엔진을 연구실에서 분석하여, 마모된 부품을 교체하고 연료 분사 시간을 조정하는 'ECU 튜닝' 과정과 같습니다. 즉, 시스템 전체를 구성하는 요소를 디버깅하고 성능을 튜닝하는 것입니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

회고는 고립된 행위가 아니라, 다른 애자일 프로세스 및 소프트웨어 엔지니어링 분야와 깊게 연결됩니다.

#### 1. 스프린트 리뷰(Sprint Review) vs 스프린트 회고(Retrospective)

| 비교 항목 | 스프린트 리뷰 (Review) | 스프린트 회고 (Retrospective) |
|:---:|:---|:---|
| **검토 대상** | **제품 (Product)** | **프로세스 (Process)** |
| **핵심 질문** | "우리가 만든 **것(What)**이 가치 있는가?" | "우리가 일하는 **방법(How)**이 효율적인가?" |
| **참여자** | 팀 + 이해관계자(Stakeholders) | **스크럼 팀만 (팀원, PO, SM)** |
| **결과물** | 업데이트된 Product Backlog | **Process Backlog (액션 아이템)** |
| **연결성** | 다음 스프린트의 '내용' 결정 | 다음 스프린트의 '품질과 속도' 결정 |

```text
       [Sprint N Finish]
             │
      ┌──────┴──────┐
      ▼             ▼
+----------+  +--------------+      +----------------+
| Review   |  | Retrospective| ---> | Process Improve|
| (Product)|  | (Process)    |      | (Next Sprint)  |
+----------+  +--------------+      +----------------+
      │             │
      ▼             ▼
 (Refined      (Action Item)
  Backlog)     (Process Backlog)
```

#### 2. 기술적 시너지: DevOps와 DORA Metrics

회고의 품질을 높이기 위해서는 주관적인 감정보다는 데이터가 필요합니다. 이때 **DORA (DevOps Research and Assessment)** 지표가 사용됩니다.

| DORA 지표 | 회고 시의 활용 방안 | 연관 액션 아이템 예시 |
|:---:|:---|:---|
| **Deployment Frequency** | 배포가 자주 안 되는 이유 분석 | "CI 파이프라인 자동화 스크립트 개선" |
| **Lead Time for Changes** | 기획~배포까지 병목 구간 파악 | "Code Review 승인 프로세스 단순화" |
| **Change Failure Rate** | 배포 후 장애 발생 원인 규명 | "통합 테스트(Integration Test) 커버리지 80%로 상향" |
| **MTTR (Mean Time to Restore)** | 장애 복구 시간 단축 방안 논의 | "장애 대응 매뉴얼(Playbook) 위키화" |

> **📢 섹션 요약 비유**: "의사가 환자의 증상(제품)을 진찰하는 것(Review)과 달리, 병원의 응급 처리 프로토콜이나 의료진 간의 소통 체계를 개선하여 다음 환자를 더 잘 치료하기 위한 병원 경영 개선(Retrospective) 활동입니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 흔한 안티패턴(Anti-patterns) 및 해결책

| 안티패턴 (Anti-pattern) | 증상 | 기술사적 처방 (Prescription) |
|:---|:---|:---|
| **Storytelling Retro** | 회고가 잡담수다로 변질됨 | **Facilitator (SM)**의 강력한 진행과 Time-boxing 준수 |
| **Blame Game** | 특정 인물을 비난하여 분위기 침체 | "비난 없는 원칙(Prime Directive)" 선포 및 **System Thinking** 교육 |
| **Complaint List** | 불만 목록만 작성하고 해결책 없음 | **Dot Voting**을 통해 우선순위 1~2개에 집중하고 Action Item 강제 할당 |
| **Action Item Decay** | 회의 직후까지만 하다가 사라짐 | 다음 스프린트 백로그에 **Enabler (가능화자)** 형태로 등록하여 관리 |

### 2. 실무 시나리오: "무력감에 빠진 팀의 회고 부활"

**[상황]**
팀원들이 회고 시간에 입을 다물고, Scrum Master(스크럼 마스터) 혼자 떠들거나 단순히 "바빴다"는 말만 반복함.

**[Root Cause Analysis (5 Whys)]**
1. 왜 말을 안 하나? → **부정적 피드백이 두려움**
2. 왜 두려워하나? → 이전에 문제를 제기했을 때 보고를 당함
3. 왜 보고를 당했나? → 관리자가 '누구의 잘못'을 찾으려 했음 (People 문제)
4. 왜 그런 분위기가 되었나? → 심리적 안전감(Psychological Safety)이 형성되지 않음

**[Architectural Decision]**
1.  **비난 금지 구역 설정**: 회의실에 'No Blame' 문구 부착 및 원격 시 이모티콘 적극 활용.
2.  **익명 피드백 도구 도입**: Mentimeter, Google Forms 등을 통해 감정적 데이터를 수집하여 객관화.
3.  **Quick Win 전략**: 해결하기 쉬운 문제(예: 노트북 성능, 모니터 추가 배치)부터 즉시 해결하여 팀에게 '회고의 효능감'을 부여.
4.  **Action Item의 가시화**: Jira 혹은 Confluence에 회고 액션을 명시하고, 다음 회고 시 성공 여부를 체크(Close)하는 절차 강제.

> **📢 섹션 요약 비유**: "막힌 하수도를 뚫기 위해 뚜껑을 열어 안을 들여다보고, 찌꺼기를 제거하거나 파이프 라인을 교체하여 물 흐름을 원활하게 만드는 '도시 배수 시스템 정비 작업'과 같습니다."

### Ⅴ. 기대효과 및 결론 (Future & Standard)

스프린트 회고를 정착시킴으로써 기업과 팀은 다음과 같은 정량적, 정성적 이득을 얻을 수 있습니다.

### 1. 정량적 기대효과 (ROI)

| 항목 | 도입 전 (As-Is) | 도입 후 (To-Be) | 개선 효과 |
|:---|:---|:---|:---|
| **반복 결함률** | 유사 버그가 주기적 재발 | 회고를 통해 패
