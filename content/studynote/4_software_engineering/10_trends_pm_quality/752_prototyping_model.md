# 752. 프로토타입 버리기 모델 vs 진화적 모델

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구사항의 불확실성을 해결하기 위해 정식 개발 전 시스템의 핵심 기능을 시제품(Prototype)으로 만들어 사용자 피드백을 받는 **반복적 생명주기 모델**이다.
> 2. **유형 분화**: 요구사항 도출 후 버려지는 **버리기 모델(Throwaway)**은 정확한 명세 확보에 주력하고, 지속적으로 보완하여 최종 제품이 되는 **진화적 모델(Evolutionary)**은 개발 속도와 비용 효율에 주력한다.
> 3. **가치**: 개발자와 사용자 간의 시각적 의사소통 도구로 기능하며, 개발 후반부에 발생하는 치명적인 요구사항 변경 리스크를 조기에 식별하고 제거한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "말로만 들어서는 알 수 없다"

사용자는 눈에 보이는 것이 없으면 자신이 진짜 무엇을 원하는지 모르는 경우가 많습니다. "일단 만들어봐"라는 요구에 대응하기 위해, 최소한의 비용으로 껍데기나 핵심 기능만 먼저 보여주는 것이 **프로토타입 (Prototype)** 모델입니다. 이는 폭포수 모델의 경직성을 극복하기 위한 첫 번째 대안으로 등장했습니다.

### 💡 비유: 견본 주택(모델하우스)과 조립식 주택

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        프로토타이핑 유형 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 버리기 모델 (모델하우스):                                                │
│     아파트 분양 전, 가짜 벽과 가구로 꾸민 집을 보여줌. 손님들이 "구조가 좋네"     │
│     하면 그 의견만 듣고, 모델하우스는 **철거(Throwaway)**함. 진짜 아파트는        │
│     처음부터 다시 정석대로 지음.                                               │
│                                                                             │
│  2. 진화적 모델 (조립식 컨테이너 집):                                         │
│     일단 살 수 있는 작은 방 하나를 먼저 지어줌. 살면서 "화장실이 필요해" 하면     │
│     옆에 덧붙이고, "2층도 필요해" 하면 위로 올림. **계속 덧붙여서(Evolution)**    │
│     결국 커다란 집을 완성함.                                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 프로토타이핑 2대 유형 비교

| 항목 | 버리기 모델 (Throwaway) | 진화적 모델 (Evolutionary) |
|:---:|:---|:---|
| **핵심 목적** | **요구사항의 정확한 식별** 및 합의 | 빠른 제품 인도 및 지속적 개선 |
| **품질 수준** | 낮음 (동작만 구현, Mockup) | **높음 (실제 제품의 기반이 됨)** |
| **주요 장점** | 리스크 최소화, 설계 정교화 | 개발 기간 단축, 비용 절감 |
| **주요 단점** | 프로토타입 제작 비용 낭비 우려 | 누더기 코드(Spaghetti) 리스크 |
| **적용 시점** | 요구사항이 전혀 모호할 때 | 요구사항이 부분적으로 명확할 때 |

### 2. 프로토타이핑 수행 절차
1. **요구사항 정의**: 핵심 요구사항만 간략히 분석.
2. **프로토타입 설계 및 구축**: 신속하게 시제품 제작 (품질보다 속도).
3. **사용자 평가**: 고객 시연 및 피드백 수집.
4. **프로토타입 정제**: 피드백 반영하여 수정.
5. **본 개발(Throwaway) 또는 확장(Evolutionary)**.

### 3. 모델 흐름도 (ASCII)

```text
    [ 초기 요구사항 ] ──▶ [ 프로토타입 개발 ] ──▶ [ 사용자 평가 ]
                                 ▲                     │
                                 │      (피드백 반영)    │ (불만족 시)
                                 └─────────────────────┘
                                         │ (만족 시)
          ┌──────────────────────────────┴──────────────────────────────┐
          ▼ (Throwaway)                                                 ▼ (Evolutionary)
    [ 프로토타입 폐기 ]                                           [ 기능 추가 및 개선 ]
    [ 정식 분석/설계/구현 ]                                        [ 최종 제품 인도 ]
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 폭포수 모델 vs 프로토타입 모델
- **폭포수**: 선형적, 결과물을 마지막에 확인. (변경 비용 높음)
- **프로토타입**: 순환적, 결과물을 중간에 확인. (변경 비용 낮음)

### 2. 기술적 시너지: RAD (Rapid Application Development)
프로토타입 모델을 극단적으로 빠르게 수행하기 위해 GUI 빌더나 로우코드(Low-code) 도구를 사용하는 방식이 **RAD**입니다. "코딩보다 조립"을 통해 피드백 루프의 속도를 극대화합니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 신규 AI 큐레이션 서비스 기획
- **상황**: 기획자가 원하는 추천 알고리즘의 UI가 실제 유저에게 편리할지 확신이 없음.
- **결단**: **버리기 모델(Throwaway) 기반 Mockup 제작**. 
- **판단**: 실제 데이터 연동 없이 Figma나 단순 HTML로 화면만 구성하여 유저 테스트(FGI) 수행. 
- **결과**: "추천 버튼이 너무 작다"는 치명적 불편함 발견. 설계 수정 후 정식 개발 착수.
- **효과**: 정식 개발 후 UI를 뒤엎는 대공사 방지로 예산 20% 절감.

### 📢 기술사적 결언
> "프로토타입은 **'실패를 저렴하게 하는 방법'**이다. 아키텍트는 프로토타입이 '장난감'으로 남지 않도록 관리해야 한다. 특히 진화적 모델을 선택할 경우, 초기 설계의 견고함(Architecture Integrity)이 무너져 나중에 유지보수 불능 상태가 되지 않도록 **'리팩토링 주기'**를 반드시 포함해야 한다. 무엇을 남기고 무엇을 버릴 것인가를 결정하는 것이 아키텍트의 통찰력이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **사용자 만족도**: 최종 제품과 요구사항 간의 일치도 90% 이상 확보.
- **리스크 감소**: 개발 중반 이후의 요구사항 변경 건수 50% 이상 감소.

### 미래 전망
미래의 프로토타이핑은 **'No-code 기반 자동 생성'**으로 진화합니다. 자연어로 요구사항을 말하면 AI가 즉시 동작하는 프로토타입을 생성해주고, 유저의 시선(Eye-tracking)을 실시간 분석하여 UI를 자동 보정하는 'Self-optimizing Prototype' 환경이 구축될 것입니다. 또한 **디지털 트윈**과 결합하여 소프트웨어뿐만 아니라 하드웨어의 동작까지 가상으로 미리 써보는 'HIL(Hardware-in-the-loop) Prototyping'이 표준이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **프로토타입**은 멋진 레고 성을 만들기 전에, **찰흙으로 대충 모양만** 먼저 만들어보는 거예요.
2. "여기에 문이 있으면 좋겠어?"라고 친구에게 물어보고, 친구가 좋다고 하면 **진짜 레고로** 튼튼하게 다시 만드는 거죠.
3. 미리 만들어보니까, 나중에 다 만든 성을 **다시 부수고 새로 지을 필요가 없어서** 정말 좋답니다!
