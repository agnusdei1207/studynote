# 667. 요구사항 검증 추적성 매트릭스 (RTM)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구사항의 생성부터 설계, 구현, 테스트, 배포에 이르기까지 전 과정의 연결 고리를 매핑하여 누락이나 왜곡이 없는지 확인하는 **요구사항 관리 도구**이다.
> 2. **추적 방향**: 사용자 요구사항이 코드에 반영되었는지 확인하는 **순방향 추적**과, 반대로 테스트 결과나 코드가 어떤 요구사항에서 기인했는지 확인하는 **역방향 추적**을 모두 지원한다.
> 3. **가치**: 프로젝트의 범위(Scope)를 완벽하게 통제하고, 변경 발생 시 영향도 분석(Impact Analysis)을 신속하게 수행하여 소프트웨어의 무결성을 보장한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "만들기로 한 걸 진짜 다 만들었는가?"

대규모 프로젝트에서는 수천 개의 요구사항이 발생합니다. 개발이 진행될수록 "이 기능은 왜 만들었지?", "테스트 안 한 요구사항은 없나?"와 같은 의문이 생깁니다. **RTM (Requirements Traceability Matrix)**은 요구사항이라는 '씨앗'이 최종 제품이라는 '열매'가 될 때까지의 모든 경로를 추적하는 족보와 같습니다.

### 💡 비유: 택배 송장 추적 시스템

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        RTM(추적성 매트릭스) 비유                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 온라인 쇼핑몰에서 물건(요구사항)을 주문함.                               │
│                                                                             │
│  1. 주문 단계: "빨간 티셔츠 1개 주문" (요구사항 생성)                           │
│  2. 물류 센터: "A-12 선반에서 티셔츠 꺼냄" (설계/분석)                          │
│  3. 배송 단계: "택배 기사님이 트럭에 실음" (구현/코딩)                          │
│  4. 도착 확인: "고객님이 직접 받고 서명함" (테스트/검증)                        │
│                                                                             │
│  → 송장 번호(ID) 하나로 지금 내 물건이 어디 있는지, 혹시 배송 안 된 건           │
│     없는지 한눈에 알 수 있는 것처럼, 요구사항도 ID로 끝까지 추적하는 것!          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. RTM의 주요 구성 요소 (Typical Columns)

| 요구사항 ID | 요구사항 내용 | 설계서 ID | 소스 파일 명 | 테스트 케이스 ID | 테스트 결과 |
|:---:|:---|:---|:---|:---|:---|
| REQ-001 | 로그인 기능 | SD-104 | Login.java | TC-01, TC-02 | **Pass** |
| REQ-002 | 비밀번호 암호화 | SD-105 | Crypto.java | TC-03 | **Fail** (수정 필요) |

### 2. 추적성의 방향성 (ASCII)

```text
    [ 사용자 요구사항 ] ──▶ [ 시스템 요구사항 ] ──▶ [ 설계/구현 ] ──▶ [ 테스트 ]
    └─────────────────────── 순방향 추적 (Forward) ───────────────────────▶
                               (누락 여부 확인)
                               
    ◀─────────────────────── 역방향 추적 (Backward) ──────────────────────┘
                               (불필요한 기능/Gold Plating 확인)
```

### 3. 변경 영향도 분석 (Impact Analysis)
특정 요구사항이 변경되었을 때, RTM을 보면 어떤 설계서를 고쳐야 하고 어떤 테스트 케이스를 다시 수행해야 하는지 즉각적으로 파악할 수 있습니다.

---

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 검증(Verification) vs 확인(Validation) 관점의 RTM
- **검증(Verification)**: "설계와 코드가 요구사항을 **올바르게 구현**했는가?" (RTM의 매핑 확인)
- **확인(Validation)**: "최종 결과물이 **사용자의 의도**에 맞는가?" (RTM의 최종 테스트 결과 확인)

### 2. 기술적 시너지: CI/CD 및 테스트 자동화
현대적인 RTM은 엑셀이 아닌 도구(Jira, DOORS)에 의해 관리됩니다. 코드 커밋 메시지에 'REQ-ID'를 적으면 자동으로 RTM이 갱신되고, 테스트 자동화 결과가 실시간으로 매트릭스에 업데이트되어 '살아있는 추적성'을 유지합니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 프로젝트 감리 대응
- **상황**: 정부 기관 발주 프로젝트의 완료 단계에서 "요구사항이 100% 반영되었음을 증명하라"는 요구를 받음.
- **결단**: **RTM 제출**.
    1. 모든 요구사항 ID에 매핑된 소스코드 위치와 테스트 결과(Pass)를 시각화.
    2. 테스트 결과가 없는 요구사항은 즉시 식별하여 추가 테스트 수행.
- **판단**: 주관적인 주장이 아닌 객관적인 '데이터 매핑'을 통해 품질 무결성을 입증하고 감리를 통과.

### 📢 기술사적 결언
> "RTM은 프로젝트의 **'품질 지도'**이자 **'보험'**이다. 개발 중에는 변경의 혼란을 막아주는 지도가 되고, 완료 후에는 품질을 보증하는 보험이 된다. 특히 요구사항과 테스트 케이스 간의 연결이 끊어지는 '추적성 단절'은 대규모 장애의 서막이 될 수 있으므로, 프로젝트 초기부터 ID 체계를 정교하게 수립하는 것이 아키텍트의 의무다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **누락 방지**: 요구사항 미반영으로 인한 결함 발생 95% 이상 예방.
- **비용 절감**: 변경 영향도 분석 시간 50% 단축 및 불필요한 과잉 개발(Gold Plating) 제거.

### 미래 전망
미래의 RTM은 **'AI 기반 자동 추적성(Automated Traceability)'**으로 발전합니다. 자연어 처리(NLP)를 통해 요구사항 문서와 소스코드를 대조하여, 사람이 일일이 연결하지 않아도 AI가 "이 함수는 REQ-102번과 80% 일치합니다"라고 추적 고리를 제안해 주는 지능형 관리 체계가 일반화될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **RTM**은 마트에서 장을 볼 때 들고 가는 **장바구니 메모지**와 같아요.
2. "사과 3개 사기"라고 적힌 걸 보고 바구니에 사과가 담겼는지(구현), 맛은 있는지(테스트) 하나씩 **체크 표시**를 하는 거죠.
3. 다 끝난 뒤에 메모지를 보면, 깜빡하고 **안 산 물건이 없는지** 한눈에 알 수 있답니다!
