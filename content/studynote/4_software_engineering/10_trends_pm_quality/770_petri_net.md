# 770. 페트리 넷 병행/비동기 시스템 정형 명세

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 독일의 카를 아담 페트리(Carl Adam Petri) 박사가 제안한 이중 그래프(Bipartite Graph) 기반의 수학적 모델링 툴로, **분산된 시스템의 비결정성(Non-determinism)**과 **병행성(Concurrency)**을 정형화(Formalization)하여 시각화하고 검증하는 정형 명세 기법이다.
> 2. **가치**: 소프트웨어의 논리적 결함 중 가장 발견하기 어려운 **교착 상태(Deadlock)**, **기아 상태(Starvation)**, **경쟁 상태(Race Condition)**를 수학적 도달 가능성 그래프(Reachability Graph)와 불변식(Invariant)을 통해 **증명(Proof)** 가능하게 한다.
> 3. **융합**: 유한 상태 기계(FSM, Finite State Machine)의 한계를 극복하여, 멀티코어 OS 스케줄링, 비동기 메시지 큐(Message Queue), 그리고 하드웨어 회로의 비동기 핸드셰이크(Handshake) 프로토콜 설계까지 아우르는 "병행성 제어의 통합 언어"이다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 철학
**페트리 넷(Petri Net, PN)**은 상태(State)와 사건(Event)을 분리된 노드로 표현하는 이중 그래프 구조를 가진 정형 명세 언어입니다. 전통적인 순차도(Flowchart)나 상태 전이도(STD, State Transition Diagram)가 제어 흐름의 순차성을 표현하는 데 집중한다면, 페트리 넷은 시스템 내의 **'사물이 얼마나 있는가(토큰)'**와 **'어떤 조건에서 사건이 발생하는가(트랜지션)'**를 분리하여 모델링함으로써, 시간이 맞물려 돌아가는 복잡한 비동기 시스템의 구조적 특성을 드러냅니다.

#### 2. 등장 배경: 비동기식 병행 컴퓨팅의 패러다임
1962년, 카를 아담 페트리 박사의 박사학위 논문 "Kommunikation mit Automaten"에서 처음 소개되었습니다. 당대의 컴퓨터 과학은 폰 노이만(Von Neumann) 구조의 **순차적 처리(Sequential Processing)**에 집중해 있었으나, 물리적 세계(물리학, 화학)와 분산 시스템(네트워크)은 본질적으로 병행적(Concurrent)입니다. 단순히 "순서대로 실행된다"는 가정으로는 설명할 수 없는 **'동시에 발생하는 이벤트 간의 인과관계와 충돌'**을 수학적으로 엄밀하게 정의하기 위해 페트리 넷이 고안되었습니다.

#### 3. 구조적 시각화: 도로 교차로와 신호 등

페트리 넷의 동작 원리를 복잡한 도로 교차로에 비유할 수 있습니다.

```text
                     [ 2중 방향 터널 (Mutual Exclusion) 모델 ]
    
    (P1: 진입 대기 구간)                 (P2: 터널 구간 - 자원)                 (P3: 진출 구간)
    ○─────────────┐                     ○─────────────────────┐                ○─────────────┐
    │             │                     │   ● ● (토큰: 차량)   │                │             │
    └───────┬─────┘                     └───────┬─────────────┘                └───────┬─────┘
            │                                   │                                      │
            │ (입력 아크: Input Arc)            │ (입력 아크)                           │ (출력 아크: Output Arc)
            ▼                                   ▼                                      ▼
        ┌───────────────────┐             ┌───────────────────┐               ┌───────────────────┐
        │   T1: 신호등(녹색) │────────────▶│  T2: 통과 제어    │──────────────▶│  T3: 진출 허용     │
        │   (Transition)     │ (Firing)    │  (Transition)     │  (Firing)     │  (Transition)     │
        └───────────────────┘             └───────────────────┘               └───────────────────┘
    
    [해설]
    1. P1(진입 대기)에 차량(토큰)이 있고, P2(터널)에 빈 공간(수용 가능)이 있어야만 T1(진입)이 발생(Firing)합니다.
    2. P2는 자원(Resource)으로서 용량이 제한되어 있어(여기선 1대), T1과 T2가 동시에 점화되는 것을 막습니다(Mutual Exclusion).
    3. 이러한 구조는 P2라는 '공유 자원'을 둘러싼 경쟁과 제어를 시각적으로 명확히 보여줍니다.
```

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 수학적 정의와 5원소 (Mathematical Definition)
페트리 넷은 5개의 튜플 $N = (P, T, I, O, M)$으로 정의됩니다. 이는 시스템의 구조(정적)와 상태(동적)를 수학적으로 엄밀하게 정의합니다.

| 구성 요소 (Component) | 기호 | 수학적 정의 | 실무적 역할 및 비유 | 내부 동작 메커니즘 |
|:---:|:---:|:---|:---|:---|
| **P (Place)** | ○ | $P = \{p_1, p_2, ..., p_n\}$ | **조건/상태 저장소** (Buffer, Memory) | 토큰을 저장하는 노드. 용량(Capacity) 제한 가능. |
| **T (Transition)** | ■ (Rect) | $T = \{t_1, t_2, ..., t_m\}$ | **이벤트/행위** (Action, Process) | 입력/출력 함수에 의해 토큰을 소비/생산. |
| **I (Input)** | → (Arrow) | $I: T \rightarrow P^\infty$ | **전제 조건** (Pre-condition) | 트랜지션이 발생하기 위해 필요한 토큰 집합. |
| **O (Output)** | → (Arrow) | $O: T \rightarrow P^\infty$ | **결과 조건** (Post-condition) | 트랜지션 발생 후 생성되는 토큰 집합. |
| **M (Marking)** | ● (Dot) | $M: P \rightarrow \mathbb{N}$ | **시스템 상태** (State Vector) | 각 플레이스에 위치한 토큰의 분포 $M(p)$. |

#### 2. 실행 의미론 (Execution Semantics): 점화 (Firing) 규칙
페트리 넷의 동작은 '점화(Firing)'라는 불연속적인 사건 발생으로 진행됩니다.

1.  **활성화 (Enabledness)**: 트랜지션 $t$에 대해, 모든 입력 플레이스 $p \in I(t)$에 최소 1개 이상의 토큰이 존재할 때 $t$는 활성화 상태가 됩니다.
    *   조건: $\forall p \in I(t), M(p) \geq 1$
2.  **점화 (Firing)**: 활성화된 $t$가 발생하면 입력 플레이스에서 토큰을 제거하고 출력 플레이스로 토큰을 이동시킵니다.
    *   변환: $M \xrightarrow{t} M'$
    *   수식: $M'(p) = M(p) - I(t, p) + O(t, p)$

#### 3. 주요 제어 구조의 ASCII 시각화와 해설

```text
    [ A. 순차 실행 (Sequential Execution) ]
    
    P1(Start)      P2(Processing)      P3(End)
       ●             (비어있음)           (비어있음)
       │                ↑                  │
       │                │                  │
    ──┴── T1 (Start)    │             ────┴── T3 (End)
                        │
    ───────────────────┴──────── T2 (Action)
    
    ▶ 해설: T1이 발생해야 P2에 토큰이 생기고, 그래야 T2가 발생 가능. 일반적인 프로그래밍 순서와 동일.

    ------------------------------------------------------------------

    [ B. 분기와 병합 (Branching & Merging) ]
    
           T_Join
              ↑
             / \
            /   \
    P_A ○──┴───┴──○ P_B
     (선택 1)      (선택 2)
        ↑            ↑
        │            │
       T1           T2
    
    ▶ 해설: 
    - 분기(Conflict): P_X에 토큰이 있으면 T1과 T2 중 '하나만' 선택 가능 (비결정적 선택).
    - 병합(Synchronization): P_A나 P_B 어느 쪽이든 토큰이 도착하면 T_Join 활성화.

    ------------------------------------------------------------------

    [ C. 병행성 (Parallelism / Concurrency) ]
    
             P_Common
                ●
                │
            ┌───┴───┐
            │ T_Fork│
            └───┬───┘
               / \
              /   \
    (Thread 1)     (Thread 2)
         P1             P2
         │             │
         T1_A          T2_B
         │             │
         P3            P4
          \             /
           \           /
            ──┴── T_Join
                 │
               P_End
    
    ▶ 해설: T_Fork가 발화하면 두 갈래(P1, P2)로 토큰이 복제되어 분산됨.
    두 갈래는 '독립적으로' 동시에 진행되며, 마지막에 T_Join에서 합류(Synchronization)하여 
    '모두 완료되었을 때' 다음 단계로 넘어감.
```

#### 4. 핵심 성질 분석 (Properties & Reachability)
페트리 넷 모델의 상태 공간(State Space)은 **도달 가능성 트리(RTG, Reachability Tree)** 혹은 그래프로 분석됩니다.

*   **도달 가능성 (Reachability)**: 초기 상태 $M_0$에서 시작하여 특정 상태 $M_{target}$에 도달할 수 있는가? (문제 해결 가능성)
*   **바운디드니스 (Boundedness)**: 플레이스 $p$의 토큰 수가 특정 값 $k$를 넘지 않는가? (버퍼 오버플로우 방지, $k=1$일 때 Safe Property)
*   **라이브니스 (Liveness)**: 어떤 트랜지션도 무한정 기다리는 상태(Deadlock)에 빠지지 않고 언젠가는 반드시 발생 가능한가?
*   **활성화 불가 (Deadlock)**: 모든 트랜지션이 비활성화된 상태. 시스템이 멈춤.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 상태 전이도(STD) vs 유한 상태 기계(FSM) vs 페트리 넷(PN)

| 비교 항목 | **FSM / STD** | **페트리 넷 (Petri Net)** |
|:---|:---|:---|
| **상태 표현** | 전체 시스템의 상태가 하나의 노드로 표현됨 (Global State) | 상태가 분산된 플레이스(Place)들의 집합으로 표현됨 (Distributed State) |
| **토큰(Token)** | 단 하나의 활성 상태(Active State)만 존재 | 여러 토큰이 존재하여 병행 상태(Concurrent States) 표현 가능 |
| **표현 능력** | 순차적 로직에는 강력하나, 병행성/동기화 표현에 부족 | 병행성, 경쟁, 동기화, 자원 공유를 직관적이고 정확히 표현 |
| **모호성(Ambiguity)** | 복잡한 시스템에서 상태 폭발(State Explosion) 발생 | 구조적 축약(Structural Reduction) 기법으로 모델 단순화 가능 |
| **검증(Verification)** | 주로 시뮬레이션에 의존 | 수학적 분석(구조 분석, 불변식 분석)을 통한 정적 검증 가능 |

#### 2. 타 영역과의 융합 시너지 (Cross-Domain Synergy)

**A. 분산 데이터베이스 (Distributed DB) & 트랜잭션 처리**
- **상관관계**: 분산 DB에서의 **2단계 로킹(2PL, Two-Phase Locking)** 프로토콜이나 **교착 상태(Deadlock)** 검출 알고리즘을 모델링하는 데 핵심적으로 사용됩니다.
- **시너지**: 트랜잭션이 자원을 요청하고 해제하는 과정을 트랜지션과 플레이스로 모델링하여, "Wait-for Graph"가 사이클을 이루어 교착 상태가 발생하는 조건을 수학적으로 증명하고 예방책을 설계할 수 있습니다.

**B. VLSI/반도체 설계 (Asynchronous Circuit Design)**
- **상관관계**: 클럭(Clock) 신호에 의존하지 않는 비동기 회로 설계에서 **신호 핸드셰이킹(Handshaking) 프로토컬**의 검증에 필수적입니다.
- **시너지**: 회로의 지연(Wire Delay)과 경쟁(Hazard) 조건을 페트리 넷으로 모델링하여, **스테이짱(Hazard)**이 발생하지 않고 데이터가 정확히 전송되는지 시뮬레이션합니다.
