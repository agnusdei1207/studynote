# 775. 정보시스템 감리 대상 및 절차

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정보시스템 감리는 막대한 공공 예산이 투입되는 IT 사업의 **투명성, 효율성, 기술적 적정성**을 확보하기 위해 제3자 전문가가 수행하는 독립적인 **IT 거버넌스(Governance) 및 품질 통제(Quality Control) 활동**입니다.
> 2. **의무 대상 및 법규**: 한국에서는 **전자정부법(E-Government Act)** 및 같은 법 시행령에 의거하여 총 사업비 5억 원 이상이거나 대국민 서비스, 다부처 공동 활용 등 **공공성이 높은 사업**은 의무적으로 감리를 수행해야 하며, 이를 통해 국가 정보화 사업의 리스크를 선제적으로 차단합니다.
> 3. **프로세스 및 메커니즘**: 단순한 문서 검토를 넘어 **예비조사(Planning) → 현장감리(Execution/Diagnosis) → 결과보고(Reporting) → 시정조치 확인(Corrective Action)**의 생명주기를 따르며, 기술적인 구현뿐만 아니라 사업 관리(Process), 운영 안정성(Operation)까지 다각도로 진단하여 시스템의 수명 주기 가치를 극대화합니다.

---

### Ⅰ. 개요 (Context & Background)

**정보시스템 감리 (IS Audit, Information Systems Audit)**는 정보화 사업의 기획, 분석, 설계, 구현, 시험, 유지보수 전 과정에 대해 제3자의 독립된 시각에서 기술적, 관리적 적정성을 평가하는 활동입니다. 건축물 준공 검사가 구조적 안전성을 확인하듯, 감리는 소프트웨어의 **비기능적 요구사항(Non-functional Requirements)**인 성능, 보안, 안정성, 그리고 사업 관리의 투명성을 검증하는 **최후의 안전장치**입니다.

**등장 배경 및 필요성**
정보화 사업은 **"눈에 보이지 않는 무형의 자산을 생산하는 과정"**이라는 특성상, 발주자(공공기관)와 사업자(개발사) 간의 심각한 **정보 비대칭성(Information Asymmetry)**이 존재합니다. 발주자는 고도의 기술적 세부 사항을 파악하기 어렵고, 사업자는 이윤 추구를 위해 품질이나 보안을 타협할 유인이 있습니다. 과거 대형 IT 프로젝트의 실패(예: 철도 통합 시스템 오류, 모바일 주민서비스 장애 등)는 대부분 "요구사항 누락", "성능 검증 부족", "보안 약점" 등에서 기인했습니다. 이를 방지하기 위해 도입된 제도가 정보시스템 감리이며, 이는 단순한 비용 절감이 아닌 **사회적 비용(Social Cost) 최소화**와 **공공 신뢰(Public Trust)** 회복을 위한 필수적인 제도적 장치입니다.

**💡 비유: 건설 현장의 독립 감리 전문가**
마치 거대한 교량을 건설할 때, 시공사가 자체 품질 관리를 하는 것과 별개로 **발주처(국가)가 고용한 독립 감리전문회사**가 설계도 대로 시공됐는지, 부식되지 않는 자재를 사용했는지, 내진 설계가 충분한지를 **현장에서 실시간으로 감독**하는 것과 같습니다. 감리원이 "줄 공사(Laying lines)"가 부실하다고 지적하면 다시 해야 하듯, IT 감리에서도 "보안 로직 미흡" 지적 시 재개발해야 합니다.

```text
      ┌─────────────────────────────────────────────────────────────────┐
      │                    [IT 감리의 시대적 배경]                       │
      ├─────────────────────────────────────────────────────────────────┤
      │                                                                 │
      │  과거 (Pre-2000s)                 현재 (Post-2010s)            │
      │  ────────────────                 ────────────────             │
      │  * 개발사 자체 검수 위주            * 제3자 전문 감리 법인 필수    │
      │  * 결과물 중심 (Output)            * 프로세스 중심 (Process)     │
      │  * 장애 발생 시 소모적 불만         * 예방적 점검 및 위험 관리     │
      │  * 납품이 곧 완료 (Delivery)       * 운영 안정성까지 책임 (O&M)   │
      │                                                                 │
      │          ▼                           ▼                         │
      │  "작동하면 된다"                  "안전하고 투명해야 한다"       │
      │                                                                 │
      └─────────────────────────────────────────────────────────────────┘
```

> 📢 **섹션 요약 비유**: 
> 정보시스템 감리는 고속도로 건설 현장에 **"공사감독관"**이 상주하는 것과 같습니다. 도로(시스템)가 예쁘게 포장됐는지보다, **다리가 무너지지 않을지(안정성), 과속 차량이 튕겨나갈지(보안), 이갈이 현상은 없는지(성능)**를 전문가의 돋보기로 미리 확인하여, 개통 후 대형 사고를 막는 역할을 합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

정보시스템 감리는 단순한 "문서 보기"가 아닌, 체계적인 **거버넌스(Governance)** 구조 위에서 작동합니다. 이 절에서는 감리의 법적/기술적 구조와 심층 절차를 다룹니다.

#### 1. 감리 대상 및 범위 (Mandatory Scope)

**전자정부법 시행령 제39조** 등에 의거하여 다음의 사업은 의무 감리 대상입니다.
1.  **총 사업비 5억 원 이상**인 정보화 사업 (단, 부가세(VAT) 제외).
2.  행정전산망 또는 대국민 서비스와 연계되어 **장애 발생 시 사회적 파장**이 큰 사업.
3.  다수 기관이 공동으로 활용하거나 표준화가 필요한 **공통 기반 구축 사업**.

#### 2. 감리 거버넌스 및 관계자 구조 (Governance Architecture)

감리는 독립성을 최우선으로 합니다. **감리법인(Auditor)**은 사업자(Developer)로부터 독립하여 발주기관(Owner)에 대한 **성실 의무(Fiduciary Duty)**를 가집니다.

```text
      [ 발주기관 (Owner) ]
             │
             │ ▲  1. 계약 (감리 의뢰)
             │ │  2. 감리 비용 지급
             │ │
             │ │
      ┌──────┴──────┐
      │             │
      │             │
      │             │
 [ 감리법인 ] ────▶ [ 수주업체 (Developer) ]
 (Auditor)     ◀─── (Auditee)
   │  ▲               ▲
   │  │ 3. 지적 및 권고 │ 1. 개발 및 납품
   │  │ 4. 승인 요청     │ 2. 산출물 제출
   │  │               │
   │  └───────┐       │
   │          │       │
   └──────────┘       │
         ▲            │
         │ 5. 시정조치 │
         │    확인 요구│
         └────────────┘

   Legend:
   -----> : 지시/지적/의뢰
   <----- : 보고/제출/응답
```

**도해 해설**:
1.  **발주기관**은 감리법인과 감리 계약을 체결하며, 수주업체와는 용역(개발) 계약을 체결합니다.
2.  **감리법인**은 개발 과정에서 산출물을 검토하거나 현장 테스트를 수행하며, 부적정 사항을 발견하면 수주업체에 **'감리 의견서'**를 통보하고 수정을 요구합니다.
3.  **수주업체**는 이를 이행한 후 **'시정 조치 결과'**를 보고해야 하며, 이것이 최종 납품 승인의 **필수 조건(Gate)**이 됩니다.

#### 3. 상세 감리 절차 및 체크포인트 (Procedure & Checkpoints)

감리는 일반적으로 **① 착수 → ② 수행 → ③ 완료**의 3단계로 진행되며, 각 단계별 핵심 관점(KPI)은 다음과 같습니다.

| 단계 | 세부 절차 | 주요 활동 (Activities) | 주요 산출물 (Deliverables) | 심층 관점 (Deep Dive) |
|:---:|:---|:---|:---|:---|
| **1. 착수** | **예비조사** (Pre-survey) | 사업 이해관계자 인터뷰, 기존 자료 분석, **감리 계획(RMP)** 수립 | 감리 계획서, 위험 식별서 | **Scope Definition**: 감리의 범위(기술/보운/관리)와 중점 점검 항목(Risk-based)을 정의 |
| **2. 수행** | **현장감리** (On-site) | 설계 검토, 소스코드 리뷰, 성능/보안 테스트, **진단서** 작성 | 중간 점검 보고서, **감리 의견서** | **Compliance Check**: 요구사항 정의서(SRS)와 설계서(HLD/LLD)의 일치성, 코드 품질 복잡도 |
| **3. 완료** | **결과보고** (Closing) | 최종 감리 결과 보고, 시정 조치 이행 여부 확인, 최종 **승인** | 감리 결과 보고서, 시정조치 확인서 | **Sign-off**: 시정 조치가 완료될 때까지 납품 승인을 보류하여 품질 강제 |

#### 4. 기술적 감리 항목 (Technical Audit Matrix)

실무 감리에서는 기술사(P.E.) 수준에서 다음과 같은 구체적인 지표를 검증합니다.

```text
      ┌───────────────────────────────────────────────────────────────┐
      │                 [Technical Audit Matrix]                      │
      ├───────────────────────────────────────────────────────────────┤
      │                                                               │
      │  1. Architecture (아키텍처)                                   │
      │     ├── Monolithic vs MSA 적합성 여부                          │
      │     ├── 결합도(Coupling) / 응집도(Cohesion) 분석               │
      │     └── 확장성(Scalability) 확보 전략 (H/W, S/W Scaling)       │
      │                                                               │
      │  2. Data (데이터)                                             │
      │     ├── 정규화(Normalization) 검증 (3NF 이상 준수 여부)        │
      │     ├── DB Lock 경합(Lock Contention) 및 Deadlock 유발 가능성   │
      │     └── 백업 및 복구(Backup & Recovery) 절차 유효성            │
      │                                                               │
      │  3. Security (보안)                                           │
      │     ├── OWASP Top 10 취약점(Injection, XSS 등) 점검           │
      │     ├── 권한 분리(SoD: Separation of Duties) 구현 여부         │
      │     └── 암호화 알고리즘 적정성 (AES-256, SHA-256 등)           │
      │                                                               │
      │  4. Performance (성능)                                        │
      │     ├── 평균 응답 시간(ART: Average Response Time) < 1초 준수  │
      │     ├── 동시 사용자 처리량(TPS: Transactions Per Second) 만족  │
      │     └── 처리율(Throughput) 병목 지점(네트워크/DB) 식별         │
      │                                                               │
      └───────────────────────────────────────────────────────────────┘
```

> 📢 **섹션 요약 비유**:
> 감리 절차는 자동차의 **"공장 출하 검수(Inspection)"** 과정과 같습니다. 엔진(서버), 브레이크(보안), 연비(성능)가 설계도서대로 조립됐는지, 연결 부위에 덜거나 느슨한 곳은 없는지 **전동 공구(테스트 툴)와 육안(코드 리뷰)으로** 꼼꼼히 확인한 뒤, 문제가 없으면 **"합격(Quality Assured)"** 인증서를 붙입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

감리는 프로젝트 관리(PMO), 품질 관리(QA), 정보 보안(ISMS) 등 타 영역과 밀접하게 연계되어 있으나, 그 **"성격"**과 **"책임 범위"**는 명확히 달라야 합니다.

#### 1. 감리(Audit) vs PMO (Project Management Office)

감리와 PMO는 혼동되기 쉽지만, **"심판(Referee)"과 "코치(Coach)"**의 차이가 있습니다.

| 비교 항목 | 감리 (IS Audit) | PMO (Project Management Office) |
|:---|:---|:---|
| **소속/위상** | **외부 제3자** (독립성 원칙) | **내부 관리 조직** (발주자 소속) |
| **주 목적** | **Compliance (준법/적정성)**<br>위험 발견, 규정 준수, 품질 보증 | **Support (지원/성공)**<br>일정 관리, 자원 조율, 의사소통 창구 |
| **시각** | 과거 지향적 (Already done?)<br>"잘못된 것을 찾음" | 미래 지향적 (To be done?)<br>"잘되도록 도움" |
| **권한** | **감리 의견 통보 및 시정 요구권**<br>(납품과 연결될 수 있음) | 권고 및 조정 권한 (강제력 없음) |
| **비유** | 경찰(단속) | 내비게이션(안내) |

**시너지(Synergy)**:
- PMO가 일상 프로젝트 관리를 잘 수행하면, 감리 단계에서 발견될 **하자(Hazard)가 줄어듭니다.**
- 반면, 감리는 PMO가 놓칠 수 있는 **"안주(Complacency)"**를 깨우치는 역할을 합니다.

#### 2. 감리(Audit) vs QA (Quality Assurance)

QA는 개발 과정 내에서의 품질 시험(Test)을 담당하고, 감리는 그 QA가 **제대로 수행되었는지를 검증(Verify)**합니다.
- **QA**: "버그가 없는지 테스트함." (1차 방어선)
- **Audit**: "QA 팀이 테스트 시나리오를 충분히 수행했는
