# 673. 기능점수 (FP) 내부논리파일(ILF) 외부연계파일(EIF)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소스코드의 양이 아닌 사용자가 느끼는 **소프트웨어의 기능적 규모**를 정량화하여 산정하는 기법으로, 구현 기술과 독립적인 ISO 표준 비용 산정 방식이다.
> 2. **데이터 기능**: 시스템 내부에서 유지되는 데이터인 **내부논리파일(ILF)**과, 타 시스템에서 참조만 하는 **외부연계파일(EIF)**을 식별하고 복잡도(RET, DET)에 따라 점수를 부여한다.
> 3. **가치**: 프로젝트 초기 단계에서 정확한 규모 예측이 가능하며, 사용자 중심의 요구사항 명세와 연동되어 개발 생산성 분석 및 비용 정산의 객관적 지표로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "코드가 아닌 가치를 측정하라"

알란 알브레히트(Alan Albrecht)가 제안한 **기능점수(FP, Function Point)**는 "개발자가 얼마나 힘들게 짰느냐"가 아니라 "사용자가 어떤 기능을 쓰느냐"를 봅니다. 이는 언어(C, Java, Python)에 따라 코드 줄 수(LOC)가 천차만별인 문제를 해결하고, 발주자와 수주자 간의 합리적인 계약 기준을 제공합니다.

### 💡 비유: 아파트 분양가 산정

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        기능점수(FP) 산정 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 지으려는 아파트의 가격(비용)을 매겨야 함.                               │
│                                                                             │
│  1. LOC 방식 (벽돌 개수):                                                    │
│     "벽돌을 10만 장 썼으니 비싸다." (벽돌이 크면 어떡하지? 비효율적임)             │
│                                                                             │
│  2. FP 방식 (평수 및 방 개수):                                               │
│     "거실 1개, 안방 1개, 화장실 2개(기능)니까 이 가격이다."                      │
│     - ILF (내 안방): 내가 직접 관리하고 청소하는 내 집 안의 방.                  │
│     - EIF (단지 내 엘리베이터): 내가 쓰긴 하지만 관리실에서 관리하는 외부 시설.     │
│                                                                             │
│  → 사용자가 실제로 사용하는 공간(기능)의 가치로 가격을 결정하는 것!               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## II. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 기능점수 구성 요소 (5대 기능)

FP는 크게 **데이터 기능**과 **트랜잭션 기능**으로 나뉩니다.

| 분류 | 기능 명칭 | 영문 명칭 | 설명 |
|:---:|:---|:---|:---|
| **데이터** | **내부논리파일** | **ILF (Internal Logical File)** | 시스템 내부에서 생성/수정/삭제되는 데이터 집합 (예: 회원 DB) |
| **기능** | **외부연계파일** | **EIF (External Interface File)** | 타 시스템에서 참조만 해오는 데이터 집합 (예: 우편번호 DB) |
| **트랜잭션**| **외부입력** | **EI (External Input)** | 데이터를 DB에 등록, 수정, 삭제하는 기능 |
| **기능** | **외부출력** | **EO (External Output)** | 계산/가공 로직을 거쳐 데이터를 외부로 내보내는 기능 |
| | **외부조회** | **EQ (External Inquiry)** | 단순 조회 및 필터링하여 데이터를 보여주는 기능 |

### 2. 복잡도 산정 기준 (Data Function 중심)
ILF와 EIF의 점수는 다음 두 가지 요소로 결정됩니다.
- **RET (Record Element Type)**: 논리적인 데이터 그룹 (예: 회원 마스터와 상세 정보).
- **DET (Data Element Type)**: 사용자 식별 가능한 개별 필드 (예: 이름, 나이, 주소).

### 3. FP 산정 프로세스 (ASCII)

```text
    [ 범위 확정 ] ──▶ [ 기능 식별 ] ──▶ [ 복잡도 산정 ] ──▶ [ 미조정 FP ]
    (Boundaries)      (5대 기능 도출)     (RET/DET/FTR)      (Unadjusted)
                                                                 │
          ▲                                                      │
          │                                                      ▼
    [ 최종 FP 산출 ] ◀── [ 보정 계수 적용 ] ◀── [ 특성 분석 ]
    (Size/Cost)         (VAF / 보정항목)      (GSC / 14가지)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. ILF vs EIF 결정적 차이
- **ILF**: 우리 시스템이 '주인'입니다. 우리가 데이터를 넣고 뺍니다.
- **EIF**: 우리 시스템은 '손님'입니다. 남의 데이터를 읽기만 합니다. (우리가 내용을 바꿀 수 없음)

### 2. 기술적 시너지: 요구사항 관리 (RTM)
요구사항 추적성 매트릭스(RTM)의 각 항목은 FP의 기능과 1:1 매핑될 수 있습니다. 이를 통해 개발 도중 요구사항이 늘어날 때(Scope Creep), "기능점수가 이만큼 늘었으니 예산도 이만큼 증액되어야 한다"는 논리적 근거를 확보할 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 공공기관 프로젝트 정산
- **상황**: 프로젝트 종료 후, 실제 구현된 기능이 제안서와 다르다는 분쟁 발생.
- **결단**: **정규 FP 산정 및 사후 정산**. 
- **판단**: 개발된 화면과 DB 테이블을 전수 조사하여 ILF, EIF, EI, EO, EQ를 재식별. 특히 외부 연동 API가 많아진 부분을 EIF로 정확히 산정하여 과업 변경에 따른 추가 비용 청구의 객관성 확보.

### 📢 기술사적 결언
> "기능점수는 소프트웨어 공학의 **'계량 경제학'**이다. 단순히 개발비 계산용 도구를 넘어, 전체 시스템의 **'데이터 중심 구조'**를 파악하는 가장 좋은 방법이다. 특히 최근의 클라우드 네이티브 환경에서는 API 연동이 많아지므로 EIF와 EO의 정밀한 구분이 프로젝트의 경제적 성패를 좌우한다. 아키텍트는 설계 초기부터 FP 산정 가이드라인을 이해하고, 거대 클래스가 아닌 유의미한 기능 단위의 모듈화를 실천해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **산정 객관성**: 사람마다 다른 견적 차이를 10% 이내로 평준화.
- **생산성 지표**: $Man-Month / FP$ 계산을 통해 우리 팀의 개발 효율을 정밀 측정.

### 미래 전망
미래의 FP 산정은 **'Automated FP'**가 주도할 것입니다. 소스코드와 아키텍처 문서를 AI가 분석하여 자동으로 기능을 분류하고 복잡도를 계산하는 ISO 표준(OMG) 기반의 자동 측정 도구들이 확산될 것입니다. 또한 마이크로서비스 간의 복잡한 통신을 어떻게 FP로 녹여낼지에 대한 신규 표준 연구가 활발히 진행되고 있습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **기능점수**는 장난감 성을 살 때, 블록이 몇 개인지(코드)가 아니라 **"방이 몇 개고 문이 몇 개인지"**로 가격을 매기는 거예요.
2. **내부논리파일**은 내 방에 있는 장난감 상자고, **외부연계파일**은 친구네 집에 있는 장난감 목록을 잠깐 빌려보는 거예요.
3. 이렇게 방 개수로 가격을 정하면, 블록을 작게 쓰든 크게 쓰든 **똑같이 공평한 가격**을 낼 수 있답니다!
