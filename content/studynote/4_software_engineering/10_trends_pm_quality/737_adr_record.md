# 737. 아키텍처 기술서 (ADR) 결정 이력 관리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 설계 과정에서 내린 중요한 아키텍처적 결정사항과 그 **맥락(Context), 근거(Rationale), 결과(Consequences)**를 기록한 짧은 문서들의 집합이다.
> 2. **지식 보존**: 시간이 흐른 뒤에도 "왜 이 기술을 선택했는가?"에 대한 답을 제공하여, 과거의 결정을 오해하고 잘못 수정하는 '아키텍처 부패'를 방지하는 지식 관리 도구이다.
> 3. **가치**: 버전 관리 시스템(Git)에 코드와 함께 저장되어 개발자 누구나 접근 가능하게 함으로써, 설계 거버넌스를 투명하게 유지하고 신규 인력의 온보딩 비용을 획기적으로 낮춘다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "사라지는 설계의 의도"

개발팀은 매일 수많은 선택을 합니다. "MySQL 대신 MongoDB를 쓰자", "Kafka를 도입하자" 등등. 하지만 1년 뒤, 그 결정을 내렸던 사람이 퇴사하면 남은 사람들은 그 이유를 모른 채 고통받거나, 함부로 설계를 뒤엎어 대형 사고를 냅니다. **ADR (Architecture Decision Record)**은 '결과'뿐만 아니라 그 이면의 **'이유'**를 기록하여 아키텍처를 영속시키는 기술입니다.

### 💡 비유: 역사책과 법전의 결합

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        ADR (아키텍처 결정 기록) 비유                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 대대로 내려오는 종갓집의 가풍(시스템)을 지켜야 함.                       │
│                                                                             │
│  1. ADR이 없다면:                                                            │
│     "왜 우리 집은 제사상에 사과를 안 올리지?" 이유를 몰라서 그냥 사과를 올림.      │
│     알고 보니 조상님 중 사과 알레르기가 있어 금지한 것이었음 (장애 발생!).        │
│                                                                             │
│  2. ADR이 있다면:                                                            │
│     낡은 일기장(ADR)을 펼쳐봄. "1920년, 장남의 알레르기 사고로 인해 사과를         │
│     금지하기로 결정함. 대신 배를 올림."                                         │
│                                                                             │
│  → 과거의 결정이 **어떤 상황(Context)**에서 **왜(Rationale)** 내려졌는지         │
│     적어둔 보물지도와 같은 것!                                                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. ADR의 표준 구조 (Michael Nygard 양식)

| 섹션 명칭 | 내용 |
|:---:|:---|
| **Title** | 결정 사항을 나타내는 짧은 제목 (예: ADR-005 API 통신에 gRPC 사용) |
| **Status** | 현재 상태 (Proposed, Accepted, Superceded, Deprecated) |
| **Context** | 결정이 필요했던 배경, 제약 사항, 비즈니스 상황 |
| **Decision** | **최종 선택한 대안**과 그 내용 |
| **Rationale** | 왜 이 대안을 선택했는가? (Trade-off 분석 내용) |
| **Consequences** | 결정 후 얻게 된 이점과 감수해야 할 불편함 (부작용) |

### 2. ADR 관리 프로세스
1. **Identify**: 결정이 필요한 아키텍처 드라이버 식별.
2. **Discuss**: 팀원들과 대안 검토 및 토론.
3. **Draft**: ADR 초안 작성 (Markdown 형식 권장).
4. **Accept**: 승인 후 Git 저장소의 `docs/adr` 폴더에 커밋.
5. **Review**: 배포 후 결과(Consequences)를 보강하거나, 새로운 ADR로 대체(Supercede).

### 3. ADR 생명주기 및 연결 구조 (ASCII)

```text
    [ ADR-001: Java 17 도입 ] (Status: Accepted)
           │
           ▼ (대체됨)
    [ ADR-012: Java 21 업그레이드 ] (Status: Accepted)
    - Context: 성능 향상 필요
    - Decision: Java 21로 전환
    - Supercedes: ADR-001 <── 연결 고리 (Traceability)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 설계서(SDD) vs ADR
- **설계서 (Design Doc)**: 시스템의 **현재 모습**을 설명하는 '사진' (정적).
- **ADR**: 시스템이 **왜 이런 모습이 되었는지**를 설명하는 '영상 기록' (동적/이력).

### 2. 기술적 시너지: GitOps 및 코드 리뷰
ADR은 코드와 같은 저장소에서 관리되어야 합니다. 새로운 ADR이 올라오면 Pull Request를 통해 모든 개발자가 리뷰하고 'Approve' 함으로써, 설계 결정이 소수의 독단이 아닌 **팀 전체의 합의**가 되는 문화를 만듭니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 마이크로서비스 간 통신 방식 결정
- **상황**: HTTP/REST를 쓸지 Kafka를 쓸지 팀 내 의견이 팽팽함.
- **결단**: **ADR 작성을 통한 의사결정 기록**. 
- **판단**: "현재 실시간 응답이 중요(Context)하므로 HTTP를 선택하되, 향후 트래픽 증가 시 확장성을 위해 인터페이스는 추상화한다(Decision). 이로 인해 서비스 간 강결합 문제는 감수한다(Consequences)."
- **효과**: 2년 뒤 "왜 Kafka 안 썼어?"라는 비판에 대해, 당시의 기술적 제약과 비즈니스 우선순위를 ADR로 명쾌하게 입증.

### 📢 기술사적 결언
> "훌륭한 아키텍처는 정답을 맞히는 것이 아니라, **'당시의 최선'을 증명하는 것**이다. 기술은 변하고 환경은 바뀌지만, 그 결정을 내렸던 **'논리적 과정'**은 영원한 자산이 된다. 아키텍트는 코드를 짜는 시간만큼이나 '왜 이렇게 짰는가'를 기록하는 데 시간을 써야 한다. ADR이 없는 프로젝트는 과거의 실수를 반복하거나, 미래의 변경을 두려워하는 비겁한 아키텍처가 될 뿐이다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **소통 비용**: 반복되는 기술적 논쟁 50% 이상 감소.
- **온보딩 속도**: 신규 개발자가 아키텍처 맥락을 파악하는 시간 수 주에서 수 일로 단축.

### 미래 전망
미래의 ADR은 **'지능형 지식 베이스'**와 통합될 것입니다. AI가 소스코드의 변경 패턴을 읽어 ADR과의 불일치를 자동으로 감지하거나, 새로운 기술을 도입할 때 과거 ADR 데이터를 바탕으로 예상되는 'Consequences'를 시뮬레이션해 주는 지능형 설계 지원 시스템이 보편화될 것입니다. 이제 ADR은 단순한 문서를 넘어 **'실행 가능한 설계 이력'**으로 진화할 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **ADR**은 장난감을 조립할 때, 왜 이 부품을 여기에 끼웠는지 적어두는 **"조립 일기장"**이에요.
2. 나중에 동생이 와서 "이거 왜 여기 있어? 빼버리자!"라고 할 때, 일기장을 보여주며 **"이건 성이 안 무너지게 받쳐주는 거야"**라고 설명해줄 수 있죠.
3. 일기장을 잘 써두면, 시간이 흘러도 우리 장난감 성이 **왜 멋진지** 모두가 알 수 있답니다!
