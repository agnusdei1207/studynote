# 733. 리포지토리 패턴 데이터 접근 추상화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도메인 모델 계층과 데이터 매핑 계층 사이의 중재자 역할을 수행하며, 데이터 저장소(DB, API 등)에 접근하는 로직을 마치 메모리 상의 컬렉션처럼 다룰 수 있게 **객체지향적으로 추상화**하는 패턴이다.
> 2. **관심사 분리**: 비즈니스 로직이 특정 DB 기술(SQL, JPA, NoSQL)에 종속되지 않게 격리하여, 도메인 모델의 순수성을 유지하고 데이터 원천의 변경에 유연하게 대응할 수 있게 한다.
> 3. **가치**: 실제 DB 없이도 가짜 객체(Mock)를 활용한 단위 테스트를 가능하게 하며, 데이터 조회 및 저장 로직을 한곳에 응집시켜 코드의 재사용성과 유지보수성을 극대화한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "코드 속에 숨어든 SQL의 위협"

비즈니스 로직(예: 이자 계산) 한복판에 `SELECT * FROM ...`과 같은 SQL 코드가 섞여 있으면, DB 스키마가 바뀔 때마다 비즈니스 코드 전체를 고쳐야 합니다. 또한 테스트할 때마다 실제 DB를 연결해야 하는 불편함이 따릅니다. **리포지토리 (Repository)** 패턴은 "데이터가 어디에 어떻게 저장되는지는 잊어라. 그저 객체를 달라고 요청하라"는 철학으로 이 문제를 해결합니다.

### 💡 비유: 도서관 사서에게 책 빌리기

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        리포지토리 패턴 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 도서관에서 "자바 프로그래밍" 책을 빌리고 싶음.                           │
│                                                                             │
│  1. 패턴 미적용 (직접 찾기):                                                 │
│     내가 직접 지하 창고에 내려가서, 수천 개의 선반(Table)을 뒤지고,               │
│     어떤 분류 체계(SQL)로 되어 있는지 공부해서 책을 꺼내옴. (힘들고 복잡함)        │
│                                                                             │
│  2. 리포지토리 적용 (사서 서비스):                                            │
│     나는 사서(Repository)에게 가서 "자바 책 한 권 주세요"라고 **말로 요청**함.     │
│     사서는 창고가 지하에 있든 별관에 있든, DB가 오라클이든 엑셀이든 상관없이       │
│     내가 원하는 **'책 객체'**를 딱 가져다줌.                                    │
│                                                                             │
│  → 창고의 복잡한 규칙(기술)을 몰라도, 사서(인터페이스)만 알면 데이터를 얻음!       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 리포지토리 패턴의 구조 (Components)
- **Domain Layer**: 리포지토리의 **인터페이스**를 정의. (비즈니스 중심)
- **Infrastructure Layer**: 특정 DB 기술을 사용하여 인터페이스를 **구현**. (기술 중심)
- **Client (Service)**: 인터페이스를 통해 데이터 요청. 실제 구현체가 무엇인지 알 필요 없음.

### 2. 주요 메서드 유형 (In-memory Collection semantics)
- `save(entity)`: 객체 저장 또는 수정.
- `findById(id)`: 식별자로 객체 조회.
- `findAll()`: 모든 객체 목록 반환.
- `remove(entity)`: 객체 삭제.

### 3. 데이터 접근 아키텍처 (ASCII)

```text
    [ Domain Service ] ──▶ [ Repository Interface ] ◀── [ Client ]
                                     ▲
                                     │ (Implementation)
    ┌────────────────────────────────┴─────────────────────────────────┐
    │                   Infrastructure Layer (Data Access)             │
    │  ┌──────────────┐      ┌─────────────┐      ┌─────────────────┐  │
    │  │ JpaRepository│      │ MybatisRepo │      │ MockRepository  │  │
    │  │ (PostgreSQL) │      │ (Legacy DB) │      │ (For Testing)   │  │
    │  └──────────────┘      └─────────────┘      └─────────────────┘  │
    └──────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. DAO (Data Access Object) vs Repository
- **DAO**: DB 테이블과 1:1로 매핑되는 **데이터 중심**의 하위 수준 API. (CRUD 위주)
- **Repository**: **도메인 객체(Aggregate)** 중심으로 데이터를 관리하는 고수준 API. 복잡한 조회 로직을 도메인 언어로 표현.

### 2. 기술적 시너지: Spring Data JPA
개발자가 인터페이스만 정의하면 프레임워크가 런타임에 구현체를 자동으로 생성해 주는 기술입니다. 리포지토리 패턴의 생산성을 극대화하며, 복잡한 쿼리는 QueryDSL과 결합하여 타입 안정성(Type Safety)까지 확보합니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 멀티 데이터 소스 통합
- **상황**: 주문 정보는 내부 RDBMS에 있고, 과거 주문 이력은 외부 클라우드 스토리지(S3)에 저장되어 있음.
- **결단**: **단일 리포지토리 인터페이스로 통합**. 
- **판단**: `OrderRepository` 인터페이스 하나를 두고, 내부 로직에 따라 RDBMS에서 가져올지 S3에서 가져올지 결정하는 구현체 작성. 
- **효과**: 서비스를 사용하는 개발자는 데이터가 어디에 흩어져 있는지 신경 쓰지 않고 `orderRepository.getHistory(id)` 하나만 호출하면 됨.

### 📢 기술사적 결언
> "리포지토리는 도메인 모델의 **'영속성 게이트웨이'**다. 이것이 없으면 비즈니스 로직은 인프라의 바다에 빠져 죽게 된다. 아키텍트는 리포지토리를 설계할 때, 단순 CRUD의 나열이 아니라 **'도메인의 언어'**를 담으려 노력해야 한다. `findByStatus('01')` 보다는 `findActiveOrders()`가 훨씬 더 가치 있는 리포지토리 설계다. 진정한 추상화는 기술을 숨기는 것이 아니라, **의도를 드러내는 것**이기 때문이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **테스트 속도**: 실제 DB 대신 In-memory 리포지토리 사용 시 테스트 속도 10배 이상 향상.
- **코드 중복 감소**: 데이터 조회 로직의 단일화로 관련 버그 30% 감소.

### 미래 전망
미래의 리포지토리는 **'자율형 데이터 브로커'**로 진화할 것입니다. 데이터의 저장 위치(Cloud, Edge, On-premise)와 비용, 성능을 고려하여 리포지토리가 실시간으로 최적의 저장소를 선택하여 데이터를 입출력하는 지능형 계층이 될 것입니다. 또한 NoSQL과 RDBMS의 경계가 허물어지는 **'다중 모델(Multi-model) 리포지토리'**가 표준 아키텍처로 자리 잡을 것입니다.
