# 729. 객체지향 결합도 (내용, 공통, 제어, 스탬프, 자료)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 두 모듈(클래스) 간의 상호 의존성을 나타내는 지표로, 한 모듈의 변경이 다른 모듈에 얼마나 영향을 미치는지를 결정하는 **소프트웨어 설계 품질의 핵심 척도**이다.
> 2. **계단식 구조**: 결합도가 낮을수록(자료 결합도) 좋으며, 높을수록(내용 결합도) 나쁘다. **내용 > 공통 > 제어 > 스탬프 > 자료** 순으로 결합도가 낮아진다.
> 3. **가치**: "느슨한 결합(Loose Coupling)"을 지향하여 모듈의 독립성을 확보함으로써, 코드의 재사용성을 높이고 유지보수 시 발생하는 사이드 이펙트(Side Effect)를 최소화한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "왜 결합도가 중요한가?"

소프트웨어 공학의 성인 글렌포드 마이어스(Glenford Myers)가 제안한 개념입니다. 시스템이 거대한 하나의 덩어리라면 수정이 불가능합니다. 시스템을 쪼갰다면, 그 조각들이 서로 얼마나 **'끈적끈적하게'** 붙어있는지가 중요합니다. 너무 끈적거리면(High Coupling) 하나를 뗄 때 옆 조각이 부서지고, 적당히 떨어져 있으면(Low Coupling) 부품 갈아 끼우듯 쉽게 고칠 수 있습니다.

### 💡 비유: 가전제품과 리모컨의 연결 방식

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        객체지향 결합도 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 자료 결합도 (최상 - 블루투스):                                             │
│     필요한 소리 조절 신호(Data)만 딱 보냄. 서로의 내부 회로는 전혀 모름.          │
│                                                                             │
│  2. 제어 결합도 (주의 - 전용 리모컨):                                          │
│     "TV 모드로 동작해!"라고 명령(Flag)을 보냄. 내가 상대의 동작 방식을 결정함.     │
│                                                                             │
│  3. 내용 결합도 (최악 - 전선 직접 꼬기):                                       │
│     상대방 가전제품의 뚜껑을 열고 내부 전선을 내 멋대로 연결함.                  │
│     상대가 내부 회로를 조금만 바꿔도 내 리모컨은 폭발함!                         │
│                                                                             │
│  → 연결 통로가 좁고 명확할수록(Data), 서로가 자유롭고 편안한 설계!               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 결합도 5단계 상세 분석 (나쁨 → 좋음)

| 단계 | 유형 | 상세 내용 | 품질 |
|:---:|:---|:---|:---:|
| **1** | **내용 (Content)** | 다른 모듈의 **내부 변수나 기능을 직접 참조**하거나 수정함 | **최악** |
| **2** | **공통 (Common)** | 여러 모듈이 하나의 **전역 변수(Global Variable)**를 공유함 | 나쁨 |
| **3** | **제어 (Control)** | 무엇을 할지 결정하는 **제어 신호(Flag)**를 전달하여 상대 로직 제어 | 낮음 |
| **4** | **스탬프 (Stamp)** | 모듈 간에 **데이터 구조(객체, 구조체)** 전체를 전달함 | 보통 |
| **5** | **자료 (Data)** | 필요한 **개별 데이터 항목(Parameter)**만 인자로 주고받음 | **최상** |

### 2. 결합도와 유지보수의 관계
- **높은 결합도**: "파문 효과(Ripple Effect)". 코드 한 줄 고쳤는데 전혀 상관없는 10개 클래스에서 에러가 남.
- **낮은 결합도**: "격리(Isolation)". 해당 모듈만 테스트하고 배포해도 안전함.

### 3. 결합도 시각화 (ASCII)

```text
    [ High Coupling ]                     [ Low Coupling ]
    (내용 결합도)                          (자료 결합도)
    
    ┌────────┐ 직접 참조 ┌────────┐        ┌────────┐ 파라미터 ┌────────┐
    │ Class A│ --------> │ Class B│        │ Class A│ --------> │ Class B│
    │ { b.x=1}           │ { int x}        │ { work(5) }        │ { int n}
    └────────┘           └────────┘        └────────┘            └────────┘
    (B의 내부를 알아야 함)                   (값만 넘겨주면 끝)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 결합도 vs 응집도 (Cohesion)
- **결합도**: 모듈 **사이**의 관계 (낮을수록 좋음).
- **응집도**: 모듈 **내부**의 관계 (높을수록 좋음).
- **시너지**: 결합도를 낮추려고 모듈을 억지로 쪼개면 응집도가 떨어질 수 있습니다. 아키텍트는 이 둘의 균형점을 찾는 '최적화'를 수행해야 합니다.

### 2. 기술적 시너지: 인터페이스(Interface)와 DIP
결합도를 낮추는 가장 강력한 도구는 **인터페이스**입니다. 구현체가 아닌 추상화된 인터페이스에 의존(Dependency Inversion Principle)하게 함으로써, 실제 클래스 간의 결합도를 '제로'에 가깝게 만들 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: "스탬프 결합도" 리팩토링
- **상황**: `calculateTax(User user)` 함수가 `User` 객체 전체를 받지만, 실제로는 `user.salary` 값 하나만 사용함.
- **결단**: **자료 결합도로 하향 조정**. 
- **판단**: `User` 객체 구조가 바뀌면(예: 이름 필드 삭제) 세금 계산 함수와 아무 상관 없는데도 컴파일 에러가 발생함.
- **조치**: `calculateTax(int salary)`로 함수 서명을 변경.
- **효과**: 세금 계산 로직이 `User` 객체로부터 완전히 독립되어 재사용성 향상.

### 📢 기술사적 결언
> "결합도는 **'시스템의 유연성'**을 결정하는 유일한 물리적 지표다. 많은 개발자가 '공통 결합도(전역 변수)'의 편리함에 유혹당하지만, 이는 아키텍처에 암세포를 심는 것과 같다. 아키텍트는 클래스 간의 대화가 오직 **'필요한 최소한의 정보'**만으로 이루어지도록 통제해야 한다. '모든 길은 자료 결합도로 통한다'는 원칙을 지키는 것만으로도 대다수의 유지보수 지옥을 예방할 수 있다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **영향도 분석 시간**: 낮은 결합도 확보 시 수정 범위 파악 시간 70% 단축.
- **코드 재사용**: 모듈 독립성 확보로 신규 프로젝트에 기존 모듈 이식성 2배 향상.

### 미래 전망
미래의 개발 환경에서는 **'컴파일 타임 결합도 체크'**가 더욱 강화될 것입니다. 아키텍처 린팅 도구가 실시간으로 결합도 단계를 계산하여, 제어 결합도 이상의 코드가 작성되면 커밋을 차단하는 자동화된 거버넌스가 정착될 것입니다. 또한 마이크로서비스 환경에서 네트워크 호출을 통한 결합도를 관리하는 **'서비스 메시'** 기술이 결합도 관리의 새로운 지평을 열 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **결합도**는 친구와 내 손이 얼마나 끈적끈적한 껌으로 붙어 있느냐 하는 거예요.
2. 껌이 너무 크면(내용 결합도) 친구가 화장실 갈 때 나도 끌려가야 해서 **너무 불편하죠.**
3. 필요한 물건만 손으로 주고받는 사이(자료 결합도)가 되어야, **서로가 편하고 자유롭게** 놀 수 있답니다!
