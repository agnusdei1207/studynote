# 729. 객체지향 결합도 (내용, 공통, 제어, 스탬프, 자료)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 모듈(클래스) 간의 상호 의존성 강도를 측정하는 척도로, 한 모듈의 내부 변경이 다른 모듈에 미치는 ** Ripple Effect (파급 효과)**의 지표이다.
> 2. **측정 기준**: 결합도는 낮을수록 우수하며, **내용 결합도(Content Coupling, 최악)부터 공통, 제어, 스탬프를 거쳐 자료 결합도(Data Coupling, 최상)**까지의 계층 구조를 가진다.
> 3. **가치**: **Loose Coupling (느슨한 결합)**을 통해 모듈의 독립성을 확보하여, 시스템의 유지보수성(Maintainability)을 획기적으로 높이고 테스트 가능성(Testability)을 보장하는 설계의 핵심 품질 지표이다.

---

### Ⅰ. 개요 (Context & Background) - [600자+]

#### 개념 및 철학
소프트웨어 공학에서 결합도(Coupling)는 두 모듈 간의 상호의존성 정도를 나타내는 척도입니다. 1974년 글렌포드 마이어스(Glenford Myers) 등이 체계화한 이 개념은, 모듈화(Modularity)의 지향점인 "변경의 국지화(Localization of Changes)"를 달성하기 위한 핵심 메트릭스(Metrics)입니다.
결합도가 높다는 것은 한 모듈이 다른 모듈의 내부 구조나 로직에 대해 너무 많이 알고 있다는 뜻이며, 이는 곧 하나의 기능을 수정하기 위해 연관된 수많은 코드를 함께 수정해야 하는 "스파게티 코드(Spaghetti Code)"의 원인이 됩니다.

#### 등장 배경 및 진화
① **초기 프로그래밍의 한계**: 초기의 폰 노이만 구조식 절차적 프로그래밍은 전역 변수(Global Variable)와 GOTO문의 무분별한 사용으로 모듈 간 경계가 모호했습니다.
② **구조적 프로그래밍의 대두**: 모듈 간 인터페이스를 명확히 하고, 데이터 은닉(Data Hiding)을 통해 변경 영향을 최소화하려는 패러다임이 등장했습니다.
③ **비즈니스 요구**: 현대의 대규모 분산 시스템(MSA)에서는 수시로 변경되는 비즈니스 로직을 유연하게 수용하기 위해 **반드시 '느슨한 결합(Loose Coupling)'** 아키텍처가 요구됩니다.

#### 💡 핵심 비유: 건물의 확장 공사
건물을 신축할 때 벽돌 하나를 떼어내려고 하는데, 그 벽돌이 철골 구조물, 전선, 수도관, 그리고 이웃집 안방 벽까지 미끈하게 얽혀 있다면 공사는 불가능합니다. 반면, **레고 블록**처럼 필요한 부분만 끼워 맞추고 떼어낼 수 있다면 유지보수가 매우 용이할 것입니다. 결합도는 이 "분리 가능성"을 나타내는 척도입니다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                          [ 결합도 계층 구조 시각화 ]                           │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│    ☣️ 나쁨 (Bad)                                   좋음 (Good) ✨             │
│                                                                              │
│    ┌───────────────────────────────────────────────┐                         │
│    │              높은 결합도 (High Coupling)       │                         │
│    │  변경 시 사이드 이펙트(Side Effect) 발생 빈번   │                         │
│    │  ⚠️ 내용 (Content)       (직접 참조/수정)        │                         │
│    │  ⚠️ 공통 (Common)        (전역 데이터 공유)      │                         │
│    │  ⚠️ 제어 (Control)       (제어 플래그 전달)      │                         │
│    │  ✅ 스탬프 (Stamp)        (데이터 구조 전달)      │                         │
│    │  ✅ 자료 (Data)          (기본형 데이터 전달)     │                         │
│    └───────────────────────────────────────────────┘                         │
│             ↓                                   ↓                            │
│    [ 강한 의존성 ]                          [ 약한 의존성 ]                   │
│    (재사용성 ↓, 유지보수성 ↓)                (재사용성 ↑, 유지보수성 ↑)       │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유:**
> 마치 복잡하게 얽힌 **'떡잡는 실'**처럼, 한 가닥을 당기면 온통 엉망이 되는 시스템이 최악의 결합도(내용 결합도)이며, **'블루투스 이어폰'**처럼 필요할 때만 신호로 연결되고 끊어지는 관계가 최상의 결합도(자료 결합도)입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive) - [1,200자+]

#### 1. 구성 요소 상세 분석 (5가지 결합도 유형)

결합도를 이해하기 위해서는 각 유형별로 데이터와 제어권이 어떻게 모듈을 넘나드는지 심층 분석해야 합니다.

| 결합도 유형 | 상세 기술적 메커니즘 | 변경 영향도 (Ripple Effect) | 품질 수준 |
|:---:|:---|:---|:---:|
| **내용 (Content)** | **최악의 결합**. A 모듈이 B 모듈의 내부 변수나 로직을 **직접 참조하거나 수정**함. (Java의 Reflection 강제 사용이나 C의 extern 남발 등) | B의 내부 구조 변경 시 A가 즉시 파손됨. | **최악** |
| **공통 (Common)** | 두 모듈 이상이 **전역 데이터(Global Data)**를 공유하여 사용함. 어느 모듈이 데이터를 수정했는지 추적 어려움. | 전역 데이터 구조 변경 시 모든 의존 모듈 전체 수정 필요. | 나쁨 |
| **제어 (Control)** | 단순히 데이터를 넘기는 것이 아니라, **"어떻게 동작해라"라는 제어 플래그(Flag)**를 넘겨서 상대방의 로직 흐름을 바꿈. | 상태 플래그 추가/삭제 시 호출하는 곳과 받는 곳 모두 로직 변경. | 낮음 |
| **스탬프 (Stamp)** | 모듈 간에 **데이터 구조(Object, Struct)** 자체를 인자로 전달함. 전체 구조를 다 쓰지 않아도 통째로 넘김. | 구조체의 멤버 변수 변경 시(이름 타입 등) 참조하는 모든 모듈 재컴파일. | 보통 |
| **자료 (Data)** | **최상의 결합**. 필요한 **스칼라 데이터(Primitive Value)**나 별도의 캡슐화된 데이터만 인자로 전달. (Call by Value/Reference) | 매개변수의 타입만 유지된다면 내부 로직 변경에 영향 없음. | **최상** |

#### 2. 결합도 ASCII 구조 다이어그램

아래 다이어그램은 모듈 A가 모듈 B를 호출할 때, 결합도의 강도에 따라 인터페이스가 어떻게 달라지는지를 보여줍니다.

```text
    【 1. 내용 결합도 (Content Coupling) 】           【 2. 공통 결합도 (Common Coupling) 】
    
     Module A                                    Module A
       │                                            │
       │ ┌──────────────┐                          │  ┌──────────────────┐
       │ │ 직접 메모리    │   (위험 직접 접근)         │  │ Global Data Area │
       └─┤ 참조/쓰기     ├───────────────────┐       │  │ (Shared State)   │
         │ (B의 내부)    │                   │       └──┤  int g_count;    │
         └──────────────┘                   │          │  ...             │
             ▼                                │          └──────────────────┘
         ┌──────────────┐                    │                ▲    ▲
         │   Module B   │                    │                │    │
         │ { private x } │                    │           Read/Write │
         └──────────────┘                    │                │    │
                                             │          ┌─────┴────┴─────┐
                                             │          │   Module B      │
                                             │          │                 │
                                             │          └─────────────────┘

    【 3. 제어 결합도 (Control Coupling) 】           【 4. 스탬프 결합도 (Stamp Coupling) 】
    
     Module A                                    Module A
       │                                            │
       │  send(flag)                                │  send(UserInfo object)
       │                                            │
       ▼                                            ▼
     ┌──────────────┐                            ┌──────────────┐
     │   Module B   │                            │   Module B   │
     │ if (flag==1) │    (A가 B의 로직을 통제)     │ userInfo.addr │ (필요없는 데이터도
     │   doThis();  │                            │ userInfo.tax  │  포함될 수 있음)
     │ else         │                            │ calcTax();   │
     │   doThat();  │                            └──────────────┘
     └──────────────┘

    【 5. 자료 결합도 (Data Coupling) - IDEAL 】
    
     Module A
       │
       │  send(int price, float taxRate)  (오직 필요한 데이터만)
       │
       ▼
     ┌──────────────┐
     │   Module B   │
     │ return p * t │
     └──────────────┘
```

#### 3. 심층 동작 원리 및 메커니즘

**① 데이터 흐름과 추상화 계층**
결합도의 본질은 "모듈 B가 모듈 A를 얼마나 '구체적으로' 아느냐"입니다.
- **내용/공통 결합도**: A는 B의 **구현(Implementation)** 세부 사항(메모리 레이아웃, 변수명)까지 압니다. B의 내부가 바뀌면 A는 멈춥니다.
- **제어 결합도**: A는 B의 **절차(Procedure)**를 알고 있습니다. A가 "왼쪽으로 가라"고 명령하는 셈입니다.
- **스탬프 결합도**: A는 B가 사용하는 **데이터 스키마(Schema)**를 압니다. 스키마가 바뀌면 영향을 받습니다.
- **자료 결합도**: A는 B가 무엇을 하는지 모르고, 단지 **입력(Input)과 출력(Output)**에만 관여합니다. 이는 **블랙박스(Black Box)** 이론에 부합합니다.

**② 안티 패턴 사례 (Ghost Update)**
전역 변수(공통 결합도)를 사용하는 시스템에서 발생하는 흔한 버그는 **Ghost Update**입니다. 모듈 A가 전역 배열 `g_buffer`를 초기화했다고 가정하지만, 사실 모듈 B가 해당 버퍼를 사용 중이어서 데이터가 뒤섞이는 현상입니다. 이는 디버깅이 거의 불가능한 잠재적 결함(Latent Defect)이 됩니다.

**③ 코드 수준의 비교 (Pseudo-code)**

```c
// [Bad] 제어 결합도 (Control Coupling)
// 호출자가 피호출자의 로직을 제어함
void processOrder(Order order, int modeFlag) { 
    if (modeFlag == 1) { // VIP Process
        applyDiscount(order, 0.3);
    } else if (modeFlag == 2) { // Normal Process
        applyDiscount(order, 0.1);
    }
    // modeFlag가 추가될 때마다 이 함수는 수정되어야 함.
}

// [Good] 자료 결합도 (Data Coupling)
// 호출자는 데이터만 전달하고, 로직은 객체 내부에 캡슐화됨
void processOrder(Order order) { 
    // Order 객체 내부 이미 로직이 캡슐화되어 있음 (다형성 활용)
    order.applyBestDiscount(); 
}
```

**📢 섹션 요약 비유:**
> 모듈 간의 결합도는 **'포장 방식'**과 같습니다. 상대방의 가방을 뒤져서 물건을 꺼내 쓰는 것(내용 결합도)은 도둑이나 다름없고, 서류 가방을 통째로 건네는 것(스탬프 결합도)은 불필요한 정보를 노출하는 위험이 있습니다. 오로지 **'명확하게 적힌 영수증(데이터)'**만 주고받는 것이 가장 안전하고 깔끔한 거래입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy) - [800자+]

#### 1. 결합도(Coupling) vs 응집도(Cohesion)의 상관관계

소프트웨어 설계의 양대 축은 결합도와 응집도입니다. 이 둘은 Trade-off 관계가 아니라, 동시에 달성해야 하는 목표입니다.

| 비교 항목 | 결합도 (Coupling) | 응집도 (Cohesion) |
|:---|:---|:---|
| **정의** | 모듈 **간(Inter-module)**의 상호 의존성 | 모듈 **내(Intra-module)** 요소들의 관련성 |
| **지향점** | **낮을수록 (Low)** 좋음 (Loose) | **높을수록 (High)** 좋음 (Tight) |
| **목표** | 모듈의 독립성 확보 | 모듈의 단일 책임(Single Responsibility) 보장 |
| **수식적 관계** | **Software Quality = f(High Cohesion, Low Coupling)** |

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    [ Good Design Quadrant (좋은 설계 사분면) ]               │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│    High Cohesion (높은 응집도)                                               │
│         │                                                                    │
│      Ⅰ │ Ⅱ                                                                  │
│   (Good) │ (Bad)                                                             │
│ ────────┼───────────                                                         │
│      Ⅲ │ Ⅳ  ⭐ 목표 지점 (High Cohesion & Low Coupling)                     │
│   (Bad) │ (Good)                                                             │
│         │                                                                    │
│    Low Coupling (낮은 결합도) ──────────────────► High Coupling (높은 결합도) │
│                                                                              │
│   Ⅰ: 기능 응집도 + 자료 결합도 (이상적)                                       │
│   Ⅱ: 우연 응집도 + 자료 결합도 (불필요한 쪼개짐)                              │
│   Ⅲ: 기능 응집
