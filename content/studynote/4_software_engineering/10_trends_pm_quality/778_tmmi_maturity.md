# 778. 소프트웨어 테스트 성숙도 모델 (TMMi)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 테스트 프로세스의 정신적, 물적 토대를 진단하고 체계적으로 고도화하기 위해 설계된 **TMMi (Test Maturity Model integration)** 프레임워크로, 단순한 테스트 수행을 넘어 '품질 보증(QA)의 산업화'를 지향합니다.
> 2. **구조**: **CMMI (Capability Maturity Model Integration)** 모델의 구조를 벤치마킹하되, 개발 라이프사이클 전반에서 테스트의 독립적인 가치와 위상을 정의한 5단계 성숙도 레벨을 제공합니다.
> 3. **가치**: 리스크 기반 테스팅(Risk-Based Testing)과 정량적 메트릭을 도입하여 '테스트의 효율성'과 '결함 조기 탐지율'을 극대화하고, 결과적으로 소프트웨어 유지보수 비용을 획기적으로 절감합니다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 철학**
**TMMi (Test Maturity Model integration)**는 소프트웨어 테스트 프로세스를 평가하고 개선하기 위한 구조적 프레임워크입니다. 이는 단순히 '버그를 찾는 행위'를 넘어, 소프트웨어의 품질을 보증하는 독립된 공학적 프로세스로서 테스트를 정의하고 관리합니다. TMMi는 **TPI (Test Process Improvement)** 및 **CMMI**의 성숙도 개념을 통합하여 발전시킨 모델로, 조직이 현재 위치한 품질 수준을 진단하고, 다음 단계로 도달하기 위한 로드맵을 제시합니다.

**2. 등장 배경: "검증(Verification)과 확인(Validation)의 간극"**
전통적인 **SW (Software)** 개발 방법론인 **CMMI (Capability Maturity Model Integration)**는 개발 프로세스 전반의 성숙도를 다루지만, '테스트'라는 영역을 독립적인 관점에서 심층적으로 다루지 못하는 한계가 있었습니다. 개발 프로세스가 성숙해도(CMMI 레벨 3~4), 테스트가 여전히 '개발자의 검토' 수준에 머물러 있다면, 고도화된 개발 프로세스에서 발생한 결함을 필터링하지 못하고 운영 환경으로 유출시키는 결과를 초래했습니다. 이에 따라 테스트를 개발로부터 분리하여, 테스트만의 전문성, 독립성, 그리고 프로세스를 고도화하려는 요구가 TMMi 탄생의 배경이 되었습니다.

**3. 💡 비유: 운전면허 학원의 체계적인 커리큘럼**
TMMi 도입 전의 테스트 현장은 '도로 연습만 무작정 시키는' 운전 학원과 같습니다. 어떤 학생은 고속도로 연습을 하고, 어떤 학생은 주차 연습만 하다가 시험장에 갑니다(임기응변식 Ad-hoc).
반면 **TMMi**는 '교육 과정'과 '평가 기준'이 체계화된 전문 자동차 학원과 같습니다.
- 초급: 기본 조작과 교통 법규(기본 테스트 정책)
- 중급: 주행 요령 및 응급 상황 대처법(테스트 설계 및 기법)
- 고급: 선회 감각 및 효율적인 주행 루트(성능 및 최적화)
이처럼 레벨별로 명확한 목표와 절차가 존재합니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│               TMMi 체계화 전후 비교: 학원 커리큘럼의 차이                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [도입 전: Ad-hoc Testing]          [도입 후: TMMi Based Testing]           │
│                                                                             │
│  (전) 막강 운전 자동차 학원             (TMMi) 프로페셔널 운전 아카데미         │
│                                                                             │
│  ┌───────────────────┐           ┌───────────────────────────────────┐      │
│  │  강사: "일단 나가봐!"  │           │  Level 1: 기본 조작/주의사항 이론      │      │
│  └───────────────────┘           │  Level 2: 기초 코스 주행 훈련        │      │
│          ↓                        │  Level 3: 도로 주행/응급 상황 시뮬레이션│      │
│  ┌───────────────────┐           │  Level 4: 주행 데이터 분석/코칭     │      │
│  │  수강생: "어디서부터 해?"│           │  Level 5: 맞춤형 드라이빙 최적화   │      │
│  └───────────────────┘           └───────────────────────────────────┘      │
│          ↓                                    ↓                              │
│   결과: 운이 좋으면 합격             결과: 체계적인 훈련을 거친            │
│        운이 나쁘면 사고 발생              안전하고 능숙한 운전자 양성        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**해설**:
위 다이어그램은 TMMi 도입을 통한 프로세스 정비의 필요성을 보여줍니다. 도입 전(Ad-hoc)은 개별 테스터의 역량에 의존하여 결과가 일관되지 않지만, TMMi는 레벨별 성숙도 단계를 통해 예측 가능한 품질 결과를 보장합니다. 이는 단순한 '노력'의 차이가 아니라 '체계(System)'의 차이입니다.

> **📢 섹션 요약 비유**: 마치 요리사가 '기분 대로' 요리하는 가게(레벨 1)가 아닌, 레시피와 매뉴얼, 그리고 고객 만족도 조사가 철저히 갖춰진 '프랜차이즈 본사(레벨 5)'로 운영 체계를 업그레이드하는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. TMMi 구성 요소 및 상세 매트릭스**
TMMi는 5단계의 성숙도 레벨로 구성되며, 각 레벨은 특정 **PA (Process Area, 프로세스 영역)**들을 달성함으로써 승격됩니다. 각 단계는 독립적이지 않으며, 누적되는 성격을 가집니다.

| 레벨 | 명칭 | 핵심 목표 | 주요 Process Areas (PA) | 상세 특징 및 내부 동작 |
|:---:|:---|:---|:---|:---|
| **1** | **Initial (초기)** | 비공식적이고 혼란스러운 프로세스 | (없음) | 테스트는 디버깅(Debugging)의 일부로 취급됨. 문서화 없음. 운영에서 결함 발견. |
| **2** | **Managed (관리)** | 프로젝트 차원의 테스트 계획 및 통제 | 1. 테스트 정책 및 목표<br>2. 테스트 계획<br>3. 테스트 전략<br>4. 테스트 모니터링 및 통제<br>5. 테스트 설계 및 실행<br>6. 테스트 환경 관리 | 프로젝트마다 테스트 계획 수립. 요구사항과 테스트 케이스의 추적성(traceability) 확보 시작. **(반복 가능)** |
| **3** | **Defined (정의)** | 조직 차원의 표준화와 프로세스 정의 | 1. 테스트 조직<br>2. 테스트 표준화<br>3. 테스트 교육 프로그램<br>4. 검토(Review)<br>5. 통합 테스트<br>6. 시스템/인수 테스트 | **전사적 표준 프로세스** 수립. 테스트 그룹의 조직적 독립성 확보. 비기능(성능/보안) 테스트 포함. |
| **4** | **Measured (측정)** | 정량적 평가와 리스크 기반 통제 | 1. 테스트 측정<br>2. 소프트웨어 품질 평가<br>3. 고급 검토<br>4. 결함 예측 | 테스트 프로세스와 제품의 품질을 **정량적 지표(메트릭)**로 관리. **DDP (Defect Detection Probability)** 등을 활용하여 결함 유출 예측. |
| **5** | **Optimization (최적화)** | 지속적 개선과 결함 예방 | 1. 결함 예방<br>2. 테스트 프로세스 개선<br>3. 품질 통제 | 통계적 품질 관리(SQC). 자동화 도구의 고도화. 결함이 발생하기 전에 예방하는 문화 정착. |

**2. 성숙도 단계별 진화 구조 (ASCII Diagram)**

```text
      ▲ [ Level 5: Optimization (최적화) ]
      │   ★ Focus: 결함 예방 (Prevention) & 지속적 개선
      │   └─► Auto-remediation, AI-driven Test Optimization
      │
      ▲ [ Level 4: Measurement (측정) ]
      │   ★ Focus: 정량적 통제 (Quantitative Control)
      │   └─► Metrics Collection (Efficiency, Coverage, DDP)
      │
      ▲ [ Level 3: Definition (정의) ]
      │   ★ Focus: 표준화 (Standardization)
      │   └─► Organization-wide Standards, Training, Peer Reviews
      │
      ▲ [ Level 2: Management (관리) ]
      │   ★ Focus: 프로젝트별 기본기 (Basic Project Management)
      │   └─► Test Planning, Requirements, Documentation
      │
      ▼ [ Level 1: Initial (초기) ]
      ★ Focus: Chaos (Ad-hoc)
      └─► "Code & Fix"
```

**해설**:
TMMi의 피라미드 구조는 테스트 역량이 저절로 높아지지 않음을 시사합니다.
- **Level 1 → 2**: '무질서'에서 '반복 가능성'을 확보하는 단계입니다. 기본적인 테스트 계획서와 요구사항 추적성(traceability matrix)이 도입됩니다.
- **Level 2 → 3**: '프로젝트 차원'의 성공을 '조직 차원'의 자산으로 확장하는 단계입니다. 핵심은 A 프로젝트의 테스트 경험이 B 프로젝트에 재사용되는 **표준화(Standardization)**입니다.
- **Level 3 → 4**: '정성적 품질'을 '정량적 수치'로 관리하는 단계입니다. 단순히 "테스트를 많이 했다"가 아니라, "잠재적 결함의 95%를 발견했다"는 수치로 입증해야 합니다.
- **Level 4 → 5**: 측정을 넘어 최적화입니다. 결함을 미리 예방하는 방식으로 업무 패턴이 변형됩니다.

**3. 핵심 메트릭: DDP (Defect Detection Probability)**
TMMi 레벨 4 이상에서 중요하게 다루는 **DDP (결함 탐지 확률)**는 테스트 프로세스의 효과성을 측정하는 척도입니다.

> **DDP 공식**:
> $$ DDP = \frac{\text{테스트 단계 발견 결함 수}}{\text{테스트 단계 발견 결함 수} + \text{운영 단계 발견 결함 수}} $$

- **실무적 해석**: DDP가 0.8(80%)라면, 개발자가 만든 결함 10개 중 8개는 테스터가 찾고 2개는 고객에게 돌아간다는 의미입니다. TMMi는 이 비율을 높이는 것을 목표로 합니다.

> **📢 섹션 요약 비유**: 마치 건물을 지을 때, 초기엔 '붕어빵 굽는 도구'로 흙을 파다가(레벨 1), 점차 '삽'과 '포크레인(레벨 2, 3)'을 갖추고, 마침내 '드론으로 지반을 스캔하여 붕괴 위험을 미리 계산하는(레벨 4, 5)' 고층 빌딩 건설 기술로 발전하는 과정과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. CMMI vs TMMi: 상호 보완적 관계 분석**

TMMi는 CMMI와 경쟁 모델이 아니라 CMMI 내부의 약점을 보완하는 확장 모델입니다. 특히 CMMI 레벨 3(정의됨)에 도달한 조직이 TMMi 레벨 2에 머무르는 경우가 많아, 실제 품질 결과는 개선되지 않는 '껍데기 성숙도' 현상이 발생합니다.

| 비교 항목 | CMMI (Capability Maturity Model Integration) | TMMi (Test Maturity Model integration) |
|:---|:---|:---|
| **주요 관점** | **Engineering & Management**: 요구사항 분석, 설계, 코딩 등 개발 전반의 프로세스 | **Testing & Quality**: 검증(Verification)과 확인(Validation)에 특화 |
| **테스트의 위상** | 개발의 하위 활동(V&V)으로 취급될 가능성 높음 | 테스트는 개발과 **동등하거나 독립적인** 평가 파트너 |
| **Process Area** | 20~25개의 PA (REQM, PMC, PI 등) | 16개의 PA (Test Policy, Organizational Test 등) |
| **성숙도 기준** | 조직 전체의 프로젝트 수행 능력 평가 | 테스트 전문성, 독립성, 자산화 수준 평가 |
| **시너지 포인트** | 프로젝트 관리 기반 제공 | 제공된 기반 위에서 품질 리스크를 실질적으로 통제 |

**2. 기술적 융합: Agile/DevOps와 TMMi의 조화**
TMMi는 전통적인 **Waterfall** 모델뿐만 아니라 **Agile (애자일)** 환경에도 적용 가능합니다. 오히려 빈번한 배포(Continuous Deployment) 환경에서는 **Shift-left Testing** (초기 단계 테스트)을 통해 결함 비용을 줄여야 하므로 TMMi의 '프로세스 정의(레벨 3)'가 더욱 중요합니다.

```text
   [ Traditional Waterfall + TMMi ]          [ DevOps / Agile + TMMi ]
   
   Req -> Dev -> Test (Very Late)       CI Pipeline
