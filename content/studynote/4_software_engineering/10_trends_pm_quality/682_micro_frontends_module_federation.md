# 682. 마이크로 프론트엔드 웹팩 연계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스(MSA)의 철학을 프론트엔드 영역으로 확장하여, 거대한 웹 애플리케이션을 독립적으로 개발, 테스트, 배포 가능한 작은 단위의 **마이크로 앱(Micro-apps)**들로 쪼개어 구성하는 아키텍처다.
> 2. **기술적 핵심**: 웹팩 5(Webpack 5)의 **모듈 페더레이션(Module Federation)** 기능을 통해, 빌드 타임이 아닌 런타임에 서로 다른 원격 컨테이너로부터 컴포넌트나 로직을 동적으로 공유하고 통합한다.
> 3. **가치**: 팀 간 기술 스택의 자율성을 보장하고, 서비스의 일부분만 수정하여 배포할 수 있는 민첩성을 제공하며, 프론트엔드 코드베이스의 거대화로 인한 유지보수 한계를 극복한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "프론트엔드 모놀리스의 붕괴"

백엔드가 MSA로 진화하는 동안 프론트엔드는 여전히 하나의 거대한 저장소에서 관리되는 모놀리식 구조를 유지하는 경우가 많았습니다. 이는 한 팀의 코드 실수가 전체 배포를 막거나, 빌드 시간이 수십 분씩 걸리는 문제를 야기했습니다. **마이크로 프론트엔드 (Micro Frontends)**는 화면의 각 영역(Header, Cart, Product 등)을 독립된 프로젝트로 분리하여 이러한 병목을 해결합니다.


## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 모듈 페더레이션 (Module Federation) 핵심 개념
웹팩 5에서 도입된 혁신적인 공유 방식으로, 라이브러리를 설치(npm install)하는 대신 URL을 통해 실시간으로 기능을 당겨옵니다.
- **Host (Shell)**: 여러 리모트 앱들을 통합하여 화면에 띄워주는 메인 컨테이너.
- **Remote**: 기능을 제공하는 개별 마이크로 앱. 자신의 빌드 산출물을 외부에서 쓸 수 있게 노출(Expose).
- **Shared**: React나 Vue 같은 공통 라이브러리를 중복 로드하지 않고 한 번만 내려받도록 설정.

### 2. 마이크로 프론트엔드 통합 전략 (Integration)
- **Build-time Integration**: 빌드 시점에 컴포넌트를 합침 (결합도 높음).
- **Server-side Integration**: 서버(Nginx, SSI)에서 HTML을 조립.
- **Run-time Integration**: 브라우저에서 JS를 통해 동적으로 로드 (가장 유연함, 모듈 페더레이션 방식).

### 3. 아키텍처 구조도 (ASCII)

```text
    [ Browser / Shell App ] (Host)
           │
    ┌──────┴───────────────────────────┐ (Module Federation)
    │                                  │
    ▼ [ Load from URL ]                ▼ [ Load from URL ]
    ┌─────────────────┐                ┌─────────────────┐
    │   Search App    │                │   Payment App   │
    │ (Team A / Vue)  │                │ (Team B / React)│
    └────────┬────────┘                └────────┬────────┘
             │                                  │
    ┌────────▼────────┐                ┌────────▼────────┐
    │  S3 / CDN (A)   │                │  S3 / CDN (B)   │ <── 독립 배포
    └─────────────────┘                └─────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. IFrame vs Module Federation

| 항목 | IFrame 방식 | 모듈 페더레이션 (MF) |
|:---:|:---|:---|
| **격리 수준** | 매우 높음 (완전 분리) | 논리적 분리 (커널/윈도우 공유) |
| **UX/성능** | 무거움, 스크롤 이슈 발생 | **매우 매끄러움, 자원 공유 최적화** |
| **데이터 공유** | postMessage 등 복잡함 | **Props나 전역 상태(Store) 공유 용이** |
| **SEO** | 어려움 | **상대적 용이 (SSR 연계 시)** |

### 2. 기술적 시너지: 모노레포 (Monorepo)
마이크로 프론트엔드는 모노레포(Nx, Turborepo)와 결합할 때 관리 효율이 극대화됩니다. 수많은 마이크로 앱들의 공통 컨벤션과 공통 UI 컴포넌트를 한곳에서 관리하면서도, 배포는 개별적으로 수행할 수 있는 '분산된 조화'를 이룹니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 대규모 이커머스 개편
- **상황**: 메인 페이지, 마이페이지, 장바구니를 각기 다른 3개 팀이 개발 중. 한 팀의 배포 지연이 전체 오픈을 막음.
- **결단**: **마이크로 프론트엔드 전환**. 
- **판단**: 쉘 앱(Shell)을 만들고 각 팀의 영역을 리모트(Remote) 앱으로 분리. 
- **효과**: 장바구니 팀은 React 18로 업그레이드하고 싶은데 메인 팀은 React 17을 고수해야 하는 상황에서, 각자 독립적인 버전을 사용하며 점진적 마이그레이션 성공.

### 📢 기술사적 결언
> "마이크로 프론트엔드는 단순한 UI 분할이 아니라 **'팀의 자율성'**에 대한 설계다. 모듈 페더레이션은 이를 가능케 하는 강력한 무기지만, 잘못 쓰면 서비스 간의 버전 충돌(Version Conflict)과 복잡한 상태 공유 지옥에 빠질 수 있다. 아키텍트는 화면을 쪼개기 전에 먼저 **'팀 사이의 비즈니스 경계'**를 DDD 관점에서 명확히 긋고, 최소한의 인터페이스로 소통하는 구조를 지향해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **배포 주기**: 일 1회 통합 배포에서 각 팀별 수시 배포로 개선.
- **빌드 속도**: 전체 빌드 대비 개별 앱 빌드 시간 80% 이상 단축.

### 미래 전망
미래의 마이크로 프론트엔드는 **'Web Components'** 표준과 더욱 밀착될 것입니다. 프레임워크에 상관없이 모든 마이크로 앱들이 표준 커스텀 엘리먼트로 동작하여, 진정한 의미의 '프레임워크 독립적 아키텍처'가 완성될 것입니다. 또한 서버 사이드에서의 모듈 페더레이션 지원이 강화되면서, SSR 환경에서도 매끄러운 분산 프론트엔드 환경이 표준으로 자리 잡을 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **마이크로 프론트엔드**는 커다란 도화지 한 장에 그림을 그리는 게 아니라, **작은 스티커들**을 모아서 붙여넣는 것과 같아요.
2. 장난감 기차(앱)를 만들 때, 바퀴 팀과 지붕 팀이 **따로 만들어서 나중에 합치기만** 하면 멋진 기차가 완성되죠.
3. 바퀴가 고장 나면 기차 전체를 버리지 않고, **그 바퀴 스티커만 떼서 새 걸로** 갈아 붙이면 된답니다!
