# 735. 도메인 주도 설계 (DDD) 전술적 설계 패턴

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 전략적 설계(Strategic Design)에서 정의된 바운디드 컨텍스트 내부를 실제로 구현하기 위해 사용하는 **객체지향 설계 패턴의 집합**으로, 비즈니스 로직을 모델 내에 견고하게 응집시키는 데 목적이 있다.
> 2. **핵심 도구**: 고유 식별자를 가진 **엔티티**, 불변 값인 **VO**, 이들의 묶음인 **애그리게이트**, 데이터 접근을 추상화하는 **리포지토리**, 그리고 특정 객체에 속하지 않는 로직을 위한 **서비스**로 구성된다.
> 3. **가치**: 기술적인 세부 사항(Persistence, Framework)보다 **비즈니스 도메인의 본질**을 코드에 투영하여, 복잡한 요구사항 변화에도 흔들리지 않는 유연하고 테스트 가능한 소프트웨어 구조를 완성한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "전략이 방향이라면, 전술은 승리다"

이벤트 스토밍이나 컨텍스트 매핑이 '어디로 갈 것인가'를 정하는 전략이라면, **전술적 설계 (Tactical Design)**는 그 안에서 '어떻게 코드를 짤 것인가'를 다룹니다. 무질서하게 흩어진 로직을 DDD의 정해진 패턴에 따라 배치함으로써, 도메인의 지식이 코드로 정확히 번역되도록 만듭니다.

### 💡 비유: 축구 전술과 선수 배치

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        DDD 전술적 설계 패턴 비유                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 엔티티 (핵심 선수): 이름과 등번호(ID)가 있는 주전 선수들.                   │
│  2. 값 객체 (장비): 선수가 입은 유니폼, 축구화. (중요하지만 갈아입을 수 있음)      │
│  3. 애그리게이트 (팀 워크): 수비진, 공격진처럼 함께 움직여야 하는 조직.           │
│  4. 리포지토리 (벤치): 선수를 경기장에 내보내거나(조회) 들여보내는(저장) 곳.      │
│  5. 서비스 (작전 타임): 특정 선수의 개인기보다 팀 전체의 움직임이 필요한 로직.     │
│                                                                             │
│  → 각자의 역할을 명확히 나누고(패턴), 정해진 규칙대로 협력하여 **승리(가치)**를    │
│     쟁취하는 것!                                                               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 전술적 설계의 5대 핵심 패턴

| 패턴 명칭 | 핵심 개념 | 주요 특징 |
|:---:|:---|:---|
| **Entity** | 식별성 (Identity) | 상태가 변해도 고유한 ID로 추적 가능 (예: 회원, 주문) |
| **Value Object** | 불변성 (Immutability) | 값 자체로 정의됨. ID 없음. 교체 방식으로 수정 (예: 주소, 금액) |
| **Aggregate** | 일관성 (Consistency) | 관련 객체의 묶음. 루트를 통해서만 접근 및 수정 |
| **Service** | 행위 (Behavior) | 상태가 없는 로직 전용 객체. 도메인 vs 애플리케이션 구분 |
| **Repository** | 영속성 (Persistence) | 도메인 모델을 저장소로부터 조회/저장하는 인터페이스 |

### 2. 패턴 간의 관계 및 의존성 구조 (ASCII)

```text
    ┌───────────────────────────────────────────────┐
    │           Application Service                 │ <── 흐름 제어 (Manager)
    └──────┬───────────────────────┬────────────────┘
           │                       │
    ┌──────▼──────────┐     ┌──────▼────────────────┐
    │ Domain Service  │     │  Repository (IF)      │ <── 인프라 추상화
    └──────┬──────────┘     └──────┬────────────────┘
           │                       │
    ┌──────▼───────────────────────▼────────────────┐
    │             Aggregate (Boundary)              │
    │  ┌─────────────────────────────────────────┐  │
    │  │         Aggregate Root (Entity)         │  │ <── 진입점
    │  ├────────────────────┬────────────────────┤  │
    │  │  Internal Entity   │   Value Objects    │  │
    │  └────────────────────┴────────────────────┘  │
    └───────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 전술적 설계 vs 클린 아키텍처
- **전술적 설계 패턴**은 클린 아키텍처의 **Entity**와 **Use Case** 계층을 구체적으로 채우는 내용물입니다.
- 리포지토리 인터페이스는 클린 아키텍처의 **Interface Adapter** 계층에 해당하며, 도메인 중심의 의존성 역전을 실현합니다.

### 2. 기술적 시너지: 객체지향 5원칙 (SOLID)
전술적 설계는 SOLID 원칙의 실천판입니다. 특히 **단일 책임 원칙(SRP)**은 엔티티와 서비스의 분리로, **의존성 역전 원칙(DIP)**은 리포지토리 패턴으로 구체화됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 레거시 코드의 DDD 전환
- **상황**: 서비스 클래스 하나에 5,000줄의 로직이 몰려 있는 '갓 서비스(God Service)' 상태.
- **결단**: **전술적 패턴으로 로직 분산**. 
- **판단**: 
    1. 데이터만 있던 테이블을 행위가 있는 **엔티티**로 격상. 
    2. 중복되는 유효성 검사 로직을 **VO** 내부로 캡슐화. 
    3. 트랜잭션의 단위를 **애그리게이트**로 재정의하여 DB 락(Lock) 충돌 방지.
- **효과**: 서비스 코드가 500줄 이하로 슬림해지며, 비즈니스 규칙의 가시성 200% 향상.

### 📢 기술사적 결언
> "전술적 설계는 단순히 코딩 가이드가 아니라 **'도메인 지식의 캡슐화 전략'**이다. 개발자가 패턴의 형식에만 집착하면 '빈약한 도메인 모델'이라는 껍데기만 남게 된다. 아키텍트는 패턴을 쓰기 이전에, **'이 로직이 도메인의 진실을 말하고 있는가?'**를 끊임없이 질문해야 한다. 올바른 전술적 설계는 코드가 곧 비즈니스 문서가 되는 '유비쿼터스 언어'의 완성 지점이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **유지보수성**: 도메인 로직 응집으로 영향도 분석 시간 60% 단축.
- **품질**: 복잡한 비즈니스 정책 누락 방지 및 테스트 커버리지 향상.

### 미래 전망
미래의 전술적 설계는 **'AI-First Tactical Design'**으로 나아갈 것입니다. AI가 도메인 스토리텔링 내용을 분석하여 최적의 엔티티-VO 구조를 설계하고, 애그리게이트 경계를 제안하며, 리포지토리와 서비스의 뼈대 코드를 자동 생성하는 시대가 올 것입니다. 개발자는 AI가 만든 전술 모델의 '비즈니스 정합성'을 최종 승인하는 고도의 '도메인 설계자'로 거듭날 것입니다.
