# 735. 디자인 바이 컨트랙트 불변 조건

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 버트런드 메이어(Bertrand Meyer)가 제안한 설계 기법으로, 클래스의 메서드를 호출하는 쪽(Client)과 실행하는 쪽(Supplier) 사이의 권리와 의무를 명확한 **계약(Contract)** 형식으로 정의하는 방식이다.
> 2. **3대 명세**: 실행 전 충족되어야 할 **선행 조건(Pre-condition)**, 실행 후 보장되는 **후행 조건(Post-condition)**, 그리고 객체의 생애 주기 내내 변하지 않아야 할 **불변 조건(Invariant)**으로 구성된다.
> 3. **가치**: 코드의 문서화와 검증을 동시에 달성하며, 버그의 책임 소재를 명확히 하여(의무 위반 시 호출자 책임, 보장 실패 시 수신자 책임) 견고한 소프트웨어 아키텍처를 구축한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "누가 잘못했는가?"

소프트웨어 에러가 났을 때, "데이터를 잘못 보낸 사람"이 잘못인지 "잘못된 데이터를 처리 못한 쪽"이 잘못인지 싸우는 경우가 많습니다. **디자인 바이 컨트랙트 (DbC, Design by Contract)**는 이를 법적 계약처럼 문서화합니다. "네가 이 조건을 지켜서 호출하면(의무), 나는 이 결과를 보장하겠다(권리)"는 약속을 코드에 박아넣는 것입니다.

### 💡 비유: 택배 배송 서비스의 계약

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        디자인 바이 컨트랙트(DbC) 비유                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 손님(Client)이 택배 회사(Supplier)에 물건을 맡김.                       │
│                                                                             │
│  1. 선행 조건 (손님의 의무): "물건은 10kg 이하여야 하고, 깨지는 물건이 아니어야    │
│     한다." (이걸 안 지키면 택배 회사는 배송을 거부할 권리가 있음)                 │
│                                                                             │
│  2. 후행 조건 (회사의 보장): "내일 오전 10시까지 상대방 집 앞에 도착시킨다."      │
│     (손님이 의무를 다했다면, 회사는 반드시 이걸 지켜야 함)                       │
│                                                                             │
│  3. 불변 조건 (언제나 지킬 약속): "배송 과정 중에 택배 상자 안의 물건 개수는      │
│     변하지 않는다." (시작부터 끝까지 항상 참이어야 함)                           │
│                                                                             │
│  → 약속을 미리 정하니, 사고가 났을 때 **누구 책임인지** 금방 알 수 있죠!          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. DbC의 3대 핵심 조건 (Assertions)

| 조건 유형 | 영문 명칭 | 책임 주체 | 특징 |
|:---:|:---|:---:|:---|
| **선행 조건** | **Pre-condition** | **호출자 (Client)** | 메서드 실행 전 반드시 참이어야 함. 어길 시 호출자의 버그. |
| **후행 조건** | **Post-condition** | **수신자 (Supplier)** | 메서드 종료 후 결과가 약속과 일치해야 함. 어길 시 수신자의 버그. |
| **불변 조건** | **Invariant** | **클래스 자체** | 객체가 생성된 후 파괴될 때까지 모든 공용 메서드 전후에 항상 참이어야 함. |

### 2. 책임 소재의 명확화 (The Blame Game)
- **Pre-condition 위반**: "데이터를 이상하게 보낸 네가 고쳐와!" (에러 핸들링 불필요, Assert 처리)
- **Post-condition 위반**: "내가 계산을 잘못했네, 코드를 수정할게!" (로직 결함)

### 3. DbC 실행 아키텍처 (ASCII)

```text
    [ Call Method ]
           │
    ┌──────▼──────┐
    │ Pre-condition│ ──(False)──▶ [ Throw Error: Client's Fault ]
    └──────┬──────┘
           │ (True)
    ┌──────▼──────┐
    │ Execute Logic│ <── Invariant must be true before & after
    └──────┬──────┘
           │
    ┌──────▼──────┐
    │Post-condition│ ──(False)──▶ [ Throw Error: Supplier's Fault ]
    └──────┬──────┘
           │ (True)
    [ Return Result ]
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 방어적 프로그래밍 vs DbC
- **방어적 프로그래밍**: "아무도 못 믿어!" 입력을 의심하고 예외 처리를 잔뜩 함 (코드가 비대해짐).
- **DbC**: "계약된 대로만 해!" 호출자가 약속을 지켰음을 신뢰하고, 지키지 않았을 때만 가차 없이 중단함 (코드가 간결해짐).

### 2. 기술적 시너지: 리스코프 치환 원칙 (LSP)
상속 관계에서 DbC는 LSP를 지키는 도구가 됩니다.
- 하위 클래스는 상위 클래스의 **선행 조건을 강화할 수 없음**. (더 까다롭게 굴면 안 됨)
- 하위 클래스는 상위 클래스의 **후행 조건을 약화할 수 없음**. (보장해주던 걸 안 해주면 안 됨)

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 스택(Stack) 클래스 설계
- **불변 조건**: `count >= 0 && count <= max_size`.
- **`push(item)` 메서드**:
    - 선행 조건: `!isFull()`. (가득 찼는데 넣으려 하면 호출자 잘못)
    - 후행 조건: `!isEmpty() && top() == item`. (넣었으면 꼭대기에 그 아이템이 있어야 함)
- **판단**: 복잡한 비즈니스 로직을 if-else로 체크하기보다, 계약 명세를 통해 클래스의 '안전 구역'을 설정함으로써 예외 상황에 대한 코딩 부담 최소화.

### 📢 기술사적 결언
> "DbC는 소프트웨어의 **'정합성(Correctness)'**을 담보하는 가장 철학적인 기법이다. 주석으로 "인자는 양수여야 합니다"라고 쓰는 대신, 코드로 계약을 명시하여 시스템이 스스로의 상태를 감시하게 해야 한다. 특히 대규모 분산 시스템이나 라이브러리 설계에서 DbC는 **'책임의 경계'**를 긋는 가장 강력한 도구다. 아키텍트는 '만약의 상황'을 대비하는 것을 넘어, '당연한 약속'이 지켜지는 구조를 설계해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **디버깅 비용**: 오류 발생 시 책임 소재 파악 시간 80% 단축.
- **문서화**: 별도의 문서 없이 코드만으로도 완벽한 사용법(API 규약) 파악 가능.

### 미래 전망
미래의 DbC는 **'자동 증명 도구'**와 결합할 것입니다. 에펠(Eiffel) 언어처럼 언어 차원에서 DbC를 지원하는 것을 넘어, Java나 C#에서도 컴파일러가 계약 위반 가능성을 미리 정적 분석하여 알려주는 기술이 보편화될 것입니다. 또한 마이크로서비스 간의 계약인 **'Consumer-Driven Contracts'**로 확장되어, 네트워크를 넘나드는 거대한 계약 기반 아키텍처가 완성될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **디자인 바이 컨트랙트**는 친구와 놀 때 **"약속 편지"**를 쓰는 것과 같아요.
2. "네가 장난감을 소중히 다루면(선행 조건), 나는 장난감을 빌려줄게(후행 조건)"라고 **미리 종이에 적어두는** 거죠.
3. 약속을 어기면 누구 잘못인지 금방 알 수 있어서, **싸우지 않고 오랫동안 사이좋게** 놀 수 있답니다!
