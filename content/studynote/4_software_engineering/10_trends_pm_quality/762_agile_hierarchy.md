# 762. 애자일 에픽, 스토리, 테마, 태스크 계층

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 방대하고 불확실한 프로젝트의 범위를 관리 가능한 단위로 쪼개어 불확실성을 해소하고, 가치 전달 속도를 최적화하기 위한 **애자일 백로그 최적화의 계층적 구조(Hierarchical Work Breakdown Structure)**이다.
> 2. **메커니즘**: 전략적 목표인 **테마(Theme)**에서 시작하여 큰 기능 묶음인 **에픽(Epic)**으로 분리되고, 이를 다시 사용자 가치 중심의 **사용자 스토리(User Story)**로 세분화하며, 최종적으로 개발자가 실행 가능한 구체적 행위인 **태스크(Task)**로 분해되는 하향식 접근(Top-Down Approach)을 따른다.
> 3. **가치**: 비즈니적 요구사항과 기술적 구현 사이의 **의미적 격차(Semantic Gap)**를 줄여주며, 프로젝트의 진척도를 가시화하고 범위 크리프(Scope Creep)를 방지하여 시장 출시 시간(Time-to-Market)을 단축시키는 핵심 거버넌스 도구이다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 정의**
애자일 Agile 프로젝트 관리에서 **계층(Hierarchy)**이란, 프로젝트의 목표를 달성하기 위해 수행해야 하는 모든 작업을 추상화 레벨에 따라 구조화한 체계를 말한다. 전통적인 폭포수 모델 Waterfall Model의 작업 분해 구조 WBS (Work Breakdown Structure)가 '작업 공정'이나 '산출물' 중심이라면, 애자일의 계층 구조는 **'사용자 가치'와 '비즈니스 목표' 중심**으로 설계된다는 것이 결정적인 차이점이다. 이 구조는 팀이 당장 무엇을 해야 할지(Task)를 명확히 하면서도, 그 작업이 왜 중요한지(Theme)를 잊지 않게 잇대어 주는 역할을 수행한다.

**2. 등장 배경: 불확실성의 관리**
소프트웨어 규모가 거대해지고 요구사항이 급변하는 환경에서, 모든 요구사항을 초기에 완벽하게 정의하는 것은 불가능하다. 따라서 "How to eat the elephant?"(코끼리를 어떻게 먹을 것인가?)라는 고난제를 해결하기 위해, 거대한 기능(Epic)을 여러 번의 반복 개발 Iteration을 통해 점진적으로 실현 가능한 작은 단위(Story)로 나누는 전략이 등장했다.

**3. 비유: 도시 건설 계획**
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                      애자일 계층 구조의 메타포                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [ Theme: "살기 좋은 혁신 도시 조성" ]          <--- 시장의 비전, 전략적 목표   │
│         │                                                                     │
│         ├── [ Epic: "대중교통 시스템 구축" ]       <--- 비용/시간이 큰 거대 프로젝트│
│         │         │                                                        │
│         │         ├── [ Story: "지하철 1호선 개통" ]  <--- 시민에게 제공하는 가치   │
│         │         │    (사용자: 시민, 목적: 빠른 이동)                          │
│         │         │         │                                              │
│         │         │         ├── [ Task: "철로铺设 시공" ]   <--- 기술적 실행 단위   │
│         │         │         ├── [ Task: "역사 건물 짓기" ]                     │
│         │         │         └── [ Task: "전력 공급 시스템 연결" ]                │
│         │         │                                                        │
│         │         └── [ Story: "버스 노선 재편" ]                                 │
│         │                                                                  │
│         └── [ Epic: "스마트 쓰레기 수거 시스템" ]                                │
│                                                                             │
│  → Theme는 'Why', Epic는 'What', Story는 'Who/What', Task는 'How'를 담당한다.   │
└─────────────────────────────────────────────────────────────────────────────┘
```

**4. 📢 섹션 요약 비유**
> 마치 거대한 정글을 개척할 때, 위성 지도(Theme)를 보고 전체 경로를 설정하고, 큰 강(Epic)을 건너기 위해 여러 개의 교각 스토리(Story)를 세우며, 그 교각을 놓기 위해 구체적으로 나무를 베고 못을 박는 태스크(Task)를 수행하는 것과 같습니다. 계층이 없다면 땅을 파는 노동자는 자신이 왜 이 일을 하는지(Theme)를 잃어버리고 방황하게 됩니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. 구성 요소 상세 분석**
애자일 계층 구조를 이루는 각 요소는 고유의 수명 주기와 관리 주체를 가진다.

| 요소 | 명칭 | 정의 및 범위 | 작성 주체 | 관리 단위 | 핵심 특징 |
|:---:|:---|:---|:---|:---|:---|
| **Lv 1** | **테마 (Theme)** | 조직의 전략적 목표나 비즈니스 가치를 달성하기 위한 일련의 에픽들의 집합. 여러 릴리즈에 걸쳐 추진됨. | PO / 경영진 | 분기/반기 단위 | **Strategic Alignment** |
| **Lv 2** | **에픽 (Epic)** | 하나의 스프린트 Sprint 내에서 완료하기 어렵고, 8~13 Story Point 이상의 규모를 가진 큰 기능 묶음. | PO | 릴리즈 단위 | **Uncertainty Container** |
| **Lv 3** | **사용자 스토리 (User Story)** | 사용자 관점에서 기능을 설명한 가치 중심의 요구사항. "누가(Who), 무엇을(What), 왜(Why)" 형식. | PO | 스프린트 단위 | **Negotiable Unit** |
| **Lv 4** | **태스크 (Task)** | 스토리를 구현하기 위한 구체적인 기술적 행위. 개발자/디자이너가 직접 수행하는 업무 단위. | 팀원 | 일/시간 단위 | **Executable Action** |

**2. 계층 분해 알고리즘 (Refinement Process)**
계층 구조의 핵심은 **Just-In-Time (JIT)** 정제 과정에 있다. 모든 것을 처음에 다 쪼개면 변경 비용이 폭증하고 맥락이 유실된다.

```text
         [ Backlog State Management Flow ]

         (Uncertain)                    (Refined)                     (Ready)
    ┌───────────────┐              ┌───────────────┐            ┌───────────────┐
    │   THEME       │              │     EPIC      │            │    STORY      │
    │ (Strategy)    │ ──Breakdown──│  (Initiative) │ ──Slicing──│  (Requirement)│
    └───────────────┘              └───────────────┘            └───────────────┘
         │                              │                            │
         │                              ▼                            ▼
         │                         * Rough Est.                * INVEST Check
         │                         * High Value?               * Tech Spec.
         │                                                        │
         │                                                        ▼
         │                                                 ┌───────────────┐
         └────────────────Refinement───────────────────────│    TASK       │
                  (Triggered: 2~3 Sprints before)          │  (Execution)  │
                                                           └───────────────┘
```

**3. 상세 동작 메커니즘**

*   **Theme Definition**: 조직의 목적 설정 (예: "이커머스 전환율 20% 증대").
*   **Epic Identification**: 목적을 달성할 핵심 기능 식별 (예: "앱 전체 리뉴얼").
*   **Story Splitting (Vertical Slicing)**: 에픽을 데이터베이스 계층, UI 계층 등으로 나누는 것이 아니라, **사용자가 완성된 기능을 경험할 수 있는 수직 조각**으로 나누는 것이 핵심이다.
    *   *잘못된 분할 (Horizontal)*: [UI 작업], [DB 작업] → 사용자는 기능을 테스트할 수 없음.
    *   *올바른 분할 (Vertical)*: [로그인 기능 완성] → UI, DB, API가 모두 포함된 배포 가능한 단위.
*   **Task Decomposition**: 스토리가 Sprint Backlog에 들어가기 직전, 팀원들이 참여하여 실제 개발 공수를 추정하기 위해 세부 작업으로 나눈다.

**4. 핵심 수식 및 규칙**
스토리 포인트 Story Point 추정 시 피보나치 수열을 활용하여 불확실성을 반영한다.
> **Sum of Story Points in Epic ≈ Cumulative Risk**
> 에픽 내의 스토리 포인트 합이 커질수록 리스크가 지수적으로 증가하므로, 보통 8~13점 이상은 반드시 쪼개야 한다고 권장한다.

**5. 📢 섹션 요약 비유**
> 마치 장인이 거대한 오토키메이 조각상(Epic)을 만들 때, 처음에는 대강의 형태를 잡고(Theme), 팔, 다리, 머리(Story) 순서로 덩어리를 쪼갠 뒤, 마지막에 정교한 끝손질(Task)을 하는 것과 같습니다. 처음부터 눈썹 하나하나(Task)를 다듬으려 하면 전체 균형(Epic)을 맞추기 어렵습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 계층 구조별 비교 분석**

| 구분 | WBS (Waterfall) | Agile Hierarchy |
|:---|:---|:---|
| **단위 분류** | 작업 공정 중심 (Task 단위) | 가치/기능 중심 (Story/Epic 단위) |
| **범위 관리** | 초기에 모든 계층을 정의 (BDUF) | 점진적 세분화 (Progressive Elaboration) |
| **변경 대응** | 하위 단위 변경 시 상위 재작업 필수 (연쇄 수정) | 하위 스토리 변경이 상위 에픽에 영향을 덜 줌 (모듈화됨) |
| **주요 지표** | 선행 작업 선후 관계 | 비즈니스 가치 및 우선순위 |

**2. 타 과목 융합 분석**

*   **Software Architecture (소프트웨어 아키텍처)**: **Microservices Architecture (MSA)**와의 상관관계가 깊다. 에픽은 한 개의 마이크로서비스 범위로 매핑되고, 스토리는 해당 서비스의 API 단위로 정의되는 경우가 많다. 잘못된 계층 구조는 "모놀리식 돼지" Monolithic Pig를 양산하게 된다.
*   **Database (데이터베이스)**: **Database Sharding (샤딩)**이나 **Normalization (정규화)** 개념과 유사하다. 데이터를 성격에 따라 쪼개는 것처럼, 작업을 독립적으로 수행 가능한 단위로 쪼개야 병렬 처리(개발)가 가능해진다.

**3. 시각적 비교: Scope Creep 방지 효과**

```text
[ Case A: 계층 구조 없이 Task만 관리 (Anti-Pattern) ]
      User Request: "검색 기능 추가"
          │
          ▼
    ┌─────────────────────┐
    │ Task: DB 테이블 설계 │ ─┐
    └─────────────────────┘  │
    ┌─────────────────────┐  │ (범위 불확실로 인한 무한 확장)
    │ Task: UI 화면 구현    │ ─┤
    └─────────────────────┘  │
    ┌─────────────────────┐  │
    │ Task: 검색 엔진 튜닝 │ ─┘
    └─────────────────────┘
    → "Task만 하다가 끝이 안 보입니다. 기능 전체의 완성 기준이 모호합니다."

[ Case B: 계층 구조를 활용한 관리 (Best Practice) ]
      User Request: "검색 기능 추가"
          │
          ▼
    ┌─────────────────────┐
    │ Story: 연관 검색어 10개 출력 │ ── [ Acceptance Criteria: "연관어 10개 뜨면 완료" ]
    └─────────────────────┘
          │
          ├── Task: 형태소 분석기 적용
          └── Task: Solr Config 수정
    → "Story라는 '가치의 완성단위'가 있어 Task의 범위를 제어합니다."
```

**4. 📢 섹션 요약 비유**
> 마치 쇼핑할 때 장바구니에 담는 목록(Theme)과, 진열대에서 물건을 집는 행위(Story), 그리고 계산대에서 바코드를 찍는 행위(Task)를 구분하는 것과 같습니다. 단순히 '바코드 찍기'만 반복하면 내가 왜 이 물건을 샀는지(목적)를 잊게 되지만, '선물하기 위해 물건을 고르고 계산한다'는 맥락(Context)이 있으면 추진력이 유지됩니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. 실무 시나리오: 신규 서비스 도입 의사결정**

*   **상황**: 핀테크 스타트업에서 "애니웨이 로그인(Biometric Auth)" 도입을 고려 중.
*   **문제**: 보안 규정과 기술적 이슈로 인해 범위가 불투명함.
*   **전략적 분해 (Practical Strategy)**:
    1.  **Theme**: "로그인 탈출률 30% 감소".
    2.  **Epic**: "생체 인증 통합 모듈 개발".
    3.  **Story 1**: "지문 인증으로 간편 로그인하기" (iOS).
    4.  **Story 2**: "얼굴 인식으로 간편 로그인하기" (Android).
    5.  **Task**: "LocalAuthentication Framework 연동", "Encrypted KeyChain Store 구현".
*   **결정**: 모든 플랫폼을 동시에 개발(Epic)하면 리스크가 크므로, iOS 지문 인증(Story 1)을 **MVP (Minimum Viable Product)**로 먼저 스프린트에 배정하여 기술적 검증을 완료 후 확장하기로 결정.

**2. 도입 체크
