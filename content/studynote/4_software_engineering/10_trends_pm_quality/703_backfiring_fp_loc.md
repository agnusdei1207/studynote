# 703. 백파이어링 FP LOC 역산

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기존에 구축된 시스템의 소스 코드 줄 수(**LOC**)를 기반으로, 해당 개발 언어의 평균적인 생산성 계수를 적용하여 역으로 **기능점수(FP)**를 산출하는 간이 비용 산정 기법이다.
> 2. **사용 목적**: 정규 기능점수 산정이 시간과 비용 측면에서 부담스러울 때, 과거 데이터를 활용하여 빠르게 소프트웨어의 규모를 추정하고 생산성을 비교하기 위해 사용한다.
> 3. **한계**: 개발 언어의 종류, 코딩 스타일, 도구 활용 능력에 따라 정확도가 크게 좌우되므로, 정밀한 계약 보다는 초기 타당성 검토나 사후 통계 분석용으로 적합하다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "FP 산정은 너무 어렵다"

정규 기능점수(Regular FP) 산정은 모든 문서와 설계서를 뒤져서 데이터 파일(ILF, EIF)과 트랜잭션(EI, EO, EQ)을 식별해야 하는 고단한 작업입니다. "이미 완성된 수백만 줄의 코드가 있는데, 이걸로 점수를 바로 알 수는 없을까?"라는 질문에 대한 답이 바로 **백파이어링(Backfiring)**입니다.

### 💡 비유: 건물 높이로 층수 맞추기

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        백파이어링(Backfiring) 비유                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 건물의 정확한 방 개수(기능점수)를 알고 싶음.                             │
│                                                                             │
│  1. 정규 산정: 건물 내부를 일일이 돌아다니며 방 개수를 직접 셈. (정확하지만 힘듦)  │
│                                                                             │
│  2. 백파이어링:                                                              │
│     건물 밖에서 키(LOC)를 잼. "보통 이 동네 건물(언어)은 3미터당 1층(FP)이니까,     │
│     30미터면 10층(10 FP) 정도 되겠군!"이라고 추측함.                           │
│                                                                             │
│  → 키(LOC)만 알면 대략적인 방 개수(FP)를 **거꾸로 계산**해낼 수 있는 지름길!      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 백파이어링 산출 공식
$$ \text{Estimated FP} = \frac{\text{Total LOC}}{\text{Average LOC per FP (Language Factor)}} $$

- **Total LOC**: 현재 시스템의 전체 소스 코드 줄 수.
- **Language Factor**: 특정 프로그래밍 언어에서 기능점수 1점을 구현하는 데 필요한 평균 코드 줄 수 (예: Java는 약 50~60, C는 약 120).

### 2. 주요 언어별 생산성 계수 (예시)

| 프로그래밍 언어 | 1 FP당 평균 LOC | 특징 |
|:---:|:---:|:---|
| **C** | 128 | 저수준 언어, 1점을 위해 많은 코드 필요 |
| **Java** | 53 | 객체지향, 중간 수준의 추상화 |
| **Python** | 40 | 고수준 언어, 적은 코드로 많은 기능 구현 |
| **SQL** | 21 | 선언적 언어, 데이터 처리에 매우 효율적 |

### 3. 백파이어링 분석 절차 (ASCII)

```text
    [ Target System ] ──▶ [ LOC Counting ] ──▶ [ Language Identification ]
    (소스코드 전수 조사)     (도구: CLOC 등)       (사용된 언어별 분류)
                                                       │
          ┌────────────────────────────────────────────┘
          ▼
    [ Apply Factor ] ──▶ [ Unadjusted FP ] ──▶ [ Adjustment ] ──▶ [ Final FP ]
    (언어별 계수 적용)      (단순 합산)          (VAF 적용)        (최종 규모)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 정규 FP vs 백파이어링 FP

| 항목 | 정규 기능점수 (IFPUG) | 백파이어링 (Backfiring) |
|:---:|:---|:---|
| **측정 기준** | 논리적 설계, 사용자 관점 | **물리적 코드 양 (LOC)** |
| **정확도** | 높음 (표준 방식) | 낮음 (경험적 추정) |
| **소요 시간** | 길음 (전문가 필요) | **매우 짧음 (자동화 가능)** |
| **언어 의존성** | 없음 | **매우 높음** |

### 2. 기술적 시너지: 생산성 분석 (Benchmarking)
백파이어링은 우리 팀의 실력을 측정하는 잣대가 됩니다. "표준 Java 프로젝트는 1FP당 53줄인데, 우리 팀은 40줄 만에 짰네?" → **팀의 역량이 높거나 재사용 모듈을 잘 활용하고 있음**을 정량적으로 증명할 수 있습니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 자산 실사 및 유지보수 단가 산정
- **상황**: 10년 된 레거시 시스템의 유지보수 재계약을 하려는데, 당시 설계서가 없어 규모를 모름.
- **결단**: **백파이어링 기법 도입**. 
- **판단**: 현재 돌아가고 있는 전체 소스코드를 도구로 스캔하여 50만 LOC 확인. 사용 언어(C#)의 계수(약 55)를 적용하여 약 9,000 FP로 추정.
- **효과**: 근거 없는 '부르는 게 값' 식의 계약이 아닌, 수치화된 규모를 바탕으로 합리적인 유지보수 인건비 산출.

### 📢 기술사적 결언
> "백파이어링은 **'차선(Second Best)'**의 지혜다. 가장 좋은 것은 설계 단계의 정규 FP 산정이지만, 현실은 늘 문서보다 코드가 앞서간다. 아키텍트는 백파이어링의 수치를 맹신하기보다, 이를 통해 시스템의 **'복잡도 경향성'**을 파악하고 향후 리팩토링이나 재공학의 우선순위를 정하는 **'전략적 나침반'**으로 활용해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **산정 비용 절감**: 정규 FP 대비 측정 비용 80% 이상 절감.
- **히스토리 복원**: 유실된 요구사항의 규모를 사후적으로 정량화 가능.

### 미래 전망
미래의 백파이어링은 **'AI 인텔리전트 스캔'**으로 진화할 것입니다. 단순히 줄 수를 세는 것이 아니라, AI가 코드의 논리적 의미를 파악하여 "이 구간은 중복 코드가 많으니 계수를 낮게 잡아야 한다"거나 "고도의 알고리즘 구간이므로 FP 가중치를 높여야 한다"라고 판단하는 지능형 규모 산정 도구가 등장할 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **백파이어링**은 친구가 만든 레고 성의 방이 몇 개인지 궁금할 때, 일일이 들어가 보지 않고 **"블록을 1,000개 썼으니까 방은 10개쯤 되겠네!"**라고 거꾸로 짐작하는 거예요.
2. 블록 하나가 큰지 작은지(언어 종류)에 따라 방 개수가 달라지니까, **어떤 블록을 썼는지** 먼저 확인해야 하죠.
3. 정확하지는 않아도, 아주 빨리 **"와! 진짜 큰 성(시스템)이구나!"**라는 걸 알 수 있는 편리한 방법이랍니다!
