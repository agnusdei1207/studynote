# 757. MoSCoW 요구사항 우선순위 판별

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 한정된 시간과 자원 내에서 최대의 가치를 창출하기 위해, 요구사항을 **Must, Should, Could, Won't**의 4가지 범주로 분류하여 개발 우선순위를 결정하는 기법이다.
> 2. **전략적 배분**: 반드시 필요한 핵심 기능(Must)에 집중하고, 부가적인 기능(Could)은 일정 지연 시 과감히 포기할 수 있는 유연한 범위(Scope) 관리의 핵심 도구이다.
> 3. **가치**: 이해관계자 간의 갈등을 조율하고 기대치를 관리하며, 비즈니스 관점에서 가장 시급하고 중요한 기능부터 시장에 출시(Time-to-Market)하게 돕는다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "다 중요하다고 말하는 고객"

요구사항 도출 단계에서 고객은 모든 기능이 "매우 중요하다"고 말합니다. 하지만 모든 것을 한꺼번에 개발할 수는 없습니다. **MoSCoW** 기법은 이러한 '중요성'의 나열을 '실행 가능성'과 '비즈니스 가치'라는 잣대로 엄격히 분류하여, 프로젝트가 산으로 가는 것을 막아줍니다.

### 💡 비유: 해외여행 짐 싸기

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        MoSCoW 우선순위 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 배낭 하나에 짐을 다 넣어야 함. (한정된 자원)                             │
│                                                                             │
│  1. Must have (여권, 지갑): 이거 없으면 비행기 못 탐. (절대적 필수)              │
│  2. Should have (카메라): 없으면 매우 아쉽지만, 일단 여행은 갈 수 있음.           │
│     (중요하지만 대체 가능하거나 다음 단계 가능)                                  │
│  3. Could have (간식, 책): 있으면 좋지만, 가방이 무거우면 제일 먼저 뺌.           │
│     (매력적이지만 포기 가능)                                                   │
│  4. Won't have (겨울 코트): 이번 여름 여행에는 절대 안 가져감.                   │
│     (이번엔 안 하기로 약속한 것)                                               │
│                                                                             │
│  → 배낭(스프린트)이 꽉 찼을 때, **무엇을 먼저 뺄지** 미리 정해두는 지혜!          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. MoSCoW 4대 분류 상세 기준

| 분류 | 영문 명칭 | 정의 | 비중 가이드 |
|:---:|:---|:---|:---:|
| **M** | **Must have** | 시스템 성공을 위한 **필수 불가결**한 요구사항 | 60% 이하 |
| **S** | **Should have** | 중요하지만 당장 없어도 운영은 가능한 사항 | 20% 내외 |
| **C** | **Could have** | 있으면 좋지만 일정에 따라 생략 가능한 사항 | 20% 내외 |
| **W** | **Won't have** | **이번 차수(Sprint)에는 구현하지 않기로** 합의함 | - |

### 2. 분류 프로세스 (Steps)
1. **List up**: 모든 요구사항을 나열.
2. **Consult**: 이해관계자들과 각 요구사항의 비즈니스 임팩트 논의.
3. **Categorize**: M, S, C, W 태그 부여.
4. **Balance Check**: 'Must' 비중이 너무 높지 않은지 확인 (보통 60% 유지 권장).

### 3. 우선순위 결정 매커니즘 (ASCII)

```text
    [ All Requirements ]
           │
    ┌──────▼────────────────────────────────┐
    │       Business Value Analysis         │ ◀── [ Stakeholder Consensus ]
    └──────┬────────────────┬───────────────┘
           │                │
    ┌──────▼──────┐  ┌──────▼─────────────┐
    │  Must Have  │  │   Should Have      │ <── Critical Path
    └──────┬──────┘  └──────┬─────────────┘
           │                │
    ┌──────▼──────┐  ┌──────▼─────────────┐
    │  Could Have │  │   Won't Have       │ <── Buffer / Future
    └─────────────┘  └────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. MoSCoW vs Kano 모델
- **MoSCoW**: **수행 가능성**과 **비즈니스 우선순위** 중심. (관리적 관점)
- **Kano**: 사용자의 **만족도**와 **품질 속성** 중심. (마케팅/심리 관점)
- **시너지**: Kano 모델로 '당연적 품질'을 찾아내어 MoSCoW의 'Must'로 지정합니다.

### 2. 기술적 시너지: 타임박싱 (Timeboxing)
애자일 스프린트에서 MoSCoW는 빛을 발합니다. 'Must'와 'Should'를 먼저 개발하고, 시간이 남으면 'Could'를 처리합니다. 만약 시간이 모자라면 'Could'를 자동으로 다음 스프린트로 이월(Won't for now)함으로써 **납기 준수**를 보장합니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 신규 쇼핑몰 MVP 출시
- **Must**: 상품 조회, 장바구니 담기, 결제 처리 (핵심 비즈니스 루프).
- **Should**: 회원 등급별 할인, 배송 추적 알림 (중요하지만 수동 처리 가능).
- **Could**: 찜한 상품 공유, 다크 모드 UI (편의 기능).
- **Won't**: AI 기반 상품 추천, 글로벌 결제 지원 (다음 고도화 과제).
- **판단**: 개발 도중 PG사 연동 이슈로 일정이 지연되자, 'Could' 항목인 다크 모드를 즉시 취소하고 'Must' 기능의 안정성 확보에 총력.

### 📢 기술사적 결언
> "우선순위 결정은 **'무엇을 할 것인가'**보다 **'무엇을 포기할 것인가'**를 결정하는 용기다. 'Must'가 100%인 프로젝트는 이미 실패한 프로젝트다. 아키텍트는 'Could'라는 **'스케줄 버퍼'**를 충분히 확보하여, 예상치 못한 기술적 난관이 닥쳤을 때 비즈니스 핵심 가치가 훼손되지 않도록 방어하는 완충 설계 능력을 보여주어야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **납기 준수율**: 유연한 범위 관리를 통해 프로젝트 완료 기한 준수율 90% 이상 확보.
- **ROI 최적화**: 고가치 기능(Must) 우선 개발로 초기 투자 대비 성과 조기 달성.

### 미래 전망
미래의 우선순위 판별은 **'AI 정밀 추천'**으로 고도화될 것입니다. AI가 유저의 실시간 피드백과 시장 트렌드 데이터를 분석하여, 현재 백로그 중 무엇이 'Must'가 되어야 하는지 데이터를 기반으로 제안해 주는 'Data-driven MoSCoW' 체계가 구축될 것입니다. 또한 개발자의 역량과 기술 부채 상황을 고려하여 실시간으로 범위를 조정하는 'Dynamic Scoping' 기술이 애자일 관리의 핵심이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **MoSCoW**는 마트에서 장을 볼 때 **"꼭 사야 할 것"**과 **"돈 남으면 살 것"**을 미리 정해두는 거예요.
2. 우유와 계란(Must)은 꼭 사야 하고, 과자(Could)는 돈이 모자라면 안 사도 괜찮다고 약속하는 거죠.
3. 이렇게 미리 정해두면, 마트에서 장바구니가 무거워져도 **제일 중요한 반찬 재료**는 잊지 않고 꼭 챙길 수 있답니다!
