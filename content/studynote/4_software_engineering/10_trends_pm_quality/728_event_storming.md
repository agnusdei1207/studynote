# 728. 이벤트 스토밍 도메인 이벤트 도출

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도메인 전문가와 개발자가 함께 모여 복잡한 비즈니스 도메인을 탐색하고, 시스템에서 발생하는 상태의 변화인 **도메인 이벤트(Domain Event)**를 중심으로 프로세스를 시각화하는 **워크숍 기반의 설계 기법**이다.
> 2. **방법론**: 화이트보드에 시간 순서대로 **오렌지색 포스트잇(이벤트)**을 붙여나가며, 명령(Command), 액터(Actor), 외부 시스템, 그리고 정책(Policy) 사이의 유기적 관계를 정의한다.
> 3. **가치**: 짧은 시간 내에 전사적 비즈니스 흐름을 정렬(Alignment)하고, 마이크로서비스(MSA)의 경계인 **바운디드 컨텍스트**를 식별하는 가장 강력한 실천적 도구이다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "기술 중심 모델링의 한계"

알베르토 브랜돌리니(Alberto Brandolini)가 제안한 **이벤트 스토밍 (Event Storming)**은 "소프트웨어의 복잡성은 도메인에 대한 이해 부족에서 온다"는 통찰에서 시작되었습니다. 전통적인 설계 회의가 데이터 구조(ERD)에 집중했다면, 이벤트 스토밍은 **'실제로 일어나는 일(Behavior)'**에 집중합니다. 형식을 파괴하고 거대한 벽면에 포스트잇을 붙이는 행위 자체가 소통의 마찰을 줄여줍니다.

### 💡 비유: 영화 필름의 프레임 나열하기

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        이벤트 스토밍 비유                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 영화(비즈니스)의 전체 내용을 빠르게 이해해야 함.                         │
│                                                                             │
│  1. 기존 방식 (대본 읽기):                                                   │
│     글자를 읽으며 상상함. (사람마다 이해가 다르고 지루함)                         │
│                                                                             │
│  2. 이벤트 스토밍 방식 (필름 나열):                                            │
│     - 중요한 장면(이벤트)을 사진 찍어 벽에 시간 순으로 쫙 붙임.                  │
│     - "주인공이 빵을 샀다(이벤트)" -> "왜? 배고파서(명령)"                       │
│     - 이 장면들을 보며 감독, 배우, 조명팀이 "아! 저기서 사고가 나겠네" 하고       │
│       한꺼번에 문제점과 흐름을 파악함.                                          │
│                                                                             │
│  → 복잡한 설명 대신 **결과물(포스트잇)**을 보며 전체 흐름을 맞추는 보물찾기!       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 포스트잇 색상별 의미 (The Grammar)

| 색상 | 요소 명칭 | 설명 | 비고 |
|:---:|:---|:---|:---|
| **오렌지** | **Domain Event** | 도메인에서 발생한 비즈니스적 사실 (과거형) | **가장 먼저 도출** |
| **파랑** | **Command** | 이벤트를 발생시키는 사용자의 의도나 명령 | - |
| **노랑** | **Aggregate** | 명령과 이벤트를 처리하는 상태의 묶음 (엔티티) | - |
| **라일락** | **Policy** | 이벤트 발생 시 수행되는 자동화된 규칙 (If-Then) | 리액티브 로직 |
| **분홍** | **External System** | 우리 시스템 밖의 제3자 시스템 (PG사, 메일 서버) | - |

### 2. 이벤트 스토밍 단계
1. **Chaotic Exploration**: 각자 알고 있는 도메인 이벤트를 무작위로 붙임.
2. **Timeline Alignment**: 중복 이벤트를 제거하고 시간 순서대로 정렬.
3. **Trigger & Rule Identification**: 이벤트를 유발하는 명령(Command)과 정책(Policy) 식별.
4. **Context Grouping**: 관련 있는 요소들을 묶어 바운디드 컨텍스트 도출.

### 3. 이벤트 스토밍 워크플로우 (ASCII)

```text
    [ Actor ] ──▶ [ Command (파랑) ] ──▶ [ Aggregate (노랑) ] ──▶ [ Event (오렌지) ]
                                                                        │
          ┌─────────────────────────────────────────────────────────────┘
          ▼
    [ Policy (라일락) ] ──▶ [ Command (파랑) ] ──▶ [ External System (분홍) ]
    ( If Event occurs )
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 이벤트 스토밍 vs 유스케이스 모델링
- **유스케이스**: 사용자와 시스템 간의 상호작용에 집중. '경계'가 명확함.
- **이벤트 스토밍**: 시스템 내부에서 일어나는 '사실'에 집중. '흐름'이 명확함.
- **시너지**: 이벤트 스토밍으로 전체 흐름을 잡고, 복잡한 구간을 유스케이스로 상세화합니다.

### 2. 기술적 시너지: 이벤트 기반 아키텍처 (EDA)
이벤트 스토밍의 산출물인 '도메인 이벤트'는 그대로 Kafka나 RabbitMQ의 **메시지 토픽(Topic)**이 됩니다. 분석 결과가 즉시 시스템 아키텍처의 설계서로 변환되는 놀라운 직결성을 보입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 복잡한 주문/배송 도메인 분석
- **상황**: 주문 취소, 반품, 교환 로직이 얽혀 있어 누구도 전체 흐름을 모름.
- **결단**: **전사 이벤트 스토밍 워크숍 개최**. 
- **판단**: 3시간 동안 전 직원이 참여하여 '주문됨', '결제됨', '배송 시작됨' 등 50개의 이벤트를 도출. 
- **결과**: "반품 신청 시 포인트 회수 로직이 누락되었다"는 치명적 정책 오류를 기획 단계에서 발견.
- **효과**: 개발 착수 전 비즈니스 룰을 완벽히 정합하고, 서비스 간의 메시지 규격 확정.

### 📢 기술사적 결언
> "이벤트 스토밍은 **'지식 습득의 가속기'**다. 코드나 문서보다 사람의 대화와 시각적 배치가 훨씬 더 많은 정보를 전달한다. 아키텍트는 포스트잇의 색깔을 맞추는 데 집착하기보다, 그 과정에서 발생하는 **'전문가들 사이의 격렬한 논쟁'**을 즐기고 그 안에서 도메인의 진실을 포착해야 한다. 모델링은 혼자 책상에서 하는 것이 아니라, 함께 벽면에서 하는 축제여야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **설계 속도**: 전통적 방식 대비 비즈니스 모델링 속도 5~10배 향상.
- **의사소통**: 이해관계자 간의 용어 불일치(Ambiguity) 90% 이상 제거.

### 미래 전망
미래의 이벤트 스토밍은 **'디지털 트윈 워크숍'**으로 진화합니다. 원격 근무 환경을 지원하는 미로(Miro)나 루시드차트와 결합하여, 실시간으로 붙인 포스트잇이 즉시 마이크로서비스의 코드로 변환되는 **'Low-code DDD'** 플랫폼이 보편화될 것입니다. 또한 AI가 워크숍 내용을 요약하고 누락된 예외 케이스(Edge Case)를 실시간 제안하는 지능형 설계 보조 시스템이 등장할 것입니다.
