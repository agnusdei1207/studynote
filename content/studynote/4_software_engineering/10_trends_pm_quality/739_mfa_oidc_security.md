# 739. MFA 인증 OIDC 인가 보안 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사용자의 신원을 다각도로 검증하는 **MFA(다요소 인증)**와, 표준화된 토큰 기반의 인증/인가 프레임워크인 **OIDC** 및 **OAuth2**를 결합하여 안전한 디지털 아이덴티티 체계를 구축하는 기술이다.
> 2. **기술 스택**: 지식(ID/PW), 소유(OTP), 존재(생체) 중 2가지 이상을 조합하는 **MFA**, 권한 위임을 다루는 **OAuth2**, 그리고 그 위에 인증 계층을 더한 **OIDC(OpenID Connect)**가 현대 웹/모바일 보안의 핵심이다.
> 3. **가치**: 패스워드 탈취 시나리오로부터 계정을 보호하고, 서비스 간의 안전한 정보 공유를 지원하여 사용자 편의성(SSO)과 시스템 보안성을 동시에 달성한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "패스워드는 이제 안전하지 않다"

공격자들은 크리덴셜 스터핑(Credential Stuffing)이나 피싱을 통해 패스워드를 쉽게 알아냅니다. 패스워드 하나만으로 모든 권한을 주는 방식은 더 이상 유효하지 않습니다. 이를 해결하기 위해 추가적인 인증 수단(MFA)을 도입하고, 개별 서비스가 패스워드를 직접 보관하지 않아도 되는 표준 인증 프로토콜(OIDC)이 필수 인프라가 되었습니다.

### 💡 비유: 금고 열기와 VIP 입장권

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        인증 및 인가 보안 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. MFA (금고 열기):                                                         │
│     비밀번호(지식)만 안다고 금고가 열리지 않음. 내 스마트폰(소유)으로 전송된        │
│     번호를 넣거나, 내 지문(존재)을 찍어야만 열림. **2개 이상의 열쇠**가 필요!      │
│                                                                             │
│  2. OAuth2 (VIP 입장권):                                                     │
│     호텔(서비스)에 갈 때, 내 신분증(ID/PW)을 보여주는 대신, 이미 검증된            │
│     여행사(인증 서버)에서 발행한 **입장권(Token)**만 보여주고 들어가는 것.        │
│                                                                             │
│  3. OIDC (입장권 위의 이름표):                                                │
│     입장권(OAuth2)에는 "입장 가능"만 적혀있지만, 그 위에 내 이름과 소속(Profile)   │
│     이 적힌 **이름표(ID Token)**를 덧붙여 내가 누구인지까지 알려주는 것.          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. MFA (Multi-Factor Authentication)의 3요소
- **Knowledge (지식)**: 본인만 알고 있는 것 (Password, PIN).
- **Possession (소유)**: 본인만 가지고 있는 것 (Smart Card, OTP, Mobile Device).
- **Inherence (존재)**: 본인의 신체적 특징 (Fingerprint, Iris, Face ID).

### 2. OIDC (OpenID Connect)와 OAuth2의 관계
- **OAuth 2.0**: **인가(Authorization)**를 위한 프레임워크. "이 앱이 내 사진첩에 접근해도 돼"라고 권한을 주는 기술.
- **OIDC**: OAuth 2.0 위에 구축된 **인증(Authentication)** 계층. "내가 누구인지" 증명하는 기술.
- **핵심 토큰**: **ID Token** (누구인가 - JWT 형식) + **Access Token** (무엇을 할 수 있는가).

### 3. 인증/인가 흐름도 (ASCII)

```text
    [ Client App ] ──(1) Login Request ──▶ [ Identity Provider (OIDC) ]
           ▲                                        │
           │ (4) Send Tokens (ID/Access)            │ (2) MFA Challenge
           │                                        ▼
    [ App Logic ] ◀─(5) Resource Access ── [ User Device / Bio ]
           │                                        │
           │ (6) Verify Token                       │ (3) Success Verify
           ▼                                        ▼
    [ Resource Server ] ◀───────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. SAML vs OIDC

| 항목 | SAML 2.0 | OIDC (OpenID Connect) |
|:---:|:---|:---|
| **데이터 형식** | XML 기반 | **JSON (JWT) 기반** |
| **적합 환경** | 기업 내 그룹웨어, 레거시 | **모바일 앱, 최신 웹 서비스, API 환경** |
| **복잡도** | 높음 | **낮음 (개발자 친화적)** |
| **전송 방식** | 주로 브라우저 포스트 | RESTful API 호출 |

### 2. 기술적 시너지: FIDO (Fast IDentity Online)
MFA의 '존재(생체)' 인증을 표준화한 기술입니다. OIDC 흐름 중에 FIDO2(WebAuthn)를 결합하면, 사용자는 패스워드 없이 지문 인식만으로 전 세계 모든 웹 서비스에 안전하게 로그인(Passwordless)할 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 글로벌 SaaS 서비스의 통합 로그인 구축
- **상황**: 구글, 페이스북 계정으로 로그인을 지원하면서, 사내 직원은 강력한 보안 적용 필요.
- **결단**: **OIDC 기반의 연합 인증(Federated ID) 및 조건부 MFA 적용**. 
- **판단**: 외부 계정 로그인은 표준 OIDC 프로토콜로 연동. 사내 직원은 관리자 페이지 접근 시에만 스마트폰 푸시 기반의 MFA를 강제하도록 정책 설정.
- **효과**: 일반 유저에게는 편리함을, 사내 관리자에게는 강력한 보안을 차별화하여 제공.

### 📢 기술사적 결언
> "인증은 **'문을 여는 행위'**이고, 인가는 **'방 안의 물건을 만지는 행위'**다. 이 둘을 명확히 구분하지 못하는 설계는 언젠가 권한 상승 공격에 무너진다. OIDC는 이 경계를 가장 현대적으로 정립한 표준이다. 아키텍트는 단순히 소셜 로그인을 붙이는 수준을 넘어, 토큰의 생명주기(TTL)와 **'토큰 탈취 시나리오'**에 대한 대응책(Revocation 등)을 반드시 설계에 포함해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **계정 탈취 방지**: MFA 도입 시 패스워드 기반 공격의 99% 이상 차단.
- **개발 생산성**: 표준 OIDC 연동으로 자체 인증 서버 구축 공수 70% 이상 절감.

### 미래 전망
미래의 인증은 **'분산 신원 증명 (DID, Decentralized ID)'**으로 진화할 것입니다. 중앙 서버(OIDC 공급자)를 믿는 대신, 블록체인에 내 신원 정보를 두고 필요한 조각만 선택적으로 증명(Verifiable Credential)하는 방식입니다. 또한 사용자의 위치, 시간, 행동 패턴을 AI가 학습하여 별도의 입력 없이도 신원을 검증하는 **'Continuous Authentication'** 기술이 보안의 새로운 표준이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **MFA**는 보물 상자를 열 때 비밀번호만 누르는 게 아니라, **내 지문까지 찍어야** 열리는 이중 잠금장치예요.
2. **OAuth2**는 놀이공원 자유이용권(토큰)과 같아서, 내 신분증을 안 보여줘도 **권한만 있으면** 모든 놀이기구를 탈 수 있게 해주죠.
3. **OIDC**는 그 자유이용권에 내 얼굴 사진(신원 정보)까지 붙여놓은 거라, **"내가 진짜 주인"**이라는 걸 더 확실히 알려주는 거랍니다!
