# 634. 구문, 분기, 조건 커버리지 포함 관계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어의 내부 논리(Logic)가 테스트 케이스(Test Case)에 의해 얼마나 실행되었는지를 정량화하는 **화이트박스 테스팅(White-box Testing)**의 핵심 지표이며, 결함 포획 능력을 계층별로 구분한다.
> 2. **구조적 포함 관계**: 구문(Statement) → 분기(Decision) 순서는 하위 호환성(포함 관계)을 가지나, 조건(Condition)과 분기(Decision)는 서로 독립적인 수학적 관계를 가지며, **MC/DC (Modified Condition/Decision Coverage)**가 이를 상호 보완하는 최상위 개념으로 자리 잡는다.
> 3. **가치**: DO-178C(항공), ISO 26262(자동차) 등 안전 관련 시스템에서 품질 보증의 객관적 수치로 활용되며, 미실행 코드(Dead Code)를 발견하고 리팩토링의 안전성을 확보하는 데 결정적인 역할을 수행한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의 및 필요성
**코드 커버리지(Code Coverage)**는 테스트 스위트(Test Suite)가 소스 코드의 얼마나 많은 부분을 실행했는지를 백분율로 나타낸 지표입니다. "테스트를 얼마나 해야 충분한가?"라는 질문에 **정량적(Quantitative)**인 기준을 제공합니다. 단순히 실행된 문장의 수를 세는 것을 넘어, 코드 내의 복잡한 분기점과 논리적 조건이 얼마나 검증되었는지를 파악함으로써 소프트웨어의 잠재적 결함을 조기에 발견할 수 있습니다.

### 💡 비유: 고속도로 톨게이트의 차선 검증
코드 실행 흐름을 차량이 달리는 **고속도로(Highway)**로 가정해 봅시다.
1.  **구문(Statement) 커버리지**: 도로의 모든 포장 구간(실행문)이 적어도 한 번은 밟혔는지 확인합니다. 단순히 지나가기만 하면 되므로, 차선 변경 없이 직진만 해도 통과할 수 있습니다.
2.  **분기(Decision) 커버리지**: 모든 **분기점(Intersection)**에서 좌회전과 우회전(True/False)을 모두 경험했는지 확인합니다. 단순히 지나가는 것을 넘어, 모든 가능한 방향으로 진입해야 합니다.
3.  **조건(Condition) 커버리지**: 복잡한 **나들목(Interchange)**의 진입 조건("차량 2대 이상이고, 요금이 지불되었을 때")을 구성하는 개별 요소들이 각각 참/거짓 상황을 모두 만족했는지 세밀하게 검증합니다.

### 📢 섹션 요약 비유
> 마치 복잡한 고속도로 톨게이트에서 하이패스 차선(고속 패스)을 별도로 운영하여 병목을 해결하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 커버리지 상세 분석 (5가지 구성 요소)
화이트박스 테스트의 커버리지는 단순함에서 엄격함으로 나아갑니다.

| 구분 요소 (Component) | 정의 (Definition) | 내부 동작 (Internal Logic) | 측정 프로토콜 (Metric) | 비유 (Analogy) |
|:---:|:---|:---|:---|:---|
| **구문 (Statement)** | 실행 가능한 명령문 | 디버깅 가능한 모든 라인이 최소 1회 수행됨 | `실행된 라인 수 / 전체 라인 수` | 모든 길을 밟고 지나감 |
| **분기 (Decision)** | 제어 구조의 결과 | `if`, `switch` 등의 전체 결과(True/False)가 모두 수행됨 | `수행된 분기 결과 수 / 전체 분기 결과 수` | 모든 갈림길의 방향 시도 |
| **조건 (Condition)** | 원자적(Boolean) 표현식 | `if (A && B)` 내부의 개별 A, B의 T/F가 모두 수행됨 | `수행된 조건 결과 수 / 전체 조건 결과 수` | 갈림길의 개별 신호등 확인 |
| **다중 조건 (Multiple)** | 모든 조건 조합 | 복합 조건 내 모든 가능한 T/F 조합을 시나리오로 수행 | `수행된 조합 수 / 전체 가능한 조합 수` | 완전한 경우의 수 탐색 |
| **MC/DC** | 수정 조건/결정 | 개별 조건이 독립적으로 전체 결과에 영향을 미침을 증명 | `독립 조건 충족 수 / 전체 조건 수` | 항공 안전성 검증 (DO-178B) |

### 2. 포함 관계의 아키텍처 (ASCII 다이어그램)
커버리지 간의 포함 관계는 단순한 상위 집합이 아닙니다. 아래 다이어그램과 같이 수학적 위계(Hierarchy)가 존재하며, **분기(Decision)와 조건(Condition)은 서로 교집합(Intersection) 관계**에 있습니다.

```text
   [ WHITE-BOX COVERAGE HIERARCHY ]
   
   ┌────────────────────────────────────────────────────────────────────────┐
   │                          Path Coverage                                 │
   │  (경로 커버리지: 모든 실행 가능한 경로 순회 - 비용 최상위)                 │
   │   ┌──────────────────────────────────────────────────────────────┐    │
   │   │         MC/DC (Modified Condition/Decision Coverage)          │    │
   │   │  (개별 조건이 전체 결과에 독립적으로 영향을 미침을 보장)            │    │
   │   │   ┌──────────────────────────────────────────────────────┐    │    │
   │   │   │              Condition Coverage                      │    │    │
   │   │   │          (조건 커버리지: 개별 피연산자 T/F)              │    │    │
   │   │   │        ┌─────────────────────────────────────┐      │    │    │
   │   │   │        │    Decision Coverage (분기 커버리지)      │      │    │    │
   │   │   │        │      (전체 조건식 결과 True/False)     │      │    │    │
   │   │   │        │      ┌─────────────────────────────┐  │      │    │    │
   │   │   │        │      │    Statement Coverage        │  │      │    │    │
   │   │   │        │      │     (구문 커버리지)             │  │      │    │    │
   │   │   │        │      └─────────────────────────────┘  │      │    │    │
   │   │   │        └─────────────────────────────────────┘      │    │    │
   │   │   └──────────────────────────────────────────────────────┘    │    │
   │   └──────────────────────────────────────────────────────────────┘    │
   └────────────────────────────────────────────────────────────────────────┘

   [ 유의점: Decision(분기)과 Condition(조건)의 상관관계 ]
   
   1. Decision ⊃ Statement (항상 참) : 모든 분기를 실행하면 구문은 무조건 실행됨.
   2. Decision ⊄ Condition (부분 집합 아님) : 분기를 다 만족해도 조건을 다 만족한 것 아님.
   3. Condition ⊄ Decision (부분 집합 아님) : 조건을 다 만족해도 분기(T/F)가 쏠릴 수 있음.
```

### 3. 심층 분석: 분기와 조건의 비포함 관계 증명 (Deep Dive)
`if (A > 0 || B > 0)` 로직을 통해 두 커버리지가 서로를 포함하지 않음을 코드와 수식으로 증명합니다.

**[시나리오 A: 조건 100% ≠ 분기 100%]**
조건 커버리지를 만족하면서 분기 커버리지를 만족시키지 못하는 경우입니다.
*   **테스트 케이스 1**: `A = 1, B = -1` → (참 || 거짓) → 전체 **True**
*   **테스트 케이스 2**: `A = -1, B = 1` → (거짓 || 참) → 전체 **True**
*   **분석**:
    *   **Condition**: `A`는 T/F를 다 가짐(OK), `B`는 T/F를 다 가짐(OK). **100%** 달성.
    *   **Decision**: 전체 식이 `True`인 경우만 존재함. `False`인 경로가 누락됨. **50%** 달성.
*   **결론**: 조건 커버리지가 100%라도 분기 커버리지는 부족할 수 있다.

**[시나리오 B: 분기 100% ≠ 조건 100%]**
분기 커버리지를 만족하면서 조건 커버리지를 만족시키지 못하는 경우입니다.
*   **테스트 케이스 1**: `A = 1, B = 1` → (참 || 참) → 전체 **True**
*   **테스트 케이스 2**: `A = -1, B = -1` → (거짓 || 거짓) → 전체 **False**
*   **분석**:
    *   **Decision**: 전체 식이 T, F 모두 실행됨. **100%** 달성.
    *   **Condition**: `A`는 T/F 다 가짐(OK), 하지만 `B`는 항상 참이거나 거짓일 때만 존재했으므로, B의 독립적인 T/F 검증이 이루어지지 않음(OR 연산 특성상 A가 참이면 B를 확인 안 함). **결함 우려**.
*   **결론**: 분기 커버리지가 100%라도 조건 커버리지는 부족할 수 있다.

### 📢 섹션 요약 비유
> 마치 건물의 내부 설계 도면에서 구조체(뼈대)가 모두 세워졌다고 해서 전기 배선(세부 논리)까지 모두 연결된 것은 아닌 것과 같습니다. 구문은 기둥을 세운 것이고, 조건/분기는 그 안의 복잡한 배선 하나하나를 테스트하는 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 커버리지 유형별 정량적·구조적 비교 분석표

| 비교 항목 | 구문 (Statement) | 분기 (Decision) | 조건 (Condition) | MC/DC (Safety Critical) |
|:---|:---|:---|:---|:---|
| **측정 단위** | 실행 라인 (Line) | 분기 노드 (Node) | 조건식 (Predicate) | 독립적 조건 영향 (Independence) |
| **목표 수준(%)** | 일반 상용: 80% | 금융/네트워크: 90%+ | 핵심 로직: 100% | 항공/원전: 100% (DO-178C) |
| **결함 발견력** | 낮음 (Low) | 중간 (Medium) | 중간 (Medium) | 매우 높음 (High) |
| **테스트 공수** | 적음 | 중간 | 중간 | 매우 많음 (Exponential) |
| **복잡도 계수** | 1 (기준) | 2~4배 | 4~8배 | Varies (조건 수에 비례) |
| **보완 대상** | Dead Code 식별 | 로직 흐름 누락 방지 | Short-circuit 회로 검증 | 우주 항공 및 보안 시스템 검증 |

### 2. 타 과목 융합 관점 (Software Engineering ↔ Computer Architecture)

**[데이터 흐름 및 최적화와의 시너지]**
*   **컴파일러 최적화 (Compiler Optimization)**: 컴파일러는 **Dead Code Elimination (죽은 코드 제거)** 최적화를 수행합니다. 이때 '구문 커버리지'가 0%인 코드 블록은 컴파일러에 의해 삭제될 수 있는 후보입니다. 반면, '조건 커버리지'를 분석하면 **Branch Prediction (분기 예측)** 실패로 인한 성능 저하 구간을 식별하는 데도 활용할 수 있습니다.
*   **하드웨어 설계 검증 (Hardware Verification)**: 소프트웨어의 커버리지 개념은 Verilog/VHDL로 작성된 하드웨어의 시뮬레이션 검증으로 확장됩니다. 회로의 모든 **Latch(래치)**와 **Flip-flop**이 토글되는지 확인하는 **Toggle Coverage**는 하드웨어의 '구문 커버리지'와 유사하며, FSM(Finite State Machine)의 상태 전이를 검증하는 것은 '분기 커버리지'와 대응됩니다.

### 📢 섹션 요약 비유
> 자동차 엔진 테스트에서 엔진이 작동하냐(구문)를 넘어, 냉각수 온도에 따라 팬이 돌아가는 논리(조건/분기)가 제대로 작동하는지 확인하는 것과 같습니다. 엔진이 돌아가도 냉각 로직이 죽어 있으면 과열되어 폭발하기 때문입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오별 의사결정 프로세스 (Decision Matrix)

**[Case A: 일반 웹 서비스 백엔드 개발]**
*   **상황**: Agile 방법론, 빈번한 릴리즈, 낮은 위험도.
*   **전략**: **구문 커버리지(Line Coverage) 80%**를 기준으로 설정.
*   **이유**: 비즈니스 로직 변경이 잦은 상황에서 MC/DC 등 고비용의 테스트를 적용하면 개발 속도가 저해됨. 단, **핵심 결제 모듈** 등에는 별도로 **분기 커버리지 100%**를 적용하는 혼합 전략(Hybrid Strategy) 사용.

**[Case B: 금융권 핵심 거래 시스템 (코어 뱅킹) ]**
*   **상황**: 대규모 트래잭션, 데이터 정합성 생명, RTO(복구 시간 목표) 0분 요구.
*   **전략**: **분기(Decision) 커버리지 100%** 필수. 복잡한 이자 계산 로직에는 **조건(Condition) 커버리지** 병행.
*   **이유**: `if`문의 한쪽分枝가 미실행되어 발생하는 잔액 계산 오류는 용납될 수 없음. 정적 분석 도구(SonarQube, Coverit
