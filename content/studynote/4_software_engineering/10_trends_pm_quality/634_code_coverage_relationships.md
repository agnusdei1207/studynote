# 634. 구문, 분기, 조건 커버리지 포함 관계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소스코드의 내부 논리 구조가 테스트 케이스에 의해 얼마나 실행되었는지를 정량적으로 측정하는 **화이트박스 테스팅(White-box Testing)**의 핵심 지표이다.
> 2. **계층 구조**: 구문(Statement) < 분기(Decision) < 조건(Condition) 커버리지 순으로 엄격함이 더해지며, 상위 커버리지를 만족한다고 해서 반드시 하위나 동급의 다른 커버리지를 100% 보장하지는 않는 복잡한 포함 관계를 가진다.
> 3. **가치**: 미실행 코드(Dead Code)를 식별하고 테스트의 충분성을 객관적으로 증명하여, 소프트웨어의 신뢰성을 보장하는 품질 척도로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

### 코드 커버리지의 필요성

"테스트를 얼마나 해야 충분한가?"라는 질문에 답하기 위해 도입된 개념입니다. 블랙박스 테스트가 명세 충족 여부를 본다면, 화이트박스 테스트의 커버리지는 소스코드의 구석구석을 모두 실행해 보았는지를 확인합니다. 특히 금융, 항공, 의료 등 고신뢰성이 요구되는 분야에서는 특정 수준 이상의 커버리지 달성이 법적/기술적 필수 요건입니다.

### 💡 비유: 미로 탐험과 지도 그리기

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                       코드 커버리지 비유                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 복잡한 미로(코드)를 탐험하여 보물을 찾고 지도를 완성해야 함.              │
│                                                                             │
│  1. 구문 커버리지: 미로의 모든 복도(문장)를 한 번씩은 걸어봤는가?               │
│  2. 분기 커버리지: 미로의 모든 갈림길(If/Else)에서 왼쪽과 오른쪽을 다 가봤는가?  │
│  3. 조건 커버리지: 갈림길의 표지판에 써진 여러 조건(A이고 B일 때)들을           │
│     하나하나(A만 참일 때, B만 참일 때 등) 다 따져봤는가?                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 커버리지 유형 정의

| 커버리지 유형 | 핵심 측정 대상 | 설명 |
|:---:|:---|:---|
| **구문 (Statement)** | 모든 실행 문장 | 프로그램 내의 모든 개별 문장이 최소 한 번 실행되었는가? |
| **분기 (Decision)** | 전체 조건식 (True/False) | `if (A > 0 && B > 0)` 전체 결과가 참인 경우와 거짓인 경우를 모두 수행했는가? |
| **조건 (Condition)** | 개별 조건식 (True/False) | `if (A > 0 && B > 0)`에서 `A > 0`의 T/F, `B > 0`의 T/F를 각각 수행했는가? |

### 2. 포함 관계 및 제약 사항

흔히 "상위 커버리지가 하위를 포함한다"고 생각하기 쉽지만, **조건 커버리지와 분기 커버리지는 서로를 완벽히 포함하지 않습니다.**

```text
    [ 커버리지 간의 관계도 ]
    
    ┌──────────────────────────────────────────┐
    │          다중 조건 커버리지 (Multiple)      │
    │   ┌──────────────────────────────────┐   │
    │   │      MC/DC (수정 조건/결정)        │   │
    │   │   ┌──────────────┐┌──────────────┐│   │
    │   │   │  분기(Decision)││  조건(Condition)││   │
    │   │   └──────────────┘└──────────────┘│   │
    │   └──────────────────────────────────┘   │
    │            ┌────────────────────┐        │
    │            │     구문(Statement) │        │
    │            └────────────────────┘        │
    └──────────────────────────────────────────┘
```

- **분기 100% → 구문 100% (O)**: 모든 분기를 다 타면 그 안의 문장은 당연히 실행됩니다.
- **조건 100% → 분기 100% (X)**: 개별 조건은 만족해도 전체 결과(Decision)가 한쪽 방향(예: 항상 True)으로만 쏠릴 수 있습니다.
- **분기 100% → 조건 100% (X)**: 전체 결과는 T/F를 다 했어도, 내부의 특정 개별 조건은 한쪽 값만 가졌을 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 예시 코드를 통한 분석
`if (A > 0 || B > 0) { action(); }`

- **Case 1: (A=1, B=1)** → 전체 True, action() 실행. **구문 100% 달성.**
- **Case 2: (A=1, B=1), (A=-1, B=-1)** → 전체 T/F 수행. **분기 100% 달성.**
- **Case 3: (A=1, B=-1), (A=-1, B=1)** → A의 T/F, B의 T/F 수행. **조건 100% 달성.** (하지만 전체 결과는 두 경우 모두 True이므로 **분기 커버리지는 50%**에 불과함!)

### 2. 고수준 커버리지와의 연결
- **MC/DC (Modified Condition/Decision Coverage)**: 조건과 분기의 단점을 상쇄하기 위해, 각 개별 조건이 전체 결과에 독립적으로 영향을 미침을 증명하는 기법 (항공 소프트웨어 표준 DO-178C 필수).

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 가이드라인
1. **일반 비즈니스 로직**: **분기 커버리지 80~90%**를 목표로 설정하여 가성비를 높임.
2. **핵심 알고리즘/보안 로직**: **조건 커버리지** 및 **경로 커버리지**를 병행하여 논리적 빈틈을 제거.
3. **미사용 코드 제거**: 커버리지 도구(Jacoco, Istanbul 등)를 통해 0%인 구간을 찾아 불필요한 코드를 정리.

### 📢 기술사적 결언
> "커버리지 수치 그 자체보다 중요한 것은 **'테스트되지 않은 영역'의 의미를 해석하는 능력**이다. 100%라는 숫자에 집착하기보다, 복잡한 논리 구조에서 발생할 수 있는 잠재적 결함을 잡기 위해 어떤 수준의 커버리지가 적절한지 판단하는 아키텍처적 통찰이 요구된다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **결함 발견율**: 구문 대비 분기 커버리지 적용 시 논리 결함 발견 확률 약 20% 향상.
- **유지보수 안정성**: 리팩토링 시 기존 로직의 실행 여부를 보장하여 사이드 이펙트 방지.

### 미래 전망
최근에는 정적 분석과 동적 테스팅이 결합되어, 코드가 수정될 때마다 영향을 받는 최소한의 테스트 세트만 골라 커버리지를 계산하는 **'지능형 커버리지 분석'**이 CI/CD 파이프라인에 통합되고 있습니다. 또한 AI가 커버리지가 낮은 구간을 분석하여 테스트 케이스를 자동 생성해 주는 기술이 상용화되고 있습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **구문 커버리지**는 일기장의 모든 글자를 다 읽어봤는지 확인하는 거예요.
2. **분기 커버리지**는 "만약 비가 오면~" 할 때 비가 올 때와 안 올 때의 이야기를 다 읽어봤는지 보는 거죠.
3. **조건 커버리지**는 "비도 오고 바람도 불면~" 할 때 비만 올 때, 바람만 불 때를 꼼꼼하게 다 따져봤는지 확인하는 가장 깐깐한 검사랍니다!
