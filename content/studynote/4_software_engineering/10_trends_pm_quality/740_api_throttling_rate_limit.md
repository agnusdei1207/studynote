# 740. API 스로틀링 Rate Limit DDoS 방어

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템이 처리할 수 있는 임계치를 넘어서는 과도한 API 요청을 제한하거나 지연시킴으로써, 서버 자원 고갈을 방지하고 전체 서비스의 가용성을 유지하는 **트래픽 제어 및 보호 기술**이다.
> 2. **제어 기법**: 특정 시간 동안의 총 요청 수를 제한하는 **Rate Limiting**과, 시스템 부하에 따라 요청 처리 속도를 조절하는 **Throttling**을 통해 정상적인 유저의 응답성을 보장한다.
> 3. **가치**: 악의적인 DDoS 공격이나 비정상적인 봇(Bot) 트래픽을 입구에서 차단하고, 공유 자원을 사용하는 멀티 테넌트 환경에서 특정 사용자의 독점을 방지하여 공정한 자원 배분을 실현한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "무제한 요청의 위험성"

인터넷에 공개된 API는 누구나 호출할 수 있습니다. 만약 한 명의 유저(혹은 공격자)가 초당 수만 번의 요청을 보낸다면 서버는 금방 마비됩니다. 이는 정당한 사용자들의 서비스 이용을 방해하는 결과로 이어집니다. 시스템을 보호하기 위한 '방어선'이 필요한데, 이것이 바로 **Rate Limiting**과 **Throttling**입니다.

### 💡 비유: 맛집의 번호표와 입장 제한

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        트래픽 제어(Rate Limit) 비유                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 아주 유명한 맛집(API 서버)이 있음.                                      │
│                                                                             │
│  1. Rate Limiting (입장권 판매 제한):                                         │
│     "한 사람당 하루에 입장권은 3장까지만 살 수 있습니다."                         │
│     - 특정 유저가 가게를 독점하는 것을 방지함.                                   │
│                                                                             │
│  2. Throttling (입장 속도 조절):                                              │
│     "가게 안이 꽉 찼으니, 5분에 10명씩만 입장시키겠습니다."                       │
│     - 주방(서버)이 감당할 수 있는 속도로 손님을 받는 것.                         │
│                                                                             │
│  → 손님이 아무리 몰려와도 **가게가 무너지지 않게** 순서대로 조절하는 기술!         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 트래픽 제어 기법 비교

| 항목 | Rate Limiting (비율 제한) | Throttling (스로틀링) |
|:---:|:---|:---|
| **핵심 목적** | 자원 오남용 방지, 공정 사용 | 시스템 과부하 보호, 안정성 유지 |
| **제한 기준** | 특정 기간 내 **총 요청 횟수** | 시스템 **처리 용량(TPS)** 및 부하 |
| **동작 방식** | 임계치 초과 시 즉시 차단 (429 에러) | 처리 속도를 늦추거나 큐에 대기시킴 |
| **적용 단위** | 유저 ID, IP 주소, API 키 | 전체 시스템, 개별 서버 노드 |

### 2. 대표적인 알고리즘 (Algorithms)
- **Token Bucket**: 양동이에 토큰을 일정 속도로 채우고, 요청 시 토큰을 소모. 버스트(Burst) 트래픽 허용 가능.
- **Leaky Bucket**: 깔대기에 물을 붓듯, 요청이 아무리 몰려도 일정한 속도로만 처리.
- **Fixed Window**: 정해진 시간(예: 1분) 단위로 횟수 카운팅. 경계 지점 트래픽 폭주 리스크 있음.
- **Sliding Window**: 고정 윈도우의 단점을 보완하여 시간에 따라 연속적으로 횟수 계산.

### 3. API 보호 아키텍처 (ASCII)

```text
    [ Client Requests ] ──▶ [ API Gateway / WAF ] ──▶ [ Backend Server ]
                                     │
          ┌──────────────────────────┴──────────────────────────┐
          ▼ (Check Policy)                                      ▼
    [ Rate Limiter ] ◀──(Lookup)──▶ [ In-memory Store ]   [ Resource Monitoring ]
    (Token Bucket)                  (Redis: Usage Data)    (CPU/MEM Load)
          │                                                     │
          └───────────▶ [ Decision ] ◀──────────────────────────┘
                           │
          ┌────────────────┴────────────────┐
          ▼ (Success)                       ▼ (Reject / Delay)
    [ Forward to App ]              [ HTTP 429 / 503 ]
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 기술적 시너지: API 게이트웨이
Rate Limiting 로직을 비즈니스 서버에 넣으면 서버가 계산하느라 더 바빠집니다. 따라서 **API 게이트웨이 (Kong, AWS API Gateway 등)**나 **사이드카 프록시(Envoy)** 레벨에서 트래픽을 먼저 걸러주는 것이 효율적입니다.

### 2. 보안 융합: DDoS 방어
Rate Limiting은 서비스 거부 공격(DoS)을 막는 최전방 방어선입니다. 지능형 **WAF (Web Application Firewall)**와 결합하여, 비정상적인 패턴을 보이는 IP를 실시간으로 블랙리스트에 올리고 트래픽을 차단하는 유기적 방어 체계를 구축합니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 오픈 API 서비스의 안정성 확보
- **상황**: 무료 유저들이 무분별한 크롤링으로 서버 부하를 일으켜 유료 유저 서비스가 느려짐.
- **결단**: **계층별 차등 Rate Limiting 적용**. 
- **판단**: 
    1. **무료 등급**: 분당 60회 제한 (1회/초).
    2. **유료 등급**: 분당 1,000회 제한 및 버스트 허용.
    3. **알고리즘**: Token Bucket을 사용하여 일시적인 트래픽 급증은 수용하되 평균 속도 통제.
- **효과**: 유료 고객의 SLA를 보장하고, 서버 인프라 비용을 예측 가능한 범위 내에서 관리.

### 📢 기술사적 결언
> "트래픽 제어는 **'시스템의 생존 본능'**을 코드로 구현하는 것이다. Rate Limit이 없는 API는 성문이 열린 성과 같아 언제든 함락될 수 있다. 아키텍트는 단순히 에러를 내뱉는 것을 넘어, **'우아한 실패(Graceful Failure)'**를 설계해야 한다. `Retry-After` 헤더를 통해 클라이언트에게 언제 다시 요청할지 알려주는 배려가 시스템의 품격과 완성도를 결정한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **가용성 향상**: 트래픽 폭증 시 서비스 중단 시간 90% 이상 감소.
- **비용 최적화**: 불필요한 봇 트래픽 차단으로 인프라 자원 낭비 제거.

### 미래 전망
미래의 트래픽 제어는 **'AI 기반 적응형 스로틀링 (Adaptive Throttling)'**으로 발전할 것입니다. 고정된 수치가 아니라, 현재 서버의 CPU 상태와 네트워크 지연 시간을 AI가 실시간 분석하여 최적의 임계치를 동적으로 조절하는 방식입니다. 또한 사용자의 '의도'를 파악하여, 정상적인 사용 패턴은 열어주고 공격성 패턴만 핀셋으로 도려내는 지능형 방어 체계가 표준이 될 것입니다.
