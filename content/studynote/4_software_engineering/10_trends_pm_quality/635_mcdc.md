# 635. MC/DC 항공/자동차 안전 표준 조건

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 각 개별 조건(Condition)이 다른 조건에 영향을 받지 않고 전체 결정(Decision) 결과에 **독립적으로 영향을 미침**을 증명하는 고수준 화이트박스 테스트 커버리지다.
> 2. **가치**: 다중 조건 커버리지($2^N$)의 기하급수적 테스트 케이스 증가 문제를 해결하면서도($N+1$), 실질적인 논리 결함을 완벽에 가깝게 찾아내는 **안전 필수 시스템(Safety-Critical System)**의 핵심 표준이다.
> 3. **표준**: 항공기 소프트웨어 보안 표준인 **DO-178C (Level A)** 및 자동차 기능 안전 표준인 **ISO 26262 (ASIL D)**에서 최고 수준의 안전성을 보장하기 위해 필수로 요구한다.

---

## Ⅰ. 개요 (Context & Background)

### 등장 배경: 효율과 안전의 트레이드오프

복잡한 논리식 `if (A || B || C)`를 완벽히 테스트하려면 $2^3 = 8$가지 경우가 필요합니다. 변수가 20개면 $2^{20} = 1,048,576$개의 케이스가 필요하여 현실적으로 불가능합니다. **MC/DC (Modified Condition/Decision Coverage)**는 "각 변수가 정말로 결과에 영향을 주는가?"만 확인하여 테스트 횟수를 획기적으로 줄이면서도(변수 개수 + 1), 다중 조건 커버리지에 준하는 안전성을 확보하기 위해 고안되었습니다.

### 💡 비유: 범인 찾기와 알리바이 증명

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        MC/DC 독립 영향 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 세 명의 용의자(A, B, C) 중 누가 범죄(결과)를 저질렀는지 확인해야 함.      │
│                                                                             │
│  1. 다중 조건: 모든 사람이 있고/없고의 모든 조합(8가지)을 다 살펴봄. (비효율)   │
│                                                                             │
│  2. MC/DC:                                                                  │
│     - B와 C가 똑같이 있을 때, A만 '있다'에서 '없다'로 바꿨더니                │
│       결과(범죄 발생 여부)가 바뀐다면?                                       │
│     - 아하! A가 결과에 "독립적으로 영향"을 주었음을 확실히 알 수 있음!          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. MC/DC 만족 조건 (4대 원칙)
MC/DC를 달성했다고 말하려면 다음 네 가지를 모두 충족해야 합니다.
1. 모든 구문(Statement)이 최소 한 번 실행됨.
2. 모든 분기(Decision) 결과(T/F)가 최소 한 번 나타남.
3. 분기 내의 각 개별 조건(Condition)이 T/F를 최소 한 번 가짐.
4. **핵심**: 각 개별 조건이 **다른 조건의 변화 없이** 전체 결정 결과를 변경시킴을 증명함.

### 2. 케이스 선정 원리: `if (A || B)` 예시

| No | A | B | Result (A \|\| B) | 비고 |
|:---:|:---:|:---:|:---:|:---|
| 1 | **F** | F | **F** | 1번과 2번 비교 시: B는 고정, A가 변할 때 결과 변함 → **A 독립성 증명** |
| 2 | **T** | F | **T** | 1번과 3번 비교 시: A는 고정, B가 변할 때 결과 변함 → **B 독립성 증명** |
| 3 | F | **T** | **T** | |
| 4 | T | T | T | (불필요한 중복 케이스) |

- **결과**: 1, 2, 3번 케이스만으로 MC/DC 100% 달성 (총 N+1개).

### 3. 커버리지 강도 비교

```text
    [ 강도: 낮음 ]                                      [ 강도: 높음 ]
    구문(SC) < 분기(DC) < 조건(CC) < MC/DC < 다중 조건(MCC)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 안전 표준과의 정합성
- **항공 (DO-178C)**: 소프트웨어 등급 중 가장 위험한 **Level A**에서 반드시 MC/DC 100%를 요구합니다.
- **자동차 (ISO 26262)**: 위험 등급인 **ASIL D**에서 강력히 권고(Highly Recommended)됩니다.
- **철도 (EN 50128)**: 최고 안전 등급인 **SIL 4**에서 사용됩니다.

### 2. 컴파일러 최적화와의 관계
현대 프로그래밍 언어의 **Short-circuit Evaluation** (`&&`에서 앞이 False면 뒤를 안 봄) 특성 때문에, 실제 실행 경로가 MC/DC 설계와 다를 수 있어 도구(Tool)를 통한 정밀한 측정이 필수적입니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 자율주행 긴급 제동 시스템(AEB)
- **논리**: `if (장애물_감지 && 속도_임계치_초과 && !운전자_개입)`
- **결단**: 단순 분기 테스트로는 "운전자 개입" 조건이 결과에 제대로 영향을 주는지(결과를 뒤집는지) 확인할 수 없음. MC/DC를 적용하여 세 가지 조건 각각이 독립적으로 제동 여부를 결정할 수 있음을 입증해야 오작동을 막을 수 있음.

### 📢 기술사적 결언
> "MC/DC는 단순히 테스트를 많이 하는 것이 아니라, **'의미 있는 테스트'를 골라내는 수학적 통찰**이다. 고신뢰 시스템 설계자라면 복잡한 논리를 단순화하려 노력하되, 남겨진 논리에 대해서는 MC/DC 수준의 엄격한 검증을 통해 '증명된 안전'을 제공해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **테스트 효율성**: 변수가 많을수록 MCC 대비 케이스 수를 지수함수적으로 줄여 비용 절감.
- **결함 제거**: 논리 연산자 오류, 조건 누락 등 치명적 논리 결함을 99% 이상 탐지.

### 미래 전망
AI와 자율주행차의 확산으로 소프트웨어의 책임이 커지면서, 과거 항공 분야에만 국한되었던 MC/DC가 일반 임베디드 및 엔터프라이즈 핵심 로직 검증으로 확산되고 있습니다. 또한 형식 검증(Formal Verification) 도구와 결합하여 소스코드를 읽는 즉시 MC/DC 케이스를 자동 생성하고 정합성을 증명하는 방향으로 발전하고 있습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **MC/DC**는 "진짜로 너 때문에 결과가 바뀐 거니?"를 물어보는 아주 똑똑한 질문법이에요.
2. 친구 세 명이서 무거운 짐을 들 때, 한 명씩만 빠져보게 해서 짐이 떨어지는지 확인하면 **누가 진짜 힘을 쓰고 있는지** 알 수 있는 것과 같아요.
3. 이렇게 하면 모든 사람이 있고 없고를 다 해보지 않아도, 비행기나 자동차가 고장 나지 않게 만드는 중요한 부분을 콕 집어서 검사할 수 있답니다!
