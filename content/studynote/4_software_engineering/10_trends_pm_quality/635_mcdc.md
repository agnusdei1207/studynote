# 635. MC/DC 항공/자동차 안전 표준 조건

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 각 개별 조건(Condition)이 다른 조건의 변화 없이 독립적으로 전체 결정(Decision) 결과에 영향을 미침을 수학적으로 입증하는 화이트박스 테스트의 최고 단계다.
> 2. **가치**: 모든 조합을 테스트하는 다중 조건 커버리지(MCC)의 조합 폭발($2^N$) 문제를 해결하면서($N+1$ 수준의 효율), 논리적 결함을 완벽에 가깝게 검증하여 **Safety-Critical System**의 신뢰성을 보장한다.
> 3. **융합**: 항공(DO-178C Level A)과 자동차(ISO 26262 ASIL D) 표준의 핵심 요건이며, CI/CD 파이프라인 내 **정적 분석(Static Analysis)** 및 **형상 검증(Formal Verification)** 도구와 결합하여 자동화된 검증 프로세스를 구축하는 추세다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 및 정의
**MC/DC (Modified Condition/Decision Coverage, 수정된 조건/결정 커버리지)**는 소프트웨어의 논리적 복잡도를 검증하는 화이트박스 테스트 기법이다. 단순히 모든 코드를 실행하는 것을 넘어, 복합 조건문(예: `A && B || C`) 내부의 각 개별 조건(Atom)이 전체 결과에 **독립적으로 영향(Independence)**을 미치는지를 검증한다. 이는 소프트웨어가 "논리적으로 올바르게 설계되었는가"를 증명하는 가장 강력한 수단 중 하나로, **SIL (Safety Integrity Level, 안전 무결성 수준)**이 높은 시스템에서 필수적으로 요구된다.

### 2. 등장 배경: 효율성과 안전성의 딜레마
복잡한 임베디드 시스템의 제어 로직은 수십 개의 변수가 복합적으로 얽혀 있다. 이를 완벽히 테스트하기 위한 **MCC (Multiple Condition Coverage, 다중 조건 커버리지)**는 변수 $N$개에 대해 $2^N$개의 테스트 케이스를 요구한다. 예를 들어 변수 20개일 경우 약 100만 개의 케이스가 필요해 현실적으로 불가능하다. 이를 해결하기 위해 NASA(미국 항공우주국)에서 항공 소프트웨어 검증을 위해 고안한 것이 MC/DC이며, 이는 최소 $N+1$개의 테스트 케이스로 MCC 수준의 검증력을 확보한다.

```text
   [논리식 테스트 비교: 효율성 vs 안전성]

   변수 N = 5개 (A, B, C, D, E) 가정
   
   ┌────────────────────────────────────────────────────────────┐
   │ 1. 완전 조합 (MCC)        : 2^5 = 32 개의 테스트 케이스    │
   │    → 안전성 100%,  비용/시간: 매우 높음 (비현실적)         │
   ├────────────────────────────────────────────────────────────┤
   │ 2. 단순 분기 (DC)         : 2 개의 테스트 케이스           │
   │    → T/F 결과만 확인,  내부 조건 검증 불가 (위험)           │
   ├────────────────────────────────────────────────────────────┤
   │ 3. MC/DC (이상적)         : 5 + 1 = 6 개의 테스트 케이스   │
   │    → 각 조건의 독립적 영향력 확인,  비용/시간: 효율적      │
   └────────────────────────────────────────────────────────────┘
```

**해설**:
상기 다이어그램은 테스트 대상 논리식의 복잡도에 따른 검증 비용과 안전성의 트레이드오프 관계를 도식화한 것이다. **MCC**는 모든 변수의 상태 조합을 검증하므로 가장 안전하지만, 변수가 증가함에 따라 테스트 케이스 수가 기하급수적으로 증가(Combinatorial Explosion)하는 문제가 있다. 반면 **DC (Decision Coverage)**는 단순히 참과 거짓 결과만 확인하므로 내부 논리의 오류를 놓칠 수 있다. **MC/DC**는 각 변수가 최소한 한 번은 다른 변수들의 상태를 고정(Coupling)한 채로 결과를 뒤집는지 확인함으로써, 선형적인 비용($N+1$)으로 최적의 검증 효율을 제공한다.

> 📢 **섹션 요약 비유**:
> MC/DC는 마치 복잡한 전자제품의 **'정밀 회로 검사'**와 같습니다. 전체 기기가 켜지고 꺼지는지(분기 검사)만 볼 것이 아니라, 각 납땜 부위(개별 조건)가 전체 흐름에 독립적으로 영향을 주는지, 하나씩 전류를 통보며(독립성 입증) 불량 여부를 판단하는 과정입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 구성 요소 및 정의 (표)

| 구성 요소 (Component) | 정의 (Definition) | 내부 동작 메커니즘 | 비고 (Note) |
|:---|:---|:---|:---|
| **Condition (조건)** | 논리식을 구성하는 최소 단위(변수, 리턴 값 등) | `True` 또는 `False`로 평가되는 `Boolean` 피연산자 | 예: `(A > 0)` |
| **Decision (결정)** | 여러 조건이 논리 연산자(`&&`, `\|\|`)로 결합된 최종 평가 식 | 내부 조건들의 조합에 따라 최종 `True/False` 산출 | 예: `(A && B) \|\| C` |
| **Coupling (결합)** | 특정 조건의 독립성을 검증하기 위해 다른 조건의 상태를 고정시키는 것 | "조건 A를 검증할 때 조건 B, C를 고정" | Short-circuit Evaluation 주의 |
| **Independence (독립성)** | 조건 P가 결정 D의 결과를 바꿀 수 있는 유일한 변화 요인임을 증명 | $D(P=T) \neq D(P=F)$ while others constant | MC/DC의 핵심 충족 조건 |

### 2. MC/DC 만족 3대 조건 (핵심 알고리즘)
MC/DC를 달성하기 위해선 다음 수학적 조건을 충족하는 테스트 셋 $T$가 존재해야 한다.

1.  **결정 커버리지 (Decision Coverage)**: 모든 결정(Decision)이 적어도 한 번은 `True`와 `False`를 반환해야 함.
2.  **조건 커버리지 (Condition Coverage)**: 모든 조건(Condition)이 적어도 한 번은 `True`와 `False` 값을 가져야 함.
3.  **독립성 영향 (Independence Effect)**: 각 조건 $C_i$에 대해, 다른 모든 조건이 고정된 상태에서 $C_i$의 값만 변화시켰을 때 결정 결과가 바뀌는 쌍(Couple)이 존재해야 함.

### 3. 상세 동작 분석 및 시나리오 (ASCII 다이어그램)

아래는 복잡한 제어 로직에서 MC/DC가 어떻게 논리적 결함을 포착하는지 보여주는 시나리오다.

```text
   [검증 대상 로직]
   결과 = (온도_높음 && 압력_높음) || (비상_스위치_ON)
   
   ┌─────────────────────────────────────────────────────────────────────┐
   │                     MC/DC 테스트 케이스 설계 전략                    │
   ├───────────┬───────────┬───────────┬───────────┬─────────────────────┤
   │   Case ID │ 온도_높음  │ 압력_높음  │비상_스위치 │      결과 (Output)  │
   ├───────────┼───────────┼───────────┼───────────┼─────────────────────┤
   │ 1 (BASE)  │   False   │   False   │   False   │      False          │
   ├───────────┼───────────┼───────────┼───────────┼─────────────────────┤
   │ 2         │ >>>True<<<│   False   │   False   │      True  (↑)      │
   │           │ (증명: 온도가 독립적으로 결과를 T로 만듦)                │
   ├───────────┼───────────┼───────────┼───────────┼─────────────────────┤
   │ 3         │   True    │ >>>True<<<│   False   │      True  (-)      │
   │           │ (주의: 온도=T, 압력=T인데 결과가 T. 독립성 미달?         │
   │           │  이전 케이스와 비교 불가.  케이스 2와 3은 같은 결과)    │
   ├───────────┼───────────┼───────────┼───────────┼─────────────────────┤
   │ 4 (조정)  │   True    │   False   │ >>>True<<<│      True  (↑)      │
   │           │ (증명: 비상 스위치만 T로 바뀌어도 결과가 T가 됨을 확인)  │
   └───────────┴───────────┴───────────┴───────────┴─────────────────────┘
```

**해설**:
위 표는 `((A && B) || C)` 형태의 논리식을 검증하는 과정이다. 단순히 T/F를 맞추는 것이 아니라 **"이 변수(온도)가 바뀌었을 때 유일하게 결과가 바뀌는 상황이 있는가?"**를 증명해야 한다. 케이스 2는 케이스 1(Base)에 비해 온도만 True로 바뀌었는데 결과가 False에서 True로 바뀌었으므로, **온도(Condition)가 결과(Decision)에 독립적 영향을 미침**을 수학적으로 증명한다. 만약 압력 변수가 논리적으로 잘못 연결되어 있다면(버그), 이 케이스에서 결과가 변하지 않아 MC/DC 커버리지 충족에 실패하게 된다. 또한, 프로그래밍 언어의 **Short-circuit Evaluation** 특성상 `A && B`에서 A가 False면 B를 평가하지 않으므로, 독립성 검증을 위해 피연산자의 순서를 고려한 신중한 케이스 설계가 요구된다.

### 4. 핵심 알고리즘 및 코드 (수식 및 스니펫)

```c
/*
 * [Safety Critical System] Braking Logic Example
 * Target Logic: A (Sensor1) || (B (Sensor2) && C (Sensor3))
 */
 
 // MC/DC Verification Pseudo-Code
 void test_mcdc_independence() {
     bool decision;
     
     // 1. Independence Check for A (Must show A changes decision alone)
     // Base: B=T, C=T (Fixed)
     decision = calculate(A=false, B=true, C=true); // Result: True (due to B&C)
     decision = calculate(A=true,  B=true, C=true); // Result: True
     // Note: A cannot be proven independent if B and C are both True
     // Strategy: Set B=F, C=F
     decision = calculate(A=false, B=false, C=false); // Result: False
     decision = calculate(A=true,  B=false, C=false); // Result: True -> PROVED
     
     // 2. Independence Check for B (Must show B changes decision alone)
     // Strategy: Fix A=F, C=T
     decision = calculate(A=false, B=false, C=true);  // Result: False
     decision = calculate(A=false, B=true,  C=true);  // Result: True -> PROVED
 }
```

> 📢 **섹션 요약 비유**:
> 마치 **'연쇄 폭탄 해체 장치'**의 각 전선이 실제로 연결되어 있는지 확인하는 과정입니다. 단순히 스위치를 눌러보는 게 아니라(결정 확인), 다른 전선은 건드리지 않은 채 특정 전선 한 가닥만 끊었을 때 폭발이 멈추는지(결과 변화) 확인하여, 해당 전선이 실제로 작동에 핵심적인 역할을 하고 있는지 입증하는 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 심층 기술 비교: MC/DC vs 경계값 분석

| 비교 항목 | **MC/DC (Modified Condition/Decision Coverage)** | **경계값 분석 (Boundary Value Analysis)** | **정형 검증 (Formal Verification)** |
|:---|:---|:---|:---|
| **분류** | 구조적 (Structural) 화이트박스 테스트 | 명세기반 (Specification-based) 블랙박스 | 수학적 증명 (Mathematical Proof) |
| **검증 대상** | 코드 내부의 **논리적 결합(Logic Coupling)** | 입력 데이터의 **범위(Range)** 및 경계 | 모델의 **상태(State)** 및 속성(Property) |
| **결함 발견 능력** | 논리 연산자 오류(`&&` vs `\|\|`), 조건 누락 | 오프-바이-원(Off-by-one) 에러, 한계치 미달 | 데드락, 라이브락, 수학적 모순 |
| **비용/난이도** | 높음 (자동화 도구 필수) | 중간 (테이블 작성 가능) | 매우 높음 (전문가 지식 필요) |
| **주요 적용 분야** | 항공, 원자력, 자율주행(SW Safety) | UI, 일반 비즈니스 로직 | 주문형 반도체, 핵심 제어 알고리즘 |

### 2. 과목 융합 관점: 컴파일러 최적화와의 상관관계
고급 언어(High-Level Language)의 소스코드상 MC/DC가 100%라 하더라도, 컴파일러의 **최적화(Optimization)** 과정에서 코드가 변형되면 바이너리(Binary) 수준에서는 MC/DC가 깨질 수 있다.

*   **최적화 기법**: 컴파일러는 `(A && B)`에서 A가 False면 B를 평가하지 않는 **Dead Code Elimination**을 수행한다.
*   **갭(Gap)**: 소스코드에서는 조건 B의 독립성을 테스트하려 했으나, 실제 기계어에서는 B가 실행되지 않아 커버리지가 측정되지 않는 문제가 발생한다.
*   **해결책**: 소스 코드 기반 검증과 오브젝트 코드 기반 검증을 모두 수행하거나, **`objdump`** 등을 통해 최적화된 코드의 흐름을 분석해야 한다.

### 3. 타 산업
