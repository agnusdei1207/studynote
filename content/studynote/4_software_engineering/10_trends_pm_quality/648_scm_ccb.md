# 648. 소프트웨어 형상 관리 (SCM) 통제 위원회 CCB

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 형상 관리(SCM, Software Configuration Management)는 개발 수명주기 전반의 변경 사항을 체계적으로 식별, 통제, 감사하여 시스템의 무결성(Integrity)과 가시성(Visibility)을 확보하는 거버넌스 체계이다.
> 2. **거버넌스**: 형상 통제 위원회(CCB, Configuration Control Board)는 변경 요청(CR, Change Request)에 대한 기술적/비즈니스적 타당성을 심의하고 승인하는 최종 의사결의 기구로, 프로젝트의 범위(Scope)와 품질(Quality)을 방어하는 방어막 역할을 수행한다.
> 3. **가치**: "누가, 언제, 무엇을, 왜 변경했는가"에 대한 완벽한 추적성(Traceability)을 제공하여, **베이스라인(Baseline)**의 안정성을 보장하고 잠재적인 개발 재작업(Rework) 비용을 최소화한다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 철학
소프트웨어 개발은 본질적으로 '변경의 연속'이다. 요구사항의 변화, 환경의 변화, 오류의 수정 등 끊임없이 코드와 문서가 변경된다. SCM은 이러한 무질서한 변경을 체계적으로 관리하여 소프트웨어의 진화 과정을 통제 가능한 상태로 유지하는 학문이자 기술이다. 특히 개발자들의 자의적인 변경으로 인해 발생하는 **"Integration Hell" (통합 지옥)** 현상을 방지하는 핵심 안전장치이다.

#### 2. 💡 비유: 건축 설계 변경과 건축위원회
초고층 빌딩을 짓는다고 가정해 보자.
- **형상 항목(CI, Configuration Item)**: 철근, 유리, 내부 배선도, 설계도면 등 건축에 쓰이는 모든 자재와 문서.
- **형상 관리(SCM)**: 자재가 입고될 때마다 검수하고, 설계 변경이 있을 때마다 도면의 버전을 관리하며, 누가 어떤 자재를 어디에 썼는지 기록하는 시스템.
- **형상 통제 위원회(CCB)**: "주요 구조벽을 옮기고 싶다"는 시공사의 요청이 들어왔을 때, 구조적 안전 여부, 비용 증가분, 공기 지연 여부를 따져보고 승인 여부를 결정하는 **건축 심의 위원회**.

#### 3. 등장 배경 및 필요성
① **기존 한계**: 소규모 개발 시절에는 파일 복사본을 날짜로 저장하는 수준으로 충실했으나, 대규모 분산 개발 환경에서는 "누가 마스터 소스를 덮어썼는가?" 하는 사고가 빈번했다.
② **혁신적 패러다임**: 중앙 집중식 저장소(Repository)와 변경 통제 절차(Change Control Procedure)를 도입하여, 병렬 개발 환경에서도 충돌을 최소화하고 변경을 가시화했다.
③ **현재 비즈니스 요구**: 규제가 심한 항공우주, 의료, 금융권 등에서는 소프트웨어 결함으로 인한 생명/재산 피해를 막기 위해 **매우 엄격한 형상 관리 규정(CAPA, Corrective and Preventive Action) 준수를 의무화**하고 있다.

#### 📢 섹션 요약 비유
> "마치 수많은 실들이 엉키지 않도록 각 색실마다 번호를 매기고, 자수 도안을 수정할 때마다 디자이너와 바느질사가 함께 검토하여 자수 놀이가 망가지지 않게 관리하는 작업과 같습니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 형상 관리 시스템 (CMS)의 4대 주요 활동 구성

형상 관리는 단순한 버전 관리를 넘어 아래와 같은 프로세스로 순환한다.

| 활동 (Activity) | 정의 (Definition) | 세부 내부 동작 (Internal Mechanism) | 주요 산출물 (Artifact) |
|:---:|:---|:---|:---|
| **① 형상 식별**<br>(Identification) | 관리 대상(CI)을 선정하고 명명 규칙 부여 | - 기능적/물리적 특징 분석<br>- 네이밍 컨벤션 정의<br>- 속성(버전, 작성자) 정의 | CI 리스트, 식별 명세서 |
| **② 형상 통제**<br>(Control) | 변경을 심의하고 승인/반려 절차 수행 | - **CCB 심의 (핵심)**<br>- 베이스라인 설정/해제<br>- 변경 예방 및 잠금(Lock) | 변경 승인서, CR 로그 |
| **③ 형상 감사**<br>(Auditing) | 변경 내용이 요구사항과 부합하는지 검증 | - 기능적/물리적 감사(FCA/PCA)<br>- 무결성 검사(Integrity Check) | 감사 보고서, 시험 성적서 |
| **④ 상태 기록**<br>(Status Accounting) | 형상의 변경 이력과 현재 상태를 관리 | - 베이스라인 추적<br>- 변경 통계 및 보고 | 상태 보고서(Status Report) |

#### 2. 아키텍처: 형상 통제 위원회(CCB) 중심의 데이터 흐름

아래 다이어그램은 개발자가 변경을 요청부터 배포까지 어떻게 흘러가는지를 보여준다.

```text
   [ DEVELOPER ]          [ CM MANAGER ]         [ CCB ]           [ QA/TEST ]
        │                       │                    │                  │
        │  ① CR (Change Req)   │                    │                  │
        │ ─────────────────────▶│                    │                  │
        │  (변경 요청서 작성)    │                    │                  │
        │                       │                    │                  │
        │                       │  ② Impact Analysis │                  │
        │                       │ ──────────────────▶│                  │
        │                       │  (영향도 분석 보고)  │                  │
        │                       │                    │                  │
        │                       │                    │  ③ Review/Decision
        │                       │                    │ ◀────────────────│
        │                       │                    │                  │
        │                       │  ④ Approve/Reject  │                  │
        │  ⑤ Notify            │ ◀───────────────────│                  │
        │ ◀─────────────────────│                    │                  │
        │                       │                    │                  │
        │  ⑥ Execute (Coding)   │                    │                  │
        │ ──────────────────────────────────────────────────────────────▶│
        │ (Basline Update)      │                    │                  │ ⑦ Verify
        │                       │                    │                  │ (Test/Build)
        │                       │                    │                  │
```

**[다이어그램 해설]**
1. **요청(①)**: 개발자가 결함이나 기능 추가에 대한 CR(Change Request)을 형상 관리자(CM Manager)에게 제출한다.
2. **분석(②)**: 관리자는 해당 변경이 소스코드, DB 스키마, 인터페이스 등에 미칠 영향도(Impact)를 분석하여 CCB에 상정한다.
3. **심의(③)**: CCB(이해관계자 포함)는 기술 타당성, 일정 영향, 비용(Benefit/Cost)을 검토하여 승인/보류/기각을 결정한다.
4. **통제(④~⑤)**: 승인 결과를 개발자에게 통보하고, 형상 저장소(Repository)의 해당 모듈에 대한 잠금(Lock)을 해제하거나 권한을 부여한다.
5. **실행 및 검증(⑥~⑦)**: 개발자가 변경을 수행하고, QA 팀은 이를 테스트하여 새로운 베이스라인(Baseline)을 생성한다.

#### 3. 베이스라인(Baseline)의 기계적 동작 원리
베이스라인은 "이후 개발의 기준이 되는 공식적인 기술 자산 집합"이다.
- **동작 원리**: 소스 코드 v1.0 + 설계서 v1.0 + DB Script v1.0이 하나의 논리적 스냅샷(Snapshot)으로 묶인다.
- **수학적 표현**:
  $$ B_i = \{ (CI_{name}, CI_{version}) \mid CI \in \text{Project}, \text{verified}(CI) = true \} $$
- **기능적 특징**: 베이스라인이 설정되면, 해당 구성 요소는 변경 불가 상태(Frozen)가 되며, CCB의 승인 없이는 덮어쓰기(Overwrite)가 불가능하도록 시스템(Lock)이 제어한다.

#### 📢 섹션 요약 비유
> "마치 복잡한 고속도로 톨게이트에서 하이패스 차선(일반 변경)과 긴급 차선(긴급 변경)을 운영하되, 과속 방지턱(베이스라인)을 설치어 무분별하게 진입하는 차량을 통제하는 것과 같습니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 심층 기술 비교: 버전 관리 시스템(VCS) vs 형상 관리(SCM)

많은 혼동이 있으나, 기술사 시험에서는 이 둘을 명확히 구분해야 한다.

| 비교 항목 | VCS (Version Control System) | SCM (Software Configuration Management) |
|:---|:---|:---|
| **대상** | **소스 코드(Source Code)** 중심 | **모든 산출물(코드+문서+Data+Environment)** |
| **목표** | 변경 이력 관리 및 병합(Concurrent) | 시스템 무결성 및 품질 보증(Quality Assurance) |
| **범위** | 기술적 도구(Tool) | **관리 프로세스 + 도구 + 조직(인력)** |
| **결과** | Diff, Patch, Commit Log | **Baseline, CM Audit Report, Release Note** |
| **핵심** | "How to merge?" | "When to release?" |

#### 2. 융합 관점: DevOps 및 CI/CD와의 시너지

현대의 애자일 환경에서 CCB는 육각형(Waterfall)의 느린 의사결정 구조가 아닌, **자동화된 파이프라인과 결합**한다.
- **시너지**: Git Branching 전략을 통해 개발 브랜치에서의 변경은 자유롭게 허용하지만, `Main/Master` 브랜치로의 Merge(즉, Production Baseline 변경)는 **Pull Request(PR)** 형태를 통해 Code Review와 같은 가상 CCB 절차를 거치게 한다.
- **자동화 통제**: AI-Code Review 도구가 자동으로 잠재적 버그를 탐지하여 CCB의 심의 부담을 줄인다.
- **오버헤드 경고**: CCB의 승인 절차가 너무 복잡하면 CI/CD의 빠른 피드백 루프를 차단하여 '데브옵스(DevOps)'의 가치를 훼손할 수 있으므로, **승인 권한의 위임(Delegation)**이 필수적이다.

#### 📢 섹션 요약 비유
> "VCS는 개인 작가가 쓰는 연습장과 수정 테이프라면, SCM은 출판사에서 원고를 검토하고 책으로 펴내는(GA)까지의 **출판 프로세스 전체**와 같습니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 시나리오: "Gold Master" 직전 결함 발견 및 긴급 변경

**상황**: 차세대 시스템 릴리즈(D-Day) 3일 전, 보안 팀에서 핵심 모듈에 **취약점(Vulnerability)**이 발견되어 긴급 패치를 요청함.

**의사결정 매트릭스 (Decision Matrix)**
| 옵션 (Option) | 조치 내용 | 기술적 리스크 | 일정 리스크 | 비용 | CCB 판단 |
|:---|:---|:---:|:---:|:---:|:---|
| **A. 유지보수** | 취약점 무시하고 릴리즈 후 패치 | **극히 높음 (해킹 가능)** | 낮음 | 중간 | **기각 (Reject)** |
| **B. 긴급 변경** | 즉시 패치 후 2일간 재테스트 | 중간 (회귀 버그 우려) | **높음 (지연)** | 높음 | **조건부 승인** |
| **C. 우회조치** | 해당 기능만 비활성화(WAF 등) | 낮음 | 낮음 | 낮음 | **검토 (Plan B)** |

**CCB 의사결정 과정**:
1. **기술 아키텍트 의견**: "해당 취약점은 Critical 등급이므로 A안은 불가능합니다. C안은 성능 저하가 있으나, 안정성을 위해 현재로서는 B안을 권장합니다."
2. **일정 관리자 의견**: "B안 실행 시 오픈이 2일 연기됩니다. 대신 마케팅에 오픈 지집 사유를 통보해야 합니다."
3. **최종 결론**: **B안(긴급 변경) 채택**. 단, CCB는 관련 모듈에 대해 **"Hotfix Branch"**를 생성하여 격리된 환경에서 패치하도록 지시하고, 24시간 동안 전담 QA를 배치하여 **회귀 테스트(Regression Test)**를 집중 수행한다.

#### 2. 도입 체크리스트 (Practical Checklist)

**[기술적 측면]**
- [ ] **저장소 모델 선정**: 중앙집중식(SVN) vs 분산형(Git). 현재 프로젝트 규모와 보안 정책에 부합하는가?
- [ ] **Branch 전략 수립**: Git-flow, Trunk-based 등 전략에 따른 CCB 승인 지점(merge point) 정의.
- [ ] **CI 연동**: 빌드 실패 시 자동으로 배포가 차단되는 잠금장치(Jenkins Pipeline) 구성.

**[운영/보안 측면]**
- [ ] **CCB 구성원 정의**: PM, 기술 리드, QA 매니저, 고객사 대표가 포함되었는가?
- [ ] **권한 분리**: 개발자(Development)와 운영자(Release)의 권한을 분리하여 **"검증 없는 배포"**를 원천 차단했는가?
- [ ] **감사 추적성**: 누가 언제 코드를 배포했는지 로그가 위변조 불가능하게 남는가?

#### 📢 섹션 요약 비
