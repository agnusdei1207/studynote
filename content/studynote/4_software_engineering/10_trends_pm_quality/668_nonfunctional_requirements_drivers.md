# 668. 비기능 요구사항 아키텍처 드라이버

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템이 '무엇(What)'을 하는지가 아닌, '어떻게(How)' 동작해야 하는지를 규정하는 성능, 보안, 가용성 등의 **품질 속성(Quality Attributes)**이다.
> 2. **드라이버**: 아키텍처 설계에 결정적인 영향을 미치는 핵심 요구사항을 **아키텍처 드라이버(Architecture Drivers)**라 하며, 이는 비기능 요구사항, 비즈니스 제약, 기술적 제약으로 구성된다.
> 3. **가치**: 아키텍처의 근거(Rationale)를 명확히 하고, 상충하는 품질 속성 간의 트레이드오프(Trade-off)를 분석하여 최적의 구조적 결단을 내리게 한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "보이지 않는 요구사항의 힘"

기능 요구사항이 집의 '방 개수'나 '거실 구조'라면, 비기능 요구사항은 '내진 설계', '단열 성능', '보안 시스템'과 같습니다. 방이 아무리 많아도 지진에 무너진다면 살 수 없는 집이 되듯, 소프트웨어 또한 기능이 아무리 화려해도 느리거나 자주 죽는다면 가치를 잃습니다. 아키텍처 드라이버는 이러한 '보이지 않는 핵심'을 찾아내어 설계의 방향을 결정합니다.

### 💡 비유: 자동차의 엔진 설계

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        아키텍처 드라이버 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 새로운 자동차(시스템)를 설계해야 함.                                    │
│                                                                             │
│  1. 기능 요구사항: 사람 5명이 타고, 짐칸이 있어야 함. (모든 차의 기본)           │
│                                                                             │
│  2. 아키텍처 드라이버 (비기능):                                               │
│     - "최고 시속 300km를 내야 함" (성능)                                      │
│     - "한 번 충전으로 500km를 가야 함" (효율성)                               │
│     - "사고 시 운전자가 절대 다치면 안 됨" (안전성)                            │
│                                                                             │
│  → 이 드라이버(목표)에 따라, 경차 엔진을 쓸지 슈퍼카 엔진을 쓸지 결정됨!          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 아키텍처 드라이버의 4대 구성 요소

| 요소 | 설명 | 예시 |
|:---:|:---|:---|
| **핵심 기능** | 시스템의 존재 이유인 핵심 비즈니스 로직 | 결제 처리, 실시간 스트리밍 |
| **품질 속성** | 시스템의 운영 품질 기준 (비기능 요구사항) | 응답 시간 < 1초, 가용성 99.99% |
| **비즈니스 제약** | 사업적 환경으로 인한 제약 사항 | 예산 1억 이내, 3개월 내 출시 필수 |
| **기술적 제약** | 기술적 환경이나 표준으로 인한 제약 | Java 사용 필수, AWS 클라우드 기반 |

### 2. 품질 속성 시나리오 (Quality Attribute Scenario)
비기능 요구사항은 모호하기 쉽습니다(예: "빨라야 함"). 이를 구체화하기 위해 시나리오 기법을 씁니다.
- **자극원**: 사용자, 시스템 오류
- **자극**: 버튼 클릭, 서버 다운
- **환경**: 정상 운영 중, 과부하 시
- **대상**: 데이터베이스, 캐시 서버
- **응답**: 데이터 조회 완료, 예비 서버 기동
- **측정**: 2초 이내 완료, 1분 내 복구

### 3. 드라이버 도출 및 설계 흐름 (ASCII)

```text
    [ Biz Vision ] ──▶ [ Architecture Drivers ] ──▶ [ Design Decisions ]
    (사업 목표)           (핵심 품질/제약)            (패턴/기술 선택)
                               │                         │
                               ▼                         ▼
                        [ Trade-off 분석 ] ◀─────── [ Architecture ]
                        (성능 vs 보안 등)            (구조적 산출물)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 기능 vs 비기능 요구사항

| 구분 | 기능 요구사항 (Functional) | 비기능 요구사항 (Non-functional) |
|:---:|:---|:---|
| **핵심 질문** | 무엇을 하는가? (What) | 어떻게 작동하는가? (How) |
| **검증 방법** | 유스케이스 테스트 | 성능/부하/보안 테스트 |
| **설계 영향** | 모듈 로직 설계에 영향 | **전체 구조(Architecture)**에 영향 |

### 2. 기술적 시너지: 아키텍처 평가 (ATAM)
비기능 요구사항이 잘 반영되었는지 확인하기 위해 ATAM (Architecture Trade-off Analysis Method) 기법을 사용합니다. 품질 속성 간의 충돌(예: 보안을 강화하면 성능이 떨어짐)을 분석하여 최적의 합의점을 찾습니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 글로벌 뱅킹 앱 아키텍처 설계
- **드라이버 1**: 초당 1만 건의 트랜잭션 처리 (성능).
- **드라이버 2**: 데이터 유실 0% 및 강력한 암호화 (보안/무결성).
- **결단**: **트레이드오프 분석**. 보안을 위해 복잡한 암호화를 쓰되, 성능 저하를 막기 위해 하드웨어 가속기(HSM)와 인메모리 DB(Redis)를 도입하기로 결정.
- **판단**: 비기능 요구사항이 단순한 희망 사항이 아닌, 구체적인 하드웨어 및 소프트웨어 스택 결정의 근거가 됨.

### 📢 기술사적 결언
> "아키텍처의 성패는 기능을 얼마나 많이 넣느냐가 아니라, **'상충하는 비기능 요구사항들 사이에서 얼마나 현명한 결단을 내리느냐'**에 달려 있다. 드라이버가 명확하지 않은 아키텍처는 뿌리 없는 나무와 같아, 운영 단계의 작은 바람에도 쉽게 흔들린다. 아키텍트는 항상 '왜 이 기술을 썼는가?'라는 질문에 대해 아키텍처 드라이버를 근거로 답할 수 있어야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **운영 안정성**: 초기 설계 단계부터 가용성을 고려하여 장애 시간 80% 단축.
- **확장성**: 유연성(Flexibility) 드라이버 반영으로 신규 기능 추가 비용 절감.

### 미래 전망
최근에는 **'지속적 아키텍처 (Continuous Architecture)'**가 강조되면서, 비기능 요구사항을 코드로 검증하는 **'아키텍처 피트니스 함수 (Fitness Functions)'** 기술이 발전하고 있습니다. 배포 파이프라인(CI/CD)에서 시스템의 성능이나 보안 지표가 드라이버 기준에 미달하면 배포를 자동 차단하는 수준으로 진화하고 있습니다.

2. **비기능 요구사항**은 배터리가 오래가고, 떨어뜨려도 액정이 깨지지 않게 튼튼하게 만드는 거예요.
