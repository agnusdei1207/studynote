# 555. 이벤트 소싱 (Event Sourcing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트 소싱(Event Sourcing)은 객체의 현재 상태(State)를 직접 저장하는 대신, 해당 상태를 변화시킨 **모든 사건(Event)들을 순차적으로 기록**하여 데이터의 전체 이력을 보존하는 영속성 패턴이다.
> 2. **가치**: 데이터의 무결성과 완벽한 감사 추적(Audit Trail)을 보장하며, 과거 특정 시점의 상태를 완벽히 재현(Replay)하거나 다양한 형태의 읽기 전용 모델(View)을 자유롭게 생성할 수 있는 유연성을 제공한다.
> 3. **융합**: CQRS(명령 조회 분리) 패턴과 찰떡궁합을 이루며, 메시지 브로커(Kafka 등)와 결합되어 마이크로서비스(MSA) 간의 강력한 데이터 정합성 보장 기술로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: "데이터는 흐름이다." 전통적인 방식이 DB 테이블의 값을 덮어쓰는(Update) 것이라면, 이벤트 소싱은 "주문됨", "결제됨", "배송지 변경됨"과 같은 개별 사건들을 추가 전용(Append-only) 로그로 저장하는 방식이다.

- **💡 비유**: 이벤트 소싱은 **"은행 통장 내역"**과 같습니다. 은행은 내 '최종 잔액'만 저장하지 않습니다. "1만원 입금", "5천원 출금" 같은 거래 내역(이벤트)을 모두 기록합니다. 거래 내역을 처음부터 다 더하면 최종 잔액(현재 상태)이 나오는 원리와 같습니다.

- **등장 배경**:
  1. **데이터 유실의 공포**: `Update` 문을 실행하면 이전 데이터는 영원히 사라진다. "누가, 언제, 왜 이 값을 바꿨는가?"에 대한 답을 얻기 위해 모든 역사가 필요해졌다.
  2. **복잡한 도메인 분석**: 비즈니스가 고도화되면서 단순한 현재 값보다 사용자의 행동 패턴(이벤트 흐름) 분석이 더 큰 가치를 갖게 되었다.
  3. **분산 시스템의 정합성**: 여러 서비스에 흩어진 데이터 상태를 맞추기 위해, "무슨 일이 일어났는지"에 대한 명확한 증거(Event) 공유가 절실해졌다.

- **📢 섹션 요약 비유**: 현재 모습만 찍은 사진(RDBMS) 대신, 태어날 때부터 지금까지의 모든 과정을 담은 동영상(이벤트 소싱)을 소장하는 방식입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 이벤트 소싱의 핵심 구성 요소

| 요소 명칭 | 역할 및 정의 | 비유 |
|:---|:---|:---|
| **Event Store** | 이벤트를 순서대로 저장하는 Append-only DB | 일기장 / 장부 |
| **Aggregate** | 이벤트를 적용하여 현재 상태를 계산하는 도메인 객체 | 현재의 나 |
| **Replay** | 저장된 이벤트를 순차적으로 재생하여 상태 복원 | 과거 회상 |
| **Snapshot** | 성능 향상을 위해 특정 시점의 상태를 저장해둠 | 책갈피 |

---

### 2. 이벤트 소싱 동작 및 상태 복원 아키텍처

이벤트가 쌓이고 현재 상태로 조립되는 과정을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      이벤트 소싱 (Event Sourcing) 및 상태 재생 구조               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. Event Storage (Append Only)]                                            │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ #1: OrderCreated (id: 101, user: A)                                   │  │
 │  │ #2: ItemAdded (id: 101, item: Phone)                                  │  │
 │  │ #3: AddressUpdated (id: 101, addr: Seoul)                             │  │
 │  └──────┬────────────────────────────────────────────────────────────────┘  │
 │         │                                                                   │
 │         ▼ [2. State Replay (재생)]                                           │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │  Order Aggregate (Memory)                                             │  │
 │  │  - apply(#1) ──▶ { status: CREATED }                                  │  │
 │  │  - apply(#2) ──▶ { status: CREATED, items: [Phone] }                  │  │
 │  │  - apply(#3) ──▶ { status: CREATED, items: [Phone], addr: Seoul }     │  │
 │  └──────┬────────────────────────────────────────────────────────────────┘  │
 │         │                                                                   │
 │         ▼ [3. Current State (현재 상태)]                                      │
 │  ┌──────────────────────────────────────┐                                   │
 │  │  Final Order Info: [Phone] at Seoul  │                                   │
 │  └──────────────────────────────────────┘                                   │
 │                                                                             │
 │  ※ 가치: 중간에 데이터가 꼬여도 처음부터 다시 재생하면 완벽히 복구됨                  │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 핵심은 **불변성 (Immutability)**이다. **Event Store**에 한 번 적힌 이벤트는 절대 수정되거나 삭제되지 않는다. 시스템에 장애가 나거나 데이터 정합성이 의심될 때, 저장된 이벤트를 1번부터 차례대로 꺼내어 빈 객체에 적용(**apply**)하면 100% 동일한 **현재 상태**를 얻을 수 있다. 성능을 위해 수백만 개의 이벤트를 매번 재생할 수는 없으므로, 1,000개 단위로 **Snapshot**을 찍어두는 최적화 기법을 병행한다.

---

### 심층 동작 원리: 원자적 저장과 발행 (Transactional Outbox)

이벤트 소싱이 메시지 큐와 결합될 때의 신뢰성 보장 원리다.
1. **상황**: DB에 이벤트는 저장했는데, Kafka로 메시지를 쏘는 데 실패하면? (데이터 불일치)
2. **해결**: 이벤트 저장과 메시지 발행을 하나의 '로컬 트랜잭션'으로 묶거나, DB 로그를 읽어 자동으로 메시지를 쏘는 **CDC** 기술을 활용한다.
3. **가치**: "기록된 것은 반드시 전파된다"는 강력한 신뢰 관계를 형성한다.

- **📢 섹션 요약 비유**: 편지를 써서 우체통에 넣는 것(저장)과 우체부가 이를 수거해가는 것(발행)이 하나의 완벽한 시스템으로 연결되어 있어야 편지가 유실되지 않는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 상태 중심 저장 (RDBMS) vs 이벤트 소싱 비교

| 비교 항목 | 상태 중심 (CRUD) | 이벤트 소싱 (Event Sourcing) |
|:---|:---|:---|
| **저장 내용** | 결과값 (Current Value) | **원인 및 과정 (History)** |
| **수정 방식** | 덮어쓰기 (Update) | **추가하기 (Append)** |
| **감사(Audit)** | 별도 로그 테이블 필요 | **자체적으로 완벽한 감사 증적** |
| **성능** | 조회(Read)에 유리 | **쓰기(Write)에 유리**, 조회 시 가공 필요 |
| **비유** | 성적표 (최종 점수) | 수능 시험지 (풀이 과정 전체) |

---

### 기술 융합: 이벤트 소싱과 CQRS

이벤트 소싱의 유일한 단점인 '조회 성능'을 CQRS가 해결한다.
- **조합**: 이벤트 소싱으로 데이터를 쓰고(Write), 이 이벤트를 구독하여 조회용 DB(Read Model)를 미리 만들어둔다.
- **효과**: "모든 이력을 보관"하면서도 "검색은 번개처럼 빠른" 꿈의 아키텍처가 완성된다.

- **📢 섹션 요약 비유**: 매번 비디오를 처음부터 돌려보는 게 힘들어서, 주요 명장면만 사진(Read Model)으로 찍어 벽에 붙여두고 바로 감상하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '금융 거래 시스템'의 무결성 보장 설계

- **문제 상황**: 고객의 포인트가 증발했다는 항의가 들어왔으나, DB에는 현재 잔액만 있고 언제 왜 깎였는지 로그가 남지 않아 대응 불가능.
- **의사결정 (이벤트 소싱 적용)**:
  1. **Event Store 구축**: 모든 포인트 변동을 `PointEarned`, `PointUsed`, `PointExpired` 이벤트로 정의하여 저장한다.
  2. **무결성 검증**: 매일 밤 전체 이벤트를 다시 계산(Replay)하여 현재 DB 잔액과 일치하는지 자동 대조한다.
  3. **시간 여행 (Time Travel)**: "3월 1일 오전 10시 당시의 잔액"을 확인하고 싶다면, 그 시간대까지의 이벤트만 재생하여 즉시 확인한다.
  4. **결과**: 데이터 변조가 원천적으로 불가능해지며, 고객에게 1원 단위의 정확한 변동 이력을 근거로 제시하여 신뢰를 얻는다.

### 도입 체크리스트
- **Event Schema Evolution**: 1년 전 이벤트 형식과 지금 형식이 다를 때 어떻게 처리할 것인가? (Upcasting 전략)
- **Snapshot Frequency**: 성능과 저장 공간 사이의 타협점(예: 이벤트 500개당 스냅샷 1개)을 찾았는가?
- **Storage Selection**: 추가 전용(Append-only)에 최적화된 저장소(EventStoreDB, Kafka 등)를 선정했는가?

### 안티패턴
- **이벤트 내 비즈니스 로직**: 이벤트 자체에 계산 수식을 넣는 행위. 이벤트는 오직 '발생한 사실' 데이터여야 한다.
- **무분별한 적용**: 게시판의 '조회수'처럼 중요도가 낮고 변경이 잦은 데이터에 도입하여 인프라 비용을 낭비하는 경우.

- **📢 섹션 요약 비유**: 모든 영수증을 다 모으는 것은 좋지만, 껌 한 통 산 영수증(사소한 데이터)까지 평생 보관하려다 집이 쓰레기장(저장소 부족)이 되는 것은 피해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Reliability)** | 데이터 복구 성공률 100% | 모든 히스토리가 남아있어 어떠한 장애 상황에서도 상태 복원 가능 |
| **정량적 (Performance)** | 쓰기 처리량(TPS) 향상 | 락(Lock) 없는 Append-only 저장으로 쓰기 성능 극대화 |
| **정성적 (Compliance)** | 법적 감사 증적 완벽 대응 | 규제 기관에 데이터의 모든 변경 이력을 즉시 제출 가능 |
| **정성적 (Analytics)** | 비즈니스 통찰력 확장 | 사용자의 행동 흐름을 분석하여 새로운 마케팅 기회 발굴 |

### 미래 전망
- **Cloud-Native Event Store**: 클라우드 벤더가 이벤트의 순서와 저장을 완벽히 보장하는 서버리스 이벤트 스토어를 출시하여 도입 문턱이 낮아질 것이다.
- **AI Replay Analysis**: 수조 개의 이벤트를 AI가 Replay하며 이상 징후를 탐지하고, 미래의 상태를 예측하는 '시뮬레이션 아키텍처'가 대세가 될 것이다.

### 참고 표준
- **Domain-Driven Design (Evans)**: 이벤트 소싱의 철학적 배경
- **Reactive Streams**: 이벤트 전파의 기술적 표준
- **Event Store DB API**: 실무 구현의 레퍼런스

- **📢 섹션 요약 비유**: "현재는 과거의 결과물입니다." 이벤트 소싱은 소프트웨어에 '기억'이라는 영혼을 불어넣어, 단순한 기계가 아닌 역사를 가진 유기체로 성장시키는 위대한 설계입니다.
