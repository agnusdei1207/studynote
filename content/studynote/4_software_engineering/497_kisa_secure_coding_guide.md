# 497. 행정안전부/KISA 소프트웨어 개발 보안 가이드 (47개 보안 약점)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 행정안전부/KISA 가이드는 대한민국 공공기관 정보시스템 구축 시 의무적으로 준수해야 하는 **7대 분야 47개 보안 약점 제거 기준**을 정의한 국내 표준 소프트웨어 개발 보안 지침이다.
> 2. **가치**: CWE와 OWASP 등 글로벌 표준을 국내 개발 환경에 맞춰 체계화하였으며, 설계 및 구현 단계에서 소스 코드의 근본적인 취약점을 제거함으로써 전자정부 서비스의 안전성과 신뢰성을 보장한다.
> 3. **융합**: '소프트웨어 보안성 검토' 제도와 직결되어 감리 및 보안 진단의 핵심 준거로 활용되며, 현대의 시큐어 코딩(Secure Coding) 교육 및 자동화 진단 도구(SAST)의 표준 룰셋(Rule-set)으로 기능한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 본 가이드는 SW 개발 단계에서 발생할 수 있는 보안 약점을 사전에 제거하여 안전한 SW를 개발하기 위한 목적의 지침서다. 2012년 12월 '정보시스템 구축·운영 지침' 개정을 통해 공공사업에 의무화되었으며, 현재는 47개 보안 약점을 기준으로 진단 및 조치가 이루어진다.

- **💡 비유**: 행안부 가이드는 **"대한민국 모든 공공 건물을 지을 때 반드시 지켜야 하는 '건축 안전 법규 47개 조항'"**과 같습니다. 건물의 용도나 디자인이 달라도, 이 47가지 안전 기준을 통과하지 못하면 준공 허가(서비스 오픈)를 받을 수 없는 엄격한 기준입니다.

- **등장 배경**:
  1. **사이버 위협의 지능화**: 홈페이지 변조, 개인정보 유출 등 침해 사고의 70% 이상이 애플리케이션 보안 약점을 악용하여 발생함에 따라 근본적인 대책이 필요해졌다.
  2. **국가 정보보안 수준 상향**: 전자정부 서비스의 세계화에 발맞춰 글로벌 수준(CWE 기반)의 보안 코딩 표준을 확립하고자 했다.
  3. **개발 단계 보안 내재화**: 운영 단계의 사후 조치보다 개발 단계의 사전 예방이 훨씬 경제적(수정 비용 1/30 수준)이라는 인식 확산.

- **📢 섹션 요약 비유**: 사고가 난 뒤에 고치는 '응급 처치'가 아니라, 유전자 설계 단계부터 병에 걸리지 않도록 관리하는 '근본적 예방 의학' 체계를 국가 차원에서 확립한 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SW 개발 보안 7대 분야 및 47개 보안 약점 구성

가이드는 기술적 특성에 따라 보안 약점을 7개의 큰 카테고리로 분류한다.

| 분야 (Category) | 약점 개수 | 주요 핵심 내용 | 비유 |
|:---|:---:|:---|:---|
| **1. 입력 데이터 검증 및 표현** | 15개 | SQLi, XSS, 경로 조작 등 외부 입력값 필터링 미비 | 검역소의 불량 검사 |
| **2. 보안 기능** | 16개 | 인증, 인가, 암호화, 권한 관리 등의 기능적 결함 | 보안 장치의 설계 오류 |
| **3. 시간 및 상태** | 2개 | Race Condition 등 병렬 실행 시의 공유 자원 문제 | 신호등 없는 교차로 |
| **4. 에러 처리** | 3개 | 에러 메시지를 통한 시스템 정보 노출 | 도둑에게 약점 말하기 |
| **5. 코드 오류** | 5개 | 널 포인터 역참조, 부적절한 자원 해제 등 | 기계의 내부 부품 결함 |
| **6. 캡슐화** | 5개 | 민감한 데이터 노출, 디버그 코드 방치 등 | 벽에 난 구멍 |
| **7. API 오용** | 1개 | 보안에 취약한 API의 무분별한 사용 | 불량 도구 사용 |

사업 기획부터 감리까지 보안 가이드가 적용되는 프로세스를 시각화한다.

**[다이어그램 해설]** 공공 SW 사업에서 47개 보안 약점 준수는 선택이 아닌 필수다. 개발자는 구현 단계에서 가이드에 정의된 코딩 표준을 지켜야 하며, 완료 시점에 반드시 **보안 약점 진단 도구(CC인증 제품)**를 사용하여 정적 분석을 수행해야 한다. 감리 단계에서는 이 진단 결과 리포트를 확인하고, 미조치된 항목이 있을 경우 '적합' 판정을 내리지 않는다. 최종적으로 모든 약점이 조치되었음을 증명하는 '이행 확인서'가 있어야만 시스템 오픈이 가능하다.

### 심층 동작 원리: 7대 분야별 대표 약점 분석

1. **입력 데이터 검증 (CWE-20)**: 사용자의 입력이 SQL 쿼리나 HTML 출력을 변조하지 못하도록 `prepared statement`를 쓰거나 특수문자를 치환(Escaping)한다.
2. **보안 기능 (인증/암호)**: 비밀번호는 솔트(Salt)를 포함한 일방향 해시를 쓰고, 중요 데이터 전송 시에는 TLS 1.2 이상의 암호화 통신을 강제한다.
3. **시간 및 상태 (Race Condition)**: 멀티쓰레드 환경에서 동일 자원에 동시 접근 시 데이터가 꼬이지 않도록 동기화(Synchronization) 처리를 한다.

- **📢 섹션 요약 비유**: 건물 설계도에 소화기 위치를 표시하고(보안 기능), 창문에 방범창을 달며(입력 검증), 전선이 꼬이지 않게 정리(코드 오류)하는 꼼꼼한 안전 관리 지침과 같습니다.

### 융합 시너지: 정보시스템 감리와 개발 보안

정보시스템 감리는 행안부 가이드 준수 여부를 확인하는 최후의 보루다.
- **감리 관점**: 단순한 '버그' 탐지가 아니라, 사업 범위 내의 모든 소스 코드가 보안 기준을 통과했는지 '품질' 관점에서 접근한다.
- **연계 효과**: 감리 결과 보고서에 보안 약점 조치 결과가 포함됨으로써, 발주처는 안심하고 대금을 지급하고 운영팀은 보안 부채(Security Debt) 없이 시스템을 인수받는다.

- **📢 섹션 요약 비유**: 시험을 보기 전 오답 노트를 완벽히 정리(행안부 가이드 준수)하고, 시험장 입구에서 선생님께 최종 확인(감리)을 받는 완벽한 합격 전략과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 공공기관 차세대 시스템 보안 진단 및 조치

- **문제 상황**: 차세대 시스템 오픈을 1개월 앞두고 실시한 보안 약점 진단에서 3,000건의 '미조치' 결과가 쏟아져 나옴.
- **의사결정 (행안부 가이드 기반 조치)**:
  1. **우선순위 분류**: 7대 분야 중 '입력 데이터 검증' 및 '보안 기능' 관련 약점(SQLi, 암호화 미흡)을 최우선 'Fatal' 등급으로 분류하여 즉시 조치한다.
  2. **공통 모듈 수정**: 특정 로직에서 반복되는 약점(예: 에러 처리 미흡)은 개별 코드 수정 대신 공통 에러 페이지 처리 필터(Filter)를 적용하여 일괄 해결한다.
  3. **수동 진단 병행**: 도구로 탐지되지 않는 '권한 관리 결함'이나 '비즈니스 로직 우회'는 전문가가 수동으로 점검하여 가이드의 취지를 충실히 이행한다.
  4. **증적 관리**: 각 조치 전후의 소스 코드를 캡처하여 '보안 약점 제거 확인서'를 작성, 감리법인과 KISA의 승인을 득한다.

### 도입 체크리스트
- **CC인증 도구**: 사용하는 진단 도구가 국정원 CC인증을 받은 제품인가?
- **진단 범위**: 운영 소스뿐 아니라 테스트 코드, 샘플 코드까지 전수 진단 대상에 포함했는가?
- **조치 적절성**: 단순히 경고를 지우기 위해 코드를 삭제한 것이 아니라, 가이드에서 권고하는 안전한 패턴으로 수정했는가?

### 안티패턴
- **예외 처리 오남용**: 조치가 어려운 항목에 대해 근거 없이 '오탐(False Positive)' 처리하여 보고서에서 누락시키는 행위.
- **구현 후 일괄 진단**: 개발이 다 끝난 뒤에야 처음으로 도구를 돌리는 행위. (수정 범위가 너무 넓어져 오픈 일정을 맞추기 불가능해짐)

- **📢 섹션 요약 비유**: 시험 공부를 하나도 안 하다가 시험 전날 밤에 벼락치기를 하며 모르는 문제는 다 "오타일 거야"라고 무시해버리는 위험한 태도와 같습니다.
