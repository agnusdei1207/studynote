# 575. 객체지향 설계와 디자인 패턴 (GoF) 총정리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 객체지향 설계(OOD)는 데이터와 행위를 하나로 묶은 '객체'를 중심으로 복잡성을 관리하는 패러다임이며, 디자인 패턴(GoF)은 이를 기반으로 실전에서 마주치는 설계 문제들에 대한 **검증된 정석적 해결책(Best Practice)**을 정형화한 것이다.
> 2. **가치**: SOLID 원칙을 통해 코드의 '건강함'을 유지하고, 23가지 GoF 패턴을 통해 시스템의 '유연성'과 '확장성'을 확보함으로써, 변화하는 비즈니스 요구사항에 최소한의 수정으로 대응할 수 있는 **지속 가능한 소프트웨어**를 구축한다.
> 3. **융합**: 아키텍처 패턴(MVC, MSA 등)이 거시적 골격이라면, 디자인 패턴은 그 골격을 채우는 미시적 근육과 같으며, 현대의 클린 아키텍처와 리팩토링 기술은 이 모든 객체지향 원리의 총체적 실현이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 객체지향 설계는 "어떻게 하면 코드를 부품처럼 잘 만들고 조립할 수 있을까?"에 대한 고민에서 시작되었다. **SOLID 원칙**이 설계의 '도덕책'이라면, **GoF 23 패턴**은 그 도덕을 지키며 효율적으로 일하는 '업무 매뉴얼'과 같다.

- **💡 비유**: 객체지향 설계와 디자인 패턴은 **"전문가들로 구성된 오케스트라"**와 같습니다. 각 연주자(객체)가 자기 악기만 잘 다루는 것(SRP)도 중요하지만, 지휘자(패턴)의 지휘에 따라 서로 박자를 맞추고 화음을 내는 약속(디자인 패턴)이 있어야만 웅장한 교향곡(소프트웨어)이 완성되는 것과 같습니다.

- **등장 배경**:
  1. **소프트웨어의 복잡도 임계점**: 함수 단위의 절차적 프로그래밍으로는 수백만 줄의 코드를 관리할 수 없게 되면서 '상태와 행위의 캡슐화'가 대안으로 떠올랐다.
  2. **재사용성의 한계**: 단순히 코드를 복사해서 쓰는 수준을 넘어, 클래스 간의 '관계 구조'를 재사용하려는 요구가 커졌다.
  3. **전문가 지식의 표준화**: 경험 많은 설계자들의 머릿속에만 있던 '설계 팁'들을 문서화하여 초보 개발자들도 일정 수준 이상의 설계를 할 수 있게 가이드할 필요가 생겼다.

- **📢 섹션 요약 비유**: 뒤죽박죽 섞인 부품 상자(나쁜 코드)를 잘 분류된 전용 부품 함(좋은 설계)으로 바꾸어, 누구나 쉽게 조립하고 확장할 수 있게 만드는 설계의 '표준화 공정'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 객체지향 설계의 근간: SOLID 원칙 요약

디자인 패턴은 아래 5가지 원칙을 지키기 위해 고안되었다.

| 원칙 | 핵심 목표 | 패턴과의 연결성 |
|:---|:---|:---|
| **SRP** | 단일 책임 | 클래스가 비대해지면 **Strategy, Command** 등으로 분리 |
| **OCP** | 개방-폐쇄 | **Decorator, Strategy**를 통해 코드 수정 없이 기능 확장 |
| **LSP** | 치환 가능성 | **Template Method** 등 상속 기반 패턴의 안정성 보장 |
| **ISP** | 인터페이스 분리 | **Adapter** 등을 통해 필요한 기능만 노출 |
| **DIP** | 의존성 역전 | **Factory, Bridge** 패턴의 핵심 아키텍처 기반 |

---

### 2. GoF 23 디자인 패턴의 구조적 맵

세 가지 분류가 시스템의 다른 측면을 담당한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      GoF 23 디자인 패턴 통합 체계도                                │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [생성 (Creational)] ─▶ "어떻게 만들 것인가?"                                    │
 │  - Singleton, Factory Method, Abstract Factory, Builder, Prototype          │
 │                                                                             │
 │ [구조 (Structural)] ─▶ "어떻게 엮을 것인가?"                                    │
 │  - Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy          │
 │                                                                             │
 │ [행위 (Behavioral)] ─▶ "어떻게 대화할 것인가?"                                  │
 │  - Strategy, Observer, State, Command, Template Method, Iterator,           │
 │    Mediator, Memento, Visitor, Interpreter, Chain of Responsibility         │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 핵심 매커니즘:                                                         │  │
 │  │ 1. 상속보다는 합성을 사용하라 (Composition over Inheritance)               │  │
 │  │ 2. 인터페이스에 의존하라 (Program to an Interface)                        │  │
 │  │ 3. 변하는 것과 변하지 않는 것을 분리하라 (Encapsulate Variation)            │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 중심 철학은 **'유연성'**이다. **생성 패턴**은 객체 생성 시의 결합도를 낮추고, **구조 패턴**은 큰 덩어리의 객체들을 조립식으로 관리하며, **행위 패턴**은 실행 시점의 로직을 동적으로 바꿀 수 있게 한다. 이 모든 패턴의 밑바닥에는 **인터페이스(Interface)**를 통한 소통이 자리 잡고 있다. 구체적인 클래스 이름 대신 인터페이스라는 '약속'을 바라보게 함으로써, 언제든 부품을 갈아 끼울 수 있는 '열린 구조'를 만드는 것이 디자인 패턴의 최종 지향점이다.

---

### 심층 동작 원리: 패턴 간의 협력 사례

실제 시스템에서는 여러 패턴이 섞여서 시너지를 낸다.
1. **Abstract Factory + Singleton**: 테마별 부품을 만드는 공장(Factory)은 시스템에 딱 하나(Singleton)만 있으면 된다.
2. **Composite + Iterator**: 폴더 구조(Composite)를 순회하며(Iterator) 파일을 찾는다.
3. **Strategy + Template Method**: 전체 알고리즘 골격은 부모가 잡고(Template), 세부 알고리즘은 런타임에 외부에서 주입(Strategy)받는다.

- **📢 섹션 요약 비유**: 숟가락(패턴 A)과 젓가락(패턴 B)이 세트(융합)가 되었을 때 비로소 맛있는 식사(완벽한 기능)가 가능한 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 디자인 패턴 간의 목적 차이 분석

| 비교 그룹 | 패턴 A | 패턴 B | 결정적 차이 |
|:---|:---|:---|:---|
| **생성** | **Factory Method** | **Abstract Factory** | 단일 객체 생성 vs **연관된 제품군** 생성 |
| **구조** | **Adapter** | **Proxy** | 인터페이스 **변환** vs **접근 제어/대리** |
| **행위** | **Strategy** | **State** | **사용자가 선택**하는 로직 vs **내부 상태가 결정**하는 로직 |
| **행위** | **Command** | **Strategy** | **무엇(What)**을 할지 캡슐화 vs **어떻게(How)** 할지 캡슐화 |

---

### 기술 융합: 프레임워크 속에 녹아든 디자인 패턴

우리가 쓰는 프레임워크는 사실 디자인 패턴의 전시장이다.
- **Spring Framework**: **Singleton** (Bean 관리), **Proxy** (AOP/트랜잭션), **Template Method** (JdbcTemplate), **Strategy** (Resolver들).
- **React / Vue**: **Observer** (상태 변화 감지), **Component (Composite)** 구조.

- **📢 섹션 요약 비유**: 우리가 타는 자동차(프레임워크)의 엔진과 바퀴 안에는 이미 수천 개의 정교한 기계 공학 패턴(디자인 패턴)이 숨어 있어 우리가 편하게 운전할 수 있는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 잦은 요건 변경이 예상되는 '이벤트 시스템' 설계

- **문제 상황**: 매주 새로운 할인 행사와 경품 지급 방식이 추가되어야 하며, 특정 조건에서는 알림톡이, 특정 조건에서는 메일이 발송되어야 함.
- **의사결정 (디자인 패턴 믹스 전략)**:
  1. **행위**: 할인 계산은 **Strategy** 패턴으로 분리하여 런타임에 교체 가능하게 한다.
  2. **구조**: 알림 발송은 **Observer** 패턴을 적용하여, 새로운 알림 채널(SMS 등)이 추가되어도 메인 로직은 수정하지 않게 설계한다.
  3. **생성**: 각 이벤트 객체는 복잡한 속성 설정이 필요하므로 **Builder** 패턴을 통해 가독성 있게 생성한다.
  4. **보안/로깅**: 모든 이벤트 실행 전후의 기록은 **Proxy** 패턴(Spring AOP)을 통해 비즈니스 로직과 분리하여 처리한다.

### 도입 체크리스트
- **Simplicity vs Pattern**: 단순한 기능에 억지로 패턴을 적용하여 복잡성만 키우지는 않았는가?
- **Pattern Language**: 팀원들끼리 "여기에 파사드(Facade)를 씁시다"라고 했을 때 동일한 설계 그림을 떠올리는가?
- **Documentation**: 적용된 패턴의 의도와 구조를 클래스 다이어그램 등으로 남겼는가?

### 안티패턴
- **Pattern Happy**: "모든 패턴을 한 번씩 다 써보겠다"는 식의 공부용 코드를 실운영에 올리는 행위. (유지보수의 재앙)
- **Hardcoding Patterns**: 패턴의 구조를 지키기 위해 인터페이스를 만들었으나, 정작 내부에서 구체 클래스를 `new`로 생성하여 결합도를 끊지 못하는 경우.

- **📢 섹션 요약 비유**: 요리 도구가 많다고 좋은 요리사가 아닙니다. 칼 한 자루(단순 코드)로도 충분한 곳에 굳이 푸드 프로세서(복잡한 패턴)를 꺼내 설거지 거리만 늘리는 실수를 하지 말아야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Complexity)** | 코드 응집도 2배 향상 | 단일 책임 원칙과 패턴화를 통해 모듈 간 결합도 대폭 하락 |
| **정량적 (Time)** | 기능 확장 시간 50% 단축 | 기존 코드를 건드리지 않는 플러그인 식 확장으로 개발 가속화 |
| **정성적 (Standard)** | 아키텍처 일관성 유지 | 누구나 이해할 수 있는 표준 설계 구조를 통해 코드 품질 유지 |
| **정성적 (Comm.)** | 고차원 의사소통 가능 | 설계 토론 시 구구절절 설명 대신 패턴 이름으로 합의 도달 |

### 미래 전망
- **Micro-Patterning**: 클라우드 네이티브와 서버리스 환경에 특화된 경량 패턴들이 GoF의 자리를 보완하며 발전할 것이다.
- **AI-Driven Design**: AI가 비즈니스 로직의 변동성을 예측하여, 적절한 지점에 디자인 패턴을 자동 제안하고 코드를 생성해주는 기술이 성숙될 것이다.

### 참고 표준
- **GoF (Gang of Four)**: 디자인 패턴의 시초이자 영원한 고전
- **Refactoring to Patterns (J. Kerievsky)**: 패턴 적용의 실전 가이드
- **SOLID (Robert C. Martin)**: 디자인 패턴의 사상적 기반

- **📢 섹션 요약 비유**: 디자인 패턴은 시대를 관통하는 '설계의 철학'입니다. 기술의 이름은 바뀌어도 그 이면의 구조적 지혜는 소프트웨어가 존재하는 한 영원히 빛날 것입니다.
