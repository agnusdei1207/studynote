# 555. 헥사고날 아키텍처 (Hexagonal Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 헥사고날 아키텍처 (포트와 어댑터 아키텍처)는 시스템의 **핵심 비즈니스 로직(Inside)을 외부 기술 환경(Outside)으로부터 분리**하고, 이들을 인터페이스(Port)와 구현체(Adapter)를 통해 연결하는 대칭적 설계 방식이다.
> 2. **가치**: 데이터베이스, 사용자 인터페이스, 외부 서비스 등을 시스템에 갈아 끼울 수 있는 '플러그인'으로 취급함으로써, 기술적 변경이 비즈니스 로직에 영향을 주지 않는 **기술 독립성**과 높은 **테스트 용이성**을 확보한다.
> 3. **융합**: 클린 아키텍처와 어니언 아키텍처의 기술적 토대가 되며, 특히 현대의 도메인 주도 설계(DDD)와 마이크로서비스(MSA) 환경에서 서비스 내부를 견고하게 구축하는 핵심 구조로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 앨리스터 코번(Alistair Cockburn)이 제안한 아키텍처로, 시스템을 내부(비즈니스 도메인)와 외부(인프라, UI 등)로 나눈다. 핵심 로직은 '포트(Port)'라는 구멍을 뚫어두고, 외부 기술들은 '어댑터(Adapter)'를 통해 이 구멍에 꽂히는 구조다.

- **💡 비유**: 헥사고날 아키텍처는 **"여행용 멀티 어댑터"**와 같습니다. 내 노트북(비즈니스 로직)은 그대로인데, 일본에 가면 110V 어댑터(MySQL용 어댑터)를 끼우고 유럽에 가면 220V 어댑터(Oracle용 어댑터)를 끼워서 사용하는 것과 같습니다. 노트북 내부 부품을 바꿀 필요는 없는 것과 같습니다.

- **등장 배경**:
  1. **계층형 아키텍처의 한계**: 전통적인 3계층 구조는 비즈니스 로직이 데이터베이스(하위 계층)에 강하게 종속되어, DB를 바꾸려면 로직까지 다 고쳐야 하는 문제가 있었다.
  2. **자동화 테스트의 어려움**: UI나 DB가 연결되어야만 테스트가 가능한 구조는 느리고 깨지기 쉬웠다. 외부 요인을 배제한 순수 로직 테스트가 절실해졌다.
  3. **다양한 인터페이스 수용**: 웹뿐만 아니라 앱, CLI, 테스트 스크립트 등 다양한 경로로 들어오는 요청을 동일한 로직으로 처리해야 할 필요성이 커졌다.

- **📢 섹션 요약 비유**: 소프트웨어를 거대한 하나의 기계가 아니라, 부품을 언제든 뺏다 꼈다 할 수 있는 '모듈형 조립 키트'로 바라보는 혁신적인 발상의 전환입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 헥사고날 아키텍처의 3대 구성 요소

| 요소 명칭 | 역할 및 정의 | 비유 |
|:---|:---|:---|
| **도메인 (Domain/Inside)** | 핵심 비즈니스 로직과 엔티티가 위치하는 순수 영역 | 우주선의 조종실 |
| **포트 (Port)** | 도메인으로 들어가거나 나가는 통로 (인터페이스) | 조종실의 표준 버튼/소켓 |
| **어댑터 (Adapter/Outside)** | 외부 기술을 포트 규격에 맞게 변환하는 구현체 | 버튼을 누르는 손 또는 외부 센서 |

---

### 포트와 어댑터 동작 아키텍처

내부 로직을 중심으로 외부 요소들이 대칭적으로 배치된 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      헥사고날 (Ports and Adapters) 아키텍처 모델                  │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Driving Side (Left)]           [Core (Inside)]          [Driven Side (Right)] │
 │ (Primary Adapters)              (Business Logic)         (Secondary Adapters)  │
 │                                                                             │
 │  ┌──────────────┐              ┌──────────────┐              ┌──────────────┐  │
 │  │ Web Controller│ ────▶       │  Input Port  │              │ Database     │  │
 │  └──────────────┘ (Call)       │ (Interface)  │              │ Implementation│  │
 │                                └──────┬───────┘              └──────▲───────┘  │
 │  ┌──────────────┐                     │                             │          │
 │  │ Mobile App   │ ────▶        ┌──────┴──────┐              ┌──────┴──────┐  │
 │  └──────────────┘              │   Service   │              │ Output Port │  │
 │                                │ (Business)  │ ────(Call)──▶│ (Interface) │  │
 │  ┌──────────────┐              └──────┬───────┘              └─────────────┘  │
 │  │ Test Script  │ ────▶               │                             │          │
 │  └──────────────┘              ┌──────┴──────┐              ┌──────▼──────┐  │
 │                                │   Domain    │              │ External API │  │
 │                                │   Entity    │              │ Adapter      │  │
 │                                └─────────────┘              └──────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처는 크게 세 영역으로 나뉜다. 1) **Driving Side (왼쪽)**: 사용자의 요청을 도메인이 이해할 수 있게 변환해주는 영역이다. 웹 요청이든 테스트 코드든 동일한 **Input Port**를 호출한다. 2) **Core (중앙)**: 외부 기술을 전혀 모르는 순수한 비즈니스 로직의 집합이다. 3) **Driven Side (오른쪽)**: 도메인이 필요로 하는 외부 기능을 수행하는 영역이다. 도메인은 **Output Port** 인터페이스만 호출하고, 실제 DB 저장이나 이메일 발송은 구체적인 **Driven Adapter**가 처리한다. 이 구조 덕분에 비즈니스 로직은 "어떤 DB를 쓰는지" 전혀 모르는 완벽한 고립을 달성한다.

---

### 심층 동작 원리: 의존성 역전 (Dependency Inversion)

도메인을 인프라로부터 해방시키는 핵심 원리다.
1. **문제**: 로직이 데이터를 저장하려면 DB 클래스를 참조해야 하므로, 도메인이 DB에 종속된다.
2. **해결**: 도메인 쪽에 `SaveData`라는 인터페이스(Port)를 정의한다. DB 모듈(Adapter)이 이 인터페이스를 상속받아 구현한다.
3. **가치**: 의존성의 화살표가 **외부에서 내부(도메인)**로 향하게 되어, 도메인은 아무런 외부 라이브러리 없이도 독립적으로 존재하고 테스트될 수 있다.

- **📢 섹션 요약 비유**: 전등(도메인)이 벽의 콘센트(포트)에 직접 연결되는 것이 아니라, 전등은 플러그(포트)만 가지고 있고 벽의 전선(어댑터)이 그 플러그에 맞게 시공되는 순서와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 헥사고날 vs 전통적 레이어드 아키텍처 비교

| 비교 항목 | 레이어드 (Layered) | 헥사고날 (Hexagonal) |
|:---|:---|:---|
| **중심 관점** | 기술적 계층 (UI-Service-DB) | **비즈니스 도메인 중심** |
| **의존성 방향** | 상위 계층 ──▶ 하위 계층 | **외부 계층 ──▶ 내부 코어** |
| **대칭성** | 수직적/비대칭적 | **수평적/대칭적** (좌우 대칭) |
| **테스트** | DB가 있어야 테스트 가능 | **Mock만으로 비즈니스 로직 100% 테스트** |
| **비유** | 샌드위치 (차례로 쌓기) | 소켓과 플러그 (끼워 맞추기) |

---

### 기술 융합: DDD(도메인 주도 설계)와의 시너지

헥사고날 아키텍처는 DDD의 사상을 코드로 옮기기 위한 가장 적합한 그릇이다.
- **Ubiquitous Language**: 코어 영역의 엔티티와 서비스 코드를 비즈니스 용어만으로 채울 수 있게 해준다.
- **Bounded Context**: 서비스 간의 경계를 헥사고날 내부의 '도메인' 단위로 명확히 분리하여 MSA 전환을 돕는다.

- **📢 섹션 요약 비유**: 요리사(도메인 전문가)가 주방 도구(DB/인프라)의 종류와 상관없이 오직 레시피(로직)에만 집중하여 최고의 맛을 낼 수 있게 환경을 조성하는 주방 설계 철학입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 외부 연동 서비스가 수시로 바뀌는 '알림 시스템' 설계

- **문제 상황**: 현재는 이메일로만 알림을 보내는데, 앞으로 카카오톡, 라인, 슬랙 등 수많은 채널이 추가될 예정임. 기존 코드는 이메일 라이브러리와 뒤섞여 확장이 매우 어려운 상황.
- **의사결정 (헥사고날 스타일 적용)**:
  1. **포트 정의**: 도메인 내부에 `NotificationPort`라는 인터페이스를 만들고 `send(Message msg)` 메서드를 정의한다.
  2. **코어 로직 보호**: 알림을 보내는 비즈니스 로직은 오직 `NotificationPort`만 호출하도록 작성한다. (이메일인지 슬랙인지 알 필요 없음)
  3. **어댑터 구현**: `EmailAdapter`, `SlackAdapter`, `KakaoAdapter`를 각각 독립적인 모듈로 개발하여 포트를 상속받게 한다.
  4. **결과**: 새로운 알림 채널이 추가되어도 기존 비즈니스 코드는 전혀 손대지 않고, 새로운 어댑터만 하나 더 꽂음으로써 확장을 완료한다. (Open-Closed Principle 준수)

### 도입 체크리스트
- **Dependency Flow**: 도메인 코드 내에 `import` 문 중 인프라나 프레임워크와 관련된 것이 없는가?
- **Mapping Overhead**: 도메인 엔티티와 DB 엔티티(JPA 등)를 분리하여 매핑하는 오버헤드를 팀이 감당할 준비가 되었는가?
- **Test Coverage**: 외부 DB 연결 없이 비즈니스 로직만으로 돌아가는 단위 테스트가 충분히 작성되었는가?

### 안티패턴
- **Pass-through Port**: 로직 없이 단순히 호출만 전달하는 무의미한 포트가 남발되는 경우.
- **도메인 내 인프라 노출**: 포트의 파라미터로 DB의 특정 예외 클래스나 웹의 `HttpRequest` 객체를 그대로 넘기는 행위. (캡슐화 파괴)

- **📢 섹션 요약 비유**: 만능 어댑터를 쓴다면서 정작 내부 전선은 특정 전압에만 견디게 설계하는(기술 종속성 잔재) 실수를 범해서는 안 됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Test)** | 테스트 속도 10배 향상 | 인프라 의존성 없는 순수 유닛 테스트로 CI 성능 극대화 |
| **정량적 (Agility)** | 기술 교체 시간 70% 단축 | 특정 벤더 종속 탈피로 DB나 미들웨어 교체 리스크 최소화 |
| **정성적 (Standard)** | 비즈니스 로직 가시성 확보 | 기술적 노이즈가 제거된 코드를 통해 업무 로직 파악 용이 |
| **정성적 (Resilience)** | 시스템 유연성 극대화 | 다양한 인터페이스(API, Event, Batch)를 동일 로직으로 수용 |

### 미래 전망
- **Architecture-as-Code**: 헥사고날 구조를 AI가 자동으로 유지하고 감시하여, 계층 위반 시 자동으로 리팩토링하는 지능형 설계 도구가 보편화될 것이다.
- **Polyglot Portability**: 포트와 어댑터의 분리를 통해, 동일한 비즈니스 로직을 온프레미스, 클라우드, 에지 컴퓨팅 등 어떤 환경으로도 즉시 이전(Migration)하는 기술이 표준이 될 것이다.

### 참고 표준
- **Alistair Cockburn의 원전 블로그**: 헥사고날 아키텍처의 탄생
- **Clean Architecture (Robert C. Martin)**: 헥사고날을 대중화시킨 이론서
- **Ports and Adapters Pattern**: 실무 구현 패턴의 표준 명칭

- **📢 섹션 요약 비유**: 기술의 유행은 바람처럼 변하지만, 기업의 비즈니스 가치는 바위처럼 견고해야 합니다. 헥사고날 아키텍처는 그 바위를 보호하고 바람에 유연하게 대처하는 가장 영리한 방법입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 내가 만든 게임 캐릭터(비즈니스 로직)는 그대로인데, **조이패드(어댑터)를 쓰든 키보드(어댑터)를 쓰든 똑같이 재미있게 게임을 할 수 있게 만드는 규칙**이에요.
2. 캐릭터에게는 "조종 도구가 무엇이든 상관없어, 나는 점프만 하면 돼!"라는 약속(포트)만 만들어주는 거죠.
3. 이렇게 하면 나중에 새로운 최신 게임기가 나와도 캐릭터를 다시 만들 필요 없이 바로 놀 수 있답니다!
