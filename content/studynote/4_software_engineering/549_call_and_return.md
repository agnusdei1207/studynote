# 549. 호출 및 복귀 스타일 (Call and Return)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 호출 및 복귀 (Call and Return) 아키텍처 스타일은 제어의 흐름이 **상위 컴포넌트에서 하위 컴포넌트로 명시적으로 이동했다가 처리가 끝나면 다시 되돌아오는** 계층적·절차적 설계 방식이다.
> 2. **가치**: 시스템을 추상화 수준에 따라 수직적으로 분할하여 복잡도를 낮추며, 특히 **Layered Architecture**는 각 계층의 책임을 명확히 분리함으로써 유지보수성과 교체 용이성을 확보하는 데 결정적인 역할을 한다.
> 3. **융합**: 고전적인 메인 프로그램 방식부터 현대적인 객체지향 설계, 클라이언트-서버 구조의 근간이 되며, 최근에는 계층 간 결합도를 더 낮추기 위해 인터페이스 기반의 의존성 주입(DI) 기술과 결합되어 사용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: Call and Return 스타일은 가장 직관적이고 널리 쓰이는 아키텍처 방식이다. 프로그램이 실행되면서 필요한 기능을 가진 함수나 객체를 부르고(Call), 결과를 받으면(Return) 다음 단계로 진행하는 방식이다. 대표적으로 **Main Program/Subroutine** 방식과 **Layered Architecture**가 이 범주에 속한다.

- **💡 비유**: Call and Return은 **"회사의 업무 지시 체계"**와 같습니다. 사장님(Main)이 부장님(Subroutine)을 불러 보고서를 작성하라고 시키면, 부장님은 일을 마친 뒤 결과물(Return)을 들고 다시 사장님께 갑니다. 사장님은 그 결과를 보고 다음 일을 결정합니다.

- **등장 배경**:
  1. **절차적 프로그래밍의 성숙**: 복잡한 기능을 한 덩어리로 만들지 않고, 의미 있는 작은 단위(Routine)로 나누어 관리하려는 분할 및 정복 전략이 정립되었다.
  2. **추상화 계층의 필요성**: 하드웨어나 DB 같은 저수준 작업과 비즈니스 로직 같은 고수준 작업을 분리하여, 서로의 영역을 침범하지 않게 하려는 요구가 커졌다.
  3. **재사용성 확보**: 공통적으로 쓰이는 기능을 서브루틴으로 만들어두고 필요할 때마다 호출하여 중복 코드를 제거하고자 했다.

- **📢 섹션 요약 비유**: 복잡한 주문을 한꺼번에 처리하는 것이 아니라, 카운터(상위)에서 주방(하위)으로 주문서를 넘기고 음식을 돌려받는 정돈된 서비스 체계와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 세부 스타일 비교

| 스타일 명칭 | 구조적 특징 | 주요 장점 | 비유 |
|:---|:---|:---|:---|
| **Main Program/Subroutine** | 메인이 하위 루틴들을 계층적으로 호출 | 이해하기 쉬운 단방향 흐름 | 대장과 쫄병들 |
| **Layered Architecture** | 시스템을 독립된 수평 계층으로 분할 | **계층 간 독립성**, 유지보수 우수 | 7층 건물 (각 층 기능 다름) |
| **Client-Server** | 클라이언트가 요청하고 서버가 응답 | 서비스 분산 처리, 확장성 | 손님과 점원 |

---

### 2. Layered Architecture의 핵심 아키텍처

가장 대표적인 Call and Return의 변형으로, 계층 간 통신 규칙을 강제한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      표준 4계층 아키텍처 (Layered Architecture)                  │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. Presentation Layer] (UI)                                                │
 │      │  - 사용자 요청 수용, 결과 출력                                          │
 │      ▼                                                                      │
 │ [2. Business/Service Layer]                                                 │
 │      │  - 핵심 비즈니스 로직 처리, 트랜잭션 관리                                 │
 │      ▼                                                                      │
 │ [3. Persistence/Data Access Layer]                                          │
 │      │  - DB 연동 로직 (DAO, Repository)                                     │
 │      ▼                                                                      │
 │ [4. Database Layer] (Storage)                                               │
 │         - 실제 데이터 저장 및 관리                                             │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 핵심 원칙:                                                            │  │
 │  │ 1. 인접 계층만 호출 가능: 1층에서 3층으로 직접 점프 금지 (Decoupling)        │  │
 │  │ 2. 상위에서 하위로만 호출: 역방향 호출은 순환 의존성 유발로 금지              │  │
 │  │ 3. 하위 계층은 상위 계층을 모름: 하위는 오직 서비스 제공에만 집중             │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Layered Architecture는 Call and Return의 원리를 수직적 계층으로 정립한 것이다. **Presentation Layer**는 오직 **Service Layer**만 호출할 수 있고, 그 아래의 DB가 무엇인지 알 필요가 없다. 이러한 **'관심사 분리'** 덕분에 DB를 Oracle에서 MySQL로 바꿔도 오직 3계층(Persistence)만 수정하면 될 뿐, 상위 계층은 아무런 영향을 받지 않는다. 다만, 모든 요청이 계층을 하나씩 통과해야 하므로 약간의 성능 오버헤드가 발생할 수 있다는 것이 특징이다.

---

### 심층 동작 원리: 제어의 역전 (Inversion of Control, IoC)

Call and Return 스타일의 결합도를 낮추는 현대적 기술이다.
1. **전통적 방식**: 상위 모듈이 하위 모듈의 인스턴스를 직접 생성하고 호출함. (강한 결합)
2. **IoC 방식**: 상위 모듈은 '인터페이스'만 호출하고, 실제 구현체는 외부(컨테이너)가 넣어줌.
3. **가치**: "내가 부를 테니 너는 대기해"가 아니라 "나는 이 규격을 원하니 누구든 이 규격에 맞는 애를 데려다줘"라는 유연한 구조로 변모한다.

- **📢 섹션 요약 비유**: 사장이 특정 직원(구현체)을 직접 지목해 일시키는 게 아니라, "이 자격증(인터페이스) 있는 사람 아무나 와서 하세요"라고 공고를 내는 스마트한 인사 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Call and Return vs 독립 컴포넌트(Event-driven) 비교

| 비교 항목 | Call and Return (호출형) | Independent Components (이벤트형) |
|:---|:---|:---|
| **소통 방식** | **명시적 호출 (Direct)** | **메시지 발행/구독 (Indirect)** |
| **결합도** | 높음 (대상을 알아야 함) | **매우 낮음 (대상을 몰라도 됨)** |
| **동작 제어** | 호출자가 흐름을 결정 | 발생한 이벤트가 흐름을 유발 |
| **장점** | 로직 추적이 쉬움, 예측 가능 | 유연한 확장, 실시간성 |
| **비유** | 전화를 걸어 지시하기 | 게시판에 할 일 적어두기 |

---

### 기술 융합: 클린 아키텍처(Clean Architecture)와 계층

Call and Return 원칙을 극한으로 끌어올린 형태다.
- **의존성 규칙**: 모든 의존성은 안쪽(비즈니스 엔티티)을 향해서만 흐른다.
- **효과**: 프레임워크나 DB 같은 외부 요인이 바뀌어도 핵심 로직은 절대 변하지 않는 견고한 Call/Return 구조를 완성한다.

- **📢 섹션 요약 비유**: 어떤 날씨(외부 환경)에도 변하지 않는 단단한 성의 안방(핵심 로직)을 만드는 설계 철학입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 복잡한 레거시 'Main Program'의 계층화 리팩토링

- **문제 상황**: 만 줄이 넘는 하나의 파일에 UI 처리와 SQL 쿼리가 범벅되어 있어, 작은 버그 하나 고치는 데도 한 달이 걸리는 상황.
- **의사결정 (Layered Style 적용)**:
  1. **횡적 분리**: 화면 관련 코드는 Controller로, 계산 로직은 Service로, SQL은 Repository로 쪼개어 각각의 계층(Layer)을 만든다.
  2. **DTO 도입**: 계층 간 데이터를 주고받을 때 DB 테이블 구조를 그대로 넘기지 않고, 딱 필요한 정보만 담은 전용 바구니(DTO)를 사용하여 정보 은닉을 실천한다.
  3. **인터페이스 추출**: Service가 Repository의 구체적인 클래스를 알지 못하게 인터페이스를 두어, 테스트 시 가짜 데이터(Mock)를 넣기 쉽게 만든다.
  4. **결과**: 리팩토링 후, UI를 웹에서 모바일로 바꿔도 Business Layer와 Database Layer는 재사용할 수 있는 '진정한 Call and Return'의 이점을 누린다.

### 도입 체크리스트
- **Layer Violation**: 상위 계층에서 하위를 건너뛰고 최하위로 직접 접근하는 '지름길' 코드가 있는가? (계층 침범 엄격 통제)
- **Granularity**: 계층을 너무 많이 나눠서 오히려 코드가 파편화되고 복잡해지지는 않았는가? (적정 계층 유지)
- **Cohesion**: 각 계층 내부의 클래스들이 계층의 이름에 걸맞은 역할만 수행하고 있는가?

### 안티패턴
- **Sinkhole Anti-pattern**: 상위 계층이 하위 계층으로 요청을 전달만 하고 아무런 로직도 수행하지 않는 무의미한 계층이 반복되는 경우. (계층 병합 고려 필요)
- **Circular Dependency**: A 계층이 B를 부르고, 다시 B가 A를 부르는 순환 구조. 시스템 붕괴의 원인이 된다.

- **📢 섹션 요약 비유**: 7층 건물을 지었는데 엘리베이터가 없고 층마다 사다리(계층 침범)로만 오르내린다면, 그 건물은 곧 무너질 위험한 건축물입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Quality)** | 코드 가독성 60% 향상 | 명확한 계층 구조를 통해 신규 개발자의 시스템 파악 시간 단축 |
| **정량적 (Cost)** | 유지보수 비용 절감 | 특정 계층만 교체 가능하여 전체 재개발 리스크 방지 |
| **정성적 (Standard)** | 개발 표준화 기틀 마련 | 팀원 모두가 동일한 호출 규칙을 따르므로 협업 효율 증대 |
| **정성적 (Stability)** | 오류 파급 효과 차단 | 계층 간 독립성을 통해 에러가 다른 계층으로 번지는 것을 억제 |

### 미래 전망
- **Micro-Layering**: 컨테이너와 서버리스 환경에 맞춰 계층 자체가 서비스 단위로 쪼개지는 초미세 계층화 기술이 발전할 것이다.
- **Polyglot Persistence Layer**: 하나의 아키텍처 내에서 용도에 따라 서로 다른 DB(RDBMS, NoSQL, Graph)를 다층적으로 호출하는 구조가 일반화될 것이다.

### 참고 표준
- **Pattern-Oriented Software Architecture (Buschmann et al.)**: Layered 패턴의 정석
- **DDD (Domain-Driven Design)**: 도메인 중심의 계층 설계 가이드
- **Hexagonal Architecture**: 계층 구조를 외부 연동 관점에서 재해석한 스타일

- **📢 섹션 요약 비유**: 수직적인 질서와 명확한 약속이 주는 안정감, 그것이 바로 수십 년간 Call and Return 스타일이 소프트웨어 세계를 지배해온 강력한 힘입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 학교에서 선생님이 **반장에게 "출석부 가져와"라고 시키고, 반장이 가져오면 "고맙다"고 말하는 것**과 같아요.
2. 각자 맡은 층(계층)에서 자기 일만 열심히 하고, 윗사람의 명령에만 대답하는 **아주 예의 바른 설계 방법**이랍니다.
3. 이렇게 질서가 잘 잡혀 있으면, 학교(프로그램)가 시끄럽지 않고 아주 조용하고 안전하게 돌아가요!
