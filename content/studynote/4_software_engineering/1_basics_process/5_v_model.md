# V-모델 (V-Model) 검증과 확인 (Verification & Validation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: V-모델은 폭포수 모델의 확장판으로, 개발 단계와 테스트 단계를 **1:1로 매칭**시켜 개발의 좌측(하향)과 테스트의 우측(상향)이 'V'자 형태를 이루는 소프트웨어 생명주기 모델이다.
> 2. **가치**: 설계 단계에서부터 테스트 계획을 수립하여 **결함을 조기에 발견**하도록 강제하며, 요구사항이 어떻게 검증될지를 명확히 규정하여 소프트웨어의 신뢰성을 극대화한다.
> 3. **융합**: **검증(Verification)**과 **확인(Validation)**의 명확한 구분을 통해 프로젝트의 기술적 무결성과 비즈니스 목적 부합성을 동시에 만족시키는 고도화된 품질 관리 체계를 제공한다.

---

## Ⅰ. V-모델 (V-Model)의 개요

### 1. 정의
- 개발 생명주기의 각 단계마다 상응하는 테스트 단계가 존재하며, 이를 통해 요구사항 분석부터 인수 테스트까지 논리적 연결성을 강조하는 모델이다.

### 2. 💡 비유: 범죄 수사와 재판
- **개발(왼쪽)**: 증거를 수집하고 논리를 짜는 과정.
- **테스트(오른쪽)**: 짜놓은 논리가 법리(요구사항)에 맞는지 재판하는 과정.
- 기소 내용(설계)이 부실하면 나중에 판결(인수 테스트)에서 무죄가 나오듯, 각 단계는 서로를 거울처럼 비춘다.

---

## Ⅱ. V-모델의 구조 및 매핑 메커니즘 (ASCII)

V-모델은 왼쪽의 상세화 과정과 오른쪽의 추상화(통합) 과정이 대칭을 이룬다.

```text
    [ 개발 단계 / Verification ]              [ 테스트 단계 / Validation ]
    
    요구사항 분석 (Requirements) ───────────────▶ 인수 테스트 (Acceptance)
           │                                          ▲
           ▼                                          │
      시스템 설계 (System Design) ────────────▶ 시스템 테스트 (System)
              │                                    ▲
              ▼                                    │
         상세 설계 (Detailed Design) ──────▶ 통합 테스트 (Integration)
                 │                                ▲
                 ▼                                │
            모듈 구현 (Coding) ───────────▶ 단위 테스트 (Unit)
```

### 1. Verification (검증) - 좌측
- **질문**: "제품을 **제대로** 만들고 있는가?" (Are we building the product right?)
- **활동**: 설계서 리뷰, 코드 인스펙션, 워크쓰루 등 주로 '개발 과정의 무결성'을 체크한다.

### 2. Validation (확인) - 우측
- **질문**: "**제대로 된** 제품을 만들었는가?" (Are we building the right product?)
- **활동**: 실제 소프트웨어를 실행하여 사용자의 '최종 요구사항'을 만족하는지 체크한다.

---

## Ⅲ. V-모델의 테스트 단계별 상세 분석

| 테스트 단계 | 대응 개발 단계 | 주요 검증 내용 |
|:---|:---|:---|
| **단위 테스트** | 구현 (Coding) | 개별 모듈/함수의 내부 로직, 인터페이스 검증 |
| **통합 테스트** | 상세 설계 | 모듈 간 상호작용, 인터페이스 정합성, 데이터 흐름 |
| **시스템 테스트** | 시스템 설계 | 기능/비기능 요구사항, 성능, 보안, 하드웨어 연동 |
| **인수 테스트** | 요구사항 분석 | 사용자의 비즈니스 목적 달성 여부, 계약 부합성 |

- **📢 섹션 요약 비유**: 자동차 설계도와 부품이 맞는지 확인하는 것이 단위 테스트라면, 완성된 차를 고객이 운전해보고 만족하는지 확인하는 것이 인수 테스트입니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. 테스트의 조기화 (Early Testing)
- **기술사적 결단**: 실무에서 V-모델을 적용한다는 것은 단순히 나중에 테스트를 하겠다는 뜻이 아니라, **요구사항 분석 단계에서 이미 인수 테스트 시나리오를 작성 시작**해야 함을 의미한다.
- **인사이트**: 결함 발견 시점이 늦어질수록 수정 비용은 기하급수적으로 늘어난다. V-모델은 이 비용을 최소화하기 위한 가장 강력한 정형 모델이다.

### 2. 모델의 한계와 보완 (W-모델로의 진화)
- **비판**: V-모델도 결국 테스트 실행은 구현 이후에 몰려 있다는 단점이 있다.
- **대안**: 개발 활동과 테스트 활동을 병행하여, 개발 산출물이 나올 때마다 즉시 테스트 활동(정적 분석 등)을 수행하는 **W-모델**로 보강하여 운영하는 것이 현대적 추세다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **품질 신뢰성**: 단계별 철저한 검증으로 고장률이 낮은 견고한 시스템 구축.
- **추적성(Traceability)**: 요구사항부터 테스트 결과까지의 연관 관계가 명확하여 책임 추적 가능.

### 2. 미래 전망
최근의 **TDD(Test Driven Development)**나 **BDD(Behavior Driven Development)**는 사실상 V-모델의 정신을 코드 수준에서 극단적으로 단축하여 구현한 것이다. 자동화된 테스트 파이프라인 속에서도 '요구사항과 테스트의 대칭성'이라는 V-모델의 핵심 철학은 소프트웨어 품질 공학의 영원한 금과옥조로 남을 것이다.

- **📢 섹션 요약 비유**: 꼼꼼하게 단추를 끼워 내려갔다가(개발), 다시 아래서부터 제대로 끼워졌는지 확인하며 올라오는(테스트) 정성스러운 바느질과 같습니다.

### 👶 어린이를 위한 3줄 비유 설명

2. 단어를 외울 때(개발) 미리 '이 단어가 시험에 나오면 이렇게 써야지'라고 문제(테스트 계획)를 만들어두는 거죠.
