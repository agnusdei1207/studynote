# 44-53. SW 비용 산정 기법 (COCOMO, FP, Delphi)

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 공학(SW Engineering)에서 비용 산정은 단순한 견적이 아닌 프로젝트의 생존을 결정짓는 **확률적 예측 모델링** 활동이다.
> 2. **가치**: LOC (Line of Code) 기반의 하드웨어적 사고에서 벗어나, FP (Function Point)와 같은 사용자 가치 중심의 산정으로 전환하여 **요구사항 변경(Agile 환경)에 유연한 대응**이 가능하다.
> 3. **융합**: PMBOK(관리)의 자원 관리와 데이터베이스(데이터)의 복잡도 지수를 결합하여, 정성적 위험을 정량적 비용(Risk Exposure)으로 변환하는 의사결정 지원 도구로 활용된다.

---

### Ⅰ. 개요 (Context & Background)

소프트웨어 비용 산정(Cost Estimation)은 "얼마나 많은 노력(Effort, 인월)이 필요하며, 언제 완료(Delivery Date)되는가?"에 대한 수학적이고 통계적인 답을 구하는 과정입니다. 소프트웨어는 보이지 않는 논리(Logic)의 집합이므로, 건축물과 같은 물리적 자재 비용 산정과는 근본적인 차이가 있습니다. 이를 해결하기 위해 과거의 데이터를 기반으로 한 경험적 모델(Empirical Model)과 전문가의 직관을 수리화한 알고리즘이 결합되어 왔습니다.

초기 컴퓨팅 환경에서는 성능이 저조한 하드웨어를 극복하기 위해 코드를 최적화하는 데 집중했으므로, 소스 코드의 물리적 길이인 LOC를 기준으로 삼았습니다. 그러나 비즈니스 로직이 복잡해지고 4세대 언어(4GL)나 코드 제네레이터가 등장하면서, 코드가 짧아지더라도 기능은 방대해지는 역설(Inverse Productivity)이 발생했습니다. 이에 따라 사용자 관점에서 '기능'의 개수를 세는 방식인 FP가 등장했고, 이를 보다 정교화하기 위해 전문가 패널의 편견을 보정하는 델파이 기법이 접목되었습니다.

```ascii
[소프트웨어 비용 산정 패러다임의 진화]

    [ 1960~70s: LOC 중심 ]           [ 1980s: 알고리즘 기반 ]           [ 1990s~: 가치 기반 ]
    +------------------+           +------------------+             +------------------+
    | 물리적 코드 양   |  --->    | 수학적 회귀 분석 |  ------>    | 사용자 기능 가치 |
    | (Hardware View)  |          | (COCOMO 모델)    |             | (Function Point) |
    +------------------+           +------------------+             +------------------+
           |                                |                                |
    "타일을 몇 장 썼나?"              "건물의 용도와 규모는?"          "방이 몇 개인가?"
```

**💡 비유**: 이는 '그림 그리기'를 의뢰받은 화가가 견적을 내는 것과 같습니다.
*   **LOC 방식**은 "물감을 몇 튜브 썼니?"로 계산하는 것입니다.
*   **COCOMO**는 "캔버스 크기와 작가의 숙련도에 따라 시간을 계산"하는 것입니다.
*   **FP**는 "인물이 나오는지, 풍경화인지(기능의 종류)"에 따라 가격을 매기는 것입니다.

**📢 섹션 요약 비유**: 소프트웨어 비용 산정은 **'보이지 않는 집'을 짓는 데 드는 노동비를 예측하는 일'**입니다. 단순히 벽돌(코드)을 몇 개 쌓았는지가 아니라, 그 집이 단층 주택인지 고층 빌딩인지(규모), 그리고 거실이 몇 개인지(기능)에 따라 공임이 책정되는 복잡한 건설 견적 과정입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

소프트웨어 비용 산정의 핵심은 불확실성(Uncertainty)을 어떻게 수식으로 좁히느냐에 달려 있습니다. 크게 전문가의 휴리스틱(Heuristic)을 활용하는 방식과 수학적 알고리즘을 활용하는 방식, 그리고 사용자 가치를 정량화하는 방식으로 나뉩니다.

#### 1. 전문가 판단 기반 (Expert Judgment)
전문가의 경험을 수치화하여 불확실성을 줄이는 방식입니다.

*   **델파이 기법 (Delphi Method)**: 피드백이 없는 단순 합의가 아닌, 익명성을 보장하여 **'Groupthink'(집단 사고)**를 방지하는 통계적 예측법입니다.
    *   **절차**: 조사자 선정 → 1차 질문 → 결과 통계(중앙값, 사분위수) → 2차 질문(수렴될 때까지 반복) → 합의 도출.
*   **와이드밴드 델파이 (Wideband Delphi)**: 일반 델파이 기법에 시나리오 회의(Work Session)를 결합하여, 전문가들이 단순히 수치만 제시하는 것이 아니라 **WBS(Work Breakdown Structure)**를 기반으로 토론하도록 유도합니다. 이를 통해 편차를 30% 이하로 줄일 수 있습니다.

#### 2. 수학적 알고리즘 기반 (Algorithmic Model)
가장 대표적인 **COCOMO (Constructive Cost Model)**는 보헴(Barry Boehm)이 제안한 모델로, 프로젝트 특성에 따라 기본 공식(Basic)을 수정합니다.

*   **상관식 (E = a × (LOC)^b)**
    *   `E`: Effort (Person-Month, 노력 공수)
    *   `LOC`: KLOC (Kilo Lines of Code)
    *   `a`, `b`: 프로젝트 특성 계수

*   **모드별 계수 비교 (Table)**

| 모드 (Mode) | 특징 (Characteristics) | 규모 (Size) | 계수 (a, b) | 비고 |
|:---|:---|:---|:---|:---|
| **Organic** (유기적) | 사내 업무, 소규모, 복잡도 낮음 | < 50 KLOC | 2.4, 1.05 | 개발자 자율성 높음 |
| **Semi-Detached** (반분리) | 유틸리티, 중간 복잡도, 중간 규모 | < 300 KLOC | 3.0, 1.12 | 일반적 산업용 SW |
| **Embedded** (내장형) | 실시간 처리, 하드웨어 제약, 고복잡 | > 300 KLOC | 3.6, 1.20 | 항공우주, 방위, OS |

*   **COCOMO II**: 재사용(Reuse)성과 비즈니스 요구를 반영하기 위해 3단계 모델로 진화했습니다.
    1.  **Application Composition**: 프로토타이핑 단계, Object Points 사용.
    2.  **Early Design**: 초기 설계 단계, Function Points 사용.
    3.  **Post-Architecture**: 상세 설계 후, LOC 및 17개의 비용 견인인자(Cost Drivers) 적용.
        *   예: `RELY`(신뢰도 요구 수준), `DATA`(데이터베이스 크기), `CPLX`(복잡도)

#### 3. 기능 점수 기반 (Function Point)
코드 양이 아닌 **사용자가 인식하는 기능(Function)**의 개수를 기준으로 합니다. 언어 독립적(Language Independent)이므로, COBOL로 짜든 Java로 짜든 같은 기능이라면 동일한 점수를 부여합니다.

```ascii
[ FP (Function Point) 산정 구조도 ]

    [ 사용자 요구 (User Requirements) ]
                 |
      +----------+----------+
      |                     |
  [ 데이터 기능 ]      [ 트랜잭션 기능 ]
  (Data Function)     (Transaction Function)
      |                     |
  +---+---+           +-----+-----+
  |       |           |     |     |
 ILF     EIF         EI    EO    EQ
(내부)  (외부)      (입력) (출력) (조회)

  [ Count ] --- [ Complexity (Low/Mid/High) ] --- [ UFP (Unadjusted FP) ]
                                                              |
                                                      +-------+-------+
                                                      |               |
                                              [ VAF (Value Adj Factor) ]
                                              (14개 GSC: 일반적 시스템 특성)
                                                      |
                                               [ Final FP ]
                                          FP = UFP × (0.65 + 0.01 × Σ(GSC))
```

**FP 핵심 요소 심층 분석**:
1.  **ILF (Internal Logical File)**: 시스템 내부에서 관리하는 데이터 집합 (예: 회원 DB 테이블).
2.  **EIF (External Interface File)**: 외부 시스템에서 관리하지만 내 시스템이 참조하는 데이터 (예: 타행 잔고 조회).
3.  **EI (External Input)**: 데이터를 내부 로직으로 가공하여 ILF를 갱신하는 입력 (예: 회원가입 API).
4.  **EO (External Output)**: 데이터 가공 처리 후 외부로 내보내는 출력 (예: 월간 판매 리포트 PDF).
5.  **EQ (External Inquiry)**: 단순 조회 및 출력 (가공 로직 최소화, 예�: 상세 조회).

**📢 섹션 요약 비유**: COCOMO는 **'건설 현장의 토공 계산기'**와 같습니다. 땅의 넓이(LOC)를 재고, 땅의 성질(모드)을 고려해 불도저로 얼마나 파야 할지 시간을 잽니다. 반면 FP는 **'인테리어 옵션 견적서'**입니다. 싱크대가 몇 개고, 전등이 몇 개인지(ILF/EI)를 세어서 주거 가치를 매깁니다. 델파이는 이 견적서를 쓸 때 **'시공 전문가 회의'**를 열어 한 명의 실수가 큰 사고로 이어지지 않게 상호 검증하는 절차입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

각 산정 기법은 독립적으로 존재하지 않으며, 프로젝트의 라이프사이클(Life Cycle)과 밀접하게 연관됩니다. 또한, 데이터베이스의 성능이나 네트워크 대역폭과 같은 타 영역의 지표와도 연결됩니다.

#### 1. 기술적 비교 분석 (Table)

| 구분 | LOC (Line of Code) | FP (Function Point) | COCOMO |
|:---|:---|:---|:---|
| **측정 단위** | 소스 코드 라인 수 (물리적) | 사용자 기능 수 (논리적) | 인월 (Person-Month) |
| **측정 시점** | 코딩 완료 후 (사후) | 요구사항 분석 단계 (사전) | 설계 단계 (중기) |
| **언어 의존성** | 높음 (C vs Python 차이 큼) | 없음 (언어 무관) | 있음 (LOC를 입력으로 사용 시) |
| **생산성 지표** | LOC/PM (라인/월) | FP/PM (기능/월) | 수식에 의해 자동 산출 |
| **주요 용도** | 소규모 프로젝트, 유지보수 | 대규모 통합 프로젝트, 견적 | 통계적 예측, 일정 산정 |

#### 2. 타 과목 융합 분석 (Subject Integration)

*   **데이터베이스 (DB)**: FP 산정 시 **ILF(내부 논리 파일)**와 **EIF(외부 인터페이스 파일)**의 개수는 곧 데이터베이스의 **Entity(개체)** 수와 1:1 매핑됩니다. 정규화(Normalization)가 잘 된 데이터 모델일수록 ILF의 복잡도가 낮아져 FP 점수가 안정화됩니다.
*   **운영체제 (OS)**: COCOMO의 **Embedded(내장형)** 모드는 OS의 자원 관리(Resource Management)와 밀접합니다. 메모리 제약이 심한 임베디드 시스템일수록 코드 최적화 비용이 비싸지므로 `b` 지수(스케일링 팩터)가 1.2로 높게 책정됩니다.
*   **네트워크 (Network)**: **EQ(External Inquiry)**나 **EO(External Output)**가 빈번한 시스템(채팅, 실시간 모니터링)은 네트워크 트래픽이 FP 점수에 반영되지 않으므로, 별도로 네트워크 대역폭 비용을 산정(예: Mbps당 비용)하여 합산해야 합니다.

```ascii
[ 산정 기법 간의 시계열적 융합 관계 ]

   [ 초기 요청 ]  ---(Delphi/Wideband)--->  [ 상세 요구사항 정의 ]
         |                                          |
         | (예측)                                   | (측정)
         v                                          v
   [ 개략 견적 ] <--------------------------- [ FP 분석 ]
   (COCOMO Basic)                                |
         |                                        | (코드 변환)
         v                                        v
   [ 설계 단계 ] ---(COCOMO II)-------------> [ 상세 설계 ]
         |                                        |
         | (작업량 산출)                          | (구현)
         v                                        v
   [ 코딩 ] <----------------------------- [ 구현 소스코드 ]
                                          (LOC 측정)
         |
         v
   [ 실제 투입 공수 (Actual Effort) ]
      ^                                      |
      |-------------- 피드백 (Feedback Loop) ----|
      (향후 프로젝트를 위한 보정 계수 α 업데이트)
```

**📢 섹션 요약 비유**: 이 기법들은 **'여행 경로 계획'**을 세울 때 서로 다른 역할을 합니다. **델파이**는 여행 가자들이 모여 "어디가 좋을까?"라고 머리를 맞대는 **'회의'**입니다. **FP**는 "우리가 몇 개의 도시를 방문할 거야?"를 세는 **'일정표 작성'**입니다. **COCOMO**는 "도로의 상태와 운전자의 숙련도를 고려해 몇 시간 걸릴까?"를 계산하는 **'내비게이션 예상 시간'**입니다. 이 셋을 모두 합쳐야만 야간에 운전할 때 발생할 수 있는 위험(변동성)을 피할 수 있습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서는
