# 프로토타입 모델 (Prototype Model) 상세 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로토타입 모델(Prototype Model)은 시스템의 핵심 기능을 구현한 **시제품(Prototype)**을 신속히 제작하여 사용자의 피드백을 받고, 이를 바탕으로 요구사항을 정제하며 완성도를 높여가는 점진적 생명주기 모델이다.
> 2. **가치**: 폭포수 모델의 최대 약점인 '불명확한 요구사항'과 '사용자 소외' 문제를 해결하며, 실제 작동하는 모델을 통해 **개발 위험을 조기에 감소**시키고 사용자 만족도를 극대화한다.
> 3. **융합**: 현대의 애자일(Agile) 방법론과 MVP(Minimum Viable Product) 전략의 정신적 모태이며, UX/UI 디자인 중심의 현대적 앱 개발 프로세스에서 필수적인 검증 단계로 자리 잡았다.

+++

## Ⅰ. 프로토타입 모델 (Prototype Model)의 개요

### 1. 정의
- 본격적인 개발에 앞서 사용자의 요구사항을 정확히 파악하기 위해 견본품(시제품)을 만들어 최종 시스템의 모습을 예측하는 모델이다.

### 2. 등장 배경: "백문이 불여일견"
- 사용자는 자신이 무엇을 원하는지 문서만 보고는 정확히 알지 못하는 경우가 많다.
- "일단 만들어 봐라, 그럼 내가 고쳐주겠다(I'll know it when I see it)"라는 사용자의 심리를 공학적으로 수용한 결과다.

### 3. 💡 비유: 모델하우스
- 아파트를 짓기 전에 만드는 **'모델하우스'**와 같다.
- 설계도만 봐서는 방 크기가 감이 안 오지만, 모델하우스를 직접 걸어보고 주방 구조를 보면 "여기에 수납장을 더 넣어주세요"라고 정확한 요구(요구사항 정제)를 할 수 있는 것과 같다.

+++

## Ⅱ. 프로토타입 모델의 반복적 프로세스 (ASCII)

요구사항이 확정될 때까지 순환 루프를 도는 것이 특징이다.

```text
    [ 1. 요구사항 수집 ] ──┐
           ▲               │
           │               ▼
    [ 4. 고객의 평가 ] ◀─ [ 2. 빠른 설계 및 ]
           │              [ 프로토타입 구축 ]
           │               │
           └─────── [반복] ──┘
                   │ (만족 시)
                   ▼
    [ 5. 상세 설계 및 구현 ] ──▶ [ 6. 테스트 및 유지보수 ]
```

**[핵심 단계별 활동]**
1. **빠른 설계 (Quick Design)**: 세부적인 내부 로직보다는 사용자 인터페이스(UI)와 데이터 흐름 위주로 설계한다.
2. **프로토타입 구축**: 실제 데이터베이스 연동 등은 생략하거나 가짜 데이터(Mock-up)를 사용하여 눈에 보이는 기능만 신속히 만든다.
3. **사용자 평가**: 사용자가 직접 시제품을 써보며 수정 사항과 추가 요구사항을 쏟아낸다.

+++

## Ⅲ. 프로토타입의 유형 분석

프로토타입을 대하는 목적에 따라 크게 두 가지로 나뉜다.

| 구분 | 폐기형 프로토타입 (Throw-away) | 진화형 프로토타입 (Evolutionary) |
|:---|:---|:---|
| **목적** | 요구사항의 명확화 및 타당성 검토 | **실제 제품의 근간**으로 활용 |
| **운명** | 확인 후 버림 (본 개발은 새로 시작) | 지속적으로 보완하여 최종 제품 완성 |
| **품질** | 외형 중심 (Quick & Dirty) | 내실 중시 (확장성, 견고성) |
| **비유** | 건축 전의 스케치 | 레고 로봇의 뼈대부터 완성까지 |

- **📢 섹션 요약 비유**: 맛보기 스푼으로 아이스크림 맛만 보여주는 것이 폐기형이라면, 빵 시트부터 구워가며 생크림을 올리는 과정 전체를 공유하는 것이 진화형입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. 장점 vs 단점의 트레이드오프
- **장점**: 개발자와 사용자 간의 **의사소통 가시성** 확보, 요구사항 변경에 따른 재작업 비용 절감.
- **단점**: 사용자가 시제품을 완성품으로 착각하여 조기 출시를 압박하거나, 프로토타입 제작에 과도한 시간과 비용이 소요될 수 있음.

### 2. 기술사적 결단
- **인사이트**: 모든 프로젝트에 프로토타입을 적용할 필요는 없다. 
- **결단 시나리오**:
  - **신규 비즈니스 모델**: 시장 반응을 알 수 없으므로 **필수**.
  - **정형화된 관리 시스템**: 요구사항이 명확하므로 생략 가능.
- **실무 팁**: "프로토타입의 목적은 학습이다"라는 원칙을 세우고, 불필요한 고퀄리티 작업에 매몰되지 않도록 범위를 제한(Scoping)해야 한다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **리스크 감소**: 구현 불가능한 기능을 초기에 발견하여 프로젝트 실패 확률 감소.
- **심리적 안정**: 사용자가 진행 상황을 눈으로 보게 되어 개발팀에 대한 신뢰도 향상.

### 2. 미래 전망
최근에는 피그마(Figma)와 같은 디자인 도구의 발전으로 실제 코딩 없이도 고해상도(Hi-Fi) 프로토타이핑이 가능해졌다. 또한 코드가 자동으로 생성되는 로우코드(Low-code) 기술과 결합하여, **'프로토타입이 곧 제품'**이 되는 진화형 모델의 경계가 더욱 허물어질 것이다.

- **📢 섹션 요약 비유**: 어둠 속에서 촛불 하나를 들고 길을 밝히며 한 걸음씩 나아가는 정성스러운 탐험과도 같은 모델입니다.

+++

### 📌 관련 개념 맵 (Knowledge Graph)
- **나선형 모델 (Spiral Model)**: 프로토타입에 '위험 분석'을 더한 확장판.
- **MVP (Minimum Viable Product)**: 현대 스타트업이 채택한 진화형 프로토타입의 실전판.
- **사용자 인터페이스 (UI/UX)**: 프로토타입이 가장 큰 힘을 발휘하는 영역.
