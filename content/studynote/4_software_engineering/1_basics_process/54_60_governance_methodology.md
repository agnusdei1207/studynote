# 54-60. 거시적 관리와 방법론 (Brooks, EA, PMO)

## # 거시적 관리와 방법론 (Brooks, EA, PMO)

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 프로젝트의 비선형적 성장 이슈를 이해하고, 전사적 차원(EA)에서 구조를 정의하며, 전담 조직(PMO)을 통해 프로젝트를 통제하는 시스템 엔지니어링 거버넌스 체계임.
> 2. **가치**: 브룩스의 법칙(Brooks's Law)을 통해 인력 추가의 함정을 피하고, EA (Enterprise Architecture)를 통해 중복 투자를 방지하며, PMO (Project Management Office)를 통해 조직의 프로젝트 성공률을 획기적으로 개선함.
> 3. **융합**: 경영학적 리스크 관리와 IT 아키텍처 설계, 그리고 애자일(Agile) 방법론 등 현대 개발 프로세스가 결합된 기업 경영의 최적화 모델.

---

### Ⅰ. 개요 (Context & Background)

#### 개념 및 철학
소프트웨어 공학(Software Engineering)에서 단순히 코딩(Coding) 기량을 넘어 프로젝트와 조차 전체를 관리하는 '거시적 안목'은 필수적입니다. 이는 **"도시를 설계할 때 건물 하나의 미관보다는 도로와 상하수도, 전력망의 연결성을 고려해야 한다"**는 원리와 같습니다. 여기서 핵심이 되는 세 가지 축은 **브룩스의 법칙(Brooks's Law)**, **전사적 아키텍처(Enterprise Architecture)**, 그리고 **프로젝트 관리 사무소(PMO, Project Management Office)**입니다.

#### 등장 배경
1.  **소프트웨어 위기(Software Crisis)의 지속**: 1960년대 이후 프로젝트 규모가 커지면서 일정과 비용이 초과되는 문제가 만연했습니다. 특히, **프레드 브룩스(Fred Brooks)**가 '맨먼스 미신(The Mythical Man-Month)'에서 지적한 것처럼, "지체되는 프로젝트에 인력을 투입하는 것은 불을 기름으로 끄는 것"과 같은 현상이 발생했습니다.
2.  **정보화의 파편화**: 기업 내各部门(Dept)마다 개별적으로 시스템을 구축하면서 데이터 중복(Data Redundancy)과 시스템 간 비호환성 문제가 대두되었습니다. 이를 해결하기 위해 경영 전략과 IT 전략을 정렬(Alignment)하는 **EA (Enterprise Architecture)** 개념이 등장했습니다.
3.  **프로젝트 통제의 필요성**: 수많은 프로젝트를 개별적으로 운영할 경우 자원이 낭비되고 품질이 편차되므로, 이를 통합 관리하고 표준화된 **방법론(Methodology)**을 적용할 전담 조직인 **PMO (Project Management Office)**의 필요성이 강조되었습니다.

> **📢 섹션 요약 비유**: 기업 IT 관리의 역사는 마치 단순히 가옥을 짓던 것에서 시작해, 도시 계획(EA)을 수립하고 시공사를 감독하는 건설 관리공사(PMO)를 두며, 인건비를 무리하게 늘리면 공사가 더 늦어진다는 교훈(Brooks)을 얻는 과정과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 브룩스의 법칙 (Brooks's Law)의 메커니즘
프레드 브룩스(Fred Brooks)는 IBM OS/360 개발 팀을 관리하며 깨달은 바를 정리하여, **"지체되는 소프트웨어 프로젝트에 인력을 추가로 투입하는 것은 프로젝트를 더욱 지연시킬 뿐이다"**라고 정의했습니다.

*   **수학적 모델**: 팀 내 의사소통 채널(Channel)의 수 $C$는 팀원 수 $n$에 대해 다음과 같이 증가합니다.
    $$ C = \frac{n(n-1)}{2} $$
    즉, 인원이 2명에서 3명으로 늘어나면 채널은 1개에서 3개로 늘어나는 것이 아니라, 3명에서 4명으로 늘어날 때 3개에서 6개로 급증하는 기하급수적(Quadratic) 복잡도를 가집니다.

#### 2. 전사적 아키텍처 (EA, Enterprise Architecture) 구조
EA는 경영 층면의 전략과 IT 기술 층면의 구현을 매핑한 청사진(Blueprint)입니다.

| 구성 요소 | 상세 설명 및 역할 | 내부 동작 및 메커니즘 | 주요 산출물 |
|:---|:---|:---|:---|
| **비즈니스 아키텍처 (BA)** | 기업의 전략, 조직, 프로세스 정의 | **전략 → 프로세스**: 비즈니스 목표 달성을 위한 기능 분석(FEA) | 조직도, 프로세스 맵 |
| **데이터 아키텍처 (DA)** | 정보 자산의 구조와 흐름 정의 | **엔티티 관계**: 데이터의 저장, 전송, 표준화 규칙 정의 | ERD, 데이터 사전 |
| **애플리케이션 아키텍처 (AA)** | 비즈니스 기능을 수행하는 SW 시스템 정의 | **시스템 연계**: 각 시스템 간 인터페이스 및 의존성 관리 | 시스템 현황도, 인터페이스 명세서 |
| **기술 아키텍처 (TA)** | HW, NW, SW 등 IT 인프라 정의 | **서비스 제공**: 애플리케이션 구동을 위한 기술 플랫폼 | 네트워크 구성도, 서버 사양 |

#### 3. PMO (Project Management Office) 유형별 거버넌스
PMO는 프로젝트를 단순히 관리하는 것을 넘어, 조직의 전략적 자산인 '프로젝트 관리 역량'을 강화하는 조직입니다.

*   **지원형 (Supportive PMO)**: 템플릿, 방법론, 교육 제공 (통제력 약함).
*   **통제형 (Controlling PMO)**: 강제성 있는 표준 프로세스 준수 감사 (合规 audit).
*   **지시형 (Directive PMO)**: PMO가 직접 프로젝트 관리자(PM)를 임명하고 수행 (통제력 강함).

#### 4. 방법론 테일러링 (Methodology Tailoring)
표준 방법론(예: RUP, Scrum)은 '기성복'과 같습니다. 이를 프로젝트의 규모, 위험도, 조직 문화에 맞게 수정(Tailoring)하여 사용해야 합니다. 예를 들어, **RAD (Rapid Application Development)** 모델을 적용할 때도, 대외 시스템 연동이 많으면 문서화(Documentation) 비중을 높이는 식의 조정이 필요합니다.

```ascii
      [ 브룩스의 법칙 시뮬레이션 : 의사소통 경로의 폭발 ]

      3인 팀 (채널 3개)          4인 팀 (채널 6개)          6인 팀 (채널 15개)

      A ------- B                A --------- B              A --------- B
       \       /                  \       /  \               \       / | \
        \     /                    \     /    \               \     /  |   \
         \   /                      \   /      \               \   /   |    \
          C                          D          C               F -- D -- C -- E
      
      [Communication: 3]          [Communication: 6]        [Communication: 15]
      
      ※ 인원 2배 증가 시, 업무량은 2배가 아니라 5배(15/3)로 증가함.
```

> **📢 섹션 요약 비유**: 브룩스의 법칙은 "요리사가 모자라는데 주방에 새로운 요리사를 투입하면, 레시피를 설명해주느라 기존 요리사의 속도가 느려져 오히려 요리가 늦어진다"는 원리입니다. EA는 "도시의 가스관, 전기, 수도를 포함한 지하 지도"이며, PMO는 "각 공구(프로젝트)에 사람을 배치하고 품질을 검사하는 본사"입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. PMO 유형별 기술적/관리적 특성 비교

| 구분 | 지원형 (Supportive) | 통제형 (Controlling) | 지시형 (Directive) |
|:---|:---:|:---:|:---:|
| **권한 수준 (Level)** | 낮음 (Low) | 중간 (Medium) | 높음 (High) |
| **핵심 역할** | 컨설턴트 (Consultant) | 감사관 (Auditor) | 집행자 (Executor) |
| **방법론 적용** | 권장 (Recommend) | 강제 권고 (Enforce) | 직접 수행 (Direct) |
| **주요 산출물** | 가이드, 교육 자료 | 평가 보고서, KPI 지표 | 프로젝트 계획서, 성과 보고 |
| **적합 상황** | 초기 도입 단계, 숙련된 PM 부재 | 성숙한 조직, 품질 안정화 필요 | 대규모 프로젝트, 위험도 높음 |

#### 2. EA 프레임워크 비교 (Zachman vs TOGAF)

| 비교 항목 | 잭맨 프레임워크 (Zachman Framework) | TOGAF (The Open Group Architecture Framework) |
|:---|:---|:---|
| **성격** | **분류(Taxonomy)** 중심: 건물의 부재 자재를 나열하는 '카탈로그' | **공정(Process)** 중심: 건물을 짓는 '건설 매뉴얼' |
| **구조** | 2차원 6x6 매트릭스 (What, How, Where, Who, When, Why) | ADM (Architecture Development Method) 순환 주기 |
| **활용성** | 현황 파악 및 갭(Gap) 분석에 유리 | 실무 구현 및 이행(Implementation)에 유리 |
| **융합 관점** | **정보 시스템(IS)** 계획 수립 시 기준 | **시스템 엔지니어링(SE)** 및 **ITIL**과 연계 운영 |

#### 3. 기술 융합 분석 (SE + Management)
*   **시스템 엔지니어링(SE) 관점**: 브룩스의 법칙은 시스템의 복잡도(Complexity) 관리 문제입니다. 이를 해결하기 위해 모듈화(Modularity)와 결합도 낮추기(Low Coupling)를 통해 추가 인원 투입 없이 생산성을 높이는 아키텍처가 필요합니다.
*   **비즈니스 관점**: EA는 단순히 IT 구조가 아니라 **BPR (Business Process Reengineering)**의 도구입니다. 비즈니스 프로세스가 변하면 IT 아키텍처도 유기적으로 변해야 하며, 이를 PMO가 통제합니다.

```ascii
        [ 전사적 거버넌스 실행 흐름 : Strategy -> Execution ]

  ┌───────────────────────┐
  │  전사 전략 (Strategy)  │
  └───────────┬───────────┘
              │
              ▼
  ┌───────────────────────┐       ┌───────────────────────┐
  │  EA (Blueprint)        │ ────▶ │  PMO (Controller)      │
  │  - Target Architecture │       │  - Methodology        │
  │  - Gap Analysis        │       │  - Resource Mgmt      │
  └───────────────────────┘       └───────────┬───────────┘
              ▲                          │
              │                          ▼
  ┌───────────────────────┐       ┌───────────────────────┐
  │  Actual Implementation │ ◀─── │  Tailored Process     │
  │  (Projects)            │       │  - Methods            │
  └───────────────────────┘       │  - Templates          │
                                   └───────────────────────┘
```

> **📢 섹션 요약 비유**: EA와 PMO의 관계는 "도시 설계도(EA)"와 "교통 통제 센터(PMO)"의 관계와 같습니다. 아무리 훌륭한 설계도가 있어도, 이를 집행하거나 감독하는 조직이 없으면 무단으로 건물을 짓거나(개별 프로젝트 방치), 도로가 막히는 문제가 발생합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 시나리오 및 의사결정

**시나리오 A: 납기가 임박한 대규모 공공 SI 프로젝트**
*   **상황**: 핵심 개발자 2명이 퇴사하고 납기가 3개월 남았음. 경영진은 인력 10명 추가 투입을 제안.
*   **기술사적 판단**: **인력 추가 반대**. 브룩스의 법칙에 따라 기존 멤버가 신규 인원을 교육(Onboarding)하는 비용이 득보다 큽니다.
*   **대안**: 프로젝트 범위(Scope)에서 우선순위를 선정하여 필수 기능만 먼저 납품하거나, 남은 핵심 개발자의 사기를 위해 인센티브 제공 및 병목 구간의 병렬 처리를 검토.

**시나리오 B: 중견 기업의 IT 투자 중복 발생**
*   **상황**: 영업팀 CRM, 인사팀 HR 시스템 구축 시 각각 다른 클라우드 서비스와 DB를 사용하여 데이터 연동이 안 됨.
*   **기술사적 판단**: **EA 도입 필요**. 각 사업부(Unit)가 독립적인 'Silos(사일로)'를 형성하고 있음.
*   **대안**: 기업 차원의 **데이터 아키텍처(DA)**를 수립하여 표준 DB 플랫폼을 정의하고, 향후 구축 시 이를 준수하도록 **통제형 PMO**를 통해 예산 승인 권한을 부여.

#### 2. 방법론 테일러링 실무 체크리스트
프로젝트 관리자는 다음 기준에 따라 방법론을 수정(Tailoring)해야 합니다.

| 구분 | 검토 항목 | 질문 (Question) |
|:---|:---|:---|
| **규모** | 프로젝트 인원 (MM) | 10인 미만인가? 100인 이상인가
