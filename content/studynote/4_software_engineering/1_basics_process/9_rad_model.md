# RAD (Rapid Application Development) 모델

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RAD 모델은 강력한 소프트웨어 개발 도구(CASE)와 사용자 참여 기법(JAD)을 활용하여, **60~90일이라는 극히 짧은 기간** 내에 시스템을 구축하는 고속 개발 모델이다.
> 2. **가치**: 불필요한 문서화 대신 **동작하는 프로토타입**과 재사용 가능한 컴포넌트(Component)에 집중하여, 비즈니스 환경 변화에 즉각적으로 대응할 수 있는 속도 경쟁력을 제공한다.
> 3. **융합**: 현대 로우코드(Low-code) 플랫폼과 애자일(Agile)의 프로토타이핑 정신의 원형이며, 기술적 위험도가 낮고 요구사항이 명확한 비즈니스 시스템 구축에 최적화된 전략이다.

+++

## Ⅰ. RAD (Rapid Application Development) 모델의 개요

### 1. 정의
- 1991년 제임스 마틴(James Martin)이 제안한 모델로, 정해진 짧은 기간 내에 개발을 완료하기 위해 CASE 도구, 컴포넌트 재사용, 사용자 참여를 극대화하는 순차적 프로세스 모델이다.

### 2. 등장 배경: "시간은 곧 돈이다"
- 전통적인 폭포수 모델은 개발 기간이 너무 길어, 시스템이 완성될 쯤에는 이미 비즈니스 환경이 바뀌어 있는 '소프트웨어 지연' 현상이 심각했다.
- "완벽한 문서보다 빨리 돌아가는 프로그램이 먼저다"라는 절박함이 RAD를 탄생시켰다.

### 3. 💡 비유: 조립식 주택 건설
- RAD는 **"조립식 주택"**을 짓는 것과 같다.
- 벽돌을 하나하나 구워서 쌓는 대신(밑바닥 코딩), 이미 공장에서 만들어진 벽면과 창문(컴포넌트)을 가져와서 중장비(CASE 도구)로 빠르게 조립한다. 집주인(사용자)은 매일 현장에 와서 배치를 확인하고 즉시 수정 의견을 낸다.

+++

## Ⅱ. RAD 모델의 핵심 단계 및 구성 요소 (ASCII)

RAD는 분석보다는 '구축'과 '피드백'의 병렬적 수행을 강조한다.

```text
    [ 1. JAD (Joint Application Design) ]
    - 사용자와 개발자가 한자리에 모여 요구사항을 즉석에서 결정
              │
    ┌─────────▼─────────┐
    │  [ 2. 반복적 구축 ] │ <──▶ [ CASE 도구 활용 ]
    │  (Construction)   │      (Code Generator, GUI Builder)
    └─────────┬─────────┘
              │ (지속적 피드백)
    ┌─────────▼─────────┐
    │  [ 3. 실행 및 인도 ]│ ──▶ [ 60~90일 이내 완료 ]
    │  (Cutover)        │
    └───────────────────┘
```

### 1. 3대 핵심 성공 요소
| 요소 | 상세 설명 | 역할 |
|:---|:---|:---|
| **CASE 도구** | 설계부터 코드 생성까지 자동화하는 소프트웨어 | 개발 속도의 비약적 향상 |
| **JAD 기법** | 사용자와 전문가가 집중 토론하는 합동 회의 | 의사결정 지연 최소화 |
| **컴포넌트 재사용** | 이미 검증된 소프트웨어 부품 활용 | 품질 안정성 및 기간 단축 |

+++

## Ⅲ. RAD 모델의 장단점 및 적용 가이드

### 1. 장점 vs 단점
| 구분 | 장점 (Pro) | 단점 (Con) |
|:---|:---|:---|
| **비즈니스** | 시장 선점 가능 (매우 빠른 출시) | 대규모 시스템에는 부적합 (관리 어려움) |
| **품질** | 사용자 만족도 매우 높음 | 특정 툴(CASE)에 대한 의존도 및 종속성 |
| **인력** | 소수 정예 전문가의 생산성 극대화 | 프로젝트 관리에 강력한 리더십 요구 |

### 2. 성공을 위한 전제 조건
- **기술적 성숙도**: 팀원들이 CASE 도구와 재사용 라이브러리에 매우 능숙해야 함.
- **사용자 가용성**: 사용자가 개발 기간 내내 피드백을 줄 수 있는 시간이 확보되어야 함.
- **모듈화 가능성**: 시스템이 독립적인 기능들로 깔끔하게 쪼개질 수 있어야 함.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. Time-box 관리 기법
- **기술사적 결단**: RAD의 핵심은 **'Time-boxing'**이다. 기능이 덜 만들어졌더라도 약속한 90일이 되면 무조건 종료(또는 릴리스)해야 한다. 
- **인사이트**: "완벽하지 않아도 좋다, 하지만 늦어서는 안 된다"는 비즈니스적 판단이 공학적 설계보다 우선시되는 모델임을 명확히 인지해야 한다.

### 2. 현대적 변천: Low-code / No-code
- **판단**: 과거의 CASE 도구는 현재의 **OutSystems, Mendix** 혹은 클라우드 기반의 **Serverless 아키텍처**로 진화했다. 
- **결론**: 현대의 IT 거버넌스에서 RAD는 단순한 개발 모델을 넘어, 비즈니스 현업 부서가 직접 앱을 개발하는 **'Citizen Developer'** 환경의 이론적 토대가 되고 있다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **비용 절감**: 짧은 개발 기간으로 인한 인건비 및 관리비 최소화.
- **요구사항 정합성**: JAD를 통해 '사용자가 정말로 원했던' 시스템이 나올 확률 증대.

### 2. 미래 전망
AI(ChatGPT, Claude 등)가 코드를 자동 생성해주는 시대가 오면서, RAD 모델은 다시금 전성기를 맞이하고 있다. 인간은 JAD를 통해 '무엇을 만들지' 결정만 하고, AI라는 궁극의 CASE 도구가 '어떻게'를 순식간에 해결해주는 **'AI-driven RAD'**가 미래 소프트웨어 공학의 표준이 될 것이다.

- **📢 섹션 요약 비유**: 정석대로 바둑을 두는 것이 폭포수라면, 상대의 수에 따라 즉각적으로 대응하며 번개처럼 대국을 끝내는 속기 바둑이 바로 RAD입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **CASE (Computer Aided SE)**: RAD를 가능케 하는 기술적 수단.
- **JAD (Joint Application Design)**: RAD의 의사결정 메커니즘.
- **애자일 방법론 (Agile)**: RAD의 정신을 계승한 현대 방법론.

### 👶 어린이를 위한 3줄 비유 설명
1. RAD는 **'번개처럼 빠른 로봇 만들기'**와 같아요.
2. 처음부터 하나하나 깎아 만드는 게 아니라, 미리 만들어진 로봇 팔다리를 가져와서 전용 기계로 뚝딱 조립하는 거죠.
3. "이 로봇이 맞나요?"라고 계속 물어보면서 아주 빨리 만들기 때문에, 생일 파티 전까지 멋진 선물을 금방 완성할 수 있답니다!
