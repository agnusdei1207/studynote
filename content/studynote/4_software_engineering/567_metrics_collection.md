# 567. 메트릭 (Metrics) - 시계열 데이터 수집 (Prometheus, Grafana)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메리트릭 (Metrics)은 시간의 흐름에 따른 시스템의 상태 변화를 수치화한 **시계열 데이터 (Time-series Data)**로, 수집(Pull/Push), 저장(TSDB), 시각화 단계를 거쳐 시스템의 가용성과 성능을 정량적으로 관리하는 기술이다.
> 2. **가치**: 실시간 지표 분석을 통해 장애 징후를 사전에 탐지하고 임계치 기반의 **지능형 알람(Alerting)**을 수행하며, 인프라의 확장성(Scalability) 및 자원 활용 효율성을 최적화하는 의사결정 근거를 제공한다.
> 3. **융합**: 프로메테우스(Prometheus)의 강력한 다차원 데이터 모델과 그라파나(Grafana)의 시각화 역량이 결합되어, 클라우드 네이티브 환경의 표준 모니터링 스택을 형성하며 SRE 실천의 핵심 도구로 활용된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 메트릭은 "시스템이 내뱉는 숫자들의 기록"이다. CPU 사용률, 초당 트랜잭션 수(TPS), 응답 시간(Latency), 에러율 등 시스템의 건강 상태를 나타내는 지표들을 일정한 시간 간격으로 측정하여 저장한 데이터다.
- **💡 비유**: 메트릭은 환자의 상태를 실시간으로 보여주는 **"병원 중환자실의 생체 신호 모니터"**와 같습니다. 심박수(CPU), 혈압(Network), 산소포화도(Memory)가 그래프로 그려지며, 수치가 정상 범위를 벗어나면 즉시 경보음(Alert)이 울리는 것과 같습니다.
- **등장 배경**:
    1. **정량적 관리의 필요성**: "시스템이 좀 느린 것 같다"는 주관적 판단 대신, "응답 시간이 200ms에서 500ms로 늘어났다"는 객관적 지표가 필요해졌다.
    2. **동적 인프라의 복잡성**: 오토스케일링으로 서버 대수가 수시로 변하는 환경에서, 각 서버의 상태를 개별적으로 보는 대신 '서비스 전체의 합산 수치'를 추적해야 했다.
    3. **대규모 데이터 처리**: 매초 발생하는 수만 개의 지표를 일반 관계형 DB(RDBMS)에 저장하기엔 오버헤드가 너무 컸으며, 시계열 전용 저장소(TSDB)가 요구되었다.

- **📢 섹션 요약 비유**: 과거에는 고장 난 뒤에 수리하는 사후 약방문이었다면, 메트릭은 실시간 수치 변화를 관찰하여 사고를 미연에 방지하는 '스마트 예방 정비' 시스템입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

메트릭 수집 및 시각화는 수집 엔진(Prometheus)과 시각화 엔진(Grafana)의 협업으로 이루어진다.

| 구성 요소 | 역할 및 정의 | 핵심 기술 | 비유 |
|:---:|:---|:---|:---|
| **Exporter** | 대상 시스템의 지표를 추출 | Node Exporter, JMX Exporter | 건강 검진 센서 |
| **Prometheus Server** | 데이터 수집 및 시계열 저장 | **Pull 방식**, TSDB, PromQL | 종합 상황실 및 장부 |
| **TSDB** | 시계열 데이터 최적화 저장소 | 효율적인 압축 및 인덱싱 | 날짜별 기록 보관소 |
| **Grafana** | 다차원 데이터 시각화 | Dashboard, Panel, Variable | 대형 전광판 |
| **Alertmanager** | 임계치 도달 시 알림 전송 | Slack, Email, PagerDuty 연동 | 비상 사이렌 |

#### [메트릭 수집 및 시각화 아키텍처 (Prometheus-Grafana Stack)]

지표가 생성되어 대시보드에 그려지기까지의 엔드-투-엔드 흐름을 보여준다.

```text
 ┌──────────────────────────┐          ┌──────────────────────────────────────┐
 │ Target (App/Server)      │          │      Prometheus Server               │
 │ ┌──────────────────────┐ │          │ ┌──────────────────────────────────┐ │
 │ │  /metrics endpoint   │◀──(Pull)───┼─▶│ Retrieval Engine (Scraper)      │ │
 │ └──────────────────────┘ │          │ └──────────────┬───────────────────┘ │
 └──────────────────────────┘          │                ▼                     │
                                       │ ┌──────────────────────────────────┐ │
 ┌──────────────────────────┐          │ │  TSDB (Time Series Database)     │ │
 │ Pushgateway (Ephemeral)  │──(Push)─▶│ │  - Metrics: cpu_usage{node="A"}  │ │
 └──────────────────────────┘          │ └──────────────┬───────────────────┘ │
                                       └────────────────┼─────────────────────┘
                                                        │
                                                        ▼ (PromQL Query)
 ┌──────────────────────────┐          ┌──────────────────────────────────────┐
 │      Grafana             │          │      Alertmanager                    │
 │ ┌──────────────────────┐ │          │ ┌──────────────────────────────────┐ │
 │ │  Rich Dashboards     │◀───────────┤ │  Notification (Slack/SMS)        │ │
 │ └──────────────────────┘ │          │ └──────────────────────────────────┘ │
 └──────────────────────────┘          └──────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **Pull 기반 수집**: 프로메테우스는 대상 서버에 직접 접속하여 지표를 긁어오는(Pull) 방식을 취한다. 이는 대상 서버의 부하를 조절하기 쉽고 중앙 집중형 관리가 용이하다.
2. **다차원 데이터 모델**: 지표명뿐만 아니라 `{node="A", env="prod"}`와 같은 **레이블(Label)**을 사용하여 데이터를 입체적으로 분류한다.
3. **TSDB 저장**: 시계열 데이터에 특화된 구조로, 시간이 지날수록 오래된 데이터는 요약하고 최신 데이터는 상세히 기록하는 등의 압축 기술을 사용한다.
4. **시각화 및 경보**: 그라파나는 **PromQL** 쿼리를 통해 저장소의 데이터를 가져와 화려한 그래프로 변환하고, 조건이 맞으면 Alertmanager가 실시간 알람을 보낸다.

#### 심층 동작 원리: 메트릭의 4가지 유형 (Prometheus 기준)
- **Counter**: 누적 수치 (예: 총 요청 수). 오직 증가하거나 초기화만 됨.
- **Gauge**: 가변 수치 (예: 현재 메모리 사용량). 오르내릴 수 있음.
- **Histogram**: 관찰 범위를 구간(Bucket)별로 나누어 빈도 측정 (예: 응답 시간 분포).
- **Summary**: 히스토그램과 유사하나 클라이언트 측에서 분위수(Quantile)를 계산함.

- **📢 섹션 요약 비유**: 숫자의 형태에 따라 누적 기록(Counter)인지, 현재 상태(Gauge)인지 구분하여 기록하는 꼼꼼한 가계부 작성법과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. Push 방식 vs Pull 방식 비교

| 비교 항목 | Push 방식 (Pushgateway, InfluxDB) | Pull 방식 (Prometheus) |
|:---:|:---|:---|
| **주도권** | 각 서버가 직접 전송 | **중앙 서버가 긁어옴** |
| **유연성** | 짧게 끝나는 배치 작업에 유리 | 대규모 상시 서비스 관리에 유리 |
| **네트워크 부하** | 서버가 많아지면 중앙 부하 가중 | 중앙 서버가 수집 주기 제어 가능 |
| **비유** | 전단지를 집집마다 돌리기 | 우체부가 편지를 수거하러 가기 |

#### 2. 로그(Logging)와의 상호 보완성
- **메트릭**: "문제가 발생했다"는 사실을 빠르게 인지 (응답 시간 증가 발견).
- **로그**: "무슨 문제가 발생했는지" 상세 원인 파악 (NullPointerException 확인).
- **시너지**: 메트릭으로 알람을 받고, 해당 시점의 로그를 추적하여 최단 시간에 복구(MTTR 감소)한다.

- **📢 섹션 요약 비유**: 메트릭이 체온계(현상 포착)라면, 로그는 혈액 검사 결과지(정밀 분석)와 같아 이 둘이 합쳐져야 완벽한 진단이 가능합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 자원 낭비 없는 오토스케일링 정책 수립**
    - **문제**: CPU 기반 스케일링을 했더니, 정작 네트워크 지연 시에는 서버가 늘어나지 않아 장애 발생.
    - **의사결정**: 커스텀 메트릭 도입. **Prometheus-Adapter**를 통해 '동시 접속자 수'와 '응답 시간 지연' 수치를 쿠버네티스 HPA에 연동.
    - **결과**: 비즈니스 지표에 기반한 정밀한 스케일링으로 가용성 20% 향상 및 클라우드 비용 15% 절감.

- **도입 체크리스트**:
    1. **Retention Policy**: 원시 데이터를 얼마나 오래 보관할 것인가? (장기 보관 시 Thanos, VictoriaMetrics 검토)
    2. **High Cardinality**: 레이블에 시퀀스 번호나 타임스탬프를 넣어 인덱스 폭발이 일어나지 않게 관리하는가?
    3. **Exporter Health**: 메트릭을 뽑아주는 에이전트 자체가 죽었을 때의 감시 체계가 있는가?

- **안티패턴**: 
    - 모든 서비스의 모든 메서드 호출을 메트릭으로 남기는 행위 (메모리 부족으로 모니터링 시스템 마비).
    - 알람 임계치를 너무 낮게 잡아 매일 수백 개의 알람이 오는 '알람 피로(Alert Fatigue)' 상태 방치.

- **📢 섹션 요약 비유**: 너무 예민한 화재 경보기는 양치기 소년이 되고, 너무 둔한 경보기는 대형 참사를 부릅니다. 적절한 '골든 지표'를 선정하는 것이 운영의 묘미입니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량적 기대효과**:
    - **장애 인지 시간**: 수동 인지 대비 평균 90% 이상 단축.
    - **TCO 절감**: 자원 사용 패턴 분석을 통한 인프라 최적화로 불필요한 과금 방지.
- **미래 전망**: **OpenTelemetry** 표준의 확산으로 벤더 종속성이 사라지고, 수집된 메트릭을 학습한 AI가 스스로 시스템 성능을 튜닝하는 **Self-Tuning 인프라**가 보편화될 것이다.
- **참고 표준**: CNCF Prometheus 표준, OpenMetrics 규격.

- **📢 섹션 요약 비유**: 감(Feeling)으로 운전하던 시대를 지나, 모든 수치가 실시간으로 분석되는 비행기 조종석(Grafana)에 앉아 안정적인 비행(운영)을 하는 것과 같습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 메트릭은 우리 몸의 **"온도계와 맥박수"**처럼, 컴퓨터가 얼마나 힘든지 숫자로 보여주는 거예요.
2. **프로메테우스**는 이 숫자들을 매일매일 일기장에 기록하고, **그라파나**는 그 숫자를 예쁜 그래프로 그려주는 마법의 칠판이랍니다.
3. 숫자가 너무 높아져서 컴퓨터가 아프려고 하면, "삐릭삐릭!" 하고 알려줘서 미리 고칠 수 있게 도와줘요!
