# 481. Insecure Design (안전하지 않은 설계)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Insecure Design (안전하지 않은 설계)은 구현(Coding)의 완성도와 무관하게, 비즈니스 로직이나 시스템 아키텍처 자체가 보안 결함을 내포하고 있어 **구조적으로 공격에 취약한 상태**를 의미한다.
> 2. **가치**: 2021년 OWASP Top 10에 신설된 항목으로, 완벽한 코딩(Perfect Coding)이 부실한 설계(Bad Design)를 구제할 수 없음을 강조하며 '시프트 레프트 (Shift-Left)' 보안의 정점을 보여준다.
> 3. **융합**: 위협 모델링(Threat Modeling), 보안 디자인 패턴, 비즈니스 로직 분석과 결합되며, 단순한 도구 스캔(SAST/DAST)만으로는 발견할 수 없는 '논리적 허점'을 설계 단계에서 제거하는 것이 핵심이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: Insecure Design은 "잘못 만들어진 것(Implementation)"이 아니라 "잘못 설계된 것(Design)"에 집중한다. 예를 들어, 비밀번호 복구 질문으로 '초등학교 이름'을 사용하는 것은 코딩상의 오류는 없지만, 설계상으로는 매우 취약하다. 이처럼 시스템의 기본 구조나 흐름 자체가 공격자에게 유리하게 설계된 모든 경우가 여기에 해당한다.

- **💡 비유**: Insecure Design은 **"금고 문은 세상에서 가장 튼튼한 강철로 만들었지만(완벽한 구현), 금고 벽면이 얇은 석고보드로 되어 있어 옆을 뚫고 들어올 수 있게 설계된 방(부실한 설계)"**과 같습니다.

- **등장 배경**:
  1. **구현 보안의 성숙**: SAST, DAST 등 자동화 도구의 발달로 단순한 코딩 실수(인젝션 등)는 줄어들었으나, 아키텍처적 결함으로 인한 대규모 사고는 여전히 빈번했다.
  2. **비즈니스 로직 공격의 증가**: 해커들이 코드의 버그를 찾기보다 시스템의 프로세스(예: 이벤트 쿠폰 중복 발급 로직)를 악용하기 시작했다.
  3. **Secure SDLC의 고도화**: 보안을 개발 마지막 단계가 아닌, 기획과 설계 단계부터 내재화해야 한다는 인식이 확산되었다.

- **📢 섹션 요약 비유**: 요리사가 재료를 아무리 정성껏 손질해도(코딩), 레시피 자체가 독버섯을 넣으라고 되어 있다면(설계) 그 음식은 위험할 수밖에 없다는 원리와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Insecure Design의 주요 원인 및 사례

| 원인 유형 | 설명 | 실제 사례 | 비유 |
|:---|:---|:---|:---|
| **논리적 우회** | 비즈니스 프로세스의 허점 노출 | 결제 단계에서 브라우저 '뒤로 가기'를 통해 결제 없이 완료 | 줄 서지 않고 새치기하기 |
| **과도한 정보 노출** | 에러 메시지나 응답에 너무 많은 정보 포함 | 로그인 실패 시 "아이디는 맞지만 비번이 틀림"이라고 알려줌 | 도둑에게 비밀번호 힌트 주기 |
| **신뢰 경계 미흡** | 외부 입력을 무비판적으로 신뢰함 | 관리자 페이지 주소(URL)를 알기만 하면 누구나 접속 가능 | 아는 척하면 들여보내 주기 |
| **부적절한 리소스 제한** | 자원 할당량(Quota) 제한 없음 | 무제한 파일 업로드로 서버 용량 고갈 유발 | 무한 리필 식당에서 음식 다 싸가기 |

### Secure Design vs Insecure Design 아키텍처 비교 (비밀번호 복구 로직)

설계의 차이가 보안성을 어떻게 결정하는지 시각화하면 다음과 같다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      비밀번호 복구 로직: 설계 결함 vs 안전한 설계                 │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Insecure Design - 취약한 설계]        [Secure Design - 안전한 설계]         │
 │                                                                             │
 │  1. 아이디 입력                       1. 아이디 입력                         │
 │  2. 보안 질문 응답 (출신학교?) ──▶     2. 등록된 이메일/번호로 일회용 토큰 발송     │
 │     (SNS로 추측 가능)                 (외부 채널 인증 강화)                  │
 │  3. 즉시 새 비밀번호 설정               3. 토큰 검증 및 일정 시간 내 변경 강제      │
 │  4. 성공 (공격자 탈취 용이)             4. 성공 (다중 인증으로 보호)             │
 │                                                                             │
 │  ┌───────────────────────┐            ┌───────────────────────┐            │
 │  │ ⚠ 설계적 결함 요소    │            │ ✅ 보안 설계 요소      │            │
 │  │ - 추측 가능한 정보 활용│            │ - Out-of-Band 인증    │            │
 │  │ - 단일 단계 검증      │            │ - 시간 제한 (TTL)      │            │
 │  │ - 과도한 편의성 우선  │            │ - 보안과 편의성의 균형 │            │
 │  └───────────────────────┘            └───────────────────────┘            │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 왼쪽의 취약한 설계는 코딩상으로는 아무런 버그가 없을 수 있다(입력값 검증도 잘 하고, DB 연동도 정석적일 수 있다). 하지만 **보안 질문**이라는 요소 자체가 사회 공학적 공격이나 SNS 정보 수집으로 쉽게 뚫릴 수 있는 아키텍처적 결함이다. 반면 오른쪽의 안전한 설계는 이메일이나 SMS 같은 별도의 경로(Out-of-Band)를 통해 일회용 토큰을 전송하고, 시간 제한을 두는 등 다층 방어 체계를 구축한다. 이는 '어떻게 코딩하느냐'의 문제가 아니라 '어떤 흐름으로 시스템을 만드느냐'의 설계 문제다.

---

### 심층 동작 원리: 보안 디자인 패턴 활용

안전한 설계를 위해 다음과 같은 정립된 패턴들을 적용해야 한다.
1. **최소 권한 원칙 (Least Privilege)**: 모든 주체는 업무 수행에 꼭 필요한 최소한의 권한만 가져야 한다.
2. **심층 방어 (Defense in Depth)**: 한 겹의 보안이 뚫려도 다음 겹이 막을 수 있도록 다중 계층 보안을 구축한다.
3. **실패 안전 (Fail-Safe)**: 보안 장치가 고장 나거나 시스템에 오류가 발생했을 때, 기본값은 '거부(Deny)'여야 한다. (예: 방화벽 오류 시 모든 트래픽 차단)

- **📢 섹션 요약 비유**: 은행 창구에 방탄 유리를 설치하고(심층 방어), 직원에게는 자기 금고 열쇠만 주고(최소 권한), 정전이 되면 모든 문이 자동으로 잠기게(Fail-Safe) 만드는 설계 원칙과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Insecure Design vs Software Vulnerabilities 비교

| 비교 항목 | Insecure Design (설계 결함) | Implementation Flaws (구현 약점) |
|:---|:---|:---|
| **발생 시점** | 기획 및 설계 단계 | 코딩 및 구현 단계 |
| **식별 도구** | 위협 모델링, 디자인 리뷰 | SAST, DAST, 퍼징 (Fuzzing) |
| **수정 방법** | 아키텍처 재설계, 비즈니스 로직 수정 | 코드 수정, 패치 적용 |
| **특징** | 코드는 깨끗할 수 있으나 논리가 틀림 | 의도는 맞으나 코딩 실수가 있음 |
| **대표 사례** | 취약한 비밀번호 찾기 질문 | SQL Injection, Buffer Overflow |

---

### 융합 시너지: 위협 모델링과의 결합

Insecure Design을 잡기 위한 가장 강력한 도구는 **위협 모델링 (Threat Modeling)**이다.
- STRIDE 모델 등을 통해 설계도상의 데이터 흐름을 분석하고, 각 접점에서 발생 가능한 부정적인 시나리오를 사전에 차단한다.
- 이는 자동화 도구가 할 수 없는 '인간 전문가의 통찰'이 필요한 영역이다.

- **📢 섹션 요약 비유**: 맞춤법 검사기(SAST)로는 소설의 오타를 고칠 수 있지만, 소설의 줄거리(설계)가 앞뒤가 맞지 않는 것은 편집자(위협 모델링)가 직접 읽어보고 고쳐야 한다는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '무료 배송' 쿠폰 무제한 발급 사고 예방

- **문제 상황**: 이벤트 페이지에서 쿠폰 발급 버튼을 연타하면 1인 1매 제한 로직을 우회하여 수십 장의 쿠폰이 발급되는 현상 발생 가능.
- **의사결정 (설계 개선)**:
  1. **상태 관리 강화**: 클라이언트 단의 체크가 아닌, 서버 측 DB 트랜잭션과 Redis 분산 락(Distributed Lock)을 사용하여 중복 요청을 원천 차단한다.
  2. **비즈니스 한도 설정 (Throttling)**: 전체 시스템 차원에서 분당 발급 가능한 쿠폰 수에 상한을 두어 예상치 못한 대량 발급 사고를 예방한다.
  3. **이상 탐지 설계**: 단시간에 특정 IP에서 대량 요청이 올 경우 계정을 일시 정지하는 로직을 아키텍처에 포함한다.

### 도입 체크리스트
- **Threat Modeling**: 프로젝트 시작 시 공식적인 위협 모델링 세션을 진행했는가?
- **Security Library**: 검증된 보안 라이브러리와 인증 패턴을 사용하고 있는가?
- **Abuse Case Analysis**: "사용자가 이렇게 쓰겠지"가 아니라 "나쁜 사용자가 이렇게 괴롭히겠지"라는 Abuse Case를 분석했는가?

### 안티패턴
- **보안을 나중에 (Security as an Afterthought)**: 기능을 다 만든 뒤에 보안팀에 검토를 요청하는 행위. 이 단계에서 설계 결함이 발견되면 전체를 갈아엎어야 한다.
- **사용자 편의성 과잉**: 보안 질문을 너무 쉽게 만들거나 로그인을 영구 유지하게 하는 등, 편의성을 위해 보안의 기본 원칙을 포기하는 행위.

- **📢 섹션 요약 비유**: 배를 다 만든 뒤에 "이 배가 물에 뜰까요?"라고 묻는 것이 아니라, 설계도 단계에서 부력 계산을 마쳐야 한다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Cost)** | 수정 비용 극대화 절감 | 운영 단계 대비 설계 단계에서의 수정 비용 1/100 수준 유지 |
| **정량적 (Quality)** | 근본적 취약점 제거 | 도구로 찾을 수 없는 논리적 결함 90% 이상 사전 차단 |
| **정성적 (Resilience)** | 사이버 레질리언스 확보 | 공격을 받아도 시스템의 핵심 로직은 무너지지 않는 강인함 확보 |
| **정성적 (Trust)** | 서비스 신뢰도 향상 | 비즈니스 로직 우회 사고가 없는 탄탄한 서비스 이미지 구축 |

### 미래 전망
- **AI-Driven Secure Design**: AI가 시스템 아키텍처 다이어그램을 분석하여 "이 구조는 권한 우회 가능성이 있습니다"라고 경고해주는 지능형 설계 지원 도구가 등장할 것이다.
- **Privacy by Design (PbD)**: 개인정보 보호를 설계 단계부터 기본값으로 설정하는 법적 요구가 강화됨에 따라, Insecure Design 방어는 단순 보안을 넘어 법적 생존 전략이 될 것이다.

### 참고 표준
- **IEEE 26702**: 시스템 공학 프로세스 표준
- **NIST SP 800-160**: Systems Security Engineering (SSE) 가이드라인
- **OWASP ASVS**: Application Security Verification Standard (설계 검증 기준 포함)

- **📢 섹션 요약 비유**: 튼튼한 성벽(방화벽) 뒤에 숨는 시대는 지나갔으며, 이제는 성 내부의 길 하나하나가 미로처럼 설계되어 적이 들어와도 길을 잃게 만드는 '지능형 요새 설계'가 핵심이 될 것입니다.
