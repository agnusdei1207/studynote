# 123-130. 클라우드 네이티브와 최신 방법론 (Serverless, DDD, INVEST)

> **핵심 인사이트**: 이제 인프라는 보이지 않는 공기처럼 당연한 것이 되었다. 서버리스를 통해 '관리할 서버'를 없애고, 클라우드에 최적화된 12-Factor 설계 지침을 따르며, DDD를 통해 비즈니스의 본질을 코드에 녹여낸다. 개발의 첫 단추인 사용자 스토리는 INVEST 원칙을 통해 측정 가능하고 가치 있는 단위로 벼려진다.

---

## Ⅰ. 서버리스 (Serverless)와 FaaS
개발자가 서버를 관리하거나 프로비저닝하지 않고, 코드(함수)만 작성하여 실행하는 모델입니다.

* **FaaS (Function as a Service)**: 특정 이벤트 발생 시에만 함수가 실행되고, 실행된 시간만큼만 비용을 지불하는 방식. (예: AWS Lambda)
* **장점**: 인프라 운영 부담 제로, 완벽한 자동 확장(Auto-scaling).

---

## Ⅱ. 클라우드 네이티브 앱의 기준: 12-Factor
현대적인 클라우드 기반 앱(SaaS)을 만들기 위한 12가지 설계 원칙입니다. (Heroku 제안)

* **주요 항목**: 
  - **Codebase**: 하나의 코드 베이스로 여러 환경 배포.
  - **Dependencies**: 명시적 의존성 관리.
  - **Config**: 설정을 코드와 분리하여 환경 변수에 저장.
  - **Stateless**: 프로세스는 아무것도 공유하지 않는 상태 비저장 구조.
  - **Concurrency**: 프로세스 모델을 통한 수평 확장.

---

## Ⅲ. 도메인 주도 설계 (DDD)의 애자일 연계
비즈니스 전문가와 개발자가 똑같은 언어(**Ubiquitous Language**)를 사용하여, 소프트웨어의 복잡성을 비즈니스 도메인 중심으로 해결하는 방식입니다.

* **Bounded Context**: 큰 도메인을 경계가 뚜렷한 작은 맥락으로 나누어 마이크로서비스(MSA)의 단위로 삼습니다.

---

## Ⅳ. 요구사항 품질 기준: INVEST
좋은 사용자 스토리(User Story)가 갖춰야 할 6가지 특성입니다.

* **I (Independent)**: 다른 스토리와 독립적이어야 함.
* **N (Negotiable)**: 협상 가능해야 함. (구체적 구현은 미정)
* **V (Valuable)**: 사용자에게 명확한 가치를 주어야 함.
* **E (Estimable)**: 노력을 추정할 수 있어야 함.
* **S (Small)**: 스프린트 내에 끝낼 수 있을 만큼 작아야 함.
* **T (Testable)**: 테스트 가능해야 함. (인수 기준 명확)

---

## Ⅴ. 개념 맵 및 요약

```ascii
[클라우드 네이티브 개발의 삼각형]

      [ 아키텍처 (12-Factor) ]
             /      \
            /        \
  [ 구현 (Serverless) ] ── [ 분석 (DDD/INVEST) ]
```

📢 **섹션 요약 비유**: **서버리스**는 집에서 직접 밥(서버 관리)을 하는 대신, 필요할 때만 배달 음식을 시켜 먹고 치우는 것과 같습니다. **12-Factor**는 어떤 나라에 가도 즉시 적응할 수 있는 '글로벌 표준 여행 매뉴얼'이며, **DDD**는 외국인(비즈니스 팀)과 대화할 때 번역기 없이 서로 통하는 전문 용어로 소통하는 것입니다. **INVEST**는 장을 볼 때 "맛있는 거 사 와"라고 모호하게 말하는 대신 "사과 3개 사 와"라고 구체적이고 실행 가능하게 적어주는 장바구니 목록과 같습니다.
