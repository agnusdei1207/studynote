# 61-72. 스크럼(Scrum) 프레임워크와 역할

> ### 핵심 인사이트 (3줄 요약)
> 1. **본질**: **스크럼(Scrum)**은 복잡한 문제 해결을 위해 투명성, 검사, 적응의 3대 기둥을 기반으로 한 경험적 프로세스 관리(EMP) 프레임워크입니다.
> 2. **가치**: 계획 주도(Waterfall) 방식 대비 **TTM(Time-to-Market)**을 30~50% 단축하고, 불확실성에 유연하게 대응하여 **ROI(Return on Investment)**를 극대화합니다.
> 3. **융합**: DevOps의 CI/CD 파이프라인과 연동하여 고객 피드백 루프를 자동화하며, 최근 AI 기반 의사결정 도구와 결합하여 백로그 관리의 효율을 높이는 추세입니다.

---

### Ⅰ. 개요 (Context & Background)

**개념 및 철학**
**스크럼(Scrum)**은 1986년 하버드 비즈니스 리뷰에 소개된 'Takeuchi and Nonaka'의 논문에서 유래하여, 1990년대 소프트웨어 개발(SW) 분야에 도입된 **애자일(Agile)** 방법론의 실천 프레임워크입니다. 단순한 프로세스가 아닌 럭비 경기의 스크럼처럼 팀원들이 공통의 목표를 향해 밀집하여 협업하는 메타포를 기반으로 합니다. 이는 **린(Lean)** 사고의 낭비 제거 원칙을 소프트웨어 개발에 적용한 것으로, 불확실한 환경에서 **반복(Iterative)**과 **점진(Incremental)**적인 개발을 통해 가치를 전달합니다.

**등장 배경**
1.  **기존 한계**: 폭포수 모델(Waterfall Model)은 초기 요구사항이 완벽해야 하며, 변경 비용(Cost of Change)이 프로젝트 후반부로 갈수록 기하급수적으로 증가하는 문제가 있었습니다.
2.  **혁신적 패러다임**: 2001년 애자일 선언(Agile Manifesto) 이후, '실패하는 것보다 빨리 배우라(Fail Fast, Learn Faster)'는 철학이 등장했습니다. 스크럼은 이를 실현하기 위해 **검증(Inspection)**과 **적응(Adaptation)** 주기를 최소화했습니다.
3.  **현재의 비즈니스 요구**: 스타트업과 대기업을 막론하고 **MVP(Minimum Viable Product, 최소 기능 제품)**를 통해 시장성을 빠르게 검증하고, **Pivot(피벗, 방향 전환)**이 필요한 현대 비즈니스 환경에 필수적인 프로세스로 자리 잡았습니다.

**💡 비유**
건물을 지을 때 설계도를 완벽하게 그린 후 한 번에 짓는 방식(폭포수)과 달리, 자동차를 만들 때 엔진 → 차체 → 도색 순으로 조립하며 부품마다 시동을 걸어보고 수정하는 방식(애자일/스크럼)과 같습니다.

📢 **섹션 요약 비유**: 스크럼은 '알 수 없는 미지의 망망대해'에서 항해하는 방식입니다. 나침반 없이 한 방향으로만 쭉 가는 것이 아니라, 매일 바람과 파도를 확인(검사)하며 돛의 각도를 조정(적응)하여 목적지에 도달하는 항해술입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

스크럼 프레임워크는 **역할(Roles)**, **이벤트(Events)**, **산출물(Artifacts)**, **규칙(Rules)**로 구성됩니다. 그중 핵심 구성 요소를 심층 분석합니다.

#### 1. 핵심 구성 요소 상세 분석

| 구성 요소 (Component) | 상세 명칭 (Full Name) | 핵심 역할 및 책임 (Role & Responsibility) | 주요 산출물/포인트 (Key Output) | 비유 (Analogy) |
|:---:|:---|:---|:---|:---|
| **역할** | **PO (Product Owner)** | 제품의 **Vision**과 **Backlog** 관리 총괄. <br>• **Stakeholder(이해관계자)** 요구사항 수렴 <br>• **ROI(투자 대비 수익)** 최대화를 위한 우선순위 결정 | **Product Backlog** <br>(User Stories, Requirements) | 영화의 **감독** (어떤 영화를 찍을지 결정) |
| **역할** | **SM (Scrum Master)** | **Servant Leader(섬기는 리더)**. <br>• **Process** 관리 및 개선 <br>• **Impediment(장애물)** 제거 및 외부 방해 차단 <br>• 팀이 **Scrum Theory**를 이해하도록 코칭 | 팀의 **성숙도** <br>(Velocity 향상) | 경기의 **심판** (규칙 지키고, 경기 흐름 유지) |
| **역할** | **Developers** | **Self-organizing(자기조직화)** 팀. <br>• 매 스프린트마다 **Done(완료)** 기준의 Increment 생성 <br>• 기술적 의사결정 권한 보유 | **Potentially Releasable Increment** <br>(Working Software) | 경기장의 **선수들** (실제로 골을 넣는 주체) |
| **이벤트** | **Sprint (스프린트)** | 1~4주의 고정된 짧은 기간. <br>• 변경 불가(No Changes) 보장을 통한 집중 <br>• 기한 내 '완성된' 결과물 강제 | **Increment** (제품의 일부) | **하루**라는 시간 단위 (해가 지면 끝) |
| **산출물** | **Increment (증분)** | 모든 **Sprint Backlog** 항목 완료 시 결과물. <br>• **Definition of Done(DoD)** 충족 필수 <br>• 이전 증분과 합쳐져 **Integrated Product** 형성 | Integrated Product (Useable SW) | 자동차의 **완성된 부품 조립체** |

#### 2. 스크럼 프로세스 아키텍처 (ASCII Diagram)

스크럼은 **Product Backlog**에서 가치 높은 항목을 선택하여 **Sprint Backlog**로 옮기고, 이를 **Sprint** 기간 동안 개발하여 **Increment**를 생성하는 **피드백 루프(Feedback Loop)** 구조입니다.

```ascii
                     [ Product Backlog ]
                              +
                  (PO에 의해 우선순위 부여)
                              |
              +---------------+---------------+
              |  Sprint Planning (Event #1)   |
              +---------------+---------------+
                              |
                              v
                     [ Sprint Backlog ]
                              +
            (Team에 의해 태스크 단위로 세분화)
                              |
      +-----------------------+-----------------------+
      |   <----------- SPRINT (Timebox: 1~4 Wks) >   |
      |   +---------------------------------------+   |
      |   |  Daily Scrum (15min, Event #2)       |   |
      |   |  [ 계획 -> 진행 -> 장애물 파악 ]     |   |
      |   +---------------------------------------+   |
      |                       |                       |
      |                       v                       |
      |   +---------------------------------------+   |
      |   |      Development (Build & Test)      |   |
      |   +---------------------------------------+   |
      |                       |                       |
      +-----------------------+-----------------------+
                              |
              +---------------+---------------+
              |   Sprint Review (Event #3)    |
              |  (Stakeholder 대상 시연/피드백) |
              +---------------+---------------+
                              |
              +---------------+---------------+
              |  Sprint Retrospective (Event#4)|
              |   (프로세스 개선/도입 방법 논의) |
              +---------------+---------------+
                              |
                              v
                        [ Increment ]
                              +
            (Potentially Releasable Product Slice)
                              |
                              +----> [ 다음 Sprint로 Feedback ]
```

#### 3. 핵심 동작 메커니즘 및 수식

**1) Definition of Done (DoD)와 Velocity**
스크럼에서 '완료(Done)'의 정의는 매우 중요합니다. 단순히 코딩이 끝난 것이 아니라, 테스트, 리뷰, 배포 준비까지 완료된 상태를 의미합니다.

*   **Velocity (속도)**: 팀이 한 스프린트 동안 완료한 스토리 포인트(Story Point)의 합계입니다.
    $$ \text{Average Velocity} = \frac{\sum \text{Story Points Completed (Last N Sprints)}}{N} $$
    이 값은 향후 스프린트 계획 수립 시 예측 가능한 작업량을 산정하는 데 사용됩니다.

**2) 번다운 차트 (Burndown Chart) 수식**
남은 작업량(Remaining Work)이 시간 경과에 따라 어떻게 감소하는지 시각화한 차트입니다. 이상적인 그래프는 0점을 향해 일직선으로 내려가는 형태입니다.

$$ y = -(\frac{\text{Total Work}}{\text{Total Days}}) \times x + \text{Initial Work} $$
*(여기서 $x$는 경과 일수, $y$는 남은 작업량)*

📢 **섹션 요약 비유**: 스크럼의 구조는 마치 **고속도로 요금소의 하이패스 시스템**과 유사합니다. **PO**는 어느 차선이 효율적인지 정하고, **SM**은 톨게이트 시스템(게이트) 오작동을 수리하며, **팀**은 그 차선을 통해 통과하는 차량들입니다. **스프린트**는 하루 단위로 정산되는 집계 기간이며, **번다운 차트**는 터미널에서 확인하는 당일 교통량 처리 현황판과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 전통적 모델 vs 스크럼 심층 비교

| 비교 항목 (Criteria) | Waterfall Model (전통적 모델) | Scrum Framework (애자일) |
|:---|:---|:---|
| **개발 주기** | **Linear (직렬적)**: 타이트하게 연결된 단계 | **Iterative (반복적)**: 짧은 주기의 순환 |
| **요구사항 변경** | **Cost 매우 높음**: 후반부 변경은 재설계를 의미 | **Cost 낮음**: 매 Sprint마다 우선순위 조정 가능 |
| **투명성** | 낮음 (관리자 보고 위주) | 높음 (**Daily Scrum**, **Artifact** 시각화) |
| **팀 문화** | **Command & Control** (상명하복) | **Self-Organization** (자율 책임) |
| **품질 관리** | 마지막 테스트 단계에서 확인 (결함 발견 지연) | 매 **Sprint**마다 **DoD** 기반 품질 확인 |
| **납품 주기** | 프로젝트 종료 시 | 최소 1~4주마다 **Increment** 납품 |

#### 2. 타 영역 융합 분석

**1) 소프트웨어 공학(SE) & DevOps와의 시너지**
스크럼의 **Increment**는 단순한 소스 코드가 아닌, **배포 가능한 형태(Deployable)**여야 합니다. 이는 **CI/CD (Continuous Integration/Continuous Deployment)** 파이프라인과 필연적으로 결합합니다.
*   **연계 지점**: Sprint Review 통과 코드 → 자동화된 CI 파이프라인 투입 → Integration Test → Staging Deploy.
*   **효과**: 스크럼에서 계획한 기능을 즉시 운영 환경에 릴리스하여 **MTTR (Mean Time To Restore)**을 단축시킵니다.

**2) 데이터베이스(DB)와의 갈등 해결**
스크럼은 스키마 변경이 잦은 반면, RDBMS는 변경에 취약합니다.
*   **해결책**: **NoSQL**(Schemaless) 도입 또는 **Refactoring**(데이터 마이그레이션)을 Sprint Backlog에 포함하여 점진적 DB 변경을 수행합니다.

📢 **섹션 요약 비유**: 스크럼과 전통적 방식의 차이는 **'식당 주문 방식'**과 같습니다. 폭포수는 코스 요령(전체 메뉴를 한 번에 주문, 요리, 서빙)이라 메뉴를 바꿀 수 없지만, 스크럼은 호텔 뷔페(여러 번 와서 원하는 음식을 그때그때 가져 먹음)처럼 고객이 먹는 것을 확인하고 다음 메뉴를 고를 수 있습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 스크럼을 도입할 때는 단순히 형식을 따라 하는 '좀비 스크럼(Zombie Scrum)'을 경계해야 합니다.

#### 1. 실무 시나리오 및 의사결정 매트릭스

**상황: 이슈 관리 도구(Jira 등)를 도입하여 스크럼을 시작하려 함.**

1.  **문제 상황**: 개발팀이 스프린트 중간에 PO가 긴급 이슈를 추가하여 **일정이 자주 지연**되고, 팀원들이 지쳐감(Burnout).
2.  **의사결정 과정**:
    *   **진단**: 스크럼의 핵심 원칙인 'Sprint 중간 변경 금지'가 위배됨.
    *   **해결**: **Buffer(버퍼)** 용량을 20% 확보하거나, 긴급 이슈는 별도의 **Kanban Board(칸반 보드)**로 분리하여 관리.
    *   **결과**: Sprint Backlog의 안정성을 확보하고, 팀의 몰입도를 회복함.

#### 2. 도입 체크리스트 (Checklist)

| 구분 | 점검 항목 | 세부 내용 |
|:---|:---|:---|
| **기술적** | 자동화 인프라 | CI/CD 파이프라인 구축 여부 (코드 통합 자동화) |
| | Artifact 관리 | **Version Control(Git)**과 Backlog의 연동 상태 |
| **운영적** | 회의 효율화 | **Daily Scrum**이 15분을 넘지 않고 진행되는가? |
| | 역할 분리 | PO와 SM의 역할이 한 사람에게 중복되어 권한이 집중되지 않는가? |
| **보안적** | 접근 통제 | Product Backlog의 노출 범위(특히 민감한 요구사항) 관리 |

#### 3. 안티패턴 (Anti-Patterns)

*   **Proxy PO(대리인 PO)**: 실제 결정권을 가진 경영진이 뒤에 있고, 현장의 PO는 전달만 하는 경우. **의사
