# 106-113. 플랫폼 엔지니어링과 가시성 (IDP, Observability)

> **핵심 인사이트**: 복잡한 클라우드 인프라가 개발자의 짐이 되어서는 안 된다. 플랫폼 엔지니어링은 개발자가 인프라를 몰라도 스스로 서비스를 띄울 수 있는 셀프 서비스(IDP)를 제공하며, 가시성 기술은 마이크로서비스라는 거대한 미로 속에서 장애의 원인을 찰나의 순간에 찾아내는 나침반 역할을 한다.

---

## Ⅰ. 플랫폼 엔지니어링 (Platform Engineering)
클라우드 네이티브 시대에 개발자 경험(DevEx)을 극대화하기 위해 탄생한 분야입니다.

* **IDP (Internal Developer Platform)**: 개발자가 인프라 티켓을 끊고 기다리는 대신, 버튼 클릭 한 번으로 서버, DB, 네트워크를 생성할 수 있게 해주는 **내부 셀프 서비스 포털**입니다.
* **가치**: 개발자는 비즈니스 로직에만 집중하고, 플랫폼 팀은 공통 기반 기술을 고도화하여 조직 전체의 생산성을 높입니다.

---

## Ⅱ. 가시성 (Observability)
시스템 외부에서 내부의 상태를 얼마나 잘 파악할 수 있는가를 의미합니다. 단순한 모니터링보다 훨씬 심화된 개념입니다.

### 가시성의 3대 기둥 (M.L.T)
1. **Metrics (지표)**: CPU 사용률, 응답 시간 등 수치화된 데이터. (무슨 일이 일어나는가?)
2. **Logs (로그)**: 특정 시점에 발생한 이벤트의 상세 기록. (왜 일어났는가?)
3. **Traces (추적)**: 마이크로서비스(MSA) 환경에서 하나의 요청이 거쳐 간 모든 서비스의 경로 기록. (어디서 지연되는가?)

---

## Ⅲ. 카오스 엔지니어링 (Chaos Engineering)
* **정의**: 시스템의 회복력(Resilience)을 확인하기 위해, **운영 환경에 고의로 장애를 주입**하여 견고함을 테스트하는 기법입니다. (예: 넷플릭스의 Chaos Monkey)
* **목적**: 예측하지 못한 장애 상황에서도 시스템이 자동으로 복구되거나 안정적으로 유지되는지 확인합니다.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[플랫폼 엔지니어링의 목표]

  [ Developer ] ───( IDP 이용 )───> [ Self-Service Infra ]
                                            │
  ┌─────────────────────────────────────────┴────────────────────────────────┐
  ▼ (가시성 확보)                                                             ▼ (안정성 확보)
 [ Metrics / Logs / Traces ] <── ( Observability ) ── [ Chaos Engineering ]
```

📢 **섹션 요약 비유**: **플랫폼 엔지니어링**은 밀키트(IDP)를 제공하는 주방 보조와 같습니다. 요리사(개발자)는 재료 손질 없이 바로 요리만 하면 되죠. **가시성**은 건물의 전력, 수도 상황을 한눈에 보여주는 '중앙 관제실'이며, **추적(Tracing)**은 수천 개의 미로에서 택배 상자가 어디서 멈췄는지 찾아내는 'GPS 추적기'입니다. **카오스 엔지니어링**은 소방관들이 실제 건물에 불을 질러보고 대피 훈련을 하는 '실전 화재 훈련'과 같습니다.
