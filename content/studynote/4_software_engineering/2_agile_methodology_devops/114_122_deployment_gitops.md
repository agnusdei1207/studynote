# 114-122. 현대적 배포 전략과 GitOps (Canary, Blue/Green)

> **핵심 인사이트**: 배포는 '모 아니면 도'가 아니다. 위험을 최소화하기 위해 사용자의 일부에게만 새 버전을 노출하거나(Canary), 똑같은 환경을 두 개 만들어 번갈아 사용하는(Blue/Green) 정교한 전략이 필요하다. 특히 GitOps는 인프라의 모든 상태를 코드로 관리하여, 누구나 신뢰할 수 있는 단 하나의 진실(Git)을 바탕으로 시스템을 운영하게 해 준다.

---

## Ⅰ. 무중단 배포 전략

1. **Blue/Green 배포**:
   - 똑같은 환경을 두 개(Blue: 현재, Green: 신규) 준비합니다. 
   - 신규 버전 테스트 완료 후, 로드밸런서 설정만 바꿔서 트래픽을 한꺼번에 Green으로 넘깁니다. (빠른 롤백 가능)
2. **카나리 배포 (Canary)**:
   - 광부들이 탄광에 들어갈 때 카나리아를 먼저 보냈듯, 전체 사용자의 **1~5%에게만 신규 버전을 노출**하여 이상 여부를 살핀 뒤 점진적으로 확대합니다.
3. **롤링 배포 (Rolling)**:
   - 서버를 한 대씩 순서대로 업데이트하는 방식입니다. 가용성이 유지되지만, 배포 중 구버전과 신버전이 공존하는 시기가 발생합니다.

---

## Ⅱ. GitOps
"Git을 운영의 유일한 진실의 원천(Source of Truth)으로 삼는 운영 방식"입니다.

* **원리**: 인프라의 현재 상태를 기술한 선언적 코드(YAML 등)를 Git에 커밋하면, 에이전트(ArgoCD 등)가 Git과 실제 인프라를 비교하여 자동으로 동기화합니다.
* **장점**: 모든 변경 이력이 Git 로그에 남으며, 명령어가 아닌 '커밋'만으로 인프라를 통제할 수 있습니다.

---

## Ⅲ. 관련 배포 기술
* **피처 플래그 (Feature Flag)**: 코드를 배포한 뒤에도, 실시간으로 특정 기능을 끄거나 켤 수 있게 해주는 스위치입니다. (런타임 제어)
* **섀도우 배포 (Shadow)**: 신규 버전을 배포하되 실제 응답은 주지 않고, 실트래픽을 복사해서 성능 테스트만 수행하는 고도의 기법입니다.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[무중단 배포 비교]

1. Blue/Green: [ Blue ] ──( LB 스위칭 )──> [ Green ] (전체 교체)
2. Canary    : [ v1 (95%) ] + [ v2 (5%) ]  (부분 테스트)
3. Rolling   : [ v2 ] [ v1 ] [ v1 ] ──> [ v2 ] [ v2 ] [ v1 ] (순차 교체)
```

📢 **섹션 요약 비유**: **Blue/Green**은 새 아파트를 옆에 똑같이 지어놓고 한 번에 이사 가는 것이고, **카나리**는 신제품 음료를 전국에 팔기 전 특정 편의점에서만 먼저 팔아 반응을 보는 시식 행사입니다. **롤링**은 낡은 전구를 하나씩 순서대로 갈아 끼우는 것이며, **GitOps**는 모든 도시 계획을 설계도(Git)에 적어두면 로봇이 알아서 설계도대로 도시를 고치는 '자동화 건축 시스템'입니다.
