# 97-105. DevOps와 SRE 운영 철학 (CI/CD, Error Budget)

> **핵심 인사이트**: 개발자는 '변화'를 원하고 운영자는 '안정'을 원한다. 이 숙명적인 갈등을 해결하기 위해 DevOps는 문화를 바꾸고 자동화(CI/CD)를 도입했으며, SRE는 '에러 예산'이라는 수학적 지표를 통해 안정성과 속도 사이의 합리적인 합의점을 찾아냈다.

---

## Ⅰ. DevOps (Development + Operations)
개발(Dev)과 운영(Ops)이 하나로 통합되어, 서비스의 생명주기 전체를 공동 책임지는 문화이자 실천 방식입니다.

* **CAMS 모델**: 
  - **Culture (문화)**: 소통과 공유.
  - **Automation (자동화)**: CI/CD, IaC.
  - **Measurement (측정)**: 데이터 기반 의사결정.
  - **Sharing (공유)**: 지식과 도구의 오픈.

---

## Ⅱ. CI / CD (지속적 통합 및 배포)
* **CI (Continuous Integration)**: 개발자가 코드를 커밋하면 자동으로 빌드와 테스트가 수행되어 품질을 즉시 확인하는 과정.
* **CD (Continuous Delivery/Deployment)**: 테스트 완료된 코드를 운영 환경에 자동으로 배포할 준비를 하거나(Delivery), 실제로 배포(Deployment)하는 과정.

---

## Ⅲ. SRE (Site Reliability Engineering)
구글이 정립한 운영 철학으로, **"소프트웨어 엔지니어가 운영을 설계하면 어떨까?"**라는 질문에서 시작되었습니다.

1. **에러 예산 (Error Budget)**: 100% 완벽한 서비스는 없다. 정해진 가용성(SLO, 예: 99.9%)을 제외한 나머지 0.1%의 불확실성을 '새로운 기능을 출시하는 데 쓸 수 있는 예산'으로 간주합니다. 예산이 다 떨어지면 출시를 멈추고 안정을 챙깁니다.
2. **SLI / SLO / SLA**:
   - **SLI**: 성능 측정 지표 (지연 시간 등).
   - **SLO**: 우리가 목표로 하는 수치 (예: 지연 시간 100ms 이내 99%).
   - **SLA**: 고객과 맺은 법적 약속 (어길 시 배상).
3. **Toil (토일)**: 단순 반복적인 운영 작업. SRE는 이를 자동화하여 전체 업무의 50% 미만으로 유지해야 합니다.

---

## Ⅳ. DevSecOps
* 보안을 개발 마지막 단계에 검사하는 게 아니라, 기획부터 운영까지 **전 과정에 보안을 내재화**하는 방식입니다. (보안의 좌측 이동, Shift-Left)

---

## Ⅴ. 개념 맵 및 요약

```ascii
[DevOps 무한 루프]

  [ Plan ] ──> [ Code ] ──> [ Build ] ──> [ Test ] ──┐
     ^                                               │
     │                                               ▼
  [ Monitor ] <── [ Operate ] <── [ Deploy ] <── [ Release ]
```

📢 **섹션 요약 비유**: **DevOps**는 주방(개발)과 홀(운영)의 벽을 허문 '오픈 키친' 식당입니다. 요리사가 서빙도 고민하고, 웨이터가 주방 상황을 실시간으로 아는 구조입니다. **CI/CD**는 식재료가 들어오자마자 자동으로 세척되고 조리되어 손님 상에 오르는 '자동화 컨베이어 벨트'입니다. **SRE**는 식당의 위생 점수(SLO)를 정해두고, 점수가 깎이지 않는 선에서만 신메뉴 개발(에러 예산)을 허용하는 깐깐한 '데이터 기반 점장님'입니다.
