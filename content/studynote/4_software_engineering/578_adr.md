# 578. 아키텍처 결정 기록 (Architecture Decision Records, ADR)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 아키텍처 결정 기록 (ADR)은 소프트웨어 설계 과정에서 내린 중요한 기술적 결정과 그 결정이 내려진 **배경(Context), 대안(Alternatives), 결과(Consequences)**를 기록하는 가벼운 문서화 기법이다.
> 2. **가치**: 설계 당시의 의사결정 맥락을 보존하여 미래의 개발자가 "왜 이렇게 설계했는가?"라는 질문에 스스로 답을 찾게 하며, 지식의 휘발성을 방지하고 아키텍처의 일관성을 유지하는 강력한 **설계 거버넌스 도구**로 작동한다.
> 3. **융합**: '코드로 관리되는 문서(Documentation as Code)'의 철학을 따르며, Git과 같은 버전 관리 시스템을 통해 코드와 함께 관리됨으로써 애자일한 개발 환경에서도 아키텍처 문서를 항상 최신 상태로 유지하게 한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: ADR은 시스템 설계 중 마주치는 수많은 갈림길에서 특정 길을 선택한 이유를 기록한 '일지'다. 마이클 니가드(Michael Nygard)가 제안한 이 기법은, 백과사전식 명세서 대신 **'결정의 순간'**에 집중하여 아키텍처를 문서화한다.

- **💡 비유**: ADR은 **"항해 일지나 등산로의 이정표"**와 같습니다. 길을 잃었을 때 예전 사람들이 왜 이 길로 갔는지 적어둔 쪽지를 보고 "아, 저쪽은 늪지대(기술적 한계)라서 이 산길(현재의 기술)을 택했구나"라고 이해하며 올바른 방향을 잡는 것과 같습니다.

- **등장 배경**:
  1. **암묵적 지식의 전수 한계**: 시간이 지나 설계자가 떠나면, 소스 코드만 보고는 당시의 의사결정 의도를 알 수 없어 시스템을 고치기가 매우 두려워진다. (Fear-based Development 방지)
  2. **전통적 문서의 사장**: 수백 페이지의 아키텍처 정의서는 개발 속도를 따라가지 못해 금방 쓸모없는 종이 뭉치가 된다.
  3. **반복되는 논쟁 방지**: 동일한 기술적 이슈에 대해 매번 처음부터 다시 논쟁하는 비효율을 줄이고, 과거의 결론을 자산으로 활용하고자 했다.

- **📢 섹션 요약 비유**: "왜?"라는 질문에 대답해주는 친절한 과거의 나를 만나는 시간, 그것이 ADR이 주는 가장 큰 선물입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. ADR의 표준 구조 (The Nygard Template)

하나의 ADR은 짧고(1~2페이지), 명확하며, 논리적이어야 한다.

| 구성 요소 | 설명 | 비유 |
|:---|:---|:---|
| **Title** | 결정 사항을 한눈에 알 수 있는 짧은 제목 (예: ADR-001) | 사건 이름 |
| **Status** | 현재 상태 (Proposed, Accepted, Superseded, Deprecated) | 진행 단계 |
| **Context** | 결정이 내려진 당시의 상황, 기술적 문제, 요구사항 | 사건 배경 |
| **Decision** | 우리가 내린 최종 선택과 해결 방법 | 최종 판결 |
| **Consequences** | 결정 이후의 영향 (좋은 점과 나쁜 점 모두 기록) | 판결의 효과 |

---

### 2. ADR 관리 및 수명 주기 아키텍처

결정이 시간이 흐름에 따라 어떻게 진화하는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      ADR (아키텍처 결정 기록) 버전 관리 및 상태 변화              │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Decision Point] ────▶ [1. Proposed] ────▶ [2. Accepted] ────▶ [3. Active]   │
 │      │                    (검토 중)           (공식 적용)         (시스템 반영)  │
 │      │                                                                      │
 │      └──────────▶ [4. Rejected] (반려됨: 근거와 함께 보관하여 중복 논쟁 방지)      │
 │                                                                             │
 │ [Future Point] ──────▶ [5. Superseded] (대체됨) ◀── [New ADR-015 작성]        │
 │                                   │                                         │
 │                                   └───────▶ [6. Deprecated] (폐기됨)         │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 핵심 원칙:                                                            │  │
 │  │ 1. Immutable History: 한번 Accepted된 ADR은 내용을 고치지 않음              │  │
 │  │ 2. New Record for Change: 변경이 필요하면 새 ADR을 써서 이전 것을 대체함      │  │
 │  │ 3. Git-based: 코드와 나란히 `/docs/adr` 폴더에 Markdown으로 보관함           │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ADR은 살아있는 역사책이다. 결정이 내려지면 **Accepted** 상태로 저장되고, 나중에 더 좋은 기술이 나와서 결정을 바꿔야 한다면 예전 문서를 고치는 게 아니라 **새로운 ADR**을 작성한다. 이때 예전 문서는 **Superseded (대체됨)** 상태로 표시하여, 우리 팀이 어떤 역사를 거쳐 현재의 기술 스택에 도달했는지 선명한 **발자취(Trail)**를 남긴다. 이를 통해 신규 입사자는 코드의 변천사를 완벽히 이해할 수 있다.

---

### 심층 동작 원리: 결과(Consequences)의 투명한 기록

ADR의 가장 큰 가치는 '트레이드 오프 (Trade-off)'의 기록이다.
1. **장점 기록**: 이 기술을 써서 성능이 20% 향상됨.
2. **단점 기록**: 대신 설정이 복잡해지고 특정 벤더에 종속됨 (Vendor Lock-in).
3. **가치**: "세상에 완벽한 기술은 없다"는 전제하에, 당시의 최선이 무엇이었는지 정직하게 기록함으로써 미래의 책임 추궁이 아닌 생산적 논의를 이끈다.

- **📢 섹션 요약 비유**: 약을 처방받을 때 효능(장점)뿐만 아니라 부작용(단점)까지 처방전에 적어두어야, 나중에 몸 상태가 변했을 때 정확한 진단이 가능한 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### ADR vs 아키텍처 정의서 (Arch. Description)

| 비교 항목 | 아키텍처 정의서 (전통적) | 아키텍처 결정 기록 (ADR) |
|:---|:---|:---|
| **중심 내용** | 시스템의 현재 '상태' (Snapshot) | 시스템이 도달한 '과정' (Process) |
| **작성 시점** | 프로젝트 초기 또는 종료 시 | **결정이 일어나는 모든 순간** |
| **업데이트** | 매우 어려움 (관리 포기 빈번) | **매우 쉬움 (추가만 하면 됨)** |
| **형식** | 수십 장의 문서 / PPT | 1~2장의 Markdown 파일 |
| **비유** | 완성된 건물의 사진 | 건물을 지은 목수의 작업 일지 |

---

### 기술 융합: GitOps와 ADR

ADR은 코드로 취급되어 배포 파이프라인과 결합된다.
- **Git Flow 연동**: ADR 승인 과정 자체가 Pull Request (PR)를 통해 이루어진다.
- **가치**: 아키텍처 결정이 동료들의 코드 리뷰를 거치게 되며, 머지(Merge)되는 순간 공식적인 팀의 자산으로 확정된다.

- **📢 섹션 요약 비유**: 새로운 가훈(보안 정책)을 정할 때 온 가족이 투표(PR 리뷰)하고, 결정되면 가계부(Git) 첫 장에 딱 붙여놓는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '마이크로서비스 DB 선정'에 대한 ADR 작성

- **문제 상황**: 서비스 초기, NoSQL을 쓸지 RDBMS를 쓸지 팀 내 의견이 팽팽하게 맞섬.
- **의사결정 (ADR 실천)**:
  1. **ADR-003 작성**: "주문 도메인 저장소로 PostgreSQL 선정" 제목으로 초안 작성.
  2. **Context 기술**: 현재 데이터 구조가 복잡하고 트랜잭션 정합성이 비즈니스 최우선 순위임을 명시.
  3. **Alternatives**: MongoDB와 비교 분석한 내용을 포함. (확장성은 좋으나 정합성 요건 미달)
  4. **Consequences**: 추후 데이터가 폭증하면 수평 확장에 비용이 들 수 있음을 미리 단점으로 기록.
  5. **승인**: 팀 미팅 후 **Accepted**로 상태 변경 후 `docs/adr/003-use-postgresql.md` 저장.
  6. **결과**: 1년 뒤 새로운 개발자가 "왜 우리 서비스는 MongoDB 안 써요?"라고 물었을 때, 말로 설명할 필요 없이 이 문서를 보여줌으로써 5분 만에 설득 완료.

### 도입 체크리스트
- **Simplicity**: 문서 작성이 10분 내로 끝날 만큼 템플릿이 간결한가?
- **Searchability**: ADR 번호와 태그를 통해 원하는 결정을 쉽게 검색할 수 있는가?
- **Code Alignment**: 실제 코드의 구조가 최신 ADR의 결정 사항과 일치하는가?

### 안티패턴
- **Post-hoc Documentation**: 이미 다 만들어놓고 나중에 "증적용"으로 억지로 쓰는 ADR. (결정 당시의 생생한 맥락이 사라짐)
- **Too Granular**: "변수명을 camelCase로 한다"와 같은 코딩 컨벤션까지 ADR로 쓰는 행위. (이런 건 Linter 설정으로 관리해야 함)

- **📢 섹션 요약 비유**: 일기는 그날그날 써야 진심이 담기듯, ADR도 결정이 내려진 그 뜨거운 순간에 기록해야 가치가 있습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Time)** | 회의 시간 30% 단축 | 과거 결정을 다시 논의하는 불필요한 반복 중단 |
| **정량적 (Agility)** | 온보딩 속도 2배 향상 | 신규 인력이 시스템의 역사와 원리를 읽는 것만으로 파악 |
| **정성적 (Standard)** | 기술 부채 가시화 | 결정의 부작용을 기록함으로써 향후 개선해야 할 지점을 명확히 인지 |
| **정성적 (Rational)** | 합리적 조직 문화 정착 | 개인의 고집이 아닌 논리적 근거(Context) 기반의 의사결정 습관화 |

### 미래 전망
- **Integrated ADR Tools**: IDE 내에서 코드를 수정하면 관련 ADR이 옆에 뜨거나, ADR을 기반으로 코드가 올바르게 짜였는지 체크하는 플러그인이 보편화될 것이다.
- **AI Knowledge Assistant**: 수백 개의 ADR을 학습한 AI가 "과거 005번 결정에 따르면 이 방식은 부작용이 큽니다"라고 실시간 조언해주는 환경이 구축될 것이다.

### 참고 표준
- **Documenting Architecture Decisions (M. Nygard)**: ADR의 표준 원전
- **ISO/IEC 42010**: Rationale(근거) 기록을 아키텍처 표준의 필수 요소로 정의
- **adr-tools**: ADR 생성을 돕는 표준 CLI 도구

- **📢 섹션 요약 비유**: "역사를 잊은 민족에게 미래는 없다"는 말은 소프트웨어에도 해당합니다. ADR은 우리가 만든 시스템의 과거를 기록하여, 더 나은 미래로 나아가게 돕는 아키텍처의 사관(史官)입니다.
