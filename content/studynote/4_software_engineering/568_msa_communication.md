# 568. 마이크로서비스 통신 패턴 (Req-Res, Pub-Sub, Saga)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스 통신 패턴은 분산된 서비스 간에 데이터를 주고받고 상태를 동기화하는 전략적 방식으로, 상호작용의 시급성과 데이터 정합성 요건에 따라 **동기(Req-Res)**, **비동기(Pub-Sub)**, **분산 트랜잭션(Saga)** 패턴을 선택적으로 적용한다.
> 2. **가치**: 서비스 간의 결합도를 제어하여 시스템의 확장성을 확보하며, 특히 네트워크 지연이나 부분적 장애가 전체 시스템 붕괴로 이어지는 연쇄 장애를 방지하는 동시에 복잡한 비즈니스 로직의 데이터 일관성을 유지한다.
> 3. **융합**: REST/gRPC(동기), Kafka/RabbitMQ(비동기) 등 기술 스택과 결합되며, 현대 아키텍처에서는 가용성을 극대화하기 위해 '최종 일관성(Eventual Consistency)'을 수용하는 메시지 기반 통신이 주류가 되고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 마이크로서비스는 각자 독립된 프로세스로 돌아가기 때문에 "어떻게 대화하느냐"가 시스템 성능과 안정성의 90%를 결정한다. 
  - **Request-Response**: 즉각적인 대답이 필요할 때 직접 묻고 답하는 방식.
  - **Pub-Sub**: 소식을 게시판에 뿌리고 필요한 사람만 보게 하는 비동기 방식.
  - **Saga**: 여러 서비스에 걸친 복잡한 업무를 단계별로 처리하고 실패 시 되돌리는 비즈니스 트랜잭션 방식.

- **💡 비유**: 
  - **Req-Res**: 상대방에게 전화를 걸어 "지금 바로 대답해!"라고 하는 **직통 전화**와 같습니다.
  - **Pub-Sub**: 잡지에 광고를 내면 구독자들이 각자 편한 시간에 읽는 **정기 구독**과 같습니다.
  - **Saga**: 여러 정거장을 거쳐야 하는 기차 여행에서, 중간에 사고가 나면 출발지까지 다시 되돌아오는 **환불 규정이 있는 장거리 여행**과 같습니다.

- **등장 배경**:
  1. **분산 시스템의 한계**: 모든 서비스를 동기(Req-Res)로 연결하면 하나만 느려져도 전체가 멈추는 '성능 병목'이 발생했다.
  2. **데이터 파편화**: DB가 쪼개지면서 여러 서비스의 데이터를 한꺼번에 고쳐야 할 때 데이터가 꼬이는(Inconsistency) 문제가 발생했다.
  3. **확장성 요구**: 서비스가 늘어날수록 호출 관계가 복잡해지는 '스파게티 통신'을 정리할 표준화된 패턴이 필요해졌다.

- **📢 섹션 요약 비유**: 복잡한 도시의 교통 체증을 해결하기 위해 지하철(동기), 버스(비동기), 그리고 환승 시스템(Saga)을 적재적소에 배치하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 통신 방식별 특징 비교

| 패턴 | 통신 성격 | 주요 기술 | 결합도 | 비유 |
|:---|:---|:---|:---:|:---|
| **Request-Response** | 동기 (Synchronous) | HTTP/REST, gRPC | 높음 | 실시간 대화 |
| **Pub-Sub** | 비동기 (Asynchronous) | Kafka, RabbitMQ | **낮음** | 단체 문자 메시지 |
| **Saga** | 트랜잭션 관리 | Event/Command 믹스 | 중간 | 단계별 계약 이행 |

---

### 2. Saga 패턴 동작 아키텍처 (Choreography vs Orchestration)

분산된 DB 사이의 데이터 정합성을 맞추는 두 가지 전략을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Saga 패턴: 분산 트랜잭션 및 보상 로직 시퀀스                 │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [성공 시나리오]                                                               │
 │ 주문 서비스 ──(주문생성)──▶ 결제 서비스 ──(결제성공)──▶ 재고 서비스 ──▶ [완료]      │
 │                                                                             │
 │ [실패 및 보상 시나리오 (Compensating Transaction)]                            │
 │                                                                             │
 │ 1. 주문 생성 (OK)                                                            │
 │ 2. 결제 승인 (OK)                                                            │
 │ 3. 재고 부족 (Fail!) ───┐                                                    │
 │                         │ [보상 이벤트 발행]                                  │
 │                         ▼                                                   │
 │ 4. 결제 취소 (Undo) ◀───┘                                                    │
 │ 5. 주문 거절 (Undo) ◀───┘                                                    │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 구현 방식:                                                            │  │
 │  │ - 코레오그래피 (Choreography): 중앙 통제 없이 서비스 간 이벤트로 연쇄 반응    │  │
 │  │ - 오케스트레이션 (Orchestration): 중앙 매니저가 각 단계의 성공/실패를 지휘    │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** MSA에서 2PC(2-Phase Commit) 같은 강력한 트랜잭션은 성능상 사용하기 어렵다. 대신 **Saga 패턴**은 로컬 트랜잭션을 순차적으로 실행한다. 만약 중간 단계(재고 서비스)에서 실패가 발생하면, 앞서 성공했던 단계들(결제, 주문)에게 **보상 트랜잭션 (Compensating Transaction)** 이벤트를 보내 이미 수정된 데이터를 원복시킨다. 이를 통해 기술적인 '원자성' 대신 비즈니스적인 **'최종 일관성'**을 달성한다.

---

### 심층 동작 원리: 메시지 큐를 통한 Pub-Sub의 신뢰성

비동기 통신에서 데이터 유실을 막는 장치들이다.
1. **At-least-once**: 메시지가 최소 한 번은 전달됨을 보장. (재전송 메커니즘)
2. **Idempotency (멱등성)**: 동일한 메시지를 두 번 받아도 시스템 상태가 변하지 않게 설계. (중복 처리 방지)
3. **Dead Letter Queue (DLQ)**: 처리에 실패한 메시지를 별도로 모아 나중에 수동 조치 가능케 함.

- **📢 섹션 요약 비유**: 편지가 배달 중에 사라지지 않게 등기 우편(재전송)으로 보내고, 만약 주소를 못 찾으면 우체국 보관함(DLQ)에 넣어두어 나중에라도 확인할 수 있게 하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 동기(Sync) vs 비동기(Async) 통신 선택 가이드

| 고려 요소 | 동기 통신 (Req-Res) | 비동기 통신 (Pub-Sub) |
|:---|:---|:---|
| **응답성** | 즉시 확인 필요 시 (로그인 등) | 지연 처리 가능 시 (알림, 통계 등) |
| **시스템 부하** | 호출 시점에 자원 점유 (Blocking) | **Buffering을 통해 부하 분산 가능** |
| **복잡도** | 상대적으로 낮음 | 높음 (메시지 인프라 관리 필요) |
| **추천 상황** | 사용자 대기형 서비스 | **대량 트래픽 처리, 장애 격리 우선 시** |

---

### 기술 융합: CQRS와 통신 패턴의 결합

조회와 명령을 분리하여 통신 효율을 극대화한다.
- **명령 (Command)**: 비동기 Pub-Sub을 통해 데이터 변경 사항을 전파.
- **조회 (Query)**: 동기 Req-Res를 통해 미리 준비된 최적화된 데이터를 즉시 반환.
- **가치**: "쓰기는 좀 늦어도 되지만, 읽기는 빨라야 한다"는 실제 사용자 경험에 최적화된 통신 구조를 만든다.

- **📢 섹션 요약 비유**: 신문을 만드는 과정(쓰기)은 밤새 천천히 진행되어도 되지만, 독자가 신문을 펼치는 순간(읽기)에는 모든 내용이 이미 인쇄되어 있어야 하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 장애에 강한 '배달 주문 시스템' 통신 설계

- **문제 상황**: '배달' 대행사 시스템이 자주 느려지는데, 이 영향으로 '주문' 앱까지 멈춰서 사용자들이 결제를 못 하고 나가는 현상 발생.
- **의사결정 (통신 패턴 최적화)**:
  1. **동기 탈피**: 주문 서비스와 배달 서비스 간의 Req-Res 연결을 끊고, 사이에 **메시지 큐(Kafka)**를 둔다.
  2. **상태 분리**: 주문이 완료되면 'DeliveryRequested' 이벤트를 발행하고 즉시 사용자에게 "주문 완료"를 리턴한다.
  3. **재시도 전략**: 배달 서비스가 죽어있더라도 이벤트는 큐에 안전하게 쌓이며, 배달 시스템 복구 시 밀린 배달 요청을 순차적으로 처리한다.
  4. **가시성**: 사용자가 배달 상태를 궁금해하므로, 배달 서비스가 진행 상황을 이벤트로 던지면 주문 서비스가 이를 받아 자신의 읽기 전용 DB에 업데이트해둔다. (CQRS 결합)

### 도입 체크리스트
- **Latency Budget**: 동기 호출이 3단계를 넘어가 전체 응답 시간이 1초를 초과하고 있지는 않은가?
- **Message Integrity**: 메시지 브로커 장애 시 데이터 유실을 막기 위한 고가용성(HA) 구성이 되어 있는가?
- **Traceability**: 서비스 간에 메시지가 오갈 때 'Correlation ID'를 사용하여 전체 흐름을 추적할 수 있는가?

### 안티패턴
- **Sync Overkill**: 모든 통신을 REST API로만 하려다가 시스템 전체가 가장 느린 서비스 속도에 맞춰지는 경우.
- **Zombie Saga**: 보상 트랜잭션(Undo) 로직을 제대로 짜지 않아, 중간에 에러가 났는데 돈은 빠져나가고 물건은 안 오는 데이터 불일치 상태 방치.

- **📢 섹션 요약 비유**: 서로 손을 꼭 잡고 걷다가(동기) 한 명이 넘어지면 다 같이 넘어지는 위험한 상황보다는, 서로 무전기(비동기)를 들고 일정한 거리를 유지하며 걷는 것이 훨씬 안전한 행군 방법입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Performance)** | 사용자 체감 속도 3배 향상 | 무거운 후속 작업의 비동기 전환을 통해 즉각적인 응답 제공 |
| **정량적 (Stability)** | 장애 전파 지수 80% 하락 | 서비스 간 느슨한 결합을 통해 부분 장애 시에도 핵심 기능 유지 |
| **정성적 (Integrity)** | 데이터 최종 정합성 보장 | Saga 패턴을 통한 복잡한 분산 업무의 논리적 무결성 확보 |
| **정성적 (Flexibility)** | 서비스 확장성 확보 | 새로운 구독자 서비스를 추가하는 것만으로 신규 기능 확장 가능 |

### 미래 전망
- **Event Mesh**: 특정 클라우드에 종속되지 않고 전 세계 모든 서비스 간에 이벤트를 실시간으로 라우팅해주는 거대한 '이벤트 그물망'이 표준 인프라가 될 것이다.
- **No-code Saga Management**: 비즈니스 흐름을 드래그 앤 드롭으로 설계하면, 복잡한 보상 트랜잭션 코드를 AI가 자동으로 생성하고 관리해주는 로우코드 플랫폼이 보편화될 것이다.

### 참고 표준
- **Reactive Streams**: 비동기 데이터 처리 표준
- **CloudEvents**: 클라우드 간 이벤트 정보 교환 표준
- **AMQP / gRPC Specification**: 통신 기술 규격

- **📢 섹션 요약 비유**: 수만 번의 통화(Req-Res)보다 한 번의 공지(Pub-Sub)가 효율적일 때가 있고, 꼼꼼한 확인(Saga)이 필수일 때가 있습니다. 상황에 맞는 통신 언어를 고르는 것이 아키텍트의 예술입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **Req-Res**는 친구에게 "지금 바로 사탕 줘!"라고 말하고 대답을 기다리는 **직접 대화**예요.
2. **Pub-Sub**은 친구들에게 "나 사탕 생겼어!"라고 큰 소리로 외치면, 사탕이 먹고 싶은 친구들만 달려오는 **소문 내기**랍니다.
3. **Saga**는 친구들과 사탕을 나눠 먹기로 약속했는데, 한 명이 못 먹게 되면 미안해서 다 같이 사탕을 다시 돌려주는 **착한 약속 지키기** 규칙이에요!
