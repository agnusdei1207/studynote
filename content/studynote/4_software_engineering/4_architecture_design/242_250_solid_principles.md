# 242-250. 객체지향 설계 원칙 (SOLID)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어의 위기는 변경의 불편함에서 비롯되며, SOLID는 **결합도(Coupling)**는 낮추고 **응집도(Cohesion)**를 높여 변경을 유연하게 수용하는 소프트웨어 설계의 구조적 철학이다.
> 2. **가치**: 유지보수 비용을 최소화하고 시스템의 수명을 연장하며, 테스트 주도 개발(TDD, Test Driven Development) 및 애자일(Agile) 프로세스의 기술적 토대가 된다.
> 3. **융합**: 클린 아키텍처(Clean Architecture), 마이크로서비스 아키텍처(MSA, Microservices Architecture)의 설계 기반이며, **CI/CD (Continuous Integration/Continuous Deployment)** 파이프라인의 안정성을 보장하는 핵심 요소이다.

---

### Ⅰ. 개요 (Context & Background)

**개념 및 정의**
객체지향 설계 원칙인 **SOLID**는 로버트 C. 마틴(Robert C. Martin)이 정립한 5가지 핵심 원칙의 앞자를 딴 말이다. 이는 소프트웨어 시스템이 변경에 유연하도록 하고, 이해하기 쉬우며 유지보수가 용이하도록 만드는 것을 목적으로 한다. 단순한 코딩 스타일을 넘어, 시스템의 아키텍처를 결정짓는 메타 설계 지침이다.

**등장 배경 및 철학**
초기 프로그래밍은 절차적(Procedural)으로 이루어져 모듈 간 의존성이 강해,一处 수정이 발생하면 전체 시스템에 **파급 효과(Side Effect)**가 미치는 '스파게티 코드(Spaghetti Code)' 문제가 있었다. 이를 해결하기 위해 캡슐화(Encapsulation)와 상속(Inheritance)을 적극 활용하나, 잘못된 상속과 의존 관계는 오히려 코드를 더 복잡하게 만들었다. SOLID는 객체지향 언어(Java, C#, C++ 등)의 특성을 극대화하면서도 재사용성을 저해하는 요인을 제거하기 위해 등장했다. 또한 실무에서는 **DRY (Don't Repeat Yourself)**, **KISS (Keep It Simple, Stupid)**, **YAGNI (You Aren't Gonna Need It)** 원칙과 결합하여 과도한 설계(Over-engineering)를 방지하는 가이드라인으로 활용된다.

```ascii
[소프트웨어 위기와 설계 원칙의 진화]

+------------------+          +---------------------+          +--------------------+
|   절차적 프로그래밍   |  ---->  |   객체지향(OO) 등장    |  ---->  |   SOLID/Agile 정착  |
+------------------+          +---------------------+          +--------------------+
| • 강한 결합도            |          | • 캡슐화/상속 도입      |          | • 유연한 아키텍처      |
| • 수정의 파급효과        |          | • 재사용성 증대         |          | • TDD, CI/CD 기반     |
| • 유지보수 어려움       |          | • But, 잘못된 사용      |          | • 지속 가능한 시스템   |
+------------------+          +---------------------+          +--------------------+
```

📢 **섹션 요약 비유**: 마치 건축에서 '내력벽(구조를 지탱하는 벽)'과 '비내력벽(간막이)'을 철저히 구분하여 설계하는 것과 같습니다. SOLID는 언제든지 remodel(변경)이 가능하도록 벽을 조립식으로 나누고 전기 배선을 쉽게 바꿀 수 있도록 설계하는 '건축 구조 기준'입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. SOLID 상세 분석**

SOLID는 5가지 원칙의 조합이며, 각각은 높은 응집도와 낮은 결합도를 달성하기 위한 구체적인 수단이다.

| 약어 | 전체 명칭 (Full Name) | 핵심 정의 | 위배 시 증상 | 실무 적용 포인트 |
|:---:|:---|:---|:---|:---|
| **SRP** | Single Responsibility Principle | "클래스는 오직 하나의 책임만 가져야 한다." | 하나의 클래스를 수정할 때 여러 이유로 인해 수정이 발생함 | UI 로직, 비즈니스 로직, DB 접근 로직 분리 (MVC 패턴 등) |
| **OCP** | Open-Closed Principle | "소프트웨어 개체는 확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 한다." | 기능 추가 시 기존 안정 코드를 수정해야 하여 Bug 유발 | 인터페이스(Interface)를 통한 추상화 및 다형성(Polymorphism) 활용 |
| **LSP** | Liskov Substitution Principle | "자식 클래스는 부모 클래스를 대체할 수 있어야 한다." | 자식 타입 사용 시 논리적 오류 발생 (상속의 오용) | `Rectangle`과 `Square` 관계처럼, `is-a` 관계가 성립되지 않는 상속 제거 |
| **ISP** | Interface Segregation Principle | "클라이언트는 자신이 사용하지 않는 인터페이스에 의존하지 않아야 한다." | 사용하지 않는 메서드 구현 강제로 인한 코드 번잡증 | 범용 인터페이스를 역할별로 세분화 (Role-based Interfaces) |
| **DIP** | Dependency Inversion Principle | "상위 모듈은 하위 모듈에 의존해서는 안 된다. 추상화에 의존해야 한다." | 하위 레벨의 변경이 상위 레벨의 비즈니스 로직 흐름을 막음 | `new` 연산자 사용 지양 및 DI (Dependency Injection) 컨테이너 활용 |

**2. 심층 동작 원리: OCP와 DIP의 시너지**
OCP와 DIP는 현대적인 프레임워크(Spring, Django 등)의 핵심 동작 원리이다.

```ascii
[OCP(개방-폐쇄) 및 DIP(의존 역전) 구조도]

+---------------------+                 Uses             +--------------------+
|   Client (High Level) |  -------------------------------->  <<Interface>>     |
|   (Business Logic)    |   (Abstract Dependency Rule)      |  IServiceProvider  |
+---------------------+                                   +--------------------+
        ^                                                     ^
        |                                                     | Implements
        | Inherited/Injected                                  |
        |                                                     |
+----------------+           +----------------+      +--------------------+
| ConcreteService|           | LegacyService  |      | NewService         |
| (Current Impl) |           | (Old Impl)     |      | (Future Impl)      |
+----------------+           +----------------+      +--------------------+

[상황: NewService 추가]
1. 기존 Client 코드는 수정하지 않음 (OCP 만족).
2. Client는 Interface에만 의존하므로 구현체 교체 자유로움 (DIP 만족).
```
*(해설: 상위 모듈은 추상화된 `<<Interface>>`에만 의존하며, 실제 구현체인 `ConcreteService`나 `NewService`는 외부(DI Container)에서 주입받는다. 이를 통해 신규 기능 추가 시 비즈니스 로직(Client)의 변경을 막고(Fixed), 새로운 구현체만 늘려가는(Open) 방식이 가능해진다.)*

**3. 실무 설계 3대 원칙 (K.Y.D)**
SOLID가 구조적 원칙이라면, 아래 세 가지는 코딩 실천의 지침이다.
- **DRY (Don't Repeat Yourself)**: 로직의 중복은 변경 지점을 여러 곳으로 만들어 오류의 원인이 된다. 공통 모듈로의 추출이 필수적이다.
- **KISS (Keep It Simple, Stupid)**: 복잡한 설계보다는 단순하고 명확한 구조가 성능과 유지보수에 유리하다. ("Premature optimization is the root of all evil")
- **YAGNI (You Aren't Gonna Need It)**: "나중에 필요할 것"이라는 예측으로 현재 불필요한 기능을 구현하지 말라는 원칙이다. 과잉 설계(Over-engineering)를 방지한다.

📢 **섹션 요약 비유**: 자동차 설계와 같습니다. **SRP**는 엔진, 핸들, 타이어를 각각 독립된 모듈로 만드는 것이고, **OCP/DIP**는 '규격화된 배선 연결부'를 통해 어떤 브랜드의 부품이든 장착 가능하게 만드는 것입니다. **KISS/DRY**는 불필요하게 복잡한 기계 장치 없이 가장 단순한 방식으로 목적지까지 이동하게 하는 설계 철학입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 절차적 프로그래밍 vs SOLID 기반 OOP**

| 비교 항목 | 절차적 프로그래밍 (Procedural) | SOLID 기반 객체지향 (OOD) |
|:---|:---|:---|
| **데이터 흐름** | 중앙 집중식 데이터 저장소, 함수가 데이터를 처리 | 데이터와 행위(Method)가 객체(Object) 내에 캡슐화 |
| **결합도(Coupling)** | **높음 (High)**: 함수 간 호출 순서에 강한 의존 | **낮음 (Low)**: 인터페이스를 통한 느슨한 결합 |
| **확장성(Scalability)** | 낮음: 코드 추가 시 기존 로직 수정 필수 | 높음: 상속 및 구현(Implements)을 통한 기능 확장 |
| **테스트 용이성** | 어려움: 모의 객체(Mock) 생성 불가능 | 쉬움: 인터페이스를 통한 모의 객체 주입 가능 |

**2. 아키텍처 관점: 모놀리스 vs MSA (Microservices Architecture)**
SOLID는 거대한 모놀리식 애플리케이션을 쪼개는 **MSA**의 이론적 배경이 된다.
- **SRP & ISP**: 마이크로서비스를 "도메인별 역할"로 쪼개는 근거가 된다. 하나의 서비스가 너무 많은 역할(Big Ball of Mud)을 담당하면 안 된다.
- **DIP**: 서비스 간 통신 시 구체적인 서비스 주소가 아닌 **API Gateway**나 추상화된 서비스 인터페이스를 통해 통신하게 한다.

```ascii
[MSA에서의 SRP/ISP 적용 관점]

[Monolithic (Bad)]          [Microservices (Good)]
+------------------+         +-------------+  +-------------+  +-------------+
|  E-Commerce App  |         | Order Svc   |  | User Svc    |  | Product Svc |
|------------------|         |(Order Mgmt) |  |(User Auth)  |  |(Inventory)  |
| - Order Logic    |         +-------------+  +-------------+  +-------------+
| - User Logic     |                 ^                ^                ^
| - Product Logic  |                 |                |                |
+------------------+                 +-------(Event Bus/Rest API)------+
(Check: 하나의 클래스/서비스가 너무 많은 책임을 짊지지고 있음)
```

*(해설: 왼쪽의 단일 애플리케이션은 SRP를 위배하여 변경이 위험하지만, 오른쪽은 책임을 분리하여 서비스 간 독립성을 보장한다. 단, 서비스 간 통신 비용과 복잡도가 증가하므로 DIP를 통해 통신 계층을 추상화해야 한다.)*

📢 **섹션 요약 비유**: 마치 거대한 종합백화점(모놀리스)을 전문점(마이크로서비스) 촌으로 재개발하는 것과 같습니다. 각 전문점(서비스)은 자신의 장사(책임)만 챙기며, 손님(클라이언트)은 쇼핑몰 앱(인터페이스)을 통해 상점을 이용합니다. 이는 한 가게가 리모델링 중이라도 다른 가게에 영향을 주지 않는 구조입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. 실무 시나리오 및 의사결정**
- **상황 A (SRP 위배)**: 데이터베이스 접속 코드와 UI 렌더링 코드가 하나의 클래스에 섞여 있어 DB를 Oracle에서 MySQL로 변경할 때 화면 코드까지 수정해야 하는 문제 발생.
    - **해결**: Repository 패턴 등을 적용하여 데이터 소스 접근 계층(DAL)을 분리.
- **상황 B (OCP 위배)**: 새로운 결제 수단(간편결제) 추가 시 기존 `PaymentService` 클래스의 `if-else` 블록을 수정하여 테스트 코드 깨짐.
    - **해결**: `PaymentStrategy` 인터페이스를 정의하고 `KakaoPay`, `NaverPay` 클래스를 추가하여 다형성으로 처리.
- **상황 C (DIP 적용)**: 날씨 API를 사용하는 로직에서 특정 `WeatherApi` 클래스를 `new`로 생성하여 테스트 시 실제 API 호출로 인한 지연 발생.
    - **해결**: 생성자를 통해 `IWeatherProvider` 인터페이스를 주입받고, 운영환경에서는 실제 구현체, 테스트에서는 가짜(Stub) 구현체를 주입.

**2. 도입 체크리스트**

| 구분 | 체크항목 |
|:---|:---|
| **기술적** | • 로직 변경 시 연관된 클래스가 2개 이상 수정되는가? (SRP 체크) <br> • 인터페이스를 추가하고 구현체만 늘려서 기능을 확장할 수 있는가? (OCP 체크) <br> • 상위 모듈이 하위 클래스 Import를 하고 있는가? (DIP 위배 여부) |
| **운영/보안** | • 불필요한 의존성 주입으로 인한 메모리 누수나 성능 저하가 없는가? <br> • 인터페이스 분리(ISP)로 인해 발생하는 클래스 파일 관리 부담이 감당 가능한가? |

**3. 안티패턴 (Anti-pattern)**
- **God Object**: 하나의 클래스가 시스템의 거의 모든 기능을 다 처리하려고 시도하는 것. (SRP 위배)
- **Shotgun Surgery**: 하나의 기능을 수정하기 위해 파일 여러 곳을 수정해야 하는 코드. (응집도 저하)
- **YAGNI 위반**: "혹시 몰라서" 미리 만들어둔 추상화 계층이 실제로는 불필요하게 복잡성만 증가시키는 경우.

📢 **섹션 요약 비유**: 도시 계획 설계와 같습니다. **SRP를 무시한 설계**는 주택가 한복판에 공장을
