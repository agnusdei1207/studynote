# 232-241. UML(Unified Modeling Language) 다이어그램

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 시스템의 설계도면으로, 객체지향 분석 및 설계(OOAD) 결과를 표준화된 시각 언어로 명세화하는 도구.
> 2. **가치**: 이해관계자(개발자, 분석가, 클라이언트) 간의 의사소통 오류를 최소화하여 개발 생산성을 30% 이상 향상시키고, 유지보수 비용을 절감.
> 3. **융합**: MDA(Model Driven Architecture) 기반의 코드 자동 생성 및 DevOps 파이프라인 내 설계 문서화와 연계됨.

---

### Ⅰ. 개요 (Context & Background)

UML (Unified Modeling Language)은 객체지향 시스템의 산출물을 명세하기 위한 표준화된 그래픽 언어입니다. 1997년 OMG (Object Management Group)에서 표준으로 채택되었으며, 래츠(Rumbaugh), 제이콥슨(Jacobson), 뷸라(Booch) 등 세 명의 방법론가가 제안한 기법을 통합하여 탄생했습니다. 단순한 그림 그리기 도구가 아니라, 모델 기반 아키텍처(MDA)의 핵심 도구로서 시스템의 요구사항 분석부터 구현, 테스트까지 전체 수명 주기(SDLC)를 아우릅니다.

**💡 비유: 건축 설계도면**
UML은 건축가가 짓는 집의 **"평면도(구조), 전기 배선도(흐름), 인테리어 시공 매뉴얼(상세)"**를 그리는 것과 같습니다. 현장(코드)에서 일하는 시공工人(개발자)과 주인(클라이언트) 모두가 같은 그림을 보고 일해야 집이 무너지지 않듯, UML은 소프트웨어라는 건물을 지키는 공용 언어입니다.

**등장 배경**
1.  **기존 한계**: 1990년대 중반까지만 해도 50여 가지가 넘는 객체지향 분석/설계 방법론(BN O-O, OMT 등)이 난립하여 개발자 간의 혼선이 극심했습니다.
2.  **혁신적 패러다임**: 세계 3대 방법론가가 자신들의 기법을 통합하여 'Unified(통합된)' 언어를 만들었고, 이는 시각적 모델링의 표준이 되었습니다.
3.  **현재 비즈니스 요구**: 대규모 분산 시스템 및 클라우드 환경에서 복잡한 로직을 문서화하고, AI 기반 코드 생성 및 자동화된 테스트를 위한 '소스코드 그 자체'로서의 모델링이 요구됩니다.

```ascii
[ UML Evolution & Standardization ]

  (과거: 방법론의 난립)           (현재: 표준화된 시스템)
 +-------+       +-------+       +-------+       +-------+
 | OMT   |       | Booch |       | OOSE  |  -->  |  UML  |
 +-------+       +-------+       +-------+       +-------+
    |               |               |              |
    +-------+-------+-------+-------+              |
            |   Unification (1995-1997)             |
            v                                       v
 [ Confusion ] ------------------------> [ OMG Standard ]
 [ Fragmented]                           [ MDA / Code Gen ]
```

**해설**:
위 다이어그램은 UML이 탄생하기 전, 객체지향 방법론이 서로 호환되지 않아 개발자들에게 '바벨탑' 혼란을 주었던 상황을 보여줍니다. 이를 해결하기 위해 OMT, Booch, OOSE 등 주요 기법이 통합(Unification)되어 OMG 표준으로 등록되었음을 시각화했습니다. 현재는 단순한 문서를 넘어 MDA(Model Driven Architecture)의 핵심 소스로 활용됨을 나타냅니다.

> **📢 섹션 요약 비유**: UML 도입은 **"각기 다른 언어를 쓰는 공구점들이 급격히 성장하자, 국제 표준 기구가 '공구 규격'을 정해 누구든지 전 세계 어디서나 호환되는 부품을 만들게 한 것"**과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

UML은 크게 **정적인 구조(Structure)**를 다루는 구조적 다이어그램과 **동적인 흐름(Behavior)**을 다루는 행위 다이어그램으로 분류됩니다. 이를 통해 시스템을 입체적으로 정의합니다.

#### 1. 구성 요소 분류 (구조 vs 행위)

| 분류 | 다이어그램 명칭 (Abbreviation) | 핵심 역할 | 주요 관심사 |
|:---:|:---|:---|:---|
| **Structural** | **Class Diagram** | 시스템의 뼈대 | 객체의 속성, 메서드, 클래스 간 관계(상속, 연관) |
| | **Component Diagram** | 구현 단위 모듈화 | 물리적 코드 컴포넌트(.jar, .dll) 간 의존성 |
| | **Deployment Diagram** | 하드웨어 매핑 | 서버, 노드, 장비 간 물리적 배치 |
| **Behavioral** | **Use Case Diagram** | 사용자 요구사항 | 액터와 시스템 간의 기능적 인터랙션 |
| | **Sequence Diagram** | 시간 순서별 메시지 교환 | 객체 간 메서드 호출 순서 및 시간 |
| | **State Machine Diagram** | 상태 변화 | 이벤트에 따른 객체 내부 상태 전이 |
| | **Activity Diagram** | 흐름 제어 | 업무 프로세스 및 제어 로직 흐름 |

#### 2. 핵심 구조: 클래스 다이어그램 (Class Diagram)

객체지향의 핵심인 클래스를 정의하며, 시스템의 가장 기본이 되는 정적 틀입니다.

```ascii
[ Class Structure & Relationships Detail ]

         [ User ]                    [ Account ]                    [ Bank ]
      (Class Name)                 (Class Name)                  (Class Name)
      +-------------+              +-------------+               +-------------+
      | - id: String |              | - balance:Fl|               | - code: Int |
      | - name: Str  |              | - number:Str|               +-------------+
      +-------------+              +-------------+
              |                           ^                           ^
              | 1                0..1 *   |  Aggregation              |
              +-------------------------->+ (Empty Diamond)           |
              | Association               |                           |
              |                           +---------------------------+
              |                               Composition             |
              |                           (Filled Diamond)            |
              v                                                         v
      +-------------+              +-------------+               +-------------+
      | login()     |              | deposit()   |               | audit()     |
      | logout()    |              | withdraw()  |               |             |
      +-------------+              +-------------+               +-------------+

      Legend:
      -----> : Dependency (Uses)
      -----> : Association (Field reference)
      <>---- : Aggregation (Has-a, weak lifecycle link)
      <#>---- : Composition (Part-of, strong lifecycle link)
      --|>-- : Generalization (Inheritance)
```

**해설**:
이 다이어그램은 **은행 시스템(Banking System)**의 일부를 모델링한 것입니다.
1.  **User**는 **Account**를 소유하며, 일반적인 연관(Association) 관계를 가집니다(1:N).
2.  **Account**는 **Bank**와 'Aggregation(집합)' 또는 'Composition(합성)' 관계를 가질 수 있습니다.
    *   **집합(Aggregation)**: 계좌(Account)는 은행(Bank)에 속하지만, 은행이 사라진다고 계좌 객체가 즉시 소멸되지 않는 약한 결합 (예: 금융 지주사 구조).
    *   **합성(Composition)**: 주문(Order)과 항목(Line Item)처럼, Account가 삭제되면 관련 Transaction이 모두 삭제되는 강한 결합 (다이어그램상 채워진 마름모).
3.  각 박스 내부는 **[속성(Properties)]**과 **[오퍼레이션(Methods)]**으로 나뉘며 접근 제어자(+/-, private/public)를 명시합니다.

**심층 동작 원리 (Multiplicity & Visibility)**
*   **Multiplicity (다중도)**: `1`, `0..1`, `*`, `0..*` 등을 사용하여 연관된 객체의 개수를 명시합니다. 예를 들어, 하나의 User는 여러 개의 Account(`0..*`)를 가질 수 있습니다.
*   **Visibility (가시성)**:
    *   `+` (Public): 모든 객체가 접근 가능.
    *   `-` (Private): 자기 자신만 접근 가능 (캡슐화).
    *   `#` (Protected): 자식 클래스(상속받은 클래스)까지 접근 가능.

> **📢 섹션 요약 비유**: 클래스 다이어그램은 **"자동차 설계도면에서 엔진, 바퀴, 핸들 등 부품들의 규격과 서로 어떻게 연결되는지를 명시한 '조립 도면'"**과 같습니다. 여기서 합성(Composition)은 용접된 부품이고, 집합(Aggregation)은 나사로 체결된 부품에 비유할 수 있습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

개발자는 상황에 따라 적절한 다이어그램을 선택해야 합니다. 특히 **시스템 구조**를 볼 것인가, **동적 흐름**을 볼 것인가에 따라 전략이 달라집니다.

#### 1. 행위 다이어그램 비교 분석 (Sequence vs Activity)

| 구분 | 시퀀스 다이어그램 (Sequence Diagram) | 액티비티 다이어그램 (Activity Diagram) |
|:---|:---|:---|
| **핵심 초점** | **"누가 누구에게 언제 메시지를 보내는가?"** (상호작용) | **"무슨 일을 어떤 순서로 처리하는가?"** (흐름/로직) |
| **시간 개념** | 명시적 (Y축이 시간의 흐름) | 암시적 (화살표 방향) |
| **주요 요소** | 객체(Object), 생명선(Lifeline), 메시지(Message) | 액션(Action), 결정 노드(Decision), 병행 처리(Fork/Join) |
| **장점** | 객체 간 메시지 호출 스택을 명확히 확인 가능 | 복잡한 로직의 분기(IF/ELSE), 반복(Loop) 시각화에 유리 |
| **단점** | 복잡한 분기 로직이 들어가면 다이어그램이 너무 넓어짐 | 메서드 호출 인자나 리턴 값 같은 세부 정보 표현이 어려움 |

#### 2. 객체 상태 모델링 (State Machine Diagram)

객체가 **어떤 상태(State)**에 있으며, **어떤 이벤트(Event)**로 상태가 **전이(Transition)**되는지를 정의합니다. UI 화면 전이나 통신 세션 관리에 필수적입니다.

```ascii
[ Order Processing State Machine ]

      +------------------+
      |   [Start]        |
      +---------+--------+
                |
             (Create Order)
                v
      +------------------+       +------------------+
      | [Checking]       |------>| [Failed]         |
      | Payment Valid?   | <Fail | (End)            |
      +-------+----------+       +------------------+
              |
            (Success)
              |
              v
      +------------------+
      | [Shipped]        |<---------------------------+
      +-------+----------+                             |
              |                                      (Return)
              v
      +------------------+       +------------------+
      | [Delivered]      |------>| [Closed]         |
      +------------------+       +------------------+
```

**해설**:
이 다이어그램은 '주문(Order)' 객체의 생애 주기를 나타냅니다.
1.  주문 생성 후, 결제 검증(Checking) 상태로 들어갑니다.
2.  실패(Fail) 시 종료되고, 성공 시 출고(Shipped) 상태로 전이됩니다.
3.  배송(Delivered) 후 완료(Closed)되지만, 반품(Return) 이벤트 발생 시 다시 이전 단계로 돌아가거나 별도 로직을 타게 됩니다.
이처럼 상태 다이어그램은 특정 상태에서 수행 가능한 동작과 불가능한 동작을 명확히 하여 논리 오류를 방지합니다.

#### 3. 타 과목(컴퓨터 구조/DB)과의 융합
*   **DB 스키마 설계**: 클래스 다이어그램은 ERD (Entity Relationship Diagram)로 변환됩니다. 클래스는 엔티티가 되며, 연관 관계는 외래키(Foreign Key) 전략(1:1, 1:N)으로 매핑됩니다.
*   **네트워크 프로토콜**: 배포 다이어그램(Deployment Diagram)은 서버, 방화벽, 로드밸런서(Load Balancer) 등의 물리적 위상을 나타내어 네트워크 아키텍처와 직접 연결됩니다.

> **📢 섹션 요약 비유**: 시퀀스 다이어그램은 **"탁구 게임 영상(선수 간의 공 주고받는 기록)"**이라면, 액티비티 다이어그램은 **"게임 규칙서(어떤 점수에서 어떤 행동을 해야 하는지)"**와 같습니다. 상태 다이어그램은 야구 게임에서 **"이닝 진행 상황판(회초, 회말, 아웃 카운트)"**과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서는 단순히 그림을 그리는 것이 목적이 아니라, 문서화 자동화, 코드 리뷰, 클라이언트 커뮤니케이션을 위해 활용합니다.

#### 1. 실무 시나리오 및 의사결정
1.  **신규 프로젝트 입찰/제안**:
    *   **상황**: 고객이 비즈니스 흐름을 이해하지 못함.
    *   **판단**: 복잡한 설계보다는 **Use Case Diagram**과 **Activity Diagram**을 활용해 업무 흐름을 시각화하고 기능 목록을 명확히 하여 신뢰도를 높임.
2.  **레거시 시스템 리팩토링**:
    *   **상황**: 유지보수가 불가능한 스파게티 코드(C언어나 전역 변수 남발) 발견.
    *   **전략**: 리버스 엔지니어링 도구를 통해 기존 코드를 **Class Diagram**으로 변환하여 구조를 파악한 후, 모듈 간 결합도를 낮추는 방향으로 재설계.
3.  **분산 시스템 오
