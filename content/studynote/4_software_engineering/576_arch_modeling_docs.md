# 576. 아키텍처 모델링 및 문서화 기법 (UML, ADR)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 아키텍처 모델링은 시스템의 복잡한 구조와 행위를 표준화된 기호(UML 등)로 시각화하는 과정이며, 문서화는 설계 당시의 **의사결정 근거(Rationale)**와 맥락을 기록하여 미래의 유지보수자가 시스템을 온전히 이해하도록 돕는 활동이다.
> 2. **가치**: 텍스트만으로 전달하기 힘든 추상적 설계를 구체화하여 이해관계자 간의 오해를 줄이고, 특히 **ADR (Architecture Decision Records)**을 통해 "왜 이 기술을 선택했는가"에 대한 역사를 남김으로써 설계의 일관성과 추적성을 확보한다.
> 3. **융합**: 시각적 표현 도구인 UML과 맥락 기록 도구인 ADR이 결합되어 '살아있는 문서화'를 실현하며, 현대의 Agile 및 DevOps 환경에서는 코드로 관리되는 문서(Documentation as Code) 기술과 융합되고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **아키텍처 모델링 (UML)**: 소프트웨어의 청사진을 그리는 표준 언어다. 클래스 간의 정적 관계나 객체 간의 동적 흐름을 그림으로 표현한다.
  - **아키텍처 의사결정 기록 (ADR)**: 설계 과정에서 내린 중요한 결정들을 짧은 텍스트 파일로 남기는 기법이다.

- **💡 비유**: 모델링과 문서화는 **"건축 설계도와 회의록"**과 같습니다. **UML**이 건물의 구조와 배관을 그린 도면이라면, **ADR**은 "왜 이 건물을 내진 설계로 지었는지", "왜 철근 대신 탄소 섬유를 썼는지"에 대한 이유를 적어둔 일지와 같습니다.

- **등장 배경**:
  1. **암묵적 지식의 위험**: 설계자의 머릿속에만 있는 지식은 설계자가 퇴사하는 순간 사라진다. 이를 '명시적 지식'으로 자산화해야 했다.
  2. **복잡도 증가**: 수만 개의 클래스가 얽힌 시스템을 코드만 보고 파악하는 것은 불가능에 가깝다. 추상화된 시각 자료가 필요해졌다.
  3. **Agile 환경의 문서화 부족**: 거창한 명세서(Spec)를 쓸 시간은 없지만, 핵심 결정 사항은 기록해야 하는 요구사항을 충족하기 위해 가벼운 ADR 기법이 부상했다.

- **📢 섹션 요약 비유**: 보물을 숨겨둔 장소의 지도(UML)만 주는 것이 아니라, 왜 그 장소를 선택했는지와 함정을 피하는 법(ADR)까지 상세히 알려주는 완벽한 인수인계서입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 모델링 및 문서화 기법 비교

| 구분 | UML (Unified Modeling Lang.) | ADR (Arch. Decision Record) |
|:---|:---|:---|
| **성격** | **시각적 (Visual)**, 형식적 | **텍스트적 (Textual)**, 실무적 |
| **주요 내용** | 시스템의 구조, 행위, 배치 등 | 결정 사항, 맥락, 대안, 결과 |
| **핵심 도구** | Class, Sequence, State Diagram | Markdown 파일, Git Repository |
| **가치** | **현재 상태**의 명확한 전달 | **과거 결정**의 근거 보존 |

---

### 2. ADR(Architecture Decision Record) 구성 요소

하나의 ADR은 짧고 간결하게 유지되어야 한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      ADR (아키텍처 의사결정 기록) 표준 템플릿                     │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ 1. Title: [ADR-005] 마이크로서비스 간 통신을 위해 Kafka 도입                    │
 │                                                                             │
 │ 2. Status: Proposed ──▶ Accepted ──▶ Superseded (상태 전이)                   │
 │                                                                             │
 │ 3. Context (맥락):                                                           │
 │    - 기존 HTTP 통신 시 결제 서비스 지연이 주문 서비스로 전파됨.                   │
 │    - 실시간 통계 처리를 위한 대량의 이벤트 데이터 발생 중.                       │
 │                                                                             │
 │ 4. Decision (결정):                                                          │
 │    - 메시지 브로커로 Apache Kafka를 도입하여 비동기 Pub-Sub 구조로 전환.         │
 │                                                                             │
 │ 5. Consequences (결과):                                                      │
 │    - 장점: 서비스 간 결합도 저하, 확장성 및 내구성 향상.                         │
 │    - 단점: 메시지 인프라 관리 비용 증가, 최종 일관성 문제 발생.                  │
 │                                                                             │
 │  ※ 핵심: "나중에 온 사람이 이 문서를 보고 왜 이렇게 짰는지 납득할 수 있는가?"      │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ADR의 정수는 **Consequences (결과)** 섹션이다. 어떤 기술을 선택했을 때 얻는 이득뿐만 아니라 감수해야 할 '부작용'까지 솔직하게 기록한다. 이는 나중에 시스템에 문제가 생겼을 때 "아, 당시 이런 한계를 알고도 선택했구나"라고 설계자의 의도를 파악하게 하여, 불필요한 비난 대신 합리적인 대안(Superseded)을 찾는 토대가 된다. 모든 ADR은 코드와 함께 **Git**에서 버전 관리되어야 한다.

---

### 심층 동작 원리: UML의 다각도 표현 (Views)

시스템을 입체적으로 설명하기 위해 3대 다이어그램을 주로 활용한다.
1. **Class Diagram**: 객체들의 정적인 관계와 구조 설명. (건축물 단면도)
2. **Sequence Diagram**: 시간 흐름에 따른 메시지 교환 시퀀스. (업무 시나리오)
3. **Deployment Diagram**: 소프트웨어가 물리적 서버에 배치되는 구조. (도시 계획도)

- **📢 섹션 요약 비유**: 물건의 모양(Class)만 보여주는 게 아니라, 어떻게 움직이는지(Sequence), 그리고 어디에 놓이는지(Deployment)까지 보여주는 입체 설계도입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 문서화의 3대 패러다임 변화

| 시대 | 방식 | 특징 | 비유 |
|:---|:---|:---|:---|
| **과거** | **Big Upfront Doc** | 개발 전 수백 페이지 명세서 작성 | 백과사전 출판 |
| **현재** | **Just-in-Time Doc** | 필요한 시점에 가벼운 UML/ADR 작성 | 포스트잇 메모 |
| **미래** | **Living Documentation** | 코드에서 문서 자동 추출, AI 가이드 | 스스로 자라는 나무 |

---

### 기술 융합: C4 모델과 UML

UML이 너무 복잡하다는 비판을 해결하기 위해 **C4 모델**이 융합된다.
- **Level 1 (Context)**: 시스템과 외부 사용자/시스템의 관계.
- **Level 2 (Container)**: 웹 앱, 모바일 앱, DB 등 거시적 실행 단위.
- **Level 3 (Component)**: 컨테이너 내부의 주요 컴포넌트들.
- **Level 4 (Code)**: 상세 클래스 구조 (UML 활용).

- **📢 섹션 요약 비유**: 구글 맵에서 세계 지도(Level 1)부터 우리 집 앞 골목(Level 4)까지 줌인(Zoom-in)하며 보여주는 직관적인 시각화 방식입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '레거시 탈출'을 위한 아키텍처 복원 프로젝트

- **문제 상황**: 20년 된 시스템을 클라우드로 이전해야 하는데, 문서가 하나도 없고 코드는 스파게티라 구조 파악이 안 됨.
- **의사결정 (모델링 및 문서화 전략)**:
  1. **Reverse Modeling**: 코드를 읽어 **Sequence Diagram**을 역으로 그려 현재의 핵심 업무 흐름을 시각화한다.
  2. **ADR 소급 작성**: 과거 개발자들과 인터뷰하여 "왜 이 DB를 썼는지" 등 중요한 결정 사항을 `ADR-000` 번호부터 역순으로 기록한다.
  3. **To-be C4 모델링**: 클라우드 환경의 목표 구조를 C4 모델의 Container 레벨로 설계하여 이해관계자 승인을 득한다.
  4. **GitOps 통합**: 모든 문서를 Markdown과 PlantUML로 작성하여 소스 코드와 함께 Git에 저장하고, 코드 수정 시 문서 수정도 의무화한다.

### 도입 체크리스트
- **Freshness**: 코드는 변했는데 문서(UML)는 옛날 상태로 방치되고 있지 않은가? (주기적 업데이트 필수)
- **Accessibility**: ADR 문서가 Wiki나 Git에 잘 정리되어 누구나 5초 안에 찾을 수 있는가?
- **Level of Detail**: 너무 사소한 코드 변경까지 ADR로 남겨 문서가 쓰레기장이 되지는 않았는가? (아키텍처적 결정에만 집중)

### 안티패턴
- **Write-only Documentation**: 아무도 읽지 않는 길고 지루한 문서를 쓰는 데 시간을 낭비하는 행위.
- **Diagram without Text**: 그림만 있고 설명(Context)이 없어, 보는 사람마다 해석이 달라지는 경우.

- **📢 섹션 요약 비유**: "백 마디 말보다 한 장의 그림이 낫다"고 하지만, 그 그림이 왜 그려졌는지 설명하는 "한 줄의 글"이 없으면 그 그림은 미궁이 됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Time)** | 온보딩 시간 50% 단축 | 신규 인력이 시스템 구조를 파악하는 속도 획기적 개선 |
| **정량적 (Bug)** | 설계 미스에 의한 장애 감소 | 시각적 검토를 통해 컴포넌트 간 부적절한 결합 사전 식별 |
| **정성적 (Standard)** | 지식의 영속성 확보 | 개인의 머릿속 지식을 조직의 자산으로 전환 (KM) |
| **정성적 (Rational)** | 합리적 의사결정 문화 | 과거의 실패나 성공의 이유(ADR)를 근거로 최선의 기술 선택 가능 |

### 미래 전망
- **AI-Auto-ADR**: 개발 회의 녹취록과 코드 변경 이력을 AI가 분석하여, 초안 ADR을 자동으로 작성해주는 기술이 도입될 것이다.
- **Digital Twin Architecture**: 실제 시스템의 실시간 상태가 아키텍처 모델(UML)에 실시간으로 투영되어, 설계와 운영의 간극이 제로가 되는 '디지털 트윈 아키텍처'가 실현될 것이다.

### 참고 표준
- **OMG UML 2.5.1**: 시각적 모델링의 글로벌 표준
- **Architecture Decision Records (Michael Nygard)**: ADR 기법의 창시자 가이드
- **IEEE 42010**: 시스템 및 소프트웨어 아키텍처 기술 표준

- **📢 섹션 요약 비유**: 기록되지 않은 아키텍처는 존재하지 않는 것과 같습니다. 모델링과 문서화는 우리가 만든 디지털 건축물이 시간이 흘러도 빛바래지 않게 지켜주는 '기록의 마법'입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **모델링**은 멋진 성을 짓기 전에 **"어디에 창문을 내고 어디에 문을 만들지" 미리 그려보는 그림 지도**와 같아요.
2. **ADR**은 "왜 성벽을 나무 대신 돌로 만들었는지" **그 이유를 일기장에 꼼꼼히 적어두는 것**과 같답니다.
3. 이 지도와 일기장이 있으면, 나중에 성을 고칠 때나 동생에게 성을 물려줄 때 아주 큰 도움이 돼요!
