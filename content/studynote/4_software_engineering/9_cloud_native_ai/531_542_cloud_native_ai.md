# 531-542. 클라우드 네이티브 및 AI 아키텍처 심화

> **핵심 인사이트**: 클라우드 네이티브는 "이주 후 떠나는 수"아파트아트 아니라, **여행 수속(Scalability)**과 **회복력(Resilience)**을 위한 **애프스케일(Scaling)** 원칙으로 설계합니다. **타력성 있는**(Economics of scale)** 서비스들은 애플리케이션의 부하에 따라 **독립적으로 확장**할 수 있도록(Decouple)합니다. 이를 통해 **비즈니스 능력(Decompose by Business Capability)** 또는 **DDD 하위 도메인(Decompose by Subdomain)** 방식을 사용합니다:
* **DDD 기반**: 풍부한 비즈니스 맥락 파악, 경계 컨텍스트 명확
* **MSA 식별**: 각 서비스는 특정 비즈니스 기능에 집중
* **통신 방식**: 동기 vs 비동기, 상황에 맞게 선택

* **데이터 관리**: 서비스별 DB + 이벤트 기반 동기화
---

## Ⅱ. MSA 분해 전략

### 1. 비즈니스 능력 기반 분해
DDD의 **전략적 설계** 활동을 기반으로 서비스를 식별합니다.

```
주문(Order) 도메인 → 주문 컨텍스트 → 주문 서비스
       ↓
   ┌───────────┐    ┌─────────────┐
   │  주문      │    │   재고   │
   │            │    │   배송   │
   │            │    │   결제   │
   └────────────┘    └─────────────┘
       ↓
     배송 서비스 ←──→ 재고 서비스 ←──→ 결제 서비스
```
### 2. 하위 도메인 기반 분해
DDD의 **전술적 설계** 활동을 통해 각 하위 도메인에서 비즈니스 규칙, 불변식, 용어를 추출합니다:
* **장점**: DDD와 높은 정합도, 유지보수 향상
* **한계**: 초기 설계 복잡도 증가, 경계 파악 어려움

### 3. 분산 모놀리스 (안티패턴) ⬇️ **독립 배포 불가**
* **특징**: 겉리게 않, MSA처럼 비즈니스 로직이 분산됨
* **발생 원인**: 경계 파악 실패 시 팀 간 의존성 즡
* **안티패턴**: 분산 모놀리스 (마이크로서비스들이 동일한 기술 스택을 사용하는 것처럼, 독립 배포의 장점이 사라집니다만 개별 팀이 각자 다른 기술 스택을 사용할 수 있습니다

```

## Ⅲ. 서비스 간 통신

### 1. 동기 통신 (REST API, gRPC)
* **REST API**: HTTP 기반, 리소스 중심적 설계
* **gRPC**: Protocol Buffers 기반, 고성능/저지연, 통신
* **장점**: HTTP 친숭한 범용, 디버깅 욒이
* **한계**: HTTP 오버헤드, gRPC 학습 곡요 높음

### 2. 비동기 통신(메시지 큐)
* **RabbitMQ**: AMQP 기반, 브로커 중심
* **Kafka**: 고성능 분산 스트리밍, 높은 처리량
* **장점**: 느슐린 결합, 높은 확장성
* **한계**: 메시지 순서 보장 복잡성 증가
### 3. API 게이트웨이 역할
* **인증/인가 통합** 인증/인가 처리)
* **라우팅**: 요청을 적절한 서비스로 분산
* **통합(Aggregation)**: 여러 서비스 응답을 단일 응답으로 결합
* **장점**: 클라이언트 단순화, 마이크로서비스 간 결합 감소
* **한계**: 게이트웨이가 단일 장애점(SPOF)
이 될 수 있습니다
   부하 분산 가능성

### 4. 분산 모놀리스 ⬇️ **독립 배포 불가**
* **특징**: 겉리 격차되 모놀리스 수 있는 서비스를 단일 코드베이스로 수정
* **발생 원인**: 공통 코드 변경이 여러 서비스에 영향
* **안티패턴**: 분산 모놀리스 - 모든 서비스가 동일한 기술 스택을 사용해야 함

## Ⅳ. 이벤트 기반 아키텍처 (EDA)
### 1. 핵심 구성
* **이벤트 생산자(Event Producer)**
* **이벤트 브로커(Event Broker)**
* **이벤트 소비자(Event Consumer)**

### 2. 이벤트 버스
이벤트를 발행/구독하는 **이벤트 버스**(Event Bus)를 통해 전파합니다.

* **대표 도구**: Kafka, RabbitMQ, AWS EventBridge
* **장점**: 느슰한 결합, 높은 확장성
* **한계**: 이벤트 순서 보장, 디버깅 어려움 수 있음

### 3. 이벤트 소싱(Event Sourcing)
상태 변경 이력을 **이벤트 스트림**(Append-only)**으로 저장합니다.
* **장점**: 완전한 감사,(Audit trail), 제공
* **한계**: 저장 공간 증가, 재생 성능 이슘(이벤트 재생) 필요
### 4. CQRS와의 연계
이벤트 소싱을 사용하여 **결정적 일관성(Eventual Consistency)**을 달성 수 있습니다를 읽기 모델로 동기화합니다.

* **장점**: 읽기/쓰기 분리 최적화
* **한계**: 결과적 일관성(Eventual Consistency)으로 즉시 조회되지 않을 수 있음
```

## Ⅴ. 분산 트랜잭션 관리
### 1. 2PC (Two-Phase Commit)
* **1단계(Prepare)**: 코디네이터가 준비 상태
* **2단계(Commit)**: 모든 참여자가 트랜잭션 커밋
* **문제점**: 1단계 실패 시 전체 롽아뱽 2단계도 실패
* **MSA 한계**: 서비스 간 강결합 필요
* **성능**: 2PC는 동기화 오버헤드가 발생
* **적용 사례**: 금융 거래, 주문/결제 시 2PC 필요

  (재고 → 주문 → 결제)

* **대안**: 사가 패턴 (로컬 트랜잭션들의 연속된 체인)
```

## Ⅵ. 사가 패턴 (분산 트랜잭션)
### 1. 오케스트레이션 사가 (Orchestration Saga)
* **중앙 통제기**(Orchestrator)가 전체 흐름을 관리
* **장점**: 명확한 흐름 제어, 실패 시 롤백 로직 구현 욀이
* **한계**: 오케스트레이터가 단일 실패 지점(SPOF)이 됨
* **코레오그래피(Choreography Saga)**
* 이벤트 기반 분산 조정
* 각 서비스이 자율적으로 이벤트를 발행/구독
* **장점**: 단일 실패 지점이 격리, 중앙 통제기 의존성 제거
* **한계**: 전체 흐름 파악이 어렵고, 이벤트 순서 보장 복잡성 증가
### 2. 보상 트랜잭션(Compensating Transaction)
* **개념**: 정상 트랜잭션을 롤백하는 **역방향 연산**
* **용도**: 사가의 실패 시 이전 상태로 복원
* **예시**: 주문 취소 → 재고 차감기 → 결제 취소 + 이메일 발송
```
**핵심 설계 원칙**: 모든 보상 트랜잭션은 **멱등(idempotent)**하게 설계해야 합니다
```

## Ⅵ. 서비스 디스커버리(Service Discovery)
### 1. 문제
MSA 환경에서 서비스의 IP/Port가 동적으로 변경됩니다.
* **해결 방안**: 서비스 레지스트리(Service Registry)를 통해 동적으로 발격니다

### 2. 패턴 비교
| **클라이언트 사이드** | **서버 사이드** |
|---| ---|--- |-------------------|----------------|--------------------------|
| **질문** | Service Registry | **Service Registry** |
| **결과** | IP 목록 반환 | 로드밸런싱하여 라우팅 |
| **장점** | 언어/플랫폼 무관 | 구현 단순 | **장점** - 캐싱 가능 | 고가용성 | **한점** - 레지스트리와의 추가 통신 필요
| **단점** - 구현 복잡도 증
 | **장점** - 클라이언트 구성이 단순함 |
| **단점** - 추가 인프라 구성 요소 필요 |
| **도구** | Eureka(Spring Cloud), Consul(HashiCorp) |
| **선택 기준**: 환경에 따라 적절한 선택

```

## Ⅶ. API 게이트웨이 (API Gateway)
### 1. 핵심 기능
* **인증/인가(Authentication/Authorization)
* **라우팅(Routing)**: 요청을 적절한 서비스로 전달
* **로드 밸런싱(Load Balancing)**: 트래픽을 여러 인스턴스에 분산
* **속도 제한(Rate Limiting)**: 과도한 트래픽으로부터 DDoS 방어
* **통합(Aggregation)**: 여러 서비스 호출 결과를 단일 응답으로 결합
* **API 버전 관리**: 여러 버전의 API를 단일 엔드포인트로 관리

* **로깅(Logging)**: 요청/응답 로그를 중앙 로그 시스템으로 전송
* **캐싱(Caching)**: 자주 사용 빈번한 응답 캐싱

* **변환(Transformation)**: 요청/응답 형식 변환 (예: XML to JSON)

* **모니터링(Monitoring)**: 성능 메트릭 수집 및 분석

* **Circuit Breaker**: 장애 전파 방지을 위해 Circuit Breaker 패턴 적용
### 2. 장점
* **단일 진입점**(Single Entry Point): 클라이언트가 단순해집
* **공통 기능 중앙화**: 인증, 로깅, 모니터링 등을 한 곳에서 구현
* **보안 강화**: 게이트웨이에서 인증/인가를 수행하고 공격 표면 감소
* **성능 최적화**: 캐싱, 로드밸런싱, 변환 등을 통해 게이트웨이 수행
* **관심사 분리**: 비즈니스 로직은 프론트엔드, 인증/인가에서 분리됨

### 3. 대표 도구
* **Kong**: Nginx 기반, 고성능, 풌러그인 지원
* **Spring Cloud Gateway**: Spring 생태계, 통합 Java
* **AWS API Gateway**: AWS 관리형 서비스
* **Azure API Management**: Azure 통합 API 관리 플랫폼
* **Envoy**: CNCF 프로젝, Istio의 데이터 플레인으로 구성 가능
```

## Ⅷ. BFF (Backend for Frontend)
### 1. 개념
클라이언트별로 전용 API 게이트웨이를 두는 것입니다. 각 BFF는 특정 클라이언트 유형에 최적화된 응답을 반환합니다.
### 2. 장점
* **최적화된 페이로드**: 모바일과 데스크탑, 필요한 데이터만 전송
* **간소한화된 응답**: 웹용은 HTML, 모바일은 JSON) - 필요한 데이터만 반환
* **독립 확장**: 각 BFF를 독립적으로 확장/축가할 수 있음
* **단순화된 클라이언트 코드**: 공통 인증/인가 로직을 한 곳에 모아도 필요 없음
### 3. 구현 예시
```
[BFF 구조]
┌───────────────────────┐
│    Web Client      │    Mobile Client   │
│      (React)          │      (React Native)    │
└────────┬───────┬───┘─────────┘
           │                  │
           ▼                  ▼
     ┌──────────────────┐──────────────────┐
     │   Web BFF       │    Mobile BFF     │
     │ (GraphQL/REST) │   (GraphQL/REST)  │
     └────────┬───────┘──────────────┘
              │                  │
              ▼                  ▼
        ┌───────────────────────────┐
        │    Business Services      │
        │  (User, Order, Payment...)   │
        └───────────────────────────┘
```
### 4. 도구
* **Apollo Federation**: GraphQL 기반 BFF 프레임워크
* **Next.js API Routes**: Next.js의 API 라우트 기능
```

## Ⅸ. 외부화된 구성 관리(Externalized Configuration)
### 1. 개념
설정 정보를 코드에서 분리하여 외부(환경 변수, Config Server)에서 관리합니다.
### 2. 구현 방식
* **Config Server**: 중앙 집중형 구성 서버 (Spring Cloud Config)
* **환경 변수**: 컨테이너/오케스트레이션 전달

### 3. 장점
* **환경 독립성**: 개발/테스트/운영 환경 분리
* **동적 갱신**: 재배포 없이 설정 변경 가능
* **보안 강화**: 민감 정보 코드에서 분리
* **12-Factor App 준수**: 구성을 환경 변수로 외부화

## Ⅹ. 서비스 메시(Service Mesh)와 사이드카(Sidecar)
### 1. 서비스 메시란 의
마이크로서비스 간 통신을 인프라 계층에서 처리하는 **인프라스트** 계층입니다.
* **구성**: 데이터 평면(Data Plane), + 사이드카 프록시(Sidecar Proxy)
* **주요 구현**: Istio, Linkerd, Consul Connect
### 2. 사이드카 패턴
```
┌────────────┐       ┌────────────┐       ┌────────────┐
│   Service A │◄─────►│   Sidecar  │◄─────►│   Service B │
│  (Pod A)   │       │   Proxy    │       │  (Pod B)   │
└────────────┘       └────────────┘       └────────────┘
        │                   │                   │
        │    Mesh Layer    │
        │  (Istio/Linkerd) │
        └───────────────────┘
```
### 3. 핵심 기능
* **트래픽 관리**: 라우팅, 로드 밸런싱, 카나리 배포
* **보안**: mTLS 상호 인증, 서비스 간 통신 암호화
* **관찰 가능성(Observability)**: 메트릭, 로그, 분산 추적 수집
* **회복력(Resilience)**: 서킷 브레이커, 타임아웃/재시도 적용
```

## Ⅺ. 트래픽 라우팅, 카나리 배포(Canary Deployment)
### 1. 개념
소수 사용자에게 신규 버전을 노출하여 안정성을 검증하고 점진적으로 전체 사용자로 전환합니다.
### 2. 구현 방식
```
┌───────────────────────┐       ┌───────────────────────┐
│   Load Balancer        │       │   Load Balancer         │
│ (V1: 10% 트래픽)  │       │ (V2: 1% 트래픽)  │
└───────────┬───────────┘       └───────────┬───────────┘
        │                       │                       │
        ▼                       ▼                       ▼
   ┌───────────────────────┐       ┌───────────────────────┐
   │   Service V1          │       │   Service V2          │
   │ (90% 트래픽)         │       │ (10% 트래픽)         │
   └───────────────────────┘       └───────────────────────┘
```
### 3. 장점
* **위험 최소화**: 문제가 있는 경우 소수 사용자만 영향을 받음
* **빠르른 롤백**: 문제 발견 시 빠르게 롤백
* **실제 테스트**: 운영 트래픽으로 실제 환경 테스트 가능
### 4. 한계
* **데이터 일관성**: v1과 v2 간 데이터 동기화 필요 (양방향 복제)
* **복잡성 증가**: 트래픽 복제 및 라우팅 규칙 추가 필요
```

## Ⅻ. 블루-그린 배포(Blue-Green Deployment)
### 1. 개념
두 개 동일한 환경(Blue/Green)을 운영하며 전환하는 무중단 배포 방식입니다.
### 2. 흐름
```
┌─────────────┐       ┌─────────────┐
│   Blue      │       │   Green      │
│ (Active)    │       │ (Standby)   │
└─────────────┘       └─────────────┘
        │
        │ Switch
        ▼
┌─────────────┐
│   Load Balancer  │
│ (Switch when)  │
│   ready)        │
└─────────────┘
```
### 3. 장점
* **무중단 배포**(Zero-downtime): 전환 시 서비스 중단 없음
* **빠른른 롤백**(Instant Rollback): 문제 발생 시 즉시 이전 버전으로 전환 가능
* **안전한 환경**: Green 환경에서 완전히 테스트 후 프로덕션으로 전환 가능
### 4. 한계
* **리소스 요구**: 두 배 환경 유지에 두 배의 리소스 필요
* **전환 시간**: 트래픽 전환 시 약간의 지연(latency spike)이 발생할 수 있음
```

## Ⅺ. 롤링 업데이트(Rolling Update)
### 1. 개념
순차적으로 인스턴스를 교체하며 서비스 중단을 최소화하는 배포 방식입니다.
### 2. 흐름
```
┌─────────────┐       ┌─────────────┐       ┌─────────────┐
│   Instance 1│ ──► │   Instance 2│ ──► │   Instance 3│
│  (v1)      │       │  (v2)      │       │  (v3)      │
└─────────────┘       └─────────────┘       └─────────────┘

              │
              ▼
        [New Instance]

### 3. 장점
* **무중단 배포**: 서비스 중단 시간 최소화
* **점진적 확장**: 트래픽 증가에 따라 점진적 확장
* **리소스 효율**: 신규 인스턴스만 추가하면, 기존 리소스를 재사할 수 있음
### 4. 한계
* **복잡성**: 롤링 로직이 복잡할 수 있음
* **버전 호환성**: 인스턴스 간 버전 호환성 검증 필요

## Ⅻ. 섀도우 배포(Shadow Deployment)
### 1. 개념
프로덕션 트래픽을 실시간으로 복제하여 **섀도우(shadow)** 환경에서 테스트하는 기법입니다.
### 2. 특징
* **미러링 트래픽**: 실제 요청을 섀도우 서비스로 복사하여 전송
* **실제 환경 테스트**: 운영 트래픽, 사용하여 새 로직을 검증
* **무중단**: 섀도우 서비스가 문제가 있어도 실서비스로 롤백
 영향만 받음
### 3. 장점
* **안전한 테스트**: 실제 트래픽으로 철저한 테스트 가능
* **문제 조기 발겶**: 섀도우 환경에서 문제가 득러워도 배포 전에 발경 가능
* **성능 비교**: 새/구 버전 성능 비교 욄 수 있음
### 4. 한계
* **리소스 소요**: 트래픽을 복제하여 두 배의 리소스가 필요
* **복잡성**: 구성이 복잡할 수 있음(특히 트래픽 미러링 도구)

## Ⅹ. 분산 로그 수집(Distributed Logging)
### 1. 도전
MSA에서 각 서비스가 자체 로그를 생성합니다.
* **로그 분산**: 여러 서버에 로그가 분산되어 있습니다
* **형식 불일**: 서비스마다 로그 형식이 다를 수 있음
* **분석 어려웤**: 개별 로그만으로는 전체 흐름 파악이 어려움
### 2. 아키텍처
```
┌───────────────┐        ┌───────────────┐        ┌───────────────┐
│   Service A   │        │   Service B   │        │   Service C   │
│               │        │               │        │               │
│   ┌─────┐        │   └─────┐        │   └─────┐        │
│   Log       │        │   Log          │        │   Log          │
│   Agent     │        │   Agent       │        │   Agent       │
└───────────┘        └───────────────┘        └───────────────┘
        │                        │                        │
        │    Log Aggregation     │
        │   (ELK/EFK Stack)    │
        └─────────────────────┘
```
### 3. 핵심 구성
* **로그 수집기(Log Agent/Shipper)**: 각 서비스에 설치
* **중앙 저장**: ELK(Elasticsearch, Logstash, Kibana), EFK(Elastic Fluentd, Kibana)
* **분석 도구**: Kibana 대시보드, Elasticsearch 쿼리
* **알림**: 로그 패턴 감지 시 알림
### 4. 로그 협회 표준화
* **JSON**: 구조화된 로그 포맷
* **Common Log Format**: 표준화된 형식(예: Logback, JSON)
* **Correlation ID**: 요청 간 연관성을 추적하기 위한 ID 사용
* **MDC(Mapped Diagnostic Context)**: 분산 추적을 위한 ID
* **Trace ID**: 전체 트랜잭션의 고유 ID
* **Span ID**: 개별 서비스 내부 작업의 ID
```

## Ⅹ. 분산 추적(Distributed Tracing)
### 1. 문제
MSA에서 요청이 여러 서비스을 통과할 수 있습니다.
* **호출 체인**: 어디서, 무슨 호출이, 어느서 지연되 증가
* **디버깅**: 호출이 깊이 알수록 전파가 어렵음
* **모니터링**: 분산 시스템의 전체 흐름 파앝이 어려움
### 2. 해결 방안: 분산 추적(Distributed Tracing)
```
┌───────────────────┐        ┌───────────────────┐
│   Service A        │        │   Service B        │
│                   │        │                   │
│   ┌─────────┐        │   └─────────┐        │   └─────────┐
│   │ Span 1 │        │   │ Span 1 │        │   │ Span 2 │
│   │       │        │   │       │        │   │       │
│   └─────────┘        │   └─────────┘        │   └─────────┘
        │                       │                       │
        │    Trace Context     │
        │   (Trace ID + Span IDs)  │
        └───────────────────────────┘
```
### 3. 구현 도구
* **OpenTelemetry**: 벤더 표준, CNCF 프로젝
* **Jaeger**: Uber 개발, 분산 추적 시스템
* **Zipkin**: Twitter 분산 추적 시스템
* **AWS X-Ray**: AWS 분산 추적 서비스
```

## Ⅹ. 컨텍스트 전파(Context Propagation)
### 1. 개념
분산 추적을 위해 **Trace ID와 Span ID**를 요청 간 서비스 간 전달하는 메커니즘입니다.
### 2. 전파 방식
* **HTTP Header**: `X-Trace-ID`, `X-Span-ID` 헤더
* **메시지 헤더**: 메시지 메타데이터에 추적 정보 포함
* **gRPC Metadata**: gRPC 인터셉트 Metadata 전파
```
┌───────────────────────────────────────────────────────┐
│                   Client Request                      │
│  X-Trace-ID: abc123                              │
│  X-Span-ID: span1                                │
└───────────────────┬───────────────────────────────┘
                    │
                    ▼
          ┌───────────────────────────────────────┐
          │            Service A                  │
          │  Span ID: span1 (Parent: abc123)    │
          │  Span ID: span2 (Parent: abc123)    │
          └───────────────────┬───────────────────┘
                              │
                              ▼
                    ┌───────────────────┐
                    │   Service B       │
                    │ Span ID: span3     │
                    │ (Parent: abc123) │
                    └───────────────────┘
```
### 3. 구현 예시
```python
# HTTP Header 전파
headers = {
    'X-Trace-ID': 'abc123',
    'X-Span-ID': 'span1'
}

# 다음 서비스 호출 시
response = requests.get('http://service-b/api', headers=headers)
new_span_id = 'span2'
response.headers['X-Span-ID'] = new_span_id
```

## Ⅹ. 탄력성(Resiliency) 패턴
### 1. 회복력이 필요한 이유
* **네트워크 장애**: 서비스 간 통신 실패

* **Retry**: 일시적 장애 시 재시도
* **Circuit Breaker**: 지속적 장애 시 차단
* **Timeout**: 응답 대기 시간 제한
* **Fallback**: 대체 기능 제공
* **Bulkhead**: 리소스 격리

## Ⅹ. 서킷 브레이커(Circuit Breaker)
### 1. 상태 머신
```
    ┌─────────┐      (실패 임계값 초과)      ┌─────────┐
    │ Closed  │ ─────────────────────────────▶│  Open   │
    │ (정상)  │                                │ (차단)  │
    └─────────┘                                └─────────┘

CircuitBreaker circuitBreaker = CircuitBreaker.of("serviceA", custom);

## Ⅹ. 타임아웃 및 재시도(Timeout & Retry)
### 1. 타임아웃 전략
* **연결 타임아웃**: 연결 설정 타임아웃
* **읽기 타임아웃**: 읽기 작업 타임아웃
* **쓰기 타임아웃**: 쓰기 작업 타임아웃
### 2. 재시도 전략
* **고정 간격(Fixed Interval)**: 일정한 대기 시간
* **지수 백오프(Exponential Backoff)**: 대기 시간이 지수적으로 증가
* **지터 백오프(Jitter)**: 무작위성 추가한 지수 백오프
```python
# 지수 백오프 예시
def exponential_backoff(attempt):
    base_delay = 100  # 100ms
    max_delay = 10000  # 10s
    delay = min(base_delay * (2 ** attempt), max_delay)
    time.sleep(delay / 1000)
```

## Ⅹ. 벌크헤드(Bulkhead)
### 1. 개념
선박의 격리(Bulkhead) 패턴은 **격벽실**에서 물이 차오르 때 침수을 막기 위해, 시스템을 격리된 구역으로 나누는 것입니다.
* **스레드 풀 격리**: 각 서비스별 전용 스레드 풀
* **프로세스 격리**: 각 서비스별 독립 프로세스
* **컨테이너 격리**: 각 서비스별 독립 컨테이너
### 2. 효과
```
[격리 없는 경우]
┌───────────────────────────────────────────────────┐
│              Shared Thread Pool                   │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐   │
│  │ Service A │  │ Service B │  │ Service C │   │
│  │ (느림)   │  │ (정상)    │  │ (대기)     │   │
└──┴───────────┴──┴───────────┴──┴───────────┴──┘
│  Service D (대기) - 리소스 부족으로 차단됨            │
└───────────────────────────────────────────────────┘

[격리된 경우]
┌───────────────┐  ┌───────────────┐  ┌───────────────┐
│   Thread Pool   │  │   Thread Pool   │  │   Thread Pool   │
│   (Service A)  │  │   (Service B)  │  │   (Service C)  │
│  ┌───────────┐ │  ┌───────────┐ │  ┌───────────┐  │
│  │ Service A │ │  │ Service B │ │  │ Service C │  │
│  │ (느림)   │ │  │ (정상)    │ │  │ (대기)     │  │
└──┴───────────┘ └──┴───────────┘ └──┴───────────┘  │
│  Service D (정상 - 영향 없음)                          │
└───────────────────────────────────────────────────┘

## Ⅹ. 서버 사이드 렌더링(SSR)
### 1. CSR vs SSR
| **CSR (Client-Side Rendering)** | **SSR (Server-Side Rendering)** |
|---|---|
| 브라우저에서 JavaScript로 렌더링 | 서버에서 HTML 렌더링 |
| 초기 로딩 빠름 | 초기 로딩은 빠름, |
| SEO 불리함 | SEO 친화화 (검색 엔진 최적화) |
| SPA(Single Page Application) 적합 | 콘텐츠 사이트, 블로그에 적합 |

### 2. SSR 프레임워크
* **Next.js**: React 기반 SSR 프레임워크
* **Nuxt.js**: Vue.js 기반 SSR 프레임워크
* **SvelteKit**: Svelte 기반 SSR 프레임워크
* **Remix**: React 기반 SSR (Cloudflare)
### 3. 하이드레이션(Hydration)
SSR로 렌더링된 HTML이 클라이언트에서 JavaScript를 실행 가능하게 하는 과정입니다.
```
SSR HTML → JavaScript 번환 → 이벤트 리스너 → 상호작용 가능
```

📢 **섹션 요약 비유**: 클라우드 네이티브 아키텍처는 **도시 계획**과 같습니다. **서비스 분해**는 구획을 쪼개하는 것(DDD 기반 vs 비즈니스 능력 기반). **통신**은 동기(REST)와 비동기(메시지 큐)를 상황에 맞게 선택합니다. **데이터**은 서비스별 DB + 이벤트 동기화를 통해 일관성을 유지합니다. **신뢰성**은 Circuit Breaker, Retry, Bulkhead 패턴으로 확보합니다. **배포**는 Canary, Blue-Green, Rolling, Shadow 전략을 통해 안전하게 릴리즈합니다. **가시성**은 로그, 메트릭, 분산 추적으로 시스템 상태를 파악합니다.
