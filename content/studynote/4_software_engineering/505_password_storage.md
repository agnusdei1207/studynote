# 505. 비밀번호 저장 방식 - KDF(PBKDF2, bcrypt, scrypt, Argon2) 및 Salt

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비밀번호 저장 보안의 핵심은 평문을 직접 저장하지 않고, **솔트(Salt)를 추가한 후 KDF (키 유도 함수)를 통해 수천 번 이상의 반복 연산(Stretching)**을 수행하여 공격자의 무차별 대입 및 레인보우 테이블 공격을 기술적으로 지연시키는 데 있다.
> 2. **가치**: GPU나 ASIC을 이용한 고속 병렬 연산 해독 위협에 대응하기 위해 단순히 계산량(Time)뿐만 아니라 메모리 사용량(Memory Hardness)까지 강제하는 알고리즘을 선택함으로써 데이터 유출 시에도 계정 안전성을 보장한다.
> 3. **융합**: 현대 아키텍처에서는 PHC (Password Hashing Competition) 우승작인 **Argon2**가 최상위 표준으로 권장되며, 이는 시스템의 가용 자원에 맞춰 보안 강도를 동적으로 조절할 수 있는 유연성을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 비밀번호 저장 방식은 "어떻게 하면 유출되어도 읽을 수 없게 만들 것인가"를 다룬다. 단순 해시(SHA-256 등)의 취약점을 보완하기 위해 사용자별 고유 값인 **솔트(Salt)**를 섞고, **KDF**를 통해 해시 연산을 수만 번 반복(Stretching)하여 공격자의 해독 비용을 물리적으로 높인다.

- **💡 비유**: 비밀번호 저장은 **"음식 레시피를 암호화하는 것"**과 같습니다. 단순히 재료를 섞는 것(Hash)이 아니라, 사람마다 다른 비밀 소스(Salt)를 넣고 수천 번 정성껏 저어서(Stretching) 원래 재료가 무엇이었는지 아무도 흉내 내지 못하게 만드는 과정입니다.

- **등장 배경**:
  1. **레인보우 테이블(Rainbow Table) 공격**: 미리 계산된 해시 값 테이블을 이용해 1초 만에 비밀번호를 복구하는 공격이 성행하면서 솔트(Salt) 도입이 필수화되었다.
  2. **하드웨어의 비약적 발전**: GPU나 특수 칩(ASIC)을 사용하면 일반 CPU보다 수만 배 빠르게 해시를 계산할 수 있게 되어, 단순 해시는 더 이상 안전하지 않게 되었다.
  3. **보안 가이드라인의 강화**: NIST SP 800-63 및 KISA 가이드에서 비밀번호 저장 시 반드시 검증된 KDF 사용을 권고하기 시작했다.

- **📢 섹션 요약 비유**: 도둑이 열쇠를 하나하나 넣어보는 속도를 늦추기 위해, 열쇠 구멍을 아주 복잡한 미로(KDF)로 만들어버리는 방어 전략과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 비밀번호 보안 강화 기술의 진화

| 기술 명칭 | 핵심 원리 | 해결하고자 하는 문제 | 비유 |
|:---|:---|:---|:---|
| **Hash** | 일방향 함수 연산 | 평문 노출 방지 | 과일 주스 만들기 |
| **Salt** | 사용자별 랜덤 문자열 추가 | **레인보우 테이블 무력화** | 같은 주스에 다른 향신료 첨가 |
| **Stretching** | 해시 연산 반복 실행 | **무차별 대입(Brute-force) 지연** | 주스를 수천 번 젓기 |
| **Memory Hard** | 대량의 메모리 점유 강제 | **GPU/ASIC 고속 연산 차단** | 젓는 동안 큰 대야(메모리) 사용 강제 |

---

### 안전한 비밀번호 저장 아키텍처 (Argon2 기준)

사용자 입력이 저장소에 도달하기까지의 보안 강화 흐름을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      KDF 기반 비밀번호 해싱 및 검증 프로세스                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [사용자 입력] : "password123!"                                               │
 │        │                                                                    │
 │        ▼ 1. 사용자별 고유 Salt 생성 (CSPRNG 활용)                              │
 │  ┌──────────────┐                                                           │
 │  │  Salt: abcde │                                                           │
 │  └──────┬───────┘                                                           │
 │        │                                                                    │
 │        ▼ 2. KDF 연산 (Stretching + Memory Hardness)                         │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │  Argon2 / bcrypt / PBKDF2                                             │  │
 │  │  - Iterations: 10,000+ times                                          │  │
 │  │  - Memory: 64MB+ usage                                                │  │
 │  │  - Parallelism: 4 threads                                             │  │
 │  └──────────────────┬────────────────────────────────────────────────────┘  │
 │                    │                                                        │
 │                    ▼ 3. 최종 저장 (Hash Result)                              │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ $argon2id$v=19$m=65536,t=3,p=4$abcde$xyz123...                        │  │
 │  │ (알고리즘 버전, 파라미터, 솔트, 결과값이 모두 포함된 표준 포맷)               │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 안전한 저장은 3가지 요소를 결합한다. 1) **솔트 (Salt)**는 동일한 비밀번호를 가진 사용자들이라도 DB상에 서로 다른 해시 값으로 저장되게 하여 대량 해독을 막는다. 2) **KDF 연산**은 단순히 CPU를 쓰는 단계를 넘어, **메모리(Memory)**를 대량으로 점유하게 설계된다. 이는 메모리가 부족한 GPU나 전용 하킹 장비가 병렬로 빠르게 연산하는 것을 물리적으로 방해한다. 3) 최종 결과물은 알고리즘의 설정값(파라미터)과 솔트가 포함된 **표준 문자열**로 저장되어, 나중에 서버 환경이 바뀌어도 검증이 가능하도록 설계된다.

---

### 심층 동작 원리: KDF 알고리즘별 특징 비교

현대 보안 아키텍처에서 선택할 수 있는 4대 알고리즘의 내부 특성이다.

1. **PBKDF2**: NIST 표준. 반복 횟수(Iteration)만 조절 가능. 구조가 단순하여 ASIC 공격에 취약할 수 있으나 호환성이 매우 높음.
2. **bcrypt**: 1999년 발표. 'Work Factor'를 통해 연산량을 조절. 현재 가장 널리 쓰이는 표준이나, 메모리 하드니스 기능은 약함.
3. **scrypt**: bcrypt의 단점을 보완하여 메모리 사용량을 대폭 늘림. ASIC 공격에 매우 강하지만 파라미터 설정이 까다로움.
4. **Argon2**: 2015년 PHC 우승작. 시간, 메모리, 병렬성을 모두 제어 가능. **현존하는 가장 강력한 비밀번호 해싱 알고리즘.**

- **📢 섹션 요약 비유**: PBKDF2가 단순히 높은 계단을 오르는 것이라면, Argon2는 무거운 짐(메모리)을 지고 여러 명이 동시에 보폭을 맞춰서(병렬성) 험난한 산길을 넘는 것과 같습니다.

### 알고리즘 선택 의사결정 매트릭스

| 선택 기준 | PBKDF2 | bcrypt | scrypt | Argon2 (Recommended) |
|:---|:---:|:---:|:---:|:---:|
| **보안 강도** | 보통 | 높음 | 매우 높음 | **최상** |
| **GPU 공격 저항력** | 낮음 | 보통 | 높음 | **매우 높음** |
| **표준 준수 여부** | NIST 표준 | 업계 표준 | 암호화폐 활용 | **PHC 우승작** |
| **구현 난이도** | 쉬움 | 쉬움 | 보통 | 보통 |
| **적합한 상황** | 레거시 호환성 중시 | 일반 웹 서비스 | 고도의 보안 요구 | **최신 신규 프로젝트** |

---

### 융합 시너지: Pepper와 Pepper 관리

솔트(Salt)는 DB에 저장되지만, **페퍼 (Pepper)**는 소스 코드나 별도의 보안 서버(KMS)에 보관하는 비밀 값이다.
- **동작**: `Hash(Password + Salt + Pepper)`
- **효과**: DB가 통째로 유출되어도 소스 코드(Pepper)를 모르면 공격자는 해독을 시작조차 할 수 없다.

- **📢 섹션 요약 비유**: 보물 상자 옆에 열쇠(Salt)를 두는 것이 불안해서, 열쇠의 절반(Pepper)은 집주인이 직접 들고 다니는 이중 장치와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 커머스 플랫폼의 비밀번호 보안 고도화

- **문제 상황**: 현재 SHA-256 단일 해시로 비밀번호를 저장 중이며, 보안 진단 결과 GPU 기반 해독 공격에 1시간 내로 뚫릴 위험성 경고 받음.
- **의사결정 (마이그레이션 전략)**:
  1. **알고리즘 선정**: 신규 가입 및 비밀번호 변경 시 **Argon2id** (Argon2i와 Argon2d의 장점을 결합) 적용을 원칙으로 한다.
  2. **파라미터 최적화**: 서버 사양을 고려하여 해싱 연산에 약 0.5초가 소요되도록 메모리(64MB)와 반복 횟수를 조정한다. (사용자는 0.5초 지연을 거의 못 느끼지만 공격자는 1,000배 이상 느려짐)
  3. **점진적 전환 (Lazy Migration)**: 기존 사용자가 로그인할 때 평문 비밀번호를 가로채서 즉시 Argon2로 다시 해싱하여 DB를 업데이트한다. (일괄 변환은 평문이 없으므로 불가능)
  4. **Pepper 도입**: 전사 공통 Pepper를 HSM(하드웨어 보안 모듈)에서 관리하여 DB 유출 사고에 대비한다.

### 도입 체크리스트
- **CSPRNG**: 솔트 생성 시 예측 가능한 `Random()` 함수가 아닌 안전한 `SecureRandom`을 쓰고 있는가?
- **Work Factor**: 하드웨어 성능 발전에 따라 주기적으로 Iteration 횟수를 상향 조정하고 있는가?
- **Unique Salt**: 모든 사용자에게 동일한 솔트를 쓰는 것이 아니라, ID마다 고유한 솔트를 부여하는가?

### 안티패턴
- **단순 해시 맹신**: SHA-256이나 SHA-512만 쓰면 안전하다고 생각하는 행위. (Stretching 없이는 무의미함)
- **클라이언트 측 해싱**: 브라우저에서 해시를 해서 보내면, 그 해시값 자체가 비밀번호 역할을 하게 되어 '해시 가로채기(Pass-the-hash)' 공격에 그대로 노출된다. 해싱은 반드시 **서버 측**에서 수행해야 한다.

- **📢 섹션 요약 비유**: "우리 집 자물쇠는 신제품이니까 괜찮아"라고 자만하기보다, 주기적으로 자물쇠 비밀번호를 바꾸고 최신 보안 트렌드를 살피는 부지런함이 가장 안전한 방패입니다.

### 미래 전망
- **Quantum-Resistant KDF**: 양자 컴퓨터의 연산 능력에 대응하기 위해 해시 길이를 2배 이상 늘리고(SHA-512 등), 격자 기반의 새로운 유도 함수가 연구될 것이다.
- **Hardware-Enforced KDF**: 모바일 기기의 Secure Enclave나 서버의 TEE(Trusted Execution Environment) 내부에서만 해싱이 일어나게 하여 메모리 덤프 공격까지 차단하는 기술이 보편화될 것이다.

### 참고 표준
- **NIST SP 800-132**: Recommendation for Password-Based Key Derivation
- **RFC 9106**: Argon2 Password Hashing Function
- **OWASP Password Storage Cheat Sheet**: 실무 표준 가이드

- **📢 섹션 요약 비유**: 수동으로 맷돌을 돌려 가루를 만들던 시대(단순 해시)에서, 정밀한 화학 공정(Argon2)을 통해 누구도 재현할 수 없는 고유의 물질을 만들어내는 고차원 보안 시대로 진입했습니다.

1. 비밀번호를 그대로 적어두면 도둑이 금방 읽을 수 있으니까, **복잡한 기계에 넣어서 알아볼 수 없는 낙서(해시)로 바꿔서 저장하는 것**과 같아요.
2. 이때 사람마다 다른 "비밀 소금(Salt)"을 팍팍 뿌리고 수만 번 휘저어주면, 도둑이 아무리 똑같이 흉내 내려고 해도 절대 따라 할 수 없게 된답니다.
3. 이렇게 정성 들여 섞어놓은 비밀번호는 설령 도둑이 훔쳐 가도 무엇인지 알아낼 수 없어 우리 계정을 안전하게 지켜줘요!
