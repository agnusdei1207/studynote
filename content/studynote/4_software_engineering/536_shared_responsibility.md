# 536. 클라우드 보안 책임 공유 모델 (Shared Responsibility Model)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클라우드 보안 책임 공유 모델은 클라우드 서비스 제공자(CSP)와 고객(Customer) 사이의 **보안 통제 책임을 명확히 구분하여 정의한 보안 거버넌스 체계**다.
> 2. **가치**: 고객이 "인프라 전체를 CSP가 다 지켜주겠지"라고 오해하여 발생하는 보안 공백을 방지하며, 서비스 모델(IaaS, PaaS, SaaS)에 따라 변하는 책임의 경계를 이해함으로써 안전한 클라우드 운영 환경을 구축하게 한다.
> 3. **융합**: 클라우드 컴플라이언스(ISMS-P, SOC2) 인증의 근간이며, 클라우드 네이티브 보안 도구(CSPM, CWPP) 선정 및 운영 정책 수립의 기준점이 된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 책임 공유 모델은 "클라우드 **의(of)** 보안"은 CSP가, "클라우드 **내(in)** 보안"은 고객이 책임진다는 원칙이다. CSP는 물리적 센터와 하이퍼바이저를 보호하고, 고객은 그 위에서 돌아가는 데이터와 애플리케이션, 설정값을 책임진다.

- **💡 비유**: 책임 공유 모델은 **"아파트 임대 계약"**과 같습니다. 아파트 건설사(CSP)는 건물의 뼈대, 엘리베이터, 공동 현관문(인프라)의 안전을 책임집니다. 하지만 내 집 안의 현관문을 잠그고(암호 설정), 귀중품을 금고에 보관하며(데이터 암호화), 낯선 사람을 들여보내지 않는 것(접근 제어)은 살고 있는 세입자(고객)의 책임인 것과 같습니다.

- **등장 배경**:
  1. **클라우드 확산에 따른 혼선**: 사용자들이 클라우드를 쓰면서 "모든 것이 자동 보안"이라고 착각하여 S3 버킷을 공개 설정하는 등 관리 소홀로 인한 사고가 급증했다.
  2. **복잡한 기술 스택**: 클라우드 환경은 하드웨어부터 소프트웨어까지 계층이 다양하여, 사고 발생 시 책임 소재를 가리기 위한 법적·기술적 기준이 필요해졌다.
  3. **효율적 자원 배분**: CSP가 모든 데이터 내용까지 검열할 수 없으므로(개인정보 보호), 영역을 나누어 각자 가장 잘할 수 있는 보안에 집중하도록 설계되었다.

- **📢 섹션 요약 비유**: 사고가 났을 때 "네 탓, 내 탓" 하며 싸우지 않도록, 미리 보안의 담장을 정확히 쳐두는 합리적인 보안 계약서입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 클라우드 서비스 모델별 책임 범위 변화

서비스 유형에 따라 고객이 챙겨야 할 보안 영역의 깊이가 달라진다.

| 영역 (Layer) | **IaaS** (EC2 등) | **PaaS** (RDS, Beanstalk) | **SaaS** (M365, Slack) | 비유 |
|:---|:---:|:---:|:---:|:---|
| **데이터 및 정보** | **고객** | **고객** | **고객** | 보물 상자 내용물 |
| **애플리케이션** | **고객** | **고객** / CSP | CSP | 방 안의 가구 |
| **운영체제 (OS)** | **고객** | CSP | CSP | 방의 도배/장판 |
| **네트워크 설정** | **고객** (Security Group) | CSP / **고객** | CSP | 방문 잠금 |
| **물리 인프라** | CSP | CSP | CSP | 아파트 외벽/지반 |

---

### 책임 공유 모델 상세 아키텍처 (AWS/Azure 표준 모델)

하부 인프라와 상부 자원의 책임 경계를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      클라우드 보안 책임 공유 모델 (Conceptual)                   │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │  [고객 책임: 클라우드 '내'의 보안] (Security **IN** the Cloud)                   │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ 1. 고객 데이터 (암호화, 무결성 보장)                                     │  │
 │  │ 2. 플랫폼, 애플리케이션, IAM (Identity & Access)                        │  │
 │  │ 3. 운영체제, 네트워크 설정 (방화벽 규칙), 방화벽 설정                      │  │
 │  │ 4. 클라이언트 및 서버 측 데이터 암호화                                    │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 │  ----------------------- [ 책임의 경계 (The Boundary) ] ---------------------- │
 │                                                                             │
 │  [CSP 책임: 클라우드 '자체'의 보안] (Security **OF** the Cloud)                 │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ 1. 소프트웨어: 컴퓨팅, 스토리지, DB, 네트워킹 가상화 계층                  │  │
 │  │ 2. 하드웨어/글로벌 인프라: 리전, 가용 영역, 엣지 로케이션                  │  │
 │  │ 3. 물리적 보안: 데이터 센터 보안, 출입 통제, 재해 복구 시스템              │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처는 크게 두 덩어리로 나뉜다. **CSP(AWS 등)**는 물리적인 데이터 센터 건물을 짓고, 서버 하드웨어를 관리하며, 그 위에서 돌아가는 가상화 소프트웨어의 무결성을 책임진다. **고객**은 이 인프라 위에 얹어지는 모든 것을 책임진다. 특히 가장 상단의 **'데이터'**는 어떤 모델(IaaS/PaaS/SaaS)을 쓰더라도 100% 고객의 책임이다. 이는 CSP가 고객의 프라이버시를 위해 데이터 내용에 접근하지 않기 때문이다. 따라서 데이터 암호화와 접근 권한 설정(IAM)은 고객 보안의 핵심이다.

---

### 심층 동작 원리: 서비스 모델별 보안 위임 (Security Delegation)

1. **IaaS**: 고객은 OS 설치부터 패치, 방화벽 설정까지 직접 해야 하므로 자유도는 높으나 보안 부담이 가장 크다.
2. **PaaS**: CSP가 OS 패치와 런타임 보안을 대신 해주므로 고객은 코드와 데이터에만 집중하면 된다. (보안 위임 발생)
3. **SaaS**: 대부분의 보안을 CSP가 관리하며, 고객은 "누가 이 소프트웨어에 접속할 수 있는가(IAM)"와 "어떤 데이터를 넣는가"만 관리하면 된다.

- **📢 섹션 요약 비유**: 텐트를 치고 자느냐(IaaS), 풀옵션 오피스텔에 사느냐(PaaS), 호텔 서비스를 이용하느냐(SaaS)에 따라 내가 직접 문단속을 해야 하는 범위가 달라지는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 책임 공유 모델의 오해와 진실

| 구분 | 일반적인 오해 | 진실 (현실) |
|:---|:---|:---|
| **백업** | "CSP가 알아서 다 복구해주겠지" | **데이터 백업 및 복구 주체는 고객임** |
| **방화벽** | "클라우드는 기본적으로 안전해" | **포트를 열고 닫는 설정(Sec Group)은 고객 몫** |
| **암호화** | "내 데이터는 자동으로 암호화돼" | **암호화 옵션을 켜고 키를 관리하는 것은 고객임** |
| **로그** | "사고 나면 CSP가 로그 다 주겠지" | **필요한 로그를 설정하고 보관하는 것도 고객 책임** |

---

### 기술 융합: 책임 공유 모델과 자동화 도구

고객의 책임 영역을 더 쉽고 완벽하게 이행하기 위해 자동화 도구가 쓰인다.
- **CSPM (Configuration Management)**: 고객의 책임인 '잘못된 설정'을 실시간으로 감시하여 수정해줌.
- **CASB (Cloud Access Security Broker)**: SaaS 환경에서 고객의 데이터가 외부로 유출되는지(고객 책임 영역)를 감시함.

- **📢 섹션 요약 비유**: 내 집 안의 방범은 내 책임(고객 영역)이지만, 이를 더 잘하기 위해 세스코나 캡스(보안 솔루션)를 불러서 관리하는 지혜가 필요하다는 뜻입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 클라우드 도입 초기 기업의 보안 전략 수립

- **문제 상황**: 온프레미스만 운영하던 기업이 AWS로 이전하면서, 보안팀은 모든 것을 AWS에 맡기려 하고 개발팀은 설정 권한을 남발하여 보안 홀 발생.
- **의사결정 (책임 공유 모델 기반 가이드)**:
  1. **책임 매트릭스(RACI) 작성**: 우리 회사가 사용하는 서비스별(EC2, S3, RDS 등)로 보안 책임 주체를 명문화하여 전사에 공표한다.
  2. **Identity 중심 보안**: 네트워크 경계는 CSP가 제공하는 기능을 쓰되, '사용자 권한(IAM)'은 우리 회사의 핵심 책임임을 인지하고 정기 감사를 수행한다.
  3. **Managed Service 우선 활용**: 보안 역량이 부족하다면 IaaS보다는 CSP가 보안을 더 많이 책임져주는 PaaS(RDS 등)를 우선 채택하여 리스크를 CSP에게 위임한다.
  4. **Compliance Map**: ISMS-P 인증 심사 시, CSP로부터 '보안 준수 증명서(SOC Report)'를 받아 우리 책임이 아닌 인프라 보안 영역의 증적을 갈음한다.

### 도입 체크리스트
- **Contract Review**: CSP와의 계약서에 명시된 책임 한계와 배상 범위(SLA)를 정확히 파악하고 있는가?
- **Configuration Hygiene**: 보안 그룹의 'All Open(0.0.0.0/0)' 정책이 우리 책임 하에 방치되고 있지는 않은가?
- **Encryption of PII**: DB 내의 개인정보를 암호화하여 '클라우드 내 보안' 의무를 다하고 있는가?

### 안티패턴
- **Passivity**: "유명한 구글/아마존 클라우드니까 해킹 안 당하겠지"라는 안일함. (클라우드 인프라 자체가 뚫리는 일보다 고객의 설정 실수가 100배 많음)
- **Log Ignorance**: CSP가 제공하는 인프라 로그(CloudTrail 등)를 우리 책임으로 수집하지 않아 침해 사고 원인 분석이 불가능한 상태.

- **📢 섹션 요약 비유**: "튼튼한 금고를 샀으니까 비밀번호를 '1234'로 해도 되겠지"라고 생각하는 것이 얼마나 위험한지 깨닫는 것이 클라우드 보안의 시작입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 설정 오류 사고 90% 예방 | 책임 범위를 명확히 인지하고 집중 관리함으로써 관리 소홀 차단 |
| **정량적 (Cost)** | 보안 운영 비용 최적화 | CSP가 책임지는 물리/가상화 계층 보안 인력 및 장비 비용 절감 |
| **정성적 (Compliance)** | 외부 감사 신뢰도 확보 | 책임 소재를 근거로 한 체계적인 증적 관리를 통해 인증 통과 용이 |
| **정성적 (Agility)** | 비즈니스 속도 향상 | 보안 책임을 명확히 분담함으로써 빠른 인프라 확장 및 서비스 출시 가능 |

### 미래 전망
- **Shared Fate (책임 공동체)**: 단순히 "네 책임이야"라고 떠넘기는 것이 아니라, 구글 클라우드 등에서 제안하는 것처럼 CSP가 고객의 설정을 적극 가이드하고 함께 책임지는 '상생 보안' 모델로 발전할 것이다.
- **Autonomous Configuration**: 인공지능이 책임 공유 모델 상의 고객 영역을 스스로 진단하고, 최적의 보안 설정을 자율적으로 적용하는 '무인 보안 운영'이 현실화될 것이다.

### 참고 표준
- **AWS Shared Responsibility Model**: 업계에서 가장 널리 인용되는 표준 모델
- **ISO/IEC 27017**: 클라우드 서비스 보안 통제 표준
- **CSA Cloud Controls Matrix (CCM)**: 클라우드 보안 통제 및 책임 맵핑 도구

- **📢 섹션 요약 비유**: 예전에는 나 혼자 성을 지켰다면, 이제는 땅과 성벽은 거인이 빌려주고(CSP), 나는 그 성 안의 평화를 지키는 '분업 보안'의 시대가 완성되었습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 학교 사물함(클라우드)을 쓸 때, **사물함 자체가 튼튼하고 열쇠 구멍이 안 부서지게 관리하는 건 학교(CSP) 선생님의 책임**이에요.
2. 하지만 사물함 안에 **소중한 물건을 잘 정리해두고, 자물쇠를 꼭 잠그는 건 사물함을 쓰는 나(고객)의 책임**이랍니다.
3. 내가 자물쇠를 안 잠가서 물건을 잃어버리면 학교 탓을 할 수 없으니, 내 물건은 내가 잘 지키기로 약속해요!
