# 569. 회복력 패턴 (Circuit Breaker, Bulkhead, Retry)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 회복력 패턴 (Resilience Patterns)은 분산 시스템에서 특정 서비스의 장애가 전체 시스템으로 확산되는 **연쇄 장애(Cascading Failure)를 방지**하고, 일시적인 오류로부터 시스템이 스스로 복구될 수 있도록 돕는 설계 기법이다.
> 2. **가치**: 장애를 "완벽히 막는 것"이 아니라 **"장애를 격리하고 수용하는 것"**에 집중하며, 실패한 요청을 자동으로 다시 시도하거나(Retry), 문제가 있는 서비스로의 경로를 즉시 차단(Circuit Breaker)하여 시스템 가용성을 유지한다.
> 3. **융합**: 서비스 메시(Istio), Resilience4j 라이브러리와 결합되어 인프라 및 코드 수준에서 구현되며, 마이크로서비스(MSA) 아키텍처의 안정성을 지탱하는 가장 실천적인 보안 및 가용성 전략이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 회복력은 시스템에 충격(장애)이 가해졌을 때 얼마나 빨리 정상 상태로 돌아오는가를 말한다. 
  - **Circuit Breaker**: 과부하된 전선을 차단하는 누전 차단기처럼, 장애 서비스로의 요청을 끊음.
  - **Bulkhead**: 배의 침수를 막는 격벽처럼, 특정 서비스의 장애가 다른 서비스의 자원(Thread 등)을 갉아먹지 못하게 격리함.
  - **Retry**: 일시적인 네트워크 오류 시 자동으로 다시 요청함.

- **💡 비유**: 회복력 패턴은 **"병원의 응급 처치 및 격리 시스템"**과 같습니다. 감염병 환자(장애 서비스)가 발생했을 때, 병원 전체로 퍼지지 않게 병동을 폐쇄하고(Circuit Breaker), 다른 환자들과 섞이지 않게 격리실에 가두며(Bulkhead), 치료가 안 되면 잠시 후 다시 약을 투여하는(Retry) 것과 같습니다.

- **등장 배경**:
  1. **분산 시스템의 불확실성**: 수많은 서비스가 네트워크로 연결된 환경에서는 단 하나의 서비스 지연이 전체 호출 체인을 마비시키는 '눈사태 효과(Snowball Effect)'가 빈번했다.
  2. **일시적 오류의 증가**: 클라우드 환경의 일시적 네트워크 순단이나 타임아웃은 빈번하므로, 이를 '치명적 장애'가 아닌 '지나가는 소나기'로 처리할 방법이 필요했다.
  3. **사용자 경험 보호**: 전체 기능 중 일부가 안 되더라도 나머지 기능은 정상 작동하는 '우아한 성능 저하(Graceful Degradation)'가 비즈니스의 필수 요건이 되었다.

- **📢 섹션 요약 비유**: 태풍이 불 때 나무가 부러지지 않게 유연하게 흔들리게 하거나, 가지 하나가 꺾여도 나무 전체가 죽지 않게 보호하는 자연의 생존 지혜를 소프트웨어에 적용한 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 3대 핵심 회복력 패턴 분석

| 패턴 명칭 | 작동 원리 | 핵심 목표 | 비유 |
|:---|:---|:---|:---|
| **Circuit Breaker** | 에러율 임계치 초과 시 호출 차단 및 Fallback 실행 | **연쇄 장애 방지**, 장애 서비스 휴식 제공 | 집안의 누전 차단기 |
| **Bulkhead** | 서비스별로 사용할 수 있는 자원(Thread 등)을 분리 | **장애 격리**, 특정 부하가 전체 전이 차단 | 배의 방수 격실 |
| **Retry** | 실패 시 일정 간격을 두고 재시도 (Exponential Backoff) | **일시적 오류 극복**, 데이터 정합성 유지 | 다시 전화 걸기 |

---

### 2. Circuit Breaker 상태 전이 아키텍처

장애를 감지하고 복구하는 내부 로직을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Circuit Breaker (누전 차단기) 상태 머신                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │      [CLOSED] (정상) ──────────────────────────▶ [OPEN] (차단)                │
 │          │   ▲        [에러율 > 임계치]           │   │                     │
 │          │   │                                   │   │                     │
 │          │   └───────────────────────────────────┘   │                     │
 │          │           [성공률 > 기준치]               │ [대기 시간 경과]       │
 │          │                                           │                     │
 │          ▼                                           ▼                     │
 │      [HALF-OPEN] (확인) ◀────────────────────────────┘                     │
 │          │                                                                  │
 │          └───── (일부 요청만 허용하여 상태 확인) ──────────────────────────────┘  │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 주요 상태 설명:                                                        │  │
 │  │ - CLOSED: 요청을 모두 통과시킴. 에러를 카운트함.                            │  │
 │  │ - OPEN: 요청을 즉시 거절하고 Fallback(대체 로직) 실행.                       │  │
 │  │ - HALF-OPEN: 장애가 고쳐졌는지 확인하기 위해 소량의 트래픽만 흘려보냄.         │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Circuit Breaker는 세 가지 상태를 가진다. 1) 평소에는 **CLOSED** 상태로 모든 요청을 보내다가, 특정 시간 동안 에러가 너무 많이 발생하면 **OPEN** 상태로 바뀐다. 이때부터는 서버를 호출하지 않고 "죄송합니다. 잠시 후 이용해주세요"라는 **Fallback** 메시지를 즉시 반환한다. 2) 일정 시간이 흐른 뒤 시스템은 **HALF-OPEN** 상태가 되어, '간'을 보듯 몇 개의 요청만 보내본다. 3) 이 요청들이 성공하면 다시 **CLOSED**로 돌아가고, 실패하면 다시 **OPEN**으로 돌아간다. 이 메커니즘은 장애가 난 서비스가 회복될 시간을 벌어주는 동시에, 사용자에게는 무한 대기가 아닌 빠른 응답을 보장한다.

---

### 심층 동작 원리: Bulkhead (격벽) 패턴의 자원 격리

하나의 서비스가 모든 자원을 독점하지 못하게 막는 기술이다.
1. **Thread Pool 분리**: 서비스 A용 쓰레드 10개, 서비스 B용 10개를 따로 둔다.
2. **효과**: 서비스 B가 느려져서 쓰레드 10개를 다 써버려도, 서비스 A는 자기 몫인 10개의 쓰레드로 평소처럼 작동할 수 있다.
3. **가치**: "옆집에 불이 나도 우리 집(자원)까지 태우지 못하게" 벽을 치는 전략이다.

- **📢 섹션 요약 비유**: 뷔페 식당에서 한 사람이 음식을 다 가져가지 못하게 1인당 접시 수(자원)를 제한하여, 모든 손님이 골고루 식사할 수 있게 만드는 규칙과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 회복력 패턴 간의 시너지

| 조합 | 상호 보완 효과 |
|:---|:---|
| **Retry + Circuit Breaker** | 일시적 오류는 다시 시도하되, 계속 안 되면 차단기가 작동하여 무의미한 시도 중단 |
| **Bulkhead + Timeout** | 자원을 격리한 상태에서 응답이 너무 늦으면 끊어버려, 자원 회수 속도 극대화 |
| **Circuit Breaker + Fallback** | 차단기가 열렸을 때, 에러 대신 캐시된 데이터나 기본값을 보여주어 사용자 경험 유지 |

---

### 기술 융합: 카오스 엔지니어링 (Chaos Engineering)

회복력 패턴이 실제 작동하는지 검증하기 위한 기술이다.
- **원리**: 실운영 환경에서 일부러 서버를 끄거나 지연을 발생시킨다 (예: Netflix의 Chaos Monkey).
- **가치**: 우리가 설계한 Circuit Breaker가 정말로 0.1초 만에 작동하는지 눈으로 확인하여 시스템의 자신감을 확보한다.

- **📢 섹션 요약 비유**: 건물에 화재 경보기가 잘 작동하는지 확인하기 위해, 정기적으로 모의 소방 훈련(카오스 엔지니어링)을 실시하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '추천 서비스' 장애로 인한 메인 화면 속도 저하 방지

- **문제 상황**: 메인 화면 로딩 시 '상품 추천' API를 부르는데, 추천 서버가 DB 부하로 인해 응답이 10초 이상 걸리면서 쇼핑몰 메인 페이지 자체가 안 뜨는 대형 장애 발생.
- **의사결정 (회복력 패턴 적용)**:
  1. **Circuit Breaker 도입**: 추천 서비스 에이전트에 차단기를 설치하고, 5번 연속 타임아웃 발생 시 30초간 호출을 전면 차단하도록 설정한다.
  2. **Fallback 설계**: 추천 서비스 호출이 차단되면 에러 페이지를 띄우는 대신, 미리 준비된 '실시간 인기 순위(정적 데이터)'를 보여주도록 로직을 구현한다.
  3. **Bulkhead 적용**: 추천 서비스를 위한 전용 HTTP 커넥션 풀을 분리하여, 추천 서버가 느려져도 메인 상품 조회 쓰레드는 영향을 받지 않게 한다.
  4. **결과**: 추천 서비스는 죽어있어도 메인 페이지는 0.5초 만에 정상 로딩되며, 사용자들은 장애 사실을 거의 인지하지 못한 채 쇼핑을 계속한다.

### 도입 체크리스트
- **Threshold Tuning**: 차단기가 열리는 기준(에러율 50% 등)이 너무 민감하여 정상적인 상황에서도 차단되지는 않는가?
- **Exponential Backoff**: Retry 시 매번 같은 간격이 아닌, 1초, 2초, 4초 순으로 간격을 늘려 대상 서버의 부담을 줄이고 있는가?
- **Monitoring**: 차단기가 열리고 닫히는 현황을 실시간으로 관제(Grafana 등)하고 있는가?

### 안티패턴
- **Infinite Retry**: 성공할 때까지 무한정 다시 시도하는 행위. 죽어가는 서버에 확인사살(DDoS 공격)을 하는 꼴이 된다.
- **No Fallback**: 차단기만 달아놓고 에러를 그대로 노출하는 경우. 사용자는 여전히 서비스 실패를 경험하게 된다.

- **📢 섹션 요약 비유**: "안 되면 될 때까지"라는 무식한 고집(무한 재시도)보다는, "안 되면 잠시 쉬었다가 다시 하자"는 지혜로운 양보(Circuit Breaker)가 더 높은 성공률을 보장합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Availability)** | 가동 시간(Uptime) 99.99% 달성 | 부분 장애 발생 시에도 핵심 서비스 중단 없이 운영 가능 |
| **정량적 (Performance)** | 응답 속도 안정화 | 지연된 서비스로의 요청을 즉시 거절(Fail-fast)하여 전체 속도 유지 |
| **정성적 (Reliability)** | 시스템 강건성(Robustness) 확보 | 어떤 상황에서도 시스템이 최소한의 기능을 수행한다는 신뢰 형성 |
| **정성적 (Agility)** | 운영 부담 감소 | 일시적인 오류는 시스템이 스스로 해결하므로 야간 호출(On-call) 감소 |

### 미래 전망
- **Self-Healing Infrastructure**: 머신러닝이 트래픽 패턴을 분석하여, 장애가 발생하기 직전에 회복력 파라미터를 동적으로 조절하는 '자율 복구 시스템'이 실현될 것이다.
- **Network-level Resilience**: 애플리케이션 코드가 아닌 네트워크 서비스 메시(Istio 등) 레이어에서 모든 회복력 패턴이 완벽히 자동화되어 투명하게 적용될 것이다.

### 참고 표준
- **Reactive Manifesto**: 회복력(Resilient)을 현대 시스템의 4대 핵심 가치 중 하나로 정의
- **Resilience4j / Hystrix**: 실무 구현을 위한 대표적 라이브러리 가이드
- **Chaos Engineering Principles**: 회복력 검증 방법론 표준

- **📢 섹션 요약 비유**: 과거의 시스템이 '부러지지 않는 단단한 강철'을 지향했다면, 현대의 시스템은 '충격을 흡수하고 다시 일어나는 유연한 대나무'와 같은 회복력을 지향합니다.
