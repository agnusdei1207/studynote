# 687. 정적 사이트 생성 (SSG) Jamstack

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 빌드 타임에 미리 모든 페이지를 완성된 HTML 파일로 생성하여 CDN을 통해 배포하는 **정적 사이트 생성(SSG)** 방식과, 이를 기반으로 JavaScript, API, Markup을 결합한 현대적 웹 아키텍처인 **Jamstack**이다.
> 2. **탈중앙화**: 서버 측 런타임 의존성을 제거하고, 동적 기능은 클라이언트의 자바스크립트와 외부 서비스(API)로 위임함으로써 **보안성, 성능, 확장성**을 동시에 확보한다.
> 3. **가치**: 데이터베이스나 서버 장애 리스크를 원천 차단하고, 캐싱 효율을 극대화하여 가장 빠르고 저렴하게 고품질의 웹 서비스를 운영할 수 있게 한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "서버가 없는 웹의 시대"

전통적인 웹은 사용자가 접속할 때마다 서버가 바쁘게 DB를 뒤져 페이지를 만들었습니다. 하지만 이는 서버 부하와 보안 취약점을 야기했습니다. **SSG (Static Site Generation)**는 "바뀌지 않는 페이지라면 미리 다 만들어두자"는 역발상에서 시작되었습니다. 이를 아키텍처로 정립한 것이 **Jamstack (JavaScript, APIs, Markup)**입니다.

### 💡 비유: 주문 제작 도시락 vs 편의점 도시락

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        SSG 및 Jamstack 비유                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 전통적 방식 (주문 제작 도시락):                                            │
│     손님이 올 때마다 주방(Server)에서 재료를 다듬고 요리해서 줌.                 │
│     손님이 갑자기 1,000명 오면 주방이 마비됨.                                   │
│                                                                             │
│  2. SSG/Jamstack 방식 (편의점 도시락):                                        │
│     공장(Build Time)에서 이미 도시락을 다 만들어서 전국 편의점(CDN)에 뿌려둠.     │
│     손님은 그냥 집어 가기만 하면 됨(0초 로딩).                                  │
│     - 반찬이 부족하면(동적 기능)? 옆에 있는 렌지(API)를 써서 직접 데워 먹음.      │
│                                                                             │
│  → 미리 다 만들어두고, 배달망(CDN)을 통해 가장 빠르게 전달하는 것!               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. Jamstack의 3대 구성 요소
- **JavaScript**: 브라우저에서 실행되며 모든 동적인 상호작용 담당.
- **APIs**: DB 처리, 인증, 결제 등 모든 서버 측 로직은 재사용 가능한 API로 처리.
- **Markup**: 미리 빌드된 템플릿(HTML)으로, Git에서 관리되고 CDN으로 서비스됨.

### 2. SSG의 작동 프로세스
1. **Develop**: 로컬 환경에서 코드와 콘텐츠(Markdown 등) 작성.
2. **Commit**: Git 저장소에 푸시.
3. **Build**: CI/CD 도구가 모든 조합을 계산하여 정적 HTML 생성.
4. **Deploy**: 전 세계 CDN 노드에 결과물 배포.

### 3. Jamstack 아키텍처 구조도 (ASCII)

```text
    [ Git Repo ] ──▶ [ Build Server ] ──▶ [ Static Assets ] ──▶ [ CDN ]
    (Code/MD)         (Gatsby/Zola)        (HTML/CSS/JS)         (Edge)
                                                                   │
    ┌──────────────────────────────────────────────────────────────┘
    │ (Fast Response)
    ▼
    [ User Browser ] ──(REST/gRPC)──▶ [ External APIs ]
                                       (Auth0, Stripe, Algolia)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. SSR vs SSG 상세 비교

| 항목 | SSR (Server-Side) | SSG (Static Site) |
|:---:|:---|:---|
| **생성 시점** | **요청 시 (Request Time)** | **빌드 시 (Build Time)** |
| **속도 (TTFB)** | 중간 (서버 연산 필요) | **매우 빠름 (즉시 응답)** |
| **보안** | 데이터베이스 노출 위험 | **매우 안전 (공격 대상 없음)** |
| **데이터 갱신** | 실시간 반영 유리 | 빌드 프로세스 재실행 필요 |
| **적합한 서비스** | SNS, 주식 정보 | 블로그, 문서, 랜딩 페이지 |

### 2. 기술적 시너지: Headless CMS
Jamstack 환경에서 관리자 페이지는 **Headless CMS** (Strapi, Contentful 등)가 담당합니다. 콘텐츠를 수정하면 Webhook이 발생하여 자동으로 전체 사이트를 다시 빌드(Re-build)하는 파이프라인이 구축됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 기업 기술 블로그 구축
- **상황**: 보안 사고 위험이 없고, 트래픽 폭증에도 견디며, 유지비가 거의 없는 블로그 원함.
- **결단**: **Jamstack (Zola + GitHub Pages + Cloudflare) 채택**. 
- **판단**: 모든 글을 Markdown으로 관리하여 Git에 저장. 검색 기능은 Algolia API로, 댓글은 외부 위젯으로 처리. 서버 운영 비용 0원 달성.
- **효과**: 전 세계 어디서 접속해도 1초 이내 로딩, 서버 관리 포인트 완전 제거.

### 📢 기술사적 결언
> "Jamstack은 웹의 **'뿌리(Static)'**로 돌아가는 동시에 **'미래(API-first)'**를 결합한 아키텍처다. 서버가 없다는 것은 관리할 포인트가 줄어든다는 것이며, 이는 곧 **'기술 부채의 소멸'**을 의미한다. 대규모 데이터 변경이 잦은 서비스에는 부적합할 수 있으나, 정적 콘텐츠의 비중이 높은 80% 이상의 웹 서비스에게 Jamstack은 가장 효율적인 엔지니어링 정답지다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **성능**: Google Lighthouse 성능 점수 90점 이상 달성 용이.
- **비용**: 전통적 서버 호스팅 대비 인프라 비용 80~100% 절감.

### 미래 전망
미래의 Jamstack은 **'분산형 엣지 컴퓨팅'**과 결합하여 진화하고 있습니다. 전체를 다시 빌드하는 대신 변경된 페이지만 엣지에서 즉시 생성하는 **ISR (Incremental Static Regeneration)** 기술이 보편화되면서, SSG의 '느린 갱신' 한계까지 극복해 나가고 있습니다. 이제 정적 사이트는 더 이상 '정적'이지 않은 강력한 플랫폼이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **SSG**는 친구들이 집에 놀러 오기 전에 미리 모든 간식을 **예쁜 접시에 다 담아두는 것**과 같아요.
2. 친구들이 오자마자 바로 먹을 수 있어서 기다릴 필요가 전혀 없죠.
3. 만약 새로운 과자를 가져오고 싶으면(업데이트), 접시를 다시 세팅해야 하지만 **미리 준비해둔 덕분**에 잔치가 아주 순조롭게 진행된답니다!
