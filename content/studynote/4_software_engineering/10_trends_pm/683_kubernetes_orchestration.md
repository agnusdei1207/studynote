# 683. 쿠버네티스 (K8s) 오케스트레이션

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 수많은 컨테이너의 배치, 확장, 네트워크, 업데이트를 자동화하여 거대한 분산 시스템을 효율적으로 관리하는 **오픈소스 컨테이너 오케스트레이션 플랫폼**이다.
> 2. **선언적 운영**: 사용자가 "앱이 3개 떠 있어야 해"라고 상태를 정의(YAML)하면, 쿠버네티스가 **Control Plane**을 통해 실제 상태를 이에 맞게 유지하는 **조정(Reconciliation)** 프로세스를 수행한다.
> 3. **가치**: 자동 복구(Self-healing), 무중단 배포, 수평 확장(Auto-scaling)을 기본 제공하여, 인프라의 복잡성에 상관없이 애플리케이션의 고가용성과 운영 안정성을 보장한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "도커만으로는 부족하다"

도커 컨테이너를 한두 개 띄우는 것은 쉽습니다. 하지만 서비스 규모가 커져서 수천 개의 컨테이너를 수십 대의 서버에 나누어 띄워야 한다면 어떨까요? "어느 서버에 여유가 있지?", "컨테이너가 죽으면 누가 다시 띄우지?", "서로 어떻게 통신하지?"와 같은 관리의 한계에 부딪힙니다. 구글의 Borg 프로젝트 노하우를 바탕으로 탄생한 **쿠버네티스(Kubernetes, K8s)**는 이 모든 고민을 해결해 주는 '인프라의 뇌'입니다.

### 💡 비유: 오케스트라 지휘자와 단원들

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        쿠버네티스(K8s) 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 컨테이너 (단원):                                                         │
│     악기(앱)를 연주하는 개별 연주자들.                                          │
│                                                                             │
│  2. 쿠버네티스 (지휘자):                                                     │
│     악보(YAML 설계도)를 보고 전체 조화를 맞춤.                                   │
│     - 연주자가 아파서 못 나오면 대타를 즉시 투입(Self-healing).                  │
│     - 관객이 많아지면 연주자를 더 많이 배치(Auto-scaling).                       │
│     - 무대 장치를 연주 중에 조용히 교체(무중단 배포).                             │
│                                                                             │
│  → 지휘자만 있으면 연주자가 수백 명이라도 완벽한 음악(서비스)이 나옴!              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 쿠버네티스 아키텍처 구성 요소 (Control vs Worker)

| 분류 | 명칭 | 핵심 역할 |
|:---:|:---|:---|
| **Control Plane** | **API Server** | 모든 명령의 입구, 중앙 관리 지점 |
| (마스터 노드) | **etcd** | 클러스터의 모든 상태를 저장하는 Key-Value 저장소 |
| | **Scheduler** | 어떤 컨테이너를 어느 노드에 띄울지 결정 |
| | **Controller Mgr** | 실제 상태를 원하는 상태로 맞추는 감시자 |
| **Worker Node** | **Kubelet** | 마스터의 명령을 받아 컨테이너를 직접 관리 |
| (실행 노드) | **Kube-proxy** | 컨테이너 간 네트워크 통신 및 로드밸런싱 |
| | **Pod** | 쿠버네티스의 최소 배포 단위 (하나 이상의 컨테이너 묶음) |

### 2. 선언적 명령과 조정 루프 (ASCII)

```text
    [ YAML (Desired State) ] ──▶ [ API Server ] ──▶ [ etcd (Save) ]
                                       ▲
                                       │ (Continuous Watch & Compare)
                                       ▼
    [ Controller Manager ] ◀── [ Reconciliation Loop ] ──▶ [ Actual State ]
                                       │
                                       ▼ (Fix Drift)
                               [ Worker Nodes (Pods) ]
```

### 3. 주요 오브젝트 유형
- **Deployment**: 앱 배포 및 업데이트 전략 관리 (버전 관리).
- **Service**: Pod들에 접근하기 위한 고정된 IP/DNS 제공 (로드밸런싱).
- **Ingress**: 외부 트래픽을 서비스로 연결하는 관문 (L7 라우팅).
- **ConfigMap/Secret**: 앱 설정과 보안 정보를 코드와 분리하여 주입.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 오케스트레이션 도구 비교

| 항목 | Kubernetes (K8s) | Docker Swarm | HashiCorp Nomad |
|:---:|:---|:---|:---|
| **주도권** | 시장 표준 (De facto) | 도커 내장, 단순함 | 멀티 클라우드 유연성 |
| **확장성** | 매우 높음 (거대 커뮤니티) | 소규모 적합 | 단일 바이너리 가벼움 |
| **난이도** | 높음 (학습 곡선 가파름) | 낮음 | 중간 |

### 2. 기술적 시너지: 클라우드 네이티브와 GitOps
쿠버네티스는 그 자체로 거대한 API 덩어리입니다. 따라서 Git에 YAML 파일을 올려두면 자동으로 클러스터에 반영되는 **GitOps (ArgoCD 등)**와 결합했을 때, 인프라의 가시성과 복구 능력이 극대화됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 무중단 롤링 업데이트 (Rolling Update)
- **상황**: 서비스 중단 없이 신규 버전을 배포해야 함.
- **결단**: **Deployment 오브젝트 활용**. 
- **판단**: `maxSurge`(동시 추가 가능 개수)와 `maxUnavailable`(동시 삭제 가능 개수) 값을 설정. 쿠버네티스가 새 버전 Pod를 하나 띄우고 건강(Health Check)을 확인한 뒤, 옛날 버전 Pod를 하나 지우는 과정을 반복.
- **효과**: 배포 중에도 서비스 가용성은 100% 유지되며, 실패 시 이전 버전으로의 자동 롤백 지원.

### 📢 기술사적 결언
> "쿠버네티스는 운영의 복잡성을 해결하기 위해 **'또 다른 복잡성'**을 도입한 도구다. 따라서 모든 프로젝트에 K8s가 정답은 아니다. 하지만 현대의 클라우드 아키텍처에서 K8s는 단순한 컨테이너 관리자가 아닌, 컴퓨팅, 네트워크, 스토리지를 추상화한 **'클라우드 운영체제(Cloud OS)'**로 진화했다. 엔지니어는 K8s의 내부 동작 원리를 이해함으로써, 어떤 클라우드 벤더에서도 통용되는 진정한 '인프라 이식성'을 확보할 수 있다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **운영 안정성**: 장애 시 자동 복구 기능을 통해 가용성 99.99% 달성 용이.
- **비용 최적화**: 자원 사용률 모니터링 기반의 HPA (Horizontal Pod Autoscaler)로 인프라 비용 30% 이상 절감.

### 미래 전망
미래의 K8s는 **'Serverless Kubernetes'**로 향하고 있습니다. 사용자가 노드 관리조차 신경 쓰지 않는 AWS Fargate와 같은 모델이 주류가 될 것입니다. 또한 K8s 상단에서 복잡한 통신을 담당하는 **'서비스 메시(Service Mesh)'**와, 특정 도메인의 운영 지식을 코드로 담은 **'오퍼레이터(Operator)'** 패턴이 결합되어, 인간의 개입이 거의 없는 '자율 운영 인프라' 시대가 열릴 것입니다.
