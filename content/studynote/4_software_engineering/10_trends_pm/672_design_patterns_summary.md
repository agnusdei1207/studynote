# 672. 디자인 패턴 생성, 구조, 행위 3대 분류

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 설계에서 반복적으로 발생하는 문제들에 대해 선배 개발자들이 정립해놓은 **검증된 해결책(Best Practice)의 카탈로그**이다.
> 2. **3대 분류**: 객체의 생성을 캡슐화하는 **생성(Creational)**, 클래스/객체 조합을 통한 구조를 정의하는 **구조(Structural)**, 객체 간의 상호작용과 책임 분산을 다루는 **행위(Behavioral)** 패턴으로 나뉜다.
> 3. **가치**: 설계의 재사용성을 높이고 개발자 간의 공통 언어(Common Language)를 제공하여 유지보수 비용을 낮추며, SOLID와 같은 객체지향 원칙을 실제 코드로 구현하는 구체적 방법론이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "바퀴를 다시 발명하지 마라"

1994년 에리히 감마(Erich Gamma) 등 4명의 저자(Gang of Four, GoF)가 23개의 패턴을 정리하며 대중화되었습니다. 디자인 패턴은 특정 프로그래밍 언어에 종속된 기술이 아니라, 객체지향의 추상화, 상속, 다형성을 활용하여 **유연하고 확장성 있는 구조**를 만드는 지혜의 집약체입니다.

### 💡 비유: 건축의 표준 평면도와 인테리어 양식

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        디자인 패턴 3대 분류 비유                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 멋진 집을 지으려 함.                                                   │
│                                                                             │
│  1. 생성 (Creational - 가구 주문):                                           │
│     "침대를 어디서 어떻게 가져올까?" 직접 만들지, 공장에 맡길지(Factory),        │
│     세상에 하나뿐인 명품을 쓸지(Singleton) 결정하는 것.                         │
│                                                                             │
│  2. 구조 (Structural - 방 배치):                                             │
│     "방과 거실을 어떻게 연결할까?" 좁은 문에 큰 가구를 넣기 위해 어댑터를 쓰거나   │
│     여러 방을 묶어 큰 거실로 만드는 등 집의 뼈대를 구성하는 것.                  │
│                                                                             │
│  3. 행위 (Behavioral - 생활 규칙):                                           │
│     "가족끼리 어떻게 소통할까?" 벨을 누르면(Event) 문을 열어주거나(Observer),    │
│     심부름을 시키는(Command) 등 사람들 사이의 상호작용 규칙을 정하는 것.          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 디자인 패턴 3대 분류 및 대표 패턴

| 분류 | 핵심 목적 | 대표적인 GoF 패턴 |
|:---:|:---|:---|
| **생성 (Creational)** | 객체 생성의 복잡성을 숨기고 적절한 방식으로 생성 | Singleton, Factory Method, Abstract Factory, Builder, Prototype |
| **구조 (Structural)** | 더 큰 구조를 형성하기 위해 클래스/객체 합성 | Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy |
| **행위 (Behavioral)** | 객체 간의 알고리즘이나 책임 분배 정의 | Observer, Strategy, Command, State, Template Method, Iterator |

### 2. 분류별 핵심 철학 (ASCII)

```text
    [ 생성 패턴 ] : "How to create?" ──▶ [ 캡슐화 ]
    (결합도 낮추기 위해 생성 로직을 분리)
    
    [ 구조 패턴 ] : "How to compose?" ──▶ [ 유연성 ]
    (상속보다는 합성을 통해 더 큰 구조 형성)
    
    [ 행위 패턴 ] : "How to interact?" ──▶ [ 결합도 분리 ]
    (객체 간의 메시지 전달 방식을 체계화)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 디자인 원칙(SOLID) vs 디자인 패턴(Pattern)
- **SOLID**: 우리가 지향해야 할 **도덕책(원칙)**. (예: OCP - 확장에 열려있어야 함)
- **Pattern**: 그 원칙을 지키기 위한 **실전 기술(수단)**. (예: Strategy 패턴을 쓰면 OCP를 지키기 쉬움)

### 2. 기술적 시너지: 프레임워크와의 결합
현대의 프레임워크는 디자인 패턴의 거대한 집합체입니다.
- **Spring**: Singleton(Bean 관리), Proxy(AOP), Template Method(JdbcTemplate) 등.
- **React/Vue**: Component(Composite), Props/Emit(Observer) 등.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 프로젝트 복잡도 관리
- **상황**: 신규 기능이 추가될 때마다 기존 코드를 대대적으로 수정해야 함.
- **결단**: **행위 패턴(Strategy)** 도입. 
- **판단**: if-else 지옥을 벗어나기 위해 변하는 부분을 전략 클래스로 캡슐화. 이를 통해 기존 코드는 건드리지 않고 새로운 전략 클래스만 추가하여 기능 확장 (OCP 준수).

### 📢 기술사적 결언
> "패턴은 **'약'**이자 **'독'**이다. 적재적소에 쓰면 아키텍처의 구원자가 되지만, 패턴을 쓰기 위해 설계를 억지로 맞추는 '패턴 만능주의(Pattern-itis)'는 과도한 복잡성(Over-engineering)을 초래한다. 가장 훌륭한 설계자는 수많은 패턴을 알고 있되, **'가장 단순한 해결책'**부터 검토하는 사람이다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적/정성적 기대효과
- **유지보수 효율**: 구조적 일관성 확보로 코드 리뷰 및 수정 시간 30% 이상 단축.
- **협업 가독성**: "여기 옵저버 패턴 썼어" 한마디로 수백 줄의 로직 흐름 전달 가능.

### 미래 전망
최근에는 GoF의 고전적 패턴을 넘어, 클라우드 환경에 최적화된 **'클라우드 디자인 패턴'** (Circuit Breaker, Sidecar, CQRS 등)이 더욱 중요해지고 있습니다. 또한 언어 자체의 발전(함수형 프로그래밍 지원 등)으로 일부 패턴은 언어의 기본 문법(예: Java의 Stream, Lambda)에 흡수되어 사라지거나 변형되는 추세입니다.
