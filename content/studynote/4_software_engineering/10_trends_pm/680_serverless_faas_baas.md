# 680. 서버리스 (Serverless) FaaS, BaaS

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 개발자가 서버의 프로비저닝, 관리, 확장을 신경 쓸 필요 없이 비즈니스 로직(코드)에만 집중할 수 있게 하는 **클라우드 컴퓨팅 실행 모델**이다.
> 2. **구성**: 로직 단위의 실행 환경인 **FaaS (Function as a Service)**와 데이터베이스, 인증 등 백엔드 기능을 API로 제공받는 **BaaS (Backend as a Service)**로 나뉜다.
> 3. **가치**: 실제 코드가 실행되는 시간 동안만 비용을 지불하는 **사용량 기반 과제(Pay-as-you-go)**와 무한한 자동 확장을 통해 운영 오버헤드를 획기적으로 줄인다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "서버가 없다는 뜻인가?"

서버리스(Serverless)는 물리적인 서버가 없다는 뜻이 아니라, 개발자가 서버의 **존재를 의식할 필요가 없다**는 뜻입니다. 과거에는 접속자가 늘어나면 서버를 늘리고(Scaling), OS 패치를 하는 등 부수적인 업무가 많았습니다. 서버리스는 이러한 '운영적 부담'을 클라우드 제공사(AWS, Azure, GCP)에게 완전히 위임합니다.

### 💡 비유: 자가용 운전 vs 카카오 택시 호출

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        서버리스(Serverless) 비유                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 기존 방식 (자가용):                                                      │
│     내가 직접 차를 사고(서버 구매), 기름 넣고(전력/인프라), 엔진 오일 갈고(관리)    │
│     운전해야 함. 안 타는 시간에도 주차비(기본 비용)가 나감.                      │
│                                                                             │
│  2. 서버리스 방식 (택시 호출):                                                │
│     내가 차를 관리할 필요 없음. 목적지에 가고 싶을 때 부르기만(Event 호출) 함.     │
│     택시 기사님(Cloud Provider)이 알아서 운전해주고, 나는 딱 **이동한 거리**만큼만  │
│     요금을 내면 끝! 이동 안 할 때는 돈이 전혀 안 듦.                             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. FaaS vs BaaS 상세 비교

| 항목 | FaaS (Function as a Service) | BaaS (Backend as a Service) |
|:---:|:---|:---|
| **정의** | 비즈니스 로직을 함수 단위로 배포 및 실행 | 서버측 기능을 API/SDK로 제공받아 사용 |
| **핵심 예시** | **AWS Lambda**, Google Cloud Functions | **Firebase**, AWS Cognito, DynamoDB |
| **제어권** | 개발자가 코드 로직을 직접 작성 | 제공된 서비스의 설정값만 조절 |
| **특징** | **상태 비저장(Stateless)**, 이벤트 기반 | 데이터 저장, 사용자 인증 등 공통 기능 |

### 2. 서버리스의 핵심 메커니즘
- **이벤트 기반 (Event-Driven)**: HTTP 요청, 파일 업로드, DB 변경 등이 트리거가 되어 함수 실행.
- **상태 비저장 (Stateless)**: 함수는 실행 후 즉시 소멸. 상태 정보는 외부 DB(BaaS)에 저장 필수.
- **Cold Start**: 오랫동안 호출되지 않던 함수를 실행할 때, 컨테이너를 띄우는 데 발생하는 약간의 지연 시간.

### 3. 서버리스 아키텍처 흐름도 (ASCII)

```text
    [ Event Source ] ──▶ [ Trigger ] ──▶ [ FaaS (Lambda) ] ──▶ [ BaaS (DB/Auth) ]
    (S3 파일 업로드)      (이벤트 감지)      (이미지 썸네일 생성)     (결과값 DB 저장)
                                              │
                                              └─▶ [ Output ]
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 가상머신(VM) vs 컨테이너(K8s) vs 서버리스(FaaS)

| 항목 | 가상머신 (IaaS) | 컨테이너 (CaaS/PaaS) | 서버리스 (FaaS) |
|:---:|:---|:---|:---|
| **추상화 단위** | 하드웨어 (OS 포함) | 애플리케이션 (Runtime 포함) | **함수 (Logic 전용)** |
| **확장 속도** | 분 단위 (느림) | 초 단위 | **밀리초 단위 (매우 빠름)** |
| **비용 모델** | 할당된 자원 시간당 과금 | 할당된 리소스 기반 과금 | **실제 실행 횟수/시간 과금** |
| **운영 부담** | 매우 높음 | 중간 | **거의 없음** |

### 2. 기술적 시너지: 마이크로서비스 (MSA)
서버리스는 MSA를 구현하는 가장 순수한 형태입니다. 각 기능을 독립된 함수로 배포함으로써 서비스 간의 결합도를 완벽히 분리할 수 있습니다. 단, 수천 개의 함수를 관리해야 하는 복잡성은 숙제로 남습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 일시적 대량 트래픽 처리 (이벤트 응모)
- **상황**: 1년에 단 하루, 특정 시간에만 100만 명의 유저가 몰리는 경품 이벤트 페이지.
- **결단**: **서버리스 아키텍처 채택**. 
- **판단**: 평소에는 서버를 하나도 띄우지 않아 비용 0원 유지. 이벤트 시작 시 수만 개의 람다 함수가 동시 기동하여 트래픽 처리. 인프라 증설 계획(Capacity Planning) 없이도 완벽 대응.

### 📢 기술사적 결언
> "서버리스는 **'운영의 추상화'**를 완성한 모델이다. 하지만 '만능 열쇠'는 아니다. 실행 시간이 길거나(Long-running), 실시간 응답이 매우 중요하여 Cold Start를 견딜 수 없는 시스템에는 부적합하다. 아키텍트는 비즈니스 로직의 **'실행 패턴'**과 **'경제성'**을 면밀히 분석하여, 컨테이너 기반의 상시 운영과 서버리스 기반의 이벤트 처리를 적절히 혼합하는 하이브리드 전략을 구사해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **비용 절감**: 유휴 자원 낭비 제거로 평균 인프라 비용 40~70% 절감.
- **생산성**: 인프라 관리 시간 0화로 비즈니스 로직 개발 집중도 향상.

### 미래 전망
미래의 서버리스는 **'Edge Computing'**과 결합하여 전 세계 유저와 가장 가까운 CDN 노드에서 로직이 실행되는 형태로 진화하고 있습니다. 또한 Cold Start 문제를 머신러닝 기반의 '예측적 인스턴스 예열' 기술로 해결함으로써, 미션 크리티컬한 실시간 서비스 영역까지 서버리스의 영토가 확장될 것입니다.
