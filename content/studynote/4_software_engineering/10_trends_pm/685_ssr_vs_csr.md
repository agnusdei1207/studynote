# 685. 서버 사이드 렌더링 (SSR) vs CSR

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 웹 페이지를 화면에 그리는 과정인 렌더링을 서버에서 수행하여 완성된 HTML을 보내느냐(**SSR**), 혹은 브라우저에서 자바스크립트를 통해 동적으로 생성하느냐(**CSR**)의 차이다.
> 2. **트레이드오프**: **SSR**은 초기 로딩 속도가 빠르고 검색 엔진 최적화(SEO)에 유리하지만 서버 부하가 크고, **CSR**은 초기 로딩은 느리나 이후 전환이 매끄럽고 사용자 상호작용(UX)이 뛰어나다.
> 3. **가치**: 서비스의 성격(콘텐츠 중심 vs 앱 중심)에 따라 적절한 렌더링 전략을 선택하고, 최근에는 Next.js와 같은 프레임워크를 통해 두 방식의 장점을 결합한 하이브리드 방식이 주류를 이룬다.

---

## Ⅰ. 개요 (Context & Background)

### 웹 렌더링의 변천사

초기 웹은 서버에서 모든 것을 그려서 보내주는 **SSR** 방식이 전부였습니다. 이후 스마트폰의 등장과 브라우저 성능 향상으로 PC 앱처럼 매끄러운 동작을 제공하는 **CSR** (Single Page Application, SPA)이 대세가 되었습니다. 하지만 CSR의 느린 첫 화면과 검색 봇이 내용을 읽지 못하는 문제(SEO)가 부각되면서, 다시 SSR의 장점을 현대적으로 재해석하려는 움직임이 활발해졌습니다.

### 💡 비유: 배달 음식 vs 밀키트 요리

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        웹 렌더링 방식 비유                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. SSR (서버 사이드 - 배달 음식):                                            │
│     식당(서버)에서 요리를 다 해서 그릇에 담아 배달해줌.                           │
│     - 장점: 받자마자 바로 먹을 수 있음 (빠른 초기 화면).                         │
│     - 단점: 식당 주인이 바쁘면 배달이 늦고, 그릇을 매번 새로 받아야 함.            │
│                                                                             │
│  2. CSR (클라이언트 사이드 - 밀키트):                                         │
│     식당에서는 재료(Data)와 설명서(JS)만 보내주고, 내가 직접 조리함.               │
│     - 장점: 한 번 조리법을 익히면 내 맘대로 빠르게 요리 가능 (매끄러운 UX).        │
│     - 단점: 처음 재료를 받고 조리 기구를 세팅하는 데 시간이 걸림 (초기 로딩 지연). │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. SSR (Server-Side Rendering) 작동 방식
1. 브라우저가 서버에 페이지 요청.
2. 서버는 DB에서 데이터를 가져와 **완성된 HTML**을 생성.
3. 브라우저는 전달받은 HTML을 즉시 화면에 그림 (FP - First Paint 빠름).
4. 이후 JS 파일이 로드되면 상호작용 가능 (TBT - Total Blocking Time 발생 가능).

### 2. CSR (Client-Side Rendering) 작동 방식
1. 브라우저가 서버에 요청. 서버는 **빈 HTML**과 대용량 JS 전달.
2. 브라우저가 JS를 실행하여 화면 틀을 만듦 (이때까지 하얀 화면).
3. JS가 API 서버에서 데이터를 가져와 화면을 채움.
4. 모든 로드 완료 후 상호작용 및 페이지 이동 가능.

### 3. 렌더링 흐름 비교 (ASCII)

```text
    [ SSR Flow ]                                [ CSR Flow ]
    
    Server -> HTML(Content) -> Browser          Server -> HTML(Empty) -> Browser
               │                                           │
               ▼                                           ▼
    [ 화면 즉시 표시 ]                           [ 하얀 화면 (Loading) ]
               │                                           │
               ▼                                           ▼
    Server -> JavaScript -> Browser              Server -> JavaScript -> Browser
               │                                           │
               ▼                                           ▼
    [ 상호작용 가능 ]                            [ 화면 렌더링 & 상호작용 ]
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. SSR vs CSR 상세 비교

| 항목 | SSR (전통적/Next.js 등) | CSR (React, Vue, Angular) |
|:---:|:---|:---|
| **초기 로딩 속도** | **빠름** (HTML 미리 생성) | 느림 (JS 실행 대기) |
| **페이지 이동** | 느림 (매번 서버 호출) | **매우 빠름** (부분 갱신) |
| **SEO (검색 엔진)** | **매우 우수** | 미흡 (봇이 JS 실행 못 할 경우) |
| **서버 부하** | 높음 (서버가 렌더링 전담) | 낮음 (클라이언트가 렌더링) |
| **사용자 경험** | 투박함 (깜빡임 발생) | 매끄러움 (앱 같은 느낌) |

### 2. 기술적 시너지: Hydration
현대적인 프레임워크(Next.js, Nuxt.js)는 SSR로 첫 화면을 빠르게 보여준 뒤, 브라우저에서 JS가 실행되면서 이벤트 리스너를 연결하는 **Hydration** 과정을 거칩니다. 이를 통해 SSR의 속도와 CSR의 동적인 UX를 동시에 달성합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 서비스 성격에 따른 선택
- **뉴스, 블로그, 이커머스 상품 페이지**: 검색 결과에 잘 노출되어야 하고 초기 로딩이 매출과 직결되므로 **SSR**이 필수적임.
- **사내 관리 시스템, 대시보드, 메신저**: 로그인 후 사용하며 검색 노출이 필요 없고, 잦은 화면 전환과 복잡한 상호작용이 중요하므로 **CSR**이 유리함.
- **판단**: "Contents is King"이라면 SSR, "Interaction is King"이라면 CSR을 선택하되, 최근 트렌드는 중요한 첫 페이지만 SSR로 처리하는 **부분적 SSR** 전략이 대세임.

### 📢 기술사적 결언
> "렌더링 전략의 선택은 **'성능 지표(LCP, FID)'**와 **'비즈니스 목표'** 사이의 최적점을 찾는 과정이다. 무조건 최신 SPA 기술이 좋다는 맹신을 버리고, 사용자가 우리 서비스를 만나는 첫 접점이 어디인지 분석해야 한다. 결국 SSR과 CSR은 대립 관계가 아니라, 웹의 접근성과 성능을 극대화하기 위해 상호 보완적으로 배치해야 할 **아키텍처적 카드**들이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **이탈률 감소**: SSR 적용 시 초기 로딩 속도 향상으로 사용자 이탈률 20~30% 감소.
- **마케팅 지표**: 검색 엔진 랭킹 상승으로 인한 유기적 트래픽 유입 증대.

### 미래 전망
최근에는 **Server Components (RSC)** 기술이 등장하여, 서버에서 렌더링하는 컴포넌트와 클라이언트에서 실행되는 컴포넌트를 코드 레벨에서 명확히 분리하고 있습니다. 또한 브라우저의 성능 한계를 극복하기 위해 **Edge Rendering** (유저와 가까운 서버에서 렌더링) 기술이 보편화되면서, SSR과 CSR의 경계가 완전히 허물어지는 '유비쿼터스 렌더링' 시대로 진화하고 있습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **SSR**은 만화책을 살 때 **그림이 다 그려진 책**을 사는 거라, 사자마자 바로 읽을 수 있어요.
2. **CSR**은 **색칠 공부 책**을 사는 거라, 내가 직접 색칠을 다 해야 예쁜 그림(화면)을 볼 수 있죠.
3. 처음엔 만화책이 편하지만, 내가 직접 색칠하는 건 **내 맘대로 바꿀 수 있어서** 나중에는 더 재미있게 놀 수 있답니다!
