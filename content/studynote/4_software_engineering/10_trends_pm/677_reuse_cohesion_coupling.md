# 677. 소프트웨어 재사용 및 모듈화 응집도/결합도

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템을 독립적인 단위인 **모듈(Module)**로 분할하여 복잡도를 관리하고, 한 번 검증된 로직을 반복적으로 활용하여 개발 효율과 품질을 높이는 **소프트웨어 설계의 핵심 원리**이다.
> 2. **설계 목표**: 모듈 내부 요소들이 얼마나 밀접하게 관련되어 있는지를 나타내는 **응집도(Cohesion)는 높이고**, 모듈 간의 상호 의존성을 나타내는 **결합도(Coupling)는 낮추는 것**이 이상적이다.
> 3. **가치**: 변경의 영향 범위를 국소화(Localization)하여 유지보수성을 극대화하고, 신규 프로젝트 구축 시 기존 자산을 활용함으로써 Time-to-Market을 단축한다.

---

## Ⅰ. 개요 (Context & Background)

### 모듈화(Modularity)의 철학

"나누어 정복하라(Divide and Conquer)". 거대한 소프트웨어를 하나의 통짜 코드로 짜는 것은 불가능합니다. 시스템을 의미 있는 기능 단위로 쪼개고, 그 부품들이 서로의 내부 사정을 모른 채 정해진 약속(Interface)으로만 소통하게 만드는 것이 좋은 설계의 시작입니다. 이 과정에서 설계의 품질을 측정하는 잣대가 바로 응집도와 결합도입니다.

### 💡 비유: 레고 블록 vs 찰흙 인형

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        응집도와 결합도 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 좋은 설계 (레고 블록):                                                   │
│     - 응집도 높음: 블록 하나는 그 자체로 완벽한 모양과 기능을 가짐.              │
│     - 결합도 낮음: 블록과 블록은 홈(Interface)으로만 연결됨. 하나를 빼거나       │
│       바꿔도 다른 블록이 부서지지 않음. (재사용성 최고)                          │
│                                                                             │
│  2. 나쁜 설계 (찰흙 인형):                                                   │
│     - 응집도 낮음: 팔다리가 몸통에 대충 붙어 있고 경계가 모호함.                 │
│     - 결합도 높음: 팔 하나를 떼어내려 하면 어깨와 몸통까지 다 뭉개짐.            │
│       일부분만 따로 떼어서 다른 인형에 붙이는 것(재사용)이 불가능함.              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 응집도 (Cohesion) : 내부의 결속력 (높을수록 좋음)

| 단계 | 유형 | 설명 | 품질 |
|:---:|:---|:---|:---|
| **High** | **기능적 (Functional)** | 모듈 내부의 모든 요소가 하나의 기능만을 수행 | **최상** |
| | **순차적 (Sequential)** | 한 요소의 출력이 다음 요소의 입력으로 사용됨 | 우수 |
| | **통신적 (Communicational)** | 동일한 입출력 데이터를 사용하여 서로 다른 기능 수행 | 양호 |
| | **절차적 (Procedural)** | 정해진 순서에 따라 여러 기능을 순차적으로 수행 | 보통 |
| | **시간적 (Temporal)** | 특정 시간에 처리되어야 하는 활동들을 모아둠 | 낮음 |
| | **논리적 (Logical)** | 유사한 성격의 활동들을 논리적으로 묶음 | 낮음 |
| **Low** | **우연적 (Coincidental)** | 아무런 관련 없는 활동들이 모여 있음 | **최악** |

### 2. 결합도 (Coupling) : 외부와의 의존성 (낮을수록 좋음)

| 단계 | 유형 | 설명 | 품질 |
|:---:|:---|:---|:---|
| **Low** | **자료 (Data)** | 파라미터를 통해서만 데이터를 주고받음 | **최상** |
| | **스탬프 (Stamp)** | 모듈 간에 데이터 구조(객체 등)를 전달함 | 우수 |
| | **제어 (Control)** | 제어 신호(Flag)를 전달하여 상대의 로직을 결정함 | 양호 |
| | **외부 (External)** | 외부 소프트웨어나 통신 프로토콜 공유 | 보통 |
| | **공통 (Common)** | 전역 변수(Global Variable)를 여러 모듈이 공유 | 낮음 |
| **High** | **내용 (Content)** | 다른 모듈의 내부 데이터나 로직을 직접 참조 | **최악** |

### 3. 설계 메커니즘 (ASCII)

```text
    [ Low Cohesion / High Coupling ]      [ High Cohesion / Low Coupling ]
    (나쁜 설계: 스파게티)                   (좋은 설계: 컴포넌트)
    
       Module A      Module B                Module A      Module B
      ┌────────┐    ┌────────┐              ┌────────┐    ┌────────┐
      │ 로직1  │───▶│ 로직3  │              │ 로직1  │    │ 로직3  │
      │ 로직2  │◀───│ 로직4  │              │ 로직2  │    │ 로직4  │
      └────────┘    └────────┘              └────┬───┘    └────┬───┘
          │ (직접 참조) │                          │ (API)      │
          ▼            ▼                          ▼            ▼
      [ Global Data Store ]                 [ Standard Interface ]
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 정보 은닉(Information Hiding)과 캡슐화
- **응집도**를 높이기 위해서는 캡슐화를 통해 관련 데이터를 한곳에 모아야 합니다.
- **결합도**를 낮추기 위해서는 정보 은닉을 통해 내부 구현을 감추고 인터페이스만 노출해야 합니다.

### 2. 기술적 시너지: 마이크로서비스 (MSA)
MSA는 결합도를 낮추기 위한 궁극의 아키텍처입니다. 서비스 간의 메모리 공유를 원천 차단하고 오직 네트워크 API로만 통신하게 함으로써 '자료 결합도' 수준의 독립성을 강제합니다. 반면, 서비스 내부적으로는 '기능적 응집도'를 갖도록 도메인 단위로 쪼개는 것이 설계의 핵심입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 전역 변수 지옥 탈출
- **상황**: 수백 개의 함수가 `GlobalConfig` 객체를 직접 수정하여, 값 하나가 바뀌면 시스템 어디가 터질지 모름 (공통 결합도).
- **결단**: **리팩토링 실시**. 
- **판단**: 전역 변수를 제거하고, 필요한 데이터만 파라미터로 넘기는 '자료 결합도' 방식으로 전환. 설정값 변경이 필요한 경우 별도의 `ConfigService`를 통해 캡슐화된 메서드로만 접근하게 하여 응집도 향상.

### 📢 기술사적 결언
> "응집도와 결합도는 단순한 수치가 아니라 **'변경에 대한 태도'**다. 응집도가 높은 모듈은 수정 시 '나만 고치면 되는 확신'을 주고, 결합도가 낮은 모듈은 수정 시 '남이 안 깨질 거라는 안심'을 준다. 아키텍처 설계 시 이 두 가지 잣대를 항상 들이대어, 시스템이 거대한 아메바처럼 엉겨 붙지 않도록 끊임없이 감시해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **유지보수성**: 변경 영향도 분석 시간 70% 단축.
- **테스트 용이성**: 낮은 결합도 덕분에 Mock 객체 활용이 쉬워져 테스트 커버리지 90% 이상 확보 가능.

### 미래 전망
미래의 소프트웨어 공학은 **'자율적 모듈화'**로 나아가고 있습니다. AI가 코드를 분석하여 결합도가 높은 구간을 자동으로 찾아내고, 응집도를 높일 수 있는 클래스 분할 방안을 제안하는 지능형 설계 도구들이 발전하고 있습니다. 또한 서버리스(Serverless) 환경처럼 함수 단위의 극단적인 모듈화가 보편화되면서, 응집도와 결합도의 원칙은 더욱 중요해질 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **응집도**는 가방 안이 얼마나 잘 정리되어 있느냐 하는 거예요. 필통에는 필기구만, 도시락통에는 음식만 들어있는 게 응집도가 높은 거죠.
2. **결합도**는 친구와 내 손이 끈적끈적한 껌으로 붙어 있느냐 하는 거예요. 껌으로 붙어 있으면(결합도 높음) 친구가 화장실 갈 때 나도 끌려가야 해서 불편하답니다.
3. 내 가방을 잘 정리하고, 친구와 적당한 거리를 유지해야 **서로가 편하고 자유롭게** 놀 수 있는 것과 같아요!
