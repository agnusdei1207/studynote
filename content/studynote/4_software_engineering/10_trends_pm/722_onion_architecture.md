# 722. 어니언 아키텍처 도메인 코어 격리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 제프리 팔레르모(Jeffrey Palermo)가 제안한 아키텍처로, 시스템의 중심에 **도메인 모델(Domain Core)**을 두고 의존성이 밖에서 안으로만 향하게 하여 외부 환경의 변화로부터 핵심 로직을 완벽히 격리하는 **계층형 아키텍처의 진화 모델**이다.
> 2. **제어 흐름**: 데이터베이스나 외부 서비스가 중심이 아닌, **도메인 엔티티와 서비스가 중심**이 되어 인터페이스를 정의하고, 외부 계층이 이를 구현(DIP)함으로써 '기술'이 '비즈니스'에 종속되게 만든다.
> 3. **가치**: 인프라(DB, UI, API)를 단순한 '세부 사항'으로 취급하여 교체가 용이하게 하며, 핵심 도메인 로직을 순수하게 유지하여 시스템의 장기적인 유연성과 테스트 가능성을 확보한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "데이터베이스 중심 설계의 탈피"

전통적인 계층형 아키텍처는 DB 계층이 가장 밑바닥에 있고 모든 계층이 그 위에 쌓이는 구조였습니다. 이는 DB가 바뀌면 시스템 전체가 흔들리는 결과를 초래했습니다. **어니언 아키텍처 (Onion Architecture)**는 이를 뒤집어, 양파의 심장부처럼 가장 안쪽에 비즈니스의 본질(도메인)을 두고, 나머지는 껍질처럼 감싸는 구조를 취합니다.

### 💡 비유: 양파의 구조와 보호

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        어니언 아키텍처 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 아주 소중한 "양파 씨앗(도메인 로직)"을 심어서 키우려 함.                 │
│                                                                             │
│  1. 도메인 코어 (씨앗): 양파의 가장 깊숙한 핵심. 비즈니스의 진짜 가치.            │
│  2. 도메인 서비스 (속껍질): 씨앗을 도와주는 도우미들.                          │
│  3. 애플리케이션 서비스 (중간껍질): 외부에서 씨앗을 쓰기 위한 통로.              │
│  4. 인프라/UI (겉껍질): 세상과 맞닿는 부분. 언제든 마르거나 벗겨질 수 있음.       │
│                                                                             │
│  → 겉껍질이 상하거나 바뀐다고 해서(DB 교체), 안쪽의 씨앗이 변하지 않도록          │
│     철저하게 감싸서 보호하는 것!                                               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 어니언 아키텍처의 계층 구조

| 계층 | 명칭 | 포함 요소 | 특징 |
|:---:|:---|:---|:---|
| **Center** | **Domain Model** | 엔티티(Entity), 값 객체(VO) | 비즈니스의 본질, 의존성 0 |
| **Inner** | **Domain Services** | 도메인 로직 처리를 위한 인터페이스 | 모델을 활용한 순수 로직 |
| **Middle** | **Application Services** | 유스케이스 로직, 오케스트레이션 | 시스템의 동작 정의 |
| **Outer** | **Infrastructure / UI** | DB 구현체, 웹 컨트롤러, 외부 API | **언제든 바뀔 수 있는 부속품** |

### 2. 의존성 역전 원칙 (DIP)의 적용
어니언 아키텍처에서 모든 계층은 중심 방향(Core)의 인터페이스만 알면 됩니다.
- **기존**: 애플리케이션 계층 ──▶ 데이터베이스 계층 (직접 의존)
- **어니언**: 애플리케이션 계층 ──▶ 도메인 인터페이스 ◀── 데이터베이스 구현체 (역전)

### 3. 아키텍처 시각화 (ASCII)

```text
    ┌─────────────────────────────────────────────────────────────┐
    │  Outer Ring: Infrastructure, UI, Tests, External Services   │
    │  ┌───────────────────────────────────────────────────────┐  │
    │  │  Middle Ring: Application Services (Use Cases)        │  │
    │  │  ┌───────────────────────────────────────────────┐    │  │
    │  │  │  Inner Ring: Domain Services (Interfaces)      │    │  │
    │  │  │  ┌───────────────────────────────────────┐    │    │  │
    │  │  │  │      CORE: Domain Model (Entities)    │    │    │  │
    │  │  │  └───────────────────────────────────────┘    │    │  │
    │  │  └───────────────────────────────────────────────┘    │  │
    │  └───────────────────────────────────────────────────────┘  │
    └───────────────────────────┬─────────────────────────────────┘
                                │
                    의존성 방향: Outside-In (Inside-Out으로만 향함)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 클린 아키텍처 vs 어니언 아키텍처 vs 헥사고날
- **공통점**: 도메인 중심, 의존성 역전(DIP)을 통한 관심사 분리.
- **어니언**: 계층을 '원'으로 표현하며, 도메인 모델의 순수성을 강조.
- **헥사고날**: '포트와 어댑터'를 통해 외부 연동의 대칭성을 강조.
- **클린**: 계층 간 데이터 전달을 위한 '경계(Boundary)'를 구체적으로 명시.

### 2. 기술적 시너지: 도메인 주도 설계 (DDD)
어니언 아키텍처는 DDD를 구현하기 위한 최적의 '그릇'입니다. 바운디드 컨텍스트(Bounded Context) 내부를 어니언 구조로 설계함으로써, 비즈니스 언어와 기술 언어를 완벽히 격리할 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: DB 기술 스택의 불확실성 대응
- **상황**: 프로젝트 초기인데 RDBMS를 쓸지 NoSQL을 쓸지 결정되지 않음.
- **결단**: **어니언 아키텍처 기반 인터페이스 설계**. 
- **판단**: 도메인 서비스 계층에 `UserRepository` 인터페이스만 정의하고 개발 착수. 모든 비즈니스 로직은 이 인터페이스만 바라보고 코딩.
- **효과**: 개발 중반에 DB가 결정되었을 때, 인프라 계층에 해당 구현체만 추가하여 즉시 연동. 기존 로직 수정 0%.

### 📢 기술사적 결언
> "어니언 아키텍처는 **'비즈니스의 영속성'**을 보장하는 설계다. 기술은 유행에 따라 변하지만, 비즈니스 규칙은 상대적으로 오래 지속된다. 아키텍트는 핵심 로직이 특정 라이브러리나 기술 스택에 오염되지 않도록 **'심장부(Core)'**를 성역화해야 한다. '어디에 저장하는가'보다 '무엇을 하는가'가 먼저 코드로 표현될 때, 비로소 진화 가능한 소프트웨어가 탄생한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **코드 재사용성**: 도메인 모델의 이식성 증대로 멀티 플랫폼(Web/Mobile) 대응 비용 절감.
- **테스트 용이성**: 인프라 없이도 핵심 로직 검증 가능 (Mocking 최소화).

### 미래 전망
미래의 아키텍처는 **'AI-Managed Core'**로 발전할 것입니다. 도메인 모델의 비즈니스 규칙을 AI가 이해하고, 외부 인터페이스 변화에 맞춰 어댑터 코드를 자동으로 생성해 주는 수준에 도달할 것입니다. 또한 서버리스 환경에서 각 '양파의 겹'들이 독립적인 마이크로 실행 단위로 배포되는 극단적인 분산 구조가 연구되고 있습니다.

2. 겉 포장지(UI/DB)가 찢어지거나 색깔이 바뀌어도, 안쪽 상자 속에 든 **진짜 선물(비즈니스 로직)**은 항상 깨끗하고 안전하죠.
