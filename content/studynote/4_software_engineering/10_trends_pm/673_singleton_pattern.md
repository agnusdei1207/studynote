# 673. 싱글톤 패턴 단일 인스턴스 보장

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클래스의 인스턴스가 오직 하나만 생성되도록 보장하고, 어디서든 이 인스턴스에 접근할 수 있는 글로벌 접근점을 제공하는 **생성(Creational) 패턴**이다.
> 2. **제어**: 생성자를 **Private**으로 제한하여 외부에서의 임의 생성을 막고, 정적(Static) 메서드를 통해 내부에 미리 생성된 동일한 객체만을 반환하도록 관리한다.
> 3. **가치**: 커넥션 풀, 로그 기록기, 환경 설정 등 시스템 전체에서 공유되어야 하는 자원의 중복 생성을 방지하여 **메모리 낭비를 줄이고 데이터 일관성**을 유지한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "왜 하나만 있어야 하는가?"

프로그램이 돌아가는 동안 설정 파일(Config)을 읽는 객체가 100개나 생길 필요가 있을까요? 데이터베이스 연결을 관리하는 객체가 매번 새로 생기면 서버 자원은 금방 고갈될 것입니다. **싱글톤(Singleton)**은 "이 객체는 시스템에 딱 하나만 존재한다"는 것을 문법적으로 강제하여 관리의 효율성을 높입니다.

### 💡 비유: 마을의 하나뿐인 공중전화

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        싱글톤 패턴 비유                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 마을 사람들이 외부와 연락을 해야 함.                                     │
│                                                                             │
│  1. 일반적인 방식: 연락하고 싶은 사람마다 집에 전화기를 새로 삼.                 │
│     (돈도 많이 들고 전화번호 관리도 안 됨)                                      │
│                                                                             │
│  2. 싱글톤 방식: 마을 광장에 딱 하나뿐인 '공중전화(Singleton)'를 설치함.         │
│     누구든지 연락이 필요하면 광장으로 와서 그 전화기를 사용함.                  │
│     전화기는 하나이므로 유지비도 적고 관리가 매우 쉬움!                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 싱글톤 패턴의 3대 구성 요소
1. **Private Constructor**: 외부에서 `new` 키워드로 인스턴스를 생성할 수 없게 차단.
2. **Static Instance**: 클래스 내부에서 유일한 인스턴스를 보관하는 정적 변수.
3. **Static Method**: 외부에서 유일한 인스턴스를 얻을 수 있는 유일한 통로 (예: `getInstance()`).

### 2. 구현 방식 및 고려 사항 (Java 예시)

| 방식 | 특징 | 장단점 |
|:---:|:---|:---|
| **Eager Initialization** | 클래스 로딩 시 즉시 생성 | 단순하지만, 사용하지 않아도 메모리 점유 |
| **Lazy Initialization** | 요청 시점에 생성 | 자원 효율적이지만, **멀티스레드 위험** |
| **Thread Safe (Lazy)** | `synchronized` 키워드 사용 | 안전하지만, 성능 저하 발생 |
| **Bill Pugh Solution** | Static Inner Class 활용 | **가장 권장됨** (성능 + Thread Safe) |

### 3. 싱글톤 구조도 (ASCII)

```text
    ┌─────────────────────────────┐
    │          Singleton          │
    ├─────────────────────────────┤
    │ - instance: Singleton       │ <── Private Static 변수
    ├─────────────────────────────┤
    │ - Singleton()               │ <── Private 생성자
    │ + getInstance(): Singleton  │ <── Public Static 메서드
    └──────────────┬──────────────┘
                   │
                   ▼
    [ Client ] ──▶ "Singleton.getInstance()" 호출 ──▶ 유일한 객체 반환
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 전역 변수 vs 싱글톤
- **전역 변수**: 이름은 하나지만 실수로 다른 값을 덮어쓸 수 있고, 언제 초기화될지 제어하기 힘듭니다.
- **싱글톤**: 객체 생성 시점을 조절할 수 있고(Lazy), 상속이나 인터페이스 구현 등 객체지향의 이점을 그대로 누릴 수 있습니다.

### 2. 기술적 시너지: Spring Framework
스프린트 컨테이너는 기본적으로 모든 빈(Bean)을 **싱글톤 스코프**로 관리합니다. 개발자가 복잡한 싱글톤 코드를 직접 짤 필요 없이, 프레임워크가 객체의 단일성과 생명주기를 책임져 줍니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 애플리케이션 로그 매니저
- **상황**: 수많은 클래스에서 로그를 남기려 함. 각자 로그 파일을 열면 파일 락(Lock) 충돌 발생.
- **결단**: **싱글톤 로그 매니저 도입**.
- **판단**: 모든 클래스가 동일한 `Logger.getInstance()`를 통해 메시지를 보냄으로써, 하나의 파일에 순차적으로 안전하게 로그를 기록함.

### 📢 기술사적 결언
> "싱글톤은 가장 이해하기 쉬운 패턴인 동시에 가장 남용되기 쉬운 패턴이다. 단일 인스턴스를 보장한다는 장점이 있지만, 전역 상태를 만들기 때문에 **객체 간의 결합도(Coupling)**를 높이고 단위 테스트를 어렵게 만드는 주범이 되기도 한다. 따라서 무조건적인 싱글톤 도입보다는, 의존성 주입(DI) 프레임워크를 통해 관리되는 싱글톤을 활용하는 것이 현대적 아키텍처의 정석이다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **메모리 절감**: 빈번하게 사용되는 무거운 객체의 중복 생성 억제로 Heap 메모리 사용량 최적화.
- **성능 향상**: 객체 생성 및 가비지 컬렉션(GC) 부하 감소.

### 미래 전망
최근에는 마이크로서비스(MSA)와 컨테이너 환경이 확산되면서, '단일 프로세스 내의 싱글톤'을 넘어 '클러스터 내의 유일성'을 보장해야 하는 요구가 늘고 있습니다. 이를 위해 Redis나 ZooKeeper를 활용한 **분산 락(Distributed Lock)** 기반의 싱글톤 개념이 클라우드 아키텍처 설계의 중요한 화두가 되고 있습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **싱글톤**은 학교에서 딱 하나뿐인 **"교장 선생님의 도장"**과 같아요.
2. 선생님들마다 도장을 따로 가지고 있으면 혼란스러우니까, 중요한 서류에는 오직 그 **하나뿐인 도장**만 찍기로 약속한 거죠.
3. 도장이 어디 있는지 모두가 알고 있어서, 필요할 때마다 그곳에 가서 빌려 쓰는 규칙이랍니다!
