# 700. 서버리스 FaaS 아키텍처 제약

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트에 의해 기동되고 실행 완료 후 즉시 소멸하는 **상태 비저장(Stateless) 함수** 단위의 컴퓨팅 모델로, 운영 오버헤드를 극단적으로 줄이지만 설계상 엄격한 제약 조건을 동반한다.
> 2. **주요 제약**: 오랫동안 호출되지 않은 함수 실행 시 발생하는 **콜드 스타트(Cold Start)** 지연, 짧은 최대 실행 시간(Timeout), 메모리 및 임시 저장 공간의 물리적 한계가 존재한다.
> 3. **가치**: 제약 사항을 극복하기 위한 비동기 설계와 상태 외부화(BaaS 활용)를 통해, 인프라 관리 부담 없이 무한 확장이 가능한 **경제적·탄력적 클라우드 아키텍처**를 실현한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "자유에는 책임이 따른다"

서버리스(FaaS)는 개발자에게 "서버 관리에서의 해방"이라는 엄청난 자유를 주었습니다. 하지만 이 자유를 누리기 위해서는 '서버가 언제든 사라질 수 있고, 자원이 넉넉하지 않다'는 현실적인 제약을 받아들여야 합니다. 700번 키워드에서는 서버리스를 성공적으로 도입하기 위해 반드시 고려해야 할 기술적 한계들을 짚어봅니다.

### 💡 비유: 팝업 스토어의 운영 규칙

```text

│                                                                             │
│  [상황] 백화점 한복판에 1일 단기 팝업 스토어(FaaS)를 열기로 함.                  │
│                                                                             │
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. FaaS의 4대 핵심 기술적 제약

| 제약 항목 | 상세 내용 | 아키텍처적 영향 |
|:---:|:---|:---|
| **Cold Start** | 비활성 상태의 컨테이너를 로드하는 데 걸리는 지연 시간 | 실시간 응답(Low Latency) 서비스에 부적합 |
| **Execution Time** | 보통 5분~15분 이내로 실행 제한 (AWS Lambda 15분) | 대용량 배치 처리나 무거운 연산 수행 불가 |
| **Statelessness** | 로컬 파일이나 메모리에 데이터 저장 불가 | 모든 상태를 외부 캐시(Redis)나 DB에 저장 필수 |
| **Resource Limit** | 할당 가능한 메모리(최대 10GB) 및 패키지 크기 제한 | 무거운 라이브러리나 딥러닝 모델 탑재 어려움 |

### 2. 콜드 스타트(Cold Start) 메커니즘 (ASCII)

```text
    [ Event ] ──▶ [ Find Worker ] ──▶ [ Download Code ] ──▶ [ Start Runtime ]
                                                                    │
          ▲ (이 구간이 Cold Start: 수백ms ~ 수초)                     │
          └─────────────────────────────────────────────────────────┘
                                     ▼
    [ Run Handler ] ──▶ [ Result ] ──▶ [ Keep Warm (Pause) ] ──▶ [ Stop ]
                                              (Next Init is Warm Start!)
```

### 3. 제약 극복을 위한 설계 전략
- **Warming**: 주기적으로 가짜 요청을 보내 인스턴스를 유지 (Provisioned Concurrency).
- **Step Functions**: 긴 작업을 여러 개의 작은 함수로 쪼개어 워크플로우로 연결.
- **BaaS 연동**: 데이터 저장은 DynamoDB, 메시지 전달은 SQS 등 전용 서비스 활용.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 컨테이너(K8s) vs 서버리스(FaaS) 제약 관점

| 항목 | 컨테이너 (K8s Pod) | 서버리스 (FaaS) |
|:---:|:---|:---|
| **상태 유지** | 로컬 디스크/메모리 일부 활용 가능 | **완전 불가능 (Ephemeral)** |
| **기동 지연** | 거의 없음 (항상 기동 중) | **발생 가능 (Cold Start)** |
| **확장 방식** | 수동/자동 설정 (HPA) | **완전 자동 (Request 단위)** |
| **비용** | 유휴 시간에도 비용 발생 | **실제 실행 시간만 과금** |

### 2. 기술적 시너지: 이벤트 기반 아키텍처 (EDA)
FaaS는 제약이 많지만, **이벤트 기반 설계**와 결합하면 무서운 폭발력을 냅니다. 짧게 치고 빠지는 함수의 특성이 '비동기 이벤트 처리'와 완벽히 일치하기 때문입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 이미지 썸네일 자동 생성 시스템
- **상황**: 유저가 올리는 사진의 크기가 제각각이고 언제 몰릴지 모름.
- **결단**: **FaaS 기반 비동기 처리**. 
- **판단**: 사진 업로드(S3 이벤트) → 람다 실행 → 썸네일 생성 → S3 저장. 
- **효과**: 실행 시간이 1초 내외이므로 Timeout 제약 무관. 상태는 S3에 저장되므로 Stateless 제약 극복. 사용량만큼만 비용 지불하여 비용 최적화.

### 📢 기술사적 결언
> "FaaS의 제약은 단점이 아니라 **'설계의 가이드라인'**이다. Stateless를 강제함으로써 시스템의 수평 확장을 원천적으로 보장하고, 실행 시간을 제한함으로써 비효율적인 로직을 쪼개도록 유도한다. 아키텍트는 서버리스의 한계를 명확히 인지하고, **'Cold Start가 비즈니스 크리티컬한가?'**와 **'함수 간 의존성 뒤엉킴(Nano-services)을 어떻게 막을 것인가?'**를 깊이 고민해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **운영 비용**: 서버 유지보수 인건비 및 유휴 서버 비용 50% 이상 절감.
- **유연성**: 트래픽 0에서 수만 TPS까지 1분 내 자동 대응.

### 미래 전망
미래의 서버리스는 **'WebAssembly (Wasm)'**와 결합하여 Cold Start 문제를 밀리초(ms) 단위로 해결할 것입니다. 또한 **'Stateful Serverless'** 기술의 발전으로 함수의 상태를 인프라 레이어에서 자동으로 관리해 주는 모델이 등장할 것입니다. 이제 서버리스는 '단순 작업용'을 넘어 엔터프라이즈의 '메인 아키텍처'로 확고히 자리 잡을 것입니다.
