# 675. 전략 패턴 알고리즘 교체

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동일한 목적을 가진 여러 알고리즘군을 정의하고, 각각을 캡슐화하여 런타임에 필요에 따라 자유롭게 교체하여 사용할 수 있게 만드는 **행위(Behavioral) 패턴**이다.
> 2. **OCP 준수**: 알고리즘을 사용하는 클라이언트 코드를 수정하지 않고도 새로운 전략을 추가할 수 있어, **개방-폐쇄 원칙(Open-Closed Principle)**을 가장 잘 구현하는 패턴 중 하나다.
> 3. **가치**: 복잡한 조건문(if-else, switch)을 제거하고 객체 간의 결합도를 낮추어 코드의 가독성과 유지보수성을 획기적으로 향상시킨다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "변하는 것과 변하지 않는 것의 분리"

하나의 기능을 수행하는 방법이 여러 가지일 때(예: 결제 수단 선택, 길 찾기 경로 선택), 이를 코드 내부에 조건문으로 길게 나열하면 코드가 비대해지고 수정이 힘들어집니다. **전략(Strategy)** 패턴은 이 '방법(알고리즘)' 자체를 별도의 객체로 분리하여 관리함으로써 변화에 유연하게 대응합니다.

### 💡 비유: 게임 캐릭터의 무기 장착

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        전략 패턴 비유                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 컨텍스트 (캐릭터):                                                       │
│     캐릭터는 "공격한다"는 행위는 변하지 않음. 하지만 "어떻게" 공격할지는           │
│     캐릭터가 결정하지 않고 장착된 무기에 맡김.                                   │
│                                                                             │
│  2. 전략 (무기):                                                             │
│     검, 활, 마법 지팡이 등 다양한 무기들. 각 무기는 자신만의 공격 로직(알고리즘)을   │
│     가지고 있음.                                                              │
│                                                                             │
│  3. 전략 교체:                                                               │
│     캐릭터 소스코드를 고치지 않고도, 손에 들고 있는 무기만 바꿔 끼우면(런타임)     │
│     공격 방식이 칼질에서 마법 발사로 즉시 바뀜!                                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 전략 패턴의 주요 구성 요소
- **Strategy (전략 인터페이스)**: 모든 지원하는 알고리즘에 대한 공통 인터페이스 정의.
- **ConcreteStrategy (구체적 전략 클래스)**: 실제 알고리즘을 구현한 클래스들.
- **Context (컨텍스트)**: 전략 객체를 참조하며, 필요에 따라 전략을 실행하는 주체.

### 2. 전략 패턴 구조도 (ASCII)

```text
    ┌───────────────┐             ┌──────────────────┐
    │    Context    │◇───────────▶│     Strategy     │
    ├───────────────┤             ├──────────────────┤
    │ - strategy    │             │ + execute()      │
    ├───────────────┤             └────────▲─────────┘
    │ + setStrategy()│                      │ (Interface)
    │ + performTask()│             ┌────────┴──────────┐
    └───────────────┘             │                   │
           │               ┌──────┴──────┐     ┌──────┴──────┐
           │ execute()     │ ConcreteStrA │     │ ConcreteStrB │
           └──────────────▶├──────────────┤     ├──────────────┤
                           │ + execute()  │     │ + execute()  │
                           └──────────────┘     └──────────────┘
```

### 3. 구현의 핵심: 위임 (Delegation)
Context는 직접 알고리즘을 구현하지 않고, 가지고 있는 Strategy 객체에게 실행을 **위임**합니다. 이를 통해 상속(Inheritance)의 경직성을 극복하고 합성(Composition)의 유연함을 얻습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 전략 패턴 vs 상태 패턴 (State Pattern)
- **Strategy**: 클라이언트가 **알고리즘을 직접 선택**하여 컨텍스트에 주입합니다. (결제 시 '카드'를 직접 고름)
- **State**: 객체의 상태에 따라 **내부에서 전략이 자동으로 변합니다**. (자판기 돈 투입 여부에 따라 동작이 바뀜)

### 2. 기술적 시너지: 의존성 주입 (DI)
Spring과 같은 프레임워크에서 전략 패턴은 빛을 발합니다. 설정 파일이나 어노테이션을 통해 어떤 구체 전략을 주입할지 결정할 수 있어, 소스코드의 수정 없이 시스템의 동작을 완전히 바꿀 수 있습니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 다국적 배송비 계산 시스템
- **상황**: 국내 배송, 해외 배송(항공), 해외 배송(해상) 등 국가와 무게에 따라 계산 로직이 다름.
- **결단**: **전략 패턴 적용**.
- **판단**: `ShippingCalculator`를 컨텍스트로 두고, `DomesticStrategy`, `AirStrategy`, `SeaStrategy`를 구현. 새로운 국가나 배송 수단이 추가되어도 기존 계산 로직을 건드리지 않고 신규 전략 클래스만 추가하여 대응. 유닛 테스트 시에도 각 전략별로 독립적인 검증이 가능해져 품질 확보 용이.

### 📢 기술사적 결언
> "전략 패턴은 **'만약(If)'을 '객체(Object)'로 치환**하는 마법이다. 코드가 복잡해지는 첫 번째 징후는 조건문의 중첩이다. 전략 패턴을 통해 이 조건들을 독립적인 클래스로 격리하는 순간, 시스템은 거대한 덩어리에서 조립 가능한 블록 구조로 진화한다. 이는 현대 아키텍처가 지향하는 **'컴포넌트화'**의 가장 밑바닥에 흐르는 핵심 철학이다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **복잡도 감소**: 복잡한 조건문 제거로 순환 복잡도(Cyclomatic Complexity) 50% 이상 감소.
- **확장성 확보**: 신규 알고리즘 추가 시 기존 코드 영향도 0%.

### 미래 전망
최근 자바스크립트나 파이썬 같은 언어에서는 클래스 없이 **고차 함수(Higher-order Function)**를 사용하여 전략 패턴을 더 간결하게 구현합니다. 함수를 파라미터로 넘기는 행위 자체가 전략 패턴의 본질인 '행위의 교체'를 의미하며, 이는 함수형 프로그래밍과 객체지향 프로그래밍이 만나는 지점에서 더욱 강력한 설계 도구로 진화하고 있습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **전략 패턴**은 리모컨 하나로 **TV도 켜고, 에어컨도 켜고, 로봇 청소기도 켜는 것**과 같아요.
2. 리모컨(컨텍스트)은 "켜기" 버튼 하나만 있으면 되고, **어떤 가전제품(전략)**을 연결하느냐에 따라 하는 일이 달라지죠.
3. 가전제품을 새로 사도 리모컨을 새로 살 필요 없이, 그냥 **새로운 가전제품만 연결**하면 바로 쓸 수 있답니다!
