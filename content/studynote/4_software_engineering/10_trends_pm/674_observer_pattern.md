# 674. 옵저버 패턴 이벤트 발행-구독

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 한 객체(**Subject**)의 상태가 변하면 그 객체에 의존하는 다른 객체들(**Observers**)에게 자동으로 알림을 보내고 상태를 갱신하게 만드는 **행위(Behavioral) 패턴**이다.
> 2. **느슨한 결합**: 주제(Subject)는 관찰자(Observer)들의 구체적인 클래스를 알 필요 없이 인터페이스만 알면 되므로, 객체 간의 의존성을 최소화하는 **발행-구독(Publish-Subscribe)** 구조를 형성한다.
> 3. **가치**: 일대다(1:N) 의존 관계에서 데이터의 일관성을 유지하며, 새로운 관찰자를 동적으로 추가하거나 제거하기 쉬운 유연한 확장성을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "변화에 어떻게 반응할 것인가?"

뉴스 앱에서 실시간 속보가 떴을 때, 앱이 1초마다 서버에 "새 뉴스 있어요?"라고 물어보는 것(Polling)은 매우 비효율적입니다. 대신, 서버가 새 뉴스가 생겼을 때 구독자들에게 "새 뉴스 나왔어요!"라고 알려주는 것(Push)이 훨씬 현명합니다. **옵저버(Observer)** 패턴은 이러한 '알림 체계'를 객체 설계 수준에서 구현한 것입니다.

### 💡 비유: 유튜브 채널 구독

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        옵저버 패턴 비유                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 주제 (유튜버):                                                           │
│     영상을 올리는 사람. 누가 내 영상을 보는지 일일이 알 필요는 없지만,             │
│     구독자 명단(List of Observers)은 가지고 있음.                                │
│                                                                             │
│  2. 관찰자 (구독자):                                                         │
│     종 모양 아이콘(구독)을 눌러둔 시청자. 유튜버가 영상을 올리면(상태 변경),       │
│     스마트폰으로 알림(Notification)을 받음.                                     │
│                                                                             │
│  3. 상호작용:                                                                │
│     유튜버는 알림만 보내고 끝! 영상을 볼지 말지는 구독자가 알아서 결정함.          │
│     구독을 취소하면 더 이상 알림이 오지 않음.                                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 옵저버 패턴의 주요 구성 요소
- **Subject (주제)**: 관찰자들을 등록/삭제하는 메서드를 제공하며, 상태 변경 시 `notify()`를 호출.
- **Observer (관찰자 인터페이스)**: 업데이트 메시지를 받기 위한 공통 메서드(예: `update()`) 정의.
- **ConcreteSubject / ConcreteObserver**: 실제 비즈니스 로직이 구현된 클래스들.

### 2. 옵저버 패턴 구조도 (ASCII)

```text
    ┌────────────────┐             ┌──────────────────┐
    │    Subject     │◇───────────▶│     Observer     │
    ├────────────────┤             ├──────────────────┤
    │ + attach(obs)  │             │ + update(data)   │
    │ + detach(obs)  │             └────────▲─────────┘
    │ + notify()     │                      │ (Implementation)
    └───────▲────────┘             ┌────────┴─────────┐
            │ (Inheritance)        │ ConcreteObserver │
    ┌───────┴────────┐             ├──────────────────┤
    │ConcreteSubject │             │ + update(data)   │
    ├────────────────┤             └──────────────────┘
    │ - state: Int   │ <──── 상태 변화 시 notify() 호출
    └────────────────┘
```

### 3. Push 방식 vs Pull 방식
- **Push**: Subject가 변경된 데이터를 Observer에게 직접 파라미터로 넘겨줌 (간편함).
- **Pull**: Subject는 알림만 보내고, Observer가 Subject의 데이터를 직접 가져감 (자율성 높음).

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 옵저버(Observer) vs 발행-구독(Pub-Sub)
엄격히 구분하면 미묘한 차이가 있습니다.
- **Observer**: Subject와 Observer가 서로를 어느 정도 알고 있음 (직접 통신).
- **Pub-Sub**: 중간에 **이벤트 버스(Message Broker)**가 있어 보낸 사람과 받는 사람이 서로의 존재를 전혀 모름 (완전한 비동기/분산).

### 2. 기술적 시너지: 반응형 프로그래밍 (RxJava, RxSwift)
현대 프론트엔드와 모바일 앱은 옵저버 패턴의 확장인 **Reactive Streams**를 기반으로 합니다. 데이터의 흐름(Stream)을 관찰하고 있다가 값이 들어오면 비동기적으로 처리하는 방식은 모두 옵저버 패턴의 철학을 계승한 것입니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 주식 시장 전광판 시스템
- **상황**: 삼성전자 주가가 변할 때마다 모바일 앱, 웹 브라우저, 대형 전광판이 동시에 업데이트되어야 함.
- **결단**: **옵저버 패턴 적용**.
- **판단**: 주가 데이터(Subject)가 변할 때 `notifyObservers()`를 실행. 앱, 웹, 전광판(Observers)은 각자의 출력 환경에 맞게 `update()` 로직을 수행. 주식 종목이 수만 개로 늘어나도 관찰자 구조는 변하지 않으므로 유지보수성 탁월.

### 📢 기술사적 결언
> "옵저버 패턴은 **'관심사의 분리'**를 통해 시스템의 유연성을 극대화하는 핵심 도구다. 특히 상태 변화가 빈번하고 그 영향을 받는 객체가 가변적인 상황에서 하드코딩된 호출을 제거하는 유일한 해법이다. 다만, 관찰자가 너무 많아지면 알림 전송에 따른 성능 저하와 순환 참조(Circular Reference) 리스크가 있으므로, **비동기 처리**와 **약한 참조(Weak Reference)** 활용을 고려해야 한다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적/정성적 기대효과
- **응집도 향상**: 알림 로직과 알림을 받는 로직이 분리되어 각자의 기능에만 집중.
- **재사용성**: 동일한 Subject에 대해 무수히 많은 종류의 Observer를 쉽게 갈아 끼울 수 있음.

### 미래 전망
클라우드 네이티브와 MSA 환경에서는 옵저버 패턴이 네트워크를 넘어 **'이벤트 기반 아키텍처 (EDA)'**로 확장되고 있습니다. Kafka나 RabbitMQ 같은 메시지 브로커를 활용하여 서비스 간 상태를 동기화하는 기법은 전사적 수준의 옵저버 패턴 구현이라 할 수 있으며, 이는 현대 분산 시스템의 표준 설계 모델이 되고 있습니다.
