# 142-147. 요구사항 분석 및 모델링 (Structured, OOA)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구사항 분석은 이해관계자의 모호한 욕구를 **SRS (Software Requirements Specification)**라는 정밀한 청사진으로 변환하는 공학적 과정입니다.
> 2. **가치**: 구조적 분석(Structured Analysis)의 기능 중심 모델링과 OOA (Object-Oriented Analysis)의 객체 중심 모델링을 병행하여 개발 초기 요구사항 결함으로 인한 유지보수 비용을 80% 이상 절감합니다.
> 3. **융합**: 데이터 흐름을 시각화하는 **DFD (Data Flow Diagram)**와 실세계를 매핑하는 **UML (Unified Modeling Language)**을 활용하여 개발자, 설계자, 클라이언트 간의 인지 격차를 해소합니다.

---

### Ⅰ. 개요 (Context & Background)

**요구사항 분석(Requirements Analysis)**은 고객이 원하는 기능적·비기능적 요구사항을 수집한 후, 이들 사이의 모순, 중복, 누락을 식별하고 정제하여 시스템의 논리적 모델을 구축하는 단계입니다. 단순히 요구사항을 나열하는 수준을 넘어, 시스템의 **'무엇(What)'**을 정의하는 핵심 활동입니다.

소프트웨어 위기(Software Crisis) 이후 대형 프로젝트의 복잡성을 관리하기 위해 등장했습니다. 초기에는 프로세스 중심의 구조적 분석이 주류였으나, GUI 환경의 확산과 비즈니스 변동성의 증가로 인해 객체지향 분석(OOA)이 표준으로 자리 잡았습니다. 현대 실무에서는 두 방법론을 융합하여 안정성과 유연성을 동시에 확보합니다.

이 과정은 엉킨 실타래를 풀어 하나의 실(Story)로 엮어내는 과정과 같습니다.

#### ASCII: 분석의 위치와 중요성
```ascii
[소프트웨어 생명주기(SDLC) 내 요구사항 분석의 위상]

1. 요구사항 공학 (Requirements Engineering)
    └──> 타당성 검토 (Feasibility Study)
    └──> 요구사항 도출 (Elicitation)
    └──> [ ★ 요구사항 분석 및 모델링 (Analysis & Modeling) ★ ]
           │
           ├── 구조적 분석 (Structured)
           │    └── DFD, DD, Mini-Spec
           │
           └── 객체지향 분석 (OOA)
                └── UML (Use Case, Class, Sequence...)
    └──> 요구사항 정의 (Specification) -> SRS
```
*(해설: SDLC 전 과정에서 분석 단계는 추상적인 요구를 구체적인 설계로 넘어가는 가장 중요한 관문입니다. 이 단계의 품질이 코딩과 테스트의 난이도를 결정합니다.)*

**📢 섹션 요약 비유**: 요구사항 분석은 마치 건축을 위해 **'허름한 스케치'를 '정밀한 설계 도면'과 '시공 상세서'로 바꾸는 작업**과 같습니다. 건축주가 "방이 많았으면 좋겠다"고 말하는 막연한 소망을, 벽돌 한 치의 오차 없이 배치된 도면으로 만드는 과정입니다.

### Ⅱ. 구조적 분석 (Structured Analysis)

구조적 분석은 1970년대 **Yourdon**과 **DeMarco**에 의해 정립된 기법으로, 시스템을 기능(Function) 중심으로 분해하여 데이터의 흐름을 추적하는 방식입니다. 실무 레거시 시스템이나 임베디드 시스템에서 여전히 강력한 분석 도구로 사용됩니다.

#### 1. 핵심 구성 요소 (Components)
구조적 분석은 시스템을 **DFD (Data Flow Diagram)**로 시각화하고, 그 구성 요소를 **DD (Data Dictionary)**로 정의하며, 최하위 프로세스를 **Mini-Spec**으로 명세합니다.

| 구성 요소 | 명칭 (Full Name) | 역할 | 실무 기술/표기 |
|:---:|:---|:---|:---|
| **Process** | 프로세스 | 데이터를 가공하거나 변환하는 기능(원형) | P1.0, P2.1 (번호 체계) |
| **Data Flow** | 데이터 흐름 | 데이터가 이동하는 경로(화살표) | `Order_Info`, `Validated_ID` |
| **Data Store** | 데이터 저장소 | 데이터가 저장되는 장소(평행선 두 줄) | `Master_File`, `Log_DB` |
| **Terminator** | 단말/외부 개체 | 시스템과 상호작용하는 외부 요인(사각형) | `User`, `Sensor`, `External_API` |

#### 2. 핵심 원리: Leveling (계층화)
복잡한 시스템을 이해하기 위해 **Top-Down** 방식으로 단계별로 상세화하는 기법입니다.
1.  **Context Diagram**: 시스템 전체를 하나의 박스로 보고 외부와의 인터페이스만 표현 (Level 0).
2.  **Level 1 DFD**: 주요 기능별로 프로세스를 분해.
3.  **Level 2+ DFD**: 복잡한 프로세스를 다시 하위 기능으로 분해.

#### 3. 심층 기술: 자료 사전 (Data Dictionary)
DD는 DFD상의 모든 데이터 요소의 **정의(Definition)**와 **구성(Composition)**을 명확히 합니다. 실무에서는 DB 스키마 정의나 API 계약서의 기초가 됩니다.

*   **문법 (BNF 스타일)**:
    *   `=` : is defined as (정의됨)
    *   `+` : and (연결, 순서)
    *   `{ }` : iteration of (반복, 0회 이상)
    *   `[ | ]` : selection (선택, 둘 중 하나)
    *   `( )` : optional (선택, 0회 또는 1회)

**Code Snippet: DD 예시**
```text
고객주문 = 주문번호 + 주문일자 + 배송지정보 + { 주문상품 } + 결제정보
주문상품 = 상품코드 + 상품명 + 수량 + (단위당가격)
결제정보 = [ 카드결제 | 무통장입금 | 간편결제 ]
카드결제 = 카드사명 + 카드번호 + 할부개월수
```

#### ASCII: DFD 상세 구조 예시
```ascii
[ Level 1 DFD 예시: 주문 처리 시스템 ]

  ┌─────────┐                 ┌─────────────┐
  │ 고객    │───────────────>│  1. 주문접수 │
  │(Actor)  │  주문요청(D1)   │ (Process)   │
  └─────────┘                 └──────┬──────┘
                                     │
                                     │ 유효성검사(D2)
                                     ▼
                              ┌─────────────┐
                              │  2. 재고확인 │───────────────────┐
                              └──────┬──────┘                   │
                                     │ 재고데이터(D3)            │
                                     ▼                           │
                              ┌─────────────┐                   │
                              │  3. 배송처리 │                   │
                              └──────┬──────┘                   │
                                     │                          ▼
                                     │   배송지시서(D4)    ┌──────────────┐
                                     └──────────────────>│ 4. 통계관리   │
                                                          │ (Batch)      │
                                                          └──────────────┘
```
*(해설: 데이터는 외부(Actor)에서 들어와 프로세스를 거쳐 가공되고, 저장소(Data Store)에 저장되거나 다시 흐름으로 나갑니다. 화살표에 이름이 있어야 하며, 프로세스는 동작을 나타내는 명사+동사 형태여야 합니다.)*

**📢 섹션 요약 비유**: 구조적 분석은 마치 **'공장의 컨베이어 벨트 공정도'**를 그리는 것과 같습니다. 원재료(입력 데이터)가 어떤 공정(프로세스)을 거쳐 어떻게 변형되고, 완제품(출력 데이터)으로 나오는지를 **흐름(Flow)** 중심으로 파악하는 방식입니다.

### Ⅲ. 객체지향 분석 (OOA, Object-Oriented Analysis)

OOA는 시스템을 **정적 구조**, **동적 행위**, **기능적 요구사항**의 세 가지 관점에서 모델링합니다. 이는 **OMT (Object Modeling Technology)** 방법론과 **UML (Unified Modeling Language)** 표준을 따릅니다.

#### 1. 3대 모델링 뷰 (Three Views of OOA)
| 뷰(View) | 목적 | 주요 다이어그램 | 핵심 포인트 |
|:---|:---|:---|:---|
| **정적 모델링** | 시스템의 구조적 스켈레톤 정의 | **Class Diagram** | 클래스, 속성, 연관관계(Association) |
| **동적 모델링** | 시간 흐름에 따른 상태 및 상호작용 | **Statechart**, **Sequence Diagram** | 상태 전이, 메시지 전달 순서 |
| **기능 모델링** | 사용자 관점의 기능적 요구사항 명세 | **Use Case Diagram** | 액터, 시스템 경계 |

#### 2. 심층 분석: Use Case Diagram (유스케이스 다이어그램)
유스케이스는 시스템이 제공해야 하는 기능을 **'액터(Actor)'**와 **'유스케이스(Use Case)'** 간의 관계로 표현합니다.

*   **Include (포함) 관계**: 필수적인 하위 기능 분리. (예: "로그인"은 "정보 수정"에 포함됨)
*   **Extend (확장) 관계**: 선택적이거나 조건부인 기능. (예: "VIP 등급 판별"은 "결제"의 흐름을 확장)

#### 3. 심층 분석: Class vs Sequence
*   **Class Diagram**: 시스템의 **'뼈대'**. 데이터와 메서드를 캡슐화하고 클래스 간의 연관(Association), 의존(Dependency), 상속(Generalization)을 정의합니다.
*   **Sequence Diagram**: 시스템의 **'맥박'**. 객체 간 메시지 호출의 순서를 시간 축으로 표현하여 **비즈니스 로직 흐름**을 검증합니다.

#### ASCII: OOA의 3가지 관점 통합
```ascii
[ 객체지향 분석(OOA)의 3차원적 관점 ]

               ┌──────────────────────────────────────────┐
               │            시스템의 본질 (Real World)    │
               └─────────────────────┬────────────────────┘�
        ┌─────────────────────────────┼─────────────────────────────┐
        ▼                             ▼                             ▼
┌───────────────┐           ┌───────────────┐           ┌───────────────┐
│  정적 (Static) │           │  동적 (Dynamic)│           │  기능 (Functional)│
└───────┬───────┘           └───────┬───────┘           └───────┬───────┘
        │                           │                           │
        │ "어떤 구조로?"            │ "어떻게 움직이는?"         │ "무엇을 하는?"
        │                           │                           │
        ▼                           ▼                           ▼
  ┌─────────┐                 ┌─────────┐                 ┌─────────┐
  │  Class  │  (Entity,       │ Sequence│  (Message        │ Use Case│  (Req,
  │ Diagram │   Boundary)     │ Diagram │   Flow)          │ Diagram │   Scope)
  └─────────┘                 └─────────┘                 └─────────┘
```
*(해설: OOA는 시스템을 입체적으로 봅니다. Class Diagram로 구조를 잡고, Sequence Diagram로 로직을 엮고, Use Case Diagram으로 요구사항을 커버하는지 확인합니다.)*

**📢 섹션 요약 비유**: 객체지향 분석은 **'공장의 로봇과 장비 설계도'**를 만드는 것입니다. 로봇(객체) 하나하나가 어떤 팔(메서드)과 머리(속성)를 가지고 있는지 설계(정적)하고, 사이렌이 울리면 어떤 순서로 동작하는지 시뮬레이션(동적)하는 것입니다.

### Ⅳ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

구조적 분석과 객체지향 분석은 상호 배타적이지 않으며, 실무에서는 프로젝트의 성격에 따라 적절히 섞어 사용합니다.

#### 1. 기술 비교 분석 (Structured vs. OOA)

| 구분 | 구조적 분석 (Structured) | 객체지향 분석 (OOA) |
|:---|:---|:---|
| **중심 관점** | **Data Flow (데이터의 흐름)** | **Object (데이터와 행위의 캡슐화)** |
| **표기법** | **DFD**, DD, Mini-Spec | **UML** (Use Case, Class, Sequence, etc.) |
| **결합도** | 기능(Function) 중심으로 데이터가 이동 | 데이터가 객체 내부에 은폐됨 (Encapsulation) |
| **장점** | 프로세스 흐름 파악 용이 (단순 시스템 적합) | 재사용성, 유지보수성 우수 (복잡 시스템 적합) |
| **단점** | 데이터와 프로세스의 분리로 인한 변동성 취약 | 학습 곡선이 높고 초기 모델링 비용이 큼 |
| **실무 적용** | 데이터 처리 위주 시스템(성적처리, 회계) | 대규모 분산 시스템, 실시간 시스템, UI/UX 시스템 |

#### 2. 융합 분석 (Synergy)
최신 **Agile (애자일)** 방법론에서는 이 두 가지를 혼합하여 **Hybrid Modeling**을 수행합니다.
*   예: 초기 요구사항 수집 단계에서는 구조적 **DFD**를 사용하여 전체 비즈니스 흐름을 빠르게 이해하고, 상세 설계 단계에서는 **Class Diagram**과 **Sequence Diagram**을 사용하여 개발자가 바로 코딩할 수 있도록 구체화합니다.

#### ASCII: Hybrid Approach 흐름
```ascii
[ 현대적 하이브리드 요구사항 분석 프로세스 ]

요구사항 도출
      │
      ├─────────────────┐
      ▼                 ▼
[ Use Case 작성 ]  [ 전체 맥락 이해 ]
      │                 │ (Context Level DFD 작성)
      ▼                 │
[ 객체 추출 ]           │
      │
