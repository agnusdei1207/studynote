# 135-141. 요구사항 도출 기법 (Persona, Journey Map)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구사항 공학(Requirements Engineering)의 도출 단계에서 단순 질문을 넘어 사용자의 **잠재적 니즈(Latent Needs)**와 **비관심 영역(Indifference Zone)**을 파악하기 위해 정성적(Qualitative) 연구 방법론을 적용하는 기술이다.
> 2. **가치**: 페르소나(Persona)와 여정 지도(Journey Map)를 활용하여 **"사용자는 자신이 원하는 것을 정확히 모른다"**는 헨리 포드의 명제를 해결하여, 프로젝트 후반부의 요구사항 변경 비용을 약 40% 이상 절감한다.
> 3. **융합**: UI/UX(User Interface/User Experience) 설계, 애자일(Agile) 스크럼의 사용자 스토리(User Story) 정의, 그리고 AI 기반의 추천 시스템 서비스 시나리오 설계에 필수적인 베이스가 된다.

---

### Ⅰ. 개요 (Context & Background) - [500자+]

**개념 및 철학**
요구사항 도출(Elicitation)은 이해관계자(Stakeholders)로부터 시스템이 충족해야 할 요구조건과 제약조건을 식별하는 과정이다. 전통적인 방식(인터뷰, 설문)은 사용자가 **의식적으로 인지하고 표현할 수 있는 요구사항(Explicit Requirements)**에 국한된다. 그러나 실제 비즈니스 가치를 창출하는 핵심은 사용자 스스로도 인지하지 못하는 **습관적 행동 패턴이나 무의식적 불편함**에 있다. 페르소나와 여정 지도는 이러한 사용자의 맥락(Context)을 시각화하고 공감(Empathy)하여 **"잠재적 요구사항"**을 끄집어내는 기법이다.

**💡 비유**
사용자는 마치 의사에게 통증을 호소하는 환자와 같다. "배가 아프다"는 말(명시적 요구사항)만 듣고는 약만 처방할 뿐, 근본 원인(잠재적 요구사항)을 찾지 못한다. 의사가 환자의 생활 습관(여정)을 청취하고 가상의 시나리오를 통해 병증을 유도하듯, 개발자도 페르소나를 통해 환자(사용자)의 입장이 되어보아야 진짜 병(니즈)을 찾을 수 있다.

**등장 배경**
1. **기존 한계**: 워터폴(Waterfall) 모델 시절의 두괄식 정의 방식은 개발 완료 후에야 "내가 원한 건 이게 아닌데"라는 임팩트 있는 한 줄 평을 불러왔다.
2. **혁신적 패러다임**: 디자인 씽킹(Design Thinking)과 UX 중심 개발(UCD, User-Centered Design)이 대두되며, 공감(Empathize) 단계의 기술적 도구로 페르소나와 여정 지도가 도입되었다.
3. **현재의 비즈니스 요구**: 빅데이터(Big Data) 시대가 도래하며, 단순 기능 구현을 넘어 사용자 경험에 기반한 서비스 차별화가 핵심 경쟁력이 되었다.

**ASCII 도표: 의식 수준별 요구사항 분류**

```ascii
+-----------------------+-----------------------------------------------+
|     수준 (Level)       |                     설명 (Description)        |
+-----------------------+-----------------------------------------------+
|   ▲ 의식 (Conscious)   | [명시적 요구] 사용자가 말로 직접 요구하는 기능 |
|      (위 - Iceberg)    |   ex) "로그인 버튼을 만들어주세요"             |
+-----------------------+-----------------------------------------------+
|   -- 수면 (Submerged) -| --------------------------------------------- |
+-----------------------+-----------------------------------------------+
|   ▼ 무의식 (Unconscious)| [잠재적 요구] 행동 관찰로만 발견 가능한 니즈   |
|      (아래 - Iceberg)  |   ex) 버튼 위치가 손가락이 닿기 불편한 거리   |
|                       |   -> 도출 기법: 페르소나, 여정 지도, 섀도잉   |
+-----------------------+-----------------------------------------------+
```

> **해설**: 요구사항은 빙산과 같다. 수면 위에 드러난 것은 전체의 20%에 불과하며, 나머지 80%의 치명적인 잠재적 요구사항은 수면 아래 가라앉아 있다. 페르소나와 여정 지도는 이 잠수함 빙산을 탐지하는 소나(Sonar) 역할을 한다.

**📢 섹션 요약 비유**: 전통적인 요구사항 도출은 **퀴즈쇼의 정답자와 같아서** 질문을 하면 대답만 주고 받지만, 페르소나/여정 지도 기법은 **탐정이 용의자의 일상을 미행하며 동선을 파악하는 것과 같습니다.**

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive) - [1,000자+]

이 섹션에서는 요구사항 도출을 위한 핵심 도구인 페르소나와 여정 지도의 구조와 생성 알고리즘을 다룬다.

**1. 핵심 구성 요소 비교**

| 구분 (Category) | 페르소나 (Persona) | 여정 지도 (Customer Journey Map, CJM) | 프로토타이핑 (Prototyping) |
| :--- | :--- | :--- | :--- |
| **정의 (Definition)** | 가상의 사용자 아바타 (Avatar) | 시간 흐름에 따른 사용자 경로 | 요구사항의 가시화 (Tangible) |
| **목적 (Purpose)** | "누가" 시스템을 쓰는지 집중 | "어떻게" 서비스를 경험하는지 집중 | "무엇을" 구현할지 확인 |
| **구성 요소 (Elements)** | 인구통계학적 특성, 목표, 성격, 인용구 | 단계(Stage), 터치포인트(Touchpoint), 감정曲线 | UI 스케치, 와이어프레임, 인터랙션 |
| **산출물 형태 (Format)** | A4 1장 시트 (프로필, 사진) | 1row x Ncols 시각화 맵 | Figma, Axure, 종이 목업 |
| **생성 시점 (Timing)** | 프로젝트 초기, 요구사항 분석 시 | 페르소나 생성 후, 시나리오 작성 시 | 설계 및 구현 단계 전반 |

**2. 페르소나(Persona) 설계 알고리즘**

페르소나는 실제 데이터를 기반으로 통계적 추론을 거쳐 생성된 '가상의 인물'이다. 막연한 '사용자' 대신 구체적인 '김대리'를 대상으로 함으로써 팀 내 커뮤니케이션 효율을 높인다.

**ASCII 다이어그램: 페르소나 구조 (Archetype)**

```ascii
+-----------------------------------------------------------------------+
|                         PERSONA PROFILE                               |
+-----------------------------------------------------------------------+
| [Photo]  Name: 김철수 (The Power User)                                |
|          Role: Senior Logistics Manager                               |
+-----------------------------------------------------------------------+
| 1. Demographics (인구통계학적 특성)                                   |
|    - Age: 42 / Male / Married                                         |
|    - Tech Savviness: High (Use Excel Macros)                          |
+-----------------------------------------------------------------------+
| 2. Goals (목표) & Frustrations (좌절)                                 |
|    - Goal: "재고 현황을 실시간으로 파악하고 싶다"                      |
|    - Frustration: "기존 시스템은 리포트 생성에 5분이나 걸려"           |
+-----------------------------------------------------------------------+
| 3. Bio / Quote (인용구)                                               |
|    "나는 복잡한 설정 메뉴는 싫어. 딱 한 번의 클릭으로 보고서가        |
|     내 메일로 날아오길 바라."                                          |
+-----------------------------------------------------------------------+
| 4. Environment (환경)                                                 |
|    - 사무실, 듀얼 모니터 사용, 업무 시간 09:00~19:00                   |
+-----------------------------------------------------------------------+
```

> **해설**: 위 구조는 단순한 신상정보가 아니다. '좌절(Frustration)' 항목은 **Pain Point(통증 지점)**를 직접 정의하여 개발팀이 해결해야 할 핵심 과제를 명시한다. '인용구(Quote)'는 개발자가 기능 구현 시 고려해야 할 **UX 톤앤매너(Tone & Manner)**를 제시한다.

**3. 사용자 여정 지도(User Journey Map) 작성 원리**

사용자가 서비스를 인지하고, 사용하고, 떠나기까지의 과정을 단계별로 나누고, 각 지점에서의 감정과 행동을 매핑한다.

**ASCII 다이어그램: 고객 여정 지도 (CJM Framework)**

```ascii
        [시간 축: Time Flow] -------------------------------------------------->

+----------+----------------+-------------------+----------------+---------------+
|  Stage   |    Awareness   |     Consideration |     Purchase   |    Retention  |
+----------+----------------+-------------------+----------------+---------------+
| Behavior | SNS Ad Click   | Compare Specs     | App Install    | Use Feature   |
+----------+----------------+-------------------+----------------+---------------+
| Touch    | Instagram      | Review Site       | Play Store     | Push Noti     |
|  Point   |                |                   |                |               |
+----------+----------------+-------------------+----------------+---------------+
| Emotion  | Interest       | Confusion         | Excitement     | Frustration   |
|  Curve   |       /\       |          \/       |      /\        |       \/      |
|          |      /  \      |                     |    /    \      |   /      \    |
+----------+----------------+-------------------+----------------+---------------+
|  Pain    | "Link is too   | "Too much text    | "Sign-up       | "Cannot find  |
|  Point   |  small"        |  to read"         |  error"        |  logout btn"  |
+----------+----------------+-------------------+----------------+---------------+
| Opportunity|  -> Make      | -> Video Guide    | -> Social      | -> Add        |
| (Oppty)   |  Button Bigger |   Addition        | Login OAuth    | Floating Menu |
+----------+----------------+-------------------+----------------+---------------+
```

> **해설**: 이 맵은 개발자에게 "어디서(Oppty)" 개입해야 할지를 보여준다. 예를 들어, 고객이 '혼란(Confusion)'을 느끼는 Consideration 단계에서 텍스트 대신 **비디오 가이드(Video Guide)**를 제공하면 이탈률을 줄일 수 있음을 시사한다. 감정 곡선(Emotion Curve)이 하락하는 구간이 개선이 시급한 **최우선 개발 대상**이 된다.

**📢 섹션 요약 비유**: **페르소나**는 영화 감독이 캐스팅 단계에서 주연 배우의 캐릭터 시트를 작성하는 것이고, **여정 지도**는 작가가 영화의 각본(Scene)을 시간 순서대로 그려 넣는 스토리보드와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy) - [비교표 2개+]

**1. 도출 기법 심층 비교 (Qualitative vs Quantitative)**

요구사항 도출 기법은 크게 질적(정성적) 연구와 양적(정량적) 연구로 나뉜다. 현대적 방식(페르소나 등)은 정성적 데이터에 의존하지만, 이를 검증하기 위해 양적 데이터가 병행되어야 한다.

| 비교 항목 | 정성적 방법 (Qualitative) | 정량적 방법 (Quantitative) |
|:---|:---|:---|
| **대표 기법** | 인터뷰(Interview), 관찰(Observation), 페르소나, 섀도잉 | 설문지(Survey), A/B 테스트, 웹 로그 분석(Web Analytics) |
| **데이터 성격** | "왜(Why)"에 집중 (심층적, 주관적) | "얼마나(How much)"에 집중 (넓은 범위, 객관적) |
| **표본 크기** | 소수 (5~10명) | 다수 (수백~수만 명) |
| **장점** | 사용자의 숨겨진 동기, 심리적 요인 발견 가능 | 통계적 유의성, 일반화된 경향 파악 가능 |
| **단점** | 연구자의 편견 개입 가능, 일반화 어려움 | 맥락(Context)을 놓치기 쉬움 |
| **시나리오** | 새로운 혁신적 아이디어 도출 시 | 기능의 우선순위를 정하거나 검증할 때 |

**2. 타 과목 융합 관점**

1.  **데이터베이스 (DB)**: 여정 지도의 각 단계에서 발생하는 **이벤트(Event)** 로그를 수집하기 위한 스키마(Schema) 설계에 영향을 준다. (예: "사용자가 결제 버튼을 누른 시점"을 로깅하는 테이블 구조)
2.  **네트워크 및 보안**: 섀도잉(Shadowing) 기법을 통해 사용자의 업무 흐름을 관찰할 때, **정보보호 및 개인정보이슈(Privacy)**가 발생할 수 있으므로 데이터 익명화(Anonymization) 전략이 동반되어야 한다.
3.  **인공지능 (AI)**: 수집된 사용자 행동 데이터를 기반으로 **추천 시스템(Recommender System)**을 구축할 때, 페르소나별 군집화(Clustering) 알고리즘의 학습 데이터로 활용된다.

**ASCII 다이어그램: 방법론의 시너지**

```ascii
      [ 폭 (Breadth) ]
            ↑
            |      설문조사 (Survey) - "50%가 불편함을 느낌"
            |
            +---------------------------> 정량적 (Quantitative)
            |
            |
<-------+---+---------------------> 정성적 (Qualitative)
        |   |
        |   |      페르소나 (Persona) - "김철수는 로딩이 느려서 이탈함"
        |
        ↓
      [ 깊이 (Depth) ]
```

> **해설**: 요구사항 도출은 양 날개가 모두 필요하다. 페르소나로 "왜" 불만족하는지(깊이)를 파악한 후, 설문으로 그것이 얼마나 보편적인 문제인지(넓이)를 검증해야 한다.

**📢 섹션 요약 비유**: 페르소나와 여정 지도는 **지하철 공사**와 같습니다. 지하를 파는 굴착기(정성적 연구)가 땅속 지형(잠재적 니즈)을
