# 156-162. 요구사항 관리와 범위 통제 (RTM, Scope Creep)

## # 요구사항 관리와 범위 통제

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구사항은 정적인 문서가 아니라 생명체와 같아서, **RTM (Requirements Traceability Matrix, 요구사항 추적 행렬)**을 통해 발생부터 소멸까지의 생애주기를 수직/수평적으로 추적해야 함.
> 2. **가치**: 관리되지 않은 변경은 프로젝트 실패의 주범이며, 체계적인 **범위 통제 (Scope Control)**를 통해 비용 초과를 최소화하고 시스템의 일관성을 보장함.
> 3. **융합**: **형상 관리 (Configuration Management)** 시스템과 연동하여 요구사항 변경 버전을 관리하고, CI/CD 파이프라인과 결합하여 개발된 기능이 요구사항을 충족하는지 자동 검증하는 지향점 필요.

---

### Ⅰ. 개요 (Context & Background)

**개념**
소프트웨어 공학에서 **요구사항 관리 (Requirements Management)**란 이해관계자의 요구를 수집, 분석, 명세한 후, 이를 시스템 전체 생명주기 동안 추적하고 변경을 통제하는 활동을 말합니다. 단순히 문서를 작성하는 것을 넘어, "어떤 기능이 요청되었는지(What)"와 "그것이 왜 필요한지(Why)", 그리고 "그것이 어디에 구현되었는지(Where)"를 끊김 없이 연결하는 정합성 확보 활동입니다. 특히, **RTM (Requirements Traceability Matrix, 요구사항 추적 행렬)**은 이러한 연결 고리를 시각화한 핵심 도구입니다.

**💡 비유**
건축에서 설계도면의 변경 사항이 현장의 시공 방법과 자재 리스트, 그리고 최종 검사 checklist까지 일치해야 하는 것과 같습니다.

**등장 배경**
1.  **기존 한계**: 폭포수 모델 초기에는 문서화된 요구사항이 개발 과정에서 유실되거나 임의로 변경되어 "요구사항과 구현물 간의 괴리"라는 치명적인 문제가 발생.
2.  **혁신적 패러다임**: 형상 관리(VC)와 추적성(Traceability) 개념이 도입되며, 요구사항 하나가 코드와 테스트 케이스까지 이어지는 **End-to-End 추적**이 중요시됨.
3.  **현재의 비즈니스 요구**: 애자일(Agile) 환경에서도 변경이 잦은 요구사항의 스프린트별 추적과, 긴밀한 **CI/CD (Continuous Integration/Continuous Deployment, 지속적 통합/배포)** 파이프라인 내의 정합성 검증이 필수적으로 요구됨.

**📢 섹션 요약 비유**: 요구사항 관리는 마치 거대한 도서관의 '분류 관리 시스템'과 같습니다. 책(요구사항) 한 권이 어느 책장(설계)에 꽂히고, 누가(개발자) 읽었으며, 반납되었는지(테스트) 실시간으로 파악하지 않으면, 책은 금세 소실되거나 분실됩니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 (표)**
| 구성 요소 | 역할 | 내부 동작 | 주요 프로토콜/포맷 | 비유 |
|:---|:---|:---|:---|:---|
| **Requirements Repository** | 요구사항 저장소 | 모든 요구사항의 원천 버전 관리 | JSON, XML, DB | 법전 |
| **RTM (Traceability Matrix)** | 추적 매트릭스 | 요구사항 ID와 산출물 ID의 매핑 관계 유지 | Matrix (.xlsx, .csv) | 거대한 엑셀 지도 |
| **CCB (Change Control Board)** | 변경 통제 위원회 | 변경 요청의 승인/반려 심의 및 의사결정 | Meeting Minutes, Workflow | 대법원 |
| **Impact Analysis Tool** | 영향 분석 도구 | 의존성 그래프 탐색하여 변경 파급 범위 계산 | Dependency Graph | 지진 해일 예보 시스템 |
| **Baseline** | 기준선 | 특정 시점의 승인된 요구사항 집합 (Snapshot) | Version Tag (v1.0) | 암벽 등반의 확보 지점 |

**ASCII 구조 다이어그램: 요구사항 추적 구조도**

아래 다이어그램은 요구사항이 상위에서 하위 시스템으로, 그리고 다시 테스트 단계로 이어지는 **이중 추적성(Double Traceability)** 구조를 보여줍니다.

```ascii
   [ Stakeholder Needs ]  (1)
          |
          v
   +--------------------------+
   |  System Specification    |  (2) Requirement Analysis
   +--------------------------+
          |
          +------> [ Design Artifact ] ----> [ Source Code ]
          |           (3)                        (4)
          |
          v
   +---------------------------------------------------------+
   |             RTM (Requirements Traceability Matrix)       |
   |---------------------------------------------------------|
   | ID  | Description       | Design Ref | Code Ref | Test   |
   |-----|-------------------|------------|----------|--------|
   | R-01| User Login        | D-05       | Auth.c   | TC-01  |
   | R-02| Payment Processing| D-12       | Pay.c    | TC-15  |
   +---------------------------------------------------------+
          ^
          |
   [ Test Case & Verification ]  (5)
```

**다이어그램 해설**
1.  **추적 흐름**: 사용자의 니즈(1)가 시스템 명세(2)로 정의되고, 이를 바탕으로 설계(3)와 코드(4)가 생성됩니다.
2.  **매핑의 중심**: **RTM**은 중앙에서 이 모든 과정을 연결합니다. 예를 들어, `R-01(로그인)` 요구사항이 변경되면, RTM을 통해 즉시 `D-05` 설계와 `Auth.c` 코드, 그리고 `TC-01` 테스트 케이스를 수정해야 함을 알 수 있습니다.
3.  **검증**: 역방향(Backward Traceability) 추적을 통해 최종 테스트 케이스(5)가 초기 요구사항을 모두 커버하는지 누락(Orphan)을 발견합니다.

**심층 동작 원리: 변경 요청的生命주기**
1.  **Request (식별)**: 이해관계자가 CR (Change Request, 변경 요청) 제기.
2.  **Classify (분류)**: 단순 오타인지, 아키텍처를 뒤집는 변경인지 분류.
3.  **Analyze (분석)**: **영향 분석 (Impact Analysis)** 수행. 연관된 요구사항, 모듈, 예상 공수(Man-Month) 산출.
4.  **Approve (승인)**: CCB (Change Control Board, 변경 통제 위원회)에서 승인.
5.  **Update (갱신)**: SRS (Software Requirements Specification) 수정 -> RTM 업데이트 -> 베이스라인 재설정.
6.  **Verify (검증)**: 수정된 요구사항이 개발 및 테스트에 반영되었는지 확인.

**핵심 알고리즘 & 코드: 의존성 분석 의사코드**
```python
# 영향 분석(Impact Analysis)을 위한 의존성 그래프 탐색 알고리즘
def analyze_impact(changed_req_id, rtm_graph):
    affected_items = []
    
    # 1. 해당 요구사항과 직접 연결된 하위 아이템 검색 (BFS)
    queue = [changed_req_id]
    visited = set()

    while queue:
        current = queue.pop(0)
        if current in visited: continue
        
        visited.add(current)
        # RTM에서 해당 요구사항을 참조하는 모든 산출물 탐색
        linked_items = rtm_graph.find_dependencies(current)
        
        for item in linked_items:
            affected_items.append(item)
            if item.type == "REQUIREMENT": # 연쇄 반응
                queue.append(item.id)
                
    return affected_items
```

**📢 섹션 요약 비유**: 요구사항 추적은 마치 'DNA 검색'과 같습니다. 범인(요구사항)이 현장(코드)에 남긴 머리카락(함수명)을 통해 거슬러 올라가면, 그 범인의 조상(상위 요구사항)과 친척(연관 모듈)까지 모두 파악할 수 있습니다. RTM은 이 DNA 데이터베이스입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**심층 기술 비교: 수직 vs 수평 추적성**

| 구분 | 수직 추적성 (Vertical) | 수평 추적성 (Horizontal) |
|:---|:---|:---|
| **정의** | 상위 요구사항과 하위 요구사항 간의 계층적 연결 | 요구사항과 다른 산출물(설계, 코드, 테스트) 간의 연결 |
| **핵심 목적** | **"필요한 것을 다 구현했는가?"** (Completeness) | **"제대로 구현했는가?"** (Correctness) |
| **주요 관점** | 요구사항 분해 및 세분화 과정 | 요구사항 실현 및 검증 과정 |
| **끊어졌을 때** | 사용자 요구 누락, 기능 미구현 발생 | 기능이 다르게 구현되거나 테스트 누락 발생 |
| **비유** | 조상을 찾아가는 계보도 | 같은 세대의 친구 관계도 |

**과목 융합 관점**
1.  **SW Engineering + QA (품질 관리)**: 요구사항 추적은 테스트 커버리지와 직결됩니다. RTM이 없으면 **테스트 시나리오**가 실제 요구사항을 100% 커버하는지 증명할 수 없습니다. 이는 ISO 26262(기능 안전) 등 인증 심사에서 탈락 사유가 됩니다.
2.  **SW Engineering + DevOps**: 현대의 **Application Lifecycle Management (ALM, 응용 프로그램 수명 주기 관리)** 도구(Jira, GitHub 등)는 이 추적성을 자동화합니다. 커밋 메시지에 이슈 ID를 포함시켜 코드와 요구사항을 자동으로 연결(Code Traceability)하는 방식이 융합의 정점입니다.

**안티패턴 분석**
*   **Scope Creep (범위 늪/크리프)**: 관리 없이 요구사항이 '늪'에 빠지듯 계속 늘어나는 현상.
    *   *증상*: 일정은 그대로인데 기능은 20% 추가됨.
    *   *결과*: 팀 번아웃(Burnout) 및 품질 저하.
*   **Gold Plating (금도금/골드 플래팅)**: 요청하지 않은 기능을 '고급스럽게' 추가하는 행위.
    *   *오해*: "사용자가 좋아하겠지?"라는 개발자의 착각.
    *   *결과*: 불필요한 복잡도 증가, 잠재적 버그 발생, 유지보수 비용만 상승.

**📢 섹션 요약 비유**: 프로젝트의 범위를 관리하는 것은 마치 '식단 관리'와 같습니다. **범위 크리프**는 다이어트 중인데 샐러드 위에 스테이크와 케이크를 몰래 올리는 것(칼료 증가)이고, **골드 플래팅**은 샐러드를 먹기 좋게 꾸민답시고 비싼 트러플 오일을 뿌려 오히려 소화 불량을 일으키는 것입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**실무 시나리오: 의사결정 과정**

1.  **상황**: 프로젝트 종료 1주 전, 고객사 대표가 "결제 모듈에 '간편 결제'를 추가하고 싶다"는 요청(CR)을 해옴.
2.  **분석 (문제점)**:
    *   **기술적**: 현재 결제 모듈은 PG사의 '승인' API만 사용 중. '등록/발급' API 추가 개발 필요.
    *   **일정적**: 보안 검증(VA) 기간이 최소 2주 소요됨. 종료 1주 전이므로 일정 불가.
    *   **법적**: 개인정보 처리 방침(PIMS) 변경 및 동의 절차 필요.
3.  **기술사적 판단 및 대응**:
    *   **판단**: **수용 불가 (Reject)**. 단, 협상은 필요.
    *   **전략**:
        *   RTM과 영향 분석 보고서를 근거로 현재 일정 및 보안 리스크 설명.
        *   "단계적 도입" 제안: Phase 1에서는 기존 기능으로 안정적으로 출시하고, '간편 결제'는 Phase 2 업데이트(1달 후)로 일정 변경 제안.
        *   결과: 프로젝트 범위(Scope) 방어와 고객 만족도 동시 확보.

**도입 체크리스트**

*   [ ] **기술적 준비**: 요구사항 관리 도구(Jira, Redmine, Doors) 도입 및 형상 관리(Git)와의 연동 설정 완료?
*   [ ] **프로세스 정의**: CR(Change Request) 접수부터 CCB 승인까지의 워크플로우 문서화?
*   [ ] **운영/보안적**: 변경된 요구사항에 대한 보안 위협 모델링(Security Threat Modeling) 재수행 프로세스 포함?

**안티패턴: 잘못된 관리 사례**
*   **"말로만 변경하기"**: 회의록에만 "변경하기로 함"이라고 적고, 요구사항 문서(SRS)와 코드를 수정하지 않음.
    *   *치명적 결함*: 나중에 "그런 거 안 했는데?"라는 분쟁 발생 시 증거 불가. 형상 관리의 핵심 위반.

**📢 섹션 요약 비유**: 기술사의 판단은 마치 '비행기 기장의 결정'과 같습니다. 날씨가 나빠지거나(리스크), 연료가 부족할 때(일정) 무리하게 이륙(변경 수용)하면 추락합니다. 승객(고객)을 안전하게 목적지로 데려다주는 것이 우선이므로, "지금은 갈 수 없으니 잠시 기다려라"라고 단호하게 말할 용기가 필요합니다.
