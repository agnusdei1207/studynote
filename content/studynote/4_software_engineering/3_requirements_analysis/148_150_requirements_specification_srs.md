# 148-150. 요구사항 명세와 명세서(SRS)

> **핵심 인사이트**: 명세는 개발의 '계약서'다. 모호한 자연어의 한계를 극복하기 위해 정형 명세 기법을 활용하며, 완성된 명세서(SRS)는 정확하고 완전하며 추적이 가능해야 한다. 잘 쓰인 SRS 한 권이 수백 번의 회의보다 더 명확한 지침이 된다.

---

## Ⅰ. 요구사항 명세 (Specification) 기법
분석된 요구사항을 공식적인 문서로 기록하는 활동입니다.

1. **비정형 명세 (Informal)**: 자연어(한국어, 영어)나 그림으로 설명. 이해가 쉽지만 모호성이 높음.
2. **정형 명세 (Formal)**: 수학적 기호(Z, VDM 등)와 논리를 사용. 정확하고 검증이 가능하지만 작성과 이해가 어려움.

---

## Ⅱ. 소프트웨어 요구사항 명세서 (SRS)
사용자의 요구를 바탕으로 시스템의 기능, 성능, 제약 사항을 상세히 기술한 문서입니다.

* **주요 포함 내용**: 시스템 개요, 기능적 요구, 비기능적 요구(성능, 보안 등), 데이터 요구, 인터페이스 요구.

---

## Ⅲ. 좋은 명세서(SRS)의 6대 품질 특성 (IEEE 830)
1. **정확성 (Correctness)**: 명세된 내용이 실제 요구와 일치하는가?
2. **명확성 (Unambiguous)**: 누구나 읽어도 한 가지 의미로 해석되는가?
3. **완전성 (Completeness)**: 빠진 기능 없이 모든 요구가 담겼는가?
4. **일관성 (Consistency)**: 요구사항 간에 서로 충돌이 없는가?
5. **수정 용이성 (Modifiable)**: 나중에 고치기 쉬운 구조인가?
6. **추적 가능성 (Traceable)**: 각 요구사항의 기원(Source)과 이후 구현/테스트 단계를 쫓을 수 있는가?

---

## Ⅳ. 개념 맵 및 요약

```ascii
[SRS의 위치와 역할]

  [ 요구 도출 ] ──> [ 분석 ] ──> [ SRS 명세 ] ──> [ 설계/구현 ]
                                     │
                                     ▼
                          ( 테스트의 기준선이 됨 )
```

📢 **섹션 요약 비유**: **요구사항 명세**는 건축 설계도를 그리는 작업입니다. **비정형 명세**는 "튼튼하고 예쁜 집"이라고 말하는 것이고, **정형 명세**는 미터법과 수치를 사용해 정확한 도면을 그리는 것입니다. **SRS**는 공사 시작 전 집주인과 업자가 사인하는 '최종 공사 계약서'이며, 이 계약서가 정확하고 꼼꼼해야(품질 특성) 나중에 엉뚱한 집이 지어지는 사고를 막을 수 있습니다.
