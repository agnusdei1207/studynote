# 148-150. 요구사항 명세와 명세서(SRS)

> **1. 본질 (Essence)**: 요구사항 명세는 모호한 자연어로 된 고객의 니즈를, 개발자가 오해 없이 구현할 수 있도록 수학적 논리나 엄격한 구조로 변환하여 '계약'화하는 과정이다.
> **2. 가치 (Value)**: IEEE 830 표준을 준수하는 견고한 SRS (Software Requirements Specification)는 재작업(Rework) 비용을 80% 이상 절감하며, 시스템의 테스트 가능성(Traceability)과 품질을 보장하는 유일한 출발점이다.
> **3. 융합 (Synergy)**: DB 설계의 ERD(Entity Relationship Diagram) 작성, 보안 설계의 위협 모델링(Threat Modeling), 그리고 AI 모델의 학습 데이터 정의(Data Definition)와 직접 연결되는 프로젝트의 '법전' 역할을 수행한다.

---

### Ⅰ. 개요 (Context & Background)

**개념 정의 및 철학**
요구사항 명세(Specification)란, 이해관계자(Stakeholders)의 추상적인 요구를 시스템이 구현해야 할 구체적이고 검증 가능한 형태로 기술하는 활동이다. 소프트웨어 공학(Software Engineering)의 "Garbage In, Garbage Out" 원리에 따라, 명세 단계에서의 오류는 코딩 단계에서의 오류보다 수십 배 비싼 수정 비용을 유발한다. 따라서 본 단계는 개발이 아닌 '계약'의 성격을 띠며, 기술적, 법적 분쟁을 방지하는 방패 역할을 한다.

**등장 배경: 자연어의 모호성과 형식적 기법의 부상**
초기 소프트웨어 개발은 구두 전달이나 자유로운 문서로 충분했으나, 시스템이 대형화되고 복잡해짐에 따라 '문맥 의존적'인 자연어의 한계가 드러났다. 이를 해결하기 위해 수학적 기호를 사용하여 논리적 모순을 자동으로 검증할 수 있는 '정형 명세(Formal Specification)' 기법이 도입되었으나, 진입 장벽이 높아 실무에서는 구조화된 자연어와 UML(Unified Modeling Language)을 혼용하는 '준정형(Semi-formal)' 기법이 주류를 이루게 되었다.

```ascii
[요구사항 명세의 진화 과정]

1. 비정형 (Informal)
   └─ 방식: 자유로운 에세이 형식
   └─ 한계: "빠르다" -> (1ms? 1ns?) 모호성 발생으로 인한 개발-운영 간 갈등

          ⬇️ (비용 증가, 복잡도 증가)

2. 준정형 (Semi-formal) [현재 실무 주류]
   └─ 방식: 템플릿화된 자연어 + UML 다이어그램 (Use Case, Sequence Diagram)
   └─ 장점: 가독성과 구조화의 균형

          ⬇️ (안전성, 핵심 시스템 등장)

3. 정형 (Formal)
   └─ 방식: 수학적 논리(Z-notation, VDM, B-Method)
   └─ 장점: 기계적 검증(Proof) 가능, 추론(Reasoning) 가능
   └─ 단점: 높은 학습 곡선 및 작성 비용
```

> **📢 섹션 요약 비유**: 요구사항 명세는 '주문 제작' 과정입니다. **비정형 명세**는 "멋지고 넓은 집을 지어주세요"라고 말하는 것이고, **정형 명세**는 "내력벽은 15cm, 창호는 이중유리, 전선은 3.5sq를 사용하며, 하중 지지력은 500kg/m²을 충족하세요"라고 설계도면과 수치로 정밀하게 명시하는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**명세 기법의 상세 분류 및 구조**
요구사항 명세는 크게 자연어를 기반으로 한 비정형 기법과 수학적 모델을 기반으로 한 정형 기법으로 나뉜다. 실무에서는 이를 적절히 섞어 사용하며, SRS 문서는 이러한 내용을 체계적으로 담는 그릇(Repository)이다.

**1. 명세 기법의 종류 및 메커니즘**

| 구분 | 기법 (Technique) | 내부 동작 메커니즘 | 특징 및 용도 | 프로토콜/표준 |
|:---:|:---|:---|:---|:---|
| **비정형** | 자연어 기반 서술 | 문맥(Context)에 의존하여 해석 | 일반적이고 유연함. 초기 아이디어 도출용. | - |
| **준정형** | **UML (Unified Modeling Language)** | 객체지향 메타모델을 이용한 시각화 | 구조적 시각화. Use Case, Sequence 등. | **OMG UML 2.x** |
| **정형** | **Z-notation** | 집합론(Set Theory)과 1차 논리(First Order Logic) 사용 | 상태(State)와 연산(Operation)의 수학적 정의. 항공, 의료 등 안전 임계 시스템에 사용. | ISO/IEC 13568 |
| **정형** | **VDM (Vienna Development Method)** | 전후 조건(Pre/Post-condition) 명시 | 실행 가능한 명세 언어, 모델 검증 가능. | ISO/IEC 13817 |

**2. SRS (Software Requirements Specification) 문서 구조**
IEEE 830 표준에 따른 SRS는 단순한 기능 나열을 넘어, 시스템의 외부 행동을 완전히 정의해야 한다.

```ascii
[ SRS (Software Requirements Specification) 표준 구조도 (IEEE 830) ]
         +-------------------------------------------------+
         |          1. 서론 (Introduction)                  |
         |   └─ 1.3 정의, 1.4 참조자료, 1.5 개요            |
         +-------------------+-----------------------------+
                             |
         +-------------------v-----------------------------+
         |          2. 전체 묘사 (Overall Description)      |
         |   └─ 2.1 제품 관점, 2.2 기능, 2.3 사용자 특성   |
         +-------------------+-----------------------------+
                             |
         +-------------------v-----------------------------+
         |       3. 상세 요구사항 (Detailed Requirements)   |
         |   └─ 3.1 기능적 요구 (Functional Requirements)  | --> Input/Output/Processing 명시
         |   └─ 3.2 비기능적 요구 (Non-Functional)          | --> Performance, Security, Reliability
         |   └─ 3.3 인터페이스 (Interfaces)                | --> UI, Hardware, API
         +-------------------+-----------------------------+
                             |
         +-------------------v-----------------------------+
         |       4. 부록 정보 (Appendix)                    |
         +-------------------------------------------------+
```

*(도해 해설)*: 위 구조는 폭포수 모델(Waterfall Model)의 기반이 되며, 특히 **3.1 기능적 요구**는 "시스템이 무엇을 하는가(What)"를 명시하고, **3.2 비기능적 요구**는 "어떻게 수행하는가(How)"에 대한 제약 조건(속도, 보안 등)을 정의한다. 이 구조가 견고해야 후속 설계 단계에서의 오해를 예방할 수 있다.

**3. 핵심 기술적 원리: 정형 명세 예시 (Z-notation 개념)**
자연어로는 모호할 수 있는 "로그인" 상태를 정형 기법으로 정의하면 다음과 같다. (개념적 표현)

```text
[사용자 상태 정의]
User == [ name : STRING; password : STRING; login_status : BOOL ]

[로그인 함수 정의]
Login_Success(u: User) ==
    PRE u.password = stored_password
    THEN u'.login_status = TRUE
    POST u'.name = u.name
```
이는 '사전 조건(PRE)'이 충족될 때 '사후 조건(POST)'이 보장됨을 수학적으로 증명할 수 있게 한다.

> **📢 섹션 요약 비유**: **SRS 문서**는 건축에서의 '시공 시방서(施工 細目書)'와 같습니다. 단순히 "부엌을 짓는다"는 개요(1장)가 아니라, 수도관은 지름 15mm를 쓰고, 배수구는 5도 기울이며, 싱크대 대리석은 두께 20mm를 사용하라는 구체적인 자재 및 시공 기준(3장)을 담고 있는 문서입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**명세 기법의 심층 비교 및 타 영역과의 시너지**

| 비교 항목 | 비정형 명세 (Informal) | 정형 명세 (Formal) |
|:---:|:---|:---|
| **표현력** | 일상어로 쉬운 의사소통 가능 | 수학적 기호로 엄밀한 표현 가능 |
| **모호성(Ambiguity)** | 높음 (해석 차이 발생 가능) | 낮음 (논리적 모순 자동 검증) |
| **작성 난이도** | 낮음 (누구나 작성 가능) | 높음 (수학적 지식 필요) |
| **도구 지원** | 워드프로세서 수준 | 모델 검증기(Model Checker), 증명 보조기(Proof Assistant) |
| **적용 분야** | 일반 업무용 시스템 | 항공, 원자력, 의료기기 등 Safety-Critical System |

**융합 관점의 분석**

1.  **데이터베이스 (DB)와의 연관성**: 요구사항 명세의 **데이터 요구사항(Data Requirements)** 항목은 곧 DB의 **스키마(Schema)** 설계로 이어진다. 정형 명세에서 정의한 '타입(Type)'과 '제약 조건(Constraint)'은 ERD 다이어그램으로 변환되며, 관계형 데이터베이스(RDBMS)의 무결성 제약조건(Primary Key, Foreign Key)의 기원이 된다.
2.  **프로젝트 관리 (PM)와의 연관성**: SRS의 **추적 가능성(Traceability)** 항목은 프로젝트 관리에서 범위 관리(Scope Management)의 핵심이다. 요구사항 ID가 부여되어야 WBS(Work Breakdown Structure)와 매핑되며, 기획 변경 시 영향도 분석(Impact Analysis)이 수학적으로 가능해진다.

**SRS 품질 특성의 상관관계 분석**
IEEE 830의 6대 품질 특성은 서로 독립적이지 않고 상충(Trade-off) 관계에 있다.

```ascii
[품질 특성 간 상충 관계]

          수정 용이성 (Modifiable)
                  ▲
                  │(상충: 내용이 상세하면 수정이 번거로움)
                  │
    완전성 (Completeness) ────▶  명확성 (Unambiguous)
                  │
                  │(상충: 비즈니스 용어를 그대로 쓰면 이해는 쉬우나 기술적 명확성은 떨어질 수 있음)
                  ▼
          일관성 (Consistency)
```

*(도해 해설)*: SRS 작성 시 가장 어려운 점은 **수정 용이성**과 **완전성**의 균형이다. 너무 상세히 적으면 완전성은 높아지지만, 요구사항이 바뀔 때마다 수십 페이지의 문서를 수정해야 하므로 유지보수가 어려워진다. 이를 해결하기 위해 '요구사항 관리 도구(Requirements Management Tool, 예: DOORS, JIRA)'를 사용하여 데이터베이스화하고 문서를 자동 생성하는 방식을 취한다.

> **📢 섹션 요약 비유**: **비정형 명세와 정형 명세의 선택**은 자동차 수리 매뉴얼과 같습니다. 일반적인 운전자를 위한 안내서는 "엔진 소리가 크면 정비소를 가세요"라는 **비정형** 가이드면 충분하지만, 정비사를 위한 매뉴얼은 "볼트의 토크를 50Nm로 조이고, 클리어런스는 0.05mm로 유지하라"는 **정형** 수치가 필요합니다. SRS는 이 두 가지를 적절히 섞어서 기획자와 개발자 모두가 만족하는 '지능형 매뉴얼'이 되어야 합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**실무 시나리오: 대규모 공공 시스템 SRS 검토**

A 공공기관에서 금융 결제 시스템의 구축을 의뢰하였다. 명세서를 검토하던 중 "시스템은 빠른 응답 속도를 제공해야 한다"라는 비기능 요구사항을 발견했다. 이는 전형적인 **나쁜 명세(Bad Specification)** 사례다.

1.  **문제 진단**: '빠르다'는 주관적인 용어로, 개발사는 "0.5초"를 고객은 "0.1초"를 기대할 수 있어 인수(Inception) 단계에서 분쟁이 발생함.
2.  **의사결정 과정 (재명세)**:
    *   **정량화**: "전체 거래의 90%가 1초 이내에 응답해야 하며, 최대 3초를 초과하면 안 된다."
    *   **구체화**: "TPS (Transactions Per Second)는 최소 1,000건을 지원해야 한다."
    *   **검증 가능성 추가**: "성능 테스트 시 JMeter(또는 LoadRunner)를 통해 1,000명의 동시 사용자를 시뮬레이션하여 기준 통과 시 인수한다."는 조항을 추가.
3.  **결과**: 명확한 SLA (Service Level Agreement) 정의로 향후 분쟁 여지를 차단함.

**도입 체크리스트**

| 구분 | 점검 항목 | 확인 사항 |
|:---:|:---|:---|
| **기술적** | **수정 용이성** | 요구사항의 변경이 발생했을 때, 연관된 설계/코드 위치를 즉시 파악할 수 있는 ID 관리(Traceability Matrix)가 되어 있는가? |
| **기술적** | **일관성 (Consistency)** | "보안 로그는 영구 보관한다"와 "개인정보는 1년 뒤 파기한다"와 같은 모순은 없는가? |
| **운영/보안** | **완전성 (Completeness)** | 예외 상황(Exception, 예: 네트워크 단절, DB 서버 다운)에 대한 시스템의 행동(Failover,
