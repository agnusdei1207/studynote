# 151-155. 요구사항 확인 및 검증 (V&V, Review)

### # 요구사항 확인 및 검증 (V&V, Review)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 생명 주기(SDLC) 전반에 걸쳐 산출물의 무결성을 보장하기 위해 **V&V (Verification & Validation)** 체계를 수립하고, 문서 단계에서 결함을 조기에 발견하는 **정적 검토(Static Review)** 기법을 적용하는 것이다.
> 2. **가치**: 방대한 코드를 리팩토링하는 비용 대비, 요구사항 단계에서 오류를 수정하는 비용은 약 100분의 1에 불과하며(Boehm's Model), 초기 결함 제거를 통해 프로젝트 리스크(Risk)를 획기적으로 절감할 수 있다.
> 3. **융합**: 품질 관리(Quality Assurance)와 **DevOps (Development and Operations)**의 시프트-레프트(Shift-Left) 전략과 연계되며, **CI/CD (Continuous Integration/Continuous Deployment)** 파이프라인 내에서 자동화된 정적 분석 툴(SonarQube 등)과 결합하여 현대적으로 진화하고 있다.

---

### Ⅰ. 개요 (Context & Background)

소프트웨어 위기(Software Crisis) 이후, 소프트웨어의 복잡도가 기하급수적으로 증가함에 따라 '코딩 후 디버깅' 방식으로는 고품질의 소프트웨어를 개발하기 어렵게 되었다. 이에 따라 소프트웨어 개발 수명 주기(Software Development Life Cycle, SDLC)의 초기 단계, 즉 요구사항 분석 및 설계 단계에서부터 품질을 통제하려는 노력이 강조되었다. 여기서 핵심이 되는 개념이 **V&V (Verification & Validation)**이다.

*   **Verification (검증)**: '우리가 제품을 올바르게 만들고 있는가?'에 대한 답을 찾는 활동이다. 각 개발 단계의 산출물(요구사항 명세서, 설계서 등)이 이전 단계의 명세와 정확히 일치하는지, 규칙에 부합하는지를 확인하는 **프로세스 중심**의 활동이다.
*   **Validation (확인)**: '우리가 올바른 제품을 만들고 있는가?'에 대한 답을 찾는 활동이다. 최종 소프트웨어가 실제 사용자의 비즈니스 요구와 기대를 충족시키는지를 확인하는 **사용자 중심**의 활동이다.

이러한 V&V 활동 중, 코드를 실행하지 않고 산출물을 분석하여 결함을 찾아내는 **정적 테스트(Static Test)** 기법, 즉 **Peer Review (동료 검토)**는 가장 경제적이고 효과적인 품질 보증 수단으로 평가받는다. IBM 등의 연구에 따르면, 설계 단계에서 발견된 결함을 수정하는 비용은 테스트 단계에서 발견되었을 때보다 10~100배 저렴하다.

```ascii
[SDLC 단계별 V&V 적용 포인트]

  요구사항 분석 ────────> 설계 (Design) ────────> 구현 (Coding) ────────> 테스트 (Testing)
        │                      │                      │                      │
   ▼ Validation          ▼ Verification          ▼ Verification          ▼ Validation
 (올바른 것인가?)      (제대로 되었는가?)        (제대로 되었는가?)     (올바른 것인가?)
        │                      │                      │                      │
   [사용자 요구 확인]    [설계 명세 검토]          [코드 리뷰]          [사용자 인수 테스트]
   ◁ 동적/정적           ◁ 정적 리뷰 중점         ◁ 정적 리뷰(인스펙션)  ◁ 동적 테스트 중점
```

**📢 섹션 요약 비유**: 소프트웨어 개발은 **거대한 빌딩을 짓는 공사**와 같습니다. V&V는 공사 중간중간 **설계도면대로 기둥이 세워졌는지(검증, Verification)** 확인하고, 완공 후에 이 건물이 **실제 거주자가 살기에 편안한지(확인, Validation)** 확인하는 과정입니다. 공사가 진행될수록 수정 비용이 기하급수적으로 늘어나므로, 청사진을 보고 설계 오류를 잡아내는 '정적 검토'가 매우 중요합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

요구사항 검증의 핵심은 **V&V 프로세스**를 구조화하고, **정적 검토(Static Review)** 기법을 체계적으로 수행하는 데 있다. 이 섹션에서는 V&V의 상세 메커니즘과 다양한 리뷰 기술의 내부 동작을 다룬다.

#### 1. V&V 메커니즘 상세 분석

**Verification**과 **Validation**은 서로 보완적이며 목표와 시점이 다르다.

| 구분 | Verification (검증) | Validation (확인) |
|:---|:---|:---|
| **질문** | "Are we building the product right?"<br>(제품을 제대로 만들고 있는가?) | "Are we building the right product?"<br>(올바른 제품을 만들고 있는가?) |
| **목표** | 명세(Specification) 준수 여부 확인 | 사용자 요구(Needs) 충족 여부 확인 |
| **대상** | 요구사항 명세서, 설계서, 소스코드, 테스트 케이스 등 | 최종 빌드(Built)된 소프트웨어 시스템 |
| **방법** | 리뷰(Review), 정적 분석, 워크쓰루, 인스펙션 | 테스팅(Testing), 벤치마킹, 사용자 설문 |
| **담당** | 주로 개발자, 설계자, QA 엔지니어 | 주로 최종 사용자, 고객, 운영자 |

#### 2. 정적 검토(Static Review) 기법의 세분화

정적 검토는 프로그램을 실행하지 않고 산출물을 검토하는 방식으로, **리뷰(Review)** 기법이 핵심이다. 비용 대비 효과가 가장 높은 결함 발견 방법이다.

**A. 인스펙션 (Inspection)**
가장 형식화되고 엄격한 검토 방식이다.
*   **목적**: 결함(Defect)을 발견하고 제거하는 것이 주목적이며, 교육은 부수적 효과이다.
*   **참여자 및 역할**:
    *   **Moderator (중재자)**: 리뷰를 주관하며, 진행을 통제하고 준비/시간을 관리함. (반드시 숙련자가 맡음)
    *   **Author (작성자)**: 산출물 작성자로서, 검토 시 질문에 답변하나 논리적 방어만 함.
    *   **Reviewer (검토자)**: 산출물을 사전 검토하고 체크리스트에 따라 결함을 식별함.
    *   **Scribe (기록자)**: 회의 중 발견된 결함 내용을 기록함.
*   **프로세스**: Planning -> Overview(교육) -> Preparation(사전 독려) -> **Inspection Meeting(본 회의)** -> Rework(수정) -> Follow-up(확인).

**B. 워크쓰루 (Walkthrough)**
비교적 비공식적인 검토 방식으로, 작성자가 주도한다.
*   **목적**: 결함 발견뿐만 아니라 참여자 간의 **정보 공유와 교육(Informal)**에 중점을 둔다.
*   **진행**: 작성자가 참여자에게 문서나 코드를 직접 설명하며 진행(Scenario 시나리오 방식 사용).
*   **특징**: 인스펙션보다 절차가 간단하며, 특정 시나리오에 따라 논리 흐름을 따라가며 확인한다.

**C. 동료 검토 (Peer Review / Desk Checking)**
가장 가볍고 빠른 형태이다.
*   **특징**: 1~2명의 동료가 자발적으로 문서를 읽고 코멘트를 남기는 형태. 별도의 회의가 없거나 매우 짧다.

```ascii
[정적 검토(Static Review) 방식 비교]

  낮음 ◄───────────── 형식성(Formality) 및 비용 ────────────► 높음
  │                                                                  │
  └─── [동료 검토] ──── [워크쓰루] ──────────────────── [인스펙션] ────┘
       (Desk Check)     (Peer + Author)              (Formal Team)
  
  [ 특징 비교 ]
  ┌─────────────┬────────────────┬──────────────────┬─────────────────┐
  │    구분      │   동료 검토    │    워크쓰루      │    인스펙션      │
  ├─────────────┼────────────────┼──────────────────┼─────────────────┤
  │   주도      │     개인       │    작성자        │   중재자(Moderator) │
  │   목적      │ 간단한 오류    │ 교육+정보 공유   │   결함 발견(중점)  │
  │   산출물    │   코멘트       │   회의록(약식)   │ 결함 리포트(상세) │
  │   문서화    │   필수 아님    │     선택적       │     필수(엄격)    │
  └─────────────┴────────────────┴──────────────────┴─────────────────┘
```

#### 3. 핵심 알고리즘 및 프로세스: 인스펙션 수행 절차

실무에서 가장 효과적인 **인스펙션(Inspection)** 프로세스를 알고리즘적으로 정리하면 다음과 같다.

1.  **Planning (계획)**: 검토 대상 선정, 팀 구성, 자료 배포.
2.  **Overview (개요)**: 작성자가 배경 설명 (참고: 팀원 간 이해도가 높으면 생략 가능).
3.  **Preparation (준비)**: 검토자들이 개별적으로 산출물을 읽으며 결함을 **체크리스트**에 기록. (가장 중요한 단계)
4.  **Meeting (회의)**:
    *   Moderator 주도로 진행.
    *   결함만을 논의하며 해결책(How to fix)은 논의하지 않는다 (시간 낭비 방지).
    *   Author는 방어만 하고 결함을 인정함.
5.  **Rework (재작업)**: Author가 발견된 결함을 수정.
6.  **Follow-up (사후 검토)**: Moderator가 수정이 완료되었는지 확인.

**📢 섹션 요약 비유**: 요구사항 검증 과정은 **출판된 책을 내기 전 교정 과정**과 같습니다. '인스펙션'은 교정 교사들이 모여 빨간 펜으로 오탈자를 빠짐없이 찾아내는 '공식 정오 회의'와 같고, '워크쓰루'는 작가 친구들이 모여 "이 문맥이 자연스러워?"라며 가볍게 의견을 나누는 '독서 모임'과 같습니다. 정적 검토는 책이 인쇄되어 나가기 전에(배포 전에) 오류를 잡는 가장 효율적인 방법입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

요구사항 검증 기법은 단순히 문서를 읽는 행위를 넘어, 프로젝트의 성공과 실패를 가르는 중요한 품질 관리(Quality Management) 활동이다. 특히 현대의 애자일(Agile) 환경과 전통적인 폭포수(Waterfall) 환경에서 V&V의 적용 방식은 큰 차이를 보인다.

#### 1. 기술 비교: V&V와 테스팅

| 비교 항목 | Verification (검증) | Validation (확인) |
|:---|:---|:---|
| **검증 시점** | SDLC 초기(요구/설계)부터 지속적 | SDLC 후반(구현/시스템) 단계 |
| **테스트 유형** | 정적 분석 (Static Analysis), 리뷰 | 동적 테스트 (Dynamic Testing) |
| **도구 예시** | **LINT**, 정적 코드 분석 툴(SonarQube, Checkmarx) | 단위 테스트(JUnit), 통합 테스트, 사용자 UAT |
| **결함 유형** | 논리적 오류, 명세 불일치, 코딩 표준 위반 | 요구 누락, UI/UX 문제, 성능 저하 |
| **주요 지표** | 결함 밀도(Defect Density), 커버리지 | 결함 제거 효율(DRE), 사용자 만족도 |

#### 2. 방법론 융합: Agile vs Waterfall

```ascii
[방법론별 V&V 적용 빈도 및 방식]

    [Waterfall (폭포수)]               [Agile (애자일)]
  ─────────────────────           ──────────────────────
  Phase Gate 방식                   Iteration(반복) 방식
        │                                 │
        ▼                                 ▼
    ┌───────┐      (긴 간격)        ┌─────────────┐
    │ 계획  │─────────────>         │ 1. Sprint   │──┐
    └───────┘                      └─────────────┘  │
        │                                            │ (매일/주)
        ▼                                            ▼
    ┌───────┐      (긴 간격)        ┌─────────────┐  │
    │ 설계  │─────────────>         │ 2. Sprint   │──┤
    └───────┘                      └─────────────┘  │
        │                                            ▼
        ▼                                      ┌─────────────┐
    ┌───────┐      (긴 간격)                │ 3. Review   │←─┘ (지속적 피드백)
    │ 구현  │─────────────>                └─────────────┘
    └───────┘
        │
        ▼
   [최종 테스트] (한 번의 기회)

  ※ Waterfall은 각 단계 종료 시 '확인(Validation)'이 중요하며,
  Agile은 짧은 주기마다 '검증(Verification)'과 '리뷰'가 지속적으로 일어남.
```

*   **Waterfall 모델**: 각 단계(Sign-off)가 끝날 때 대규모 **인스펙션(Inspection)**을 통해 확실한 넘겨주기(Hand-off)를 수행한다. 이때의 V&V는 단계 간의 'Gatekeeper' 역할을 한다.
*   **Agile 모델**: 문
