# 181-190. 현대적 요구사항 기법과 AI (Story Map, Lean Canvas, BDD)

> **핵심 인사이트**
> 1. **본질**: 요구사항 공학은 정적인 문서 작성이 아니라, **'사용자 경험의 흐름'을 시각화**하고 **'행동 중심의 공유 언어'**를 구축하는 과정으로 진화했다.
> 2. **가치**: **User Story Mapping**을 통해 릴리즈 파장(RW)을 최소화하여 시장 투입 시간을 단축하고, **BDD(Behavior-Driven Development)**로 개발-테스트 간의 **Translation Loss(전달 손실)**를 0%에 가깝게 낮춘다.
> 3. **융합**: **LLM(Large Language Model)** 기반의 생성형 AI를 도입하여 비정형 요구사항 데이터를 정형 **SRS(Software Requirements Specification)**로 자동 변환함으로써 요구공학의 생산성을 획기적으로 높인다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념 및 철학
현대 소프트웨어 요구사항 정의는 **"What(무엇)"**을 나열하는 수준을 넘어, 사용자의 **"Why(왜)"**와 **"How(어떻게)"**의 행동 패턴을 모델링하는 방향으로 패러다임이 전이되었습니다. 전통적인 **Waterfall Model(폭포수 모델)** 하에서의 두꺼운 **SRS(Software Requirements Specification)** 문서는 변경에 유연하지 못하다는 비판을 받았습니다. 이를 해결하기 위해 **Agile Methodology(애자일 방법론)**에서는 요구사항을 **Backlog(백로그)** 형태로 관리하되, 이를 단순한 리스트가 아닌 **2차원 맵**과 **행위 기반 테스트**로 구체화하는 기법들이 주류가 되었습니다.

#### 💡 비유
건물을 지을 때, 예전에는 설계도면의 숫자만 나열했다면, 이제는 입주자가 살아가는 **'생활 동선 시뮬레이션'**을 먼저 그려보고, 그 다음 벽과 문의 구조를 설계하는 것과 같습니다.

#### 등장 배경
1.  **기존 한계**: 긴밀하게 변하는 비즈니스 환경에서 수백 페이지의 요구사항 문서는 **Obsolete(구식)**가 되기 일쑤였으며, 개발자와 기획자 간의 해석 차이로 인한 **Bug(결함)**가 발생했다.
2.  **혁신적 패러다임**: **Lean Startup(린 스타트업)** 운동과 **Agile(애자일)** 선언 이후, **MVP(Minimum Viable Product, 최소 기능 제품)**를 빠르게 만들고 검증하는 문화가 정착되면서 요구사항도 '흐름'과 '가치' 중심으로 재편되었다.
3.  **현재 요구**: 최근에는 **Generative AI(생성형 인공지능)**의 발전으로, 고객의 대화 내용을 자동으로 **User Story(사용자 스토리)**나 **Test Case(테스트 케이스)**로 변환하는 **Intelligent Requirements Engineering(지능형 요구공학)**이 요구되고 있다.

> **📢 섹션 요약 비유**: 애자일 요구사항 정의는 마치 넷플릭스가 영화를 '장르별로 분류(Theme)'하고, 시리즈를 '시즌별로 묶으며(Epic)', 각 회차를 '줄거리(Story)'로 요약하는 것과 같습니다. 이제는 AI가 시청자의 취향을 분석해서 다음 에피소드를 자동으로 작성해주는 시대로 접어들었습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

현대적 요구사항 기법은 **계층 구조(Hierarchy)**, **2차원 시각화(Visualization)**, **구조적 명세(Specification)**의 세 가지 핵심 축으로 구성됩니다.

#### 1. 구성 요소 (표)

| 요소명 | 역할 | 내부 동작 및 상세 | 프로토콜/형식 | 비유 |
|:---|:---|:---|:---|:---|
| **Theme (테마)** | 전략적 목표 | 비즈니스 목적을 달성하기 위한 가장 상위의 컨테이너. 여러 Epic을 포함함. | 전략 문서 | 폴더 |
| **Epic (에픽)** | 대형 기능 단위 | 하나의 스프린트(Sprint)에서 완료하기 어려운 큰 작업. 분해가 필요함. | 백로그 아이템 | 챕터 |
| **User Story (사용자 스토리)** | 구현 단위 | "사용자 역할(User Role) + 원하는 기능(Function) + 가치(Value)" 형식. | `~로서, ~하기 위해, ~것이다` | 문장 |
| **Task (태스크)** | 실행 단위 | 스토리를 실현하기 위한 개발자의 구체적 행동(코딩, DB 설계 등). | Jira Ticket, TODO | 단어 |
| **Acceptance Criteria (인수 조건)** | 완료 기준 | 스토리가 완료되기 위해 충족해야 하는 기준(주로 BDD 형식). | Given-When-Then | 체크리스트 |

#### 2. 요구사항 계층 구조 다이어그램

```ascii
+---------------------------------------------------------------+
|                    [ Vision / Business Strategy ]            |
+-----------------------------+---------------------------------+
                              │
                              ▼
+---------------------------------------------------------------+
|  [ THEME: Market Expansion ]  (전략적 방향)                   |
|                                                               |
|   +-------------------+        +-------------------+          |
|   |   EPIC A          |        |   EPIC B          |          |
|   | (User Management) |        | (Payment System)  |          |
|   +--------+----------+        +---------+---------+          |
|            |                            |                    |
|     +------+------+            +--------+--------+            |
|     | Story 1      |            | Story 1         |            |
|     | (Login)      |            | (Credit Card)   |            |
|     +------+------+            +--------+--------+            |
|            |                            |                    |
|     +------+------+            +--------+--------+            |
|     | Task 1,2     |            | Task 1,2        |            |
|     +------+------+            +--------+--------+            |
+---------------------------------------------------------------+
```

**[해설]**
위 다이어그램은 요구사항의 **Top-Down(하향식)** 분해 과정을 나타냅니다.
1.  가장 상위의 비전이 존재하고, 이를 **Theme**라는 큰 주제로 나눕니다.
2.  Theme는 여러 **Epic**으로 나뉘며, 각 Epic은 여전히 개발하기에 너무 큽니다.
3.  Epic은 **Sprint(스프린트)** 주기(보통 2주) 내에 구현 가능한 크기인 **User Story**로 쪼개집니다.
4.  각 User Story는 개발자가 실제로 작업할 수 있는 **Task** 단위로 분해되어 집행 단위가 됩니다.

#### 3. 핵심 알고리즘 및 작성 원칙: INVEST & 3C

**User Story**는 다음의 **INVEST** 원칙을 준수해야 하며, **3C** 프로세스를 거칩니다.

*   **INVEST Model**: Independent(독립적), Negotiable(협상 가능), Valuable(가치 있음), Estimable(추정 가능), Small(작음), Testable(테스트 가능).
*   **3C Process**:
    1.  **Card (카드)**: 요구사항의 핵심을 짧게 기록.
    2.  **Conversation (대화)**: 개발자와 기획자가 상세 내용을 구두로 소통.
    3.  **Confirmation (확인)**: 테스트를 통해 완료를 확인.

> **📢 섹션 요약 비유**: 요구사항을 관리하는 것은 마치 거대한 고기를 정리하는 것과 같습니다. **Epic**은 통째로 소를 사오는 것이고, **User Story**는 소를 부위별로(안심, 우둔살 등) 숙성가게에서 자르는 것이며, **Task**는 집에서 가위로 먹기 좋은 크기로 썰어 불판에 올리는 과정입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

이 섹션에서는 요구사항을 시각화하는 **Story Mapping**과 비즈니스를 검증하는 **Lean Canvas**, 그리고 명세를 자동화하는 **AI**를 융합 분석합니다.

#### 1. 심층 기술 비교: Story Map vs. Flat Backlog

| 비교 항목 | Flat Backlog (일반 백로그) | User Story Map (스토리 맵) |
|:---|:---|:---|
| **시각화** | 1차원 리스트 (Vertical) | 2차원 매트릭스 (Time vs. Priority) |
| **시야** | 일부 스토리에 집중 (Tunnel Vision) | 전체 **User Journey(사용자 여정)** 파악 가능 |
| **우선순위** | 상위에 있는 것만 우선 (순서 변경 번거로움) | 상위 행(MVP)부터 배포 계획 수립 용이 |
| **간섭 관리** | 중복/누락 발견 어려움 | 흐름상 끊어지는 지점(데드존) 발견 쉬움 |
| **주요 용도** | 스프린트 단위 실행 | 제품 릴리즈 계획, MVP 정의 |

#### 2. 린 캔버스 (Lean Canvas)와 요구사항의 연결

**Lean Canvas**는 비즈니스 모델을 9개 블록으로 요약한 것으로, 요구사항이 **'왜'** 필요한지를 정당성을 부여합니다.

1.  **Problem (문제 정의)**: 해결할 사용자의 페인 포인트 -> **(파생)** -> User Story의 `Value` 부분.
2.  **Solution (해결책)**: 제품의 핵심 기능 -> **(파생)** -> Epic 및 User Story.
3.  **Key Metrics (핵심 지표)**: 성공 측정 기준 -> **(파생)** -> Acceptance Criteria (KPI).

#### 3. AI 기반 요구사항 자동화 (LLM 활용)

최신 트렌드인 **LLM(Large Language Model)**을 활용한 요구공학 파이프라인은 다음과 같습니다.

**ASCII 다이어그램: AI-Driven Requirements Pipeline**

```ascii
    [Input Sources]               [ Processing (LLM) ]               [Output Artifacts]
    
 +------------------+       +-----------------------------+       +---------------------+
 | 1. Customer      |       | GPT-4 / Claude / Llama      |       | 1. Structured       |
 |    Interview Log |  -->  | (Context Understanding)     |  -->  |    User Stories     |
 +------------------+       |                             |       +---------------------+
 | 2. Competitor    |       | 1. Ambiguity Resolution     |       | 2. BDD Test Cases   |
 |    Analysis PDF  |  -->  | 2. Conflict Detection       |  -->  |    (Gherkin Syntax) |
 +------------------+       | 3. Gap Analysis             |       +---------------------+
 | 3. Legacy Code   |       +-----------------------------+       | 3. Sequence Diag.   |
 |    Comments      |             ▲      │                    |    (Auto-gen)        |
 +------------------+             │      │                    +---------------------+
                                 │      │
                     (Prompt Engineering) (RAG - Internal Docs)
```

**[해설]**
이 다이어그램은 비정형 데이터가 **SRS(Software Requirements Specification)**로 전환되는 흐름을 보여줍니다.
1.  **데이터 수집**: 고객 인터뷰 녹취록, 경쟁사 분석 PDF, 레거시 코드 주석 등을 입력으로 사용합니다.
2.  **LLM 처리**: 모델은 내부적으로 모호함을 해결하고, 요구사항 간의 충돌을 감출하며, 기능 갭(Gap)을 분석합니다.
3.  **결과물 생성**: 자연어 요구사항을 **Gherkin Syntax**(Given-When-Then)로 변환하여 **BDD(Behavior-Driven Development)** 테스트 코드를 자동 생성합니다.

> **📢 섹션 요약 비유**: **린 캔버스**는 요리사가 메뉴를 기획할 때 '이 메뉴가 돈이 되는가?'를 확인하는 **수익성 계산기**이고, **스토리 맵**은 식사 코스를 디자인하는 **플레이팅 도면**입니다. **AI 기반 자동화**는 요리사가 생각하는 재료 목록을 보고 자동으로 조리법(레시피)과 영양성분표(테스트 케이스)를 작성해주는 **스마트 키오스크**와 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 현대적 요구사항 기법을 도입할 때의 전략적 의사결정 프로세스를 다룹니다.

#### 1. 실무 시나리오: 스타트업의 핀테크 서비스 기획

*   **상황**: 신규 핀테크 앱을 3개월 안에 출시해야 하며, 요구사항이 매일 변경됨.
*   **문제점**: 기획서가 60페이지에 달해 개발자가 읽지도 않고, 기획자와 개발자가 '로그인'의 의미(단순 인증 vs 본인 인증 + KYC)를 달리 이해하여 개발이 지연됨.
*   **해결 과정**:
    1.  **린 캔버스** 작성 회의를 통해 불필요한 기능(혁신적 요소)을 제거하고 핵심 문제 해결에 집중.
    2.  **유저 스토리 맵** 워크숍을 통해 거래 흐름을 2차원으로 시각화하여 최소 기능(MVP) 범위를 정의 (상위 2개 행만 우선 개발).
    3.  **BDD 도입**: Cucumber와 같은 도구를 사용해 기획자가 작성한 시나리오(Given-When-Then)를 그대로 테스트 자동화 코드와 연결.
*   **결과**: 스펙 오차로 인한 재작업(Rework)이 40% 감소하고, 릴리즈 주기가 2배 빨라짐.

#### 2. 도입 체크리스트 (Checklist)

| 구분 | 체크항목 | 검증 포인트 |
|:---|:---|:---|
| **기술적** | **도구 호환성** | Jira, Confluence, Azure DevOps 등 현재 **ALM(Application Lifecycle Management)** 툴과의 연동성 |
| **기술적** | **테스트 자동화** | BDD 프레임워크(Cucumber, SpecFlow)가 CI/CD 파이프라인에 완전히 통합되었는가? |
| **운영적** | **참여자 교육** | PO(Product Owner)가 Gherkin
