# 295-301. 시스템 신뢰성과 결함 허용 설계

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 신뢰성 engineering은 단순한 '고장 방지'가 아닌, 수학적 확률 모델을 기반으로 한 위험 관리이며, 결함 허용은 고장을 전체 시스템의 붕괴로 연결되지 않게 격리하는 아키텍처적 핵심 전략입니다.
> 2. **가치**: MTBF (Mean Time Between Failures)를 최대화하고 MTTR (Mean Time To Repair)을 최소화하여 비즈니스 연속성을 보장하며, 특히 금융/항공/의료 분야에서 Safety-Critical System의 생명을 지키는 절대적 지표입니다.
> 3. **융합**: OS의 이중화(Dual System), 분산 DB의 샤딩(Sharding) 및 복제(Replication), 그리고 AI의 앙상블(Ensemble) 학습 기법까지 모두 이 '신뢰성 모델링'과 '결함 허용'의 수학적 원리를 기반으로 작동합니다.

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**
시스템 신뢰성(Reliability)은 시스템이规定된 기간 동안规定的된 조건에서 의도한 기능을 수행할 확률로 정의됩니다. 이는 단순한 '품질'의 개념을 넘어 **RAS (Reliability, Availability, Serviceability)**라는 정량적 지표로 관리됩니다.
결함 허용(Fault Tolerance)은 시스템의 일부 구성 요소에 결함(Fault)이 발생하거나 오류(Error)가 발생하더라도, 전체 시스템이 정상적인 서비스를 중단 없이 계속 제공할 수 있는 능력을 의미합니다.

**💡 비유**
시스템 신뢰성 설계는 '복잡한 교차로'를 설계하는 것과 같습니다. 아무리 사고가 나지 않게 하려 해도 결국 사고는 납니다. 중요한 것은 사고가 났을 때 전체 교통이 마비되지 않도록 우회 도로(중복성)를 만들어두고, 사고 차량을 신속하게 치우는 메커니즘(복구)을 갖추는 것입니다.

**등장 배경**
① **하드웨어적 한계**: 물리적 부품의 수명이 다하거나(Soft Error/Hard Error) 물리적 환경(전압 스파이크, 온도)에 의해 고장이 발생할 수밖에 없음.
② **소프트웨어 복잡도 폭증**: 모듈 간 결합도가 높아지면서 Side Effect로 인한 장애가 연쇄적으로 확산(Cascading Failure)되는 현상 발생.
③ **고가용성 요구**: 24/7 무중단 서비스가 필수인 핀테크, 클라우드 시대에서 단순 장애 예방을 넘어 '격리'와 '회복' 능력이 핵심 경쟁력으로 부상.

**📢 섹션 요약 비유**
우리가 자동차의 '브레이크'를 신뢰할 수 있는 이유는, 브레이크가 끊어지는 것을 막는 것만큼이나 끊어졌을 때를 대비해 '이중 파이프라인'과 '주차 브레이크'라는 비상장치를 설계해 두었기 때문입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

결함 허용 설계는 수학적 확률 모델 위에 구축됩니다.

**1. 신뢰도 모델링 수식 (Reliability Modeling)**
시스템 전체의 신뢰도 $R_{sys}$는 개별 부품의 신뢰도 $R_i$와 시스템 구조에 따라 결정됩니다. ($t$: 시간, $\lambda$: 고장률)

*   **직렬 시스템 (Series System)**: 모든 부품이 정상 작동해야 시스템이 살아있음. 가장 취약한 부품(Markov Chain의 Weak Link)이 전체를 좌우함.
    $$ R_{series}(t) = R_1(t) \times R_2(t) \times \dots \times R_n(t) $$
    *(Ex: 단일 엔진 비행기)*

*   **병렬 시스템 (Parallel System)**: 하나만 살아있어도 시스템 유지. **Active Redundancy** 기반.
    $$ R_{parallel}(t) = 1 - \prod_{i=1}^{n} (1 - R_i(t)) $$
    *(Ex: 쌍발 엔진 비행기)*

**2. 구성 요소 및 기술 분석표**

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 (Internal Mechanism) | 프로토콜/기법 (Protocol/Tech) | 비유 (Analogy) |
|:---|:---|:---|:---|:---|
| **Redundancy (중복성)** | 시스템의 여유 설비 확보 | Hot Standby(항시 대기) 또는 Cold Standby(수동 대기) 방식으로 예비 자원 투입 | RAID (Redundant Array of Independent Disks) | 스페어 타이어 |
| **Heartbeat (하트비트)** | 시스템 생존 여부 감지 | 주기적으로 신호를 송수신하여 응답이 없을 경우 Failover 판단 | Keepalived Protocol | 심박동 체크 |
| **Voting (투표기)** | 다수결로 결과 선정 | N개의 모듈 결과를 비교하여 다수가 맞는 결과를 채택(N-Modular Redundancy) | Triple Modular Redundancy (TMR) | 3심 재판 |
| **Checkpoint (체크포인트)** | 상태 저장 및 복구 지점 | 일정 주기마다 메모리/레지스터 상태를 안전한 저장소에 스냅샷 | Chandy-Lamport Algorithm | 게임 세이브 포인트 |

**3. 결함 허용 기술 아키텍처 (ASCII Diagram)**

```ascii
          [ 결함 허용 시스템의 계층적 아키텍처 ]

  +---------------------------------------------------------------+
  |                   Application Layer                            |
  |  (N-Version Programming: 서로 다른 알고리즘으로 동일 문제 해결)  |
  +---------------------------+-----------------------------------+
                              | 투표 (Voting)
  +---------------------------v-----------------------------------+
  |               Middleware / OS Layer (Failover Manager)         |
  |   +----------------+          +----------------+               |
  |   | Active Node    | <HB>     | Standby Node   |               |
  |   | (Primary)      |--------->| (Secondary)    |               |
  |   |  [CPU][Mem]    |  Heartbeat|  [CPU][Mem]    |               |
  |   +-------+--------+          +--------+-------+               |
  |           |                             |                       |
  +-----------|-----------------------------|-----------------------+
              | Shared Storage              |
  +-----------v-----------------------------v-----------------------+
  |            Data Layer (RAID / Replicated DB)                   |
  |    (Data Consistency & Durability 보장)                        |
  +---------------------------------------------------------------+
```

**[도해 설명]**
이 아키텍처는 크게 3단계로 결함을 격리합니다.
1.  **애플리케이션 계층**: **N-Version Programming** 기술을 적용하여 동일한 입력에 대해 서로 다른 로직으로 연산 후, **Voter(투표기)**를 통해 오류 가능성이 있는 결과를 사전에 필터링합니다. (논리적 결함 허용)
2.  **미들웨어/OS 계층**: **Active-Standby** 구조를 통해 Active 노드에 장애가 발생하면, **Heartbeat** 신호 단절을 감지한 매니저가 즉시 Standby 노드로 전환(**Failover**)하여 서비스 중단을 방지합니다. (물리적 결함 허용)
3.  **데이터 계층**: **RAID**나 **DB Replication**을 통해 데이터 자체의 손실을 방지합니다. 노드가 바뀌어도 데이터는 동일성을 유지해야 하므로 가장 깊이 있는 중복성이 필요합니다.

**핵심 알고리즘: Triple Modular Redundancy (TMR)**
3개의 모듈이 같은 연산을 수행하고 다수결(2개 이상 일치)을 채택합니다. 신뢰도 $R$인 부품 3개를 사용할 때, TMR 시스템의 신뢰도 $R_{TMR}$은 다음과 같습니다. (단, $R > 0.5$)

$$ R_{TMR} = R^3 + 3R^2(1-R) = 3R^2 - 2R^3 $$

**📢 섹션 요약 비유**
이는 마치 중요한 상을 수상하는 결심 과정과 같습니다. 심사위원 3명(N-Version)이 각자 다른 방식으로 심사를 하더라도, 결과가 틀린 사람 한 명 때문에 시상식을 멈추는 것이 아니라, 다수가 선택한 결과에 따라 상을 수여(Voting)하여 행사를 진행하는 것입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

시스템의 장애 대응 방식은 크게 '고장을 견디는 능력'과 '고장 났을 때의 행동 양식'으로 나뉩니다.

**1. 심층 기술 비교표**

| 구분 | **Fault Tolerance (결함 허용)** | **High Availability (고가용성)** |
|:---|:---|:---|
| **목표** | 장애 발생 시에도 **중단 없는 서비스** (Zero Downtime) | 장애 발생 시 **최소한의 다운타임** (RTO 최소화) |
| **복잡도** | 매우 높음 (하드웨어/소프트웨어 중복 설계 필요) | 상대적으로 낮음 (주로 소프트웨어/클러스터 레벨) |
| **비용** | 고비용 (Active-Active, TMR 등) | 상대적으로 저비용 (Active-Standby) |
| **적용 분야** | 항공 우주, 원자력 발전소, 의료 장비 | 웹 서비스, 클라우드, 일반 비즈니스 시스템 |
| **지표** | Safety(안전성), Reliability(신뢰성) | Availability (가용성: 99.99% 등) |

**2. Fail 기법 비교 및 시각화**

Fail 기법은 시스템이 장애를 겪었을 때 어떤 태도를 취하느냐를 정의한 설계 철학입니다.

```ascii
      [ 장애 발생 시 시스템 상태 전이도 (State Transition Diagram) ]

    (정상 상태: Normal State)
           │
           ├──> 장애 감지 (Fault Detected)
           │
   ┌───────┴───────┬────────────┬──────────────┐
   ▼               ▼            ▼              ▼
[Fail-Safe]    [Fail-Soft]   [Fail-Over]    [Fail-Silent]
 안전 정지       기능 저하      자동 전환       무응답
   │              │            │              │
   ▼              ▼            ▼              ▼
 [STOP]        [Degrade]     [Recovery]     [Timeout]
 피해 0         성능 하락      서비스 유지    타임아웃
```

*   **Fail-Safe (페일 세이프)**: 장애 시 시스템을 안전한 쪽으로 즉시 정지시킴. (Ex: 철도 신호기, 고압 가스 밸브)
    *   *원리*: Energy being removed logic.
*   **Fail-Soft (페일 소프트)**: 장애를 격리하여 일부 기능을 포기하더라도 핵심 서비스는 유지함. (Ex: 스트리밍 화질 낮추기, 쇼핑몰 장바구니 임시 비활성화)
*   **Fail-Over (페일 오버)**: 예비 시스템으로 서비스를 자동 이전함. (Ex: DB Replication 승격)

**3. 타 영역과의 융합 (OS & 네트워크)**
*   **OS와의 융합**: OS의 **Cluster** 기술(예: Linux Heartbeat)은 결함 허용의 소프트웨어적 구현체입니다. 하나의 노드가 죽으면 다른 노드가 IP(VIP)를 인수(Takeover)하는 과정은 Fail-Over의 정석입니다.
*   **네트워크와의 융합**: **OSPF (Open Shortest Path First)**나 **BGP (Border Gateway Protocol)** 프로토콜은 링크 장애 시 우회 경로를 설정하는 **네트워크 레벨의 결함 허용**을 제공합니다.

**📢 섹션 요약 비유**
시스템을 '사람'에 비유하자면, **결함 허용(Fault Tolerance)**은 '후각을 잃어도 후각을 대신할 수신을 동원하는' 능력이고, **Fail-Over**는 '오른손을 다쳤을 때 왼손으로 즉시 식사를 대신하는' 반사 신경이며, **Fail-Safe**는 '어지러워지면 무조건 바닥에 주저앉아 넘어지는 것을 방지하는' 안전 본능입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. 실무 시나리오: 대형 이커머스 장바구니 서비스 설계**

*   **상황**: `고객 주문 시스템`의 장애로 인한 주문 불가 사태 발생. 이 때 기술사로서 어떤 아키텍처를 제안할 것인가?
*   **의사결정 과정**:
    1.  **요구사항 분석**: 장바구니 서비스는 재고 관련 트랜잭션과 직결되어 있어, **데이터 정합성(Consistency)**이 최우선이다.
    2.  **전략 수립**: 단순한 로드 밸런싱(Load Balancing)만으로는 DB 트랜잭션 중단을 막을 수 없다.
    3.  **아키텍처 제안**: **Active-Active Cluster**와 **Fail-Over** 조합.
        *   웹 서버는 **Active-Active** (분산 처리)
        *   WAS는 세션 공유를 위한 **Sticky Session** 또는 세션 클러스터링
        *   DB는 **Fail-Over** 기반의 이중화 (Real-time Replication)

**2. 결함 허용 설계 체크리스트**

| 구분 | 항목 | 설명 |
|:---|:---|:---|
| **기술적** | **단일 장애점(SPOF) 제거** | 하나의 장비 고장으로 전체가 멈추는 지점이 없는지 검토 |
| | **Graceful Degradation** | 일부 기능 장애 시 핵심 기능만이라도 떨구지 않고 제공하는지 |
