# 405-411. 시스템, 인수 및 회귀 테스트

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 생명주기의 마지막 관문인 시스템 및 인수 테스트는 요구사항 충족 여부와 비즈니스 가치 실현을 검증하는 절대적 기준이다.
> 2. **가치**: 회귀 테스트(Regression Test)는 소프트웨어 진화 과정에서 발생하는 '의도치 않은 파장'을 차단하여 시스템의 신뢰성(Reliability)과 무결성(Integrity)을 유지하는 핵심 안전망이다.
> 3. **융합**: CI/CD (Continuous Integration/Continuous Deployment) 파이프라인 내 자동화된 회귀 테스트와 가상화된 인수 테스트 환경은 DevOps 및 클라우드 아키텍처와 결합하여 빠르고 안정적인 배포를 가능하게 한다.

---

### Ⅰ. 개요 (Context & Background)

**개념 및 철학**
소프트웨어 테스트는 단순히 결함을 찾는 과정이 아니라, **"만들어진 시스템이 명세된 요구사항을 충족하는가?"**를 엔지니어링적으로 증명하는 활동입니다. 단위(Unit) 및 통합(Integration) 테스트를 거쳐 모듈 간의 연결이 완료된 후, 우리는 **'전체 시스템'**이라는 거시적 관점과 **'최종 사용자'**라는 실용적 관점에서의 검증이 필요합니다. 이때 수행되는 것이 **시스템 테스트(System Test)**와 **인수 테스트(Acceptance Test)**입니다. 또한, 소프트웨어는 수정과 개선을 거듭하며 변경되는데, 이 과정에서 기존 기능이 망가지는 현상(회귀, Regression)을 방지하는 것이 **회귀 테스트(Regression Test)**입니다.

**등장 배경**
1.  **기존 한계**: 모듈 단위 테스트는 통과했으나, 전체 시스템 레벨에서 발생하는 성능 저하, 보안 취약점, 환경 의존적 오류가 발견되는 경우가 빈번했습니다.
2.  **혁신적 패러다임**: V-Model 등의 엔지니어링 모델을 통해 각 개발 단계별 검증을 체계화하고, 최종 단계에서의 '검증(Validation)'과 '확인(Verification)'의 중요성이 강조되었습니다.
3.  **현재의 비즈니스 요구**: 빠른 시장 변화 속에서 기능 추가나 버그 수정이 기존 서비스에 미치는 영향을 최소화하면서도, 사용자에게 품질 높은 제품을 인계해야 하는 요구가 증대되었습니다.

**💡 비유**
자동차 제조 공장을 떠올려 보겠습니다. 엔진, 타이어, 브레이크가 각각 잘 작동하더라도(단위 테스트), 조립된 차가 안전하게 도로를 달릴 수 있는지(시스템 테스트) 확인해야 하며, 실제 구매자가 운전해 보고 만족해야(인수 테스트) 출고할 수 있습니다. 또한, 엔진 튜닝을 했는데 브레이크가 고장 났다면(회귀 오류) 낭패를 보겠죠.

📢 **섹션 요약 비유**: 소프트웨어 개발의 완성도를 검증하는 과정은, 마치 **자동차 제조사가 조립 완료된 차량을 시험장에서 굴려보고(시스템), 고객에게 인도하기 전 시승을 시키며(인수), 이후 정비를 할 때마다 다른 부품에 이상이 생기지 않았는지 전수 점검하는(회귀) '최종 품질 관리 시스템'과 같습니다.**

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

이 섹션에서는 각 테스트 유형의 구성 요소, 내부 동작 메커니즘, 그리고 실무적인 수행 절차를 심도 있게 분석합니다.

#### 1. 구성 요소 및 역할

| 요소명 | 역할 | 내부 동작 및 특징 | 주요 프로토콜/도구 | 비유 |
|:---|:---|:---|:---|:---|
| **ST (System Test)** | 요구사항 충족 여부 전체 검증 | 블랙박스 테스트 기반, 기능/비기능(성능, 보안) 전수 검증 | JIRA, Selenium, JMeter | 자동차 전자제어장치(ECU) 통합 점검 |
| **AT (Acceptance Test)** | 최종 사용자 인수 결정 | UAT(User Acceptance Test), 실제 비즈니스 프로세스 시뮬레이션 | QTP, Cucumber (BDD) | 고객의 차량 인수 검사 |
| **RT (Regression Test)** | 변경으로 인한 부작용 방지 | 이미 통과한 테스트 케이스 재실행 또는 영향도 분석 기반 실행 | Jenkins CI, AutoIT | 정비 후 안전 장치 재확인 |
| **Test Harness** | 테스트 실행 및 자동화 환경 | Stub/Driver를 통해 모듈을 제어하고, 로그를 수집하는 프레임워크 | JUnit, TestNG | 테스트 주행장비 및 로봇 |

#### 2. 테스트 수행 레벨 및 흐름 아키텍처

아래 다이어그램은 소프트웨어가 개발된 후 배포되기까지의 테스트 레벨(Layer)과 데이터 흐름을 도식화한 것입니다.

**다이어그램 해설**:
1.  **Integration Test (통합 테스트)**: 개별 모듈이 합쳐질 때 발생하는 인터페이스 오류를 찾아냅니다. 이를 통과해야 비로소 '시스템'이 구성됩니다.
2.  **System Test (시스템 테스트)**: 완성된 시스템에 대해 기능적(Functional) 요구사항뿐만 아니라, 성능(Performance), 보안(Security) 같은 비기능적(Non-Functional) 요구사항을 포괄적으로 검증합니다. 이때는 내부 구조를 모르는 **블랙박스 테스트(Black-box Test)** 방식이 주로 사용됩니다.
3.  **Acceptance Test (인수 테스트)**: 사용자의 관점에서 실제 시나리오를 수행하며 비즈니스 요구를 충족하는지 확인합니다. 이 단계를 통과해야 배포(Deployment)가 승인됩니다.
4.  **Regression Test (회귀 테스트)**: 배포 후 운영 중이나, 새로운 버전 개발 시(CI/CD 파이프라인 상), 코드 변경으로 인해 기존의 정상적이던 기능에 문제(Side Effect)가 발생하지 않았는지 확인하는 피드백 루프입니다.

#### 3. 핵심 심층 동작 원리: 회귀 테스트 전략 (Selection & Impact)

회귀 테스트는 모든 테스트를 다시 수행하기에 비용이 너무 큽니다. 따라서 **영향도 분석(Impact Analysis)**을 통해 효율성을 확보해야 합니다.

*   **Retest All Strategy**: 모든 테스트 케이스를 재실행합니다. 가장 안전하지만 시간과 비용이 과다하게 소요되어 대규모 시스템에서는 비현실적입니다.
*   **Selective Strategy**: 변경된 모듈과 연관된 테스트 케이스만 선별적으로 수행합니다. 이를 위해 **의존성 매트릭스(Dependency Matrix)**를 활용합니다.

```python
# [Pseudo Code] Impact Analysis based Regression Test Selection
def select_regression_tests(changed_modules, all_tests, dependency_map):
    """
    변경된 모듈(changed_modules)에 의해 영향받는 테스트 목록을 추출합니다.
    dependency_map: {Module_ID: [Test_IDs]}
    """
    affected_tests = set()
    
    for module in changed_modules:
        # 의존성 맵에 있는 모듈이라면 연결된 테스트 케이스 추출
        if module in dependency_map:
            for test_id in dependency_map[module]:
                affected_tests.add(test_id)
                
    return list(affected_tests)

# Example Usage
# changed_modules = ['AuthModule', 'DB_Schema_User']
# selected_tests = select_regression_tests(changed_modules, ..., dependency_map)
# print(f"Execute Regression Tests: {selected_tests}")
```

**핵심 메커니즘**:
1.  **변경 감지**: 버전 관리 시스템(Git 등)과 연동하여 변경된 소스 코드의 범위를 식별합니다.
2.  **의존성 추적**: A 모듈이 변경되면 B 모듈을 거쳐 C 테스트가 실패할 수 있는 '집합'을 빠르게 계산합니다.
3.  **우선순위 부여**: 비즈니스 임팩트가 큰 'Critical Path'상의 테스트를 먼저 수행하여 핵심 기능의 회귀 여부를 조기에 탐지합니다.

📢 **섹션 요약 비유**: 회귀 테스트의 전략적 수행은 마치 **고속도로에서 특정 구간의 도로를 보수 공사할 때, 그 도로와 연결된 모든 진입로(링크)를 미리 시뮬레이션하여 정체가 발생할 수 있는 구간을 선별적으로 통제하고, 우회 도로(Critical Path)가 끊기지 않았는지 확인하는 '첨단 교통 통제 시스템'과 같습니다.**

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

테스트 전략은 단순한 검증을 넘어 비용(Cost), 품질(Quality), 시간(Time)의 삼위일체를 최적화하는 과정입니다.

#### 1. 테스트 레벨 심층 비교 (System vs Acceptance)

| 구분 | System Test (시스템 테스트) | Acceptance Test (인수 테스트) |
|:---|:---|:---|
| **시점** | 통합 테스트 완료 후, 출시 전 | 시스템 테스트 완료 후, 운영 이양 전 |
| **시행자** | 개발 조직 내 QA (Quality Assurance) 팀 | **고객(Customer)** 또는 최종 사용자(User) |
| **관점** | 개발자/스펙 관점 (Specification) | 사용자/비즈니스 관점 (Usage/Value) |
| **범위** | 기능 + 성능 + 보안 + 회복력 등 기술적 요소 전반 | 실제 업무 프로세스, UI/UX 편의성, 법적 규제 준수 |
| **환경** | 개발사의 테스트 환경 (Staging)에 준하는 환경 | 실제 운영 환경(Production)과 동일하거나 근접한 환경 |
| **통과 기준** | SRS (Software Requirement Spec) 만족 여부 | 계약상의 인수 기준 및 ROI 충족 여부 |

#### 2. 타 과목 융합 관점

1.  **SW Engineering + OS (Operating System)**:
    *   시스템 테스트의 **성능 테스트(Performance Test)**는 OS의 커널 레벨 자원 할당(CPU Scheduling, Memory Paging)을 분석합니다. 소프트웨어의 응답 속도가 느려질 때, 이가 **애플리케이션 로직의 문제인지, 혹은 OS의 Thrashing 현상인지**를 판단하는 융합적 사고가 필요합니다.
2.  **SW Engineering + DB (Database)**:
    *   회귀 테스트 수행 시 데이터베이스의 무결성(Data Integrity)이 중요합니다. 테스트 데이터(Test Data)의 **Rollback** 전략과 트랜잭션 격리 수준(Isolation Level)을 설정하지 않으면, 반복되는 테스트가 쓰레기 데이터(Garbage Data)를 양산하여 신뢰성을 떨어뜨립니다.
3.  **SW Engineering + AI (Artificial Intelligence)**:
    *   최근에는 **AI 기반 테스트 생성(AI-based Test Generation)**이 활용됩니다. 과거의 버그 데이터를 학습하여 회귀 버그가 발생할 확률이 높은 코드 구간을 예측하고 테스트 케이스를 추천해 줌으로써, 인간 테스터의 영향도 분석 부담을 줄이는 시너지를 창출합니다.

📢 **섹션 요약 비유**: 시스템 테스트와 인수 테스트의 관계는 **자동차 엔지니어와 운전자의 시선 차이**와 같습니다. 엔지니어는 엔진 회전수(RPM)와 연비(스펙)를 보고 기술적 완벽성을 판단하지만, 운전자는 운전대의 감각과 좌석의 편안함(체감)을 보고 만족도를 결정합니다. 두 관점이 모두 충족될 때 비로소 완벽한 자동차가 탄생합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무 현장에서는 무한한 자원을 쓸 수 없으므로, **위험 기반 테스트(Risk-Based Testing)**와 **자동화 전략**이 필수적입니다.

#### 1. 실무 시나리오 및 의사결정 매트릭스

**상황**: 금융권 핀테크 서비스의 핵심 결제 모듈에 로직 변경이 발생함.

*   **문제점**: 결
