# 397-404. 단위 및 통합 테스트 전략

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: **단위 테스트(Unit Test)**는 소프트웨어의 최소 기능 단위(모듈, 함수, 클래스)를 격리하여 내부 로직의 정확성을 검증하는 화이트박스 테스트이며, **통합 테스트(Integration Test)**는 이러한 단위들이 결합되는 지점에서 발생하는 인터페이스 오류와 데이터 흐름을 검증하는 단계이다.
> 2. **가치**: 결함을 조기에 발견하여 **수정 비용(Cost of Fix)**을 기하급수적으로 낮추며, 특히 모듈 간 의존성을 제어하는 **테스트 대역(Test Double, Stub/Driver)** 전략은 병렬 개발을 촉진하고 시스템 안정성을 확보하는 핵심 기술이다.
> 3. **융합**: CI/CD 파이프라인의 **테스트 자동화(Testing Automation)**와 연계되며, **정적 분석(Static Analysis)** 도구와 결합하여 코드 품질을 사전에 관리하는 모던 소프트웨어 엔지니어링(MSA)의 필수 프로세스이다.

---

### Ⅰ. 개요 (Context & Background)

소프트웨어 개발 수명 주기(SDLC, Software Development Life Cycle)에서 테스트는 단순한 오류 탐지를 넘어 시스템의 신뢰성을 보장하는 핵심 프로세스입니다. **단위 테스트(Unit Test)**는 시스템의 최소 단위인 모듈(Module)이나 컴포넌트(Component)가 명세서대로 정확히 동작하는지를 개발자 주도로 검증하는 과정입니다. 이는 내부 구조와 로직을 드러내어 검증하는 **화이트박스 테스트(White-box Test)**에 속합니다.

반면, **통합 테스트(Integration Test)**는 단위 테스트를 통과한 개별 모듈들을 결합(Combine)하여, 모듈 간 인터페이스(Interface)에서 발생할 수 있는 데이터 손실, 프로토콜 불일치, 타이밍 오류 등을 찾아내는 과정입니다. 모듈이 개별적으로는 정상 작동하더라도, 통합 시도 시 **사이드 이펙트(Side Effect)**나 **커플링(Coupling)** 문제로 실패하는 경우가 빈번하기 때문입니다.

이 과정에서 미완성된 모듈이나 실제 환경 구현이 어려운 요소를 대신할 **스텁(Stub)**과 **드라이버(Driver)**와 같은 **테스트 대역(Test Double)**의 활용 능력이 시스템의 출시 기간을 단축시키는 결정적 기술이 됩니다.

> **💡 비유**: 자동차 제작 과정에서 엔진, 타이어, 핸들을 각각 정밀 검사(단위 테스트)하는 것은 기본이지만, 이들을 조립했을 때 핸들이 바퀴를 제대로 조종하는지 확인(통합 테스트)하지 않으면 차는 운행될 수 없습니다.

#### 📢 섹션 요약 비유
마치 건물을 지을 때, 벽돌(단위) 하나하나의 강도를 테스트하는 것은 **단위 테스트**이고, 이 벽돌들을 쌓아 올려 기둥과 벽이 서로 지지하고 있는지 접합 부분을 검사하는 것은 **통합 테스트**와 같습니다. 재료는 좋아도 붙인 곳이 무너지면 건물이 무너지기 때문입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

테스트의 깊이를 확보하기 위해서는 모듈 간 의존성을 제어하는 기술이 필수적입니다. 특히 통합 테스트 시 **아직 개발되지 않은 모듈**을 시뮬레이션하는 기술이 아키텍처의 핵심입니다.

#### 1. 핵심 구성 요소 및 테스트 대역 (Test Double) 비교
하나의 모듈을 테스트하기 위해 주변 모듈의 도움이 필요할 때, 이를 대체하는 가상의 모듈을 **테스트 대역**이라 합니다.

| 요소명 (Element) | 역할 (Role) | 사용 통합 전략 | 내부 동작 (Behavior) | 프로토콜/특징 |
|:---|:---|:---|:---|:---|
| **Stub (스텁)** | 하위 모듈 대역 | **하향식 (Top-down)** | 상위 모듈의 호출(Call)에 응답하여 **미리 정의된 값( Dummy Data)**만 반환 | "가짜 서버", 수동적(Passive) |
| **Driver (드라이버)** | 상위 모듈 대역 | **상향식 (Bottom-up)** | 테스트 대상 모듈에게 **입력(Input)을 주고 결과를 검증** | "가짜 클라이언트", 능동적(Active) |
| **Test Target (테스트 대상)** | 검증 대상 | 중간 | 실제 비즈니스 로직 수행 | 실제 코드(Real Code) |

#### 2. 통합 테스트 전략 아키텍처 ASCII
통합 테스트는 크게 하향식, 상향식, 그리고 혼합형(Sandwich)으로 나뉩니다. 각 방식에 따라 Stub과 Driver의 배치가 결정됩니다.

```ascii
         [ Top-Down Integration (하향식) ]
        (주요 제어 로직부터 검증, 하위 Stub 필요)
    +------------------------------------------+
    |  [Main Controller] (Real Code)           | <--- 테스트 시작점
    +----------------+-------------------------+
                     | 1. Call
    +----------------v-------------------------+
    |  [Module A] (Real Code)                  |
    +----------------+-------------------------+
                     | 2. Call (Interface)
    +----------------v-------------------------+
    |  [Sub-Module STUB] (Fake Code)           | <--- 하위 모듈 미완성 시 대체
    |  -> Returns: "Fixed Test Data"           |
    +------------------------------------------+


         [ Bottom-Up Integration (상향식) ]
        (기초 모듈부터 검증, 상위 Driver 필요)
    +------------------------------------------+
    |  [TEST DRIVER] (Fake Code)               | <--- 하위 모듈 호출 및 검증
    |  -> Input: "Test Data A"                 |
    +----------------+-------------------------+
                     | 1. Call
    +----------------v-------------------------+
    |  [Low-Level Module] (Real Code)          |
    +----------------+-------------------------+
                     | 2. Return Result
    +----------------v-------------------------+
    |  [TEST DRIVER] Checks Result             |
    +------------------------------------------+
```

**다이어그램 해설**:
1.  **하향식(Top-Down)**: 메인 루프나 주요 제어 로직이 복잡할 때 유리합니다. 상위 모듈이 개발되었으나 하위 모듈이 준비되지 않았다면, **Stub**을 만들어 하위 모듈이 "정상적으로 응답하는 것처럼" 가정하여 테스트를 진행합니다. 이때 주의할 점은 Stub의 로직이 너무 단순하면 실제 하위 모듈과의 연동 시 버그를 놓칠 수 있다는 점입니다.
2.  **상향식(Bottom-up)**: 데이터 처리나 핵심 알고리즘이 복잡할 때 유리합니다. 가장 하위 모듈을 먼저 개발하고, 이를 호출해줄 **Driver**를 만들어 테스트합니다. Driver는 단순히 입력값을 넣고 출력값을 확인하는 반복적인 수행을 자동화할 수 있습니다.

#### 3. 심층 동작 원리 및 코드 예시
**스텁(Stub)**과 **드라이버(Driver)**의 구현적 차이를 코드 수준에서 이해해야 합니다.

*   **Stub 로직**: "나는 호출되면 무조건 성공(0 혹은 True)을 반환한다." (상태 반환 중심)
*   **Driver 로직**: "나는 대상 모듈에 여러 파라미터를 넣고, 리턴값이 기대값과 같은지 Assert(단정)한다." (행위 검증 중심)

```python
# [Python Pseudo-code]

# 1. Stub (하위 모듈 대역)
def get_database_connection_stub(config):
    # 실제 DB에 연결하지 않고, 가상의 Connection 객체 반환
    # -> 하향식 통합 시 "DB 연결 성공" 가정
    print("[Stub] DB Connection Faked.")
    return {"status": "connected", "ping": "10ms"} 

# 2. Test Target (중간 모듈)
def fetch_user_data(user_id, db_connector):
    conn = db_connector() # Stub 호출
    if conn["status"] == "connected":
        return f"Data for {user_id}"
    return "Error"

# 3. Driver (상위 모듈 대역/테스트 러너)
def test_driver():
    # 테스트 케이스 정의
    test_cases = ["user_01", "user_02", "admin"]
    
    # 루프를 돌며 테스트 대상 호출 및 결과 검증
    for user in test_cases:
        result = fetch_user_data(user, get_database_connection_stub)
        assert "Data" in result # 결과 검증
        print(f"[Driver] Test Passed for {user}")
```

#### 📢 섹션 요약 비유
하향식은 CEO(메인 모듈)가 먼저 취임하고, 아직 부서장이 없는 빈 자리에 임시 직원(Stub)을 앉혀서 업무 흐름이 맞는지 확인하는 방식입니다. 상향식은 말단 직원(하위 모듈)들이 먼저 입사하여 실력을 검증받고, 이들을 관리할 가상의 팀장(Driver)이 팀원들을 시뮬레이션하여 업무 능력을 평가하는 방식입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

단위 테스트와 통합 테스트의 선택은 프로젝트의 성격과 아키텍처 스타일(MSA vs Monolithic)에 따라 달라집니다.

#### 1. 심층 기술 비교 (Unit Test vs Integration Test)

| 비교 항목 | 단위 테스트 (Unit Test) | 통합 테스트 (Integration Test) |
|:---|:---|:---|
| **관점** | 미시적 (Micro) | 거시적 (Macro) |
| **대상** | 함수, 메서드, 클래스 단일 기능 | 모듈 간 인터페이스, 데이터 흐름 |
| **결함 발견** | 로직 오류, 경계값 오류, 알고리즘 오류 | 인터페이스 불일치, 데이터 손실, 타이밍 |
| **실행 속도** | 매우 빠름 (ms 단위) | 상대적으로 느림 (외부 리소스/네트워크 포함) |
| **비용** | 낮음 (개발자가 코드 작성 중 바로 수행) | 높음 (환경 구축, 복잡한 디버깅 필요) |
| **Mocking** | 외부 의존성을 차단하고 격리 (Isolation) | 실제 의존성을 통합하거나 Stub/Driver 사용 |

#### 2. 과목 융합 관점 (OS & Network)
*   **OS (운영체제) 융합**: 통합 테스트 시 **Deadlock(교착 상태)**이나 **Race Condition(경쟁 상태)**과 같은 문제는 단위 테스트에서는 발견하기 어렵습니다. 스레드 간의 동기화가 이루어지는 지점인 통합 과정에서만 이러한 병행 제어(Concurrency) 이슈가 드러납니다.
*   **Network (네트워크) 융합**: 마이크로서비스 아키텍처(MSA, Microservices Architecture) 환경에서 통합 테스트는 네트워크 지연(Latency)이나 패킷 손실과 같은 **API 통신 오류**를 검증해야 합니다. 이때 **Service Virtualization (서비스 가상화)** 기술이 고도화된 Stub의 형태로 사용됩니다.

#### 📢 섹션 요약 비유
단위 테스트는 현미경으로 세포 하나하나를 관찰하여 세균(버그)을 찾는 **생물학 실험**이라면, 통합 테스트는 여러 장기가 연결되어 혈액이 흐르는지 확인하는 **수술실 시나리오**와 같습니다. 현미경으로는 혈류의 속도나 막힌 혈관을 찾을 수 없듯이, 각각의 관점이 다릅니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서는 무조건적인 테스트가 아닌, 효율적인 테스트 전략 수립이 필요합니다.

#### 1. 실무 시나리오 및 의사결정 매트릭스
*   **상황 A: 새로운 핵심 알고리즘 개발**
    *   **전략**: 알고리즘 자체의 복잡도가 높으므로 **상향식(Bottom-up)** 접근이 유리합니다. 기본 함수를 Driver로 완벽히 검증한 후 조립합니다.
*   **상황 B: UI/UX가 중요한 웹 애플리케이션 개발**
    *   **전략**: 사용자 시나리오 흐름이 중요하므로 **하향식(Top-down)** 접근을 사용합니다. 메인 화면부터 Stub을 통해 프로토타입을 빠르게 보여주며 통합합니다.

#### 2. 도입 체크리스트 (Checklist)
*   [ ] **테스트 커버리지(Test Coverage)**: 최소 80% 이상의 라인 커버리지 목표.
*   [ ] **자동화(Automation)**: Jenkins, GitLab CI 등과 연동하여 코드 커밋 시 자동 실행.
*   [ ] **테스트 데이터 관리**: Stub과 Driver가 사용할 데이터 셋(Data Set)이 현실성 있는지 확인.

#### 3. 안티패턴 (Anti-Pattern)
*   **Stub Over-reliance**: 너무 많은 기능을 Stub으로 대체하면, 실제 통합 시 "작동하지 않는(Nothing Works)" 상태에 빠질 수 있습니다. Stub은 최소한의 응답만 제공해야 합니다.
*   **Big Bang Integration**: 모든 모듈을 다 만든 뒤 한 번에 테스트하는 방식은 **핀셋으로 바늘 찾기**와 같습니다. 오류 위치 파악이 불가능하여 절대 권장하지 않습니다.

#### 📢 섹션 요약 비유
자동차를 조립할 때 모든 부품을 다 맞춰놓고 시동을 걸어보는 것(Big Bang)은, 시동이 걸리지 않았을 때 배터리 문제인지, 엔진 문제인지, spark plug 문제인지 알 수 없어 악몽과 같습니다. 부품마다(MSA) 테스트하고 조립하면서 연결 확인을 하는 것이 공장의 정상적인 라인입니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과 (ROI)

| 구
