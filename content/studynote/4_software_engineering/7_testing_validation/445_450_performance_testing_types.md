# 445-450. 성능 테스트의 유형과 목적

## 핵심 인사이트 (3줄 요약)
> **1. 본질**: 성능 테스트(Performance Test)는 시스템이 기능적 요구사항을 충족하는지를 넘어, 예상 및 예상치 못한 부하 상황에서의 **안정성, 신뢰성, 확장성(Scale-up/Scale-out)**을 검증하는 비기능적 테스트의 핵심이다.
> **2. 가치**: 사전에 병목 지점(Bottleneck)을 발견하고 시스템 한계점(Limit)을 명확히 함으로써, 서비스 장애로 인한 막대한 비즈니스 손실(매출 타격, 브랜드 신뢰도 하락)을 방지하고 최적의 자원 투자를 가능하게 한다.
> **3. 융합**: CI/CD 파이프라인 자동화, 클라우드의 오토스케일링(Auto-scaling) 정책 수립, APM(Application Performance Monitoring) 도구와의 연계를 통해 현대적인 DevOps 운영 기반을 구축한다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 철학**
소프트웨어가 단순히 "돌아가는 것"에서 "빠르고 안정적으로 돌아가는 것"으로 요구사항이 고도화됨에 따라, **성능 테스트(Performance Test)**는 선택이 아닌 필수 생존 전략이 되었다. 이는 단순히 속도를 측정하는 것이 아니라, 시스템의 **부하 처리 능력**, **한계점(Limit)**, **회복 탄력성(Resilience)**을 과학적으로 분석하는 활동이다. 기능적 결함이 없더라도 응답 지연이 발생하면 이용자는 이탈하므로, 실사용자의 트래픽 패턴을 시뮬레이션하여 잠재적인 위험을 사전에 제거해야 한다.

**2. 등장 배경 및 필요성**
- **① 기존 한계**: 과거의 몬리식 개발(Manual Test)이나 간단한 기능 테스트로는 대용량 트래픽 발생 시 발생하는 `Race Condition`, `Deadlock`, `Memory Leak` 등의 동적 오류를 발견하기 불가능했다.
- **② 혁신적 패러다임**: 24시간 365일 서비스가 상식이 된 시대에, 사소한 메모리 누수라도 장기간 누적되면 시스템 전체를 다운시키게 됨에 따라, 장기간의 부하 테스트와 극한 상황 테스트가 도입되었다.
- **③ 비즈니스 요구**: 쿠폰 발행이나 티켓 예매와 같이 순간 트래픽이 폭증하는 이벤트 비즈니스가 증가함에 따라, 순간 부하(Spike)와 장기 부하(Soak)에 대한 대응력이 곧 매출로 직결되게 되었다.

**3. 주요 용어 정리**
- **VU (Virtual User)**: 실제 사용자를 흉내 내기 위해 성능 테스트 도구가 생성하는 가상의 사용자 스레드.
- **TPS (Transactions Per Second)**: 서버가 1초 동안 처리하는 트랜잭션의 건수로, 시스템 처리 성능의 핵심 지표.
- **Latency (지연 시간)**: 요청이 전송되어 응답을 수신하기까지의 대기 시간.

📢 **섹션 요약 비유**: 자동차 엔지니어가 신차를 출시하기 전에 단순히 시동이 걸리는지 확인하는 것(기능 테스트)에 그치지 않고, 고속도로에서의 최고 속도, 장시간 주행 시 과열 여부, 충돌 시의 안전도를 실험하는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

성능 테스트는 단순히 요청을 보내는 것이 아니라, 측정 대상 시스템(SUT: System Under Test)의 행동 양식을 관찰하고 분석하는 정교한 과정이다.

**1. 성능 테스트 지향점별 세부 구성**
성능 테스트는 목적에 따라 다음과 같이 5가지 주요 유형으로 세분화된다.

| 구분 | 목적 (Objective) | 부하 패턴 (Load Pattern) | 주요 확인 포인트 (Key Metrics) | 비유 |
|:---|:---|:---|:---|:---|
| **부하 테스트**<br>(Load Test) | 예상 트래픽에서 시스템이 정상 작동하는지 확인 | 점진적 증가 (Ramp-up) → 목표 TPS 유지 | 응답 시간稳定性, 자원利用率(CPU/Memory) | 정원 15명 탑승 후 속도 저하 여부 확인 |
| **스트레스 테스트**<br>(Stress Test) | 시스템 파괴 시점 및 복구 능력 확인 | 설계 용량 초과 과부하 (Overload) | 붕괴 지점(Breaking Point), 데이터 무결성 | 30명을 태워 줄이 끊어지는지 확인 |
| **스파이크 테스트**<br>(Spike Test) | 급격한 트래픽 변화에 대한 반응 속도 확인 | 순간적 폭증 (Sudden Jump & Drop) | 응답 지연 발생 여부, 복구 속도 | 15명이 동시에 뛰어드는 충격 견디기 |
| **내구성 테스트**<br>(Endurance/Soak Test) | 장기간 운영 시 메모리 누수 등 확인 | 장시간 정상 부하 지속 (Long duration) | 메모리 릭, Garbage Collection 빈도 | 24시간 연속 주행 시 엔진 과열 여부 |
| **볼륨 테스트**<br>(Volume Test) | 대용량 데이터 처리에 따른 성능 저하 확인 | 데이터 양(Dataset Size) 증가 | 디스크 I/O, 배치 처리 시간, 검색 속도 | 짐을 만재하고 언덕을 오르는 힘 확인 |

**2. 성능 테스트 수행 프로세스 및 아키텍처**
성능 테스트는 일반적으로 **Generator(부하 생성자) → SUT(대상 시스템) → Monitor(모니터링)**의 구조를 갖는다.

```ascii
+-------------------+          +----------------------+          +-------------------+
|   Load Generator  |          |   System Under Test  |          |  Monitoring DB    |
| (JMeter / Locust) |          | (WAS + DB + Cache)   |          | (Prometheus/Grafana)|
+-------------------+          +----------------------+          +-------------------+
         |                              ^                               ^
         |  ① VU 스레드 생성 및 요청     |  ② 트래픽 처리               |  ③ 매트릭 수집
         +------------------------------>                               |
                                                                       |
         +---------------------------------------------------------------+
         |  ④ 보고서: TPS, Latency, Error Rate, Resource Usage Graph    |
         +---------------------------------------------------------------+
```

**[다이어그램 해설]**
위 구조는 성능 테스트의 표준적인 데이터 흐름을 도식화한 것이다.
1. **부하 생성(Generate)**: `Load Generator`는 `VU (Virtual User)`를 시뮬레이션하여 `SUT (System Under Test)`로 HTTP/TCP 요청을 쏜다. 이때 **Think Time(사고 시간)**을 설정하여 실제 사용자의 행동 패턴을 흉내 낸다.
2. **시스템 처리(Process)**: 대상 시스템(WAS, DB, 네트워크)은 들어오는 요청을 큐(Queue)에 쌓고 스레드(Thread)나 프로세스(Process)를 할당하여 처리한다. 이 과정에서 `Context Switching` 비용과 `Lock` 대기 시간이 발생한다.
3. **모니터링(Monitor)**: OS 레벨의 CPU, Memory, Disk I/O 네트워크 사용량과 WAS 레벨의 `Thread Pool` 상태, `JVM Heap` 메모리 상태를 실시간으로 수집하여 병목 지점을 식별한다.

**3. 핵심 지표 및 분석 로직**
성능 테스트 결과는 다음 3가지 지표의 상관관계로 분석한다.
- **Throughput (처리량)**: 시간당 처리 건수. 병목이 발생하면 처리량이 정체되거나 하락한다.
- **Latency (지연 시간)**: 요청에 대한 응답 속도. 부하가 증가할수록 지수적으로 증가하는 경향이 있다.
- **Error Rate (오류율)**: 부하가 한계를 넘어서면 HTTP 500 등의 오류가 발생하기 시작한다.

```python
# 가상의 부하 테스트 스크립트 예시 (Locust Framework)
from locust import HttpUser, task, between

class WebsiteUser(HttpUser):
    # 사용자 간 대기 시간 설정 (Think Time)
    wait_time = between(1, 5)

    @task(3) # 3:1 가중치로 자주 호출
    def index_page(self):
        self.client.get("/")

    @task(1)
    def about_page(self):
        self.client.get("/about")
```

📢 **섹션 요약 비유**: 성능 테스트는 마치 다리 위에 트럭을 한 대씩, 혹은 수십 대를 연달아 올려보내어 다리가 견디는지, 처지는지, 무너지는지를 확인하는 '교량 내구성 실험'과 같습니다. 특히 스파이크 테스트는 갑자기 다리 위로 군인들이 구보를 해서 다루는 것과 같은 극한 상황 시뮬레이션입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 성능 테스트 유형 심층 비교 분석**

| 비교 항목 | 부하 테스트 (Load) | 스트레스 테스트 (Stress) | 스파이크 테스트 (Spike) | 내구성 테스트 (Soak) |
|:---|:---|:---|:---|:---|
| **주요 목적** | 설계된 SLA 준수 여부 확인 | 시스템 파손 지점 및 복구력 확인 | 급격한 변화에 대한 안정성 확인 | 시간 경과에 따른 자원 누수 확인 |
| **부하 형태** | 점진적 상승 → 유지 (Ramp-up) | 급격한 과부하 → 초과 지속 | 순간적 폭증 → 급격히 감소 | 낮은 수준의 장시간 지속 |
| **지표 관점** | TPS, 응답 시간(RT) | CPU 100% 도달 시점, OOM 발생 | 부하 급변 시 쓰레드 풀 동작 | 메모리 사용량 추이, GC 횟수 |
| **결과물** | **"목표 성능을 달성함"** | **"1200 TPS에서 Deadlock 발생"** | **"동시 접속자 5만 시 Timeout 발생"** | **"8시간 후 메모리 누수로 장애 발생"** |

**2. 기술 스택 융합 분석**
- **① OS & Computer Architecture**: 성능 테스트는 단순히 애플리케이션의 문제가 아니다. **Context Switching(문맥 교환)** 빈도, **Cache Miss**, **Thrashing(스래싱)** 현상 등 OS 커널 레벨의 병목을 찾아내야 한다. 예를 들어, Load Average가 코어 수보다 높게 유지되면 CPU 병목임을 즉시 파악해야 한다.
- **② Database & Network**: 데이터베이스의 `Lock Contention`이나 `Index Scan` 여부, 네트워크의 `Bandwidth` 포화 상태, `TCP Retransmission` 비율 등을 분석하여 병목이 WAS인지 DB인지 네트워크인지 격리(Isolation)해야 한다.

📢 **섹션 요약 비유**: 부하 테스트가 '계절용 타이어'로 평소 주행을 테스트하는 것이라면, 스트레스 테스트는 '겨울철 빙판길 주행 테스트'이고, 내구성 테스트는 '자동차 10만 km 무상점검'과 같이 서로 다른 목적과 시점을 가진 종합 검진입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. BMT (Benchmark Test)를 통한 제품 선정 전략**
**BMT (Benchmark Test)**는 동일한 환경 조건하에서 서로 다른 벤더의 HW나 SW를 비교하여 최적의 성능을 가진 제품을 선정하는 과정이다. 기술사는 단순히 "A제품이 빠르다"는 결론을 내려서는 안 되며, **"단위 비용당 성능(Price/Performance)"**과 **"확장성"**을 함께 고려해야 한다.

**2. 실무 시나리오 및 의사결정**

| 시나리오 | 문제 상황 (Problem) | 기술사적 판단 (Decision) | 해결 방안 (Solution) |
|:---|:---|:---|:---|
| **쿠폰 오픈 이벤트** | 이벤트 시작 직전, 대기열 시스템의 응답 시간이 급격히 증가하여 페이지 로딩 실패 | **Connnection Pool 고갈** 및 DB **Lock 경합** 판단 | WAS 스레드 수 최적화 및 Redis를 활용한 분산 락(Distributed Lock) 적용 |
| **배치 시스템 장애** | 매일 새벽 2시 배치 작업 시 WAS 메모리 사용량이 점진적으로 증가하여 OOM 발생 | **Memory Leak** 판단 | JVM Heap Dump 분석 도구를 통해 미해제 참조 객체 탐지 및 코드 수정 |
| **BMT 진행** | 신규 DB 도입 시 A사(OLAP 특화)와 B사(OLTP 특화) 비교 | 업무 특성이 **OLTP(대량 트랜잭션)**임을 고려 | 단순 조회 속도가 빠른 A사보다 트랜잭션 처리량(TPS)이 높은 B사 선정 권고 |

**3. 성능 테스트 도입 체크리스트**
- **기술적 검토**:
  - [ ] 테스트 환경이 운영 환경과 동일한 사양(H/W, S/W Version)인가?
  - [ ] 캐시(Cache)는 초기화되었는가? (Warm-up 필요)
  - [ ] 로그 레벨이 운영과 동일한가? (DEBUG 모드는 성능 저하 유발)
- **운영/보안적 검토**:
  - [ ] 테스트 데이터가 개인정보 보호법에 위반되지 않는가? (가명 처리 필요)
  - [ ] 장애 발생 시 운영 서버로의 트래픽 전환(Failover) 절차가 수립되었는가?

**4. 안티패턴 (Anti-Pattern)
