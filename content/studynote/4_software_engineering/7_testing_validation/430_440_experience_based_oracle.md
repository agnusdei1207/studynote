# 430-440. 경험 기반 테스팅과 테스트 오라클

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 명세 기반 테스트의 한계를 넘어, 전문가의 암묵지(Implicit Knowledge)와 리스크 분석을 통해 문서화되지 않은 잠재적 결함을 발견하는 **경험 기반 테스트 전략**.
> 2. **가치**: 애자일(Agile) 환경에서 빈번한 요구사항 변경 유연성을 확보하며, **Heuristic (휴리스틱)** 오라클을 활용해 자동화가 어려운 복잡한 시스템의 결함 판별 정확도를 극대화함.
> 3. **융합**: 정적 분석 도구와의 결합을 통해 초기 결함 제거 비용을 절감하고, CI/CD 파이프라인 내에서 동적 테스트의 신뢰성을 보장하는 필터링 메커니즘 역할 수행.

---

### Ⅰ. 개요 (Context & Background)

**개념 및 정의**
경험 기반 테스팅(Experience-based Testing)은 요구사항 명세서(SRS)나 설계 문서에만 의존하는 기존 블랙박스 테스트의 기계적 한계를 극복하기 위해, 테스터가 과거에 축적한 도메인 지식, 기술적 직관, 그리고 유사 시스템에서의 결함 패턴을 적극 활용하는 테스트 접근법입니다. 이는 **SWEBOK (Software Engineering Body of Knowledge)** 에서 정의하는 테스트 설계 기법 중 하나로, 테스트 케이스 설계(Design)와 테스트 수행(Execution)이 별도의 단계가 아닌 동시에 일어나는 인지적 과정(Cognitive Process)에 중점을 둡니다.

특히, 현대의 소프트웨어 개발 life cycle이 **CI/CD (Continuous Integration/Continuous Deployment)** 와 같은 애자일 방법론으로 전환됨에 따라, 문서화보다는 코드 실체와 실제 동작에 집중하는 **ET (Exploratory Testing, 탐색적 테스팅)** 의 중요성이 대두되고 있습니다.

**💡 기술적 비유**
이는 베테랑 의사가 초진 환자의 검진 결과(명세서)만 보고 진단하는 것을 넘어, 환자의 표정, 과거 병력, 그리고 임상 감각을 통해 CT 촬영에서 잡히지 않는 희귀 질병을 발견해내는 '임상적 진단(Clinical Diagnosis)' 과정과 유사합니다.

**등장 배경 및 필요성**
1. **기존 한계**: 100% 코드 커버리지를 달성해도 '의도하지 않은 동작'은 잡아내지 못함(할당된 기능은 수행하나 시스템 전체의 맥락을 파괴하는 버그).
2. **혁신적 패러다임**: 테스터를 '시나리오 실행자'가 아닌 '리스크 탐색가(Risk Explorer)'로 격상시키는 전략적 전환.
3. **비즈니스 요구**: 단축된 출시 기간(Sprint) 내에서 문서화 비용을 줄이고 결함 발견률(Defect Detection Rate)을 극대화해야 하는 시장적 압박.

📢 **섹션 요약 비유**: 경험 기반 테스팅은 **'GPS 없이 낯선 도시를 여행하는 베테랑 여행가'**와 같습니다. 지도(명세서)는 없지만, 과거 여행 경험을 통해 골목길(엣지 케이스)에 숨겨진 맛집(기능)이나 위험 지역(버그)을 본능적으로 찾아내고 유연하게 대처하는 능력입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

경험 기반 테스팅은 단순히 '대충 해보는 것'이 아니라, 체계적인 **사고 사이클(Thinking Cycle)** 과 **테스트 오라클(Test Oracle)** 의 전략적 배치를 전제로 합니다.

**1. 구성 요소 상세 분석**

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 (Mechanism) | 관련 프로토콜/형식 | 비유 (Analogy) |
|:---|:---|:---|:---|:---|
| **Charter (차터)** | 탐색 명세서 | 무엇을 테스트할지, 어떤 버그를 찾을지 목적과 범위를 명시하여 테스팅이 산으로 가는 것을 방지 | 문서/자유 형식 | 탐사 보고서 |
| **Timebox (타임박스)** | 리소스 관리 | 정해진 시간(예: 90분) 내에 집중하게 하여 번아웃을 방지하고 효율을 극대화 | Scrum 단위 | 모래 시계 |
| **Session Report (회고록)** | 지식 공유 | 발견한 버그, 느낀 점, 추가된 아이디어를 기록하여 팀 지식화 | HTML/Log | 항해 일지 |
| **Heuristic (휴리스틱)** | 의사결정 규칙 | "CRUD SANS..." 등 검증해야 할 속성 목록을 기억하여 빠트린 부분 없이 점검 | 멘탈 리스트 | 스마트 쇼핑 리스트 |
| **Oracle (오라클)** | 참/거짓 판별 기준 | 현재 결과가 올바른지 판단할 확률적 기준(True/False/Maybe) 제공 | 비교 알고리즘 | 심사위원 |

**2. 탐색적 테스팅 (ET) 프로세스 ASCII 다이어그램**

탐색적 테스팅은 선형적인 프로세스가 아닌 **동시적이고 반복적인 루프(Loop)** 로 구성됩니다.

```ascii
         +-------------------------------------------------------+
         |              [ 탐색적 테스팅 싱글 루프 ]               |
         +-------------------------------------------------------+
             |                                      |
   (1) Learn & Plan                      (3) Observation
             |                                      |
     +-------v-------+                      +-------v-------+
     |   탐지 & 학습  | <------------------- |   수행 & 조작 |
     | (Learn/Test   |    (Real-time        |   (Execute &  |
     |   Chartering) |     Feedback)        |    Operate)   |
     +-------+-------+                      +-------+-------+
             |                                      |
             |                                      |
             v                                        v
    (4) Interpretation                         (5) Test Oracle
             |                                      |
     +-------v-------+                      +-------v-------+
     |   해석 & 분석  |                      | [판단] Pass?  |
     | (Reasoning)   |                      | (Consistency) |
     +---------------+                      +---------------+
                                                   |
              +---------------+---------------------+
              |               |                     |
              v               v                     v
         [Bug Log]    [New Charter Idea]      [Coverage Map]
```

**(해설)**
1. **Learn & Plan**: 테스터는 애플리케이션을 탐색하며 시스템의 동작 방식을 학습합니다.
2. **Execute & Operate**: 학습한 내용을 바탕으로 즉석에서 테스트 케이스를 수행합니다.
3. **Observation**: 시스템의 반응(Response)과 로그(Log), 화면 상태를 관찰합니다.
4. **Interpretation**: 관찰된 결과가 의도한 것인지, 혹은 새로운 정보인지 해석합니다.
5. **Test Oracle**: 해석된 결과를 바탕으로 **결함(Defect)** 여부를 판별합니다. 이때 "오류가 있다"는 확신이 서면 로그를 기록하고, 확신이 서지 않으면 다시 '학습' 단계로 피드백하여 탐색 범위를 좁힙니다.

**3. 테스트 오라클 (Test Oracle)의 심층 원리**

테스트 오라클은 테스트 대상의 실제 결과(Expected Result)가 올바른지를 판단하는 **검증 메커니즘**입니다. 현실적으로 모든 입력에 대해 정답을 아는 '참 오라클(True Oracle)'은 존재하기 어렵기 때문에, 공학적으로는 **Heuristic Oracle**을 주로 활용합니다.

**핵심 알고리즘: 일관성 오라클 (Consistency Oracle) 예시**
회귀 테스트(Automated Regression Test)에서 가장 널리 쓰이는 방식입니다.

```python
# [Pseudo Code] Consistency Oracle Logic
def validate_response(current_response, baseline_response, tolerance):
    """
    이전 버전의 정답셋(Baseline)과 현재 응답을 비교하여 허용 오차(Tolerance) 내의 변화만 허용.
    """
    if current_response == baseline_response:
        return "PASS (Consistent)"
    
    # 변화가 감지되었을 경우, 휴리스틱(경험적 규칙) 적용
    # 예: 타임스탬프, 세션 ID 등 동적으로 변해야 하는 속성은 무시
    delta = calculate_delta(current_response, baseline_response)
    
    if delta <= tolerance:
        return "PASS (Within Tolerance)"
    else:
        return "FAIL (Potential Regression Detected)"
```

📢 **섹션 요약 비유**: 경험 기반 테스팅 아키텍처는 **"요리사의 감각 루프"**와 같습니다. 레시피(명세)대로만 요리하는 것이 아니라, 맛을 보며(테스트 수행) 간이 부족하면 바로 소금을 더 넣고(Present 결과 해설), 다시 맛을 보는(Oracle 판단) 과정을 요리가 완성될 때까지(타임박스) 고속으로 반복합니다. 여기서 '오라클'은 바로 요리사의 혀입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 심층 기술 비교: 명세 기반 vs 경험 기반**

| 비교 항목 | 명세 기반 테스팅 (Specification-based) | 경험 기반 테스팅 (Experience-based) |
|:---|:---|:---|
| **입력(Input)** | SRS, ID, 설계 문서 | 테스터의 스킬, 시스템 동작 관찰, 유사 경험 |
| **결정 기준** | 명세된 요구사항 커버리지 (100% 달성 여부) | 발견된 결함의 심각도 및 리스크 완화 정도 |
| **Test Oracle** | 명확함 (True Oracle에 가까움) | 모호함 (Heuristic Oracle 활용) |
| **AI 활용도** | 낮음 (이미 정의된 경계) | **높음** (AI가 로그를 학습하여 변칙 패턴 탐지 지원) |
| **주요 지표** | Code Coverage, Pass Rate | **Defect Detection Rate (결함 발견율)** |
| **재현성** | 높음 (스크립트 존재 시) | 상대적으로 낮음 (테스터의 기억 의존) |

**2. 융합 관점 분석 (Convergence)**

*   **[SE + AI] 지능형 오라클 (AI-based Oracle)**: 최근 **NLP (Natural Language Processing)** 기술을 접목하여 테스터가 생성한 불확실한 테스트 결과 로그를 분석하고, "정상 패턴"을 학습하여 **Heuristic Oracle**의 정확도를 높이는 기술이 도입되고 있습니다. 이는 과거의 데이터(Cloud Log)를 기반으로 새로운 버그의 신뢰도를 점수화합니다.
*   **[SE + DevOps] 좌우명 테스트 (Touring Test)**: "The Essential Guide to Et"라는 책에서 언급하는 **'소프트웨어 투어(Software Tour)'** 개념은 경험 기반 테스트를 구조화하여 DevOps 문화에 녹여냅니다. 예를 들어, "욕심쟁이 기능 투어(Greedy Feature Tour)"는 최신 기능만 집공 공격하여 리스크를 빠르게 식별합니다.

📢 **섹션 요약 비유**: 명세 기반 테스트는 **"정해진 악보를 연주하는 오케스트라"**라면, 경험 기반 테스팅은 **"재즈 즉흥 연주(Jam Session)"**와 같습니다. 악보가 없어도 기본 코드(이론)를 바탕으로 서로 신호를 주고받으며(Collaboration), 그 순간의 흐름에 맞춰 가장 아름다운 하모니(결함 발견)를 만들어냅니다. 둘은 대립이 아니라 완벽한 하모니를 이루어야 합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. 실무 시나리오 및 의사결정 매트릭스**

*   **시나리오 A: 신규 기능의 스프린트 완료 직전**
    *   *상황*: 명세서에 없는 UI 애니메이션 버그 발견.
    *   *전략*: 문서화된 테스트 케이스(명세 기반)는 이를 못 잡음. 테스터의 '직관'으로 버그 리포트를 작성. 이때 "트레이드오프(Trade-off)" 분석 필요.
    *   *결정*: 버그의 시각적 영향도가 높으면 Fix, 낮으면 Backlog로 이동. **오라클의 정의**가 프로젝트의 성공을 좌우함.

*   **시나리오 B: 레거시 시스템의 문서 부재 상태**
    *   *상황*: 10년 된 시스템의 결함 분석. 문서가 전혀 없음.
    *   *전략*: **RBT (Risk-Based Testing)** 적용. 장애 발생 시 피해가 큰 핵심 모듈(결제, DB Write)에 경험 많은 시니어 테스터 투입.
    *   *결정*: 새로 문서를 만들 시간이 없으므로 '탐색적 테스팅'을 통해 결함 맵을 먼저 그리고, 그 결과를 새로운 명세서로 역설계(Reverse Engineering).

**2. 도입 체크리스트**

| 구분 | 체크항목 | 설명 |
|:---|:---|:---|
| **기술적** | **Heuristic List 보유 여부** | 테스터가 공통의 속성(예: FEW - CRUD, Exploratory, World)을 공유하고 있는가? |
| | **Tool Chain** | 캡처 도구(Screely, Snipping Tool), 세션 관리 툴(Pluke, Session Tester)이 준비되었는가? |
| **운영/보안** | **테스트 데이터 분리** | 탐색 과정에서 운영 DB에 영향을 주지 않도록 샌드박스 환경이 격리되었는가? |
| | **인증 정보 관리** | 테스터가 시스템의 권한을 넘어(Privilege Escalation) 탐
