# 412-419. 블랙박스 테스트 기법 (Boundary, Pairwise)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 내부 구조(Internal Structure)의 노출을 차단하고, 소프트웨어의 **입력(Input)**과 **출력(Output)**에 대한 **명세(Specification)**만을 기반으로 결함을 검출하는 기법.
> 2. **가치**: 무한한 입력 도메인 중 **동등 분할(Equivalence Partitioning)**과 **경계값 분석(Boundary Value Analysis)**을 통해 테스트 케이스를 최적화하여, 적은 비용으로 높은 결함 검출률(Detection Rate)을 달성.
> 3. **융합**: **페어와이즈 테스트(Pairwise Testing)**와 같은 조합 최적화 알고리즘을 통해 수억 개의 조합(Combination Explosion) 문제를 해결하고, CI/CD 파이프라인의 자동화된 테스트 단계에서 필수적인 전략임.

---

### Ⅰ. 개요 (Context & Background)

블랙박스 테스트(Black-box Test)는 소프트웨어의 내부 로직이나 코드 구조를 알지 못한 상태, 즉 '검은 상자'로 간주하고 수행하는 테스트 방법론입니다. 이는 **SW 개발 생명주기(SDLC, Software Development Life Cycle)** 전반의 품질 보증(QA, Quality Assurance) 단계에서 사용자 관점의 요구사항이 충족되었는지를 검증하는 핵심적인 기법입니다.

#### 작동 원칙 및 철학
이 방식의 핵심 철학은 **"소프트웨어는 사용자가 기대하는 기능을 수행해야 한다"**는 것입니다. 개발자의 관점(코드 복잡도, 분기覆盖率)이 아닌, **외부 관점(입력 데이터, 사용자 시나리오, 명세서)**에서의 무결성을 확인합니다. 이를 위해 **기능 명세서(Functional Specification)**를 분석하여 입력값에 따른 예상 결과(Expected Result)를 정의하고, 실제 결과(Actual Result)를 비교합니다.

#### 등장 배경
1.  **기존 한계**: 화이트박스(White-box) 테스트는 코드 라인 수준의 검증은 가능하지만, 요구사항 누락이나 사용자 관점의 UX 결함을 발견하기 어려움.
2.  **혁신적 패러다임**: 소프트웨어 공학(Software Engineering)이 발전하며, 내부 구조보다 '요구사항 충족 여부'가 비즈니스 성공에 더 직결됨이 인식됨.
3.  **현재의 비즈니스 요구**: 대규모 시스템에서는 모든 조합을 테스트할 수 없으므로, **위험 기반 테스트(Risk-Based Testing)**의 일환으로 효율적인 데이터 선정 기법(Boundary, Pairwise)이 필수적이 되었음.

```ascii
      [ 테스터의 관점 ]
             │
             ▼
    +-------------------------+
    │   ⚫ Specification       │ (입력/출력 명세)
    │   (요구사항 정의서)       │
    +-------------------------+
             │
             │  입력 자극 (Input)
             ▼
    +-------------------------+
    │  ╔════════════════════╗  │
    │  ║    BLACK  BOX      ║  │  <-- 내부 로직 미확인 (Hidden Logic)
    │  ║                    ║  │      (알고리즘, 코드, 변수)
    │  ╚════════════════════╝  │
    +-------------------------+
             │
             │  결과 반응 (Output)
             ▼
    [   ✅ Pass or ❌ Fail    ]
```
*도해: 블랙박스 테스트의 기본 개념도. 내부 로직은 은폐된 상태에서 입력 명세에 따른 출력 결과만을 검증함.*

**해설**: 위 다이어그램과 같이 블랙박스 테스트는 시스템의 내부 구조를 '블랙홀'처럼 취급합니다. 테스터는 오직 유효한 입력과 무효한 입력을 시스템에 전달하고, 시스템이 명세에 정의된 대로 올바른 출력을 내놓는지, 혹은 적절한 에러 메시지를 출력하는지에만 집중합니다. 이는 시스템이 "무엇을 하는가(What)"에 중점을 둡니다.

> **📢 섹션 요약 비유**: 블랙박스 테스트는 **자동차의 성능을 시험하는 운전자**와 같습니다. 운전자는 자동차 내부의 엔진 실린더 배열이나 ECU 연산 알고리즘(내부 구조)을 알 필요가 없습니다. 단지 핸들을 조작하고(입력), 브레이크를 밟았을 때(입력) 자동차가 안전하게 멈추는지(출력), 혹은 엑셀을 밟으면 가속이 되는지만 확인하면 됩니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

블랙박스 테스트의 품질은 **"어떤 테스트 데이터(Test Data)를 선정하느냐"**에 달려 있습니다. 이를 위해 **동등 분할(Equivalence Partitioning)**, **경계값 분석(Boundary Value Analysis)**, **페어와이즈 테스트(Pairwise Testing)**의 세 가지 핵심 기술을 심층 분석합니다.

#### 1. 구성 요소 상세 분석

| 모듈 (요소) | 역할 | 내부 동작 및 전략 | 프로토콜/도구 | 비유 |
|:---:|:---|:---|:---|:---|
| **EP**<br>(동등 분할) | 입력 도메인의 클래스 분류 | 입력 조건을 유효/무효로 구분하고, 각 클래스를 대표하는 값을 선택하여 **중복 테스트 회피** | JUnit, QTest | 한 입 먹어보고 전체 맛(Seasoning)을 아는 것 |
| **BVA**<br>(경계값 분석) | 오류 밀집 지역 타격 | 결함은 주로 '경계'에서 발생한다는 경험칙에 기반하여 **Min, Min-1, Min+1, Max, Max+1** 등을 테스트 | Boundary Scanner | 절단선 바로 위/아래를 자르며 재단 오류 확인 |
| **Pairwise**<br>(상호 작용 테스트) | 조합 폭발 문제 해결 | 모든 변수 조합이 아닌 **'두 개'의 변수 간 조합**이 커버되도록 **직교 배열(Orthogonal Array)** 또는 생성 알고리즘 활용 | PICT, ACTS | 색상, 옵션 등 모든 조합 중 핵심 디자인만 매칭 |

#### 2. 심층 동작 원리: 결함의 밀도 분포

블랙박스 테스트는 입력 데이터의 **공간(Domain)**을 얼마나 전략적으로 커버하는지가 관건입니다.

1.  **동등 분할 (Equivalence Partitioning)**:
    *   **개념**: 입력 데이터가 만족하는 조건에 따라 여러 개의 집합으로 분할합니다. 각 분할(Partition)은 **동등한 특성**을 가지므로, 하나의 값이 올바르게 처리된다면 나머지 값도 올바를 것이라는 가정하에 **대표값(대푯값)** 하나만 테스트합니다.
    *   **효과**: 테스트 케이스 수를 획기적으로 줄이면서도(무한한 입력 -> 유한한 그룹), 결함 검출 능력을 유지합니다.

2.  **경계값 분석 (Boundary Value Analysis)**:
    *   **개념**: 프로그래밍 오류(Off-by-one error)는 주로 **'>= 10'**인지 **'> 10'**인지와 같이 경계선(Boundary)에서 발생합니다. 따라서 유효 범위의 바로 안쪽과 바로 바깥쪽(Invalid) 값을 포함하여 테스트합니다.
    *   **전략**: 단일 조건 검증 시 `Min`, `Min+`, `Nominal`, `Max-`, `Max` 등 5가지 포인트를 공략합니다.

```ascii
[ Input Domain Mapping: 1 ~ 100 (Valid) ]

Invalid Zone    Valid Zone            Invalid Zone
      │             │                       │
      ▼             ▼                       ▼
   -999 ... 0 | 1 [2...99] 100 | 101 ... 999
             │   ▲       ▲   │
             │   │       │   │
    Target Points: │       │   └─ (Max+1) : Invalid Check
        (BVA Focus) │       └───── (Max)  : Valid Edge
                   └───────────── (Min)  : Valid Edge
```
*도해: 경계값 분석(BVA)에 따른 입력 공간의 포인트 선정 전략. 경계선(Min/Max)과 그 바로 바깥(Min-1, Max+1)이 핵심 타겟팅 지점임.*

**해설**: 위 다이어그램은 1~100을 유효 범위로 갖는 시스템의 경계값 분석을 시각화한 것입니다. 단순히 중앙값(50)만 테스트하는 것으로는 부족합니다. **0(무효)**, **1(최소 유효)**, **100(최대 유효)**, **101(무효)** 경계에서 프로그래머의 실수로 인한 버그가 다수 발생합니다. 이를 통해 테스트 집중도를 최적화합니다.

#### 3. 핵심 알고리즘: 페어와이즈 테스트 (Pairwise Testing)

시스템의 복잡도가 높아질수록(N개의 파라미터), 모든 조합(N-way)을 테스트하는 것은 불가능합니다. 페어와이즈는 **"대부분의 결함은 2개의 인자가 상호작용할 때 발생한다"**는 가설(실험적 검증됨)에 기반합니다.

*   **수식적 접근**:
    전체 조합 수: $V^P$ (V: 값의 수, P: 파라미터 수)
    페어와이즈 조합 수: $\approx V^2 \times \frac{P \times (P-1)}{2}$ (매우 큰 폭으로 감소)

```ascii
[ Pairwise Scenario: Web Browser Test ]

Variables:
1. OS: [Windows, Mac, Linux]
2. Browser: [Chrome, Firefox, Safari]
3. Language: [EN, KR, JP]

(Combination Optimization)
----------------------------------------------------
   Case   |   OS    | Browser | Language | Coverage
----------------------------------------------------
   1      | Windows |  Chrome |   EN     | (OS,Br), (Br,Lan), (OS,Lan)
   2      | Windows | Firefox |   KR     |
   3      | Windows | Safari  |   JP     |
   4      |   Mac   |  Chrome |   KR     |
   5      |   Mac   | Firefox |   JP     |
   6      |   Mac   | Safari  |   EN     |
   7      | Linux   |  Chrome |   JP     |
   8      | Linux   | Firefox |   EN     |
   9      | Linux   | Safari  |   KR     |
----------------------------------------------------
Result: 27 Cases (All) → 9 Cases (Pairwise Opt.)

🔎 Mechanism: Every pair (e.g., Windows+Chrome, Chrome+KR) appears at least once.
```
*도해: 페어와이즈 테스트의 조합 최적화 예시. 3개의 변수(각 3개 값)는 총 27가지 조합을 만들 수 있으나, 모든 '쌍(Pair)'을 커버하는 데에는 9가지 케이스로 충분함.*

**해설**: ASCII 다이어그램과 같이 페어와이즈는 **전수 조합(Exhaustive Testing)**의 기하급수적인 비용을 선형적인 수준으로 끌어내리면서도, 가장 잠재력이 높은 결함(2개 인자 간의 상호작용 버그)을 검출해 냅니다. 이는 **OATS(Orthogonal Array Testing System)**의 응용이며, CI/CD 환경에서 테스트 시간을 단축시키는 핵심 기술입니다.

> **📢 섹션 요약 비유**: 동등 분할은 **국수를 다 익었는지 확인하기 위해 국수 한 가닥만 뜯어 보는 것**과 같습니다. 경계값 분석은 **토너먼트 시드의 경계선에 있는 선수들(1번 시드와 16번 시드)**의 경기 결과를 집중 검토하는 것입니다. 페어와이즈는 **많은 사람들이 앉는 자리 배치에서, 서로 싫어하는 사람들끼리의 '쌍(Pair)'이 최소 한 번씩은 마주 보도록 자리를 배치하는 최적화 알고리즘**과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

블랙박스 테스트 기법들은 서로 독립적이지 않으며, 특정 상황에 따라 각각의 장단점이 명확히 구분됩니다. 여기서는 기법 간의 비교와 다른 영역(보안, 성능)과의 융합 관계를 분석합니다.

#### 1. 심층 기술 비교 분석표

| 비교 항목 | 동등 분할 (EP) | 경계값 분석 (BVA) | 페어와이즈 (Pairwise) |
|:---|:---|:---|:---|
| **핵심 전략** | 영역(Domain) 대표화 | 오류 밀집 지역(Edge) 타격 | 조합(Combination) 최소화 |
| **결함 검출률** | 낮음 ~ 중간 | **높음 (단일 변수 오류)** | **높음 (변수 간 상호작용)** |
| **테스트 케이스 수** | 가장 적음 | 적음 ~ 중간 (EP x 4배) | 효율적으로 최적화됨 |
| **적용 가능 시점** | 초기 smoke test | 단위/통합/인수 테스트 | 설정이 복잡한 시스템 테스트 |
| **주요 약점** | 경계선 옆의 결함을 놓침 | 중간 영역의 논리적 오류 놓침 | 3개 이상의 변수 연관 오류 놓칠 수 있음 |
