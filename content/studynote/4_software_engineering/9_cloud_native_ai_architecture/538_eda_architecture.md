# 538. 이벤트 기반 아키텍처 (EDA): 생산자, 브로커, 소비자

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트 기반 아키텍처 (Event-Driven Architecture, EDA)는 상태의 변화(Event)를 감지하고 전파하여, 시스템 구성 요소들이 **비동기적으로 상호작용**하게 하는 분산 소프트웨어 아키텍처 패턴이다.
> 2. **가치**: 생산자(Producer)와 소비자(Consumer) 사이의 시간적·공간적 결합도를 완전히 제거하여 시스템의 확장성과 유연성을 극대화하며, 실시간 데이터 처리 및 대규모 트래픽 폭주 시에도 안정적인 가용성을 보장한다.
> 3. **융합**: 메시지 브로커(Kafka, RabbitMQ)를 핵심 인프라로 사용하며, 현대의 마이크로서비스(MSA) 환경에서 데이터 정합성 보장을 위한 Saga 패턴 및 CQRS 패턴의 핵심 근간으로 작동한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: EDA는 "사건(Event)이 시스템을 움직이는 원동력"이 되는 구조다. 누군가 상품을 주문하면(이벤트 발생), 그 소식을 들은 결제 팀, 배송 팀, 마케팅 팀이 각자 자기 할 일을 알아서 시작하는 방식이다.

- **💡 비유**: EDA는 **"교내 방송 시스템"**과 같습니다. 교무실(생산자)에서 "지금부터 점심시간입니다"라고 방송(이벤트)을 하면, 이 방송을 들은 학생(소비자)들은 급식실로 가고, 선생님들은 교무실로 가며, 매점 주인은 판매 준비를 합니다. 교무실은 학생들이 무엇을 하는지 일일이 확인할 필요가 없습니다.

- **등장 배경**:
  1. **동기식 호출의 한계**: 서비스가 늘어날수록 API 호출 체인이 길어지고, 하나만 느려져도 전체가 멈추는 'Blocking' 문제를 해결해야 했다.
  2. **실시간 비즈니스 요구**: 사용자의 행동에 즉각적으로 반응하는 마케팅이나 사기 탐지(FDS)와 같은 실시간 처리가 중요해졌다.
  3. **시스템 간 독립성 확보**: 새로운 기능(소비자)을 추가할 때 기존 시스템(생산자)의 코드를 수정하지 않고도 확장이 가능한 구조가 절실해졌다.

- **📢 섹션 요약 비유**: 전화를 걸어 일일이 지시하는 방식(동기)에서, 게시판에 할 일을 적어두면 필요한 사람들이 보고 움직이는 방식(이벤트 기반)으로 업무 시스템이 바뀐 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. EDA의 3대 구성 요소 및 역할

| 구성 요소 | 주요 역할 | 특징 | 비유 |
|:---|:---|:---|:---|
| **이벤트 생산자 (Producer)** | 상태 변화를 감지하고 이벤트 발행 | 소비자가 누구인지 몰라도 됨 (Decoupling) | 소식을 알리는 사람 |
| **이벤트 브로커 (Broker)** | 이벤트를 수집, 보관, 라우팅 | 메시지 큐, 고가용성 저장소 역할 | 우체국 / 게시판 |
| **이벤트 소비자 (Consumer)** | 필요한 이벤트를 구독하여 로직 수행 | 자신의 처리 속도에 맞춰 메시지 소비 | 소식을 듣고 일하는 사람 |

---

### 2. EDA 표준 동작 및 통신 아키텍처

이벤트가 흐르는 전 과정을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      이벤트 기반 아키텍처 (EDA) 논리 흐름도                       │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Event Producer]           [Event Broker]               [Event Consumers]   │
 │ (Order Service)           (Kafka / RabbitMQ)             (Subscribers)      │
 │        │                          │                             │           │
 │  1. 상태 변화 발생  ──▶  ┌──────────────────┐  ──(Push/Pull)──▶  ┌────────┐  │
 │     "주문 #101"          │  Event Channel / │                  │ 결제 앱 │  │
 │     (Event Object)       │  Topic           │                  └────────┘  │
 │        │                 └────────┬─────────┘                  ┌────────┐  │
 │        ▼                          │             ──(Push/Pull)──▶  │ 배송 앱 │  │
 │  [Local Database]                 │                            └────────┘  │
 │  (Commit 완료)                    │                            ┌────────┐  │
 │                                   └─────────────(Push/Pull)──▶  │ 통계 앱 │  │
 │                                                                └────────┘  │
 │                                                                             │
 │  ※ 가치: 생산자는 소비자의 성공 여부를 기다리지 않음 (Fire and Forget)             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 핵심은 **이벤트 브로커**를 통한 **간접 소통**이다. 1) **생산자**는 주문 데이터를 자신의 DB에 저장한 직후, "주문이 생성되었다"는 최소한의 정보(Event)를 브로커에게 던지고 자기 할 일을 끝낸다. 2) **브로커**는 이 메시지를 토픽(Topic)별로 분류하여 안전하게 보관한다. 3) 여러 **소비자**들은 자신이 관심 있는 토픽을 구독하고 있다가, 메시지가 들어오는 즉시 각자의 비즈니스 로직을 실행한다. 이 과정에서 생산자와 소비자는 서로의 존재를 전혀 몰라도 시스템이 완벽하게 돌아간다.

---

### 심층 동작 원리: 이벤트 스트림 처리 (Stream Processing)

단순한 전달을 넘어 실시간 분석으로 확장되는 원리다.
1. **CEP (Complex Event Processing)**: 여러 개의 이벤트를 조합하여 의미 있는 패턴을 찾아냄. (예: 5분 내 3번 결제 실패 시 사기 의심)
2. **Windowing**: 시간 단위(1분, 1시간)로 이벤트를 묶어 합계나 평균을 계산.
3. **가치**: 데이터가 데이터베이스에 저장되기 전에 '흐르는 상태'에서 비즈니스 가치를 즉시 창출한다.

- **📢 섹션 요약 비유**: 개별 물방울(이벤트)을 보는 것을 넘어, 거대한 강물의 흐름(스트림)을 보고 홍수(장애)를 예견하거나 수력 발전(통계)을 하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 동기식(Request-Response) vs 이벤트 기반(EDA)

| 비교 항목 | 동기식 아키텍처 | 이벤트 기반 아키텍처 (EDA) |
|:---|:---|:---|
| **결합도** | 높음 (대상을 알아야 함) | **매우 낮음 (익명성 보장)** |
| **제어 흐름** | 호출자가 흐름을 결정 | **이벤트가 흐름을 유발** |
| **확장성** | 서버 부하 전파 위험 | **수평적 확장 매우 용이** |
| **정합성** | 즉시 일관성 (Strong) | **최종 일관성 (Eventual)** |
| **비유** | 1:1 대화 (질문-답변) | 뉴스 속보 (발표-시청) |

---

### 기술 융합: EDA와 서버리스 (Serverless)

서버리스는 EDA의 가장 이상적인 실행 환경이다.
- **Trigger**: S3 파일 업로드나 DB 변경 이벤트가 발생하는 순간, 서버리스 함수(AWS Lambda)가 자동으로 깨어나 로직을 실행한다.
- **가치**: 인프라 관리 없이 오직 '이벤트'에만 반응하는 초경량, 초효율 아키텍처를 구현한다.

- **📢 섹션 요약 비유**: 손님이 올 때만 불이 켜지는 자동 센서등(서버리스)을 이벤트(사람의 움직임)로 작동시키는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 트래픽 변동이 심한 '티켓 예매 시스템' 설계

- **문제 상황**: 예매 시작 시각에 사용자가 폭주하여 '결제 서버'와 '문자 발송 서버'가 동시에 과부하로 다운됨.
- **의사결정 (EDA 도입 전략)**:
  1. **완충 지대(Buffer) 구축**: 주문과 결제 사이에 **Kafka**를 도입한다.
  2. **비동기 처리**: 주문 서비스는 티켓을 선점한 후 '예약 완료' 이벤트를 발행하고 사용자에게 응답을 보낸다. (결제 결과는 나중에 알림)
  3. **Throttling 효과**: 결제 서버는 자신이 감당할 수 있는 속도(예: 초당 100건)로만 Kafka에서 메시지를 가져와 처리한다.
  4. **결과**: 트래픽이 수만 배 몰려도 시스템은 다운되지 않고, 단지 메시지 처리 대기 시간만 길어지는 형태로 안정성을 유지한다.

### 도입 체크리스트
- **Idempotency**: 동일한 이벤트를 두 번 처리해도 결과가 똑같은가? (네트워크 재전송 대비)
- **Monitoring (Lag)**: 브로커에 쌓여있는 이벤트 수(Lag)가 해소되지 않고 계속 쌓이지는 않는가?
- **Exactly-once**: 메시지가 유실되거나 중복되지 않음을 보장하는 브로커 설정(Ack 정책)이 되어 있는가?

### 안티패턴
- **이벤트를 통한 강한 결합**: 이벤트 데이터(Payload)에 너무 많은 정보를 담아, 생산자를 고칠 때 소비자도 다 고쳐야 하는 상황. (최소한의 데이터만 전송 권장)
- **무분별한 비동기**: 즉각적인 결과 확인이 필요한 로그인이나 인증 기능까지 비동기로 처리하여 사용자 경험을 망치는 행위.

- **📢 섹션 요약 비유**: 모든 대화를 무전기(비동기)로만 하려다가 긴급 상황에서 골든타임을 놓칠 수 있습니다. 전화(동기)와 무전기(비동기)의 적절한 조화가 실력입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Performance)** | 응답 속도 3배 향상 | 무거운 후속 작업을 비동기로 분리하여 메인 로직 가속화 |
| **정량적 (Cost)** | 인프라 비용 절감 | 피크 트래픽에 맞춰 서버를 증설하는 대신 큐를 통한 순차 처리로 대응 |
| **정성적 (Resilience)** | 장애 복원력 극대화 | 특정 컴포넌트 장애가 전체 시스템 다운으로 번지는 것을 원천 차단 |
| **정성적 (Flexibility)** | 비즈니스 확장성 확보 | 기존 로직 수정 없이 새로운 서비스(구독자)를 꼽기만 하면 확장 완료 |

### 미래 전망
- **Event Mesh**: 특정 클라우드에 종속되지 않고 전 세계 모든 서비스 간에 이벤트를 실시간으로 라우팅해주는 글로벌 이벤트 망이 대세가 될 것이다.
- **AI-Managed Events**: AI가 실시간 이벤트 스트림을 분석하여, 장애 징후 시 스스로 트래픽을 우회시키거나 복구하는 자율 아키텍처가 실현될 것이다.

### 참고 표준
- **Reactive Manifesto**: 반응성, 회복력 있는 시스템 설계 지침
- **CloudEvents**: 클라우드 간 이벤트 정보를 교환하기 위한 공통 포맷 표준
- **AMQP / MQTT**: 메시지 전달을 위한 표준 프로토콜

- **📢 섹션 요약 비유**: 물결이 번져나가는 호수처럼, 작은 변화가 시스템 전체의 생동감을 만드는 EDA는 현대 마이크로서비스 생태계의 가장 아름답고 강력한 소통 방식입니다.
