# 573. 타임아웃 (Timeout) 및 재시도 (Retry) 백오프(Backoff) 전략

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 타임아웃 (Timeout)은 무한 대기로 인한 자원 고갈을 막기 위한 **강제 종료 임계 시간**이며, 재시도 (Retry)는 일시적 오류 발생 시 **일정 간격을 두고 다시 요청**하여 성공 확률을 높이는 상호 보완적 회복 기법이다.
> 2. **가치**: 네트워크 순단이나 일시적 부하 상황에서 시스템의 **가용성을 자동 복구**하며, 지수 백오프(Exponential Backoff)와 지터(Jitter) 전략을 통해 재시도 요청이 특정 시점에 몰리는 '천둥치는 무리(Thundering Herd)' 현상을 방지한다.
> 3. **융합**: 서킷 브레이커와 결합하여 과도한 재시도로 인한 대상 서버의 '확인사살'을 예방하며, 서비스 메시 및 탄력성 라이브러리를 통해 선언적으로 구현된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 
    - **Timeout**: 호출자가 응답을 기다리는 최대 시간. 이 시간을 넘기면 실패로 간주하고 자원(쓰레드 등)을 회수한다.
    - **Retry**: 요청이 실패했을 때, "잠깐 기다렸다가 다시 해보자"는 논리다.
    - **Backoff**: 재시도 사이의 대기 시간을 조절하는 알고리즘이다.
- **💡 비유**: **"전화 걸기"**와 같습니다. 상대방이 전화를 안 받으면 30초만 벨을 울리고 끊는 것이 **타임아웃**이고, 1분 뒤에 다시 거는 것이 **재시도**입니다. 이때 매번 1분마다 거는 게 아니라 1분, 2분, 4분으로 간격을 늘려가는 것이 **백오프** 전략입니다.
- **등장 배경**:
    1. **분산 환경의 불안정성**: 네트워크 패킷 손실, 찰나의 서버 재시작 등 '일시적 오류(Transient Error)'는 항상 발생한다.
    2. **자원 점유 문제**: 타임아웃이 없으면 에러가 난 서버를 기다리느라 정상인 서버들의 쓰레드까지 모두 묶여 시스템이 정지된다.
    3. **복구 탄력성**: 사소한 오류로 전체 트랜잭션을 취소하기보다, 자동으로 재시도하여 사용자에게 매끄러운 경험을 제공하고자 했다.

- **📢 섹션 요약 비유**: 무작정 문을 두드리는(무한 대기) 대신, 응답이 없으면 잠시 물러나서(Timeout) 숨을 고르고 나중에 다시 정중하게 노크(Retry Backoff)하는 성숙한 소통 방식입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

재시도 전략의 핵심은 대상 서버에 부담을 주지 않으면서 성공 확률을 높이는 알고리즘 설계에 있다.

| 전략 명칭 | 특징 및 원리 | 수식/로직 | 비유 |
|:---:|:---|:---|:---|
| **Fixed Delay** | 항상 일정한 간격으로 재시도 | $Delay = Constant$ | 1분마다 전화하기 |
| **Exponential Backoff** | 재시도 횟수에 따라 대기 시간을 기하급수적으로 늘림 | $Delay = Base \times 2^{count}$ | 1, 2, 4, 8분 간격 |
| **Jitter (지터)** | 대기 시간에 무작위성(Random) 추가 | $Delay = Backoff + Random$ | 1분 2초, 3분 50초... |
| **Max Attempts** | 무한 루프 방지를 위한 최대 횟수 제한 | $Count \le Max$ | 3번만 해보고 포기 |

#### [지수 백오프와 지터(Jitter)의 트래픽 분산 아키텍처]

다수의 클라이언트가 동시에 재시도할 때 발생하는 트래픽 스파이크를 어떻게 억제하는지 보여준다.

```text
 [ 1. Without Jitter (Thundering Herd) ]
 요청량
   ▲      Retry 1      Retry 2      Retry 3
   │        ┃            ┃            ┃
   │        ┃            ┃            ┃
   └────────┸────────────┸────────────┸────────▶ 시간
     (모든 클라이언트가 정확히 같은 시점에 재요청 ──▶ 서버 2차 마비)

 [ 2. With Jitter (Distributed Traffic) ]
 요청량
   ▲      Retry 1      Retry 2      Retry 3
   │      ┎┒ ┎┒        ┎┒ ┎┒        ┎┒ ┎┒
   │    ┎┚┖┚┖┚┒      ┎┚┖┚┖┚┒      ┎┚┖┚┖┚┒
   └────┸┸┸┸┸┸──────┸┸┸┸┸┸──────┸┸┸┸┸┸─────▶ 시간
     (무작위 시간 추가로 트래픽이 분산됨 ──▶ 서버 회복 시간 확보)
```

**[다이어그램 해설]**
1. **천둥치는 무리 현상**: 네트워크 장애 복구 직후, 수천 대의 클라이언트가 동시에 재시도를 수행하면 서버는 다시 부하를 견디지 못하고 죽게 된다. 이를 막기 위해 **Jitter**를 섞어야 한다.
2. **지수 백오프**: $2^n$ 형태로 대기 시간을 늘려 서버가 물리적으로 복구될 시간을 벌어준다.
3. **지터(Jitter)의 마법**: 대기 시간에 `random()` 값을 더함으로써, 클라이언트들의 재시도 시점이 겹치지 않게 분산(Smoothing)시킨다. 이는 현대 클라우드 아키텍처(AWS, Google 등)의 필수 권장 사항이다.

#### 심층 동작 원리: 멱등성 (Idempotency)의 전제 조건
- **원리**: 재시도는 "같은 작업을 여러 번 해도 결과가 같아야 한다"는 멱등성이 보장될 때만 안전하다.
- **실무**: 결제 API처럼 중복 실행 시 큰일 나는 작업은 재시도 전 반드시 **Unique Key(Idempotency Key)**를 대조하여 중복 처리를 방지해야 한다.

- **📢 섹션 요약 비유**: 똑똑한 우체부는 주인이 부재중일 때 매시간 정각에 오는 게 아니라, "금방 오겠지" 하며 조금 이따가 오고, 또 없으면 "오후에 오겠지" 하며 한참 뒤에 무작위로 방문하는 것과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 타임아웃의 두 종류: Connection vs Read

| 비교 항목 | Connection Timeout | Read (Socket) Timeout |
|:---:|:---|:---|
| **발생 시점** | TCP 핸드쉐이크 단계 (연결 맺기) | 데이터 전송 중 (응답 기다리기) |
| **주요 원인** | 서버 다운, 네트워크 단절 | 서버 내부 연산 지연, DB 락 |
| **대응 전략** | 즉시 다른 노드로 전환 | 재시도 혹은 서킷 브레이킹 |
| **비유** | 가게 문이 잠겨 있음 | 주문했는데 음식이 안 나옴 |

#### 2. 서킷 브레이커와의 시너지
- **Retry**: "한 번만 더 믿어보자" (희망적)
- **Circuit Breaker**: "지금은 건드리지 말자" (냉소적)
- **융합**: 재시도를 수 차례 반복해도 안 되면 서킷 브레이커가 작동하여 **Open** 상태로 전환. 이는 죽어가는 서버에 계속 재요청을 보내 확인사살(Retry Storm)하는 것을 막는 '자비로운 보호'다.

- **📢 섹션 요약 비유**: 계속 고집을 부려보는 끈기(Retry)와, 상황이 안 좋으면 깔끔하게 포기하는 냉철함(Circuit Breaker)이 조화를 이루어야 건강한 시스템입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 일시적인 DB 커넥션 풀 부족으로 인한 에러**
    - **문제**: 갑작스러운 트래픽으로 DB 연결이 0.5초간 실패하여 결제 요청 100건이 취소됨.
    - **의사결정**: 결제 모듈에 **Exponential Backoff (Base 500ms, Max 3회)** 적용.
    - **결과**: 100건 중 95건은 0.5초 뒤의 재시도에서 성공 처리됨. 사용자에게는 에러 화면 없이 단지 0.5초의 지연만 노출. 비즈니스 손실 방어.

- **도입 체크리스트**:
    1. **Fail-fast**: 타임아웃을 너무 길게(예: 30초) 잡아서 쓰레드 풀을 다 말려버리고 있지는 않은가? (최대한 짧게 권장)
    2. **Side Effect**: `POST`, `PATCH` 요청에 대해 무분별하게 재시도를 허용하고 있지는 않은가? (데이터 중복 생성 위험)
    3. **Cancellation**: 사용자가 취소 버튼을 눌렀을 때 진행 중인 재시도 루프가 즉시 중단되는가?

- **안티패턴**: 
    - **무조건적 재시도**: 401(인증 실패), 400(잘못된 요청)은 백번 재시도해도 실패한다. **5xx 계열** 에러에만 선별적으로 적용해야 한다.
    - **Recursive Retry**: 에러 핸들러 내부에서 자기 자신을 다시 호출하는 방식. (Stack Overflow 위험)

- **📢 섹션 요약 비유**: 벽에 머리를 박는다고 벽이 뚫리지 않습니다(4xx 에러). 벽이 무너질 때까지 기다리는 것보다, 벽이 수리되는 시간(Backoff)을 주는 것이 공학입니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량적 기대효과**:
    - **가용성 향상**: 일시적 오류에 의한 서비스 중단 발생률 90% 이상 감소.
    - **리소스 효율**: 타임아웃 최적화를 통한 좀비 프로세스 방지 및 메모리 회수 속도 향상.
- **미래 전망**: **Adaptive Timeout** 기술의 등장으로, 고정된 값이 아닌 실시간 네트워크 품질(RTT)과 서버 부하에 따라 AI가 최적의 타임아웃과 백오프 시간을 조절하는 자율형 회복 인프라가 실현될 것이다.
- **참고 표준**: **AWS SDK Retry Policy**, **Google Cloud API Design Guide (Retry Section)**.

- **📢 섹션 요약 비유**: 기다림의 미학인 타임아웃과, 다시 도전하는 용기인 재시도는 분산 시스템이라는 거친 바다에서 배가 뒤집히지 않게 평형을 잡아주는 '스테빌라이저'입니다.

1. **타임아웃**은 친구에게 대답을 기다릴 때, **"10초 안에 대답 안 하면 딴 데 갈 거야!"**라고 미리 정해놓는 규칙이에요.
