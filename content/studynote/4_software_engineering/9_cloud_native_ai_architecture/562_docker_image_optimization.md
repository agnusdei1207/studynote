# 562. 도커 이미지 최적화: Multi-stage Build, Distroless

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도커 이미지 최적화는 애플리케이션 실행에 꼭 필요한 바이너리와 라이브러리만 남기고 불필요한 OS 도구 및 빌드 아티팩트를 제거하여, **이미지 크기를 줄이고 보안 공격 표면(Attack Surface)을 최소화**하는 작업이다.
> 2. **가치**: **Multi-stage Build**를 통해 빌드 환경과 실행 환경을 분리하여 '기름기를 뺀' 가벼운 이미지를 생성하며, **Distroless** 이미지를 사용하여 쉘(Shell)조차 없는 극단적인 격리 환경을 구축함으로써 보안성을 극대화한다.
> 3. **융합**: CI/CD 파이프라인의 전송 속도 향상, 클라우드 스토리지 비용 절감, 그리고 컨테이너 보안 하드닝(Hardening) 기술과 결합되어 현대적 클라우드 네이티브 운영의 필수 공정으로 자리 잡고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 도커 이미지는 계층(Layer) 구조로 되어 있다. 최적화는 이 계층에서 '낭비'를 줄이는 과정이다. 컴파일할 때만 필요한 도구(GCC, JDK 등)를 최종 실행 이미지에서 빼버리고, 해커가 침투해도 쓸 수 있는 도구(curl, bash 등)를 아예 삭제하는 기법들을 포함한다.

- **💡 비유**: 도커 이미지 최적화는 **"이사 짐 줄이기"**와 같습니다. 새집(운영 서버)으로 갈 때, 가구 조립에 썼던 망치와 전동 드릴(빌드 도구)은 버리고 완성된 가구(바이너리)만 챙겨가는 것입니다. 짐이 가벼워야 이사 트럭(네트워크)이 빨리 달리고, 도둑이 들어와도 훔쳐갈 가구 외에 연장(Shell)이 없어 2차 범죄를 못 저지르게 만드는 것과 같습니다.

- **등장 배경**:
  1. **배포 속도 저하**: 서비스가 MSA로 쪼개지면서 컨테이너 수가 늘어났고, GB 단위의 무거운 이미지는 배포 지연의 주범이 되었다.
  2. **보안 위협 증대**: 이미지 안에 포함된 불필요한 패키지들이 CVE(취약점)의 온상이 되어, 관리해야 할 보안 구멍이 너무 많아졌다.
  3. **비용 최적화**: 클라우드 레지스트리 저장 비용과 데이터 전송 비용을 줄이기 위한 경제적 요구가 커졌다.

- **📢 섹션 요약 비유**: 두꺼운 백과사전(일반 이미지)에서 내가 공부할 페이지(핵심 앱)만 찢어서 가볍게 들고 다니는 효율적이고 안전한 학습법입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 이미지 최적화 주요 기술 비교

| 기술 명칭 | 핵심 동작 원리 | 주요 효과 | 비유 |
|:---|:---|:---|:---|
| **Multi-stage Build** | 한 Dockerfile 내에서 여러 `FROM` 사용 | 빌드 부산물 제거, 최종 이미지 슬림화 | 주방에서 요리 후 접시만 서빙 |
| **Distroless** | OS 도구(Shell, Package Mgr) 제거 | **공격 표면 최소화**, 보안 극대화 | 문도 창문도 없는 철통 금고 |
| **Alpine Linux** | 초경량 기반 이미지 (5MB) 사용 | 이미지 크기 획기적 감소 | 초소형 텐트 설치 |
| **Layer Merge** | `&&` 연산자로 명령어 통합 | 불필요한 중간 레이어 생성 방지 | 영수증 여러 장을 한 장으로 합치기 |

---

### 2. Multi-stage Build 아키텍처 흐름

빌드 과정과 실행 과정을 분리하여 무결한 실행 이미지를 만드는 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Multi-stage Build를 통한 이미지 생성 프로세스                │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Stage 1: Build (Heavy)]                     [Stage 2: Run (Slim)]          │
 │  ┌────────────────────────┐                  ┌────────────────────────┐     │
 │  │ FROM maven:3.8-jdk-11  │                  │ FROM openjdk:11-jre    │     │
 │  │ (빌드용 거대 이미지)      │                  │ (실행용 경량 이미지)      │     │
 │  ├────────────────────────┤                  ├────────────────────────┤     │
 │  │ 1. 소스 코드 복사       │                  │                        │     │
 │  │ 2. 의존성 다운로드      │                  │ 4. 결과물만 복사        │     │
 │  │ 3. 컴파일 (mvn build)   │ ──(app.jar)──▶   │    (COPY --from=build) │     │
 │  │    ▶ 생성물: app.jar    │                  │ 5. 앱 실행 (CMD)        │     │
 │  └────────────────────────┘                  └────────────────────────┘     │
 │                                                                             │
 │  ※ 가치: 800MB 이미지에서 100MB 이미지로 압축 (빌드 도구는 1단계에서 소멸됨)        │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 하나의 `Dockerfile` 안에서 두 번의 `FROM` 구문을 사용한다. 1) **1단계(build)**에서는 모든 컴파일 도구가 포함된 무거운 이미지를 가져와 소스를 빌드한다. 2) **2단계(run)**에서는 실제 실행에 필요한 최소한의 런타임(JRE 등)만 포함된 이미지를 새로 시작한다. 3) 1단계에서 만들어진 결과물(`app.jar`)만 쏙 빼서 2단계 이미지로 복사한다. 최종 결과물에는 소스 코드나 빌드 툴이 전혀 포함되지 않아 **크기가 작고 보안이 강력**해진다.

---

### 심층 동작 원리: Distroless (디스트로리스) 이미지

보안 전문가들이 가장 추천하는 '껍데기 없는' 이미지 기술이다.
1. **정의**: 패키지 매니저(apt, yum), 쉘(bash, sh) 등 표준 리눅스 배포판의 일반적인 도구들을 모두 제거하고, 오직 애플리케이션과 그 의존성 라이브러리만 담은 이미지.
2. **보안 가치**: 해커가 앱의 취약점을 통해 컨테이너 안으로 들어와도, `ls`, `cat`, `curl` 같은 명령어를 아예 쓸 수 없어 추가 공격(횡적 이동)이 불가능하다.
3. **가동 방식**: 구글(Google)에서 제공하는 `gcr.io/distroless/static` 등이 대표적이다.

- **📢 섹션 요약 비유**: 도둑이 금고 안으로 순간 이동에 성공했더라도, 금고 안에 손도 발도 못 움직이게 꽉 찬 콘크리트(Distroless)가 부어져 있어 아무것도 못 하는 상황과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 이미지 기반 (Base Image) 선택 매트릭스

용도와 보안 요구치에 따라 베이스 이미지를 결정한다.

| 베이스 이미지 | 크기 | 보안성 | 호환성 | 추천 상황 |
|:---|:---:|:---:|:---:|:---|
| **Ubuntu / Debian** | 대 (100MB+) | 낮음 | **최상** | 초기 개발, 복잡한 라이브러리 필요 시 |
| **Alpine Linux** | **소 (5MB)** | 높음 | 보통 | 일반적인 MSA 서비스 (musl libc 이슈 주의) |
| **Distroless** | 중소 (20MB+) | **최상** | 낮음 | **보안이 극도로 중요한 금융/결제 서비스** |

---

### 기술 융합: 이미지 최적화와 이미지 스캐닝 (SCA)

이미지를 줄이는 것은 보안 검사 효율을 높인다.
- **Scanning Time**: 1,000개의 패키지가 든 이미지를 스캔하는 것보다, 10개의 핵심 라이브러리만 든 이미지를 스캔하는 것이 훨씬 빠르고 정확하다.
- **가치**: "관리할 것이 적어야 보안이 잘 된다"는 미니멀리즘 보안을 실현한다.

- **📢 섹션 요약 비유**: 수만 명의 관중 속에서 범인을 찾는 것보다, 단 10명이 모인 방 안에서 범인을 찾는 것이 훨씬 쉬운 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 자바 스프링 부트 앱의 '다이어트 및 하드닝'

- **문제 상황**: 배포 한 번에 5분이 걸리고, 이미지 스캔 시마다 수백 개의 보안 취약점 경고가 쏟아져 관리가 불가능함.
- **의사결정 (최적화 전략)**:
  1. **Multi-stage 도입**: 빌드와 실행을 분리하여 JDK 대신 JRE 이미지를 최종 단계에 사용한다. (이미지 크기 50% 감소)
  2. **JRE 튜닝**: `jlink` 도구를 사용하여 우리 앱이 쓰지 않는 자바 표준 라이브러리 모듈까지 도려낸 '맞춤형 JRE'를 생성한다.
  3. **Distroless 전환**: 최종 실행 이미지를 구글의 distroless 이미지로 교체하여 쉘 접근을 원천 차단한다.
  4. **Layer 최적화**: 자주 변하지 않는 라이브러리 레이어를 하단에, 자주 변하는 소스 코드 레이어를 상단에 배치하여 도커 캐시 효율을 극대화한다.

### 도입 체크리스트
- **Compatibility**: Alpine 사용 시 C 라이브러리(glibc vs musl) 호환성 문제가 없는가?
- **Debugability**: Distroless 이미지는 쉘이 없어 디버깅이 힘들다. `kubectl debug`나 임시 사이드카를 통한 대안이 마련되어 있는가?
- **Scan Result**: 최적화 후 이미지 스캔 시 발견되는 CVE 숫자가 유의미하게 줄어들었는가?

### 안티패턴
- **단일 레이어에 모든 것 쓰기**: 모든 명령을 한 줄에 다 넣으려다 가독성이 떨어지는 경우. (논리적 단위로 묶는 것이 정석)
- **개발 도구 포함 배포**: 편의를 위해 `vim`, `net-tools` 등을 운영 이미지에 남겨두는 행위. (해커를 위한 선물 상자)

- **📢 섹션 요약 비유**: 집을 지을 때 썼던 사다리(빌드 도구)를 거실에 그대로 두고 손님을 맞는 무신경함을 버려야 합니다. 집(이미지)은 오직 거주(실행)를 위한 공간이어야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Size)** | 이미지 크기 80% 감소 | 수백 MB 단위에서 수십 MB 단위로 이미지 슬림화 |
| **정량적 (Speed)** | 배포 속도 3배 향상 | 네트워크 전송 및 이미지 Pull 시간 단축을 통한 빠른 스케일링 |
| **정성적 (Security)** | 보안 취약점 90% 제거 | 불필요한 OS 패키지 제거로 CVE 발생 원천 차단 |
| **정성적 (Stability)** | 일관된 실행 환경 확보 | 필요한 최소 자원만 정의되어 환경에 따른 변수 최소화 |

### 미래 전망
- **Unikernels**: OS 커널조차 없이 오직 앱과 런타임만으로 구성된 '유니커널' 기술이 성숙되어 컨테이너 보안의 종착역이 될 것이다.
- **WebAssembly (Wasm) Containers**: 도커보다 더 가볍고 보안이 강력한 Wasm 단위의 배포가 보편화되어, 이미지 최적화 개념이 한 단계 더 도약할 것이다.

### 참고 표준
- **Docker Best Practices**: 도커 공식 이미지 최적화 가이드
- **GoogleContainerTools/distroless**: 오픈소스 표준 디스트로리스 프로젝트
- **OCI (Open Container Initiative)**: 컨테이너 이미지 규격 표준

- **📢 섹션 요약 비유**: "덜어냄으로써 얻는 완벽함." 도커 이미지 최적화는 단순한 용량 줄이기가 아니라, 소프트웨어의 본질만 남기는 공학적 예술입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **이미지 최적화**는 소풍 가방을 쌀 때, **필요 없는 장난감은 다 빼고 맛있는 도시락만 가볍게 챙기는 것**과 같아요.
2. 가방이 가벼워야 산 정상(서버)까지 빨리 올라갈 수 있고, 가방 안에 칼이나 가위 같은 위험한 물건이 없어야 다치지 않는답니다.
3. 이렇게 꼭 필요한 것만 쏙쏙 골라 담으면, 우리 로봇 친구들이 훨씬 더 똑똑하고 안전하게 일할 수 있어요!
