# 537. 안티패턴: 분산 모놀리스 (Distributed Monolith)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 모놀리스(Distributed Monolith)는 시스템을 마이크로서비스(MSA) 형태로 쪼갰음에도 불구하고, 서비스 간의 **강한 결합도(Tight Coupling)**로 인해 독립적인 배포와 운영이 불가능하고 전체 시스템을 동시에 수정·배포해야 하는 **아키텍처적 안티패턴**이다.
> 2. **가치**: 모놀리식의 단점(배포 병목)과 분산 시스템의 단점(네트워크 복잡도, 관리 오버헤드)만을 결합한 최악의 상태를 경고하며, 진정한 MSA로 나아가기 위한 결합도 제거(Decoupling)와 인터페이스 설계의 중요성을 역설한다.
> 3. **융합**: 공유 데이터베이스 사용, 지나치게 잦은 동기 통신(Chatty API), 공통 라이브러리에 대한 과도한 의존이 결합되어 발생하며, 이를 해결하기 위해 비동기 통신과 서비스별 독립 DB 원칙을 재정립해야 한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 분산 모놀리스는 "껍데기만 MSA"인 시스템이다. 서비스들은 네트워크로 분리되어 있지만, 로직이나 데이터가 끈끈하게 엮여 있어 서비스 A를 배포하려면 반드시 B와 C도 함께 배포해야만 정상 작동하는 상태를 말한다.

- **💡 비유**: 분산 모놀리스는 **"서로 밧줄로 꽁꽁 묶인 수십 명의 육상 선수들"**과 같습니다. 각자 트랙(독립 서비스)에 서 있기는 하지만, 한 명이 넘어지거나 멈추면 나머지 사람들도 줄줄이 끌려가거나 멈춰야 합니다. 차라리 혼자 뛰는 것(모놀리식)보다 훨씬 느리고 위험합니다.

- **등장 배경**:
  1. **잘못된 분해 전략**: 비즈니스 도메인(DDD)에 대한 이해 없이 기술적 계층이나 무분별한 기준으로 서비스를 쪼갰을 때 발생한다.
  2. **데이터 소유권 무시**: 여러 서비스가 하나의 거대한 공통 DB 테이블을 직접 참조하면서 논리적 경계가 무너졌다.
  3. **과도한 동기 호출**: 서비스 간에 수십 번의 REST 호출이 쇠사슬처럼 엮여(Chained Call), 지연 시간이 누적되고 장애가 순식간에 전파되는 구조가 되었다.

- **📢 섹션 요약 비유**: 복잡한 기계를 쪼개서 부품화했다고 생각했는데, 사실은 부품들이 강력 본드로 붙어 있어 하나만 교체하려 해도 기계 전체를 부숴야 하는 설계 실패 사례입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 분산 모놀리스의 주요 징후 (Symptoms)

이 중 3개 이상 해당한다면 분산 모놀리스 상태로 볼 수 있다.

| 징후 | 상세 설명 | 아키텍처적 문제 |
|:---|:---|:---|
| **동시 배포 필수** | 서비스 A 배포 시 B도 같이 배포해야 함 | 배포 독립성 상실 |
| **공유 데이터베이스** | 여러 서비스가 하나의 DB 스키마를 공유 | 데이터 결합도 최상 |
| **채티(Chatty) API** | 기능 하나를 위해 서비스 간 호출이 너무 많음 | 네트워크 지연 및 가용성 저하 |
| **공통 코드 종속** | 공통 라이브러리 수정 시 모든 서비스 재빌드 | 코드 수준의 강결합 |

---

### 2. 분산 모놀리스 발생 구조 아키텍처

왜 배포가 묶이는지 기술적인 원인을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      분산 모놀리스 (Distributed Monolith)의 강결합 구조            │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Service A] ──────(Tight Sync Call)──────▶ [Service B] ──────(Call)──▶ [C]  │
 │      │                                        │                             │
 │      │ [Shared Library Dependency]            │                             │
 │      ▼                                        ▼                             │
 │  ┌──────────────────────────────────┐        ┌──────────────────┐           │
 │  │      Common Domain Model Jar     │        │   **Shared DB**  │           │
 │  │ (A와 B가 동일한 클래스 구조 공유)   │        │ (Schema Sharing) │           │
 │  └──────────────────────────────────┘        └──────────────────┘           │
 │                                                       ▲                     │
 │  [재앙의 시나리오]                                      │                     │
 │  1. Shared DB의 컬럼명 하나 변경 ───────────────────────┘                     │
 │  2. 이와 연결된 Service A, B, C의 소스 코드가 모두 깨짐                         │
 │  3. 전체 서비스를 동시에 빌드하고 한날한시에 배포해야 함 (Big-bang Release)      │
 │                                                                             │
 │  ※ 결과: MSA의 장점은 사라지고 분산 시스템의 고통만 남음                         │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 분산 모놀리스의 주범은 **공유 자원**이다. 특히 **Shared DB** 아키텍처는 치명적이다. 서비스 A가 자신의 로직을 바꾸기 위해 DB 스키마를 변경하면, 그 DB를 같이 쓰던 서비스 B와 C는 즉시 런타임 에러가 발생한다. 또한 **Common Jar** 같은 공통 라이브러리에 도메인 로직을 넣어 공유하면, 라이브러리 버전 하나 올릴 때마다 수십 개의 서비스를 한꺼번에 배포해야 하는 '배포 지옥'에 빠진다. 이는 물리적으로만 떨어져 있을 뿐, 논리적으로는 거대한 덩어리(Monolith)인 상태다.

---

### 심층 동작 원리: 연쇄 장애 (Cascading Failure)

분산 모놀리스가 가용성에 미치는 파괴적 영향이다.
1. **상황**: 서비스 A가 B를 부르고, B는 C를 부르는 동기적 체인(Chain).
2. **현상**: 가장 말단인 C가 응답이 0.5초만 늦어져도, 상위의 B와 A의 쓰레드(Thread)가 모두 대기 상태로 차오른다.
3. **결과**: 순식간에 전체 시스템의 모든 서버가 응답 불능 상태에 빠진다.
4. **교훈**: 통신 경로를 끊거나 비동기로 전환하지 않으면 분산 시스템은 모놀리식보다 훨씬 위험하다.

- **📢 섹션 요약 비유**: 전등 수백 개를 직렬로 연결해놓은 것과 같습니다. 전구(서비스) 하나만 나가도 집 전체가 캄캄해지는 암전 사고가 일어납니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Monolithic vs Distributed Monolith vs MSA

| 비교 항목 | 모놀리식 | 분산 모놀리스 | 마이크로서비스 (MSA) |
|:---|:---|:---|:---|
| **네트워크 비용** | 낮음 (내부 호출) | **높음 (잦은 원격 호출)** | 보통 (최적화된 호출) |
| **배포 단위** | 전체 하나 | **전체 묶음 (강제)** | **개별 독립적** |
| **장애 격리** | 불가 | **불가 (연쇄 장애)** | **가능 (Circuit Break)** |
| **운영 난이도** | 낮음 | **최상 (복잡한데 위험함)** | 보통 (자동화 전제) |
| **비유** | 거대한 바위 | 바구니에 담긴 달걀들 | 하늘을 나는 새 떼 |

---

### 기술 융합: 안티패턴 탈출을 위한 '내성' 설계

분산 모놀리스를 진정한 MSA로 리팩토링하는 기법들이다.
- **Database Per Service**: 데이터를 물리적/논리적으로 완전히 쪼개어 API로만 소통하게 한다.
- **Consumer-Driven Contracts**: 서비스 간 통신 규약을 명확히 하고, 버전 관리를 통해 하위 호환성을 유지한다.
- **Asynchronous Events**: 꼭 필요한 경우가 아니면 직접 호출 대신 메시지 큐를 통한 비동기 방식으로 전환한다.

- **📢 섹션 요약 비유**: 서로 묶인 밧줄을 끊고(Decoupling), 각자 무전기(비동기 메시지)를 들려주어 자기 페이스대로 뛸 수 있게 만드는 '독립성 선언'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '주문-재고-결제' 강결합 시스템의 MSA 정상화

- **문제 상황**: 주문 서비스 배포 시마다 재고와 결제 서비스를 항상 같이 배포해야 함. 하나라도 배포 순서가 틀리면 데이터가 꼬임.
- **의사결정 (리팩토링 전략)**:
  1. **스키마 분리**: 주문 DB 내에 섞여 있던 재고 테이블을 물리적으로 분리된 재고 서비스 전용 DB로 이전한다.
  2. **API 추상화**: 주문 서비스가 재고 수량을 직접 SQL로 조회하던 코드를 삭제하고, 재고 서비스의 `GET /stock/{id}` API 호출로 대체한다.
  3. **비동기 이벤트 도입**: 주문 완료 시 결제 서버를 직접 부르지 않고, 'OrderCreated' 이벤트를 Kafka로 발행한다. 결제 서버는 이를 받아 자신의 속도대로 처리한다.
  4. **버전 관리**: API 규격 변경 시 `v1`, `v2`를 동시에 유지하여, 호출하는 쪽에서 대응할 시간을 벌어준다. (Co-existence)

### 도입 체크리스트
- **Deployment Independence**: 서비스 A를 배포할 때, 서비스 B의 관리자에게 "지금 배포해도 되나요?"라고 물어봐야 하는가? (그렇다면 분산 모놀리스임)
- **Data Encapsulation**: 우리 서비스의 DB 테이블을 다른 서비스가 직접 `Select` 하고 있지는 않은가?
- **Shared Lib Scope**: 공통 라이브러리에 비즈니스 로직이 들어있지는 않은가? (공통 라이브러리는 유틸리티로 한정해야 함)

### 안티패턴
- **통합 테스트 맹신**: 분산 모놀리스 환경에서는 단위 테스트보다 통합 테스트에 집착하게 된다. (구조가 불안하기 때문). 이는 개발 속도를 더욱 늦추는 악순환이 된다.
- **과도한 중앙 제어**: 모든 서비스의 흐름을 하나의 거대한 오케스트레이터가 통제하는 방식. (지능형 모놀리스의 탄생)

- **📢 섹션 요약 비유**: 부모님과 따로 살기로(분산) 해놓고, 매끼 식사 메뉴(로직)와 용돈(데이터)을 일일이 허락받아야 한다면 그것은 독립이 아니라 '불편한 동거'일 뿐입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Agility)** | 배포 대기 시간 80% 감소 | 동시 배포 협의 과정 제거를 통한 즉각적 기능 반영 |
| **정량적 (MTBF)** | 장애 발생 빈도 하락 | 서비스 간 결합 제거로 한 곳의 장애가 전체로 번지는 것 방지 |
| **정성적 (Freedom)** | 기술 스택의 자율성 | 서비스 간 의존성이 없으므로 각 팀이 최적의 기술을 자유롭게 선택 |
| **정성적 (Standard)** | 아키텍처 원칙 정립 | 분산 시스템의 핵심인 '독립성'과 '격리'의 가치를 조직 전체가 인지 |

### 미래 전망
- **Self-Healing Architecture**: AI가 서비스 간의 호출 패턴을 감시하여, 강한 결합이 감지되는 지점을 실시간으로 찾아내고 비동기 구조로 변경을 제안할 것이다.
- **Contract-first Development**: 코드를 짜기 전 API 계약(Contract)을 확정하고 이를 어기면 빌드 자체가 안 되는 '계약 중심 개발'이 분산 모놀리스 예방의 표준이 될 것이다.

### 참고 표준
- **Microservices Anti-patterns (Chris Richardson)**: 분산 모놀리스의 체계적 정의
- **The Twelve-Factor App**: 배포 독립성을 위한 12가지 원칙
- **Domain-Driven Design (Eric Evans)**: 경계(Boundary) 설정의 이론적 기초

- **📢 섹션 요약 비유**: 분산 모놀리스는 성장의 진통이자, 설계의 나태함이 부른 대가입니다. 이를 깨고 나가는 과정이 바로 '진정한 클라우드 네이티브 엔지니어'로 거듭나는 길입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **분산 모놀리스**는 친구들과 기차 놀이를 할 때, **옆 친구 옷을 손으로 꽉 붙잡고 있어서 한 명만 넘어져도 다 같이 엉덩방아를 찧는 것**과 같아요.
2. 기차 칸처럼 따로 떨어져 있는 것 같지만, 사실은 옷이 엉켜 있어서 마음대로 움직일 수 없는 상태랍니다.
3. 옷을 놓고 각자 자기 속도대로 신나게 달려야(MSA), 진짜 멋진 기차 놀이가 완성되는 거예요!
