# 572. 서킷 브레이커 (Circuit Breaker) - 상태 기반 장애 확산 차단

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서킷 브레이커 (Circuit Breaker)는 분산 시스템에서 특정 서비스의 호출 에러율이 임계치를 넘을 경우, **상태를 차단(Open)하여 추가 호출을 즉시 거절**함으로써 연쇄 장애(Cascading Failure)를 방지하는 회복력 패턴이다.
> 2. **가치**: 장애 서비스로 향하는 트래픽을 차단하여 해당 서비스가 회복될 시간을 벌어주는 동시에, 호출자(Caller)에게는 빠른 실패(Fail-fast)와 **폴백(Fallback)** 응답을 제공하여 전체 시스템 가용성을 유지한다.
> 3. **융합**: 현대 자바 진영에서는 **Resilience4j**가 사실상 표준 라이브러리로 쓰이며, 서비스 메시(Istio)의 사이드카 프록시 레벨에서도 인프라적으로 자동화되어 구현된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 전기 회로의 누전 차단기에서 유래한 패턴으로, 하위 서비스의 응답 지연이나 에러가 상위 서비스의 쓰레드(Thread)를 모두 점유하여 전체가 마비되는 것을 막기 위해 통로를 물리적으로 끊어버리는 논리적 스위치다.
- **💡 비유**: 서킷 브레이커는 집안의 **"누전 차단기(두꺼비집)"**와 같습니다. 특정 가전제품에서 합선이 발생했을 때, 집 전체에 불이 나지 않도록 차단기가 즉시 전기를 끊어버리는 것과 같습니다. 문제가 해결되면 다시 올릴 수 있습니다.
- **등장 배경**:
    1. **분산 트랜잭션의 한계**: 서비스 A ──▶ B ──▶ C로 이어지는 동기 호출 구조에서 C의 지연은 A의 전멸로 이어진다.
    2. **자원 고갈(Resource Exhaustion)**: 네트워크 타임아웃 대기 시간이 길어지면 호출자의 커넥션 풀과 쓰레드가 순식간에 고갈된다.
    3. **자동화된 복구 요구**: 관리자가 개입하지 않아도 시스템 스스로 장애를 감지하고, 서비스가 정상화되면 자동으로 통로를 다시 여는 지능적 제어가 필요해졌다.

- **📢 섹션 요약 비유**: 사고가 난 터널(장애 서비스)로 차들이 계속 진입하여 갇히는 2차 사고를 막기 위해, 터널 입구에서 진입 금지 신호(Circuit Open)를 보내고 우회 도로(Fallback)를 안내하는 똑똑한 교통 관제 시스템입니다.

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

서킷 브레이커는 **상태 머신 (State Machine)** 구조로 작동하며 세 가지 핵심 상태를 순환한다.

| 상태 (State) | 상세 설명 | 동작 매커니즘 | 비유 |
|:---:|:---|:---|:---|
| **CLOSED** | 정상 상태 | 모든 요청을 통과시키며 성공/실패율 기록 | 파란불 (정상 통행) |
| **OPEN** | **차단 상태** | 모든 요청을 즉시 거부(Fail-fast)하고 폴백 실행 | 빨간불 (진입 금지) |
| **HALF-OPEN** | 확인 상태 | 소량의 요청만 보내 서비스가 회복되었는지 테스트 | 황색불 (주의 통행) |

#### [서킷 브레이커 상태 전이 다이어그램]

장애 발생부터 복구까지의 논리적 흐름과 전이 조건을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Circuit Breaker Finite State Machine (FSM)              │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │      [ CLOSED ] ─────────────────────────────▶ [ OPEN ]                     │
 │          │   ▲        (에러율 > 임계치)           │   │                     │
 │          │   │                                   │   │                     │
 │          │   └───────────────────────────────────┘   │                     │
 │          │           (성공률 > 기준치)               │ (대기 시간 경과)       │
 │          │                                           │                     │
 │          ▼                                           ▼                     │
 │      [ HALF-OPEN ] ◀─────────────────────────────────┘                     │
 │          │                                                                  │
 │          └───── (일부 요청만 허용하여 상태 확인) ──────────────────────────────┘  │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 주요 설정 파라미터 (Resilience4j 예시):                                 │  │
 │  │ 1. FailureRateThreshold: 차단기가 열리는 에러 백분율 (예: 50%)             │  │
 │  │ 2. WaitDurationInOpenState: OPEN에서 HALF-OPEN으로 가기 전 대기 시간       │  │
 │  │ 3. SlidingWindowSize: 에러율을 계산할 최근 요청 개수 (예: 100개)           │  │
 │  │ 4. PermittedNumberOfCallsInHalfOpenState: 상태 확인용 테스트 요청 수       │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **정상(Closed) ──▶ 차단(Open)**: 최근 100개의 요청 중 50% 이상이 에러이거나 느리면, 시스템은 "위험"으로 간주하고 즉시 차단기를 내린다.
2. **차단(Open) ──▶ 확인(Half-Open)**: 무작정 닫아두는 게 아니라, 설정된 대기 시간(예: 30초)이 지나면 다시 기회를 주기 위해 '확인' 상태로 바꾼다.
3. **확인(Half-Open) ──▶ 정상(Closed)**: 테스트로 보낸 몇 개의 요청이 성공하면 다시 통로를 활짝 연다. 실패하면 다시 차단한다.

#### 심층 동작 원리: 폴백 (Fallback) 전략
- **개념**: 차단기가 열렸을 때(Open), 에러 메시지 대신 반환할 '차선책' 로직이다.
- **구현**: 캐시된 데이터 반환, 정적 메시지(예: 점검 중), 혹은 기능의 일부 비활성화 등을 통해 사용자 이탈을 방지한다.

- **📢 섹션 요약 비유**: 전기가 나갔을 때 당황하지 않고 미리 준비해둔 비상용 촛불(Fallback)을 켜서 어둠을 밝히는 슬기로운 대처법과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 주요 라이브러리 비교: Hystrix vs Resilience4j

| 비교 항목 | Netflix Hystrix (구형) | Resilience4j (현대적) |
|:---:|:---|:---|
| **상태** | Maintenance Mode (개발 중단) | **Active Development (표준)** |
| **기반 기술** | 자체 쓰레드 풀 및 복잡한 구조 | **Java 8 함수형 인터페이스, 경량화** |
| **오버헤드** | 높음 (Thread Context Switching) | **매우 낮음** |
| **의존성** | 무거움 (Archaius 등 동반) | 매우 가벼움 (Vavr만 의존) |
| **비유** | 웅장하지만 낡은 성벽 | 작지만 견고하고 빠른 최신 방패 |

#### 2. 서비스 메시(Istio)와의 기술 융합
- **Application Level (R4j)**: 코드 내에서 비즈니스 로직별로 세밀한 차단 조건을 정의 가능.
- **Infrastructure Level (Istio)**: 코드 수정 없이 모든 트래픽에 대해 일괄적으로 L7 서킷 브레이킹 적용.
- **시너지**: 인프라 레벨의 거친 방어와 앱 레벨의 정교한 방어를 중첩하여 완벽한 탄력성을 구축한다.

- **📢 섹션 요약 비유**: 보초병(코드 수준)이 수상한 자를 개별적으로 심문하는 것과, 성문(인프라 수준)을 굳게 잠그는 두 가지 방어선이 조화롭게 작동하는 요새와 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 외부 이메일 발송 서버의 지연 장애**
    - **문제**: 가입 완료 후 이메일 인증 메일 발송이 늦어지자, 회원가입 완료 버튼을 누른 사용자들이 로딩 창에서 멈춰있다가 이탈함.
    - **의사결정**: 이메일 발송 모듈에 서킷 브레이커 적용. 에러율 30% 시 차단기 작동 및 비동기 처리(Message Queue)로 폴백.
    - **결과**: 이메일 서버 장애 시에도 "가입 성공! 메일은 5분 내로 도착합니다" 메시지와 함께 즉시 가입 완료 처리. 사용자 경험 보존.

- **도입 체크리스트**:
    1. **Sliding Window**: 시간 기반(Time-based)으로 에러율을 잴 것인가, 횟수 기반(Count-based)으로 잴 것인가?
    2. **Exception Filtering**: 404 에러(사용자 실수)는 서킷 브레이커 카운트에서 제외하고, 503 에러(서버 장애)만 포함하도록 필터링했는가?
    3. **Threshold Tuning**: 트래픽이 적을 때는 한두 번의 에러로 차단기가 열릴 수 있으므로, 최소 요청 수(MinimumNumberOfCalls)를 설정했는가?

- **안티패턴**: 
    - 모든 `Exception`에 대해 차단기를 작동시키는 행위 (정상적인 비즈니스 예외까지 장애로 오인).
    - 폴백 로직 내부에 또 다른 네트워크 호출을 넣는 행위 (폴백마저 장애가 나면 대책 없음).

- **📢 섹션 요약 비유**: 도둑이 들었다고 해서(장애) 집안의 모든 사람(다른 서비스)을 잠재울 필요는 없습니다. 오직 도둑이 들어온 창문(문제 구간)만 봉쇄하는 정밀한 통제가 필요합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량적 기대효과**:
    - **MTTI (평균 장애 인지 시간)**: 시스템 자동 차단으로 즉시 장애 전파 확인.
    - **자원 활용도**: 쓰레드 덤프(Thread Dump) 분석 결과, 대기 중인 좀비 쓰레드 90% 감소.
- **미래 전망**: **AI-Driven Threshold** 기술의 발전으로, 고정된 임계값이 아닌 시간대별 트래픽 특성에 따라 차단 조건이 동적으로 변하는 자율형 서킷 브레이커가 보편화될 것이다.
- **참고 표준**: **Reactive Manifesto** (회복력의 핵심 기법으로 명시).

- **📢 섹션 요약 비유**: "실패를 대하는 태도가 시스템의 품격을 결정합니다." 서킷 브레이커는 장애를 두려워하지 않고 우아하게 수용하는 법을 가르쳐주는 소프트웨어의 성숙한 철학입니다.
