# 552. 오케스트레이션 사가 (Orchestration Saga)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 오케스트레이션 사가(Orchestration Saga)는 분산 트랜잭션의 실행 흐름을 **중앙의 오케스트레이터(지휘자)**가 관리하며, 각 서비스에 명령을 내리고 결과에 따라 다음 단계나 보상 트랜잭션을 결정하는 방식이다.
> 2. **가치**: 비즈니스 로직이 복잡할수록 전체 흐름을 한눈에 파악하기 용이하며, 서비스 간의 직접적인 순환 의존성(Cyclic Dependency)을 제거하여 대규모 마이크로서비스(MSA) 환경의 트랜잭션 관리를 체계화한다.
> 3. **융합**: 상태 머신(State Machine) 기술과 결합되어 각 트랜잭션의 현재 위치를 정확히 추적하며, Camunda나 AWS Step Functions와 같은 전문 워크플로우 엔진을 통해 고가용성 제어 평면을 구축한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 오케스트레이션 사가는 중앙에 '매니저'를 두는 방식이다. 매니저가 "A님, 이거 하세요", "성공했으니 B님, 저거 하세요"라고 시키고, 만약 실패하면 "C님, 아까 한 거 취소하세요"라고 지시하는 구조다.

- **💡 비유**: 오케스트레이션 사가는 **"오케스트라 지휘자와 연주자들"**과 같습니다. 연주자(서비스)들은 서로 쳐다보지 않고 오직 지휘자(오케스트레이터)의 손짓만 보고 연주합니다. 지휘자가 연주 순서를 바꾸거나 멈추게 하는 등 전체 곡의 흐름을 독점적으로 제어합니다.

- **등장 배경**:
  1. **코레오그래피의 한계**: 서비스가 많아지면 누가 어떤 이벤트를 던지고 받는지 추적하기가 너무 복잡해지는 '스파게티 이벤트' 현상이 발생했다.
  2. **복잡한 비즈니스 정책**: 조건에 따라 흐름이 수시로 변하는 업무를 구현하기 위해 중앙 집중식 제어 판넬이 필요해졌다.
  3. **가시성 확보**: "현재 이 주문이 어느 단계에서 멈춰있는가?"에 대한 답을 즉시 얻기 위해 중앙화된 상태 관리 시스템이 요구되었다.

- **📢 섹션 요약 비유**: 수많은 부서가 얽힌 대규모 프로젝트에서, 각 부서가 알아서 협력하기를 기대하기보다 유능한 PM(오케스트레이터) 한 명을 두어 모든 진행 상황을 조율하게 하는 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 오케스트레이션 사가의 구성 요소

| 요소 | 역할 및 정의 | 비유 |
|:---|:---|:---|
| **Saga Orchestrator** | 트랜잭션의 상태를 저장하고 다음 행동을 결정하는 두뇌 | 사령관 / 지휘자 |
| **Participant Service** | 오케스트레이터의 명령을 받아 로컬 작업을 수행하는 서비스 | 일선 부대 / 연주자 |
| **Command Channel** | 오케스트레이터가 서비스에게 명령을 보내는 통로 | 무전기 지시 |
| **Reply Channel** | 서비스가 작업 결과를 오케스트레이터에게 보고하는 통로 | 결과 보고 |

---

### 2. 오케스트레이션 사가 동작 아키텍처

중앙 지휘자를 중심으로 한 별 모양(Star)의 통신 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      오케스트레이션 사가 (Centralized Control) 구조               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Order Service] ──────(Start Saga)──────▶ [Saga Orchestrator]               │
 │                                                  │                          │
 │           ┌──────────────────────────────────────┼────────────────────────┐ │
 │           │ (1. Command: "Charge Money")         │ (3. Command: "Deduct") │ │
 │           ▼                                      ▼                        │ │
 │    ┌────────────┐                         ┌────────────┐                  │ │
 │    │ 결제 서비스 │ ──────(2. Reply: "OK")──▶ │ 재고 서비스 │ ────(4. Fail)───┘ │
 │    └────────────┘                         └────────────┘                  │
 │                                                  │                          │
 │           ┌──────────────────────────────────────┘                          │
 │           ▼ [5. 보상 명령 하달 (Compensation)]                                │
 │    ┌──────────────────────────────────┐                                     │
 │    │  "결제 서비스님, 아까 결제한 거 취소!"  │                                     │
 │    └──────────────────────────────────┘                                     │
 │                                                                             │
 │  ※ 핵심: 개별 서비스는 다음 단계가 무엇인지 모르며, 오직 지휘자에게만 보고함         │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 중심에는 **Saga Orchestrator**라는 상태 머신이 있다. 1) 주문이 시작되면 오케스트레이터는 결제 서비스에 명령을 보낸다. 2) 결제 완료 보고가 오면, 오케스트레이터는 내부 로직에 따라 다음 단계인 재고 서비스에 명령을 내린다. 3) 만약 재고 서비스에서 '실패' 보고가 오면, 오케스트레이터는 사전에 정의된 시퀀스에 따라 결제 서비스에게 **보상 트랜잭션(취소)**을 실행하라고 명령한다. 이 모든 과정에서 서비스들은 서로의 존재를 알 필요가 없으며, 오직 중앙의 지휘관과만 소통한다.

---

### 심층 동작 원리: 상태 머신 (State Machine) 활용

오케스트레이터 내부에서 트랜잭션을 관리하는 방식이다.
1. **정의**: 현재 트랜잭션이 '결제대기', '재고확인중', '취소중' 등 어떤 상태에 있는지 정의된 도표.
2. **원리**: 각 서비스로부터 오는 응답(이벤트)을 '상태 전이(State Transition)'의 트리거로 사용한다.
3. **가치**: 서버가 재시작되어도 DB에 저장된 상태 값을 보고 멈췄던 지점부터 다시 트랜잭션을 이어 나갈 수 있다. (내구성 보장)

- **📢 섹션 요약 비유**: 게임기에서 '세이브 포인트'를 만드는 것과 같습니다. 보스 몬스터(장애)를 만나 죽더라도, 마지막으로 저장된 장소(현재 상태)에서 다시 도전할 수 있게 해줍니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 오케스트레이션 vs 코레오그래피 (Choreography)

분산 트랜잭션의 두 숙명을 비교한다.

| 비교 항목 | 오케스트레이션 (지휘형) | 코레오그래피 (반응형) |
|:---|:---|:---|
| **중앙 통제자** | **있음** (매니저 존재) | 없음 (자율적 연쇄 반응) |
| **결합도** | 중간 (오케스트레이터에 집중) | **매우 낮음** |
| **추적성** | **우수** (중앙 로그 확인 가능) | 어려움 (여러 서비스 로그 대조) |
| **복잡도 대응** | **복잡한 로직 관리에 유리** | 단순한 로직 관리에 유리 |
| **비유** | 사령관의 일괄 지휘 | 소문 듣고 각자 행동 |

---

### 기술 융합: 오케스트레이션과 BPMN (Business Process Model)

비즈니스 흐름을 시각적으로 설계하고 바로 실행한다.
- **BPMN 2.0**: 비즈니스 프로세스를 그리는 표준 기호.
- **가치**: 기획자가 그린 흐름도(BPMN)를 오케스트레이터 엔진(Camunda 등)에 그대로 넣으면, 코딩 없이도 서비스 간의 사가 트랜잭션 흐름이 완성된다.

- **📢 섹션 요약 비유**: 악보(BPMN)를 그리면 컴퓨터가 알아서 악기(서비스)들을 연주시키는 자동 연주 피아노와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 복잡한 '할부 승인 및 사은품 증정' 트랜잭션 설계

- **문제 상황**: 결제 승인 후 사은품 재고를 확인하고, 배송 지연 안내를 보내며, 포인트 적립까지 해야 하는 복잡한 업무. 실패 시 조건별로 취소 범위가 다름.
- **의사결정 (오케스트레이션 적용 전략)**:
  1. **오케스트레이터 구축**: '주문-Saga-매니저' 모듈을 별도로 생성한다.
  2. **상태값 영속화**: 오케스트레이터는 모든 단계를 처리할 때마다 로컬 DB에 현재 진행 상태를 저장한다. (장애 대응)
  3. **비동기 통신**: 명령과 응답은 메시지 큐를 통해 비동기로 처리하여 성능 저하를 방지한다.
  4. **가시성**: 관리자 도구에서 "현재 500번 주문은 사은품 재고 확인 단계에서 3분째 대기 중"임을 시각적으로 확인 가능하게 한다.

### 도입 체크리스트
- **SPOF 방지**: 중앙 오케스트레이터 서버가 죽으면 전체 업무가 마비되므로, 고가용성(HA) 구성이 되어 있는가?
- **Logic Placement**: 비즈니스 로직을 오케스트레이터에 너무 많이 넣고 있지는 않은가? (오케스트레이터는 '흐름'만 관리해야 함)
- **Time-out 관리**: 특정 서비스가 대답이 없을 때, 오케스트레이터가 얼마나 기다렸다가 보상 로직을 시작할 것인가?

### 안티패턴
- **뚱뚱한 오케스트레이터 (Smart Orchestrator)**: 모든 비즈니스 계산 로직을 오케스트레이터에 다 넣어서 거대한 모놀리식을 만드는 행위.
- **직접 DB 조작**: 오케스트레이터가 다른 서비스의 DB를 직접 건드리는 행위. (반드시 API나 메시지로 지시해야 함)

- **📢 섹션 요약 비유**: 매니저(오케스트레이터)가 지시만 해야지, 직접 부엌에 들어가서 칼질(개별 서비스 로직)까지 하려고 들면 주방은 엉망이 되고 매니저는 과부하로 쓰러집니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (MTTR)** | 장애 복구 시간 50% 단축 | 중앙 상태 추적을 통해 병목 및 오류 지점 즉시 식별 |
| **정량적 (Scalability)** | 시스템 확장성 확보 | 새로운 단계 추가 시 오케스트레이터 설정만 변경하여 유연한 대응 |
| **정성적 (Standard)** | 비즈니스 로직 가시성 | 흩어진 코드를 뒤지지 않아도 전체 흐름을 한눈에 파악 가능 |
| **정성적 (Consistency)** | 데이터 정합성 강화 | 중앙 통제하에 누락 없는 보상 트랜잭션 수행 보장 |

### 미래 전망
- **Serverless Orchestration**: AWS Step Functions처럼 인프라 관리 없이 비즈니스 워크플로우만 정의하면 되는 서버리스 오케스트레이션이 대세가 될 것이다.
- **AI-Managed Sagas**: AI가 과거의 트랜잭션 성공/실패 데이터를 분석하여, 장애 확률이 높은 경로를 우회하거나 보상 로직을 지능적으로 최적화할 것이다.

### 참고 표준
- **Saga Pattern (Chris Richardson)**: 마이크로서비스 패턴의 표준 가이드
- **BPMN 2.0 (ISO/IEC 19510)**: 비즈니스 프로세스 모델링 표준
- **State Machine Architecture**: 전산학의 고전적 상태 제어 모델

- **📢 섹션 요약 비유**: "혼돈 속의 질서." 오케스트레이션 사가는 파편화된 서비스들이 각자 연주하는 불협화음을, 중앙의 지휘를 통해 하나의 아름다운 교향곡으로 완성하는 지휘자의 미학입니다.
