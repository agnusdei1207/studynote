# 550. 사가 (Saga) 패턴 및 최종 일관성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사가 (Saga) 패턴은 마이크로서비스(MSA) 환경에서 여러 서비스에 걸친 분산 트랜잭션을 구현하기 위해, **각 서비스의 로컬 트랜잭션을 순차적으로 실행**하고 실패 시 이전에 성공한 작업들을 취소하는 **보상 트랜잭션 (Compensating Transaction)**을 수행하는 기법이다.
> 2. **가치**: 2PC의 동기적 블로킹 문제를 해결하여 시스템의 높은 가용성과 성능을 유지하며, 데이터의 즉시 일관성 대신 시간의 흐름에 따라 결과가 일치되는 **최종 일관성 (Eventual Consistency)** 모델을 통해 비즈니스 정합성을 보장한다.
> 3. **융합**: 메시지 큐(Kafka, RabbitMQ)를 이용한 비동기 이벤트 통신과 결합되며, 구현 방식에 따라 중앙 통제형(Orchestration)과 분산 반응형(Choreography) 모델로 나뉘어 다양한 비즈니스 워크플로우에 적용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 사가 패턴은 "실패를 되돌리는 연속된 거래"다. 하나의 커다란 트랜잭션을 쪼개어, 각 서비스가 자기 할 일을 하고 다음 서비스에 소식을 알린다. 만약 중간에 문제가 생기면, 왔던 길을 되돌아가며 이미 처리된 일들을 '취소' 처리한다.

- **💡 비유**: 사가 패턴은 **"패키지 여행의 예약 및 취소 환불 규정"**과 같습니다. 1) 비행기 예약, 2) 호텔 예약, 3) 가이드 예약을 순서대로 진행합니다. 만약 3번 가이드 예약이 꽉 차서 실패하면, 이미 결제된 1번과 2번을 가서 취소하고 환불받는 과정이 바로 사가 패턴입니다.

- **등장 배경**:
  1. **분산 DB의 정합성 난제**: MSA에서는 서비스마다 DB가 따로 있어 고전적인 `BEGIN-COMMIT`을 쓸 수 없다.
  2. **2PC의 성능 한계**: 모든 노드를 멈춰 세우는 2PC 방식은 대규모 트래픽 환경에서 성능 저하와 장애 전파의 주범이 되었다.
  3. **가용성 중심의 설계 (CAP 이론)**: 일관성을 조금 양보하더라도(최종 일관성), 시스템이 멈추지 않고(가용성) 돌아가게 하려는 현대적 요구사항이 반영되었다.

- **📢 섹션 요약 비유**: "완벽한 동시 성공"을 포기하는 대신, "실패하면 책임지고 원래대로 돌려놓겠다"는 신사적인 약속을 기반으로 한 협력 모델입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 사가 패턴의 두 가지 구현 모델

| 비교 항목 | 코레오그래피 (Choreography) | 오케스트레이션 (Orchestration) |
|:---|:---|:---|
| **제어 방식** | 의사결정이 **분산**됨 (이벤트 구독 방식) | 중앙의 **컨트롤러(Orchestrator)**가 지휘 |
| **복잡도** | 단순함 (중앙 서버 불필요) | 복잡함 (중앙 서버 및 상태 관리 필요) |
| **결합도** | 매우 낮음 | 중간 (중앙 서버에 의존) |
| **추천 상황** | 서비스 개수가 적고 흐름이 단순할 때 | **복잡한 비즈니스 로직, 대규모 시스템** |
| **비유** | 무도회장 댄서들의 합 맞추기 | 오케스트라 지휘자와 연주자 |

---

### 2. 사가 패턴 동작 아키텍처 (보상 트랜잭션 흐름)

작업이 실패했을 때 역순으로 취소되는 메커니즘을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Saga 패턴: 보상 트랜잭션 (Rollback) 흐름                    │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 정상 흐름 (Happy Path)]                                                   │
 │  주문 서비스 ──(T1)──▶ 결제 서비스 ──(T2)──▶ 재고 서비스 ──(T3)──▶ [완료]         │
 │                                                                             │
 │ [2. 실패 및 보상 흐름 (Failure & Compensation)]                               │
 │                                                                             │
 │  주문 생성 (T1) 성공                                                         │
 │      │                                                                      │
 │      ▼                                                                      │
 │  결제 승인 (T2) 성공                                                         │
 │      │                                                                      │
 │      ▼                                                                      │
 │  재고 부족 (T3) **실패!!** ──┐                                                │
 │                             │ [보상 이벤트 발행]                              │
 │                             ▼                                               │
 │  결제 취소 (C2) ◀───────────┘ (T2에 대응하는 보상 작업)                        │
 │      │                                                                      │
 │      ▼                                                                      │
 │  주문 거절 (C1) ◀───────────── (T1에 대응하는 보상 작업)                        │
 │                                                                             │
 │  ※ 가치: ACID의 Atomicity를 비동기 이벤트와 보상 로직으로 흉내 냄 (Semantic)      │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 사가 아키텍처의 핵심은 **보상 트랜잭션 (Compensating Transaction)**이다. 일반적인 DB 롤백은 시스템이 알아서 해주지만, 사가 패턴에서의 롤백은 개발자가 직접 `cancelPayment()`, `cancelOrder()` 같은 취소 로직을 코드로 짜야 한다. **재고 부족**이라는 실패가 확정되는 순간, 시스템은 이전에 성공했던 **결제**와 **주문** 서비스에게 "방금 한 일 취소해!"라는 메시지를 보낸다. 이를 통해 결국 전체 시스템은 '모두 실패'한 상태와 동일한 데이터 정합성을 맞추게 된다.

---

### 심층 동작 원리: 멱등성 (Idempotency)과 중복 방지

비동기 메시지 기반이므로 반드시 지켜야 할 원칙이다.
1. **상황**: 네트워크 문제로 '보상 이벤트'가 두 번 전송될 수 있음.
2. **해결**: 모든 서비스는 "이미 취소된 주문인가?"를 먼저 확인하고, 중복 요청은 무시한다.
3. **가치**: 보상 트랜잭션이 여러 번 실행되어도 결과가 바뀌지 않게(Idempotent) 설계하여 데이터 오염을 막는다.

- **📢 섹션 요약 비유**: "취소해!"라는 말을 두 번 들어도, 이미 취소했다면 두 번째 말은 웃으며 넘기는 여유로운 업무 태도가 필요하다는 뜻입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 2PC vs Saga 패턴 상세 비교

| 비교 항목 | 2-Phase Commit (2PC) | Saga 패턴 |
|:---|:---|:---|
| **일관성** | 강력한 일관성 (Immediate) | **최종 일관성 (Eventual)** |
| **가용성** | 낮음 (Locking 발생) | **매우 높음 (Non-blocking)** |
| **데이터 격리** | 높음 (작업 중 남이 못 봄) | **낮음 (중간 상태가 남에게 노출됨)** |
| **복구 방식** | DB 자동 롤백 | **애플리케이션 보상 트랜잭션** |
| **비유** | 만장일치 투표 | 일단 수락 후 사후 환불 |

---

### 기술 융합: Saga와 메시지 브로커 (Kafka)

사가 패턴의 신뢰성은 메시지 전달 보장에 달렸다.
- **Transactional Outbox 패턴**: 로컬 DB 저장과 이벤트 발행을 하나의 트랜잭션으로 묶어, "DB는 저장됐는데 메시지는 안 나가는" 불상사를 막는다.
- **가치**: 인프라 장애 시에도 큐(Queue)에 쌓인 메시지를 통해 결국에는 사가 흐름이 완결되도록 보장한다.

- **📢 섹션 요약 비유**: 약속을 잊어버리지 않게 비서(Kafka)에게 메모를 남기고, 비서가 상대방이 확인했는지 끝까지 챙기게 하는 꼼꼼한 비즈니스 파트너십입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 대규모 이커머스의 '쿠폰 사용 결제' 프로세스 설계

- **문제 상황**: 사용자가 쿠폰을 적용해 결제하는데, 쿠폰 서버와 결제 서버가 분리되어 있음. 쿠폰은 소진됐는데 결제가 실패하면 쿠폰을 돌려줘야 함.
- **의사결정 (Saga 모델 선정)**:
  1. **모델 선택**: 흐름이 명확하고 추적이 용이해야 하므로 **Orchestration Saga**를 선택한다.
  2. **중재자(Orchestrator) 배치**: '주문 서비스'가 지휘자가 되어 각 서비스에 명령을 내린다.
  3. **보상 로직 설계**: '결제 실패' 통보 시, 주문 서비스가 쿠폰 서비스에 `restoreCoupon(couponId)` API를 호출하도록 설계한다.
  4. **시각화**: 분산 추적(Jaeger)을 통해 사가 흐름이 어디서 멈췄고 보상 트랜잭션이 잘 돌았는지 모니터링한다.

### 도입 체크리스트
- **Semantic Rollback**: 실제 세상의 행위를 되돌리는 로직(예: 이메일 발송 취소 등은 불가능하므로 사과 메일 발송 등)이 정의되었는가?
- **Isolation Level**: 트랜잭션 중간에 다른 사용자가 중간 상태의 데이터를 봐도 비즈니스적으로 문제가 없는가? (Pivot 데이터 보호)
- **Retry Policy**: 보상 트랜잭션 자체가 실패할 경우를 대비한 무한 재시도나 운영자 개입 절차가 있는가?

### 안티패턴
- **Cyclic Dependencies**: 코레오그래피 방식에서 A ──▶ B ──▶ A 순으로 이벤트가 돌아 무한 루프에 빠지는 경우.
- **Over-orchestration**: 너무 사소한 로직까지 지휘자가 통제하여 중앙 병목을 만드는 상황.

- **📢 섹션 요약 비유**: 모든 사소한 가사 노동까지 아빠(Orchestrator)가 지시하는 게 아니라, "밥 다 됐다"는 소리에 아이들이 알아서 손을 씻는(Choreography) 유연함도 섞여야 좋은 집안(시스템)이 됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Perf)** | 시스템 가용성 2배 향상 | 블로킹 없는 처리로 대량 트래픽 수용 능력 증대 |
| **정량적 (Reliability)** | 데이터 정합성 유지 | 실패를 가정한 보상 설계를 통해 분산 환경의 비즈니스 무결성 확보 |
| **정성적 (Standard)** | 현대적 MSA 표준 확립 | 분산 트랜잭션의 실무적인 해결책으로서 팀 전체의 아키텍처 수준 상향 |
| **정성적 (Stability)** | 장애 격리 능력 강화 | 특정 서비스 장애가 다른 서비스의 자원(Lock)을 잡고 늘어지는 현상 제거 |

### 미래 전망
- **Autonomous Compensation**: 머신러닝이 과거의 보상 이력을 학습하여, 가장 성공률 높은 복구 경로를 스스로 선택하는 '지능형 사가'가 등장할 것이다.
- **Native Saga Frameworks**: 언어나 프레임워크 차원에서 상태 머신과 보상 로직을 자동 관리해주는 고수준의 라이브러리가 더욱 성숙될 것이다.

### 참고 표준
- **Sagas (Hector Garcia-Molina)**: 1987년 제안된 논문 (사가의 기원)
- **Microservices Patterns (Chris Richardson)**: 실무형 사가 패턴의 정석
- **Eventual Consistency Guide**: 분산 시스템 일관성 지침

- **📢 섹션 요약 비유**: "완벽함보다 회복력을 선택하라." 사가 패턴은 불확실한 분산 네트워크라는 바다 위에서, 돛이 찢어져도 다시 기워 항해를 계속할 수 있게 해주는 선원들의 끈기 있는 생존 수칙입니다.

3. 이렇게 서로 소식을 전하며 도와주면, 나중에 다 같이 웃으면서 보물을 찾을 수 있게 된답니다!
