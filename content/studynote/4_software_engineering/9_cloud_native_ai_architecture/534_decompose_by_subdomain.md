# 534. 하위 도메인에 따른 분해 (Decompose by Subdomain)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하위 도메인에 따른 분해는 도메인 주도 설계(DDD)의 핵심 기법을 활용하여, 비즈니스의 복잡한 문제를 논리적으로 구분되는 **하위 도메인(Core, Supporting, Generic)**과 언어의 맥락이 일치하는 **Bounded Context**를 기준으로 마이크로서비스를 나누는 전략이다.
> 2. **가치**: 데이터의 중복과 모델의 혼선을 방지하여 시스템의 **정밀한 논리 구조**를 형성하며, 각 서비스가 독립적인 도메인 모델을 가짐으로써 비즈니스 요건 변화에 가장 유연하고 견고하게 대응할 수 있는 기반을 제공한다.
> 3. **융합**: 전략적 설계(Bounded Context, Context Map)와 전술적 설계(Entity, Value Object, Aggregate)가 결합되어 서비스 내부와 외부의 구조적 완결성을 동시에 달성한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: "데이터와 언어가 어디까지 통하는가?"를 기준으로 서비스를 나누는 방식이다. 똑같은 '상품'이라는 단어라도 주문팀과 재고팀에서 쓰는 의미가 다를 수 있음을 인정하고, 그 의미가 같은 영역(Bounded Context)을 하나의 마이크로서비스로 묶는다.

- **💡 비유**: 하위 도메인별 분해는 **"사전(Dictionary)별 분류"**와 같습니다. '차'라는 단어가 자동차 전시장에서는 탈 것(자동차)을 의미하고, 찻집에서는 마시는 음료(녹차)를 의미하듯, 서로 말이 통하는 사람들끼리 방(서비스)을 나누어 대화의 오해를 없애는 방식입니다.

- **등장 배경**:
  1. **모델의 비대화**: 모든 기능을 하나의 '거대 도메인 모델'로 관리하다 보니, 속성이 수백 개인 클래스가 탄생하고 수정 시마다 사이드 이펙트가 발생하는 문제를 해결해야 했다.
  2. **언어의 모호성**: 부서마다 같은 단어를 다른 의미로 사용하여 발생하는 의사소통 비용과 시스템 로직의 오류를 박멸하고자 했다.
  3. **데이터 소유권 명확화**: 어떤 서비스가 특정 데이터의 '진짜 주인'인지를 도메인 논리에 따라 명확히 규정하여 데이터 불일치를 막고자 했다.

- **📢 섹션 요약 비유**: 복잡한 법전을 '민법', '형법', '상법'으로 나누어 각 법전 안에서는 용어의 의미가 명확하게 통하게 만드는 '논리적 구획 정리' 기술입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 하위 도메인의 3가지 유형

비즈니스의 중요도에 따라 자원을 다르게 배분한다.

| 유형 | 설명 | 전략적 가치 | 비유 |
|:---|:---|:---|:---|
| **Core Domain** | 기업의 핵심 경쟁력이자 차별화 포인트 | **직접 고품질 개발 (최우선)** | 맛집의 비밀 소스 |
| **Supporting Domain** | 핵심은 아니지만 비즈니스에 필수적인 기능 | 외주 또는 주니어 개발자 배정 | 식당의 서빙 시스템 |
| **Generic Domain** | 모든 기업이 공통으로 쓰는 일반적 기능 | **기성 솔루션(SaaS) 도입 권장** | 식당의 결제 포스기 |

---

### 2. DDD 기반 서비스 분해 아키텍처 (Bounded Context)

언어와 모델의 경계가 서비스의 경계가 되는 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      DDD Bounded Context 기반 서비스 분할 모델                    │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [전체 비즈니스 영역 (Domain)]                                                  │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │  [Context A: 주문]             │  [Context B: 배송]                     │  │
 │  │  - 언어: "상품" = 구매 대상      │  - 언어: "상품" = 운송 물품             │  │
 │  │  - 핵심: 가격, 할인, 수량       │  - 핵심: 무게, 부피, 주소               │  │
 │  │  ┌──────────────┐             │  ┌──────────────┐                     │  │
 │  │  │ Order Service│ ◀──(Event)──┼─▶│ Ship Service │                     │  │
 │  │  └──────┬───────┘             │  └──────┬───────┘                     │  │
 │  └─────────┼─────────────────────┴─────────┼─────────────────────────────┘  │
 │            ▼                               ▼                                │
 │     [Order DB]                      [Shipping DB]                           │
 │     (주문 특화 테이블)                 (배송 특화 테이블)                       │
 │                                                                             │
 │  ※ 가치: "상품" 객체를 하나로 합치지 않고, 각자의 문맥에 맞게 두 개로 쪼갬          │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 핵심은 **Bounded Context (제한된 맥락)**다. 주문 서비스와 배송 서비스는 각자의 **하위 도메인**을 가지며, 동일한 이름의 데이터라도 각자의 문맥에 맞는 '속성'만 골라 별도의 모델(Entity)로 정의한다. 이렇게 하면 주문 서비스의 상품 클래스에 배송용 속성(무게 등)이 섞여 들어와 코드가 지저분해지는 것을 막을 수 있다. 서비스 간의 데이터 동기화는 서로의 내부를 알 필요 없이 **이벤트(Event)**나 **Context Map**을 통해 느슨하게 연결된다.

---

### 심층 동작 원리: 컨텍스트 매핑 (Context Mapping)

쪼개진 서비스들이 서로 협력하는 방식을 정의하는 기술이다.
1. **Shared Kernel**: 두 컨테이너가 공통으로 사용하는 모델 일부를 공유. (결합도 주의)
2. **Customer-Supplier**: 한쪽이 데이터를 주고 한쪽이 받는 상하 관계.
3. **Anti-corruption Layer (ACL)**: 다른 서비스의 지저분한 모델이 우리 서비스로 침범하지 못하게 중간에 변환 계층을 둠.
4. **가치**: 서비스들을 쪼갠 후 다시 조립할 때의 '외교적 관계'를 명확히 설계한다.

- **📢 섹션 요약 비유**: 서로 다른 나라(서비스)가 무역을 할 때, 화폐(데이터 형식)를 환전해주는 환전소(ACL)를 두어 자국의 경제(도메인 모델)를 보호하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비즈니스 능력별 분해 vs 하위 도메인별 분해 상세 비교

| 비교 항목 | 비즈니스 능력별 (Capability) | 하위 도메인별 (Subdomain) |
|:---|:---|:---|
| **분석 대상** | **조직도, 부서 R&R** | **언어의 의미, 비즈니스 규칙** |
| **장점** | 서비스 획정이 빠르고 직관적임 | **데이터 정합성 및 로직 응집도 최상** |
| **단점** | 도메인 내부의 복잡성 해결 미흡 | 초기 분석에 많은 시간과 전문가 필요 |
| **비유** | 부서별로 사무실 나누기 | 전문 용어 사전별로 팀 나누기 |

---

### 기술 융합: DDD와 클린 아키텍처

하위 도메인으로 분해된 서비스 내부는 클린 아키텍처로 완성된다.
- **Inner Core**: Bounded Context 내부의 가장 순수한 '엔티티'와 '도메인 서비스'.
- **Outer Layers**: 데이터베이스나 외부 API 같은 '어댑터' 계층.
- **가치**: 서비스 간의 경계(DDD)뿐만 아니라 서비스 내부의 기술적 경계(Clean Arch)까지 확보하여 완벽한 모듈화를 실현한다.

- **📢 섹션 요약 비유**: 마을(시스템)을 구역(DDD)별로 잘 나누는 것을 넘어, 각 구역의 집(서비스) 내부도 튼튼하게 설계(클린 아키텍처)하는 완벽한 도시 건설 계획입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 쇼핑몰의 '상품' 데이터 모델 분해

- **문제 상황**: '상품' 테이블에 판매가, 재고량, 배송비, 상품평, 이미지 경로 등이 모두 들어있어 수정할 때마다 온갖 서비스에서 에러가 발생함.
- **의사결정 (하위 도메인별 분해)**:
  1. **맥락 분리**: '전시 컨텍스트'(이미지, 설명), '판매 컨텍스트'(가격, 옵션), '물류 컨텍스트'(재고, 무게)로 하위 도메인을 나눈다.
  2. **Bounded Context 획정**: 각 컨텍스트를 독립된 마이크로서비스로 분리하고, 각자 'Product'라는 이름의 서로 다른 클래스를 갖게 한다.
  3. **데이터 분산**: 하나의 거대 테이블을 3개의 서비스 전용 DB로 쪼개고, 상품 ID로만 연관 관계를 맺는다.
  4. **결과**: 가격 정책을 바꿀 때 배송 시스템이나 상품평 시스템을 건드릴 필요가 전혀 없는 '완벽한 격리'를 달성한다.

### 도입 체크리스트
- **Ubiquitous Language**: 현업과 개발자가 동일한 단어를 쓰고 있는가? (단어 사전 구축 여부)
- **Bounded Context Size**: 서비스가 너무 크지는 않은가? (변경 이유가 여러 개라면 더 쪼개야 함)
- **Domain Expert Participation**: 아키텍처 설계 과정에 비즈니스 전문가가 참여하여 업무 경계를 검증했는가?

### 안티패턴
- **Shared Kernel 남발**: 서비스가 쪼개졌음에도 공통 라이브러리에 핵심 모델을 다 넣어 공유하는 행위. (결국 다시 하나로 뭉쳐짐)
- **아네믹 도메인 (Anemic Domain)**: 데이터만 있고 로직은 없는 껍데기 엔티티. (진정한 DDD 분해가 아님)

- **📢 섹션 요약 비유**: 샴쌍둥이처럼 몸이 붙어있는 서비스들을 떼어낼 때, 신경(데이터 흐름)과 장기(비즈니스 로직)를 정확히 어디서 끊어야 할지 결정하는 정밀한 수술과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Quality)** | 코드 응집도 3배 향상 | 관련 있는 로직이 한 곳에 모여 가독성 및 가용성 극대화 |
| **정량적 (Complexity)** | 시스템 복잡도 지수 하락 | Bounded Context 간의 명확한 인터페이스를 통해 인지 부하 감소 |
| **정성적 (Standard)** | 도메인 지식의 명문화 | 코드를 읽는 것만으로도 비즈니스 업무 규칙을 완벽히 파악 가능 |
| **정성적 (Flexibility)** | 변화에 대한 강건함 | 하위 도메인 단위의 독립적 진화를 통해 시스템 전체의 장기 생존성 확보 |

### 미래 전망
- **AI-Augmented DDD**: 대량의 비즈니스 로그와 소스 코드를 AI가 분석하여, 숨겨진 Bounded Context를 자동으로 찾아내어 리팩토링 안을 제시해줄 것이다.
- **Living Domain Models**: 비즈니스 정책이 바뀌면 실제 코드가 아닌 '도메인 모델 그래프'를 수정하는 것만으로 시스템이 변경되는 모델 기반 아키텍처가 실현될 것이다.

### 참고 표준
- **Domain-Driven Design (Eric Evans)**: DDD의 창시자이자 원전
- **Implementing Domain-Driven Design (Vaughn Vernon)**: 실무 구현 가이드 (IDDD)
- **Context Mapping Patterns**: 서비스 간 협력 패턴 정의

- **📢 섹션 요약 비유**: "언어가 사고를 결정한다." 하위 도메인별 분해는 소프트웨어의 언어를 정제하여 시스템이 비즈니스와 똑같은 생각을 하고 움직이게 만드는 고차원의 지능형 설계입니다.
