# 574. 벌크헤드 (Bulkhead) - 스레드 풀 격리를 통한 장애 전파 방지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 벌크헤드 (Bulkhead) 패턴은 선박의 침수를 막는 격벽 구조에서 유래한 기술로, 시스템의 **자원(쓰레드 풀, 커넥션 풀 등)을 서비스별로 논리적으로 분할**하여 특정 컴포넌트의 장애가 전체 시스템의 리소스 고갈로 이어지지 않게 하는 격리 전략이다.
> 2. **가치**: 특정 서비스의 응답 지연이나 부하가 다른 정상적인 서비스의 자원까지 잠식하는 '자원 기아(Resource Starvation)' 현상을 방지하며, 장애 상황에서도 핵심 기능을 유지하는 **부분적 가용성(Graceful Degradation)**을 보장한다.
> 3. **융합**: 마이크로서비스(MSA)의 서킷 브레이커와 함께 사용되어 '장애 차단'과 '자원 보호'라는 이중 방어선을 형성하며, 최근에는 컨테이너 쿼터(Quota) 및 서비스 메시(Sidecar) 설계를 통해 인프라 수준에서 구현된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 벌크헤드는 하나의 거대한 자원 풀을 여러 개의 작은 풀로 쪼개는 것이다. 서비스 A, B, C가 하나의 쓰레드 풀을 공유하다가 A에서 지연이 발생하면 B와 C도 일할 쓰레드가 없어 멈추게 되는데, 벌크헤드는 이를 막기 위해 각 서비스에 전용 쓰레드 주차장을 지정해주는 방식이다.
- **💡 비유**: 벌크헤드는 **"대형 선박의 격벽"**과 같습니다. 배 밑바닥이 하나의 통자로 되어 있으면 구멍이 하나만 나도 배 전체가 침수되지만, 여러 개의 칸막이(Bulkhead)로 나누어 놓으면 한 칸이 물에 잠겨도 나머지 칸의 부력으로 배가 가라앉지 않고 버틸 수 있는 것과 같습니다.
- **등장 배경**:
    1. **공유 자원의 딜레마**: 서버의 CPU와 메모리, 쓰레드 개수는 한정되어 있다. 특정 서비스의 무한 루프나 느린 DB 쿼리가 이 한정된 자원을 독점하는 것을 막아야 했다.
    2. **Cascading Failure 방지**: 하위 시스템의 지연이 상위 시스템으로 전파되어 전체 서비스가 다운되는 도미노 현상을 물리적으로 차단할 방법이 필요했다.
    3. **예측 가능한 운영**: 서비스별로 최대 사용 가능한 자원을 미리 할당함으로써, 특정 서비스의 폭주가 다른 서비스의 SLA(Service Level Objective)를 해치지 않게 관리하고자 했다.

- **📢 섹션 요약 비유**: 부페 식당에서 한 사람이 음식을 싹쓸이하지 못하도록 1인당 접시 수(자원 한도)를 정해두어, 모든 손님이 골고루 식사할 수 있게 만드는 공정한 배분 전략입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

벌크헤드는 크게 **소프트웨어 레벨 (Thread Pool)**과 **인프라 레벨 (Deployment)**에서 구현된다.

| 격리 수준 | 상세 설명 | 구현 기술 | 비유 |
|:---:|:---|:---|:---|
| **Thread-level** | 하나의 앱 내에서 서비스별 쓰레드 풀 분리 | Resilience4j, Hystrix | 한 지붕 세 가족 (방 나누기) |
| **Process-level** | 서비스마다 독립적인 프로세스/컨테이너 할당 | Docker, Kubernetes Pod | 각자 자기 집에서 살기 |
| **Cluster-level** | 물리 서버나 클러스터 자체를 물리적 분리 | Multi-region, Rack 분리 | 다른 도시에서 살기 |

#### [벌크헤드 패턴 적용 전후의 자원 상태 비교]

특정 서비스의 지연이 전체 시스템 쓰레드에 미치는 영향을 시각화한다.

```text
 [ 1. Before: Shared Thread Pool ]
 전체 쓰레드 (Total: 10)
 ┌──────────────────────────────────┐
 │ [A][A][A][A][A][A][A][A][A][A]   │  ◀── 서비스 A 장애(지연) 발생
 └──────────────────────────────────┘
   ▶ 결과: 서비스 B, C가 쓸 쓰레드가 없어 전체 시스템 마비 (Deadlock)

 [ 2. After: Bulkhead Applied ]
 격리된 쓰레드 풀 (Total: 10)
 ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
 │ Pool A (4)   │  │ Pool B (3)   │  │ Pool C (3)   │
 │ [A][A][A][A] │  │ [B][ ][ ]    │  │ [C][C][ ]    │
 └──────────────┘  └──────────────┘  └──────────────┘
   ▲ 서비스 A 풀 고갈(Full)  ▲ B는 여유로움     ▲ C는 여유로움
   ▶ 결과: 서비스 A는 지연되지만, B와 C는 자기 몫의 자원으로 정상 서비스 유지
```

**[다이어그램 해설]**
1. **자원 독점 방지**: 서비스 A에서 병목이 발생하더라도, A가 점유할 수 있는 최대 쓰레드 수는 4개로 제한된다.
2. **부분적 장애 수용**: A를 호출하는 사용자들은 응답을 못 받을 수 있지만, B와 C를 이용하는 대다수의 사용자는 장애를 전혀 느끼지 못한다.
3. **Fail-fast**: Pool A가 꽉 차면 더 이상의 요청을 대기시키지 않고 즉시 거절(Rejected)함으로써, 전체 시스템의 응답성이 급격히 무너지는 것을 방지한다.

#### 심층 동작 원리: 세마포어(Semaphore) vs 쓰레드 풀(Thread Pool) 격리
- **Semaphore 방식**: 별도의 쓰레드를 만들지 않고 카운터만 두어 진입 인원 제한. (오버헤드 적음, 네트워크 지연 대응 미흡)
- **Thread Pool 방식**: 완전한 독립 쓰레드 할당. (오버헤드 발생하나 타임아웃과 함께 쓰여 네트워크 지연 차단에 탁월)

- **📢 섹션 요약 비유**: 은행 창구에서 번호표를 뽑을 때, 일반 예금 창구와 대출 상담 창구를 분리하여 대출 상담이 길어져도 일반 예금 업무가 멈추지 않게 만드는 것과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. Bulkhead vs Circuit Breaker (회복력 시너지)

| 비교 항목 | 벌크헤드 (Bulkhead) | 서킷 브레이커 (Circuit Breaker) |
|:---:|:---|:---|
| **중점 목표** | **자원 점유 보호** (Isolation) | **장애 서비스 호출 차단** (Prevention) |
| **동작 시점** | 요청이 들어오는 순간 (상시 격리) | 에러율이 임계치를 넘는 순간 (동적 차단) |
| **방어 대상** | 우리 서버의 **쓰레드/메모리** | **상대 서버의 부하** 및 연쇄 장애 |
| **비유** | 전용 차선 확보 | 고속도로 입구 폐쇄 |

#### 2. 쿠버네티스(K8s) 리소스 제한과의 융합
- **Resource Quota**: CPU/Memory 상한선을 두어 특정 파드(Pod)가 노드 전체 자원을 고갈시키지 않게 함.
- **Sidecar Isolation**: 서비스 메시(Envoy)의 Outlier Detection 기능을 통해 특정 엔드포인트의 자원 사용량을 제어.
- **시너지**: 코드 수준(R4j)의 벌크헤드와 인프라 수준(K8s)의 리소스 제한이 결합되어 '난공불락의 요새'를 구축한다.

- **📢 섹션 요약 비유**: 개별 방에 방화문(Bulkhead)을 다는 것뿐만 아니라, 건물 전체의 전기 용량(Resource Quota)을 나누어 관리함으로써 한 방의 전열기 과부하가 아파트 전체의 정전으로 번지지 않게 하는 이중 보호 시스템입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 외부 서드파티 결제사 응답 지연 장애**
    - **문제**: 특정 결제사(예: A카드)의 망 지연으로 결제 요청 쓰레드가 수천 개 대기 상태에 빠짐. 이로 인해 계좌 이체 등 다른 결제 수단까지 모두 먹통이 됨.
    - **의사결정**: 결제 수단별로 **Bulkhead** 적용. A카드 전용 쓰레드 20개, 계좌 이체 전용 20개로 분리.
    - **결과**: A카드 결제는 지연되지만, 계좌 이체 및 간편 결제 서비스는 100% 정상 작동 유지. 피해 범위를 특정 결제 수단으로 한정함.

- **도입 체크리스트**:
    1. **Pool Size Tuning**: 리틀의 법칙(Little's Law) 등을 활용하여 서비스별 적정 쓰레드 개수를 과학적으로 산출했는가?
    2. **Wait Queue Size**: 큐 길이를 너무 길게 잡으면 여전히 메모리 부족 위험이 있다. 적절히 짧은 큐 길이를 설정했는가?
    3. **Monitoring**: 서비스별 자원 사용량과 거절률(Rejection Rate)을 실시간으로 관찰하고 있는가?

- **안티패턴**: 
    - **너무 작은 풀**: 부하가 조금만 늘어도 멀쩡한 서비스를 거절하여 가용성 저하.
    - **격리 없는 만능 풀**: 성능 향상을 위해 하나의 거대 풀을 고집하는 행위 (장애 전파에 무방비).

- **📢 섹션 요약 비유**: "계란을 한 바구니에 담지 마라." 벌크헤드는 자원이라는 소중한 자산을 여러 바구니에 나누어 담아, 하나가 떨어져도 나머지는 지켜내는 가장 기본적인 위험 분산 전략입니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량적 기대효과**:
    - **장애 격리 지수**: 특정 모듈 장애 시 타 모듈 영향도 0% 달성 가능.
    - **시스템 가동 시간**: 연쇄 장애로 인한 전체 다운타임 90% 이상 예방.
- **미래 전망**: **Serverless(FaaS)** 환경에서는 함수 단위로 자원이 자동 격리되므로 벌크헤드가 인프라의 기본 속성이 되고 있으며, 더 나아가 하드웨어 수준에서 자원을 나누는 **Virtualization 가속 기술**과 결합될 것이다.
- **참고 표준**: **Reactive Manifesto** (탄력적인 시스템의 필수 요건으로 격리(Isolation) 강조).

- **📢 섹션 요약 비유**: 수동으로 문을 닫던 과거의 수밀문에서, 물이 새면 자동으로 닫히는 지능형 격벽 시스템으로 진화하듯 소프트웨어 자원 관리도 '자율형 격리' 시대로 나아가고 있습니다.
