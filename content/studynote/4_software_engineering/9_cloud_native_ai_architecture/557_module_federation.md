# 557. 모듈 페더레이션 (Module Federation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모듈 페더레이션(Module Federation)은 Webpack 5에서 도입된 자바스크립트 아키텍처로, **빌드 시점이 아닌 런타임(Runtime)에** 서로 다른 애플리케이션 간에 코드와 모듈을 동적으로 공유하고 실행할 수 있게 하는 기술이다.
> 2. **가치**: 마이크로 프론트엔드(Micro Frontends)를 구현하는 가장 효율적인 수단이며, 공통 라이브러리(React, Vue 등)의 중복 로드를 방지하는 '싱글톤 의존성' 관리 기능을 통해 웹 애플리케이션의 성능과 개발 유연성을 동시에 확보한다.
> 3. **융합**: 독립적인 배포 파이프라인과 결합되어 '무중단 업데이트'를 가능케 하며, 현대 웹 아키텍처에서는 마이크로서비스(MSA)의 철학을 프론트엔드 영역으로 완벽히 확장하는 기술적 교량 역할을 한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 모듈 페더레이션은 시스템을 "연맹(Federation)" 체제로 만드는 것이다. 각 앱은 독립적으로 존재하지만, 필요할 때 옆 동네(다른 앱)의 컴포넌트나 함수를 자기 것처럼 가져다 쓸 수 있다. 중요한 것은 이 모든 과정이 사용자가 사이트를 보고 있는 **실시간(Runtime)**에 일어난다는 점이다.

- **💡 비유**: 모듈 페더레이션은 **"스마트폰의 앱 위젯"**이나 **"레고 블록의 실시간 교체"**와 같습니다. 내 휴대폰 화면(Host)은 그대로인데, 날씨 앱(Remote)에서 제공하는 위젯(Component)을 실시간으로 가져와 화면에 띄우는 것과 같습니다. 날씨 앱이 업데이트되면 내 화면의 위젯도 배포 없이 즉시 최신 상태가 됩니다.

- **등장 배경**:
  1. **코드 공유의 딜레마**: 공통 라이브러리를 공유하려면 `npm install`을 해야 했는데, 이는 라이브러리가 바뀔 때마다 모든 앱을 다시 빌드하고 배포해야 하는 고통을 유발했다. (Build-time dependency의 한계)
  2. **마이크로 프론트엔드의 부상**: 거대한 웹 앱을 쪼개서 운영하고 싶지만, 쪼개진 앱들 간의 통신과 공통 자원 관리가 너무 복잡했다.
  3. **번들 사이즈 최적화 요구**: 여러 마이크로 앱이 각자 동일한 React 라이브러리를 포함하여 전송함으로써 웹 페이지가 무거워지는 현상을 해결해야 했다.

- **📢 섹션 요약 비유**: 책을 통째로 복사해서 나눠주는(빌드 타임 공유) 대신, 도서관(원격 서버)에 있는 책의 페이지를 필요할 때만 실시간으로 펼쳐보는(런타임 공유) 스마트한 지식 공유 시스템입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 모듈 페더레이션의 핵심 역할 (Roles)

| 역할 명칭 | 설명 | 비유 |
|:---|:---|:---|
| **Host (호스트)** | 다른 앱의 모듈을 수입(Import)하여 사용하는 주체 | 음식을 주문하는 손님 |
| **Remote (리모트)** | 자신의 모듈을 외부에서 쓸 수 있게 노출(Expose)하는 주체 | 음식을 제공하는 식당 |
| **Bidirectional** | 호스트인 동시에 리모트 역할을 모두 수행하는 앱 | 서로 음식을 나눠 먹는 친구들 |

---

### 2. 모듈 페더레이션 동작 및 의존성 관리 아키텍처

브라우저에서 서로 다른 빌드 결과물이 어떻게 조립되는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Module Federation 런타임 조립 및 로딩 구조                   │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [App 1: Header Service (Remote)]           [App 2: Main Dashboard (Host)]    │
 │  ┌────────────────────────┐                ┌────────────────────────┐       │
 │  │ Exposes:               │                │ Remotes:               │       │
 │  │  - './Header' ────────┼────────────────┼▶ './Header' from App 1  │       │
 │  └────────────────────────┘                └───────────┬────────────┘       │
 │                                                        │                    │
 │           ┌────────────────────────────────────────────┘                    │
 │           ▼ [Browser Runtime]                                               │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │  1. App 2(Host) 로드                                                   │  │
 │  │  2. App 1(Remote)의 remoteEntry.js를 비동기로 호출                      │  │
 │  │  3. **Shared Dependency Check**:                                      │  │
 │  │     - "Host에 React 18이 있네? Remote의 React는 다운로드 안 함!" (Skip)   │  │
 │  │  4. 런타임 상에서 Header 컴포넌트 렌더링                                  │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 │  ※ 핵심 기술: remoteEntry.js (노출된 모듈의 지도 역할을 하는 매니페스트 파일)      │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 꽃은 **Shared Dependency (의존성 공유)** 기능이다. **App 1**과 **App 2**가 모두 React를 사용한다면, 웹팩은 브라우저에서 이를 감지하여 단 한 번만 React를 로드하도록 제어한다. 1) **Host** 앱이 실행되면 **Remote** 앱의 **remoteEntry.js** 파일을 먼저 읽어 어떤 모듈을 가져올 수 있는지 파악한다. 2) 사용자가 헤더를 보려 할 때, 필요한 코드 조각만 비동기로 가져와 실시간으로 화면에 끼워 넣는다. 이 모든 과정이 별도의 라이브러리 없이 웹팩 자체 엔진으로 처리된다.

---

### 심층 동작 원리: 컨테이너 레퍼런스 (Container Reference)

모듈 페더레이션이 물리적으로 연결되는 방식이다.
1. **Global Variable**: Remote 앱은 자신의 모듈을 특정 전역 변수(예: `window.app1`) 아래에 등록한다.
2. **Asynchronous Import**: Host 앱은 해당 전역 변수를 통해 모듈을 비동기적으로 가져온다. (`import('app1/Header')`)
3. **가치**: 서로 다른 도메인에 배포된 자바스크립트 파일들이 마치 하나의 프로젝트 내에 있는 파일처럼 자연스럽게 섞이게 된다.

- **📢 섹션 요약 비유**: 각자 자기 명함(remoteEntry)을 게시판에 붙여두고, 필요할 때 명함에 적힌 주소로 전화해서 도움(코드)을 받는 프로들의 네트워크와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 모듈 페더레이션 vs 이전 기술 비교

| 비교 항목 | 전통적인 npm 배포 | 아이프레임 (iframe) | 모듈 페더레이션 |
|:---|:---|:---|:---|
| **업데이트 시점** | 빌드 타임 (전체 재배포 필요) | **런타임 (즉시)** | **런타임 (즉시)** |
| **코드 공유** | 패키지 중복 로드 가능성 높음 | 공유 불가능 (완전 격리) | **지능형 공유 (Singleton)** |
| **UX 통합** | 완벽함 (단일 앱) | 부자연스러움 (스크롤, 통신 제약) | **완벽함 (심리스한 조립)** |
| **비유** | 매번 새 차로 바꾸기 | 차 뒤에 리어카 매달기 | **필요할 때 부품만 순간 이동** |

---

### 기술 융합: 모듈 페더레이션과 서버 사이드 렌더링 (SSR)

최신 기술인 **Streaming SSR**과 결합되어 성능을 극대화한다.
- **원리**: 서버에서 여러 마이크로 앱의 HTML을 미리 렌더링하여 보내고, 브라우저에서 모듈 페더레이션으로 인터랙션을 활성화(Hydration)한다.
- **가치**: 검색 엔진 최적화(SEO)와 런타임 유연성을 동시에 잡는 고난도 아키텍처를 완성한다.

- **📢 섹션 요약 비유**: 손님이 오기 전 밑반찬(SSR)을 미리 깔아두고, 손님이 앉자마자 따뜻한 메인 요리(런타임 모듈)를 서빙하는 고급 한정식 서비스와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 커머스의 '공통 내비게이션' 관리 전략

- **문제 상황**: '장바구니', '마이페이지', '검색' 등 수십 개의 팀이 각자 앱을 운영 중인데, 상단 내비게이션 바의 '로고' 하나를 바꾸려면 수십 개의 팀이 동시에 배포 일정을 잡아야 함.
- **의사결정 (모듈 페더레이션 적용)**:
  1. **Remote App 생성**: 내비게이션 바만 담당하는 'Layout-App'을 별도로 만든다.
  2. **Module Expose**: Layout-App에서 `Header` 컴포넌트를 모듈 페더레이션으로 노출한다.
  3. **Host App 설정**: 모든 서비스 팀 앱에서 Layout-App을 Remote로 등록하고 헤더를 실시간 수입하여 렌더링한다.
  4. **결과**: 로고를 바꿀 때 오직 Layout-App 하나만 배포하면, 수십 개의 서비스 화면이 1초 만에 최신 로고로 자동 업데이트된다.

### 도입 체크리스트
- **Version Compatibility**: 공유되는 라이브러리(React 등)의 메이저 버전이 서로 호환되는가? (Breaking Change 주의)
- **Error Handling**: Remote 앱의 서버가 다운되었을 때 Host 앱이 멈추지 않도록 '에러 바운더리' 처리가 되어 있는가?
- **Performance Path**: 런타임에 여러 JS 파일을 로드함에 따른 초기 로딩 속도 저하를 방지하기 위해 Preload 전략을 썼는가?

### 안티패턴
- **Circular Remotes**: 앱 A는 B를 수입하고, 다시 B는 A를 수입하는 순환 참조 구조. (무한 로딩의 주범)
- **Fat Shared Libs**: 사용하지도 않는 수많은 라이브러리를 모두 'Shared'로 설정하여 설정 파일만 수백 줄이 되는 경우.

- **📢 섹션 요약 비유**: 친구들끼리 물건을 빌려 쓰기로(공유) 했으면, 최소한 빌려줄 물건이 고장 나지 않았는지(버전 호환성)는 항상 확인해야 서로 싸우지 않는다는 점을 잊지 마세요.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Velocity)** | 배포 생산성 5배 향상 | 의존성 문제로 인한 팀 간 일정 조율 시간 완전 제거 |
| **정량적 (Size)** | 번들 크기 30% 감소 | 공통 라이브러리 싱글톤 관리를 통한 중복 코드 전송 차단 |
| **정성적 (Standard)** | 전사 일관된 UI/UX 유지 | 중앙에서 관리되는 컴포넌트를 실시간 강제 적용 가능 |
| **정성적 (Agility)** | 비즈니스 실험 가속화 | 신기능 조각(Micro-module)을 특정 화면에 즉시 꽂아 테스트 가능 |

### 미래 전망
- **Framework Agnostic Federation**: React에서 만든 모듈을 Vue에서 아무런 설정 없이 그대로 사용하는 '프레임워크 독립적 페더레이션' 기술이 표준화될 것이다.
- **Edge Module Federation**: CDN 엣지 노드에서 사용자에게 가장 적합한 모듈 조각을 실시간으로 조립하여 내려주는 '초저지연 조립 아키텍처'가 실현될 것이다.

### 참고 표준
- **Webpack 5 Module Federation**: 기술의 오리지널 명세
- **Module Federation Universe**: 에코시스템 공유 저장소
- **W3C Import Maps**: 브라우저 표준 코드 공유 기술 (유사 개념)

- **📢 섹션 요약 비유**: 과거의 웹이 '완성된 조각상'을 배달하는 것이었다면, 이제는 '스스로 조립되는 나노 로봇'을 배달하는 시대로 변했습니다. 모듈 페더레이션은 그 로봇들의 조립 매뉴얼입니다.
