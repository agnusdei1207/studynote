# 535. 서비스 간 동기 통신 (REST API, gRPC)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스(MSA) 간 동기 통신은 요청(Request)을 보낸 서비스가 응답(Response)이 올 때까지 대기하며 실행 흐름을 맞추는 방식으로, **REST API (텍스트/JSON)**와 **gRPC (이진/HTTP2)**가 대표적인 기술이다.
> 2. **가치**: 호출 즉시 결과를 확인해야 하는 비즈니스 로직(예: 로그인, 실시간 결제 승인)에 적합하며, 특히 gRPC는 Protocol Buffers를 통한 고성능 직렬화와 타입 안전성을 제공하여 내부 서비스 간 통신 효율을 극대화한다.
> 3. **융합**: API 게이트웨이의 라우팅 기술 및 서비스 메시(Service Mesh)의 트래픽 제어와 결합되며, 장애 전파를 막기 위해 서킷 브레이커(Circuit Breaker)와 같은 회복력 패턴과 반드시 병행 설계되어야 한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 동기 통신은 "질문과 답변이 한 세트"로 일어나는 통신이다. 클라이언트 서비스가 서버 서비스에게 데이터를 요청하면, 서버가 답을 줄 때까지 클라이언트는 다른 일을 하지 않고 기다린다(Blocking).

- **💡 비유**: 동기 통신은 **"전화 통화"**와 같습니다. 전화를 걸면 상대방이 받을 때까지 수신음을 듣고 기다려야 하며, 상대방이 말을 끝내야 나도 다음 말을 할 수 있는 실시간 소통 방식입니다.

- **등장 배경**:
  1. **MSA 통신 표준화**: 서비스들이 서로 쪼개지면서 이들 간의 대화법이 중요해졌다. 가장 널리 퍼진 HTTP 인프라를 활용한 **REST**가 표준으로 자리 잡았다.
  2. **성능 한계 극복**: JSON 기반의 REST는 사람이 읽기는 좋으나 기계 간의 대량 통신에서는 데이터 크기가 크고 파싱이 느린 단점이 있었다. 이를 해결하기 위해 구글이 이진(Binary) 통신 방식인 **gRPC**를 제안했다.
  3. **실시간성 요구**: 결과값을 즉시 받아서 다음 로직을 처리해야 하는 동기적 업무(예: 재고 확인 후 결제 진행)가 여전히 시스템의 핵심을 차지한다.

- **📢 섹션 요약 비유**: 서로 손을 꼭 잡고 발맞춰 걷는 연인들처럼, 서비스들이 서로의 속도에 맞춰 긴밀하게 소통하는 방식입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. REST API vs gRPC 기술 아키텍처 비교

| 비교 항목 | REST API | gRPC (Google Remote Procedure Call) |
|:---|:---|:---|
| **기반 프로토콜** | HTTP 1.1 / 2 | **HTTP/2 (Streaming 지원)** |
| **페이로드 포맷** | **JSON / XML (Text)** | **Protocol Buffers (Binary)** |
| **통신 방식** | 리소스 중심 (GET, POST...) | 서비스/함수 중심 (RPC) |
| **엄격함** | 느슨함 (Loosely typed) | **엄격함 (Strongly typed, IDL)** |
| **비유** | 편지지에 글써서 보내기 | 규격화된 택배 박스에 물건만 담기 |

---

### 2. gRPC 동작 메커니즘 (IDL 기반)

gRPC가 어떻게 언어에 상관없이 고성능 통신을 구현하는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      gRPC / Protocol Buffers 작동 프로세스                        │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 서비스 정의 (.proto)]                                                     │
 │  service Order { rpc create(OrderReq) returns (OrderRes); }                  │
 │                                                                             │
 │           ┌─────────────────────────┴─────────────────────────┐             │
 │           ▼ [Code Generation]                                 ▼             │
 │  [Service A (Java Client)]                        [Service B (Go Server)]   │
 │  ┌──────────────────┐                             ┌──────────────────┐      │
 │  │   Stub (Proxy)   │                             │   Skeleton       │      │
 │  └────────┬─────────┘                             └────────▲─────────┘      │
 │           │                                                │                │
 │           │ ────(2. ProtoBuf 직렬화: Binary Data)────────▶ │                │
 │           │                                                │                │
 │           │ ◀───(3. HTTP/2 다중화 전송: Response) ─────────  │                │
 │           ▼                                                ▼                │
 │                                                                             │
 │  ※ 가치: JSON 대비 데이터 크기 1/3 하락, 직렬화 속도 10배 이상 향상                 │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** gRPC는 **Protocol Buffers**라는 언어 중립적인 인터페이스 정의 언어(IDL)를 사용한다. 1) `.proto` 파일에 서비스 규격을 적으면 각 언어별로 코드가 자동 생성된다. 2) 클라이언트는 마치 자기 로컬 함수를 부르듯 **Stub**을 호출한다. 3) 데이터는 텍스트가 아닌 **이진(Binary)** 형태로 압축되어 **HTTP/2**를 타고 전송된다. HTTP/2는 하나의 연결에 여러 요청을 동시에 싣는 '멀티플렉싱'을 지원하여 네트워크 효율을 극대화한다.

---

### 심층 동작 원리: REST의 무상태성과 캐싱

REST가 인터넷의 표준이 된 비결이다.
1. **Stateless**: 각 요청은 필요한 모든 정보를 담고 있어 서버가 상태를 유지할 필요가 없다. (확장성)
2. **Cacheable**: GET 요청의 결과는 브라우저나 CDN 단계에서 캐싱될 수 있다. (네트워크 부하 절감)
3. **Uniform Interface**: URL만 보면 무엇을 하는지 알 수 있는 표준화된 인터페이스를 제공한다.

- **📢 섹션 요약 비유**: 누구에게나 통하는 세계 공용어(REST)를 쓰느냐, 아니면 특정 팀원들끼리만 아는 초고속 수화(gRPC)를 쓰느냐의 차이와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 서비스 간 통신 기술 선택 매트릭스

| 선택 기준 | REST API 권장 | gRPC 권장 |
|:---|:---|:---|
| **대상 독자** | 외부 파트너, 일반 웹 브라우저 | **내부 마이크로서비스 간 (Internal)** |
| **성능 요구** | 보통 | **매우 높음 (저지연, 고처리량)** |
| **개발 편의성** | 높음 (툴과 문서화 풍부) | 보통 (Proto 관리 오버헤드) |
| **언어 환경** | 상관 없음 | 다양한 언어 간 엄격한 규약 필요 시 |

---

### 기술 융합: 동기 통신의 연쇄 장애와 서킷 브레이커

동기 통신의 가장 큰 숙제는 '기다림'이 '전멸'로 이어지는 것이다.
- **Cascading Failure**: 서비스 A ──▶ B ──▶ C 구조에서 C가 느려지면 A까지 쓰레드가 고갈되어 죽는다.
- **해결**: **Resilience4j** 같은 도구를 융합하여, 하위 서비스가 일정 수준 이상 에러를 내면 즉시 호출을 차단(Circuit Breaker)하고 기본값(Fallback)을 돌려준다.

- **📢 섹션 요약 비유**: 친구(하위 서비스)가 대답이 없다고 해서 무작정 집 앞에서 기다리는(Blocking) 게 아니라, "지금은 바쁜가 보다" 하고 나중에 다시 오기로(Circuit Break) 약속하는 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 내부 서비스 간 gRPC 도입 및 외부 REST 노출

- **문제 상황**: 모바일 앱에서 주문을 넣으면 내부적으로 수십 개의 마이크로서비스가 통신하는데, JSON 처리 비용 때문에 서버 CPU 부하가 높고 반응이 느림.
- **의사결정 (혼합 통신 아키텍처)**:
  1. **Edge 영역**: 외부 브라우저나 앱과 통신하는 관문은 호환성이 좋은 **REST API**를 유지한다. (OpenAPI/Swagger 활용)
  2. **Core 영역**: 마이크로서비스들끼리 소통하는 내부 망은 **gRPC**로 전면 전환한다.
  3. **Gateway Translation**: API Gateway에서 외부의 REST 요청을 내부의 gRPC 호출로 변환(Transcoding)해주는 프록시 설정을 추가한다.
  4. **결과**: 외부 연동의 편의성은 지키면서, 내부 시스템의 처리 용량(TPS)은 2배 이상 확보한다.

### 도입 체크리스트
- **Load Balancing**: gRPC는 HTTP/2를 사용하므로 L4 장비가 아닌 L7 로드밸런서(Envoy 등)가 필요한데, 인프라가 준비되었는가?
- **Proto Governance**: 여러 팀이 공유하는 `.proto` 파일의 버전 관리와 배포 프로세스가 정립되었는가?
- **Observability**: 이진 데이터라 눈으로 확인이 어려우므로, `grpcurl`이나 전용 모니터링 도구가 갖춰졌는가?

### 안티패턴
- **Deep Sync Chain**: A가 B를 부르고, B가 C를 부르고, C가 D를 부르는 식으로 3단계 이상 동기로 엮는 행위. (하나만 삐끗해도 전체 장애)
- **Large JSON over REST**: 수십 MB의 데이터를 굳이 JSON 텍스트로 주고받는 행위. (네트워크 대역폭 낭비)

- **📢 섹션 요약 비유**: 가족끼리는 암호(gRPC)로 빠르게 소통하되, 외부 손님에게는 친절한 안내문(REST)을 써 붙이는 것이 가장 똑똑한 소통 전략입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Perf)** | 네트워크 트래픽 60% 절감 | gRPC의 이진 압축을 통해 전송 데이터량 대폭 감소 |
| **정량적 (Latency)** | 서비스 응답 속도 향상 | JSON 파싱 오버헤드 제거 및 HTTP/2 멀티플렉싱 효과 |
| **정성적 (Standard)** | 인터페이스 정의의 명확화 | IDL(Proto)을 통해 서비스 간의 '계약'을 강제하여 개발 오류 방지 |
| **정성적 (Agility)** | 폴리글랏 개발 가속화 | 자동 생성된 Stub 코드를 통해 언어가 달라도 즉시 통신 연동 가능 |

### 미래 전망
- **Unified API Mesh**: REST와 gRPC를 넘어 GraphQL까지 하나의 논리적 망으로 묶어주는 지능형 커넥터 기술이 발전할 것이다.
- **Web-native gRPC**: 브라우저에서 gRPC를 직접 호출하는 표준(gRPC-Web)이 성숙되어, 프론트엔드와 백엔드 간의 통신 속도 경쟁이 가속화될 것이다.

### 참고 표준
- **RFC 7231**: Hypertext Transfer Protocol (HTTP/1.1)
- **HTTP/2 (RFC 7540)**: gRPC의 전송 계층 표준
- **Protocol Buffers Language Guide**: 데이터 직렬화 표준

- **📢 섹션 요약 비유**: 손을 잡고 걷는 것(동기)은 정겹고 확실하지만, 가끔은 무전기(비동기)를 써야 할 때가 있습니다. 이 둘을 자유자재로 다루는 것이 진정한 아키텍트의 실력입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **REST**는 친구에게 **"한글로 쓴 긴 편지"**를 보내는 것과 같아서, 읽기는 쉽지만 종이를 많이 써요.
2. **gRPC**는 친구와 우리만 아는 **"비밀 숫자 암호"**로 대화하는 거라, 종이도 적게 들고 대화가 아주 빨라요.
3. 둘 다 친구가 대답할 때까지 기다려야 하는 방식이라, 친구가 바쁘면 나도 같이 기다려야 한답니다!
