# 563. 쿠버네티스 (Kubernetes) 오브젝트 아키텍처 (Pod, Service, Deployment, Ingress)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쿠버네티스 (Kubernetes) 오브젝트 아키텍처는 선언적 API(Declarative API)를 기반으로 컨테이너화된 애플리케이션의 실행 단위(Pod), 배포 관리(Deployment), 네트워크 노출(Service/Ingress)을 계층적으로 조직화한 추상화 모델이다.
> 2. **가치**: 인프라의 상태를 "원하는 상태(Desired State)"로 정의하고 컨트롤러 루프를 통해 이를 "현재 상태(Actual State)"와 일치시킴으로써 가용성 보장, 자동 스케일링, 무중단 배포를 실현한다.
> 3. **융합**: 클라우드 네이티브(Cloud Native) 생태계의 운영체제 역할을 수행하며, 마이크로서비스(MSA) 간의 복잡한 통신 및 생명주기 관리를 표준화된 매니페스트(YAML)를 통해 자동화한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 쿠버네티스(Kubernetes, K8s) 오브젝트 아키텍처는 클러스터의 상태를 관리하기 위한 영속적인 엔티티(Entities)들의 집합이다. 사용자가 "원하는 상태"를 YAML 파일에 기술하여 API 서버에 제출하면, 쿠버네티스는 해당 오브젝트들을 생성하고 유지하기 위해 전력을 다한다.
- **💡 비유**: 쿠버네티스 오브젝트는 **"스마트 빌딩의 중앙 제어 시스템"**과 같습니다. 관리자가 "온도를 24도로 유지해(Desired State)"라고 설정하면, 시스템이 센서를 통해 현재 온도를 체크(Actual State)하고 에어컨이나 히터를 가동(Controller Loop)하여 목표를 맞추는 것과 같은 원리입니다.
- **등장 배경**:
    1. **컨테이너 관리의 한계**: 수백, 수천 개의 도커 컨테이너를 수동으로 관리하고 장애 복구를 수행하는 것은 불가능에 가까워졌다.
    2. **추상화 요구**: 하드웨어 서버와 네트워크 장비의 세부 사항을 감추고, 개발자가 애플리케이션 정의에만 집중할 수 있는 논리적 모델이 필요했다.
    3. **불변 인프라(Immutable Infrastructure)**: 서버를 고쳐 쓰는 대신 새로 배포하는 패러다임을 지원하기 위해 고유한 생명주기를 가진 오브젝트 모델이 정립되었다.

- **📢 섹션 요약 비유**: 마치 거대한 오케스트라의 악보(Object)를 보고 지휘자(Master Node)가 연주자(Worker Node)들을 조율하여 완벽한 화음을 만들어내는 체계적인 지휘 시스템과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

쿠버네티스의 핵심 오브젝트들은 계층 구조를 이루며 상호 보완적으로 작동한다.

| 오브젝트 명칭 | 역할 및 정의 | 핵심 특징 | 비유 |
|:---:|:---|:---|:---|
| **Pod (파드)** | 최소 실행 단위 | 하나 이상의 컨테이너가 IP와 볼륨을 공유 | 호텔의 '방' (침대, TV 공유) |
| **Deployment (디플로이먼트)** | 상태/버전 관리 | 레플리카셋(ReplicaSet)을 관리하며 롤링 업데이트 수행 | '관리 사무소' (인원수 조절) |
| **Service (서비스)** | 네트워크 노출 | 파드 집합에 대한 고정 IP와 로드밸런싱 제공 | '대표 전화번호' |
| **Ingress (인그레스)** | L7 외부 접근 | 도메인 기반 라우팅 및 SSL 종단 처리 | '안내 데스크' (주소별 길 안내) |
| **ConfigMap/Secret** | 설정 정보 관리 | 앱 코드와 설정을 분리하여 환경 이식성 확보 | '보관함' (열쇠 또는 메모) |

#### [쿠버네티스 핵심 오브젝트 레이아웃]

쿠버네티스 시스템 내에서 요청이 유입되어 실제 컨테이너에 도달하기까지의 오브젝트 간 연결 구조를 시각화한다.

```text
 [ 외부 사용자 (Internet) ]
          │
          ▼ [HTTP/HTTPS]
 ┌─────────────────────────────────────────────────────────────┐
 │  Ingress (L7 Router)                                        │
 │  - example.com/api ─▶ Service-A                             │
 │  - example.com/web ─▶ Service-B                             │
 └──────────┬────────────────────────────┬─────────────────────┘
            │                            │
            ▼ [Internal Cluster IP]      ▼
 ┌───────────────────────┐    ┌───────────────────────┐
 │ Service-A (LB)        │    │ Service-B (LB)        │
 └──────────┬────────────┘    └──────────┬────────────┘
            │                            │
            ▼ [Selector Matching]        ▼
 ┌─────────────────────────────────────────────────────────────┐
 │  Deployment (Manager)                                       │
 │  ┌───────────────────────────────────────────────────────┐  │
 │  │  ReplicaSet (Scale Logic)                             │  │
 │  │  ┌───────────────┐   ┌───────────────┐   ┌────────────┐│  │
 │  │  │ Pod (Container)│   │ Pod (Container)│   │ Pod ...    ││  │
 │  │  └───────────────┘   └───────────────┘   └────────────┘│  │
 │  └───────────────────────────────────────────────────────┘  │
 └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **Ingress**: 외부 트래픽의 관문으로, 도메인 이름이나 URL 경로를 보고 적절한 **Service**로 토스한다.
2. **Service**: 파드는 생성/삭제 시 IP가 변하므로, 고정된 VIP(Virtual IP)를 제공하여 클라이언트가 안정적으로 접근하게 한다.
3. **Deployment**: 사용자가 "파드 3개를 유지해줘"라고 선언하면 이를 감시하고, 버전 업데이트 시 기존 파드를 하나씩 죽이고 새 파드를 띄우는 전략을 관리한다.
4. **Pod**: 실제 비즈니스 로직이 돌아가는 컨테이너를 감싸고 있는 껍데기다. 서비스는 'Label Selector'를 통해 특정 라벨이 붙은 파드들을 묶어 트래픽을 분산한다.

#### 심층 동작 원리: 컨트롤러 루프 (Controller Loop)
쿠버네티스의 모든 오브젝트는 다음의 3단계 무한 루프를 통해 생존한다.
1. **관찰 (Observe)**: 현재 상태(Actual State)를 주기적으로 스캔한다. (예: 실행 중인 파드가 2개임)
2. **차이 발견 (Diff)**: 사용자가 설정한 원하는 상태(Desired State)와 대조한다. (예: 3개가 필요함)
3. **조치 (Act)**: 차이를 메우기 위한 명령을 수행한다. (예: 파드 1개를 새로 생성함)

- **📢 섹션 요약 비유**: 마치 자동 온도 조절기가 달린 방처럼, 우리가 목표를 정해주기만 하면 시스템이 알아서 수단과 방법을 가리지 않고 그 상태를 지켜내는 지능형 관리 체계입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 서비스 노출 방식의 정량적 비교

| 노출 방식 | 적용 레이어 | 주용도 | 장점 | 단점 |
|:---:|:---:|:---|:---|:---|
| **ClusterIP** | L4 | 클러스터 내부 통신 | 내부 보안성 우수 | 외부 접근 불가 |
| **NodePort** | L4 | 외부 노출 (테스트용) | 설정이 간단함 | 보안 취약, 포트 관리 어려움 |
| **LoadBalancer** | L4 | 운영 환경 외부 노출 | 클라우드 벤더 연동 | 추가 비용 발생 |
| **Ingress** | **L7** | 복잡한 라우팅, SSL | **도메인 단위 관리 가능** | 컨트롤러 별도 설치 필요 |

#### 2. 배포 전략 시너지 (Deployment Strategy)
- **Rolling Update**: 가용 자원을 유지하며 순차 교체. (무중단 서비스 필수)
- **Recreate**: 기존 버전 전체 삭제 후 신버전 투입. (데이터 정합성 중요 시 사용)
- **Blue-Green & Canary**: 쿠버네티스의 Service 레이블 변경 기능을 이용해 외부 서비스 메시(Istio 등)와 결합하여 고도의 트래픽 제어를 수행한다.

- **📢 섹션 요약 비유**: 내부 통신은 귓속말(ClusterIP)로, 외부 공개는 정식 대표전화(LoadBalancer)나 정밀한 안내 데스크(Ingress)를 통해 상황에 맞게 최적화하는 전략적 소통 구조와 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 트래픽 폭주 시의 오토스케일링 대응**
    - **문제**: 특정 이벤트 시 트래픽이 10배 급증하여 파드 응답이 지연됨.
    - **의사결정**: **HPA (Horizontal Pod Autoscaler)**를 Deployment에 적용. CPU 사용량이 70%를 넘으면 자동으로 파드 개수를 늘리도록 설정.
    - **결과**: 인프라 팀의 개입 없이도 시스템이 스스로 자원을 확장하여 서비스 가용성을 유지함.

- **도입 체크리스트**:
    1. **Resource Limits**: 모든 파드에 CPU/Memory 제한(Limit)과 요청(Request)을 명시했는가? (자원 잠식 방지)
    2. **Liveness/Readiness Probe**: 앱이 진짜 살아있는지, 요청을 받을 준비가 됐는지 체크하는 로직이 YAML에 포함됐는가?
    3. **Anti-Affinity**: 고가용성을 위해 동일한 서비스의 파드들이 서로 다른 물리 노드에 흩어지게 설정했는가?

- **안티패턴**: 
    - 파드에 직접 IP로 통신하는 행위 (Service를 거치지 않으면 장애 시 연결 끊김).
    - Deployment 없이 파드를 단독으로 생성하는 행위 (죽었을 때 자동 복구 안 됨).

- **📢 섹션 요약 비유**: "준비된 자만이 기회를 잡는다"는 말처럼, 철저한 리소스 제한과 헬스 체크 설정을 통해서만 클라우드의 파도(트래픽) 위에서 안정적인 항해(운영)가 가능합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량적 기대효과**:
    - **가동 시간(Uptime)**: 자가 치유(Self-healing) 기능을 통해 99.99% 이상의 고가용성 달성.
    - **운영 비용(OpEx)**: 자동화된 배포 및 스케일링으로 관리 인력 리소스 50% 절감.
- **미래 전망**: 서버리스 쿠버네티스(Fargate 등)의 확산으로 노드 관리 책임이 사라지고, 서비스 메시(Service Mesh)와의 완전한 결합을 통해 '인프라의 투명화'가 가속화될 전망이다.
- **참고 표준**: CNCF (Cloud Native Computing Foundation) 졸업 프로젝트 표준, OCI (Open Container Initiative) 규격.

- **📢 섹션 요약 비유**: 수동으로 바퀴를 돌리던 시대에서 자율주행 엔진으로 갈아타는 것과 같은 변화이며, 쿠버네티스는 그 자율주행의 핵심 제어 장치로 자리 잡았습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **파드(Pod)**는 장난감 로봇이 들어있는 **"작은 상자"**예요.
2. **디플로이먼트(Deployment)**는 로봇이 고장 나면 새 로봇을 꺼내주고, 친구들이 많이 오면 로봇을 더 많이 꺼내주는 **"똑똑한 창고 관리인"**이죠.
3. **서비스(Service)**는 친구들이 로봇과 놀고 싶을 때 "이쪽으로 와!"라고 알려주는 **"반짝이는 표지판"**이랍니다!
