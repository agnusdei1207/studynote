# 554. CQRS (Command Query Responsibility Segregation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CQRS (명령 및 조회 책임 분리)는 시스템의 데이터 변경 작업을 수행하는 **명령(Command) 모델**과 데이터 조회를 수행하는 **조회(Query) 모델**을 물리적 또는 논리적으로 분리하는 아키텍처 패턴이다.
> 2. **가치**: 읽기와 쓰기의 성능 요건이 상이한 분산 환경(MSA)에서, 조회 전용 DB를 별도로 구축하여 복잡한 조인(Join) 부하를 제거하고 **조회 성능을 극대화**하며 서비스의 확장성을 유연하게 관리한다.
> 3. **융합**: 이벤트 소싱(Event Sourcing) 패턴과 결합되어 데이터의 일치성을 비동기적으로 맞추며, '최종 일관성(Eventual Consistency)' 모델을 통해 대규모 동시성 처리가 필요한 이커머스 및 금융 시스템의 핵심 기술로 쓰인다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: "데이터를 저장하는 방법과 보여주는 방법을 다르게 가져가기"다. 전통적인 방식(CRUD)에서는 하나의 도메인 모델로 저장과 조회를 모두 처리했지만, CQRS는 이를 두 개의 전문화된 길로 나눈다.

- **💡 비유**: CQRS는 **"신문의 편집부와 배달부"**의 관계와 같습니다. 기사를 쓰고 편집하는 과정(Command)은 아주 꼼꼼하고 복잡한 절차를 거치지만, 독자가 신문을 읽는 과정(Query)은 이미 인쇄된 종이를 펼쳐보기만 하면 되므로 아주 빠릅니다. 독자가 기사 수정 권한을 가질 필요는 없는 것과 같습니다.

- **등장 배경**:
  1. **조회 성능의 한계**: 복잡한 비즈니스 로직이 담긴 DB 테이블을 수십 개 조인하여 화면에 뿌려주는 방식은 데이터가 늘어날수록 급격히 느려졌다.
  2. **성능 불균형**: 보통 웹 서비스는 읽기(Read) 비중이 쓰기(Write)보다 10~100배 이상 높다. 이 둘을 하나의 모델로 묶어두는 것은 비효율적이었다.
  3. **MSA와 분산 데이터**: 여러 서비스에 흩어진 데이터를 한 번에 조회해야 하는 요구사항이 생기면서, 조회만을 위한 '통합 뷰(View) DB'가 필요해졌다.

- **📢 섹션 요약 비유**: 입구와 출구가 하나인 좁은 문(CRUD) 대신, 입구는 튼튼한 철문(Command)으로 만들고 출구는 넓은 자동문(Query)으로 만들어 사람들의 흐름을 원활하게 하는 건축 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. CQRS의 단계별 구현 수준

| 단계 | 특징 | 상세 설명 | 비유 |
|:---|:---|:---|:---|
| **Level 1** | **코드 분리** | 동일 DB를 쓰되 소스 코드 상에서 CUD 클래스와 R 클래스 분리 | 한 장부에 펜 색깔만 나누기 |
| **Level 2** | **DB 분리** | 쓰기용 RDBMS와 읽기용 NoSQL을 따로 운영 | 원본 장부와 복사본 배포 |
| **Level 3** | **이벤트 결합** | 이벤트 소싱을 통해 실시간으로 읽기 전용 모델 업데이트 | 실시간 뉴스 전광판 업데이트 |

---

### 2. CQRS 통합 아키텍처 (Event-driven)

쓰기 모델의 변화가 어떻게 읽기 모델로 전파되는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      CQRS (Command Query) 분리 아키텍처 흐름                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. Command Side (Write)]                 [2. Query Side (Read)]            │
 │      │                                           ▲                          │
 │      ▼ (데이터 변경 요청)                          │ (초고속 조회)             │
 │  ┌──────────────────┐                    ┌──────────────────┐               │
 │  │  Domain Model    │                    │  Read Model      │               │
 │  │ (Write-optimized)│                    │ (Query-optimized)│               │
 │  └──────┬───────────┘                    └──────▲───────────┘               │
 │         │                                       │                           │
 │         ▼ [3. Event Bus]                        │                           │
 │  ┌──────────────────────────────────┐           │                           │
 │  │  Event: "OrderCreated"           │ ──────────┘                           │
 │  │  (Status Changed)                │ (Asynchronous Data Sync)              │
 │  └──────────────────────────────────┘                                       │
 │                                                                             │
 │  ※ 특징: 쓰기 DB는 정규화(RDBMS), 읽기 DB는 비정규화(Redis/ES) 선호                │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처는 완벽히 이원화되어 있다. 1) **Command Side**는 데이터의 무결성이 중요하므로 엄격한 **정규화**와 트랜잭션을 지원하는 RDBMS를 주로 쓴다. 2) 데이터가 바뀌면 '사건(Event)'을 발행한다. 3) **Query Side**는 이 이벤트를 받아 **Read Model**을 업데이트한다. 이때 Read Model은 조인이 필요 없도록 데이터를 미리 다 합쳐놓은(Materialized View) **NoSQL**이나 **Elasticsearch**를 사용하여 검색 속도를 비약적으로 높인다. 두 모델 사이에는 0.몇 초의 시차가 발생할 수 있는데, 이를 **최종 일관성**이라 부른다.

---

### 심층 동작 원리: 최종 일관성 (Eventual Consistency)

CQRS를 받아들이기 위해 비즈니스적으로 수용해야 할 개념이다.
1. **정의**: 데이터 변경 후 조회 시점에 아주 잠깐 예전 데이터가 보일 수 있지만, 결국에는 모든 노드가 동일한 최신 데이터를 갖게 됨.
2. **원리**: 분산 시스템에서 가용성(Availability)을 높이기 위해 즉시성(Strong Consistency)을 포기하는 전략.
3. **가치**: "지금 당장 100% 일치"보다 "언제 어디서든 0.1초 만에 응답"이 더 중요한 포털, SNS, 쇼핑몰의 표준 모델이다.

- **📢 섹션 요약 비유**: 인스타그램에 사진을 올렸을 때, 내 친구의 피드에는 1초 뒤에 나타나는 현상과 같습니다. 전 세계 모든 사람의 화면이 동시에 바뀌길 기다리면 인스타그램은 멈춰버릴 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### CRUD vs CQRS 상세 비교

| 비교 항목 | 전통적 CRUD | CQRS 패턴 |
|:---|:---|:---|
| **모델 구조** | 단일 모델 (상태 저장/조회 혼재) | **명령과 조회 모델의 완전 분리** |
| **성능 최적화** | 어려움 (조회 성능 vs 정규화 충돌) | **매우 쉬움 (각자 전용 DB 사용)** |
| **일관성** | 강력한 일관성 (Strong) | **최종 일관성 (Eventual)** |
| **복잡도** | 낮음 | **높음 (데이터 동기화 메커니즘 필요)** |
| **비유** | 만능 칼 한 자루 | 전문 요리 도구 세트 |

---

### 기술 융합: CQRS와 이벤트 소싱 (Event Sourcing)

CQRS의 단짝 친구인 이벤트 소싱과의 시너지다.
- **연결**: 쓰기 모델에서 현재 상태를 저장하는 대신 '모든 변경 이력(Event)'을 저장한다.
- **효과**: 이 이벤트를 재생(Replay)하기만 하면, 언제든 다양한 형태의 '읽기 모델'을 무한정 새로 만들어낼 수 있다. (예: 1년 전 매출 통계용 읽기 DB 즉시 생성)

- **📢 섹션 요약 비유**: 요리 완성본만 남기는 게 아니라, 요리하는 전 과정을 비디오(이벤트 소싱)로 찍어두는 것입니다. 그러면 나중에 언제든 그 비디오를 보고 다른 그릇(읽기 모델)에 요리를 다시 담아낼 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 대규모 이커머스 '상품 상세 페이지' 고도화

- **문제 상황**: 상품 정보, 리뷰, 배송 정보, 쿠폰 혜택 등을 한 화면에 보여주려니 조인이 수십 개 발생하여 페이지 로딩이 3초 이상 걸림.
- **의사결정 (CQRS 적용 전략)**:
  1. **Command DB**: 상품 등록 및 수정은 기존 Oracle DB(정규화)에서 처리한다.
  2. **Event Relay**: 데이터 변경 시 'ProductUpdated' 이벤트를 Kafka로 쏜다.
  3. **Read DB**: 이벤트를 구독하여 **MongoDB**에 상품 상세 페이지에 필요한 모든 정보를 통째로 JSON 하나로 담아둔다. (Denormalization)
  4. **결과**: 화면 호출 시 조인 없이 단 한 번의 Key-Value 조회로 모든 정보를 가져와 응답 속도를 0.1초로 단축한다.

### 도입 체크리스트
- **Latency Tolerance**: 비즈니스에서 데이터 시차를 몇 초까지 허용하는가?
- **Data Integrity**: 읽기 모델이 오염되었을 때, 쓰기 모델(원본)로부터 다시 복구할 수 있는 파이프라인이 있는가?
- **Cost**: 두 종류의 DB를 운영함에 따른 인프라 비용과 관리 공수를 감당할 수 있는가?

### 안티패턴
- **Everything is CQRS**: 단순한 관리자 설정 페이지나 트래픽이 적은 서비스에 굳이 CQRS를 도입하여 복잡성만 키우는 경우.
- **Synchronous Sync**: 쓰기 DB에 저장하고 나서 조회 DB 업데이트를 동기식으로 기다리는 행위. (CQRS의 성능 이점을 완전히 깎아먹음)

- **📢 섹션 요약 비유**: 편의점에서 껌 한 통 파는 데(단순 기능) 거대 물류 창고 관리 시스템(CQRS)을 도입하지 마세요. 도구가 문제보다 크면 배보다 배꼽이 더 커집니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Perf)** | 조회 성능 10배~100배 향상 | 인덱싱에 최적화된 데이터 구조와 비정규화 효과 |
| **정량적 (Scale)** | 읽기 전용 노드 무한 확장 | 트래픽이 몰리는 조회 서버만 별도로 증설하여 자원 효율화 |
| **정성적 (Stability)** | 쓰기/읽기 장애 격리 | 조회 서버가 다운되어도 데이터 저장(주문)은 정상 작동 유지 |
| **정성적 (Standard)** | 비즈니스 로직의 명확화 | 명령 모델은 도메인에만, 조회 모델은 UI에만 집중 가능 |

### 미래 전망
- **Polyglot Persistence Standard**: RDBMS는 기록용으로, NoSQL은 조회용으로 사용하는 혼합형 데이터 아키텍처가 모든 엔터프라이즈의 표준이 될 것이다.
- **Serverless CQRS**: 클라우드가 이벤트 전파와 읽기 모델 동기화를 알아서 처리해주는 'No-ops CQRS' 서비스가 발전할 것이다.

### 참고 표준
- **Domain-Driven Design (Eric Evans)**: CQRS의 배경이 된 설계 방법론
- **CAP Theorem**: 분산 시스템의 근본적인 선택 기준
- **Materialized View**: DB 차원의 CQRS 구현 기술

- **📢 섹션 요약 비유**: "가장 빠른 조회는 미리 계산해 둔 것을 보여주는 것이다." CQRS는 현대의 방대한 데이터를 가장 현명하게 사용자에게 전달하는 '디지털 전광판'의 철학입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **CQRS**는 장난감 상자에 장난감을 **"넣는 입구"와 "구경하는 유리창"을 따로 만드는 것**과 같아요.
2. 장난감을 정리할 때는 꼼꼼히 넣어야 하지만(Command), 구경할 때는 유리창만 슥 보면 되니까(Query) **훨씬 빠르고 편하답니다.**
3. 넣는 쪽이 복잡해도 구경하는 쪽은 항상 깔끔하게 보여줄 수 있는 똑똑한 보물 상자 규칙이에요!
