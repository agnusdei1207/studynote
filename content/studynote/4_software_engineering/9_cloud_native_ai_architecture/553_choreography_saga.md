# 553. 코레오그래피 사가 (Choreography Saga)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 코레오그래피 사가(Choreography Saga)는 중앙의 통제자 없이 서비스들이 서로의 **이벤트를 교환하며 연쇄적으로 트랜잭션을 처리**하는 분산 반응형 사가 모델이다.
> 2. **가치**: 서비스 간의 결합도를 최소화하고 중앙 병목 현상을 제거하여 시스템의 유연성과 확장성을 극대화하며, 상대적으로 단순한 비즈니스 로직을 가진 마이크로서비스(MSA) 환경에서 빠른 구현이 가능하다.
> 3. **융합**: 이벤트 기반 아키텍처(EDA)와 결합되어 비동기 메시지 브로커를 통해 소통하며, 각 서비스가 다음 단계와 보상 트랜잭션 로직을 스스로 판단하여 실행하는 '자율적 협업'을 실현한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 코레오그래피(Choreography)는 '무용 안무'를 뜻한다. 무용수들이 지휘자 없이도 서로의 동작을 보고 다음 동작을 이어가는 것처럼, 각 마이크로서비스가 다른 서비스의 성공 이벤트를 보고 자기 일을 시작하는 방식이다.

- **💡 비유**: 코레오그래피 사가는 **"릴레이 달리기"**와 같습니다. 다음 주자는 앞 주자가 달려와서 바톤(이벤트)을 넘겨주기만을 기다립니다. 선생님(지휘자)이 "지금 뛰어!"라고 시키지 않아도, 바톤을 받는 행위 자체가 뛰어야 한다는 신호가 되어 전체 경기가 진행되는 것과 같습니다.

- **등장 배경**:
  1. **오케스트레이터의 복잡성 회피**: 소규모 시스템에서 굳이 비싼 비용을 들여 중앙 관리 서버를 구축하고 운영하는 부담을 줄이고자 했다.
  2. **극단적 분산화 지향**: 중앙 통제 지점(SPOF)을 없애고, 각 서비스가 완벽한 자율권을 갖는 진정한 의미의 마이크로서비스를 구현하려 했다.
  3. **이벤트 드리븐의 확산**: Kafka와 같은 고성능 메시지 브로커가 보편화되면서, 이벤트 기반의 연쇄 반응을 구현하기가 매우 쉬워졌다.

- **📢 섹션 요약 비유**: 복잡한 매뉴얼 대신 "전등이 켜지면 박수를 친다"는 식의 간단한 약속만으로 수많은 사람이 질서 있게 움직이게 만드는 '자연스러운 질서'의 아키텍처입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 코레오그래피 사가의 구성 요소

| 요소 | 역할 및 정의 | 비유 |
|:---|:---|:---|
| **Event Producer** | 자신의 로컬 트랜잭션을 마치고 소식을 알리는 서비스 | 바톤을 넘겨주는 주자 |
| **Message Broker** | 이벤트를 나르고 보관하는 통로 (Topic 기반) | 달리기 트랙 |
| **Event Consumer** | 특정 이벤트를 기다렸다가 자기 일을 수행하는 서비스 | 바톤을 받는 다음 주자 |
| **Compensation Logic** | 실패 이벤트 감지 시 수행하는 취소 로직 | 경기 무효 시 제자리로 돌아가기 |

---

### 2. 코레오그래피 사가 동작 아키텍처 (Chained Events)

서비스들이 꼬리에 꼬리를 무는 형태로 통신하는 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      코레오그래피 사가 (Decentralized Events) 구조               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [주문 서비스] ─────▶ [메시지 브로커] ─────▶ [결제 서비스] ─────▶ [메시지 브로커] │
 │ (OrderCreated)       (Topic: Order)      (PaymentSuccess)     (Topic: Pay)  │
 │                                                                             │
 │                                                                  │          │
 │ [완료] ◀───── [메시지 브로커] ◀───── [배송 서비스] ◀──────────────────┘          │
 │               (ShipStarted)        (Topic: Ship)                            │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ⚠ 실패 시 보상 흐름 (Rollback Chain):                                     │  │
 │  │ 1. [결제 서비스] ──▶ "PaymentFailed" 이벤트 발행                           │  │
 │  │ 2. [주문 서비스] ──▶ "PaymentFailed" 구독 후 주문 상태 '취소'로 변경         │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 중앙 지휘자가 없다. **주문 서비스**는 주문을 받고 'OrderCreated' 이벤트를 발행한 뒤 잊어버린다. **결제 서비스**는 그 이벤트를 구독하고 있다가 돈을 빼고 'PaymentSuccess' 이벤트를 발행한다. 다시 **배송 서비스**가 이를 이어받는다. 만약 중간에 결제가 실패하면, 결제 서비스가 'PaymentFailed' 이벤트를 발행하고, 이 소식을 들은 주문 서비스가 스스로 주문을 취소한다. 모든 서비스가 **관심 있는 소식(Event)**에만 귀를 기울이고 반응하는 분산형 구조다.

---

### 심층 동작 원리: 발행-구독 (Pub-Sub) 모델

코레오그래피를 지탱하는 핵심 기술이다.
1. **익명성**: 주문 서비스는 내 소식을 누가 듣는지(결제팀인지 마케팅팀인지) 알 필요가 없다.
2. **느슨한 결합**: 새로운 서비스(예: 알림톡 서비스)를 추가하고 싶으면, 기존 시스템 수정 없이 'OrderCreated' 이벤트를 구독하는 서비스 하나만 새로 띄우면 된다.
3. **가치**: 시스템의 확장성이 무한대에 가까워지며, 서비스 간의 직접적인 의존성을 완벽히 제거한다.

- **📢 섹션 요약 비유**: 마을 잔치 소식을 벽보(이벤트)에 붙여두면, 고기 먹고 싶은 사람과 춤추고 싶은 사람이 각자 알아서 모이는 것과 같습니다. 이장은 누가 올지 일일이 초대장을 보낼 필요가 없습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 코레오그래피 vs 오케스트레이션 상세 비교

| 비교 항목 | 코레오그래피 (Choreography) | 오케스트레이션 (Orchestration) |
|:---|:---|:---|
| **제어 중심** | **분산 (각자 판단)** | 중앙 (지휘자 판단) |
| **구현 난이도** | 쉬움 (작은 규모에서) | 어려움 (초기 인프라 구축 필요) |
| **가시성** | **낮음 (흐름 파악이 힘듦)** | 높음 (중앙 대시보드 가능) |
| **결합도** | **최소 (가장 느슨함)** | 중간 |
| **추천 상황** | 간단한 선형 로직, 독립성 최우선 | 복잡한 분기, 중앙 통제 필수 상황 |

---

### 기술 융합: 코레오그래피와 서킷 브레이커

비동기 연쇄 반응에서 발생할 수 있는 장애를 방어한다.
- **Backpressure**: 소비자가 이벤트를 처리하지 못할 때 브로커가 속도를 조절하게 한다.
- **Dead Letter Queue (DLQ)**: 처리에 실패한 이벤트를 별도로 모아 나중에 수동으로 사가 흐름을 재개하거나 보상 트랜잭션을 유도한다.

- **📢 섹션 요약 비유**: 릴레이 주자가 넘어져서 바톤을 못 받으면, 진행 요원(DLQ)이 나타나서 바톤을 챙겨두었다가 나중에 다시 경기를 진행시키는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '회원가입 사은품 증정' 이벤트의 코레오그래피 설계

- **문제 상황**: 회원가입이 완료되면 1) 웰컴 메일 발송, 2) 5,000 포인트 적립, 3) 추천인에게 1,000 포인트 적립이 동시에 일어나야 함.
- **의사결정 (코레오그래피 적용 전략)**:
  1. **이벤트 정의**: 회원 서비스는 가입 성공 시 'UserSignedUp' 이벤트를 발행한다.
  2. **구독자 배치**: 메일 서비스와 포인트 서비스가 각각 이 이벤트를 구독하게 한다.
  3. **보상 로직 생략**: 포인트 적립이 실패해도 회원가입을 취소할 필요는 없는 '비핵심 트랜잭션'이므로, 실패 시 재시도(Retry)만 하고 사가는 종결한다.
  4. **확장**: 나중에 "가입 축하 쿠폰" 기능이 추가되면, 그저 쿠폰 서비스가 'UserSignedUp'을 구독하게 설정만 추가한다.

### 도입 체크리스트
- **Cycle Dependency**: A가 이벤트를 던져 B가 반응하고, B가 다시 A가 반응하는 이벤트를 던져 '무한 루프'에 빠지지는 않는가?
- **Observability**: 수십 개의 이벤트가 복잡하게 얽혔을 때, 전체 흐름을 추적할 수 있는 **Trace ID** 공유 체계가 잡혀 있는가?
- **Eventual Consistency Time**: 데이터가 최종적으로 일치하기까지 걸리는 시차를 비즈니스 조직에서 허용했는가?

### 안티패턴
- **스파게티 이벤트**: 누가 어떤 이벤트를 왜 구독하는지 아무도 모르는 상태. (이벤트 카탈로그 문서화 필수)
- **이벤트를 통한 강한 결합**: 이벤트 메시지 안에 너무 많은 데이터를 담아, 메시지 규격 하나 바꿀 때 모든 구독자를 다 고쳐야 하는 상황.

- **📢 섹션 요약 비유**: 소문이 꼬리에 꼬리를 물고 퍼지다 보면 원래 내용이 왜곡되거나(데이터 불일치), 소문의 발원지를 찾기 힘든(디버깅 지옥) 상황을 경계해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Scaling)** | 시스템 처리량 3배 향상 | 동기 대기가 없어 CPU와 네트워크 자원 효율 극대화 |
| **정량적 (Agility)** | 신규 기능 개발 시간 50% 단축 | 기존 시스템 수정 없이 신규 소비자 추가만으로 확장 |
| **정성적 (Standard)** | 진정한 서비스 자율성 확보 | 서비스 간 간섭 없는 독립 배포 및 기술 선택권 보장 |
| **정성적 (Resilience)** | 부분 장애에 강한 구조 | 특정 구독자가 죽어도 전체 시스템의 '메인 흐름'은 유지됨 |

### 미래 전망
- **Event-Driven AI**: AI가 실시간 이벤트 흐름을 분석하여, 다음에 어떤 서비스가 실행되어야 할지 동적으로 안무를 짜주는 '지능형 코레오그래피'가 등장할 것이다.
- **Standardized Event Schema**: 서비스 간 오해를 없애기 위해 **CloudEvents**와 같은 공통 이벤트 규격이 모든 인프라에서 기본으로 지원될 것이다.

### 참고 표준
- **Reactive Manifesto**: 반응성 및 메시지 중심 설계 표준
- **CloudEvents (CNCF)**: 이벤트 기반 상호 운용성 표준
- **AMQP / MQTT**: 메시지 전달을 위한 표준 프로토콜

- **📢 섹션 요약 비유**: "누구의 간섭도 받지 않고 흐르는 물처럼." 코레오그래피 사가는 마이크로서비스들이 각자의 개성을 뽐내면서도, 보이지 않는 이벤트의 끈으로 연결되어 하나의 거대한 생태계를 이루는 가장 자연스러운 아키텍처입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 친구들과 놀이터에서 **"도미노 게임"을 하는 것**과 같아요. 내가 첫 번째 도미노를 쓰러뜨리면, 그다음 도미노들이 줄줄이 넘어가는 규칙이죠.
2. 선생님이 시키지 않아도 **옆에 있는 도미노가 넘어지는 것(이벤트)만 보고 내가 할 일을 하는 거랍니다.**
3. 중간에 도미노 하나가 안 넘어가면, 다시 예쁘게 세워놓기로 친구들과 미리 약속해두면 아주 재미있게 놀 수 있어요!
