# 541. 클라이언트 사이드 vs 서버 사이드 디스커버리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클라이언트 사이드 디스커버리는 **호출 주체가 직접 레지스트리를 조회**하여 대상의 위치를 찾는 방식이며, 서버 사이드 디스커버리는 클라이언트가 **로드 밸런서를 호출하면 로드 밸런서가 대신 위치를 찾아 연결**해주는 방식이다.
> 2. **가치**: 클라이언트 방식(Eureka)은 로드 밸런서라는 병목 지점을 없애고 세밀한 부하 분산 로직을 가능케 하며, 서버 방식(Consul, K8s)은 클라이언트 코드가 인프라 지식으로부터 독립되어 언어에 상관없이 유연한 개발이 가능하게 한다.
> 3. **융합**: 서비스 메시(Service Mesh) 환경에서는 두 방식의 장점이 융합되어, 사이드카 프록시가 투명하게 디스커버리와 로드 밸런싱을 수행하는 형태로 발전하고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 서비스 디스커버리를 구현하는 물리적 위치에 따른 분류다. 누구의 손에 "전화번호부"를 들려줄 것인가의 문제다.

- **💡 비유**: 
  - **클라이언트 사이드**: 손님이 **"직접 맛집 지도(Registry)"**를 들고 식당을 찾아가는 것과 같습니다. (길은 잘 알지만 지도를 계속 업데이트해야 함)
  - **서버 사이드**: 손님이 **"호텔 안내 데스크(Load Balancer)"**에 가서 식당을 예약해달라고 부탁하는 것과 같습니다. (데스크만 알면 되지만, 데스크가 붐비면 대기 시간이 생김)

- **등장 배경**:
  1. **Netflix의 혁신 (Client-side)**: 대규모 트래픽을 처리하던 넷플릭스는 중앙 로드 밸런서의 부하를 줄이기 위해 클라이언트가 직접 최적의 경로를 찾는 **Eureka + Ribbon** 방식을 정립했다.
  2. **클라우드 및 쿠버네티스의 보편화 (Server-side)**: 서비스가 수만 개로 늘어나면서 모든 앱에 디커스버리 코드를 넣는 것이 부담스러워졌고, **Consul**이나 **K8s Service** 같은 인프라 중심의 방식이 선호되기 시작했다.

- **📢 섹션 요약 비유**: 내가 직접 내비게이션을 켜고 운전할 것인지(클라이언트), 아니면 목적지만 말하면 데려다주는 택시를 탈 것인지(서버)를 결정하는 네트워크 설계 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 두 방식의 구조적 차이 비교

| 비교 항목 | 클라이언트 사이드 (Client-side) | 서버 사이드 (Server-side) |
|:---|:---|:---|
| **메인 흐름** | Client ──▶ Registry ──▶ Server | Client ──▶ Load Balancer ──▶ Server |
| **로드 밸런싱** | **클라이언트 내부 라이브러리**에서 수행 | **외부 로드 밸런서** 장치에서 수행 |
| **의존성** | 클라이언트가 서비스 위치를 알아야 함 | 클라이언트는 LB 주소만 알면 됨 |
| **대표 기술** | **Netflix Eureka + Ribbon** | **AWS ELB, K8s Service, Consul** |

---

### 2. 아키텍처 상세 다이어그램

데이터가 흐르는 경로와 컴포넌트 간의 관계를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      서비스 디스커버리 구현 모델 비교 아키텍처                    │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Model A: Client-side (Eureka)]             [Model B: Server-side (K8s)]    │
 │                                                                             │
 │  ┌────────────┐      ┌────────────┐          ┌────────────┐                 │
 │  │ Client App │ ──▶  │ Service    │          │ Client App │                 │
 │  │ (w/ Ribbon)│ ◀──  │ Registry   │          └─────┬──────┘                 │
 │  └─────┬──────┘      └────────────┘                │                        │
 │        │ (Direct Call)                             ▼                        │
 │        ▼                                     ┌────────────┐                 │
 │  ┌────────────┐                              │   Load     │ ──▶ [Server A]  │
 │  │  Server A  │                              │  Balancer  │ ──▶ [Server B]  │
 │  └────────────┘                              └────────────┘                 │
 │                                                                             │
 │  - 장점: 중간 지연 없음, 분산 제어             - 장점: 클라이언트 구현 단순함      │
 │  - 단점: 언어별 라이브러리 필요                - 단점: LB 장치 관리 및 병목 가능성  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** **클라이언트 사이드** 방식은 클라이언트 앱 내부에 똑똑한 라이브러리(Ribbon 등)가 들어있다. 이 라이브러리가 레지스트리에서 받은 서버 목록 중 가장 건강한 곳을 골라 **직접(Direct)** 호출한다. 중간에 거치는 장비가 없어 매우 빠르다. 반면 **서버 사이드** 방식은 클라이언트는 멍청하다. 오직 정해진 관문(Load Balancer)에만 요청을 던지면, 관문이 레지스트리(예: K8s etcd)를 뒤져서 적절한 서버로 요청을 **대리 전달(Proxying)** 한다.

---

### 심층 동작 원리: Consul의 다목적성 (Server-side + @)

단순 주소 관리를 넘어선 서비스 메시의 기초다.
1. **Health Check**: 단순히 포트가 열렸는지가 아니라, "메모리 80% 이상인가?", "DB 연결 끊겼나?" 등 정밀 체크 수행.
2. **Key-Value Store**: 설정 정보(Config)를 실시간으로 공유하는 저장소 역할 겸임.
3. **가치**: 서버 사이드 방식을 취하면서도, 에이전트 기반의 분산 구조를 가져 가용성을 극대화한다.

- **📢 섹션 요약 비유**: 동네마다 작은 안내 데스크(Consul Agent)를 두고, 본사 중앙 센터와 실시간으로 정보를 동기화하는 촘촘한 프랜차이즈 안내 망과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 기술 선택 가이드라인

| 조직 상황 | 추천 방식 | 이유 |
|:---|:---:|:---|
| **Spring Cloud 기반 Java 환경** | **Client-side** | Eureka 연동이 매우 쉽고 넷플릭스 기술셋의 시너지가 큼 |
| **다양한 언어 혼용 (Polyglot)** | **Server-side** | 언어별로 디스커버리 라이브러리를 만들 필요가 없음 |
| **쿠버네티스 표준 운영** | **Server-side** | K8s 'Service'와 'DNS' 기능에 디스커버리가 내장되어 있음 |
| **고성능 저지연 필수** | **Client-side** | 네트워크 홉을 최소화하여 밀리초 단위의 속도 확보 |

---

### 기술 융합: 서비스 메시와 사이드카 프록시

두 방식의 단점을 보완한 최신 융합 아키텍처다.
- **원리**: 클라이언트 코드에는 디스커버리 로직이 없지만(Server-side의 장점), 바로 옆의 **사이드카 프록시(Envoy)**가 클라이언트 대신 레지스트리를 조회하여 직접 서버로 요청을 보낸다(Client-side의 성능).
- **가치**: 개발자는 코딩에만 집중하고, 인프라 계층에서 고성능 디스커버리가 투명하게 일어난다.

- **📢 섹션 요약 비유**: 길을 모르는 주인(앱) 대신, 옆에 앉은 비서(사이드카)가 지도를 보고 운전기사에게 길을 알려주는(디스커버리) 지능형 의전 시스템과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 자바 서비스와 파이썬 서비스의 상호 호출 설계

- **문제 상황**: 자바(Spring) 기반 주문 서비스가 파이썬(FastAPI) 기반의 추천 서비스를 호출해야 함. 주소는 동적으로 변함.
- **의사결정 (디스커버리 전략)**:
  1. **Eureka(Client-side) 포기**: 파이썬용 유레카 라이브러리 관리가 번거롭고 일관성이 떨어짐.
  2. **Consul 또는 K8s Service(Server-side) 채택**: 언어에 독립적인 **DNS 방식**이나 **로드 밸런서 방식**을 선택한다.
  3. **내부 통신 최적화**: 쿠버네티스의 `order-service.default.svc.cluster.local`과 같은 표준 도메인 네임을 사용하여, 서비스가 어디로 이사 가든 이름만으로 소통하게 한다.
  4. **결과**: 개발자는 `http://recommend-service/` 주소만 코드에 적으면 되고, 실제 연결은 인프라가 책임지는 깔끔한 구조를 완성한다.

### 도입 체크리스트
- **Network Overhead**: 서버 사이드 방식 도입 시 로드 밸런서가 단일 장애점(SPOF)이나 병목이 되지 않도록 이중화가 되었는가?
- **TTL (Time To Live)**: 클라이언트 사이드 방식에서 서버 목록을 캐싱할 때, 실제 서버 상태와 캐시 사이의 시차(Stale data)를 어떻게 제어할 것인가?
- **Cross-platform**: 온프레미스와 클라우드 자원이 섞여 있을 때 두 영역을 모두 지원하는 솔루션인가?

### 안티패턴
- **Smart LB, Stupid Code**: 로드 밸런서에 너무 많은 비즈니스 로직을 넣어 LB 설정이 소스 코드보다 복잡해지는 상황.
- **Library Sprawl**: 팀마다 다른 버전의 디스커버리 라이브러리를 써서, 어떤 팀은 서버를 찾는데 어떤 팀은 못 찾는 불일치 현상.

- **📢 섹션 요약 비유**: "누가 더 똑똑해야 하는가"의 문제입니다. 내가 똑똑해질지(Client-side), 아니면 똑똑한 비서를 고용할지(Server-side) 서비스의 규모와 다양성에 맞춰 선택해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Complexity)** | 설정 파일 관리 공수 80% 감소 | 수천 개의 IP 주소를 수동 관리하던 비효율 제거 |
| **정량적 (Performance)** | 네트워크 지연 시간 최적화 | 클라이언트 사이드 도입 시 불필요한 게이트웨이 경유 제거 |
| **정성적 (Standard)** | 폴리글랏 개발 환경 구축 | 언어 장벽 없는 서비스 간 소통 기반 마련 |
| **정성적 (Stability)** | 동적 환경의 안정성 확보 | 인스턴스 교체 시에도 중단 없는 트래픽 전환 실현 |

### 미래 전망
- **Zero-config Networking**: 서비스가 뜨는 순간 물리적 위치와 상관없이 신원(Identity)만으로 망이 자동 구성되는 '인텐트 기반 네트워크'와 융합될 것이다.
- **Service Mesh Native**: 별도의 디스커버리 서버 개념이 사라지고, 플랫폼(K8s 등) 자체가 완벽한 디스커버리 기능을 기본값으로 제공하게 될 것이다.

### 참고 표준
- **Netflix Open Source Software (OSS)**: 클라이언트 사이드 방식의 선구자
- **HashiCorp Consul Documentation**: 서버 사이드 및 서비스 메시 가이드
- **Kubernetes Networking Model**: 클러스터 내 디스커버리 표준

- **📢 섹션 요약 비유**: "찾는 자가 임자"인 시대에서 "알아서 찾아주는" 시대로, 서비스 디스커버리는 마이크로서비스를 더 이상 '고립된 섬'이 아닌 '유기적인 생명체'로 연결하고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **클라이언트 사이드**는 내가 직접 친구들 자리가 적힌 **"자리표"**를 들고 다니면서 친구를 찾아가는 거예요.
2. **서버 사이드**는 반장 선생님께 **"친구 어디 있어?"**라고 물어보면, 선생님이 내 손을 잡고 친구 자리까지 데려다주는 거랍니다.
3. 내가 스스로 찾으면 빠르고(클라이언트), 선생님께 물어보면 머리를 안 써도 돼서 편해요(서버)!
