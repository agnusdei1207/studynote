# 533. 비즈니스 능력에 따른 분해 (Decompose by Business Capability)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비즈니스 능력에 따른 분해는 기업의 조직도나 부서별 기능(Marketing, Sales, HR 등)을 기준으로 마이크로서비스의 경계를 나누는 **하향식(Top-down) 서비스 분할 전략**이다.
> 2. **가치**: 비즈니스 가치를 창출하는 실제 업무 단위와 소프트웨어 구조를 일치시킴으로써, 경영진의 요구사항이 시스템에 즉각 반영되는 **비즈니스 민첩성**을 확보하고 팀별 독립적인 예산과 의사결정 권한을 부여한다.
> 3. **융합**: 전사 아키텍처(EA)의 비즈니스 아키텍처 계층과 긴밀히 연계되며, 분해된 각 서비스는 내부적으로 도메인 주도 설계(DDD)를 통해 더욱 정교한 논리 구조로 발전한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: "회사가 무슨 일을 하는가?"라는 질문에서 시작하여 서비스를 나누는 방식이다. 기술적인 공통점(DB, 캐시 등)이 아니라, 업무적 연관성을 기준으로 서비스를 획정한다. 

- **💡 비유**: 비즈니스 능력별 분해는 **"백화점의 층별 구성"**과 같습니다. '의류', '가전', '식품'처럼 고객에게 제공하는 서비스 종류에 따라 층(서비스)을 나누고, 각 층에는 그 물건을 가장 잘 아는 전문 판매원(개발팀)을 배치하는 방식입니다.

- **등장 배경**:
  1. **IT와 비즈니스의 괴리**: 과거에는 'DB팀', '화면팀' 등으로 나누어 소통이 어려웠다. 이를 해결하기 위해 '비즈니스 가치'를 중심으로 조직과 코드를 재편해야 했다.
  2. **콘웨이의 법칙 (Conway's Law)**: "시스템의 구조는 그 조직의 의사소통 구조를 반영한다"는 원리에 따라, 효율적인 아키텍처를 위해 먼저 비즈니스 조직과 서비스 단위를 맞추고자 했다.
  3. **책임 경영 체계**: 특정 서비스의 매출과 비용을 명확히 추적하기 위해 서비스 단위를 비즈니스 부서 단위로 쪼갤 필요가 생겼다.

- **📢 섹션 요약 비유**: 복잡한 대가족 살림(모놀리식)을 정리할 때, '돈 관리', '빨래', '요리' 등 하는 일별로 담당자를 정하고 각자 독립된 방(서비스)을 주는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 비즈니스 능력 식별 프로세스

현실의 조직 구조를 디지털 서비스로 투영하는 단계다.

| 단계 | 수행 작업 | 핵심 질문 | 비유 |
|:---|:---|:---|:---|
| **1. 업무 분석** | 기업의 전체 가치 사슬 (Value Chain) 분석 | "우리는 무엇으로 돈을 버는가?" | 가게 업종 정하기 |
| **2. 능력 정의** | 계층형 업무 기능 목록 작성 (Function Map) | "그 일을 위해 어떤 구체적 기능이 필요한가?" | 부서 나누기 |
| **3. 서비스 매핑** | 각 업무 기능을 독립된 마이크로서비스로 전환 | "이 부서를 하나의 독립된 가게로 만들 수 있는가?" | 지점 개설하기 |

---

### 2. 비즈니스 능력별 분해 아키텍처 모델

조직의 기능이 어떻게 소프트웨어 서비스로 변환되는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Business Capability-based Decomposition                │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Enterprise Level (가치 사슬 분석)]                                            │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ 마케팅 (Marketing) │ 판매 (Sales) │ 재고 (Inventory) │ 배송 (Logistic)    │  │
 │  └────────┬──────────┴──────┬───────┴────────┬─────────┴────────┬────────┘  │
 │           │                 │                │                  │           │
 │           ▼                 ▼                ▼                  ▼           │
 │ [Service Level (MSA 분해)]                                                   │
 │  ┌────────────┐      ┌────────────┐      ┌────────────┐      ┌────────────┐ │
 │  │ 캠페인 서비스 │      │ 주문 서비스 │      │ 창고 서비스 │      │ 트래킹 서비스│ │
 │  ├────────────┤      ├────────────┤      ├────────────┤      ├────────────┤ │
 │  │ Marketing  │      │ Order      │      │ Stock      │      │ Tracking   │ │
 │  │ Team       │      │ Team       │      │ Team       │      │ Team       │ │
 │  └────────────┘      └────────────┘      └────────────┘      └────────────┘ │
 │                                                                             │
 │  ※ 특징: 각 서비스는 자신의 비즈니스 목표(KPI)와 예산을 가짐                      │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처는 기업의 **전략적 목표**를 충실히 따른다. **판매(Sales)**라는 거대한 업무는 **주문 서비스**로 실체화되며, 이 서비스는 오직 '주문'과 관련된 모든 데이터와 로직을 소유한다. 중요한 점은 개발팀의 조직 구조도 이와 일치시킨다는 것이다. 즉, 주문 서비스에 문제가 생기면 '주문 팀'에 연락하면 된다. 이는 소통 비용을 낮추고, 비즈니스 부서의 요구사항이 기술팀에 전달되는 경로를 최단거리로 만든다.

---

### 심층 동작 원리: 안정적 서비스 경계 (Stability of Boundaries)

비즈니스 능력 중심 분해가 갖는 가장 큰 장점이다.
1. **정의**: 기술(언어, 프레임워크)은 1~2년마다 변하지만, 기업의 핵심 비즈니스 능력(예: 은행의 이자 계산)은 수십 년간 변하지 않는다.
2. **효과**: 비즈니스 능력을 기준으로 서비스를 쪼개 놓으면, 기술 트렌드가 바뀌어도 서비스의 '경계'와 '역할'은 그대로 유지된다.
3. **가치**: 아키텍처의 수명을 비약적으로 연장하며, 플랫폼 이전(Migration) 시에도 로직의 혼란을 방지한다.

- **📢 섹션 요약 비유**: 집의 벽지(기술)는 자주 바꾸더라도, 거실과 주방(비즈니스 능력)의 위치를 함부로 바꾸지 않는 것과 같은 아키텍처적 지혜입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비즈니스 능력별 분해 vs 하위 도메인별 분해 (DDD)

| 비교 항목 | 비즈니스 능력별 (Capability) | 하위 도메인별 (Subdomain/DDD) |
|:---|:---|:---|
| **설계 시점** | 프로젝트 초기, 거시적 관점 | 개발 심화 단계, 미시적 관점 |
| **분류 기준** | **현실의 조직/부서 기능** | **언어의 맥락과 데이터 관계** |
| **장점** | 비즈니스 가치와 정렬 용이 | 데이터 불일치 및 중복 방지 탁월 |
| **적합한 상황** | 신규 서비스 기획, 조직 개편 병행 시 | 복잡한 레거시 현대화, 정밀한 설계 필요 시 |

---

### 기술 융합: 콘웨이의 법칙과 역콘웨이 전략

조직 구조가 아키텍처를 결정한다는 법칙을 역이용한다.
- **Reverse Conway Maneuver**: 원하는 마이크로서비스 아키텍처 모양대로 먼저 팀을 쪼갠다. (예: 결제 서비스를 만들고 싶으면 결제 팀을 먼저 독립시킨다.)
- **가치**: 조직의 소통 구조를 강제로 재편함으로써 자연스럽게 비즈니스 능력 중심의 분해가 일어나도록 유도한다.

- **📢 섹션 요약 비유**: 물길을 내고 싶으면 땅을 먼저 파는 것(조직 개편)이 아니라, 물(서비스)이 흐르는 모양대로 땅이 저절로 깎이게 만드는 자연스러운 융합 전략입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 은행의 '계정계' 시스템 MSA 전환 분해

- **문제 상황**: 수십 년 된 메인프레임 시스템을 클라우드로 옮겨야 함. 기술적 연관성만으로 나누기엔 업무가 너무 방대함.
- **의사결정 (비즈니스 능력 기반 분해)**:
  1. **능력 식별**: 은행의 3대 핵심 능력인 '수신(입금)', '여신(대출)', '외환'을 최상위 서비스 경계로 확정한다.
  2. **팀 재편**: Java 개발자 그룹 대신, '여신 전문 개발팀'과 '여신 업무 전문가(현업)'를 하나의 애자일 스쿼드로 묶는다.
  3. **데이터 소유권**: '여신' 서비스가 사용하는 데이터는 오직 여신 팀만 관리하게 하고, 다른 팀은 API로만 정보를 얻게 한다.
  4. **결과**: 금융 법규나 정책이 '대출'에 대해서만 바뀔 때, 다른 시스템 영향 없이 대출 서비스만 신속하게 업데이트하고 배포한다.

### 도입 체크리스트
- **Organization Alignment**: 현재의 우리 조직도가 비즈니스 능력을 제대로 반영하고 있는가?
- **Business Language**: 서비스의 이름과 API 명칭이 현업 부서에서도 그대로 쓰이는 단어인가?
- **Autonomy**: 각 부서(서비스)가 독자적인 배포 주기와 기술 결정을 내릴 수 있는 환경인가?

### 안티패턴
- **Technical Silo**: 'UI 서비스', 'DB 서비스'와 같이 기술 계층으로 서비스를 나누는 행위. (비즈니스 변화 대응 불가)
- **Shared Master Data**: 고객 정보를 모든 서비스가 한 테이블에서 공유하는 구조. (독립적 분해의 최대 장애물)

- **📢 섹션 요약 비유**: 오케스트라에서 '현악기 팀', '관악기 팀'으로 나누는 것(기술적 분리)도 좋지만, 영화 음악 연주회라면 '액션 장면 담당', '로맨스 장면 담당'(비즈니스 가치 분리)으로 나누는 것이 더 빠르고 정확할 수 있습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Agility)** | 요구사항 반영 시간 50% 단축 | 부서-개발팀 간 직접 소통으로 의사결정 단계 축소 |
| **정량적 (Cost)** | IT 투자 효율성 증대 | 비즈니스 가치가 높은 특정 부문에 집중적으로 자원 투자 가능 |
| **정성적 (Standard)** | 비즈니스-IT 전략 일치 (Alignment) | IT 시스템이 경영 전략의 방해 요소가 아닌 가속기가 됨 |
| **정성적 (Owner)** | 책임감 있는 운영 문화 | 서비스의 장애가 곧 우리 팀의 비즈니스 실패라는 명확한 책임 의식 |

### 미래 전망
- **Business-to-Code Automation**: 비즈니스 프로세스 모델(BPMN)을 그리면 AI가 자동으로 마이크로서비스 경계를 설계하고 보일러플레이트 코드를 생성하는 기술이 성숙될 것이다.
- **Liquid Organization**: 시장 상황에 따라 비즈니스 능력이 수시로 재정의되고, 이에 맞춰 서비스 아키텍처가 유동적으로 변하는 '액체형 아키텍처'가 실현될 것이다.

### 참고 표준
- **Conway's Law (Melvin Conway)**: 조직과 시스템 구조의 상관관계
- **VALUE CHAIN (Michael Porter)**: 비즈니스 능력 분석의 고전적 틀
- **Microservices Patterns (Chris Richardson)**: 분해 패턴의 이론적 기반

- **📢 섹션 요약 비유**: 기술은 도구일 뿐, 목적은 비즈니스입니다. 비즈니스 능력을 따라가는 아키텍처는 결코 길을 잃지 않으며, 기업을 가장 강력한 경쟁 우위로 이끕니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 우리 집 청소 역할을 나눌 때 **"방 청소 팀", "설거지 팀", "빨래 팀"으로 딱딱 자기 일만 정해서 나누는 것**과 같아요.
2. 방 청소를 잘하고 싶을 때 빨래 팀 친구에게 물어볼 필요 없이, **우리 팀 친구들끼리 상의해서 바로 청소하면 되니까 정말 빨라요.**
3. 이렇게 하는 일(비즈니스) 중심으로 팀을 나누면, 우리 집이 언제나 깨끗하고 행복해진답니다!
