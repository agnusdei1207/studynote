# 569. 분산 추적 (Distributed Tracing) - 트랜잭션 경로 추적

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 추적 (Distributed Tracing)은 마이크로서비스(MSA) 환경에서 하나의 사용자 요청이 여러 서비스를 거쳐 처리되는 **전 과정을 시각화하고 각 구간의 성능 지연을 분석**하는 옵저버빌리티 기술이다.
> 2. **가치**: 요청마다 고유한 **Trace ID**를 부여하고 서비스 간 통신 단위인 **Span**을 기록함으로써, 복잡한 호출 체인 속에서 장애 발생 지점과 성능 병목을 단번에 식별하여 평균 복구 시간(MTTR)을 획기적으로 단축한다.
> 3. **융합**: Zipkin, Jaeger와 같은 오픈소스 도구와 **OpenTelemetry (OTEL)** 표준을 통해 구현되며, 서비스 메시(Service Mesh) 및 API 게이트웨이와 결합되어 시스템 전 구간의 투명한 가시성을 제공한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 분산 추적은 "요청의 일생"을 기록하는 것이다. 사용자가 클릭한 한 번의 요청이 주문 서버, 결제 서버, 재고 서버를 거칠 때, 각 서버가 그 요청을 처리하는 데 얼마나 시간을 썼는지 꼬리에 꼬리를 물고 추적한다.
- **💡 비유**: 분산 추적은 **"택배 배송 추적 시스템"**과 같습니다. 물건(요청)이 발송되어 허브 터미널(주문 서버), 지점(결제 서버), 배송 기사(재고 서버)를 거치는 단계마다 바코드를 찍어(Span 기록), 지금 어디에 있는지와 어디서 배송이 늦어지는지 실시간으로 확인하는 것과 같습니다.
- **등장 배경**:
    1. **MSA의 디버깅 한계**: 서비스가 쪼개지면서 "주문이 실패했다"는 결과는 알지만, 그 원인이 주문 서버 자체인지, 아니면 호출한 결제 서버의 네트워크 지연인지 파악하기가 매우 어려워졌다.
    2. **성능 튜닝의 필요성**: 수십 개의 마이크로서비스가 얽힌 환경에서 전체 응답 속도를 늦추는 단 하나의 '느린 구간'을 찾아내야 했다.
    3. **분산 로그의 파편화**: 서버마다 로그가 따로 저장되어 있어, 하나의 요청에 대한 전체 맥락(Context)을 연결하는 데 엄청난 시간이 소요되었다.

- **📢 섹션 요약 비유**: 흩어진 실타래(파편화된 로그)를 하나의 긴 실(Trace ID)로 꿰어, 요청이 지나간 모든 자리를 선명하게 드러내는 디지털 수사 기술입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

분산 추적은 Trace ID를 통한 문맥 전파(Context Propagation)와 Span 단위의 시간 기록이 핵심이다.

| 용어 명칭 | 상세 정의 | 특징 | 비유 |
|:---:|:---|:---|:---|
| **Trace (트레이스)** | 하나의 요청이 완료될 때까지의 전체 경로 | 여러 개의 Span으로 구성됨 | 택배의 전체 배송 경로 |
| **Span (스팬)** | 특정 서비스 내부의 작업 단위 (단일 API 호출 등) | 시작/종료 시간, 상태 정보 포함 | 택배가 특정 터미널에 머문 시간 |
| **Trace ID** | 전체 트랜잭션을 식별하는 고유 번호 | 서비스 간 통신 시 헤더에 포함됨 | 운송장 번호 |
| **Span ID** | 개별 작업 단위를 식별하는 번호 | 부모 Span ID를 참조하여 계층 구조 형성 | 터미널 작업 고유 번호 |
| **Baggage** | 서비스 간에 전파하고 싶은 커스텀 데이터 | 사용자 ID, 권한 등 메타데이터 | 택배 상자 안의 전용 태그 |

#### [분산 추적 시스템 동작 아키텍처]

요청이 서비스 간에 흐르면서 ID가 전파되고 수집되는 과정을 보여준다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Distributed Tracing Workflow (Collector Model)         │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Client] ──▶ [Service A (Order)] ──▶ [Service B (Pay)] ──▶ [Service C (DB)]   │
 │      │              │                     │                     │           │
 │  1. 요청 발생 ──▶ 2. Trace ID 생성       4. Trace ID 전달 ──▶ 6. Trace ID 전달 │
 │                    3. Span 1 시작/종료     5. Span 2 시작/종료     7. Span 3...  │
 │                           │                     │                     │     │
 │                           └───────────┬─────────┴──────────┬──────────┘     │
 │                                       ▼                    ▼                │
 │                             ┌────────────────────────────────┐              │
 │                             │    Tracing Collector (Jaeger)  │              │
 │                             │  - 데이터 취합 및 타임라인 생성   │              │
 │                             └────────────────────────────────┘              │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 시각화 결과 (Timeline):                                               │  │
 │  │ [Span 1 (Order)] ■■■■■■■■■■■■■■■■■■■■■■ (100ms)                         │  │
 │  │   [Span 2 (Pay)]       ■■■■■■■■■■■ (50ms)                               │  │
 │  │     [Span 3 (DB)]         ■■■■ (10ms)                                    │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **Context Propagation**: 서비스 A가 B를 호출할 때 HTTP 헤더(예: `X-B3-TraceId`)에 Trace ID를 실어 보낸다. 이를 통해 물리적으로 떨어진 서버들이 하나의 작업을 처리하고 있음을 인지한다.
2. **Span Reporting**: 각 서비스는 자신이 처리한 작업의 시작/종료 시간과 결과를 Span 객체로 만들어 비동기적으로 **Collector**에게 보고한다.
3. **Trace Visualization**: Collector는 동일한 Trace ID를 가진 Span들을 시간 순서와 부모-자식 관계에 맞춰 정렬하여 간트 차트(Gantt Chart) 형태로 시각화한다.

#### 심층 동작 원리: 샘플링 (Sampling) 전략
모든 요청을 추적하면 시스템에 부하를 줄 수 있으므로 영리한 선택이 필요하다.
- **Probabilistic Sampling**: 1,000건 중 1건만 추적하도록 확률적으로 설정 (보통 0.1% ~ 1%).
- **Rate Limiting Sampling**: 초당 최대 n건만 추적.
- **Adaptive Sampling**: 시스템 부하가 낮을 땐 많이, 높을 땐 적게 자동으로 조절.

- **📢 섹션 요약 비유**: 모든 택배 상자를 엑스레이로 검사하는 게 아니라, 무작위로 몇 개만 골라 정밀 검사함으로써 물류 속도를 유지하는 것과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 주요 분산 추적 도구 비교: Zipkin vs Jaeger

| 비교 항목 | Zipkin (Twitter 개발) | Jaeger (Uber 개발) |
|:---:|:---|:---|
| **아키텍처** | 단순함, Java 기반 | **복잡하고 강력함, Go 기반** |
| **데이터 모델** | B3 Propagation 표준 | **OpenTelemetry 기반** |
| **주요 특징** | 검증된 안정성, 쉬운 설치 | **클라우드 네이티브 최적화**, CNCF 졸업 |
| **UI/UX** | 클래식하고 직관적 | **상세한 통계 및 적응형 샘플링 제공** |
| **비유** | 튼튼한 트럭 | 최첨단 자율주행 트럭 |

#### 2. 기술 융합: 서비스 메시(Istio)와의 결합
- **Auto-Tracing**: 개발자가 코드에 추적 로직을 넣지 않아도, 서비스 메시의 **Sidecar Proxy(Envoy)**가 자동으로 헤더를 읽고 Span을 생성하여 보고한다.
- **가치**: 비즈니스 로직과 관측성 로직을 완전히 분리하여 개발 생산성을 높인다.

- **📢 섹션 요약 비유**: 각자 기록하던 항해 일지를, 이제는 배(Service) 옆에 붙은 무인 드론(Sidecar)이 알아서 촬영하고 보고하는 것과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 간헐적으로 발생하는 '결제 지연' 원인 규명**
    - **문제**: 특정 고객들이 결제 버튼을 누르면 5초 이상 멈춰있다가 성공하는 현상 발생. 로그에는 에러가 없음.
    - **의사결정**: 분산 추적 툴에서 해당 Trace ID를 조회.
    - **결과**: 결제 서비스가 외부 카드사 API를 호출할 때 네트워크 타임아웃 대기 시간이 5초임을 발견. 해당 구간에 서킷 브레이커와 타임아웃 축소 정책 적용.

- **도입 체크리스트**:
    1. **Context Propagation**: 비동기 쓰레드나 메시지 큐(Kafka)를 거칠 때도 Trace ID가 유실되지 않고 전달되는가?
    2. **Storage Management**: 매일 테라바이트 단위로 쌓이는 추적 데이터를 어떻게 효율적으로 저장할 것인가? (Elasticsearch Sharding 등)
    3. **Standardization**: 도구에 종속되지 않기 위해 **OpenTelemetry** 표준 SDK를 사용하고 있는가?

- **안티패턴**: 
    - 전수 추적(100% Sampling) 설정으로 운영 서버의 CPU를 고갈시키는 행위.
    - Trace ID만 있고 관련 로그(Log)가 연결되지 않아 "어디서 늦은지는 알겠는데 왜 늦었는지는 모르는" 반쪽짜리 관측성.

- **📢 섹션 요약 비유**: 지도는 있는데 내비게이션 기능(로그 연동)이 없는 상황을 경계해야 합니다. 데이터 사이의 '맥락'을 잇는 것이 실력입니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량적 기대효과**:
    - **MTTR (평균 복구 시간)**: 장애 지점 파악 시간 90% 이상 단축.
    - **성능 최적화**: 가장 느린 서비스 구간(Bottleneck)을 수치로 파악하여 튜닝 우선순위 결정 가능.
- **미래 전망**: **OpenTelemetry**로의 수집 표준 단일화가 가속화되고, 추적 데이터를 기반으로 장애 전파 경로를 예측하는 **Graph AI 기반 장애 예방 기술**이 도입될 것이다.
- **참고 표준**: **W3C Trace Context**, **OpenTelemetry Specification**.

- **📢 섹션 요약 비유**: 어두운 밤하늘(복잡한 MSA)에 별자리(분산 추적)를 그려 넣음으로써, 우리가 어디에 있고 어디로 가야 할지 길을 잃지 않게 해주는 시스템의 북극성과 같습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 분산 추적은 친구들과 릴레이 달리기를 할 때, **누가 바톤을 전달하는 데 얼마나 시간을 썼는지 꼼꼼하게 기록하는 시계**와 같아요.
2. 바톤(Trace ID)에 이름표를 붙여서 넘겨주면, 나중에 달리기 경주가 끝나고 **누가 가장 늦게 뛰었는지 금방 알 수 있답니다.**
3. 이 시계를 보면, 우리가 왜 졌는지(장애 발생) 금방 찾아내서 다음에는 더 빨리 뛸 수 있게 도와준답니다!
