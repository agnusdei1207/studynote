# 549. 2PC (2-Phase Commit) 한계와 장애 시나리오

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 2단계 커밋 (2-Phase Commit, 2PC)은 분산 데이터베이스 환경에서 원자성(Atomicity)을 보장하기 위해 중앙 통제자가 모든 참여자의 준비 상태를 확인한 후 일괄 커밋하는 프로토콜이나, 구조적 특성상 **동기적 블로킹(Blocking)**과 **단일 장애점(SPOF)** 문제를 내포한다.
> 2. **가치**: 데이터의 강력한 일관성(Strong Consistency)이 필수적인 금융 거래 등에는 유효하나, 네트워크 지연이나 노드 장애 시 전체 시스템의 가용성을 급격히 저하시켜 현대 마이크로서비스(MSA) 환경에서는 사용이 지양된다.
> 3. **융합**: 2PC의 한계를 극복하기 위해 비차단 프로토콜인 3PC (3-Phase Commit)나, 성능과 가용성을 우선시하는 **Saga 패턴 (최종 일관성)** 기술과 융합 및 비교되어 최적의 분산 트랜잭션 전략 수립의 근거가 된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 2PC는 분산된 여러 노드에 걸친 데이터를 수정할 때, "모두 성공하거나 모두 실패"하게 만드는 가장 고전적인 방법이다. **Prepare(준비)**와 **Commit(실행)**의 두 단계로 이루어진다.

- **💡 비유**: 2PC는 **"단체 사진 촬영"**과 같습니다. 사진작가(코디네이터)가 "모두 준비됐나요?"라고 묻고(Prepare), 눈을 감은 사람(장애 노드)이 한 명이라도 있으면 찍지 않습니다. 모든 사람이 "네!"라고 대답해야만 셔터(Commit)를 누릅니다. 만약 한 명이라도 대답이 없으면 작가는 무한정 기다리게 되고, 모델들은 계속 포즈(Lock)를 잡고 있어야 합니다.

- **등장 배경**:
  1. **데이터 분산 저장**: 시스템 규모가 커지면서 데이터를 여러 DB 서버에 나눠 담게 되었고, 이들 간의 데이터 일관성을 맞출 기법이 필요해졌다.
  2. **ACID 원칙 준수 요구**: 분산 환경에서도 단일 DB처럼 완벽한 원자성을 보장하고자 하는 공학적 시도였다.
  3. **표준화된 인터페이스**: X/Open XA 표준 등과 결합하여 서로 다른 제조사의 DB 간에도 트랜잭션을 묶을 수 있게 되었다.

- **📢 섹션 요약 비유**: 완벽함을 위해 모두를 멈춰 세우는 '독재적 합의 방식'으로, 질서는 완벽하지만 속도와 유연성이 떨어지는 태생적 한계를 지니고 있습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 2PC의 단계별 프로세스 및 잠금(Lock) 메커니즘

| 단계 (Phase) | 수행 주체 | 상세 동작 | 자원 상태 |
|:---|:---:|:---|:---|
| **1. Prepare (준비)** | Coordinator | 참여자들에게 "커밋 가능?" 질의 전송 | **Lock 점유 시작** |
| **2. Vote (응답)** | Participants | 자신의 로컬 작업 성공 여부 응답 (Yes/No) | 대기 (Blocking) |
| **3. Commit (실행)** | Coordinator | 모두 Yes면 Commit, 하나라도 No면 Rollback | **Lock 해제** |

---

### 2. 2PC 장애 시나리오 및 한계 아키텍처

네트워크 장애나 노드 다운이 시스템 전체에 미치는 악영향을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      2PC의 결정적 한계: 동기적 블로킹 및 장애 전파                  │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Coordinator (통제자)]                               [Participants (참여자)]  │
 │           │                                          ┌────────────┐         │
 │           │ ────(1. Prepare)──────────────────▶      │   DB 1     │         │
 │           │                                          └──────┬─────┘         │
 │           │                                                 │ [Lock 걸림]    │
 │           │ ◀───(2. Yes 응답)─────────────────      ┌──────▼─────┐         │
 │           │                                          │   DB 2     │         │
 │           X [통제자 갑자기 다운!]                      └──────┬─────┘         │
 │           │                                                 │ [Lock 걸림]    │
 │           │                                          ┌──────▼─────┐         │
 │           │ [장애 상황]                               │   DB 3     │         │
 │           │ - DB들은 통제자의 다음 명령을 기다리며      │ (네트워크 단절) │         │
 │           │   무한 루프/블로킹 상태에 빠짐              └────────────┘         │
 │           │ - 다른 사용자들은 해당 데이터를 수정 못 함                            │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 2PC의 3대 한계:                                                       │  │
 │  │ 1. Blocking: 참여자들은 명령이 올 때까지 자원을 점유한 채 멈춤 (성능 저하)      │  │
 │  │ 2. SPOF: 코디네이터가 죽으면 전체 시스템의 트랜잭션이 중단됨                 │  │
 │  │ 3. Scalability: 참여 노드가 많아질수록 지연 시간(Latency)이 기하급수적 증가    │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 2PC의 가장 치명적인 문제는 **동기적 대기**다. 모든 참여 노드는 코디네이터의 최종 결정을 기다리며 해당 데이터에 **배타적 잠금 (Exclusive Lock)**을 건다. 만약 코디네이터가 최종 단계에서 죽거나, 특정 노드와의 네트워크가 끊기면 모든 DB는 락을 푼 채 영원히 대기하게 된다. 이 동안 다른 정상적인 요청들도 해당 데이터에 접근하지 못해 시스템 전체가 서서히 마비되는 **연쇄 지연** 현상이 발생한다.

---

### 심층 동작 원리: 불확실성 구간 (In-doubt Period)

장애 복구가 가장 어려운 마의 구간이다.
1. **정의**: 참여자가 'Yes'라고 투표한 후, 코디네이터로부터 'Commit/Rollback' 명령을 받기 전까지의 시간.
2. **문제**: 이 구간에서 코디네이터가 죽으면, 참여자는 독단적으로 커밋하거나 롤백할 수 없다. (잘못 선택하면 데이터 정합성이 깨지기 때문)
3. **가치**: 2PC가 "일관성은 지키지만 가용성은 포기한다"는 사실을 극명하게 보여주는 기술적 지점이다.

- **📢 섹션 요약 비유**: 결혼식장에서 신랑, 신부 모두 "네"라고 대답했는데, 주례 선생님이 갑자기 쓰러지신(SPOF) 상황입니다. 하객(자원)들은 예식이 끝난 건지 취소된 건지 몰라 아무도 자리를 뜨지 못하고 갇혀버리는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 2PC vs 3PC vs Saga 패턴 비교

| 비교 항목 | 2-Phase Commit (2PC) | 3-Phase Commit (3PC) | Saga 패턴 |
|:---|:---|:---|:---|
| **일관성 수준** | 강력한 일관성 (Strong) | 강력한 일관성 (Strong) | **최종 일관성 (Eventual)** |
| **작동 방식** | 동기, 블로킹 | 동기, 비블로킹 지향 | **비동기, 이벤트 기반** |
| **주요 단점** | SPOF, 성능 저하 | 통신 오버헤드 매우 큼 | 구현 복잡, 데이터 일시 불일치 |
| **가용성** | 낮음 | 중간 | **매우 높음** |
| **비유** | 만장일치제 | 중간 확인 절차 추가 | 일단 실행 후 나중에 취소 |

---

### 기술 융합: 클라우드 분산 DB와 2PC의 진화

최근 클라우드 기술은 2PC의 단점을 보완한다.
- **Paxos / Raft 알고리즘**: 중앙 통제자 없이 노드들끼리 과반수 합의(Quorum)를 통해 결정을 내린다.
- **가치**: 2PC의 SPOF 문제를 해결하여, 통제자가 죽어도 시스템이 멈추지 않는 고가용성 분산 합의 체계를 구축한다. (예: Spanner, CockroachDB)

- **📢 섹션 요약 비유**: 한 명의 선생님 말만 듣는 교실에서, 친구들끼리 다수결로 결정하는 민주적인 학생 자치회로 발전한 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 2PC 도입 여부 결정 가이드라인

- **문제 상황**: 마이크로서비스로 쪼개진 환경에서 '계좌 이체' 기능을 만들 때, 2PC를 써야 할까 말아야 할까?
- **의사결정 (아키텍처적 판단)**:
  1. **데이터 중요도**: 1원의 오차도 허용되지 않는 핵심 금융 원장인가? → **Yes라면 2PC(XA) 검토**. (단, 성능 저하 감수)
  2. **성능 요구사항**: 초당 수만 건의 트래픽을 견뎌야 하는가? → **Yes라면 2PC 절대 금지**. (Saga 패턴 채택)
  3. **장애 영향도**: 특정 서버 장애가 전체 시스템 마비로 이어져도 되는가? → **No라면 2PC 배제**.
  4. **결론**: 현대 웹 서비스의 99%는 2PC의 블로킹 리스크를 감당할 수 없다. 따라서 **Transactional Outbox**와 **Saga 패턴**을 융합하여 '성능'과 '정합성'의 균형을 맞추는 것이 정석이다.

### 도입 체크리스트
- **Lock Duration**: 트랜잭션이 한 번 돌 때 락을 점유하는 시간이 100ms 이내인가? (길면 시스템 전체 지연)
- **Coordinator HA**: 코디네이터가 죽었을 때 즉시 업무를 이어받을 대기 서버(Standby)가 준비되었는가?
- **Network Stability**: 분산 노드 간의 네트워크가 전용선급으로 안정적인가?

### 안티패턴
- **마이크로서비스 간 광범위한 2PC**: 모든 서비스를 2PC로 엮는 행위. 결국 거대한 모놀리식을 네트워크로만 떼어놓은 '분산 모놀리스'를 만드는 꼴이다.
- **긴 휴먼 프로세스를 2PC에 포함**: 사용자의 입력을 기다리는 시간을 트랜잭션에 포함하는 것. (시스템 전체 잠금 발생)

- **📢 섹션 요약 비유**: "모두가 예라고 할 때까지 기다릴게"라는 고집은 아름답지만, 배고픈 수만 명의 손님(트래픽)을 기다리게 하는 식당 주인의 고집은 결국 폐업(시스템 다운)을 부릅니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Integrity)** | 데이터 불일치 0% 달성 | 물리적으로 분리된 저장소 간의 완벽한 데이터 동기화 보장 |
| **정량적 (Compliance)** | 법적 정합성 요건 충족 | 금융 및 세무 관련 규정에서 요구하는 엄격한 트랜잭션 증명 |
| **정성적 (Standard)** | 표준 분산 아키텍처 이해 | XA 표준 등을 통한 이기종 시스템 간의 논리적 결합성 확보 |
| **정성적 (Risk Management)** | 분산 시스템의 한계 인지 | 2PC의 한계를 통해 가용성(Availability)의 소중함을 깨닫는 계기 제공 |

### 미래 전망
- **NewSQL의 지배**: 2PC의 원자성과 NoSQL의 확장성을 결합한 **NewSQL (Google Spanner 등)**이 분산 트랜잭션의 기본 인프라가 될 것이다.
- **Deterministic Scheduling**: 트랜잭션 실행 순서를 미리 결정하여 합의 과정의 오버헤드를 제로에 가깝게 줄이는 결정론적 스케줄링 기술이 도입될 것이다.

### 참고 표준
- **X/Open XA Interface**: 분산 트랜잭션의 기술 표준
- **CAP Theorem**: 일관성과 가용성 사이의 선택 이론
- **ISO/IEC 9804**: OSI-Transaction Processing 표준

- **📢 섹션 요약 비유**: 2PC는 소프트웨어 공학이 '완벽함'을 향해 내디뎠던 위대한 첫발입니다. 비록 지금은 그 한계 때문에 자리를 내주고 있지만, 그 철학은 현대의 모든 정교한 데이터 기술 속에 살아 숨 쉬고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **2PC**는 친구 10명과 함께 놀이공원에 갈 때, **"모두 준비됐니?"라고 묻고 단 한 명이라도 늦으면 아무도 출발하지 못하게 하는 규칙**이에요.
2. 모두가 "응!"이라고 대답할 때까지 대문 앞에 서서 기다려야 해서 시간이 많이 걸린답니다.
3. 만약 대답을 하려던 친구가 화장실에 가버리면(장애), 나머지 친구들은 영문도 모른 채 대문 앞에 계속 서 있어야 하는 **조금 고집불통인 약속**이에요!
