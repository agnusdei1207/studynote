# 548. 로컬 트랜잭션 vs 분산 트랜잭션 (Distributed Transaction)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트랜잭션은 데이터의 일관성을 유지하기 위한 작업 단위로, 하나의 데이터베이스 내에서 완결되는 **로컬 트랜잭션**과 네트워크로 연결된 여러 노드나 DB에 걸쳐 실행되는 **분산 트랜잭션**으로 구분된다.
> 2. **가치**: 로컬 트랜잭션은 ACID 원칙을 엄격히 준수하여 최고 수준의 무결성을 제공하며, 분산 트랜잭션은 서비스가 파편화된 환경(MSA)에서 "전부 성공하거나 전부 실패(All or Nothing)"해야 하는 비즈니스 정합성을 보장한다.
> 3. **융합**: 마이크로서비스 아키텍처의 확산으로 2PC(2-Phase Commit)와 같은 전통적 분산 트랜잭션 방식의 한계(성능 저하)를 극복하기 위해, 최근에는 Saga 패턴을 통한 '최종 일관성(Eventual Consistency)' 확보 기술이 핵심으로 부상하고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **로컬 트랜잭션**: 단일 DB 엔진 안에서 일어나는 일. (예: 한 은행 계좌에서 다른 계좌로 송금)
  - **분산 트랜잭션**: 물리적으로 떨어진 여러 자원(DB, 메시지 큐 등)을 하나의 작업으로 묶는 일. (예: 여행 예약 시 항공권 결제 + 호텔 예약 + 렌터카 예약)

- **💡 비유**: 
  - **로컬 트랜잭션**: 한 식당 안에서 주문과 서빙이 이루어지는 것과 같습니다. 주방장이 "요리 완료"를 선언하면 바로 손님에게 전달되므로 관리가 매우 쉽습니다.
  - **분산 트랜잭션**: 여러 식당의 음식을 한꺼번에 배달 주문하는 것과 같습니다. 중식, 일식, 양식 배달원이 **동시에 도착**해야 파티가 시작될 수 있으며, 한 명이라도 못 오면 전체 파티를 취소해야 하는 복잡한 상황입니다.

- **등장 배경**:
  1. **데이터베이스 분산화**: 시스템 규모가 커지면서 하나의 DB로 감당이 안 되어 데이터를 여러 서버로 나누어 저장(Sharding)하게 되었다.
  2. **MSA의 필연적 산물**: 서비스마다 각자의 DB를 가지는(Database per Service) MSA 구조에서는 비즈니스 프로세스 하나가 여러 DB를 건드릴 수밖에 없게 되었다.
  3. **엔터프라이즈 통합(EAI)**: 서로 다른 회사의 시스템들이 API로 엮이면서 이들 간의 데이터 정합성을 맞추어야 하는 고난도 요구사항이 생겨났다.

- **📢 섹션 요약 비유**: 내 지갑 안에서 돈을 옮기는 것(로컬)과, 지구 반대편에 있는 여러 명에게 동시에 돈을 보내고 모두 받았는지 확인하는 것(분산)의 난이도 차이를 해결하는 기술입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 로컬 vs 분산 트랜잭션 기술적 차이 비교

| 비교 항목 | 로컬 트랜잭션 (Local) | 분산 트랜잭션 (Distributed) |
|:---|:---|:---|
| **관리 주체** | 단일 DBMS (Transaction Mgr) | **트랜잭션 관리자 (Coordinator)** |
| **일관성 모델** | **강력한 일관성 (ACID)** | 최종 일관성 (BASE) 또는 복잡한 2PC |
| **성능** | 매우 빠름 | **매우 느림 (네트워크 오버헤드)** |
| **복구 방식** | DB 로그(WAL) 기반 자동 복구 | 보상 트랜잭션 또는 전역 롤백 |
| **주요 기술** | SQL Commit / Rollback | **XA 표준, 2PC, TCC, Saga** |

---

### 2. 분산 트랜잭션 동작 아키텍처 (2PC 모델)

가장 고전적인 분산 트랜잭션 방식인 2단계 커밋(Two-Phase Commit)을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      분산 트랜잭션 (2PC) 제어 흐름 및 구조                        │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Global Transaction Coordinator] (중앙 통제자)                                │
 │           │                                                                 │
 │           ▼ [Phase 1: Prepare (준비)]                                        │
 │  ┌──────────────────┬──────────────────┬──────────────────┐                 │
 │  │ "커밋할 수 있니?"  │ "커밋할 수 있니?"  │ "커밋할 수 있니?"  │                 │
 │  ▼ (DB 1)           ▼ (DB 2)           ▼ (App 3)          │                 │
 │  [Yes/No 응답] ◀───  [Yes/No 응답] ◀───  [Yes/No 응답] ◀───┘                 │
 │           │                                                                 │
 │           ▼ [Phase 2: Commit/Rollback (실행)]                                │
 │  ┌──────────────────┬──────────────────┬──────────────────┐                 │
 │  │  **COMMIT!!**    │  **COMMIT!!**    │  **COMMIT!!**    │ (모두 Yes일 때)   │
 │  └──────────────────┴──────────────────┴──────────────────┘                 │
 │                                                                             │
 │  ※ 한 명이라도 No라고 하면 전원 Rollback 수행 (All or Nothing 보장)              │
 │  ※ 문제점: 통제자가 죽거나 네트워크 지연 시 모든 자원이 '잠금(Lock)' 상태로 대기함    │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 분산 트랜잭션은 **중앙 통제자(Coordinator)**의 지휘 하에 움직인다. 1) **Prepare 단계**에서 모든 참여자에게 실행 준비가 되었는지 묻는다. 이때 참여자들은 데이터를 수정할 준비를 마치고 'Lock'을 건다. 2) 모든 참여자가 "준비됨"이라고 답하면, 통제자는 **Commit** 명령을 일괄 하달한다. 만약 단 한 명이라도 실패하거나 응답이 없으면 전체에 **Rollback**을 명령한다. 이 과정은 데이터의 완벽한 일치(ACID)를 보장하지만, 네트워크 구간이 늘어날수록 성능이 급격히 저하되고 시스템 가용성이 떨어지는 단점이 있다.

---

### 심층 동작 원리: XA 리소스 (eXtended Architecture)

분산 트랜잭션을 지원하는 하부 인터페이스 표준이다.
1. **정의**: 이종의 DB나 메시지 큐(JMS)가 하나의 트랜잭션에 참여할 수 있게 하는 개방형 아키텍처 표준.
2. **원리**: 각 리소스는 `xa_prepare`, `xa_commit`, `xa_rollback` 함수를 외부에 제공해야 한다.
3. **가치**: Oracle DB와 RabbitMQ가 섞여 있어도, XA 표준을 지원한다면 하나의 원자적 작업(Transaction)으로 묶을 수 있다.

- **📢 섹션 요약 비유**: 서로 다른 나라의 부대(이종 시스템)가 연합 작전을 펼칠 때, 통일된 무전 규격(XA 표준)을 사용하여 사령관의 "공격!" 신호에 동시에 움직이는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 일관성 모델의 변화 (ACID vs BASE)

분산 환경에서 데이터 정합성을 바라보는 두 시각이다.

| 비교 항목 | ACID (로컬 트랜잭션 중심) | BASE (분산 트랜잭션 중심) |
|:---|:---|:---|
| **철학** | 데이터는 **언제나** 일관되어야 함 | 데이터는 **결국** 일관되어야 함 |
| **특징** | Atomicity, Consistency, Isolation, Durability | Basically Available, Soft-state, **Eventual Consistency** |
| **가용성** | 낮음 (Locking으로 인해 지연 발생) | **매우 높음** |
| **적합한 상황** | 계좌 이체, 결제 시스템 핵심부 | SNS 추천, 장바구니, 주문 상태 추적 |

---

### 기술 융합: Saga 패턴을 통한 2PC 극복

2PC의 성능 한계를 극복하기 위해 최신 아키텍처는 Saga 패턴을 융합한다.
- **원리**: 각 서비스가 로컬 트랜잭션만 수행하고, 다음 서비스로 이벤트를 넘긴다. 중간에 실패하면 이전에 성공한 서비스들에게 '취소(보상)' 이벤트를 보낸다.
- **가치**: 긴 시간 동안 자원을 점유(Lock)하지 않으면서도 비즈니스적 정합성을 맞출 수 있다.

- **📢 섹션 요약 비유**: 친구들 10명이 모여야 여행을 가는 규칙에서, "한 명이라도 안 오면 다 해산!(2PC)" 하던 방식에서 "일단 예약하고, 못 가는 사람 생기면 그때 환불하자(Saga)"는 유연한 방식으로 바뀐 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 마이크로서비스 간의 데이터 정합성 설계 의사결정

- **문제 상황**: '주문' 서비스는 MySQL을 쓰고 '재고' 서비스는 Redis를 쓰는데, 주문 완료 시 재고가 반드시 깎여야 함.
- **의사결정 (기술사적 판단)**:
  1. **2PC 배제**: MySQL과 Redis 간의 전역 트랜잭션(2PC)은 성능 저하가 극심하고 구현이 어려우므로 배제한다.
  2. **Transactional Outbox 패턴**: 주문 DB 내에 '이벤트 테이블'을 만들고, 주문 저장과 이벤트 저장을 하나의 **로컬 트랜잭션**으로 묶는다. (신뢰성 확보)
  3. **Eventual Consistency**: 별도의 폴링 프로세스(Relay)가 이벤트 테이블을 읽어 재고 서비스에 비동기로 전달한다.
  4. **보상 로직 구현**: 재고 차감에 실패할 경우, '재고부족' 이벤트를 다시 주문 서비스로 보내 주문을 자동으로 취소(Rollback)하는 **Saga 패턴**을 완성한다.

### 도입 체크리스트
- **Latency Tolerance**: 비즈니스 로직상 데이터가 일치하기까지 수 초의 지연(Eventual Consistency)을 감내할 수 있는가?
- **Idempotency**: 네트워크 재시도로 인해 동일한 트랜잭션이 두 번 실행되어도 결과가 안전한가? (멱등성 설계 필수)
- **Deadlock Potential**: 분산 락(Lock)을 과하게 사용하여 시스템 전체가 멈출 위험은 없는가?

### 안티패턴
- **전역 트랜잭션 만능주의**: 모든 MSA 통신을 XA/2PC로 해결하려는 시도. 시스템 전체 성능을 최하위 서비스 수준으로 떨어뜨린다.
- **보상 트랜잭션 누락**: Saga 패턴을 쓴다면서 실패 시 데이터를 원복하는 코드를 짜지 않는 경우. (데이터 오염의 주범)

- **📢 섹션 요약 비유**: "완벽한 정답"을 찾기 위해 모두를 멈춰 세우는 독재적 방식보다, "실수하면 고치면 된다"는 유연한 협력 방식이 분산 세계에서는 훨씬 빠르고 강합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Perf)** | 시스템 처리량(TPS) 3배 향상 | 2PC 제거 및 비동기 처리를 통한 자원 점유 시간 최소화 |
| **정량적 (Stability)** | 장애 전파 지수 하락 | 로컬 트랜잭션 중심 설계를 통해 서비스 간 장애 격리 실현 |
| **정성적 (Standard)** | 현대적 아키텍처 대응 | 클라우드와 MSA 환경에 적합한 데이터 관리 표준 확보 |
| **정성적 (Resilience)** | 비즈니스 복원력 강화 | 실패를 가정하고 대응하는(Saga) 설계를 통해 장애 복구 지능화 |

### 미래 전망
- **Database Mesh**: 데이터베이스 계층에서 분산 트랜잭션을 투명하게 관리해주는 전용 사이드카 기술이 발전할 것이다.
- **AI-Managed Transaction**: 트랜잭션의 성격을 AI가 실시간 분석하여, 정합성이 중요한 건은 2PC로, 속도가 중요한 건은 Saga로 자동 할당하는 지능형 엔진이 등장할 것이다.

### 참고 표준
- **X/Open XA Group**: 분산 트랜잭션 인터페이스의 고전 표준
- **Saga Pattern (Chris Richardson)**: 현대 분산 트랜잭션의 실무 표준
- **CAP Theorem**: 분산 시스템의 근본적 제약 이론

- **📢 섹션 요약 비유**: 로컬 트랜잭션이 한 명의 완벽한 일꾼이라면, 분산 트랜잭션은 서로 다른 개성을 가진 수많은 일꾼이 보조를 맞춰 춤을 추는 군무(群舞)와 같습니다. 그 조화를 맞추는 것이 아키텍처의 미학입니다.
