# 543-560. 분산 트랜잭션, 옵저버빌리리티 및 서비스 메시

> **핵심 인사이트**: 서비스 메시는 **비즈니스 로직과 인프라에서 처리** 애플리케이션 계층의 문제를 애플리케이션 계층에서 해결합니다. **애플리케이션 계층을 인프라 계층으로 이동**시킩니다(Infrastructure Layer Problem) 해결이 수 있습니다들 **서비스 메시**에서 해결할 수 있습니다들 팀들은 인프라를 직접 다루지 수 있다, 사이드카 프록시가 애플리케이션의 보안, 관찰 가능성, 트래픽 관리, 라우팅, 로드 밸런싱, Canary 배포 등을 애플리케이션 계층에서 수행할 수 있습니다

---

## Ⅱ. 서비스 메시 구성요소

### 1. Control plane (Istiod)
* **Envoy**: 고성능 C/C++ 프록시
* **Pilot**: Go 기반, 경량
* **Linkerd**: Rust 기반, 보안 중심

### 2. data plane (Prometheus)
* **Metrics**: 메트릭 수집
* **GRPC**: mTLS 기반 서비스 간 통신
* **MTLS**: 상호 TLS 인증 (서비스 간 보안)

### 3. policy (Mixin/Retry)
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: reviews-api
spec:
  hosts:
  - reviews
  http:
  - reviews-api:8080
  trafficPolicy:
    - loadBalancer:
      simple: ROUND_ROBIN
    connectionPool:
      simple:
        http1MaxRequests: 3
        http2MinRequests: 1
        interval: 30s
      retries: 3
      attempts: 3
---
```

### 3. Envoy 사이드카
**특징**: Envoy는 C/++ 프록시와 높은 성능과 다양한 기능을 제공합니다.
* **장점**: 확장성, 풍부한 기능, 다양한 L7 프로토콜 지원
* **한계**: 복잡한 설정, 다양한 기능 간 충돌 가능성

### 4. Linkerd 아키텍처
```

│                     Linkerd                                   │
│  ┌─────────────────────────────────────────────────────┐  │
```

## Ⅲ. 트래픽 관리 전략
### 1. 라우팅(Routing)
* **경로 기반**: URL 패턴으로 서비스 매핑
* **헤더 기반**: HTTP 헤더로 라우팅 결정
* **가중치 기반**: 트래픽 분산 고려
* **서브셋 기반**: 쿠버네티스 내 특정 서비스로 라우팅
### 2. 로드 밸런싱(Load Balancing)
* **라운드 로빈**: 최신 연결 수, 정기적 전송
* **가중 라운드 로빈**: 정기적 최신 연결로 교체
* **일시적 라운드 로빈**: 순환 시 세션 유지
* **IP 해시 기반**: 클라이언트 IP로 특정 서버 라우팅
### 3. 카나리 배포(Canary Deployment)
```
┌───────────────────────────────────────────────────────┐
│                    Production Traffic                      │
│  ┌───────────────┐                 ┌───────────────┐  │
│  │   Service A   │                 │   Service A   │  │
│  │   (v1 - 95%)   │ ──────────────▶│   (v2 - 5%)   │  │
│  └───────────────┘                 └───────────────┘  │
│         │                                       │          │
│         ▼                                       ▼          │
│    [Load Balancer]                         [Load Balancer]    │
│         │                                       │          │
│         ▼                                       ▼          │
│   [Metrics Server] ◄──── Monitoring ──────▶ [Prometheus] │
└───────────────────────────────────────────────────────┘
```
### 4. 블루-그린 배포(Blue-Green Deployment)
```
┌───────────────────────────────────────────────────────┐
│                    Load Balancer                         │
│  ┌───────────────────────┐   ┌───────────────────────┐   │
│  │      Blue (Active)       │   │      Green (Standby)    │   │
│  │   ┌───────────────┐   │   │   ┌───────────────┐   │   │
│  │   │   Instance 1  │   │   │   │   Instance 1  │   │   │
│  │   │   Instance 2  │   │   │   │   Instance 2  │   │   │
│  │   │   Instance 3  │   │   │   │   Instance 3  │   │   │
│  │   └───────────────┘   │   │   └───────────────┘   │   │
│  └───────────────────────┘   └───────────────────────┘   │
│                         │                       │                         │
│         100% Traffic ──────────────────────────────▶ 0% Traffic │
└───────────────────────────────────────────────────────┘
```
### 5. 섀도우 배포(Shadow Deployment)
```
┌───────────────────────────────────────────────────────┐
│                  Production Traffic                     │
│                       │                                   │
│                       ▼                                   │
│                 [Service A - v1]                              │
│                       │                                   │
│                       │                                   │
│                       ▼                                   │
│                 [Service A - v2]                              │
│                  (Mirrored Traffic)                  │
│                       │                                   │
│                       ▼                                   │
│                 [Test System]                               │
└───────────────────────────────────────────────────────┘
```
* **개념**: 실제 트래픽을 복제하여 신규 버전으로 전송
* **장점**: 프로덕션 환경에서 안전하게 테스트 가능
* **용도**: 성능 테스트, 데이터 파이프라인 검증, 로그 비교
* **구현**: Service Mesh의 Traffic Mirroring 기능 활용
```

## Ⅳ. 관찰 가능성(Observability)과 보안
### 1. 메트릭 수집(Metrics)
**주요 메트릭 유형**:
| **RED** | **Counter** | 증가율 | 예: 초당 요청 수, HTTP 4xx,5xx |
| **GREEN** | **Gauge** | 측정값 | 예: 현재 메모리 사용량, 활성 연결 수 |
| **BLUE** | **Histogram** | 분포 | 예: API 응답 시간 분포 |

### 2. 로그 수집(Logging)
* **중앙화 로그**: 모든 서비스 로그를 중앙 저장소로 수집
* **구조화 로그**: JSON 형식, 표준화된 형식 사용
* **상관관계 ID**: Trace ID를 통한 모든 로그를 연결
* **ELK Stack**: Elasticsearch + Logstash + Kibana + Fluentd
* **Grafana Loki**: Grafana가 지원하는 로그 저장
* **Fluent Bit**: 로그 전달을 위한 에이전트 에이전트 비동
 처리
### 3. 분산 추적(Distributed Tracing)
* **Trace**: 하나 요청의 전체 경로
* **Span**: 하나 서비스 내부의 작업 단위
* **Context Propagation**: Trace ID/Span ID 전파
```

## Ⅶ. 분산 트랜잭션(Saga)
### 1. 2PC(2-Phase Commit) - MSA 부적합
* **1단계**: Prepare (준비 상태)
* **2단계**: Commit (모든 참여자 커밋)
* **문제**: 동기화 오버헤드, 블로킹 위험
### 2. 사가 패턴 - 로컬 트랜잭션 체인
```
┌───────────────────────────────────────────────────────────────┐
│                    Saga Pattern                              │
│                                                         │
│   ┌───────────────┐      ┌───────────────┐      ┌───────────────┐
│   │   Service A   │      │   Service B   │      │   Service C   │
│   │               │      │               │      │               │
│   │  [Local Tx 1] ───────▶│  [Local Tx 2] ───────▶│  [Local Tx 3] │
│   │   Commit ◀──────┼   Commit ◄──────┼   Commit ◄──────┤
│   └───────────────┘      └───────────────┘      └───────────────┘
│                                                         │
│   ┌───────────────────────────────────────────────────────────┐
│   │              All Succeed or All Fail                     │
│   │  - 성공: 모든 로컬 트랜잭션이 커밋됨                    │
│   │  - 실패: 보상 트랜잭션으로 롤백                              │
│   └───────────────────────────────────────────────────────────┘
└───────────────────────────────────────────────────────────────┘
```
### 3. 오케스트레이션 사가(Orchestration Saga)
* **중앙 통제기(Orchestrator)**가 전체 흐름 관리
* **장점**: 명확한 흐름 제어, 실패 시 정확한 보상
* **단점**: 단일 실패 지점(SPOF)
```python
class OrderOrchestrator:
    def create_order(order_id):
        # 1. 예약 재고
        # 2. 결제 서비스 호출
        # 3. 배송 서비스 호출
        # 4. 성공/실패 확인
        # 5. 보상 트랜잭션 필요 시 실행
```
### 4. 코레오그래피 사가(Choreography Saga)
* **이벤트 기반 분산 조정**
* **장점**: 단일 실패 지점 제거, 단순한 구현
* **단점**: 전체 흐름 파악 어려움, 디버깅 복잡
```python
# 서비스 A
async def create_order(event):
    await publish_event("OrderCreated", event)

# 서비스 B (재고)
@subscribe("OrderCreated")
async def handle_order(event):
    try:
        await reserve_inventory(event.product_id)
        await publish_event("InventoryReserved", event)
    except:
        await publish_event("InventoryReservationFailed", event)

```
### 5. 보상 트랜잭션(Compensating Transaction)
* **목적**: 실패 시 이전 상태로 복원
* **원칙**: 멱등(Idempotent), 역연산 가능해야 함
* **구현**: 각 서비스에 보상 로직 포함
```

## Ⅷ. CQRS (명령 조회 책임 분리)
### 1. 개념
* **Command Model**: 쓰기 작업 전담
* **Query Model**: 읽기 작업 전담
* **장점**: 각 모델을 독립적으로 최적화 가능
```
┌───────────────────────────────────────────────────────────────┐
│                      CQRS Pattern                              │
│  ┌─────────────────────────────────────────────────────┐        │
│  │                  Command Side                        │        │
│  │  ┌─────────────────────────────────────────────────┐        │
│  │  │               Command Model                   │        │
│  │  │  - Create Order                         │        │
│  │  │  - Update User                          │        │
│  │  │  - Delete Product                       │        │
│  │  │                                         │        │
│  │  │           [Write Database]                  │        │
│  │  │           (정규화된 구조)                  │        │
│  │  └─────────────────────────────────────────────────┘        │
│  └─────────────────────────────────────────────────────┘        │
│                          │                                       │
│              [Event Bus / Message Queue]                              │
│                          │                                       │
│                          ▼                                       │
│  ┌─────────────────────────────────────────────────────┐        │
│  │                   Query Side                         │        │
│  │  ┌─────────────────────────────────────────────────┐        │
│  │  │                Query Model                    │        │
│  │  │  - Get Order by ID                        │        │
│  │  │  - Search Orders                          │        │
│  │  │  - Get User Orders                        │        │
│  │  │                                         │        │
│  │  │           [Read Database]                    │        │
│  │  │           (비정규화된 구조)                  │        │
│  │  └─────────────────────────────────────────────────┘        │
│  └─────────────────────────────────────────────────────┘        │
└───────────────────────────────────────────────────────────────┘
```
### 2. 장점
* **성능**: 쓰기/읽기 독립 최적화
* **확장성**: 읽기 DB를 수평 확장 가능
* **복잡성**: 각 모델 단순화
### 3. 주의사항
* **일관성**: 결과적 일관성(Eventual Consistency) 관리 필요
* **동기화**: 쓰기 DB → 읽기 DB 동기화 지연
* **복잡성**: 인프라 증가
```

## Ⅸ. 이벤트 소싱(Event Sourcing)
### 1. 개념
* **상태 저장**: 현재 상태만 저장 (CRUD)
* **이벤트 저장**: 상태 변경 이력을 이벤트로 저장 (Append-only)
* **장점**: 완전한 감사(Audit Trail) 제공, 상태 재현 가능
```
[CRUD 방식]                    [Event Sourcing]
┌───────────────────┐               ┌───────────────────┐
│   Current State   │               │   Event Store    │
│   ┌───────────┐   │               │   ┌───────────┐   │
│   │  id: 123  │   │               │   │  id: 123  │   │
│   │  name: A  │   │               │   │  type: Created│   │
│   │  qty: 10 │   │  ──────────▶│   │  payload: {...}│   │
│   │  price: 100│   │               │   │  timestamp: t │   │
│   └───────────┘   │               │   └───────────┘   │
│                   │               │                   │
│   [이전 상태 덮어씀] │               │   [이벤트 추가]    │
│                   │               │                   │
└───────────────────┘               └───────────────────┘
```
### 2. 이벤트 재생(State Replay)
```
[Event Store] → Event 1 → Event 2 → Event 3 → [Current State]
     (재생)       (재생)       (재생)       (재생)
```
* **장점**: 특정 시점으로 상태 복원 가능
* **용도**: 버그 조사, 감사(Audit), 데이터 복구

## Ⅺ. 서버리스(Serverless / FaaS)

* **BaaS(Backend as a Service)**: 백엔드 기능 서비스
### 2. 장점

### 3. 한계
* **콜드 스타트**: 초기 실행 지연
* **벤더 락인(Lock-in)**: 특정 클라우드 제공자 의존
* **디버깅**: 분산 환경 디버깅 어려움

```
## Ⅻ. 콜드 스타트(Cold Start)
### 1. 문제
서버리스 함수가 오랫 시간 미사용 후 첫 요청 시 **콜드 상태**로 인한 지연이 발생합니다.
* **원인**: 컨테이너/런타임 초기화 필요
* **영향**: 사용자 경험 저하, 타임아웃 증가
### 2. 해결 방안
* **Provisioned Concurrency**: 미리 초기화된 인스턴스 유지
* **Warm-up 함수**: 초기화 로직 분리
* **최소 인스턴스 유지**: 항상 실행 중인 인스턴스 유지
* **연결 풀 재사**: 컨테이너 연결 유지
```
[콜드 스타트 문제]
           ┌─────────────────────────────────────────┐
           │         Cold Instance                    │
           │    (콜드 상태 - 초기화 필요)         │
           └─────────────────────────────────────────┘
                           │
                           ▼ (지연)
           ┌─────────────────────────────────────────┐
           │         Warm Instance                    │
           │    (따뜻 상태 - 즉시 응답)           │
           └─────────────────────────────────────────┘
```
### 3. 언어별 지원
* **AWS Lambda**: Provisioned Concurrency
* **Azure Functions**: Premium Plan
* **GCP Cloud Functions**: Min Instances
```

4. **거버넌스**: 표준화된 데이터 정책 및 규칙

```
│              [Self-Service Data Platform]               │
│                  (중앙 인프라)                             │

```
### 4. 장점
* **확장성**: 각 도메인 독립적으로 확장

```
## ⅩⅣ. 컨테이너 오케스트레이션(Container Orchestration)
```

```
## ⅤⅥ. 오퍼레이터 패턴(Operator Pattern)
### 1. 개념
사용자 정의 컨트롤러를 통해 복잡한 애플리케이션을 자동으로 관리하는 패턴입니다.
### 2. 구성 요소
* **CRD(Custom Resource Definition)**: 사용자 정의 리소스 정의
* **Controller**: 리소스 상태 관리 루프
* **Reconciler**: 현재 상태와 목표 상태 비교
* **Watch**: 리소스 변경 감시
### 3. 작동 방식
```
┌───────────────────────────────────────────────────────────────┐
│                    Operator Workflow                        │
│                                                         │
│   ┌───────────────────────────────────────────────────────┐   │
│   │                   Watch Loop                         │   │
│   │  1. Watch for changes in CRD                  │   │
│   │  2. Detect resource creation/update/delete        │   │
│   └───────────────────────────────────────────────────────┘   │
│                           │                             │
│                           ▼                             │
│   ┌───────────────────────────────────────────────────────┐   │
│   │                 Reconcile Loop                      │   │
│   │  1. Compare current state with desired state        │   │
│   │  2. Calculate diff                              │   │
│   └───────────────────────────────────────────────────────┘   │
│                           │                             │
│                           ▼                             │
│   ┌───────────────────────────────────────────────────────┐   │
│   │                 Controller Action                   │   │
│   │  1. Create/Update/Delete resources                  │   │
│   │  2. Update status                                  │   │
│   └───────────────────────────────────────────────────────┘   │
└───────────────────────────────────────────────────────────────┘
```
### 4. 사용 사례
* **데이터베이스 Operator**: PostgreSQL, MySQL Operator
* **Prometheus Operator**: Prometheus Stack 관리
* **Cert-Manager Operator**: TLS 인증서 관리
```

## ⅤⅡ. 옵저버빌리티(Observability) 아키텍처
### 1. 3대 요소
* **Metrics**: 시계열 데이터 (CPU, 메모리, 요청 수)
* **Logs**: 이벤트 로그 (애플리케이션 로그, 시스템 로그)
* **Traces**: 분산 요청 추적 (마이크로서비스 간 호출 경로)

```

│  │  ┌───────────┐   │  │  ┌───────────┐   │  │  ┌───────────┐   │  │
│  │  │Prometheus │   │  │  │   ELK     │   │  │  │   Jaeger   │   │  │
│  │  │ Grafana   │   │  │  │  Fluentd  │   │  │  │   Zipkin   │   │  │
│  │  └───────────┘   │  │  └───────────┘   │  │  └───────────┘   │  │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘  │
└───────────────────────────────────────────────────────────────┘
```
### 3. 메트릭 수집(Prometheus)
* **Pull 모델**: Prometheus가 타겟에서 메트릭 수집
* **Push 모델**: 애플리케이션이 메트릭 푸시
* **Service Discovery**: 타겟 자동 발견
* **TSDB(Time Series Database)**: 시계열 데이터 저장
### 4. 로그 수집(ELK Stack)
* **Elasticsearch**: 검색 엔진
* **Logstash/Fluentd**: 로그 수집/변환
* **Kibana/Grafana**: 시각화
* **구조화 로깅**: JSON 형식 권장
### 5. 분산 추적(Jaeger/Zipkin)
* **Trace ID**: 전체 요청의 고유 ID
* **Span ID**: 개별 작업의 ID
* **Context Propagation**: 서비스 간 컨텍스트 전파

📢 **섹션 요약 비유**: 클라우드 네이티브 아키텍처는 **확장 가능한 설계**를 지향합니다. **MSA**는 서비스를 비즈니스 능력별 분해하고, **API 게이트웨이**에서 인증/라우팅/부하분산을 담당합니다. **Service Mesh**는 **인프라 계층**에서 보안/트래픽 관리/관찰 가능성을 담당합니다. **CQRS**는 읽기/쓰기를 분리하여 성능을 최적화합니다. **이벤트 소싱**은 상태를 이벤트로 저장하여 감사 추적을 제공합니다.
 **사가 패턴**은 분산 트랜잭션을 **오케스트레이션**과 **코레오그래피** 두 가지 방식으로 관리합니다. **Kubernetes**는 컨테이너 오케스트레이션의 핵심이며, **Helm**으로 패키지 관리, **Operator**로 복잡한 앱을 자동화합니다.
 **옵저버빌리티**는 메트릭/로그/분산 추적으로 시스템 상태를 파악합니다.
