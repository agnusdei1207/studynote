# 570. Trace ID와 Span ID의 전파 (Context Propagation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 문맥 전파 (Context Propagation)는 분산 시스템에서 하나의 논리적 요청이 여러 서비스 경계를 넘나들 때, 해당 요청의 메타데이터(Trace ID, Span ID 등)를 **통신 헤더(HTTP Header 등)에 실어 지속적으로 전달**하는 기술적 프로세스다.
> 2. **가치**: 파편화된 개별 서비스의 실행 기록들을 하나의 통합된 트랜잭션으로 엮어줌으로써 인과관계를 명확히 하고, 장애 발생 시 서비스 간 호출 관계를 추적하여 **분산 환경의 가시성(Visibility)**을 완성한다.
> 3. **융합**: W3C Trace Context 표준 규격을 기반으로 작동하며, 서비스 메시(Istio) 및 분산 추적 도구(Jaeger, Zipkin)와 결합되어 지능형 장애 진단 및 성능 모니터링의 핵심 기반 데이터를 제공한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 문맥 전파는 "요청의 꼬리표 달기"다. 마이크로서비스(MSA) 환경에서 요청이 A ──▶ B ──▶ C로 흐를 때, 각 서비스가 독립적으로 로그를 남기면 나중에 이를 하나로 합칠 방법이 없다. 이를 해결하기 위해 최초 진입점에서 고유한 ID를 생성하고, 모든 서비스 호출 시 이 ID를 전파(Propagation)하는 규칙을 적용한다.
- **💡 비유**: 문맥 전파는 **"릴레이 경주의 바톤 터치"**와 같습니다. 다음 주자에게 바톤(Trace ID)을 넘겨주지 않으면, 그 선수가 아무리 빨리 뛰어도(작업 수행) 같은 팀의 기록(전체 요청 흐름)으로 인정받지 못하는 것과 같습니다. 바톤이 끝까지 유지되어야만 팀 전체의 최종 기록을 잴 수 있습니다.
- **등장 배경**:
    1. **분산 로그의 한계**: 단순히 '주문 실패' 로그만으로는 그 주문이 어느 사용자가 보낸 것인지, 어떤 결제 요청과 연관된 것인지 알 수 없는 정보 단절 문제가 발생했다.
    2. **비동기 통신의 복잡성**: 메시지 큐(Kafka) 등을 통한 비동기 호출 시, 요청의 흐름이 끊기지 않도록 관리할 표준화된 전달 방식이 필요해졌다.
    3. **표준화 요구**: 구글, MS 등 기업마다 서로 다른 헤더 명칭(X-B3-TraceId 등)을 사용하여 시스템 통합이 어려웠고, 이를 해결하기 위해 W3C 표준이 정립되었다.

- **📢 섹션 요약 비유**: 수만 조각의 퍼즐(분산 로그)을 맞추기 위해, 각 조각 뒷면에 공통된 그림 번호(Trace ID)를 적어두어 누구나 쉽게 원본 그림(트랜잭션)을 완성하게 돕는 연결 기술입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

문맥 전파는 데이터 주입(Injection)과 데이터 추출(Extraction)의 반복으로 이루어진다.

| 구성 요소 | 역할 및 정의 | 주요 파라미터 | 비유 |
|:---:|:---|:---|:---|
| **Trace Context** | 전파되어야 할 메타데이터의 묶음 | Trace ID, Span ID, Flags | 바톤에 적힌 정보 |
| **Carrier (운반체)** | 정보를 실어 나르는 매개체 | HTTP Headers, Kafka Headers | 바톤을 쥐는 손 |
| **Injector (주입기)** | 통신 시작 전 헤더에 ID를 삽입 | `traceparent`, `tracestate` | 바톤에 기록하기 |
| **Extractor (추출기)** | 통신 수신 후 헤더에서 ID를 읽어옴 | `request.headers.get()` | 바톤 읽기 |
| **Baggage** | 전 구간 공유할 사용자 정의 데이터 | User-ID, Session-Type | 바톤 안의 쪽지 |

#### [문맥 전파(Context Propagation) 상세 흐름도]

서비스 경계를 넘을 때마다 ID가 어떻게 유지되고 변하는지 시각화한다.

```text
 [ Service A (Origin) ]           [ Service B (Intermediate) ]
 ┌───────────────────────┐        ┌────────────────────────────┐
 │ 1. New Trace ID: T1   │        │ 4. Extract T1, S1          │
 │ 2. Root Span ID: S1   │        │ 5. New Child Span ID: S2   │
 │ 3. Inject to Header   │        │ 6. Inject to Header        │
 └──────────┬────────────┘        └─────────────┬──────────────┘
            │                                   │
            ▼ [ HTTP Header ]                   ▼ [ gRPC / Messaging ]
   ┌──────────────────────────┐        ┌──────────────────────────┐
   │ traceparent:             │        │ traceparent:             │
   │ 00-T1-S1-01              │        │ 00-T1-S2-01              │
   └────────────┬─────────────┘        └────────────┬─────────────┘
                │                                   │
                └──────────▶ [ Service C ] ◀────────┘
                             7. Final Processing
                             - Trace ID: T1 (고정)
                             - Parent ID: S2
                             - Current ID: S3
```

**[다이어그램 해설]**
1. **Trace ID 고정**: 요청이 시작된 순간부터 끝날 때까지 **T1**이라는 ID는 변하지 않고 유지된다. 이것이 전체 트랜잭션을 묶는 '키'가 된다.
2. **Span ID 계층화**: 각 서비스로 넘어갈 때마다 새로운 **Span ID**가 생성된다. 이때 이전 서비스의 ID를 부모(Parent)로 참조함으로써 호출의 '순서'와 '깊이'를 재구성한다.
3. **W3C 표준 헤더**: `traceparent` 필드는 `버전-TraceID-ParentSpanID-플래그` 구조로 되어 있어, 이기종 시스템 간에도 완벽하게 호환된다.

#### 심층 동작 원리: 전파 과정의 2단계 (Inject & Extract)
- **Injection**: 클라이언트 라이브러리가 API 호출 직전에 현재 스레드의 관측성 컨텍스트를 읽어 HTTP 헤더에 텍스트 형태로 주입한다.
- **Extraction**: 서버 라이브러리가 요청을 받자마자 헤더를 읽어, 자신의 스레드 로컬(Thread Local) 메모리에 해당 컨텍스트를 복원한다.

- **📢 섹션 요약 비유**: 각 나라의 통관 절차(Service Entry)마다 여권(Trace ID)에 입출국 도장(Span ID)을 찍어, 나중에 여행자가 어디를 돌아다녔는지 완벽히 추적하는 출입국 관리 시스템과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 전파 방식 비교: B3 vs W3C Trace Context

| 비교 항목 | B3 Propagation (Zipkin 표준) | W3C Trace Context (신표준) |
|:---:|:---|:---|
| **헤더 명칭** | `X-B3-TraceId`, `X-B3-SpanId` | **`traceparent`**, `tracestate` |
| **구조** | 여러 개의 개별 헤더 사용 | **하나의 압축된 헤더 사용** |
| **호환성** | 구형 시스템 및 Java 진영 우세 | **최신 클라우드 및 글로벌 표준** |
| **비유** | 여러 장의 신분증 조각 | **통합 IC 카드 신분증** |

#### 2. 네트워크 지연과의 상관관계
- **오버헤드**: 헤더에 수십 바이트의 ID 정보를 추가하는 것은 네트워크 성능에 미치는 영향이 극히 미미하다 (0.1ms 이하).
- **시너지**: **Baggage**를 통해 인증 정보나 라우팅 힌트를 함께 전파하면, 매번 인증 서버를 호출하지 않아도 되어 오히려 전체 시스템 지연 시간을 단축할 수 있다.

- **📢 섹션 요약 비유**: 배달 음식을 시킬 때 영수증(Trace ID)을 붙여 보내는 것은 아주 작은 비용이지만, 그 영수증 덕분에 배달 사고(장애)를 즉시 해결할 수 있는 엄청난 보험이 됩니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 비동기 메시지 큐(Kafka) 환경에서의 문맥 단절 해결**
    - **문제**: HTTP 요청은 잘 추적되는데, Kafka에 메시지를 넣고 꺼내는 순간 Trace ID가 사라져 이후 흐름이 끊김.
    - **의사결정**: **Manual Propagation (수동 전파)** 수행. 메시지 페이로드나 Kafka Header에 Trace 정보를 명시적으로 담아 보냄.
    - **결과**: `Web ──▶ Kafka ──▶ Batch ──▶ DB`로 이어지는 전 구간 추적망 복구.

- **도입 체크리스트**:
    1. **Thread Local Propagation**: 자바 환경에서 별도의 쓰레드 풀을 사용할 때, 부모 쓰레드의 Trace ID가 자식 쓰레드로 복사(Transfer)되는 코드가 적용되었는가?
    2. **Log Correlation**: 로그 프레임워크(Logback 등) 설정에 `%mdc{trace_id}`가 포함되어 로그와 추적이 자동으로 맵핑되는가?
    3. **Standardization**: 모든 마이크로서비스가 동일한 전파 규격(예: W3C)을 사용하고 있는가?

- **안티패턴**: 
    - Baggage에 너무 큰 데이터를 담아 전파하는 행위 (네트워크 패킷 크기 초과 및 보안 위험).
    - 게이트웨이에서만 ID를 생성하고 내부 서비스 호출 시에는 전달을 생략하는 행위.

- **📢 섹션 요약 비유**: 끊어진 철길 위로는 기차가 달릴 수 없습니다. 비동기 환경이나 멀티 쓰레드 환경이라는 끊긴 지점을 '문맥 전파'라는 다리로 완벽하게 잇는 것이 엔지니어의 핵심 역량입니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량적 기대효과**:
    - **디버깅 효율**: 서비스 간 연관 에러 추적 성공률 100% 달성.
    - **가용성 증대**: 지연 구간의 명확한 식별로 성능 튜닝 대상 선정 시간 80% 단축.
- **미래 전망**: **Auto-Instrumentation** 기술의 발전으로 개발자가 코드를 전혀 수정하지 않아도 런타임 엔진이 알아서 모든 통신에 ID를 전파하는 '투명한 추적' 환경이 보편화될 것이다.
- **참고 표준**: **W3C Trace Context Recommendation**, **OpenTelemetry Propagation API**.

- **📢 섹션 요약 비유**: 과거에는 흩어진 발자국을 찾으러 다녔다면, 이제는 모든 요청에 지워지지 않는 형광 물질(Context)을 묻혀 어두운 네트워크 속에서도 그 경로를 선명하게 관찰하는 시대를 살고 있습니다.
