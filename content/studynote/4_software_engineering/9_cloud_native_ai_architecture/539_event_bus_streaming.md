# 539. 이벤트 버스 및 스트림 프로세싱 (Kafka Streams, Flink)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트 버스(Event Bus)는 분산된 시스템 간에 이벤트를 실시간으로 중계하는 **중앙 통신 인프라**이며, 스트림 프로세싱(Stream Processing)은 이 버스를 통해 흐르는 데이터를 저장하기 전 **연속적으로 분석하고 가공**하는 연산 패러다임이다.
> 2. **가치**: 데이터가 데이터베이스에 저장되기 전 "살아있는 상태"에서 즉각적인 비즈니스 가치(예: 사기 탐지, 실시간 추천)를 창출하며, 대규모 데이터를 병렬로 처리하여 지연 시간을 최소화하고 시스템의 실시간 대응력을 극대화한다.
> 3. **융합**: Apache Kafka와 Kafka Streams, Apache Flink 등이 대표적인 기술이며, 마이크로서비스(MSA)의 비동기 통신과 결합되어 '이벤트 소싱' 및 '상태 기반 컴퓨팅'의 핵심 아키텍처로 기능한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **이벤트 버스**: 컴퓨터 내부의 버스(Bus)처럼, 시스템 전체에 이벤트를 실어 나르는 거대한 신경망이다.
  - **스트림 프로세싱**: 멈춰있는 데이터(Data at Rest)가 아닌, 흐르는 데이터(Data in Motion)를 대상으로 한 연산이다.

- **💡 비유**: 이벤트 버스와 스트림 프로세싱은 **"컨베이어 벨트와 자동 선별기"**와 같습니다. 공장 라인(이벤트 버스)을 타고 수많은 부품(데이터)이 쏟아져 들어올 때, 이를 창고에 다 쌓아놓고 나중에 분류하는 게 아니라 벨트 위에서 즉시 불량품을 골라내거나(Filtering) 개수를 세는(Aggregation) 지능형 자동화 공정과 같습니다.

- **등장 배경**:
  1. **배치 처리의 한계**: 하루에 한 번 데이터를 모아 처리하는 방식으로는 실시간으로 변하는 고객의 요구와 시장 상황에 대응할 수 없게 되었다.
  2. **빅데이터의 폭증**: 초당 수백만 건의 로그와 센서 데이터를 DB에 먼저 넣으려다가는 DB가 먼저 뻗어버리는 문제가 발생했다.
  3. **실시간 의사결정 요구**: 카드 부정 사용 탐지나 주식 매매와 같이 '0.1초의 차이'가 비즈니스 성패를 가르는 영역이 확대되었다.

- **📢 섹션 요약 비유**: 낚시터에서 물고기를 다 잡아서 아이스박스에 담은 뒤(배치) 무게를 재는 게 아니라, 흐르는 강물(스트림)에 그물을 쳐놓고 지나가는 물고기의 크기를 실시간으로 측정하는 스마트 어업 방식입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 스트림 프로세싱의 핵심 연산 모델

데이터가 흐르는 동안 수행되는 논리적 작업들이다.

| 연산 유형 | 상세 내용 | 비유 |
|:---|:---|:---|
| **Filtering** | 특정 조건에 맞는 이벤트만 골라냄 | 불량품 솎아내기 |
| **Transformation** | 데이터의 형식을 바꾸거나 값 수정 | 부품 조립 및 도색 |
| **Windowing** | 시간 단위(1분, 1시간)로 데이터를 묶음 | 상자에 제품 담기 |
| **Aggregation** | 윈도우 내의 합계, 평균 등을 계산 | 상자 무게 재기 |
| **Join** | 서로 다른 두 스트림을 연결하여 분석 | 부품 A와 B 결합하기 |

---

### 2. 이벤트 스트림 프로세싱 통합 아키텍처

이벤트가 발생하여 분석되고 저장되기까지의 파이프라인을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      이벤트 스트림 프로세싱 (Kafka/Flink) 아키텍처                │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Event Source] ──────▶ [Event Bus (Kafka)] ──────▶ [Stream Engine] ──────▶ [Sink] │
 │ (App, IoT, DB)         (Topic / Partition)         (Flink / Streams)       (DB, UI) │
 │      │                          │                         │                 │    │
 │      │ 1. 이벤트 발행            │ 2. 데이터 보관           │ 3. 실시간 연산     │ 4. 결과│
 │      │ (Producer)               │ (Log-structured)        │ (Stateful Calc)  │ 저장   │
 │      │                          │                         │                 │    │
 │      └──────────────────────────┴───────────┬─────────────┴─────────────────┘    │
 │                                            │                                    │
 │                                            ▼                                    │
 │                    ┌──────────────────────────────────┐                         │
 │                    │      State Store (로컬 저장소)      │                         │
 │                    │  - 최근 1시간 평균값 등 보관        │                         │
 │                    └──────────────────────────────────┘                         │
 │                                                                             │
 │  ※ 가치: DB 조인 없이 메모리 수준의 속도로 복잡한 통계 및 탐지 수행                   │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 핵심은 **상태 기반 연산 (Stateful Processing)**이다. **Stream Engine**은 단순히 데이터를 통과시키는 게 아니라, 자신의 로컬 메모리나 디스크에 **State Store**를 두고 "최근 5분간 들어온 이벤트의 합계"와 같은 상태를 기억한다. 이 덕분에 외부 DB를 매번 조회하지 않고도 실시간으로 "지금 결제 시도가 평소보다 10배 많음!"과 같은 고차원적인 분석 결과를 **Sink** 시스템(대시보드나 다른 앱)으로 즉시 던져줄 수 있다.

---

### 심층 동작 원리: 윈도우 (Windowing) 기술

흐르는 데이터에서 '시간의 경계'를 긋는 방법이다.
1. **Tumbling Window**: 1분 단위로 겹치지 않게 딱딱 끊어서 계산. (예: 매 분 정각 통계)
2. **Sliding Window**: 현재 시점부터 과거 1분간의 데이터를 실시간 이동하며 계산. (예: 최근 60초간의 추세)
3. **Session Window**: 사용자의 활동이 끊길 때까지를 하나의 묶음으로 처리. (예: 한 번 접속 시의 활동 로그)

- **📢 섹션 요약 비유**: 끊임없이 돌아가는 회전 초밥(스트림)에서, 매 분마다 접시 개수를 세느냐(Tumbling), 아니면 지금 내 눈앞을 지나가는 최근 10개의 접시만 보느냐(Sliding)의 차이입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Kafka Streams vs Apache Flink 비교

| 비교 항목 | Kafka Streams (라이브러리형) | Apache Flink (클러스터형) |
|:---|:---|:---|
| **배포 방식** | 별도 서버 없음 (앱 내부에 포함) | **독립된 클러스터 서버 필요** |
| **복잡도** | 낮음 (자바 개발자에게 친숙) | 높음 (전용 인프라 관리 필요) |
| **성능/확장성** | 보통 (Kafka 파티션에 종속) | **매우 높음 (대규모 병렬 처리 최강)** |
| **특이 기능** | Kafka와 완벽한 통합 | **Exactly-once 보장, 고도화된 윈도우** |
| **비유** | 자가용 (필요할 때 바로 탐) | 기차 (정해진 노선과 거대 용량) |

---

### 기술 융합: CDC와 스트림 프로세싱

데이터베이스의 변화를 실시간 이벤트로 변환한다.
- **CDC (Change Data Capture)**: DB의 로그를 읽어 데이터가 바뀌는 순간(Insert/Update) 이벤트를 발행한다.
- **가치**: 구형 시스템(레거시)을 건드리지 않고도, DB의 변화를 이벤트 버스에 태워 최신 스트림 분석 시스템과 연결할 수 있다.

- **📢 섹션 요약 비유**: 무뚝뚝한 할아버지(구형 DB) 옆에서 손자(CDC)가 할아버지가 하시는 말씀을 대신 실시간으로 동네방네 소문(이벤트) 내주는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 카드 부정 사용 실시간 탐지 (FDS) 구축

- **문제 상황**: 결제 승인 요청이 들어왔을 때, 0.5초 내에 "해당 카드가 1분 내에 지구 반대편에서 쓰인 적이 있는지" 확인해야 함. 일반 DB 조회로는 속도를 맞출 수 없음.
- **의사결정 (스트림 프로세싱 전략)**:
  1. **이벤트 버스**: 모든 결제 내역을 **Kafka**의 'Payment' 토픽으로 수집한다.
  2. **상태 관리 (Flink)**: Flink 엔진이 카드 번호별로 **마지막 결제 위치와 시간**을 자신의 로컬 상태(State)에 저장한다.
  3. **실시간 대조**: 새 결제 이벤트가 들어오면, 로컬 상태와 즉시 비교하여 물리적으로 불가능한 이동 거리(예: 한국 ──▶ 뉴욕 1분)면 즉시 '부정 의심' 이벤트를 발행한다.
  4. **결과**: 외부 DB 조회 없이 메모리 연산만으로 0.01초 내에 판단을 완료하여 고객 대기 시간을 최소화한다.

### 도입 체크리스트
- **Exactly-once**: 네트워크 오류로 메시지가 중복 처리되어도 통계 수치가 틀리지 않게 보장되는가?
- **State Size**: 상태 데이터(State)가 너무 커져서 서버 메모리를 고갈시키지는 않는가? (Check-pointing 주기 조절)
- **Order Guarantee**: 이벤트의 순서가 뒤바뀌었을 때(Late arrival) 어떻게 처리할 것인가? (Watermark 기술 적용 여부)

### 안티패턴
- **모든 데이터를 스트림으로**: 단순한 조회 업무까지 억지로 스트림 엔진을 거치게 하는 행위. (아키텍처가 너무 무거워짐)
- **Stateful 로직의 남용**: 상태를 너무 많이 저장하여 스트림 엔진 자체가 거대한 DB처럼 변해버리는 경우. (성능 저하의 지름길)

- **📢 섹션 요약 비유**: 흐르는 물에서 금을 캐는 것(스트림 분석)은 좋지만, 강물 전체를 얼려서 보관(과도한 상태 저장)하려다가는 강 전체가 멈춰버릴 수 있음을 경계해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Latency)** | 분석 지연 시간 99% 단축 | 배치 처리(시간 단위)에서 스트림 처리(밀리초 단위)로 전환 |
| **정량적 (Cost)** | 인프라 비용 효율화 | DB에 모든 로그를 쌓지 않고 필요한 요약 정보만 저장하여 스토리지 절감 |
| **정성적 (Standard)** | 데이터 기반 실시간 경영 | 데이터가 발생하는 즉시 비즈니스 통찰력을 얻어 신속한 대응 가능 |
| **정성적 (Resilience)** | 유연한 시스템 확장 | 이벤트 버스를 통한 서비스 간 결합도 제거로 장애 전파 차단 |

### 미래 전망
- **Serverless Streaming**: 데이터 양에 따라 스트림 엔진의 노드 수가 자동으로 조절되는 완전 관리형 서비스가 표준이 될 것이다.
- **AI-Streaming Convergence**: 흐르는 데이터 위에서 AI 모델이 직접 추론(Inference)을 수행하여, 사건이 일어나기도 전에 예측하는 지능형 스트림 아키텍처가 보편화될 것이다.

### 참고 표준
- **The Dataflow Model (Google)**: 현대 스트림 프로세싱의 이론적 기초
- **Apache Kafka / Flink Documentation**: 업계 기술 표준
- **CloudEvents**: 클라우드 간 이벤트 정보 교환 표준

- **📢 섹션 요약 비유**: 멈춰있는 과거(DB)를 공부하던 시대에서, 흐르는 현재(Stream)와 대화하는 시대로 정보 통신 기술은 한 단계 더 진화했습니다.
