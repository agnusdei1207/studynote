# 561-572. 서버리스(FaaS) 아키텍처

> **핵심 인사이트**: 서버리스는 **"서버를 관리하지 않는다"**는 것이 아니라, **"서버리스한 책임을 개발자에게 이동"**시키는 것입니다. 인프라 관리 부담을 줄이고, 비즈니스 로직에 집중할 수 있지만, **콜드 스타트**와 **상태 관리**의 두 가지 주요 과제를 이해해야 합니다.

---

## Ⅰ. 서버리스 개요
### 1. 정의
* **FaaS(Function as a Service)**: 함수 단위로 배포되는 클라우드 컴퓨팅 모델
* **BaaS(Backend as a Service)**: 백엔드 기능을 제공하는 서비스(Firebase, Supabase)
* **이벤트 기반**: 트리거에 의해 실행되는 구조
### 2. 특징
| 항목 | 전통 서버 | 서버리스 |
|-----|---------|----------|
| 인프라 관리 | 직접 관리 | 클라우드 제공자 |
| 확장성 | 수동 | 자동 |
| 비용 | 고정 비용 | 사용량 기반 |
| 배포 단위 | 전체 애플리케이션 | 함수 |
| 상태 관리 | 개발자 | 플랫폼/Stateful |

### 3. 주요 플랫폼
* **AWS Lambda**: 가장 널리 사용되는 FaaS
* **Azure Functions**: Azure의 서버리스 플랫폼
* **Google Cloud Functions**: GCP의 서버리스 플랫폼
* **Cloudflare Workers**: 엣지 컴퓨팅 기반 서버리스

```

## Ⅱ. 콜드 스타트(Cold Start) 문제
### 1. 문제 정의
서버리스 함수가 **유휴 상태(Idle)**에서 **활성 상태(Active)**로 전환될 때 발생하는 **초기 지연** 현상입니다.
```
[요청 도착] ───────────────▶ [Cold Start 발생]
                   │
                   ▼
        ┌───────────────────────────────────────────────────────┐
        │               Function Instance                        │
        │  ┌─────────────────────────────────────────────────┐  │
        │  │           Cold Start Process                  │  │
        │  │  1. Container 시작                     │  │
        │  │  2. 런타임 초기화                    │  │
        │  │  3. 코드 로드                          │  │
        │  │  4. 핸들러 초기화                      │  │
        │  │  5. 실행 준비                          │  │
        │  └─────────────────────────────────────────────────┘  │
        │                         │                        │
        │                         ▼                        │
        │                   [실제 실행 시작]                    │
        └───────────────────────────────────────────────────────┘
```
### 2. 지연 시간 요소
| 요소 | 지연(근사치) |
|-----|-----------|
| Container 시작 | 100-500ms |
| 런타임 초기화 | 50-200ms |
| 코드 로드 | 10-100ms |
| 핸들러 초기화 | 100-500ms |
| DB 연결 | 50-300ms |
| **총 지연** | **300-2000ms** |

### 3. 해결 방안
* **Provisioned Concurrency**: 미리 준비된 실행 인스턴스
* **Connection Pooling**: 데이터베이스 연결 풀 유지
* **초기화 최적화**: 지연 초기화(lazy initialization) 패턴
* **Warming Up**: 주기적인 워밍 요청으로 핫 상태 유지
* **최소 메모리 설정**: 필요한 최소한의 메모리만 할당

```

## Ⅲ. 상태 관리(Stateful Serverless)
### 1. Stateless vs Stateful
| 구분 | Stateless | Stateful |
|-----|----------|----------|
| 상태 저장 | 외부(DB/Cache) | 메모리/로컬 |
| 확장성 | 용이 | 제한적 |
| 복잡도 | 낮음 | 높음 |
| 지연 | 낮음 | 높음(Cold Start 시) |

### 2. Stateful 패턴
* **Durable Functions(Azure)**: 상태를 저장하는 함수
* **Step Functions**: 상태 머신 기반 워크플로우
* **AWS Step Functions**: 상태 기반 오케스트레이션
### 3. 주의사항
* **인스턴스 수 제한**: 동시 실행 인스턴스 제한
* **메모리 관리**: 메모리 누수 방지
* **타임아웃**: 적절한 타임아웃 설정
```

## Ⅳ. 이벤트 소싱(Event Sourcing) with Serverless
### 1. 개념
서버리스 환경에서 이벤트 소싱을 적용하면 **확장성**과 **내결함성**을 모두 확보할 수 있습니다.
```
┌───────────────────────────────────────────────────────────────┐
│               Serverless Event Sourcing                    │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐  │
│  │   Event Producer │  │   Event Bridge  │  │  Event Consumer │  │
│  │  (Lambda/API GW)  │  │  (EventBridge/  │  │    (Lambda)     │  │
│  │                   │  │   Kafka)        │  │                 │  │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘  │
│                                                         │
│                         ▼                               │
│              ┌─────────────────────────┐               │
│              │     Event Store         │               │
│              │   (DynamoDB/Kinesis)    │               │
│              └─────────────────────────┘               │
└───────────────────────────────────────────────────────────────┘
```
### 2. 장점
* **자동 확장**: 이벤트 양에 따라 자동 스케일링
* **비용 효율**: 사용한 만큼만 비용 지불
* **내결함성**: 클라우드 제공자의 고가용성 활용
### 3. 구현 예시
* **AWS**: API Gateway → Lambda → DynamoDB Streams → Lambda
* **Azure**: Event Grid → Functions → Cosmos DB
* **GCP**: Cloud Pub/Sub → Cloud Functions → Firestore
```

## Ⅴ. 스텝 함수(Step Functions)
### 1. 개념
여러 Lambda 함수를 **상태 머신(State Machine)**으로 오케스트레이션하는 서비스입니다.
```
┌───────────────────────────────────────────────────────────────┐
│                    Step Functions Workflow                  │
│                                                         │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐  │
│   │ Step 1  │───▶│ Step 2  │───▶│ Step 3  │───▶│ Step 4  │  │
│   │ Validate│    │ Process │    │  Store  │    │ Notify  │  │
│   └─────────┘    └─────────┘    └─────────┘    └─────────┘  │
│        │              │              │              │        │
│        ▼              ▼              ▼              ▼        │
│   [Failure]        [Success]      [Error]        [Retry]      │
│        │              │              │              │        │
│        └──────────────┴──────────────┴──────────────┘        │
└───────────────────────────────────────────────────────────────┘
```
### 2. 상태 유형
* **Task**: 단일 작업
* **Choice**: 조건부 분기
* **Parallel**: 병렬 실행
* **Map**: 반복 작업
* **Wait**: 대기
### 3. 에러 처리
* **Retry**: 자동 재시도
* **Catch**: 에러 캐치 및 대체 경로
* **Fallback**: 실패 시 대체 작업
```

## Ⅵ. 서버리스 모베 사례
### 1. 함수 설계
* **단일 책임**: 하나의 함수는 하나의 작업만 수행
* **멱등성**: 동일한 입력에 동일한 결과 보장
* **빠른 실행**: 함수 실행 시간 최소화
* **상태 비저장**: 가능한 Stateless 설계
### 2. 성능 최적화
* **패키지 크기 최소화**: 필요한 의존성만 포함
* **초기화 최적화**: 핸들러 외부에서 초기화
* **연결 재사용**: Keep-alive 연결 사용
* **환경 변수 활용**: 설정을 환경 변수로 관리
### 3. 보안
* **최소 권한**: IAM 역할에 최소 권한만 부여
* **환경 변수 암호화**: 민감 정보는 암호화
* **VPC 구성**: 필요시 VPC 내 배포
* **API 키 관리**: Secrets Manager 활용
```
