# 532. 마이크로서비스 (Microservices) 분해 패턴

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스 분해 패턴은 거대한 단일 애플리케이션(Monolithic)을 독립적으로 배포 및 실행 가능한 **작은 서비스 단위로 쪼개는 전략적 방법론**으로, 서비스 간의 응집도는 높이고 결합도는 낮추는 것을 목표로 한다.
> 2. **가치**: 비즈니스 변경에 민첩하게 대응할 수 있는 **독립적 확장성과 배포 속도**를 제공하며, 특정 서비스의 장애가 전체 시스템으로 번지지 않도록 격리함으로써 시스템의 가용성을 극대화한다.
> 3. **융합**: 도메인 주도 설계(DDD)의 Bounded Context 개념과 결합되어 서비스의 경계를 획정하며, 비즈니스 능력(Capability) 또는 하위 도메인(Subdomain) 기준의 분해 전략이 실무의 표준으로 쓰인다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 마이크로서비스 분해는 "어디를 칼로 낼 것인가"의 문제다. 모든 기능이 얽혀 있는 모놀리식 구조를 끊어낼 때, 비즈니스 기능이나 데이터의 연관성을 기준으로 명확한 경계를 나누어 각 서비스를 '독자적인 생명체'로 만드는 과정이다.

- **💡 비유**: 마이크로서비스 분해는 **"한 덩어리 찰흙으로 만든 로봇을 레고 블록 로봇으로 바꾸는 과정"**과 같습니다. 팔이 고장 났을 때 로봇 전체를 다시 빚는 게 아니라, 팔 블록만 쏙 빼서 고치거나 더 강력한 팔로 갈아 끼울 수 있도록 조각조각 잘 나누는 설계 기술입니다.

- **등장 배경**:
  1. **모놀리식의 한계**: 소스 코드가 수백만 줄로 늘어나면서 빌드 시간이 수 시간이 걸리고, 코드 한 줄 수정이 어디에 영향을 줄지 몰라 배포가 두려워지는 상황을 해결해야 했다.
  2. **팀 간 간섭 발생**: 수백 명의 개발자가 하나의 소스를 수정하면서 발생하는 코드 충돌과 배포 대기 현상을 없애기 위해 팀별 독립적인 코드 소유권이 필요해졌다.
  3. **유연한 기술 도입**: 전체 시스템을 자바로 짰더라도, 인공지능 모듈은 파이썬으로, 채팅 모듈은 Go로 짜고 싶은 '기술적 자유'를 추구하게 되었다.

- **📢 섹션 요약 비유**: 복잡한 실타래를 한 올씩 풀어서 색깔별로 정리함으로써, 필요할 때 원하는 실만 골라 쓸 수 있게 만드는 지혜로운 정리 정돈법입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 마이크로서비스 분해의 2대 표준 전략

| 분해 기준 | 상세 설명 | 장점 | 비유 |
|:---|:---|:---|:---|
| **비즈니스 능력별** | 조직의 업무 기능(주문, 결제, 배송) 중심 분해 | 비즈니스 가치와 팀 조직이 일치함 | 회사의 부서별 분리 |
| **하위 도메인별** | DDD의 Bounded Context를 기준으로 분해 | 데이터와 로직의 경계가 가장 명확함 | 전문 분야별 분리 |

---

### 2. 마이크로서비스 분해 프로세스 아키텍처

모놀리식에서 마이크로서비스로 나아가는 단계적 로직을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Monolithic to Microservices Decomposition              │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Step 1: 시스템 분석] ──▶ [Step 2: 경계 식별] ──▶ [Step 3: 서비스 획정] ──▶ [Step 4] │
 │                                                                             │
 │  ┌──────────────┐        ┌──────────────┐        ┌──────────────┐      ┌─────┐  │
 │  │  거대 단일    │        │  Bounded     │        │  독립 서비스  │      │ API │  │
 │  │  애플리케이션  │ ──▶    │  Context     │ ──▶    │  (Deployable)│ ──▶  │ GW  │  │
 │  │  (Big Ball)  │        │  (도메인 경계)  │        │  Unit A, B, C│      └─────┘  │
 │  └──────────────┘        └──────┬───────┘        └──────┬───────┘               │
 │                                 │                       │                       │
 │           ┌─────────────────────┴───────────┬───────────┴───────────┐           │
 │           ▼                                 ▼                       ▼           │
 │    [데이터 독립성 확보]               [통신 방식 정의]          [공통 기능 분리]      │
 │    - Database per Service           - REST / gRPC           - Auth, Logging     │
 │    - Join 제거 및 API 호출           - Event-driven          - Config Server     │
 │                                                                                 │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 분해 아키텍처의 핵심은 **데이터의 고립**이다. 단순히 코드만 나누는 것이 아니라, 각 서비스가 자신만의 전용 데이터베이스를 가지게 하는 **Database per Service** 원칙을 지켜야 진정한 분해가 완성된다. 이 과정에서 발생하는 서비스 간의 데이터 조회는 직접 DB를 뒤지는 것이 아니라 **API 호출**이나 **메시지 큐**를 통한 비동기 이벤트를 통해 이루어진다. 이렇게 쪼개진 서비스들을 외부에서 하나로 묶어주는 것이 **API Gateway**의 역할이다.

---

### 심층 동작 원리: 결합도(Coupling)와 응집도(Cohesion)

분해의 성공 여부를 측정하는 공학적 잣대다.
1. **높은 응집도**: "함께 변하는 코드는 한 서비스 안에 모아라." (예: 주문 처리와 주문 상태 변경은 하나로)
2. **낮은 결합도**: "서비스 하나를 고칠 때 다른 서비스를 고치지 않게 하라." (예: 주문을 고친다고 결제 코드를 만지면 안 됨)
3. **가치**: 이 균형을 잘 맞춘 분해만이 MSA의 진정한 이점인 '독립적 배포'를 가능하게 한다.

- **📢 섹션 요약 비유**: 팀워크가 좋은 사람들끼리 한 팀을 짜주되(응집도), 다른 팀의 업무 방식에는 간섭하지 않게(결합도) 조직을 개편하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 서비스 분해의 트레이드 오프(Trade-off) 분석

분해는 공짜가 아니다. 얻는 것이 있으면 잃는 것도 있다.

| 구분 | 모놀리식 (분해 전) | 마이크로서비스 (분해 후) |
|:---|:---|:---|
| **복잡도** | 코드 내부 복잡도 높음 | **인프라/네트워크 복잡도 폭증** |
| **트랜잭션** | 강력한 일관성 (ACID) | **최종 일관성 (Saga 패턴 필요)** |
| **테스트** | 통합 테스트 용이 | **전 구간 테스트(E2E) 매우 어려움** |
| **배포 속도** | 전체 배포로 인해 느림 | **부분 배포로 인해 매우 빠름** |
| **비유** | 만능 칼 한 자루 | 전문 요리 기구 세트 |

---

### 기술 융합: 스트랭글러 피그 (Strangler Fig) 패턴

기존 모놀리식을 죽이지 않고 조금씩 마이크로서비스로 옮겨가는 융합 전략이다.
- **원리**: 신규 기능은 마이크로서비스로 짜고, 기존 기능 중 하나를 골라 API Gateway에서 마이크로서비스로 경로를 돌린다.
- **가치**: 서비스 중단 없이 안전하게 시스템의 체질을 개선하는 '점진적 분해' 기술이다.

- **📢 섹션 요약 비유**: 낡은 건물을 한꺼번에 허무는 게 아니라, 기둥 하나하나를 최신식으로 교체하면서 계속 영업을 이어가는 호텔 리모델링 공법과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 거대 커머스 시스템의 '주문' 도메인 분해 전략

- **문제 상황**: '주문' 모듈에 결제, 배송, 재고, 쿠폰 로직이 모두 섞여 있어 장애 발생 시 원인 파악이 안 됨.
- **의사결정 (분해 프로세스)**:
  1. **Bounded Context 식별**: '주문 완료'와 '결제 승인'은 서로 다른 트랜잭션임을 인지하고 경계를 긋는다. (DDD 적용)
  2. **API 계약 정의**: 서비스 간 소통을 위한 인터페이스(Swagger 등)를 먼저 확정하여 팀 간 협업 기반을 닦는다.
  3. **데이터 마이그레이션**: 주문 DB에서 결제 관련 테이블을 분리하여 별도의 DB 인스턴스로 옮긴다. (Join 제거)
  4. **비동기 이벤트 도입**: 주문이 완료되면 '주문성공' 이벤트를 발행하고, 결제와 배송 서비스가 이를 구독하여 후속 처리하게 한다. (결합도 최소화)

### 도입 체크리스트
- **Latency**: 네트워크를 타고 다른 서비스를 부르는 시간이 사용자 응답 속도를 해치지는 않는가?
- **Distributed Tracing**: 요청 하나가 여러 서비스를 거칠 때 추적할 수 있는 Trace ID가 구현되어 있는가? (Zipkin, Jaeger)
- **Shared Libraries**: 공통 코드를 공유하느라 서비스 간에 다시 강한 결합이 생기지는 않았는가?

### 안티패턴
- **Distributed Monolith**: 서비스는 쪼갰지만 배포할 때는 다 같이 해야 하는 상태. (분해의 가치 상실)
- **Too Small (Nano-services)**: 너무 잘게 쪼개어 통신 오버헤드가 비즈니스 로직보다 커진 경우.

- **📢 섹션 요약 비유**: "잘게 쪼개는 것"이 목적이 아닙니다. "잘 쪼개는 것"이 목적입니다. 무의미한 조각들은 오히려 관리의 지옥을 선물할 뿐입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Velocity)** | 출시 주기 5배 단축 | 독립 배포를 통해 승인 대기 및 코드 충돌 시간 제거 |
| **정량적 (Scale)** | 인프라 비용 30% 절감 | 부하가 몰리는 특정 서비스만 정밀하게 수평 확장 |
| **정성적 (Stability)** | 시스템 회복력 강화 | 한 서비스가 죽어도 전체 사이트는 살아남는 격리 효과 확보 |
| **정성적 (Innovation)** | 기술 스택의 민주화 | 팀마다 최적의 기술을 실험하고 적용할 수 있는 문화 정착 |

### 미래 전망
- **Self-Splitting Architecture**: AI가 소스 코드의 호출 빈도와 데이터 연관성을 분석하여, 최적의 분해 지점을 스스로 추천해주는 지능형 아키텍처 도구가 등장할 것이다.
- **Wasm-based Micro-components**: 컨테이너보다 더 가벼운 웹어셈블리(Wasm) 단위의 분해가 일어나며 초저지연 분산 시스템이 가능해질 것이다.

### 참고 표준
- **Microservices Patterns (Chris Richardson)**: 분해 패턴의 교과서
- **Building Microservices (Sam Newman)**: 실무적 지침서
- **Domain-Driven Design (Eric Evans)**: 분해의 논리적 근거

- **📢 섹션 요약 비유**: 분해는 파괴가 아니라 '새로운 질서'를 만드는 예술입니다. 촘촘하게 나누고 느슨하게 연결할 때, 비로소 거대한 소프트웨어는 무한한 자유를 얻습니다.
