# 577-580. SSR, SSG, ISR, PWA, WebAssembly

> **핵심 인사이트**: 프론트엔드 렌더링 전략은 **SEO**, **성능**, **개발자 경험** 간의 트레이드오프입니다. SSR은 SEO에 유리하지만 서버 부하가 증가하고, SSG는 가장 빠르지만 동적 콘텐츠에 제약이 있습니다. ISR과 PWA는 이들의 장점을 결합한 하이브리드 접근법입니다.

---

## Ⅰ. 렌더링 전략 비교

### 1. CSR vs SSR vs SSG vs ISR
```
┌─────────────────────────────────────────────────────────────────────┐
│                    Rendering Strategies Comparison                    │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐    │
│  │       CSR         │  │       SSR        │  │       SSG        │    │
│  │  (Client-Side)    │  │  (Server-Side)   │  │  (Static Gen)    │    │
│  │  - 빈 HTML         │  │  - 완성된 HTML    │  │  - 미리 생성      │    │
│  │  - JS로 렌더링    │  │  - 서버에서 렌더  │  │  - 빌드 시 생성   │    │
│  │  - SEO 불리        │  │  - SEO 유리       │  │  - SEO 유리       │    │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘    │
│  ┌─────────────────┐  ┌─────────────────┐                        │
│  │       ISR         │  │       PWA        │                        │
│  │ (Incremental)     │  │ (Progressive)    │                        │
│  │  - 부분 재생성    │  │  - 오프라인 지원  │                        │
│  │  - TTL 기반 갱신  │  │  - 캐싱 전략      │                        │
│  └─────────────────┘  └─────────────────┘                        │
└─────────────────────────────────────────────────────────────────────┘
```

### 2. 성능 비교
| 지표 | CSR | SSR | SSG | ISR |
|-----|-----|-----|-----|-----|
| **FCP** | 느림 | 빠름 | 가장 빠름 | 빠름 |
| **TTI** | 느림 | 중간 | 빠름 | 빠름 |
| **SEO** | 불리 | 유리 | 가장 유리 | 유리 |
| **서버 부하** | 없음 | 높음 | 낮음 | 중간 |
| **동적 콘텐츠** | 용이 | 용이 | 제약 | 부분 가능 |

## Ⅱ. SSR (Server-Side Rendering)

### 1. 동작 흐름
```
[사용자 요청] ──▶ [서버] ──▶ [DB 조회] ──▶ [HTML 렌더링] ──▶ [완성된 HTML]
                                        │
                                        ▼
                              [클라이언트 하이드레이션]
                              (JavaScript 이벤트 연결)
```

### 2. 하이드레이션(Hydration)
SSR로 렌더링된 HTML에 JavaScript 이벤트를 연결하는 과정입니다.
```
[SSR HTML] ──▶ [JS 번들 로드] ──▶ [이벤트 리스너 부착] ──▶ [상호작용 가능]
```

### 3. SSR 프레임워크
* **Next.js (React)**: 가장 인기 있는 SSR 프레임워크
* **Nuxt.js (Vue.js)**: Vue.js 기반 SSR
* **SvelteKit (Svelte)**: Svelte 기반 풀스택 프레임워크
* **Remix (React)**: React 기반 SSR (Cloudflare 지원)

### 4. 장단점
| 장점 | 단점 |
|-----|-----|
| SEO 최적화 | 서버 부하 증가 |
| 초기 로딩 빠름 | TTFB 증가 |
| 소셜 미디어 공유 | 캐싱 복잡성 |

---

## Ⅲ. SSG (Static Site Generation)

### 1. 동작 흐름
```
[빌드 시점]
     │
     ▼
[데이터 페치] ──▶ [HTML 생성] ──▶ [CDN 배포]
                                        │
[요청 시점] ◀────────────────────────────┘
     │
     ▼
[CDN에서 즉시 응답]
```

### 2. 적합한 사용 사례
* **블로그/문서 사이트**: 자주 변경되지 않는 콘텐츠
* **마케팅 페이지**: 정적 랜딩 페이지
* **포트폴리오**: 개인 웹사이트
* **API 문서**: 기술 문서 사이트

### 3. SSG 도구
* **Gatsby**: React 기반 SSG
* **Hugo**: Go 기반 초고속 SSG
* **Jekyll**: Ruby 기반 (GitHub Pages)
* **Astro**: 멀티 프레임워크 지원 SSG

---

## Ⅳ. ISR (Incremental Static Regeneration)

### 1. 개념
SSG와 SSR의 장점을 결합한 하이브리드 방식입니다.
```
[요청 도착]
     │
     ▼
[캐시 확인] ──▶ [캐시 히트] ──▶ [즉시 응답]
     │
     │ [캐시 미스 또는 만료]
     ▼
[백그라운드 재생성] ──▶ [캐시 업데이트]
```

### 2. 재생성 전략
* **TTL(Time-To-Live)**: 캐시 만료 시간 설정
* **On-Demand**: 웹훅 또는 API 호출로 재생성 트리거
* **필수 재생성**: 특정 경로만 재생성

### 3. Next.js ISR 예시
```javascript
export async function getStaticProps() {
  const data = await fetchPosts();

  return {
    props: { data },
    revalidate: 60, // 60초마다 재생성
  };
}
```

## Ⅴ. PWA (Progressive Web App)

### 1. 핵심 기술
```
┌─────────────────────────────────────────────────────────────────────┐
│                        PWA Architecture                               │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐    │
│  │  Service Worker  │  │   Web App Manifest │  │   HTTPS       │    │
│  │  - 캐싱           │  │   - 앱 아이콘       │  │   - 보안       │    │
│  │  - 오프라인 지원  │  │   - 설치 가능       │  │   - 필수      │    │
│  │  - 푸시 알림      │  │   - 시작 URL        │  │              │    │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘    │
│  ┌─────────────────┐  ┌─────────────────┐                        │
│  │   Cache API     │  │   IndexedDB     │                        │
│  │   - HTTP 캐시   │  │   - 구조화된 데이터│                        │
│  │   - 정적 리소스  │  │   - 대용량 저장    │                        │
│  └─────────────────┘  └─────────────────┘                        │
└─────────────────────────────────────────────────────────────────────┘
```

### 2. 오프라인 우선(Offline-First) 패턴
* **Cache First**: 캐시 우선, 네트워크 폴백
* **Network First**: 네트워크 우선, 캐시 폴백
* **Stale While Revalidate**: 캐시 즉시 반환 + 백그라운드 갱신

### 3. 설치 가능한 웹 앱
* **Add to Home Screen**: 홈 화면에 추가
* **App Icon**: 앱 아이콘 지원
* **Splash Screen**: 스플래시 화면
* **Fullscreen**: 전체 화면 실행

## Ⅵ. WebAssembly (WASM)

### 1. 개념
브라우저에서 네이티브 바이너리 코드를 실행할 수 있는 기술입니다.
```
[C/C++/Rust/Go] ──▶ [WASM 바이너리] ──▶ [브라우저 실행]
                          │
                          ▼
              [JavaScript와 상호 운용]
```

### 2. 특징
| 항목 | 설명 |
|-----|------|
| **성능** | JavaScript보다 20-50% 빠름 |
| **이식성** | 모든 주요 브라우저 지원 |
| **보안** | 샌드박스 환경에서 실행 |
| **크기** | 바이너리 형식으로 작은 크기 |

### 3. 사용 사례
* **Figma**: 디자인 도구 (C++ → WASM)
* **Google Earth**: 3D 지도 렌더링
* **AutoCAD Web**: CAD 소프트웨어
* **FFmpeg.wasm**: 브라우저 비디오 처리
* **TensorFlow.js**: 머신러닝 추론

### 4. WASM vs JavaScript
```
┌─────────────────────────────────────────────────────────────────────┐
│                    WASM vs JavaScript                                │
│  ┌─────────────────┐                    ┌─────────────────┐        │
│  │     WASM         │                    │   JavaScript     │        │
│  │  - 컴파일 필요    │                    │  - 인터프리터     │        │
│  │  - 정적 타이핑    │                    │  - 동적 타이핑    │        │
│  │  - 고성능        │                    │  - 유연성         │        │
│  │  - 메모리 직접   │                    │  - GC 지원        │        │
│  └─────────────────┘                    └─────────────────┘        │
│                    ┌─────────────────┐                             │
│                    │   상호 운용 가능   │                             │
│                    │   (WASM ⟷ JS)    │                             │
│                    └─────────────────┘                             │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅶ. 개념 맵 및 요약

```ascii
[프론트엔드 렌더링 전략 결정 트리]

                    [콘텐츠 유형?]
                         │
           ┌─────────────┼─────────────┐
           ▼             ▼             ▼
        [정적]         [동적]       [하이브리드]
           │             │             │
           ▼             ▼             ▼
        [SSG]          [SSR]        [ISR]
           │             │             │
           ▼             ▼             ▼
      [블로그/문서]   [대시보드]   [뉴스/쇼핑몰]
           │             │             │
           └─────────────┼─────────────┘
                         ▼
                    [PWA 추가?]
                         │
           ┌─────────────┼─────────────┐
           ▼             ▼             ▼
        [오프라인]    [푸시 알림]    [설치형]
```

📢 **섹션 요약 비유**: SSR은 **"레스토랑에서 조리된 음식을 받는 것"**과 같습니다. 주문하면 이미 완성된 상태로 받습니다. SSG는 **"도시락"**과 같습니다. 미리 만들어져 있어 즉시 먹을 수 있지만 메뉴가 제한적입니다. ISR은 **"간편식"**과 같습니다. 미리 만들어져 있지만 유통기한이 지나면 새로 만듭니다. PWA는 **"모바일 앱"**과 같습니다. 설치하면 오프라인에서도 작동합니다. WebAssembly는 **"스포츠카"**와 같습니다. 무겁고 빠른 작업에 특화되어 있습니다.
