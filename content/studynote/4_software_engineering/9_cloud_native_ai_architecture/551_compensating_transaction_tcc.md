# 551. 보상 트랜잭션 (Compensating Transaction) 및 TCC 패턴

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보상 트랜잭션은 분산 환경에서 작업 실패 시 이미 완료된 작업을 논리적으로 취소(Undo)하는 기법이며, TCC (Try-Confirm-Cancel) 패턴은 이를 **'가예약(Try) - 확정(Confirm) - 취소(Cancel)'**의 3단계로 정형화한 분산 트랜잭션 해결책이다.
> 2. **가치**: 2PC의 동기적 블로킹 문제를 해결하여 높은 성능을 유지하면서도, 자원을 미리 점유(Try)하고 결과에 따라 확정 혹은 취소함으로써 비즈니스 정합성을 **최종 일관성(Eventual Consistency)** 관점에서 완벽히 보장한다.
> 3. **융합**: 마이크로서비스(MSA) 간의 금전적 거래나 재고 관리 등 엄격한 일관성이 요구되는 도메인에 적용되며, Saga 패턴의 구체적인 구현 기술 중 하나로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **보상 트랜잭션**: "실패했으니 아까 한 일을 없던 일로 하자"는 명령이다. 물리적 롤백이 불가능한 분산 환경에서 논리적으로 반대되는 행위를 수행한다.
  - **TCC (Try-Confirm-Cancel)**: 모든 서비스가 "준비됐어(Try)"라고 답해야만 실제 작업을 확정(Confirm)하고, 한 명이라도 거절하면 모두 취소(Cancel)하는 3단계 프로토콜이다.

- **💡 비유**: TCC 패턴은 **"결혼식 하객 식사 예약"**과 같습니다. 1) 식당에 "30명 자리가 있나요?(Try)" 묻고 임시 예약을 잡습니다. 2) 식이 문제없이 끝나면 "진짜 30명 갑니다(Confirm)"라고 확정합니다. 3) 만약 식이 취소되면 "예약 취소해주세요(Cancel)"라고 알려주어 식당이 다른 손님을 받게 하는 방식입니다.

- **등장 배경**:
  1. **긴 트랜잭션의 폐해**: 분산 환경에서 모든 데이터를 잠금(Lock)하면 시스템 전체가 느려진다. (2PC의 한계)
  2. **비가역적 행위의 관리**: 이메일 발송이나 외부 PG 결제처럼 DB 롤백만으로는 되돌릴 수 없는 '실제 세상의 행위'를 관리할 체계가 필요했다.
  3. **고가용성 요구**: 장애가 발생해도 시스템이 멈추지 않고, 나중에라도 데이터를 맞추는 유연한 정합성 모델이 중요해졌다.

- **📢 섹션 요약 비유**: "만장일치가 아니면 안 움직여!"라는 고집불통 방식에서, "일단 가예약하고 다 모이면 가자!"라는 영리한 예약 시스템으로의 진화입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. TCC 패턴의 3단계 메커니즘

| 단계 | 수행 작업 | 핵심 목표 | 자원 상태 |
|:---|:---|:---|:---|
| **Try (시도)** | 가용 자원 확인 및 가점유 (Reserve) | 작업 가능 여부 판단 | **가예약 상태 (Locked)** |
| **Confirm (확정)** | 실제 비즈니스 로직 완성 (Commit) | 비즈니스 가치 확정 | 완료 상태 |
| **Cancel (취소)** | 가점유 자원 해제 및 원복 (Rollback) | 자원 회수 및 정합성 유지 | 취소 상태 |

---

### 2. TCC 분산 트랜잭션 동작 아키텍처

여러 마이크로서비스가 참여하는 TCC 흐름을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      TCC (Try-Confirm-Cancel) 표준 제어 흐름                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Transaction Manager]           [Service A (Order)]         [Service B (Pay)] │
 │           │                            │                           │        │
 │  1. Try Phase (가예약) ────────────────▶ │ [Check & Reserve]         │        │
 │           │ ───────────────────────────┼─────────────────────────▶ │ [Reserve]│
 │           │                            │                           │        │
 │  2. All Try Success? ──▶ [YES]         │                           │        │
 │           │                            │                           │        │
 │  3. Confirm Phase (확정) ──────────────▶ │ [Finalize Order]          │        │
 │           │ ───────────────────────────┼─────────────────────────▶ │ [Deduct] │
 │           │                            │                           │        │
 │  [IF FAIL CASE]                        │                           │        │
 │  4. Any Try Fail? ──▶ [NO]             │                           │        │
 │           │                            │                           │        │
 │  5. Cancel Phase (취소) ───────────────▶ │ [Release Order]           │        │
 │           │ ───────────────────────────┼─────────────────────────▶ │ [Refund] │
 │                                                                             │
 │  ※ 핵심: Try 단계에서 이미 비즈니스 검증이 끝났으므로 Confirm은 반드시 성공해야 함    │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TCC 아키텍처의 정수는 **Try** 단계에서의 '자원 선점'이다. **Service B(결제)**는 실제 결제 전, 사용자의 한도가 충분한지 확인하고 그만큼의 금액을 다른 곳에 못 쓰게 '가홀딩' 한다. 모든 참여 서비스가 가예약에 성공하면 **Transaction Manager**는 일괄 **Confirm**을 내린다. 만약 중간에 **Service C(재고)**에서 "재고 없음"이라는 에러가 나면, 이미 성공한 A와 B에게 **Cancel** 명령을 내려 가예약된 자원을 즉시 해제한다. 이 방식은 DB의 물리적 Lock 시간을 최소화하여 가용성을 극대화한다.

---

### 심층 동작 원리: 보상 트랜잭션의 '논리적 롤백'

물리적으로 지워진 데이터를 복구하는 것이 아니라, 반대 효과를 내는 행위다.
1. **Insert (추가)** ──▶ **Delete (삭제)** (또는 Flag 처리)
2. **Update (차감)** ──▶ **Update (증가)** (보상 금액 추가)
3. **External API (메일 발송)** ──▶ **External API (취소/사과 메일 발송)**
4. **가치**: "이미 벌어진 일"을 기록으로 남기면서도 최종적인 상태값을 0으로 수렴시킨다.

- **📢 섹션 요약 비유**: 연필로 쓰고 지우개로 지우는 것(물리 롤백)이 아니라, 볼펜으로 쓰고(실행) 그 밑에 "위 내용은 취소함"이라고 도장을 찍는(논리 보상) 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 2PC vs TCC vs Saga 패턴 비교

분산 트랜잭션 해결책 삼대장을 명확히 구분한다.

| 비교 항목 | 2-Phase Commit (2PC) | TCC 패턴 | Saga 패턴 |
|:---|:---|:---|:---|
| **제어 주체** | DB 엔진 (인프라 계층) | **애플리케이션 계층** | 애플리케이션 계층 |
| **일관성 수준** | 강력한 일관성 (Immediate) | **높은 일관성 (가예약)** | 최종 일관성 (Eventual) |
| **작동 방식** | 동기, 블로킹 | **동기 지향 (2단계 합의)** | 비동기, 메시징 |
| **복잡도** | 낮음 (DB가 해줌) | **높음 (취소 로직 직접 구현)** | 높음 (상태 관리 필요) |
| **비유** | 만장일치 투표 | **확인 후 계약서 작성** | 일단 물건 보내고 나중에 반품 |

---

### 기술 융합: TCC와 멱등성 (Idempotency)

네트워크 재시도 시의 위험을 방지한다.
- **원리**: `Confirm`이나 `Cancel` 요청이 여러 번 중복되어 들어와도 한 번만 처리되도록 고유 트랜잭션 ID를 관리한다.
- **가치**: "취소 요청"을 두 번 받았다고 해서 돈을 두 번 돌려주는 대형 사고를 막는다.

- **📢 섹션 요약 비유**: "취소해주세요"라는 말을 백 번 들어도, 이미 취소 도장을 찍었다면 웃으며 넘기는 여유로운 업무 태도가 필요하다는 뜻입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 항공권+호텔 패키지 결제 시스템 설계

- **문제 상황**: 항공권은 성공했는데 호텔 예약이 실패하면 항공권 결제를 취소해야 함. 항공사 API와 호텔 API는 서로 다른 회사에서 운영함.
- **의사결정 (TCC 전략 적용)**:
  1. **Try**: 항공권 '가예약' API와 호텔 '객실 홀딩' API를 호출한다. (결제는 안 함)
  2. **Confirm**: 두 곳 모두 성공하면 실제 '결제 승인' API를 호출하여 티켓을 확정한다.
  3. **Cancel**: 만약 호텔 가예약이 실패하면, 이미 잡힌 항공권 가예약을 취소하는 API를 호출한다.
  4. **결과**: 고객은 "결제 중입니다"라는 메시지를 잠시 보게 되지만, 데이터가 꼬여서 항공권만 결제되고 잘 곳이 없는 낭패를 겪지 않는다.

### 도입 체크리스트
- **Business Transaction Support**: 연동하는 모든 서비스가 '가예약'과 '취소' 기능을 API로 제공하는가?
- **Isolation Level**: 가예약 상태의 자원이 다른 트랜잭션에 의해 사용되지 않도록 격리되는가? (예: 가예약된 재고는 품절 처리)
- **Time-out Policy**: `Try` 이후 너무 오랫동안 `Confirm/Cancel`이 오지 않을 경우 자동으로 취소(Auto-cancel)하는 정책이 있는가?

### 안티패턴
- **Confirm 로직의 복잡화**: Confirm 단계에서 새로운 비즈니스 검증을 수행하는 행위. (Confirm은 무조건 성공해야 하는 로직이어야 함)
- **보상 트랜잭션 누락**: "설마 실패하겠어?"라는 안일함으로 Cancel 로직 작성을 미루는 행위. (데이터 불일치의 지름길)

- **📢 섹션 요약 비유**: "가예약"은 고객과의 약속입니다. 약속을 잡았으면 반드시 지키거나(Confirm) 정중히 취소(Cancel)해야지, 아무 말 없이 방치하는 것이 가장 나쁜 설계입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Perf)** | TPS 2배 이상 향상 | 전역 Lock을 사용하지 않아 동시 처리량 증대 |
| **정량적 (Reliability)** | 데이터 정합성 오류 제로화 | 단계별 합의 과정을 통해 분산 환경의 원자성 확보 |
| **정성적 (Standard)** | 비즈니스 워크플로우 명확화 | 수동으로 관리하던 취약한 취소 로직을 아키텍처 수준으로 표준화 |
| **정성적 (Stability)** | 장애 전파 차단 | 서비스 간 동기 대기를 최소화하여 연쇄 장애 리스크 감소 |

### 미래 전망
- **Smart TCC Frameworks**: 개발자가 로직만 작성하면, 상태 전이와 타임아웃 처리를 AI가 자동으로 관리해주는 고수준 프레임워크가 보편화될 것이다.
- **Transactional Mesh**: 서비스 메시 계층에서 TCC 흐름을 투명하게 가로채어 관리하는 '인프라 레벨의 분산 트랜잭션' 기술이 발전할 것이다.

### 참고 표준
- **Sagas (H. Garcia-Molina)**: 보상 트랜잭션의 이론적 배경
- **XA Specification**: 분산 트랜잭션의 기술적 표준
- **CAP Theorem**: 일관성과 가용성 사이의 공학적 타협 이론

- **📢 섹션 요약 비유**: 세상은 완벽하지 않습니다. 모든 것이 한 번에 성공하길 바라는 마음보다, 실패했을 때 정직하게 되돌리는 용기가 더 훌륭한 시스템을 만듭니다. TCC는 그 정직함을 기술로 구현한 것입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 친구들과 놀이터에 갈 때, **"모두 갈 수 있는지 먼저 물어보고(Try), 모두 된다고 하면 출발(Confirm)하는 규칙"**이에요.
2. 만약 한 명이라도 "나 못 가!"라고 하면, 이미 신발을 신은 친구들도 다시 신발을 벗고 집에서 노는 것(Cancel)과 같답니다.
3. 이렇게 미리 확인하고 행동하면, 나중에 놀이터에서 친구를 못 만나서 슬퍼하는 일을 미리 막을 수 있어요!
