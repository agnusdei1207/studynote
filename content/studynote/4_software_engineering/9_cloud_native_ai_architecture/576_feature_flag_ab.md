# 576. 피처 플래그 (Feature Flag) 기반 A/B 테스트 및 점진적 롤아웃

> **핵심 인사이트**: 피처 플래그는 **코드 재배포 없이 런타임에 기능을 켜고 끌 수 있는 스위치**입니다. 이를 통해 A/B 테스트, 점진적 롤아웃, 카나리 배포를 코드 수정 없이 구현할 수 있어, **위험을 최소화**하고 **피드백을 빠르게 수집**할 수 있습니다.

---

## Ⅰ. 피처 플래그 개요

### 1. 정의
피처 플래그(Feature Flag)는 애플리케이션의 기능을 런타임에 켜거나 끄는 설정 메커니즘입니다.

### 2. 핵심 구성 요소
```
┌─────────────────────────────────────────────────────────────┐
│                    Feature Flag System                       │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐ │
│  │   Flag Store     │  │   Evaluation     │  │   SDK/Client     │ │
│  │   (저장소)        │  │   Engine          │  │   Library       │ │
│  │   - DB/Redis     │  │   (평가 엔진)      │  │   (SDK)          │ │
│  │   - Config File │  │   - Rules          │  │   - API Calls    │ │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### 3. 주요 플래그 유형
| 유형 | 용도 | 예시 |
|-----|------|------|
| **Boolean** | 단순 on/off | `new_checkout_enabled` |
| **Multivariate** | 여러 변형 중 선택 | `button_color: [red, blue, green]` |
| **Percentage** | 사용자 비율 | `new_feature_rollout: 10%` |

## Ⅱ. A/B 테스트 구현

### 1. A/B 테스트 흐름
```
[사용자 요청]
      │
      ▼
┌─────────────────────────────────────────────────────────────┐
│                    Feature Flag Evaluation                       │
│  ┌─────────────────┐                                        │
│  │   사용자 ID 해시 │ ──────────────────────┐               │
│  │   (User Hashing) │                         │               │
│  └─────────────────┘                          ▼               │
│                                    ┌─────────────────┐        │
│                                    │   A 그룹 (50%)    │        │
│                                    │   B 그룹 (50%)    │        │
│                                    └─────────────────┘        │
└─────────────────────────────────────────────────────────────┘
      │
      ▼
┌─────────────────┐  ┌─────────────────┐
│   Variant A      │  │   Variant B      │
│   (기존 기능)     │  │   (신규 기능)     │
└─────────────────┘  └─────────────────┘
```

### 2. 사용자 분배 알고리즘
* **해시 기반 분배**: `user_id % 100 < percentage`
* **UUID 기반 분배**: 일관된 사용자 경험 보장
* **랜덤 분배**: 간단하지만 일관성 부족

### 3. 메트릭 수집
| 메트릭 | 설명 |
|-------|------|
| **노출 수** | 각 변형별 사용자 수 |
| **전환율** | 목표 달성 비율 |
| **이탈률** | 이탈한 사용자 비율 |
| **체류 시간** | 평균 세션 시간 |

## Ⅲ. 점진적 롤아웃 전략

### 1. 롤아웃 단계
```
[1%] ──▶ [5%] ──▶ [10%] ──▶ [25%] ──▶ [50%] ──▶ [100%]
  │        │        │         │         │         │
  ▼        ▼        ▼         ▼         ▼         ▼
모니터링 ──▶ 문제 시 롤백 ──▶ 정상 시 확대 ──▶ 완료
```

### 2. 롤아웃 기준
* **에러율**: 임계치 이상 시 롤백
* **응답 시간**: P99 지연 시 롤백
* **사용자 피드백**: 부정적 피드백 급증 시 롤백
* **비즈니스 메트릭**: 전환율 저하 시 롤백

### 3. 카나리 배포와의 연계
```
┌─────────────────────────────────────────────────────────────┐
│              Canary Deployment + Feature Flags                  │
│  ┌─────────────────┐                                        │
│  │   Canary Pod 10% │ ──────▶ Feature Flag: 10%             │
│  └─────────────────┘          (동일한 사용자 그룹)           │
│                                    │                         │
│                                    ▼                         │
│                           ┌─────────────────┐                │
│                           │   통합된 롤아웃   │                │
│                           │   10% → 100%     │                │
│                           └─────────────────┘                │
└─────────────────────────────────────────────────────────────┘
```

## Ⅳ. 구현 도구

### 1. 주요 도구
| 도구 | 특징 | 적용 환경 |
|-----|------|----------|
| **LaunchDarkly** | 엔터프라이즈급, 실시간 | 대규모 서비스 |
| **Unleash** | 오픈소스, 자체 호스팅 | 온프레미스 |
| **Split.io** | A/B 테스트 특화 | 데이터 중심 |
| **Firebase Remote Config** | 모바일 특화 | 모바일 앱 |
| **AWS AppConfig** | AWS 통합 | AWS 환경 |

### 2. 코드 예시
```javascript
// Feature Flag 평가
const featureFlags = {
  'new-checkout': {
    enabled: true,
    variants: {
      'control': { percentage: 50 },
      'treatment': { percentage: 50 }
    }
  }
};

// 사용자 분배
function getVariant(userId, flagName) {
  const hash = hashCode(userId + flagName) % 100;
  const flag = featureFlags[flagName];

  let cumulative = 0;
  for (const [variant, config] of Object.entries(flag.variants)) {
    cumulative += config.percentage;
    if (hash < cumulative) return variant;
  }
  return 'control';
}
```

## Ⅴ. 모베 사례

### 1. 플래그 관리
* **수명 주기 관리**: 사용 완료된 플래그 제거
* **명명 규칙**: 일관된 플래그 이름
* **문서화**: 플래그 목적과 기록 유지

### 2. 위험 관리
* **Kill Switch**: 즉시 비활성화 가능한 메커니즘
* **모니터링**: 실시간 메트릭 대시보드
* **롤백 계획**: 사전 정의된 롤백 절차

### 3. 성능 고려사항
* **평가 지연**: 플래그 평가 오버헤드 최소화
* **캐싱**: 자주 변경되지 않는 플래그 캐싱
* **CDN 활용**: 엣지에서 평가 수행

## Ⅵ. 개념 맵 및 요약

```ascii
[Feature Flag Lifecycle]

  생성 ──▶ 활성화 ──▶ 모니터링 ──▶ 최적화 ──▶ 비활성화 ──▶ 제거
    │         │          │          │          │          │
    ▼         ▼          ▼          ▼          ▼          ▼
  [개발]   [A/B 테스트] [메트릭]  [롤아웃]   [롤백]    [정리]
```

📢 **섹션 요약 비유**: 피처 플래그는 **"조명 스위치"**와 같습니다. 방의 불을 켜고 끄는 것처럼, 코드를 다시 배포하지 않고도 기능을 켜고 끌 수 있습니다. A/B 테스트는 **"맛보기 시식"**과 같습니다. 두 그룹에게 다른 메뉴를 제공하고 어떤 것이 더 인기 있는지 확인하는 것입니다. 점진적 롤아웃은 **"수도권 확장"**과 같습니다. 작은 지역에서 시작해서 점차 전국으로 확장하는 것처럼, 소수 사용자에게서 시작해 전체로 확대합니다.
