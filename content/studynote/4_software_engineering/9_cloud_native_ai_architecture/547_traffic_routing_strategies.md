# 547. 트래픽 라우팅 전략 (Canary, Blue-Green, A/B)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트래픽 라우팅 전략은 새로운 버전의 소프트웨어를 운영 환경에 배포할 때, 서비스 중단 없이 **트래픽의 흐름을 지능적으로 제어하여 리스크를 최소화**하고 품질을 검증하는 배포 아키텍처 기법이다.
> 2. **가치**: 장애 발생 시 즉각적인 롤백(Rollback) 기능을 제공하며, 실제 사용자 데이터를 기반으로 신규 기능의 성능(Canary)과 비즈니스 효과(A/B Testing)를 사전에 검증함으로써 '무결점 배포'를 지향한다.
> 3. **융합**: 쿠버네티스(K8s)의 서비스 제어, 서비스 메시(Istio)의 가중치 기반 라우팅, 그리고 API 게이트웨이 기술과 결합되어 현대적 CD(지속적 배포) 파이프라인의 핵심 공정으로 작동한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 트래픽 라우팅은 단순히 코드를 서버에 올리는 것을 넘어, 사용자가 신기술을 만나는 '방식'을 설계하는 것이다. 구버전과 신버전을 동시에 띄워두고 트래픽을 조금씩 옮기거나(Canary), 한꺼번에 스위칭(Blue-Green)하거나, 특정 그룹에게만 노출(A/B)하는 기법들을 포함한다.

- **💡 비유**: 트래픽 라우팅 전략은 **"기차 선로의 분기점 교체"**와 같습니다. 기차가 달리는 도중에(무중단 서비스), 낡은 선로(구버전)에서 새 선로(신버전)로 기차를 안전하게 옮기는 작업입니다. 한 번에 확 옮길지(Blue-Green), 빈 차만 먼저 보내보고 안전하면 승객을 태울지(Canary) 결정하는 것과 같습니다.

- **등장 배경**:
  1. **무중단 서비스(Zero-downtime) 요구**: 점검 중 페이지를 띄우는 과거 방식은 글로벌 경쟁 시대에 고객 이탈의 주범이 되었다.
  2. **장애 파급력의 거대화**: 코드 한 줄의 실수가 전체 시스템 다운으로 이어지는 리스크를 줄이기 위해 '일부 사용자'에게만 먼저 노출하는 실험적 배포가 필요해졌다.
  3. **데이터 기반 의사결정**: "개발자가 보기에 좋은 기능"이 아니라 "실제 사용자가 좋아하는 기능"인지 통계적으로 확인(A/B Testing)하고 싶어 했다.

- **📢 섹션 요약 비유**: 사고가 날지 모르는 새 도로를 한꺼번에 개통하는 위험을 피하고, 시운전 차량부터 차례로 보내보며 안전을 확인하는 '지능형 교통 관제' 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 3대 트래픽 라우팅 전략 비교

| 전략 명칭 | 핵심 동작 원리 | 주요 장점 | 비유 |
|:---|:---|:---|:---|
| **Blue-Green** | 구버전(Blue)과 신버전(Green)을 똑같이 띄우고 한 번에 스위칭 | **빠른 롤백**, 구조 단순 | 구교량 옆에 새 교량 짓고 동시 개통 |
| **Canary** | 일부 트래픽(1%~10%)만 신버전으로 먼저 보냄 | **장애 피해 최소화**, 실전 성능 검증 | 광산의 카나리아 (위험 미리 감지) |
| **A/B Testing** | 특정 조건(나이, 지역 등)의 사용자에게만 신기능 노출 | **비즈니스 지표 검증**, UX 최적화 | 시식 행사 후 선호도 조사 |

---

### 2. 카나리(Canary) 배포 아키텍처 (Service Mesh 기반)

서비스 메시를 통해 트래픽 비율을 정밀하게 조정하는 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Service Mesh 기반 가중치 트래픽 라우팅                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Client Request] ─────▶ [API Gateway / Ingress] ────────────────────┐       │
 │                                     │                               │       │
 │                                     ▼ [Service Mesh Control Plane]  │       │
 │                    ┌──────────────────────────────────┐             │       │
 │                    │  Traffic Split Policy (Istio)    │             │       │
 │                    │  - v1 (Blue)  : 90%              │             │       │
 │                    │  - v2 (Canary): 10%              │             │       │
 │                    └──────┬────────────────────┬──────┘             │       │
 │                           │                    │                    │       │
 │         ┌─────────────────┘                    └─────────────────┐  │       │
 │         ▼                                                        ▼  │       │
 │  ┌──────────────┐                                         ┌──────────────┐  │
 │  │ Service v1   │                                         │ Service v2   │  │
 │  │ (Stable)     │                                         │ (Canary)     │  │
 │  └──────┬───────┘                                         └──────┬───────┘  │
 │         │                                                        │          │
 │         └────────────▶ [실시간 모니터링 및 비교] ◀──────────────────┘          │
 │                        (Error Rate, Latency)                                │
 │                                                                             │
 │  ※ 결정: v2의 에러율이 낮으면 10% ──▶ 50% ──▶ 100% 순차적으로 비중 확대           │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 핵심은 **가중치(Weight)** 제어다. 1) 운영 서버에 v1(기존)과 v2(신규)를 동시에 띄운다. 2) **서비스 메시(Istio 등)** 설정 파일에서 v2로 가는 트래픽을 10%로 제한한다. 3) 10%의 사용자만 v2를 경험하는 동안, 보안 및 운영팀은 v2에서 에러가 발생하는지 실시간 감시한다. 4) 문제가 없다면 비중을 점점 늘려 최종적으로 v1을 삭제한다. 만약 문제가 생기면 v2로 가는 10%의 길만 0.1초 만에 닫으면 되므로 사고 범위를 획기적으로 줄일 수 있다.

---

### 심층 동작 원리: Blue-Green 배포의 환경 분리

가장 확실하고 빠른 배포/롤백 기법이다.
1. **Idle 상태 활용**: Green(신규) 환경이 준비되는 동안 사용자 트래픽은 Blue(기존)에만 머문다.
2. **Atomic Switching**: 로드 밸런서의 포인터를 Blue에서 Green으로 바꾸는 찰나에 모든 사용자가 새 서비스를 보게 된다.
3. **가치**: 문제가 생기면 포인터만 다시 Blue로 돌리면 되므로 롤백 시간이 사실상 0에 가깝다.

- **📢 섹션 요약 비유**: 이삿짐을 다 옮겨놓고(Green 준비), 살던 집 열쇠를 반납하고 새 집 비밀번호를 입력하는(Switching) 순간 이동 이사와 같습니다.

| **추천 상황** | 미션 크리티컬 시스템 | 대규모 동시 접속 서비스 | UI/UX 개선, 추천 알고리즘 변경 |

### 기술 융합: 트래픽 라우팅과 피처 플래그 (Feature Flag)

코드 수준의 라우팅 기술과 결합된다.
- **Feature Toggle**: 코드는 배포하되 기능은 꺼둔 상태(OFF)로 둔다.
- **융합 효과**: 특정 사용자(예: 사내 직원)의 세션에만 기능을 켜서(ON) 테스트하는 '다크 런칭(Dark Launching)'이 가능해진다.

- **📢 섹션 요약 비유**: 도로는 개통해놓고 입구에 가림막을 쳐둔 뒤, VIP 손님(베타 테스터)에게만 가림막을 살짝 열어주는 프리미엄 시승식과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 커머스 앱의 '결제 모듈' 메이저 업데이트 배포

- **문제 상황**: 결제 로직을 전면 개편했는데, 한 번에 바꿨다가 결제 오류라도 나면 매출에 치명적인 타격을 입게 됨.
- **의사결정 (라우팅 전략 적용)**:
  1. **Phase 1 (Shadow Deployment)**: 트래픽은 구버전으로 보내되, 요청을 복사(Mirroring)해서 신버전에도 쏴본다. 실제 결제는 안 하고 "결과가 일치하는지"만 비교한다.
  2. **Phase 2 (Canary 1%)**: 이상 없음 확인 후, 아주 소량의 트래픽만 신버전으로 보낸다.
  3. **Phase 3 (Progressive Rollout)**: 1% ──▶ 5% ──▶ 20%로 늘려가며 매출 지표가 떨어지지 않는지 확인한다.
  4. **Phase 4 (Full Switch)**: 최종 확인 후 100% 전환하고 구버전 리소스를 회수한다.

### 도입 체크리스트
- **Sticky Session**: 카나리 배포 중 사용자가 새로고침을 했을 때 버전이 왔다 갔다 하는 '깜빡임' 현상이 발생하지 않도록 세션 유지가 되는가?
- **Data Compatibility**: 구버전과 신버전이 동일한 DB를 바라볼 때, 데이터 스키마 충돌이 발생하지 않는가? (Backward Compatibility 확보 필수)
- **Monitoring Automation**: 에러율이 임계치를 넘으면 시스템이 스스로 롤백하는 'Auto-rollback' 로직이 파이프라인에 포함되었는가?

### 안티패턴
- **Big-bang Release**: 라우팅 전략 없이 한꺼번에 전체 시스템을 교체하는 행위. 사고 발생 시 복구가 불가능해진다.
- **지표 없는 카나리**: 트래픽만 나눠놓고 정작 에러 로그나 성능 데이터를 보지 않는 무의미한 단계 나누기.

- **📢 섹션 요약 비유**: "일단 믿고 타보세요"라는 무책임한 선장(개발자)이 아니라, 구명보트와 안전망을 겹겹이 준비한 뒤 조심스럽게 항해를 시작하는 꼼꼼한 항해사의 마음가짐이 필요합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Uptime)** | 가동 시간 99.99% 달성 | 무중단 배포를 통해 배포 중단 시간(Downtime) 제로화 |
| **정량적 (Stability)** | 장애 영향 범위 90% 축소 | 카나리 배포를 통해 문제 발생 시 피해를 극소수 사용자로 한정 |
| **정성적 (Confidence)** | 배포 공포증 해소 | 언제든 즉시 롤백 가능하다는 믿음이 개발팀의 배포 속도 향상 |
| **정성적 (Customer)** | 사용자 경험 연속성 확보 | 업데이트 시 사이트 멈춤 현상 없이 부드러운 서비스 개선 제공 |

### 미래 전망
- **AI-Managed Progressive Delivery**: AI가 실시간 비즈니스 지표를 분석하여, 배포 성공 여부를 스스로 판단하고 다음 단계의 트래픽 비중을 자율 결정할 것이다.
- **Infrastructure-less Routing**: 서버 인프라 제어 없이 앱 레이어의 서비스 메시 기술만으로 완벽한 트래픽 샌드박스를 구축하는 고도의 추상화가 실현될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **Blue-Green**은 낡은 미끄럼틀 옆에 **새 미끄럼틀을 미리 만들어놓고, 한꺼번에 "자! 이제 새 거 타자!"라고 옮기는 것**과 같아요.
2. **Canary**는 새 미끄럼틀이 튼튼한지 확인하기 위해, **가장 용감한 친구 한 명만 먼저 타보게 해서 안전한지 지켜보는 것**이랍니다.
3. 이렇게 조심조심 확인하면서 바꾸면, 아무도 다치지 않고 아주 재미있게 새로운 놀이터에서 놀 수 있어요!
