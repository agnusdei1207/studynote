# 571. 탄력성 (Resiliency) 및 결함 허용 (Fault Tolerance) 패턴

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 탄력성(Resiliency)과 결함 허용(Fault Tolerance)은 시스템의 일부 컴포넌트에 장애가 발생하더라도 **전체 서비스가 붕괴되지 않고 핵심 기능을 유지하거나 신속하게 복구**할 수 있게 하는 설계 속성이다.
> 2. **가치**: 마이크로서비스(MSA) 환경에서 피할 수 없는 '부분 장애'를 수용하고, 장애가 다른 서비스로 전파되는 **연쇄 장애(Cascading Failure)**를 차단하여 비즈니스 연속성과 고객 신뢰도를 극대화한다.
> 3. **융합**: 서킷 브레이커(Circuit Breaker), 재시도(Retry), 타임아웃(Timeout), 벌크헤드(Bulkhead) 등 다양한 디자인 패턴이 결합되며, 현대 아키텍처에서는 서비스 메시(Service Mesh)를 통해 인프라 레벨에서 자동화된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 
    - **탄력성 (Resiliency)**: 시스템이 충격(장애)을 받았을 때 얼마나 빨리 원래의 정상 상태로 회복하느냐에 대한 능력.
    - **결함 허용 (Fault Tolerance)**: 결함이 발생하더라도 중단 없이 서비스를 지속적으로 제공할 수 있는 설계적 강인함.
- **💡 비유**: 탄력성은 **"오뚝이"**와 같습니다. 넘어져도 즉시 다시 일어나는 복원력을 의미합니다. 결함 허용은 **"비행기의 쌍발 엔진"**과 같습니다. 엔진 하나가 고장 나도 나머지 엔진으로 비행을 지속하여 안전하게 착륙할 수 있는 이중화 구조를 의미합니다.
- **등장 배경**:
    1. **분산 시스템의 불확실성**: 수백 개의 서비스가 네트워크로 얽힌 환경에서는 "어딘가에서 반드시 장애가 발생한다"는 전제가 필요해졌다.
    2. **눈사태 효과 (Snowball Effect)**: 하위 서비스의 지연이 상위 서비스의 쓰레드 고갈로 이어져 전체 시스템이 다운되는 비극을 막아야 했다.
    3. **사용자 경험(UX) 보호**: 전체 기능 중 추천 시스템(비핵심)이 죽더라도, 주문(핵심)은 가능하게 만드는 '우아한 성능 저하(Graceful Degradation)'가 비즈니스 생존 전략이 되었다.

- **📢 섹션 요약 비유**: 태풍이 불 때 부러지지 않게 유연하게 흔들리는 대나무와, 뿌리가 깊어 비바람을 견디는 고목의 장점을 결합하여 어떤 시련에도 무너지지 않는 '디지털 방파제'를 만드는 과정입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

탄력성 설계를 위해 실무에서 가장 많이 활용되는 4대 패턴을 계층적으로 분석한다.

| 패턴 명칭 | 핵심 목표 | 작동 원리 | 비유 |
|:---:|:---|:---|:---|
| **Timeout** | 무한 대기 방지 | 정해진 시간 내에 응답이 없으면 연결을 강제 종료 | 약속 장소에서 10분만 기다리기 |
| **Retry** | 일시적 오류 극복 | 네트워크 순단 시 일정 간격(Backoff)을 두고 재요청 | 전화가 안 걸리면 다시 걸기 |
| **Circuit Breaker** | 장애 전파 차단 | 에러율이 임계치를 넘으면 통로를 물리적으로 차단 | 집안의 누전 차단기 |
| **Bulkhead** | 리소스 격리 | 특정 서비스용 쓰레드 풀을 분리하여 자원 점유 방지 | 배의 침수를 막는 격벽 |

#### [탄력성 패턴 통합 제어 아키텍처]

요청이 유입되어 백엔드 서비스에 도달하기까지 각 패턴이 어떻게 레이어링 되는지 시각화한다.

```text
 [ User Request ]
        │
        ▼
 ┌─────────────────────────────────────────────────────────────────┐
 │ API Gateway / Service Mesh Sidecar (Control Plane)              │
 │ ┌──────────────────┐      ┌──────────────────┐      ┌────────┐ │
 │ │  1. Timeout      │ ──▶  │ 2. Retry Logic   │ ──▶  │ 3. CB  │ │
 │ │ (Max Wait Time)  │      │ (Exponential)    │      │ (State)│ │
 │ └──────────────────┘      └──────────────────┘      └────┬───┘ │
 └──────────────────────────────────────────────────────────┼──────┘
                                                            │
        ┌───────────────────( Circuit Open? )───────────────┤
        │ [YES: Fail Fast]                                  │ [NO: Success]
        ▼                                                   ▼
 ┌──────────────┐                          ┌────────────────────────────────┐
 │ 4. Fallback  │                          │ 5. Bulkhead (Isolated Pool)    │
 │ (Static Data)│                          │ ┌────────────┐  ┌────────────┐ │
 │ └─▶ "죄송함다"│                          │ │ Pool A (10)│  │ Pool B (5) │ │
 └──────────────┘                          │ └──────┬─────┘  └────────────┘ │
                                           └────────┼───────────────────────┘
                                                    ▼
                                           [ Backend Service ]
```

**[다이어그램 해설]**
1. **차단기(Circuit Breaker)와 폴백(Fallback)**: 서비스가 아픈 상태라면 굳이 호출하여 상처를 덧나게 하지 않고, 미리 준비된 '가짜 데이터'나 '안내 메시지'로 즉각 응답(Fail-fast)한다.
2. **격벽(Bulkhead)**: A 서비스가 느려져서 쓰레드를 다 써버려도, B 서비스를 위한 전용 쓰레드 풀은 살아있으므로 B는 정상 작동한다.
3. **재시도(Retry)**: 소나기처럼 금방 지나가는 에러는 사용자 몰래 다시 시도하여 성공시킨다.

#### 심층 동작 원리: 우아한 성능 저하 (Graceful Degradation)
- **원리**: 시스템 부하가 임계치에 도달하면 중요도가 낮은 기능을 자동으로 비활성화한다.
- **사례**: 이커머스에서 '실시간 추천 상품' 로직이 늦어지면 해당 영역을 빈칸으로 두거나 정적 이미지로 대체하고, '결제' 로직에 CPU 자원을 몰아준다.

- **📢 섹션 요약 비유**: 큰 화재(장애)가 발생했을 때, 박물관의 모든 보물을 다 챙기려다 다 타버리는 대신, 국보급 보물(핵심 기능)만 먼저 챙겨서 대피로로 빠져나오는 명민한 위기 관리 전략입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. Resiliency vs High Availability (HA)

| 비교 항목 | 고가용성 (HA) | 탄력성 (Resiliency) |
|:---:|:---|:---|
| **접근 방식** | **물리적 중복성** (Redundancy) | **논리적 대응** (Adaptability) |
| **주요 수단** | 클러스터링, 다중 리전 배포 | 서킷 브레이커, 타임아웃, 리팩토링 |
| **핵심 가치** | 서버가 죽지 않게 함 | **서버가 죽어도 서비스는 유지함** |
| **비유** | 전용 주차장 확보 | 유동적인 발렛 파킹 |

#### 2. 네트워크 과목과의 융합: TCP Retransmission vs Application Retry
- **L4 레벨**: 패킷 손실 시 TCP가 자동으로 재전송. (인프라 레벨)
- **L7 레벨**: 애플리케이션이 비즈니스 응답 코드를 보고 재시도 결정. (애플리케이션 레벨)
- **시너지**: 인프라 레벨의 견고함과 앱 레벨의 지능형 회복력이 결합되어 완벽한 신뢰성을 구축한다.

- **📢 섹션 요약 비유**: 튼튼한 장갑(HA)을 끼는 것만큼이나, 뜨거운 것을 만졌을 때 즉시 손을 떼는 반사 신경(Resiliency)을 기르는 것이 현대 분산 시스템의 생존 비결입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 특정 외부 API(예: 지도 서비스) 장애로 인한 시스템 마비**
    - **문제**: 지도를 가져오는 API가 지연되자 메인 화면 로딩이 멈추고 서버의 모든 워커 쓰레드가 대기 상태에 빠짐.
    - **의사결정**: **Resilience4j**를 사용하여 해당 API 호출부에 **Circuit Breaker**와 **Timeout(2s)** 적용.
    - **결과**: 지도 서비스 장애 시 2초 만에 '지도를 불러올 수 없습니다' 메시지를 띄우고 메인 화면은 정상 노출. 서버 쓰레드 점유율 정상화.

- **도입 체크리스트**:
    1. **Idempotency**: Retry를 적용할 로직이 여러 번 호출되어도 안전한가? (결제 두 번 되면 안 됨)
    2. **Backoff Strategy**: 재시도 시 지수 백오프(Exponential Backoff)를 적용하여 장애 서버를 확인사살하지 않도록 했는가?
    3. **Observability**: 차단기가 열리고 닫히는 상태를 실시간으로 모니터링하고 있는가?

- **안티패턴**: 
    - 무분별한 무한 재시도 (Infinite Retry) -> 스스로 자행하는 DDoS 공격이 됨.
    - 모든 서비스에 동일한 타임아웃 설정 -> 서비스 성격(조회 vs 대량 연산) 무시.

- **📢 섹션 요약 비유**: "실패를 계획하라(Design for Failure)." 모든 코드가 성공할 것이라는 낙관론을 버리고, 반드시 실패할 것이라는 비관론적 설계를 하는 것이 진정한 아키텍트의 성숙함입니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량적 기대효과**:
    - **가용성 증가**: 연간 장애 시간 80% 이상 단축.
    - **운영 비용**: 장애 복구에 투입되는 엔지니어 리소스 절감.
- **미래 전망**: **Chaos Engineering**이 보편화되어 실제 운영 환경에 인위적인 장애를 주입하며 탄력성을 상시 단련하는 '면역 체계형 인프라'가 주류가 될 것이다.
- **참고 표준**: **Reactive Manifesto** (반응성, 회복력 강조), **NIST SP 800-160** (시스템 레질리언스 지침).

- **📢 섹션 요약 비유**: 수동으로 펌프질을 하던 시대에서, 수압이 높아지면 알아서 밸브를 조절하는 스마트 그리드(Smart Grid)처럼 소프트웨어도 지능형 생존 체계로 진화하고 있습니다.
