# 561. 컨테이너 기반 배포 (Blue-Green, Canary)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컨테이너 기반 배포는 도커와 쿠버네티스의 불변(Immutable) 특성과 동적 라우팅 기능을 활용하여, 서비스 중단 없이 신규 버전을 출시하고 리스크를 최소화하는 **무중단 배포 아키텍처**다.
> 2. **가치**: **Blue-Green** 배포를 통해 즉각적인 롤백(Rollback) 능력을 확보하고, **Canary** 배포를 통해 일부 사용자에게만 점진적으로 노출함으로써 실제 운영 환경에서의 안정성을 검증한다.
> 3. **융합**: 서비스 메시(Istio), Ingress 컨트롤러, 그리고 GitOps 파이프라인과 결합되어 트래픽의 가중치 제어와 자동화된 배포 승인 프로세스를 실현한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 컨테이너 기술은 서버 환경을 이미지로 고정할 수 있게 해주어, 배포 전략을 훨씬 정교하게 짤 수 있게 한다. 
  - **Blue-Green**: 구버전(Blue)과 신버전(Green)을 통째로 띄워두고 한 번에 교체.
  - **Canary**: 신버전의 인스턴스를 소량만 띄워 트래픽의 일부(예: 10%)만 보내고 모니터링.

- **💡 비유**: 컨테이너 배포 전략은 **"기차 선로 교체 작업"**과 같습니다. 기차가 달리는 도중(서비스 운영 중)에 낡은 선로(Blue)를 새 선로(Green)로 안전하게 갈아 끼우는 것입니다. 한 번에 확 옮길지(Blue-Green), 빈 차만 먼저 보내보고 안전하면 승객을 태울지(Canary) 결정하는 것과 같습니다.

- **등장 배경**:
  1. **무중단 서비스(Zero-downtime) 필수화**: "서버 점검 중" 페이지는 경쟁력이 떨어지는 증거가 되었다. 24시간 끊김 없는 업데이트가 요구되었다.
  2. **장애 파급력 최소화**: 배포 후 발생하는 버그가 전체 사용자에게 영향을 주지 않도록 '방어적인 배포'가 필요해졌다.
  3. **인프라의 유연성**: 컨테이너 덕분에 동일한 사양의 서버를 1초 만에 여러 대 복제할 수 있게 되면서 복잡한 배포 전략이 가능해졌다.

- **📢 섹션 요약 비유**: 사고가 날지 모르는 새 도로를 한꺼번에 개통하는 위험을 피하고, 시운전 차량부터 차례로 보내보며 안전을 확인하는 '지능형 교통 관제' 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 컨테이너 배포 전략 비교

| 전략 명칭 | 핵심 동작 원리 | 장점 | 단점 |
|:---|:---|:---|:---|
| **Blue-Green** | 구/신 버전 완전 병렬 운영 후 L7 스위칭 | **빠른 롤백**, 구조 단순 | 리소스 비용 2배 소요 |
| **Canary** | 특정 비율(Weight)만큼 점진적 전환 | **리스크 분산**, 실전 성능 검증 | 라우팅 설정 복잡 |
| **Rolling Update** | 순차적으로 하나씩 교체 (K8s 기본) | 리소스 효율적 | 배포 중 구/신 버전 공존 기간 긺 |

---

### 2. 쿠버네티스 기반 Canary 배포 아키텍처 (Istio 활용)

서비스 메시를 통해 트래픽을 정교하게 나누는 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Kubernetes & Service Mesh Canary 배포 구조                  │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Client] ──▶ [Ingress Gateway] ───────────────────┐                         │
 │                                                   │                         │
 │                                     ▼ [Virtual Service (Weight)]            │
 │                    ┌──────────────────────────────────┐                     │
 │                    │  Traffic Split Policy            │                     │
 │                    │  - v1 (Stable): 90%              │                     │
 │                    │  - v2 (Canary): 10%              │                     │
 │                    └──────┬────────────────────┬──────┘                     │
 │                           │                    │                            │
 │         ┌─────────────────┘                    └─────────────────┐          │
 │         ▼                                                        ▼          │
 │  ┌──────────────┐                                         ┌──────────────┐  │
 │  │ Service v1   │                                         │ Service v2   │  │
 │  │ (Pod group)  │                                         │ (Pod group)  │  │
 │  └──────┬───────┘                                         └──────┬───────┘  │
 │         │                                                        │          │
 │         └────────────▶ [실시간 메트릭 비교 분석] ◀──────────────────┘          │
 │                        (Error Rate, Latency)                                │
 │                                                                             │
 │  ※ 결정: v2 에러율 < 1% 시 10% ──▶ 50% ──▶ 100% 확대 (Progressive Delivery)     │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 핵심은 **가중치 기반 라우팅**이다. 1) 기존의 안정적인 버전(v1)과 새로운 버전(v2)을 동시에 실행한다. 2) **Istio**와 같은 서비스 메시 설정에서 v2로 가는 트래픽을 10%로 제한한다. 3) 모니터링 시스템은 v2를 사용하는 10%의 사용자에게서 에러가 발생하는지 집중 감시한다. 4) 문제가 없다면 10% ──▶ 30% ──▶ 100%로 비중을 높여 배포를 완성한다. 만약 에러가 급증하면 설정 파일의 숫자만 다시 0%로 바꾸면 즉시 롤백이 완료된다.

---

### 심층 동작 원리: Blue-Green 배포의 원자적 스위칭 (Atomic Switch)

가장 직관적이고 확실한 배포 방법론이다.
1. **Prepare**: Green 환경에 신규 코드를 배포하고 헬스 체크를 마친다. (사용자는 여전히 Blue를 봄)
2. **Switch**: 로드 밸런서의 엔드포인트를 Blue 서비스에서 Green 서비스로 한 번에 바꾼다.
3. **가치**: 구버전과 신버전이 공존하는 '중간 지대'가 없어 데이터 정합성 관리가 쉽고, 장애 시 즉시 예전 포인터로 되돌릴 수 있다.

- **📢 섹션 요약 비유**: 이삿짐을 다 옮겨놓고(Green), 현관 비밀번호만 바꾸면(Switch) 즉시 새 집에서 살 수 있게 되는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 트래픽 라우팅 전략별 선택 기준

| 기준 | Rolling Update | Blue-Green | Canary |
|:---|:---:|:---:|:---:|
| **인프라 비용** | 저렴 | **비쌈 (2배)** | 보통 |
| **롤백 속도** | 느림 | **매우 빠름** | 보통 |
| **사용자 체감** | 공존 기간 발생 | 즉시 변경 | 일부만 변경 |
| **적합한 경우** | 일반적인 앱 | **미션 크리티컬 (금융 등)** | **대규모 트래픽, 신기능 검증** |

---

### 기술 융합: GitOps와 점진적 배포 (ArgoCD Rollouts)

배포 전략이 코드에 의해 자동화된다.
- **ArgoCD Rollouts**: 쿠버네티스 커스텀 컨트롤러로, Blue-Green과 Canary 배포를 선언적으로 정의한다.
- **가치**: "성공률이 99% 이상이면 10분마다 10%씩 트래픽을 늘려라"라는 복잡한 배포 시나리오를 사람의 개입 없이 기계가 수행한다.

- **📢 섹션 요약 비유**: 비행기를 조종할 때 기장이 일일이 레버를 당기는 게 아니라, 목적지를 입력하면 인공지능 부조종사(GitOps)가 알아서 이착륙과 순항을 안전하게 마치는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 메이저 업데이트 시의 '데이터 정합성' 사고 예방

- **문제 상황**: Blue-Green 배포를 하려는데, 신버전(Green)이 DB 스키마를 변경해야 함. 배포 중 구버전(Blue)과 충돌 위험 발생.
- **의사결정 (데이터 호환성 확보)**:
  1. **2-Step Schema Change**: 1단계로 구/신버전 모두 읽을 수 있는 컬럼을 추가 배포하고, 2단계로 배포 완료 후 안 쓰는 컬럼을 삭제하는 방식으로 설계한다.
  2. **Feature Toggle**: 코드 내에 기능을 끄고 켤 수 있는 스위치를 두어, 인프라 배포와 기능 활성화 시점을 분리한다.
  3. **Backward Compatibility**: API 규격이 바뀔 경우 `v1`, `v2`를 동시에 유지하여, 클라이언트가 천천히 업데이트할 시간을 벌어준다.
  4. **결과**: 인프라 배포 방식(Blue-Green)과 데이터의 무결성을 동시에 확보하여 '무중단+무사고'를 달성한다.

### 도입 체크리스트
- **Sticky Session**: 카나리 배포 중 사용자가 새로고침을 했을 때 버전이 왔다 갔다 하는 '깜빡임' 현상이 방지되는가?
- **Auto-rollback Policy**: 특정 에러 임계치 도달 시 1초 내에 자동으로 이전 버전으로 복구되는가?
- **Resource Quota**: 두 버전을 동시에 띄울 수 있는 인프라 자원(CPU/Memory) 여유가 충분한가?

### 안티패턴
- **매뉴얼 카나리**: 사람이 눈으로 로그를 보면서 수동으로 트래픽 숫자를 바꾸는 행위. (대응이 늦어 대형 사고로 번짐)
- **공유 세션 무시**: 두 버전이 세션 저장소(Redis)의 데이터 형식을 다르게 처리하여, 사용자가 튕겨 나가는 현상 방치.

- **📢 섹션 요약 비유**: 기차 선로를 바꾸는데, 기차 바퀴 규격(데이터 형식)이 맞지 않으면 아무리 훌륭한 선로(배포 전략)라도 기차는 탈선할 수밖에 없습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Uptime)** | 가동 시간 99.99% 달성 | 무중단 배포를 통해 계획된 점근 중단 시간(Downtime) 제로화 |
| **정량적 (Stability)** | 장애 영향 범위 90% 축소 | 카나리 배포를 통해 사고 발생 시 피해를 극소수 사용자로 한정 |
| **정성적 (Confidence)** | 배포 공포증 해소 | 언제든 즉시 롤백 가능하다는 믿음이 개발팀의 배포 속도 향상 |
| **정성적 (Brand)** | 서비스 신뢰도 제고 | 업데이트 시 사용자 불편 없는 매끄러운 서비스 개선 제공 |

### 미래 전망
- **Self-Healing Progressive Delivery**: AI가 실시간 비즈니스 지표(매출, 클릭률 등)를 분석하여 배포 성공 여부를 스스로 판단하고 트래픽을 조절하는 단계까지 발전할 것이다.
- **Micro-Canary**: 서비스 단위가 아닌 특정 사용자 그룹이나 지역 단위의 초미세 트래픽 제어 기술이 표준 인프라로 자리 잡을 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **Blue-Green**은 낡은 미끄럼틀 옆에 **새 미끄럼틀을 미리 만들어놓고, 한꺼번에 "자! 이제 새 거 타자!"라고 옮기는 것**과 같아요.
2. **Canary**는 새 미끄럼틀이 튼튼한지 확인하기 위해, **가장 용감한 친구 한 명만 먼저 타보게 해서 안전한지 지켜보는 것**이랍니다.
3. 이렇게 조심조심 확인하면서 바꾸면, 아무도 다치지 않고 아주 재미있게 새로운 놀이터에서 놀 수 있어요!
