# 328-335. 고품질 구현과 시큐어 코딩 (Clean Code, Secure Coding)

> **핵심 인사이트**: 코드는 기계가 읽기 위해서가 아니라 '사람이 읽기 위해서' 작성되어야 한다. 클린 코드는 유지보수라는 이름의 고통을 줄여주며, 시큐어 코딩은 개발 단계에서부터 해커의 침입 통로를 원천 봉쇄하는 가장 경제적이고 강력한 보안 대책이다.

---

## Ⅰ. 클린 코드 (Clean Code) 원칙
누구나 쉽게 읽고 이해하며 수정할 수 있는 가독성 높은 코드입니다.

* **5대 원칙**:
  1. **의미 있는 이름**: 변수와 함수명에 의도를 명확히 담기.
  2. **단일 책임**: 하나의 함수는 오직 한 가지 일만 수행.
  3. **주석 최소화**: 코드 자체로 설명되도록 작성. (주석은 '왜'에 집중)
  4. **포맷 유지**: 일관된 코딩 컨벤션 준수.
  5. **중복 제거**: 공통 로직은 모듈화. (DRY 원칙)

---

## Ⅱ. 시큐어 코딩 (Secure Coding)
소프트웨어 개발 과정에서 발생할 수 있는 보안 취약점을 최소화하기 위한 코딩 기법입니다.

* **7대 보안 약점 (KISA)**: 입력 데이터 검증 및 표현, 보안 기능, 시간 및 상태, 에러 처리, 코드 오류, 캡슐화, API 오용.
* **핵심 수칙**:
  - 외부 입력값은 반드시 검증 (SQL Injection, XSS 방어).
  - 중요 정보(비밀번호 등)는 암호화 전송 및 저장.
  - 에러 메시지에 시스템 내부 정보를 노출하지 않음.

---

## Ⅲ. 코드 리뷰 및 분석 기법
1. **코드 리뷰 (Code Review)**: 동료 개발자들이 코드를 검토하여 논리적 결함이나 가독성을 개선하는 활동. (PR 기반 검토)
2. **정적 분석 (Static Analysis)**: 프로그램을 실행하지 않고 도구(SonarQube 등)를 이용해 소스코드의 취약점과 복잡도를 분석.
3. **동적 분석 (Dynamic Analysis)**: 프로그램을 실행하면서 메모리 누수나 성능 병목을 탐지.

---

## Ⅳ. 기술 부채 (Technical Debt) 관리
* 빠른 출시를 위해 품질을 희생하며 쌓인 '나중에 고쳐야 할 빚'입니다. 
* 주기적인 **리팩토링**을 통해 이자(유지보수 비용 상승)가 불어나기 전 부채를 상환해야 합니다.

---

## Ⅴ. 개념 맵 및 요약

```ascii
[고품질 소프트웨어의 3대 축]

  [ Clean Code ] (가독성) ──┐
                            ├─> [ Sustainable SW ] (지속 가능한 개발)
  [ Secure Code ] (안전성) ──┘           │
                                        ▼
                          [ Continuous Review & Refactoring ]
```

📢 **섹션 요약 비유**: **클린 코드**는 나중에 누가 봐도 금방 물건을 찾을 수 있게 잘 정리된 '서재'와 같습니다. **시큐어 코딩**은 집을 지을 때 도둑이 들어올 만한 구멍을 미리 시멘트로 메우는 방범 공사입니다. **코드 리뷰**는 글을 다 쓴 뒤 맞춤법이나 논리 오류가 없는지 친구에게 검수받는 '교정 작업'이며, **기술 부채**는 당장 편하려고 거실에 던져둔 짐들이 쌓여 나중에 발 디딜 틈조차 없게 되는 상황과 같습니다.
