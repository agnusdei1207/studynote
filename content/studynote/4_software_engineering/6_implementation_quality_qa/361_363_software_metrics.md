# 361-363. 소프트웨어 복잡도와 메트릭 (McCabe, Halstead, CK)

> **핵심 인사이트**: "측정할 수 없는 것은 관리할 수 없다." 소프트웨어의 복잡도는 단순히 '어렵다'는 느낌이 아니라, 제어 흐름의 갈래 수(McCabe)나 사용된 연산자의 양(Halstead)으로 수치화되어야 한다. 특히 객체지향 시스템에서는 클래스 간의 얽힘(Coupling)을 CK 메트릭스로 측정하여 잠재적인 위험 요소를 조기에 차단해야 한다.

---

## Ⅰ. 맥케이브 순환 복잡도 (McCabe's Cyclomatic Complexity)
프로그램의 논리적인 복잡도를 제어 흐름 그래프(Control Flow Graph)를 통해 측정하는 기법입니다.

* **공식**: $V(G) = E - N + 2$
  - **E (Edge)**: 화살표(흐름)의 개수.
  - **N (Node)**: 노드(문장)의 개수.
* **다른 계산법**: 분기점(Predicate Node)의 개수 + 1. (또는 그래프로 둘러싸인 영역의 수)
* **의미**: 복잡도가 10을 넘으면 구조가 복잡하여 오류 발생 가능성이 높고 테스트가 어렵다고 판단합니다.

---

## Ⅱ. 할스테드 (Halstead) 소프트웨어 과학
소스코드에 사용된 **연산자(Operator)**와 **피연산자(Operand)**의 개수를 기반으로 복잡도를 측정하는 기법입니다.

* **측정 요소**:
  - $n_1, n_2$: 고유한 연산자/피연산자 수.
  - $N_1, N_2$: 전체 연산자/피연산자 수.
* **지표**: 프로그램 어휘량, 전체 길이, 볼륨(Volume), 노력(Effort) 등을 계산하여 코드의 양적인 복잡도를 산출합니다.

---

## Ⅲ. 객체지향 메트릭 (CK 메트릭스)
Chidamber와 Kemerer가 제안한 객체지향 설계의 품질 측정 지표입니다.

1. **WMC (Weighted Methods per Class)**: 클래스 내 메서드의 복잡도 합계.
2. **DIT (Depth of Inheritance Tree)**: 상속 계층의 깊이. (너무 깊으면 이해하기 힘듦)
3. **NOC (Number of Children)**: 자식 클래스의 개수.
4. **CBO (Coupling Between Objects)**: 다른 클래스와 연관된 정도. **(낮을수록 좋음)**
5. **RFC (Response For a Class)**: 클래스에서 호출할 수 있는 메서드의 총합.
6. **LCOM (Lack of Cohesion in Methods)**: 응집도 결여 지수. **(낮을수록 응집도가 높음)**

---

## Ⅳ. 개념 맵 및 요약

```ascii
[SW 복잡도 측정의 세 관점]

  ( 논리 흐름 ) [ McCabe ] : "길이 얼마나 복잡하게 꼬였나?" (V(G))
      │
  ( 양적 분석 ) [ Halstead ] : "부품(단어)을 얼마나 많이 썼나?"
      │
  ( 설계 구조 ) [ CK Metrics ] : "클래스들이 얼마나 끈끈/복잡한가?"
```

📢 **섹션 요약 비유**: **맥케이브 복잡도**는 미로(코드)에서 갈림길이 몇 개인지 세어보는 것입니다. 갈림길이 많을수록 길 잃기(버그) 쉽죠. **할스테드**는 요리 레시피에 들어가는 재료와 도구의 총 개수를 세어 요리의 난이도를 측정하는 것이며, **CK 메트릭스**는 가족 관계도(상속)와 인맥(결합도)을 분석하여 집안 싸움(오류 파급)이 날 가능성을 미리 점쳐보는 것과 같습니다.
