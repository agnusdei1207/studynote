# 336-338. 제어의 역전과 관점 지향 프로그래밍 (IoC, DI, AOP)

> **핵심 인사이트**: 좋은 프레임워크는 개발자를 자유롭게 한다. 객체 생성을 직접 하지 않고 프레임워크에 맡기는 제어의 역전(IoC)과, 비즈니스 로직과 상관없는 로그/보안 코드를 밖으로 떼어내는 관점 지향(AOP) 기술은 개발자가 오직 '핵심 가치'에만 집중할 수 있는 환경을 제공한다.

---

## Ⅰ. 라이브러리 vs 프레임워크 (IoC)
* **라이브러리 (Library)**: 개발자가 필요할 때 호출해서 쓰는 도구 모음. (주도권이 **개발자**에게 있음)
* **프레임워크 (Framework)**: 프로그램의 큰 틀을 제공하고 개발자의 코드를 대신 호출함. (주도권이 **프레임워크**에 있음 $\rightarrow$ **IoC, Inversion of Control**)

---

## Ⅱ. 의존성 주입 (DI, Dependency Injection)
객체가 직접 자신이 사용할 객체를 생성하지 않고, **외부에서 생성된 객체를 넣어주는(주입)** 방식입니다.

* **목적**: 객체 간의 **결합도(Coupling)를 낮추어** 코드 변경과 테스트를 쉽게 함.
* **방식**: 생성자 주입, Setter 주입, 필드 주입.

---

## Ⅲ. 관점 지향 프로그래밍 (AOP)
여러 모듈에 공통적으로 나타나는 **횡단 관심사 (Cross-cutting Concerns)**를 분리하여 설계하고 구현하는 방식입니다.

* **핵심 요소**:
  - **Aspect**: 횡단 관심사(로그, 보안, 트랜잭션 등)를 모듈화한 것.
  - **Advice**: 언제 무엇을 할 것인가에 대한 정의.
  - **JoinPoint / Pointcut**: Advice가 적용될 위치와 대상 메서드.
  - **Weaving**: 비즈니스 로직에 Aspect를 결합하는 과정.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[AOP의 횡단 관심사 분리]

  [ Module A ]  [ Module B ]  [ Module C ]
       │             │             │
  ─────┼─────────────┼─────────────┼─────> [ Logging Aspect ] (횡단)
  ─────┼─────────────┼─────────────┼─────> [ Security Aspect ]
       ▼             ▼             ▼
  ( 핵심 비즈니스 로직만 깔끔하게 남음 )
```

📢 **섹션 요약 비유**: **IoC**는 내가 직접 운전(라이브러리)하다가 '자율 주행차(프레임워크)'에 몸을 맡기는 변화입니다. **DI**는 내 컴퓨터 부품을 일체형으로 사지 않고, 고장 나면 램이나 그래픽카드만 따로 사서 꽂는(주입) '조립식 PC'와 같습니다. **AOP**는 집을 지을 때 거실, 안방(핵심 로직)을 만드는 일과, 모든 방에 공통으로 들어가는 전기, 수도(횡단 관심사) 공사를 분리하여 전문가에게 맡기는 효율적인 시공법과 같습니다.
