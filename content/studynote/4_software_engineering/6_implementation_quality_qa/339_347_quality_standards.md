# 339-347. 소프트웨어 품질 표준 (ISO 9126, ISO 25010)

> **핵심 인사이트**
> 1. **본질**: 소프트웨어 품질은 주관적 만족도가 아닌, ISO/IEC 25010 (Systems and software Quality Requirements and Evaluation)에서 정의하는 **8대 제품 품질 특성**과 **사용 품질(Quality in Use)**으로 정량화되는 공학적 지표이다.
> 2. **가치**: 현대 SW 아키텍처에서 단순 기능(Functionality) 만족을 넘어, **보안성(Security)**과 **호환성(Compatibility)**을 필수 요소로 통합하여 클라우드 및 분산 환경의 신뢰성을 보장하고 비즈니스 리스크를 최소화한다.
> 3. **융합**: SW 공학(SW Engineering)의 품질 관리(Quality Management)와 IT 운영의 **SLA (Service Level Agreement)**, 그리고 정보 보안의 **취약점 분석** 데이터를 연계하여 전 수명 주기(Life Cycle) 동안의 품질 지표를 통합 관리하는 방향으로 진화하고 있다.

---

### Ⅰ. 개요 (Context & Background)

소프트웨어 품질 표준은 소프트웨어의 "얼마나 좋은가?"라는 모호한 질문에 대해 엔지니어링적으로 답할 수 있는 측정 기준을 제공한다. 과거 단순한 기능 수행 여부를 넘어, 현대의 복잡한 시스템 환경에서는 보안, 성능, 유지보수성 등 다차원적인 관점에서의 평가가 요구된다.

이를 위해 국제 표준화 기구(ISO)와 국제 전기 표준 회의(IEC)는 **ISO/IEC 9126** 표준을 제정하였으며, 이를 현대적인 기술 환경에 맞춰 **ISO/IEC 25010 SQuaRE (System and software Quality Requirements and Evaluation)** 시리즈로 대체하였다. SQuaRE는 소프트웨어의 요구사항 정의, 평가, 테스트를 포함한 전 프로세스를 아우르는 통합 품질 모델이다.

**💡 비유**
이는 건물을 지을 때 '단순히 무너지지 않는다(기능)'는 기준을 넘어, '내진 설계(신뢰성)', '에너지 효율(성능)', '층간 소음(사용성)', '확장 가능성(유지보수성)' 등을 종합적으로 평가하는 '건축물 성능 등급 인증제'와 같다.

**등장 배경**
1.  **기존 한계**: 초기 소프트웨어는 기능 구현이 최우선이었으나, 규모가 커지면서 유지보수 비용 폭증 및 보안 사고 발생.
2.  **혁신적 패러다임**: 품질을 '결함 없는 코드'에서 '사용자 맥락을 충족하는 가치'로 확장하여 정의한 ISO 25010 등장.
3.  **현재의 비즈니스 요구**: 클라우드 네이티브(Cloud Native) 환경과 DevOps 자동화 파이프라인 내에서 품질 지표를 수치화하여 지속적으로 모니터링할 수 있는 기준의 필요성 대두.

```ascii
[소프트웨어 품질 표준의 진화 과정]

  [1991] ISO 9126 (6대 특성 중심)
     │
     │  + 보안 강화(해킹 위협 증가)
     │  + 호환성 강화(확장성 요구)
     ▼
  [2011] ISO/IEC 25010 (SQuaRE, 8대 특성)
     │
     └─> 제품 품질(Product Quality) + 사용 품질(Quality in Use)
```

*해설: 위 다이어그램은 소프트웨어 품질 모델이 단순한 '제품의 속성(ISO 9126)'에서 '사용자 경험과 환경적 적응성(ISO 25010)'을 포함하도록 확장되었음을 시각화한다. 특히 인터넷 보급 이후 급증한 보안 요구사항과 다양한 플랫폼 간 연결성을 반영하여 6개 특성에서 8개 특성으로 세분화된 계기가 잘 나타나 있다.*

> 📢 **섹션 요약 비유**: 소프트웨어 품질 표준의 진화는 **자동차 산업의 안전 기준 발전**과 같습니다. 초기엔 "차가 잘 굴러가는가(기능)"만 따졌지만, 지금은 "충돌 시 에어백이 터지는가(보안/신뢰성)", "다른 회사 부품이 호환되는가(호환성)", "수리가 쉬운가(유지보수성)" 등을 종합적으로 평가하는 **'안전 성능 등급 5단계'** 기준으로 발전한 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

ISO/IEC 25010은 소프트웨어 품질을 **8대 제품 품질 특성(Product Quality)**과 **5대 사용 품질(Quality in Use)**으로 계층화하여 정의한다. 이 구조는 품질 요구사항을 정의하고, 측정하고, 분석하는 기술적 체계를 제공한다.

#### 1. 8대 제품 품질 특성 상세 분석

| 분류 (Category) | 품질 특성 (Characteristic) | 내부 동작 및 정의 (Definition) | 주요 측정 메트릭 (Metrics) |
|:---:|:---|:---|:---|
| **기능 (Functional)** | **기능 적합성** (Functional Suitability) | 명시된 요구사항과 의도된 목적을 정확히 만족하는가? | 기능 커버리지(%), 기구현률 |
| **신뢰 (Reliability)** | **신뢰성** (Reliability) | 주어진 조건에서 일정 수준의 성능을 유지하는가? | MTBF (Mean Time Between Failures), MTTR |
| **사용 (Usability)** | **사용성** (Usability) | 특정 목적을 달성하기 위해 얼마나 효과적, 효율적, 만족스러운가? | 학습 시간, 조작 오류율, SUS 점수 |
| **성능 (Efficiency)** | **성능 효율성** (Performance Efficiency) | 사용된 자원 양에 비해 성능을 얼마나 잘 수행하는가? | 처리량(Throughput), 응답 시간(Latency), CPU/메모리 사용률 |
| **유지 (Maintainability)** | **유지보수성** (Maintainability) | 수정(Modiﬁcation)이 얼마나 용이한가? | MTTC (Mean Time To Change), 복잡도(Complexity) |
| **이식 (Portability)** | **이식성** (Portability) | 여러 환경(HW/SW/OS)으로 전이/적용이 가능한가? | 이식 효율, 의존성 비율 |
| **보안 (Security)** | **보안성** (Security) | **[신규]** 사람/기타 개체/시스템이 데이터/정보에 접근하는 것을 제어하는가? | 취약점 개수, 보안 공격 방지율, 암호화 강도 |
| **호환 (Compatibility)** | **호환성** (Compatibility) | **[신규]** 다른 제품/시스템과 정보를 교환하며 공존할 수 있는가? | 인터페이스 준수율, 데이터 변환 오류율 |

#### 2. 품질 모델 아키텍처

```ascii
[ISO/IEC 25010 품질 모델 구조도]
     _____________________________________________
    |            사용 품질 (Quality in Use)        |
    |  (실제 사용 환경에서의 최종 결과물 품질)     |
    |---------------------------------------------|
    |  효율성 | 유효성 | 만족도 | 자유도 | 리스크완화 |
    |______________|______________________________|
           ▲
           | 상위 수준의 인자
           | (사용자 관점)
    _______|______________________________________
    |         8대 제품 품질 (Product Quality)      |
    | (시스템 내부 동작 및 정적/동적 특성)        |
    |---------------------------------------------|
    | [1] 기능적      : 기능적합성               |
    | [2] 신뢰성      : 성숙성, 회복성, 가용성    |
    | [3] 사용성      : 인식성, 학습성, 운용성    |
    | [4] 효율성      : 시간, 자원, 행위         |
    | [5] 유지보수성  : 분석성, 수정성, 시험성    |
    | [6] 이식성      : 적응성, 설치성, 공존성    |
    | [7] 보안성      : 기밀성, 무결성, 부인방지   |
    | [8] 호환성      : 상호운용성, 존재성         |
    |______________|______________________________|
           ▲
           | 상세 하위 요소 (Sub-characteristics)
    _______|______________________________________
    |          품질 측정 (Quality Measure)         |
    |    (실무 코드/레벨에서의 메트릭 수집)       |
    |_____________________________________________|
```

*해설: 이 다이어그램은 ISO 25010의 계층 구조를 보여준다. 가장 하단에는 실제 소스 코드나 서버 로그에서 수집 가능한 수량적 데이터(품질 측정)가 있고, 이것들이 중간 단계의 8대 제품 품질 특성으로 집약된다. 이 제품 품질 특성들이 결합되어 최종적으로 사용자가 느끼는 '사용 품질(상단)'을 형성하게 된다. 예를 들어, '데이터베이스 쿼리 응답 시간(측정)'이 '성능 효율성(특성)'에 영향을 주고, 이는 최종적으로 사용자의 '만족도(사용 품질)'를 결정한다.*

#### 3. 핵심 로직: 품질 평가 프로세스 (Quality Evaluation Process)

1.  **요구사항 정의 (Requirement)**: 비즈니스 목표에 맞춰 8대 특성에 대한 우선순위(Trade-off) 설정.
2.  **메트릭 선정 (Measure)**: 각 특성을 측정할 수 있는 정량적 지표(예: MTBF, Response Time) 정의.
3.  **측정 및 분석 (Evaluation)**: 실제 SW 개발 단계 혹은 운영 환경에서 데이터 수집.
4.  **피드백 (Feedback)**: 측정된 품질 데이터가 기준을 충족하지 못했을 경우, 리팩토링이나 아키텍처 개선 수행.

> 📢 **섹션 요약 비유**: ISO 25010의 아키텍처는 **스마트폰 제조 공정의 품질 관리(QC) 단계**와 같습니다. 연구소에서 카메라 화소(기능)와 배터리 수명(성능)을 설계하고(제품 품질), 방수/충격 방지 처리(보안/신뢰성)를 거쳐 최종 제품을 만듭니다. 하지만 실제 고객이 만족하는지는 실사용 환경에서 화면이 얼마나 잘 터치되고 배터리가 얼마나 오래가는지(사용 품질) 테스트해야 최종 평가가 완료됩니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

소프트웨어 품질은 독립적으로 존재하는 것이 아니며, 시스템 아키텍처의 다른 영역과 긴밀하게 상호작용한다. 특히 최근의 **DevSecOps (Development, Security, and Operations)** 환경에서는 품질 특성 간의 트레이드오프 관계가 중요한 의사결정 포인트가 된다.

#### 1. ISO 9126 vs ISO 25010 비교 분석

| 구분 | ISO/IEC 9126 (1991) | ISO/IEC 25010 (2011, SQuaRE) |
|:---|:---|:---|
| **모델 구조** | 6대 품질 특성 (Functionality, Reliability, Usability, Efficiency, Maintainability, Portability) | 8대 품질 특성 (6대 + Security, Compatibility) |
| **보안(Security)** | '기능성(Functionality)'의 하위 개념으로 취급 (약한 중요성) | 독립적인 최상위 **보안성(Security)** 특성으로 격상 (CRITICAL) |
| **상호운용성** | '이식성(Portability)'의 일부로 간주 | 독립된 **호환성(Compatibility)** 특성으로 분리 |
| **사용자 관점** | 주로 제품 중심 (Internal Quality) | **사용 품질(Quality in Use)**을 명확히 구분하여 강조 |
| **적용 환경** | 전통적인 Waterfall 개발, Stand-alone SW | Agile, DevOps, Cloud, AI 기반 시스템 |

#### 2. 타 영역 융합 및 시너지 (Convergence)

1.  **SW 품질 & 보안 (Security Synergy)**
    *   **기술적 연계**: ISO 25010의 **보안성(Security)**은 NIST 프레임워크나 OWASP Top 10과 연동된다. 예를 들어, '입력 유효성 검사(Functional Suitability)'가 강화되면 'SQL Injection(Security Risk)'이 줄어들어 결과적으로 보안성과 신뢰성이 동시에 향상된다.
    *   **오버헤드**: 과도한 보안 강화(암호화, 로깅)는 **성능 효율성(Performance Efficiency)**에 부정적 영향을 주어 Latency를 증가시킬 수 있으므로, **Trade-off(균형)** 설계가 필수적이다.

2.  **SW 품질 & 클라우드 운영 (Operations Synergy)**
    *   **기술적 연계**: 클라우드 환경에서의 **이식성(Portability)**은 **컨테이너화(Containerization)** 기술(Docker, Kubernetes)과 직결된다. 애플리케이션의 이식성이 높으면 멀티 클라우드 전략(Vendor Lock-in 회피)을 구현할 수 있다.
    *   **메트릭**: **MTTR (Mean Time To Repair)**은 유지보수성 지표이자, SRE(Site Reliability Engineering)의 **SLI (Service Level Indicator)**로 활용된다.

```ascii
[품질 특성 간 Trade-off 관계 다이어그램]

      (높음) ────────┐─────────────────────────
                  │ │
                  ▼ ▼
    [Performance
