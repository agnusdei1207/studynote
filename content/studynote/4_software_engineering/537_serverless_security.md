# 537. 서버리스(Serverless) 및 FaaS 보안 고려사항

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서버리스 (Serverless) 보안은 인프라 관리 책임이 CSP로 넘어간 환경에서, 실행 주체인 **개별 함수(Function)의 코드 무결성, 최소 권한 설정, 그리고 이벤트 트리거의 정당성을 보호**하는 애플리케이션 중심의 보안 체계다.
> 2. **가치**: 서버 관리 부담(Patching, Hardening)은 줄어들지만, 함수 단위로 파편화된 공격 표면(Attack Surface)과 짧은 실행 시간(Ephemeral)으로 인한 가시성 부족 문제를 해결하여 클라우드 네이티브 서비스의 안전성을 보장한다.
> 3. **융합**: RASP (Runtime Self-Protection), API 게이트웨이 보안, 그리고 서버리스 특화형 취약점 분석 도구와 결합되며, **OWASP Serverless Top 10** 가이드라인을 통해 아키텍처적 결함을 사전에 차단한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 서버리스 보안은 사용자가 서버를 직접 관리하지 않는 FaaS(Function as a Service, 예: AWS Lambda) 환경에서의 보안이다. 기존의 방화벽이나 백신 같은 서버 중심 보안이 아닌, "함수에 어떤 데이터가 들어오고, 그 함수가 어떤 권한을 행사하는가"에 집중한다.

- **💡 비유**: 서버리스 보안은 **"배달 전문 공유 주방(Serverless)의 위생 관리"**와 같습니다. 주방 시설(서버 인프라)은 건물주가 관리하지만, 내가 만드는 음식(코드)에 독이 들지 않았는지, 요리사가 허락 없이 다른 냉장고(데이터)를 열어보지는 않는지 감시하는 것과 같습니다.

- **등장 배경**:
  1. **보안 경계의 세분화**: 서버 한 대를 지키던 방식에서 수천 개의 '함수'를 각각 지켜야 하는 초미세 보안 시대로 진화했다.
  2. **이벤트 중심 공격**: HTTP 요청뿐만 아니라 S3 파일 업로드, DB 변경, 메시지 큐 등 다양한 이벤트가 함수를 실행시키면서 예측 불가능한 공격 경로가 늘어났다.
  3. **전통적 도구의 무력화**: 서버리스는 실행 시간이 매우 짧아(수 초 이내), 기존의 무거운 보안 에이전트를 설치하거나 네트워크 패킷을 전수 분석하기 어렵다.

- **📢 섹션 요약 비유**: 큰 성벽을 쌓는 대신, 성 안에서 움직이는 수만 명의 일꾼(함수) 하나하나에게 각기 다른 암호를 부여하고 일거수일투족을 감시하는 '정밀 현미경 보안' 체계입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### OWASP Serverless Top 10 주요 항목 및 대응

서버리스 환경에서 가장 흔히 발생하는 보안 위협들이다.

| 위험 항목 | 설명 | 대응 방안 | 비유 |
|:---|:---|:---|:---|
| **S1: Injection** | 이벤트 데이터에 악성 명령 삽입 | 입력값 검증, 라이브러리 보안 | 불량 식재료 유입 |
| **S2: Broken Auth** | 함수 간 호출 시 인증 누락 | IAM Role 기반의 상호 인증 | 가짜 주문서 승인 |
| **S3: Over-privilege** | 함수에 과도한 권한 부여 (Admin 등) | **함수당 1개의 전용 IAM Role** | 일꾼에게 마스터키 주기 |
| **S4: Resource Info Leak** | 환경 변수 등을 통한 정보 노출 | Secret Manager 연동 | 벽에 비번 적어두기 |
| **S5: Improper Logging** | 짧은 실행 시간으로 인한 로그 누락 | 실시간 로그 수집(CloudWatch 등) | 블랙박스 고장 |

### 서버리스 보안 제어 아키텍처

함수의 실행 전, 중, 후 전 과정에서의 보호 장치를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      서버리스(Lambda) 환경의 계층적 보안 통제 구조                │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [이벤트 소스] ──────▶ [API Gateway] ──────▶ [Lambda Function] ──────▶ [자원] │
 │ (S3, SQS, Web)         (WAF 적용)           (Execution)           (DB, S3)  │
 │      │                    │                    │                    │       │
 │  ┌───┴──────┐      ┌──────┴──────┐      ┌──────┴─────────┐    ┌─────┴─────┐ │
 │  │ 1. 입력  │      │ 2. 전방 방어 │      │ 3. 런타임 보호   │    │ 4. 최소   │ │
 │  │    검증  │      │    (Throttling)│      │    (RASP/Code) │    │    권한   │ │
 │  └──────────┘      └─────────────┘      └────────┬───────┘    └───────────┘ │
 │                                                  │                          │
 │                                                  ▼                          │
 │                    ┌──────────────────────────────────┐                     │
 │                    │      통합 로그 및 이상 탐지         │                     │
 │                    │  - Cold Start 지연 분석 (DoS)     │                     │
 │                    │  - 비정상적인 외부 아웃바운드 차단   │                     │
 │                    └──────────────────────────────────┘                     │
 │                                                                             │
 │  ※ 핵심: 함수는 상태가 없으므로(Stateless), 매번 권한을 명시적으로 검증함          │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 서버리스 보안은 **이벤트 흐름**을 따라간다. 1) **입구(API Gateway)**에서 WAF를 통해 부적절한 요청을 차단하고 트래픽을 제어한다. 2) **함수 내부**에서는 코드가 실행되는 동안 라이브러리 취약점을 감시하거나 비정상적인 시스템 호출을 막는 가벼운 보안 모듈(RASP)이 작동한다. 3) 가장 중요한 것은 **최소 권한(Least Privilege)**이다. 각 함수는 자신이 접근해야 할 특정 DB 테이블이나 S3 버킷에 대해서만 '딱 맞는' IAM 권한을 가져야 한다. 4) 마지막으로 서버리스 특유의 **Cold Start**를 이용한 DoS 공격(자원 고갈)을 막기 위해 실행 시간과 메모리 사용량을 모니터링한다.

---

### 심층 동작 원리: 함수당 권한 분리 (Per-Function IAM)

서버리스 보안의 성패를 좌우하는 핵심 원칙이다.
1. **문제**: 수십 개의 함수가 하나의 커다란 '관리자 권한'을 공유하면, 함수 하나만 뚫려도 클라우드 인프라 전체가 장악된다.
2. **해결**: 모든 함수(Lambda)마다 독립적인 IAM Role을 생성한다.
   - 예: '이미지 리사이징 함수'는 오직 '입력 버킷 읽기'와 '출력 버킷 쓰기' 권한만 가짐. DB 접근 권한은 절대 주지 않음.
3. **가치**: 공격자가 함수를 장악하더라도, 그 함수가 가진 좁은 권한 밖으로는 한 발짝도 나갈 수 없게 격리(Blast Radius 축소)한다.

- **📢 섹션 요약 비유**: 수만 명의 병사에게 전 군대를 지휘할 수 있는 무전기를 주는 것이 아니라, 자기 초소의 문만 열 수 있는 작은 열쇠 하나씩만 들려주는 철저한 분업 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 컨테이너(K8s) 보안 vs 서버리스(FaaS) 보안 비교

| 비교 항목 | 컨테이너 보안 | 서버리스 보안 |
|:---|:---|:---|
| **관리 주체** | 고객이 OS/Runtime 관리 | **CSP가 인프라/Runtime 관리** |
| **보안 포커스** | 이미지 스캔, 커널 감시 | **함수 코드, IAM 권한, 이벤트** |
| **지속성** | 상대적으로 길음 (수 분~수 일) | **매우 짧음 (수 초 단위)** |
| **주요 도구** | Prisma, Aqua, Falco | **Snyk, Checkov, CloudWatch** |
| **비유** | 아파트 단지 전체 보안 | 배달 상자 내용물 보안 |

---

### 기술 융합: FinOps와 서버리스 보안

서버리스 보안 사고는 곧바로 '비용 사고'로 이어진다.
- **Denial of Wallet (DoW)**: 공격자가 무한 루프를 유발하거나 대량의 이벤트를 발생시켜, 사용자에게 엄청난 클라우드 비용 폭탄을 안기는 공격.
- **융합**: 가용성 보안(Rate Limiting)이 곧 비용 절감 기술(FinOps)이 되는 지점이다.

- **📢 섹션 요약 비유**: 수도꼭지를 틀어놓고 도망가는 장난(DoW)을 막기 위해, 한 번에 나오는 물의 양을 제한하고 사용료가 급증하면 자동으로 밸브를 잠그는 스마트 계량기와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 서버리스 기반 '이미지 처리 파이프라인' 보안 강화

- **문제 상황**: 사용자가 이미지를 업로드하면 Lambda가 실행되는데, 파일명에 특수문자를 넣어 커맨드 인젝션을 시도하거나 10GB 이상의 대용량 파일을 올려 함수 비용을 폭증시키는 사례 발생.
- **의사결정 (서버리스 보안 전략)**:
  1. **Event Validation**: S3 이벤트가 발생할 때, Lambda 코드 첫 줄에서 파일 확장자와 크기를 체크하고 기준 미달 시 즉시 종료(Return)한다.
  2. **Environment Hardening**: 환경 변수에 DB 비밀번호를 평문으로 두지 않고, AWS Secrets Manager와 연동하여 런타임에만 가져오도록 수정한다.
  3. **Concurrency Limit**: 한꺼번에 수만 개의 함수가 뜨지 않도록 '예약된 동시성(Reserved Concurrency)' 설정을 통해 서비스 거부 공격을 예방한다.
  4. **Timeout Optimization**: 평균 처리 시간이 1초인 함수라면 타임아웃을 3초 이내로 짧게 설정하여, 공격 시도가 길게 이어지지 못하게 물리적으로 차단한다.

### 도입 체크리스트
- **Least Privilege**: 모든 함수가 자신만의 최소 권한 IAM Role을 가지고 있는가? (Wildcard `*` 사용 금지)
- **Vulnerable Libraries**: 서버리스 패키지 내에 포함된 오픈소스 라이브러리에 알려진 취약점이 없는가? (SCA 스캔)
- **API Gate**: 함수가 외부로 노출될 때 반드시 인증된 API 게이트웨이를 통하는가?

### 안티패턴
- **함수 내부 대기 (Sleep)**: 함수가 외부 응답을 기다리며 계속 켜져 있는 상태. 비용 효율과 보안(타임아웃 공격) 측면에서 최악이다. 비동기 메시지 큐(SQS)를 써야 한다.
- **공용 Role 사용**: "귀찮으니까" 모든 람다에 동일한 고권한 정책을 복사해서 붙여넣는 행위.

- **📢 섹션 요약 비유**: "누군지 모르겠지만 일단 일은 시켜야지"라는 안일함이, 내 집 안방(데이터)을 털어가는 강도를 초대하는 일이 될 수 있음을 잊지 말아야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Cost)** | 보안 사고 피해액 80% 감소 | 함수 단위 격리를 통해 사고 발생 시 다른 자원으로의 전파 차단 |
| **정량적 (Efficiency)** | 인프라 보안 관리 공수 제로 | OS 패치, 서버 하드닝 등 전통적 업무를 CSP에 이양하여 핵심 로직에 집중 |
| **정성적 (Standard)** | 클라우드 네이티브 신뢰 확보 | 서버리스 특화 보안 가이드 준수를 통해 현대적 아키텍처의 안정성 입증 |
| **정성적 (Agility)** | 비즈니스 속도 향상 | 보안이 코드와 함께 배포(Security as Code)되어 빠른 기능 릴리스 가능 |

### 미래 전망
- **AI-Managed Guardrails**: 인공지능이 함수의 평소 실행 패턴(메모리 사용, 호출 간격)을 학습하여, 미세한 로직 오작동이나 인젝션 시도를 실시간으로 교정할 것이다.
- **Hardware-level Isolation**: 더 강력한 보안을 위해 함수마다 하드웨어 수준에서 격리된 신뢰 실행 환경(TEE)을 제공하는 기술이 보편화될 것이다.

### 참고 표준
- **OWASP Serverless Top 10**: 서버리스 위협의 표준 준거
- **Cloud Security Alliance (CSA) - Serverless Framework**: 보안 아키텍처 가이드라인
- **NIST SP 800-210**: General Access Control Guidance for Cloud Systems

- **📢 섹션 요약 비유**: 예전에는 큰 공장(서버)을 지키는 경비원이었다면, 이제는 흩날리는 민들레 씨앗(함수) 하나하나가 안전하게 땅에 뿌리 내릴 수 있도록 지키는 '디지털 생태계 관리자'의 시대가 되었습니다.
