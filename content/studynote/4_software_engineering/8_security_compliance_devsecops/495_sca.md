# 495. SCA (Software Composition Analysis)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SCA (소프트웨어 구성 분석)는 애플리케이션에 포함된 오픈소스 라이브러리 및 서드파티 컴포넌트의 목록을 추출하여, **알려진 보안 취약점(CVE)과 오픈소스 라이선스 준수 여부를 자동으로 식별하고 관리**하는 보안 기술이다.
> 2. **가치**: 현대 소프트웨어의 80~90% 이상이 외부 라이브러리에 의존함에 따라 발생하는 '공급망 리스크'를 관리하는 핵심 수단이며, 개발자가 직접 작성하지 않은 코드에서 비롯되는 보안 구멍을 메우는 필수 도구다.
> 3. **융합**: SBOM (Software Bill of Materials) 생성 기술, CI/CD 파이프라인의 의존성 관리 도구와 결합되며, 단순 스캔을 넘어 '전이적 의존성 (Transitive Dependency)'까지 추적하여 가시성을 확보한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: SCA는 우리가 만드는 소프트웨어의 "식재료 성분 분석"과 같다. 개발자가 직접 짠 코드가 아닌, 외부에서 가져온 오픈소스(예: Log4j, Spring, jQuery 등)에 어떤 위험이 숨어 있는지 찾아낸다. 주로 사용 중인 라이브러리의 버전 정보를 추출하여 국가 취약점 데이터베이스(NVD)와 대조하는 방식으로 동작한다.

- **💡 비유**: SCA는 **"식당에서 음식을 팔기 전, 식재료를 납품한 업체들이 유통기한을 넘기지는 않았는지(취약점), 그리고 무단으로 상표를 도용한 식재료는 없는지(라이선스) 검사하는 영양사"**와 같습니다.

- **등장 배경**:
  1. **오픈소스 활용의 보편화**: 생산성 향상을 위해 오픈소스 사용이 필수화되었으나, 정작 어떤 라이브러리가 어디에 쓰이고 있는지 파악하는 가시성(Visibility) 문제가 발생했다.
  2. **공급망 공격 (Supply Chain Attack)의 증가**: 해커들이 기업을 직접 공격하는 대신, 기업이 신뢰하고 사용하는 유명 오픈소스를 오염시켜 수만 개의 고객사를 감염시키는 전략을 취하기 시작했다.
  3. **라이선스 법적 분쟁**: 무심코 사용한 오픈소스 라이선스(GPL 등) 위반으로 인해 기업의 소스 코드를 강제로 공개해야 하거나 거액의 배상금을 지불하는 리스크가 커졌다.

- **📢 섹션 요약 비유**: 내가 만든 요리뿐만 아니라, 시장에서 사 온 소스 속에 든 방부제(취약점)까지 꼼꼼히 따져봐야 안전한 음식이 된다는 원리와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SCA의 핵심 분석 대상 및 메커니즘

SCA는 단순히 파일 이름만 보는 것이 아니라 의존성 구조 전체를 분석한다.

| 분석 대상 | 설명 | 상세 내용 | 비유 |
|:---|:---|:---|:---|
| **직접 의존성** | 개발자가 명시적으로 선언한 모듈 | `pom.xml`, `package.json`에 정의된 라이브러리 | 내가 직접 고른 식재료 |
| **간접 의존성** | 라이브러리가 내부적으로 쓰는 모듈 | **Transitive Dependency** (의존성의 의존성) | 소스 속에 이미 섞여 들어온 첨가물 |
| **알려진 취약점 (CVE)** | 이미 공표된 보안 결함 | NVD, GitHub Advisory 등의 데이터베이스와 매핑 | 유통기한이 지난 식품 목록 대조 |
| **라이선스 (License)** | 오픈소스 사용 조건 | Apache, MIT, GPL, AGPL 등 법적 조건 식별 | 식재료 원산지 및 사용료 확인 |

---

### SCA 통합 보안 파이프라인 아키텍처

SCA가 개발 프로세스의 어느 지점에서 가치를 제공하는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      SCA 기반 소프트웨어 공급망 보안 관리 아키텍처                │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 개발 단계] ──────▶ [2. 빌드 단계] ──────▶ [3. 배포 단계] ──────▶ [4. 운영 단계] │
 │      │                    │                    │                    │       │
 │  의존성 추가         빌드 서버(Jenkins)      운영 환경(K8s)        상시 모니터링   │
 │ (IDE 플러그인)        (SCA 스캔 실행)        (검증된 이미지만)      (신규 CVE 감지) │
 │      │                    │                    │                    │       │
 │      └──────────┬─────────┴──────────┬─────────┴──────────┬─────────┘       │
 │                 ▼                    ▼                    ▼                 │
 │       ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐      │
 │       │   의존성 트리 분석  │  │   SBOM 자동 생성   │  │  라이선스 준수 체크  │      │
 │       │ (NVD DB 매핑)     │  │ (SPDX / CycloneDX)│  │ (Copyleft 방지)    │      │
 │       └──────────────────┘  └──────────────────┘  └──────────────────┘      │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SCA는 SDLC 전 과정에 걸쳐 동작한다. 1) **빌드 단계**에서 SCA 도구는 패키지 매니저의 설정 파일을 읽어 의존성 그래프를 그린다. 2) 이 그래프를 기반으로 **SBOM**을 생성하여 우리 소프트웨어의 유전 정보를 문서화한다. 3) 만약 치명적인 취약점이 발견되거나, 기업 정책에 어긋나는 독소 조항(GPL 등)을 가진 라이브러리가 포함되면 빌드를 중단(Build Break)시킨다. 4) **운영 단계**에서도 새로운 취약점이 매일 발표되므로, 이미 배포된 소프트웨어에 대해서도 지속적인 '재스캔'을 수행하여 제로데이 위협에 대응한다.

---

### 심층 동작 원리: 전이적 의존성 (Transitive Dependency) 분석

SCA의 진가는 눈에 보이지 않는 의존성을 찾아내는 데 있다.
1. **의존성 풀이 (Resolution)**: `npm install`이나 `mvn install` 시점에 다운로드되는 수백 개의 하위 라이브러리를 전수 조사한다.
2. **지문 인식 (Fingerprinting)**: 파일의 해시(Hash) 값을 계산하여, 공격자가 라이브러리 이름은 그대로 두고 내용물만 바꾼 '공급망 오염'을 감지한다.
3. **버전 권고**: "현재 1.2.3 버전을 쓰고 있는데, 1.2.5 버전에서 보안 패치가 완료되었습니다"라고 정확한 업데이트 경로를 제안한다.

- **📢 섹션 요약 비유**: 내가 고용한 직원(직접 의존성)뿐만 아니라, 그 직원이 데려온 아르바이트생(간접 의존성)까지 모두 신원 조회를 하는 철저한 관리 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SAST vs SCA 비교 (코드 보안 vs 구성 보안)

| 비교 항목 | SAST (정적 분석) | SCA (구성 분석) |
|:---|:---|:---|
| **분석 대상** | **내가 작성한 소스 코드** | **남이 작성한 라이브러리 (오픈소스)** |
| **핵심 기술** | 구문 분석, 데이터 흐름 추적 | 버전 대조, 해시 비교, 라이선스 식별 |
| **발견 항목** | 코딩 실수 (SQLi, XSS 등) | 알려진 취약점 (CVE), 라이선스 위반 |
| **수정 방법** | 코드 로직 직접 수정 | **라이브러리 버전 업데이트/교체** |
| **커버리지** | 내 코드 범위 내 | **전체 소프트웨어 구성 요소 (80% 이상)** |

---

### 융합 시너지: SCA와 SBOM (Software Bill of Materials)

SCA는 SBOM을 만드는 "공장"이다.
- **SBOM의 역할**: 소프트웨어의 자재 명세서를 투명하게 공개하여, 취약점 발생 시 "우리 제품의 어떤 모델에 이 문제가 있는가?"를 즉시 파악하게 한다.
- **법적 효력**: 미국 행정명령 등 글로벌 표준에서 정부 납품 SW에 대해 SBOM 제출을 의무화함에 따라 SCA의 중요성이 법적 준거성(Compliance) 영역으로 확장되었다.

- **📢 섹션 요약 비유**: 건강 검진(SCA) 결과를 건강 기록부(SBOM)로 남겨두어, 나중에 특정 약물(취약점) 부작용 뉴스가 나왔을 때 내가 위험한지 즉시 확인하는 체계와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 전사 오픈소스 라이선스 및 취약점 거버넌스 수립

- **문제 상황**: 개발자들이 무분별하게 오픈소스를 사용하여, 배포 직전 법무팀으로부터 라이선스 위반(GPL 포함) 경고를 받고 출시가 지연됨.
- **의사결정 (SCA 거버넌스 전략)**:
  1. **사전 승인 프로세스**: SCA 도구와 연동된 '오픈소스 승인 시스템'을 구축하여, 허용된 라이선스(Apache, MIT 등)만 자동으로 통과시킨다.
  2. **Vulnerability Gate**: CVSS 점수 7.0점 이상의 취약점이 포함된 오픈소스는 빌드 단계에서 원천 차단한다.
  3. **재고 관리 (Inventory)**: 전사적으로 어떤 라이브러리가 몇 개의 프로젝트에 쓰이고 있는지 실시간 대시보드를 운영한다. (Log4j 사태 같은 비상 상황 시 10분 내 영향도 파악 목표)
  4. **고지 의무 자동화**: 제품 내에 포함되어야 하는 '오픈소스 고지문'을 SCA 분석 결과로부터 자동 생성하여 배포 프로세스에 포함한다.

### 도입 체크리스트
- **Database Freshness**: 도구가 참조하는 취약점 DB가 매일 업데이트되는가?
- **Transitive Depth**: 의존성의 끝단(마지막 자식 노드)까지 완벽하게 분석하는가?
- **Binary Scan**: 소스 코드 없이 바이너리(jar, war, container image)만으로도 분석이 가능한가?

### 안티패턴
- **버전 미지정 (Latest 맹신)**: 라이브러리 버전을 고정하지 않고 `latest`를 쓰는 행위. 빌드할 때마다 새로운 취약점이 유입될 수 있다.
- **예외 처리 남발**: 조치가 귀찮아서 "이건 영향 없음"으로 무분별하게 무시(Ignore) 버튼을 누르는 행위.

- **📢 섹션 요약 비유**: "유명한 가게 물건이니까 믿어도 되겠지"라는 안일함이 독이 되어 돌아오는 것을 방지하기 위해, 브랜드 명성보다 성분표(SCA)를 믿어야 한다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Time)** | 사고 대응 속도 10배 향상 | 신규 취약점 발표 시 자산 영향도 분석 시간 획기적 단축 |
| **정량적 (Legal)** | 라이선스 소송 리스크 0% | 카피레프트(Copyleft) 라이선스 유입 차단으로 법적 분쟁 예방 |
| **정성적 (Standard)** | 공급망 신뢰성 확보 | 고객사 및 정부 기관에 검증된 SBOM을 제공하여 사업 경쟁력 강화 |
| **정성적 (Efficiency)** | 불필요한 라이브러리 제거 | 중복되거나 쓰이지 않는 노후 컴포넌트 정리로 애플리케이션 경량화 |

### 미래 전망
- **AI-Based Remediation**: AI가 취약한 라이브러리를 발견하면, 단순히 경고하는 것을 넘어 소스 코드와 호환되는 최적의 패치 버전을 찾아 자동으로 코드를 수정해줄 것이다.
- **VEX (Vulnerability Exploitability eXchange)**: 단순히 취약점이 있다는 것을 넘어, "우리 제품 구조상 이 취약점이 실제로 공격 가능한 경로에 있는가?"를 분석하여 불필요한 패치 피로도를 줄이는 기술이 도입될 것이다.

### 참고 표준
- **NIST SP 800-161**: Supply Chain Risk Management (공급망 위험 관리 가이드)
- **OpenChain (ISO/IEC 5230)**: 오픈소스 라이선스 준수 표준
- **SPDX / CycloneDX**: SBOM 전송 및 공유를 위한 글로벌 데이터 표준

- **📢 섹션 요약 비유**: 과거에는 내가 잘하는 것에만 집중했다면, 이제는 내가 믿고 거래하는 모든 파트너(오픈소스)의 투명성까지 책임져야 생존할 수 있는 '공동체 보안'의 시대로 나아가고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 레고로 멋진 로봇을 만들 때, **내가 직접 만든 블록(내 코드)뿐만 아니라 친구에게 빌려온 블록(오픈소스) 중에 부서진 게 없는지 검사하는 도구**예요.
2. 빌려온 블록이 알고 보니 나쁜 사람이 독을 발라놓은 블록일 수도 있으니까, 매일매일 명단을 확인해서 나쁜 블록은 새 블록으로 바꿔줘야 해요.
3. 이렇게 하면 우리가 만든 로봇이 갑자기 고장 나지 않고, 친구들에게 아주 안전하게 선물할 수 있게 된답니다!
