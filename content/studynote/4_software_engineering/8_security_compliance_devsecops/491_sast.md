# 491. SAST (Static Application Security Testing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SAST (정적 애플리케이션 보안 테스팅)는 소프트웨어를 실행하지 않고 소스 코드, 바이트 코드, 또는 바이너리를 분석하여 **보안 취약점의 근본 원인이 되는 보안 약점(CWE 등)을 식별하는 화이트박스(White-box) 테스트 기법**이다.
> 2. **가치**: SDLC의 최상단인 구현(Coding) 단계에서 취약점을 발견하는 **시프트 레프트 (Shift-Left)** 보안의 핵심 도구로, 개발자가 코드를 작성하는 즉시 피드백을 제공하여 결함 수정 비용을 최소화한다.
> 3. **융합**: IDE 플러그인, CI/CD 파이프라인, 그리고 최근에는 LLM (Large Language Model) 기반의 자동 수정(Auto-remediation) 기술과 결합되어, 단순 탐지를 넘어 보안 품질 관리(Quality Gate)의 표준으로 자리 잡고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: SAST는 "정적(Static)"이라는 말 그대로 프로그램을 구동하지 않은 상태에서 소스 코드의 구문(Syntax), 구조(Structure), 데이터 흐름(Data Flow)을 전수 조사한다. 코드 내에 숨겨진 하드코딩된 암호, SQL 인젝션 취약점, 버퍼 오버플로우 가능성 등을 룰셋(Rule-set) 기반으로 찾아내는 기술이다.

- **💡 비유**: SAST는 **"건물을 짓기 전, 설계도와 자재 명세서를 꼼꼼히 훑어보며 철근이 빠지지는 않았는지, 화재에 취약한 재료를 쓰지는 않았는지 검사하는 정밀 검수"**와 같습니다.

- **등장 배경**:
  1. **사후 보안의 고비용 구조**: 테스팅이나 운영 단계에서 발견된 보안 결함은 아키텍처 수정을 동반하는 경우가 많아 해결 비용이 기하급수적으로 상승한다.
  2. **코딩 표준의 복잡성 증가**: 수천 명의 개발자가 협업하는 대규모 프로젝트에서 일관된 시큐어 코딩(Secure Coding) 수준을 유지하기 위해 자동화된 검증 도구가 필수적이 되었다.
  3. **컴플라이언스 대응**: 행정안전부의 '소프트웨어 개발 보안 가이드' 등 법적 규제 준수를 입증하기 위한 객관적 증빙 자료로 활용된다.

- **📢 섹션 요약 비유**: 수험생이 시험을 다 치고 채점을 기다리는 것이 아니라, 문제를 푸는 도중에 옆에서 선생님이 오답을 바로바로 지적해주는 효율적인 학습 도우미와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SAST의 주요 분석 기술 및 메커니즘

SAST는 단순한 텍스트 검색을 넘어 고차원적인 분석 기법을 동원한다.

| 분석 기법 | 설명 | 핵심 가치 | 비유 |
|:---|:---|:---|:---|
| **구문 분석 (Semantic)** | 코드의 의미와 문맥을 파악 | 단순 오타가 아닌 로직 결함 식별 | 문장의 맥락 파악 |
| **데이터 흐름 분석 (Data Flow)** | 외부 입력값이 민감한 함수에 도달하는 경로 추적 | SQLi, XSS 유발 경로 탐지 | 오염된 물의 흐름 추적 |
| **제어 흐름 분석 (Control Flow)** | 프로그램의 실행 실행 경로(If, Loop 등) 분석 | 실행되지 않는 코드(Dead Code)나 논리적 허점 탐지 | 미로의 모든 통로 확인 |
| **오염 분석 (Taint Analysis)** | 신뢰할 수 없는 데이터(Tainted)의 전파 확인 | 검증되지 않은 입력값의 오남용 방지 | 전염병 감염자 경로 추적 |

---

### SAST 기반 DevSecOps 통합 아키텍처

SAST가 현대적 개발 프로세스에 어떻게 녹아드는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      SAST 기반 지속적 보안 통합(Continuous Security) 아키텍처     │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. IDE 단계] ──▶ [2. Commit 단계] ──▶ [3. Build/Scan 단계] ──▶ [4. Review 단계] │
 │      │                 │                    │                    │          │
 │  플러그인 실행    Pre-commit Hook       전수 스캔 실행        취약점 확인/수정    │
 │ (실시간 가이드)    (보안 규칙 위반 차단)    (CWE/CVE 매핑)        (Build Fail?)     │
 │                                                                             │
 │           ┌───────────────────────────────────────────────────────┐         │
 │           ▼                                                       ▼         │
 │    [보안 룰셋 저장소]                                       [중앙 대시보드]     │
 │    - 행안부 47개 항목                                       - 부서별 보안 지수   │
 │    - OWASP Top 10                                          - 취약점 조치 현황   │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SAST는 개발자의 로컬 환경(IDE)부터 중앙 빌드 서버(CI/CD)까지 촘촘하게 배치된다. 1) **IDE 단계**에서는 소나린트(SonarLint) 같은 도구가 코딩과 동시에 실시간 수정을 제안한다. 2) **CI/CD 단계**에서는 소나큐브(SonarQube)나 체크마르크스(Checkmarx) 같은 엔터프라이즈 도구가 전체 코드를 분석한다. 만약 심각한(Critical) 보안 약점이 발견되면 **Quality Gate**가 작동하여 빌드를 실패시키고, 취약점이 운영 환경으로 넘어가는 것을 기술적으로 차단한다. 모든 결과는 중앙 대시보드에 기록되어 전사 보안 성숙도 지표로 관리된다.

---

### 심층 동작 원리: 데이터 흐름 분석 (Data Flow Analysis)

인젝션 취약점을 찾는 핵심 알고리즘이다.
1. **Source (입력 지점)**: 외부 사용자가 값을 입력하는 함수(예: `request.getParameter()`).
2. **Sink (위험 지점)**: 데이터가 실행되거나 저장되는 함수(예: `executeQuery()`).
3. **Taint Analysis**: Source에서 들어온 데이터가 Sink에 도달할 때까지 중간에 '검증 함수(Validator/Sanitizer)'를 거치는지 추적한다.
4. **결과**: 거치지 않고 도달하면 취약점으로 리포팅한다.

- **📢 섹션 요약 비유**: 오염된 원수가 정수기(검증 함수)를 거치지 않고 그대로 수도꼭지(Sink)로 나오는 배관 구조를 찾아내어 경고를 울리는 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SAST vs DAST (정적 vs 동적) 비교 분석

| 비교 항목 | SAST (정적 분석) | DAST (동적 분석) |
|:---|:---|:---|
| **분석 시점** | **개발 초기 (SDLC 좌측)** | **테스트/운영 (SDLC 우측)** |
| **접근 방식** | 화이트박스 (코드 내부 투시) | 블랙박스 (외부 공격 시뮬레이션) |
| **발견 대상** | 보안 약점 (근본 원인) | 보안 취약점 (실제 사고 지점) |
| **장점** | 100% 코드 커버리지, 수정 용이성 | 오탐(False Positive) 낮음, 설정 오류 발견 |
| **단점** | 높은 오탐률, 런타임 환경 반영 불가 | 코드의 모든 경로 분석 불가, 늦은 발견 |

---

### 융합 시너지: SCA (Software Composition Analysis)와의 결합

SAST는 내가 짠 코드를 검사하고, SCA는 가져다 쓴 코드(오픈소스)를 검사한다.
- **ASOC (Application Security Orchestration & Correlation)**: SAST, DAST, SCA의 결과를 하나로 묶어 중복을 제거하고 실제 위험도가 가장 높은 것부터 정렬해주는 통합 관리 체계로 진화하고 있다.

- **📢 섹션 요약 비유**: 내가 직접 쓴 보고서의 오타(SAST)를 고치는 것과, 보고서에 인용된 외부 자료의 출처와 신뢰성(SCA)을 확인하는 작업을 병행하여 완벽한 문서를 만드는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 신규 금융 프로젝트의 'Security Gate' 도입

- **문제 상황**: 수백 명의 개발자가 동시에 코딩하는 상황에서 보안 점검 인력이 부족하여 배포 전 보안 심사가 병목 현상을 일으킴.
- **의사결정 (SAST 자동화 전략)**:
  1. **자동화**: Jenkins 파이프라인에 SAST 스캔 단계를 필수 삽입한다.
  2. **차등 정책**: 신규 개발(Feature) 브랜치는 모든 보안 약점 수정 시에만 Merge를 허용하고, 기존 레거시 코드는 위험도 '중' 이상의 항목부터 단계적으로 조치한다.
  3. **개발자 중심**: 보안팀이 일방적으로 결과를 던져주는 것이 아니라, 개발자가 IDE 내에서 직접 결과를 확인하고 자가 조치할 수 있는 환경(Self-Service Security)을 구축한다.
  4. **오탐 관리**: 비즈니스 로직상 안전한 패턴은 'False Positive'로 예외 처리하여 개발자의 불필요한 피로도를 줄인다.

### 도입 체크리스트
- **Rule Customization**: 우리 프로젝트의 프레임워크(Spring, Django 등)에 최적화된 룰셋을 보유하고 있는가?
- **Integration**: 빌드 시스템 및 티켓 관리 시스템(Jira)과 유기적으로 연동되는가?
- **Speed**: 전체 빌드 시간을 저해하지 않을 정도의 빠른 스캔 속도를 보장하는가? (Incremental Scan 지원 여부)

### 안티패턴
- **도구 만능주의**: SAST가 모든 취약점을 다 잡아줄 것이라는 맹신. 로직 우회나 권한 관리 결함은 SAST로 잡기 어렵다.
- **방치된 리포트**: 수만 건의 경고가 담긴 리포트만 생성하고 아무도 조치하지 않는 '보고서를 위한 보안'.

- **📢 섹션 요약 비유**: 최신식 거짓말 탐지기(SAST)를 도입했어도, 질문이 잘못되었거나(룰셋 미비) 결과를 해석할 사람이 없다면 범인(취약점)을 잡을 수 없다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Cost)** | 결함 수정 비용 80% 절감 | 운영 단계 발견 대비 구현 단계 조치 시 비용의 획기적 절감 |
| **정량적 (Quality)** | 코드 품질 향상 | 보안뿐만 아니라 코드의 복잡도, 중복성 등 일반 품질 지표 동시 개선 |
| **정성적 (Culture)** | 보안 내재화 (Security Culture) | 개발자가 코딩 과정에서 자연스럽게 보안 지식을 학습하고 적용 |
| **정성적 (Trust)** | 대외 신뢰도 및 규제 준수 | "보안성이 검증된 소프트웨어"라는 브랜드 이미지 구축 |

### 미래 전망
- **AI-Native SAST**: 인공지능이 코드의 맥락을 완벽히 이해하여 오탐(False Positive)을 획기적으로 줄이고, 취약점 발견 시 안전한 코드로의 리팩토링 안을 즉시 제시할 것이다.
- **IAST로의 전이**: 정적 분석의 깊이와 동적 분석의 정확도를 결합한 IAST(상호작용 테스팅)가 SAST의 많은 부분을 대체하며 실시간 보안 시대를 열 것이다.

### 참고 표준
- **CWE (Common Weakness Enumeration)**: SAST 도구의 표준 분석 항목
- **OWASP ASVS**: 애플리케이션 보안 검증 표준 (정적 분석 요건 포함)
- **KISA 소프트웨어 개발 보안 가이드**: 국내 공공/민간 보안 준수 표준

- **📢 섹션 요약 비유**: 수동으로 현미경을 들여다보던 시대에서, 인공지능 로봇이 24시간 실시간으로 미세한 결함을 찾아내고 스스로 고치기까지 하는 '자율 주행형 보안 공정'으로 진화하고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **DAST (Dynamic Application Security Testing)**: SAST와 상호 보완적인 동적 분석 기술
- **CWE (Common Weakness Enumeration)**: SAST가 찾아내는 약점들의 표준 사전
- **Secure Coding**: SAST를 통해 달성하고자 하는 궁극적인 코딩 목표
- **DevSecOps**: SAST가 자동화되어 통합되는 현대적 운영 체계
