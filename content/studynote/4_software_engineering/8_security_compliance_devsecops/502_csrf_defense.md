# 502. 크로스 사이트 요청 위조 (CSRF) 방어 - 토큰 및 SameSite

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CSRF (크로스 사이트 요청 위조) 방어는 로그인된 사용자의 브라우저가 본인의 의사와 상관없이 악성 사이트의 유도로 인해 **신뢰하는 서버에 위조된 요청을 보내는 것을 차단**하는 보안 메커니즘이다.
> 2. **가치**: 사용자의 세션 권한을 도용하여 발생하는 무단 송금, 비밀번호 변경, 정보 삭제 등의 '행위 위조' 사고를 원천 봉쇄하며, 특히 'Anti-CSRF 토큰'은 서버와 클라이언트 간의 고유 비밀 값을 검증함으로써 요청의 진위성을 보장한다.
> 3. **융합**: SameSite 쿠키 속성을 통한 브라우저 차원의 자동 방어와, 서버 측의 Referer/Origin 체크 및 토큰 검증이 결합된 심층 방어(Defense in Depth) 체계 구축이 현대 웹 보안의 표준이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: CSRF 공격은 사용자의 "세션 쿠키"가 브라우저에 의해 자동으로 전송된다는 특징을 악용한다. 방어의 핵심은 브라우저가 자동으로 보내는 정보(쿠키) 외에, **사용자가 직접 의도하여 보낸 것임을 증명하는 추가적인 데이터(토큰)**를 요구하는 것이다.

- **💡 비유**: CSRF 방어는 **"은행 업무를 볼 때, 통장과 도장(세션 쿠키) 외에도 본인만 알고 있는 1회용 비밀번호(Anti-CSRF 토큰)나 지문 인식(SameSite 설정)을 한 번 더 요구하여, 다른 사람이 내 손을 빌려 몰래 서명하는 것을 막는 것"**과 같습니다.

- **등장 배경**:
  1. **쿠키 기반 인증의 한계**: 브라우저는 특정 도메인으로 요청을 보낼 때 해당 도메인의 쿠키를 무조건 함께 보내는데, 이를 이용해 공격자가 피해자의 권한으로 서버 명령을 내릴 수 있게 되었다.
  2. **소셜 엔지니어링의 정교화**: 피싱 이메일이나 악성 광고를 통해 사용자가 공격 코드가 담긴 페이지를 방문하게 하는 기법이 교묘해졌다.
  3. **브라우저 정책의 변화**: 과거에는 서버가 모든 방어를 책임졌으나, 최근에는 SameSite 같은 표준 속성을 통해 브라우저가 능동적으로 위조 요청을 거부하는 추세다.

- **📢 섹션 요약 비유**: 대문에 도어락(인증)이 있어도, 도둑이 집주인의 팔을 뒤에서 붙잡고 억지로 문을 열게 하는 상황(CSRF)을 방지하기 위해, 집주인만 아는 암구호(토큰)를 추가로 확인하는 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CSRF 3대 방어 메커니즘

| 방어 기법 | 적용 위치 | 핵심 원리 | 비유 |
|:---|:---:|:---|:---|
| **Anti-CSRF 토큰** | 서버/클라이언트 | 매 요청마다 고유한 랜덤 토큰값을 비교 검증 | 매번 바뀌는 비밀번호 |
| **SameSite 쿠키** | 브라우저 | 타 사이트에서 발생하는 요청에 쿠키 포함 여부 결정 | 집 밖으로 유출되지 않는 신분증 |
| **Referer / Origin 체크** | 서버 | 요청이 시작된 이전 페이지 주소를 확인 | "어디서 오셨나요?" 질문 |

---

### Anti-CSRF 토큰 동작 아키텍처

서버와 클라이언트가 비밀 토큰을 공유하고 검증하는 전 과정을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Anti-CSRF 토큰 기반 요청 검증 워크플로우                    │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 페이지 요청]                                [2. 정상적인 폼 제출]           │
 │  사용자 ──────▶ 서버                             사용자 ──────▶ 서버             │
 │         (GET)                                           (POST)              │
 │                                                                             │
 │  ◀─(Form + Token)─┘                             ┌───(Data + Token)───┘      │
 │  (서버는 세션에 저장)                             │ (서버는 세션 토큰과 비교)     │
 │                                                 │                           │
 │                                                 ▼ [서버 측 로직]              │
 │ [3. CSRF 공격 시도]                              If (Req.Token == Session.Token) │
 │  악성 사이트 ──▶ 서버                            { 실행 승인! }                │
 │         (POST)                                   Else                           │
 │                                                  { 요청 거부 (403 Forbidden) }  │
 │  ◀──(Data Only)───┘                             │                           │
 │  (토큰이 없으므로 실패!)                           └───────────────────────────┘  │
 │                                                                             │
 │  ※ 핵심: 공격자는 사용자의 세션 내 토큰 값을 읽어올 수 없음 (SOP 정책)             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 1) 사용자가 게시판 글쓰기 페이지를 요청하면, 서버는 랜덤하게 생성된 **Anti-CSRF 토큰**을 세션에 저장하고 페이지의 숨겨진 필드(`<input type="hidden">`)에 담아 보낸다. 2) 정상적인 사용자가 '저장' 버튼을 누르면 데이터와 함께 이 토큰이 서버로 전달되고, 서버는 세션의 값과 일치하는지 확인한다. 3) 공격자가 가짜 페이지를 통해 전송을 유도하면, 공격자는 사용자의 토큰 값을 알 수 없으므로(동일 출처 정책, SOP에 의해 차단됨) 토큰 없이 요청을 보낼 수밖에 없고, 서버는 이를 즉시 거부한다.

---

### 심층 동작 원리: SameSite 쿠키 속성 (Lax, Strict, None)

브라우저 차원에서 CSRF를 막는 가장 현대적이고 편리한 방법이다.
1. **SameSite=Strict**: 오직 현재 사이트 내에서의 이동(First-party)일 때만 쿠키를 보낸다. 가장 안전하지만 링크 클릭 이동 시 로그인이 풀리는 등 사용자 불편이 크다.
2. **SameSite=Lax (기본값)**: 다른 사이트에서 오는 링크 클릭(GET)은 허용하되, 상태를 변경하는 요청(POST 등)에는 쿠키를 보내지 않는다. 편의성과 보안의 타협점이다.
3. **SameSite=None**: 모든 사이트에서 쿠키를 보낸다. 반드시 `Secure` 속성과 함께 사용해야 하며, 사실상 CSRF 방어를 포기하는 설정이다.

- **📢 섹션 요약 비유**: 신분증(쿠키)에 특수한 칩을 달아, 우리 동네(도메인) 안에서만 작동하고 동네 밖으로 나가면 자동으로 무효화되는 지능형 신분증 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### CSRF 방어 기법 간 장단점 비교

| 기법 | 장점 | 단점 | 권장 상황 |
|:---|:---|:---|:---|
| **CSRF Token** | **가장 확실한 방어**, 상태 변경 차단 | 서버 세션 부하 증가, API 연동 복잡 | 모든 중요 상태 변경 요청 |
| **SameSite Cookie** | 코드 수정 불필요 (설정만으로 가능) | 구형 브라우저 미지원, Lax의 한계 | 기본 보안 설정으로 상시 적용 |
| **Referer 체크** | 구현이 매우 간단함 | Referer 변조 가능성, 개인정보 정책으로 누락 가능 | 보조 수단으로 활용 |
| **Double Submit Cookie** | 무상태(Stateless) 서버에 적합 | 쿠키 변조 공격에 취약할 수 있음 | REST API, MSA 환경 |

---

### 융합 시너지: REST API와 CSRF

현대의 Single Page Application (SPA) 환경에서는 전통적인 세션 토큰 방식이 어려울 수 있다.
- **Custom Header**: 요청 헤더에 `X-CSRF-TOKEN` 같은 사용자 정의 헤더를 추가한다. 브라우저의 스크립트 권한 없이는 타 사이트에서 커스텀 헤더를 보낼 수 없으므로 강력한 방어가 된다.
- **Bearer Token (JWT)**: 쿠키를 쓰지 않고 로컬 스토리지에 토큰을 저장하여 Authorization 헤더로 보내면, 브라우저가 자동 전송을 하지 않으므로 CSRF 위협에서 완전히 벗어날 수 있다. (단, XSS 위험은 증가함)

- **📢 섹션 요약 비유**: 통장 대신 현금(JWT)을 들고 다니는 것과 같습니다. 도둑이 내 도장을 뺏어갈 걱정(CSRF)은 없지만, 대신 현금을 직접 소매치기(XSS)당하지 않게 주머니를 잘 단속해야 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 인터넷 뱅킹의 이중 CSRF 방어 체계 구축

- **문제 상황**: SameSite=Lax 설정만으로는 GET 방식의 정보 유출이나 정교한 우회 공격을 완벽히 막기 어려움.
- **의사결정 (심층 방어 전략)**:
  1. **토큰 기반 방어**: 모든 송금 및 개인정보 변경 요청(POST/PUT/DELETE)에 대해 세션 당 고유한 Anti-CSRF 토큰 검증 로직을 구현한다. (Spring Security의 CSRF Filter 활용)
  2. **SameSite=Strict 강제**: 로그인이 완료된 직후 발급되는 중요 세션 쿠키에는 `SameSite=Strict`를 부여하여 외부 링크 유입을 차단한다.
  3. **재인증 (Step-up Auth)**: 실제 자금 이체 시점에는 CSRF 방어와 별개로 추가 비밀번호나 생체 인증을 요구하여 '행위의 의도성'을 최종 확인한다.
  4. **Referer 검증**: 서버 측 필터에서 `Referer` 헤더가 우리 은행 도메인이 아닌 경우 요청을 즉시 드랍(Drop) 처리한다.

### 도입 체크리스트
- **Method Restriction**: 조회(GET) 요청에서 데이터 상태를 변경(DB Update 등)하는 코드가 없는가?
- **Token Uniqueness**: 토큰이 추측하기 어려운 무작위 값이며, 매 세션 또는 매 요청마다 갱신되는가?
- **Logout Validation**: 로그아웃 시 서버의 CSRF 토큰 정보도 확실히 파기되는가?

### 안티패턴
- **GET 방식 송금**: `bank.com/transfer?to=hacker&amount=100` 식의 설계. <img> 태그 하나만으로도 공격이 성공하는 최악의 설계다.
- **CSRF 토큰을 쿠키로만 전송**: 토큰 값을 쿠키에만 담아 보내고 서버 세션과 대조하지 않는 행위. 공격자가 쿠키를 조작할 수 있다면 무용지물이다.

- **📢 섹션 요약 비유**: 현관문 비밀번호만 믿고 문을 열어두는 것이 아니라, 문을 열 때마다 집 안의 스피커로 "정말 당신 맞습니까?"라고 물어보고 대답(토큰)을 들어야 한다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 위조 요청 사고 100% 방지 | 토큰 및 SameSite 적용 시 기술적 CSRF 공격은 불가능 |
| **정량적 (Compliance)** | 법적 책임 완수 | 금융감독원 및 정보보호 준거성 가이드의 CSRF 방어 요건 충족 |
| **정성적 (Trust)** | 서비스 무결성 유지 | 사용자 모르게 정보가 변경되는 사고를 막아 고객 신뢰도 제고 |
| **정성적 (Stability)** | 비정상 트래픽 감소 | 악성 봇이나 스크립트에 의한 무분별한 요청 처리 비용 절감 |

### 미래 전망
- **Cookie-less Authentication**: 쿠키의 자동 전송 특성 자체가 보안의 취약점으로 지목됨에 따라, 쿠키를 아예 쓰지 않는 WebAuthn 기반의 인증 체계가 주류가 될 것이다.
- **Browser-Managed Authorization**: 브라우저가 사이트 간의 요청 성격을 스스로 판단하여, 민감한 요청일 경우 사용자에게 명시적인 승인 팝업을 띄우는 지능형 방어 기능이 강화될 것이다.

### 참고 표준
- **CWE-352**: Cross-Site Request Forgery (CSRF)
- **RFC 6265bis**: SameSite 쿠키 속성 명세
- **OWASP CSRF Prevention Cheat Sheet**: 분야별 상세 방어 전략

- **📢 섹션 요약 비유**: 수동으로 도장을 찍던 시대에서, 디지털 서명과 지문 인식이 결합된 '부인 방지형 스마트 계약'의 시대로 웹 보안이 진화하고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 나쁜 사람이 내 손을 몰래 잡고 **엄마한테 "내 장난감 친구 줄게"라고 억지로 편지를 쓰게 하는 것**과 같아요.
2. CSRF 방어는 엄마가 그 편지를 받았을 때, **"진짜 네가 쓴 거 맞니? 너만 아는 암호를 말해봐!"라고 한 번 더 물어보는 것**과 같아요.
3. 우리만 아는 암호(토큰)를 꼭 확인하면, 나쁜 사람의 거짓말에 엄마가 속지 않는답니다!
