# 498. 입력 데이터 검증 및 표현 (Input Validation) 원칙

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 입력 데이터 검증 (Input Validation)은 애플리케이션으로 유입되는 모든 외부 데이터를 신뢰할 수 없는 것으로 간주하고, 이를 사용하기 전 **정의된 형식, 길이, 타입, 범위에 부합하는지 확인하여 악성 코드의 실행을 원천 차단**하는 보안 원칙이다.
> 2. **가치**: SQL 인젝션, XSS, 경로 조작 등 주요 보안 취약점의 80% 이상이 부적절한 입력값 처리에서 비롯되므로, 가장 경제적이고 효과적인 1차 방어선 역할을 수행하며 데이터 무결성을 보장한다.
> 3. **융합**: '화이트리스트(Allow-list)' 방식의 검증과 '출력 인코딩(Output Encoding)' 기술이 결합되어 '입력부터 출력까지'의 안전한 데이터 수명 주기를 완성하며, 현대 아키텍처에서는 중앙 집중식 검증 라이브러리와 프레임워크 내장 기능을 통해 구현된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 입력 데이터 검증은 "Don't Trust, Verify" 철학의 실천이다. 사용자로부터 입력받는 폼(Form), URL 파라미터, HTTP 헤더, 심지어 DB에서 가져온 데이터까지 시스템 경계를 넘나드는 모든 데이터를 검사 대상으로 삼는다. 단순히 '나쁜 문자를 빼는 것'이 아니라 '정상적인 데이터만 통과시키는 것'이 핵심이다.

- **💡 비유**: 입력 데이터 검증은 **"공항 입국 심사대"**와 같습니다. 유효한 여권(형식)과 비자(권한)가 있는 사람만 통과시키고, 위험한 물건(악성 코드)을 소지했는지 엑스레이(필터링)로 검사하여 나라 안(시스템)의 안전을 지키는 것과 같습니다.

- **등장 배경**:
  1. **데이터와 명령의 혼동**: 컴퓨터는 입력받은 데이터 속에 숨겨진 특수문자(`, <, >, ;)를 데이터가 아닌 명령어로 오인하여 실행함으로써 사고가 발생한다.
  2. **사용자 환경의 다양화**: 브라우저뿐만 아니라 모바일, IoT 기기 등 다양한 경로로 데이터가 입력되면서, 클라이언트 단의 검증만으로는 서버 보안을 담보할 수 없게 되었다.
  3. **공격 기법의 고도화**: 우회 기법(Encoding 트릭 등)이 발전함에 따라, 단순한 블랙리스트 기반 필터링은 더 이상 유효하지 않게 되었다.

- **📢 섹션 요약 비유**: 현관문 앞에 튼튼한 체(Filter)를 쳐두어, 우리가 먹어도 되는 깨끗한 물(정상 데이터)만 집 안으로 들어오게 하고 불순물(악성 코드)은 걸러내는 정수 시스템과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 입력 데이터 검증의 3대 핵심 메커니즘

| 단계 | 명칭 | 상세 동작 | 비유 |
|:---|:---|:---|:---|
| **1. 구문 검증 (Syntactic)** | 형식 및 타입 체크 | 숫자인지, 날짜 형식인지, 길이는 적당한지 확인 | 옷차림과 가방 크기 검사 |
| **2. 의미 검증 (Semantic)** | 비즈니스 로직 체크 | 나이가 0~150 사이인지, 계좌 잔액보다 출금액이 적은지 확인 | 여행 목적과 체류 기간 확인 |
| **3. 정제 (Sanitization)** | 위험 요소 제거/치환 | 특수문자 제거, HTML 엔티티 치환, 이스케이프 처리 | 반입 금지 물품 압수 |

---

### 안전한 데이터 처리 흐름 아키텍처 (End-to-End)

입력 시점의 검증과 출력 시점의 표현을 결합한 다중 방어 체계를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      입력 검증 및 출력 인코딩 통합 아키텍처                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [외부 세계] ──────(Data)─────▶ [애플리케이션] ──────(Query)────▶ [데이터베이스] │
 │                                     │                                       │
 │  1. 클라이언트 검증 (UX용)            │  3. 보안 함수 실행                      │
 │     - JS 체크                       │     - Prepared Statement              │
 │                                     │     - SQL 파라미터 바인딩               │
 │  2. 서버 측 입력 검증 (보안용)         │                                       │
 │     - 화이트리스트 필터링 ◀──────────┘  4. 출력 인코딩 (표현 단계)              │
 │     - 타입/길이/범위 체크                   - HTML Entity 변환 (XSS 방지)     │
 │                                           - Content-Security-Policy 적용    │
 │                                                                             │
 │  ※ 원칙: 클라이언트는 믿지 마라 (Server-side Validation is Mandatory)           │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 안전한 데이터 처리는 4단계 레이어로 구성된다. 1) **클라이언트 검증**은 사용자 편의를 위한 것이며 보안 수단이 될 수 없다. 2) 핵심은 **서버 측 입력 검증**이다. 들어온 데이터가 '숫자 10자리'라는 규칙에 맞지 않으면 즉시 거부한다. 3) 검증된 데이터라도 DB에 보낼 때는 **Prepared Statement**를 사용하여 데이터와 쿼리 구조를 완전히 분리한다. 4) 마지막으로 데이터를 사용자 화면에 다시 뿌려줄 때는 **출력 인코딩**을 거쳐, 데이터 내부에 포함되었을지 모를 스크립트가 실행되지 않도록 문자로 치환(` < `를 ` &lt; `로) 한다.

---

### 심층 동작 원리: 화이트리스트 vs 블랙리스트

1. **블랙리스트 (Black-list)**: "나쁜 것들(예: `SELECT`, `<script>`)만 막자."
   - **문제점**: 해커는 `S-E-L-E-C-T`나 인코딩된 문자로 우회 가능. 항목이 너무 많아 관리가 불가능함.
2. **화이트리스트 (White-list)**: "허용된 것(예: 숫자, 한글, 영문 일부) 외에는 다 막자."
   - **장점**: 알려지지 않은 공격 기법도 원천 차단 가능. 가장 강력한 보안 원칙.

- **📢 섹션 요약 비유**: 모르는 사람을 일일이 기억해서 못 들어오게 막는 것(블랙리스트)보다, 미리 약속된 초대장을 가진 사람만 들여보내는 것(화이트리스트)이 훨씬 완벽한 방범인 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 입력 검증 (Validation) vs 출력 인코딩 (Encoding) vs 이스케이핑 (Escaping)

| 항목 | 사용 시점 | 주요 목적 | 대응 취약점 |
|:---|:---|:---|:---|
| **입력 검증** | 데이터가 들어올 때 | 데이터 무결성 및 형식 보장 | SQLi, 경로 조작, DoS |
| **출력 인코딩** | 데이터가 화면에 나갈 때 | 브라우저 인터프리터 오작동 방지 | **XSS (Cross-Site Scripting)** |
| **이스케이핑** | 데이터가 외부 툴로 갈 때 | 특수문자의 의미 무력화 (명령어 오인 방지) | SQLi, OS Command Injection |

---

### 기술 융합: 정규 표현식 (Regex)과 보안

정밀한 입력 검증을 위해 정규 표현식이 널리 사용되지만, 잘못 쓰면 보안 위협이 된다.
- **가치**: `^[a-zA-Z0-9+_.-]+@[a-zA-Z0-9.-]+$` 와 같이 복잡한 이메일 형식을 한 줄의 코드로 완벽하게 검증한다.
- **위협 (ReDoS)**: 복잡한 정규 표현식에 특정 패턴의 문자열을 입력하여 서버의 CPU를 고갈시키는 '정규 표현식 서비스 거부 공격'에 유의해야 한다.

- **📢 섹션 요약 비유**: 범인을 잡기 위해 몽타주(정규표현식)를 그리는 것과 같습니다. 너무 자세하면 좋지만, 몽타주가 너무 복잡하면 수사관이 지쳐 쓰러질(ReDoS) 수도 있다는 뜻입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 게시판 '파일 업로드' 기능의 보안 강화

- **문제 상황**: 사용자가 게시글에 사진을 올리는 기능에서, 해커가 `.php`나 `.jsp` 같은 실행 파일을 올려 서버를 장악하려는 시도가 예상됨.
- **의사결정 (입력 검증 원칙 적용)**:
  1. **확장자 화이트리스트**: `.jpg`, `.png`, `.gif` 외의 모든 확장자는 업로드를 거부한다. (블랙리스트 방식인 '`.php`만 막기'는 금지)
  2. **MIME 타입 검증**: 파일 확장자뿐만 아니라 실제 파일의 헤더(Magic Byte)를 읽어 실제 이미지 파일인지 재검증한다.
  3. **파일명 정제 (Sanitization)**: 파일명에 포함된 특수문자나 `../` 같은 경로 조작 문자를 제거하고, 서버 측에서 생성한 UUID로 파일명을 변경하여 저장한다.
  4. **실행 권한 제거**: 업로드된 폴더에서는 어떠한 스크립트도 실행되지 않도록 웹 서버(Nginx/Apache) 설정을 하드닝한다.

### 도입 체크리스트
- **Centralized Logic**: 검증 로직이 각 페이지에 흩어져 있지 않고 공통 라이브러리나 인터셉터에서 수행되는가?
- **Server-side Mandatory**: 클라이언트 측 검증(JS)과 별개로 서버 측 검증이 반드시 중복 수행되는가?
- **Fail-safe**: 검증 로직에서 오류 발생 시 해당 데이터를 신뢰하지 않고 '거부' 처리하는가?

### 안티패턴
- **필터링 후 재사용**: 나쁜 문자를 지우기만 하고(Replacing) 다시 사용하는 행위. 공격자는 `scr<script>ipt` 처럼 필터링 후 다시 유효한 문장이 되게 만들어 우회한다. (거부가 원칙)
- **Null Byte 혼동**: 문자열 끝을 의미하는 `%00`을 이용해 검증을 우회하는 기법에 대비하지 않는 경우.

- **📢 섹션 요약 비유**: 상한 음식을 보고 상한 부분만 도려내서 먹으려고(필터링) 애쓰기보다는, 조금이라도 의심스러우면 아예 식탁에 올리지 않는 것(거부)이 식중독을 예방하는 최선의 길입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 인젝션 공격 99% 차단 | 화이트리스트 검증과 파라미터 바인딩 결합 시 기술적 공격 불가능 |
| **정량적 (Quality)** | 데이터 쓰레기 값 유입 방지 | 정제된 데이터만 DB에 저장되어 시스템 성능 및 분석 정확도 향상 |
| **정성적 (Standard)** | 시큐어 코딩 준거성 확보 | 행안부 가이드 및 OWASP 표준을 충족하여 대외 보안 인증 획득 |
| **정성적 (Stability)** | 예외 상황 최소화 | 비정상 입력으로 인한 애플리케이션 크래시(Crash) 방지 |

### 미래 전망
- **Context-Aware Validation**: AI가 데이터의 문맥을 이해하여, 단순 형식을 넘어 비즈니스 로직상의 이상 징후(예: 평소보다 100배 큰 금액 입력)를 실시간으로 판단하는 지능형 검증이 보편화될 것이다.
- **Standardized Validation Libraries**: 개발자가 직접 정규식을 짜는 대신, 전 세계적으로 검증된 표준 보안 검증 라이브러리(ESAPI 등)를 연동하는 방식이 표준으로 자리 잡을 것이다.

### 참고 표준
- **OWASP ASVS (Application Security Verification Standard)**: 입력 검증 상세 요건 포함
- **CWE-20**: Improper Input Validation (가장 광범위한 보안 약점)
- **KISA 소프트웨어 개발 보안 가이드**: 입력 데이터 검증 및 표현 분야 15개 항목

- **📢 섹션 요약 비유**: 지금까지는 사람이 일일이 가방을 열어봤다면, 이제는 인공지능 엑스레이가 물건의 용도까지 파악해서 위험을 걸러내는 '스마트 검역 시대'로 나아가고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 컴퓨터 프로그램은 **"낯선 사람이 주는 사탕(데이터)"을 함부로 먹으면 배탈(해킹)이 날 수 있어요.**
2. 그래서 사탕을 먹기 전에 "이게 진짜 사탕 맞는지", "위험한 게 들어있지는 않은지" 꼼꼼하게 검사하는 규칙을 정해놓아야 해요.
3. 확인되지 않은 수상한 것은 "안 먹을래요!"라고 단호하게 거절하는 것이 우리 컴퓨터를 건강하게 지키는 가장 중요한 약속이랍니다!
