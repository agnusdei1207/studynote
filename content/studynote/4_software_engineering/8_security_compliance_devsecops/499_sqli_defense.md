# 499. SQL 인젝션 방어 - Prepared Statement, ORM 프레임워크 사용

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SQL 인젝션 방어의 핵심은 사용자 입력값을 SQL 쿼리의 구조를 결정하는 '명령어'가 아닌 단순한 '데이터'로 처리하도록 **쿼리와 데이터를 완전히 분리**하는 데 있다.
> 2. **가치**: Prepared Statement (파라미터화된 쿼리)는 쿼리 템플릿을 미리 컴파일하여 실행 계획을 고정함으로써 악의적인 SQL 구문 삽입을 원천 차단하며, ORM 프레임워크는 이러한 메커니즘을 내재화하여 개발자의 실수를 방지한다.
> 3. **융합**: 시큐어 코딩의 핵심 기술이며, DB 계정의 '최소 권한 원칙'과 결합하여 공격 성공 시의 피해 범위를 최소화하는 다층 방어(Defense in Depth) 전략의 필수 요소다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: SQL 인젝션 방어는 데이터베이스로 전달되는 쿼리문의 구조를 외부 사용자가 임의로 조작하지 못하게 막는 모든 기법을 말한다. 가장 강력한 방법은 **Prepared Statement**를 사용하여 쿼리의 틀(Template)을 미리 만들고, 입력값은 나중에 따로 끼워 넣는(Binding) 방식이다.

- **💡 비유**: SQL 인젝션 방어는 **"입력 칸이 정해진 양식지(Form)"**와 같습니다. 빈 종이에 아무 글자나 적게 하는 것이 아니라, '이름' 칸에는 이름만, '나이' 칸에는 숫자만 적게 하여 문장의 전체 의미(쿼리 구조)를 바꿀 수 없도록 칸막이를 쳐두는 것과 같습니다.

- **등장 배경**:
  1. **문자열 결합 방식의 위험성**: 과거 `"... WHERE id = '" + id + "'"` 식의 동적 쿼리 생성은 인젝션 공격에 무방비로 노출되었다.
  2. **DB 성능 최적화 요구**: 동일한 쿼리를 반복 실행할 때 매번 파싱(Parsing)하는 오버헤드를 줄이기 위해 'Pre-compilation' 기술이 도입되었고, 이것이 보안적으로도 완벽한 해결책임이 밝혀졌다.
  3. **생산성 향상**: 현대적인 웹 개발 환경에서 복잡한 SQL을 직접 작성하기보다 객체 중심의 ORM (Object-Relational Mapping)을 사용하면서 보안이 자연스럽게 내재화되었다.

- **📢 섹션 요약 비유**: 도둑이 열쇠 구멍에 본드를 발라 자물쇠를 망가뜨리지 못하도록, 아예 열쇠 구멍이 없는 스마트 도어락(쿼리 고정) 시스템으로 교체하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SQL 인젝션 방어 기법 비교

| 기법 | 설명 | 주요 특징 | 보안성 |
|:---|:---|:---|:---:|
| **Prepared Statement** | 쿼리 구조를 선(先) 컴파일 후 데이터 바인딩 | **가장 강력하고 권장되는 방법** | 상(High) |
| **ORM 프레임워크** | 객체와 DB를 매핑하여 자동으로 쿼리 생성 | JPA, Hibernate 등 활용, 내부적으로 PS 사용 | 상(High) |
| **Stored Procedure** | DB 내부에 미리 정의된 프로시저 호출 | 사용 시 주의 필요 (내부에서 동적 SQL 금지) | 중(Mid) |
| **Input Validation** | 입력값에서 특수문자 제거/치환 (Escaping) | 블랙리스트 방식의 한계로 보조 수단으로만 사용 | 하(Low) |

---

### Prepared Statement 동작 아키텍처

쿼리 구조와 데이터가 처리되는 시점을 분리하여 보안성을 확보하는 메커니즘을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Prepared Statement vs 동적 쿼리 실행 흐름 비교               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 동적 쿼리 (Vulnerable)]                                                   │
 │    "SELECT * FROM users WHERE id = '" + input + "'"                         │
 │    ▶ Parse ──▶ Compile ──▶ Plan ──▶ Execute (사용자 입력이 쿼리 구조를 바꿈)      │
 │                                                                             │
 │ [2. Prepared Statement (Secure)]                                            │
 │    "SELECT * FROM users WHERE id = ?"  <── 템플릿 선(先) 전송                   │
 │    ▶ 1단계: DB 엔진에서 쿼리 구조 Parse & Compile (실행 계획 고정)               │
 │    ▶ 2단계: 애플리케이션에서 input 데이터를 ? 자리에 Binding                    │
 │    ▶ 3단계: Execute (데이터는 절대 명령어로 해석되지 않음)                        │
 │                                                                             │
 │  ┌──────────────────┐          ┌──────────────────┐          ┌────────────┐ │
 │  │  App Server     │ ──(SQL)──▶ │   DB Engine      │ ──▶      │ Storage    │ │
 │  │ (Pre-compiled)  │ ──(Data)─▶ │ (Frozen Plan)    │          │            │ │
 │  └──────────────────┘          └──────────────────┘          └────────────┘ │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Prepared Statement의 핵심 보안 원리는 **'쿼리 분석 단계의 분리'**다. DB 엔진은 `?`가 포함된 쿼리를 먼저 받아 문법을 해석하고 어떻게 데이터를 찾을지 계획(Plan)을 세운다. 이 계획이 확정된 후에는 아무리 이상한 값(`' OR 1=1 --`)이 들어와도 DB 엔진은 이를 "검색할 글자"로만 취급한다. 이미 쿼리의 구조가 굳어졌기 때문에 새로운 조건절이 추가되거나 주석 처리가 되는 등의 변조가 불가능해지는 것이다.

---

### 심층 동작 원리: ORM (Object-Relational Mapping)의 보안성

개발자가 SQL을 직접 만지지 않고 객체를 조작하면 ORM이 대신 안전한 쿼리를 만든다.
1. **추상화**: `userRepository.findById(id)`를 호출하면 프레임워크가 내부적으로 `SELECT ... WHERE id = ?` 쿼리를 생성한다.
2. **자동 바인딩**: 모든 파라미터는 자동으로 Prepared Statement 방식으로 바인딩되어 전달된다.
3. **주의사항**: `query.nativeQuery("... " + input)`와 같이 ORM 내에서 직접 문자열을 결합하는 'Native Query' 기능을 쓰면 보안 효과가 사라지므로 주의해야 한다.

- **📢 섹션 요약 비유**: 수동 변속기 차량(직접 SQL)은 운전자의 미숙함으로 사고가 날 수 있지만, 자동 변속기 차량(ORM)은 시스템이 안전한 변속 시점(쿼리 생성)을 알아서 제어해주는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 기술적 방어 vs 관리적 방어 시너지

| 구분 | 방어 수단 | 시너지 효과 |
|:---|:---|:---|
| **기술적 (Primary)** | **Prepared Statement** | 기술적 공격 시도 자체를 무력화 |
| **기술적 (Secondary)** | **WAF (웹 방화벽)** | 알려진 공격 패턴을 입구에서 차단 (심층 방어) |
| **관리적 (Principle)** | **최소 권한 계정 (DB)** | 공격 성공 시에도 테이블 삭제나 권한 탈취 방지 |
| **관리적 (Principle)** | **에러 메시지 마스킹** | 공격자에게 DB 구조 정보를 주지 않음 |

---

### 융합 시너지: JPA와 데이터 접근 제어

현대적인 프레임워크는 단순히 인젝션을 막는 것을 넘어, 접근 제어와 결합된다.
- **예시**: Spring Data JPA의 `@Query` 어노테이션 사용 시 사용자의 ID를 세션에서 가져와 쿼리에 강제 바인딩함으로써, 다른 사용자의 정보를 조회하는 **IDOR (접근 제어 실패)** 취약점까지 동시에 방어할 수 있다.

- **📢 섹션 요약 비유**: 방탄복(Prepared Statement)만 입는 것이 아니라, 애초에 위험한 구역(관리자 권한)에 접근하지 못하도록 통제(최소 권한)하는 다중 보안의 힘입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 레거시 시스템의 '안전한 MyBatis' 전환 전략

- **문제 상황**: 수만 개의 XML 매퍼 파일에 `${parameter}` (직접 주입) 방식이 혼용되어 인젝션 위험이 산재함.
- **의사결정 (보안 가이드 적용)**:
  1. **표기법 강제**: MyBatis에서 직접 값을 치환하는 `${ }` 대신, Prepared Statement를 사용하는 `#{ }` 표기법으로 전수 교체한다.
  2. **정적 분석 연동**: CI/CD 파이프라인에 SAST 도구를 넣어 XML 파일 내에 `${ }`가 사용된 코드가 있으면 빌드를 중단시킨다.
  3. **예외 케이스 처리**: `ORDER BY` 절이나 테이블명 등 어쩔 수 없이 `${ }`를 써야 하는 경우는 코드 내에서 화이트리스트(Allow-list) 기반의 별도 검증 로직을 구현하도록 한다.
  4. **DB 로그 감시**: 실행되는 모든 쿼리 로그를 SIEM으로 전송하여, 비정상적인 대량 조회나 에러 발생 패턴을 모니터링한다.

### 도입 체크리스트
- **Parameter Binding**: 소스 코드 전체에서 SQL 문자열 더하기(`+`, `concat`)가 제거되었는가?
- **Least Privilege**: 애플리케이션용 DB 계정이 `DROP`, `ALTER`, `GRANT` 권한을 가지고 있지는 않은가?
- **Safe ORM Usage**: ORM을 쓰더라도 동적 쿼리 생성 부분에서 취약한 코딩 패턴이 없는가?

### 안티패턴
- **Stored Procedure 내 동적 SQL**: 프로시저 호출 자체는 안전하지만, 프로시저 내부에서 문자열을 더해 `EXECUTE` 하는 코드는 인젝션의 온상이다.
- **블랙리스트 기반 필터링**: `'`, `--`, `;` 등만 막으려 애쓰는 행위. 우회 경로가 무수히 많으므로 절대 권장하지 않는다.

- **📢 섹션 요약 비유**: "독버섯만 골라내서 요리할게요"라고 고집(블랙리스트) 부리는 것보다, 그냥 시장에서 산 검증된 느타리버섯(Prepared Statement)만 쓰는 것이 가장 현명한 요리사의 선택입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | SQLi 사고율 0% 수렴 | Prepared Statement 전수 적용 시 기술적 인젝션 공격은 불가능 |
| **정량적 (Performance)** | 쿼리 실행 속도 최적화 | 실행 계획 재사용을 통해 초당 처리 건수(TPS) 향상 효과 |
| **정성적 (Standard)** | 표준 개발 가이드 확립 | 신입 개발자도 안전하게 코딩할 수 있는 시스템적 환경 제공 |
| **정성적 (Audit)** | 컴플라이언스 완벽 대응 | 행안부 및 금융보안원 점검 항목의 '데이터 검증' 요건 충족 |

### 미래 전망
- **AI-Native Secure ORM**: 인공지능이 개발자의 의도를 파악하여, 가장 효율적이면서도 보안상 완벽한 쿼리 코드를 자동으로 생성해주는 기술이 보편화될 것이다.
- **NoSQL Injection 방어 표준화**: RDBMS뿐만 아니라 MongoDB 등 NoSQL 환경에서도 파라미터화된 쿼리와 유사한 보안 표준 API 사용이 강제될 것이다.

### 참고 표준
- **CWE-89**: Improper Neutralization of Special Elements used in an SQL Command
- **OWASP Top 10 - A03:2021**: Injection 카테고리
- **행정안전부 SW 개발 보안 가이드**: 입력 데이터 검증 및 표현 분야 준수 사항

- **📢 섹션 요약 비유**: 수동으로 빗장을 걸던 과거의 보안에서, 지문 인식 하나로 모든 권한과 데이터가 안전하게 통제되는 '디지털 잠금장치'의 표준으로 진화했습니다.
