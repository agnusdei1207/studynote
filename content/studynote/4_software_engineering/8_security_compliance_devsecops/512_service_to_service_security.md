# 512. 마이크로서비스 간 보안 (Service-to-Service) 및 mTLS

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스 간 보안은 신뢰할 수 없는 내부 네트워크 환경(Zero Trust)에서 서비스들이 서로의 신원을 확인하고 통신 데이터를 보호하기 위해 적용하는 보안 체계로, **mTLS (상호 TLS 인증)**가 그 핵심 기술이다.
> 2. **가치**: 기존의 단방향 TLS가 서버만 검증하는 것과 달리, mTLS는 클라이언트와 서버가 **서로의 디지털 인증서를 교환하여 쌍방향으로 검증**함으로써 내부 침입자에 의한 위장 공격(Spoofing)과 도청을 원천 차단한다.
> 3. **융합**: 서비스 메시(Istio, Linkerd) 인프라를 통해 자동화된 인증서 발급 및 갱신(Certificate Rotation)이 수행되며, SPIFFE 표준과 결합되어 플랫폼에 독립적인 서비스 신원 체계를 구축한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 서비스 간 보안은 "누가 나를 부르는가?"와 "내가 부르는 상대가 진짜인가?"를 확인하는 것이다. **mTLS (Mutual TLS)**는 이를 위해 클라이언트와 서버가 모두 각자의 인증서를 제시하고 상대방을 검사하는 프로토콜이다.

- **💡 비유**: mTLS는 **"두 스파이가 암호를 주고받는 접선 현장"**과 같습니다. 한쪽만 암호를 대는 게 아니라, 서로 "나 무궁화(서버)다", "나 소나무(클라이언트)다"라고 증표를 보여주고 확인이 끝난 뒤에야 비밀 대화(암호화 통신)를 시작하는 것과 같습니다.

- **등장 배경**:
  1. **경계 보안의 붕괴**: 과거에는 내부망은 안전하다고 믿었으나, 이제는 내부망 어디에든 해커가 있을 수 있다는 '제로 트러스트' 사상이 지배적이 되었다.
  2. **MSA의 복잡한 통신**: 수백 개의 서비스가 서로 얽혀 호출하는 구조에서, IP 주소만으로 상대를 신뢰하기에는 보안 리스크가 너무 커졌다.
  3. **데이터 규제 강화**: 금융이나 공공 분야에서는 내부 구간 통신도 의무적으로 암호화해야 하는 규정이 강화되었다.

- **📢 섹션 요약 비유**: 성문 입구에서만 검사하는 것이 아니라, 성 안의 모든 방에 들어갈 때마다 서로 얼굴(인증서)을 확인하고 대화하는 철저한 신원 확인 체계입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 단방향 TLS vs 상호 TLS (mTLS) 비교

| 구분 | 단방향 TLS (일반 웹) | 상호 TLS (mTLS) |
|:---|:---|:---|
| **검증 주체** | **클라이언트**만 서버를 검증 | **클라이언트와 서버가 서로** 검증 |
| **인증서 소지** | 서버만 인증서 소유 | **둘 다 개별 인증서 소유** |
| **보안 강도** | 보통 (사용자 편의 중심) | **매우 높음** (신뢰 관계 중심) |
| **비유** | 손님이 가게 간판 확인 | 수사관들이 서로 신분증 제시 |

---

### 2. mTLS 기반 서비스 간 통신 아키텍처

인증서 발급부터 세션 형성까지의 전 과정을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Service-to-Service mTLS 통신 메커니즘                        │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [서비스 A (Client)]                                     [서비스 B (Server)]   │
 │  ┌──────────────┐                                       ┌──────────────┐    │
 │  │ Cert_A / PK_A│                                       │ Cert_B / PK_B│    │
 │  └──────┬───────┘                                       └──────▲───────┘    │
 │         │                                                      │            │
 │         │ ────(1. Client Hello + Cert_A)─────────────────────▶ │            │
 │         │                                                      │            │
 │         │ ◀───(2. Server Hello + Cert_B + Request Cert_A) ───  │            │
 │         │                                                      │            │
 │         │ ────(3. Verify Cert_B & Send Signed Data)──────────▶ │            │
 │         │                                                      │            │
 │         │ ◀───(4. Verify Cert_A & Establish Secure Session)──  │            │
 │         ▼                                                      ▼            │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 자동화 인프라 (Service Mesh - Istio)                                  │  │
 │  ├───────────────────────────────────────────────────────────────────────┤  │
 │  │ - CA (Citadel): 각 파드(Pod)에 단기 인증서 자동 발급 및 배포               │  │
 │  │ - Rotation: 탈취 위험을 막기 위해 수 시간 단위로 인증서 자동 갱신           │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** mTLS의 핵심은 **디지털 인증서**를 통한 쌍방향 신뢰 구축이다. 1) 서비스 A가 B에게 통신을 요청할 때 자신의 신분증(Cert_A)을 보낸다. 2) 서비스 B도 자신의 신분증(Cert_B)을 주면서 상대방의 것을 확인한다. 3) 양쪽 모두 신뢰할 수 있는 기관(CA)에서 발급된 진짜 신분증임을 확인하면, 그때 비로소 데이터를 주고받을 암호키를 생성한다. 실무에서는 이 복잡한 인증서 관리를 개발자가 직접 하지 않고, **Istio** 같은 서비스 메시가 자동으로 수행해준다.

---

### 심층 동작 원리: SPIFFE와 서비스 신원 (Identity)

플랫폼에 종속되지 않는 일관된 서비스 이름을 부여하는 기술이다.
1. **SPIFFE ID**: `spiffe://cluster.local/ns/default/sa/order-service`와 같이 서비스의 정체성을 URI 형태로 정의한다.
2. **SVID (Document)**: 이 ID 정보를 담은 X.509 인증서.
3. **가치**: 서비스가 AWS에 있든, 온프레미스에 있든 상관없이 **"이 서비스가 주문 서비스가 맞다"**는 사실을 표준화된 방식으로 증명할 수 있게 한다.

- **📢 섹션 요약 비유**: 이사를 가거나 옷을 갈아입어도 변하지 않는 고유한 주민등록번호(SPIFFE ID)를 서비스에게 부여하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 서비스 간 보안의 3대 요소

| 요소 | 기술 및 기법 | 설명 |
|:---|:---|:---|
| **신원 (Identity)** | SPIFFE / X.509 Cert | "너는 누구인가?"를 기술적으로 정의 |
| **기밀성 (Conf.)** | TLS Encryption | 통신 내용을 암호화하여 도청 방지 |
| **인가 (Authz)** | RBAC / ABAC | "누구인지 알았으니, 어떤 행동을 허용할지" 결정 |

---

### 기술 융합: JWT와 mTLS의 하이브리드 보안

사용자의 신원과 서비스의 신원을 동시에 보장한다.
- **Outer Security (JWT)**: 사용자가 누구인지에 대한 정보를 담고 있음.
- **Inner Security (mTLS)**: 통신하는 시스템들이 신뢰할 수 있는 장비인지 보장함.
- **시너지**: "믿을 수 있는 서버(mTLS)가, 정당한 권한을 가진 사용자(JWT)의 요청을 처리함"을 이중으로 보증한다.

- **📢 섹션 요약 비유**: 비행기를 탈 때 비행기(서버)가 안전한지 확인(mTLS)하고, 승객(사용자)이 티켓(JWT)을 가지고 있는지 확인하는 이중 보안 검사대와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 금융 클라우드의 내부 망 전 구간 mTLS 적용

- **문제 상황**: 금감원 보안 가이드에 따라 내부 서비스 간 통신도 모두 암호화해야 함. 수백 개의 자바/파이썬 서비스에 일일이 인증서 로직을 넣기엔 공수가 너무 큼.
- **의사결정 (보안 전략 수립)**:
  1. **Infrastructure Level**: 서비스 코드 수정 없이 보안을 적용하기 위해 **Service Mesh (Istio)**를 전사 도입한다.
  2. **Sidecar Proxy**: 모든 서비스 옆에 **Envoy 프록시**를 배치하여, 실제 mTLS 협상과 암호화는 프록시가 전담하게 한다.
  3. **Strict Mode**: `PeerAuthentication` 설정을 **STRICT**로 두어, 평문(HTTP) 통신 시도를 원천 차단하고 오직 mTLS 통신만 수용한다.
  4. **가시성 확보**: Kiali 대시보드를 통해 어느 서비스 간에 mTLS가 활성화되어 있는지 시각적으로 모니터링한다.

### 도입 체크리스트
- **Certificate Expiry**: 인증서 유효 기간이 만료되기 전 자동으로 갱신되는 프로세스가 완벽히 작동하는가?
- **Performance Impact**: 암호화 및 복호화 연산으로 인해 서비스 지연(Latency)이 비즈니스 허용 범위 내에 있는가?
- **CA Security**: 인증서를 발급해주는 루트 CA의 개인키가 하드웨어 보안 모듈(HSM) 등에 안전하게 보관되고 있는가?

### 안티패턴
- **Permissive Mode 방치**: 도입 초기에는 편리하지만, 나중에 보안을 강화하지 않고 평문 통신을 계속 허용해두는 상태.
- **정적 인증서 사용**: 유효 기간 10년짜리 인증서를 서비스에 박아두는 행위. 유출 시 대처가 불가능하다. (짧은 주기의 자동 갱신 권장)

- **📢 섹션 요약 비유**: 현관문에 지문 인식기를 달아놓고 귀찮다고 문을 열어두는(Permissive Mode) 것은 보안 장비를 안 단 것보다 더 위험할 수 있습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 내부 위협 차단율 100% | 인증되지 않은 서비스의 모든 접근을 네트워크 레벨에서 거부 |
| **정량적 (Compliance)** | 법적 규제 준수 증명 | 전 구간 암호화 및 신원 기반 감사 로그를 통해 심사 통과 |
| **정성적 (Standard)** | 서비스 신원 체계 통일 | 플랫폼 종류와 상관없이 일관된 신뢰 모델(SPIFFE) 확보 |
| **정성적 (Agility)** | 개발팀 보안 부담 제로 | 인프라에서 보안을 처리하므로 개발자는 비즈니스 로직에만 집중 |

### 미래 전망
- **Hardware-accelerated mTLS**: CPU의 전용 명령어나 스마트 NIC을 통해 암호화 성능 저하가 제로에 가까운 mTLS 환경이 구현될 것이다.
- **Multi-mesh Federation**: 서로 다른 기업이나 다른 클라우드에 있는 서비스 메시들끼리 신뢰를 맺어, 전 세계적인 거대 제로 트러스트 망이 형성될 것이다.

### 참고 표준
- **SPIFFE (Secure Production Identity Framework for Everyone)**: 서비스 신원 표준
- **NIST SP 800-204A**: Building Secure Microservices (mTLS 강조)
- **RFC 5246 / 8446**: TLS 1.2 / 1.3 표준 명세

- **📢 섹션 요약 비유**: "누구도 믿지 않음으로써 모두를 지킨다." 서비스 간 보안은 마이크로서비스라는 거대한 오케스트라가 불협화음 없이 안전하게 연주될 수 있게 하는 보이지 않는 지휘봉입니다.
