# 474. 위협 모델링 (Threat Modeling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 위협 모델링 (Threat Modeling)은 시스템의 아키텍처를 분석하여 잠재적인 보안 위협을 식별하고, 각 위협의 위험도를 산정한 후 적절한 대응 방안을 결정하는 **구조적 보안 설계 기법**이다.
> 2. **가치**: 구현(Coding) 이전에 설계 결함을 발견함으로써 보안 사고 예방 비용을 최소화하며, "무엇을 보호해야 하는가(자산)", "누가 공격하는가(위협원)", "어떻게 공격하는가(공격 경로)"에 대한 명확한 청사진을 제공한다.
> 3. **융합**: STRIDE 모델, DFD (Data Flow Diagram) 등과 결합되어 Secure SDLC의 설계 단계를 주도하며, 최근에는 클라우드 네이티브 환경의 복잡한 마이크로서비스 간 신뢰 경계(Trust Boundary) 분석을 위해 필수적으로 적용되고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 위협 모델링은 공격자의 관점에서 시스템을 바라보고, 발생 가능한 부정적인 시나리오를 체계적으로 나열하여 방어 전략을 수립하는 과정이다. 이는 단순한 취약점 스캔이 아니라, 시스템의 비즈니스 로직과 데이터 흐름 속에 숨겨진 논리적 허점을 찾아내는 데 초점을 맞춘다.

- **💡 비유**: 위협 모델링은 **건물을 짓기 전 설계도를 보며 도둑이 들어올 수 있는 경로(창문, 환풍구, 지하 주차장 연결로)를 미리 찾아내어 잠금장치를 계획하는 보안 시뮬레이션**과 같습니다.

- **등장 배경**:
  1. **설계 결함의 치명성**: 코딩 오류보다 아키텍처 설계 결함으로 인한 보안 사고가 더 파괴적이며 해결 비용도 막대하다.
  2. **복잡해지는 인프라**: 클라우드, API, 외부 서드파티 연동 등 접점이 늘어나면서 직관적인 판단만으로는 모든 보안 위협을 감지하기 어려워졌다.
  3. **규제 준수**: 금융 및 공공 분야의 주요 시스템 구축 시 보안성 검토의 핵심 항목으로 위협 모델링 수행이 필수화되고 있다.

- **📢 섹션 요약 비유**: 전쟁터에 나가기 전 지도를 펼쳐 놓고 적군이 매복할 만한 곳과 우리 군의 취약한 보급로를 분석하여 작전 계획을 짜는 것과 같습니다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 위협 모델링 4단계 프로세스 (Adam Shostack)

위협 모델링의 대가 아담 쇼스탁이 제안한 네 가지 핵심 질문을 중심으로 프로세스가 진행된다.

| 단계 | 핵심 질문 | 주요 활동 | 비유 |
|:---|:---|:---|:---|
| **1. 모델링 (Model)** | 무엇을 만들고 있는가? | DFD (Data Flow Diagram) 작성, 신뢰 경계 설정 | 설계도 그리기 |
| **2. 식별 (Identify)** | 무엇이 잘못될 수 있는가? | STRIDE 분석, 위협 라이브러리 활용 | 범죄 시나리오 작성 |
| **3. 대응 (Mitigate)** | 위협에 대해 무엇을 할 것인가? | 제거, 전가, 완화, 수용 중 대응 방안 결정 | 방범 장치 설치 |
| **4. 검증 (Validate)** | 제대로 수행했는가? | 모델의 정확성 검토, 조치 사항 확인 | 최종 점검 |

---

### 위협 모델링의 핵심 도구: DFD (Data Flow Diagram) 및 신뢰 경계

위협 모델링은 시스템을 '데이터의 흐름' 관점에서 추상화하여 분석한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      위협 모델링 분석 아키텍처 (DFD 예시)                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │    [사용자 영역]              [DMZ / 서비스 영역]           [DB / 내부 영역]     │
 │  ┌────────────┐           ┌──────────────────┐           ┌────────────┐     │
 │  │   사용자   │           │   웹 서버 (Web)  │           │   데이터   │     │
 │  │ (브라우저)  │ ──(ID)──▶ │  (비즈니스 로직) │ ──(SQL)─▶ │   베이스   │     │
 │  └────────────┘           └──────────────────┘           └────────────┘     │
 │           │                        │                           │            │
 │           └─────── (Boundary 1) ───┴─────── (Boundary 2) ──────┘            │
 │                                                                             │
 │  [신뢰 경계 분석]                                                            │
 │  - Boundary 1: 인터넷 평면 vs 내부망 (인증 필수 지점)                          │
 │  - Boundary 2: 서비스망 vs 데이터 저장망 (접근 제어 지점)                      │
 │                                                                             │
 │  [STRIDE 매핑 예시]                                                          │
 │  - ID 전송 구간: Spoofing (위장), Info Disclosure (도청) 위협 존재             │
 │  - SQL 전송 구간: Tampering (변조), Injection 위협 존재                      │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 위협 모델링의 첫 단추는 DFD를 통해 시스템의 입출력, 프로세스, 저장소를 시각화하는 것이다. 여기서 가장 중요한 요소는 '신뢰 경계 (Trust Boundary)'다. 서로 다른 권한 수준을 가진 영역이 만나는 지점에서 위협이 발생하기 때문이다. 예를 들어 인터넷 평면에서 서버로 넘어오는 Boundary 1에서는 외부인의 'Spoofing (위장)' 위협이 높으므로 강력한 인증 메커니즘이 필요하다. 분석가는 각 흐름마다 STRIDE 위협을 대입하여 보안 요구사항을 도출한다.

---

### 심층 동작 원리: 위협 대응 전략 (The Four Responses)

식별된 위협에 대해 조직은 다음 네 가지 중 하나를 선택해야 한다.
1. **제거 (Eliminate)**: 위협을 유발하는 기능이나 구성 요소를 아예 삭제함 (가장 확실함).
2. **완화 (Mitigate)**: 보안 컨트롤(인증, 암호화, 방화벽)을 도입하여 위험을 낮춤.
3. **전가 (Transfer)**: 보험 가입이나 서드파티 서비스 이용을 통해 위험 책임을 제3자에게 넘김.
4. **수용 (Accept)**: 발생 가능성이 매우 낮거나 조치 비용이 피해액보다 클 경우, 위험을 인지하고 그대로 둠.

- **📢 섹션 요약 비유**: 도둑이 들어올 것 같을 때 아예 문을 없애버리거나(제거), 열쇠를 잠그거나(완화), 경비 업체에 맡기거나(전가), 운에 맡기는(수용) 것 중 하나를 선택하는 의사결정 과정과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 위협 모델링 분석 기법 비교 (STRIDE vs PASTA vs TRIKE)

| 기법 | 특징 | 강점 | 적합한 상황 |
|:---|:---|:---|:---|
| **STRIDE** | MS 제안, 6개 위협 관점 분석 | 이해하기 쉽고 체크리스트 위주 분석 용이 | 대부분의 일반 SW 개발 |
| **PASTA** | 비즈니스 목적 및 위험 중심 (7단계) | 비즈니스 영향도와 연계된 전략 수립 | 금융, 이커머스 등 대형 프로젝트 |
| **TRIKE** | 자산 및 권한 중심의 자동화 지향 | 자산 중요도에 따른 방어 체계 구축 | 보안 규제가 엄격한 환경 |
| **LINDDUN** | 프라이버시(Privacy) 중심 분석 | 데이터 보호 및 GDPR 대응 특화 | 개인정보 처리 시스템 |

---

### 공격 트리 (Attack Tree)와의 시너지

위협 모델링이 '어디에 위협이 있는가'를 찾는다면, 공격 트리는 '그 위협을 어떻게 현실화하는가'를 분석한다.
- **Root Node**: 최종 공격 목표 (예: 고객 정보 탈취)
- **Leaf Nodes**: 구체적인 공격 방법 (예: SQLi, 세션 하이재킹)
- **관계**: AND/OR 로직을 통해 공격 성공을 위한 필수 조건을 도식화한다.

- **📢 섹션 요약 비유**: 지도를 보고 취약점을 찾는 것(위협 모델링)과, 실제 도둑이 밧줄과 사다리를 준비해서 담을 넘는 상세 계획을 세우는 것(공격 트리)의 차이입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 자율주행 차량용 OTA (Over-The-Air) 시스템 보안

- **문제 상황**: 차량의 펌웨어를 원격으로 업데이트하는 OTA 시스템 설계 중, 해커가 악성 펌웨어를 배포할 위험이 제기됨.
- **의사결정 (위협 모델링 적용)**:
  1. **모델링**: 업데이트 서버 - 통신망 - 차량 게이트웨이 간 DFD를 작성하고 신뢰 경계를 식별한다.
  2. **식별 (STRIDE)**: 서버 위장(S), 패키지 변조(T), 업데이트 이력 부인(R), 통신 도청(I) 위협을 도출한다.
  3. **대응**: 
     - T(변조) 방지: 코드 서명(Code Signing) 및 인증서 검증 로직 추가.
     - S(위장) 방지: 상호 인증(mTLS) 적용.
  4. **검증**: 설계에 반영된 암호화 알고리즘이 현대의 연산 능력으로 해독 불가능한지 검토한다.

### 도입 체크리스트
- **적시성**: 아키텍처가 확정되기 전, 변경이 쉬운 단계에서 수행하고 있는가?
- **포괄성**: 기술적 위협뿐 아니라 비즈니스 로직(우회 결제 등)에 대한 위협도 포함했는가?
- **문서화**: 도출된 위협과 대응 방안이 개발자 티켓(Jira 등)으로 발행되어 추적 관리되는가?

### 안티패턴
- **탁상공론**: 실제 시스템 구현 방식과 동떨어진 추상적인 위협만 나열하는 것.
- **일회성 행사**: 프로젝트 초기에 한 번 하고 서랍 속에 넣어두는 문서화 활동. 시스템 변경 시 모델도 업데이트되어야 한다.

- **📢 섹션 요약 비유**: 항해를 시작할 때 한 번 지도를 본 뒤 항로가 바뀌었는데도 옛날 지도만 보고 운전하다가 암초에 부딪히는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 잠재 취약점 조기 식별 | 설계 결함으로 인한 보안 패치 발생률 60% 이상 감소 |
| **정량적 (Cost)** | 보안 비용 최적화 | 위험도가 높은 자산에 보안 예산을 집중 투자하여 효율성 극대화 |
| **정성적 (Design)** | 보안 가시성 확보 | 아키텍처의 어느 지점이 보안에 취약한지 전사적 공유 가능 |
| **정성적 (Culture)** | 보안 내재화 마인드 | 개발자와 아키텍트가 설계 단계부터 보안을 고민하게 됨 |

### 미래 전망
- **Threat Modeling as Code (TMaC)**: DFD와 위협 분석을 코드로 작성(Python, YAML 등)하여 CI/CD 파이프라인에서 자동으로 검증하는 기술이 확산될 것이다.
- **AI 기반 위협 자동 생성**: 시스템 아키텍처 다이어그램을 인식한 AI가 수천 개의 CVE(공개 취약점)와 CWE 데이터를 결합하여 맞춤형 위협 시나리오를 추천해줄 것이다.

### 참고 표준
- **NIST SP 800-154**: 가이드라인 (Draft)
- **OWASP Threat Dragon**: 오픈소스 위협 모델링 도구 표준
- **VEX (Vulnerability Exploitability eXchange)**: 취약점 활용 가능성 정보 교환 포맷

- **📢 섹션 요약 비유**: 과거에는 탐험가가 직접 길을 찾아야 했지만(수동 모델링), 이제는 GPS와 실시간 교통 정보가 목적지까지 가장 안전한 길을 알려주듯(자동화 모델링) 보안 설계도 지능화될 것입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 새로 지을 놀이터에 **나쁜 사람이 들어오지 못하게 미리 지도를 보고 "여기로 넘어올 수 있겠네!"라고 찾아보는 거예요.**
2. 창문에는 잠금장치를 달고, 문 앞에는 경비 아저씨를 세우기로 미리 계획하는 거죠.
3. 이렇게 하면 놀이터를 다 지은 다음에 다시 허물고 고치는 일 없이, 처음부터 아주 안전하게 놀 수 있답니다!
