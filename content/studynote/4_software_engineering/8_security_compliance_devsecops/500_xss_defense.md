# 500. 크로스 사이트 스크립팅 (XSS) 방어 - 입/출력값 인코딩, CSP

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: XSS (크로스 사이트 스크립팅) 방어는 공격자가 삽입한 악성 스크립트가 피해자의 브라우저에서 실행되지 않도록, 데이터가 출력되는 시점에 **특수문자를 무해한 문자로 치환(출력 인코딩)**하거나 브라우저 정책(CSP)을 통해 **스크립트 실행 자체를 통제**하는 보안 전략이다.
> 2. **가치**: 사용자의 세션 쿠키 탈취, 페이지 변조, 피싱 공격을 원천 차단하며, 특히 '출력 인코딩'은 가장 기본적이고 확실한 기술적 대응책이고 'CSP'는 실수로 누락된 취약점까지 막아주는 강력한 심층 방어막 역할을 한다.
> 3. **융합**: 입력 데이터 검증(Input Validation)과 병행되어야 하며, 현대 웹 아키텍처에서는 프레임워크(React, Angular 등)의 자동 인코딩 기능과 HTTP 보안 헤더 설정을 통해 체계적으로 관리된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: XSS 방어는 사용자가 입력한 데이터가 다시 화면에 나타날 때, 브라우저가 이를 "실행해야 할 코드"가 아닌 "단순한 글자"로 인식하게 만드는 과정이다. 주요 수단으로는 HTML 엔티티 인코딩(Entity Encoding)과 브라우저 차원의 보안 정책인 CSP (Content Security Policy)가 있다.

- **💡 비유**: XSS 방어는 **"도둑이 보낸 편지(악성 스크립트)를 읽기 전, 모든 글자를 암호화하거나 읽을 수 없는 문자로 바꾸어(인코딩), 편지 속에 든 최면 주문이 작동하지 않게 만드는 것"**과 같습니다. 또한 CSP는 **"낯선 사람의 편지는 아예 뜯어보지도 못하게 하는 보안 규정"**과 같습니다.

- **등장 배경**:
  1. **동적 콘텐츠의 증가**: 게시판, 댓글, SNS 등 사용자 참여형 서비스가 늘어나면서 악성 스크립트 유포 통로가 기하급수적으로 증가했다.
  2. **전통적 인코딩의 한계**: HTML 영역 외에도 자바스크립트 내부, URL 파라미터 등 다양한 지점(Context)에서 XSS가 발생하면서 각 상황에 맞는 정교한 인코딩 기법이 요구되었다.
  3. **브라우저 보안 기술의 진화**: 서버 측의 실수(인코딩 누락)가 있더라도 브라우저가 능동적으로 위험한 스크립트를 차단할 수 있는 정책적 대응(CSP)이 필요해졌다.

- **📢 섹션 요약 비유**: 물 속에 든 세균을 직접 걸러내는 정수기(인코딩)와, 아예 오염된 구역의 물은 마시지 못하게 폐쇄하는 경고문(CSP)을 동시에 운영하는 이중 안전 시스템과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### XSS 3단계 방어 레이어

| 단계 | 방어 기법 | 핵심 동작 | 비유 |
|:---|:---|:---|:---|
| **1단계 (입력)** | **Input Validation** | 화이트리스트 기반의 태그/속성 제한 | 위험물 반입 금지 |
| **2단계 (출력)** | **Output Encoding** | `<`, `>`, `&`, `"`, `'` 등을 HTML Entity로 변환 | 독극물을 무해한 가짜로 치환 |
| **3단계 (브라우저)** | **CSP 설정** | 신뢰할 수 있는 도메인에서만 스크립트 로드 허용 | 공인된 직원만 출입 허가 |

---

### 출력 인코딩 및 CSP 동작 아키텍처

데이터가 브라우저에 도달하여 렌더링되는 전 과정을 보호하는 아키텍처를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      XSS 통합 방어 메커니즘 (Encoding + CSP)                  │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 서버 측 처리]                                [2. 브라우저 측 렌더링]        │
 │                                                                             │
 │  사용자 데이터 (DB)                                 HTML 파서 (Browser)        │
 │        │                                              │                     │
 │        ▼ [Context-aware Encoding]                      ▼                     │
 │  ┌──────────────────┐                        ┌──────────────────┐           │
 │  │ " <script> "     │                        │  &lt;script&gt;  │           │
 │  │  치환 알고리즘    │ ─────(HTTP Response)──▶│ (단순 텍스트로 인식) │           │
 │  └──────────────────┘                        └──────────────────┘           │
 │                                                       │                     │
 │  [3. CSP 정책 적용]                                    │                     │
 │  HTTP Header: Content-Security-Policy:                │                     │
 │  "script-src 'self' trust.com;"  ◀────────────────────┘                     │
 │                                                                             │
 │  ※ 브라우저 판단:                                                            │
 │  - 인라인 스크립트 실행 금지!                                                  │
 │  - 출처가 불분명한 외부 스크립트 로드 차단!                                      │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 방어는 서버에서 시작하여 브라우저에서 완성된다. 1) 서버는 데이터를 출력할 때 `<`를 `&lt;`로, `>`를 `&gt;`로 바꾸는 **인코딩**을 수행한다. 이 결과물은 브라우저에서 실행 코드가 아닌 글자로 나타난다. 2) 서버는 응답 헤더에 **CSP (Content-Security-Policy)**를 실어 보낸다. 브라우저는 이 정책을 읽고, 페이지 내에 직접 쓰인 스크립트(Inline Script)나 검증되지 않은 외부 사이트의 스크립트가 실행되는 것을 원천 봉쇄한다. 이 두 가지가 결합되면 개발자가 실수로 인코딩을 한 군데 빠뜨려도 CSP가 '최후의 보루'로서 공격을 막아낸다.

---

### 심층 동작 원리: 컨텍스트별 인코딩 (Context-aware Encoding)

데이터가 출력되는 위치에 따라 인코딩 방식이 달라져야 한다.
1. **HTML Body**: `<div> ... </div>` 사이 출력 시. (HTML Entity 사용)
2. **HTML Attribute**: `<input value="...">` 속성 내 출력 시. (속성 탈출 방지용 인코딩)
3. **JavaScript**: `<script> var name = "..."; </script>` 내부 출력 시. (유니코드 인코딩 `\uXXXX` 사용)
4. **CSS**: `<style> .user { color: ...; } </style>` 내부 출력 시. (CSS 이스케이프 사용)

- **📢 섹션 요약 비유**: 불이 났을 때 물을 뿌리는 것(HTML 인코딩)도 좋지만, 전기 화재(자바스크립트 내 발생)에는 모래를 뿌려야 하듯(JS 인코딩) 상황에 맞는 적절한 소화 도구를 써야 한다는 점과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### CSP (Content Security Policy) 주요 지시어 비교

| 지표 (Directive) | 역할 | 보안 효과 |
|:---|:---|:---|
| **default-src** | 모든 리소스의 기본 허용 범위 설정 | 포괄적 제한 (Fallback) |
| **script-src** | 스크립트 실행 및 로드 가능 출처 제어 | **XSS 방어의 핵심** |
| **object-src** | 플러그인 (Flash 등) 로드 제어 | 악성 액티브X/플러그인 차단 |
| **style-src** | CSS 로드 가능 출처 제어 | UI 변조(Defacement) 방지 |
| **frame-ancestors** | 현재 페이지를 프레임으로 넣을 수 있는 사이트 제한 | **Clickjacking 방어** |

---

### 융합 시너지: HttpOnly 쿠키와 XSS

인코딩과 CSP가 "공격을 막는 것"이라면, **HttpOnly** 쿠키 속성은 "공격 성공 시의 피해를 막는 것"이다.
- **원리**: 쿠키에 `HttpOnly` 속성을 부여하면 자바스크립트(`document.cookie`)로 접근이 불가능해진다.
- **효과**: XSS 공격에 성공하여 스크립트가 실행되더라도, 공격자는 사용자의 세션 쿠키를 훔쳐갈 수 없어 계정 탈취가 불가능해진다.

- **📢 섹션 요약 비유**: 도둑이 집에 들어오는 것을 막는 것(XSS 방어)이 1순위지만, 혹시 들어오더라도 가장 귀중한 보석(세션 쿠키)은 절대 못 열어보는 별도의 금고(HttpOnly)에 넣어두는 이중 보안 전략입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 커뮤니티 사이트의 '안전한 에디터' 도입

- **문제 상황**: 사용자가 글을 쓸 때 HTML 태그(`<b>`, `<i>` 등)를 쓰고 싶어 하지만, 이를 허용하면 `<script>` 공격 위험이 있음.
- **의사결정 (XSS 방어 전략)**:
  1. **화이트리스트 필터링 (Server-side)**: `LucidXSS`나 `HTMLPurifier` 같은 라이브러리를 사용하여 허용된 안전한 태그와 속성(href, src 등)만 남기고 나머지는 제거한다.
  2. **엄격한 CSP 적용**: `Content-Security-Policy: default-src 'self'; script-src 'self' https://trusted-cdn.com;` 헤더를 설정하여 인라인 스크립트 실행을 금지한다.
  3. **Non-Executable Domain**: 사용자 업로드 파일(이미지 등)은 실행 권한이 없는 별도의 도메인(`static.com`)에서 서비스하여 스크립트 실행 위험을 분리한다.
  4. **DOM-based XSS 방어**: 프론트엔드 코드에서 `.innerHTML` 대신 `.textContent`나 `.innerText`를 사용하여 데이터가 HTML로 해석되는 것을 방지한다.

### 도입 체크리스트
- **Automatic Encoding**: 사용 중인 템플릿 엔진(Thymeleaf, Handlebars 등)이 기본적으로 모든 변수를 인코딩하도록 설정되어 있는가?
- **CSP Report-Only**: 실운영 환경에 CSP를 적용하기 전, `Report-Only` 모드를 통해 정상적인 기능이 차단되지 않는지 충분히 모니터링했는가?
- **Cookie Security**: 모든 민감한 쿠키에 `HttpOnly`, `Secure` 속성이 부여되었는가?

### 안티패턴
- **블랙리스트 `<script>` 제거**: `<script>` 문자열만 지우는 방식. 공격자는 `<sCrIpT>`나 `<img onerror=...>` 등으로 쉽게 우회한다.
- **unsafe-inline 허용**: CSP 설정 시 편의를 위해 `'unsafe-inline'`을 허용하는 행위. CSP의 존재 가치를 상실하게 만드는 가장 위험한 설정이다.

- **📢 섹션 요약 비유**: "나쁜 단어만 안 쓰면 돼"라고 아이를 가르치는 것(블랙리스트)보다, "미리 정해진 예의 바른 말만 써야 한다"라고 가르치는 것(화이트리스트/인코딩)이 훨씬 올바른 훈육법이라는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 계정 탈취 사고 90% 예방 | 세션 쿠키 탈취 차단을 통한 2차 피해 방지 |
| **정량적 (Compliance)** | 보안 인증 요건 충족 | ISMS-P, OWASP 등 글로벌 웹 보안 표준 완벽 준수 |
| **정성적 (Trust)** | 사용자 프라이버시 보호 | 개인정보 오남용 및 페이지 변조 없는 안전한 서비스 경험 제공 |
| **정성적 (Efficiency)** | 보안 관리 자동화 | CSP 및 프레임워크 내장 기능을 통한 시스템적 보안 통제 구현 |

### 미래 전망
- **Trusted Types API**: 브라우저 차원에서 취약한 함수(innerHTML 등)에 신뢰할 수 없는 데이터가 들어가는 것을 원천 봉쇄하는 새로운 보안 표준이 확산될 것이다.
- **AI-Managed CSP**: 사이트의 자원 호출 패턴을 분석하여 최적의 CSP 정책을 자동으로 생성하고, 정책 위반 시 지능적으로 대응하는 AI 보안 솔루션이 도입될 것이다.

### 참고 표준
- **CWE-79**: Improper Neutralization of Input During Web Page Generation ('Cross-site Scripting')
- **W3C Content Security Policy Level 3**: 최신 CSP 명세서
- **OWASP XSS Prevention Cheat Sheet**: 실무형 방어 가이드

- **📢 섹션 요약 비유**: 지금까지는 개별 창문을 단속하는 데 집중했다면, 미래에는 집 전체를 감싸는 지능형 투명 보호막(Trusted Types/CSP)이 모든 침입을 자동으로 막아주는 시대로 나아가고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 나쁜 사람이 우리 집 우체통에 **"이 편지를 읽으면 가방을 밖으로 던져라"라는 마법이 걸린 편지(악성 스크립트)를 넣어두는 것**과 같아요.
2. XSS 방어는 그 편지를 읽기 전에 **마법 글자들을 평범한 낙서로 바꿔버려서(인코딩)** 아무리 읽어도 마법이 걸리지 않게 하는 거예요.
3. 그리고 **"모르는 사람이 보낸 편지는 절대 열지 마!"라는 집안 규칙(CSP)**을 정해두면 훨씬 더 안전해진답니다!
