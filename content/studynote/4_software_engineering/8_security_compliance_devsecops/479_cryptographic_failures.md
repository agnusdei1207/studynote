# 479. Cryptographic Failures (암호화 실패 / 민감 데이터 노출)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Cryptographic Failures (암호화 실패)는 민감한 데이터를 보호하기 위해 적용된 암호화 알고리즘, 키 관리, 전송 프로토콜 등이 취약하거나 부적절하게 구현되어 **데이터의 기밀성과 무결성이 훼손되는 상태**를 의미한다.
> 2. **가치**: 2021년 OWASP Top 10에서 2위에 오를 만큼 치명적이며, 개인정보 유출 사고의 근본 원인이 되는 경우가 많아 법적 컴플라이언스(GDPR, ISMS-P 등) 준수를 위한 최우선 방어 항목이다.
> 3. **융합**: 취약한 알고리즘(DES, MD5) 사용, 하드코딩된 암호화 키, 부적절한 인증서 검증, 평문 전송(HTTP) 등을 포함하며, 현대 아키텍처에서는 양자 내성 암호(PQC) 준비 및 HSM (Hardware Security Module) 기반의 키 관리 체계 도입이 강조되고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 과거 'Sensitive Data Exposure'로 불리던 이 항목은 2021년 'Cryptographic Failures'로 명칭이 변경되면서, 데이터 노출이라는 현상보다 **'암호화 기술의 실패'라는 근본 원인**에 더 집중하게 되었다. 이는 전송 중(Data-in-Transit) 또는 보관 중(Data-at-Rest)인 데이터가 암호화 미비로 인해 공격자에게 노출되는 모든 상황을 포괄한다.

- **💡 비유**: Cryptographic Failures는 **"아무리 튼튼한 금고(애플리케이션)라도 열쇠를 금고 바로 옆에 두거나(키 관리 실패), 금고 문이 종이로 만들어져 있는(취약한 알고리즘) 상태"**와 같습니다.

- **등장 배경**:
  1. **데이터 가치의 수직 상승**: 개인정보와 비즈니스 기밀이 해커들의 주 수익원이 되면서 암호화 우회 공격이 정교해졌다.
  2. **컴퓨팅 성능의 비약적 발전**: 과거 안전하다고 믿었던 알고리즘(SHA-1, 1024비트 RSA 등)이 클라우드 컴퓨팅과 GPU 연산 능력 향상으로 인해 실시간 해독 가능 영역으로 들어왔다.
  3. **키 관리의 복잡성 증가**: 분산 환경(MSA)에서 수많은 서비스 간의 암호화 키를 안전하게 생성, 저장, 갱신(Rotation)하는 작업이 고도화된 엔지니어링 영역이 되었다.

- **📢 섹션 요약 비유**: 보물 상자를 아무리 꽁꽁 묶어도 매듭(알고리즘)이 헐겁거나 가위(해킹 툴) 한 번에 잘린다면 그 상자는 열려 있는 것과 다름없다는 원리와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Cryptographic Failures의 주요 원인 및 사례

암호화 실패는 크게 전송 보안, 저장 보안, 키 관리의 세 영역에서 발생한다.

| 영역 | 주요 취약점 | 공격 및 사고 사례 | 비유 |
|:---|:---|:---|:---|
| **전송 (Transit)** | 평문 전송 (HTTP, FTP), 취약한 프로토콜(TLS 1.0/1.1) | 공공 Wi-Fi에서의 패킷 스니핑을 통한 세션 탈취 | 엽서에 비밀번호 적어 보내기 |
| **저장 (At-Rest)** | 솔트(Salt) 없는 해시, 취약한 암호화(DES) | DB 유출 시 레인보우 테이블을 이용한 비밀번호 대량 복구 | 투명한 비닐봉투에 돈 담기 |
| **키 관리 (Key Mgmt)** | 소스코드 내 키 하드코딩, 단순한 키 생성 | GitHub 오픈소스 저장소를 통한 API 키 유출 | 대문 밑에 열쇠 숨기기 |
| **인증서 (Cert)** | 자체 서명 인증서, 만료된 인증서 무시 | 중간자 공격 (MitM)을 통한 데이터 가로채기 | 위조 신분증에 속기 |

---

### 안전한 암호화 데이터 흐름 아키텍처

민감 데이터는 생성 시점부터 파기 시점까지 전 수명 주기에 걸쳐 암호화되어야 한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      엔드-투-엔드(E2E) 데이터 암호화 아키텍처                    │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [클라이언트]              [API 게이트웨이 / 서버]             [DB / 저장소]     │
 │      │                          │                               │           │
 │  1. 데이터 입력 ────▶ 2. TLS v1.3 전송 ────▶ 3. 애플리케이션 처리 ───▶ 4. 저장      │
 │     (평문)             (전송 구간 암호화)       (필요 시 내부 암호화)    (저장 암호화) │
 │                                 │                               │           │
 │                                 ▼                               ▼           │
 │                       ┌──────────────────┐            ┌──────────────────┐  │
 │                       │   HSM / KMS      │ ◀──(Key)── │ TDE (Transparent)│  │
 │                       │ (중앙 키 관리)    │            │  Column 암호화    │  │
 │                       └──────────────────┘            └──────────────────┘  │
 │                                                                             │
 │  [핵심 보안 제어]                                                            │
 │  - 전송: HSTS 적용, 강력한 Cipher Suite (ECDHE-RSA-AES256-GCM) 사용            │
 │  - 저장: SHA-256 + Salt (비밀번호), AES-256 (개인정보)                         │
 │  - 관리: 키 순환(Rotation), 접근 제어 정책 적용                                │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 안전한 아키텍처는 다중 방어(Defense in Depth)를 기본으로 한다. 1) 전송 시에는 TLS 1.3을 통해 중간자 공격을 차단한다. 2) 서버 메모리 내에서도 민감 데이터는 최소한의 시간 동안만 평문으로 존재해야 한다. 3) DB 저장 시에는 TDE (Transparent Data Encryption) 또는 컬럼 레벨 암호화를 적용하며, 이때 사용되는 암호화 키는 애플리케이션 서버가 아닌 별도의 **KMS (Key Management Service)** 또는 **HSM (Hardware Security Module)**에서 안전하게 관리되어야 한다. 특히 비밀번호는 복호화가 불가능한 일방향 해시(Salt 포함)를 사용하는 것이 글로벌 표준이다.

---

### 심층 동작 원리: 일방향 해시와 솔팅 (Salting)

비밀번호 저장 시 단순 해시만 사용하면 레인보우 테이블(Rainbow Table) 공격에 취약하다.
1. **솔트 (Salt) 생성**: 각 사용자마다 고유한 무작위 문자열(Salt)을 생성한다.
2. **결합 및 해시**: `Hash(Password + Salt)` 과정을 거친다.
3. **반복 (Iteration)**: 키 스트레칭(Key Stretching) 기법을 통해 해시 연산을 수천 번 반복하여 무차별 대입 공격 시간을 늦춘다. (예: PBKDF2, bcrypt, Argon2 사용)

- **📢 섹션 요약 비유**: 음식에 나만의 특별한 향신료(Salt)를 뿌리고 수천 번 저어서(Stretching), 원래 재료가 무엇이었는지 아무도 알아내지 못하게 만드는 조리법과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 인코딩(Encoding) vs 해싱(Hashing) vs 암호화(Encryption) 비교

| 비교 항목 | 인코딩 (Encoding) | 해싱 (Hashing) | 암호화 (Encryption) |
|:---|:---|:---|:---|
| **목적** | 데이터 포맷 변환 (Base64 등) | 데이터 무결성 검증, 비밀번호 저장 | 데이터 기밀성 유지 (복구 가능) |
| **가역성** | **가역** (누구나 디코딩 가능) | **비가역** (복호화 불가) | **가역** (키가 있어야 복호화 가능) |
| **핵심 요소** | 공개된 알고리즘 | 일방향성, 충돌 저항성 | 암호화 키 (대칭/비대칭) |
| **보안성** | **없음** (보안 수단 아님) | 높음 (솔트 적용 시) | 높음 (키 관리 철저 시) |

---

### 현대적 기술 융합: 형태 보존 암호화 (FPE, Format Preserving Encryption)

데이터의 길이와 형식을 유지하면서 암호화하는 기술이다.
- **활용**: 카드 번호(16자리 숫자)를 암호화해도 여전히 16자리 숫자로 유지되어, 기존 DB 스키마나 레거시 시스템의 수정 없이 암호화를 적용할 수 있다.
- **장점**: 데이터 마스킹과 암호화의 이점을 동시에 제공하여 실무 활용도가 매우 높다.

- **📢 섹션 요약 비유**: 한글 문장을 영어로 번역하는 것이 아니라, 글자 모양은 그대로 두되 의미만 바보처럼 바꿔서 문장의 길이나 구조는 그대로 유지하는 암호 기술과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 소스코드 내 API Key 노출 사고 대응

- **문제 상황**: 개발자가 실수로 공공 클라우드(AWS)의 Secret Key가 포함된 코드를 GitHub Public 저장소에 푸시(Push)함.
- **의사결정 (긴급 대응)**:
  1. **무효화 (Revoke)**: 즉시 해당 키를 클라우드 콘솔에서 비활성화 또는 삭제한다. (History 삭제만으로는 부족함)
  2. **교체 (Rotate)**: 새로운 키를 발급받아 시스템에 적용한다.
  3. **환경 변수화**: 키를 소스코드에서 분리하여 서버의 환경 변수나 전용 Secret Store (Vault, AWS Secrets Manager)를 사용하도록 리팩토링한다.
  4. **사전 방지**: `git-secrets` 또는 `trufflehog` 같은 도구를 도입하여 커밋 전 민감 정보 포함 여부를 자동 스캔한다.

### 도입 체크리스트
- **Modern Algorithms**: AES-256, RSA-2048 이상, SHA-256 이상의 최신 알고리즘을 사용하고 있는가?
- **Protocol Security**: HTTP를 HTTPS로 강제 리다이렉트(HSTS) 처리하고 있는가?
- **Key Separation**: 암호화 데이터와 암호화 키를 물리적으로 분리된 저장소에 보관하고 있는가?

### 안티패턴
- **자체 암호화 (Home-grown Crypto)**: 유명 알고리즘 대신 "나만 아는 복잡한 수식"으로 암호화하는 행위. 암호학적으로 가장 위험한 자살 행위다.
- **Base64를 암호로 착각**: Base64 인코딩된 데이터를 "암호화했다"고 안심하는 행위. 이는 누구나 1초 만에 풀 수 있는 포맷 변환일 뿐이다.

- **📢 섹션 요약 비유**: 전문가가 만든 잠금장치 대신 내가 직접 만든 나무 막대기 빗장을 걸어두고 "이건 복잡해서 아무도 못 열 거야"라고 착각하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Compliance)** | 법적 과태료 예방 | 개인정보 암호화 위반 시 발생하는 매출액의 수 % 과징금 리스크 제거 |
| **정량적 (Trust)** | 브랜드 가치 보호 | 데이터 유출 사고 발생 시에도 암호화로 인해 2차 피해(평문 유출) 차단 |
| **정성적 (Security)** | 심층 방어 체계 구축 | 네트워크가 뚫리거나 DB가 유출되어도 데이터 자체는 보호되는 최후의 보루 |
| **정성적 (Standard)** | 글로벌 보안 인증 획득 | ISO 27001, SOC2 등 글로벌 보안 표준의 핵심 요건 충족 |

### 미래 전망
- **양자 내성 암호 (PQC, Post-Quantum Cryptography)**: 양자 컴퓨터의 등장으로 현재의 RSA/ECC 암호가 무력화될 것에 대비하여, Lattice 기반 암호 등 새로운 표준으로의 전환이 시작될 것이다.
- **동형 암호 (Homomorphic Encryption)**: 데이터를 암호화된 상태 그대로 연산(덧셈, 곱셈 등)할 수 있는 기술로, 개인정보 노출 없이 빅데이터 분석이 가능한 시대를 열 것이다.

### 참고 표준
- **FIPS 140-2/3**: 미국 정부의 암호 모듈 보안 요구사항 표준
- **KISA 암호 알고리즘 및 키 길이 가이드**: 국내 표준 권고안 (RSA 2048, AES 128 이상 등)

- **📢 섹션 요약 비유**: 지금까지는 튼튼한 자물쇠를 만드는 경쟁이었다면, 미래에는 자물쇠를 풀지 않고도 그 안의 물건을 만질 수 있는 마법 같은 기술(동형 암호)과 원자 폭탄급 해킹 도구에 견디는 초강력 재질(양자 내성 암호)의 대결이 될 것입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 우리가 친구랑 비밀 일기를 쓸 때, 다른 사람이 읽지 못하게 **글자를 미리 약속한 암호(숫자나 기호)로 바꿔서 쓰는 것**과 같아요.
2. 하지만 암호를 너무 쉽게 만들거나, 암호를 푸는 방법(열쇠)을 일기장 바로 옆에 두면 다른 사람에게 비밀을 들키게 돼요.
3. 소중한 내 정보(이름, 비밀번호)가 밖으로 새 나가지 않도록 아주 튼튼한 암호 상자에 담아서 보관하는 것이 중요하답니다!
