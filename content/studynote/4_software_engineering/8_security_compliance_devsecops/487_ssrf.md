# 487. SSRF (Server-Side Request Forgery)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSRF (서버 측 요청 위조)는 공격자가 서버를 조종하여, 서버가 접근할 수 있는 **내부 네트워크 자원이나 외부 시스템에 악의적인 요청을 보내게 만드는** 취약점이다.
> 2. **가치**: 2021년 OWASP Top 10에서 10위(Community 선정)로 신설되었으며, 클라우드 환경의 메타데이터 서버(IMDS)를 공격하여 관리자 권한을 탈취하거나 방화벽으로 보호된 내부망을 스캔할 수 있어 매우 치명적이다.
> 3. **융합**: 클라우드 네이티브 보안, API 보안, 네트워크 세분화(Segmentation)와 결합되며, 단순 URL 필터링을 넘어 '애플리케이션 계층에서의 명시적 허용 리스트(Allow-list)'와 '네트워크 격리'가 근본적인 방어책이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: SSRF는 서버가 외부 사용자로부터 입력받은 URL을 충분한 검증 없이 그대로 호출할 때 발생한다. 공격자는 서버를 "프록시(Proxy)"처럼 활용하여, 외부에서는 접근이 불가능한 서버 내부의 파일, DB, 또는 클라우드 설정 정보에 접근한다.

- **💡 비유**: SSRF는 **"회사 건물 안으로 들어갈 수 없는 외부인이, 회사 내부 사정을 잘 아는 비서(서버)에게 '사장님 책상 위에 있는 기밀 서류 사진 좀 찍어서 보내줘'라고 속여서 기밀을 빼내는 상황"**과 같습니다.

- **등장 배경**:
  1. **클라우드 서비스의 보편화**: AWS, Azure 등 클라우드 서버는 자신만의 설정 정보를 담은 메타데이터 서버(`169.254.169.254`)를 운영하는데, SSRF는 이 정보를 훔치는 가장 강력한 수단이다.
  2. **복잡한 아키텍처 연동**: 웹 애플리케이션이 이미지 썸네일 생성, PDF 변환, 외부 API 호출 등 다른 서버와 통신하는 기능이 많아지면서 공격 지점이 늘어났다.
  3. **내부망 신뢰의 허점**: 많은 기업이 외부망은 철저히 막지만 내부망끼리는 신뢰하는 구조를 가지고 있어, 일단 서버 하나만 장악(SSRF 성공)하면 내부망 전체가 뚫리는 구조적 문제가 있었다.

- **📢 섹션 요약 비유**: 대문은 굳게 잠겨 있지만, 집 안의 로봇 청소기(서버)를 원격 조종해서 집 구석구석을 훔쳐보게 만드는 영리한 침입 기법과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SSRF 공격의 주요 유형 및 대상

| 유형 | 설명 | 공격 대상 | 비유 |
|:---|:---|:---|:---|
| **기본 SSRF** | 서버가 내부 자원의 데이터를 응답으로 반환 | 내부 DB 데이터, 로컬 파일 시스템(`file:///`) | 비서가 서류를 직접 갖다줌 |
| **Blind SSRF** | 데이터는 안 보이지만 상태 변화로 판단 | 내부망 포트 스캔, HTTP 상태 코드(200 vs 500) | 비서에게 물어보고 표정 읽기 |
| **Cloud SSRF** | 클라우드 인프라 권한 탈취 | AWS 메타데이터 서버(`iam/security-credentials`) | 비서에게 금고 비밀번호 물어보기 |
| **Protocol Abuse** | HTTP 외의 다른 프로토콜 악용 | `gopher://`, `dict://` 등을 이용한 로컬 서비스 공격 | 비서에게 암호로 소통하기 |

---

### 클라우드 메타데이터 기반 SSRF 공격 아키텍처

클라우드 환경에서 SSRF가 어떻게 권한 탈취로 이어지는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      클라우드 IMDS (Instance Metadata Service) 공격 흐름          │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [공격자 (인터넷)]          [취약한 웹 서버]             [메타데이터 서버]         │
 │        │                         │ (Internal Network)        │              │
 │  1. URL 입력 ─────────────────▶  │                           │              │
 │     ?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/  │
 │                                  │                           │              │
 │                                  │ 2. 검증 없이 내부 요청 ────▶ │              │
 │                                  │                           │              │
 │                                  │ 3. 임시 자격증명 반환 ◀──── │              │
 │                                  │    (AccessKey, Token)     │              │
 │                                  │                           │              │
 │  4. 탈취한 키로 클라우드 장악 ◀───┘                           │              │
 │     (S3 탈취, 인스턴스 종료 등)                                │              │
 │                                                              │              │
 │  ※ 방어책: IMDSv2 적용 (Session Token 기반), SSRF Filter 적용                  │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클라우드 서버는 자신의 역할(Role)에 맞는 임시 권한을 얻기 위해 내부의 고정 IP인 `169.254.169.254`와 통신한다. 공격자가 웹 서버의 "URL 입력" 기능에 이 주소를 넣으면, 서버는 외부로 나가는 것이 아니라 자기 자신의 내부 망에 있는 메타데이터 서버에 질문을 던지게 된다. 그 결과로 받은 **IAM 임시 자격증명**은 공격자에게 전달되어, 공격자는 자신의 PC에서도 해당 클라우드 서버의 관리자 권한을 행사할 수 있게 된다. 이것이 Capital One 유출 사고와 같은 대형 참사의 주된 메커니즘이다.

---

### 심층 동작 원리: DNS Rebinding 공격

서버 측의 IP 기반 화이트리스트 필터링을 우회하는 고난도 기법이다.
1. **첫 번째 요청**: 공격자는 자신이 제어하는 도메인(`malicious.com`)을 보낸다. 서버가 DNS를 조회하면 공격자는 짧은 TTL(Time to Live)과 함께 '안전한 IP'를 알려준다.
2. **검증 통과**: 서버는 IP가 안전하므로 검증을 통과시킨다.
3. **두 번째 요청 (실행)**: 서버가 실제 접속을 위해 다시 DNS를 조회하면, 공격자는 이번에는 '내부 IP (예: 127.0.0.1)'로 응답을 바꾼다. 서버는 이미 검증됐다고 믿고 내부 자원에 접속하게 된다.

- **📢 섹션 요약 비유**: 신분증 검사를 할 때는 착한 사람의 얼굴(안전한 IP)을 보여주고, 문이 열리는 찰나에 나쁜 사람(내부 IP)으로 얼굴을 바꿔치기하여 들어가는 트릭과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SSRF vs CSRF (Cross-Side Request Forgery) 비교

| 비교 항목 | SSRF (Server-Side) | CSRF (Cross-Site) |
|:---|:---|:---|
| **요청 주체** | **서버 (Server)** | **사용자 브라우저 (Client)** |
| **공격 대상** | 서버의 내부 자원, 인접 시스템 | 사용자의 계정 권한 (송금, 비번 변경) |
| **공격 조건** | 서버의 외부 통신 기능 필요 | 사용자가 로그인된 세션이 있어야 함 |
| **방어 수단** | Allow-list 필터링, 네트워크 격리 | Anti-CSRF 토큰, SameSite 쿠키 |

---

### 기술 융합: 제로 트러스트와 SSRF 방어

전통적인 네트워크 보안(경계 보안)의 한계를 제로 트러스트로 극복한다.
- **Micro-segmentation**: 서버가 다른 내부 서버와 통신할 때도 매번 인증을 요구하고, 꼭 필요한 경로만 열어둔다.
- **Identity-based Access**: IP 주소가 아닌, 서비스 자체의 신원(Identity)을 기반으로 통신 권한을 제어하여 SSRF로 IP를 사칭해도 접근을 막는다.

- **📢 섹션 요약 비유**: 집 안에서 가족끼리는 문을 안 잠그고 살았는데(경계 보안), 이제는 가족이라도 각자 방에 들어갈 때 지문 인식을 해야 하는 철저한 통제 시스템(제로 트러스트)으로 바뀐 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 이미지 크롤링 서비스의 보안 아키텍처 설계

- **문제 상황**: 사용자가 입력한 URL의 이미지를 가져와 썸네일을 만드는 서비스에서 내부 서버 스캔 위협이 발견됨.
- **의사결정 (SSRF 방어 전략)**:
  1. **네트워크 격리 (Isolation)**: 이미지 추출 기능을 수행하는 서버를 별도의 VPC(가상 사설망)에 격리하고, 내부망으로의 모든 아웃바운드 트래픽을 차단한다.
  2. **화이트리스트 (Allow-list)**: `http`, `https` 프로토콜만 허용하고, 접근 가능한 도메인을 사전에 정의된 신뢰할 수 있는 사이트로 한정한다.
  3. **메타데이터 서버 차단**: 인스턴스 설정에서 IMDSv2를 강제하고, 홉 제한(Hop Limit)을 1로 설정하여 서버 내부에서의 릴레이 공격을 원천 차단한다.
  4. **전용 라이브러리 사용**: 리다이렉트(`302 Found`)를 자동으로 따라가지 않도록 옵션을 끄거나, 리다이렉트된 주소를 재검증하는 로직을 추가한다.

### 도입 체크리스트
- **Protocol Restriction**: `file://`, `gopher://`, `ftp://` 등 위험한 프로토콜이 차단되어 있는가?
- **Internal IP Block**: 입력받은 URL이 `127.0.0.1`, `10.0.0.0/8`, `192.168.0.0/16` 등 사설 IP 대역인지 확인하는가?
- **Cloud Hardening**: 클라우드 환경이라면 메타데이터 서비스에 대한 보안 강화 설정이 완료되었는가?

### 안티패턴
- **블랙리스트 필터링**: `localhost`나 `169.254.169.254`만 막는 방식. 공격자는 `[::]`, `0.0.0.0`, 또는 16진수 IP 표현 등으로 쉽게 우회한다.
- **클라이언트 측 URL 검증**: 프록시 툴로 서버에 직접 값을 보내면 무용지물이다. 검증은 반드시 서버 측(Server-side)에서 이루어져야 한다.

- **📢 섹션 요약 비유**: 나쁜 단어만 골라서 입을 막으려고(블랙리스트) 노력하기보다, 아예 미리 약속된 예의 바른 말만(화이트리스트) 할 수 있도록 규칙을 정하는 것이 훨씬 안전하다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 내부망 침투 위험 90% 차단 | 네트워크 격리 및 필터링 적용 시 서버를 통한 내부 공격 무력화 |
| **정량적 (Control)** | 클라우드 자산 보호 | 자격증명 탈취 시도를 차단하여 인프라 전체 장악 리스크 제거 |
| **정성적 (Standard)** | 현대적 보안 요건 충족 | OWASP Top 10 최신 기준을 준수하는 선진 보안 아키텍처 확보 |
| **정성적 (Reliability)** | 서비스 안정성 향상 | 비정상적인 내부 요청으로 인한 시스템 부하 및 충돌 예방 |

### 미래 전망
- **WAF의 지능화**: 머신러닝이 SSRF 우회 패턴을 학습하여, 변형된 IP 주소나 복잡한 리다이렉트 트릭을 실시간으로 감지할 것이다.
- **Service Mesh 기반 제어**: Istio나 Linkerd 같은 서비스 메시가 마이크로서비스 간의 모든 통신을 가시화하고 제어함으로써, 애플리케이션 코드 수정 없이도 네트워크 수준에서 SSRF를 원천 봉쇄하게 될 것이다.

### 참고 표준
- **OWASP TOP 10 - A10:2021**: SSRF 공식 카테고리
- **AWS IMDSv2**: 인스턴스 메타데이터 보안 표준
- **RFC 1918**: 사설망 IP 주소 할당 표준 (필터링 기준)

- **📢 섹션 요약 비유**: 예전에는 일 잘하는 비서에게 모든 것을 맡겼지만, 이제는 비서가 누구와 통화하는지 실시간으로 기록하고(가시성), 가족 외에는 절대 연결해주지 못하게(격리) 하는 철저한 비서 관리 시대가 되었습니다.
