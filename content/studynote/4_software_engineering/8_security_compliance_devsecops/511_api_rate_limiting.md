# 511. API Rate Limiting 및 Throttling

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: API Rate Limiting(비율 제한)과 Throttling(스로틀링)은 특정 시간 동안 클라이언트가 보낼 수 있는 **요청의 횟수를 제한하여 시스템 자원의 고갈을 방지**하고 서비스의 공정성(Fairness)을 유지하는 트래픽 제어 기술이다.
> 2. **가치**: 악의적인 DDoS 공격, 무분별한 스크래핑(Crawling), 프로그래밍 실수에 의한 무한 루프 호출로부터 서버를 보호하며, 갑작스러운 트래픽 폭주 시에도 핵심 서비스의 가용성(Availability)을 보장하는 완충 장치 역할을 한다.
> 3. **융합**: API 게이트웨이 및 서비스 메시(Service Mesh)의 핵심 기능으로 탑재되며, 토큰 버킷(Token Bucket), 리킹 버킷(Leaky Bucket) 등 다양한 알고리즘을 통해 비즈니스 요건에 맞는 정밀한 트래픽 정책을 구현한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **Rate Limiting**: 특정 주체(IP, User ID 등)가 정해진 시간 내에 수행할 수 있는 최대 요청 횟수를 제한함.
  - **Throttling**: 시스템의 과부하를 막기 위해 일시적으로 요청 속도를 늦추거나 일부를 거부함.

- **💡 비유**: Rate Limiting과 Throttling은 **"인기 맛집의 번호표 시스템"**과 같습니다. 한꺼번에 수천 명이 식당 안으로 밀고 들어오면 주방이 마비되므로, 입구에서 번호표를 나눠주고(Rate Limiting) 자리가 날 때까지만 들여보내거나(Throttling) 대기 줄이 너무 길면 예약을 더 이상 받지 않는 것과 같습니다.

- **등장 배경**:
  1. **API 경제의 확산**: 서비스가 API 중심으로 통합되면서 외부 파트너사나 공개된 앱으로부터 들어오는 트래픽 통제가 불가능해졌다.
  2. **자원 공유 모델 (Multi-tenancy)**: 하나의 서버가 여러 고객을 응대할 때, 특정 고객의 과도한 사용이 다른 고객의 성능 저하를 일으키는 '시끄러운 이웃(Noisy Neighbor)' 문제를 해결해야 했다.
  3. **보안 위협 지능화**: 브루트 포스(비번 무차별 대입)나 API를 통한 대량 정보 수집 공격이 늘어나면서, 비정상적인 호출 빈도를 감지하고 차단할 필요가 생겼다.

- **📢 섹션 요약 비유**: 수도꼭지를 끝까지 돌린다고 물이 무한정 쏟아져서 배수구가 막히지 않게, 중간에 수압 조절기(Throttling)를 달아 안전한 물 흐름을 유지하는 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 트래픽 제어 알고리즘 비교

| 알고리즘 | 작동 원리 | 특징 | 비유 |
|:---|:---|:---|:---|
| **Token Bucket** | 일정한 속도로 토큰을 채우고, 요청 시 토큰 소비 | 일시적인 트래픽 폭주(Burst) 허용 | 정해진 수량의 식권 배부 |
| **Leaky Bucket** | 요청을 큐에 담고 일정한 속도로 처리 | 일정한 응답 속도 보장, 폭주 불허 | 구멍 난 물통 (졸졸 나옴) |
| **Fixed Window** | 정해진 시간 단위(예: 1분)로 횟수 제한 | 구현 단순, 윈도우 경계 시점에 트래픽 몰림 위험 | 매시 정각 100명 선착순 |
| **Sliding Window** | 시간 윈도우를 실시간 이동하며 횟수 계산 | **가장 정교함**, 경계 문제를 해결함 | 최근 1분간의 기록 실시간 감시 |

---

### 2. API 게이트웨이 기반 제어 아키텍처

트래픽이 백엔드 서버에 도달하기 전 입구에서 걸러지는 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      중앙 집중형 API 트래픽 제어 아키텍처                         │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Client Side]              [API Gateway (PEP)]             [Backend API]    │
 │                                     │                                       │
 │  ┌────────────┐           ┌──────────────────────┐         ┌────────────┐   │
 │  │ Attacker   │ ──(Req)──▶ │ 1. Identify Client   │ ──(Req)▶ │ Order Serv │   │
 │  └────────────┘           │    (IP, API Key, ID) │         └────────────┘   │
 │                           │ 2. Check Limit (Redis)│                         │
 │  ┌────────────┐           │                      │         ┌────────────┐   │
 │  │ Normal User│ ──(Req)──▶ │ 3. Update Counter    │ ──(Req)▶ │ Pay Serv   │   │
 │  └────────────┘           │ 4. Allow or Reject   │         └────────────┘   │
 │                           └──────────┬───────────┘                          │
 │                                      │                                      │
 │                                      ▼ [Reject Case]                        │
 │                           ┌──────────────────────┐                          │
 │                           │ HTTP 429 Too Many    │                          │
 │                           │ Requests + Retry-After│                          │
 │                           └──────────────────────┘                          │
 │                                                                             │
 │  ※ 가치: Redis와 같은 인메모리 DB를 사용하여 밀리초 단위로 횟수 체크 수행           │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 트래픽 제어의 핵심은 **식별(Identification)**과 **상태 관리**다. 1) **API 게이트웨이**는 들어오는 모든 요청의 신원(API Key 등)을 확인한다. 2) 실시간으로 **Redis**와 같은 고속 저장소에 해당 클라이언트의 최근 호출 횟수를 묻는다. 3) 허용 한도를 넘었다면 백엔드 서버로 요청을 보내지 않고 그 자리에서 **HTTP 429 (Too Many Requests)** 응답을 보낸다. 이때 `Retry-After` 헤더를 통해 언제 다시 시도할 수 있는지 알려줌으로써 클라이언트의 매너 있는 재시도를 유도한다.

---

### 심층 동작 원리: 계층형 스로틀링 (Tiered Throttling)

비즈니스 가치에 따른 차등적 트래픽 관리다.
1. **사용자 등급**: 유료 회원은 분당 1,000회, 무료 회원은 10회로 차별화.
2. **서비스 중요도**: 결제 API는 넉넉하게, 홍보용 이벤트 API는 타이트하게 제한.
3. **가치**: 한정된 서버 자원을 돈을 지불한 고객이나 핵심 비즈니스 로직에 우선 배분하여 ROI를 극대화한다.

- **📢 섹션 요약 비유**: 놀이공원에서 비싼 티켓(유료 회원)을 산 사람에게 전용 대기 줄(하이패스)을 제공하여 만족도를 높이는 프리미엄 서비스 전략과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Rate Limiting vs Circuit Breaker (회복력 관점)

| 비교 항목 | Rate Limiting (비율 제한) | Circuit Breaker (차단기) |
|:---|:---|:---|
| **제어 기준** | **클라이언트의 호출 빈도** | **서버의 에러 발생률** |
| **주요 대상** | 외부에서 들어오는 유입량 통제 | 내부 서비스 간 장애 전파 방지 |
| **목적** | 자원 오남용 방지, 보안 | 시스템 회복력(Resilience) 확보 |
| **비유** | 현관문에서 입장 인원 조절 | 집안 전기가 합선되면 차단기 내림 |

---

### 기술 융합: API 보안과 Rate Limiting

단순 부하 분산을 넘어 보안 솔루션으로 융합된다.
- **WAF (Web Application Firewall)**: Rate Limiting 기능을 통해 특정 IP에서 짧은 시간에 발생하는 로그인 시도(Brute Force)나 비정상적인 데이터 수집(Scraping)을 자동 탐지 및 영구 차단한다.
- **가치**: 트래픽의 '양'을 보고 '의도'를 파악하는 지능형 보안 방어막이 된다.

- **📢 섹션 요약 비유**: 문을 세게 두드리는 사람(폭주 트래픽)을 잠시 멈추게 하는 게 아니라, 1초에 100번씩 문고리를 흔드는 수상한 사람(해커)을 경찰에 신고하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 한정판 운동화 응모 이벤트의 트래픽 설계

- **문제 상황**: 응모 시작 시각 정각에 수백만 명의 사용자가 몰려 서버가 다운될 위기. (DDoS급 부하)
- **의사결정 (제어 전략 수립)**:
  1. **Token Bucket 적용**: 평시에는 넉넉한 토큰을 주다가, 이벤트 시점에는 토큰 생성 속도를 대폭 낮춘다.
  2. **Client-side Throttling**: 앱 화면에서 응모 버튼을 한 번 누르면 10초간 비활성화하여 의미 없는 연타(Click Storm)를 막는다.
  3. **Queueing**: 게이트웨이에서 넘치는 요청은 버리지 않고 메시지 큐(Kafka)에 담아, 백엔드 서버가 감당할 수 있는 속도로 처리(Leaky Bucket 효과)하게 한다.
  4. **Dynamic Whitelist**: 사전에 인증된 '우수 고객' IP는 제한에서 예외 처리하여 핵심 충성 고객의 경험을 보호한다.

### 도입 체크리스트
- **Client Identification**: IP 주소는 변조하기 쉬우므로, API Key나 JWT 토큰 등 강력한 식별 수단을 사용하고 있는가?
- **Global vs Local**: 서버가 여러 대인 분산 환경에서 합산 횟수를 정확히 계산하고 있는가? (중앙 Redis 사용 권장)
- **Informative Response**: 429 에러 시 사용자에게 "잠시 후 시도해달라"는 명확한 안내와 재시도 가이드가 전달되는가?

### 안티패턴
- **Hard Limit Only**: 무조건 꽉 막아버리는 정책. 일시적인 트래픽 급증(Burst)도 용납하지 않으면 정상적인 서비스 이용이 불편해진다.
- **In-memory Counter**: 각 서버의 로컬 메모리에만 횟수를 기록하는 경우. 서버 10대로 분산되면 사용자는 10배의 요청을 보낼 수 있게 되어 제한의 의미가 사라진다.

- **📢 섹션 요약 비유**: 전교생이 운동장에 모일 때, 한 명씩만 줄 서게 강요(Hard Limit)하면 줄이 너무 길어지니, 가끔은 단체 입장(Burst)도 허용하는 유연함이 필요합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Availability)** | 서버 가동 시간 99.99% 달성 | 트래픽 폭주로 인한 연쇄 장애(Cascading Failure) 원천 차단 |
| **정량적 (Cost)** | 인프라 비용 절감 | 무분별한 스크래핑 봇의 트래픽을 차단하여 불필요한 서버 자원 낭비 방지 |
| **정성적 (Security)** | 계정 및 데이터 탈취 방어 | 자동화된 공격 도구의 접근 속도를 늦춰 피해 규모 최소화 |
| **정성적 (Reliability)** | 서비스 공정성 확보 | 특정 블랙 컨슈머의 자원 독점을 막아 전체 사용자 품질 균일화 |

### 미래 전망
- **AI-Based Dynamic Limiting**: AI가 실시간으로 서버의 CPU/메모리 여유와 트래픽 패턴을 분석하여, 한도(Limit)를 자동으로 늘리거나 줄이는 '자율 트래픽 제어' 기술이 보편화될 것이다.
- **Identity-Centric Throttling**: IP 기반이 아닌, 사용자의 과거 행동 점수(Trust Score)에 따라 실시간으로 권한 한도를 결정하는 초개인화 보안이 실현될 것이다.

### 참고 표준
- **RFC 6585**: 429 Too Many Requests 상태 코드 정의
- **Google API Design Guide - Quota**: 업계 표준 쿼터 관리 가이드
- **NIST SP 800-204**: Microservices 보안 및 트래픽 제어 지침

- **📢 섹션 요약 비유**: "욕심은 화를 부른다." 시스템도 인간처럼 감당할 수 있는 만큼만 받아들여야 오래갑니다. 트래픽 제어는 우리 서비스의 건강과 장수를 지키는 가장 현명한 '절제'의 미학입니다.
