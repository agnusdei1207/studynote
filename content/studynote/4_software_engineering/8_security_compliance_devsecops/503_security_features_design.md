# 503. 보안 기능 (Security Features)의 설계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보안 기능 (Security Features)의 설계는 소프트웨어가 제공하는 비즈니스 기능 외에, 정보의 기밀성·무결성·가용성(CIA)을 보장하기 위한 **인증, 인가, 암호화, 감사 등의 메커니즘을 시스템 아키텍처에 체계적으로 반영**하는 과정이다.
> 2. **가치**: 보안을 부가 기능이 아닌 시스템의 '기본 속성'으로 설계함으로써 취약점 수정 비용을 최소화(Shift-Left)하며, '최소 권한 원칙'과 '심층 방어'를 통해 공격 발생 시에도 피해 확산을 원천 차단한다.
> 3. **융합**: 행정안전부 47개 보안 약점 가이드의 '보안 기능' 분야와 직결되며, 현대 아키텍처에서는 제로 트러스트(Zero Trust) 및 MSA 보안 패턴과 결합되어 동적이고 정밀한 통제 체계를 구축한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 보안 기능 설계는 소프트웨어가 "무엇을 하는가(What to do)"를 넘어 "어떻게 안전하게 보호하는가(How to protect)"를 정의하는 단계다. 단순히 로그인 창을 만드는 것이 아니라, 비정상적인 접근을 어떻게 탐지하고, 중요 데이터를 어떤 알고리즘으로 암호화하며, 행위 이력을 어떻게 남길지 아키텍처 수준에서 결정한다.

- **💡 비유**: 보안 기능 설계는 **"건물을 지을 때 단순히 방을 만드는 것이 아니라, 각 방의 중요도에 따라 지문 인식기(인증)를 달고, 금고(암호화)를 설치하며, 복도마다 CCTV(감사)를 배치하는 보안 설계도를 그리는 것"**과 같습니다.

- **등장 배경**:
  1. **구현 중심 보안의 한계**: 코딩 단계에서 개별적으로 추가된 보안 기능은 일관성이 없고 우회하기 쉽다. 아키텍처 차원의 중앙 집중식 통제가 필요해졌다.
  2. **규제 준수 강제화**: ISMS-P, GDPR 등 법적 컴플라이언스가 '설계 단계에서의 보안성 검토'를 명시적으로 요구하기 시작했다.
  3. **비즈니스 로직 우회 공격 증가**: 기술적 취약점보다 로직의 허점을 노리는 공격이 늘어나면서, 기능 간의 상호작용을 고려한 보안 설계가 중요해졌다.

- **📢 섹션 요약 비유**: 사고가 난 뒤에 경비원을 고용하는 것이 아니라, 건물 설계 단계부터 방화벽 위치와 대피로를 계산하여 '태생적으로 안전한 건물'을 만드는 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 5대 핵심 보안 설계 요소 (Essential Security Features)

| 요소 (Component) | 역할 및 정의 | 핵심 메커니즘 | 비유 |
|:---|:---|:---|:---|
| **인증 (Authentication)** | 신원 확인 | MFA, OAuth 2.0, FIDO | 신분증 대조 |
| **인가 (Authorization)** | 권한 부여 및 통제 | RBAC, ABAC, ACL | 출입증 등급 확인 |
| **암호화 (Cryptography)** | 데이터 보호 | AES-256, RSA, SHA-256 | 비밀 금고 보관 |
| **감사 (Auditing)** | 행위 추적 및 기록 | 6하 원칙 로그, SIEM 연동 | CCTV 녹화 및 장부 기록 |
| **가용성 (Availability)** | 서비스 지속성 보장 | 부하 분산, 서킷 브레이커 | 비상 발전기 가동 |

---

### 보안 기능 설계 아키텍처 (Layered Security Design)

시스템의 각 계층에서 보안 기능이 어떻게 상호작용하는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      계층별 보안 기능 설계 및 연동 아키텍처                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 프레젠테이션 계층] ────▶ [2. 비즈니스 계층] ────▶ [3. 데이터 계층]           │
 │      │                          │                         │                 │
 │  - 입력값 검증 (XSS 방지)     - 인가 체크 (Interceptor)  - 데이터 암호화 (TDE) │
 │  - 세션 관리 (Secure Cookie) - 비즈니스 로직 유효성 검증  - 접근 이력 로깅       │
 │      │                          │                         │                 │
 │      └───────────┬──────────────┴────────────┬────────────┘                 │
 │                  ▼                           ▼                              │
 │        ┌──────────────────┐        ┌──────────────────┐                     │
 │        │   중앙 인증 서버   │        │   키 관리 센터    │                     │
 │        │   (IAM / SSO)    │        │   (KMS / HSM)    │                     │
 │        └──────────────────┘        └──────────────────┘                     │
 │                                                                             │
 │  ※ 설계 원칙: Default Deny (명시적 허용 외에는 모두 차단)                       │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 보안 기능은 파편화되지 않고 **계층화된 방어(Defense in Depth)** 체계를 갖춰야 한다. 1) 사용자 접점에서는 입력값의 무결성을 검증하고, 2) 핵심 로직 계층에서는 중앙 인증 서버(IAM)와 연동하여 사용자의 권한을 매번 확인한다. 3) 마지막 데이터 저장소에서는 민감 정보를 암호화하고 접근 로그를 남긴다. 특히 인증과 암호화 키 관리는 개별 서비스가 직접 수행하기보다 **중앙 집중식 서버(SSO, KMS)**를 통해 통제함으로써 보안 관리의 일관성과 효율성을 확보한다.

---

### 심층 동작 원리: 실패 안전 설계 (Fail-safe Design)

보안 기능이 오작동하거나 시스템에 에러가 발생했을 때의 대처 설계다.
1. **기본 거부 (Default Deny)**: 인증 로직에서 에러가 발생하면 "로그인 성공"이 아닌 "시스템 오류로 인한 거부"를 반환해야 한다.
2. **에러 마스킹 (Error Masking)**: 보안 실패 시 구체적인 이유(예: '비밀번호가 틀림')를 알려주는 대신 추상적인 메시지('아이디 또는 비밀번호를 확인하세요')를 보내 정보를 숨긴다.
3. **상태 보존 (Persistence)**: 시스템 다운 시에도 보안 설정값은 유지되어야 하며, 복구 시 보안 체크가 생략되지 않도록 시퀀스를 설계한다.

- **📢 섹션 요약 비유**: 은행 문이 고장 났을 때 문이 활짝 열리는 것이 아니라, 수리가 끝날 때까지 꽉 잠긴 상태로 유지되는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 포지티브(Positive) vs 네거티브(Negative) 보안 모델

| 비교 항목 | 포지티브 보안 모델 | 네거티브 보안 모델 |
|:---|:---|:---|
| **기본 철학** | **Allow-list (화이트리스트)** | **Deny-list (블랙리스트)** |
| **동작 원리** | 허용된 행위 외 모두 차단 | 금지된 행위 외 모두 허용 |
| **보안 수준** | 매우 높음 (미지의 위협 대응 가능) | 보통 (알려진 공격 위주 대응) |
| **관리 부담** | 높음 (정교한 정책 수립 필요) | 낮음 (공격 패턴만 업데이트) |
| **권장처** | **금융, 국가 핵심 시설, 권한 관리** | 일반 웹 트래픽 필터링 (WAF 초기 설정) |

---

### 융합 시너지: Privacy by Design (PbD)

보안 기능 설계는 개인정보 보호와 결합될 때 법적 시너지가 극대화된다.
- **최소 데이터 수집**: 기능 설계 시 주민번호 대신 대체 식별자를 사용하도록 아키텍처를 유도한다.
- **목적 기반 인가**: 단순히 '관리자' 권한이 아니라, '민감 정보 열람 사유'를 입력해야만 데이터가 복호화되는 로직을 설계에 반영한다.

- **📢 섹션 요약 비유**: 수사관이 영장(열람 사유) 없이 함부로 남의 집(데이터)을 뒤질 수 없게 만드는 엄격한 법 집행 절차를 시스템화하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 기업의 중앙 집중식 API 인가 설계

- **문제 상황**: 수십 개의 서비스가 각자 자체적으로 권한 체크를 수행하다 보니, 서비스마다 보안 수준이 다르고 일관된 권한 정책 변경이 불가능함.
- **의사결정 (보안 아키텍처 개선)**:
  1. **Policy Enforcement Point (PEP)**: API 게이트웨이 단계에서 모든 요청의 토큰(JWT) 유효성을 1차 검증하도록 중앙 집중화한다.
  2. **Policy Decision Point (PDP)**: 별도의 권한 관리 엔진(예: OPA - Open Policy Agent)을 두어, 복잡한 비즈니스 규칙(예: 평일 낮 시간에만 특정 데이터 접근 가능)을 중앙에서 관리한다.
  3. **암호화 표준화**: 모든 서비스 간 통신은 상호 TLS (mTLS)를 강제하고, 암호화 키는 하드웨어 보안 모듈(HSM)에서만 생성/관리하도록 설계 표준을 수립한다.
  4. **통합 감사 로그**: 모든 권한 요청과 결과 로그를 실시간으로 중앙 SIEM으로 전송하여 이상 접근을 실시간 모니터링한다.

### 도입 체크리스트
- **Completeness**: CIA 3요소 중 누락된 보안 기능이 없는가?
- **User Experience**: 보안 기능(MFA 등)이 비즈니스 생산성을 저해하지 않도록 적절히 설계되었는가? (Adaptive Auth 적용 고려)
- **Separation of Duties**: 보안 관리자와 시스템 관리자의 권한이 분리되어 설계되었는가?

### 안티패턴
- **단일 지점 신뢰 (Trusting Input)**: "내부망에서 오는 요청은 인증이 필요 없다"고 설계하는 행위. SSRF나 내부자 위협에 무방비가 된다.
- **하드코딩 보안**: 코드 내부에 비밀번호나 암호화 키를 직접 적어두는 설계. (환경 변수나 Secret Store 활용 필수)

- **📢 섹션 요약 비유**: "우리 직원이니까 믿어도 되겠지"라는 안일함이 기업을 망하게 하는 스파이를 키우는 길이 될 수 있음을 명심해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Cost)** | 보안 조치 비용 90% 절감 | 설계 단계에서 반영 시 사후 보완 작업 대비 압도적 효율 |
| **정량적 (Success)** | 공격 성공률 획기적 감소 | 다중 방어 및 최소 권한 적용으로 침투 성공 가능성 차단 |
| **정성적 (Compliance)** | 법적 리스크 제로화 | 국내외 보안 표준 및 법규를 설계 단계부터 완벽히 충족 |
| **정성적 (Reliability)** | 시스템 신뢰성 강화 | 데이터 변조나 권한 오남용 없는 탄탄한 비즈니스 환경 구축 |

### 미래 전망
- **Context-Aware Security**: AI가 사용자의 위치, 시간, 기기 상태, 행동 패턴을 실시간 분석하여 보안 강도를 동적으로 조절하는 지능형 설계가 주류가 될 것이다.
- **Cyber Resilience**: 공격을 완벽히 막는 것을 넘어, 공격을 당하는 도중에도 핵심 기능은 중단 없이 작동하게 하는 '회복력 중심의 보안 설계'로 진화할 것이다.

### 참고 표준
- **ISO/IEC 27034**: Application Security 표준 (보안 기능 가이드 포함)
- **NIST SP 800-160**: Systems Security Engineering (SSE)
- **행정안전부/KISA 소프트웨어 개발 보안 가이드**: 보안 기능 분야 16개 항목

- **📢 섹션 요약 비유**: 예전에는 튼튼한 성문(방화벽) 하나면 충분했지만, 이제는 성 안의 모든 보물 상자마다 각기 다른 퍼즐과 함정(지능형 보안 기능)을 설계해야 하는 정교한 보안 건축의 시대입니다.

2. 다 만든 다음에 자물쇠를 달려고 하면 상자가 망가질 수 있지만, 처음부터 그렇게 만들면 아주 튼튼하고 안전하답니다.
3. 이렇게 미리미리 안전 장치를 생각해서 물건을 만들면, 나중에 나쁜 사람이 뺏어가려고 해도 절대 열 수 없게 보호할 수 있어요!
