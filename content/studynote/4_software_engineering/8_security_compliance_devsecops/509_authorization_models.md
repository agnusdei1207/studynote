# 509. 인가(Authorization) 모델: RBAC 및 ABAC

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인가(Authorization) 모델은 인증된 주체가 특정 자원에 대해 수행할 수 있는 행위를 결정하는 보안 체계로, 정적인 역할 중심의 **RBAC**와 동적인 속성 조합 중심의 **ABAC**가 대표적이다.
> 2. **가치**: 조직의 복잡한 권한 요구사항을 체계적으로 구조화하여 '최소 권한 원칙'을 실현하며, 휴먼 에러에 의한 권한 남용을 방지하고 보안 감사를 위한 명확한 근거를 제공한다.
> 3. **융합**: 제로 트러스트(Zero Trust) 아키텍처의 핵심 판단 엔진인 PDP (Policy Decision Point)로 작동하며, 현대 클라우드 및 마이크로서비스(MSA) 환경에서 정밀한 접근 통제를 수행하는 표준 프레임워크로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **RBAC (Role-Based Access Control)**: 사용자를 개별적으로 관리하지 않고, '역할(Role)'이라는 바구니에 담아 권한을 부여한다. (예: 대리, 과장, 팀장 등)
  - **ABAC (Attribute-Based Access Control)**: 사용자, 자원, 환경의 '속성(Attribute)'들을 조합한 정책(Policy)으로 권한을 결정한다. (예: '인사팀' + '평일 낮' + '본사 접속' 시에만 접근 허용)

- **💡 비유**: 
  - **RBAC**: **"유니폼 기반 출입 통제"**와 같습니다. 경찰 유니폼을 입었으면 경찰서 내부를 다닐 수 있고, 간호사 유니폼을 입었으면 병원 약국에 들어갈 수 있는 식입니다.
  - **ABAC**: **"스마트 지문 인식 통제"**와 같습니다. 유니폼뿐만 아니라 오늘 출근 기록이 있는지, 현재 체온은 정상인지, 가방 안에 위험물은 없는지 등 여러 조건을 실시간으로 체크하여 문을 열어주는 방식입니다.

- **등장 배경**:
  1. **관리 대상의 폭증**: 사용자가 수만 명인 엔터프라이즈 환경에서 개별 권한 설정(ACL)은 불가능해졌고, 이를 그룹화(RBAC)할 필요가 생겼다.
  2. **동적 업무 환경**: 재택근무, 외주 협업 등 접속 위치와 시간대에 따른 유연한 권한 통제가 요구되면서 RBAC의 정적 구조가 한계에 부딪혔다. (ABAC의 탄생)
  3. **클라우드 및 마이크로서비스**: 자원이 수시로 생성되고 사라지는 환경에서, 이름이 아닌 '라벨(속성)' 기반의 자동화된 통제 체계가 절실해졌다.

- **📢 섹션 요약 비유**: 누구에게나 통하는 '마스터 키(RBAC)'를 줄 것인지, 상황에 따라 내용이 변하는 '스마트 카드(ABAC)'를 줄 것인지에 대한 통제 전략의 선택입니다.

## II. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. RBAC vs ABAC 아키텍처 비교

권한이 결정되는 논리적 구조의 차이를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      인가(Authorization) 모델별 결정 메커니즘                     │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. RBAC (역할 기반)]                                                        │
 │  사용자(User) ──▶ 역할(Role) ──▶ 권한(Permission) ──▶ 자원(Resource)           │
 │  (정적 맵핑)       (Admin)       (Read/Write)                                │
 │                                                                             │
 │ [2. ABAC (속성 기반)]                                                        │
 │  ┌────────────┐      ┌────────────┐      ┌────────────┐      ┌────────────┐ │
 │  │ 주체 속성  │      │ 자원 속성  │      │ 환경 속성  │      │ 행동 속성  │ │
 │  │ (소속,직급)│ +    │ (기밀도)   │ +    │ (시간,IP)  │ +    │ (Read,Del) │ │
 │  └──────┬─────┘      └──────┬─────┘      └──────┬─────┘      └──────┬─────┘ │
 │         └───────────┬───────┴─────────────┬───────┘                     │ │
 │                     ▼ [정책 엔진 (PDP)]     ◀─────────────────────────────┘ │
 │               "Is (Subject) allowed to do (Action)                          │
 │                on (Resource) in (Environment)?" ──▶ [Yes / No]              │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** **RBAC**는 계층이 단순하다. "팀장이면 결제 가능"이라는 규칙이 DB 테이블에 미리 매핑되어 있다. 관리가 매우 쉽지만, "우리 팀장이라도 집에서는 결제 못 하게 해줘"라는 예외 조건을 넣으려면 역할을 새로 만들어야 하는 **'역할 폭발 (Role Explosion)'** 문제가 발생한다. **ABAC**는 훨씬 복잡하지만 유연하다. 모든 요청 시점에 **PDP (Policy Decision Point)**가 주체, 자원, 환경 속성을 가공하여 즉석에서 가부를 판단한다. 이는 if-then 형태의 **'정책 애즈 코드 (Policy as Code)'**를 가능케 한다.

---

### 2. ABAC의 4대 속성 요소 (Attributes)

1. **주체 (Subject)**: 접근을 시도하는 주체의 속성 (ID, 역할, 부서, 보안 등급 등)
2. **자원 (Resource)**: 접근 대상의 속성 (문서 제목, 작성자, 기밀성 단계 등)
3. **환경 (Environment)**: 요청이 일어난 상황의 속성 (현재 시간, 위치, 위협 수준 등)
4. **행동 (Action)**: 수행하고자 하는 작업 (조회, 수정, 승인 등)

- **📢 섹션 요약 비유**: "인사팀 박 과장이(주체), 평일 오후 2시에(환경), 사무실에서(환경), 급여 명부를(자원), 수정하려(행동) 한다"는 전체 문장을 분석하여 도장을 찍어주는 판결 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### RBAC vs ABAC 상세 비교 매트릭스

| 비교 항목 | RBAC (Role-Based) | ABAC (Attribute-Based) |
|:---|:---|:---|
| **관리 단위** | 사용자 그룹 / 역할 | **중앙화된 정책 (Policy)** |
| **복잡도** | 낮음 (이해하기 쉬움) | 높음 (정책 언어 학습 필요) |
| **유연성** | 낮음 (정적임) | **매우 높음 (동적/세밀함)** |
| **확장성** | 역할 수 증가 시 관리 한계 | 속성 추가만으로 정책 확장 가능 |
| **비유** | 아파트 공동 현관 비번 | 상황에 따라 변하는 얼굴 인식 |

---

### 기술 융합: XACML과 OPA

ABAC를 실제로 구현하기 위한 표준 기술들이다.
- **XACML**: XML 기반의 전통적인 권한 정책 표준. 매우 강력하지만 무겁고 복잡함.
- **OPA (Open Policy Agent)**: 현대적 대안으로, **Rego**라는 경량 언어를 사용하여 클라우드 네이티브(K8s 등) 환경의 권한을 코드로 관리함.

- **📢 섹션 요약 비유**: 법전(XACML)을 들고 다니는 대신, 스마트폰 앱(OPA)으로 실시간 법률 해석을 받아 즉시 행동하는 효율성의 차이와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 의료 시스템의 개인정보 접근 제어 설계

- **문제 상황**: 의사는 환자 기록을 볼 수 있어야 하지만, 1) 자신이 담당하는 환자여야 하고 2) 진료 시간 내여야 하며 3) 외부 공공 Wi-Fi가 아닌 병원 전용망이어야 함.
- **의사결정 (인가 모델 선정 전략)**:
  1. **RBAC의 한계**: "담당 환자 여부"나 "진료 시간"과 같은 동적 조건을 RBAC(의사 역할)만으로는 해결할 수 없음.
  2. **ABAC 도입**: 속성 기반의 정책을 수립한다.
     - `Rule: Allow access IF (Subject.Role == Doctor) AND (Subject.ID == Resource.AttendingDoctor) AND (Env.Time in WorkingHours) AND (Env.IP == HospitalInternalIP)`
  3. **하이브리드 접근**: 기본적인 시스템 로그인은 RBAC로 처리하고, 민감 데이터인 '환자 기록' 조회 시에만 ABAC 엔진을 호출하여 정밀 검증을 수행한다.
  4. **결과**: 권한 관리를 위해 수천 개의 역할을 만들 필요 없이, 단 몇 줄의 정책 코드로 완벽한 보안과 법규 준수를 동시에 달성한다.

### 도입 체크리스트
- **Latency**: ABAC의 속성 조합 연산이 API 응답 속도에 영향을 주지는 않는가? (캐싱 및 고성능 엔진 필요)
- **Attribute Truth**: 정책 판단의 근거가 되는 '속성 데이터' 자체가 신뢰할 수 있는 소스(HR DB 등)에서 실시간으로 공급되는가?
- **Auditability**: 권한이 거부되었을 때, 어떤 속성값 때문에 거부되었는지 역추적하여 설명할 수 있는가?

### 안티패턴
- **역할 폭발 (Role Explosion)**: 예외 상황마다 `TeamLead_InOffice`, `TeamLead_Remote` 등 새로운 역할을 무한히 생성하는 행위. (관리가 불가능해짐)
- **데이터 기반 인가 누락**: 소스 코드 곳곳에 `if (user == writer)` 같은 하드코딩된 인가 로직을 분산시키는 행위. (중앙 집중식 정책 관리 부재)

- **📢 섹션 요약 비유**: "누구는 되고 누구는 안 된다"는 단순한 이분법(RBAC)에서 벗어나, "이런 조건일 때만 된다"는 입체적인 약속(ABAC)을 지키는 것이 데이터 경제 시대의 기본 상식입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Risk)** | 권한 남용 사고 90% 감소 | 상황에 맞는 정밀 통제를 통해 불필요한 노출 및 탈취 방어 |
| **정량적 (Audit)** | 감사 리소스 절감 | 중앙 집중식 정책 로그를 통해 "누가 왜 접근했는지" 즉시 증명 |
| **정성적 (Agility)** | 비즈니스 변경 대응력 | 코드 수정 없이 정책 파일(ABAC) 업데이트만으로 즉시 권한 조정 |
| **정성적 (Scale)** | 글로벌 확장성 확보 | 수만 명의 사용자와 기기에 대해 일관된 보안 거버넌스 유지 |

### 미래 전망
- **AI-Managed Authorization**: AI가 사용자의 평소 업무 패턴과 데이터 중요도를 학습하여, 위험 점수(Risk Score)를 실시간으로 속성에 반영하는 자율형 인가 시스템이 등장할 것이다.
- **Contextual Graph Identity**: 주체와 자원 간의 복잡한 관계망을 그래프 DB로 분석하여, 겉으로 드러나지 않는 연쇄적 권한 리스크를 사전에 차단하는 기술이 보편화될 것이다.

### 참고 표준
- **NIST SP 800-162**: Attribute Based Access Control (ABAC) Definition and Considerations
- **XACML 3.0**: Oasis 공식 인가 정책 언어 표준
- **Rego (Open Policy Agent)**: 클라우드 네이티브 정책 표준 언어

- **📢 섹션 요약 비유**: 수동으로 문을 열어주던 시대를 지나, 이제는 센서와 데이터가 보이지 않는 곳에서 우리를 지켜주는 '지능형 보안 환경'의 핵심 설계 원리가 바로 인가 모델입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **RBAC**는 학교에서 **"반장은 교무실에 들어올 수 있어"**라고 직책(역할)에 따라 문을 열어주는 거예요.
2. **ABAC**는 **"반장이라도, 숙제 검사 시간이고, 선생님이 계실 때만 들어와!"**라고 꼼꼼하게 조건을 따지는 거랍니다.
3. 이렇게 상황에 맞춰서 문을 열어주면, 우리 소중한 시험 문제(데이터)를 훨씬 더 안전하게 지킬 수 있어요!
