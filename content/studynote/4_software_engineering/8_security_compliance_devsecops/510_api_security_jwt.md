# 510. API 보안 관리: OAuth 2.0, OIDC, JWT

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: API 보안 관리는 분산된 시스템 환경에서 제3자 애플리케이션이나 모바일 기기가 서버 자원에 접근할 때, **OAuth 2.0(권한 위임), OIDC(신원 확인), JWT(자가 수용적 토큰)**를 결합하여 안전한 인증 및 인가 체계를 구축하는 기술이다.
> 2. **가치**: 사용자의 비밀번호를 직접 공유하지 않고도 특정 자원에 대한 접근 권한을 안전하게 위임(Delegation)하며, 토큰 자체에 담긴 정보를 통해 별도의 세션 DB 조회 없이도 고속의 무상태(Stateless) 검증을 실현한다.
> 3. **융합**: 마이크로서비스(MSA) 아키텍처의 핵심 보안 게이트웨이 기술이며, 토큰의 서명(Signature) 및 만료(Expiration) 검증을 통해 위변조와 재사용 공격을 방어하는 현대 웹 보안의 표준이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **OAuth 2.0**: "내 데이터를 타사 앱이 대신 가져가게 해줘"라는 **권한 위임** 표준.
  - **OIDC (OpenID Connect)**: OAuth 2.0 위에 **신원 확인(ID)** 기능을 얹은 것.
  - **JWT**: 인증 정보를 JSON 형태로 담아 디지털 서명한 **토큰 규격**.

- **💡 비유**: 
  - **OAuth 2.0**: 호텔 발렛 파킹 시 차 키 전체가 아닌 **발렛용 키(Access Token)**만 주는 것과 같습니다. (인가)
  - **OIDC**: 발렛 키를 줄 때 내 **신분증(ID Token)**을 보여주고 "내가 이 차 주인 맞다"고 확인받는 과정입니다. (인증)
  - **JWT**: 그 키에 "이 키는 1시간만 유효함"이라고 적혀 있고 호텔의 **공식 직인(서명)**이 찍혀 있는 것과 같습니다.

- **등장 배경**:
  1. **비밀번호 공유의 위험**: 타사 앱에 내 원본 계정 비밀번호를 알려줘야 했던 과거의 보안 위협을 해결해야 했다.
  2. **Stateless 서버 요구**: 수백만 명의 사용자를 처리하기 위해 서버 메모리에 세션을 저장하지 않는 가벼운 인증 방식이 필요해졌다.
  3. **모바일 및 API 경제**: 웹 브라우저를 넘어 다양한 앱과 서비스들이 서로의 API를 호출하는 생태계가 구축되면서 표준화된 보안 프로토콜이 절실해졌다.

- **📢 섹션 요약 비유**: 복잡한 디지털 세상에서 신분증(OIDC)과 출입증(OAuth)을 하나의 스마트 토큰(JWT)에 담아, 전 세계 어디서든 나를 증명하고 일하는 시스템입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. API 보안 3대 기술의 역할 분담

| 기술 | 주요 역할 | 산출물 | 비유 |
|:---|:---|:---:|:---|
| **OIDC** | 사용자가 누구인지 확인 (Authentication) | **ID Token** | 주민등록증 |
| **OAuth 2.0** | 자원에 대한 접근 권한 부여 (Authorization) | **Access Token** | 출입 허가증 |
| **JWT** | 인증 정보를 담는 표준 데이터 포맷 | JSON String | 봉인된 봉투 |

---

### 2. API 인증 및 인가 아키텍처 흐름

토큰이 발행되어 검증되는 전 과정을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      OAuth 2.0 / OIDC 및 JWT 기반 API 보안 메커니즘               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 로그인 및 권한 요청]                                                       │
 │  사용자 ──────▶ [인증 서버 (IdP)]                                             │
 │                 (Auth Server)                                               │
 │  ◀───(JWT 발급)──┘                                                           │
 │  (Access Token + ID Token)                                                  │
 │                                                                             │
 │ [2. API 호출]                                                                │
 │  사용자 ──────(HTTP Header: Authorization: Bearer <JWT>)──────▶ [API 서버]     │
 │                                                               (Resource)    │
 │                                                                             │
 │  [3. 서버 측 JWT 검증 로직]                                                    │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ 1. 서명 검증: "서버의 비밀키로 해시한 값과 토큰의 서명이 일치하는가?"         │  │
 │  │ 2. 만료 확인: "현재 시간이 exp(Expiration Time)를 넘지 않았는가?"           │  │
 │  │ 3. 권한 확인: "이 토큰의 scope에 요청한 기능이 포함되어 있는가?"             │  │
 │  └──────────────────┬────────────────────────────────────────────────────┘  │
 │                    │                                                        │
 │                    ▼ [검증 성공 시]                                          │
 │           ┌────────────────────────┐                                        │
 │           │  데이터 반환 (200 OK)   │                                        │
 │           └────────────────────────┘                                        │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 핵심은 **검증의 독립성**이다. 1) **인증 서버**는 사용자를 확인한 후 그의 권한 정보를 담은 **JWT**를 발행한다. 2) 사용자는 이 토큰을 들고 **API 서버**를 호출한다. 3) 이때 API 서버는 인증 서버에 매번 물어볼 필요가 없다. 토큰에 포함된 **서명(Signature)**을 자신의 공개키나 비밀키로 확인하기만 하면 된다. 서명이 맞고 **만료 시간**이 지나지 않았다면, 토큰 내부의 권한 정보(Scope)를 믿고 데이터를 즉시 내준다. 이 과정이 DB 조회 없이 0.1초 만에 이루어지는 것이 JWT의 강점이다.

---

### 심층 동작 원리: JWT의 3단계 구조와 무결성

1. **Header**: 알고리즘(HS256 등)과 토큰 타입 정보.
2. **Payload**: 실제 데이터(User ID, Role, Expire). **(Base64 인코딩일 뿐이므로 암호화된 것이 아님에 주의!)**
3. **Signature**: `Header + Payload + Secret Key`를 조합하여 생성.
4. **가치**: 누군가 Payload의 내용을 살짝 고치면(예: Role을 Admin으로), 서버에서 검증 시 Signature 값이 달라지므로 위변조가 즉시 탄로 난다.

- **📢 섹션 요약 비유**: 내용물이 비치는 투명한 편지봉투(Payload)이지만, 봉투 입구에 절대 위조할 수 없는 서버만의 마법 인장(Signature)이 찍혀 있는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Opaque Token vs JWT 비교

| 비교 항목 | 불투명 토큰 (Opaque) | 자가 수용적 토큰 (JWT) |
|:---|:---|:---|
| **정보 포함** | 없음 (의미 없는 문자열) | **있음 (사용자 정보 포함)** |
| **검증 방식** | 매번 인증 서버에 조회 (Introspection) | **서버 자체 로직으로 검증** |
| **보안성** | 토큰 자체로 정보 유출 없음 | 토큰 탈취 시 정보 노출 위험 |
| **확장성** | 낮음 (인증 서버 부하) | **매우 높음 (Stateless)** |
| **비유** | 은행 번호표 | 봉인된 성적표 |

---

### 기술 융합: API Gateway와 JWT

현대 MSA 보안의 핵심 연동 방식이다.
- **Gateway의 역할**: 외부에서 들어오는 모든 JWT를 게이트웨이에서 1차 검증하고, 내부 서비스들에는 검증된 정보만 'Plain Header'로 넘겨준다.
- **가치**: 수십 개의 마이크로서비스가 각자 복잡한 JWT 검증 로직을 가질 필요 없이, 입구에서 보안을 일원화한다.

- **📢 섹션 요약 비유**: 호텔 정문에서 가짜 초대장을 가진 사람을 미리 걸러내어(Gateway), 안방(백엔드 서비스)은 오직 진짜 손님만 맞이하게 하는 효율적인 경비 시스템입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 토큰 탈취 사고에 대비한 'Refresh Token' 설계

- **문제 상황**: JWT는 한 번 발행하면 서버에서 강제로 취소(Revoke)하기 어렵다는 단점이 있음. 만료 시간을 길게 잡으면 탈취 시 위험하고, 짧게 잡으면 사용자가 자꾸 로그인해야 함.
- **의사결정 (토큰 전략 고도화)**:
  1. **Access Token (Short-lived)**: 수명이 15분인 짧은 JWT를 발급하여 API 호출에 사용한다.
  2. **Refresh Token (Long-lived)**: 수명이 2주인 토큰을 발급하여 별도의 보안 저장소(HttpOnly Cookie 등)에 보관한다.
  3. **갱신 프로세스**: Access Token이 만료되면, Refresh Token을 인증 서버에 보내 새로운 Access Token을 받아온다.
  4. **강제 로그아웃**: 사용자가 기기를 분실하면 서버 DB에서 해당 Refresh Token을 삭제하여, 해커가 더 이상 새로운 Access Token을 발급받지 못하게 차단한다.

### 도입 체크리스트
- **Signature Algorithm**: HS256(대칭키)보다 보안성이 높은 **RS256(비대칭키)**을 사용하여 인증 서버와 API 서버의 키 노출 리스크를 분리했는가?
- **Sensitive Data**: JWT 페이로드에 비밀번호나 개인정보(주민번호 등)를 평문으로 담고 있지는 않은가?
- **None Algorithm**: 헤더의 `alg: "none"` 설정을 허용하여 서명 검증을 무력화하는 보안 허점이 있지는 않은가?

### 안티패턴
- **LocalStorage 저장**: 자바스크립트로 접근 가능한 LocalStorage에 JWT를 저장하는 행위. (XSS 한 번에 모든 계정 권한이 털림)
- **무기한 만료 시간**: "로그인이 귀찮다"는 이유로 토큰 수명을 1년으로 설정하는 행위. (탈취된 토큰이 영원한 마스터 키가 됨)

- **📢 섹션 요약 비유**: "잃어버려도 금방 쓸모없어지는 일회용 카드(Access)"와 "금고에 잘 보관했다가 카드를 새로 뽑는 마스터 카드(Refresh)"를 나누어 쓰는 똑똑한 자산 관리 전략입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Performance)** | 인증 속도 50% 향상 | 세션 DB 조회 및 네트워크 통신 제거를 통한 지연 시간 단축 |
| **정량적 (Scale)** | 서버 확장성 무제한 확보 | 서버 간 세션 공유(Clustering) 없이도 전 세계 서버에서 즉시 인증 가능 |
| **정성적 (Standard)** | 글로벌 연동 편의성 | 구글, 카카오 등 외부 플랫폼과의 간편한 소셜 로그인 구현 |
| **정성적 (Security)** | 계정 탈취 피해 국소화 | 짧은 만료 시간과 세밀한 Scope 관리를 통해 사고 영향 범위 제한 |

### 미래 전망
- **JWE (JSON Web Encryption)**: 페이로드의 내용까지 완전히 암호화하여 JWT 탈취 시에도 정보 유출이 0%인 기술이 보편화될 것이다.
- **AI-Driven Token Revocation**: AI가 사용자의 접속 위치와 패턴을 분석하여, 평소와 다른 곳에서 JWT가 쓰이면 즉시 Refresh Token을 무효화하는 능동적 방어가 실현될 것이다.

### 참고 표준
- **RFC 6749**: The OAuth 2.0 Authorization Framework
- **OpenID Connect Core 1.0**: OIDC 공식 명세서
- **RFC 7519**: JSON Web Token (JWT) 표준 규격

- **📢 섹션 요약 비유**: 열쇠(Password)를 주고받던 원시 시대에서, 디지털 서명과 유효 기간이 담긴 스마트 통행증(JWT)을 주고받는 고도의 신뢰 사회로 API 보안이 완성되었습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **OAuth 2.0**은 내 보물 상자를 친구에게 보여줄 때, **"딱 5분 동안만, 책만 읽을 수 있는 마법 열쇠"**를 빌려주는 거예요.
2. **JWT**는 그 열쇠에 **"이건 가짜가 아니야!"라고 적힌 선생님의 비밀 도장**이 찍혀 있는 거랍니다.
3. 이 도장 덕분에 친구는 선생님께 매번 물어보지 않아도 마음껏 책을 읽을 수 있고, 시간이 지나면 열쇠는 저절로 사라져서 아주 안전해요!
