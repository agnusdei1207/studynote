# 475. STRIDE 모델

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: STRIDE 모델은 Microsoft에서 고안한 **6가지 보안 위협 분류 체계**로, Spoofing(위장), Tampering(변조), Repudiation(부인), Information Disclosure(정보 유출), Denial of Service(서비스 거부), Elevation of Privilege(권한 상승)의 앞 글자를 따서 명명되었다.
> 2. **가치**: 보안 전문가가 아닌 개발자나 아키텍트도 시스템의 각 구성 요소(엔티티, 프로세스, 데이터 흐름, 저장소)에서 발생 가능한 위협을 누락 없이 식별할 수 있도록 직관적인 가이드라인을 제공한다.
> 3. **융합**: 위협 모델링의 '식별' 단계에서 DFD (Data Flow Diagram)와 결합하여 기술적 보안 요구사항을 도출하는 표준 도구로 활용되며, 식별된 위협은 DREAD 모델을 통해 우선순위가 결정된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: STRIDE는 시스템의 설계 단계에서 "어떤 것이 잘못될 수 있는가?"라는 질문에 답하기 위해 사용되는 위협 식별 프레임워크다. 정보 보안의 3요소(CIA: Confidentiality, Integrity, Availability)와 그 이상의 보안 속성을 공격자의 관점에서 재해석한 모델이다.

- **💡 비유**: STRIDE는 **"범죄 수사의 6대 유형 체크리스트"**와 같습니다. 경찰이 사건 현장을 조사할 때 '도난(변조)', '신분 위조(위장)', '알리바이 조작(부인)' 등 정해진 항목을 하나하나 대조하여 범죄의 단서를 놓치지 않는 것과 같은 원리입니다.

- **등장 배경**:
  1. **보안 약점의 표준화**: 위협을 체계적으로 분류하지 않으면, 분석가의 주관에 따라 특정 위협은 과하게 고려되고 다른 치명적인 위협은 무시되는 불균형이 발생한다.
  2. **비보안 전문가의 참여**: 개발 단계에서 보안을 내재화하기 위해서는 보안 전문가가 아니더라도 쉽게 이해하고 적용할 수 있는 보편적인 위협 모델이 필요했다.
  3. **Microsoft SDL의 핵심 도구**: 윈도우 운영체제의 보안성을 획기적으로 높이기 위해 MS가 SDL 프로세스에 통합하면서 전 세계적인 표준으로 자리 잡았다.

- **📢 섹션 요약 비유**: 복잡한 미로에서 길을 찾을 때, '왼쪽 벽만 타고 가기'처럼 단순하지만 확실한 규칙을 적용하여 빠짐없이 탐색하는 방식과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### STRIDE 6대 위협 요소 및 대응 보안 속성

STRIDE의 각 항목은 공격자가 침해하고자 하는 특정 보안 속성과 1:1로 매칭된다.

| 위협 (Threat) | 설명 | 위반되는 보안 속성 | 대응 방안 (예시) |
|:---|:---|:---|:---|
| **S: Spoofing (위장)** | 다른 사용자나 시스템인 것처럼 속임 | **인증 (Authentication)** | 강력한 인증 (MFA), 디지털 서명 |
| **T: Tampering (변조)** | 데이터나 코드를 무단으로 수정함 | **무결성 (Integrity)** | 해시(Hash), HMAC, 접근 제어 |
| **R: Repudiation (부인)** | 행위 자체를 부정함 (영수증 조작 등) | **부인 방지 (Non-repudiation)** | 로깅/감사, 전자 서명 |
| **I: Info Disclosure (유출)** | 인가되지 않은 정보 노출 | **기밀성 (Confidentiality)** | 암호화 (AES), 최소 권한 원칙 |
| **D: DoS (서비스 거부)** | 시스템을 마비시켜 이용 불가하게 함 | **가용성 (Availability)** | 처리율 제한(Rate Limit), 필터링 |
| **E: Elev. Priv (권한 상승)** | 낮은 권한으로 높은 권한 행위 수행 | **권한 부여 (Authorization)** | RBAC, 최소 권한 설정 |

---

### STRIDE와 DFD (Data Flow Diagram) 요소 매핑 아키텍처

STRIDE는 DFD의 구성 요소별로 적용 가능한 위협이 다르다는 특징을 이용하여 분석 효율을 높인다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      STRIDE - DFD 구성 요소별 위협 매핑                         │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │    [DFD 요소]              [적용 가능한 STRIDE 위협]           [보안 속성]      │
 │  ┌────────────┐           ┌──────────────────────┐       ┌──────────────┐   │
 │  │ 외부 엔티티 │           │  S  ,      R         │       │ 인증, 부인방지 │   │
 │  │  (사용자)  │           └──────────────────────┘       └──────────────┘   │
 │  └────────────┘                      │                          │           │
 │        │                             ▼                          │           │
 │     [데이터 흐름]          ┌──────────────────────┐       ┌──────────────┐   │
 │  (네트워크 패킷)          │  T  ,  I  ,  D       │       │ 무결성,기밀성,가용성 │  │
 │        │                   └──────────────────────┘       └──────────────┘   │
 │        ▼                             │                          │           │
 │    [프로세스]             ┌──────────────────────┐       ┌──────────────┐   │
 │    (코드 실행)            │ S, T, R, I, D, E     │       │ 전체 보안 속성 │   │
 │        │                   └──────────────────────┘       └──────────────┘   │
 │        ▼                             │                          │           │
 │    [데이터 저장소]          ┌──────────────────────┐       ┌──────────────┐   │
 │    (DB, 파일)             │  T  ,  I  ,  D       │       │ 무결성,기밀성,가용성 │  │
 │                           └──────────────────────┘       └──────────────┘   │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** DFD의 각 노드에 STRIDE 위협을 무작위로 대입하는 것이 아니라, 요소의 특성에 맞는 위협을 선별적으로 대입한다. 예를 들어, **외부 엔티티(사용자)**는 인증의 대상이므로 '위장(S)'과 '부인(R)' 위협이 핵심이다. 반면, **프로세스(프로그램)**는 시스템의 모든 자원에 접근할 수 있는 핵심 주체이므로 STRIDE의 모든 항목(S, T, R, I, D, E)이 검토 대상이 된다. **데이터 흐름**과 **저장소**는 전송/보관 중인 데이터의 보안이 중요하므로 '변조(T)', '유출(I)', '가용성(D)' 위협에 집중한다. 이 매핑 규칙을 통해 분석 시간을 단축하고 정확도를 높일 수 있다.

---

### 심층 동작 원리: 위협 구체화 과정

1. **엔티티 식별**: DFD를 통해 분석 대상 시스템의 경계를 확정한다.
2. **STRIDE 대입**: 위 매핑 표에 따라 각 요소에 해당 위협을 대입한다.
   - *질문 예시*: "외부 사용자가 관리자인 것처럼 속일 수 있는가? (Spoofing)"
3. **위협 시나리오 작성**: "공격자가 세션 쿠키를 탈취하여(I) 다른 사용자로 위장(S)한 뒤, 관리자 권한으로 상승(E)하여 DB 데이터를 변조(T)함"과 같은 상세 흐름을 작성한다.
4. **보안 컨트롤 매핑**: 각 위협을 방어하기 위한 기술적 대책(SSL/TLS, MFA, HMAC 등)을 설계에 반영한다.

- **📢 섹션 요약 비유**: 건강 검진 시 모든 사람에게 똑같은 검사를 하는 것이 아니라, 위(저장소)는 내시경, 심장(프로세스)은 초음파 등 부위별로 가장 적합한 정밀 검사 항목을 골라 실행하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### STRIDE 항목별 융합 비교 (보안 3요소 연계)

| STRIDE 요소 | CIA 3요소 대응 | 주요 공격 사례 | 현대적 방어 기술 |
|:---|:---|:---|:---|
| **S / E** | **인증 / 권한** | 세션 하이재킹, 무차별 대입 | OAuth 2.0, FIDO2, RBAC |
| **T / R** | **무결성 / 부인방지** | 웹 파라미터 변조, 로그 삭제 | 전자서명, Blockchain (H/W Root of Trust) |
| **I** | **기밀성** | 패킷 스니핑, SQL 인젝션 | 데이터 마스킹, AES-256 암호화 |
| **D** | **가용성** | HTTP Flood, Slowloris | DDoS 완화 서비스 (WAF), Auto-scaling |

---

### STRIDE vs PASTA (위협 식별 관점의 비교)

STRIDE는 기술적/상세 위협 식별에 강점이 있고, PASTA는 비즈니스 영향도와 위험 수용 여부를 결정하는 거버넌스적 관점이 강하다.

- **📢 섹션 요약 비유**: STRIDE가 자동차 부품 하나하나의 결함을 찾는 정밀 검사라면, PASTA는 차 사고가 났을 때 보험금과 수리비를 계산하여 폐차할지 고칠지 결정하는 경제적 판단과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 모바일 뱅킹 송금 기능의 위협 모델링

- **문제 상황**: 모바일 뱅킹 앱에서 송금 요청 시 금액을 0원으로 바꾸거나, 타인의 계좌에서 돈이 빠져나가게 하려는 시도가 예상됨.
- **의사결정 (STRIDE 적용)**:
  1. **T (Tampering)**: 클라이언트에서 서버로 가는 패킷 내 금액 필드를 변조할 수 있음. → **해결**: 패킷 전체에 대한 HMAC 검증 및 서버 측 최종 금액 재검증.
  2. **S (Spoofing)**: 공격자가 가짜 인증 토큰을 만들어 서버에 전송. → **해결**: 서명된 JWT(JSON Web Token) 및 기기 바인딩 인증 적용.
  3. **R (Repudiation)**: 송금 후 "내가 보낸 적 없다"고 주장하며 환불 요구. → **해결**: 송금 시점에 사용자 바이오 인증 값과 함께 전자 서명된 로그 보관.

### 도입 체크리스트
- **신뢰 경계 설정**: 신뢰할 수 없는 외부(인터넷)와 내부망 사이의 경계가 명확히 표시되었는가?
- **누락 확인**: DFD의 모든 요소에 대해 STRIDE 체크리스트를 최소 1회 이상 대조했는가?
- **도구 활용**: MS Threat Modeling Tool 등을 활용하여 분석 결과의 일관성을 확보했는가?

### 안티패턴
- **단순 나열**: "Spoofing 위험이 있음"이라고만 적고, 구체적으로 어떤 데이터를 어떻게 속이는지에 대한 설명이 없는 경우.
- **위협 과소평가**: "내부망이라서 DoS(서비스 거부) 공격은 없을 것"이라며 특정 항목을 임의로 제외하는 행위.

- **📢 섹션 요약 비유**: "도둑 조심하세요"라는 말만 하는 것이 아니라, "2층 창문의 걸쇠가 헐거우니 이쪽으로 들어올 수 있습니다"라고 구체적으로 짚어줘야 실질적인 방어가 가능하다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Quality)** | 보안 요구사항 누락 방지 | 설계 단계에서 보안 요구사항 도출률 80% 이상 향상 |
| **정량적 (Cost)** | 결함 수정 비용 절감 | 구현 전 설계 수정을 통해 사후 사고 조치 비용 1/10 수준으로 절감 |
| **정성적 (Standard)** | 공통 보안 언어 확립 | 개발-보안-운영팀 간의 위협에 대한 공통 이해 기반 마련 |
| **정성적 (Education)** | 보안 인식 제고 | 체크리스트를 채워가는 과정 자체가 개발자에게 보안 교육 효과 제공 |

### 미래 전망
- **AI-Enhanced STRIDE**: AI가 소스 코드의 아키텍처를 분석하여 자동으로 DFD를 그리고, 최신 취약점 데이터베이스와 매핑된 STRIDE 위협 리포트를 즉시 생성해줄 것이다.
- **IoT/CPS 특화 STRIDE**: 하드웨어 제어권 탈취나 물리적 파괴 위협을 포함하는 CPS(Cyber-Physical Systems)용 STRIDE 확장 모델이 표준화될 것이다.

### 참고 표준
- **NIST SP 800-154**: 위협 모델링 가이드
- **IEEE 26702**: 시스템 공학 프로세스 내 보안 활동 가이드

- **📢 섹션 요약 비유**: 흑백 지도(수동 분석)가 실시간으로 교통 정보와 사고 구역을 알려주는 스마트 내비게이션(AI STRIDE)으로 진화하여, 사고 없는 디지털 항해를 도울 것입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 우리 집 보안을 위해 **"도둑이 들어올 수 있는 6가지 방법"**이라는 마법의 체크리스트를 들고 집 안 구석구석을 살펴보는 거예요.
2. "가짜 열쇠를 쓰나?(위장)", "창문을 깨나?(변조)", "몰래 훔쳐보나?(유출)" 같은 질문을 던지며 위험한 곳을 찾아요.
3. 이 6가지 질문에 다 대답하고 나면, 우리 집을 아주 튼튼한 성으로 만들 수 있는 완벽한 계획이 세워진답니다!
