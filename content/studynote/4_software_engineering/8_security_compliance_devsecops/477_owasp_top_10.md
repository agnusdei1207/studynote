# 477. OWASP Top 10 (2021 기준 주요 취약점)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OWASP Top 10은 전 세계 보안 전문가들의 의견과 실제 데이터를 바탕으로 웹 애플리케이션에서 발생하는 가장 치명적인 **10가지 보안 취약점 유형**을 3~4년 주기로 발표하는 표준 리포트다.
> 2. **가치**: 보안 위협의 우선순위를 제시하여 기업이 한정된 자원으로 최대의 방어 효과를 낼 수 있도록 돕는 실무 가이드라인이며, PCI-DSS 등 주요 보안 컴플라이언스의 핵심 준거 기준으로 활용된다.
> 3. **융합**: 2021년판은 '디자인 단계의 보안(Insecure Design)'을 신설하는 등 Secure SDLC의 상위 단계(Shift-Left) 활동을 강조하며, 단순 기술적 결함을 넘어 아키텍처와 비즈니스 로직 전반의 보안 내재화를 요구하고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **概念**: OWASP (Open Web Application Security Project) Top 10은 웹 보안의 '사실상 표준(De-facto Standard)'으로 통하는 문서다. 전 세계에서 수집된 수십만 건의 취약점 데이터를 분석하여, 발생 빈도와 영향도가 가장 높은 10개 카테고리를 선정한다.

- **💡 비유**: OWASP Top 10은 **"전 세계에서 가장 많이 발생하는 10대 질병 리포트"**와 같습니다. 보건 당국이 이 리포트를 보고 예방 접종 우선순위를 정하듯, 개발팀은 이 리포트를 보고 가장 위험한 보안 구멍부터 메우게 됩니다.

- **등장 배경**:
  1. **폭발적인 웹 서비스 증가**: 웹이 비즈니스의 중심이 되면서 공격 대상(Attack Surface)이 급격히 늘어났고, 일관된 보안 기준이 필요해졌다.
  2. **보안 약점의 가시화**: 수많은 취약점 중 무엇이 정말 위험한지 우선순위를 명확히 하여 개발 현장의 혼란을 줄이고자 했다.
  3. **패러다임의 변화 반영**: 2017년판 대비 2021년판은 단순 인젝션 위주의 공격에서 권한 제어와 설계 결함 위주로 보안의 초점이 이동했음을 보여준다.

- **📢 섹션 요약 비유**: 유행하는 전염병이 매년 바뀌듯, 해커들의 공격 수법도 시대에 따라 변하므로 주기적으로 '최신 보안 유행 리포트'를 확인해야 한다는 원리와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### OWASP Top 10: 2021년 주요 항목 분석

2021년판은 기존의 개별 취약점 나열 방식에서 '근본 원인' 중심의 카테고리 통합 방식으로 개편되었다.

| 순위 | 항목명 (2021 기준) | 주요 내용 및 특징 | 비유 |
|:---|:---|:---|:---|
| **A01** | **Broken Access Control** | 인가되지 않은 데이터 접근 및 권한 탈취 | 열쇠 없는 방 출입 |
| **A02** | **Cryptographic Failures** | 민감 데이터 노출 및 암호화 실패 | 투명한 금고 |
| **A03** | **Injection** | SQL, OS 커맨드 주입 등 (기존 1위에서 하락) | 불량 재료 투입 |
| **A04** | **Insecure Design** | **[신설]** 설계 단계의 보안 결함 (아키텍처 문제) | 부실한 건물 설계도 |
| **A05** | **Security Misconfiguration** | 잘못된 보안 설정, 기본 패스워드 사용 | 열려 있는 뒷문 |
| **A06** | **Vulnerable Components** | 취약한 오픈소스 및 라이브러리 사용 | 중고 불량 부품 사용 |
| **A07** | **Auth Failures** | 인증 실패 (패스워드 무차별 대입 등) | 가짜 신분증 통과 |
| **A08** | **Integrity Failures** | SW 및 데이터 무결성 검증 실패 (CI/CD 보안) | 오염된 소스 코드 |
| **A09** | **Logging/Monitoring** | 불충분한 로깅 및 모니터링 실패 | CCTV 고장 |
| **A10** | **SSRF** | 서버 측 요청 위조 (서버 내부 자원 공격) | 내부자 사칭 요청 |

### 보안 패러다임의 변화: 2017 vs 2021 아키텍처 비교

2021년판은 기술적 결함(Implementation)보다 관리적/설계적(Architecture/Management) 결함을 더 상위로 배치했다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      OWASP Top 10 패러다임의 진화 (2017 vs 2021)              │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │    [2017년 - 기술 결함 중심]          [2021년 - 근본 원인/설계 중심]            │
 │  ┌──────────────────────┐           ┌──────────────────────┐                │
 │  │ 1. Injection (SQLi)  │           │ 1. Access Control    │ ◀── 권한 강화   │
 │  ├──────────────────────┤           ├──────────────────────┤                │
 │  │ 2. Broken Auth       │ ────▶     │ 2. Crypto Failures   │ ◀── 데이터 보호 │
 │  ├──────────────────────┤           ├──────────────────────┤                │
 │  │ 3. Sensitive Data    │           │ 4. Insecure Design   │ ◀── [New] 설계  │
 │  └──────────────────────┘           └──────────────────────┘                │
 │                                                                             │
 │  [주요 특징]                                                                 │
 │  - 개별 기술(XSS 등) 통합 ──▶ 큰 카테고리(Injection 등)로 묶임                   │
 │  - 개발 후반부 점검 ──▶ 설계/기획 단계(Shift-Left) 보안 강조                     │
 │  - 단순 공격 방어 ──▶ 무결성 및 공급망 보안(A08) 영역 확장                       │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 위 다이어그램은 보안의 초점이 어떻게 이동했는지 보여준다. 과거에는 SQL 인젝션처럼 특정 코딩 실수를 잡아내는 것이 1순위였으나, 현재는 **'취약한 접근 제어(Access Control)'**가 1위로 올라섰다. 이는 클라우드와 MSA 환경에서 복잡해진 권한 관리가 가장 큰 위험 요소가 되었음을 시사한다. 특히 **'Insecure Design'**의 신설은 코드가 아무리 깨끗해도 설계 자체가 잘못되면(예: 비밀번호 복구 로직의 논리적 결함) 보안이 무너진다는 '시프트 레프트(Shift-Left)' 철학을 반영한 것이다.

---

### 심층 동작 원리: 데이터 기반의 순위 산정 (Data-Driven)

1. **취약점 발생 빈도 (Occurrence Rate)**: 실제 스캔 및 테스트에서 얼마나 자주 발견되는가?
2. **공격 용이성 (Exploitability)**: 공격자가 해당 취약점을 이용하기 얼마나 쉬운가?
3. **기술적 영향도 (Technical Impact)**: 사고 발생 시 시스템에 입히는 타격은 어느 정도인가?
4. **설문 데이터 결합**: 통계 데이터에 나타나지 않는 미래의 위협을 위해 전문가들의 설문 조사를 반영한다. (A10 SSRF가 이 방식으로 선정됨)

- **📢 섹션 요약 비유**: 건강 검진 데이터 통계(빈도)에 최신 의학 논문(설문)을 더해 올해의 10대 주의 질병을 정하는 정밀한 분석 과정과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### OWASP Top 10 vs CWE (Common Weakness Enumeration) 비교

| 비교 항목 | OWASP Top 10 | CWE (보안 약점 사전) |
|:---|:---|:---|
| **성격** | **우선순위 리포트** (인식 제고용) | **포괄적 분류 체계** (백과사전식) |
| **개수** | 10개 카테고리 (압축적) | 900개 이상의 개별 약점 (상세함) |
| **용도** | 교육, 거버넌스, 최소 보안 준수 기준 | 정밀 진단 도구(SAST)의 룰셋, 기술 분석 |
| **관계** | OWASP의 각 항목은 여러 개의 CWE를 포함함 | CWE는 OWASP를 분류 체계 중 하나로 참조함 |

---

### 융합 시너지: 클라우드 보안 환경으로의 확장

현대 웹 보안은 단순 브라우저-서버 통신을 넘어 컨테이너, API, 서버리스 환경으로 확장되었다.
- **A06 (Vulnerable Components)**: 오픈소스 보안뿐 아니라 Docker 이미지 내의 취약점 스캔으로 확장.
- **A10 (SSRF)**: 클라우드 메타데이터 서버에 접근하여 인스턴스 권한을 탈취하는 공격으로 발전.

- **📢 섹션 요약 비유**: 집 안의 도둑만 걱정하다가, 이제는 집으로 오는 택배 상자 속의 위험(공급망)과 원격 제어 가전제품의 해킹(클라우드)까지 걱정해야 하는 시대적 변화를 반영한 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 신규 서비스 런칭 전 '최소 보안 가이드라인' 수립

- **문제 상황**: 일정이 촉박한 상황에서 모든 보안 약점을 검토할 수 없음.
- **의사결정 (OWASP Top 10 활용)**:
  1. **A01/A07 집중**: 로그인이 필요한 모든 페이지에 인가 체크 로직(Interceptor)이 있는지 전수 점검한다.
  2. **A03 (Injection) 방어**: MyBatis/JPA 사용 시 파라미터 바인딩 원칙을 강제하여 SQLi를 원천 차단한다.
  3. **A06 (Supply Chain)**: 빌드 파이프라인에 SCA(Software Composition Analysis) 도구를 넣어 알려진 취약점이 있는 라이브러리 사용 시 빌드를 중단시킨다.

### 도입 체크리스트
- **가시성**: 우리 조직의 보안 가이드라인이 OWASP Top 10의 최신 버전을 반영하고 있는가?
- **도구 연동**: 사용하는 보안 스캐너(SAST/DAST)가 OWASP Top 10 기준의 리포팅을 지원하는가?
- **교육**: 개발자들이 각 항목의 개념과 방어 코드를 실무 수준에서 이해하고 있는가?

### 안티패턴
- **목록 맹신**: "Top 10에 없으니 이 취약점은 고치지 않아도 된다"고 판단하는 행위. Top 10은 최소한의 기준일 뿐이다.
- **2017년판 고수**: 여전히 예전 기준(예: XSS를 별도 항목으로 관리 등)에 머물러 최신 공격 트렌드를 놓치는 행위.

- **📢 섹션 요약 비유**: 시험에 꼭 나오는 핵심 10문제만 공부하고 "나는 만점을 받을 거야"라고 자신하는 것은 위험하며, 최소한 80점은 맞기 위한 전략으로 이해해야 한다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Coverage)** | 주요 취약점 80% 이상 예방 | 상위 10개 항목만 방어해도 전체 웹 보안 사고의 대다수 예방 가능 |
| **정량적 (Audit)** | 컴플라이언스 대응 속도 | ISMS, PCI-DSS 등 외부 감사 시 보안 대응의 객관적 근거 확보 |
| **정성적 (Standard)** | 글로벌 표준 호환성 | 해외 파트너사나 고객사에게 글로벌 수준의 보안성을 증명 |
| **정성적 (Shift-Left)** | 설계 단계 보안 강화 | 'Insecure Design' 인식을 통해 개발 초기 결함 감소 |

### 미래 전망
- **AI-Specific OWASP Top 10**: LLM 애플리케이션의 확산에 따라 'Prompt Injection' 등이 포함된 AI 특화 Top 10이 별도로 중요해질 것이다. (이미 OWASP Top 10 for LLM 존재)
- **API-First Security**: 웹 페이지 중심에서 API 중심(A01, A10 강조)으로 보안의 무게추가 완전히 이동할 것이다.

### 참고 표준
- **PCI-DSS**: 카드 결제 보안 표준 (OWASP Top 10 준수 필수)
- **KISA 정보보호 관리체계 (ISMS-P)**: 국내 보안 인증의 애플리케이션 보안 심사 기준

- **📢 섹션 요약 비유**: 과거에는 성벽만 높게 쌓으면 됐지만(인젝션 방어), 이제는 성 내부의 복잡한 통로와 출입증 관리(권한/설계)가 더 중요해진 요새 방어 전략과 같습니다.
