# 480. Injection (인젝션 / SQLi, OS Command, NoSQL 등)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인젝션 (Injection)은 신뢰할 수 없는 외부 입력값이 애플리케이션의 인터프리터나 쿼리 엔진에 전달될 때, 이를 데이터가 아닌 **실행 가능한 명령어로 오인하여 처리**함으로써 발생하는 보안 취약점이다.
> 2. **가치**: 한때 OWASP Top 10 부동의 1위였을 만큼 파괴력이 크며, 단 한 번의 성공으로 데이터베이스 전체 탈취(SQLi)나 서버 장악(OS Command Injection)이 가능하여 가장 즉각적인 대응이 필요한 항목이다.
> 3. **융합**: 고전적인 SQL Injection 외에도 NoSQL, LDAP, XML(XXE), OS Command 등 다양한 경로로 확장되고 있으며, 현대적 방어의 핵심은 '입력값 필터링'이 아닌 '명령어와 데이터의 완전한 분리(Parameterized Query)'에 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 인젝션은 "데이터와 명령의 혼동"에서 비롯된다. 애플리케이션이 외부 사용자의 입력을 받아 쿼리문이나 명령행 인수를 만들 때, 입력값에 포함된 특수문자(`, ;, &, | 등)가 쿼리의 구조를 변경하여 개발자가 의도하지 않은 악의적인 동작을 수행하게 만든다.

- **💡 비유**: 인젝션은 **"음식 주문서(쿼리)에 '피자 1판(데이터)'이라고 적는 대신, '피자 1판, 그리고 금고 열쇠를 카운터에 두세요(악의적 명령)'라고 적어 보내 요리사가 그대로 따르게 만드는 상황"**과 같습니다.

- **등장 배경**:
  1. **동적 쿼리 생성 방식**: 과거에 문자열 더하기(+) 방식으로 SQL문을 생성하던 관행이 인젝션의 온상이 되었다.
  2. **이기종 시스템 연동**: 웹 애플리케이션이 DB뿐만 아니라 시스템 쉘, 디렉터리 서비스(LDAP), NoSQL 등 다양한 인터프리터와 소통하면서 공격 표면이 넓어졌다.
  3. **공격 도구의 대중화**: `sqlmap` 같은 자동화 도구가 보급되면서 초보 해커들도 복잡한 인젝션 공격을 손쉽게 수행할 수 있게 되었다.

- **📢 섹션 요약 비유**: 편지 봉투에 적힌 주소(데이터)를 우체부가 읽고 배달하는 것이 아니라, 주소 자리에 적힌 "이 편지를 읽는 즉시 현금을 송금하라"는 명령을 우체부가 직접 실행해버리는 사고와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 인젝션의 주요 유형 및 공격 메커니즘

| 유형 | 공격 대상 | 공격 원리 | 비유 |
|:---|:---|:---|:---|
| **SQL Injection** | 관계형 DB (RDBMS) | `OR '1'='1'` 등을 삽입하여 인증 우회 및 데이터 조회 | 잠긴 문 뒤에서 "내가 주인이다"라고 거짓말하기 |
| **OS Command Inj.** | 서버 운영체제 쉘 | `; rm -rf /` 등을 삽입하여 시스템 명령어 강제 실행 | 택배 송장에 "이 집을 폭파하라"고 적기 |
| **NoSQL Injection** | MongoDB 등 NoSQL | `{"$gt": ""}` 등을 삽입하여 조건절 무력화 | "0보다 큰 숫자 다 내놔"라고 떼쓰기 |
| **LDAP Injection** | 디렉터리 서비스 | `*` 등을 삽입하여 사용자 정보 대량 검색 | 명부에서 모든 이름에 체크 표시하기 |

---

### SQL Injection 공격 및 방어 아키텍처 (Parameterized Query)

가장 효과적인 방어책은 데이터와 쿼리 템플릿을 분리하는 것이다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      SQL Injection 방어: Prepared Statement 원리               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [공격자 입력] : ' OR '1'='1                                                   │
 │                                                                             │
 │  1. 취약한 방식 (문자열 결합)                                                   │
 │     query = "SELECT * FROM users WHERE id = '" + input + "'";               │
 │     ▶ 결과: SELECT * FROM users WHERE id = '' OR '1'='1' (전체 조회 성공!)     │
 │                                                                             │
 │  2. 안전한 방식 (Parameterized Query)                                        │
 │     query = "SELECT * FROM users WHERE id = ?";   <── 쿼리 미리 컴파일(고정)     │
 │     stmt.setString(1, input);                      <── 입력값은 단순 데이터로 취급 │
 │                                                                             │
 │  [동작 아키텍처]                                                              │
 │  ┌───────────────┐          ┌──────────────────┐          ┌───────────────┐ │
 │  │ 애플리케이션  │ ──(Template)─▶ │   DB 엔진       │ ◀─(Validate)─ │  데이터베이스 │ │
 │  │ (Pre-compile) │ ──(Data)─────▶ │ (Query Plan 고정)│          │               │ │
 │  └───────────────┘          └──────────────────┘          └───────────────┘ │
 │                                                                             │
 │  ※ DB 엔진은 ? 자리에 들어오는 값을 명령어로 해석하지 않고 오직 문자열로만 처리함    │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 취약한 방식은 사용자 입력이 들어온 후에야 전체 쿼리문이 완성되므로, 입력값이 쿼리의 구조를 바꿀 수 있다. 반면 **Prepared Statement (또는 Parameterized Query)**는 DB 엔진에 쿼리의 템플릿(`SELECT ... WHERE id = ?`)을 먼저 보낸다. DB 엔진은 이 템플릿에 대한 실행 계획(Query Plan)을 미리 세워 고정시킨다. 이후 전달되는 사용자 입력(`input`)은 그 구조 안에 갇힌 '단순한 문자열 데이터'로만 취급되므로, 어떠한 특수문자가 포함되어도 명령어로 실행될 수 없다. 이것이 인젝션을 막는 가장 근본적이고 강력한 아키텍처적 장치다.

---

### 심층 동작 원리: Blind SQL Injection (중급 이상)

쿼리 결과가 화면에 직접 보이지 않을 때 사용하는 기법이다.
1. **Boolean-based**: `1=1`일 때와 `1=2`일 때의 페이지 반응 차이를 보고 데이터를 한 글자씩 추론한다.
2. **Time-based**: `sleep(5)` 명령을 인젝션하여, 응답이 5초 뒤에 오는지 확인함으로써 조건문이 참인지 거짓인지 판별한다.

- **📢 섹션 요약 비유**: 상대방의 눈을 가리고 질문했을 때, 대답 대신 "맞으면 박수를 치고(Boolean)", "맞으면 10초 뒤에 대답해달라(Time-based)"고 약속하여 정보를 조금씩 캐내는 심문 기법과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SQL Injection vs NoSQL Injection 비교

| 비교 항목 | SQL Injection | NoSQL Injection (MongoDB 예시) |
|:---|:---|:---|
| **공격 수단** | SQL 특수문자 (`, --, #, ;) | JSON 객체, 연산자 ($gt, $ne, $where) |
| **방어 핵심** | Prepared Statement 사용 | 라이브러리 내장 필터링, 스키마 유효성 검사 |
| **발생 원인** | 문자열 결합을 통한 쿼리 생성 | 입력값을 그대로 쿼리 객체(Object)로 사용 |
| **위험도** | 데이터 전체 유출, 권한 탈취 | 데이터 조회 우회, 스크립트 실행(Javascript) |

---

### 현대적 기술 융합: ORM (Object-Relational Mapping)과 인젝션

JPA, Hibernate, Sequelize 같은 ORM 프레임워크는 기본적으로 파라미터화된 쿼리를 사용하여 인젝션을 방어한다.
- **주의점**: ORM을 사용하더라도 `Native Query` 기능을 쓰면서 직접 문자열을 결합하거나, `order by` 절에 동적으로 입력값을 넣을 경우 여전히 인젝션 위험이 존재한다.

- **📢 섹션 요약 비유**: 최신 자동화 주방 기계(ORM)를 쓰면 안전하지만, 기계 설정을 무시하고 직접 칼(Native Query)을 휘두르면 여전히 손을 다칠 수 있다는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 레거시 PHP 게시판의 보안 고도화

- **문제 상황**: 10년 전 작성된 게시판 소스에서 `mysql_query("... WHERE title = '$input'")` 식의 취약 코드가 수천 군데 발견됨.
- **의사결정 (인젝션 방어 전략)**:
  1. **단기 조치 (WAF)**: 웹 방화벽(WAF)에 인젝션 패턴 차단 룰을 적용하여 외부 공격을 일시적으로 막는다.
  2. **중기 조치 (PDO 도입)**: DB 연결 방식을 PDO(PHP Data Objects)로 교체하고, 모든 동적 쿼리를 Prepared Statement로 리팩토링한다.
  3. **보완 조치 (입력값 검증)**: 숫자가 들어와야 할 곳에 문자가 들어오지 못하도록 타입 검증 로직을 추가한다.
  4. **최소 권한 DB 계정**: 웹 앱이 사용하는 DB 계정에서 `DROP TABLE`, `GRANT` 등 관리자 권한을 삭제하여 공격 성공 시의 피해를 최소화한다.

### 도입 체크리스트
- **Query Parameterization**: 모든 SQL 쿼리에서 문자열 더하기(+)를 제거하고 파라미터 바인딩을 사용하고 있는가?
- **Escaping/Sanitization**: OS 명령 실행이나 LDAP 조회 시 특수문자를 적절히 이스케이프 처리하고 있는가?
- **Safe APIs**: `system()`, `exec()` 같이 쉘을 직접 호출하는 함수 대신 보안이 강화된 라이브러리 함수를 사용하는가?

### 안티패턴
- **블랙리스트 필터링**: `' OR '`나 `SELECT` 단어를 막는 방식. 공격자는 `SeLeCt`나 `/**/` 같은 우회 기법으로 쉽게 뚫을 수 있다. (화이트리스트 방식 권장)
- **클라이언트 측 필터링**: 자바스크립트로 특수문자를 막는 행위. 해커는 프록시 도구(Burp Suite)로 브라우저를 거치지 않고 직접 서버에 공격 페이로드를 보낸다.

- **📢 섹션 요약 비유**: 도둑이 못 들어오게 "검은 옷 입은 사람 금지"라고 써 붙이는 것(블랙리스트)은 소용없으며, 오직 "초대장 있는 사람만 입장(화이트리스트)"시키는 것이 안전하다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 치명적 결함 100% 차단 | Prepared Statement 적용 시 기술적 인젝션 위협은 이론상 완벽 차단 |
| **정량적 (Performance)** | 쿼리 실행 속도 향상 | DB 엔진에서 쿼리 실행 계획을 재사용하므로 반복 요청 시 성능 이득 |
| **정성적 (Safety)** | 시스템 무결성 보장 | 데이터 무단 변조 및 삭제 시도를 차단하여 서비스 신뢰성 유지 |
| **정성적 (Standard)** | 시큐어 코딩 표준 준수 | KISA 및 OWASP 가이드라인을 충족하여 보안 감사 통과 |

### 미래 전망
- **AI 기반 코드 리뷰**: AI가 정적 분석(SAST) 단계에서 인젝션 가능성이 있는 쿼리 패턴을 탐지하고, 자동으로 Prepared Statement 코드로 변환해줄 것이다.
- **RPA 및 자동화 공격의 심화**: 해커들이 생성형 AI를 활용하여 특정 WAF(웹 방화벽) 룰을 우회하는 정교한 인젝션 페이로드를 실시간으로 생성하여 공격할 것에 대비해야 한다.

### 참고 표준
- **CWE-89**: SQL Injection 약점 정의
- **CWE-78**: OS Command Injection 약점 정의
- **행정안전부 SW 개발 보안 가이드**: 입력 데이터 검증 및 표현 항목

- **📢 섹션 요약 비유**: 예전에는 창과 방패의 싸움이었다면, 미래에는 인공지능 창과 인공지능 방패가 초당 수만 번씩 부딪히는 지능형 보안 전쟁이 될 것입니다.
