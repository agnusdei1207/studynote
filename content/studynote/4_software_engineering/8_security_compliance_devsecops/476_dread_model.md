# 476. DREAD 모델

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DREAD 모델은 식별된 보안 위협의 위험도를 **5가지 정량적 지표**인 Damage (잠재적 피해), Reproducibility (재현 가능성), Exploitability (공격 용이성), Affected Users (영향을 받는 사용자), Discoverability (발견 가능성)를 기준으로 산정하는 리스크 평가 프레임워크다.
> 2. **가치**: 주관적일 수 있는 보안 위험을 1~10점 사이의 수치로 객관화하여, 제한된 자원 내에서 어떤 위협을 우선적으로 해결해야 하는지에 대한 의사결정 우선순위를 제공한다.
> 3. **융합**: STRIDE 모델을 통해 식별된 위협들을 필터링하고 분류하는 후속 단계로 활용되며, 현대의 위협 모델링에서는 점수 산정의 복잡성으로 인해 하이-미디움-로우(High-Medium-Low) 방식과 병행되거나 자동화된 위험 산정 도구의 알고리즘으로 내재화되고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: DREAD는 위협 모델링의 결과물로 도출된 수많은 위협 중, 실제로 시스템에 가장 치명적이고 시급한 조치가 필요한 항목이 무엇인지 판별하기 위한 평가 모델이다. 각 지표의 점수를 합산하거나 평균을 내어 리스크의 크기를 결정한다.

- **💡 비유**: DREAD는 **"응급실의 트리아지(Triage, 환자 분류)"**와 같습니다. 환자의 출혈 정도(Damage), 통증 수치(Exploitability), 나이 및 기저질환(Affected Users) 등을 종합적으로 따져서 가장 먼저 수술실로 보낼 사람을 결정하는 것과 같은 이치입니다.

- **등장 배경**:
  1. **리스크 우선순위의 필요성**: 모든 보안 위협을 동시에 해결하는 것은 비용과 시간 면에서 불가능하므로, 선택과 집중을 위한 객관적 기준이 필요했다.
  2. **의사소통의 도구**: 개발자와 경영진에게 "이 위협은 10점 만점에 9점이니 당장 고쳐야 합니다"라고 정량적으로 설명하기 위한 수단으로 고안되었다.
  3. **Microsoft SDL의 초기 위험 평가 모델**: STRIDE와 함께 보안 설계의 완성도를 높이는 핵심 기법으로 널리 사용되었다.

- **📢 섹션 요약 비유**: 수많은 숙제 중에서 마감 기한, 난이도, 성적 비중을 따져서 오늘 밤에 꼭 해야 할 숙제부터 골라내는 스마트한 학습 계획표와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### DREAD 5대 평가 지표 (Risk Factors)

DREAD는 다음 5가지 질문을 통해 위험도를 측정한다. (각 1~10점 부여)

| 지표 (Factor) | 질문 내용 | 점수 기준 (높은 점수 예시) | 비유 |
|:---|:---|:---|:---|
| **D: Damage (잠재적 피해)** | 공격 성공 시 피해가 얼마나 큰가? | 전체 데이터 유출, 시스템 파괴 | 폭탄의 파괴력 |
| **R: Reproducibility (재현성)** | 공격을 다시 시도했을 때 성공률은? | 언제든 100% 재현 가능함 | 다시 시도해도 백발백중 |
| **E: Exploitability (공격 용이성)** | 공격을 수행하기 얼마나 쉬운가? | 초보자도 툴 하나로 가능함 | 열린 대문 넘기 |
| **A: Affected Users (영향 범위)** | 얼마나 많은 사용자가 피해를 입나? | 전체 가입자(100%) | 전염병의 확산 속도 |
| **D: Discoverability (발견 가능성)** | 취약점을 찾아내기 얼마나 쉬운가? | 브라우저 주소창만 봐도 알 수 있음 | 길가에 떨어진 지갑 |

---

### DREAD 위험도 산정 공식 및 프로세스

최종 위험도 점수는 각 지표의 합을 지표의 개수(5)로 나눈 평균값으로 정의하는 것이 일반적이다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      DREAD 리스크 평가 매트릭스 및 프로세스                     │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │  Risk Score = (Damage + Repro + Exploit + Affected + Discover) / 5           │
 │                                                                             │
 │   [위협 식별] ────▶ [지표별 점수 부여] ────▶ [우선순위 결정] ────▶ [대응 수립]  │
 │   (STRIDE 등)       (1 ~ 10 Scale)         (High / Mid / Low)               │
 │                                                                             │
 │  ┌───────────┬───┬───┬───┬───┬───┬─────────┬────────┐                     │
 │  │ 위협 명칭 │ D │ R │ E │ A │ D │ Total   │ Rating │                     │
 │  ├───────────┼───┼───┼───┼───┼───┼─────────┼────────┤                     │
 │  │ SQL Inj.  │ 10│ 10│ 8 │ 10│ 7 │ 45 / 5  │  9.0   │ ◀─── Critical      │
 │  ├───────────┼───┼───┼───┼───┼───┼─────────┼────────┤                     │
 │  │ XSS       │ 5 │ 8 │ 9 │ 4 │ 9 │ 35 / 5  │  7.0   │ ◀─── High          │
 │  ├───────────┼───┼───┼───┼───┼───┼─────────┼────────┤                     │
 │  │ Path Trav.│ 3 │ 4 │ 2 │ 1 │ 5 │ 15 / 5  │  3.0   │ ◀─── Low           │
 │  └───────────┴───┴───┴───┴───┴───┴─────────┴────────┘                     │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 위 표는 세 가지 서로 다른 위협에 대해 DREAD 지표를 적용한 예시다. **SQL 인젝션**은 데이터베이스 전체를 파괴할 수 있어 Damage(10)가 크고 영향 범위(10)도 넓어 최종 9.0점의 'Critical' 등급을 받는다. 반면 **경로 추적(Path Traversal)**은 발견하기도 어렵고(Discoverability 5) 피해도 제한적이라면 3.0점의 'Low' 등급으로 분류되어 나중에 조치해도 되는 항목이 된다. 이 정량적 숫자는 개발자와 보안 담당자 사이의 주관적 논쟁을 줄여주는 강력한 근거가 된다.

---

### 심층 동작 원리: 지표 간의 상관관계

1. **공격자 관점 (E, D)**: Exploitability와 Discoverability는 공격자가 들이는 노력과 기술 수준을 나타낸다. 이 점수가 높을수록 공격 빈도가 높아질 확률이 크다.
2. **피해자 관점 (D, A)**: Damage와 Affected Users는 사고 발생 시 기업이 입게 될 실질적인 손실(금전, 평판)을 나타낸다.
3. **신뢰성 관점 (R)**: Reproducibility는 해당 취약점이 일시적인 우연인지, 아니면 구조적인 결함인지를 판별하게 해준다.

- **📢 섹션 요약 비유**: 태풍의 위험도를 매길 때 풍속(Damage), 경로상의 인구수(Affected Users), 발생 빈도(Reproducibility)를 종합하여 경보 단계를 결정하는 기상 예보 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### DREAD vs CVSS (Common Vulnerability Scoring System) 비교

현대 보안 실무에서는 DREAD보다 CVSS가 더 널리 표준으로 사용되지만, 설계 단계에서는 DREAD가 여전히 유효하다.

| 비교 항목 | DREAD 모델 | CVSS (공통 취약점 평가 시스템) |
|:---|:---|:---|
| **사용 시점** | **설계/개발 단계** (위협 모델링 중) | **배포/운영 단계** (이미 발견된 취약점) |
| **평가 주체** | 내부 개발팀 및 아키텍트 | 보안 분석가 및 전문 기관 (NVD 등) |
| **특징** | 비즈니스 맥락과 사용자 영향을 중시함 | 기술적 복잡성과 환경 요인을 상세 분리함 |
| **주요 활용** | 내부 개발 우선순위 결정 | 패치 적용 시급성 판단 및 공표 |

---

### 리스크 평가의 간소화 트렌드

DREAD의 5가지 지표에 일일이 점수를 매기는 것이 번거롭다는 현장의 피드백으로 인해, 최근에는 다음과 같은 변형 모델이 사용된다.
- **Damage x Probability**: 피해액과 발생 확률만 곱하는 단순 모델.
- **High / Medium / Low**: 점수 대신 3단계 등급으로만 분류.
- **OWASP Risk Rating**: 위협원(Threat Agent)과 취약점(Vulnerability) 요인을 더 상세히 나누어 평가.

- **📢 섹션 요약 비유**: 정밀한 체성분 분석기(DREAD)를 사용하다가, 시간이 없을 때는 몸무게와 거울에 비친 모습만으로 건강 상태를 체크(간소화 모델)하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 커머스 플랫폼의 신규 결제 모듈 도입

- **문제 상황**: 위협 모델링 결과 '결제 금액 변조'와 '관리자 페이지 주소 노출' 두 가지 위협이 식별됨. 예산 문제로 이번 스프린트에는 하나만 고칠 수 있음.
- **의사결정 (DREAD 점수 산정)**:
  1. **위협 A (금액 변조)**: Damage(10 - 직접 금전 손실), Reproducibility(10 - 툴로 상시 가능), Affected Users(5 - 결제 시도자만). → **DREAD Avg: 8.5**
  2. **위협 B (주소 노출)**: Damage(3 - 주소만 암), Reproducibility(10), Affected Users(1 - 관리자만). → **DREAD Avg: 4.8**
  3. **결론**: 위협 A를 즉시 조치하고, 위협 B는 다음 스프린트 백로그로 넘김.

### 도입 체크리스트
- **점수 기준 합의**: 조직 내에서 무엇이 1점이고 무엇이 10점인지에 대한 구체적인 'Scoring Guide'가 마련되어 있는가?
- **주관성 배제**: 여러 명의 분석가가 매긴 점수의 평균을 사용하여 개인의 편향을 줄이고 있는가?
- **맥락 반영**: 동일한 취약점이라도 '공개 서버'와 '내부 폐쇄망'에서의 점수가 다르게 산정되는가?

### 안티패턴
- **Discoverability의 함정**: "아무도 못 찾을 거야"라고 생각해서 Discoverability 점수를 낮게 주어 전체 위험도를 낮추는 행위 (Security by Obscurity).
- **무조건적 10점**: 모든 위협이 무서워 보여서 전부 10점을 주면, 결국 우선순위 결정이라는 DREAD 본연의 목적을 상실하게 됨.

- **📢 섹션 요약 비유**: 모든 환자가 아프다고 소리 지른다고 해서 모두에게 1등으로 수술을 해줄 수는 없으므로, 차분하게 검사 수치(Scoring Guide)를 봐야 한다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Prioritizing)** | 자원 배분 최적화 | 위험도 상위 20% 위협에 보안 예산 80%를 집중 투여 |
| **정량적 (Speed)** | 의사결정 속도 향상 | 수치화된 근거를 통해 부서 간 이견 조율 시간 단축 |
| **정성적 (Communication)** | 투명한 리스크 관리 | 경영진에게 보안 현황을 숫자로 보고하여 가시성 확보 |
| **정성적 (Standard)** | 일관된 보안 수준 유지 | 평가자 주관에 상관없이 일정한 기준의 보안 필터링 가능 |

### 미래 전망
- **Context-Aware Scoring**: AI가 시스템의 비즈니스 가치와 데이터의 민감도를 실시간으로 파악하여 DREAD 점수를 자동 보정해주는 기술이 등장할 것이다.
- **Dynamic Risk Rating**: 위협이 발견된 시점의 최신 해킹 트렌드(Exploit 키트 유통 여부 등)를 반영하여 점수가 실시간으로 변하는 유동적 리스크 모델이 도입될 것이다.

### 참고 표준
- **ISO/IEC 27005**: 정보보안 리스크 관리 표준
- **NIST SP 800-30**: 연방 정보 시스템 위험 평가 가이드라인

- **📢 섹션 요약 비유**: 종이 지도 위에 펜으로 위험 지역을 그리던 방식에서, 실시간 기상 위성과 센서 데이터가 결합된 '디지털 리스크 트윈'으로 발전하여 재난을 미리 막게 될 것입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 우리 동네에 나타난 **나쁜 악당들이 얼마나 무서운지 점수를 매겨서 순서를 정하는 규칙**이에요.
2. "힘이 센가?(피해)", "자주 나타나나?(재현)", "잡기 쉬운가?(발견)" 같은 5가지 질문에 점수를 줘서 가장 높은 점수가 나온 악당부터 먼저 잡으러 가는 거죠.
3. 이렇게 하면 소중한 우리 영웅들(개발자)의 힘을 낭비하지 않고, 가장 위험한 문제부터 똑똑하게 해결할 수 있답니다!
