# 478. Broken Access Control (취약한 접근 제어)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Broken Access Control (취약한 접근 제어)은 사용자가 자신의 권한 범위를 벗어나 다른 사용자의 데이터에 접근하거나, 관리자 기능을 실행하는 등 **접근 통제 정책이 적절히 강제되지 않는 상태**를 의미한다.
> 2. **가치**: 2021년 OWASP Top 10에서 부동의 1위를 차지할 만큼 발생 빈도와 위험도가 높으며, 이를 방어하는 것은 단순히 기술적 결함을 고치는 것을 넘어 비즈니스 로직 전반에 '최소 권한 원칙 (Least Privilege)'을 설계적으로 녹여내는 작업이다.
> 3. **융합**: IDOR (Insecure Direct Object Reference)와 특권 상승 (Privilege Escalation) 위협을 포함하며, 현대의 제로 트러스트 (Zero Trust) 아키텍처와 RBAC (Role-Based Access Control) / ABAC (Attribute-Based Access Control) 모델을 통해 서버 측에서 중앙 집중식으로 관리되어야 한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 접근 제어는 누가(Subject) 어떤 자원(Object)에 대해 어떤 행위(Action)를 할 수 있는지 결정하는 보안 체계다. Broken Access Control은 이 규칙이 무너져서, 공격자가 URL 수정, 세션 조작, API 직접 호출 등을 통해 타인의 정보를 훔치거나 시스템 설정을 변경하는 취약점이다.

- **💡 비유**: Broken Access Control은 **"아파트 현관문은 열쇠로 잠겨 있지만, 베란다 창문이나 뒷문이 열려 있어 누구나 들어갈 수 있는 상태"** 또는 **"일반 사원이 사장님의 결재 도장을 몰래 가져와서 자기 마음대로 서류에 도장을 찍는 것"**과 같습니다.

- **등장 배경**:
  1. **복잡한 비즈니스 로직**: 웹 서비스가 고도화되면서 사용자 유형(일반, 우수, 관리자, 파트너 등)과 자원의 소유권 관계가 복잡해져 권한 체크 누락이 빈번해졌다.
  2. **API 중심 아키텍처**: 클라이언트 사이드 렌더링(React, Vue 등)이 보편화되면서, 서버 측에서 데이터를 내려줄 때 권한 검증을 소홀히 하고 클라이언트 단의 UI 제어에만 의존하는 경향이 생겼다.
  3. **데이터 중심 공격의 증가**: 개인정보 유출이 해커들의 주된 수익원이 되면서, 대량의 데이터를 한꺼번에 탈취할 수 있는 접근 제어 취약점이 주 타겟이 되었다.

- **📢 섹션 요약 비유**: 입구에서 아무리 신분증 검사를 철저히 해도(인증), 건물 안의 개별 사무실 문이 다 열려 있다면(인가 실패) 보안은 무의미하다는 원리와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Broken Access Control의 주요 유형 및 메커니즘

접근 제어 실패는 크게 수평적/수직적 권한 상승과 비즈니스 로직 우회로 나뉜다.

| 유형 | 설명 | 공격 사례 | 비유 |
|:---|:---|:---|:---|
| **수평적 권한 상승 (IDOR)** | 동일한 권한 수준의 다른 사용자 데이터 접근 | URL의 `user_id=100`을 `101`로 바꿔 타인의 프로필 보기 | 옆집 열쇠로 우리 집 문 열기 |
| **수직적 권한 상승** | 하위 사용자가 상위 사용자의 기능 수행 | 일반 사용자가 `/admin/delete_user` 경로로 직접 접근 | 사원이 인사부장의 컴퓨터 사용 |
| **비즈니스 로직 우회** | 시스템 프로세스의 허점을 이용한 접근 | 결제 단계 점프, 무료 사용자용 유료 기능 접근 | 기차표 없이 플랫폼에 들어가기 |
| **CORS 및 API 오설정** | 잘못된 도메인 허용으로 인한 데이터 유출 | `Access-Control-Allow-Origin: *` 설정으로 타 사이트에서 데이터 탈취 | 아무나 복사할 수 있는 복사기 |

---

### 접근 제어 실패 시나리오 아키텍처 (IDOR 예시)

IDOR (Insecure Direct Object Reference)는 접근 제어 실패의 가장 대표적인 사례다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      IDOR (부적절한 직접 객체 참조) 공격 메커니즘               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │    [공격자 (Attacker)]          [취약한 서버 (Vulnerable Server)]    [DB]      │
 │           │                                │                          │     │
 │  1. 로그인 (User A)                         │                          │     │
 │           │ ─────────────────────────────▶ │                          │     │
 │           │                                │                          │     │
 │  2. 타인(User B)의 정보 요청                   │                          │     │
 │     GET /api/order/500                     │                          │     │
 │     (본인 주문은 499번임)                    │                          │     │
 │           │ ─────────────────────────────▶ │                          │     │
 │           │                                │ 3. 권한 체크 누락          │     │
 │           │                                │    (세션 A와 주문 500 매핑 확인 안함) │
 │           │                                │                          │     │
 │           │                                │ 4. DB 쿼리                │     │
 │           │                                │    SELECT * FROM orders  │     │
 │           │                                │    WHERE order_id = 500  │ ──▶ │
 │           │                                │                          │     │
 │  5. User B의 정보 탈취 완료!                   │                          │     │
 │           │ ◀───────────────────────────── │                          │     │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 위 시나리오에서 공격자는 정상적으로 본인의 계정(User A)으로 로그인(인증)했다. 그러나 서버로 데이터를 요청할 때, 자신의 주문 번호가 아닌 타인의 주문 번호(500)를 직접 참조(Direct Reference)하여 보낸다. 서버가 "이 요청을 보낸 사람이 주문 500의 주인인가?"를 확인하는 **인가(Authorization)** 과정을 누락하면, 공격자는 타인의 민감 정보를 그대로 열람하게 된다. 이를 방어하기 위해서는 `WHERE order_id = 500 AND owner_id = current_user_id` 처럼 소유권 검증 쿼리를 반드시 수행해야 한다.

---

### 심층 동작 원리: 중앙 집중식 접근 제어 메커니즘

접근 제어는 코드 곳곳에 산재해 있으면 누락될 위험이 크므로, 중앙 집중화된 아키텍처가 필수적이다.
1. **인터셉터/필터 (Interceptor/Filter)**: 모든 요청의 입구에서 사용자의 역할(Role)과 요청 경로를 대조하여 기본적인 접근 권한을 필터링한다.
2. **정책 기반 제어 (Policy-based)**: RBAC (Role-Based) 또는 ABAC (Attribute-Based) 모델을 사용하여 정책을 데이터로부터 분리 관리한다.
3. **데이터 소유권 검증 (Data-level)**: DB 조회 시 현재 세션 사용자의 ID를 조건절에 강제로 포함시켜 다른 객체 참조를 원천 차단한다.

- **📢 섹션 요약 비유**: 건물의 모든 방마다 보안 요원을 세우는 대신, 중앙 통제실에서 마스터 키 카드의 권한에 따라 각 방의 문이 자동으로 열리고 닫히게 관리하는 스마트 빌딩 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### RBAC (역할 기반) vs ABAC (속성 기반) 접근 제어 모델

| 비교 항목 | RBAC (Role-Based) | ABAC (Attribute-Based) |
|:---|:---|:---|
| **제어 기준** | 사용자의 정적인 역할 (Admin, User) | 사용자, 자원, 환경의 동적인 속성 |
| **복잡도** | 낮음 (관리하기 쉬움) | 높음 (정밀한 정책 수립 필요) |
| **유연성** | 낮음 (역할이 늘어날수록 관리 폭발) | 매우 높음 (시간, IP, 기기 등에 따른 제어) |
| **적합한 환경** | 일반적인 웹 서비스, 소규모 조직 | 국방, 금융, 대규모 클라우드 환경 |

---

### 융합 보안: 제로 트러스트 (Zero Trust)와 접근 제어

현대 보안은 "한 번 인증됐다고 해서 무조건 신뢰하지 않는다"는 제로 트러스트 원칙을 적용한다.
- **연속적 검증**: 세션이 유지되는 동안에도 매 요청마다 권한을 다시 확인한다.
- **최소 권한 원칙**: 사용자가 업무를 수행하는 데 꼭 필요한 최소한의 권한만 일시적으로 부여한다.

- **📢 섹션 요약 비유**: 예전에는 성문만 통과하면 성 안을 마음대로 다닐 수 있었지만, 이제는 성 안의 모든 건물과 방에 들어갈 때마다 매번 신분증을 다시 보여줘야 하는 철저한 보안 검색과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 의료 정보 시스템의 환자 기록 조회 보안

- **문제 상황**: 의사가 환자 기록을 조회할 때, URL의 환자 번호만 바꾸면 자신이 담당하지 않는 환자의 기록까지 볼 수 있는 결함이 발견됨.
- **의사결정 (접근 제어 강화)**:
  1. **간접 참조 도입**: 실제 DB ID 대신 사용자의 세션 내에서만 유효한 임시 랜덤 토큰(Map 기반)을 사용하여 직접적인 ID 추측을 차단한다.
  2. **쿼리 레벨 제어**: 조회 쿼리에 의사 ID와 환자 담당 관계 테이블을 Join 하여 소유권이 없는 데이터는 검색 결과에 나오지 않게 한다.
  3. **ABAC 적용**: 긴급 상황 시에만 일시적으로 권한을 확장하는 'Break-glass' 정책을 수립하여 예외 상황도 안전하게 통제한다.

### 도입 체크리스트
- **Default Deny**: 권한이 명시적으로 허용되지 않은 모든 요청은 기본적으로 거부(Deny)되도록 설정되어 있는가?
- **Server-side Validation**: 클라이언트에서 버튼을 숨기는 것으로 만족하지 않고, 서버 API 내부에서 권한을 재검증하고 있는가?
- **Metadata Protection**: 디렉터리 리스팅이나 `.git` 등 민감한 설정 파일에 대한 외부 접근이 차단되어 있는가?

### 안티패턴
- **보안 수단으로서의 식별자**: GUID나 UUID가 추측하기 어렵다는 이유만으로 권한 체크를 생략하는 행위. (노출 시 무방비 상태가 됨)
- **클라이언트 사이드 신뢰**: "관리자 메뉴가 안 보이니까 일반 사용자는 못 들어오겠지"라고 생각하며 서버 URL 접근 제어를 소홀히 하는 행위.

- **📢 섹션 요약 비유**: 보이지 않는 비밀번호를 썼다고 안심하고 금고 문을 열어두는 것이 아니라, 비밀번호를 알아도 지문 인식이 있어야만 열리도록 이중 장치를 해야 한다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 데이터 유출 사고 예방 | 타인 정보 접근 시도로 인한 데이터 노출 위험 99% 차단 |
| **정량적 (Cost)** | 유지보수 효율성 | 중앙 집중식 권한 관리로 정책 변경 시 소스 수정 최소화 |
| **정성적 (Trust)** | 사용자 프라이버시 보호 | "내 정보가 남에게 보이지 않는다"는 고객의 강력한 신뢰 확보 |
| **정성적 (Compliance)** | 법적 준거성 확보 | 개인정보보호법 상의 접근 통제 의무 사항 완벽 준수 |

### 미래 전망
- **AI 기반 이상 접근 탐지**: 평소와 다른 패턴의 권한 요청(짧은 시간에 수천 개의 ID를 순차 조회 등)을 AI가 실시간으로 탐지하여 접근을 차단하는 기술이 보편화될 것이다.
- **Decentralized Identity (DID)**: 사용자가 자신의 권한을 직접 제어하고, 서비스는 필요한 최소한의 속성만 검증하는 자기 주권형 접근 제어 모델이 확산될 것이다.

### 참고 표준
- **NIST SP 800-162**: 속성 기반 접근 제어(ABAC) 가이드라인
- **OAuth 2.0 / OpenID Connect**: 서비스 간 권한 위임 및 인증 표준 프로토콜

- **📢 섹션 요약 비유**: 과거에는 튼튼한 열쇠만 있으면 됐지만, 미래에는 인공지능 경비원이 사람의 표정과 행동까지 분석해서 문을 열어줄지 말지 결정하는 '지능형 접근 통제' 시대가 올 것입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 우리 학교 도서관에서 **내 이름표를 달고 들어갔는데, 갑자기 교장 선생님만 볼 수 있는 비밀 서류 가방을 열어볼 수 있게 된 것**과 같아요.
2. 아무리 학생이라도 학교의 모든 곳을 다 가볼 수 있으면 안 되는 것처럼, 컴퓨터 프로그램도 자기가 가진 열쇠로 열 수 있는 문만 열게 해야 해요.
3. 이렇게 문마다 주인을 꼼꼼히 확인하면, 소중한 친구들의 비밀 편지(데이터)가 남에게 들키는 일을 막을 수 있답니다!
