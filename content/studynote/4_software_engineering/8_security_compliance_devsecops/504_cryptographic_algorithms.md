# 504. 암호화 알고리즘 (대칭키, 비대칭키, 일방향) 적용 기준

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 암호화 알고리즘은 수학적 원리를 이용하여 정보의 기밀성(Encryption), 무결성(Hashing), 인증(Signature)을 보장하는 기술로, 목적에 따라 **대칭키(속도), 비대칭키(키 교환), 일방향 해시(복구 불가)**로 구분하여 적용한다.
> 2. **가치**: 현대 컴퓨팅 환경에서 보안 사고 예방의 최후 보루이며, NIST 및 KISA 가이드에 따른 최신 표준(AES-256, RSA-2048, SHA-256 이상)과 충분한 키 길이를 준수함으로써 브루트 포스 공격으로부터 데이터를 보호한다.
> 3. **융합**: 하이브리드 암호 시스템(SSL/TLS)의 핵심 구성 요소이며, 최근에는 양자 컴퓨터의 해독 위협에 대응하는 양자 내성 암호(PQC)와 데이터 가용성을 높이는 동형 암호(Homomorphic Encryption) 기술로 진화하고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 암호화 알고리즘은 평문을 암호문으로 바꾸는 일련의 절차다. **대칭키**는 동일한 키로 암/복호화를 수행하고, **비대칭키**는 서로 다른 쌍(공개키/개인키)을 사용하며, **일방향 해시**는 복호화가 불가능한 고유 값을 생성한다.

- **💡 비유**: 
  - **대칭키**: 금고 주인이 열쇠 하나로 문을 열고 닫는 것 (빠름, 열쇠 관리 중요).
  - **비대칭키**: 우체통에 편지를 넣을 때는 누구나 가능하지만(공개키), 편지를 꺼낼 때는 집주인만 열 수 있는 것(개인키).
  - **일방향 해시**: 생과일을 갈아서 주스로 만드는 것. 주스를 보고 원래 과일이 무엇인지 알 수 있지만(무결성), 주스를 다시 과일로 되돌릴 수는 없음(비가역성).

- **등장 배경**:
  1. **데이터 기밀성 요구**: 군사/금융 목적을 넘어 개인정보 보호가 기업의 생존권과 직결되면서 강력한 암호 표준이 필요해졌다.
  2. **키 교환의 난제**: 대칭키 방식의 가장 큰 문제인 '어떻게 키를 안전하게 전달할 것인가'를 해결하기 위해 비대칭키(PKI)가 탄생했다.
  3. **연산 성능의 향상**: 하드웨어 성능이 좋아지면서 복잡한 수학적 연산을 기반으로 한 암호화 기술이 실용화되었다.

- **📢 섹션 요약 비유**: 상황에 맞는 가장 튼튼한 자물쇠와 봉인(Hash)을 선택하여, 정보라는 보물을 지키는 '디지털 수호 규칙'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 암호화 방식별 특징 및 적용 기준

| 분류 | 대표 알고리즘 | 암/복호화 키 | 특징 | 주요 용도 |
|:---|:---|:---:|:---|:---|
| **대칭키 (Symmetric)** | **AES**, ARIA, SEED | 동일함 | 연산 속도 빠름, 대량 데이터 적합 | DB 암호화, 파일 암호화 |
| **비대칭키 (Asymmetric)** | **RSA**, **ECC**, ElGamal | 다름 (Public/Private) | 연산 속도 느림, 키 배송 문제 해결 | 디지털 서명, 키 교환 |
| **일방향 해시 (One-way)** | **SHA-2/3**, BLAKE3 | 없음 (비가역) | 동일 입력 시 동일 결과, 복구 불가 | 비밀번호 저장, 무결성 체크 |

---

### 하이브리드 암호 시스템 아키텍처 (SSL/TLS 예시)

대칭키의 속도와 비대칭키의 안전한 키 교환 장점을 결합한 현대 보안의 표준 구조다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      하이브리드 암호화 및 무결성 검증 흐름도                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [송신자 / 클라이언트]                                  [수신자 / 서버]         │
 │        │                                                  │                 │
 │  1. 비대칭키로 세션키 교환 ────────(Shared Key)─────────▶ │                 │
 │     (RSA / Diffie-Hellman)                                │                 │
 │                                                           │                 │
 │  2. 대칭키로 본문 암호화 ──────────(Ciphertext)─────────▶ │ 3. 본문 복호화    │
 │     (AES-256)                                             │    (AES-256)    │
 │                                                           │                 │
 │  4. 해시로 무결성 보장 ───────────(HMAC / Hash)─────────▶ │ 5. 변조 여부 확인 │
 │     (SHA-256)                                             │    (SHA-256)    │
 │                                                           │                 │
 │  ┌───────────────────────────────────────────────────────┐ │                 │
 │  │ ⚠ 적용 기준 (Standard)                                 │ │                 │
 │  │ - 대칭키: AES-128 이상 (권장 256)                       │ │                 │
 │  │ - 비대칭키: RSA-2048 이상, ECC-224 이상                 │ │                 │
 │  │ - 일방향: SHA-256 이상 (SHA-1 사용 금지)                │ │                 │
 │  └───────────────────────────────────────────────────────┘                  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 실전에서는 한 가지 알고리즘만 쓰지 않는다. **비대칭키(RSA 등)**는 속도가 너무 느려 대량의 데이터를 암호화하기엔 부적절하므로, 통신 시작 시 아주 짧은 **'세션키(대칭키)'**를 안전하게 주고받는 용도로만 쓴다. 일단 세션키가 안전하게 공유되면, 그 뒤로는 속도가 빠른 **대칭키(AES 등)**로 실제 데이터를 암호화하여 통신한다. 여기에 데이터가 전송 도중 바뀌지 않았음을 증명하기 위해 **해시(SHA-256)** 기반의 HMAC을 결합한다. 이것이 우리가 매일 쓰는 HTTPS 통신의 근본 아키텍처다.

---

### 심층 동작 원리: 블록 암호 운영 모드 (Block Cipher Mode)

AES와 같은 대칭키 암호는 데이터를 일정 블록(128비트 등) 단위로 나누어 처리한다.
1. **ECB (Electronic Codebook)**: 각 블록을 독립적으로 암호화. 동일한 평문 블록이 동일한 암호문으로 나와 패턴이 노출됨. **(사용 금지)**
2. **CBC (Cipher Block Chaining)**: 이전 블록의 암호화 결과와 다음 평문을 XOR 연산 후 암호화. 보안성이 높으나 병렬 처리가 어려움.
3. **GCM (Galois/Counter Mode)**: CTR 모드에 무결성 검증 기능을 추가. 고속 병렬 처리가 가능하여 현대 네트워크(TLS 1.3)의 표준으로 쓰임.

- **📢 섹션 요약 비유**: 똑같은 자물쇠라도 하나씩 따로 잠그는 것(ECB)보다, 체인으로 엮어서 하나라도 건드리면 전체가 틀어지게 만드는 것(CBC/GCM)이 훨씬 안전하다는 점과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### RSA (소인수분해) vs ECC (타원곡선) 비교

현대 암호학은 효율성이 높은 ECC로 무게추가 옮겨가고 있다.

| 비교 항목 | RSA (Rivest-Shamir-Adleman) | ECC (Elliptic Curve Crypto) |
|:---|:---|:---|
| **수학적 근거** | 소인수분해의 난해함 | 타원곡선 이산대수 문제 |
| **보안 강도** | 동일 보안 수준을 위해 긴 키 필요 | **짧은 키로도 높은 보안성 제공** |
| **키 길이 예시** | **2048 비트** | **224 비트** (보안성 동일) |
| **성능 (속도)** | 복호화/서명 시 연산량 많음 | 연산 속도 빠름, 모바일 기기에 적합 |
| **가치** | 검증된 범용성, 구현 용이 | 차세대 표준, 효율적인 자원 사용 |

---

### 융합 시너지: 양자 내성 암호 (PQC) 준비

현재의 비대칭키 암호(RSA, ECC)는 양자 컴퓨터의 '쇼어 알고리즘(Shor's Algorithm)'에 의해 실시간으로 풀릴 수 있는 잠재적 위험이 있다.
- **적용 전략**: 기존 암호 체계와 새로운 격자 기반(Lattice-based) 암호를 섞어서 사용하는 하이브리드 아키텍처로의 전환이 요구된다.

- **📢 섹션 요약 비유**: 아무리 튼튼한 금고(RSA)라도 원자 폭탄(양자 컴퓨터)에는 견딜 수 없으므로, 원자 폭탄의 충격을 흡수할 수 있는 새로운 재질의 금고(PQC)를 개발하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 기업 내부 민감 데이터 저장 아키텍처 설계

- **문제 상황**: 개인정보보호법에 따라 고객의 주민등록번호와 비밀번호를 DB에 안전하게 저장해야 함.
- **의사결정 (알고리즘 선정)**:
  1. **주민등록번호 (기밀성)**: 복호화가 필요하므로 대칭키 알고리즘인 **AES-256**을 적용한다. 운영 모드는 성능과 보안을 고려해 **GCM** 모드를 선택한다.
  2. **비밀번호 (비가역성)**: 관리자도 알 필요가 없으므로 일방향 해시인 **SHA-256** (또는 그 이상)을 사용한다. 레인보우 테이블 공격 방지를 위해 사용자별 **솔트(Salt)**와 **키 스트레칭**을 의무 적용한다.
  3. **키 관리 (KMS)**: 암호화 키를 소스코드나 DB 서버에 두지 않고, 별도의 **KMS(Key Management Service)**에 보관하여 접근 권한을 분리한다.
  4. **전송 보안**: 관리자 페이지 접속 시 **TLS 1.2/1.3**을 강제하여 구간 암호화를 수행한다.

### 도입 체크리스트
- **Length**: RSA 키 길이는 최소 2048비트 이상인가?
- **Deprecated Algorithms**: MD5, SHA-1, DES 등 취약 알고리즘을 소스 코드에서 완전히 퇴출했는가?
- **Randomness**: 암호화에 사용되는 IV(Initial Vector)나 Salt가 안전한 난수 생성기(CSPRNG)를 통해 생성되는가?

### 안티패턴
- **자체 암호화 로직**: "나만 아는 알고리즘"을 만들어 쓰는 행위. 학술적으로 검증되지 않은 암호는 해커의 좋은 먹잇감이 된다.
- **IV 고정 사용**: AES-CBC 등에서 초기화 벡터(IV)를 고정값으로 쓰면 동일 평문에 대해 동일 암호문이 나와 추측이 가능해진다.

- **📢 섹션 요약 비유**: 검증된 유명 브랜드의 자물쇠를 쓰는 대신, 내가 직접 만든 나무 빗장을 걸어두고 "이건 세상에 하나뿐이라 못 열 거야"라고 착각하는 보안 불감증을 버려야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 데이터 복호화 시도 차단 | 현대 연산 능력으로 해독에 수천 년이 걸리는 강도의 암호 확보 |
| **정량적 (Compliance)** | 법적 과태료 예방 | 개인정보 보호법의 기술적 보호 조치 의무사항 완벽 준수 |
| **정성적 (Trust)** | 대외 서비스 신뢰도 향상 | "암호화가 철저한 안전한 서비스"라는 브랜드 이미지 구축 |
| **정성적 (Interoperability)** | 글로벌 표준 호환성 | 해외 기관 및 서비스와 연동 시 암호화 통신 호환성 확보 |

### 미래 전망
- **Homomorphic Encryption (동형 암호)**: 데이터를 암호화한 채로 연산(통계 분석 등)이 가능한 기술이 실용화되어, 개인정보 침해 없이 빅데이터 분석이 가능해질 것이다.
- **Post-Quantum Standard**: 2024~2025년을 기점으로 NIST에서 PQC 표준을 확정함에 따라, 전 세계적인 암호 체계 교체(Crypto Agility) 사업이 본격화될 것이다.

### 참고 표준
- **FIPS 140-2/3**: 미국 연방 정보 처리 표준 (암호 모듈 검증)
- **KISA 암호 알고리즘 및 키 길이 가이드**: 국내 보안 준수 표준
- **NIST SP 800-57**: Recommendation for Key Management

- **📢 섹션 요약 비유**: 수동 자물쇠에서 디지털 지문 인식으로, 이제는 뇌파 인식(미래 암호)으로 발전하듯 암호 기술은 인간의 상상력보다 빠르게 해커의 위협을 앞서 나갈 것입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **KMS (Key Management Service)**: 암호화 알고리즘보다 더 중요한 '키'를 관리하는 곳
- **TLS (Transport Layer Security)**: 알고리즘들이 결합되어 사용되는 실전 프로토콜
- **솔팅 및 스트레칭 (Salting & Stretching)**: 해시 알고리즘의 보안성을 높이는 기법
- **PKI (Public Key Infrastructure)**: 비대칭키 암호를 지탱하는 신뢰 인프라

## 👶 어린이를 위한 3줄 비유 설명
1. **대칭키**는 나랑 친구랑 똑같은 열쇠를 하나씩 나눠 갖고 비밀 상자를 여는 거예요.
2. **비대칭키**는 친구가 내 우체통(공개키)에 비밀 편지를 넣으면, 나만 가진 내 방 열쇠(개인키)로 열어보는 거랍니다.
3. **해시**는 과일을 주스로 만드는 마법이라서, 주스를 보고 어떤 과일인지 맞힐 수는 있지만 주스를 다시 과일로 만들 수는 없어요. 비밀번호를 숨길 때 최고죠!
