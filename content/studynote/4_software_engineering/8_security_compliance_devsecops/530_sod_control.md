# 530. 보안 조직 분리 (SoD) 및 SW 통제 로직

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보안 조직 분리 (Segregation of Duties, SoD)는 하나의 비즈니스 프로세스나 위험한 작업을 **두 명 이상의 독립된 주체가 나누어 수행**하도록 설계하여, 단독 행위에 의한 부정, 오류, 권한 남용을 방지하는 내부 통제 원칙이다.
> 2. **가치**: "결제 요청자와 승인자의 분리", "개발자와 운영자의 분리" 등 이해 상충 지점을 기술적으로 격리함으로써 내부 공모에 의한 횡령이나 고의적 시스템 파괴 리스크를 최소화하고 업무의 투명성을 확보한다.
> 3. **융합**: IAM(계정 및 권한 관리) 시스템의 RBAC/ABAC 모델과 결합되며, 현대 아키텍처에서는 CI/CD 파이프라인의 '승인 워크플로우' 및 '이중 검토(Two-person control)' 로직을 통해 소프트웨어적으로 강제된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: SoD는 "혼자서 다 하게 두지 마라"는 원칙이다. 시스템의 특정 기능을 수행할 때 필요한 권한을 쪼개어 서로 다른 사람에게 배분함으로써, 한 사람이 실수하거나 나쁜 마음을 먹어도 시스템 전체가 위험에 빠지지 않게 하는 방어 기제다.

- **💡 비유**: SoD는 **"핵미사일 발사 키"**와 같습니다. 미사일을 쏘려면 두 명의 장교가 각자의 열쇠를 동시에 꽂고 돌려야 하듯, 중요한 비즈니스 결정(예: 거액 송금)도 요청자와 승인자가 각자의 시스템 권한을 행사해야만 완료되도록 만드는 것입니다.

- **등장 배경**:
  1. **내부 부정 사고의 빈번화**: 정당한 권한을 가진 단일 사용자가 감시 없이 중요 데이터를 조작하여 발생하는 기업 금융 사고가 늘어났다.
  2. **컴플라이언스 준수**: 상장사 회계 감사(SOX), 금융 보안 표준(ISMS-P) 등에서 직무 분리를 시스템적으로 구현할 것을 법적 의무로 규정하고 있다.
  3. **운영 안정성 확보**: 개발자가 직접 운영 서버에 접속하여 소스를 수정할 경우 발생하는 휴먼 에러와 보안 홀을 차단하기 위한 '개발-운영 분리' 요구가 커졌다.

- **📢 섹션 요약 비유**: 축구 경기에서 심판과 선수가 동일 인물일 수 없듯이, 시스템에서도 일을 하는 사람과 감시하는 사람의 역할을 엄격히 나누는 공정성 확보 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 직무 분리(SoD) 시나리오 매트릭스

소프트웨어 설계 시 반드시 고려해야 할 충돌 지점들이다.

| 프로세스 | 직무 A (요청/수행) | 직무 B (승인/검토) | 통제 목표 |
|:---|:---|:---|:---|
| **구매 및 회계** | 구매 주문 작성자 | 대금 결제 승인자 | 허위 구매 및 횡령 방지 |
| **시스템 운영** | **소프트웨어 개발자** | **운영 환경 배포자** | 검증되지 않은 코드 배포 차단 |
| **보안 관리** | 일반 사용자 관리자 | 보안 감사 로그 열람자 | 관리자의 권한 오남용 감시 |
| **데이터 관리** | 데이터 입력/수정자 | 데이터 무결성 검증자 | 원본 데이터 임의 변조 방지 |

---

### 2. SoD 기반 워크플로우 통제 아키텍처

소프트웨어 로직이 어떻게 두 명의 승인을 강제하는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      이중 승인 (Two-Person Control) 기반 SoD 로직 구조            │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [주체 1 (Requester)] ────▶ [1. 작업 요청 등록] ────▶ [중앙 통제 엔진]             │
 │      │                       (State: PENDING)         │ (IAM / Workflow)    │
 │      │                                                │                     │
 │      └───────────────────(자기 승인 시도)────────────▶ [2. SoD Validation]    │
 │                                                       │  If (User == Owner) │
 │                                                       │  Then [REJECT]      │
 │                                                       ▼                     │
 │ [주체 2 (Approver)]  ◀─── [3. 승인 대기 알림] ──────── [검증 완료]             │
 │      │                                                                      │
 │      └──────────────────▶ [4. 승인 수행] ───────▶ [5. 작업 최종 실행]          │
 │                           (State: APPROVED)        (State: COMPLETED)       │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 소프트웨어 통제 규칙 (Constraints):                                     │  │
 │  │ 1. Static SoD: 상충되는 두 역할을 동시에 한 명에게 부여 금지                 │  │
 │  │ 2. Dynamic SoD: 역할은 가졌더라도 동일한 건에 대해 두 번 행동 금지            │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SoD 아키텍처의 핵심은 **상태 관리**와 **신원 대조**다. 1) **주체 1**이 작업을 요청하면 시스템은 해당 건을 '승인 대기' 상태로 둔다. 2) 시스템은 **SoD Validation** 로직을 통해 요청자와 승인자가 동일 인물인지, 혹은 특수 관계인지(정적 SoD 위반 여부)를 실시간 체크한다. 3) **주체 2**가 승인 버튼을 누를 때 비로소 최종 명령이 실행된다. 이는 단순히 '두 번 클릭'하는 게 아니라, '서로 다른 두 개의 신뢰(Trust)'가 합쳐져야 문이 열리는 논리적 구조다.

---

### 심층 동작 원리: 정적 SoD vs 동적 SoD

1. **정적 SoD (Static SoD)**: 사용자가 권한을 할당받는 시점에 통제. (예: 인사 DB에서 '회계' 역할과 '감사' 역할을 동시에 부여할 수 없게 막음)
2. **동적 SoD (Dynamic SoD)**: 사용자가 실제 업무를 수행하는 런타임에 통제. (예: 내가 올린 결재 기안은 내가 승인 버튼을 누를 수 없게 코드로 막음)
3. **가치**: 정적 SoD는 권한의 과잉 부여를 막고, 동적 SoD는 권한이 있더라도 특정 상황에서의 오남용을 막는다.

- **📢 섹션 요약 비유**: 처음부터 경찰과 범인 역할을 동시에 못 맡게 하는 것(정적)과, 경찰이라도 자기 가족의 사건은 수사하지 못하게 제척하는 것(동적)의 차이입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 직무 분리(SoD) vs 최소 권한 원칙 (Least Privilege)

둘은 찰떡궁합이지만 지향점이 살짝 다르다.

| 비교 항목 | 최소 권한 원칙 (Least Privilege) | 직무 분리 원칙 (SoD) |
|:---|:---|:---|
| **초점** | **수평적** 제한 (필요한 것만 줌) | **수직적/프로세스적** 제한 (나눠서 줌) |
| **방어 대상** | 외부 침입자의 권한 확대 방지 | **내부자의 공모 및 부정 방지** |
| **구현 방식** | 정밀한 ACL, 권한 최소화 | **승인 워크플로우, 역할 분리** |
| **비유** | 자기 방 열쇠만 주기 | 현관문을 두 개의 열쇠로 잠그기 |

---

### 기술 융합: DevSecOps와 SoD의 자동화

클라우드 네이티브 환경에서 SoD는 코드로 구현된다.
- **Pull Request Review**: 개발자가 짠 코드는 반드시 다른 개발자(또는 리뷰어)의 승인이 있어야만 메인 코드에 합쳐진다. (Code Level SoD)
- **Deployment Gate**: 빌드 서버(자동화)가 코드를 만들고, 배포 서버(권한 분리)가 실제 운영기에 올린다. 개발자는 배포 버튼을 직접 누를 수 없다.

- **📢 섹션 요약 비유**: 요리사(개발자)가 음식을 만들면, 시식가(리뷰어)가 독이 없는지 확인하고, 서버(배포 자동화)가 손님 상에 내놓는 완벽한 분업 시스템과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 은행 '거액 송금 시스템'의 SoD 로직 설계

- **문제 상황**: 은행 전산팀 직원이 테스트 계정을 이용해 허위 이체 요청을 보내고 직접 승인하여 자금을 횡령할 리스크 상존.
- **의사결정 (SoD 통제 로직 구현)**:
  1. **Dual Control 로직**: 송금 금액이 1억 원 이상일 경우, 시스템적으로 '관리자 승인' 단계를 강제로 생성한다.
  2. **Identity 대조**: 승인 API 호출 시 `session.user_id`와 요청 테이블의 `requester_id`를 비교하여 동일하면 `403 Forbidden`을 반환한다.
  3. **Role Exclusion**: IAM 시스템에서 '송금 요청자' 그룹과 '송금 승인자' 그룹을 상호 배타적(Mutually Exclusive)으로 설정한다.
  4. **Audit Trail**: 모든 승인 반려 및 수락 이력을 IP와 타임스탬프를 포함하여 위변조 방지 로그에 기록한다.

### 도입 체크리스트
- **Exception Path**: 긴급 상황(Break-glass) 시 SoD를 일시 우회할 수 있는 안전하고 기록 가능한 절차가 있는가?
- **Hierarchy Check**: 직속 상하 관계가 아닌 독립된 부서 간의 교차 승인이 이루어지고 있는가?
- **Automated Validation**: 승인 로직이 UI에서만 막혀 있는 게 아니라, 서버 API 레벨에서 단단히 검증되고 있는가?

### 안티패턴
- **명목상의 승인**: 승인자가 내용 확인 없이 '일괄 승인' 버튼만 누르는 관행. (시스템은 SoD 준수이나 실질적으론 위반)
- **과도한 분리 (Gridlock)**: 사소한 작업까지 모두 3~4명의 승인을 받게 하여 비즈니스 속도를 심각하게 저하시키는 경우. (Risk 기반의 차등 적용 필요)

- **📢 섹션 요약 비유**: "빨리 가려면 혼자 가고, 멀리 가려면 함께 가라"는 말처럼, 보안 사고 없이 오랫동안 사업을 하려면 조금 느리더라도 서로를 확인하는 '동행 보안'이 필수입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Risk)** | 내부 부정 사고 80% 감소 | 단독 범행이 불가능한 구조를 통해 횡령 및 조작 차단 |
| **정량적 (Quality)** | 배포 결함률 하락 | 이중 검토(Peer Review)를 통해 개발자의 단순 실수 방지 |
| **정성적 (Compliance)** | 외부 감사 수검 신뢰도 확보 | 법규에서 요구하는 직무 분리 통제 항목에 대한 기술적 증빙 마련 |
| **정성적 (Standard)** | 책임 경영 체계 확립 | 각 단계별 책임자를 명확히 하여 사고 시 원인 파악 및 수습 신속화 |

### 미래 전망
- **AI-Managed SoD**: AI가 전사 권한 지도를 분석하여, 자신도 모르게 형성된 '위험한 권한 조합(Toxic Combinations)'을 실시간으로 찾아내고 경고할 것이다.
- **Blockchain Approval**: 승인 이력을 블록체인에 기록하여, 승인자가 나중에 "나는 승인한 적 없다"고 부인하는 것을 원천 차단하는 기술이 도입될 것이다.

### 참고 표준
- **NIST SP 800-53 AC-5**: Separation of Duties (연방 정부 보안 통제 항목)
- **ISO/IEC 27001 제어 항목 A.6.1.2**: 직무 분리 지침
- **COBIT**: IT 거버넌스 프레임워크 (SoD 강조)

- **📢 섹션 요약 비유**: SoD는 불신의 기술이 아니라, 인간의 불완전함을 인정하고 서로를 지켜주는 '상호 존중의 기술'입니다. 잘 설계된 SoD는 우리 조직을 가장 투명하고 단단하게 만듭니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 우리 반 청소 당번을 정할 때, **"청소 검사하는 친구"와 "청소를 하는 친구"를 서로 다른 사람으로 정하는 것**과 같아요.
2. 만약 내가 청소하고 내가 검사하면 대충 하고 "다 했다!"라고 거짓말할 수 있으니까, **서로 확인해주는 규칙**을 만든 거랍니다.
3. 이렇게 둘이서 힘을 합치고 서로 도와주면, 우리 교실이 훨씬 더 깨끗하고 공평해져요!
