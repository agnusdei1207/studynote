# 501. XSS 유형 - Reflected, Stored, DOM-based XSS

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: XSS (크로스 사이트 스크립팅)는 악성 스크립트가 피해자에게 전달되는 경로와 실행 시점에 따라 **반사형(Reflected), 저장형(Stored), DOM 기반(DOM-based)**의 세 가지 주요 유형으로 분류된다.
> 2. **가치**: 각 유형별로 공격의 전파 범위와 지속성이 다르기 때문에, 개발자는 자신의 시스템 아키텍처에서 데이터가 어떻게 흐르고 렌더링되는지 파악하여 유형별 맞춤 방어 전략을 수립해야 한다.
> 3. **융합**: 고전적인 서버 사이드 렌더링 환경에서는 반사형과 저장형이 주를 이루었으나, 현대의 클라이언트 사이드 렌더링(SPA) 환경에서는 브라우저 내부 스크립트 조작을 통한 DOM 기반 XSS의 비중이 급격히 높아지고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: XSS는 공격자가 웹 사이트에 악성 스크립트를 삽입하여, 해당 사이트를 방문하는 사용자의 브라우저에서 스크립트가 강제로 실행되게 하는 공격이다. 이때 악성 코드가 서버에 저장되는지, 아니면 요청 파라미터에 포함되어 즉시 되돌아오는지, 혹은 브라우저 메모리 내에서만 조작되는지에 따라 유형이 갈린다.

- **💡 비유**: 
  - **Reflected**: 도둑이 나에게 **"이 편지를 읽어봐"라고 건네준 종이**에 최면 주문이 적혀 있어 읽자마자 가방을 넘겨주는 상황.
  - **Stored**: 도둑이 **마을 게시판에 최면 주문**을 붙여놓아, 게시판을 보는 모든 사람이 홀리는 상황.
  - **DOM-based**: 도둑이 나에게 준 **돋보기(브라우저 스크립트)**가 알고 보니 글자를 왜곡해서 보여주어, 내가 엉뚱한 행동을 하게 만드는 상황.

- **등장 배경**:
  1. **사용자 입력의 동적 출력**: 사용자의 검색어, 이름, 댓글 등을 화면에 그대로 보여주는 기능이 늘어나면서 인젝션의 통로가 되었다.
  2. **자바스크립트의 비중 확대**: 웹 앱의 로직이 서버에서 브라우저(Client)로 이동하면서, 서버를 거치지 않는 취약점(DOM-based)이 새롭게 부상했다.
  3. **공격 타겟의 변화**: 서버 자체를 공격하기보다, 서버가 신뢰하는 '사용자'의 세션이나 권한을 탈취하는 것이 해커들에게 더 효율적인 수익원이 되었다.

- **📢 섹션 요약 비유**: 독을 묻힌 화살을 직접 쏘느냐(Reflected), 우물에 독을 타느냐(Stored), 아니면 물 마시는 컵 자체를 오염시키느냐(DOM-based)의 차이와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### XSS 3대 유형 비교 및 특징

| 유형 | 데이터 저장 위치 | 주요 전파 경로 | 지속성 | 특징 |
|:---|:---:|:---|:---:|:---|
| **Reflected (반사형)** | 없음 (일회성) | URL 파라미터, 이메일 링크 | 낮음 | 가장 흔함, 피싱 이메일에 악용 |
| **Stored (저장형)** | **서버 DB / 파일** | 게시판, 댓글, 사용자 프로필 | **높음** | 접속자 전원 감염 가능, 파괴력 큼 |
| **DOM-based (DOM 기반)** | 클라이언트 메모리 | URL Fragment (`#`), 로컬 스토리지 | 낮음 | 서버 로그에 남지 않음 (Client-only) |

---

### XSS 유형별 공격 아키텍처 비교

데이터가 어떤 경로로 흐르는지에 따른 구조적 차이를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      XSS 주요 유형별 데이터 흐름 아키텍처                        │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │  [1. Reflected XSS]                                                         │
 │     공격자 ──(URL with Script)──▶ 피해자 ──(Request)──▶ 서버 ──(Response)──▶ 피해자 │
 │                                                               (즉시 반사 실행)  │
 │                                                                             │
 │  [2. Stored XSS]                                                            │
 │     공격자 ──(Script)──▶ 서버(DB 저장)                                        │
 │                           │                                                 │
 │     피해자 ──────────────▶ 서버 ──(Stored Script 포함 Page)──▶ 피해자            │
 │                                                               (지속적 실행)    │
 │                                                                             │
 │  [3. DOM-based XSS]                                                         │
 │     공격자 ──(URL with #Script)──▶ 피해자 ──▶ 브라우저 내부 JS 로직 실행 ──▶ 피해자 │
 │                                            (서버 전송 없이 DOM에서 조작)        │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 
1) **반사형**은 서버가 입력받은 스크립트를 응답 페이지에 그대로 담아 보낼 때 발생한다. 공격자는 피해자가 클릭하도록 교묘하게 유도한다. 
2) **저장형**은 악성 코드가 서버 DB에 영구적으로 저장되므로, 해당 페이지를 방문하는 모든 사용자가 피해를 입는 가장 위험한 형태다. 
3) **DOM 기반**은 데이터가 서버로 전송되지 않고 브라우저의 자바스크립트가 URL의 해시값(`#`) 등을 읽어 처리하는 과정에서 발생한다. 서버는 이 요청을 알 수 없으므로(WAF가 잡기 어려움) 클라이언트 단의 시큐어 코딩이 더욱 중요하다.

---

### 심층 동작 원리: DOM-based XSS의 핵심 (Sources & Sinks)

서버를 거치지 않는 공격의 핵심 메커니즘이다.
1. **Sources (입력원)**: 공격자가 조작할 수 있는 자바스크립트 객체.
   - `location.search`, `location.hash`, `document.referrer`, `localStorage` 등.
2. **Sinks (실행점)**: 데이터를 실행 가능한 코드로 바꾸는 위험한 함수.
   - `eval()`, `setTimeout()`, `innerHTML`, `document.write()` 등.
3. **취약점 발생**: Source에서 받은 데이터를 검증 없이 Sink 함수에 전달할 때 브라우저 내부에서 공격이 완성된다.

- **📢 섹션 요약 비유**: 무전기(Source)로 받은 명령을 확인도 안 하고 바로 확성기(Sink)로 내보내서 부대 전체에 가짜 명령(공격)을 전달하는 사고와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### XSS vs CSRF (Cross-Site Request Forgery)

두 공격은 이름이 비슷하여 자주 혼동되지만, 공격의 본질이 다르다.

| 비교 항목 | XSS (Cross-Site Scripting) | CSRF (Cross-Site Request Forgery) |
|:---|:---|:---|
| **공격 목표** | **사용자 브라우저 내 데이터(쿠키) 탈취** | **사용자의 권한 도용 (비정상 요청 실행)** |
| **스크립트 실행** | 필수 (악성 코드가 실행됨) | 불필요 (이미지 태그 호출만으로도 가능) |
| **방어 수단** | 출력 인코딩, CSP | Anti-CSRF 토큰, SameSite 쿠키 |
| **비유** | 사용자가 가짜 주문서에 서명하게 함 | 사용자의 손을 빌려 몰래 주문서에 사인함 |

---

### 기술 융합: Modern Framework (React/Vue)와 XSS

최신 프레임워크는 설계 단계에서 XSS 방어를 내재화하고 있다.
- **자동 인코딩**: 중괄호(`{{ }}` 또는 `{ }`) 내의 변수는 자동으로 인코딩되어 출력된다.
- **DangerouslySetInnerHTML**: 개발자가 명시적으로 위험을 인지하고 함수를 호출하지 않는 한, HTML 주입이 기본적으로 차단된다.
- **주의점**: 그럼에도 불구하고 `href` 속성에 `javascript:` 구문을 넣는 등의 시도는 여전히 XSS를 유발할 수 있다.

- **📢 섹션 요약 비유**: 최신 자동차(현대 프레임워크)는 안전벨트(자동 인코딩)가 기본으로 채워져 나오지만, 운전자가 굳이 안전벨트를 풀고(DangerouslySet...) 운전하면 여전히 사고가 날 수 있는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 사내 위키(Wiki)의 저장형 XSS 사고 조사 및 조치

- **문제 상황**: 직원이 작성한 위키 문서에 접속하면 갑자기 타 사이트로 리다이렉트되거나 로그인 창이 다시 뜨는 현상 발생.
- **의사결정 (조치 프로세스)**:
  1. **유형 판별**: DB 조회 결과 특정 문서의 본문에 `<script>location.href=...</script>` 코드가 저장되어 있음을 확인. (Typical **Stored XSS**)
  2. **긴급 격리**: 해당 문서를 비공개 처리하고, 이미 탈취되었을 가능성이 있는 세션 쿠키를 무효화하기 위해 전사 로그아웃 처리를 수행한다.
  3. **근본 조치 (서버 측)**: 위키 렌더링 엔진에 HTML Sanitizer (예: DOMPurify)를 도입하여, 화이트리스트 기반의 안전한 태그만 남기고 나머지는 제거하도록 수정한다.
  4. **심층 방어**: 모든 서비스 도메인에 `Content-Security-Policy` 헤더를 적용하여 신뢰할 수 없는 외부 도메인으로의 데이터 전송을 차단한다.

### 도입 체크리스트
- **Context-aware Encoding**: 데이터가 HTML 태그 사이, 속성 안, 자바스크립트 변수 안 중 어디에 위치하는지에 따라 다른 인코딩을 적용하고 있는가?
- **HttpOnly Cookie**: 모든 세션 쿠키에 HttpOnly 속성을 부여하여 XSS 성공 시에도 세션 탈취를 방지하고 있는가?
- **Scanner Verification**: SAST/DAST 도구가 세 가지 XSS 유형을 모두 탐지할 수 있는 룰셋을 가지고 있는가?

### 안티패턴
- **필터링 후 실행**: `<script>` 문자열만 지우고 실행하는 방식. 공격자는 `<scr<script>ipt>` 처럼 지워진 후 다시 완성되는 코드로 우회한다.
- **서버 검증 맹신**: DOM-based XSS는 데이터가 서버로 가지 않으므로, 서버 측 필터링만 믿고 클라이언트 단 검증을 소홀히 하면 낭패를 본다.

- **📢 섹션 요약 비유**: 우물 물(서버 데이터)만 정수한다고 안심하지 말고, 물컵(브라우저 DOM)에 묻은 먼지까지 닦아내야 진정으로 깨끗한 물을 마실 수 있다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 계정 탈취 사고 90% 예방 | HttpOnly 쿠키와 인코딩 결합 시 쿠키 탈취 불가능 |
| **정량적 (Compliance)** | 개인정보보호법 준수 | 악성 스크립트를 통한 개인정보 무단 수집 및 유출 시도 차단 |
| **정성적 (Reliability)** | 서비스 가용성 확보 | 스크립트 실행으로 인한 페이지 무한 루프, 다운 현상 방지 |
| **정성적 (Standard)** | 시큐어 코딩 문화 정착 | 모든 출력 지점에 대한 검증 프로세스를 통해 개발 품질 상향 평준화 |

### 미래 전망
- **Trusted Types의 보편화**: 구글이 주도하는 'Trusted Types'가 표준화되어, 브라우저가 보안 정책에 위반되는 데이터 타입의 DOM 주입을 원천 거부하게 될 것이다.
- **AI-Native XSS Defense**: AI가 사이트의 전체적인 데이터 흐름을 이해하여, 룰셋 없이도 맥락상 위험한 스크립트 실행을 실시간으로 차단하는 지능형 방어 체계가 도입될 것이다.

### 참고 표준
- **OWASP TOP 10 - A03:2021**: Injection (XSS 포함)
- **CWE-79**: XSS 공식 분류
- **W3C Content Security Policy (CSP)**: 표준 보안 정책 명세

- **📢 섹션 요약 비유**: 수동으로 도둑을 쫓던 과거에서, 집 구조 자체가 침입자에게는 미로처럼 작동하고 센서가 실시간으로 대응하는 '스마트 보안 건축'의 시대로 진화하고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **반사형**은 나쁜 친구가 보낸 편지를 읽자마자 손이 멋대로 움직여 내 간식을 전해주는 거예요.
2. **저장형**은 학교 게시판에 적힌 거짓말을 본 친구들 모두가 가짜 장난에 속아 넘어가는 거랍니다.
3. **DOM 기반**은 내 안경에 나쁜 마법이 걸려 있어서, 착한 글자도 나쁜 명령으로 잘못 읽게 만드는 거예요. 모두 조심해야겠죠?
