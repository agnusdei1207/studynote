# 492. DAST (Dynamic Application Security Testing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DAST (동적 애플리케이션 보안 테스팅)는 애플리케이션이 실행 중인 상태에서 외부로부터 다양한 공격 페이로드를 주입하고 반응을 분석하여 **런타임 취약점을 식별하는 블랙박스(Black-box) 테스트 기법**이다.
> 2. **가치**: 소스 코드 없이도 웹 인터페이스나 API를 통해 실제 공격자의 관점에서 보안 상태를 점검하며, 서버 설정 오류, 인증 메커니즘의 결함, 그리고 실제 실행 경로에서만 발생하는 취약점을 정확히 찾아낸다.
> 3. **융합**: 웹 취약점 스캐너, 프록시 도구(Burp Suite 등), 그리고 QA 단계의 자동화 테스트와 결합되어 배포 전 최종 보안 관문 역할을 수행하며, IAST와 상호 보완적으로 사용되어 오탐률을 낮춘다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: DAST는 "동적(Dynamic)"이라는 명칭대로 프로그램을 실제로 가동하면서 테스트를 진행한다. HTTP/HTTPS 프로토콜을 통해 요청을 보내고 서버의 응답 코드, 응답 본문, 처리 시간 등을 관찰하여 취약점을 판단한다. 애플리케이션의 내부 구조를 모르는 상태에서 진행되므로 블랙박스 테스팅의 범주에 속한다.

- **💡 비유**: DAST는 **"완성된 건물 밖에서 도둑이 되어 직접 담을 넘어보고, 창문을 흔들어보고, 열쇠 구멍에 여러 가지 가짜 열쇠를 넣어보며 침입 가능한 틈을 찾는 실전 침투 훈련"**과 같습니다.

- **등장 배경**:
  1. **현실적인 공격 대응**: 소스 코드 분석(SAST)만으로는 알 수 없는 웹 서버 설정, 데이터베이스 연동상의 취약점, 런타임 환경의 보안 미비점을 보완해야 했다.
  2. **언어 독립적 보안성 확보**: 소스 코드가 어떤 언어(Java, Python, PHP 등)로 작성되었든 관계없이, 표준 웹 프로토콜을 사용하는 모든 서비스에 대해 일관된 테스트가 필요했다.
  3. **낮은 오탐률 선호**: 실제 공격이 성공했을 때만 취약점으로 리포팅하므로, 개발자가 수정해야 할 우선순위를 정하기에 SAST보다 유리하다.

- **📢 섹션 요약 비유**: 요리사가 레시피를 검사하는 것이 아니라, 완성된 요리를 직접 먹어보고 맛이 이상하거나(취약점 발생) 배탈이 나는지(서버 오류) 확인하는 실전 시식 테스트와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### DAST의 주요 테스팅 프로세스 및 메커니즘

DAST 도구는 크게 탐색(Crawl)과 공격(Audit)의 두 단계를 거친다.

| 단계 | 수행 작업 | 상세 내용 | 비유 |
|:---|:---|:---|:---|
| **1. 탐색 (Crawling)** | 사이트 구조 파악 | 모든 URL, 폼(Form), 파라미터, 헤더를 수집 | 건물의 모든 출입구 확인 |
| **2. 입력값 추출** | 공격 지점(Point) 식별 | 사용자 입력이 가능한 모든 변수 리스트업 | 열려 있는 창문 목록 작성 |
| **3. 공격 (Fuzzing)** | 페이로드 주입 | SQLi, XSS, Path Traversal 등 공격 구문 전송 | 가짜 열쇠 끼워보기 |
| **4. 응답 분석** | 취약점 판단 | HTTP 500 에러, 특정 문자열 노출 등을 체크 | 문이 열리는지 확인 |
| **5. 리포팅** | 결과 및 가이드 생성 | 발견된 취약점의 위험도와 조치 방안 제시 | 방범 리포트 작성 |

---

### DAST 자동화 테스팅 아키텍처

DAST가 테스트 및 스테이징 환경에 어떻게 통합되는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      DAST 기반 동적 보안 검증 및 자동화 아키텍처                  │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [CI/CD 파이프라인]                                [테스트 대상 환경]           │
 │        │                                          (Staging / QA)            │
 │        ▼                                                │                   │
 │  ┌──────────────┐          ┌──────────────────┐         │   ┌─────────────┐ │
 │  │  Build &    │          │   DAST Scanner   │         │   │  Web App    │ │
 │  │  Deploy     │ ──▶      │ (ZAP, Burp, 등)  │ ◀──(HTTP)─┼─▶ │  (Runtime)  │ │
 │  └──────────────┘          └──────────────────┘         │   └─────────────┘ │
 │                                     │                   │          │        │
 │                                     ▼                   │          ▼        │
 │                            ┌──────────────────┐         │   ┌─────────────┐ │
 │                            │  취약점 리포트    │ ◀───────┘   │  DB / Server │ │
 │                            │  (Critical?)     │             │  (Config)   │ │
 │                            └──────────────────┘             └─────────────┘ │
 │                                                                             │
 │  ※ 주요 점검 항목: SQLi, XSS, CSRF, 세션 관리, 보안 헤더, SSL/TLS 설정 등         │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** DAST는 애플리케이션이 정상적으로 배포된 **스테이징 또는 QA 환경**에서 가동된다. CI/CD 툴이 배포를 완료하면 DAST 스캐너가 구동되어 실제 운영 환경과 유사한 조건에서 공격을 시도한다. 이때 단순히 코드상의 결함뿐만 아니라, 웹 서버(Nginx, Apache)의 설정 오류나 DB 권한 문제 등 인프라와 관련된 취약점도 함께 잡아낸다. 특히 자동화된 DAST는 UI 테스트 시나리오(Selenium 등)와 결합하여 로그인이나 복잡한 비즈니스 로직 내부의 보안성까지 점검할 수 있다.

---

### 심층 동작 원리: 차등적 분석 (Differential Analysis)

DAST가 취약점을 확신하는 핵심 논리다.
1. **정상 요청**: 정상적인 값을 보내 응답(A)을 기록한다.
2. **변조 요청**: 특정 공격 구문(예: `' OR '1'='1`)을 포함해 요청을 보내고 응답(B)을 기록한다.
3. **대조**: 응답 A와 B의 본문 길이나 포함된 단어를 비교한다. 만약 B에서 로그인에 성공하거나 DB 에러가 발생했다면 취약점으로 확정한다.

- **📢 섹션 요약 비유**: "주인님"이라고 불렀을 때와 "도둑놈"이라고 불렀을 때 집 안의 대답이 같다면, 그 집의 신분 확인 시스템(인증)이 고장 났다고 판단하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SAST vs DAST vs IAST 비교 (보안 테스팅 관점)

| 비교 항목 | SAST (정적) | DAST (동적) | IAST (상호작용) |
|:---|:---|:---|:---|
| **분석 환경** | 소스 코드 | **실행 중인 앱** | 코드 + 런타임 에이전트 |
| **지식 수준** | 화이트박스 (내부 앎) | **블랙박스 (내부 모름)** | 그레이박스 (둘 다 활용) |
| **오탐률** | 높음 (False Positive) | **낮음 (실제 발생 확인)** | 매우 낮음 |
| **발견 항목** | CWE (보안 약점) | **CVE / OWASP (취약점)** | 두 영역 모두 |
| **수정 가이드** | 코드 라인 지정 가능 | **HTTP 요청/응답 정보** | 코드 라인 및 흐름 제시 |

---

### 기술 융합: API 보안과 DAST

최근의 웹은 화면(HTML)보다 데이터(JSON/API) 중심이다.
- **REST/GraphQL 스캔**: DAST 도구는 Swagger(OpenAPI) 명세서를 읽어 들여, 사람이 일일이 클릭하지 않아도 모든 API 엔드포인트에 대해 자동으로 인젝션이나 권한 상승 테스트를 수행한다.

- **📢 섹션 요약 비유**: 앞문(웹 화면)으로만 들어가 보던 도둑이, 이제는 환풍구나 하수구(API) 설계도를 입수해서 모든 구멍에 연기를 피워보며 어디로 새 나오는지(데이터 유출) 확인하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 정기 보안 점검 자동화 및 탐지 한계 극복

- **문제 상황**: 매달 수행하는 모의해킹 인력이 부족하고, 자동화 스캐너는 로그인 페이지 뒤쪽의 기능을 점검하지 못하는 한계 발생.
- **의사결정 (DAST 최적화 전략)**:
  1. **인증 연동**: DAST 도구에 로그인 매크로 또는 인증 토큰(JWT) 정보를 주입하여 권한이 필요한 페이지(Admin 등)까지 깊숙이 스캔하도록 설정한다.
  2. **API 기반 스캔**: 프론트엔드와 백엔드가 분리된 구조에서는 백엔드 API 명세서(Swagger)를 기반으로 직접 호출 테스트를 수행한다.
  3. **WAF 연동 테스트**: 실운영 환경과 동일한 방화벽 설정을 두고 DAST를 돌려, 우리 방어 체계가 실제 공격 패턴을 얼마나 잘 차단하는지 검증한다.
  4. **수동 모의해킹 병행**: 로직 우회나 다단계 인증(MFA) 우회처럼 DAST가 못 잡는 복잡한 결함은 전문 보안 요원이 수동으로 점검한다.

### 도입 체크리스트
- **Coverage**: 우리 서비스의 모든 페이지와 API가 스캔 범위에 포함되었는가?
- **Safety**: 테스트 중 DB 데이터가 삭제되거나 시스템이 다운될 위험(Side effect)을 관리하고 있는가?
- **Alerting**: 스캔 결과 중 즉각 조치가 필요한 Critical 항목이 개발자에게 실시간으로 전달되는가?

### 안티패턴
- **운영 DB 대상 스캔**: 테스트용 데이터가 아닌 실제 고객 데이터가 있는 DB를 대상으로 DAST를 돌리는 행위. 공격 구문 때문에 데이터가 변조되거나 삭제될 수 있다.
- **보고서 양 불리기**: 스캐너가 뱉어낸 수천 개의 '정보성(Info)' 경고를 모두 심각한 취약점으로 둔갑시켜 개발팀에 넘기는 행위.

- **📢 섹션 요약 비유**: 도둑 들지 말라고 방범창을 흔들어보는 것(DAST)은 좋지만, 너무 세게 흔들어서 창문을 깨버리거나 집 안의 물건을 부수면 안 된다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Accuracy)** | 실제 취약점 탐지율 향상 | 런타임에서만 나타나는 설정 및 인증 오류 90% 이상 식별 |
| **정량적 (Resource)** | 보안 검수 리소스 절감 | 단순 반복적인 취약점 점검을 자동화하여 전문가 인력을 고난도 분석에 집중 |
| **정성적 (Reality)** | 실제 공격 대응력 강화 | 공격자의 관점에서 시스템을 바라봄으로써 실전적인 방어 체계 구축 |
| **정성적 (Standard)** | 글로벌 보안 규격 준수 | PCI-DSS, ISO 27001 등에서 요구하는 정기적 동적 점검 요건 충족 |

### 미래 전망
- **Autonomous DAST**: AI가 사이트 구조를 스스로 학습하고, 인간 모의해커처럼 비즈니스 로직의 취약한 연결 고리를 찾아내어 창의적인 공격 시나리오를 생성할 것이다.
- **Shift-Right Security**: 운영 환경에서도 실제 트래픽에 영향을 주지 않는 범위 내에서 상시로 DAST를 구동하여, 새로운 제로데이 취약점 발생 시 즉시 탐지하는 체계로 발전할 것이다.

### 참고 표준
- **OWASP WSTG (Web Security Testing Guide)**: DAST 수행의 교과서적 가이드
- **PTES (Penetration Testing Execution Standard)**: 침투 테스트 표준 프로세스

- **📢 섹션 요약 비유**: 과거에는 정해진 시간에만 성벽을 검사했다면, 미래에는 24시간 내내 보이지 않는 투명 인간(AI 스캐너)이 성벽 주위를 돌며 작은 틈이라도 생기면 즉시 경보를 울리는 체계로 나아갈 것입니다.

1. 새로 산 자전거가 튼튼한지 알아보기 위해, **직속으로 타보기도 하고 울퉁불퉁한 길도 달려보며 어디 고장 난 곳은 없는지 확인하는 것**과 같아요.
2. 멈춰있는 자전거를 눈으로만 보는 것보다, 직접 페달을 밟아봐야 체인이 빠지는지 브레이크가 잘 듣는지 정확히 알 수 있겠죠?
3. 이렇게 실제로 써보면서 문제를 찾아내면, 나중에 길을 가다가 갑자기 자전거가 망가져서 다치는 일을 미리 막을 수 있답니다!
