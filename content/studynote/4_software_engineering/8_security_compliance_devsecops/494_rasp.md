# 494. RASP (Runtime Application Self-Protection)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RASP (런타임 애플리케이션 자가 보호)는 애플리케이션이 실행되는 런타임 환경에 상주하며, **입출력 데이터와 내부 실행 흐름을 실시간으로 감시하여 공격 시도를 스스로 탐지하고 차단하는** 보안 기술이다.
> 2. **가치**: 네트워크 단에서 동작하는 WAF(웹 방화벽)가 알 수 없는 '애플리케이션 내부의 맥락(Context)'을 이해함으로써, 제로데이 공격이나 고도로 변조된 페이로드를 오탐 없이 정교하게 방어할 수 있다.
> 3. **융합**: 클라우드 네이티브 보안, 서버리스 아키텍처, 그리고 IAST 테스팅 도구와 기술적 기반을 공유하며, 외부 방화벽에만 의존하던 기존의 '경계 보안'을 넘어 '애플리케이션 중심의 능동적 방어'를 실현한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: RASP는 애플리케이션의 "면역 체계"와 같다. 외부의 공격이 네트워크 장비를 뚫고 서버 안으로 들어왔을 때, 애플리케이션 자체가 "이 데이터는 내 로직을 파괴하려는 독성 물질이다"라고 판단하여 스스로 실행을 멈추거나 공격자의 세션을 끊어버리는 기술이다.

- **💡 비유**: RASP는 **"성문을 지키는 경비병(WAF)을 지나 성 안으로 잠입한 스파이가 국왕의 음식에 독을 타려는 순간, 음식을 먹기 전 미리 맛을 보고 독을 감지하여 식사를 중단시키는 전담 시종"**과 같습니다.

- **등장 배경**:
  1. **네트워크 보안의 한계**: WAF는 암호화된 트래픽(HTTPS) 내부나 복잡하게 인코딩된 공격 구문을 분석하는 데 한계가 있고, 정상적인 비즈니스 로직 우회 공격을 잡아내지 못한다.
  2. **시스템 복잡성 증가**: 클라우드와 MSA 환경에서는 고정된 네트워크 경계가 사라지면서, 각 애플리케이션이 스스로를 보호해야 할 필요성이 커졌다.
  3. **알려지지 않은 위협(Zero-day)**: 패턴 기반의 방어 체계는 새로운 공격에 취약하므로, 공격 패턴이 아닌 '행위의 비정상성'을 감시하는 기술이 요구되었다.

- **📢 섹션 요약 비유**: 도둑이 창문을 넘는 것을 막는 담장(WAF)도 필요하지만, 도둑이 집 안에 들어와서 금고를 만지는 순간 손목을 잡아채는 내부 경보 장치(RASP)가 더 확실한 최후의 보루가 된다는 원리입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### RASP의 구성 요소 및 방어 메커니즘

RASP는 애플리케이션과 한 몸처럼 움직이기 위해 '인스트루멘테이션' 기술을 사용한다.

| 구성 요소 | 역할 | 상세 동작 | 비유 |
|:---|:---|:---|:---|
| **RASP 에이전트** | 통합 및 감시 | WAS(JVM, .NET 등) 시작 시 로드되어 핵심 라이브러리 함수를 가로챔 | 몸속의 백혈구 |
| **정책 엔진** | 판단 기준 | 들어온 데이터가 SQLi, XSS 등의 공격 패턴인지 로직 위반인지 대조 | 면역 정보 데이터베이스 |
| **차단기 (Blocker)** | 실행 제어 | 공격 확정 시 예외(Exception)를 발생시켜 함수 실행을 즉시 중단 | 혈전 생성 및 차단 |
| **보고 모듈** | 가시성 제공 | 공격 시도의 상세 정보(공격자 IP, 소스 위치, 페이로드)를 SIEM으로 전송 | 통증 신호 전달 |

---

### RASP vs WAF 방어 지점 비교 아키텍처

두 보안 기술이 보호하는 영역의 차이를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      WAF(경계 보안)와 RASP(런타임 보안)의 협력 구조               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [외부 공격자] ────▶ [인터넷] ────▶ [WAF / IPS] ────▶ [웹 서버 / WAS] ─────▶ [DB] │
 │                                      │                 │                    │
 │                                      │                 │  ┌──────────────┐  │
 │                                      │                 │  │    RASP      │  │
 │                                      │                 │  │ (에이전트)    │  │
 │                                      │                 │  └──────┬───────┘  │
 │                                      │                 │         │          │
 │  1. 네트워크 단 방어 ─────────────────┘                 │         │          │
 │     - 패턴 매칭 (Signature)                             │         │          │
 │     - IP 차단, DDoS 방어                                 │         │          │
 │                                                         │         │          │
 │  2. 애플리케이션 내부 방어 ──────────────────────────────┘         │          │
 │     - SQL 쿼리 실행 직전 검증 ──────────────────────────────────────┘          │
 │     - 시스템 쉘 호출 감시                                                     │
 │     - 비정상적인 메모리 접근 차단                                              │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** **WAF**는 성문 밖에서 검문소를 운영하며 대량의 공격을 1차적으로 걸러낸다. 하지만 공격자가 교묘하게 위장하거나 암호화된 터널을 통해 들어오면 WAF를 통과할 수 있다. **RASP**는 성 안(WAS 내부)에서 대기하다가, 데이터가 실제로 **DB 쿼리로 변환되거나 시스템 명령어로 실행되기 직전**에 이를 가로챈다. 이때 RASP는 "이 데이터가 실제로 DB 엔진에 도달하면 어떤 영향을 줄지"에 대한 맥락을 100% 알고 있으므로, WAF가 놓친 정교한 공격을 '실행 직전'에 완벽히 차단한다.

---

### 심층 동작 원리: 함수 후킹 (Function Hooking)

RASP가 공격을 인지하는 물리적 방식이다.
1. **바이트코드 수정**: Java의 경우 에이전트가 JVM에 로드될 때 `java.sql.Statement.executeQuery()`와 같은 메서드에 자신의 코드를 삽입한다.
2. **컨텍스트 분석**: 쿼리가 실행되기 전, RASP는 호출 스택(Call Stack)을 분석하여 이 요청이 정상적인 사용자로부터 온 것인지, 아니면 변조된 것인지 판단한다.
3. **능동적 대응**: 공격으로 판단되면 쿼리 전송을 중단하고 로그를 남기며, 사용자에게는 "보안 정책 위반" 에러를 반환한다.

- **📢 섹션 요약 비유**: 수도꼭지에서 물이 나오기 바로 직전에 필터(RASP)를 달아서, 물속에 나쁜 세균(공격 페이로드)이 섞여 있는 것이 확인되면 0.1초 만에 밸브를 잠가버리는 원리와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### WAF vs RASP 비교 분석

| 비교 항목 | WAF (Web Application Firewall) | RASP (Runtime Protection) |
|:---|:---|:---|
| **설치 위치** | 네트워크 경계 (In-front of App) | **애플리케이션 내부 (Inside App)** |
| **분석 대상** | HTTP 요청/응답 패킷 | **코드 실행 흐름, 변수 상태, DB 쿼리** |
| **오탐률** | 높음 (맥락 파악 불가로 인한 과잉 차단) | **매우 낮음 (실행 결과 예측 가능)** |
| **성능 영향** | 네트워크 지연(Latency) 유발 가능 | **애플리케이션 응답 시간 소폭 증가** |
| **제로데이 대응** | 취약하함 (패턴 업데이트 필요) | **강력함 (비정상 행위 자체를 차단)** |

---

### 기술 융합: Cloud-Workload Protection (CWPP)

RASP는 현대 클라우드 보안 모델의 핵심 요소로 통합되고 있다.
- **Serverless Security**: 서버 주소를 알 수 없는 Lambda 환경에서 RASP는 코드 내부에 포함되어 스스로를 보호하는 유일한 수단이 된다.
- **Container Security**: 사이드카(Sidecar) 패턴으로 RASP가 배치되어 컨테이너 간의 동적인 통신 보안을 담당한다.

- **📢 섹션 요약 비유**: 성벽을 높게 쌓는 방식에서, 성 안의 모든 사람에게 방탄조끼(RASP)를 입히고 독극물 감지 센서를 하나씩 들려주는 개인 보안 시대로 변하고 있는 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 패치가 불가능한 레거시 시스템의 긴급 방어

- **문제 상황**: 중단할 수 없는 핵심 서비스에서 치명적인 라이브러리 취약점(예: Log4j)이 발견되었으나, 코드 수정 및 재배포에 수일이 소요되는 상황.
- **의사결정 (RASP 도입 및 가상 패치)**:
  1. **Hot-fixing**: RASP 에이전트를 WAS에 즉시 로드하여 해당 취약점을 유발하는 함수(Sink)에 대한 필터링 정책을 적용한다. (코드 수정 없이 방어 가능)
  2. **가상 패치 (Virtual Patching)**: WAF보다 훨씬 정교하게 해당 취약점 경로만 선택적으로 차단하여 비즈니스 중단 없이 보안성을 확보한다.
  3. **가시성 분석**: 공격자가 어떤 서버의 어떤 소스 코드 라인을 통해 침투를 시도했는지 실시간 리포트를 받아, 향후 근본적인 패치 작업의 우선순위를 정한다.

### 도입 체크리스트
- **Compatibility**: 사용하는 WAS 버전, 프레임워크 버전과 충돌이 없는가?
- **Performance**: 서비스 응답 속도 저하가 비즈니스 허용 범위(보통 5ms 이내) 내에 있는가?
- **Reliability**: 보안 에이전트 오류 시 애플리케이션 전체가 멈추지 않도록 하는 'Fail-safe' 옵션이 있는가?

### 안티패턴
- **단일 솔루션 맹신**: RASP만 있으면 WAF나 IPS가 필요 없다고 생각하는 행위. RASP는 L7 보안에 특화되어 있어 L3/L4 수준의 네트워크 공격(DDoS 등)은 막지 못한다.
- **무분별한 차단 설정**: 런타임 특성을 제대로 분석하지 않고 모든 경고를 '차단(Block)'으로 두면, 정상적인 비즈니스 트래픽이 끊겨 장애가 발생할 수 있다.

- **📢 섹션 요약 비유**: 아무리 훌륭한 해독제(RASP)가 있어도, 애초에 독이 든 음식(네트워크 공격)이 들어오지 못하게 입구를 막는 것(WAF)이 가장 기본이 되어야 한다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 오탐 처리 비용 95% 절감 | WAF의 수많은 경고성 알람을 실제 실행 가능한 위험으로 필터링 |
| **정량적 (Agility)** | 사고 대응 속도 향상 | 취약점 발표 시 코드 수정 없이 런타임 정책만으로 수 분 내 방어 가능 |
| **정성적 (Precision)** | 정밀 타격 방어 | 공격 경로(URL -> Controller -> SQL) 전 과정을 가시화하여 근본 분석 지원 |
| **정성적 (Resilience)** | 자가 치유 능력 확보 | 공격을 받아도 서비스 중단 없이 스스로 공격자만 골라내는 강인함 확보 |

### 미래 전망
- **Self-Healing App**: 공격을 감지하면 RASP가 AI와 연동하여 자동으로 보안 패치 코드를 생성하고 스스로를 업데이트하는 자가 치유 애플리케이션으로 진화할 것이다.
- **Edge RASP**: CDN이나 Edge 컴퓨팅 지점에서 가벼운 RASP 로직이 동작하여, 사용자에게 더 가까운 곳에서 지능형 보안을 제공하게 될 것이다.

- **Gartner Hype Cycle**: Application Security 영역의 핵심 기술로 등재

- **📢 섹션 요약 비유**: 수동으로 약을 먹어 병을 고치던 시대에서, 몸속에 상주하며 암세포가 생기는 즉시 자동으로 제거하는 '나노 로봇 보안' 시대로 나아가고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 컴퓨터 프로그램 안에 **"나쁜 마법을 막는 투명 보호막"**을 씌워두는 것과 같아요.
2. 밖에서 나쁜 사람이 아무리 이상한 명령을 보내도, 프로그램이 직접 "이건 우리 주인을 괴롭히는 명령이야!"라고 알아차리고 스스로 멈춘답니다.
3. 누가 도와주지 않아도 컴퓨터 스스로가 나쁜 사람을 골라낼 수 있어서, 우리 소중한 정보가 훨씬 더 안전해지는 거예요!
