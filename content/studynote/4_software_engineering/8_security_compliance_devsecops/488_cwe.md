# 488. CWE (Common Weakness Enumeration)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CWE (공통 보안 약점 사전)는 소프트웨어 및 하드웨어에서 발생하는 **보안 취약점의 근본 원인이 되는 '보안 약점(Weakness)'**들을 체계적으로 분류하고 목록화한 오픈 커뮤니티 기반의 지식 베이스다.
> 2. **가치**: 보안 전문가, 개발자, 도구 제작자 간의 공통 언어를 제공하여, 소스 코드 분석(SAST)이나 아키텍처 설계 시 보안 결함을 식별하고 예방하는 기준점(Baseline) 역할을 한다.
> 3. **융합**: CVE (이미 발견된 취약점 사례)의 부모 격이며, CVSS (위험도 평가)와 결합하여 취약점 관리의 전 과정을 표준화한다. 특히 'CWE Top 25'는 기업의 시큐어 코딩 룰셋 수립의 핵심 지표로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: CWE는 MITRE Corporation에서 관리하는 데이터베이스로, 소프트웨어의 설계, 아키텍처, 또는 코드 구현 단계에서 보안 사고를 유발할 수 있는 기술적 결함을 분류한다. 취약점(CVE)이 "특정 제품의 특정 버그"라면, 보안 약점(CWE)은 "버그가 발생하는 근본적인 유형(예: 입력값 검증 미흡)"을 말한다.

- **💡 비유**: CWE는 **"의학 사전에 등록된 수천 가지 질병의 원인(세균 감염, 유전적 결함 등) 목록"**과 같습니다. 특정 환자의 병명(CVE)은 제각각일 수 있지만, 그 병을 일으킨 근본적인 원인(CWE)을 알면 예방법을 세울 수 있는 것과 같은 원리입니다.

- **등장 배경**:
  1. **보안 의사소통의 혼선**: 같은 취약점을 두고 개발자는 "코딩 실수", 보안팀은 "인젝션 위협"이라고 다르게 부르던 문제를 해결하기 위해 표준 명칭이 필요해졌다.
  2. **보안 도구의 표준화**: 코드 분석 도구(SAST)들이 각자 다른 기준으로 결과를 리포팅하던 것을 CWE 번호로 통합하여 도구 간의 성능 비교와 결과 통합을 가능케 했다.
  3. **예방 중심의 보안 (Shift-Left)**: 이미 발생한 사고(CVE)를 처리하는 수준을 넘어, 사고의 씨앗(CWE)을 설계 단계부터 뿌리 뽑기 위한 지식 체계가 요구되었다.

- **📢 섹션 요약 비유**: 수만 가지 사고의 개별 사례를 외우기보다, 사고가 일어나는 100가지 근본 이유를 정리한 '안전 백과사전'을 통해 사고를 미리 방지하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CWE의 계층적 구조 (CWE Hierarchy)

CWE는 추상화 수준에 따라 부모-자식 관계의 트리 구조를 가진다.

| 계층 (Level) | 명칭 | 설명 | 예시 |
|:---|:---|:---|:---|
| **Root (Top)** | **Pillar** | 가장 높은 수준의 추상화된 분류 | CWE-707 (Improper Neutralization) |
| **Middle** | **Class** | 기술적 특성에 따른 중간 분류 | CWE-74 (Injection) |
| **Middle** | **Base** | 구체적인 구현상의 문제점 | CWE-89 (SQL Injection) |
| **Bottom** | **Variant** | 특정 언어/환경에 종속된 세부 약점 | CWE-564 (Hibernate SQL Injection) |

---

### CWE - CVE - CAPEC 연계 아키텍처

보안 사고의 원인, 결과, 공격 방법을 연결하는 생태계를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      취약점 관리를 위한 표준 지식 체계 연계도                     │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │    [CWE (보안 약점)] ◀──────(Parent)───────┐                               │
 │    "근본적인 원인"                          │                               │
 │    (예: 버퍼 오버플로우)                     │                               │
 │           │                                 │                               │
 │           ▼ (Manifests as)                  │ (Categorizes)                 │
 │                                             │                               │
 │    [CVE (보안 취약점)] ──────▶ [CVSS (위험 점수)]                           │
 │    "실제 사고 사례"             "심각도 평가"                                │
 │    (예: Windows OS의                     (0.0 ~ 10.0)                       │
 │     CVE-2024-xxxx)                                                          │
 │           ▲                                                                 │
 │           │ (Exploited by)                                                  │
 │                                                                             │
 │    [CAPEC (공격 패턴)]                                                       │
 │    "해커의 공격 방법"                                                        │
 │    (예: 스택 스매싱 기법)                                                    │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 보안 생태계는 세 가지 축으로 돌아간다. 1) **CWE**는 "어떤 실수로 인해 약점이 생겼는가"라는 개발자 관점의 사전이다. 2) **CVE**는 그 약점이 실제 소프트웨어 제품(Windows, Chrome 등)에서 발현되어 보고된 고유 식별자다. 3) **CAPEC**은 그 약점을 해커가 어떤 기술로 요리(공격)하는지를 설명한다. 예를 들어, **CWE-89 (SQLi)**라는 약점이 **CVE-2023-1234**라는 특정 앱의 취약점으로 나타나고, 공격자는 **CAPEC-66 (SQL Injection)** 공격 패턴을 사용하여 침투한다. 이 세 가지를 연결함으로써 방어자는 원인 분석부터 공격 차단까지 일관된 전략을 세울 수 있다.

---

### 심층 동작 원리: CWE View (관점)

사용자의 목적에 따라 CWE를 다르게 그룹화하여 보여준다.
1. **CWE-699 (Software Development)**: 개발자가 코딩 중 참고하기 좋은 관점 (언어별, 라이브러리별).
2. **CWE-1000 (Research View)**: 학술적으로 취약점의 상관관계를 분석하기 위한 정밀 관점.
3. **CWE-1400 (Top 25)**: 현재 시점에서 가장 위험하고 빈번한 25가지 약점만 추린 실무용 관점.

- **📢 섹션 요약 비유**: 같은 백과사전이라도 요리사에게는 레시피 위주로(개발 관점), 의사에게는 병리학 위주로(연구 관점) 목차를 다시 짜서 보여주는 맞춤형 가이드북과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### CWE vs CVE vs CVSS 비교

| 항목 | CWE (Weakness) | CVE (Vulnerability) | CVSS (Scoring) |
|:---|:---|:---|:---|
| **질문** | "무엇이 문제의 뿌리인가?" | "어디서 사고가 났는가?" | "얼마나 위험한가?" |
| **관리 주체** | MITRE, Community | MITRE, CNA (승인 기관) | FIRST (포럼) |
| **ID 예시** | CWE-79 | CVE-2021-44228 | 10.0 (Critical) |
| **활용** | **예방**, 도구 개발, 교육 | **대응**, 패치, 정보 공유 | **우선순위 결정** |

---

### 융합 시너지: CWE 기반의 SAST 도구 고도화

정적 분석 도구(SAST)는 소스 코드를 스캔하여 발견한 문제에 CWE ID를 태깅한다.
- **가치**: 서로 다른 회사의 도구 A와 도구 B를 쓸 때, 리포트 양식은 달라도 동일한 CWE 번호를 통해 중복을 제거하고 탐지 범위를 비교할 수 있다.
- **정량 지표**: "우리 회사는 올해 CWE Top 25 중 20개를 탐지할 수 있는 수준으로 보안 역량을 높였다"와 같은 정량적 보고가 가능해진다.

- **📢 섹션 요약 비유**: 전 세계 모든 병원이 질병 분류 기호(CWE)를 통일함으로써, 어떤 병원에서 진찰받아도 정확한 병명과 치료법을 공유할 수 있는 국가 의료 전산망과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 전사 시큐어 코딩 가이드라인 수립

- **문제 상황**: 개발팀마다 보안 코딩 기준이 제각각이라 보안 심사 때마다 마찰이 발생함.
- **의사결정 (CWE 기반 표준화)**:
  1. **Baseline 선정**: 최신 'CWE Top 25'를 기반으로 우리 조직이 반드시 막아야 할 'Must-Fix' 약점 목록을 정의한다.
  2. **도구 커스터마이징**: 사내 SAST 도구의 룰셋을 CWE ID와 매핑하여, 개발자가 코드 수정 가이드를 CWE 공식 문서(예시 코드 포함)에서 직접 찾을 수 있게 링크한다.
  3. **교육 과정 개편**: 개발자 보안 교육 시 "CWE-20 (부적절한 입력값 검증)"과 같은 표준 용어를 사용하여 보안 마인드를 상향 평준화한다.
  4. **공급망 관리**: 외부 업체에 SW 개발 발주 시, "CWE Top 25 약점이 포함되지 않아야 함"을 계약서 상의 품질 요건으로 명시한다.

### 도입 체크리스트
- **ID 매핑**: 현재 사용 중인 보안 도구들이 최신 CWE 버전(현재 4.x 이상)을 지원하는가?
- **Prioritization**: 수백 개의 CWE 중 우리 서비스 성격(웹, 임베디드 등)에 맞는 핵심 약점 그룹을 선정했는가?
- **Knowledge Sharing**: 사내 Wiki에 주요 CWE별 'Best Practice'와 'Bad Practice' 코드 스니펫을 구축했는가?

### 안티패턴
- **번호만 나열**: 취약점 리포트에 CWE 번호만 적고, 그것이 우리 시스템의 비즈니스 로직에서 어떤 실제 위협이 되는지 설명하지 않는 경우.
- **오탐 방치**: CWE-20(입력 검증)은 매우 광범위하여 SAST 도구에서 오탐(False Positive)이 많이 발생한다. 이를 기계적으로 다 고치라고 강요하면 개발 생산성이 급격히 떨어진다.

- **📢 섹션 요약 비유**: 사전에 나온 모든 단어를 외우라고 강요하기보다, 일상 대화에 꼭 필요한 필수 단어(Top 25)부터 익히게 하는 것이 효율적인 학습법이라는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 보안 결함 밀도 감소 | 설계 및 구현 단계에서 CWE 기반 검토 수행 시 잠재 취약점 70% 사전 제거 |
| **정량적 (Cost)** | 도구 통합 비용 절감 | 표준 ID 사용으로 이기종 보안 도구 간 데이터 통합 및 비교 용이 |
| **정성적 (Standard)** | 전사 보안 거버넌스 확립 | 보안팀과 개발팀 간의 명확한 기술적 합의 기준 마련 |
| **정성적 (Education)** | 보안 전문가 양성 | CWE의 방대한 공격 사례와 예시 코드를 통한 실무 역량 강화 |

### 미래 전망
- **Hardware CWE (CWE for H/W)**: CPU 보안 버그(Spectre, Meltdown) 이후 하드웨어 설계 단계의 약점을 다루는 CWE 항목이 급격히 늘어나며 반도체 보안의 핵심이 될 것이다.
- **AI-Native CWE**: AI 모델의 취약점(Adversarial Attack 등)을 분류하는 'CWE for AI'가 표준화되어 지능형 시스템 보안의 근간이 될 것이다.

### 참고 표준
- **CWE Top 25 Most Dangerous Software Weaknesses**: 매년 발표되는 핵심 지표
- **ISO/IEC 29147**: 취약점 공개 프로세스 표준 (CWE 참조)
- **행정안전부/KISA 소프트웨어 개발 보안 가이드**: CWE를 국내 실정에 맞게 47개 항목으로 재구성

- **📢 섹션 요약 비유**: 지금까지는 땅에 떨어진 열매(취약점 사례)를 줍는 데 급급했다면, 이제는 뿌리와 줄기(보안 약점 사전)의 상태를 관리하여 처음부터 건강한 나무(소프트웨어)를 키우는 시대로 진화했습니다.
