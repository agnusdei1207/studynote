# 515. 쿠버네티스 (Kubernetes) 보안

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쿠버네티스 보안은 컨테이너 오케스트레이션 환경의 복잡성을 관리하기 위해 **4C 모델(Cloud, Cluster, Container, Code)**을 기반으로 계층적 방어를 구축하고, 제어 평면과 데이터 평면 전반에 걸쳐 선언적 보안 정책을 적용하는 체계다.
> 2. **가치**: **RBAC(권한 관리), Network Policy(네트워크 격리), PSA(파드 보안 승인)**를 통해 공격자의 횡적 이동(Lateral Movement)을 차단하고, API 서버에 대한 비정상적인 접근을 물리적으로 제한하여 클러스터 전체의 무결성을 유지한다.
> 3. **융합**: 클라우드 네이티브 보안 도구(CSPM, CWPP) 및 서비스 메시와 결합되어 운영되며, 'Admission Control'을 통한 정책 자동화로 "설정 오류에 의한 보안 사고"를 원천 방지한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 쿠버네티스 보안은 수천 개의 컨테이너가 유기적으로 움직이는 클러스터 환경을 보호하는 것이다. 단순히 서버 하나를 지키는 수준을 넘어, 컨테이너 간의 통신, 관리자의 명령(API), 그리고 민감한 설정 정보(Secret)를 입체적으로 방어해야 한다.

- **💡 비유**: 쿠버네티스 보안은 **"대규모 아파트 단지의 통합 보안 시스템"**과 같습니다. 단지 입구(Cloud)부터 동별 출입구(Cluster), 개별 가구의 현관문(Container), 그리고 집 안의 금고(Code)까지 층층이 보안 장치를 두는 것과 같습니다. 특히 관리사무소(API Server)의 열쇠를 누가 가졌는지(RBAC)가 가장 중요합니다.

- **등장 배경**:
  1. **오케스트레이션의 복잡성**: 수동으로 관리할 수 없는 방대한 자원들이 생기면서, 보안 설정 하나가 클러스터 전체를 무너뜨리는 위험이 커졌다.
  2. **공격 표면(Attack Surface)의 확대**: API 서버, etcd 저장소, Kubelet 등 해커가 노릴 수 있는 접점이 늘어났다.
  3. **내부망 신뢰의 위험성**: 일단 클러스터 내부로 들어온 해커가 네트워크 정책이 없는 틈을 타 다른 파드(Pod)의 데이터를 훔쳐가는 사고가 빈번해졌다.

- **📢 섹션 요약 비유**: 보이지 않는 거대한 성벽을 쌓는 대신, 성 안의 모든 방과 통로에 실시간으로 변하는 지능형 잠금장치(Policy)를 다는 현대적 요새화 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 쿠버네티스 보안의 3대 핵심 기둥

| 기술 요소 | 주요 역할 | 상세 메커니즘 | 비유 |
|:---|:---|:---|:---|
| **RBAC** | 권한 제어 | 누가(User/SA) 무엇을(Pod/Secret) 할 수 있는지(Verb) 정의 | 직급별 출입 카드 |
| **Network Policy** | 네트워크 격리 | 파드 간의 L3/L4 트래픽 허용/차단 규칙 정의 | 방 사이의 방화벽 |
| **PSA (Pod Sec. Adm.)** | 파드 실행 통제 | 보안 기준(Privileged, Baseline, Restricted)에 따른 파드 생성 제어 | 엄격한 입주 심사 |

---

### 2. 쿠버네티스 API 서버 보안 아키텍처

모든 명령의 중심인 API 서버를 보호하는 3단계 필터를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Kubernetes API Request Lifecycle 보안 구조                  │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [명령 요청] ──▶ [1. Authentication] ──▶ [2. Authorization] ──▶ [3. Admission] ──▶ [etcd] │
 │ (kubectl)       (인증: 누구냐?)          (인가: 권한 있나?)      (승인: 정책 맞나?)   (저장) │
 │                                                                             │
 │  ┌────────────┐      ┌────────────┐      ┌────────────┐      ┌────────────┐ │
 │  │  User /    │      │  Cert /    │      │  RBAC /    │      │  PSA /     │ │
 │  │  Service   │      │  Token     │      │  ABAC      │      │  Validating│ │
 │  │  Account   │      │  Verify    │      │  Check     │      │  Webhooks  │ │
 │  └────────────┘      └────────────┘      └────────────┘      └────────────┘ │
 │                                                                             │
 │  ※ 핵심: 3단계를 모두 통과해야만 실제 클러스터 자원(Pod 등)이 생성/변경됨             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 쿠버네티스의 모든 활동은 API 서버를 통한다. 1) **인증** 단계에서 유효한 인증서나 토큰을 가졌는지 확인한다. 2) **인가(RBAC)** 단계에서는 "이 사용자가 파드를 삭제할 권한이 있는가?"를 따진다. 3) 마지막 **Admission Control** 단계가 가장 강력한데, 권한이 있는 사람이라도 "루트 권한으로 실행되는 위험한 파드"를 만들려고 하면 정책 위반으로 거절한다. 이 3중 필터를 통해 휴먼 에러와 악의적 명령을 동시에 차단한다.

---

### 심층 동작 원리: 네트워크 정책 (Network Policy)

클러스터 내부의 '제로 트러스트'를 실현하는 기술이다.
1. **Default Deny**: 기본적으로 모든 파드 간 통신을 차단한다.
2. **Whitelist**: "주문 파드는 오직 결제 파드와만 대화할 수 있다"는 식으로 명시적인 통로만 열어준다.
3. **가치**: 웹 서버 파드가 해킹당해도, 네트워크 정책이 있다면 해커는 내부망의 DB 파드로 접근할 수 없어 2차 피해를 막는다.

- **📢 섹션 요약 비유**: 아파트 복도에서 옆집 사람을 만날 수 없게 칸막이를 치고, 오직 인터폰(정해진 포트)으로만 대화하게 만드는 철저한 격리 정책과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### PSP (Pod Security Policy) vs PSA (Pod Security Admission)

쿠버네티스 버전 업데이트에 따른 보안 기술의 진화를 이해해야 한다.

| 비교 항목 | 구 방식 (PSP) | 신 방식 (PSA) |
|:---|:---|:---|
| **복잡도** | 매우 높음 (설정이 까다로움) | **낮음 (표준화된 프로파일 제공)** |
| **적용 단위** | 개별 리소스 | **네임스페이스 (Namespace) 단위** |
| **운영 방식** | 복잡한 RBAC 연동 필요 | 레이블(Label) 설정만으로 즉시 적용 |
| **상태** | v1.25에서 완전히 삭제됨 | **v1.23 이후 표준으로 정착** |
| **비유** | 직접 설계하는 특수 보안 장치 | 기성품 보안 등급 (상/중/하) |

---

### 기술 융합: eBPF와 쿠버네티스 가시성

네트워크 정책의 한계를 넘어서는 최신 기술 융합이다.
- **Cilium / Falco**: 커널 레벨에서 작동하는 eBPF 기술을 사용하여, 네트워크 패킷뿐만 아니라 컨테이너 내부에서 실행되는 '명령어'까지 실시간 감시한다.
- **가치**: "누가 파일을 읽었는가", "누가 비정상적인 쉘을 띄웠는가"를 클러스터 수준에서 완벽하게 관측(Observability)한다.

- **📢 섹션 요약 비유**: 성벽(Network Policy)을 쌓는 것을 넘어, 성 안의 모든 사람의 혈관 속에 나노 로봇(eBPF)을 넣어 건강 상태를 실시간 체크하는 초정밀 방역 시스템입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 내부자 위협에 대비한 'Secret' 관리 강화

- **문제 상황**: 클러스터 관리자가 `kubectl get secrets` 명령으로 모든 서비스의 DB 비밀번호를 평문(Base64)으로 훔쳐볼 수 있는 리스크 발견.
- **의사결정 (K8s 보안 고도화 전략)**:
  1. **RBAC 최소화**: 개발자에게 `Cluster-admin` 권한을 주지 않고, 자기 네임스페이스 내에서만 활동 가능한 제한적 Role을 부여한다.
  2. **etcd 암호화**: 쿠버네티스 저장소인 etcd 자체를 KMS(Key Management Service)와 연동하여 암호화(Encryption at Rest)한다.
  3. **외부 Vault 연동**: 비밀번호를 K8s 내부에 두지 않고 **HashiCorp Vault** 등 외부 전문 솔루션에서 런타임에만 주입받도록 아키텍처를 변경한다.
  4. **Audit Log 감시**: "누가 Secret을 조회했는가"를 24시간 로깅하고 이상 징후 시 보안 팀에 즉각 알람을 보낸다.

### 도입 체크리스트
- **Pod Security Levels**: 모든 네임스페이스에 `privileged`, `baseline`, `restricted` 중 적절한 보안 라벨이 붙어 있는가?
- **Root Usage**: `securityContext` 설정에서 `runAsNonRoot: true`가 강제되고 있는가?
- **Resource Quota**: 특정 파드가 자원을 독점하여 클러스터를 마비시키지 않도록 쿼터(Quota)가 설정되었는가?

### 안티패턴
- **Default 네임스페이스 공용 사용**: 모든 앱을 한곳에 몰아넣는 행위. 격리(Isolation) 효과가 사라진다.
- **API 서버 외부 노출**: 공인 IP로 API 서버를 열어두는 행위. (반드시 VPN이나 전용망을 통해서만 접근해야 함)

- **📢 섹션 요약 비유**: 아무리 튼튼한 금고(etcd)를 샀어도, 금고 비밀번호(관리자 권한)를 모든 직원에게 알려준다면 이미 보안은 실패한 것입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 횡적 이동 공격 90% 차단 | 네트워크 정책을 통한 공격 경로의 물리적 폐쇄 |
| **정량적 (Ops)** | 보안 설정 오류 70% 감소 | PSA와 Admission Controller를 통한 보안 표준 자동 강제 |
| **정성적 (Compliance)** | 클라우드 보안 인증 통과 | 금융권 및 공공 클라우드 가이드라인의 복잡한 요건 충족 |
| **정성적 (Resilience)** | 클러스터 생존성 향상 | 일부 파드 오염 시에도 전체 시스템으로의 전이 차단 및 자가 치유 |

### 미래 전망
- **Serverless Kubernetes (Fargate)**: 노드 관리 자체를 제거하여, 인프라 보안 책임을 CSP에 완전히 넘기는 형태가 대세가 될 것이다.
- **AI-Driven Admission Control**: AI가 실시간 위협 트렌드를 학습하여, 위험해 보이는 파드 생성을 사전에 차단하는 지능형 정책 엔진이 도입될 것이다.

### 참고 표준
- **CIS Kubernetes Benchmark**: 업계 최고의 보안 설정 체크리스트
- **NIST SP 800-190**: Application Container Security Guide
- **NSA/CISA Kubernetes Hardening Guidance**: 미 정부 발행 보안 지침

- **📢 섹션 요약 비유**: 쿠버네티스 보안은 "설정의 미학"입니다. 수만 개의 나사를 조이는 복잡한 과정 속에서 단 하나의 나사(보안 정책)도 헐겁지 않게 조이는 꼼꼼함이 위대한 클라우드 성벽을 만듭니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 우리 반 친구들이 다 같이 쓰는 **"거대한 로봇(쿠버네티스)"**이 있는데, 이 로봇은 아주 복잡해서 조종하는 규칙이 필요해요.
2. 아무나 로봇의 중요 부품을 만지지 못하게 **비밀 신분증(RBAC)**을 확인하고, 로봇 안의 방들을 **튼튼한 벽(Network Policy)**으로 나누는 거예요.
3. 이렇게 꼼꼼하게 규칙을 정하면, 나쁜 스파이 로봇이 들어와도 우리 로봇을 망가뜨리지 못하게 지킬 수 있답니다!
