# 496. SBOM (Software Bill of Materials) 포맷 - SPDX, CycloneDX

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SBOM (소프트웨어 자재 명세서)은 소프트웨어를 구성하는 모든 오픈소스 라이브러리, 직접 개발한 모듈, 그리고 의존 관계 정보를 기계가 읽을 수 있는 형식으로 기록한 **디지털 부품 상세 리스트**다.
> 2. **가치**: 취약점(CVE) 발생 시 영향도를 즉각 파악하여 대응 시간을 최소화하고, 오픈소스 라이선스 리스크를 예방하며, 기업 간 소프트웨어 거래 시 투명성과 신뢰성을 증명하는 '디지털 유통 증명서' 역할을 한다.
> 3. **융합**: SPDX (ISO 표준, 상세함)와 CycloneDX (보안 특화, 경량화)가 글로벌 표준으로 경쟁 및 공존하고 있으며, 현대 아키텍처에서는 CI/CD 파이프라인에서 SCA 도구를 통해 자동으로 생성·검증되는 '보안 애즈 코드 (Security as Code)'의 핵심 산출물이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: SBOM은 소프트웨어의 "성분표"다. 우리가 가공식품을 살 때 뒷면의 원재료 함량을 확인하듯, 소프트웨어 패키지 내부에 어떤 오픈소스가 들어있는지, 그 버전은 무엇인지, 어떤 라이선스 조건이 붙어 있는지를 표준화된 포맷(JSON, XML, Tag-value 등)으로 나열한 문서다.

- **💡 비유**: SBOM은 **"항공기나 자동차를 만들 때 들어가는 수만 개의 부품(나사 하나까지)의 제조사, 규격, 제조 일자를 기록한 정밀 설계 명세서"**와 같습니다. 부품 결함(취약점)으로 인한 리콜 상황이 발생했을 때, 어떤 차량이 위험한지 즉시 골라낼 수 있게 해줍니다.

- **등장 배경**:
  1. **가시성(Visibility)의 부재**: 현대 소프트웨어의 90% 이상이 외부 컴포넌트로 구성되면서, 정작 우리 서비스에 어떤 위험한 라이브러리가 숨어있는지 파악하기 불가능해졌다.
  2. **국가 차원의 보안 규제**: 미국 백악관 행정명령(EO 14028) 등에서 정부에 납품되는 모든 소프트웨어에 대해 SBOM 제출을 의무화하면서 기술적 선택이 아닌 법적 필수 사항이 되었다.
  3. **대규모 사고의 교훈**: Log4j(Log4Shell) 사태 당시, SBOM이 없던 기업들은 수천 개의 서버를 일일이 수동 점검하느라 대응에 수주가 걸리는 뼈아픈 경험을 했다.

- **📢 섹션 요약 비유**: 사고가 난 뒤에 가방을 다 쏟아서 내용물을 확인하는 것이 아니라, 가방 겉면에 붙은 상세 목록(SBOM)만 보고도 위험 물품이 있는지 바로 알아내는 체계적인 관리법과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SBOM의 2대 표준 포맷 비교: SPDX vs CycloneDX

현재 시장을 양분하고 있는 두 포맷은 탄생 배경과 목적이 다르다.

| 비교 항목 | SPDX (Software Package Data Exchange) | CycloneDX |
|:---|:---|:---|
| **주도 기관** | Linux Foundation | OWASP (Open Web App Security Project) |
| **핵심 목적** | **라이선스 준수** 및 IP 관리 (ISO/IEC 5962) | **보안 분석** 및 취약점 관리 (SCA 특화) |
| **특징** | 매우 상세하고 엄격함 (방대한 데이터) | 가볍고 자동화에 유리함 (보안 중심) |
| **주요 활용** | 엔터프라이즈 SW 배포, 법적 증빙 | DevSecOps 파이프라인, 취약점 실시간 스캔 |
| **포맷 지원** | Tag-value, JSON, YAML, XML | JSON, XML, Protobuf |

---

### SBOM 생성 및 유통 아키텍처 (Supply Chain Trust)

SBOM이 소프트웨어 공급망의 각 단계에서 어떻게 생성되고 활용되는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      SBOM 기반 소프트웨어 공급망 신뢰 아키텍처                    │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [공급자 (Vendor)]           [유통/검증 (CI/CD)]           [수요자 (User/Gov)] │
 │        │                          │                          │              │
 │  1. 소스 개발 및 ──────▶ 2. SCA 도구 구동  ──────▶ 3. 배포 및 SBOM 전달        │
 │     외부 라이브러리         (SPDX/CycloneDX)          (Digital Signature)   │
 │     도입 (In-bound)         (SBOM 생성)               │                     │
 │        │                          │                          ▼              │
 │        ▼                          ▼                ┌──────────────────┐     │
 │  ┌────────────┐           ┌──────────────────┐     │  지속적 모니터링   │     │
 │  │  Build     │           │  무결성 검증      │     │  (VEX 연동)      │     │
 │  └────────────┘           └──────────────────┘     └──────────────────┘     │
 │                                   │                          │              │
 │                                   └───────(Trust Chain)──────┘              │
 │                                                                             │
 │  ※ VEX (Vulnerability Exploitability eXchange): 취약점의 실제 영향도 정보 공유    │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SBOM 아키텍처의 핵심은 **자동화**와 **서명**이다. 1) 개발 과정에서 외부 라이브러리가 유입되면, 2) 빌드 파이프라인에 통합된 SCA 도구가 이를 감지하여 표준 포맷의 SBOM 파일을 생성한다. 이때 생성된 SBOM은 위변조 방지를 위해 공급자의 **디지털 서명**이 포함되어야 한다. 3) 최종 사용자는 전달받은 SBOM을 자신의 취약점 관리 시스템에 등록한다. 이후 새로운 CVE가 발표되면, 사용자는 서버를 뒤질 필요 없이 등록된 SBOM 데이터베이스만 검색하여 조치 대상을 즉시 확정한다.

---

### 심층 동작 원리: SBOM의 필수 구성 요소 (NTIA 권고)

미국 NTIA(국립통신정보청)는 유효한 SBOM을 위해 최소 7가지 요소를 정의한다.
1. **Supplier Name**: 소프트웨어를 만든 공급자 명칭
2. **Component Name**: 라이브러리나 모듈의 정확한 명칭
3. **Version**: 해당 부품의 버전 정보
4. **Unique Identifier**: 다른 부품과 구별되는 고유 값 (CPE, PURL 등)
5. **Dependency Relationship**: 부품 간의 상하 관계 (A가 B를 포함함)
6. **Author of SBOM Data**: 이 명세서를 작성한 주체
7. **Timestamp**: 명세서 작성 일시

- **📢 섹션 요약 비유**: 택배 송장에 보낸 사람, 받는 사람, 물건 이름, 수량뿐만 아니라 상자 안에 든 작은 물건들의 상세 리스트까지 적어두어 투명성을 확보하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SBOM vs VEX (Vulnerability Exploitability eXchange)

SBOM이 "무엇이 들어있는가"를 말한다면, VEX는 "그게 진짜 위험한가"를 말한다.

| 항목 | SBOM (성분표) | VEX (상태 정보) |
|:---|:---|:---|
| **내용** | 라이브러리 A 버전 1.0 포함됨 | 라이브러리 A의 취약점은 우리 구조상 영향 없음 |
| **가치** | 잠재적 위협 식별 | **오탐(False Positive) 제거**, 패치 우선순위 결정 |
| **관계** | "이 재료가 들어갔어요" | "그 재료는 익혀 먹어서 안전해요" |

---

### 기술 융합: 클라우드 네이티브와 SBOM

컨테이너 환경(Docker, K8s)에서의 SBOM 활용도가 급증하고 있다.
- **Image Scan**: Docker 이미지를 빌드할 때 이미지 레이어별 SBOM을 생성하여, 컨테이너 내부에 숨겨진 OS 패키지 취약점까지 관리한다.
- **Admission Control**: 쿠버네티스 배포 시, 신뢰할 수 있는 서명된 SBOM이 없는 컨테이너는 실행을 거부하는 정책(Opa 등)을 적용한다.

- **📢 섹션 요약 비유**: 요리 재료뿐만 아니라 요리를 담은 그릇(컨테이너)의 유해 성분까지 모두 기록하여 완벽한 위생 상태를 보장하는 스마트 주방과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 기업 납품을 위한 보안 신뢰성 증명

- **문제 상황**: 국내 중소 SW 기업 C사가 미국 대기업에 솔루션을 수출하려는데, 상대방 측에서 "보안 신뢰성을 위해 전체 제품의 SBOM을 CycloneDX 포맷으로 제출하라"고 요구함.
- **의사결정 (SBOM 구축 프로세스)**:
  1. **도구 도입**: CI/CD 파이프라인(GitLab CI)에 오픈소스 SBOM 생성 도구(Syft, Trivy 등)를 통합한다.
  2. **포맷 결정**: 상대방의 요구에 맞춰 보안 특화 기능이 강한 **CycloneDX JSON** 포맷을 표준으로 정한다.
  3. **의존성 정리**: 개발자가 개인적으로 다운로드하여 쓰고 있는 라이브러리들을 모두 공식 패키지 매니저 관리 하로 옮겨 '그림자 의존성(Shadow Dependency)'을 제거한다.
  4. **지속적 관리**: 릴리스할 때마다 자동으로 SBOM을 생성하고, 여기에 회사 명의의 전자 서명을 첨부하여 고객사에 API 형태로 공유한다.

### 도입 체크리스트
- **Completeness**: 간접 의존성(Transitive Dependency)이 누락 없이 포함되는가?
- **Format Validity**: 생성된 파일이 SPDX나 CycloneDX 표준 스키마 검증(Validation)을 통과하는가?
- **Integration**: 취약점 스캔 도구가 생성된 SBOM을 직접 읽어 들일 수 있는가?

### 안티패턴
- **수동 작성**: 엑셀에 사람이 손으로 라이브러리 이름을 적는 행위. 소프트웨어 변경 속도를 따라갈 수 없어 며칠만 지나도 '가짜 문서'가 된다.
- **빌드 시점 누락**: 배포 직전에만 SBOM을 만드는 행위. 개발 중간에 이미 취약점이 유입될 기회를 놓치게 된다.

- **📢 섹션 요약 비유**: 가계부를 한 달에 한 번 기억을 더듬어 쓰는 것이 아니라, 결제할 때마다 영수증이 자동으로 기록되어 완벽한 장부가 되게 하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 취약점 탐지 시간 95% 단축 | 수천 대 서버 스캔 대신 SBOM DB 검색으로 단 수 초 만에 영향도 파악 |
| **정량적 (Compliance)** | 해외 수출 장벽 제거 | 미 행정명령 등 글로벌 보안 규제 요구사항 완벽 충족 |
| **정성적 (Standard)** | 소프트웨어 투명성 향상 | "무엇으로 만들었는지 안다"는 사실만으로 개발 품질 및 신뢰도 상승 |
| **정성적 (Legal)** | 라이선스 리스크 관리 | 상용 소프트웨어 개발 시 GPL 등 금지 라이선스 유입 차단 |

### 미래 전망
- **Dynamic SBOM**: 애플리케이션 실행 중에 동적으로 로드되는 모듈까지 실시간으로 반영하는 '런타임 SBOM' 기술이 발전할 것이다.
- **AI-Managed Supply Chain**: AI가 전 세계의 SBOM 데이터를 통합 분석하여, 특정 국가나 업체의 라이브러리에 집중된 리스크를 예측하고 대체 라이브러리를 추천해주는 서비스가 등장할 것이다.

### 참고 표준
- **ISO/IEC 5962**: SPDX 국제 표준
- **NTIA Minimum Elements for SBOM**: SBOM 필수 구성 요소 가이드라인
- **CycloneDX Specification**: OWASP 공식 명세서

- **📢 섹션 요약 비유**: 지금까지는 내 가방 속 물건만 잘 챙기면 됐지만, 이제는 내가 산 물건이 어디서 왔고 누가 만들었는지까지 증명해야 하는 '투명한 디지털 신뢰 시대'의 신분증이 바로 SBOM입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **SCA (Software Composition Analysis)**: SBOM을 생성하고 분석하는 핵심 기술
- **VEX (Vulnerability Exploitability eXchange)**: SBOM의 정보를 보완하는 취약점 영향도 전송 표준
- **Supply Chain Attack**: SBOM 도입의 가장 강력한 동기가 된 공격 유형
- **CPE / PURL**: SBOM 내에서 부품을 식별하는 표준 명칭 체계
