# 484. Identification and Authentication Failures (인증 및 세션 관리 실패)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Identification and Authentication Failures (인증 및 세션 관리 실패)는 사용자의 신원을 확인하는 **인증(Authentication) 과정이나, 한 번 인증된 상태를 유지하는 세션(Session) 관리 메커니즘이 취약**하여 공격자가 타인의 계정을 도용하거나 세션을 탈취할 수 있는 상태를 의미한다.
> 2. **가치**: 2021년 OWASP Top 10에서 7위에 위치하며, 무차별 대입 공격(Brute Force), 취약한 비밀번호 정책, 세션 고정(Session Fixation), 자동화된 크리덴셜 스터핑(Credential Stuffing) 등 계정 탈취의 직접적인 원인이 되는 치명적 취약점이다.
> 3. **융합**: MFA (Multi-Factor Authentication), FIDO (Fast IDentity Online), OAuth 2.0 등 현대적 인증 표준과 결합되며, 단순 비밀번호를 넘어 '무엇을 알고 있는가(지식)', '무엇을 가지고 있는가(소유)', '무엇인가(생체)'의 다중 요소 결합이 방어의 핵심이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 과거 'Broken Authentication'으로 불리던 이 항목은 2021년 'Identification and Authentication Failures'로 명칭이 변경되면서, 신원 식별(Identification) 단계의 중요성까지 포괄하게 되었다. 사용자가 누구인지 확인하는 문(인증)과, 문을 통과한 후 들고 있는 통행증(세션)이 얼마나 안전한지를 다룬다.

- **💡 비유**: Identification and Authentication Failures는 **"호텔 로비에서 이름만 대면 방 키를 내주거나(취약한 인증), 방 키가 너무 단순해서 누구나 복사할 수 있고, 심지어 체크아웃한 뒤에도 그 키로 계속 방에 들어갈 수 있는 상황(세션 관리 실패)"**와 같습니다.

- **등장 배경**:
  1. **크리덴셜 스터핑 공격의 유행**: 다른 사이트에서 유출된 아이디/비번 조합을 자동화 도구로 무차별 대입하는 공격이 기하급수적으로 늘어났다.
  2. **모바일 및 API 환경 확산**: 웹 브라우저의 쿠키 기반 세션 외에도 JWT (JSON Web Token) 등 다양한 토큰 기반 인증 방식이 쓰이면서 구현 오류가 빈번해졌다.
  3. **비밀번호 관리의 한계**: 사용자들이 복잡한 비밀번호를 기억하기 어려워하면서, 쉬운 비밀번호 사용 및 돌려막기 관행이 지속되고 있다.

- **📢 섹션 요약 비유**: 성문 입구에서 얼굴 확인도 대충 하고, 한 번 들어온 사람에게 준 출입증이 유효기간도 없이 영원히 쓸 수 있다면 그 성은 곧 도둑들의 놀이터가 될 것이라는 원리와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 인증 및 세션 실패의 주요 유형

| 유형 | 설명 | 공격 및 사고 사례 | 비유 |
|:---|:---|:---|:---|
| **취약한 비밀번호** | 단순한 비번 허용, 초기 비번 미변경 | `123456`, `password` 등으로 관리자 계정 탈취 | 누구나 아는 암호 |
| **무차별 대입 (Brute Force)** | 로그인 시도 횟수 제한 없음 | 수만 번의 시도 끝에 비밀번호 무작위 적중 | 열쇠 수천 개 돌려보기 |
| **세션 노출 및 고정** | URL에 세션 ID 포함, 로그인 전후 세션 동일 | 공공 PC에서 세션 ID를 가로채 타인 계정 접속 | 흘린 통행증 줍기 |
| **불충분한 MFA** | 다중 인증 미적용 또는 우회 가능 | 아이디/비번만으로 모든 중요 기능 수행 | 카드 없이 비밀번호만 입력 |
| **취약한 세션 만료** | 로그아웃 후에도 세션이 서버에 남음 | 사용자가 나간 뒤에도 세션 토큰으로 재접속 성공 | 체크아웃 후에도 열리는 방 |

---

### 안전한 인증 및 세션 관리 아키텍처 (MFA & JWT)

다중 요소 인증과 무상태(Stateless) 토큰의 안전한 관리 체계를 시각화하면 다음과 같다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      현대적 다중 인증(MFA) 및 세션 보호 아키텍처                  │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [사용자] ──(1.ID/PW)──▶ [인증 서버] ◀──(2.OTP/Push)──▶ [사용자 기기]           │
 │    │                       │                               │                │
 │    │ ◀──(3.Access Token)── ┘                               │                │
 │    │                                                       │                │
 │    ▼ [리소스 서버 (API)]                                     │                │
 │  ┌───────────────────────────────────────────────────────┐ │                │
 │  │ 4. 요청 (with JWT Bearer Token)                       │ │                │
 │  │    - signature 검증 필수                               │ │                │
 │  │    - exp (만료시간) 체크                                │ ┘                │
 │  │    - jti (재사용 방지) 확인                              │                  │
 │  └───────────────────────────────────────────────────────┘                  │
 │                                                                             │
 │  [핵심 보안 메커니즘]                                                         │
 │  - Session ID: HttpOnly, Secure, SameSite 속성 필수 적용                      │
 │  - Password: Argon2, bcrypt 등 강력한 알고리즘으로 저장                         │
 │  - Brute Force: 계정 잠금(Lockout) 또는 CAPTCHA 적용                          │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 안전한 인증은 1단계(ID/PW)를 넘어 2단계(MFA)를 필수화한다. 사용자의 스마트폰으로 전송된 OTP나 푸시 알림을 승인해야만 최종 인증이 완료된다. 인증 완료 후 발행되는 **세션 토큰(JWT 등)**은 기밀성 보장을 위해 HTTPS(Secure 속성)를 통해서만 전송되어야 하며, 자바스크립트로 접근할 수 없도록 **HttpOnly** 속성이 부여되어야 한다. 또한 토큰 내부에는 짧은 만료 시간(`exp`)과 고유 식별자(`jti`)를 포함하여 탈취된 토큰의 재사용 및 피해 범위를 최소화한다.

---

### 심층 동작 원리: 크리덴셜 스터핑 (Credential Stuffing) 방어

공격자가 유출된 계정 정보를 대량으로 자동 입력할 때 이를 차단하는 로직이다.
1. **속도 제한 (Rate Limiting)**: 단일 IP 또는 계정에 대해 초당 로그인 시도 횟수를 엄격히 제한한다.
2. **평판 기반 차단 (IP Reputation)**: 이미 알려진 해커의 IP나 Tor 네트워크로부터의 접근을 차단한다.
3. **행위 분석 (Behavioral Analysis)**: 인간이 아닌 기계적인 입력 패턴(일정한 간격, 비정상적인 브라우저 환경 등)을 감지하여 차단하거나 CAPTCHA를 띄운다.

- **📢 섹션 요약 비유**: 현관문 앞에 카메라를 달고, 짧은 시간에 수백 번 열쇠를 넣어보는 사람이 있으면 경찰에 신고하고 입구를 봉쇄하는 지능형 방범 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 지식(Knowledge) vs 소유(Possession) vs 존재(Inherence) 기반 인증

| 요소 유형 | 설명 | 예시 | 강점 및 약점 |
|:---|:---|:---|:---|
| **지식 (Something you know)** | 머릿속에 기억하는 정보 | 비밀번호, PIN, 패턴 | 기억하기 쉬움 / 유출 및 망각 위험 |
| **소유 (Something you have)** | 물리적으로 소유한 물건 | 스마트폰(OTP), 하드웨어 키, SMS | 도용 어려움 / 분실 시 인증 불가 |
| **존재 (Something you are)** | 신체적 특징 (생체 정보) | 지문, 홍채, 안면 인식, 목소리 | 유일성, 편의성 / 변경 불가(유출 시 치명적) |

---

### 현대적 기술 융합: FIDO (Fast IDentity Online)와 패스워드리스

비밀번호의 한계를 극복하기 위해 생체 인식과 공개키 암호화(PKI)를 결합한 표준이다.
- **동작**: 서버에는 비밀번호 대신 공개키만 저장하고, 사용자는 자신의 기기(스마트폰 등)에서 지문으로 개인키를 풀어 서명함으로써 인증한다.
- **가치**: 서버가 해킹당해도 비밀번호가 없으므로 유출될 정보가 없으며, 피싱 공격에 완벽히 면역된다.

- **📢 섹션 요약 비유**: 자물쇠와 열쇠를 주고받는 대신, 내 지문으로만 열리는 금고를 내가 들고 다니면서 "이게 제 금고 맞죠?"라고 확인만 시켜주는 가장 안전한 인증법과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 사내 시스템의 세션 하이재킹 사고 대응

- **문제 상황**: 직원이 사내 카페 Wi-Fi를 쓰던 중, 암호화되지 않은 세션 쿠키를 공격자에게 탈취당해 내부 인사 시스템 권한을 빼앗김.
- **의사결정 (세션 보안 강화)**:
  1. **쿠키 속성 강제**: 모든 세션 쿠키에 `Secure`(HTTPS 전용)와 `HttpOnly`(XSS 방지) 속성을 즉시 적용한다.
  2. **세션 바인딩**: 세션 생성 시 사용자의 IP와 User-Agent 정보를 결합하여, 중간에 기기나 장소가 바뀌면 세션을 즉시 무효화한다.
  3. **로그아웃 고도화**: 클라이언트 단의 쿠키 삭제뿐만 아니라, 서버 측 세션 저장소(Redis 등)에서도 세션 데이터를 확실히 삭제하도록 로직을 수정한다.
  4. **동시 접속 제어**: 동일 계정으로 여러 곳에서 중복 로그인하는 것을 제한하거나 알림을 보낸다.

### 도입 체크리스트
- **MFA Implementation**: 모든 관리자 및 외부 노출 시스템에 MFA가 적용되어 있는가?
- **Password Policy**: 길이(10자 이상), 복잡도, 사전 단어 금지 정책이 강제되고 있는가?
- **Session Timeout**: 일정 시간 활동이 없으면 자동으로 세션이 종료되는가? (Idle Timeout 적용)

### 안티패턴
- **비밀번호 평문 저장**: "우리 DB는 안전해"라고 믿고 암호화 없이 비번을 저장하는 행위. (사고 발생 시 법적 파산 가능성)
- **URL에 세션 ID 노출**: `.../mypage?sid=abc123...` 식의 설계. 브라우저 기록이나 레퍼러(Referer) 로그를 통해 세션이 100% 유출된다.

- **📢 섹션 요약 비유**: "우리 집은 골목 안쪽에 있어서 도둑이 못 찾을 거야"라고 안심하며 대문을 열어두는 근거 없는 자신감이 가장 위험하다는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 계정 탈취율 90% 이상 감소 | MFA 및 강력한 비번 정책 적용 시 자동화 공격의 대부분 차단 |
| **정량적 (Compliance)** | 법적 책임 경감 | 개인정보보호법 상의 인증 보호 조치 의무를 충실히 이행 |
| **정성적 (Trust)** | 서비스 신뢰도 향상 | "내 계정이 안전하다"는 믿음을 고객에게 주어 이탈 방지 |
| **정성적 (Standard)** | 글로벌 인증 통과 | ISMS-P, ISO 27001 등 보안 인증의 가장 핵심 요건 달성 |

### 미래 전망
- **Passwordless의 보편화**: 비밀번호가 완전히 사라지고 Passkey와 같은 기기 기반 인증이 표준이 될 것이다.
- **Continuous Authentication**: 로그인 시점뿐만 아니라, 사용 중의 타이핑 습관, 마우스 움직임 등을 AI가 계속 분석하여 본인 여부를 실시간으로 검증하는 상시 인증 시스템이 도입될 것이다.

### 참고 표준
- **NIST SP 800-63**: Digital Identity Guidelines (인증 가이드라인의 바이블)
- **FIDO2 / WebAuthn**: 웹 표준 인증 프로토콜
- **OAuth 2.0 / OpenID Connect**: 위임 권한 및 인증 표준

- **📢 섹션 요약 비유**: 과거에는 성문을 한 번만 잘 통과하면 됐지만, 미래에는 성 안을 걸어 다니는 걸음걸이(행위 분석)만으로도 본인인지 확인하는 물 흐르듯 자연스러운 보안 시대가 올 것입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 게임에 로그인할 때 **"나만 아는 비밀번호"를 입력하는데, 이게 너무 쉬우면 나쁜 사람이 내 아이템을 다 훔쳐 갈 수 있어요.**
2. 그래서 비밀번호 말고도 내 스마트폰에 온 숫자(OTP)를 한 번 더 확인하는 이중 잠금장치를 하는 게 좋아요.
3. 한 번 들어갔다고 해서 문을 계속 열어두지 말고, 컴퓨터를 끌 때는 "안녕히 계세요" 하고 문을 꽉 잠그고(로그아웃) 나오는 습관이 중요하답니다!
