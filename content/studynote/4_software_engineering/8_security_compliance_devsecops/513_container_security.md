# 513. 컨테이너 보안: 이미지 스캔, Non-root, 샌드박스

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컨테이너 보안은 가상화된 애플리케이션 실행 환경의 무결성을 보장하기 위해 **빌드(이미지 스캔), 실행(권한 최소화), 격리(커널 샌드박스)**의 3단계 계층적 방어 체계를 구축하는 기술이다.
> 2. **가치**: 컨테이너가 호스트 운영체제의 커널을 공유한다는 구조적 취약점을 극복하기 위해 **Root 권한 실행을 엄격히 금지**하고, 네임스페이스(Namespace) 기반의 강력한 격리를 통해 침해 사고 시 피해가 호스트나 다른 컨테이너로 전이되는 것을 방지한다.
> 3. **융합**: CI/CD 파이프라인의 이미지 취약점 자동 스캔 기술과 결합되며, 최근에는 gVisor, Kata Containers 등 고립성을 강화한 차세대 샌드박스 기술과 융합되어 보안성을 극대화한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 컨테이너 보안은 도커(Docker)나 쿠버네티스 환경에서 앱이 돌아가는 '그릇'을 안전하게 만드는 것이다. 가상 머신(VM)보다 가볍지만 하이퍼바이저 수준의 격리가 없기 때문에, 더 촘촘한 소프트웨어적 방어막이 필요하다.

- **💡 비유**: 컨테이너 보안은 **"아파트 단지의 가구별 보안"**과 같습니다. 1) 택배(이미지) 안에 위험물이 없는지 미리 검사하고, 2) 집주인이 아닌 외부인(Non-root)에게는 열쇠를 주지 않으며, 3) 옆집과 벽(Namespace)을 두껍게 만들어 소음이나 화재(공격)가 번지지 않게 하는 것과 같습니다.

- **등장 배경**:
  1. **컨테이너의 급격한 확산**: MSA 전환으로 컨테이너 수가 폭증하면서 관리되지 않은 이미지에서 대규모 취약점이 발견되기 시작했다.
  2. **커널 공유의 취약성**: 컨테이너 하나가 뚫리면 공유하는 호스트 커널을 통해 서버 전체가 장악되는 '컨테이너 탈출(Container Breakout)' 공격이 등장했다.
  3. **공급망 위협**: 공개된 컨테이너 허브(Docker Hub 등)에서 내려받은 이미지가 악성코드에 오염된 사례가 늘어났다.

- **📢 섹션 요약 비유**: 가벼운 텐트(컨테이너)를 치고 야영을 할 때, 텐트 입구(권한)를 잘 단속하고 주변에 울타리(샌드박스)를 쳐서 야생동물(해커)로부터 자신을 보호하는 지혜입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 컨테이너 보안의 3대 핵심 메커니즘

| 보안 영역 | 주요 기술 및 원칙 | 상세 내용 | 비유 |
|:---|:---|:---|:---|
| **이미지 보안** | **이미지 스캐닝 (Scanning)** | 취약한 패키지(CVE) 및 비밀번호 포함 여부 검사 | 공항 수하물 검색 |
| **실행 권한 보안** | **Non-root Execution** | 컨테이너 내부 프로세스를 일반 사용자로 실행 | 관리자 비밀번호 숨기기 |
| **인프라 격리** | **Namespace & Cgroups** | 프로세스, 네트워크, 자원의 논리적/물리적 격리 | 방음벽 및 개인 울타리 |

---

### 2. 컨테이너 생명주기별 보안 아키텍처

빌드부터 운영까지 전 단계에 걸친 보안 가드레일을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Container Security Pipeline (Build to Run)              │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. Build 단계] ──────▶ [2. Shipping 단계] ──────▶ [3. Running 단계]          │
 │      │                         │                         │                  │
 │  ┌───┴──────────┐        ┌──────┴──────────┐        ┌────┴──────────────┐   │
 │  │ 이미지 스캐닝 │        │ 이미지 서명      │        │ 런타임 보안 감시   │   │
 │  │ (Trivy, Snyk)│        │ (Cosign, Notary)│        │ (Falco, AppArmor) │   │
 │  └──────────────┘        └──────────────────┘        └───────────────────┘   │
 │                                                                             │
 │  [런타임 보안 핵심 설정]                                                       │
 │  - **USER 1000** (Non-root 강제)                                            │
 │  - **Read-only Root FS** (파일 변조 차단)                                    │
 │  - **Capabilities 제거** (불필요한 시스템 권한 회수)                            │
 │                                                                             │
 │  ※ 핵심: 샌드박스 내부의 공격자가 밖으로 나갈 수 있는 모든 문을 폐쇄함              │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 핵심은 **심층 방어(Defense in Depth)**다. 1) **이미지 스캐닝**은 이미 알려진 취약점(CVE)이 있는 라이브러리의 유입을 막는다. 2) **이미지 서명**은 우리가 신뢰하는 이미지만 실행되도록 보장한다. 3) **런타임 보안**은 가장 결정적인 단계로, 설령 앱이 해킹당해도 **USER 1000** 설정 덕분에 해커는 관리자 권한을 얻지 못한다. 또한 **Read-only Root FS**는 해커가 악성 파일을 컨테이너에 새로 설치하는 것을 물리적으로 불가능하게 만든다.

---

### 심층 동작 원리: 네임스페이스(Namespace)와 샌드박스

컨테이너가 서로를 보지 못하게 만드는 '투명 인간' 기술이다.
1. **Namespaces**: 프로세스 ID(PID), 네트워크(Net), 마운트(Mnt) 등을 분리하여, 컨테이너 A가 B의 프로세스를 죽이거나 네트워크를 훔쳐보는 것을 막는다.
2. **Cgroups (Control Groups)**: 특정 컨테이너가 CPU나 메모리를 100% 써서 다른 컨테이너를 굶겨 죽이는(DoS) 것을 방지한다.
3. **가치**: 한 서버에 수백 개의 컨테이너가 있어도, 각자는 자신이 이 컴퓨터를 혼자 쓰고 있다고 착각하게 만든다.

- **📢 섹션 요약 비유**: 수영장의 레인(Namespace)과 같습니다. 옆 레인 사람이 아무리 빨리 헤엄쳐도(데이터 처리) 내 물살(자원)을 방해하지 못하게 벽을 세우는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 일반 실행 vs 일반 컨테이너 vs 강화된 샌드박스 비교

보안 강도가 높아질수록 성능 오버헤드는 증가한다.

| 비교 항목 | 일반 호스트 프로세스 | 일반 컨테이너 (Docker) | 강화된 샌드박스 (gVisor) |
|:---|:---|:---|:---|
| **격리 수준** | 없음 | 보통 (커널 공유) | **높음 (커널 대리 호출)** |
| **보안 경계** | 없음 | 리눅스 커널 자원 제어 | **사용자 정의 커널(Sentry)** |
| **성능** | 최상 | 우수 | 약간 저하 (시스템 콜 오버헤드) |
| **비유** | 거실에서 다 같이 생활 | 파티션만 쳐놓은 사무실 | **방음 처리된 독서실** |

---

### 기술 융합: 컨테이너 보안과 시큐어 코딩

이미지만 깨끗하다고 보안이 끝나는 것이 아니다.
- **Minimal Base Image**: Alpine 리눅스나 Distroless 이미지를 사용하여 공격자가 쓸 수 있는 도구(curl, ping, shell)를 아예 빼버린다.
- **가치**: 해커가 침투에 성공해도 정작 실행할 명령어가 없어 아무것도 못 하게 만드는 '불모지 보안' 전략이다.

- **📢 섹션 요약 비유**: 도둑이 집에 들어왔는데, 집 안에 숟가락 하나조차 없는 텅 빈 집(Distroless)을 만들어 도둑질할 의욕을 꺾어버리는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 금융권 결제 컨테이너의 보안 하드닝(Hardening)

- **문제 상황**: 결제 정보를 처리하는 컨테이너가 뚫릴 경우, 동일 노드에 있는 다른 서비스로의 횡적 이동(Lateral Movement) 리스크가 큼.
- **의사결정 (보안 강화 전략)**:
  1. **Rootless 도입**: 도커 데몬 자체를 일반 사용자 권한으로 구동하여, 컨테이너 탈출 시에도 호스트 관리자 권한 획득을 차단한다.
  2. **Security Context 설정**: 쿠버네티스 YAML에 `runAsNonRoot: true`와 `allowPrivilegeEscalation: false`를 명시하여 권한 상승 시도를 원천 봉쇄한다.
  3. **Syscall 필터링**: `Seccomp` 프로파일을 적용하여, 웹 서비스에 필요 없는 위험한 시스템 콜(예: `mount`, `ptrace`)을 커널 레벨에서 거부한다.
  4. **자동화 스캔 연동**: 취약점 점수가 High 이상인 이미지는 `kubectl apply` 단계에서 자동으로 거부(Admission Control)되도록 설정한다.

### 도입 체크리스트
- **User ID**: Dockerfile에 `USER` 명령어가 포함되어 있으며, Root(ID 0)가 아닌 아이디를 사용하고 있는가?
- **Base Image**: 불필요한 패키지가 잔뜩 든 `Ubuntu/CentOS` 대신 가벼운 `Alpine`이나 `Distroless`를 썼는가?
- **Secrets**: 이미지 레이어 안에 비밀번호나 API Key가 구워져(Hardcoded) 있지는 않은가?

### 안티패턴
- **--privileged 옵션 사용**: 개발이 편하다는 이유로 이 옵션을 켜는 행위. 컨테이너에게 호스트의 모든 권한을 주는 '보안 자살 행위'다.
- **Latest 태그 사용**: 어떤 버전이 배포되는지 알 수 없어 취약점 관리가 불가능해진다. 반드시 고유한 버전 태그를 사용해야 한다.

- **📢 섹션 요약 비유**: "현관문은 디지털 도어락으로 바꿨는데, 집 열쇠를 우체통(이미지 레이어)에 넣어두는" 바보 같은 실수를 하지 말아야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 컨테이너 탈출 성공률 0% 지향 | Non-root 및 Seccomp 적용 시 시스템 장악 위험 원천 차단 |
| **정량적 (Compliance)** | 보안 심사 통과율 향상 | 자동화된 이미지 스캔 증적을 통해 공급망 보안 요건 충족 |
| **정성적 (Standard)** | 안전한 배포 표준 확립 | 개발자 모두가 동일한 보안 가이드라인(Dockerfile 표준) 준수 |
| **정성적 (Resilience)** | 신속한 사고 대응 | 이미지 레이어 분석을 통해 취약한 부위만 신속히 교체 및 재배포 |

### 미래 전망
- **Hardware-enforced Isolation**: CPU의 기밀 컴퓨팅(Confidential Computing) 기능을 사용하여 메모리 내부의 데이터까지 보호하는 강력한 컨테이너 격리가 일반화될 것이다.
- **AI-Managed Runtime Security**: AI가 컨테이너의 정상적인 행위 패턴을 학습하여, 평소와 다른 아주 미세한 이상 징후만 보여도 즉시 컨테이너를 격리(Self-healing)하는 기술이 도입될 것이다.

### 참고 표준
- **CIS Docker Benchmark**: 도커 보안 설정의 정석
- **NIST SP 800-190**: Application Container Security Guide
- **PCI-DSS**: 컨테이너 환경의 카드 결제 보안 표준

- **📢 섹션 요약 비유**: 컨테이너 보안은 "작은 것들에 대한 완벽한 불신"입니다. 가장 작은 단위인 컨테이너 하나하나를 깐깐하게 관리할 때, 거대한 클라우드 대륙 전체가 평화로워질 수 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 도시락 상자(컨테이너)를 쌀 때, **나쁜 세균(취약점)이 없는지 돋보기로 먼저 검사하는 것**과 같아요.
2. 상자 안에 든 장난감 로봇에게 **위험한 칼(관리자 권한)을 주지 말고, 안전하게 놀 수 있는 도구만 쥐여주는 규칙**이 있답니다.
3. 이렇게 하면 도시락 상자가 튼튼해져서, 우리 맛있는 음식(데이터)을 끝까지 안전하게 지킬 수 있어요!
