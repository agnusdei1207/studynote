# 485. Software and Data Integrity Failures (소프트웨어 및 데이터 무결성 실패)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Software and Data Integrity Failures (소프트웨어 및 데이터 무결성 실패)는 **코드나 데이터가 비정상적으로 수정되거나 검증 없이 신뢰되는 상태**를 의미하며, 특히 CI/CD 파이프라인, 자동 업데이트, 역직렬화(Deserialization) 과정에서의 보호 미비를 포괄한다.
> 2. **가치**: 2021년 OWASP Top 10에서 8위에 위치하며, SolarWinds 사태와 같은 대규모 공급망 공격의 원인이 되는 '소프트웨어 무결성'과, 신뢰할 수 없는 데이터가 시스템 상태를 파괴하는 '데이터 무결성'을 동시에 강조한다.
> 3. **융합**: 디지털 서명(Digital Signature), 코드 서명(Code Signing), 안전한 역직렬화 기법과 결합되며, 현대 아키텍처에서는 빌드부터 배포까지 전 과정의 무결성을 보장하는 '증명 가능한 소프트웨어 공급망' 구축이 핵심이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 과거 'Insecure Deserialization' 항목이 2021년 'Software and Data Integrity Failures'로 통합·확장되었다. 이는 단순히 특정 기술의 허점이 아니라, **"우리가 사용하는 소프트웨어 부품과 입력받는 데이터가 정말로 믿을 수 있는 것인가?"**에 대한 근본적인 신뢰의 문제를 다룬다.

- **💡 비유**: Software and Data Integrity Failures는 **"정품 소프트웨어를 설치했다고 믿었는데 해커가 몰래 바꿔치기한 가짜 프로그램이 깔리거나(소프트웨어 무결성), 택배 상자 안에 폭탄이 들어있는데 내용물 확인도 안 하고 집 안으로 들여놓는 것(데이터 무결성)"**과 같습니다.

- **등장 배경**:
  1. **공급망 공격 (Supply Chain Attack)의 급증**: 해커들이 기업을 직접 공격하기보다, 기업이 신뢰하는 업데이트 서버나 오픈소스 저장소를 오염시켜 수만 개의 고객사를 한꺼번에 감염시키는 전략을 취하기 시작했다.
  2. **CI/CD 자동화의 명암**: 빌드와 배포가 자동화되면서 사람이 개입할 틈이 줄어들었고, 파이프라인 중간에 악성 코드가 삽입될 경우 검증 없이 운영 환경까지 배포되는 리스크가 커졌다.
  3. **객체 역직렬화의 위험성**: 네트워크를 통해 전달되는 객체(Object) 데이터가 서버 내부에서 복원되는 과정에서 악성 코드가 실행되는 취약점이 지속적으로 발견되었다.

- **📢 섹션 요약 비유**: 아무리 보안이 철저한 성이라도, 성으로 들어오는 보급품 상자 속에 숨어 들어온 스파이나 독이 든 음식(오염된 데이터/코드)에는 속수무책일 수 있다는 원리와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 소프트웨어 및 데이터 무결성 실패의 주요 유형

| 유형 | 설명 | 공격 사례 | 비유 |
|:---|:---|:---|:---|
| **소프트웨어 업데이트 위조** | 검증되지 않은 경로로 자동 업데이트 수행 | 업데이트 서버를 해킹하여 전 세계 PC에 악성코드 유포 | 가짜 패치 붙이기 |
| **부적절한 역직렬화** | 신뢰할 수 없는 데이터를 객체로 복원 | 자바 역직렬화 과정에서 원격 코드 실행(RCE) 발생 | 트로이 목마 목마 들여오기 |
| **파이프라인 오염** | CI/CD 도구 자체의 보안 설정 미흡 | 소스 코드가 빌드되는 과정에서 악성 바이너리 삽입 | 공장 조립 라인에 불량품 넣기 |
| **데이터 무결성 미검증** | 서명되지 않은 설정 파일이나 데이터 신뢰 | 설정 파일 변조를 통해 시스템 권한 탈취 | 위조된 명령서 따르기 |

---

### 무결성 보장을 위한 안전한 CI/CD 아키텍처

코드의 생성부터 배포까지 각 단계에서 무결성을 검증하는 '체인 오브 커스터디(Chain of Custody)'를 구축한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      무결성 검증 기반의 Secure Software Supply Chain            │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [개발자] ──▶ [Git Repo] ──▶ [Build Server] ──▶ [Artifact Repo] ──▶ [운영 환경] │
 │    │            │               │                 │                 │       │
 │  1. 서명 커밋  2. 코드 검증    3. 빌드 및 서명   4. 서명 확인 및 저장  5. 검증 후 실행 │
 │    (GPG)       (Scan)          (Code Sign)       (Integrity Check)  (Verify)    │
 │                                                                             │
 │  [주요 무결성 제어 기술]                                                       │
 │  - 코드 서명: 빌드된 바이너리에 대해 신뢰할 수 있는 CA의 인증서로 서명            │
 │  - 해시 체크: 파일 전송 전후의 Hash(SHA-256) 값을 대조하여 변조 여부 확인        │
 │  - SLSA Framework: 소프트웨어 공급망 수준을 높이기 위한 보안 등급 표준 적용       │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 무결성 확보의 핵심은 **디지털 서명**이다. 개발자가 코드를 올릴 때(GPG 서명), 빌드 서버가 결과물을 만들 때(코드 서명), 그리고 실제 운영 서버에서 이를 실행하기 직전에 이 서명들이 유효한지 확인하는 과정이 반복되어야 한다. 만약 중간에 공격자가 바이너리를 1바이트라도 수정하면 서명 검증 단계에서 실패(Verify Fail)하게 되어 실행이 차단된다. 이는 "우리가 의도한 그 소프트웨어가 맞는가?"를 기계적으로 증명하는 아키텍처다.

---

### 심층 동작 원리: 안전하지 않은 역직렬화 (Insecure Deserialization)

데이터를 객체로 바꾸는 과정에서 발생하는 무결성 실패를 다룬다.
1. **직렬화 (Serialization)**: 메모리상의 객체를 파일이나 네트워크로 전송하기 위해 바이트 스트림으로 변환하는 과정.
2. **역직렬화 (Deserialization)**: 전달받은 데이터를 다시 메모리상의 객체로 복원하는 과정.
3. **취약점**: 공격자가 역직렬화되는 데이터 내부에 '객체가 생성될 때 자동으로 실행되는 메서드(Magic Method)'를 조작하여 악성 코드를 심으면, 서버는 데이터를 읽는 것만으로도 공격자의 명령을 실행하게 된다.

- **📢 섹션 요약 비유**: 가구 조립 세트(직렬화된 데이터)를 샀는데, 설명서대로 조립했더니 가구가 완성되는 것이 아니라 갑자기 숨겨져 있던 스프링이 튀어나와 주인을 공격하는(악성 코드 실행) 함정과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 기밀성(Confidentiality) vs 무결성(Integrity) vs 가용성(Availability) 실패 비교

| 보안 요소 | 실패 시 현상 | 무결성 실패와의 관계 |
|:---|:---|:---|
| **기밀성** | 데이터가 남에게 읽힘 | 데이터가 변하지 않았어도 유출된 것임 |
| **무결성** | **데이터가 변조됨** | **이 항목의 핵심 (코드 바꿔치기, 값 조작 등)** |
| **가용성** | 시스템이 멈춤 | 무결성이 깨진 코드가 실행되어 시스템이 파괴될 때 발생 가능 |

---

### 기술 융합: SLSA (Supply-chain Levels for Software Artifacts)

Google이 주도하는 소프트웨어 공급망 보안 프레임워크다.
- **Level 1**: 빌드 프로세스가 문서화됨.
- **Level 2**: 빌드 서비스가 호스팅되고 빌드 증명(Provenance)이 서명됨.
- **Level 3**: 빌드 환경이 격리되고 코드가 다른 곳에서 빌드되지 않았음을 보장함.
- **Level 4**: 두 명 이상의 작업자가 승인해야 배포 가능 (Hermetic Build).

- **📢 섹션 요약 비유**: 음식의 원산지 표기(Level 1)를 넘어, 조리 과정을 CCTV로 녹화하고(Level 2), 외부인이 출입할 수 없는 주방에서(Level 3) 두 명의 요리사가 동시에 검수하는(Level 4) 최고급 위생 시스템과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 자동 업데이트 기능을 통한 악성코드 유포 방지

- **문제 상황**: 우리 회사의 유틸리티 소프트웨어가 업데이트될 때마다 고객사 PC에서 바이러스 백신 경고가 발생함. 조사 결과 업데이트 서버의 바이너리가 변조되었을 가능성 제기됨.
- **의사결정 (무결성 강화 전략)**:
  1. **코드 서명 의무화**: 빌드 서버에서 생성된 모든 업데이트 파일에 대해 EV (Extended Validation) 코드 서명을 적용한다.
  2. **클라이언트 검증 로직**: 업데이트 클라이언트는 파일을 다운로드한 후, 서버에서 별도로 받아온 해시값(SHA-512)과 대조하고 서명이 유효하지 않으면 설치를 즉시 중단한다.
  3. **전송 채널 보호**: 업데이트 메타데이터를 전송할 때 HTTPS를 사용하고, HSTS를 통해 인증서 변조 시도를 차단한다.
  4. **역직렬화 방어**: 업데이트 정보를 JSON으로 주고받을 때, 임의의 클래스로 역직렬화하는 기능을 끄고 화이트리스트 기반의 단순 데이터 매핑만 허용한다.

### 도입 체크리스트
- **Signature Verification**: 서드파티 라이브러리나 도구를 가져올 때 디지털 서명을 확인하는 절차가 있는가?
- **Pipeline Security**: CI/CD 도구(Jenkins, GitHub Actions 등)의 접근 권한이 엄격히 관리되고 로그가 남는가?
- **SCA Integration**: 사용 중인 오픈소스의 무결성을 검증하기 위해 SCA 도구를 활용하고 있는가?

### 안티패턴
- **HTTP 업데이트**: 업데이트 파일을 암호화되지 않은 HTTP로 내려받는 행위. 중간에 누구나 파일을 바꿔치기할 수 있다.
- **범용 역직렬화 허용**: `readObject()` 같은 함수를 쓰면서 들어오는 데이터의 타입을 제한하지 않는 행위.

- **📢 섹션 요약 비유**: "믿을 만한 친구가 준 선물이니까"라며 내용물을 확인도 안 하고 덥석 받는 것이 가장 위험한 보안 불감증이라는 점과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 공급망 사고 99% 예방 | 서명 및 해시 검증 도입 시 바이너리 변조를 통한 공격 원천 차단 |
| **정량적 (Reliability)** | 배포 신뢰도 향상 | 잘못된 버전이나 오염된 코드가 운영 환경에 배포되는 에러율 감소 |
| **정성적 (Compliance)** | 글로벌 보안 요건 충족 | 미국 행정명령 등 최신 글로벌 소프트웨어 공급망 보안 가이드 준수 |
| **정성적 (Brand)** | 제품 무결성 이미지 확보 | "이 회사의 소프트웨어는 변조되지 않았다"는 강력한 신뢰 마케팅 가능 |

### 미래 전망
- **Blockchain for Integrity**: 소프트웨어 빌드 이력과 서명 데이터를 블록체인(Hyperledger 등)에 기록하여 위변조가 절대 불가능한 투명한 공급망을 구축할 것이다.
- **AI-Based Anomaly Detection in CI/CD**: 평소의 빌드 패턴과 다른 코드 변경이나 비정상적인 라이브러리 유입을 AI가 실시간으로 탐지하여 파이프라인을 중단시키는 지능형 무결성 보호가 확산될 것이다.

### 참고 표준
- **NIST SP 800-218**: Secure Software Development Framework (SSDF)
- **SLSA (Supply-chain Levels for Software Artifacts)**: 공급망 보안 등급 표준
- **CWE-502**: 역직렬화 취약점 정의

- **📢 섹션 요약 비유**: 예전에는 문을 잘 잠그는 것이 보안의 전부였다면, 이제는 문으로 들어오는 모든 공기와 물(데이터/코드)의 성분을 실시간으로 분석하여 독극물을 가려내는 정밀 검역 시스템의 시대가 되었습니다.
