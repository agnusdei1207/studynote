# 558. 서버리스 아키텍처 (BaaS, FaaS)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서버리스(Serverless)는 서버가 없다는 뜻이 아니라, 개발자가 서버의 프로비저닝, 패치, 스케일링 등 **인프라 관리를 전혀 신경 쓰지 않고 코드 구현에만 집중**할 수 있게 하는 클라우드 컴퓨팅 실행 모델이다.
> 2. **가치**: 요청이 올 때만 자원을 할당하는 **이벤트 중심(Event-driven)** 구조를 통해 '사용하지 않을 때 비용 0원'이라는 극단적인 비용 효율성과 무한한 자동 확장성(Auto-scaling)을 제공한다.
> 3. **융합**: 서드파티 서비스를 API로 호출하는 **BaaS (Backend as a Service)**와 비즈니스 로직을 함수 단위로 실행하는 **FaaS (Function as a Service)**가 결합되어, 현대 마이크로서비스(MSA)의 가장 경량화된 구현체로 쓰인다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 서버리스는 클라우드 서비스 제공자(CSP)가 인프라를 완전히 추상화한 상태다. 개발자는 단순히 코드(함수)를 업로드하거나 필요한 기능(DB, 인증)을 API로 불러쓰기만 하면 된다. 인프라의 모든 복잡성은 CSP가 책임진다.

- **💡 비유**: 서버리스는 **"자판기나 배달 음식"**과 같습니다. 직접 요리하기 위해 주방을 치우고 재료를 사고 불 조절을 하는(서버 관리) 과정 없이, 버튼만 누르면 음식이 나오거나(BaaS) 주문이 올 때만 요리사가 와서 딱 한 그릇만 만들고 떠나는(FaaS) 시스템입니다.

- **등장 배경**:
  1. **운영 오버헤드 감소**: 서버 패치나 OS 업데이트 등 비즈니스 가치와 상관없는 '단순 반복 작업(Toil)'에서 개발자를 해방시키고자 했다.
  2. **유휴 자원 낭비 방지**: 트래픽이 없는 밤 시간대에도 켜져 있는 가상 서버 비용을 절감하기 위해 '실행 시간 단위 과금' 모델이 필요해졌다.
  3. **초고속 확장성**: 갑작스러운 트래픽 폭주 시 수초 내에 수천 개의 인스턴스를 늘려야 하는 현대 웹 환경의 요구사항을 충족하기 위해 탄생했다.

- **📢 섹션 요약 비유**: 발전기를 직접 돌려 전기를 만드는(온프레미스) 시대를 지나, 벽에 있는 콘센트에 플러그만 꽂으면(서버리스) 쓴 만큼만 요금을 내고 무제한 전기를 쓰는 시대로 진화한 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. BaaS vs FaaS 핵심 비교

서버리스는 크게 두 가지 서비스 형태로 나뉜다.

| 구분 | BaaS (Backend as a Service) | FaaS (Function as a Service) |
|:---|:---|:---|
| **정의** | 앱에 필요한 기능을 API로 제공받음 | 사용자가 직접 짠 로직을 함수로 실행 |
| **제어권** | 매우 낮음 (기성품 사용) | 중간 (코드는 직접 작성) |
| **대표 사례** | Firebase(인증/DB), AWS Cognito | AWS Lambda, Google Cloud Functions |
| **주요 용도** | 로그인, 푸시 알림, 데이터 저장 | **비즈니스 로직, 데이터 변환** |
| **비유** | 완성된 반찬 사 오기 | 레시피대로 요리만 하기 |

---

### 2. 서버리스 동작 및 과금 아키텍처

이벤트가 발생했을 때만 자원이 소환되는 메커니즘을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      서버리스 (Event-driven) 실행 흐름 및 구조                    │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [이벤트 발생 (Trigger)] ─────▶ [서버리스 플랫폼] ─────▶ [코드 실행 (Invoke)]      │
 │  - HTTP 요청 (API GW)          (관리 및 배분)           (FaaS: Lambda)       │
 │  - 파일 업로드 (S3)                   │                    │                │
 │  - 타이머 (Cron)                     ▼                    ▼                │
 │                                ┌────────────┐       ┌────────────┐          │
 │                                │  Warm Pool │ ──▶   │  Container │          │
 │                                │  (Ready)   │       │  (Running) │          │
 │                                └────────────┘       └──────┬─────┘          │
 │                                                            │                │
 │  [과금 청구 방식] ◀─────────────────────────────────────────┘                │
 │  - (실행 횟수) x (메모리 크기) x (실행 시간: ms 단위)                          │
 │                                                                             │
 │  ※ 핵심 특징: No Request = No Cost (요청이 없으면 비용 0원)                   │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 서버리스 아키텍처의 핵심은 **이벤트 트리거 (Trigger)**다. 평소에는 아무런 자원도 차지하지 않다가, 사용자가 사진을 올리거나 API를 호출하는 '사건'이 터지면 그제서야 플랫폼이 컨테이너를 잠깐 띄워 코드를 실행한다. 이때 처음 실행할 때 발생하는 약간의 지연 시간을 **Cold Start**라고 하며, 이미 띄워진 자원을 재사용하는 것을 **Warm Start**라고 한다. 과금은 서버의 대수가 아니라, 코드가 돌아간 **밀리초(ms) 단위의 시간**에 대해서만 매겨지므로 극단적인 가성비를 자랑한다.

---

### 심층 동작 원리: 상태 없음 (Statelessness)

서버리스 설계를 위해 반드시 지켜야 할 원칙이다.
1. **휘발성 환경**: 함수가 한 번 실행되고 나면 그 안의 변수나 파일은 모두 사라진다. (재시작 시 보장 안 됨)
2. **상태 저장**: 보존해야 할 데이터가 있다면 반드시 외부의 **BaaS** (DynamoDB, S3 등)에 저장해야 한다.
3. **가치**: 상태가 없기 때문에 플랫폼은 요청이 1개든 100만 개든 상관없이 동일한 함수를 무한히 복제하여 동시에 실행할 수 있다.

- **📢 섹션 요약 비유**: 일회용 장갑과 같습니다. 한 번 쓰고 버려야지, 장갑에 뭔가를 적어두고 나중에 다시 끼려고 하면(상태 보존 시도) 이미 쓰레기통에 가 있어 찾을 수 없는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 가상 머신(VM) vs 컨테이너 vs 서버리스

| 비교 항목 | Virtual Machine (IaaS) | Container (CaaS/PaaS) | Serverless (FaaS) |
|:---|:---|:---|:---|
| **추상화 단위** | 하드웨어 (OS 포함) | 애플리케이션 (OS 공유) | **함수 (로직만)** |
| **확장 속도** | 분 단위 (느림) | 초 단위 (빠름) | **밀리초 단위 (매우 빠름)** |
| **관리 책임** | OS 패치, 런타임 등 전체 | 런타임, 라이브러리 | **코드 로직만** |
| **비용 모델** | 할당된 시간 (고정비) | 할당된 자원 (고정비) | **실행된 시간 (변동비)** |
| **비유** | 집 사기 | 집 월세 내기 | 호텔 방 하루 빌리기 |

---

### 기술 융합: 서버리스와 마이크로서비스 (Micro-MSA)

서버리스는 MSA를 구현하는 가장 작은 부품이 된다.
- **초소형 서비스**: 서비스 전체를 서버에 띄우는 대신, 각 API 엔드포인트 하나하나를 독립된 Lambda 함수로 만든다.
- **가치**: 특정 API(예: 조회)만 트래픽이 몰려도 그 함수만 알아서 수만 개로 늘어나 대응하므로 인프라 효율이 극대화된다.

- **📢 섹션 요약 비유**: 거대한 기계 장치(모놀리식) 대신, 수만 개의 아주 작은 나노 로봇(서버리스)들이 흩어져 있다가 명령이 떨어지면 한곳에 모여 임무를 수행하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 이미지 썸네일 자동 생성 파이프라인 구축

- **문제 상황**: 사용자가 원본 이미지를 올리면 웹용, 모바일용 썸네일을 즉시 만들어야 함. 트래픽이 불규칙하여 전용 서버를 두기엔 돈이 아깝고, 이벤트 때는 수만 명의 업로드가 몰림.
- **의사결정 (서버리스 아키텍처 적용)**:
  1. **이벤트 트리거**: 사용자가 S3 버킷에 파일을 저장하는 행위를 **Trigger**로 설정한다.
  2. **FaaS 연동**: 파일 저장 즉시 미리 작성한 이미지 리사이징 **Lambda 함수**를 호출한다.
  3. **BaaS 활용**: 생성된 썸네일 경로는 **DynamoDB(BaaS)**에 저장하고, 실제 파일은 다시 **S3**에 보관한다.
  4. **결과**: 사용자가 한 명도 없을 때는 비용이 0원이며, 10만 명이 동시에 올려도 람다가 10만 개 떠서 동시에 처리하므로 지연 시간이 거의 없다.

### 도입 체크리스트
- **Cold Start**: 지연 시간에 민감한 서비스인가? (Provisioned Concurrency 등 대책 필요)
- **Vendor Lock-in**: 특정 CSP의 특화 서비스(BaaS)에 너무 종속되어 나중에 다른 클라우드로 옮기기 힘들지 않은가?
- **Execution Limit**: 함수 실행 시간이 플랫폼 제한(보통 15분)을 넘지는 않는가? (배치 작업 등에는 부적합)

### 안티패턴
- **Long-running Task**: 1시간 동안 돌아가는 연산을 억지로 서버리스로 돌리는 행위. 비용이 일반 서버보다 훨씬 비싸진다.
- **Heavy Initialization**: 함수 한 번 실행하는데 수십 개의 라이브러리를 로드하여 Cold Start 시간을 늘리는 비효율적 코딩.

- **📢 섹션 요약 비유**: 100m 달리기를 하는데 무거운 배낭(과도한 라이브러리)을 메고 뛰는 선수는 결코 좋은 성적(빠른 응답)을 낼 수 없음을 명심해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Cost)** | 인프라 비용 최대 90% 절감 | 유휴 자원에 대한 지불이 사라지고 실행된 만큼만 정산 |
| **정량적 (Scale)** | 수평 확장성 무제한 확보 | 트래픽 규모에 상관없이 CSP가 관리하는 인프라 한도까지 자동 확장 |
| **정성적 (Agility)** | 개발 생산성 극대화 | 서버 인프라 고민 없이 비즈니스 로직 작성 및 즉시 배포 가능 |
| **정성적 (Stability)** | 고가용성 기본 내재 | CSP가 관리하는 분산 환경에서 돌아가므로 장애에 강함 |

### 미래 전망
- **Full-stack Serverless**: DB, 인증뿐만 아니라 AI 연산, 블록체인 노드 등 모든 백엔드 기능이 서버리스 API화 되어 조립만으로 서비스를 만드는 시대가 올 것이다.
- **Cold Start 박멸**: 하드웨어 가속 및 런타임 최적화 기술로 서버리스의 고질적 문제인 초기 지연 시간이 제로에 수렴하게 될 것이다.

### 참고 표준
- **CloudEvents**: 서버리스 이벤트 정보 교환을 위한 상호 운용성 표준
- **AWS Lambda / Google Functions Documentation**: 업계 기술 표준
- **CNCF Serverless Whitepaper**: 서버리스 기술 정의 및 분류 체계

- **📢 섹션 요약 비유**: "서버"라는 물리적 족쇄를 끊어내고, 오직 "비즈니스 가치(Code)"만이 자유롭게 떠다니며 가치를 창출하는 것, 그것이 서버리스가 꿈꾸는 소프트웨어 공학의 미래입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **서버리스**는 내가 직접 요리하지 않고 **"마법의 요정"에게 음식을 주문하는 것**과 같아요.
2. 요정은 내가 음식을 먹고 싶을 때만 나타나서 뚝딱 만들어주고, **내가 먹은 만큼만 사탕(돈)을 주면 된답니다.**
3. 요정에게 주방(서버)을 빌려줄 필요도 없고, 요정이 수천 명으로 늘어날 수도 있어서 아주 많은 친구들과 파티를 열 때 최고예요!
