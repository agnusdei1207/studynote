# 471-476. 웹 보안과 상태 관리 (HTTPS, Cache, Cookie)

> **핵심 인사이트**: 웹은 원래 정보를 '공유'하기 위해 만들어졌으나, 이제는 '보안'과 '기억'이 더 중요해졌다. 데이터를 암호화하는 HTTPS, 반복되는 데이터를 저장해두는 캐시, 그리고 로그인 상태를 유지하는 쿠키와 세션은 현대 웹 서비스의 3대 필수 지지대이다.

---

## Ⅰ. HTTPS (HTTP Secure)
HTTP 프로토콜에 SSL/TLS 암호화 계층을 추가한 방식입니다. 포트 **443**번을 사용합니다.

* **제공 가치**:
  - **기밀성**: 패킷을 가로채도 내용을 읽을 수 없습니다.
  - **무결성**: 데이터가 중간에 변조되었는지 확인합니다.
  - **인증**: 접속한 사이트가 진짜인지 증명합니다.
* **작동 방식**: 공개키 암호화(대칭키 전달용)와 대칭키 암호화(실제 데이터 전송용)를 혼합하여 사용합니다.

---

## Ⅱ. 웹 캐싱 (Web Caching) 메커니즘
동일한 데이터를 매번 서버에서 가져오는 낭비를 줄이기 위해 로컬이나 프록시에 임시 저장하는 기술입니다.

* **Cache-Control 헤더**:
  - `max-age`: 캐시가 유효한 시간(초).
  - `no-store`: 절대 캐시하지 마라.
  - `no-cache`: 캐시를 써도 되지만, 쓰기 전에 서버에 "바뀐 거 없어?"라고 물어봐라.
* **유효성 검증**:
  - **ETag**: 파일의 고유 해시값. 값이 같으면 서버는 `304 Not Modified` 응답을 줍니다.
  - **Last-Modified**: 파일이 마지막으로 수정된 날짜.

---

## Ⅲ. 상태 유지 기술 (Cookie vs Session)
HTTP의 Stateless 특성을 극복하고 사용자를 기억하기 위한 방법입니다.

### 1. 쿠키 (Cookie)
* **저장 위치**: **클라이언트(브라우저)**의 로컬 파일.
* **특징**: 서버가 브라우저에게 "이거 보관해둬"라고 주면, 브라우저는 다음 요청부터 매번 서버에 쿠키를 함께 보냅니다. 보안에 취약하여 민감한 정보는 담지 않습니다.

### 2. 세션 (Session)
* **저장 위치**: **서버**의 메모리나 데이터베이스.
* **특징**: 서버에 실제 정보를 저장하고, 클라이언트에게는 그 정보의 열쇠인 **세션 ID**만 쿠키로 넘겨줍니다. 쿠키보다 보안성이 훨씬 높습니다.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[웹 서비스 운영의 3요소]

1. 보안 (HTTPS)  : [ Client ] <==(SSL/TLS 가로막기)==> [ Server ]
2. 성능 (Cache)  : [ Client ] ──(이미지 있어?)──> [ Local Storage ]
3. 기억 (Cookie) : [ Client ] --(내 ID는 123이야)--> [ Server ]
```

📢 **섹션 요약 비유**: **HTTPS**는 편지를 보낼 때 투명한 봉투 대신 금고에 넣어 열쇠와 함께 보내는 것입니다. **캐시**는 매일 먹는 우유를 매번 마트에 가지 않고 냉장고에 넣어두고 꺼내 먹는 지혜입니다. **쿠키**는 단골 카페에서 내 얼굴을 보고 "늘 드시던 걸로 드릴까요?"라고 묻는 것처럼, 서버가 나를 알아볼 수 있게 내 주머니에 넣어둔 이름표와 같습니다.
