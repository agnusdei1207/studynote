# 477-481. 현대적 웹 API와 실시간 기술

> **핵심 인사이트**: 데이터 통신은 '웹 페이지' 중심에서 '데이터(API)' 중심으로 변했다. 표준이 된 REST부터, 필요한 데이터만 골라 담는 GraphQL, 서버 간 초고속 대화를 위한 gRPC, 그리고 양방향 실시간 대화가 가능한 WebSocket까지 상황에 맞는 도구 선택이 중요하다.

---

## Ⅰ. 현대 웹 API 기술

### 1. REST (Representational State Transfer)
* **개념**: HTTP 메서드(GET, POST 등)와 URI(주소)를 결합하여 데이터(자원)를 처리하는 가장 대중적인 아키텍처입니다.
* **장점**: 단순하고 별도의 설치가 필요 없으며 누구나 이해하기 쉽습니다.

### 2. GraphQL
* **개념**: 페이스북(메타)에서 만든 쿼리 언어입니다.
* **차별점**: REST는 서버가 주는 대로 받아야 했지만, GraphQL은 클라이언트가 **"나 이거, 이거만 줘"**라고 요청할 수 있습니다. (Overfetching 방지)

### 3. gRPC (Google Remote Procedure Call)
* **개념**: 구글에서 만든 고성능 RPC 프레임워크입니다.
* **특징**: JSON 대신 **Protocol Buffers**라는 이진 포맷을 쓰고, **HTTP/2** 위에서 동작합니다. REST보다 압도적으로 빠르고 작아 마이크로서비스(MSA) 내부 통신에 주로 쓰입니다.

---

## Ⅱ. 실시간 양방향 통신 기술

### 1. WebSocket (웹소켓)
* **특징**: 하나의 TCP 연결을 맺은 후, **전이중(Full-Duplex) 양방향 통신**을 지속합니다.
* **용도**: 실시간 채팅, 주식 차트, 온라인 게임 등 실시간 데이터 교환이 필수인 곳에 쓰입니다.

### 2. SSE (Server-Sent Events)
* **특징**: 서버에서 클라이언트로만 데이터를 쏘는 **단방향** 실시간 기술입니다. 
* **장점**: 웹소켓보다 가볍고 HTTP 표준을 그대로 사용하므로 구현이 쉽습니다.
* **용도**: 뉴스 피드 업데이트, 알림 푸시 등.

---

## Ⅲ. 기술 비교 요약

| 기술 | 방식 | 데이터 형식 | 주요 용도 |
| :--- | :--- | :--- | :--- |
| **REST** | 단방향(요청/응답) | JSON / XML | 일반적인 웹 서비스 API |
| **GraphQL** | 단방향(요청/응답) | JSON | 복잡한 데이터 구조 조회 |
| **gRPC** | 양방향 지원 | **바이너리** | 서버 간 초고속 내부 통신 |
| **WebSocket** | **양방향(전이중)** | 텍스트 / 바이너리 | 채팅, 게임, 실시간 차트 |

---

## Ⅳ. 개념 맵 및 요약

```ascii
[API 및 실시간 통신 패러다임]

1. [ Client ] --(주세요)--> [ Server ] (REST / GraphQL)
   [ Client ] <--(드려요)--- [ Server ]

2. [ Client ] <========> [ Server ] (WebSocket)
   (빨대를 꽂아놓고 양쪽에서 수시로 주고받음)

3. [ Server ] --------> [ Client ] (SSE / Push)
   (서버가 일방적으로 새 소식을 알려줌)
```

📢 **섹션 요약 비유**: **REST**는 식당 메뉴판에서 정해진 세트 메뉴를 주문하는 것이고, **GraphQL**은 뷔페에서 먹고 싶은 것만 접시에 골라 담는 것입니다. **gRPC**는 주방장들끼리 자기들만의 암호(바이너리)로 빠르게 대화하는 무전기입니다. **WebSocket**은 전화기를 아예 귀에 대고 계속 떠드는 채팅이고, **SSE**는 라디오 방송처럼 서버가 보내주는 소식을 듣기만 하는 것입니다.
