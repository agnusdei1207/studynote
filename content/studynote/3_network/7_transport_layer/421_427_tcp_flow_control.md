# 421-427. TCP 흐름 제어와 슬라이딩 윈도우

> **핵심 인사이트**: TCP는 상대방이 소화할 수 있을 만큼만 음식을 입에 넣어주는 아주 배려심 깊은 프로토콜이다. '수신 윈도우'라는 정보를 주고받으며 속도를 조절하고, '네이글 알고리즘'과 같은 최적화 기법을 통해 작은 패킷이 네트워크를 낭비하지 않도록 관리한다.

---

## Ⅰ. TCP 슬라이딩 윈도우 (Sliding Window)
수신 측의 확인 응답(ACK)을 기다리지 않고도 송신 측이 한 번에 보낼 수 있는 데이터의 양을 조절하는 기법입니다.

* **동작 원리**: 
  - 수신자는 ACK 패킷을 보낼 때마다 자신의 남은 버퍼 크기인 **Window Size (RWND)**를 헤더에 적어 보냅니다.
  - 송신자는 이 RWND 값을 넘지 않는 선에서 데이터를 연속해서 보냅니다.
* **윈도우 스케일 (Window Scale)**: 기본 헤더의 윈도우 필드는 16비트(64KB)뿐입니다. 이를 기가비트급 고속망에서도 쓸 수 있도록 배수 옵션을 써서 최대 1GB까지 늘리는 기술입니다.

---

## Ⅱ. 어리석은 윈도우 증후군 (Silly Window Syndrome)
데이터보다 헤더가 더 커서 네트워크 효율이 극도로 떨어지는 비정상적인 현상입니다.

### 1. 송신 측 문제와 해결 (네이글 알고리즘)
* **현상**: 애플리케이션이 1바이트씩 찔끔찔끔 데이터를 만들어낼 때마다 즉시 패킷을 만들어 보내면, 배보다 배꼽(헤더 40바이트)이 더 큰 낭비가 생깁니다.
* **해결 (Nagle's Algorithm)**: 보낼 데이터가 아주 작으면, 이전에 보낸 패킷의 ACK가 올 때까지 전송을 멈추고 데이터를 버퍼에 모았다가 한꺼번에 보냅니다. (단, 실시간 게임 등에서는 반응 속도를 위해 끄기도 합니다.)

### 2. 수신 측 문제와 해결 (클라크 해결책)
* **현상**: 수신자의 버퍼가 꽉 찼다가 겨우 1바이트의 여유가 생겼을 때, 송신자에게 "윈도우 1바이트 열렸어!"라고 알리면 송신자는 또 1바이트짜리 비효율적인 패킷을 보냅니다.
* **해결 (Clark's Solution)**: 버퍼에 어느 정도 유의미한 공간(최대 세그먼트 크기 MSS의 절반 이상)이 생길 때까지는 윈도우 크기를 강제로 **0**이라고 거짓말하며 기다립니다.

---

## Ⅲ. 지연된 확인 응답 (Delayed ACK)
* **개념**: 패킷 하나 받을 때마다 매번 ACK를 보내지 않고, 조금 기다렸다가 다음 패킷까지 몰아서 하나의 ACK로 대답하거나, 수신자가 보낼 데이터가 있을 때 거기에 얹어서 보내는(Piggybacking) 기술입니다.

---

## Ⅳ. 영 윈도우(Zero Window)와 탐색 (Persist Timer)
수신자가 "나 지금 꽉 찼어(RWND=0)"라고 알린 뒤, 나중에 버퍼가 비었을 때 보내는 알림 패킷이 유실되면 두 라우터는 영원히 서로를 기다리는 데드락(Deadlock)에 빠집니다.

* **해결**: 송신자는 주기적으로 **ZWP (Zero Window Probe)** 패킷을 보내어 "정말 아직도 0이니?"라고 쿡 찔러보며 확인합니다.

---

## Ⅴ. 개념 맵 및 요약

```ascii
[TCP 흐름 제어의 핵심 흐름]

  [ 송신자 (Sender) ] <────── (ACK + Window Size: 2000) ────── [ 수신자 (Receiver) ]
          │                                                       ▲
          │ (1) "오케이, 너가 2000만큼 여유 있구나!"                     │
          │ (2) 1000바이트 패킷 전송 (ACK 대기 안 함)                   │
          │ (3) 1000바이트 패킷 또 전송 (총 2000 전송 완료)              │
          ▼                                                       │
  [ 송신 중단 및 대기 ] ──────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: **슬라이딩 윈도우**는 부모님(송신자)이 아이(수신자) 입에 밥을 넣어주는 것과 같습니다. 아이가 입안의 밥을 삼켜서 공간(Window)이 생겼다고 알려줄 때만 다음 숟가락을 넣습니다. **네이글 알고리즘**은 밥 한 톨씩 주는 게 감질나서 숟가락이 찰 때까지 모아서 한 번에 넣어주는 요령이고, **클라크 해결책**은 입안에 밥 한 톨만큼 공간 났다고 성급하게 다음 숟가락을 벌리지 않는 아이의 인내심입니다.
