# 프레임화 (Framing) - 비트 스터핑, 바이트 스터핑

> 1. **본질**: 물리 계층에서 전달되는 연속적이고 끊임없는 비트 스트림(Bit Stream)을 수신측이 의미 있는 정보 단위로 인식할 수 있도록, 시작과 끝을 구분하는 플래그(Flag)를 삽입하여 **프레임(Frame)** 단위로 구조화하는 기술이다.
> 2. **가치**: 데이터 블록의 경계를 명확히 설정함으로써 에러 검출(CRC) 및 흐름 제어의 기본 단위를 제공하며, 데이터 내에 플래그와 동일한 패턴이 나타나도 혼동하지 않게 하는 **투명성(Transparency)**을 보장한다.
> 3. **융합**: HDLC, PPP 등 고전적인 데이터 링크 프로토콜부터 현대 이더넷(Ethernet)의 프리앰블(Preamble) 및 가드 인터벌 설계의 논리적 모태가 된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: **프레임화(Framing)**는 '어디서부터 어디까지가 한 묶음인가'를 약속하는 것이다. 주로 특수 비트 패턴인 **플래그(Flag)**를 양 끝에 붙인다. 이때 실제 데이터 안에 플래그와 똑같은 모양이 섞여 있으면 수신기가 혼란에 빠지는데, 이를 막기 위해 가짜 비트나 바이트를 끼워 넣는 것이 **스터핑(Stuffing)** 기술이다.

- **💡 비유**: 긴 문장에서 단어 사이를 띄어 쓰는 것과 같습니다. 문장 시작에 "시작", 끝에 "끝"이라는 단어를 붙이기로 했는데, 만약 내 이름이 "끝돌이"라면 컴퓨터는 내 이름을 읽다가 문장이 끝난 줄 알겠죠? 그래서 이름 중간에 "진짜 아님"이라는 표시를 살짝 끼워 넣는 것이 스터핑입니다.

- **등장 배경**:
  1. **동기화의 한계**: 물리 계층은 비트 하나하나만 보낼 뿐, 어디가 메시지의 시작인지 알려주지 않는다.
  2. **투명성 문제**: 사용자가 어떤 데이터를 보내더라도(바이너리 파일 등) 시스템이 제어 문자와 혼동하지 않고 있는 그대로 전달해야 한다는 '데이터 투명성' 요구가 발생했다.
  3. **효율적 복구**: 통신 에러 발생 시 전체 데이터를 버리는 대신, 손상된 프레임만 찾아내어 다시 받기 위해 단위를 나누어야 했다.

- **📢 섹션 요약 비유**: 끊임없이 쏟아지는 구슬(비트)들을 상자(프레임)에 담아 포장하고, 상자 안에 든 구슬이 포장지 문양(플래그)과 헷갈리지 않게 별도의 표식(스터핑)을 하는 물류 관리 기술입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 프레임 구성 방식의 분류

| 방식 | 설명 | 특징 | 비유 |
|:---|:---|:---|:---|
| **문자 카운트** | 헤더에 프레임의 전체 길이를 숫자로 표기 | 단순하나 숫자 비트가 깨지면 대참사 | 택배 송장에 "구슬 10개 있음" 적기 |
| **바이트 스터핑** | 제어 문자 앞에 ESC 문자를 삽입 | 문자 위주 프로토콜 (PPP 등) | "끝"이라는 단어 앞에 "가짜" 붙이기 |
| **비트 스터핑** | 특정 비트 패턴 사이에 '0'을 강제 삽입 | 비트 위주 프로토콜 (HDLC 등) | 1이 5개 나오면 무조건 0 끼워넣기 |

---

### 비트 스터핑 (Bit Stuffing) 메커니즘

가장 널리 쓰이는 HDLC 프로토콜의 예시이다. 플래그 패턴은 `01111110` (0 하나, 1 여섯 개, 0 하나)이다.

```text
 [ 송신측: '0' 삽입 로직 ]
 데이터에 1이 연속으로 5개 나타나면, 6번째에 무조건 '0'을 하나 집어넣는다.
 
 원본: 0 1 1 1 1 1 1 0  (데이터 내에 플래그와 똑같은 패턴 존재)
 변환: 0 1 1 1 1 1 0 1 0 (5번째 1 뒤에 '0' 스터핑)
 
 -----------------------------------------------------------
 
 [ 수신측: '0' 제거 로직 ]
 1이 연속으로 5개 나온 뒤, 그 다음 비트를 확인한다.
 1. 다음 비트가 '0'이면 -> 스터핑된 비트이므로 삭제 (정상 데이터 복원)
 2. 다음 비트가 '1'이고 그 다음이 '0'이면 -> '01111110' 패턴이므로 프레임의 종료(Flag)로 인식
```

**[다이어그램 해설]** 이 방식을 쓰면 실제 데이터 안에 1이 아무리 많아도 전송로상에는 절대로 1이 6개 이상 연속될 수 없다(플래그 제외). 따라서 수신기는 1이 6개 연속되는 순간 "아, 이제 프레임이 끝났구나!"라고 100% 확신할 수 있게 된다.

---

### 바이트 스터핑 (Byte Stuffing / Character Stuffing)

문자 단위로 통신하는 경우, 제어 문자와 데이터 문자를 구별하기 위해 **ESC(Escape)** 문자를 활용한다.

```text
 [ 바이트 스터핑 규칙 ]
 
 1. 데이터에 FLAG가 있으면: [ESC] [FLAG] 로 변환
 2. 데이터에 ESC가 있으면:  [ESC] [ESC]  로 변환
 
 [ 프레임 예시 ]
 [FLAG] [Header] [Payload (ESC FLAG)] [Trailer] [FLAG]
                    ↓ (스터핑 후)
 [FLAG] [Header] [ESC ESC ESC FLAG] [Trailer] [FLAG]
```

**[다이어그램 해설]** 수신기는 ESC를 만나는 순간 "다음에 오는 문자는 제어용이 아니라 순수 데이터다"라고 해석한다. 이 방식은 비트 단위 연산이 어려운 초기 하드웨어 환경에서 소프트웨어적으로 처리하기 매우 편리했다.

- **📢 섹션 요약 비유**: "끝"이라는 외침이 진짜 종료인지 아니면 이름의 일부인지를 구별하기 위해, 데이터 중간에 "이건 진짜 이름이야"라는 꼬리표(ESC/스터핑 비트)를 달아주는 약속입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비트 스터핑 vs 바이트 스터핑 비교

| 항목 | 비트 스터핑 (Bit-oriented) | 바이트 스터핑 (Byte-oriented) |
|:---|:---|:---|
| **처리 단위** | 비트 (Bit) | 바이트 (Byte/Character) |
| **오버헤드** | 매우 낮음 (1비트 단위 추가) | 상대적으로 높음 (8비트 추가) |
| **투명성** | 완벽함 (모든 비트 패턴 수용) | 텍스트 중심 (바이너리 시 오버헤드 증가) |
| **주요 프로토콜** | HDLC, SDLC, CAN Bus | PPP, SLIP, BISYNC |
| **하드웨어 부하** | 전용 하드웨어 로직 필요 (Shift Reg.) | 소프트웨어 처리에 적합 |

### 과목 융합 관점

- **컴퓨터 구조 (CA)**: 비트 스터핑은 실시간성이 중요하므로 CPU가 처리하지 않고 NIC(네트워크 카드)의 직렬화/역직렬화(SerDes) 칩셋에서 하드웨어 게이트로 즉시 처리한다.
- **보안 (Security)**: 스터핑 로직의 허점을 이용해 비정상적으로 긴 프레임을 보내거나 프레임 경계를 무너뜨리는 'Framing Attack'이 존재한다. 이를 방지하기 위해 최대 프레임 길이 제한(MTU) 검증이 병행되어야 한다.

- **📢 섹션 요약 비유**: 바이트 스터핑이 "단어 단위로 검사하는 교정가"라면, 비트 스터핑은 "글자 획 하나하나를 감시하는 정밀 판독기"와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 임베디드 시리얼 통신(UART) 프레임 설계**: MCU 간에 가변 길이 데이터를 주고받아야 한다. 구분자 없이 보내면 데이터가 밀릴 때 복구가 안 된다. 기술사는 **바이트 스터핑**을 적용한 단순 프로토콜을 설계한다. 데이터 시작은 `0x02(STX)`, 끝은 `0x03(ETX)`으로 정하고, 본문에 해당 값이 나오면 `0x1B(ESC)`를 붙여 데이터 투명성을 확보한다.

2. **시나리오 — 자동차 제어망(CAN Bus) 오류 분석**: CAN 통신 도중 특정 구간에서 에러 프레임이 빈번하게 발생한다. 분석 결과, 비트 스터핑이 너무 자주 일어나 프레임 길이가 길어지면서 타이밍 마진을 초과한 것이 원인이었다. 기술사는 비트 패턴을 분석하여 **스터핑 발생 확률을 줄이는 스크램블링(Scrambling)**을 적용하거나 전송 속도를 조정하여 신뢰성을 높인다.

### 도입 체크리스트
- **데이터의 성격**: 텍스트 위주인가(바이트 방식 가능), 아니면 순수 바이너리/이미지인가(비트 방식 권장)?
- **실시간성 요구**: 하드웨어 가속이 가능한 환경인가? (불가능하면 단순 문자 카운트 방식 고려)
- **오버헤드 허용 범위**: 대역폭이 극도로 좁은가? (그렇다면 비트 스터핑이 유리)

### 안티패턴
- **스터핑 없는 가변 길이 프레임**: "데이터가 대충 이쯤 끝나겠지"라고 짐작하는 설계는 잡음 하나에 전체 통신 세션을 마비시킨다.
- **ESC 문자를 스터핑하지 않음**: 플래그만 스터핑하고 정작 ESC 자체를 스터핑하지 않으면, 데이터에 ESC가 포함될 때 다음 문자가 무조건 데이터로 오인되어 시스템이 꼬이게 된다.

- **📢 섹션 요약 비유**: 포장지를 아끼려다 내용물을 다 쏟는 것(프레임 경계 상실)보다, 약간의 오버헤드를 감수하고 확실한 봉투(스터핑)를 만드는 것이 네트워크 엔지니어의 기본기입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (Raw Stream) | 도입 후 (Framing) | 개선 효과 |
|:---|:---|:---|:---|
| **데이터 투명성** | 특정 값 전송 제한됨 | 어떤 데이터든 전송 가능 | 응용 서비스 자유도 **100%** |
| **에러 복구** | 처음부터 다시 시작 | 해당 프레임만 재전송 | 복구 시간 **80% 이상** 단축 |
| **동기화 유지** | 비트 밀림 시 복구 불가 | 프레임마다 동기 재설정 | 링크 가동률 극대화 |

### 미래 전망
현대 고속 네트워크(Ethernet, PCIe)는 물리적 신호 자체에 강력한 **부호화(8B/10B, 64B/66B)**를 적용하여 데이터 패턴에 관계없이 경계를 식별할 수 있게 진화했다. 하지만 '데이터 투명성'과 '경계 식별'이라는 프레임화의 핵심 철학은 상위 계층의 캡슐화(Encapsulation) 원리로 계승되어 현대 통신의 근간을 유지하고 있다.

### 참고 표준
- **ISO/IEC 13239 (HDLC)**: 비트 스터핑 기반의 전 세계 데이터 링크 표준
- **IETF RFC 1661 (PPP)**: 인터넷 연결을 위한 바이트 스터핑 표준

- **📢 섹션 요약 비유**: 비트의 바다에서 의미 있는 섬(정보)을 찾아내기 위해 등대(플래그)를 세우고 안개(중복 패턴)를 걷어내는 지혜로운 길잡이 기술입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **프레임화**는 수많은 구슬(데이터)을 색깔별로 나누어 상자에 담고, 상자 겉면에 "여기부터 여기까지가 한 세트야"라고 표시하는 거예요.
2. 그런데 구슬 중에 상자 겉면의 표시와 똑같이 생긴 가짜 구슬이 섞여 있으면 헷갈리겠죠?
3. 그래서 가짜 구슬 옆에 "이건 가짜야!"라는 작은 쪽지(스터핑)를 붙여서 받는 사람이 헷갈리지 않게 도와주는 거랍니다!
