# 흐름 제어 및 에러 제어 (Flow & Error Control)

> 1. **본질**: 송신측의 전송 속도가 수신측의 처리 능력을 초과하지 않도록 조절하는 **흐름 제어(Flow Control)**와, 전송 중 손상되거나 분실된 프레임을 탐지하고 복구하는 **에러 제어(Error Control)**의 통합 체계이다.
> 2. **가치**: 수신측 버퍼 오버플로우(Overflow)를 방지하고 데이터의 무결성(Integrity)을 확보하여, 불안정한 물리적 회로 위에서 신뢰할 수 있는 논리적 연결(Reliable Link)을 형성한다.
> 3. **융합**: 데이터 링크 계층(L2)의 핵심 기능인 ARQ(Automatic Repeat Request) 메커니즘을 정의하며, 이는 전송 계층(L4)의 TCP 윈도우 제어 및 혼잡 제어 알고리즘의 기초가 된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: **흐름 제어**는 수신자가 "천천히 보내!"라고 말하는 속도 조절이고, **에러 제어**는 "다시 보내!"라고 말하는 복구 절차이다. 이 둘은 주로 **확인 응답(ACK)**과 **재전송(Retransmission)** 기법을 결합하여 구현된다.

- **💡 비유**: 흐름 제어는 정수기 물을 컵에 담을 때, 물이 넘치지 않게 수도꼭지를 조절하는 것과 같습니다. 에러 제어는 컵에 담긴 물에 먼지가 들어갔을 때(에러), 그 물을 버리고 새 물을 다시 담는(재전송) 것과 같습니다.

- **등장 배경**:
  1. **성능 불일치**: 슈퍼컴퓨터가 보낸 데이터를 구형 단말기가 받으면 처리 속도를 못 따라가 데이터가 유실된다.
  2. **물리적 잡음**: 전송 선로의 노이즈로 인해 비트가 뒤집히거나 프레임 자체가 사라지는 현상이 빈번하다.
  3. **효율적 자원 활용**: 무조건 재전송하는 것이 아니라, 어떤 단위로 어떻게 효율적으로 다시 보낼지에 대한 전략이 필요했다.

- **📢 섹션 요약 비유**: 대화의 상대방이 잘 알아듣고 있는지(에러 제어)와 말이 너무 빠르지는 않은지(흐름 제어)를 끊임없이 확인하며 조절하는 "지능형 대화 규칙"입니다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 흐름 제어 방식: 정지-대기 vs 슬라이딩 윈도우

| 방식 | 동작 원리 | 특징 | 비유 |
|:---|:---|:---|:---|
| **정지-대기 (Stop-and-Wait)** | 하나 보내고 ACK 올 때까지 대기 | 구현은 쉬우나 전송 효율(Throughput)이 매우 낮음 | 한 입 먹여주고 씹는 거 확인 후 다음 입 주기 |
| **슬라이딩 윈도우 (Sliding Window)** | ACK 없이 여러 개를 한꺼번에 전송 | 송수신측 버퍼(Window) 크기를 맞춰 효율 극대화 | 뷔페에서 접시에 여러 개를 담아 한꺼번에 먹기 |

---

### 에러 제어 (ARQ: Automatic Repeat Request) 방식 비교

손상된 데이터를 어떻게 복구할 것인가에 대한 3대 전략이다.

```text
 [ ARQ 메커니즘 시각화 ]
 
 1. Stop-and-Wait ARQ:
    송신: [P1] ──▶ 수신 (에러!)
    송신: (타임아웃) ──▶ [P1 다시 전송]
    
 2. Go-Back-N ARQ (GBN):
    송신: [P1, P2, P3, P4, P5] ──▶ (P3 에러 발생!)
    송신: [P3, P4, P5] (P3 이후 모든 프레임 싹 다 다시 전송)
    
 3. Selective Repeat ARQ (SR):
    송신: [P1, P2, P3, P4, P5] ──▶ (P3 에러 발생!)
    송신: [P3] (에러 난 P3만 쏙 골라서 다시 전송)
```

**[다이어그램 해설]** **GBN**은 수신측에서 순서가 틀린 프레임을 아예 받지 않으므로 구현이 단순하지만, 멀쩡한 프레임까지 다시 보내는 낭비가 있다. **SR**은 수신측에 재정렬 버퍼가 필요해 구현이 복잡하지만, 필요한 것만 다시 보내므로 대역폭 효율이 가장 높다.

---

### ARQ 핵심 구성 요소

1. **순서 번호 (Sequence Number)**: 각 프레임에 번호를 매겨 중복 수신이나 분실을 감지함.
2. **타임아웃 (Timeout)**: 일정 시간 동안 응답이 없으면 분실로 간주하고 자동 재전송.
3. **부정 응답 (NAK/NACK)**: "데이터가 망가졌으니 다시 줘"라고 명시적으로 요청.
4. **긍정 응답 (ACK)**: "잘 받았어, 다음 거 줘"라고 통보.

- **📢 섹션 요약 비유**: 물건을 배달할 때, 하나씩 확인하고 다음 걸 보내느냐(Stop-and-Wait), 일단 한 박스 보내고 빠진 것만 다시 보내느냐(SR)의 물류 전략 차이입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### ARQ 방식별 정량적 비교

| 비교 항목 | Stop-and-Wait | Go-Back-N | Selective Repeat |
|:---|:---|:---|:---|
| **전송 효율** | 매우 낮음 | 우수 (Window 크기 비례) | **최상** |
| **수신측 버퍼** | 1개 (단순) | 1개 (단순) | **다수 (재정렬 버퍼 필요)** |
| **순서 번호 범위** | 0, 1 (1비트) | $2^n - 1$ | $2^{n-1}$ |
| **복잡도** | 최저 | 중간 | 높음 |
| **주 사용처** | 초기 모뎀, 단순 센서 | 초기 HDLC, TCP 기본 | **현대 고속 통신 (Wi-Fi, 5G)** |

### 과목 융합 관점

- **전송 계층 (L4)**: TCP(Transmission Control Protocol)는 L2의 슬라이딩 윈도우 원리를 전역 네트워크로 확장하여 구현한다. 다만, 네트워크 혼잡 상황에 따라 윈도우 크기를 동적으로 바꾸는 '혼잡 제어'가 추가된다.
- **알고리즘**: 수신된 프레임을 원래 순서대로 조립하는 과정에서 힙(Heap)이나 우선순위 큐(Priority Queue) 같은 자료구조가 효율적인 SR 구현을 위해 사용된다.

- **📢 섹션 요약 비유**: GBN이 "하나라도 틀리면 군기가 빠졌다며 전체 기합을 주는 교관"이라면, SR은 "실수한 사람만 불러서 따로 가르치는 친절한 선생님"과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 위성 통신 지연 문제**: 지연 시간(Latency)이 매우 긴 위성 통신망에서 Stop-and-Wait을 쓰면, 응답 기다리는 데 시간을 다 써서 속도가 거의 안 나온다. 기술사는 **슬라이딩 윈도우**와 **Selective Repeat ARQ**를 도입하도록 설계를 변경한다. 윈도우 크기를 위성 링크의 대역폭-지연 곱(BDP)만큼 충분히 키워, 응답이 오기 전에도 끊임없이 데이터를 밀어넣어 효율을 극대화한다.

2. **시나리오 — 초소형 IoT 센서의 전력 절감**: 배터리 수명이 10년 이상 가야 하는 센서가 전파 환경이 좋은 유선 선로에 연결되어 있다. 기술사는 복잡한 SR 대신 가장 단순한 **Stop-and-Wait**을 선택한다. 연산량이 적어 CPU 부하를 낮추고 전력 소모를 최소화하는 것이 속도보다 더 중요하기 때문이다.

### 도입 체크리스트
- **BER (Bit Error Rate)**: 에러가 잦은 선로인가? (잦다면 GBN은 비효율적, SR 필수)
- **RTT (Round Trip Time)**: 왕복 시간이 긴가? (길면 윈도우 크기를 대폭 키워야 함)
- **수신단 메모리**: 수신 장비가 수 메가바이트의 재정렬 버퍼를 가질 여유가 있는가?

### 안티패턴
- **Window Size를 1로 고정**: 고속 광랜 환경에서 윈도우를 1로 쓰면 1Gbps 선로라도 실제 속도는 수 kbps로 떨어진다.
- **타임아웃 설정을 너무 짧게**: 네트워크 지연으로 ACK가 조금만 늦게 와도 중복 데이터를 계속 보내게 되어 망 전체에 혼잡을 야기한다.

- **📢 섹션 요약 비유**: 고속도로 통행료를 한 대 지날 때마다 내게 할지(Stop-and-Wait), 하이패스로 한꺼번에 정산할지(Sliding Window) 결정하는 것이 시스템 성능의 핵심입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (에러 방치) | 도입 후 (Flow/Error Control) | 개선 효과 |
|:---|:---|:---|:---|
| **전송 신뢰성** | 에러 시 데이터 폐기 | 100% 무결성 복구 | 데이터 손실률 **0%** 달성 |
| **처리량 (Throughput)** | 낮은 효율 | 선로 대역폭 근접 활용 | 효율 **5~10배 향상** |
| **자원 보호** | 버스트 트래픽에 시스템 다운 | 버퍼 안정성 유지 | 시스템 가용성 확보 |

### 미래 전망
현대 통신은 하드웨어 성능이 좋아짐에 따라 거의 대부분 **Selective Repeat**와 **고정밀 슬라이딩 윈도우**를 기본으로 장착하고 있다. 최근에는 AI를 활용하여 네트워크의 RTT와 BER을 실시간 예측하고, 최적의 윈도우 크기와 재전송 전략을 스스로 결정하는 **지능형 적응형 ARQ** 기술이 6G 표준화 과정에서 활발히 논의되고 있다.

### 참고 표준
- **IEEE 802.3**: 이더넷의 흐름 제어 (PAUSE 프레임) 규격
- **ISO 4335**: HDLC의 흐름 및 에러 제어 절차 표준

- **📢 섹션 요약 비유**: 물결치는 바다 위에서 배가 침몰하지 않게 평형을 잡고(흐름 제어), 빠진 짐을 다시 건져 올리는(에러 제어) 행위는 통신의 영원한 숙명이자 완성입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **흐름 제어**는 엄마가 밥을 먹여줄 때 아이가 다 씹기도 전에 또 주지 않도록, 아이의 입 모양을 보며 속도를 맞추는 거예요.
2. **에러 제어**는 밥을 먹다가 콩을 떨어뜨리면(에러), 엄마가 다시 콩을 집어서 입에 넣어주는 것과 같답니다.
3. 이렇게 서로 신호를 주고받아야 밥을 맛있고 안전하게(신뢰성) 다 먹을 수 있어요!
