# 4B/5B, 8B/10B 부호화 (Block Coding)

> 1. **본질**: m비트의 데이터 블록을 n비트(n > m)의 부호어(Codeword)로 변환하여 전송함으로써, 신호 내에 충분한 전이(Transition)를 확보하고 **DC 균형(DC Balance)**을 유지하는 논리적 부호화 체계이다.
> 2. **가치**: 0이나 1이 길게 이어지는 현상을 구조적으로 차단하여 수신측의 동기화를 보장하며, 사용되지 않는 부호어 조합을 통해 물리 계층에서의 제어 신호(Control Symbol) 및 에러 탐지 기능을 제공한다.
> 3. **융합**: 고속 이더넷(Fast Ethernet)부터 기가비트 이더넷, USB 3.0, PCIe, 광채널(Fibre Channel) 등 현대 고속 직렬 통신(SerDes) 인터페이스의 성능과 신뢰성을 지탱하는 핵심 표준 기술이다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: **블록 부호화(Block Coding)**는 데이터를 일정한 크기의 덩어리(Block)로 나누고, 이를 미리 약속된 더 큰 덩어리로 바꾸어 보내는 방식이다. **4B/5B**는 4비트를 5비트로, **8B/10B**는 8비트를 10비트로 변환한다.

- **💡 비유**: 4글자짜리 암호를 보낼 때, 중간에 엉뚱한 글자 하나를 섞어 5글자로 만드는 것과 같습니다. 이 '섞인 글자' 덕분에 암호가 너무 단순해지는 것을 막고(동기화), 가짜 암호(에러)를 쉽게 구별할 수 있게 됩니다.

- **등장 배경**:
  1. **NRZ의 한계**: 고속 통신에서 NRZ를 쓰면 0이나 1이 연속될 때 동기화가 깨진다. 이를 데이터 자체를 가공하여 해결하고자 했다.
  2. **DC 편중 문제**: 전압이 한쪽으로 쏠리면 장비가 손상될 수 있다. 8B/10B는 0과 1의 개수를 맞춰 이 문제를 완벽히 해결한다.
  3. **대역폭과 신뢰성의 타협**: 맨체스터 방식은 대역폭을 2배(100% 오버헤드)나 쓰지만, 블록 부호화는 20~25%의 오버헤드만으로도 유사한 동기화 성능을 낸다.

- **📢 섹션 요약 비유**: 날것의 데이터(Raw Data)를 안전한 포장지(Extra Bits)로 감싸서, 험난한 통신 경로(Physical Medium)에서도 내용물이 변질되지 않게 보호하는 진보된 포장 기술입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 4B/5B 부호화 메커니즘 (Fast Ethernet)

4B/5B는 16가지의 데이터 조합을 32가지의 부호어 중 "0이 3개 이상 연속되지 않는" 최적의 16개 조합에 매핑한다.

```text
 [ 4B/5B 매핑 예시 ]
  Data (4-bit)  │  Code (5-bit)  │  설명
 ───────────────┼────────────────┼────────────────────────────
  0000 (0)      │  11110         │  0이 연속되지 않게 배치
  0001 (1)      │  01001         │  전이(Transition) 확보
  ...           │  ...           │
  1111 (F)      │  10101         │  
 ───────────────┼────────────────┼────────────────────────────
  제어 기호      │  11000 (J)     │  프레임 시작(Start of Stream)
  (Control)     │  10001 (K)     │  
```

**[다이어그램 해설]** 4B/5B는 4비트 데이터를 5비트로 늘림으로써 총 32개의 조합을 확보한다. 이 중 가장 '건강한' 16개를 골라 데이터에 할당하고, 나머지는 **JK(시작), TR(끝), I(대기)** 등 네트워크 제어용 심볼로 활용한다. 만약 정의되지 않은 나머지 10여 개의 조합이 수신되면 즉시 에러로 처리한다. 오버헤드는 25%이며, 보통 NRZ-I나 MLT-3와 결합하여 물리 계층 신호를 완성한다.

+++

### 8B/10B 부호화와 불일치(Running Disparity) 제어

8B/10B(IBM 개발)는 더욱 정교하게 0과 1의 개수를 조절하여 **DC 밸런스**를 맞춘다.

```text
 [ 8B/10B의 3B/4B + 5B/6B 결합 구조 ]
 
   8-bit Data ──▶ [ 5B/6B Encoder ] ──┬──▶ 10-bit Codeword
                └─ [ 3B/4B Encoder ] ──┘
                          │
              ┌───────────┴───────────┐
              │  Running Disparity     │ ◀── 핵심 제어 로직
              │  (현재까지 1이 더 많은가? 0이 더 많은가?)
              └───────────────────────┘
```

**[다이어그램 해설]** 8B/10B는 데이터를 3비트와 5비트로 쪼개 각각 부호화한 뒤 합친다. 이때 가장 중요한 기술이 **런닝 디스파리티(Running Disparity)**이다. 현재까지 보낸 비트 중 1이 많으면 다음에는 0이 더 많은 부호어를 선택하고, 반대면 1이 많은 부호어를 선택한다. 결과적으로 장기적인 관점에서 0과 1의 개수는 정확히 일치하게 되어, 전선에 흐르는 평균 전압은 완벽히 0V가 된다.

+++

### 블록 부호화의 3대 가치

1. **Self-Clocking**: 어떤 데이터가 오더라도 0이나 1이 일정 개수 이상 연속되지 않으므로 수신측이 박자를 놓치지 않는다.
2. **DC Balance**: 8B/10B 등을 통해 신호의 전압 편중을 막아 변압기 및 광센서의 오동작을 방지한다.
3. **Control Symbol**: 데이터와 섞이지 않는 별도의 제어 신호(K-코드)를 통해 프레임의 경계를 명확히 구분한다.

- **📢 섹션 요약 비유**: 짐을 실을 때 왼쪽과 오른쪽의 무게를 실시간으로 재가며(Disparity) 배치를 바꾸어, 배(통신 선로)가 한쪽으로 기울어지지 않게 관리하는 일등 항해사와 같습니다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### 4B/5B vs 8B/10B vs Manchester

| 비교 항목 | Manchester | 4B/5B | 8B/10B |
|:---|:---|:---|:---|
| **오버헤드** | 100% (매우 높음) | 25% | 25% |
| **효율성** | 낮음 (10Mbps용) | 높음 (100Mbps용) | 최적 (1Gbps 이상) |
| **DC 밸런스** | 완벽함 | 불완전 (DC 성분 남음) | **완벽함 (RD 제어)** |
| **동기화** | 비트 단위 강제 전이 | 패턴 매핑으로 확보 | 패턴 매핑으로 확보 |
| **주 사용처** | 10Base-T | 100Base-TX, FDDI | GbE, USB 3.0, PCIe |

### 과목 융합 관점

- **컴퓨터 구조 (CA)**: 고속 직렬 버스인 PCIe 3.0부터는 오버헤드를 줄이기 위해 128B/130B 부호화를 사용한다. 이는 1.5%의 극소량 오버헤드만으로도 동기화를 유지하는 초고도화된 블록 부호화 기술이다.
- **전자 회로**: 8B/10B를 통해 구현된 DC Balance는 AC 커플링(AC Coupling) 설계를 가능하게 하여, 서로 다른 전압 시스템을 가진 칩 간의 안전한 통신을 돕는다.

- **📢 섹션 요약 비유**: 맨체스터가 1kg 물건을 포장하는 데 1kg 포장지를 쓰는 낭비라면, 블록 부호화는 가벼운 뽁뽁이(25% 비트)만으로도 같은 보호 효과를 내는 효율적인 포장법입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 기가비트 스위치의 광모듈 발열**: 광모듈(SFP)에서 데이터 전송 시 0과 1의 불균형으로 인해 레이저 다이오드에 과도한 바이어스 전류가 흘러 발열이 심해진다. 이를 해결하기 위해 **8B/10B** 로직이 정상 작동하는지 확인하고, RD(Running Disparity) 제어를 통해 광출력의 평균 레벨을 안정화하여 수명을 연장한다.

2. **시나리오 — USB 3.0 인식 불량**: 고속 데이터 전송 중 간헐적인 연결 끊김이 발생한다. 분석 결과, 물리 계층의 **8B/10B** 복호화 과정에서 정의되지 않은 'Invalid Symbol'이 빈번하게 감지되었다. 이는 선로 노이즈가 블록 부호화의 규칙을 깨뜨린 것이므로, 케이블 차폐를 강화하거나 이퀄라이저(Equalizer) 설정을 조정해야 한다.

### 도입 체크리스트
- **대역폭 오버헤드 감당 가능 여부**: 물리 속도의 20% 이상을 제어 비트로 할당할 여유가 있는가?
- **DC Balance 필수성**: 선로 중간에 변압기나 광 분리기가 존재하는가? (존재한다면 8B/10B 이상 필수)
- **제어 심볼 요구 사항**: 데이터 외에 'IDLE', 'ERROR', 'START' 등의 상태를 물리적으로 구분해야 하는가?

### 안티패턴
- **8B/10B 사용 시 Scrambling 누락**: 블록 부호화가 패턴을 섞어주긴 하지만, 특정 데이터가 반복될 때 발생하는 전자파 간섭(EMI)까지 완벽히 잡지는 못한다. 따라서 고속 설계에서는 8B/10B 전 단계에서 스크램블링을 병행하는 것이 정석이다.
- **오버헤드 간과**: 1Gbps 통신을 위해 8B/10B를 쓰면 실제 선로에는 1.25Gbps의 신호가 흘러야 한다. 하드웨어 설계 시 이 25%의 속도 증가분을 계산에 넣지 않으면 타이밍 위반이 발생한다.

- **📢 섹션 요약 비유**: 포장지가 두꺼워지면(오버헤드) 튼튼하긴 하겠지만, 트럭(대역폭)에 실을 수 있는 알맹이 양이 줄어든다는 점을 항상 계산해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (NRZ) | 도입 후 (8B/10B) | 개선 효과 |
|:---|:---|:---|:---|
| **신뢰성** | 패턴에 따른 동기 상실 빈번 | 모든 데이터에서 동기 유지 | 링크 안정성 **99.99%** 달성 |
| **정확도** | 에러 탐지 불가 | 규칙 위반 심볼 즉시 탐지 | 물리 계층 에러 탐지 능력 확보 |
| **효율** | 100% 오버헤드(Manchester) | 25% 오버헤드로 성능 유지 | 유효 대역폭 **75% 향상** |

### 미래 전망
8B/10B는 수십 년간 고속 통신의 제왕이었으나, 10Gbps 이상의 초고속 영역에서는 25%의 오버헤드조차 아까워지기 시작했다. 이에 따라 최신 표준인 **128B/130B(PCIe 3.0+)**나 **64B/66B(10G Ethernet)**처럼 오버헤드를 2~3%로 극단적으로 낮춘 고효율 블록 부호화가 주류가 되고 있다.

### 참고 표준
- **IEEE 802.3z**: 기가비트 이더넷의 8B/10B 규격
- **ANSI X3.230**: 파이버 채널(Fibre Channel)의 부호화 표준

- **📢 섹션 요약 비유**: 처음에는 튼튼한 나무 상자(8B/10B)로 포장하다가, 이제는 얇지만 질긴 특수 비닐(64B/66B)로 포장하여 더 많은 짐을 싣는 방향으로 진화하고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **런닝 디스파리티 (Running Disparity)**: 8B/10B에서 DC 균형을 맞추는 핵심 알고리즘.
- **K-코드 (K-Codes)**: 블록 부호화에서 제공하는 특수 제어 심볼 체계.
- **64B/66B 부호화**: 10기가비트 이더넷에서 사용하는 고효율 블록 부호화.
- **Scrambling (스크램블링)**: 블록 부호화와 융합하여 EMI를 줄이는 필수 기술.

## 👶 어린이를 위한 3줄 비유 설명
1. **블록 부호화**는 4개짜리 레고 블록을 전송할 때, 더 튼튼하게 연결하려고 가짜 블록 하나를 더 끼워 5개로 만드는 것과 같아요.
2. 이 가짜 블록 덕분에 블록들이 길게 일렬로 늘어서지 않고 예쁜 모양을 유지해서, 받는 사람이 훨씬 쉽게 잡을 수 있답니다.
3. 전기가 한쪽으로 쏠리지 않게 무게 중심을 맞춰주는 아주 똑똑한 포장 규칙이라고 생각하면 돼요!
