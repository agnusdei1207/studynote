# 681-689. SSL/TLS 프로토콜과 핸드셰이크

> **핵심 인사이트**: HTTPS의 심장인 TLS는 느리고 복잡한 공개키 암호화를 '통성명(Handshake)'에만 사용하고, 실제 대화는 빠른 대칭키로 전환하는 영리한 전략을 쓴다. 특히 최신 버전인 TLS 1.3은 이 과정을 단 한 번의 왕복(1-RTT)으로 줄여 보안과 속도를 모두 잡았다.

---

## Ⅰ. SSL / TLS 개요
전송 계층과 응용 계층 사이에서 데이터를 암호화하는 표준 프로토콜입니다. (SSL은 구식 명칭이며 현재는 **TLS**가 표준)

* **제공 기능**: 기밀성(암호화), 무결성(HMAC), 인증(인증서).

---

## Ⅱ. TLS 핸드셰이크 (Handshake)
통신 시작 전 암호 알고리즘과 키를 협상하는 과정입니다.

1. **Client Hello**: "나 접속하고 싶어. 내가 쓸 수 있는 암호 목록(Cipher Suite)이야."
2. **Server Hello**: "반가워. 암호는 이걸로(예: AES-GCM) 하자. 내 **인증서** 받아."
3. **Key Exchange**: 서로의 정보를 섞어 **세션키(대칭키)**를 만듭니다. (디피-헬만 등 활용)
4. **Finished**: "이제부터 이 세션키로 암호화해서 대화하자!"

---

## Ⅲ. 주요 기술 및 최신 트렌드

### 1. 전방향 안전성 (PFS, Perfect Forward Secrecy)
* **개념**: 나중에 서버의 개인키가 해커에게 털리더라도, **과거에 가로챘던 통신 내용은 복호화할 수 없게** 하는 성질입니다.
* **방법**: 매 접속마다 일회용 세션키를 생성하여 폐기하는 방식(ECDHE)을 사용합니다.

### 2. TLS 1.3의 혁신
* **성능**: 핸드셰이크 과정을 2-RTT에서 **1-RTT**로 단축. (재접속 시 0-RTT 지원)
* **보안**: 취약한 암호 방식(RSA 키 교환 등)을 완전히 제거하고 AEAD(GCM 모드 등)만 허용.

### 3. ECH (Encrypted Client Hello)
* **SNI (Server Name Indication)**: 접속하려는 사이트 주소가 평문으로 노출되는 약점을 해결하기 위해, 첫 패킷부터 암호화하여 국가 검열이나 엿보기를 방어하는 기술입니다.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[TLS 1.3 핸드셰이크 흐름]

  [ Client ] ───── ( Client Hello + Key Share ) ────> [ Server ]
                                                         │
  [ Client ] <──── ( Server Hello + Key Share ) ───── [ Server ]
                                                         │
  [ Client ] <========= ( 암호화 데이터 전송 ) ========> [ Server ]
```

📢 **섹션 요약 비유**: **TLS 핸드셰이크**는 두 나라의 외교관이 만나 비밀 회담을 하기 전, 어떤 암호 체계를 쓸지 정하고 신분증(인증서)을 확인하는 '사전 조율' 과정입니다. **PFS**는 매일 회담이 끝나면 암호 장부를 불태워버려서, 훗날 비밀번호가 노출되어도 예전 대화 내용을 아무도 모르게 하는 '철저한 보안 수칙'입니다. **TLS 1.3**은 복잡한 의전 절차를 생략하고 만나자마자 본론으로 들어가는 '속전속결 실무형 회담'과 같습니다.
