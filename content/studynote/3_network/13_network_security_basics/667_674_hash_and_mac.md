# 667-674. 해시 함수와 메시지 인증(MAC)

> **1. 본질**: 임의 길이의 메시지를 고정 길이의 축약값으로 변환하는 **해시 함수(Hash Function)**의 일방향성과 압축성을 기반으로, 비밀키를 결합한 **메시지 인증 코드(MAC, Message Authentication Code)**를 통해 데이터 무결성과 송신자 인증을 동시에 달성하는 암호학적 프로토콜입니다.
> **2. 가치**: 네트워크 전송 구간에서 데이터 위변조를 탐지(무결성)하고, **HMAC (Hash-based MAC)**와 같은 알고리즘을 통해 비밀키를 공유한 당사자만 메시지를 생성할 수 있음을 보장하여 ** Replay Attack(재전송 공격)** 및 위조를 방지합니다.
> **3. 융합**: OSI 7계층의 전송 계층(TCP/IP) 및 응용 계층(HTTPS/IPsec)에서 **TLS (Transport Layer Security)** 핸드셰이크 및 데이터 교환 검증에 핵심적으로 사용되며, 블록체인의 블록 연결(Merkle Tree) 및 비밀번호 저장(BCrypt 등) 시스템의 기초가 됩니다.

+++

### Ⅰ. 개요 (Context & Background)

해시 함수(Hash Function)는 암호학의 핵심 primitive로서, 가변 길이의 입력 데이터(M)를 받아 고정된 길이의 출력 값(h)으로 매핑하는 함수입니다. 수식으로는 `h = H(M)`으로 표현하며, **'지문(Fingerprint)'** 또는 **'다이제스트(Digest)'**라고 불립니다. 일반적인 암호화 알고리즘과 달리 **일방향성(One-wayness)**을 가지므로, 해시값을 알더라도 원본 메시지를 복구하는 것은 계산상 불가능해야 합니다. 이러한 특성 때문에 무결성 검증과 비밀번호 저장에 필수적으로 사용됩니다.

**💡 비유**
해시 함수는 거대한 소설책(임의 길이 데이터)을 그 책의 고유한 ISBN 번호(고정 길이 해시값) 하나로 압축하여 표현하는 것과 같습니다. 소설의 내용이 쉼표 하나만 바뀌어도 ISBN 번호는 완전히 달라지기 때문에, ISBN 번호만 보고 책의 내용이 위조되었는지 즉시 확인할 수 있습니다.

**등장 배경**
1.  **기존 한계**: 초기 네트워크 보안은 메시지 암호화(기밀성)에 집중되었으나, 중간자 공격자가 암호문을 임의로 수정하거나 재전송하는 경우 이를 탐지할 수 없는 문제가 존재했습니다.
2.  **혁신적 패러다임**: 1970년대 후반부터 시작된 암호학적 해시 함수 연구(데이터 압축을 위한 것이 아닌 보안을 위한)는 MD5, SHA-1을 거쳐 SHA-2/3로 발전하며 '무결성'이라는 새로운 보안 축을 제공했습니다.
3.  **현재의 비즈니스 요구**: 클라우드 및 블록체인 환경에서 '신뢰'의 최소 단위로 작용하며, **PCI-DSS** 등 보안 규정에서 비밀번호 저장 시 반드시 솔트(Salt)와 해시를 사용하도록 강제하고 있습니다.

**📢 섹션 요약 비유**
해시 함수는 복잡한 미로(원본 데이터)의 입구와 출구를 연결하는 '순간 이동 장치'와 같습니다. 들어가는 길(해시화)은 쉽고 빠르지만, 나오는 길(역추적/Pre-image Attack)은 존재하지 않아 미로를 다시 설계하지 않는 한 원래 입구를 찾을 수 없습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

해시 함수와 MAC의 내부는 수학적 비트 연산으로 가득 찬 복잡한 구조를 가집니다. 특히 **SHA-2 (Secure Hash Algorithm 2)** 계열과 같은 현대적 알고리즘은 **Merkle-Damgård 구조** 혹은 더 강화된 구조를 사용하여 충돌 저항성을 확보합니다.

#### 1. 구성 요소 및 동원 요소

| 요소명 | 역할 | 내부 동작 및 프로토콜 | 주요 파라미터/비유 |
|:---|:---|:---|:---|
| **Compression Function** | 메시지 블록을 압축하는 핵심 엔진 | 입력된 블록과 이전 상태(Chaining Value)를 비트 논리 연산(AND, OR, XOR, SHIFT)으로 섞어 새로운 상태를 생성함. | `f(CV_i, Block_i) -> CV_i+1` <br>*(고기 분쇄기에 비유)* |
| **IV (Initialization Vector)** | 해시 연산의 시작값 | 알고리즘에 미리 정의된 고정된 상수 값. 첫 번째 블록 연산에 입력됨. | SHA-256 IV: 0x6a09e667... <br>*(첫 물 붓는 양)* |
| **Salt (솔트)** | Rainbow Table 공격 방지 | 원본 데이터(비번) 앞뒤에 붙는 무작위 문자열. 사용자마다 고유값을 가져야 함. | Length: 16~32Bytes <br>*(요리할 때 넣는 장물)* |
| **Secret Key (비밀키)** | MAC 생성을 위한 공유 값 | HMAC 생성 시 해시 함수 입력에 포함되는 비밀 값. 공격자가 절대 알면 안 됨. | Length: Hash Block Size (SHA-2: 512/1024bit) |
| **Padding** | 블록 크기 맞춤 | 마지막 블록이 정해진 크기(예: 512bit)가 되도록 1과 0을 채우고 길이 정보를 붙이는 과정. | MD-Strengthening <br>*(빈 곳 채우기)* |

#### 2. 해시 함수 구조 다이어그램 (SHA-2 가상화)

아래는 **Merkle-Damgård 구조**를 기반으로 한 일반적인 해시 함수의 파이프라인입니다. 데이터는 블록 단위로 나뉘어 처리되며, 이전 결과값이 다음 블록의 연산에 영향을 주어 **눈사태 효과(Avalanche Effect)**를 유발합니다.

```ascii
                                   [ 해시 함수 파이프라인 (Hash Pipeline) ]

  원본 메시지 (Message)
  ──────────────────────────────────────────────────────────────────────
  | Block 1 (512bit) | Block 2 (512bit) | ... | Block N (512bit) + Pad |
  ──────────────────────────────────────────────────────────────────────
         │                  │                          │
         ▼                  ▼                          ▼
  ┌──────────────┐    ┌──────────────┐         ┌──────────────┐
  │  IV (초기값) │    │  Chaining    │         │  Chaining    │
  │   + Logic    │───>│  Value(CV)   │────...──>│  Value(CV)   │
  │              │    │   + Logic    │         │   + Logic    │
  └──────────────┘    └──────────────┘         └──────────────┘
         │                                        │
         └────────────> Feed Forward <────────────┘
                           │
                           ▼
                  ┌─────────────────────┐
                  │  Final Hash Value   │
                  │  (예: 256 bits)     │
                  └─────────────────────┘
```

**[다이어그램 해설]**
1.  **분할 및 패딩**: 메시지는 512비트나 1024비트 크기의 블록으로 나뉘고, 마지막 블록은 패딩(Padding) 과정을 거쳐 정해진 크기를 맞춥니다.
2.  **압축 함수(Compression Function)**: 각 블록은 이전 단계에서 넘어온 체이닝 값(IV 또는 중간 결과)과 함께 비트 연산을 수행합니다. 이때 1비트의 입력 변화가 출력에 거대한 변화를 일으키도록 설계된 비혼형(Confusion)과 확산(Diffusion) 과정이 수행됩니다.
3.  **체이닝(Chaining)**: 블록 1의 연산 결과가 블록 2의 입력으로 들어가는 방식입니다. 따라서 뒤쪽 데이터의 해시값은 앞쪽 모든 데이터의 영향을 받습니다.

#### 3. HMAC (Hash-based MAC) 심층 분석

단순히 `H(Key || Message)` 방식은 **Length Extension Attack**에 취약합니다. 이를 해결하기 위해 **HMAC**은 다음과 같은 이중 해싱 구조를 사용합니다. (RFC 2104 표준)

**핵심 알고리즘 코드 및 수식**
```python
# HMAC 의사 코드 (Pseudo-Code)
def hmac_sha256(key, message):
    block_size = 64  # SHA-256의 블록 크기 (bytes)
    
    # 1. 키 처리: 키가 블록보다 길면 해시로 압축, 짧으면 0으로 패딩
    if len(key) > block_size:
        key = sha256(key)
    else:
        key = key.ljust(block_size, b'\x00')
        
    # 2. 내부/외부 패딩 생성 (XOR 연산)
    o_key_pad = bytes(x ^ 0x5c for x in key)  # Outer Pad
    i_key_pad = bytes(x ^ 0x36 for x in key)  # Inner Pad
    
    # 3. 해시 계산: Hash( Key_out || Hash( Key_in || Message ) )
    inner_hash = sha256(i_key_pad + message)
    return sha256(o_key_pad + inner_hash)
```

#### 4. HMAC 구조 다이어그램

```ascii
             [ HMAC 동작 구조 (RFC 2104) ]

  비밀키 (Secret Key) ───┬── [ 0x36 iPad XOR ] ─┐
                         │                      ▼
  Message (Data) ────────┴──> [   Hash   ] ───> [ Inner Hash ]
                                                      │
                         ┌────────────────────────────┘
                         ▼
  비밀키 (Secret Key) ───┴── [ 0x5c oPad XOR ] ─┐
                         │                      ▼
                         └──────────> [   Hash   ] ───> [ HMAC Tag ]
```

**[다이어그램 해설]**
1.  **Inner Pad (0x36)**: 비밀키와 `0x36`으로 XOR 된 값 뒤에 메시지를 붙여 해시합니다. 이는 내부 잠금 장치를 거는 과정입니다.
2.  **Outer Pad (0x5c)**: 위에서 나온 해시값 뒤에, 비밀키와 `0x5c`으로 XOR 된 값을 붙여 다시 한번 해시합니다. 외부 잠금 장치를 거는 과정입니다.
3.  **안정성**: 이 구조는 공격자가 해시 함수 자체의 약점을 이용하더라도 비밀키를 알 수 없게 만듭니다.

**📢 섹션 요약 비유**
해시 함수의 **Compression Function**은 '고기 분쇄기'와 같아서, 어떤 고기(데이터)를 넣어도 나오는 것은 다진 고기(해시값) 형태입니다. **HMAC**은 고기를 넣기 전에 특수 양념(비밀키)을 미리 갈아서 분쇄기 내부를 코팅하고, 나온 다진 고기에 다시 그 양념을 발라 겉을 감싸는 복잡한 과정이라, 양념 비율을 모르는 누구도 맛을 똑같이 낼 수 없습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

해시 기술은 단순히 보안을 넘어 시스템 전반의 성능과 신뢰성에 영향을 미칩니다. 특히 다른 암호 기술과의 융합 지점에서 결정적인 역할을 수행합니다.

#### 1. 심층 기술 비교: Digital Signature vs MAC

| 비교 항목 | MAC (Message Authentication Code) | Digital Signature (전자서명) |
|:---|:---|:---|
| **기반 기술** | 대칭키 암호 (Hash + Secret Key) | 비대칭키 암호 (Hash + Private Key) |
| **핵심 속성** | 무결성 + **인증(Authentication)** | 무결성 + **인증** + **부인 방지(Non-repudiation)** |
| **속도 (TPS)** | 매우 빠름 (단순 Hash 연산) | 느림 (모듈러 지수 연산 포함) |
| **키 관리 복잡도** | 높음 (1:N 통신 시 N개의 키 필요) | 낮음 (공개키 1쌍으로 해결) |
| **대표 프로토콜** | **HMAC-SHA256**, CMAC | RSA-PSS, ECDSA |
| **주요 사용처** | TLS 1.2/1.3 (Record Protocol), API 토큰 검증 | 전자 문서 서명, 블록체인 트랜잭션 서명 |

#### 2. 타 과목 융합 관점

**[OS/DB 관점]: Database Indexing**
해시 함수의 빠른 검색 특성(O(1))을 활용하여 데이터베이스의 **Hash Index** 구현에 사용됩니다. 하지만 암호학적 해시(SHA-2)는 속도가 느리므로 DB 인덱싱에는 **MurmurHash**나 **CityHash**와 같은 비암호적 해시함수(Non-cryptographic Hash)를 주로 사용합니다. 이는 보안보다 성능이 중요한 영역에서의 기술 선택입니다.

**[네트워크 관점]: SSH/TLS Handshake**
SSH 프로토콜에서는 클라이언트와 서버가 **Diffie-Hellman (DH)** 키 교환을 수행한 후, 그 결과값으로 생성된 세션 키를 해시 함수(SHA-256)를 통해 **Session ID** 및 **Exchange Hash**를 생성하여 이를 **Fingerprint**로 사용합니다. 사용자는 이 해시값을 눈으로 확인하여 'Man-in-the-Middle Attack'을 방지합니다.

**📢 섹션 요약 비유**
MAC은 **비밀 번호를 사용한 서명 도장**입니다. (도장을 가진 사람만 찍을 수 있음, 서로 비밀번호 공유 필요). 전자서명은 **관공서에 등록된 인감 도장**입니다. (내 도장이지만 공개된 인감증명서로 내가 찍은 것임을 제3자도 증명 가능, 복사가 불가능한 증명서). DB 해싱은 **도서관의 분류 번호표**와 같아 책 제목을 기억하지 않아도 번호 하나로 책을 즉시 찾아줍니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무 시스템에서 해시 함수를 선정하고 적용하는 과정은 보안과 성능의
