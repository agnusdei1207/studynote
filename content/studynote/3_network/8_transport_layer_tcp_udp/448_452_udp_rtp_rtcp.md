# 448-452. UDP 헤더와 실시간 전송 프로토콜(RTP)

> **핵심 인사이트**: UDP는 복잡한 기능을 덜어낸 '초경량 배달부'이다. UDP 자체만으로는 부족한 실시간 성질(순서, 시간)을 보완하기 위해 RTP라는 전용 가방을 씌워 동영상과 음성 데이터를 끊김 없이 전달한다.

---

## Ⅰ. UDP (User Datagram Protocol) 헤더 구조
UDP 헤더는 오직 **8바이트**로 구성되어 매우 가볍습니다.

* **Source / Destination Port (각 16bit)**: 앱 식별 번호.
* **Length (16bit)**: 헤더와 데이터를 합친 총 길이.
* **Checksum (16bit)**: 데이터 무결성 검사용. (선택 사항이지만 대부분 사용)

---

## Ⅱ. UDP의 활용과 장점
* **오버헤드 최소화**: 20바이트인 TCP 헤더에 비해 훨씬 가벼워 전송 효율이 높습니다.
* **실시간성**: 재전송을 하지 않으므로 지연 시간(Jitter)이 일정해야 하는 스트리밍에 유리합니다.
* **브로드캐스트/멀티캐스트**: 일대다 통신이 가능합니다. (TCP는 불가능)

---

## Ⅲ. RTP (Real-time Transport Protocol)
UDP는 속도는 빠르지만, 패킷이 뒤섞여 들어오면 동영상이 거꾸로 재생될 수 있습니다. 이를 해결하기 위해 UDP 위에 얹어 쓰는 프로토콜입니다. (RFC 3550)

### 1. RTP의 역할
* **순서 번호 (Sequence Number)**: UDP가 섞어놓은 패킷들을 수신 측에서 순서대로 정렬하게 해 줍니다.
* **타임스탬프 (Timestamp)**: 소리가 영상보다 늦게 나오는 일을 막기 위해, 데이터가 생성된 시점을 기록하여 동기화(Lip-sync)합니다.
* **페이로드 유형 (Payload Type)**: 현재 데이터가 어떤 코덱(MP3, H.264 등)으로 압축되었는지 알려줍니다.

### 2. RTCP (RTP Control Protocol)
* RTP와 짝을 이루어 동작하며, 데이터 자체는 보내지 않고 **전송 품질(QoS)을 모니터링**합니다.
* 수신자가 "지금 패킷 유실률이 10%야"라고 알려주면, 송신자가 화질을 낮추는 등의 조치를 취할 수 있게 합니다.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[실시간 스트리밍 패킷의 계층적 포장]

  [ IP Header ] [ UDP Header ] [ RTP Header ] [ Video/Audio Data ]
       │              │              │                │
       │              │              │                └─ 실제 영상 조각
       │              │              └─ 순서(1,2,3), 시간(10:00) 정보
       │              └─ 포트 번호 (앱 식별)
       └─ IP 주소 (집 주소)
```

📢 **섹션 요약 비유**: **UDP**는 목적지 주소만 적힌 '오토바이 퀵 서비스'입니다. 빠르긴 한데 물건 순서가 바뀌거나 하나쯤 잃어버려도 그냥 가버립니다. **RTP**는 오토바이에 실린 상자마다 "이건 1번 조각, 10시 정각에 개봉하세요"라고 적힌 '번호표'를 붙이는 것과 같습니다. **RTCP**는 수신자가 기사님에게 "지금 길이 좀 막히네요, 천천히 오세요"라고 피드백을 주는 무전기 역할을 합니다.
