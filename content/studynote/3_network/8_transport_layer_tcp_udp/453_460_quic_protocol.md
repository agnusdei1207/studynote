# 453-460. 차세대 전송 혁신: QUIC 프로토콜

> **핵심 인사이트**: 인터넷 속도는 빨라졌지만, 40년 전 설계된 TCP는 여전히 느린 핸드셰이크와 패킷 하나만 밀려도 전체가 멈추는(HOL Blocking) 문제를 안고 있다. 이를 해결하기 위해 구글이 UDP라는 자유로운 도화지 위에 TCP의 장점과 최신 보안(TLS 1.3)을 결합해 새로 그린 그림이 바로 QUIC이다.

---

## Ⅰ. QUIC (Quick UDP Internet Connections) 개요
TCP의 성능 한계를 극복하기 위해 **UDP**를 기반으로 구현된 새로운 전송 계층 프로토콜입니다. (RFC 9000 표준)

* **배경**: 커널 수준에서 수정이 힘든 TCP 대신, 애플리케이션 수준(User-space)에서 빠르게 업데이트할 수 있는 UDP를 운송 수단으로 선택했습니다.
* **적용**: HTTP/3의 근간이 되는 기술입니다.

---

## Ⅱ. QUIC의 3대 혁신 기능

### 1. 연결 지연 단축 (Low Latency Handshake)
* **TCP + TLS**: 연결을 맺고(3-Way) 보안 설정을(TLS) 하는 데 여러 번의 왕복(RTT)이 필요했습니다.
* **QUIC**: 연결 설정과 암호화 협상을 동시에 진행합니다.
  - **1-RTT**: 처음 접속할 때 단 한 번의 왕복으로 통신 준비 완료.
  - **0-RTT**: 이전에 접속했던 서버라면, 기다리지 않고 첫 패킷부터 데이터를 실어 보냅니다.

### 2. HOL (Head-of-Line) 블로킹 해결
* **문제**: TCP는 하나의 패킷이 유실되면 그 뒤에 도착한 패킷들이 멀쩡해도 애플리케이션으로 올라가지 못하고 줄줄이 대기해야 합니다.
* **QUIC 해결**: 여러 개의 독립적인 **스트림(Stream)**을 사용합니다. 1번 스트림 패킷이 유실되어도 2, 3번 스트림은 영향을 받지 않고 즉시 처리됩니다.

### 3. 연결 마이그레이션 (Connection Migration)
* **문제**: TCP는 IP 주소가 바뀌면(Wi-Fi $\leftrightarrow$ LTE 전환) 연결이 끊어지고 다시 핸드셰이크를 해야 합니다.
* **QUIC 해결**: IP 주소 대신 **Connection ID**라는 고유 번호로 세션을 식별합니다. 사용자의 IP가 바뀌어도 ID가 같으면 연결이 끊기지 않고 부드럽게 유지됩니다.

---

## Ⅲ. 보안과 신뢰성 (TLS 1.3 기본 내장)
* QUIC은 설계 단계부터 **TLS 1.3** 보안 프로토콜을 필수적으로 포함합니다. 
* 헤더의 일부 정보까지 암호화하여 중간자 공격(MITM)이나 경로상의 정보 노출을 최소화합니다.
* 패킷 번호가 재전송 시에도 중복되지 않게 설계되어, 중복 ACK 판별과 타임아웃 계산이 훨씬 정확해졌습니다.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[TCP vs QUIC 연결 설정 비교]

1. TCP + TLS 1.2 (느림)
   [ SYN ] ──>
   [ SYN/ACK ] <──
   [ ACK + ClientHello ] ──>
   [ ServerHello + KeyShare ] <──
   (총 3회 왕복 후 데이터 전송)

2. QUIC (매우 빠름)
   [ Initial + ClientHello ] ──>
   [ Handshake + ServerHello ] <──
   (단 1회 왕복 후 데이터 전송 시작!)
```

📢 **섹션 요약 비유**: **TCP**는 옛날식 관공서 업무입니다. 1번 창구에서 번호표 뽑고, 2번 창구에서 서류 내고, 3번 창구에서 도장 받아야 비로소 일을 시작합니다. **QUIC**은 모든 서류를 미리 앱으로 전송하고, 가서 얼굴만 한 번 보여주면(1-RTT) 바로 업무가 끝나는 혁신적인 '원스톱 서비스'입니다. 심지어 카페 Wi-Fi에서 쓰다 밖으로 나가도 내 대기 번호(Connection ID)는 그대로 살아있는 아주 편리한 시스템입니다.
