# 297-300. IPv4 주소 체계와 클래스 (Classful Addressing)

> **핵심 인사이트**
> 1. **본질**: **IPv4 (Internet Protocol version 4)**는 32비트 논리 주소 체계로, 계층적 라우팅을 위해 **Network ID**와 **Host ID**로 구성된다. 초기 클래스풀(Classful) 설계는 고정된 비트 경계를 통해 주소 할당의 단순함을 추구했다.
> 2. **가치**: 이 체계는 라우터의 **라우팅 테이블(Routing Table)** 크기를 줄이고 초기 인터넷 확장에 기여했으나, 비효율적인 주소 공간 활용(약 40% 낭비)이라는 구조적 한계를 내재하고 있어 **CIDR (Classless Inter-Domain Routing)** 등으로 진화하게 된다.
> 3. **융합**: 네트워크 계층(Layer 3)의 핵심 식별자로서, **TCP (Transmission Control Protocol)** 통신의 출발지/목적지를 결정하며, **OSPF (Open Shortest Path First)** 등의 IGP 프로토콜과 **NAT (Network Address Translation)** 기술의 기반이 된다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 철학
**IP 주소(Internet Protocol Address)**는 인터넷에 연결된 호스트(Host)나 라우터(Router)와 같은 장비가 서로를 식별하기 위해 사용하는 논리적인 주소(Logical Address)입니다. 물리적인 **MAC 주소(Media Access Control Address)**가 2계층(Data Link Layer)에서의 통신에 사용된다면, IP 주소는 3계츤(Network Layer)에서 **End-to-End 전달**을 담당합니다.

IPv4는 32비트, 즉 2^32개(약 43억 개)의 주소 공간을 제공합니다. 이 체계의 핵심 철학은 **계층적 주소 지정(Hierarchical Addressing)**을 통해 라우팅의 효율성을 높이는 것입니다. 전체 주소를 '어디 네트워크(네트워크 부)'인지와 '그 네트워크 내의 누구(호스트 부)'인지로 나눔으로써, 라우터는 모든 개별 호스트의 경로를 저장할 필요 없이 네트워크 대표 경로만 알면 되기 때문입니다.

#### 2. 도입 배경 및 한계
1980년대 초, 인터넷의 전신인 ARPANET이 확장되면서, 단순한 **Flat Addressing** 구조로는 대규모 네트워크를 관리할 수 없게 되었습니다. 이를 해결하기 위해 IETF는 **Classful Addressing** 체계를 도입하여 주소를 A, B, C, D, E 5개의 클래스로 분류했습니다. 하지만 이 방식은 네트워크 규모에 따라 '너무 크거나 너무 작은' 주소 블록만 할당할 수 있는 **One-size-fits-all** 문제가 있었습니다. 예를 들어, 호스트가 300대뿐인 중소기업에 클래스 B(약 65,000개 할당)를 부여하면 64,700개의 IP가 낭비되는 **Internal Fragmentation(내부 단편화)**이 발생했습니다. 이러한 비효율은 1993년 **CIDR**와 **VLSM (Variable Length Subnet Mask)**의 등장으로 이어졌습니다.

#### 3. 표기법
사람이 이해하기 쉽도록 32비트를 8비트(옥텟) 단위로 4개로 나누고, 이를 10진수로 변환하여 점(.)으로 구분하는 **DDN (Dotted Decimal Notation)** 방식을 사용합니다. 비트 연산 시 주의해야 할 점은, 컴퓨터 내부에서는 여전히 이진수(Binary)로 처리되므로 **서브넷 마스크(Subnet Mask)**와의 **AND 연산**을 통해 네트워크 주소를 추출한다는 점입니다.

> 📢 **섹션 요약 비유**
> IP 주소 체계는 **'우편물 주소 시스템'**과 같습니다. "서울특별시 강남구"와 같은 지역 명칭(Network ID)이 없다면, 우체국(라우터)은 전 세계 모든 집의 위치를 외워야만 편지를 배달할 수 있습니다. IP 주소는 이처럼 지역(네트워크)과 번지(호스트)를 조직화하여 우편 배달 시스템이 효율적으로 돌아가도록 설계된 논리적 체계입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 클래스별 비트 구조 및 할당 원칙
Classful Addressing은 주소의 최상위 비트(High-Order Bit) 패턴을 확인하여 즉시 클래스를 식별할 수 있게 설계되었습니다.

**표 1: IPv4 클래스별 상세 스펙**

| 클래스 | 최상위 비트 패턴 | 주소 범위 (First Octet) | 네트워크 비트 | 호스트 비트 | 네트워크 수 | 호스트 수 (가용) | 주 용도 |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---|
| **A** | `0xxxxxxx` | 0 ~ 127 | 8 | 24 | 126 (2^7-2) | 16,777,214 (2^24-2) | 대규모 (백본) |
| **B** | `10xxxxxx` | 128 ~ 191 | 16 | 16 | 16,384 (2^14) | 65,534 (2^16-2) | 중규모 (대학/기업) |
| **C** | `110xxxxx` | 192 ~ 223 | 24 | 8 | 2,097,152 (2^21) | 254 (2^8-2) | 소규모 (SME/가정) |
| **D** | `1110xxxx` | 224 ~ 239 | N/A | N/A | N/A | N/A | 멀티캐스트 |
| **E** | `1111xxxx` | 240 ~ 255 | N/A | N/A | N/A | N/A | 연구/예약 |

> **참고**: 클래스 A의 0번 네트워크(`0.0.0.0/8`)는 "현재 네트워크"를 의미하며, 127번(`127.0.0.0/8`)은 **Loopback**으로 예약되어 있어 실제 할당 가능한 네트워크는 126개입니다.

#### 2. 아키텍처 다이어그램
아래 다이어그램은 32비트 IPv4 주소가 각 클래스에서 어떻게 Network ID와 Host ID로 분리되는지를 시각적으로 보여줍니다.

```ascii
     +---------------------------------------------------+
     |              IPv4 32-bit Address Space            |
     +---------------------------------------------------+

Class A (0xxxxxxx):
  |0|      Network ID (7 bits)       |        Host ID (24 bits)         |
  +---------------------------------------------------+
  | 0    1 . 0 . 0 . 0  ~  126 . 255 . 255 . 255       |
  |<- Large Networks ->|<------- Huge Hosts ---------->|

Class B (10xxxxxx):
  |1 0|   Network ID (14 bits)      |        Host ID (16 bits)         |
  +---------------------------------------------------+
  |128 . 0 . 0 . 0  ~  191 . 255 . 255 . 255           |
  |<------ Medium Networks ------>|<----- Medium Hosts ----->|

Class C (110xxxxx):
  |1 1 0|    Network ID (21 bits)    |    Host ID (8 bits)    |
  +---------------------------------------------------+
  |192 . 0 . 0 . 0  ~  223 . 255 . 255 . 255           |
  |<--------- Many Small Networks --------->|<- Small ->|

Legend: 'x' = Variable bit
```

**다이어그램 해설**:
이 구조에서 **네트워크 마스크(Network Mask)**는 기본적으로 고정된 길이를 가집니다. 클래스 A는 `/8` (255.0.0.0), 클래스 B는 `/16` (255.255.0.0), 클래스 C는 `/24` (255.255.255.0)가 기본 마스크입니다. 라우터는 패킷을 수신하면 목적지 IP의 첫 번째 옥텟을 보고 클래스를 판단한 후, 해당하는 기본 마스크를 적용하여 목적지 네트워크를 찾습니다. 이 과정은 하드웨어적으로 매우 빠르게 처리되지만, **Classless** 환경에서는 더 이상 첫 번째 비트만으로는 클래스를 판단할 수 없으므로 항상 마스크 정보가 별도로 필요합니다.

#### 3. 심층 동작 원리: AND 연산과 서브넷 판별
IP 통신에서 가장 기초적이고 중요한 연산은 **서브넷 마스크와의 AND 연산**입니다. 송신 호스트는 목적지 IP와 자신의 서브넷 마스크를 AND 연산하여 **목적지 네트워크 주소(Destination Network Address)**를 도출합니다.

```python
# Pseudo-Code for Subnet Identification
def is_same_subnet(dest_ip, my_ip, subnet_mask):
    # 논리적 AND 연산: 대응하는 비트가 둘 다 1일 때만 1
    dest_network = dest_ip & subnet_mask
    my_network   = my_ip & subnet_mask
    
    if dest_network == my_network:
        return "Local Delivery (ARP Request)"
    else:
        return "Remote Delivery (Gateway IP)"
```

이 로직은 **OSI 7계층** 모델에서 계층 간 데이터 캡슐화와 전달의 핵심입니다.

> 📢 **섹션 요약 비유**
> 이 구조는 마치 **'전화번호부 시스템'**과 같습니다. **지역번호(Network ID)**가 길면(클래스 A), 그 안에 포함될 수 있는 **가입자 번호(Host ID)**의 자릿수는 짧아집니다. 반대로 지역번호가 짧으면, 그 지역에 거주할 수 있는 가입자 수가 엄청나게 늘어나는 구조입니다. 전화국(라우터)은 지역번호만 보고 통신을 연결할지, 다른 지역으로 중계할지를 결정합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. Classful vs Classless (CIDR) 비교
초기 인터넷의 Classful 방식은 주소 낭비가 심하여 **CIDR (Classless Inter-Domain Routing)**로 전환되었습니다. 이 두 체계의 기술적 차이는 **Aggregation(집약)**과 **Routing Table Expansion**과 직결됩니다.

**표 2: Classful Addressing vs Classless Addressing (CIDR)**

| 비교 항목 | Classful Addressing (과거) | Classless Addressing (CIDR, 현재) |
|:---|:---|:---|
| **비트 경계** | 고정적 (8, 16, 24 비트 단위) | 가변적 (임의의 비트 길이 `/n` 사용) |
| **표기법** | `192.168.1.0` (Class C 유추) | `192.168.1.0/24` (명시적 마스크) |
| **주소 낭비** | 심각 (Class B 할당 시 64K 개 필요 여부 무시) | 최소화 (필요한 만큼만 할당, e.g., `/27`) |
| **라우팅 테이블** | 작음 (개념적으로 단순함) | 최적화 필요 (**Route Aggregation**으로 크기 억제) |
| **예시** | Class C: `.xxx` | `/24` (Class C와 동일) ~ `/30` (Point-to-Point) |

#### 2. 네트워크 계층(Layer 3)과 전송 계층(Layer 4)의 시너지
IP 주소(계층 3)는 목적지까지의 '길 찾기'를 담당하고, **Port 번호(계층 4)**는 도착 후의 '어플리케이션 식별'을 담당합니다. 이 두 계층의 융합은 소켓 통신(`Socket = IP + Port`)에서 완성됩니다. 예를 들어, `192.168.1.1:80`은 IP 주소가 웹 서버 네트워크까지 전달하고, Port 80이 그 안의 HTTP 프로세스를 찾아갑니다.

> 📢 **섹션 요약 비유**
> Classful 방식은 **'옷장 정리'**에 비유할 수 있습니다. S, M, L 사이즈로만 옷을 분류해두면, 약간 뚱뚱한 사람에게는 L이 꽉 끼고, 작은 사람에게는 L이 너무 커서 공간을 낭비하게 됩니다. CIDR은 **'수재봉(High-End Tailoring)'**처럼 사람의 체형(네트워크 규모)에 딱 맞는 옷(주소 공간)을 만들어 주어 낭비를 없애는 방식입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 시나리오: 주소 설계 및 할당
네트워크 설계자는 IP 주소를 할당할 때, 단순히 호스트 수만 맞추는 것이 아니라 **추가 확장성**과 **보안**을 고려해야 합니다.

*   **문제 1: 본사 500명, 지사 5곳(각 20명)의 네트워크 구축**
    *   **잘못된 판단**: 본사에 Class B(65,534개)를 할당한다. (IP 주소 멸망 원인)
    *   **올바른 판단**: 본사에 Class C(`192.168.1.0/24`, 254개)를 할당하고, 지사에는 각각 `/27`(30개 사용 가능) 서브넷을 Class C 하나에서 쪼개어 할당한다. (**VLSM** 적용)
*   **문제 2: 공인 IP(Public IP) 부족 문제**
    *   전 세계적으로 IPv4 주소가 고갈됨에 따라, 외부 통신에는 **공인 IP**를, 내부망에는 **사설 IP(Private IP)**를 사용하는 **NAT (Network Address Translation)** 기술이 필수적이 되었습니다. RFC 1918에 정의된 사설 대역(A: `10.0.0.0/8`, B: `172.16.0.0/12`, C: `192.168.0.0/16`)을 적극 활용하여 공인 IP 절약 전략을 세워야 합니다.

#### 2. 도입 체크리스트
1.  [ ] **주소 요구량 분석**: 현재 호스트 수 + 향후 2년 간 증가율을 반영하여 블록 크기 산출
2.  [ ] **라우팅 프로토콜 호환성**: 사용할 라우팅 프로토콜(RIP, OSPF, BGP 등)이 Classful 라우팅을 지원하는지(Classful RIP 등) Classless를
