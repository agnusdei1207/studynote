# 307-311. 네트워크 주소 변환(NAT)과 횡단 기술

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: NAT (Network Address Translation)는 IPv4 주소 고갈 문제를 해결하고 내부 네트워크의 보안을 강화하기 위해, 패킷의 IP 주소(및 포트)를 경계 장비에서 변경하는 주소 변환 기술입니다.
> 2. **가치**: 하나의 공인 IP로 다수의 사설 단말을 인터넷에 연결함으로써 IP 할당 비용을 절감하고, 내부 토폴로지를 은닉하여 외부 공격으로부터 방어하는 계층적 보안 효과를 제공합니다.
> 3. **융합**: VoIP나 WebRTC와 같은 P2P 서비스에서 NAT 장벽을 넘기 위한 STUN/TURN/ICE 프로토콜과 연계되며, 방화벽(Firewall) 정책과 밀접하게 결합된 네트워크 보안의 핵심 요소입니다.

---

### Ⅰ. 개요 (Context & Background)

NAT는 사설 네트워크(Private Network)에서 사용하는 사설 IP 주소를 공용 네트워크(Internet)에서 사용하는 공인 IP 주소로 변환하는 기술입니다. 이 기술의 핵심 철학은 **'주소 보존'**과 **'보안 은닉'**에 있습니다. IPv4 (Internet Protocol version 4)의 주소 공간(약 43억 개)은 인터넷의 폭발적인 성장으로 인해 고갈되었으나, NAT를 통해 C클래스(예: 192.168.x.x) 사설 대역을 무제한으로 내부에서 재사용함으로써 주소 부족 문제를 해결했습니다. 또한, 외부에서는 내부의 사설 IP를 직접 알 수 없으므로, 일종의 방화벽 역할을 수행하여 내부망의 구조를 보호하는 효과를 가집니다.

```ascii
[ NAT 개념적 위치도 ]

    ( 내부 사설망 - Inside Network )          ( NAT 장비 : 경계 )           ( 공용 인터넷 - Outside )
                                     
   PC A: 10.1.1.5  --------------------->   [ 변환 테이블 ]  ---------------------->  203.0.113.5
   (Source: 10.1.1.5)                       (Src: 10.1.1.5              (Src: 1.1.1.1)
                                            -> 1.1.1.1)
                                            
   [ 원격 서버 ]
   (Dst: 203.0.113.5)
```
*도해: 내부에서 발생한 패킷의 소스 IP가 NAT 장비를 통과하며 공인 IP로 변경되어 인터넷으로 향하는 흐름을 보여줍니다.*

NAT 동작은 주로 L3 계층(네트워크 계층)과 L4 계층(전송 계층)의 헤더를 수정합니다. 변환 정보는 내부 메모리에 유지되며, 이를 통해 통신 세션을 추적합니다. 이는 마치 복잡한 고속도로 톨게이트에서 하이패스 차선(고속 패스)을 별도로 운영하여, 수많은 차량(내부 IP)들이 소수의 진입로(공인 IP)를 효율적으로 사용하도록 하는 것과 같습니다.

📢 **섹션 요약 비유**: NAT는 마치 건물 관리실에서 외부에서 오는 택배나 방문객을 위해 각 호수(사설 IP)를 '001호, 002호' 대신 '김대표, 이팀장'이라는 **별칭(공인 IP)**으로 등록해두고, 건물 밖으로 나갈 때는 별칭을 사용했다가 다시 돌아올 때는 호수로 바꿔주는 **지능형 우편물 배달 시스템**과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

NAT는 매핑(Mapping) 방식에 따라 Static, Dynamic, PAT로 나뉘며, 각 방식은 상이한 메커니즘과 성능 특성을 가집니다. 이 섹션에서는 각 방식의 구조와 PAT의 핵심인 포트 변환 로직을 분석합니다.

#### 1. 구성 요소 및 매핑 방식 비교

| 구분 | Static NAT (1:1) | Dynamic NAT (Pool) | PAT (NAPT) (1:N) |
|:---|:---|:---|:---|
| **정의** | 고정된 1:1 IP 매핑 | 공인 IP Pool에서 임시 할당 | 1개 공인 IP + 포트 분할 |
| **Full Name** | Static Network Address Translation | Dynamic Network Address Translation | Port Address Translation |
| **변환 대상** | IP 주소만 변환 | IP 주소만 변환 | **IP 주소 + 포트 번호** 변환 |
| **주요 용도** | 외부 공개 서버(DMZ) | 일반 사무실 인터넷 연결 (구형) | 가정/기업 공유기 (가장 보편적) |
| **세션 수** | 공인 IP 개수로 제한됨 | 공인 IP Pool 개수로 제한됨 | 약 6만 개(포트 범위)까지 확장 가능 |
| **비유** | 단독 주택 전용 번호 | 공동 주택 공용 전화번호 대기열 | **회사 대표번호 + 내선번호** |

#### 2. PAT (Port Address Translation) 트래픽 처리 메커니즘

PAT는 IP 주소뿐만 아니라 TCP/UDP 헤더의 포트 번호(16비트, 0~65535)를 함께 수정하여 수천 개의 세션을 하나의 공인 IP로 멀티플렉싱(Multiplexing)합니다.

```ascii
[ PAT 변환 프로세스 및 테이블 구조 ]

    내부망 (Inside Local)                  NAT 라우터 (Translation Table)              외부망 (Inside Global)
+----------------------+    +---------------------------------------------------------+    +---------------------+
| PC A: 192.168.1.10   |    | [ Inside Local  | Inside Global  | Outside  | Protocol ]|    | Web Server         |
| Port: 50001          |--->| 192.168.1.10:50001 <-> 1.1.1.1:20001 <-> 8.8.8.8:443 TCP|    | 8.8.8.8:443         |
+----------------------+    +---------------------------------------------------------+    +----------+----------+
      |                                                                                          |
      |     (1) Request: Src=192.168.1.10:50001 -> Dst=8.8.8.8:443                                  |
      |     (2) NAT Changes Src -> 1.1.1.1:20001 (Unique Port Mapping)                            |
      |     (3) Internet: Src=1.1.1.1:20001 -> Dst=8.8.8.8:443                                     |
      |     (4) Return: Src=8.8.8.8:443 -> Dst=1.1.1.1:20001                                       |
      |     (5) NAT Lookup: Finds 20001 mapped to 192.168.1.10:50001                               |
      +<-------------------------------------------------------------------------------------------+
      (6) Delivered: 192.168.1.10:50001
```

**해설**:
1.  **① 패킷 수신 및 분석**: NAT 장비는 내부 인터페이스로부터 패킷을 수신하면, 헤더의 Source IP와 Source Port를 추출합니다.
2.  **② 테이블 생성 및 포트 할당**: 현재 사용하지 않는 가용한 공인 포트(예: 20001)를 선택하여 NAT 테이블을 생성/갱신합니다. `hash(IP, Port)` 알고리즘을 사용하여 충돌을 방지합니다.
3.  **③ 헤더 변환 및 전송**: IP 헤더의 Source IP를 공인 IP로, TCP/UDP 헤더의 Source Port를 할당된 포트로 변경(Checksum 재계산 포함)하여 외부로 전송합니다.
4.  **④ 역방향 복원**: 외부에서 들어오는 응답 패킷의 Destination Port(20001)를 키(Key)로 테이블을 검색하여 원래의 사설 IP와 포트로 복원 후 내부로 전달합니다.

#### 3. 핵심 알고리즘: 포트 할당 및 충돌 처리
PAT의 가장 큰 기술적 난제는 포트 고갈과 충돌 방지입니다. 실무에서는 포트 할당 시 다음과 같은 로직을 적용합니다.

```python
# NAT Port Allocation Logic (Pseudo-code)
# Full Name: PAT (Port Address Translation) Allocation Algorithm

global_port_pool = range(1024, 65535) # 가용 포트 범위 (Well-known 제외)
nat_table = {}

def allocate_port(source_ip, source_port):
    # 기존 세션 확인
    if (source_ip, source_port) in nat_table:
        return nat_table[(source_ip, source_port)] # 기존 포트 재사용
    
    # 새로운 포트 할당 (랜덤 혹은 순차)
    try:
        new_port = select_random_port(global_port_pool)
        nat_table[(source_ip, source_port)] = new_port
        return new_port
    except OutOfPortsError:
        # 포트 고갈 시 처리 (PAT Port Overloading 제한)
        trigger_log("Global Port Pool Exhausted")
        return ERROR
```

📢 **섹션 요약 비유**: PAT는 마치 **거대한 쇼핑몰의 '휴대품 보관 서비스'**와 같습니다. 수백 명의 고객(내부 IP)이 각자의 가방(데이터)을 맡기지만, 관리 직원(NAT)은 그 가방들에 **번호표(포트 번호)**를 붙여서 아주 좁은 보관함(공인 IP) 한 칸에 모두 빽빽하게 쌓아둡니다. 찾을 때는 번호표만 보고 정확한 고객에게 돌려주죠.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

NAT는 단순한 주소 변환을 넘어 보안, 서비스 품질(QoS), 그리고 애플리케이션 계층 프로토콜과 복잡하게 얽혀 있습니다.

#### 1. 기술적 비교 분석: NAT vs Proxy

| 비교 항목 | **NAT (L3/L4)** | **Proxy Server (L7)** |
|:---|:---|:---|
| **작동 계층** | OSI 3계층(IP), 4계층(Port) | OSI 7계층(Application) |
| **클라이언트 인식** | **무지**: 클라이언트는 NAT의 존재를 모름 (투명함) | **인식**: 클라이언트는 Proxy 설정을 해야 함 |
| **성능 부하** | 매우 낮음 (하드웨어 스위칭 수준) | 높음 (컨텐츠 캐싱 및 검수 로직) |
| **보안 기능** | 기본 은닉 (내부망 보호) | 심층 방어 (DMZ, 캐시, 인증 등) |
| **P2P 친화성** | **NAT Traversal 필요** (주소 변환으로 인한 연결 제한) | 상대적으로 유리함 (중계 서버 역할) |

#### 2. NAT와 애플리케이션 계층의 상호작용 (ALG)
표준 NAT는 IP/Port만 수정하므로, 데이터 패킷(Payload) 내부에 IP 주소 정보를 포함하는 프로토콜(예: FTP, SIP)은 작동하지 않습니다. 이를 해결하기 위해 **ALG (Application Level Gateway)** 기능이 탑재된 NAT가 필요합니다.

```ascii
[ FTP의 Passive Mode 및 ALG 개입 순서 ]

  1. FTP Client -----------------> FTP Server (Control Connection: Port 21)
     (PORT 192,168,1,10,195,12)  "내부 IP:Port로 접속해라"

  2. [ NAT-ALG 개입 ]
     - 라우터가 Control 패킷 내용을 스니핑(Sniffing)
     - 내부 IP(192.168.1.10)를 공인 IP(1.1.1.1)로 변환하여 패킷 내용 자체를 수정
     - 변환된 패킷: "PORT 1,1,1,1,..." 로 서버에 전달

  3. FTP Server ----------------> NAT Router (Data Connection Attempt)
     (Target: 1.1.1.1:50001)     -> NAT Table에 의해 자동으로 내부 PC로 전달
```
*도해: ALG가 FTP 제어 채널의 페이로드를 수정하여 데이터 연결이 성립되는 과정을 보여줍니다.*

#### 3. 타 과목 융합 관점 (OS 및 보안)
*   **OS와의 관계**: OS의 TCP/IP 스택은 패킷을 생성할 때 기본 게이트웨이로 NAT 공유기의 MAC 주소를 지정합니다. 따라서 OS는 NAT의 존재를 모르며, 단순히 기본 게이트웨이가 넓은 인터넷과 연결된 '다음 홉'인 것으로만 인식합니다.
*   **보안(Security)과의 관계**: NAT는 inbound 패킷(외부->내부)을 기본적으로 차단합니다. 요청하지 않은 패킷이 들어오면 NAT 테이블에 매핑이 없으므로 폐기합니다. 이는 자연스러운 **Stateful Inspection** 상태 방화벽 효과를 제공합니다. 하지만 보안 정책이 복잡할 경우 **Firewall과 NAT의 순서**가 중요합니다. (Packet Flow: In -> NAT -> Firewall vs In -> Firewall -> NAT)

📢 **섹션 요약 비유**: ALG는 마치 **검열 책임자가 편지 봉투의 주소만 바꾸는 게 아니라, 편지지 내용에 적힌 '나의 집 좌표'까지 외부에서 이해할 수 있는 '공공 주소'로 교정해주는 행정 절차**와 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무 네트워크 설계 시 NAT는 필수 불가결하지만, 문맥에 따라 적절한 유형을 선택하고 문제를 해결해야 합니다.

#### 1. 실무 시나리오 및 의사결정 프로세스

| 시나리오 | 요구사항 | 권장 기술 | 의사결정 근거 |
|:---|:---|:---|:---|
| **웹 서비스 공개** | 외부에서 접속 가능해야 함, 고정 IP 필요 | **Static NAT (1:1)** | 외부 사용자는 특정 공인 IP로 접속해야 하므로 매핑이 고정적이어야 함. |
| **사무실 인터넷 공유** | 직원 100명이 동시 접속, 비용 절감 | **Dynamic NAT / PAT** | IP Pool 관리의 오버헤드보다 PAT가 효율적이며 하나의 공인 IP로 충분함. |
| **화상 회의(ZOOM, WebRTC)** | 내부 클라이언트 간 P2P 연결 필요 | **STUN/TURN/ICE** | Symmetric
