# 286-291. IPv4 프로토콜과 헤더 구조 및 단편화

### # IPv4 프로토콜과 헤더 구조 및 단편화 (IPv4 Protocol and Fragmentation)
#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터넷의 표준 전달 계층인 **IPv4 (Internet Protocol version 4)**는 **비연결형(Connectionless)**, **데이터그램(Datagram)** 방식으로 작동하며 신뢰성보다는 전달 효율성과 단순성을 최우선으로 설계되었습니다.
> 2. **가치**: 라우터의 처리 부하를 최소화하여 고속 라우팅을 가능하게 하며, **MTU (Maximum Transmission Unit)** 차이로 발생하는 전송 장애를 **단편화(Fragmentation)** 기술로 극복하여 인터넷의 상호 연결성을 보장합니다.
> 3. **융합**: 상위 계층인 **TCP (Transmission Control Protocol)**와 결합하여 신뢰성을 확보하고, 하위 계층의 **Ethernet**과 같은 물리적 프레임 크기 제약을 추상화하여 네트워크 계층의 독립성을 유지합니다.

---

### Ⅰ. 개요 (Context & Background)
**IPv4 (Internet Protocol version 4)**는 인터넷 프로토콜 슈트(Internet Protocol Suite)의 핵심 네트워크 계층(Network Layer) 프로토콜로, 1981년 **RFC 791**으로 표준화되었습니다. 그 근본 철학은 **'끝점 간의 전달(End-to-End Delivery)'**입니다. **TCP (Transmission Control Protocol)**가 연결 지향적(Connection-oriented)이라면, IPv4는 상대방과 미리 연결을 설정(Handshake)하지 않고 데이터를 독립적으로 전송하는 **비연결형(Connectionless)** 모델을 따릅니다. 이는 각 패킷이 독립적인 **데이터그램(Datagram)**으로 취급됨을 의미하며, 네트워크의 중간 노드(라우터)는 연결 상태를 유지할 필요 없이 순수히 헤더 정보만으로 다음 홉(Next Hop)을 결정합니다. 이러한 비연결성과 최선 노력(Best Effort) 전달 특성은 인터넷의 규모를 확장 가능한(Scalable) 것으로 만들었으나, 패킷 손실, 순서 뒤바뀜 등의 신뢰성 문제는 상위 계층에 전가하게 됩니다.

이러한 구조는 복잡한 제어 로직이 라우터에 있지 않고 네트워크 엣지(Edge, 호스트)에 존재한다는 **'엔드 투 엔드 원칙(End-to-End Argument)'**의 완벽한 구현체입니다.

```ascii
[데이터 전송 철학 비교]

 ┌─────────────────────────────────────────────────────────────────────┐
│  📡 전송 모델 비교: TCP (연결 지향형) vs IPv4 (비연결형)               │
└─────────────────────────────────────────────────────────────────────┘

  [TCP Model]                     [IPv4 Model]
  (Connection-Oriented)           (Connectionless / Datagram)

  1. SYN (연결 요청)              
     └───────● 연결 확립 ───────┐   (없음) ───┐
  2. ACK (승인)                  │             │
  3. DATA (순서 보장 전송)       │             ▼
  4. FIN (연결 종료)             │   1. DATA ●─┐
                                │             │
  [Logic is Heavy]              │   2. DATA ●─┤ (독립적)
  [Reliable]                    │   3. DATA ●─┘
                                │
```
**해설**: 위 다이어그램은 TCP의 연결 설정 과정(SYN/ACK/FIN)과 IPv4의 즉시 전송 방식을 대조합니다. TCP는 사전에 논리적 회선을 확보하고 데이터를 쓰나 드나들이지만, IPv4는 편지를 우편함에 넣는 것처럼 즉시 네트워크에 던지는 차이를 보여줍니다.

**💡 비유**: TCP는 전화를 걸어 "여보세요?" 하고 통화를 시작하는 것이라면, IPv4는 엽서를 우체통에 넣어버리는 것과 같습니다. 수신자가 엽서를 받았는지, 중간에 잃어버렸는지 우체국(라우터)은 신경 쓰지 않고 오직 배달만 담당합니다.

**등장 배경**: 초기 ARPANET의 연결 지향적 방식(NCP)은 전 세계적인 규모 확장에 한계가 있었습니다. **DARPA (Defense Advanced Research Projects Agency)**는 군사적 목적에 따라 **폭탄 맹스**(일부 노드 파괴)에도 견딜 수 있는 유연한 패킷 교환망이 필요했고, 이로 인해 **TCP/IP** 아키텍처가 탄생했습니다. 현재 **IoT (Internet of Things)**의 폭발적 증가와 IPv4 주소 고갈 문제는 **IPv6 (Internet Protocol version 6)**로의 전환을 요구하고 있으나, 여전히 인터넷의 주류는 IPv4입니다.

> **📢 섹션 요약 비유**: IPv4는 우편 시스템의 '표준 우편 서비스'와 같습니다. 편지(패킷)를 보내기 위해 우체국(라우터)과 예약을 하지 않고, 그냥 편지에 주소(IP)를 쓰고 함에 넣으면 됩니다. 편지가 순서대로 도착하든, 길에서 잃어버리든 우체국은 보장하지 않는 '최선을 다해 보내는' 시스템입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)
IPv4의 핵심은 **헤더(Header)** 구조에 있습니다. 헤더는 패킷의 **메타데이터(Metadata)**를 담고 있으며, 크기는 **가변적(20~60 Bytes)**입니다. 이 섹션에서는 20바이트 기본 헤더의 필드를 뜯어보고, 데이터가 쪼개지는 단편화(Fragmentation)의 내부 메커니즘을 분석합니다.

#### 1. 구성 요소 상세 분석 (Component Analysis)

| 요소명 (Field) | 비트 (Bit) | 역할 (Role) | 내부 동작 및 프로토콜 (Mechanism) | 실무적 포인트 (Note) |
|:---:|:---:|:---|:---|:---|
| **Version** | 4 | IP 버전 식별 | 값 4 (IPv4), 6 (IPv6). | 헤더 파싱의 첫 단계. |
| **IHL**<br>(Internet Header Length) | 4 | 헤더 길이 | **4바이트(32비트) 워드** 단위. 5 = 20B, 15 = 60B. | 옵션 사용 시 이 값이 증가하며 라우터 처리 지표. |
| **ToS**<br>(Type of Service) | 8 | 서비스 품질 | 우선순위(Prec) 및 지연(D), 처리량(T), 신뢰성(R) 비트. | 현대 네트워크에서는 **DSCP (Differentiated Services Code Point)** 및 **ECN (Explicit Congestion Notification)**로 재활용됨. |
| **Total Length** | 16 | 패킷 총 길이 | 헤더 + 데이터. 최대 65,535 바이트. | **MTU**와 직접적으로 비교되어 단편화 여부 결정. |
| **ID**<br>(Identification) | 16 | 패킷 식별자 | 단편화된 조각들이 같은 원본임을 증명하는 고유 번호. | **DF/MF** 플래그와 함께 단편화 재조립의 핵심 키. |
| **Flags** | 3 | 단편화 제어 | **DF (Don't Fragment)**: 쪼개지 금지.<br>**MF (More Fragments)**: 뒤에 더 조각 있음.<br>(1 bit 예비) | **PMU (Path MTU Discovery)** 시 DF=1 설정하여 쪼개지지 않은 최대 크기를 탐색. |
| **Fragment Offset** | 13 | 조각 위치 | **8바이트 단위** 블록 번호. 원본 데이터의 시작점 위치. | 13비트로 최대 $8 \times 8191 = 65,528$ 바이트까지 표현 가능. |
| **TTL**<br>(Time to Live) | 8 | 수명 제한 | 초 단위(원래)가 아닌 **홉(Hop) 수**로 해석됨. 라우터 통과 시 1 감소. | 0이 되면 패킷 폐기 및 **ICMP Time Exceeded** 메시지 송신. 루프 방지. |
| **Protocol** | 8 | 상위 프로토콜 | **1=ICMP, 6=TCP, 17=UDP**, 89=OSPF 등. | MUX(멀티플렉서) 역할. 수신 호스트가 어떤 상위 계층 소켓에 전달할지 결정. |
| **Header Checksum** | 16 | 헤더 무결성 | 헤더 부분만 검증. **TTL** 변화로 라우터마다 재계산 필요. | 연산 오버헤드를 줄이기 위해 데이터 페이로드는 검증 안 함. |
| **Src/Dst Addr** | 32/32 | 발신/착신 주소 | 네트워크 ID(NetID) + 호스트 ID(HostID). | 라우팅의 핵심 결정 요소. |

#### 2. 단편화(Fragmentation) 메커니즘

단편화는 **MTU (Maximum Transmission Unit)**의 불일치에서 발생합니다. 송신 호스트나 중간 라우터는 패킷이 다음 링크의 MTU를 초과하면, 이를 여러 개의 **프래그먼트(Fragment)**로 쪼개야 합니다. 이때 **Layer 3 (IP Layer)**는 데이터를 8바이트 단위의 블록으로 자르는 것을 원칙으로 합니다. 이는 13비트 오프셋 필드를 효율적으로 사용하기 위함입니다.

```ascii
[단편화(Fragmentation) 구조도 및 데이터 레이아웃]

 원본 패킷 (Total Length: 4000 Bytes, ID: 2468)
 ┌───────────────────────────────────────────────────────────────────────┐
│  IP Header (20B) │ Data Payload (3980B)                                │
└───────────────────────────────────────────────────────────────────────┘
           │
           │  [Router encountering MTU=1500 Link]
           │  (Total 4000B > 1500B -> Must Fragment)
           ▼
 ┌──────────────────────────────────────────────────────────────────────────────┐
│ ⚙️  계산 과정:                                                                 │
│ 1. 최대 데이터 공간: 1500 (MTU) - 20 (Header) = 1480 Bytes                     │
│ 2. 1480은 8의 배수이므로 오프셋 계산 가능: 1480 / 8 = 185                      │
│ 3. 마지막 조각 제외: 3980 - 1480 - 1480 = 1020 Bytes 남음                     │
└──────────────────────────────────────────────────────────────────────────────┘

   ↓ 쪼개기 (Fragmentation)

 ┌────────────────────────────┐  ┌────────────────────────────┐  ┌───────────────────────────┐
│ #1 Fragment (Forwarded)     │  │ #2 Fragment (Forwarded)    │  │ #3 Fragment (Forwarded)   │
├────────────────────────────┤  ├────────────────────────────┤  ├───────────────────────────┤
│ ID: 2468                    │  │ ID: 2468                    │  │ ID: 2468                  │
│ MF: 1 (More to come)        │  │ MF: 1 (More to come)        │  │ MF: 0 (Last one)          │
│ Offset: 0 (0 * 8)           │  │ Offset: 185 (1480/8)        │  │ Offset: 370 (2960/8)      │
│ Length: 1500 (20+1480)      │  │ Length: 1500 (20+1480)      │  │ Length: 1040 (20+1020)    │
├────────────────────────────┤  ├────────────────────────────┤  ├───────────────────────────┤
│ IP Hdr │  Data (1480B)      │  │ IP Hdr │  Data (1480B)     │  │ IP Hdr │  Data (1020B)    │
│        │ [Block 0~184]      │  │        │ [Block 185~369]   │  │        │ [Block 370~End]   │
└────────────────────────────┘  └────────────────────────────┘  └───────────────────────────┘
```

**해설**: 다이어그램은 4000바이트 패킷이 1500바이트 MTU를 가진 네트워크 구간을 통과하며 3등분되는 과정입니다. 핵심은 **Offset (오프셋)**이 '실제 바이트 수'가 아니라 '8바이트 블록 단위'로 기록된다는 점입니다. 두 번째 조각의 오프셋이 1480이 아니라 185인 이유가 여기에 있습니다. 수신 측은 이 오프셋에 8을 곱하여 실제 위치를 복원합니다.

#### 3. 재조립(Reassembly) 및 타이머
재조립은 **최종 목적지 호스트**에서만 이루어집니다. 중간 라우터는 재조립하지 않습니다.
*   **버퍼링**: 수신 호스트는 `ID`가 같고 `Src IP`가 같은 패킷들을 모읍니다.
*   **정렬**: `Fragment Offset`을 사용하여 데이터를 순서대로 배열합니다.
*   **완료 확인**: `MF=0`인 패킷이 도착하면 재조립이 완료된 것으로 판단하고 상위 계층(TCP/UDP)으로 전달합니다.
*   **실패 처리**: 일정 시간(보통 30~60초) 내에 모든 조각이 도착하지 않으면, **메모리 누수 방지**를 위해 모든 조각을 폐기하고 **ICMP Time Exceeded (Reassembly Timeout)** 오류를 발생시킵니다.

> **📢 섹션 요약 비유**: 인터넷은 다양한 너비의 도로(링크)로 연결된 고속도로 시스템입니다. IPv4 헤더는 화물 트럭에 붙은 운송장입니다. 만약 큰 트럭이 좁은 다리(MTU)를 지나지 못하면, 분리 컨테이너 센터(라우터)에서 트럭을 여러 대의 소형 트럭(단편화)으로 나눕니다. 이때 운송장에 "이 화물은 전체 짐 중 몇 번째 칸(Offset)"이라는 **넘버링 태그(ID, Offset)**를 붙여서,
