# 266-271. 초기 광역 통신망(WAN): X.25와 프레임 릴레이

> 1. **본질**: 패킷 교환 기술의 진화는 '신뢰성(Reliability)이 물리 계층에 있는가, 상위 계층에 있는가'에 대한 책임 소재의 이동입니다. X.25는 비신뢰 회선을 위해 프로토콜 스택이 무거운 복잡한 에러 복구를 수행하지만, 프레임 릴레이는 광케이블과 같은 저오율(Low Bit Error Rate) 회선을 가정하여 에러 복구를 단말(End-point)으로 이관(E2E Principle)함으로써 고속화를 달성했습니다.
> 2. **가치**: X.25는 낡은 구리선(Copper Wire) 환경에서 안정적인 데이터 통신을 가능하게 했으나 64Kbps의 속도 한계가 있었습니다. 프레임 릴레이는 이론적으로 최대 45Mbps(T3/E3 수준)까지 처리량을 높이면서도 전용선(Leased Line) 대비 평균 30~50% 이상의 비용 절감 효과를 제공하며 1990년대 기업 WAN의 표준으로 자리 잡았습니다.
> 3. **융합**: 데이터 링크 계층(Layer 2)의 **VC (Virtual Circuit)** 개념과 **CIR (Committed Information Rate)**와 같은 트래픽 약속(Traffic Policing) 기술은 현대의 **MPLS (Multiprotocol Label Switching)**와 **QoS (Quality of Service)** 설계의 시초가 되었습니다.

---

### Ⅰ. 개요 (Context & Background)

광역 통신망(WAN, Wide Area Network)의 초기 역사는 물리적 매체의 품질과 프로토콜의 부담 사이의 줄다리기였습니다. 1970~80년대, 기업들은 지사 간 연결을 위해 **전용선(Leased Line)**을 사용하거나 공중전화망(PSTN)을 이용하는 **다이얼업(Dial-up)** 방식을 사용했습니다. 전용선은 안정적이었으나 거리에 비례해 비용이 기하급수적으로 증가한다는 한계가 있었고, 다이얼업은 저렴했으나 56Kbps 수준의 낮은 대역폭과 긴 연결 설정 시간(Latency)으로 인해 실무적 한계가 명확했습니다.

이러한 문제를 해결하기 위해 등장한 것이 **패킷 교환망(Packet Switched Network)**입니다. 다수의 사용자가 하나의 통신사 망(Cloud)을 공유하여 비용을 절감하되, 논리적인 독립 회선을 확보하여 전용선처럼 사용하는 기술입니다. 초기 통신 환경인 아날로그 구리선은 잡음(Noise)과 간섭이 심해 비트 오류율(BER, Bit Error Rate)이 높았습니다(약 $10^{-5}$ 수준). 따라서 망 자체가 패킷마다 에러를 검사하고 수정해주는 무겁지만 안전한 **X.25** 프로토콜이 탄생했습니다. 하지만 1990년대 들어 광케이블(Optical Fiber)이 보급되어 물리적 신뢰도가 획기적으로 향상되자($10^{-9}$ 이하), 이제는 망의 개입을 줄이고 속도에 집중하는 **프레임 릴레이(Frame Relay)**로 패러다임이 이동했습니다.

**💡 비유**: X.25는 울퉁불퉁한 비포장 도로에서 포장 상자를 하나하나 확인하며 운행하는 '완벽주의 화물 트럭'이고, 프레임 릴레이는 평탄한 고속도로가 깔리자 검문소를 통과하며 목적지만 확인하고 전속 질주하는 '고속 택배'입니다.

**📢 섹션 요약 비유**: 마치 비포장 모래사장길에서는 조심해서 운전해야 하지만(안정성 중시), 고속도로가 완공되면 속도를 내는 것이 효율적인 것과 같습니다. 통신선의 품질(도로)이 프로토콜(운전 방식)의 성격을 결정지었다는 점이 핵심입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. X.25 (ITU-T Recommendation X.25)의 심층 구조
X.25는 **DTE (Data Terminal Equipment)**인 컴퓨터와 **DCE (Data Circuit-terminating Equipment)**인 패킷 교환기 간의 인터페이스를 정의합니다. 가장 큰 특징은 계층 2(Layer 2)와 계층 3(Layer 3)에서 강력한 흐름 제어와 에러 복구를 수행한다는 점입니다.

*   **구성 요소 상세**:
    *   **PLP (Packet Layer Protocol, Layer 3)**: 패킷의 다중화(Multiplexing) 및 라우팅 담당. 가상 회선(VC) 번호인 LCN(Logical Channel Number)을 관리합니다.
    *   **LAPB (Link Access Procedure Balanced, Layer 2)**: HDLC(High-level Data Link Control) 기반의 프로토콜. 프레임의 순서 번호(Sequence Number)를 관리하며, 프레임 전송 시마다 ACK(Acknowledgment)를 받지 못하면 재전송(ARQ, Automatic Repeat reQuest)을 수행합니다.

#### 2. 프레임 릴레이 (Frame Relay)의 고속화 메커니즘
프레임 릴레이는 X.25에서 복잡한 에러 제어(Flow/Error Control) 기능을 제거하고, **LAPF (Link Access Procedure for Frame Mode Bearer Services)** Core 기능만 남겨 경량화했습니다. 이를 **'복잡한 라우터(Router) 내부 기능을 단말(Host)으로 떠넘기기'**라고 볼 수 있습니다.

```ascii
+-----------------------------------------------------------------------+
|                     [ 에러 복구 책임 소재 이동 ]                       |
+-----------------------------------------------------------------------+

[ X.25 아키텍처 ]
Source PC ----> [ Node 1: Check/ErrorFix ] --(Error Free)---> [ Node 2 ] --...
                  ^ 여기서 에러를 잡고 고침 (큰 지연 발생)

[ Frame Relay 아키텍처 ]
Source PC ----> [ Node 1: Relay Only ] ---(Fast)---> [ Node 2: Relay ] --...
                  ^ 에러가 있으면 그냥 폐기 (Discard)
                  ^ 재전송은 출발지 PC가 알아서 TCP/IP로 처리함
```

#### 3. 핵심 파라미터 및 동작 원리

| 구분 | 요소 | 설명 및 내부 동작 |
|:---:|:---|:---|
| **주소** | **DLCI (Data Link Connection Identifier)** | 10비트 필드. [0~1023] 범위 사용. 프레임 릴레이 망 내부에서 가상 회선을 식별하는 식별자. **LMI (Local Management Interface)**를 통해 스위치와 매핑됨. |
| **성능** | **CIR (Committed Information Rate)** | 약속된 정보 전송률. 통신사가 보장하는 평균 대역폭 (bps 단위). 이 크기 내의 트래픽은 100% 전송 보장됨. |
| **성능** | **Bc (Committed Burst Size)** | CIR을 기준으로 특정 시간(Tc) 내에 보낼 수 있는 최대 허용 버스트 크기. |
| **정책** | **DE (Discard Eligibility)** | Bc를 초과하여 전송되는 트래픽(Excess Burst)에 붙는 표시. 망 혼잡 시 가장 먼저 폐기됨. |
| **제어** | **FECN/BECN (Forward/Backward Explicit Congestion Notification)** | 혼잡 통지 비트. 수신측(FECN) 또는 송신측(BECN)에게 "망이 붐빈다"고 알려주며 전송 속도를 줄이도록 유도함. |

#### 4. 프레임 릴레이 포맷 (Hex Dump Level)
실제 프레임 릴레이 헤더는 2바이트 구조로 매우 단순합니다.

```ascii
|<-- Flag (1 octet) -->|<----- Address (2 octets) ----->|<- Data ->|<-FCS->|
+---------------------------------------------------------------+
  DLCI (6) | CR | EA | DLCI (4) | FECN | BECN | DE | EA | ...
  (Upper)      |           (Lower)
```

*   **DLCI**: 10비트로 나뉘어 저장됨.
*   **EA (Extended Address)**: 0이면 다음 바이트가 주소의 일부, 1이면 주소 필드 종료.

**📢 섹션 요약 비유**: X.25가 중간중간 택배 기사가 물건이 깨졌는지 확인하고 다시 포장하는 무거운 과정을 거친다면, 프레임 릴레이는 '겉 주소(DLCI)만 확인하고 무조건 보낸 뒤, 물건이 깨지면 보낸 사람이 알아서 다시 보내라'는 방식으로 업무를 분담하여 속도를 높인 것입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. X.25 vs Frame Relay 정량적/구조적 비교

| 비교 항목 | X.25 | Frame Relay |
|:---|:---|:---|
| **개발 시대** | 1970년대 (구리선 위주) | 1990년대 (광케이블 도입) |
| **오류 제어** | **망 담당 (Link-by-Link)**. 노드마다 CRC 및 ARQ 수행. | **단말 담당 (End-to-End)**. 중간 노드는 Relay만 수행. 에러 시 프레임 폐기. |
| **흐름 제어** | **Sliding Window** (XON/XOFF, 패킷 모드) | **혼잡 알림 기반** (FECN/BECN). |
| **전송 속도** | 최대 64Kbps (구형 교환기 기준) | 56Kbps ~ 45Mbps (T3/E3 수준) |
| **지연 시간(Latency)** | 매우 높음 (노드 통과시마다 처리 지연) | 매우 낮음 (스위칭 지연만 발생) |
| **프레임 길이** | 가변 (Packet 단위) | 가변 (Frame 단위) |
| **오버헤드** | 높음 (헤더+Protocol 복잡도) | 낮음 (헤더 2Bytes 최소화) |
| **적용 분야** | POS, 금융망 (높은 신뢰성 필요 시) | 인터넷, LAN 간 연결 (고속성 필요 시) |

#### 2. 타 과목(OS/Network) 융합 관점
프레임 릴레이의 가장 중요한 설계 철학은 **'End-to-End Argument (단대단 인자)'**입니다. 인터넷의 핵심인 TCP(Transmission Control Protocol) 역시 데이터의 신뢰성을 보장하기 위해 **재전송(Retransmission)**과 **순서화(Sequencing)**을 담당합니다. 프레임 릴레이는 물리적 망이 좋아졌다는 전제 하에, 중간 라우터(계층 2/3)의 복잡한 기능을 제거하고, **TCP(계층 4)에게 에러 복구를 완전히 위임**하여 인터넷의 고속화에 기여했습니다. 이는 OSI 7계층에서 각 계층의 독립성과 책임 분리를 보여주는 대표적인 사례입니다.

**📢 섹션 요약 비유**: 마치 현대 자동차가 비포장도로에서는 4륜 구동(X.25)으로 험난한 길을 견디지만, 평탄한 고속도로(광케이블)에서는 2륜 구동(프레임 릴레이)으로 연비와 속도를 올리는 것과 같습니다. 도로 상태(물리 계층)에 따라 차량의 구동 방식(프로토콜)을 다르게 설계하는 전략적 선택입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 기술적/운영적 도입 체크리스트
기업에서 프레임 릴레이 망을 구축할 때 다음 사항을 고려해야 합니다.

| 구분 | 항목 | 고려사항 |
|:---|:---|:---|
| **기술** | **CIR 설계** | CIR을 평균 트래픽보다 낮게 설정하면 비용은 절감되지만 피크 타임에 패킷 손실이 발생함. 평균 대역폭의 1.2~1.5배를 권장. |
| **기술** | **LMI 호환성** | Cisco, ANSI, Q933a 등 LMI 타입이 통신사 장비와 일치해야 DLCI 매핑이 정상적으로 됨. |
| **운영/보안** | **Burst Traffic 처리** | 비즈니스 특성상 간헐적으로 대용량 전송이 발생한다면 CIR + EBS 구성이 필수적임. |

#### 2. 실무 시나리오: 지사 LAN 연결
서울 본사와 5개 지사를 연결하려 할 때, 전용선(Leased Line) 5개를 까는 것보다 통신사의 프레임 릴레이 망에 접속하여 **단 하나의 액세스 회선(Access Line)**으로 **여러 개의 PVC (Permanent Virtual Circuit)**를 설정하여 지사별로 분리된 통신망을 구축하는 것이 비용 효율적입니다.

#### 3. 안티패턴 (Anti-Pattern)
*   **CIR 초과 의존**: "망이 텅 비니까 CIR 넘어도 괜찮겠지"라는 생각은 위험합니다. 망이 혼잡해지는 시간대에 데이터가 유실되어 재전송이 발생하면, 오히려 평균 속도가 CIR 이하로 떨어지는 **Global Synchronization** 현상이 발생할 수 있습니다.

```ascii
      [ 공중망 (Frame Relay Cloud) ]
       /        |          \
    [PVC 1]   [PVC 2]     [PVC 3]
       |         |           |
   (서울)     (부산)      (대구)

하나의 물리적 포트(WAN Interface) 안에 여러 개의 DLCI로 분리된 논리적 경로가 공존.
```

**📢 섹션 요약 비유**: 마치 큰 빌딩(공중망)에 입주한 회사들처럼, 모두가 같은 입구(회선)를 쓰지만 사무실(PVC)은 서로 다르게 배정받아 보안과 트래
