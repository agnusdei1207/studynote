# 188-191. 오류 제어(Error Control)와 FEC/ARQ

> **핵심 인사이트**: 전송 중인 데이터는 반드시 노이즈로 인해 훼손될 수 있다. 이를 극복하는 방법은 수신자가 잉여 데이터를 활용해 스스로 고치거나(FEC), 아니면 틀렸으니 다시 보내라고 요청하는 것(ARQ) 두 가지뿐이다.

---

## Ⅰ. 오류 제어 (Error Control) 개요
물리적인 전송 매체(구리선, 광케이블, 전파 등)를 통과하는 신호는 감쇠, 왜곡, 잡음(Noise)의 영향을 받아 송신자가 보낸 0이 1로, 1이 0으로 뒤바뀌는 **에러(Error)**가 필연적으로 발생합니다. 데이터 링크 계층은 이 에러를 **찾아내고(Detection)** 나아가 **수정(Correction)**하여 상위 계층에 깨끗한 데이터를 넘겨줄 책임이 있습니다.

### 비트 에러율 (BER, Bit Error Rate)
* **개념**: 전송된 전체 비트 수에 대해 에러가 발생한 비트 수의 비율입니다.
* **공식**: $BER = \frac{\text{에러가 발생한 비트 수}}{\text{전송된 총 비트 수}}$
* 네트워크 매체의 품질을 평가하는 가장 직관적인 지표로, 광케이블은 BER이 매우 낮고($10^{-12}$ 이하), 무선 통신은 BER이 상대적으로 높습니다.

---

## Ⅱ. 순방향 에러 수정 (FEC, Forward Error Correction)
송신자가 원본 데이터에 복잡한 수학적 알고리즘으로 계산된 **'잉여 비트(Redundant Bits)'를 잔뜩 붙여서** 전송하는 방식입니다.

* **동작 원리**: 수신자는 이 잉여 비트들을 풀어헤쳐서, 어디서 에러가 났는지 스스로 찾아내고(Detect) 송신자에게 물어보지 않고 **자체적으로 고쳐버립니다(Correct).**
* **장점**: 수신자가 송신자에게 다시 보내달라고 할 필요가 없으므로 **재전송 지연(Delay)**이 발생하지 않습니다.
* **단점**: 에러를 스스로 고치기 위해 보내야 할 '잉여 데이터'의 양이 너무 많아서, 실제 데이터를 보낼 대역폭이 낭비됩니다.
* **용도**: 수신자가 응답(ACK/NAK)을 보낼 수 없는 단방향 통신(위성 방송, TV)이나, 재전송을 하면 시간이 너무 오래 걸려 의미가 없는 초장거리 통신(심우주 탐사 등)에서 주로 쓰입니다.

---

## Ⅲ. 역방향 에러 수정 (BEC) / ARQ
수신자는 에러를 고칠 능력 없이 오직 **'에러가 났는지 안 났는지 확인(Detection)'**만 할 수 있는 최소한의 꼬리표(Parity, CRC 등)만 붙여서 전송하는 방식입니다.

* **동작 원리 (ARQ, Automatic Repeat reQuest)**: 
  수신자가 꼬리표를 검사해 보니 에러가 발견되었습니다. 그러면 수신자는 해당 프레임을 버리고, 송신자에게 **"에러 났으니까 다시 보내!(NAK)"** 라고 자동으로 재전송을 요청합니다.
* **장점**: 잉여 비트를 조금만 붙여도 되므로 네트워크 대역폭 효율이 높습니다.
* **단점**: 에러가 날 때마다 재전송을 해야 하므로 시간이 지연됩니다.
* **용도**: 우리가 쓰는 대부분의 인터넷 환경(이더넷, Wi-Fi, TCP)은 모두 에러 정정을 ARQ(재전송)에 의존하고 있습니다.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[오류 제어 방식 비교: FEC vs ARQ]

1. FEC (Forward Error Correction) : "내가 고칠게!"
   [ 송신자 ] ========(Data + 아주 두꺼운 힌트북)========> [ 수신자 ]
                                                   (힌트북을 풀어서 스스로 복원)

2. ARQ (Automatic Repeat reQuest) : "다시 보내줘!"
   [ 송신자 ] ========(Data + 짧은 검사표)===============> [ 수신자 ]
                                                   (검사해보니 틀림, 파기!)
              <==========(NAK: 오류남, 재전송 요청)========
   [ 송신자 ] ========(동일한 Data 다시 전송)==============>
```

📢 **섹션 요약 비유**: **FEC**는 친구에게 띄어쓰기와 오타 교정을 위해 앞뒤 문맥과 부연 설명을 A4용지 10장 분량으로 잔뜩 써서 편지를 보내는 것입니다. 글씨가 좀 번져도 친구가 문맥으로 알아서 고쳐 읽을 수 있습니다. 반면 **ARQ**는 딱 한 장만 보내고, 친구가 글씨가 번져서 못 읽겠으면 "야, 다시 써서 보내!" 하고 전화를 거는 방식입니다.
