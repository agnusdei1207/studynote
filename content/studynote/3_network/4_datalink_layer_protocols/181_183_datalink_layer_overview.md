# 181-183. 데이터 링크 계층(Data Link Layer)의 역할과 하위 계층

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: **OSI (Open Systems Interconnection)** 7계층 중 2계층에 해당하며, 1계층의 신뢰할 수 없는 물리적 비트 스트림을 논리적이고 신뢰 가능한 전송 단위인 **프레임 (Frame)**으로 캡슐화하고 제어하는 계층입니다.
> 2. **가치**: 인접 노드 간(Node-to-Node)의 전송 오류를 검출/수정하고, 전송 속도를 조절하여 네트워크의 효율성과 안정성을 보장하며, **MAC (Media Access Control)** 주소를 통해 물리적 기기 식별을 수행합니다.
> 3. **융합**: 이더넷(Ethernet), Wi-Fi와 같은 다양한 물리적 매체의 상이성을 **LLC (Logical Link Control)** 부계층을 통해 상위 계층(IP)으로부터 추상화하여, 데이터 링크 계층의 기술이 발전하도(L2 스위칭, 가상화) 하부 구조의 변화에 유연하게 대응합니다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 및 정의
**데이터 링크 계층 (Data Link Layer)**은 물리 계층(Physical Layer, Layer 1)이 제공하는 원시적인 전송 매체 위에서, 네트워크 계층(Network Layer, Layer 3)으로부터 받은 데이터를 **프레임 (Frame)**이라는 논리적 단위로 변환하여 전송하는 역할을 합니다. 물리 계층이 단순히 비트(0과 1)를 전기 신호나 광신호로 바꿔 보내는 '무식한 파이프'라면, 데이터 링크 계층은 이 데이터에 의미를 부여하고, 수신처를 확인하며, 전송 중 발생하는 오류를 복구하는 '지능적인 우체국'과 같습니다.

이 계층은 소프트웨어적인 절차와 하드웨어적인 제어가 결합되는 지점으로, **NIC (Network Interface Card)**와 같은 장치 드라이버 형태로 구현됩니다.

### 💡 비유: 고속도로 톨게이트 시스템
데이터 링크 계층은 고속도로의 톨게이트 시스템과 유사합니다.
*   **1계층(도로)**: 그냥 차가 다니는 길.
*   **2계층(톨게이트)**: 차량(데이터)을 검사하고, 목적지를 확인하며, 차량이 겹치지 않도록 정리(흐름 제어)하고 통행료를 징수하거나 차량 상태를 점검(오류 제어)하는 곳.

### 등장 배경
① **물리 계층의 한계**: 물리 계층만으로는 신호 감쇠, 노이즈(Noise) 등으로 인한 비트 오류를 해결할 수 없으며, 데이터의 시작과 끝을 구분할 수 없어 수신자가 해석 불가능함.
② **신뢰성 있는 전송의 필요**: 컴퓨터 통신의 발달로 단순한 신호 전송을 넘어, 데이터 무결성(Integrity)과 전송 효율성이 필수적인 요구 사항이 되었음.
③ **매체 접근 제어(MAC)의 등장**: 여러 대의 컴퓨터가 하나의 선(공유 매체)을 사용할 때, 누가 말할지 순서를 정하지 않으면 충돌이 발생하여 데이터가 파괴됨. 이를 방지하기 위한 규칙이 필요해짐.

### 📢 섹션 요약 비유
데이터 링크 계층은 **"전기 신호라는 '액자'를 깨지지 않게 포장하고, 받는 사람 주소를 정확히 붙여서 배달하는 물류 센터 팀장"**입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

데이터 링크 계층은 프로토콜의 복잡도를 줄이고 표준화를 유도하기 위해 IEEE 802 표준에 따라 **LLC (Logical Link Control)**와 **MAC (Media Access Control)**라는 두 개의 부계층(Sub-layer)으로 나뉩니다.

### 1. 구성 요소 (상세 테이블)

| 구성 요소 | 전체 명칭 (Abbreviation) | 주요 역할 | 상세 동작 메커니즘 | 주요 프로토콜/표준 | 기능적 비유 |
|:---:|:---:|:---|:---|:---:|:---|
| **LLC** | **Logical Link Control** | 상위 계층 인터페이스 추상화 | 네트워크 계층(IP 패킷)으로부터 데이터를 받아 **SAP (Service Access Point)**를 통해 제어 정보를 추가하고, 하위 MAC 계층으로 전달. 매체 종류(유선/무선)에 상관없이 일관된 서비스 제공. | IEEE 802.2 | **물류 총괄 매니저**: 고객(IP)의 요청을 받아 택배를 준비하고 하부 직원에게 지시. |
| **MAC** | **Media Access Control** | 매체 접근 제어 및 주소 지정 | 물리적 매체(선, 전파)에 실제로 프레임을 전송하기 전 **CSMA/CD**나 **CSMA/CA** 알고리즘을 통해 전송권 확보. **MAC 주소(BIA)**를 기반으로 프레임의 송수신자 식별 및 캡슐화 수행. | IEEE 802.3 (Ethernet)<br>IEEE 802.11 (Wi-Fi) | **창고 현장 반장**: 실제 트럭(매체)에 짐을 싣고, 누가 먼저 싣는지(순서)를 조율하고 바코드(MAC 주소)를 읽음. |

### 2. ASCII 구조 다이어그램: IEEE 802 계층 구조

아래 다이어그램은 데이터 링크 계층이 상위 계층과 하위 계층 사이에서 어떻게 분리되어 동작하는지를 시각화한 것입니다.

```ascii
+-----------------------------------------------------------------------+
| [ 3계층: Network Layer (IP 패킷) ]                                     |
|         |                                                             |
|         +---> (SAP: Service Access Point)                            |
+-------------------------|---------------------------------------------+
| [ 2계층: Data Link Layer ]                                             |
|                         |                                              |
|  +======================|===========================================+  |
|  |  LLC (Logical Link Control) - IEEE 802.2                       |  |
|  |  - 목적지 SAP (DSAP) / 출발지 SAP (SSAP)                        |  |
|  |  - 제어 정보 (Control): 흐름/오류 제어                          |  |
|  +======================|===========================================+  |
|                         |                                              |
|  +======================|===========================================+  |
|  |  MAC (Media Access Control) - IEEE 802.3 / 802.11              |  |
|  |  - MAC 주소 (Source/Dest Address)                              |  |
|  |  - 프레임 체크 시퀀스 (FCS/CRC)                                 |  |
|  |  - 매체 접근 알고리즘 (CSMA/CD, CSMA/CA)                        |  |
|  +======================|===========================================+  |
|                         |                                              |
+-------------------------|---------------------------------------------+
                          v
+-----------------------------------------------------------------------+
| [ 1계층: Physical Layer (비트/전기신호/광신호) ]                        |
+-----------------------------------------------------------------------+
```

#### 다이어그램 해설
위 구조는 데이터가 송신 호스트에서 수신 호스트로 이동할 때의 캡슐화 과정을 보여줍니다.
1.  **LLC 계층**은 네트워크 계층으로부터 전달받은 IP 패킷에 **SAP**(어떤 상위 프로토콜용인지 식별자, 예: IP 번호 0x06) 정보를 덧붙입니다. 이때, 물리적인 매체가 이더넷이든 와이파이든 상관없이 동일한 인터페이스를 제공합니다.
2.  **MAC 계층**으로 내려온 데이터는 이제 **MAC 헤더**와 **트레일러**가 추가됩니다. 헤더에는 송수신자의 **MAC 주소**(48비트 하드웨어 고유 번호)가 포함되고, 트레일러에는 오류 검출을 위한 **FCS (Frame Check Sequence)**가 계산되어 붙습니다.
3.  최종적으로 완성된 **프레임**은 1계층으로 전달되어 물리적 신호로 변환되어 매체를 타고 날아갑니다.

### 3. 핵심 동작 원리 및 기술

#### A. 프레이밍 (Framing) 및 비트 스터핑
데이터 링크 계층은 비트열을 프레임 단위로 묶어야 합니다. 프레임의 시작과 끝을 알리기 위해 **플래그 바이트(Flag Byte, 예: 01111110)**를 사용하는데, 데이터 내부에 이 플래그와 같은 패턴이 나타나면 문제가 됩니다. 이를 해결하기 위해 **비트 스터핑(Bit Stuffing)** 기법을 사용합니다.
*   *동작*: 송신 측은 데이터 부분에 연속된 '1'이 5개 오면 무조건 '0'을 하나 삽입합니다. 수신 측은 '1'이 5개 오면 뒤의 '0'을 제거하고 원래 데이터로 복원합니다.

#### B. 오류 제어 (Error Control) & ARQ (Automatic Repeat reQuest)
전송 중 오류가 발생했는지는 **CRC (Cyclic Redundancy Check)** 등을 통해 확인합니다.
*   **Stop-and-Wait ARQ**: 한 프레임을 보내고 ACK (Acknowledge)를 받을 때까지 기다림. 비효율적.
*   **Go-Back-N ARQ**: 여러 프레임을 연속으로 보내고, 오류가 발생한 프레임부터 다시 모두 전송.
*   **Selective Repeat ARQ**: 오류가 발생한 특정 프레임만 재전송. 가장 효율적이나 버퍼 관리가 복잡함.

#### C. 흐름 제어 (Flow Control)
수신자의 처리 속도가 느려 데이터를 잃어버리는 것을 방지합니다.
*   **Sliding Window**: 수신자가 수용 가능한 버퍼 크기(Window Size)만큼 송신자가 미리 프레임을 보낼 수 있게 하여, 매번 확인 응답을 기다리는 대기 시간을 줄이고 효율을 높입니다.

### 📢 섹션 요약 비유
데이터 링크 계층의 구조는 **"하나는 고객 응대(LLC), 하나는 현장 작업(MAC)으로 나뉘어 운영되는 배송 센터"**와 같습니다. 고객(상위 계층)은 센터가 트럭으로 보내든 비행기로 보내든(MAC 방식) 상관없이 그저 물건을 맡기기만 하면 됩니다(LLC의 역할).

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. LLC vs MAC 심층 기술 비교

| 비교 항목 | LLC (Logical Link Control) | MAC (Media Access Control) |
|:---|:---|:---|
| **핵심 관심사** | "무엇을" 보낼 것인가? (프로토콜 독립성) | "어떻게" 보낼 것인가? (매체 접근) |
| **표준** | IEEE 802.2 | IEEE 802.3 (Ethernet), 802.11 (Wi-Fi), 802.15 (Bluetooth) |
| **하드웨어 의존성** | **Low**: 소프트웨어(드라이버)에서 주로 처리 | **High**: NIC (Network Interface Card) 칩셋에서 처리 |
| **주요 필드** | DSAP, SSAP, Control | Source MAC, Dest MAC, Type/Length, FCS |
| **오류 제어** | 기본적인 흐름 제어 및 오류 복구 논리 관리 | 물리적 충돌 회피 및 재전송 기능 수행 |
| **가상화 기술 연계** | 가상 랜(VLAN) 태깅 시 논리적 그룹 관리 | VLAN ID를 기반으로 실제 포트 분리 및 스위칭 |

### 2. 과목 융합 관점

#### A. 네트워크 (L3)와의 시너지
데이터 링크 계층의 **MAC 주소**는 네트워크 계층의 **IP 주소**와 매핑되어 사용됩니다. **ARP (Address Resolution Protocol)**가 이 역할을 수행합니다. 데이터 링크 계층이 로컬 네트워크(LAN) 내에서의 '정확한 배달'을 보장한다면, 네트워크 계층은 전 세계를 넘나드는 '길 찾기(Routing)'를 담당합니다. 만약 L2 계층의 **MTU (Maximum Transmission Unit)** 설정이 잘못되면, L3 패킷이 조각나(Fragmentation) 성능이 저하되는 오버헤드가 발생합니다.

#### B. 운영체제 (OS)와의 연계
운영체제 커널의 네트워크 스택(Stack)에서 데이터 링크 계층은 인터럽트 처리와 **DMA (Direct Memory Access)** 제어를 담당합니다. 패킷 수신 시 카드가 CPU에 인터럽트를 걸면 OS가 이를 핸들링하여 상위 계층으로 데이터를 넘깁니다. 최신 OS에서는 CPU 부하를 줄이기 위해 하드웨어가 L2 계층 처리(예: Chksum Offload, LRO/LSO)를 대신 수행하는 기능을 활용합니다.

### 📢 섹션 요약 비유
LLC와 MAC의 관계는 **"사람 언어(LLC)와 통신 기기(MAC)"**의 차이와 같습니다. 사용자는 상대방에게 한국어로 "안녕(LLC)"이라고 말하면 되고, 이를 전기 신호(유선), 적외선(무선), 우편(물리) 중 무엇을 이용해 보낼지는 통신사(MAC)가 알아서 결정합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오 및 의사결정

#### 시나리오 A: 공장 자동화 환경에서의 충돌 문제
**상황**: 유선 이더넷 기반의 공장 제어 시스템에서 특정 구간에서 패킷 손실율이 급증하여 로봇 제어가 지연됨.
**원인 분석**: 허브(Hub)를 사용하는 레거시 환경에서 **CSMA/CD (Carrier Sense Multiple Access with Collision Detection)** 방식이 빈번한 충돌을 유발함.
**기술사
