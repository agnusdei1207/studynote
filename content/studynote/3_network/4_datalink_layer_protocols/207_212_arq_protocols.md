# 207-212. 자동 재전송 요청 (ARQ) 프로토콜

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: ARQ (Automatic Repeat reQuest)는 신뢰할 수 없는 물리 계층 위에서 오류 제어(Error Control)와 흐름 제어(Flow Control)를 수행하여 데이터 무결성을 보장하는 핵심 데이터 링크 계층 기술입니다.
> 2. **가치**: 단순한 '정지-대기(Stop-and-Wait)' 방식의 낮은 링크 효율(Utility)을 극복하기 위해 윈도우(Window) 기반의 파이프라이닝을 도입하여, 대역폭 효율을 극대화하고 네트워크 처리량(Throughput)을 획기적으로 향상시킵니다.
> 3. **융합**: TCP (Transmission Control Protocol)의 신뢰성 전송 메커니즘의 근간이 되며, 현재의 5G/6G 무선 통신 환경에서 하이브리드 ARQ(HARQ) 형태로 진화하여 물리적 계층(MAC Layer)과 밀접하게 결합하여 동작하고 있습니다.

---

### Ⅰ. 개요 (Context & Background)

**개념 및 철학**
ARQ (Automatic Repeat reQuest)는 데이터 전송 중에 발생하는 비트 오류(Bit Error)나 프레임 손실(Frame Loss)을 수신 측이 감지하고, 송신 측에게 자동으로 재전송을 요청하여 신뢰성을 확보하는 기술입니다. 이는 종단 간(End-to-End) 신뢰성을 위한 TCP의 기반이자, 무선 구간에서 필수적인 오류 제어 수단입니다.

**등장 배경 및 패러다임 변화**
초기의 단순한 '정지-대기(Stop-and-Wait)' 방식은 RTT (Round Trip Time)가 긴 환경에서 링크 활용도가 1% 이하로 떨어지는 치명적 한계가 있었습니다. 이를 극복하기 위해 '슬라이딩 윈도우(Sliding Window)' 개념이 도입되었고, 단순히 연속으로 보내는 것(Go-Back-N)에서 에러난 프레임만 선별적으로 재전송하는(Selective Repeat) 고도화된 알고리즘으로 발전했습니다. 현재는 혼잡 제어(Congestion Control)와 결합된 형태로 인터넷의 근간을 이루고 있습니다.

**💡 비유**
데이터 전송은 소포 배송과 같습니다. 중간에 소포가 찢어지거나(오류) 사라지면(분실), 배송 담당자가 알아서 다시 보내주지 않으면 고객(수신자)은 물건을 받을 수 없습니다. ARQ는 이 "다시 보내주세요"라는 절차를 자동화한 시스템입니다.

```ascii
     [송신자]                                  [수신자]
    (Source)                                  (Sink)
        |                                        |
        |  ----(1) 데이터 패킷 전송----->         |
        |                                        |
        |  X----(2) 노이즈로 인한 손상/분실----> X  (오류 감지)
        |                                        |
        |  <----(3) NAK 또는 타임아웃------       |
        |                                        |
        |  ----(4) 재전송(Retransmission)----->   |
        |                                        |
```
*해설: 위 다이어그램은 ARQ의 가장 기본적인 피드백 루프(Feedback Loop)를 보여줍니다. 송신자는 데이터를 보내고 대기하며, 수신자는 CRC (Cyclic Redundancy Check) 등을 통해 오류를 검출합니다. 오류 발생 시 NAK (Negative Acknowledgement)를 보내거나, 송신자는 타이머 만료(Time-out)를 통해 데이터 유실을 감지하고 재전송을 수행합니다.*

**📢 섹션 요약 비유**: ARQ 시스템은 마치 **소포를 보내고 송장 번호로 도착 여부를 확인하는 택배 시스템**과 같습니다. 만약 물건이 분실되었다면(에러), 택배 기사가 알아서 찾아서 다시 배달해주는(재전송) 시스템입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

ARQ의 효율성을 결정짓는 핵심은 **'슬라이딩 윈도우(Sliding Window)'** 메커니즘과 재전송 시점을 결정하는 **'제어 프레임(Control Frame)'**의 처리 방식입니다.

#### 1. 구성 요소 및 상세 비교

| 구분 | Stop-and-Wait ARQ | Go-Back-N (GBN) ARQ | Selective Repeat (SR) ARQ |
|:---:|:---:|:---:|:---:|
| **윈도우 크기** | $W=1$ | $W < 2^N - 1$ (N: 시퀀스 비트) | $W \le 2^{N-1}$ |
| **송신 버퍼** | 1개 (1 프레임만 저장) | $N$개 (윈도우 내 프레임 저장) | $N$개 (미확인 프레임 저장) |
| **수신 버퍼** | 1개 (순서대로만 처리) | **1개** (순서가 맞지 않으면 버림) | **$N$개** (순서가 틀려도 보관) |
| **오류 복구** | 타임아웃 시 유실된 1개만 재전송 | 누락된 프레임부터 **모두 재전송** | 누락된 **특정 프레임만 선택 재전송** |
| **링크 효율** | $\frac{1}{1+2a}$ (매우 낮음) | 높음 (순차 전송 유지) | 최고 (비순차적 수신 허용) |
| **비용/복잡도** | 구현 단순, 메모리 적음 | 수신측 버퍼 로직 단순함 | **양쪽 모두 버퍼/정렬 로직 복잡** |
| **주요 용도** | 초근거리 통신, 간단한 프로토콜 | TCP 표준 (구현 용이성) | 고위성 통신, 고속 네트워크 |

> *$a = \frac{Propagation\ Time}{Transmission\ Time}$ (전파 지연 대 전송 시간의 비율)*

#### 2. 핵심 제어 매커니즘: 피기배킹 (Piggybacking) 및 NAK

**피기배킹 (Piggybacking)**
양방향 통신(Full-Duplex) 환경에서 수신자가 별도의 ACK 프레임을 보내는 대신, 자신이 보내야 할 데이터 프레임의 헤더에 ACK 정보를 실어 보내는 기법입니다. 이는 대역폭(Bandwidth)을 절약하고 오버헤드(Overhead)를 줄이는 효율적인 방식입니다.

**NAK (Negative Acknowledgement) vs Timeout**
- **NAK**: 수신자가 명시적으로 "에러가 났다"고 알림. 송신자는 즉시 재전송 가능하여 대기 시간을 줄일 수 있음.
- **Timeout**: NAK가 손실되었을 경우를 대비한 최후의 보험. 타이머가 만료되면 해당 프레임을 무조건 재전송.

#### 3. ASCII 아키텍처: 슬라이딩 윈도우 및 버퍼 상태

```ascii
[송신자 버퍼 상태 (GBN 예시, Base=1, NextSeqNum=5)]
┌───────────────────────────────────────────────────────┐
│  [송신 완료, ACK 대기 중]     │        [전송 가능]      │
│  Frame 1  Frame 2  Frame 3   │  Frame 4  Frame 5      │
│   (Sent)   (Sent)   (Sent)   │   (Usable)  (Usable)   │
└───────────────────────────────────────────────────────┘
   ↑ Base                     ↑ NextSeqNum          ↑ Base+N

[수신자 버퍼 상태 (GBN vs SR 비교)]

[Case A: Go-Back-N (GBN)]           [Case B: Selective Repeat (SR)]
수신 버퍼 크기 = 1                   수신 버퍼 크기 = N
┌──────────────┐                   ┌───────────────────────┐
│   Seq 3 (O)  │                   │ Seq 3 (X - 재전송요청) │ <- 예상
└──────────────┘                   │ Seq 4 (O - 버퍼링)    │ <- 보관 중
                                   │ Seq 5 (O - 버퍼링)    │ <- 보관 중
                                   └───────────────────────┘
```
*해설: 위 다이어그램은 각 ARQ 방식의 핵심 차이인 '수신 버퍼'의 동작을 시각화했습니다. GBN(왼쪽)은 수신자가 오직 순차적인 패킷(여기서는 3번)만 받아들이며, 4, 5번이 도착해도 버퍼 공간이 없으므로 폐기(Discard)합니다. 반면 SR(오른쪽)은 수신자에게 버퍼가 있어, 3번이 누락되더라도 4, 5번을 저장하고 있다가 3번이 도착하면 순서대로 재조립(Reassembly)하여 상위 계층으로 전달합니다.*

#### 4. 핵심 알고리즘 및 수식

**효율성 계산 (Utilization)**
정지-대기 방식의 효율 $\eta$은 다음과 같습니다.
$$ \eta = \frac{1}{1 + 2a} $$
여기서 $a$가 크면(전파 지연이 길면) 효율이 급격히 떨어집니다.
파이프라이닝(GBN/SR)을 사용할 경우 효율은 다음과 같이 극대화됩니다.
$$ \eta \approx 1 \quad (\text{when } W \gg 2a) $$

**심층 동작 원리**
1.  **송신 (Send)**: 윈도우 크기($W$)만큼 연속적으로 프레임 전송.
2.  **수신 및 검사 (Receive & Check)**: 수신자는 시퀀스 번호(Sequence Number) 확인.
3.  **오류 처리 (Error Handling)**:
    -   **GBN**: 수신자는 `ExpectedSeqNum`과 일치하지 않는 프레임을 수신하면 모두 버리고, 마지막으로 성공적으로 받은 프레임의 `ACK`를 계속 보냄(Cumulative ACK). 송신자는 타임아웃 후 `Base`부터 `NextSeqNum-1`까지 전부 재전송.
    -   **SR**: 수신자는 누락된 프레임에 대해 `NAK`를 보내거나 `Selective ACK(SACK)`를 보냄. 송신자는 해당 프레임만 재전송.

**📢 섹션 요약 비유**: GBN은 **연필을 떨어뜨리면 그 뒤에 쓴 모든 페이지를 다시 찢고 처음부터 다시 쓰는** 작문 방식입니다. SR은 **수정 테이프를 활용하여 틀린 글자 부분만 정확하게 지우고 다시 쓰는** 방식입니다. 당연히 후자가 효율적이지만 수정 테이프(버퍼)를 항상 준비해야 하는 비용이 듭니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

ARQ 방식은 단순히 데이터 링크 계층의 문제를 넘어, 전체 네트워크 성능과 밀접한 관련이 있습니다.

#### 1. 심층 기술 비교: 정량적 분석

| 비교 항목 | Stop-and-Wait | Go-Back-N (GBN) | Selective Repeat (SR) |
|:---|:---:|:---:|:---:|
| **최대 링크 이용률** | 낮음 (Long RTT에서 치명적) | 높음 (윈도우 크기에 비례) | 최고 (오버헤드 최소화) |
| **재전송 트래픽 양** | 1 프레임 | $N$ 프레임 (중복 전송 다수) | 1 프레임 (필요시만) |
| **수신측 버퍼 요구사항** | 없음 (거의) | 1 프레임 (버퍼링 불필요) | $N$ 프레임 (대용량 버퍼 필수) |
| **순차 전송 보장 (In-Order Delivery)** | 완전 보장 | 완전 보장 (버퍼링 불필요) | 재조립(Reordering) 로직 필요 |
| **구현 복잡도** | 매우 낮음 ($O(1)$) | 낮음 ($O(N)$) | 높음 ($O(N \log N)$ - 정렬/버퍼 관리) |

#### 2. 타 영역(Subject) 융합 분석

**1) 네트워크 (Network) - TCP와의 관계**
TCP의 신뢰성은 기본적으로 GBN 방식의 변형입니다. TCP는 'Cumulative ACK'를 사용하여, 누락된 세그먼트 이후의 모든 패킷을 재전송하는 Fast Retransmit 메커니즘을 사용합니다. 다만, 최신 OS의 TCP 스택은 `SACK (Selective ACK)` 옵션을 사용하여 SR ARQ의 장점을 일부 흡수하여, 수신자가 정확히 어떤 블록을 받았는지 송신자에게 알려줌으로써 불필요한 재전송을 줄입니다.

**2) 컴퓨터 구조 (Architecture) - Memory & Cache**
CPU의 캐시(Cache) 일관성 유지나 메모리 인터럽트 처리에서도 유사한 개념이 발견됩니다.
-   **Stop-and-Wait**: 싱글 스레드 동기 처리(Synchronous Call)와 유사. 하나의 작업이 끝날 때까지 대기.
-   **Pipelining (GBN/SR)**: CPU의 파이프라이닝(Pipelining)이나 아웃오더 실행(Out-of-Order Execution)과 유사. SR ARQ의 수신 버퍼링과 재조립은 CPU가 명령어를 순서에 맞게 재배열(Reorder Buffer)하여 완결성을 유지하는 것과 같은 원리입니다.

```ascii
[TCP Header의 SACK (Selective ACK) 옵션 예시]
┌──────────────────────────────────────────────────────────────┐
│ ... Kind=5 (SACK) Length=10 ┤ Left Edge=1000 │ Right Edge=1500│
│ ... Kind=5 (SACK) Length=10 ┤ Left Edge=3000 │ Right Edge=3500│
└──────────────────────────────────────────────────────────────┘
(의미: 1000
