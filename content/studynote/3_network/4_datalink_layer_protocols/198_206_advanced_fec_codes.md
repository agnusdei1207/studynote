# 198-206. 고급 오류 정정 기술 (FEC와 HARQ)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**= 전송로의 노이즈(Noise)로 인한 비트 열화를 수학적 잉여성(Redundancy)을 통해 수신측에서 자체적으로 복원하는 **FEC (Forward Error Correction)** 기술과, 복원 실패 시 재전송을 결합한 **HARQ (Hybrid Automatic Repeat Request)**의 메커니즘.
> 2. **가치**= 무선 채널의 낮은 신뢰성을 극복하여 **0.0001% 이하의 BER (Bit Error Rate)**을 달성하고, 재전송 오버헤드를 최소화하여 5G 및 Wi-Fi 6의 **Gbps급 초고속 처리량**을 보장하는 핵심 원천 기술.
> 3. **융합**= 무선 통신(Physical Layer)의 변조 방식과 결합하여 채널 용량을 극대화하며, 데이터 링크 계층의 흐름 제어(Flow Control)와 TCP 재전송 효율을 결정짓는 **OSI 계층간 시너지**의 핵심.

### Ⅰ. 개요 (Context & Background)

통신망은 이상적인 환경이 아닌, 열 잡음(Thermal Noise), 간섭(Interference), 페이딩(Fading) 등으로 가득 찬 적대적인 환경입니다. 약간의 오류가 발생했을 때 이를 감지하고 폐기하는 수동적인 방식(Parity Bit, CRC)은 너무 느리거나 비효율적입니다. 이를 해결하기 위해 수신측이 오류를 스스로 수정할 수 있는 정보를 미리 보내는 기술이 **FEC (Forward Error Correction)**입니다.

FEC는 데이터에 $k$ 비트의 정보에 $n-k$ 비트의 패리티(Parity)를 더해 $n$ 비트의 코드워드(Codeword)를 만드는 과정인 **채널 코딩(Channel Coding)**을 기반으로 합니다. 우주 통신(Voyager)에서 시작된 이 기술은 모뎀의 **변조(Modulation)** 기술과 투투(Two-Two)로 발전하며, 유선(XDSL)에서 무선(5G, Wi-Fi 6)으로 그 영역을 확장했습니다.

```ascii
[통신 시스템에서의 오류 제어 위치]

발신측                           수신측
------+-------+               +-------+------+
 데이터|  코딩부 |--(전송)---->|  복호부|---> 데이터
      |  (ECC) |   (Noise)   |  (ECC) |
------+-------+               +-------+------+
      +                       +
      | CRC (오류 검출용)      | CRC (검출)
      | FEC (오류 정정용)      | FEC (정정)
      
(병렬 구조): FEC가 먼저 오류를 수정하고, 
남은 오류는 CRC가 감지하여 상위 계층(Arq)으로 처리 위임
```

*도입 서술*: 위 다이어그램은 통신 시스템의 코딩 블록을 도식화한 것입니다. 데이터는 출발하기 전에 코딩부를 거쳐 방탄 조끼(패리티 비트)를 입습니다. 노이즈가 섞인 채로 수신측에 도착하면, 복호부는 이 조끼를 이용해 총알 구멍(오류)을 메우려 시도합니다. FEC가 실패하면 상위 계층인 TCP가 재전송을 요청하게 됩니다.

**💡 비유**: 마치 배우가 대본을 외워서 무대에 서는 것과 같습니다. 앵커(ARQ) 방식은 말을 더듬을 때마다 대본을 보고 다시 말하는(재전송) 방식이지만, FEC는 애초에 대사를 110% 외워서 나와서 혀가 꼬여도 문맥으로 즉각적으로 복구하는 '애드리브 실력'을 갖춘 것입니다.

**등장 배경**:
1.  **기존 한계**: ARQ만으로는 위성 통신처럼 신호 왕복 시간(RTT)이 긴 환경에서 심각한 대기 시간(Latency) 발생.
2.  **혁신적 패러다임**: 1948년 클로드 샤논의 **A Mathematical Theory of Communication**으로 증명된 '채널 용량 이론'에 기반하여, 오류를 완벽히 수정할 수 있는 이론적 가능성 제시.
3.  **현재 비즈니스 요구**: 5G, 자율주행차 등 1ms 미만의 지연 시간과 99.999%의 신뢰성이 요구되는 URLLC(Ultra-Reliable Low Latency Communications) 환경에서 필수적 요소.

**📢 섹션 요약 비유**: "오류 정정 기술은 마치 배송되는 유리 세라믹 그릇(데이터)을 포장할 때, 뻣뻣한 스티로폼(패리티 비트)으로 감싸서 운반 과정에서 생긴 금이 간 부분을 수령자가 현장에서 마법으로 복구하는 과정입니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 블록 코드 (Block Codes) 구성
블록 코드는 $k$ 비트의 메시지 블록을 독립적으로 $n$ 비트의 코드워드로 변환합니다. 대표적인 해밍 코드와 리드-솔로몬 코드의 구조를 분석합니다.

| 구성 요소 | 해밍 코드 (Hamming Code) | 리드-솔로몬 코드 (Reed-Solomon Code) | 골레이 코드 (Golay Code) |
|:---:|:---|:---|:---|
| **기본 단위** | 비트 (Bit) | 심볼 (Symbol, m-bit) | 비트 (Bit) |
| **수학적 원리** | 2의 멱수를 이용한 패리티 행렬 | 갈루아 필드(Galois Field) 상의 다항식 연산 | 이진 23차원 공간 |
| **정정 능력** | 1비트 정정 / 2비트 검출 | $t$개 심볼 오류 정정 (버스트 에러 강함) | 3비트 정정 (23비트 중) |
| **오버헤드** | 적음 ($2^m - 1$ 비트 길이) | 큼 (N, K 설정에 따라 가변) | 고정 (코드율 약 0.5) |
| **주요 용도** | ECC RAM, 버스 내부 | CD/DVD, QR코드, DSL, 위성 통신 | 심우주 통신 (Voyager) |

```ascii
[리드-솔로몬 코드의 데이터 구조 예시 (N, K) = (255, 223)]

+----------------------- 데이터 부분 (K) -----------------------+--- 패리티 (2t) ---+
|  223 Bytes (Original Data)                                  |   32 Bytes (P)   |
+-------------------------------------------------------------+------------------+
                            ↑
                        인코딩 과정
-------------------------------------------------------------
                         ↓
+---------------------------- 전송되는 패킷 (N) ---------------------------+
| [Data_1] [Data_2] ... [Data_223] [Parity_1] ... [Parity_32]             |
+--------------------------------------------------------------------------+
                (버스트 에러 발생 시, 패리티 부분을 통해 Data 손실 복구 가능)
```
*도입 서술*: 리드-솔로몬 코드는 데이터를 바이트 단위의 '심볼'로 취급합니다. 위 다이어그램과 같이 패리티를 뒤에 붙이지만, 이 단순한 패리티가 32바이트가 넘는 데이터 손실까지 복원할 수 있는 강력한 연산 능력을 가집니다. 긁힌 CD를 읽을 수 있는 것이 바로 이 기술 덕분입니다.

### 2. 길쌈 코드 및 터보 코드 (Convolutional & Turbo)
블록 코드가 텍스트를 문단 단위로 끊어서 검사한다면, 길쌈 코드는 흐름에 초점을 맞춥니다.

```ascii
[길쌈 인코더 (k=3, Rate 1/2) 구조 예시]

입력 비트 스트림 (t시점) --->(+)--->[x]-- 출력 1 (상위 비트)
       ^    ^             ^   ^
       |    |             |   |
      [ ]  [ ]   (메모리, D Flip-Flop)  x
       |    |             |   |
       v    v             v   v
      (+)--->[x]-- 출력 2 (하위 비트)
       
(<--- 시간의 흐름)
이전 2비트의 상태와 현재 비트를 XOR 연산하여 2비트를 출력.
(Rate 1/2: 1비트 입력당 2비트 출력 -> 오버헤드 2배)
```
*도입 서술*: 위 다이어그램은 기본적인 길쌈 인코더를 보여줍니다. 시프트 레지스터(메모리)를 통해 이전 데이터가 현재 데이터에 영향을 주며 출력됩니다. 수신측은 **비터비 알고리즘(Viterbi Algorithm)**을 사용하여 이 거미줄 같은 경로를 거슬러 올라가 가장 가능성 높은 원래 경로를 추적(Trellis Decoding)합니다.

**터보 코드 (Turbo Code)**는 이 길쌈 인코더를 두 개 사용하여(Parallel Concatenation) 서로의 결과물을 교환해가며 복호 성능을 극한으로 끌어올린 방식입니다. 이론적 한계인 **샤논 한계(Shannon Limit)**에 매우 근접하는 성능을 자랑하며 3G/4G 시대를 열었습니다.

### 3. 현대의 LDPC 및 폴라 코드 (5G Standard)
데이터 전송량이 기가바이트 단위로 넘어가면서 연산 복잡도를 낮추고 병렬 처리가 가능한 코드가 필요해졌습니다.

**A. LDPC (Low Density Parity Check Code)**
희소 행렬(Sparse Matrix)을 기반으로 하는 확률적 코드입니다. 각 비트 노드가 서로 메시지를 주고받으며 본인의 값을 0일지 1일지 결정하는 **Belief Propagation (신뢰 전파)** 알고리즘을 사용합니다.

```ascii
[LDPC의 패리티 검사 행렬 (Tanner Graph) 개념도]

  비트 노드 (Variable Nodes)          검사 노드 (Check Nodes)
   (데이터 비트)                         (패리티 규칙)
    +      +      +      +                +      +
    | b1   | b2   | b3   | b4            | c1    | c2
    +      +      +      +                +      +
     \    /  \  /    \  /                /  \    /  \
      \  /    \/      \/                /    \  /    \
       \/     /\      /\               /      \/      \
      /\     /  \    /  \             /      /\       \
     /  \   /    \  /    \           /      /  \       \
    +    + +      +       +         +      +    +       +
    | c1   | c2   | c3     |       |  Rule | Rule      |
    +      +      +       +         +      +      +
    
(각 검사 노드는 연결된 비트들의 합이 0이 되어야 한다는 규칙을
반복적으로 전파하며 오류를 정정함)
```

**B. 폴라 코드 (Polar Code)**
채널을 얼음처럼 차가운(신뢰도 0%) 채널과 뜨거운(신뢰도 1%) 채널로 완전히 분극(Polarization)시켜서, 정보 비트는 좋은 채널에만 할당하고 나머지는 고정된 값(Frozen Bit)으로 채워버리는 기술입니다. 짧은 길이의 메시지(제어 신호 등)에 매우 강합니다.

**핵심 파라미터 및 코드 예시**:
```python
# [LDPC 성능 지표 예시]
# Rate = k / n (코드율, 전송 효율)
# Eb/N0 (비트당 에너지 대 잡음 스펙트럼 밀도비) - 낮을수록 좋음
# Shannon Limit: Eb/N0 = -1.6 dB (BPSK 기준)

# 5G NR (New Radio) LDPC 설정
BG1_BaseGraph = 1 # 대용량 블록 (K > 640)
Zc = 384 # Lifting Size (병렬 처리를 위한 기본 단위)
# H 행렬은 Zc 크기의 서브행렬로 구성되어 고속 병렬 연산 가능
```

**📢 섹션 요약 비유**: "길쌈 코드는 끊임없이 흐르는 강물의 물결을 망원경으로 관찰하는 것처럼 연속적이지만, LDPC는 수천 명의 사람들이 각자 자신의 이웃과만 대화하며 전체 소문을 바로잡아가는 '점진적 민주주의' 시스템과 같습니다. 폴라 코드는 반 친구들 중 확실한 애들만 과제를 맡기고 나머지는 면제시키는 철저한 능력주의 배분입니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 고급 오류 정정 기술 심층 비교
| 비교 항목 | 블록 코드 (Block) | 길쌈/터보 코드 (Convolutional) | LDPC (5G Data) | 폴라 코드 (5G Control) |
|:---:|:---|:---|:---|:---|
| **코딩 구조** | 대수학적 행렬 (Algebraic) | 시스템 함수 (Systematic) | 희소 행렬 (Graph-based) | 채널 분극 (Recursive) |
| **에러 유형** | 랜덤/버스트 복합 | 랜덤 에러에 강함 | 랜덤 에러 극강 | 랜덤 에러 강함(단문) |
| **연산 복잡도** | 낮음 (단순 행렬 연산) | 높음 (비터비 복잡도 $O(2^v)$) | 높지만 **병렬화 용이** | 중간 (Successive Cancel) |
| **지연 시간** | 짧음 (블록 단위) | 길음 (트렐리스 추적) | 짧음 (H/W 가속) | 매우 짧음 |
| **주요 사용처** | 메모리(ECC), 저장 매체 | 3G/4G 데이터, 위성 | **5G eMBB (데이터)** | **5G URLLC (제어)** |

### 2. 과목 융합 관점 (Linkage)
- **운영체제(OS)와의 관계**: **TCP/IP 스택**에서 오류가 발생하면 TCP 레벨에서 패킷 재전송이 발생합니다. 물리 계층(PHY)의 FEC 성능이 좋지 않으면 상위 계층(T
