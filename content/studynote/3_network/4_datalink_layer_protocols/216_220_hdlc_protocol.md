# 216-220. HDLC 프로토콜의 구조와 동작 모드

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**= **HDLC (High-Level Data Link Control)**는 OSI 7계층의 데이터 링크 계층(Data Link Layer)에서 가장 널리 사용되는 **비트 지향형(Bit-oriented)** 프로토콜로, 비트 스터핑(Bit Stuffing) 기법을 통해 데이터의 투명성을 보장합니다.
> 2. **가치**= 신뢰성 있는 전송을 위해 ** sliding window (슬라이딩 윈도우)** 기반의 효율적인 흐름 제어와 **ARQ (Automatic Repeat reQuest)** 기반의 오류 제어를 하나의 프레임 내에서 통합하여 처리하는 고성능 프로토콜입니다.
> 3. **융합**= 현대의 **PPP (Point-to-Point Protocol)**나 프레임 릴레이(Frame Relay)의 모체가 되는 기술로, LAN의 이더넷과 WAN의 전용선 프로토콜을 잇는 가교 역할을 수행했습니다.

### Ⅰ. 개요 (Context & Background)

**HDLC (High-Level Data Link Control)**는 **ISO (International Organization for Standardization)**에서 표준화한 데이터 링크 계층의 통신 규약입니다. 과거의 문자 지향형(Character-oriented) 프로토콜인 BISYNC 등이 특정 문자 코드(예: ASCII)에 의존하여 투명성이 떨어지는 문제를 해결하기 위해, 모든 데이터를 비트 단위의 연속적인 흐름으로 취급하는 **비트 동기식(Bit Synchronous)** 방식을 채택했습니다.

이 프로토콜은 단순한 전송을 넘어, 물리 계층(Physical Layer)의 오류를 극복하고 네트워크 계층(Network Layer)에 신뢰할 수 있는 서비스를 제공하는 핵심적인 역할을 합니다. 특히, 전이중(Full-Duplex) 통신 환경에서 데이터 전송과 제어 정보를 동시에 처리할 수 있는 구조를 가지고 있어, 고효율의 WAN 구축에 필수적인 요소로 자리 잡았습니다.

#### 배경 및 필요성
1.  **기존 한계**: 초기 프로토콜들은 제어 문자가 데이터 내에 포함될 경우 이를 구분하지 못하는 문제가 있었습니다.
2.  **혁신적 패러다임**: 'Flag Sequence'라는 고유한 비트 패턴과 'Bit Stuffing' 기법을 도입하여, 데이터 내용에 무관하게 프레임의 시작과 끝을 명확히 인식하는 투명한 데이터 전송을 실현했습니다.
3.  **비즈니스 요구**: 급증하는 네트워크 트래픽과 장거리 통신에서의 높은 신뢰성 요구에 따라, 효율적인 오류 제어와 흐름 제어가 가능한 표준 규격의 필요성이 대두되었습니다.

📢 **섹션 요약 비유**: HDLC는 마치 고속도로의 '스마트 톨게이트 시스템'과 같습니다. 과거에는 차량의 종류(문자)를 일일이 확인해 통행료를 받았다면, HDLC는 차량이나 화물의 내용물(데이터)에 상관없이, 차량이 통과하는 시점에 고유한 태그(Flag)만으로 진입/퇴장을 자동으로 처리하고, 동시에 교통 상황(흐름 제어)과 사고 여부(오류 제어)까지 실시간으로 관리하는 시스템입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

HDLC의 아키텍처는 통신에 참여하는 노드의 역할과 그들이 맺는 관계(모드)에 따라 결정됩니다. 이는 네트워크 토폴로지 설계의 기본이 되는 중요한 요소입니다.

#### 1. 국(Station)의 종류 및 역할

| 구분 | 정의 | 역할 | 비유 |
|:---:|:---|:---|:---|
| **Primary Station (주국)** | 링크의 제어를 주도하는 스테이션 | 명령(Command) 프레임을 전송하고, 링크 전체의 복구(Recovery)와 흐름 제어를 총괄함. | **지휘관** (명령권자) |
| **Secondary Station (종국)** | 주국의 제어를 받는 스테이션 | 주국의 명령(Poll)에 대해서만 응답(Response)할 수 있으며, 자발적인 전송은 불가능함. | **사병** (명령 수행자) |
| **Combined Station (혼성국)** | 주국과 종국의 기능을 모두 가짐 | 명령과 응답을 모두 생성할 수 있으며, 링크 제어에 대해 대등한 권한을 가짐. | **대표** (대등한 협력자) |

#### 2. 링크 동작 모드 (Link Configuration Modes)

```ascii
[ HDLC 동작 모드 간의 관계도 ]

        [비동기식/균형/점대점]
        +-----------------------+
        |   ABM (Asynchronous Balanced Mode)     |
        |   Primary <==(Combined)==> Primary    |   (현대 인터넷 표준)
        +-----------------------+
                     ▲
                     | (진화)
                     |
        +-----------------------+
        |   NRM (Normal Response Mode)          |
        |   Primary <========> Secondary        |   (다중점, 멀티드롭)
        |   (Poll/Final 절차 필요)               |
        +-----------------------+

        [ARM (Asynchronous Response Mode): NRM과 유사하나 Secondary의 자발 전송 허용]
```

**상세 해설**:
1.  **NRM (Normal Response Mode, 정규 응답 모드)**: 1개의 주국과 여러 종국으로 구성된 **다중점(Multi-point)** 회선에서 주로 사용됩니다. 종국은 주국의 폴링(Polling)比特을 받을 때만 전송 권한을 획득하여 응답합니다. 기본적으로 불평등한 관계입니다.
2.  **ARM (Asynchronous Response Mode, 비동기 응답 모드)**: 주국과 종국의 관계는 유지하되, 종국이 주국의 폴링 없이도 자발적으로 데이터를 전송할 수 있습니다. 하지만 여전히 주국이 논리적인 제어권을 쥐고 있습니다.
3.  **ABM (Asynchronous Balanced Mode, 비동기 균형 모드)**: 두 **혼성국(Combined Station)** 간의 **점대점(Point-to-Point)** 통신에 사용됩니다. 양쪽이 모두 대등한 권한을 가지며, 별도의 허가 절차 없이 언제든지 데이터를 주고받을 수 있습니다. **PPP (Point-to-Point Protocol)** 등 현대 WAN 프로토콜이 채택하는 표준 모드입니다.

#### 3. HDLC 프레임 구조와 비트 스터핑 (Bit Stuffing)

HDLC의 핵심 메커니즘은 프레임의 시작과 끝을 나타내는 고유한 패턴을 사용하면서, 데이터 필드 내에 동일한 패턴이 나타나는 것을 방지하는 기술입니다.

```ascii
[ HDLC 프레임 구조 (Standard Frame Format) ]

 8 Bits   :  8/16 Bits :  8/16 Bits  : Variable Length : 16/32 Bits : 8 Bits
+---------+-----------+-------------+----------------+------------+---------+
|  Flag   | Address   |  Control    |  Information   |   FCS      |  Flag   |
|01111110 | (주소)    |  (제어)     |  (데이터)      | (CRC)      |01111110 |
+---------+-----------+-------------+----------------+------------+---------+
   ▲                                                    ▲
   └─────────────── 투명성 보장을 위한 Bit Stuffing ─────────────┘

[ 비트 스터핑(0-bit Insertion) 원리 ]

Sender (송신측)                              Receiver (수신측)
   |                                            |
   v                                            v
11011111111000111111...                   11011111111000111111...
   |                                            |
[Scan 5 consecutive 1s]                  [Scan 5 consecutive 1s]
   |                                            |
   v                                            v
11011111011000111111...                   (Stuffed '0' removed)
(강제로 '0' 삽입)                            |
   |                                            v
   |                                    11011111111000111111...
   |                                    (Original Data Restored)
```

**심층 동작 원리**:
*   **Flag (플래그)**: `01111110` (Hex 7E) 패턴을 사용하여 프레임의 경계를 정의합니다.
*   **Bit Stuffing (비트 스터핁)**: 송신측은 Flag 패턴과 혼동을 방지하기 위해, **데이터 부분 내에 연속된 '1'이 5개가 나타나면 무조건 '0'을 강제로 삽입**합니다. 수신측은 연속된 '1'이 5개 나오면 뒤의 '0'이 stuffing 비트임을 인지하고 이를 제거한 뒤 데이터를 복원합니다.
*   **Address (주소 필드)**: NRM/ARM에서는 종국의 주소를 식별하지만, ABM(점대점)에서는 상대방 주소를 식별하는 데 사용됩니다.
*   **Control (제어 필드)**: 프레임의 유형(I/S/U)과 송수신 순서 번호(N(S), N(R))를 포함하는 뇌부에 해당합니다.
*   **FCS (Frame Check Sequence)**: 주로 **CRC-CCITT (Cyclic Redundancy Check)** 다항식을 사용하여 프레임 전체의 무결성을 검증합니다.

📢 **섹션 요약 비유**: HDLC의 비트 스터핑은 마치 '구슬 공예'에서 '색상 분리 탭'을 넣는 것과 같습니다. 투명한 유리구슬(0)과 불투명한 구슬(1)을 꿰다가, 불투명한 구슬이 연속으로 5개 나오면 이것이 끝 부분(Flag)인지 실제 데이터인지 헷갈리지 않도록, 의도적으로 다른 색의 탭(0)을 하나 끼워 넣어 물리적/논리적 구분을 명확히 하는 과정과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

HDLC는 이론적 완성도가 높지만, 실무 환경에서는 다른 프로토콜과의 상호작용과 비교를 통해 선택적으로 사용됩니다.

#### 1. 프레임 유형별 심층 기술 비교

Control 필드의 포맷에 따라 HDLC 프레임은 세 가지로 분류됩니다.

| 구분 | I-Frame (Information) | S-Frame (Supervisory) | U-Frame (Unnumbered) |
|:---:|:---|:---|:---|
| **기능** | **데이터 전송 + 제어** (Piggybacking) | **흐름 제어 + 에러 제어** | **링크 관리 (연결/해제)** |
| **Control 필드** | N(S) 송신번호 / N(R) 수신번호 포함 | N(R) 수신번호만 포함 (데이터 없음) | 순서 번호 없음 (Modifer 포함) |
| **상세 동작** | 슬라이딩 윈도우 기반으로 ACK를 끼워 보냄 | RR, RNR, REJ, SREJ 코드로 상태 알림 | SABM, DISC 등으로 모드 설정/종료 |
| **데이터 필드** | **있음** | **없음** | **없음** (일부 정보만 존재) |

#### 2. 오류 제어 기법 비교 (S-Frame 활용)

```ascii
[ S-Frame의 비교: Go-Back-N vs Selective Repeat ]

     Sender                       Network (Channel)                    Receiver
        |                              |                                  |
   [0] [1] [2]                         |                                  |
        |----[0]---->[OK]------------->|----[0]-------------------------->|
        |                              |                          (Ack 0) |
        |----[1]---->[Error/X]-------->|--------(Lost/Error)------------->|
        |                              |                                  |
   [2] [3] [4]                         |                                  |
        |----[2]---------------------->|----[2]-------------------------->|
        |                              |                                  |
        |   Sender receives S-Frame (REJ or SREJ)                          |
        |<------------------[S-Frame (REJ 1)]------------------------------|
        |   (Go-Back-N: "1번부터 다시 보내라")                            |
        |   vs                                                            |
        |<------------------[S-Frame (SREJ 1)]-----------------------------|
        |   (Selective Repeat: "1번만 다시 보내라")                        |
```

**기술적 분석**:
*   **REJ (Reject)**: **Go-Back-N ARQ** 방식에 사용됩니다. 에러가 발생한 프레임부터 가장 최근까지 보낸 모든 프레임을 재전송합니다. 구현이 간단하지만 대역폭 낭비가 발생할 수 있습니다.
*   **SREJ (Selective Reject)**: **Selective Repeat ARQ** 방식에 사용됩니다. 에러가 발생한 특정 프레임만 재전송을 요청합니다. 대역폭 효율은 높지만 수신측의 버퍼 관리(Reordering Buffer)가 복잡해집니다.

#### 3. OSI 7계층 모델과의 융합 (Convergence)
HDLC는 계층 간의 명확한 분리와 협력을 보여줍니다.
*   **네트워크 계층(L3)과의 관계**: HDLC의 Information 필드는 네트워크 계층의 패킷(IP 패킷 등)을 캡슐화하여 전달합니다. 데이터 링크 계층의 신뢰성(L2)이 네트워크 계층의 비연결성/IP의 비신뢰성을 보완하는 기반 기술입니다.
*   **물리 계층(L1)과의 관계**: HDLC는 전기적 신호를 비트 단위로 조직화합니다. 물리 계층의 비트 전송 속도(Baud rate)나 인터페이스 규격(V.35, RS-232)과 독립적으로 설계되어, 이기종 장비 간의 호환성을 보장합니다.

📢 **섹션 요약 비유**: HDLC의 프레임 구분은 마치 '우편물 분류 시스템'과 같습니다. **I-Frame**은 편지(데이터)를 보내면서도 수신 확인 도장(ACK)을 찍어서 보내는 '등기 우편물'입니다. **S-Frame**은 편지는 없고 "주소지 못 찾아서 반송(REJ)"이거나 "여유 공간 없어서 보류(RNR)한다는" 엽서만 보내는 '공문'입니다. **U-Frame**은 "우편함을 새로 설치한다(SABM)"거나 "우편 서비스를 종료한다(DISC)"는 우체국의 행정 규정 공지입니다.
