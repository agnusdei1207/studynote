# 213-215. 흐름 제어와 슬라이딩 윈도우 (Flow Control and Sliding Window)

> **1. 본질 (Essence)**: 데이터 링크 계층 및 전송 계층에서 송신자의 전송률이 수신자의 처리 속도를 초과하여 발생하는 데이터 손실과 네트워크 혼잡을 방지하기 위해, 수신자의 버퍼 상태에 따라 송신 속도를 동적으로 조절하는 메커니즘입니다.
>
> **2. 가치 (Value)**: Stop-and-Wait ARQ (정지-대기 ARQ) 방식의 대역폭 낭비를 최소화하여, 파이프라이닝(Pipelining) 기술을 통해 회선의 이용 효율을 극대화하고 네트워크 처리량(Throughput)을 획기적으로 증대시킵니다.
>
> **3. 융합 (Convergence)**: TCP (Transmission Control Protocol)의 혼잡 제어(Congestion Control)와 OSI 7계층의 데이터 흐름 관리의 기반이 되며, 신뢰성 있는 데이터 전송을 위한 ARQ (Automatic Repeat reQuest) 기술과 결합하여 인터넷의 안정성을 책임집니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 및 정의
**흐름 제어 (Flow Control)**는 송신자(Transmitter)가 데이터를 전송하는 속도가 수신자(Receiver)가 데이터를 수신하고 처리하는 속도보다 빠를 때, 수신자의 버퍼(Buffer)가 오버플로우(Overflow)되어 데이터가 손실되는 것을 방지하기 위해 전송 속도를 규제하는 기술입니다. 이는 단순히 "속도를 줄이는 것"이 아니라, 수신자의 **수신 윈도우 (Receive Window)** 크기, 즉 가용 버퍼 공간을 고지하여 송신자가 전송량을 최적화하게 하는 피드백 시스템입니다.

### 2. 기술적 배경 및 필요성
데이터 통신 환경에서 송신자의 GPU/CPU 성능과 네트워크 대역폭은 매우 빠르게 발전하고 있으나, 수신자의 처리 능력은 상대적으로 제한적일 수 있습니다. 만약 송신자가 제약 없이 데이터를 쏘아 보낸다면, 수신자는 처리하지 못한 패킷을 저장할 공간이 부족해져 **패킷 드롭 (Packet Drop)**이 발생합니다. 이는 궁극적으로 송신자가 데이터를 재전송하게 만들어 네트워크 효율을 저하시키는 주요 원인이 됩니다. 흐름 제어는 이러한 비효율을 방지하기 위해 '평균 8번 정도' 명시된 기술사 필수 개념입니다.

### 💡 비유 (Analogy)
수도꼭지에서 흐르는 물(데이터)의 양을 조절하는 것과 같습니다. 물이 나가는 배수구(수신자 처리 속도)가 막혔는데도 수도꼭지를 최대로 틀어놓으면, 세면대(버퍼)가 넘처 물이 바닥으로 흘러넘치게 됩니다.

### 3. 도입 방식의 비교
흐름 제어를 구현하는 대표적인 방식으로는 피드백을 기반으로 하는 **슬라이딩 윈도우 (Sliding Window)**와 하드웨어 신호를 사용하는 **XON/XOFF**가 있습니다.

| 구분 | Stop-and-Wait (정지-대기) | Sliding Window (슬라이딩 윈도우) |
| :--- | :--- | :--- |
| **전송 방식** | 1개 프레임 전송 후 ACK 대기 | 미리 정해진 개수(Frame Count)만큼 연속 전송 |
| **회선 효율** | 매우 낮음 (대기 시간이 길음) | 매우 높음 (대기 시간 동안도 데이터 전송) |
| **버퍼 요구** | 최소 1개 (Frame) | 윈도우 크기($W$)만큼 필요 |

📢 **섹션 요약 비유**: 흐름 제어는 **물탱크의 수위 조절기**와 같습니다. 배수구(수신자)의 처리 속도가 느려도 수도꼭지(송신자)를 잠금으로써 물탱크(버퍼)가 넘치는 것을 방지합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 슬라이딩 윈도우 프로토콜 (Sliding Window Protocol)의 구조
슬라이딩 윈도우 프로토콜은 **ARQ (Automatic Repeat reQuest)** 기술의 일종으로, 전송 효율을 높이기 위해 송신자와 수신자 간에 '윈도우(Window)'라는 논리적인 전송 가능 범위를 설정하고 관리합니다. 이 윈도우는 데이터의 전송 순서와 확인 응답(ACK)에 따라 좌우로 미끄러지듯(Sliding) 이동한다고 하여 붙여진 이름입니다.

#### 핵심 구성 요소 및 매개변수 (Parameters)
| 요소명 | 역할 및 내부 동작 | 프로토콜/비고 |
|:---|:---|:---|
| **송신 윈도우 (Send Window)** | 송신자가 ACK 수신 없이 연속으로 보낼 수 있는 시퀀스 번호의 범위. 크기 $N$이면 최대 $N$개 프레임 전송 가능. | TCP 헤더의 Window Size 필드 |
| **수신 윈도우 (Receive Window)** | 수신자가 수신 가능한 시퀀스 번호의 범위. 수신 버퍼의 여유 공간을 의미함. | rwnd (Receiver Window) |
| **시퀀스 번호 (Seq No)** | 전송되는 각 프레임(데이터)에 부여되는 고유 번호. 순서 보장 및 오류 제어에 사용. | 32bit (TCP) |
| **Piggybacking (피기백킹)** | 데이터 프레임에 ACK 필드를 실어서 보내는 기법. 별도의 ACK 패킷을 줄여 효율 증대. | 양방향 통신 시 유리 |
| **L.Frame / R.Frame** | 윈도우의 경계. Left는 확인 응답받은 가장 오래된 프레임, Right는 다음에 전송할 마지막 프레임. | Window Size = $L - R$ |

### 2. 상세 동작 메커니즘 및 ASCII 다이어그램
슬라이딩 윈도우의 핵심은 **파이프라이닝(Pipelining)**입니다. 송신자는 ACK를 받지 않아도 윈도우 크기($W$) 내에서 프레임을 연속으로 전송합니다.

**도입 서술**: 아래 다이어그램은 윈도우 크기가 3인 시스템의 동작을 도식화한 것입니다. 송신자는 1, 2, 3번을 연속 전송하고 ACK를 기다리는 사이, 수신자로부터 ACK 1을 받으면 윈도우가 1칸 이동하여 4번 프레임을 전송할 수 있는 권한을 획득하게 됩니다.

```ascii
   송신 측 (Sender)                              수신 측 (Receiver)
+---------------------+                   +---------------------+
|  [1] [2] [3] [4] [5]|                   |   Buffer: [ ] [ ]   |
+---------------------+                   +---------------------+

[Phase 1: 윈도우 크기 3 설정 및 초기 전송]
| 송신 윈도우 | 미전송 데이터
|   [1] [2] [3] |  [4] [5]...
   ^           ^
   |-----------|
     전송 가능한 최대 3개

1) 프레임 1, 2, 3 전송 (ACK를 기다리지 않고 연속 발사)

[Phase 2: 수신자가 프레임 1, 2 수신 및 ACK 3 전송]
   송신 윈도우:      [1] [2] |  [3] [4] [5]...
   ACK 3 수신 시 -> 윈도우 슬라이딩(Slide) 발생

   송신 윈도우 (이동 후):    [3] [4] [5] |  [6]...
                               ^       ^
                               |-------|
                              새로 전송 가능한 영역 확보 (4, 5번 전송 가능)

[Phase 3: 누적 확인 응답 (Cumulative ACK)]
만약 수신자가 ACK 5를 보냈다면: "1번부터 4번까지 모두 잘 받았으니, 이제 5번을 달라"는 의미.
송신자의 윈도우는 즉시 5번 다음으로 이동.
```

#### ③ 다이어그램 심층 해설
위 다이어그램에서 볼 수 있듯이, 송신 윈도우는 **전송 가능한 프레임의 집합**을 나타냅니다. 수신자가 정상적으로 프레임 $N$을 받으면 ACK $N+1$을 송신자에게 반환합니다. 송신자가 이 ACK를 받는 순간, 윈도우의 좌측 경계(L)가 오른쪽으로 이동하며, 우측 경계(R) 역시 같은 폭만큼 이동합니다. 이렇게 확보된 새로운 공간만큼 송신자는 즉시 다음 프레임들을 전송합니다. 이 과정이 끊임없이 반복되며 데이터가 흐르게 됩니다.

### 3. 핵심 알고리즘 및 윈도우 관리
윈도우의 크기는 네트워크 대역폭과 대기 시간(Latency)의 곱인 **BDP (Bandwidth-Delay Product)**에 의해 결정됩니다.
$$ \text{최적 윈도우 크기} \ge \text{대역폭 (Bandwidth)} \times \text{왕복 지연 시간 (RTT)} $$

**코드 레벨 이해 (Pseudo-code)**:
```python
# Sender Side Logic
Send_Window_Base = 0    # 가장 오래된 확인되지 않은 패킷
Next_Seq_Num = 0        # 다음에 보낼 패킷 번호
Window_Size = 4         # 윈도우 크기

while True:
    # 윈도우에 여유가 있는 만큼 전송
    while (Next_Seq_Num < Send_Window_Base + Window_Size):
        send_packet(Next_Seq_Num)
        Next_Seq_Num += 1
    
    # ACK 대기 (Non-blocking)
    Ack_Num = receive_ack()
    
    # 윈도우 슬라이딩 (Cumulative ACK 지원)
    if (Ack_Num > Send_Window_Base):
        Send_Window_Base = Ack_Num 
        # Base가 이동함에 따라 위 while 루프가 다시 돌며 패킷 전송
```

📢 **섹션 요약 비유**: 슬라이딩 윈도우는 **매끄러운 물류 컨베이어 벨트**와 같습니다. 작업자(수신자)는 일감을 처리하면서 "지금부터 5개까지 보내와"라고 요청하고, 쌓여있는 박스들을 하나씩 처리할 때마다 컨베이어 벨트에 새 박스를 올릴 수 있는 슬롯이 하나씩 생겨납니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 심층 기술 비교: Stop-and-Wait vs. Sliding Window
정지-대기(Stop-and-Wait) 방식은 구현이 간단하지만, 네트워크의 성능을 극히 저하시킵니다. 슬라이딩 윈도우는 이를 극복한 고급 기술입니다.

| 비교 항목 | Stop-and-Wait (정지-대기) | Sliding Window (슬라이딩 윈도우) |
|:---|:---|:---|
| **전송 효율 (Utilization)** | $U = \frac{1}{1 + 2a}$ (매우 낮음) | $U = 1$ (윈도우 크기가 충분할 때) |
| **필요 버퍼 크기** | 1 (단일 프레임) | $W$ (Window Size) 이상 |
| **프레임 순서** | 항상 보장 | 시퀀스 번호로 순서 보장 |
| **오류 복구** | Timeout 시 단순 재전송 | Selective Repeat (선택적 재전송) 또는 Go-Back-N |
| **네트워크 부하** | Header(H)가 Data(D)에 비해 큼 | D/H 비율을 높여 효율적 |

> *$a = \frac{\text{전송 지연}}{\text{프로세싱 지연}}$*
> 슬라이딩 윈도우는 윈도우 크기 $W \ge 1 + 2a$일 때 100% 효율을 낼 수 있습니다.

### 2. 타 과목 융합 관점
1.  **운영체제 (OS)와의 시너지**: 흐름 제어는 데이터 링크 계층의 물리적 회선 제어뿐만 아니라, **OS의 프로세스 간 통신(IPC)**에서도 발생합니다. 생산자-소비자(Producer-Consumer) 문제에서 세마포어(Semaphore)를 통해 버퍼를 관리하는 것은 소프트웨어적 흐름 제어의 완벽한 예시입니다.
2.  **네트워크 계층과의 시너지**: 흐름 제어가 송신-수신 간의 **1:1 문제**를 해결한다면, 혼잡 제어(Congestion Control)는 네트워크 내부의 **라우터나 링크가 포화 상태**가 되는 것을 방지하는 **1:N (전체 네트워크) 문제**를 다룹니다. 슬라이딩 윈도우는 이 두 가지 목적을 모두 달성하기 위해 사용됩니다.

📢 **섹션 요약 비유**: 정지-대기 방식은 **사각형 펜스를 연속해서 설치하는 것**과 같아서 못을 하나 박을 때마다 해머를 내려놓고 쉬어야 합니다. 슬라이딩 윈도우는 **미리 못을 여러 개 입에 물고 작업하는 것**과 같아서, 못을 하나 박을 때마다 바로 다음 못을 박을 수 있어 작업 속도가 훨씬 빠릅니다.
