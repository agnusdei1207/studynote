# OAuth 2.0 - 인가 프레임워크

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 제3자 인가 프레임워크
> 2. **가치**: 비밀번호 없는 접근
> 3. **융합**: OpenID Connect, SSO

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

OAuth 2.0은 제3자 애플리케이션이 사용자 데이터에 접근하도록 인가합니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    OAuth 2.0                                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              OAuth 2.0 흐름                                    │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │                                                      │   │ │
│   │   │  [User]                                             │   │ │
│   │   │     │                                                │   │ │
│   │   │     │ 1. 인가 요청                                    │   │ │
│   │   │     ▼                                                │   │ │
│   │   │  [Client App] ──────▶ [Authorization Server]        │   │ │
│   │   │                            │                          │   │ │
│   │   │                            │ 2. 로그인 & 동의          │   │ │
│   │   │                            ▼                          │   │ │
│   │   │  [User] ◀───────────────┘                          │   │ │
│   │   │     │                                                │   │ │

│   │   │     ▼                                                │   │ │
│   │   │  [Client App] ──────▶ [Authorization Server]        │   │ │
│   │   │     │                    │                          │   │ │
│   │   │     │ 4. 액세스 토큰    │                          │   │ │
│   │   │     ▼                    ▼                          │   │ │
│   │   │  [Resource Server] ◀───┘                          │   │ │
│   │   │                                                      │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │

│   │   │  +------------+----------+----------+               │   │ │
│   │   │  │ Auth Code  │ 웹       │ 안전     │               │   │ │
│   │   │  │ Implicit   │ SPA      │ 레거시   │               │   │ │
│   │   │  | Client Cred│ 서버     │ 기계     │               │   │ │
│   │   │  │ Refresh    │ 갱신     │ 장기     │               │   │ │
│   │   │  +------------+----------+----------+               │   │ │
│   │   │                                                      │   │ │

```

> **해설**: OAuth 2.0은 액세스 토큰으로 제한된 접근을 인가합니다.

**💡 비유**: OAuth는 위임장입니다. 대신 일을 처리하도록 위임합니다.

**📢 섹션 요약 비유:** OAuth = 인가!

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 항목 | 설명 |
|:---|:---|
| **Client** | 클라이언트 앱 |
| **Resource Owner** | 사용자 |
| **Auth Server** | 인가 서버 |
| **Resource Server** | 리소스 서버 |

**비교표: OAuth vs OIDC**

```
+------------+----------+----------+
| 항목       │ OAuth 2.0│ OIDC     │
+------------+----------+----------+
| 목적       │ 인가     │ 인증     │
| 토큰       │ AT/RT    │ ID Token │
| 사용자     │ 없음     │ 있음     │
+------------+----------+----------+
```

**핵심 알고리즸:** 인가 코드, 토큰 교환

**📢 섹션 요약 비유:** PKCE = 모바일 필수!

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**기술 비교: 인증 방식**

| 항목 | OAuth | SAML | API Key |
|:---|:---|:---|:---|
| **용도** │ 인가 │ SSO │ API |
| **형식** │ JSON │ XML │ 문자열 |
| **웹** │ 적합 │ 적합 │ 간단 |

**📢 섹션 요약 비유:** 소셜 로그인 = OAuth!

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1:** 소셜 로그인 → OAuth 2.0 + OIDC
**시나리오 2:** API 접근 → Client Credentials
**시나리오 3:** SPA → Auth Code + PKCE

**도입 체크리스트:** Grant 타입, 스코프, 보안

**📢 섹션 요약 비유:** 웹 = Auth Code!
