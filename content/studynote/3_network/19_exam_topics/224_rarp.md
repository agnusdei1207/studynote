# RARP (Reverse ARP) - 역방향 주소 해석

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: MAC 주소 → IP 주소 변환
> 2. **가치**: 디스크리스 워크스테이션
> 3. **융합**: BOOTP, DHCP로 진화

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

RARP(Reverse Address Resolution Protocol)는 MAC 주소를 IP 주소로 변환하는 프로토콜입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    RARP (Reverse ARP)                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              RARP 동작 과정                                    │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │                                                      │   │ │
│   │   │  디스크리스 워크스테이션      RARP 서버              │   │ │
│   │   │  (MAC: aa:bb:cc:dd:ee:ff)                           │   │ │
│   │   │                                                      │   │ │
│   │   │  1. RARP Request (Broadcast):                       │   │ │
│   │   │     "My MAC is aa:bb:cc:dd:ee:ff, what's my IP?"    │   │ │
│   │   │     ───────────────────────────────────────>         │   │ │
│   │   │                                                      │   │ │
│   │   │  2. RARP Reply (Unicast):                           │   │ │
│   │   │     "Your IP is 192.168.1.100"                      │   │ │
│   │   │     <──────────────────────────────────────          │   │ │
│   │   │                                                      │   │ │
│   │   │  용도: 디스크리스 부팅, IP 없는 장비                 │   │ │
│   │   │                                                      │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              ARP vs RARP                                       │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │                                                      │   │ │
│   │   │  프로토콜 │ 요청            │ 응답             │   │ │
│   │   │  ────────┼─────────────────┼─────────────────│   │ │
│   │   │  ARP     │ IP → MAC        │ MAC              │   │ │
│   │   │  RARP    │ MAC → IP        │ IP               │   │ │
│   │   │                                                      │   │ │
│   │   │  방향:                                                 │   │ │
│   │   │  • ARP: 통신 전 MAC 확인                             │   │ │
│   │   │  • RARP: 부팅 시 IP 확인                              │   │ │
│   │   │                                                      │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────────────┘
```

> **해설**: RARP는 MAC 주소로 IP를 할당받습니다.

**💡 비유**: RARP는 신분증(MAC)으로 주민번호(IP)를 찾는 것입니다.

**📢 섹션 요약 비유:** RARP = MAC → IP!

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 항목 | ARP | RARP |
|:---|:---|:---|
| **요청** | IP 알 때 | IP 모를 때 |
| **응답** | MAC | IP |
| **서버** | 없음 | RARP 서버 |

**비교표: 진화**

```
+------------+----------+----------+
| 프로토콜   │ 특징     │ 상태     │
+------------+----------+----------+
| RARP      │ MAC→IP   │ 레거시   │
| BOOTP     │ IP 할당  │ 레거시   │
| DHCP      │ 동적 할당│ 표준     │
+------------+----------+----------+
```

**핵심 알고리즸:** 브로드캐스트 요청, 서버 응답

**📢 섹션 요약 비유:** RARP → BOOTP → DHCP!

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**기술 비교: RARP 한계**

| 한계 | 설명 |
|:---|:---|
| **서버 필요** | RARP 서버 필수 |
| **네트워크 제한** | 동일 네트워크만 |
| **기능 제한** | IP만 할당 |

**📢 섹션 요약 비유:** DHCP = RARP + 더 많은 기능!

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1: 레거시** → RARP (현재 거의 사용 안함)
**시나리오 2: 디스크리스** → DHCP + PXE
**시나리오 3: IoT** → DHCP

**도입 체크리스트:** DHCP로 대체 권장

**📢 섹션 요약 비유:** RARP는 역사적 프로토콜!

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**정량/정성 기대효과**: 디스크리스 부팅

**미래 전망:** DHCP로 완전 대체

**참고 표준:** RFC 903

**📢 섹션 요약 비유:** RARP는 BOOTP/DHCP의 조상!
