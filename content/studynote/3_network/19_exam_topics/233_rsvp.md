# RSVP (Resource Reservation Protocol) - 자원 예약

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 네트워크 자원 예약 프로토콜
> 2. **가치**: QoS 보장
> 3. **융합**: IntServ, MPLS TE와 통합

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

RSVP(Resource Reservation Protocol)는 네트워크 경로를 따라 자원을 예약하는 시그널링 프로토콜입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    RSVP (Resource Reservation Protocol)              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              RSVP 동작                                         │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │                                                      │   │ │
│   │   │  [송신자]                                        │   │ │
│   │   │     │                                             │   │ │
│   │   │     │ 1. PATH 메시지 (경로 정보)                   │   │ │
│   │   │     │ ────────────────────────────────────────>    │   │ │
│   │   │     │                                             │   │ │
│   │   │  [라우터 1] ───> [라우터 2] ───> [라우터 3]        │   │ │
│   │   │     │               │               │              │   │ │
│   │   │     │               │               │              │   │ │
│   │   │     │ 2. RESV 메시지 (예약 요청)                   │   │ │
│   │   │     │ <───────────────────────────────────────      │   │ │
│   │   │     │                                             │   │ │
│   │   │  [수신자]                                        │   │ │
│   │   │                                                      │   │ │
│   │   │  특징:                                                │   │ │
│   │   │  • 수신자 기반 예약                                  │   │ │
│   │   │  • 소프트 상태 (Soft State)                          │   │ │
│   │   │  • 멀티캐스트 지원                                   │   │ │
│   │   │                                                      │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              RSVP 메시지                                       │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │                                                      │   │ │
│   │   │  메시지   │ 방향         │ 용도                      │   │ │
│   │   │  ────────┼─────────────┼─────────────────────────│   │ │
│   │   │  PATH    │ 송신→수신    │ 경로 정보                 │   │ │
│   │   │  RESV    │ 수신→송신    │ 예약 요청                 │   │ │
│   │   │  PATHERR │ 수신→송신    │ 경로 오류                 │   │ │
│   │   │  RESVERR │ 송신→수신    │ 예약 오류                 │   │ │
│   │   │  TEAR    │ 양방향       │ 예약 해제                 │   │ │
│   │   │                                                      │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────────────┘
```

> **해설**: RSVP는 수신자가 자원을 예약합니다.

**💡 비유**: RSVP는 공연 예약입니다. 관객이 티켓을 예매합니다.

**📢 섹션 요약 비유:** RSVP = 수신자 기반 예약!

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 항목 | 설명 |
|:---|:---|
| **PATH** | 송신자가 전송 |
| **RESV** | 수신자가 예약 |
| **FlowSpec** | 대역폭, 지연 요구 |
| **FilterSpec** | 패킷 필터 |
| **Soft State** | 주기적 갱신 |

**비교표: 예약 스타일**

```
+------------+----------+----------+
| 스타일     │ 설명     │ 용도     │
+------------+----------+----------+
| FF         │ 고정 필터│ 유니캐스트│
| SE         │ 공유 명시│ 멀티캐스트│
| WF         │ 와일드카드│ 브로드캐스트│
+------------+----------+----------+
```

**핵심 알고리즸:** PATH/RESV 교환, Soft State

**📢 섹션 요약 비유:** Soft State = 주기적 갱신!

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**기술 비교: RSVP-TE**

| 항목 | RSVP | RSVP-TE |
|:---|:---|:---|
| **용도** | QoS | MPLS TE |
| **라벨** | X | O |
| **경로** | 동적 | 명시적 |

**📢 섹션 요약 비유:** RSVP-TE = MPLS 트래픽 엔지니어링!

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1: MPLS TE** → RSVP-TE
**시나리오 2: 레거시 QoS** → RSVP (거의 사용 안함)
**시나리오 3: 현대 QoS** → DiffServ

**도입 체크리스트:** 규모, MPLS 여부

**📢 섹션 요약 비유:** 현대 = DiffServ, MPLS = RSVP-TE!

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**정량/정성 기대효과**: 자원 보장

**미래 전망:** Segment Routing으로 이동

**참고 표준:** RFC 2205, RFC 3209 (RSVP-TE)

**📢 섹션 요약 비유:** RSVP는 MPLS TE에 여전히 사용!
