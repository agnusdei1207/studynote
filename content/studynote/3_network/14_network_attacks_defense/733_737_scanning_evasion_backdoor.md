# 733-737. 보안 정찰 및 우회 기법 (Scanning, Tunneling, Backdoor)

> 1. **본질**: 사이버 공격의 생명주기(Cyber Kill Chain) 중 초기 단계인 '정찰(Reconnaissance)'과 '무기화(Weaponization)', 그리고 '지속성 유지(Persistence)'를 위한 핵심 기술적 메커니즘.
> 2. **가치**: 방화벽(Firewall)과 침입 탐지 시스템(Intrusion Detection System, IDS)의 작동 원리를 역이용하여, 99.9%의 보안 장비가 탐지하지 못하는 '스텔스' 성능을 확보함.
> 3. **융합**: 네트워크 계층(L3/L4)의 패킷 구조와 암호학, 그리고 운영체제(Operating System, OS)의 네트워크 스택 내부 동작을 종합적으로 분석해야 방어 가능.

+++

### Ⅰ. 개요 (Context & Background) - [600자+]

**개념 및 정의**
보안 정찰 및 우회 기술은 해커가 본격적인 공격을 감행하기 전, 타겟 시스템의 취약점을 식별하거나 보안 장비의 감시망을 피하는 기술을 의미합니다. 단순한 포트 확인을 넘어, 방화벽의 설정 규칙을 우회하거나 암호화 통계를 악용하여 악성 코드를 은폐하는 고도화된 기법을 포함합니다. 이는 물리적인 전쟁에서 적진의 지형을 파악하고 은밀한 통로를 확보하는 과정과 기술적으로 동일합니다.

**💡 비유**
보안 정찰은 강력한 보안을 자랑하는 성벽을 앞에 둔 침입자가, 성문의 위치(개방 포트)를 확인하고, 성 밑을 통해 침입하는 굴을 파거나(터널링), 내부에 숨겨둔 첩자(백도어)와 연락하는 과정입니다.

**등장 배경**
1.  **보안 장비의 발전**: 초기의 단순 패킷 필터링에서 최신의 DPI(Deep Packet Inspection) 및 상태 추적(Stateful Inspection) 방화벽으로 진화하면서, 단순한 공격은 차단됨.
2.  **규칙의 역이용**: 보안 장비가 '정상 트래픽'으로 간주하는 특징(예: TCP 3-Way Handshake 완성, 특정 포트 허용)을 악용하여 트래픽을 위장함.
3.  **지속적 원격 제어의 필요성**: 1회성 공격이 아닌, 장기적으로 타겟 시스템을 장악하고 데이터를 유출하기 위한 C2(Command & Control) 채널 유지 필요성 대두.

📢 **섹션 요약 비유**: 성 밖에 숨어 있는 도둑이 성벽에 경비병(보안 장비)이 어디에 배치되어 있는지, 그리고 어느 구간에 경비가 허술한지(취약점)를 밤중에 손전등을 비추며 살피는 과정입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive) - [1,200자+]

이 섹션에서는 포트 스캐닝의 세부 메커니즘과 터널링, 백도어의 기술적 아키텍처를 분석합니다.

#### 1. 구성 요소 (표)

| 요소명 | 역할 | 내부 동작 메커니즘 | 주요 프로토콜/도구 | 비유 |
|:---|:---|:---|:---|:---|
| **Port Scanner** | 서비스 식별 | TCP/IP 스택의 응답 플래그(RST, SYN-ACK) 분석을 통해 포트 상태(Open/Closed/Filtered) 판별 | TCP SYN, FIN, XMAS, NULL | 문을 두드려보는 행위 |
| **Tunneling Protocol** | 캡슐화 및 우회 | 허용된 프로토콜(HTTP/HTTPS)의 페이로드(Payload) 영역에 비인가 프로토콜이나 데이터를 숨겨 전송 | HTTP, DNS, ICMP | 피자 박스에 숨겨진 밀반입품 |
| **Backdoor** | 지속적 접근 | OS 부팅 과정(Runlevel)이나 정상 프로세스에 삽입되어 인증 없는 쉘(Shell) 접근 제공 | Netcat, Meterpreter, Web Shell | 땅굴 입구에 숨겨진 비밀 문 |
| **C2 Server** | 원격 명령 제어 | 악성코드에게 명령을 내리거나 탈취한 데이터를 수집하는 중앙 통제소 (Beaconing 신호 송수신) | HTTPS, DNS Tunneling | 군 작전 지휘소 |

#### 2. 포트 스캐닝 및 스텔스 기술 (ASCII 다이어그램)

Nmap(Network Mapper) 등을 이용한 스캐닝은 대상 시스템의 네트워크 스택 구현 차이를 이용합니다.

```ascii
[ Stealth SYN Scan (Half-Open Scan) Flow ]

Attacker                        Target System
   |                                 |
   |      1. SYN (Port 80)           |
   |-------------------------------->|
   |                                 | [SYN_RECEIVED State]
   |      2. SYN/ACK (Port Open)     |
   |<--------------------------------|
   |                                 |
   |      3. RST (Reset Immediately) |
   |-------------------------------->|
   |            [Connection Closed]  |
   |                                 |
```

**해설**:
**SYN Scan**은 TCP 연결 설정(3-Way Handshake)의 3단계(ACK)를 완료하지 않고, 2단계(SYN/ACK) 수신 후 즉시 **RST (Reset)** 패킷을 보내 연결을 끊습니다. 이때 대상 시스템은 Application Layer 로그를 남기지 않으므로 '스텔스'라고 불립니다. 반면, **FIN/XMAS/NULL Scan**은 RFC 표준을 위반하는 플래그 조합을 보내 방화벽이나 OS의 예외 처리(Exception Handling) 루틴을 유발하여 포트 상태를 유추합니다.

#### 3. 터널링 및 데이터 은폐

보안 장비는 특정 포트(80, 443)를 통과하는 트래픽을 신뢰하는 경향이 있습니다. 이를 악용한 기술입니다.

**코어 코드 예시 (HTTP Header Tunneling)**
```http
POST /api/upload HTTP/1.1
Host: target-server.com
Content-Type: multipart/form-data
X-C2-Command: whoami; cat /etc/passwd  <!-- 악성 명령이 헤더나 페이로드에 숨겨짐 -->
```

#### 4. C2 (Command & Control) 및 백콘 (Backdoor)

**Beaconing 신호**: 감염된 호스트는 주기적으로 C2 서버에 접속을 시도하거나, 특정 패턴의 DNS 질의를 보내 "나는 살아잉니다"라고 신고합니다.

```ascii
[ C2 Beaconing & Persistence Logic ]

Infected Host                    Firewall/IDS           C2 Server
     |                                |                      |
     |---(1) Keep-alive Beacon------->| [Looks Normal HTTPS] |
     |    (Encrypted "Heartbeat")     |---------->           |
     |                                |                      |
     |                                |                      |---(2) Check Task Queue
     |                                |                      |    (Is there a command?)
     |                                |                      |
     |<------(3) Download Module------|<----------           |
     |    (Encrypted Payload)         |                      |
     |                                |                      |
     |---(4) Exfiltrate Data--------->|---------->           |
```

**해설**:
최신 백도어는 **Cobalt Strike**나 **Metasploit**과 같은 프레임워크를 사용하여 트래픽을 암호화합니다. 일반 방화벽은 이를 정상적인 HTTPS 웹 트래픽으로 인식하여 통과시킵니다. 이를 **Evasion (우회)** 라고 하며, 정규 표현식(Regex) 기반의 탐지를 우회하기 위해 **패킷 파편화(Packet Fragmentation)** 기술을 쓰기도 합니다. 패킷을 쪼개서 보내면 IDS는 패킷 재조합(Reassembly) 과정에서 CPU 부하를 겪거나 설정에 따라 중간 패킷을 검사하지 못할 수 있습니다.

📢 **섹션 요약 비유**: 터널링은 마치 금지된 약물을 운반할 때, 경찰 단속을 피하기 위해 정상적인 화물 트럭(프로토콜) 안에 가장 안쪽에 숨겨서 국경을 넘는 것과 같습니다. 백도어는 도둑이 집안에 침입한 뒤, 나중에 언제든 다시 들어올 수 있도록 현관문 잠금장치를 주기적으로 열어두는 자동 장치를 설치하는 것입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy) - [800자+]

보안 정찰 기법은 단순히 네트워크를 스캔하는 것이 아니라, OS의 버전(OS Fingerprinting)까지 식별하여 융합적인 공격 경로를 설정합니다.

#### 1. 심층 기술 비교

| 구분 | 포트 스캐닝 (Scanning) | 터널링 (Tunneling) | 백도어 (Backdoor) |
|:---|:---|:---|:---|
| **주 목적** | 정보 수집 (정찰) | 우회 및 은폐 (회피) | 지속성 (Persistence) |
| **네트워크 계층** | L3/L4 (IP/TCP) | L4/L7 (Application Payload) | L7 (App) / OS Kernel |
| **탐지 난이도** | 중간 (로그 남음) | 상당 (DPI 필요) | 상당 (행태 분석 필요) |
| **주요 방어기술** | Network IDS (NIDS), Port Knocking | Next-Gen FW, SSL Inspection | EDR (Endpoint Detection), Integrity Check |

#### 2. 과목 융합 관점 (OS & 네트워크)
스캐닝은 **운영체제(OS)**의 **TCP/IP 스택 구현 방식 차이**를 이용합니다. 예를 들어, Linux와 Windows는 RFC 비표준 플래그(FIN scan)를 받았을 때 보내는 응답 패킷이 다릅니다. 이를 통해 해커는 OS 버전까지 식별 가능합니다.
또한, 백도어는 **시스템 해킹** 기술인 **Rootkit**과 결합하여 프로세스 숨김(Hiding Process)이나 파일 숨김 기능을 수행합니다. 단순 네트워크 차단으로는 막을 수 없으며 **종합적인 보안 관제**가 필요합니다.

📢 **섹션 요약 비유**: 의사가 환자를 진단할 때 단순히 겉만 보는 게 아니라(Scanning), MRI 등을 찍어 내부 구조를 확인하고(OS Fingerprinting), 수술 후 재활을 위해 몸 속에 약물 전달 장치를 심는(Backdoor) 것과 같은 복합적인 과정입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision) - [900자+]

#### 1. 실무 시나리오 및 대응

**시나리오 A: 정상적인 업무용 포트인 80/443 포트로부터의 이상 징후**
*   **상황**: 대내망 서버가 평소와 다른 패턴으로 해외 IP로 대량의 아웃바운드 HTTPS 트래픽을 발생시킴. 내용은 암호화되어 있어 내부 확인 불가.
*   **판단**: 정상 웹 서비스가 아니라 **C2 Beaconing** 신호일 가능성이 높음. 터널링을 통한 데이터 유출이나 원격 제어 시도일 수 있음.
*   **대응**: 네트워크 트래픽 베이스라인(Baseline) 설정을 통해 이상 탐지를 자동화하고, SSL Inspection(HTTPS 복호화 검사)을 도입하여 페이로드 내부까지 검사해야 함.

**시나리오 B: 스텔스 스캔 탐지 실패**
*   **상황**: 방화벽 로그에는 아무런 기록이 없으나, 서버 시스템 로그에는 비정상적인 접속 시도 패턴이 남아 있음.
*   **판단**: SYN Scan 등 스텔스 스캔 사용될 것. 방화벽은 연결이 완료되지 않은 패킷을 로그로 남기지 않도록 설정되었을 수 있음.
*   **대응**: 방화벽 뿐만 아니라 **HIDS (Host-based Intrusion Detection System)**를 도입하여 서버 내부에서의 접속 요청을 모니터링해야 함.

#### 2. 도입 체크리스트 (보안 강화)

| 구분 | 항목 | 설명 |
|:---|:---|:---|
| **기술적** | **Port Knocking** | 방화벽의 모든 포트를 닫고, 특정 순서로 패킷을 보내야만 포트가 열리도록 설정하여 무작위 스캔 차단 |
| | **SSL/TLS Inspection** | 들어오고 나가는 HTTPS 트래픽을 중간에서 복호화하여 터널링 악성코드 탐지 (성능 저하 고려 필요) |
| | **Honeypot (허니팟)** | 가짜 서버를 운영하여 스캐닝 및 공격 시도를 유인하고 공격자 정보 수집 |
| **운영적** | **비인가 장치 탐지** | 무선 AP(Rogue AP)나 연결된 불법 장비를 주기적으로 스캔하여 제거 |
| | **무결성 검사** | 주요 시스템 파일의 해시(Hash) 값을 주기적으로 확인하여 백도어 삽입 여부 확인 |

#### 3. 안티패턴 (Anti-Pattern)

*   **방화벽 로그만 신뢰**: "방화벽 로그에 없으니 안전하다"는 판단은 치명적입니다. L4 방화벽은 세션이 완성되지 않은 SYN 스캔을 로그로 남기지 않을 수 있기 때문입니다.
*   **폐쇄망 방역 무시**: 인터넷 연결이 없는 폐쇄망이라도 내부자에 의한 스캐닝이나 USB를 통해 유입된 백도어가 터널링(예: DNS Tunneling)을 시도할 수 있습니다.

📢 **섹션 요약 비유**: 보안은 마치 화장실에 들어갈 때 문을 잠그는 것(방화벽)만으로는 부족합니다. 누군가 문틈으로 훔쳐보는지(스캐닝), 환풍창을 통해 누군가 들어오는지(백도어), 혹은 배관을 통해 침입하는지(터널링)를 모두 감시해야 하는 종합
