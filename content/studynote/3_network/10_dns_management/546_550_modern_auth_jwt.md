# 546-550. 최신 인증 및 토큰 기술 (OAuth, JWT, X.509)

> **핵심 인사이트**: 이제는 매번 서버에 로그인할 필요가 없다. 한 번 받은 '금고 열쇠(Access Token)'만 있으면 여러 앱을 자유롭게 드나들 수 있다(OAuth). 또한, 서버가 정보를 저장하지 않아도 스스로 증명이 가능한 '스마트 이름표(JWT)'와 인터넷상의 '전자 신분증(X.509)'이 현대 보안의 핵심이다.

---

## Ⅰ. OAuth 2.0 (Open Authorization)
자신의 서비스 계정 비밀번호를 제3자 앱에 알려주지 않고도, 그 앱이 내 정보(사진, 친구 목록 등)에 접근할 수 있도록 권한을 위임하는 프레임워크입니다.

* **참여 주체**:
  - **Resource Owner**: 사용자 (나)
  - **Client**: 제3자 애플리케이션 (예: 맛집 앱)
  - **Authorization Server**: 인증 서버 (예: 구글, 카카오)
  - **Resource Server**: 실제 데이터가 있는 서버
* **결과물**: **Access Token**. 이 토큰만 있으면 정해진 범위 내에서 데이터를 가져올 수 있습니다.

---

## Ⅱ. JWT (JSON Web Token)
인증에 필요한 정보들을 JSON 객체에 담아 **디지털 서명**을 한 후, 클라이언트가 직접 들고 다니게 하는 토큰입니다.

* **구조**: `Header.Payload.Signature`
  - **Header**: 어떤 암호화 알고리즘을 썼는가?
  - **Payload**: 누구인지, 권한이 무엇인지 담긴 실제 데이터 (Claim).
  - **Signature**: 서버가 가진 비밀키로 만든 서명. 위변조 방지.
* **장점 (Stateless)**: 서버가 "누가 로그인했는지" 일일이 메모리(세션)에 적어둘 필요가 없습니다. 토큰만 열어보면 서버가 직접 발행한 것인지 알 수 있기 때문입니다.

---

## Ⅲ. OpenID Connect (OIDC)
* OAuth 2.0가 '권한(Authorization)'에 집중했다면, OIDC는 그 위에 '인증(Authentication, 신원 확인)' 기능을 얹은 것입니다.
* **ID Token**이라는 JWT를 추가로 발행하여, 이 사용자가 누구인지 명확히 알려줍니다.

---

## Ⅳ. X.509 v3 디지털 인증서
공개키 기반 구조(PKI)에서 사용되는 **표준 인증서 규격**입니다.

* **포함 정보**: 주체(사용자) 정보, 발행자(CA) 서명, **사용자의 공개키**, 유효 기간 등.
* **용도**: HTTPS 웹사이트 보안, 전자 서명, 코드 사인 등.

---

## Ⅴ. 개념 맵 및 요약

```ascii
[JWT 인증의 흐름]

1. [ User ] ──(로그인 ID/PW)──> [ Server ]
2. [ User ] <───(JWT 발급)───── [ Server ] (서버는 세션 저장 안함)
3. [ User ] ───(JWT 전송)─────> [ Server ] (매 요청마다 토큰 제출)
4. [ Server ] (서명을 검증하고 데이터 즉시 처리)
```

📢 **섹션 요약 비유**: **OAuth**는 호텔에서 방 비밀번호(ID/PW)를 주는 대신, 특정 층만 갈 수 있는 '임시 카드키(Token)'를 만들어주는 것입니다. **JWT**는 '자유이용권 팔찌'와 같습니다. 놀이공원 직원(서버)이 일일이 장부를 확인하지 않아도, 팔찌에 찍힌 도장(서명)만 보고 "응, 우리 손님 맞네" 하고 통과시켜 줍니다. **X.509**는 국가가 발행한 '전자 주민등록증'이며, **OIDC**는 카드키에 이름과 사진(ID)까지 박아둔 것입니다.
