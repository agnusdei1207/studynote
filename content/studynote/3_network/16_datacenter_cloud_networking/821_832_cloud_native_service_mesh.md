# 821-832. 클라우드 네이티브와 서비스 메시 (CNI, Istio)

> **핵심 인사이트**: 컨테이너가 수천 개로 늘어나면 사람이 일일이 네트워크를 연결할 수 없다. CNI는 포드(Pod)가 태어나자마자 자동으로 주소를 부여하고, 서비스 메시는 그 수많은 파편화된 서비스들 사이의 대화(통신, 보안, 모니터링)를 인프라 차원에서 알아서 관리해 준다.

---

## Ⅰ. CNI (Container Network Interface)
쿠버네티스와 같은 컨테이너 오케스트레이터가 네트워크 플러그인과 대화하기 위한 표준 규격입니다.

### 주요 CNI 플러그인
1. **Flannel**: 가장 단순한 오버레이(VXLAN) 방식. 설정이 쉽지만 보안 기능이 부족함.
2. **Calico**: BGP 라우팅 기반으로 성능이 뛰어나며, 강력한 **네트워크 정책(Network Policy)**을 지원함.
3. **Cilium**: 리눅스 커널의 **eBPF** 기술을 사용하여 고성능 로드밸런싱과 보안 가시성을 제공함. (차세대 표준)

---

## Ⅱ. 서비스 메시 (Service Mesh)
마이크로서비스(MSA) 간의 복잡한 통신을 관리하기 위해 도입된 전용 인프라 계층입니다.

* **핵심 아키텍처: 사이드카 (Sidecar)**
  - 애플리케이션 코드 안에 통신 로직을 넣지 않고, 컨테이너 바로 옆에 작은 프록시(**Envoy**)를 붙여 모든 통신을 대행하게 합니다.
* **주요 기능**:
  - **트래픽 제어**: 카나리 배포, 서킷 브레이커 (장애 전파 방지).
  - **보안**: 서비스 간 자동 암호화 통신 (**mTLS**).
  - **가시성**: 어느 서비스가 느린지 한눈에 파악.
* **대표 솔루션**: **Istio (이스티오)**.

---

## Ⅲ. 인그레스와 이그레스 (Ingress / Egress)
* **Ingress**: 클러스터 외부에서 내부 서비스로 들어오는 트래픽 통제. (주로 L7 로드밸런싱)
* **Egress**: 클러스터 내부에서 외부 인터넷으로 나가는 트래픽 통제.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[사이드카 프록시 구조]

  [ Pod 1 ]                     [ Pod 2 ]
  ┌───────────────┐             ┌───────────────┐
  │ [ App 1 ]     │             │ [ App 2 ]     │
  │      ↕        │             │      ↕        │
  │ [ Sidecar ] <────( mTLS )────> [ Sidecar ]  │
  └───────────────┘             └───────────────┘
```

📢 **섹션 요약 비유**: **CNI**는 신도시 아파트(Pod)가 지어질 때마다 자동으로 수도와 전기를 연결해 주는 '도시 기반 시설팀'입니다. **서비스 메시**는 아파트 단지 내의 모든 집마다 '비서(Sidecar)'를 한 명씩 배치하는 것과 같습니다. 이 비서들은 서로 암호(mTLS)로 대화하고, 택배(데이터)가 너무 많이 몰리면 입구에서 컷(서킷 브레이커)하기도 하며, 누가 언제 왔는지 꼼꼼히 기록(모니터링)하는 유능한 관리인입니다.
