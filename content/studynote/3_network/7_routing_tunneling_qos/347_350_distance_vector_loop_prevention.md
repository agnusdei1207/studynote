# 347-350. 거리 벡터(Distance Vector) 라우팅과 루프 방지

### # 거리 벡터 라우팅 (Distance Vector Routing) 및 루프 방지 기법

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 거리 벡터는 **벨만-포드(Bellman-Ford)** 알고리즘에 기반하여, 라우터가 이웃과만 라우팅 테이블을 교환하는 분산형 '소문(Rumor)' 방식의 라우팅 프로토콜입니다.
> 2. **가치**: 설정이 간단하고 소형 네트워크에서 리소스 사용량이 적으나, 수렴 속도가 느리고 **무한 루프(Count to Infinity)** 문제 발생 가능성이 있어 대규모 네트워크에는 부적합합니다.
> 3. **융합**: 이 문제를 해결하기 위해 링크 스테이트(Link State) 방식인 **OSPF**나 IS-IS 등으로 진화하거나, 기존 DV 방식에 스플릿 호라이즌(Split Horizon) 등의 안전장치를 추가합니다.

+++

### Ⅰ. 개요 (Context & Background) - [500자+]

**개념 및 철학**
거리 벡터(Distance Vector) 라우팅은 네트워크 내의 모든 라우터가 자신이 알고 있는 전체 네트워크의 거리 정보를 직접 이웃에게만 전파하는 방식입니다. 마치 "내 집에서 서울역까지 10km야"라는 정보만 이웃에게 알려주고, 이웃은 이를 바탕으로 "내 집에서 너 집 거리 1km + 너 집에서 서울역 10km = 11km"라고 계산하여 경로를 수립하는 방식입니다. 각 라우터는 목적지 네트워크와 그 거리(보통 **Hop Count**), 그리고 다음 홉(Next Hop) 정보를 담은 벡터를 유지 관리합니다.

**등장 배경**
초기 인터넷(**ARPANET**) 환경에서는 네트워크 토폴로지가 단순하고 라우터의 하드웨어 성능(메모리, CPU)이 제한적이었습니다. 이에 따라 모든 라우터가 전체 지도를 가질 필요 없이, 이웃과만 정보를 주고받아 메모리를 적게 쓰는 **RIP (Routing Information Protocol)** 같은 거리 벡터 방식이 널리 사용되었습니다. 그러나 네트워크 규모가 커지면서 정보 전파의 지연으로 인한 루프(Loop) 문제가 대두되었습니다.

**💡 비유: 고대 소문 통신망**
이는 마치 왕조 시대에 변방 장수들이 "수도까지 가려면 동쪽 성을 거쳐 5일이 걸린다"는 정보를 이웃 성주에게만 전달하는 것과 같습니다. 이웃은 이 말을 듣고 자신의 성에서 출발하는 기간을 더해 다시 옆 성에 전파합니다.

**📢 섹션 요약 비유**
거리 벡터 라우팅은 **"동네 방범대장들이 서로 모여서, 자신이 아는 길 정보만 쪽지에 써서 이웃에게 전달하는 소문 통신망"**과 같습니다. 정보가 한 명을 거쳐갈 때마다 왜곡되거나 시간이 오래 걸릴 수 있는 구조적 한계를 가집니다.

```ascii
[RIP 정보 전파 과정]

 [Router A]          [Router B]          [Router C]
(10.1.1.0/24)  --->  (From A: 1Hop)  --->  (From B: 2Hops)
 Direct Link          Learned Link        Learned Link
(A: "내가 출발지")    (B: "A를 거치면 1홉") (C: "B를 거치면 2홉")
```
*그림 1: 거리 벡터의 정보 누적 과정. 홉(Hop) 수가 증가할 때마다 정보의 신뢰도나 거리가 산술적으로 더해집니다.*

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive) - [1,000자+]

거리 벡터 라우팅의 핵심은 **벨만-포드(Bellman-Ford)** 알고리즘에 있으며, 주기적인 업데이트와 수렴(Convergence) 과정을 거칩니다. 여기서는 대표적인 프로토콜인 **RIP (Routing Information Protocol)**를 중심으로 분석합니다.

**구성 요소 및 내부 동작**

| 요소명 | 역할 | 내부 동작 및 프로토콜 | 비유 |
|:---|:---|:---|:---|
| **Routing Table** | 경로 정보 저장소 | 목적지, Next Hop, Metric(홉 수) 저장. 주기적으로 갱신됨. | 여행자의 수첩 |
| **Distance Metric** | 비용 측정 기준 | RIP는 **Hop Count** 사용. 대역폭이나 지연 고려 안 함. | 지나가는 다리 개수 |
| **Update Timer** | 정보 전파 주기 | 기본 **30초**마다 전체 라우팅 테이블 브로드캐스트/멀티캐스트. | 정기적인 뉴스 속보 |
| **Invalid Timer** | 경로 무효화 | 업데이트가 180초 동안 없으면 해당 경로를 '가능성 없음'으로 표시. | 뉴스 단통 끊김 |
| **Flush Timer** | 정보 삭제 | 무효화 후 120초가 지나면 테이블에서 완전히 삭제. | 전화번호부 지우기 |

**심층 동작 원리: 수렴(Convergence)의 딜레마**
라우터는 초기에 직접 연결된 네트워크만 인지합니다. 주기적으로 이웃에게 테이블을 받으면, 수학적으로 **$D(x, y) = \min(D(x, v) + d(v, y))$** 공식을 적용하여 최단 경로를 업데이트합니다. 여기서 $x$는 현재 라우터, $y$는 목적지, $v$는 이웃 라우터입니다.

그러나 이웃이 보내준 정보가 이미 오래되었거나(Stale), 네트워크 회선이 끊긴 상황에서 잘못된 정보가 들어오면 **Count to Infinity (무한대로 세기)** 현상이 발생합니다. 예를 들어, A라우터가 B라우터를 통해 목적지로 가는데, B와 목적지 사이 선이 끊겼을 때, A는 여전히 B를 통해 갈 수 있다고 착각하고 B에게 "목적지로 가는 길 물어봐", B는 "A를 통해서 가(사실 A는 자신을 통해 가려 함)"라며 서로 떠넘기는 패킷 순환 고리에 빠지게 됩니다.

```ascii
[Count to Infinity 현상 시뮬레이션]

  1. 초기 상태 (정상)
  [Net 10.0.0.0] <---(1 hop)---> [Router B] <---(1 hop)---> [Router A]
                                      ^                      |
                                      |                      |
                          (Link Cut / 선 끊김)             (2 hops via B)

  2. 장애 발생 후 업데이트 지연
  [Router B]는 10.0.0.0를 잃음.
  하지만 [Router A]는 아직 "B를 거치면 10.0.0.0에 2홉"이라는 오래된 정보를 가짐.

  3. 잘못된 정보 교환 (Loop 형성)
  B는 A에게 "10.0.0.0로 가는 길 알려줘"라고 묻거나,
  A가 B에게 "내가 10.0.0.0에 갈 수 있어(거리 2)"라고 알림.
  B는 "A를 거치면 3홉이면 되겠구나"라고 업데이트함.
  -> 다음 사이클에서 A는 4, B는 5... 무한히 증가 (RIP는 15에서 멈춤)
```

**핵심 알고리즘 코드 (의사코드)**
```python
# 벨만-포드 기반 거리 벡터 업데이트 로직
def update_routing_table(neighbor_table):
    for dest in neighbor_table:
        # 이웃이 알려준 거리 + 나까지 오는 거리(1)
        new_dist = neighbor_table[dest].metric + 1
        
        # 내 테이블에 없거나, 새로운 경로가 더 짧으면 갱신
        if dest not in my_table or new_dist < my_table[dest].metric:
            my_table[dest] = Route(dest, next_hop=neighbor, metric=new_dist)
            # 경로 변경 시 트리거드 업데이트 발생
            send_triggered_update()
```

**📢 섹션 요약 비유**
이 동작 방식은 **"지도를 보지 않고 앞사람의 뒷모습만 보고 따라가는 열차 행렬"**과 같습니다. 맨 앞의 사람이 낭떠러지로 떨어졌는데도, 뒷사람들은 아직 "앞사람이 쭉 가고 있으니 길이 있겠지"라고 착각하며 계속 앞으로 전진하다가 공중에 뜨게 되는(무한 루프) 상황이 연출됩니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy) - [비교표 2개+]

거리 벡터의 루프 문제를 해결하기 위해 다양한 기법이 존재하며, 이를 다른 라우팅 알고리즘과 비교 분석합니다.

**루프 방지 기법 심층 비교 (DV 내부 기술)**

| 기법 (Mechanism) | 동작 원리 | 장점 (Pros) | 단점 (Cons) | 적용 Protocol |
|:---|:---|:---|:---|:---|
| **Maximum Hop Count** | 홉 수가 15를 넘으면 **Infinity(16)**로 간주하여 경로 차단 | 루프가 물리적으로 무한히 돌지 않음 (시스템 붕괴 방지) | 직경이 15 이상인 네트워크 구성 불가 | RIP |
| **Split Horizon** | 정보를 받은 인터페이스로 다시 그 정보를 **광고하지 않음** | 직관적이며 설정 불필요. 2개 노드 간 루프 즉시 방지 | 3개 이상의 노드가 얽힌 복잡한 루프는 방지 못함 | RIP, IGRP |
| **Poison Reverse** | 정보를 받은 인터페이스로 **'거리 무한대(16)'**로 광고 | 상대방 경로를 즉시 삭제(Purge) 유도. Split Horizon보다 강력 | 불필요한 업데이트 패킷 증가 (대역폭 낭비) | RIP |
| **Hold-Down Timer** | 경로가 다운되면 일정 시간(180s) 동안 해당 경로 정보 **갱신 불가** | 플래핑(Flapping) 방지 및 불안정한 정보 수용 차단 | 수렴 시간이 더 길어짐 (복구 늦음) | RIP |
| **Triggered Updates** | 네트워크 변화 발생 시 **즉시** 업데이트 패킷 전송 | 수렴 속도 향상 (Count to Infinity 시간 단축) | 네트워크가 불안정할 경우 트래픽 급증 유발 | RIP v2 |

**과목 융합 관점: DV vs Link State (OSPF)**

| 구분 | Distance Vector (RIP) | Link State (OSPF) |
|:---|:---|:---|
| **지식 범위** | **부분 지식(Partial Map)**: 이웃의 정보만 알음 | **전체 지식(Full Map)**: AS(Autonomous System) 내 모든 토폴로지 파악 |
| **알고리즘** | Bellman-Ford (산술적 계산) | Dijkstra (SPF Tree 구성) |
| **수렴 속도** | 느림 (Loop 방지 로직으로 인해 지연됨) | 빠름 (변화를 감지하면 LSA 플러딩으로 즉시 전파) |
| **리소스** | CPU 낮음, 메모리 낮음 | CPU 높음(최단 경로 계산), 메모리 높음(전체 DB) |
| **활용 분야** | 소형 사무실, 초소형 네트워크 (SOHO) | 대형 기업용, 데이터 센터, ISP 백본 |

**📢 섹션 요약 비유**
**스플릿 호라이즌**은 "네가 나한테 가르쳐준 소식을, 내가 다시 너한테 가르쳐주지는 않겠다"는 약속이고, **포이즌 리버스**는 "그 소식은 사실 뻥이야, 잊어버려"라고 강력하게 부정하는 전략입니다. 반면 **OSPF**는 이런 소문 체계를 버리고 모든 주민에게 전체 지도를 배포하여 자신이 길을 찾는 **GPS 내비게이션** 시스템과 같습니다.

```ascii
[루프 방지 기법의 시각적 비교]

(1) Normal State        (2) Split Horizon       (3) Poison Reverse
  A ---(1 hop)--> B       A --(Info)---> B        A --(Poison)--> B
  A <---(Block)---- B     A X--(Return)-- B       A <--(Inf 16)-- B

  Legend:
  ---> : Normal Update
  X---- : Blocked Interface (Split Horizon)
  <--- : Infinity Metric (Poison Reverse)
```

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision) - [800자+]

실무에서 거리 벡터 프로토콜(RIP)을 도입하거나 유지보수할 때는 명확한 트레이드오프 관계를 이해해야 합니다. 특히 최신 네트워크 아키텍처에서는 RIP가 거리 사라지고 있으나, 레거시 시스템이나 특수한 목적으로 여전히 사용됩니다.

**실무 시나리오 및 의사결정**

1.  **소형 지사 네트워크 (SOHO) 구축**
    *   **상황**: 라우터가 2~3대뿐이고, 링크 대역폭이 넉넉하며 관리자가 전문 네트워크 엔지니어가 아님.
    *   **결정**: 설정의 단순성(Simplicity)이 최우선이므로 **RIP v2** 사용이 적합함. 라우팅 테이블을 수동으로 관리하는 것보다 자동화가 중요.
    *   **주의**: 최대 홉 수 15개 제한에 유의하여 네트워크 직경을 확인해야 함.

2.  **혼재 라우팅 환경 (Redistribution)**
    *   **상황**: 본사는 **OSPF**를 쓰지만,
