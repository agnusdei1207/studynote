# 380-384. IPsec 보안 프레임워크

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: IPsec (Internet Protocol Security)은 **L3 (Network Layer)** 계층에서 통신 보안을 제공하는 표준화된 아키텍처로, 패킷의 출처 인증, 무결성 검증, 기밀성(암호화)을 포트(Port) 번호 변경 없이 처리하는 'OSI 3계층 보안'의 정점입니다.
> 2. **가치**: 애플리케이션 수정 없이 네트워크 장비(Router/Firewall) 레벨에서 VPN (Virtual Private Network)을 구축하여, 공용 인터넷망을 마치 전용선처럼 안전하게 활용할 수 있게 하며, **AH(Authentication Header)**와 **ESP(Encapsulating Security Payload)**를 통해 유연한 보안 정책 구현이 가능합니다.
> 3. **융합**: OSI 7계층의 보안 프로토콜 중 가장 강력한 성능을 자랑하며, **L2TP**나 **GRE** 같은 터널링 프로토콜과 결합하여 **Site-to-Site VPN** 및 **Remote Access VPN**의 핵심 전송 수단으로 활용됩니다.

---

### Ⅰ. 개요 (Context & Background)

**IPsec (Internet Protocol Security)**은 IETF (Internet Engineering Task Force)에서 표준화한, IP (Internet Protocol) 패킷 계층에서 보안 서비스를 제공하는 프로토콜 스위트(Suite)입니다. 기존 IP 패킷이 Plain Text로 전송되는 인터넷의 구조적 취약점을 해결하기 위해 설계되었으며, OSI 7계층 중 **L3 (Network Layer)**에서 동작하므로 상위 애플리케이션의 변경 없이 투명하게 보안을 적용할 수 있다는 특징이 있습니다.

IPsec은 단일 프로토콜이 아니라 **AH(Authentication Header)**, **ESP(Encapsulating Security Payload)**, **IKE(Internet Key Exchange)** 등 여러 보안 프로토콜의 집합체입니다. L2 계층의 SSL/TLS가 '애플리케이션' 단위의 보안에 집중한다면, IPsec은 '네트워크' 자체를 보호하는 데 중점을 둡니다.

**작동 모드 (Mode of Operation)**
IPsec은 두 가지 주요 모드로 동작하며, 이 선택은 트래픽의 성격과 게이트웨이 존재 여부에 따라 결정됩니다.

1.  **전송 모드 (Transport Mode)**:
    *   **정의**: IP 패킷의 **페이로드(Payload)** 부분만 보안 처리합니다. 원본 IP 헤더는 그대로 둡니다.
    *   **용도**: 주로 **Host-to-Host** (엔드포인트 간) 통신에 사용됩니다. IPsec을 지원하는 최종 서버나 PC끼리 직접 통신할 때 사용합니다.
    *   **오버헤드**: 헤더 추가가 최소화되어 효율적입니다.

2.  **터널 모드 (Tunnel Mode)**:
    *   **정의**: **원본 IP 패킷 전체를 새로운 IP 패킷의 페이로드로 캡슐화(Encapsulation)**합니다. 따라서 새로운 **외부 IP 헤더(Outer IP Header)**가 추가됩니다.
    *   **용도**: **Gateway-to-Gateway** (Site-to-Site VPN) 구축 시 필수적입니다. 내부 IP 주소를 외부에 노출시키지 않고 보안 통신이 가능합니다.
    *   **구조**: `[New IP Header] [IPsec Header] [Original IP Header] [Data]`

> **💡 비유: 편지 봉투와 박스**
> *   **전송 모드**: 편지 봉투 안의 내용만 암호화하여 보내는 것. 주소는 그대로.
> *   **터널 모드**: 편지 봉투 그 자체를 튼튼한 금고 상자에 넣고, 그 상자에 다시 주소를 붙여서 보내는 것. 연예인이 이사할 때 박스 안에 짐을 넣고 박스 겉에 새 주소를 쓰는 것과 같습니다.

**등장 배경 및 필요성**
인터넷 초창기에는 암호화된 통신이 선택 사항이었으나, 전자상거래, 기업 데이터 센터의 분산, 재택근무의 확산으로 인해 **데이터 도청(Sniffing)**, **위장(Spoofing)**, **변조(Tampering)**에 대한 방어가 필수가 되었습니다. 특히 기업 간의 사설망 구축 비용을 줄이기 위해 공용 인터넷을 사설망처럼 쓰고자 하는 **VPN (Virtual Private Network)**의 핵심 기술로 IPsec이 자리 잡았습니다.

```ascii
┌─────────────────────────────────────────────────────────────────────┐
│                     IPsec 동작 모드 비교 (Architecture)              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [Transport Mode] (Host-to-Host)                                   │
│  ┌─────────┐ ┌───────────────┐ ┌───────────────────────────┐       │
│  │ IP Hdr  │ │  IPsec Hdr    │ │     Payload (Data)        │       │
│  │ (Original)│ (AH/ESP)      │ │     (Encrypted/Auth)      │       │
│  └─────────┘ └───────────────┘ └───────────────────────────┘       │
│   └─ 변경 없음      └─ 보안 처리                                        │
│                                                                     │
│---------------------------------------------------------------------│
│                                                                     │
│  [Tunnel Mode] (Gateway-to-Gateway / VPN)                           │
│  ┌─────────┐ ┌───────────────┐ ┌─────────┐ ┌───────────┐ ┌───────┐ │
│  │New IP Hr│ │  IPsec Hdr    │ │Org IP Hr│ │  Payload  │ │ ESP...│ │
│  │ (Gateway)│ │ (AH/ESP)     │ │ (Hidden)│ │ (Data)    │ │       │ │
│  └─────────┘ └───────────────┘ └─────────┘ └───────────┘ └───────┘ │
│      └─ 외부 헤더      └─ 보안 처리      └──── 원본 패킷 전체 캡슐화 ───┘    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```
*   **해설**: 위 다이어그램은 전송 모드와 터널 모드의 패킷 구조적 차이를 보여줍니다. 터널 모드에서는 원본 IP 헤더가 보안 영역 안으로 들어가기 때문에, 중간에 있는 라우터나 공격자는 최종 목적지(Internal IP)를 확인할 수 없어 **익명성**과 **보안성**이 훨씬 높습니다. 이는 기업 본사와 지사 간 연결에 반드시 필요한 구조입니다.

📢 **섹션 요약 비유**: IPsec의 도입은 마치 일반 우편물을 보내던 우편 시스템에 **등기 우편 시스템과 은행 금고 차량**을 도입하는 것과 같습니다. 단순히 편지 봉투를 봉하는 것(전송 모드)을 넘어, 내용물을 아무도 볼 수 없는 금고에 넣고 운송 차량을 통제함으로써(터널 모드), 공용 도로(인터넷) 위에서도 사설 창고(본사) 간의 물류이동을 안전하게 보장하는 것입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

IPsec 아키텍처의 핵심은 **어떤 보안 서비스를 제공할 것인가(Protocol)**와 **어떻게 키를 관리할 것인가(Key Management)**, 그리고 **이 정보를 어떻게 유지할 것인가(SA)**로 나뉩니다. 가장 중요한 두 가지 프로토콜인 **AH**와 **ESP**의 상세 기술을 비교 분석합니다.

#### 1. AH (Authentication Header)
AH는 데이터의 무결성(Integrity)과 출처 인증(Authentication)을 제공하지만, **기밀성(Confidentiality, 암호화)은 제공하지 않습니다.**

*   **구조**: AH는 원본 IP 헤더와 페이로드 사이에 삽입됩니다.
*   **동작 원리**: 송수신 측이 공유하고 있는 **비밀 키(Secret Key)**와 데이터를 합쳐 **ICV (Integrity Check Value)**라는 해시값을 생성하여 헤더에 붙입니다. 수신자는 이를 다시 계산하여 변조 여부를 확인합니다.
*   **한계 (NAT Traversal 불가)**: AH는 IP 헤더의 불변 필드를 포함하여 전체에 대해 ICV를 계산합니다. 만약 중간에 NAT 장비가 IP 주소를 변경하면, 수신자가 계산한 ICV 값과 원본 ICV 값이 달라지므로 패킷을 폐기합니다. 따라서 AH는 NAT 환경(대부분의 가정/사무실 네트워크)에서 사용하기 매우 어렵습니다.

#### 2. ESP (Encapsulating Security Payload)
ESP는 **무결성, 인증뿐만 아니라 암호화(기밀성)까지 제공**하는 실질적인 표준입니다. AH보다 구조가 복잡하지만 강력합니다.

*   **구조**: ESP Header, Payload(암호화됨), ESP Trailer, ESP Auth Data로 구성됩니다.
*   **동작 원리**:
    1.  **암호화**: 데이터(Payload)와 패딩(Padding), 패딩 길이, 다음 헤더 필드를 암호화 알고리즘(예: **AES-CBC**, **AES-GCM**)을 이용해 암호화합니다.
    2.  **인증**: 암호화된 부분과 ESP Header를 포함하여 무결성 검사를 수행합니다. (단, IP 헤더는 인증 범위에서 제외될 수 있어 NAT와 상호 운용 가능합니다)

**구성 요소 비교표**

| 구분 | AH (Authentication Header) | ESP (Encapsulating Security Payload) |
| :--- | :--- | :--- |
| **주요 역할** | 데이터 무결성 및 송신자 인증 | **암호화(기밀성)** + 무결성 및 인증 |
| **암호화 지원** | ❌ 지원 안 함 (데이터 노출) | ✅ 지원 (데이터 은폐) |
| **NAT 호환성** | ❌ 불가능 (IP Header 포함 검증) | ✅ 가능 (NAT-T 지원 시) |
| **필드 구조** | Next Header / Payload Len / Reserved / ICV | SPI, Sequence Num / Payload Data (Enc) / Pad / Pad Len / NextHdr / ICV |
| **사용 사례** | 거의 사용되지 않음 (특수 목적) | **VPN 표준** (대부분의 IPsec 구현체) |

**심층 동작: SA (Security Association)와 데이터 처리 플로우**

IPsec은 **SA (Security Association)**라는 논리적 연결을 통해 통신을 보호합니다. SA는 일방향(One-way) 개념이므로 양방향 통신을 위해서는 2개의 SA가 필요합니다. SA의 정보는 **SPI (Security Parameter Index)**, 목적지 IP 주소, 보안 프로토콜(AH/ESP)의 조합으로 식별됩니다.

```ascii
┌─────────────────────────────────────────────────────────────────────┐
│                   IPsec 처리 플로우 (Outbound Process)               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. IP Packet Generation                                           │
│  ┌───────────────┐ ┌───────────────────────────────────────┐       │
│  │   IP Header   │ │         TCP/UDP Data (Payload)        │       │
│  └───────────────┘ └───────────────────────────────────────┘       │
│           │                                                          │
│           ▼                                                          │
│  2. IPsec Engine (SPD & SAD Lookup)                                 │
│  ┌─────────────────────────────────────────────────────────┐       │
│  │  - SPD (Security Policy Database): "보안 정책 확인"        │       │
│  │    -> 이 패킷은 IPsec 처리가 필요한가? (Match?)           │       │
│  │  - SAD (Security Association Database): "키 및 알고리즘"   │       │
│  │    -> 어떤 SPI, Key, 알고리즘을 쓸지 조회               │       │
│  └─────────────────────────────────────────────────────────┘       │
│           │                                                          │
│           ▼                                                          │
│  3. Encryption & Authentication (ESP)                               │
│  ┌───────────────┐ ┌───────────────────────────────────────┐       │
│  │   IP Header   │ │ [ESP Hdr] [Encrypted Payload] [Auth]   │       │
│  └───────────────┘ └───────────────────────────────────────┘       │
│     (Original)         └───── AES-256 / HMAC-SHA-256 ─────┘       │
│           │                                                          │
│           ▼                                                          │
│  4. Tunneling (If Tunnel Mode)                                      │
│  ┌───────────────┐ ┌───────────────────────────────────────────┐   │
│  │  New IP Header│ │  [Whole Packet above]                    │   │
│  └───────────────┘ └───────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```
*   **해설**:
    1.  **SPD (Security Policy Database)**는 방화벽 규칙처럼 들어오고 나가는 패킷을 필터링하여 "IPsec 처리 대상"인지 결정합니다.
    2.  대상으로 결정되면 **SAD (Security Association Database)**를 참조하여 이 패킷을 캡슐화하고 암호화하는 데 필요한 세부 정보(암호화 키, 수명, 알고리즘)를 가져옵니다.
    3.  **AES (Advanced Encryption Standard)**나 **3DES (Triple Data Encryption Standard)** 등의 알고리즘을 사용하여 페이로드를 암호화하고, **HMAC (Hash-based Message Authentication Code)**을 통해 인증 데이터를 생성합니다.
    4.  이 과정은 라우터의 CPU나 전용 하드웨어 가속기(ASIC/HSM)에서 고속으로 처리됩니다.

📢 **섹션 요약 비유**: AH와 ESP의 관계는 **'봉인 테이프'와 '금고'**의 차이와 같습니다. AH는 택배 상자의 테이프가 뜯겼는지 확인하여 변조 여부만 알 수 있지만, 내용물이 훤히 보입니다. 반면 ESP는 상자 자체를 투명하지 않은 금고로 교체하여 내용물을 숨기고(암호화), 금고 문을 열었는지까지 확인합니다(인증). SA는 이 금고를 열 수 있는 **'일회용 열쇠
