> **핵심 인사이트**
> 1. **본질**: 소프트웨어 아키텍처(Software Architecture)는 시스템의 구성 요소, 요소 간 관계, 설계 원칙을 통해 시스템의 全生命주기品質을 결정하는 뼈대 구조이며, SOLID 원칙과 모듈화는 응집도를 높이고 결합도를 낮추는 설계 철학의 핵심이다.
> 2. **가치**: 좋은 아키텍처는 요구사항 변경에 유연하게 대응하여 기능 추가 비용을 50 이상 절감하고, 버그 발생률을 80 이상 감소시키며, 신규 개발자 onboarding 시간을 단축한다.
> 3. **융합**: 클라우드 네이티브, 마이크로서비스, 이벤트 구동 아키텍처(EDA)가 전통적 모놀리식 설계를 대체하며, 각 패러다임의 트레이드오프를 깊이 이해하는 것이 기술사적 판단의 핵심이다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

소프트웨어 아키텍처(Software Architecture)란 소프트웨어 시스템을 구성하는 구성 요소(Component), 구성 요소 간의 관계(Relationship), 구성 요소와 환경 간의 상호 작용(Interaction), 그리고 이러한 설계를 지배하는 원칙(Principle)을 포괄하는 상위 수준 구조를 말한다. 건축에서 구조 도면이 붕어남 없이 건물을 지탱하는 핵심인 것처럼, 소프트웨어에서도 아키텍처는 시스템의 확장성, 유지보수성, 성능, 보안 등 全반면에 영향을 미치는根本적 결정이다.

### 문제의식: 왜 아키텍처가 중요한가

소프트웨어 개발 초기에는 "동작하면 그만"이라는 마인드셋으로 코드를 작성할 수 있었다. 그러나 시스템 규모가 성장하고 팀이 확대되면, 초기 설계의欠陥이 모든 곳에 발목을 잡는다. 한 모듈의 변경이 다른 모듈의 예기치 않은 오류를 유발하고, 새로운 기능 추가가 기존 기능의 버그를 야기하는 "레고 블록이 아니라 찰흙 같은 코드"가 탄생한다. 이것이 기술 부채(Technical Debt)이며, 아키텍처 원칙은 이 기술 부채를 원천적으로 억제하는 설계 철학이다.

### SOLID 원칙의 등장 배경

로버트 마틴(Robert C. Martin)이 정리한 SOLID 원칙은 2000년대 초 객체지향 설계의最佳实践을 모은 것이다. 과거에는 상속(Inheritance)을 통한 코드 재사용이美德로 여겨졌으나, 실제로는 결합도가 높아져 변경이 어려워지는 문제가 발생했다. SOLID 원칙은 이Paradigm의 전환을 통해 "변경에는 닫혀 있고, 확장에는 열려 있는" 유연한 설계를 가능하게 한다.

SOLID 원칙이 기존 절차적 설계의 한계를 어떻게 극복하는지를 시각화하면, 객체지향의 "캡슐화와 추상화"가 如何して 결합도를 줄이는지 명확히 드러난다.

```text
┌─────────────────────────────────────────────────────────────────┐
│        절차적 설계 vs 객체지향 설계 (SOLID 적용 전/후)              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [절차적 설계 (Procedural Design)]                               │
│                                                                 │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐                   │
│  │ 기능 A   │──▶│ 기능 B   │──▶│ 기능 C   │                   │
│  │ (함수)   │   │ (함수)   │   │ (함수)   │                   │
│  └──────────┘   └──────────┘   └──────────┘                   │
│       │              │              │                          │
│       └──────────────┴──────────────┘                          │
│                    │                                            │
│               공유 데이터 (Global State)                         │
│               → A 변경 → B, C에 파급 효과 (Ripple Effect)       │
│               → 단위 테스트 어려움                               │
│               → 기능 추가 시 예측 불가능한 버그                   │
│                                                                 │
│  [객체지향 설계 (SOLID 적용)]                                    │
│                                                                 │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐                   │
│  │ 객체 A   │   │ 객체 B   │   │ 객체 C   │                   │
│  │ ┌──────┐│   │ ┌──────┐│   │ ┌──────┐│                   │
│  │ │상태   ││   │ │상태   ││   │ │상태   ││                   │
│  │ │행동   ││   │ │행동   ││   │ │행동   ││                   │
│  │ └──────┘│   │ └──────┘│   │ └──────┘│                   │
│  └────┬────┘   └────┬────┘   └────┬────┘                   │
│       │ Interfaces │             │                          │
│       └────────────┴─────────────┘                          │
│                    │                                            │
│            느슨한 결합 (Loose Coupling)                         │
│            → A 변경이 B, C에 영향 없음                         │
│            → 독립적 단위 테스트 가능                            │
│            → 기능 추가/변경 용이                                │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 절차적 설계에서 함수 A, B, C가 공유 데이터(Global State)에 직접 접근하면, A의 변경이 B와 C의 동작에 예기치 않은 영향을 미친다. 반면 SOLID 원칙을 적용한 객체지향 설계에서는 각 객체가 자신의 상태와 행동을 캡슐화하고, 객체 간 통신은 명시적 인터페이스(Interface)를 통해서만 이루어진다. 이 구조의 핵심 이점은 한 객체의 내부가 변경되어도 인터페이스가 동일하면 의존하는 객체에 영향이 없다는 점이다. 실무에서 이것은 "A 모듈의 데이터베이스 로직을 교체해도 B 모듈의 비즈니스 로직이 영향받지 않는다"는 것을 보장한다.

### 아키텍처 품질을 결정하는 두 가지 축

응집도(Cohesion)와 결합도(Coupling)는 아키텍처 품질을 결정하는 가장 기본적인 두 척도다. 응집도는 하나의 모듈 내부에서 요소들이 얼마나 밀접하게 관련되어 있는지를 나타내고, 결합도는 서로 다른 모듈 간 얼마나 느슨하게 연결되어 있는지를 나타낸다. 좋은 설계는 "높은 응집도 + 낮은 결합도"를 목표로 한다.

### 비유

소프트웨어 아키텍처는 도시 계획과 같다. 초기에는 작은 마을이라 어떤 구조로든 살 수 있지만, 도시가 성장하면道路(接口), 지역구분(모듈화), 기반시설(공통 인프라)이 체계적으로规划设计되어야 한다. SOLID 원칙은 이때 각 지역이自成적으로發展하면서도 전체 도시와 어떻게 조화를 이루는지를定義する原則集である.

📢 섹션 요약 비유: 아키텍처 원칙을 지키지 않은 코드는 마치熨斗大手術装潢없이増改築を続けた古い建物のようで、壁の一カ所を取り壊すだけで建物全体が揺れ動く危うさがある.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SOLID 원칙 상세 분석

| 원칙 | 축약 | 핵심 내용 | 안티패턴 예시 | 적용 효과 |
|:---|:---|:---|:---|:---|
| **단일 책임 원칙 (SRP)** | S | 한 클래스는 하나의 책임만 져야 한다 | 모든 비즈니스 로직을 한 클래스에 때려넣음 (God Object) | 변경 대상이 명확화 |
| **개방-폐쇄 원칙 (OCP)** | O | 확장에는 열려 있고, 수정에는 닫혀 있어야 한다 | if-else로 새 기능 추가마다 분기 증가 | 기능 추가 시 기존 코드 변경 없음 |
| **리스코프 치환 원칙 (LSP)** | L | 자식 클래스는 부모 클래스로 대체 가능해야 한다 | 하위 클래스에서 부모 동작을 기대와 다르게 재정의 | 다형성 안전성 확보 |
| **인터페이스 분리 원칙 (ISP)** | I | 클라이언트는 사용하지 않는 메서드에 의존하면 안 된다 | 큰 단일 인터페이스에 불필요한 메서드强迫 | 불필요한 의존성 제거 |
| **의존성 역전 원칙 (DIP)** | D | 고수준 모듈이 저수준 모듈에 의존하지 않고, 둘 다 추상에 의존한다 | concretas 클래스에 직접 의존 | 결합도 최소화, 테스트 용이성 |

### OCP와 전략 패턴(Strategy Pattern)의 결합

개방-폐쇄 원칙(OCP)을 위반하는 전형적인 안티패턴은 새로운 결제 수단 추가마다 if-else 분기를 증가시키는 것이다. 이러한 분기는 결제 수단이 추가될 때마다 테스트해야 할 경로가指数的に増加하고, 기존 테스트의 재실행이 필수적이어서버그 도입 위험이 높아진다.

OCP를 지키면서 결제 수단을 확장하는 구체적 구현을 살펴보면, 전략 패턴(Strategy Pattern)이如何して 변경에 열려 있는 설계를 가능하게 하는지 이해할 수 있다.

```text
┌─────────────────────────────────────────────────────────────────┐
│      OCP 적용: 전략 패턴(Strategy Pattern)을 통한 결제 시스템       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [안티패턴: OCP 위반]                                           │
│                                                                 │
│  public void pay(String type, int amount) {                     │
│      if (type.equals("CREDIT")) {                               │
│          // 신용카드 결제 로직                                   │
│      } else if (type.equals("KAKAO")) {                        │
│          // 카카오 결제 로직                                     │
│      } else if (type.equals("NAVER")) {                        │
│          // 네이버 결제 로직                                     │
│      }  // 새로운 결제 수단 추가마다 이 메서드 수정 → OCP 위반     │
│  }                                                              │
│                                                                 │
│  [올바른 설계: OCP 준수]                                         │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │              <<interface>>                               │   │
│  │            PaymentStrategy                              │   │
│  │  ┌─────────────────────────────────────────────┐        │   │
│  │  │ + pay(amount: int): void                    │        │   │
│  │  └─────────────────────────────────────────────┘        │   │
│  │         ▲              ▲              ▲                  │   │
│  │         │              │              │                  │   │
│  │  ┌──────┴─────┐  ┌──────┴─────┐  ┌──────┴─────┐         │   │
│  │  │CreditPayment│  │KakaoPayment│  │NaverPayment│         │   │
│  │  └────────────┘  └────────────┘  └────────────┘         │   │
│  │                                                         │   │
│  │  ※ 새로운 결제 수단 추가 시?                              │   │
│  │    → NewPaymentStrategy 인터페이스 구현 클래스 생성       │   │
│  │    → 기존 코드 수정 ZERO                                  │   │
│  │    → OCP 완전 준수                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  public class Checkout {                                        │
│      private PaymentStrategy strategy;  // DIP: 추상에 의존      │
│                                                                 │
│      public void setStrategy(PaymentStrategy strategy) {        │
│          this.strategy = strategy;                              │
│      }                                                          │
│                                                                 │
│      public void execute(int amount) {                          │
│          strategy.pay(amount);  // OCP: 확장에는 열려있음       │
│      }                        // 수정에는 닫혀 있음              │
│  }                                                              │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단 안티패턴에서는 새로운 결제 수단이 추가될 때마다 pay() 메서드를 수정해야 하며, 이 변경은 기존 테스트의 재실행과 회귀 테스트(Regression Test) 부담을 증가시킨다. 하단의 올바른 설계에서는 PaymentStrategy 인터페이스가 결제 행동의 "추상"을定義하고, 각 결제 수단이 이 인터페이스를 구현한다. Checkout 클래스는 구체적인 결제 수단 클래스가 아니라 추상(인터페이스)에 의존하므로(DIP 원칙), 새로운 결제 수단이 추가되어도 Checkout 자체는 변경되지 않는다. 실무에서 이 구조는 결제 모듈의 단위 테스트가 매우 용이해지며, 각 전략 클래스를 독립적으로 배포할 수 있다는 Advantages가 있다.

### DRY, KISS, YAGNI 원칙

SOLID 외에 반드시 숙지해야 할 설계 원칙이 있다. DRY(Don't Repeat Yourself)는相同的 지식(knowledge)이 두 곳 이상에 중복되지 않도록 하는 원칙으로, 중복은 하나의 변경이 다른 곳에서의 갱신을 잊어버리게 하여 불일치를 야기한다. KISS(Keep It Simple, Stupid)는設計をシンプルに保という警告であり、複雑な解决方法よりも単純な解决方法を目指す。 YAGNI(You Aren't Gonna Need It)은 현재 필요하지 않은 기능을 미리 구현하지 말고,真正 필요할 때 구현하라는 원칙이다.

### 모놀리식 vs 마이크로서비스 아키텍처 비교

시스템 규모가 커짐에 따라 모놀리식(Monolithic) 아키텍처의 한계가 드러난다. 모놀리식에서는 일부 모듈의 장애가 전체 시스템 장애로 확대되고, 한 모듈의 deploy가 전체 시스템의 가용성을 위협한다. 마이크로서비스 아키텍처(MSA)는 이 문제를 각 기능을 독립적 서비스로 분리하여 해결하지만, 서비스 간 네트워크 통신 지연, 분산 트랜잭션, 데이터 일관성 등의 새로운 복잡성이 발생한다.

두 아키텍처 패턴의 구조적 차이를 비교하면, 모놀리식이 단일 배포 단위인 반면 MSA는 독립적 배포 단위라는 점이根本적으로 다르다.

```text
┌─────────────────────────────────────────────────────────────────┐
│          모놀리식 vs 마이크로서비스 아키텍처 비교                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [모놀리식 아키텍처]                                             │
│                                                                 │
│  ┌───────────────────────────────────────────────────────┐     │
│  │                    단일 배포 단위                        │     │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐  │     │
│  │  │ 주문    │  │ 결제    │  │ 배송    │  │ 상품    │  │     │
│  │  │ 모듈    │  │ 모듈    │  │ 모듈    │  │ 모듈    │  │     │
│  │  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘  │     │
│  │       └────────────┴────────────┴────────────┘        │     │
│  │                      │                                  │     │
│  │              ┌───────┴───────┐                          │     │
│  │              │  단일 DB      │                          │     │
│  │              └───────────────┘                          │     │
│  └───────────────────────────────────────────────────────┘     │
│                                                                 │
│  - 배포: 전체 애플리케이션了一起 배포                            │
│  - 확장: 특정 모듈만扩展不可 (전체 확장의 해야 함)               │
│  - 장애 영향: 한 모듈 버그 → 전체 시스템 장애 가능              │
│                                                                 │
│  [마이크로서비스 아키텍처 (MSA)]                                │
│                                                                 │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐     │
│  │ 주문    │    │ 결제    │    │ 배송    │    │ 상품    │     │
│  │ 서비스  │    │ 서비스  │    │ 서비스  │    │ 서비스  │     │
│  │ [独自 DB]│    │ [独自 DB]│    │ [独自 DB]│    │ [独自 DB]│     │
│  └────┬────┘    └────┬────┘    └────┬────┘    └────┬────┘     │
│       │              │              │              │           │
│       └──────────────┴──────────────┴──────────────┘           │
│                           │                                      │
│                    ┌──────┴──────┐                               │
│                    │ API Gateway │                               │
│                    └─────────────┘                               │
│                                                                 │
│  - 배포: 각 서비스 독립 배포 가능                                 │
│  - 확장: 특정 서비스만 확장 가능                                  │
│  - 장애 격리: 한 서비스 장애 → 해당 기능のみ影響                 │
│  - Trade-off: 네트워크 지연, 분산 트랜잭션 복잡                   │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 모놀리식에서는 모든 모듈이 동일한 프로세스 내에서 동작하고 단일 데이터베이스를 공유한다. 이 구조는 개발 초기에는简单하지만, 시스템 규모가 커질수록 배포 시간 증가, 부분적 확장 어려움, 장애 영향 범위 확대 등의 문제에 직면한다. MSA에서는 각 서비스가 독립적 프로세스로 동작하고 자신의 데이터베이스를 보유한다. 서비스 간 통신은 네트워크를 통한 API 호출로 이루어지며, 이를 통해 특정 서비스의 확장 또는 장애를他のサービス로부터 독립시킬 수 있다. 그러나 분산 환경에서의 트랜잭션 일관성( Saga 패턴, CQRS 등)과 네트워크 병목 관리가 새로운設計課題이다.

📢 섹션 요약 비유: 모놀리식 아키텍처는 하나의 큰 레고 블록으로 만든 작품으로,一部分을 바꾸려면 전체를 다시分解しなければならない. MSA는 작은 레고 블록들로 조립한 작품으로,特定のブロックだけを別の色に取り替えることができるが、ブロック間の接着剂(네트워크 통신)가 필요하다는 단점이 있다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### Layered vs MVC vs Clean Architecture 비교

아키텍처 패턴은 系统의構造를 어떻게 组织화할 것인가에 대한 해결 방법이다. Layered Architecture는 가장 전통적인 패턴으로, 표현-비즈니스-데이터 계층으로 분리한다. MVC(Model-View-Controller)는 Layered의 변형으로, 웹 애플리케이션에서広く使われています。 Clean Architecture는 로버트 마틴이 제안한もので, 의존성 방향을 외부에서 내부로만 흐르게 하여 핵심 비즈니스 규칙을 외부 변화로부터 보호한다.

세 패턴의 구조적 차이를 비교하면, Clean Architecture가他の两者と 달리 의존성 방향을 명시적으로 제어한다는 점이 다르다.

```text
┌─────────────────────────────────────────────────────────────────┐
│       Layered vs MVC vs Clean Architecture 비교                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [Layered Architecture]           [MVC Architecture]          │
│                                                                 │
│  ┌─────────────────────┐           ┌─────────────────────┐       │
│  │  Presentation Layer│           │      View           │       │
│  │  (UI, API)         │           │  (HTML, JSP)        │       │
│  ├─────────────────────┤           ├─────────────────────┤       │
│  │  Business Layer    │           │    Controller      │       │
│  │  (Logic)           │           │  (Request 처리)    │       │
│  ├─────────────────────┤           ├─────────────────────┤       │
│  │  Data Access Layer │           │     Model           │       │
│  │  (DB)              │           │  (비즈니스 로직, DB)│       │
│  └─────────────────────┘           └─────────────────────┘       │
│        │ 의존성: 위에서 아래            │ 의존성: Controller→Model │
│                                                                 │
│  [Clean Architecture]                                            │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐     │
│  │                    External Agencies                     │     │
│  │              (UI, DB, External Interfaces)               │     │
│  ├─────────────────────────────────────────────────────────┤     │
│  │                    Interface Adapters                    │     │
│  │         (Controllers, Gateways, Presenters)              │     │
│  ├─────────────────────────────────────────────────────────┤     │
│  │                    Application Business Rules            │     │
│  │              (Use Cases, Application Services)           │     │
│  ├─────────────────────────────────────────────────────────┤     │
│  │                    Enterprise Business Rules             │     │
│  │                  (Entities, Domain Models)               │     │
│  └─────────────────────────────────────────────────────────┘     │
│         ▲ 모든 의존성은 안쪽(Entities)을 향함                     │
│         ▲ 외부 변화가 내부로浸透하지 않음                          │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Layered와 MVC는 의존성이 위에서 아래(또는 전형적으로前方에서後方으로)로 흐르지만, Clean Architecture는 이 의존성 방향을 반대로 뒤집는다. 핵심은 Enterprises Business Rules(Entities)가 외부 세계에 대해何も知らない这一点이다. UI가 바뀌어도, 데이터베이스가 교체되어도 Entities는 변경되지 않아야 한다. 이것이 바로 "비즈니스 핵심을 외부 변화로부터 보호한다"는 Clean Architecture의 핵심 가치다. 실무에서 이는 새로운 UI 기술(React에서 Vue로)의 도입이나 데이터베이스 변경(MySQL에서 PostgreSQL로)이 핵심 비즈니스 로직에 영향을 주지 않다는 것을 보장한다.

### 과목 융합 관점

- **데이터베이스 정규화**: Clean Architecture의 Entities 설계 시 정규화(Normalization) 원칙을 적용하면, 데이터 모델의冗長성을 제거하고 변경의 영향을最小化할 수 있다. 정규화가 과도하게 적용되면 조인 비용이 증가하는 트레이드오프가 존재한다.
- **네트워크 프로토콜**: MSA에서 서비스 간 통신은 HTTP/gRPC/Kafka 등 다양한 프로토콜을 사용하며,これらの選択がシステム性能と複雑性に直接影響する。

📢 섹션 요약 비유: 아키텍처 패턴을 선택하는 것은 목공점에서家具의構造를 결정하는 것과 같다. Layered는 전통的な抬い家具, MVCは一般的なテーブル家具, Clean Architectureはまるで家具의根本(構造骨架)과装飾(외관)을 완전히 분리하여、골조만 완성해도功能은하고, 나중에外观만 변경할 수 있는 模块式 시스템이다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 시나리오 1: 과도한 엔지니어링(Over-Engineering) 함정

**문제 상황**: 소규모 내부 도구 개발에 Clean Architecture를 도입하여 5명 팀이 3개월간架构만 设计하고 실제 기능 开发에 들어가지 못하는 상황. 과도한抽象화가 오히려 생산성을 저해하고 있다.

**기술사적 판단**: 아키텍처 복잡성은 시스템 규모와 팀 규모에 비례해서 선택해야 한다. 5명 이하 소규모 팀의 내부 도구라면 Layered Architecture 또는 간단한 MVC로 충분하며, YAGNI 원칙에 따라未来的인 확장성을 미리 구현하지 않아야 한다. 기술사로서 "모든 프로젝트에 동일한 아키텍처를 적용하는 것" 자체가 안티패턴임을 인식하고,プロジェクトの性質에 맞는 최적의設計を選択する 능력이 필요하다.

### 시나리오 2: 레거시 시스템의 마이크로서비스 전환

**문제 상황**: 15년 된 모놀리식 ERP 시스템을 MSA로 전환하려는 계획. 현재 시스템은 200만 줄 이상의 코드, 50명 이상의 개발 인력이 투입되어 있으며, 장애 시 전체 시스템이 멈춘다.

**기술사적 판단**: MSA 전환은strangler fig 패턴(외부에서 내부로慢慢地 교체)을 적용해야 한다. 한 번에 전체 시스템을 전환하면 실패 시，恢复할 방법이 없다. 또한 모든 모듈이 MSA에 적합한 것은 아니므로, 변경 빈도가 높고 독립적 배포가 필요한 模块부터先行 적용해야 한다. 전환 비용이 전환으로 인한 절감 비용을上回ることを検証する分析가事前에 필요하다.

### 도입 체크리스트

**기술적 항목**:
- 현재 시스템 규모와 팀 규모에 적합한 아키텍처 패턴을 선택했는가?
- SOLID 원칙 위배 코드가 전체의 20%를 초과하지 않는가?
- 새 기능 추가 시 기존 테스트의 80 이상 재실행해야 하는가?(OCP 위반 신호)
- 모듈 간 순환 참조(Circular Dependency)가 존재하는가?

**운영·보안적 항목**:
- 아키텍처 결정의 근거(Architecture Decision Record, ADR)가 문서화되어 있는가?
- 주요 설계 결정에 대한トレードオフ分析이 존재하는가?
- 장애 발생 시影響範囲을 제한할 수 있는 설계가 되어 있는가?

### 안티패턴

**God Object 안티패턴**: 하나의 클래스에 모든 책임(бизнес 로직, DB 접근, API 호출, 뷰 렌더링까지)을 때려넣는 경우. SRP(단일 책임 원칙) 명백한 위반이며, 테스트 불가능하고 유지보수 불가능한 코드로 바로 이어진다.

**강한 결합도 안티패턴**: конкрет 클래스 간 직접 의존. 새로운 구현으로 교체하려면 의존하는 모든 클래스를 수정해야 하는 상황. DIP(의존성 역전 원칙)을 적용하여 추상에 의존하게 리팩토링해야 한다.

📢 섹션 요약 비유: 아키텍처 안티패턴은 마치 요리사 한 명이 주방的所有器具的使用까지 모두 관리해야 하는状况に似ている。 包丁(도마), 팔炉(레인지), 재고 관리(장보기)까지 한 사람이 하면 당연히 지친다. 요리사(бизнес 로직)에게最好的 환경은 필요한 도구를接口으로 받을 수 있게하는 주방環境(아키텍처)이다.

## 📌 관련 개념 맵 (Knowledge Graph)

- **SOLID 원칙**: SRP, OCP, LSP, ISP, DIP 5개 원칙의 머리글자로, 객체지향 설계의核心 가이드라인
- **策略 패턴 (Strategy Pattern)**: OCP를 실현하는 대표적 디자인 패턴으로, 알고리즘을 교환 가능하게 하는 구조
- **모놀리식 vs MSA**: 두 아키텍처 접근 방식의 트레이드오프를 이해하는 것은 아키텍처選択の核心
- **Clean Architecture**: 의존성 방향을 외부에서 내부로 흐르게 하여 핵심 비즈니스 규칙을 보호하는 설계 방법론
- **기술 부채 (Technical Debt)**: 아키텍처 원칙 미준수로 인한 장기적 개발 비용 증가 현상
- **도메인 주도 설계 (DDD)**:业务 개념을코드구조에 반영하여ビジネス邏輯と기술 구현을 Align시키는 설계 방법론

## 👶 어린이를 위한 3줄 비유 설명

1. 소프트웨어 아키텍처는 마치「大장난꾸러기 장난감 상자整理」と同じで、類玩具(모듈)끼리在一起了整理하면 원하는玩具を素早く能找到し、玩具が壊れる心配もなくせる。
2. SOLID 원칙은 장난감을 정리하는 규칙이에요. 각 상자에一つの 종류만 넣고(SRP), 같은 종류의 장난감은新しい种类が追加されても既存の规则は変わらない(OCP).
3. 만약 장난감이全部 섞여 있으면(severe coupling), 레고 캐릭터의 팔을 찾으려고 전체 상자를 뒤져야 하는데, architecture 원칙을 따르면 캐릭터 팔 상자에서 바로找到할 수 있어 Efficiency가 좋아진다.
