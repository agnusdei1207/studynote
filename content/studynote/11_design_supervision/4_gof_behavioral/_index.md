> **핵심 인사이트**
> 1. **본질**: GoF 행위 패턴(Behavioral Pattern)은 객체 간 책임 분배와 알고리즘 할당을 정의하여, 시스템의 행위(Behavior)를 구성 요소 간 어떻게 분배하느냐에 초점을 맞춘다.
> 2. **가치**: Observer 패턴은 event-driven 시스템의 핵심 토대이며, State 패턴은 복잡한 상태 전이 로직을 명확히 하고, Strategy 패턴은 알고리즘 교체 가능성을 열어 성능 튜닝의 유연성을 제공한다.
> 3. **융합**: 행위 패턴은 modern event-driven architecture, reactive programming, distributed system의事件處理 로직設計에서根本적인 역할을 한다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 행위 패턴의 정의와 범위

행위 패턴(Behavioral Pattern)은 GoF 23 패턴 중에서도 특히 객체 간 상호 작용과 책임 분배에 관심을 두는 패턴군이다. 생성 패턴이 "무엇을 생성할 것인가"에, 구조 패턴이 "어떻게 구성할 것인가"에 집중한다면, 행위 패턴은 "어떻게 협력할 것인가"와 "어떤 객체가 어떤 행위를 담당할 것인가"라는 질문에 답한다. 이 패턴들은 알고리즘의 할당, 상태 전이의 관리,事件通知의 전달 방식 등을建模하여 시스템의 行事的側面을構造적으로 관리할 수 있게 한다.

### 왜 행위 패턴이 중요한가

절차적 프로그램에서는 함수가 알고리즘을 포함하고, 이 함수를 호출하면 원하는 동작이 수행된다. 그러나 시스템이 복잡해지면 하나의 함수에서 모든 알고리즘과 상태를 관리하게 되어, 변경과 테스트가 극도로 어려워진다. 행위 패턴은 이러한 "과도한 책임"을 여러 객체로 분산시켜, 각 객체가 단일 책임만을 담당하게 함으로써 변경의 영향을局所化한다.

### 행위 패턴이 필요한 상황

행위 패턴이 빛을 발하는 대표적인 상황은 다음과 같다. 첫째, UI와 business logic의 분리(Observer 패턴)가 필요한 상황이다. 버튼 클릭이 여러 UI 요소의 동시 업데이트를 유발해야 하는데, 버튼이 모든 대상을 직접 알면 강한 결합이 발생한다. 둘째, 객체의 상태에 따라 행동이 급격히 변하는 상황(State 패턴)이다. 주문 시스템에서 "주문접수 → 결제완료 → 배송중 → 배송완료" 각 상태에서 허용되는 행동이 다를 때, 상태 전이 로직을 if-else로 구현하면 변경이 어렵다. 셋째, 여러 알고리즘 중 런타임에 하나를 선택해야 하는 상황(Strategy 패턴)이다. 결제 수단별 할인이 각각 다른 알고리즘으로 계산되어야 할 때, 이를 직접 구현하면 결제 수단 추가마다 로직을 수정해야 한다.

Observer 패턴의 필요성을 시각화하면, event sender와多个 receiver 간의 직접 연결이 어떻게 복잡한 결합을 유발하는지, 그리고 Observer 패턴이 이를 어떻게 해소하는지 명확히 드러난다.

```text
┌─────────────────────────────────────────────────────────────────┐
│     직접 연결 (Without Observer) vs Observer 패턴 적용 비교          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [Without Observer: 직접 연결]                                   │
│                                                                 │

│          │              → OCP 위반!                         │       │
│          ▼                                               ▼       │
│      ┌───────┐     ┌───────┐     ┌───────┐     ┌───────┐      │

│                                                                 │

│                                                                 │

│         │Subject│───────────────────▶│ <<interface>>   │       │

```

**[다이어그램 해설]** 직접 연결 방식에서 새로운 UI 요소(UI D)를 추가하려면 Subject 클래스에 직접 UI D에 대한 참조와通知 로직을 추가해야 한다. 이는 Subject가 자신의粒도 이상의 역할을 담당하게 하여 SOLID의 SRP(단일 책임 원칙)를 위반하고, 새로운 요소 추가마다 Subject를 수정해야 하므로 OCP(개방-폐쇄 원칙)도 위반한다. Observer 패턴에서는 Subject가 Observer 인터페이스만 알면 되고, 새로운 Observer는 Subject에 attach()만 하면 자동으로通知を受け取る。 Subject는 자신의 핵심责任(상태 관리)에 집중할 수 있다. 실무에서 이는 SNS의포스팅投稿 → 팔로워全員通知, 주식 가격 변동 → 등록된投資자全员通知 등에 직접 적용된다.

### 비유

행위 패턴은 마치管弦楽団の指帥と各楽器奏者の関係に似ている。 指挥者(Subject)가全体の音を统一的に管理하면서도、各楽器(Observers)은指挥者に 개별적으로 연결되어 있다. 楽器が追加されるたびに乐团を再構成するのではなく、新しい奏者が指挥者看一眼就能理解知道自己该何时演奏一样、 Observer 패턴では Subject가 특정 상태 변경을認知하면 등록된 모든 Observer에게 자동으로 전달된다.

📢 섹션 요약 비유: 행위 패턴은 마치宴会에서司会者가 Guests(Observers)에게お知らせを一斉送信する仕組みと同じで、招待客名单 Guests list)만管理하면 Individual每位客人에게個別に連絡を取る필요가 없다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 핵심 행위 패턴 상세

#### 1. Observer 패턴

Observer 패턴은 한 객체(Subject)의 상태가 변경되었을 때, 해당 객체에 등록된 여러 객체(Observer)들에게 자동으로通知하고, 자동更新할 수 있도록 하는 패턴이다. 이 패턴의 핵심 가치는 "발행-구독(Pub-Sub)" 구조를 통해 결합도를最低化하는 것이다.

Observer 패턴의 구조와 상태 전이 과정을 시각화하면, Subject와 Observer 간의 느슨한 결합 관계와通知传播机制가 명확히 드러난다.

```text
┌─────────────────────────────────────────────────────────────────┐
│                    Observer 패턴 구조 및 동작                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [Subject 클래스]               [Observer 인터페이스]             │
│  ┌─────────────────────┐      ┌─────────────────────┐         │
│  │ «abstract»          │      │ <<interface>>        │         │
│  │ Subject            │──────▶│ Observer            │         │
│  ├─────────────────────┤      ├─────────────────────┤         │
│  │ +attach(Observer)  │      │ +update(subject)   │         │
│  │ +detach(Observer)  │      └──────────┬──────────┘         │
│  │ +notify()          │                 │ △                    │
│  │ #state             │                 │ │                    │
│  └──────────┬──────────┘                 │ │                    │
│             │                            │ │                    │
│  ┌──────────┴──────────┐         ┌───────┴───────┐            │
│  │ ConcreteSubject     │         │               │            │
│  ├─────────────────────┤         │┌─────────┐ ┌──┴───────┐   │
│  │ +getState()        │         ││ConcreteA│ │ConcreteB │   │
│  │ +setState(newState)│─────────▶││Observer │ │Observer  │   │
│  │   { notify(); }    │         │└─────────┘ └──────────┘   │
│  └─────────────────────┘                                         │
│                                                                 │
│  동작 흐름:                                                      │
│  ─────────                                                      │
│  1. ConcreteSubject.attach(concreteA)  // Observer 등록         │
│  2. ConcreteSubject.attach(concreteB)                           │
│  3. ConcreteSubject.setState(newValue)  // 상태 변경             │
│  4. ConcreteSubject.notify()  // 등록된 Observer에게一斉通知    │
│     │                                                         │
│     │  concreteA.update(this)  // Pull 방식: subject에서 직접 가져옴  │
│     │  concreteB.update(this)                               │
│                                                                 │
│  변형: Pull vs Push                                               │
│  ─────────                                                      │
│  Pull: Observer가 update() 호출 후 subject에서 필요 데이터 가져옴     │
│  Push: Subject가 update() 시 필요한 데이터를 인자로 전달           │
│                                                                 │
│  핵심 이점:                                                      │
│  · Subject 확장 시 기존 Observer 수정 불필요 (OCP)                │
│  · Observer는 Subject의 구현细节를 알 필요 없음 (느슨한 결합)     │
│  · 런타임에 Observer 등록/해제 가능 (动态性)                     │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Subject는 Observer 목록을 유지하고, 상태 변경 시 notify() 메서드를 호출하여 모든 Observer의 update()를一斉実行する。 핵심은 Subject와 Observer가 서로에 대해 추상(인터페이스)만 알 뿐, 구체적 구현을 모른다는 점이다. 따라서 ConcreteSubject가 변경되어도 ConcreteObserver에는 영향이 없고, 새로운 Observer 추가도 기존 Subject를 수정하지 않고 가능하다. Pull 방식에서는 Observer가 update() 호출 후 subject.getState()로 필요한 데이터를 능동적으로 가져오고, Push 방식에서는 Subject가 변경 데이터 자체를 update() 인자로 전달한다. Pull 방식이 더 느슨한 결합을 유지하지만, Push 방식은 한 번의 통신으로 데이터 동기화가 완료되는 장점이 있다. 실무에서는 MVC(Model-View-Controller)에서 Model(Subject)과 View(Observer) 간의自動更新, event-driven 시스템, SNS의팔로우通知 시스템 등에广泛应用된다.

#### 2. State 패턴

State 패턴은 객체의 내부 상태에 따라その объек트의 행동이 자동으로 변경되는 패턴이다. if-else나 switch-case로 상태를 처리하면 상태 전이 로직이 하나의 클래스에 집중되어 복잡해지고, 새로운 상태 추가 시 기존 코드를大幅 수정해야 한다. State 패턴은 각 상태를独立된クラス(ConcreteState)로建模하고, 상태 전이 자체도 각 State 클래스에 위임함으로써この問題を解決する。

State 패턴의 구조와 주문 시스템 적용 예를 시각화하면, 상태 전이 로직이 어떻게 각 State 클래스로 분산되는지 명확히 드러난다.

```text
┌─────────────────────────────────────────────────────────────────┐
│              State 패턴: 주문 시스템 적용 예                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [State 인터페이스]                                              │
│  ┌─────────────────────┐                                        │
│  │ <<interface>>       │                                        │
│  │ State              │                                        │
│  ├─────────────────────┤                                        │
│  │ +handle(Context)   │                                        │

│             △                                                    │
│             │                                                    │

│ ┌──────┐ ┌──────┐ ┌──────┐       ┌──────┐                    │
│ │Order  │ │Payment│ │Ship-  │       │Deliv- │                    │
│ │Received│ │Completed│ │pingState│       │ered  │                    │

│  ┌─────────────────────┐                                        │
│  │ Order               │                                        │

│  │ -state: State      │◀──────────────────────────────┐        │
│  │ +setState(state)   │                               │        │
│  │ +request() {       │                               │        │
│  │    state.handle() │                               │        │
│  │  }                 │                               │        │
│  └─────────────────────┘                               │        │
│                                                         │        │
│  상태 전이 흐름:                                            │        │
│  ──────────                                              │        │
│  OrderReceived ──(결제 완료)──▶ PaymentCompleted           │        │
│      │                                    │                 │        │
│      │                              (배송 시작)            │        │
│      ▼                                    ▼                 │        │
│  [handle()에서                              ShippingState    │        │
│   다음 상태로 전이] ──────────────────────────────────▶ Delivered │
│                                                                 │
│  if-else 방식 vs State 패턴 비교:                                │
│  ─────────────────────────                                      │
│  // 안티패턴: if-else                                           │
│  if (state == "RECEIVED") { ... }                              │
│  else if (state == "PAID") { ... }                            │
│  else if (state == "SHIPPING") { ... }  // 새로운 상태마다 추가  │
│                                                                 │
│  // 올바른 설계: State 패턴                                       │
│  // 새 상태 추가 시: 새 State 클래스 추가만으로 충분              │
│  // 기존 상태 처리 코드 수정 불필요 (OCP)                          │
│                                                                 │
│  핵심 이점:                                                      │
│  · 각 상태의 행위와 전이 로직이 해당 상태 클래스에 캡슐화         │
│  · 새로운 상태 추가 시 기존 코드 수정 불필요 (OCP)                │
│  · 상태 전이 테스트를 각 State 클래스별로 독립적으로 수행 가능    │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Context(Order) 클래스는 현재 상태 객체(State)에 행위 처리를 위임한다. 각 ConcreteState(OrderReceived, PaymentCompleted, ShippingState, Delivered)는 자신의 handle() 메서드에서 적절한 행위를 수행하고, 다음 상태로의 전이도 직접 결정한다. 따라서 OrderReceived 상태에서는 결제 관련 행위만 처리하고, ShippingState에서는 배송 관련 행위만 처리한다. 새로운 상태(예: 반품 상태 Returned)가 추가되어도 기존 상태 클래스들은 변경되지 않으며, Returned 클래스만 추가하면 된다. 이는 OCP의完美한 실현이며, if-else로 상태를 처리하는 안티패턴에서 발생하는 "함께 변경해야 하는 이유"를分離한다. 실무에서는 주문/결제/배송 workflow, 게임 캐릭터 상태 관리(Idle, Walking, Running, Jumping 등), 워크프루/workflows engine実装 등에广泛应用된다.

#### 3. Strategy 패턴

Strategy 패턴은 알고리즘군을 정의하고, 각각을 캡슐화하여 교환 가능하게 만드는 패턴이다. 상위 섹션의 2_architecture_principles에서 이미 상세히 다루었으므로, 여기서는 행위 패턴 관점에서의 핵심 메커니즘만 요약한다. Strategy 패턴에서 각 알고리즘(Strategy)은 교환 가능한 객체로建模되고, Context 객체는 Strategy 인터페이스만 알면 된다.

#### 4. Command 패턴

Command 패턴은 요청을 객체로 캡슐화하여, 요청의 파라미터화, 대기열 등록, 실행 취소(Undo) 기능을 가능하게 하는 패턴이다. 요청을 객체로 변환함으로써, 요청을发送した時点と실행한時点を分離できる。

Command 패턴의 구조와 Undo/Redo 적용 예를 시각화하면, 요청의 캡슐화가 어떻게 실행 취소 기능을可能하게 하는지 명확히 드러난다.

```text
┌─────────────────────────────────────────────────────────────────┐
│                 Command 패턴: Undo/Redo 적용 예                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [Command 인터페이스]                                             │
│  ┌─────────────────────┐                                        │
│  │ <<interface>>       │                                        │
│  │ Command            │                                        │
│  ├─────────────────────┤                                        │
│  │ +execute()         │ ← 요청 실행                             │
│  │ +undo()            │ ← 실행 취소                            │
│  └──────────┬──────────┘                                        │
│             △                                                    │
│             │                                                    │
│  ┌──────────┼──────────┬─────────────┐                           │
│  │          │          │             │                           │
│  ▼          ▼          ▼             ▼                           │
│ ┌──────┐ ┌──────┐ ┌──────┐      ┌──────┐                      │
│ │CopyCmd│ │PasteCmd│ │CutCmd│      │UndoMgr│                      │
│ └──────┘ └──────┘ └──────┘      └──────┘                      │
│                                               │                  │
│  동작 흐름:                                      │                  │
│  ─────────                                      │                  │
│  1. 사용자 "복사" 클릭                                              │
│  2. CopyCommand.execute() → 클립보드에 内容 저장                   │
│  3. UndoManager에 Command 삽입                                      │
│                                               │                  │
│  4. 사용자 "실행 취소" 클릭                                         │
│  5. UndoManager.pop() → PasteCommand.undo() 실행                  │
│  6. 이전 상태 복원                                                 │
│                                                                 │
│  [Invoker]                      [Receiver]                        │
│  ┌───────────┐    execute()    ┌───────────┐                     │
│  │  Button   │───────────────▶│  Editor   │                     │
│  │ (요청자) │                │  (수신자) │                     │
│  └───────────┘                └───────────┘                     │
│       │                                                        │
│       │ Button 클릭 → Command.execute() 호출                     │
│       │ = 요청과 실행의 분리                                      │
│                                                                 │
│  핵심 이점:                                                      │
│  · 요청을 객체로 캡슐화 → 지연 실행, 대기열 등록 가능              │
│  · execute/undo 쌍으로 동작 취소/재실행 기능용이                  │
│  · 새 명령 추가 시 기존 Command/Invoker 수정 불필요 (OCP)        │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Command 패턴의 핵심은 "요청을 객체로 변환"하는 것이다. Button(Invoker)은 어떤具体的な 명령이 실행될지 모르며, 단지 Command.execute()만 호출한다. CopyCommand, PasteCommand, CutCommand 등은 execute()에서 구체적 동작을 정의하고, undo()에서 그 동작을 취소하는 방법을 정의한다. UndoManager는 실행된 Command 객체를 스택에保存하고, undo() 요청 시 pop()하여对应 Command의 undo()를実行する。 이를 통해 복수의 동작을 연속으로 취소하거나, 다시 실행(Redo)하는 것이 가능해진다. 실무에서는テキスト편집기의 실행 취소/다시 실행, 매크로 녹화/재생, 数据库 트랜잭션 관리(ROLLBACK) 등에广泛应用된다.

#### 5. Template Method 패턴

Template Method 패턴은 알고리즘의骨架를 상위 클래스에서 정의하고, 일부 단계를 하위 클래스에서 재정의하도록 하는 패턴이다. 알고리즘의構造는 고정하고, 일부 구체적 구현만 하위 클래스에서 채우는 방식으로, "流程을 정의하고 세부 구현은委嘱"한다.

#### 6. Iterator 패턴

Iterator 패턴은 collection의 내부 구조를 알 필요 없이, 요소에 순차적으로 접근할 수 있는 방법을 제공하는 패턴이다. 배열, 리스트, 트리, 해시맵 등 다양한 자료구조에 대해 동일한 인터페이스로遍历 가능한 추상화를 제공한다.

### 행위 패턴 간 관계 맵

행위 패턴들은 각각 다른 행위적 측면을建模하지만,相互에协同 동작하기도 한다. Observer 패턴으로	event-driven 행위를 처리하고, State 패턴으로 상태 전이를 관리하며, Strategy 패턴으로 알고리즘을 교체하고, Command 패턴으로 요청을 캡슐화하는 등의 조합이 가능하다.

📢 섹션 요약 비유: 행위 패턴은 마치木 marionetteの各糸の引き方を定義する marionette choreographyのように、各オブジェクトがどのような役割を持ち、他のオブジェクトと如何にして通信するかを体系的に管理する方法论を 제공한다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### Observer vs Mediator vs Event-Driven Architecture 비교

| 비교 항목 | Observer | Mediator | Event-Driven |
|:---|:---|:---|:---|
| **핵심 역할** | 1:N 자동通知 | N:N 중재 조정 | 비동기 event 처리 |
| **결합도** | Subject ↔ Observer (느슨) | 모든 객체 ↔ Mediator (중간) | Publisher ↔ Subscriber (매우 느슨) |
| **동기/비동기** | 주로 동기 | 주로 동기 | 비동기 위주 |
| **적합 상황** | 실시간 상태 동기화 | 복잡한 객체 상호작용 | 확장성 중요한 분산 시스템 |
| **한계** | Observer 많을 때通知洪水 | Mediator bottleneck 가능 | event 순서/처리 보장 어려움 |

세 패턴 모두 객체 간通信을 관리하지만, 관점의 차이가 있다. Observer는 "자동通知"에, Mediator는 "중재 조정"에, Event-Driven은 "비同步通信"에重点을 둔다.

Observer 패턴과 Event-Driven Architecture의 구조적 차이를 시각화하면, Publisher-Subscriber 모델이如何して 더 느슨한 결합을実現하는지 명확히 드러난다.

```text
┌─────────────────────────────────────────────────────────────────┐
│    Observer 패턴 vs Event-Driven Architecture 비교                 │
├─────────────────────────────────────────────────────────────────�─┤
│                                                                   │
│  [Observer 패턴 (동기, 직접 호출)]                                  │
│                                                                   │
│      ┌─────────┐                                                    │
│      │ Subject │                                                    │
│      │ notify()│                                                    │
│      └────┬────┘                                                    │
│           │ 직접 호출 (동기)                                        │
│     ┌─────┴─────┐                                                  │
│     ▼           ▼                                                 │
│  ┌──────┐  ┌──────┐                                              │
│  │ Obs A│  │ Obs B│  ← Subject가 직접 update() 호출               │
│  └──────┘  └──────┘  ← Observer 목록 관리 의무: Subject에게       │
│                                                                   │
│  [Event-Driven Architecture (비동기, Message Broker 경유)]           │
│                                                                   │
│   ┌─────────┐         ┌──────────────┐        ┌─────────────┐   │
│   │Publisher│────────▶│Message Broker│◀───────│Subscriber A │   │
│   │(Event   │ event   │ (Kafka,     │ fetch  └─────────────┘   │
│   │Source)  │ posting │  RabbitMQ)  │        ┌─────────────┐   │
│   └─────────┘         └──────────────┘◀──────│Subscriber B │   │
│                                    │         └─────────────┘   │
│                                 async                           │
│                              통신                                     │
│                                                                   │
│  핵심 차이:                                                        │
│  ────────                                                        │
│  · Observer: Subject가 Observer 목록 관리 (결합 있음)               │
│  · EDA: Publisher와 Subscriber가 서로 모름 (매우 느슨한 결합)        │
│  · EDA: Message Broker가事件을 保存/분배 (확장성↑, 내구성↑)         │
│  · EDA: 비동기 처리로 시스템 반응성 향상                           │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Observer 패턴에서 Subject는 직접 Observer 목록을 관리하고,notify() 호출 시 모든 Observer의 update()를 직접 동기 호출한다. 이는 Subject가 Observer의存在를認知한다는 점에서 결합이 존재한다. Event-Driven Architecture에서는 Publisher가 event를 Message Broker에投稿し、Subscriber가 Message Broker에서 event를 가져온다(pull)またはMessage Broker가 subscriber에게 event를 push한다. Publisher와 Subscriber는 서로에 대해 알지 못하며, 오직 Message Broker의 topic/event format만 알면 된다. 이 구조는 Subscriber의数量的 변화(Scale Up/Down)가 Publisher에게 영향하지 않는다는 Advantages이 있다. 그러나 event 순서 보장과 exactly-once 처리, dead letter queue 관리 등 추가적인 복잡성이 발생한다. 실무에서는 대규모 분산 시스템(전자상거래 주문 처리, 실시간 분석 파이프라인 등)에서 EDA가 선호되고, 내부 모듈 간의緊密한協調이 필요한 상황에서는 Observer 패턴이 적합하다.

### 과목 융합 관점

- **운영체제**: Observer 패턴은 이벤트 핸들링(Interrupt Handler →注册的 Handler 호출)과 동일한 구조
- **네트워크**: EDA는 마이크로서비스 간 비동기通信의 표준 아키텍처
- **데이터베이스**: Trigger(사건) → stored procedure(처리)의 연쇄 동작은 행위 패턴의一種

📢 섹션 요약 비유: Observer 패턴은宴会에서司会者가 개별 참석자에게 직접 메모를 전달하는 것이고, EDA는宴会 안내소(메시지 브로커)에 메모를 두고, 참석자가 알아서 찾아가는 것에 비유할 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 시나리오 1: 주가 변동 실시간 알림 시스템 구축

**문제 상황**: 여러 금융 상품(주가, 환율, 암호화폐)에 대해 등록된 사용자들에게 실시간으로 가격 변동을通知해야 한다. 특정 가격 도달 시 알림, 이동평균선 교차 알림 등 다양한 조건이 존재한다.

**기술사적 판단**: Observer 패턴을 기반으로, 각 금융 상품이 Subject가 되고, 사용자가 Observer로 등록되는 구조를設計한다. 각 조건(가격 도달, 이동평균선 교차 등)은 Strategy 패턴으로 캡슐화하여 condition check 알고리즘을 런타임에 교체 가능하게 한다. 대량 사용자에게의通知는Message Queue(Kafka)를 통해 비동기로 전송하여 시스템 부하를分散한다. 이를 통해 Subject(금융 상품)의 핵심 기능(가격 업데이트)과Observer 관리/通知 로직을 분리할 수 있다.

### 시나리오 2: 워크플로우 엔진 상태 관리

**문제 상황**: 문서 승인 시스템에서 "검토중 → 수정요청 → 재검토 → 승인 → 배포" 등 다양한 상태 전이와 각 상태에서 가능한 행위가 존재한다. 새로운 상태 추가 시 기존 코드를 최대한変更하지 않아야 한다.

**기술사적 판단**: State 패턴을 적용하여 각 상태(검토중, 수정요청, 재검토, 승인, 배포)를 ConcreteState 클래스로建模한다. 각 State 클래스에서 가능한 행위(handle())와 다음 상태 전이(nextState())를 정의하고, Context(문서)가 현재 State 객체에 모든 처리를 위임한다. 새로운 상태(예: 보류 상태)가 추가되어도 기존 State 클래스들은変更不要하며, 보류 상태 클래스만 추가하면 된다. 이는 SOLID의 OCP를完美히 실현하는 설계 판단이다.

### 도입 체크리스트

**기술적 항목**:
- Observer 패턴 사용 시 notify() 호출 빈도가 시스템 성능에 영향을 미치는가? (通知洪水 방지 필요)
- State 패턴 적용 시 상태 전이 조건이 명확히 정의되어 있는가?
- Command 패턴 사용 시 undo/redo 범위가 명확히界定되어 있는가?
- 각 패턴 적용이 시스템 복잡성을 늘리지 않는가?

**운영·보안적 항목**:
- Observer 등록/해제가 동적으로 이루어질 때, 메모리 누수 가능성은 없는가?
- 분산 시스템에서 Event-Driven 패턴 적용 시 event 순서 보장이 중요한가?
- Command 패턴의 execute()가 idempotent(멱등)한가? (재실행 가능성 고려)

### 안티패턴

**안티패턴: 과도한 Observer 등록** 하나의 Subject에 너무 많은 Observer가 등록되면, 상태 변경 시通知洪水가 발생하여 성능 저하를 유발한다. 해결책으로通知 batching(일정 시간 또는 일정 수량 모아서通知) 또는 Dead Observer 패턴(Observer를 해제 가능한 구조)을 적용해야 한다.

**안티패턴: 상태 전이 중첩** State 패턴에서 상태 전이 핸들러 내에서 또 다른 상태 전이를 유발하면, 상태 전이 스택이 깊어져 추적이 어려워지고 예기치 않은 동작이 발생할 수 있다. 해결책으로 상태 전이는 Context 레벨에서 한 번만 처리하거나, 비동기 큐를 통해 처리한다.

📢 섹션 요약 비유: 행위 패턴의 남용은 마치宴会에서司会자가 Attendee listを細部和らえる同時に宴会の進行까지管理하려고 몸两份을 쓰려는 것과 같이、 결국은司会者가 과부하 되어崩溃하거나,Attendee에게 잘못된 안내를 줄 수 있다.

### 미래 전망

## 📌 관련 개념 맵 (Knowledge Graph)

- **Observer**: 1:N 자동通知 패턴으로, event-driven 시스템의基礎
- **State**: 상태 전이 로직을 상태 클래스로 분산하여 관리하는 패턴
- **Strategy**: 알고리즘군을 캡슐화하여 교체 가능하게 하는 패턴
- **Command**: 요청을 객체로 캡슐화하여 실행 취소/대기열 기능을 가능하게 하는 패턴
- **Template Method**: 알고리즘骨架를 상위 클래스에 정의하고 세부 구현을 하위 클래스에 위임하는 패턴
- **Iterator**: collection 내부 구조를 알 필요 없이 요소에 순차 접근하는 추상화
- **Event-Driven Architecture**: 비同步/event 기반 통신으로 결합도를極限まで 낮춘 아키텍처 스타일

## 👶 어린이를 위한 3줄 비유 설명

1. Observer 패턴은 "유튜브 채널 구독"이라고 생각하면 돼요. 채널主(Subject)가 새 영상을 올리면, 구독자全员(Observers)에게 자동으로알림이 간다. 새 구독자가 늘어도 채널主는 그냥 영상만 올리면 돼요.
2. State 패턴은 게임 캐릭터의 상태와 같아요. " idle" 상태에서는 가만히 서 있고, "점프" 상태에서는 하늘로 올라가고, "걷기" 상태에서는 발을 굴러. 상태가 바뀌면 자동으로 그 상태에 맞는 행동이 실행된다.
3. Command 패턴은 "실행 취소(Ctrl+Z)"와 같아요. 어떤 행동(복사, 자르기 등)을 "실행"하면 그 행동이カード에 적혀서 Stack에 쌓이고, 실행 취소를 하면Cards를 꺼내서 이전 상태로 되돌린다.
