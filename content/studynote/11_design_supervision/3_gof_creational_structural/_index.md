## Ⅰ. 개요 및 필요성 (Context & Necessity)

### GoF 디자인 패턴 개요

GoF(Gang of Four) 패턴은 1994년 에릭 감마, 리처드 헬름, 랄프 존슨, 존 블리시데스(Gamma, Helm, Johnson, Vlissides)가 저술한 「Design Patterns: Elements of Reusable Object-Oriented Software」에서 체계화된 23가지 设计模式이다. 이 패턴들은 객체지향 설계에서 반복적으로 발생하는 문제에 대한 검증된 해결 방법론으로, 구체적 상황에서의 "무엇을"보다는 "왜 그렇게 설계하는가"의 원칙에重点을 둔다.

### 분류 체계

GoF 23패턴은 목적(Purpose)과 범위(Scope)에 따라 세 가지로 분류된다. 목적에 따라서는 생성(Creational), 구조(Structural), 행위(Behavioral)로 나뉘며, 범위에 따라서는 클래스(클래스 간 상속을 통한 정적 관계)와 객체(객체 합성을 통한 동적 관계)로 구분된다.

GoF 패턴의全体系를 시각화하면, 생성-구조-행위 패턴이 클래스와 객체 범위로 어떻게 분포되어 있는지 확인 할 수 있다.

```text
┌─────────────────────────────────────────────────────────────────┐
│                    GoF 23 패턴 분류 체계                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  목적별 (Purpose)                범위별 (Scope)                  │
│  ─────────────                ──────────────                   │
│                                                                 │
│  ┌─────────────────────────┐    ┌─────────────┬─────────────┐   │
│  │     생성 (Creational)    │    │   클래스    │    객체     │   │
│  │  객체 생성 메커니즘 제어    │    │  (정적)    │  (동적)    │   │
│  ├─────────────────────────┤    ├─────────────┼─────────────┤   │
│  │ ·Factory Method         │    │     O      │     O      │   │
│  │ ·Abstract Factory       │    │            │     O      │   │
│  │ ·Builder               │    │            │     O      │   │
│  │ ·Prototype            │    │            │     O      │   │
│  │ ·Singleton            │    │     O      │            │   │
│  ├─────────────────────────┤    ├─────────────┼─────────────┤   │
│  │    구조 (Structural)     │    │            │            │   │
│  │  클래스/객체 구성 결합    │    │            │            │   │
│  ├─────────────────────────┤    ├─────────────┼─────────────┤   │
│  │ ·Adapter               │    │     O      │     O      │   │
│  │ ·Bridge               │    │     O      │     O      │   │
│  │ ·Composite            │    │            │     O      │   │
│  │ ·Decorator           │    │            │     O      │   │
│  │ ·Facade              │    │            │     O      │   │
│  │ ·Flyweight           │    │            │     O      │   │
│  │ ·Proxy               │    │            │     O      │   │
│  ├─────────────────────────┤    ├─────────────┼─────────────┤   │
│  │    행위 (Behavioral)     │    │            │            │   │
│  │  객체 간 책임 분배/알고리즘│    │            │            │   │
│  │  (다음 섹션에서 상세)    │    │            │            │   │
│  └─────────────────────────┘    └─────────────┴─────────────┘   │
│                                                                 │
│  ※ O는 해당 범위에 속함을 표시                                    │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** GoF 패턴全体系에서 생성 패턴은 객체 생성의 책임을 "누가"負うかに焦点を当てる。 Factory Method는 하위 클래스에 생성 책임委譲하고, Abstract Factory는 관련 객체 군(제품군 전체)의 생성责任を抽象化する。 구조 패턴은已有의 클래스나 객체를 어떻게 조합하여より大きな構造を作るかに焦点を当てる。 Adapter는兼容性없는 接口를 변환하고, Composite는 部分-전체 관계를建模하며, Decorator는 객체에 동적으로 responsibilities를 추가한다. 각 패턴은특정 문제 상황에서繰り返し 발생하는 설계 문제에 대한 검증된 해결책이다.

### 왜 패턴이 필요한가

패턴을 모르고 설계하면 각 개발자가 동일한 문제에 대해 다른 해결책을 시도하고, 그 결과로 일관성 없는 코드 구조가 탄생한다. 예를 들어, 동일 시스템 내에서 "객체 생성 방식"이 여기서는 new 연산자, 저기서는 static factory method, 또다른 곳에서는 singleton 직접 구현처럼 제각각이면 코드를 이해하고 유지보수하기 어려워진다. 패턴은 이러한 "설계의 관례"를 표준화하여 개발자 간 Communication 비용을 줄이고, 검증된 구조를 재사용하여 버그 가능성을 낮춘다.

### 비유

GoF 패턴은 마치 요리 레시피와 같다. "如何して美味しい 찌개を作るか"には各料理장의 pribadi 경험과手艺이 있지만, 가장 맛있는 찌개には공통적인 재료放入順序と火候管理が存在하듯이, 소프트웨어設計에도 "반복적으로 좋은 결과를 만드는構造적 관례"가 존재한다. 이 관례를 레시피처럼 정리해둔 것이 GoF 패턴集이다.

📢 섹션 요약 비유: GoF 패턴을 모르면 요리할 때마다 receitas를 처음부터 찾아봐야 하지만, 패턴을 알면"Just do it"처럼 익은 레시피를 떠올리며，迅速かつ一貫した料理を作れる".

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 생성 패턴 (Creational Patterns)

#### 1. Factory Method 패턴

Factory Method 패턴은 객체를 생성하는 인터페이스를 정의하되, 하위 클래스(Subclass)에서 어떤 클래스의 인스턴스를 생성할지를 결정하도록 하는 패턴이다. 이를 통해 "생성하는 클래스"와 "생성되는 클래스" 간의 결합도를 낮추고, 새로운 제품 유형 추가 시 기존 코드를変更不要にする。

Factory Method 패턴의 구조를 시각화하면, Creator 클래스가 product 생성 책임을 subclass에 위임하는 구조가 명확히 드러난다.

```text
┌─────────────────────────────────────────────────────────────────┐
│              Factory Method 패턴 구조                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [Creator (생성자)]                 [Product (제품)]              │
│  ┌─────────────────────┐           ┌─────────────────────┐       │
│  │ «abstract»         │           │ «abstract»           │       │
│  │ Creator            │           │ Product             │       │
│  │ + factoryMethod()  │──────────▶│ + operation()       │       │
│  │   {abstract}       │           └─────────────────────┘       │
│  │ + anOperation()    │                    △                    │
│  └─────────────────────┘                    │                    │
│          △                                   │                    │
│          │                                   │                    │
│  ┌───────┴───────┐                  ┌───────┴───────┐            │
│  │               │                  │               │            │
│  ┌─────────┐  ┌─────────┐    ┌─────────────┐  ┌─────────────┐  │
│  │Concreate│  │Concreate│    │ ConcreteProd│  │ ConcreteProd│  │
│  │  or A   │  │  or B   │    │    A       │  │    B       │  │
│  ├─────────┤  ├─────────┤    ├─────────────┤  ├─────────────┤  │
│  │factoryM │  │factoryM │    │ operation()│  │ operation()│  │
│  │:Product │  │:Product │    │ (구현)     │  │ (구현)     │  │
│  └─────────┘  └─────────┘    └─────────────┘  └─────────────┘  │
│                                                                 │
│  동작 흐름:                                                      │
│  client ──▶ Creator.anOperation()                                │
│               │                                                 │
│               │ factoryMethod() 호출 (하위 클래스에서 구현)        │
│               ▼                                                 │
│            Product 인스턴스 반환                                  │
│               │                                                 │
│               ▼                                                 │
│            product.operation() 수행                              │
│                                                                 │
│  핵심 이점:                                                      │
│  · 새 제품 추가 시: Product + ConcreteProd 추가 → Creator 변경 불필요 │
│  · 기존 코드 수정 없이 새 제품군 도입 가능 (OCP 준수)              │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Creator 클래스는工厂 메서드(factoryMethod)의 구현을 하위 클래스에게 위임한다. 상위 Creator는 어떤产品在作られるかには関心が 없고, 오직 Product 인터페이스만 알고 있다. 하위 클래스(ConcreateorA/B)가 어떤 구체적 Product를 생성할지决定한다. 따라서 새로운 제품(ConcreteProductC)을 추가할 때 Creator의 코드를変更하지 않아도 된다. 이는 SOLID의 OCP(개방-폐쇄 원칙)를 직접 실현하는 패턴이다. 실무에서는 Logger 생성(콘솔용, 파일용, 네트워크용), Database 연결 생성(MySQL용, PostgreSQL용)等)에广泛应用된다.

#### 2. Abstract Factory 패턴

Abstract Factory 패턴은 서로 관련이 있는 여러 객체群을 생성するための抽象적工廠을 제공하는 패턴이다. Factory Method가 단일 제품의 생성에 집중한다면, Abstract Factory는 "제품군(가족)" 전체의 생성을 추상화한다. 예를 들어, UI 프레임워크에서 "Windows 스타일 버튼/체크박스/입력상자" 또는 "Mac 스타일 버튼/체크박스/입력상자"를整套로 생성해야 할 때 이 패턴이 사용된다.

Abstract Factory 패턴의 구조를 시각화하면, Abstract Factory가 제품군 전체를 생성하는 接口를 제공하고, 구체적 Factory가 각_OS风格에 맞는整套 제품을 생성하는 것이 명확히 드러난다.

```text
┌─────────────────────────────────────────────────────────────────┐
│            Abstract Factory 패턴 구조                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [Abstract Factory]          [Abstract Products]                  │
│  ┌─────────────────────┐    ┌──────────┐ ┌──────────┐          │
│  │ «abstract»          │    │Button   │ │CheckBox │          │
│  │ GUIFactory          │───▶├──────────┤├──────────┤          │
│  ├─────────────────────┤    └──────────┘ └──────────┘          │
│  │ +createButton()     │           △           △                 │
│  │ +createCheckBox()   │           │           │                 │
│  └─────────────────────┘           │           │                 │
│          △                         │           │                 │
│          │                         │           │                 │
│  ┌───────┴───────┐                 │           │                 │
│  │               │          ┌──────┴──────┐ ┌──┴───────┐        │
│  ┌─────────┐  ┌─────────┐  │ WinButton   │ │WinCheckBx │        │
│  │WinFacory│  │MacFactor│  │ (구현)      │ │(구현)    │        │
│  ├─────────┤  ├─────────┤  └─────────────┘ └──────────┘        │
│  │createBtn│  │createBtn│                                        │
│  │→WinBtn  │  │→MacBtn  │                                        │
│  └─────────┘  └─────────┘                                        │
│                                                                 │
│  사용 예시:                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  GUIFactory factory = new WinFactory();                 │   │
│  │  Button btn = factory.createButton();                   │   │
│  │  CheckBox cb = factory.createCheckBox();                │   │
│  │  // Windows 스타일整套 UI 생성                            │   │
│  │                                                         │   │
│  │  // Mac 스타일 전환 시:                                  │   │
│  │  factory = new MacFactory();  // 다른 Factory로 교체     │   │
│  │  btn = factory.createButton(); // Mac 버튼 자동 생성     │   │
│  │  // Button, CheckBox 타입은 동일 → client 코드 변경 불필요 │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  핵심 이점:                                                      │
│  · 제품군 전체가 동시에 교환됨 → 일관된 UI 스타일 보장             │
│  · 새 제품 추가 시: Abstract Factory + 모든 구체 Factory 수정 필요  │
│    (단, 일반적으로 제품군 자체는 고정됨)                            │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Abstract Factory의 핵심 가치는 "제품군 전체의 일관성"이다. WinFactory는 WinButton과 WinCheckBox를生成하고, MacFactory는 MacButton과 MacCheckBox를生成한다. Client 코드는 구체적 Factory 구현체를 교체함으로써 전체 UI 스타일을 한 번에切换할 수 있다. 그러나 새로운 제품 유형(예: RadioButton)을 추가하려면 Abstract Factory 인터페이스와 모든 구체적 Factory 구현체를 수정해야 하므로, 제품군이 비교적 고정적인 환경에서 효과적이다. 실무에서는 테마 지원 UI, 다국어 리소스 관리, 数据库 연동 구성 등 "相关 객체 군"의 생성 관리를 위해 사용된다.

#### 3. Singleton 패턴

Singleton 패턴은 클래스의 인스턴스가 오직 하나만 존재하도록 보장하는 패턴이다. 데이터베이스 연결 풀, 설정 관리자, 로깅 시스템 등 "애플리케이션 전역에서 유일한 존재"가 필요한 경우에 사용된다.

#### 4. Builder 패턴

Builder 패턴은 복잡한 객체의 생성 과정과 표현 방법을 분리하여, 동일한 생성 공정에서 다른 표현 결과를作られる模式이다. 예를 들어, 게임 캐릭터 생성 시 "머리/몸/다리"를 선택하는 공정과人物模具 самиを 분리하여, 같은 공정으로 서로 다른 외형의 캐릭터를 생성할 수 있다.

### 구조 패턴 (Structural Patterns)

#### 1. Adapter 패턴

Adapter 패턴은 호환성이 없는 接口를 가진 클래스들이 함께動作할 수 있도록 변환(interfaces)하는 패턴이다. 전기 플러그 어댑터(220V → 110V 변환)와 같은 원리로, 기존 클래스의 接口를client가 원하는 接口로 변환한다.

Adapter 패턴의 동작 원리를 시각화하면, 기존 인터페이스가 client가 요구하는 인터페이스로 어떻게 변환되는지 명확히 드러난다.

```text
┌─────────────────────────────────────────────────────────────────┐
│                  Adapter 패턴 동작 원리                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [클라이언트 요구 接口]            [기존 클래스 (Adaptee)]          │
│  ┌─────────────────────┐    ┌─────────────────────┐            │
│  │ Target Interface    │    │ Adaptee             │            │
│  │ (client가 사용하려는 │    │ (이미 존재하는 클래스)│            │
│  │  接口)              │    │ ·existingMethod()   │            │
│  │ + request()        │    └──────────┬──────────┘            │
│  └──────────┬──────────┘                 │                     │
│              │                             │                     │
│              │         ┌──────────────────┘                     │
│              │         │ Adapater가 중개                          │
│              ▼         ▼                                          │
│       ┌─────────────────────┐                                    │
│       │      Adapter        │                                    │
│       ├─────────────────────┤                                    │
│       │ + request() {       │                                    │
│       │     adaptee.existing│                                    │
│       │     _method();       │                                    │
│       │   }                  │                                    │
│       └─────────────────────┘                                    │
│                                                                 │
│  실생활 비유:                                                     │
│  ─────────────────                                                │
│  한국 충전기 (220V) ──▶ [어댑터] ──▶ 미국 콘센트 (110V)            │
│  ─────────────────                                                │
│  Legacy 시스템 ──▶ [Adapter] ──▶ New Client Interface            │
│                                                                 │
│  핵심 이점:                                                      │
│  · 기존 코드 수정 없이 Legacy ↔ 신 시스템 연동 가능               │
│  · client 코드는 Target Interface만 알면 됨                      │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Adapter 패턴은 기존 Adaptee 클래스의 메서드를 호출하고, 그 결과를 client가 원하는 포맷으로 변환하여 반환한다. 핵심은 Adaptee를 직접 수정하지 않고도 interface 호환성을 확보한다는 점이다. 예를 들어, 기존의 LegacyPaymentGateway(kakaoPay() 메서드만 있는)를 새로운 PaymentProcessor(request() 메서드 요구) 인터페이스에 맞추려면 Adapter를 создать하여 kakaoPay()를 request()로 변환하면 된다. 실무에서는 레거시 시스템 연동, 써드파티 라이브러리 통합, 또는 接口版本 차이 관리에广泛应用된다.

#### 2. Decorator 패턴

Decorator 패턴은 객체에 동적으로 responsibilities를 추가할 수 있도록 하는 패턴이다. 상속(Inheritance)을 통한 기능 추가가 컴파일 시점에 고정되는 것과 달리, Decorator는 실행 시점에 기능 조합을 동적으로 변경할 수 있다. 커피에 우유, 시럽, 휘핑크림 등을 선택적으로 추가하는 커피 좌석 시스템과 동일한 원리다.

Decorator 패턴의 구조를 시각화하면, 기본 객체(Order.base)를 Decorator들이 순차적으로 감싸면서(dynamic wrapping) 기능이 추가되는 구조가 명확히 드러난다.

```text
┌─────────────────────────────────────────────────────────────────┐
│               Decorator 패턴 구조 및 동작                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Component 인터페이스:                                           │
│  ┌─────────────────────┐                                        │
│  │ «abstract»          │                                        │
│  │ Component          │                                        │
│  │ + operation()      │                                        │
│  └──────────┬──────────┘                                        │
│             △                                                   │
│             │                                                   │
│  ┌──────────┴──────────┐     ┌─────────────────────────────┐  │
│  │ ConcreteComponent   │     │ «abstract» Decorator        │  │
│  │ (기본 객체)          │◀────│ + component: Component     │  │
│  │ · operation()       │     │ + operation()              │  │
│  │   "기본 커피"        │     │   {子类 구현}               │  │
│  └─────────────────────┘     └──────────┬──────────────────┘  │
│                                         │ △                     │
│                     ┌───────────────────┼────────────────────┐  │
│                     │                   │                     │  │
│           ┌─────────┴────────┐ ┌────────┴────────┐ ┌────────┴─┐│
│           │ MilkDecorator    │ │ SyrupDecorator  │ │WhipDecor ││
│           │ (우유 추가)       │ │ (시럽 추가)      │ │ator      ││
│           ├──────────────────┤ ├─────────────────┤ ├──────────┤│
│           │ operation() {    │ │ operation() {   │ │operation ││
│           │   component.op()│ │  component.op() │ │(){compo  ││
│           │   + "우유"       │ │   + "시럽"      │ │nent.op() ││
│           │ }                │ │ }               │ │ + "휘핑" ││
│           └──────────────────┘ └─────────────────┘ └──────────┘│
│                                                                 │
│  동작 예시:                                                      │
│  ─────────                                                      │
│  Component coffee = new MilkDecorator(                           │
│                      new SyrupDecorator(                         │
│                          new ConcreteComponent()));              │
│  coffee.operation();                                            │
│  // 출력: "기본 커피 + 시럽 + 우유"                              │
│                                                                 │
│  핵심 이점:                                                      │
│  · 서브클래스 없이 기능 조합을 동적으로 변경                       │
│  · 각 Decorator가 단일 책임 → 단위 테스트 용이                   │
│  · 순서 변경으로 다양한 조합 가능                                 │
└─────────────────────────────────────────────────────────────────┘
```

**[다디어그램 해설]** Decorator 패턴의 핵심은 "감싸는(wrapping)" 동작이다. ConcreteComponent에 MilkDecorator가 감싸고, 그 위에 SyrupDecorator가 감싸는 형태로 객체가 형성된다. operation() 호출 시에는 가장 바깥쪽 Decorator의 operation()이 먼저 실행되고, 내부의 component.operation()을 호출한 후 자신의 추가 동작을 덧붙인다. 따라서 "기본 커피 + 시럽 + 우유"의 출력 순서는 Decorator 적용의 역순(우유 → 시럽 → 기본 커피)이 아닌, 감싸는 순서의 역순으로 구성된다. 실무에서는 로깅 추가, 캐싱, 트랜잭션 관리, 입력값 검증 등을 decorator로 구현하여 핵심 비즈니스 로직과 부가 기능을 분리한다.

#### 3. Facade 패턴

Facade 패턴은 복잡한 하위 시스템에 대해 단순화된 接口를 제공하여, client가 하위 시스템의复杂性를 알 필요 없이 사용할 수 있도록 하는 패턴이다. 아파트 관리사무소(퍼사드)가 개별 전기, 상하수도, 가스 시설(하위 시스템)을대신 관리하고, resident에게는 간단한 창구 하나만 제공하는 것과 같은 원리다.

Facade 패턴의 역할을 시각화하면, 복잡한子系统群 앞에 단순한窓口(Facade)가 놓여 client의 접근을 정리하는 구조가 명확히 드러난다.

```text
┌─────────────────────────────────────────────────────────────────┐
│                  Facade 패턴 역할 시각화                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [Client]                    [Facade]                            │
│     │                         │                                 │
│     │  simpler interface       │                                 │
│     │  ───────────────────────│                                 │
│     │ ────────────────────────▶│                                 │
│     │                         │                                 │
│     │                         ▼                                 │
│     │              ┌─────────────────────┐                     │
│     │              │   ComputerFacade    │                     │
│     │              │ (컴퓨터 시작Facade)  │                     │
│     │              ├─────────────────────┤                     │
│     │              │ + start() {          │                     │
│     │              │     cpu.start();     │                     │
│     │              │     memory.load();   │                     │
│     │              │     disk.read();     │                     │
│     │              │   }                  │                     │
│     │              │ + stop() { ... }    │                     │
│     │              └──────────┬──────────┘                     │
│     │                         │                                 │
│     │            ┌────────────┴────────────┐                    │
│     │            ▼            ▼            ▼                    │
│     │      ┌─────────┐  ┌─────────┐  ┌─────────┐              │
│     │      │   CPU   │  │ Memory  │  │  Disk   │              │
│     │      └─────────┘  └─────────┘  └─────────┘              │
│     │         ▲            ▲            ▲                       │
│     │         │            │            │                       │
│     │         └────────────┴────────────┘                       │
│     │                   ▼                                        │
│     │            Complex Subsystems                              │
│     │            (복잡한 하위 시스템들)                            │
│                                                                 │
│  Client 관점:                                                   │
│  ──────────                                                    │
│  without Facade:                                                │
│    cpu.start(); memory.load(); disk.read(); // 세부 지식 필요     │
│                                                                 │
│  with Facade:                                                   │
│    computer.start(); // 간단한 호출만으로 충분                    │
│                                                                 │
│  핵심 이점:                                                      │
│  · client 코드 단순화                                           │
│  · 하위 시스템과 client 간 결합도 감소                            │
│  · 하위 시스템 개별 사용도 여전히 가능                              │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Client가 직접 Subsystem 클래스들(CPU, Memory, Disk)을 제어하려면 각 클래스의 使用方法和순서를 정확히 알고 있어야 한다. 그러나 Facade는 이 복잡한 interaction을 단일 메서드(start(), stop())로 캡슐화하여 Client에게 제공한다. 핵심은 Facade가 하위 시스템을 "대신 관리"할 뿐 아니라, 호출 순서와 방법을 제어하여 시스템 전체가 정상적으로 동작하도록协调한다는 점이다. 실무에서는 복잡한 라이브러리/프레임워크의初期化过程, 旅游 booking 시스템의 항공+호텔+보험パッケージ、금융 시스템의 계좌개설流程等에广泛应用된다.

📢 섹션 요약 비유: 생성 패턴은 "전세�� 제조 공장"에서 자동차를 만드는 방법(Factory)을 결정하는 것처럼オブジェクト作成の責任を谁来負うかに焦点を当てる。 구조 패턴은 "자동차에 옵션(에어컨,썬루프)을 추가하는 방법"처럼既存の構造に機能を追加 또는 변형하는 데焦点を当てる.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### Factory Method vs Abstract Factory 비교

| 비교 항목 | Factory Method | Abstract Factory |
|:---|:---|:---|
| **생성 대상** | 단일 제품 (single product) | 제품군 전체 (product family) |
| **확장 방식** | 하위 클래스에서 제품 클래스 선택 | 새 제품군 추가 시Factory 계층 전체 수정 |
| **적합 상황** | 제품 유형이 자주 변경되는 경우 | 제품 조합(가족)이 고정인 경우 |
| **코드 복잡도** | 상대적으로 단순 | 인터페이스와 구현체 수가 많아짐 |

두 패턴의 선택 기준은 "확장하려는 대상이 제품 개별인가, 제품 군인가"이다. 새로운 결제 수단(카카오, 네이버, 토스)이 추가된다면 → Factory Method, OS별 UI整套 변경(Windows, Mac, Linux)이라면 → Abstract Factory가 적합하다.

### Adapter vs Decorator vs Facade 비교

| 비교 항목 | Adapter | Decorator | Facade |
|:---|:---|:---|:---|
| **목적** | 接口 호환성 확보 (접합) | 기능 동적 추가 (장식) | 접근 단순화 (단순화) |
| **변환 방향** | 기존 → client 요구 | 기존 + α | 복잡한 subsystem → 단순窓口 |
| **동적/정적** | 정적 (구성 시决定) | 동적 (실행 시 조합) | 정적 (提供者は固定) |
| **기존 类 수정** | 불필요 | 불필요 | 불필요 |
| **주使用처** | 레거시 연동 | 로깅, 트랜잭션, 검증 | 복잡系统의 간단한入口 |

### 과목 융합 관점

- **컴퓨터구조**: Adapter 패턴은Hardware의 interface 변환(HW Interface Adapter)과 동일한 개념으로, 레거시 주변기기연결에서 필수적
- **네트워크**:Decorator는 OSI 7계층에서 각 계층이 상위 계층에 서비스를 제공하는 구조와 유사
- **데이터베이스**: Facade 패턴은 ORM(Object-Relational Mapping) 프레임워크가 복잡한 SQL를 감추는 것과 동일한 역할

📢 섹션 요약 비유: Adapterは「変換기」(여러国の 전기 플러그 변환)와 같고, Decoratorは「장식 추가」(케이크에 초콜릿, 딸기添え)와 같으며, Facade는「경비원」(아파트 전체 시설 대신 입주민에게簡単な窓口提供服务)와 같다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 시나리오 1: 레거시 결제 시스템에 신규 PG사 연동

**문제 상황**: 기존 시스템이 KG 이니시스(PG사) 전용으로 개발되어 있고, 신규 카카오 페이 연동이 필요한 상황. 레거시 코드를 최대한 유지하면서 연동해야 한다.

**기술사적 판단**: Adapter 패턴을 적용하여 기존 PaymentGateway( KG 이니시스 전용)를 감싸는 KakaoPaymentAdapter를 생성한다. 이 Adapter가 KG 이니시스의 processPayment() 메서드를 호출하고, 그 결과를 카카오 페이 요구 포맷으로 변환하여 반환한다. 기존 코드의 내부 로직은変更不要이며, 새로운 PG사 추가 시 기존 코드 수정 없이 Adapter만 추가하면 된다(OCP 준수). 기술사로서 레거시 시스템 연동 시 "변경보다 추가" 원칙을 적용하면 리스크를 최소화할 수 있다.

### 시나리오 2: 대용량 로그 시스템에 동적 필터링 기능 추가

**문제 상황**: 기존 Logger 시스템에 로그 레벨 필터링, 로그 Rotation, 원격 전송 기능을 동적으로 조합해야 한다. 기능 조합이 런타임에 결정된다.

**기술사적 판단**: Decorator 패턴을 적용한다. 기본 Logger을 감싸는 LevelFilterDecorator(레벨 필터), RotationDecorator(로테이션), RemoteTransportDecorator(원격 전송)를 순차적으로 적용한다. 조합은 설정 파일 또는 코드에서 동적으로 결정되므로, 새로운 필터 추가 시 기존 Logger 코드를 수정하지 않고 Decorator만 추가하면 된다. 이는 SOLID의 OCP를 직접 실현하는 설계 판단이다.

### 도입 체크리스트

**기술적 항목**:
- 객체 생성 로직이 여러 곳에 중복되어 있는가? ( Builder 또는 Factory 적용 검토)
- 특정 클래스가 다른 구체적 클래스에 강하게 결합되어 있는가? (Abstract Factory 또는DI 적용 검토)
- 기존 시스템과 새 시스템의 接口가 호환되지 않는가? (Adapter 적용 검토)
- 기능 추가 시 기존 클래스를 직접 수정하는 경우가 많은가? (Decorator 적용 검토)

**운영·보안적 항목**:
- Singleton 사용 시 멀티스레드 환경에서 인스턴스 생성 충돌 가능성이 있는가? (Thread-safe Singleton 검토)
- Factory 패턴 사용 시 구체적 제품 클래스가 너무 많아져 관리자가 복잡해지는 상황인가?

### 안티패턴

**安티패턴: 너무 많은 Singleton**: 애플리케이션 전역에서 Singleton을 남발하면 테스트가 어려워지고, 암묵적 의존성이 증가하여 코드를 이해하기 어려워진다. Singleton은 정말 全局唯하나의 상태가 필요한 경우에만 사용해야 한다.

**安티패턴: 과도한 Adapter**: 단순한 接口変換에도すべてAdapter를 적용하면 类階層が過度に複雑화된다. 호환성이 정말 필요한 경우에만 Adapter를 적용하고, 단순히 接口가 다른 경우에는 통합하는 것이 더 나은 경우도 있다.

📢 섹션 요약 비유: 패턴도 약이다. 적절한 용량과 용도에 맞게 먹으면疗效がありますが(설계 장점 극대화), 남용하면副作用가 발생한다(과도한 abstraction으로 인한 복잡성 증가).

### 미래 전망

향후 生成 및 構造 패턴은 AI 코드 生成 도구와의 협업으로 진화할 것이다. AI가 Boilerplate 객체 생성 코드를 자동生成하고, 개발자는 패턴 적용 대상 여부를 판단하는 것에 집중하게 될 것이다. 또한 Spring Boot, Angular, React 등 현대 프레임워크가 이미 패턴을 내부에抽象화하고 있어, 개발자가 직접 패턴을 구현하기보다 프레임워크의 제공 기능을 활용하는 쪽으로主流가 이동하고 있다.

### 관련 표준 및 참고 자료

- GoF, 「Design Patterns: Elements of Reusable Object-Oriented Software」: 23가지 패턴의 원조
- Christopher Dennis, 「Patterns of Enterprise Application Architecture」: 엔터프라이즈 系统 설계 패턴
- Martin Fowler, 「Refactoring」: 기존 코드를 패턴 기반으로 리팩토링하는 方法論

📢 섹션 요약 비유: 생성 패턴은 요리 레시피에서 재료 선택 방법을 표준화하는 것이고, 구조 패턴은 기본 요리에 곁들이는 재료 조합을 표준화하는 것이다. 둘 다 "문제를 해결한 경험의結晶"이다.

## 📌 관련 개념 맵 (Knowledge Graph)

- **Factory Method**: 객체 생성 책임을 하위 클래스에 위임하여 OCP를 실현하는 生成 패턴
- **Abstract Factory**: 관련 객체 군(제품군) 전체의 생성責任을抽象化する 生成 패턴
- **Singleton**: 애플리케이션 전역에서 유일한 인스턴스를 보장하는 生成 패턴
- **Adapter**: 호환성 없는 接口를 변환하여 호환 가능하게 하는 構造 패턴
- **Decorator**: 객체에 동적으로 responsibilities를 추가하는 構造 패턴
- **Facade**: 복잡한 子 시스템을 단순化した 接口로 제공하는 構造 패턴
- **SOLID 원칙**: 패턴 적용의根本 원리로, 특히 OCP와 DIP가 패턴選別に重要

## 👶 어린이를 위한 3줄 비유 설명

1. Factory Method 패턴은 "장난감 공장에서 어떤 장난감을 만들지 子클래스가 결정하는 것"과 같아요. 공장 엄마는 "장난감 만들어!"만 말하고, 어떤 장난감을 만들지는 아기 공장(서브클래스)이 정해요.
2. Abstract Factory 패턴은 "한국식 떡국 재료整套와 일본식 오니기라멘 재료整套을 동시에 선택해서 세트트로 만드는 것"과 같아요.整套가 다맞아야 맛이協調한다.
3. Adapter 패턴은 "여러 나라 충전기를 한국 콘센트에 맞게 변환해주는 어댑터와 같아요. 기기의自体はKorea specification를몰라도 어댑터가 변환해주니 편하다.
