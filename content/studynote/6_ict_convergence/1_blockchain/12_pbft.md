# PBFT (Practical BFT): 비동기 네트워크를 위한 실무적 비잔틴 합의

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PBFT(Practical Byzantine Fault Tolerance)는 1999년 카스트로(Castro)와 리스코프(Liskov)가 제안한 알고리즘으로, 지수적 복잡도를 가졌던 기존 BFT를 다항 시간 복잡도($O(n^2)$)로 개선하여 실무 적용이 가능하게 만든 **상태 기계 복제(SMR)** 합의 프로토콜이다.
> 2. **가치**: 거래가 합의되는 즉시 확정되는 **'즉각적 완결성(Immediate Finality)'**을 제공하며, 분산 노드 중 1/3 미만의 비잔틴 장애(배신/오작동)가 발생해도 시스템의 전체 신뢰성을 보장한다.
> 3. **융합**: 하이퍼레저 패브릭(Hyperledger Fabric), R3 Corda 등 허가형(Permissioned) 엔터프라이즈 블록체인의 표준 합의 모델로 자리 잡았으며, 텐더민트(Tendermint) 등 현대적 BFT 알고리즘의 모태가 되었다.

+++

## Ⅰ. PBFT (Practical BFT) 개요

### 1. 정의
- 비동기 분산 시스템에서 비잔틴 장애를 허용하며 합의를 이루는 실무 중심의 알고리즘이다.
- 모든 노드가 동일한 순서로 요청을 실행하여 모든 정상 노드의 상태를 일치시키는 **복제 상태 기계(Replicated State Machine)** 원리를 따른다.

### 2. 💡 비유: "3심제 재판과 교차 확인"
- PBFT는 매우 엄격한 **3심제 재판**과 같습니다. 판사(리더)가 판결문을 써서 배심원(노드)들에게 보내면, 배심원들은 서로 "너도 같은 판결문을 받았니?"라고 두 번이나 서로 대조하며 확인합니다. 배심원의 2/3 이상이 똑같은 판결문을 받았다고 서로 확인 도장을 찍었을 때만 비로소 판결이 확정됩니다. 판사가 스파이라 하더라도 배심원들이 서로 대조해보는 과정에서 거짓말이 들통나게 됩니다.

### 3. 주요 특징
- **강한 일관성 (Strong Consistency)**: 합의된 블록은 결코 뒤집히지 않음.
- **처리량 (Throughput)**: PoW에 비해 압도적으로 빠르나, 노드 수 증가 시 통신량 급증.
- **참여 제한**: 모든 노드의 신원을 알고 있어야 하므로 프라이빗 환경에 적합.

📢 **섹션 요약 비유**: 서로를 100% 믿지 못하는 파트너들이 모여 사업을 할 때, 모든 계약서에 대해 서로가 서로의 사인을 일일이 확인하여 누구도 몰래 내용을 고치지 못하게 하는 '철저한 교차 검증 체계'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. PBFT의 시스템 모델 및 상태

| 요소 | 역할 | 특징 | 비유 |
|:---|:---|:---|:---|
| **Primary (리더)** | 클라이언트의 요청을 받아 순서 할당 및 전파 | 매 뷰(View)마다 교체될 수 있음 | 회의 주관자 |
| **Backups (백업 노드)** | 리더의 제안을 검증하고 투표 수행 | 리더의 배신 감시 및 투표 참여 | 회의 참석자 |
| **View (뷰)** | 현재 리더가 통치하는 기간 | 리더 장애 시 View Change 발생 | 의장의 임기 |
| **Quorum (정족수)** | 합의에 필요한 최소 동의 수 | 전체 $n$개 중 $2f + 1$개 이상 | 의결 정족수 |

### 2. PBFT의 3단계 합의 프로토콜 프로세스 (ASCII)

```text
┌─────────────────────────────────────────────────────────────┐
│               [ PBFT 3-Phase Protocol ]                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Client      Primary       Backup 1      Backup 2      Backup 3
│     │            │             │             │             │
│  1. Request ───▶ │             │             │             │
│     │            │             │             │             │
│     │      2. Pre-prepare ───▶ │ ──────────▶ │ ──────────▶ │
│     │            │             │             │             │
│     │            │ ◀── 3. Prepare (All-to-All) ──────────▶ │
│     │            │             │             │             │
│     │            │ ◀── 4. Commit (All-to-All) ───────────▶ │
│     │            │             │             │             │
│  5. Reply ◀───── ┴ ─────────── ┴ ─────────── ┴ ─────────── ┘
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 
1. **Pre-prepare**: 리더가 요청에 순서 번호($n$)를 붙여 모든 노드에 보냄. 리더가 정직하다는 전제하에 합의를 시작함.
2. **Prepare**: 각 노드는 리더의 메시지를 받고, 다른 모든 노드에게 "나는 이 순서($n$)에 동의한다"고 알림. $2f$개의 일치하는 메시지를 받으면 합의 준비가 완료됨.
3. **Commit**: 준비가 된 노드들은 다시 모든 노드에게 "나는 이제 이 거래를 확정할 준비가 됐다"고 알림. 다시 $2f+1$개의 일치하는 메시지를 받으면 로컬 원장에 거래를 기록(Commit)함.
- 이 과정에서 각 단계마다 $O(n^2)$의 메시지 전송이 발생하며, 이를 통해 리더의 배신이나 노드의 장애를 극복함.

### 3. 심층 동작 메커니즘: View Change
- **장애 탐지**: 리더가 일정 시간 동안 메시지를 보내지 않거나 잘못된 정보를 보내면, 백업 노드들은 타임아웃을 감지함.
- **리더 교체**: 노드들이 "리더를 바꾸자"는 메시지를 주고받아 $2f+1$개 이상의 동의가 모이면 다음 순번의 노드가 새로운 리더(Primary)가 됨. 이를 통해 시스템의 **Liveness(지속성)**를 보장함.

📢 **섹션 요약 비유**: 회의 주관자가 딴소리를 하거나 자리를 비우면, 참석자들이 서로 눈빛을 교환하여 주관자를 즉시 교체하고 회의를 이어가는 '자동 비상대책 시스템'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. PoW vs PBFT 심층 비교

| 비교 항목 | 작업 증명 (PoW) | PBFT (Practical BFT) |
|:---:|:---|:---|
| **합의 방식** | 해시 파워 경쟁 (연산) | 다수결 투표 (메시지 교환) |
| **완결성** | 확률적 (시간 경과 필요) | 즉각적 (합의 즉시 확정) |
| **확장성** | 노드 수 무제한 | 노드 수 제한 (일반적으로 수십 개) |
| **익명성** | 높음 (누구나 참여) | 낮음 (신원 확인 필수) |
| **에너지 효율** | 매우 낮음 | 매우 높음 |

### 2. PBFT vs Tendermint
- **Tendermint**: PBFT를 현대적으로 개량하여, 블록 생성 과정에서 투표를 결합하고 리더를 매 라운드 교체하도록 단순화함. 코스모스(Cosmos) 네트워크의 핵심 합의 엔진으로 사용됨.

📢 **섹션 요약 비유**: PoW가 "누가 더 힘이 센가"를 겨루는 씨름 시합이라면, PBFT는 "누가 더 많은 찬성표를 얻었나"를 따지는 의회 투표와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오: 하이퍼레저 패브릭(Hyperledger Fabric) 적용
- **문제**: 기업 간 무역 금융에서 거래가 뒤집히면(Fork) 법적 분쟁이 발생함.
- **해결**: PBFT 기반의 Orderer 서비스를 사용하여 거래가 체인에 기록되는 즉시 완결성을 갖게 함으로써, 롤백(Rollback) 걱정 없는 안전한 기업용 블록체인 네트워크를 구축함.

### 2. 기술사적 판단 (Insight)
- **노드 수의 한계**: PBFT는 메시지 복잡도가 $O(n^2)$이므로, 노드가 100개만 넘어가도 네트워크 부하가 극심해짐. 따라서 대규모 퍼블릭 체인보다는 소수의 허가된 노드가 운영하는 **프라이빗/컨소시엄 블록체인**에 최적화된 기술임.
- **Checkpointing**: 무한히 쌓이는 메시지 로그를 관리하기 위해 일정 주기마다 상태를 스냅샷 찍고 로그를 삭제하는 체크포인팅 기술이 실무 운영의 핵심임.

📢 **섹션 요약 비유**: 모든 국민이 모여 투표하는 건 불가능하지만, 각 도의 대표자 20명이 모여 결정하는 건 매우 빠르고 확실한 것과 같은 '엘리트 합의 모델'입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성적 기대효과
- **확정적 금융 서비스**: 결제 즉시 승인이 완료되어야 하는 실시간 금융 인프라 구현 가능.
- **낮은 오버헤드**: 컴퓨팅 자원을 낭비하지 않고 오직 네트워크 통신만으로 신뢰를 구축하여 운영 비용 절감.

### 2. 미래 전망
- **BFT 최적화 연구**: HotStuff와 같이 통신 복잡도를 $O(n)$으로 줄여 수백 개 이상의 노드에서도 동작하는 차세대 BFT로 진화 중.
- **분산 금융(DeFi) 인프라**: 중앙 기관 없는 금융 거래의 최종 확정 수단으로서 BFT 계열 알고리즘의 중요성은 더욱 커질 것임.

📢 **섹션 요약 비유**: PBFT는 디지털 세상에서 **'완벽하게 굳은 콘크리트'**와 같아서, 한 번 합의된 결과는 절대로 흔들리지 않는 튼튼한 기초가 됩니다.

### 📌 관련 개념 맵
- **비잔틴 장애 허용 (BFT)**: PBFT가 구현하고자 하는 이론적 목표.
- **텐더민트 (Tendermint)**: PBFT를 개량한 현대적 합의 엔진.
- **하이퍼레저 패브릭**: PBFT를 주요 합의 방식으로 사용하는 기업용 프레임워크.
- **즉각적 완결성 (Finality)**: PBFT의 가장 큰 기술적 장점.

### 👶 어린이를 위한 3줄 비유 설명
1. **서로 확인하는 친구들**: 친구들이 모여서 놀이를 할 때, 한 친구가 룰을 말하면 다른 모든 친구가 서로 "너도 그 룰 들었어?"라고 두 번이나 물어보고 확인하는 방법이에요.
2. **도장 쾅쾅!**: 친구들의 2/3 이상이 "응, 똑같이 들었어!"라고 하면 그 규칙은 절대 바꿀 수 없는 진짜 약속이 된답니다.
3. **가짜는 안 돼**: 누가 거짓말을 하거나 중간에 없어져도, 나머지 친구들이 서로 확인을 꼼꼼히 하니까 아무도 속일 수 없어요.
