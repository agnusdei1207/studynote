# BFT 합의 3단계: 배신자를 걸러내는 세 번의 약속과 검증

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: BFT 합의 3단계는 분산 노드들이 악의적인 공격이나 메시지 오류가 있는 환경에서도 동일한 상태에 도달하기 위해 거치는 **Pre-prepare, Prepare, Commit**의 순차적 메시지 교환 절차다.
> 2. **가치**: 리더 노드가 거짓말을 하거나 일부 노드가 배신하더라도, 노드 간의 **교차 검증(All-to-All)**을 통해 전체의 2/3 이상이 동의한 정당한 거래만 원장에 기록함을 보장한다.
> 3. **융합**: 분산 상태 기계 복제(SMR) 이론의 핵심이며, 하이퍼레저 패브릭이나 텐더민트 등 엔터프라이즈 블록체인의 **즉각적 완결성(Finality)**을 구현하는 기술적 토대가 된다.

+++

## Ⅰ. BFT 합의 3단계 개요

### 1. 정의
- 비잔틴 장애 허용(BFT) 환경에서 신뢰할 수 없는 노드들 간에 하나의 합의된 값을 도출하기 위한 구체적인 통신 프로토콜 단계다.
- 각 단계는 이전 단계의 결과가 유효함을 수학적으로 확인하고 다음 단계로 전이되는 구조를 가진다.

### 2. 💡 비유: "3심제 재판과 투표"
- BFT 3단계는 **매우 중요한 판결을 내리는 재판 과정**과 같습니다.
  - **1단계(Pre-prepare)**: 판사가 판결문을 써서 모든 배심원에게 나누어줍니다. (안건 상정)
  - **2단계(Prepare)**: 배심원들이 서로 "너도 같은 판결문을 받았니?"라고 확인하며 "나도 동의해"라고 외칩니다. (상호 확인)
  - **3단계(Commit)**: 배심원 과반수가 동의했다는 게 확인되면, 마지막으로 "이제 진짜로 장부에 적자!"라고 확정 도장을 찍습니다. (최종 확정)

### 3. 단계별 기본 전제
- **Primary (Leader)**: 안건을 제안하는 노드.
- **Backup (Nodes)**: 제안을 검증하고 투표하는 노드들.
- **$3f+1$ 조건**: $f$개의 비잔틴 노드가 있어도 합의가 가능하려면 전체 노드는 $3f+1$개 이상이어야 함.

📢 **섹션 요약 비유**: 한 명의 지휘관 말만 믿고 돌격하는 게 아니라, 모든 병사가 서로 옆 사람의 명령지를 세 번 대조해보고 움직이는 '극강의 신중함'입니다.

## Ⅱ. 단계별 상세 동작 원리 (Deep Dive)

### 1. BFT 3단계 메시지 교환 프로세스 상세

| 단계명 | 주요 메시지 | 역할 및 동작 | 성공 조건 |
|:---|:---|:---|:---|
| **1. Pre-prepare** | `<PRE-PREPARE, v, n, d>` | 리더가 요청에 순번($n$)을 붙여 전파. | 리더의 서명이 유효함 |
| **2. Prepare** | `<PREPARE, v, n, d, i>` | 각 노드가 리더의 메시지를 인정하고 모두에게 전파. | **$2f$ 개**의 동일 메시지 수신 |
| **3. Commit** | `<COMMIT, v, n, d, i>` | 합의 준비 완료를 선언하고 최종 서명을 교환. | **$2f+1$ 개**의 동일 메시지 수신 |

### 2. 메시지 교환 아키텍처 다이어그램 (ASCII)

```text
┌─────────────────────────────────────────────────────────────┐
│               [ BFT 3-PHASE MESSAGE FLOW ]                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Leader (P)      Node 1 (B)      Node 2 (B)      Node 3 (B)│
│      │               │               │               │      │
│  [ Pre-prepare ] ───▶│──────────────▶│──────────────▶│ (1)  │
│      │               │               │               │      │
│      │               │◀── [ Prepare ]───────────────▶│ (2)  │
│      │               │ (All-to-All broadcast)        │      │
│      │               │               │               │      │
│      │               │◀── [ Commit ] ───────────────▶│ (3)  │
│      │               │ (All-to-All broadcast)        │      │
│      │               │               │               │      │
│      ▼               ▼               ▼               ▼      │
│  [ EXECUTE ]     [ EXECUTE ]     [ EXECUTE ]     [ EXECUTE ]│
│                                                             │
└───────────────────────────┬─────────────────────────────────┘
                            │
            ┌───────────────┴───────────────┐
    [ Strong Consistency ]          [ View Change Safety ]
   (모든 노드 상태 일치)           (리더 교체 시에도 무결 유지)
```

**[다이어그램 해설]** 
1. **Pre-prepare**: 리더가 "이 거래를 10번 순서로 처리하자"라고 제안한다.
2. **Prepare**: 각 노드는 리더의 제안이 자신의 장부 순서와 맞는지 확인하고, 다른 모든 노드에게 "나는 10번 순서 제안에 찬성한다"고 보낸다. 노드들은 서로의 찬성표를 수집하며 과반수($2f+1$)가 모였는지 확인한다.
3. **Commit**: 과반수 찬성이 확인되면, "이제 이 거래를 영구히 기록하자"고 최종 확정 메시지를 보낸다. 다시 한번 과반수 확정표가 모이면 비로소 자신의 로컬 원장에 데이터를 적는다. 이 촘촘한 그물망 같은 교차 확인 덕분에 리더가 일부 노드에게만 거짓말을 해도 나머지 정직한 노드들에 의해 즉시 발각된다.

### 3. 심층 동작 원리: Prepared Certificate (준비 증명)
- 특정 노드가 $2f$개의 `Prepare` 메시지를 받으면 **'Prepared State'**가 된다. 이는 "나만 찬성하는 게 아니라, 네트워크의 정족수가 이 안건을 알고 동의했다"는 암호학적 증거가 된다. 이 증거가 있어야만 마지막 `Commit` 단계로 넘어갈 수 있다.

📢 **섹션 요약 비유**: 투표함에 내 표를 넣기 전에, 다른 친구들의 투표용지를 미리 훑어보고 "아, 다들 나랑 같은 생각이구나"라는 확신(증명)을 얻어야만 최종 도장을 찍는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. BFT 3단계 vs Paxos/Raft (CFT)

| 비교 항목 | Paxos / Raft (CFT) | BFT 3단계 (PBFT 등) |
|:---:|:---|:---|
| **장애 가정** | 노드 중단 (Crash)만 고려 | **배신/악성 행위 (Byzantine) 고려** |
| **통신 단계** | 2단계 (Prepare -> Accept) | **3단계 (Pre-prep -> Prep -> Commit)** |
| **메시지 복잡도** | $O(n)$ | **$O(n^2)$ (기하급수적 증가)** |
| **정족수(Quorum)** | $f+1$ (과반수) | **$2f+1$ (2/3 이상)** |
| **신뢰 환경** | 신뢰할 수 있는 관리 서버군 | 신뢰할 수 없는 분산 네트워크 |

### 2. 과목 융합 관점 (네트워크 & 보안)
- **네트워크 (Throughput)**: 3단계 합의는 메시지 교환량이 노드 수의 제곱에 비례하므로 네트워크 대역폭을 많이 소모한다. 따라서 실무에서는 노드 수를 수십 개 이내로 제한하거나, 계층형 구조를 사용하여 부하를 분산한다.
- **보안 (Safety & Liveness)**: BFT 3단계는 **Safety(합의의 정확성)**를 최우선으로 한다. 즉, 잘못된 합의가 일어날 바에는 시스템이 잠시 멈추는(Liveness 희생) 길을 택하는 엄격한 보안 모델이다.

📢 **섹션 요약 비유**: Raft가 "아픈 친구는 빼고 다수결하자"는 효율 중심이라면, BFT는 "거짓말쟁이가 있을지 모르니 세 번씩 꼬치꼬치 캐묻자"는 보안 중심의 차이가 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오: 프라이빗 블록체인 노드 확장 설계
- **상황**: 기업 연합 블록체인에서 노드 수를 10개에서 100개로 늘리려 한다.
- **적용**: **BFT 3단계의 통신 부하** 고려. 100개 노드가 All-to-All 통신을 하면 네트워크 마비가 올 수 있다. 기술사는 이때 모든 노드가 합의에 참여하는 대신, 무작위로 선출된 20개의 '검증인 그룹'만 BFT 3단계를 수행하고 나머지는 결과만 복제하는 **'위원 선출형 BFT'** 아키텍처를 제안해야 한다.

### 2. 기술사적 판단 (Insight)
- **즉각적 완결성(Instant Finality)**: PoW는 확률적이라 거래가 뒤집힐 수 있지만, BFT 3단계는 `Commit`이 끝나는 순간 거래가 영구 확정된다. 이는 **금융권 실시간 결제**나 **증권 정산** 시스템에서 가장 중요하게 평가받는 기술적 속성이다.
- **View Change의 비용**: 리더가 죽었을 때 새로운 리더를 뽑는 'View Change' 과정은 3단계 합의보다 훨씬 복잡하고 오래 걸린다. 실무 운영 시에는 리더 노드의 상태 모니터링(Keep-alive)을 가장 높은 우선순위로 관리해야 한다.

📢 **섹션 요약 비유**: 모든 사원이 결재 서류를 돌려보는 건 비효율적이므로, 신뢰받는 '운영위원회'를 뽑아 그들끼리 빠르게 3심제를 진행하게 하는 것이 실무의 지혜입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성적 기대효과
- **무신뢰 협업 실현**: 서로 경쟁 관계인 기업들이 중앙 관리자 없이도 100% 신뢰할 수 있는 공동 장부를 가질 수 있게 한다.
- **데이터 일관성 보장**: 전 세계 노드가 단 1초의 오차도 없이 동일한 상태를 유지하는 '글로벌 동기화' 달성.

### 2. 미래 전망
- **HotStuff 합의 알고리즘**: 페이스북 리브라에서 제안된 방식으로, 3단계를 선형적(Star topology)으로 개선하여 통신 효율을 $O(n)$으로 낮춘 차세대 BFT 기술이 확산될 것이다.
- **하이브리드 BFT**: PoS의 거버넌스와 BFT의 빠른 완결성을 결합한 모델(예: Casper FFG)이 퍼블릭 체인의 표준으로 자리 잡을 전망이다.

📢 **섹션 요약 비유**: BFT 3단계는 디지털 세상에서 **'누구도 부정할 수 없는 진실'**을 만들어내는 가장 정교하고 튼튼한 거푸집입니다.

+++

### 📌 관련 개념 맵
- **PBFT**: 3단계 합의를 실무적으로 구현한 최초의 알고리즘.
- **비잔틴 장애 허용 (BFT)**: 3단계 합의가 해결하려는 상위 이론.
- **즉각적 완결성 (Finality)**: 3단계 합의를 통해 얻는 기술적 이득.
- **View Change**: 리더 장애 시 대처하는 비상 매커니즘.

### 👶 어린이를 위한 3줄 비유 설명
1. **친구들의 세 번 확인**: 술래잡기 규칙을 정할 때, 한 친구가 말하면 다들 "나도 들었어!"라고 대답하고, 마지막으로 "진짜 이대로 하는 거다!"라고 손가락 걸고 약속하는 3단계 과정이에요.
2. **거짓말쟁이는 못 껴요**: 친구 한두 명이 일부러 이상한 규칙을 말해도, 나머지 착한 친구들이 서로 물어보며 확인하니까 가짜 규칙은 금방 사라져요.
3. **약속은 바로 쾅쾅**: 세 번의 확인이 끝나면 그 약속은 절대로 바꿀 수 없는 진짜 규칙이 된답니다.
