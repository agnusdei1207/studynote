# 아카이브 노드 (Archive Node): 블록체인의 모든 과거를 기억하는 디지털 도서관

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 아카이브 노드(Archive Node)는 블록체인의 모든 블록 데이터뿐만 아니라, 각 시점에서의 **모든 계정 잔액 및 상태 변화 이력(Historical State)**을 전부 보관하는 특수 노드다.
> 2. **가치**: 일반 풀 노드가 최신 상태만 빠르게 조회하는 데 최적화된 반면, 아카이브 노드는 "3년 전 특정 블록 시점에 A 계정의 잔액이 얼마였는가?"와 같은 **과거 데이터 쿼리**를 즉시 처리할 수 있게 한다.
> 3. **융합**: 블록 익스플로러(Etherscan 등), 온체인 데이터 분석 플랫폼, 세무 및 감사 시스템의 핵심 백엔드로 활용되며, 수십 TB에 달하는 **스토리지 폭발(Storage Bloat)** 문제를 감당하기 위한 고도의 인프라 기술이 요구된다.

+++

## Ⅰ. 아카이브 노드 (Archive Node) 개요

### 1. 정의
- 블록체인 상의 모든 트랜잭션 실행 결과로 생성된 모든 중간 상태(Intermediate States)를 삭제(Pruning)하지 않고 영구히 저장하는 노드다.
- 블록체인의 '현재 결과'뿐만 아니라 그 결과에 도달하기까지의 **'모든 과정'**을 데이터베이스화한 실체다.

### 2. 💡 비유: "일기장의 수정 내역까지 보관하는 박물관"
- 일반 풀 노드가 **일기장의 마지막 페이지(현재 잔액)**를 보여주는 것이라면, 아카이브 노드는 일기장 첫 장부터 지금까지 **글자를 썼다 지웠다 한 모든 흔적과 중간 기록**을 다 보관하고 있는 박물관과 같습니다. 100일 전 점심에 무엇을 먹었는지(과거 상태) 알고 싶을 때, 아카이브 노드라는 박물관에 가면 그날의 일기장을 즉시 펼쳐 볼 수 있습니다.

### 3. 등장 배경
- **정밀한 데이터 분석 요구**: 특정 프로토콜의 과거 성장 지표나 사고 발생 시점의 자금 흐름 추적 필요.
- **DApp 서비스 지원**: 블록 익스플로러나 지갑 앱에서 유저의 과거 활동 내역을 실시간으로 보여주기 위한 인프라 요구.

📢 **섹션 요약 비유**: 현재의 결론만 아는 '뉴스'와 달리, 역사의 모든 장면을 슬로우 모션으로 돌려볼 수 있는 '디지털 기록 저장소'입니다.

## Ⅱ. 아키텍처 및 데이터 보관 원리 (Deep Dive)

### 1. 노드 유형별 저장 데이터 비교

| 항목 | 풀 노드 (Full Node) | 아카이브 노드 (Archive Node) |
|:---|:---|:---|
| **Raw Blocks** | 전체 보유 | 전체 보유 |
| **Current State** | 최신 상태만 보유 | **전체 역사 상태 보유** |
| **Historical Query** | 과거 연산을 재수행해야 함 (느림) | **즉시 조회 가능 (매우 빠름)** |
| **저장 용량 (ETH 기준)** | 약 1~2 TB | **약 15~20 TB 이상 (급증 중)** |
| **주요 용도** | 네트워크 합의, 개인 거래 | 분석, 감사, 인프라 서비스 |

### 2. 아카이브 노드의 상태 저장 구조 다이어그램 (ASCII)

```text
┌─────────────────────────────────────────────────────────────┐
│                [ ARCHIVE NODE DATA STRUCTURE ]              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [ Block N ] ──▶ [ Block N+1 ] ──▶ [ Block N+2 ] (Chain)    │
│       │               │                 │                   │
│       ▼               ▼                 ▼                   │
│  ┌───────────┐   ┌───────────┐     ┌───────────┐            │
│  │ State @N  │   │ State @N+1│     │ State @N+2│ (Snapshots)│
│  │ (Balance) │   │ (Balance) │     │ (Balance) │            │
│  └───────────┘   └───────────┘     └───────────┘            │
│        │               │                 │                  │
│        └───────────────┴─────────────────┘                  │
│               ( All Kept in DB )                            │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│  [ 특징 ]                                                   │
│  1. 상태가 변하지 않은 노드도 블록별로 색인화하여 보관      │
│  2. 과거 시점의 스마트 컨트랙트 실행 시뮬레이션 가능        │
│  3. 디스크 I/O와 스토리지 비용이 기하급수적으로 높음        │
└───────────────────────────┬─────────────────────────────────┘
                            │
            ┌───────────────┴───────────────┐
    [ Full Traceability ]           [ Resource Intensive ]
   (완벽한 과거 추적 능력)         (막대한 유지비용 발생)
```

**[다이어그램 해설]** 
일반 풀 노드는 새로운 블록이 들어오면 이전 상태(State) 데이터 중 더 이상 필요 없는 부분을 지우거나 덮어써서 용량을 아낀다. 하지만 아카이브 노드는 **모든 블록 시점의 상태 스냅샷**을 고유한 인덱스와 함께 저장한다. 이 덕분에 유저가 `eth_getBalance(block_height)`와 같은 명령어로 과거 시점을 조회할 때, 복잡한 재연산 과정 없이 DB에서 즉시 값을 꺼내 줄 수 있다. 이 대가로 아카이브 노드는 일반 노드보다 수십 배 많은 저장 공간을 소모한다.

### 3. 심층 동작 원리: 상태 전이와 재연산 (Re-execution)
- 일반 풀 노드도 이론적으로는 과거 데이터를 알 수 있다. 하지만 이를 위해서는 제네시스 블록부터 해당 시점까지의 모든 거래를 **다시 실행**해봐야 하므로 시간이 수 시간에서 수일이 걸린다. 아카이브 노드는 이 재연산 결과를 미리 다 저장해둔 '정답지'라고 볼 수 있다.

📢 **섹션 요약 비유**: 수학 문제를 풀 때마다 계산 과정을 다 지우는 친구(풀 노드)와, 모든 연습장 노트를 버리지 않고 모아두어 언제든 예전 풀이를 찾아보는 친구(아카이브 노드)의 차이입니다.

| **비용 부담** | 하드웨어 유지비 감당 불가 | 유료 API 호출 수익으로 충당 |

### 2. 과목 융합 관점 (데이터베이스 & 분석)
- **데이터베이스 (Big Data)**: 아카이브 노드는 전형적인 빅데이터 저장소의 특성을 가진다. 쓰기 성능보다 읽기 성능과 데이터 보존성이 중요하며, 이를 위해 효율적인 **인덱싱 알고리즘**과 고성능 저장 장치(NVMe) 융합이 필수다.
- **분석 (On-chain Forensics)**: 자금 세탁 방지(AML)나 해킹 자금 추적 시, 공격자가 거쳐 간 모든 중간 지갑의 과거 상태를 파악하기 위해 아카이브 노드 데이터가 수사 기관의 결정적 증거로 쓰인다.

📢 **섹션 요약 비유**: 아카이브 노드가 '전 세계 도서관의 모든 책을 보관하는 국립중앙도서관'이라면, 일반 노드는 '내게 필요한 책만 빌려보는 동네 도서관'과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오: 블록 익스플로러(Explorer) 구축
- **상황**: 우리 회사가 자체 메인넷을 만들고, 이더스캔 같은 조회 사이트를 운영하려 한다.
- **적용**: **아카이브 노드 구축 및 인덱서(Indexer) 연동**. 단순히 노드만 돌리는 게 아니라, 아카이브 노드에서 나오는 방대한 로우 데이터를 SQL이나 NoSQL로 가공하여 빠르게 검색할 수 있는 '인덱서'를 결합해야 한다. 사용자가 "내 지갑의 1년 전 수익률"을 물어볼 때 아카이브 노드가 그 기반 데이터를 제공한다.

### 2. 기술사적 판단 (Insight)
- **상태 폭발(State Bloat)의 위기**: 블록체인이 길어질수록 아카이브 노드 운영은 불가능에 가까워진다. 기술사는 이를 해결하기 위해 모든 노드가 아카이브가 될 필요가 없는 **'상태 만료(State Expiry)'** 정책이나, 데이터를 조각내어 저장하는 **'에라져 코딩(Erasure Coding)'** 기반 분산 저장 아키텍처를 대안으로 제시해야 한다.
- **중앙화의 위협**: 아카이브 노드 운영비가 너무 비싸져서 소수 기업(Infura 등)만 운영하게 되면, 과거 데이터에 대한 '접근 권한'을 특정 기업이 독점하는 중앙화 문제가 발생한다. 이를 막기 위해 **Portal Network**와 같이 과거 데이터를 P2P로 나누어 보관하는 기술 연구가 실무의 핵심이다.

📢 **섹션 요약 비유**: 모든 기록을 다 보관하는 게 좋지만 창고 비용이 너무 비싸지므로, 마을 사람들이 기록을 나누어 보관하거나(분산 저장) 아주 오래된 기록은 마이크로필름으로 변환하는(상태 압축) 지혜가 필요합니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성적 기대효과
- **데이터 투명성 극대화**: 블록체인의 모든 역사를 1원 단위까지 완벽하게 증명하고 검증 가능.
- **금융 혁신 지원**: 복잡한 과거 데이터 기반의 대출 심사, 파생상품 설계, 세무 자동화 서비스의 기반 마련.

### 2. 미래 전망
- **스테이트리스 시대의 아카이브**: 미래의 노드들이 현재 상태만 저장하게 되면(Stateless), 과거를 기억하는 아카이브 노드는 별도의 **'기록 보관 서비스(History Providers)'**라는 독립된 산업군으로 분화될 것이다.
- **ZK-Proof 기반 압축 아카이브**: 수십 TB의 역사를 단 몇 GB의 영지식 증명으로 압축하여, 아카이브 노드의 기능을 일반 PC에서도 수행하게 하는 혁신이 일어날 전망이다.

📢 **섹션 요약 비유**: 아카이브 노드는 블록체인 문명의 **'기록 유산 센터'**이며, 우리가 지나온 모든 길을 잊지 않게 해주는 가장 거대한 디지털 메모리입니다.

### 📌 관련 개념 맵
- **풀 노드 (Full Node)**: 아카이브 노드의 기본 토대가 되는 노드.
- **이더리움 3대 트리**: 아카이브 노드가 보관하는 상태 데이터의 구조.
- **프루닝 (Pruning)**: 아카이브 노드가 거부하는 데이터 삭제 기술.
- **이더스캔 (Etherscan)**: 아카이브 데이터를 사용자에게 보여주는 대표적 서비스.
