# 마이크로 프론트엔드, 이벤트 소싱, CQRS: 마이크로서비스의 복잡성을 통제하는 현대적 설계 패턴

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스 아키텍처(MSA)가 백엔드(서버)를 잘게 쪼갰다면, **마이크로 프론트엔드**는 화면(UI)까지 기능별로 잘게 쪼개어 독립 배포하는 기술이며, **이벤트 소싱**과 **CQRS**는 쪼개진 DB들 사이의 데이터 불일치와 조회 속도 저하를 막기 위한 데이터 관리 패턴이다.
> 2. **가치**: 이 패턴들은 수백 명의 개발자가 각자의 파트(결제, 장바구니, 리뷰)를 동시에 개발하고 배포해도 시스템이 엉키지 않게 만들며, 수백만 건의 읽기(Read)와 쓰기(Write) 요청을 분리해 극강의 성능 확장을 가능케 한다.
> 3. **융합**: 단일 기술로는 구현이 불가능하며, 메시지 큐(Apache Kafka)를 통한 비동기 통신망과 브라우저 레벨의 컴포넌트 융합(웹팩 모듈 페더레이션)이 유기적으로 결합되어 완성되는 최상위 아키텍처 디자인이다.

---

## Ⅰ. 현대적 앱 아키텍처 설계 패턴 개요

### 1. 정의
- **마이크로 프론트엔드 (Micro Frontends)**: 백엔드의 MSA 사상을 프론트엔드 웹 화면으로 확장하여, 웹 페이지를 여러 개의 독립적인 조각(결제 버튼, 추천 상품 등)으로 나누어 각자 다른 팀이 개발하고 하나로 합치는 아키텍처.
- **이벤트 소싱 (Event Sourcing)**: 데이터의 '최종 상태'만 DB에 덮어쓰는 것이 아니라, 상태를 변화시킨 모든 '이벤트(행동)들의 기록'을 순서대로 블록체인처럼 끊임없이 쌓아 저장하는 데이터 모델링 기법.
- **CQRS (Command and Query Responsibility Segregation)**: 상태를 변경하는 **명령(Command/Write)** 모델과 상태를 조회하는 **조회(Query/Read)** 모델을 물리적, 논리적으로 완전히 분리하는 아키텍처.

### 💡 비유: "은행 통장과 인터넷 쇼핑몰"
- **이벤트 소싱**: 내 통장 잔고가 5,000원(최종 상태)이라는 것만 연필로 썼다 지웠다 하는 것이 아니라, "1만 원 입금, 3천 원 출금, 2천 원 출금"이라는 **입출금 내역 전체(이벤트)를 지워지지 않는 잉크로 꼼꼼히 기록**하는 것입니다. 과거 내역을 쫙 더하면 항상 현재 잔고(5,000원)가 나옵니다.
- **CQRS**: 식당에서 주문받는 직원(Command)과 요리 완성 여부를 알려주는 전광판(Query)을 완전히 분리한 것입니다. 백종원 식당처럼 주문 줄과 음식 받는 줄을 따로 만들어 속도를 극대화합니다.
- **마이크로 프론트엔드**: 네이버 메인 화면을 만들 때 한 명이 다 그리는 게 아니라, 쇼핑팀, 뉴스팀, 웹툰팀이 **각자의 네모난 스티커를 만들어와서 빈 도화지에 덕지덕지 붙여 하나의 화면을 완성**하는 것입니다.

### 3. 등장 배경
- **프론트엔드 모놀리식의 한계**: 백엔드를 아무리 잘게 쪼개도, 화면(프론트) 소스 코드가 한 덩어리면 결국 한 부분만 수정해도 전체 화면을 다시 배포해야 하는 병목이 발생함.
- **분산 트랜잭션의 한계**: DB가 여러 개로 쪼개진 MSA에서 "주문은 성공했는데 결제 DB가 죽었을 때" 이를 롤백(Rollback)하는 과정(2PC)이 너무 느리고 복잡해, 비동기 이벤트 기반의 대안이 절실해짐.

📢 **섹션 요약 비유**: 이 3가지 패턴은 거대한 코끼리(모놀리식 앱)를 여러 마리의 날쌘 치타(MSA)로 조각낼 때, 치타들이 서로 싸우지 않고 하나의 무리처럼 똑똑하게 사냥하도록 만드는 완벽한 '전술 교본'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 마이크로 프론트엔드 아키텍처 (ASCII)

```text
┌─────────────────────────────────────────────────────────────┐
│                 [ MICRO FRONTENDS ARCHITECTURE ]            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [ Client Browser ] (Shell App / App Shell)                │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ ┌───────────────────────────────────────────────────┐ │  │
│  │ │  [ Header Team (React) ] - /header.js             │ │  │
│  │ └───────────────────────────────────────────────────┘ │  │
│  │                                                       │  │
│  │ ┌──────────────────────┐  ┌─────────────────────────┐ │  │
│  │ │ [ Product Team ]     │  │ [ Review Team ]         │ │  │
│  │ │ (Vue.js) - /prod.js  │  │ (Angular) - /review.js  │ │  │
│  │ │                      │  │                         │ │  │
│  │ └──────────────────────┘  └─────────────────────────┘ │  │
│  └───────────────────────────────────────────────────────┘  │
│       ▲                        ▲                        ▲   │
│       │                        │                        │   │
│  [ Product Backend ]    [ Review Backend ]     [ User Backend ]
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 
과거에는 프론트엔드 팀이 React 하나로 모든 화면을 짰다. **마이크로 프론트엔드**에서는 '쇼핑몰 앱'이라는 빈 껍데기(Shell)만 띄워놓고, 상품 설명 파트는 상품팀 서버에서(`Vue.js`), 리뷰 파트는 리뷰팀 서버에서(`Angular`) 실시간으로 스크립트를 불러와 하나로 조합(Composition)한다. 리뷰팀이 코드를 수정하고 배포해도, 상품팀은 1초의 멈춤 없이 자기 일을 계속할 수 있다. (WebPack Module Federation 기술 활용)

### 2. 이벤트 소싱 및 CQRS 통합 아키텍처 (ASCII)

```text
┌─────────────────────────────────────────────────────────────┐
│             [ EVENT SOURCING & CQRS ARCHITECTURE ]          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [ User ] ──(1. Command: "Buy Item")──▶ [ Command API ]    │
│                                                 │           │
│                                                 ▼ (2. Save) │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                [ EVENT STORE (Write DB) ]             │  │
│  │  Log 1: ItemAdded (+$100)                             │  │
│  │  Log 2: CouponApplied (-$10)                          │  │
│  │  (Append-only. No Updates/Deletes!)                   │  │
│  └──────────────────────────┬────────────────────────────┘  │
│                             │                               │
│                             ▼ (3. Publish Event)            │
│                 [ Event Bus (Kafka) ]                       │
│                             │                               │
│                             ▼ (4. Consume & Project)        │
│  ┌───────────────────────────────────────────────────────┐  │
│  │               [ READ DATABASE (Query DB) ]            │  │
│  │  View Table: User A Cart Total = $90                  │  │
│  │  (Optimized for fast SELECT queries)                  │  │
│  └──────────────────────────▲────────────────────────────┘  │
│                             │                               │
│   [ User ] ◀──(5. Read: "Show Cart")──── [ Query API ]      │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 
사용자가 물건을 사면(Command), 시스템은 카트 총액을 90달러로 `UPDATE` 치는 것이 아니라, "100달러 담음", "10달러 깎음"이라는 **이벤트 자체를 Event Store에 추가(Append)**만 한다(이벤트 소싱). 수정과 삭제가 없어 쓰기(Write) 속도가 빛의 속도다.
이 이벤트는 **Kafka**를 타고 넘어가서 읽기 전용 DB에 "총액 90달러"라는 결괏값으로 갱신(Projection)된다. 사용자가 장바구니를 볼 때(Query)는 이 읽기 전용 DB만 쳐다본다. 쓰기 DB(Command)와 읽기 DB(Query)가 완벽히 쪼개진 이 형태가 바로 **CQRS**다.

📢 **섹션 요약 비유**: 글을 쓰는 작가(Command)와 책을 읽어주는 성우(Query)를 완전히 다른 사람으로 고용하여, 작가가 글을 쓰는 동안 성우가 쉴 새 없이 책을 읽어줄 수 있게 만든 분업의 극치입니다.

## Ⅲ. 융합 비교 및 기술 분석

### 1. 전통적 CRUD 모델 vs 이벤트 소싱(Event Sourcing) 비교

| 비교 항목 | 전통적 CRUD (RDBMS) | 이벤트 소싱 (Event Sourcing) |
|:---:|:---|:---|
| **저장 방식** | 상태 값을 계속 덮어씀 (UPDATE, DELETE) | **과거의 이벤트를 순차적으로 누적 (APPEND ONLY)** |
| **데이터 크기** | 현재 상태 1개만 존재하여 크기 작음 | 모든 이력이 쌓이므로 용량이 거대함 |
| **감사(Audit)** | 로그 테이블을 따로 만들어야 알 수 있음 | **저장소 자체가 완벽한 로그이자 히스토리 (블록체인과 유사)** |
| **복원 능력** | 백업본이 없으면 과거로 돌아갈 수 없음 | **이벤트를 과거 특정 시점까지만 다시 재생(Replay)하면 언제든 복원 가능** |

### 2. 과목 융합 관점 (분산 데이터베이스 & 운영체제)
- **최종 일관성 (Eventual Consistency)**: CQRS 환경에서는 쓰기 DB에 데이터가 들어가고 읽기 DB로 동기화되기까지 수백 밀리초(ms)의 시차가 발생한다. 은행 잔고가 아주 잠깐 덜 깎여 보일 수 있다는 뜻이다. 기술사는 이 불일치를 인정하고 "결국에는 데이터가 똑같아진다(Eventual Consistency)"는 분산 시스템의 **CAP 정리(BASE 속성)** 사상을 비즈니스 기획자에게 납득시켜야 한다.
- **스트림 처리 (Stream Processing)**: 이벤트 소싱의 로그를 실시간으로 읽어 읽기 DB로 밀어 넣는 작업은 Apache Kafka나 Flink 같은 실시간 스트리밍 빅데이터 기술과 완벽하게 융합된다.

📢 **섹션 요약 비유**: CRUD가 '연필로 썼다 지우개를 쓰는 장부'라면, 이벤트 소싱은 '만년필로 한 줄 한 줄 이어서 쓰는 일기장'입니다. 지저분하고 길어지지만, 내 과거를 완벽하게 증명해 줍니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오: 토스/배달의민족 같은 트래픽 폭주 시스템
- **상황**: 치킨 반값 할인 이벤트 때 초당 10만 명의 사용자가 앱을 켜서 식당 목록을 봄(Read). 동시에 1만 명이 주문 버튼을 누름(Write). 기존 서버는 읽기 요청 때문에 DB가 뻗어서 주문이 안 됨.
- **적용**: **CQRS 패턴을 통한 Read/Write 완전 분리**.
  - **Write DB**: 정확성이 생명인 RDBMS(MySQL)로 구축하여 주문 이벤트만 미친 듯이 받아냄(Command).
  - **Read DB**: 읽기 속도가 극강인 NoSQL(Redis, MongoDB)을 10대 복제하여 배치함(Query).
  - 식당 목록과 메뉴판은 무조건 Read DB(Redis)에서만 가져가게 하여 Write DB의 부하를 0으로 만듦.
- **결과**: 읽기 트래픽이 100배 폭증해도 주문(결제) 시스템은 전혀 타격을 받지 않는 완벽한 방어선 구축 성공.

### 2. 기술사적 판단 (Insight)
- **오버 엔지니어링의 치명적 함정**: 이벤트 소싱과 CQRS는 개발 난이도가 상상을 초월한다. 단순히 게시판이나 사내 결재 시스템을 만드는데 이 패턴을 적용하면, 개발 기간이 3배로 늘고 유지보수 지옥에 빠진다(Anti-pattern). 기술사는 **조회(Read) 트래픽이 쓰기(Write) 트래픽보다 100배 이상 많거나, 주문/결제처럼 도메인 지식이 극도로 복잡한(DDD) 핵심 코어 시스템**에만 이 수술 칼(CQRS)을 대는 절제력을 보여야 한다.
- **마이크로 프론트엔드의 CSS/JS 충돌**: 서로 다른 팀이 만든 프론트엔드를 한 화면에 합치면 CSS 클래스명이 겹쳐서 남의 버튼 디자인을 찌그러뜨리는 대참사가 난다. 실무에서는 **Shadow DOM**이나 **BEM(Block Element Modifier)** 방법론을 강제하여 CSS 격리성(Isolation)을 확보하는 아키텍처 규칙이 선행되어야 한다.

📢 **섹션 요약 비유**: 명검(CQRS)은 용(수백만 트래픽)을 잡을 때만 뽑아야 합니다. 파리(사내 시스템)를 잡겠다고 명검을 마구 휘두르면 내 손목(개발팀)만 날아갈 뿐입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성적 기대효과
- **독립적 스케일링**: 트래픽의 95%를 차지하는 조회(Read) 서버만 100대로 늘리고 쓰기(Write) 서버는 2대만 유지하여, 막대한 클라우드 데이터베이스 라이선스 비용 절감.
- **데이터 타임머신**: 이벤트 소싱을 통해 과거 1년 전 오후 3시에 시스템의 상태가 어땠는지 100% 완벽하게 재구성(Replay)할 수 있어, 금융권의 해킹 추적 및 감사(Audit)에 극강의 무기로 작용.

### 2. 미래 전망
- **서버리스와의 결합**: 이벤트를 받아 뷰(View) 테이블을 만드는 작업(Projection)을 AWS Lambda 같은 서버리스 함수로 처리하여 인프라 비용을 제로화하는 Serverless-CQRS 아키텍처가 대세가 될 것이다.
- **프론트엔드의 Edge 배포**: 마이크로 프론트엔드의 각 조각들이 중앙 서버가 아니라 Cloudflare 같은 엣지 네트워크(CDN)에서 렌더링되어 클라이언트에게 0.01초 만에 꽂히는 Edge-side Rendering으로 진화할 전망.

📢 **섹션 요약 비유**: 이 3가지 패턴은 소프트웨어라는 레고 성을 지을 때, 무작정 통째로 본드를 칠하지 않고 언제든 층을 나누고 부품을 갈아 끼울 수 있게 만든 '현대 건축 공학의 마스터피스'입니다.

- **[최종 일관성 (Eventual Consistency)](#)**: CQRS의 부작용으로, 데이터가 동기화되기 전의 찰나의 불일치를 허용하는 개념.

### 👶 어린이를 위한 3줄 비유 설명
1. **각자 그리는 모자이크 (마이크로 프론트엔드)**: 커다란 도화지에 한 명이 다 그리지 않고, 철수는 나무, 영희는 구름을 따로 그려서 나중에 스티커처럼 딱 붙여 완성하는 그림이에요.
2. **절대 안 지우는 일기장 (이벤트 소싱)**: 일기장에 쓴 글을 지우개로 절대 지우지 않고, 무조건 밑에다 계속 새로운 줄을 추가해서 적기 때문에 내 과거의 실수를 모두 기억할 수 있어요.
3. **주문과 요리의 분리 (CQRS)**: 손님에게 주문만 받는 아르바이트생과 주방에서 요리만 하는 요리사를 아예 따로 뽑아서, 식당에 손님이 엄청 몰려도 절대 우왕좌왕하지 않는 똑똑한 식당이랍니다.
