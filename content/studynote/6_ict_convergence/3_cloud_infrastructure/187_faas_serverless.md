# FaaS 및 서버리스 (Serverless): 서버의 존재를 지우고 오직 비즈니스 로직에만 집중하는 클라우드의 진화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: FaaS(Function as a Service)는 서버리스(Serverless) 아키텍처를 구현하는 대표적인 클라우드 서비스 모델로, 개발자가 서버 인프라를 전혀 관리하지 않고 오직 특정 이벤트 발생 시 실행될 '함수(Function) 단위의 코드'만 업로드하여 구동하는 방식이다.
> 2. **가치**: 서버를 24시간 띄워놓지 않으므로 트래픽이 0일 때는 요금이 0원이며, 트래픽이 폭증하면 클라우드 벤더가 1초 만에 함수를 수만 개로 복제(Scale-out)해 주어 극강의 비용 효율성과 오토스케일링을 제공한다.
> 3. **융합**: 마이크로서비스 아키텍처(MSA), API 게이트웨이, 이벤트 주도 아키텍처(EDA)와 결합하여 현대적인 클라우드 네이티브 애플리케이션 구축의 핵심 동력으로 작용한다.

+++

## Ⅰ. FaaS (Function as a Service) 및 서버리스 개요

### 1. 정의
- **서버리스 (Serverless)**: '서버가 없다'는 뜻이 아니라, 서버의 관리(프로비저닝, 패치, 스케일링)를 클라우드 제공자(AWS, GCP 등)가 100% 대행하여 개발자 눈에는 서버가 보이지 않게 추상화한 아키텍처 철학.
- **FaaS**: 이 서버리스 철학을 구현한 구체적인 서비스(예: AWS Lambda, Google Cloud Functions). 애플리케이션을 여러 개의 짧은 함수(Function)로 쪼개어, HTTP 요청이나 DB 변경 같은 '이벤트'가 발생할 때만 잠시 실행되고 종료되는 모델이다.

### 💡 비유: "수도꼭지 요금과 동전 넣는 안마 의자"
- **기존 서버 (IaaS/PaaS)**는 **'집에 있는 정수기 렌탈'**과 같습니다. 물을 마시든 안 마시든 한 달 내내 켜져 있고 매달 똑같은 렌탈료를 내야 합니다. 손님이 100명 오면 정수기를 100대 미리 빌려놔야 합니다.
- **서버리스 (FaaS)**는 **'동전 넣고 쓰는 휴게소 안마 의자'**와 같습니다. 내가 의자에 앉아 동전을 넣은(이벤트 발생) 딱 10분 동안만 기계가 작동하고, 그 10분 치 요금만 냅니다. 아무도 안 앉으면 돈이 전혀 들지 않습니다.

### 3. 등장 배경
- **유휴 자원의 완벽한 제거**: PaaS도 결국 서버를 띄워두어야 하므로 트래픽이 없는 새벽 시간대에는 서버 임대료가 낭비됨. 이 낭비마저 0으로 만들려는 경제적 욕구.
- **개발 생상성 극대화**: 운영체제 패치나 로드밸런서 설정 등 인프라 업무(Ops)를 완전히 없애고, 개발자(Dev)가 오직 코드 작성에만 집중할 수 있는 궁극의 환경 요구.

📢 **섹션 요약 비유**: 복잡한 식당 주방(서버)을 운영하는 대신, 주문(이벤트)이 들어올 때만 마법처럼 요리사(함수)가 나타나 요리를 끝내고 사라지는 '마법의 주방'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. FaaS 동작 메커니즘 (Event-Driven Architecture)

| 컴포넌트 | 상세 내용 | 동작 예시 |
|:---:|:---|:---|
| **Event Source** | 함수를 깨우는 방아쇠(Trigger) | 사용자가 API Gateway로 HTTP 요청, S3에 사진 업로드 |
| **FaaS Platform**| 코드가 실행되는 환경 (컨테이너) | 이벤트 수신 시 즉시 함수 컨테이너를 띄우고 코드를 실행 |
| **Function Code** | 개발자가 작성한 순수 비즈니스 로직 | "업로드된 사진의 사이즈를 줄여라" |
| **Backend Services**| 결과가 저장되거나 연동되는 서비스 | DB에 상태 업데이트, 푸시 알림 전송 |

### 2. 서버리스 아키텍처 vs 전통적 아키텍처 (ASCII)

```text
┌─────────────────────────────────────────────────────────────┐
│                 [ SERVERLESS ARCHITECTURE ]                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [ Traditional Monolithic ]      [ Serverless (FaaS) ]     │
│                                                             │
│         [ Load Balancer ]              👤 User Click        │
│                │                             │              │
│       ┌────────┴────────┐                    ▼ (Event)      │
│       ▼                 ▼              [ API Gateway ]      │
│  [ VM Server ]     [ VM Server ]         │        │         │
│  (Always ON)       (Always ON)           ▼        ▼         │
│  ┌─────────┐       ┌─────────┐      [Lambda]  [Lambda]      │
│  │ Auth    │       │ Auth    │     (Sign-in)  (Upload)      │
│  │ Upload  │       │ Upload  │       │          │           │
│  │ Payment │       │ Payment │       ▼          ▼           │
│  └─────────┘       └─────────┘      [DB]      [S3 Bucket]   │
│                                   (Runs ONLY when invoked)  │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 
기존 아키텍처는 로드밸런서 뒤에 항상 켜져 있는 서버(VM)가 모든 로직을 처리한다. 트래픽이 없어도 월 임대료가 나간다. 반면 **서버리스 아키텍처**는 사용자가 버튼을 클릭하면 **API Gateway**가 이를 이벤트로 인식하여 해당하는 **Lambda 함수**만 깨운다. 이 함수는 임시 컨테이너 안에서 수 밀리초 만에 실행되어 로직을 처리하고 즉시 소멸한다. 이 구조는 진정한 의미의 초단위 과금(Pay-per-invocation)과 무한대 오토스케일링을 가능케 한다.

### 3. 심층 동작 원리: 콜드 스타트 (Cold Start)
- FaaS의 가장 큰 기술적 한계다. 
- 함수가 한동안 호출되지 않으면 클라우드 벤더는 자원 절약을 위해 해당 컨테이너를 죽인다. 이후 새로운 요청이 오면, 컨테이너를 새로 만들고 언어 런타임(Java, Node.js)을 로딩하는 데 1~3초의 지연 시간이 발생한다. 이를 **콜드 스타트**라 하며, Java나 C# 같은 무거운 언어일수록 심하다.

📢 **섹션 요약 비유**: 평소에는 스위치를 끄고 자다가(비용 절감), 누군가 벨을 누르면(이벤트) 부스스 일어나 옷을 챙겨 입고 나가는 시간(콜드 스타트 지연)이 존재하는 구조입니다.

## Ⅲ. 융합 비교 및 기술 분석

### 1. CaaS (컨테이너/K8s) vs FaaS (서버리스) 상세 비교

| 비교 항목 | CaaS (Container as a Service) | FaaS (Function as a Service) |
|:---:|:---|:---|
| **기본 단위** | 컨테이너 (Docker) 묶음 | **단일 함수 (Code Snippet)** |
| **상태 유지** | Stateful 및 장기 실행(배치) 가능 | **Stateless 필수, 최대 실행 시간 제한 (예: 15분)** |
| **운영 부담** | K8s 클러스터 관리 및 스케일링 설정 필요 | **운영 부담 제로 (벤더가 100% 관리)** |
| **비용 구조** | 서버가 띄워진 '시간' 기반 과금 | **함수가 '호출된 횟수' 및 '실행 밀리초' 기반 과금** |
| **확장 속도** | 수 초 ~ 수 분 | **수 밀리초 단위로 수만 개 동시 복제** |

### 2. 과목 융합 관점 (소프트웨어 공학 & 운영체제)
- **소프트웨어 공학 (마이크로서비스 극대화)**: MSA를 넘어, 하나의 API 엔드포인트가 하나의 함수와 매칭되는 나노 서비스(Nano-services) 아키텍처로 진화하게 만든다.
- **운영체제 (MicroVM / Firecracker)**: 클라우드 벤더는 하나의 강력한 물리 서버에서 수만 명 고객의 함수를 안전하게 격리 실행하기 위해, KVM 기반의 초경량 가상화 기술인 **MicroVM(예: AWS Firecracker)**을 운영체제 커널 레벨에서 융합하여 0.1초 만에 안전한 실행 환경을 찍어낸다.

📢 **섹션 요약 비유**: K8s(CaaS)가 '요리사 팀을 고용해 우리 주방에서 관리하는 것'이라면, FaaS는 '배달의 민족으로 음식만 주문하는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오: 미디어 썸네일 자동 생성 시스템
- **상황**: 사용자가 인스타그램 같은 앱에 사진 원본(5MB)을 올리면, 즉시 3가지 크기의 썸네일(모바일용, 웹용, 섬네일용)을 만들어야 함. 트래픽은 예측 불가능함.
- **적용**: **AWS Lambda 및 S3 기반 서버리스 아키텍처**.
  1. 사용자가 원본 사진을 **S3 버킷**에 업로드.
  2. 업로드 완료 이벤트가 **Lambda 함수**를 즉시 트리거.
  3. 함수가 사진을 3가지 사이즈로 리사이징하여 다시 S3에 저장.
- **결과**: 별도의 이미지 처리 서버 24시간 운영 없이, 업로드될 때만 1초 미만으로 과금되어 인프라 비용 95% 절감. 트래픽이 100배 폭증해도 Lambda가 자동 스케일링되어 지연 없음.

### 2. 기술사적 판단 (Insight)
- **콜드 스타트 방어 설계**: 1초의 지연이 치명적인 사용자 로그인 API 같은 곳에 FaaS를 쓰면 UX가 망가진다. 기술사는 언어를 콜드 스타트가 빠른 **Node.js, Go, Rust**로 변경하거나, 클라우드 벤더가 제공하는 **'프로비저닝된 동시성(Provisioned Concurrency - 미리 컨테이너를 켜두는 기능)'**을 활성화하여 콜드 스타트를 아키텍처 단에서 제거해야 한다.
- **Vendor Lock-in 및 테스트의 어려움**: FaaS는 특정 클라우드 벤더의 생태계(AWS S3, DynamoDB 등)와 깊게 결합된다. 또한 로컬 PC에서 테스트하기가 매우 까다롭다. 기술사는 **Serverless Framework** 같은 추상화 도구를 사용하여 이식성을 확보하고 로컬 디버깅 환경을 구축해야 한다.

📢 **섹션 요약 비유**: 서버리스는 완벽해 보이지만, '장거리 마라톤(오래 걸리는 작업)'을 시키면 중간에 쓰러지고(시간 초과), 벤더의 입맛(Lock-in)에 맞게 편식한다는 치명적 약점을 설계 시 반드시 고려해야 합니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성적 기대효과
- **TCO(총소유비용) 혁신**: 간헐적 트래픽 워크로드의 경우, 항상 켜두는 IaaS 대비 인프라 비용을 최대 80~90% 절감 가능.
- **NoOps (운영 제로) 실현**: 인프라 관리팀의 역할을 애플리케이션 개발자가 흡수하여(DevOps의 진화), 팀의 모든 리소스를 비즈니스 가치 창출에만 쏟을 수 있음.

### 2. 미래 전망
- **서버리스 컨테이너의 융합**: FaaS의 실행 시간 제약(15분)과 상태 유지 불가 단점을 극복하기 위해, 컨테이너 자체를 서버리스로 띄우는 **AWS Fargate, Google Cloud Run**과 같은 기술이 주류로 자리 잡으며 CaaS와 FaaS의 경계가 허물어지고 있다.
- **엣지 서버리스 (Edge FaaS)**: AWS Lambda@Edge나 Cloudflare Workers처럼, 함수가 중앙 데이터센터가 아닌 전 세계에 흩어진 CDN 엣지 노드에서 실행되어 사용자와의 물리적 거리를 0으로 만드는 초저지연 서버리스로 진화할 것이다.

📢 **섹션 요약 비유**: 서버리스(FaaS)는 코딩의 귀찮은 잡일(인프라)을 구름 속으로 완전히 치워버리고, 개발자에게 오직 '순수한 창조(로직)'의 즐거움만 남겨준 클라우드 혁명의 클라이맥스입니다.

+++

### 📌 관련 개념 맵
- **IaaS / PaaS / SaaS**: FaaS가 속해 있는 클라우드 서비스 모델의 거대한 계보.
- **클라우드 네이티브**: FaaS를 활용하여 100% 클라우드의 이점을 뽑아내는 개발 사상.
- **마이크로서비스 (MSA)**: FaaS가 아주 작은 단위로 쪼개어 구현하고자 하는 아키텍처.
- **프로비저닝된 동시성 (Provisioned Concurrency)**: FaaS의 가장 큰 약점인 콜드 스타트를 돈으로 해결하는 기능.

### 👶 어린이를 위한 3줄 비유 설명
1. **서버가 사라졌어요**: 복잡한 컴퓨터(서버)를 내가 직접 관리할 필요 없이, 요리 레시피(코드)만 인터넷에 올려두면 끝나는 마법이에요.
2. **동전 넣고 쓰는 컴퓨터**: 24시간 내내 켜놔서 돈이 줄줄 새는 게 아니라, 손님이 내 버튼을 누른 그 짧은 1초 동안만 요금이 나가는 아주 알뜰한 시스템이랍니다.
3. **천 명의 요리사 복제**: 갑자기 손님이 1만 명 몰려와도 걱정 없어요! 구름(클라우드) 아저씨가 순식간에 요리사를 1만 명으로 복제해서 음식을 다 만들어주고 다시 사라진답니다.
