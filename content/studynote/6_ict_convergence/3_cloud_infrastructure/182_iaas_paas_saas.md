# 클라우드 서비스 모델 (IaaS, PaaS, SaaS): IT 자원 관리 책임의 점진적 위임

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클라우드 서비스 모델은 제공자(CSP, Cloud Service Provider)와 사용자 간의 'IT 인프라 관리 책임'을 어느 계층(Layer)까지 나눌 것인지에 따라 IaaS, PaaS, SaaS로 구분되는 아키텍처 분류 체계다.
> 2. **가치**: 사용자는 핵심 비즈니스 로직(Application) 개발에만 집중하고, 네트워크, 스토리지, 서버, 심지어 운영체제(OS)와 런타임 환경까지의 번거로운 유지보수를 클라우드 벤더에게 아웃소싱하여 개발 민첩성(Agility)을 극대화할 수 있다.
> 3. **융합**: 전통적인 3대 모델을 넘어, 모바일/웹 앱의 백엔드를 통째로 제공하는 BaaS와 함수 단위로 실행되는 서버리스 기반의 FaaS로 진화하며, 기업의 마이크로서비스(MSA) 전환을 가속화하고 있다.

---

## Ⅰ. 클라우드 서비스 모델 개요

### 1. 정의
- 클라우드 컴퓨팅 환경에서 사용자에게 제공되는 IT 자원의 추상화(Abstraction) 수준과 제어 권한의 범위를 규정한 표준 모델(NIST 기준).
- 'Pizza as a Service' 비유로 유명하며, 집에서 피자를 직접 만드는 것(온프레미스)부터, 냉동 피자를 사서 오븐에 굽는 것(IaaS), 피자 배달을 시키는 것(PaaS), 식당에 가서 사 먹는 것(SaaS)에 이르는 책임의 스펙트럼이다.

### 💡 비유: "집 짓기와 인테리어의 권한 위임"
- **On-Premise**: 내가 직접 땅을 사고, 철근을 세우고, 시멘트를 바르고, 가구를 넣고 요리까지 모두 다 해야 하는 **'자가 건축'**.
- **IaaS**: 뼈대와 벽만 있는 **'빈 창고 임대'**. 전기와 수도(서버, 네트워크)는 들어오지만, 벽지(OS)부터 가구(데이터)까지 내가 다 꾸며야 합니다.
- **PaaS**: 풀옵션 가구와 조리기구가 완비된 **'레지던스 호텔 임대'**. 나는 요리 재료(애플리케이션 코드)만 들고 가서 요리만 하면 됩니다.
- **SaaS**: 요리까지 다 되어서 나오는 **'최고급 레스토랑'**. 나는 자리에 앉아 맛있게 먹기(소프트웨어 사용)만 하면 됩니다.

### 3. 등장 배경
- **개발 생산성(Time-to-Market) 요구**: 서버 하드웨어를 구매하고 OS를 세팅하는 데 수주가 걸리는 과거 방식으로는, 하루 단위로 앱을 배포해야 하는 스타트업의 속도를 맞출 수 없음.
- **클라우드 벤더의 서비스 고도화**: 아마존(AWS)이 처음엔 가상 서버(IaaS)만 빌려주다가, 점차 DB, 머신러닝 환경 등 고부가가치 플랫폼(PaaS)을 얹어 팔기 시작하며 시장이 세분화됨.

📢 **섹션 요약 비유**: IT라는 거대한 짐가방에서, 내가 들고 갈 핵심(코드)만 남기고 나머지 무거운 짐(인프라)을 남에게 돈을 주고 맡기는 '점진적 짐 덜어내기' 모델입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 공동 책임 모델 (Shared Responsibility Model)

| 계층 (Stack) | On-Premise | IaaS (Infra) | PaaS (Platform) | SaaS (Software) |
|:---|:---:|:---:|:---:|:---:|
| **Applications (앱)** | User | User | User | **Vendor** |
| **Data (데이터)** | User | User | User | **Vendor** |
| **Runtime (실행 환경)**| User | User | **Vendor** | **Vendor** |
| **Middleware (미들웨어)**| User | User | **Vendor** | **Vendor** |
| **O/S (운영체제)** | User | User | **Vendor** | **Vendor** |
| **Virtualization (가상화)**| User | **Vendor** | **Vendor** | **Vendor** |
| **Servers (서버)** | User | **Vendor** | **Vendor** | **Vendor** |
| **Storage (스토리지)**| User | **Vendor** | **Vendor** | **Vendor** |
| **Network (네트워크)**| User | **Vendor** | **Vendor** | **Vendor** |

*(User = 사용자 관리 책임, Vendor = 클라우드 사업자 관리 책임)*

### 2. 서비스 모델별 대표 사례 및 구조 (ASCII)

**[다이어그램 해설]** 
**IaaS**는 하이퍼바이저(가상화)까지만 벤더가 제공한다. 사용자는 그 위에 올라갈 OS(리눅스/윈도우)를 직접 고르고 패치해야 한다. **PaaS**는 OS와 Java, Node.js 같은 런타임 환경까지 벤더가 세팅해 준다. 개발자는 오직 코드만 올려서(Deploy) 실행하면 된다. **SaaS**는 이미 완성된 소프트웨어(예: 구글 문서도구, 세일즈포스)를 브라우저로 접속해 사용하는 형태로, 인프라나 코드에 대한 접근 권한이 아예 없다.

### 3. 심층 동작 원리: BaaS (Backend as a Service) 와 FaaS (Function as a Service)
- **BaaS (예: Firebase)**: 모바일 앱을 만들 때 회원가입, 푸시 알림, 데이터베이스 등 반복적으로 필요한 백엔드(서버) 기능을 API 형태로 가져다 쓰는 모델이다.
- **FaaS (예: AWS Lambda)**: 서버리스(Serverless)의 핵심으로, 서버를 띄워두지 않고 '특정 이벤트(클릭, 파일 업로드)'가 발생했을 때만 코드가 담긴 함수(Function)가 아주 잠깐 실행되고 사라지는 궁극의 PaaS 진화형이다.

📢 **섹션 요약 비유**: 내가 요리사(개발자)라면, 재료를 직접 씻고 다듬는 것(IaaS)보다 씻겨진 밀키트(PaaS)를 쓰면 요리에만 집중할 수 있고, 손님(사용자) 입장에서는 그냥 식당(SaaS)에 가는 게 최고입니다.

## Ⅲ. 융합 비교 및 기술 분석

### 1. IaaS vs PaaS vs SaaS 선택 기준

| 비교 항목 | IaaS (인프라) | PaaS (플랫폼) | SaaS (소프트웨어) |
|:---:|:---|:---|:---|
| **타겟 고객** | 시스템 엔지니어 (Infra) | 앱 개발자 (Developer) | 일반 비즈니스 사용자 (End User) |
| **유연성/통제권** | **매우 높음 (OS 마음대로 조작)** | 중간 (런타임 제약 있음) | 매우 낮음 (설정만 변경 가능) |
| **주요 사용 사례** | 기존 온프레미스 시스템 이전 (Lift & Shift) | 신규 모바일/웹 앱 빠른 개발 | 그룹웨어, ERP, CRM 도입 |
| **벤더 종속성(Lock-in)**| 상대적으로 낮음 (이전 쉬움) | **높음 (특정 벤더 DB/API 종속됨)** | 매우 높음 (데이터 빼기 힘듦) |

### 2. 과목 융합 관점 (소프트웨어 공학 & 보안)
- **클라우드 네이티브 (Cloud Native)**: 기존 코드를 그대로 IaaS 서버에 올리는 방식(Rehost)은 클라우드의 이점을 살리지 못한다. 코드를 마이크로서비스로 쪼개고 컨테이너(Docker)로 감싸 PaaS나 FaaS 위에 올리는 **리팩토링(Refactor)**이 현대 소프트웨어 공학의 필수 융합이다.
- **SaaS 보안 (CASB)**: 직원들이 회사 기밀 문서를 노션이나 슬랙 같은 외부 SaaS에 마구 올리는 것을 통제하기 위해, 클라우드 접근 보안 중개(CASB, Cloud Access Security Broker) 솔루션이 기업 보안의 필수품으로 떠올랐다.

📢 **섹션 요약 비유**: 자유(IaaS)를 누리려면 그만큼의 책임(관리)을 져야 하고, 편안함(PaaS, SaaS)을 원한다면 규칙(벤더 종속)에 순응해야 하는 철저한 기브 앤 테이크입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오: 엔터프라이즈 하이브리드 전략
- **상황**: 대형 은행이 노후화된 IT 인프라를 클라우드로 이전하려 하나, 핵심 고객 데이터베이스는 규제상 퍼블릭 클라우드 반출이 불가함.
- **적용**: **IaaS / PaaS 하이브리드 믹스 전략**.
  - **프런트엔드 (웹/앱 서버)**: 트래픽 변동이 심하므로 AWS의 컨테이너 기반 **PaaS** (EKS 등)에 올려 오토스케일링 적용.
  - **백엔드 (레거시 연동)**: 기존 리눅스 서버 환경과 똑같이 맞춰야 하는 내부 결제 모듈은 AWS EC2(**IaaS**)에 그대로 마이그레이션(Lift & Shift).
  - **데이터베이스 (DB)**: 사내 전산실(On-premise)에 남겨두고, 클라우드와 전용선(AWS Direct Connect)으로 연결.
- **결과**: 보안 규제를 100% 준수하면서도, 모바일 뱅킹 접속 폭주 시 서버 다운을 막는 탄력성 확보.

### 2. 기술사적 판단 (Insight)
- **Vendor Lock-in 딜레마**: AWS의 DynamoDB(PaaS)나 Lambda(FaaS)를 떡칠해서 앱을 짜면 개발은 10배 빨라지지만, 나중에 구글 클라우드(GCP)로 이사 가려 할 때 코드를 처음부터 다시 짜야 하는 참사가 발생한다. 기술사는 속도가 생명인 스타트업에는 **PaaS/FaaS**를 권장하되, 장기적 비용과 독립성이 중요한 엔터프라이즈에는 **IaaS 기반의 쿠버네티스(K8s) 자체 구축**을 제안하는 아키텍처적 결단을 내려야 한다.
- **Shared Responsibility (공동 책임의 함정)**: 많은 기업이 "IaaS(AWS)를 샀으니 해킹당하면 아마존 책임이겠지"라고 착각한다. 클라우드 벤더는 '데이터센터의 철조망(물리적 보안)'만 책임질 뿐, 방화벽 포트를 실수로 열어놔서 해킹당한 책임은 100% 사용자에게 있다. 

📢 **섹션 요약 비유**: 클라우드 벤더가 제공하는 꿀(PaaS)을 너무 많이 먹으면 그들의 둥지에서 평생 벗어날 수 없는 꿀벌이 됩니다. 적절한 거리 두기(IaaS+컨테이너)가 장기적 생존 전략입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성적 기대효과
- **IT 인력의 고도화**: 서버 랜선을 꼽고 OS 패치를 하던 단순 인프라 인력들이, 애플리케이션의 비즈니스 로직과 클라우드 아키텍처를 설계하는 고급 인력으로 전환됨.
- **SaaS 구독 경제**: 소프트웨어 불법 복제가 사라지고, 월 구독료를 통해 기업의 안정적이고 지속 가능한 현금 흐름(Cash Flow)을 창출.

### 2. 미래 전망
- **XaaS (Everything as a Service)**: AI 모델 자체를 API로 빌려 쓰는 **MaaS (Model as a Service)**, 데스크톱 PC 화면 자체를 스트리밍하는 **DaaS (Desktop as a Service)** 등 세상의 모든 하드웨어와 소프트웨어가 '서비스화'되어 구독 경제로 편입될 것이다.
- **엣지(Edge)의 역습**: 모든 것을 중앙 클라우드(IaaS/PaaS)로 모으던 트렌드에서, 지연 시간을 줄이기 위해 다시 기기 단말이나 지역 기지국으로 서비스 모델을 분산시키는 분산형 클라우드 패러다임이 부상 중이다.

📢 **섹션 요약 비유**: 클라우드 서비스 모델은 복잡한 IT 기술을 전기나 수도처럼 벽에 꽂기만 하면 나오는 '보편적 유틸리티'로 탈바꿈시킨 21세기 최고의 비즈니스 발명품입니다.

- **[마이크로서비스 (MSA)](../../4_software_engineering/xx_msa.md)**: 클라우드 환경(PaaS)의 장점을 100% 뽑아내기 위한 소프트웨어 아키텍처.

### 👶 어린이를 위한 3줄 비유 설명
1. **내 맘대로 빈방 (IaaS)**: 텅 빈 튼튼한 방을 빌려주는 거예요. 벽지도 내가 바르고 가구도 직접 사 와야 하지만 내 맘대로 다 꾸밀 수 있어요.
2. **가구 풀옵션 방 (PaaS)**: 예쁜 침대와 책상이 다 준비된 방이에요. 나는 그냥 내 옷과 장난감(앱 코드)만 들고 들어가서 살면 돼서 아주 편해요.
3. **뷔페 식당 (SaaS)**: 아예 요리사가 다 만들어 놓은 음식을 숟가락만 들고 가서 맛있게 먹고 나오는 곳이에요. (예: 넷플릭스, 구글 문서)
