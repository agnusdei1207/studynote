# MQTT: 초경량 Pub/Sub 구조로 만물을 잇는 사물인터넷 메시징 표준

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MQTT는 대역폭이 제한되고 신뢰성이 낮은 네트워크 환경에서 동작하는 IoT 기기를 위해 설계된 TCP 기반의 초경량(Lightweight) 퍼블리시/서브스크라이브(Pub/Sub) 메시징 프로토콜이다.
> 2. **가치**: 클라이언트 간의 직접 연결 없이 '브로커(Broker)'를 통해 메시지를 중계함으로써, 수만 개의 센서가 네트워크 상태와 무관하게 비동기적으로 데이터를 주고받을 수 있는 극강의 확장성(Scalability)을 제공한다.
> 3. **융합**: 서비스 품질(QoS) 레벨 0, 1, 2를 지원하여 데이터의 중요도에 따라 전송 보장 수준을 선택할 수 있으며, 클라우드 플랫폼(AWS IoT, Azure IoT)의 기본 수집 프로토콜로 사용된다.

## Ⅰ. MQTT (Message Queuing Telemetry Transport) 개요

### 1. 정의
- IBM과 Eurotech가 공동 개발하고 현재 OASIS 표준으로 지정된 사물인터넷용 경량 메시징 프로토콜이다.
- HTTP의 무거운 헤더(Header) 구조를 버리고, 최소 2바이트의 고정 헤더만을 사용하여 배터리 전력과 네트워크 대역폭 소모를 극소화했다.

### 💡 비유: "우체국 사서함과 구독 잡지"
- **기존의 통신(HTTP)**이 내가 직접 친구 집에 찾아가서 **"혹시 편지 온 거 있어?"라고 계속 물어보는(Polling) 방식**이라면,
- **MQTT**는 **우체국(브로커)에 내 전용 사서함(토픽)을 만들어두고 "잡지가 도착하면 우리 집으로 배달해 줘"라고 구독(Subscribe) 신청을 해두는 것**과 같습니다. 잡지사(퍼블리셔)가 우체국에 잡지를 보내면 우체국이 알아서 구독자들에게 쫙 뿌려주기 때문에, 구독자는 하루 종일 기다리거나 계속 물어볼 필요가 없습니다.

### 3. 등장 배경
- **열악한 네트워크 환경**: 사막의 송유관 감시 센서 등 연결이 수시로 끊기고 대역폭이 좁은 위성 통신 환경에서 쓸 가벼운 규격이 필요.
- **다대다(M:N) 통신의 복잡성 증가**: 수많은 센서와 수많은 서버가 1:1로 직접 연결(Point-to-Point)되는 구조는 시스템을 복잡하게 만들고 확장을 가로막음.

📢 **섹션 요약 비유**: 수만 명의 사람들이 얽혀있는 복잡한 밧줄을 끊어내고, 중앙의 교통순경(브로커)에게 모든 길 안내를 맡겨 도로(네트워크)의 정체를 뚫어낸 혁신입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. MQTT 3대 핵심 구성 요소

| 구성 요소 | 역할 | 상세 동작 | 비유 |
|:---:|:---|:---|:---|
| **Publisher (발행자)** | 메시지 생성 | 센서 데이터를 측정하여 특정 '토픽'의 이름표를 붙여 브로커에 전송. | 신문사 |
| **Broker (브로커)** | 메시지 중계 | 발송된 메시지를 받아, 해당 토픽을 구독하는 모든 기기에 전달. | 우체국 / 분류 센터 |
| **Subscriber (구독자)** | 메시지 수신 | 브로커에게 특정 토픽을 구독하겠다고 등록한 뒤, 메시지가 오면 수신. | 정기 구독자 |

### 2. Pub/Sub 아키텍처 및 토픽 필터링 (ASCII)

```text
┌─────────────────────────────────────────────────────────────┐
│                 [ MQTT Pub/Sub ARCHITECTURE ]               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [ Sensor Node ]                            [ Mobile App ] │
│   (Publisher)                                (Subscriber)   │
│        │                                           ▲        │
│        │ 1. Publish:                               │        │
│        │ Topic: "home/livingroom/temp"             │ 3. Forward Message │
│        │ Payload: "24.5C"                          │        │
│        ▼                                           │        │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                     MQTT BROKER                       │  │
│  │  (Topic Registry & Message Routing Engine)            │  │
│  │                                                       │  │
│  │   Topic: "home/livingroom/temp" -> [Mobile App, DB]   │  │
│  │   Topic: "home/+/temp"          -> [BigData Server]   │  │
│  └──────────────────────────┬────────────────────────────┘  │
│                             │                               │
│        ▲ 2. Subscribe       │ 4. Forward Message            │
│        │  Topic: "home/#"   ▼                               │
│  ┌─────────────────────────────┐                            │
│  │   [ Cloud DB Server ]       │                            │
│  │   (Subscriber)              │                            │
│  └─────────────────────────────┘                            │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 
발행자(센서)는 자신이 만든 데이터를 누가 받을지 신경 쓰지 않고 브로커로 보낸다. 구독자(앱, DB) 역시 누가 데이터를 만들었는지 모르며 브로커에게 특정 토픽(`home/livingroom/temp`)만 달라고 요청한다. 이때 **와일드카드(+, #)**를 사용하여 여러 토픽을 한 번에 구독할 수도 있다. (`+`는 단일 레벨 통과, `#`는 하위 모든 레벨 통과). 이 완벽한 **비동기성(Asynchrony)**과 **디커플링(Decoupling)**이 MQTT의 심장이다.

### 3. 심층 동작 원리: 서비스 품질 (QoS, Quality of Service)
네트워크 상태가 나쁠 때 데이터 유실을 막기 위해 브로커와 클라이언트 간의 메시지 전송 보장 수준을 3단계로 정의한다.
- **QoS 0 (At most once)**: "최대 한 번". 메시지를 보내고 끝. 유실될 수 있음 (가장 빠름, 센서 데이터).
- **QoS 1 (At least once)**: "최소 한 번". 수신 확인(PUBACK)이 올 때까지 계속 보냄. 중복 수신 가능.
- **QoS 2 (Exactly once)**: "정확히 한 번". 4단계 핸드셰이크(PUBREC, PUBREL, PUBCOMP)를 통해 중복과 유실을 완벽히 차단 (가장 느림, 결제/제어 데이터).

📢 **섹션 요약 비유**: 전단지를 문 앞에 대충 던져두고 가기(QoS 0), 수령 사인을 받을 때까지 계속 찾아가기(QoS 1), 신분증을 확인하고 영수증까지 꼼꼼히 챙겨주기(QoS 2)의 차이입니다.

## Ⅲ. 융합 비교 및 기술 분석

### 1. MQTT vs HTTP 상세 비교

| 비교 항목 | MQTT | HTTP / REST API |
|:---:|:---|:---|
| **통신 모델** | Pub/Sub (비동기, 이벤트 기반) | Request/Response (동기, 클라이언트 주도) |
| **연결 지속성** | 연결 유지 (Stateful, TCP Keep-alive) | 단발성 연결 (Stateless) |
| **헤더 크기** | 최소 2바이트 (극도로 가벼움) | 수백 바이트 이상 (무거움) |
| **배터리 소모** | 매우 적음 | 많음 |
| **1:N 데이터 분배**| 브로커가 자동으로 N명에게 배포 | 서버가 N번의 응답을 개별 처리해야 함 |

### 2. 과목 융합 관점 (네트워크 & 메시지 큐)
- **네트워크 (L4 연결)**: 물리적으로는 **TCP/IP** 위에서 동작하며, 연결을 유지하기 위해 주기적으로 PINGREQ/PINGRESP 패킷(Keep-alive)을 주고받아 방화벽의 세션 단절을 방지한다.
- **메시지 큐 (Kafka 연동)**: 수백만 개의 IoT 기기가 보내는 MQTT 메시지를 클라우드 브로커가 받은 후, 대용량 분산 처리 시스템인 **Apache Kafka**로 넘겨 빅데이터 파이프라인을 완성하는 아키텍처가 업계 표준이다.

📢 **섹션 요약 비유**: HTTP가 '주문할 때마다 매번 전화를 걸어 메뉴를 말하는 것'이라면, MQTT는 '전용 회선을 열어두고 언제든 짧게 신호만 보내는 무전기'와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오: 커넥티드 카(Connected Car) 원격 제어
- **상황**: 수만 대의 자동차에서 실시간 상태(GPS, 배터리)를 수집하고, 앱으로 문 열림/시동 켜기 명령을 내려야 함. 차량은 터널에 들어가면 네트워크가 끊김.
- **적용**: **MQTT 아키텍처 설계**.
  - 상태 데이터 수집: `QoS 0`으로 설정하여 부하 최소화.
  - 차량 제어 명령: `QoS 1` 또는 `QoS 2`로 설정하여 명령 누락 방지.
  - **LWT (Last Will and Testament)**: 차량이 터널에서 예기치 않게 끊기면, 브로커가 미리 설정된 유언(LWT) 메시지를 관제 센터에 발행하여 "차량 연결 끊김"을 즉시 알림.
- **결과**: 신뢰성 높은 차량 원격 제어 및 네트워크 단절에 대한 완벽한 예외 처리 구현.

### 2. 기술사적 판단 (Insight)
- **브로커의 확장성(SPOF 방지)**: 모든 통신이 브로커를 거치므로, 브로커가 죽으면 전체 시스템이 마비된다. 기술사는 로드 밸런서(L4) 뒤에 **RabbitMQ, EMQX, Mosquitto** 등의 엔터프라이즈급 브로커 클러스터링을 구성하여 수평적 확장성을 확보해야 한다.
- **보안 취약점**: MQTT 자체는 암호화를 제공하지 않으므로 패킷 스니핑에 취약하다. 실무에서는 반드시 **TLS/SSL(MQTTS - 포트 8883)** 기반의 종단간 암호화를 적용하고, 클라이언트 인증서(X.509) 기반 접근 제어를 구현해야 한다.

📢 **섹션 요약 비유**: 우체국(브로커)이 하나만 있으면 마비되므로 여러 우체국을 짓고 연결(클러스터링)하며, 중요한 편지는 금고(TLS)에 담아 보내는 철저한 설계가 필요합니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성적 기대효과
- **트래픽 및 전력 절감**: 기존 HTTP 대비 데이터 대역폭을 1/10 수준으로 줄이고 기기 배터리 수명을 극대화.
- **비동기 유연성**: 데이터 생성자와 소비자가 서로 몰라도 동작하므로 새로운 서비스(앱) 추가 시 센서 코드를 수정할 필요가 없는 유연성 확보.

### 2. 미래 전망
- **MQTT-SN (for Sensor Networks)**: TCP조차 부담스러운 초소형 센서를 위해 UDP 기반으로 더욱 경량화된 버전 확장.
- **MQTT v5.0**: 사용자 속성(User Properties), 공유 구독(Shared Subscriptions) 등 클라우드 네이티브 환경에 맞춘 엔터프라이즈 기능 대폭 강화.

📢 **섹션 요약 비유**: MQTT는 수억 개의 사물들이 동시에 말해도 절대 엉키지 않게 만들어주는 사물인터넷 세상의 '완벽한 교통망'입니다.

+++

### 📌 관련 개념 맵
- **CoAP (Constrained Application Protocol)**: MQTT의 영원한 라이벌인 UDP/REST 기반 경량 프로토콜.
- **TCP/IP**: MQTT가 기반으로 하는 신뢰성 있는 전송 계층.
- **Kafka (Apache Kafka)**: MQTT 브로커 뒤에서 대규모 데이터를 받아주는 분산 스트리밍 플랫폼.
- **퍼블리시/서브스크라이브 패턴**: MQTT 아키텍처의 근본이 되는 소프트웨어 디자인 패턴.

### 👶 어린이를 위한 3줄 비유 설명
1. **우체국 사서함**: 내가 우체국에 내 이름이 적힌 상자를 만들어두고 "만화책이 오면 보내주세요~"라고 부탁해두는 기술이에요.
2. **알아서 척척**: 만화책 회사(센서)는 누가 책을 받는지 몰라도 일단 우체국(브로커)에 다 넘겨요. 그럼 우체국이 알아서 신청한 사람들에게 쫙 배달해 준답니다.
3. **편지 확인 도장**: 아주 중요한 편지는 "잘 받았어?"라고 우체국에서 몇 번이고 확인해서 절대 잃어버리지 않게 해주는 기능(QoS)도 있어요.
