## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Shadow Paging은 페이지 단위로 데이터를 복사하여 변경 시 원본 대신 사본을 수정하고, 커밋 시 포인터만 전환하는 복구 기법이다.
> 2. **가치**: 로그 기반 복구보다 복구 시간이 짧고, UNDO가 필요 없으며, 시스템 장애 시 즉시 복구 가능하다.
> 3. **융합**: SSD의 Copy-on-Write와 유사한 원리로, 버전 관리 시스템의 스냅샷과도接通 점이 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념
Shadow Paging은 페이지 단위로 동작하는 복구 기법으로, 데이터를 수정할 때 원본 페이지(Current Page)를 그대로 두고, 수정된 데이터를 사본 페이지(Shadow Page)에 기록한다. 커밋 시 원본 페이지에 대한 포인터를 사본 페이지로 전환하고, 장애 발생 시에는 사본 페이지 포인터를 사용하여 복구한다.

### 필요성
로그 기반 복구는 장애 발생 시 로그를 분석하여 UNDO/REDO를 수행해야 하므로 복구 시간이 길어질 수 있다. Shadow Paging은 이러한 분석 시간이 필요 없어 즉각적인 복구가 가능하다.

### 섹션 요약 비유
Shadow Paging은 文書の推敲と 같아서, 원본을 보존한 채修正版을 작성하고, 完成되면 원본을替换한다. 문제 발생 시 원본을 보면 된다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Shadow Paging 동작

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    Shadow Paging 동작 과정                                │
  ├───────────────────────────────────────────────────────────────────────┤
  │
  │   [페이지 테이블 구조]                                                │
  │
  │   ┌─────────────────────────────────────────────────────────────┐    │
  │   │  현재 페이지 테이블 (Current Page Table)                         │    │
  │   │  Logical Page 0 ──▶ Physical Page 3                         │    │
  │   │  Logical Page 1 ──▶ Physical Page 7                         │    │
  │   │  Logical Page 2 ──▶ Physical Page 5                         │    │
  │   └─────────────────────────────────────────────────────────────┘    │
  │
  │   [페이지 P2 수정 시]                                                │
  │
  │   원본 Physical Page 5 ──▶ Shadow Page 9 (사본 생성)                │
  │   Shadow Page 9을 수정                                                │
  │
  │   [커밋 완료 후]                                                    │
  │
  │   현재 페이지 테이블 업데이트:                                          │
  │   Logical Page 2 ──▶ Physical Page 9 (사본)                         │
  │   원본 Physical Page 5는 나중에 GC로 정리                            │
  │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Shadow Paging의 핵심은"원본 보존"과"포인터 전환"이다. 페이지 수정 시 원본 페이지를 그대로 두고 사본을 생성하여 수정한다. 커밋 시에는 현재 페이지 테이블의 포인터만 사본 페이지로 전환하면 즉시 복구가 완료된다. 장애 발생 시에는 페이지 테이블의 포인터만 확인하면 되고, 로그 분석이 필요 없다. 그러나缺点으로는 모든 수정에 대해 페이지 단위의 사본을 생성해야 하므로 스토리지 오버헤드가 크고, 쓰기放大效应가 발생할 수 있다.

### 섹션 요약 비유
Shadow Paging은 그림의 디지털 편집과 같다. 원본 레이어를 보존한 채副本 레이어를修正하고, 完成되면副本 레이어를可視화한다. 문제가 있으면 원본 레이어로 돌아가면 된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: Shadow Paging vs 로그 기반 복구

| 구분 | Shadow Paging | 로그 기반 복구 |
|:---|:---|:---|
| **복구 시간** | 거의 即時 | 로그 길이에 따라 수분~수시간 |
| **스토리지 오버헤드** | 페이지 단위 사본 | 로그 레코드 |
| **쓰기 overhead** | 페이지 전체 복사 | 변경된 필드만 기록 |
| ** implementação** | 복잡 | 비교적 단순 |

### 섹션 요약 비유
Shadow Paging vs 로그 기반 복구는 문서 편집기의"复制全部 새 문서"vs"변경사항만 기록"과 같다.前者은即時 복구 가능하지만 스토리지가 많이 들고, 後자는 스토리지는 적게 들지만 복구에 시간이 걸린다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

Shadow Paging은 쓰기보다 읽기가 많은 환경에서는 오히려 overhead가 클 수 있다. 따라서 대규모 배치 쓰기보다 짧은 트랜잭션에서 더 эффектив적으로使用될 수 있다.

### 섹션 요약 비유
Shadow Paging의误用は الكتابة过多的场合의"全部コピー"와같다. 1페이지만修正하는데 100페이지全部をコピー하면 효율이 떨어진다.
