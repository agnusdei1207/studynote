## 핵심 인사이트 (3줄 요약)
> 1. **본질**: UNDO는 실패한 트랜잭션을 이전 상태로 되돌리는 역방향 복구이고, REDO는 완료된 트랜잭션을 재실행하여 영구 저장하는 순방향 복구다.
> 2. **가치**: 시스템 장애의 유형(커밋 이전/이후)에 따라 적절한 복구 전략을 선택하여 데이터 무결성을 보장한다.
> 3. **융합**: UNDO는 타임머신과, REDO는 재실행 디버깅과 동일한 원리다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념
UNDO는 장애 발생 시 실패한 트랜잭션의 변경을 이전 상태로 되돌리는 연산이다. 트랜잭션이 부분 완료된 상태에서 장애가 발생하면, 로그에 기록된 이전 값(old_value)을 사용하여 원래 상태를 복원한다. REDO는 장애 발생 시 완료된 트랜잭션의 변경을 재실행하는 연산이다. 커밋 로그가 있지만 실제 데이터 변경이 디스크에 기록되지 않았을 경우, 로그의 이후 값(new_value)을 사용하여 변경을 재현한다.

### 필요성
시스템 장애 시 어떤 트랜잭션이 완료되었고 어떤 것이 실패했는지 판단해야 하며, 각각에 대해 적절한 복구 동작을 수행해야 한다. UNDO/REDO 없이는 장애 후 데이터 일관성을 보장할 수 없다.

### 비유
UNDO는"时光倒流"와 같아서, 실수하기 전 모습으로 돌아가는 것이고, REDO는"다시 하기"와 같아서, 제대로 한 일을再度実行하는 것이다.

### 섹션 요약 비유
UNDO/REDOは映画編集の"逆戻し"と"最初から 재생"の違いと同類です。问题是"どのシーンまで正常だったか"を把握することから始まります。

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### UNDO 동작

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    UNDO 동작 예시                                        │
  ├───────────────────────────────────────────────────────────────────────┤
  │
  │   [원본 로그]                                                        │
  │   <UPDATE, T1, X, old=100, new=200>  ← T1이 X를 100→200로 변경     │
  │
  │   [장애 상황]                                                        │
  │   - T1은 COMMIT 로그 없이 장애 발생                                   │
  │   → T1은 실패한 트랜잭션 → UNDO 필요                                 │
  │
  │   [UNDO 수행]                                                        │
  │   - 로그의 old_value=100을 X에 기록                                  │
  │   - <CLR, T1, X, 100> 기록 (Compensation Log Record)                │
  │
  │   [결과]                                                            │
  │   - X = 100 (이전 상태로 복원)                                        │
  │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** UNDO의 핵심은"이전 값으로 복원"이다. 실패한 트랜잭션의 로그 레코드에는 이전 값(old_value)이 기록되어 있으므로, 이를 사용하여 장애 발생 전 상태를 재현한다. UNDO 수행 자체도 로그에 기록되는데, 이를 CLR(Compensation Log Record)이라 한다. CLR을 기록하는 이유는 UNDO 도중再度장애가 발생했을 때,"여기까지 UNDO 완료"를 알려서 중복 UNDO를 방지하기 위함이다.

### REDO 동작

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    REDO 동작 예시                                        │
  ├───────────────────────────────────────────────────────────────────────┤
  │
  │   [원본 로그]                                                        │
  │   <UPDATE, T1, X, old=100, new=200>  ← T1이 X를 100→200로 변경     │
  │   <COMMIT, T1>                     ← T1 커밋 완료                    │
  │
  │   [장애 상황]                                                        │
  │   - T1은 COMMIT 완료, but 디스크에 X=200 미기록                      │
  │   → REDO 필요                                                        │
  │
  │   [REDO 수행]                                                        │
  │   - 로그의 new_value=200을 X에 기록                                  │
  │   - CLR은 불필요 (이미 COMMIT 완료)                                   │
  │
  │   [결과]                                                            │
  │   - X = 200 (커밋된 상태로 재현)                                       │
  │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** REDO의 핵심은"이후 값으로 재현"이다. 커밋 로그가 있다는 것은"이 트랜잭션은 완료되었음"을 의미하지만, 장애 발생 전 디스크에 실제 데이터가 기록되지 않았을 수 있다. 이 경우 로그의 이후 값(new_value)을 사용하여 변경을 재현한다. UNDO와 달리 REDO는 트랜잭션이 완료된 상태에서 실행되므로 CLR을 기록할 필요가 없다.

### UNDO+REDO 통합

| 상황 | 장애 시 동작 |
|:---|:---|
| **COMMIT 로그 없음** | UNDO |
| **COMMIT 로그 있음 + 데이터 미기록** | REDO |
| **COMMIT 로그 있음 + 데이터 기록됨** | 불필요 |

### 섹션 요약 비유
UNDO+REDO統合은 영화編集での"逆戻し"と"最初から 재생"の選択と 같다. 正常終了したシーンまで戻って(UNDO)、再び再生し(REDO)、最终的な正常状態 достичь.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: UNDO vs REDO

| 구분 | UNDO | REDO |
|:---|:---|:---|
| **방향** | 역방향 (이전 값) | 순방향 (이후 값) |
| **대상** | 실패 트랜잭션 | 완료 트랜잭션 |
| **CLR 필요 여부** | 필요 | 불필요 |
| **로그 레코드** | old_value 포함 | new_value 포함 |

### 섹션 요약 비유
UNDOは"書き込む 直前"の状態への戻りで、REDOは"書き込むべきだった"の状態への進んだ適用である.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 — BANKING 시스템**: 송금 트랜잭션 T1이 A계좌 차감(UNDO 대상)과 B계좌 입금(REDO 대상)을 모두 수행하고 COMMIT했다. 장애 발생 시:
- COMMIT 로그 있다 → B계좌 입금은 REDO
- A계좌 차감은 이미 디스크에 반영됨 (불필요)

### 섹션 요약 비유
送金色이 완료된 후의 장애は、「both A계좌 차감과 B계좌 입금이 완료」された場合と类似している. その 후の障害에서는 完了しなかった操作만 REDO하면 된다.

## Ⅴ. 결론

### 섹션 요약 비유
UNDO/REDO는 正座와 같다. 실수(실패 트랜잭션)하면 一歩下がり(UNDO), 제대로 한 일은 다시確認(REDO)하여、最終的な正常状態を維持する.

## 👶 어린이를 위한 3줄 비유 설명
1. UNDO는 **折纸の例え**で、比喩적으로言えば "やり直し" のrollbackで、失敗したら 以前的状态に戻す 것이다.
2. REDO는 **同じ道をもう一度歩く**と 같아서、目的地に到着できなかった(障害発生)場合、記録を基に 再び歩いて(REDO)、目的地に到達する 것이다.
3. 봇은 이 두 가지를 모두 사용하여、문제가 생겨도 이전 상태로 돌거나(UNDO), 다시 작업을 수행(REDO)하여最終結果를 完成시킨다!
