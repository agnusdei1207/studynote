# 425. 해시 인덱스(Hash Index) - O(1)의 속도

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해시 인덱스는 인덱스 키 값을 해시 함수(Hash Function)에 통과시켜 얻은 결과값(해시 코드)을 메모리의 **버킷(Bucket) 주소로 사용하여 데이터를 즉시 찾아가는 인덱스 기술**이다.
> 2. **가치**: 데이터 양에 상관없이 정확히 일치하는 값($=$ 연산)을 찾는 경우 **상수 시간($O(1)$)** 내에 결과를 반환하는 극강의 속도를 자랑한다.
> 3. **융합**: 충돌(Collision) 해결을 위한 체이닝(Chaining) 기법과 메모리 기반 DB 기술이 융합되어, 고속 캐싱이나 대용량 로그 분석의 포인트 쿼리 최적화에 사용된다.

+++

### Ⅰ. 해시 인덱스의 작동 원리

1. **해싱**: 키 값(예: 'Kim')을 입력하면 고정된 길이의 숫자(예: 5)를 뱉어냅니다.
2. **버킷 매핑**: 숫자 5에 해당하는 메모리 공간(버킷)을 즉시 조회합니다.
3. **충돌 처리**: 만약 'Lee'도 해시값이 5라면, 같은 버킷 안에 연결 리스트(Linked List)로 데이터를 매답니다. (Chaining)

+++

### Ⅱ. 해시 구조 및 체이닝 시각화 (ASCII Model)

```text
[ Hash Index & Bucket Chaining ]

  Key Value ──▶ [ Hash Function ] ──▶ Bucket Index
  
  (Index)    (Buckets)
  ┌───┐      ┌───────────────┐
  │ 0 │ ──▶  │               │
  ├───┤      ├───────────────┤
  │ 1 │ ──▶  │ [Data: Kim]   │──▶ [Data: Lee] ✅ (Chaining)
  ├───┤      ├───────────────┤
  │ 2 │ ──▶  │ [Data: Park]  │
  └───┘      └───────────────┘
```

+++

### Ⅲ. 특징 및 한계

- **장점**: 
    - 정확히 일치하는 값을 찾는 등가 조인(Equi-join)에서 B-Tree보다 훨씬 빠릅니다.
    - 구현이 상대적으로 단순하며 인메모리 환경에서 성능이 극대화됩니다.
- **한계**: 
    - **범위 검색(Range Scan) 불가**: 해시값은 순서가 없으므로 "어떤 값보다 큰 데이터"를 찾을 수 없습니다.
    - **부분 일치 검색 불가**: 키 전체를 해싱하므로 앞 글자만 보고 찾는 검색이 안 됩니다.
    - **해시 충돌**: 너무 많은 데이터가 한 버킷에 몰리면 성능이 급격히 저하됩니다 ($O(1) \to O(N)$).

- **📢 섹션 요약 비유**: 해시 인덱스는 **'번호 키 사물함'**과 같습니다. 내 이름(키)을 입력하면 즉시 몇 번 사물함(주소)인지 알려주는 기계와 같습니다. 사물함 번호만 알면 바로 달려가 물건을 꺼낼 수 있어 아주 빠르지만, "이름이 '김'씨로 시작하는 사물함들을 다 열어봐"라거나 "10번부터 20번까지 사물함을 순서대로 열어봐" 같은 명령은 수행할 수 없는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Collision]**: 서로 다른 키가 같은 해시값을 가질 때 발생하는 문제.
- **[Chaining]**: 충돌 발생 시 데이터를 줄줄이 엮어 보관하는 기법.
- **[Equality Search]**: 해시 인덱스가 유일하게 왕좌를 차지하는 검색 영역.

📢 **마무리 요약**: **Hash Index**는 특수 목적의 고속도로입니다. 범위 검색이 필요 없는 단순 조회 시스템에서 압도적인 효율을 발휘하는 핀포인트 인덱스 기술입니다.
