# 531. 2PC의 블로킹 문제 - 분산 합의의 아킬레스건

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 2PC(2단계 커밋)의 블로킹(Blocking) 문제는 트랜잭션 조율자인 **코디네이터가 장애를 일으켰을 때, 모든 참여 노드가 지시를 기다리며 해당 자원의 잠금(Lock)을 해제하지 못하고 무한 대기하는 현상**이다.
> 2. **가치**: 분산 시스템의 가용성(Availability)을 심각하게 훼손하는 요인으로 작용하며, 강한 일관성을 지키기 위해 지불해야 하는 가장 비싼 대가다.
> 3. **융합**: 이를 해결하기 위해 타임아웃 매커니즘을 강화한 3단계 커밋(3PC)이나, 최종적 정합성을 지향하는 비동기 기반의 Saga 패턴 등이 대안으로 융합되어 사용된다.

+++

### Ⅰ. 2PC 블로킹의 발생 시나리오

1. **준비 단계 완료**: 코디네이터가 모든 참여자(Node A, B)로부터 "Ready" 응답을 받았습니다.
2. **코디네이터 급사**: 조정자가 최종 "COMMIT" 명령을 노드들에게 보내기 직전에 전원이 꺼지거나 네트워크가 단절됩니다.
3. **참여자의 딜레마**: 노드 A와 B는 "준비는 끝났는데, 커밋을 해야 할지 롤백을 해야 할지" 모르는 상태로 남습니다.
4. **결과**: 다른 트랜잭션들이 해당 데이터를 쓰려 해도 락 때문에 모두 멈춰버리는 **시스템 마비**가 발생합니다.

+++

### Ⅱ. 블로킹 장애 구조 시각화 (ASCII Model)

```text
[ 2PC Blocking Disaster ]

  [ Coordinator ] ──▶ (Crash! 💥 After receiving READY)
         │
  ----------------------- (Silence) -----------------------
         │
  ┌──────┴──────────────┐
  ▼ [ Node A ]          ▼ [ Node B ]
  - Status: READY       - Status: READY
  - Lock: 🔒 Held       - Lock: 🔒 Held
  - Action: WAITING...  - Action: WAITING... ⌛️
  
  "누구도 스스로 결정할 수 없어 시스템 전체가 굳어버림"
```

+++

### Ⅲ. 블로킹 문제의 파급 효과

- **가용성 붕괴**: 특정 노드의 장애가 아닌, '조율자의 부재'가 클러스터 전체의 가동 중단으로 이어집니다. (SPOF 문제)
- **자원 고갈**: 대기 중인 트랜잭션들이 늘어나면서 커넥션 풀과 메모리가 고갈됩니다.
- **복구의 난해함**: 코디네이터가 복구되어 로그를 확인하기 전까지는 수동으로 락을 푸는 행위 자체가 데이터 불일치를 유발할 수 있어 매우 위험합니다.

- **📢 섹션 요약 비유**: 2PC 블로킹은 **'결혼식 도중 쓰러진 주례 선생님'**과 같습니다. 신랑과 신부가 "네!"라고 서약까지 마쳤는데(Prepare), 주례 선생님이 성혼 선포(Commit) 직전에 쓰러지신 상황입니다. 하객들(다른 트랜잭션)은 결혼식이 끝난 건지 아닌지 몰라 식장을 떠나지 못하고, 다른 커플들은 예식장을 쓰지 못한 채 문밖에서 하염없이 기다리는 아수라장이 되는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[SPOF (Single Point of Failure)]**: 코디네이터에게 집중된 장애 위험.
- **[3PC]**: 블로킹 문제를 해결하기 위해 '예비 커밋' 단계를 추가한 프로토콜.
- **[Quorum]**: 코디네이터 없이도 과반수 합의로 결정을 내리는 현대적 대안.

📢 **마무리 요약**: **2PC Blocking**은 분산 무결성의 어두운 면입니다. 일관성을 얻기 위해 가용성을 포기해야 하는 이 숙명을 이해할 때, 비로소 강건한 분산 아키텍처를 설계할 수 있습니다.
