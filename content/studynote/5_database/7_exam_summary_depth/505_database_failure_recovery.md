# 505. 데이터베이스 장애 유형과 복구 범위 - 위기 대응의 매뉴얼

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 장애는 그 범위와 심각도에 따라 **트랜잭션 장애, 시스템 장애, 미디어 장애**로 분류되며, 각 유형에 최적화된 복구 전략(Undo, Redo, Restore)이 요구된다.
> 2. **가치**: 장애의 성격을 정확히 진단하여 복구 시간(RTO)과 데이터 유실(RPO)을 최소화하며, 시스템의 영속성(Durability)을 물리적으로 증명한다.
> 3. **융합**: 로그 파일(Redo/Undo), 체크포인트, 백업 기술이 융합되어 어떤 재난 상황에서도 최신 일관된 상태로 데이터베이스를 되살리는 생존 아키텍처를 구성한다.

+++

### Ⅰ. 데이터베이스 장애 3대 유형 비교

| 장애 유형 | 발생 원인 | 영향 범위 | 복구 방법 |
|:---|:---|:---|:---|
| **트랜잭션 장애** | 논리적 오류, 입력값 위반 | 해당 트랜잭션 1개 | **Undo** (작업 취소) |
| **시스템 장애** | 정전, HW 고장, 커널 패닉 | 메모리 내 전체 데이터 증발 | **Redo + Undo** (로그 복구) |
| **미디어 장애** | 디스크 물리적 파손 | 데이터 파일 물리적 유실 | **Restore + Redo** (백업 복원) |

+++

### Ⅱ. 장애 상황 및 복구 로직 시각화 (ASCII Flow)

```text
[ Database Failure Spectrum ]

  (Minor) ◀─────────────────────────────────────────────▶ (Major)
  
  [ Transaction Fail ]     [ System Crash ]          [ Media Failure ]
  - SQL Error              - Power Outage            - Disk Head Damage
  - DB Alive ✅            - RAM Erased 💥           - File Deleted 💥
          │                        │                         │
          ▼                        ▼                         ▼
  [ Action: Undo ]         [ Action: ARIES ]         [ Action: Restore ]
  "하나만 취소"             "로그로 기억 복원"          "백업본으로 교체" ✅
```

+++

### Ⅲ. 유형별 복구 매커니즘 상세

1. **트랜잭션 장애**: 비즈니스 로직 상의 문제로 트랜잭션이 중단된 경우입니다. Undo 로그를 이용해 해당 트랜잭션이 바꾼 부분만 깨끗이 지우면 끝납니다.
2. **시스템 장애**: 전원이 꺼지면서 메모리(Buffer Pool)에만 써둔 데이터가 날아간 경우입니다. WAL 로그를 보고, 커밋된 건 다시 실행(Redo)하고 못한 건 취소(Undo)하여 맞춥니다.
3. **미디어 장애**: 가장 치명적인 상태입니다. 로그 파일까지 같은 디스크에 있었다면 복구가 불가능할 수 있으므로, 평소 외부 장치에 보관한 백업본(Dump)과 아카이브 로그를 가져와서 처음부터 재현해야 합니다.

- **📢 섹션 요약 비유**: 장애 유형은 **'부상의 정도'**와 같습니다. 트랜잭션 장애는 '손가락 가시 박힌 정도(가시만 뽑으면 됨)', 시스템 장애는 '깜빡 잠든 정도(일어나서 어제 일 기억해내기)', 미디어 장애는 '사고로 팔다리가 부러진 정도(새 부품으로 교체하고 재활 훈련하기)'에 비유할 수 있습니다. 상처에 맞는 치료법(복구 방식)을 골라야 환자(데이터)가 빨리 일어날 수 있습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[RTO (Recovery Time Objective)]**: 장애 발생 후 복구까지 걸리는 시간 목표.
- **[RPO (Recovery Point Objective)]**: 어느 시점의 데이터까지 살려낼 것인가에 대한 목표.
- **[Idempotency]**: 복구 작업을 여러 번 해도 결과가 같아야 한다는 복구의 철칙.

📢 **마무리 요약**: **Database Recovery Scope**는 복구 기술의 지도입니다. 평소 로그를 쌓고 백업을 챙기는 모든 행위가 바로 이 지도 위에서 시스템의 생명을 연장하는 숭고한 과정입니다.
