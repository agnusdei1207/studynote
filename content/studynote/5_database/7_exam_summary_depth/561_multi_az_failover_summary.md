# 561. 클라우드 DB 고가용성(Multi-AZ) - 자동 페일오버의 원리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클라우드 DB 고가용성은 물리적으로 격리된 **다중 가용 영역(Multi-AZ)에 주 서버(Primary)와 대기 서버(Standby)를 배치하고, 데이터를 실시간 동기 복제**하여 장애 발생 시 즉시 서비스를 전환하는 기술이다.
> 2. **가치**: 데이터 센터 화재나 네트워크 전면 장애 시에도 **RPO=0(데이터 유실 제로)**과 **낮은 RTO(빠른 복구)**를 달성하여 서비스 연속성을 보장한다.
> 3. **융합**: DNS CNAME 기반의 엔드포인트 자동 전환 기술과 헬스 체크(Health Check) 프로토콜이 융합되어, 애플리케이션 수정 없이도 매끄러운 **자동 페일오버(Failover)**를 실현한다.

+++

### Ⅰ. Multi-AZ 아키텍처의 작동 원리

1. **동기 복제 (Synchronous Replication)**: 사용자가 데이터를 쓸 때 주 서버와 대기 서버 모두에 기록이 완료되어야 성공으로 응답합니다. (정합성 보장)
2. **상시 감시 (Heartbeat)**: 클라우드 인프라가 주 서버의 건강 상태를 밀리초(ms) 단위로 체크합니다.
3. **자동 전환 (Failover)**: 주 서버 장애 감지 시, 대기 서버를 주 서버로 승격시키고 DB 주소(DNS)를 새 서버로 연결합니다.

+++

### Ⅱ. 자동 페일오버 매커니즘 시각화 (ASCII Model)

```text
[ Cloud Multi-AZ High Availability ]

      (User Application) ──▶ [ DNS Endpoint ] ✅
                                     │
          ┌──────────────────────────┴──────────────────────────┐
          ▼ (Active)                                           ▼ (Standby)
    [ Primary DB (AZ-1) ] ──────(Sync Replication)──────▶ [ Standby DB (AZ-2) ]
          │                                                    │
  (--- 💥 CRASH! ---)                                          │
          │                                                    │
          ▼ [ Failover Triggered ]                             ▼
    [ TERMINATED ]           [ DNS Pointing Update ] ──▶ [ New Primary ] 👑 ✅
```

+++

### Ⅲ. 실무적 고려사항: 성능과 정합성

- **Write Latency**: 동기 복제 방식이므로 네트워크 거리에 따른 쓰기 지연 시간이 발생할 수 있습니다. (가용 영역 간 고속 전용망 필수)
- **Zero-Downtime**: 애플리케이션 레벨에서 재시도(Retry) 로직과 커넥션 풀 설정을 적절히 관리해야 전환 시 찰나의 순간을 견딜 수 있습니다.
- **백업 자동화**: 대기 서버에서 스냅샷 백업을 수행함으로써 주 서버의 운영 부하를 완전히 분리할 수 있습니다.

- **📢 섹션 요약 비유**: 멀티 AZ 고가용성은 **'스페어 타이어를 장착한 자동차'**와 같습니다. 단순히 트렁크에 타이어(백업)를 싣고 다니는 게 아니라, 주행 중에 타이어가 터지자마자 차를 멈추지 않고 즉시 보조 타이어가 내려와서 주행을 이어가는 최첨단 시스템과 같습니다. 승객(사용자)은 차가 아주 잠깐 덜컹거리는 것(지연)은 느껴도 목적지까지 안전하게 도착할 수 있습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[RTO (Recovery Time Objective)]**: 장애 시 전환되는 데 걸리는 시간 (보통 수 분 이내).
- **[RPO (Recovery Point Objective)]**: 동기 복제 덕분에 0에 수렴하는 데이터 유실 시점.
- **[Read Replica]**: Multi-AZ Standby와 달리 읽기 분산 목적으로 비동기 복제되는 노드.

📢 **마무리 요약**: **Multi-AZ High Availability**는 클라우드 DB의 약속입니다. 물리적 한계를 넘어서는 이중화와 자동화된 전환 기술을 통해, 데이터는 어떤 시련 속에서도 서비스를 멈추지 않습니다.
