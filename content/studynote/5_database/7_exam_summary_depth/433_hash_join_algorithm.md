# 433. 해시 조인(Hash Join) - 대용량 조인의 지름길

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해시 조인은 작은 테이블(Build Input)을 메모리에 **해시 테이블(Hash Table)로 구축한 뒤, 큰 테이블(Probe Input)을 한 행씩 읽으며 해시 맵을 탐색**하여 짝을 찾는 고속 조인 방식이다.
> 2. **가치**: 인덱스가 없어도 대량의 데이터를 매우 빠르게 조인할 수 있으며, CPU 연산과 메모리 자원을 적절히 활용하여 **디스크 I/O 병목을 획기적으로 줄여준다.**
> 3. **융합**: '해시 함수'라는 수학적 기법과 '인메모리 버퍼' 기술이 융합된 결과물이며, 현대 RDBMS와 데이터 웨어하우스(DW)의 대규모 집계 및 통계 질의에서 사실상의 표준 조인 방식이 되었다.

+++

### Ⅰ. 해시 조인의 2단계 프로세스

1. **빌드 단계 (Build Phase)**: 
    - 두 테이블 중 더 작은 쪽(선행 테이블)을 읽어 메모리에 해시 테이블을 만듭니다. 
    - 조인 키 값을 해시 함수에 넣어 버킷 주소로 변환하여 데이터를 담습니다.
2. **프로브 단계 (Probe Phase)**: 
    - 더 큰 쪽(후행 테이블)을 한 행씩 읽으며 조인 키를 동일한 해시 함수에 넣습니다. 
    - 결과로 나온 주소의 해시 테이블 버킷을 즉시 확인하여 일치하는 데이터가 있는지 체크합니다.

+++

### Ⅱ. 해시 조인 아키텍처 시각화 (ASCII Flow)

```text
[ Hash Join Workflow ]

  1. Build Phase (Small Table)      2. Probe Phase (Large Table)
  ┌──────────┐                      ┌──────────┐
  │ Dept (S) │ ──▶ [ Hash Func ] ──▶│ Data Row │ ◀── DeptID = 10?
  └──────────┘          │           └────┬─────┘
                        ▼                ▼
                ┌───────────────┐   [ Hash Func ]
                │ Hash Table    │        │
                ├───────────────┤        │ (Match!)
                │ [10] [Sales]  │ ◀──────┘
                │ [20] [HR   ]  │ ──▶ ✅ Result Found!
                └───────────────┘
```

+++

### Ⅲ. 성능 최적화 및 주의사항

- **Build Input 크기**: 해시 테이블을 만드는 테이블이 가급적 작아야 메모리(PGA) 내에서 모든 처리가 끝나(In-Memory Hash Join) 성능이 가장 좋습니다.
- **메모리 부족 (Grace Hash Join)**: 메모리가 부족하면 데이터를 디스크로 밀어냈다가 다시 읽어야 하므로 성능이 급격히 떨어집니다.
- **등가 조인 전용**: 해시 함수를 쓰기 때문에 **`=`(Equal) 조건일 때만** 사용 가능합니다. 범위 검색($<, >$)에는 사용할 수 없습니다.

- **📢 섹션 요약 비유**: 해시 조인은 **'친구의 얼굴을 미리 사진첩(해시 테이블)에 넣어두고, 지나가는 군중(큰 테이블)을 한 명씩 대조해보는 것'**과 같습니다. 전체 군중을 일일이 기억할 필요 없이, 내 손안의 사진첩만 보고 쓱 훑어내면 되니 엄청나게 빠릅니다. 단, 사진첩이 너무 두꺼워 내 가방(메모리)에 안 들어가면 그때부터 고생이 시작됩니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[In-Memory Hash Join]**: 모든 해시 테이블이 메모리에 올라가는 이상적인 상태.
- **[Equi-Join]**: 해시 조인이 빛을 발하는 유일한 조인 유형.
- **[Hash Collision]**: 해시 조인 성능을 저해하는 버킷 충돌 현상.

📢 **마무리 요약**: **Hash Join**은 데이터베이스의 물리적 조인 중 가장 강력한 파괴력을 가집니다. 메모리라는 비싼 자원을 쓰는 만큼, 대용량 데이터를 처리할 때 가장 빠른 응답 속도를 보장합니다.
