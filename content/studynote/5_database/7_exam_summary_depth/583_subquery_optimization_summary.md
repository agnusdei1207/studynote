# 583. 서브쿼리 최적화 - 비상관 vs 상관 서브쿼리의 성능 차이

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서브쿼리는 쿼리 내부에 중첩된 질의이며, **비상관 서브쿼리는 독립적으로 실행**되어 결과를 메인 쿼리에 제공하는 반면, **상관 서브쿼리는 메인 쿼리의 행을 하나씩 참조하여 반복 실행**되는 구조적 차이가 있다.
> 2. **가치**: 상관 서브쿼리는 표현력이 풍부하지만 대량 데이터 처리 시 **중첩 루프(Nested Loop)와 같은 부하**를 유발하므로, 성능을 위해 조인(Join)이나 윈도우 함수로 변환하는 최적화가 필수적이다.
> 3. **융합**: 서브쿼리 펼치기(Unnesting) 및 세미 조인(Semi-join) 최적화 기술이 옵티마이저에 융합되어, 복잡한 서브쿼리를 효율적인 실행 계획으로 재구성한다.

+++

### Ⅰ. 비상관 vs 상관 서브쿼리 상세 비교

| 구분 | 비상관 서브쿼리 (Non-correlated) | 상관 서브쿼리 (Correlated) |
|:---|:---|:---|
| **상호 의존성** | 독립적 (메인 쿼리와 관련 없음) | **상호 의존적** (메인 쿼리의 컬럼 참조) |
| **실행 순서** | **서브쿼리 1번 실행** ──▶ 메인 쿼리 | 메인 쿼리 행마다 서브쿼리 실행 (Loop) |
| **성능** | 비교적 양호 (결과 캐싱 가능) | **데이터량에 비례하여 부하 급증** 💥 |
| **예시** | `WHERE ID IN (SELECT ID FROM ...)` | `WHERE Sal > (SELECT AVG(Sal) FROM ... WHERE T1.ID = T2.ID)` |

+++

### Ⅱ. 실행 프로세스 비교 시각화 (ASCII Model)

```text
[ 1. 비상관 (Single Run) ] ✅           [ 2. 상관 (Recursive Run) ] 💥

  (Subquery Result)                      Main Table: [ Row 1 ][ Row 2 ][ Row 3 ]
         │                                     │         │         │
         ▼ (Cache once)                        ▼         ▼         ▼
  [ Filter List ]                        [ Subquery ][ Subquery ][ Subquery ] ✅
         │                               (메인 쿼리 행의 수만큼 반복 호출!) 
         ▼
  (Apply to Main Query)
```

### Ⅲ. 성능 최적화(Tuning) 전략

1. **Unnesting (서브쿼리 풀기)**: 중첩된 서브쿼리를 일반 조인 문장으로 바꿉니다. 옵티마이저가 가장 선호하는 방식입니다.
2. **EXISTS 활용**: `IN` 연산자 대신 `EXISTS`를 사용하여, 첫 번째 일치하는 데이터 발견 시 서브쿼리 실행을 즉시 멈추게 합니다 (Short-circuit).
3. **윈도우 함수로 대체**: 상관 서브쿼리를 이용한 그룹별 집계 연산은 윈도우 함수(`AVG OVER`)로 바꾸면 조인 없이 단 한 번의 테이블 스캔으로 끝낼 수 있습니다. ✅

- **📢 섹션 요약 비유**: 서브쿼리 최적화는 **'심부름 보내기'**와 같습니다. 비상관 서브쿼리는 시장에 가서 "사과 10개 사와"라고 시키고(1번 실행), 가져온 사과를 요리하는 것과 같습니다. 반면 상관 서브쿼리는 요리사가 칼질을 한 번 할 때마다(메인 쿼리 행) 시장에 전화를 걸어 "이 사과 깎아도 되나요?"라고 물어보는(반복 실행) 아주 답답하고 느린 방식입니다. 이 전화를 줄이고 미리 허락을 받아두는 것(조인/최적화)이 성능의 핵심입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Scalar Subquery]**: 결과가 단 하나인 서브쿼리 (주로 SELECT 절에 사용).
- **[Semi-join]**: 서브쿼리 존재 여부만 체크하고 결합하는 고속 조인 방식.
- **[View Merging]**: 서브쿼리나 뷰를 메인 쿼리에 합쳐서 최적화하는 기법.

📢 **마무리 요약**: **Subquery Optimization**은 쿼리의 '깊이'를 '넓이'로 바꾸는 작업입니다. 중첩된 로직을 수평적인 데이터 흐름으로 전환할 때, 데이터베이스는 비로소 숨겨진 성능을 발휘합니다.
