# 497. 툼스톤(Tombstone) - 분산 삭제의 약속된 증거

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 툼스톤(Tombstone)은 분산 데이터베이스에서 데이터를 즉시 물리적으로 삭제하는 대신, **해당 레코드가 삭제되었음을 나타내는 특수 마커(비석)를 남겨두는 지연 삭제 기법**이다.
> 2. **가치**: 노드 간 동기화 지연이나 장애 발생 시, 삭제된 데이터가 다른 노드로부터 다시 부활하는 **'데이터 좀비' 현상을 방지**하여 분산 환경의 데이터 정합성을 유지한다.
> 3. **융합**: LSM-Tree 기반의 스토리지 엔진과 결합되어 삭제를 '쓰기'의 일종으로 처리하며, 주기적인 콤팩션(Compaction) 과정을 통해 툼스톤과 원본 데이터를 최종적으로 정리한다.

+++

### Ⅰ. 툼스톤이 필요한 이유: 데이터 부활 문제

- **상황**: 복제본이 3개(A, B, C)인 환경에서 데이터 X를 삭제하려 합니다.
- **문제**: 노드 C가 장애인 동안 A와 B에서 X를 삭제했습니다. 나중에 C가 복구되어 A에게 "나 X 가지고 있어!"라며 복구 데이터를 보내면, A는 X가 원래 없었던 건지 삭제된 건지 알 수 없어 X를 다시 생성해 버릴 수 있습니다.
- **해결**: "X는 삭제됨"이라는 **툼스톤(비석)**을 세워두면, C가 나타나도 "그건 죽은 데이터야!"라고 명확히 알려주어 부활을 막습니다.

+++

### Ⅱ. 툼스톤 생명주기 시각화 (ASCII Model)

```text
[ Tombstone Lifecycle ]

  1. DELETE Request ──▶ [ Data X ]
                          │
  2. Write Tombstone ──▶ [ Tombstone X ] (Status: Deleted, TS: 10:00) ✅
                          │
  3. Replication ──────▶ All Nodes receive "Tombstone X"
                          │
  4. GC / Compaction ──▶ (After Grace Period...) 
                          │ 툼스톤과 원본 데이터를 물리적으로 소멸 💾
                          ▼
                       [ Empty Space ] ✅
```

+++

### Ⅲ. 툼스톤 관리 시 주의사항

- **저장 공간 낭비**: 삭제가 빈번할수록 툼스톤 데이터가 쌓여 디스크 공간을 점유하고 읽기 성능을 저하시킬 수 있습니다.
- **Compaction 설정**: 툼스톤이 모든 노드에 전파될 충분한 시간(보통 수일)을 확보한 뒤에만 물리적으로 삭제해야 합니다.
- **Read Latency**: 쿼리 시 실제 데이터와 툼스톤을 함께 필터링해야 하므로 조회 속도에 영향을 줄 수 있습니다.

- **📢 섹션 요약 비유**: 툼스톤은 **'철거 예정지의 빨간 딱지'**와 같습니다. 건물을 바로 허물 수 없는 상황에서, "이 건물은 곧 사라질 것"이라는 딱지(툼스톤)를 붙여두어 외부인이 입주(데이터 부활)하는 것을 막는 것입니다. 동네 정비의 날(Compaction)이 되면 비로소 딱지가 붙은 건물들을 한꺼번에 철거하여 빈터를 만듭니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[LSM-Tree]**: 툼스톤을 삭제 로그로 활용하는 물리 저장 구조.
- **[GC Grace Seconds]**: 툼스톤이 유지되어야 하는 최소 유예 시간.
- **[Bloom Filter]**: 툼스톤 조회 오버헤드를 줄이기 위한 보조 기술.

📢 **마무리 요약**: **Tombstone**은 분산 시스템의 정직한 기록입니다. 눈에 보이지 않는 비석을 통해 데이터의 과거를 명확히 정리함으로써, 분산 데이터의 무결성을 완성합니다.
