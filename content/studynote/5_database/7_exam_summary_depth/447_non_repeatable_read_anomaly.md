# 447. 비반복 읽기(Non-Repeatable Read) - 변덕스러운 데이터

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비반복 읽기는 한 트랜잭션 내에서 **동일한 데이터를 두 번 읽었을 때, 그 사이 다른 트랜잭션이 해당 데이터를 수정하고 커밋함에 따라 읽기 결과가 서로 다르게 나타나는 현상**이다.
> 2. **가치**: 데이터의 '값'이 변하는 일관성 문제를 다루며, 정산이나 통계 보고서처럼 트랜잭션 내내 데이터가 불변해야 하는 비즈니스 로직에 심각한 논리적 오류를 유발한다.
> 3. **융합**: 격리 수준 'Read Committed' 이하에서 발생하며, 이를 해결하기 위해 'Repeatable Read' 격리 수준이나 MVCC 스냅샷 기술이 융합되어 사용된다.

+++

### Ⅰ. 비반복 읽기의 발생 시나리오

1. **최초 조회**: 트랜잭션 T1이 사용자 A의 점수(100점)를 읽습니다.
2. **타인의 수정**: 그 사이 다른 트랜잭션 T2가 사용자 A의 점수를 200점으로 고치고 **커밋**합니다.
3. **재조회**: T1이 업무 처리를 위해 한 번 더 사용자 A의 점수를 읽었는데, 이번에는 '200점'이 나옵니다.
4. **결과**: T1은 분명 한 트랜잭션 안에서 같은 사람을 조회했는데 값이 바뀌어 있어 로직이 꼬이게 됩니다 (일관성 결여).

+++

### Ⅱ. 비반복 읽기 타임라인 시각화 (ASCII Flow)

```text
[ Non-Repeatable Read Scenario ]

  Time | Transaction T1 (Reporter) | Transaction T2 (Updater) | Actual DB
  -----|---------------------------|--------------------------|----------
   t1  | Read Score ──▶ [100]      |                          |   100
   t2  |                           | Update Score = 200       |   100 (in mem)
   t3  |                           | 🟢 COMMIT!               |   200 ✅
   t4  | Read Score ──▶ [200] 💥   |                          |   200
  -----------------------------------------------------------------------
  * Result: T1 sees different values for the SAME record in ONE transaction.
```

+++

### Ⅲ. 대응 방안 및 기술적 해결

- **Repeatable Read 격리 수준**: 트랜잭션이 처음 읽은 데이터의 상태를 끝까지 유지하도록 잠금(S-Lock)을 걸거나 스냅샷을 고정합니다.
- **MVCC (Multi-Version Concurrency Control)**: 데이터를 수정할 때 이전 버전을 보관해 둡니다. T1은 남이 데이터를 수만 번 고쳐도 자기가 시작한 시점의 '과거 버전'만 계속 읽어 일관성을 유지합니다.

- **📢 섹션 요약 비유**: 비반복 읽기는 **'장부 결산 중 숫자 고치기'**와 같습니다. 내가 장부를 펴서 1번 페이지의 금액을 확인하고(100원), 2번 페이지를 정리하고 있는데, 누가 몰래 와서 1번 페이지의 숫자를 200원으로 고쳐놓고 도망간(Commit) 상황입니다. 다시 1번 페이지를 봤을 때 숫자가 바뀌어 있으니 결산 결과가 엉망이 되는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Inconsistency]**: 비반복 읽기가 유발하는 데이터의 논리적 모순.
- **[Read Committed]**: 이 현상이 발생할 수 있는 가장 높은 격리 수준.
- **[Snapshot Isolation]**: 비반복 읽기를 막는 현대적 동시성 기술의 핵심.

📢 **마무리 요약**: **Non-Repeatable Read**는 데이터의 신뢰를 무너뜨립니다. 한 트랜잭션 안에서는 세상이 변해도 내 데이터만큼은 변하지 않아야 한다는 원칙을 지키는 것이 견고한 시스템의 기초입니다.
