# 565. 인메모리 DB의 영속성 전략 - 속도와 안전의 공존

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인메모리 DB(In-memory DB)의 영속성 전략은 모든 데이터를 RAM에서 처리하면서도, **휘발성 문제를 해결하기 위해 로그 기록(Logging)과 스냅샷(Snapshotting) 기술을 사용하여 디스크에 데이터를 백업하는 매커니즘**이다.
> 2. **가치**: 서비스 가동 중 발생하는 체크포인트(Checkpoint) 지연을 최소화하기 위해 **Copy-On-Write(COW)**나 백그라운드 비차단 방식(Non-blocking)을 활용하여, 고성능 응답 속도를 유지하면서도 장애 시 완벽한 복구를 보장한다.
> 3. **융합**: 고성능 연산 기술과 WAL(Write-Ahead Logging) 프로토콜이 융합되어, 메모리의 초고속 성능과 디스크의 영구적 안정성을 동시에 달성한다.

+++

### Ⅰ. 인메모리 DB의 2대 영속성 보장 기술

1. **로그 기반 (Transaction Logging / AOF)**:
    - 데이터 변경이 일어날 때마다 그 내역을 파일에 순차적으로 기록합니다. 
    - **장점**: 데이터 유실이 거의 없습니다.
    - **단점**: 로그 파일이 계속 커지며, 복구 시 로그를 하나씩 재실행(Redo)하느라 시간이 오래 걸립니다.
2. **스냅샷 기반 (Snapshotting / RDB)**:
    - 특정 시점의 메모리 데이터 전체를 통째로 디스크에 덤프(Dump)합니다.
    - **장점**: 복구 속도가 매우 빠릅니다.
    - **단점**: 스냅샷 주기 사이의 데이터는 장애 시 유실될 위험이 있습니다.

+++

### Ⅱ. 비차단 체크포인트 아키텍처 시각화 (ASCII Model)

서비스 중단 없이 백업을 수행하기 위해 자식 프로세스를 활용하는 방식입니다 (Redis 등).

```text
[ Non-blocking Checkpoint (Copy-on-Write) ]

  [ Main Process (RW) ] ──▶ [ Active Memory (RAM) ] ✅ "Normal Operation"
           │                         │
           ▼ (Fork!)                 │ (Update happens...)
  [ Child Process (RO) ] ────────────┘
           │
           └─▶ [ Snapshot Creation ] ──▶ [ Disk Storage ] 💾 ✅
               (Snapshot of T0 point)
               
  * COW: 메인 프로세스가 데이터를 고쳐도, 자식은 복사본(T0)을 보고 안전하게 백업.
```

+++

### Ⅲ. 지연 성능 최소화를 위한 최적화 기법

- **비동기 로깅**: 로그를 디스크에 쓰는 작업을 별도 스레드에 맡겨 사용자 응답 시간을 단축합니다.
- **증분 스냅샷**: 전체가 아닌 마지막 스냅샷 이후의 변경분만 저장하여 I/O 부하를 줄입니다.
- **NVM (Non-Volatile Memory) 활용**: 하드웨어적으로 전원이 꺼져도 데이터가 보존되는 고속 메모리를 사용하여 소프트웨어적 백업 오버헤드를 근본적으로 제거합니다.

- **📢 섹션 요약 비유**: 인메모리 DB의 영속성 전략은 **'공연장의 실시간 녹화'**와 같습니다. 관객(사용자)에게 최고의 몰입감(속도)을 주기 위해 무대 위에서는 쉴 새 없이 공연이 펼쳐지지만, 만약의 사고를 대비해 무대 뒤에서는 카메라로 전체 장면을 찍어두거나(Snapshot), 대본에 실시간으로 수정 사항을 적어두는(Logging) 것입니다. 특히 배우가 움직이면서도 방해받지 않게 멀리서 조용히 찍는 기술(Non-blocking)이 훌륭한 공연의 핵심입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[AOF (Append Only File)]**: 변경 로그를 계속 덧붙이는 방식.
- **[RDB (Redis Database)]**: 특정 시점의 스냅샷을 저장하는 방식.
- **[Persistence Trade-off]**: 더 안전하게 기록할수록(Sync) 속도는 느려지는 물리적 한계.

📢 **마무리 요약**: **In-memory DB Persistence**는 속도라는 마법을 부리기 위한 안전장치입니다. 정교한 체크포인트 설계를 통해 메모리의 찬란함과 디스크의 묵직함을 조화롭게 운영해야 합니다.
