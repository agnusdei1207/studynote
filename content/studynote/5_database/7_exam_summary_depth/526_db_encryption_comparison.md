# 526. DB 암호화 방식 비교 - API, Plug-in, TDE의 조화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DB 암호화는 데이터를 안전하게 보호하기 위해 적용되는 기술로, **적용 위치와 주체에 따라 API 방식(애플리케이션), Plug-in 방식(DB 커널), TDE 방식(DB 엔진 스토리지)**으로 구분된다.
> 2. **가치**: 데이터의 기밀성을 확보하여 유출 시에도 내용을 해독할 수 없게 하며, 성능 오버헤드와 애플리케이션 수정 비용 사이의 최적점을 선택하여 보안 시스템을 구축한다.
> 3. **융합**: 각 방식은 상호 배타적인 것이 아니며, 중요도에 따라 특정 컬럼은 API 암호화를 적용하고 전체 데이터 파일은 TDE로 보호하는 다층 암호화 전략으로 융합된다.

+++

### Ⅰ. 3대 암호화 방식 상세 비교

| 비교 항목 | API 방식 | Plug-in 방식 | TDE 방식 |
|:---|:---|:---|:---|
| **암호화 위치** | 애플리케이션 서버 | **DB 서버 (커널)** | **DB 서버 (스토리지)** |
| **수행 주체** | 애플리케이션(JAVA/C 등) | DBMS 확장 모듈 | DBMS 엔진 자체 |
| **수정 범위** | **애플리케이션 소스 수정** | 거의 없음 (SQL 일부) | **전혀 없음 (Transparent)** |
| **성능 부하** | 앱 서버 CPU 분산 | DB 서버 CPU 부하 | DB 서버 I/O 부하 |
| **색인 검색** | 인덱스 활용 어려움 | 인덱스 활용 가능(일부) | **인덱스 활용 100% 가능** |

+++

### Ⅱ. 암호화 방식별 위치 시각화 (ASCII Model)

```text
[ Database Encryption Points ]

  (Application)  ────(1) API 방식 (가장 앞단 보안)
       │
  [ Network ]
       │
  [ DB Server ]  ────(2) Plug-in 방식 (커널 레벨 통제)
       │
  [ Storage ]    ────(3) TDE 방식 (디스크 저장 시 박제) ✅
```

+++

### Ⅲ. 실무적 선택 가이드

1. **TDE (Transparent Data Encryption)**: 디스크 분실이나 백업 파일 유출 대비에 최적. 성능 영향이 가장 적고 운영이 편리함.
2. **API 방식**: DB 서버가 뚫려도 데이터 기밀성을 유지해야 하는 초고도 보안 데이터(비밀번호 등)에 적합. 단, 개발 공수가 큼.
3. **Plug-in 방식**: 소스 수정은 어렵지만, 특정 컬럼에 대해 세밀한 암호화 정책을 적용하고 싶을 때 사용.

- **📢 섹션 요약 비유**: DB 암호화 방식은 **'편지의 보안'**과 같습니다. API 방식은 '편지를 쓰자마자 암호로 바꾸는 것'이고, Plug-in은 '우체국(DB)에 도착했을 때 암호로 바꾸는 것'이며, TDE는 '편지를 금고(디스크)에 넣을 때 금고 자체가 자동으로 내용을 암호화하는 것'과 같습니다. 앞단에서 할수록 안전하지만(API), 금고가 알아서 해주는 게(TDE) 가장 편리하고 빠릅니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[At-rest Encryption]**: TDE와 같이 저장된 상태의 데이터를 보호하는 것.
- **[In-transit Encryption]**: 네트워크 전송 구간(TLS)을 보호하는 것.
- **[Key Management]**: 암호화 방식보다 훨씬 더 중요한 암호 키(Key)의 안전한 보관.

📢 **마무리 요약**: **Encryption Methods**는 보안의 깊이를 결정합니다. 비즈니스 영향도와 보안 위협 모델을 분석하여, 가장 효율적이고 강력한 암호화 믹스(Mix)를 구성해야 합니다.
