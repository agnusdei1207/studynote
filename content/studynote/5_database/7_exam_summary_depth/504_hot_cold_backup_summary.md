# 504. 데이터베이스 백업 전략 - Hot Backup vs Cold Backup

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 백업은 시스템 장애나 데이터 오염에 대비하여 데이터의 복사본을 생성하는 것으로, **서비스 가동 중에 수행하는 핫 백업(Hot)과 서비스를 중단하고 수행하는 콜드 백업(Cold)**으로 나뉜다.
> 2. **가치**: 핫 백업은 24/7 무중단 서비스를 지원하는 현대 인프라의 필수 요건이며, 콜드 백업은 완벽한 정적인 시점의 데이터 일관성을 보장하는 가장 단순하고 확실한 방법이다.
> 3. **융합**: 아카이브 로그(Archive Log) 기술과 융합된 핫 백업은 특정 시점 복구(PITR)를 가능케 하여, 데이터 유실을 최소화하는 비즈니스 연속성 계획(BCP)의 핵심이 된다.

+++

### Ⅰ. 핫 백업 vs 콜드 백업 상세 비교

| 비교 항목 | 핫 백업 (Hot / Online) | 콜드 백업 (Cold / Offline) |
|:---|:---|:---|
| **서비스 상태** | **가동 중 (Online)** | **중단 (Shutdown)** |
| **데이터 일관성** | 아카이브 로그를 통해 보정 필요 | **물리적으로 완벽히 일치** |
| **복구 난이도** | 상대적으로 복잡함 | 매우 단순 (파일 복사) |
| **주요 용도** | 24시간 가동 서비스 | 배치 시스템, 초기 구축 단계 |
| **필수 전제** | **ARCHIVELOG 모드 설정** | 없음 |

+++

### Ⅱ. 백업 방식별 아키텍처 시각화 (ASCII Model)

```text
[ 1. Hot Backup: Running... ]         [ 2. Cold Backup: Stopped... ]

  (Application Traffic) ✅              (Application Traffic) ❌
          │                                     │
  ┌───────▼───────┐                     ┌───────▼───────┐
  │ DB Instance   │                     │  DB Stopped   │
  ├───────────────┤                     ├───────────────┤
  │ [Data] [Log]  │ ──▶ [Backup]        │ [Data] [Log]  │ ──▶ [Backup]
  └───────────────┘      (Online)       └───────────────┘      (Offline)
```

### Ⅲ. 핫 백업의 핵심: 아카이브 로그

- **문제점**: 백업이 진행되는 동안에도 데이터는 계속 변합니다. (백업 시작 시점과 끝 시점의 파일 내용이 불일치)
- **해결**: 백업 중에 발생한 모든 변경 로그를 별도의 파일(**Archive Log**)로 남겨둡니다. 
- **복구 시**: 백업본을 복원한 뒤, 아카이브 로그를 재실행(Redo)하여 조각난 시간을 하나로 맞춥니다.

- **📢 섹션 요약 비유**: 핫 백업은 **'달리는 기차 안에서 짐 옮기기'**와 같습니다. 기차가 멈추지 않아 승객(사용자)은 편하지만, 짐을 옮기는 동안 기차가 흔들려 짐이 섞일 수 있으므로 짐의 이동 경로를 꼼꼼히 기록(로그)해 두어야 합니다. 콜드 백업은 **'종점역에 멈춘 기차에서 짐 옮기기'**입니다. 아주 안전하고 편하지만, 짐을 다 옮길 때까지 기차가 출발하지 못해 승객들이 기다려야 하는 단점이 있습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[PITR (Point-In-Time Recovery)]**: 핫 백업과 로그를 이용해 원하는 시각으로 되돌리는 기술.
- **[RTO / RPO]**: 백업 방식에 따라 결정되는 복구 목표 수치.
- **[Incremental Backup]**: 바뀐 부분만 골라 백업하여 핫 백업의 부하를 줄이는 기술.

📢 **마무리 요약**: **Backup Strategy**는 데이터의 생명 보험입니다. 서비스의 성격과 중단 허용 시간을 고려하여, 가장 확실한 정합성과 가용성을 보장하는 백업 믹스를 설계해야 합니다.
