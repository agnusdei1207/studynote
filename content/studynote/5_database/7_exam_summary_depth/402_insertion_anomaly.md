# 402. 삽입 이상 (Insertion Anomaly) - 원치 않는 데이터의 강요

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 삽입 이상은 데이터베이스에 **새로운 정보를 저장할 때, 해당 정보와는 직접적인 관련이 없는 불필요한 데이터까지 강제로 함께 입력해야만 하는 비정상적인 상태**를 말한다.
> 2. **가치**: 정규화되지 않은 테이블 구조의 취약성을 드러내는 지표이며, 데이터 입력의 유연성을 떨어뜨리고 NULL 값의 남발을 초래하는 주범이다.
> 3. **융합**: 개체 무결성 및 참조 무결성과 연결되어 있으며, 테이블 분해(Decomposition)를 통해 각 개체가 독립적으로 삽입될 수 있는 환경을 구축함으로써 해결한다.

+++

### Ⅰ. 삽입 이상의 발생 원인

- **근본 원인**: 서로 다른 성격의 두 개 이상의 엔티티(Entity)가 하나의 테이블에 섞여 있기 때문입니다.
- **상황**: (학번, 학생이름, 학과명, 학과전화번호)가 한 테이블에 있는 경우.
- **문제 발생**: 신설 학과가 생겨서 학과명과 전화번호만 등록하고 싶은데, 아직 해당 학과에 소속된 '학생(학번)'이 없으면 기본키(학번) 제약 조건 때문에 학과 정보 자체를 아예 입력할 수 없습니다.

+++

### Ⅱ. 삽입 이상 시나리오 시각화 (ASCII Model)

```text
[ Unnormalized Table: Student_Dept ]

  ┌────────┬──────────┬──────────┬──────────┐
  │ 학번(PK)│ 학생이름  │ 학과명    │ 학과번호  │
  ├────────┼──────────┼──────────┼──────────┤
  │ 202601 │ Kim      │ CS       │ 02-111   │
  │ NULL   │ NULL     │ AI       │ 02-222   │ ◀── 💥 삽입 불가!
  └────────┴──────────┴──────────┴──────────┘
  * PK인 '학번'이 없으면 'AI 학과' 정보는 세상에 존재할 수 없음.
```

+++

### Ⅲ. 해결 방안: 테이블 분리

- **정규화 수행**: 학생 테이블과 학과 테이블을 분리하여 각각의 기본키를 갖게 합니다.
- **결과**: 학과 테이블에 신설 학과 정보를 먼저 넣고, 나중에 학생이 생기면 외래키로 연결하면 됩니다. 데이터 간의 불필요한 종속성이 제거됩니다.

- **📢 섹션 요약 비유**: 삽입 이상은 **'세트 메뉴 강매'**와 같습니다. 나는 콜라(학과 정보)만 한 잔 마시고 싶은데, 가게에서 "햄버거(학생 정보)를 같이 주문하지 않으면 콜라를 아예 팔지 않겠다"고 고집을 부리는 상황입니다. 배가 안 고픈데도 억지로 햄버거를 사야 하거나(NULL 입력), 아니면 콜라를 포기해야 하는(저장 실패) 불합리한 규칙입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Entity Integrity]**: PK가 NULL이면 안 된다는 원칙 (삽입 이상의 기술적 장벽).
- **[Normal Form]**: 삽입 이상을 해결하기 위한 단계적 처방전.
- **[Data Consistency]**: 삽입 이상이 해결되었을 때 얻게 되는 데이터의 건강 상태.

📢 **마무리 요약**: **Insertion Anomaly**는 설계의 결함을 알리는 경고음입니다. 데이터가 독자적으로 태어날 권리를 보장해 주는 것이 데이터베이스 설계자의 기본 의무입니다.
