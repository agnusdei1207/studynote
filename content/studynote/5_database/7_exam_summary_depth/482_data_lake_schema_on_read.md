# 482. 데이터 레이크와 Schema-on-Read - 날것의 데이터 활용법

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 레이크(Data Lake)는 정형, 반정형, 비정형 데이터를 가공 없이 **원시 형태(Raw Format) 그대로 대량 저장**하며, 데이터를 읽는 시점에 스키마를 정의하는 **Schema-on-Read** 방식을 취한다.
> 2. **가치**: 데이터를 담기 전에 구조를 미리 고민할 필요가 없어 **수집 속도가 압도적으로 빠르며**, 미래에 어떤 분석 요구가 생기더라도 원본 데이터를 활용해 대응할 수 있는 무한한 유연성을 제공한다.
> 3. **융합**: 클라우드 오브젝트 스토리지(S3 등) 및 하둡(HDFS)과 융합되어 저비용 고효율의 빅데이터 기반을 형성하며, 인공지능(AI) 학습을 위한 풍부한 원천 데이터를 공급한다.

+++

### Ⅰ. Schema-on-Write (DW) vs Schema-on-Read (Lake)

- **Schema-on-Write (DW)**: 저장할 때 엄격한 스키마에 맞춰야 함. (정제된 데이터, 빠른 조회, 높은 비용)
- **Schema-on-Read (Lake)**: 일단 원본 그대로 던져두고, 분석 쿼리를 날릴 때 비로소 "이 데이터는 이런 구조야"라고 정의함. (날것의 데이터, 유연한 분석, 낮은 비용)

+++

### Ⅱ. 데이터 레이크 저장 매커니즘 시각화 (ASCII Model)

```text
[ Data Lake: Raw Storage Flow ]

  (Various Sources)       [ Data Lake Storage ]       (Analysis Time)
  ┌────────────┐        ┌~~~~~~~~~~~~~~~~~~~~~┐      ┌───────────────┐
  │ JSON Log   │ ──▶    │  { "id": 1, ... }   │      │ SQL / Python  │
  ├────────────┤        │  (Original Image)   │ ──▶  │ Define Schema │ ✅
  │ Image File │ ──▶    │  [ CSV Content ]    │      │ at Runtime    │
  └────────────┘        └~~~~~~~~~~~~~~~~~~~~~┘      └───────────────┘
                         (Schema-less Land)           (Schema-on-Read)
```

+++

### Ⅲ. 실무적 의의: 왜 레이크인가?

1. **데이터 유실 방지**: ETL 과정에서 필요 없다고 버려지는 데이터가 없습니다. 5년 뒤에 그 데이터가 중요해질 때 레이크에서 다시 꺼내 쓸 수 있습니다.
2. **데이터 사이언스 최적화**: 머신러닝 모델은 정제된 통계값보다 가공되지 않은 '날것'의 특징들을 학습할 때 더 좋은 성능을 낼 수 있습니다.
3. **비용 혁신**: 모든 데이터를 비싼 RDBMS에 넣는 대신, 저렴한 클라우드 저장소에 담아 경제적인 데이터 자산화를 실현합니다.

- **📢 섹션 요약 비유**: 데이터 레이크는 **'편집 전의 영상 원본 데이터'**와 같습니다. 완성된 뉴스(DW)는 보기 편하지만 편집자의 의도대로 잘려 나간 부분이 많습니다. 원본 영상(Lake)은 길고 지루할 수 있어도, 나중에 감독이 바뀌었을 때 전혀 다른 영화(새로운 분석 결과)를 만들어낼 수 있는 무궁무진한 재료가 됩니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Raw Format]**: 변환되지 않은 원시 데이터 상태.
- **[Object Storage]**: 데이터 레이크의 물리적 기반 (S3, Azure Blob 등).
- **[Data Swamp]**: 관리(메타데이터)가 안 되어 썩어버린 데이터 레이크의 비극.

📢 **마무리 요약**: **Data Lake & Schema-on-Read**는 데이터의 잠재력을 보존하는 기술입니다. 미래의 불확실한 질문에 답하기 위해, 오늘의 진실을 있는 그대로 간직하는 거대한 호수입니다.
