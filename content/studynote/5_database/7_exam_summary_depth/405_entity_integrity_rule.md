# 405. 개체 무결성 (Entity Integrity) - 데이터 식별의 절대 법칙

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 개체 무결성은 관계형 데이터베이스의 모든 테이블은 반드시 **기본키(Primary Key)를 가져야 하며, 기본키를 구성하는 어떠한 속성도 NULL 값을 가질 수 없다**는 제약 조건이다.
> 2. **가치**: 테이블 내의 모든 튜플(행)을 유일하게 식별할 수 있음을 보장하여, 데이터의 중복을 방지하고 정확한 데이터 접근 및 수정을 가능케 한다.
> 3. **융합**: 참조 무결성과 함께 데이터베이스 무결성의 양대 산맥을 이루며, 실무적으로는 인덱스(Index) 생성의 기준이 되어 검색 성능과 직결된다.

+++

### Ⅰ. 개체 무결성의 2대 원칙

1. **유일성 (Uniqueness)**: 모든 행은 기본키 값에 의해 서로 구별되어야 합니다. 똑같은 키 값을 가진 행이 두 개 존재할 수 없습니다.
2. **최소성 (Minimality)**: 기본키는 행을 구별하는 데 꼭 필요한 속성들로만 구성되어야 합니다.
3. **NOT NULL (비공백성)**: 기본키는 개체를 식별하는 '이름표'와 같으므로, 이름표가 비어있는(NULL) 상태로 존재할 수 없습니다.

+++

### Ⅱ. 개체 무결성 위반 사례 시각화 (ASCII Model)

```text
[ Table: User ]

  ┌──────────┬──────────┬──────────┐
  │ User_ID  │ Name     │ Email    │
  ├──────────┼──────────┼──────────┤
  │   101    │ Kim      │ k@ai.com │
  │   101    │ Lee      │ l@ai.com │ ◀── 💥 유일성 위반 (Duplicate PK)
  │   NULL   │ Park     │ p@ai.com │ ◀── 💥 비공백성 위반 (NULL PK)
  └──────────┴──────────┴──────────┘
  * 위와 같은 데이터는 개체 무결성이 깨진 상태로, 저장 자체가 거부되어야 함.
```

+++

### Ⅲ. 실무적 의의: 왜 중요한가?

- **데이터 정합성**: 기본키가 중복되거나 NULL이면, 특정 사용자의 정보를 수정하려 할 때 어떤 행을 고쳐야 할지 알 수 없게 됩니다.
- **관계의 연결고리**: 다른 테이블에서 이 테이블을 참조(외래키)할 때, 식별자가 불분명하면 관계를 맺을 수 없습니다.
- **성능**: DBMS는 기본키에 대해 자동으로 고유 인덱스(Unique Index)를 생성하여 검색 속도를 최적화합니다.

- **📢 섹션 요약 비유**: 개체 무결성은 **'국민의 주민등록번호'**와 같습니다. 대한민국 국민이라면 누구나 고유한 번호(기본키)를 가져야 하고, 번호가 없는 국민(NULL)이나 번호가 똑같은 두 명의 국민(중복)이 있어서는 안 됩니다. 이 번호가 확실해야만 병원 기록도 찾고 세금도 정확히 매길 수 있는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Primary Key]**: 개체 무결성을 구현하는 구체적 도구.
- **[Unique Index]**: 개체 무결성을 물리적으로 강제하는 장치.
- **[Tuple Identification]**: 개체 무결성이 지향하는 궁극의 목적 (행 식별).

📢 **마무리 요약**: **Entity Integrity**는 데이터베이스의 질서 그 자체입니다. "모든 존재는 이름이 있어야 하고, 그 이름은 유일해야 한다"는 단순한 진리가 거대한 데이터 시스템을 지탱하는 힘입니다.
