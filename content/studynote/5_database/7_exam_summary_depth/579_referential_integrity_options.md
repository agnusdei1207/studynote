# 579. 참조 무결성 유지 옵션 - 관계의 운명을 결정하다

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 참조 무결성 유지 옵션은 부모 테이블의 기본키(PK)가 삭제되거나 수정될 때, **이를 참조하고 있는 자식 테이블의 외래키(FK) 데이터를 어떻게 처리할 것인지 정의하는 비즈니스 규칙**이다.
> 2. **가치**: 데이터 간의 **연쇄적 변화(Cascade)**나 **삭제 방지(Restrict)**를 자동화하여, 애플리케이션의 개입 없이 DB 엔진 차원에서 데이터 정합성을 완벽하게 보호한다.
> 3. **융합**: CASCADE, RESTRICT, SET NULL, NO ACTION 등의 옵션이 외래키 제약 조건과 융합되어, 복잡한 엔티티 간의 생명주기를 물리적으로 통제한다.

+++

### Ⅰ. 주요 참조 무결성 옵션 상세

| 옵션 명칭 | 부모 데이터 변경/삭제 시 자식의 동작 | 주요 활용 유즈케이스 |
|:---|:---|:---|
| **CASCADE** | **동일하게 변경/삭제됨** | 주문 삭제 시 주문 상세도 자동 삭제 |
| **RESTRICT** | **수행 거부** (자식이 있으면 부모 손 못 댐) | 부서원 있는 부서 정보 삭제 방지 |
| **SET NULL** | 자식의 외래키 값을 **NULL**로 변경 | 부서 삭제 시 사원의 소속을 미지정 상태로 둠 |
| **SET DEFAULT** | 자식의 외래키 값을 **기본값**으로 변경 | 담당자 삭제 시 기본 공용 계정으로 자동 지정 |
| **NO ACTION** | 위반 시 트랜잭션 종료 시점에 에러 발생 | RESTRICT와 유사하나 체크 시점의 차이 |

+++

### Ⅱ. CASCADE 동작 매커니즘 시각화 (ASCII Model)

```text
[ Referral Integrity: ON DELETE CASCADE ]

  (Parent: Board)                  (Child: Comment)
  ┌────────────┬──────────┐        ┌────────┬──────────┬──────────┐
  │ Post_ID(K) │ Title    │        │ Com_ID │ Content  │ Post_ID  │
  ├────────────┼──────────┤        ├────────┼──────────┼──────────┤
  │   101      │ Hello!   │ ◀───Ref─│  5001  │ Nice!    │   101    │
  │   102      │ Bye...   │        │  5002  │ Good!    │   101    │
  └────────────┴──────────┘        └────────┴──────────┴──────────┘

  (Action) DELETE FROM Board WHERE Post_ID = 101;
  (Result) Board 101번 삭제 ──▶ Comment 5001, 5002번 자동 연쇄 삭제! 💥 ✅
```

+++

### Ⅲ. 설계 시 고려사항: 안전과 편의의 균형

- **CASCADE의 위험성**: 편의성은 높지만, 실수로 부모 데이터를 지웠을 때 자식 데이터 수백만 건이 순식간에 사라지는 대참사가 발생할 수 있습니다. (중요 데이터는 RESTRICT 권장)
- **비즈니스 정합성**: 도메인 규칙에 따라 "부모 없는 자식이 가능한가?"를 먼저 고민하고 `SET NULL` 혹은 `RESTRICT`를 결정해야 합니다.
- **성능 영향**: 연쇄 작업이 많아지면 단일 `DELETE` 명령이 내부적으로 거대한 트랜잭션을 유발하여 시스템 부하가 커질 수 있습니다.

- **📢 섹션 요약 비유**: 참조 무결성 옵션은 **'가족 여행의 규칙'**과 같습니다. `CASCADE`는 "아빠가 안 가면 아이들도 다 안 가는 것"이고, `RESTRICT`는 "아이가 울며 매달려 있으면 아빠가 외출을 못 하는 것"이며, `SET NULL`은 "아빠가 안 가더라도 아이는 혼자 놀게 두는 것"과 같습니다. 어떤 규칙을 택하느냐에 따라 가족(데이터)의 운명이 달라집니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Referential Integrity]**: 외래키가 지켜야 할 근본적인 규칙.
- **[Foreign Key Constraint]**: 옵션을 설정하는 구체적인 SQL 문법 공간.
- **[Orphan Record]**: 옵션 설정이 잘못되어 부모를 잃어버린 불쌍한 데이터.

📢 **마무리 요약**: **Referential Options**는 관계의 완성입니다. 데이터 간의 연결 고리를 어떻게 끊고 이을 것인지에 대한 세심한 설계가 시스템의 품격을 결정합니다.
