# 470. 해시 샤딩(Hash Sharding) - 공평한 데이터의 분배

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해시 샤딩은 샤드 키 값을 해시 함수(Hash Function)에 통과시켜 얻은 결과값에 따라 **데이터를 여러 샤드에 무작위로, 그러나 균등하게 분산 저장하는 기법**이다.
> 2. **가치**: 특정 범위에 데이터가 몰리는 핫스팟(Hotspot) 현상을 방지하여 **대규모 쓰기 부하를 모든 서버에 고르게 분산**시키며, 가장 단순하면서도 효과적인 샤딩 전략을 제공한다.
> 3. **융합**: 고정된 해시 방식의 한계를 극복하기 위해 일관된 해싱(Consistent Hashing) 기술과 융합되어, 노드 추가/삭제 시 데이터 재배치 비용을 최소화하는 현대적 분산 DB의 기반이 된다.

+++

### Ⅰ. 해시 샤딩의 작동 원리

1. **키 선택**: 분산 기준이 될 샤드 키(예: User_ID)를 선정합니다.
2. **해시 연산**: `Hash(User_ID)`를 통해 고유한 난수 값을 생성합니다.
3. **샤드 결정**: `Hash(User_ID) % Shard_Count` 연산을 통해 해당 데이터가 저장될 물리적 노드 번호를 결정합니다.

+++

### Ⅱ. 해시 샤딩 프로세스 시각화 (ASCII Model)

```text
[ Hash Sharding Logic ]

  (Data) { ID: 501, Name: "A" }
           │
           ▼ [ Hash Function ] ──▶ 123,456,789
           │
           ▼ [ Modulo Operation (%) ] ──▶ 123... % 3 = 0
           │
    ┌──────┴──────┬─────────────┐
    ▼             ▼             ▼
 [ Shard 0 ]   [ Shard 1 ]   [ Shard 2 ]
 ┌─────────┐   ┌─────────┐   ┌─────────┐
 │ ID: 501 │   │         │   │         │ ✅ "Evenly Distributed"
 └─────────┘   └─────────┘   └─────────┘
```

+++

### Ⅲ. 장점과 한계점

| 항목 | 특징 |
|:---|:---|
| **장점** | 데이터가 물리적으로 완벽하게 분산되어 **부하 균형(Load Balance)**이 뛰어남 |
| **단점** | **범위 검색(Range Query)에 취약**함. (연속된 데이터가 서로 다른 샤드에 있음) |
| **확장성** | 샤드 대수가 바뀌면 해시 매핑 결과가 다 바뀌어 대규모 데이터 이동 필요 |

- **📢 섹션 요약 비유**: 해시 샤딩은 **'화카(카드 섞기) 후 나눠주기'**와 같습니다. 카드의 숫자(데이터 순서)와 상관없이 무작위로 섞어서 사람들에게 나눠주면, 모든 사람이 거의 비슷한 수의 카드를 가지게 되는 공평한 방식입니다. 하지만 내가 '다이아몬드 1부터 10까지'를 모으고 싶다면(범위 검색), 모든 사람의 손패를 다 확인해야 하는 번거로움이 생기는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Hotspot]**: 해시 샤딩이 가장 성공적으로 방어하는 장애 현상.
- **[Consistent Hashing]**: 해시 샤딩의 확장성 문제를 해결한 진화된 알고리즘.
- **[Data Skew]**: 샤드 키 선정이 잘못되어 해시 후에도 한쪽으로 쏠리는 현상.

📢 **마무리 요약**: **Hash Sharding**은 분산 저장의 기본값입니다. 정렬의 이점을 포기하는 대신 얻는 '균형'과 '속도'는 대규모 트래픽을 견뎌야 하는 현대 아키텍처의 강력한 엔진이 됩니다.
