# 578. 수퍼타입/서브타입 물리 설계 - 논리 계층의 물리적 이식

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 수퍼타입/서브타입 물리 설계는 논리 모델의 계층 관계를 실제 DB 테이블로 변환하는 과정으로, **One-to-One(1:1 분리), Roll-up(수퍼타입 통합), Roll-down(서브타입 통합)**의 3가지 전략이 있다.
> 2. **가치**: 데이터의 정합성 유지 비용과 조회 성능(Join 부하) 사이의 최적 균형을 찾으며, 워크로드의 성격(전체 조회 위주 vs 개별 조회 위주)에 따라 최적의 물리 구조를 결정한다.
> 3. **융합**: 객체 지향의 상속 개념을 관계형 표 구조로 투영하는 핵심 기술이며, 데이터 정규화와 반정규화의 전략적 선택지가 융합되는 지점이다.

+++

### Ⅰ. 3대 물리 변환 전략 상세 비교

| 전략 명칭 | 변환 방식 | 장점 | 단점 |
|:---|:---|:---|:---|
| **One-to-One** | 수퍼/서브 각각 테이블 생성 | **정규화 완벽**, 관리 용이 | **Join 부하 발생** |
| **Roll-up** | 수퍼타입으로 모든 서브타입 통합 | **조회 성능 최상** (Join 없음) | NULL 값 증가, 테이블 비대화 |
| **Roll-down** | 서브타입별로 수퍼타입 속성 중복 | 개별 서브타입 조회 최적화 | 전체 조회(UNION) 부하 큼 |

+++

### Ⅱ. 물리 변환 모델 시각화 (ASCII Model)

```text
[ Logical Model: Person (Super) - Student/Emp (Sub) ]

  1. One-to-One (분리) ✅        2. Roll-up (통합) ✅        3. Roll-down (분산) ✅
  ┌──────────┐                 ┌────────────────┐          ┌──────────────┐
  │ Person   │                 │   Integrated   │          │ Student_Tbl  │
  └────┬─────┘                 │   Person       │          ├──────────────┤
  ┌────┴────┬────────┐         │ (All Columns)  │          │ Employee_Tbl │
  ▼         ▼        │         └────────────────┘          └──────────────┘
 [Std]     [Emp]     │         (Single Big Table)          (Duplicate Super Info)
```

+++

### Ⅲ. 실무적 선택 기준

- **One-to-One 선택**: 각 서브타입의 데이터가 방대하고 독립적인 업무가 뚜렷할 때. (정규화 중시)
- **Roll-up 선택**: 전체 대상을 한꺼번에 조회하는 빈도가 압도적으로 높을 때. (성능 중시)
- **Roll-down 선택**: 수퍼타입 정보만 따로 볼 일이 거의 없고, 각 서브타입별 고유 업무만 존재할 때.

- **📢 섹션 요약 비유**: 수퍼타입/서브타입 설계는 **'가족 앨범 만들기'**와 같습니다. `One-to-One`은 가족 공통 앨범 하나와 개인별 앨범을 따로 두는 것이고(정리가 잘 되지만 둘 다 펴봐야 함), `Roll-up`은 거대한 가족 앨범 한 권에 모든 사진을 다 때려 넣는 것이며(찾기는 편하지만 앨범이 너무 무거움), `Roll-down`은 아예 가족 공통 사진을 복사해서 각자의 개인 앨범에 다 나눠 넣어주는(중복이 생기지만 내 것만 보기엔 최고) 방식입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[ISA Relationship]**: 논리 모델에서의 상속 관계 표현.
- **[Discriminator]**: 통합 테이블(Roll-up)에서 서브타입을 구분하는 속성.
- **[Partial/Total Specialization]**: 물리 설계 시 NULL 허용 여부를 결정짓는 논리 제약.

📢 **마무리 요약**: **Super/Subtype Physical Design**은 데이터의 '그릇'을 정하는 일입니다. 비즈니스 쿼리의 흐름을 정확히 읽고, 데이터의 응집도와 조회 속도의 최적점을 찾아내는 것이 설계자의 핵심 역량입니다.
