# 456. WAL(Write-Ahead Logging) - 기록의 절대 우선순위

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: WAL은 데이터베이스의 모든 변경 사항을 **실제 데이터 파일에 반영하기 전에, 반드시 복구용 로그(Redo Log)에 먼저 기록**해야 한다는 트랜잭션 처리 원칙이다.
> 2. **가치**: 디스크의 무거운 무작위 쓰기(Random I/O)를 가벼운 순차 쓰기(Sequential I/O)로 대체하여 시스템 성능을 높이며, 어떠한 시점의 시스템 크래시에서도 데이터 복구를 보장한다.
> 3. **융합**: '지연된 쓰기(No-Force)' 정책을 가능케 하는 기술적 근간이며, 현대적인 모든 관계형 및 비관계형 데이터베이스 엔진의 신뢰성을 지탱하는 표준 프로토콜이다.

+++

### Ⅰ. WAL의 2대 핵심 규칙

1. **Log Before Data**: 어떤 데이터 페이지가 디스크에 쓰여지기 전에, 해당 페이지의 모든 변경 내역을 담은 로그 레코드가 비휘발성 저장소(로그 파일)에 먼저 기록되어야 합니다.
2. **Commit Before Acknowledgement**: 사용자가 `COMMIT`을 요청하면, 실제 데이터를 쓰는 대신 관련 로그를 디스크에 동기화(Flush)하는 시점에 성공 응답을 보냅니다.

+++

### Ⅱ. WAL 동작 메커니즘 시각화 (ASCII Flow)

```text
[ WAL 기반의 고속 쓰기 프로세스 ]

  1. UPDATE SQL ──▶ [ Log Buffer (Mem) ] ──▶ 로그 작성 (순간 완료)
  2. Buffer Pool ──▶ 데이터 페이지 수정 (Dirty)
       │
  3. COMMIT 요청 ──▶ [ Redo Log (Disk) ] 💾 순차 쓰기 완료! ✅
       │             (이 시점에 사용자에게 "성공" 알림)
       ▼
  4. 지연된 반영 ──▶ [ Data File (Disk) ] 💾 (나중에 천천히 기록)
```

+++

### Ⅲ. WAL이 가져다주는 이점

- **성능 (Performance)**: 데이터 파일은 인덱스 위치를 찾느라 헤드가 바쁘게 움직여야 하지만, 로그는 무조건 맨 끝에 덧붙여 쓰므로 I/O가 매우 빠릅니다.
- **원자성 및 영속성 보장**: 정전이 발생해도 "로그"라는 완벽한 증거가 남아있으므로, 커밋된 건 Redo로 살리고 커밋 안 된 건 Undo로 지울 수 있습니다.

- **📢 섹션 요약 비유**: WAL은 **'계약서와 메모장'**의 관계와 같습니다. 두꺼운 계약서 원본(데이터 파일)을 매번 수정하여 도장을 찍으려면 시간이 오래 걸립니다. 대신, "오늘 계약 내용을 메모지(로그)에 적어서 금고에 먼저 넣어둔 뒤" 계약이 성사되었다고 선포하고, 실제 계약서 정리는 나중에 한가할 때 한꺼번에 하는 지혜로운 업무 방식입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Fsync]**: 메모리 상의 로그를 디스크 물리 공간으로 강제 저장하는 시스템 콜.
- **[Steal/No-Force]**: WAL 덕분에 채택할 수 있게 된 유연한 메모리 관리 정책.
- **[Redo Log]**: WAL의 규칙을 따르는 가장 대표적인 기록물.

📢 **마무리 요약**: **WAL Protocol**은 현대 데이터베이스의 생존 원칙입니다. "로그가 곧 데이터"라는 믿음 아래, 속도와 안전이라는 두 마리 토끼를 완벽하게 잡아낸 위대한 설계입니다.
