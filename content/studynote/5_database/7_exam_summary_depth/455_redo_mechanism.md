# 455. Redo(리두) - 멈추지 않는 데이터의 기록

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Redo(리두)는 트랜잭션이 성공적으로 완료(Commit)되었으나, 시스템 장애로 인해 **실제 데이터 파일에 아직 기록되지 못한 변경 사항을 다시 수행(Replay)하기 위해 남기는 로그**다.
> 2. **가치**: "한 번 성공한 거래는 전원이 꺼져도 절대 잃어버리지 않는다"는 **영속성(Durability)**을 보장하며, 복구 시 로그를 따라가며 데이터베이스를 최신 상태로 전진(Rollforward)시킨다.
> 3. **융합**: 온라인 Redo 로그와 아카이브 로그(Archive Log) 기술이 결합되어, 단순 장애 복구뿐만 아니라 수년 전 시점으로의 데이터 복원(PITR)까지 가능케 하는 생존 아키텍처를 구성한다.

+++

### Ⅰ. Redo의 작동 배경

- **상황**: DB는 성능을 위해 메모리(Buffer Pool)에서 작업을 하고, 디스크 쓰기는 나중에 몰아서 합니다 (No-Force 정책).
- **장애**: 데이터를 디스크에 쓰기 전 정전이 나면 메모리의 데이터는 증발합니다.
- **해결**: 데이터를 쓰기 전, 훨씬 가벼운 **변경 로그(Redo Log)**를 디스크에 먼저 순차적으로 기록해 둡니다 (WAL 프로토콜).

+++

### Ⅱ. Redo 복구 프로세스 시각화 (ASCII Flow)

```text
[ Redo Recovery Flow ]

  1. 트랜잭션 성공 (Commit) ──▶ [ Redo Log (Disk) ]에 "A=50" 기록 ✅
  2. [ Buffer Pool (Mem) ] ──▶ "A=50" 상태 (Dirty Page)
  
  ( --- 💥 SYSTEM CRASH --- )
  
  3. 재기동 후 [ Data File (Disk) ] 확인 ──▶ "A=10" (옛날 값) 😱
  4. [ Redo Log ] 스캔 ──▶ "A=50" 기록 발견!
  5. **REDO 실행** ──▶ [ Data File ]을 "A=50"으로 갱신 ✅
  "약속된 일을 끝까지 완수함"
```

+++

### Ⅲ. 온라인 Redo 로그 vs 아카이브 로그

- **Online Redo Log**: 현재 활발히 기록 중인 로그. 파일이 꽉 차면 처음부터 다시 덮어쓰는 순환(Round-robin) 구조입니다.
- **Archive Log**: 덮어쓰기 전의 온라인 로그를 별도의 안전한 장소로 복사해 둔 파일입니다. 미디어 장애(디스크 파손) 시 백업본과 합쳐져 완벽한 복구를 가능케 합니다.

- **📢 섹션 요약 비유**: Redo는 **'은행의 업무 일지'**와 같습니다. 고객에게 입금 영수증(Commit)을 끊어줬는데, 직원이 퇴근 전 장부에 옮겨 적기 전에 도둑이 들어 장부를 훼손했다면, 낮에 써둔 '업무 일지(로그)'를 보고 장부를 다시 정확하게 채워 넣는 것과 같습니다. 영수증을 준 이상, 그 돈은 반드시 통장에 기록되어야 한다는 약속을 지키는 행위입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Durability]**: Redo가 수호하는 트랜잭션의 핵심 가치.
- **[Rollforward]**: 로그를 통해 과거에서 현재로 데이터를 밀어붙이는 복구 방식.
- **[WAL]**: 로그를 먼저 써야 한다는 대원칙.

📢 **마무리 요약**: **Redo**는 데이터베이스의 강한 의지입니다. 어떤 재난 상황에서도 약속된 결과는 반드시 지켜낸다는 철학이 담긴 기록 기술입니다.
