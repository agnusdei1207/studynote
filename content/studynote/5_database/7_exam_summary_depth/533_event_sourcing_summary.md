# 533. 이벤트 소싱(Event Sourcing) - 변화의 모든 발자취

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트 소싱은 애플리케이션의 최종 상태만을 저장하는 전통적 방식과 달리, **상태를 변화시키는 모든 사건(Event)을 발생 순서대로 저장하고, 이를 재실행(Replay)하여 현재 상태를 복원하는 데이터 관리 기법**이다.
> 2. **가치**: 데이터의 **완벽한 감사 이력(Audit Trail)**을 제공하며, 과거 특정 시점으로의 복구가 용이하고, 쓰기 작업이 단순 삽입(Append-only)으로 이루어져 쓰기 성능이 매우 뛰어나다.
> 3. **융합**: CQRS 아키텍처와 결합되어 명령(Command) 계층의 영속성 모델로 사용되며, Kafka와 같은 메시지 스트리밍 플랫폼과 융합되어 실시간 비즈니스 이벤트 망을 형성한다.

+++

### Ⅰ. 이벤트 소싱의 작동 원리

- **이벤트 저장소 (Event Store)**: '주문됨', '결제됨', '배송 시작됨'과 같은 비즈니스 이벤트를 불변(Immutable) 상태로 저장합니다.
- **상태 복원 (Rehydration)**: 현재의 잔액이나 주문 상태를 알기 위해, 저장된 이벤트를 처음부터 끝까지 순차적으로 재생합니다.
- **스냅샷 (Snapshot)**: 이벤트가 너무 많아질 경우, 특정 시점의 상태를 미리 저장해두어 복원 속도를 높입니다.

+++

### Ⅱ. 이벤트 소싱 아키텍처 시각화 (ASCII Flow)

```text
[ Event Sourcing Workflow ]

  (Command) ──▶ [ Business Logic ] ──▶ [ New Event Generated ]
                                              │
          ┌───────────────────────────────────┘
          ▼
  [ Event Store (Append-only) ] ✅ "The Source of Truth"
  1. User Created (10:00)
  2. Order Placed (10:05)
  3. Payment Done (10:10)
          │
          ├─▶ (Replay) ──▶ [ Current State ] ✅ "Order Status: Paid"
          └─▶ (Stream) ──▶ [ Read Models / Dashboards ]
```

+++

### Ⅲ. 실무적 장점과 과제

- **장점**: 
    - **데이터 유실 제로**: "왜 이 수치가 나왔지?"에 대한 모든 근거가 남아있습니다.
    - **동시성 최적화**: 업데이트가 없고 삽입만 있으므로 데이터 충돌(Locking)이 거의 없습니다.
- **과제**: 
    - **조회 복잡도**: 현재 상태를 알기 위해 연산이 필요하므로 CQRS와의 결합이 필수적입니다.
    - **버전 관리**: 비즈니스 로직이 바뀌었을 때 과거 이벤트들을 어떻게 처리할 것인지(Event Versioning) 고민이 필요합니다.

- **📢 섹션 요약 비유**: 이벤트 소싱은 **'은행 통장의 거래 내역'**과 같습니다. 우리가 잔액(최종 상태)만 적힌 수첩을 들고 다니는 게 아니라, 입금과 출금 내역이 한 줄 한 줄 적힌 통장을 가진 것과 같습니다. 잔액이 의심스러울 때 처음부터 더해보고 확인(Replay)할 수 있고, 중간에 내역을 지울 수도 없는(불변) 완벽한 기록 방식입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Append-only]**: 수정과 삭제 없이 오직 추가만 하는 데이터의 성질.
- **[Replay]**: 과거 이벤트를 다시 실행하여 상태를 만드는 행위.
- **[Aggregate]**: 관련 이벤트를 묶어 비즈니스 규칙을 강제하는 단위.

📢 **마무리 요약**: **Event Sourcing**은 데이터의 '결과'가 아닌 '과정'에 집중합니다. 모든 변화의 흔적을 자산으로 만듦으로써, 가장 투명하고 강력한 비즈니스 지능을 실현합니다.
