# 398. 이행 함수 종속과 제3정규형(3NF) - 직접 종속의 원칙

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 제3정규형(3NF)은 제2정규형을 만족하면서, 일반 속성 간에 발생하는 **'이행 함수 종속(Transitive Dependency)'을 제거하여 모든 일반 속성이 기본키에 직접적으로만 종속되게 하는 과정**이다.
> 2. **가치**: 기본키가 아닌 속성들끼리의 종속 관계를 끊어냄으로써 데이터 변경 시 발생하는 부작용을 최소화하고, 데이터의 독립성과 관리 효율성을 높인다.
> 3. **융합**: "기본키를 통하지 않고는 어떠한 속성도 결정할 수 없다"는 원칙을 실현하며, 실무적인 정규화의 가장 보편적인 도달 지점이 된다.

+++

### Ⅰ. 이행 함수 종속의 정의와 위험성

- **이행 함수 종속 (Transitive Dependency)**: 속성 $A \to B$ 이고 $B \to C$ 일 때, 결과적으로 $A \to C$ 가 성립하는 경우입니다 (단, $B$ 는 기본키가 아님).
- **발생 사례**: (학번) ──▶ (학과명) ──▶ (학과전화번호). 학번을 알면 학과번호를 알 수 있지만, 이는 '학과명'을 거쳐서 알게 되는 간접적인 관계입니다.
- **위험성**: 학과 전화번호가 바뀌면 해당 학과의 모든 학생 레코드를 다 고쳐야 하며(갱신 이상), 학생이 한 명도 없는 신설 학과는 번호를 저장할 수 없는(삽입 이상) 문제가 생깁니다.

+++

### Ⅱ. 3정규화 과정 시각화 (ASCII Model)

간접적인 종속 고리를 끊어 테이블을 물리적으로 분리합니다.

```text
[ 2NF (with Transitive Dependency) ]
  ┌────────┬──────────┬─────────────┐
  │ 학번(K)│ 학과명    │ 학과번호     │
  └────┬───┴────┬─────┴─────────────┘
       │        └─▶ [ 이행 종속 ] 💥
       └──────────▶ [ 직접 종속 ] ✅

[ 3NF (Normalized) ]
  ┌────────┬──────────┐      ┌──────────┬──────────┐
  │ 학번(K)│ 학과명    │      │ 학과명(K)│ 학과번호  │
  └────────┴──────────┘      └──────────┴──────────┘
```

+++

### Ⅲ. 제3정규형(3NF)의 성립 조건

1. **제2정규형(2NF)**을 만족해야 합니다.
2. 기본키가 아닌 모든 속성 간에 **이행적 함수 종속이 없어야** 합니다.
3. 즉, 모든 비주요 속성이 기본키에 대하여 **비이행적으로 종속**되어야 합니다.

- **📢 섹션 요약 비유**: 제3정규형은 **'사촌의 친구 정리하기'**와 같습니다. 우리 집 주소록(테이블)에는 우리 가족(직접 종속) 정보만 있어야 합니다. 사촌(학과명)을 통해 알게 된 사촌의 친구(학과번호) 정보까지 우리 집 주소록에 다 적어두면, 사촌이 이사 갈 때 그 친구 연락처까지 잃어버리거나 엉망이 됩니다. 사촌의 친구 정보는 사촌네 집 주소록(학과 테이블)에 따로 적어두는 것이 맞습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Direct Dependency]**: 3NF가 지향하는 깔끔한 관계.
- **[Anomaly Prevention]**: 이행 종속 제거를 통해 얻는 가장 큰 기술적 이득.
- **[Lossless Join]**: 분해된 테이블을 다시 합쳐도 데이터 손실이 없음을 보장함.

📢 **마무리 요약**: **3NF**는 데이터의 '간접 연결'을 허용하지 않습니다. 모든 정보가 오직 자신의 주인(기본키)에게만 직접 귀속될 때, 데이터베이스는 가장 건강한 상태를 유지합니다.
