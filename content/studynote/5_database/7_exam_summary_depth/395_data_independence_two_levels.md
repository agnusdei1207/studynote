# 395. 데이터 독립성 2단계 - 논리와 물리적 변화의 차단

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 독립성은 상위 계층의 스키마를 변경하지 않고도 하위 계층의 스키마를 변경할 수 있는 성질로, **논리적 데이터 독립성**과 **물리적 데이터 독립성**의 두 단계로 구분된다.
> 2. **가치**: 응용 프로그램과 데이터 구조 사이의 의존성을 제거하여, 데이터 저장 방식이나 논리적 구조가 바뀌어도 **응용 프로그램을 수정할 필요가 없게 함으로써 유지보수 비용을 획기적으로 낮춘다.**
> 3. **융합**: ANSI/SPARC 3단계 아키텍처(외부, 개념, 내부 스키마)의 계층 간 매핑(Mapping) 기술을 통해 물리적 실체로 구현된다.

+++

### Ⅰ. 데이터 독립성의 2가지 단계

#### 1. 논리적 데이터 독립성 (Logical Data Independence)
- **정의**: 개념 스키마(전체 논리 구조)가 변경되어도 외부 스키마(사용자 뷰)에는 영향을 주지 않는 성질.
- **예시**: 새로운 테이블 컬럼이 추가되어도 기존 사용자의 조회 화면(View)은 수정하지 않아도 됨.

#### 2. 물리적 데이터 독립성 (Physical Data Independence)
- **정의**: 내부 스키마(물리적 저장 구조)가 변경되어도 개념 스키마에 영향을 주지 않는 성질.
- **예시**: 성능 향상을 위해 인덱스를 추가하거나 데이터 파일 경로를 옮겨도, 사용자가 사용하는 SQL 쿼리는 바꿀 필요가 없음.

+++

### Ⅱ. 데이터 독립성 아키텍처 시각화 (ASCII Model)

계층 사이의 **매핑(사상)** 정보가 독립성을 지탱하는 핵심입니다.

```text
[ Data Independence Mapping Layer ]

  [ External Schema ] (사용자 뷰)
          │
  ========│======== [ 논리적 데이터 독립성 ] ✅ (응용 프로그램 보호)
          │
  [ Conceptual Schema ] (전체 논리 구조)
          │
  ========│======== [ 물리적 데이터 독립성 ] ✅ (저장 효율성 보호)
          │
  [ Internal Schema ] (물리적 저장 방식)
```

+++

### Ⅲ. 독립성이 필요한 이유: 종속성(Dependency)의 타파

- **파일 시스템의 한계**: 과거에는 데이터 파일 구조가 바뀌면 모든 프로그램을 새로 짜야 했습니다. (데이터 종속성)
- **DBMS의 해법**: 데이터 독립성을 통해 데이터 관리의 유연성을 확보하고, 데이터 무결성 보장을 중앙 집중화할 수 있게 되었습니다.

- **📢 섹션 요약 비유**: 데이터 독립성은 **'전기 콘센트 표준화'**와 같습니다. 발전소(물리 계층)가 화력이든 원자력이든 바뀌어도, 혹은 집안 인테리어(논리 계층)를 바꿔도, 콘센트 모양(인터페이스)만 그대로라면 가전제품(응용 프로그램)을 그대로 꽂아 쓸 수 있는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Mapping (사상)]**: 독립성을 구현하기 위한 계층 간 변환 규칙.
- **[ANSI/SPARC]**: 데이터 독립성 이론을 정립한 3단계 아키텍처 표준.
- **[Data Abstraction]**: 복잡한 물리 구조를 숨기고 단순한 논리 구조만 보여주는 추상화 과정.

📢 **마무리 요약**: **Data Independence**는 DBMS의 존재 이유입니다. 변화무쌍한 물리/논리 환경 속에서도 서비스의 안정성을 지켜주는 데이터베이스의 가장 위대한 공학적 배려입니다.
