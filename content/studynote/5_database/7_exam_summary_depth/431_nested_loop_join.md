# 431. 중첩 루프 조인(Nested Loop Join) - 조인의 가장 기초

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 중첩 루프 조인(NL Join)은 외부 테이블(Driving)의 각 행을 하나씩 읽으며, **내부 테이블(Driven)의 인덱스를 반복적으로 탐색하여 조건에 맞는 데이터를 찾아 결합하는 방식**이다.
> 2. **가치**: 소량의 데이터를 조인할 때 응답 속도가 매우 빠르며, 인덱스가 잘 구성되어 있다면 추가적인 메모리나 정렬 없이도 효율적인 수행이 가능하다.
> 3. **융합**: '순차적 접근'과 '인덱스 랜덤 액세스'가 결합된 형태이며, 선행 테이블의 크기가 작고 후행 테이블의 조인 컬럼에 인덱스가 있을 때 최고의 효율을 낸다.

+++

### Ⅰ. 중첩 루프 조인의 작동 매커니즘

1. **외부 테이블(Outer/Driving) 스캔**: 조건에 맞는 첫 번째 행을 찾습니다.
2. **내부 테이블(Inner/Driven) 탐색**: 찾은 행의 조인 키 값을 가지고 내부 테이블의 인덱스를 뒤져 짝을 찾습니다.
3. **반복**: 외부 테이블의 모든 대상 행에 대해 위 과정을 반복합니다.

+++

### Ⅱ. NL 조인 프로세스 시각화 (ASCII Model)

```text
[ Nested Loop Join Logic ]

  Outer Table (Dept)         Inner Table (Emp)
  ┌────────────┐             ┌────────────────┐
  │ ID: 10     │──▶ (Loop 1) ──▶ [ Index on DeptID ] ──▶ ✅ Found 5 rows
  ├────────────┤             └────────────────┘
  │ ID: 20     │──▶ (Loop 2) ──▶ [ Index on DeptID ] ──▶ ✅ Found 2 rows
  └────────────┘             
  
  * For each row in Dept:
      Scan Emp Index where Dept.ID = Emp.DeptID
```

+++

### Ⅲ. 성능 최적화 포인트 (튜닝 가이드)

- **선행 테이블 선정**: 데이터량이 적은 테이블을 먼저 읽도록 순서를 조정합니다 (`/*+ ORDERED */` 등).
- **인덱스 필수**: 내부 테이블의 조인 속성에는 반드시 **인덱스**가 있어야 합니다. 없으면 매 루프마다 전체 테이블을 스캔(Full Scan)하는 대참사가 발생합니다.
- **랜덤 액세스 비용**: 조인 건수가 많아지면 내부 테이블로의 랜덤 액세스 부하가 커지므로, 이때는 해시 조인(Hash Join)으로 전환을 고려해야 합니다.

- **📢 섹션 요약 비유**: 중첩 루프 조인은 **'전화번호부에서 친구 찾기'**와 같습니다. 내 휴대폰 주소록(외부 테이블)에 있는 이름들을 하나씩 보면서, 두꺼운 종이 전화번호부(내부 테이블)의 색인(인덱스)을 뒤져서 주소를 찾아내는 것과 같습니다. 친구가 몇 명 없을 때는 이 방식이 가장 빠르지만, 친구가 만 명이라면 번호부를 만 번 펼쳐봐야 하므로 아주 힘든 작업이 됩니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Driving Table]**: 조인을 주도하는 바깥쪽 루프 테이블.
- **[Random Access]**: NL 조인의 숙명적인 물리적 특징.
- **[Single Block Read]**: 인덱스 탐색 시 발생하는 주요 I/O 방식.

📢 **마무리 요약**: **Nested Loop Join**은 소량 데이터 결합의 왕입니다. 인덱스라는 날개를 달아줄 때 가장 민첩하게 움직이는 데이터베이스의 기본 조인 알고리즘입니다.
