# 534. Saga 패턴 - 분산 환경의 유연한 트랜잭션

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사가(Saga) 패턴은 마이크로서비스(MSA)와 같은 분산 환경에서, 여러 서비스에 걸친 긴 트랜잭션을 **독립적인 로컬 트랜잭션들의 연속체로 처리하고 실패 시 보상 트랜잭션(Compensating Transaction)으로 원복하는 기법**이다.
> 2. **가치**: 2단계 커밋(2PC)의 락(Lock) 병목 문제를 해결하여 시스템의 **가용성과 확장성**을 높이며, '결과적 일관성(Eventual Consistency)' 모델을 실무에 적용한다.
> 3. **융합**: 이벤트 주도 아키텍처(EDA)와 결합되어 중앙 통제(Orchestration) 또는 자율적 협력(Choreography) 방식으로 비즈니스 정합성을 융합 관리한다.

+++

### Ⅰ. Saga 패턴의 핵심: 보상 트랜잭션

- **문제**: 분산 시스템에서는 한꺼번에 `ROLLBACK`을 날리는 것이 불가능합니다. (이미 일부 서비스는 커밋을 마쳤기 때문)
- **해결 (Compensating Transaction)**: 어떤 단계가 실패하면, 이전에 성공했던 단계들을 거꾸로 찾아가며 **반대되는 작업(예: 결제 완료 ──▶ 결제 취소)**을 수행하여 시스템을 원래의 일관된 상태로 되돌립니다.

+++

### Ⅱ. Saga 아키텍처 시각화 (ASCII Flow)

중앙에서 지휘하는 오케스트레이션 방식의 예시입니다.

```text
[ Saga Orchestration Workflow ]

  [ Saga Manager ] ─── (1) 주문 생성 ──▶ [ 주문 서비스 ] (Success ✅)
         │                                     │
         │ ◀─────── (2) 주문 완료 응답 ────────┘
         │
         │ ─── (3) 결제 요청 ──▶ [ 결제 서비스 ] (Fail! 💥)
         │                                     │
         │ ◀─────── (4) 결제 실패 응답 ────────┘
         │
         │ ─── (5) 주문 취소(보상) ──▶ [ 주문 서비스 ] ✅ "Rollback Simulated"
         ▼
  (Result) : "전체 트랜잭션 안전하게 취소됨"
```

### Ⅲ. 2PC vs Saga 패턴 비교

| 비교 항목 | 2단계 커밋 (2PC) | Saga 패턴 |
|:---|:---|:---|
| **일관성** | 강력한 일관성 (ACID) | **결과적 일관성 (BASE)** |
| **성능 (TPS)** | 낮음 (대기 시간 김) | **높음 (비동기 처리)** |
| **구현 난이도** | 낮음 (DB 엔진 지원) | **높음 (보상 로직 직접 개발)** |
| **가용성** | 낮음 (SPOF 존재) | **높음 (독립적 실행 가능)** |

- **📢 섹션 요약 비유**: Saga 패턴은 **'택배 릴레이 배달'**과 같습니다. 첫 번째 기사가 옆 동네 기사에게 물건을 넘기고(이벤트), 그 기사가 다음 기사에게 넘깁니다. 만약 마지막 동네가 폐쇄되어 배달에 실패했다면(장애), 기사들이 역순으로 다시 물건을 돌려주며(보상 트랜잭션) 최종적으로 발송인에게 물건을 되돌려주는 유연한 시스템입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Choreography]**: 중앙 통제자 없이 서비스끼리 이벤트를 주고받는 방식.
- **[Orchestration]**: 중앙 매니저가 전체 흐름을 제어하는 방식.
- **[Eventual Consistency]**: 사가가 지향하는 데이터 정합성 철학.

📢 **마무리 요약**: **Saga Pattern**은 MSA의 생존 전략입니다. 완벽한 동기화의 환상을 버리고, 유연한 보상 로직을 통해 거대한 분산 시스템의 평화를 지켜내는 고도의 설계 기법입니다.
