# 445. 갱신 손실(Lost Update) - 사라져버린 데이터의 비극

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 갱신 손실은 두 개 이상의 트랜잭션이 동일한 데이터를 동시에 수정할 때, **먼저 완료된 트랜잭션의 수정 사항이 나중에 완료된 트랜잭션에 의해 덮어씌워져 사라지는 현상**이다.
> 2. **가치**: 데이터베이스 정합성 파괴의 가장 치명적인 형태이며, 고립성(Isolation) 원칙을 정면으로 위반하여 비즈니스 로직(재고 부족, 잔액 불일치 등)에 막대한 오류를 초래한다.
> 3. **융합**: 낙관적 락(Optimistic Lock)과 비관적 락(Pessimistic Lock) 기술이 융합되어 이를 방어하며, 현대 분산 시스템의 최종 쓰기 승리(LWW) 정책의 리스크를 대변한다.

+++

### Ⅰ. 갱신 손실의 발생 시나리오

1. **동시 판독**: 트랜잭션 T1과 T2가 동시에 데이터 A(값: 100)를 읽습니다.
2. **개별 수정**: T1은 A에 50을 더하고(150), T2는 A에 30을 더합니다(130).
3. **덮어쓰기**: T1이 먼저 150을 저장하고 커밋합니다. 직후 T2가 130을 저장하고 커밋합니다.
4. **결과**: 최종값은 130이 되며, T1의 "+50" 연산은 영원히 사라집니다. (기대값: 180)

+++

### Ⅱ. 갱신 손실 타임라인 시각화 (ASCII Flow)

```text
[ Lost Update Timeline ]

  Time | Transaction T1 (Add 50) | Transaction T2 (Add 30) | DB Value
  -----|-------------------------|-------------------------|---------
   t1  | Read A (100)            |                         |   100
   t2  |                         | Read A (100)            |   100
   t3  | A = 100 + 50 (150)      |                         |   100
   t4  | Update A = 150 & Commit |                         |   150
   t5  |                         | A = 100 + 30 (130)      |   150
   t6  |                         | Update A = 130 & Commit |   130 💥
  -------------------------------------------------------------------
  * Result: T1's update is LOST. Final value is incorrect!
```

+++

### Ⅲ. 해결을 위한 방어 전략

- **비관적 락 (Pessimistic Lock)**: 데이터를 읽는 순간 `FOR UPDATE`로 잠가버려, 남이 읽지도 못하게 막습니다.
- **낙관적 락 (Optimistic Lock)**: 데이터에 '버전(Version)'을 두고, 수정 시 내가 읽은 버전과 현재 DB 버전이 같을 때만 수정을 허용합니다.
- **원자적 연산**: `SET balance = balance + 50` 처럼 현재 DB 값을 기준으로 연산하는 쿼리를 사용하여 덮어쓰기 위험을 줄입니다.

- **📢 섹션 요약 비유**: 갱신 손실은 **'공동 문서 편집기의 저장 덮어쓰기'**와 같습니다. 내가 한 시간 동안 공들여 보고서를 고쳐서 저장했는데(T1), 마침 옆자리 동료도 똑같은 문서를 열어놓고 오타 하나 고친 뒤 저장 버튼을 누르는 바람에(T2), 내가 공들인 한 시간 분량의 내용이 싹 날아가고 동료가 고친 오타 한 자만 남게 되는 허망한 상황입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Isolation Anomaly]**: 고립성이 깨졌을 때 나타나는 대표적 증상.
- **[Last Writer Wins]**: 갱신 손실을 허용하는 대신 속도를 택하는 분산 시스템 정책.
- **[Compare and Swap (CAS)]**: 낙관적 락을 구현하는 하드웨어 수준의 원자적 연산.

📢 **마무리 요약**: **Lost Update**는 시스템의 신뢰를 갉아먹는 암세포와 같습니다. 정교한 Locking 전략을 통해 데이터의 '생존권'을 보장하는 것이 아키텍트의 숙명입니다.
