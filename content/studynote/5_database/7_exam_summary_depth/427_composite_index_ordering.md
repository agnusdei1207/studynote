# 427. 결합 인덱스(Composite Index) - 순서가 만드는 성능

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 결합 인덱스는 두 개 이상의 컬럼을 합쳐서 하나의 인덱스로 구성한 것으로, **인덱스 생성 시 지정한 컬럼의 순서에 따라 데이터가 정렬되고 검색 효율이 결정**되는 특수한 다중 컬럼 인덱스다.
> 2. **가치**: 여러 개의 단일 인덱스를 만드는 것보다 저장 공간과 쓰기 부하를 줄이면서도, 복합 조건절을 가진 쿼리의 성능을 극적으로 향상시킨다.
> 3. **융합**: '가장 먼저 정렬될 기준'을 맨 앞에 두는 전략이 융합되어야 하며, 선행 컬럼이 조건절에 누락될 경우 인덱스 전체가 무용지물이 될 수 있는 강력한 제약 조건을 가진다.

+++

### Ⅰ. 결합 인덱스의 정렬 원리

- **계층적 정렬**: 첫 번째 컬럼으로 먼저 정렬하고, 첫 번째 컬럼 값이 같으면 두 번째 컬럼으로 정렬하는 방식입니다.
- **Access Rule**: 반드시 인덱스의 **첫 번째 컬럼(Leading Column)이 조건절(`WHERE`)에 포함**되어야 인덱스 스캔이 시작됩니다.

+++

### Ⅱ. 결합 인덱스 구조 시각화 (ASCII Model)

인덱스가 `(Dept, Name)` 순서로 생성된 경우의 데이터 정렬 모습입니다.

```text
[ Composite Index: (Dept, Name) ]

  ┌────────┬──────────┐
  │ Dept   │ Name     │
  ├────────┼──────────┤
  │ CS     │ Alice    │ ✅ (Dept 우선 정렬)
  │ CS     │ Bob      │
  │ Math   │ Kim      │ ✅ (Dept 변경 시 Name 다시 정렬)
  │ Math   │ Zack     │
  └────────┴──────────┘

  [ Query Efficiency ]
  1. WHERE Dept='CS' AND Name='Bob'  ──▶ Best Performance (Both used) ✅
  2. WHERE Dept='CS'                 ──▶ High Performance (Leading used) ✅
  3. WHERE Name='Bob'                ──▶ Index Not Used ❌ (Leading missing)
```

+++

### Ⅲ. 최적의 컬럼 순서 선정 가이드

1. **카디널리티 (Cardinality)**: 선택도(Selectivity)가 높은 컬럼(중복이 적은 것)을 앞에 두는 것이 유리합니다.
2. **조회 빈도**: 자주 사용되는 컬럼을 우선순위로 배치합니다.
3. **연산자 특성**: `=` 조건으로 쓰이는 컬럼을 앞에, `BETWEEN` 이나 `>` 같은 범위 연산 컬럼을 뒤에 배치해야 인덱스 스캔 범위를 최소화할 수 있습니다.

- **📢 섹션 요약 비유**: 결합 인덱스는 **'사전의 단어 배열'**과 같습니다. 사전은 '알파벳 첫 글자' 순으로 정렬된 뒤, 그 안에서 '두 번째 글자' 순으로 정렬됩니다. 내가 'Apple'을 찾으려면 'A'라는 첫 글자(선행 컬럼)를 알아야만 사전 인덱스를 쓸 수 있습니다. 첫 글자를 모른 채 '두 번째 글자가 p인 단어'를 찾으려 하면 사전 전체를 다 뒤져야 하는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Leading Column]**: 인덱스의 성패를 쥐고 있는 첫 번째 컬럼.
- **[Index Skip Scan]**: 선행 컬럼이 없어도 인덱스를 타게 돕는 특수 기술 (성능은 낮음).
- **[Selectivity]**: 컬럼이 데이터를 얼마나 잘 걸러내는지 나타내는 수치.

📢 **마무리 요약**: **Composite Index**는 정교한 설계의 산물입니다. 비즈니스 쿼리의 흐름을 완벽히 이해하고 컬럼의 '순서'를 정할 때, 인덱스는 비로소 최고의 성능을 발휘합니다.
