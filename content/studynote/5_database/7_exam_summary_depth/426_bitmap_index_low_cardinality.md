# 426. 비트맵 인덱스(Bitmap Index) - 비트로 계산하는 검색

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비트맵 인덱스는 데이터의 각 고유 값에 대해 **비트 배열(Bit Vector)을 생성하고, 특정 조건에 해당하는 행의 비트만 1로 표시하여 데이터를 인덱싱하는 기술**이다.
> 2. **가치**: 값의 종류가 적은(Low Cardinality) 컬럼에 특화되어 있으며, 여러 조건이 결합된 쿼리를 **비트 연산(AND, OR, NOT)만으로 초고속 처리**할 수 있어 데이터 웨어하우스(DW) 환경에 최적이다.
> 3. **융합**: 압축 기술과 결합되어 대용량 데이터에서도 저장 공간을 적게 차지하며, 다차원 분석(OLAP) 도구의 핵심 조회 엔진으로 융합된다.

+++

### Ⅰ. 비트맵 인덱스의 작동 원리

- **구성**: 각 컬럼 값마다 하나의 비트맵이 존재합니다.
- **매핑**: 테이블의 $n$번째 행이 해당 값을 가지면 $n$번째 비트를 1로, 아니면 0으로 세팅합니다.
- **연산**: "서울 지역(1010) AND 남성(1100)" 쿼리가 오면 CPU는 즉시 비트 연산을 수행하여 결과(1000)를 도출합니다.

+++

### Ⅱ. 비트맵 인덱스 구조 시각화 (ASCII Model)

```text
[ Data Table ]           [ Bitmap Index: Gender ]
  Row 1: Male              Male   : 1 0 1 1 0
  Row 2: Female            Female : 0 1 0 0 1
  Row 3: Male
  Row 4: Male            [ Bitmap Index: Region ]
  Row 5: Female            Seoul  : 1 1 0 0 0
                           Busan  : 0 0 1 1 1

[ Multi-Condition Query ]
  "Gender='Male' AND Region='Seoul'"
  Male (10110) AND Seoul (11000) = (10000) ──▶ ✅ Row 1 Found!
```

+++

### Ⅲ. 장점 및 한계

- **장점**: 
    - 저장 공간 효율성: 값의 종류가 적을수록 인덱스 크기가 매우 작습니다.
    - 고속 연산: 복잡한 논리 조건을 하드웨어 레벨의 비트 연산으로 해결합니다.
- **한계**: 
    - **수정 부하**: 한 행만 수정해도 해당 블록의 비트맵 전체에 락이 걸려 쓰기 성능이 매우 낮습니다. (OLTP 부적합)
    - **High Cardinality 취약**: 값의 종류가 수백만 개면 비트맵 개수도 수백만 개가 되어 관리가 불가능합니다.

- **📢 섹션 요약 비유**: 비트맵 인덱스는 **'출석부의 체크 표시'**와 같습니다. 이름 대신 '남학생 출석부', '여학생 출석부'를 따로 둔 뒤, 해당되는 칸에만 체크(1)를 해두는 것입니다. "안경 쓴(1010) 남학생(1100)"을 찾으려면 두 출석부를 겹쳐보고 둘 다 체크된 칸만 확인하면 되는 아주 빠르고 명쾌한 방식입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Low Cardinality]**: 비트맵 인덱스가 가장 사랑하는 데이터 특성.
- **[Bitwise Operation]**: CPU 친화적인 비트맵의 검색 엔진.
- **[Read-Only/DW]**: 비트맵 인덱스가 주로 서식하는 환경.

📢 **마무리 요약**: **Bitmap Index**는 분석의 효율성을 위해 태어났습니다. 비트라는 단순한 언어로 거대한 데이터의 바다를 가로지르는 지능형 탐색 기술입니다.
