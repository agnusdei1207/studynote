# 415. DDL(Data Definition Language) - 구조의 창조와 파괴

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DDL(데이터 정의 언어)은 데이터베이스의 논리적/물리적 구조인 **스키마(Schema)를 정의, 수정, 삭제하기 위해 사용하는 SQL 명령어**들의 집합이다.
> 2. **가치**: 데이터가 담길 '그릇'을 만드는 역할을 하며, 실행 즉시 데이터 사전(Data Dictionary)에 반영되어 시스템의 골격을 확정 짓는다.
> 3. **특징**: 대부분의 DBMS에서 **자동 커밋(Auto-commit)** 방식으로 작동하여 롤백(Rollback)이 불가능하므로, 운영 환경에서 사용 시 극도로 신중해야 하는 명령어군이다.

+++

### Ⅰ. DDL의 주요 명령어와 기능

| 명령어 | 기능 | 특징 |
|:---|:---|:---|
| **CREATE** | 테이블, 인덱스, 뷰 등 객체 **생성** | 제약 조건(PK, FK) 정의 포함 |
| **ALTER** | 기존 객체의 구조 **변경** | 컬럼 추가/삭제, 타입 변경 |
| **DROP** | 객체를 완전히 **삭제** | 스키마와 데이터 모두 소멸 |
| **TRUNCATE** | 테이블의 데이터만 **고속 삭제** | 공간은 유지하되 데이터만 즉시 초기화 |
| **RENAME** | 객체의 명칭 변경 | 논리적 이름 수정 |

+++

### Ⅱ. DDL 실행의 물리적 효과 (ASCII Flow)

DDL 명령은 서비스 데이터가 아닌 '시스템 카탈로그'를 건드리는 무거운 작업입니다.

```text
[ DDL Execution Workflow ]

  (Admin SQL) : "DROP TABLE Backup_Log;"
       │
       ▼ [ Database Engine ]
  ┌───────────────────────────────────────────┐
  │ 1. Metadata Lock (접근 차단)               │ 🔒
  │ 2. Update Data Dictionary (카탈로그 갱신)   │
  │ 3. Physical File Delete (공간 회수)        │
  └───────────────────────────────────────────┘
       │
       ▼
  (System Catalog) : "Backup_Log table removed" ✅ (Auto-committed)
```

+++

### Ⅲ. DML과의 결정적 차이: 롤백 불가

- **트랜잭션 제외**: DML(INSERT 등)은 실수하면 `ROLLBACK`으로 되돌릴 수 있지만, DDL은 실행하는 순간 DB 엔진이 강제로 `COMMIT`을 날립니다.
- **TRUNCATE vs DELETE**: 
    - `DELETE`: 로그를 하나하나 남기며 지움 (느림, 롤백 가능).
    - `TRUNCATE`: 로그 없이 데이터 블록 자체를 초기화 (매우 빠름, **롤백 절대 불가**).

- **📢 섹션 요약 비유**: DDL은 **'건축 공사'**와 같습니다. 집 안에 가구를 들이는 것(DML)은 마음대로 위치를 바꿀 수 있지만, 벽을 허물거나(DROP) 방을 하나 더 만드는(ALTER) 것은 한 번 하면 되돌리기 어렵고 집 전체 구조(스키마)에 큰 영향을 주는 것과 같습니다. 공사를 시작하기 전 설계도를 수만 번 확인해야 하는 이유입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Auto-Commit]**: DDL의 가장 무서운 성질.
- **[Metadata Lock]**: DDL 수행 시 일반 사용자의 조회를 막는 잠금 현상.
- **[Schema Evolution]**: DDL을 통해 DB를 조금씩 발전시켜 나가는 과정.

📢 **마무리 요약**: **DDL**은 데이터베이스의 헌법과 같습니다. 국가의 기틀을 세우듯, 데이터의 틀을 정의하는 DDL 명령어 하나하나에 시스템의 운명이 달려 있습니다.
