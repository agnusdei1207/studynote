# 512. 버퍼 히트율(Buffer Cache Hit Ratio) - 메모리 활용의 성적표

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 버퍼 히트율은 사용자가 요청한 데이터가 물리적인 디스크가 아닌, **데이터베이스 메모리 영역(Buffer Cache)에 이미 존재하여 즉시 반환된 비율**을 나타내는 성능 지표다.
> 2. **가치**: 디스크 I/O라는 가장 느린 병목 구간을 얼마나 효과적으로 회피하고 있는지를 보여주며, 시스템 전체의 응답 속도와 처리량을 결정짓는 핵심 척도가 된다.
> 3. **융합**: 메모리 크기(SGA/PGA) 설정과 SQL 튜닝 기술이 융합된 결과물이며, 통상적으로 **95~99% 이상의 높은 수치를 유지**하는 것이 고성능 DB 운영의 목표다.

+++

### Ⅰ. 버퍼 히트율의 계산 원리

- **공식**: $Hit Ratio = \left( 1 - \frac{\text{Physical Reads}}{\text{Logical Reads}} \right) \times 100$
    - **Logical Reads**: 메모리에서 읽은 총 횟수 (메모리 히트 + 디스크 읽기 후 메모리 적재).
    - **Physical Reads**: 메모리에 없어 디스크에서 직접 읽어온 횟수.
- **의미**: 100번의 데이터 요청 중 몇 번이나 "메모리에서 바로 찾았는가"를 의미합니다.

+++

### Ⅱ. 버퍼 캐시 작동 매커니즘 시각화 (ASCII Model)

```text
[ Data Request Flow ]

  (User SQL) ──▶ [ Database Buffer Cache (RAM) ]
                          │
          ┌───────────────┴───────────────┐
          ▼ (Found! ✅)                   ▼ (Not Found! ❌)
    [ CACHE HIT ]                   [ CACHE MISS ]
    - Speed: Nanoseconds            - Speed: Milliseconds 💥
    - Result Delivered              - Physical Disk Read ──▶ [ Disk ] 💾
                                    - Load to Cache
```

+++

### Ⅲ. 히트율이 낮을 때의 원인과 대책

1. **메모리 부족**: 버퍼 캐시 크기가 너무 작아 자주 쓰이는 데이터가 계속 밀려나는 경우 (LRU 알고리즘). ──▶ **메모리 증설**.
2. **비효율적 SQL**: 인덱스를 타지 않고 전체 테이블을 읽어(Full Scan) 메모리 공간을 한꺼번에 오염시키는 경우. ──▶ **SQL 튜닝**.
3. **대량 배치 작업**: 평소엔 좋다가 대규모 이관 작업 시 일시적으로 급락할 수 있습니다. ──▶ **워크로드 분리**.

- **📢 섹션 요약 비유**: 버퍼 히트율은 **'요리사의 재료 손질 상태'**와 같습니다. 손님이 주문했을 때(쿼리) 필요한 재료가 이미 손질되어 도마 위에 있다면(Cache Hit) 요리는 순식간에 완성됩니다. 하지만 재료가 없어 멀리 있는 시장(Disk)까지 다녀와야 한다면(Cache Miss) 손님은 한참을 기다려야 합니다. 도마가 얼마나 큰지(메모리 크기), 요리사가 얼마나 미리 준비를 잘하는지(SQL 튜닝)가 식당의 성적표(히트율)를 결정합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[LRU (Least Recently Used)]**: 버퍼 캐시를 관리하는 가장 대중적인 알고리즘.
- **[Logical I/O vs Physical I/O]**: 히트율을 계산하는 두 가지 I/O 관점.
- **[Full Table Scan]**: 버퍼 히트율을 떨어뜨리는 주범 중 하나.

📢 **마무리 요약**: **Buffer Cache Hit Ratio**는 DB 튜닝의 나침반입니다. 90% 아래로 떨어지는 히트율은 시스템의 위기 신호이며, 이를 올리기 위한 메모리와 쿼리의 조화가 고성능 아키텍처의 핵심입니다.
