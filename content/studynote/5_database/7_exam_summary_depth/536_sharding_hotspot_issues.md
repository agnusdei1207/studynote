# 536. 샤드 키 불균형과 핫스팟 - 분산의 적, 쏠림 현상

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 샤드 키 불균형은 부적절한 샤드 키(Shard Key) 선정으로 인해 **특정 노드에만 데이터와 트래픽이 집중되는 현상**으로, 해당 노드가 병목 지점이 되어 시스템 전체의 성능을 저하시키는 **핫스팟(Hotspot)** 문제를 유발한다.
> 2. **가치**: 균등한 분산(Uniform Distribution)의 중요성을 환기하며, 순차적 키(날짜, 증가하는 ID) 사용의 위험성과 카디널리티(Cardinality) 분석의 필요성을 공학적으로 입증한다.
> 3. **융합**: 솔팅(Salting) 기술, 해시 샤딩(Hash Sharding), 그리고 동적 리밸런싱 기술이 융합되어, 고정된 물리적 한계를 넘어 유연하고 균형 잡힌 분산 저장 체계를 구축한다.

+++

### Ⅰ. 핫스팟(Hotspot)의 발생 원인과 증상

- **주요 원인**: 
    - **저카디널리티 키**: 값의 종류가 너무 적어 특정 샤드에 데이터가 쏠림. (예: 성별, 거주 리전)
    - **순차적 키 (Monotonic Keys)**: 시간순이나 숫자순 키를 쓰면 최신 데이터가 항상 마지막 샤드에만 저장되어 쓰기 부하가 집중됨.
- **주요 증상**: 
    - 특정 노드만 CPU/IO 100% 점유. 💥
    - 다른 노드들은 유휴 상태인데도 전체 시스템 응답 속도가 느려짐.
    - 특정 샤드의 저장 공간 부족으로 인한 장애 발생.

+++

### Ⅱ. 핫스팟 발생 및 해결(Salting) 시각화 (ASCII Model)

```text
[ Hotspot Case: Time-based Key ]
  (Writes) ──▶ [ Shard 1 ] [ Shard 2 ] [ Shard 3 ] ◀── 💥 10:00~11:00 데이터 집중!
                 (Empty)      (Empty)      (BUSY!)

[ Solution: Salting (Random Prefix) ] ✅
  Raw Key: "10:00" ──▶ [ Hash/Salt ] ──▶ "A-10:00", "B-10:00", "C-10:00"
                                              │          │          │
          ┌───────────────────────────────────┴──────────┼──────────┘
          ▼                                              ▼
    [ Shard 1 ]                [ Shard 2 ]         [ Shard 3 ]
    (ID: A-*)                  (ID: B-*)           (ID: C-*)
    ✅ "Evenly Distributed"     ✅ "Balanced"       ✅ "Balanced"
```

+++

### Ⅲ. 핫스팟 대처 및 예방 전략

1. **해시 샤딩 (Hash Sharding)**: 키 값을 해싱하여 무작위로 분산. (범위 검색은 포기해야 함)
2. **솔팅 (Salting)**: 키 앞에 무작위 접두사(Salt)를 붙여 강제로 쪼개는 기법.
3. **복합 샤드 키 (Compound Key)**: 카디널리티가 낮은 키와 높은 키를 결합하여 유일성을 확보.
4. **동적 리밸런싱 (Dynamic Rebalancing)**: 부하를 감지하여 핫스팟이 생긴 샤드의 데이터를 한가한 노드로 자동 이동 (MongoDB Balancer 등).

- **📢 섹션 요약 비유**: 샤드 키 불균형은 **'대형 마트의 특정 계산대만 줄이 길게 늘어선 상황'**과 같습니다. 입구(샤드 키)에서 손님들을 "오늘 생일인 분은 1번 계산대, 아닌 분은 2번" 식으로 나누면(저카디널리티), 1번 계산대만 터져나가고 2번은 노는 상황이 벌어집니다. 이를 막으려면 번호표를 무작위로 나눠주거나(Hash), 줄이 긴 곳의 손님을 옆으로 옮겨주는(Rebalancing) 똑똑한 안내가 필요합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Cardinality]**: 샤드 키 선정 시 가장 먼저 고려해야 할 '값의 다양성'.
- **[Data Skew]**: 데이터가 한쪽으로 기울어져 있는 물리적 상태.
- **[Hot Shard]**: 트래픽 폭주로 비명을 지르고 있는 특정 서버 노드.

📢 **마무리 요약**: **Sharding Hotspot**은 분산 시스템의 고질병입니다. 데이터의 성질을 깊이 있게 이해하고 '공평한 분배'의 규칙을 설계할 때, 시스템은 비로소 무한한 확장성을 누릴 수 있습니다.
