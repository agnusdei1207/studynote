# 424. 클러스터드 인덱스(Clustered Index) - 데이터의 물리적 질서

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클러스터드 인덱스는 인덱스의 리프 노드(Leaf Node)가 곧 실제 데이터 페이지이며, **인덱스 키의 순서대로 데이터가 물리적으로 정렬되어 저장되는 인덱스**다.
> 2. **가치**: 데이터가 이미 정렬되어 있으므로 범위 검색(Range Scan)과 특정 값 탐색 시 추가적인 데이터 페이지 이동(Fetch) 없이 즉시 결과를 반환하는 **최고의 조회 성능**을 제공한다.
> 3. **융합**: 테이블당 단 하나만 존재할 수 있으며(물리적 정렬의 유일성), 주로 기본키(Primary Key) 설정 시 자동으로 생성되어 데이터베이스 접근의 주 경로(Main Path)가 된다.

+++

### Ⅰ. 클러스터드 인덱스의 작동 원리

- **데이터와 인덱스의 일치**: 인덱스가 곧 테이블 그 자체입니다. 별도의 인덱스 파일이 있는 게 아니라, 테이블의 데이터 블록을 인덱스 구조로 재배열합니다.
- **리프 노드의 정체**: 넌클러스터드 인덱스의 리프 노드는 '주소'를 담고 있지만, 클러스터드 인덱스의 리프 노드는 **'실제 행(Row) 데이터'**를 통째로 담고 있습니다.
- **물리적 정렬**: 새로운 데이터가 삽입되면 인덱스 순서에 맞는 물리적 위치를 찾아 끼워 넣습니다. (Page Split 발생 가능)

+++

### Ⅱ. 클러스터드 인덱스 구조 시각화 (ASCII Model)

```text
[ Clustered Index Structure ]

          [ Root Node ]
         /      |      \
  [Branch 1] [Branch 2] [Branch 3]
     /   \      /   \      /   \
  ┌───────┐┌───────┐┌───────┐┌───────┐
  │Data 1 ││Data 3 ││Data 5 ││Data 7 │ 
  │Data 2 ││Data 4 ││Data 6 ││Data 8 │ 
  └───────┘└───────┘└───────┘└───────┘
  (Leaf Nodes are the ACTUAL DATA PAGES) ✅
  (Physical Order: 1, 2, 3, 4, 5, 6, 7, 8) ✅
```

+++

### Ⅲ. 장점과 주의사항

- **장점**: 
    - 조회(특히 Range Query) 성능이 압도적으로 빠릅니다.
    - 데이터 페이지를 한 번만 읽으면 모든 컬럼을 다 가져올 수 있습니다.
- **주의사항**: 
    - 삽입/수정/삭제 시 물리적 정렬을 유지해야 하므로 오버헤드가 큽니다.
    - 키 값이 큰 컬럼을 클러스터드 키로 잡으면, 이를 참조하는 모든 보조 인덱스의 크기가 덩달아 커지는 부작용이 있습니다.

- **📢 섹션 요약 비유**: 클러스터드 인덱스는 **'사전 그 자체'**와 같습니다. 단어들이 가나다순으로 이미 정렬되어 본문이 쓰여 있으므로, 단어를 찾으면 그 자리에서 바로 뜻(데이터)을 읽을 수 있습니다. 책 뒤의 인덱스(보조 인덱스)를 보고 본문을 다시 펼칠 필요가 없는 가장 효율적인 구조입니다. 단, 중간에 새 단어를 넣으려면 책 페이지들을 다 밀어내야 하는 수고가 따릅니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Primary Key]**: 클러스터드 인덱스의 가장 흔한 주인.
- **[Page Split]**: 정렬 순서대로 데이터를 넣기 위해 페이지가 쪼개지는 현상.
- **[Sequential I/O]**: 클러스터드 인덱스가 지향하는 고속 입출력 방식.

📢 **마무리 요약**: **Clustered Index**는 데이터베이스 성능 설계의 기본 중의 기본입니다. 가장 자주 조회되는 기준을 클러스터드 인덱스로 설정하는 것만으로도 시스템 전체 성능을 비약적으로 끌어올릴 수 있습니다.
