# 462. 2단계 커밋(2PC) - 분산 트랜잭션의 합의

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 2단계 커밋(2PC)은 물리적으로 분리된 여러 데이터베이스 노드 간의 트랜잭션 원자성을 보장하기 위해, **모든 노드가 준비 완료를 보고했을 때만 최종 커밋을 수행하는 동기식 조율 프로토콜**이다.
> 2. **가치**: "전체가 성공하거나 아니면 전체가 실패한다"는 ACID 원칙을 분산 환경에서 실현하며, 글로벌 트랜잭션 매니저(GTM)를 통해 파편화된 데이터의 정합성을 수호한다.
> 3. **융합**: 조정자(Coordinator)와 참여자(Participant) 모델로 작동하며, 현대의 XA 표준 인터페이스를 통해 이기종 데이터베이스 간의 통합 트랜잭션을 가능케 한다.

+++

### Ⅰ. 2단계 커밋의 상세 프로세스

1. **제1단계: 준비 (Prepare Phase)**:
    - 조정자가 모든 노드에게 "커밋 가능합니까?"라고 질의합니다.
    - 각 노드는 락(Lock)을 획득하고 작업을 준비한 뒤 "Ready" 혹은 "No"로 응답합니다.
2. **제2단계: 커밋 (Commit Phase)**:
    - **전원 Ready인 경우**: 조정자가 최종 "COMMIT" 명령을 하달합니다.
    - **한 명이라도 No인 경우**: 조정자가 전체 "ROLLBACK"을 지시합니다.

+++

### Ⅱ. 2PC 아키텍처 및 장애 상황 시각화 (ASCII Flow)

```text
[ 2PC Normal Commitment ]

  [ Coordinator ]           [ Node A ]           [ Node B ]
        │                      │                    │
        │ ── 1. PREPARE ────▶  │                    │
        │ ── 1. PREPARE ────────────────────────▶  │
        │                      │                    │
        │ ◀── 2. READY ──────  │                    │
        │ ◀── 2. READY ──────────────────────────  │
        │                      │                    │
        │ ── 3. COMMIT! ────▶  │                    │
        │ ── 3. COMMIT! ────────────────────────▶  │
        ▼                      ✅                   ✅
```

+++

### Ⅲ. 2PC의 한계: 블로킹(Blocking) 문제

- **동기적 대기**: 코디네이터가 준비 완료 후 갑자기 다운되면, 모든 참여 노드는 지시를 기다리며 해당 자원의 락을 쥔 채 무한 대기 상태에 빠집니다.
- **가용성 저하**: 네트워크 지연이나 노드 장애에 매우 취약하여 시스템 전체의 응답 속도를 떨어뜨리는 원인이 됩니다.
- **대안**: 이를 해결하기 위해 3단계 커밋(3PC)이나 비동기 방식인 **Saga 패턴** 등이 사용되기도 합니다.

- **📢 섹션 요약 비유**: 2PC는 **'단체 결혼식의 주례'**와 같습니다. 주례 선생님(조정자)이 신랑, 신부 수십 쌍에게 한 명씩 "동의하십니까?"라고 묻습니다(Prepare). 단 한 명이라도 "아니오!"라고 대답하거나, 화장실에 가서 대답이 없으면 결혼식 전체가 중단되고 아무도 부부가 될 수 없습니다(Rollback). 모두가 "예!"라고 해야만 비로소 혼인 선언(Commit)이 이루어집니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Global Transaction ID]**: 분산 트랜잭션을 식별하는 고유 번호.
- **[XA Standard]**: 분산 트랜잭션을 위한 산업 표준 규격.
- **[Blocking]**: 2PC의 가장 치명적인 아킬레스건.

📢 **마무리 요약**: **2-Phase Commit**은 분산 데이터의 정석입니다. 무겁고 느리지만, 완벽한 원자성을 요구하는 금융 결제 등의 핵심 업무에서는 여전히 가장 신뢰받는 프로토콜입니다.
