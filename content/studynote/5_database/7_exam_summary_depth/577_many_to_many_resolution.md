# 577. 다대다(N:M) 관계 해소 - 교차 릴레이션의 탄생

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다대다(N:M) 관계 해소는 관계형 데이터베이스에서 직접 구현이 불가능한 두 엔티티 간의 복잡한 관계를, **교차 릴레이션(Intersection Entity / Mapping Table)을 도입하여 두 개의 일대다(1:N) 관계로 분해**하는 과정이다.
> 2. **가치**: 데이터의 중복을 방지하고 참조 무결성을 유지하며, 관계 자체에 별도의 속성(예: 수강 신청의 '성적', 주문의 '수량')을 부여할 수 있는 유연한 설계를 가능케 한다.
> 3. **융합**: 논리적 모델링의 관계 정의와 물리적 스키마의 제약 조건이 융합되는 지점이며, 모든 관계형 DB 설계의 필수적인 정규화 단계다.

+++

### Ⅰ. 다대다 관계의 문제점

- **직접 구현 불가**: 관계형 모델은 하나의 컬럼에 여러 값을 담을 수 없으므로(제1정규형 위반), 어느 한쪽에 외래키를 두는 방식으로는 N:M 관계를 표현할 수 없습니다.
- **데이터 중복 및 이상 현상**: 무리하게 합치면 엄청난 데이터 중복과 삽입/삭제 이상이 발생합니다.

+++

### Ⅱ. 교차 릴레이션을 이용한 분해 시각화 (ASCII Model)

```text
[ Logical: N:M Relation ]
  [ Student ]  M <──(Register)──> N  [ Course ]

[ Physical: Intersection Entity ] ✅
  ┌───────────┐ (1)      (N) ┌─────────────┐ (N)      (1) ┌───────────┐
  │  Student  │──────────────│  Enrollment │──────────────│  Course   │
  │ - Stud_ID │              │ - Stud_ID(FK)│              │ - Cour_ID │
  └───────────┘              │ - Cour_ID(FK)│              └───────────┘
                             │ - Grade      │ ✅ "Attribute of relation"
                             └─────────────┘
```

+++

### Ⅲ. 교차 릴레이션 설계의 특징

1. **복합키 (Composite Key) 구성**: 보통 두 부모 테이블의 기본키(PK)를 가져와 자신의 기본키로 삼습니다.
2. **관계 속성 보관**: 학생도 아니고 과목도 아닌, '수강'이라는 관계에서만 발생하는 데이터(예: 수강 시점, 점수)를 저장하는 장소가 됩니다.
3. **참조 무결성**: 두 부모 테이블과 강력한 외래키(FK) 관계를 맺어, 유령 데이터의 발생을 원천 차단합니다.

- **📢 섹션 요약 비유**: 다대다 관계 해소는 **'결혼식의 혼인 서약서'**와 같습니다. 신랑(엔티티 A)과 신부(엔티티 B)는 서로 다대다(수많은 인연) 관계가 가능하지만, 법적(DB)으로 이를 증명하려면 두 사람의 이름이 동시에 적힌 '서약서(교차 릴레이션)' 한 장이 필요합니다. 서약서에는 결혼 날짜나 장소(관계 속성)를 적을 수 있고, 이 종이 한 장 덕분에 두 사람의 관계가 명확하게 정리되는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Associative Entity]**: 교차 릴레이션의 또 다른 학술적 명칭.
- **[Mapping Table]**: 개발 실무에서 주로 부르는 용어.
- **[Referential Integrity]**: 교차 테이블을 통해 지켜지는 핵심 무결성 가치.

📢 **마무리 요약**: **N:M Decomposition**은 관계의 실체화입니다. 추상적인 연결을 구체적인 테이블로 변환함으로써, 데이터베이스는 비로소 복잡한 현실 세계를 정교하게 담아낼 수 있습니다.
