# 583. 파라미터 스니핑(Parameter Sniffing) - 최적화의 역습

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파라미터 스니핑은 옵티마이저가 저장 프로시저의 첫 실행 시 입력된 **파라미터 값을 훔쳐보고(Sniffing), 그 값에 최적화된 실행 계획을 수립하여 캐시에 저장하는 매커니즘**이다.
> 2. **가치**: 대다수의 경우 성능 향상에 기여하지만, 데이터 분포가 불균형한 경우 **처음 들어온 특이값에 고정된 실행 계획**이 일반적인 다른 호출 시 성능을 급격히 떨어뜨리는 부작용을 낳는다.
> 3. **융합**: 실행 계획 재사용 기술과 통계 기반 최적화 기술이 융합된 결과물이며, 이를 해결하기 위해 적응형 쿼리 실행(Adaptive Query Execution)이나 재컴파일 힌트가 활용된다.

+++

### Ⅰ. 파라미터 스니핑의 양면성

- **긍정적 효과**: 리터럴 SQL과 달리 바인드 변수를 사용하는 프로시저는 파싱 부하를 줄여야 합니다. 옵티마이저는 실제 값을 보고 "아, 이 값이면 인덱스를 타는 게 좋겠군"이라고 판단하여 최적의 계획을 세웁니다.
- **부정적 효과 (캐시 오염)**:
    - 처음 들어온 값이 전체의 0.01%인 경우 ──▶ **인덱스 스캔** 계획 수립 및 캐싱. ✅
    - 다음 값이 전체의 90%인 경우 ──▶ 캐시된 **인덱스 스캔** 사용 ──▶ 💥 대량 랜덤 I/O 발생으로 성능 폭락.

+++

### Ⅱ. 파라미터 스니핑 장애 매커니즘 시각화 (ASCII Model)

```text
[ Parameter Sniffing & Plan Pollution ]

  Step 1: First Run (Param: 'VIP')
  - Data Distribution: 'VIP' is very rare (1 row).
  - Optimizer: "Index Scan is Best!" ──▶ [ Cache Plan: INDEX SCAN ] ✅

  Step 2: Second Run (Param: 'NORMAL')
  - Data Distribution: 'NORMAL' is very common (1,000,000 rows).
  - Executor: "Use Cached Plan!" ──▶ [ Execution: INDEX SCAN ] 💥
  
  * Result: 100만 건을 인덱스로 하나씩 읽느라 시스템 마비! 😱
```

+++

### Ⅲ. 해결 방안 및 대응 기술

1. **RECOMPILE 힌트**: 호출 시마다 실행 계획을 새로 짜도록 강제합니다. (CPU 부하 증가 감수)
2. **OPTIMIZE FOR 힌트**: 특정 값이나 평균적인 분포를 기준으로 실행 계획을 세우도록 옵티마이저에게 가이드를 줍니다.
3. **로컬 변수 사용**: 프로시저 내부에서 파라미터를 로컬 변수에 다시 담아 사용하면, 옵티마이저가 값을 스니핑하지 못하고 평균 통계치를 사용하게 됩니다.
4. **Adaptive Cursor Sharing**: 오라클 등에서 지원하는 기능으로, 파라미터에 따라 여러 개의 실행 계획을 캐시에 유지하고 선택적으로 사용합니다. ✅

- **📢 섹션 요약 비유**: 파라미터 스니핑은 **'첫인상으로 선입견 갖기'**와 같습니다. 요리사(옵티마이저)가 첫 손님(첫 파라미터)이 아주 조금 먹는 것을 보고 소량의 재료만 준비해뒀는데(인덱스 스캔 계획), 다음 손님이 단체 관광객(대량 데이터)으로 왔음에도 불구하고 "첫 손님이 조금 먹었으니 이번에도 조금일 거야"라고 고집을 부리며 준비를 안 하다가 주방이 마비되는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Bind Peeking]**: 파라미터 스니핑의 기술적 원인 행위.
- **[Plan Stability]**: 힌트를 통해 실행 계획을 고정하여 스니핑 리스크를 제거하는 전략.
- **[Data Skew]**: 데이터 쏠림 현상이 심할수록 스니핑 문제는 치명적이 됨.

📢 **마무리 요약**: **Parameter Sniffing**은 효율을 위한 지혜가 독이 된 사례입니다. 데이터의 분포를 명확히 이해하고, 상황에 맞는 쿼리 재사용 전략을 수립하는 것이 고성능 DB 운영의 묘미입니다.
