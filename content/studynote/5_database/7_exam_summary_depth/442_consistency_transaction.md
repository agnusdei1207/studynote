# 442. 일관성(Consistency) - 데이터 정합성의 수호

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 일관성은 트랜잭션이 성공적으로 수행된 후에도 데이터베이스가 **미리 정의된 모든 무결성 제약 조건(Constraints)과 비즈니스 규칙을 만족하며, 논리적으로 모순이 없는 상태를 유지**해야 함을 의미한다.
> 2. **가치**: "트랜잭션 실행 전이 정상이면, 실행 후도 정상이어야 한다"는 원칙을 통해, 데이터 간의 수치적 합계나 관계의 정당성을 보장하여 시스템의 신뢰를 유지한다.
> 3. **융합**: 데이터 무결성(Integrity) 관리 시스템과 연동되며, 애플리케이션의 비즈니스 로직과 DB의 물리적 제약 조건이 융합되어 달성된다.

+++

### Ⅰ. 일관성의 주요 보장 대상

- **무결성 제약 조건**: 도메인 제약, 개체 무결성(PK), 참조 무결성(FK) 등.
- **명시적 비즈니스 규칙**: "총 잔액의 합계는 변하지 않아야 한다", "재고는 음수가 될 수 없다" 등.
- **상태 전이 정당성**: 한 상태에서 다음 상태로 변할 때 정의된 규칙을 위반하지 않음.

+++

### Ⅱ. 일관성 유지 과정 시각화 (ASCII Model)

```text
[ Consistent State Transition ]

  (Consistent State 1) ──▶ [ Transaction ] ──▶ (Consistent State 2)
  - A + B = 1000           - A transfer 100    - A + B = 1000 (여전히 동일) ✅
  - PK: Unique             - Commit            - PK: Unique (여전히 유일) ✅

  * 만약 트랜잭션 도중 규칙이 깨지면?
  ──▶ [ DBMS Intervention ] ──▶ ROLLBACK ──▶ (Consistent State 1) ✅
```

+++

### Ⅲ. 일관성 확보의 주체

| 주체 | 역할 |
|:---|:---|
| **DBMS (데이터베이스)** | 기본키, 외래키, CHECK 제약 조건 등을 강제하여 물리적 일관성 유지 |
| **개발자 (응용 프로그램)** | 복잡한 비즈니스 로직(예: 선착순 정원 관리)을 트랜잭션으로 묶어 논리적 일관성 유지 |

- **📢 섹션 요약 비유**: 일관성은 **'장부 결산'**과 같습니다. 매일 돈이 들어오고 나가는 거래(트랜잭션)가 수천 건 발생해도, 장부를 마감할 때 "들어온 돈과 나간 돈의 차이가 금고의 잔액과 딱 맞아야 한다"는 원칙이 지켜지는 것입니다. 만약 1원이라도 틀리면 장부 전체를 다시 확인(Rollback)하여 항상 정답인 상태를 유지하는 꼼꼼함입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Integrity Constraint]**: 일관성을 지탱하는 DB 레벨의 약속들.
- **[Validation]**: 트랜잭션 종료 시 일관성을 검증하는 과정.
- **[Strong Consistency]**: 분산 환경에서 모든 노드가 즉시 동일한 상태가 되는 고차원 일관성.

📢 **마무리 요약**: **Consistency**는 데이터의 '옳음'을 증명합니다. 단순한 값의 저장을 넘어, 데이터 간의 유기적인 질서를 유지하는 것이 일관성의 핵심입니다.
