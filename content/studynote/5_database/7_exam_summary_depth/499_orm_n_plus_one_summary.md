# 499. ORM과 N+1 문제 - 객체와 관계의 간극

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ORM(Object-Relational Mapping)은 객체 지향 언어의 객체와 관계형 데이터베이스의 테이블을 자동으로 매핑해주는 기술이며, **N+1 문제**는 1번의 쿼리로 N개의 데이터를 가져왔을 때 연관된 데이터를 조회하기 위해 **N번의 추가 쿼리가 발생하는 성능 저하 현상**이다.
> 2. **가치**: SQL 중심의 개발에서 비즈니스 로직 중심의 개발로 패러다임을 전환하여 생산성을 높이지만, 내부적인 동작 원리를 모를 경우 시스템 성능을 심각하게 훼손할 수 있는 양날의 검이다.
> 3. **융합**: JPA(Java Persistence API), Hibernate 등의 표준 기술과 결합되어 사용되며, **패치 조인(Fetch Join)**이나 **배치 사이즈(Batch Size)** 최적화 기술을 통해 N+1 문제를 융합 해결한다.

+++

### Ⅰ. N+1 문제의 발생 원인: 지연 로딩 (Lazy Loading)

- **원인**: 부모 엔티티를 조회할 때 자식 엔티티를 즉시 가져오지 않고, 실제 사용할 때마다 쿼리를 날려 가져오기 때문에 발생합니다.
- **상황**: "모든 게시글(10개)을 조회하고, 각 게시글의 작성자 이름을 출력하라."
- **결과**:
    1. 게시글 전체 조회 쿼리 (1번)
    2. 각 게시글마다 작성자 조회 쿼리 (10번)
    ──▶ **총 11번 (1 + N)의 쿼리 발생!** 💥

+++

### Ⅱ. N+1 문제 시각화 및 해결책 (ASCII Model)

```text
[ N+1 Problem Flow ]                     [ Optimized: Fetch Join ]

  User SELECT * FROM Board;                User SELECT b FROM Board b 
     │                                     JOIN FETCH b.author;
     ▼ (Returns 10 Boards)                    │
  For each board:                             ▼ (Returns everything)
     SELECT * FROM User WHERE id=?; 💥        ONE SINGLE SQL QUERY! ✅
     (Looping 10 times...)                    (All data in one go)
```

+++

### Ⅲ. 주요 해결 전략

1. **Fetch Join (패치 조인)**: SQL의 `JOIN`을 사용하여 연관된 엔티티를 한꺼번에 조회합니다. (가장 확실한 방법)
2. **Entity Graph**: 어노테이션이나 설정으로 특정 시점에 함께 조회할 대상을 지정합니다.
3. **Batch Size 설정**: `WHERE id IN (...)` 쿼리를 사용하여 N번의 쿼리를 1번으로 묶어 처리합니다. (In-clause 활용)

- **📢 섹션 요약 비유**: ORM의 N+1 문제는 **'심부름센터의 비효율'**과 같습니다. 시장에서 물건 10개를 사오라고 시켰는데(1번 쿼리), 심부름꾼이 물건 하나를 집어 올 때마다 "이거 포장지는 어떻게 할까요?"라고 매번 집에 전화를 걸어 물어보는(N번 추가 쿼리) 것과 같습니다. 시장에 가기 전 미리 포장지 옵션까지 다 정해서 알려주면(Fetch Join) 전화 한 통 없이 한꺼번에 일을 끝낼 수 있는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Impedance Mismatch]**: 객체 지향과 관계형 모델 간의 근본적인 불일치.
- **[Persistence Context]**: 영속성 컨텍스트. ORM이 엔티티를 관리하는 1차 캐시 공간.
- **[JPQL]**: 객체 지향 쿼리 언어. N+1 문제 해결을 위한 패치 조인의 주무대.
