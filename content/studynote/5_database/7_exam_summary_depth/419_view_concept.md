# 419. 뷰(VIEW) - 논리적으로 존재하는 가상 테이블

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 뷰(View)는 물리적으로 데이터를 저장하지 않고, **하나 이상의 기본 테이블로부터 유도된 가상의 테이블(Virtual Table)**로, SQL 질의를 미리 정의해 둔 저장된 쿼리이다.
> 2. **가치**: 복잡한 조인(Join)이나 계산 로직을 단순화하여 사용자에게 제공하며, 민감한 컬럼을 제외한 뷰를 생성함으로써 강력한 **데이터 보안 및 접근 제어** 수단으로 활용된다.
> 3. **융합**: 논리적 데이터 독립성을 구현하는 핵심 도구이며, 성능 향상을 위해 결과를 물리적으로 저장하는 구체화된 뷰(Materialized View) 기술로 확장 융합된다.

+++

### Ⅰ. 뷰(View)의 주요 특징

- **가상성**: 물리적인 저장 공간을 차지하지 않습니다 (정의만 데이터 사전에 저장).
- **독립성**: 원본 테이블 구조가 바뀌어도 뷰 정의만 수정하면 응용 프로그램을 고칠 필요가 없습니다 (논리적 독립성).
- **제약성**: 인덱스 생성이 불가능하며, 집계 함수 등이 포함된 복잡한 뷰는 데이터 삽입/수정/삭제가 제한됩니다.

+++

### Ⅱ. 뷰의 작동 아키텍처 시각화 (ASCII Model)

뷰는 원본 데이터를 바라보는 '창(Window)'과 같습니다.

```text
[ View Layered Architecture ]

  (User View: "Emp_Basic") : [ ID, Name, Dept ]
          │
  ========│======== [ VIEW Definition ] ✅ (Pre-defined SQL)
          │
  [ Base Table: Employee ] (Physical Data)
  - ID, Name, Social_ID(Hidden), Dept, Salary(Hidden)
  ┌────┬──────┬──────────┬──────┬────────┐
  │ ID │ Name │ SocialID │ Dept │ Salary │ 💾 (Disk)
  └────┴──────┴──────────┴──────┴────────┘
```

+++

### Ⅲ. 뷰 사용의 장점과 단점

| 장점 | 단점 |
|:---|:---|
| **보안 강화**: 필요한 컬럼만 노출 | **성능 저하**: 쿼리 실행 시마다 정의된 SQL이 재실행됨 |
| **복잡성 은닉**: 긴 Join 문을 단순 조회로 변경 | **갱신 제약**: 모든 뷰를 통해 데이터를 수정할 순 없음 |
| **데이터 독립성**: 구조 변경에 유연하게 대응 | **인덱스 부재**: 검색 속도 최적화에 한계 |

- **📢 섹션 요약 비유**: 뷰는 **'가전제품의 리모컨'**과 같습니다. 텔레비전 내부의 복잡한 회로(기본 테이블)를 우리가 다 알 필요는 없습니다. 리모컨의 전원, 채널 버튼(뷰의 컬럼)만 누르면 기계가 알아서 내부 회로를 작동시켜 화면을 보여주듯, 우리는 뷰라는 간편한 인터페이스를 통해 복잡한 데이터를 쉽게 다룰 수 있습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Logical Data Independence]**: 뷰가 제공하는 최고의 아키텍처적 가치.
- **[Base Table]**: 뷰의 근간이 되는 실제 물리 테이블.
- **[Materialized View]**: 뷰의 결과를 디스크에 저장하여 성능을 높인 특수 뷰.

📢 **마무리 요약**: **View**는 데이터베이스의 추상화 계층입니다. 복잡함을 숨기고 보안을 높임으로써, 사용자가 데이터를 더 쉽고 안전하게 소비할 수 있는 창구를 제공합니다.
