# 590. 클럭 스큐와 구글 트루타임 - 시간 오차의 정복

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클럭 스큐(Clock Skew)는 분산 시스템의 노드 간 물리적 시간이 서로 어긋나는 현상이며, 구글 트루타임(TrueTime)은 이를 해결하기 위해 **원자시계와 GPS를 활용하여 시간 오차 범위를 명시적으로 제공하는 고정밀 시간 API**다.
> 2. **가치**: 글로벌 분산 환경에서도 "누가 먼저 기록했는가"를 절대적으로 판별할 수 있게 하며, 인위적인 대기 시간(Commit Wait)을 통해 **전역적 강력한 일관성(External Consistency)**을 보장한다.
> 3. **융합**: 하드웨어 정밀 시계와 소프트웨어 합의 알고리즘이 융합되어, 거리와 네트워크 지연을 극복하는 현대 분산 데이터베이스의 가장 정교한 시간 관리 기법이 된다.

+++

### Ⅰ. 클럭 스큐(Clock Skew)의 난제

- **물리적 한계**: 수정 진동자의 미세한 차이, 온도 변화 등으로 인해 모든 서버의 시계는 조금씩 다르게 흐릅니다. (NTP로도 수십 ms 오차 발생)
- **데이터 위험**: 서버 A(10:00:01)와 서버 B(10:00:02)가 있을 때, 실제로는 A가 먼저 데이터를 썼음에도 B의 시계가 더 빨라 B가 먼저인 것처럼 기록되어 데이터가 꼬일 수 있습니다.

+++

### Ⅱ. 구글 트루타임(TrueTime) 작동 시각화 (ASCII Model)

트루타임은 현재 시간이 아닌, 현재 시간이 포함된 **'범위'**를 반환합니다.

```text
[ TrueTime API Concept ]

  1. Query: "지금 몇 시니?"
  2. Answer: [ Earliest: 10:00:01.001, Latest: 10:00:01.007 ] ✅
             (Uncertainty Window: 6ms)

[ Global Commit Process ]
  - Node A: 쓴다! ──▶ Commit Time = T_latest (10:00:01.007)
  - **Commit Wait**: ⏳ 10:00:01.007 이 확실히 지날 때까지 대기
  - Finish: "이제 전 세계 누구도 이 시간보다 과거의 데이터를 볼 수 없음" ✅
```

+++

### Ⅲ. 트루타임의 물리적 인프라

- **원자 시계 (Atomic Clocks)**: 시간이 흐르는 속도가 일정하며, 주기적으로 다른 원자 시계들과 대조하여 오차를 보정합니다.
- **GPS 수신기**: 위성으로부터 정밀한 시간 신호를 받아 절대 시각을 유지합니다.
- **이중화**: 각 데이터 센터마다 두 장비를 모두 설치하여 장애에 대비합니다.

- **📢 섹션 요약 비유**: 클럭 스큐와 트루타임은 **'전 세계 동시 퀴즈 대회의 타이머'**와 같습니다. 각 나라의 시계가 조금씩 다르면 누가 먼저 버튼을 눌렀는지 판정 시비가 붙습니다. 구글은 모든 도시에 '완벽하게 일치하는 정밀 시계(TrueTime)'를 보급하고, 만약 시계가 1초 정도 틀릴 가능성이 있다면 1초를 억지로 기다렸다가(Commit Wait) 결과를 발표함으로써, 누가 진짜 1등인지 아무도 이의를 제기할 수 없게 만드는 완벽한 심판 시스템을 구축한 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Uncertainty Window ($\epsilon$)]**: 트루타임이 보장하는 시간 오차의 폭.
- **[Commit Wait]**: 일관성을 위해 오차 범위만큼 인위적으로 멈추는 행위.
- **[Google Spanner]**: 트루타임 기술을 세계 최초로 상용화한 데이터베이스.

📢 **마무리 요약**: **TrueTime**은 분산 시스템의 철학적 승리입니다. 시간의 불확실성을 인정하고 이를 기술적으로 통제함으로써, 지구 전체를 하나의 데이터베이스로 묶는 위대한 도약을 이루어냈습니다.
