# 483. ETL 병목 현상 - 적재 전 변환의 한계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ETL 병목은 데이터를 목적지(DW)에 넣기 전 별도의 중간 서버에서 **복잡한 정제 및 변환(Transform) 과정을 거쳐야 하므로 발생하는 시간적/자원적 지체 현상**이다.
> 2. **가치**: 데이터의 신선도(Freshness)를 떨어뜨리는 주범이며, 특히 폭발적으로 증가하는 비정형 데이터를 처리할 때 중간 변환 서버의 하드웨어 한계로 인해 시스템 전체의 가동을 멈추게 할 수 있다.
> 3. **융합**: 이를 해결하기 위해 컴퓨팅 자원이 풍부한 타겟 DB 내부에서 변환을 수행하는 **ELT** 방식이나, 실시간 로그 기반의 **CDC** 기술이 대안으로 융합되고 있다.

+++

### Ⅰ. ETL 병목이 발생하는 3대 포인트

1. **중간 서버 부하**: 수백만 건의 데이터를 정규화하고 중복 제거하는 연산이 단일 ETL 서버의 CPU/RAM을 고갈시킵니다.
2. **네트워크 전송 지연**: 추출(E) 후 변환(T) 서버로 보내고, 다시 적재(L) 서버로 보내는 두 번의 대량 데이터 이동이 발생합니다.
3. **배치 윈도우(Batch Window) 초과**: 데이터량이 너무 많아 밤새도록 돌려도 업무 시작 전까지 적재를 끝내지 못하는 상황이 발생합니다.

+++

### Ⅱ. ETL 병목 아키텍처 시각화 (ASCII Model)

```text
[ Traditional ETL Bottleneck ]

  (Source DB) ──▶ [ Network ] ──▶ [ ETL Server ] ──▶ [ Network ] ──▶ (Target DW)
       │                             │                                  │
       │                       💥 [ TRANSFORM ] 💥                      │
       │                       (CPU / Memory Spike)                     │
       └──────────────────────────────┴─────────────────────────────────┘
                                 Time Lag (Hours/Days)
```

+++

### Ⅲ. 병목 해결을 위한 현대적 전략

- **ELT 전환**: "일단 담고(Load), 타겟의 MPP(대규모 병렬 처리) 엔진으로 구워라(Transform)."
- **증분 추출 (Incremental Load)**: 전체를 다 가져오지 않고 바뀐 부분만 가져와 변환 부하를 줄입니다.
- **스트리밍 ETL**: 배치 단위가 아닌 데이터 한 건 한 건을 실시간으로 흐르게 하여(Kafka 등 활용) 지연을 분산시킵니다.

- **📢 섹션 요약 비유**: ETL 병목은 **'식당 주방의 좁은 조리대'**와 같습니다. 재료(데이터)는 트럭째 들어오는데, 요리사가 좁은 조리대(ETL 서버)에서 일일이 껍질을 까고 다듬느라(변환) 손님 상(DW)에 요리가 나가는 시간이 계속 늦어지는 것과 같습니다. 이 문제를 해결하려면 재료를 일단 큰 식탁(Data Lake)에 다 올려두고 여러 명의 요리사가 동시에 달려들어 조리하는 방식이 필요합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Batch Window]**: ETL이 허용되는 시간적 범위.
- **[Staging Area]**: 병목을 완화하기 위해 임시로 데이터를 쌓아두는 공간.
- **[MPP (Massive Parallel Processing)]**: ETL 병목을 힘으로 밀어붙여 해결하는 현대적 컴퓨팅 기술.

📢 **마무리 요약**: **ETL Bottleneck**은 데이터 엔지니어링의 고질적인 숙제입니다. 인프라의 한계를 기술적 선후 관계(ELT)의 변화로 극복하는 것이 데이터 시대를 살아가는 영리한 전술입니다.
