# 454. Undo(언두) - 과거를 기억하는 복구의 힘

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Undo(언두)는 데이터가 변경될 때 **변경 전의 원본 데이터를 보관하는 임시 저장소(Undo Segment)**로, 트랜잭션의 취소(Rollback)와 데이터 일관성 유지를 위해 사용된다.
> 2. **가치**: 사용자가 작업을 취소하면 즉시 원상 복구할 수 있는 **원자성(Atomicity)**을 제공하며, MVCC 환경에서 다른 사용자에게 '마지막 커밋 상태'의 데이터를 보여주는 조회 원천이 된다.
> 3. **융합**: Redo 로그가 장애 발생 후 '다시 실행'하기 위한 미래 지향적 기록이라면, Undo는 '되돌리기' 위한 과거 지향적 기록으로 트랜잭션 관리의 핵심 균형을 이룬다.

+++

### Ⅰ. Undo의 2대 핵심 역할

1. **트랜잭션 롤백 (Rollback)**: 
    - 트랜잭션 수행 도중 오류가 나거나 사용자가 취소 명령을 내리면, Undo 영역에 저장된 '이전 값'을 가져와 실제 데이터 블록을 원상태로 되돌립니다.
2. **읽기 일관성 (Read Consistency)**: 
    - 트랜잭션 A가 데이터를 수정 중일 때, 다른 트랜잭션 B가 같은 데이터를 조회하면 수정 중인 값이 아닌 Undo 영역의 '커밋된 이전 값'을 제공합니다.

+++

### Ⅱ. Undo 데이터 흐름 시각화 (ASCII Model)

```text
[ Undo Process: Update Scenario ]

  1. Original Data: [X = 100]
  2. Transaction Starts: UPDATE SET X = 200
       │
       ├─▶ [ Undo Segment ] ◀── 기존 값 '100'을 복사해 둠 ✅
       └─▶ [ Data Block ]   ◀── 새 값 '200'으로 덮어씀
       │
  3. 다른 사용자 조회 ──▶ [ Undo Segment ]에서 '100'을 읽어감 (일관성) ✅
       │
  4. 💥 ROLLBACK 명령 ──▶ [ Undo Segment ]의 '100'을 다시 [ Data Block ]에 씀
```

+++

### Ⅲ. Undo 관리 시 주의사항 (Snapshot Too Old)

- **공간 재활용**: Undo 영역은 한정된 크기를 순환하며 사용합니다. 커밋된 트랜잭션의 Undo 데이터는 다른 데이터에 의해 덮어씌워질 수 있습니다.
- **오류 발생**: 만약 어떤 트랜잭션이 너무 길어져서, 그 트랜잭션이 읽어야 할 과거 데이터가 이미 다른 작업에 의해 덮어씌워졌다면 `ORA-01555 (Snapshot too old)` 에러가 발생합니다.

- **📢 섹션 요약 비유**: Undo는 **'포토샵의 작업 내역(History)'**과 같습니다. 그림을 그리다가 실수하면 `Ctrl+Z`를 눌러 이전 상태로 돌아가게 해주는 고마운 기능입니다. 또한 내가 그림을 고치는 동안 다른 사람들에게 "방금 전까지는 이랬어"라고 이전 그림을 보여주는 전시실 역할도 수행합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Rollback Segment]**: Undo 데이터가 저장되는 물리적 공간.
- **[Atemporal Consistency]**: 시간이 흘러도 변하지 않는 논리적 일관성.
- **[Steal Policy]**: 커밋 안 된 데이터를 디스크에 쓸 수 있게 하여 Undo를 필수로 만드는 버퍼 정책.

📢 **마무리 요약**: **Undo**는 데이터베이스의 안전장치입니다. 모든 변화의 그림자를 남김으로써, 언제든 진실된 과거로 돌아갈 수 있는 신뢰의 길을 열어줍니다.
