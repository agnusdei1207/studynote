# 441. 원자성(Atomicity) - 전부 아니면 전무

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 원자성은 트랜잭션과 관련된 작업들이 **더 이상 분할할 수 없는 하나의 논리적 단위**임을 보장하며, 내부의 모든 연산이 완벽히 성공하거나 아니면 아예 수행되지 않은 상태(All or Nothing)여야 함을 의미한다.
> 2. **가치**: 시스템 장애나 중단 시 데이터가 '하다 만' 중간 상태로 남는 것을 방지하여, 데이터베이스의 **논리적 무결성**을 수호한다.
> 3. **융합**: DBMS의 **회복 관리자(Recovery Manager)**가 담당하며, Undo 로그와 롤백(Rollback) 매커니즘을 통해 물리적으로 구현된다.

+++

### Ⅰ. 원자성의 주요 특징

- **원자적 단위**: 여러 개의 SQL 문장이 하나의 트랜잭션으로 묶여, 외부에서는 마치 하나의 거대한 명령처럼 보입니다.
- **상태 전이**: 트랜잭션은 오직 **Commit(완료)** 또는 **Abort(중단/취소)** 두 가지 결과만 가질 수 있습니다.
- **책임 주체**: 애플리케이션이 `Commit`을 호출하기 전까지는 모든 변경이 임시 상태이며, 오류 발생 시 DBMS가 원상 복구를 책임집니다.

+++

### Ⅱ. 원자성 보장 매커니즘 시각화 (ASCII Flow)

```text
[ Atomicity: Rolling back to Consistency ]

  (Start) ──▶ [ SQL 1: Money Out ] ✅
                 │
                 ▼ [ SQL 2: Money In ] ──▶ 💥 CRASH!
                 │
  ┌──────────────┴──────────────┐
  │  Atomicity Check (Recovery) │
  └──────────────┬──────────────┘
                 ▼
  [ UNDO Process ] ──▶ [ SQL 1 취소 ] ──▶ (Original State) ✅
  "반쪽짜리 거래는 허용하지 않는다!"
```

+++

### Ⅲ. 실무적 의의: 복구의 근거

- **Undo 로그**: 변경 전 데이터를 미리 적어두어 원자성을 지키는 생명줄입니다.
- **장애 대응**: 정전 등으로 서버가 꺼졌을 때, 재부팅 후 커밋 마크가 없는 작업들을 모두 지워버리는 근거가 원자성 원칙입니다.

- **📢 섹션 요약 비유**: 원자성은 **'폭탄 해제 작업'**과 같습니다. 선을 여러 개 자르는 복잡한 과정 중 하나라도 실패하면(장애), 작업이 성공한 게 아니라 폭탄이 터져서 아무것도 안 한 것보다 못한 상태가 됩니다. 이를 막기 위해 "완벽하게 해제하거나, 아니면 처음부터 건드리지 않은 상태로 두라"고 명령하는 엄격한 규칙입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Commit/Rollback]**: 원자성을 실현하는 수단.
- **[Shadow Paging]**: 원자성을 구현하는 또 다른 기술적 방법.
- **[ACID]**: 원자성이 첫 번째 글자를 차지하는 트랜잭션 4대 속성.

📢 **마무리 요약**: **Atomicity**는 데이터베이스 신뢰의 마지노선입니다. 어떠한 불완전한 결과도 허용하지 않는 이 단호함이 데이터의 순수성을 유지합니다.
