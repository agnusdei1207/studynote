# 556. 복제 지연(Replication Lag) - 읽기 불일치의 극복

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 복제 지연은 마스터(Master) 노드에 기록된 데이터가 네트워크 지연이나 슬레이브(Slave)의 처리 부하로 인해 **복제본에 반영되기까지 발생하는 시간차**를 의미한다.
> 2. **가치**: "방금 쓴 글이 조회 결과에 나타나지 않는" **읽기 후 쓰기(Read-your-writes) 불일치** 문제를 유발하며, 시스템 가용성과 데이터 정합성 사이의 트레이드오프를 결정짓는 핵심 변수다.
> 3. **융합**: 준동기 복제(Semi-sync), 지능형 라우팅, 그리고 세션 일관성 기술이 융합되어 사용자 경험을 해치지 않는 수준의 실무적 정합성 망을 구축한다.

+++

### Ⅰ. 복제 지연이 발생하는 원인

- **네트워크 지연**: 물리적 거리가 먼 리전 간 복제 시 패킷 전송 시간 증가.
- **슬레이브 병목**: 슬레이브 서버의 사양이 낮거나, 읽기 트래픽이 너무 많아 복제 로그를 처리할 여력이 없는 경우.
- **대량의 쓰기**: 마스터에서 한꺼번에 수백만 건을 수정하면, 이를 하나씩 따라가는 슬레이브는 뒤로 밀릴 수밖에 없습니다 (Single-threaded SQL thread 한계).

+++

### Ⅱ. 읽기 불일치 현상 시각화 (ASCII Flow)

```text
[ Replication Lag Disaster ]

  1. User A: Write(ID=1, Val='Hi') ──▶ [ Master ] ──▶ Success! ✅
                                         │
  2. Master ──▶ (Syncing...) ──▶ [ Slave ] 💥 (Lag: 2 seconds)
                                         │
  3. User A: Read(ID=1)? ────────────▶ [ Slave ] ──▶ Result: Empty ❌
  "분금 썼는데 왜 없지? 멘붕!" 😱
```

+++

### Ⅲ. 불일치 이슈 극복을 위한 대응 전략

1. **준동기 복제 (Semi-synchronous)**: 마스터가 쓰기 성공을 알리기 전, 최소 하나의 슬레이브가 로그를 받았음을 확인할 때까지 기다립니다. (지연 감소 효과)
2. **Read-your-own-writes (세션 일관성)**: 자신이 수정한 데이터에 대해서는 일정 시간 동안 강제로 **마스터 노드**에서 읽게 하여 불일치를 체감하지 못하게 합니다. ✅
3. **지능형 라우팅**: 복제 지연 시간(Seconds_Behind_Master)을 상시 체크하여, 지연이 심한 슬레이브는 로드 밸런서에서 즉시 제외합니다.
4. **멀티 스레드 복제**: 슬레이브가 로그를 읽어 적용할 때 여러 스레드를 병렬로 사용하여 처리 속도를 높입니다.

- **📢 섹션 요약 비유**: 복제 지연은 **'생방송과 TV 중계 사이의 시차'**와 같습니다. 야구장에서 홈런이 터진 순간(Master Write), 집에서 TV로 보는 사람(Slave Read)은 통신 속도 때문에 1~2초 뒤에야 환호하게 됩니다. 만약 옆집에서 환호성 소리가 들렸는데 우리 집 TV에는 아직 타자가 준비 중이라면(읽기 불일치), 시청자는 혼란에 빠집니다. 이를 막기 위해 중요한 순간에는 야구장 중계석(마스터)에서 직접 정보를 듣게 하는 것이 해결책인 셈입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Eventual Consistency]**: 복제 지연이 허용된 시스템의 최종 정합성 상태.
- **[Seconds_Behind_Master]**: 복제 지연 정도를 나타내는 대표적 모니터링 지표.
- **[Binary Log]**: 복제 지연의 원천 데이터가 담긴 로그 파일.

📢 **마무리 요약**: **Replication Lag**는 분산 DB의 숙명입니다. 기술적 완벽함보다는 사용자 시나리오에 맞는 정교한 **라우팅 전략**을 통해 데이터의 신뢰를 유지해야 합니다.
