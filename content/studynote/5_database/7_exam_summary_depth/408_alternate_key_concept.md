# 408. 대체 키 (Alternate Key) - 든든한 후보 식별자

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 대체 키는 후보키(Candidate Key)들 중에서 **기본키(Primary Key)로 선정되지 못하고 남은 나머지 키**들을 말하며, 기본키와 마찬가지로 데이터의 유일성과 최소성을 모두 만족한다.
> 2. **가치**: 기본키를 대신하여 데이터를 식별할 수 있는 **보조 식별자** 역할을 수행하며, 시스템 설계 시 유니크 인덱스(Unique Index) 설정을 통해 정합성을 보조하는 데 사용된다.
> 3. **융합**: 기본키가 물리적 인덱싱과 관계 맺기에 집중한다면, 대체 키는 논리적 데이터의 유일성을 중복 보장하는 안전장치로 융합되어 작동한다.

+++

### Ⅰ. 대체 키의 정의 및 성립 과정

- **후보키 (Candidate Key)**: 튜플을 유일하게 식별할 수 있는 최소한의 속성 집합들의 모임.
- **기본키 (Primary Key)**: 후보키 중 설계자가 대표로 선정한 단 하나의 키.
- **대체 키 (Alternate Key)**: `후보키 - 기본키`. 즉, 아쉽게 2등을 한 키들이지만 능력(식별력)은 기본키와 동등합니다.

+++

### Ⅱ. 키의 관계 구조 시각화 (ASCII Model)

```text
[ Relationship of Key Types ]

  ┌─────────────────────────────────────────────────┐
  │  Candidate Keys (후보키 전체)                     │
  │  ┌──────────────────────┬──────────────────────┐  │
  │  │  Primary Key (1등)   │  Alternate Keys (2등) │  │
  │  │  (기본키)            │  (대체 키 / 보조키)    │  │
  │  └──────────────────────┴──────────────────────┘  │
  └─────────────────────────────────────────────────┘

  * 예시: 학생(학번, 주민번호, 이름...)
    - 후보키: {학번}, {주민번호}
    - 기본키 선정: {학번}
    - 대체 키 확정: {주민번호} ✅
```

+++

### Ⅲ. 실무적 활용과 주의사항

- **보조 인덱스 활용**: 대체 키 속성에 `UNIQUE` 제약 조건을 걸면, DBMS는 자동으로 인덱스를 생성하여 기본키만큼은 아니더라도 매우 빠른 검색 성능을 보장합니다.
- **기본키 변경 시 대안**: 만약 운영 도중 기본키 체계를 바꿔야 하는 상황이 오면, 이미 유일성이 보증된 대체 키가 가장 강력한 다음 기본키 후보가 됩니다.
- **NULL 비허용 권장**: 학술적으로 대체 키는 NULL을 가질 수 없으나(후보키의 일종이므로), 실무 DB 엔진의 `UNIQUE` 제약 조건은 NULL을 허용하는 경우가 많아 설계 시 주의해야 합니다.

- **📢 섹션 요약 비유**: 대체 키는 **'국가대표 선발전의 2위 선수'**와 같습니다. 1위 선수(기본키)가 메인 경기에 출전하지만, 2위 선수(대체 키)도 실력은 1위와 대등하여 언제든 주전을 대신해 경기를 뛸 수 있는 능력을 갖추고 있는 것과 같습니다. 비록 대표는 아니지만, 팀(테이블)의 전력을 든든하게 받쳐주는 실력자입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Candidate Key]**: 대체 키의 모태가 되는 전체 후보군.
- **[Unique Constraint]**: 대체 키의 유일성을 물리적으로 강제하는 수단.
- **[Secondary Key]**: 실무에서 대체 키를 부르는 또 다른 이름 (보조키).

📢 **마무리 요약**: **Alternate Key**는 기본키의 든든한 조력자입니다. 데이터의 유일성을 다각도에서 검증함으로써, 시스템의 신뢰도를 한 층 더 끌어올리는 역할을 수행합니다.
