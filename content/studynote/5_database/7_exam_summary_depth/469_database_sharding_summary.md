# 469. 샤딩(Sharding)과 파티셔닝 - 수평 확장의 필살기

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 샤딩은 거대한 하나의 데이터베이스를 물리적으로 분리된 여러 개의 작은 단위인 **샤드(Shard)로 나누어 서로 다른 서버에 분산 저장하고 관리하는 수평적 확장(Scale-out) 기술**이다.
> 2. **가치**: 단일 서버의 하드웨어 한계(CPU, I/O)를 돌파하여 트래픽과 데이터량 증가에 선형적으로 대응할 수 있게 하며, 장애 범위를 특정 샤드로 국한시키는 격리 효과를 제공한다.
> 3. **융합**: 샤드 키(Shard Key) 기반의 라우팅 기술과 데이터 리밸런싱 기술이 융합되어, 현대 클라우드 네이티브 아키텍처의 대규모 분산 처리를 가능케 한다.

+++

### Ⅰ. 샤딩과 파티셔닝의 차이

- **파티셔닝 (Partitioning)**: 보통 '단일 데이터베이스 서버 내'에서 테이블을 논리적으로 쪼개는 것. (예: 월별 매출 테이블 분리)
- **샤딩 (Sharding)**: 테이블을 쪼개어 '서로 다른 물리적 서버'에 나누어 담는 것. (Shared-Nothing 구조)

+++

### Ⅱ. 샤딩 아키텍처 시각화 (ASCII Flow)

```text
[ Database Sharding Architecture ]

      (Application) ──▶ [ Sharding Proxy / Router ]
                               │ (Check Shard Key)
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
    [ Shard 1 (Node) ]   [ Shard 2 (Node) ]   [ Shard 3 (Node) ]
    (IDs: 1 ~ 1000)      (IDs: 1001 ~ 2000)   (IDs: 2001 ~ 3000) ✅
    ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
    │  Data Part A │     │  Data Part B │     │  Data Part C │
    └──────────────┘     └──────────────┘     └──────────────┘
```

+++

### Ⅲ. 샤딩 설계 시 핵심 과제

1. **샤드 키 선정**: 데이터가 특정 서버에 몰리는 **핫스팟(Hotspot)** 현상을 방지하기 위해 카디널리티가 높고 균등한 분산이 가능한 키를 골라야 합니다.
2. **조인 연산의 제한**: 서로 다른 샤드에 있는 테이블끼리의 조인은 네트워크 비용이 매우 커서 가급적 피해야 합니다 (Global Table 활용 등).
3. **리밸런싱 (Rebalancing)**: 데이터가 늘어나 샤드를 추가할 때, 기존 데이터를 다시 나누는 작업이 매우 복잡하고 시스템 부하를 유발합니다.

- **📢 섹션 요약 비유**: 샤딩은 **'거대 도서관의 서가 분리'**와 같습니다. 한 건물(서버)에 책이 너무 많아 감당이 안 되면, 역사책은 A건물, 과학책은 B건물로 건물을 따로 짓고 책을 옮기는 것입니다. 찾아오는 사람(쿼리)은 자기가 찾는 책의 종류(샤드 키)만 알면 어느 건물로 가야 할지 바로 알 수 있어 혼잡함이 사라집니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Scale-out]**: 서버를 추가하여 성능을 높이는 방식.
- **[Shared Nothing]**: 노드 간 자원 공유 없이 독립적으로 작동하는 원칙.
- **[Consistent Hashing]**: 노드 추가/제거 시 데이터 이동을 최소화하는 알고리즘.

📢 **마무리 요약**: **Sharding**은 빅데이터 시대의 생존 전략입니다. 관리의 복잡성을 대가로 무한한 확장성을 얻는, 현대 아키텍처의 가장 강력한 근육입니다.
