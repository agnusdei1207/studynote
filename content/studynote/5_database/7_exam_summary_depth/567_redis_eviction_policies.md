# 567. Redis 데이터 삭제 정책(Eviction) - 한정된 메모리의 관리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Redis의 데이터 삭제 정책(Eviction Policy)은 메모리가 가득 찼을 때(maxmemory 도달 시), **새로운 데이터를 수용하기 위해 기존 키 중 어떤 것을 삭제할지 결정하는 알고리즘**이다.
> 2. **가치**: 캐시의 효율(Hit Ratio)을 높이기 위해 최근 사용 빈도(LFU)나 시점(LRU)을 고려하여 **가치가 낮은 데이터를 우선 제거**하며, 시스템 중단 없는 안정적인 캐시 운영을 보장한다.
> 3. **융합**: 비즈니스 워크로드의 특성(시간 중심 vs 빈도 중심)에 맞춰 최적의 정책을 선택하는 것이 메모리 기반 DB 아키텍처의 핵심 튜닝 요소다.

+++

### Ⅰ. Redis의 주요 삭제(Eviction) 알고리즘

| 알고리즘 | 설명 | 특징 |
|:---|:---|:---|
| **Noeviction** | 어떤 키도 지우지 않음 | 메모리 꽉 차면 쓰기 오류 발생 |
| **Allkeys-LRU** | 모든 키 중 **가장 오랫동안 쓰이지 않은 것** 삭제 | 범용적으로 가장 많이 사용됨 |
| **Volatile-LRU** | TTL(만료시간)이 설정된 키 중에서만 LRU 적용 | 중요 데이터 보호에 유리 |
| **Allkeys-LFU** | 모든 키 중 **사용 빈도가 가장 낮은 것** 삭제 | 특정 기간 반짝 인기 있는 데이터 방어 |
| **Random** | 무작위로 삭제 | 성능 부하는 낮으나 효율성 떨어짐 |

+++

### Ⅱ. LRU vs LFU 매커니즘 시각화 (ASCII Model)

```text
[ Redis Memory Eviction Strategy ]

  1. LRU (Least Recently Used) - "시점 중심"
     [ A (10:00) ]  [ B (10:05) ]  [ C (10:10) ]
     ──▶ Delete [ A ] ✅ (가장 오래전에 접근됨)

  2. LFU (Least Frequently Used) - "빈도 중심"
     [ A (Hits: 100) ] [ B (Hits: 5) ] [ C (Hits: 50) ]
     ──▶ Delete [ B ] ✅ (가장 적게 사용됨)

  [ Decision Center: Maxmemory Policy ]
  * If Memory Full ──▶ Apply Policy ──▶ ✅ Free Space for New Data
```

+++

### Ⅲ. 실무 운영 가이드

- **캐시 용도라면**: `allkeys-lru` 설정을 추천합니다. 자주 찾는 데이터가 메모리에 오래 머물게 하는 가장 합리적인 방법입니다.
- **특정 주기성 데이터**: 매일 특정 시간에만 조회수가 폭발하는 데이터가 많다면 `allkeys-lfu`가 더 효과적일 수 있습니다.
- **안정성 확보**: `maxmemory`를 실제 물리 메모리의 70~80% 정도로 설정하여, OS의 스왑(Swap) 발생을 막아야 Redis의 초고속 성능이 유지됩니다.

- **📢 섹션 요약 비유**: Redis의 삭제 정책은 **'책상 정리법'**과 같습니다. 책상(메모리)이 좁아서 새 책을 놓을 자리가 없을 때, `LRU`는 "가장 오랫동안 안 펼쳐본 책"을 치우는 것이고, `LFU`는 "거의 읽지도 않고 장식용으로 꽂아둔 책"부터 치우는 것입니다. 나에게 지금 진짜 필요한 책(데이터)이 무엇인지에 따라 정리 규칙을 잘 정해야 공부(비즈니스)를 빨리 끝낼 수 있습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Cache Hit Ratio]**: 삭제 정책이 직접적으로 영향을 미치는 성능 지표.
- **[TTL (Time To Live)]**: 정책과 별개로 시간이 지나면 지워지게 하는 자동 소멸 시간.
- **[Lazy Freeing]**: 삭제 작업을 백그라운드에서 천천히 수행하여 메인 스레드 부하를 줄이는 기술.

📢 **마무리 요약**: **Redis Eviction Policy**는 메모리 경제학의 지혜입니다. 한정된 자원을 가장 가치 있는 데이터로 채우는 이 규칙이 고성능 서비스의 심장을 뛰게 합니다.
