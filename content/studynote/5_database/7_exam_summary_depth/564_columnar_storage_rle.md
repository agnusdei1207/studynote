# 564. 컬럼 기반 스토리지와 RLE - 대규모 분석의 고속 압축

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컬럼 기반 스토리지(Columnar Storage)는 데이터를 행 단위가 아닌 **열(Column) 단위로 모아 물리적으로 저장하는 방식**이며, RLE(Run-Length Encoding)는 동일한 값이 연속될 때 그 값과 반복 횟수만 기록하여 용량을 줄이는 대표적 압축 기술이다.
> 2. **가치**: 분석 쿼리 시 필요한 열만 선택적으로 읽어 **디스크 I/O를 획기적으로 줄이며**, 유사한 데이터가 인접한 특성을 이용한 고효율 압축을 통해 스토리지 비용을 절감한다.
> 3. **융합**: 데이터 웨어하우스(DW) 및 빅데이터 처리 엔진(Parquet, ORC 포맷 등)과 융합되어, 수천억 건의 데이터를 대상으로 하는 실시간 집계 분석의 성능 기반이 된다.

+++

### Ⅰ. 컬럼 기반 스토리지의 성능 원리

- **데이터 국부성 (Data Locality)**: 같은 컬럼의 데이터들이 디스크 상에 나란히 모여 있습니다.
- **I/O 효율성**: `SELECT SUM(Price)...` 쿼리 실행 시, 이름이나 주소 등 불필요한 컬럼은 아예 건드리지 않고 'Price' 컬럼 블록만 순차적으로 읽어 들입니다. ✅
- **압축률 극대화**: 같은 컬럼에는 비슷한 데이터 타입과 중복된 값이 많아, 행 기반(Row-based)보다 훨씬 강력한 압축이 가능합니다.

+++

### Ⅱ. RLE(Run-Length Encoding) 시각화 (ASCII Model)

연속된 데이터의 중복을 '값 + 개수'로 치환하는 마법입니다.

```text
[ Raw Column Data ]
  "Seoul", "Seoul", "Seoul", "Seoul", "Busan", "Busan", "Daegu"
  (Total 7 items, Uncompressed)

[ RLE Compressed Data ] ✅
  ("Seoul", 4), ("Busan", 2), ("Daegu", 1)
  (데이터 종류와 반복 횟수만 기록 ──▶ 획기적 용량 감소!)

[ Physical Layout Comparison ]
  Row-based: [ID1, Name1, City1][ID2, Name2, City2]...
  Col-based: [City1, City2, City3...][ID1, ID2, ID3...]
```

+++

### Ⅲ. 실무적 의의: 분석 전용 포맷

- **분석 쿼리 특화**: 대량의 행을 대상으로 소수의 컬럼만 집계하는 전형적인 BI 워크로드에서 행 기반 DB보다 수백 배 빠른 속도를 냅니다.
- **포맷의 표준**: Apache Parquet이나 ORC와 같은 오픈소스 파일 포맷은 내부적으로 컬럼 기반 저장과 RLE를 포함한 다양한 압축 알고리즘을 사용합니다.
- **주의**: 한 행 전체를 빈번하게 수정하거나 추가해야 하는 OLTP성 업무에는 오히려 성능이 떨어집니다 (모든 컬럼 파일을 각각 열어야 하기 때문).

- **📢 섹션 요약 비유**: 컬럼 기반 스토리지는 **'사전에서 특정 알파벳만 골라내기'**와 같습니다. 행 기반이 '문장별로 쓰인 책'이라면, 컬럼 기반은 '첫 번째 글자들만 모은 책', '두 번째 글자들만 모은 책'으로 나누어 둔 것입니다. 만약 모든 단어의 첫 글자가 'A'라면(RLE 적용), "첫 글자 책에는 A가 100번 나옵니다"라고 한 줄만 적으면 끝나는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Vectorized Execution]**: 압축된 컬럼 데이터를 CPU 레벨에서 한꺼번에 처리하는 기술.
- **[Parquet / ORC]**: 컬럼 기반 저장을 구현한 대표적인 빅데이터 파일 포맷.
- **[Aggregation Speed]**: 컬럼 기반 스토리지가 추구하는 최고의 가치.

📢 **마무리 요약**: **Columnar Storage & RLE**는 현대 데이터 분석의 엔진입니다. 데이터의 물리적 배치를 바꾸고 중복을 제거하는 단순한 논리가 빅데이터 세상을 움직이는 거대한 파워가 됩니다.
