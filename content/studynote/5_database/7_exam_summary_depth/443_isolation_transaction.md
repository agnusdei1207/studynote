# 443. 고립성(Isolation) - 독립적인 실행의 보호막

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 고립성은 동시에 실행되는 여러 트랜잭션이 **서로의 중간 상태(Uncommitted Data)를 볼 수 없게 격리**하여, 각각의 트랜잭션이 마치 데이터베이스를 혼자 독점하여 사용하는 것처럼 보장하는 속성이다.
> 2. **가치**: 다중 사용자 환경에서 발생할 수 있는 갱신 손실, 오손 읽기 등의 동시성 이상 현상을 방지하여 **트랜잭션의 독립적 무결성**을 확보한다.
> 3. **융합**: 락킹(Locking), 타임스탬프, MVCC 등 다양한 **병행 제어(Concurrency Control)** 기술이 융합되어 구현되며, 성능을 위해 고립의 정도를 조절하는 격리 수준(Isolation Level)과 연결된다.

+++

### Ⅰ. 고립성의 핵심 목적: 이상 현상 방지

고립성이 확보되지 않으면 다음과 같은 문제가 발생합니다.
- **Dirty Read**: 다른 트랜잭션이 작업 중인(미확정) 데이터를 읽는 실수.
- **Non-repeatable Read**: 내가 읽은 데이터를 남이 고쳐서 다시 읽을 때 값이 변하는 현상.
- **Phantom Read**: 없던 데이터가 갑자기 나타나는 유령 현상.

+++

### Ⅱ. 고립성 매커니즘 시각화 (ASCII Model)

트랜잭션 사이에는 보이지 않는 논리적 벽이 존재해야 합니다.

```text
[ Isolation Barrier in Database ]

  Transaction A ──▶ [ SQL Work ] ──▶ (Intermediate State)
                                            ┃
          (Barrier: No Peeking!) ◀━━━━━━━━━━╋━━━━━━━━━━▶ (Barrier: No Peeking!)
                                            ┃
  Transaction B ──▶ [ SQL Work ] ──▶ (Intermediate State)

  * A가 COMMIT 하기 전까지 B는 A의 변화를 알 수 없음. ✅
```

+++

### Ⅲ. 성능과 고립성의 트레이드오프

- **완벽한 고립 (Serializable)**: 데이터는 가장 안전하지만, 모든 사용자가 줄을 서야 하므로 매우 느립니다.
- **느슨한 고립 (Read Committed 등)**: 데이터 정합성 리스크가 약간 있지만, 여러 사용자가 동시에 빠르게 작업할 수 있습니다.
- **현대적 해결 (MVCC)**: 데이터를 여러 버전으로 관리하여, 읽기 작업이 쓰기 작업을 방해하지 않게 함으로써 성능과 고립성을 동시에 잡습니다.

- **📢 섹션 요약 비유**: 고립성은 **'시험장의 칸막이'**와 같습니다. 수백 명의 학생(트랜잭션)이 동시에 시험을 보지만, 옆 사람이 답안지에 무엇을 적고 있는지(중간 상태) 볼 수 없게 칸막이(Isolation)를 높게 세우는 것입니다. 내가 내 시험지에만 집중할 수 있을 때, 비로소 정확한 성적(정합성)이 나올 수 있는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Isolation Level]**: 고립의 강도를 결정하는 설정 (Level 0~3).
- **[Concurrency Control]**: 고립성을 지키기 위한 기술적 투쟁.
- **[MVCC]**: 락 없이 고립성을 구현하는 현대적 마법.

📢 **마무리 요약**: **Isolation**은 데이터베이스의 평화를 유지하는 경계선입니다. 서로 간섭하지 않는 독립된 실행 환경을 제공함으로써, 복잡한 동시성 속에서도 데이터의 진실을 수호합니다.
