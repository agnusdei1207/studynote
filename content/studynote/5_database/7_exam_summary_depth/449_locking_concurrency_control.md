# 449. 동시성 제어 잠금(Locking) - 읽기와 쓰기의 교통정리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 잠금(Locking)은 특정 데이터 항목에 대해 트랜잭션이 접근 권한을 선점하여, **동시에 다른 트랜잭션이 데이터를 훼손하지 못하도록 상호 배제(Mutual Exclusion)를 구현하는 물리적 장치**이다.
> 2. **가치**: 공유 락(S)과 배타 락(X)의 조합을 통해 "읽기끼리는 허용하되, 쓰기는 독점한다"는 규칙을 강제하여, 데이터 정합성과 동시성 사이의 균형을 유지한다.
> 3. **융합**: 2단계 락킹 프로토콜(2PL)의 기초가 되며, 락의 단위(Granularity) 설정에 따라 전체 시스템의 처리량(Throughput)이 결정되는 핵심 성능 변수로 작용한다.

+++

### Ⅰ. S-락(공유)과 X-락(배타)의 상세 규칙

| 락 유형 | 용도 | 호환성 (Current Lock vs New Req) |
|:---|:---|:---|
| **S-락 (Shared Lock)** | 데이터 **읽기** 시 사용 | **S-락끼리는 허용** (동시 읽기 가능) |
| **X-락 (Exclusive Lock)** | 데이터 **쓰기** 시 사용 | **누구와도 호환 안 됨** (독점) |

+++

### Ⅱ. 락 호환성 매트릭스 시각화 (ASCII Model)

```text
[ Lock Compatibility Matrix ]

               Transaction B 요청
               ┌──────┬──────┐
               │  S   │  X   │
        ┌──────┼──────┼──────┤
  Trans │  S   │  O   │  X   │  <-- S-락끼리는 사이좋게 공유! ✅
  action ├──────┼──────┼──────┤
    A   │  X   │  X   │  X   │  <-- X-락은 누구도 끼어들 수 없음! 🔒
  점유중 └──────┴──────┴──────┘
```

+++

### Ⅲ. 락킹의 한계: 데드락과 오버헤드

- **교착 상태 (Deadlock)**: 두 트랜잭션이 서로가 쥔 락을 기다리며 영원히 멈추는 현상.
- **블로킹 (Blocking)**: X-락을 쥔 트랜잭션이 길어지면 다른 모든 트랜잭션이 줄줄이 대기하여 성능이 급락합니다.
- **락 에스컬레이션**: 너무 많은 레코드 락이 발생하면 시스템이 메모리 절약을 위해 페이지나 테이블 전체 락으로 수준을 높여버리는 현상입니다.

- **📢 섹션 요약 비유**: 락킹은 **'도서관의 책 이용 규칙'**과 같습니다. S-락은 '열람실에서 책 읽기'와 같아서 여러 명이 동시에 읽어도 문제가 없지만, X-락은 '책을 집으로 빌려 가기'와 같아서 내가 빌려 간 동안에는 아무도 그 책을 볼 수 없습니다. 내가 반납(Unlock)할 때까지 다른 사람들은 하염없이 기다려야 하는 엄격한 규칙입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Lock Manager]**: DBMS 내부에서 락의 상태를 관리하는 모듈.
- **[Lock Granularity]**: 행, 페이지, 테이블 등 락을 거는 데이터의 크기 단위.
- **[2PL]**: 락을 걸고 푸는 시점을 정한 프로토콜.

📢 **마무리 요약**: **Locking**은 데이터베이스의 법률입니다. 읽기의 자유와 쓰기의 책임을 조율하는 이 매커니즘이 다중 사용자 환경의 평화를 지탱합니다.
