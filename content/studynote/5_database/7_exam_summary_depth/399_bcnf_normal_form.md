# 399. BCNF(Boyce-Codd Normal Form) - 결정자의 권위

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: BCNF는 제3정규형을 보완한 강력한 정규형으로, **릴레이션 내의 모든 결정자(Determinant)가 반드시 후보키(Candidate Key)여야 한다는 원칙**을 가진다.
> 2. **가치**: 3NF를 만족하더라도 여러 개의 복합 후보키가 겹쳐 있을 때 발생할 수 있는 이상 현상을 원천 차단하여, 데이터 정합성을 한 단계 더 높은 수준으로 끌어올린다.
> 3. **융합**: "결정자이면 키여야 한다"는 단순하면서도 엄격한 규칙을 통해, 논리적 설계의 빈틈을 메우는 정규화의 최종 관문(일반적 실무 기준) 역할을 한다.

+++

### Ⅰ. BCNF의 필요성 (3NF의 한계)

- **문제 상황**: 복합키가 존재하고, 기본키의 일부가 다른 일반 속성에 의해 결정되는 경우 발생합니다.
- **사례**: (학생, 과목)이 기본키인데, 특정 교수님은 한 과목만 가르치고(교수 ──▶ 과목), 한 과목은 여러 교수가 가르칠 수 있을 때.
- **이상 현상**: 교수님이 담당 과목을 바꾸면 해당 수업을 듣는 모든 학생 레코드를 고쳐야 하는 등 3NF만으로는 해결 안 되는 중복이 남습니다.

+++

### Ⅱ. BCNF 정규화 과정 시각화 (ASCII Model)

키가 아닌 속성이 결정자 노릇을 하는 구조를 파괴하여 분리합니다.

```text
[ 3NF (but not BCNF) ]
  ┌────────┬──────────┬──────────┐
  │ 학생(K) │ 과목(K)   │ 교수      │
  └────┬───┴────▲─────┴────┬─────┘
       │        └──────────┘ [ 교수가 과목을 결정함 ] 💥
       └───────────────────▶ [ 후보키가 아닌 결정자 존재 ]

[ BCNF (Normalized) ]
  ┌────────┬──────────┐      ┌──────────┬──────────┐
  │ 학생(K) │ 교수(K)   │      │ 교수(K)   │ 과목      │
  └────────┴──────────┘      └──────────┴──────────┘
  (이제 모든 결정자는 각 테이블의 키가 됨 ✅)
```

+++

### Ⅲ. BCNF의 성립 조건 및 특징

1. **제3정규형(3NF)**을 만족해야 합니다.
2. 모든 함수 종속 관계 $X \to Y$ 에 대하여, **$X$ 는 반드시 후보키(Candidate Key)**여야 합니다.
3. **Strong 3NF**: 3NF보다 엄격하며, 종속성 보존(Dependency Preservation)이 안 되는 경우가 발생할 수 있어 설계 시 주의가 필요합니다.

- **📢 섹션 요약 비유**: BCNF는 **'반장 선거의 자격'**과 같습니다. 반 아이들의 번호(결정자)를 매길 수 있는 권한은 오직 정식으로 선출된 '반장(후보키)'에게만 있어야 합니다. 반장도 아닌 평범한 학생이 번호를 매기려 들면(후보키가 아닌 결정자), 교실 질서(데이터 정합성)가 어지러워지므로 이를 엄격히 금지하고 따로 격리하는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Determinant]**: 다른 속성을 결정짓는 속성.
- **[Candidate Key]**: 튜플을 유일하게 식별할 수 있는 최소한의 속성 집합.
- **[Lossless Decomposition]**: BCNF 분해 시 데이터 유실이 없어야 함.

📢 **마무리 요약**: **BCNF**는 "결정권자는 반드시 자격을 갖춰야 한다"는 데이터의 질서입니다. 키와 결정자를 일치시킴으로써 데이터베이스의 논리적 완결성을 완성합니다.
