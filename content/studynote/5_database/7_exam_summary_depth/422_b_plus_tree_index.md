# 422. B+Tree 인덱스 - 리프 노드 순차 연결의 미학

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: B+Tree는 B-Tree를 개량하여 **모든 실제 데이터(또는 포인터)를 최하위 리프 노드(Leaf Node)에만 저장**하고, 이들 리프 노드들을 형제 포인터로 연결하여 **순차적 탐색을 극대화한 자료구조**다.
> 2. **가치**: 리프 노드가 양방향 연결 리스트(Doubly Linked List)로 이어져 있어, 특정 범위의 데이터를 찾는 **인덱스 레인지 스캔(Index Range Scan)** 시 루트부터 재탐색할 필요 없이 옆으로 이동하며 데이터를 광속으로 수집한다.
> 3. **융합**: 디스크 I/O를 최소화하는 블록 단위 저장 기술과 정렬된 데이터 유지 기술이 융합되어, 현대 거의 모든 RDBMS의 기본 인덱스 알고리즘으로 채택되었다.

+++

### Ⅰ. B+Tree의 구조적 특징

- **Root & Branch Node**: 실제 데이터는 없고, 하위 노드로 가기 위한 가이드(인덱스 키) 정보만 보관합니다.
- **Leaf Node**: 실제 데이터(또는 레코드의 물리적 주소인 ROWID)를 저장하며, 정렬된 상태를 유지합니다.
- **Sequential Link**: 리프 노드끼리 포인터로 연결되어 있어, 범위를 스캔할 때 매우 유리합니다.

+++

### Ⅱ. B+Tree 리프 노드 연결 시각화 (ASCII Model)

```text
[ B+Tree Structure ]

          [ Root Node ]
         /      |      \
  [Branch 1] [Branch 2] [Branch 3]
     /   \      /   \      /   \
  ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
  │ L1│─▶ L2│─▶ L3│─▶ L4│─▶ L5│─▶ L6│  <-- Sequential Linked Leaves ✅
  └───┘ └───┘ └───┘ └───┘ └───┘ └───┘
    ▲                               ▲
    └────────── Range Scan ─────────┘
```

+++

### Ⅲ. 왜 B+Tree를 쓰는가? (B-Tree와의 차이)

1. **레인지 스캔 최적화**: B-Tree는 범위를 찾을 때 노드를 위아래로 계속 오가야 하지만, B+Tree는 시작점만 찾으면 옆으로 쭉 훑으면 끝납니다.
2. **풀 스캔 효율**: 모든 데이터가 리프 노드에 모여 있어, 인덱스 전체를 읽을 때 디스크 I/O가 선형적으로 발생하여 빠릅니다.
3. **높은 팬아웃(Fan-out)**: 중간 노드에 데이터를 두지 않으므로 한 블록에 더 많은 포인터를 담을 수 있고, 이는 트리의 높이를 낮춰 탐색 속도를 높입니다.

- **📢 섹션 요약 비유**: B+Tree는 **'도서관의 인덱스 카드함'**과 같습니다. 카드함의 서랍(Branch)을 열어 원하는 카테고리를 찾으면, 그 안에는 모든 책의 위치가 적힌 카드들(Leaf)이 번호순으로 나란히 꽂혀 있습니다. 내가 100번부터 200번까지의 책을 찾는다면, 100번 카드를 찾은 뒤 그 뒤에 있는 카드들을 순서대로(Sequential Link) 넘겨보기만 하면 되는 아주 편리한 시스템입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Index Range Scan]**: B+Tree의 장점을 가장 잘 활용하는 검색 방식.
- **[Leaf Split]**: 데이터 삽입 시 리프 노드가 꽉 차서 쪼개지는 현상.
- **[Clustering Factor]**: 인덱스 정렬 순서와 실제 데이터 정렬 순서의 일치도.

📢 **마무리 요약**: **B+Tree**는 단순한 트리를 넘어 '범위의 탐색'을 완성한 구조입니다. 리프 노드들의 순차적 연결이라는 작은 디테일이 대규모 데이터 조회 성능의 거대한 차이를 만들어냅니다.
