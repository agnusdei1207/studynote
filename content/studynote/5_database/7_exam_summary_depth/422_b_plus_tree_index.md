# 422. B+Tree 인덱스 - 리프 노드 순차 연결의 미학

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: B+Tree (B-Plus Tree)는 **B-Tree (Balanced Tree)**의 변형으로, 모든 실제 데이터(또는 데이터 포인터)를 트리의 가장 하위인 **리프 노드(Leaf Node)**에 배치하고, 상위 내부 노드(Internal Node)는 인덱스 키만 저장하여 **트리의 높이를 낮추고 팬아웃(Fan-out)을 극대화**한 자료구조다.
> 2. **가치**: 리프 노드들이 **순차 연결 리스트(Sequential Linked List)** 형태로 서로 연결되어 있어, 단일 키 검색은 물론 **범위 검색(Range Query)** 수행 시 획기적인 성능 향상을 제공하며, 디스크 블록 접근 횟수를 최소화한다.
> 3. **융합**: 운영체제의 **페이지(Page)** 단위 I/O 최적화와 데이터베이스의 **정렬(Ordering)** 요구사항이 결합되어, 대용량 트랜잭션 처리(OLTP) 환경에서 RDBMS의 표준 인덱싱 메커니즘으로 자리 잡았다.

---

### Ⅰ. 개요 (Context & Background)

**B+Tree (B-Plus Tree)**는 데이터베이스 시스템에서 가장 널리 사용되는 인덱스 자료구조로, 파일 시스템과 데이터베이스 인덱싱을 위해 설계되었다. 기존의 B-Tree가 내부 노드와 리프 노드 모두에 데이터를 저장하여 범위 검색 시 비효율적인 탐색 경로를 생성하는 문제를 해결하기 위해 고안되었다.

B+Tree의 핵심 철학은 **"데이터의 저장과 탐색 경로의 분리"**에 있다. 상위 노드들은 오직 **"어디로 가야 원하는 데이터를 찾을 수 있는가(Routing)"**에만 집중하며, 실제 데이터는 트리의 바닥에 있는 리프 노드에 모여 있다. 이러한 구조는 디스크와 같은 느린 보조기억장치에서 **블록(Block) 또는 페이지(Page)** 단위로 데이터를 읽어야 하는 환경에서 절대적인 유리함을 제공한다. 리프 노드의 크기를 디스크 블록 크기에 맞춤으로써, 한 번의 I/O로 최대한 많은 키(Key)를 가져올 수 있게 하는 것이 설계의 핵심이다.

**💡 비유**
B+Tree는 **"거대한 박물관의 전시 가이드북"**과 같다. 1층 안내 데스크(Root Node)에서는 "고대 미술품은 3층입니다"라고만 알려주고, 3층 입구(Branch Node)에서는 "왼쪽 방은 그리스, 오른쪽 방은 로마입니다"라고 안내한다. 실제 유물(Actual Data)을 보려면 결국 가장 안쪽의 전시실(Leaf Node)로 가야 하며, 이 전시실들은 서로 연결된 통로(Linked List)로 이어져 있어 관람객이 한 방에서 다음 방으로 이동하기가 매우 편리하다.

**등장 배경**
1. ① 기존 B-Tree 등장: 데이터 삽입/삭제/검색의 균형을 맞추려는 노력.
2. ② 디스크 I/O 병목: 데이터베이스의 크기가 메모리를 초과하면서 디스크 접근이 성능의 병목이 됨.
3. ③ 순차 접근의 필요성: 실무 비즈니스는 단건 조회보다 "금일 주문 내역 전체"와 같은 범위 조회가 빈번함.

**📢 섹션 요약 비유**: B+Tree는 마치 고층 빌딩에서 **'내비게이션 앱(탐색 경로)'과 '실제 상점(데이터)'을 분리**해 놓은 것과 같습니다. 엘리베이터(Internal Node)는 층만 안내해 줄 뿐이고, 실제 쇼핑은 옥상 정원(Leaf Node)에서 이루어지며, 옥상들은 서로 이어진 통로로 연결되어 있어 이동이 편리합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 구성 요소 상세 분석

B+Tree는 크게 루트 노드, 내부 노드(Branch), 리프 노드로 구성되며, 각 노드는 디스크 섹터(Sector)나 페이지(Page) 크기에 맞춰 설계된다.

| 요소명 | 역할 | 내부 동작 및 특징 | 프로토콜/키 관리 | 비유 |
|:---|:---|:---|:---|:---|
| **루트 노드 (Root Node)** | 탐색의 시작점 | 트리 진입 유일한 경로. 메모리에 상주 캐싱되는 경우가 많음. | 최상위 키 범위 보유 | 빌딩 1층 로비 |
| **내부 노드 (Internal Node)** | 라우팅(Routing) | 실제 데이터 없이 `Key + Child Pointer`만 존재. 팬아웃을 높임. | Key를 기준으로 하위 브랜치 분기 | 각 층의 중앙 현관 |
| **리프 노드 (Leaf Node)** | 데이터 저장소 | **실제 데이터(또는 ROWID)** 저장. **순차 연결(LinkedList)됨.** | 정렬된 키 저장 및 `Next/Prev Pointer` 보유 | 실제 전시실들 |
| **Order (차수)** | 노드 용량 | 자식 노드를 가질 수 있는 최대 개수. 노드의 크기(Page Size)에 따라 결정됨. | Block Size / Key Size | 방에 들어갈 수 있는 문의 개수 |

#### 2. B+Tree 구조 및 데이터 흐름 다이어그램

아래는 B+Tree의 논리적 구조를 도식화한 것이다. 리프 노드들끼리의 연결(`<===>`)이 핵심임에 유의하라.

```text
   [ B+Tree Logical Structure (High Fan-out) ]

       [Root Node (Level 1)]
       Key: 50
      /          \
  [Branch L]    [Branch R]
 Keys: 10,30    Keys: 70,90
 /   |   \      /   |   \
(1)  (2)  (3)  (4)  (5)  (6)
▼    ▼    ▼    ▼    ▼    ▼
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│Leaf 1│◀─│Leaf 2│◀─│Leaf 3│  │Leaf 4│◀─│Leaf 5│◀─│Leaf 6│
│[~10] │ │[11~30]│ │[31~50]│ │[51~70]│ │[71~90]│ │[91~] │
└──────┘ └──────┘ └──────┘ └──────┘ └──────┘ └──────┘
  ▲                                                        ▲
  │                   Doubly Linked List                   │
  └────────────────────────────────────────────────────────┘
          Sequential Scan Direction (Range Query Optimized)

(1) Internal Node Routing: 50보다 작으니 왼쪽, 크니 오른쪽으로 분기.
(2) Leaf Node Access: 실제 데이터(RowID) 획득.
(3) Sequential Link: Leaf 2 -> Leaf 3 이동 시 부모 노드 재방문 불필요.
```

**다이어그램 해설**:
1.  **Root & Branch**: 상위 노드들은 데이터의 실제 위치를 모르지만, **"이 키 범위는 이쪽으로 가라"**는 판단만 수행한다. 데이터를 저장하지 않기 때문에 하나의 노드(디스크 블록)에 수백~수천 개의 포인터를 담을 수 있어 트리의 높이(Height)를 3~4 수준으로 낮출 수 있다.
2.  **Leaf Node**: 모든 데이터는 여기에 있다. 리프 노드는 항상 정렬된 상태를 유지한다.
3.  **Sequential Link**: 다이어그램 하단의 화살표(`<─>`)처럼, 리프 노드들은 양방향 연결 리스트(Doubly Linked List)로 연결되어 있다. 이는 `WHERE id BETWEEN 100 AND 200` 쿼리가 들어왔을 때, 100을 찾은 뒤에는 부모 노드로 올라가지 않고 옆으로만 이동하면 되므로 매우 강력한 최적화가 된다.

#### 3. 심층 동작 원리 및 복잡도
- **탐색 (Search)**: 루트에서 시작하여 리프까지 이동(Height 횟수만큼 I/O 발생). 리프에서 데이터를 찾으면 종료. 시간 복잡도는 $O(\log N)$. 여기서 N은 전체 키의 개수가 아니라 전체 노드 개수에 근접한 효율을 가짐.
- **삽입 (Insert)**: 리프 노드에 데이터를 삽입. 리프가 꽉 차면 **리프 분할(Leaf Split)** 발생. 분할 시 중간 키가 부모 노드로 전파(Propagation)됨. 이 과정에서 루트까지 분할이 전파되면 트리 높이가 1 증가.
- **삭제 (Delete)**: 리프에서 데이터 제거. 노드의 사용률이 50% 이하로 떨어지면 형제 노드와 **병합(Merge)**하거나 **재분배(Redistribution)** 수행.

#### 4. 핵심 메커니즘: Leaf Split (수식 및 예시)

리프 노드의 용량이 $L$이고, 현재 저장된 키 수가 $L$개일 때 새로운 키가 삽입되면 다음 과정이 발생한다.

1.  **Overflow Detection**: $Count == L$ 확인.
2.  **Allocate New Node**: 디스크에 새 블록 할당.
3.  **Split Keys**: 기존 $L$개 + 새 키 1개를 총 $(L+1)$개로 보고 중앙값($\lceil (L+1)/2 \rceil$)을 기준으로 분할.
4.  **Update Parent**: 분할된 기준값을 상위 내부 노드에 삽입.
5.  **Link Update**: 기존 리프의 다음 포인터(Next Pointer)가 새 리프를 가리키도록 수정, 새 리프의 Next Pointer는 기존 Next가 가리키던 곳을 설정.

```sql
-- pseudo-code for B+Tree Insert
IF leaf.is_full() THEN
    new_leaf = allocate_node()
    split_keys(leaf, new_leaf, new_key)
    median = get_median_key(leaf, new_leaf)
    insert_into_parent(parent, median, new_leaf) // Recursive propagation
    link_leaf(leaf, new_leaf) // Maintain Linked List
ELSE
    insert_sorted(leaf, new_key)
END IF
```

**📢 섹션 요약 비유**: B+Tree의 아키텍처는 마치 **"초대형 공장의 컨베이어 벨트 시스템"**과 같습니다. 중간 관리자(Internal Node)는 작업자가 어디에 배치되어야 할지만 지시할 뿐 직접 물건을 만지지 않으며, 실제 작업자(Leaf Node)들은 일렬로 늘어선 컨베이어 벨트(Linked List) 위에서 연속적으로 작업을 처리합니다. 덕분에 한 구역에서 다음 구역으로 이동할 때 복잡한 승인 절차(부모 노드 방문) 없이 즉시 이동이 가능합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 기술 비교 분석표

B+Tree의 장단점을 타 자료구조와 비교하여 분석한다.

| 구분 | B+Tree (B-Plus Tree) | B-Tree (Balanced Tree) | Hash Index |
|:---|:---|:---|:---|
| **저장 위치** | 데이터 리프(Leaf) 전용 | 데이터 모든 노드에 분산 | 해시 테이블(Bucket) |
| **순차 접근** | **매우 우수** (리프 연결로 $O(1)$ 스캔) | **불량** (트리 순회 필요) | **불가능** (Full Scan 필요) |
| **범위 검색** | **지원** (시작점 찾고 리프 탐색) | **미흡** (노드마다 데이터 확인 필요) | **미지원** |
| **높이 (Height)** | **낮음** (내부 노드가 키만 저장) | 상대적으로 높음 | N/A |
| **공간 효율** | 내부 노드가 작아 팬아웃이 높음 | 내부 노드가 커서 팬아웃이 낮음 | Bucket Load Factor에 따름 |

#### 2. 과목 융합 관점

-   **[OS (Operating System)]**: **Page Fault (페이지 부재)** 최소화와의 시너지. B+Tree의 노드 크기는 보통 OS의 페이지 크기(4KB, 8KB, 16KB)에 맞춰 설계된다. 팬아웃이 높다는 것은 하나의 페이지에 더 많은 자식 포인터를 저장할 수 있음을 의미하며, 이는 곧 **"데이터 하나를 찾기 위해 읽어야 하는 디스크 페이지 수를 줄인다(I/O Reduction)"**는 것으로 직결된다.
-   **[Network]**: 대규모 분산 DB(Sharding) 환경에서, 네트워크 호출 비용(RTT)이 크기 때문에 B+Tree와 같이 **"적은 횟수로 정확한 위치를 찾는"** 구조가 필수적이다. 만약 Hash를 사용하여 Random I/O가 유발된다면 네트워크 병목이 심각해질 수 있다.

#### 3. 성능 메트릭 (Metrics)
-   **Fan-out**: $M = \lfloor \frac{Page\_Size}{Key\_Size + Pointer\_Size} \rfloor$. Page Size가 클수록 Fan-out은 제곱급으로 증가하여 트리 높이를 낮춘다.
-   **Clustering Factor**: 인덱스 순서와 실제 테이블 데이터 순서의 일치도. B+Tree 인덱스를 통해 테이블 액세스 시 이 값이 좋으면 Physical I/O가 획기적으로 감소한다.

**📢 섹션 요약 비유**: B+Tree는 **"인기 있는 쇼핑몰의 동선 설계"**와 같습니다. B-Tree는 상품이 여기저기 흩어져 있어서 가끔씩만 찾는 사람에게는 편하지만, 쇼핑물을 구경하는 사람(범위 검색)에게는 비효율적입니다. 반면 B+Tree는 상품(데이터)을 모두 **1층 전시관(리프 노드)**에 모아두고, 전시관끼리 연결 통로를 뚫어놓아 구경하기 쉽게 만든 **지능적 동선 구조**입니다.
