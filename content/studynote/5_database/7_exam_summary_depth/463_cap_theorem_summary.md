# 463. CAP 이론 - 분산 시스템의 삼각 딜레마

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CAP 이론은 분산 데이터 저장소에서 **일관성(Consistency), 가용성(Availability), 분단 허용성(Partition Tolerance)**이라는 세 가지 속성을 동시에 모두 만족시키는 것은 불가능하며, 오직 두 가지만 선택할 수 있다는 이론이다.
> 2. **가치**: 네트워크 장애(P)가 발생할 수밖에 없는 현실에서, 데이터 정합성(C)을 우선할 것인지 서비스 무중단(A)을 우선할 것인지 결정하는 **아키텍처 설계의 기준**이 된다.
> 3. **융합**: NoSQL 시스템의 다양성을 설명하는 근거가 되며, 최근에는 평상시의 성능까지 고려한 PACELC 이론으로 융합 확장되고 있다.

+++

### Ⅰ. CAP의 3가지 요소 정의

1. **일관성 (Consistency)**: 모든 노드가 어떤 시점에서도 동일한 데이터를 보아야 함. (쓰기 성공 즉시 모든 노드에서 최신 값 읽기 가능)
2. **가용성 (Availability)**: 일부 노드 장애 시에도 모든 요청에 대해 성공/실패 응답을 보낼 수 있어야 함. (서비스 무중단)
3. **분단 허용성 (Partition Tolerance)**: 노드 간 네트워크가 끊어지는 '분단' 상황에서도 시스템이 계속 동작해야 함. (분산 시스템의 필수 조건)

+++

### Ⅱ. CAP 트레이드오프 시각화 (ASCII Model)

```text
[ CAP Theorem Triangle ]

           Consistency (C)
               / \
              /   \
          CP /     \ CA (RDBMS - 단일망)
            /       \
           /_________\
Availability(A)  AP  Partition Tolerance(P)

  - CP: 정합성 중시 (네트워크 문제 시 서비스 중단). 예: HBase, MongoDB
  - AP: 가용성 중시 (네트워크 문제 시 낡은 데이터 허용). 예: Cassandra
  - CA: 분산 불가능 (네트워크 장애가 없다고 가정). 예: 단일 MySQL
```

+++

### Ⅲ. 실무적 선택의 딜레마

- **현실적인 제약**: 분산 시스템을 구축한다면 **P(분단 허용성)**는 선택이 아닌 필수입니다. 따라서 실제 고민은 **CP vs AP**의 싸움이 됩니다.
- **금융 업무**: 돈이 틀리면 안 되므로 **CP**를 택하고, 점검 중 메시지를 띄우는 것이 낫습니다.
- **소셜 서비스**: 댓글이나 '좋아요'가 1초 늦게 보여도 서비스가 멈추면 안 되므로 **AP**를 택하는 것이 유리합니다.

- **📢 섹션 요약 비유**: CAP 이론은 **'맛집의 메뉴 선택'**과 같습니다. "싸고(A), 맛있고(C), 양 많은(P)" 세 가지를 모두 만족하는 메뉴는 없습니다. 분산 환경(양 많은)에서 최고의 맛(일관성)을 고수하려면 가격이 비싸거나 기다려야(가용성 포기) 하고, 빨리 먹으려면(가용성) 맛이 조금 떨어지는 것(일관성 포기)을 감수해야 하는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Eventually Consistent]**: AP 시스템이 지향하는 "언젠가는 일치함" 상태.
- **[Network Partition]**: CAP 이론이 작동하게 만드는 주범.
- **[PACELC]**: 장애 시(P)와 평상시(E)를 나누어 설명하는 CAP의 확장판.

📢 **마무리 요약**: **CAP Theorem**은 분산 아키텍처의 물리 법칙입니다. 이 한계를 인정하고 비즈니스 가치에 가장 부합하는 최적의 두 가지를 골라내는 것이 진정한 설계의 묘미입니다.
