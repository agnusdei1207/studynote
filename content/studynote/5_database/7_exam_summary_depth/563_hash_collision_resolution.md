# 563. 해시 충돌과 해결 기법 - 체이닝과 개방 주소법의 승부

## # 해시 충돌과 해결 기법 (Hash Collision & Resolution)

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해시 충돌(Collision)은 서로 다른 키(Key)가 **해시 함수(Hash Function)**에 의해 동일한 슬롯(Slot) 주소를 반환하는 필연적인 현상이며, 이를 해결하기 위한 **체이닝(Chaining)**과 **개방 주소법(Open Addressing)**은 상반된 메모리 관리 전략을 제시합니다.
> 2. **가치**: 적재율(Load Factor)이 증가함에 따라 조회 성능이 $O(1)$에서 $O(N)$으로 열화되는 것을 방지하고, **캐시 라인(Cache Line)** 효율과 메모리 오버헤드 사이의 트레이드오프를 최적화하여 데이터베이스 인덱스의 처리량(Throughput)을 보장합니다.
> 3. **융합**: **DBMS(데이터베이스 관리 시스템)**의 인메모리 인덱싱과 **OS(Operating System)**의 페이지 테이블 관리, 그리고 분산 캐시 시스템의 일관성 유지를 위한 핵심 메커니즘으로 활용됩니다.

---

### Ⅰ. 개요 (Context & Background)

해시 테이블(Hash Table)은 키-값(Key-Value) 쌍을 저장하는 자료구조로, 이상적인 해시 함수는 모든 키를 고유한 주소로 매핑하지만, **피전 현상(Pigeonhole Principle)**에 의해 저장할 항목의 수가 버킷(Bucket)의 수보다 많으면 필연적으로 두 개 이상의 키가 동일한 주소를 가리키는 **충돌(Collision)**이 발생합니다.

이 충돌을 해결하지 못하면 데이터 덮어쓰기가 발생하거나 검색이 불가능해집니다. 이를 해결하기 위해 크게 두 가지 패러다임이 존재합니다.
1.  **체이닝 (Chaining)**: 연결 리스트(Linked List)나 트리(Tree)를 사용하여 해당 버킷에 데이터를 묶어서 관리하는 **'외부 저장(Overflow Area 활용)'** 방식.
2.  **개방 주소법 (Open Addressing)**: 해시 테이블 내의 빈 공간을 찾아 데이터를 삽입하는 **'내부 저장(Closed Hashing)'** 방식.

과거에는 메모리가 비싸 체이닝을 선호했으나, 현대에는 **CPU 캐시(CPU Cache)**의 지역성(Locality)이 중요해짐에 따라 메모리 공간을 더 효율적으로 활용하고 캐시 미스(Cache Miss)를 줄이는 개방 주소법이 많이 연구되고 있습니다.

#### 💡 핵심 비유
해시 충돌 해결은 **'만료된 주차권 찾기'**와 같습니다.
-   **Chaining**: 주차 구역(슬롯)에 차가 이미 있으면, 그 옆에 빈 공간을 만들어 연달아 세우는 방식입니다. 차를 찾을 때 해당 구역에 세워진 차들을 하나하나 확인해야 합니다.
-   **Open Addressing**: 내 자리에 다른 차가 있으면, 바로 옆자리, 그 다음자리로 순회하며 비어 있는 곳에 주차하는 방식입니다. 나중에 차를 찾을 때 내가 어디로 이동했는지 추적해야 합니다.

#### 📢 섹션 요약 비유
해시 충돌 해결 전략은 **'만원 식당의 자리 배정'**과 같습니다. 체이닝은 테이블에 자리가 없으면 의자를 추가로 끌어다 붙여서 앉히는 방식(공간 유연성)이고, 개방 주소법은 예약된 자리에 누가 앉아있으면 옆 테이블의 빈자리를 순서대로 찾아가 앉히는 방식(공간 효율성)입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

해시 충돌 해결 기법의 동작 메커니즘을 구성 요소별로 심층 분석합니다.

#### 1. 구성 요소 및 상세 비교

| 구분 | Chaining (체이닝 / Separate Chaining) | Open Addressing (개방 주소법) |
|:---|:---|:---|
| **기본 철학** | 충돌이 발생해도 해당 버킷에 **연결**(Chain)하여 저장 | 충돌 발생 시 **해시 테이블 내**의 다른 빈 슬롯을 탐색 |
| **주요 자료구조** | 연결 리스트(Linked List), 레드-블랙 트리(Red-Black Tree) | 배열(Array) 기반의 선형/이차/이중 탐사 |
| **삭제 연산** | 매우 간단함 (노드 분리) | 복잡함 (Lazy Deletion 등의 기법 필요) |
| **메모리 오버헤드** | 포인터(Pointer) 저장을 위한 추가 메모리 필요 | 포인터 없이 배열만 사용하므로 메모리 절약 |
| **적재율(Load Factor)** | $\alpha > 1$ 가능 (요소 개수 > 버킷 수) | $\alpha < 1$ 유지 필요 (항상 빈 공간 확보 필수) |
| **캐시 친화적 성능** | 낮음 (포인터 추적으로 인한 참조 지역성 저하) | 높음 (연속된 메모리 영역 탐색으로 캐시 히트 유리) |

#### 2. 아키텍처 다이어그램 (ASCII)

아래는 동일한 키(Key) 충돌이 발생했을 때, 두 방식이 데이터를 저장하는 물리적 구조의 차이를 보여줍니다.

```text
[HASH FUNCTION: h(k) = k mod 5]
Scenario: Key 1, Key 6 both map to Index 1.

+-------------------------+       +-------------------------+
|     1. Chaining Model   |       | 2. Open Addressing Model|
|     (Linked List)       |       |     (Linear Probing)    |
+-------------------------+       +-------------------------+

 Buckets (Array)               Buckets (Array)
 +-----+  +-----+               +-----+  +-----+
 |  0  |  |  1  | --------+     |  0  |  |  1  | (Key 1)
 +-----+  +-----+         |     +-----+  +-----+
          |  HEAD |       |                     |
          +-------+       v                     +--> +-----+
          |  Key 1 | [Next]--->               |  2  | (Key 6)
          +-------+       |                    +-----+
                          |                    (Collision Resolved:
          +-------+       v                     Probing to next slot)
          |  Key 6 | [NULL]
          +-------+

 [메모리 포인터]                [연속 메모리 접근]
 외부 영역에 추가 할당           내부 빈 공간 순차 탐색
```

#### 3. 심층 동작 원리 및 알고리즘

**A. 개방 주소법의 탐사(Probing) 시퀀스**
개방 주소법은 빈 슬롯을 찾는 방법에 따라 성능이 크게 달라집니다.

1.  **Linear Probing (선형 탐사)**:
    -   $h(k, i) = (h(k) + i) \mod S$
    -   $i$를 0, 1, 2... 순서대로 증가시키며 바로 옆 칸을 확인.
    -   **문제점**: 1차 군집화(Primary Clustering). 이미 데이터가 뭉쳐 있는 곳에 또 데이터가 뭉치는 현상 발생.
2.  **Quadratic Probing (이차 탐사)**:
    -   $h(k, i) = (h(k) + c_1 \cdot i + c_2 \cdot i^2) \mod S$
    -   충돌 시 제곱수만큼 떨어진 칸으로 이동.
    -   **문제점**: 군집화는 완화되나, 테이블의 절반 이상이 차면 빈 슬롯을 찾지 못할 수 있음.
3.  **Double Hashing (이중 해싱)**:
    -   $h(k, i) = (h_1(k) + i \cdot h_2(k)) \mod S$
    -   두 개의 독립적인 해시 함수를 사용하여 이동 거리를 결정. 군집화 효과가 가장 적음.

**B. 체이닝의 진화 (Tree 기반)**
Java의 `HashMap` 등 최신 구현체는 체이닝 리스트의 길이가 임계값(보통 8)을 넘어가면, 연결 리스트를 **Red-Black Tree(레드-블랙 트리)**로 변환합니다. 이는 최악의 경우 $O(N)$인 리스트 탐색을 $O(\log N)$으로 개선합니다.

#### 📢 섹션 요약 비유
개방 주소법의 탐사 방식은 **'주차장에서 빈자리 찾기'**와 같습니다. 선형 탐사는 옆자리가 차면 바로 옆을 보고, 이차 탐사는 한 칸 띄고, 그 다음은 두 칸 띄고 보는 방식입니다. 이중 해싱은 오늘의 날씨나 요일처럼 랜덤한 요인을 섞어 다른 방향으로 움직이는 고급 주차 전략과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

해시 충돌 해결 기법은 단순 자료구조를 넘어 DBMS와 OS의 성능을 좌우합니다.

#### 1. 성능 메트릭스 비교 (Load Factor 대비 Time Complexity)

| 적재율 ($\alpha = N/S$) | Chaining (평균) | Open Addressing (성공) | Open Addressing (실패) |
|:---:|:---:|:---:|:---:|
| **0.5** | $< 1.0$ | $< 1.5$ | $< 2.0$ |
| **0.75** | $\approx 1.5$ | $\approx 2.0$ | $\approx 5.0$ |
| **0.9** | $\approx 2.0$ | $\approx 5.0$ | $\approx 50.0+$ |

*   **Chaining**: 적재율이 높아져도 성능 저하가 완만함 (포인터 따라가는 비용은 상대적 일정).
*   **Open Addressing**: 적재율이 높아질수록 탐사 길이가 기하급수적으로 증가함. 특히 실패(Serch Miss) 시 성능이 급격히 악화됨.

#### 2. DBMS 및 OS 융합 관점

**A. PostgreSQL Index & Chaining**
과거 PostgreSQL의 Hash Index는 체이닝 방식을 사용하여 버킷 오버플로우를 처리했습니다. 하지만 디스크 I/O 관점에서 연속되지 않은 페이지를 읽는 체이닝은 **Random Access**를 유발하여 성능 병목을 일으켰습니다. 이로 인해 실무에서는 B-Tree 인덱스가 더 선호되는 원인이 되었습니다.

**B. Redis & Open Addressing**
인메모리 DB인 Redis는 메모리를 효율적으로 쓰고 CPU 캐시 히트율을 높이기 위해 개방 주소법을 변형한 방식을 주로 사용합니다. 포인터를 따라가는 메모리 낭비를 줄이고 연속된 메모리 공간을 활용하여 **Latency**를 최소화합니다.

**C. OS Page Table**
운영체제의 페이지 테이블 구현 시, 체이닝 방식을 사용하면 Context Switching 시 페이지 테이블 전체를 스캔해야 하는 비용이 발생할 수 있어, 하드웨어 지원을 받는 **TLB(Translation Lookaside Buffer)**와 결합한 최적화된 방식이 필요합니다.

#### 📢 섹션 요약 비유
적재율에 따른 성능 차이는 **'만원 지하철'**과 같습니다. 체이닝은 칸막이를 설치해 사람을 밀어 넣는 방식이라 꽉 차도 그나비 들어가지만, 개방 주소법은 좌석 배정이라 꽉 차면 자리를 찾느라 기차가 출발하는 것을 놓칠 수 있습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

시스템 아키텍트는 비즈니스 요구사항에 따라 두 기법 중 하나를 선택해야 합니다.

#### 1. 실무 시나리오 및 의사결정 매트릭스

| 시나리오 | 추천 기법 | 이유 (Decision Factor) |
|:---|:---|:---|
| **실시간 캐시 시스템 (Redis)** | **Open Addressing** | 낮은 **Latency**와 높은 **Cache Locality**이 중요. 메모리 낭비를 줄여야 함. |
| **디스크 기반 DB 인덱스** | **Chaining (Tree-based)** | Random Access 비용이 크기 때문에, 연속된 메모리 공간보다는 포인터를 통한 논리적 연결이 유리함. |
| **대량 데이터 로그 적재** | **Chaining** | 데이터 양이 예측 불가능하고 **Load Factor**가 1.0 이상 초과 가능해야 함. |
| **임베디드 시스템 (저메모리)** | **Open Addressing** | 포인터 변수 저장 공간조차 아까운 환경. 배열 하나로 해결해야 함. |

#### 2. 도입 체크리스트

-   **[ ] 데이터 분포도 확인**: 입력 데이터의 해시 분포가 고른지? (편중되면 Open Addressing에서 긴 탐사 발생)
-   **[ ] 메모리 허용 범위**: 메모리 여유가 있어 Load Factor를 낮게(0.5 이하) 유지할 수 있는가? (Open Addressing 조건)
-   **[ ] 삭제 빈도**: 데이터 삭제가 빈번한가? (Chaining이 유리, Open Addressing은 복잡한 로직 필요)
-   **[ ] 동시성 제어 (Concurrency)**: 멀티스레드 환경인가? (Chaining은 Lock이 좁혀지지만, Open Addressing은 재귀적 탐사 시 Lock 범위가 넓어질 수 있음)

#### 3. 안티패턴 (Anti-Pattern)

-   **Linear Probing 과신**: 데이터가 많아지면 군집화(Clustering)로 인해 **Performance Degradation**이 급격히 일어남. 반드시 Quadratic이나 Double Hashing을 고려해야 함.
-   **리해싱(Rehashing) 지연**: Load Factor가 한계치(Open Addressing 시 0.7~0.8)를 넘었음에도 테이블 크기를 늘리지 않으면 조회 성능이 **Sluggish**해짐.
