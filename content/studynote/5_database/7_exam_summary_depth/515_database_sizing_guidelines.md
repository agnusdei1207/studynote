# 515. 데이터베이스 사이징(Sizing) - 미래를 설계하는 자원 산정

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 사이징(Sizing)은 비즈니스 요구사항(사용자 수, 데이터 발생량)을 기반으로 시스템이 안정적으로 돌아가기 위해 필요한 **CPU, 메모리, 스토리지 자원을 공학적으로 계산하고 가용성 여유율을 반영하는 과정**이다.
> 2. **가치**: 자원 부족으로 인한 성능 지연을 예방하고, 반대로 과도한 투자로 인한 비용 낭비(Over-provisioning)를 막아 **최적의 TCO(총소유비용)**를 실현한다.
> 3. **융합**: 트랜잭션 복잡도 분석, 피크 타임 트래픽 예측, 데이터 보존 정책 기술이 융합되어, 장애 상황에서도 서비스 연속성을 보장하는 견고한 인프라 기초를 형성한다.

+++

### Ⅰ. 주요 자원별 산정 기준

| 자원 항목 | 핵심 산정 근거 | 여유율 적용 (Best Practice) |
|:---|:---|:---|
| **CPU** | 초당 트랜잭션(TPS), 쿼리 복잡도 | Peak 시 사용률이 **50~60%**를 넘지 않도록 설정 |
| **Memory** | Hot Data(자주 쓰는 데이터) 크기, 인덱스 전체 크기 | 전체 데이터의 **10~20%**가 RAM에 상주하도록 권장 |
| **Storage** | 초기 용량 + (일일 증가량 x 보존 기간) | RAID 구성 손실분 및 **20~30%의 추가 여유 공간** 확보 |

+++

### Ⅱ. 사이징 프로세스 및 여유율 시각화 (ASCII Model)

```text
[ Database Capacity Planning ]

  (Input: Biz Requirement)

  - 5 Years Retention
          │

  ├───────────────────────────┤

  └─────────────┬─────────────┘

      [ Final System Sizing ] ✅ (안정적 운영 보장)
```

### Ⅲ. 가용성 여유율 산정 시 고려사항

1. **이중화 방식**: Active-Active 구조인지 Active-Standby 구조인지에 따라 필요한 물리 자원 대수가 결정됩니다.
2. **백업 및 복구 공간**: 운영 데이터 외에도 스냅샷, 아카이브 로그, 풀 백업본을 저장할 별도의 독립적인 공간(보통 운영의 2배 이상)이 필요합니다.
3. **인덱스 및 오버헤드**: 순수 데이터 값 외에도 인덱스, 임시 세그먼트(정렬용), 시스템 카탈로그가 차지하는 공간(약 30% 이상)을 반드시 포함해야 합니다.

- **📢 섹션 요약 비유**: 데이터베이스 사이징은 **'명절날 가족 식사 준비'**와 같습니다. 평소 식구 수(평균 부하)에 딱 맞춰 음식을 준비하면, 갑자기 손님이 오거나(Peak 트래픽) 누군가 음식을 쏟았을 때(장애 발생) 대처할 길이 없습니다. 조금 넉넉하게 재료를 사고(여유율 반영), 만약을 대비해 여분의 그릇(가용성 자원)을 챙겨두어야 비로소 완벽한 잔치(안정적 서비스)가 가능한 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[TCO (Total Cost of Ownership)]**: 사이징을 통해 최적화해야 할 전체 비용 지표.
- **[Benchmark]**: TPC-C 등을 활용해 하드웨어의 실제 처리 능력을 검증하는 과정.
- **[Scalability]**: 사이징 예측이 틀렸을 때를 대비한 클라우드의 탄력적 확장 능력.

📢 **마무리 요약**: **Database Sizing**은 불확실한 미래에 대한 기술적 보험입니다. 치밀한 데이터 분석을 바탕으로 적정한 여유를 설계할 때, 시스템은 어떤 위기 속에서도 흔들리지 않는 가용성을 증명합니다.
