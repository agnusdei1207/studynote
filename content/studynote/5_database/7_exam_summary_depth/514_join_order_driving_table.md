# 514. 조인 순서와 드라이빙 테이블 - 집합의 크기가 성능을 결정한다

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 조인 순서(Join Order)는 여러 테이블을 결합할 때 어느 테이블을 먼저 읽을지 정하는 것이며, **드라이빙 테이블(Driving Table)**은 조인의 시작점이 되는 첫 번째 테이블을 의미한다.
> 2. **가치**: 필터 조건에 의해 걸러진 **결과 집합이 가장 작은 테이블을 드라이빙으로 선정**함으로써, 후행 테이블(Driven)에 대한 탐색 횟수를 최소화하여 쿼리 전체의 I/O 비용을 획기적으로 낮춘다.
> 3. **융합**: 중첩 루프 조인(NL Join) 환경에서 특히 결정적인 성능 변수가 되며, 옵티마이저 힌트(`LEADING`, `ORDERED`)를 통해 수동으로 제어하여 실행 계획을 최적화할 수 있다.

+++

### Ⅰ. 드라이빙 테이블 선정의 대원칙

- **작은 집합 우선 (Smallest Set First)**: 테이블 전체 크기가 아닌, **WHERE 절의 조건에 의해 필터링된 후의 크기**가 작은 테이블을 먼저 읽어야 합니다.
- **인덱스 전략 연계**: 후행 테이블(Driven)의 조인 컬럼에는 반드시 효율적인 인덱스가 존재해야 조인 속도가 보장됩니다.

+++

### Ⅱ. 조인 순서에 따른 부하 차이 시각화 (ASCII Model)

```text
[ Case 1: Big Table First (Bad) ]      [ Case 2: Small Table First (Good) ]

  Driving: Table A (1,000 Rows)          Driving: Table B (10 Rows) ✅
     │                                      │
     ▼ (Loop 1,000 times)                   ▼ (Loop 10 times) ✅
  Driven: Table B (Index Scan)           Driven: Table A (Index Scan)
  ------------------------------         ------------------------------
  Total I/O: High (1,000 probes)         Total I/O: Low (10 probes) ✅
```

+++

### Ⅲ. 조인 순서 결정 요인

1. **조건절의 존재**: 상수 조건(`WHERE Dept='IT'`)이 있는 테이블이 우선순위가 높습니다.
2. **조인 연산자**: 해시 조인의 경우 메모리에 올릴 '빌드 테이블'을 작게 잡기 위해 조인 순서를 조정합니다.
3. **인덱스 유무**: 인덱스가 없는 테이블을 드라이빙으로 써서 풀 스캔 후, 인덱스가 있는 테이블을 프로브(Probe) 하는 것이 유리할 때가 많습니다.

- **📢 섹션 요약 비유**: 조인 순서 선정은 **'전화번호부에서 친구 10명 찾기'**와 같습니다. 내 수첩(작은 집합)에 적힌 10명의 이름을 먼저 보고(Driving), 두꺼운 전화번호부(큰 집합)의 인덱스를 10번 뒤지는 것(Driven)은 쉽습니다. 하지만 전화번호부의 수만 명을 한 명씩 다 보면서(Driving), 수첩에 그 이름이 있는지 대조하는 것은 엄청나게 미련한 짓입니다. 작은 것을 먼저 쥐는 것이 조인의 핵심입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Driving vs Driven]**: 조인의 주인과 종속 관계.
- **[Filter Predicate]**: 드라이빙 테이블 선정을 돕는 조건절.
- **[Nested Loop Join]**: 조인 순서에 가장 민감하게 반응하는 조인 방식.

📢 **마무리 요약**: **Join Order**는 SQL 튜닝의 절반을 차지합니다. 어떤 데이터를 먼저 손에 쥐느냐에 따라 시스템의 응답 시간이 수백 배 달라질 수 있음을 명심해야 합니다.
