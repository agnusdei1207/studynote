# 448. 유령 읽기(Phantom Read) - 보이지 않던 데이터의 습격

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 유령 읽기는 한 트랜잭션 내에서 **동일한 조건의 범위 조회(Range Query)를 두 번 수행했을 때, 다른 트랜잭션의 INSERT로 인해 첫 번째 조회에는 없던 행이 두 번째 조회에 나타나는 현상**이다.
> 2. **가치**: 개별 행의 수정을 다루는 비반복 읽기와 달리 '데이터 집합의 변화'를 다루며, 집계 데이터나 통계적 일관성을 해치는 고차원적인 동시성 이상 현상이다.
> 3. **융합**: 최고 격리 수준인 'Serializable'에서만 완벽히 방지되며, 현대 DB에서는 **넥스트 키 락(Next-Key Lock)**이나 MVCC 기반의 정교한 스냅샷 제어를 통해 이를 융합 해결한다.

+++

### Ⅰ. 유령 읽기의 발생 시나리오

1. **범위 조회**: 트랜잭션 T1이 "나이가 20세 이상인 회원 수"를 조회합니다 (결과: 5명).
2. **타인의 삽입**: 그 사이 다른 트랜잭션 T2가 "나이가 25세인 신규 회원"을 추가하고 **커밋**합니다.
3. **재조회**: T1이 다시 같은 조건으로 회원 수를 세어보니 이번에는 '6명'이 나옵니다.
4. **결과**: T1은 데이터가 갑자기 유령처럼 나타난 것에 혼란을 느끼며, 집계의 일관성이 깨지게 됩니다.

+++

### Ⅱ. 유령 읽기 타임라인 시각화 (ASCII Flow)

```text
[ Phantom Read Scenario ]

  Time | Transaction T1 (Counter) | Transaction T2 (Inserter) | Actual DB
  -----|--------------------------|---------------------------|----------
   t1  | SELECT count(*) WHERE age > 20 ──▶ [5명]            |   5명
   t2  |                          | INSERT (name: 'New', age: 25)
   t3  |                          | 🟢 COMMIT!                |   6명 ✅
   t4  | SELECT count(*) WHERE age > 20 ──▶ [6명] 💥         |   6명
  -----------------------------------------------------------------------
  * Result: A "Phantom" record appeared in T1's second query.
```

+++

### Ⅲ. 차단 기술: 넥스트 키 락 (Next-Key Lock)

- **원리**: 단순히 데이터가 있는 행만 잠그는 것이 아니라, 데이터와 데이터 사이의 빈 공간(Gap)까지 몽땅 잠가버립니다.
- **효과**: 다른 트랜잭션이 해당 범위(Gap)에 새로운 데이터를 끼워 넣으려고 해도 락 때문에 대기하게 되어 유령의 발생을 원천 차단합니다.
- **MySQL 예시**: InnoDB 엔진은 Repeatable Read 격리 수준에서도 이 넥스트 키 락을 사용하여 유령 읽기를 대부분 방어합니다.

- **📢 섹션 요약 비유**: 유령 읽기는 **'학생 출석 체크'**와 같습니다. 선생님(T1)이 교실 안의 학생 수를 다 세었는데(5명), 잠시 뒤에 복도에 있던 전학생(T2)이 몰래 들어와 앉아 있는 바람에 다시 세어보니 인원이 늘어난(6명) 상황입니다. 이를 막으려면 출석 체크가 끝날 때까지 교실 문을 잠가(Next-Key Lock) 아무도 못 들어오게 해야 합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Range Lock]**: 특정 범위 전체를 잠그는 기술.
- **[Serializable]**: 모든 읽기/쓰기를 직렬화하여 유령을 막는 최고 레벨.
- **[Gap Lock]**: 인덱스 레코드 사이의 빈 공간을 보호하는 락.

📢 **마무리 요약**: **Phantom Read**는 데이터의 '집합적 진실'을 위협합니다. 보이지 않는 빈틈까지 관리하는 세밀한 잠금 전략이 대규모 정산 시스템의 무결성을 완성합니다.
