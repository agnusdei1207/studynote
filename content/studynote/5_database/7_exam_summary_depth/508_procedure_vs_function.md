# 508. 저장 프로시저 vs 사용자 정의 함수 - 절차적 SQL의 정석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 저장 프로시저(Procedure)와 사용자 정의 함수(Function)는 SQL을 로직으로 묶어 **DB 서버에 저장하고 컴파일하여 사용하는 절차적 모듈**이지만, 실행 목적과 반환 방식에서 결정적 차이가 있다.
> 2. **가치**: 반복적인 쿼리를 하나의 명령어로 재사용하여 생산성을 높이며, 미리 컴파일된 실행 계획을 공유함으로써 **네트워크 트래픽 감소와 성능 최적화**를 달성한다.
> 3. **융합**: 트랜잭션 관리와 복잡한 비즈니스 로직 처리는 프로시저가, 단순 계산이나 SQL 문장 내의 표현식 확장은 함수가 담당하며 상호 보완적으로 융합된다.

+++

### Ⅰ. 프로시저 vs 함수 상세 비교

| 비교 항목 | 저장 프로시저 (Procedure) | 사용자 정의 함수 (Function) |
|:---|:---|:---|
| **반환값 (Return)** | 0개 이상 (OUT 파라미터 활용) | **반드시 1개의 값 반환** |
| **호출 방식** | `EXECUTE` 명령으로 단독 실행 | **SQL 문장(SELECT 등)의 일부**로 사용 |
| **트랜잭션** | `COMMIT`, `ROLLBACK` 가능 | **트랜잭션 제어 원칙적 불가** |
| **주요 용도** | 복잡한 비즈니스 프로세스, 배치 | 단순 계산, 형식 변환, 데이터 가공 |
| **특징** | 서버 사이드 로직의 중심 | SQL의 기능을 확장하는 도구 |

+++

### Ⅱ. 컴파일 및 실행 매커니즘 시각화 (ASCII Model)

```text
[ Pre-compiled Execution Architecture ]

  ┌───────────────────────▼─────────────────────────────────────┐

  └───────────────────────┬─────────────────────────────────────┘
                          ▼
                 [ Actual Data Access ] ──▶ ✅ Fast Response
```

### Ⅲ. 공통적인 장점: 성능과 보안

1. **절차적 최적화**: SQL의 선언적 한계를 넘어 `IF`, `LOOP` 등을 사용하여 고도의 로직을 DB 근처에서 수행합니다.
2. **보안 강화**: 테이블에 직접 접근 권한을 주지 않고, 특정 프로시저/함수 실행 권한만 부여하여 데이터를 보호(캡슐화)할 수 있습니다.
3. **네트워크 부하 절감**: 수백 줄의 SQL을 네트워크로 보내는 대신, 짧은 호출 명령 한 줄만 보내어 통신 효율을 높입니다.

- **📢 섹션 요약 비유**: 프로시저와 함수는 **'완제품 요리와 소스'**의 관계와 같습니다. 프로시저는 '밀키트 세트'와 같아서, 물만 붓고 끓이면(호출) 복잡한 요리가 한꺼번에 완성(프로세스 처리)되는 방식입니다. 반면 함수는 '만능 양념장'과 같아서, 내가 직접 요리하는 중간에(SQL 문장 안에서) 한 숟갈씩 떠넣어 맛을 내는(값 반환) 용도로 쓰입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Procedure]**: '어떻게(How)' 처리할 것인가에 집중한 로직 덩어리.
- **[UDF (User Defined Function)]**: '무엇(What)'을 계산해낼 것인가에 집중한 도구.
- **[Parameter Sniffing]**: 프로시저 성능 저하의 주요 원인이 되는 실행 계획 오염 현상.

📢 **마무리 요약**: **Stored Object**는 데이터베이스의 지능입니다. 로직을 데이터 근처로 전진 배치함으로써, 시스템 전체의 기동성과 정합성을 완벽하게 지켜냅니다.
