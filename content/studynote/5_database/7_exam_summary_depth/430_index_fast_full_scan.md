# 430. 인덱스 패스트 풀 스캔(Index Fast Full Scan) - 속도에 올인한 탐색

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인덱스 패스트 풀 스캔(IFFS)은 인덱스의 논리적 구조(B+Tree 순서)를 무시하고, **디스크에 저장된 물리적 순서대로 모든 인덱스 블록을 멀티 블록 I/O 방식으로 한꺼번에 읽어 들이는 고속 스캔 방식**이다.
> 2. **가치**: 테이블 전체를 읽는 것보다 훨씬 작은 크기의 인덱스만 읽으면서도 **병렬 처리(Parallel)와 멀티 블록 읽기**를 활용해 극단적인 조회 속도를 제공하며, 주로 `COUNT(*)`나 커버링 인덱스 쿼리에 사용된다.
> 3. **특징**: 논리적 순서로 읽지 않기 때문에 **결과 데이터가 정렬(Sort)되어 있지 않으며**, 반드시 필요한 모든 컬럼이 인덱스에 포함되어 있을 때만 수행 가능하다.

+++

### Ⅰ. 작동 원리: 논리보다 물리

- **일반 풀 스캔 (Index Full Scan)**: 리프 노드를 처음부터 끝까지 연결 포인터를 따라 읽습니다. (논리적 정렬 보장, Single Block Read 위주)
- **패스트 풀 스캔 (Index Fast Full Scan)**: 디스크 상의 인덱스 세그먼트 전체를 순차적으로 긁어옵니다. (정렬 보장 안 됨, Multi-Block Read 활용)

+++

### Ⅱ. 스캔 방식 비교 시각화 (ASCII Model)

```text
[ Index Full Scan (Logical) ]        [ Index Fast Full Scan (Physical) ]

  ┌──▶┌───┐ ┌──▶┌───┐                  ┌───────┬───────┬───────┐
  │   │ L1│─┘   │ L2│                  │ Blk 1 │ Blk 2 │ Blk 3 │
  └───┴───┘     └───┘                  └───────┴───────┴───────┘
  (하나씩 순서대로 따라감)               (통째로 한꺼번에 읽음) ✅
  Result: [1, 2, 3, 4...] (Sorted)     Result: [3, 1, 4, 2...] (Unsorted)
```

+++

### Ⅲ. 언제 사용되는가? (Use Case)

1. **대량의 집계**: `SELECT COUNT(*) FROM Large_Table;` (인덱스만 다 읽으면 답이 나옴)
2. **커버링 인덱스 쿼리**: `SELECT Name, Dept FROM User;` (두 컬럼이 결합 인덱스로 묶여 있을 때)
3. **병렬 처리 극대화**: 여러 프로세스가 인덱스 블록을 나누어 동시에 읽어야 할 때.

- **📢 섹션 요약 비유**: 인덱스 패스트 풀 스캔은 **'시험 기간의 속독법'**과 같습니다. 한 문장 한 문장 논리적인 흐름을 따라 정독하는 게 아니라(일반 풀 스캔), 필요한 단어(인덱스 컬럼)만 찾기 위해 책장을 수십 장씩 한꺼번에 넘기며 눈으로 훑는 것입니다. 내용은 다 읽을 수 있고 속도는 엄청나게 빠르지만, 머릿속에서 정보가 순서대로 정리(정렬)되지는 않는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Multi-Block Read]**: IFFS 성능의 핵심 물리적 기술.
- **[Parallel Query]**: IFFS와 찰떡궁합인 대규모 병렬 처리.
- **[Covering Index]**: IFFS가 실행되기 위한 논리적 전제 조건.

📢 **마무리 요약**: **Index Fast Full Scan**은 정렬을 포기하고 속도를 얻은 실용적인 기법입니다. 데이터의 양이 거대해질수록, 논리적인 우아함보다 물리적인 파괴력을 선택하는 이 방식이 빛을 발합니다.
