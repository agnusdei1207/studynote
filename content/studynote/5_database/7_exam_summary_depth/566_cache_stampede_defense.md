# 566. 캐시 스탬피드(Cache Stampede) - 분산 시스템의 압사 사고 방지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 캐시 스탬피드는 대규모 트래픽 환경에서 **특정 인기 데이터의 캐시가 만료되는 순간, 수많은 요청이 일제히 DB로 몰려들어 시스템 전체를 마비시키는 현상**이다.
> 2. **가치**: 단일 장애 지점(SPOF)이 된 데이터베이스의 과부하를 방어하며, **Mutex 락**이나 **확률적 조기 갱신(PER)** 기술을 통해 서비스 가용성을 보장한다.
> 3. **융합**: 분산 락 기술과 확률 통계 기반의 알고리즘이 융합되어, 조회 성능은 유지하면서도 예측 불가능한 트래픽 피크 타임의 장애를 선제적으로 회피한다.

+++

### Ⅰ. 캐시 스탬피드의 발생 원인 (Thundering Herd)

- **상황**: 캐시 서버에 'Hot Key' (예: 실시간 검색어 1위)가 저장되어 있습니다.
- **사건**: 해당 키의 TTL(만료 시간)이 종료되어 데이터가 사라집니다.
- **결과**: 그 찰나에 들어온 수만 건의 요청이 캐시 미스(Miss)를 내고, 한꺼번에 원본 DB로 질의를 던져 DB를 다운시킵니다. 💥

+++

### Ⅱ. 회피 기법 1: Mutex Lock (분산 락)

가장 직관적인 방법으로, DB에 접근하는 통로를 1개로 제한합니다.

```text
[ Cache Stampede Defense: Mutex ]

  (Users) ──▶ [ Cache (Expired) ] 💥
                 │
          ┌──────┴──────┬───────────────┐
          ▼             ▼               ▼
  (Request 1)   (Request 2) ... (Request N)
     │ (Lock ✅)    │ (Wait... ⌛️)   │ (Wait... ⌛️)
     ▼              └───────┬───────┘
  [ Database ]              │
     │                      │
     └─▶ [ Update Cache ] ──┘ ──▶ ✅ "Everyone reads from Cache again"
```

+++

### Ⅲ. 회피 기법 2: 확률적 조기 갱신 (PER)

- **원리**: 캐시가 실제로 만료되기 직전에, 들어오는 요청 중 일부가 확률적으로 미리 캐시를 갱신하게 만드는 알고리즘입니다.
- **공식**: `currentTime - (slot * gap * log(random())) > expiryTime`
- **장점**: 락을 거는 오버헤드 없이도 자연스럽게 캐시 만료 시점을 분산시켜 '절벽 현상'을 막습니다. ✅

- **📢 섹션 요약 비유**: 캐시 스탬피드는 **'유명 맛집의 재료 소진'**과 같습니다. 맛집 사장님이 재료가 떨어졌다고 선언하는 순간(캐시 만료), 줄 서 있던 수천 명의 손님(요청)이 한꺼번에 항의하러 주방(DB)으로 몰려들어 주방이 쑥대밭이 되는 것과 같습니다. 이를 막으려면 번호표(Mutex)를 나눠주어 한 명씩만 주방장과 대화하게 하거나, 재료가 완전히 떨어지기 전에 눈치 빠른 직원(PER)이 미리 장을 보러 다녀오는 전략이 필요합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Hot Key]**: 스탬피드의 주된 대상이 되는 인기 데이터.
- **[PER (Probabilistic Early Recomputation)]**: 통계적으로 캐시 만료를 회피하는 세련된 기법.
- **[Jitter]**: 만료 시간에 미세한 난수를 더해 만료 시점을 분산시키는 단순한 방법.

📢 **마무리 요약**: **Cache Stampede Recovery**는 대규모 서비스의 안전핀입니다. 단순한 캐시 적용을 넘어 '만료의 순간'을 공학적으로 설계할 때, 시스템은 비로소 진정한 안정성을 얻게 됩니다.
