# 527. 데이터베이스 접근 제어(DAM) - 실시간 감시와 통제

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DAM(Database Activity Monitoring)은 데이터베이스로 유입되는 모든 쿼리와 접속 활동을 **실시간으로 가로채어 분석하고, 사전에 정의된 보안 정책에 따라 허용 또는 차단하는 보안 기술**이다.
> 2. **가치**: 인가된 사용자의 오남용이나 SQL 인젝션 같은 공격을 즉시 탐지하며, 누가 어떤 데이터를 조회했는지에 대한 **완벽한 감사 증적(Audit Trail)**을 남겨 부인 방지를 실현한다.
> 3. **융합**: 네트워크 스니핑(Sniffing), 게이트웨이(Proxy), 에이전트(Agent) 기술이 융합되어 DB 본체에 부하를 주지 않으면서도 철저한 외부 감시망을 형성한다.

+++

### Ⅰ. DAM의 주요 구현 방식

1. **Gateway (Proxy) 방식**: 애플리케이션과 DB 사이의 통로에 위치하여 모든 패킷을 직접 중계하고 제어함. (강력한 차단 가능)
2. **Sniffing (Mirroring) 방식**: 네트워크 스위치에서 복제된 패킷을 분석함. (DB 성능 부하 0, 하지만 실시간 차단은 어려움)
3. **Agent 방식**: DB 서버 내부에 설치되어 커널 레벨에서 활동을 기록함. (로컬 접속까지 감시 가능)

+++

### Ⅱ. DAM 작동 아키텍처 시각화 (ASCII Model)

```text
[ Database Access Control Pipeline ]

  (User / App) ──▶ [ DAM Gateway ] ──▶ [ Database Engine ]
                          │                   │
          ┌───────────────┴───────────────────┘
          ▼
  [ Security Policy Engine ] ◀─── (Context: User, Query, Time)
          │
          ├─▶ (If OK) ──▶ ✅ Forward Query
          └─▶ (If Bad) ──▶ ❌ Block & Alert! 💥
          │
          ▼
  [ Secure Audit Storage ] ──▶ 💾 Immutable Logs
```

+++

### Ⅲ. DAM의 핵심 보안 기능

- **SQL 인젝션 차단**: 비정상적인 쿼리 패턴을 감지하여 실행 전 차단.
- **개인정보 조회 통제**: 하루에 조회할 수 있는 개인정보 건수를 제한(Threshold)하여 대량 유출 방지.
- **권한 오남용 감시**: 평소에 안 보던 테이블을 갑자기 대량 조회하는 행위 등을 AI로 탐지.
- **우회 접속 차단**: 배스천 호스트나 특정 IP를 거치지 않은 직접 접속을 원천 봉쇄.

- **📢 섹션 요약 비유**: DAM은 **'은행 창구의 무장 보안 요원'**과 같습니다. 은행원(DB)에게 돈(데이터)을 찾으러 온 모든 고객(사용자)의 신분증을 확인하고, 가방 안의 내용물(쿼리 의도)을 검사하며, 수상한 행동을 하면 그 자리에서 즉시 제지(차단)하는 역할을 합니다. 또한 그날 누가 다녀갔는지 모든 영상을 기록(감사 로그)하여 사고에 대비합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[SQL Injection Prevention]**: DAM의 가장 고전적이고 중요한 기능.
- **[Audit trail]**: 사후 분석을 위해 남겨지는 정교한 행위 기록.
- **[VPD vs DAM]**: DB 엔진 내장형 보안(VPD)과 외부 솔루션형 보안(DAM)의 공존.

📢 **마무리 요약**: **DAM**은 데이터베이스의 외부 면역 체계입니다. 데이터의 입구를 장악하고 모든 움직임을 기록할 때, 비로소 데이터 주권과 보안의 신뢰가 완성됩니다.
