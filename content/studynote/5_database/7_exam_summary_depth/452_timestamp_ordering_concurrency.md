# 452. 타임스탬프 순서(Timestamp Ordering) - 시간의 선후 관계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 타임스탬프 순서는 잠금(Lock)을 사용하지 않고, 트랜잭션이 시스템에 들어온 순서대로 부여된 **고유의 타임스탬프(TS)를 기준으로 연산의 실행 순서를 결정하는 동시성 제어 기법**이다.
> 2. **가치**: 데이터를 기다리게 하는 대신 규칙 위반 시 즉시 롤백하므로 **교착 상태(Deadlock)가 원천적으로 발생하지 않으며**, 읽기 전용 작업이 많은 환경에서 효율적이다.
> 3. **융합**: 데이터 항목마다 마지막 읽기($R-TS$)와 쓰기($W-TS$) 시간을 기록하여, 나중에 온 트랜잭션이 과거의 데이터를 건드리지 못하게 하는 수학적 질서를 확립한다.

+++

### Ⅰ. 타임스탬프 순서 기법의 작동 규칙

트랜잭션 $T$가 데이터 $Q$에 접근할 때의 판단 로직입니다.

1. **Read(Q)**:
    - `TS(T) < W-TS(Q)` 이면: 미래의 트랜잭션이 이미 $Q$를 바꿨으므로, $T$는 거절되고 **Rollback**.
    - 아니면: 허용하고 `R-TS(Q)`를 `MAX(R-TS(Q), TS(T))`로 갱신.
2. **Write(Q)**:
    - `TS(T) < R-TS(Q)` 이면: 미래의 트랜잭션이 이미 $Q$를 읽어갔으므로, $T$는 거절되고 **Rollback**.
    - `TS(T) < W-TS(Q)` 이면: 미래의 트랜잭션이 이미 $Q$에 썼으므로, $T$는 거절되고 **Rollback**.
    - 아니면: 허용하고 `W-TS(Q)`를 `TS(T)`로 갱신.

+++

### Ⅱ. 타임스탬프 로직 시각화 (ASCII Flow)

```text
[ Timestamp Decision Process ]

  (Transaction T) ──▶ [ Operation Request ]
                             │
          ┌──────────────────┴──────────────────┐
          ▼ (Check logic)                       ▼
  [ IF TS(T) < Data_TS ]                [ IF TS(T) >= Data_TS ]
          │                                     │
          ▼                                     ▼
  [ ❌ ROLLBACK & RESTART ]             [ ✅ ALLOW & UPDATE TS ]
  "이미 늦었습니다! 다시 시도하세요."      "순서가 맞군요. 진행하세요."
```

+++

### Ⅲ. 장단점 분석

- **장점**: 락 관리 오버헤드가 없고, 교착 상태가 절대로 발생하지 않습니다.
- **단점**: 트랜잭션 간 충돌이 잦으면 반복적인 롤백으로 인해 자원이 낭비되고, 먼저 온 트랜잭션이 계속 뒤로 밀리는 **기아 현상(Starvation)**이 생길 수 있습니다.

- **📢 섹션 요약 비유**: 타임스탬프 순서는 **'번호표가 있는 은행 창구'**와 같습니다. 10번 손님이 업무를 보고 있는데 9번 손님이 뒤늦게 달려와서 "내가 먼저 할래!"라고 떼를 써도, 은행원(DBMS)은 "이미 10번 번호표가 처리되었으니 9번은 유효하지 않습니다. 다시 번호표를 뽑아오세요(Rollback)"라고 단호하게 거절하는 질서 정연한 방식입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Deadlock Free]**: 타임스탬프 기법의 가장 큰 물리적 강점.
- **[Thomas Write Rule]**: 늦게 온 쓰기가 과거의 것이라면 무시하고 진행하는 성능 최적화 규칙.
- **[Logical Clock]**: 물리적 시간 대신 단조 증가하는 카운터를 사용하는 방식.

📢 **마무리 요약**: **Timestamp Ordering**은 락의 복잡함을 시간의 선후 관계로 치환한 지혜입니다. 대기 없는 빠른 처리가 필요한 분산 환경의 핵심 알고리즘으로 자리 잡고 있습니다.
