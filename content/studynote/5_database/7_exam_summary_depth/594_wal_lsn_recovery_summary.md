# 594. WAL과 LSN 기반 복구 - 데이터 무결성의 물리적 증명

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: WAL(Write-Ahead Logging)은 데이터를 디스크에 쓰기 전 로그를 먼저 기록하는 규약이며, **LSN(Log Sequence Number)은 각 로그 레코드에 부여되는 고유 번호로 데이터 페이지의 최신성 여부를 판단하는 절대 기준**이다.
> 2. **가치**: LSN 기반 체크포인트를 통해 복구 시작 지점을 최적화하고, 장애 후 **롤 포워드(Roll-forward)** 과정을 통해 디스크 데이터와 로그 간의 불일치를 수학적으로 해결하여 무결성을 완성한다.
> 3. **융합**: 로그 플러시 기술, 페이지 레벨의 LSN 추적 기술, 그리고 비휘발성 저장 기술이 융합되어 어떤 시점의 크래시에서도 일관된 복구를 가능케 한다.

+++

### Ⅰ. WAL과 LSN의 상호작용 원리

- **WAL**: "로그가 디스크에 먼저 박히지 않으면, 데이터 페이지는 절대 디스크로 나갈 수 없다"는 철칙.
- **LSN**: 로그가 생성될 때마다 1씩 증가하는 일련번호. 
- **pageLSN**: 모든 데이터 페이지 헤더에 해당 페이지를 마지막으로 수정한 로그의 LSN을 저장. ✅
- **복구 로직**: `IF (Log_LSN > Page_LSN) THEN REDO;` (로그가 더 최신이면 다시 실행하라!)

+++

### Ⅱ. LSN 기반 복구 프로세스 시각화 (ASCII Model)

```text
[ Recovery Logic: LSN Comparison ]

  (Redo Log File)
  [ LSN: 500 | Update A=20 ] ──▶ (1) 읽기
           │
           ▼ (2) 비교
  (Data Page A)
  [ pageLSN: 450 ] ◀────────────┘
  
  (Conclusion)
  450 < 500 : "데이터 페이지가 로그보다 과거의 것이군!"
  ──▶ [ ⚙️ REDO ACTION ] ──▶ Page A updated to 20 & pageLSN to 500 ✅
```

+++

### Ⅲ. 미디어 장애와 롤 포워드 (Roll-forward)

- **상황**: 디스크 파손(미디어 장애)으로 백업본을 복원한 상태 (시점 T0).
- **롤 포워드**: T0 이후부터 현재까지의 모든 로그(LSN 기반)를 순차적으로 재실행하여 장애 직전 상태로 데이터를 전진시키는 과정입니다.
- **체크포인트의 역할**: LSN 1000번까지는 디스크 반영이 확실하다는 표식을 남겨, 복구 시 1번부터 읽는 삽질을 막아줍니다.

- **📢 섹션 요약 비유**: WAL과 LSN은 **'공사 현장의 작업 일지와 벽돌 도장'**과 같습니다. 벽돌을 쌓기 전 반드시 일지(로그)에 먼저 적어야 하고(WAL), 벽돌 하나를 쌓을 때마다 일지의 몇 번째 줄인지 번호(LSN)를 벽돌에 도장(pageLSN)으로 찍어두는 것입니다. 사고가 나서 벽이 무너져도, 일지의 번호와 벽돌의 도장을 대조해보면 어디까지가 진짜인지, 어디부터 다시 쌓아야(Redo) 하는지 완벽하게 알 수 있는 원리입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Flushed LSN]**: 디스크에 안전하게 저장된 마지막 로그 번호.
- **[Dirty Page]**: 로그는 써졌지만 데이터 파일엔 아직 반영 안 된 메모리 페이지.
- **[Recovery Path]**: LSN을 따라가며 데이터를 치유하는 최단 경로.

📢 **마무리 요약**: **WAL & LSN**은 데이터베이스의 블랙박스이자 시계입니다. 이 정교한 번호 체계가 있기에 데이터베이스는 시간의 풍파 속에서도 변치 않는 무결성을 유지할 수 있습니다.
