# 450. 2단계 잠킹 프로토콜(2PL) - 직렬성의 보증

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 2단계 잠킹(2PL, Two-Phase Locking)은 트랜잭션 내에서 모든 잠금(Lock) 요청이 첫 번째 해제(Unlock) 연산보다 먼저 일어나야 한다는 규칙을 통해, **비직렬 스케줄의 직렬 가능성(Serializability)을 수학적으로 보장하는 규약**이다.
> 2. **가치**: 데이터베이스 엔진이 충돌 직렬 가능성을 실시간으로 강제할 수 있는 가장 표준적인 알고리즘이며, 동시성 제어의 이론적 완성도를 현실에 구현한다.
> 3. **융합**: 확장 단계(Growing)와 축소 단계(Shrinking)의 명확한 구분을 통해 데이터 일관성을 지키며, 교착 상태(Deadlock) 가능성을 내포하는 트레이드오프를 가진다.

+++

### Ⅰ. 2PL의 2가지 핵심 단계

1. **확장 단계 (Growing Phase)**: 
    - 트랜잭션이 필요한 잠금을 계속해서 획득해 나가는 단계입니다. 
    - 이 단계에서는 **잠금을 해제할 수 없습니다.**
2. **축소 단계 (Shrinking Phase)**: 
    - 잠금을 하나씩 반납하는 단계입니다. 
    - 이 단계가 시작되면(첫 번째 Unlock 발생) **절대 새로운 잠금을 요청할 수 없습니다.**

+++

### Ⅱ. 2PL 상태 변화 시각화 (ASCII Model)

```text
[ 2PL Phase Transition ]

  Number of
  Locks Held
      ▲
      │        [ 1. 확장 단계 (Growing) ]
      │             (Lock 가능, Unlock 불가)
      │               /───┐
      │              /    │  <-- [ Lock Point ] (모든 락 획득 시점)
      │             /     │
      │            /      │  [ 2. 축소 단계 (Shrinking) ]
      │           /       │       (Unlock 가능, Lock 불가)
      │          /        └───\
      │         /              \
      └─────────┴────────────────┴──────────────────▶ Time
```

+++

### Ⅲ. 2PL의 한계와 변형

- **데드락 발생**: 2PL은 직렬 가능성은 보장하지만, 서로의 락을 기다리는 교착 상태를 예방하지는 못합니다.
- **연쇄 복귀 위험**: 기본 2PL은 커밋 전에 락을 풀 수 있어, 그 데이터를 읽은 다른 트랜잭션이 롤백 시 함께 취소되는 연쇄 복귀가 발생할 수 있습니다.
- **해결책 (Strict 2PL)**: 갱신 연쇄 복귀를 막기 위해, 배타 락(X-Lock)을 트랜잭션 종료(Commit/Abort) 시까지 쥐고 있는 방식이며 현대 DB의 표준입니다.

- **📢 섹션 요약 비유**: 2PL은 **'뷔페 식당의 접시 담기'**와 같습니다. 자리에 앉아 먹기 시작(Unlock)하기 전까지는 먹고 싶은 음식을 접시에 계속 담을 수 있지만(확장), 일단 한 입이라도 먹기 시작하면(축소) 다시는 음식을 담으러 갈 수 없다는 엄격한 규칙입니다. 이 규칙 덕분에 접시 위의 음식들(트랜잭션 결과)이 중간에 섞이지 않고 온전하게 유지되는 것입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Lock Point]**: 트랜잭션의 실질적인 선후 관계가 결정되는 지점.
- **[Serializability]**: 2PL을 지키는 스케줄이 얻게 되는 성질.
- **[Cascading Rollback]**: 기본 2PL이 가진 구조적 취약점.

📢 **마무리 요약**: **2-Phase Locking**은 병행 제어의 헌법입니다. 단계적 제어라는 단순한 원칙을 통해 데이터베이스라는 거대하고 복잡한 시스템에 확고한 질서를 부여합니다.
