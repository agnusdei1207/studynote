# 451. 교착 상태(Deadlock)와 해결 기법 - 멈춰버린 트랜잭션

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 교착 상태(Deadlock)는 두 개 이상의 트랜잭션이 **서로가 점유한 자원의 잠금(Lock)이 풀리기를 기다리며 무한 대기에 빠지는 교착 현상**이다.
> 2. **가치**: 시스템이 영원히 멈추는 것을 방지하기 위해, 타임스탬프 기반의 예방 기법(Wait-Die, Wound-Wait)이나 탐지 후 희생자 선정(Victim Selection)을 통한 복구 전략이 필수적이다.
> 3. **융합**: 운영체제의 프로세스 동기화 이론이 DB 트랜잭션 레벨로 확장된 것이며, 자원 할당 그래프의 사이클(Cycle) 존재 여부를 통해 수학적으로 판별한다.

+++

### Ⅰ. 교착 상태의 예방 및 회피 기법

| 기법 명칭 | 작동 원리 | 특징 |
|:---|:---|:---|
| **Wait-Die** | 늙은 트랜잭션은 기다리고(Wait), 젊은 놈은 스스로 죽음(Die) | **비선점(Non-preemptive)** 방식 |
| **Wound-Wait** | 늙은 놈이 젊은 놈을 찔러 죽이고(Wound) 자원 탈취, 젊은 놈은 대기 | **선점(Preemptive)** 방식 |

+++

### Ⅱ. 교착 상태 탐지 및 복구 시각화 (ASCII Model)

DBMS는 주기적으로 **대기 그래프(Wait-for Graph)**를 그려 사이클을 감시합니다.

```text
[ Deadlock Detection: Wait-for Graph ]

  T1 ──(Wait Resource A)──▶ T2 ──(Wait Resource B)──▶ T3
  ▲                                                   │
  └───────────────(Wait Resource C)───────────────────┘ 💥 Cycle!

  [ Recovery Process ]
  1. Detect Cycle ✅
  2. Select Victim (T3) ──▶ 비용이 가장 적은 트랜잭션 선정
  3. Abort & Rollback T3 ──▶ 🔓 Release Locks
  4. T1, T2 Resume ✅
```

+++

### Ⅲ. 실무적 의의: 희생자 선정(Victim Selection)

- **선정 기준**: 
    - 트랜잭션의 실행 시간 (짧은 놈 우선).
    - 수정한 레코드 수 (적게 고친 놈 우선).
    - 롤백 시 발생하는 비용 (가벼운 놈 우선).
- **기아 현상 (Starvation)**: 한 트랜잭션이 계속해서 희생자로 선정되어 영원히 끝나지 않는 문제. (회수 기반 우선순위 보정으로 해결)

- **📢 섹션 요약 비유**: 교착 상태는 **'좁은 외나무다리에서 마주친 두 염소'**와 같습니다. 둘 다 비키지 않고 버티면 아무도 다리를 건널 수 없습니다. 이때 "나이가 어린 염소가 물러나자(Wait-Die)"거나, "힘센 염소가 어린 염소를 밀어내고 먼저 지나가자(Wound-Wait)"는 규칙을 정해두거나, 정 안 되면 주인이 나타나 한 마리를 들어내어(Victim Selection) 길을 터주는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Wait-for Graph]**: 데드락 탐지의 핵심 알고리즘 도구.
- **[Livelock]**: 데드락은 아니지만 서로 양보만 하느라 진행이 안 되는 상태.
- **[Lock Timeout]**: 가장 원시적이지만 확실한 데드락 탈출구.

📢 **마무리 요약**: **Deadlock Handling**은 동시성의 그림자를 걷어내는 기술입니다. 완벽한 탐지와 공정한 복구를 통해 시스템의 가용성을 무중단으로 유지해야 합니다.
