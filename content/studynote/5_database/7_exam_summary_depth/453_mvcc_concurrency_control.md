# 453. MVCC 다중 버전 읽기 일관성 - 락 없는 성능의 비밀

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MVCC(Multi-Version Concurrency Control)는 하나의 데이터에 대해 **여러 버전의 스냅샷을 유지함으로써, 읽기 작업과 쓰기 작업이 서로를 방해하지 않고 동시에 수행**되도록 하는 병행 제어 기술이다.
> 2. **가치**: "읽기는 쓰기를 막지 않고, 쓰기는 읽기를 막지 않는다"는 원칙을 실현하여, 락(Lock) 경합으로 인한 성능 저하를 방지하고 높은 동시 처리량을 제공한다.
> 3. **융합**: 트랜잭션 번호(SCN 등)와 Undo 영역(롤백 세그먼트)을 결합하여 각 트랜잭션에게 '자신이 시작된 시점'의 일관된 데이터 뷰를 제공하는 현대 RDBMS의 핵심 아키텍처다.

+++

### Ⅰ. MVCC의 작동 매커니즘

- **버전 생성**: 데이터를 수정할 때 원본을 덮어쓰지 않고, 새로운 버전의 데이터를 생성하거나 이전 데이터를 Undo 영역에 보관합니다.
- **읽기 기준**: 트랜잭션이 시작될 때 고유한 번호(타임스탬프)를 부여받고, 자신보다 낮은 번호(과거)를 가진 커밋된 데이터 중 가장 최신 것만 읽습니다.
- **쓰기 규칙**: 쓰기는 여전히 락을 사용하여 한 명만 수행하게 하지만, 읽는 사람들은 락을 기다리지 않고 과거 버전을 봅니다.

+++

### Ⅱ. MVCC 일관된 읽기 시각화 (ASCII Model)

```text
[ MVCC: Concurrent Read and Write ]

  Data [A] Version 10 (Last Committed)
       │
  T1 (Reader, TS:11) ──▶ Read [A] ──▶ Returns Ver 10
       │
  T2 (Writer, TS:12) ──▶ Update [A] ──▶ Creates Ver 12 (Not Committed yet)
       │
  T1 (Reader, TS:11) ──▶ Read [A] ──▶ Returns Ver 10! ✅ (Wait X, Consistency O)
       │
  T2 (Writer, TS:12) ──▶ COMMIT! ──▶ Ver 12 is now the latest
```

+++

### Ⅲ. MVCC의 장점과 도전 과제

- **장점**: 
    - 조회(SELECT) 성능이 비약적으로 향상됩니다.
    - 읽기-쓰기 간의 데드락이 발생하지 않습니다.
- **도전 과제**: 
    - 여러 버전을 보관해야 하므로 저장 공간(Undo 영역) 소모가 커집니다.
    - 더 이상 필요 없는 과거 버전을 정리하는 **Vacuum(가비지 컬렉션)** 프로세스가 추가적으로 필요합니다.

- **📢 섹션 요약 비유**: MVCC는 **'여러 층으로 쌓인 도화지'**와 같습니다. 화가(쓰기 트랜잭션)가 맨 윗장에 새로운 그림을 그리고 있어도, 구경꾼(읽기 트랜잭션)들은 그 아래층에 이미 완성된 이전 그림들을 아무 방해 없이 감상할 수 있는 것과 같습니다. 화가가 그림을 다 그리면(Commit) 그때부터 구경꾼들에게는 새 그림이 가장 윗장으로 보이게 됩니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Snapshot Isolation]**: MVCC가 제공하는 데이터 고립 수준의 실체.
- **[Undo Segment]**: 과거 버전 데이터가 머무는 임시 거처.
- **[Non-blocking Read]**: MVCC가 세상에 준 가장 큰 선물.

📢 **마무리 요약**: **MVCC**는 데이터베이스의 동시성을 한 단계 진화시킨 기술입니다. 성능과 정합성이라는 두 마리 토끼를 잡은 이 아키텍처가 현대 대용량 서비스의 기반이 되고 있습니다.
