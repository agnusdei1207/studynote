# 391. 릴레이션 스키마(Intension) - 데이터의 논리적 뼈대

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 릴레이션 스키마(Schema)는 데이터베이스 테이블의 이름과 그 안에 포함된 속성(Attribute)들의 이름 및 데이터 타입(Domain)을 정의한 **논리적 설계도**다.
> 2. **가치**: 데이터의 구조적 정적 정의를 담당하며, 시간이 지나도 쉽게 변하지 않는 **내포(Intension)**적 속성을 지닌다.
> 3. **융합**: 스키마 정의를 통해 데이터 무결성 제약 조건을 수립하고, 응용 프로그램과 물리적 저장소 사이의 데이터 독립성을 구현하는 기초가 된다.

+++

### Ⅰ. 릴레이션 스키마의 구성 요소

- **릴레이션 이름**: 테이블의 식별자.
- **속성 (Attribute)**: 데이터의 최소 논리적 단위 (열, Column).
- **도메인 (Domain)**: 각 속성이 가질 수 있는 원자 값들의 집합과 타입.
- **차수 (Degree)**: 속성의 전체 개수.

+++

### Ⅱ. 릴레이션 스키마 시각화 (ASCII Model)

```text
[ Relation Schema: Student ]

  Relation Name: Student
  Attributes   : (Stud_ID, Name, Major, Grade)
  Domains      : (Integer, String, String, Integer)
  
  ┌──────────┬──────────┬──────────┬──────────┐
  │ Stud_ID  │ Name     │ Major    │ Grade    │  <-- Schema (Structure)
  └──────────┴──────────┴──────────┴──────────┘
  (Degree: 4)
```

+++

### Ⅲ. 스키마의 특징: 내포 (Intension)

- **정적 성격**: 데이터베이스 설계 단계에서 확정되며, 특별한 관리적 필요(Schema Evolution)가 없는 한 변하지 않습니다.
- **제약 조건의 터전**: 기본키(PK), 외래키(FK) 등 모든 무결성 규칙이 스키마 레벨에서 정의됩니다.

- **📢 섹션 요약 비유**: 릴레이션 스키마는 **'빈 아파트의 설계도'**와 같습니다. 방이 몇 개인지(차수), 안방의 크기는 얼마인지(도메인), 거실과 부엌은 어떻게 연결되는지(구조)를 정해둔 것입니다. 입주민(데이터)이 들어오기 전부터 이미 정해져 있는 건물의 뼈대와 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Attribute]**: 스키마를 구성하는 최소 단위.
- **[Domain]**: 데이터 타입의 확장된 개념.
- **[Intension]**: 스키마를 일컫는 또 다른 학술적 용어 (내포).

📢 **마무리 요약**: **Relation Schema**는 데이터의 정체성을 정의합니다. 튼튼한 스키마 설계가 데이터 무결성과 시스템 안정성의 출발점입니다.
