# 514. 팩트 테이블과 차원 모델 - 데이터 분석의 골격

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 팩트 테이블(Fact Table)은 비즈니스 프로세스에서 발생하는 **수치적 측정값(측정 지표)**을 담는 테이블이며, 차원 모델은 이를 분류하고 설명하는 **맥락 정보(차원)**로 감싸는 분석용 설계 기법이다.
> 2. **가치**: "누가, 언제, 어디서, 무엇을(차원)" 했고 "그 결과가 얼마(팩트)"인지를 명확히 분리함으로써, 방대한 데이터를 다각도로 자유롭게 집계하고 분석할 수 있는 기반을 제공한다.
> 3. **융합**: 외래키(FK)를 통해 팩트와 차원이 결합되어 스타 스키마를 형성하며, 이는 현대 모든 데이터 웨어하우스(DW)와 비즈니스 인텔리전스(BI) 시스템의 논리적 표준이 된다.

+++

### Ⅰ. 팩트(Fact)와 차원(Dimension)의 구성

1. **팩트 테이블 (Fact Table)**:
    - **Measures**: 매출액, 판매량, 온도, 클릭 수 등 계산 가능한 숫자 데이터.
    - **Keys**: 연관된 모든 차원 테이블을 가리키는 외래키들의 집합.
    - **특징**: 데이터량이 매우 크며(수억 건 이상), 주로 '추가'만 일어납니다.
2. **차원 테이블 (Dimension Table)**:
    - **Attributes**: 이름, 주소, 카테고리, 날짜 속성 등 텍스트 정보.
    - **특징**: 팩트 데이터를 필터링하거나 그룹화하는 기준이 되며, 상대적으로 데이터량이 작습니다.

+++

### Ⅱ. 다차원 모델 시각화 (ASCII Model)

팩트를 중심으로 차원이 별 모양으로 둘러싼 구조입니다.

```text
[ Star Schema Dimensional Model ]

      (언제?) [ Dim_Time ] ─────┐
                               │
      (어디서?) [ Dim_Store ] ──┼──▶ [ Fact_Sales ] (Measure: Sales_Amt) ✅
                               │
      (무엇을?) [ Dim_Product ] ┘
```

+++

### Ⅲ. 설계 시 고려사항 (그레인 설정)

- **Grain (단위)**: 팩트 테이블의 한 행이 무엇을 의미하는지 정의하는 것이 가장 중요합니다. (예: 영수증 행 단위 vs 매장별 일일 합계 단위)
- **정규화 여부**: 조인 성능을 위해 차원 테이블은 가급적 비정규화(Star Schema)하여 관리하거나, 필요 시 계층화(Snowflake)합니다.

- **📢 섹션 요약 비유**: 팩트와 차원은 **'가계부 정리'**와 같습니다. 내가 쓴 '돈의 액수(Fact)'를 가운데 적고, 그 옆에 '언제', '어디서', '무엇을 위해' 썼는지(Dimension)를 기록하는 것과 같습니다. 이렇게 정리해 두면 나중에 "지난달에 커피값으로 총 얼마 썼지?"라는 질문에 바로 대답할 수 있는 것과 같은 원리입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Granularity]**: 데이터의 상세화 정도.
- **[Conformed Dimension]**: 여러 팩트 테이블이 공유하는 공통 차원.
- **[Star Schema]**: 팩트와 차원을 잇는 가장 단순하고 빠른 물리 구조.

📢 **마무리 요약**: **Fact & Dimension**은 분석의 알파이자 오메가입니다. 숫자의 의미를 맥락으로 풀어내는 이 정교한 설계가 데이터로부터 가치 있는 전략을 뽑아내는 원천이 됩니다.
