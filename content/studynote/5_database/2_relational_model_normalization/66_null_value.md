# 66. NULL 값 - 아직 알려지지 않거나 해당 없는 값

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 관계형 데이터베이스에서 데이터의 값이 존재하지 않거나, 아직 정해지지 않았거나, 혹은 해당 속성에 값이 적용될 수 없는 상태를 나타내는 **특수 마커(Marker)**다.
> 2. **논리적 특성**: 숫자 '0'이나 공백 문자('')와는 완전히 다른 개념이며, 비교 연산 시 참(True)도 거짓(False)도 아닌 **알 수 없음(Unknown)**을 반환하는 **3치 논리(Three-Valued Logic, 3VL)**의 핵심 원천이 된다.
> 3. **가치**: 현실 세계의 불완전한 정보를 데이터베이스에 수용할 수 있게 하지만, 인덱스 활용 저하나 연산 오류를 유발할 수 있어 **`NOT NULL` 제약 조건** 등을 통한 정교한 설계와 제어가 필수적이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**NULL 값**은 속성에 저장된 실제 값이 없음을 의미하는 정보 부재의 표현입니다. 1970년대 에드거 커드가 관계형 모델을 보완하기 위해 도입한 개념으로, 크게 두 가지 의미를 내포합니다.
- **알 수 없는 값 (Unknown Value)**: 값은 존재하지만 아직 시스템이 모르는 상태. (예: 신입사원의 핸드폰 번호 미등록)
- **해당 없는 값 (Inapplicable Value)**: 속성 자체가 해당 개체에 적용되지 않는 상태. (예: 미혼자의 '배우자 이름' 칸)

### 2. 💡 비유: 택배 송장의 비어있는 칸
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        NULL 값 개념 비유                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 숫자 0: 배송비가 '0원'인 무료 배송. (값이 0으로 존재함)                      │
│  2. 공백(''): 받는 사람 이름 칸에 스페이스바만 침. (내용물은 없으나 칸은 채움)      │
│  3. NULL: 송장 칸 자체가 깨끗하게 비어있음.                                     │
│                                                                             │
│  - 비어있는 칸을 보고 "배송비가 0원인가요?"라고 물으면 "모름(Unknown)"이 정답!     │
│  - "받는 사람이 누군가요?"라고 물어도 "아직 적지 않음(Unknown)"이 정답!           │
│                                                                             │
│  → NULL은 값이 아니라, **'값이 아직 여기에 오지 않았다'**는 상태 표시등입니다.      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **현실의 불확실성**: 모든 데이터가 입력 시점에 완벽할 수 없으므로, 값이 없는 상태를 정의할 표준적인 방법 필요.
- **구조적 유연성**: 테이블 구조상 특정 개체에게는 필요 없는 속성(예: 법인 사업자의 '성별')을 비워두기 위함.

- **📢 섹션 요약 비유**: NULL은 데이터 상자 안이 비어있음을 나타내는 '빈 상자 태그'와 같습니다. 상자 자체가 없는 것과는 다릅니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. NULL의 3치 논리 (Three-Valued Logic, 3VL) 연산표

NULL이 포함된 논리 연산의 결과는 매우 독특합니다.

| 연산 ($A$ AND $B$) | True | False | **Unknown (NULL)** |
|:---:|:---:|:---:|:---:|
| **True** | True | False | **Unknown** |
| **False** | False | False | False |
| **Unknown** | **Unknown** | False | **Unknown** |

- **핵심**: 하나만 False 면 전체가 False 지만, 그 외에 NULL이 끼어들면 결과는 항상 **Unknown**이 됩니다.

### 2. SQL 내 NULL 처리 메커니즘 (ASCII)

NULL은 직접 비교가 불가능하며 특수 명령어를 사용해야 함을 시각화한다.

```text
    [ Data Sample ]
    ID: 1, Sal: 1000
    ID: 2, Sal: NULL
    
    Q1: WHERE Sal = 1000  --> { ID: 1 } (Match)
    Q2: WHERE Sal = NULL  --> { Empty } (Failure! 직접 비교 불가)
    Q3: WHERE Sal IS NULL --> { ID: 2 } (Success! 마커 확인)
    
    [ NULL 연산 법칙 ]
    - NULL + 100 = NULL (모르는 값에 100을 더해도 여전히 모름)
    - SUM(Sal) = 1000 (집계 함수는 기본적으로 NULL을 무시하고 계산)
```

### 3. 심층 동작 원리: 물리적 저장 방식
- **Bitmap**: 블록 헤더에 NULL 비트맵을 두어 어떤 컬럼이 NULL인지 표시함.
- **Length 0**: 컬럼의 길이를 0으로 저장하여 공간을 절약함.
- **Fixed-length**: 고정 길이 컬럼(INT 등)에서는 특수 비트 패턴이나 블록 내의 특정 오프셋 정보를 통해 NULL임을 인지함.

### 4. 핵심 기술: NULL 전이 (Null Propagation)
- 산술 연산이나 문자열 결합 시, 피연산자 중 하나라도 NULL이면 결과 전체가 NULL이 되는 성질.
- 예: `10 + NULL = NULL`, `'Hello ' || NULL = NULL` (단, DB 제품마다 문자열 처리는 다를 수 있음)

- **📢 섹션 요약 비유**: 3치 논리는 'OX 퀴즈에서 모르는 문제가 나오면 세모(△)를 치는 것'과 같습니다. 세모가 섞이면 채점 결과도 세모가 됩니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. NULL vs 0 vs 공백('')

| 비교 항목 | NULL | 0 (Zero) | 공백 ('') |
|:---:|:---|:---|:---|
| **성격** | **상태 (Marker)** | **값 (Numeric Value)** | **값 (String Value)** |
| **길이/크기** | 정의되지 않음 (또는 0) | 데이터 타입 크기 점유 | 문자열 길이에 따라 점유 |
| **비교 방식** | **IS NULL** | `=`, `<>` | `=`, `<>` |
| **의미** | 알 수 없음 / 해당 없음 | 없음 / 영(Zero) | 빈 문자열 |

### 2. 과목 융합 관점: 이산수학 및 프로그래밍
- **이산수학 융합**: NULL은 부분 함수(Partial Function)에서 정의되지 않은 입력값에 대한 출력 결과로 해석될 수 있다.
- **프로그래밍 융합**: 자바의 `Optional`, 코틀린의 `Nullable Type(? )`, Swift의 `Optional` 등은 DB의 NULL 개념을 언어 차원에서 안전하게 다루기 위해 진화한 형태다.

- **📢 섹션 요약 비유**: NULL이 '기억 상실'이라면, 0은 '빈 통장'이고, 공백은 '빈 종이'입니다. 상태와 값의 차이를 이해해야 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 집계 함수의 NULL 함정 피하기
- **상황**: 사원들의 평균 연봉을 구하려 함. 신입사원 1명의 연봉이 NULL임.
- **판단**: `AVG(salary)`는 NULL인 사원을 아예 분모에서 제외함. (전체 사원 수로 나누지 않음)
- **결단**: 비즈니스 요건에 따라 **`NVL(salary, 0)` 또는 `COALESCE` 함수를 적용**. NULL을 0으로 치환하여 전체 인원 대비 평균을 구할 것인지, 아니면 정말 무시할 것인지 명확히 설계.

### 2. 안티패턴: 인덱스 컬럼의 NULL 허용
- **증상**: 검색 조건으로 자주 쓰이는 '상태값' 컬럼에 NULL을 허용함.
- **결과**: 일반적인 B-Tree 인덱스는 NULL 값을 저장하지 않으므로, `IS NULL` 조건 검색 시 인덱스를 타지 못하고 Full Scan 발생.
- **조치**: 가능한 **`DEFAULT` 값을 설정**하여 NULL 발생을 억제하거나, 반드시 NULL이 필요하다면 Partial Index나 특수 처리를 고려.

### 📢 기술사적 결언
> "NULL은 데이터베이스 설계자의 **'세심함'**을 테스트하는 시금석이다. NULL을 무분별하게 허용하는 것은 설계를 포기하는 것과 다름없다. 기술사는 **`NOT NULL` 제약 조건**을 기본으로 하되, 비즈니스상 데이터 부재가 허용되는 곳에만 엄격하게 NULL을 허용해야 한다. 또한, 3치 논리의 특성으로 인해 쿼리 결과가 예상과 다르게 나올 수 있음을 인지하고, **Null-safe 한 SQL**을 작성하는 습관을 가져야 한다."

- **📢 섹션 요약 비유**: 인덱스에서 NULL을 찾는 것은 '목차에 없는 페이지를 찾으려고 책 전체를 넘기는 것'과 같이 고된 일입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (적절한 NULL 관리 시)

| 지표 | 관리 미흡 시 | 최적 관리 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **쿼리 정확도** | 논리 오류 발생 가능 | **100% 신뢰 가능** | 데이터 정합성 오류 80% 감소 |
| **검색 속도** | 인덱스 활용 저하 | **Index Only Scan 가능** | 성능 5~10배 향상 |
| **코드 복잡도** | Null 체크 코드 남발 | **간결한 비즈니스 로직** | 개발 생산성 및 유지보수성 향상 |

### 2. 미래 전망
- **Non-nullable by Default**: 최신 데이터 언어나 일부 차세대 DB에서는 모든 속성을 기본적으로 `NOT NULL`로 설정하고, 명시적으로만 허용하는 방향으로 진화 중.
- **AI Null Imputation**: 비어있는 NULL 값을 주변 데이터의 문맥을 파악하여 AI가 가장 확률 높은 값으로 자동 채워주는 **지능형 데이터 보정** 기술 발전.

- **📢 섹션 요약 비유**: 미래에는 NULL이 단순한 '빈칸'을 넘어, 왜 비어있는지에 대한 '사연'까지 품고 있는 지능적 데이터로 발전할 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **NULL**은 장난감 상자에 이름표를 붙였는데, 아직 **안에 무엇을 넣을지 결정을 안 해서** 비어있는 상태예요.
2. 숫자 "0"은 사과가 한 개도 없다는 뜻이지만, "NULL"은 **사과가 있는지 없는지도 아직 모른다는 뜻**이죠.
3. 나중에 무엇이 들어올지 모르니까, NULL이 보이면 **"조금 더 기다려보자!"**라고 생각하면 된답니다!
