# 64. 도메인 (Domain) - 속성의 원자적 값의 집합

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 관계형 데이터 모델의 릴레이션에서 특정 속성이 가질 수 있는 **동일한 타입의 원자적(Atomic) 값들의 전체 집합**을 의미한다.
> 2. **무결성 수호**: 속성에 입력되는 값의 범위와 형식을 규정함으로써, 잘못된 데이터의 유입을 원천 차단하는 가장 기초적인 **도메인 무결성(Domain Integrity)** 제약 조건의 토대가 된다.
> 3. **가치**: 데이터의 의미적 정합성을 보장하며, 서로 다른 속성이더라도 동일한 도메인을 공유할 경우에만 비교나 조인(Join) 연산이 논리적으로 타당함을 입증하는 기준이 된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**도메인 (Domain)**은 릴레이션의 한 속성에 나타날 수 있는 모든 가능한 값들의 집합입니다. 예를 들어, '성별'이라는 속성의 도메인은 {남, 여}가 될 수 있고, '점수'의 도메인은 {0 이상 100 이하의 정수}가 될 수 있습니다.

### 2. 💡 비유: 가전제품의 전압 규격
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        도메인(Domain) 개념 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 속성 (콘센트 구멍): 벽에 붙어있는 구멍들. (데이터가 들어올 자리)              │
│  2. 도메인 (전압 규격): "이 구멍에는 오직 '220V'만 꽂아야 한다"는 약속.           │
│                                                                             │
│  - 만약 110V 제품(잘못된 데이터)을 억지로 꽂으려 하면, 규격(도메인)에 맞지 않아    │
│    작동하지 않거나 고장(무결성 위반)이 납니다.                                   │
│                                                                             │
│  → 도메인은 데이터가 들어오기 전, 미리 정해둔 **'값의 합격 기준'**과 같습니다.     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 정합성 확보**: '나이' 칸에 '-5'가 들어오거나 '이름' 칸에 숫자가 들어오는 등의 비논리적 상황 방지 필요.
- **연산의 유효성 검증**: '사원번호'와 '연봉'은 둘 다 숫자 타입일 수 있지만, 의미(도메인)가 다르므로 이 둘을 더하거나 비교하는 연산은 논리적으로 오류임을 판별하기 위함.

- **📢 섹션 요약 비유**: 도메인은 데이터들이 입고 있는 '유니폼'과 같아서, 어떤 팀(속성)에 속해야 하는지 명확히 구분해줍니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 도메인의 구성 요소 및 제약 형태

| 구성 요소 | 상세 내용 | 비유 |
|:---:|:---|:---|
| **데이터 타입** | 정수(INT), 실수(FLOAT), 문자열(VARCHAR), 날짜(DATE) 등. | 재질 (철, 나무) |
| **값의 범위** | 0 ~ 100, 특정 문자열 리스트({A, B, C}). | 크기 제한 |
| **원자성 (Atomicity)** | 더 이상 쪼개질 수 없는 단일 값의 성질. | 쪼개지지 않는 알갱이 |
| **기본값 (Default)** | 값이 입력되지 않았을 때 자동으로 채워지는 값. | 예비 부품 |

### 2. 도메인과 속성의 매핑 도해 (ASCII)

하나의 도메인이 여러 속성에서 재사용될 수 있음을 시각화한다.

```text
    [ Domain: Emp_ID_Type ] : { 1000 ~ 9999 사이의 정수 }
               │
    ┌──────────┴──────────┐
    │                     │
    ▼                     ▼
    [ Attr: 사원번호 ]     [ Attr: 추천인번호 ]
    (Employee Table)      (Employee Table)
    
    * 서로 다른 속성이지만 같은 도메인을 사용하므로, 
      "사원번호 = 추천인번호" 라는 비교 연산이 논리적으로 성립함.
```

### 3. 심층 동작 원리: 도메인 무결성 (Domain Integrity)
- **메커니즘**: DBMS는 새로운 튜플이 삽입되거나 수정될 때, 해당 값이 속성에 정의된 도메인 규칙을 만족하는지 실시간 검사함.
- **SQL 구현**: `CREATE DOMAIN` (표준 SQL) 또는 `CHECK` 제약 조건, `NOT NULL`, `Data Type` 정의 등을 통해 구현.

### 4. 핵심 기술: 사용자 정의 도메인 (User-defined Domain)
- 단순한 `INT` 타입을 넘어, `Score`라는 이름의 도메인을 정의하고 "0~100 사이"라는 규칙을 부여.
- 여러 테이블에서 '점수'와 관련된 모든 속성에 이 `Score` 도메인을 적용하면 관리가 일원화됨.

- **📢 섹션 요약 비유**: 도메인 무결성은 '입구에서 신분증과 자격증을 대조하여 통과시키는 보안 요원'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 도메인(Domain) vs 데이터 타입(Data Type)

| 비교 항목 | 데이터 타입 (Data Type) | 도메인 (Domain) |
|:---:|:---|:---|
| **범위** | 물리적/기술적 형식 (시스템 관점) | **논리적/의미적 범위 (비즈니스 관점)** |
| **예시** | `INTEGER` | `학생 성적 (0~100 정수)` |
| **포괄 관계** | 도메인이 데이터 타입을 포함함 | 데이터 타입은 도메인의 일부 요소임 |
| **목적** | 메모리 할당 및 연산 방식 결정 | **데이터의 올바름(정합성) 보장** |

### 2. 과목 융합 관점: 프로그래밍 언어 및 컴파일러
- **프로그래밍 언어 융합**: 도메인은 강한 타입 시스템(Strong Typing)을 가진 언어의 **사용자 정의 자료형(User-defined Type)**과 유사하다.
- **컴파일러 융합**: 도메인 체크 과정은 컴파일러의 **타입 검사(Type Checking)** 단계와 논리적으로 동일하며, 런타임에 데이터 오류를 사전에 차단하는 역할을 한다.

- **📢 섹션 요약 비유**: 데이터 타입이 '그릇의 재질'이라면, 도메인은 '그릇에 담길 음식의 종류'입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 일관성 있는 도메인 관리의 중요성
- **상황**: '전화번호' 속성이 회원 테이블에는 `VARCHAR(20)`으로, 배송 테이블에는 `VARCHAR(15)`로 되어 있음.
- **판단**: 두 속성의 도메인이 일치하지 않아 조인 시 데이터가 잘리거나 오류 발생 가능성 높음.
- **결단**: **전사 데이터 표준(Data Standard)**을 수립하고, 모든 '전화번호' 관련 속성은 동일한 도메인(길이, 형식)을 따르도록 스키마 리팩토링 수행.

### 2. 안티패턴: 범용 타입(`TEXT`, `BLOB`)의 남발
- **증상**: 도메인을 정의하기 귀찮아서 모든 문자열을 `TEXT` 타입으로 선언.
- **결과**: 나이 칸에 "스무살"이라는 문자가 들어와도 막지 못함. 옵티마이저가 데이터 크기를 가늠하지 못해 실행 계획이 꼬임.
- **조치**: 데이터의 성격에 맞는 **가장 타이트한 도메인**을 설정하여 시스템의 자정 능력을 강화.

### 📢 기술사적 결언
> "도메인은 데이터베이스의 **'최소 안전장치'**다. 코드 레벨에서 체크하는 정합성은 우회될 수 있지만, DB 엔진 레벨의 도메인 제약은 타협이 불가능하기 때문이다. 기술사는 데이터 모델링 초기 단계부터 **공통 도메인 추출**에 힘써야 하며, 이는 곧 시스템 전체의 **데이터 거버넌스**와 **메타데이터 관리**의 품질로 이어진다는 사실을 명심해야 한다."

- **📢 섹션 요약 비유**: 도메인을 느슨하게 잡는 것은 '정수기 필터를 빼고 물을 마시는 것'과 같아, 결국 시스템 전체에 오염된 데이터가 퍼지게 됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (철저한 도메인 설계 시)

| 지표 | 설계 미흡 시 | 최적 설계 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 클렌징 비용** | 높음 (사후 교정) | **낮음 (원천 차단)** | 운영 비용 50% 이상 절감 |
| **시스템 신뢰도** | 낮음 (데이터 불일치) | **최상 (정합성 보장)** | 비즈니스 판단 정확도 향상 |
| **개발 생산성** | 예외 처리 코드 남발 | **깔끔한 비즈니스 로직** | 코드 가독성 및 가용성 증대 |

### 2. 미래 전망
- **Semantic Domain**: AI가 데이터의 의미를 파악하여, 타입이 다르더라도 논리적으로 같은 도메인임을 인지하고 자동으로 변환/매칭해주는 기술 발전.
- **Ontology-based DB**: 단순 집합을 넘어 데이터 간의 고차원적 관계(상위/하위 개념)를 도메인 레벨에서 관리하는 지식 기반 저장소 확산.

- **📢 섹션 요약 비유**: 미래의 도메인은 단순한 벽을 넘어, 데이터를 스스로 정화하고 올바른 길로 안내하는 '스마트 게이트'가 될 것입니다.
