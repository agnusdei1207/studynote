# 77. 사용자 정의 무결성 (User-defined Integrity) - 비즈니스 규칙 준수

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 개체, 참조, 도메인 무결성과 같은 표준 제약 조건만으로는 표현할 수 없는 기업 고유의 복잡한 **비즈니스 로직과 업무 규칙**을 데이터베이스 계층에서 강제하는 기법이다.
> 2. **구현 수단**: 주로 **트리거(Trigger)**, **저장 프로시저(Stored Procedure)**, 그리고 복합적인 **CHECK 제약 조건** 등을 활용하여 데이터의 상태 변화 시점에 정교한 검증 로직을 실행한다.
> 3. **가치**: 애플리케이션의 파편화된 검증 로직을 데이터 중심부로 결착시켜, 어떤 경로(API, 직접 SQL 등)로 접근하더라도 동일한 비즈니스 정합성을 보장하는 **최종 수비수** 역할을 수행한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**사용자 정의 무결성 (User-defined Integrity)**은 특정 데이터베이스 환경에서 사용자가 정의한 모든 제약 조건을 의미합니다. 표준적인 무결성(PK, FK 등)이 시스템적인 '형식'을 맞춘다면, 사용자 정의 무결성은 실질적인 '업무 내용'의 올바름을 검사합니다.

### 2. 💡 비유: 은행 대출 심사의 특수 규정
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        사용자 정의 무결성 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 기본 무결성 (공통 규칙): "신분증이 있어야 한다(PK)", "사인을 해야 한다".      │
│  2. 사용자 정의 무결성 (우리 은행만의 특수 규칙):                              │
│     - "대출 금액은 연봉의 2배를 넘을 수 없다."                                  │
│     - "최근 3개월 내에 연체 기록이 있으면 대출 불가."                            │
│                                                                             │
│  → 법에는 없지만, **'우리 조직의 정해진 약속'**을 어기면 거래를 승인하지 않는 것과 │
│     같습니다. 데이터가 비즈니스적으로 **'말이 되는지'** 끝까지 따져보는 것이죠!    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **현실의 복잡성**: 단순한 타입 체크(INT)나 범위 체크(0~100)만으로는 해결되지 않는 동적인 업무 규칙 속출. (예: 재고보다 많은 주문은 반려)
- **중앙 집중식 로직 관리**: 여러 애플리케이션(Web, Mobile, Batch)이 동일한 DB를 공유할 때, 검증 로직을 DB에 한 번만 작성하여 일관성 유지 필요.

- **📢 섹션 요약 비유**: 사용자 정의 무결성은 규격화된 기성복(표준 제약)이 아닌, 내 몸(비즈니스)에 딱 맞게 제작한 '맞춤복'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 사용자 정의 무결성의 주요 구현 방식

| 기술 수단 | 상세 메커니즘 | 적용 사례 |
|:---:|:---|:---|
| **Complex CHECK** | 다중 컬럼 간의 수식 비교. | `CHECK (start_date < end_date)` |
| **Trigger** | INSERT/UPDATE 발생 시 자동 실행되는 코드. | 재고 차감 시 마이너스 확인 후 자동 주문 |
| **Assertion** | 여러 테이블에 걸친 복합적인 제약 정의. | "전체 사원 급여 합계가 예산을 넘지 말 것" |
| **UDF (Function)** | 사용자 정의 함수를 통한 논리 검증. | 복잡한 세금 계산식 적용 여부 확인 |

### 2. 사용자 정의 무결성 처리 흐름도 (ASCII)

업무 규칙이 데이터 반영 전 어떻게 개입하는지 시각화한다.

```text
    [ Client Request: 주문 생성 ]
           │
    ┌──────▼────────────────── DBMS Engine ───────────────────────────────┐
    │                                                                     │
    │  1. [ Basic Check ] : 도메인, PK, FK 검증 (통과)                     │
    │                                                                     │
    │  2. [ User-defined Check: Trigger Invoked ]                         │
    │     ┌───────────────────────────────────────────────────────────┐   │
    │     │ IF (주문수량 > 현재재고) THEN                             │   │
    │     │    RAISE_ERROR('재고 부족');                              │   │
    │     │ ELSE                                                      │   │
    │     │    UPDATE Inventory SET Stock = Stock - 주문수량;         │   │
    │     │ END IF;                                                   │   │
    │     └──────────────┬────────────────────────────────────────────┘   │
    │                    │ (성공 시에만 아래 진행)                         │
    └────────────────────┼────────────────────────────────────────────────┘
                         ▼
    ┌──────────────── Persistent Storage ────────────────────────────────┐
    │  [ Data Updated Successfully ]                                     │
    └────────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: 상태 전이 검증 (Transition Constraint)
- **개념**: 단순 현재 값이 아니라, "값이 어떻게 변하는가"를 감시함.
- **예시**: "직급이 '부장'인 사원은 '차장'으로 강등될 수 없다"는 규칙.
- **구현**: 트리거 내부에서 `:OLD.position`과 `:NEW.position` 값을 비교하여 논리적 타당성 검사.

### 4. 핵심 기술: 데이터 무결성 서비스 계층화
- **DB 레벨**: 최후의 강력한 보루. 성능 저하를 고려하여 핵심 규칙만 선별 적용.
- **API 레벨**: 빠른 사용자 피드백을 위해 서버 측 로직으로 1차 검증 수행.

- **📢 섹션 요약 비유**: 상태 전이 검증은 '비디오 판독기(VAR)'와 같아서, 동작의 전후 맥락을 파악해 반칙(오류)을 잡아냅니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 표준 제약 조건(Standard) vs 사용자 정의 제약(User-defined)

| 비교 항목 | 표준 제약 조건 (PK, FK, NULL) | 사용자 정의 제약 (Trigger 등) |
|:---:|:---|:---|
| **정의 시점** | 테이블 생성 시 (DDL) | 필요 시 언제든 추가 가능 |
| **표현력** | 정적이고 단순함 | **매우 동적이고 복합적임** |
| **성능 오버헤드** | 거의 없음 (최적화됨) | **코드 복잡도에 따라 발생 가능** |
| **관리 주체** | DBMS 엔진 아키텍처 | 데이터베이스 관리자 / 개발자 |

### 2. 과목 융합 관점: 소프트웨어 공학 및 보안
- **소프트웨어 공학(SE) 융합**: 사용자 정의 무결성은 **비즈니스 인텔리전스(BI)**를 데이터 계층에 내재화하는 과정이며, **객체-관계 매핑(ORM)** 프레임워크와의 로직 중복 문제를 해결해야 할 과제로 안고 있다.
- **보안(Security) 융합**: 논리적인 허점을 이용한 **우회 공격(Logical Bypass)**을 막는 가장 강력한 수단이다. 예를 들어, 가격 컬럼에 마이너스(-)를 넣어 환불을 유도하는 공격을 사용자 정의 무결성으로 차단할 수 있다.

- **📢 섹션 요약 비유**: 표준 제약이 '법률'이라면, 사용자 정의 무결성은 조직의 '사규'입니다. 법보다 더 깐깐할 때가 많습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 은행 이체 시의 잔액 검증 트리거
- **상황**: A 계좌에서 B 계좌로 돈을 보낼 때, A의 잔액이 마이너스가 되면 안 됨.
- **판단**: 애플리케이션에서 `SELECT`로 잔액을 확인하고 `UPDATE`를 하면 그 짧은 찰나에 다른 거래가 끼어들어 잔액이 꼬일 수 있음(Race Condition).
- **결단**: **DB 트리거를 통한 사용자 정의 무결성 강제**. 업데이트 직전 엔진 레벨에서 잔액을 확인하고, 규칙 위반 시 트랜잭션을 통째로 취소하여 원자성과 정합성을 완벽히 수호함.

### 2. 안티패턴: 너무 무거운 트리거 로직
- **증상**: 트리거 하나에서 로그 기록, 외부 메일 발송, 다른 테이블 10개 수정을 동시에 수행.
- **결과**: 데이터 한 줄 넣는 데 수 초가 걸리고, 트랜잭션 락이 길어져 시스템 전체 성능 마비.
- **조치**: 사용자 정의 무결성은 **데이터의 논리적 정합성 검증**에 집중하고, 무거운 부수 작업은 메시지 큐(Message Queue) 등을 이용해 비동기로 분리.

### 📢 기술사적 결언
> "사용자 정의 무결성은 **'살아있는 비즈니스'를 데이터로 가두는 틀**이다. 업무 규칙이 바뀌면 무결성 규칙도 변해야 한다. 기술사는 모든 로직을 DB에 넣으려는 욕심을 버리고, **엔진의 처리 성능**과 **비즈니스의 정합성** 사이에서 최적의 '로직 배치 전략'을 수립해야 한다. DB는 데이터의 최종 수비수로서 결코 타협할 수 없는 '핵심 정합성'만을 사용자 정의 무결성으로 담아내야 한다."

- **📢 섹션 요약 비유**: 트리거 남발은 '현관문을 열 때마다 온 집안 청소를 다 해야 하는 규칙'과 같아, 출입(데이터 처리)을 너무 힘들게 만듭니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (사용자 정의 무결성 최적화 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **논리적 데이터 오류** | 상존 (앱 버그 시) | **0% (엔진 강제)** | 데이터 품질 신뢰도 극대화 |
| **시스템 통합 비용** | 개별 검증 로직 구현 | **중앙 집중식 검증 활용** | 개발 및 유지보수 비용 40% 절감 |
| **보안 취약점** | 비즈니스 로직 우회 가능 | **원천 차단** | 금융/결제 사고 예방 |

### 2. 미래 전망
- **AI-Managed Business Rules**: 자연어로 된 업무 규정을 AI가 파악하여 실시간으로 사용자 정의 무결성 규칙(SQL Check/Trigger)으로 변환하고 적용하는 기술 등장.
- **Smart Contract Integration**: 블록체인의 스마트 컨트랙트와 DB의 무결성 엔진을 연동하여, 기관 간 거래의 업무 규칙을 투명하게 강제하는 아키텍처 확산.

- **📢 섹션 요약 비유**: 미래의 무결성 시스템은 단순한 감시를 넘어, 비즈니스의 흐름을 스스로 학습하고 최적의 경로로 유도하는 '지능형 비즈니스 코치'가 될 것입니다.
