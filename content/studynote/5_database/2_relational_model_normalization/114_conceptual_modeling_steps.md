# 114. 개념적 모델링 단계 - 주제 영역과 핵심 엔티티 도출

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 전체 데이터베이스 설계의 첫 단추로, 비즈니스의 핵심 가치를 생성하는 **주제 영역(Subject Area)**을 정의하고 그 안에서 관리해야 할 가장 기초적인 **핵심 엔티티(Core Entity)**를 식별하는 추상화 단계다.
> 2. **전략적 접근**: 복잡한 세부 사항은 잠시 접어두고, 비즈니스의 '주인공(핵심 개체)'과 '줄거리(관계)' 위주로 스케치함으로써 사용자 요구사항을 데이터 관점으로 빠르게 정형화한다.
> 3. **가치**: 특정 기술이나 DBMS에 독립적인 논리 구조를 확립하여 의사소통의 기준점을 제공하며, 향후 논리/물리 설계의 흔들리지 않는 **데이터 골격**을 형성한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**개념적 모델링 단계**는 사용자의 요구사항을 분석하여 데이터베이스의 전체적인 윤곽을 그리는 과정입니다. 기술적인 용어보다는 비즈니스 용어를 사용하며, **ERD(Entity-Relationship Diagram)**를 통해 시각화하는 것이 특징입니다.

### 2. 💡 비유: 지도의 구획 정리와 주요 랜드마크 표시
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        개념적 모델링 비유                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 새로운 도시(정보 시스템)를 건설하려고 함.                                 │
│                                                                             │
│  1. 주제 영역 설정 (구획 정리): "여기는 주거 단지, 여기는 상업 지구입니다."        │
│  2. 핵심 엔티티 도출 (랜드마크): "주거 단지엔 '아파트'와 '공원'이 핵심입니다."     │
│  3. 관계 설정 (도로망): "아파트에서 공원까지는 '산책로'로 연결됩니다."             │
│                                                                             │
│  → 하수구 위치(물리 설계)를 고민하기 전, **'어떤 동네에 무엇이 있는지'**를          │
│     크게 그려보는 것이 바로 개념적 모델링입니다!                                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **복잡성 제어**: 처음부터 모든 테이블과 컬럼을 정의하려 하면 전체 숲(비즈니스 흐름)을 놓치기 쉬움.
- **현업 소통의 도구**: IT 비전공자인 비즈니스 담당자도 자신의 업무가 데이터로 어떻게 표현되는지 검증할 수 있어야 함.

- **📢 섹션 요약 비유**: 개념적 모델링은 '어떤 요리를 만들지 메뉴를 정하고 주재료를 준비하는 단계'와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 개념적 모델링의 4대 핵심 활동

| 활동 단계 | 상세 내용 | 비유 |
|:---:|:---|:---|
| **주제 영역 정의** | 비즈니스를 관리 가능한 단위로 그룹화. (예: 인사, 영업, 재고) | 마을 나누기 |
| **핵심 엔티티 식별** | 각 영역의 주인공이 되는 개체 추출. (예: 사원, 상품, 창고) | 주인공 뽑기 |
| **핵심 관계 설정** | 엔티티 간의 굵직한 연결 고리 정의. (예: 사원은 부서에 소속됨) | 인연 맺기 |
| **ERD 작성** | 위 내용들을 시각적 다이어그램으로 통합. | 지도 그리기 |

### 2. 개념적 모델링의 계층 구조 (ASCII)

요구사항에서 주제 영역, 엔티티로 구체화되는 과정을 시각화한다.

```text
    [ 비즈니스 요구사항 : 쇼핑몰 운영 ]
           │
    ┌──────▼────────────────────────────────┐
    │ 1. [ 주제 영역: 영업 ]  [ 주제 영역: 물류 ] │ (Subject Area)
    └──────┬──────────────────────┬─────────┘
           │                      │
    ┌──────▼──────┐        ┌──────▼──────┐
    │  고 객 (E)   │        │  상 품 (E)   │ (Core Entities)
    └──────┬──────┘        └──────┬──────┘
           │       ┌─────────┐    │
           └───────┤  주 문  ├────┘         (Relationship)
                   └─────────┘
```

### 3. 심층 동작 원리: 주제 영역(Subject Area)의 효용
- **관리성**: 수백 개의 엔티티를 한 번에 다루는 대신, 10~20개 내외의 주제 영역으로 나누어 집중도를 높임.
- **상관관계**: 주제 영역 간의 인터페이스를 정의함으로써 데이터 중복과 간섭을 사전에 방지함.

### 4. 핵심 기술: 엔티티 후보 식별 (Entity Discovery)
- **명사 추출법**: 요구사항 명세서의 명사들을 나열하고, 속성이 아닌 '독립적 실체'인 것들을 엔티티로 격상.
- **검증 규칙**: 유일한 식별자가 있는지, 2개 이상의 인스턴스가 존재하는지, 업무에 꼭 필요한 정보인지 확인.

- **📢 섹션 요약 비유**: 주제 영역은 '도서관의 장르별 서가'와 같고, 엔티티는 '그 서가에 꽂힐 책의 종류'입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 개념적 모델링 vs 논리적 모델링

| 비교 항목 | 개념적 모델링 (Conceptual) | 논리적 모델링 (Logical) |
|:---:|:---|:---|
| **주요 독자** | 현업 담당자, 경영진, 설계자 | 개발자, 데이터베이스 관리자 |
| **상세도** | 핵심 엔티티 및 관계 중심 | **모든 속성, 정규화, 타입** 정의 |
| **데이터 모델** | ER 모델 (기술 독립적) | **관계형 모델 (기술 종속적)** |
| **목표** | 비즈니스 **범위(Scope)** 확정 | 시스템 **구조(Structure)** 확정 |

### 2. 과목 융합 관점: 소프트웨어 공학 및 아키텍처
- **소프트웨어 공학(SE) 융합**: 개념적 모델링은 **유스케이스(Use Case)** 분석을 데이터 관점에서 재해석한 것이며, 도메인 중심 설계(DDD)의 **전략적 설계(Bounded Context)** 단계와 긴밀히 연결된다.
- **데이터 아키텍처(DA) 융합**: 기업 전체의 데이터 표준을 정의하는 **전사 데이터 모델(Enterprise Data Model)** 수립 시 개념적 모델링의 주제 영역 체계가 핵심 가이드가 된다.

- **📢 섹션 요약 비유**: 개념 모델이 '아이디어 스케치'라면, 논리 모델은 '치수가 기입된 설계도'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 복잡한 금융 비즈니스의 개념 모델링
- **상황**: 은행의 여신, 수신, 외환, 카드 업무를 통합하는 DB 구축.
- **판단**: 한꺼번에 설계하면 반드시 실패함.
- **결단**: **주제 영역별 점진적 모델링 수행**. '공통 고객'과 '상품'이라는 핵심 엔티티를 먼저 정의한 뒤, 여신/수신 등의 주제 영역별로 상세화를 진행하여 데이터의 고립(Silo) 현상을 방지함.

### 2. 안티패턴: 개념 단계를 건너뛰고 테이블부터 만들기
- **증상**: 요구사항 듣자마자 "Create Table..." 명령어를 날림.
- **결과**: 나중에 정산 업무를 하려니 '거래 주체' 정보가 누락되어 스키마를 통째로 뜯어고쳐야 함.
- **조치**: **반드시 주제 영역과 핵심 엔티티 맵을 먼저 완성**하고 현업의 승인을 받은 뒤 상세 설계(논리)에 착수할 것.

### 📢 기술사적 결언
> "개념적 모델링은 **'데이터의 가치'**를 결정하는 상위 기획 단계다. 무엇을 기억하고 무엇을 버릴 것인지를 정하는 철학적 행위이기 때문이다. 기술사는 도구의 화려함에 현혹되지 말고, 비즈니스의 본질을 꿰뚫는 **'핵심 명사'**를 찾아내는 직관력을 길러야 한다. 개념 모델링에서 정립된 **주제 영역 체계**가 향후 시스템의 수평적 확장(MSA)과 데이터 거버넌스의 성패를 좌우하는 기준점이 됨을 명심해야 한다."

- **📢 섹션 요약 비유**: 개념 모델링을 생략하는 것은 '재료도 안 사고 요리부터 시작하는 것'과 같아, 중간에 요리를 포기하게 만들 수 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (개념 모델링 준수 시)

| 지표 | 미준수 시 | 준수 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **요구사항 누락율** | 20% 이상 | **1% 미만** | 설계 재작업 비용 80% 감소 |
| **의사소통 비용** | 높음 (용어 혼선) | **낮음 (시각적 일치)** | 협의 시간 50% 단축 |
| **모델 유연성** | 경직됨 | **확장 가능** | 신규 비즈니스 수용력 증대 |

### 2. 미래 전망
- **AI-Powered Concept Mapping**: 텍스트 형태의 비즈니스 정책서를 AI가 읽고 자동으로 주제 영역과 핵심 엔티티 후보를 제안하는 **지능형 모델링 어시스턴트** 상용화.
- **Semantic Linkage**: 기업 내 산재한 모든 데이터의 개념 모델을 하나로 엮어, 데이터의 위치에 상관없이 비즈니스 용어로 검색하는 **데이터 카탈로그** 기술 발전.

- **📢 섹션 요약 비유**: 미래에는 기술이 발전해도 '우리가 무엇을 관리해야 하는가'라는 개념적 질문은 여전히 인간의 가장 가치 있는 창의적 작업으로 남을 것입니다.
