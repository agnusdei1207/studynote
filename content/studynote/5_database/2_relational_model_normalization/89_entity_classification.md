# 89. 엔티티 분류 (Core, Action Entity) - 핵심 엔티티와 행위 엔티티

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 내의 개체들을 업무적 성격과 발생 시점에 따라 비즈니스의 근간이 되는 **핵심 엔티티(Core/Base)**와 업무 수행 과정에서 생성되는 **행위 엔티티(Action)**로 분류하는 모델링 기법이다.
> 2. **구조적 위계**: 핵심 엔티티는 독립적으로 존재하며 부모 역할을 수행하는 반면, 행위 엔티티는 핵심 엔티티 간의 상호작용 결과로 발생하여 부모를 참조하는 자식 역할을 수행한다.
> 3. **가치**: 엔티티의 성격을 명확히 분류함으로써 데이터 모델의 가독성을 높이고, 데이터 정규화 및 라이프사이클 관리(Archiving 등) 전략을 수립하는 데 결정적인 기준을 제공한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **핵심 엔티티 (Core/Base/Master Entity)**: 업무의 대상이 되는 실체로, 다른 엔티티 없이 독립적으로 생성되는 기본 데이터를 의미합니다. (예: 고객, 상품, 사원)
- **행위 엔티티 (Action/Active Entity)**: 두 개 이상의 핵심 엔티티가 업무적으로 상호작용하여 발생하는 데이터를 의미합니다. (예: 주문, 예약, 결제)

### 2. 💡 비유: 영화 제작의 '배우'와 '명대사'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        엔티티 분류 비유                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 핵심 엔티티 (배우): "유재석", "이광수"라는 실체.                             │
│     - 촬영이 없어도(업무 전에도) 이들은 이미 존재합니다.                          │
│                                                                             │
│  2. 행위 엔티티 (명대사/NG장면): 배우들이 만나서 촬영할 때 발생하는 기록.           │
│     - 배우(핵심)들이 모여서 연기를 해야만 '대사(행위)' 데이터가 생깁니다.           │
│                                                                             │
│  → 핵심 엔티티는 **'세상의 주인공'**이고, 행위 엔티티는 그들이 만들어내는           │
│     **'사건의 기록'**과 같습니다.                                               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 흐름의 시각화**: 수백 개의 테이블 중 어떤 것이 중심이고 어떤 것이 파생 데이터인지 한눈에 파악하기 위해 체계적 분류 필요.
- **성능 및 저장 전략 차별화**: 마스터 데이터(핵심)는 양이 적고 조회가 잦지만, 트랜잭션 데이터(행위)는 양이 폭발적으로 늘어나므로 관리 방식이 달라야 함.

- **📢 섹션 요약 비유**: 핵심 엔티티는 '주어'이고, 행위 엔티티는 그 주어가 만들어낸 '목적어와 동사의 흔적'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 엔티티 유형별 상세 비교

| 분류 | 영문 명칭 | 주요 특징 | 데이터 예시 | 비유 |
|:---:|:---|:---|:---|:---|
| **핵심 엔티티** | Core / Master | 독립적 생존, 정적(Static), 적은 데이터량. | 고객, 상품, 부서, 창고 | 뿌리 |
| **개념 엔티티** | Concept | 조직의 규칙이나 분류 체계 정의. | 코드, 등급, 카테고리 | 줄기 |
| **행위 엔티티** | Action | 핵심들 간의 관계에서 발생, 동적, 대량 발생. | 주문, 배송, 입금, 상담 | 열매 |

### 2. 엔티티 발생 흐름도 (ASCII)

핵심 엔티티에서 행위 엔티티가 파생되는 논리적 순서를 시각화한다.

```text
    [ Core Entity A: 고객 ]       [ Core Entity B: 상품 ]
               │                         │
               └───────────┬─────────────┘
                           │
                < Action Entity: 주문 >
               ┌───────────┴─────────────┐
               │                         │
    [ Action Entity: 배송 ]      [ Action Entity: 결제 ]
    (주문이 있어야 발생)           (주문이 있어야 발생)
```

### 3. 심층 동작 원리: 발생 시점에 따른 라이프사이클
1. **정적 데이터 (핵심)**: 시스템 구축 시점에 기초 데이터로 적재되며, 데이터 수정 빈도가 낮음.
2. **동적 데이터 (행위)**: 비즈니스가 가동되는 동안 실시간으로 생성되며, 특정 시간이 지나면 아카이빙(Archiving) 대상이 됨.

### 4. 핵심 기술: 중심 엔티티 (Main Entity) 식별
- 행위 엔티티 중에서도 업무의 핵심이 되는 엔티티를 별도로 분류하여 관리.
- 예: 쇼핑몰의 '주문', 은행의 '계좌이체' 등은 행위 엔티티이면서도 전체 시스템의 가장 중요한 **Main 엔티티**가 됨.

- **📢 섹션 요약 비유**: 핵심 엔티티는 '무대 장치'이고, 행위 엔티티는 무대 위에서 벌어지는 '연극 내용'입니다.

+++

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 핵심(Core) vs 행위(Action) 저장소 전략 비교

| 비교 항목 | 핵심 엔티티 관리 | 행위 엔티티 관리 |
|:---:|:---|:---|
| **저장 매체** | 고속 메모리 / SSD (조회 위주) | 대용량 스토리지 / Cloud (저장 위주) |
| **인덱스 전략** | 다양한 검색 조건에 인덱스 부여 | **Partitioning** 및 날짜 기반 인덱스 |
| **백업 주기** | 변경 시 즉시 백업 | 정기적인 Batch 백업 및 CDC 활용 |
| **데이터 독립성** | 매우 높음 (참조되는 쪽) | 낮음 (부모 개체에 종속) |

### 2. 과목 융합 관점: 소프트웨어 공학 및 빅데이터
- **소프트웨어 공학(SE) 융합**: 핵심 엔티티는 **도메인 엔티티(Domain Entity)**로, 행위 엔티티는 **트랜잭션 스크립트(Transaction Script)**나 이벤트 로그로 모델링된다.
- **빅데이터 융합**: 데이터 웨어하우스(DW)의 **Dimension Table**은 핵심 엔티티에, **Fact Table**은 행위 엔티티에 대응되어 다차원 분석의 토대가 된다.

- **📢 섹션 요약 비유**: 핵심 엔티티가 '사전의 단어'라면, 행위 엔티티는 그 단어들로 만들어진 '매일매일의 일기'입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 행위 엔티티의 파티셔닝(Partitioning) 설계
- **상황**: '주문' 테이블의 데이터가 매일 수백만 건씩 쌓여 조회가 느려짐.
- **판단**: '주문'은 행위 엔티티이므로 시간이 지남에 따라 데이터 가치가 하락함.
- **결단**: **월 단위 범위 파티셔닝(Range Partition) 적용**. 최신 데이터는 고속 영역에, 1년 이상 된 데이터는 저가형 스토리지로 분리(Tiering)하여 성능과 비용을 동시에 최적화함.

### 2. 안티패턴: 핵심 엔티티와 행위 엔티티의 혼용
- **증상**: '고객' 테이블에 '최근 주문 상품' 정보를 속성으로 직접 저장함.
- **결과**: 고객이 주문할 때마다 고객 테이블 전체에 락이 걸려 다른 정보 수정이 막히고, 중복 데이터로 인한 정합성 파괴.
- **조치**: **핵심(고객)과 행위(주문상품)를 엔티티 단위로 명확히 분리**하고 1:N 관계로 설계할 것.

### 📢 기술사적 결언
> "엔티티 분류는 데이터베이스의 **'질서'**를 잡는 첫 단추다. 무엇이 변하지 않는 가치(핵심)이고 무엇이 흐르는 물(행위)인지 구분하지 못하면 데이터의 홍수에 휩쓸리게 된다. 기술사는 모델링 시 **엔티티의 생명 주기**를 먼저 예측하고, 대량으로 발생하는 행위 엔티티에 대해서는 **쓰기 성능 최적화와 파티셔닝** 전략을, 핵심 엔티티에 대해서는 **데이터 표준화와 캐싱** 전략을 차별화하여 적용하는 '전략적 설계자'가 되어야 한다."

- **📢 섹션 요약 비유**: 분류 없는 모델링은 '박물관 유물(핵심)과 오늘 들어온 택배 박스(행위)를 한 방에 섞어두는 것'과 같이 관리를 불가능하게 만듭니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (유형별 차별화 관리 시)

| 지표 | 관리 미흡 시 | 최적 관리 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **스토리지 비용** | 전량 고비용 SSD 사용 | **계층별 분산 저장** | 인프라 비용 50% 절감 |
| **시스템 응답 속도** | 대량 데이터로 인해 저하 | **파티셔닝으로 일관성 유지** | 성능 안정성 확보 |
| **운영 가시성** | 전체 구조 파악 힘듦 | **비즈니스 흐름 명확** | 분석 및 보고 속도 향상 |

### 2. 미래 전망
- **Event-Driven Entity**: 행위 엔티티가 단순 테이블이 아닌 **이벤트 스트림(Kafka 등)**으로 관리되며, 실시간으로 핵심 엔티티의 상태를 업데이트하는 실시간 정합성 아키텍처 확산.
- **Immutable Core Entities**: 핵심 엔티티의 이력을 모두 보존하기 위해, 수정 대신 새로운 행을 쌓는 **Append-only Master Data** 관리 기법 부상.

- **📢 섹션 요약 비유**: 미래의 엔티티 시스템은 정지된 기록이 아니라, 핵심이라는 뼈대 위에서 행위라는 근육이 실시간으로 움직이는 '살아있는 생명체'가 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **핵심 엔티티**는 우리 집에 원래부터 있는 **"가구"**와 같아요. 침대, 책상처럼 자리를 지키고 있죠.
2. **행위 엔티티**는 우리가 가구를 쓰면서 생기는 **"먼지나 낙서"**와 같아요. 매일매일 조금씩 생겨나는 것들이죠.
3. 가구(핵심)는 튼튼하게 잘 관리하고, 먼지(행위)는 쌓이지 않게 자주 정리해줘야 집이 깨끗해진답니다!
