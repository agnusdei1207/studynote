# 110. 중복 컬럼 및 중복 관계 - 조인 횟수 감소 전략

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 조인(Join) 연산으로 인한 성능 저하를 해결하기 위해, 부모 테이블의 특정 컬럼을 자식 테이블에 복제하거나(**중복 컬럼**), 여러 단계를 거쳐야 하는 조인 경로를 직접 연결하는(**중복 관계**) 반정규화 기법이다.
> 2. **성능 극대화**: 데이터의 논리적 최소성보다는 **조회 편의성**에 집중하여, 한 번의 I/O나 최소한의 테이블 접근으로 원하는 정보를 얻게 함으로써 응답 속도를 비약적으로 향상시킨다.
> 3. **가치**: 트래픽이 집중되는 핵심 비즈니스 경로(Hot Path)의 병목을 제거하는 강력한 도구이며, 수반되는 데이터 업데이트 부하와 일관성 유지 비용을 설계 단계에서 정밀하게 관리해야 한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **중복 컬럼 (Redundant Column)**: 조인 없이도 값을 알 수 있도록, 다른 테이블에 이미 존재하는 속성을 현재 테이블에 추가로 저장하는 기법입니다.
- **중복 관계 (Redundant Relationship)**: 조인 경로가 너무 길 때(A-B-C), 중간 테이블을 건너뛰고 끝단끼리 직접 연결하는(A-C) 외래 키를 추가하는 기법입니다.

### 2. 💡 비유: 택배 주소지와 연락처 기록
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        중복 컬럼 및 관계 비유                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 중복 컬럼 (운송장 정보):                                                  │
│     - 원래 고객 이름은 '고객 명부'에만 있어야 하지만, 택배 박스(주문) 위에도      │
│       이름을 적어둠. (중복 컬럼)                                               │
│     - 매번 명부를 찾아보지(Join) 않아도 누구 건지 바로 알 수 있음! (속도)         │
│                                                                             │
│  2. 중복 관계 (직통 전화):                                                   │
│     - 사장님께 보고하려면 대리-과장-부장을 거쳐야 함. (A-B-C 관계)                │
│     - 급한 일 처리를 위해 사장님과 나 사이의 직통 라인을 만듦. (A-C 중복 관계)     │
│                                                                             │
│  → 복잡한 절차를 생략하고 **'즉시 결과를 얻기 위해'** 정보를 겹쳐두는 지혜입니다!   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **조인의 물리적 한계**: 테이블 규모가 수천만 건을 넘어서면, 인덱스가 있더라도 조인 자체의 연산 부하가 시스템 전체의 병목이 됨.
- **분산 아키텍처의 요구**: 마이크로서비스(MSA) 간의 조인은 네트워크 통신을 유발하므로, 성능을 위해 필수적인 데이터를 미리 복제(Data Replication)해 두어야 함.

- **📢 섹션 요약 비유**: 중복 컬럼과 관계는 데이터베이스의 '즐겨찾기'와 같아서, 복잡한 경로를 거치지 않고 바로 목적지에 도달하게 해줍니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 기법별 적용 기준 및 사례

| 기법 명칭 | 적용 방식 | 주요 사례 | 기대 효과 |
|:---:|:---|:---|:---|
| **중복 컬럼 추가** | 부모의 일반 속성을 자식에 복사. | 주문 테이블에 '상품명' 저장. | 조인 횟수 1회 감소. |
| **중복 관계 추가** | 조인 경로를 단축하는 FK 추가. | 수강-과목-학과 관계에서 수강에 '학과코드' 추가. | 다단계 조인 제거. |
| **파생 컬럼 추가** | 계산된 결과를 미리 저장. | 주문 테이블에 '총 결제 금액' 저장. | SUM/AVG 연산 제거. |

### 2. 중복 관계 아키텍처 도해 (ASCII)

다단계 조인 경로를 직접 연결로 단축하는 과정을 시각화한다.

```text
    [ 정규화 모델: 2-Hop Join ]
    (학생) ─<소속>─ (학과) ─<개설>─ (과목)
    * 학생이 듣는 과목의 학과명을 알려면 2번 조인해야 함.
    
    [ 중복 관계 모델: 1-Hop Join ]
    (학생) ─<소속>─ (학과) ──┐
      │                     │
      └──────<직접 관계>────▶ (과목)
    * 과목 테이블에 '소속학과코드'를 직접 넣음으로써, 
      학생과 과목만 조인해도 학과 정보를 즉시 파악 가능.
```

### 3. 심층 동작 원리: 데이터 일관성 비용 (Consistency Cost)
- **현상**: 부모 테이블의 데이터(예: 상품명)가 바뀌면, 이를 중복 저장한 수만 건의 자식 테이블(예: 주문)도 모두 고쳐야 함.
- **판단**: 데이터 변경이 매우 드물고(Low Mutation), 조회가 압도적으로 많은(High Read) 속성이 반정규화의 최적 후보임.

### 4. 핵심 기술: 트리거(Trigger)를 이용한 자동 동기화
- 중복 컬럼의 정합성을 위해 부모 테이블 업데이트 시 자식 테이블을 함께 고치는 트리거 로직을 구현함.
- (참고: 성능을 위해 비동기 배치로 처리하기도 함)

- **📢 섹션 요약 비유**: 중복 관계는 '환승역을 거치지 않고 한 번에 가는 급행열차 노선'을 새로 까는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 중복 컬럼 vs 뷰(View) vs 인덱스(Index)

| 비교 항목 | 중복 컬럼 (Denorm) | 가상 뷰 (View) | 커버링 인덱스 (Index) |
|:---:|:---|:---|:---|
| **물리 저장** | **실제 데이터 중복 저장** | 저장 공간 없음 (논리) | 인덱스 공간 사용 |
| **성능 체감** | 최상 (조인 제거) | 보통 (쿼리만 단순화) | 상 (테이블 접근 제거) |
| **정합성 유지** | 개발자 수동 관리 | **자동 (원본 참조)** | **자동** |
| **추천 상황** | 조인 자체가 너무 무거울 때 | 쿼리 가독성이 필요할 때 | 인덱스만으로 조회 가능할 때 |

### 2. 과목 융합 관점: 소프트웨어 공학 및 분산 시스템
- **소프트웨어 공학(SE) 융합**: 중복 컬럼은 디자인 패턴 중 **프록시(Proxy)**나 **캐시(Cache)** 패턴의 데이터 계층 버전이다.
- **분산 시스템 융합**: 서로 다른 데이터베이스 인스턴스 간 조인이 불가능한 환경에서, 데이터를 비동기적으로 복제하여 로컬에 두는 **CQRS(Command Query Responsibility Segregation)** 패턴의 핵심 기술이다.

- **📢 섹션 요약 비유**: 뷰가 '거울'이라면, 중복 컬럼은 '복사기로 찍어낸 종이'입니다. 거울은 원본을 비추지만, 종이는 원본이 변해도 스스로 변하지 않습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 사용자 프로필 조회의 성능 병목 해결
- **상황**: '댓글' 목록 조회 시 매번 '작성자 이름'과 '프로필 이미지'를 가져오기 위해 '회원' 테이블과 조인함. 댓글이 많아질수록 서버 부하 급증.
- **판단**: 회원 이름과 이미지는 거의 변하지 않는 정적 데이터임.
- **결단**: **'댓글' 테이블에 '작성자이름', '프로필이미지URL'을 중복 컬럼으로 추가**. 조인을 0회로 만들어 댓글 로딩 속도를 5배 이상 단축하고, 회원이 이름을 바꿀 때만 배치를 통해 천천히 반영하는 정책 수립.

### 2. 안티패턴: 자주 변하는 속성의 중복 저장
- **증상**: '현재 재고 수량'을 '주문' 테이블에 중복 저장함.
- **결과**: 재고는 실시간으로 변하므로, 주문이 들어올 때마다 과거 모든 주문 행을 고칠 수 없어 데이터 불일치 대폭증.
- **조치**: **변동성이 큰 데이터는 절대 중복 저장하지 말고**, 조인 성능을 위해 인덱스나 인메모리 캐시를 활용할 것.

### 📢 기술사적 결언
> "중복은 **'빚'**이다. 오늘 성능을 빌려 쓰고, 내일 정합성 유지라는 이자를 내는 행위이기 때문이다. 기술사는 단순히 성능이 빨라진다고 좋아하는 것이 아니라, 이 중복으로 인해 발생할 **데이터 오염의 확률**과 **업데이트 오버헤드**를 정밀하게 시뮬레이션해야 한다. '정규화로 정교하게 쌓고, 반정규화로 날카롭게 깎는다'는 원칙 하에, 비즈니스 가치가 가장 높은 지점에만 중복의 마법을 부려야 한다."

- **📢 섹션 요약 비유**: 무분별한 중복 컬럼은 '모든 문서에 날짜와 날씨를 다 적어두는 것'과 같아, 나중에 날씨 정보가 틀렸음을 알게 되었을 때(데이터 수정) 모든 문서를 파기해야 할 수도 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (중복 전략 최적 적용 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **쿼리당 조인 횟수** | 평균 4.5회 | **1.2회** | 조인 연산 부하 70% 감소 |
| **CPU 사용량** | 90% (병목 발생) | **40%** | 시스템 안정성 및 수용량 증대 |
| **데이터 정합성** | 100% (정규화) | **99.9% (보완책 적용)** | 성능 대비 신뢰도 방어 |

### 2. 미래 전망
- **Smart Replication**: AI가 쿼리 통계를 보고 어느 컬럼을 중복 저장할지 스스로 결정하고 백그라운드에서 동기화하는 **자율형 중복 관리** 시스템의 등장.
- **Event-Driven Duplication**: Kafka와 같은 이벤트 버스를 통해 부모 데이터의 변경이 발생 즉시 수백 개의 자식 테이블 중복 컬럼으로 전파되는 **실시간 복제 아키텍처** 보편화.

- **📢 섹션 요약 비유**: 미래의 데이터 중복은 단순한 복사가 아니라, 데이터 스스로가 가장 필요한 곳으로 분신을 보내는 '지능형 분신술'이 될 것입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **반정규화 기법**: 중복 컬럼이 속한 상위 분류.
- **조인 (Join)**: 중복을 통해 피하고자 하는 대상.
- **데이터 정합성 (Consistency)**: 중복 저장 시 희생되는 가치.
- **CQRS 패턴**: 중복 저장을 아키텍처 레벨로 격상한 사례.

### 👶 어린이를 위한 3줄 비유 설명
1. **중복 컬럼**은 내 가방 이름표에 **"우리 집 주소"**까지 미리 적어두는 거예요.
2. 매번 지갑 속의 주소 카드를 꺼내 확인(조인)하지 않아도, 가방만 보면 **금방 집을 찾을 수** 있죠.
3. 대신 이사가면 가방에 적힌 주소를 **전부 다 지우고 새로 써야 해서** 조금 번거로울 수 있답니다!
