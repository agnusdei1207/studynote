# 108. 반정규화 기법 - 테이블 통합, 분할, 속성 및 관계 중복

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템의 입출력(I/O) 성능을 최적화하기 위해 테이블의 구조를 변경하거나 데이터의 복사본을 전략적으로 배치하는 **물리적 설계의 구체적 방법론**이다.
> 2. **유형별 접근**: 조인을 줄이기 위한 **테이블 통합/속성 중복**, 데이터 경합을 줄이기 위한 **테이블 분할**, 그리고 연산 속도를 높이기 위한 **파생/요약 컬럼 추가**로 대별된다.
> 3. **가치**: 단순히 데이터를 합치는 수준을 넘어, 하드웨어 자원의 병목 지점을 정확히 타격하여 **조회 성능의 극대화**를 달성하며, 수반되는 데이터 불일치 리스크를 관리하는 보완 설계가 필수적이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**반정규화 기법**은 정규화된 데이터 모델이 실무 환경의 대량 트래픽이나 복잡한 쿼리를 견디지 못할 때, 이를 해결하기 위해 동원되는 구체적인 기술적 수단들입니다. 크게 테이블 단위, 속성(컬럼) 단위, 관계 단위의 기법으로 나뉩니다.

### 2. 💡 비유: 가구 배치와 집 정리 전략
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        반정규화 기법 비유                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 테이블 통합 (거실 합치기): 거실과 주방 사이의 벽을 허물어 동선을 줄임.         │
│  2. 테이블 분할 (개인 방 나누기): 너무 붐비는 거실을 피해 각자 방에서 공부함.      │
│  3. 속성 중복 (복사본 두기): 주방에도 시계가 있지만, 거실에도 시계를 하나 더 둠.   │
│  4. 파생 컬럼 (미리 계산): 매번 영수증을 다 더하지 않고, 지갑에 총액을 적어둠.     │
│                                                                             │
│  → 집의 구조(스키마)를 **'살기 편하게(성능)'** 고치는 다양한 공사 방법들입니다!    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **현실적 제약**: 이론적 정규화는 조인 횟수를 늘려 쿼리 실행 계획을 복잡하게 만듦.
- **분산 환경의 도래**: 서로 다른 서버(노드)에 있는 테이블 간의 조인은 비용이 너무 커서, 아예 데이터를 복사해두는 것이 유리할 때가 많음.

- **📢 섹션 요약 비유**: 반정규화 기법은 상황에 따라 망치(분할)를 쓸지, 풀(통합)을 쓸지 결정하는 '성능 튜닝의 도구 상자'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 반정규화 기법의 3대 분류 및 상세 내용

| 분류 | 주요 기법 | 적용 내용 | 비유 |
|:---:|:---|:---|:---|
| **테이블 단위** | **통합 (Merge)** | 1:1, 1:N 관계의 테이블을 하나로 합침. | 방 합치기 |
| | **분할 (Split)** | 수직 분할(컬럼 단위), 수평 분할(행 단위 - 파티셔닝). | 가벽 세우기 |
| **속성(컬럼) 단위** | **중복 컬럼 추가** | 조인 없이 조회하도록 부모 속성을 자식에 복사. | 보조 배터리 |
| | **파생 컬럼 추가** | 합계, 평균 등 미리 계산된 값을 컬럼에 저장. | 미리 계산기 |
| **관계 단위** | **중복 관계 추가** | 여러 단계를 거치는 조인을 위해 직접 링크 생성. | 지름길 터널 |

### 2. 기법별 데이터 레이아웃 변화 (ASCII)

중복 컬럼 추가와 수직 분할의 차이를 시각화한다.

```text
    [ Case 1: 중복 컬럼 추가 ]
    Order (OID, Date, Cust_ID) + (Cust_Name) <--- (Customer 테이블에서 복사)
    * 효과: 고객 이름을 알기 위해 Customer 테이블로 갈 필요 없음.
    
    [ Case 2: 테이블 수직 분할 ]
    Employee_Base (EID, Name, Dept)  <--- (자주 조회되는 필드)
    Employee_Detail (EID, Photo, Resume) <--- (가끔 조회되는 무거운 필드)
    * 효과: 기본 정보 조회 시 디스크 I/O 크기 대폭 감소.
```

### 3. 심층 동작 원리: 테이블 수평 분할 (Horizontal Partitioning)
- **원리**: 행(Tuple) 단위로 데이터를 쪼개어 별도의 물리적 저장 공간에 분산.
- **방식**: 범위(Range), 해시(Hash), 리스트(List) 기준 사용.
- **효과**: 특정 기간이나 지역 데이터만 조회할 때 관련 파티션만 읽어 성능 극대화 (**Partition Pruning**).

### 4. 핵심 기술: 요약 테이블 (Summary Table)
- 원본 테이블의 상세 내역을 주기적으로 집계하여 별도의 테이블에 관리.
- 데이터 정합성 유지를 위해 **CDC (Change Data Capture)**나 배치(Batch) 작업 연동이 필수적.

- **📢 섹션 요약 비유**: 수직 분할은 '매일 입는 옷은  دم 옷장에, 계절 옷은 창고에' 넣는 계층형 정리법과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 테이블 통합 vs 테이블 분할의 전략적 선택

| 비교 항목 | 테이블 통합 (Consolidation) | 테이블 분할 (Fragmentation) |
|:---:|:---|:---|
| **주요 목적** | **조인(Join) 횟수 감소** | **I/O 경합 및 잠금(Lock) 감소** |
| **성능 혜택** | 조회 속도 향상 | 동시성 및 관리 효율 향상 |
| **위험 요소** | 테이블 비대화, 데이터 중복 | 관리 복잡도 증가, 조인 필요성 발생 |
| **추천 상황** | 조인이 너무 잦은 경우 | 컬럼 수가 너무 많거나 대용량인 경우 |

### 2. 과목 융합 관점: OS 및 네트워크
- **운영체제(OS) 융합**: 테이블 분할은 OS의 **병렬 파일 시스템** 활용도를 높이며, 서로 다른 디스크에 데이터를 배치하여 I/O 대역폭을 극대화한다.
- **네트워크(NW) 융합**: 중복 컬럼 기법은 분산 시스템에서 **네트워크 홉(Hop)** 수를 줄여, 마이크로서비스 간 통신 부하를 줄이는 핵심 아키텍처 패턴이 된다.

- **📢 섹션 요약 비유**: 통합은 '한 바구니에 다 담아 한 번에 옮기기'이고, 분할은 '여러 명이 나눠서 가볍게 나르기'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 대규모 이커머스의 '상품 정보' 반정규화
- **상황**: 상품 상세 페이지 노출 시 상품 정보, 카테고리, 이미지 주소 등 5개 테이블 조인 발생.
- **판단**: 조회 트래픽이 쓰기보다 100배 이상 많음. 조인 부하가 서버의 70% 차지.
- **결단**: **상품 엔티티에 카테고리명과 메인 이미지 경로를 중복 컬럼으로 추가**. 조인 횟수를 5회에서 1회로 줄여 사용자 체감 속도를 0.5초에서 0.05초로 단축.

### 2. 안티패턴: 무분별한 '합계' 컬럼의 실시간 업데이트
- **증상**: "조회 성능이 중요하다"며 모든 주문 발생 시 실시간으로 '회원 누적 구매액' 업데이트.
- **결과**: 결제 시점에 해당 회원 레코드에 강력한 락(Lock)이 걸려, 정작 결제 시스템이 느려지는 **쓰기 지연** 발생.
- **조치**: 실시간성이 극도로 중요하지 않다면 **배치 작업으로 분리**하거나, **In-memory 캐시**를 활용하여 기록하고 나중에 한꺼번에 DB에 반영할 것.

### 📢 기술사적 결언
> "반정규화 기법은 **'맞춤형 튜닝'**의 정수다. 모든 약에는 부작용이 있듯이, 모든 기법에는 리스크가 따른다. 기술사는 단순히 컬럼을 복사하는 작업자가 아니라, **데이터의 갱신 빈도(Mutability)**와 **동시성 제어 비용**을 계산하여 '수술' 부위를 정확히 결정하는 아키텍트가 되어야 한다. 기법 적용 후에는 반드시 **데이터 일관성 체크 스크립트**를 마련하여, 성능을 위해 정직함을 버리지 않았는지 스스로 검증해야 한다."

- **📢 섹션 요약 비유**: 중복 관계 추가는 '복잡한 숲속 길에서 나무를 베어 지름길을 만드는 것'과 같아, 관리가 안 되면 나중에 숲 전체를 망칠 수도 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (기법 최적 적용 시)

| 지표 | 적용 전 | 적용 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **응답 속도 (Latency)** | 1.2초 | **0.08초** | 사용자 만족도 비약적 향상 |
| **I/O 처리량** | 1,000 IOPS | **300 IOPS** | 스토리지 부하 70% 절감 |
| **데이터 안정성** | - | **보완책(Trigger) 확보** | 시스템 신뢰도 유지 |

### 2. 미래 전망
- **Software-Defined Denormalization**: 사용자의 질의를 머신러닝이 분석하여, 물리적 저장 레이아웃을 스스로 통합/분할하는 **자율 운전형 스토리지 엔진** 보편화.
- **Materialized Views in Cloud**: 클라우드 벤더가 제공하는 관리형 구체화 뷰 기술을 통해, 사용자가 직접 반정규화 기법을 고민하지 않아도 논리 스키마를 성능과 자동 분리해주는 추세.

- **📢 섹션 요약 비유**: 미래의 기법은 고정된 칼날이 아니라, 상황에 따라 형태를 바꾸는 '스마트 나이프'처럼 변할 것입니다.
