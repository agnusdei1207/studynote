# 73. 무결성 제약 조건 (Integrity Constraints) - 데이터 정확성 보장 규칙

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스에 저장된 데이터가 현실 세계의 실제 값과 일치하며, 논리적인 오류가 없음을 보장하기 위해 DBMS가 강제하는 **선언적 규칙(Declarative Rules)**의 집합이다.
> 2. **3대 핵심 유형**: 튜플의 유일 식별을 보장하는 **개체 무결성**, 테이블 간의 관계를 보호하는 **참조 무결성**, 속성 값의 범위를 규정하는 **도메인 무결성**을 통해 다층적인 방어 체계를 구축한다.
> 3. **가치**: 애플리케이션 버그나 사용자의 실수로 인한 데이터 오염(Garbage in, Garbage out)을 엔진 레벨에서 원천 차단하여, 기업 정보 자산의 **신뢰성**과 **영속성**을 담보한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**무결성 제약 조건 (Integrity Constraints)**은 데이터베이스의 상태가 항상 '유효한 상태'를 유지하도록 하는 제약입니다. 데이터의 삽입, 수정, 삭제 시마다 DBMS는 이 규칙을 검사하여, 규칙에 어긋나는 모든 연산을 거부(Reject)합니다.

### 2. 💡 비유: 은행의 입출금 규칙
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        무결성 제약 조건 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 개체 무결성: 모든 고객은 고유한 '계좌번호'가 있어야 함. (중복/빈칸 불가)       │
│  2. 참조 무결성: 돈을 보내려는 '상대방 계좌'는 반드시 실재해야 함. (허상 금지)     │
│  3. 도메인 무결성: 송금액은 반드시 '숫자'여야 하며, '0원 이상'이어야 함. (형식 준수)│
│                                                                             │
│  → 이 규칙들이 하나라도 어긋나면 은행 전산은 거래를 취소합니다.                  │
│     덕분에 우리의 예금 잔액이 엉망진창이 되지 않고 안전하게 보관되는 것이죠!       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 신뢰도 하락 방지**: 파일 시스템 시절, 데이터가 여러 곳에 흩어져 있어 값이 서로 달라지는 '데이터 불일치'가 빈번하게 발생함.
- **애플리케이션 로직 독립**: 모든 프로그램마다 중복으로 구현하던 검사 로직을 DB 엔진으로 일원화하여 관리 효율성 및 보안성 강화.

- **📢 섹션 요약 비유**: 무결성 제약 조건은 데이터베이스라는 성을 지키는 '철저한 문지기'와 같아, 자격 없는 데이터는 절대로 들여보내지 않습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 무결성 제약 조건의 주요 분류

| 분류 | 상세 설명 | 구현 예시 (SQL) |
|:---:|:---|:---|
| **개체 무결성 (Entity)** | 기본 키(PK)는 NULL일 수 없고 유일해야 함. | `PRIMARY KEY` |
| **참조 무결성 (Referential)** | 외래 키(FK)는 참조하는 테이블의 PK값이거나 NULL이어야 함. | `FOREIGN KEY` |
| **도메인 무결성 (Domain)** | 속성 값은 정해진 데이터 타입과 범위 내에 있어야 함. | `NOT NULL`, `CHECK`, `DEFAULT` |
| **사용자 정의 무결성** | 비즈니스 로직에 따른 특수한 제약 사항. | `TRIGGER`, `ASSERTION` |

### 2. 무결성 검사 메커니즘 도해 (ASCII)

데이터 변경 요청 시 DBMS가 수행하는 검사 흐름을 시각화한다.

```text
    [ SQL Request: INSERT/UPDATE ]
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Integrity Manager ] : 검사 엔진 가동                          │
    │                                                                 │
    │  1. [ Type Check ] : 도메인이 일치하는가? (INT vs STR)           │
    │  2. [ Null Check ] : 필수 입력값이 비어있지 않은가? (NOT NULL)    │
    │  3. [ Unique Check ] : 기존 데이터와 겹치지 않는가? (PK/UK)       │
    │  4. [ Reference Check ] : 부모 데이터가 실재하는가? (FK)          │
    │                                                                 │
    └──────┬──────────────────────────┬───────────────────────────────┘
           │ (모두 통과 시)             │ (하나라도 위반 시)
    ┌──────▼──────┐            ┌──────▼──────┐
    │ Commit      │            │ Rollback    │
    │ (DB 반영)    │            │ (Error 반환) │
    └─────────────┘            └─────────────┘
```

### 3. 심층 동작 원리: 상태 무결성 vs 과도 무결성
- **상태 무결성 (State)**: 정지된 상태의 DB가 만족해야 하는 조건. (예: 현재 잔액은 0원 이상이어야 함)
- **과도 무결성 (Transition)**: 상태가 변할 때(Update) 만족해야 하는 조건. (예: 월급은 이전보다 깎일 수 없음 - Trigger로 구현)

### 4. 핵심 기술: 선언적 무결성 (Declarative) vs 절차적 무결성 (Procedural)
- **선언적**: 테이블 생성 시 `CONSTRAINT` 문구로 명시. DBMS가 자동 관리. (권장)
- **절차적**: 트리거(Trigger)나 저장 프로시저(Stored Procedure)를 통해 코드로 검사 로직 작성. 복잡한 업무 규칙 적용 시 사용.

- **📢 섹션 요약 비유**: 선언적 무결성은 '건물의 설계도'에 정해진 규칙이고, 절차적 무결성은 '경비원이 그때그때 상황을 보고 판단하는 수칙'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 무결성(Integrity) vs 보안(Security)

| 비교 항목 | 데이터 무결성 (Integrity) | 데이터 보안 (Security) |
|:---:|:---|:---|
| **목적** | 데이터의 **정확성 / 일관성** | 데이터의 **기밀성 / 허가** |
| **대상** | 권한 있는 사용자의 실수/오류 | **권한 없는 사용자의 불법 접근** |
| **위협** | 버그, 시스템 장애, 잘못된 입력 | 해킹, 정보 유출, 악성 코드 |
| **방어 수단** | 제약 조건, 트랜잭션, 복구 | 암호화, 접근 제어(DCL), 인증 |

### 2. 과목 융합 관점: 소프트웨어 공학 및 테스팅
- **소프트웨어 공학(SE) 융합**: 무결성 제약 조건은 **계약에 의한 설계(Design by Contract)** 사상을 데이터 계층에서 구현한 것이며, 전제 조건(Pre-condition)을 명확히 함으로써 시스템 신뢰도를 높인다.
- **테스팅(Testing) 융합**: 데이터 무결성 제약 조건은 **네거티브 테스트(Negative Testing)**를 자동화한 효과를 준다. 잘못된 값을 넣었을 때 시스템이 안정적으로 거부하는지를 엔진이 보장하기 때문이다.

- **📢 섹션 요약 비유**: 무결성이 '음식이 상하지 않게 신선함을 유지하는 것'이라면, 보안은 '음식을 아무나 훔쳐먹지 못하게 감시하는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 성능 향상을 위한 제약 조건 해제 유혹
- **상황**: 대량의 데이터 배치 입력 시, FK 검사 때문에 속도가 너무 느림.
- **판단**: 제약 조건을 끄고(Disable) 입력하면 빠르지만, 입력 도중 장애가 나면 데이터 정합성이 깨짐.
- **결단**: **임시로 제약 조건을 '비검사(NOVALIDATE)' 모드로 전환하여 대량 입력 후, 완료 즉시 다시 활성화(Enable)**. 속도를 챙기되 데이터의 최종 정합성은 반드시 검증하는 절차 수립.

### 2. 안티패턴: DB 무결성을 무시하고 앱에서만 체크함
- **증상**: "로직이 복잡하니 자바 코드에서 다 체크할게요. DB는 그냥 저장만 하세요."
- **결과**: 직접 SQL로 데이터를 수정하거나 다른 시스템과 연동될 때 앱의 체크 로직이 우회되어, 원인 모를 '유령 데이터'와 '불일치 데이터'가 쌓임.
- **조치**: **핵심 비즈니스 정합성은 반드시 DB 레벨의 제약 조건(PK, FK, Check)으로 결착**하여 시스템의 최후 보루를 확보할 것.

### 📢 기술사적 결언
> "무결성 제약 조건은 데이터베이스의 **'양심'**이다. 양심을 저버린 데이터는 정보로서의 가치를 잃고 기업의 리스크가 된다. 기술사는 단순 구현을 넘어 **데이터 거버넌스** 관점에서 무결성을 바라봐야 한다. 시스템의 성능(Performance)과 정합성(Integrity) 사이의 트레이드오프를 정교하게 설계하되, 어떠한 극한의 장애 상황에서도 **비즈니스의 논리적 일관성**만큼은 양보하지 않는 강단 있는 아키텍처를 제시해야 한다."

- **📢 섹션 요약 비유**: 앱에서만 체크하는 것은 '가드레일 없이 운전자의 실력에만 의존하는 고속도로'와 같습니다. 한 번의 졸음(장애)이 대형 사고로 이어집니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (제약 조건 도입 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 정제(Cleaning) 비용** | 매우 높음 | **최소화** | 운영 리소스 60% 이상 절감 |
| **비즈니스 판단 정확도** | 낮음 (오류 데이터 혼재) | **높음** | 의사결정 신뢰도 극대화 |
| **장애 복구 탄력성** | 수동 데이터 보정 필수 | **자동 정합성 유지** | 시스템 안정성 및 가용성 향상 |

### 2. 미래 전망
- **Self-healing Integrity**: AI가 데이터의 패턴을 분석하여, 누락된 무결성 제약 조건을 스스로 제안하거나 데이터 오염 시 자동으로 최적값을 추론하여 보정하는 지능형 무결성 관리.
- **Semantic Integrity in NoSQL**: 스키마리스 환경의 NoSQL 진영에서도 성능 저하 없이 논리적 무결성을 보장하기 위한 **Sidecar Integrity Proxy** 기술 발전.

- **📢 섹션 요약 비유**: 미래의 무결성 시스템은 단순한 금지 규칙을 넘어, 데이터를 스스로 치유하고 올바른 상태로 유도하는 '지능형 면역 체계'로 발전할 것입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **개체 무결성 (Entity Integrity)**: 무결성의 근본.
- **참조 무결성 (Referential Integrity)**: 연결의 무결성.
- **도메인 무결성 (Domain Integrity)**: 값의 무결성.
- **ACID 원칙**: 무결성을 보장하기 위한 트랜잭션 성질.

### 👶 어린이를 위한 3줄 비유 설명
1. **무결성 제약 조건**은 일기장을 쓸 때 **"맞춤법을 꼭 지키고, 거짓말을 적지 말자"**라고 정해둔 규칙이에요.
2. 이 규칙이 있어야 나중에 일기를 읽을 때 **내용을 100% 믿을 수** 있고 헷갈리지 않죠.
3. 컴퓨터가 이 규칙을 꼼꼼하게 검사해주기 때문에, 우리들의 소중한 정보가 **엉터리가 되지 않게** 지켜준답니다!
