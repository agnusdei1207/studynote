# 113. 데이터 모델링과 성능 - 정규화와 성능의 상관관계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 성능은 하드웨어 성능이나 튜닝 이전에, 데이터의 논리적·물리적 구조를 결정하는 **모델링 단계**에서 이미 80% 이상 결정되는 구조적 특성을 가진다.
> 2. **트레이드오프**: 정규화는 중복을 제거하여 **쓰기(Write) 성능과 정합성**을 높이지만 조인(Join) 증가로 읽기 속도를 늦출 수 있으며, 반정규화는 **읽기(Read) 성능**을 극대화하지만 관리 복잡도를 높인다.
> 3. **가치**: 성능 병목의 원인인 디스크 I/O를 최소화하기 위해 **정규화를 통한 블록 밀도 향상**과 **전략적 반정규화를 통한 조인 횟수 조절** 사이의 최적의 균형점을 찾는 것이 아키텍트의 핵심 역량이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**데이터 모델링과 성능**은 데이터베이스 설계가 실제 시스템의 운영 속도와 안정성에 미치는 상관관계를 다룹니다. "설계가 잘못되면 어떤 튜닝으로도 성능을 살릴 수 없다"는 것이 데이터 공학의 정설입니다.

### 2. 💡 비유: 자동차 프레임과 엔진 튜닝
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        모델링과 성능 비유                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 데이터 모델링 (자동차 프레임): 차의 뼈대와 공기 역학적 구조.                  │
│  2. SQL 튜닝 (엔진 튜닝): 엔진의 출력을 높이고 기름(자원)을 조절함.               │
│                                                                             │
│  - 만약 차의 뼈대가 트럭처럼 무겁고 투박하다면(모델링 부실),                     │
│  - 엔진을 아무리 슈퍼카 급으로 튜닝해도(SQL 튜닝),                               │
│  - 시속 300km(고성능)를 낼 수 없고 기름만 낭비하게 됩니다.                        │
│                                                                             │
│  → 성능의 근본은 **'가볍고 튼튼한 뼈대(올바른 모델링)'**에서 나옵니다!             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 양의 폭증**: 단순한 쿼리 최적화로는 해결할 수 없는 테라바이트(TB) 단위의 데이터 환경 도래.
- **비즈니스 민첩성**: 요구사항이 바뀔 때마다 성능 저하 없이 유연하게 구조를 변경할 수 있는 '고성능 구조' 설계 필요.

- **📢 섹션 요약 비유**: 모델링은 성능이라는 건물을 올리기 전 지반을 다지는 공사이며, 지반이 약하면 건물이 높이 올라갈수록(데이터가 많아질수록) 무너집니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 정규화가 성능에 미치는 긍정적/부정적 영향

| 구분 | 긍정적 측면 (Positive) | 부정적 측면 (Negative) |
|:---:|:---|:---|
| **I/O 효율** | **중복 제거로 블록 당 레코드 수 증가**. (I/O 횟수 감소) | - |
| **정합성** | 한곳만 수정하면 되므로 **Update 성능 우수**. | - |
| **조회 연산** | 데이터가 명확히 분리되어 인덱스 활용 용이. | **테이블 간 Join 증가**로 연산량 상승. |
| **유연성** | 쿼리 실행 계획이 단순해짐. | 복잡한 요구사항에서 SQL 가독성 저하. |

### 2. 성능 중심 데이터 모델링 프로세스 (ASCII)

성능을 고려한 모델링의 단계별 체크포인트를 시각화한다.

```text
    [ Logical Modeling ]
           │
    ┌──────▼──────────────────────────────────┐
    │ 1. [ 정규화 수행 ] : 중복 제거 및 무결성 확보 │ (Write 성능 최적화)
    └──────┬──────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────┐
    │ 2. [ 용량 산정 ] : 데이터량 및 트래픽 분석    │ (병목 지점 예측)
    └──────┬──────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────┐
    │ 3. [ 성능 튜닝 ] : 인덱스, 파티셔닝, 반정규화 │ (Read 성능 최적화)
    └──────┬──────────────────────────────────┘
           │
    [ 최적 성능의 물리 스키마 완성 ]
```

### 3. 심층 동작 원리: 블록 밀도와 Full Scan 성능
- **원리**: 정규화로 테이블이 얇아지면(Column 수가 줄어들면), 하나의 디스크 블록에 더 많은 행(Row)을 담을 수 있음 (**Blocking Factor 향상**).
- **효과**: 전체 데이터를 스캔할 때 읽어야 할 물리적 블록 수가 줄어들어, 오히려 정규화된 테이블이 비정규화된 테이블보다 조회 속도가 빠를 수 있음.

### 4. 핵심 기술: 성능 패턴 모델링
- **자주 사용되는 경로(Hot Path)**에는 조인을 최소화하는 반정규화 적용.
- **업데이트가 잦은 속성**은 별도 테이블로 분리하여 락(Lock) 경합 최소화.

- **📢 섹션 요약 비유**: 정규화는 '불필요한 지방을 빼서 근육(밀도)을 만드는 과정'이며, 이는 시스템의 기초 대사량(기본 성능)을 높여줍니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 성능을 위한 모델링 전략 비교

| 비교 항목 | 성능 저하 모델링 (Anti-pattern) | 성능 최적 모델링 (Best-practice) |
|:---:|:---|:---|
| **속성 정의** | 거대한 통합 테이블 (Fat Table) | **주제별 정규화 테이블 (Thin Table)** |
| **관계 설정** | 다단계 조인 유발 (Deep Path) | **중복 관계 추가 (Short-cut)** |
| **데이터 타입** | 범용 문자열 위주 (VARCHAR) | **도메인에 맞는 최적 타입 (INT, DATE)** |
| **대량 데이터** | 단일 대형 테이블 | **파티셔닝 적용 테이블** |

### 2. 과목 융합 관점: 컴구 및 알고리즘
- **컴퓨터 구조 융합**: 모델링에 따른 데이터 배치는 CPU의 **캐시 히트율(L1/L2 Cache)**에 지대한 영향을 미치며, 인접한 데이터 배치(Locality)가 연산 성능을 좌우한다.
- **알고리즘 융합**: 정규화된 구조는 옵티마이저가 **조인 순서(Join Ordering)**를 결정할 때 훨씬 더 정교한 수학적 최적화 알고리즘을 적용할 수 있는 기회를 제공한다.

- **📢 섹션 요약 비유**: 최적 타입 선택은 '꼭 맞는 박스에 짐을 담는 것'과 같아, 남는 공간(메모리 낭비) 없이 꽉꽉 채워 옮길 수 있게 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 금융 시스템의 마스터 데이터 설계
- **상황**: 고객의 '현재 등급' 정보가 조인 시마다 성능을 갉아먹음.
- **판단**: 등급은 한 달에 한 번만 변함. 조회가 쓰기보다 10,000배 많음.
- **결단**: **전략적 반정규화 수행**. 주문 테이블에 등급 정보를 중복 저장하되, 등급 변경 시 배치로 동기화하는 정책 수립. 조인 오버헤드를 제거하여 피크 타임 응답 속도 확보.

### 2. 안티패턴: "성능 때문에 무조건 다 합치자"는 만능주의
- **증상**: 모델링 초기부터 모든 테이블을 하나로 합침.
- **결과**: 데이터가 늘어날수록 한 행의 길이가 너무 길어져(Row Chaining), 작은 정보 하나를 읽을 때도 수십 개의 블록을 읽어야 하는 성능 재앙 초래.
- **조치**: **선(先) 정규화, 후(後) 반정규화** 원칙을 고수하고, 데이터의 **Selectivity(선택도)**를 분석하여 물리적 레이아웃을 결정할 것.

### 📢 기술사적 결언
> "데이터 모델링은 **'성능의 설계도'**다. 쿼리 튜닝이 전술이라면 모델링은 전략이다. 기술사는 단순히 정규화 규칙을 지키는 것에 만족하지 말고, **비즈니스 트랜잭션의 흐름**을 데이터 구조에 미리 녹여내야 한다. 특히 현대의 **In-memory DB나 컬럼 지향 DB** 환경에서는 기존의 로우 지향 모델링 상식이 오히려 독이 될 수 있음을 인지하고, 저장 매체의 특성에 부합하는 **물리적 최적화 설계** 역량을 갖춰야 한다."

- **📢 섹션 요약 비유**: 쿼리 튜닝은 '빨리 뛰는 연습'이고, 모델링은 '내리막길(성능 가속 구조)을 만드는 것'입니다. 길이 좋으면 누구나 빨리 달릴 수 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (성능 기반 모델링 시)

| 지표 | 설계 미흡 시 | 설계 최적화 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **물리적 I/O 발생량** | 100% | **30% 이하** | 리소스 사용량 70% 절감 |
| **최대 동시 접속자** | 1,000명 (지연 발생) | **10,000명 이상** | 시스템 확장성 10배 향상 |
| **운영 유지보수 비용** | 잦은 긴급 튜닝 필요 | **안정적 구조 유지** | 관리 TCO 대폭 절감 |

### 2. 미래 전망
- **Autonomous Modeling for Performance**: 실시간 쿼리 워크로드를 분석하여 최적의 성능을 내는 스키마로 스스로 리팩토링하는 **AI 자율 운영 DB** 기술의 진화.
- **Hardware-Aware Modeling**: SSD의 병렬성과 NVM의 비휘발성을 고려하여, 소프트웨어 계층의 모델링이 하드웨어 물리 구조와 일체화되는 **하드웨어 가속형 모델링** 트렌드 확산.

- **📢 섹션 요약 비유**: 미래의 성능 모델링은 굳어있는 조각이 아니라, 데이터의 흐름에 따라 스스로 최적의 모양을 갖추는 '변신하는 유기체'가 될 것입니다.

1. **데이터 모델링과 성능**은 장난감 상자를 얼마나 **"똑똑하게 배치하느냐"**의 문제예요.
2. 자주 가지고 노는 장난감을 문 앞에 두면(반정규화), 매번 장난감 방 깊숙이 들어가지 않아도 되어 **시간이 아주 많이 절약**되죠.
3. 하지만 상자를 너무 크게 만들면 무거워서 들기 힘들듯이, **적당한 크기로 잘 나누는 것**이 성능의 비결이랍니다!
