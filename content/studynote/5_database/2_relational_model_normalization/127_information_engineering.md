# 127. 정보 공학 방법론 (Information Engineering) - 데이터 중심 개발 (James Martin)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기업의 정보 시스템을 구축할 때 프로세스보다는 **데이터(Data)를 중심**으로 하여, 기업 전체의 비즈니스 전략과 IT를 정렬(Alignment)시키는 체계적인 공학적 방법론이다.
> 2. **James Martin의 철학**: 프로세스는 수시로 변하지만 데이터의 본질적 구조는 변하지 않는다는 원칙 하에, **ISP(정보 전략 계획)**부터 시스템 구축까지 단계별로 데이터를 설계하고 관리한다.
> 3. **가치**: 데이터 기반의 하향식(Top-down) 접근을 통해 시스템 간 중복을 방지하고 상호 운용성을 확보하며, 전사적 통합 정보 시스템 구축의 표준 가이드를 제공한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**정보 공학 방법론 (Information Engineering, IE)**은 1980년대 초 제임스 마틴(James Martin)에 의해 정립된 방법론입니다. 기업 경영 목표를 달성하기 위해 필요한 '데이터'를 먼저 정의하고, 이를 처리하는 '프로세스'를 나중에 설계하는 **데이터 중심(Data-driven)** 접근 방식을 취합니다.

### 2. 💡 비유: 가구 배치 vs 집의 뼈대 설계
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        정보 공학 방법론 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 프로세스 중심 (가구 배치): "소파를 어디 둘까?", "TV는 어디에?"              │
│     - 가구(프로세스)는 유행에 따라 수시로 바뀜. 옮길 때마다 힘듦.                 │
│                                                                             │
│  2. 데이터 중심 (뼈대 설계): "벽은 어디에 세우고, 기둥은 어디에 박을까?"          │
│     - 기둥(데이터)은 한 번 세우면 100년은 감. 가구 배치는 기둥 안에서 자유로움.     │
│                                                                             │
│  → 정보 공학 방법론은 **'변하지 않는 튼튼한 뼈대(데이터)'**를 먼저 세우고,         │
│     그 위에서 업무(프로세스)가 자유롭게 움직이게 만드는 건축 공법입니다!           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **소프트웨어 위기 극복**: 유지보수 비용이 개발 비용을 초과하는 문제 해결을 위해 재사용성이 높은 데이터 구조 설계 필요.
- **전사적 통합 요구**: 부서별로 쪼개진 정보 시스템(Islands of Information)을 하나로 통합하기 위한 공통 분모(Data) 중심의 설계 사상 대두.

- **📢 섹션 요약 비유**: 정보 공학 방법론은 '요리 순서'를 고민하기 전, '가장 좋은 식재료'를 미리 확보하고 관리하는 셰프의 지혜와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 정보 공학 방법론의 4대 핵심 단계

| 단계 명칭 | 약어 | 주요 활동 | 비유 |
|:---:|:---:|:---|:---|
| **정보 전략 계획** | **ISP** | 기업 목표 분석, 전사적 데이터 모델 수립. | 도시 계획 |
| **업무 영역 분석** | **BAA** | 데이터와 프로세스의 상관성 분석 (CRUD 매트릭스). | 구역 설계 |
| **업무 시스템 설계** | **BSD** | 사용자 인터페이스, 논리적 데이터 설계. | 상세 도면 |
| **시스템 구축** | **SC** | 실제 코딩, 물리 DB 생성, 테스트. | 실제 시공 |

### 2. 정보 공학의 데이터-프로세스 정렬 (ASCII)

데이터와 업무 기능이 어떻게 매핑되는지 시각화한다.

```text
    [ Strategy ] ──▶ [ Data Model (Entity) ] ──▶ [ Application (Process) ]
           │                   │                          │
    ┌──────▼───────────────────▼──────────────────────────▼──────┐
    │  [ CRUD Matrix ] : 데이터와 프로세스의 유기적 결합 감시        │
    ├──────────────┬──────────┬──────────┬──────────┬────────────┤
    │  Entity \ Func │  영업    │  물류    │  회계    │  인사      │
    ├──────────────┼──────────┼──────────┼──────────┼────────────┤
    │  고 객        │    C     │    R     │    R     │    -       │
    │  주 문        │    C     │    U     │    R     │    -       │
    │  사 원        │    -     │    -     │    R     │    C       │
    └──────────────┴──────────┴──────────┴──────────┴────────────┘
    * C:Create, R:Read, U:Update, D:Delete (누락이나 중복을 즉각 탐지)
```

### 3. 심층 동작 원리: 하향식(Top-down) 설계
- **원리**: 최고 경영층의 목표(Mission)로부터 시작하여 세부 엔티티와 속성으로 내려가는 하향식 접근.
- **장점**: 지엽적인 문제에 매몰되지 않고, 기업 전체의 비즈니스 가치와 IT가 완벽하게 일치하게 됨.

### 4. 핵심 기술: 상관 분석 (CRUD Matrix)
- 데이터 엔티티와 업무 프로세스를 행과 열로 배치하여, 어떤 업무에서 데이터를 생성(C)하고 소비(R)하는지 전수 조사.
- 데이터의 고립(Silo)을 방지하고 불필요한 프로세스를 제거하는 핵심 도구.

- **📢 섹션 요약 비유**: 하향식 설계는 '왕의 명령(목표)으로부터 백성들의 행동(데이터)까지 일관된 법도(구조)를 세우는 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 정보 공학 방법론 vs 구조적 방법론 vs 객체 지향 방법론

| 비교 항목 | 정보 공학 (IE) | 구조적 방법론 (SA/SD) | 객체 지향 (OO) |
|:---:|:---|:---|:---|
| **중심 축** | **데이터 (Data)** | **프로세스 (Function)** | **객체 (Data+Func)** |
| **접근 방식** | **하향식 (Top-down)** | 하향식 (기능 분해) | 상향식/하향식 혼합 |
| **주요 도구** | ERD, CRUD Matrix | DFD, DD | Class, Sequence Diagram |
| **강점** | 전사 통합 시스템 | 단위 시스템 기능 구현 | 복잡한 로직 재사용성 |

### 2. 과목 융합 관점: 경영 전략 및 소프트웨어 공학
- **경영 전략 융합**: 정보 공학은 마이클 포터의 **가치 사슬(Value Chain)** 분석과 결합하여, 핵심 가치 활동에 필요한 데이터 인프라를 우선적으로 구축하는 전략적 도구가 된다.
- **소프트웨어 공학(SE) 융합**: 정보 공학의 ISP 단계는 **BPR(비즈니스 프로세스 재설계)**의 데이터적 근거를 제공하며, 현대의 엔터프라이즈 아키텍처(EA) 수립의 모태가 되었다.

- **📢 섹션 요약 비유**: 구조적 방법론이 '파이프 흐름'을 설계한다면, 정보 공학은 '저수지의 물 관리'를 설계하고, 객체 지향은 '스스로 헤엄치는 물고기'를 설계합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 전사 통합 데이터 웨어하우스(DW) 구축
- **상황**: 부서별로 데이터가 파편화되어 전사 실적 합산이 불가능한 대기업.
- **판단**: 각 부서의 요구사항을 따로 받으면 또다시 파편화됨.
- **결단**: **James Martin의 정보 공학 방법론 적용**. 먼저 기업 전체의 핵심 엔티티(고객, 상품, 자산)를 정의하는 ISP를 수행하고, 모든 부서가 공유하는 공통 데이터 모델을 수립한 뒤 이를 기반으로 부서별 마트를 구축하여 '전사 싱글 뷰' 실현.

### 2. 안티패턴: 형식적인 CRUD 매트릭스 작성
- **증상**: 설계 막판에 문서 제출용으로 엑셀 칸을 대충 채움.
- **결과**: 데이터 생성 주체가 불분명하여 나중에 고아 데이터(Orphan)가 발생하거나, 여러 시스템에서 동시에 같은 데이터를 생성하여 정합성이 깨지는 사고 발생.
- **조치**: **설계 초기 BAA 단계**에서 실제 업무 규칙을 바탕으로 CRUD 매트릭스를 정교하게 작성하고, 이를 통해 데이터 소유권(Data Ownership)을 명확히 할 것.

### 📢 기술사적 결언
> "정보 공학 방법론은 **'IT의 산업 혁명'**이었다. 주먹구구식 코딩에서 벗어나 '공학적 설계'라는 개념을 데이터 중심으로 안착시켰기 때문이다. 기술사는 유행하는 신기술(Cloud, NoSQL 등) 이전에, 제임스 마틴이 강조한 **'데이터의 영속성'**과 **'비즈니스 전략과의 정렬'**이라는 본질적 가치를 모델링에 녹여내야 한다. 정보 공학의 엄격한 상하 위계 구조가 현대의 애자일과 충돌할 수 있으나, **전사적 데이터 표준과 거버넌스**라는 IE의 유산은 영원히 유효한 아키텍처 원칙임을 잊지 말아야 한다."

- **📢 섹션 요약 비유**: CRUD 매트릭스를 소홀히 하는 것은 '누가 요리를 하고 누가 서빙을 할지 안정하고 주방을 여는 것'과 같아 큰 혼란을 야기합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (정보 공학 준수 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 중복도** | 50% 이상 | **10% 이하** | 저장 공간 및 관리 비용 절감 |
| **시스템 통합 용이성** | 매우 낮음 | **높음 (공통 모델 활용)** | 차세대 전환 비용 40% 절감 |
| **비즈니스 정렬도** | 미흡 (IT 따로 노름) | **완벽 (목표 기반 설계)** | IT 투자 수익률(ROI) 극대화 |

### 2. 미래 전망
- **Modern IE with MSA**: 거대한 IE를 마이크로서비스 단위로 쪼개어, 각 서비스 내에서는 데이터 중심으로 설계하고 서비스 간에는 이벤트로 소통하는 **진화형 정보 공학** 부상.
- **AI-Driven ISP**: 기업의 전략 문서와 과거 데이터를 AI가 학습하여, 최적의 주제 영역과 엔티티 후보를 자동으로 제안하는 **지능형 정보 전략 계획** 기술 발전.

- **📢 섹션 요약 비유**: 미래의 정보 공학은 굳어있는 법전이 아니라, 비즈니스의 변화를 실시간으로 학습하고 데이터의 뿌리를 스스로 뻗어 나가는 '지능형 유기체'가 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **정보 공학 방법론**은 아주 커다란 레고 성을 지을 때, **"바닥에 박을 튼튼한 기둥"**부터 먼저 만드는 거예요.
2. 장난감 방 구조(업무)는 바뀔 수 있어도, 기둥(데이터)이 튼튼하면 **성을 훨씬 더 크고 멋지게** 고칠 수 있거든요.
3. 똑똑한 아저씨 제임스 마틴이 만든 이 방법 덕분에, 복잡한 컴퓨터 세상이 **엉망진창이 되지 않고** 잘 돌아간답니다!
