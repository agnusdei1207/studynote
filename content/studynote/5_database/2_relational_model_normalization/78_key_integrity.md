# 78. 키 무결성 (Key Integrity) - 식별자의 고유성과 최소성 유지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 릴레이션 내에서 튜플을 구별하기 위해 정의된 **키(Key)**가 가져야 할 근본적인 성질인 **유일성(Uniqueness)**과 **최소성(Minimality)**이 데이터의 삽입, 수정, 삭제 과정에서도 변함없이 유지되어야 한다는 원칙이다.
> 2. **수행 역할**: 데이터 중복 유입을 물리적으로 차단하여 시스템 내 데이터 개체들이 서로 섞이거나 모호해지는 것을 방지하며, 데이터 간의 명확한 연결(Relationship)을 가능케 하는 신뢰의 기반을 제공한다.
> 3. **가치**: 개체 무결성의 하위 개념이자 실무적 구현체로서, 잘못된 키 값의 변경이나 중복된 식별자 생성을 엔진 차원에서 거부하여 데이터베이스의 **구조적 견고함**을 담보한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**키 무결성 (Key Integrity)**은 데이터베이스의 키 제약 조건이 항상 만족되는 상태를 말합니다. 특히 후보 키(Candidate Key)와 기본 키(Primary Key)가 가져야 할 고유 식별 능력이 훼손되지 않도록 보장하는 규약입니다.

### 2. 💡 비유: 아파트의 '도어락 비밀번호'와 '관리 대장'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        키 무결성 비유                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 유일성 유지: "옆집과 우리 집의 비밀번호는 반드시 달라야 한다."                │
│     - 만약 비밀번호가 겹치면(키 무결성 파괴), 남의 집에 들어가는 대사고 발생!      │
│                                                                             │
│  2. 최소성 유지: "비밀번호는 문을 여는 데 꼭 필요한 숫자만으로 구성한다."          │
│     - 비밀번호가 너무 길거나(불필요한 속성 포함), 일부만 눌러도 열리면 안 됨.      │
│                                                                             │
│  → 키 무결성은 우리 집(데이터)을 찾고 보호하는 **'유일한 열쇠'**가 복제되거나       │
│     망가지지 않게 관리하는 보안 규칙과 같습니다.                                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 식별 오류 방지**: 동일한 주문 번호가 두 개 생기면 결제와 배송 시스템이 엉망이 됨.
- **물리적 성능 최적화**: 키의 유일성이 깨지면 인덱스 검색 효율이 급감하고, 최악의 경우 무한 루프나 시스템 크래시 유발 가능.

- **📢 섹션 요약 비유**: 키 무결성은 데이터베이스라는 도시에 '주소 중복'이 발생하지 않게 감시하는 '공간 관리의 법도'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 키 무결성 보호의 2대 축

| 항목 | 상세 내용 | 비유 |
|:---:|:---|:---|
| **유일성 보호** | 모든 튜플의 키 값은 유일해야 함. 중복 수정을 시도할 경우 Rollback. | 복제 금지 |
| **최소성 보호** | 설계된 키 속성 중 일부를 임의로 제거하거나 무의미한 속성을 추가할 수 없음. | 규격 준수 |

### 2. 키 무결성 위반 탐지 구조 (ASCII)

데이터 변경 시 키의 유효성을 검사하는 물리적 과정을 시각화한다.

```text
    [ Transaction: Update ID from 1 to 2 ]
           │
    ┌──────▼────────────────── DBMS Storage ──────────────────────────────┐
    │                                                                     │
    │  1. [ Index Lookup ] : 인덱스 트리에서 '2'라는 값이 이미 있는지 확인. │
    │     ┌───────────────────────────────────────────────────────────┐   │
    │     │ [ B+Tree Index ]                                          │   │
    │     │ - Key 1: RID_A                                            │   │
    │     │ - Key 2: RID_B  <─── "이미 존재함!" (Conflict Detected)    │   │
    │     └──────────────┬────────────────────────────────────────────┘   │
    │                    │                                                │
    │  2. [ Exception Raise ] : "Key Integrity Violation" 에러 발생.       │
    │                                                                     │
    └────────────────────┬────────────────────────────────────────────────┘
                         ▼
    ┌──────────────── Transaction Result ────────────────────────────────┐
    │  [ Rollback ] - 이전 상태(ID=1)로 복구하여 무결성 유지                │
    └────────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: 고유 인덱스(Unique Index) 강제
- **물리적 보장**: 사용자가 `UNIQUE`나 `PRIMARY KEY`를 선언하면, DBMS는 **고유 인덱스**를 생성함.
- **잠금(Locking)**: 키 무결성 검사 중에는 해당 키 값에 대한 **인덱스 잠금**을 걸어, 찰나의 순간에 다른 트랜잭션이 동일한 키를 넣지 못하게 원자성을 보장함.

### 4. 핵심 기술: 키 업데이트 정책 (Key Update Policy)
- 기본적으로 기본 키 값은 **업데이트하지 않는 것**이 키 무결성 유지에 가장 유리함.
- 부득이하게 변경 시, 이를 참조하는 모든 외래 키(FK)와의 정합성을 함께 따지는 복합 무결성 검사가 수행됨.

- **📢 섹션 요약 비유**: 키 무결성 검사는 '새로운 차가 주차장에 들어올 때, 이미 등록된 번호판인지 입구에서 즉시 대조하는 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 키 무결성 vs 도메인 무결성 vs 개체 무결성

| 비교 항목 | 키 무결성 (Key) | 도메인 무결성 (Domain) | 개체 무결성 (Entity) |
|:---:|:---|:---|:---|
| **초점** | **식별 능력의 유지** | 값의 범위와 형식 | 기본 키의 존재(NOT NULL) |
| **핵심 규칙** | **중복 금지** | 유효값 준수 | 빈칸 금지 + 중복 금지 |
| **범위** | 키로 지정된 속성들 | 전체 속성 | 오직 기본 키(PK) |
| **관계** | 개체 무결성의 실무적 수단 | 데이터 품질의 기초 | 무결성 체계의 최상위 원칙 |

### 2. 과목 융합 관점: 이산수학 및 알고리즘
- **이산수학 융합**: 키 무결성은 함수의 **일대일 대응(Injective Function)** 성질을 집합 간 매핑에서 강제하는 논리다.
- **알고리즘 융합**: 키 무결성 검사는 **해시 충돌(Hash Collision)** 방지 알고리즘 및 B-Tree의 **Unique Key Path** 탐색 최적화 기술과 직접적으로 연동된다.

- **📢 섹션 요약 비유**: 도메인이 '재료의 신선도'라면, 키 무결성은 '재료마다 붙은 고유한 바코드의 정확성'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 동시성 높은 환경에서의 키 중복 사고
- **상황**: 두 명의 사용자가 동시에 같은 아이디로 가입 버튼을 누름.
- **판단**: 애플리케이션의 `SELECT` 후 `INSERT` 로직으로는 0.001초 차이의 중복을 막을 수 없음.
- **결단**: **데이터베이스의 키 무결성(Unique Constraint)에 전적으로 의존**. 애플리케이션 코드는 중복 에러가 발생했을 때 사용자에게 친절한 메시지를 보여주는 예외 처리에만 집중.

### 2. 안티패턴: 대량 데이터 로드 시의 인덱스(키 무결성) 무시
- **증상**: 속도를 위해 모든 제약 조건과 인덱스를 제거하고 데이터를 밀어 넣음.
- **결과**: 입력은 빠르지만, 나중에 인덱스를 다시 만들려 할 때 중복 데이터가 발견되어 인덱스 생성 실패 및 데이터 전수 조사 발생.
- **조치**: **입력 전 데이터의 유일성을 애플리케이션에서 1차 검증**하거나, 배치 로드 전용 정렬 기능을 사용하여 키 무결성을 논리적으로 먼저 확보할 것.

### 📢 기술사적 결언
> "키 무결성은 데이터베이스의 **'최소한의 자존심'**이다. 키가 무너진 DB는 엑셀 파일보다 못하다. 기술사는 단순히 유일성 제약을 거는 데 그치지 않고, **인덱스 파편화**와 **키 변경의 비용**이 시스템 전체 성능에 미치는 파급력을 계산해야 한다. 특히 분산 시스템에서는 **결과적 일관성(Eventual Consistency)** 환경에서도 키 무결성을 어떻게 보장할 것인가라는 고차원적 정합성 전략을 수립해야 한다."

- **📢 섹션 요약 비유**: 제약 조건 없이 데이터를 넣는 것은 '검문 없이 난민을 수용하는 것'과 같아, 나중에 누가 누구인지 파악하는 데 엄청난 비용이 듭니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (키 무결성 엄격 준수 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **중복 데이터 발생** | 빈번 (1~2%) | **0% (완벽 차단)** | 데이터 정합성 100% 확보 |
| **참조 무결성 오류** | 연쇄 발생 | **원천 방어** | 데이터 간 연결성 신뢰도 향상 |
| **운영 관리 비용** | 사후 보정 비용 과다 | **시스템 자동 관리** | 유지보수 공수 40% 절감 |

### 2. 미래 전망
- **Blockchain-based Key Integrity**: 분산 환경에서 키의 유일성을 수학적 증명(Zero-knowledge Proof 등)을 통해 보장하고, 위변조가 불가능한 키 무결성 체계 구축.
- **Self-remediating Integrity**: 키 중복이 의심되는 데이터 유입 시, AI가 과거 이력을 분석하여 자동으로 키를 보정하거나 식별자를 재부여하는 자율 치유형 무결성 엔진 등장.

- **📢 섹션 요약 비유**: 미래의 키 무결성은 단순한 자물쇠를 넘어, 데이터의 탄생부터 소멸까지를 감시하는 '지능형 지문 시스템'으로 진화할 것입니다.
