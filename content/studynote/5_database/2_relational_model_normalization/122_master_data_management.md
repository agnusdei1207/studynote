# 122. 마스터 데이터 관리 (MDM, Master Data Management) - 전사 기준 정보 통합

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기업의 여러 비즈니스 프로세스에서 공통으로 사용되는 핵심적인 기준 정보(고객, 상품, 자산, 장소 등)를 하나의 통합된 저장소에서 중앙 집중적으로 관리하여 **단일 진실 공급원(Single Source of Truth, SSOT)**을 구축하는 기술 및 프로세스다.
> 2. **해결 과제**: 시스템별로 흩어져 서로 중복되고 불일치하는 마스터 데이터의 파편화 현상을 해결하며, 데이터의 중복 제거(De-duplication), 표준화, 동기화를 통해 전사적 데이터 정합성을 확보한다.
> 3. **가치**: 데이터 품질 향상을 통해 분석의 정확도를 높이고, IT 시스템 간의 연동 비용을 절감하며, 규제 준수(Compliance) 및 비즈니스 변화에 대한 민첩한 대응을 가능케 하는 데이터 거버넌스의 핵심 요소다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**마스터 데이터 관리 (MDM)**는 기업 내에 산재한 마스터 데이터를 통합하고 정제하여 일관된 정보를 전 시스템에 배포하는 체계입니다. 트랜잭션 데이터(주문, 결제 등)의 근간이 되는 '명사(Noun)' 중심의 정보를 다룹니다.

### 2. 💡 비유: 가전제품 서비스 센터의 '고객 장부'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        MDM 개념 비유                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 한 고객이 에어컨 수리도 맡기고, 냉장고도 새로 샀음.                       │
│                                                                             │
│  1. MDM 미사용:                                                              │
│     - 수리팀 장부: "홍길동(서울시 강남구)"                                      │
│     - 판매팀 장부: "홍길동(서울시 서초구 - 이사 감)"                             │
│     - 두 장부의 주소가 달라 나중에 경품을 보낼 때 어디로 보낼지 헷갈림.             │
│                                                                             │
│  2. MDM 사용:                                                                │
│     - **'중앙 통합 장부'** 하나만 관리함.                                       │
│     - 판매팀이 주소를 바꾸면 수리팀 장부도 자동으로 바뀜! (싱글 뷰)                │
│                                                                             │
│  → 모든 부서가 **'똑같은 최신 정보'**를 보고 일하게 만드는 것이 MDM입니다!         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **시스템 사일로(Silo) 심화**: ERP, CRM, SCM 등 개별 시스템 도입으로 인해 시스템마다 같은 고객의 정보가 다르게 저장됨.
- **분석 신뢰도 하락**: "우리 회사 고객이 총 몇 명인가?"라는 질문에 시스템마다 답변이 다른 상황 발생.

- **📢 섹션 요약 비유**: MDM은 기업이라는 유기체가 공유하는 '단 하나의 뇌(기억)'를 만드는 작업입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 마스터 데이터의 주요 대상 분류

| 분류 | 상세 내용 | 예시 |
|:---:|:---|:---|
| **고객 (Party)** | 비즈니스 거래 주체. | 개인고객, 법인, 협력사, 임직원 |
| **상품 (Product)** | 판매 또는 관리의 대상. | 완제품, 원자재, 서비스, 부품 |
| **장소 (Location)** | 물리적/논리적 거점. | 매장, 창고, 공장, 지사 주소 |
| **자산 (Asset)** | 기업의 소유물. | 설비, 차량, 지식재산권, IT 장비 |

### 2. MDM 시스템 통합 아키텍처 (ASCII)

여러 소스 시스템에서 데이터를 수집하여 통합하고 배포하는 허브 구조를 시각화한다.

```text
    [ Source Systems ]       [ MDM Hub ]              [ Downstream ]
    ┌──────────┐             ┌────────────────────┐             ┌──────────┐
    │ CRM (고객)│ ──(Extract)▶│ 1. [ Data Cleansing]│             │ BI / DW  │
    └──────────┘             │    (정제 및 표준화)   │ ──(Publish)▶└──────────┘
    ┌──────────┐             │ 2. [ Matching / DL ]│             ┌──────────┐
    │ ERP (상품)│ ──(Extract)▶│    (중복 제거/매칭)  │ ──(Push)───▶│ SCM (물류)│
    └──────────┘             │ 3. [ Master Record ]│             └──────────┘
    ┌──────────┐             │    (황금 레코드 생성) │             ┌──────────┐
    │ Legacy   │ ──(Extract)▶│ 4. [ Governance ]   │ ──(Sync)───▶│ Web/App  │
    └──────────┘             └────────────────────┘             └──────────┘
```

### 3. 심층 동작 원리: 골든 레코드 (Golden Record) 생성
- **매칭(Matching)**: 이름, 연락처, 주소가 유사한 데이터들을 확률적/결정적 알고리즘으로 비교.
- **머징(Merging)**: 여러 데이터 중 가장 최신의, 가장 정확한 속성값들만 뽑아 하나의 완벽한 레코드를 조립.
- **서바이버십(Survivorship)**: 어떤 시스템의 데이터를 우선할지 규칙(Rule)에 따라 결정.

### 4. 핵심 기술: 데이터 식별자 매핑 (Cross-Reference)
- 각 시스템마다 다른 PK(`CRM_ID`, `ERP_ID`)를 MDM의 `GLOBAL_MASTER_ID`와 연결하는 매핑 테이블 관리.

- **📢 섹션 요약 비유**: 골든 레코드는 '여러 명의 목격자 진술을 합쳐 그린 단 한 장의 정확한 몽타주'와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. MDM vs DW (데이터 웨어하우스)

| 비교 항목 | MDM (기준 정보 관리) | DW (데이터 분석 저장소) |
|:---:|:---|:---|
| **데이터 성격** | **명사 (Who, What)** | **동사 (How much, When)** |
| **주요 작업** | **실시간 동기화 / 정제** | 과거 이력 적재 / 통계 |
| **방향성** | 소스 시스템으로 재배포 (양방향) | 분석을 위한 수집 (단방향) |
| **사용 시점** | **트랜잭션 실행 중** | 트랜잭션 종료 후 (사후 분석) |

### 2. 과목 융합 관점: 소프트웨어 공학 및 보안
- **소프트웨어 공학(SE) 융합**: MDM은 MSA(마이크로서비스 아키텍처)에서 각 서비스가 공통으로 참조하는 **공통 서비스(Shared Service)**나 **캐시 레이어**의 논리적 기반이 된다.
- **보안(Security) 융합**: 개인정보 보호법(GDPR 등) 대응을 위해 산재한 고객 정보를 한곳에서 통제(잊힐 권리 행사 등)하는 데 MDM 체계가 필수적이다.

- **📢 섹션 요약 비유**: MDM이 '표준 규격 나사'를 만드는 공장이라면, DW는 그 나사로 만든 '기계들의 실적'을 기록하는 일기장입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 제조 대기업의 전사 상품 코드 통합
- **상황**: 본사와 해외 공장의 상품 코드가 달라 재고 파악 불능.
- **판단**: 단순히 DB를 합치는 것은 불가능하며, 각자의 시스템 코드를 유지하면서 '통합 코드'가 필요함.
- **결단**: **MDM 허브 구축**. 전 세계 모든 공장의 상품 정보를 MDM으로 수집하여 중복을 제거한 뒤 'Global Single SKU'를 발급. 각 시스템은 기존 코드를 쓰되, 내부적으로 MDM ID를 참조하게 하여 실시간 전 세계 재고 가시성 확보.

### 2. 안티패턴: 기술 도구에만 의존하는 MDM 구축
- **증상**: 비싼 MDM 솔루션만 도입하면 데이터가 저절로 정제될 것이라 기대.
- **결과**: "고객명"이 누구는 '삼성', 누구는 '삼성전자'인 비즈니스 정체성 문제를 기술이 해결해주지 못해 시스템 도입 실패.
- **조치**: 솔루션 도입 전, **데이터 관리 위원회**를 구성하여 표준 용어와 정제 규칙을 정하는 **데이터 거버넌스(Governance)** 프로세스를 먼저 수립할 것.

### 📢 기술사적 결언
> "MDM은 **'기업의 디지털 자아'**를 찾는 과정이다. 우리 회사가 다루는 고객과 상품이 정확히 누구이고 무엇인지 정의하지 못하는 IT는 사상누각이다. 기술사는 단순히 마스터 데이터를 동기화하는 엔지니어를 넘어, **비즈니스 아키텍처와 데이터 아키텍처를 일치시키는 거버넌스 전문가**가 되어야 한다. MDM은 구축보다 **운영과 품질 유지**가 수백 배 어려우며, 이를 위해 조직적인 문화와 프로세스 혁신이 동반되어야 함을 명심해야 한다."

- **📢 섹션 요약 비유**: 거버넌스 없는 MDM 솔루션은 '글씨를 못 쓰는 사람에게 비싼 만년필을 사주는 것'과 같아, 도구의 가치를 발휘할 수 없습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (MDM 성공 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 불일치 해소 시간** | 주 단위 (수작업) | **실시간 / 즉시** | 업무 생산성 기하급수적 향상 |
| **마케팅 타겟 정확도** | 낮음 (중복 발송) | **높음 (Single View)** | 마케팅 ROI 30% 이상 증가 |
| **데이터 정제 비용** | 상시 발생 (비효율) | **원천 관리 (저비용)** | IT 운영 비용 절감 |

### 2. 미래 전망
- **Cloud-Native MDM**: SaaS 기반의 MDM이 보편화되어 중소기업도 저비용으로 전역 마스터 데이터를 관리하는 시대 도래.
- **Graph-based MDM**: 단순 속성 매칭을 넘어, 데이터 간의 복잡한 연결 고리(가족, 지인, 법인 관계 등)를 그래프 DB로 관리하여 통찰력을 높이는 **관계 중심 MDM** 진화.

- **📢 섹션 요약 비유**: 미래의 MDM은 고정된 데이터베이스를 넘어, 전 세계에 흩어진 데이터들 사이에서 '진실'만을 골라내는 '지능형 필터'가 될 것입니다.
