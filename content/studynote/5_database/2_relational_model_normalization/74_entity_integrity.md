# 74. 개체 무결성 (Entity Integrity) - 기본 키의 유일성과 비널성 보장

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 관계형 데이터베이스의 모든 릴레이션에서 **기본 키(Primary Key)**로 지정된 속성은 어떠한 경우에도 **NULL 값을 가질 수 없으며**, 릴레이션 내에서 반드시 **유일(Unique)**해야 한다는 규칙이다.
> 2. **존재의 이유**: 데이터베이스 내의 각 튜플(행)을 명확하게 식별하고 추적할 수 있게 함으로써, 현실 세계의 개체가 시스템 상에서 모호함 없이 단 하나로 존재함을 기술적으로 증명한다.
> 3. **가치**: 무결성 체계의 가장 기초가 되는 '제1원칙'이며, 개체 무결성이 무너지면 데이터 간의 참조나 업데이트 시 대상이 특정되지 않아 시스템 전체의 신뢰성이 붕괴된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**개체 무결성 (Entity Integrity)**은 릴레이션의 기본 키를 구성하는 그 어떤 속성값도 NULL 일 수 없으며, 중복될 수 없음을 규정하는 제약 조건입니다. 이는 개체를 식별하는 '최소한의 증명'입니다.

### 2. 💡 비유: 국민의 '주민등록번호'와 '생존 증명'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        개체 무결성 비유                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 국가 행정 시스템에 국민 정보를 등록함.                                   │
│                                                                             │
│  1. 규칙 1 (NOT NULL): "번호 없는 국민은 없다."                                │
│     - 이름만 있고 주민번호가 비어있으면(NULL), 그 사람이 실재하는지 증명 불가.       │
│                                                                             │
│  2. 규칙 2 (Unique): "똑같은 번호를 가진 사람은 없다."                          │
│     - 번호가 겹치면 세금을 누구에게 걷어야 할지, 투표를 누가 했는지 꼬임.           │
│                                                                             │
│  → 개체 무결성은 데이터라는 시민들이 각자 **'확실한 이름표'**를 달고 있게           │
│     강제하는 법과 같습니다.                                                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **식별 불가능의 공포**: 데이터가 수백만 건인데 기본 키가 비어있거나 중복되면, 특정 행을 수정(Update)하거나 삭제(Delete)할 때 엉뚱한 데이터가 처리되는 사고 발생.
- **집합론적 완결성**: 수학적 집합의 원소는 서로 구별 가능해야 한다는 원리를 관계형 모델에 물리적으로 이식.

- **📢 섹션 요약 비유**: 개체 무결성은 '신분증 없는 사람의 입국을 금지하고, 위조 신분증을 잡아내는 출입국 관리소'와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 개체 무결성의 2대 강제 조건

| 제약 내용 | 상세 설명 | 비유 |
|:---:|:---|:---|
| **NOT NULL** | 기본 키의 어떤 부분도 비어있을 수 없음. | 빈칸 제출 금지 |
| **Unique** | 동일한 기본 키 값을 가진 튜플이 존재할 수 없음. | 복제 인간 금지 |

### 2. 개체 무결성 준수 vs 위반 사례 (ASCII)

테이블 수준에서 규칙이 어떻게 작동하는지 시각화한다.

```text
    [ Relation: Student (PK: SID) ]
    
    (A) 준수 사례: 정상 등록
    ┌──────┬──────┬──────┐
    │ SID  │ Name │ Dept │
    ├──────┼──────┼──────┤
    │ 101  │ Kim  │ CS   │
    │ 102  │ Lee  │ IT   │
    └──────┴──────┴──────┘
    
    (B) 위반 사례: DBMS에서 Reject 발생
    ┌──────┬──────┬──────┐
    │ SID  │ Name │ Dept │
    ├──────┼──────┼──────┤
    │ 101  │ Park │ Math │ <--- 위반! (SID '101' 중복)
    │ NULL │ Choi │ Eng  │ <--- 위반! (기본 키에 NULL 입력)
    └──────┴──────┴──────┘
```

### 3. 심층 동작 원리: 기본 키와 인덱스의 결합
- **메커니즘**: 사용자가 `PRIMARY KEY`를 선언하면, DBMS는 내부적으로 해당 컬럼에 대해 **Unique Index**를 자동으로 생성함.
- **검사 과정**: 새로운 데이터 삽입 시, 인덱스(B-Tree 등)를 고속 탐색하여 동일 값이 있는지 $O(\log N)$ 속도로 체크하고, NULL 인지도 즉시 판별함.

### 4. 핵심 기술: 복합 기본 키에서의 개체 무결성
- 여러 속성을 묶어 PK를 만든 경우, **구성 속성 중 단 하나라도 NULL이 되어서는 안 됨**.
- 예: {학번, 수강년도}가 PK일 때, 학번만 적고 수강년도를 안 적는 것은 개체 무결성 위반임.

- **📢 섹션 요약 비유**: 복합 키에서의 개체 무결성은 '아이디와 비밀번호가 모두 있어야 로그인이 되는 것'과 같아, 둘 중 하나라도 없으면 신원을 인정받지 못합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 개체 무결성(Entity) vs 참조 무결성(Referential)

| 비교 항목 | 개체 무결성 (Entity) | 참조 무결성 (Referential) |
|:---:|:---|:---|
| **적용 범위** | **단일 릴레이션 내부** | **두 릴레이션 간의 관계** |
| **핵심 키** | **기본 키 (PK)** | **외래 키 (FK)** |
| **NULL 허용** | **절대 불가** | **조건부 허용** (부모 없을 시) |
| **목표** | 데이터의 **존재 증명** | 데이터의 **연결 보장** |

### 2. 과목 융합 관점: 자료구조 및 보안
- **자료구조 융합**: 개체 무결성은 해시 테이블의 **Key 충돌 방지** 원리와 맥을 같이 하며, 물리적으로는 **클러스터드 인덱스**의 리프 노드 유일성을 보장한다.
- **보안(Security) 융합**: 기본 키의 중복 방지는 **재전송 공격(Replay Attack)**이나 중복 결제 사고를 막는 애플리케이션 보안의 최하단 방어선 역할을 한다.

- **📢 섹션 요약 비유**: 개체 무결성이 '나 자신의 건강'이라면, 참조 무결성은 '친구와의 신뢰 관계'입니다. 내가 건강해야 친구도 사귈 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 레거시 데이터 통합 시의 무결성 충돌
- **상황**: 두 회사의 고객 데이터를 합치려는데, 양쪽의 고객번호(PK)가 겹침.
- **판단**: 그대로 합치면 개체 무결성 위반으로 데이터가 입력되지 않음.
- **결단**: **새로운 인조 키(Surrogate Key)를 생성하여 통합 PK로 설정**. 기존 PK는 '이전시스템ID'라는 일반 속성으로 보존함으로써 개체 무결성을 우회적으로 해결하고 데이터 정합성을 확보함.

### 2. 안티패턴: 기본 키 없이 테이블 생성
- **증상**: "빨리 개발해야 하니까"라며 PK 없이 테이블만 만들고 데이터를 쌓음.
- **결과**: 나중에 특정 행을 지우려고 해도 같은 데이터가 여러 개 있어 지울 수 없거나, 조인 성능이 기하급수적으로 느려져 시스템 전체가 마비됨.
- **조치**: **모든 테이블에는 반드시 PK를 정의**해야 하며, 비즈니스 키가 마땅치 않다면 Sequence나 Identity 컬럼이라도 반드시 추가할 것.

### 📢 기술사적 결언
> "개체 무결성은 데이터베이스 세계의 **'존재론적 정당성'**이다. 존재가 증명되지 않은 데이터는 유령과 같다. 기술사는 화려한 튜닝 기술 이전에, 설계된 모든 릴레이션이 **완벽한 개체 무결성**을 갖추고 있는지부터 점검해야 한다. 특히 분산 환경에서는 노드 간의 PK 중복을 막는 것이 시스템의 성패를 가르므로, **글로벌 유일 식별자(GUID/UUID)** 전략을 수립하는 데 심혈을 기울여야 한다."

- **📢 섹션 요약 비유**: PK 없는 테이블은 '주소 없는 집들의 도시'와 같아, 아무리 편지를 보내도(쿼리 질의) 배달되지 않거나 엉뚱한 곳에 전달됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (개체 무결성 확보 시)

| 지표 | 도입 전 (파일 기반) | 도입 후 (PK 기반) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 중복율** | 약 5~10% | **0% (Engine 차단)** | 데이터 정합성 100% 확보 |
| **개별 행 접근 속도** | 수 초 (Full Scan) | **수 ms (Index 활용)** | 검색 성능 수천 배 향상 |
| **운영 신뢰도** | 낮음 (데이터 의구심) | **매우 높음** | 데이터 기반 의사결정 가능 |

### 2. 미래 전망
- **Self-Generating PK**: 개발자가 신경 쓰지 않아도 DB 엔진이나 클라우드 계층에서 최적의 물리 구조를 가진 PK를 자동으로 생성하고 관리하는 지능형 스키마 기술 등장.
- **Biometric Entity Integrity**: 사물인터넷(IoT) 기기나 생체 정보를 직접 PK로 변환하여, 물리적 실체와 데이터 실체를 일치시키는 초정밀 무결성 체계 발전.

- **📢 섹션 요약 비유**: 기술이 아무리 변해도 '너는 누구냐?'라는 질문에 명확히 답할 수 있는 개체 무결성의 원칙은 정보 사회의 근간으로 남을 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **개체 무결성**은 우리 반 모든 친구가 **"이름표를 꼭 달고 있어야 하고, 똑같은 이름표는 없어야 한다"**는 약속이에요.
2. 이름표가 없으면 선생님이 누구인지 모르고, 이름표가 겹치면 엉뚱한 사람에게 선물을 줄 수도 있기 때문이죠.
3. 이 약속 덕분에 컴퓨터는 우리들을 **헷갈리지 않고 정확하게** 찾아낼 수 있답니다!
