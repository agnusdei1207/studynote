# 86. 순환 관계 (Recursive Relationship) - 자기 자신을 참조하는 관계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동일한 개체 타입(Entity Type) 내에서 서로 다른 개체 인스턴스들끼리 관계를 맺는 특수한 형태이며, 논리적으로는 **자기 자신을 참조**하는 관계다.
> 2. **용도**: 조직의 상하 관계(팀장-팀원), 부품의 포함 관계(BOM: Bill of Materials), 소셜 네트워크의 팔로워 관계 등 **계층 구조나 네트워크 구조**를 표현할 때 필수적으로 사용된다.
> 3. **가치**: 불필요하게 유사한 개체를 중복 생성하지 않고도 데이터 간의 무한한 깊이를 가진 연결성을 확보하게 해주며, 물리적으로는 테이블 내의 **Self-Join**을 통해 구현된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**순환 관계 (Recursive Relationship)**는 하나의 개체가 자기 자신과 관계를 맺는 것입니다. ERD에서는 개체에서 나온 선이 다시 동일한 개체로 돌아가는 형태로 그려집니다. 이를 **셀프 참조(Self-reference)**라고도 합니다.

### 2. 💡 비유: 가계도와 '부모-자식' 관계
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        순환 관계 개념 비유                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 가문의 족보를 만들고 있음.                                              │
│                                                                             │
│  1. 개체: 오직 '사람' 하나뿐임.                                               │
│  2. 관계: "사람"은 다른 "사람"의 '부모'이거나 '자식'임.                          │
│                                                                             │
│  - 할아버지도 사람, 아버지도 사람, 나도 사람입니다.                              │
│  - "사람" 테이블 하나만 있으면, 그 안에서 누가 누구의 부모인지 화살표만           │
│    연결(순환 관계)하면 수천 년의 가계도를 다 그릴 수 있습니다!                   │
│                                                                             │
│  → 새로운 종류의 데이터(개체)를 만들지 않고도 **'나와 같은 존재'**들 사이의        │
│     연결을 표현하는 것이 순환 관계입니다.                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **계층적 구조의 정형화**: 회사 조직도처럼 끝이 정해지지 않은 상하 관계를 하나의 테이블 구조로 효율적으로 담기 위함.
- **부품 구조 관리(BOM)**: 자동차라는 제품 밑에 엔진이 있고, 엔진 밑에 실린더가 있는 '포함의 연쇄'를 표현하기 위해 도입.

- **📢 섹션 요약 비유**: 순환 관계는 '거울 속에 비친 나와 악수하는 것'과 같아서, 같은 부류끼리 서로 돕고 엮이는 관계를 말합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 순환 관계의 역할(Role) 정의

순환 관계에서는 참여하는 개체가 동일하므로, 각자가 수행하는 **역할(Role)**을 명시하는 것이 매우 중요합니다.

| 예시 상황 | 역할 A (Parent Side) | 역할 B (Child Side) | 비유 |
|:---:|:---|:---|:---|
| **조직도** | 관리자 (Manager) | 부하직원 (Subordinate) | 지시와 보고 |
| **BOM** | 모부품 (Parent Part) | 자식부품 (Component) | 조립과 분해 |
| **SNS** | 팔로잉 대상 (Followee) | 팔로워 (Follower) | 영향력 전파 |

### 2. 순환 관계 도해 (ASCII)

ERD 표기법과 물리적 테이블 내의 연결 구조를 시각화한다.

```text
    [ ERD Representation ]
    
       ┌───────────┐
       │   사 원    │◀─────┐
       └─────┬─────┘      │
             │            │ (1:N Recursive)
          ┌──▼──┐         │ "관리한다"
          │관리자│─────────┘
          └─────┘
    
    [ Physical Table: Employee ]
    ┌────┬─────┬──────────┐
    │ EID│ Name│ Mgr_EID  │ <--- (FK가 본인 테이블의 PK를 참조)
    ├────┼─────┼──────────┤
    │ 01 │ Kim │   NULL   │ (사장님)
    │ 02 │ Lee │    01    │ (Kim이 Lee의 관리자)
    │ 03 │ Park│    01    │ (Kim이 Park의 관리자)
    │ 04 │ Choi│    02    │ (Lee가 Choi의 관리자)
    └────┴─────┴──────────┘
```

### 3. 심층 동작 원리: 셀프 조인 (Self-Join)
- **원리**: 쿼리 시 동일한 테이블을 가상의 별칭(Alias)으로 두 번 호출하여 결합함.
- **예시**: `SELECT a.Name as 사원, b.Name as 팀장 FROM Emp a, Emp b WHERE a.Mgr_ID = b.EID;`
- **효과**: 한 번의 쿼리로 계층의 한 단계를 타고 올라가거나 내려갈 수 있음.

### 4. 핵심 기술: 계층형 쿼리 (Hierarchical Query)
- **문제**: 순환 관계가 깊어지면(예: 증손자까지) 일반 조인으로는 한계가 있음.
- **해결**: Oracle의 `CONNECT BY`나 표준 SQL의 **Recursive CTE (WITH RECURSIVE)**를 사용하여 트리를 탐색하듯 데이터를 순회함.

- **📢 섹션 요약 비유**: 셀프 조인은 '내 왼쪽 주머니에서 꺼낸 명함을 오른쪽 주머니에 있는 사람 명단과 대조해보는 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 일반 관계 vs 순환 관계

| 비교 항목 | 일반 관계 (Binary) | 순환 관계 (Recursive) |
|:---:|:---|:---|
| **참여 개체 수** | **2개 이상** (예: 학생, 과목) | **오직 1개** (예: 사원) |
| **구조적 형태** | 수평적 연결 | **수직적 / 계층적 연결** |
| **FK 위치** | 상대방 테이블 | **자기 자신 테이블** |
| **데이터 탐색** | 단순 조인 | 재귀 탐색 (Recursion) |

### 2. 과목 융합 관점: 자료구조 및 알고리즘
- **자료구조 융합**: 순환 관계는 전산학의 **트리(Tree)** 및 **유향 비순환 그래프(DAG)**를 관계형 데이터베이스로 구현한 실체다.
- **알고리즘 융합**: 순환 관계 데이터를 처리하기 위해 **깊이 우선 탐색(DFS)**이나 **너비 우선 탐색(BFS)** 로직이 DB 엔진 내부에 탑재되어 작동한다.

- **📢 섹션 요약 비유**: 일반 관계가 '친구 사이의 악수'라면, 순환 관계는 '러시아 인형(마트료시카)처럼 자기 안에 자기를 품고 있는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 카테고리 분류 체계 설계
- **상황**: 대분류-중분류-소분류가 무한히 깊어질 수 있는 상품 카테고리를 설계함.
- **판단**: 각 분류를 테이블로 만들면 확장이 불가능함.
- **결단**: **순환 관계를 적용한 단일 'Category' 테이블 설계**. `parent_id` 속성을 통해 상위 카테고리를 무한히 참조하게 함으로써, 비즈니스가 성장해도 스키마 수정 없이 분류를 추가할 수 있는 유연성 확보.

### 2. 안티패턴: 순환 참조의 무한 루프 (Cycle)
- **증상**: A가 B의 팀장인데, 실수로 B를 A의 팀장으로 등록함.
- **결과**: 계층형 쿼리 수행 시 끝이 나지 않는 무한 루프에 빠져 서버 CPU가 폭주함.
- **조치**: 애플리케이션이나 DB 제약 조건(Trigger)을 통해 **사이클(Cycle) 발생 여부를 사전에 검증**하는 로직 결착.

### 📢 기술사적 결언
> "순환 관계는 **'단순함으로 복잡함을 다스리는 모델링의 정수'**다. 단 하나의 테이블 구조로 무한한 계층을 표현할 수 있기 때문이다. 기술사는 순환 관계를 설계할 때 반드시 **루트 노드(Root Node)의 NULL 처리**와 **데이터 순환(Cycle) 방지** 전략을 수립해야 한다. 또한, 성능 최적화를 위해 **경로 정보(Path)**를 미리 저장하는 방식(Materialized Path)과 순수 재귀 쿼리 사이의 트레이드오프를 판단할 줄 알아야 한다."

- **📢 섹션 요약 비유**: 사이클 사고는 '꼬리에 꼬리를 무는 뱀'과 같아서, 길을 잃고 영원히 제자리걸음을 하게 만듭니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (순환 관계 활용 시)

| 지표 | 도입 전 (계층별 분리) | 도입 후 (순환 관계) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **스키마 복잡도** | 테이블 다수 발생 | **단 1개로 통합** | 설계 생산성 획기적 향상 |
| **데이터 확장성** | 계층 깊이 제한됨 | **무한 깊이 지원** | 비즈니스 유연성 극대화 |
| **유지보수 비용** | 코드 수정 잦음 | **쿼리 하나로 해결** | 운영 비용 절감 |

### 2. 미래 전망
- **Graph DB로의 전이**: 순환 관계가 극도로 복잡한(네트워크형) 데이터는 관계형의 한계를 넘어 **네오포제이(Neo4j)** 같은 그래프 DB로 관리하는 아키텍처가 확산될 것임.
- **Auto-Hierarchization**: AI가 산재한 개체들 사이의 관계를 분석하여 스스로 순환 관계를 제안하고 계층 구조를 최적화하는 기능 고도화.

- **📢 섹션 요약 비유**: 미래에는 데이터 스스로가 자신의 뿌리를 찾아가는 '지능형 계층망'이 구축될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **순환 관계**는 장난감 기차들이 **"자기랑 똑같은 기차 친구"**랑 손을 잡는 거예요.
2. 기차 한 칸이 다른 한 칸을 끌어주고, 그 친구가 또 다른 친구를 끌어주면 **아주아주 긴 기차**가 되죠.
3. 이 약속만 잘 지키면 기차 칸이 아무리 많아져도 **절대 끊어지지 않고** 멀리 갈 수 있답니다!
