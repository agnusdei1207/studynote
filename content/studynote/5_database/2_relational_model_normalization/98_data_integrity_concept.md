# 98. 데이터 무결성 (Data Integrity) - 일관성, 정확성, 유효성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스에 저장된 데이터가 현실 세계의 실제 값과 일치하며, 어떠한 논리적 모순도 없이 **정확성(Accuracy), 일관성(Consistency), 유효성(Validity)**을 유지하는 상태다.
> 2. **수행 체계**: 데이터의 입력, 수정, 삭제 시점에 DBMS 엔진이 미리 정의된 **무결성 제약 조건**을 실시간 검증함으로써, 부적절한 데이터가 저장소에 침투하는 것을 원천 차단한다.
> 3. **가치**: 정보 시스템의 신뢰도를 결정짓는 최후의 보루이며, 무결성이 확보된 데이터만이 비즈니스 의사결정의 올바른 근거(Single Source of Truth)가 될 수 있다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**데이터 무결성 (Data Integrity)**은 데이터베이스의 생명과도 같습니다. 데이터가 전 생애주기 동안 원래의 형태를 유지하고, 비즈니스 규칙을 완벽히 준수하며, 사용자에게 항상 믿을 수 있는 정보를 제공하는 성질입니다.

### 2. 💡 비유: 정수기 필터와 물의 품질
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        데이터 무결성 개념 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 데이터베이스 (물탱크): 우리가 정보를 담아두는 거대한 저장소.                 │
│  2. 무결성 제약 조건 (정수기 필터):                                            │
│     - 흙탕물(잘못된 값)이 들어오려 하면 필터가 꽉 막아버림. (차단)               │
│     - 깨끗한 물(유효한 값)만 통과시켜 탱크에 채움. (저장)                        │
│                                                                             │
│  → 필터가 망가지면 물탱크 전체가 오염되듯, 무결성이 깨진 DB는 아무리 데이터가      │
│     많아도 마실 수 없는 물(가치 없는 정보)이 가득한 장소일 뿐입니다!               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **다중 사용자 환경의 위험**: 여러 명이 동시에 데이터를 고칠 때 발생하는 충돌로 인해 데이터가 꼬이는 현상 방지 필요.
- **분산 시스템의 정합성**: 데이터가 여러 서버에 복제되어 있을 때, 모든 곳의 값이 같아야 한다는 '일관성' 요구 증대.

- **📢 섹션 요약 비유**: 데이터 무결성은 데이터베이스라는 국가의 '법치주의'와 같아서, 정해진 규칙(제약)대로만 데이터가 움직이게 강제합니다.

## II. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 무결성의 3대 핵심 속성

| 속성 | 상세 의미 | 보장 메커니즘 |
|:---:|:---|:---|
| **정확성 (Accuracy)** | 중복이나 누락 없이 실제 값과 일치함. | **Unique, NOT NULL** 제약 |
| **일관성 (Consistency)** | 데이터 간의 논리적 모순이 없음. | **Foreign Key, Transaction** |
| **유효성 (Validity)** | 정해진 범위와 형식을 준수함. | **Data Type, CHECK** 제약 |

### 2. 무결성 수호 레이어 (ASCII)

데이터가 시스템에 유입되어 저장되기까지의 다층 방어 체계를 시각화한다.

```text
    [ User Input ]
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 1. [ Application Layer ] : UI 레벨 형식 검사 (1차 필터)           │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 2. [ DBMS Engine Layer ] : 제약 조건(Constraints) 검증 (2차 필터) │
    │    - PK, FK, CHECK, Trigger 실시간 가동                          │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 3. [ Transaction Layer ] : 연산의 원자성(ACID) 보장 (3차 필터)    │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    [ Persistent Data ] : **무결성이 확보된 데이터 안착**
```

### 3. 심층 동작 원리: 참조 무결성과 도메인 무결성의 협업
- **상황**: 주문 테이블에 '상품코드'를 넣으려 함.
- **도메인 무결성**: 입력된 값이 '숫자 5자리' 형식을 맞췄는지 검사.
- **참조 무결성**: 그 5자리 숫자가 실제 '상품 마스터' 테이블에 존재하는지 검사.
- **결과**: 두 검사를 모두 통과해야만 "의미 있는 주문"으로 인정됨.

### 4. 핵심 기술: 데이터 거버넌스와 무결성
- 무결성은 기술적 설정을 넘어, 기업 전체의 **데이터 표준(Standard)** 관리와 직결됨.
- 메타데이터 관리 시스템(MDR)을 통해 모든 시스템의 무결성 규칙을 동기화함.

- **📢 섹션 요약 비유**: 무결성 검사는 '입국 심사'와 같아서, 비자 형식(도메인)도 맞아야 하고 실제 신원(참조)도 확실해야 통과시켜 줍니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 무결성(Integrity) vs 가용성(Availability)의 트레이드오프

| 비교 항목 | 무결성 중심 설계 | 가용성 중심 설계 |
|:---:|:---|:---|
| **최우선 순위** | **데이터의 정확함** | **중단 없는 서비스** |
| **장애 시 대응** | 연산 차단 및 롤백 (안전) | 일단 처리 후 나중에 보정 (속도) |
| **분산 환경 (CAP)** | **일관성(C)** 선택 | 가용성(A) 선택 (BASE 철학) |
| **적용 분야** | 금융, 결제, 인사 시스템 | SNS, 로그 수집, 게임 서버 |

### 2. 과목 융합 관점: 이산수학 및 보안
- **이산수학 융합**: 무결성 제약 조건은 릴레이션이라는 집합 위에 정의된 **술어(Predicate)**이며, 모든 튜플은 이 술어를 '참'으로 만들어야 한다는 논리 체계를 가진다.
- **보안(Security) 융합**: 무결성은 CIA 보안 3요소의 핵심이며, 데이터의 위변조를 막는 **디지털 서명(Digital Signature)**이나 **해시 체크섬(Hash Checksum)** 기술의 논리적 목표점이 된다.

- **📢 섹션 요약 비유**: 무결성이 '완벽한 정답'을 찾는 것이라면, 가용성은 '일단 대화'를 이어가는 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 마이크로서비스 간 데이터 정합성 유지
- **상황**: 주문 서비스와 재고 서비스가 서로 다른 DB를 씀. 물리적 FK를 걸 수 없음.
- **판단**: DB 엔진의 무결성 기능을 쓸 수 없으므로 논리적 데이터 오염 위험 상존.
- **결단**: **Saga 패턴 또는 TCC(Try-Confirm-Cancel) 도입**. 애플리케이션 레벨에서 분산 트랜잭션을 관리하여, 물리적 제약 없이도 최종적인 **데이터 무결성(Eventual Integrity)**을 달성함.

### 2. 안티패턴: "에러 나니까 제약 조건 다 빼자"
- **증상**: 개발 단계에서 자꾸 에러가 난다는 이유로 PK, FK를 모두 제거하고 개발.
- **결과**: 서비스 오픈 1년 후, 부모 없는 데이터와 중복 데이터가 전체의 30%를 차지하여 어떤 통계 보고서도 믿을 수 없는 '데이터 쓰레기장' 탄생.
- **조치**: 제약 조건은 **'귀찮은 방해물'이 아니라 '데이터를 살리는 생명줄'**임을 인지하고, 초기 설계부터 강력하게 적용할 것.

### 📢 기술사적 결언
> "무결성은 데이터베이스 아키텍처의 **'영혼'**이다. 영혼 없는 육체(데이터)는 금방 부패한다. 기술사는 단순히 SQL 제약 조건을 거는 기술자를 넘어, 비즈니스 프로세스 전체를 관통하는 **데이터 정합성 전략**을 수립해야 한다. 특히 현대의 복잡한 하이브리드 클라우드 환경에서는 실시간 무결성(Strong)과 지연 무결성(Eventual) 사이의 경계를 명확히 획정하고, 이를 기술적으로 뒷받침하는 **데이터 가시성(Observability)**을 확보해야 한다."

- **📢 섹션 요약 비유**: 제약 조건을 빼는 것은 '사고가 잦다고 안전벨트를 없애는 것'과 같이 어리석은 행동입니다.

### 2. 미래 전망
- **AI-Driven Integrity Guard**: 기계 학습을 통해 데이터의 맥락을 파악하고, 명시된 규칙 외의 이상 징후를 실시간 감지하여 차단하는 **지능형 무결성 보호** 기술 발전.
- **Quantum-resistant Integrity**: 양자 컴퓨터 시대에 대비하여, 데이터의 무결성을 증명하는 암호화 알고리즘이 더욱 고도화될 것임.

- **📢 섹션 요약 비유**: 미래의 무결성 시스템은 단순한 벽을 넘어, 데이터 스스로가 자신의 진실성을 증명하는 '자율 정화 시스템'으로 진화할 것입니다.
