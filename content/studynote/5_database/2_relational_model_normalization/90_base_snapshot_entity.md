# 90. 기본 엔티티와 스냅샷 엔티티 (Base vs Snapshot Entity)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터의 최신 상태를 유지하는 **기본 엔티티(Base)**와, 특정 시점의 데이터 상태를 사진을 찍듯 고정하여 보관하는 **스냅샷 엔티티(Snapshot)**로 구분하여 관리하는 모델링 기법이다.
> 2. **상호 보완**: 기본 엔티티는 실시간 트랜잭션의 정확성을 책임지며, 스냅샷 엔티티는 과거 시점의 데이터를 재현하거나 통계 분석 시 데이터의 변동 가능성을 차단하여 **일관된 분석 결과**를 보장한다.
> 3. **가치**: 데이터의 이력(History) 관리와 대규모 통계 쿼리의 성능 최적화를 동시에 달성하며, 특히 금융이나 물류 시스템에서 과거 특정일의 정산 데이터를 복원하는 데 필수적인 아키텍처다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **기본 엔티티 (Base Entity)**: 개체의 현재 가장 최신 정보를 담고 있는 마스터 성격의 데이터입니다. (예: 현재 잔액, 현재 주소)
- **스냅샷 엔티티 (Snapshot Entity)**: 과거 특정 시점(월말, 연말, 혹은 특정 이벤트 시점)의 기본 엔티티 정보를 그대로 복사하여 저장한 데이터입니다. (예: 2023년 12월 말 잔액)

### 2. 💡 비유: 거울과 사진첩
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        기본 vs 스냅샷 비유                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 기본 엔티티 (거울): 지금 내 모습을 그대로 비춤.                             │
│     - 내가 웃으면 거울 속 나도 웃음. (실시간 업데이트)                           │
│                                                                             │
│  2. 스냅샷 엔티티 (사진첩): 1년 전 내 모습을 찍어둔 사진.                       │
│     - 지금 내가 아무리 변해도 사진 속 모습은 그대로임. (과거 상태 보존)            │
│                                                                             │
│  → 시스템 운영에는 **'거울(최신)'**이 필요하고, 보고서를 쓰거나 추억을 되새길 땐    │
│     **'사진(스냅샷)'**이 필요합니다!                                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 변동성의 문제**: "지난달 매출 보고서"를 뽑으려는데, 그 사이 고객의 주소나 등급이 바뀌어버리면 당시의 정확한 실적 산출이 불가능해짐.
- **분석 성능 최적화**: 수천만 건의 이력(Log)을 매번 합산하여 과거 상태를 계산하는 것보다, 시점별 합계(Snapshot)를 미리 저장해두는 것이 훨씬 빠름.

- **📢 섹션 요약 비유**: 기본 엔티티는 '현재 진행형'이고, 스냅샷 엔티티는 '과거 완료형'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 엔티티 유형별 특성 비교

| 항목 | 기본 엔티티 (Base) | 스냅샷 엔티티 (Snapshot) |
|:---:|:---|:---|
| **데이터 성격** | **현재 상태 (Current State)** | **과거 상태 (Point-in-Time)** |
| **갱신 방식** | 수시 업데이트 (Overwrite) | **Append-only (수정 불가)** |
| **식별자(PK)** | 개체 고유 식별자 | **{개체 고유 식별자 + 스냅샷 일시}** |
| **주요 목적** | 실시간 업무 처리, CRUD | 결산, 통계 분석, 감사(Audit) |

### 2. 기본-스냅샷 동기화 아키텍처 (ASCII)

업무 처리가 일어난 뒤 정기적으로 스냅샷이 생성되는 과정을 시각화한다.

```text
    [ 실시간 트랜잭션 ]
           │
    ┌──────▼──────────────────┐          ┌──────────────────────────┐
    │  Base Entity: Account   │          │  Snapshot: Monthly_Bal   │
    ├──────┬────────┬─────────┤          ├──────┬────────┬──────────┤
    │ ID   │ Name   │ Balance │          │ ID   │ Month  │ Balance  │
    ├──────┼────────┼─────────┤          ├──────┼────────┼──────────┤
    │ A01  │ Kim    │ 5000    │ ──(Job)─▶│ A01  │ 24-01  │ 4500     │
    └──────┴────────┴─────────┘ (Monthly)│ A01  │ 24-02  │ 5000     │
                                         └──────┴────────┴──────────┘
    (현재는 5000원이지만,                 (지난달, 지지난달의 상태를
     실시간으로 계속 변함)                 기록으로 남겨둠)
```

### 3. 심층 동작 원리: 스냅샷 생성 주기 (Granularity)
- **주기 설정**: 비즈니스 요구에 따라 일단위(Daily), 월단위(Monthly), 혹은 무작위 시점(Event-driven)으로 생성.
- **생성 방식**:
    1. **Full Snapshot**: 모든 레코드를 복사 (공간 소모 큼, 조회 단순).
    2. **Delta Snapshot**: 변경된 레코드만 복사 (공간 효율적, 복원 시 연산 필요).

### 4. 핵심 기술: 이력 관리와 스냅샷의 결합
- **Log-based**: 모든 변경 로그(History)를 남기되,
- **Snapshot-based**: 조회 성능을 위해 특정 시점의 중간 합계를 스냅샷으로 유지. (예: Redis의 RDB 스냅샷과 AOF 로그의 관계와 유사)

- **📢 섹션 요약 비유**: 스냅샷은 '영화의 매 프레임을 다 저장하는 대신, 1초마다 한 장씩 사진을 남겨두는 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 스냅샷(Snapshot) vs 로그 이력(History Log)

| 비교 항목 | 스냅샷 (Snapshot) | 로그 이력 (Transaction Log) |
|:---:|:---|:---|
| **저장 단위** | **전체 상태 (State)** | **개별 행위 (Event)** |
| **용량** | 중복 데이터로 인해 큼 | 발생 횟수에 비례함 |
| **복원 속도** | **즉시 복원** | 로그를 처음부터 재실행해야 함 |
| **활용** | 경영진 보고서, 결산 | 장애 복구, 상세 원인 추적 |

### 2. 과목 융합 관점: 가상화 및 운영체제
- **운영체제(OS) 융합**: 파일 시스템의 **Copy-on-Write (CoW)** 기술을 이용한 스냅샷 생성 방식은 데이터베이스의 스냅샷 엔티티 생성 메커니즘의 물리적 모태가 된다.
- **클라우드 융합**: 가상 머신(VM)이나 컨테이너의 **이미지 스냅샷**과 유사하게, 데이터베이스 전체 상태를 보존하여 다른 리전으로 복제하거나 테스트 환경을 즉시 구축하는 데 활용된다.

- **📢 섹션 요약 비유**: 스냅샷이 '중간 점수'라면, 로그는 '점수를 얻기까지의 모든 플레이 기록'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 연말 결산을 위한 스냅샷 테이블 설계
- **상황**: 12월 31일 자의 재고 가치를 산정해야 함.
- **판단**: 1월 1일이 되는 순간 출고가 발생하면 12월 31일 자 데이터는 거울에서 사라짐.
- **결단**: **12월 31일 23:59분에 'Yearly_Inventory_Snapshot' 테이블 생성**. 당시의 단가, 수량, 창고 위치를 통째로 얼려(Freeze) 두어, 나중에 회계 감사를 받을 때 완벽한 증거 자료로 제출함.

### 2. 안티패턴: 기본 엔티티에 과거 정보 덧쓰기
- **증상**: "주소가 바뀌었으니 기존 주소 위에 새 주소를 덮어쓰자."
- **결과**: 과거 배송 사고 조사 시, 당시 고객이 어디 살았는지 알 수 없게 됨.
- **조치**: 중요한 속성은 **이력 테이블**을 별도로 두거나, **유효 기간(Start_Date, End_Date)** 정보를 포함한 기본 엔티티 설계를 통해 선후 관계를 명확히 할 것.

### 📢 기술사적 결언
> "기본 엔티티와 스냅샷 엔티티의 조화는 **'공간으로 시간을 사는 행위'**다. 저장 공간을 더 써서 과거로 돌아가는 속도를 얻기 때문이다. 기술사는 비즈니스의 **재현 요구사항**을 분석하여 스냅샷의 밀도(Density)를 결정해야 한다. 너무 잦은 스냅샷은 스토리지 비용 폭증을 부르고, 너무 드문 스냅샷은 데이터의 신뢰성을 떨어뜨린다. 결국 '어느 시점의 데이터가 비즈니스적으로 가치 있는가'를 판단하는 것이 아키텍트의 혜안이다."

- **📢 섹션 요약 비유**: 무분별한 스냅샷은 '매초 사진을 찍어서 사진첩이 터져나가는 것'과 같아 정작 원하는 사진을 찾기 힘들게 만듭니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (스냅샷 전략 도입 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **보고서 생성 시간** | 수 시간 (연산 필요) | **수 초 (즉시 조회)** | 업무 효율성 90% 향상 |
| **데이터 정합성 분쟁** | 빈번 발생 | **증거 데이터 확보** | 법적/회계적 리스크 해소 |
| **운영계 DB 부하** | 대량 분석 쿼리로 높음 | **스냅샷 조회로 분산** | 서비스 안정성 확보 |

### 2. 미래 전망
- **Blockchain Snapshots**: 스냅샷 데이터의 무결성을 증명하기 위해 해시값을 블록체인에 기록하는 **증명 가능한 이력 관리** 기술 확산.
- **Streaming Snapshots**: 배치가 아닌 실시간 스트림(Change Data Capture)을 통해 실시간에 가까운 스냅샷(Continuous View)을 유지하는 기술이 메인스트림이 될 것임.

- **📢 섹션 요약 비유**: 미래의 데이터베이스는 시간을 자유자재로 넘나드는 '타임머신' 기능을 내장하여, 과거의 진실을 언제든 현재로 소환하는 지능형 저장소가 될 것입니다.
