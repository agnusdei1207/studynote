# 102. 제2정규형 (2NF, Second Normal Form) - 부분 함수적 종속성 제거

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 릴레이션이 제1정규형(1NF)을 만족하면서, 기본 키(Primary Key)가 아닌 모든 속성이 기본 키 전체에 **완전 함수적으로 종속(Full Functional Dependency)**되어야 한다는 원칙이다.
> 2. **작동 원리**: 기본 키가 복합 키(Composite Key)인 경우, 키의 일부분에만 종속된 속성들(**부분 함수적 종속**)을 찾아내어 별도의 테이블로 분리함으로써 데이터 중복을 제거한다.
> 3. **가치**: 특정 정보만 업데이트하려 할 때 발생하는 갱신 이상이나, 일부 정보가 없어 전체 행을 입력하지 못하는 삽입 이상 문제를 해결하여 데이터의 정합성을 한 단계 격상시킨다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**제2정규형 (2NF)**은 복합 키를 가진 릴레이션에서 발생할 수 있는 중복과 모순을 해결하는 단계입니다. "기본 키의 부분집합이 결정자가 되어서는 안 된다"는 것이 핵심입니다.

### 2. 💡 비유: 수강증과 개인정보
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        2NF 개념 비유                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] '학생'과 '과목' 정보를 하나의 수강증에 다 적고 있음.                      │
│                                                                             │
│  1. 기본 키: {학번 + 과목번호} (누가 어떤 과목을 듣는지)                         │
│  2. 부분 종속 (문제 발생):                                                    │
│     - '학생 성명'은 {학번}만 알면 나옴. {과목번호}는 몰라도 됨. (부분 종속)        │
│     - '강의실 위치'는 {과목번호}만 알면 나옴. {학번}은 필요 없음. (부분 종속)      │
│                                                                             │
│  → 수강증 하나에 학생 정보와 강의실 정보를 다 적어두면, 학생이 개명을 할 때        │
│     그 학생이 듣는 모든 수강증(수백 장)을 다 고쳐야 함! (갱신 이상)                │
│                                                                             │
│  → 해결: [학생], [과목], [수강] 세 장의 종이로 나누어 적는 것이 바로 2NF!          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **복합 키 설계의 함정**: 여러 속성을 묶어 PK를 만들 때, 키의 일부에만 걸려 있는 부가 정보들이 중복을 일으킴.
- **데이터 일관성 붕괴**: 특정 과목의 강의실을 바꿨는데, 수강생이 없는 과목은 강의실 정보를 저장할 수 없는(삽입 이상) 문제 발생.

- **📢 섹션 요약 비유**: 2NF는 '한 지붕 아래 살지만 남남인 데이터들'을 각자의 집으로 독립시켜주는 분가(分家) 작업입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 용어 정리

| 용어 | 상세 정의 | 비유 |
|:---:|:---|:---|
| **완전 함수적 종속** | 기본 키 전체($X, Y$)를 알아야만 결정되는 종속 관계. | 부부가 모두 동의해야 성사되는 계약 |
| **부분 함수적 종속** | 기본 키 중 일부($X$ 또는 $Y$)만 알아도 결정되는 관계. | 남편 혼자 결정해도 되는 일 |
| **결정자 (Determinant)** | 다른 속성을 결정짓는 속성 (화살표의 시작점). | 번호표 |

### 2. 2NF 변환 과정 도해 (ASCII)

부분 종속성을 분리하여 정합성을 높이는 과정을 시각화한다.

```text
    [ 1NF 릴레이션: 수강성적 ]
    PK: { 학번, 과목번호 }
    ┌──────┬──────────┬──────┬────────┐
    │ 학번 │ 과목번호  │ 성적 │ 강의실  │
    ├──────┼──────────┼──────┼────────┤
    │ 101  │   D01    │  A   │  301호 │ <--- (강의실은 과목번호에만 종속)
    │ 101  │   D02    │  B   │  405호 │
    │ 102  │   D01    │  A   │  301호 │ <--- 중복 발생!
    └──────┴──────────┴──────┴────────┘
               │
               ▼ [ 2NF 변환: 부분 종속 제거 ]
    ┌────────────────────────┐      ┌────────────────────┐
    │   [ 성적 테이블 ]       │      │   [ 과목 테이블 ]    │
    ├──────┬──────────┬──────┤      ├──────────┬─────────┤
    │ 학번 │ 과목번호  │ 성적 │      │ 과목번호  │ 강의실   │
    ├──────┼──────────┼──────┤      ├──────────┼─────────┤
    │ 101  │   D01    │  A   │      │   D01    │  301호  │
    │ 101  │   D02    │  B   │      │   D02    │  405호  │
    │ 102  │   D01    │  A   │      └──────────┴─────────┘
    └──────┴──────────┴──────┘
    (강의실 정보는 이제 딱 한 번만 기록됨)
```

### 3. 심층 동작 원리: 무손실 분해 (Lossless Join)
- **원리**: 테이블을 나눌 때 공통 속성(Join Key)을 남겨두어, 다시 합쳤을 때 원래의 모든 정보를 100% 복원 가능해야 함.
- **위 예시**: `과목번호`가 두 테이블 사이의 다리(Bridge) 역할을 하여 무손실 분해를 보장함.

### 4. 핵심 기술: 함수적 종속성 다이어그램 (FD Diagram)
- 설계 단계에서 화살표를 그려 키의 일부에서 뻗어 나가는 선이 있는지 확인.
- 키 전체가 아닌 부분에서 출발하는 모든 화살표는 2NF 위반 대상임.

- **📢 섹션 요약 비유**: 무손실 분해는 '종이를 찢을 때 글자가 잘리지 않게, 나중에 다시 붙여 읽을 수 있도록 조심해서 자르는 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 1NF vs 2NF vs 3NF 비교

| 정규형 | 주요 타겟 (제거 대상) | 핵심 원리 |
|:---:|:---|:---|
| **1NF** | **다중값 (Multivalued)** | 속성 값의 원자화 |
| **2NF** | **부분 함수적 종속** | 기본 키 전체에 대한 종속성 확보 |
| **3NF** | **이행 함수적 종속** | 키가 아닌 속성 간의 종속성 제거 |

### 2. 과목 융합 관점: 소프트웨어 공학 및 운영체제
- **소프트웨어 공학(SE) 융합**: 2NF는 객체 지향의 **응집도(Cohesion)**를 높이는 리팩토링 과정과 유사하다. 하나의 클래스가 두 가지 서로 다른 목적(학생과 과목)을 가지지 않게 분리하는 것과 같다.
- **운영체제(OS) 융합**: 2NF를 통한 데이터 분리는 메모리 상의 **Cache Locality**를 변화시킨다. 자주 바뀌는 데이터(성적)와 잘 안 바뀌는 데이터(강의실)를 분리하여 캐시 효율을 높일 수 있다.

- **📢 섹션 요약 비유**: 1NF가 '재료 손질'이라면, 2NF는 '재료별 보관'입니다. 3NF는 '상한 재료 골라내기'로 이어집니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 마스터-디테일 구조의 정립
- **상황**: 주문(Order) 테이블에 상품명과 단가를 함께 저장 중임.
- **판단**: 상품명은 상품코드(복합키의 일부)에만 종속되므로 2NF 위반.
- **결단**: **상품 마스터 테이블을 별도로 구축**. 주문 테이블에는 상품코드(FK)만 남김으로써, 상품 가격이 변동될 때 모든 주문 내역을 뒤지지 않아도 되는 견고한 시스템 구축.

### 2. 안티패턴: 복합 키 사용 시의 설계 방임
- **증상**: "복합 키니까 유일하겠지"라며 키 구성 속성 중 하나에만 걸려 있는 10개 이상의 속성을 방치.
- **결과**: 테이블이 거대해지고(Wide Table), 중복 데이터 입력 시 오타가 발생해도 엔진이 잡아내지 못해 정합성 박살.
- **조치**: 복합 키를 사용할 때는 반드시 **모든 일반 속성이 키 전체와 관련 있는지**를 매의 눈으로 검증해야 함.

### 📢 기술사적 결언
> "제2정규형은 **'책임의 소재'**를 분명히 하는 단계다. 기본 키가 그 행의 모든 정보를 온전히 책임지고 있는지를 따지기 때문이다. 기술사는 단순히 정규화 수식을 푸는 것을 넘어, **데이터의 생명 주기(Lifecycle)**가 다른 정보들이 한 테이블에 섞여 있는지를 포착해내야 한다. 2NF를 통해 확보된 데이터의 독립성은 향후 **시스템 확장(Scale-out)**과 **API 기반의 서비스 분리** 시 가장 강력한 자산이 된다."

- **📢 섹션 요약 비유**: 2NF 위반은 '동생 일기장에 내 비밀을 적어두는 것'과 같아, 동생이 이사(데이터 변경)가면 내 비밀도 같이 위험해집니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (2NF 준수 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 수정 시간** | 수십 초 (Full Scan) | **밀리초 (Point Update)** | 갱신 성능 100배 향상 |
| **데이터 정합성** | 90% (수동 보정 필요) | **100% (구조적 보장)** | 이상 현상 원천 차단 |
| **개발 가독성** | 거대 테이블 분석 난해 | **주제별 명확한 엔티티** | 유지보수 편의성 증대 |

### 2. 미래 전망
- **Microservices Data Decomposition**: MSA 환경에서 서비스별 DB를 쪼갤 때, 2NF의 '부분 종속성 분리' 원리가 서비스 경계를 획정하는 논리적 가이드로 더욱 중요해짐.
- **Auto-Normalization Engine**: AI가 쿼리 패턴을 분석하여 부분 종속성을 자동으로 감지하고 테이블 분할을 권고하는 **지능형 스키마 튜닝** 도구 상용화.

- **📢 섹션 요약 비유**: 기술이 아무리 변해도 '각자의 정보는 각자의 자리에'라는 2NF의 질서는 정보 관리의 기본 매너로 남을 것입니다.
