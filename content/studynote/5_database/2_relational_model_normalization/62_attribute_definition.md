# 62. 속성 (Attribute / Column / Degree) - 릴레이션의 열

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 관계형 데이터 모델의 릴레이션에서 개체의 고유한 특성이나 상태를 나타내는 논리적 최소 단위이며, 사용자 관점에서는 **열(Column)** 또는 필드(Field)에 해당한다.
> 2. **구조적 역할**: 릴레이션 스키마의 구성 요소로서 데이터의 의미와 형식을 규정하며, 한 릴레이션이 가진 속성의 총 개수를 **차수(Degree)**라고 정의하여 릴레이션의 복잡도를 나타낸다.
> 3. **가치**: 속성은 상호 독립적이어야 하며, 도메인(Domain)이라는 유효 값 범위를 가짐으로써 데이터베이스의 **무결성(Integrity)**을 유지하는 일차적인 규제 메커니즘으로 작동한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**속성 (Attribute)**은 릴레이션을 구성하는 각각의 세로 열을 의미합니다. 현실 세계의 개체가 가지는 구체적인 성질(예: 학생의 '이름', '나이', '전공')을 데이터베이스 언어로 표현한 것입니다.

### 2. 💡 비유: 인적 사항 기입란의 '항목'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        속성(Attribute) 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 회원 가입 양식을 작성 중임.                                             │
│                                                                             │
│  1. 항목 명칭: [아이디], [비밀번호], [이메일] (속성 이름)                        │
│  2. 기입 칸: 각 항목 아래에 내용을 적는 세로줄 전체. (Column)                   │
│  3. 차수 (Degree): 이 가입 양식에 총 몇 개의 질문 항목이 있는가?                 │
│                                                                             │
│  → 속성은 데이터라는 집을 짓기 위한 **'기둥의 종류'**와 같습니다.                 │
│     기둥이 3개면 차수가 3인 집(릴레이션)이 되는 것이죠!                           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **정형 데이터의 명확화**: 비구조화된 텍스트에서 필요한 정보만 쏙쏙 뽑아내어 규격화된 틀(Schema)에 넣기 위해 도입.
- **데이터 분석의 기초**: "모든 사원의 '연봉' 합계를 구해라"와 같이 특정 성격의 데이터만 집계하기 위한 분류 기준 필요.

- **📢 섹션 요약 비유**: 속성은 데이터라는 원석을 깎아 만든 '분류표'이자, 정보의 성격을 규정하는 '이름표'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 속성의 주요 특징 및 제약

| 특징 | 상세 설명 | 비유 |
|:---:|:---|:---|
| **원자성 (Atomicity)** | 속성 값은 더 이상 쪼개질 수 없는 하나의 값이어야 함. | 쪼개지지 않는 원자 |
| **무순서성** | 속성 간의 좌우 배치는 논리적으로 의미가 없음. | 순서 상관없는 기둥 |
| **명칭 유일성** | 한 릴레이션 내에서 속성 이름은 중복될 수 없음. | 중복 없는 이름표 |
| **차수 (Degree)** | 릴레이션을 구성하는 속성의 총 개수. | 건물의 방 개수 |

### 2. 속성과 도메인의 관계 도해 (ASCII)

속성이 데이터의 형식을 어떻게 규정하는지 시각화한다.

```text
    [ Relation: Student ]
    ┌─────────────┬─────────────┬─────────────┐
    │  Name (Attr)│  Age (Attr)  │ Grade (Attr)│ <--- Degree = 3
    ├─────────────┼─────────────┼─────────────┤
    │    Kim      │     20      │      A      │
    │    Lee      │     22      │      B      │
    └──────┬──────┴──────┬──────┴──────┬──────┘
           │             │             │
    ┌──────▼──────┐┌─────▼──────┐┌─────▼──────┐
    │ Domain: STR ││ Domain: INT ││ Domain: CHR │ (데이터 타입 제약)
    │ ('가'~'힣')  ││ (0 ~ 150)   ││ ('A' ~ 'F') │ (범위 제약)
    └─────────────┘└────────────┘└─────────────┘
```

### 3. 심층 동작 원리: 속성의 원자성 (Atomicity)
- **개념**: 관계형 모델의 대전제로, 한 칸(Cell)에는 반드시 하나의 값만 들어가야 함.
- **위반 사례**: '취미' 속성에 '독서, 등산' 두 값을 넣는 경우.
- **해결**: 중복 속성을 분리하여 새로운 릴레이션을 만들거나(1NF), 속성을 여러 개(취미1, 취미2)로 나누어야 함.

### 4. 핵심 기술: 차수(Degree)의 변화와 스키마 진화
- **차수의 증가**: `ALTER TABLE ADD column_name` 명령을 통해 속성이 추가되면 차수가 증가함. 이는 릴레이션 스키마의 구조적 변경을 의미함.
- **성능 영향**: 차수가 너무 큰 테이블(Wide Table)은 디스크 I/O 시 불필요한 데이터를 많이 읽게 되어 성능 저하를 유발할 수 있음 (Row Store의 한계).

- **📢 섹션 요약 비유**: 원자성은 '한 그릇에 한 종류의 음식만 담는 깔끔함'이며, 차수는 '식탁에 놓인 접시의 개수'와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 속성(Attribute) vs 튜플(Tuple)

| 비교 항목 | 속성 (Attribute) | 튜플 (Tuple) |
|:---:|:---|:---|
| **방향** | **세로 (Vertical)** | **가로 (Horizontal)** |
| **의미** | 데이터의 **종류 / 성격** | 데이터의 **실체 / 사례** |
| **수량 지표** | **차수 (Degree)** | **카디널리티 (Cardinality)** |
| **변화 빈도** | 낮음 (정적 구조) | 높음 (동적 데이터) |

### 2. 과목 융합 관점: 소프트웨어 공학 및 보안
- **소프트웨어 공학(SE) 융합**: 속성은 객체 지향 프로그래밍의 **멤버 변수(Member Variable)** 또는 필드(Field)와 1:1로 매핑된다.
- **보안(Security) 융합**: 특정 민감한 속성(예: 주민번호, 급여)에 대해서만 접근을 차단하거나 암호화하는 **열 수준 보안(Column-Level Security)**은 속성 단위 관리의 핵심 보안 응용이다.

- **📢 섹션 요약 비유**: 속성이 '질문의 종류'라면, 튜플은 '사람들이 써낸 답변'입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 신규 비즈니스 요구에 따른 속성 추가
- **상황**: 기존 회원 테이블에 '마케팅 수신 동의 여부'를 추가해야 함.
- **판단**: 속성을 추가하면 차수가 늘어나고, 기존 수억 건의 데이터에 NULL이나 기본값이 채워지는 물리적 작업 발생.
- **결단**: **온라인 DDL 기술을 활용하거나, 자주 변하는 속성은 확장 테이블로 분리**. 운영 중인 서비스의 중단을 최소화하면서 차수를 유연하게 확장하는 전략 수립.

### 2. 안티패턴: 다중값 속성(Multivalued Attribute)의 방치
- **증상**: "귀찮으니까 그냥 콤마(,)로 구분해서 다 때려 넣자."
- **결과**: 특정 값으로 검색 시 `LIKE '%검색어%'`를 써야 하므로 인덱스를 타지 못해 성능 폭망. 데이터 수정 시에도 문자열 파싱 부하 발생.
- **조치**: **제1정규화(1NF)를 철저히 준수**. 다중값은 별도의 자식 테이블로 추출하여 부모-자식 관계(FK)로 연결.

### 📢 기술사적 결언
> "속성은 데이터베이스의 **'언어'**다. 비즈니스를 어떤 속성으로 정의하느냐에 따라 시스템의 표현력이 결정된다. 기술사는 단순히 컬럼을 늘리는 데 급급하지 말고, 각 속성이 가진 **도메인의 무결성**과 **원자성**을 어떻게 지킬 것인지 고민해야 한다. 특히 현대의 데이터 레이크 환경에서는 속성 정보(Catalog)를 잘 관리하는 것이 곧 데이터의 자산 가치를 결정짓는 **메타데이터 거버넌스**의 핵심임을 잊지 말아야 한다."

- **📢 섹션 요약 비유**: 다중값 속성은 '한 칸에 여러 명의 이름을 적어둔 출석부'와 같아, 출석 체크(검색)를 할 때마다 혼란을 줍니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (명확한 속성 정의 시)

| 지표 | 정의 미흡 시 | 정의 최적화 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 정합성** | 낮음 (잘못된 값 유입) | **높음 (도메인 제약)** | 무결성 오류 90% 차단 |
| **검색 효율** | 낮음 (풀 스캔 빈번) | **높음 (속성 기반 인덱스)** | 조회 속도 10배 이상 향상 |
| **유지보수성** | 스파게티 구조 | **직관적인 스키마** | 인계인수 및 확장 용이 |

### 2. 미래 전망
- **Dynamic Attribute (Schema-less)**: 고정된 속성 외에 유동적인 속성을 처리하기 위한 **JSON 타입 속성** 지원이 RDBMS의 표준 기능으로 정착 중.
- **AI-Generated Metadata**: 수만 개의 속성 중 의미가 유사한 것을 AI가 찾아내어 통합하거나 설명(Description)을 달아주는 **지능형 카탈로그** 기술 발전.

- **📢 섹션 요약 비유**: 미래의 속성은 고정된 기둥이 아니라, 필요에 따라 늘어났다 줄어드는 '스마트 모듈'처럼 진화할 것입니다.

2. 이 제목들이 있어야 나중에 **"이름만 보고 싶어!"** 할 때 이름 칸만 쏙 골라서 볼 수 있죠.
