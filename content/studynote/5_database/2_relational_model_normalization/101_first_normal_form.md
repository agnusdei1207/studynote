# 101. 제1정규형 (1NF, First Normal Form) - 속성의 원자성 확보

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 관계형 데이터베이스의 모든 릴레이션이 갖춰야 할 가장 기본적인 구조적 요건으로, 릴레이션의 모든 속성 값이 더 이상 분해될 수 없는 **단일 값(Atomic Value)**만으로 구성되도록 강제하는 단계다.
> 2. **수행 과정**: 하나의 칸(Cell)에 여러 개의 값이 들어간 **다중값 속성(Multivalued Attribute)**이나, 비슷한 속성이 반복되는 **중복 그룹**을 제거하여 각각 별도의 튜플이나 테이블로 분리한다.
> 3. **가치**: 데이터 검색 및 수정 시 복잡한 문자열 파싱이나 논리적 모순을 제거하며, 이후 단계의 고차 정규화(2NF, 3NF 등)를 수행할 수 있는 논리적 평면 구조를 확립한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**제1정규형 (1NF)**은 릴레이션 스키마 설계의 첫 번째 단계입니다. "릴레이션의 모든 속성이 원자 값(Atomic Value)으로만 되어 있으면 제1정규형을 만족한다"라고 정의합니다.

### 2. 💡 비유: 서랍장의 한 칸에 한 물건만 넣기
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        1NF 개념 비유                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 장난감 상자를 정리하고 있음.                                            │
│                                                                             │
│  1. 비정규형 (엉망진창): '취미'라는 상자 한 칸에 [축구, 영화, 독서]를 몽땅 넣음.   │
│     - 나중에 "축구공만 꺼내줘!"라고 하면 상자 안을 다 뒤져서 골라내야 함.         │
│                                                                             │
│  2. 제1정규형 (원자화):                                                       │
│     - 상자 칸을 나누거나, 아예 다른 줄에 [축구], [영화], [독서]를 하나씩 따로 담음.│
│     - 이제 "축구" 칸만 딱 보면 되니까 찾기가 훨씬 쉬워짐!                        │
│                                                                             │
│  → 복잡하게 뭉쳐있는 데이터를 **'더 이상 쪼갤 수 없는 알갱이'**로 나누는 것이 1NF!  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **검색 효율성 저하**: 한 컬럼에 여러 값이 쉼표(,) 등으로 묶여 있으면 `LIKE %...%` 검색을 써야 하며, 이는 인덱스를 타지 못해 성능을 심각하게 저하시킴.
- **데이터 업데이트의 난해함**: '독서'만 '글쓰기'로 바꾸려 해도 전체 문자열을 읽어서 수정 후 다시 써야 하는 번거로움 발생.

- **📢 섹션 요약 비유**: 1NF는 데이터를 '가루'가 될 때까지 쪼개어, 컴퓨터가 다루기 가장 편한 알갱이 상태로 만드는 작업입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 1NF 위반 사례와 해결 전략

| 위반 유형 | 상세 내용 | 해결 방법 |
|:---:|:---|:---|
| **다중값 속성** | 한 칸에 {값1, 값2} 처럼 여러 데이터 존재. | 새로운 행(Tuple)으로 분리하거나 테이블 분할. |
| **중복 그룹** | 취미1, 취미2, 취미3 처럼 유사 속성 반복. | 속성을 하나로 합치고 여러 행으로 전개. |
| **복합 속성** | 주소(시+구+동)처럼 의미가 섞인 속성. | 시, 구, 동을 각각의 독립 속성으로 분해. |

### 2. 1NF 변환 과정 도해 (ASCII)

비정규형 테이블이 어떻게 1NF로 변하는지 시각화한다.

```text
    [ 비정규형: 학생_취미 ]
    ┌────┬─────┬─────────────────────┐
    │학번 │성명 │ 취미 (다중값)        │
    ├────┼─────┼─────────────────────┤
    │101 │ Kim │ 축구, 영화, 독서     │ <--- 위반!
    │102 │ Lee │ 음악                 │
    └────┴─────┴─────────────────────┘
               │
               ▼ [ 1NF 변환: 모든 속성을 원자화 ]
    ┌────┬─────┬──────────┐
    │학번 │성명 │ 취미      │
    ├────┼─────┼──────────┤
    │101 │ Kim │ 축구      │ <--- 행을 분리하여
    │101 │ Kim │ 영화      │      한 칸에 하나씩만 저장
    │101 │ Kim │ 독서      │
    │102 │ Lee │ 음악      │
    └────┴─────┴──────────┘
    * PK가 {학번}에서 {학번, 취미} 복합키로 변경됨.
```

### 3. 심층 동작 원리: 기본 키(Primary Key)의 재정의
- **이슈**: 데이터를 쪼개다 보면 기존의 단일 PK만으로는 튜플을 식별할 수 없게 됨. (위 예시에서 학번 101이 세 줄 나옴)
- **해결**: 원자화된 속성을 포함하여 **복합 키(Composite Key)**를 형성하거나, 인조 키(Surrogate Key)를 도입하여 유일성을 확보해야 함.

### 4. 핵심 기술: 데이터 타입의 엄격성
- 1NF는 단순히 형태만 쪼개는 것이 아니라, 도메인(Domain)의 **원자성**을 엔진 차원에서 보장받는 단계임.
- 현대의 Postgres 등에서 지원하는 'Array'나 'JSON' 타입은 편의상 1NF를 물리적으로 우회하지만, 논리적 설계 관점에서는 여전히 1NF 준수가 권장됨.

- **📢 섹션 요약 비유**: 복합 키 형성은 '같은 이름의 상자가 여러 개 생기니, 상자마다 번호를 붙여서 구분하는 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 1NF 준수 vs 미준수 시의 연산 효율

| 비교 항목 | 1NF 미준수 (Comma Separated) | 1NF 준수 (Normalized) |
|:---:|:---|:---|
| **검색 (Search)** | `WHERE col LIKE '%A%'` (Full Scan) | **`WHERE col = 'A'` (Index Scan)** |
| **수정 (Update)** | 문자열 파싱 및 결합 필요 | **단일 행 즉시 수정** |
| **통계 (Count)** | 애플리케이션 연산 필수 | **`COUNT(*)` 한 줄로 끝** |
| **데이터 정합성** | '축구 '와 ' 축구'를 다르게 인식 가능 | **도메인 제약으로 일관성 유지** |

### 2. 과목 융합 관점: 자료구조 및 컴파일러
- **자료구조 융합**: 1NF는 가변 길이의 리스트 구조를 고정 크기의 **플랫한 배열(Flat Array)** 구조로 바꾸는 작업이며, 이는 메모리 접근 시 **Offset 계산**을 단순화하여 하드웨어 성능을 극대화한다.
- **컴파일러 융합**: 텍스트를 토큰(Token)으로 분리하는 **어휘 분석(Lexical Analysis)** 과정은 비정형 데이터를 1NF 형태의 유효한 심볼로 변환하는 정규화의 철학과 닮아 있다.

- **📢 섹션 요약 비유**: 1NF는 '덩어리 고기'를 '다짐육'으로 만드는 것과 같아, 요리(연산)할 때 훨씬 고르게 빨리 익습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 다중 연락처 관리 설계
- **상황**: 고객이 휴대폰, 집전화, 회사번호 등 여러 연락처를 가짐.
- **판단**: `phone1`, `phone2` 컬럼을 만드는 중복 그룹 방식이나, 한 칸에 다 넣는 방식 모두 지양해야 함.
- **결단**: **고객연락처 엔티티를 별도로 분리**. `고객ID + 연락처구분코드`를 PK로 하는 1NF 테이블을 설계하여, 향후 이메일이나 SNS 계정이 추가되어도 스키마 변경 없이 유연하게 대응함.

### 2. 안티패턴: JSON 컬럼에 모든 비즈니스 로직 매몰
- **증상**: "정규화하기 귀찮으니 그냥 JSON 타입 하나에 다 때려 넣자."
- **결과**: 초기 개발은 빠르나, 나중에 특정 속성값으로 조인(Join)하거나 대량 집계 시 성능이 기하급수적으로 느려짐. (사실상의 파일 시스템 회귀)
- **조치**: **검색과 조인의 대상이 되는 핵심 속성은 반드시 1NF로 추출**하고, 단순 정보성 데이터만 선택적으로 JSON 활용.

### 📢 기술사적 결언
> "제1정규형은 관계형 데이터베이스의 **'입장권'**이다. 원자성이 확보되지 않은 데이터는 관계형 엔진의 강력한 기능(Index, Join, Optimizer)을 제대로 누릴 수 없기 때문이다. 기술사는 편의성이라는 유혹에 넘어가 데이터를 뭉뚱그려 저장하는 **'논리적 게으름'**을 경계해야 한다. 1NF라는 탄탄한 기초 공사 위에 2NF, 3NF의 상부 구조를 세울 때 비로소 거대한 데이터의 하중을 견디는 시스템이 완성된다."

- **📢 섹션 요약 비유**: JSON에 다 넣는 것은 '정리하기 귀찮아서 모든 물건을 검은 비닐봉지에 담아두는 것'과 같아, 나중에 내용물을 확인하려면 봉지를 일일이 다 열어봐야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (1NF 준수 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **인덱스 활용률** | 0% (LIKE 검색) | **100% (Equal 검색)** | 조회 성능 100배 이상 향상 |
| **데이터 수정 시간** | Read-Parse-Write | **Direct Update** | 쓰기 오버헤드 70% 감소 |
| **데이터 무결성** | 중복/누락 위험 | **원천 차단** | 데이터 품질 신뢰도 확보 |

### 2. 미래 전망
- **Automatic Atomicity Extraction**: 비정형 텍스트나 로그에서 AI가 자동으로 의미 있는 속성을 추출하여 1NF 테이블로 정규화해주는 **자율형 ETL** 기술 발전.
- **Polyglot Storage**: 1NF의 정교함과 비정형의 유연성을 하드웨어 레벨에서 통합하여, JSON 내 데이터도 1NF처럼 초고속 검색하는 **인덱싱된 도큐먼트 엔진** 진화.

- **📢 섹션 요약 비유**: 미래에는 기술이 발전해도 '무엇이 최소 단위인가'를 정의하는 1NF의 원자성 원칙은 모든 데이터 아키텍처의 근본 지침으로 남을 것입니다.
