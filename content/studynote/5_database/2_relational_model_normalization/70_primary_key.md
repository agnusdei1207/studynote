# 70. 기본 키 (Primary Key, PK) - 후보 키 중 설계자가 선택한 메인 식별자

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 릴레이션 내에서 튜플을 유일하게 식별하기 위해 후보 키(Candidate Key) 중 데이터베이스 설계자가 최종적으로 선정한 **대표 식별자**다.
> 2. **제약 조건**: 모든 튜플에 대해 고유한 값을 가져야 하며(**유일성**), 단 하나의 속성이라도 누락되면 안 되는 **최소성**을 만족함은 물론, 절대로 빈 값을 허용하지 않는 **NULL 불가** 원칙을 고수한다.
> 3. **가치**: 데이터베이스 정합성의 근간인 **개체 무결성(Entity Integrity)**을 보장하며, 물리적으로는 **클러스터드 인덱스(Clustered Index)**의 기준이 되어 검색 및 조인 성능을 결정짓는 핵심 지표로 작동한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**기본 키 (Primary Key, PK)**는 한 릴레이션에서 특정 튜플을 유일하게 구별할 수 있는 가장 중요한 속성입니다. 후보 키 중에서 비즈니스적 가치와 물리적 효율성을 고려하여 선택된 '단 하나'의 키입니다.

### 2. 💡 비유: 대한민국 국민의 '주민등록번호'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        기본 키(Primary Key) 비유                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 수천만 명의 국민 중 '홍길동' 한 명을 정확히 찾아야 함.                   │
│                                                                             │
│  1. 후보군: {지문}, {주민번호}, {운정면허번호}, {여권번호} 등. (후보 키)           │
│  2. 대표 선정: 국가 행정의 편의를 위해 **'주민번호'**를 메인 식별자로 정함. (기본 키) │
│                                                                             │
│  - 규칙 1: 주민번호가 똑같은 사람은 절대 없음. (유일성)                          │
│  - 규칙 2: 대한민국 국민이라면 누구나 주민번호가 있어야 함. (NULL 불가)            │
│                                                                             │
│  → 기본 키는 데이터 세계에서 그 존재를 증명하는 **'유일무이한 출생 증명서'**입니다.   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 식별의 유일성 보장**: 데이터가 중복되거나 뒤섞이는 현상을 방지하기 위한 강제 규약 필요.
- **관계 형성의 기준**: 서로 다른 테이블을 연결(Join)할 때, 참조의 대상이 되는 확실한 기준점이 요구됨.
- **물리적 저장 최적화**: 디스크 상에서 데이터를 정렬하고 검색 속도를 높이기 위한 물리적 정렬 기준(Primary Index) 확보.

- **📢 섹션 요약 비유**: 기본 키는 릴레이션이라는 거대한 도시에서 집을 정확히 찾아가게 해주는 '대표 지번(Address)'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 기본 키의 성질 및 개체 무결성 제약

| 성질 | 상세 설명 | 비유 |
|:---:|:---|:---|
| **유일성** | 중복된 값을 가질 수 없음. | 세상에 단 하나뿐인 번호 |
| **최소성** | 키를 구성하는 속성 집합이 더 이상 쪼개질 수 없는 최소 단위여야 함. | 군더더기 없는 증명서 |
| **개체 무결성** | **NULL 값을 절대 가질 수 없음**. (Entity Integrity) | 존재하지 않는 사람은 번호도 없음 |
| **불변성** | 한 번 부여된 기본 키 값은 가급적 변경되지 않아야 함. | 이름은 바뀌어도 주민번호는 그대로 |

### 2. 기본 키와 물리적 인덱스 구조 도해 (ASCII)

기본 키가 물리적 데이터 배치를 어떻게 결정하는지 시각화한다.

```text
    [ Primary Key: Student_ID ]
    ┌─────────────┐
    │ PK: S001    │ ───▶ [ Disk Page A ] : {S001, Kim, CS}
    ├─────────────┤
    │ PK: S002    │ ───▶ [ Disk Page A ] : {S002, Lee, IT}
    ├─────────────┤
    │ PK: S003    │ ───▶ [ Disk Page B ] : {S003, Park, Math}
    └─────────────┘
    * Clustered Index: 기본 키의 순서대로 실제 데이터가 물리적으로 정렬됨.
    * 혜택: 범위 검색(Range Scan) 및 특정 데이터 탐색 시 I/O 횟수 최소화.
```

### 3. 심층 동작 원리: 기본 키 선정의 기준 (Selection Criteria)
설계자는 후보 키들 중 다음 기준에 따라 PK를 선정합니다.
1. **짧은 길이**: 인덱스 크기를 줄여 메모리 효율 극대화. (숫자형 선호)
2. **변하지 않는 값**: 참조 무결성 유지를 위해 갱신이 거의 없는 속성.
3. **보안성**: 주민번호처럼 노출 시 위험한 정보보다는 시스템이 생성한 **인조 키(Surrogate Key)** 선호.

### 4. 핵심 기술: 복합 기본 키 (Composite PK)
- 여러 속성을 묶어 하나의 PK로 지정하는 경우.
- 예: {학생ID, 학기, 과목ID} $\to$ 한 학생이 특정 학기에 특정 과목을 듣는 것은 유일함.
- 주의: 속성이 너무 많아지면 성능 저하를 유발하므로 2~3개 이내로 제한 권장.

- **📢 섹션 요약 비유**: 개체 무결성은 '이름표가 없는 물건은 분실물 센터(장애)로 가야 하는 원칙'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 기본 키(Primary) vs 대체 키(Alternate)

| 비교 항목 | 기본 키 (Primary Key) | 대체 키 (Alternate Key) |
|:---:|:---|:---|
| **지위** | 릴레이션의 대표 식별자 | 기본 키가 되지 못한 후보자 |
| **NULL 허용** | **절대 불가 (NOT NULL)** | **허용 가능** (비즈니스 규칙에 따름) |
| **개수** | **오직 1개** | 여러 개 존재 가능 |
| **인덱스** | 보통 클러스터드 인덱스 생성 | 보통 넌-클러스터드 인덱스 생성 |

### 2. 과목 융합 관점: 자료구조 및 보안
- **자료구조 융합**: 기본 키는 DBMS 내부에서 **B+Tree** 자료구조를 통해 관리되며, 검색 복잡도를 $O(\log N)$으로 고정하는 물리적 근거가 된다.
- **보안(Security) 융합**: 민감한 비즈니스 키 대신 의미 없는 **UUID**나 **Auto-increment** 값을 PK로 사용하는 것은 데이터 추론 공격을 방어하는 보안 설계의 일환이다.

- **📢 섹션 요약 비유**: 기본 키가 '정식 여권'이라면, 대체 키는 '학생증'이나 '사원증'과 같습니다. 나를 증명할 순 있지만 국가적 공신력(참조 기준)은 여권이 가집니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 마이크로서비스(MSA) 환경의 PK 설계
- **상황**: 여러 서비스로 분산된 DB 환경에서 중복되지 않는 PK를 생성해야 함.
- **판단**: 각 DB의 `Sequence`나 `Auto-increment`는 서버 간 중복 발생 위험이 있음.
- **결단**: **UUID v7 또는 Snowflake ID 기술 채택**. 시간 정보와 작업자 ID를 조합하여 전 세계적으로 유일한 PK를 보장하고, 분산 환경에서의 데이터 병합(Merge) 리스크를 원천 차단함.

### 2. 안티패턴: 가변적인 비즈니스 데이터를 PK로 사용
- **증상**: '이메일 주소'나 '로그인 ID'를 PK로 지정.
- **결과**: 사용자가 정보를 바꾸고 싶을 때, 이를 참조하는 수많은 하위 테이블의 외래 키(FK)까지 연쇄 수정(Cascading)해야 하여 시스템 전체 부하 및 잠금(Lock) 발생.
- **조치**: 비즈니스 데이터는 Unique 제약 조건으로 관리하고, PK는 시스템이 관리하는 **무의미한 일련번호(Sequnce ID)**를 사용할 것.

### 📢 기술사적 결언
> "기본 키는 데이터베이스라는 생태계의 **'절대적 좌표'**다. 좌표가 흔들리면 그 위에 세워진 모든 관계(Relationship)가 붕괴한다. 기술사는 단순히 유일한 속성을 찾는 것을 넘어, **데이터의 휘발성, 물리적 저장 밀도, 분산 환경에서의 생성 비용**을 종합적으로 고려하여 PK를 설계해야 한다. PK 설계의 정석은 '작고, 변하지 않으며, 존재가 확실한(NOT NULL)' 값을 찾는 것임을 잊지 말아야 한다."

- **📢 섹션 요약 비유**: 잘못된 PK 선정은 '모래 위에 집을 짓는 것'과 같아, 비즈니스가 커질수록(데이터가 쌓일수록) 기반부터 무너지게 됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (PK 최적화 시)

| 지표 | 도입 전 (Full Scan) | 도입 후 (PK Index) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **단일 행 검색 속도** | 수 초 ~ 분 | **밀리초 (ms)** | 체감 성능 1,000배 이상 향상 |
| **데이터 정합성 오류** | 빈번 발생 | **0% (Engine 강제)** | 데이터 품질 신뢰도 확보 |
| **스토리지 효율** | 중복 데이터 산재 | **최소화된 구조 유지** | 관리 비용 절감 |

### 2. 미래 전망
- **Identity-as-a-Service**: 클라우드 계층에서 전역 유일 ID를 발급해주는 서비스가 보편화되어, DB 엔진의 PK 생성 부하를 외부로 위임하는 구조 확산.
- **AI-Driven PK Mapping**: 서로 다른 시스템의 데이터를 통합할 때, PK가 다르더라도 논리적으로 같은 개체임을 AI가 판별하여 매핑해주는 **지능형 엔티티 레졸루션** 기술 발전.

- **📢 섹션 요약 비유**: 기술이 아무리 변해도 '나를 나라고 증명할 확실한 증거'라는 기본 키의 본질은 정보 문명의 영원한 유산이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **기본 키**는 학교에서 우리에게 하나씩 나누어주는 **"우리만의 고유한 번호(학번)"**와 같아요.
2. 이름이 같은 친구는 있을 수 있어도, 이 번호는 **세상에 단 한 명**에게만 주어지죠.
3. 이 번호는 **절대 비어있으면 안 되고(NULL 불가)**, 한 번 정해지면 졸업할 때까지 **바뀌지 않는 약속**이랍니다!
