# 79. NULL 무결성 (Null Integrity) - 필수 데이터의 부재 방지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 릴레이션의 특정 속성이 반드시 값을 가져야 함을 보장하기 위해, 데이터 입력이나 수정 시 **NULL 값(데이터 부재)의 유입을 시스템적으로 차단**하는 원칙이다.
> 2. **제약 수단**: SQL의 **`NOT NULL`** 제약 조건을 통해 구현되며, 비즈니스 로직상 반드시 존재해야 하는 핵심 정보(예: 이름, 연락처, 주문일자 등)의 정합성을 확보하는 최소한의 장치다.
> 3. **가치**: 정보의 불완전성으로 인한 쿼리 연산 오류(3치 논리 문제)와 집계 결과의 왜곡을 방지하며, 데이터베이스가 비즈니스의 실질적인 상태를 완벽하게 반영하도록 강제한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**NULL 무결성 (Null Integrity)**은 특정 속성값이 NULL 이 될 수 없음을 보장하는 제약 조건입니다. 이는 도메인 무결성의 한 변형으로 볼 수 있으며, 해당 속성이 개체를 설명하는 데 있어 '선택 사항'이 아닌 **'필수 사항'**임을 규정합니다.

### 2. 💡 비유: 회원 가입 신청서의 '필수 항목(*)'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        NULL 무결성 비유                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 온라인 쇼핑몰에 가입 신청서를 제출함.                                    │
│                                                                             │
│  1. 일반 항목 (NULL 허용): [추천인 아이디]. (없어도 가입됨)                      │
│  2. 필수 항목 (NOT NULL): [아이디], [비밀번호], [전화번호].                      │
│                                                                             │
│  - 만약 필수 항목을 비워두고(NULL) '제출' 버튼을 누르면, 시스템은 즉시            │
│    "필수 항목을 입력하세요!"라며 거부함.                                         │
│                                                                             │
│  → NULL 무결성은 데이터베이스가 **'구멍 뚫린 정보'**를 가지지 않도록 막아주는       │
│     기초적인 검문소와 같습니다.                                                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **연산의 모호성 해결**: 산술 연산이나 비교 연산 시 NULL이 끼어들면 결과가 `Unknown`이 되어버리는 논리적 결함을 원천 차단하기 위함.
- **데이터 활용도 극대화**: 마케팅 메일을 보내야 하는데 '이메일' 주소가 없는 데이터가 섞여 있으면, 전체 캠페인의 효율과 정확도가 떨어지기 때문.

- **📢 섹션 요약 비유**: NULL 무결성은 '앙꼬 없는 찐빵'이 만들어지지 않게 감시하는 요리사의 원칙과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. NULL 무결성의 물리적 강제 (SQL)

| 제약 형식 | 상세 내용 | 비유 |
|:---:|:---|:---|
| **NOT NULL** | 특정 컬럼에 NULL 값이 들어오는 것을 절대 금지. | 빈 칸 금지 |
| **DEFAULT** | 값이 입력되지 않았을 때 미리 정해둔 값을 채워 NULL 발생 방지. | 예비 부품 끼우기 |
| **Primary Key** | 개체 무결성에 의해 자동으로 `NOT NULL`과 `Unique` 성질 부여. | 신분증 필수 지참 |

### 2. NULL 무결성 검증 흐름 (ASCII)

데이터 변경 시 NULL 여부를 판단하는 논리적 공정을 시각화한다.

```text
    [ Client Request: INSERT {ID: 5, Name: NULL} ]
           │
    ┌──────▼────────────────── DBMS Engine ───────────────────────────────┐
    │                                                                     │
    │  1. [ Column Definition Scan ] : Name 컬럼 설정 확인.                │
    │     - Name: VARCHAR(20) NOT NULL  <─── "필수 항목 발견!"             │
    │                                                                     │
    │  2. [ Integrity Check ] : 입력값 검사.                               │
    │     - Input Value is NULL? ──▶ (YES)                                │
    │                                                                     │
    │  3. [ Exception Handling ] : "Column 'Name' cannot be null"         │
    │                                                                     │
    └────────────────────┬────────────────────────────────────────────────┘
                         ▼
    ┌──────────────── Transaction Result ────────────────────────────────┐
    │  [ Abort ] - 데이터 오염 방지를 위해 해당 요청을 완전히 거부           │
    └────────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: 3치 논리(3VL) 방어
- **원리**: NULL 무결성을 통해 데이터가 오직 `True` 또는 `False`의 결과만 내도록 유도.
- **효과**: `WHERE salary > 3000` 쿼리 실행 시, 연봉이 NULL인 사람이 결과에서 누락되어 통계가 틀려지는 현상을 근본적으로 해결함.

### 4. 핵심 기술: NULL과 인덱스 성능
- 많은 DBMS(특히 Oracle)는 전체 컬럼 값이 NULL인 경우 인덱스에 저장하지 않음.
- NULL 무결성이 지켜진 컬럼은 인덱스 스캔 효율이 극대화되며, **Index Only Scan**을 통해 테이블 접근 없이 결과 도출이 가능함.

- **📢 섹션 요약 비유**: NULL 무결성은 '길을 찾을 때 표지판이 없는 구간이 없도록' 미리 모든 곳에 이정표를 세워두는 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. NULL 무결성 vs 기본값(Default) 설정

| 비교 항목 | NULL 무결성 (NOT NULL) | 기본값 설정 (DEFAULT) |
|:---:|:---|:---|
| **동작 방식** | 잘못된 입력을 **거부**함. | 값이 없으면 **대신 채워줌**. |
| **강제성** | 강함 (사용자 책임) | 유연함 (시스템 책임) |
| **사용 사례** | 아이디, 비밀번호, 고유 코드 | 가입일자(SYSDATE), 상태(0), 수량(1) |
| **결과** | NULL이 절대 없음. | NULL을 방지하지만 유의미한 값 유지. |

### 2. 과목 융합 관점: 소프트웨어 공학 및 보안
- **소프트웨어 공학(SE) 융합**: **방어적 프로그래밍(Defensive Programming)** 관점에서 DB의 NOT NULL 제약은 애플리케이션의 NullPointerException을 방지하는 가장 확실한 백엔드 안전장치다.
- **보안(Security) 융합**: 필수 정보의 누락은 감사 추적(Audit Trail)의 단절을 야기한다. NULL 무결성은 모든 행위에 대한 **책임 추적성(Accountability)**을 확보하는 보안의 기초 데이터 품질 요건이다.

- **📢 섹션 요약 비유**: NOT NULL이 '입장을 거절하는 깐깐한 경비원'이라면, DEFAULT는 '빈손으로 온 손님에게 입장권을 쥐여주는 안내원'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 기존 테이블에 NOT NULL 제약 추가하기
- **상황**: 이미 수만 건의 데이터가 있는 '주소' 컬럼을 필수(NOT NULL)로 바꾸고 싶음.
- **판단**: 기존 데이터 중 주소가 비어있는(NULL) 것이 하나라도 있으면 제약 조건 생성 실패.
- **결단**: **먼저 NULL 데이터를 찾아서 '주소미상' 등의 더미 값으로 일괄 업데이트 수행**. 그 후 `ALTER TABLE`을 통해 NOT NULL 제약을 부여하여 향후 유입될 데이터의 품질 보장.

### 2. 안티패턴: 모든 컬럼에 습관적으로 NULL 허용
- **증상**: 설계를 고민하기 귀찮아서 모든 컬럼을 Nullable 하게 만듦.
- **결과**: `WHERE` 절 조건마다 `IS NOT NULL`을 붙여야 하는 지저분한 코드가 양산되고, 데이터 집계 시 원인 불명의 수치 오류 발생.
- **조치**: **기본 설정을 NOT NULL로 가져가되**, 정말로 값이 없을 수 있는 경우에만 '화이트리스트' 방식으로 NULL을 허용하는 **Strict Design** 지향.

### 📢 기술사적 결언
> "NULL 무결성은 데이터베이스 설계자의 **'성실함'**을 보여준다. 빈틈없는 설계는 빈틈없는 비즈니스를 만든다. 기술사는 단순히 에러를 피하기 위해 NULL을 허용할 것이 아니라, **비즈니스의 필수 도메인**을 정확히 식별하여 데이터의 '밀도'를 높여야 한다. 데이터에 구멍(NULL)이 많을수록 인덱스의 힘은 약해지고, 분석의 칼날은 무뎌진다는 사실을 명심해야 한다."

- **📢 섹션 요약 비유**: 구멍 난 양말(NULL 데이터)을 신고 달리면(쿼리 수행), 언젠가는 발이 아프거나 넘어지게(시스템 장애) 됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (NULL 무결성 강화 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **프로그램 런타임 에러** | 빈번 (NPE 발생) | **현저히 감소** | 앱 안정성 40% 향상 |
| **통계/집계 정확도** | 왜곡 가능성 높음 | **100% 신뢰** | 정확한 리포팅 가능 |
| **데이터 정제 비용** | 높음 (결측치 처리) | **최소화** | 데이터 분석 공수 절감 |

### 2. 미래 전망
- **Non-nullable by Default**: 최신 데이터 언어(Rust 등)의 철학을 따라, 모든 DB 테이블 생성 시 기본값이 NOT NULL이 되고 NULL 허용 시에만 `OPTIONAL`을 선언하는 아키텍처 확산.
- **Context-aware Null Blocking**: 무조건 막는 것이 아니라, 사용자의 역할이나 상황에 따라 동적으로 필수 값을 체크하는 **지능형 컨텍스트 무결성** 기술 발전.

- **📢 섹션 요약 비유**: 미래의 데이터 관리에서도 '명확하지 않은 정보는 정보가 아니다'라는 NULL 무결성의 원칙은 더욱 엄격해질 것입니다.
