# 91. 슈퍼타입과 서브타입 엔티티 (Supertype & Subtype Entity) - 상속 관계 모델링

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 공통적인 속성을 가진 상위 개체(**슈퍼타입**)와 특정 조건에 따라 고유한 속성을 분리하여 가지는 하위 개체(**서브타입**)들 간의 계층적 관계를 정의한 고도의 추상화 모델링 기법이다.
> 2. **제약 조건**: 모든 개체가 반드시 서브타입에 속해야 하는지(**Total/Partial**), 그리고 한 개체가 여러 서브타입에 동시에 속할 수 있는지(**Disjoint/Overlapping**)에 따라 데이터 무결성의 수준이 결정된다.
> 3. **가치**: 객체지향의 **상속(Inheritance)** 개념을 데이터 모델에 투영하여 중복 속성을 제거하고, 비즈니스 규칙을 명확히 함으로써 시스템의 확장성과 유지보수성을 극대화한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**슈퍼타입과 서브타입 (Supertype & Subtype)**은 엔티티들 간의 공통점과 차이점을 논리적으로 분류하여 표현하는 기법입니다. 이를 **ISA (is-a)** 관계라고도 부릅니다.
- **슈퍼타입**: 서브타입들의 공통 속성을 통합하여 관리하는 부모 엔티티.
- **서브타입**: 슈퍼타입의 속성을 상속받으면서 자신만의 특수한 속성을 가지는 자식 엔티티.

### 2. 💡 비유: 가전제품 카테고리
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        슈퍼타입/서브타입 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 슈퍼타입 (가전제품): 모든 제품이 공통으로 가지는 정보.                       │
│     - [모델번호], [제조사], [가격].                                            │
│                                                                             │
│  2. 서브타입 (냉장고, TV): 각 제품만의 특별한 정보.                             │
│     - 냉장고: [용량], [문 개수].                                               │
│     - TV: [화면 크기], [해상도].                                               │
│                                                                             │
│  → 모두가 **'가전제품'**이지만, 종류에 따라 **'궁금한 내용'**이 다를 때 사용하는     │
│     똑똑한 분류법입니다!                                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 중복의 딜레마**: 공통 속성을 모든 테이블에 복사하자니 낭비가 심하고, 하나로 합치자니 특정 유형에만 필요한 빈칸(NULL)이 너무 많아짐.
- **비즈니스 복잡도 증가**: '사원' 중에서도 '정규직', '계약직', '아르바이트' 등 유형별로 관리해야 할 업무 규칙이 고도화됨.

- **📢 섹션 요약 비유**: 슈퍼타입/서브타입은 '몸통(슈퍼)'은 하나인데 상황에 따라 다른 '옷(서브)'을 갈아입는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 슈퍼타입/서브타입 모델의 제약 조건 분류

| 구분 기준 | 유형 | 상세 내용 | 비유 |
|:---:|:---|:---|:---|
| **완전성 (Completeness)** | **Total (전체)** | 슈퍼타입 개체는 반드시 하나의 서브타입에 속해야 함. | 미분류 허용 안 함 |
| | **Partial (부분)** | 어느 서브타입에도 속하지 않는 개체가 존재 가능. | 기타(Etc) 허용 |
| **배타성 (Disjointness)** | **Disjoint (배타)** | 한 개체는 **오직 하나**의 서브타입에만 속함. | 짬짜면 불가 |
| | **Overlapping (중복)** | 한 개체가 **여러 개**의 서브타입에 동시 참여 가능. | 다중 전공 가능 |

### 2. 슈퍼타입/서브타입 논리 구조 도해 (ASCII)

분류자(Discriminator)를 통해 서브타입을 결정하는 구조를 시각화한다.

```text
           [ Supertype: 사 원 ]
           ┌──────────────────┐
           │ - 사번 (PK)      │
           │ - 성명           │
           │ - 구분코드 (D)   │ <--- [ Discriminator ]
           └────────┬─────────┘
                    │
          ┌─────────┴─────────┐
          ▼ (D='Full')        ▼ (D='Part')
    ┌─────────────┐     ┌─────────────┐
    │ Sub: 정규직  │     │ Sub: 계약직  │
    ├─────────────┤     ├─────────────┤
    │ - 호봉       │     │ - 계약종료일  │
    │ - 수당       │     │ - 파견업체    │
    └─────────────┘     └─────────────┘
```

### 3. 심층 동작 원리: 분류자 (Discriminator)
- **개념**: 슈퍼타입 개체가 어떤 서브타입에 속하는지 결정해주는 속성입니다.
- **역할**: 쿼리 시 어떤 서브타입 테이블과 조인(Join)해야 할지 옵티마이저에게 길을 안내하는 지표가 됩니다.

### 4. 핵심 기술: 물리적 변환 전략 (Mapping to Tables)
1. **One-to-One (Single Table)**: 모든 정보를 하나의 큰 테이블로 합침. (성능 좋음, NULL 많음)
2. **Roll-down (Subtype Table)**: 슈퍼타입 속성을 각 서브타입 테이블에 복사. (조회 빠름, 중복 발생)
3. **Roll-up (Supertype Table)**: 공통 부분과 개별 부분을 각각 테이블로 유지. (가장 정규화된 형태)

- **📢 섹션 요약 비유**: 분류자는 '식당 입구의 번호표'와 같아서, 번호에 따라 한식(서브1)으로 갈지 일식(서브2)으로 갈지 정해줍니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 물리적 변환 전략 비교 매트릭스

| 전략 명칭 | 적용 방식 | 장점 | 단점 |
|:---:|:---|:---|:---|
| **Single Table** | Super+Sub 통합 | Join 불필요, 조회 최고 | NULL 과다, 테이블 비대화 |
| **Subtype Partition** | Sub별 별도 테이블 | 특정 타입 집중 조회 시 유리 | 전체 조회 시 Union 필요 |
| **Identity Entity** | Super/Sub 1:1 관계 | 저장 효율성, 정규화 준수 | 잦은 Join으로 성능 저하 |

### 2. 과목 융합 관점: 소프트웨어 공학 및 아키텍처
- **소프트웨어 공학(SE) 융합**: 객체 지향의 **다형성(Polymorphism)**을 데이터 계층에서 구현하는 표준 기법이며, ORM의 상속 전략(Table-per-class 등)과 직접 연동된다.
- **클라우드 아키텍처 융합**: 데이터 양이 폭증하는 클라우드 환경에서는 성능을 위해 **Single Table 전략**을 선호하며, 대신 Sparse 한 컬럼 관리를 위해 **컬럼 기반 저장소(Columnar Store)**를 활용하기도 한다.

- **📢 섹션 요약 비유**: 변환 전략은 '가족 일기를 한 권에 다 쓸지(Single), 아이들마다 따로 쓸지(Partition)'를 결정하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 결제 수단(카드, 계좌, 포인트)의 모델링
- **상황**: 핀테크 앱에서 다양한 결제 수단을 지원해야 함.
- **판단**: 모든 결제 수단은 '결제금액', '승인번호'라는 공통점이 있지만, 카드번호나 은행계좌번호 등은 각자 다름.
- **결단**: **슈퍼타입(Payment)과 서브타입(Card, Bank, Point) 모델링 적용**. 공통 승인 로직은 슈퍼타입을 통해 처리하고, 상세 정산 로직은 서브타입별로 분리하여 데이터 정합성과 확장성 확보.

### 2. 안티패턴: 무분별한 공통 속성 추출
- **증상**: "사원"과 "자동차"가 둘 다 '이름'과 '가격'을 가진다고 해서 "이름가격개체"라는 이상한 슈퍼타입을 만듦.
- **결과**: 업무적으로 아무 관련 없는 데이터가 섞여 논리 모델이 붕괴되고 쿼리 가독성 최악.
- **조치**: **의미적으로 IS-A 관계가 성립하는 경우**에만 슈퍼타입/서브타입을 사용할 것.

### 📢 기술사적 결언
> "슈퍼타입/서브타입 모델링은 **'데이터의 혈통'**을 정의하는 작업이다. 혈통이 명확해야 비즈니스 규칙이 꼬이지 않는다. 기술사는 단순히 테이블을 쪼개는 기술을 넘어, **배타적 관계와 중복적 관계**의 미묘한 차이를 비즈니스 요건에서 읽어낼 줄 알아야 한다. 물리 설계 단계에서는 시스템의 **Read/Write 비중**을 고려하여 성능과 유연성 사이의 '최적의 변환 전략'을 도출하는 아키텍처적 결단력을 갖춰야 한다."

- **📢 섹션 요약 비유**: 억지 슈퍼타입은 '바퀴가 있다고 자동차와 자전거를 같은 엔티티로 묶는 것'과 같아, 정작 수리(유지보수)할 때 큰 혼란을 줍니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (정교한 모델링 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 중복도** | 높음 (공통 속성 산재) | **최소화 (통합 관리)** | 저장 효율 30% 향상 |
| **코드 가독성** | 복잡한 CASE문 남발 | **명확한 상속 구조 활용** | 개발 생산성 50% 향상 |
| **스키마 유연성** | 신규 타입 추가 시 대공사 | **서브타입만 추가** | 요구사항 변경 대응력 극대화 |

### 2. 미래 전망
- **Schema-on-Read와의 결합**: 슈퍼타입은 정형으로, 서브타입의 고유 속성은 JSONB 등 비정형으로 담는 **하이브리드 슈퍼/서브타입 모델**이 대세가 될 것임.
- **AI-Driven Classification**: 방대한 원시 데이터를 분석하여 AI가 스스로 공통 분모를 찾아 슈퍼타입을 제안하는 **지능형 데이터 모델링** 기술 고도화.

- **📢 섹션 요약 비유**: 미래의 데이터 모델은 고정된 족보가 아니라, 비즈니스의 변화에 따라 스스로 세포 분열하고 진화하는 '살아있는 유기체'가 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **슈퍼타입/서브타입**은 동물들을 분류할 때 **"큰 바구니와 작은 바구니"**를 나누는 것과 같아요.
2. "동물"이라는 큰 바구니에는 다리 개수를 적고, "사자" 바구니에는 갈기 모양을, "새" 바구니에는 날개 색깔을 적는 거죠.
3. 이렇게 하면 모든 동물에게 날개 색깔을 물어보는 **바보 같은 실수를 하지 않고 똑똑하게 정리**할 수 있답니다!
