# 81. 개체 (Entity) - 현실 세계의 독립적 존재

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 현실 세계에서 유무형의 형태로 존재하며, 다른 것과 명확히 구별되는 **독립적인 정보의 단위**로서 데이터베이스에 저장하고 관리해야 할 대상(사람, 사물, 장소, 사건 등)이다.
> 2. **구성 요소**: 공통된 속성들의 틀을 정의하는 **개체 타입(Entity Type)**과, 실제 데이터 값이 채워진 구체적인 사례인 **개체 인스턴스(Entity Instance)**로 구성되며, ERD에서는 **사각형**으로 표기한다.
> 3. **가치**: 데이터베이스 설계의 가장 기초적인 원재료이며, 개체를 정확히 식별하고 정의하는 과정이 곧 비즈니스 요구사항을 데이터 구조로 변환하는 핵심적인 **추상화(Abstraction)** 단계가 된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**개체 (Entity)**는 데이터베이스에 기록될 가치가 있는 실체입니다. 유일한 식별자에 의해 구별 가능해야 하며, 업무 프로세스에서 반드시 필요로 하는 정보를 담고 있어야 합니다.

### 2. 💡 비유: 레고 블록 세트의 피규어와 부품
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        개체(Entity) 개념 비유                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 개체 타입 (피규어 틀): "경찰관"이라는 레고 인형의 설계도.                    │
│     - 모든 경찰관 피규어는 모자, 제복, 이름표 칸이 있음. (공통 속성)              │
│                                                                             │
│  2. 개체 인스턴스 (실제 인형): "김 순경", "박 경위"라는 실제 레고 인형들.         │
│     - 설계도에 따라 실제 얼굴과 이름이 그려진 실체.                              │
│                                                                             │
│  → 데이터베이스라는 레고 세상을 만들기 위해 가장 먼저 준비해야 할 **'등장인물'**이   │
│     바로 개체입니다!                                                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **관리 대상의 명확화**: 전산화하려는 비즈니스 도메인에서 무엇을 데이터로 남길 것인지(예: '책'을 남길 것인가, '도서 대출 행위'를 남길 것인가)를 결정하기 위해 도입.
- **논리적 분리**: 서로 다른 성격의 데이터(예: 학생 정보와 성적 정보)를 섞이지 않게 별도의 주머니(엔티티)에 담아 관리하기 위함.

- **📢 섹션 요약 비유**: 개체는 비즈니스라는 문장에서 '주어' 역할을 수행하는 문장의 주인공들입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 개체의 분류 및 특징

| 분류 기준 | 유형 | 상세 내용 | 예시 |
|:---:|:---|:---|:---|
| **유형/무형** | **유형 개체** | 물리적으로 형태가 있는 것. | 자동차, 사원, 상품 |
| | **무형 개체** | 개념적으로만 존재하는 것. | 부서, 교과목, 계좌 |
| | **사건 개체** | 업무 수행 중 발생하는 것. | 주문, 예약, 결제 |
| **자립성** | **강한 개체** | 독자적으로 식별 및 존재 가능. | 학생 |
| | **약한 개체** | 다른 개체의 존재에 의존함. | 부양가족 (사원이 있어야 존재) |

### 2. 개체 타입 vs 개체 인스턴스 vs 개체 집합 (ASCII)

개념의 위계 구조를 시각화한다.

```text
    [ Entity Type ] (Schema)
    : Student { SID, Name, Dept }  <-- "틀" 정의
           │
           ▼
    [ Entity Instances ] (Data)
    : i1 = { 101, "Kim", "CS" }    <-- "실체" 1
    : i2 = { 102, "Lee", "Math" }  <-- "실체" 2
           │
           ▼
    [ Entity Set ] (Relation)
    ┌──────┬──────┬──────┐
    │ SID  │ Name │ Dept │         <-- "모임" 전체
    ├──────┼──────┼──────┤
    │ 101  │ Kim  │ CS   │
    │ 102  │ Lee  │ Math │
    └──────┴──────┴──────┘
```

### 3. 심층 동작 원리: 개체 식별성 (Identifiability)
- **원리**: 모든 개체 인스턴스는 유일하게 식별 가능해야 함.
- **수단**: 하나 이상의 속성(Attribute)을 묶어 **유일 식별자(UID)**를 부여함.
- **영향**: 식별자가 없는 개체는 데이터베이스에서 관리할 수 없으며, 중복 데이터로 간주되어 무결성을 해치게 됨.

### 4. 핵심 기술: 엔티티 프로파일링 (Entity Profiling)
- 설계 단계에서 개체가 가진 속성들을 분석하여 최적의 타입(INT, VARCHAR 등)을 지정하고, 예상되는 데이터 크기를 가늠하는 과정.
- 개체의 발생 빈도(Frequency)에 따라 물리적인 파티셔닝 전략을 미리 수립함.

- **📢 섹션 요약 비유**: 개체 타입은 '붕어빵 틀'이고, 개체 인스턴스는 '실제 구워져 나온 붕어빵'입니다. 팥이 들었든 크림이 들었든 모두 붕어빵이라는 개체 타입에 속합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 개체(Entity) vs 속성(Attribute) vs 관계(Relationship)

| 비교 항목 | 개체 (Entity) | 속성 (Attribute) | 관계 (Relationship) |
|:---:|:---|:---|:---|
| **역할** | **행위의 주체** | 개체의 특징 | 개체 간의 상호작용 |
| **비유** | **명사 (사람, 사물)** | 형용사 (키, 색상) | 동사 (사다, 가르치다) |
| **ERD 표기** | **사각형** | 타원 | 마름모 |
| **식별자** | 가짐 | 없음 | 보통 개체의 식별자를 빌려 씀 |

### 2. 과목 융합 관점: 소프트웨어 공학 및 운영체제
- **소프트웨어 공학(SE) 융합**: 개체는 객체 지향 프로그래밍의 **클래스(Class)**와 대응되며, 비즈니스 로직을 수행하는 **엔티티 객체(Domain Object)**로 구체화된다.
- **운영체제(OS) 융합**: 대량의 개체 데이터를 저장할 때, 개체의 크기에 따라 OS의 **디스크 블록(Block)** 내에 몇 개의 개체를 담을 것인지(Blocking Factor) 결정하는 물리적 최적화가 필요하다.

- **📢 섹션 요약 비유**: 개체가 '자동차'라면, 속성은 '차 색깔'이고, 관계는 '차가 도로를 달린다'는 현상입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 핵심 개체(Core Entity)의 추출
- **상황**: 병원 예약 시스템을 설계함.
- **판단**: 환자, 의사, 약품은 눈에 보이는 '유형 개체'이므로 찾기 쉬움. 하지만 '진료 내역'이나 '처방'은 눈에 보이지 않지만 매우 중요한 '사건 개체'임.
- **결단**: **'처방'을 독립된 엔티티로 분리**. 처방 일시, 처방전 번호 등의 고유 속성을 부여하여 진료 행위의 이력을 완벽하게 추적할 수 있도록 설계함.

### 2. 안티패턴: 너무 거대한 개체 (God Entity) 설계
- **증상**: '사용자' 엔티티 하나에 기본 정보, 로그인 정보, 구매 정보, 배송지 주소를 모두 넣음.
- **결과**: 데이터 중복이 발생하고, 주소 하나를 고칠 때마다 사용자 행 전체에 락(Lock)이 걸려 성능 저하.
- **조치**: **엔티티 분할(Normalization)**을 통해 업무 성격에 맞게 쪼개고, 관계(FK)로 연결하여 응집도를 높임.

### 📢 기술사적 결언
> "개체는 데이터베이스의 **'벽돌'**이다. 벽돌의 규격과 강도가 명확해야 튼튼한 집을 지을 수 있다. 기술사는 단순히 명사를 뽑아내는 수준을 넘어, 해당 개체가 **영속적으로 보관될 가치**가 있는지, 그리고 **유일하게 식별 가능한 핵심 속성**이 존재하는지를 철저히 검증해야 한다. 엔티티 정의 단계에서의 작은 실수(예: 식별자 누락)가 훗날 시스템 전체를 다시 짜야 하는 대공사로 이어진다는 사실을 명심해야 한다."

- **📢 섹션 요약 비유**: 거대 개체를 만드는 것은 '집 안의 모든 가구를 하나의 큰 상자에 붙여놓은 것'과 같아, 의자 하나를 옮기려 해도 집 전체를 들썩여야 하는 비효율을 낳습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (명확한 개체 정의 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 정합성** | 낮음 (정보 혼재) | **높음 (독립적 관리)** | 데이터 오류 70% 감소 |
| **요구사항 변경 대응** | 어려움 (구조 복잡) | **유연함 (엔티티만 수정)** | 유지보수 속도 2배 향상 |
| **시스템 확장성** | 구조적 한계 | **신규 엔티티 추가 용이** | 비즈니스 민첩성 확보 |

### 2. 미래 전망
- **Micro-Entities**: MSA 아키텍처 환경에서 개체들이 아주 작게 쪼개져 서비스별로 분산 관리되는 추세 가속화.
- **AI-Discovery of Entities**: 비정형 텍스트(계약서, 상담록)에서 AI가 자동으로 엔티티를 추출하고 관계를 설정하는 **자동 모델링 기술** 보편화.

- **📢 섹션 요약 비유**: 미래의 개체는 단순한 저장 단위를 넘어, 스스로의 상태와 관계를 AI와 소통하는 '스마트 데이터 입자'가 될 것입니다.
