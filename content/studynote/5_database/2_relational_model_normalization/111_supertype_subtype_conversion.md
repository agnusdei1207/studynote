# 111. 슈퍼타입/서브타입 변환 - One-to-One, Roll-up, Roll-down

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 개념 모델링의 상속 구조(ISA 관계)를 관계형 데이터베이스의 물리적 테이블 구조로 매핑하는 기술이며, 데이터의 **집적도와 접근 성능**을 고려한 전략적 선택이다.
> 2. **3대 전략**: 개별 테이블로 유지하는 **One-to-One**, 슈퍼타입으로 통합하는 **Roll-up(Single Table)**, 서브타입으로 속성을 내려보내는 **Roll-down(Table-per-type)**으로 구분된다.
> 3. **가치**: 비즈니스의 데이터 활용 패턴(전체 조회 위주 vs 개별 타입 위주)에 최적화된 물리 레이아웃을 결정함으로써, 조인 비용을 줄이고 스토리지 효율을 극대화한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**슈퍼타입/서브타입 변환**은 논리 모델의 계층 구조를 물리 테이블로 변환하는 기법입니다. 관계형 DB는 상속(Inheritance) 기능을 직접 지원하지 않으므로, 성능과 정합성 사이의 트레이드오프를 따져 세 가지 방식 중 하나를 선택합니다.

### 2. 💡 비유: 가족의 일기 쓰기 방식
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        변환 전략 비유 (가족 일기)                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. One-to-One (각자 쓰기): 부모님 일기장 따로, 자녀 일기장 따로 씀.             │
│     - 사생활(데이터 격리) 보호에 좋지만, 온 가족 일과를 보려면 다 모아야 함.       │
│                                                                             │
│  2. Roll-up (가족 일기 통합): 큰 일기장 하나에 온 가족 내용을 다 적음.           │
│     - 한 권만 보면 끝나서 정말 빠름! (통합 조회 유리)                            │
│     - 자녀가 없는 집은 일기장 뒷부분이 텅 비게 됨. (NULL 발생)                   │
│                                                                             │
│  3. Roll-down (자녀에게 상속): 부모님 일과를 자녀 일기장에 각각 복사해서 씀.      │
│     - 자녀 일기장만 봐도 부모님 소식을 알 수 있음. (개별 조회 유리)               │
│                                                                             │
│  → 데이터의 **'조회 패턴'**에 따라 일기장을 합칠지 나눌지 결정하는 것입니다!       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **조인(Join) 오버헤드**: 슈퍼/서브 관계를 무조건 쪼개면 조회할 때마다 조인이 발생하여 응답 속도가 떨어짐.
- **NULL 값의 경제성**: 대규모 데이터에서 불필요한 컬럼(NULL)이 많아지면 디스크 낭비와 Full Scan 부하가 커짐.

- **📢 섹션 요약 비유**: 변환 전략은 '관리의 정교함'과 '조회의 속도' 사이에서 가장 효율적인 타협점을 찾는 설계의 묘미입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 3대 변환 전략 상세 분석

| 전략 명칭 | 물리적 구현 방식 | 장점 | 단점 |
|:---:|:---|:---|:---|
| **One-to-One** | 슈퍼/서브를 각각 테이블로 유지. | 정규화 준수, 관리 명확. | 잦은 Join 발생, 성능 저하. |
| **Roll-up** | 슈퍼타입 테이블에 모든 속성 통합. | **조인 없음, 성능 최상**. | NULL 과다, 테이블 비대화. |
| **Roll-down** | 각 서브타입 테이블에 슈퍼 속성 복사. | 타입별 조회 최적화. | 전체 조회 시 Union 필요. |

### 2. 물리적 변환 아키텍처 도해 (ASCII)

하나의 논리 모델이 세 가지 물리 구조로 변하는 모습을 시각화한다.

```text
    [ Logic Model ] : 사원(Super) - 정규직(Sub), 계약직(Sub)
    
    (A) One-to-One (기본)
    ┌──────────┐ 1:1 ┌──────────┐
    │ 사 원 (T) │ ───│ 정규직 (T)│
    └──────────┘ 1:1 └──────────┘
                 └───│ 계약직 (T)│
                     └──────────┘
    
    (B) Roll-up (상향 통합)
    ┌──────────────────────────────────┐
    │ [ 사 원 테이블 (All Columns) ]    │ <--- Single Table
    └──────────────────────────────────┘
    
    (C) Roll-down (하향 분할)
    ┌───────────┐      ┌───────────┐
    │ 정규직 (T) │      │ 계약직 (T) │ <--- Subtype Tables
    │ (+사원속성)│      │ (+사원속성)│
    └───────────┘      └───────────┘
```

### 3. 심층 동작 원리: 분류자(Discriminator)의 역할
- **Roll-up 전략** 시, 해당 행이 정규직인지 계약직인지 구분하는 '구분코드' 컬럼이 필수임.
- **인덱스 활용**: 구분코드에 인덱스를 생성하여 특정 서브타입 데이터만 빠르게 필터링함.

### 4. 핵심 기술: 데이터 타입의 희소성 (Sparsity) 관리
- Roll-up 시 발생하는 대량의 NULL 값을 효율적으로 처리하기 위해, DBMS의 **Sparse Column** 최적화 기술이나 컬럼 기반 저장 구조를 활용하기도 함.

- **📢 섹션 요약 비유**: Roll-up은 '모든 공구를 큰 공구함 하나에 다 넣는 것'이고, Roll-down은 '드라이버 세트와 망치 세트를 각각 가방에 넣는 것'입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 전략 선택을 위한 의사결정 매트릭스

| 고려 요소 | One-to-One 권장 | Roll-up 권장 | Roll-down 권장 |
|:---:|:---:|:---:|:---:|
| **데이터량** | 적음 | **많음** | 많음 |
| **조회 패턴** | 개별/통합 혼재 | **통합 조회가 압도적** | **타입별 조회가 압도적** |
| **서브타입 수** | 많음 | **적음 (2~3개)** | 적음 |
| **속성 공통성** | 낮음 | **높음** | 낮음 |

### 2. 과목 융합 관점: 소프트웨어 공학 및 운영체제
- **소프트웨어 공학(SE) 융합**: ORM(JPA/Hibernate)의 상속 전략인 **SINGLE_TABLE(Roll-up)**, **JOINED(One-to-One)**, **TABLE_PER_CLASS(Roll-down)**와 정확히 일치하는 개념이다.
- **운영체제(OS) 융합**: Roll-up 구조는 데이터의 **Spatial Locality**를 높여 한 번의 디스크 블록 읽기로 모든 정보를 가져오게 함으로써, OS 캐시 히트율을 극대화한다.

- **📢 섹션 요약 비유**: 설계 패턴을 고르는 것은 '책을 장르별로 꽂을지(Roll-down), 아니면 한 권의 백과사전으로 만들지(Roll-up)'를 정하는 지식 분류법과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 사용자 계정(개인, 법인) 테이블 설계
- **상황**: 쇼핑몰 앱에서 개인 회원과 법인 회원을 관리해야 함.
- **판단**: 두 회원은 로그인 ID, 비번 등 공통점이 90% 이상이고, 관리자 화면에서 전체 회원을 검색하는 일이 잦음.
- **결단**: **Roll-up 전략(Single Table) 채택**. 'Member' 테이블 하나에 통합하고, 법인에만 필요한 사업자번호 등은 NULL 허용으로 설계. 조인 없는 초고속 전체 검색 성능 확보.

### 2. 안티패턴: 복잡한 서브타입의 무리한 Roll-up
- **증상**: 서브타입이 10개가 넘고 각자 속성도 수십 개인 경우를 하나로 합침.
- **결과**: 테이블 컬럼이 수백 개(Wide Table)가 되어 한 행의 길이가 너무 긺. 데이터 조회 시 불필요한 컬럼 로드로 인해 I/O 효율 급락.
- **조치**: 이럴 경우 **One-to-One 방식**으로 분리하여 데이터의 밀도를 유지할 것.

### 📢 기술사적 결언
> "슈퍼타입/서브타입 변환은 **'물리적 수렴과 발산'**의 미학이다. 논리적으로는 분화된 비즈니스를 물리적으로 어떻게 응축(Roll-up)할 것인가를 결정하기 때문이다. 기술사는 이론적인 정규화 수칙에 얽매이지 말고, **애플리케이션의 SQL 프로파일링 결과**에 근거하여 '성능 중심의 모델 변환'을 주도해야 한다. 변환 전략의 선택이 곧 시스템의 **확장성(Scalability)**과 **응답성(Responsiveness)**을 결정짓는 핵심 설계 포인트임을 명심해야 한다."

- **📢 섹션 요약 비유**: 무리한 통합은 '전교생의 일기장을 한 권으로 묶는 것'과 같아, 책이 너무 무거워져서 아무도 들 수 없게(I/O 임계치 도달) 됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (최적 변환 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **집계 쿼리 속도** | 수 초 (Join 발생) | **밀리초 (ms)** | 10배 이상의 성능 향상 |
| **스토리지 점유** | 중복 데이터 상존 | **압축 및 통합 관리** | 용량 효율성 20% 개선 |
| **개발 가독성** | 복잡한 Join SQL | **단순 Select SQL** | 코드 복잡도 감소 |

### 2. 미래 전망
- **JSON Hybrid Storage**: 슈퍼타입은 고정 컬럼으로, 서브타입의 다양한 고유 속성은 **JSONB(Postgres)** 타입으로 담는 '동적 스키마' 방식이 차세대 표준으로 부상.
- **AI-Driven Schema Refactoring**: 운영 중인 DB의 쿼리 로그를 AI가 분석하여, 실시간으로 Roll-up에서 One-to-One으로 구조를 변경하는 **자율 튜닝 스키마** 기술 발전.

- **📢 섹션 요약 비유**: 미래의 변환 기술은 필요할 때만 합쳐지고 다시 나뉘는 '액체 괴물'과 같은 유연함을 지향할 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **슈퍼타입/서브타입 변환**은 장난감 상자를 정리할 때, **"모두 한 큰 상자에 담을지, 아니면 따로 나눌지"** 정하는 거예요.
2. 같이 놀 때는 한 상자에 있는 게 편하지만(Roll-up), 상자가 너무 무거워지면 **나눠 담는 게(One-to-One)** 더 좋겠죠.
3. 우리 친구들이 장난감을 **얼마나 많이, 어떻게 가지고 노는지** 보고 가장 좋은 방법을 고르는 거랍니다!
