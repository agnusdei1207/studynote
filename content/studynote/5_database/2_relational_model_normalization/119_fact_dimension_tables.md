# 119. 팩트 테이블(Fact Table)과 차원 테이블(Dimension Table)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 차원 모델링의 양대 핵심 구성 요소로, 비즈니스 활동의 수치적 결과물인 **팩트(Fact)**와 그 수치가 발생한 맥락(시간, 장소, 대상 등)을 설명하는 **차원(Dimension)**으로 데이터를 이원화하여 관리한다.
> 2. **상호작용**: 팩트 테이블은 대량의 트랜잭션 수치와 각 차원을 가리키는 외래 키(FK)들의 조합으로 구성되며, 차원 테이블은 사용자 질의의 조건(Filter)과 그룹화(Group by)의 기준이 되는 다양한 속성 정보를 제공한다.
> 3. **가치**: 데이터의 '의미'와 '수치'를 분리함으로써 저장 공간을 효율적으로 사용하고, 수조 건의 팩트 데이터에 대한 다차원 분석 조인 성능을 최적화하는 물리적 기반을 마련한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **팩트 테이블 (Fact Table)**: 비즈니스의 핵심 지표인 측정값(Measure)을 담은 테이블입니다. (예: 매출액, 수량, 클릭 수)
- **차원 테이블 (Dimension Table)**: 팩트에 담긴 숫자가 '언제', '어디서', '누구에 의해' 발생했는지에 대한 환경 정보를 담은 테이블입니다. (예: 날짜, 매장, 고객, 상품)

### 2. 💡 비유: 가계부의 '금액'과 '메모'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        팩트 vs 차원 테이블 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 팩트 테이블 (금액 적는 칸):                                               │
│     - "10,000원", "20,000원"... 오직 숫자들만 가득함.                           │
│     - 칸이 아주 좁고(컬럼 수 적음), 매일매일 엄청나게 많이 적힘. (행 수 많음)      │
│                                                                             │
│  2. 차원 테이블 (메모 및 분류):                                               │
│     - "식비", "스타벅스 판교점", "친구와 커피" ... 상세한 설명들.                │
│     - 칸이 넓고(컬럼 수 많음), 분류가 정해져 있어 자주 안 바뀜. (행 수 적음)      │
│                                                                             │
│  → 숫자를 모아둔 **'통장(팩트)'**과 그 돈의 사연을 적은 **'일기장(차원)'**의 관계!   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 비대화 해소**: 모든 상세 텍스트 정보를 수억 건의 트랜잭션 행마다 적어두면 저장 공간이 폭발함. 이를 코드(ID)화하여 차원 테이블로 분리함.
- **분석 유연성 확보**: "상품명"이 바뀌더라도 팩트 테이블은 수정할 필요 없이 차원 테이블 한 곳만 고치면 과거 모든 실적에 반영됨.

- **📢 섹션 요약 비유**: 팩트 테이블은 '과녁의 중심(점수)'이고, 차원 테이블은 '과녁을 쏜 방향(화살의 각도)'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 두 테이블의 물리적 특성 비교

| 특성 항목 | 팩트 테이블 (Fact) | 차원 테이블 (Dimension) |
|:---:|:---|:---|
| **저장 내용** | 수치 데이터 (Measures), FK 조합 | 속성 데이터 (Attributes), 설명 텍스트 |
| **행 수 (Rows)** | **매우 많음** (수억 ~ 수조 건) | 상대적으로 적음 (수백 ~ 수백만 건) |
| **열 수 (Columns)** | 적음 (수치와 키 위주) | **매우 많음** (상세 속성 전개) |
| **데이터 증분** | 초 단위로 추가됨 (Add-only) | 가끔 추가되거나 수정됨 (SCD) |

### 2. 테이블 간 연결 구조 도해 (ASCII)

스타 스키마 환경에서 두 테이블이 어떻게 결합되는지 시각화한다.

```text
    [ 차원 테이블: 매장 ]             [ 차원 테이블: 날짜 ]
    ┌─────────────┐                 ┌─────────────┐
    │ 매장ID (PK)  │ ──┐         ┌── │ 날짜ID (PK)  │
    │ 매장명, 지역  │   │         │   │ 년, 월, 일   │
    └─────────────┘   │         │   └─────────────┘
                      │         │
                ┌─────▼─────────▼─────┐
                │   [ 팩트 테이블 ]     │
                ├─────────────────────┤
                │ - 매장ID (FK)        │ <--- (Foreign Keys)
                │ - 날짜ID (FK)        │
                ├─────────────────────┤
                │ - 판매수량 (Measure)  │ <--- (Measures)
                │ - 매출액   (Measure)  │
                └─────────────────────┘
```

### 3. 심층 동작 원리: 대리 키 (Surrogate Key)의 사용
- 차원 테이블은 운영 DB의 PK(자연 키) 대신, DW 자체에서 부여한 정수형 일련번호인 **대리 키**를 주로 사용함.
- **이유**: 운영 시스템에서 PK가 바뀌어도 DW의 과거 이력을 보존하기 위함 (Slowly Changing Dimension 대응).

### 4. 핵심 기술: 팩트의 입도 (Granularity)
- 팩트 테이블 한 행이 나타내는 비즈니스 활동의 최소 단위. (예: 영수증 전체 vs 영수증 내 품목 하나)
- 입도가 세밀할수록(Atomic) 더 다양한 분석이 가능하지만, 데이터량이 기하급수적으로 늘어남.

- **📢 섹션 요약 비유**: 입도 설정은 '돋보기의 배율'과 같아서, 너무 낮으면 흐릿하고 너무 높으면 전체를 보기 힘듭니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 팩트의 종류에 따른 분석

| 팩트 유형 | 상세 내용 | 예시 |
|:---:|:---|:---|
| **Additive (가산)** | 모든 차원에 대해 합산 가능. | 매출액, 수량 |
| **Semi-Additive** | 특정 차원(주로 시간)에 대해서만 합산 불가. | **재고량** (오늘 10개, 내일 10개라고 합계 20개 아님) |
| **Non-Additive** | 합산이 불가능하고 비율 등으로만 사용. | 할인율, 온도, 단가 |

### 2. 과목 융합 관점: 통계 및 인덱스 아키텍처
- **통계학 융합**: 팩트 테이블은 **모집단**의 역할을 수행하며, 차원 테이블은 모집단을 쪼개어 보는 **층화(Stratification)**의 기준이 된다.
- **인덱스 융합**: 팩트 테이블은 대량의 외래 키를 가지므로, 조인 성능을 위해 **비트맵 인덱스(Bitmap Index)**나 **스타 변환(Star Transformation)** 기술이 엔진 레벨에서 적극적으로 활용된다.

- **📢 섹션 요약 비유**: 가산 팩트는 '돈'과 같아서 모으면 커지지만, 비가산 팩트는 '온도'와 같아서 모은다고 뜨거워지지 않습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: Slowly Changing Dimension (SCD) 처리
- **상황**: 고객이 경기도에서 서울로 이사를 감.
- **판단**: 과거 경기도 거주 시절의 매출을 지금 서울 매출로 집계하면 지역 통계가 왜곡됨.
- **결단**: **SCD Type 2 적용**. 기존 차원 행은 닫고(Expire), 새로운 주소를 가진 새 행을 생성. 팩트 테이블은 당시의 차원 ID를 유지함으로써 과거의 진실과 현재의 상태를 완벽히 격리 관리함.

### 2. 안티패턴: 팩트 테이블에 텍스트 속성 직접 저장
- **증상**: "조인이 귀찮다"며 팩트 테이블 수억 행마다 '상품명'이나 '고객주소'를 텍스트로 다 적어둠.
- **결과**: 데이터량이 10배 이상 폭증하고, 상품명이 오타로 인해 서로 다르게 적히는 등 데이터 오염 발생.
- **조치**: **무조건 차원 테이블로 분리**하고 숫자형 FK만 남겨 데이터 밀도를 극대화할 것.

### 📢 기술사적 결언
> "팩트와 차원의 구분은 **'사건'과 '맥락'**을 분리하는 철학적 통찰이다. 팩트는 거짓말을 하지 않는 비즈니스의 결과이고, 차원은 그 결과를 해석하는 인간의 관점이다. 기술사는 단순히 테이블을 설계하는 것이 아니라, 기업의 **성과 지표(Metric)**가 무엇인지 정의하고 이를 가장 빠르고 유연하게 추출할 수 있는 **차원의 축**을 세워야 한다. 이 견고한 이원적 구조가 현대 **빅데이터 분석과 AI 인사이트**의 가장 강력한 원재료가 됨을 명심해야 한다."

- **📢 섹션 요약 비유**: 팩트 테이블에 글자를 적는 것은 '수표에 편지를 쓰는 것'과 같아, 돈(데이터)의 가치를 떨어뜨리고 처리만 힘들게 만듭니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (팩트/차원 분리 시)

| 지표 | 도입 전 (통합 테이블) | 도입 후 (분리 모델) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **스토리지 사용량** | 100% | **20~30%** | 공간 효율 3배 이상 향상 |
| **쿼리 튜닝 유연성** | 매우 낮음 | **차원 변경만으로 대응** | 분석 민첩성 확보 |
| **데이터 정합성** | 불일치 잦음 | **차원 테이블 통합 관리** | 데이터 품질 신뢰도 향상 |

### 2. 미래 전망
- **OBT (One Big Table)의 재등장**: 클라우드 고성능 스토리지(BigQuery 등)에서는 조인 비용을 줄이기 위해 다시 팩트와 차원을 합친 거대 테이블을 쓰기도 하지만, 논리적 관점에서의 **팩트/차원 개념**은 여전히 데이터 거버넌스의 표준임.
- **Semantic Layer**: 사용자가 테이블 구조를 몰라도 자연어로 질문하면 팩트와 차원을 AI가 알아서 매핑해주는 **지능형 시맨틱 계층**이 분석 도구의 핵심이 될 것임.

- **📢 섹션 요약 비유**: 미래의 데이터 저장소는 우리가 '결과'를 물으면 수조 개의 '팩트'를 뒤져 '차원'이라는 그릇에 예쁘게 담아 내놓는 자동 요리 머신이 될 것입니다.
