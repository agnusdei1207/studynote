# 109. 테이블 통합 및 분할 (Merge & Decomposition)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 성능 최적화를 위해 논리적으로 분리된 테이블을 하나로 합치거나(**통합**), 하나의 거대한 테이블을 관리 효율성과 성능을 위해 여러 개로 쪼개는(**분할**) 물리적 재구성 기법이다.
> 2. **이원적 전략**: 테이블 통합은 조인(Join) 횟수를 줄여 조회 성능을 극대화하며, 테이블 분할은 특정 컬럼이나 행에 집중되는 I/O 병목 및 데이터 경합(Contention)을 분산시켜 전체 시스템의 동시성을 향상시킨다.
> 3. **가치**: 데이터의 사용 패턴(Access Pattern)에 따라 테이블의 물리적 경계를 유연하게 조정함으로써, 정규화의 논리적 이점을 유지하면서도 하드웨어의 물리적 한계를 극복하는 핵심 설계 기술이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **테이블 통합 (Merge)**: 1:1 관계나 1:N 관계에 있는 테이블들을 하나의 테이블로 합쳐 조인 없이 조회 가능하게 만드는 작업입니다.
- **테이블 분할 (Decomposition)**: 테이블의 속성들을 나누어 별도의 테이블로 만드는 **수직 분할(Vertical)**과, 튜플들을 나누어 저장하는 **수평 분할(Horizontal)**로 구분됩니다.

### 2. 💡 비유: 가방 싸기와 짐 나누기
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        테이블 통합 vs 분할 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 테이블 통합 (한 가방에 다 담기):                                           │
│     - 외출할 때 지갑, 폰, 차 키를 각각 다른 주머니에 넣지 않고 한 파우치에 담음.    │
│     - 한 번에 쏙 꺼낼 수 있어 정말 빠름! (조회 속도 향상)                         │
│                                                                             │
│  2. 테이블 분할 (짐 나눠 들기):                                               │
│     - 너무 무거운 배낭 하나를 메는 대신, 작은 가방 여러 개로 나눠서 친구들과 나눠 듦.│
│     - 걷기도 편하고(I/O 분산), 가방 하나가 고장 나도 다른 건 멀쩡함. (가용성)      │
│                                                                             │
│  → 데이터의 **'무게(크기)'**와 **'꺼내는 빈도'**에 따라 가방을 합칠지 나눌지         │
│     결정하는 것이 바로 통합과 분할입니다!                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **Wide Table의 비효율**: 한 테이블에 수백 개의 컬럼이 있으면, 단 두 개만 조회하려 해도 블록 전체를 읽어야 하는 I/O 낭비 발생.
- **Hot-Spot 문제**: 특정 인기 테이블에 모든 사용자가 몰려 락(Lock) 경합이 심해지는 현상 해결 필요.

- **📢 섹션 요약 비유**: 통합은 '이동 시간'을 줄이는 것이고, 분할은 '기다리는 시간'을 줄이는 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 테이블 통합 및 분할의 상세 기법

| 기법 유형 | 상세 내용 | 적용 효과 |
|:---:|:---|:---|
| **1:1 테이블 통합** | 두 테이블을 하나로 합침. | 조인 제거, 프로세스 단순화 |
| **1:N 테이블 통합** | 부모 데이터를 자식에 포함 (N 쪽으로 통합). | 다량의 조인 부하 해소 |
| **수직 분할 (Vertical)** | 자주 쓰는 컬럼과 안 쓰는 컬럼(LOB 등) 분리. | 디스크 I/O 절감, 캐시 효율 상승 |
| **수평 분할 (Horizontal)** | 특정 기간/지역별로 데이터를 행 단위 분리. | 파티셔닝(Partitioning) 효과 |

### 2. 테이블 수직 분할 아키텍처 도해 (ASCII)

하나의 비대한 테이블이 사용 빈도와 데이터 크기에 따라 나뉘는 과정을 시각화한다.

```text
    [ Original Table: Employee (Very Wide) ]
    ┌────┬─────┬─────┬────────┬────────┬───────┐
    │ EID│ Name│ Dept│ Photo  │ Resume │ Intro │
    └────┴─────┴─────┴────────┴────────┴───────┘
               │
               ▼ [ Vertical Decomposition ]
    ┌────────────────────┐      ┌───────────────────────────────┐
    │ [ Employee_Basic ]  │      │ [ Employee_Files ]            │
    ├────┬─────┬─────────┤      ├────┬────────┬────────┬────────┤
    │ EID│ Name│ Dept    │      │ EID│ Photo  │ Resume │ Intro  │
    └────┴─────┴─────────┘      └────┴────────┴────────┴────────┘
    (자주 조회되는 필드)           (가끔 조회되는 무거운 필드)
    * 효과: Basic 테이블 스캔 시 블록 당 레코드 수가 많아져 I/O 성능 급상승.
```

### 3. 심층 동작 원리: 블록 밀도와 I/O (Blocking Factor)
- **통합의 논리**: 두 번의 I/O(부모 블록 + 자식 블록)를 한 번의 I/O로 끝내려는 시도.
- **분할의 논리**: 필요한 컬럼만 모아둔 작은 블록을 읽음으로써 한 번의 I/O로 더 많은 '유효 데이터'를 가져오려는 시도.

### 4. 핵심 기술: 분할 테이블의 뷰(View) 통합
- 물리적으로 분할된 테이블들을 사용자가 일일이 알기 힘들므로, **UNION ALL**을 사용한 뷰를 생성하여 논리적으로는 하나의 테이블처럼 보이게 관리함.

- **📢 섹션 요약 비유**: 수직 분할은 '자주 쓰는 필기구는 필통에, 가끔 쓰는 물감은 서랍에' 따로 두어 책상을 넓게 쓰는 지혜입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 테이블 통합 vs 테이블 분할 비교

| 비교 항목 | 테이블 통합 (Merge) | 테이블 분할 (Split) |
|:---:|:---|:---|
| **조인(Join)** | 감소 (최소화) | 증가 (필요시 발생) |
| **I/O 단위 크기** | 증가 | **감소** |
| **동시성(Concurrency)** | 낮음 (Lock 범위 큼) | **높음 (Lock 분산)** |
| **정규화 관점** | 정규화 위반 가능성 높음 | 정규화 원칙 유지 (1:1 분리) |

### 2. 과목 융합 관점: OS 및 파일 시스템
- **운영체제(OS) 융합**: 테이블 수평 분할은 OS의 **파일 파티셔닝** 및 **디스크 스트라이핑(RAID)** 기술과 결합하여 병렬 I/O 성능을 극대화한다.
- **보안(Security) 융합**: 민감 정보(급여, 주민번호)를 수직 분할하여 별도 테이블에 격리하고 접근 권한을 따로 설정함으로써, **최소 권한의 원칙**을 데이터 계층에서 구현할 수 있다.

- **📢 섹션 요약 비유**: 통합이 '단체 관광 버스'라면, 분할은 '개인용 승용차 여러 대'입니다. 상황에 따라 정체(Lock)가 다르게 발생합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 대용량 이미지 데이터를 포함한 게시판 튜닝
- **상황**: 게시글 조회 시 썸네일과 원본 이미지 경로, 본문 내용(Long Text) 때문에 목록 로딩이 너무 느림.
- **판단**: 본문과 이미지는 용량이 커서 한 행의 길이가 너무 긺. 목록 조회 시 비효율적 I/O 발생.
- **결단**: **테이블 수직 분할 단행**. [번호, 제목, 작성자, 날짜]만 담은 'Board_List'와 [본문, 이미지]를 담은 'Board_Content'로 분리. 목록 조회 성능 10배 향상.

### 2. 안티패턴: 무분별한 1:N 테이블 통합
- **증상**: "조인이 싫다"며 모든 주문 내역(N)을 회원 테이블(1)에 컬럼으로(주문1, 주문2...) 넣음.
- **결과**: 회원이 주문을 더 많이 하면 테이블 구조를 바꿔야 하고, 주문이 적은 회원은 빈 공간(NULL)으로 메모리 낭비 심화.
- **조치**: 1:N 관계는 가급적 **중복 컬럼 기법**이나 **인덱스 최적화**로 해결하고, 테이블 통합은 1:1 관계 위주로 제한적 수행.

### 📢 기술사적 결언
> "테이블 구조의 재구성은 **'데이터의 무게중심'**을 옮기는 작업이다. 통합은 조회의 무게를 줄이고, 분할은 관리의 무게를 나눈다. 기술사는 단순히 성능 수치에 매몰되지 말고, **데이터의 휘발성, 보안 등급, 그리고 트랜잭션의 입도(Granularity)**를 종합적으로 고려하여 물리적 경계를 획정해야 한다. 특히 분산 환경에서는 분할된 테이블이 서로 다른 노드에 위치하여 **네트워크 오버헤드**를 유발할 수 있음을 항상 경계해야 한다."

- **📢 섹션 요약 비유**: 잘못된 통합은 '한 바구니에 달걀과 돌을 담는 것'과 같아, 관리가 불편해지고 결국 사고(정합성 파괴)가 날 확률이 높습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (재구성 최적화 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **Full Scan 속도** | 수 초 ~ 분 | **밀리초 (ms)** | I/O 효율 50% 이상 개선 |
| **시스템 동시성** | Lock 경합 빈번 | **분산 처리 가능** | 동시 사용자 수용량 2배 증가 |
| **관리 용이성** | 거대 테이블 관리 부담 | **주제별 독립 관리** | 백업 및 복구 시간 단축 |

### 2. 미래 전망
- **Self-organizing Tables**: AI가 실시간 쿼리 패턴을 모니터링하여, 사용자가 명령하지 않아도 테이블을 수직/수평으로 분할하여 배치하는 **자율 주행 스토리지** 기술 발전.
- **Virtual Merging**: 물리적으로는 분할되어 있으나 메모리 계층(In-memory)에서 실시간으로 통합된 것처럼 보여주는 **가상화 저장 기술**이 클라우드 DB의 핵심 경쟁력이 될 것임.

- **📢 섹션 요약 비유**: 미래의 테이블은 고정된 상자가 아니라, 데이터의 흐름에 따라 스스로 합쳐지고 쪼개지는 '변신 로봇'과 같은 유연함을 갖추게 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **테이블 통합**은 따로 놀던 두 친구를 **"한 팀"**으로 묶어주는 거예요. 손을 잡고 있으니 부를 때 한꺼번에 올 수 있죠.
2. **테이블 분할**은 너무 무거운 가방을 멘 친구의 짐을 **"작은 가방 여러 개"**에 나눠서 가볍게 만들어주는 거예요.
3. 상황에 맞춰서 잘 묶고 나누면, 우리가 필요한 물건을 **훨씬 더 빨리, 힘 안 들이고** 찾을 수 있답니다!
