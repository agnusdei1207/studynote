# 99. 무결성의 종류 - 상태 무결성(Static)과 과도 무결성(Dynamic)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 무결성을 감시하는 시점에 따라, 정지된 데이터의 최종 상태를 검사하는 **상태 무결성(Static)**과 데이터가 변경되는 과정의 타당성을 검사하는 **과도 무결성(Dynamic)**으로 분류된다.
> 2. **차별적 메커니즘**: 상태 무결성은 PK, FK 등 선언적 제약 조건으로 구현되어 데이터의 '현재 모습'을 규제하며, 과도 무결성은 트리거(Trigger) 등을 활용하여 '이전 값과 이후 값의 논리적 변화'를 통제한다.
> 3. **가치**: 두 무결성 체계의 조화를 통해 데이터의 단순한 저장 형식뿐만 아니라, 비즈니스의 흐름과 시간적 인과관계까지 데이터베이스 레벨에서 완벽하게 수호할 수 있다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **상태 무결성 (Static Integrity)**: 데이터베이스가 어느 시점에 있든 항상 만족해야 하는 조건입니다. 데이터의 '값' 그 자체에 집중합니다. (예: 나이는 항상 0보다 커야 함)
- **과도 무결성 (Dynamic/Transition Integrity)**: 데이터가 한 상태에서 다른 상태로 변할 때(Update) 만족해야 하는 조건입니다. 데이터의 '변화 양상'에 집중합니다. (예: 월급은 이전보다 줄어들 수 없음)

### 2. 💡 비유: 학교 성적표와 전학 규칙
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        상태 vs 과도 무결성 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 상태 무결성 (정지 화면):                                                  │
│     - "성적표의 점수는 반드시 0점에서 100점 사이여야 한다."                      │
│     - 지금 이 순간 성적표에 적힌 숫자가 유효한지만 봅니다.                       │
│                                                                             │
│  2. 과도 무결성 (움직이는 화면):                                               │
│     - "학년은 반드시 1학년 다음엔 2학년으로만 올라가야 한다."                    │
│     - 1학년이었던 학생이 갑자기 4학년이 되는 '이상한 점프'를 막습니다.             │
│                                                                             │
│  → 상태는 **'지금 모습'**을, 과도는 **'변화하는 과정'**을 감시하는 눈입니다!         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **비즈니스 로직의 고도화**: 단순히 "값의 범위"를 넘어서, "업무의 선후 관계"나 "상태 전이 규칙"을 데이터 레벨에서 강제해야 할 필요성 증대.
- **데이터 보안 및 이력 관리**: 승인된 결제 상태가 다시 미승인으로 돌아가는 등의 불법적인 상태 변경을 원천 차단하기 위함.

- **📢 섹션 요약 비유**: 상태 무결성은 '현재의 신분'을 확인하는 것이고, 과도 무결성은 '어디서 와서 어디로 가는지' 행적을 확인하는 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 무결성 유형별 세부 분류 및 구현

| 분류 | 상세 유형 | 설명 | 주요 구현 수단 |
|:---:|:---|:---|:---|
| **상태 무결성** | 도메인 무결성 | 속성 값의 타입과 범위 규제. | Data Type, CHECK |
| (Static) | 개체 무결성 | 기본 키의 유일성 및 NOT NULL 보장. | PRIMARY KEY |
| | 참조 무결성 | 외래 키의 참조 대상 실재 여부 보장. | FOREIGN KEY |
| **과도 무결성** | 상태 전이 무결성 | 이전 값과 새 값의 논리적 비교. | **TRIGGER** |
| (Dynamic) | 인과 무결성 | 선행 행위가 있어야 후행 행위 가능. | 응용 로직 + TRIGGER |

### 2. 무결성 작동 아키텍처 (ASCII)

상태와 과도 무결성이 검증되는 시점의 차이를 시각화한다.

```text
    [ Current State ] ──────────▶ [ Transition Process ] ─────────▶ [ New State ]
    (Salary = 3000)               (UPDATE to 2500)                 (Salary = 2500)
           │                             │                                │
    ┌──────▼──────┐              ┌──────▼──────┐                  ┌──────▼──────┐
    │상태 무결성 검사│              │과도 무결성 검사│                  │상태 무결성 검사│
    │(3000 > 0?)  │              │(2500 < 3000?)│                  │(2500 > 0?)  │
    └──────┬──────┘              └──────┬──────┘                  └──────┬──────┘
           │ (OK)                        │ (Violation!)                   │
           └─────────────────────────────┼────────────────────────────────┘
                                         ▼
                                   [ ROLLBACK! ]
                        "월급 삭감은 비즈니스 규칙 위반입니다."
```

### 3. 심층 동작 원리: 트리거(Trigger)를 통한 과도 무결성 구현
- **원리**: 데이터 변경 직전(BEFORE) 또는 직후(AFTER)에 이전 행의 값(`:OLD`)과 새로운 행의 값(`:NEW`)을 가로채어 비교 연산 수행.
- **예시**: `IF :NEW.status = 'CANCEL' AND :OLD.status = 'DELIVERED' THEN RAISE_ERROR;`
- **의미**: "배송 완료된 주문은 취소 상태로 바꿀 수 없다"는 비즈니스 로직을 물리적으로 강제.

### 4. 핵심 기술: 선언적 제약 조건의 한계 극복
- 상태 무결성(PK, FK)은 SQL 문법만으로 선언 가능하여 성능이 좋으나 복잡한 로직 표현 불가.
- 과도 무결성은 성능 오버헤드가 발생하더라도 비즈니스의 **'Flow'**를 수호하는 데 필수적임.

- **📢 섹션 요약 비유**: 트리거는 '어제의 나'와 '오늘의 나'를 비교하여, 나쁜 쪽으로 변하지 않았는지 체크하는 마음의 거울과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 상태 무결성(Static) vs 과도 무결성(Dynamic)

| 비교 항목 | 상태 무결성 | 과도 무결성 |
|:---:|:---|:---|
| **관심 시점** | **현재 시점의 단면 (Snapshot)** | **변화하는 시점 (Update)** |
| **비교 대상** | 정의된 규칙 (Schema) | **이전 상태의 값 (:OLD)** |
| **표현력** | 단순하고 정적임 | 복잡하고 동적임 |
| **성능 오버헤드** | 최소화됨 | 상대적으로 높음 (연산 수반) |

### 2. 과목 융합 관점: 이산수학 및 소프트웨어 공학
- **이산수학 융합**: 과도 무결성은 **유한 오토마타(Finite State Machine)**의 상태 전이도와 개념적으로 일치하며, 허용되지 않는 상태로의 전이를 막는 함수로 정의될 수 있다.
- **소프트웨어 공학(SE) 융합**: 과도 무결성 로직은 DDD(도메인 주도 설계)의 **애그리게이트(Aggregate)** 내 비즈니스 인베리언트(Invariant)를 데이터베이스 계층으로 투영한 결과다.

- **📢 섹션 요약 비유**: 상태 무결성이 '건축물의 하중 견디기'라면, 과도 무결성은 '엘리베이터의 층별 이동 규칙'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 주문 상태 머신(State Machine) 설계
- **상황**: 주문의 상태가 '결제대기 $\to$ 결제완료 $\to$ 배송중 $\to$ 배송완료' 순으로만 가야 함.
- **판단**: 상태 무결성(CHECK)만으로는 '결제완료'에서 다시 '결제대기'로 돌아가는 것을 막을 수 없음.
- **결단**: **과도 무결성 트리거 작성**. 현재 상태와 변경될 상태를 매트릭스로 대조하여, 허용되지 않은 방향의 업데이트를 원천 차단함.

### 2. 안티패턴: 상태 무결성만 믿고 과거 데이터를 무시함
- **증상**: "지금 데이터가 유효하면 됐지"라며 변화의 과정을 검증하지 않음.
- **결과**: 해커가 DB에 침투하여 포인트 잔액을 직접 수정(Update)할 때, 획득 이력이 없는데 잔액만 늘어나는 논리적 모순 방어 실패.
- **조치**: 중요한 자산 데이터에는 **과도 무결성 검증과 이력 로그(Audit Trail)**를 반드시 병행 설계할 것.

### 📢 기술사적 결언
> "무결성은 데이터의 **'공간적 정합성(Static)'**과 **'시간적 정합성(Dynamic)'**의 합작품이다. 공간만 보고 시간을 무시하면 논리가 무너지고, 시간만 보고 공간을 무시하면 형식이 무너진다. 기술사는 시스템의 중요도에 따라 두 무결성 수준을 전략적으로 배치해야 한다. 특히 현대의 **이벤트 소싱(Event Sourcing)** 아키텍처에서는 모든 것이 과도 무결성의 관점으로 재편되고 있음을 인지하고, 데이터의 '변경 궤적'을 수호하는 설계 역량을 갖춰야 한다."

- **📢 섹션 요약 비유**: 상태 무결성만 있는 DB는 '정지된 사진들만 있는 앨범'이고, 과도 무결성까지 갖춘 DB는 '앞뒤 맥락이 완벽한 영화'와 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (복합 무결성 적용 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **비즈니스 로직 오류** | 빈번 (상태 역전 등) | **0% (엔진 수준 방어)** | 업무 프로세스 안정성 극대화 |
| **부정 취득/조작 위험** | 상존 | **원천 차단 (맥락 검증)** | 데이터 보안 수준 2단계 격상 |
| **운영 가시성** | 원인 파악 힘듦 | **변경 경로 명확** | 장애 대응 속도 향상 |

### 2. 미래 전망
- **AI-Based Behavioral Integrity**: 과거의 정상적인 데이터 전이 패턴을 AI가 학습하여, 규칙에 명시되지 않았더라도 비정상적인 전이(Anomaly Path)를 감지하여 차단하는 지능형 무결성 보호.
- **Immutable State Engines**: 모든 변화를 이벤트로 기록하고, 현재 상태를 이벤트의 합으로 도출함으로써 과도 무결성을 아키텍처 수준에서 강제하는 **함수형 데이터베이스** 확산.

- **📢 섹션 요약 비유**: 미래의 무결성 시스템은 단순한 감시자를 넘어, 데이터의 과거와 미래를 연결하는 '지능형 스토리텔러'가 될 것입니다.
