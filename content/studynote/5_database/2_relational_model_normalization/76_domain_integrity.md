# 76. 도메인 무결성 (Domain Integrity) - 속성 값의 유효 범위 보장

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 릴레이션의 특정 속성에 입력되는 모든 값은 정의된 **도메인(Domain)**에 속한 값이어야 하며, 데이터 타입, 길이, 허용 범위, NULL 여부 등의 규칙을 반드시 준수해야 한다는 원칙이다.
> 2. **수호 메커니즘**: DBMS 엔진 차원에서 데이터의 형식을 강제하여 비논리적인 값(예: 나이가 음수이거나 이름에 숫자가 포함됨)의 유입을 원천 차단하는 **가장 기초적인 방어선** 역할을 수행한다.
> 3. **가치**: 데이터의 **원자성(Atomicity)**을 유지하고 속성의 의미적 정합성을 확보함으로써, 쿼리 연산의 신뢰도를 높이고 애플리케이션의 예외 처리 부담을 경감시킨다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**도메인 무결성 (Domain Integrity)**은 릴레이션 내의 모든 속성 값이 해당 속성에 정의된 도메인 규칙을 만족해야 함을 의미합니다. 이는 개별 컬럼(속성) 수준에서 데이터의 '품질'과 '형식'을 정의하는 규약입니다.

### 2. 💡 비유: 자판기의 동전 투입구
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        도메인 무결성 비유                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 음료수 자판기에 돈을 넣고 있음.                                         │
│                                                                             │
│  1. 도메인 (투입구 규격): 100원, 500원 동전만 들어갈 수 있는 크기와 두께.         │
│  2. 무결성 검사:                                                              │
│     - 500원 동전을 넣으면? 찰칵! 통과 (무결성 만족)                             │
│     - 단추나 종이 조각을 넣으면? 퉤! 뱉어냄 (무결성 위반)                        │
│                                                                             │
│  → 도메인 무결성은 데이터가 들어오기 전, 미리 정해둔 **'모양과 자격'**을            │
│     엄격하게 심사하는 것과 같습니다.                                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 오염의 조기 발견**: 잘못된 형식의 데이터가 한 번 DB에 저장되면, 이를 나중에 찾아내고 수정하는 비용은 삽입 시 차단하는 비용의 수십 배에 달함.
- **표준화된 관리**: '날짜'나 '금액' 등 반복적으로 쓰이는 데이터 형식을 전사적으로 통일하여 데이터 활용성을 높이기 위함.

- **📢 섹션 요약 비유**: 도메인 무결성은 '입구에서 신분증의 유효기간과 형식을 대조하는 자동 개찰구'와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 도메인 무결성을 구현하는 4대 제약 기술

| 구현 기술 | 상세 역할 | 비유 |
|:---:|:---|:---|
| **Data Type** | 정수, 문자열, 날짜 등 물리적 데이터 형식 지정. | 재질 검사 |
| **NOT NULL** | 필수 입력 속성에 빈 값(NULL) 유입 차단. | 공란 거부 |
| **CHECK** | 값의 범위나 패턴(정규표현식 등)을 논리적으로 제한. | 수치 제한 |
| **DEFAULT** | 값이 없을 때 자동으로 채워지는 기본값 정의. | 예비 부품 |

### 2. 도메인 무결성 검증 아키텍처 (ASCII)

데이터가 속성에 안착하기 전 거치는 검문 과정을 시각화한다.

```text
    [ Raw Data Input ] : { Name: "Kim", Age: -5, Grade: "Z" }
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Domain Integrity Filter ]                                    │
    │                                                                 │
    │  1. Type Check: Age is Number? ─────▶ (Pass)                    │
    │  2. Range Check: Age > 0? ──────────▶ (Fail! Error Raised)      │
    │  3. Value Check: Grade in {A,B,C}? ──▶ (Fail! Error Raised)      │
    │                                                                 │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (Fail 시 연산 즉시 중단)
    ┌──────▼──────┐
    │ Transaction │ 
    │ Rollback    │
    └─────────────┘
```

### 3. 심층 동작 원리: 원자성(Atomicity)과 도메인
- **원칙**: 도메인 무결성은 속성 값이 '원자적'이어야 함을 전제로 함.
- **내용**: 하나의 속성에 여러 개의 값(Set)이나 복합적인 구조가 들어오는 것을 막아, 관계형 모델의 근간인 제1정규형(1NF)을 물리적으로 지원함.

### 4. 핵심 기술: 사용자 정의 도메인 (User-defined Domain)
- **SQL 표준**: `CREATE DOMAIN Score AS INT CHECK (VALUE BETWEEN 0 AND 100);`
- **장점**: 'Score'라는 도메인을 한 번 정의해두면, 국어/영어/수학 테이블 등 여러 곳에서 재사용 가능하여 유지보수성이 극대화됨.

- **📢 섹션 요약 비유**: 체크 제약조건은 '어린이용 풀장에 어른이 들어오지 못하게 키를 제한하는 표지판'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 무결성 유형별 비교

| 비교 항목 | 도메인 무결성 (Domain) | 개체 무결성 (Entity) | 참조 무결성 (Referential) |
|:---:|:---|:---|:---|
| **관심 대상** | **개별 속성(Column) 값** | 행(Row)의 식별자 | 테이블 간의 연결 |
| **핵심 도구** | **Data Type, CHECK** | **Primary Key** | **Foreign Key** |
| **검증 시점** | 모든 DML 발생 시 | 삽입/수정 시 | 삭제/수정 시 |
| **비유** | **물건의 규격** | 물건의 고유 번호 | 물건들 간의 조립 |

### 2. 과목 융합 관점: 프로그래밍 및 보안
- **프로그래밍 융합**: 도메인 무결성은 객체지향 프로그래밍의 **캡슐화(Encapsulation)** 및 **Setter 메서드 내 검증 로직**을 데이터 계층으로 옮겨온 것과 같다.
- **보안(Security) 융합**: 잘못된 도메인 입력을 통한 **SQL Injection**이나 **Buffer Overflow** 공격을 방어하는 가장 근본적이고 강력한 입력 값 검증(Input Validation) 수단이다.

- **📢 섹션 요약 비유**: 도메인 무결성이 '개별 부품의 품질 검사'라면, 개체/참조 무결성은 '완성품의 조립 상태 검사'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 복잡한 비즈니스 규칙의 도메인화
- **상황**: 사원번호는 반드시 'E'로 시작하고 뒤에 5자리 숫자가 와야 함.
- **판단**: 단순 문자열(`VARCHAR`)로 선언하면 잘못된 값이 들어올 확률이 매우 높음.
- **결단**: **정규표현식을 활용한 CHECK 제약조건 부여**. `CHECK (emp_id ~ '^E[0-9]{5}$')`와 같은 규칙을 DB에 심어, 애플리케이션의 버그로부터 데이터를 수호함.

### 2. 안티패턴: 형식만 맞춘 '가짜' 도메인 설계
- **증상**: 날짜 데이터를 `VARCHAR(8)`에 '20240132'와 같이 저장함.
- **결과**: 문자열 타입 제약은 통과하지만, 실제로는 존재하지 않는 날짜임. 날짜 연산 시 에러 발생 및 정렬 순서 뒤섞임.
- **조치**: 반드시 **의미에 부합하는 전용 타입(DATE, TIMESTAMP)**을 사용하고, 추가적인 비즈니스 제약은 CHECK로 보완할 것.

### 📢 기술사적 결언
> "도메인 무결성은 데이터베이스의 **'최소한의 품격'**이다. 품격 없는 데이터는 분석의 가치를 잃는다. 기술사는 단순 데이터 타입을 넘어 **데이터의 의미적 경계(Semantic Boundary)**를 도메인으로 정의할 줄 알아야 한다. 특히 최근의 빅데이터 환경에서는 데이터의 유효성을 검증하는 것이 시스템 전체 성능보다 중요할 때가 많으므로, **강력한 도메인 정의**를 통해 데이터의 자정 능력을 키우는 설계를 지향해야 한다."

- **📢 섹션 요약 비유**: 전용 타입을 쓰지 않는 것은 '우유를 페인트 통에 담아두는 것'과 같아, 겉모양은 비슷할지 몰라도 나중에 큰 혼란을 야기합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (도메인 무결성 최적화 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 에러율** | 약 3~5% | **0.1% 미만** | 정합성 획기적 개선 |
| **앱 검증 코드량** | 방대함 (If문 남발) | **간소함 (비즈니스 집중)** | 개발 생산성 30% 향상 |
| **쿼리 최적화 효율** | 낮음 (값 예측 불가) | **높음 (범위 한정 가능)** | 쿼리 실행 속도 향상 |

### 2. 미래 전망
- **AI-Based Domain Discovery**: 수천 개의 속성 중 의미가 같은 것들을 AI가 찾아내어 자동으로 공통 도메인을 제안하고 정합성을 맞추는 기술 고도화.
- **Complex Domain Support**: 지리 정보(GIS), 생체 정보, 벡터 데이터 등 복잡한 데이터 구조에 대한 고차원적 도메인 무결성 검증 엔진이 보편화될 전망.

- **📢 섹션 요약 비유**: 미래의 도메인 시스템은 단순한 거름망을 넘어, 데이터의 성격을 스스로 파악하고 최적의 모양으로 가꿔주는 '지능형 정원사'가 될 것입니다.
