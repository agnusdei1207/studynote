# 103. 제3정규형 (3NF, Third Normal Form) - 이행 함수적 종속성 제거

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 릴레이션이 제2정규형(2NF)을 만족하면서, 기본 키(Primary Key)가 아닌 속성들 간에 존재하는 **이행 함수적 종속성(Transitive Functional Dependency)**을 제거한 상태다.
> 2. **작동 원리**: 속성 $A$가 $B$를 결정하고, $B$가 $C$를 결정하여 결과적으로 $A$가 $C$를 결정하게 되는 ($A \to B \to C$) 관계에서, 중간 매개체인 $B$를 별도의 릴레이션으로 분리한다.
> 3. **가치**: 일반 속성의 변경이 다른 속성에 연쇄적인 영향을 미치는 **갱신 이상**을 방지하며, 실무 데이터베이스 설계에서 정합성과 성능의 균형을 맞추는 가장 보편적인 정규화 목표 지점이 된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**제3정규형 (3NF)**은 "기본 키가 아닌 속성들(Non-key attributes)은 오직 기본 키에만 종속되어야 한다"는 원칙을 따릅니다. 즉, 일반 속성끼리 서로 결정하고 종속되는 '이행적' 관계를 허용하지 않습니다.

### 2. 💡 비유: 사원 명부와 부서 정보
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        3NF 개념 비유                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 이행 종속 발생 (문제 상황):                                               │
│     - [사원번호]를 알면 [부서번호]를 알 수 있음. (A -> B)                      │
│     - [부서번호]를 알면 [부서위치]를 알 수 있음. (B -> C)                      │
│     - 결과적으로 [사원번호]만 알아도 [부서위치]가 결정됨. (A -> C)               │
│                                                                             │
│  - 만약 부서 위치가 바뀌면, 그 부서에 속한 수천 명의 사원 정보를 다 고쳐야 함!     │
│  - 사원이 한 명도 없는 부서는 위치 정보를 저장할 방법이 없음.                   │
│                                                                             │
│  → 해결: [사원] 테이블에는 부서번호만 남기고, [부서] 테이블을 따로 만들어          │
│     위치 정보를 관리함. 이것이 바로 3NF!                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **간접 종속의 폐해**: 기본 키와 직접적인 관련이 없는 정보들이 테이블에 기생하면서 데이터 중복을 야기함.
- **논리적 응집도 향상**: 테이블의 주제를 더욱 선명하게 하여, "사원 테이블에는 사원 정보만, 부서 테이블에는 부서 정보만" 담기 위함.

- **📢 섹션 요약 비유**: 3NF는 '건너건너 아는 사이'를 정리하고, '직계 가족'끼리만 모여 살게 하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 용어 및 조건

| 용어 | 상세 정의 | 비유 |
|:---:|:---|:---|
| **이행 함수적 종속** | $X \to Y$ 이고 $Y \to Z$ 일 때, $X \to Z$ 가 성립하는 관계. | 친구의 친구는 내 친구 |
| **비주요 속성** | 기본 키나 후보 키에 포함되지 않는 일반 속성. | 조연 배우 |
| **결정자 (Determinant)** | 종속 관계에서 왼쪽(화살표 시작)에 위치한 속성. | 원인 |

### 2. 3NF 변환 과정 도해 (ASCII)

중간 매개 속성을 결정자로 하는 새로운 테이블을 추출하는 과정을 시각화한다.

```text
    [ 2NF 릴레이션: 사원_부서 ]
    PK: { 사원번호 }
    ┌──────────┬─────┬──────────┬──────────┐
    │ 사원번호  │ 성명 │ 부서번호  │ 부서위치  │
    ├──────────┼─────┼──────────┼──────────┤
    │   101    │ Kim │   D01    │  서울    │ <--- (부서위치는 부서번호에 종속)
    │   102    │ Lee │   D01    │  서울    │ <--- 중복 발생!
    │   103    │ Park│   D02    │  부산    │
    └──────────┴─────┴──────────┴──────────┘
               │
               ▼ [ 3NF 변환: 이행 종속 제거 ]
    ┌────────────────────┐      ┌────────────────────┐
    │   [ 사원 테이블 ]    │      │   [ 부서 테이블 ]    │
    ├──────────┬─────┬───┤      ├──────────┬─────────┤
    │ 사원번호  │ 성명 │D# │      │ 부서번호  │ 부서위치  │
    ├──────────┼─────┼───┤      ├──────────┼─────────┤
    │   101    │ Kim │D01│      │   D01    │  서울    │
    │   102    │ Lee │D01│      │   D02    │  부산    │
    │   103    │ Park│D02│      └──────────┴─────────┘
    └──────────┴─────┴───┘
    (부서가 서울에서 대전으로 이사 가도, 부서 테이블 한 줄만 고치면 끝!)
```

### 3. 심층 동작 원리: 비이행적 종속의 강제
- **조건**: $R$이 3NF이려면, $R$의 모든 함수적 종속성 $X \to A$에 대해 다음 중 하나를 만족해야 함.
    1. $X$가 슈퍼 키(Super Key)이다.
    2. $A$가 $R$의 주요 속성(Prime Attribute)이다.
- **의미**: 일반 속성이 결정자가 되는 상황을 차단하여 정보의 독립성을 확보함.

### 4. 핵심 기술: 데이터 무결성과 3NF
- **갱신 이상 해결**: 부서 위치 변경 시 발생하던 대규모 데이터 불일치 위험 제거.
- **삭제 이상 해결**: 마지막 사원을 지워도 부서 정보(위치 등)는 유지됨.

- **📢 섹션 요약 비유**: 3NF는 '한 문장에 주어가 두 개(PK와 일반 결정자) 있는 비문'을 두 개의 명확한 문장으로 나누는 과정입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 2NF vs 3NF vs BCNF 비교

| 정규형 | 주요 타겟 | 차이점 요약 |
|:---:|:---|:---|
| **2NF** | **부분** 함수 종속 | **PK의 일부**가 결정자인 경우 제거. |
| **3NF** | **이행** 함수 종속 | **일반 속성**이 결정자인 경우 제거. |
| **BCNF** | **모든** 결정자 자격 | **후보 키가 아닌** 결정자(주요 속성 결정 포함) 제거. |

### 2. 과목 융합 관점: 소프트웨어 공학 및 디자인 패턴
- **소프트웨어 공학(SE) 융합**: 3NF는 객체 지향의 **데메테르의 법칙(Law of Demeter)**과 유사하다. "직접적인 친구(PK)하고만 대화하라"는 원칙을 데이터 구조로 구현한 것이다.
- **디자인 패턴 융합**: 3NF로 분리된 테이블 구조는 **컴포지트 패턴(Composite Pattern)**이나 **브리지 패턴(Bridge Pattern)**처럼 엔티티 간의 결합도를 낮추고 유연성을 높인다.

- **📢 섹션 요약 비유**: 2NF가 '자식의 독립'이라면, 3NF는 '사촌과의 재산 분리'입니다. 가까운 사이일수록 경계를 명확히 해야 평화(정합성)가 유지됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 우편번호와 주소 모델링
- **상황**: 고객 테이블에 [고객ID, 이름, 우편번호, 기본주소, 상세주소]가 있음.
- **판단**: [우편번호 $\to$ 기본주소]라는 종속성이 존재하므로 3NF 위반.
- **결단**: 이론적으로는 우편번호 마스터 테이블로 분리해야 하지만, 실무에서는 주소 조회가 너무 빈번하고 우편번호 체계가 거의 변하지 않음. **전략적 비정규화(3NF 위반 허용)**를 택하여 성능을 우선시하되, 정기적인 데이터 클렌징으로 정합성 보완.

### 2. 안티패턴: "귀찮으니까 조인하지 말고 컬럼 더 만들자"
- **증상**: 3NF를 무시하고 반복되는 코드명, 부서명 등을 사원 테이블에 다 때려 넣음.
- **결과**: 나중에 코드명이 바뀌었을 때 일부 데이터만 수정되어 통계 데이터가 엉망이 됨.
- **조치**: **핵심 마스터 정보는 반드시 3NF를 준수**하여 분리하고, 성능 이슈는 인덱스나 캐시로 해결할 것.

### 📢 기술사적 결언
> "제3정규형은 현대 관계형 데이터베이스 설계의 **'가장 실천적인 표준'**이다. 3NF를 달성했다는 것은 데이터의 **논리적 자립**을 완성했다는 뜻이다. 기술사는 기계적인 분해에 그치지 않고, 각 속성이 비즈니스 도메인에서 어떤 **의미적 결정권**을 갖는지 파악해야 한다. 정규화의 깊이는 비즈니스의 **변경 빈도**와 비례해야 하며, 3NF는 그 변화의 파고를 견디는 가장 견고한 방파제가 될 것이다."

- **📢 섹션 요약 비유**: 3NF를 지키지 않는 것은 '한 바구니에 계란(사원)과 돌(부서 정보)을 같이 담는 것'과 같아, 돌을 옮기려다 계란을 깨뜨릴 수 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (3NF 준수 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 중복도** | 30~50% | **5% 미만** | 저장 효율성 및 정합성 향상 |
| **수정 영향도 분석** | 전수 조사 필요 | **해당 테이블만 확인** | 유지보수 공수 60% 절감 |
| **보고서 정확도** | 데이터 불일치로 불확실 | **100% 신뢰 가능** | 의사결정 품질 향상 |

### 2. 미래 전망
- **Microservices Orchestration**: 서비스별로 쪼개진 DB 사이의 정합성을 맞추기 위해, 3NF를 넘어선 **글로벌 메타데이터 관리** 기술이 핵심 경쟁력이 됨.
- **Denormalized Storage Engines**: 내부적으로는 3NF로 관리하되, 물리적으로는 비정규화하여 저장하는 **PAX**나 **Columnar Store**의 최적화 기술이 3NF의 단점(Join 부하)을 보완하며 발전할 것임.

- **📢 섹션 요약 비유**: 기술이 발전해도 '한 속성은 한 주제에만 충성한다'는 3NF의 명쾌한 논리는 데이터 세계의 헌법으로 남을 것입니다.
