# 120. 데이터 역엔지니어링 (Data Reverse Engineering) - 기존 스키마 분석 및 복원

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이미 운영 중인 물리적 데이터베이스(스키마)로부터 데이터의 구조, 관계, 제약 조건을 추출하여 상위 단계인 **논리적 또는 개념적 데이터 모델(ERD)**로 역산하여 복원하는 프로세스다.
> 2. **필요성**: 설계 문서가 없거나 현행화되지 않은 레거시 시스템의 구조를 파악하고, 차세대 시스템 구축이나 클라우드 마이그레이션 시 **AS-IS 분석**의 신뢰성을 확보하기 위한 필수 관문이다.
> 3. **가치**: 데이터베이스 내에 숨겨진 비즈니스 로직과 데이터 간의 종속성을 시각화함으로써, 데이터 자산의 가시성을 높이고 시스템 통합 및 리팩토링의 정확한 가이드를 제공한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**데이터 역엔지니어링 (Data Reverse Engineering)**은 '완성된 제품(물리 DB)'을 뜯어보며 '설계도(ER 모델)'를 다시 그리는 과정입니다. DBMS의 시스템 카탈로그 정보를 읽어 테이블, 컬럼, 인덱스, 외래 키 정보를 추출하고 이를 다이어그램으로 변환합니다.

### 2. 💡 비유: 유적지 발굴과 설계도 복원
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        데이터 역엔지니어링 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 수천 년 된 고대 성(레거시 DB)을 발견함. 설계도는 잃어버린 지 오래임.      │
│                                                                             │
│  1. 정엔지니어링: 설계도를 보고 성을 짓는 것. (일반적인 개발)                    │
│  2. 역엔지니어링:                                                             │
│     - 성벽의 두께와 방의 개수를 잼. (물리 스키마 추출)                           │
│     - 복도가 어디로 연결되는지 확인. (관계 분석)                                │
│     - 이를 바탕으로 원래 이 성이 어떤 모양이었는지 도면을 그림. (모델 복원)        │
│                                                                             │
│  → 낡고 복잡한 시스템을 **'눈에 보이는 그림'**으로 바꿔서 정체를 밝히는 작업입니다!  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **설계 문서의 부재**: 오랜 기간 운영되며 문서와 실제 DB가 따로 노는 '현행화 실패' 문제 심각.
- **차세대 전환 사업**: 기존 시스템을 버리고 새로 만들려면, 현재 데이터가 어떻게 얽혀 있는지 알아야만 데이터를 안전하게 옮길(Migration) 수 있음.

- **📢 섹션 요약 비유**: 데이터 역엔지니어링은 데이터베이스의 'X-ray 촬영'과 같아, 겉으로 보이지 않는 내부의 뼈대(구조)를 선명하게 보여줍니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 데이터 역엔지니어링의 주요 단계

| 단계 | 수행 활동 | 주요 도구 |
|:---:|:---|:---|
| **1. 추출 (Extraction)** | DBMS 카탈로그(Dictionary)에서 스키마 정보 수집. | SQL Query, CASE Tool |
| **2. 분석 (Analysis)** | 추출된 속성명, 타입 등을 통해 논리적 의미 유추. | 전문가 통찰 |
| **3. 관계 복원 (Linking)** | FK 제약 조건이나 데이터 패턴으로 관계(Line) 생성. | ERD 자동 생성기 |
| **4. 모델 시각화** | 상위 수준의 ER 다이어그램으로 도식화. | ERwin, DA#, SQL Developer |

### 2. 역엔지니어링 프로세스 도해 (ASCII)

물리 구조에서 개념 구조로 거슬러 올라가는 흐름을 시각화한다.

```text
    [ Physical DB ] (Oracle, MySQL, etc.)
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 1. [ Dictionary Scan ] : TABLE, COLUMN, PK, FK 정보 읽기         │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (Metadata XML/SQL)
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 2. [ Logical Inference ] : 물리 약어(CUST_NM)를 논리명(고객성명)으로 변환 │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (Logical Schema)
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 3. [ Conceptual Abstraction ] : 주제 영역별 엔티티 그룹화 및 시각화 │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    [ Conceptual ERD ] : **현행화된 최종 설계도 복원**
```

### 3. 심층 동작 원리: 고아 관계(Orphan Relation) 찾기
- **문제**: 성능상의 이유로 물리적 외래 키(FK)를 걸지 않은 레거시 DB가 많음.
- **해결 기술**:
    1. **이름 매칭**: 서로 다른 테이블에서 컬럼명이 같은 것(예: `USER_ID`)을 찾아 연결 제안.
    2. **데이터 분석**: 실제로 들어있는 값들을 비교하여 부모-자식 관계가 성립하는지 통계적으로 검증.

### 4. 핵심 기술: 메타데이터 매핑 (Word Dictionary)
- 물리 컬럼명(`ORD_DT`)을 보고 논리 속성명(`주문일자`)으로 자동 변환하기 위해, 사전에 정의된 표준 단어 사전을 대조하는 기술 활용.

- **📢 섹션 요약 비유**: 역엔지니어링은 '암호화된 편지(물리 이름)'를 '누구나 읽을 수 있는 평서문(논리 이름)'으로 번역하는 해독 과정입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 정엔지니어링(Forward) vs 역엔지니어링(Reverse)

| 비교 항목 | 정엔지니어링 (Forward) | 역엔지니어링 (Reverse) |
|:---:|:---|:---|
| **시작점** | 사용자 요구사항 / 설계도 | **실제 운영 중인 DB** |
| **방향** | 추상화 $\to$ 구체화 | **구체화 $\to$ 추상화** |
| **목표** | 새로운 시스템 구축 | **현행 시스템 파악 및 개선** |
| **정확도** | 설계자의 의도에 따름 | **100% 현상 반영 (진실)** |

### 2. 과목 융합 관점: 소프트웨어 공학 및 보안
- **소프트웨어 공학(SE) 융합**: 역엔지니어링은 **유지보수(Maintenance)** 단계의 핵심 기술이며, 기존 코드를 분석하여 모델을 추출하는 **정적 분석(Static Analysis)** 기법과 데이터베이스 관점에서 맥을 같이 한다.
- **보안(Security) 융합**: 역엔지니어링을 통해 데이터베이스 내에 **민감 정보(PII)**가 어디에 어떤 관계를 맺고 저장되어 있는지 전수 조사하여, 보안 정책 수립의 기초 자료로 활용한다.

- **📢 섹션 요약 비유**: 정엔지니어링이 '새 옷을 맞추는 것'이라면, 역엔지니어링은 '오래된 옷의 실밥을 풀어 패턴을 분석하는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 차세대 프로젝트의 데이터 분석
- **상황**: 20년 된 금융 시스템을 신규 구축하려 함. 테이블만 3,000개임.
- **판단**: 담당자들도 전체 구조를 다 아는 사람이 없음.
- **결단**: **전수 데이터 역엔지니어링 실시**. 도구를 사용하여 물리 DB를 긁어온 뒤, 표준 단어 사전을 적용해 ERD를 복원함. 3,000개의 테이블 중 안 쓰이는 유령 테이블 500개를 찾아내어 마이그레이션 대상에서 제외, 프로젝트 비용 20% 절감.

### 2. 안티패턴: 도구의 결과물을 100% 믿는 과신
- **증상**: 역엔지니어링 툴이 그려준 ERD를 그대로 최종 분석 문서로 제출.
- **결과**: 툴은 '물리적 연결'만 보여줄 뿐, 실제 비즈니스의 '의미적 연결'은 알지 못함. 엉뚱한 선들이 가득한 그림이 되어 분석에 방해만 됨.
- **조치**: 툴의 결과물은 **'초안'**으로만 사용하고, 반드시 업무 전문가가 수작업으로 주제 영역을 나누고 논리적 의미를 부여하는 **'정제 작업'**을 거칠 것.

### 📢 기술사적 결언
> "데이터 역엔지니어링은 **'과거의 거울을 통해 미래를 보는 기술'**이다. 지저분한 레거시 DB라고 무시할 것이 아니라, 그 안에 녹아있는 십수 년의 비즈니스 노하우를 '논리의 언어'로 구출해내야 한다. 기술사는 도구의 사용법보다 **데이터의 생명 주기와 의미적 종속성**을 파악하는 안목을 가져야 하며, 역엔지니어링 결과를 바탕으로 **데이터 표준화와 품질 개선**이라는 더 큰 그림을 그려낼 수 있어야 한다."

- **📢 섹션 요약 비유**: 도구에만 의존하는 역엔지니어링은 '번역기로 시를 번역하는 것'과 같아, 글자는 맞을지 몰라도 마음(비즈니스 의미)은 전해지지 않습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (역엔지니어링 수행 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **AS-IS 분석 시간** | 수개월 (수작업) | **수 일 (자동화)** | 분석 생산성 10배 향상 |
| **데이터 마이그레이션 정확도** | 불안정 (누락 위험) | **최상 (전수 파악)** | 전환 사고 90% 예방 |
| **IT 자산 가시성** | 어두움 (블랙박스) | **밝음 (시각화)** | 지식 자산화 및 인수인계 용이 |

### 2. 미래 전망
- **AI-Enhanced Reverse Engineering**: 단순 매칭을 넘어, AI가 쿼리 로그와 실제 데이터를 딥러닝으로 분석하여 물리적 제약이 없는 **논리적 관계를 99% 확률로 추론**해주는 기술 발전.
- **Real-time Live Modeling**: DB 스키마가 변경되는 즉시 역엔지니어링이 수행되어 문서(ERD)가 실시간으로 자동 현행화되는 **Continuous Modeling** 환경 보편화.

- **📢 섹션 요약 비유**: 미래의 역엔지니어링은 죽은 유적을 발굴하는 것이 아니라, 살아있는 생명체의 변화를 실시간으로 기록하는 '디지털 트윈' 기술로 진화할 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **데이터 역엔지니어링**은 완성된 레고 성을 보고, 거꾸로 **"이 성을 만들려면 어떤 블록들이 필요했는지" 설계도를 다시 그리는 것**과 같아요.
2. 성이 너무 복잡해서 누가 만들었는지 모를 때, **하나씩 뜯어보며 비밀 통로를 찾아내는** 탐험 같은 일이죠.
3. 이 일을 잘하면 나중에 성을 더 크고 멋지게 고칠 때 **실수하지 않고 잘 고칠 수** 있답니다!
