# 212. 충돌 직렬 가능성 (Conflict Serializable)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트랜잭션 스케줄이 **충돌(Conflict)** 연산의 순서를 직렬 스케줄(Serial Schedule)과 동일하게 유지하는 성질을 말하며, 데이터베이스의 정합성을 보장하는 가장 강력한 기준이다.
> 2. **가치**: 비직렬(Non-serial) 스케줄의 정당성을 수학적(그래프 이론)으로 증명하여 DBMS가 동시성 제어(Concurrency Control)의 안전성을 판단하는 필수 조건으로 활용된다.
> 3. **융합**: **OS (Operating System)**의 세마포어(Semaphore)나 **네트워크**의 분산 락(Distributed Lock)과 연결되는 이론적 토대이며, **Deadlock(교착 상태)** 방지 알고리즘의 근간이 된다.

---

### Ⅰ. 개요 (Context & Background)

**충돌 직렬 가능성(Conflict Serializability)**은 데이터베이스 트랜잭션 관리의 핵심 개념으로, **여러 트랜잭션이 동시에 실행되더라도 최종 결과가 어떤 순서로 실행하든 한 번에 하나씩 실행한 것과 동일해야 함**을 보장하는 엄격한 제약 조건입니다.

**💡 비유: 고속도로 톨게이트의 차량 정리**
동시에 여러 차량이 톨게이트를 통과하더라도(동시성), 통행료 정산 내역이 차량들이 하나씩 순서대로 통과했을 때의 기록과 정확히 일치해야 한다는 원칙입니다. 만약 두 차량이 충돌 위험이 있는 상황(같은 차로 진입)이라면, 진입 순서가 결과에 결정적 영향을 미치게 됩니다.

**등장 배경 및 필요성**
① **기존 한계**: 초기 데이터베이스는 단순한 직렬 실행(Serial Execution)만 지원하여 처리량(Throughput)이 극히 낮았음.
② **혁신적 패러다임**: 트랜잭션을 겹쳐 실행(Interleaving)하여 성능을 높이되, 데이터 무결성(Integrity)은 깨지지 않아야 한다는 **직렬 가능성(Serializability)** 이론 등장.
③ **비즈니스 요구**: 금융 거래나 재고 관리 등 `Read(읽기)`와 `Write(쓰기)`가 혼재되는 복잡한 OLTP(Online Transaction Processing) 환경에서 안전한 병행 제어가 필수적임.

**📢 섹션 요약 비유**
> 마치 악보를 보는 여러 피아노 연주자들이 동시에 연주하더라도, 악보의 화음(데이터 상태)이 어긋나지 않도록 서로 건반을 치는 순서를 맞추는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

충돌 직렬 가능성을 판단하기 위해서는 먼저 **충돌(Conflict)**의 정의와 **우선순위 그래프(Precedence Graph)** 분석이 필요합니다.

#### 1. 핵심 구성 요소 (Components)

| 요소 (Component) | 역할 (Role) | 내부 동작 (Mechanism) | 비유 (Analogy) |
|:---:|:---|:---|:---|
| **Conflict (충돌)** | 순서 종속성 판별 | 서로 다른 트랜잭션이 **동일 데이터**에 대해 최소 하나 이상의 **쓰기(Write)** 연산을 수행할 때 발생 | 두 사람이 동시에 같은 종이에 글씨 쓰기 |
| **Swap (교환)** | 순서 재배치 | 두 연산이 **비충돌(Non-conflict)**이라면 순서를 바꿔도 스케줄의 결과(Result)가 변하지 않음 | 순서 바꿔도 합이 같은 덧셈 |
| **Precedence Graph** | 순환 의존성 시각화 | 트랜잭션을 노드(Node), 충돌 순서를 방향성 있는 엣지(Edge)로 표현한 DAG(Directed Acyclic Graph) | 작업 간의 선후 관계도 |
| **Serial Schedule** | 비교 기준 | 트랜잭션이 시간상 겹침 없이 하나씩 순차적으로 실행되는 스케줄 | 줄 서서 차례대로 하는 식사 |
| **Cyclic (순환)** | 불가능 판정 | 그래프에서 사이클(Cycle)이 발생하면 순서 모순이 생겨 직렬 변환 불가능 | 닭이 먼저냐 달걀이 먼저냐의 모순 |

#### 2. 충돌(Conflict)의 세부 조건
두 연산 $O_i \in T_k$와 $O_j \in T_m$이 충돌하려면 다음 **3가지 조건**을 동시에 만족해야 합니다.
1. 서로 다른 트랜잭션에 속해야 한다. ($k \neq m$)
2. 동일한 데이터 항목(Data Item)에 접근해야 한다.
3. 적어도 하나의 연산이 **쓰기(Write)** 연산이어야 한다.

```text
[연산 유형별 충돌 여부 매트릭스]

  T1 연산   T2 연산      충돌 여부?      설명 (Reason)
  -------  -------     ------------    ---------------------------------
   Read     Read          NO            읽기는 데이터를 변경하지 않음 (상태 유지)
   Read     Write         YES           Read 시점에 따라 값이 달라짐 (Dirty Read 위험)
  Write     Read          YES           Write 후 Read 시 이전 값 vs 새 값 상이
  Write     Write         YES           최종 DB 상태가 최종 Write 연산자에 따라 달라짐
```

#### 3. 우선순위 그래프 (Precedence Graph) 분석 로직

**도입 서술**: 스케줄 $S$가 충돌 직렬 가능한지 확인하는 표준적인 알고리즘은 우선순위 그래프를 생성하여 사이클을 검사하는 것입니다. 이는 그래프 이론의 **Topological Sort(위상 정렬)** 가능성 여부와 같습니다.

```text
[Precedence Graph 생성 및 분석 알고리즘]

  스케줄 S: T1: R(A), T2: W(A), T1: W(A), T3: R(A), T2: R(B), T3: W(B)

  1. 연산 분석 및 Edge 생성:
     - T1(A) read ─┐
                    ├─ T2(A) write  (충돌: R-W) => Edge: T1 -> T2
     - T2(A) write ─┼─ T1(A) write  (충돌: W-W) => Edge: T2 -> T1
                    │
                    └─ T3(A) read   (충돌: W-R) => Edge: T2 -> T3

     - T2(B) read ─┬─ T3(B) write  (충돌: R-W) => Edge: T2 -> T3
                    │

  2. 결과 그래프 구조:
     T1 ◄─────┐
      │       │
      └──────► T2
              │
              └──────► T3

  [해설] 위 그래프는 T1 → T2, T2 → T1 간의 순환(Cycle)이 존재합니다.
  따라서 이 스케줄 S는 **Conflict Serializable 하지 않음(Non-CS)**.
  DBMS는 이러한 스케줄이 발생하지 않도록 Abort하거나 Lock을 강제합니다.
```

**📢 섹션 요약 비유**
> 충돌 직렬 가능성 검증은 **'GPS 경로 탐색'**과 같습니다. 모든 도로(연산)가 단방향일 때, 출발지에서 목적지까지 되돌아오지 않고(사이클 없음) 계속 나아갈 수 있는 경로(직렬 순서)가 존재하는지 확인하는 과정입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 직렬 가능성 분류 체계 (Hierarchy)

모든 스케줄은 충돌 직렬 가능성 여부에 따라 계층적으로 구분됩니다.

| 구분 (Classification) | 정의 (Definition) | 성능 (Performance) | 검증 복잡도 (Cost) |
|:---|:---|:---:|:---:|
| **Serial (직렬)** | 트랜잭션이 동시에 실행되지 않음 | ⭐ (낮음) | 매우 낮음 |
| **Conflict Serializable (CS)** | 충돌 연산의 순서가 직렬과 일치 | ⭐⭐⭐ (높음) | 낮음 (그래프 검사) |
| **View Serializable (VS)** | 최종 결과 상태(View)가 직렬과 일치 | ⭐⭐⭐ (높음) | 매우 높음 (NP-Complete 문제 포함) |
| **Non-Serializable** | 결과 불일치, 정합성 위반 | 최대 (위험함) | - |

* **핵심 차이**: CS는 VS의 상위 호환이지만, VS 중에 CS가 아닌 경우(예: 블라인드 라이트 등 특수 케이스)가 존재합니다. 실무 DBMS는 오버헤드와 구현 단순화를 위해 **CS**를 주로 보장합니다.

#### 2. 기술 스택 융합 분석 (Cross-Domain Analysis)

*   **💡 OS (Operating System)와의 연계**: **세마포어(Semaphore)**나 **뮤텍스(Mutex)**는 CS를 보장하기 위한 저수준(Low-level) 도구입니다. 공유 자원에 대한 접근 제어 원리가 동일합니다.
*   **💡 네트워크 (Network)와의 연계**: 분산 DB 환경에서는 **2PC (Two-Phase Commit)** 프로토콜이 사용되며, 이때 글로벌 충돌 직렬 가능성을 보장하기 위해 트랜잭션 타임스탬프나 전역 락 매니저가 필요합니다.
*   **💡 보안 (Security)과의 연계**: **Covert Channel(은닉 채널)** 차단을 위해 고립 레벨(Isolation Level)을 조절할 때, CS 보장 여부가 보안 정책의 중요한 변수가 됩니다.

**📢 섹션 요약 비유**
> 직렬 가능성의 단계는 **'요리사의 레시피 엄격함'**과 같습니다. '충돌 직렬 가능성'은 "순서는 바뀌어도 되지만, 재료를 넣는 타이밍은 지켜라"는 규칙이고, 단순 '뷰 직렬 가능성'은 "결국 맛만 같으면 되고, 과정은 상관없다"는 관대한 규칙입니다. 실무에서는 맛보다 과정(안전성)이 중요하므로 전자를 따릅니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 시나리오 및 의사결정 (Scenarios)

**시나리오 A: 금융 이체 시스템의 계좌 합계**
*   **문제**: 두 트랜잭션이 동시에 `Account A`에서 출금하여 `Account B`로 입금할 때, `Account A`의 잔고를 읽는(Read) 시점이 겹침.
*   **판단**: 이는 `Read(A) -> Write(A)` 충돌입니다.
*   **결정**: 충돌 직렬 가능성을 보장하기 위해 **Lock-Based Protocol(잠금 기반 프로토콜)** 적용. T1이 `A`를 수정하는 동안 T2는 대기해야 함.
*   **지표**: **Dirty Read(반복 가능하지 않은 읽기)** 방지, 데이터 정합성 100% 보장.

**시나리오 B: 통계 집계 쿼리 vs 실시간 업데이트**
*   **문제**: 리포트 생성(Read)이 오래 걸려 실시간 업데이트(Write)가 Blocking(지연)됨.
*   **판단**: 엄격한 CS 보장(Repeatable Read)이 성능 저하를 유발함.
*   **결과**: **MVCC (Multi-Version Concurrency Control)** 도입 검토. 읽기는 이전 버전(Snapshot)을 읽게 하여 Write와 충돌하지 않도록 함으로써, CS를 보장하면서도 동시성을 확보.

#### 2. 도입 체크리스트 (Checklist)

| 구분 | 체크 항목 | 설명 |
|:---|:---|:---|
| **설계** | **Locking Strategy** | 2PL (Two-Phase Locking)이나 MVCC 등 CS 보장 방안이 있는가? |
| **운영** | **Isolation Level** | `READ COMMITTED`는 CS를 보장하지 못할 수 있음. `REPEATABLE READ`나 `SERIALIZABLE` 필요성 검토. |
| **성능** | **Deadlock Handling** | CS 보장을 위한 Lock 경쟁이 과도하여 Deadlock이 발생하는가? (Timeout 설정 필요) |
| **보안** | **Information Leakage** | Lock 대기 시간(Time-based Side-channel)을 통해 정보 유출 가능성은 없는가? |

#### 3. 안티패턴 (Anti-Patterns)

*   **"그냥 대기해" (Sleep/Wait)**: 트랜잭션 충돌 시 랜덤 시간 대기(Sleep) 후 재시도하는 로직.
    *   **결함**: 충돌이 해결된다는 보장이 없으며, 스케줄이 비직렬 상태로 남을 수 있어 데이터 오염 위험.
*   **Lock 미사용 (Optimistic but Unchecked)**: 충돌이 드물다고 판단하여 Lock 없이 진행.
    *   **결함**: 높은 확률로 Race Condition 발생. `Lost Update` 문제로 이어져 비즈니스 로직 붕괴.

**📢 섹션 요약 비유**
> 실무 적용은 **'교통정리 교통경찰'**과 같습니다. 사고(충돌)가 나지 않도록 미리 신호등(Lock)을 설치하고, 신호를 무시하고 달리는 차량(안티패턴)을 단속해야 합니다. 하지만 너무 엄격한 신호 체계는 교통 체증(성능 저하)을 유발하므로, 우회 도로(MVCC)를 활성화하는 지혜가 필요합니다.
