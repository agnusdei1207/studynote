# 258. 벡터 시계 (Vector Clock)와 타임스탬프 - 분산 데이터의 인과율

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 벡터 시계는 분산 시스템 내의 각 노드가 독자적인 카운터 배열을 유지함으로써, 데이터 변경 이벤트 간의 **선후 관계(Causality)를 추적하고 동시(Concurrent) 수정을 감지하는 논리적 시계 기술**이다.
> 2. **가치**: 물리적 시계의 오차(Clock Skew)와 상관없이 데이터의 '누가 먼저인가'를 수학적으로 증명하며, 충돌 발생 시 이를 사용자에게 알려 병합(Merge)을 유도할 수 있게 한다.
> 3. **융합**: DynamoDB, Riak 등 가용성이 높은 분산 NoSQL 저장소의 데이터 정합성 유지 및 버전 관리에 필수적으로 융합된다.

+++

### Ⅰ. 벡터 시계의 작동 원리

- **구조**: 노드가 $N$개라면, 각 데이터 복제본은 $[v_1, v_2, \dots, v_n]$ 형태의 정수 배열을 메타데이터로 가집니다.
- **갱신 규칙**:
    1. 노드 $i$에서 데이터 수정 시, 자신의 카운터 $v_i$를 1 증가시킵니다.
    2. 다른 노드로 전송 시 이 벡터 전체를 함께 보냅니다.
    3. 데이터를 받은 노드는 자신의 벡터와 받은 벡터의 각 요소 중 최댓값을 선택하여 업데이트합니다.

+++

### Ⅱ. 충돌 감지 매커니즘 (ASCII Flow)

```text
[벡터 시계를 이용한 충돌 판별]

  Event A: [1, 0]  VS  Event B: [1, 1]
  ──▶ B의 모든 요소가 A보다 크거나 같음: "B는 A 이후에 일어난 사건" (OK!)

  Event C: [2, 1]  VS  Event D: [1, 2]
  ──▶ 엇갈림 발생! (C는 첫 번째가 크고, D는 두 번째가 큼)
  ──▶ 💥 [ CONFLICT DETECTED ] : "두 노드가 서로 모른 채 동시에 수정함"
```

+++

### Ⅲ. 실무적 의의: 충돌 해결 (Conflict Resolution)

- **LWW (Last Writer Wins)**: 타임스탬프가 가장 늦은 것을 선택하여 덮어쓰는 단순한 방식. (데이터 유실 위험)
- **Semantic Merge**: 벡터 시계로 충돌을 감지한 뒤, 쇼핑몰 장바구니처럼 두 데이터를 합치는(사과 1개 + 바나나 1개) 고도의 병합 로직.

- **📢 섹션 요약 비유**: 벡터 시계는 **'여러 명의 작가가 쓰는 공동 소설의 판본 관리'**와 같습니다. 내가 쓴 2페이지가 친구가 쓴 1페이지 뒤에 오는 건 명확하지만(인과관계), 서로 모르는 상태에서 각자 3페이지를 써버렸다면 누가 진짜 3페이지인지 싸우게 됩니다(충돌). 이때 벡터 시계는 "둘 다 동시에 썼으니 내용을 합쳐라!"라고 신호를 주는 편집자 역할을 합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Lamport Clock]**: 벡터 시계의 전신인 단순 정수형 논리 시계.
- **[Causality]**: 사건의 선후 관계를 나타내는 인과율.
- **[Versioning]**: 데이터의 변화 흐름을 추적하는 기술.

📢 **마무리 요약**: **Vector Clock**은 분산 시스템의 시간 경찰입니다. 물리적 시간의 부정확함을 극복하고, 데이터의 정당한 역사(History)를 수호합니다.
