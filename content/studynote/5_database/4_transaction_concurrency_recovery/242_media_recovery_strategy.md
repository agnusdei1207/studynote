# 242. 미디어 회복 (Media Recovery) - 최악의 상황을 위한 대비

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 미디어 회복은 디스크 파손이나 파일 시스템 오염 등 **비휘발성 저장 장치의 물리적/논리적 결함으로 인해 데이터베이스 파일이 손상되었을 때, 물리적 백업본과 트랜잭션 로그를 결합하여 데이터를 복원하는 절차**이다.
> 2. **가치**: 데이터베이스의 **ACID 특성 중 내구성(Durability)**을 최후의 수단에서 보장하며, **RPO (Recovery Point Objective, 복구 목표 시점)**를 '0(최소 손실)'에 근접하게 유지하여 비즈니스 연속성을 확보한다.
> 3. **융합**: OS의 파일 시스템 체크sum, 스토리지 계층의 **RAID (Redundant Array of Independent Disks)** 기술, 그리고 클라우드 환경의 스냅샷 기술과 상호 보완적으로 작동하는 데이터 보호의 최종 방어선이다.

+++

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 정의**
미디어 회복(Media Recovery)은 데이터베이스의 저장 매체(디스크, 스토리지)가 파손되었거나 데이터 파일이 삭제되는 등, **미디어 장애(Media Failure)**가 발생했을 때 수행하는 복구 메커니즘이다. 단순한 소프트웨어 오류(Transaction Failure)와 달리, 디스크 블록 자체가 접근 불가능한 상황을 가정하므로, 백업된 데이터 이미지를 이용한 **Restore(복원)** 단계와 로그를 이용한 **Recover(복구)** 단계로 명확히 구분된다.

**2. 등장 배경 및 필요성**
- **한계**: 기존의 **ACID (Atomicity, Consistency, Isolation, Durability)** 특성을 유지하기 위한 롤백(Rollback) 메커니즘은 메모리상의 트랜잭션만 관리하며, 디스크 파일의 물리적 손상에는 무력하다.
- **혁신**: 시간 순서대로 기록된 **REDO Log (Redo Log)**를 백업 파일에 순차적으로 재적용(Roll-forward)하는 기법을 도입하여, 과거 특정 시점의 백업을 최신 상태로 진화시킬 수 있게 되었다.
- **비즈니스 요구**: 24/7 비중단 서비스 환경에서, 하드웨어 교체 시 발생하는 수시간의 다운타임은 치명적이므로, 데이터 완전성과 신속한 복구가 필수적이다.

**3. 핵심 구성 요소**
미디어 회복을 위해 다음과 같은 인프라가 선행되어야 한다.
- **전체 백업 (Full Dump)**: 데이터베이스 전체를 특정 시점(T0)에 덤프한 파일. 복구의 기준점(Base)이 된다.
- **아카이브 로그 (Archive Log)**: 온라인 리두 로그가 덮어쓰기 전에 영구 보관된 로그 파일의 집합으로, T0 이후의 모든 변경 이력을 담고 있다.

> **📢 섹션 요약 비유**: 미디어 회복 시스템을 구축하는 것은 **'부서진 도자기를 복원하기 위한 설계도와 일지 보관 시스템'**을 만드는 것과 같습니다. 도자기가 깨졌을 때(장애 발생), 처음 만들었던 설계도대로 다시 만들고(백업 복원), 매일 기록해둔 일지(아카이브 로그)를 보며 그날 그날 칠해온 무늬를 그대로 다시 그려 넣어(롤포워드) 깨지기 직전의 상태로 되돌리는 준비 과정입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. 미디어 회복 아키텍처**
미디어 회복은 크게 **Restore(물리적 복구)** 단계와 **Recover(논리적 복구)** 단계로 나뉜다. 아키텍처는 백업 관리자, 아카이버(Archiver), 그리고 복구 프로세스(Redo Applicator)가 협력하는 구조를 가진다.

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 (Mechanism) | 관련 프로토콜/명령어 | 비유 |
|:---|:---|:---|:---|:---|
| **Data File Dump** | 복구 기준점 | DB 페이지 블록을 물리적으로 복사 | `RMAN> BACKUP DATABASE` | 깨진 도자기의 원형 틀 |
| **Archive Log** | 변경 이력 재현 | Commit된 트랜잭션의 Redo Record 기록 | Log Switch & Copy | 도자기에 칠하는 순서대로의 일지 |
| **Redo Log Buffer** | 메모리 변경 로그 | Cache에서 디스크로의 변경 사항 전달 로그 | LGWR(Log Writer) | 임시로 적어두는 메모장 |
| **Control File** | 구조 관리자 | 데이터 파일과 로그 파일의 매핑 정보(SCN) 저장 | System Metadata | 목차(Index) |
| **Recovery Manager** | 복구 총괄 | Restore 스케줄링 및 Redo Apply 수행 | `RMAN> RECOVER DATABASE` | 복원 작업을 지휘하는 도공 |

**2. 복구 프로세스 상세 흐름 (ASCII Architecture Diagram)**
미디어 회복은 데이터베이스가 마운트(Mount)된 상태에서 수행되며, 다음과 같은 데이터 흐름을 따른다.

```text

[ Phase 1: Restore (Physical Restore) ]
+----------------+          +----------------+          +----------------+
|   Backup Media |  (Read)  |   Target Disk  | (Write)  |   DB Data File |
| (T0 Image Dump)| --------▶ | (New Volume)   | ------->  |   (COPY ONLY)  |
+----------------+          +----------------+          +----------------+
       ▲                             |                             |
       |                         1. File Copy                   2. Old Data
       |                         (Non-logged)                   ( overwritten )
       
[ Phase 2: Roll-Forward (Logical Recovery) ]
+----------------+          +----------------+          +----------------+
| Archive Logs   | (Apply   |   Redo Engine  | (Update)  |   DB Data File |
| (T0 ~ T_last)  | =======> |  (SMON/RMAN)   | =======>  |   (Applying)   |
+----------------+          +----------------+          +----------------+
       ▲                             |                             |
       |                         3. Redo Apply                 4. Data Changes
       |                         (SCN Sequencing)              (Replicating)

[ Phase 3: Roll-Back (Undo Uncommitted) ]
+----------------+          +----------------+          +----------------+
| Undo Segments  | (Fetch   |   Undo Engine  | (Revert)  |   DB Data File |
| (for Active TX)| --------> |   (Transaction)| ------->  |   (Clean Up)   |

                                     ▲
                                     |
                               5. Consistency Check
                               (Rollback uncommitted TXs)
```

**3. 심층 동작 원리 및 알고리즘**
- **SCN (System Change Number) 기반 동기화**: 복구는 SCN이라는 시계열 번호를 기준으로 수행된다. 백업 파일은 SCN=1000 시점이며, 장애 발생 시점이 SCN=5000이라고 가정할 때, 복구 시스템은 1001부터 5000까지의 로그를 순차적으로 적용한다.
- **로그 적용(Log Apply) 로직**:
    1. `Restore Phase`: 백업본을 디스크에 복사한다. 이 시점의 데이터는 일관성이 있으나 과거의 데이터이다.
    2. `Roll-forward Phase`: 아카이브 로그와 온라인 리두 로그를 읽어 데이터 블록에 변경 사항을 재적용(Redo)한다. 이 과정에서 커밋된 트랜잭션은 모두 반영된다.
    3. `Roll-back Phase`: 장애 발생 시점에 활성화되어 있었던(Commit되지 않은) 트랜잭션은 Undo 세그먼트를 이용해 롤백하여 데이터의 일관성을 맞춘다.

> **📢 섹션 요약 비유**: 이 과정은 **'영화 촬영 테이프와 편집본을 이용한 장면 복구'**와 같습니다. 우선 촬영 초기 원본(백업)을 가져온 뒤, 지금까지 찍어둔 모든 추가 컷(아카이브 로그)을 시간 순서대로 합치고(Roll-forward), 마지막으로 촬영 중이던 장면은 아직 완성되지 않았으므로 삭제해버리는(Roll-back) 심사 과정을 거쳐 완성된 영화(일관된 DB)를 복원합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 심층 기술 비교: Crash Recovery vs. Media Recovery**
데이터베이스 복구 기술은 장애의 유형에 따라 다르게 작동한다. 아래는 인스턴스 장애(Crash)와 미디어 장애(Media)의 비교 분석표이다.

| 비교 항목 (Metric) | Crash Recovery (인스턴스 장애) | Media Recovery (미디어 장애) |
|:---|:---|:---|
| **장애 원인** | 정전, 프로세스 강제 종료(OS 재부팅 등) | 디스크 헤드 충돌, 파일 시스템 손상 |
| **손상 범위** | 메모리(Dirty Buffer) 및 일부 디스크 | 데이터 파일 전체(물리적 파손) |
| **복구 데이터 소스** | **Online Redo Log (Active/Current)** | **Backup Dump + Archive Log** |
| **복구 속도** | 매우 빠름 (초~수분 내) | 상대적으로 느림 (파일 복사 시간 소요) |
| **필수 전제 조건** | 로그 파일의 무결성 | 주기적인 물리적 백업 필수 |
| **데이터 손실 가능성** | 없음 (Commit된 데이터 보장) | 백업 이후 로그 없을 시 손실 발생 |

**2. 타 과목 융합 관점 (Convergence)**
- **OS (Operating System)와의 융합**: 파일 시스템(File System) 수준의 **fsck** 나 **CHKDSK**는 논리적 손상은 복구할 수 있으나 섹터 파손에 대해서는 미디어 회복이 필요하다. OS 레벨의 **RAID (Redundant Array of Independent Disks)** 기술과 결합하여 디스크 자체의 중복성을 확보한 뒤에도 최종 방어선으로 DB 레벨의 미디어 회복이 필요하다.
- **네트워크와의 융합**: 원격지 재해 복구(Disaster Recovery)를 위해 네트워크 대역폭을 이용해 **Standby DB**로 아카이브 로그를 실시간 전송(Shipping)하는 기술이 미디어 회복과 강력한 시너지를 낸다.

> **📢 섹션 요약 비유**: 크래시 복구는 **'작업 중 갑작스러운 정전으로 컴퓨터가 꺼졌을 때, 자동 저장 문서로 되살리는 기능'**과 같다면, 미디어 회복은 **'하드디스크 자체가 날아가버려서, 외부 하드에 백업해둔 파일을 가져와서 다시 설치하는 작업'**과 같습니다. 전자는 전원만 다시 연결하면 되지만 후자는 장치 교체 및 복구가 필요한 근본적인 차이가 있습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. 실무 시나리오 및 의사결정 프로세스**
- **상황**: 금융권 핵심 트랜잭션 서버에서 RAID 5 구성의 디스크 2개가 동시에 고장 나 데이터베이스가 마운트되지 않음 (Double Failure).
- **의사결정 트리**:
    1. **물리적 하드웨어 교체**: 새 디스크를 장착하고 OS 레벨 볼륨을 재구성한다.
    2. **백업본 확인**: 가장 최근의 '합리적 무결성을 가진 백업(RMAN Backup)' 식별. (예: 24시간 전 전체 백업)
    3. **복구 시나리오 수립**: 
        - RMAN (Recovery Manager)을 사용하여 데이터 파일을 새 볼륨에 `RESTORE`.
        - `UNTIL TIME` 절을 사용하여 장애 발생 1분 전까지 `RECOVER` 명령 수행.
    4. **검증**: 데이터 무결성 체크(`DBMS_REPAIR` 등) 후 서비스 오픈.
- **기술적 판단**: 단순 백업만 덮어씌우면 24시간 치 데이터가 유실되므로, 반드시 **Archive Log Mode**를 활성화하여 로그를 연속 적용해야 한다.

**2. 도입 및 운영 체크리스트**
| 구분 | 체크 항목 (Checklist Item) | 상세 설명 |
|:---|:---|:---|
| **기술적** | **Archive Log Mode** | 데이터베이스가 아카이브 로그 모드로 운영되어야 함. (NOARCHIVELOG 시 복구 불가) |
| **기술적** | **Multiplexed Redo Log** | 로그 멤버를 다른 디스크에 중복 저장하여 로그 파일 자체의 파손에 대비 |
| **운영적** | **백업 주기 정책** | 전체 백업(Level 0)과 증분 백업(Level 1)의 주기 설정 |
| **보안적** | **백업 파일 암호화** | 미디어 유출 시 데이터 노출 방지를 위한 암호화(TDE) 적용 여부 |

**3. 안티패턴 (Anti-pattern)**
- ❌ **Archiving을 하지 않은 상태에서의 Media Recovery**: 로그가 덮어씌워져 복구 시점이 백업 시점으로 고정됨.
- ❌ **Control File 미백업**: 데이터 파일과 로그 파일의 매핑 정보가 손실되면 복구 시작 자체가 불가능해짐.

> **📢 섹션 요약 비유**: 미디어 회복 전략을 수립하는 것은 **'자동차 사고 대비 보험 및 블랙박스 설치'**와 같습니다. 사고가 나면 당황하지 않고 블랙박스(아카이브 로그) 영상과 견적서(백업)를 근거로 차량을 원래 상태로 수리(복구)하듯이, 평소에 보험(백업 정책)과 블랙박스(로그 관리)를 제대로 설치해두
