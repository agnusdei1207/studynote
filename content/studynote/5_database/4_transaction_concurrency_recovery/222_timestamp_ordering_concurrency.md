# 222. 타임스탬프 순서 (Timestamp Ordering) 기법
## # [타임스탬프 순서 기법]
### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트랜잭션의 진입 순서를 논리적 시간 값으로 부여하여, 명시적인 락(Lock) 매커니즘 없이 **충돌 직렬 가능성(Conflict Serializability)**을 보장하는 비관적 동시성 제어 기술이다.
> 2. **가치**: **Deadlock (교착 상태)**이 근원적으로 발생 불가능한 구조를 가지며, 락 관리 오버헤드가 없어 고속 처리가 필요한 분산 데이터베이스 환경에서 유리하다.
> 3. **융합**: 분산 시스템의 **TrueTime (Google Spanner)** 및 **Lamport Timestamp**와 결합하여 글로벌 트랜잭션 순서를 보장하거나, MVCC (Multi-Version Concurrency Control)의 기본 베이스로 활용된다.
+++

## Ⅰ. 개요 (Context & Background)

### 개념 및 정의
**타임스탬프 순서 (Timestamp Ordering, TO)** 기법은 트랜잭션 간의 순서를 결정하기 위해 데이터 항목에 대한 잠금을 사용하는 대신, 트랜잭션이 시작될 때 시스템이 부여한 유일한 **타임스탬프 (Timestamp, TS)**를 기준으로 충돌을 해결하는 기법이다.
이 기법은 모든 데이터 항목 $Q$에 대해 마지막으로 읽거나 쓴 트랜잭션의 시간 정보인 **R-TS(Q) (Read-Timestamp)**와 **W-TS(Q) (Write-Timestamp)**를 유지하며, 이를 통해 **"과거의 트랜잭션이 미래의 상태를 변경하려는 시도"**를 사전에 차단한다.

### 💡 비유: 수정할 수 없는 공문서 번호
각 트랜잭션은 발행된 순서대로 고유한 '공문서 번호(타임스탬프)'를 부여받는다. 데이터베이스는 이 번호를 보고 "100번 문서보다 200번 문서가 나중이다"라는 것을 즉시 알아본다. 만약 200번을 먼저 처리했는데, 나중에 100번이 끼어들어 수정을 시도하면 "이미 시간이 앞서간 일이 발생했으므로 100번은 무효(Rollback)"라고 판단한다.

### 등장 배경 및 필요성
1.  **기존 Locking 방식의 한계**: 2PL (Two-Phase Locking) 등은 락 획득을 위해 대기해야 하며, 이 과정에서 **Deadlock**이 발생할 수 있다.
2.  **기술적 패러다임 변화**: "기다림(Wait)" 대신 "즉시 롤백(Restart)"을 통해 시스템의 처리량(Throughput)을 높이고 복잡한 락 관리 오버헤드를 제거하고자 하는 **Non-blocking** 기술이 대두되었다.
3.  **비즈니스 요구**: 분산 데이터베이스 환경에서 노드 간의 합의 없이도 독립적으로 트랜잭션의 선후 관계를 판단할 수 있는 확장 가능한(Scalable) 기술이 필요했다.

### 📢 섹션 요약 비유
이 기법은 마치 **'번호표 순서대로만 입장이 가능한 철도역 자동개찰기'**와 같습니다. 뒷번호(나중 타임스탬프) 사람이 먼저 지나갔는데, 앞번호(이전 타임스탬프) 사람이 뒤늦게 와서 "내가 먼저 들어갈 거야"라고 소리쳐도 개찰기는 "이미 다른 사람이 지나갔으니 입장 불가"라고 면제권을 회수(Rollback)해버리는 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (Components)
| 요소 (Component) | 역할 (Role) | 내부 동작 및 파라미터 | 프로토콜/형식 | 비유 |
|:---|:---|:---|:---:|:---|
| **TS(T)** (Timestamp) | 트랜잭션 식별자 및 순서 | 트랜잭션 시작 시 시스템에 부여되는 단조 증가하는 값 (Logical Clock) | Integer | 발행 번호표 |
| **W-TS(Q)** | 데이터 쓰기 시점 | 데이터 항목 $Q$에 마지막으로 쓰기 성공한 트랜잭션의 $TS$ 값 | $Q$의 Header | 최종 수정일 |
| **R-TS(Q)** | 데이터 읽기 시점 | 데이터 항목 $Q$를 마지막으로 읽기 성공한 트랜잭션의 $TS$ 값 | $Q$의 Header | 최종 열람일 |
| **Scheduler** | 순서 검증기 | 연산 요청마다 $TS(T)$와 $W/R\text{-}TS(Q)$를 비교하여 수행/중단 결정 | Validation Logic | 출입 통제관 |
| **Rollback Module** | 충돌 처리 | 위반 트랜잭션을 즉시 Abort하고, 나중에 재시작(Restart) 스케줄링 | Abort Protocol | 퇴장 직원 |

### ASCII 구조 다이어그램 + 해설

**1. 데이터 항목의 구조**
각 데이터베이스의 레코드(데이터 항목)는 사용자의 데이터 외에도 메타데이터로서 타임스탬프 정보를 포함하고 있다.

```text
[Data Item Structure in Memory/Disk]

+-------------------------------------------------------+
|  Data Item Q: "Employee Salary"                       |
+-------------------------------------------------------+
|  User Data: 5000 (Value)                              |
+-------------------------------------------------------+
|  Metadata (Timestamps):                               |
|   - W-TS(Q): 105              <-- Last Written by T105 |
|   - R-TS(Q): 102              <-- Last Read by T102    |
+-------------------------------------------------------+
```

**해설**:
위 구조에서 볼 수 있듯이, 실제 값 외에 **W-TS(Q)**와 **R-TS(Q)**라는 두 가지 중요한 제어 정보가 존재한다. 새로운 트랜잭션 $T110$이 접근하면 자신의 타임스탬프(110)가 이 두 값보다 큰지(미래인지) 확인한다. 만약 $T101$이 접근한다면(101 < 105/102), 본인이 과거의 시간대에 속해 있음을 인지하고 규칙에 따라 처리된다.

**2. 핵심 알고리즘 및 수행 흐름**

트랜잭션 $T$가 데이터 $Q$를 읽거나 쓸 때의 **확인(Validation)** 절차는 다음과 같다.

```text
[Timestamp Ordering Algorithm Flow]

  ┌─────────────────────────────────────────────┐
  │  Transaction T requests READ(Q)             │
  └───────────────┬─────────────────────────────┘
                  │
                  ▼
          ┌──────────────────┐
          │ Check: TS(T)     │
          │   < W-TS(Q) ?    │  <-- 과거의 트랜잭션이 미래에 쓰인 데이터를 읽으려 함?
          └────┬────────┬────┘
               │ NO     │ YES (Past Read -> Future Write Violation)
               ▼        ▼
      ┌──────────┐  ┌──────────────────┐
      │ PERFORM  │  │ REJECT & ROLLBACK│
      │ Read(Q)  │  │ T                │
      │          │  │ (Ignore)         │
      └──────────┘  └──────────────────┘
      
      [If Perform]
      Update: R-TS(Q) <- MAX(R-TS(Q), TS(T))

  ┌─────────────────────────────────────────────┐
  │  Transaction T requests WRITE(Q)            │
  └───────────────┬─────────────────────────────┘
                  │
                  ▼
          ┌──────────────────┐
          │ Check: TS(T)     │
          │   < R-TS(Q) ?    │  <-- 과거의 트랜잭션이 미래에서 읽힌 데이터를 덮어씀?
          │   OR             │
          │ TS(T) < W-TS(Q) ?│  <-- 과거의 트랜잭션이 미래에 쓰인 데이터를 다시 씀?
          └────┬────────┬────┘
               │ NO     │ YES (Conflict Detected)
               ▼        ▼
      ┌──────────┐  ┌──────────────────┐
      │ PERFORM  │  │ REJECT & ROLLBACK│
      │ Write(Q) │  │ T                │
      │          │  │ (Ignore)         │
      └──────────┘  └──────────────────┘
      
      [If Perform]
      Update: W-TS(Q) <- TS(T)
```

**해설**:
1.  **Read(Q) 검증**: 만약 $T$의 타임스탬프가 $Q$의 **W-TS**보다 작다면($TS(T) < W-TS(Q)$), 이는 "이미 나중에 쓰인 값"을 "과거의 트랜잭션"이 읽으려는 상황이므로 **시간 역순 불가능**으로 간주하여 거절(Rollback)한다.
2.  **Write(Q) 검증**: 두 가지 경우를 모두 검사한다.
    *   $TS(T) < R\text{-}TS(Q)$: "이미 나중에 읽힌 값"을 "과거의 트랜잭션"이 덮어쓰려는 경우 (읽기 모순).
    *   $TS(T) < W\text{-}TS(Q)$: "이미 나중에 쓰인 값"을 "과거의 트랜잭션"이 다시 덮어쓰려는 경우 (쓰기 모순).
    *   이 중 하나라도 해당되면 Rollback한다. 그렇지 않다면 수행하고 해당 타임스탬프 값을 갱신한다.

### 심층 동작 원리 (Thomas Write Rule 포함)
기본적인 TO 기법은 쓰기 충돌 시 무조건 롤백하지만, **Thomas Write Rule**이라는 변형 알고리즘이 존재한다. 이는 "늦게 도착한 쓰기 연산(이미 과거의 시간을 가진 연산)"이 기존 데이터보다 오래되었다면, 굳이 롤백하지 않고 **단순히 무시(Ignore)**하는 최적화 기법이다. 이는 "이미 최신 데이터가 있으니 굳이 옛날 데이터로 덮어쓸 필요 없다"는 논리로, 불필요한 트랜잭션 중단을 줄여준다.

### 📢 섹션 요약 비유
이 매커니즘은 **'위조지폐 단속반'**과 같습니다. 누군가 1990년에 돈을 썼다고 주장하는데, 그 돈에 찍힌 위조 방지 장치(타임스탬프)가 2050년 기술로 되어 있다면, 단속반은 "너는 시간 여행자가 아니라 사기꾼이다"라고 판단하여 그 거래를 강제로 취소시켜버립니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 심층 기술 비교 (Locking vs. Timestamp)

| 구분 | Locking (e.g., 2PL) | Timestamp Ordering (TO) |
|:---|:---|:---|
| **매커니즘** | 공유(S) / 배타(X) 락을 사용하여 상호 배제 | 논리적 시계(Counter/Clock)를 사용하여 순서 부여 |
| **대기 정책** | **Wait (기다림)**: 락을 획득할 때까지 대기 | **No-Wait (즉시 판단)**: 충돌 시 즉시 Rollback 및 Restart |
| **Deadlock** | 발생 가능 (주기적 탐지/예방 필요) | **발생하지 않음 (Deadlock-free)** |
| **Starvation** | 발생 가능하지만 해결 알고리즘 존재 | 충돌이 빈번하면 계속 Rollback되어 기아(Starvation) 발생 가능 |
| **Concurrency** | 락 획득 구간에 따라 병행성 제약 발생 | 타임스탬프만 다르면 매우 높은 병행성 가능 |
| **주요 사용처** | 일단위 트랜잭션, 충돌이 빈번한 OLTP | 분산 DB, 실무 시스템, 충돌이 드문 환경 |

### 과목 융합 관점
1.  **운영체제 (OS)와의 융합**: 타임스탬프 기법은 **Lamport Logical Clock**이나 **Vector Clock**의 개념을 DBMS에 특화시킨 것이다. 분산 시스템에서 이벤트의 선후 관계(Happened-Before)를 파악하는 핵심 원리를 공유한다.
2.  **네트워크와의 융합**: 분산 데이터베이스 환경에서 물리적 클럭 싱크(NTP, Network Time Protocol)가 불가능할 때, 트랜잭션의 순서를 보장하기 위한 핵심 프로토콜로 사용된다. (예: Google Spanner의 TrueTime 기반 트랜잭션 커밋)

### 비교 분석표 (성능 및 오버헤드)

| 지표 | Locking 기반 | Timestamp 기반 | 의사결정 포인트 |
|:---:|:---:|:---:|:---|
| **CPU Overhead** | 락 매니저 관리, 대기 큐 관리 | 타임스탬프 비교 연산 (매우 가벼움) | **TO 승** (연산 단순) |
| **Memory Overhead** | 락 테이블(Lock Table) 유지 | 모든 데이터에 R/W-TS 유지 | 데이터 양이 많으면 TO 부담 |
| **Response Time** | 평균적으로 느림 (대기 시간 포함) | 빠름 (대기 없음) / 최악의 경우 롤백 반복 | **TO 승** (응답 시간 중시) |
| **Throughput** | 충돌 시 대기하므로 처리량 저하 가능성 | 충돌 시 폐기하므로 낭비 발생 가능성 | 충돌률이 낮을 때 **TO 승** |

### 📢 섹션 요약 비유
Locking은 **'철길 건널목 차단기'**입니다. 한 기차가 지나가면(True) 다른 기차는 멈춰서 기다려야 해서 전체 흐름이 느려집니다. 반면 타임스탬프는 **'하이패스 차선'**입니다. 그냥 달리기만 하면 되지만, 만약 삐끅해서 룰을 어기면 즉시 "벌금(Rollback)"을 물리고 퇴장당합니다. 차선이 넓고 차가 별로 안 막힌다면 하이패스가 훨씬 빠릅니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오 및 의사결정

1.  **시나리오:
