# 252. Saga 패턴 - MSA 환경의 유연한 결합

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사가(Saga) 패턴은 분산된 여러 마이크로서비스에 걸친 긴 트랜잭션을 처리하기 위해, **각 서비스별 로컬 트랜잭션을 순차적으로 실행하고 실패 시 보상 트랜잭션(Compensating Transaction)을 통해 정합성을 맞추는 기법**이다.
> 2. **가치**: 2단계 커밋(2PC)의 락(Lock) 병목 문제를 해결하여 시스템의 **가용성과 성능을 극대화**하며, '결과적 일관성(Eventual Consistency)' 모델을 실무에 적용한다.
> 3. **융합**: 이벤트 주도 아키텍처(EDA) 및 메시지 브로커(Kafka 등)와 결합되어, 복잡한 비즈니스 프로세스를 비동기적으로 조율하는 현대 MSA의 표준 트랜잭션 관리 전략으로 자리 잡았다.

+++

### Ⅰ. Saga 패턴의 작동 원리

- **정의**: 하나의 거대한 분산 트랜잭션을 여러 개의 독립적인 로컬 트랜잭션들의 연속체로 정의합니다.
- **실패 대응 (보상 트랜잭션)**: 중간에 특정 단계가 실패하면, 이미 성공했던 앞선 단계들의 결과를 취소하는 별도의 API(보상 트랜잭션)를 호출하여 시스템을 원상 복구합니다.

+++

### Ⅱ. Saga 패턴의 두 가지 유형 (ASCII Diagram)

1. **Choreography (코레오그래피)**: 중앙 통제자 없이 서비스들끼리 이벤트를 주고받으며 알아서 움직이는 방식.
2. **Orchestration (오케스트레이션)**: 중앙의 'Saga 인스턴스'가 각 서비스에게 명령을 내리고 상태를 관리하는 방식.

```text
[Orchestration 기반 Saga 흐름]

  [ Saga Manager ] ─── 1. 주문 생성 요청 ──▶ [ 주문 Service ]
         │                                      │ (Success)
         │ ◀────── 2. 주문 완료 이벤트 ─────────┘
         │
         │ ─── 3. 결제 처리 요청 ──▶ [ 결제 Service ]
         │                                      │ (Fail! 💥)
         │ ◀────── 4. 결제 실패 이벤트 ─────────┘
         │
         │ ─── 5. 주문 취소(보상) ──▶ [ 주문 Service ]
         ▼                                      ✅ (Rollback 효과)
```

+++

### Ⅲ. 2PC vs Saga 패턴

| 비교 항목 | 2단계 커밋 (2PC) | Saga 패턴 |
|:---|:---|:---|
| **일관성** | 강한 일관성 (ACID) | 결과적 일관성 (BASE) |
| **성능 (TPS)** | 낮음 (Lock 대기 심함) | **높음** (비동기 처리) |
| **복잡도** | 낮음 (DB 엔진 지원) | **높음** (개발자가 보상 로직 구현) |
| **적합 환경** | 금융 코어 시스템 | 일반 이커머스, 소셜 서비스 |

- **📢 섹션 요약 비유**: Saga 패턴은 **'택배 릴레이 배달'**과 같습니다. 첫 번째 기사가 옆 동네 기사에게 물건을 넘기고(이벤트), 그 기사가 다음 기사에게 넘깁니다. 만약 마지막 동네가 폐쇄되었다면(장애), 역순으로 다시 물건을 돌려주며(보상 트랜잭션) 최종적으로 발송인에게 물건을 되돌려주는 유연한 시스템입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Compensating Transaction]**: Saga의 심장부.
- **[Eventual Consistency]**: 사가가 지향하는 데이터 철학.
- **[Exactly-once]**: 메시지 큐와 결합 시 반드시 고민해야 할 데이터 신뢰성.

📢 **마무리 요약**: **Saga 패턴**은 MSA를 성공시키기 위한 필수 도구입니다. 강한 일관성을 고집하기보다, 비즈니스 성격에 맞는 유연한 정합성 전략을 수립하는 것이 핵심입니다.
