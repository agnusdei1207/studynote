# 241. 검사점 (Checkpoint) 회복 기법

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 검사점(Checkpoint) 기법은 로그 기반 회복 시 **로그 전체를 조사해야 하는 비효율을 해결**하기 위해, 주기적으로 메모리 버퍼의 내용을 디스크에 강제 동기화하고 그 시점을 로그에 기록하는 기술이다.
> 2. **가치**: 장애 복구 시 로그를 읽어야 할 시작점을 최근 검사점으로 제한함으로써 **복구 시간(RTO)을 획기적으로 단축**시킨다.
> 3. **융합**: ARIES 복구 알고리즘의 기초가 되며, 시스템 성능 부하를 줄이기 위해 백그라운드에서 조금씩 데이터를 내리는 'Fuzzy Checkpointing'으로 융합 진화했다.

+++

### Ⅰ. 체크포인트의 필요성

- **문제점**: 로그가 수 기가바이트(GB) 쌓여있을 때 장애가 나면, DB는 수년 전의 로그부터 읽어야 할 수도 있습니다. 이는 복구에 수 시간이 걸리게 만듭니다.
- **해결책**: 주기적으로 "여기까지는 디스크에 안전하게 저장 완료!"라는 도장(Checkpoint Record)을 찍어둡니다.

+++

### Ⅱ. 체크포인트 실행 과정 (ASCII Flow)

```text
[Checkpoint 수행 프로세스]

  1. 트랜잭션 일시 정지 (또는 큐잉)
  2. [ Buffer Pool ]의 모든 더티 페이지(Dirty Page)를 [ Data File ]로 Flush ──▶ 💾
  3. 로그 파일에 <CHECKPOINT LSN> 레코드 기록
  4. 로그의 시작 지점(Redo Start) 정보 갱신
  5. 트랜잭션 재개
```

+++

### Ⅲ. 회복 시 활용

- **Redo 범위**: 체크포인트 이후에 커밋된 트랜잭션만 재실행합니다.
- **Undo 범위**: 체크포인트 시점에 실행 중이었거나 그 이후에 시작된 트랜잭션 중 커밋되지 않은 것을 취소합니다.

- **📢 섹션 요약 비유**: 체크포인트는 **'게임의 세이브 포인트'**와 같습니다. 보스를 만나기 전 세이브를 해두면, 죽었을 때 게임 처음부터 다시 시작할 필요 없이 세이브 지점부터 바로 이어서 할 수 있는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Dirty Page]**: 디스크와 내용이 다른 메모리 페이지.
- **[LSN]**: 체크포인트 위치를 지정하는 로그 순서 번호.
- **[Fuzzy Checkpoint]**: 서비스 중단 없이 수행하는 고도의 기법.

📢 **마무리 요약**: **Checkpoint**는 복구의 효율성을 결정짓는 마침표입니다. 적절한 체크포인트 주기를 설정하는 것이 안정적인 시스템 운영의 핵심입니다.
