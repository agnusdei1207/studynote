# 203. 갱신 손실 (Lost Update) - 동시 수정의 대착사

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 갱신 손실(Lost Update)은 다중 트랜잭션 환경에서 **후속 트랜잭션의 커밋(Commit)이 선행 트랜잭션의 변경 내용을 무효화**시키는 동시성 제어(Convurrency Control)의 실패 현상이다.
> 2. **가치**: 재고 관리, 포인트 적립 등 데이터 정합성이 생명인 금융/ERP 시스템에서 **치명적인 데이터 유실**을 야기하며, 이를 해결하기 위해 DBMS의 격리 수준(Isolation Level) 조정 및 락(Lock) 전략 수립이 필수적이다.
> 3. **융합**: 분산 데이터베이스(Distributed DB) 환경에서의 **일관성(Consistency)** 모델과 연결되며, 단일 서버의 로컬 락을 넘어 분산 락(Distributed Lock)이나 버전 관리 전략으로 확장된다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 정의**
갱신 손실(Lost Update)은 데이터베이스 트랜잭션(Transaction)의 **ACID 특성 중 원자성(Atomicity)과 격리성(Isolation)**이 보장되지 않을 때 발생하는 대표적인 동시성 문제입니다. 구체적으로, 두 개 이상의 트랜잭션이 동일한 데이터 항목을 읽고 수정(Write)할 때, 나중에 수행된 트랜잭션이 먼저 커밋된 트랜잭션의 변경 사항을 인식하지 못하고 단순히 덮어쓰는 경우를 말합니다. 이는 데이터베이스의 논리적 정합성을 깨뜨리는 주범입니다.

**2. 기술적 배경 및 필요성**
초기 데이터베이스 시스템이나 낮은 격리 수준(Read Uncommitted)을 사용하는 환경에서는 성능을 위해 쓰기 락(Write Lock)을 즉시 획득하지 않거나, 체크포인트(Checkpoint) 전의 dirty data를 허용하는 경우가 많았습니다. 그러나 전자상거래(E-commerce) 및 핀테크(FinTech) 시스템이 발전하면서, '재고 부족 과다 판매'나 '잔액 증발'과 같은 갱신 손실 오류는 막대한 금전적 손실로 이어지게 되었습니다. 이를 방지하기 위해 트랜잭션 처리 모니터(TPM, Transaction Processing Monitor)와 로킹 매니저(Locking Manager)의 중요성이 강조되었습니다.

**3. 💡 직관적 비유**
A4 용지에 적힌 계산식 두 사람이 지우개로 지우고 동시에 숫자를 적는 상황과 같습니다. 둘 다 자신이 쓴 숫자가 최종값이라고 믿지만, 나중에 쓴 사람의 글씨 위로 덮어씌워져 먼저 쓴 사람의 노력은 사라집니다.

> **📢 섹션 요약 비유**: 갱신 손실은 **'마라톤 경기에서 두 명의 심판이 각자 다른 타이머로 시간을 재다가, 마지막에 멈춘 타이머 하나만 기록부에 넣어버려 먼저 기록된 시간을 사라지게 만드는 행정 착오'**와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. 발생 메커니즘 심층 분석**
갱신 손실은 주로 트랜잭션이 **Read-Modify-Write(읽기-수정-쓰기)** 과정을 수행할 때 발생합니다. 문제의 핵심은 트랜잭션이 데이터를 읽은 시점(t_read)과 쓰는 시점(t_write) 사이의 간격(Gap)입니다. 이 간격 사이에 다른 트랜잭션이 이미 커밋해버린다면, 이후에 커밋되는 트랜잭션은 이전 변경 사항을 '잃어버리게' 됩니다.

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 (Internal Logic) | 관련 프로토콜/명령어 | 비유 |
|:---|:---|:---|:---|:---|
| **User Application** | 트랜잭션发起 | 데이터를 조회하고 비즈니스 로직을 수행한 뒤 SQL을 생성 | SQL (`SELECT`, `UPDATE`) | 손님 |
| **Transaction Manager (TM)** | 트랜잭션 관리 | 트랜잭션의 시작(`BEGIN`)과 종료(`COMMIT`)을 관리하고 경계를 설정 | 2단계 커밋(2PC) | 관리자 |
| **Lock Manager (LM)** | 동시성 제어 | 자원(Data Item)에 대한 락(X-Lock, S-Lock)을 부여 및 해제 | 락 요청/해제 Protocol | 열쇠 관리실 |
| **Data Buffer (Cache)** | 임시 저장 | 디스크 I/O를 줄이기 위해 데이터를 메모리에 일시 저장 | LRU Algorithm | 작업 책상 |
| **Recovery Manager (RM)** | 로그 관리 | 변경 사항을 Redo/Undo 로그에 기록하여 회복 가능하게 함 | WAL (Write-Ahead Logging) | 블랙박스 |

**2. 갱신 손실 발생 라이프사이클 (ASCII Diagram)**

아래 다이어그램은 표준적인 **`READ COMMITTED`** 격리 수준이나 락이 없는 상황에서 발생하는 전형적인 Lost Update 시나리오입니다.

```text
[갱신 손실 (Lost Update) 상세 타임라인 & 상태 변이]

  Time│ Transaction T1 (A: +50)      │ Transaction T2 (A: +30)      │ DB Value (A)
  ─────┼────────────────────────────┼────────────────────────────┼─────────────
   t1  │ S-Lock 획득 (Read 100)     │                              │ 100
   t2  │ [Local Calc: 100+50 = 150] │ S-Lock 획득 (Read 100)       │ 100
   t3  │ X-Lock 획득 요청            │ [Local Calc: 100+30 = 130]  │ 100
   t4  │ X-Lock 획득 성공            │ X-Lock 대기 (Locked by T1)   │ 100
   t5  │ UPDATE Value = 150          │ [Wait...]                   │ 150 (T1 Write)
   t6  │ COMMIT (Lock Release)      │ X-Lock 획득 성공             │ 150
   t7  │                            ⚠️ UPDATE Value = 130         │ 130 (Overwrite!)
   t8  │                            │ COMMIT                       │ 130
  ─────┴────────────────────────────┴────────────────────────────┴─────────────
  
  💥 Critical Failure:
  t7 시점에 T2는 T1이 변경한 150을 읽는 것이 아니라,
  자신이 t2에 읽은 100을 기반으로 130을 덮어썼습니다.
  결과적으로 T1의 업데이트(+50)는 소멸되었습니다.
```

**3. 해설: 왜 이런 일이 발생하는가?**
위 다이어그램에서 T2는 t4 시점에 T1의 락이 풀릴 때까지 대기하지만, 문제는 **이미 t2 시점에 값을 읽어버린 상태**라는 점입니다. T1이 커밋하고 락을 풀자마자 T2는 자신이 예전에 읽었던 값(100)에 연산을 수행한 결과(130)로 데이터를 덮어씁니다. 이는 DBMS가 기본적으로 **'마지막 커밋이 승리한다(Last Commit Wins)'**는 정책을 따르기 때문이며, T2의 입장에서는 자신이 수정하는 시점에 기존 값(150)을 다시 읽어서 더하는(200) 로직이 수행되지 않았기 때문입니다.

> **📢 섹션 요약 비유**: 이 과정은 **'두 명의 화가가 같은 캔버스를 그리는데, 뒤에 온 화가가 앞 사람이 그린 그림을 보지 않고 자신의 밑그림 위에서 덧칠을 해버려 그림을 망가뜨리는 것'**과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Analysis)

**1. 동시성 제어 전략 비교 (Locking vs. Non-Locking)**
갱신 손실을 방지하기 위해 데이터베이스 설계자들은 다양한 전략을 사용합니다. 여기서는 크게 **비관적 락(Pessimistic Locking)**과 **낙관적 락(Optimistic Locking)**, 그리고 **원자적 연산(Atomic Update)**을 비교합니다.

| 비교 항목 | 비관적 락 (Pessimistic Lock) | 낙관적 락 (Optimistic Lock) | 원자적 연산 (Atomic Update) |
|:---|:---|:---|:---|
| **기본 철학** | 충돌이 발생할 것이라고 가정하고 미리 잠금 | 충돌이 드물 것이라고 가정하고 버전으로 검증 | 애초에 Read-Modify 단계를 없앰 |
| **핵심 기술** | `SELECT ... FOR UPDATE` (X-Lock) | `UPDATE ... WHERE ver = N` | `UPDATE SET col = col + 1` |
| **성능 (Low Contention)** | 낮음 (락 획득/해제 오버헤드) | 높음 (별도의 락 관리 없음) | 매우 높음 (DB 엔진 레벨 처리) |
| **성능 (High Contention)** | 중간 (순차 처리 보장) | 매우 낮음 (재시도가 빈번하여 실패율 증가) | 높음 (Queue 내에서 직렬화 처리) |
| **구현 난이도** | 쉬움 (DB 자원 의존) | 보통 (앱 레벨 재시도 로직 필요) | 쉬움 (쿼리 작성법) |
| **Deadlock 위험** | 있음 (순환 대기 발생 가능) | 없음 | 없음 |

**2. OS/컴퓨터 구조와의 융합 관점**
갱신 손실은 데이터베이스뿐만 아니라 **운영체제(OS)의 파일 시스템**이나 **공유 메모리(Shared Memory)** 영역에서도 발생합니다. 예를 들어, 두 프로세스가 **세마포어(Semaphore)** 없이 공유 메모리의 카운터를 증가시키려 할 때 `MOV` 명령어와 `ADD` 명령어 사이에 컨텍스트 스위칭(Context Switching)이 발생하면, 메모리 상의 값은 한 번만 증가하게 됩니다(경쟁 상태, Race Condition). 이를 해결하기 위해 하드웨어 레벨에서는 `Test-and-Set`이나 `Compare-and-Swap (CAS)` 같은 **원자적 명령어**를 제공합니다. 이는 DBMS의 `SELECT FOR UPDATE`와 동일한 맥락의 하드웨어적 락킹 기술입니다.

> **📢 섹션 요약 비유**: 갱신 손실 방지 전략은 **'회의실 사용'**과 같습니다. 비관적 락은 **'문을 걸어 잠그고 사용하는 것'**이고, 낙관적 락은 **'들어가기 전에 사용 예약표를 확인하고 내 순서가 아니면 나오는 것'**이며, 원자적 연산은 **'미리 신청서를 내면 사무실 직원이 알아서 처리해주는 위탁 시스템'**과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. 실무 시나리오별 의사결정 트리**
기술사(PE) 시나리오에서 갱신 손실 방지 기술을 선택할 때는 **충돌 빈도(Conflict Probability)**와 **잠재적 손실 규모(Impact)**를 기준으로 판단해야 합니다.

```text
[의사결정 매트릭스: 어떤 락을 사용할 것인가?]

   Conflict Rate  Low  ────────────────────────  High
                      │
                      │
  Impact  High    ─────┼─────┐  [Case A: 재고/금융]
  (Loss)                 │     │  -> 전략: Pessimistic Lock
                          │     │     (SELECT FOR UPDATE)
                          │     │
                      ─────┼─────┘
  Impact  Low          │     │  [Case B: 조회수/좋아요]
  (Non-Critical)       │     │  -> 전략: Atomic OR Pessimistic
                          │        (SET count = count + 1)
                          │
                ┌─────────┴─────────┐
                │  [Case C: 협업 편집] │
                │  -> 전략: Optimistic │
                │     (Version Column)│
                └────────────────────┘
```

**2. 안티패턴 및 주의사항 (Anti-Patterns)**
- **안티패턴 1: 애플리케이션 레벨의 `if` 검증**
    ```java
    // 잘못된 예
    int stock = getStock(); // 100
    if (stock > orderQty) { // 100 > 10 (True)
        updateStock(stock - orderQty); // 다른 트랜잭션이 100 -> 90로 만들었는데? 갱신 손실 발생
    }
    ```
    해결책: 쿼리 레벨에서 검증을 수행하거나(`WHERE stock >= ?`), 트랜잭션 격리 수준을 높여야 합니다.

- **안티패턴 2: 트랜잭션 격리 수준의 오남용**
    `SERIALIZABLE` 격리 수준을 사용하면 갱신 손실은 막을 수 있지만, 성능 저하가 심각하여 실무에서 사용하기 어렵습니다. 목적에 맞는 락 전략을 사용하는 것이 중요합니다.

**3. 도입 체크리스트**
- [ ] **정합성 요구도**: 데이터 정합성이 틀어졌을 때 비즈니스적 손해가 규정상 불허되는가? (예: 재고, 잔고)
- [ ] **트래픽 패턴**: 동시 수정 시도가 몰리는 이벤트 시점에 DB CPU가 100%에 달해 Deadlock이 빈번한가?
- [ ] **잠금 대기 시간**: 비관적 락 사용 시 대기 시간이 사용자 경험(UX)에 악영향을 주는가?
- [ ] **재시도 로직**: 낙관적 락 사용 시 충돌 발생 시 유저에게 알리고 재시도할 것인가?

> **📢 섹션 요약 비유**: 실무 대응은 **'교통 체증 관리'**와 같습니다. 사소한 충돌은 원활하게 흘러가게 하다가(낙관적 락), 정말 위험한 구
