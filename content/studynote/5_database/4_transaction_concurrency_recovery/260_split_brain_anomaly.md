# 260. 스플릿 브레인 (Split Brain) - 분산 시스템의 뇌 분리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스플릿 브레인(Split Brain)은 네트워크 파티셔닝(Network Partitioning)으로 인해 분산 시스템의 노드들이 서로의 존재를 인지하지 못해, **두 개 이상의 노드가 동시에 '리더(Master)'임을 선언하며 독립적으로 운영되는 데이터 일관성 파괴 상태**를 의미한다.
> 2. **가치**: 이 현상은 데이터 양방향 충돌(Write Conflict)을 유발하여 **복구 불가능한 데이터 훼손(Data Corruption)**을 초래하므로, 쿼럼(Quorum) 기반의 투표 메커니즘과 펜싱(Fencing) 기술을 통해 반드시 방지해야 한다.
> 3. **융합**: 분산 시스템의 **CAP 정리**(Consistency, Availability, Partition Tolerance)에서 P(분단 내성)가 발생할 때 C(일관성)와 A(가용성) 중 무엇을 선택할 것인가를 결정하는 핵심적인 아키텍처 문제이다.

+++

### Ⅰ. 개요 (Context & Background)

**개념**
스플릿 브레인(Split Brain)이란 분산 컴퓨팅 환경, 특히 **HA (High Availability)** 클러스터에서 네트워크 장애나 통신 두절로 인해 노드 간의 연결이 끊어졌을 때, 격리된 각 노드가 자신이 유일한 생존자라고 판단하여 독립적으로 마스터(Master) 또는 리더(Leader) 역할을 수행하는 현상을 말한다. 이는 마치 사람의 뇌가 좌우로 분리된 상태에서 서로 다른 신호를 보내는 것과 유사하여 붙여진 이름이다.

**💡 비유**
한 회사의 사장이 비행기 타는 동안 통신두절이 되자, 본사 점장과 지사 점장이 각각 "사장이 실종되었으니 내가 사장이다"라고 선언하고 서로 다른 결재를 내리는 상황과 같습니다. 회사(시스템)는 두 명의 사장을 섬기게 되어 재무(데이터)가 엉망이 됩니다.

**등장 배경**
1.  **기존 한계**: 단일 서버(Single Point of Failure, SPOF)를 방지하기 위해 이중화(Active-Standby) 구성을 도입했으나, 두 노드를 연결하는 '감시 네트워크(Heartbeat Link)'가 단일 경로로 구성될 경우 물리적 선로 끊김에 취약했다.
2.  **혁신적 패러다임**: 단순한 이중화를 넘어, 네트워크 분단 상황에서도 데이터 일관성을 지키기 위해 '자동 장애 조치(Failover)' 시스템에 **지능적인 투표(Quorum) 및 격리(Fencing)** 메커니즘을 통합하는 설계 패턴이 등장했다.
3.  **현재의 비즈니스 요구**: 클라우드 환경과 대규모 분산 DB에서, 가용성을 유지하면서도 데이터가 깨지는 것을 절대 허용할 수 없는 금융/결제 시스템의 **RPO (Recovery Point Objective, 복구 시점 목표)**를 0으로 유지해야 하는 필수 요구사항이 되었다.

> **📢 섹션 요약 비유**: 스플릿 브레인의 발생 배경은 **"통신 두절로 인해 두 명의 함장이 각자의 함교에서 '내가 선장이다'라고 외치며 배를 서로 다른 방향으로 몰고 가는 해난 사고"**와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 (표)**

| 요소명 (Component) | 역할 (Role) | 내부 동작 (Mechanism) | 관련 프로토콜/기술 |
|:---|:---|:---|:---|
| **Heartbeat Link** | 생존 확인 | 1초(기본) 간격으로 UDP/TCP 패킷을 교환하여 상대방의 생존 여부를 감시 | Pacemaker, Corosync |
| **Failover Logic** | 자동 승격 | Heartbeat 타임아웃(예: 5회 실패) 시 즉시 Active 상태로 전환하는 알고리즘 수행 | VIP (Virtual IP) 재할당 |
| **Split Brain** | 치명적 장애 | 네트워크 파티셔닝 발생 시, 양쪽 노드가 모두 Active가 되어 데이터 쓰기 경합을 일으킴 | 원치 않는 이중화 |
| **Quorum (정족수)** | 분단 방지 | 전체 노드 수의 과반수(과반수 투표, Majority Vote) 이상의 응답이 있어야 리더 자격을 부여 | Raft, Paxos, ZAB |
| **Fencing (펜싱)** | 물리적 격리 | 이전 마스터가 데이터베이스나 디스크에 접근하지 못하도록 전원 차단(STONITH)하거나 SCSI 예약을 걸어 차단 | IPMI, SCSI Fence |

**ASCII 구조 다이어그램 + 해설**
아래 다이어그램은 네트워크 분단(WAN Link 끊김) 발생 시, **Shared Storage (공유 저장소)** 환경에서 데이터가 깨지는 과정을 도식화한 것이다.

```text
[ 스플릿 브레인 발생 시 데이터 훼손 메커니즘 ]

      ┌──────────────────────────┐
      │       Public Network      │
      │  (Client User Requests)   │
      └─────────────┬─────────────┘
            ┌───────┴───────┐
            ▼               ▼
      [ Node A ]       [ Node B ]  ▲ 보기: 둘 다 "나 살았음!" 선언
      (Active)         (Active)    │
            │               │
            │ Heartbeat     │ Heartbeat
      ┌─────┴─────┐   ┌─────┴─────┐
      │  Switch   │ X │  Switch   │  ◀─── 문제 발생: 링크 단절 (Partition)
      └───────────┘   └───────────┘
            │               │
            └───────┬───────┘
                    ▼
          [ Shared SAN Storage ]
          (혹은 분산 DB 데이터 파일)

       ⚠️ Write Conflict 발생 포인트 ⚠️
       ========================================
       Node A: File의 Offset 0에 "ABC" 기록  ┐
                                     │  충돌!
       Node B: File의 Offset 0에 "XYZ" 기록  ┘
       ========================================
       Result: 데이터가 뒤집히거나 블록 단위로 깨짐.
```

**[다이어그램 해설]**
1.  **정상 상태**: Node A가 Active이며, VIP를 소유하고 사용자 요청을 처리. Node B는 Standby 상태로 Heartbeat만 수신.
2.  **단계 1 (네트워크 단절)**: 중앙의 Heartbeat Link가 물리적 장애로 끊어짐. Node A는 "B가 죽었다"고 착각하지만, Node B는 여전히 살아있고 "A가 응답이 없으니 내가 리더다"라고 판단.
3.  **단계 2 (양방향 승격)**: Node B가 자신을 Active로 승격시키고, 자신의 IP를 VIP로 설정하거나 공유 스토리지에 대한 쓰기 권한을 획득.
4.  **단계 3 (데이터 파괴)**: 이때 클라이언트 요청이 분산되어 A와 B에 동시에 도달하게 되면, 두 노드는 동일한 데이터베이스 파일에 서로 다른 트랜잭션(Log)을 기록하게 되어 메타데이터가 꼬이고 복구 불가능한 상태가 됨.

**심층 동작 원리 (과정)**
1.  **감지 (Detection)**: `missed-heartbeat-count * heartbeat-interval` 시간이 초과되면 클러스터 멤버십(Cluster Membership) 모듈이 상태를 'UNKNOWN'으로 변경.
2.  **후보 지명 (Candidate)**: 고립된 노드의 코디네이터(Coordinator)가 리더 선출을 위한 투표 요청(Request Vote)을 브로드캐스팅.
3.  **오인 (Misinterpretation)**:
    *   파티션 A (1대): "나는 1표 받았으니 100% 찬성이네, 리더다!"
    *   파티션 B (1대): "나도 1표 받았으니 100% 찬성이네, 리더다!"
4.  **데이터 분기 (Divergence)**: 두 리더가 각자의 WAL(Write-Ahead Log)에 커밋을 기록하면서 논리적인 데이터 세계가 갈라짐.
5.  **충돌 (Collision)**: 네트워크 복구 시 두 노드가 다시 연결되면서, 서로 모순된 데이터를 동기화하려 시도하며 데이터베이스가 크래시(Crash) 발생.

**핵심 알고리즘: 쿼럼 (Quorum) 계산식**
스플릿 브레인을 방지하기 위해 시스템은 다음 공식을 사용한다.
$$ Q = \lfloor \frac{N}{2} \rfloor + 1 $$
여기서 $N$은 전체 노드 수, $Q$는 과반수(Quorum)이다.
노드가 연결된 파티션의 노드 수가 $Q$ 미만이라면, 그 파티션은 자체적으로 서비스를 중지(Stop)해야 한다.

> **📢 섹션 요약 비유**: 스플릿 브레인의 아키텍처 원리는 **"두 대의 비행기 조종석이 동시에 작동하여, 하나는 상승 하나는 하강 조작을 함으로써 비행기를 공중 분해시키는 시스템 결함"**과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**심층 기술 비교 (정량적 분석표)**

| 비교 항목 | 정상 HA 장애 조치 (Normal Failover) | 스플릿 브레인 (Split Brain) |
|:---|:---|:---|
| **마스터 수** | 오직 1개만 존재 | 2개 이상 존재 (다중 마스터) |
| **일관성** | 강한 일관성 보장 | 데이터 비일관 (Data Inconsistency) |
| **가용성** | 순간적 중단 후 복구 | 겉보기엔 정상 작동함 (가용성 유지) |
| **데이터 무결성** | 보안 (안전함) | **파괴됨 (치명적)** |
| **복구 난이도** | 자동 복구 용이 | 수동 개입 필요 시간이 매우 김 |
| **주요 원인** | 마스터 정지 (Power Off) | **네트워크 파티셔닝 (Link Cut)** |

**과목 융합 관점**
1.  **OS (운영체제)**: 커널 레벨의 파일 시스템(OCFS2, GFS2)에서는 **DLM (Distributed Lock Manager)**을 사용하여 스플릿 브레인 시 서로 다른 노드가 동일한 파일 블록에 쓰지 못하도록 락(Lock)을 관리한다.
2.  **네트워크**: 네트워크 토폴로지 설계 시 **Redundant Network (이중화)**를 필수로 구축한다. 단일 경로가 끊기더라도 라우팅 프로토콜(OSPF/BGP)이 우회 경로를 찾아 Heartbeat가 끊기지 않게 하는 것이 스플릿 브레인 예방의 1차 수단이다.

> **📢 섹션 요약 비유**: 일반적인 장애 조치는 **"부재장 때 사장이 권한을 대리인에게 위임하고 휴가를 떠나는 것"**이라면, 스플릿 브레인은 **"사장이 없는 사이 두 명의 사장이 동시에 등장하여 상반된 명령을 내리는 기업의 최악의 컴플리케이션 상황"**과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**실무 시나리오**
1.  **상황**: 금융권 핵심 결제 시스템의 Active-Active 클러스터에서 데이터 센터 간 광케이블이 절단됨.
2.  **문제**: 두 데이터 센터가 각각 자신이 결제 마스터라고 판단하여 동일한 계좌의 잔액을 동시에 차감(이중 출금).
3.  **의사결정**: **쿼럼(Quorum) 장치(Witness Server)를 도입**하여, 과반수 투표를 얻지 못한(광케이블 끊어진) 쪽 데이터 센터는 즉시 **자폐(Self-Fencing)**하여 서비스를 중단시킴으로써 데이터 일관성을 우선시함.

**도입 체크리스트**
- [ ] **설계**: 노드 수는 반드시 홀수(1, 3, 5...)로 구성하였는가?
- [ ] **네트워크**: Heartbeat 전용 망을 일반 트래픽과 물리적/논리적으로 분리(Isolation)하였는가?
- [ ] **장애 복구**: STONITH(Shoot The Other Node In The Head)가 설정되어 있어 자동 전원 차단이 가능한가?
- [ ] **테스트**: 네트워크 카블을 물리적으로 뽑는 'Chaos Engineering' 테스트를 수행하였는가?

**안티패턴 (Anti-Pattern)**
- **❌ 잘못된 사용**: "쿼럼 설정이 복잡하니까, 노드 2대(A, B)만으로 Active-Active 구성을 하고 IP만 다르게 두자."
    - **결과**: 네트워크가 끊기는 순간 100% 스플릿 브레인이 발생하여 데이터가 즉시 망가진다. 반드시 **과반수(Majority) 로직**이 필요하다.

> **📢 섹션 요약 비유**: 실무 대응은 **"원자력 발전소의 안전 장치"**와 같습니다. 문제가 발생했을 때 무조건 발전을 계속(가용성) 하려 하기보다, 시스템이 멈추더라도 폭발(데이터 파괴)을 막기 위해 먼저 스스로 멈추는 **안전핀(Fencing)이 우선**되어야 합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**정량/정성 기대효과 (표)**

| 구분 | 도입 전 (스플릿 브레인 허용) | 도입 후 (Quorum & Fencing
