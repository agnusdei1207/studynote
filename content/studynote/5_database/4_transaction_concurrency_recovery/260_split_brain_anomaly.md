# 260. 스플릿 브레인 (Split Brain) - 분산 시스템의 뇌 분리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스플릿 브레인(Split Brain)은 네트워크 단절로 인해 분산 클러스터의 노드들이 서로의 상태를 확인하지 못하고, **두 개 이상의 노드가 동시에 자신이 '리더(Master)'라고 오판하여 독립적으로 작동하는 재앙적 상태**를 말한다.
> 2. **가치**: 데이터 양방향 쓰기로 인한 **복구 불가능한 데이터 파괴**를 유발하며, 이를 방지하기 위해 쿼럼(Quorum) 기반의 과반수 투표와 펜싱(Fencing) 기술이 동원된다.
> 3. **융합**: 고가용성(HA) 아키텍처의 설계 시 가장 먼저 고려해야 할 '안전핀'이며, 분산 합의 알고리즘(Raft 등)의 존재 이유가 된다.

+++

### Ⅰ. 스플릿 브레인의 발생 원인

- **상황**: 하트비트(Heartbeat) 네트워크 전용 선로가 끊어지거나 스위치가 장애를 일으켰을 때 발생합니다.
- **매커니즘**:
    1. 마스터(A)와 대기조(B)가 연결 단절.
    2. B는 "A가 죽었네? 내가 마스터 해야지!"라며 승격.
    3. A는 멀쩡히 살아있으므로 여전히 마스터로 동작.
    4. **결과**: 하나의 시스템에 두 명의 왕이 존재.

+++

### Ⅱ. 스플릿 브레인 시각화 (ASCII Diagram)

```text
[스플릿 브레인으로 인한 데이터 분기]

      (User Requests)
       /          \
  [ Node A ]  X─── [ Node B ]
  (Active)    단절   (Active)
      |              |
  [ DB File A ]  [ DB File B ] ◀── 💥 데이터가 서로 다르게 저장됨!
```

+++

### Ⅲ. 해결책: 쿼럼과 펜싱

1. **Quorum (정족수)**: 노드를 항상 홀수(3, 5)로 구성하여, 과반수(2, 3)의 표를 얻지 못한 그룹은 스스로 서비스를 중단하게 만듭니다.
2. **Fencing (격리)**: 새로운 리더가 선출되면, 구형 리더의 전원을 물리적으로 차단(STONITH)하거나 스토리지 접근을 막아버립니다.

- **📢 섹션 요약 비유**: 스플릿 브레인은 **'두 명의 기사가 동시에 왕관을 쓰고 영토를 다스리는 상황'**과 같습니다. 한 나라이지만 법이 두 개가 되어 백성(데이터)들이 큰 혼란에 빠집니다. 이를 막기 위해 반드시 원로원(쿼럼)의 과반수 찬성이 있어야 왕이 될 수 있게 하고, 가짜 왕은 즉시 유배(펜싱) 보내야 합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[HA (High Availability)]**: 고가용성을 지키려다 만나는 가장 무서운 적.
- **[STONITH]**: "Shoot The Other Node In The Head" - 강제 전원 차단 기법.
- **[Raft/Paxos]**: 스플릿 브레인을 논리적으로 예방하는 알고리즘.

📢 **마무리 요약**: **Split Brain**은 분산 시스템의 사형 선고입니다. 완벽한 합의 구조와 물리적 격리 체계를 갖추는 것만이 시스템의 생명을 연장하는 길입니다.
