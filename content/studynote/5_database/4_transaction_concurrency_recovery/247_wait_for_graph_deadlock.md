# 247. 교착 상태 탐지 대기 그래프 (Wait-for Graph)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 대기 그래프(Wait-for Graph)는 트랜잭션 간의 자원 대기 관계를 노드(트랜잭션)와 간선(대기 방향)으로 표현한 **방향 그래프(Directed Graph)**이다.
> 2. **가치**: 그래프 내에서 **사이클(Cycle)의 존재 여부를 수학적으로 판별**함으로써 교착 상태의 발생을 확실하게 진단하고, 어떤 트랜잭션을 중단할지 결정하는 근거를 제공한다.
> 3. **융합**: 그래프 알고리즘의 사이클 탐지 기술이 데이터베이스 엔진의 데드락 감시 스레드와 결합되어 동시성 제어의 안정성을 책임진다.

+++

### Ⅰ. 대기 그래프(Wait-for Graph)의 구성

- **노드 (Node)**: 현재 실행 중인 각 트랜잭션($T_i, T_j$)을 의미합니다.
- **간선 (Edge)**: $T_i \to T_j$ 는 "트랜잭션 $T_i$가 트랜잭션 $T_j$가 점유한 자원을 기다리고 있음"을 의미합니다.
- **생성 규칙**: 트랜잭션이 자원 할당을 요청했으나 거절당해 대기 상태로 들어갈 때 간선을 추가하고, 자원을 할당받으면 간선을 제거합니다.

+++

### Ⅱ. 사이클 탐지를 통한 데드락 판별 (ASCII Diagram)

```text
[Wait-for Graph의 사이클 구조]

     ┌───────── (Wait Resource X) ─────────┐
     ▼                                     │
  ┌────┐                                ┌────┐
  │ T1 │ ────(Wait Resource Y)──────▶ │ T2 │
  └────┘                                └────┘
     ▲                                     │
     └───────── (Wait Resource Z) ─────────┘
     
  * 판별: T1 -> T2 -> T1 방향의 닫힌 루프(Cycle) 형성.
  * 결론: 데드락 발생 확정! 누군가 죽어야만 풀림.
```

+++

### Ⅲ. 탐지 후의 복구 전략

1. **희생자 선정 (Victim Selection)**: 비용(Undo량, 소요 시간 등)이 가장 적은 트랜잭션을 골라 롤백시킵니다.
2. **락 해제**: 희생자가 가진 모든 락을 즉시 해제하여 대기 중인 다른 트랜잭션들이 전진할 수 있게 합니다.
3. **재실행**: 희생된 트랜잭션은 잠시 후 자동으로 다시 시작되도록 관리합니다.

- **📢 섹션 요약 비유**: 대기 그래프는 **'꼬리에 꼬리를 무는 미행'**과 같습니다. A는 B를 쫓고, B는 C를 쫓는데, C가 다시 A를 쫓고 있다면 이들은 영원히 제자리를 돌게 됩니다. 이 순환 고리를 끊기 위해 한 명을 대열에서 이탈(Rollback)시키는 것이 유일한 해결책입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[사이클 탐지]**: DFS(깊이 우선 탐색) 등을 활용한 알고리즘적 접근.
- **[진공 프로세스]**: 사이클을 끊어 자원을 회수하는 과정.
- **[분산 대기 그래프]**: 여러 DB 서버에 걸친 거대 그래프 관리의 난제.

📢 **마무리 요약**: **Wait-for Graph**는 데드락을 시각화하고 해결하는 가장 명쾌한 도구입니다. 복잡한 시스템일수록 이 그래프를 효율적으로 관리하는 능력이 중요합니다.
