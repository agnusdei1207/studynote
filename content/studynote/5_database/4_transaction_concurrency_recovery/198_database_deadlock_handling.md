# 198. 데이터베이스 교착 상태 (Deadlock) - 발생 원리와 해결 알고리즘

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 교착 상태(Deadlock)는 두 개 이상의 트랜잭션이 서로가 점유하고 있는 락(Lock)을 해제하기만을 무한정 기다리며 시스템의 진행이 완전히 멈추는(Stalemate) 현상이다.
> 2. **메커니즘**: 이는 2단계 락킹(2PL) 프로토콜을 사용할 때 필연적으로 발생할 수밖에 없는 부작용이며, 주로 **상호 배제(Mutual Exclusion)**와 **환형 대기(Circular Wait)** 조건이 맞물려 일어난다.
> 3. **해결책**: 이를 해결하기 위해 타임스탬프 기반으로 미리 예방하는 **Wait-Die / Wound-Wait** 기법을 사용하거나, 대기 그래프(Wait-for Graph)를 그려 데드락을 탐지(Detection)한 후 희생자(Victim)를 롤백시키는 방식을 사용한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**데이터베이스 교착 상태 (Deadlock)**는 다중 프로그래밍 환경에서 복수의 트랜잭션들이 각자 자원(데이터 행, 페이지 등)에 락(Lock)을 건 상태에서, 상대방이 가진 락이 풀리기만을 서로 기다림으로써 누구도 다음 단계로 진행하지 못하는 무한 대기 상태를 말합니다.

### 💡 비유: 좁은 골목길의 자동차 교착
폭이 좁은 골목길 한가운데서 두 대의 자동차가 마주쳤습니다. 양쪽 모두 뒤로 후진(Rollback)하지 않고 "네가 먼저 비켜!"라며 경적만 울리며 기다린다면, 두 차는 영원히 그 골목길을 빠져나갈 수 없습니다. 누군가 한 명(Victim)이 양보(Rollback)해야만 상황이 풀립니다.

### 2. 등장 배경
- **2PL (Two-Phase Locking)의 맹점**: 트랜잭션의 직렬성(Serializability)을 보장하기 위해 도입된 2PL은 커밋 직전까지 락을 풀지 않고 모으기만(Growing phase) 하므로, 트랜잭션들이 락을 교차로 요구할 때 데드락 구조를 형성하기 쉽습니다.
- **상호 배제와 비선점**: 데이터 무결성을 위해 배타 락(X-Lock)이 걸린 데이터는 다른 트랜잭션이 강제로 빼앗을(선점, Preemption) 수 없기 때문에, 결국 무한 대기로 이어집니다.

- **📢 섹션 요약 비유**: 락(Lock)이 데이터 보호를 위한 '자물쇠'라면, 데드락(Deadlock)은 두 사람이 서로의 열쇠를 삼켜버려 둘 다 문을 열지 못하는 모순적 상황입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 데드락 발생 메커니즘 (환형 대기)

데드락은 주로 두 개 이상의 테이블/행(Row)을 서로 다른 순서로 업데이트할 때 발생합니다. 이를 **교차 업데이트(Cross Update)**라고도 합니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                       [데이터베이스 데드락 발생 타임라인]                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  시간(t)     트랜잭션 T1                         트랜잭션 T2                    │
│  ─────────────────────────────────────────────────────────────────────────  │
│   t1      UPDATE 계좌A (A에 X-Lock 획득)                                       │
│   t2                                      UPDATE 계좌B (B에 X-Lock 획득)       │
│   t3      UPDATE 계좌B 시도                                                   │
│           (T2의 B 락 해제 대기 ⏳)                                             │
│   t4                                      UPDATE 계좌A 시도                    │
│                                           (T1의 A 락 해제 대기 ⏳)             │
│                                                                             │
│  [상태] T1은 T2를 기다리고, T2는 T1을 기다리는 **환형 대기 (Circular Wait)** 형성  │
│         → 무한 대기 (Deadlock!)                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2. 데드락 예방 알고리즘 (Deadlock Prevention)

트랜잭션이 시작될 때 부여받는 **타임스탬프 (Timestamp, 나이)**를 활용하여, 대기(Wait)를 허용할지 아니면 한쪽을 죽일지(Rollback) 미리 결정함으로써 환형 대기를 원천적으로 차단합니다. **늙은(오래된) 트랜잭션에 우선권을 주는 것**이 핵심입니다.

#### ① Wait-Die (비선점 / 기다리거나 죽거나)
- **원칙**: 늙은 트랜잭션이 젊은 트랜잭션을 기다릴 수는 있지만, 젊은 트랜잭션은 늙은 트랜잭션을 기다리지 못하고 자살(Die/Rollback)합니다.
- T_old 가 T_young 의 자원을 요구할 때: **Wait (기다림)**
- T_young 이 T_old 의 자원을 요구할 때: **Die (젊은 쪽이 롤백 후 재시작)**

#### ② Wound-Wait (선점 / 상처입히거나 기다리거나)
- **원칙**: 늙은 트랜잭션은 기다리지 않고 젊은 트랜잭션을 찔러서(Wound/강제 롤백) 자원을 빼앗습니다. 반면, 젊은 트랜잭션은 늙은 트랜잭션을 기다립니다.
- T_old 가 T_young 의 자원을 요구할 때: **Wound (젊은 쪽 강제 롤백시키고 자원 탈취)**
- T_young 이 T_old 의 자원을 요구할 때: **Wait (기다림)**

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                 [Wait-Die 와 Wound-Wait 비교 (T1: 늙음, T2: 젊음)]              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황 1: 👴T1(늙음)이 👶T2(젊음)의 자원 요청]                                   │
│  - Wait-Die: T1은 너그러이 기다려줌 (Wait)                                    │
│  - Wound-Wait: T1이 T2를 찔러 죽이고 뺏음 (Wound: T2 롤백)                      │
│                                                                             │
│  [상황 2: 👶T2(젊음)가 👴T1(늙음)의 자원 요청]                                   │
│  - Wait-Die: T2가 분수를 알고 스스로 죽음 (Die: T2 롤백)                        │
│  - Wound-Wait: T2는 감히 못 뺏고 기다림 (Wait)                                │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 데드락 탐지 알고리즘 (Deadlock Detection)

현대의 대부분 RDBMS(Oracle, MySQL 등)는 예방보다 탐지(Detection) 방식을 씁니다. 데드락 예방은 롤백이 너무 잦아 오버헤드가 크기 때문입니다.
1. **대기 그래프 (Wait-for Graph) 유지**: 데이터베이스 엔진은 내부적으로 어떤 트랜잭션이 어떤 트랜잭션을 기다리고 있는지 방향성 그래프를 그립니다.
2. **사이클 탐지**: 주기적으로 그래프에 **사이클(Cycle, 순환 고리)**이 있는지 검사합니다.
3. **희생자(Victim) 선정 및 롤백**: 사이클이 발견되면, 롤백 비용이 가장 적은 트랜잭션(가장 최근에 시작했거나, 수정한 데이터가 가장 적은 트랜잭션)을 희생자(Victim)로 선정하여 강제 롤백(Rollback)시킵니다. 그러면 락이 풀려 꼬인 실타래가 풀립니다.

- **📢 섹션 요약 비유**: Wait-Die/Wound-Wait가 교차로 진입 전 '나이순'으로 우선순위를 정해 사고를 아예 막는 교통경찰이라면, 탐지 기법은 일단 차들을 엉키게 둔 뒤 하늘에서 드론(탐지기)으로 보고 가장 치우기 쉬운 차 한 대(Victim)를 크레인으로 빼내는 방식입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 데드락 예방 vs 탐지 아키텍처 비교

| 비교 항목 | 데드락 예방 (Prevention) | 데드락 탐지 (Detection) |
|:---|:---|:---|
| **설계 사상** | 비관적 (데드락은 잦을 것이고 치명적이다) | 낙관적 (데드락은 드물다, 발생하면 그때 처리하자) |
| **자원 활용률** | 낮음 (불필요한 선제 롤백으로 CPU/I/O 낭비 심함) | **높음** (실제 충돌 시에만 조치하므로 효율적) |
| **알고리즘** | Wait-Die, Wound-Wait 등 타임스탬프 활용 | Wait-for Graph 기반 사이클 탐색 (DFS 등) |
| **적용 시스템** | 실시간성이 극도로 중요한 군사/우주 시스템 | 일반적인 상용 RDBMS (Oracle, MySQL, SQL Server) |
| **오버헤드 발생 시점** | 매 트랜잭션 자원 요청 시마다 검사 로직 동작 | 백그라운드 스레드(Deadlock Detector)가 주기적 동작 |

### 2. 타 영역과의 연계 관점
- **운영체제(OS) 데드락**: OS에서도 프로세스 간 자원(프린터, 메모리 등) 할당 시 데드락이 발생하며 식사하는 철학자 문제(Dining Philosophers), 은행원 알고리즘, 자원 할당 그래프 등 DB 데드락과 수학적/논리적 기반을 완전히 공유합니다.
- **분산 시스템 데드락 (Global Deadlock)**: 클라우드 분산 DB에서는 노드 A와 노드 B에 걸쳐 데드락이 발생할 수 있습니다. 단일 노드의 Wait-for Graph만으로는 탐지가 불가능하므로, 글로벌 코디네이터(Coordinator)가 각 노드의 로컬 그래프를 취합하여 글로벌 사이클을 탐지해야 하는 복잡한 문제가 생깁니다.

- **📢 섹션 요약 비유**: 빈대가 무섭다고 초가삼간을 다 태우는 것(예방)보다, 빈대가 나타났을 때 약을 쳐서 잡는 것(탐지)이 현대 데이터베이스의 기본 철학입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 데드락 튜닝 시나리오
- **문제 상황**: e커머스 시스템에서 세일 기간 중 동시에 여러 상품을 결제하는 주문 로직에서 `ORA-00060: deadlock detected while waiting for resource` 또는 MySQL의 `Deadlock found when trying to get lock` 에러가 초당 수십 건씩 폭증.
- **원인 분석**: 애플리케이션 코드가 장바구니에 담긴 상품 재고를 차감(UPDATE)할 때, 사용자 A는 [상품1, 상품2] 순서로 차감하고, 사용자 B는 [상품2, 상품1] 순서로 차감하게 되어 교차 대기(Cross Wait)가 발생함.
- **해결책 (설계 개선)**:
  1. **정렬 후 락 획득 (Global Ordering)**: 애플리케이션에서 다수의 행을 업데이트할 때, **반드시 일관된 순서(예: 상품 ID 오름차순)로 정렬한 뒤 업데이트**하도록 코드를 수정합니다. 방향이 일치하면 환형 대기(사이클)가 수학적으로 절대 발생할 수 없습니다.
  2. **트랜잭션 분할 및 트랜잭션 시간 최소화**: 트랜잭션 시작과 커밋 사이의 시간(외부 API 호출 등)을 최소화하여 락 점유 시간을 줄입니다.

### 2. 도입 및 운영 체크리스트
- **재시도 (Retry) 로직 필수**: RDBMS는 데드락 탐지 시 Victim을 롤백하고 즉시 에러를 앱으로 던집니다. 애플리케이션은 이 에러 코드(Deadlock)를 캐치하면, 수 초 후 **자동 재시도(Retry)**하도록 스프링(Spring) `@Retryable` 등의 AOP 로직을 반드시 구현해야 합니다.
- **인덱스 확인**: 테이블 락을 피하고 레코드(Row) 락을 유도하기 위해, UPDATE/DELETE 조건(WHERE 절)에 적절한 인덱스가 있는지 확인합니다. 인덱스가 없으면 풀 스캔 과정에서 락 범위가 과도하게 넓어져 데드락 확률이 급증합니다.

### 3. 안티패턴
- **비즈니스 로직 중 사용자 응답 대기**: 화면에서 트랜잭션을 열고 행에 락을 건 채로 사용자의 '확인' 버튼 클릭을 기다리는 행위 (Interactive Transaction). 이는 락을 무한정 붙들고 있어 데드락과 성능 저하의 주범이 됩니다.

- **📢 섹션 요약 비유**: 병목을 푸는 가장 확실한 방법은 경찰을 부르는 것(DB 롤백)이 아니라, 모든 차량이 처음부터 정해진 차선과 순서대로만(ID 정렬 업데이트) 이동하도록 룰을 정하는 것입니다.

### 2. 결론
데드락은 시스템이 비정상적으로 동작해서 발생하는 버그라기보다는, 데이터의 일관성(Consistency)을 엄격하게 지키려는 **락킹 메커니즘의 필연적인 기회비용**입니다. DBMS 레벨에서 제공하는 탐지 및 희생자 처리 기능은 사후약방문일 뿐이며, 근본적인 해결책은 데이터 접근 순서 단일화, 인덱스 최적화, 그리고 낙관적 락(MVCC) 활용 등 **애플리케이션 아키텍처 레벨의 정교한 트랜잭션 설계**에 있습니다.

- **📢 섹션 요약 비유**: 데드락은 질병이라기보다 과도한 안전벨트가 꼬여서 숨을 못 쉬는 현상에 가깝습니다. 이를 푸는 열쇠는 안전벨트를 푸는 것이 아니라, 꼬이지 않게 매는 순서(정렬)를 지키는 것입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **병행 제어 및 락 (Concurrency Control)**: 데드락을 유발하는 원인이 되는 2PL (2단계 락킹) 메커니즘.
- **MVCC (다중 버전 동시성 제어)**: 읽기와 쓰기 사이의 데드락은 완전히 방지해주지만, 쓰기와 쓰기 간의 데드락은 여전히 발생할 수 있음.
- **운영체제 데드락 (OS Deadlock)**: 자원 할당 그래프와 예방/회피 알고리즘의 컴퓨팅 이론적 기반.

## 👶 어린이를 위한 3줄 비유 설명
1. **데드락**은 철수와 영희가 서로 "네가 가진 장난감 주면 내 거 줄게"라고 버티면서, 둘 다 아무것도 못하고 멈춰버린 바보 같은 상황이에요.
2. 데이터베이스는 이런 상황을 막기 위해 늙은(먼저 온) 사람에게 양보하게 하거나(Wait-Die), 아예 경찰(DB)이 와서 한 명의 장난감을 뺏어버린 뒤(탐지 및 롤백) 나중에 다시 주기도 해요.
3. 이 상황을 피하는 제일 좋은 방법은 처음부터 장난감을 고르는 **'순서(오름차순 등)'를 똑같이 맞추는** 거랍니다!
