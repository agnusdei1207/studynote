# 194. 병행 제어와 락 (Locking) - 동시성 관리와 데이터 보호

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다중 사용자 환경에서 여러 트랜잭션이 동시에 동일한 데이터에 접근할 때, 상호 간섭으로 인한 데이터 오염을 방지하고 일관성을 유지하기 위해 데이터의 사용권을 제어하는 **병행 처리 관리 기술**이다.
> 2. **메커니즘**: 데이터 접근 전 반드시 **락(Lock)**을 획득하게 하여 타 트랜잭션의 접근을 차단하거나 제한하며, 읽기 전용인 **공유 락(Shared)**과 수정 전용인 **배타 락(Exclusive)**을 조합하여 동시성과 정합성의 균형을 맞춘다.
> 3. **가치**: 갱신 손실(Lost Update)이나 현황 파악 오류를 원천 차단하며, 효율적인 락 설계는 시스템의 응답 속도와 사용자 수용량(Scalability)을 결정짓는 핵심 아키텍처 요소가 된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**병행 제어 (Concurrency Control)**는 데이터베이스라는 공용 자원을 여러 사람이 동시에 쓰면서도 '나 혼자 쓰고 있는 것'처럼 느끼게 해주는 질서 유지 기법입니다. 그 핵심 수단이 바로 데이터를 잠그는 **락(Lock)**입니다.

### 💡 비유: 화장실 사용과 전용 열쇠
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        병행 제어 및 락 (Lock) 비유                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 카페에 화장실(데이터)이 하나 있는데 손님(트랜잭션)이 많음.                │
│                                                                             │
│  1. 병행 제어가 없다면?                                                       │
│     - 한 명이 볼일 보는데 다른 사람이 문을 벌컥 열고 들어옴. (데이터 오염)         │
│                                                                             │
│  2. 락(Lock)을 사용한다면?                                                    │
│     - 화장실 들어가기 전 카운터에서 '열쇠(Lock)'를 받아감.                       │
│     - 내가 열쇠를 쥐고 있는 동안 다른 손님은 열쇠가 나올 때까지 대기함.            │
│     - 볼일이 끝나고 열쇠를 반납(Unlock)해야 다음 사람이 쓸 수 있음.               │
│                                                                             │
│  → **'한 번에 한 명씩만 안전하게'** 자원을 쓰도록 통제하는 것이 락입니다!          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 일관성 붕괴 방지**: 비행기 좌석 예약 시 두 명이 동시에 같은 자리를 예약하는 '이중 예약' 사고 등을 막아야 함.
- **처리량(Throughput) 극대화**: 무조건 한 줄로 세우기보다는, 읽기(Select)는 같이 하고 쓰기(Update)만 막는 식의 효율적인 공존 방법 필요.

- **📢 섹션 요약 비유**: 병행 제어는 데이터베이스라는 도로의 '교차로 신호등'과 같아서, 충돌(Collision)을 막고 흐름을 조절합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 락(Lock)의 주요 유형 및 호환성

| 락 유형 | 영문 명칭 | 목적 | 호환성 (남이 같이 잡을 수 있는가?) |
|:---:|:---|:---|:---|
| **공유 락** | **Shared (S)** | 읽기 전용. | **Yes (S-S 호환)**. 읽는 건 다 같이 가능. |
| **배타 락** | **Exclusive (X)** | 쓰기/수정용. | **No (X-Any 불허)**. 혼자만 독점해야 함. |

### 2. 병행 제어 미흡 시 발생하는 문제 도해 (ASCII)

락이 제대로 작동하지 않을 때 발생하는 '갱신 분실' 시나리오를 시각화한다.

```text
    [ Time ]    [ Transaction 1 ]       [ Data (A=100) ]       [ Transaction 2 ]
       │               │                       │                       │
       t1              │ ───(Read A: 100)──▶   │                       │
       t2              │                       │ ◀───(Read A: 100)───  │
       t3              │ ───(Update A+10)──▶   │ (A=110)               │
       t4              │                       │ ◀───(Update A+20)───  │ (A=120)
       t5              │ ───(COMMIT)           │                       │
       t6              │                       │ ◀───(COMMIT)          │
       │               │                       │                       │
    [ 결과 ] A는 130이 되어야 하는데 **120**이 됨! (T1의 +10 작업이 사라짐 = 갱신 분실)
```

### 3. 심층 동작 원리: 2단계 잠금 규약 (2-Phase Locking, 2PL)
- **확장 단계 (Growing)**: 트랜잭션이 필요한 락을 하나씩 계속 얻기만 하는 단계.
- **수축 단계 (Shrinking)**: 얻었던 락을 하나씩 해제하는 단계. (한 번 해제하면 다시 얻을 수 없음)
- **효과**: 트랜잭션의 직렬 가능성(Serializability)을 수학적으로 보장하는 표준 규약.

### 4. 핵심 기술: 데드락(Deadlock) 탐지
- 두 트랜잭션이 서로 상대방이 쥔 락을 기다리며 영원히 멈추는 현상. DBMS는 주기적으로 'Wait-for Graph'를 그려 순환 고리를 찾아내고 한 쪽을 강제 종료(Kill)시킴.

- **📢 섹션 요약 비유**: 2PL은 '쇼핑 카트에 물건을 다 담을 때까지는 절대 계산(해제)하지 않는 규칙'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 낙관적 병행 제어 vs 비관적 병행 제어

| 비교 항목 | 낙관적 (Optimistic) | 비관적 (Pessimistic) |
|:---:|:---|:---|
| **가정** | 충돌이 거의 없을 것이다. | **충돌이 잦을 것이다**. |
| **제어 방식** | 종료 시점에 버전 체크. | **시작부터 Lock 획득**. |
| **장점** | 오버헤드 적음, 동시성 높음. | 데이터 정합성 보장 확실. |
| **단점** | 롤백 발생 시 재작업 큼. | 대기 시간 길어짐, 성능 저하. |
| **비유** | **일단 먹고 나중에 계산** | **선불 결제 후 식사** |

### 2. 과목 융합 관점: 운영체제 및 알고리즘
- **운영체제(OS) 융합**: 병행 제어는 OS의 **상호 배제(Mutual Exclusion)**와 공유 자원 관리 기법(Monitor, Semaphore)을 데이터 영속성 계층으로 확장한 모델이다.
- **알고리즘 융합**: 데드락 탐지 알고리즘은 그래프 이론의 **Cycle Detection (DFS 활용)**을 기반으로 하며, 타임아웃 기반의 회피 전략은 분산 시스템의 합의 알고리즘과 원리를 공유한다.

- **📢 섹션 요약 비유**: 낙관적 제어는 '자율 주행'과 같고, 비관적 제어는 '모든 교차로에 경찰을 배치하는 것'과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 수강 신청 시스템의 동시성 튜닝
- **상황**: 정원 50명인 강의에 5,000명이 동시 클릭.
- **판단**: 낙관적 제어 사용 시 마지막에 4,950명이 실패하여 서버 부하 폭증.
- **결단**: **비관적 락(SELECT ... FOR UPDATE) 적용**. 수량 체크 단계부터 락을 잡아 정원이 차는 순간 이후 요청을 즉시 차단. 속도는 조금 느려지더라도 데이터의 엄격한 선착순 정합성 확보.

### 2. 안티패턴: 트랜잭션 내 대기 시간(Think Time) 포함
- **증상**: 락을 잡은 상태에서 외부 API를 호출하거나 사용자의 입력을 기다림.
- **결과**: 외부 시스템 지연 시 DB의 락 점유 시간이 길어져 다른 모든 사용자의 쿼리가 줄줄이 대기(Enqueue Wait)하며 시스템 전체 중단.
- **조치**: **트랜잭션은 가장 짧고 굵게**. 모든 준비를 마친 뒤에 DB 연산을 시작하고, `COMMIT`을 가장 빠르게 수행하여 락을 즉시 해제할 것.

### 📢 기술사적 결언
> "병행 제어는 데이터베이스의 **'교통 제어 시스템'**이다. 속도보다는 사고 없는 흐름이 우선이기 때문이다. 기술사는 단순히 락을 거는 것을 넘어, **락의 입도(Granularity)**와 **지속 시간**이 동시성 지표인 **TPS(Transaction Per Second)**에 미치는 상관관계를 계산해야 한다. 락은 정합성을 위한 수호자이지만, 과도한 락은 시스템을 굳게 만드는 독이 될 수 있음을 인지하고 **MVCC와 같은 비차단(Non-blocking) 기술**을 적극적으로 설계에 녹여내야 한다."

- **📢 섹션 요약 비유**: 락을 너무 크게(테이블 단위) 잡는 것은 '화장실 한 칸 쓰려고 건물 전체 문을 잠그는 것'과 같아 엄청난 민폐(성능 저하)를 줍니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (병행 제어 최적화 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 오염 발생률** | 상존 | **0% (Lock 보장)** | 비즈니스 정합성 완벽 확보 |
| **최대 동시 처리량** | 낮음 (경합 심함) | **높음 (세밀한 제어)** | 시스템 수용량 5배 증가 |
| **장애 조치 비용** | 수동 데이터 복구 | **시스템 자동 통제** | 운영 리스크 획기적 절감 |

### 2. 미래 전망
- **Distributed Lockless Algorithms**: 네트워크 지연이 큰 분산 DB에서 락 없이도 순서를 보장하는 **HLC(Hybrid Logical Clock)**와 **Deterministic Concurrency Control** 기술 고도화.
- **AI-Managed Lock Granularity**: 쿼리 패턴을 분석하여 레코드 락이 유리할지 페이지 락이 유리할지 AI가 실시간으로 결정하는 **자율 제어 엔진** 보편화.

- **📢 섹션 요약 비유**: 미래의 병행 제어는 우리가 싸우지 않게 말리지 않아도, 데이터들이 스스로 투명해지거나 길을 비켜주며 공존하는 '초협력 지능'이 될 것입니다.
