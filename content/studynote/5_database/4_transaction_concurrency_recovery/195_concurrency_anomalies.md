# 195. 병행 수행의 이상 현상 (Concurrency Anomalies) - 동시성 제어 실패 시 발생하는 데이터 무결성 위협

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 병행 수행의 이상 현상(Concurrency Anomalies)은 여러 트랜잭션이 동시에 실행될 때, 적절한 격리성(Isolation)이 보장되지 않아 발생하는 **데이터 불일치 및 무결성 훼손 문제**이다.
> 2. **유형**: 대표적인 이상 현상으로 다른 트랜잭션의 갱신을 덮어쓰는 갱신 손실(Lost Update), 확정되지 않은 데이터를 읽는 오손 읽기(Dirty Read), 반복 읽기 시 값이 달라지는 모순성(Non-Repeatable Read), 새로운 데이터가 나타나는 유령 읽기(Phantom Read)가 있다.
> 3. **가치**: 이러한 이상 현상을 정확히 이해하는 것은, 시스템의 트랜잭션 고립화 수준(Isolation Level)을 적절히 설정하고 애플리케이션 레벨의 락(Lock) 전략을 설계하기 위한 **필수적인 이론적 기반**이 된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**병행 수행의 이상 현상 (Concurrency Anomalies)**은 다중 사용자 데이터베이스 환경에서 여러 트랜잭션(Transaction)이 동시(Concurrent)에 실행되면서 같은 데이터에 접근할 때, **순차적으로 실행되었을 때와 다른 잘못된 결과를 초래하는 논리적 오류**를 의미합니다.

### 💡 비유: 공동 문서 편집의 혼란
마치 클라우드 문서(Google Docs 등)에서 동시 편집 기능이 제대로 작동하지 않을 때, 내가 쓴 문장을 다른 사람이 덮어써서 날아가버리거나(갱신 손실), 다른 사람이 쓰다가 지운 임시 문장을 내가 복사해 가는(오손 읽기) 혼란스러운 상황과 같습니다.

### 2. 등장 배경
- **동시성(Concurrency)과 일관성(Consistency)의 상충**: 시스템의 처리량(Throughput)을 높이기 위해 트랜잭션을 병행(Interleaving) 실행해야 하지만, 이는 필연적으로 데이터 자원 접근의 충돌을 야기합니다.
- **격리성 위배**: ACID 원칙 중 격리성(Isolation)이 100% 보장되는 직렬화(Serializable) 스케줄은 성능 저하가 커서 현실적으로 사용하기 어렵습니다. 따라서 의도적으로 격리 수준을 낮추게 되고, 이 틈을 타 이상 현상이 발생하게 됩니다.

- **📢 섹션 요약 비유**: 여러 명의 화가가 하나의 캔버스에 동시에 그림을 그릴 때, 각자의 붓터치 순서를 조율하지 않으면 캔버스가 엉망진창(데이터 오염)이 되는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 이상 현상 (Anomaly) 구성 요소

| 이상 현상 | 역할 (문제 정의) | 내부 원인 | SQL 표준 격리 수준 | 비유 |
|:---|:---|:---|:---|:---|
| **갱신 손실 (Lost Update)** | 하나의 트랜잭션이 갱신한 내용을 다른 트랜잭션이 덮어써서 무효화됨 | 두 트랜잭션이 동시에 같은 데이터를 읽고 각각 수정 후 저장 시 발생 | 완벽히 차단해야 함 (모든 레벨) | 내 예금 입금을 남의 출금이 덮어씀 |
| **오손 읽기 (Dirty Read)** | 다른 트랜잭션이 수정 중이나 아직 커밋되지 않은 데이터를 읽음 | 트랜잭션 T1이 수정 후 롤백했는데, 그 사이 T2가 읽어감 | Read Uncommitted 허용 | 임시 보관함의 편지를 진짜인 줄 알고 읽음 |
| **모순성 / 반복 불가능 읽기 (Non-Repeatable Read)** | 한 트랜잭션 내에서 같은 데이터를 두 번 읽을 때 값이 달라짐 | T1이 읽고 있는 도중 T2가 해당 데이터를 **수정(UPDATE)** 하고 커밋함 | Read Committed까지 허용 | 아침에 본 뉴스 기사가 점심에 바뀌어 있음 |
| **유령 읽기 (Phantom Read)** | 이전에 없던 새로운 데이터(행)가 갑자기 나타남 | T1이 범위 검색 중 T2가 그 범위 내에 새로운 데이터를 **삽입(INSERT)**함 | Repeatable Read까지 허용 | 명부에 없던 유령 회원이 갑자기 추가됨 |

### 2. 이상 현상의 동작 원리와 타임라인 다이어그램

각 이상 현상이 발생하는 과정을 시간 축(Time)에 따라 다이어그램으로 분석합니다.

#### ① 갱신 손실 (Lost Update)

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                      [갱신 손실 (Lost Update) 타임라인]                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  시간(t)     트랜잭션 T1 (입금)                트랜잭션 T2 (출금)               │
│  ─────────────────────────────────────────────────────────────────────────  │
│   t1      Read(A) : A=100                                                   │
│   t2                                      Read(A) : A=100                   │
│   t3      A = A + 50 (A=150 계산)                                           │
│   t4                                      A = A - 30 (A=70 계산)            │
│   t5      Write(A) : A=150 기록                                             │
│   t6                                      Write(A) : A=70 기록 ◀ T1 덮어씀 │
│   t7      Commit                                                            │
│   t8                                      Commit                            │
│                                                                             │
│  [결과] T1이 입금한 50은 사라지고, A의 최종 값은 70이 됨 (데이터 정합성 파괴)      │
└─────────────────────────────────────────────────────────────────────────────┘
```

**[해설]** 갱신 손실은 두 트랜잭션이 동일한 변수(A)를 읽고 산술 연산을 한 후, 각자 덮어쓰는(Write) 과정에서 선행 트랜잭션의 갱신이 무효화되는 가장 치명적인 오류입니다. 이는 데이터베이스 내부 락(Lock)이나 애플리케이션의 비관적/낙관적 락을 통해 100% 방어해야 합니다.

#### ② 오손 읽기 (Dirty Read)

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        [오손 읽기 (Dirty Read) 타임라인]                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  시간(t)     트랜잭션 T1                         트랜잭션 T2                    │
│  ─────────────────────────────────────────────────────────────────────────  │
│   t1      Read(A) : A=100                                                   │
│   t2      Write(A) : A=200 (미확정)                                           │
│   t3                                      Read(A) : A=200 ◀ 오손 읽기 발생 │
│   t4                                      A = A * 2 (A=400)                 │
│   t5      Rollback! (A는 100으로 원복)                                         │
│   t6                                      Write(A) : A=400                  │
│   t7                                      Commit                            │
│                                                                             │
│  [결과] T2는 T1이 취소(Rollback)한 허구의 데이터를 기반으로 연산을 수행함         │
└─────────────────────────────────────────────────────────────────────────────┘
```

**[해설]** 오손 읽기는 다른 트랜잭션이 아직 Commit하지 않은(Uncommitted) 중간 상태의 데이터를 읽음으로써 발생합니다. T1이 모종의 이유로 Rollback을 수행하면, T2는 데이터베이스 역사상 존재하지 않았던 값을 기반으로 영구적인 변화를 주게 되어 연쇄 복귀(Cascading Rollback)를 유발하거나 데이터 오염을 일으킵니다.

#### ③ 반복 불가능 읽기 (Non-Repeatable Read)

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                 [반복 불가능 읽기 (Non-Repeatable Read) 타임라인]               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  시간(t)     트랜잭션 T1                         트랜잭션 T2                    │
│  ─────────────────────────────────────────────────────────────────────────  │
│   t1      Read(A) : A=100                                                   │
│   t2                                      Read(A) : A=100                   │
│   t3                                      Write(A) : A=300                  │
│   t4                                      Commit                            │
│   t5      Read(A) : A=300 ◀ 값이 변경됨!                                     │
│   t6      Commit                                                            │
│                                                                             │
│  [결과] T1은 동일한 트랜잭션 내에서 A를 두 번 읽었는데 결과가 100 → 300으로 다름   │
└─────────────────────────────────────────────────────────────────────────────┘
```

**[해설]** 모순성(Inconsistency)이라고도 불리며, 한 트랜잭션 내에서 일관된 읽기를 보장받지 못하는 상황입니다. 주로 다른 트랜잭션이 기존 행을 **UPDATE** 하거나 **DELETE** 할 때 발생합니다. 총합을 구하거나 정밀한 분석 리포트를 생성할 때 데이터의 무결성이 흔들리게 됩니다.

#### ④ 유령 읽기 (Phantom Read)

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        [유령 읽기 (Phantom Read) 타임라인]                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  시간(t)     트랜잭션 T1 (집계)                 트랜잭션 T2 (신규 가입)           │
│  ─────────────────────────────────────────────────────────────────────────  │
│   t1      Select COUNT(*) (조건: 부서=A)                                     │
│           → 결과: 10명                                                        │
│   t2                                      Insert (부서=A 신입사원 추가)        │
│   t3                                      Commit                            │
│   t4      Select COUNT(*) (조건: 부서=A)                                     │
│           → 결과: 11명 ◀ 유령(Phantom) 데이터 등장                            │
│   t5      Commit                                                            │
│                                                                             │
│  [결과] T1의 동일 조건(WHERE) 검색 결과에 이전에 없던 Row가 새로 나타남           │
└─────────────────────────────────────────────────────────────────────────────┘
```

**[해설]** 유령 읽기는 다른 트랜잭션이 새로운 데이터를 **INSERT** 함으로써 발생합니다. Repeatable Read 격리 수준에서는 기존 데이터에 대한 변경(Update)은 막을 수 있지만, 새로운 행이 삽입되는 것을 막기 위해서는 테이블 전체나 범위(Range/Gap) 단위의 락이 필요합니다.

- **📢 섹션 요약 비유**: 갱신 손실은 내 파일에 친구가 덮어쓰기 하는 것, 오손 읽기는 친구의 초안을 베껴 숙제하는 것, 반복 불가능 읽기는 읽던 책의 페이지 내용이 바뀌는 것, 유령 읽기는 빈 줄에 새로운 글이 솟아나는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 이상 현상과 트랜잭션 고립화 수준 (Isolation Level)의 상관관계

ANSI/ISO SQL 표준은 이러한 이상 현상들이 발생하는 여부에 따라 4단계의 격리 수준을 정의합니다.

| 고립화 수준 (Isolation Level) | 오손 읽기 (Dirty Read) | 반복 불가능 읽기 (Non-Repeatable Read) | 유령 읽기 (Phantom Read) | 동시성 (성능) |
|:---|:---:|:---:|:---:|:---:|
| **Read Uncommitted (LV 0)** | 발생 (O) | 발생 (O) | 발생 (O) | 가장 높음 (최고) |
| **Read Committed (LV 1)** | 방지 (X) | 발생 (O) | 발생 (O) | 높음 (Oracle 기본) |
| **Repeatable Read (LV 2)** | 방지 (X) | 방지 (X) | 발생 (O) | 보통 (MySQL 기본) |
| **Serializable (LV 3)** | 방지 (X) | 방지 (X) | 방지 (X) | 가장 낮음 (최하) |

> *참고*: **갱신 손실 (Lost Update)**은 모든 격리 수준에서 데이터베이스가 자체적으로 배타 락(X-Lock) 등을 사용하여 기본적으로 방어합니다.

### 2. 동시성 제어 기법과의 연계

- **락킹 (Locking)**: 비관적(Pessimistic) 접근법. S-Lock(공유 락)과 X-Lock(배타 락)을 통해 이상 현상을 물리적으로 차단합니다. 2단계 락킹(2PL)을 통해 직렬성을 보장합니다.
- **다중 버전 동시성 제어 (MVCC)**: 낙관적(Optimistic) 읽기 접근법. 데이터를 덮어쓰지 않고 변경 시마다 스냅샷(버전)을 생성하여 읽기 트랜잭션에 과거 버전을 제공함으로써, 락 없이 오손 읽기와 반복 불가능 읽기를 방어합니다.

- **📢 섹션 요약 비유**: 이상 현상들은 막아야 할 '도둑'이며, 고립화 수준(Isolation Level)은 경비의 '경계 태세 등급', 동시성 제어 기법은 경비원이 사용하는 '방패와 CCTV'에 해당합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오
- **결제 시스템 (갱신 손실 방지)**: 사용자의 포인트 차감과 적립이 동시에 일어나는 시스템에서 Lost Update가 발생하면 회사에 막대한 금전적 손실이 발생합니다. DB 기본 락에 의존하기보다, 애플리케이션 레벨에서 `SELECT ... FOR UPDATE`를 활용한 비관적 락이나, Version 컬럼을 이용한 낙관적 락을 명시적으로 구현해야 합니다.
- **일일 마감 집계 배치 (유령 읽기 방어)**: 밤 12시에 그날의 총매출을 집계(SUM)하는 배치 프로그램이 도는 중, 심야 결제 건이 INSERT되면 12시 정각 기준의 정확한 정산이 깨집니다(Phantom Read). 이 경우 배치 트랜잭션을 Serializable로 격리하거나, 특정 시간대 기준의 MVCC 스냅샷을 활용하여 집계 무결성을 보장해야 합니다.

### 2. 도입 체크리스트
- **비즈니스 정합성 파악**: 이 서비스가 재무/결제 등 1원 단위의 무결성이 중요한가(Serializable 검토), 아니면 SNS 피드처럼 약간의 불일치(Dirty Read/Non-Repeatable)를 감수하더라도 동시성이 중요한가 파악.
- **RDBMS 기본 격리 수준 확인**: 운영 중인 DB(Oracle은 Read Committed, MySQL InnoDB는 Repeatable Read)의 기본 격리 수준이 비즈니스 요구사항의 이상 현상을 커버하는지 검증.

### 3. 안티패턴
- **무조건적인 Serializable 사용**: 무결성을 명분으로 모든 트랜잭션 수준을 최고(Serializable)로 올리면, 테이블 레벨 락에 준하는 대기열이 발생하여 트래픽이 몰릴 때 시스템 전체가 타임아웃(Timeout)으로 뻗어버리는 심각한 장애(장애 파급)를 초래합니다.

- **📢 섹션 요약 비유**: 완벽한 무균실(Serializable)을 만들면 병(이상 현상)은 없겠지만 숨이 막혀 살 수 없습니다. 서비스의 특성에 맞게 적절한 면역력(격리 수준)을 갖추고 활동성(성능)을 보장하는 것이 아키텍트의 역할입니다.

### 1. 정량/정성 기대효과

| 구분 | 이상 현상 관리 실패 시 | 철저한 격리 수준 최적화 적용 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 교착 상태(Deadlock) 및 Timeout 급증 | TPS (초당 트랜잭션) 최적화 | 처리량(Throughput) **5배 이상 증가** |
| **정량** | 금전적 데이터 불일치 빈발 | 데이터 무결성 결함 0 달성 | 갱신 손실로 인한 장애 비용 **100% 절감** |
| **정성** | 개발자의 DB 신뢰도 하락 | 예측 가능한 트랜잭션 동작 보장 | 안정적인 애플리케이션 아키텍처 구축 |

### 2. 결론
병행 수행의 이상 현상은 다중 사용자 환경에서 피할 수 없는 '물리적 마찰'입니다. 이를 무작정 회피하기보다는 **갱신 손실**, **오손 읽기**, **모순성**, **유령 읽기**의 발생 원리와 구조를 정확히 이해하고, DBMS가 제공하는 격리 수준(Isolation Level)과 MVCC 아키텍처를 지렛대 삼아 **'데이터 정합성'과 '동시 처리 성능' 사이의 최적의 균형점(Trade-off)**을 찾는 것이 데이터베이스 엔지니어링의 핵심입니다.

- **📢 섹션 요약 비유**: 이상 현상이라는 '질병'을 정확히 진단해야, 불필요한 전신 마취(전체 락) 없이 정교한 표적 치료(최적 고립화 수준 적용)가 가능해집니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **ACID 원칙**: 격리성(Isolation)을 위배할 때 발생하는 것이 이상 현상임.
- **고립화 수준 (Isolation Level)**: 이상 현상을 허용하는 정도에 따라 4단계로 나눈 ANSI SQL 표준.
- **병행 제어 및 락 (Concurrency Control)**: 이상 현상을 제어하기 위한 물리적 통제 수단 (공유 락, 배타 락).
- **MVCC (다중 버전 동시성 제어)**: 락 없이 오손 읽기와 모순성을 해결하는 현대 DBMS의 핵심 아키텍처.

## 👶 어린이를 위한 3줄 비유 설명
1. **갱신 손실(Lost Update)**은 내가 공책에 적은 답을 짝꿍이 자기 지우개로 지우고 자기 답을 덮어써 버려서 내 답이 날아가는 속상한 상황이에요.
2. **오손 읽기(Dirty Read)**는 친구가 정답을 쓰고 아직 확인(Commit)도 안 한 오답 노트를 슬쩍 보고 베꼈는데, 친구가 나중에 정답을 바꿔버려서 나 혼자 틀리는 상황이죠.
3. 이런 엉망진창인 일(이상 현상)을 막기 위해 데이터베이스는 **"차례대로, 확인된 것만 보자!"**라는 규칙(격리 수준)을 만들어서 모두가 싸우지 않고 사이좋게 공책을 나눠 쓸 수 있게 해준답니다!
