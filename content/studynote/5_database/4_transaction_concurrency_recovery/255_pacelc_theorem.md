# 255. PACELC 정리 - CAP 이론의 완성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PACELC 정리는 CAP 이론의 한계를 보완하여, **네트워크 분할(P) 시의 트레이드오프(A/C)뿐만 아니라, 정상 상황(E)에서의 지연 시간(L)과 일관성(C)의 상충 관계**까지 정의한 현대적 분산 시스템 이론이다.
> 2. **가치**: "평상시에는 얼마나 빠른가?"라는 성능적 측면을 이론 체계에 편입시킴으로써, 실제 서비스 운영 환경에서의 데이터베이스 동작 방식을 더욱 정교하게 모델링한다.
> 3. **융합**: NoSQL의 고성능 복제(Replication) 전략과 결합되어, 초저지연(Ultra Low Latency) 서비스 설계를 위한 핵심 아키텍처 가이드가 된다.

+++

### Ⅰ. PACELC 정리의 구조 (해독)

**P(Partition) A(Availability) C(Consistency) / E(Else) L(Latency) C(Consistency)**

1. **If P (네트워크 분단 시)**: 
    - **A** (가용성)를 택할 것인가? 
    - **C** (일관성)를 택할 것인가? (CAP 이론과 동일)
2. **Else (정상 작동 시)**:
    - **L** (지연 시간)을 최소화할 것인가?
    - **C** (일관성)를 유지할 것인가?

+++

### Ⅱ. PACELC 유형 분류 (ASCII Flow)

```text
[PACELC 기반의 시스템 분류]

  If Partition(P)?      Else(E) Normal?
  ┌──────┴──────┐       ┌──────┴──────┐
  ▼             ▼       ▼             ▼
  A             C       L             C
 (AP)          (CP)    (EL)          (EC)

  - [PA / EL]: 장애 시 응답 우선, 평소에도 속도 우선 (Cassandra, DynamoDB)
  - [PA / EC]: 장애 시 응답 우선, 평소에는 정합성 우선 (MongoDB)
  - [PC / EC]: 장애 시 서비스 중단, 평소에도 정합성 우선 (BigTable, HBase)
```

+++

### Ⅲ. 왜 'EL'과 'EC'가 중요한가?

- **정상 상황의 비용**: 네트워크가 멀쩡해도, 모든 노드에 데이터를 복제하느라 기다리면 사용자는 '느리다'고 느낍니다 (EC). 
- **속도의 유혹**: 복제가 완료되기 전에 사용자에게 응답을 주면 빠르지만(EL), 바로 뒤에 읽은 사람은 옛날 데이터를 볼 수 있습니다.
- **실무 선택**: 현대의 글로벌 서비스(넷플릭스, 아마존 등)는 대부분 **PA/EL** 구조를 택하여 극한의 성능을 추구합니다.

- **📢 섹션 요약 비유**: PACELC는 **'철저한 식당 주인'**과 같습니다. "손님이 너무 많아 재료가 떨어지면(P), 맛없게라도 팔지(A) 아예 문을 닫을지(C)"를 고민할 뿐만 아니라, "평소에도(E), 대충 빨리 요리할지(L) 시간이 걸려도 완벽한 맛을 낼지(C)"까지 규칙을 정해둔 것입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Latency]**: 정상 상황에서 지불해야 하는 가장 큰 대가.
- **[Replication Factor]**: L과 C 사이를 조절하는 물리적 수치.
- **[DynamoDB]**: PACELC 이론을 실제 인프라에 구현한 대표적 사례.

📢 **마무리 요약**: **PACELC**는 분산 시스템의 현실적인 지도입니다. 장애뿐만 아니라 **평상시의 성능**까지 고려한 설계만이 진정한 고가용성을 완성합니다.
