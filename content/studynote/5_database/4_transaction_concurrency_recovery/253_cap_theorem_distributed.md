# 253. CAP 정리 (CAP Theorem) - 분산 시스템의 삼자 대립

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CAP 정리는 분산 데이터 시스템이 **일관성(Consistency), 가용성(Availability), 분단 허용성(Partition Tolerance)**이라는 세 가지 속성을 동시에 모두 만족시킬 수 없음을 증명한 이론이다.
> 2. **가치**: 네트워크 장애(P)가 불가피한 분산 환경에서, 시스템 아키텍처가 데이터 정합성(C)을 우선할 것인지 아니면 서비스 무중단(A)을 우선할 것인지 결정하는 **전략적 판단 기준**을 제공한다.
> 3. **융합**: NoSQL의 폭발적 성장을 이론적으로 뒷받침하며, 현대 클라우드 인프라 설계 시 비즈니스 요구사항에 맞는 최적의 데이터베이스 선정의 근거가 된다.

+++

### Ⅰ. CAP의 3가지 요소 정의

1. **일관성 (Consistency)**: 모든 노드가 어떤 시점에서도 동일한 데이터를 보아야 함. (쓰기 직후 어디서 읽어도 최신 값 반환)
2. **가용성 (Availability)**: 일부 노드가 장애가 나더라도, 모든 요청에 대해 성공/실패 응답을 보낼 수 있어야 함. (무중단 서비스)
3. **분단 허용성 (Partition Tolerance)**: 노드 간 네트워크가 끊어지더라도 시스템이 계속 동작해야 함. (분산 시스템의 필수 전제)

+++

### Ⅱ. CAP 트레이드오프 (ASCII Triangle)

네트워크 분할(P) 상황은 반드시 발생하므로, 현실에서는 **CP** 또는 **AP** 중 하나를 선택해야 합니다.

```text
[CAP Theorem Triangle]

           Consistency (C)
               / \
              /   \
          CP /     \ CA (RDBMS - 단일망 환경)
            /       \
           /_________\
Availability(A)  AP  Partition Tolerance(P)

  - CP: 네트워크 장애 시 일관성을 위해 서비스 중단 (HBase, MongoDB 기본)
  - AP: 네트워크 장애 시 가용성을 위해 낡은 데이터 반환 허용 (Cassandra, DynamoDB)
```

+++

### Ⅲ. 분산 시스템에서의 선택

- **전제**: 분산 시스템이라면 **P(분단 허용성)**는 포기할 수 없는 기본값입니다.
- **CP 선택 (C+P)**: 데이터가 틀리면 안 되는 금융, 결제 등. 네트워크 문제 시 "현재 시스템을 이용할 수 없습니다"라고 응답하는 것이 정석입니다.
- **AP 선택 (A+P)**: 소셜 미디어 게시글, 실시간 댓글 등. 네트워크 문제 시 "조금 늦게 반영되더라도 일단 서비스를 유지하는 것"이 사용자 경험상 유리합니다.

- **📢 섹션 요약 비유**: CAP 정리는 **'분식집의 메뉴 선택'**과 같습니다. "싸고(A), 맛있고(C), 양 많은(P)" 세 가지를 모두 만족하는 메뉴는 세상에 없습니다. 양이 많으려면(P) 비싸지거나 맛이 조금 떨어지는 것을 감수해야 하듯, 시스템도 환경에 맞춰 가장 중요한 가치를 골라야 합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[P 선택의 강제성]**: 분산 시스템은 무조건 P를 가져가야 함.
- **[PACELC]**: CAP 정리의 한계를 보완한 확장 이론.
- **[NoSQL]**: CAP 정리의 결과로 탄생한 다양한 데이터 저장소들의 집합.

📢 **마무리 요약**: **CAP Theorem**은 분산 아키텍처의 기본 물리 법칙입니다. 이 한계를 인정하고 비즈니스에 가장 이득이 되는 **Trade-off**를 설계하는 것이 뛰어난 아키텍트의 자질입니다.
