# 253. CAP 정리 (CAP Theorem) - 분산 시스템의 삼자 대립

### # [253. CAP 정리 (CAP Theorem)]
#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: **CAP 정리 (CAP Theorem)**는 분산 컴퓨팅 시스템이 **일관성 (Consistency, C)**, **가용성 (Availability, A)**, **분단 허용성 (Partition Tolerance, P)**의 세 가지 보장을 동시에 100% 충족할 수 없음을 수학적으로 증명한 정리입니다.
> 2. **가치**: **RDBMS (Relational Database Management System)**의 한계를 극복한 **NoSQL (Not Only SQL)**의 등장을 이론적으로 뒷받침하며, 비즈니스의 특성(금융 vs SNS)에 따라 **CP** 계열(데이터 정합성 중시) 또는 **AP** 계열(서비스 연속성 중시) 시스템을 선택하는 핵심 설계 지침을 제공합니다.
> 3. **융합**: 클라우드 네이티브 아키텍처에서 **고가용성 (High Availability, HA)** 요구사항과 데이터 무결성 제약 사이의 타협점(Trade-off)을 설계하는 데 필수적인 원리이며, **PACELC** 이론 등과 연계됩니다.

+++

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 철학
**CAP 정리**는 2000년 에릭 브루어(Eric Brewer)가 제안하고 2002년에 증명된 이론으로, 분산 시스템에서 데이터 일관성과 서비스 가용성은 상충 관계에 있음을 정의합니다. 네트워크 통신은 기본적으로 신뢰할 수 없으며(Fallible), **분단 (Partition)**, 즉 노드 간 통신 두절은 필연적으로 발생합니다. 따라서 시스템 설계자는 네트워크 장애 발생 시 '오래된 데이터를 보여주고 서비스를 유지할 것인가(A)' 아니면 '서비스를 중단하고라도 데이터 정합성을 지킬 것인가(C)'를 선택해야 합니다.

#### 2. 등장 배경
① **기존 한계 (RDBMS의 확장성)**: 기존의 관계형 데이터베이스는 **ACID (Atomicity, Consistency, Isolation, Durability)** 특성을 통해 강력한 일관성을 보장했지만, 수평 확장(Scaling-out)에 한계가 있어 대규모 트래픽 처리에 어려움을 겪음.
② **혁신적 패러다임 (NoSQL의 부상)**: 수천 대의 서버에서 데이터를 분산 저장하는 대용량 웹 서비스(구글, 아마존 등)가 등장하며, 일관성을 일부 희생하고서라도 확장성과 가용성을 확보하려는 움직임이 시작됨.
③ **현재의 비즈니스 요구**: 글로벌 서비스에서는 24/365 무중단 서비스가 필수적이므로, 일관성 수준을 조정(Tunable Consistency)할 수 있는 분산 데이터베이스 설계가 표준이 됨.

#### 📢 섹션 요약 비유
CAP 정리는 **"분식집의 메뉴 선택"**과 같습니다. "맛있고(C), 싸고(A), 양이 많은(P)" 메뉴는 세상에 없습니다. 분식집이든 고급 레스토랑이든, 사장님은 셋 중 어느 하나를 희생해야만 합니다. 시스템 설계자도 이 세 가지를 동시에 가질 수 없으며, 서비스의 성격에 맞춰 하나를 포기해야 합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 구성 요소 상세 분석
CAP 정리의 핵심은 세 가지 속성이 정확히 무엇을 의미하는지, 특히 분산 환경에서 어떻게 충돌하는지 이해하는 것입니다.

| 속성 | 전체 명칭 (약어) | 정의 및 내부 동작 | 프로토콜/특징 | 비유 |
|:---|:---|:---|:---|:---|
| **C** | **Consistency (일관성)** | 모든 노드가 동일한 시점에 동일한 데이터를 반환. 선형성(Linearizability) 보장. 쓰기 완료 후 모든 읽기에서 최신 값 조회. | 동기식 복제 (Synchronous Replication), 2PC (Two-Phase Commit) | 단일 장부; 어느 지점에서 조회해도 잔고가 같음. |
| **A** | **Availability (가용성)** | (비잔존 장애 노드 제외) 모든 노드가 유효한 요청에 대해 반드시 응답. 에러나 타임아웃을 반환하지 않음. | 비동기식 복제 (Asynchronous Replication), Eventually Consistent | 24시간 편의점; 문이 닫히지 않아 언제든 들어갈 수 있음. |
| **P** | **Partition Tolerance (분단 허용성)** | 노드 간 네트워크 분단이 발생해도 시스템이 동작. 메시지 유실이나 지연이 발생해도 시스템이 붕괴하지 않음. | Gossip Protocol, Quorum (N/2+1) | 끊어지지 않는 다리; 다리가 끊어져도 배로 우회하여 연결 유지. |

#### 2. CAP 상관관계 및 트레이드오프 다이어그램
분산 시스템에서 네트워크 장애(P)는 물리적 현실이므로 선택 불가능합니다. 따라서 설계 시 **CA**, **CP**, **AP** 중 하나를 선택해야 합니다.

```text
      [CAP Theorem System Selection Map]

            Consistency (C)
            (정확한 데이터)
                  ▲
                  │
                  │  [CA System]
                  │  (Monolithic RDBMS on Stable LAN)
                  │  - Redis Sentinel (Stable mode)
                  │
                  ├───────────────────────┐
                  │                       │
                  │                       │
   [CP System]    │           [AP System]
 (정합성 우선)     │        (가용성 우선)
   "Stop or       │       "Stale Data OK"
    Correct"      │
   HBase,         │        Cassandra, CouchDB
   MongoDB,       │        DynamoDB, DNS
   Redis Cluster  │
                  │
                  │
      Partition Tolerance (P)
      (네트워크 장애 감내 & 분산 환경)
```

#### 3. 심층 동작 원리 (시나리오 분석)
**상황**: 노드 1과 노드 2가 네트워크로 연결된 상태에서, 두 노드 사이의 네트워크가 단절됨(P 발생). 사용자가 노드 1에 데이터를 쓰려고 함.

1.  **CP 시스템 동작 (Consistency 우선)**:
    -   노드 1은 "노드 2에게 데이터를 복제할 수 없음"을 감지.
    -   데이터 불일치를 막기 위해 **쓰기 작업을 거부(Block)**하거나, 노드 2와 복구될 때까지 대기(Lock).
    -   **결과**: 사용자는 에러를 경험하지만, 데이터는 깨지지 않음. (금융 거래)
2.  **AP 시스템 동작 (Availability 우선)**:
    -   노드 1은 "노드 2와 연결되지 않음"에도 불구하고, **즉시 쓰기를 승인(Ack)**하고 로컬에 저장.
    -   나중에 네트워크가 복구되면 노드 2에게 변경 사항을 전파(Async Replication).
    -   **결과**: 사용자는 즉시 응답을 받지만, 일시적으로 노드 1과 2의 데이터가 다를 수 있음(Soft State).

#### 📢 섹션 요약 비유
이는 **"지갑을 잃어버린 경우"**와 같습니다. **CP(일관성)**는 "돈이 조금이라도 사라질까 봐 밖에 나가지도 못하고(장애), 구석구석까지 샅샅이 뒤져서 찾을 때까지 집에만 있는(중단)" 전략입니다. **AP(가용성)**는 "일부 잃더라도 우선 버스를 타고 출근은 가자(서비스 유지). 나중에 생각해보자(추후 동기화)"는 전략입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 심층 기술 비교표: RDBMS vs NoSQL

| 비교 항목 | **RDBMS (CA/CP 지향)** | **NoSQL (AP 지향 또는 Tunable CP)** |
|:---|:---|:---|
| **대표 제품** | **MySQL, Oracle, PostgreSQL** | **Cassandra, DynamoDB, Couchbase, MongoDB** |
| **CAP 기본값** | **C+A (단일 노드/안정적인 LAN)**<br>또는 **C+P (클러스터링 시)** | **A+P** (기본적으로 Eventual Consistency)<br>일부는 **C+P** 설정 가능 |
| **일관성 모델** | 강한 일관성 (Strong Consistency) <br>항상 최신 데이터 보장 | 최종 일관성 (Eventual Consistency) <br>시간 차 두고 동기화 (BASE 이론) |
| **확장성 (Scale)** | 수직 확장 (Scale-up) <br>고성능 서버 1대 필요 | 수평 확장 (Scale-out) <br>저사양 서버 다수 추가 가능 |
| **데이터 구조** | 스키마 고정 (Schema-on-write) <br>정형화된 테이블 | 스키마 자유 (Schema-on-read) <br>JSON/Key-Value/Column |
| **트랜잭션** | **ACID** 보장 (복잡한 쿼리/Join 지원) | **ACID** 지원 어려움 (또는 제한적) <br>**BASE** (Basically Available, Soft state, Eventually consistent) |
| **적합 분야** | 은행, 재고 관리, 결제 시스템 | SNS 피드, 실시간 로그 분석, IoT 센서 데이터, 캐싱 |

#### 2. 과목 융합 관점 (OS & Network)
-   **네트워크 (Network) 관점**: **P**는 **TCP/IP** 계층의 물리적 한계에서 기인합니다. **FLP Impossibility (Fischer, Lynch, Paterson)**는 비동기 메시지 전달 환경에서 합의(Consensus)가 불가능함을 증명했으며, 이는 네트워크 지연(Latency)과 패킷 손실이 분산 합의 알고리즘(예: **Raft**, **Paxos**)의 복잡도를 결정함을 의미합니다.
-   **운영체제 (OS) 관점**: 분산 파일 시스템(예: **Google GFS**, **HDFS**)은 대용량 처리를 위해 **CP**에 가까운 설계(쓰기 일관성 보장)를 채택하는 반면, 캐싱 시스템(예: **Memcached**, **Redis Cluster**)은 속도를 위해 **AP**나 **AP-like** 성향(일관성보다 응답 속도)을 가집니다.

#### 3. 의사결정 매트릭스 (TPS vs Latency)
-   **CP 시스템**: 네트워크 복구 시까지 **Latency**가 급격히 증가하거나 Timeout 발생. 데이터 정확도가 중요한 금융 거래(단건 TPS 낮음, 안정성 중요).
-   **AP 시스템**: 네트워크 장애 시에도 **Latency**는 일정하게 유지. 하지만 데이터 최신성이 떨어질 수 있어, 대량의 조회 요청(높은 TPS)을 처리하는 뉴스 피드에 적합.

#### 📢 섹션 요약 비유
자동차의 **서스펜션** 설계와 같습니다. **스포츠카(CP)**는 노면(네트워크)의 울퉁불퉁함을 그대로 체감하면서라도 코너링(데이터 정합성)을 정확히 해야 합니다. 반면 **오프로드 자동차(AP)**는 노면이 험해도(데이터 불일치) 타이어가 지면에서 떨어지지 않고 계속 굴러가야(서비스 가용성) 합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 시나리오 및 의사결정

**[시나리오 1: 핀테크 송금 시스템 (CP 선정) ]**
-   **상황**: A 계좌에서 B 계쳐로 100만 원 이체 중 DB 분단 발생.
-   **의사결정**: **일관성(C)** 우선. 분단 발생 시 **트랜잭션 중단 (Abort)**.
-   **이유**: 돈이 사라지거나 이중 지급되면 치명적. "시스템 점검 중" 메시지를 띄우더라도 데이터 무결성이 최우선.
-   **솔루션**: **Consistent Hashing**을 기반으로 하되, 쓰기 시 **Quorum(N/2 + 1)**을 엄격히 적용하는 **HBase** 또는 **MySQL Cluster** 사용.

**[시나리오 2: 쇼핑몰 상품 리뷰 서비스 (AP 선정)]**
-   **상황**: 사용자가 리뷰를 작성했으나, 지역 서버 간 네트워크 지연 발생.
-   **의사결정**: **가용성(A)** 우선. 일단 리뷰 작성 완료 처리 후 백그라운드 동기화.
-   **이유**: 리뷰가 1초 늦게 보이는 것은 비즈니스에 큰 타격이 아님. 서비스 중단(Erro 500)보다 사용자 경험(UX) 우선.
-   **솔루션**: **Write-anywhere, Read-anyone** 모델의 **Cassandra** 또는 **DynamoDB** 사용. **Vector Clock** 등을 이용해 충돌 해소.

**[시나리오 3: 전자상거래 장바구니 (하이브리드/BASE)]**
-   **상황**: 장바구니 담기 기능.
-   **의사결정**: AP 모델 사용하되, 결제 단계로 넘어갈 때 CP 시스템(재고 DB)과 동기화.
-   **솔루션**: 사용자 세션에 정보를 저장(AP)하다가 주문 시점에 재고 시스템(CP)에 확인.

#### 2. 도입 체크리스트

| 구분 | 체크항목 | CP 시스템 (예: HBase) | AP 시스템 (예: Cassandra) |
|:---|:---|:---|:---|
| **기술적** | **데이터 양** | TB ~ PB 급 대용량 가능 (Wide-column) | TB ~ PB 급 대용량 가능 |
| | **쓰기 패턴** | 단일 쓰기 지점이 명확할 때 유리 | 다중 지점에서의 쓰기가 빈번할 때 유리 |
| |
