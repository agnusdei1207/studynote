# 199. 분산 트랜잭션 (Distributed Transaction) - 2PC, 3PC 그리고 Saga 패턴

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 트랜잭션은 물리적으로 분리된 여러 데이터베이스나 서비스에 걸쳐 있는 데이터 조작을 **하나의 논리적 작업 단위(원자성 보장)**로 묶어 처리하는 기술이다.
> 2. **전통적 해법**: 분산 환경에서 완벽한 원자성(ACID)을 보장하기 위해 코디네이터 중심의 합의 프로토콜인 **2단계 커밋(2PC, Two-Phase Commit)**과 **3단계 커밋(3PC)**이 고안되었다.
> 3. **현대적 해법**: 마이크로서비스 아키텍처(MSA) 환경에서는 2PC의 락 대기(Blocking)로 인한 성능 저하를 극복하기 위해, 개별 로컬 트랜잭션을 연속 실행하고 실패 시 **보상 트랜잭션(Compensating Transaction)**을 호출하는 **Saga 패턴(결과적 일관성 기반)**이 표준으로 자리 잡았다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**분산 트랜잭션 (Distributed Transaction)**은 둘 이상의 네트워크상 독립된 노드(데이터베이스, 메시지 큐 등)가 참여하는 트랜잭션입니다. 모든 노드에서의 작업이 완벽하게 모두 성공(Commit)하거나, 하나라도 실패하면 모두 취소(Rollback)되어야 하는 글로벌 원자성(Global Atomicity)을 요구합니다.

### 💡 비유: 해외여행 패키지 예약
항공권, 호텔, 렌터카를 각각 다른 여행사 시스템을 통해 동시에 예약해야 합니다. 항공권은 결제됐는데 호텔 방이 없어서 취소된다면, 이미 결제된 항공권과 렌터카도 취소수수료 없이 완벽하게 물러(Rollback) 줘야만 '여행 패키지 예약'이라는 하나의 트랜잭션이 안전하게 보장됩니다.

### 2. 등장 배경
- **시스템 스케일 아웃(Scale-out)과 MSA**: 단일 모놀리식(Monolithic) 데이터베이스로는 글로벌 트래픽을 감당할 수 없게 되면서 데이터베이스를 샤딩(Sharding)하거나 MSA로 쪼개게 되었습니다.
- **분할된 데이터 간의 정합성 위기**: 주문 DB에서는 결제가 완료되었는데, 배송 DB에는 네트워크 오류로 데이터가 들어가지 않으면 회사의 신뢰도에 치명적인 타격을 줍니다. 분산된 자원을 하나로 묶는 통제자(Coordinator)가 필요해졌습니다.

- **📢 섹션 요약 비유**: 각기 다른 악기를 연주하는 단원(로컬 DB)들이 각자 자기 악보만 보고 연주하면 소음이 되지만, 지휘자(트랜잭션 코디네이터)의 통제에 맞춰 한 번에 시작하고 멈추면 아름다운 교향곡(분산 트랜잭션)이 됩니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 2단계 커밋 (2PC, Two-Phase Commit) 프로토콜

2PC는 분산 트랜잭션의 표준(XA 프로토콜 등)입니다. 전체 트랜잭션을 지휘하는 **코디네이터(Coordinator)**와 실제 데이터를 저장하는 **참여자(Participant / RM)** 로 구성됩니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        [2단계 커밋 (2PC) 동작 메커니즘]                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [Phase 1: 준비 단계 (Prepare Phase)]                                         │
│                                                                             │
│  Coordinator               Participant A (주문 DB)      Participant B (배송 DB) │
│      │                           │                            │             │
│      ├──── 1. Prepare 요청 ─────▶│                            │             │
│      ├──── 1. Prepare 요청 ──────────────────────────────────▶│             │
│      │                           │ (자원 락킹 & 로컬 로그 플러시) │             │
│      │◀─── 2. Prepared (OK) ─────┤                            │             │
│      │◀─── 2. Prepared (OK) ──────────────────────────────────┤             │
│                                                                             │
│  [Phase 2: 커밋 단계 (Commit Phase) - 모든 참여자가 OK 했을 경우]                   │
│                                                                             │
│      ├──── 3. Commit 요청 ──────▶│                            │             │
│      ├──── 3. Commit 요청 ───────────────────────────────────▶│             │
│      │                           │ (실제 커밋 & 락 해제)        │             │
│      │◀─── 4. Done (완료) ───────┤                            │             │
│      │◀─── 4. Done (완료) ────────────────────────────────────┤             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**[작동 원리]**
1. **Prepare**: 코디네이터가 "너희들 커밋할 준비(락 획득, 디스크 쓰기 등) 다 됐어?"라고 묻습니다. 하나라도 NO를 외치거나 타임아웃되면 2단계는 전원 Rollback으로 결정됩니다.
2. **Commit / Rollback**: 전원 OK가 모이면 코디네이터는 "좋아, 다 같이 커밋해!"라고 명령을 내립니다.

**[2PC의 치명적 단점 - 블로킹 (Blocking)]**
- **코디네이터 단일 장애점 (SPOF)**: 참여자가 Prepare를 마친 상태에서 코디네이터가 갑자기 죽어버리면, 참여자는 커밋해야 할지 롤백해야 할지 몰라서 데이터에 **락(Lock)을 건 채로 무한정 대기(Blocking)**하게 됩니다.

### 2. 3단계 커밋 (3PC, Three-Phase Commit)

2PC의 블로킹 문제를 해결하기 위해, Prepare와 Commit 사이에 **Pre-Commit** 단계를 하나 더 추가한 비블로킹(Non-blocking) 프로토콜입니다.

1. **CanCommit (투표)**: "커밋할 수 있니?" (락 없이 의향만 물음)
2. **PreCommit (준비)**: 전원 OK면 "이제 진짜 준비해(락 걸기)"라고 지시.
3. **DoCommit (커밋)**: 실제 커밋 수행.

**[핵심 아이디어]**: 코디네이터가 죽더라도, 참여자들끼리 현재 자신이 어느 단계에 있는지 통신하여 타임아웃 발생 시 스스로 롤백하거나 커밋할 수 있도록 상태를 분산시킨 것입니다. 그러나 네트워크 분할(Network Partition) 발생 시 데이터 불일치 위험이 더 커서 **실무에서는 거의 쓰이지 않습니다.**

### 3. Saga 패턴 (MSA 환경의 분산 트랜잭션)

Saga 패턴은 거대한 글로벌 트랜잭션(2PC)을 포기하고, 각 마이크로서비스의 **로컬 트랜잭션을 연속적으로 이어서** 비즈니스를 완성하는 패턴입니다. (결과적 일관성, Eventual Consistency 기반)

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                [Saga 패턴 코레오그래피 (Choreography) 방식 타임라인]              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [정상 흐름 (Success)]                                                       │
│  주문 서비스 ───(이벤트)──▶ 결제 서비스 ───(이벤트)──▶ 배송 서비스                 │
│  (주문 생성)                (카드 결제)                 (배송 준비)                │
│                                                                             │
│  [실패 및 보상 트랜잭션 흐름 (Failure & Compensation)]                           │
│  주문 서비스 ───(이벤트)──▶ 결제 서비스 ───(이벤트)──▶ 배송 서비스 (재고 부족 ❌)      │
│  (주문 생성)                (카드 결제 완료)                                     │
│                                                                             │
│  (주문 취소) ◀──(이벤트)──  (결제 취소) ◀──(이벤트)──  [배송 실패 이벤트 발행]       │
│  보상 트랜잭션               보상 트랜잭션                                        │
└─────────────────────────────────────────────────────────────────────────────┘
```

**[작동 원리]**
- 각 서비스는 데이터베이스의 S-Lock, X-Lock을 쓰지 않습니다. 자기 DB에만 커밋하고 이벤트를 발행합니다.
- 파이프라인 중간(예: 배송)에 에러가 발생하면, 역순으로 앞선 서비스들에게 **보상 트랜잭션(Compensating Transaction, 예: 결제 취소 API, 주문 취소 API)**을 호출하여 논리적으로 데이터를 원복시킵니다.

- **📢 섹션 요약 비유**: 2PC가 모두가 손을 잡고 동시에 결승선을 넘어야 하는 '2인 3각 달리기'라면, Saga 패턴은 각자 자기 구간을 달리는 '이어달리기(계주)'이며 바통을 떨어뜨리면 역주행해서 처음으로 돌아가는 방식입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 2PC vs Saga 패턴 아키텍처 비교

| 비교 항목 | 2PC (Two-Phase Commit) | Saga 패턴 |
|:---|:---|:---|
| **일관성 수준** | 강한 일관성 (Strong Consistency / ACID) | 결과적 일관성 (Eventual Consistency / BASE) |
| **락 (Lock) 점유** | 전역 락 사용 (성능 심각한 저하, 데드락 위험) | 로컬 락만 사용 (글로벌 락 없음, 성능 우수) |
| **장애 시 롤백 방식** | DBMS 차원의 자동 롤백 (Undo) | 애플리케이션 차원의 보상 트랜잭션 (수동 개발) |
| **아키텍처 적합성** | 단일 벤더의 모놀리식, 사내망 (Intranet) | 클라우드 네이티브, MSA, 외부 API 연동 환경 |
| **격리성 (Isolation)** | 완벽히 보장됨 (Dirty Read 없음) | **보장 안 됨 (Dirty Read 노출됨)** |

### 2. Saga 패턴의 두 가지 구현 방식
- **코레오그래피 (Choreography)**: 중앙 통제자 없이 서비스들이 비동기 이벤트(Kafka, RabbitMQ)를 주고받으며 알아서 다음 행동을 결정합니다. (유연하지만 흐름 추적이 어려움)
- **오케스트레이션 (Orchestration)**: 중앙의 오케스트레이터(Saga Execution Coordinator)가 상태 머신(State Machine)을 가지고 각 서비스에 API 호출 명령을 내립니다. (관리가 편하지만 단일 장애점이 될 수 있음)

- **📢 섹션 요약 비유**: 2PC는 법원의 '가압류'처럼 확실하지만 숨막히는 강제 집행이고, Saga는 '신용 카드 취소'처럼 일단 돈은 빠져나갔다가 나중에 다시 통장으로 돌려주는 유연한 신용 거래입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오와 의사결정
- **시나리오 1: 은행의 계좌 이체 시스템 (강력한 정합성 필수)**: 우리 은행 A지점 계좌에서 B지점 계좌로 돈을 이체합니다. 중간에 돈이 증발하거나 허공에 떠 있는 시간(Eventual Consistency)을 고객이 절대 용납하지 않습니다.
  - **결정**: MSA라 하더라도 이체 도메인만큼은 2PC 프로토콜(XA 트랜잭션)을 사용하거나, 단일 RDBMS 노드에 모아 로컬 트랜잭션(ACID)으로 강하게 묶어야 합니다.
- **시나리오 2: 배달 앱의 주문-결제-가게 전달 (확장성 필수)**: 사용자가 치킨을 주문합니다. 결제가 끝나고 가게 사장님에게 오더가 전달되기까지 1~2초의 지연이나, 취소 시 카드 환불에 수 분이 걸리는 것은 고객이 납득할 수 있는 비즈니스 정책입니다.
  - **결정**: 글로벌 락(2PC)을 걸면 트래픽 폭주 시 시스템이 다운됩니다. 아파치 카프카(Kafka)를 활용한 이벤트 기반의 Saga 패턴(코레오그래피)을 적용하여 TPS를 극대화해야 합니다.

### 2. Saga 패턴 도입 시 필수 체크리스트
- **멱등성 (Idempotency) 보장**: 네트워크 재전송으로 인해 보상 트랜잭션 API가 여러 번 호출되더라도 결과가 한 번 호출된 것과 같아야 합니다. (예: `결제_취소(주문ID=123)`를 두 번 보내도 두 번 환불되면 안 됨)
- **가시성(Isolation) 부재 대책**: Saga는 중간 상태가 외부에 노출됩니다. 결제는 완료되었으나 배송이 취소되기 직전(보상 트랜잭션 실행 전)의 찰나에 사용자가 화면을 새로고침하면 "결제 완료"로 보일 수 있습니다. 이를 막기 위해 상태값에 "결제_확인중(Pending)" 등의 중간 상태(Semantic Lock)를 비즈니스 로직으로 구현해야 합니다.

### 3. 안티패턴
- **외부 SaaS API에 2PC 적용 시도**: 외부의 결제 대행사(PG) API나 메일 발송 API를 2PC 글로벌 트랜잭션에 묶으려 하는 행위. 외부 시스템은 우리의 Prepare 요청에 맞춰 자신의 DB에 락을 걸어주지 않습니다. 외부 연동은 무조건 Saga(보상 트랜잭션)로 접근해야 합니다.

- **📢 섹션 요약 비유**: 모든 것에 2PC를 쓰는 것은 집 앞 편의점에 가면서도 장갑차를 타는 격이며, 모든 것에 Saga를 쓰는 것은 중요한 현금 다발을 그물망 가방에 넣고 다니는 것과 같습니다. 비즈니스 성격에 맞는 도구 선택이 아키텍트의 역량입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과

| 구분 | 글로벌 2PC 강제 적용 시 | 도메인별 Saga 패턴 전환 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 피크 타임 분산 락 대기로 TPS 수십 수준 | 로컬 트랜잭션 단위 비동기 처리 | TPS **수천~수만 단위 확장 (100배+)** |
| **정량** | 타 서비스 지연이 내 서비스 타임아웃 유발 | MSA 간 완벽한 장애 격리 | 시스템 전체 장애 파급율 **0%** |
| **정성** | DB 스키마 분리가 불가능하여 모놀리식 고착 | DB/서비스별 독립적 배포 가능 | 진정한 마이크로서비스 조직 민첩성 확보 |

### 2. 결론
분산 트랜잭션은 시스템 설계에서 피할 수 있다면 최대한 피하는 것이 상책입니다 (트랜잭션 바운더리 최적화). 그러나 피할 수 없는 비즈니스 요구사항 앞에서는, 2PC의 동기식 강한 일관성과 Saga 패턴의 비동기식 결과적 일관성 사이에서 줄타기를 해야 합니다. 현대 클라우드 네이티브 환경에서는 성능과 가용성을 위해 **Saga 패턴과 보상 트랜잭션**이 실질적인 표준으로 자리 잡았으며, 이를 뒷받침하기 위해 **아웃박스 패턴(Outbox Pattern)**과 메시지 브로커(Kafka)의 결합 설계가 필수적으로 동반되어야 합니다.

- **📢 섹션 요약 비유**: 분산 트랜잭션 아키텍처는 과거의 '철통같은 중앙 집권 통제(2PC)'에서, 현대의 '각자의 책임과 약속(보상)을 믿는 자율적 시민 사회(Saga)'로 진화하고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **ACID 원칙**: 분산 환경에서 완벽히 달성하기 매우 어려운 트랜잭션의 이상적인 4원칙.
- **CAP 정리 (CAP Theorem)**: 2PC(일관성, C 선호)와 Saga(가용성, A 선호)의 근본적인 트레이드오프를 설명하는 분산 시스템 이론.
- **폴리글랏 퍼시스턴스 (Polyglot Persistence)**: 서비스마다 RDB, NoSQL 등 각기 다른 DB를 쓰게 되면서 2PC 적용이 불가능해지고 Saga가 강제되는 배경.

## 👶 어린이를 위한 3줄 비유 설명
1. **분산 트랜잭션**은 친구 3명이 동시에 온라인 게임에서 교환 버튼을 눌러야만 아이템이 바뀌고, 한 명이라도 튕기면 전부 원래대로 돌아가야 하는 규칙이에요.
2. **2PC**는 방장이 "준비됐어?" 묻고 다 "응!" 하면 "그럼 교환!" 하고 외치는 방식인데, 방장이 갑자기 화장실에 가면 다들 버튼을 누른 채 무작정 기다려야(Blocking) 해요.
3. **Saga 패턴**은 일단 각자 알아서 교환을 시작하고, 만약 마지막 친구가 안 된다고 하면 이미 교환한 친구들이 "아쉽다, 다시 돌려주자(보상 트랜잭션)"라며 스스로 되돌리는 똑똑한 방법이에요!
