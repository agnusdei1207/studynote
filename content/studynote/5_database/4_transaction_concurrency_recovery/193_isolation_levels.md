# 193. 격리 수준 (Isolation Level) - 병행 제어의 정합성 단계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 여러 트랜잭션이 동시에 실행될 때, 한 트랜잭션이 변경한 데이터를 다른 트랜잭션에 어느 정도까지 노출할 것인지 결정하는 **데이터 고립의 강도** 설정이다.
> 2. **상충 관계**: 격리 수준이 높을수록 데이터 정합성은 완벽해지지만 시스템의 동시 처리 성능(Concurrency)은 저하되며, 낮을수록 속도는 빨라지나 **Dirty Read, Non-Repeatable Read, Phantom Read**와 같은 이상 현상이 발생한다.
> 3. **가치**: 비즈니스의 중요도(금융 정산 vs 단순 게시글 조회)에 따라 성능과 정합성 사이의 최적 균형점(Trade-off)을 선택할 수 있는 아키텍처적 유연성을 제공한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**격리 수준 (Isolation Level)**은 트랜잭션 간의 '벽'의 높이입니다. 벽이 낮으면 옆방(다른 트랜잭션) 소리가 다 들리지만 이동이 편하고, 벽이 높으면 조용(정합성)하지만 방을 드나들기가 매우 까다로워집니다.

### 💡 비유: 영화관 매표소의 줄 서기
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        격리 수준 단계별 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. Read Uncommitted (난장판): 옆 사람이 돈을 꺼내기도 전에 내 표를 끊음.         │
│     - 그 사람이 갑자기 표 안 산다(Rollback)고 하면, 내 표는 가짜가 됨! (Dirty Read)│
│                                                                             │
│  2. Read Committed (기본 예의): 옆 사람이 결제를 끝낸(Commit) 것만 봄.           │
│     - 하지만 내가 다시 확인했을 때 그 사람이 표를 또 샀으면 수량이 달라짐.          │
│                                                                             │
│  3. Repeatable Read (철저함): 내가 확인하는 동안은 남이 표를 못 고치게 함.        │
│     - 근데 내 눈을 피해 구석에서 새로 표를 산 사람은 못 막음! (Phantom Read)      │
│                                                                             │
│  4. Serializable (완벽주의): 매표소에 한 번에 딱 한 명만 들어오게 함.             │
│     - 가장 정확하지만, 줄이 엄청나게 길어짐 (성능 최악).                          │
│                                                                             │
│  → **'정확함'**을 위해 **'기다림(성능)'**을 얼마나 견딜 수 있는가의 문제입니다!      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **동시성의 딜레마**: 모든 작업을 순차적으로 처리하면 데이터는 완벽하지만 시스템은 너무 느림.
- **이상 현상의 제어**: 병행 처리 시 발생하는 논리적 오류들을 단계별로 정의하고, 필요한 만큼만 차단하기 위함.

- **📢 섹션 요약 비유**: 격리 수준은 '데이터 보안 등급'과 같아서, 등급이 올라갈수록 비밀(데이터 일관성)은 잘 지켜지지만 절차는 복잡해집니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. ANSI/ISO 표준 격리 수준과 발생 가능한 이상 현상

| 격리 수준 | Dirty Read | Non-Repeatable Read | Phantom Read | 비유 |
|:---:|:---: |:---: |:---: |:---|
| **Read Uncommitted** | 발생 | 발생 | 발생 | 무법 지대 |
| **Read Committed** | **방지** | 발생 | 발생 | 신뢰의 시작 |
| **Repeatable Read** | **방지** | **방지** | 발생 | 철저한 감시 |
| **Serializable** | **방지** | **방지** | **방지** | 완벽한 격리 |

### 2. 주요 이상 현상(Anomalies) 도해 (ASCII)

데이터 정합성이 깨지는 3대 시나리오를 시각화한다.

```text
    (1) Dirty Read (오염된 읽기)
    T1: Update A=100 (Uncommitted) ──▶ T2: Read A=100
    T1: ROLLBACK (A는 다시 0)       ──▶ T2가 읽은 100은 '가짜'!
    
    (2) Non-Repeatable Read (불일치 읽기)
    T1: Read A=50 ──▶ T2: Update A=100 & COMMIT ──▶ T1: Read A 다시 해보니 100!
    * 한 트랜잭션 내에서 같은 값을 두 번 읽었는데 결과가 다름.
    
    (3) Phantom Read (유령 읽기)
    T1: Count(*) = 10 ──▶ T2: Insert new row & COMMIT ──▶ T1: Count(*) 다시 보니 11!
    * 없던 데이터가 유령처럼 나타남.
```

### 3. 심층 동작 원리: 스냅샷 읽기 (MVCC)
- **원리**: 많은 DBMS(Oracle, MySQL InnoDB)는 Lock 대신 데이터의 과거 버전을 보관하는 **Undo 세그먼트**를 활용함.
- **작동**: 쿼리 시작 시점의 타임스탬프를 기준으로, 그 이후에 변경된 데이터는 무시하고 과거 데이터를 읽어 격리성을 구현함.

### 4. 핵심 기술: Next-Key Lock (MySQL 특화)
- `Repeatable Read` 수준임에도 불구하고, 인덱스 레인지에 락을 걸어 `Phantom Read`까지 방지하는 MySQL 만의 독특한 최적화 기술.

- **📢 섹션 요약 비유**: MVCC는 '타임머신'과 같아서, 현재 데이터가 바뀌고 있어도 나는 내가 일을 시작했을 때의 '과거 사진'을 보고 평화롭게 작업하는 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 비즈니스 요건에 따른 격리 수준 선택 가이드

| 비즈니스 성격 | 권장 격리 수준 | 이유 |
|:---:|:---|:---|
| **은행 송금, 결제** | **Serializable** 또는 강력한 Lock | 1원의 오차도 허용 불가. |
| **일반 ERP, 재고 관리** | **Repeatable Read** | 자신이 보고 있는 데이터의 일관성 유지 필요. |
| **게시판, SNS, 일반 웹** | **Read Committed (표준)** | 동시 접속자가 많으므로 성능 중시. |
| **로그 수집, 통계 엔진** | **Read Uncommitted** | 데이터가 조금 틀려도 속도가 최우선. |

### 2. 과목 융합 관점: 소프트웨어 공학 및 운영체제
- **소프트웨어 공학(SE) 융합**: 격리 수준은 **동시성 디자인 패턴**의 전략 선택과 같으며, 낙관적 락(Optimistic)과 비관적 락(Pessimistic)의 선택 기준이 된다.
- **운영체제(OS) 융합**: 격리 수준의 구현은 OS의 **판독자-기록자 문제(Readers-Writers Problem)**를 해결하는 세마포어 알고리즘의 변형이며, 자원 점유 시간(Holding Time)을 제어하여 시스템 데드락 발생 확률을 조율한다.

- **📢 섹션 요약 비유**: 격리 수준 선택은 '방음 공사'와 같아서, 도서관(금융)은 완벽한 방음이 필요하고 광장(SNS)은 소음이 섞여도 상관없습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: Oracle(Read Committed)에서 Phantom Read 방어
- **상황**: 오라클의 기본값은 Read Committed라 유령 읽기가 발생할 수 있음.
- **판단**: 중요한 통계 쿼리 도중 데이터가 추가되면 합계가 틀려짐.
- **결단**: **`SET TRANSACTION ISOLATION LEVEL SERIALIZABLE` 명시**. 쿼리 수행 동안 전체 데이터셋의 스냅샷을 고정하여, 외부의 INSERT로부터 완벽히 보호된 정적 결과 도출.

### 2. 안티패턴: 모든 서비스에 SERIALIZABLE 강제
- **증상**: "안전한 게 최고"라며 전사 공통 격리 수준을 최고 단계로 설정.
- **결과**: 트랜잭션 하나가 길어질 때마다 다른 모든 사용자가 대기하며 `ORA-08177: can't serialize access` 에러와 함께 서비스 마비.
- **조치**: **기본은 Read Committed 로 유지**하되, 정말 정교한 작업이 필요한 특정 로직에만 높은 격리 수준이나 `SELECT ... FOR UPDATE` 구문을 부분적으로 적용할 것.

### 📢 기술사적 결언
> "격리 수준은 데이터베이스의 **'사회적 거리두기'**다. 트랜잭션들이 서로에게 미치는 영향 범위를 설정하기 때문이다. 기술사는 단순한 이론적 정의를 넘어, **사용하는 DBMS 제품의 기본 격리 수준**이 무엇이며(Oracle: RC, MySQL: RR) 그 엔진이 이상 현상을 어떻게 물리적으로 방어하는지 꿰뚫어야 한다. 성능(Performance)과 신뢰(Correctness)라는 두 마리 토끼 사이에서 비즈니스 가치를 수호하는 **'최적의 타협가'**가 되어야 함을 잊지 말아야 한다."

- **📢 섹션 요약 비유**: 격리 수준을 무시하는 개발은 '옆 사람의 일기장을 훔쳐보며 내 일기를 쓰는 것'과 같아, 결국 남의 실수까지 내 기록에 남게 됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (격리 수준 최적화 시)

| 지표 | 활용 미흡 시 | 최적 활용 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 정합성 사고** | 잦은 논리 오류 | **0% (수준별 통제)** | 데이터 품질 신뢰도 극대화 |
| **시스템 응답 속도** | 병목 심함 | **최적 동시성 확보** | 서비스 처리량 2~3배 향상 |
| **개발 복잡도** | 수동 락 관리 | **선언적 격리 활용** | 코드 가독성 및 생산성 증대 |

### 2. 미래 전망
- **AI-Adaptive Isolation**: 쿼리의 중요도와 실시간 시스템 부하를 분석하여, 트랜잭션마다 최적의 격리 수준을 자동으로 할당하는 **지능형 격리 제어**.
- **Distributed Consistent Snapshots**: 전 세계에 분산된 노드 간에도 물리적 시차 없이 동일한 격리 수준을 보장하는 **글로벌 타임스탬프(TrueTime)** 기술 고도화.

- **📢 섹션 요약 비유**: 미래의 격리 수준은 우리가 벽을 쌓지 않아도, 데이터들이 스스로 서로의 영역을 존중하며 평화롭게 공존하는 '투명한 고립' 기술로 진화할 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **격리 수준**은 친구들과 한 교실에서 공부할 때, **"서로의 공부를 방해하지 않게 칸막이를 세우는 것"**과 같아요.
2. 칸막이가 낮으면 친구가 연필을 깎는 소리가 들리지만(데이터 변화), 칸막이가 아주 높으면 **나 혼자 방에 있는 것처럼** 집중할 수 있죠.
3. 이 칸막이 높이를 잘 조절해야 친구들과 **사이좋게 지내면서도 공부(데이터 처리)**를 정확하게 끝낼 수 있답니다!
