# 240. 그림자 페이징 (Shadow Paging) 기법 - 로그 없는 복구의 예술

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 그림자 페이징(Shadow Paging)은 트랜잭션의 원자성(Atomicity)을 보장하기 위해 **원본 페이지 테이블(Shadow Table)**을 보존하고, 변경 사항은 **현 페이지 테이블(Current Table)**이 가리키는 새로운 디스크 공간에 기록하여, 커밋 시점에 포인터만 교체하는 로그 없는 회복(Log-less Recovery) 기법이다.
> 2. **가치**: 로그 파일(Log Sequential File) 관리에 따른 I/O 오버헤드와 버퍼 관리 복잡도를 제거하며, 장애 발생 시 복잡한 로그 분석(Redo/Undo) 없이 포인터만 원복하면 되므로 복구 시간이 극단적으로 단축된다.
> 3. **융합**: 현대 컴퓨팅의 **Copy-On-Write (COW)**, **SSD (Solid State Drive)**의磨损 레벨링, 그리고 **ZFS**나 **Btrfs** 같은 파일 시스템 스냅샷(Snapshot) 기술의 이론적 기초로 사용되며, NoSQL이나 임베이드 데이터베이스의 효율적인 저장소 구조로 응용된다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념 및 정의
**그림자 페이징(Shadow Paging)**은 데이터베이스 회복(Recovery) 기법의 일종으로, 트랜잭션 수행 중 데이터의 변경을 직접 원본에 반영하지 않고, **새로운 페이지**에 기록함으로써 **이전 상태(Shadow)를 보존**하는 방식입니다. 기존의 로그 기반 회복(Log-based Recovery) 방식이 데이터 변경 전후의 이미지를 로그 파일에 기록하는 반면, 그림자 페이징은 **물리적 디스크 블록**의 할당 정책을 통해 회복성을 확보합니다. 이는 데이터베이스의 일관성을 유지하면서도 로그 관리 시스템(Log Manager)의 부담을 델 수 있는 대안이 됩니다.

#### 등장 배경
1.  **기존 로그 방식의 한계**: 전통적인 **Undo/Redo Logging**은 트랜잭션마다 로그 레코드를 작성하고, 주기적으로 **FLUSH** 연산을 수행해야 하므로 쓰기 증폭(Write Amplification) 현상이 발생하고 병목이 생길 수 있습니다.
2.  **물리적 스냅샷의 필요성**: 메모리의 페이지 테이블(Page Table) 개념을 디스크 관리로 확장하여, 트랜잭션 단위로 디스크 상태의 스냅샷을 물리적으로 생성하고자 하는 패러다임이 등장했습니다.
3.  **빠른 회복 요구**: 금융 거래나 실시간 시스템과 같이 장애 발생 후 즉각적인 서비스 재개(RTO: Recovery Time Objective)가 중요한 환경에서, 로그를 분석하여 재실행하는 시간을 줄이고자 하는 요구가 있었습니다.

#### 💡 핵심 비유
집을 리모델링할 때 **'사진 찍어두고 철거하기'**와 유사합니다. 리모델링(트랜잭션)을 시작하기 전 현재 상태의 사진(Shadow Page Table)을 찍어둡니다. 벽을 허물거나 페인트를 칠하는 등의 변경(데이터 수정)은 바닥에 새로 종이를 깔아서(새 페이지) 그립니다. 만약 마음에 들면 새 종이를 바닥에 붙이고(Commit), 마음에 안 들면 찍어둔 사진을 보고 원래대로 되돌리면(Abort) 되므로, 일일이 "어떤 벽을 어떻게 바꿨다"는 로그를 남길 필요가 없습니다.

#### 📢 섹션 요약 비유
그림자 페이징은 마치 **'문서 작성 시 '수정 추적' 대신 '파일 다른 이름으로 저장'을 사용하는 것'**과 같습니다. 원본 문서(Shadow)는 건드리지 않고 복사본(Current)에서 마음껏 수정하다가, 만족하면 복사본을 원본으로 덮어쓰고, 실패하면 복사본을 삭제하면 되기 때문입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 구성 요소 상세 분석
그림자 페이징은 운영체제의 가상 메모리 관리와 유사한 구조를 디스크에 구현합니다. 주요 구성 요소는 다음과 같습니다.

| 요소명 | 역할 및 내부 동작 | 프로토콜/특징 | 비유 |
|:---|:---|:---|:---|
| **Shadow Page Table** | 트랜잭션 시작 시점의 데이터베이스 상태를 가리키는 **원본 페이지 테이블**. 변경되지 않음 | 읽기 전용(Read-Only) | 마스터 원본 도면 |
| **Current Page Table** | 트랜잭션 수행 중 발생하는 모든 변경 사항을 가리키는 **작업용 페이지 테이블** | 읽기/쓰기 가능(RW) | 작업용 스케치북 |
| **Disk Pages (DB)** | 실제 데이터가 저장된 물리적 디스크 블록. 변경 시 새로운 페이지 할당 필요 | Random Access I/O | 건물의 실제 벽면 |
| **Master Record** | 디스크 상의 고정된 위치(주로 섹터 0)에 존재하며, 현재 유효한 페이지 테이블의 주소를 저장하는 **루트 포인터** | Atomic Update | 최신 도면의 위치를 알려주는 게시판 |

**다이어그램 해설**:
1.  **구조**: **Current Page Table**과 **Shadow Page Table**은 완전히 분리된 두 개의 페이지 디렉토리입니다.
2.  **변경 (Update)**: 트랜잭션이 페이지 4(P4)의 데이터(D4)를 변경하고자 할 때, 시스템은 새로운 빈 디스크 블록을 할당하고 그곳에 수정된 데이터 D4'를 기록합니다.
3.  **포인터 조작**: **Current Page Table** 내의 P4 엔트리는 기존 0x4D(D4의 위치) 대신 새로운 블록 주소 0x99(D4'의 위치)를 가리키도록 업데이트됩니다. 반면, **Shadow Page Table**의 P4는 여전히 0x4D를 가리키며 원본 데이터 D4를 보호합니다.
4.  **비연속성**: 새로 할당된 페이지(D4')는 기존 데이터 영역과 물리적으로 떨어져 있을 수 있으므로, 트랜잭션이 커지면 데이터 파편화가 발생할 수 있습니다.

#### 심층 동작 원리 및 의사결정 과정
1.  **트랜잭션 시작 (Begin Tx)**:
    *   현재의 Master Record가 가리키는 페이지 테이블을 읽어들입니다.
    *   이를 Shadow로 지정하고, 이 페이지 테이블을 복사하여 Current Page Table을 생성합니다.
2.  **데이터 수정 (Update)**:
    *   변경하려는 레코드가 포함된 페이지 `p`를 메모리(Buffers)로 가져옵니다.
    *   `p`에 대한 변경 사항을 기록하기 위해 **새로운 디스크 페이지 `p'`를 할당**합니다.
    *   `p`의 내용을 `p'`로 복사한 뒤, 메모리 상에서 `p'`를 수정합니다.
    *   **Current Page Table**에서 `p`의 주소를 `p'`의 주소로 갱신합니다. (Shadow Table은 변화 없음)
3.  **커밋 (Commit)**:
    *   **Current Page Table**을 디스크에 안전하게 기록합니다.
    *   가장 중요한 단계: **Master Record**의 포인터를 Shadow 주소에서 Current 주소로 **원자적(Atomic)**으로 변경합니다. (보통 단일 디스크 블록 쓰기 작업이므로 원자성 보장이 쉬움)
    *   이제 Current가 새로운 Shadow가 되며, 이전 Shadow는 가비지 컬렉션(Garbage Collection) 대상이 됩니다.
4.  **롤백 (Abort)**:
    *   아무런 일도 하지 않습니다. 단순히 메모리상의 Current Page Table을 해제하고, Master Record가 계속 가리키고 있는 기존의 Shadow Table을 그대로 사용하면 됩니다.

#### 핵심 알고리즘 스니펫 (Pseudo Code)
```python
def shadow_paging_write(page_id, new_data):
    # 1. 현재 페이지 테이블에서 해당 페이지가 메모리에 있는지 확인
    if page_id not in current_table.pages:
        old_page_addr = shadow_table.pages[page_id]
        # 2. 새로운 페이지 할당 및 복사
        new_page_addr = allocate_disk_page()
        disk_write(new_page_addr, disk_read(old_page_addr)) 
    
    # 3. 메모리 상에서 수정 및 테이블 갱신
    current_table.pages[page_id].data = new_data
    # Current Table만 새 주소를 가리킴
    current_table.pages[page_id].addr = new_page_addr 

def commit_transaction():
    # 1. Current Table 디스크에 기록
    flush_to_disk(current_table)
    
    # 2. Master Record 교체 (Atomic Operation)
    update_master_record(current_table.address)
    # 성공: 현재 트랜잭션의 변경 사항이 공식화됨
```

#### 📢 섹션 요약 비유
그림자 페이징의 수정 및 커밋 과정은 **'넷플릭스의 프로필 관리'**와 같습니다. 내가 시청 기록(데이터)을 바꾸려면 "키즈 프로필(Shadow)"을 그대로 둔 채, "나의 프로필(Current)"을 복사해서 만들어 그 안에서 수정합니다. 수정을 마치고 저장(Commit)하면 시스템은 메인 화면의 '나의 프로필' 포인터를 방금 수정된 프로필로 바꿉니다. 도중에 그만둬도 원본 '키즈 프로필'에는 아무런 영향이 없습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교: Shadow Paging vs. Log-Based Recovery
그림자 페이징은 로그 기반 방식(ARIES 등)과 구조적으로 근본적인 차이를 가집니다.

| 비교 항목 | 그림자 페이징 (Shadow Paging) | 로그 기반 회복 (Log-Based Recovery) |
|:---|:---|:---|
| **저장 공간 관리** | **할당(Allocation) 중심**: 수정 시 새 페이지를 항상 할당. <br>남은 공간(Free Space)이 계속 필요. | **재사용(In-place) 중심**: 원본 위치를 Overwrite 가능. <br>공간 효율적이지만 로그 공간 별도 필요. |
| **I/O 패턴** | **Random Write**: 새 페이지가 디스크 곳곳에 흩어지므로 <br>Seek Time이 증가하여 쓰기 성능 저하 가능. | **Sequential Write**: 로그는 연속적으로 기록되므로 <br>Sequential I/O의 이점을 봄. |
| **장애 복구 속도** |
