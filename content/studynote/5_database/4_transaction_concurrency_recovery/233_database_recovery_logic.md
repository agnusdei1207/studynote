# 233. 회복 (Recovery) - 데이터의 불사조 정신

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 회복(Recovery)은 트랜잭션(Transaction)의 **원자성(Atomicity)**과 **영속성(Durability)**을 보장하는 DBMS의 핵심 방어 메커니즘으로, 장애 발생 시 데이터베이스를 일관된 상태(Consistent State)로 복원하는 절차입니다.
> 2. **가치**: "Redo(재수행)"를 통해 커밋된 데이터의 소실을 막고, "Undo(취소)"를 통해 미완료 트랜잭션의 오염을 방지함으로써, 시스템 crash나 미디어 failure 상황에서도 데이터 무결성을 100% 유지합니다.
> 3. **융합**: 운영체제(OS)의 버퍼 관리 기법과 파일 시스템의 저널링(Journaling) 기술이 융합되어 WAL(Write-Ahead Logging) 프로토콜로 발전했으며, 이는 분산 데이터베이스의 고가용성(High Availability) 기초가 됩니다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 정의**
회복(Recovery)이란 트랜잭션 실행 중 장애(Failure)가 발생했을 때, 데이터베이스를 장애 발생 직전의 **일관성 있는 상태(Consistent State)**로 복원하는 모든 과정을 의미합니다. 단순히 데이터를 다시 불러오는 것이 아니라, "완료되지 않은 일은 없었던 것처럼(Undo)" 하고 "완료된 일은 확실하게(Redo)" 처리하는 ACID 특성의 최후 보루입니다.

**2. 등장 배경 및 필요성**
① **기존 한계**: 메모리(Main Memory)는 휘발성이므로 전원 차단 시 데이터가 증발하며, 디스크(Disk)조차 물리적 손상에 취약합니다.
② **혁신적 패러다임**: 모든 변경 사항을 '변경 불가능한 기록(Immutable Log)'으로 먼저 남겨두는 **선형 로깅(Linear Logging)** 방식이 도입되었습니다.
③ **비즈니스 요구**: 금융권, 항공 예약 등 1초의 다운타임도 허용되지 않는 미션 크리티컬(Mission-Critical) 환경에서 **RTO(Recovery Time Objective)**와 **RPO(Recovery Point Objective)**를 최소화해야 했습니다.

**3. 작동 기본 원칙**
회복 관리자(Recovery Manager)는 트랜잭션의 결과를 판단하기 위해 **LCR(Log Sequence Number)**과 **LBA(Logical Block Address)** 등의 메타데이터를 활용하며, 크게 두 가지 기본 연산을 수행합니다.

*   **Redo (재수행)**: 커밋(Commit)되었으나 장애로 인해 디스크에 반영되지 않은 변경 사항을 로그를 읽어 다시 재현합니다.
*   **Undo (취소)**: 커밋되지 않고 중단된 트랜잭션의 변경 사항을 로그를 통해 역추적하여 원래 값으로 복구합니다.

> **📢 섹션 요약 비유**: 회복 관리는 **'화재 진화 및 복구 시스템'**과 같습니다. 불이 난 후(장애 발생), 타버린 건물은 최근 찍어둔 청사진(로그)을 보고 다시 짓고(Redo), 공사 중이던 미완성 건물은 안전을 위해 철거해버리는(Undo) 작업입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. 구성 요소 상세 분석**
회복 시스템은 메모리, 디스크, 안전 저장소의 계층적 구조로 이루어집니다. 아래 표는 각 구성 요소의 역할과 내부 동작을 심층 분석한 것입니다.

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 (Internal Operation) | 프로토콜/특징 |
|:---|:---|:---|:---|
| **Log Buffer** (로그 버퍼) | 변경 로그를 메모리에 모아두는 공간 | 트랜잭션 발생 시 즉시 기록 (Force-Log at Commit) | FIFO 순서, 휘발성 |
| **Log File** (로그 파일) | 디스크 상의 영구적 기록 저장소 | WAL(Write-Ahead Logging) 원칙에 따라 DB보다 먼저 기록됨 | Append-Only |
| **Checkpoint** (체크포인트) | 복구 속도를 높이기 위한 동기화 지점 | 특정 시점의 메모리 상태(Dirty Page)를 디스크로 Flush | 주기적 실행 |
| **Recovery Manager** (회복 관리자) | 장애 시 복구 수행 주체 | ARIES 알고리즘 등을 통해 Analysis, Redo, Undo 수행 | 시스템 시작 시 자동 호출 |

**2. 회복 시스템의 논리적 아키텍처**
아래 다이어그램은 트랜잭션의 데이터 흐름과 장애 발생 시 회복 관리자가 로그를 어떻게 활용하는지를 도식화한 것입니다.

```text
[DBMS 회복 아키텍처 및 데이터 흐름도]

   ┌──────────────────────────────────────────────────────────────────┐
   │                        시스템 영역 (Volatile)                     │
   │  ┌──────────────┐       ┌──────────────────┐                     │
   │  │   Client     │──────▶│ Transaction Mgr  │                     │
   │  └──────────────┘       └────────┬─────────┘                     │
   │                                  │                               │
   │                         [Write Request]                          │
   │                                  ▼                               │
   │  ┌──────────────────────────────────────────────────────┐       │
   │  │                Buffer Pool (Data Cache)               │       │
   │  │  ┌──────────────────────┐      ┌──────────────────┐  │       │
   │  │  │ Data Page (Dirty)    │      │ Log Buffer       │  │       │
   │  │  │ (Update before Flush)│◀─────│ (Seq# 105)       │  │       │
   │  │  └──────────────────────┘      └──────┬───────────┘  │       │
   │  └─────────────────────────────────────┼───────────────┘       │
   │                                        │ Log Flush             │
   └────────────────────────────────────────┼───────────────────────┘
                                           │ WAL Protocol
                                           ▼
   ┌──────────────────────────────────────────────────────────────────┐
   │                     비휘발성 저장소 (Stable Storage)              │
   │                                                                   │
   │   ┌──────────────────────┐         ┌──────────────────────┐     │
   │   │   Log File (Disk)    │         │  Data File (Disk)    │     │
   │   │ ◀───[Append Only]─── │         │  (DB Persistent)     │     │
   │   │                      │         │                      │     │
   │   │  LSN 1 | LSN 2 | ... │         │  Page A | Page B |   │     │
   │   │  (REDO/Undo Source)  │         │  (Asynchronous)      │     │
   │   └──────────────────────┘         └──────────────────────┘     │
   │                                                                   │
   │   [장애 발생 시: Recovery Manager는 Log File을 읽어 Data File 복구]│
   └──────────────────────────────────────────────────────────────────┘
```

**다이어그램 해설**:
1.  **트랜잭션 요청**: 사용자의 데이터 변경 요청은 먼저 버퍼 풀(Buffer Pool)의 데이터 페이지를 메모리 상에서 수정합니다.
2.  **로그 기록 (WAL)**: 데이터 페이지가 디스크로 Flush되기 **전에**, 변경 사항을 기록한 로그(Log)가 반드시 디스크의 로그 파일에 기록됩니다. 이를 **WAL (Write-Ahead Logging)**이라 합니다.
3.  **비동기 Flush**: 실제 데이터 파일(Data File)에 대한 수정은 Checkpoint나 필요 시점에 비동기적으로 일어납니다.
4.  **장애 복구**: 시스템이 죽으면(Go down), 재부팅 시 회복 관리자는 가장 최신의 로그 파일을 스캔하여(Analysis), 커밋된 트랜잭션은 Redo하고, 미완료된 트랜잭션은 Undo하여 일관성을 복구합니다.

**3. 심층 동작 원리: WAL과 ARIES**
현대 DBMS는 주로 **ARIES (Algorithms for Recovery and Isolation Exploiting Semantics)** 알고리즘을 사용합니다.
*   **Analysis Phase**: 로그를 스캔하여 Checkpoint 이후 커밋된 트랜잭션과 미완료 트랜잭션을 식별합니다.
*   **Redo Phase**: LSN(Log Sequence Number)을 기준으로 Checkpoint 이후부터 로그의 끝까지, 커밋 여부와 상관없이 모든 변경을 디스크에 재반영합니다(물리적 Redo).
*   **Undo Phase**: 커밋되지 않은 트랜잭션을 로그의 역순으로 Undo하여 변경 사항을 취소합니다.

```sql
-- [실무 로그 레코드 예시: Logical Log]
-- Begin Transaction (TX_ID: 1001)
LSN: 5001 | Type: Begin | TxID: 1001
-- Update Operation
LSN: 5002 | Type: Update | TxID: 1001 | Page: 105 | BeforeVal: 'A' | AfterVal: 'B'
-- Commit
LSN: 5003 | Type: Commit | TxID: 1001
```
> 📌 **코드 해설**: `LSN 5002`는 `Page 105`를 `A`에서 `B`로 변경했다는 기록입니다. 장애 발생 시 `5002`가 디스크에 안착하지 않았다면, 이를 다시 실행(Redo)하여 `B`로 만들어야 합니다.

> **📢 섹션 요약 비유**: 이 과정은 **'작가의 집필 및 복구 과정'**과 같습니다. 작가는 원고를 바로 책으로 출판하지 않고, 매일 날짜별로 수정한 내용을 사본(로그)으로 보관해둡니다. 컴퓨터가 고장 나면, 어제까지 완성된 책(Commit)을 다시 인쇄하고(Redo), 오늘 작성하던 미완성 원고(Undo)는 폐기하여, 책의 내용이 오염되지 않게 합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 핵심 회복 기법 심층 비교**
회복 기법은 로그를 언제 디스크에 반영하느냐와 데이터를 어떻게 보관하느냐에 따라 크게 나뉩니다. 아래는 이를 정량적/구조적으로 분석한 표입니다.

| 구분 | 지연 갱신 (Deferred Update) | 즉시 갱신 (Immediate Update) | 그림자 페이징 (Shadow Paging) |
|:---|:---|:---|:---|
| **작동 방식** | 트랜잭션 완료(Commit) 시점에 DB 변경 | 트랜잭션 수행 중 실시간으로 DB 변경 (로그 기반) | 데이터 페이지 복제본(Shadow) 생성 후 스왑 |
| **Undo 필요성** | **불필요** (Commit 전 DB 변경 없음) | **필수** (Crash 전 변경된 것을 롤백) | **불필요** (원본 보존됨) |
| **Redo 필요성** | **필수** (Commit 시 변경만 반영하므로) | **필수** (메모리에만 있던 로그 재반영) | **불필요** (Commit 시 포인터만 변경) |
| **성능 특성** | 단순하지만 대용량 트랜잭션 시 메모리 부담 | 가장 일반적, 높은 동시성 제어 가능 | 오버헤드 커서 분산 DB 환경 등에 유리 |
| **대표 예시** | 연구용 prototype | **Oracle, MySQL(InnoDB)** 등 상용 DBMS | **Git** (Blob 관리), 일일 Old DBMS |

**2. Checkpoint의 성능 및 기술적 영향**
**Checkpoint**는 시스템 장애 발생 시 복구해야 할 로그의 양을 결정하는 성능의 핵심 요소입니다.

*   **Fuzzy Checkpoint (퍼지 체크포인트)**: DB 전체를 멈추지 않고, 페이지 단위로 Flush를 수행하여 시스템 가용성을 높이는 기술입니다. (OS의 Fork나 Copy-on-Write 기술과 연계)
*   **Performance Trade-off**:
    *   Checkpoint 주기가 **짧을 때**: 복구 시간(Recovery Time)은 짧아지지만, 잦은 디스크 쓰기(I/O)로 인한 운영 성능(Run-time Performance) 저하 발생.
    *   Checkpoint 주기가 **길 때**: 운영 시 성능은 높아지지만, 장애 시 처리해야 할 로그 양이 많아져 복구 시간이 급격히 증가.

**3. 과목 융합 관점 (OS & Data Structure)**
*   **OS와의 융합**: DBMS의 **WAL (Write-Ahead Logging)**은 OS 파일 시스템의 **저널링(Journaling)**과 동일한 원리입니다. `fsync()` 시스템 콜을 통해 버퍼 캐시 내용을 디스크로 강제로 내리는 과정이 회복의 핵심입니다.
*   **자료구조와의 융합**: 로그 파일의 무결성을 위해 **LSN (Log Sequence Number)**이라는 단조 증가하는 논리적 타임스탬프를 사용하며, 이는 동시성 제어에서 발생하는 교착상태(Deadlock) 해결 순서 결정에도 활용됩니다.

> **📢 섹션 요약 비유**: 이는 **'게임 세이브 슬롯'** 전략과 같습니다. '그림자 페이징'은 '수정할 때마다 세이브 파일을 새로 만드는 방식'(용량 많이 먹음, 복구 빠름), '지연 갱신'은 '한 스테이지를 다 클리어해야 저장하는 방식', '체크포인트'는 '게임 중간에 자동 저장되는 포인트'입니다. 자동 저장이 너무 잦으면 게임이 버벅거리지만(성능 저하), 너무 드물면 죽었을 때 처음부터 다시 해야 합니다(복구 시간 증가).

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. 실무 시나리오: 서버 장애 및 미디어 복구**

| 시나리오 | 상황 묘사 | 기술적 의사결정 (Decision) |
|:---|:---|:---|
| **A. 셧다운 복구** | 정전으로 DBMS 강제 종료됨 (Memory Data Loss) | **Redo + Undo** 수행. WAL 로그를 확인하여
