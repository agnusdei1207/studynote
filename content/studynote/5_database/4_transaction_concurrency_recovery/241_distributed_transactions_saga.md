# 분산 트랜잭션의 난제 - 2단계 커밋(2PC)과 Saga 패턴의 대립

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 트랜잭션은 물리적으로 분리된 여러 데이터베이스(또는 마이크로서비스)에 걸쳐 실행되는 작업들의 **원자성(Atomicity, 모두 성공하거나 모두 실패함)**을 보장하기 위한 아키텍처적 조율 매커니즘이다.
> 2. **가치**: 전통적인 **2단계 커밋(2PC)**은 글로벌 코디네이터를 통해 완벽한 강한 일관성(Strong Consistency)을 보장하지만 치명적인 락킹(Locking) 병목을 유발한다. 이에 대한 대안으로 등장한 **사가(Saga) 패턴**은 비동기 이벤트와 보상 트랜잭션(Compensating Transaction)을 통해 가용성(Availability)과 성능을 극대화한다.
> 3. **융합**: 사가 패턴은 이벤트 소싱(Event Sourcing) 및 카프카(Kafka) 같은 분산 메시지 브로커와 완벽히 융합되어, 현대 MSA(Microservices Architecture) 환경에서 '결과적 일관성(Eventual Consistency)'을 수용하는 글로벌 표준 데이터 파이프라인으로 자리 잡았다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **분산 트랜잭션**: 2개 이상의 독립된 네트워크 노드나 DB에서 수행되는 트랜잭션 묶음.
  - **2PC (Two-Phase Commit)**: 중앙 관리자(Coordinator)가 모든 노드에게 준비(Prepare)를 묻고, 전원이 동의하면 커밋(Commit)을 지시하는 동기식 합의 프로토콜.
  - **Saga 패턴**: 중앙 통제 없이, 각 서비스가 자신의 로컬 트랜잭션을 실행하고 다음 서비스로 이벤트를 넘기며 릴레이식으로 작업을 이어가는 비동기 아키텍처 패턴.

- **💡 비유**: 2PC가 "결혼식장에서 주례(코디네이터)가 신랑과 신부에게 각각 동의를 묻고(Prepare), 둘 다 '네'라고 대답할 때까지 하객들을 꼼짝 못 하게 묶어둔 뒤 부부 선언(Commit)을 하는 것"이라면, Saga 패턴은 "택배 기사들이 각자 자기 구역만 배달하고 다음 기사에게 물건을 토스하는데, 중간에 물건이 파손되면 즉시 반품(보상 트랜잭션) 스티커를 붙여서 뒤로 되돌려 보내는 릴레이 시스템"입니다.

- **등장 배경**:
  1. **모놀리식(Monolithic)의 해체**: 과거에는 하나의 거대한 RDBMS 안에서 `BEGIN ~ COMMIT`으로 쉽게 원자성을 지켰으나, 시스템이 도메인별로 쪼개지면서(MSA) DB도 쪼개져 단일 트랜잭션 제어가 불가능해졌다.
  2. **2PC의 성능 파괴**: 분산된 DB를 2PC로 묶었더니, 하나의 노드가 네트워크 장애로 응답하지 않으면 나머지 99개의 노드가 락(Lock)을 쥔 채 영원히 대기(Blocking)하는 치명적인 연쇄 마비가 발생했다.
  3. **가용성(Availability)의 시대**: 아마존, 넷플릭스 등 빅테크는 "1명의 결제 오류를 막기 위해 99명의 결제를 대기시키는 것은 미친 짓"이라고 판단하고, 일관성(C)을 조금 포기하더라도 절대 멈추지 않는 가용성(A) 중심의 Saga 패턴을 창안해 냈다.

- **📢 섹션 요약 비유**: 10명이 묶여서 3각 4각 달리기를 하는 것(2PC)은 발만 맞으면 완벽하지만 한 명만 넘어져도 다 같이 쓰러집니다. 반면 바통 터치식 릴레이(Saga)는 각자 최고 속도로 뛰되, 바통을 놓치면 뛰었던 길을 되돌아가서 다시 주워오는(보상) 방식입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 2단계 커밋 (2PC, Two-Phase Commit)의 딜레마

글로벌 트랜잭션 매니저(GTM)가 주도하는 전통적 방식이다.

- **1단계 (Prepare Phase)**: GTM이 모든 DB 노드에게 "지금부터 데이터를 쓸 건데 준비됐니?"라고 묻는다. 각 노드는 락(X-Lock)을 걸고 Undo/Redo 로그를 기록한 뒤 "준비 완료(Ready)"를 응답한다.
- **2단계 (Commit Phase)**: 모든 노드로부터 "Ready"를 받으면 GTM은 "모두 Commit 해!"라고 지시한다. 단 한 노드라도 "No"나 "응답 없음"이 오면 전원에게 "Rollback 해!"라고 지시한다.
- **치명적 약점 (Blocking)**: 만약 1단계에서 노드들이 락을 걸고 "Ready"를 보냈는데, 갑자기 GTM(주례)이 심장마비로 죽어버리면? 노드들은 커밋할지 롤백할지 지시를 못 받아 영원히 락을 쥐고 멈춰버리는 **단일 장애점(SPOF) 지옥**에 빠진다.

---

### 2. Saga (사가) 패턴 - MSA의 구원자

Saga는 긴 트랜잭션을 여러 개의 짧은 '로컬 트랜잭션'으로 쪼개고, 이를 이벤트(Event) 메시지로 연결하는 방식이다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 Saga 패턴의 정상 흐름과 롤백(보상 트랜잭션) 흐름        │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 정상 흐름 (Happy Path) ]                                          │
  │  (사용자 주문)                                                      │
  │  1. [주문 DB] -> 주문 상태 'PENDING'으로 INSERT (로컬 Commit)          │
  │     └─(주문생성 이벤트 발행)─▶ Kafka                                  │
  │                                │                                 │
  │  2. [결제 DB] ◀─(이벤트 수신)──┘ 결제 승인 INSERT (로컬 Commit)        │
  │     └─(결제완료 이벤트 발행)─▶ Kafka                                  │
  │                                │                                 │
  │  3. [재고 DB] ◀─(이벤트 수신)──┘ 재고 차감 UPDATE (최종 완료)           │
  │                                                                  │
  │                                                                  │
  │  [ 에러 발생 시 보상 흐름 (Compensating Transaction) ]                  │
  │  3. [재고 DB] ◀─(이벤트 수신)──┘ 재고 부족 에러 발생! (ROLLBACK 불가)   │
  │     └─(재고부족 이벤트 발행)─▶ Kafka                                  │
  │                                │                                 │
  │  4. [결제 DB] ◀─(이벤트 수신)──┘ "결제 취소(환불)" 로컬 트랜잭션 수행!     │
  │     └─(환불완료 이벤트 발행)─▶ Kafka                                  │
  │                                │                                 │
  │  5. [주문 DB] ◀─(이벤트 수신)──┘ 주문 상태 'CANCELED'로 UPDATE 마감.  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Saga의 핵심은 **"이미 커밋된 남의 DB 트랜잭션을 강제로 롤백(Rollback)시킬 방법은 우주상에 존재하지 않는다"**는 현실을 인정하는 것이다. 결제 DB가 커밋을 쳤는데 3번째 단계인 재고 DB에서 에러가 나면, 결제 DB를 롤백시키는 게 아니라 **'결제 취소(환불)'라는 새로운 반대(보상) 트랜잭션을 실행**하여 논리적으로 원상 복구시키는 것이다. 각 단계는 락(Lock)을 쥐고 대기하지 않으므로 트래픽을 무한대로 소화할 수 있다.

---

### 3. Saga 패턴의 두 가지 구현 방식

1. **코레오그래피 (Choreography, 안무)**: 중앙 통제자 없이, 각 서비스가 이벤트를 발행하고 다른 서비스가 구독(Pub/Sub)하며 춤을 추듯 알아서 움직이는 방식. (단순하지만 서비스가 많아지면 이벤트 추적이 불가능해짐)
2. **오케스트레이션 (Orchestration, 지휘)**: 중앙에 'Saga Orchestrator(지휘자)'라는 별도 서비스를 두고, 이 지휘자가 "결제팀 결제해, 재고팀 차감해"라고 명시적으로 API를 호출하며 보상 로직까지 통제하는 방식. (복잡한 엔터프라이즈 환경의 표준)

- **📢 섹션 요약 비유**: 코레오그래피는 눈빛만 보고 패스하는 길거리 농구라면, 오케스트레이션은 감독(오케스트레이터)이 선수들의 멱살을 잡고 전술을 지시하는 프로 농구입니다. 서비스가 5개가 넘어가면 무조건 오케스트레이션을 써야 합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 트랜잭션 제어 아키텍처 비교

| 비교 항목 | 2단계 커밋 (2PC) | Saga 패턴 (보상 트랜잭션) |
|:---|:---|:---|
| **일관성 수준** | 강한 일관성 (Strong Consistency, ACID) | 결과적 일관성 (Eventual Consistency, BASE) |
| **락킹 (Locking)**| 글로벌 범위의 트랜잭션 락 유발 (병목 심각) | 로컬 DB 락만 존재 (대기 시간 0화) |
| **장애 복구** | DBMS 커널이 알아서 롤백 처리 | 개발자가 명시적인 '취소(보상) API' 코딩 필요 |
| **격리성 (Isolation)**| 완벽한 격리 보장 (오손 읽기 없음) | 격리성 깨짐. 중간 상태('결제됨-재고확인중')가 외부에 노출됨 |
| **사용 도메인** | 단일 망 내부의 엄격한 금융 코어 원장 | 글로벌 MSA 환경의 커머스, 숙박, 스트리밍 서비스 |

### 과목 융합 관점

- **이벤트 소싱(Event Sourcing)과의 융합**: Saga 패턴에서 이벤트를 잃어버리면(Message Loss) 영원히 환불을 받지 못하는 참사가 생긴다. 이를 방지하기 위해 데이터를 업데이트함과 동시에 이벤트를 발행하는 행위를 원자적으로 묶는 **트랜잭셔널 아웃박스(Transactional Outbox)** 패턴이나 **CDC (Change Data Capture)** 기술이 필수적으로 결합된다.
- **분산 시스템 이론 (CAP 정리)**: 2PC는 CAP 정리에서 네트워크 파티션(P)이 발생했을 때 가용성을 버리고 일관성(C)을 택한(CP 시스템) 대표적 사례이며, Saga는 일관성을 조금 늦추더라도 끝까지 서비스를 멈추지 않는 가용성(A)을 택한(AP 시스템) 현대의 타협안이다.

- **📢 섹션 요약 비유**: 2PC는 한 치의 오차도 허용치 않는 수술실의 '절대 동기화 매뉴얼'이고, Saga는 환자가 많을 때 일단 붕대부터 감고 나중에 차트를 정리하는 응급실의 '전시 대응 프로토콜'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 항공/호텔 예약 패키지 결제 시스템 (Orchestration Saga 도입)

- **문제 상황**: 고객이 '항공권+호텔+렌터카' 패키지를 결제할 때, 항공사 API, 호텔 API, 렌터카 API를 2PC로 묶으려 했으나, 타사 API는 2PC 스펙(XA)을 지원하지 않을 뿐만 아니라 호출 시 기본 5초씩 걸려 WAS가 타임아웃으로 모조리 터져버림.
- **의사결정**:
  1. **오케스트레이터(Saga Execution Coordinator) 구축**: 예약 패키지를 총괄하는 주문 마이크로서비스가 지휘자가 됨.
  2. 고객 결제 완료 시, 항공사, 호텔, 렌터카 API를 비동기 이벤트(Kafka)로 찔러 로컬 트랜잭션을 각각 수행하게 함.
  3. 만약 렌터카 예약에서 "모든 차가 매진됨" 에러가 리턴되면, 오케스트레이터는 이미 예약이 완료된 항공사와 호텔 측에 즉각 **"예약 취소 API (보상 트랜잭션)"**를 쏘아 전체 비즈니스 정합성을 원상 복구함.

### 도입 체크리스트
- **비즈니스적 '중간 상태' 허용 여부 (Semantic Lock)**: Saga는 트랜잭션이 끝날 때까지 1~2초가 걸린다. 그 사이 고객이 마이페이지를 새로고침하면 "돈은 빠져나갔는데, 주문은 완료 안 된 상태"를 볼 수 있다. 기획팀과 합의하여 화면에 "결제 확인 중입니다"라는 임시 상태(Semantic Lock) 뱃지를 띄울 준비가 되었는가?
- **보상 트랜잭션의 멱등성 (Idempotence)**: 재시도 폭풍(Retry Storm)으로 인해 "환불 이벤트"가 네트워크를 타고 2번 전송될 수 있다. 환불 API가 2번 호출되어도 환불은 1번만 되도록(멱등성) 방어 로직(Unique Key 검사)이 코딩되어 있는가?
- **피벗 트랜잭션 (Pivot Transaction)**: Saga 과정 중 "이 단계가 성공하면 절대 돌아갈 수 없고 끝까지 가야 한다"는 지점(예: 이미 고객에게 커피를 만들어버림)이 존재한다. 이를 피벗 포인트라 하며, 이 이후의 에러는 자동 환불이 아니라 수동(CS 고객센터 개입)으로 처리해야 한다는 아키텍처적 한계를 인지해야 한다.

### 안티패턴: MSA 도입 후 분산 트랜잭션을 무시한 "배 째라"식 강제 분리
- "모놀리식을 MSA로 쪼개는 게 유행이라서" DB를 5개로 찢어놓고, 2PC나 Saga 패턴에 대한 아무런 설계 없이 각 DB에 개별 트랜잭션을 날리는 행위. 3번째 DB에서 에러가 나면 앞에 커밋된 2개의 DB는 공중에 붕 떠버리고, 개발자는 매일 밤 SQL로 수동 업데이트를 치며 정합성을 맞추느라 퇴사하게 된다. 분산 트랜잭션 제어 능력이 없다면, MSA 쪼개기를 중단하고 모놀리식 DB(Shared Database)로 돌아가는 것이 100번 낫다.

- **📢 섹션 요약 비유**: 무거운 항아리(모놀리식)를 옮기기 힘들다고 5조각으로 깨부수는 것(MSA)은 쉽습니다. 문제는 그 조각난 파편들 안에 있는 물(데이터 일관성)이 밖으로 새지 않게 고무줄(Saga)로 완벽하게 묶을 실력이 있느냐입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 2PC 기반 강한 결합 | Saga 기반 비동기 느슨한 결합 | 개선 효과 |
|:---|:---|:---|:---|
| **가용성 (Availability)**| 하나의 DB 장애가 전사 시스템 락 유발 | 로컬 장애가 타 서비스로 전파되지 않음 | **SPOF 제거 및 시스템 생존율(Uptime) 극대화** |
| **처리량 (TPS)** | 글로벌 락(Lock) 대기로 극도의 병목 | 락(Lock) 없는 비동기 이벤트 릴레이 | **동시 결제 처리량 100배 이상 선형적 확장** |
| **개발 복잡도** | DB 엔진이 알아서 커밋/롤백 해줌 | 취소(보상) 로직을 개발자가 모두 짜야 함 | (단점) **개발/테스트 기간 2~3배 증가 및 복잡성** |

### 미래 전망
- **Saga 프레임워크의 성숙**: 개발자가 직접 보상 로직의 분기문을 짜던 고통을 줄이기 위해, Camunda, Netflix Conductor, AWS Step Functions 같은 BPF(비즈니스 프로세스 플로우) 엔진들이 Saga의 오케스트레이션을 GUI 코딩 없는(Low-code) 형태로 관리해 주는 솔루션으로 진화하고 있다.
- **TCC (Try-Confirm/Cancel) 패턴**: Saga의 격리성(Isolation) 부재로 인한 중간 상태 노출 문제를 막기 위해, 데이터에 임시 락(예: 재고 -1 임시 차감)을 거는 Try 단계를 추가하여 2PC와 Saga의 장점을 합친 TCC 패턴이 알리바바 등 중국 대형 커머스 코어 엔진을 중심으로 대중화되고 있다.

- **📢 섹션 요약 비유**: 분산 트랜잭션의 세계에서는 공짜 점심이 없습니다. 빛보다 빠른 성능(Saga)을 얻으려면, 혹시 모를 파편화(에러)를 수습하기 위한 길고 복잡한 취소 영수증(보상 트랜잭션)을 직접 써 내려가는 고된 장인 정신이 반드시 수반되어야 합니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 철수, 영희, 민수가 돈을 모아서 게임기를 사기로 했어요. 옛날 방식(**2PC**)은 세 명이 동시에 손에 돈을 꽉 쥐고 눈치만 보다가, 한 명이라도 돈을 안 내면 다 같이 "취소!"하고 흩어지는 깐깐한 방식이에요.
2. 하지만 사람이 100명이면 동시에 손을 내밀기가 너무 힘들잖아요? 그래서 요즘 쓰는 **Saga 패턴**은 일단 철수가 낸 돈을 저금통에 넣고 영희한테 넘겨요.
3. 그러다 민수 차례에서 민수가 돈이 없으면? 저금통에 넣었던 돈을 다시 철수와 영희에게 "환불(보상 트랜잭션)"해 주면서 쿨하게 취소하는 릴레이 방식이랍니다!
