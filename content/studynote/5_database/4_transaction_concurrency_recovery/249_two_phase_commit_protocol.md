# 249. 2단계 락킹 (2PC, Two-Phase Commit) - 동기적 합의의 정석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 2단계 커밋(2PC)은 분산 데이터베이스 환경에서 여러 노드 간의 트랜잭션 원자성을 보장하기 위해, **모든 참여 노드가 준비를 완료했을 때만 최종 커밋을 수행하는 동기식 조율 프로토콜**이다.
> 2. **가치**: "모두가 성공하거나 아니면 모두가 취소한다"는 강한 일관성을 실현하며, 글로벌 트랜잭션 매니저(GTM)를 통해 분산된 자원의 상태를 일치시킨다.
> 3. **융합**: XA 리소스 매니저 인터페이스를 통해 서로 다른 벤더의 DB 간에도 트랜잭션을 묶을 수 있으나, 코디네이터 장애 시 시스템 전체가 멈추는 '블로킹(Blocking)' 문제라는 치명적 약점을 가진다.

+++

### Ⅰ. 2단계 커밋의 프로세스

트랜잭션을 조율하는 **조정자 (Coordinator)**와 실제 작업을 수행하는 **참여자 (Participant)** 간의 협동입니다.

1. **제1단계: 준비 (Prepare Phase)**:
    - 조정자가 모든 참여자에게 "커밋할 준비가 됐니?"라고 묻습니다.
    - 참여자들은 락을 걸고 연산을 수행한 뒤 "Ready" 혹은 "No"로 응답합니다.
2. **제2단계: 커밋 (Commit Phase)**:
    - 모든 참여자가 "Ready"라면 조정자는 "Commit!"을 지시합니다.
    - 한 명이라도 "No"라면 조정자는 "Rollback!"을 지시합니다.

+++

### Ⅱ. 2PC 아키텍처 시각화 (ASCII Flow)

```text
[2PC의 정상 커밋 시나리오]

  [ Coordinator ]           [ Node A ]           [ Node B ]
        │                      │                    │
        │ ─── 1. PREPARE ───▶  │                    │
        │ ─── 1. PREPARE ────────────────────────▶  │
        │                      │                    │
        │ ◀── 2. READY ──────  │                    │
        │ ◀── 2. READY ──────────────────────────  │
        │                      │                    │
        │ ─── 3. COMMIT! ───▶  │                    │
        │ ─── 3. COMMIT! ────────────────────────▶  │
        │                      │                    │
        ▼                      ✅                   ✅
```

+++

### Ⅲ. 2PC의 한계점 (Blocking Problem)

- **동기적 대기**: 한 노드가 응답하지 않으면 다른 모든 노드는 락을 쥔 채 무한정 기다려야 합니다. (가용성 저하)
- **SPOF 위험**: 조정자(Coordinator)가 장애를 일으키면 시스템 전체가 마비될 수 있습니다.
- **성능 병목**: 네트워크 라운드트립이 많아 트랜잭션 처리 시간이 길어집니다.

- **📢 섹션 요약 비유**: 2PC는 **'단체 결혼식의 주례'**와 같습니다. 주례 선생님이 신랑, 신부 수십 쌍에게 한 명씩 "동의하십니까?"라고 묻습니다. 만약 단 한 명이라도 "아니오!"라고 대답하거나 화장실에 가서 대답이 없으면, 결혼식 전체가 중단되고 아무도 부부가 될 수 없습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[원자성]**: 2PC가 존재하는 근본 이유.
- **[XA]**: 2PC를 구현하기 위한 표준 산업 규격.
- **[3PC]**: 2PC의 블로킹 문제를 해결하기 위해 고안된 대안.

📢 **마무리 요약**: **2PC**는 고전적이고 강력한 분산 무결성 도구입니다. 하지만 가용성이 중요한 현대 MSA 환경에서는 그 무거움 때문에 **Saga 패턴** 등으로 대체되는 추세입니다.
