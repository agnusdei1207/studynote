# 244. LSN (Log Sequence Number) - 데이터의 시간표

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: LSN(Log Sequence Number)은 데이터베이스 로그 파일 내의 **각 로그 레코드에 부여되는 유일하고 단조 증가하는 식별 번호**이다.
> 2. **가치**: 데이터 페이지의 최신성 여부를 판단하는 척도가 되며, 복구 시 중복 연산을 방지하고 로그 파일 내의 정확한 물리적 위치를 찾아내는 이정표 역할을 한다.
> 3. **융합**: 페이지 헤더의 `pageLSN`과 로그의 `LSN`을 비교하는 매커니즘을 통해, 복구 알고리즘이 "이 변경 사항이 이미 디스크에 반영되었는가?"를 0.1초 만에 판별하게 돕는다.

+++

### Ⅰ. LSN의 정의와 물리적 실체

- **정의**: 로그가 생성될 때마다 1씩 증가하거나, 로그 파일의 바이트 오프셋(Byte Offset)을 값으로 가집니다.
- **pageLSN**: 모든 데이터 페이지의 헤더에는 해당 페이지를 마지막으로 수정한 로그의 LSN이 저장되어 있습니다.

+++

### Ⅱ. LSN을 활용한 복구 판단 (ASCII Logic)

재부팅 후 Redo를 수행할지 결정하는 핵심 로직입니다.

```text
[LSN 비교를 통한 복구 판단]

  (Log Record)  [ LSN: 500 ] : "Set A = 20"
                       │
                       ▼
  (Data Page A) [ pageLSN: 400 ]
  
  Logic: IF Log.LSN (500) > Page.pageLSN (400) THEN
            "데이터가 과거 것이네! Redo 실행!" ──▶ ✅
         ELSE
            "이미 디스크에 반영됐군. 패스!" ──▶ ⏭️
```

+++

### Ⅲ. LSN의 주요 기능

1. **복구 범위 결정**: 체크포인트 LSN을 통해 어디서부터 로그를 읽을지 정합니다.
2. **트랜잭션 순서 보장**: 분산 환경이나 복제 환경에서 연산의 선후 관계를 명확히 합니다.
3. **버퍼 플러시 제어**: WAL 프로토콜을 지키기 위해, 특정 데이터 페이지를 쓰기 전 로그의 LSN이 디스크에 먼저 박혔는지 검사합니다.

- **📢 섹션 요약 비유**: LSN은 **'공정표의 일련번호'**와 같습니다. 건물을 지을 때 벽돌을 500번까지 쌓기로 했는데, 사고가 나서 다시 확인해 보니 벽에 '400번' 도장이 찍혀 있다면, 401번부터 다시 쌓기 시작하면 되는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Flushed LSN]**: 디스크에 안전하게 저장된 마지막 로그 번호.
- **[Dirty Page Table]**: LSN 정보를 바탕으로 복구할 페이지를 모아둔 표.
- **[WAL]**: LSN이 탄생하게 된 근본 배경.

📢 **마무리 요약**: **LSN**은 무질서한 로그의 세계에 질서를 부여하는 번호표입니다. 이 작은 숫자가 모여 거대한 데이터베이스의 무결성을 지탱합니다.
