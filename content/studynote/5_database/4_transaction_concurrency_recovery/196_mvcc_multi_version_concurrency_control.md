# 196. 다중 버전 동시성 제어 (MVCC) - 락 없는 읽기 일관성의 구현

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다중 버전 동시성 제어(MVCC, Multi-Version Concurrency Control)는 데이터를 갱신할 때 기존 데이터를 덮어쓰지 않고 **이전 버전(스냅샷)을 유지**하여, 읽기 작업과 쓰기 작업이 서로를 차단(Blocking)하지 않도록 하는 동시성 제어 기법이다.
> 2. **가치**: "읽기가 쓰기를 막지 않고, 쓰기가 읽기를 막지 않는다"는 원칙을 실현하여, 락(Lock) 기반 동시성 제어의 치명적 단점인 성능 저하와 교착 상태(Deadlock) 가능성을 획기적으로 낮춘다.
> 3. **구조**: Oracle의 Undo 세그먼트, PostgreSQL의 다중 튜플, MySQL(InnoDB)의 Undo 로그 등 RDBMS마다 구현 방식은 다르나, **시간(SCN/XID)**을 기준으로 트랜잭션에게 일관된 과거 시점의 뷰를 제공하는 핵심 원리는 동일하다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**MVCC (Multi-Version Concurrency Control)**는 동시성 제어를 위해 락(Lock)에만 의존하는 대신, **데이터의 여러 버전을 동시에 관리**하는 접근법입니다. 읽기 트랜잭션은 락을 기다릴 필요 없이 자신이 시작된 시점(또는 쿼리가 시작된 시점)에 맞는 올바른 버전의 데이터를 찾아 읽습니다.

### 💡 비유: 위키백과(Wikipedia)의 역사 보기
락 기반 시스템이 "한 명이 문서를 편집하는 동안 다른 사람은 문서를 아예 볼 수 없는 도서관 대출 시스템"이라면, MVCC는 "누군가 문서를 최신 버전으로 편집하고 있더라도, 다른 사람들은 '버전 역사'를 통해 과거의 문서를 언제든 조회할 수 있는 위키백과"와 같습니다.

### 2. 등장 배경
- **2PL (2단계 락킹)의 한계**: 전통적인 락킹 시스템에서는 데이터를 수정(X-Lock) 중일 때 다른 사용자가 읽을(S-Lock) 수 없었고, 읽는 중(S-Lock)일 때 수정(X-Lock)할 수도 없었습니다. 이는 대규모 트래픽에서 심각한 병목(Blocking)을 유발했습니다.
- **성능과 정합성의 딜레마 극복**: 데이터 정합성을 포기할 수 없었던 시스템 엔지니어들은, 갱신 전의 데이터를 어딘가에 백업해두면 쓰기 중에도 읽기가 가능하지 않을까 하는 발상에서 MVCC 아키텍처를 탄생시켰습니다.

- **📢 섹션 요약 비유**: MVCC는 마치 시간을 되돌리는 타임머신처럼, 트랜잭션마다 각자의 시간대(스냅샷)에 맞는 평행 우주(다중 버전)를 제공하여 충돌을 없앱니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. MVCC 구성 요소

| 요소명 | 역할 | 작동 원리 | RDBMS 구현 예시 | 비유 |
|:---|:---|:---|:---|:---|
| **현재 데이터 블록 (Data Block)** | 가장 최신의 데이터 상태 저장 | 쓰기 트랜잭션이 직접 갱신 (In-place update) | 테이블 스페이스, 페이지 | 최신 버전 원고 |
| **Undo 세그먼트 / 로그 (Undo Log)** | 갱신되기 전의 과거 데이터 스냅샷 저장 | 변경 전 원본 데이터를 기록 체인으로 연결 | Oracle(Undo), MySQL(Undo) | 편집 이전의 백업 파일 |
| **트랜잭션 ID (XID / SCN)** | 데이터 버전을 식별하는 논리적 시간 스탬프 | 트랜잭션 시작 시 부여, 커밋 순서 결정 | SCN (Oracle), XID (PostgreSQL) | 원고의 발행 타임스탬프 |
| **읽기 일관성 (Read Consistency) 엔진** | 사용자에게 맞는 버전을 조립하여 반환 | CR(Consistent Read) 블록 생성 | Buffer Cache 내부 로직 | 타임머신 뷰어 |
| **가비지 컬렉터 (Garbage Collector)** | 더 이상 참조되지 않는 과거 버전 삭제 | 오래된 튜플/언두 영역을 주기적으로 회수 | PostgreSQL (Vacuum), MySQL (Purge) | 휴지통 비우기 |

### 2. MVCC 심층 동작 원리 (Oracle / MySQL InnoDB 방식)

MVCC 환경에서 데이터를 갱신(UPDATE)하고 다른 트랜잭션이 이를 읽는(SELECT) 과정을 다이어그램으로 살펴봅니다. (Out-of-place 방식을 쓰는 PostgreSQL과 달리, 가장 대중적인 In-place + Undo 방식을 기준으로 합니다.)

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    [MVCC 기반의 읽기 일관성 동작 메커니즘]                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│    [시간: T1] SCN: 100 기준 데이터베이스 상태                                   │
│    테이블 데이터 블록: [ ID: 1, 잔액: 1000 ]                                   │
│                                                                             │
│    [시간: T2] 트랜잭션 A 시작 (UPDATE 잔액=500 WHERE ID=1)                    │
│    1. A가 기존 데이터(잔액 1000)를 Undo 세그먼트로 복사                       │
│    2. A가 테이블 블록의 데이터를 500으로 변경 (X-Lock 획득, 미커밋 상태)          │
│                                                                             │
│          테이블 블록 (현재)           Undo 세그먼트 (과거 버전)               │
│          ┌────────────────┐         ┌────────────────────┐                │
│          │ ID: 1          │ 포인터   │ ID: 1              │                │
│          │ 잔액: 500      │────────▶│ 잔액: 1000         │                │
│          │ SCN: 미확정(A)  │         │ SCN: 100           │                │
│          └────────────────┘         └────────────────────┘                │
│                                                                             │
│    [시간: T3] 트랜잭션 B 시작 (SELECT 잔액 FROM 계좌 WHERE ID=1)              │
│    1. B는 테이블 블록에 접근함.                                                │
│    2. 데이터가 트랜잭션 A에 의해 수정 중(미커밋)임을 감지함.                       │
│    3. B는 대기(Block)하지 않고, 포인터를 따라 Undo 세그먼트로 이동함.             │
│    4. B의 시작 시점(T3)보다 오래된 SCN 100의 데이터(잔액: 1000)를 읽음.           │
│       → 이를 **CR(Consistent Read) 블록**이라고 함.                          │
│                                                                             │
│    [시간: T4] 트랜잭션 A 커밋 (SCN: 110 발급)                                  │
│    1. 테이블 블록의 SCN이 110으로 확정됨.                                      │
│    2. 이후 시작되는 새로운 트랜잭션은 500을 읽게 됨.                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**[해설]**
MVCC의 핵심은 덮어쓰기가 발생할 때 **기존 값을 Undo(롤백) 영역에 남겨둔다는 것**입니다. 읽기 작업을 수행하는 트랜잭션 B는 현재 데이터 블록에 누군가 락(X-Lock)을 걸어두었더라도 기다리지 않습니다. 대신, 데이터 블록 헤더에 기록된 트랜잭션 정보를 보고 아직 커밋되지 않았음을 확인한 뒤, 연결된 Undo 세그먼트를 찾아가 자신이 볼 자격이 있는 '과거 스냅샷'을 재구성(CR 블록)하여 읽습니다. 결과적으로 "쓰기가 읽기를 차단하지 않는" 동시성이 달성됩니다.

### 3. RDBMS별 MVCC 구현 아키텍처 차이

- **Oracle / MySQL (InnoDB)**: 원본 데이터는 제자리에 업데이트(In-place update)하고, 이전 값은 별도의 **Undo 세그먼트**에 저장합니다. 장점은 테이블 크기가 불필요하게 커지지 않는다는 것이며, 단점은 장기 실행 트랜잭션이 있을 때 Undo 영역이 가득 차서 `Snapshot too old` 에러가 발생할 수 있다는 점입니다.
- **PostgreSQL**: 동일한 테이블 내에 새로운 튜플(행)을 삽입(Out-of-place update)하고 구버전 튜플도 그대로 남겨둡니다. 튜플 헤더의 트랜잭션 ID(xmin, xmax)로 가시성을 판단합니다. 장점은 롤백이 매우 빠르다는 것이나, 단점은 테이블 내에 죽은 튜플(Dead Tuple)이 쌓여 주기적으로 **Vacuum (가비지 컬렉션)** 작업을 해주지 않으면 성능이 심각하게 저하된다는 점입니다.

- **📢 섹션 요약 비유**: 오라클/MySQL은 '별도의 보관함(Undo)'에 구버전을 정리하는 스타일이고, PostgreSQL은 '같은 서랍(테이블)'에 새 문서를 계속 쌓아두고 나중에 대청소(Vacuum)를 하는 스타일입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 비관적 락(2PL) vs MVCC 아키텍처 비교

| 비교 항목 | 2PL (2단계 락킹) | MVCC (다중 버전 제어) |
|:---|:---|:---|
| **읽기와 쓰기 충돌** | 심각 (서로 차단함) | **없음** (서로 차단하지 않음) |
| **읽기 일관성 보장 방식** | Shared Lock (S-Lock) 유지 | Undo 영역/버전 스냅샷 활용 |
| **동시성 (성능)** | 낮음 (대기 시간 김) | 매우 높음 (대기 없음) |
| **추가 공간 오버헤드** | 없음 (제자리 수정) | 높음 (Undo Segment 또는 Dead Tuple 공간 필요) |
| **주요 한계점** | 교착 상태(Deadlock) 빈발 | 장기 트랜잭션 시 Undo 고갈 (Snapshot too old) |

### 2. 고립화 수준(Isolation Level)과 MVCC의 연계
MVCC는 각 고립화 수준을 어떻게 구현할까요?
- **Read Committed**: 매 쿼리(Statement)가 시작될 때마다 새로운 스냅샷을 생성합니다. 따라서 쿼리와 쿼리 사이에 다른 트랜잭션이 커밋하면 값이 달라질 수 있습니다(Non-Repeatable Read 발생).
- **Repeatable Read**: 트랜잭션(Transaction)이 시작되는 첫 시점에 단 하나의 스냅샷을 생성하고 끝날 때까지 그 스냅샷만 봅니다. 따라서 중간에 다른 트랜잭션이 커밋하더라도 내 눈에는 일관된 과거만 보이므로 반복 읽기가 보장됩니다. (MySQL InnoDB는 이를 통해 Phantom Read까지 방어합니다.)

- **📢 섹션 요약 비유**: 2PL이 한 길밖에 없는 좁은 터널이라면, MVCC는 차량마다 서로 다른 고도로 날아갈 수 있게 해주는 입체 교차로(다중 버전) 시스템입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오
- **대용량 배치 리포트 추출 (Snapshot too old)**: 수억 건의 데이터를 읽어 리포트를 생성하는 쿼리가 3시간 동안 실행되고 있습니다. 그 사이 OLTP 서비스에서 무수히 많은 UPDATE가 발생하고 커밋됩니다. 오라클 환경의 경우, 3시간 전의 과거 버전을 유지하기 위해 Undo 세그먼트를 꽉 채우다가 결국 공간이 부족해지면 `ORA-01555: Snapshot too old` 에러를 뱉고 3시간짜리 쿼리가 죽어버립니다.
  - **해결책**: Undo 테이블스페이스의 크기를 늘리고 `UNDO_RETENTION` 파라미터를 크게 설정하거나, 배치 쿼리 자체의 수행 시간을 튜닝하여 줄여야 합니다.
- **PostgreSQL의 성능 저하 (Vacuum Bloat)**: PostgreSQL 기반 서비스에서 잦은 UPDATE와 DELETE로 인해 디스크 사용량이 폭증하고 쿼리 속도가 느려졌습니다.
  - **해결책**: 이는 Dead Tuple이 쌓여 물리적 테이블 비대화(Bloat)가 일어난 현상입니다. Auto-vacuum 주기를 공격적으로 튜닝하고 트랜잭션을 짧게 유지해야 합니다.

### 2. 안티패턴
- **의미 없는 롱 트랜잭션(Long Transaction) 방치**: 트랜잭션을 시작해 놓고 사용자 입력을 기다리거나 외부 API 호출(대기 시간 수십 초)을 수행하는 것은 MVCC에서 최악의 안티패턴입니다. 긴 트랜잭션 하나 때문에 DBMS는 그 시점 이후의 모든 데이터 버전을 지우지 못하고(Garbage Collection 불가) 계속 끌어안고 있어야 하므로, 시스템 전체의 메모리와 디스크 성능을 심각하게 저하시킵니다.

- **📢 섹션 요약 비유**: 긴 트랜잭션을 방치하는 것은 식당에서 밥을 다 먹고도 자리를 비켜주지 않아 밖에서 대기하는 손님들(시스템 자원)을 지치게 만드는 민폐 행위와 같습니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과

| 구분 | MVCC 미적용 (Lock 기반) | MVCC 기반 아키텍처 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 트랜잭션 동시 실행 시 TPS 급감 | 쓰기 작업 중에도 읽기 TPS 유지 | 대규모 OLTP 환경에서 TPS **수십 배 향상** |
| **정량** | 교착 상태(Deadlock) 타임아웃 잦음 | 읽기-쓰기 간 데드락 원천 차단 | 데드락 발생률 **90% 이상 감소** |
| **정성** | Read 분석과 OLTP 쓰기 분리 강제 | 동일 DB에서 유연한 혼합 워크로드 | 하이브리드 운영 유연성 확보 |

### 2. 결론
MVCC는 현대 RDBMS(Oracle, MySQL InnoDB, PostgreSQL 등)를 지탱하는 가장 중요한 동시성 제어 엔진입니다. 데이터 무결성을 보장하면서도 극단적인 동시성을 끌어내기 위해 '공간(Undo)을 희생하여 시간(대기)을 번다'는 아키텍처적 트레이드오프의 교과서적 사례입니다. 개발자와 DBA는 자신이 사용하는 DBMS의 MVCC 내부 구현체(In-place vs Out-of-place)를 명확히 이해하고, 긴 트랜잭션 관리와 Undo/Vacuum 최적화에 힘써야 합니다.

- **📢 섹션 요약 비유**: MVCC는 현대 데이터베이스라는 대도시에 거대한 '차원 분리' 인프라를 구축하여, 출근길(쓰기)과 퇴근길(읽기) 차량들이 서로 부딪히지 않게 만들어준 마법의 교차로입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **격리 수준 (Isolation Level)**: MVCC가 스냅샷을 생성하고 유지하는 기준점(Statement vs Transaction).
- **이상 현상 (Concurrency Anomalies)**: MVCC가 락 없이도 방어해내는 데이터 오염 문제들.
- **트랜잭션 복구 (Recovery)**: MVCC를 위한 Undo 영역은 롤백 및 시스템 장애 복구 시에도 동일하게 활용됨.
- **가비지 컬렉션 (Vacuum)**: PostgreSQL 아키텍처에서 구버전 데이터를 청소하는 필수 매커니즘.

+++

## 👶 어린이를 위한 3줄 비유 설명
1. **MVCC**는 친구가 내 일기장을 고치고 있을 때, 내가 기다리지 않도록 **'고치기 전 일기장 복사본'**을 보여주는 마법이에요.
2. 덕분에 쓰는 친구와 읽는 친구가 서로 비켜달라고 싸울(Blocking) 필요 없이 **동시에 숙제**를 할 수 있답니다.
3. 대신 다 쓴 복사본(Undo)들은 너무 많이 쌓이기 전에 휴지통에 제때제때 버려줘야(Vacuum) 책상이 깨끗하게 유지돼요!
