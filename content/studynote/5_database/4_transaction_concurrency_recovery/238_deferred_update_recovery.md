# 238. 지연 갱신 (Deferred Update) - 성공할 때까지 기다리는 미덕

### # 지연 갱신 (Deferred Update)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 지연 갱신(Deferred Update)은 트랜잭션(Transaction)의 원자성(Atomicity)을 보장하기 위해, 트랜잭션이 **COMMIT(확정)되기 전까지 모든 변경 사항을 비휘발성 로그에만 기록하고 실제 데이터베이스(DB)에는 반영하지 않는 회복 기법**입니다.
> 2. **가치**: 장애 발생 시 실제 DB가 훼손되지 않았음을 보장하므로 **Undo(취소) 로직이 전혀 필요 없어 No-Undo 속성**을 가지며, 이를 통해 복구 절차를 극도로 단순화하고 Rollback 비용을 제로(0)로 만듭니다.
> 3. **융합**: ARIES(Algorithms for Recovery and Isolation Exploiting Semantics)와 같은 현대적 복구 알고리즘의 기저에 깔린 'No-Steal, Force' 버퍼 관리 전략과 직결되며, 고신뢰성(High Availability)이 요구되는 분산 데이터베이스 설계의 핵심 철학입니다.

---

### Ⅰ. 개요 (Context & Background)

지연 갱신은 데이터베이스 회복(Recovery) 관리자가 트랜잭션의 중간 상태를 데이터베이스에 반영하는 것을 허용하지 않는 보수적이지만 강력한 접근 방식입니다. 기존의 즉시 갱신(Immediate Update) 방식이나 Shadow Paging 방식과 비교할 때, 지연 갱신은 **'트랜잭션의 완료 여부가 불확실한 상태에서는 물리적 데이터를 절대 변경하지 않는다'**는 철학을 따릅니다.

이 방식은 주로 **WAL (Write-Ahead Logging)** 프로토콜의 변형이나, 로그 기반의 시스템에서 장애 복구의 오버헤드를 줄이기 위해 채택됩니다. 특히, 시스템 crash(충돌) 발생 시 디스크 상의 데이터베이스는 항상 `Commit`된 트랜잭션의 결과만 담고 있다는 보장을 전제로 하므로, 복구 로직은 `COMMIT` 마크가 있는 로그 레코드에 대해서만 **Redo(재수행)** 연산을 수행하면 됩니다.

**💡 비유: "주문 제작 의류 서비스"**
이는 고객이 입금 완료(Commit)를 하기 전까지는 재단사가 천을 한 번도 자르지 않고, 설계도(Log)만 먼저 작성해두는 것과 같습니다. 고객이 마음을 바꿔도(트랜잭션 실패/Abort) 천은 손상되지 않았으므로 다시 사용할 수 있습니다.

**등장 배경 및 필요성**:
1.  **기존 한계**: 즉시 갱신 방식에서는 장애 발생 시, 미반영된 변경을 취소하기 위해 Undo 로그를 역추적해야 하므로 복구 시간이 길고 로직이 복잡했습니다.
2.  **혁신적 패러다임**: "실제 DB는 트랜잭션 완료 시에만 변경한다"는 **No-Undo** 속성을 도입하여, 복구 시 Undo 과정을 완전히 배제하여 복구 속도를 획기적으로 개선했습니다.
3.  **비즈니스 요구**: 금융 거래나 항공 예약 등 데이터 일관성이 생명이며, Rollback 비용이 높은 시스템에서 복구의 신뢰성과 단순성을 극대화해야 했습니다.

> **📢 섹션 요약 비유**:
> 지연 갱신은 마치 **'게임의 자동 저장 포인트'**와 같습니다. 보스를 물리치기 전까지(Commit 전)는 아무리 노력해도 진척도가 0%인 상태로 유지되다가, 보스를 쓰러뜨리는 순간(Checkpoint/Commit)에야 비로소 '클리어' 데이터로 확정되어 저장됩니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

지연 갱신의 핵심은 **버퍼 관리(Buffer Management)**와 **로그 순차 기록(Sequential Logging)**의 조합에 있습니다. 이 방식을 구현하기 위해서는 데이터베이스 시스템이 기본적으로 **No-Steal(노스틸)** 버퍼 관리 정책을 따라야 합니다. 즉, 커밋되지 않은 트랜잭션의 데이터를 디스크(비휘발성 저장소)로 내보내지 않아야 합니다.

#### 1. 구성 요소 (주요 모듈 및 동작)

| 요소명 (Component) | 역할 (Role) | 내부 동작 (Internal Behavior) | 주요 프로토콜/명령어 | 관련 비유 |
|:---|:---|:---|:---|:---|
| **Log Manager** | 로그 버퍼 관리 및 디스크 기록 | 트랜잭션의 모든 변경(`Update`, `Insert`)을 순차적으로 버퍼에 적재 후, 로그 파일(Log File)로 플러시(Flushing) | `LSN (Log Sequence Number)` 할당, Force Log at Commit | 사건의 기록자 |
| **Recovery Manager (RM)** | 장애 복구 수행 | 시스템 재시작 시 로그 스캔을 통해 Redo할 트랜잭션 식별 및 데이터베이스 반영 | `Redo(T)`, `Analysis` Phase | 복구 반장 |
| **Buffer Manager** | 메모리 캐시 관리 | **No-Steal 정책** 강제. Uncommitted 데이터는 디스크로 쓰지 않음(Flush 금지). 커밋 시에만 디스크로 내보냄 가능 | `Pin`, `Unpin` Page | 출입구 관리자 |
| **Transaction Processor** | 트랜잭션 로직 실행 | 사용자 요청을 받아 로그에만 기록(Timestamp 등 기록) 후 종료 시점에 `COMMIT` 로그 작성 | `BEGIN`, `COMMIT`, `ABORT` | 주문 접수처 |

#### 2. ASCII 아키텍처: 지연 갱신의 데이터 흐름 (Data Flow)

```text
  [지연 갱신 (Deferred Update) 시스템 아키텍처]

  ┌─────────────── 사용자 요청 (User Request) ─────────────────┐
  │ T1: A=A-100 (Write)                                       │
  └───────────────────────────┬───────────────────────────────┘
                              ▼
  ┌─────────────────────────────────────────────────────────────┐
  │  [Transaction Processor] & [Log Buffer (Volatile Memory)]  │
  │  ┌─────────────────────────────────────────────────────┐   │
  │  │ 1. [T1] Write A (100 -> 0) ──▶ 메모리 버퍼에만 기록  │   │
  │  │ 2. [T1] COMMIT ────────────▶ 로그 버퍼에 기록       │   │
  │  └─────────────────────────────────────────────────────┘   │
  └───────────────────────┬───────────────────────────────────┘
                           │ (Force Log at Commit)
                           ▼
  ┌─────────────────────────────────────────────────────────────┐
  │      [Stable Storage (Disk)] / Log File                     │
  │  ┌─────────────────────────────────────────────────────┐   │
  │  │ LSN 100: [T1 Start]                                  │   │
  │  │ LSN 101: [T1 Update: A=0] (Before/After Image)      │   │
  │  │ LSN 102: [T1 COMMIT] ◀── 이 시점에 디스크에 영구 저장 │   │
  │  └─────────────────────────────────────────────────────┘   │
  └───────────────────────┬───────────────────────────────────┘
                           │ (실제 DB 반영은 나중에 or Lazy)
                           ▼
  ┌─────────────────────────────────────────────────────────────┐
  │      [Actual Database Files] (Data Pages)                   │
  │  ┌─────────────────────────────────────────────────────┐   │
  │  │ 아직 A=100 (변경 전 상태 유지)                        │   │
  │  │ (Commit 로그 확인 후 Background 또는 Recovery 시 반영)│   │
  │  └─────────────────────────────────────────────────────┘   │
  └─────────────────────────────────────────────────────────────┘
```

**[도해 해설]**
1.  **로그 선행 기록(Force Log)**: 트랜잭션이 커밋되는 순간, 데이터베이스 파일이 아닌 **로그 파일(Log File)**에 `COMMIT` 기록과 변경 값(새 값 New Value)이 먼저 안전하게 저장됩니다. 이때 `LSN(Log Sequence Number)`을 사용하여 순서를 보장합니다.
2.  **No-Steal (훔치지 말라)**: 그림에서처럼 커밋 전까지 데이터 페이지는 디스크로 내려가지 않습니다. 따라서 디스크의 데이터베이스 페이지는 항상 "커밋된 상태"이거나 "초기 상태"로 존재합니다.
3.  **실제 반영 타이밍**: 로그에만 기록되므로, 실제 데이터 파일에 대한 수정은 사용자에게 응답을 보낸 후 비동기적으로(Asynchronously) 일어나거나, Recovery 시점에 일어날 수 있습니다.

#### 3. 심층 동작 원리 및 복구 알고리즘

지연 갱신의 복구 알고리즘은 **Redo-Only** 특성을 가집니다.

*   **단계 1: 분석 (Analysis)**
    *   로그 파일을 스캔하여 가장 마지막에 성공한 `COMMIT` 로그를 찾습니다.
    *   만약 `COMMIT` 로그가 없는 트랜잭션이 존재한다면, 이 트랜잭션은 DB에 아무런 영향을 주지 않은 상태이므로 **무시(Ignore)**합니다. (Undo 불필요)

*   **단계 2: 재실행 (Redo)**
    *   로그에 `<T, Commit>` 기록이 있는 모든 트랜잭션에 대해, 해당 트랜잭션의 연산을 순서대로 다시 실행하여 데이터베이스에 반영합니다.
    *   이미 반영된 연산이라도(중복 실행), 로그의 갱신 값(Before/After Image 중 After)을 그대로 덮어쓰므로 결과적으로 일관성 있는 상태가 됩니다. (`Redo는 멱등성 Idempotence`을 가짐)

#### 4. 핵심 코드 (의사코드)

```sql
-- [트랜잭션 처리 로직]
FUNCTION Begin_Transaction(T):
    Log_Write(T, "Start")
    Main_Memory_Buffer.state = ACTIVE

FUNCTION Write_Operation(T, DataItem, Value):
    -- 지연 갱신 핵심: DB에 바로 쓰지 않음
    Log_Write(T, DataItem, OldValue, Value) 
    Main_Memory_Buffer[DataItem] = Value
    -- DB Disk에 쓰지 않음 (No-Steal)

FUNCTION Commit_Transaction(T):
    Log_Write(T, "Commit") 
    Flush_Log_to_Disk() -- 반드시 디스크 로그에 기록
    -- 실제 DB 반영은 나중에 (Deferred)

-- [복구 관리자 로직 (Restart)]
FUNCTION Recovery():
    Uncommitted_List = Scan_Log_For_No_Commit()
    Committed_List = Scan_Log_For_Commit()
    
    -- Uncommitted는 No-Undo이므로 아무것도 안 함
    -- Committed는 Redo만 수행
    FOR each Txn IN Committed_List:
        Redo(Txn) 
```

> **📢 섹션 요약 비유**:
> 지연 갱신의 아키텍처는 **'청소부의 출퇴근 카드'**와 같습니다. 청소부(트랜잭션)가 청소를 다 하지 않았다면(Commit 전), 회사는 출퇴근 기록(로그)은 남겨두되 정작 사무실(실제 DB)은 청소하지 않은 원래 상태로 둡니다. 그러다 청소부가 "완료(Commit)"를 찍고 나가야만 비로소 사무실이 깨끗해지는 것입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

지연 갱신 방식은 데이터베이스의 전통적인 회복 기법인 즉시 갱신(Immediate Update)과 대조적이며, **OS의 메모리 관리(Copy-on-Write)** 및 **분산 시스템의 2단계 커밋(2PC, Two-Phase Commit)**과도 깊은 연관이 있습니다.

#### 1. 심층 기술 비교: 지연 갱신 vs 즉시 갱신

| 구분 | 지연 갱신 (Deferred Update) | 즉시 갱신 (Immediate Update) |
|:---|:---|:---|
| **DB 반영 시점** | `COMMIT` 시점 이후 (로그 먼저) | 연산 수행 즉시 (로그 먼저 기록 후) |
| **Undo 필요성** | **불필요 (No-Undo)**<br>DB는 커밋 전까지 변경되지 않음 | **필요 (Undo)**<br>장애 시 미반영 변경을 롤백해야 함 |
| **Redo 필요성** | **필요 (Redo)**<br>장애 시 커밋된 트랜잭션을 다시 반영 | **필요 가능성 있**<br>메모리 내 변경이 디스크로 반영되기 전 장애 시 |
| **버퍼 정책** | **No-Steal** (Uncommitted page flush 금지) | **Steal 허용** (버퍼 공간 확보를 위해 디스크로 내보낼 수 있음) |
| **복구 복잡도** | 낮음 (Redo만 하면 됨) | 높음 (Undo와 Redo를 모두 고려해야 함 - LSN 체이닝) |
| **주요 사용처** | 중소 규모 시스템, 단순한 로깅 시스템 | 고성능 OLTP, 대규모 DB (오라클 등) |

#### 2. ASCII 다이어그램: 장애 시나리오 비교

```text
[장애 발생 시 복구 로직 차이 비교]

시나리오: T1이 A=100으로 변경 후, COMMIT 직전 시스템 다운.

┌──────────────────────────────────────────────────────────────┐
│ 1. 지연 갱신 (Deferred Update)                               │
├──────────────────────────────────────────────────────────────┤
│ Log: [Start T1] -> [Write A=100] -> (Commit 전 다운)          │
│ DB Disk:  A = 50 (변경 없음)                                 │
│                                                              │
│ [복구 Action]                                                │
│  ──▶ T1의 로그에 'COMMIT'이 없음.                            │
│  ──▶ 'Undo' 필요 없음 (DB는 이미 50이므로)                  │
│  ──▶ 결과: A = 50
