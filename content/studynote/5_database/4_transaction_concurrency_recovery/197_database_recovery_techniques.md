# 197. 데이터베이스 회복 기법 (Recovery) - 장애로부터의 생존과 영속성 보장

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 회복(Recovery)은 정전, 시스템 크래시, 디스크 오류 등 예기치 않은 장애가 발생했을 때, 데이터베이스를 장애 발생 이전의 **일관된(Consistent) 상태로 복원**하는 핵심 메커니즘이다.
> 2. **원리**: 트랜잭션의 ACID 특성 중 **원자성(Atomicity)**을 보장하기 위해 완료되지 않은 작업은 취소(Undo)하고, **영속성(Durability)**을 보장하기 위해 완료된 작업은 재실행(Redo)한다. 이를 위해 **WAL(Write-Ahead Logging)** 프로토콜이 필수적으로 사용된다.
> 3. **아키텍처**: 현대의 RDBMS는 체크포인트(Checkpoint)를 통해 복구 시간을 단축하며, **ARIES(Algorithm for Recovery and Isolation Exploiting Semantics)** 알고리즘을 표준으로 채택하여 정교하고 신속한 복구를 수행한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**데이터베이스 회복 (Recovery)**은 장애(Failure)로 인해 손상되거나 불일치 상태에 빠진 데이터베이스를, 트랜잭션 로그(Log)와 백업 데이터를 활용하여 장애가 발생하기 직전의 정상적인 상태로 되돌리는 과정입니다.

### 💡 비유: 블랙박스와 타임머신
사고가 났을 때 비행기의 블랙박스(로그 파일)를 열어 비행 궤적을 확인한 뒤, 정상 비행을 마친 기록(Commit)은 그대로 인정(Redo)하고, 사고로 마무리되지 못한 기록(Uncommitted)은 이륙 전 상태로 되돌리는(Undo) 것과 같습니다.

### 2. 장애의 유형과 등장 배경
- **트랜잭션 장애 (Transaction Failure)**: 논리적 오류, 데이터 입력 오류, 데드락 등으로 트랜잭션 내부에서 발생. (Rollback으로 해결)
- **시스템 장애 (System Failure)**: 정전, 하드웨어 크래시, OS 패닉 등 메인 메모리(버퍼) 데이터가 날아가는 장애. (Redo/Undo 복구 필요)
- **미디어 장애 (Media Failure)**: 디스크 헤드 손상 등 비휘발성 저장장치 자체가 물리적으로 파손되는 장애. (덤프 백업 + 로그 아카이브 복구 필요)

모든 데이터 변경을 매번 디스크에 직접 쓰면(I/O) 성능이 끔찍하게 느려집니다. 그래서 메모리(버퍼 풀)에서 먼저 데이터를 바꾸고 나중에 디스크에 쓰는데, 이 시차 때문에 정전이 발생하면 **"커밋했는데 디스크에 안 써진 데이터"**와 **"커밋 안 했는데 디스크에 써진 데이터"**가 생겨납니다. 이를 바로잡는 기술이 회복 기법입니다.

- **📢 섹션 요약 비유**: 메모리는 빠르지만 건망증이 심한 단기 기억이고 디스크는 느리지만 확실한 장기 기억인데, 회복 기법은 이 둘 사이의 기억 상실증(장애)을 일기장(로그)을 보며 치료하는 의사입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 핵심 구성 요소

| 요소명 | 역할 | 작동 원리 | 비유 |
|:---|:---|:---|:---|
| **WAL (Write-Ahead Logging)** | 데이터 갱신 전, 로그부터 안전하게 디스크에 기록하는 프로토콜 | 트랜잭션 커밋은 '데이터 기록'이 아닌 '로그 플러시(Flush)'로 완료됨 | 행동하기 전에 계획서부터 제출하기 |
| **Redo (재실행)** | 영속성(Durability) 보장 | 장애 발생 시 커밋된 트랜잭션의 변경 사항을 로그를 보고 디스크에 재적용 | 완료된 숙제 다시 쓰기 |
| **Undo (취소)** | 원자성(Atomicity) 보장 | 장애 발생 시 커밋되지 않은 트랜잭션의 변경 사항을 로그를 보고 원상 복구 | 쓰다 만 숙제 지우개로 지우기 |
| **체크포인트 (Checkpoint)** | 복구 소요 시간(RTO) 단축 | 주기적으로 메모리 버퍼의 변경된 페이지(Dirty Page)를 디스크에 동기화 | 게임 중간 저장 지점 |
| **LSN (Log Sequence Number)** | 로그 레코드의 고유 식별자 | 로그의 순서를 보장하여 Redo/Undo의 멱등성(Idempotency) 유지 | 일기장의 순서 번호 매기기 |

### 2. WAL 프로토콜과 버퍼 동기화 원리

데이터베이스는 성능을 위해 버퍼 풀(메모리)에서 데이터를 조작합니다. 이때 **WAL 프로토콜**은 다음 두 가지 절대 규칙을 지킵니다.
1. 데이터를 디스크에 쓰기(Flush) 전에, **반드시 그 변경 내용이 담긴 로그부터 디스크에 기록**해야 한다. (Undo 보장용)
2. 트랜잭션이 커밋되면, **반드시 해당 트랜잭션의 로그부터 디스크에 기록(Sync)**해야 한다. (Redo 보장용)

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                      [메모리 버퍼와 WAL 기반 로그 기록 흐름]                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [사용자 쿼리] UPDATE 계좌 SET 잔액 = 0 WHERE ID = 1;                         │
│                                                                             │
│  1. 메모리 (Buffer Pool)                                                     │
│     데이터 페이지: [ ID:1, 잔액: 1000 ]  ──변경──▶ [ ID:1, 잔액: 0 ] (Dirty)   │
│     로그 버퍼:     [ LSN:101, T1 시작 ]                                        │
│                   [ LSN:102, T1 변경 (Before: 1000, After: 0) ]             │
│                                                                             │
│  2. 커밋 (COMMIT) 발생!                                                       │
│     ▶ 데이터 페이지는 아직 디스크에 안 써도 됨. (I/O 병목 방지)                     │
│     ▶ 단, **로그 버퍼**는 즉시 디스크(Log File)로 플러시(Flush) 해야 함! (WAL)   │
│                                                                             │
│  3. 디스크 (Disk Storage)                                                    │
│     로그 파일: [ LSN:101 ... LSN:102, T1 커밋 ] ◀ 영구 저장 완료!               │
│                                                                             │
│     ... (시간 경과 후 또는 체크포인트 발생 시) ...                                 │
│                                                                             │
│  4. 체크포인트 (Checkpoint) 동기화                                             │
│     ▶ 메모리의 Dirty 데이터 페이지 [ ID:1, 잔액: 0 ]을 데이터 파일에 기록         │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. ARIES 알고리즘 3단계 페이즈 (현대 DB의 회복 표준)

ARIES(Algorithm for Recovery and Isolation Exploiting Semantics)는 IBM이 개발한 회복 알고리즘으로, 현대 RDBMS(Oracle, DB2, SQL Server 등)의 기반이 됩니다. 시스템 장애로 재부팅 시 3단계를 거칩니다.

1. **Analysis Phase (분석 단계)**: 마지막 체크포인트 시점부터 로그를 끝까지 스캔하여, 장애 발생 시점에 어떤 트랜잭션이 '활성 상태(Active)'였고 어떤 데이터 페이지가 'Dirty' 상태였는지(어디부터 Redo 해야 하는지) 분석합니다.
2. **Redo Phase (재실행 단계)**: 로그를 순방향(Forward)으로 스캔하며, **성공/실패 여부를 떠나 일단 장애 발생 직전의 상태로 DB를 똑같이 재현**합니다. (Repeating History 패턴)
3. **Undo Phase (취소 단계)**: 로그를 역방향(Backward)으로 스캔하며, 분석 단계에서 알아낸 '커밋되지 않은(Active) 트랜잭션'들의 작업을 모두 취소(원상 복구)합니다. 이때 Undo 과정 자체도 로그(CLR, Compensation Log Record)로 남겨 장애 중첩 시 무한 루프를 방지합니다.

- **📢 섹션 요약 비유**: ARIES는 붕괴된 건물을 복구할 때, 먼저 현장 사진(로그)을 보고 붕괴 직전의 뼈대까지 일단 그대로 다시 지어놓고(Redo), 그 후 설계도에 없던 불법 증축 부분만 철거(Undo)하는 방식과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 로그 기반 회복 기법 비교 (지연 갱신 vs 즉시 갱신)

| 비교 항목 | 지연 갱신 (Deferred Update) | 즉시 갱신 (Immediate Update) |
|:---|:---|:---|
| **데이터 기록 시점** | 트랜잭션 **부분 완료(Commit) 후**에만 DB에 기록 | 트랜잭션 **실행 도중**에도 수시로 DB에 기록 |
| **Undo 작업 필요성** | **불필요 (No Undo)** (디스크에 미커밋 데이터가 없음) | **필요 (Undo 필수)** (디스크에 미커밋 데이터가 섞여 있음) |
| **Redo 작업 필요성** | 필요 | 필요 |
| **로그 구조** | 새 값(After Image)만 필요 | 이전 값(Before Image)과 새 값 모두 필요 |
| **현실성** | 메모리가 무한해야 하므로 실무 사용 불가 | 버퍼 캐시 관리의 유연성으로 **대부분의 DBMS 채택** |

### 2. 그림자 페이징 (Shadow Paging) 기법
로그를 사용하지 않는 대안적 복구 기법입니다. 데이터를 변경할 때 현재 디렉토리(Current Directory)가 가리키는 디스크 블록을 직접 덮어쓰지 않고, 새로운 블록을 할당받아 수정한 뒤, 커밋 시점에 디렉토리 포인터만 그림자 디렉토리(이전 버전)에서 새 디렉토리로 원자적으로 교체합니다.
- **장점**: Redo/Undo 오버헤드가 없어 커밋/복구가 매우 빠릅니다. (Copy-On-Write 방식)
- **단점**: 데이터 파편화(Fragmentation)가 심해져 디스크 연속 스캔 속도가 급감하며, 다중 사용자 병행 제어가 어렵습니다. (SQLite나 구형 파일시스템에서 제한적으로 사용)

- **📢 섹션 요약 비유**: 즉시 갱신이 틀릴 때마다 지우개로 지우며(Undo) 한 장에 쓰는 방식이라면, 그림자 페이징은 틀릴 때마다 새 종이에 처음부터 다시 쓰고(Copy) 다 쓴 뒤 예전 종이를 버리는 방식입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오
- **대규모 시스템 크래시 발생 후 기동 시간(MTTR) 지연**: 정전으로 DB가 다운된 후 재부팅했는데, ARIES Redo 과정이 1시간 이상 걸려 서비스 오픈을 못하는 상황입니다.
  - **진단**: 체크포인트 주기가 너무 길어 장애 발생 시 Redo 해야 할 로그 양이 방대해졌기 때문입니다.
  - **해결책**: DBA는 `FAST_START_MTTR_TARGET` (Oracle) 같은 파라미터를 조절하여, 체크포인트 주기를 짧게(자주 발생하게) 튜닝해야 합니다. 단, 평상시 디스크 I/O가 증가하여 성능이 소폭 하락하는 트레이드오프를 감수해야 합니다.
- **랜섬웨어 및 미디어(디스크) 파손**: 디스크 어레이가 물리적으로 파괴되거나 랜섬웨어에 감염되었습니다. 시스템 장애 복구(메모리 날아감)를 넘어서는 재난입니다.
  - **해결책**: 매일 새벽 받아둔 전체 백업(Full Dump) 파일을 복원하고, 장애 직전까지 별도 스토리지에 복제해 둔 아카이브 로그(Archived Redo Log)를 적용해 롤 포워드(Roll-forward) 복구를 수행해야 합니다 (Point-in-Time Recovery).

### 2. 안티패턴
- **운영 DB의 No-Logging 모드 남용**: 대량의 데이터 적재(Bulk Insert) 시 속도를 높이기 위해 힌트나 테이블 속성으로 Nologging을 설정하는 경우가 있습니다. 이 상태에서 미디어 장애가 발생하면 해당 데이터는 **절대 복구할 수 없습니다**. 일회성 배치 데이터에만 제한적으로 사용해야 합니다.
- **동일 디스크에 데이터 파일과 로그 파일 혼재**: 데이터 파일(랜덤 I/O)과 트랜잭션 로그 파일(순차 I/O)을 같은 물리적 디스크에 배치하면 I/O 병목이 발생하며, 디스크 손상 시 둘 다 잃어버려 복구가 불가능해집니다. 반드시 분리 격리해야 합니다.

- **📢 섹션 요약 비유**: 체크포인트 주기를 조율하는 것은 등산을 할 때 '세이브 포인트(산장)'를 얼마나 자주 들를지 결정하는 것과 같습니다. 너무 안 들르면 추락 시 처음부터 다시 올라와야 하고, 너무 자주 들르면 정상 도착(평시 성능)이 늦어집니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과

| 구분 | 회복/백업 아키텍처 미비 | 체계적인 WAL 및 아카이브 설계 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 디스크 장애 시 RPO 수일 단위 | 아카이브 로그 적용 시 RPO 0 달성 | 데이터 유실(Data Loss) **제로(0)화** |
| **정량** | 장애 시 복구(MTTR) 시간 예측 불가 | 타겟 MTTR 설정 및 튜닝 | 복구 시간(RTO) **수 분 이내로 보장** |
| **정성** | 재난 발생 시 비즈니스 영속성 타격 | 비즈니스 연속성 계획(BCP) 확립 | 엔터프라이즈급 신뢰성 및 규제 준수 |

### 2. 결론
데이터베이스 회복 기술은 평상시에는 드러나지 않지만 시스템의 신뢰성을 담보하는 최후의 보루입니다. **WAL 프로토콜**을 통한 디스크 I/O 최적화와 **ARIES 기반의 Redo/Undo 알고리즘**은 트랜잭션의 원자성과 영속성을 완벽히 조화시킨 컴퓨터 공학의 걸작입니다. 아키텍트는 단순 기능 구현을 넘어, 장애 상황에서의 RPO(목표 복구 시점)와 RTO(목표 복구 시간)를 만족할 수 있도록 로그 물리 볼륨 분리, 체크포인트 튜닝, 아카이빙 정책을 시스템 설계 초기 단계부터 통합해야 합니다.

- **📢 섹션 요약 비유**: 비행기가 추락해도 블랙박스가 튼튼하면 다음 비행은 안전해집니다. 회복 기술은 데이터베이스가 어떤 충격에도 다시 부활할 수 있게 만드는 강력한 불사조의 심장(로그)입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **트랜잭션 ACID**: 회복 기술이 궁극적으로 수호하고자 하는 원자성(A)과 영속성(D).
- **버퍼 매니저 (Buffer Manager)**: 디스크 I/O 성능을 위해 메모리 캐시를 관리하며, Checkpoint와 WAL 플러시를 통제하는 모듈.
- **스토리지 엔진 (Storage Engine)**: InnoDB 등 하부 엔진이 트랜잭션 로그와 Redo/Undo 동작을 실질적으로 구현함.
- **고가용성 분산 DB (High Availability)**: 단일 노드 복구를 넘어, 실시간 복제(Replication)를 통해 장애 시간을 0으로 수렴시키는 클라우드 환경의 발전 방향.

## 👶 어린이를 위한 3줄 비유 설명
1. **데이터베이스 장애**는 열심히 그림(데이터)을 그리고 있는데 갑자기 정전이 돼서 캄캄해지는 상황이에요.
2. **로그(Log)**는 불이 꺼지기 전에 "어떤 색을 어디에 칠했다"고 꼬박꼬박 적어둔 작은 수첩이에요.
3. 데이터베이스는 이 수첩을 보고 **이미 완성한 곳은 다시 칠하고(Redo)**, **그리다 만 곳은 지우개로 지워서(Undo)** 예쁜 그림을 완벽하게 되살려 낸답니다!
