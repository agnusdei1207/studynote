# 208. 트랜잭션 스케줄 (Schedule / History)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트랜잭션 스케줄(Schedule)은 여러 트랜잭션의 연산(Read, Write)들이 실행되는 **시간적 순서를 정의한 논리적 흐름**이다.
> 2. **가치**: 직렬(Serial) 스케줄의 일관성을 유지하면서도, 비직렬(Non-serial) 스케줄의 병렬 처리 이점을 극대화하여 시스템의 응답 속도를 높이는 판단 근거가 된다.
> 3. **융합**: '직렬 가능성(Serializability)' 이론과 결합되어, 인터리빙(Interleaving) 방식의 병행 수행이 결과적으로 안전한지 수학적으로 증명하는 도구로 쓰인다.

+++

### Ⅰ. 트랜잭션 스케줄의 개념

- **정의**: 동시에 실행되는 여러 트랜잭션의 연산들을 하나의 타임라인에 배치한 순서입니다.
- **필요성**: CPU와 디스크 자원을 놀리지 않고 여러 트랜잭션을 겹쳐서 실행(Concurrency)하기 위해, 어떤 연산을 먼저 할지 결정해야 합니다.

+++

### Ⅱ. 스케줄의 분류 (Classification)

#### 1. 직렬 스케줄 (Serial Schedule)
- 트랜잭션별로 연산을 순차적으로 수행합니다. (T1 완료 후 T2 시작)
- **특징**: 일관성은 100% 보장되지만, 동시성이 없어 성능이 매우 낮습니다.

#### 2. 비직렬 스케줄 (Non-serial Schedule)
- 트랜잭션들의 연산을 섞어서 실행(Interleaving)합니다.
- **특징**: 자원 활용도가 높으나, 잘못 섞으면 데이터가 꼬일 위험이 큽니다.

#### 3. 직렬 가능 스케줄 (Serializable Schedule)
- 비직렬 방식이지만, 그 결과가 어떤 직렬 스케줄과 동일한 경우입니다.
- **목표**: 모든 동시성 제어의 궁극적인 지향점입니다.

```text
[직렬 vs 비직렬 스케줄 시각화]

  (Serial)         (Non-serial / Interleaved)
  T1: R(A)         T1: R(A)
  T1: W(A)         T2: R(A) ◀─ 섞임!
  T2: R(A)         T1: W(A)
  T2: W(A)         T2: W(A)
```

+++

### Ⅲ. 스케줄의 정당성 판별: 직렬 가능성 (Serializability)

- **충돌 직렬 가능 (Conflict Serializability)**: 연산의 순서를 바꿔도 결과에 영향이 없는 연산들끼리 재배치하여 직렬 스케줄로 만들 수 있는지 확인합니다.
- **뷰 직렬 가능 (View Serializability)**: 두 스케줄이 동일한 데이터를 읽고 쓰며, 최종 결과가 같은지 비교합니다. (충돌 직렬보다 넓은 개념)

- **📢 섹션 요약 비유**: 스케줄은 **'여러 명의 요리사가 주방 하나를 나눠 쓰는 시간표'**와 같습니다. 한 명이 요리를 다 끝내고 다음 사람이 들어오는 게 '직렬'이고, 불과 칼을 나눠 쓰며 복잡하게 움직이는 게 '비직렬'입니다. 시간표를 잘 짜야 요리가 타지 않고(데이터 무결성) 빨리(성능) 나옵니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[인터리빙]**: 연산을 교차로 배치하는 핵심 기술.
- **[직렬화]**: 비직렬을 직렬처럼 안전하게 만드는 과정.
- **[회복 가능성]**: 장애 시 롤백이 가능한 스케줄인지 판단하는 또 다른 척도.

📢 **마무리 요약**: 스케줄은 병행 제어의 이론적 뼈대입니다. 시스템은 **Serializable**한 스케줄을 생성하여 성능과 정확성의 완벽한 균형을 유지해야 합니다.
