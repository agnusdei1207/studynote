# 250. 코디네이터와 참여자 - 분산 조율의 이중주

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 시스템에서 코디네이터(Coordinator)는 전체 트랜잭션의 **의사결정을 주도하는 관리자**이며, 참여자(Participant)는 자신의 로컬 자원에서 **실제 연산을 수행하고 결과를 보고하는 하부 노드**이다.
> 2. **가치**: 코디네이터는 분산된 노드들의 상태를 수집하여 'Global Decision'을 내리고, 참여자는 'Local Atomicity'를 책임짐으로써 시스템 전체의 정합성을 완성한다.
> 3. **융합**: 이들의 상호작용은 2PC나 3PC 프로토콜의 메시지 교환 규칙에 의해 정의되며, 코디네이터의 생존 여부가 분산 가용성의 결정적 변수가 된다.

+++

### Ⅰ. 역할 정의 (Roles)

#### 1. 코디네이터 (Coordinator / Manager)
- **책임**: 트랜잭션 시작, Prepare 메시지 전송, 결과 취합, 최종 Commit/Rollback 결정 및 전파.
- **특징**: 전체 트랜잭션의 문맥(Context)을 소유하며, 모든 참여자의 응답을 기다리는 '지휘자'입니다.

#### 2. 참여자 (Participant / Resource Manager)
- **책임**: 코디네이터의 지시에 따른 로컬 연산 수행, 준비 상태 보고, 지시에 따른 최종 확정 또는 취소.
- **특징**: 데이터가 실제로 저장된 물리적 노드이며, 자신의 로컬 로그에 상태를 기록하여 복구에 대비합니다.

+++

### Ⅱ. 코디네이터-참여자 상호작용 (ASCII Flow)

```text
[Coordinator vs Participant Interaction]

  (Coordinator)              (Participant 1)        (Participant 2)
        │                          │                      │
        │ ─── 1. Prepare ───▶      │                      │
        │ ─── 1. Prepare ──────────────────────────────▶  │
        │                          │                      │
        │ ◀── 2. Ready/No ───      │                      │
        │ ◀── 2. Ready/No ──────────────────────────────  │
        │                          │                      │
        │ (Decision: Commit)       │                      │
        │                          │                      │
        │ ─── 3. Commit ────▶      │                      │
        │ ─── 3. Commit ───────────────────────────────▶  │
        ▼                          ✅                     ✅
```

### Ⅲ. 아키텍처적 고려사항

1. **내결함성 (Fault Tolerance)**: 참여자가 죽으면 코디네이터는 해당 트랜잭션을 롤백하면 되지만, **코디네이터가 죽으면** 모든 참여자는 지시를 기다리며 락을 쥔 채 굳어버립니다.
2. **로그 기록**: 코디네이터는 결정 사항을 자신의 로그에 먼저 써야 하며, 참여자 역시 자신의 준비 상태를 로그에 영구 저장해야 장애 후 복구가 가능합니다.

- **📢 섹션 요약 비유**: 코디네이터와 참여자는 **'합창단의 지휘자와 단원'**과 같습니다. 지휘자가 손을 들어 신호를 주면(Prepare), 단원들은 숨을 들이마시고 부를 준비를 마친 뒤 고개를 끄덕입니다(Ready). 지휘자가 팔을 휘두를 때(Commit) 비로소 노래가 시작됩니다. 지휘자가 갑자기 쓰러지면 단원들은 입만 벌린 채 얼어붙게 됩니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Global Transaction ID]**: 코디네이터가 발급하는 전체 작업의 이름표.
- **[Vote]**: 참여자가 코디네이터에게 보내는 가부 결정권.
- **[SPOF]**: 코디네이터에게 집중된 장애 리스크.

📢 **마무리 요약**: **Coordinator & Participant** 구조는 분산 시스템의 통치 모델입니다. 이들의 긴밀한 통신이 데이터베이스의 거대한 무결성을 만들어냅니다.
