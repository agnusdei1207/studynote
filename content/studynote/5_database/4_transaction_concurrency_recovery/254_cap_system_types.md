# 254. CP vs AP vs CA 시스템 - 분산 데이터베이스의 성격

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CAP 정리에 따라 분산 시스템은 **일관성(C), 가용성(A), 분단 허용성(P)** 중 두 가지만을 최우선으로 선택할 수 있으며, 이에 따라 시스템의 성격이 결정된다.
> 2. **가치**: 데이터의 절대적 정확성이 필요한 곳(CP), 서비스의 끊김 없는 연결이 중요한 곳(AP) 등 비즈니스 도메인에 최적화된 데이터 저장소를 선택하는 혜안을 제공한다.
> 3. **융합**: 하드웨어 인프라(P)와 소프트웨어 알고리즘(C, A)의 설계 철학이 융합된 결과물이며, 현대 폴리글랏 퍼시스턴스(Polyglot Persistence) 아키텍처의 구성 요소가 된다.

+++

### Ⅰ. 유형별 상세 특징 및 사례

#### 1. CP 시스템 (Consistency + Partition Tolerance)
- **철학**: "네트워크가 끊기면, 데이터가 틀릴 수 있으니 차라리 서비스를 멈추겠다."
- **특징**: 강력한 일관성을 보장하지만, 장애 발생 시 일부 사용자의 요청이 거절될 수 있습니다.
- **사례**: **HBase, MongoDB(기본), BigTable, Redis**.

#### 2. AP 시스템 (Availability + Partition Tolerance)
- **철학**: "네트워크가 끊겨도, 일단 서비스는 유지하겠다. 데이터는 나중에 맞추면 된다."
- **특징**: 뛰어난 가용성을 제공하지만, 사용자가 잠시 낡은 데이터를 볼 수 있습니다 (Eventually Consistent).
- **사례**: **Cassandra, DynamoDB, CouchDB**.

#### 3. CA 시스템 (Consistency + Availability)
- **철학**: "네트워크는 절대 끊기지 않는다고 가정하고, 일관성과 가용성을 모두 잡겠다."
- **특징**: 분산 시스템이 아닌 **단일 노드 시스템**에 가깝습니다. 네트워크 분할 상황(P)이 오면 시스템 전체가 무너집니다.
- **사례**: **전통적 RDBMS (MySQL, Oracle - 단일 클러스터)**.

+++

### Ⅱ. 시스템 유형 비교 (ASCII Flow)

```text
[네트워크 분리(P) 시나리오 대응 비교]

      [ User Request ]
             │
      ┌──────┴──────┐
      ▼             ▼
  [ Node A ]  X─[Partition]─X  [ Node B ]
  (Updated)                    (Old Data)

  - CP 선택 시: Node B는 "에러" 응답 ──▶ (일관성 수호)
  - AP 선택 시: Node B는 "옛날 값" 응답 ──▶ (가용성 수호)
```

+++

### Ⅲ. 실무적 선택 가이드

- **금융/결제/원장**: 무조건 **CP** 시스템 지향.
- **SNS/댓글/조회수**: 사용자 경험을 위해 **AP** 시스템 지향.
- **설정 정보/코디네이터**: 정합성이 생명이므로 **CP** 시스템(etcd, ZooKeeper) 필수.

- **📢 섹션 요약 비유**: CP는 **'법규를 철저히 지키느라 조금 늦는 모범생'**이고, AP는 **'일단 일을 빨리 처리하고 나중에 실수를 바로잡는 실무가'**이며, CA는 **'섬에 갇혀서 밖의 세상과 소통할 필요가 없는 평화로운 마을 주민'**과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Eventually Consistent]**: AP 시스템의 핵심 키워드.
- **[Strong Consistency]**: CP 시스템의 자부심.
- **[분산 시스템의 Fallacy]**: 네트워크는 절대 끊기지 않는다는 믿음(CA의 착각)을 경계해야 함.

📢 **마무리 요약**: **CP, AP, CA**의 구분은 데이터베이스 선정의 '바이블'입니다. 시스템이 처한 물리적 환경과 비즈니스 가치를 고려한 현명한 선택이 필요합니다.
