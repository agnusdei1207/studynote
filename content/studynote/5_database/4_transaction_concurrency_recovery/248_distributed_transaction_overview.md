# 248. 분산 트랜잭션 (Distributed Transaction) - 파편화된 원자성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 트랜잭션은 물리적으로 분리된 **둘 이상의 데이터베이스 노드나 서비스에 걸쳐 수행되는 작업들을 하나의 논리적인 단위로 묶는 기술**이다.
> 2. **가치**: "전체가 성공하거나 전체가 실패해야 한다"는 원자성을 네트워크 너머의 시스템들 사이에서도 강제하여 데이터 불일치 위험을 방어한다.
> 3. **융합**: 마이크로서비스 아키텍처(MSA)의 확산과 함께 2단계 커밋(2PC)이나 사가(Saga) 패턴과 같은 고도의 조율 프로토콜과 결합되어 진화하고 있다.

+++

### Ⅰ. 분산 트랜잭션의 개념 및 특징

- **정의**: 로컬 DB 1대가 아닌, 여러 대의 서버(Node)가 참여하는 트랜잭션입니다.
- **특징**:
    - **네트워크 의존**: 노드 간 메시지 교환으로 인해 지연 시간(Latency)이 발생합니다.
    - **부분 실패 위험**: 일부 노드는 성공하고 일부는 실패했을 때 이를 복구하는 과정이 매우 복잡합니다.
    - **코디네이터의 존재**: 각 노드의 상태를 취합하고 최종 결정을 내리는 관리자가 필요합니다.

+++

### Ⅱ. 분산 트랜잭션 아키텍처 (ASCII Flow)

```text
[분산 트랜잭션의 기본 구조]

     [ Global Transaction Manager (GTM) ] ◀───▶ [ Client ]
               │              │
       ┌───────┴──────────────┴───────┐
       ▼                              ▼
  [ Node A (DB) ]                [ Node B (DB) ]
  (Local Transaction)            (Local Transaction)
```

+++

### Ⅲ. 분산 트랜잭션 제어 기법

1. **2단계 커밋 (2PC)**: 모든 노드가 준비될 때까지 기다렸다가 한꺼번에 커밋합니다. (강한 일관성)
2. **Saga 패턴**: 각 단계별로 로컬 커밋을 치고, 실패 시 보상 트랜잭션을 통해 원복합니다. (결과적 일관성)
3. **TCC (Try-Confirm/Cancel)**: 예약(Try) 단계를 거쳐 확정 또는 취소를 수행하는 비즈니스 레벨의 제어입니다.

- **📢 섹션 요약 비유**: 분산 트랜잭션은 **'여러 명이 한꺼번에 발을 맞춰 뛰는 단체 줄넘기'**와 같습니다. 한 명이라도 발이 걸리면(부분 실패), 줄을 멈추고 다 같이 처음부터 다시 시작해야 하는 긴밀한 협동이 필요합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[원자성]**: 분산 환경에서 가장 지키기 어려운 가치.
- **[CAP 정리]**: 분산 트랜잭션이 성능과 일관성 사이에서 고민하게 만드는 근거.
- **[XA 표준]**: 분산 트랜잭션을 위한 이기종 시스템 간의 통신 규약.

📢 **마무리 요약**: **Distributed Transaction**은 현대 분산 아키텍처의 숙제이자 핵심입니다. 시스템의 복잡도를 감수하더라도 데이터의 무결성을 지키기 위한 최후의 보루입니다.
