## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Saga 패턴은 분산 환경에서 여러 서비스에 걸친 트랜잭션을 관리하기 위해, 각 단계마다 보상 트랜잭션(Compensating Transaction)을 정의하여 부분 실패 시 이전 단계를 역순으로 취소한다.
> 2. **가치**: 2PC의 blocking 문제 없이 분산 트랜잭션의 결과적 일관성을 보장한다.
> 3. **융합**: 마이크로서비스 아키텍처에서跨서비스 데이터 일관성을 달성하는標準 패턴이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념
Saga 패턴은 1987년 Hector Garcia-Molina와 Kenneth Salem이 발표한 개념으로, 长年全球 분산 트랜잭션을 관리하기 위한 방법론이다. 전체를单个의 원자적 트랜잭션으로 묶는 대신, 각 단계의"보상(Compensation)"을 미리 정의하여, 어떤 단계에서 실패하면 이전 단계의 보상을 역순으로 실행한다.

### 필요성
마이크로서비스 환경에서는 여러 서비스에データを跨서 작업이 필요한 경우가 많다. 그러나 2PC는 blocking 문제와 결합 실패 위험이 있어, 대규모 분산 환경에서는 비실용적이다. Saga는 이 문제를 보상 트랜잭션으로 해결한다.

### 섹션 요약 비유
Saga 패턴은旅行代理店の予約システムと같て、各プロバイダー（航空会社、酒店、保险）を個別に予約し、いずれかが失敗した場合は以前のプロバイダーを逆向にキャンセルして、全体の整合性を保つ。

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Saga 동작 구조

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    Saga 패턴 동작 과정                                    │
  ├───────────────────────────────────────────────────────────────────────┤
  │
  │   [정상 흐름]                                                        │
  │
  │   Step 1: 항공권 예약          ──▶ 성공                                │
  │       │                                                             │
  │       ▼                                                             │
  │   Step 2: 호텔 예약            ──▶ 성공                                │
  │       │                                                             │
  │       ▼                                                             │
  │   Step 3: 보험 가입           ──▶ 성공                                │
  │       │                                                             │
  │       ▼                                                             │
  │   완료                                                                  │
  │
  │   [실패 및 보상 흐름]                                                  │
  │
  │   Step 1: 항공권 예약          ──▶ 성공                                │
  │       │                                                             │
  │       ▼                                                             │
  │   Step 2: 호텔 예약            ──▶ ❌ 실패!                            │
  │       │                                                             │
  │       ▼                                                             │
  │   Compensate Step 1: 항공권 예약 취소  ──▶ 성공                        │
  │       │                                                             │
  │       ▼                                                             │
  │   실패 처리                                                          │
  │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Saga의 핵심은"사전 보상 정의"다. 각 단계에서 어떤 일이 실패할 경우를 대비하여, 해당 단계의"undo"操作을 미리 정의한다. 위 예시에서 호텔 예약이 실패하면, 이미 성공한 항공권 예약을 취소하는 보상 트랜잭션을 실행한다. 중요한 점은,Saga는 원자성을 보장하지 않는다. 즉, 전체가 성공하거나 또는 보상까지 모두 완료된"결과적 일관성(Eventual Consistency)"만을 보장한다. 중간에 보상 자체도 실패하면"사건이 발생한" 상황이 되며, 이는 비즈니스 수준에서 별도로 처리해야 한다.

### Saga vs 2PC

| 구분 | Saga | 2PC |
|:---|:---|:---|
| **Blocking** | 없음 | 있음 |
| **원자성** | 결과적 일관성 | 완전한 원자성 |
| **복구** | 보상 트랜잭션 | 자동 UNDO/REDO |
| **지연** | 각 단계별 독립 | 전체 대기 |

### 섹션 요약 비유
Saga와 2PC의 차이는"보험 가입 확인"과"현장 결제"의 차이와 같다. Sagaは各段階で部分確認して问题和場合は以前をキャンセル하지만、2PCは全场打完後に全結果を同時に确定する。

## Ⅲ. 융합 비교 및 다각도 분석

### Saga 구현 방식

| 방식 | 설명 | 특징 |
|:---|:---|:---|
| **Choreography** | 각 서비스가 이벤트를发布하고他のサービスがサブスクライブする | 分散型、拡張性 높음 |
| **Orchestration** | 중앙 오케스트레이터가 전체 흐름을 조정 | 중앙集权적, 순서 명확 |

### 섹션 요약 비유
Choreographyは各县が勝手に广播して各自判断하고、OrchestrationはNHKが全程을 연출指挥하는 것과 같다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 — 전자상거래 주문**: 주문 서비스가 Saga 오케스트레이터로 동작하여, (1) 재고 확인 → (2) 결제 처리 → (3) 배송 예약의 단계를 순차 실행한다. 어느 단계에서 실패하면 이전 단계의 보상을 실행한다.

### 안티패턴
- **보상 미정의**: 실패 시 보상 트랜잭션이 정의되지 않으면"半完成状態"에 놓일 수 있다.
- **보상 실패**: 보상 자체가 실패하면 비즈니스 수준에서의 조치가 필요하다.

### 섹션 요약 비유
Saga 패턴의限界는部分保険의限界と같다. 全行程のどの段階で问题发生しても以前を全部无にできない可能性がある。

## Ⅴ. 결론

Saga 패턴은 분산 환경에서 2PC의 한계를 보완하는 유연한 방법을 제공하지만, 원자성이 아닌 결과적 일관성만을 보장한다는 한계를 가진다. 따라武士缺的应用场景에 선택적으로 使用해야 한다.

### 섹션 요약 비유
Saga 패턴은旅行スケジュールの卷物と같다. 어디든問題发生으면 previousき位置まで戻って卷物を消し、全体の一贯性を保とうとする。

## 👶 어린이를 위한 3줄 비유 설명
1. Saga 패턴은 **롤러코스터의 사전 점검**과 같아서, 각 턱에서 안전장치가 작동하지 않으면 이전 턱에서 내려야 해요.
2. 놀이기구에서 한 단계에서 고장이 나면 이전 단계도 취소되는데, 이것이 바로 보상 트랜잭션이에요.
3. 하지만 놀이기구를 타다가 고장이 나면(部分 실패) 완전히 처음부터 다시 타야 하는 것처럼(결과적 일관성), Saga도 완전한 원자성을 보장하지는 않아요!
