## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 중첩 트랜잭션(Nested Transaction)은 부모 트랜잭션 내에 자식 트랜잭션이 존재하는 계층 구조로, 각 자식은 독립적으로 COMMIT/ROLLBACK이 가능하다.
> 2. **가치**: 대규모 작업의 部分실패 시 전체 롤백 없이 해당 부분만 재시도할 수 있다.
> 3. **융합**: 마이크로서비스 아키텍처의 Saga 패턴과 유사한 部分복구 개념을 적용한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념
중첩 트랜잭션(Nested Transaction)는 상위 트랜잭션(Parent) 내에 하위 트랜잭션(Child)이 존재하는 트랜잭션 모델이다. 자식 트랜잭션은 독립적으로 COMMIT이나 ROLLBACK이 가능하며, COMMIT은 부모에게만 알려지고, 부모가 롤백되면 자식의 결과도 모두 취소된다.

### 필요성
복잡한 비지니스 로직에서 전체 작업의 部分만 실패했을 때, 나머지 성공한 部分까지 롤백하면 비효율적이다. 중첩 트랜잭션은 部分실패 시 해당 부분만 재시도할 수 있게 한다.

### 섹션 요약 비유
중첩 트랜잭션은 복합 악기의 연주와 같다. 첼로(부모)가 하모니카(자식)를 포함한 合奏全程을 管理하지만, 하모니카만 문제가 있으면 하모니카만 다시 불면 된다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 동작 구조

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    중첩 트랜잭션 구조                                    │
  ├───────────────────────────────────────────────────────────────────────┤
  │
  │   [트랜잭션 계층]                                                    │
  │
  │   Parent Transaction                                                │
  │       │                                                             │
  │       ├── Child Transaction 1                                        │
  │       │       └── Sub-Child 1-1                                    │
  │       │       └── Sub-Child 1-2                                    │
  │       │                                                             │
  │       └── Child Transaction 2                                        │
  │               └── Sub-Child 2-1                                    │
  │
  │   [제약사항]                                                        │
  │   • 자식 COMMIT은 부모에게만 알려짐 (상위 트랜잭션이 최종 결정)         │
  │   • 부모 ROLLBACK 시 모든 자식의 결과도 취소                           │
  │   • 자식 ROLLBACK 시 다른 자식은 영향받지 않음                         │
  │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 중첩 트랜잭션의 핵심은"부분 커밋"과"계층적 롤백"이다. Child1이 COMMIT되어도 Parent가 최종 결정권을 가지므로, Parent가 ROLLBACK되면 Child1의 결과도 취소된다. 그러나 Child1이 ROLLBACK되어도 Parent와 Child2는影响받지 않는다. 이것이"부분 실패 시 부분 재시도"를 가능케 한다. 그러나 대부분의 RDBMS는 중첩 트랜잭션을 지원하지 않으며, 필요하다면 SAVEPOINT를 사용한다.

### 섹션 요약 비유
중첩 트랜잭션은 조직도와 같다. 차장(Parent)이 팀장(Child)을 管理하고, 팀장이사원(Sub-Child)을 管理한다. 차장이 퇴사하면(ROLLBACK) 팀장과 사원 모두 대체하지만, 팀장이 퇴사하면(Child ROLLBACK) 차장과 다른 팀은影響받지 않는다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: Flat vs Nested vs Saga

| 구분 | Flat | Nested | Saga |
|:---|:---|:---|:---|
| **구조** | 단일 레벨 | 계층적 | choreographed |
| **부분 롤백** | 전체만 | 가능 | compensating 연산 |
| **ACID 보장** | 완전 | 부분적 | 결과적 |

### 섹션 요약 비유
세 트랜잭션 모델은 조직 구조와 같다. Flatは全て同じ部署で、Nestedは課と班の階層が、Sagaは各チームが独立して通信する。

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 — 복잡한 주문 처리**: 주문 트랜잭션(Parent) 내에 결제 트랜잭션(Child1)과 재고 확인(Child2)가 있다. 결제 실패 시 Child1만 롤백하고, 다른 商品으로 재시도할 수 있다.

### 섹션 요약 비유
中철트랜잭션の취약성은 조직의atar政策と 같다. 상급자가 변경되면(CHILD 롤백) 하급자의 업무도全部取り消される場合がある。

---

## Ⅴ. 결론

중첩 트랜잭션은 유연한 오류 처리가 가능한 장점이 있지만, 대부분의 RDBMS에서 미지원이므로 실무에서는 SAVEPOINT나 Saga 패턴으로우회하는 것이 일반적이다.

### 섹션 요약 비유
중첩 트랜잭션은Delegation과같아서,Manager가Task를Delegate하면Subordinate는독립적으로作業하지만, Manager가Task를취소하면Subordinate의작업도无效다.

## 👶 어린이를 위한 3줄 비유 설명
1. 중첩 트랜잭션은 **게임의 미니게임**과 같아서, 메인 게임 안에 미니게임이 있고, 미니게임에서 지면 그 미니게임만 다시 하면 돼요.
2. 그런데 메인 게임이 종료되면(부모 롤백) 미니게임 결과도 함께 사라져요.
3. 만약 미니게임에서 지면(자식 롤백) 다른 미니게임에는影響없이 계속 할 수 있어요!
