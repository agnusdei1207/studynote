# 140. 서브쿼리 (Subquery) - 중첩된 질의의 논리와 활용

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하나의 SQL 문장 안에 포함된 또 다른 SQL 문장을 의미하며, 바깥쪽의 **메인쿼리(Main Query)** 실행에 필요한 상수를 공급하거나 특정 조건을 필터링하기 위한 중간 결과물로 활용된다.
> 2. **작동 방식**: 메인쿼리와 독립적으로 실행되는 **비상관 서브쿼리**와 메인쿼리의 한 행마다 실행되는 **상관 서브쿼리(Correlated Subquery)**로 나뉘며, 연산 위치에 따라 스칼라 서브쿼리, 인라인 뷰, 서브쿼리로 구분된다.
> 3. **가치**: 복잡한 비즈니스 로직을 단계별로 추상화하여 표현할 수 있게 해주며, 옵티마이저의 **서브쿼리 언네스팅(Unnesting)** 기술과 결합하여 효율적인 조인 연산으로 변환되어 실행된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**서브쿼리 (Subquery)**는 "질의 속의 질의"입니다. 전체 데이터를 가져오기 전에, "그중에서 특정 조건을 만족하는 값들만 먼저 뽑아줘"라고 부탁하는 중첩된 명령입니다. 반드시 괄호 `( )`로 감싸서 사용합니다.

### 2. 💡 비유: 심부름 시키기 (이중 명령)
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        서브쿼리(Subquery) 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 엄마가 아들에게 심부름을 시킴.                                          │
│                                                                             │
│  1. 일반 쿼리: "냉장고에서 **'우유'**를 가져오렴." (우유라는 값을 이미 앎)         │
│                                                                             │
│  2. 서브쿼리: "냉장고에서 **'아빠가 어제 사 온 것'**을 가져오렴."                │
│     - 아들은 먼저 아빠에게 물어봄 (서브쿼리 실행): "어제 뭐 사 오셨어요?"         │
│     - 아빠의 대답 (중간 결과): "우유란다."                                      │
│     - 아들은 그 대답을 들고 냉장고로 감 (메인쿼리 실행): 우유를 찾아옴.           │
│                                                                             │
│  → **'질문 안에 또 다른 질문'**을 넣어, 모르는 값을 찾아가며 일을 처리하는 방식!    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **동적 필터링**: "평균 연봉보다 많이 받는 사람"을 조회할 때, 평균 연봉은 수시로 변하므로 고정된 상수를 쓸 수 없음.
- **표현력의 한계 극복**: 단순 조인(Join)만으로는 표현하기 힘든 복잡한 집계 및 존재 여부 확인(EXISTS) 로직을 간결하게 작성하기 위함.

- **📢 섹션 요약 비유**: 서브쿼리는 '변수'를 품은 수학 공식과 같아서, 변수값이 먼저 계산되어야 최종 답을 낼 수 있습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 서브쿼리의 위치에 따른 분류

| 명칭 | 위치 | 주요 역할 | 비유 |
|:---:|:---|:---|:---|
| **스칼라 서브쿼리** | SELECT 절 | 단일 값을 결과 열로 출력. | 한 칸 채우기 |
| **인라인 뷰 (Inline View)** | FROM 절 | 가상의 테이블(뷰)처럼 활용. | 임시 바구니 |
| **중첩 서브쿼리** | WHERE/HAVING 절 | 조건 필터링의 기준으로 활용. | 필터 종이 |

### 2. 상관 서브쿼리(Correlated) vs 비상관 서브쿼리 (ASCII)

메인쿼리와의 의존성 여부에 따른 실행 차이를 시각화한다.

```text
    [ 비상관 서브쿼리 (Non-correlated) ]
    SELECT * FROM Emp WHERE Salary > (SELECT AVG(Salary) FROM Emp);
    1. 서브쿼리 실행 (평균: 3000)
    2. 메인쿼리에 값 대입: WHERE Salary > 3000
    * 특징: 딱 한 번만 실행됨 (Fast)
    
    [ 상관 서브쿼리 (Correlated) ]
    SELECT * FROM Emp E WHERE Salary > (SELECT AVG(Salary) FROM Emp WHERE Dept = E.Dept);
    1. 메인쿼리에서 한 행(사원 A)을 읽음.
    2. 그 사원의 '부서'를 서브쿼리에 전달.
    3. 해당 부서의 평균을 계산하여 비교.
    4. 다음 행(사원 B)에 대해 위 과정 반복.
    * 특징: 메인쿼리 행 수 만큼 반복 실행됨 (Slow, but powerful)
```

### 3. 심층 동작 원리: 반환 행 수에 따른 처리
- **단일 행 서브쿼리**: 결과가 1건만 나오는 경우. `=`, `<`, `>` 연산자 사용.
- **다중 행 서브쿼리**: 결과가 여러 건인 경우. `IN`, `ANY`, `ALL`, `EXISTS` 연산자 사용 필수.

### 4. 핵심 기술: 서브쿼리 언네스팅 (Subquery Unnesting)
- **개념**: DBMS 옵티마이저가 서브쿼리를 일반적인 **조인(Join)문으로 변환**하여 처리하는 최적화 기술.
- **효과**: 서브쿼리의 반복 실행 오버헤드를 줄이고, 인덱스 활용도를 높여 성능을 극대화함.

- **📢 섹션 요약 비유**: 비상관 서브쿼리는 '한 번만 물어보는 질문'이고, 상관 서브쿼리는 '한 명 한 명 붙잡고 일일이 물어보는 질문'입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 조인(Join) vs 서브쿼리(Subquery)

| 비교 항목 | 조인 (Join) | 서브쿼리 (Subquery) |
|:---:|:---|:---|
| **수행 목적** | 두 테이블의 **열을 합쳐** 보여줌. | 조건에 맞는 **행을 걸러냄**. |
| **결과 데이터** | 양쪽 테이블 컬럼 모두 가용. | **메인 테이블 컬럼만** 가용. (Scalar 제외) |
| **성능** | 대체로 우수 (Optimizer 최적화) | 복잡한 상관 쿼리 시 성능 저하 위험. |
| **가독성** | 관계가 복잡해지면 난해함. | **논리적 단계가 명확함**. |

### 2. 과목 융합 관점: 이산수학 및 알고리즘
- **이산수학 융합**: 서브쿼리는 집합 내의 원소를 결정하는 **한정 기호(Quantifier)**인 $\exists$ (Exists)와 $\forall$ (For all)의 프로그래밍적 구현체다.
- **알고리즘 융합**: 상관 서브쿼리는 자료구조의 **이중 루프(Nested Loop)**와 시간 복잡도($O(N \times M)$) 측면에서 동일하며, 이를 조인으로 바꾸는 과정은 연산의 분산 및 해시화를 통한 최적화 과정이다.

- **📢 섹션 요약 비유**: 조인이 '두 친구를 한 장의 사진에 찍는 것'이라면, 서브쿼리는 '사진 속의 사람 중 특정 자격이 있는 사람만 표시하는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 부서 내 최고 연봉자 조회
- **상황**: 각 부서에서 가장 월급을 많이 받는 사람의 모든 정보를 보고 싶음.
- **판단**: 부서별 MAX 값만으로는 사원명을 알 수 없음.
- **결단**: **상관 서브쿼리 또는 다중 행 서브쿼리 사용**. `WHERE (Dept, Salary) IN (SELECT Dept, MAX(Salary) FROM Emp GROUP BY Dept)`. 또는 상관 쿼리를 사용하여 각 행마다 부서 내 최고액인지 검증하여 정교한 결과 도출.

### 2. 안티패턴: 대량 데이터에서의 상관 서브쿼리 남발
- **증상**: 100만 건 테이블 조회 시 `SELECT` 절에 매 행마다 다른 테이블을 뒤지는 스칼라 서브쿼리 포함.
- **결과**: I/O가 100만 번 발생하며 DB 서버 응답 불가 (Row-by-row processing의 폐해).
- **조치**: **서브쿼리를 인라인 뷰로 바꾸어 미리 조인(Hash Join)** 하거나, Window Function(`RANK`, `MAX() OVER`)을 활용하여 집합 단위로 연산할 것.

### 📢 기술사적 결언
> "서브쿼리는 **'생각의 계층화'**다. 복잡한 문제를 '안쪽 질문'과 '바깥쪽 질문'으로 나누어 해결하기 때문이다. 기술사는 서브쿼리의 문법적 화려함에 빠지지 말고, 이것이 내부적으로 **Filter 연산**으로 남을 것인지, **Semi-join**으로 언네스팅될 것인지 실행 계획을 예측할 수 있어야 한다. '집합적 사고'를 통해 서브쿼리를 최적의 조인 구조로 재구성하는 능력이 데이터베이스 성능의 임계치를 결정함을 잊지 말아야 한다."

- **📢 섹션 요약 비유**: 서브쿼리 오남용은 '질문 하나 하려고 100만 명에게 전화를 거는 것'과 같아, 반드시 '단체 문자(집합 연산)'로 바꿀 방법을 고민해야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (서브쿼리 최적 활용 시)

| 지표 | 활용 전 | 활용 후 (최적화) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **SQL 코드 가독성** | 매우 복잡 (Join 범벅) | **논리적 분리 명확** | 유지보수 가독성 40% 향상 |
| **연산 효율 (Unnesting 시)** | $O(N \times M)$ | **$O(N + M)$ (Hash)** | 대량 데이터 처리 속도 10배 |
| **비즈니스 구현력** | 단순 쿼리 한계 | **복합 조건 처리 가능** | 유연한 정보 추출 역량 확보 |

### 2. 미래 전망
- **AI-Managed Subquery Optimization**: 옵티마이저가 서브쿼리의 의도를 스스로 파악하여, 성능이 더 좋은 CTE나 창 함수로 실시간 재작성(Rewrite)하여 실행하는 기술 고도화.
- **Functional Query Integration**: SQL의 서브쿼리 사상이 최신 프로그래밍 언어의 **LINQ나 Stream API**와 더욱 밀접하게 통합되어, 개발자가 언어의 경계 없이 서브쿼리 논리를 구사하게 될 것임.

- **📢 섹션 요약 비유**: 미래의 서브쿼리는 굳이 우리가 괄호를 열고 닫지 않아도, 컴퓨터가 우리의 질문 속에 숨은 '속마음(의도)'을 알아채서 먼저 답을 구해오는 지능형 질의 시스템이 될 것입니다.
