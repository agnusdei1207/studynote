# 167. 인덱스 선정 기준 - 선택도, 카디널리티, 활용 빈도

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 조회 성능을 최적화하기 위해, 어떤 컬럼에 인덱스를 생성할 것인지 결정하는 전략적 판단 과정이며, **선택도(Selectivity)**와 **카디널리티(Cardinality)**라는 정량적 지표를 근거로 한다.
> 2. **최적의 조건**: 중복된 값이 적고(High Cardinality), 검색 결과가 전체의 5~10% 이내로 좁혀지는(Low Selectivity) 컬럼이 최우선 선정 대상이며, 조인(Join)과 정렬(Sort)에 자주 사용되는 컬럼일수록 인덱스 가치가 높다.
> 3. **가치**: 무분별한 인덱스 생성이 유발하는 입력/수정 성능 저하를 방지하고, 최소한의 인덱스로 최대한의 쿼리 가독성과 물리적 응답 속도를 확보하는 **데이터 아키텍처 효율화**의 핵심이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**인덱스 선정 기준**은 테이블의 수많은 컬럼 중 "어떤 컬럼에 길잡이(Index)를 세울 것인가"를 정하는 규칙입니다. 인덱스는 공짜가 아니므로(저장 공간 소모, 수정 성능 저하), 반드시 **'투자 대비 효과(ROI)'**가 확실한 컬럼만 선별해야 합니다.

### 2. 💡 비유: 도서관의 '검색 키워드' 선정
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        인덱스 선정 기준 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 수만 권의 책이 있는 도서관에서 '찾기 쉬운 목록'을 만들려 함.              │
│                                                                             │
│  1. 나쁜 기준 (성별): "남자 주인공 책", "여자 주인공 책"                         │
│     - 목록을 봐도 절반(50%)이나 나옴. 결국 서가를 다 뒤져야 함. (선택도 낮음)      │
│                                                                             │
│  2. 좋은 기준 (고유번호/제목): "ISBN 번호", "특이한 제목"                       │
│     - 목록을 보면 딱 그 책 한 권만 나옴. (카디널리티 높음 / 선택도 높음)           │
│                                                                             │
│  → **'한 번에 범위를 팍 줄여줄 수 있는'** 정보에만 인덱스라는 이름표를             │
│     붙여주는 것이 지혜로운 도서관 관리법입니다!                                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **저장 및 연산 비용의 발생**: 인덱스는 테이블과 별도의 공간을 차지하며, 데이터가 변할 때마다 인덱스도 계속 업데이트해줘야 하므로 시스템에 큰 부담을 줌.
- **옵티마이저의 한계**: 인덱스가 너무 많으면 옵티마이저가 오히려 잘못된 길을 선택(Over-modeling)할 위험이 커짐.

- **📢 섹션 요약 비유**: 인덱스 선정은 '전쟁터에서 가장 중요한 전략적 요충지에만 초소를 세우는 것'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 인덱스 선정을 위한 3대 핵심 지표

| 지표명 | 정의 및 공식 | 이상적인 상태 | 비유 |
|:---:|:---|:---:|:---|
| **카디널리티 (Cardinality)** | 전체 행 중 중복을 제외한 유일한 값의 개수. | **높을수록 좋음** (Unique) | 지문의 유일성 |
| **선택도 (Selectivity)** | (1 / 카디널리티) * 100 | **낮을수록 좋음** (1% 미만) | 정밀 타격 |
| **분포도 (Distribution)** | 데이터가 전체 테이블에 흩어진 정도. | **고르게 퍼져 있을수록** 유리 | 모래 속 진주 |

### 2. 컬럼 성격에 따른 인덱스 적합성 도해 (ASCII)

어떤 컬럼이 인덱스로서 가치가 높은지 시각화한다.

```text
    [ High Value ] <─────────────────────────────────────> [ Low Value ]
    
    (1) 사원번호 / 주문ID        (2) 핸드폰번호 / 이메일        (3) 성별 / 결제여부
    ┌──────────────────┐        ┌──────────────────┐        ┌──────────────────┐
    │  Cardinality: Max│        │  Cardinality: High│        │  Cardinality: Min│
    │  Selectivity: Low│        │  Selectivity: Low │        │  Selectivity: High│
    └──────────────────┘        └──────────────────┘        └──────────────────┘
    * 선정 1순위 (PK/UK)         * 선정 2순위 (일반 Index)     * 선정 제외 (Full Scan 유리)
```

### 3. 심층 동작 원리: 결합 인덱스 (Composite Index)의 우선순위
- **원칙**: 여러 컬럼을 묶어 인덱스를 만들 때는 **선택도가 낮은(변별력이 높은) 컬럼을 앞순서**에 배치해야 함.
- **이유**: 루트 노드에서부터 범위를 크게 걸러낼수록 리프 노드까지의 탐색 깊이와 I/O 횟수가 줄어들기 때문.

### 4. 핵심 기술: 활용 빈도 분석
- 카디널리티가 조금 낮더라도, 해당 컬럼이 서비스의 핵심 기능(예: '진행중'인 주문 조회)에서 매 초마다 수천 번씩 조회된다면 인덱스를 생성하는 것이 유리함.

- **📢 섹션 요약 비유**: 결합 인덱스의 순서는 '주소록 정리'와 같아서, [도시 > 동 > 번지] 순으로 정리해야 가장 빨리 찾을 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 인덱스 생성 vs 미생성 시의 기회비용

| 비교 항목 | 인덱스 생성 시 | 인덱스 미생성 시 |
|:---:|:---|:---|
| **조회 성능 (SELECT)** | **비약적 향상** | 데이터 양에 따라 급격히 저하 |
| **입력 성능 (INSERT)** | 저하 (인덱스 분할 발생) | **최상 (Append 전용)** |
| **수정 성능 (UPDATE)** | 저하 (인덱스 재배치) | 상대적으로 우수 |
| **디스크 관리** | 인덱스 파일 관리 부하 | 테이블 파일만 관리 |

### 2. 과목 융합 관점: 통계학 및 검색 알고리즘
- **통계학 융합**: 인덱스 선정은 데이터의 **표준 편차와 분산**을 분석하는 것과 같으며, 데이터가 편중되어 있다면(Skewed Data) 단순 선택도 수치보다 **히스토그램**을 통한 미세 조정이 더 중요하다.
- **알고리즘 융합**: 인덱스 탐색은 $O(\log N)$의 **이진 탐색(Binary Search)**을 기반으로 하며, 인덱스가 없는 조회는 $O(N)$의 **선형 탐색(Linear Search)**이 되므로 데이터 규모가 커질수록 격차는 지수적으로 벌어진다.

- **📢 섹션 요약 비유**: 인덱스는 '공부 효율'과 같아서, 무작정 다 외우기(전체 인덱스)보다 핵심 키워드만 파악하는 것이 성적(성능)이 더 잘 나옵니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 쇼핑몰 주문 테이블 인덱스 전략
- **상황**: 주문 테이블에 `주문번호`, `고객ID`, `주문일자`, `상태(결제완료/취소)` 컬럼이 있음.
- **판단**: `주문번호`는 PK이므로 당연히 생성. `고객ID`는 한 고객이 수십 번 주문하므로 카디널리티가 높음. `상태`는 단 2~3종류뿐이라 변별력 없음.
- **결단**: **고객ID와 주문일자를 묶은 결합 인덱스 생성**. "누가 언제 샀나"라는 빈번한 질문에 최적화. `상태` 컬럼은 인덱스에서 제외하되, 필요한 경우 비트맵 인덱스(Bitmap)나 파티셔닝을 고려하여 설계.

### 2. 안티패턴: 모든 WHERE 조건 컬럼에 개별 인덱스 생성
- **증상**: 쿼리에 나오는 모든 컬럼마다 하나씩 단일 인덱스를 만듦.
- **결과**: 인덱스 머지(Index Merge) 오버헤드가 발생하고, 테이블 데이터보다 인덱스 용량이 더 커지는 '배보다 배꼽이 큰' 상황 초례.
- **조치**: **자주 함께 쓰이는 컬럼들을 분석하여 '최소한의 결합 인덱스'**로 통합 설계할 것.

### 📢 기술사적 결언
> "인덱스 선정은 **'데이터의 가치'**를 선별하는 철학적 과정이다. 어떤 데이터가 사용자의 핵심 질문에 답을 줄 수 있는지 결정하는 일이기 때문이다. 기술사는 도구(DBMS)가 알아서 해주길 기다리기보다, **비즈니스 도메인의 쿼리 패턴**을 완벽히 이해하고 **선택도라는 수학적 근거**를 바탕으로 아키텍처를 세워야 한다. 인덱스를 하나 만드는 것은 '성능의 빚'을 지는 행위임을 인지하고, 그 빚을 상쇄할 만큼의 '조회 이익'이 있는지 냉정하게 계산해야 한다."

- **📢 섹션 요약 비유**: 인덱스를 너무 많이 만드는 것은 '모든 문장에 형광펜을 칠하는 공부법'과 같아, 결국 중요한 게 무엇인지 알 수 없게 만듭니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (최적 인덱스 선정 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **평균 쿼리 응답 시간** | 수 초 | **밀리초 (ms)** | 체감 성능 100배 향상 |
| **디스크 I/O 발생량** | 10,000 Blocks | **5 Blocks 이내** | 리소스 절감 및 수명 연장 |
| **데이터 정합성 유지** | - | **유니크 인덱스로 보장** | 비즈니스 무결성 강화 |

### 2. 미래 전망
- **Automatic Indexing (Oracle ADW 등)**: 머신러닝이 실시간 쿼리를 분석하여, 사람이 개입하지 않아도 최적의 인덱스를 생성하고 안 쓰는 인덱스를 삭제하는 **자율 운영 인덱스** 보편화.
- **Learned Indexes**: 데이터 분포를 학습한 인공 신경망 모델 자체가 인덱스 역할을 대신하여, B-Tree보다 훨씬 작은 메모리로 더 빠른 검색을 지원하는 차세대 기술 연구 중.

- **📢 섹션 요약 비유**: 미래의 인덱스는 고정된 책의 목차가 아니라, 우리가 찾는 물건을 미리 알고 우리 손 앞에 가져다주는 '자율 주행 로봇 팔'이 될 것입니다.
