# 182. 정적 SQL vs 동적 SQL - 성능과 유연성의 상충 관계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소스 코드 내에 SQL 문이 고정되어 컴파일 시점에 구조가 확정되는 **정적 SQL**과, 실행 시점에 사용자 입력이나 조건에 따라 문자열 결합 등으로 문장이 동적으로 생성되는 **동적 SQL**의 대결이다.
> 2. **상충 관계**: 정적 SQL은 강력한 타입 체크와 실행 계획의 안정성을 제공하여 고성능 처리에 유리하지만, 동적 SQL은 복잡한 검색 조건에 유연하게 대응할 수 있는 대신 하드 파싱 부하와 보안 취약성(SQL Injection)의 위험을 안고 있다.
> 3. **가치**: 대다수 트랜잭션은 정적 SQL(바인드 변수 활용)로 표준화하여 시스템 안정성을 꾀하고, 다차원 검색과 같은 가변적 요건에만 동적 SQL을 제한적으로 활용하는 전략적 분배가 중요하다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **정적 SQL (Static SQL)**: 프로그램 개발 시점에 이미 SQL 문법이 완성되어 있는 형태입니다. 변하는 값은 오직 바인드 변수(`:var`)로만 처리됩니다.
- **동적 SQL (Dynamic SQL)**: 프로그램 실행 중에 문자열을 조립하여(`str = "SELECT " + col + " FROM ..."`) 실행하는 형태입니다.

### 💡 비유: 기성복 vs 맞춤 정장
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        정적 vs 동적 SQL 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 정적 SQL (기성복):                                                        │
│     - 사이즈(구조)가 이미 정해져 있고, 입는 사람(값)만 바뀜.                      │
│     - 옷을 고르고 입는 속도가 매우 빠름. (안정적 성능)                            │
│     - 내 몸이 아주 특이하면(복잡한 조건) 옷이 안 맞을 수 있음.                    │
│                                                                             │
│  2. 동적 SQL (맞춤 정장):                                                      │
│     - 손님이 오면 그때부터 치수를 재고 가위질을 해서 옷을 만듦.                   │
│     - 어떤 체형(조건)이 와도 완벽하게 딱 맞는 옷을 제작 가능. (유연성)             │
│     - 하지만 옷 한 벌 만드는 데 시간이 너무 오래 걸림! (하드 파싱 부하)            │
│                                                                             │
│  → **'미리 준비된 안정성'**과 **'그때그때 맞추는 유연성'** 사이의 선택입니다!       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **개발 생산성의 요구**: 조건이 수십 개인 '상세 검색' 화면을 수천 개의 정적 SQL로 다 만들 수 없으므로, 동적으로 문장을 생성할 도구 필요.
- **DB 서버의 비명**: 무분별한 동적 SQL 사용으로 인해 매번 실행 계획을 새로 짜느라(Hard Parsing) CPU가 타버리는 성능 위기 도래.

- **📢 섹션 요약 비유**: 정적 SQL은 '정해진 코스 요리'이고, 동적 SQL은 '내 맘대로 골라 담는 뷔페'와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 정적 vs 동적 SQL 상세 특성 비교

| 비교 항목 | 정적 SQL (Static) | 동적 SQL (Dynamic) |
|:---:|:---|:---|
| **문장 확정 시점** | 컴파일 단계 | **런타임(실행) 단계** |
| **구문 분석** | 사전 검증 가능 | 실행 시점에야 오류 확인 가능 |
| **파싱 형태** | **소프트 파싱** (바인드 변수 권장) | **하드 파싱** 발생 가능성 높음 |
| **보안성** | 우수 (구조 불변) | **취약** (SQL Injection 위험) |
| **유연성** | 낮음 (정해진 쿼리만 수행) | **매우 높음** (테이블명도 가변 가능) |

### 2. 동적 SQL의 하드 파싱 유발 구조 도해 (ASCII)

리터럴 값을 결합할 때 발생하는 캐시 오염 과정을 시각화한다.

```text
    [ Dynamic SQL with Literals ]
    1. "SELECT * FROM User WHERE ID = " + "101" ──▶ Hash: 'A1B2' (New Plan)
    2. "SELECT * FROM User WHERE ID = " + "102" ──▶ Hash: 'C3D4' (New Plan)
    3. "SELECT * FROM User WHERE ID = " + "103" ──▶ Hash: 'E5F6' (New Plan)
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Library Cache : Overflow! ]                                  │
    │  - 각 SQL 마다 다른 자리를 차지하여 소중한 캐시 메모리 고갈.        │
    │  - CPU는 계속 계획만 짜느라(Hard Parsing) 서비스 응답 불가.        │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: 바인드 변수(Bind Variable)의 중재
- **해결책**: 동적 SQL이라도 문자열 결합 대신 `WHERE ID = ?` 처럼 바인드 변수를 사용하면, 문장 구조가 같아져서 **소프트 파싱** 혜택을 누릴 수 있음.
- **원리**: SQL 텍스트의 해시 값이 동일하게 유지되도록 강제하는 기술.

### 4. 핵심 기술: SQL Injection 방어
- 동적 SQL 작성 시 `id = ' + input + '` 처럼 문자열을 직접 붙이면, 악의적인 사용자가 `1' OR '1'='1` 등을 입력하여 보안을 뚫을 수 있음. 바인드 변수는 이를 원천 차단함.

- **📢 섹션 요약 비유**: 바인드 변수는 '빈칸 채우기 문제지'와 같아서, 답(값)이 바뀌어도 문제(구조)는 그대로라 선생님(DB)이 채점하기 편합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 사용 패턴에 따른 성능 체감 지수

| 시나리오 | 최적의 SQL 형태 | 이유 |
|:---:|:---: |:---|
| **단건 조회 / 반복 트랜잭션** | **정적 SQL** | 최소한의 파싱 비용, 극강의 속도. |
| **복합 조건 검색 (Search)** | **동적 SQL + Bind** | 유연한 WHERE 절 생성, 캐시 공유. |
| **관리자용 대량 집계** | 동적 SQL + Literal | 변수 피킹 부작용 방지, 정확한 통계 활용. |

### 2. 과목 융합 관점: 소프트웨어 공학 및 보안
- **소프트웨어 공학(SE) 융합**: 정적 SQL은 **MyBatis**나 **JPA**의 정적 메서드에 해당하며, 동적 SQL은 **Criteria API**나 **QueryDSL**의 동적 쿼리 생성 로직과 맥을 같이 한다.
- **보안(Security) 융합**: 동적 SQL의 오남용은 **OWASP Top 10**의 부동의 1위인 Injection 공격의 통로가 되며, 'PrepareStatement'를 통한 파라미터화된 쿼리 사용은 현대 보안 아키텍처의 절대 규범이다.

- **📢 섹션 요약 비유**: 정적 SQL은 '튼튼한 정문'이고, 동적 SQL은 '필요할 때만 여는 비밀 통로'입니다. 관리가 안 되면 비밀 통로로 도둑(해커)이 듭니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 다중 조건 검색 필터 구현
- **상황**: [이름, 나이, 지역, 가입일...] 20개의 필터 중 사용자가 선택한 것만 검색해야 함.
- **판단**: 모든 조합(2의 20승 개)의 정적 SQL을 만드는 것은 불가능.
- **결단**: **바인드 변수를 지원하는 동적 SQL 프레임워크(QueryDSL 등) 사용**. 문장은 동적으로 생성하되, 값은 반드시 바인드 변수로 전달하여 라이브러리 캐시 효율을 높이고 하드 파싱 부하를 방어함.

### 2. 안티패턴: 루프(Loop) 내부에서의 동적 SQL 생성
- **증상**: `for` 문을 돌면서 매번 `EXECUTE IMMEDIATE "SELECT ..."`를 수행.
- **결과**: 루프 횟수만큼 하드 파싱이 발생하여 DB 서버의 CPU 사용률이 100%에 도달, 전체 시스템 마비.
- **조치**: **SQL은 루프 밖에서 한 번만 확정**하거나, 반드시 바인드 변수를 사용하여 소프트 파싱을 유도할 것.

### 📢 기술사적 결언
> "정적 SQL과 동적 SQL의 구분은 **'성능의 보수주의'와 '기능의 진보주의'**의 대결이다. 시스템의 안정성을 위해서는 정적 SQL이라는 튼튼한 뿌리가 필요하지만, 풍부한 사용자 경험을 위해서는 동적 SQL이라는 유연한 가지가 필요하기 때문이다. 기술사는 도구의 편리함에 매몰되지 말고, **Library Cache 의 생존력**을 계산하며 동적 SQL의 범위를 통제해야 한다. '유연함을 허용하되, 물리적 비용은 바인딩으로 상쇄한다'는 하이브리드 전략이 고도화된 DB 설계의 정석임을 잊지 말아야 한다."

- **📢 섹션 요약 비유**: 동적 SQL을 쓰는 것은 '즉석요리'와 같아, 손님이 몰릴 때는 미리 만들어 둔 요리(정적 SQL)를 내놓는 요령이 필요합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (SQL 전략 최적화 시)

| 지표 | 활용 미흡 시 | 최적 활용 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **CPU 파싱 점유율** | 70% (Hard) | **5% 미만 (Soft)** | 서버 처리량 10배 이상 향상 |
| **보안 사고 발생률** | 상존 (Literal) | **0% (Prepared)** | 기업 자산 및 신뢰도 보호 |
| **개발 유연성** | 낮음 (하드코딩) | **매우 높음 (동적 생성)** | 비즈니스 요구사항 즉시 대응 |

### 2. 미래 전망
- **AI-Driven Static Caching**: 동적 쿼리의 패턴을 AI가 학습하여, 자주 발생하는 형태를 미리 정적 실행 계획으로 구체화해두는 **지능형 하이브리드 파싱**.
- **Template-based SQL Engines**: 문자열 결합이 아닌 논리 템플릿 기반으로 쿼리를 생성하여, 컴파일 타임에 모든 동적 경로를 검증하는 고안전성 엔진의 확산.

- **📢 섹션 요약 비유**: 미래의 SQL은 우리가 문장을 조립하지 않아도, 우리의 의도(Context)에 맞춰 가장 완벽하고 안전한 형태의 정적 코드로 스스로를 재구성하는 지능형 언어가 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **정적 SQL**은 이미 다 만들어진 **"완성품 레고 성"**과 같아요. 바로 가지고 놀 수 있어서 아주 빠르죠.
2. **동적 SQL**은 내가 놀고 싶은 대로 그때그때 **"블록을 조립해서 성을 만드는 것"**과 같아요. 내 맘대로 만들 수 있지만 시간이 좀 걸려요.
3. 이 두 방법을 잘 섞어서, 자주 노는 성은 미리 만들어두고 **특별한 성만 새로 조립하는 것**이 가장 똑똑한 방법이랍니다!
