# 157. 문자열 및 수치 함수 - 데이터 가공과 수치 연산

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스에 저장된 원천 데이터를 사용자 요구에 맞는 형태로 변형하거나 복잡한 계산을 수행하기 위해 제공되는 **단일 행 함수(Scalar Function)**의 대표적인 두 축이다.
> 2. **기능적 유연성**: **문자열 함수**는 텍스트 결합, 추출, 치환을 통해 비정형 속성을 정제하며, **수치 함수**는 반올림, 절대값, 나머지 연산 등을 통해 정밀한 산술 결과를 도출한다.
> 3. **가치**: 애플리케이션 레벨의 로직 부하를 데이터베이스 엔진으로 이관하여 네트워크 전송량을 줄이고, 데이터의 표현(Presentation) 형식을 표준화하여 시스템의 응답성과 정합성을 제고한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **문자열 함수 (String Functions)**: 문자 데이터를 조작하여 대소문자 변경, 길이 측정, 특정 문자 추출 등을 수행하는 함수입니다.
- **수치 함수 (Numeric Functions)**: 숫자 데이터를 입력받아 수학적 처리를 수행하고 결과를 반환하는 함수입니다.

### 2. 💡 비유: 가공 공장의 '조각칼'과 '저울'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        문자열 및 수치 함수 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 원목(Raw Data)을 가져와서 예쁜 가구(Information)를 만들려 함.            │
│                                                                             │
│  1. 문자열 함수 (조각칼):                                                     │
│     - 너무 긴 나무는 자르고(SUBSTR), 매끄럽게 다듬고(TRIM), 이름을 새김(CONCAT).    │
│     - 겉모양을 보기 좋게 만드는 '데이터의 미용사'입니다.                          │
│                                                                             │
│  2. 수치 함수 (정밀 저울/계산기):                                              │
│     - 무게를 정확히 재고, 소수점은 반올림(ROUND)하거나 버림(TRUNC).               │
│     - 숫자를 정확한 비즈니스 규칙에 맞게 가공하는 '데이터의 회계사'입니다.           │
│                                                                             │
│  → 복잡한 기계를 돌리지 않아도, **'도구 하나로 즉석에서 데이터를 가공'**             │
│     해주는 편리한 작업 도구들입니다!                                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 정제(Cleaning)의 필연성**: 입력 단계에서 제각각인 텍스트('kim', 'KIM')를 하나로 통일하거나, 불필요한 공백을 제거해야 하는 요건 발생.
- **비즈니스 산식의 DB 이관**: 할인율 계산, 부가세 처리 등 공통적인 수학 공식들을 모든 쿼리에서 일관되게 적용하기 위함.

- **📢 섹션 요약 비유**: 단일 행 함수는 '개별 데이터에 붙여주는 작은 스티커나 액세서리'와 같아서, 데이터 하나하나를 빛나게 해줍니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 문자열 함수 카테고리 및 예시

| 함수명 | 상세 역할 | 예시 |
|:---:|:---|:---|
| **UPPER / LOWER** | 대/소문자로 강제 변환. | 'kim' $\to$ 'KIM' |
| **SUBSTR (SUBSTRING)** | 특정 위치에서 지정한 길이만큼 추출. | '20260316' $\to$ '2026' |
| **INSTR** | 특정 문자가 나타나는 위치 찾기. | 'admin@gmail.com' $\to$ '@' 위치(6) |
| **LPAD / RPAD** | 남는 공간을 특정 문자로 채우기. | '123' $\to$ '00123' (LPAD) |
| **REPLACE** | 특정 문자를 다른 문자로 치환. | '010-1234' $\to$ '0101234' |

### 2. 주요 수치 함수 카테고리 및 예시

| 함수명 | 상세 역할 | 예시 |
|:---:|:---|:---|
| **ABS** | 절대값 반환. | -10 $\to$ 10 |
| **ROUND** | 특정 자리수에서 반올림. | 123.456 $\to$ 123.5 |
| **TRUNC** | 특정 자리수에서 절삭(버림). | 123.456 $\to$ 123.4 |
| **MOD** | 나머지 연산. | 10 $\div$ 3 의 나머지 $\to$ 1 |
| **CEIL / FLOOR** | 올림 / 내림 (정수형). | 10.1 $\to$ 11 / 10 |

### 3. 함수 실행 아키텍처 (ASCII)

데이터가 엔진을 통과하며 가공되는 흐름을 시각화한다.

```text
    [ Disk: Raw Block ]
    Data: { Name: ' kim ', Price: 1234.56 }
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ SQL Engine : Scalar Function Processor ]                     │
    │  1. TRIM(Name)    : ' kim ' ──▶ 'kim'                           │
    │  2. UPPER(Name)   : 'kim'   ──▶ 'KIM'                           │
    │  3. ROUND(Price)  : 1234.56 ──▶ 1235                            │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    [ Output: Formatted Row ]
    Result: { Name: 'KIM', Price: 1235 }
```

### 4. 심층 동작 원리: 함수 기반 인덱스 (FBI)
- **문제**: `WHERE UPPER(Name) = 'KIM'` 처럼 조건절에 함수를 쓰면 일반 인덱스를 탈 수 없음.
- **해결**: 함수를 적용한 결과 자체를 인덱스로 미리 만들어두는 **Function-Based Index** 기술을 통해 검색 성능 보장.

- **📢 섹션 요약 비유**: 함수 기반 인덱스는 '별명을 미리 사전(Index)에 등록해두어, 별명으로 불러도 즉시 주인을 찾아주는 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. SQL 가공 vs Application(Java/Python) 가공

| 비교 항목 | SQL 레벨 가공 (함수) | App 레벨 가공 |
|:---:|:---|:---|
| **데이터 전송량** | **최소화 (가공 후 전송)** | 최대화 (전체 데이터 전송) |
| **CPU 부하** | DB 서버 부하 증가 | App 서버 부하 분산 |
| **일관성** | **전 시스템 동일 규칙 적용** | 언어마다 로직 다를 위험 |
| **유연성** | 쿼리 수정으로 즉시 반영 | 코드 배포 필요 |

### 2. 과목 융합 관점: 통계 및 보안
- **통계 융합**: 수치 함수 중 `SQRT`(제곱근), `POWER`(거듭제곱) 등은 데이터의 **표준 편차**를 구하거나 **정규화(Normalization)**를 수행하는 수학적 기초가 된다.
- **보안 융합**: 문자열 함수인 `SUBSTR`과 `RPAD`를 조합하여 '900101-1******' 처럼 주민번호 뒷자리를 가리는 **마스킹(Masking)** 기술이 개인정보 보호의 필수 수단으로 활용된다.

- **📢 섹션 요약 비유**: SQL 가공은 '공장에서 미리 조립해서 배송하는 것'이고, App 가공은 '재료를 다 받아서 집에서 조립하는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 카드 번호 형식 통일 및 보안 마스킹
- **상황**: 고객이 입력한 카드 번호가 어떤 곳은 `-`가 있고 어떤 곳은 없음. 화면에는 뒷자리 4자리만 보여줘야 함.
- **판단**: 저장된 형태가 달라도 출력은 일관되어야 함.
- **결단**: **REPLACE와 SUBSTR 조합 활용**. `REPLACE(card_no, '-')`로 형식을 맞춘 뒤, `RPAD(SUBSTR(card_no, 1, 12), 16, '*')`을 적용하여 데이터 정제와 보안을 한 번에 해결.

### 2. 안티패턴: 조건절(WHERE) 왼쪽 항의 함수 남발
- **증상**: `WHERE SUBSTR(Order_Date, 1, 6) = '202603'`.
- **결과**: `Order_Date` 컬럼에 인덱스가 있어도 사용하지 못하고 Full Scan 발생.
- **조치**: **함수를 오른쪽 항으로 이동하거나 범위 조건으로 변경**. `WHERE Order_Date >= '20260301' AND Order_Date <= '20260331'`. 인덱스 효율을 극대화하는 'Sargable' 쿼리 작성 습관화.

### 📢 기술사적 결언
> "문자열과 수치 함수는 데이터베이스의 **'표현력'**을 좌우한다. 데이터는 가공될 때 비로소 정보(Information)로서의 생명력을 얻기 때문이다. 기술사는 단순 함수 사용을 넘어, 함수 사용이 **인덱스 스캔과 실행 계획**에 미치는 물리적 파급력을 계산해야 한다. '가장 우아한 쿼리'는 화려한 함수를 많이 쓴 쿼리가 아니라, **최소한의 가공으로 최대한의 성능**을 뽑아내는 '절제의 미학'이 담긴 쿼리임을 명심해야 한다."

- **📢 섹션 요약 비유**: 잘 드는 칼(함수)을 가졌다고 모든 재료를 다 썰어버리면(과도한 가공), 요리의 본맛(성능)을 잃게 됩니다.
