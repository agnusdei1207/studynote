# 146. 계층형 질의 (Hierarchical Query) - 트리 구조 탐색과 CONNECT BY

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동일한 테이블 내에 존재하는 부모-자식 간의 수직적 관계를 논리적으로 연결하여, 데이터의 전체적인 **계층적 위계(Hierarchy)**를 트리 형태로 탐색하고 출력하는 특수한 질의 방식이다.
> 2. **메커니즘**: 시작 지점을 정하는 **`START WITH`**와 부모-자식 연결 규칙을 정의하는 **`CONNECT BY`** 절을 사용하며, `LEVEL` 가상 컬럼을 통해 데이터의 깊이를 파악하고 시각화할 수 있다.
> 3. **가치**: 조직도, 부품 구성표(BOM), 게시판 댓글 구조 등 현실의 복잡한 계층 데이터를 조인(Join)의 한계를 넘어 무한한 깊이까지 효율적으로 처리하는 **순환적 사고의 정수**다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**계층형 질의 (Hierarchical Query)**는 하나의 테이블에 부모 행과 자식 행이 섞여 있을 때, 이를 논리적으로 엮어 '뿌리(Root)'부터 '잎(Leaf)'까지 순차적으로 찾아가는 질의입니다. 주로 오라클의 `CONNECT BY` 문법이나 표준 SQL의 `WITH RECURSIVE` 문법을 사용합니다.

### 💡 비유: 가계도 족보 읽기
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        계층형 질의 개념 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 시골 종갓집에서 거대한 족보(테이블)를 펼쳤음.                             │
│                                                                             │
│  1. 시작점 (START WITH): "시조 할아버지(루트)"부터 찾기 시작함.                  │
│  2. 연결 규칙 (CONNECT BY): "할아버지의 자식은 아버지, 아버지의 자식은 나."        │
│  3. 깊이 (LEVEL): 할아버지는 1단계, 아버지는 2단계, 나는 3단계.                  │
│                                                                             │
│  → 계층형 질의는 수천 명의 자손 중 **'누가 누구의 후손인지'** 줄기를 따라          │
│     한눈에 보여주는 똑똑한 돋보기와 같습니다!                                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **현실 세계의 수직 관계**: 회사 조직(사장-이사-팀장), 컴퓨터 폴더 구조 등 계층적 데이터가 도처에 존재함.
- **조인의 한계 극복**: 5단계 깊이의 조직을 일반 조인으로 짜려면 테이블을 5번 불러와야 하지만, 계층형 질의는 **단 한 번의 선언**으로 해결 가능.

- **📢 섹션 요약 비유**: 계층형 질의는 '실타래의 끝'을 잡고 쭉 당겨서 전체 엉킴(관계)을 풀어내는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 계층형 질의의 핵심 구문 (Oracle 기준)

```sql
SELECT LEVEL, LPAD(' ', (LEVEL-1)*2) || Name as Tree_Name
FROM   Employee
START WITH Mgr_ID IS NULL           -- (1) 시작 지점 (최상위 사장님)
CONNECT BY PRIOR EID = Mgr_ID;      -- (2) 연결 방식 (부모PK = 자식FK)
```

| 구문 요소 | 상세 역할 | 비유 |
|:---:|:---|:---|
| **START WITH** | 계층 구조의 시작이 되는 행을 지정. | 뿌리 찾기 |
| **CONNECT BY** | 자식 행을 어떻게 찾을지 규칙 설정. | 가지 뻗기 |
| **PRIOR** | '이전 행(부모)'의 값을 가리키는 키워드. | "우리 아빠가..." |
| **LEVEL** | 현재 행의 계층적 깊이를 나타내는 가상 컬럼. | 촌수 계산 |

### 2. 계층 탐색 방향 도해 (ASCII)

탐색 방향에 따른 데이터 흐름의 차이를 시각화한다.

```text
    [ Top-Down : 정방향 탐색 ]          [ Bottom-Up : 역방향 탐색 ]
    (사장)                              (나)
      │                                  │
    ┌─┴─┐  (CONNECT BY PRIOR PK=FK)    ┌─┴─┐  (CONNECT BY PK=PRIOR FK)
    (부장)                              (과장)
      │                                  │
    (대리)                              (사장)
    * 용도: 조직 전체 조망               * 용도: 상위 결재권자 추적
```

### 3. 심층 동작 원리: 재귀적 방문 (Recursive Visit)
- **메커니즘**: 첫 번째 행을 결과셋에 담고, 그 행의 PK를 PRIOR 값으로 삼아 다음 자식을 검색함. 더 이상 자식이 없을 때까지 깊이 우선 탐색(DFS) 방식으로 진행.
- **사이클 방지**: `NOCYCLE` 옵션을 통해 A가 B의 부모인데 B가 다시 A의 부모로 등록된 '순환 오류' 발생 시 무한 루프 차단.

### 4. 핵심 기술: ORDER SIBLINGS BY
- 계층 구조(줄기)를 깨뜨리지 않으면서, 동일 레벨의 형제(Siblings)들끼리만 정렬을 수행하는 특수 정렬 기술.

- **📢 섹션 요약 비유**: PRIOR 키워드는 '부모님의 손'을 잡는 것과 같아서, 손을 놓지 않고 끝까지 계층을 따라가게 해줍니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. CONNECT BY vs Recursive CTE (표준 SQL)

| 비교 항목 | Oracle CONNECT BY | Recursive CTE (WITH) |
|:---:|:---|:---|
| **표준성** | 오라클 전용 문법 | **ANSI/ISO 표준 문법** |
| **가독성** | 매우 간결하고 직관적 | 조금 장황하고 복잡함 |
| **유연성** | 특정 방향 탐색 최적화 | **복잡한 수식 및 연산 결합 유리** |
| **지원 DBMS** | Oracle, Tibero 등 | **PostgreSQL, SQL Server, MySQL 8.0+** |

### 2. 과목 융합 관점: 자료구조 및 알고리즘
- **자료구조 융합**: 계층형 질의는 메모리 상의 **트리(Tree) 순회**를 SQL로 구현한 것이며, 결과 집합은 트리의 **전위 순회(Pre-order Traversal)** 결과와 유사한 순서를 갖는다.
- **알고리즘 융합**: 대량의 계층 데이터 조회 시 성능을 높이기 위해 **경로 열거(Path Enumeration)** 모델이나 **중첩 집합(Nested Set)** 모델과 같은 별도의 모델링 기법과 성능 대결을 펼친다.

- **📢 섹션 요약 비유**: CONNECT BY가 '자동 응답기'라면, CTE는 '직접 회로를 설계하는 무전기'와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 제조 공정의 BOM (Bill of Materials) 전개
- **상황**: 자동차 한 대를 만들기 위한 수만 개의 부품 계층 구조를 뽑아야 함.
- **판단**: 부품 밑에 하위 부품이 몇 단계나 있을지 알 수 없음.
- **결단**: **계층형 질의 적용**. `SYS_CONNECT_BY_PATH` 함수를 사용하여 [자동차 > 엔진 > 실린더] 처럼 전체 조립 경로를 텍스트로 생성하고, 각 부품의 소요량을 계층별로 합산하여 정확한 자재 소요량 산출.

### 2. 안티패턴: 순환 참조(Cycle) 체크 누락
- **증상**: 데이터 입력 실수로 사원이 자기 자신을 관리자로 등록하거나, 순환 고리 형성.
- **결과**: 쿼리 수행 시 CPU가 100%를 치며 서버가 멈춤 (Infinite Loop).
- **조치**: **`NOCYCLE` 키워드 사용**을 습관화하고, `CONNECT_BY_ISCYCLE` 가상 컬럼을 조회하여 잘못된 데이터를 찾아내어 정제할 것.

### 📢 기술사적 결언
> "계층형 질의는 데이터베이스에서 **'시간과 공간의 인과관계'**를 풀어내는 열쇠다. 평면적인 테이블에 입체적인 생명력을 불어넣기 때문이다. 기술사는 문법의 편리함에 앞서, 계층형 쿼리가 유발하는 **랜덤 I/O 부하**를 인지하고 **인덱스 설계**를 통해 이를 완화해야 한다. 특히 무한 재귀의 위험성을 통제하는 **거버넌스적 시각**이 시스템의 안정성을 결정짓는 핵심 역량이 됨을 명심해야 한다."

- **📢 섹션 요약 비유**: 계층형 쿼리 최적화는 '미로 찾기'와 같아서, 지도가 좋아도(인덱스) 길을 잘못 들면(무한 루프) 영영 빠져나오지 못할 수 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (계층 질의 도입 시)

| 지표 | 도입 전 (Join 방식) | 도입 후 (Hierarchical) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **코드 가독성** | 매우 복잡 (Self-join 중첩) | **간결한 선언형** | 유지보수 생산성 3배 향상 |
| **탐색 깊이** | 고정됨 (Join 횟수만큼) | **무제한 (Dynamic)** | 비즈니스 확장성 확보 |
| **시스템 부하** | 반복적인 테이블 스캔 | **최적화된 트리 순회** | 리소스 효율성 증대 |

### 2. 미래 전망
- **Graph Database의 융합**: 관계형의 계층형 질의 한계를 넘어, 복잡한 네트워크 관계를 네이티브하게 처리하는 **Cypher(Neo4j)** 문법의 사상이 SQL 표준에 스며드는 추세.
- **AI-Logic Discovery**: 계층 관계가 명시되지 않은 데이터들 사이에서 AI가 계층 구조를 스스로 추론하여 가상의 부모-자식 관계를 제안하는 지능형 모델링 기술 발전.

- **📢 섹션 요약 비유**: 미래의 계층형 질의는 단순한 나무(Tree)를 넘어, 전 세계 데이터가 거대한 뿌리로 연결된 '디지털 숲'을 탐험하는 지도가 될 것입니다.
