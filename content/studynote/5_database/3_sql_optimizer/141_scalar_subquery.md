# 141. 스칼라 서브쿼리 (Scalar Subquery) - SELECT 절의 단일 값 연산

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SELECT 절에서 사용되어 **단 하나의 행과 하나의 열(Single Value)**만을 반환하는 서브쿼리로, 메인쿼리의 각 행마다 하나의 컬럼 값처럼 취급되어 출력된다.
> 2. **성능 요체**: 매 행마다 실행되는 오버헤드를 줄이기 위해 DBMS는 **서브쿼리 캐싱(Caching)** 기술을 활용하지만, 입력값의 종류가 많을 경우 성능이 급격히 저하되므로 조인(Join)으로의 변환 가능성을 항상 검토해야 한다.
> 3. **가치**: 코드 조견표(Lookup Table)나 합계 등 간단한 연관 정보를 조인 없이 간결하게 조회할 때 유용하며, 쿼리의 복잡도를 낮추고 가독성을 높여주는 효과가 있다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**스칼라 서브쿼리 (Scalar Subquery)**는 함수의 결과값처럼 단일 값을 리턴하는 서브쿼리입니다. 만약 결과가 2건 이상 나오면 런타임 에러가 발생하고, 결과가 없으면 NULL을 반환합니다.

### 2. 💡 비유: 학생 명찰에 '담임 선생님 성함' 적기
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        스칼라 서브쿼리 비유                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 학생들의 명단(메인쿼리)을 한 줄씩 인쇄하고 있음.                          │
│                                                                             │
│  - "이름: 김철수, 반: 1반, [담임 선생님 성함]"                                  │
│  - 이때 [담임 선생님 성함] 칸을 채우기 위해,                                    │
│  - 매번 선생님 명단(서브쿼리)으로 달려가서 "1반 선생님 누구세요?"라고 물어봄.      │
│  - 선생님이 대답해주면 그 이름을 명찰의 **'딱 한 칸'**에 적어 넣음!               │
│                                                                             │
│  → 행마다 **'빈칸 하나를 채우기 위해'** 옆방에 가서 정답을 구해오는 방식입니다.     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **조인의 복잡도 완화**: 단순히 코드명 하나를 가져오기 위해 거대한 테이블 전체를 조인하는 것이 쿼리 가독성을 해칠 때 사용.
- **유연한 데이터 가공**: 특정 행의 값과 전체 평균을 한 줄에 비교해서 보여주고 싶을 때 유용.

- **📢 섹션 요약 비유**: 스칼라 서브쿼리는 쿼리라는 문장 속의 '대명사'와 같아서, 그 자리에 들어갈 구체적인 단어를 실시간으로 검색해 옵니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 스칼라 서브쿼리의 실행 특징

| 특징 | 상세 내용 | 주의 사항 |
|:---:|:---|:---|
| **단일값 보장** | 1 Row, 1 Column 만 반환해야 함. | 결과가 2개면 SQL 에러 발생. |
| **상관 관계** | 주로 메인쿼리의 컬럼을 인자로 받음 (상관 쿼리). | 인덱스 부재 시 성능 재앙. |
| **NULL 처리** | 매칭되는 값이 없으면 에러가 아닌 NULL 리턴. | Outer Join 과 유사한 효과. |
| **위치** | 주로 SELECT 절, 간혹 WHERE/ORDER BY 절. | 연산량이 많은 곳에선 지양. |

### 2. 스칼라 서브쿼리 작동 아키텍처 (ASCII)

메인쿼리의 데이터가 흐를 때 서브쿼리가 어떻게 개입하는지 시각화한다.

```text
    [ Main Query Scan ]
    Row 1: {ID: 101, Dept: 'A'} ──▶ [ Subquery Execution ] ──▶ Result: 'Sales'
    Row 2: {ID: 102, Dept: 'B'} ──▶ [ Subquery Execution ] ──▶ Result: 'IT'
    Row 3: {ID: 103, Dept: 'A'} ──▶ [ Cache Hit! ] ──────────▶ Result: 'Sales'
                                    (동일한 'A' 입력 시 실행 생략)
    
    [ Output Result Set ]
    ┌─────┬──────┬────────────┐
    │ ID  │ Dept │ Dept_Name  │  <--- (Scalar Subquery Column)
    ├─────┼──────┼────────────┤
    │ 101 │  A   │ Sales      │
    │ 102 │  B   │ IT         │
    │ 103 │  A   │ Sales      │
    └─────┴──────┴────────────┘
```

### 3. 심층 동작 원리: 서브쿼리 캐싱 (Subquery Caching)
- **원리**: 서브쿼리의 입력값(Input)과 출력값(Output)을 메모리의 해시 테이블에 저장해 둠.
- **효과**: 동일한 입력값이 들어오면 서브쿼리를 재실행하지 않고 캐시에서 즉시 반환하여 성능 최적화.
- **한계**: 입력값의 종류(Cardinality)가 너무 많으면 캐시 히트율이 떨어져 매 행마다 실제 I/O가 발생함.

### 4. 핵심 기술: Outer Join 으로의 변환
- 스칼라 서브쿼리는 논리적으로 **Left Outer Join**과 동일함.
- 옵티마이저가 판단하기에 서브쿼리 부하가 크다면, 내부적으로 Outer Join으로 구조를 바꾸어 실행함.

- **📢 섹션 요약 비유**: 서브쿼리 캐싱은 '자주 묻는 질문(FAQ)의 답을 머릿속에 외워두어 서재(DB)까지 안 가도 되게 하는 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 스칼라 서브쿼리 vs 레프트 조인 (Left Join)

| 비교 항목 | 스칼라 서브쿼리 | 레프트 조인 (Left Join) |
|:---:|:---|:---|
| **가독성** | **매우 높음** (컬럼 하나로 보임) | 복잡함 (Join 구문 추가) |
| **성능 (대량 데이터)** | 캐시 실패 시 느림 | **일반적으로 더 빠름 (Hash Join)** |
| **데이터 보존** | 원본 행 무조건 보존 | 원본 행 보존 (매칭 없으면 NULL) |
| **데이터 증폭** | **절대 없음** (1:1 보장) | 조인 조건에 따라 행이 늘어날 수 있음 |

### 2. 과목 융합 관점: 소프트웨어 공학 및 운영체제
- **소프트웨어 공학(SE) 융합**: 스칼라 서브쿼리는 코드 레벨의 **Lazy Loading**이나 **함수 호출**과 유사한 개념이며, 도메인 모델에서 간단한 연관 속성을 가져올 때의 논리 구조를 대변한다.
- **운영체제(OS) 융합**: 서브쿼리 캐싱은 OS의 **TLB(Translation Lookaside Buffer)**나 페이지 교체 알고리즘과 원리가 유사하며, 메모리와 성능 사이의 교환(Trade-off) 전략을 따른다.

- **📢 섹션 요약 비유**: 스칼라 서브쿼리가 '필요할 때마다 사전 앱을 켜서 단어를 찾는 것'이라면, 조인은 '단어가 다 적힌 전용 사전을 미리 복사해두는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 코드 명칭(Code Name) 조회 최적화
- **상황**: '주문' 테이블 100만 건을 조회하면서 '주문상태코드'의 '코드명'을 가져와야 함.
- **판단**: 코드 종류는 '결제, 배송, 완료' 3가지뿐임.
- **결단**: **스칼라 서브쿼리 적용**. 코드 종류가 매우 적으므로 서브쿼리 캐시 히트율이 99.9%에 달함. 복잡한 조인 구문 없이도 코드명을 초고속으로 가져올 수 있는 최적의 선택.

### 2. 안티패턴: 대량의 고유 식별자(ID) 기반 스칼라 서브쿼리
- **증상**: 100만 명의 회원 목록을 뽑으면서 각 회원의 '최종 로그인 IP'를 스칼라 서브쿼리로 조회.
- **결과**: 모든 회원의 ID가 다르므로 캐시가 전혀 작동 안 함. 100만 번의 랜덤 I/O 발생으로 시스템 행(Hang) 발생.
- **조치**: **반드시 일반 Join(Hash Join)으로 전환**하여 집합 연산으로 처리할 것.

### 📢 기술사적 결언
> "스칼라 서브쿼리는 **'양날의 검'**이다. 문법적 간결함 뒤에 숨겨진 'Row-by-Row' 연산의 위험성을 간과해서는 안 되기 때문이다. 기술사는 데이터의 **카디널리티(종류의 수)**를 먼저 분석하여, 캐싱 효과를 누릴 수 있는 지점에는 스칼라 서브쿼리를, 집합적 처리가 필요한 지점에는 조인을 배치하는 '전략적 분배' 능력을 갖춰야 한다. 성능 튜닝의 핵심은 결국 **반복 연산을 집합 연산으로 바꾸는 것**임을 명심해야 한다."

- **📢 섹션 요약 비유**: 스칼라 서브쿼리를 함부로 쓰는 것은 '배달 한 번에 음식 하나씩만 가져오는 것'과 같아, 배달료(I/O 비용) 때문에 파산할 수 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (스칼라 서브쿼리 최적 활용 시)

| 지표 | 활용 미흡 시 | 최적 활용 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **쿼리 가독성** | Join 범벅 (난해) | **컬럼 단위 명확** | 유지보수 효율 40% 향상 |
| **소량 데이터 조회 속도** | 조인 오버헤드 | **캐시 활용 즉시 반환** | 응답성 2배 향상 |
| **데이터 무결성** | 조인 시 행 증폭 위험 | **1:1 관계 강제** | 쿼리 결과 신뢰도 확보 |

### 2. 미래 전망
- **Smart Adaptive Query Rewrite**: 옵티마이저가 런타임에 캐시 히트율을 실시간 측정하여, 스칼라 서브쿼리가 비효율적이면 자동으로 조인으로 실행 계획을 바꾸는 **적응형 쿼리 최적화** 고도화.
- **In-memory Vector Processing**: 서브쿼리 캐시 정보를 벡터 연산(SIMD)과 결합하여, 반복 실행 오버헤드 자체를 하드웨어 레벨에서 지워버리는 기술 발전 중.

- **📢 섹션 요약 비유**: 미래의 스칼라 서브쿼리는 우리가 고민하지 않아도 컴퓨터가 '가장 효율적인 기억법'을 찾아 답을 채워주는 똑똑한 자동 완성 시스템이 될 것입니다.
