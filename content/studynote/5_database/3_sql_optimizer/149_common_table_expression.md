# 149. CTE (Common Table Expression) - 공통 테이블 식과 가독성 향상

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 복잡한 SQL 쿼리 내에서 반복적으로 사용되거나 논리적으로 분리된 서브쿼리를 별도의 이름을 가진 **임시 결과 집합**으로 정의하여 재사용할 수 있게 하는 **`WITH`** 절 기반의 구문이다.
> 2. **구조적 이점**: 중첩된 인라인 뷰(Inline View)를 하향식(Top-down)의 평면적인 구조로 리팩토링하여 쿼리의 가독성을 획기적으로 높이며, 동일한 CTE를 한 쿼리 내에서 여러 번 참조할 수 있어 코드 중복을 제거한다.
> 3. **가치**: 일반적인 정적 CTE 외에도 자기 자신을 참조하는 **재귀적(Recursive) CTE**를 통해 계층형 데이터를 탐색할 수 있게 하며, 복잡한 비즈니스 로직을 모듈화하여 유지보수 비용을 절감한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**CTE (Common Table Expression)**는 질의문이 실행되는 동안에만 존재하며, `WITH` 키워드로 시작하여 정의되는 이름이 있는 임시 결과 집합입니다. '이름이 있는 서브쿼리'라고 이해하면 쉽습니다.

### 2. 💡 비유: 수학 문제 풀 때의 '치환' 연산
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        CTE (Common Table Expression) 비유                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 매우 복잡한 수학 방정식을 풀어야 함.                                     │
│                                                                             │
│  1. 일반 서브쿼리: 식 안에 괄호를 계속 넣어서 식이 3미터쯤 길어짐.                │
│     - 읽기도 힘들고 어디가 끝인지 헷갈림.                                        │
│                                                                             │
│  2. CTE 방식 (치환):                                                          │
│     - "먼저 **A = (복잡한 식)** 이라고 하자." (이게 CTE 정의)                    │
│     - "이제 최종 정답은 **A + 100** 이다." (이게 메인쿼리)                       │
│                                                                             │
│  → 복잡한 덩어리에 **'이름표'**를 붙여서 따로 떼어놓고, 나중에 그 이름만 불러서      │
│     간결하게 문제를 푸는 지혜로운 방식입니다!                                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **인라인 뷰의 늪(Spaghetti Query)**: 서브쿼리가 5~6단계씩 겹치면 작성자조차 로직을 파악하기 힘든 가독성 위기 봉착.
- **코드 재사용성 요구**: 같은 서브쿼리를 조인 조건과 필터 조건에서 중복해서 써야 할 때, 이를 한 번만 정의하고 재활용하고 싶은 니즈 발생.

- **📢 섹션 요약 비유**: CTE는 쿼리라는 요리 레시피에서 '미리 만들어 둔 만능 양념장'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. CTE의 기본 구문 구조

```sql
WITH CTE_Name (Col1, Col2) AS (
    -- 서브쿼리 정의
    SELECT ... FROM ... WHERE ...
)
SELECT * FROM CTE_Name; -- 메인쿼리에서 참조
```

| 구성 요소 | 상세 역할 | 비유 |
|:---:|:---|:---|
| **WITH** | CTE의 시작을 알리는 키워드. | "잠깐 주목!" |
| **CTE 명칭** | 임시 테이블로 사용할 이름. | 별명 짓기 |
| **AS ( )** | 실제 데이터를 뽑아내는 쿼리 본체. | 내용물 담기 |
| **참조 쿼리** | 정의된 CTE를 일반 테이블처럼 사용. | 별명 부르기 |

### 2. 서브쿼리 vs CTE 가독성 비교 도해 (ASCII)

중첩 구조가 어떻게 평면 구조로 바뀌는지 시각화한다.

```text
    [ Subquery Style : Inside-Out ]
    SELECT ...
    FROM (SELECT ... 
          FROM (SELECT ... ) -- 괄호 지옥
         )
    
    [ CTE Style : Top-Down ]
    WITH 
    Step1 AS (SELECT ...),  -- 1단계: 재료 준비
    Step2 AS (SELECT ... FROM Step1), -- 2단계: 1단계 결과 사용
    Step3 AS (SELECT ... FROM Step2)  -- 3단계: 최종 조립
    SELECT * FROM Step3;
    * 위에서 아래로 흐르는 논리적 독해 가능!
```

### 3. 심층 동작 원리: 구체화(Materialization) vs 인라인화
- **Inline화**: 옵티마이저가 CTE를 다시 서브쿼리로 풀어서 메인쿼리와 합침.
- **Materialization**: CTE 결과를 임시 테이블에 딱 한 번 써두고 여러 번 읽음. (반복 호출 시 유리)
- **성능**: 동일 CTE를 2번 이상 참조할 때, 테이블 스캔 횟수를 1회로 줄여주는 최적화 효과 발생.

### 4. 핵심 기술: 재귀적 CTE (Recursive CTE)
- `WITH RECURSIVE` 문법을 통해 자기 자신을 호출하며 트리나 그래프 구조를 탐색하는 기술. (오라클의 CONNECT BY를 대체하는 표준 방식)

- **📢 섹션 요약 비유**: 재귀적 CTE는 '거울 두 개를 마주 보게 하여 끝없는 복도를 만드는 마법'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. CTE vs 임시 테이블 (Temporary Table)

| 비교 항목 | CTE (Common Table Expr) | 임시 테이블 (Temp Table) |
|:---:|:---|:---|
| **생명 주기** | **단일 쿼리 실행 종료 시** | 세션 종료 또는 명시적 삭제 시 |
| **저장 위치** | 주로 메모리 (일시적) | 디스크/Temp 영역 (물리적) |
| **오버헤드** | 매우 낮음 | DDL 발생으로 인한 약간의 부하 |
| **복잡도** | 간결함 (SQL 일부) | 복잡함 (별도 생성/관리 필요) |

### 2. 과목 융합 관점: 소프트웨어 공학 및 함수형 프로그래밍
- **소프트웨어 공학(SE) 융합**: CTE는 코드 가독성을 위한 **리팩토링(Refactoring)** 기법 중 '설명용 변수 도입'과 맥을 같이 하며, 쿼리의 복잡도를 제어하는 핵심 수단이다.
- **함수형 프로그래밍 융합**: CTE의 정의와 참조 방식은 함수형 언어의 **파이프라인(Pipeline)** 연산이나 **Lazy Evaluation** 구조와 유사하여, 데이터의 흐름을 선언적으로 정의한다.

- **📢 섹션 요약 비유**: CTE가 '한 번 쓰고 버리는 일회용 컵'이라면, 임시 테이블은 '파티 내내 들고 다니는 개인용 텀블러'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 복잡한 정산 로직의 단계별 분리
- **상황**: [매출 집계] $\to$ [세금 계산] $\to$ [수수료 차감] $\to$ [최종 정산액]을 구해야 함.
- **판단**: 한 쿼리에 다 때려 넣으면 수수료율 하나 바꿀 때마다 쿼리 전체가 흔들림.
- **결단**: **Multi-CTE 적용**. `WITH Sales AS (...), Tax AS (SELECT ... FROM Sales), Fee AS (SELECT ... FROM Tax)`. 각 단계를 CTE로 명확히 분리하여, 중간 단계의 검증이 쉽고 로직 변경이 용이한 '유지보수형 쿼리' 구축.

### 2. 안티패턴: 대량 데이터의 무분별한 CTE 참조
- **증상**: 수억 건의 테이블을 가진 CTE를 메인쿼리에서 여러 번 조인.
- **결과**: 옵티마이저가 매번 CTE를 재실행(Re-evaluation)하도록 결정할 경우 I/O 폭증.
- **조치**: **Materialized Hint**를 쓰거나, 성능이 감당 안 될 정도로 크다면 **물리적인 임시 테이블**에 데이터를 먼저 넣어두고 쓰는 방식으로 전환.

### 📢 기술사적 결언
> "CTE는 SQL을 **'문학적 코드'**로 승화시킨다. 기계가 읽기 좋은 중첩 구조를 인간이 읽기 좋은 선형 구조로 바꿔주기 때문이다. 기술사는 단순히 `WITH` 절을 쓰는 것을 넘어, **CTE가 유발하는 메모리 사용량과 구체화 전략**이 실행 계획에 어떤 영향을 주는지 꿰뚫어야 한다. 가독성을 위해 성능을 희생할 것인지, 아니면 성능을 위해 가독성을 양보할 것인지 결정하는 **'아키텍처적 균형'**의 중심에 CTE가 있음을 명심해야 한다."

- **📢 섹션 요약 비유**: 너무 긴 CTE 명칭은 '주객전도'가 되어 쿼리를 더 어지럽게 만들 수 있으니, 이름은 짧고 명확하게(Step1, Step2 등) 짓는 미덕이 필요합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (CTE 활용 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **코드 분석 시간** | 30분 이상 (복잡 쿼리) | **5분 이내** | 가독성 및 생산성 6배 향상 |
| **디버깅 정확도** | 낮음 (어디서 틀렸는지 모름) | **매우 높음 (단계별 확인)** | 로직 오류 수정 속도 향상 |
| **코드 재사용성** | 수동 복사 붙여넣기 | **CTE 이름 참조** | 중복 코드 획기적 제거 |

### 2. 미래 전망
- **Dynamic CTE Generation**: 사용자 인터페이스의 드래그 앤 드롭 동작에 따라 AI가 최적의 CTE 체인을 자동으로 생성하여 백엔드 쿼리를 구성하는 기술 고도화.
- **Distributed CTE Materialization**: 클라우드 분산 DB에서 CTE의 결과물을 여러 노드에 분산 캐싱하여, 전역적으로 재사용하는 **공유 메모리형 CTE** 아키텍처 진화.

- **📢 섹션 요약 비유**: 미래의 CTE는 단순한 문법을 넘어, 우리가 생각하는 논리의 흐름을 컴퓨터가 실시간으로 지도(Graph)로 그려주는 인터랙티브한 설계 도구가 될 것입니다.
