# 185. 라이브러리 캐시 래치 및 락 - 공유 메모리 동기화와 경합

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 여러 사용자가 동시에 동일한 SQL의 실행 계획을 조회하거나 새로운 계획을 생성할 때, 공유 메모리인 **라이브러리 캐시**의 데이터 일관성을 유지하기 위해 사용되는 DBMS 내부의 초경량 동기화 메커니즘이다.
> 2. **계층적 보호**: 캐시 구조 탐색을 위한 **래치(Latch)**, 특정 SQL 객체(LCO)의 정의를 보호하는 **락(Lock)**, 그리고 실행 중인 객체의 메모리 이탈을 방지하는 **핀(Pin)**이라는 3단계 제어 체계를 통해 동시성을 제어한다.
> 3. **가치**: 하드 파싱(Hard Parsing)이나 부적절한 바인드 변수 사용으로 인한 경합(Contention) 지점을 식별하여 시스템의 '동시 처리 능력'을 극대화하며, 대규모 시스템의 응답 지연(Wait Event) 해소의 핵심 열쇠가 된다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**라이브러리 캐시 래치 및 락**은 공유 메모리 안에서 데이터가 엉키지 않게 관리하는 '질서 유지 도구'입니다. 수천 명의 사용자가 동시에 "이 쿼리 계획 좀 보여줘!"라고 외칠 때, 한 명씩 차례대로 계획을 꺼내 가거나 수정할 수 있게 줄을 세웁니다.

### 💡 비유: 도서관의 '대출 관리대'와 '독서대'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        라이브러리 캐시 동기화 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 래치 (도서관 입구 번호표):                                                │
│     - "나 책(실행 계획) 찾으러 들어간다!"라고 알리는 행위.                        │
│     - 번호표를 받아야만 서가 구역(Hash Bucket)에 들어갈 수 있음.                  │
│                                                                             │
│  2. 락 (특정 책 대출):                                                       │
│     - "이 책 내가 보고 있으니까, 아무도 가져가서 내용을 고치지 마!"                │
│     - 다른 사람이 책 제목(SQL 구조)을 바꾸려 할 때 방어함.                        │
│                                                                             │
│  3. 핀 (책상에 책 펼쳐두기):                                                  │
│     - "이 책 지금 읽는 중이니까, 직원이 와서 책을 치우지(Aging-out) 못하게 해!"    │
│     - 쿼리가 실행되는 동안 메모리에서 실행 계획이 삭제되는 것을 막음.               │
│                                                                             │
│  → 아주 좁은 문(Memory)을 수많은 사람이 **'싸우지 않고 이용하게'** 돕는 규칙들입니다! │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **공유 자원의 경합(Contention)**: 모든 사용자가 SGA라는 하나의 메모리판을 같이 쓰기 때문에, 누군가 데이터를 쓰는 동안 다른 사람이 읽으면 데이터가 깨지는(Corruption) 현상 발생.
- **하드 파싱의 부작용**: 실행 계획을 새로 만들 때는 메모리 구조를 대대적으로 고쳐야(Write) 하므로, 다른 모든 사용자의 조회를 멈추게 하는 병목 발생.

- **📢 섹션 요약 비유**: 래치와 락은 '메모리라는 고속도로'의 '신호등과 차단기'와 같아서, 사고 없이 차(데이터)들이 흐르게 합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 라이브러리 캐시 3단계 보호 메커니즘

| 구분 | 역할 | 수준 | 특징 |
|:---:|:---|:---|:---|
| **Latch (래치)** | 캐시의 **해시 버킷** 구조 탐색 보호. | 물리적 | 매우 짧은 시간 점유. Spin 발생 가능. |
| **Lock (락)** | **SQL 객체(LCO)의 정의** 자체를 보호. | 논리적 | 구문 분석 및 DDL 시 획득. |
| **Pin (핀)** | 실행 중인 **Heap(메모리 덩어리)** 고정. | 실행 | 쿼리 수행 중 실행 계획 삭제 방지. |

### 2. 동기화 객체 획득 프로세스 도해 (ASCII)

사용자가 SQL을 날려 실행 계획을 찾기까지의 보안 검색 과정을 시각화한다.

```text
    [ SQL Request ]
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 1. [ Library Cache Latch ] 획득 : 해시 버킷을 뒤질 권한 확보      │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (Bucket Search...)
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 2. [ Library Cache Lock ] 획득 : 해당 SQL 객체에 대한 독점/공유권  │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (Plan Found)
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 3. [ Library Cache Pin ] 획득 : 실행 계획 메모리 고정 및 실행     │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    [ Execution Result ] ──▶ [ Release Pin/Lock/Latch ]
```

### 3. 심층 동작 원리: Spin 과 Wait
- **원리**: 래치를 얻으려는데 다른 프로세스가 이미 쥐고 있다면, 즉시 잠들지 않고 CPU를 빠르게 회전하며(Spin) 다시 시도함.
- **부하**: 스핀 횟수가 많아지면 CPU 사용량이 폭증하며, 결국 잠들게(Sleep) 되면 컨텍스트 스위칭 비용까지 발생함.

### 4. 핵심 기술: 뮤텍스 (Mutex)
- 현대의 오라클(10g R2 이상)은 무거운 래치 대신 훨씬 가볍고 동시성이 뛰어난 **뮤텍스**를 라이브러리 캐시 동기화에 적극 도입하여 성능을 개선함.

- **📢 섹션 요약 비유**: 래치는 '초스피드 대기표'이고, 락은 '책 대여 예약'입니다. 대기표가 안 나오면 입구(CPU)에서 발만 동동 구르게 됩니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 래치(Latch) vs 락(Lock) 의 결정적 차이

| 비교 항목 | 래치 (Latch) | 락 (Lock / Enqueue) |
|:---:|:---|:---|
| **대상** | 공유 메모리 데이터 구조 | 데이터베이스 오브젝트 (테이블, SQL) |
| **큐 관리** | 없음 (먼저 잡는 놈이 임자) | **Queue(대기행렬) 있음** |
| **데드락** | 발생 불가 (순차 획득 규칙) | **발생 가능** |
| **지속 시간** | 나노초 ~ 마이크로초 | 트랜잭션 종료 시까지 (길 수 있음) |

### 2. 과목 융합 관점: 운영체제 및 자료구조
- **운영체제(OS) 융합**: 래치는 OS의 **Spinlock**과 논리적으로 동일하며, 임계 구역(Critical Section) 진입을 제어하는 원자적(Atomic) 명령을 수행한다.
- **자료구조 융합**: 라이브러리 캐시 래치는 **해시 테이블의 버킷 단위**로 쪼개져 관리(Child Latches)되는데, 이는 경합 범위를 좁히는 '세밀한 잠금(Fine-grained Locking)' 전략의 대표 사례다.

- **📢 섹션 요약 비유**: 래치가 '화장실 문 잠금 장치'라면, 락은 '화장실 건물 출입증'입니다. 잠금 장치가 고장 나면 줄이 길어집니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 'Library Cache Latch' 경합 장애 조치
- **상황**: 동시 접속자가 늘어날 때마다 `latch: library cache` 대기 이벤트가 발생하며 시스템 응답 불가.
- **판단**: 대량의 리터럴 SQL 유입으로 인해 하드 파싱이 빈번해지며, 라이브러리 캐시를 갱신하려는 래치 획득 시도가 충돌 중.
- **결단**: **바인드 변수 적용 및 CURSOR_SHARING 파라미터 조정**. 래치를 잡아야 하는 하드 파싱 횟수 자체를 줄여 경합 원천 제거. 메모리 증설보다 쿼리 구조 표준화가 정답임을 증명.

### 2. 안티패턴: 대량 파싱 중에 수행되는 DDL 작업
- **증상**: 트래픽이 많은 낮 시간에 테이블에 컬럼을 추가(`ALTER TABLE`)함.
- **결과**: 해당 테이블과 연관된 라이브러리 캐시의 모든 쿼리 계획이 무효화(Invalidation)되고, 수천 명의 사용자가 동시에 재파싱을 시도하며 래치/락 경합 폭발.
- **조치**: **스키마 변경은 반드시 서비스 한산 시간대에 수행**. 변경 후에는 실행 계획을 미리 워밍업(Pre-parsing)하여 피크 타임 부하 분산.

### 📢 기술사적 결언
> "라이브러리 캐시의 동기화 메커니즘은 **'보이지 않는 질서'**다. 이 질서가 무너지는 이유는 기술적 결함보다 **'무분별한 쿼리 작성'**이라는 인재(人災)인 경우가 많다. 기술사는 단순히 대기 이벤트를 분석하는 것을 넘어, **파싱의 단가와 빈도**가 전체 공유 자원에 미치는 역학 관계를 이해해야 한다. **뮤텍스나 래치 경합**을 제어하는 최선의 전략은 '더 똑똑한 동기화'가 아니라 **'동기화가 필요 없는 소프트 파싱'**의 정착임을 명심해야 한다."

- **📢 섹션 요약 비유**: 신호등(래치)을 튼튼하게 만드는 것보다, 차(쿼리)들이 서로 길을 비켜주게(바인드 변수) 만드는 것이 가장 좋은 교통 튜닝입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (경합 해소 시)

| 지표 | 도입 전 (경합 발생) | 도입 후 (안정) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **CPU Wait Time** | 50% 이상 (Spin 부하) | **5% 미만** | 연산 효율 극대화 |
| **최대 동시 접속자** | 500명 (Hanging) | **5,000명 이상** | 시스템 확장성 확보 |
| **시스템 가용성** | 돌발 장애 빈번 | **매우 안정적** | 서비스 신뢰도 확보 |

### 2. 미래 전망
- **Latch-free Architectures**: 공유 메모리 구조 자체를 비차단(Wait-free) 자료구조로 재설계하여, 래치 경합 자체를 물리적으로 제거하는 차세대 DB 엔진 진화.
- **In-memory Vector Execution Plans**: 실행 계획을 중앙 메모리가 아닌 각 CPU 코어의 로컬 메모리에 분산 복제하여 동기화 부하를 0으로 만드는 기술 연구 중.

- **📢 섹션 요약 비유**: 미래의 동기화 기술은 우리가 줄을 서지 않아도, 투명한 유령처럼 서로를 통과하며 자기 길을 가는 초능력 같은 모습이 될 것입니다.
