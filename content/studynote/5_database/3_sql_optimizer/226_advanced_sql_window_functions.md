# 고급 SQL 문법과 윈도우 함수 심층 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 윈도우 함수(Window Function, 분석 함수)는 관계형 데이터베이스에서 각 행(Row)의 세부 정보를 유지하면서도, 특정 기준(Partition)으로 나눈 '창문(Window)'을 통해 **행 간의 관계(누계, 순위, 이동 평균)를 연산**하는 고급 SQL 기술이다.
> 2. **가치**: 기존 `GROUP BY`가 그룹화된 단일 결과 행만을 반환하여 원본 데이터를 상실하던 한계를 극복하며, 복잡한 자기 조인(Self-join)과 스칼라 서브쿼리로 수십 줄에 달하던 비효율적인 SQL을 단 한 줄의 직관적인 코드로 최적화한다.
> 3. **융합**: 데이터 웨어하우스(DW) 환경에서 대규모 데이터의 다차원(OLAP) 분석을 위한 `ROLLUP`, `CUBE` 구문과 결합되어, RDBMS 엔진이 실시간 비즈니스 인텔리전스(BI) 집계 엔진으로 기능하게 하는 핵심 원동력이 된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 윈도우 함수는 데이터 집합에 대해 부분적인 집계나 순위, 행 간의 참조 연산을 수행하지만, 반환 결과에서 입력된 원본 행들의 개수와 디테일(Detail)을 그대로 유지하는 선언적 함수다.

- **💡 비유**: `GROUP BY`가 "우리 반 전체 학생들의 평균 키를 구하고 학생 명부는 파쇄기에 버려버리는 방식"이라면, `윈도우 함수`는 "각 학생의 학생부를 그대로 둔 채, 학생 이름 옆에 '우리 반 평균 키와의 차이'와 '반 석차'를 도장으로 예쁘게 찍어주는 방식"입니다.

- **등장 배경**:
  1. **RDBMS의 절차적 한계**: SQL은 비절차적 언어이므로, C나 Java처럼 "이전 행의 값을 변수에 저장했다가 다음 행에 더하는(누적 합계)" 식의 절차적(Sequential) 데이터 처리가 불가능하여 극도의 서브쿼리 남용을 초래했다.
  2. **분석 쿼리 작성의 고통**: "부서별 급여 1등 직원의 이름과 급여를 구하라"는 단순한 비즈니스 질문조차, 과거에는 부서별 MAX 급여를 임시 뷰로 만들고 원본 테이블과 다시 조인해야 하는 비효율적인 연산을 강제했다.
  3. **표준의 정립**: 이러한 데이터 분석의 고통을 덜기 위해 ANSI/ISO SQL:2003 표준부터 윈도우 함수 기능이 공식 도입되었고, 현재 모든 주요 RDBMS(Oracle, MySQL 8.0+, PostgreSQL)가 이를 완벽히 지원한다.

- **📢 섹션 요약 비유**: 윈도우 함수는 복잡한 수식이 얽힌 엑셀(Excel)의 드래그 채우기(Auto Fill) 기능을 데이터베이스 엔진 가장 깊숙한 곳에서 SQL 단 한 줄로 실행할 수 있게 해주는 마법 지팡이입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 윈도우 함수의 기본 신택스 (Syntax)와 3대 구성 요소

윈도우 함수는 항상 `OVER()` 절을 동반하여 연산의 대상을 정의한다.

```sql
SELECT 
    함수명(컬럼) OVER (
        [PARTITION BY 컬럼]      -- 1. 창문을 나눌 기준 (그룹화)
        [ORDER BY 컬럼]         -- 2. 창문 안에서 데이터를 정렬할 기준
        [ROWS / RANGE BETWEEN]  -- 3. 연산을 수행할 행의 정확한 범위 (프레이밍)
    )
FROM 테이블;
```

| 구성 요소 | 역할 및 동작 원리 | 엑셀(Excel) 비유 |
|:---|:---|:---|
| **PARTITION BY** | 전체 결과 집합을 논리적으로 작은 윈도우(그룹)로 나눔 (`GROUP BY`와 유사하나 원본 행은 보존됨) | 시트를 부서별로 쪼갠 탭(Tab) 만들기 |
| **ORDER BY** | 파티션 내부에서 연산이 적용될 순서를 결정 (누적합이나 순위 계산 시 필수) | 탭 안에서 데이터를 날짜순 정렬 (오름차순) |
| **ROWS BETWEEN** | (프레이밍) 현재 행을 기준으로 "위쪽 2줄부터 아래쪽 1줄까지" 연산할 윈도우의 크기를 동적으로 설정 | 수식에서 참조할 셀의 범위를 마우스로 드래그 |

---

### 2. 핵심 윈도우 함수 카테고리

1. **순위(Ranking) 함수**
   - `ROW_NUMBER()`: 값이 같아도 무조건 고유한 1, 2, 3등 부여 (페이징 처리에 유용).
   - `RANK()`: 동점자에게 같은 등수를 주고 다음 등수는 건너뜀 (예: 1등, 1등, 3등).
   - `DENSE_RANK()`: 동점자에게 같은 등수를 주되, 다음 등수를 건너뛰지 않고 빽빽하게 부여 (예: 1등, 1등, 2등).

2. **순서/탐색(Navigation) 함수**
   - `LAG(컬럼, N)`: 현재 행 기준으로 N번째 '이전' 행의 값을 가져옴 (전일 대비 매출 증감액 계산에 필수).
   - `LEAD(컬럼, N)`: 현재 행 기준으로 N번째 '다음' 행의 값을 가져옴.
   - `FIRST_VALUE()` / `LAST_VALUE()`: 파티션 내에서 정렬된 기준 첫 번째 / 마지막 값을 가져옴.

3. **그룹/집계(Aggregation) 함수 + OVER()**
   - `SUM() OVER(ORDER BY ...)`: 1월부터 현재 월까지의 **누적 합계(Running Total)**를 계산.
   - `AVG() OVER(ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)`: 주식 차트에서 많이 쓰이는 **3일 이동 평균(Moving Average)**을 단숨에 계산.

---

### 3. 다차원 OLAP 집계 함수 (ROLLUP, CUBE, GROUPING SETS)

`GROUP BY` 절을 확장하여 부서별, 연도별 데이터의 '소계(Subtotal)'와 '총계(Grand Total)'를 서버 단에서 한 번에 뿜어내는 기능이다.

- **`ROLLUP(A, B)`**: 계층적 구조(상하 관계)를 갖는 데이터 집계. (A, B 집계) -> (A의 소계) -> (전체 총계) 반환.
- **`CUBE(A, B)`**: 가능한 모든 차원(차수)의 조합에 대한 소계를 계산. (A, B), (A), (B), (전체) 반환. (연산량이 기하급수적으로 커짐)
- **`GROUPING SETS(A, B)`**: 롤업이나 큐브처럼 자동 조합하지 않고, 개발자가 명시적으로 지정한 그룹(A의 그룹, B의 그룹)만 집계.

- **📢 섹션 요약 비유**: 윈도우 함수가 나만의 기준(OVER)으로 옆 사람의 시험지를 슬쩍 커닝(LAG)하거나 앞사람까지의 점수들을 더하는(SUM OVER) 디테일한 기술이라면, 다차원 집계(ROLLUP)는 교장 선생님을 위해 학년별 평균, 반별 평균, 전교 평균을 큰 덩어리로 턱턱 요약해 내는 기술입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 조인(Join) 방식과 윈도우 함수 성능 심층 비교

"최근 3일 치 이동 평균"을 구하는 로직의 아키텍처 비교.

| 구분 | 일반 SQL (Self-Join + Subquery) | 윈도우 함수 (ROWS BETWEEN) |
|:---|:---|:---|
| **작동 원리** | 3일 치 범위를 찾기 위해 테이블을 자기 자신과 조인(`ON A.date BETWEEN B.date-2 AND B.date`) | 데이터를 한 번 정렬(Sort)한 후, 메모리 창(Window)을 이동시키며 스캔 |
| **디스크 I/O** | 원본 테이블을 최소 2번 이상 읽음 (비용 N^2) | **원본 테이블을 단 1번만 읽음 (비용 N)** |
| **메모리(PGA) 부하** | 카티션 곱(Cartesian Product)에 가까운 조인 메모리 소모 | `ORDER BY`에 의한 정렬 부하 발생 |
| **코드 가독성** | 최소 10줄 이상의 복잡한 중첩 쿼리 | 단 1줄의 `AVG() OVER(...)` 구문으로 완결 |

### 과목 융합 관점

- **SQL 옵티마이저 (Optimizer)**: 윈도우 함수는 옵티마이저의 실행 계획 트리에서 가장 마지막 단계(조인과 일반 `GROUP BY`가 모두 끝난 직후)에 연산된다. 정렬 부하(`ORDER BY`)가 크기 때문에 인덱스를 활용한 **소트 머지(Sort Merge) 최적화**가 필수적이다.
- **분산 시스템 (Spark/Hadoop)**: 대용량 데이터 분산 프레임워크인 Apache Spark의 DataFrame API에서도 윈도우 함수(`Window.partitionBy()`)를 완벽히 융합 지원한다. 이때 파티션별로 데이터가 물리적인 노드로 셔플(Shuffle)되어 분산 연산된다.

- **📢 섹션 요약 비유**: 옛날에는 달리는 기차의 속도 변화를 알기 위해 기차역마다 사람을 세워두고 일일이 기록을 취합(Self-Join)해야 했다면, 윈도우 함수는 기차 머리에 달린 속도계(메모리 연산)가 어제와 오늘의 속도를 스스로 기억하고 실시간으로 계산해 내는 혁신입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 커머스 앱의 유저 재방문(Retention) 및 퍼널(Funnel) 분석

- **문제 상황**: 데이터 분석가가 "고객이 상품을 장바구니에 담은 직후 10분 이내에 결제 페이지로 넘어간 전환율"을 구하려 함. RDBMS에서 유저의 행동 로그 수천만 건을 자기 조인(Self-Join)하려다 DB CPU 100% 장애 유발.
- **의사결정 (LEAD 함수 도입)**:
  1. 행동 로그 테이블을 1회 풀 스캔(Full Scan)하면서 `LEAD(행동명) OVER (PARTITION BY 유저ID ORDER BY 접속시간)` 윈도우 함수 사용.
  2. 현재 로그가 '장바구니 담기'이면서 동시에 방금 구한 **다음 행(LEAD)** 로그가 '결제 페이지'인 행만 `WHERE` 조건으로 필터링.
  3. `시간 차이(다음 행 시간 - 현재 시간)`가 10분 이내인 데이터만 카운트. (실행 시간 10분 -> 2초 내로 단축)

### 도입 체크리스트
- **정렬(Sort) 메모리 공간 (PGA/Work_mem)**: 윈도우 함수 내에 `ORDER BY`가 들어있으면 데이터베이스는 메모리(PGA) 상에서 대규모 정렬을 시도한다. RAM이 부족하면 느린 디스크 기반의 Temp 영역으로 스왑(Swap)이 발생해 치명적인 성능 저하가 일어나므로, 파티션 인덱스가 적절히 타는지 확인해야 한다.
- **실행 순서(Logical Order of Execution) 인지**: 윈도우 함수는 `WHERE` 절이나 `HAVING` 절 안에서 절대 사용할 수 없다. (윈도우 함수가 그보다 나중에 실행되기 때문). 따라서 윈도우 함수의 결과를 필터링하려면 반드시 서브쿼리(인라인 뷰)나 CTE(WITH 절)로 한 번 감싸서 바깥쪽 쿼리의 `WHERE` 절에 써야 한다.

### 안티패턴: 단순 순번 매기기에 ROWNUM이나 복잡한 변수 남용
- 오라클의 `ROWNUM`이나 MySQL의 `@var:=@var+1` 같은 로컬 변수를 사용하여 억지로 그룹 내 순위를 매기는 구식 기법. 데이터 정합성을 깨뜨리기 쉽고 조인이 섞이면 순서가 엉망이 된다. 그룹 내 순번 부여는 무조건 `ROW_NUMBER() OVER(PARTITION BY ...)`를 표준으로 사용해야 유지보수성과 성능을 동시에 잡을 수 있다.

- **📢 섹션 요약 비유**: 망치(일반 SQL)로 나사를 박으려다 손을 다치는 것보다, 전동 드라이버(윈도우 함수)라는 고급 도구의 사용법과 실행 순서(전원을 켜고 돌려야 함)를 익히면 작업 생산성이 100배 달라집니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 레거시 SQL 코딩 방식 | 윈도우 함수 / 다차원 그룹핑 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **디스크 스캔 (I/O)** | 잦은 서브쿼리와 자기 조인으로 다중 스캔 | 단 1번의 스캔(One-pass)으로 처리 | **물리/논리적 I/O 연산 비용 90% 극감** |
| **코드 라인 수 (LOC)** | 50줄 이상의 스파게티 쿼리 | 5~10줄의 선언적 문법으로 요약 | **가독성 향상 및 쿼리 리뷰(Review) 공수 최소화** |
| **데이터 분석 역량** | DBA의 개입 없이 복잡한 지표 추출 불가 | 분석가가 직접 시계열/퍼널 지표 즉시 도출 | **현업 부서의 데이터 자생력(Data Literacy) 폭발적 강화** |

### 미래 전망
- **실시간 스트리밍(Streaming) SQL과의 병합**: Apache Flink, ksqlDB와 같은 초당 수백만 건의 스트림 프로세싱 환경에서, 끊임없이 들어오는 무한한 데이터 스트림을 자르는(Tumbling/Sliding) 역할로서 윈도우 함수의 `ROWS BETWEEN` 프레이밍(Framing) 개념이 스트리밍 엔진의 핵심 문법으로 그대로 차용되고 있다.
- **클라우드 데이터 웨어하우스 최적화**: Snowflake나 BigQuery 같은 최신 분산 엔진은 윈도우 함수의 정렬 부하(Sort)를 여러 노드에 완벽하게 병렬(Parallel) 분산 처리하는 방향으로 최적화되어, 테라바이트 급의 분석 함수도 수 초 만에 끝내는 시대를 열고 있다.

- **📢 섹션 요약 비유**: 윈도우 함수는 개발자와 데이터베이스 모두에게 씌워진 '답답한 조인(Join)의 족쇄'를 풀어버린 해방의 열쇠입니다. 이 강력한 언어를 다루는 자만이 진정한 데이터 분석의 마법사로 거듭날 수 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 선생님이 체육 시간에 아이들의 번호를 매길 때, 반 전체 아이들을 키순으로 줄 세우고(일반 쿼리) 종이에 적는 건 너무 평범해요.
2. **윈도우 함수**는 반 아이들을 그대로 둔 채로 홀로그램 창문(Window)을 씌워요. 창문 안에서는 아이들이 "남학생/여학생(Partition)"으로 딱 나뉘고, 그 속에서 "가장 빠른 순서(Order)"대로 1등, 2등 딱지가 실시간으로 아이들 머리 위에 뿅! 하고 생겨나는 마법이에요.
3. 또 **LAG/LEAD**라는 주문을 외우면, 내 머리 위에 '내 앞사람 점수'랑 '내 뒷사람 점수'가 자동으로 둥둥 떠올라서 친구들과 점수 차이를 바로 계산할 수 있답니다!
