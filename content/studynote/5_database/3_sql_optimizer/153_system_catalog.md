# 153. 시스템 카탈로그 (System Catalog) - 메타데이터의 물리적 저장소

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스를 구성하는 모든 객체(테이블, 뷰, 인덱스, 사용자 등)에 대한 정의 정보와 통계 데이터를 담고 있는 **'데이터베이스의 데이터베이스'**로, DBMS가 스스로를 관리하기 위해 생성하고 유지하는 특수 시스템 테이블들의 집합이다.
> 2. **역할**: 사용자가 SQL을 실행할 때 접근 권한을 확인하고, 옵티마이저가 최적의 실행 경로를 결정하기 위한 기초 자료(행 수, 데이터 분포 등)를 제공하며, 시스템의 무결성과 일관성을 유지하는 뇌의 중추 역할을 수행한다.
> 3. **가치**: 데이터베이스 시스템의 **자기 기술(Self-describing)** 능력을 실현하며, 데이터 아키텍처의 투명성을 확보하고 도구(CASE Tool, BI)들이 시스템 구조를 자동으로 파악할 수 있는 표준 창구를 제공한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**시스템 카탈로그 (System Catalog)**는 DBMS 내의 모든 메타데이터를 저장하는 저장소입니다. 사용자 데이터가 담긴 '일반 테이블'과 달리, '테이블의 이름이 무엇인지', '컬럼 타입이 무엇인지'와 같은 메타 정보를 담고 있습니다. 흔히 **데이터 사전(Data Dictionary)**이라고도 불립니다.

### 2. 💡 비유: 거대 도서관의 '대출 관리 장부'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        시스템 카탈로그 개념 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 수억 권의 책(데이터)이 있는 도서관을 운영함.                             │
│                                                                             │
│  1. 일반 서가: 소설, 잡지 등 실제 읽을거리(사용자 데이터)가 꽂힌 곳.               │
│  2. 시스템 카탈로그 (관리 장부):                                               │
│     - "A 구역에는 과학 책이 500권 있다." (통계 정보)                             │
│     - "철수는 3층 열람실에 들어갈 수 없다." (권한 정보)                          │
│     - "책의 제목은 반드시 10글자 이내여야 한다." (제약 조건)                     │
│                                                                             │
│  → 사서(DBMS)가 도서관을 엉망진창이 되지 않게 **'스스로를 관리하기 위해'**          │
│     꼭 쥐고 있는 마법의 장부입니다!                                             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **시스템 자가 제어 필요성**: 수천 개의 테이블이 얽힌 환경에서 매번 소스 코드를 뒤져 구조를 파악할 수 없으므로, DB 내부에 정보를 공식화하여 보관.
- **옵티마이저의 탄생**: 데이터를 가장 빠르게 찾는 경로를 계산하기 위해선, 각 테이블에 데이터가 얼마나 들어있는지(Metadata) 알아야 함.

- **📢 섹션 요약 비유**: 시스템 카탈로그는 데이터베이스 시스템이 자기 자신을 들여다보는 '거울'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 카탈로그에 저장되는 주요 정보 분류

| 분류 | 상세 항목 | 주요 용도 |
|:---:|:---|:---|
| **객체 정보** | 테이블명, 컬럼명, 뷰 정의, 인덱스 구조. | SQL 구문 분석 및 객체 확인 |
| **무결성 정보** | PK/FK 제약 조건, NOT NULL, CHECK. | 데이터 입력 시 정합성 검증 |
| **보안 정보** | 사용자 계정, 비밀번호, 권한(Grant). | 접근 제어 및 침입 차단 |
| **통계 정보** | 행 수(Card), 블록 수, 컬럼 분포도. | **옵티마이저의 경로 결정** |

### 2. 시스템 카탈로그 접근 아키텍처 (ASCII)

사용자의 요청이 카탈로그를 거쳐 처리되는 흐름을 시각화한다.

```text
    [ User Query ] : "SELECT * FROM Emp"
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ SQL Parser ] : 카탈로그 조회                                   │
    │  - "Emp 라는 테이블이 실제로 존재하는가?" (Schema Check)          │
    │  - "사용자가 조회 권한을 가졌는가?" (Security Check)              │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (Validation Success)
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Optimizer ] : 카탈로그 통계 조회                               │
    │  - "Emp 테이블은 100만 건이네? 인덱스를 타는 게 빠르겠군."        │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    [ Real Data Access ] : **물리 디스크의 Emp 테이블 데이터 읽기**
```

### 3. 심층 동작 원리: 읽기 전용과 자동 갱신
- **읽기 전용 (Read-Only)**: 사용자는 일반적인 `INSERT`, `DELETE` 명령어로 카탈로그를 직접 고칠 수 없음 (DB 파괴 위험 방지).
- **자동 갱신**: 사용자가 `CREATE TABLE`을 수행하거나 `ANALYZE` 명령을 날리면 DBMS가 내부적으로 카탈로그 정보를 최신화함.

### 4. 핵심 기술: 데이터 사전 뷰 (Dictionary Views)
- 사용자가 복잡한 시스템 테이블을 쉽게 볼 수 있도록 제공하는 가상 테이블.
- Oracle: `USER_TABLES`, `ALL_COLUMNS`, `DBA_INDEXES` 등.
- MySQL: `INFORMATION_SCHEMA`.

- **📢 섹션 요약 비유**: 카탈로그는 '출입국 심사대'와 같아서, 허가된 사람(권한)과 존재하는 목적지(테이블)인지 확인한 후에만 통과시켜 줍니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 시스템 카탈로그 vs 일반 사용자 테이블

| 비교 항목 | 시스템 카탈로그 | 일반 사용자 테이블 |
|:---:|:---|:---|
| **저장 내용** | **메타데이터** (구조, 통계) | **실제 데이터** (값) |
| **생성 주체** | **DBMS 엔진** | 응용 프로그램 / 개발자 |
| **갱신 방식** | DDL 실행 시 자동 갱신 | DML(Insert/Update) 실행 시 |
| **직접 수정** | **불가능 (치명적)** | 가능 |
| **비유** | **도서관 관리 지침서** | **도서관에 있는 소설책** |

### 2. 과목 융합 관점: 운영체제 및 보안
- **운영체제(OS) 융합**: OS의 **파일 시스템 메타데이터(Inode, FAT)**와 논리적으로 동일하며, 물리적 저장 장치의 위치와 속성을 관리하여 데이터 추상화를 실현한다.
- **보안(Security) 융합**: 시스템 카탈로그는 **접근 제어 행렬(Access Control Matrix)**의 물리적 구현소이며, 카탈로그 자체가 해킹될 경우 전체 시스템의 통제권을 상실하므로 가장 강력한 보호 대상이 된다.

- **📢 섹션 요약 비유**: 일반 테이블이 '통장의 잔액'이라면, 카탈로그는 '은행의 계좌 개설 규칙과 보안 보안 카드 번호'입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: SQL 성능 급격 저하 원인 분석
- **상황**: 잘 돌아가던 쿼리가 갑자기 느려짐. 테이블 구조는 바뀐 게 없음.
- **판단**: 데이터가 급증했는데 카탈로그 내의 '통계 정보'가 과거에 머물러 있을 가능성 농후.
- **결단**: **카탈로그 통계 정보 갱신(Analyze/Gather Stats) 수행**. 옵티마이저가 최신 행 수와 데이터 분포를 카탈로그에서 다시 읽게 하여, 최적의 인덱스 실행 계획을 재생성하도록 유도.

### 2. 안티패턴: 카탈로그 테이블에 대한 직접 UPDATE 시도
- **증상**: 관리자 권한으로 시스템 테이블에 직접 들어가 컬럼명을 강제로 고침.
- **결과**: 관련된 인덱스, 뷰, 권한 정보와의 연결 고리가 끊겨 데이터베이스 인스턴스 전체가 깨짐(Corruption).
- **조치**: **반드시 표준 DDL(`ALTER TABLE`)을 사용**하고, 시스템 카탈로그는 오직 조회(`SELECT`) 목적으로만 활용할 것.

### 📢 기술사적 결언
> "시스템 카탈로그는 데이터베이스의 **'디지털 자아'**다. 자아를 잃은 DB는 자신이 무엇을 저장하고 있는지조차 알지 못하기 때문이다. 기술사는 카탈로그 조회를 통해 **전사 데이터 아키텍처의 현행화 상태**를 진단하고, 옵티마이저가 카탈로그 정보를 바탕으로 내리는 '판단의 근거'를 해석할 줄 알아야 한다. 카탈로그 관리 능력이 곧 **DB 장애 복구 및 성능 최적화**의 기술적 깊이를 결정함을 명심해야 한다."

- **📢 섹션 요약 비유**: 카탈로그 통계 갱신을 잊는 것은 '지도가 10년 전 것인데 최단 경로를 찾으려는 운전자'와 같아 길을 헤매게(성능 저하) 됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (카탈로그 활용 시)

| 지표 | 활용 미흡 시 | 최적 활용 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **실행 계획 정확도** | 50% (추측성) | **99% (데이터 기반)** | 쿼리 성능 안정성 확보 |
| **스키마 분석 시간** | 수 일 (수작업) | **수 분 (SQL 조회)** | 시스템 파악 효율 극대화 |
| **데이터 정합성** | 오류 방치 | **실시간 제약 검증** | 무결성 사고 원천 차단 |

### 2. 미래 전망
- **AI-Driven Self-Tuning Catalog**: 사람이 시키지 않아도 AI가 데이터 변화 패턴을 감지하여 실시간으로 카탈로그 통계를 미세 조정(Micro-stats)하는 기술 고도화.
- **Open Metadata Standard (Atlas, Amundsen)**: 특정 벤더의 카탈로그를 넘어, 전사적 데이터 계보(Lineage)와 연동되는 **연합 메타데이터 카탈로그** 환경 보편화.

- **📢 섹션 요약 비유**: 미래의 시스템 카탈로그는 죽어있는 장부가 아니라, 데이터의 흐름을 실시간으로 기록하고 학습하여 스스로 진화하는 '살아있는 데이터 뇌'가 될 것입니다.
