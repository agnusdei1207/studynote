# 145. 윈도우 함수 (Window Function) - 분석 및 순위 연산

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 행과 행 사이의 관계를 정의하여 집계, 순위, 분석 연산을 수행하는 함수로, 결과 집합의 행 수(Cardinality)를 줄이지 않으면서도 각 행에 대해 집합적 연산 결과를 부여하는 **고급 SQL 분석 도구**다.
> 2. **작동 구조**: **`OVER`** 절을 통해 연산의 범위인 '윈도우'를 지정하며, **`PARTITION BY`**로 그룹을 나누고 **`ORDER BY`**로 정렬 순서를 정하여 이동 평균, 누적 합계, 순위 등을 계산한다.
> 3. **가치**: 복잡한 셀프 조인(Self-join)이나 서브쿼리 없이도 실시간 비즈니스 통계를 산출할 수 있게 하며, 쿼리의 가독성을 높이고 옵티마이저의 연산 효율을 극대화하는 **현대적 SQL 아키텍처**의 정수다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**윈도우 함수 (Window Function)**는 분석 함수(Analytic Function)라고도 불립니다. 일반 집계 함수(`SUM`, `AVG`)가 `GROUP BY`를 통해 행을 합쳐버리는 것과 달리, 윈도우 함수는 **원래의 행 형태를 그대로 유지**하면서 그 옆에 집계 결과를 나란히 보여줍니다.

### 2. 💡 비유: 전교생 성적표와 반별 등수
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        윈도우 함수 개념 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 전교생 1,000명의 성적표를 인쇄하고 있음.                                 │
│                                                                             │
│  1. 일반 집계 (GROUP BY): "우리 반 평균 점수는 80점이야."                       │
│     - 반별로 한 줄씩만 요약됨. (개별 학생 정보는 사라짐)                         │
│                                                                             │
│  2. 윈도우 함수 (OVER):                                                       │
│     - "내 점수는 90점인데, 우리 반(Partition) 안에서는 1등(Rank)이야."           │
│     - 1,000명의 성적표가 다 출력되면서, 각자 옆에 반별 평균과 등수가 적힘!         │
│                                                                             │
│  → 전체를 유지하면서 **'특정 범위(Window)'** 내의 비교 결과를 덧붙여주는           │
│     똑똑한 메모장과 같습니다!                                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **복잡한 통계 요구**: "직전 3개월 매출 평균", "현재 행까지의 누적 합계" 등 행 간 연산 요구 증가.
- **조인 부하 감소**: 과거에는 이런 정보를 얻기 위해 같은 테이블을 여러 번 셀프 조인해야 했으나, 성능과 가독성 문제가 심각했음.

- **📢 섹션 요약 비유**: 윈도우 함수는 '망원경'과 같아서, 전체 풍경(데이터)은 그대로 보면서 특정 구역(Window)만 확대해서 분석하게 해줍니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 윈도우 함수의 구문 구조

```sql
SELECT 컬럼, 
       함수(인자) OVER (PARTITION BY 그룹컬럼 ORDER BY 정렬컬럼 WINDOWING절)
FROM 테이블;
```

| 구성 요소 | 상세 역할 | 비유 |
|:---:|:---|:---|
| **PARTITION BY** | 데이터를 분석할 소그룹으로 나눔. (필수 아님) | 반별로 나누기 |
| **ORDER BY** | 윈도우 안에서의 행 순서를 결정. | 성적순 줄 세우기 |
| **WINDOWING** | 현재 행을 기준으로 연산할 물리적 범위 지정. (ROWS/RANGE) | 내 앞뒤 3명만 |

### 2. 윈도우 함수 작동 도해 (ASCII)

누적 합계(Cumulative Sum)가 계산되는 물리적 흐름을 시각화한다.

```text
    [ Data Set ]        [ Window: Current Row + Preceding ]      [ Result ]
    ┌────┬─────┐        ┌─────────────────────────────────┐      ┌────────┐
    │ 날짜│매출 │        │ Window Logic (Running Total)    │      │ 누적합 │
    ├────┼─────┤        ├─────────────────────────────────┤      ├────────┤
    │ 01 │ 100 │ ──▶    │ [100]                           │ ──▶  │  100   │
    │ 02 │ 150 │ ──▶    │ [100 + 150]                     │ ──▶  │  250   │
    │ 03 │ 200 │ ──▶    │ [100 + 150 + 200]               │ ──▶  │  450   │
    └────┴─────┘        └─────────────────────────────────┘      └────────┘
    * SQL: SUM(매출) OVER (ORDER BY 날짜 ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
```

### 3. 주요 함수의 분류
1. **순위 함수**: `RANK`(공동 순위 건너뜀), `DENSE_RANK`(공동 순위 유지), `ROW_NUMBER`(무조건 순차).
2. **집계 함수**: `SUM`, `AVG`, `COUNT`, `MAX`, `MIN` (OVER 절과 결합).
3. **행 순서 함수**: `LAG`(이전 행), `LEAD`(다음 행), `FIRST_VALUE`, `LAST_VALUE`.
4. **비율 함수**: `RATIO_TO_REPORT`(백분율), `PERCENT_RANK`.

### 4. 핵심 기술: 윈도우 버퍼링 및 캐싱
- DBMS 엔진은 윈도우 연산을 위해 메모리에 데이터를 일시적으로 버퍼링하며, 한 번의 스캔으로 여러 윈도우 연산을 끝내기 위해 **병렬 처리 최적화** 기법을 사용함.

- **📢 섹션 요약 비유**: LAG 함수는 '내 뒤에 누가 오는지 백미러로 확인하는 것'과 같아, 데이터의 흐름을 파악하게 해줍니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. GROUP BY vs WINDOW FUNCTION

| 비교 항목 | GROUP BY | WINDOW FUNCTION (OVER) |
|:---:|:---|:---|
| **결과 행 수** | 그룹당 **1행**으로 압축 | **원본 행 수 유지** |
| **정보 보존** | 개별 행 정보 소실 | **개별 정보 + 집계 정보** 공존 |
| **조인 필요성** | 상세 데이터와 비교 시 Join 필수 | **Join 없이 한 문장으로 가능** |
| **비유** | 요약 보고서 | 주석이 달린 전수 조사서 |

### 2. 과목 융합 관점: 통계 및 알고리즘
- **통계학 융합**: 윈도우 함수는 시계열 분석의 **이동 평균(Moving Average)**과 **지수 평활법**을 SQL 레벨에서 직접 구현하는 도구가 된다.
- **알고리즘 융합**: 윈도우 함수는 자료구조의 **슬라이딩 윈도우(Sliding Window)** 알고리즘을 확장한 것이며, 정렬 연산의 시간 복잡도($O(N \log N)$) 내에서 대부분의 분석 연산을 완료한다.

- **📢 섹션 요약 비유**: GROUP BY가 '평균 신장'을 구하는 것이라면, 윈도우 함수는 '친구들 사이에서 내 키가 몇 번째인지'를 구하는 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 일자별 매출 증감율 분석
- **상황**: "오늘 매출이 어제보다 몇 % 올랐는가?"를 모든 날짜에 대해 보고해야 함.
- **판단**: 일반 조인으로는 전날 데이터를 가져오기 위해 테이블을 한 번 더 읽어야 함.
- **결단**: **LAG 함수 사용**. `(Sales - LAG(Sales) OVER(ORDER BY Date)) / LAG(Sales) OVER(ORDER BY Date) * 100`. 테이블을 단 한 번만 읽고도(Single Pass) 모든 날짜의 증감율을 초고속으로 산출.

### 2. 안티패턴: ORDER BY 없는 집계 윈도우 함수
- **증상**: `SUM(Amt) OVER (PARTITION BY Dept)` 처럼 순서 없이 사용.
- **결과**: 부서 전체 합계가 모든 행에 똑같이 찍힘. (물론 의도한 것일 수도 있지만, 누적 합계가 필요할 땐 치명적 실수)
- **조치**: **누적 연산이 필요하면 반드시 ORDER BY**를 포함하고, 정지된 집계가 필요하면 윈도우 절을 신중히 선택할 것.

### 📢 기술사적 결언
> "윈도우 함수는 SQL의 **'제3의 혁명'**이다. 조인과 서브쿼리에 갇혀있던 집합적 사고를 **'관계적 사고'**로 확장했기 때문이다. 기술사는 단순 순위 매기기를 넘어, **데이터의 파이프라인 연산** 과정에서 윈도우 함수가 어떻게 중간 연산 부하를 줄이는지 이해해야 한다. 특히 실시간 대용량 분석 환경에서 **Self-join을 윈도우 함수로 대체**하는 것만으로도 시스템 성능을 수십 배 개선할 수 있는 'SQL 전략가'가 되어야 한다."

- **📢 섹션 요약 비유**: 윈도우 함수를 모르는 개발자는 '손가락으로 숫자를 세는 사람'이고, 아는 개발자는 '엑셀 수식을 다루는 사람'만큼 차이가 납니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (윈도우 함수 활용 시)

| 지표 | 도입 전 (Self-join 위주) | 도입 후 (Window Func) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **쿼리 수행 시간** | 수 초 (I/O 과다) | **밀리초 (Single Scan)** | 응답 속도 5~10배 향상 |
| **SQL 코드 라인 수** | 100줄 (복잡) | **10줄 (간결)** | 유지보수 생산성 극대화 |
| **데이터 정합성** | 조인 실수 잦음 | **함수 자체 보장** | 로직 오류 0% 실현 |

### 2. 미래 전망
- **Stream Processing Windowing**: 정지된 DB뿐만 아니라 Flink나 Spark Streaming 등에서 실시간 유입 데이터에 대한 윈도우 함수 지원이 데이터 아키텍처의 표준이 됨.
- **AI-Predictive Windowing**: 과거 윈도우 패턴을 기반으로 AI가 다음 행의 값을 예측하여 채워주는 **예측형 윈도우 함수** 기술 연구 중.

- **📢 섹션 요약 비유**: 미래의 윈도우 함수는 단순한 계산기를 넘어, 데이터의 과거와 현재를 연결하여 미래의 궤적을 그려주는 '지능형 나침반'이 될 것입니다.
