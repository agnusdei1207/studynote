# 178. 조건절 푸시다운 (Pushing Predicates) - 조인 전 필터링 최적화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 뷰(View)나 서브쿼리가 포함된 조인 쿼리에서, 바깥쪽(Main Query)에 정의된 필터 조건을 뷰의 내부로 밀어 넣어(Push-down) 조인 연산이 수행되기 전에 미리 데이터를 걸러내는 최적화 기법이다.
> 2. **작동 원리**: 옵티마이저가 뷰를 해체(Merging)할 수 없는 상황에서도 바깥쪽의 상수를 안쪽 쿼리의 조건으로 전달함으로써, 불필요한 전체 테이블 스캔을 방지하고 인덱스 스캔을 유도한다.
> 3. **가치**: 중간 결과 집합의 크기를 최소화하여 메모리 사용량과 I/O 부하를 획기적으로 줄이며, 복잡한 계층 구조나 집계 뷰가 포함된 대형 시스템 쿼리의 성능을 결정짓는 핵심 지능이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**조건절 푸시다운 (Pushing Predicates)**은 옵티마이저의 '먼저 거르기' 전략입니다. 데이터를 몽땅 가져온 뒤에 거르는 것이 아니라, 거름망(조건)을 최대한 원천 데이터(Table)에 가깝게 배치하여 연산 대상을 줄이는 기술입니다.

### 💡 비유: 뷔페 접시에 음식 담기
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        조건절 푸시다운 비유                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 뷔페에서 '초밥 세트(View)'를 가져오려 함. 그런데 나는 '오이'를 못 먹음.     │
│                                                                             │
│  1. 푸시다운 미적용 (비효율):                                                 │
│     - 일단 만들어진 초밥 100세트를 식탁으로 다 가져옴. (조인 수행)                │
│     - 그중 오이가 든 걸 하나씩 골라냄. (사후 필터링)                             │
│     - 힘만 들고 버리는 음식이 너무 많음. (I/O 낭비)                              │
│                                                                             │
│  2. 푸시다운 적용 (최적화):                                                   │
│     - 주방장에게 미리 말함: "초밥 세트 만드실 때 오이는 빼고(Push) 주세요!"        │
│     - 주방에서 이미 정제된 5세트만 식탁으로 옴.                                  │
│                                                                             │
│  → 필요한 조건(오이 제외)을 **'최대한 안쪽(주방)'**으로 밀어 넣어 일을 줄이는        │
│     똑똑한 주문 방식입니다!                                                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **뷰의 블랙박스화**: 뷰 내부에 `GROUP BY`, `ROWNUM`, `UNION` 등이 있으면 엔진이 뷰의 괄호를 풀지(Merge) 못해 성능이 급락하는 문제 발생.
- **분산 DB 환경**: 원격지 DB에서 데이터를 다 가져온 뒤 거르는 것은 네트워크 낭비이므로, 원격지에서 미리 걸러서 결과만 보내줘야 함.

- **📢 섹션 요약 비유**: 조건절 푸시다운은 '필터지를 깔대기의 입구(원천)'에 설치하여 맑은 물만 흐르게 하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 조건절 푸시다운의 2대 유형

| 유형 | 상세 설명 | 비유 |
|:---:|:---|:---|
| **상수 푸시다운** | `WHERE id = 10` 같은 고정 값을 뷰 내부로 전달. | "10번 손님 것만 준비해!" |
| **조인 조건 푸시다운** | 드라이빙 테이블의 키 값을 뷰 내부로 실시간 전달. | "한 명 올 때마다 그 사람 거 준비해!" |

### 2. 푸시다운 작동 프로세스 도해 (ASCII)

뷰 내부에 조건이 주입되는 논리적 흐름을 시각화한다.

```text
    [ User Query ]
    SELECT * FROM (SELECT * FROM Sales) V  <--- (View V)
    WHERE  V.Order_Date = '20260316';
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Optimizer : Predicate Pushdown Check ]                       │
    │  - "V는 머징이 안 되네? 그럼 날짜 조건을 안으로 던져볼까?"        │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (Pushing...)
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Transformed Internal View ]                                  │
    │  SELECT * FROM Sales **WHERE Order_Date = '20260316'**          │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (Execution)
    [ Real Scan ] : **Sales 테이블의 날짜 인덱스를 즉시 사용 가능!**
```

### 3. 심층 동작 원리: 뷰 머징(View Merging)과의 관계
- **차이**: 뷰 머징은 뷰의 껍데기(괄호) 자체를 없애는 것이고, 푸시다운은 껍데기는 유지하되 알맹이(SQL)에 조건만 쏙 집어넣는 것임.
- **우선순위**: 옵티마이저는 보통 뷰 머징을 먼저 시도하고, 머징이 불가능할 때 푸시다운을 최후의 수단으로 선택함.

### 4. 핵심 기술: PUSH_PRED 힌트
- 옵티마이저가 비용 계산 실수로 푸시다운을 포기할 때, 개발자가 `/*+ PUSH_PRED(view_alias) */` 힌트를 사용하여 강제로 밀어 넣는 기술.

- **📢 섹션 요약 비유**: 뷰 머징이 '벽을 허무는 리모델링'이라면, 푸시다운은 '벽에 구멍을 내서 물건(조건)을 건네주는 것'입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 조건절 위치에 따른 성능 차이 (Before vs After)

| 비교 항목 | 사후 필터링 (Filter) | 조건절 푸시다운 (Push) |
|:---:|:---|:---|
| **수행 시점** | 조인/집계 연산 완료 후 | **조인/집계 연산 시작 전** |
| **인덱스 활용** | 불가능 (임시 테이블 대상) | **가능 (원천 테이블 인덱스)** |
| **데이터량** | 100만 건 (전체) | **100건 (조건 일치분)** |
| **비유** | 수확 후 썩은 거 골라내기 | **심을 때부터 좋은 씨앗만 심기** |

### 2. 과목 융합 관점: 컴파일러 및 분산 시스템
- **컴파일러 융합**: 푸시다운은 추상 구문 트리(AST)에서 조건 노드를 리프 노드로 하강시키는 **노드 최적화(Node Lowering)** 기술의 일종이다.
- **분산 시스템 융합**: MSA 환경에서 API 호출 시 검색 조건을 파라미터로 넘기는 것은 **'API 레벨의 푸시다운'**이며, 이는 네트워크 오버헤드를 줄이는 가장 기초적인 분산 아키텍처 전략이다.

- **📢 섹션 요약 비유**: 푸시다운은 '전화로 미리 주문(사전 필터링)'하는 것과 같아, 매장에 가서 기다리는(사후 필터링) 시간을 줄여줍니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 복잡한 결산 뷰의 조회 속도 개선
- **상황**: [매출-세금-수수료]를 합산하는 거대 뷰를 '특정 매장' 조건으로 조회하는데 1분이 걸림.
- **판단**: 뷰 내부에 `GROUP BY`가 있어 뷰 머징이 실패했고, 전 세계 모든 매장의 데이터를 다 합산한 뒤 마지막에 특정 매장만 필터링 중.
- **결단**: **PUSH_PRED 힌트 적용**. 매장 ID 조건을 집계 연산 전인 뷰 내부의 `Sales` 테이블 스캔 단계로 밀어 넣음. 1분 걸리던 결산이 0.1초 만에 특정 매장 데이터만 읽고 종료됨.

### 2. 안티패턴: 푸시다운을 저해하는 뷰 내부의 ROWNUM 사용
- **증상**: 뷰 정의 시 정렬을 위해 무심코 `WHERE ROWNUM <= 100` 등을 포함.
- **결과**: `ROWNUM`은 데이터의 순번을 고정시키므로 바깥쪽의 조건이 안으로 들어오면 순서가 바뀔 위험이 있어 옵티마이저가 푸시다운을 원천 차단함.
- **조치**: **뷰 내부의 가상 컬럼이나 윈도우 함수 사용 주의**. 계층 구조나 정렬이 꼭 필요하다면 뷰 밖에서 조건을 처리하거나, 분석용 인라인 뷰를 최소화할 것.

### 📢 기술사적 결언
> "조건절 푸시다운은 데이터베이스의 **'선견지명'**이다. 나중에 할 일을 미리 당겨서 처리하기 때문이다. 기술사는 쿼리 실행 계획에서 **'VIEW PUSHED PREDICATE'** 오퍼레이션을 찾아내는 예리한 눈을 가져야 한다. 단순히 쿼리를 잘 짜는 것을 넘어, 옵티마이저가 뷰의 장벽에 막혀 헤매지 않도록 **논리적 통로(힌트, 구조 개선)**를 열어주는 '성능 아키텍트'의 역할을 수행해야 한다."

- **📢 섹션 요약 비유**: 푸시다운을 아는 것은 '숙제를 몰아서 하지 않고 틈틈이 미리 해두는 우등생'의 습관과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (푸시다운 활성화 시)

| 지표 | 활용 전 | 활용 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **불필요한 Block I/O** | 100,000 | **100 이내** | 리소스 99.9% 절감 |
| **메모리(PGA) 점유** | 수십 MB (임시 집합) | **수 KB (정제 데이터)** | 동시성 수용량 확대 |
| **시스템 체감 성능** | 거부감 (느림) | **쾌적함 (즉시)** | 비즈니스 민첩성 강화 |

### 2. 미래 전망
- **Automatic Predicate Propagation**: AI 옵티마이저가 뷰뿐만 아니라 복잡한 프로시저나 외부 연동 시스템(API)까지 조건을 자동으로 전파하여 최적화하는 **전방위적 푸시다운** 진화.
- **Predicate Pushdown to Hardware**: DB 소프트웨어가 아닌 스토리지 컨트롤러(FPGA) 레벨에서 조건절을 인지하고 데이터를 걸러서 올려보내는 **Smart SSD** 연계 기술 보편화.

- **📢 섹션 요약 비유**: 미래의 푸시다운은 우리가 질문을 하기도 전에, 우리의 마음(조건)을 읽어 데이터 소스들이 알아서 몸을 정제하는 '이심전심 최적화' 시스템이 될 것입니다.
