# 165. 인덱스 스캔 방식 - Unique, Range, Full, Fast Full Scan

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인덱스의 B-Tree 구조를 활용하여 데이터를 검색하는 다양한 물리적 방법론이며, 조건의 정밀도와 인덱스 구성에 따라 옵티마이저가 최적의 스캔 방식을 선택한다.
> 2. **유형별 특성**: 정확한 1건을 찾는 **Unique Scan**, 범위를 탐색하는 **Range Scan**, 정렬 순서를 활용하는 **Full Scan**, 그리고 멀티 블록 I/O로 인덱스 전체를 훑는 **Fast Full Scan** 등으로 분화된다.
> 3. **가치**: 테이블 전체를 읽는 무거운 연산을 최소한의 리프 블록 탐색으로 대체하여 응답 속도를 획기적으로 개선하며, 효율적인 스캔 방식의 선택이 SQL 튜닝의 성패를 좌우한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**인덱스 스캔 (Index Scan)**은 인덱스라는 사전의 목차를 보고 필요한 정보가 있는 페이지(Block)만 쏙쏙 골라 읽는 행위입니다. 인덱스 구조(B-Tree)의 상단(Root)부터 하단(Leaf)까지 어떻게 타고 내려가느냐에 따라 이름이 달라집니다.

### 2. 💡 비유: 전공 서적에서 정보 찾기
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        인덱스 스캔 방식 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. Unique Scan (페이지 번호로 찾기):                                          │
│     - "77페이지를 펴라." -> 딱 한 번에 정확한 지점을 찾음. (제일 빠름)             │
│                                                                             │
│  2. Range Scan (키워드 범위로 찾기):                                          │
│     - "사과~오렌지 단어가 있는 곳을 다 찾아라." -> 관련 구역을 쭉 훑음.            │
│                                                                             │
│  3. Full Scan (목차 전체 정독):                                               │
│     - "목차의 처음부터 끝까지 다 읽으면서 ㄱㄴㄷ 순서대로 정리해라."               │
│                                                                             │
│  4. Fast Full Scan (목차 찢어서 한꺼번에 보기):                                 │
│     - "순서 상관없으니 목차 페이지를 몽땅 뜯어와서 미친 듯이 읽어라!" (병렬)        │
│                                                                             │
│  → 내가 가진 정보(조건)가 얼마나 정확한지에 따라 **'찾는 기술'**이 달라집니다!      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **대량 데이터 검색의 비효율**: 데이터가 수억 건일 때, 무조건 처음부터 끝까지 읽는(Full Table Scan) 것은 불가능함.
- **다양한 검색 조건 대응**: 단건 조회, 범위 조회, 정렬 조회 등 사용자의 다양한 요구에 최적화된 물리적 경로 제공 필요.

- **📢 섹션 요약 비유**: 인덱스 스캔은 '데이터 바다에서 보물을 찾는 잠수부의 기술'과 같아, 상황에 맞는 장비(스캔 방식)를 골라야 합니다.

## II. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 4대 주요 인덱스 스캔 방식 상세 비교

| 방식 | 발동 조건 | 작동 원리 | 비유 |
|:---:|:---|:---|:---|
| **Unique Scan** | PK 또는 Unique 인덱스에 **`=`** 조건 사용. | Root $\to$ Branch $\to$ Leaf 로 수직 탐색 후 종료. | 핀셋으로 집기 |
| **Range Scan** | 인덱스 선두 컬럼이 **부등호, LIKE, BETWEEN** 조건. | 수직 탐색 후 필요한 만큼 Leaf 노드를 수평 탐색. | 줄줄이 사탕 |
| **Full Scan** | 인덱스는 있는데 조건이 없거나 부적절할 때. | Leaf 노드의 처음부터 끝까지 수평 탐색 (정렬 활용). | 책장 넘기기 |
| **Fast Full Scan** | 인덱스 컬럼만 조회할 때 (Multi-block I/O). | 인덱스 블록을 물리적 순서대로 한꺼번에 읽음. | 속독하기 |

### 2. 인덱스 스캔 경로 도해 (ASCII)

수직 탐색(Vertical)과 수평 탐색(Horizontal)의 차이를 시각화한다.

```text
          [ ROOT BLOCK ]
               │
      ┌────────┴────────┐
 [ BRANCH ]        [ BRANCH ]
      │                 │
 ┌────┴────┐       ┌────┴────┐
[L1] ──▶ [L2] ──▶ [L3] ──▶ [L4]  (LEAF BLOCKS: Linked List)
  ▲        │         │        ▲
  │        └─(Range)─┘        │
  └──────────(Full)───────────┘
  
  * Unique: ROOT -> BRANCH -> L2 (딱 하나 찾고 끝!)
  * Range : ROOT -> BRANCH -> L2 (시작점) -> L3 (끝점까지 옆으로 이동)
  * Full  : L1 -> L2 -> L3 -> L4 (순서대로 몽땅)
```

### 3. 심층 동작 원리: 인덱스 스킵 스캔 (Index Skip Scan)
- **개념**: 결합 인덱스에서 선두 컬럼 조건이 없더라도, 선두 컬럼의 종류(NDV)가 적으면 이를 건너뛰며(Skip) 하위 컬럼 인덱스를 활용하는 지능형 기술.
- **효과**: 인덱스를 새로 만들지 않고도 기존 인덱스를 재활용하는 유연성 제공.

### 4. 핵심 기술: 인덱스 역방향 스캔 (Index Range Scan Descending)
- `ORDER BY Desc` 요청 시, Leaf 노드의 이중 연결 리스트(Double Linked List)를 거꾸로 타고 올라가는 기술. 별도의 정렬(Sort) 연산 없이 결과 반환 가능.

- **📢 섹션 요약 비유**: Skip Scan은 '아는 길은 건너뛰고 모르는 길(필요한 데이터)만 골라 가는 징검다리 건너기'와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 인덱스 Full Scan vs Fast Full Scan

| 비교 항목 | Index Full Scan | Index Fast Full Scan |
|:---:|:---|:---|
| **I/O 방식** | Single Block I/O | **Multi-block I/O (병렬)** |
| **결과 정렬** | **정렬 보장 (Index 순서)** | 정렬 보장 안 됨 (물리 순서) |
| **속도** | 보통 | **매우 빠름** |
| **제약** | 조건 상관없음 | **SELECT 절에 인덱스 컬럼만** 있어야 함 |

### 2. 과목 융합 관점: 자료구조 및 알고리즘
- **자료구조 융합**: 인덱스 스캔은 **B+Tree** 자료구조의 탐색 알고리즘을 따르며, Leaf 노드가 **Linked List**로 연결되어 있어 범위 검색($O(\log N) + K$)에 최적화되어 있다.
- **알고리즘 융합**: Fast Full Scan은 파일 시스템의 **순차적 읽기(Sequential Read)** 알고리즘을 차용한 것이며, 처리량을 높이기 위해 OS 레벨의 **Read-ahead** 메커니즘을 적극 활용한다.

- **📢 섹션 요약 비유**: Full Scan이 '계단을 하나씩 오르는 것'이라면, Fast Full Scan은 '엘리베이터로 층마다 짐을 한꺼번에 싣는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 인덱스가 있는데 왜 안 탈까? (Range Scan 실패)
- **상황**: 사원명(`Name`) 컬럼에 인덱스가 있는데 `WHERE Name LIKE '%KIM'` 쿼리가 너무 느림.
- **판단**: 실행 계획 확인 시 `FULL TABLE SCAN` 중. 패턴의 시작이 와일드카드(`%`)라 인덱스의 시작점을 찾을 수 없음.
- **결단**: **쿼리 변경 또는 인덱스 재설계**. `WHERE Name LIKE 'KIM%'`으로 전방 일치로 바꾸어 `Index Range Scan`을 유도하거나, 역방향 인덱스(Reverse Index) 등을 검토하여 검색 효율성 확보.

### 2. 안티패턴: 인덱스 컬럼 가공으로 인한 Scan 거부
- **증상**: `WHERE TO_CHAR(Reg_Date, 'YYYY') = '2026'`.
- **결과**: `Reg_Date` 인덱스가 있어도 엔진이 값을 다 변환해봐야 하므로 `Index Range Scan` 불가.
- **조치**: **컬럼은 원형 그대로 유지**. `WHERE Reg_Date >= TO_DATE('20260101') AND ...`. 인덱스의 수직 탐색이 가능하도록 조건을 표준화할 것.

### 📢 기술사적 결언
> "인덱스 스캔 방식은 데이터베이스의 **'물리적 지능'**을 보여주는 지표다. 데이터의 위치를 얼마나 영리하게 찾아내느냐가 시스템의 생사(Performance)를 결정하기 때문이다. 기술사는 단순한 문법적 매칭을 넘어, **데이터의 카디널리티와 클러스터링 팩터**가 각 스캔 방식의 효율에 미치는 파급력을 계산해야 한다. 가장 빠른 Unique Scan부터 가장 강력한 Fast Full Scan까지, 옵티마이저의 도구 상자를 자유자재로 다루는 능력이 **최고 수준의 튜닝 전문가**를 만든다."

- **📢 섹션 요약 비유**: 스캔 방식을 오해하는 것은 '서울 가는데 부산행 기차를 타는 것'과 같아, 노력(I/O)은 많이 해도 목적지(결과)에 늦게 도착하게 됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (최적 스캔 방식 선택 시)

| 지표 | 활용 전 | 활용 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **블록 I/O 횟수** | 10,000 (Full Scan) | **3~5 (Unique Scan)** | I/O 부하 99% 이상 감소 |
| **CPU 사용량** | 90% (연산 과부하) | **10% 이하** | 시스템 안정성 확보 |
| **응답 시간** | 수 초 ~ 수 분 | **밀리초 (ms)** | 사용자 체감 성능 혁신 |

### 2. 미래 전망
- **AI-Managed Index Scan Selection**: 과거 실행 이력과 실시간 시스템 부하를 학습한 AI가, 특정 시점에는 Full Scan이 오히려 낫다고 판단하여 실행 계획을 동적으로 바꾸는 **자율형 스캔 최적화**.
- **Storage-level Scan Offloading**: DB 엔진이 아닌 저장 장치(SSD/NVMe) 내부에서 직접 인덱스 스캔과 필터링을 수행하여 CPU로의 데이터 전송 자체를 줄이는 **Near-Data Processing** 기술 확산.

- **📢 섹션 요약 비유**: 미래의 인덱스 스캔은 우리가 길을 묻지 않아도, 데이터들이 스스로 정렬하여 우리의 손바닥 위로 뛰어 올라오는 '자율 주행 데이터' 시스템이 될 것입니다.
