# 166. 전체 테이블 스캔 vs 인덱스 스캔 - 선택 기준과 손익분기점

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 엔진이 질의 결과를 얻기 위해 테이블의 모든 데이터 블록을 순차적으로 읽을지(**Full Scan**), 아니면 인덱스를 통해 필요한 행의 주소(ROWID)만을 추출하여 접근할지(**Index Scan**) 결정하는 물리적 선택의 문제다.
> 2. **성능 결정 요인**: 전체 데이터를 뭉텅이로 읽어오는 **Multi-block I/O(Full)**와 하나씩 콕 찝어 읽는 **Single-block I/O(Index)**의 효율성 대결이며, 데이터의 양과 **선택도(Selectivity)**에 따라 어느 쪽이 유리한지 결정되는 지점을 '손익분기점'이라 한다.
> 3. **가치**: 인덱스가 항상 빠르다는 고정관념을 깨고, 대량 데이터 집계 시에는 오히려 전체 스캔이 효율적일 수 있음을 이해함으로써 리소스 낭비를 방지하고 시스템 전반의 처리량(Throughput)을 최적화한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **전체 테이블 스캔 (Full Table Scan)**: 테이블의 HWM(High Water Mark) 아래에 있는 모든 데이터 블록을 읽는 방식입니다.
- **인덱스 스캔 (Index Scan)**: 인덱스를 검색하여 데이터의 위치(ROWID)를 먼저 찾은 뒤, 해당 블록만 테이블에서 가져오는 방식입니다.

### 2. 💡 비유: 전교생 중에 '김철수' 찾기
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        스캔 방식 선택 비유                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 운동장에 전교생 1,000명이 모여 있음.                                     │
│                                                                             │
│  1. 인덱스 스캔 (출석부 활용):                                                │
│     - 출석부(Index)를 보고 "김철수는 3분대 5번째에 있다"라고 주소(ROWID) 확인.      │
│     - 정확히 그 지점으로 달려가서 철수를 데려옴.                                  │
│     - **한두 명 찾을 때는 최고**지만, 500명을 찾아야 하면 출석부 보느라 더 느려짐.    │
│                                                                             │
│  2. 전체 스캔 (줄대로 훑기):                                                  │
│     - 출석부 안 보고, 맨 앞줄부터 끝줄까지 확 훑으면서 철수들을 골라냄.             │
│     - 한꺼번에 여러 명을 훑을 수 있어(Multi-block), **많은 인원 찾을 때 유리**.     │
│                                                                             │
│  → **'찾아야 할 데이터가 얼마나 많은가'**에 따라 출석부를 볼지 말지 결정합니다!      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **I/O 방식의 차이**: 디스크에서 데이터를 읽어올 때 한 번에 한 블록을 읽는 것보다 여러 블록을 한꺼번에 읽는 것이 하드웨어적으로 유리한 구간 존재.
- **랜덤 액세스(Random I/O) 부하**: 인덱스를 타고 테이블을 뒤지는 것은 디스크 헤드가 여기저기 튀어야 하므로, 데이터가 많아지면 순차 읽기보다 훨씬 느려짐.

- **📢 섹션 요약 비유**: 전체 스캔은 '저인망 그물'로 몽땅 훑는 것이고, 인덱스 스캔은 '작살'로 정확히 쏘는 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 스캔 방식별 물리적 I/O 비교

| 비교 항목 | 전체 테이블 스캔 (Full) | 인덱스 스캔 (Index) |
|:---:|:---|:---|
| **I/O 단위** | **Multi-block I/O** (한 번에 8~128개) | **Single-block I/O** (한 번에 1개) |
| **액세스 방식** | 순차적 접근 (Sequential) | 임의 접근 (Random / ROWID) |
| **캐시 활용** | 재사용성 낮음 (Keep Buffer 제외) | **재사용성 높음 (Cache Hit 유리)** |
| **CPU 부하** | 낮음 (단순 읽기) | 높음 (인덱스 구조 탐색 연산) |

### 2. 스캔 방식의 손익분기점 도해 (ASCII)

데이터 추출 비율에 따른 비용 변화를 시각화한다.

```text
    Cost
      ^
      │          / (Index Scan Cost : 기하급수적 상승)
      │         /
      │        /
      │       /
      │      /
      │     * <--- **[ 손익분기점 ]** (보통 전체의 5~15%)
      │    / __________________________ (Full Scan Cost : 완만하거나 일정)
      │   /
      └──────────────────────────────────▶ % of Data (추출 비율)
    
    * 소량 추출: Index Scan 이 압도적으로 저렴.
    * 대량 추출: Full Scan 이 Multi-block I/O 덕분에 더 저렴.
```

### 3. 심층 동작 원리: 클러스터링 팩터 (Clustering Factor)
- **개념**: 인덱스의 정렬 순서와 실제 테이블의 데이터 저장 순서가 얼마나 일치하는지를 나타내는 수치.
- **영향**: 일치도가 높으면(Good CF) 인덱스 스캔의 손익분기점이 올라가며(더 많이 읽어도 유리), 일치도가 낮으면 조금만 읽어도 Full Scan 이 나음.

### 4. 핵심 기술: 병렬 쿼리 (Parallel Query)
- 전체 테이블 스캔 시 여러 개의 프로세스(Slave)를 띄워 테이블을 조각내어 동시에 읽는 기술. 대용량 DW 환경에서 전체 스캔을 선호하게 만드는 결정적 요인.

- **📢 섹션 요약 비유**: 손익분기점은 '택시(인덱스)와 버스(전체 스캔)'의 관계와 같아, 사람이 많아지면 버스가 훨씬 경제적입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. OLTP vs OLAP 환경에서의 선택 전략

| 환경 유형 | 주요 쿼리 성격 | 권장 스캔 방식 | 이유 |
|:---:|:---|:---:|:---|
| **OLTP (운영)** | 특정 고객 조회, 소량 변경 | **Index Scan** | 빠른 응답 속도, 동시성 확보 |
| **OLAP (분석)** | 월간 매출 집계, 전수 통계 | **Full Scan** | 대량 데이터 처리량(Throughput) 중시 |

### 2. 과목 융합 관점: 하드웨어 및 운영체제
- **하드웨어 융합**: SSD와 NVMe의 보급으로 랜덤 I/O 비용이 낮아지면서 인덱스 스캔의 손익분기점이 과거 HDD 시절보다 높아지는(인덱스가 더 유리해지는) 경향이 있다.
- **운영체제(OS) 융합**: 전체 스캔의 Multi-block I/O는 OS의 **파일 시스템 캐시(Page Cache)**와 시너지를 내어, 한 번 읽은 인접 블록을 메모리에서 즉시 반환하는 효과를 극대화한다.

- **📢 섹션 요약 비유**: 인덱스는 '전화번호부'이고 전체 스캔은 '마을 전수조사'입니다. 한 명 찾을 땐 전화번호부가 최고지만, 마을 전체의 평균 수입을 잴 땐 전수조사가 빠릅니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 인덱스가 있는데도 Full Scan을 타는 쿼리 분석
- **상황**: 조회 조건 컬럼에 인덱스가 분명히 있는데, 옵티마이저가 자꾸 `FULL TABLE SCAN`을 선택함.
- **판단**: 추출하려는 데이터가 전체 테이블의 30%를 넘거나, 인덱스 통계 정보가 실제보다 데이터가 더 많다고 알려주고 있음.
- **결단**: **실제 데이터 비율 확인 및 통계 갱신**. 만약 대량 데이터를 가져오는 게 맞다면 옵티마이저의 판단이 옳은 것이므로 강제로 인덱스를 태우지(Hint) 말 것. 오히려 병렬 쿼리(`PARALLEL`) 힌트를 주어 전체 스캔 속도를 높이는 것이 정답.

### 2. 안티패턴: "인덱스는 무조건 선(善)"이라는 맹신
- **증상**: 성능을 올리겠다고 모든 컬럼에 인덱스를 걸고, 모든 쿼리에 인덱스 힌트를 삽입.
- **결과**: 데이터 입력(Insert) 속도가 처참하게 떨어지고, 대량 조회 시 인덱스를 타느라 Single-block I/O 늪에 빠져 시스템 전체 응답 지연.
- **조치**: **데이터의 분포도(Selectivity)를 분석**. 15% 이상의 데이터를 읽는 쿼리는 인덱스를 제거하거나 전체 스캔을 유도하는 '유연한 튜닝' 실시.

### 📢 기술사적 결언
> "스캔 방식의 선택은 **'규모의 경제'**를 이해하는 것이다. 소량의 정밀함(Index)과 대량의 효율성(Full) 사이에서 최적의 균형점을 찾는 행위이기 때문이다. 기술사는 인덱스 유무에만 집착하지 말고, **테이블의 HWM 위치와 데이터의 집중도**를 파악하여 시스템의 병목을 진단해야 한다. 때로는 인덱스를 과감히 버리고 **병렬 전체 스캔과 파티셔닝**으로 승부하는 것이 진정한 '대용량 데이터 아키텍트'의 자세임을 명심해야 한다."

- **📢 섹션 요약 비유**: 바늘로 수천 개의 구멍을 뚫는 것보다, 송곳으로 한 번에 뚫는 게 빠를 때가 있음을 아는 것이 튜닝의 지혜입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (최적 스캔 방식 선택 시)

| 지표 | 활용 미흡 시 | 최적 활용 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **배치 작업 시간** | 수 시간 (Index 오남용) | **수 분 (Full Scan 전환)** | 처리량 10배 향상 |
| **온라인 응답 속도** | 불안정 (Full Scan 유입) | **밀리초 (Index 유도)** | 서비스 안정성 확보 |
| **디스크 수명/부하** | 랜덤 액세스로 과부하 | **순차 I/O로 안정화** | HW 운영 효율 극대화 |

### 2. 미래 전망
- **Columnar Storage Scan**: 행 단위가 아닌 열 단위로 데이터를 읽는 클라우드 DB(BigQuery, Snowflake)에서는 '전체 스캔'의 정의가 특정 컬럼만 읽는 방식으로 바뀌어 효율성이 극대화됨.
- **In-memory Vector Scan**: 모든 데이터를 메모리에 올리고 인덱스 없이도 CPU의 벡터 연산(SIMD)으로 순식간에 전체를 훑어버리는 **No-Index 아키텍처** 확산.

- **📢 섹션 요약 비유**: 미래의 스캔은 우리가 고민하지 않아도, 데이터의 양에 따라 '작살'이 될지 '그물'이 될지 빛의 속도로 변신하는 지능형 채집 시스템이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **인덱스 스캔**은 사탕 바구니에서 **"딱 내가 좋아하는 맛 한 개"**만 쏙 집어내는 거예요.
2. **전체 스캔**은 바구니를 통째로 쏟아서 **"좋아하는 사탕이 어디 있나"** 한꺼번에 훑어보는 거고요.
3. 사탕이 한두 개만 필요할 땐 손가락(인덱스)이 빠르지만, **한 주먹 넘게 필요할 땐** 그냥 쏟는 게(전체) 훨씬 빠르답니다!
