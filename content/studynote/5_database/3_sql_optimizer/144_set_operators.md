# 144. 집합 연산자 - UNION, INTERSECT, EXCEPT

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 두 개 이상의 질의 결과(Result Set)를 하나의 결과 집합으로 결합하기 위한 연산자로, 수학적 집합론의 **합집합(Union), 교집합(Intersect), 차집합(Except/Minus)** 개념을 SQL 문법으로 구현한 것이다.
> 2. **제약 및 특성**: 결합하려는 모든 질의는 **컬럼의 수와 데이터 타입이 상호 호환(Compatible)**되어야 하며, `UNION`, `INTERSECT`, `EXCEPT`는 결과에서 중복 행을 자동 제거하는 정렬 연산이 수반된다.
> 3. **가치**: 서로 다른 테이블에 흩어진 유사 성격의 데이터를 통합하여 일관된 뷰를 생성하며, 특히 `UNION ALL`을 통해 중복 제거 오버헤드 없이 대량의 데이터를 고속으로 병합할 수 있다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**집합 연산자 (Set Operators)**는 여러 개의 SELECT 문 결과를 하나로 합치는 도구입니다. 조인(Join)이 테이블을 옆으로(수평) 붙인다면, 집합 연산자는 테이블을 위아래로(수직) 쌓는 개념입니다.

### 2. 💡 비유: 전교생 명단 합치기
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        집합 연산자 개념 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] '1반 학생 명부'와 '2반 학생 명부'가 있음.                                │
│                                                                             │
│  1. UNION (합집합): 두 반 학생을 모두 모아 한 줄로 세움. 중복된 애는 한 번만!      │
│  2. UNION ALL: 중복이든 아니든 일단 다 모아서 세움. (제일 빠름)                  │
│  3. INTERSECT (교집합): 1반에도 있고 2반에도 있는 (전학생 등) 학생만 골라냄.      │
│  4. EXCEPT (차집합): 1반 학생 중 2반에는 없는 순수 1반 학생만 골라냄.             │
│                                                                             │
│  → 데이터를 **'어떤 바구니에 담을지'** 결정하는 집합 놀이와 같습니다!              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 분산의 수직적 통합**: 지점별 매출 테이블, 월별 로그 테이블 등 구조는 같으나 데이터가 나뉜 경우를 통합 조회하기 위함.
- **복합 필터링 구현**: "A 조건 만족 또는 B 조건 만족"하는 데이터를 각각의 쿼리로 짠 뒤 하나로 합치는 것이 논리적으로 명확할 때 사용.

- **📢 섹션 요약 비유**: 집합 연산자는 '서로 다른 통에 든 구슬들을 규칙에 따라 하나의 통에 쏟아붓는 것'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 주요 집합 연산자 상세 특징

| 연산자 | 수학적 의미 | 중복 제거 여부 | 정렬 여부 | 비유 |
|:---:|:---|:---: |:---:|:---|
| **UNION** | 합집합 ($\cup$) | **Yes** | Yes | 정제된 모임 |
| **UNION ALL** | 합집합 (Multi-set) | **No** | **No** | 무조건 합치기 |
| **INTERSECT** | 교집합 ($\cap$) | **Yes** | Yes | 공통분모 찾기 |
| **EXCEPT (MINUS)** | 차집합 ($-$) | **Yes** | Yes | 소외된 쪽 찾기 |

### 2. 집합 연산 작동 구조 도해 (ASCII)

데이터가 수직으로 병합되는 논리적 과정을 시각화한다.

```text
    [ Query 1 Result ]          [ Query 2 Result ]
    ┌──────────┐                ┌──────────┐
    │    A     │                │    B     │
    │    B     │                │    C     │
    └──────────┘                └──────────┘
           │                           │
           └───────────┬───────────────┘
                       ▼
    [ Final Result Set ]
    (1) UNION     : { A, B, C } (B 중복 제거)
    (2) UNION ALL : { A, B, B, C } (그대로)
    (3) INTERSECT : { B } (겹치는 것만)
    (4) EXCEPT    : { A } (1에는 있지만 2에는 없는 것)
```

### 3. 심층 동작 원리: 컬럼 호환성 (Column Compatibility)
- **제약**: SELECT 절의 컬럼 개수가 동일해야 하며, 대응하는 컬럼끼리 데이터 타입이 일치하거나 암시적 형변환이 가능해야 함.
- **정렬 부하**: `UNION`, `INTERSECT`, `EXCEPT`는 중복을 찾기 위해 내부적으로 **Sort(정렬)** 또는 **Hash(해시)** 연산을 수행하므로, CPU와 메모리를 소모함.

### 4. 핵심 기술: UNION ALL의 성능상 이점
- 중복 체크 과정을 생략하므로, 데이터가 중복될 리 없거나 중복되어도 상관없는 경우 `UNION`보다 압도적으로 빠름.
- **분산 DB**에서 여러 노드의 결과를 합칠 때 가장 선호되는 연산자.

- **📢 섹션 요약 비유**: UNION ALL은 '그냥 쏟아붓기'이고, UNION은 '쏟아부으면서 같은 게 있는지 눈으로 일일이 확인하기'입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 조인(Join) vs 집합 연산(Set)

| 비교 항목 | 조인 (Join) | 집합 연산 (Set) |
|:---:|:---|:---|
| **결합 방향** | **수평적 (Horizontal)** | **수직적 (Vertical)** |
| **컬럼 수** | 증가함 (A+B) | **불변 (A와 동일)** |
| **조건** | 매칭 조건(ON) 필수 | 조건 없음 (구조만 맞으면 됨) |
| **비유** | 두 사람의 정보를 한 줄에 적기 | 두 사람의 정보를 다른 줄에 적기 |

### 2. 과목 융합 관점: 이산수학 및 운영체제
- **이산수학 융합**: 집합 연산자는 집합론의 **기본 연산 법칙(교환, 결합, 분배 법칙)**을 충실히 따르며, 이를 통해 복잡한 질의를 논리적으로 단순화할 수 있다.
- **운영체제(OS) 융합**: 대량의 집합 연산 시 메모리가 부족하면 디스크의 **Temp Segment**를 활용한 외부 정렬(External Sort)이 발생하며, 이는 OS의 I/O 스케줄링 성능에 의존하게 된다.

- **📢 섹션 요약 비유**: 조인이 '결혼해서 한 등본에 나오는 것'이라면, 집합 연산은 '같은 아파트 단지에 입주해서 이웃이 되는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 통합 매출 보고서 쿼리
- **상황**: '오프라인매출' 테이블과 '온라인매출' 테이블이 분리되어 있음.
- **판단**: 두 테이블의 컬럼 구조가 거의 비슷함.
- **결단**: **UNION ALL 사용**. `SELECT 'OFF' as Type, Amt FROM Offline UNION ALL SELECT 'ON', Amt FROM Online`. 오프라인과 온라인 매출이 겹칠 리 없으므로, 정렬 부하가 없는 `UNION ALL`을 써서 시스템 자원을 아끼고 응답 속도 확보.

### 2. 안티패턴: UNION을 쓰면서 중복 체크를 또 함
- **증상**: `SELECT ... WHERE ID NOT IN (SELECT ID FROM ...)` 로직을 짠 뒤, 결과에 다시 `UNION`을 사용.
- **결과**: 이미 논리적으로 중복을 걸렀음에도 `UNION`이 다시 정렬 연산을 수행하여 CPU 낭비.
- **조치**: 쿼리 논리상 중복이 발생하지 않는다면 무조건 **`UNION ALL`**을 기본으로 사용할 것.

### 📢 기술사적 결언
> "집합 연산자는 데이터베이스의 **'조직력'**을 강화한다. 파편화된 데이터에 하나의 이름표를 달아 통합된 가치를 부여하기 때문이다. 기술사는 단순히 결과를 합치는 것에 그치지 않고, **중복 제거(Sort/Hash)**가 물리적 리소스에 미치는 파급력을 계산해야 한다. 특히 현대의 **빅데이터 분산 처리** 환경에서는 `UNION ALL`을 통한 병렬 처리가 전체 시스템 처리량(Throughput)의 핵심 변수가 됨을 명심해야 한다."

- **📢 섹션 요약 비유**: 무분별한 UNION 사용은 '다 아는 내용을 확인하느라 책장을 계속 넘기는 수험생'과 같아, 진도(성능)를 나가지 못하게 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (집합 연산 최적 활용 시)

| 지표 | 활용 전 | 활용 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **쿼리 수행 속도** | UNION 사용 시 느림 | **UNION ALL로 3배 향상** | 자원 소모 최소화 |
| **데이터 통합 가시성** | 각기 조회 후 엑셀 합산 | **SQL로 즉시 통합** | 업무 자동화 및 정확도 향상 |
| **유지보수 용이성** | 복잡한 IF문 코드 | **명확한 집합 논리** | 코드 가독성 50% 개선 |

### 2. 미래 전망
- **Schema-less Set Operations**: 데이터 타입이 조금 달라도 AI가 의미를 파악하여 자동으로 형변환하고 합쳐주는 **지능형 집합 엔진** 발전.
- **Stream Set Operations**: 정지된 데이터뿐만 아니라 실시간으로 쏟아지는 데이터 스트림(Kafka 등) 사이에서 윈도우 기반의 집합 연산을 수행하는 기술이 표준화될 것임.

- **📢 섹션 요약 비유**: 미래의 집합 연산은 우리가 일일이 통을 붓지 않아도, 데이터들 스스로가 비슷한 친구를 찾아 뭉치고 흩어지는 '액체형 데이터 클러스터'가 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **집합 연산자**는 여러 상자에 든 장난감들을 **"하나의 커다란 상자에 쏟아붓는 것"**과 같아요.
2. 똑같은 게 있으면 하나만 남길지(UNION), 아니면 몽땅 다 넣을지(UNION ALL) **약속**만 하면 되죠.
3. 이 약속을 잘 지키면 흩어져 있던 장난감들을 **한눈에 다 볼 수 있어서** 정말 편하답니다!
