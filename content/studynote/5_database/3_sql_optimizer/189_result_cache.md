# 189. 결과 캐시 (Result Cache) - 쿼리 결과의 메모리 저장과 재활용

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 실행 계획을 캐싱하는 라이브러리 캐시와 달리, SQL 쿼리나 함수의 **최종 실행 결과(Result Set)**를 메모리에 직접 저장하여 동일 요청 시 연산 과정을 완전히 생략하고 즉시 반환하는 고속 조회 기술이다.
> 2. **무효화 메커니즘**: 캐시된 결과의 기반이 되는 원천 테이블에 단 하나의 행이라도 변경(DML)이 발생하면, 정합성을 위해 해당 캐시를 즉시 **무효화(Invalidation)** 처리하는 엄격한 동기화 원칙을 따른다.
> 3. **가치**: 읽기 비중이 압도적으로 높고 데이터 변경이 드문 기준 정보 조회나 복잡한 통계 연산의 응답 속도를 0에 가깝게 단축시키며, DB 서버의 CPU 및 블록 I/O 부하를 획기적으로 경감한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**결과 캐시 (Result Cache)**는 데이터베이스판 '정답지'입니다. 시험 문제를 풀 때마다 계산하는 게 아니라, "이 문제의 답은 42야"라고 답 자체를 메모리에 포스트잇으로 붙여두는 방식입니다.

### 💡 비유: 자주 시키는 '배달 음식'의 포장 완료 상태
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        결과 캐시 (Result Cache) 비유                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 손님들이 매일 똑같은 '비빔밥'을 수천 그릇 주문함.                        │
│                                                                             │
│  1. 일반 방식 (No Cache):                                                    │
│     - 주문 올 때마다 나물 볶고, 밥 짓고, 고추장 넣고 비빔. (연산 수행)             │
│     - 1,000명이 주문하면 똑같은 비빔질을 1,000번 반복함. (I/O & CPU 낭비)          │
│                                                                             │
│  2. 결과 캐시 방식:                                                          │
│     - 첫 주문 때 비빔밥을 하나 완벽하게 비벼서 그릇째 보관함. (Result Store)      │
│     - 다음 손님이 오면 요리할 필요 없이 보관함에서 그릇만 꺼내 줌! (Instant)       │
│     - 단, 재료(Source Table)가 바뀌면(DML) 보관된 비빔밥은 버려야 함! (Invalid)    │
│                                                                             │
│  → 계산 과정조차 아까운 **'반복되는 정답'**을 메모리에 모셔두는 최고급 기술입니다!   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **극한의 성능 요구**: 1ms(밀리초)조차 아까운 실시간 대시보드나 고빈도 코드 조회 업무 증가.
- **CPU 리소스의 효율적 사용**: 데이터 블록 I/O는 줄였어도, 수억 번의 필터링과 조인 연산 자체가 CPU에 부담을 주는 현상 해결 필요.

- **📢 섹션 요약 비유**: 결과 캐시는 '과정'이 아닌 '결과'를 사랑하는 데이터베이스의 지름길입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 결과 캐시의 두 가지 유형

| 유형 | 저장 위치 | 상세 특징 |
|:---:|:---|:---|
| **Server Result Cache** | **DB 서버 SGA** | 모든 세션이 공유. 복잡한 집계 결과 공유에 최적. |
| **Client Result Cache** | **WAS / 클라이언트 메모리** | 네트워크 전송 비용(Round-trip)까지 제로화. 특정 세션 전용. |

### 2. 결과 캐시 작동 및 무효화 도해 (ASCII)

데이터 변경 시 캐시가 파괴되는 정합성 유지 과정을 시각화한다.

```text
    [ SELECT /*+ RESULT_CACHE */ ... FROM Code_Table ]
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  1. [ Cache Lookup ] : 메모리에 결과가 있는가?                    │
    └──────┬───────────────────────┬──────────────────────────────────┘
           │ [ YES : Hit ]         │ [ NO : Miss ]
           ▼                       ▼
    [ Return Result ]       [ Execute Query ] ──▶ [ Save to SGA Cache ]
    (0.001 sec)             (1.500 sec)
           │                       │
    ┌──────┴───────────────────────┴──────────────────────────────────┐
    │  [ Event: UPDATE Code_Table ] ──▶ **[ Cache Invalidation ]**    │
    │  - "재료가 바뀌었으니, 보관된 결과물은 더 이상 진실이 아니다!"      │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: 종속성 추적 (Dependency Tracking)
- **원리**: 결과 캐시는 자기가 참조한 테이블 목록을 '종속성 리스트'로 관리함.
- **무효화**: 해당 리스트에 있는 테이블 중 하나라도 `COMMIT`이 발생하는 순간, 연관된 모든 캐시 엔트리를 `STALE` 상태로 변경하거나 즉시 삭제함.

### 4. 핵심 기술: 결과 캐시 래치 (Result Cache Latch)
- 캐시 결과물을 읽거나 쓸 때 발생하는 아주 짧은 메모리 잠금. 너무 많은 쿼리가 한꺼번에 캐시를 쓰려 하면 래치 경합이 발생할 수 있음.

- **📢 섹션 요약 비유**: 결과 캐시는 '유통기한이 엄격한 우유'와 같아서, 원천 데이터가 바뀌는 순간 바로 상해버려(무효화) 버려야 합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 라이브러리 캐시 vs 결과 캐시

| 비교 항목 | 라이브러리 캐시 (Library Cache) | 결과 캐시 (Result Cache) |
|:---:|:---|:---|
| **캐싱 대상** | **실행 계획 (How)** | **데이터 결과 (What)** |
| **재실행 여부** | **계획대로 재실행함 (I/O 발생)** | **실행 안 함 (I/O 없음)** |
| **무효화 민감도** | 보통 (DDL 시에만) | **매우 높음 (DML 시 무효화)** |
| **최적 업무** | 일반 트랜잭션 전체 | **변경 적고 연산 복잡한 조회** |

### 2. 과목 융합 관점: 소프트웨어 공학 및 분산 시스템
- **소프트웨어 공학(SE) 융합**: 결과 캐시는 디자인 패턴의 **메모이제이션(Memoization)** 기법을 데이터베이스 레벨에서 구현한 것이며, 함수의 멱등성(Idempotency)을 전제로 성능을 가속한다.
- **분산 시스템 융합**: MSA의 **API 캐싱**이나 Redis를 활용한 **분산 캐시** 전략은 DB의 결과 캐시 사상을 애플리케이션 계층으로 확장한 형태이며, 데이터 일관성(Consistency) 유지 비용과 성능 간의 트레이드 오프를 관리한다.

- **📢 섹션 요약 비유**: 라이브러리 캐시가 '공식 암기'라면, 결과 캐시는 '정답 암기'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 코드성 데이터 및 대형 집계 쿼리 튜닝
- **상황**: 전 시스템에서 공통으로 참조하는 '공통코드' 테이블 조회량이 전체 DB 부하의 30%를 차지함.
- **판단**: 코드는 하루에 한 번 바뀔까 말까 함.
- **결단**: **RESULT_CACHE 힌트 적용**. `SELECT /*+ RESULT_CACHE */ ... FROM Common_Code`. 첫 번째 조회만 DB를 읽고, 이후 수백만 번의 요청은 메모리에서 즉시 반환. DB 서버의 CPU 사용률을 극적으로 낮추고 응답성 확보.

### 2. 안티패턴: DML이 빈번한 테이블에 결과 캐시 적용
- **증상**: 초당 수백 건의 주문이 들어오는 '주문' 테이블 조회 쿼리에 결과 캐시 설정.
- **결과**: 캐시가 생기자마자 무효화되고, 무효화 처리 부하가 오히려 조인을 수행하는 것보다 더 큰 오버헤드 유발. 시스템 전체 성능 퇴보.
- **조치**: **수정 빈도가 높은 테이블은 절대 캐싱 금지**. 오직 'Read-Mostly' 성격의 정적인 데이터에만 적용할 것.

### 📢 기술사적 결언
> "결과 캐시는 데이터베이스 **'성능의 극한'**을 추구하는 도구다. 연산 비용 자체를 0으로 수렴시키기 때문이다. 기술사는 결과 캐시의 속도에 현혹되지 말고, **데이터의 휘발성(Volatility)**을 냉정하게 측정해야 한다. 무효화 오버헤드가 캐싱 이득을 상쇄하는 '임계 지점'을 파악하고, **정합성 우선순위**에 따라 서버 캐시와 클라이언트 캐시를 조율하는 '전략적 캐싱 아키텍트'가 되어야 한다."

- **📢 섹션 요약 비유**: 결과 캐시를 잘못 쓰는 것은 '매초 내용이 바뀌는 뉴스 전광판을 사진 찍어 보여주는 것'과 같아, 내용도 틀리고 사진 찍느라 바빠 정작 뉴스를 못 보게 만듭니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (결과 캐시 적정 활용 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **평균 쿼리 응답 시간** | 100ms | **1ms 이내** | 체감 속도 100배 향상 |
| **논리적 I/O (LIO)** | 1,000 Blocks | **0 Blocks (Mem Read)** | 스토리지 부하 제로화 |
| **CPU 가용량** | 80% 점유 | **20% 미만** | 인프라 확장 없이 수용량 증대 |

### 2. 미래 전망
- **Semantic Caching**: 쿼리가 100% 일치하지 않아도, AI가 논리적 포함 관계를 분석하여 기존 캐시 결과를 부분적으로 재활용하는 **지능형 결과 재구성** 기술 발전.
- **Memory-Centric DBMS**: 모든 데이터를 인메모리에 보관하고, 모든 연산 결과를 영구적인 캐시 형태로 관리하는 **결과 중심 아키텍처**로의 패러다임 전환.

- **📢 섹션 요약 비유**: 미래의 결과 캐시는 단순한 보관함을 넘어, 우리가 무엇을 물어보든 과거의 수많은 정답을 조합해 즉석에서 새 정답을 만들어내는 '데이터 연금술'이 될 것입니다.
