# 뷰(View)와 구체화된 뷰(MView) - 쿼리 재작성과 스냅샷

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 뷰(View)는 물리적 공간을 차지하지 않고 저장된 SQL 문장만 존재하는 **논리적 가상 테이블(Virtual Table)**이며, 구체화된 뷰(MView, Materialized View)는 그 SQL의 집계 연산 결과를 디스크에 물리적인 **스냅샷(Snapshot)으로 영구 저장**하는 스토리지 객체이다.
> 2. **가치**: 일반 뷰는 복잡한 조인을 추상화하여 보안 캡슐화(Encapsulation)와 쿼리 단순성을 제공하고, MView는 수천만 건의 데이터를 묶는 팩트/차원 테이블 조인(OLAP) 시 매번 계산하는 오버헤드를 원천 제거하여 응답 시간을 수백 배(ms 단위) 단축시킨다.
> 3. **융합**: 옵티마이저의 **'쿼리 재작성 (Query Rewrite)'** 기능과 융합되어, 사용자가 원본 테이블에 무거운 쿼리를 날려도 DB 엔진이 백그라운드에서 이를 가로채어 이미 계산해 둔 MView의 결과를 몰래 반환하는 극강의 투명한 최적화 아키텍처를 완성한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **뷰 (View)**: 다른 테이블을 기반으로 만들어진 가상의 테이블. 쿼리문(`SELECT`) 자체만 DB 딕셔너리에 저장되며 데이터는 원본 테이블에 있다.
  - **구체화된 뷰 (Materialized View, MView)**: 뷰의 쿼리 결과(Result Set)를 디스크에 실체화(Materialize)하여 물리적 공간을 할당받아 저장한 객체. 원본 데이터가 변경되면 동기화(Refresh)가 필요하다.

- **💡 비유**: 일반 뷰가 "건물 옥상에서 바깥 풍경을 볼 수 있는 창문(그때그때 밖을 내다봐야 함)"이라면, 구체화된 뷰는 "바깥 풍경을 찍어서 벽에 걸어둔 고화질 사진 액자(밖을 볼 필요 없이 사진만 보면 됨)"와 같습니다.

- **등장 배경**:
  1. **보안과 접근 통제**: 말단 사원이 '전 직원 연봉 테이블'을 접근해야 할 때, 연봉 컬럼만 쏙 뺀 '이름과 직급'만 보이는 가상의 창문(View)을 뚫어주어 원본 테이블 접근 권한을 막기 위해 탄생했다.
  2. **OLAP 시스템의 집계 지연**: 데이터 웨어하우스(DW) 환경에서 경영진 대시보드를 띄울 때마다 매일 1억 건의 매출 테이블을 `SUM()`하고 `GROUP BY` 하느라 10분씩 걸려 속 터지는 사태가 발생했다.
  3. **공간과 시간의 트레이드오프**: 디스크 용량이 넉넉해짐에 따라, CPU를 혹사시켜 매번 계산(시간 낭비)하는 대신 결과를 디스크 공간에 구워버리는(MView) '공간으로 시간을 사는' 철학이 대세가 되었다.

- **📢 섹션 요약 비유**: 일반 뷰는 '매번 요리사에게 똑같은 찌개를 끓여달라고 적어둔 주문서'이고, MView는 '요리사가 어젯밤에 100인분을 미리 끓여서 냉동실에 얼려둔 밀키트(데우기만 하면 즉시 완성)'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 일반 뷰 (View)의 내부 실행 원리 (뷰 머징)

사용자가 `SELECT * FROM 사원_뷰`라는 쿼리를 실행하면, 데이터베이스 엔진(파서)은 딕셔너리에 저장된 `사원_뷰`의 정의(`SELECT 이름, 부서 FROM 사원테이블`)를 가져와 사용자의 쿼리와 병합(Merge)하는 치환 작업을 수행한 후, 결국 원본 '사원테이블'을 향한 실제 물리 디스크 I/O를 발생시킨다. (물리적 성능 향상 효과는 0%이다).

---

### 2. 구체화된 뷰 (MView) 아키텍처와 쿼리 재작성 (Query Rewrite)

MView는 물리적 테이블과 똑같이 취급되므로 내부에 자신만의 B-Tree 인덱스를 독자적으로 가질 수 있다. 가장 큰 마법은 **옵티마이저의 쿼리 재작성(Query Rewrite)** 기능에서 나타난다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 MView 쿼리 재작성 (Query Rewrite) 메커니즘             │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ MView 생성 (사전 작업) ]                                          │
  │  CREATE MATERIALIZED VIEW 월별매출_MV AS                           │
  │  SELECT 월, 지역, SUM(매출액) FROM 원본매출테이블 GROUP BY 월, 지역;         │
  │   - 디스크에 12건의 요약 데이터(스냅샷) 영구 저장됨.                       │
  │                                                                  │
  │  [ 사용자 쿼리 요청 (실시간) ]                                        │
  │  Client ──▶ SELECT SUM(매출액) FROM 원본매출테이블 WHERE 월 = '2026-03'; │
  │                                                                  │
  │  [ ⭐️ 옵티마이저의 가로채기 (Rewrite) ]                               │
  │  1. 옵티마이저 판단: "원본 테이블(1억 건)을 읽으면 5분 걸린다."              │
  │  2. MView 카탈로그 조회: "어? 저 질문의 답, 이미 '월별매출_MV'에 다 계산돼 있잖아?"│
  │  3. 내부 SQL 강제 치환 (Rewrite):                                   │
  │     SELECT 매출액 FROM 월별매출_MV WHERE 월 = '2026-03';               │
  │                                                                  │
  │  [ 결과 반환 ]                                                      │
  │  원본 대신 MView를 스캔하여 0.001초 만에 Client에게 최종 합계 반환!           │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 사용자는 MView의 이름조차 알 필요가 없다. 원래 짜던 대로 원본 테이블을 향해 무거운 `GROUP BY` 쿼리를 날린다. 그러면 똑똑한 **CBO(비용 기반 옵티마이저)**가 백그라운드에서 딕셔너리를 뒤져 "내가 미리 만들어둔 정답지(MView)가 이 쿼리를 포괄(Cover)할 수 있는가?"를 수학적으로 판별한다. 조건이 맞는다면 1억 건짜리 원본으로 가려던 쿼리 실행 계획을 강제로 비틀어 12건짜리 MView 테이블로 연결(Rewrite)해 버린다. 이는 애플리케이션 코드를 1줄도 수정하지 않고 DB의 물리적 튜닝만으로 시스템을 구원하는 최고의 아키텍처 패턴이다.

---

### 3. MView의 동기화 메커니즘 (Refresh Strategy)

원본 테이블에 새 데이터(INSERT)가 들어왔을 때, MView(액자 사진)를 어떻게 최신 상태로 바꿀 것인가에 대한 전략이다.

| 동기화 시점 (When) | 설명 | 동기화 방식 (How) | 설명 및 트레이드오프 |
|:---|:---|:---|:---|
| **ON COMMIT** | 원본에 트랜잭션이 커밋되는 즉시 뷰를 동기화. | **COMPLETE (전체 갱신)** | 뷰 안의 데이터를 다 지우고 쿼리를 처음부터 다시 계산하여 덮어씀. (가장 무식함, 오래 걸림) |
| **ON DEMAND** | 관리자가 지정한 야간 시간(배치)에 주기적으로 동기화. (일반적) | **FAST (빠른 갱신 / 증분 갱신)** | 원본 테이블에 **MView Log(변경 추적 로그)**를 달아두고, 바뀐(Delta) 부분만 뷰에 살짝 더하거나 뺌. (매우 빠름) |

- **📢 섹션 요약 비유**: 매일 아침 가계부를 쓸 때, 어제까지 쓴 내용 다 지우고 1월 1일부터 통장을 다시 다 더해보는 것이 완전 갱신(Complete)이고, 어제 쓴 최종 금액에다가 오늘 쓴 영수증 1장 값만 더해서 적는 것이 빠른 증분 갱신(Fast Refresh)입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 인덱스 vs 일반 뷰 vs 구체화된 뷰(MView) 심층 비교

성능 튜닝을 위한 3대 무기의 차이점 비교.

| 비교 항목 | B-Tree 인덱스 (Index) | 일반 뷰 (View) | 구체화된 뷰 (MView) |
|:---|:---|:---|:---|
| **물리적 저장 공간** | 필요 (인덱스 트리 노드 저장) | 없음 (딕셔너리에 SQL 텍스트만) | **크게 필요 (결과 데이터를 테이블로 통째로 저장)** |
| **적용 목적** | 소수 데이터(Point Query)의 초고속 색인 | SQL 가독성, 복잡한 조인 추상화, 보안 통제 | **대량 데이터의 복잡한 집계(Aggregation) 회피** |
| **동기화 지연** | 없음 (트랜잭션 시 실시간 갱신) | 없음 (항상 원본을 즉시 조회함) | **존재함 (배치 Refresh로 인한 결과적 일관성 타협)** |
| **성능 향상 원리** | 디스크 랜덤 I/O를 O(log N)으로 탐색 | **성능 향상 없음 (오히려 뷰 머징 실패 시 저하)** | **사전 계산(Pre-computation)으로 CPU 연산 비용 = 0화** |

### 과목 융합 관점

- **분산 데이터베이스 (Replication)**: MView는 본래 성능 최적화용으로 만들어졌으나, 원격지에 있는 원본 테이블을 내 지역 데이터베이스로 '복사'해오기 위한 **스냅샷 복제(Snapshot Replication)** 기술의 원형으로 활용되었다. 한국 DB에서 `CREATE MVIEW ... AS SELECT * FROM 미쿡DB_사원테이블`을 걸어두고 매일 동기화하는 식이다.
- **캐시 아키텍처 (OS/Memory)**: MView는 CPU의 연산 결과를 미리 메모리(또는 디스크)에 담아두고 재사용하는 소프트웨어 공학의 '메모이제이션(Memoization)' 패턴이자 캐싱(Caching) 전략이 관계형 데이터베이스 엔진에 네이티브로 구현된 완벽한 사례다.

- **📢 섹션 요약 비유**: 인덱스가 책 맨 뒤의 '찾아보기'라면, 일반 뷰는 무거운 원본 책을 안 보이게 덮어둔 '반투명 색안경'이고, MView는 아예 가장 중요한 핵심 내용만 형광펜으로 칠해서 따로 뽑아낸 얇은 '족집게 요약 노트'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 통신사의 월말 요금 청구 시스템 부하 분산

- **문제 상황**: 매달 1일, 3,000만 가입자의 한 달 치 통화/데이터 사용 로그 100억 건을 조인하여 요금을 청구서 대시보드에 뿌려주는데, WAS와 DB가 과부하(OOM)로 뻗어버리고 대기 시간이 수십 분에 달함. (전형적인 OLAP 병목)
- **의사결정 (MView + Fast Refresh 도입)**:
  1. 원본 로그 테이블(100억 건)에 `MATERIALIZED VIEW LOG` 생성하여 변경분 캡처 인프라 마련.
  2. '가입자별, 월별 합계'를 구하는 MView 생성 (데이터 크기가 100억 건 -> 3,000만 건으로 1/300 축소됨).
  3. 야간에 고객이 뜸한 새벽 2시마다 `Refresh Fast ON DEMAND` 배치 작업을 돌려, 어제 하루치 추가된 통화 로그 델타(Delta) 값만 MView에 합산시킴.
  4. 1일 아침 고객들은 무거운 조인 없이, MView라는 요약 테이블만 단순히 읽어가게 되어 청구서 조회 시간이 0.1초로 극단적 단축 달성.

### 도입 체크리스트
- **데이터 최신성(Freshness) 요구 사항**: 뷰를 조회하는 비즈니스 요건이 "방금 결제한 1원까지 무조건 실시간으로 맞아야 하는가?" (이 경우 MView의 야간 배치는 쓸 수 없고, ON COMMIT 갱신을 해야 하는데 쓰기 부하가 심해짐). 아니면 "어제까지의 매출 동향 파악용인가?" (MView 도입 최적의 시기).
- **Fast Refresh 제약 조건 확인**: MView를 '빠른 증분 갱신(Fast)'으로 만들려면 원본 쿼리에 제약이 매우 많다. 아우터 조인(Outer Join), 서브쿼리, 윈도우 함수, 복잡한 UNION이 들어가면 Fast Refresh가 불가능해 매번 전체를 다 지우고 다시 부어 넣는 Complete 갱신만 가능하여 배치 창(Batch Window) 타임아웃을 유발할 수 있다. 
- **Query Rewrite 옵티마이저 호환성**: 원본 쿼리에 바인드 변수(`?`)를 심하게 남용하거나 데이터 타입이 MView의 정의와 미세하게 다르면, 옵티마이저는 "안전하지 않다"고 판단하여 쿼리 가로채기(Rewrite)를 스스로 포기해 버린다. 실행 계획(EXPLAIN)을 철저히 검증해야 한다.

### 안티패턴: 단순 가독성 향상을 위한 일반 뷰(View)의 무한 중첩 (View within View)
- 개발 편의성을 위해 뷰(A)를 만들고, 그 뷰(A)를 참조하는 뷰(B)를 만들고, 다시 뷰(C)가 B를 조인하는 극악의 연쇄 중첩 스파게티. 옵티마이저가 쿼리 변환(View Merging)을 시도하다가 계층이 너무 깊어 포기하고, 각각의 뷰를 독립된 덩어리로 메모리에 올려버려 카티션 곱(Cartesian Product)과 디스크 스왑 폭발을 일으키는 관계형 DB 성능 파괴의 1순위 주범이다.

- **📢 섹션 요약 비유**: 포장지(일반 뷰)를 겹겹이 5번 싸놓고 물건을 꺼내라고 하면 로봇(옵티마이저)이 포장지를 뜯다가 멈춰버립니다. 필요한 물건은 처음부터 투명하고 단순한 하나의 박스(순수 SQL 조인)나, 아예 완성된 조립품 상자(MView)에 담아 주어야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 일반 SQL 조인 / 일반 View | 구체화된 뷰 (MView) 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **대규모 집계 (OLAP) 속도**| 수억 건 디스크 스캔 및 CPU 그룹핑 연산 | 사전 계산된 요약 블록 단순 메모리 리드 | **대시보드 응답 시간 99% 이상 단축 (분 -> 밀리초)** |
| **주간 트랜잭션 (OLTP) 부하**| 무거운 집계가 주간 피크타임 자원 강탈 | 야간 배치(Refresh)로 무거운 연산 이전 | **주간 메인 트랜잭션 서비스 안정성(TPS) 보호** |
| **코드 이관성** | 앱 소스에 수십 줄 SQL 하드코딩 필요 | Query Rewrite로 소스 수정 없이 적용 | **애플리케이션(WAS) 배포 없이 무중단 DB 성능 향상** |

### 미래 전망
- **실시간 스트리밍 구체화 뷰 (Streaming Materialized View)**: 과거의 MView는 야간 배치(Batch)로 동기화되는 한계가 있었으나, 현재 ksqlDB, Materialize, RisingWave 등 스트림 데이터베이스가 등장하여, Kafka로 들어오는 이벤트 스트림을 10밀리초 이내에 실시간으로 MView에 즉각 증분 연산(Incremental View Maintenance, IVM)해버리는 혁신이 상용화되었다.
- **클라우드 DW의 자동 구체화 (Auto MView)**: Snowflake, BigQuery 같은 클라우드 웨어하우스는 시스템이 사용자의 반복적인 조인 패턴을 머신러닝으로 분석한 뒤, 관리자의 승인 없이도 스스로 MView를 백그라운드에 구워버리고 쿼리를 Rewrite 해주는 '자율 최적화' 기능의 핵심 무기로 이를 진화시켰다.

- **📢 섹션 요약 비유**: 구체화된 뷰(MView)는 거대하고 무거운 빙하(데이터 레이크) 속에서, 사람들이 마시기 좋게 미리 얼음을 캐내어 정수해 둔 '가장 순수하고 시원한 얼음물 저장소'로서, 정보화 시대 속도 경쟁의 궁극적 승리 공신입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **일반 뷰(View)**는 식당 메뉴판이에요. "김치찌개"라고 적혀 있지만 메뉴판 자체는 종이일 뿐이라, 주문하면 그때부터 주방에서 요리사가 찌개를 10분 동안 팔팔 끓여야 한답니다.
2. 하지만 **구체화된 뷰(MView)**는 편의점에 있는 "전자레인지 3분 햄버거"예요. 공장에서 이미 요리(집계 연산)를 다 끝내서 포장해 둔 진짜 음식(물리적 저장)이라서, 데우기만 하면 1초 만에 배불리 먹을 수 있죠!
3. 로봇 종업원(옵티마이저)은 우리가 김치찌개를 시켰을 때 주방이 너무 바쁘면, "손님, 미리 만들어둔 햄버거(MView)로 드릴게요!" 하고 몰래 바꿔서(Query Rewrite) 우리를 안 기다리게 해주는 최고의 센스쟁이랍니다.
