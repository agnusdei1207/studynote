# 177. 쿼리 변환 (Query Transformation) - 옵티마이저의 SQL 재작성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사용자가 입력한 원본 SQL 문장을 옵티마이저가 논리적으로는 동일하지만 물리적으로 더 효율적인 실행이 가능한 형태의 새로운 SQL로 **스스로 고쳐 쓰는(Rewriting)** 지능형 최적화 기술이다.
> 2. **주요 기법**: 중첩된 서브쿼리를 조인으로 바꾸는 **서브쿼리 언네스팅**, 가상 테이블 괄호를 푸는 **뷰 머징**, 그리고 조인 전 데이터를 미리 거르는 **조건절 푸시다운** 등이 대표적이다.
> 3. **가치**: 개발자가 작성한 SQL의 표현 방식에 상관없이 DBMS가 최적의 성능을 보장하게 하며, 복잡한 비즈니스 로직을 단순한 물리적 연산으로 변환하여 시스템의 전반적인 처리량을 극대화한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**쿼리 변환 (Query Transformation)**은 옵티마이저 내부의 '번역기'이자 '편집자'입니다. 사용자가 다소 비효율적인 방식으로 질문(SQL)을 던지더라도, 엔진이 "아, 이 질문은 이렇게 고쳐서 물어보는 게 훨씬 빠르겠군!"이라며 쿼리를 최적화된 내부 언어로 변환합니다.

### 💡 비유: 서투른 외국인의 질문과 가이드의 통역
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        쿼리 변환 (Query Transformation) 비유                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 한 외국인 관광객이 가이드에게 아주 복잡하게 길을 물어봄.                   │
│                                                                             │
│  1. 원본 질문 (Original Query):                                              │
│     - "그쪽으로 가서 저기를 본 다음에, 거기 있는 사람에게 물어보고 왼쪽으로..."      │
│     - 너무 장황하고 비효율적임.                                                  │
│                                                                             │
│  2. 변환된 질문 (Transformed Query):                                          │
│     - 가이드(Optimizer): "아, 결국 **'판교역 1번 출구'** 가고 싶다는 거죠?"         │
│     - 장황한 설명을 생략하고, 가장 짧은 지명(최적 경로)으로 질문을 재정의함.         │
│                                                                             │
│  → 사용자가 어떻게 묻든 **'가장 빠른 정답을 얻을 수 있는 문장'**으로                │
│     엔진이 스스로 고쳐 쓰는 마법입니다!                                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **개발자 숙련도 편차**: 모든 개발자가 튜닝 전문가가 아니므로, 작성된 SQL의 비효율성을 시스템이 스스로 보정해줘야 함.
- **성능의 일관성 유지**: 논리적으로 같은 질문이라면, 어떤 문법(`IN` vs `EXISTS` vs `JOIN`)을 쓰더라도 동일하게 빠른 성능을 내기 위함.

- **📢 섹션 요약 비유**: 쿼리 변환은 '데이터베이스용 맞춤법 교정기'이자 '문장 요약기'와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 쿼리 변환의 3대 핵심 기법

| 기법명 | 상세 내용 | 비유 |
|:---:|:---|:---|
| **서브쿼리 언네스팅** | 중첩된 서브쿼리(Unnest)를 일반 조인(Join)으로 변환. | 겹쳐진 상자 풀기 |
| **뷰 머징 (View Merging)** | 인라인 뷰의 괄호를 풀고 메인쿼리와 합침. | 칸막이 치우기 |
| **조건절 푸시다운** | 바깥쪽의 필터 조건을 안쪽 테이블로 미리 전달. | 입구 컷 (사전 필터링) |

### 2. 서브쿼리 언네스팅 변환 도해 (ASCII)

`IN` 서브쿼리가 어떻게 일반 `Join`으로 변하는지 시각화한다.

```text
    [ Original SQL ]
    SELECT * FROM Dept D
    WHERE  D.ID IN (SELECT E.Dept_ID FROM Emp E);
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Query Transformer ] : 언네스팅 시도                           │
    │  - "IN 서브쿼리는 중복이 발생할 수 있으니 Semi-Join 으로 바꾸자!" │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (Transformed)
    [ Internal SQL ]
    SELECT D.* 
    FROM   Dept D, (SELECT DISTINCT Dept_ID FROM Emp) E  <--- (Join 으로 변환)
    WHERE  D.ID = E.Dept_ID;
```

### 3. 심층 동작 원리: 휴리스틱(Heuristic) vs 비용 기반(Cost-based)
- **Heuristic**: 조건만 맞으면 무조건 변환. (예: 단순 뷰 머징)
- **Cost-based**: 변환했을 때와 안 했을 때의 비용을 비교하여 더 유리한 쪽을 선택. (예: 복잡한 서브쿼리 언네스팅)

### 4. 핵심 기술: 조건절 이행 (Transitive Predicate Generation)
- `A.id = B.id AND B.id = 10` 이라는 조건이 있을 때, 옵티마이저가 스스로 `A.id = 10`이라는 조건을 추가로 생성하여 A 테이블을 먼저 필터링하는 기술.

- **📢 섹션 요약 비유**: 쿼리 변환은 '3단 합체 로봇이 분리되어 싸우는 게 나을지, 합쳐서 싸우는 게 나을지' 판단하는 전투 지휘관의 지능입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 뷰 머징(Merging) vs 뷰 구체화(Materializing)

| 비교 항목 | View Merging (변환) | View Materializing (비변환) |
|:---:|:---|:---|
| **작동 방식** | 뷰를 풀어서 메인과 합침. | 뷰 결과를 임시 테이블에 먼저 담음. |
| **장점** | **인덱스 활용도 극대화**. | 복잡한 집계 연산 부하 격리. |
| **단점** | 조인 순서가 꼬일 수 있음. | **추가 I/O 및 메모리 소모**. |
| **최적 상황** | 단순 조인 및 필터링 위주. | 대량 데이터의 `GROUP BY` 포함 시. |

### 2. 과목 융합 관점: 컴파일러 및 알고리즘
- **컴파일러 융합**: 쿼리 변환은 프로그래밍 언어 컴파일러의 **코드 최적화(Optimization)** 단계인 '데드 코드 제거', '루프 언롤링' 등과 철학적 궤를 같이 한다.
- **알고리즘 융합**: 조건절 푸시다운(Push-down)은 탐색 공간을 사전에 줄이는 **가지치기(Pruning)** 알고리즘의 일종이며, 조인 연산의 시간 복잡도를 $O(N \times M)$에서 $O(n \times m)$으로 낮추는 결정적 수단이다.

- **📢 섹션 요약 비유**: 변환이 '이름을 부르면 바로 오는 친구'라면, 비변환은 '메모지에 이름을 적어두고 나중에 부르는 친구'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 뷰(View)를 사용한 쿼리의 급격한 성능 저하
- **상황**: 공통으로 사용하는 대형 뷰를 조인했는데, 평소보다 100배 느려짐.
- **판단**: 실행 계획 확인 결과, 뷰 내부에 `ROWNUM`이나 `DISTINCT`가 있어 옵티마이저가 '뷰 머징'을 포기하고 전체 데이터를 임시 테이블에 담고 있음.
- **결단**: **쿼리 변환 저해 요소 제거**. 뷰 정의에서 불필요한 `DISTINCT`를 빼거나, `NO_MERGE` 힌트 대신 `PUSH_PRED` 힌트를 사용하여 외부 조건을 뷰 내부로 강제 주입. 뷰를 풀어서 인덱스를 타게 함으로써 성능 복구.

### 2. 안티패턴: 쿼리 변환을 믿고 SQL을 너무 복잡하게 작성
- **증상**: "옵티마이저가 다 고쳐주겠지"라며 10단계 중첩 서브쿼리와 꼬인 로직을 방치.
- **결과**: 변환 규칙의 한계를 벗어나면 옵티마이저는 최악의 경로를 선택하며, 변환 과정 자체(Transformation Time)가 수 초 이상 걸려 하드 파싱 부하 발생.
- **조치**: **가급적 사람이 읽기에도 단순한 표준 SQL 작성**. 시스템의 보조 지능(변환)은 최후의 보루로 남겨두고, 명확한 논리 구조를 지향할 것.

### 📢 기술사적 결언
> "쿼리 변환은 데이터베이스의 **'최적화 자정 작용'**이다. 인간의 불완전한 질문을 기계의 완벽한 실행으로 교정하기 때문이다. 기술사는 옵티마이저가 내린 최종 실행 계획을 보고, **어떤 변환 기법이 적용되었는지(혹은 왜 실패했는지)** 역으로 추적할 수 있어야 한다. 변환의 원리를 꿰뚫을 때, 비로소 힌트 하나로 수조 개의 연산을 지배하는 **'SQL의 마스터'**가 될 수 있음을 명심해야 한다."

- **📢 섹션 요약 비유**: 쿼리 변환은 '구겨진 종이(SQL)를 반듯하게 펴서(Rewriting) 내용을 읽기 쉽게 만드는 과정'입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (쿼리 변환 활성화 시)

| 지표 | 활용 전 | 활용 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **인덱스 효율성** | 부분적 활용 | **전방위적 활용 (Unnesting)** | 성능 10배 이상 향상 |
| **CPU 파싱 부하** | 낮음 (단순 실행) | 조금 높음 (최적 경로 탐색) | **전체 응답 시간은 단축** |
| **개발 유연성** | 튜닝된 SQL 고집 | **자유로운 문법 사용** | 생산성 및 가독성 확보 |

### 2. 미래 전망
- **Deep-learning Query Rewriter**: 과거의 수억 건 쿼리 실행 데이터를 학습한 신경망이, 정해진 규칙을 넘어 창의적인(?) 쿼리 변환을 시도하는 **AI 기반 SQL 재작성**.
- **Cross-Engine Transformation**: 클라우드 멀티 DB 환경에서, Oracle 쿼리를 Snowflake나 Postgres에 최적화된 문법으로 실시간 변환하여 실행하는 **이기종 쿼리 가상화** 기술 안착.

- **📢 섹션 요약 비유**: 미래의 쿼리 변환은 우리가 '뭐 좀 찾아줘'라고 웅얼거리기만 해도, 데이터의 바다에서 가장 빠르고 정확한 질문으로 다듬어 정답을 건네주는 '지능형 필터'가 될 것입니다.

2. 질문이 예쁘고 정확해지면, 정답을 아는 요정(DB)이 **훨씬 더 빨리 대답**을 해줄 수 있거든요.
3. 우리가 조금 헷갈리게 말해도 **컴퓨터가 알아서 똑똑하게 고쳐주니** 정말 고마운 기능이랍니다!
