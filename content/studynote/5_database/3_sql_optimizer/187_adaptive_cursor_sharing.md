# 187. 적응형 커서 공유 (ACS) - 변수 값에 따른 지능형 실행 계획 선택

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 바인드 변수 피킹(Peeking)의 한계를 극복하기 위해 도입된 기술로, 바인드 변수에 입력된 값의 데이터 분포에 따라 옵티마이저가 **런타임에 가장 적합한 실행 계획을 동적으로 선택**하여 공유할 수 있게 하는 지능형 메커니즘이다.
> 2. **진화된 메커니즘**: 특정 쿼리를 **Bind Sensitive** 커서로 지정하여 실행 통계를 감시하고, 입력 값에 따라 성능 차이가 크다고 판단되면 **Bind Aware** 커서로 승격시켜 값의 범위별로 서로 다른 실행 계획을 생성 및 보관한다.
> 3. **가치**: 데이터 편중(Skewness)이 심한 컬럼에 대해 하나의 고정된 계획만을 강요하던 기존 방식에서 탈피하여, 소프트 파싱의 효율성과 실행 계획의 최적성이라는 두 마리 토끼를 동시에 잡는 고도화된 최적화 전략이다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**적응형 커서 공유 (Adaptive Cursor Sharing, ACS)**는 옵티마이저가 '학습'을 하는 과정입니다. 처음에는 바인드 변수 피킹을 통해 대충 계획을 짰더라도, 실제 실행 결과를 지켜보며 "아, 이 값은 인덱스가 낫고 저 값은 전체 스캔이 낫구나!"라고 판단하여 계획을 분리해주는 기법입니다.

### 💡 비유: 손님의 식사량에 맞춰 밥공기를 바꾸는 식당
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        적응형 커서 공유 (ACS) 비유                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 손님마다 먹는 양(Data Amount)이 천차만별인 식당임.                      │
│                                                                             │
│  1. 기존 방식 (피킹만 할 때):                                                 │
│     - 첫 손님이 밥을 한 숟가락만 먹음. ──▶ 주방장이 "아, 여긴 소식가만 오나 봐!"   │
│     - 그 뒤로 모든 손님에게 간장 종지만 한 그릇(Index Scan)에 밥을 줌.            │
│     - 대식가가 오면 밥을 수십 번 퍼 날라야 해서 힘들어 죽음(성능 장애).            │
│                                                                             │
│  2. ACS 방식 (적응형):                                                       │
│     - 주방장이 지켜봄: "어? 저 손님은 그릇이 너무 작아서 고생하네?"                │
│     - 다음부턴 많이 먹는 손님용 '대접(Full Scan)' 계획을 하나 더 만들어둠.         │
│     - 손님이 오면 **'체격(바인드 값)'**을 보고 적절한 그릇을 골라서 줌!            │
│                                                                             │
│  → 상황을 보고 **'유연하게 대처'**하여 모두가 만족(최적 성능)하게 만드는 지능형      │
│     서빙 시스템입니다!                                                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **바인드 피킹의 복불복 리스크**: 첫 번째 실행 시 입력된 값에 따라 전체 시스템 성능이 결정되는 불안정성 해결 필요.
- **데이터 불균형(Skew) 대응**: 90%가 '정상'이고 1%가 '취소'인 상태 데이터에서, 하나의 실행 계획으로 두 경우를 모두 만족시킬 수 없는 한계 극복.

- **📢 섹션 요약 비유**: ACS는 '한 번의 경험으로 단정 짓지 않고, 여러 번 겪어보며 맞춤형 정답지를 늘려가는 신중한 선생님'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. ACS 커서의 상태 전이 단계

| 단계 | 명칭 | 상태 정의 및 역할 |
|:---:|:---|:---|
| **1단계** | **Bind Sensitive** | 피킹이 일어난 커서. 실행 통계(CPU, I/O)를 감시하기 시작함. |
| **2단계** | **Monitoring** | 다른 바인드 값이 들어올 때 성능 차이가 발생하는지 체크. |
| **3단계** | **Bind Aware** | 값에 따라 계획이 달라야 함을 인지. **값의 범위(Selectivity)별로 계획 생성**. |

### 2. ACS 작동 프로세스 도해 (ASCII)

성능 모니터링을 통해 계획이 분화되는 과정을 시각화한다.

```text
    [ Execution 1 : :v1 = 'Minority' ] ──▶ [ Plan A : Index Scan ]
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Optimizer Monitoring ] : "음, 이 쿼리는 바인드 값에 민감하군." │
    │  - Tagging as **'Bind Sensitive'**                              │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    [ Execution 2 : :v1 = 'Majority' ] ──▶ [ Stats Conflict! ]
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Optimizer Decision ] : "안 되겠다! 계획을 하나 더 짜야겠어."   │
    │  - Transition to **'Bind Aware'**                               │
    │  - Create [ Plan B : Full Scan ]                                │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    [ Execution 3 : New Input ] ──▶ **값의 선택도를 판단하여 A 또는 B 선택 실행**
```

### 3. 심층 동작 원리: 선택도 큐브 (Selectivity Cube)
- **원리**: ACS는 바인드 변수 값 자체를 저장하는 것이 아니라, 해당 값이 가질 것으로 예상되는 **선택도(Selectivity)**의 범위를 저장함.
- **작동**: 새로운 값이 들어오면 그 값의 선택도를 계산하고, 이미 저장된 '선택도 범위' 안에 있으면 해당 계획을 재사용, 없으면 하드 파싱 수행.

### 4. 핵심 기술: IS_BIND_SENSITIVE & IS_BIND_AWARE
- `V$SQL` 뷰를 통해 현재 특정 SQL이 ACS의 관리를 받고 있는지, 지능형 모드로 전환되었는지 실시간으로 모니터링할 수 있는 가상 컬럼 기술.

- **📢 섹션 요약 비유**: 선택도 큐브는 '옷 사이즈(S, M, L)'와 같아서, 손님의 키를 재보고 맞는 사이즈의 옷(계획)을 꺼내 주는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 바인드 피킹 vs 적응형 커서 공유 (ACS)

| 비교 항목 | Bind Variable Peeking | Adaptive Cursor Sharing (ACS) |
|:---:|:---|:---|
| **계획 개수** | 오직 1개 (최초 값 기준) | **여러 개 (값의 범위 기준)** |
| **판단 시점** | 최초 하드 파싱 시 단 1회 | **실행 통계 누적에 따른 지속적 판단** |
| **성능 안정성** | 낮음 (복불복 장애 위험) | **높음 (값에 따른 최적화)** |
| **오버헤드** | 거의 없음 | 모니터링 및 추가 파싱 비용 발생 |

### 2. 과목 융합 관점: 머신러닝 및 제어 공학
- **머신러닝 융합**: ACS는 데이터베이스 내부에 구현된 기초적인 **온라인 학습(Online Learning)** 시스템이며, 피드백 루프를 통해 모델(실행 계획)을 정교화하는 강화 학습의 사상을 일부 담고 있다.
- **제어 공학 융합**: 시스템의 출력값(실행 성능)을 관찰하여 입력 제어 로직(파싱 전략)을 변경하는 **피드백 제어(Feedback Control)** 시스템의 전형적인 사례다.

- **📢 섹션 요약 비유**: 피킹이 '첫인상으로 평생을 판단하는 것'이라면, ACS는 '사귀어보며 상대의 다양한 면모를 알아가는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 불균형한 데이터 상태에서의 안정적 서비스
- **상황**: 고객 상담 테이블의 '처리여부' 컬럼에 '미처리(1%)'와 '완료(99%)'가 섞여 있음.
- **판단**: 미처리를 조회할 땐 인덱스가 빠르고, 완료를 조회할 땐 전체 스캔이 빠름.
- **결단**: **ACS 활성화 및 모니터링**. 쿼리가 처음 '미처리' 값으로 피킹되어 인덱스 계획을 잡더라도, 나중에 '완료' 조회 시 성능 저하를 감지한 ACS가 전체 스캔 계획을 추가로 생성하게 함으로써 전천후 성능 안정성 확보.

### 2. 안티패턴: ACS 부하를 고려하지 않은 과도한 바인드 변수
- **증상**: 수천 개의 인자를 가진 복잡한 쿼리에 ACS를 무분별하게 적용.
- **결과**: ACS가 매번 선택도를 계산하고 계획을 분화하느라 파싱 오버헤드가 커지고, Shared Pool 메모리가 자식 커서들로 가득 차서 단편화 발생.
- **조치**: **핵심 컬럼 위주로 인덱스 전략 수립**. ACS는 만능이 아니므로, 가급적 데이터 분포를 균일하게 가져가는 모델링을 선행하고 정 안 되는 곳에만 ACS의 혜택을 받을 것.

### 📢 기술사적 결언
> "적응형 커서 공유(ACS)는 데이터베이스의 **'학습 지능'**을 증명한다. 정적인 규칙이 아닌 동적인 데이터 흐름에 반응하기 때문이다. 기술사는 ACS가 제공하는 유연함 이면에 숨겨진 **커서 폭발(Cursor Explosion)**의 위험성을 경계해야 한다. 실행 계획이 너무 많이 쪼개지지 않는지 상시 모니터링하고, 필요하다면 **SQL Plan Baseline**과 결합하여 검증된 계획들 안에서만 ACS가 동작하도록 통제하는 '지능형 관리자'가 되어야 한다."

- **📢 섹션 요약 비유**: ACS를 너무 믿고 쿼리를 막 짜는 것은 '시험 문제가 매번 바뀔 텐데 선생님의 암기력만 믿고 공부 안 하는 학생'과 같이 위험합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (ACS 성공적 정착 시)

| 지표 | 도입 전 | 도입 후 (ACS) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **쿼리 응답 시간 편차** | 0.1초 ~ 100초 | **0.1초 ~ 1초** | 서비스 신뢰도 획기적 개선 |
| **하드 파싱 빈도** | 높음 (Literal 전환 시) | **낮음 (Bind 유지)** | CPU 가용 자원 확보 |
| **운영 관리 비용** | 수동 힌트 주입 빈번 | **자동 최적화** | DBA 업무 효율 향상 |

### 2. 미래 전망
- **Deep Adaptive Execution**: 단순히 선택도 범위가 아니라, 실시간 시스템 부하와 사용자 중요도까지 고려하여 계획을 바꾸는 **초적응형 실행 엔진** 등장.
- **Self-Healing Plan Management**: 성능이 떨어진 커서를 AI가 즉시 격리하고 최적의 계획을 재탐색하여 배포하는 **자가 치유형 커서 관리** 보편화.

- **📢 섹션 요약 비유**: 미래의 조인 공유 기술은 우리가 아무리 변덕스러운 질문을 던져도, 우리의 상황을 미리 알고 가장 완벽한 대답을 준비하는 '독심술사'가 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **적응형 커서 공유**는 컴퓨터가 처음 본 친구랑만 놀지 않고, **"새로운 친구가 올 때마다 그 친구가 좋아하는 놀이"**를 기억해두는 거예요.
2. 운동장에서 술래잡기를 좋아하는 친구에겐 운동화(인덱스)를 빌려주고, 수영을 좋아하는 친구에겐 오리발(전체 스캔)을 빌려주는 식이죠.
3. 이렇게 하면 어떤 친구가 놀러 와도 **당황하지 않고 가장 재미있게(빠르게)** 놀 수 있답니다!
