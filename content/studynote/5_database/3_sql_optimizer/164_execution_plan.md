# 164. 실행 계획 (Execution Plan) - 쿼리 실행의 물리적 단계와 도식화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 옵티마이저가 SQL을 실행하기 위해 수립한 상세한 **물리적 작업 명세서**로, 어떤 테이블부터 읽을지, 어떤 인덱스를 사용할지, 조인은 어떤 방식으로 할지 등의 연산 절차를 계층적으로 보여준다.
> 2. **구성 요소**: 데이터를 찾는 경로인 **액세스 경로(Access Path)**, 두 집합을 합치는 **조인 기법(Join Method)**, 그리고 각 단계의 예상 부하를 나타내는 **비용(Cost)과 카디널리티**로 이루어져 있다.
> 3. **가치**: 느린 SQL의 병목 지점(Bottleneck)을 정확히 진단하게 해주는 유일한 'X-ray'이며, 개발자가 의도한 대로 DB가 동작하는지 검증하고 성능을 개선하는 **SQL 튜닝의 핵심 산출물**이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**실행 계획 (Execution Plan)**은 SQL 문장이라는 '주문서'를 받은 DBMS 요리사가 주방에서 '요리하는 순서'를 적어놓은 레시피와 같습니다. 사용자는 `EXPLAIN PLAN FOR` 또는 `AUTOTRACE` 기능을 통해 이 내부 로직을 엿볼 수 있습니다.

### 2. 💡 비유: 가구 조립 설명서
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        실행 계획 (Execution Plan) 비유                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 복잡한 이케아 가구(결과 데이터)를 조립하려 함.                            │
│                                                                             │
│  1. SQL: "가구 완제품을 거실에 놓아줘!" (사용자의 선언적 요청)                   │
│  2. 실행 계획:                                                               │
│     - 1단계: 창고(Table) A구역에서 나사(Index)를 먼저 찾음.                     │
│     - 2단계: 다리 판재와 상판을 해시 조인(Join)으로 결합함.                      │
│     - 3단계: 다 만들어진 가구를 거실로 배달(Output)함.                           │
│                                                                             │
│  → 실행 계획은 가구를 만드는 과정 중 **'어느 단계가 제일 힘든지(Cost)'**와          │
│     **'순서가 뒤바뀌진 않았는지'**를 보여주는 설계 도면입니다!                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **SQL의 블랙박스 해소**: SQL은 "무엇"만 정의하므로, 성능이 안 나올 때 엔진 내부에서 무슨 일이 벌어지는지 알 수 없는 답답함 해결 필요.
- **체계적인 성능 관리**: 감(Guess)에 의존하는 튜닝이 아닌, 객관적인 수치(Cost, Card)에 기반한 공학적인 성능 개선 체계 구축.

- **📢 섹션 요약 비유**: 실행 계획은 쿼리라는 환자의 속을 들여다보고 어디가 아픈지(느린지) 찾아내는 '정밀 진단서'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 실행 계획의 계층적 읽기 규칙
- **원리**: 안쪽에서 바깥쪽으로(Inside-Out), 위에서 아래로(Top-Down) 읽습니다.
- **의미**: 가장 들여쓰기가 깊게 된 단계가 가장 먼저 실행되는 물리적 연산입니다.

### 2. 실행 계획 기본 도해 (ASCII)

전형적인 인덱스 스캔 후 조인되는 계획의 구조를 시각화한다.

```text
    ---------------------------------------------------------------------------
    | Id  | Operation                    | Name           | Rows  | Cost (%CPU)|
    ---------------------------------------------------------------------------
    |   0 | SELECT STATEMENT             |                |     1 |     3 (0)  |
    |   1 |  NESTED LOOPS                |                |     1 |     3 (0)  |
    |   2 |   TABLE ACCESS BY INDEX ROWID| EMP            |     1 |     2 (0)  |
    |*  3 |    INDEX RANGE SCAN          | EMP_ID_IDX     |     1 |     1 (0)  | <--- (1) 최초 실행
    |   4 |   TABLE ACCESS BY INDEX ROWID| DEPT           |     1 |     1 (0)  |
    |*  5 |    INDEX UNIQUE SCAN         | DEPT_PK        |     1 |     0 (0)  | <--- (2) 조인 대조
    ---------------------------------------------------------------------------
    * Predicate Information: 3 - access("ID"=7), 5 - access("E.DID"="D.DID")
```

### 3. 3대 핵심 분석 요소 (The Big 3)
1. **Operation**: 무엇을 하는가? (Table Scan, Index Scan, Join, Sort 등)
2. **Cardinality (Rows)**: 해당 단계에서 처리되는 예상 행 수. 이 숫자가 틀리면 전체 계획이 꼬임.
3. **Cost**: 해당 단계를 수행하는 데 드는 상대적인 비용. (작을수록 좋음)

### 4. 핵심 기술: 조건 처리(Predicates)의 분리
- **Access Predicate**: 인덱스를 타게 만드는 결정적인 조건. (빠름)
- **Filter Predicate**: 데이터를 가져온 뒤 체크하는 부가적인 조건. (느림)

- **📢 섹션 요약 비유**: Access Predicate는 '주소를 알고 집을 찾아가는 것'이고, Filter Predicate는 '일단 집안에 들어가서 물건을 하나씩 확인하는 것'입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 예상 실행 계획 vs 실제 실행 통계

| 비교 항목 | Explain Plan (예상) | Trace / Runtime (실제) |
|:---:|:---|:---|
| **확인 시점** | 쿼리 실행 전 (정적) | **쿼리 실행 후 (동적)** |
| **정확도** | 통계에 기반한 추측 | **100% 진실** |
| **용도** | 설계 단계의 검증 | **장애 원인 분석 및 튜닝** |
| **비유** | 예상 경로와 도착 시간 | **블랙박스에 기록된 실제 경로** |

### 2. 과목 융합 관점: 자료구조 및 운영체제
- **자료구조 융합**: 실행 계획은 내부적으로 **방향성 비순환 그래프(DAG)**로 관리되며, 각 노드는 트리 구조의 부모-자식 관계를 형성한다.
- **운영체제(OS) 융합**: 실행 계획의 각 단계는 OS의 **프로세스/스레드**와 연동되어 실행되며, 특히 병렬 실행 계획(`PX COORDINATOR`)은 OS 스케줄러의 자원 분배 전략에 직접적인 영향을 받는다.

- **📢 섹션 요약 비유**: 예상 계획이 '여행 전 지도 보기'라면, 실제 통계는 '다녀온 뒤 영수증 확인하기'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 튜닝의 첫걸음, '나쁜 계획' 식별하기
- **상황**: 평소 1초 걸리던 쿼리가 1분으로 늘어남.
- **판단**: 실행 계획 확인 결과, 기존 `INDEX SCAN`이 `FULL TABLE SCAN`으로 바뀌어 있음.
- **결단**: **카디널리티 오차(Misestimation) 확인**. 데이터 분포가 변했는데 통계가 갱신 안 되어 옵티마이저가 착각함. 통계를 다시 수집하여 실행 계획을 원래의 인덱스 기반으로 되돌림.

### 2. 안티패턴: 실행 계획 없이 쿼리만 고치기 (Trial and Error)
- **증상**: 쿼리가 느리면 실행 계획도 안 보고 `DISTINCT`를 붙여보거나 힌트를 이것저것 막 넣어봄.
- **결과**: 우연히 빨라질 순 있으나 원인을 모르니 나중에 재발하며, 오히려 다른 쿼리에 악영향을 줌.
- **조치**: **반드시 실행 계획을 먼저 확인**하여, 어느 지점에서 I/O가 튀는지(Cost가 높은지) 수치로 확인한 뒤 '핀포인트 튜닝'을 실시할 것.

### 📢 기술사적 결언
> "실행 계획은 데이터베이스와 대화하는 **'청진기'**다. 엔진의 숨소리(Cost)와 맥박(Rows)을 들을 수 있기 때문이다. 기술사는 단순 명령어를 외우는 것을 넘어, 실행 계획의 **각 단계가 물리적 블록 I/O와 메모리 점유**에 어떤 의미를 갖는지 해석할 수 있어야 한다. 실행 계획을 100% 이해하고 통제할 수 있는 능력이 곧 **최고 수준의 DB 아키텍트**를 결정짓는 기준선임을 명심해야 한다."

- **📢 섹션 요약 비유**: 실행 계획을 무시하고 튜닝하는 것은 '눈을 감고 환자를 수술하는 의사'와 같아 매우 위험한 도박입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (실행 계획 활용 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **장애 원인 파악 시간** | 수 시간 (추측) | **수 분 (계획 확인)** | 문제 해결 속도 90% 향상 |
| **SQL 성능 안정성** | 예측 불가능 | **정량적 통제 가능** | 서비스 신뢰도 극대화 |
| **인프라 비용** | 무분별한 증설 | **쿼리 효율화로 절감** | HW 투자 비용 30% 절감 |

### 2. 미래 전망
- **Visual Execution Analysis**: 텍스트 형태의 계획을 넘어, 데이터의 흐름과 속도를 빛의 움직임으로 보여주는 **3D 시각화 실행 계획** 도구의 발전.
- **Plan Stability AI**: 엔진이 스스로 '좋은 계획'을 기억해두었다가, 통계가 변해도 성능이 검증된 과거 계획을 유지하는 **지능형 계획 고정(Baseline Management)** 기술 고도화.

- **📢 섹션 요약 비유**: 미래의 실행 계획은 단순한 결과 통보가 아니라, "이 길로 가면 막히니까 저 길로 가는 게 어때요?"라고 먼저 말을 거는 스마트 내비게이션이 될 것입니다.
