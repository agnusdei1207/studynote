# 186. SQL 커서 관리 - 세션 및 공유 커서의 생명주기

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SQL 문장이 실행되는 메모리 상의 작업 영역이며, 서버 메모리(SGA)에 저장되어 여러 사용자가 공유하는 **공유 커서(Shared Cursor)**와 개별 사용자 프로세스(PGA)에서 실행 상태를 관리하는 **세션 커서(Session Cursor)**로 이원화되어 관리된다.
> 2. **생명주기**: [Open $\to$ Parse $\to$ Bind $\to$ Execute $\to$ Fetch $\to$ Close]의 엄격한 절차를 따르며, 빈번한 Parse와 Close 부하를 줄이기 위한 **커서 캐싱(Cursor Caching)** 기술이 성능 최적화의 핵심이다.
> 3. **가치**: 커서의 효율적인 관리는 하드 파싱과 소프트 파싱의 빈도를 직접적으로 제어하며, 시스템 자원(Memory, CPU)의 효율적 분배와 대량 동시 요청에 대한 데이터베이스의 수용력을 결정한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**커서 (Cursor)**는 SQL 문을 처리하기 위해 할당된 전용 메모리 공간입니다. SQL이라는 '프로그램'이 돌아가기 위한 '프로세스 메모리'라고 이해하면 쉽습니다.

### 💡 비유: 도서관 책 읽기와 필기장
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        SQL 커서 (Cursor) 개념 비유                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 공유 커서 (도서관의 전공 서적):                                            │
│     - 모든 학생이 같이 보는 책. (SGA 상의 실행 계획)                             │
│     - 누군가 이미 해석해 둔 주석(최적화 결과)이 달려 있어 다들 편하게 봄.           │
│                                                                             │
│  2. 세션 커서 (개인 필기장):                                                  │
│     - 책을 읽으면서 내가 어디까지 읽었는지(Fetch Pointer),                        │
│     - 어떤 단어를 찾고 있는지(Bind Variable) 적어두는 나만의 메모장. (PGA)        │
│                                                                             │
│  → **'공통의 지식(Shared)'**을 보면서, **'나만의 실행 상태(Session)'**를            │
│     유지하는 데이터베이스의 작업 단위입니다!                                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **반복 연산의 효율화**: 한 번 실행한 쿼리를 다시 실행할 때, 메모리 주소값만 알면 즉시 실행할 수 있도록 연결 고리(Handle)가 필요해짐.
- **상태 관리의 정밀성**: 수만 건의 데이터를 한꺼번에 못 가져오므로, "현재 몇 번째 행을 읽고 있는가"라는 포인터 정보 보관 필수.

- **📢 섹션 요약 비유**: 커서는 '데이터라는 바다를 항해하는 배의 조종실'과 같아서, 목적지(결과)까지 가는 모든 경로와 상태를 관리합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 커서의 이중 구조 상세 비교

| 구분 | 공유 커서 (Shared Cursor) | 세션 커서 (Session Cursor) |
|:---:|:---|:---|
| **저장 위치** | **SGA (Library Cache)** | **PGA (Private SQL Area)** |
| **공유 여부** | 모든 사용자가 공유 가능. | 해당 사용자(세션)만 사용 가능. |
| **저장 내용** | SQL 문장, Parse Tree, **실행 계획**. | 바인드 변수 값, **현재 행 포인터**. |
| **비유** | 영화 필름 원본 | 영화관의 영사기 상태 |

### 2. 커서 생명주기 및 작동 프로세스 도해 (ASCII)

SQL이 실행되는 단계별 커서의 역할을 시각화한다.

```text
    [ Application ]         [ Session Cursor (PGA) ]      [ Shared Cursor (SGA) ]
           │                           │                             │
    1. [ OPEN ]     ────▶   새 작업 영역 할당   │                             │
           │                           │                             │
    2. [ PARSE ]    ────▶   문법/권한 체크      ◀─────(Match?)─────▶  실행 계획 공유
           │                           │                             │
    3. [ BIND ]     ────▶   변수 값 대입 (:v1)  │                             │
           │                           │                             │
    4. [ EXECUTE ]  ────▶   SQL 엔진 구동       │                             │
           │                           │                             │
    5. [ FETCH ]    ────▶   데이터 한 건씩 인출 ◀──────(I/O)──────▶  Table / Index
           │                           │                             │
    6. [ CLOSE ]    ────▶   작업 영역 해제      │                             │
```

### 3. 심층 동작 원리: 커서 캐싱 (Cursor Caching)
- **원리**: 쿼리가 종료되어도 `CLOSE` 하지 않고, 세션 메모리에 핸들을 유지함.
- **효과**: 다음에 같은 쿼리 요청 시 `OPEN/PARSE` 단계를 건너뛰고 즉시 `EXECUTE`로 진입 (Recursive CPU 소모 방지).

### 4. 핵심 기술: 부모/자식 커서 (Parent & Child)
- SQL 텍스트가 같아도 사용자 환경(NLS_SET 등)이 다르면, 하나의 부모 커서 아래 여러 개의 자식 커서(실제 실행 계획)를 주렁주렁 매달아 관리함.

- **📢 섹션 요약 비유**: 커서 캐싱은 '자주 전화하는 번호를 단축번호로 등록해두는 것'과 같아, 번호를 일일이 누르는(파싱) 수고를 덜어줍니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 하드 파싱 vs 소프트 파싱 vs 커서 캐싱

| 비교 항목 | 하드 파싱 | 소프트 파싱 | 커서 캐싱 (Bypass) |
|:---:|:---|:---|:---|
| **SGA 조회** | 실패 (계획 생성) | **성공 (계획 공유)** | **불필요 (이미 쥐고 있음)** |
| **CPU 부하** | 최상 | 보통 | **최저** |
| **속도** | 느림 | 빠름 | **가장 빠름** |
| **비유** | 문제 처음 풀기 | 문제지 다시 보기 | **답을 외워서 즉답하기** |

### 2. 과목 융합 관점: 운영체제 및 프로그래밍
- **운영체제(OS) 융합**: 세션 커서는 OS의 **파일 디스크립터(File Descriptor)**나 프로세스의 **Handle** 개념과 정확히 일치하며, 시스템 자원의 한정성(Open Cursors 제한)으로 인해 철저한 반납이 요구된다.
- **프로그래밍 융합**: 자바의 `PreparedStatement`는 커서 관리의 객체 지향적 구현체이며, 이를 재사용하는 **DBCP(Database Connection Pool)**는 세션 커서 캐싱 효과를 전사적으로 확장한 기술이다.

- **📢 섹션 요약 비유**: 소프트 파싱이 '도서관에서 책을 찾는 것'이라면, 커서 캐싱은 '책을 아예 내 가방에 넣어두는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 'Maximum Open Cursors Exceeded' 에러 조치
- **상황**: 서비스 운영 중 갑자기 DB 연결이 끊기며 'ORA-01000' 에러 발생.
- **판단**: 애플리케이션 소스 코드에서 SQL 실행 후 `close()`를 호출하지 않아, 세션 커서가 계속 쌓여 임계치(Limit)를 초과함.
- **결단**: **리소스 반납 로직 점검 및 세션 커서 캐싱 설정**. `finally` 블록에서 커서를 명시적으로 닫게 하고, 자주 쓰는 쿼리는 `SESSION_CACHED_CURSORS` 파라미터를 통해 엔진이 안전하게 관리하도록 유도.

### 2. 안티패턴: 대량의 Ad-hoc 쿼리로 인한 자식 커서 폭발
- **증상**: 동일 SQL인데 자식 커서(Child)가 수천 개씩 생김.
- **결과**: `v$sql_shared_cursor` 조회 시 'Mismatch' 사유가 가득하며, 라이브러리 캐시 탐색 속도가 급격히 저하됨.
- **조치**: **환경 설정 통일**. 바인드 변수의 데이터 타입을 일치시키고, 세션별로 다른 NLS 설정을 통일하여 불필요한 자식 커서 생성을 억제할 것.

### 📢 기술사적 결언
> "커서 관리는 데이터베이스 **'메모리 거버넌스'**의 핵심이다. 커서는 단순한 통로가 아니라 시스템의 모든 유한한 자원(CPU, Memory, Latch)이 집약되는 교차점이기 때문이다. 기술사는 단순한 SQL 실행력을 넘어, **커서의 생명주기가 애플리케이션 커넥션 풀과 어떻게 연동**되는지 꿰뚫어야 한다. 완벽하게 관리되는 커서야말로 **무중단 고가용성 시스템**을 지탱하는 가장 정교한 기초 공사임을 잊지 말아야 한다."

- **📢 섹션 요약 비유**: 커서를 닫지 않는 것은 '수도꼭지를 틀어놓고 집을 나가는 것'과 같아, 결국 수조(메모리)가 바닥나 시스템이 멈추게 됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (커서 최적 관리 시)

| 지표 | 활용 전 | 활용 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **파싱 CPU 점유율** | 40% 이상 | **5% 미만** | 업무 처리량 2배 향상 |
| **시스템 장애 발생율** | 빈번 (Cursor Leak) | **0% (안정적 관리)** | 서비스 안정성 확보 |
| **쿼리 응답 지연** | 간헐적 발생 | **일정한 성능 유지** | 사용자 경험(UX) 최적화 |

### 2. 미래 전망
- **Autonomous Cursor Management**: 엔진이 앱의 누수(Leak)를 스스로 감지하여 장시간 미사용 커서를 강제 회수하거나, 비효율적인 공유 구조를 실시간 재조정하는 자율 기술 탑재.
- **Persistent Session Cursors**: 서버 재기동 후에도 사용자의 이전 커서 상태를 보존하여, 재접속 시 파싱 단계를 완전히 생략하는 **'영구 커서'** 기술의 클라우드 도입.

- **📢 섹션 요약 비유**: 미래의 커서는 단순한 메모리 칸이 아니라, 우리가 어떤 길(데이터)을 가려는지 미리 알고 길을 닦아놓는 '지능형 네비게이터'가 될 것입니다.
