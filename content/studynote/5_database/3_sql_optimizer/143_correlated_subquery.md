# 143. 상관 서브쿼리 (Correlated Subquery) - 메인쿼리와의 의존적 실행

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서브쿼리 내부에서 메인쿼리의 컬럼을 참조하여 실행되는 형태로, 메인쿼리의 한 행(Row)이 읽힐 때마다 서브쿼리가 해당 행의 값을 인자로 받아 **반복적으로 수행**되는 중첩 질의 방식이다.
> 2. **작동 특성**: 비상관 서브쿼리와 달리 서브쿼리 단독으로는 실행이 불가능하며, 메인쿼리와 서브쿼리가 **데이터 한 건마다 상호작용**하며 필터링을 수행하므로 주로 **`EXISTS`**나 **`IN`** 연산과 결합하여 존재 여부를 확인하는 데 쓰인다.
> 3. **가치**: "자신이 속한 부서의 평균보다 많이 받는 사원"과 같이 개별 행의 맥락(Context)에 의존하는 복잡한 조건을 선언적으로 표현하게 해주며, 성능 최적화를 위해 **서브쿼리 언네스팅(Unnesting)**과의 조화가 필수적이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**상관 서브쿼리 (Correlated Subquery)**는 서브쿼리가 메인쿼리의 바깥쪽 값을 참조하는 형태입니다. 메인쿼리가 서브쿼리에 '입력값'을 던져주고, 서브쿼리는 그 값을 바탕으로 결과를 내어 다시 메인쿼리에 '필터링 결과'를 돌려주는 방식입니다.

### 2. 💡 비유: 한 명씩 붙잡고 물어보는 '입구 조사'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        상관 서브쿼리 비유                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 파티장에 입장하려는 사람(메인쿼리)들을 검사하고 있음.                     │
│                                                                             │
│  1. 비상관 (단순 조사): "키가 180 넘는 사람만 들어오세요."                       │
│     - 기준(180)이 이미 정해져 있어 한 번만 공지하면 끝! (Fast)                  │
│                                                                             │
│  2. 상관 (상호 조사): "자기 가족 중에 제일 키 큰 사람만 들어오세요."               │
│     - 입장객 A가 오면: 그 사람의 가족 명단(서브쿼리)을 뒤져서 A가 1등인지 확인.     │
│     - 입장객 B가 오면: 다시 B의 가족 명단을 뒤져서 B가 1등인지 확인.               │
│                                                                             │
│  → 사람 한 명 올 때마다 **'그 사람의 조건'**에 맞춰 질문을 새로 던지는 방식입니다!   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **행 단위 맥락(Context)의 필요성**: 집합 전체의 통계치가 아니라, 각 행이 속한 그룹이나 특정 속성값에 따라 기준이 변하는 복잡한 필터링 요구사항 발생.
- **존재 여부 확인 (Existence Check)**: 특정 데이터가 다른 테이블에 "있는지 없는지"를 효율적으로 판단하기 위해 도입.

- **📢 섹션 요약 비유**: 상관 서브쿼리는 '개인 맞춤형 질문'과 같아서, 대답하는 사람(행)마다 질문 내용이 조금씩 달라집니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 상관 서브쿼리의 실행 메커니즘

| 단계 | 수행 활동 | 비유 |
|:---:|:---|:---|
| **1. 메인 행 추출** | 메인쿼리에서 한 건의 데이터를 읽음. | 면접자 1명 호명 |
| **2. 값 전달** | 해당 행의 특정 컬럼 값을 서브쿼리의 변수로 전달. | 이력서 전달 |
| **3. 서브 실행** | 전달받은 값을 조건으로 서브쿼리 연산 수행. | 이력서 기반 질문 |
| **4. 조건 판정** | 서브쿼리의 결과가 참(True)이면 메인 행을 결과에 포함. | 합격 여부 결정 |

### 2. 상관 서브쿼리 논리 구조 도해 (ASCII)

메인쿼리와 서브쿼리 간의 데이터 루프를 시각화한다.

```text
    [ Main Query: Employee (E) ]
    ┌───────────────────────────────────────────┐
    │ Row 1: {Name: 'Kim', Dept: 'Sales', Sal: 500}  ──┐
    │ Row 2: {Name: 'Lee', Dept: 'IT', Sal: 700}     ──┼──┐ (Step 1: Get Dept)
    └───────────────────────────────────────────┘    │  │
                                                     │  │
    ┌────────────────────────────────────────────────▼──▼─────────────┐
    │ [ Correlated Subquery ]                                         │
    │ SELECT AVG(Sal) FROM Employee WHERE Dept = **E.Dept**           │ <--- (Step 2: Compare)
    └────────────────────────────────────────────────┬──┬─────────────┘
                                                     │  │
    ┌────────────────────────────────────────────────▼──▼─────────────┐
    │ Result: Kim (Sales Avg: 450) -> OK (Include)                    │ (Step 3: Output)
    │ Result: Lee (IT Avg: 800)    -> Fail (Exclude)                  │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: EXISTS 연산자와의 결합
- **EXISTS**: 서브쿼리의 결과가 '한 건이라도' 있으면 참을 반환.
- **단락 평가 (Short-circuit)**: 상관 서브쿼리와 EXISTS가 결합하면, 조건을 만족하는 첫 번째 행을 찾는 즉시 서브쿼리 실행을 멈추므로 매우 효율적임.

### 4. 핵심 기술: 서브쿼리 언네스팅 (Unnesting)
- **개념**: 옵티마이저가 상관 서브쿼리를 **세미 조인(Semi-Join)**이나 일반 조인으로 바꾸어 실행하는 것.
- **효과**: 행 단위 반복 실행(Loop)을 집합 단위 연산(Hash Join)으로 승격시켜 성능 폭증 유도.

- **📢 섹션 요약 비유**: EXISTS 는 '찾는 물건이 창고에 하나라도 있는지 확인하는 즉시 문을 닫고 나오는 똑똑한 창고지기'와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 비상관 서브쿼리 vs 상관 서브쿼리

| 비교 항목 | 비상관 서브쿼리 (Independent) | 상관 서브쿼리 (Dependent) |
|:---:|:---|:---|
| **실행 순서** | 서브쿼리(1회) $\to$ 메인쿼리 | **메인 행(N회) $\leftrightarrow$ 서브 행** |
| **의존성** | 독자적으로 실행 가능 | **메인 없이는 실행 불가** |
| **성능** | 대체로 빠름 | 데이터가 많을수록 느려짐 (N번 반복) |
| **비유** | "1등 점수가 몇 점이지?" (정답지 확인) | "네가 1등이니?" (한 명씩 대조) |

### 2. 과목 융합 관점: 알고리즘 및 프로그래밍
- **알고리즘 융합**: 상관 서브쿼리는 전형적인 **중첩 루프(Nested Loop)** 방식이며, 시간 복잡도는 $O(N_{main} \times N_{sub})$에 수렴한다. 최적화 과정은 이 루프를 인덱싱된 탐색이나 해시로 치환하는 과정이다.
- **프로그래밍 융합**: 상관 서브쿼리는 람다식이나 스트림의 **Filter(row -> condition)** 함수와 논리적으로 동일하며, 외부 스코프의 변수를 참조하는 **클로저(Closure)**와 유사한 성격이 있다.

- **📢 섹션 요약 비유**: 비상관이 'TV 중계'라면 상관은 '일대일 영상 통화'입니다. 통화량이 많아질수록 통신망(DB 자원) 부하가 심해집니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 부서별 최신 주문 고객 조회
- **상황**: 각 부서(또는 지역)에서 가장 최근에 주문을 완료한 고객 리스트를 뽑아야 함.
- **판단**: 단순히 조인만으로는 '가장 최근'이라는 행 단위 필터링이 복잡함.
- **결단**: **상관 서브쿼리 적용**. `SELECT * FROM Orders O WHERE Order_Date = (SELECT MAX(Date) FROM Orders WHERE Dept = O.Dept)`. 각 행의 부서 내에서 최대 날짜인지 실시간 대조하여 정확한 최신 데이터 추출.

### 2. 안티패턴: 대량 테이블에서의 중첩된 상관 쿼리
- **증상**: 1,000만 건 테이블 조회 시 `SELECT` 절이나 `WHERE` 절에 상관 서브쿼리를 2~3개씩 중첩.
- **결과**: I/O가 기하급수적으로 발생(1,000만 * 1,000만)하며 시스템 전체 중단.
- **조치**: **상관 쿼리를 분석용 함수(Window Function)로 전환**하거나, 인라인 뷰를 통한 미리 조인(Pre-join) 방식으로 아키텍처 개선.

### 📢 기술사적 결언
> "상관 서브쿼리는 **'문맥적 필터링'**의 정수다. 데이터 그 자체가 아닌 데이터들 사이의 관계적 위치를 조건으로 쓰기 때문이다. 기술사는 상관 서브쿼리의 편리한 문법 뒤에 숨겨진 **'반복 수행의 공포'**를 경계해야 한다. 옵티마이저가 이를 **조인으로 변환(Unnesting)** 할 수 있도록 쿼리를 힌트와 함께 가이드하거나, **인덱스 스캔 효율**을 극대화하여 반복 연산의 단가를 낮추는 '물리적 튜닝' 역량을 발휘해야 한다."

- **📢 섹션 요약 비유**: 상관 서브쿼리를 함부로 쓰는 것은 '전교생 한 명씩 교무실로 불러서 가족 사항을 묻는 것'과 같아, 학교(DB) 운영을 마비시킬 수 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (상관 쿼리 최적 활용 시)

| 지표 | 활용 미흡 시 | 최적 활용 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **복잡 비즈니스 구현** | 구현 불가 또는 복잡 | **SQL 한 문장으로 해결** | 개발 민첩성 50% 향상 |
| **데이터 정확도** | 중복 결과 발생 가능 | **정교한 행 단위 필터링** | 분석 품질 신뢰도 극대화 |
| **쿼리 리팩토링** | 하드코딩 빈번 | **선언적 관계 표현** | 코드 가독성 및 유지보수성 향상 |

### 2. 미래 전망
- **AI-Rewriting Engine**: 사람이 짠 복잡한 상관 서브쿼리를 AI가 실시간으로 분석하여, 성능이 최적화된 **CTE나 Window Function**으로 자동 변환하여 실행하는 지능형 엔진 등장.
- **In-memory Correlation Cache**: 상관 쿼리의 중간 결과값을 메모리 내 벡터 엔진이 초고속 캐싱하여, 반복 실행 부하 자체를 0에 가깝게 만드는 하드웨어 가속 기술 발전.

- **📢 섹션 요약 비유**: 미래의 상관 서브쿼리는 우리가 일일이 질문을 던지지 않아도, 데이터들끼리 서로의 맥락을 알아보고 알아서 짝을 맺어주는 '지능형 연결망'이 될 것입니다.
