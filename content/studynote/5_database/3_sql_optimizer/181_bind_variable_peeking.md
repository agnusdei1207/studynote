# 181. 바인드 변수 피킹 (Bind Variable Peeking) - 변수 값에 따른 실행 계획의 명암

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SQL이 처음 실행될 때(하드 파싱), 옵티마이저가 바인드 변수(`:var`)에 할당된 실제 값을 몰래 훔쳐보고(Peeking), 그 값의 데이터 분포를 바탕으로 최적의 실행 계획을 결정하는 기술이다.
> 2. **딜레마**: 첫 번째 값에 최적화된 실행 계획이 메모리(Library Cache)에 공유되면서, 데이터 분포가 전혀 다른 다른 값이 들어올 때도 부적절한 계획을 강제 적용하게 되어 성능 저하를 유발할 수 있다.
> 3. **가치**: 대다수 균등한 데이터 분포에서는 파싱 부하를 획기적으로 줄여주지만, 데이터 편중(Skewness)이 심한 컬럼에서는 양날의 검이 되므로 **적응형 커서 공유(ACS)** 등의 보완 기술과 함께 신중히 다뤄야 한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**바인드 변수 피킹 (Bind Variable Peeking)**은 옵티마이저의 '호기심'입니다. 원래 바인드 변수는 어떤 값이 들어올지 모르는 상태에서 실행 계획을 짜야 하지만, 처음 딱 한 번만 변수 안의 알맹이를 확인하여 "아, 이런 데이터가 주로 들어오겠구나"라고 추측하는 기법입니다.

### 💡 비유: 첫 손님의 입맛으로 정한 '오늘의 메뉴'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        바인드 변수 피킹 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 한 식당(DB)에서 손님이 올 때마다 "좋아하는 거(Bind Var)"를 요리해줌.      │
│                                                                             │
│  1. 피킹 발생 (첫 손님):                                                      │
│     - 첫 손님이 "매운 거요!"라고 함. ──▶ 주방장(Optimizer)이 '매운 떡볶이'를      │
│       만드는 최적의 경로(실행 계획)를 짜고 레시피를 게시판에 붙임.                 │
│                                                                             │
│  2. 문제 발생 (두 번째 손님):                                                 │
│     - 다음 손님은 "안 매운 거요!"라고 함.                                       │
│     - 그런데 주방장은 게시판 레시피를 보고 **'매운 떡볶이'** 방식 그대로 요리함.    │
│     - 두 번째 손님은 입맛에 안 맞아 화가 남(성능 저하)!                          │
│                                                                             │
│  → 첫 번째로 들어온 값에 따라 **'전체 시스템의 운명이 결정'**되는 복불복 시스템!    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **파싱 오버헤드 감소**: 매번 값을 보고 쿼리를 새로 짜면(Hard Parsing) CPU 가 견디지 못하므로, 한 번 짠 계획을 재사용하고 싶은 니즈.
- **통계 정보 활용의 한계**: 바인드 변수만 쓰면 옵티마이저가 히스토그램(데이터 분포)을 전혀 쓰지 못해 바보가 되는 것을 방지하기 위해 도입.

- **📢 섹션 요약 비유**: 바인드 변수 피킹은 '예고편만 보고 영화 전체 내용을 짐작하는 관객'과 같아서, 가끔은 대박을 치지만 가끔은 낚이기도 합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 피킹의 작동 메커니즘 단계

| 단계 | 수행 활동 | 상태 |
|:---:|:---|:---|
| **1. 최초 실행** | SQL이 처음 들어옴. 바인드 변수의 실제 값(Literal)을 읽음. | **Hard Parsing** |
| **2. 계획 수립** | 읽은 값을 바탕으로 히스토그램을 참조하여 최적 경로 선정. | **Optimization** |
| **3. 캐시 등록** | 생성된 계획을 라이브러리 캐시에 변수와 함께 보관. | **Caching** |
| **4. 이후 실행** | 변수 값이 바뀌어도 캐시된 계획을 그대로 꺼내 씀. | **Soft Parsing** |

### 2. 피킹으로 인한 성능 변동 도해 (ASCII)

데이터 분포 불균형(Skew) 상황에서의 위험성을 시각화한다.

```text
    [ Table: Member (Status: 99% Active, 1% Dormant) ]
    [ Index: Status_Idx ]
    
    Query: SELECT * FROM Member WHERE Status = :v1;
    
    (A) 최초 입력 :v1 = 'Dormant' (1%)
        - Optimizer: "오, 데이터가 아주 적네? Index 가 최고지!"
        - Plan: **INDEX RANGE SCAN** (Saved in Cache)
    
    (B) 다음 입력 :v1 = 'Active' (99%)
        - Optimizer: "게시판에 INDEX 쓰라고 되어있네? 그대로 하자!"
        - 현실: 99% 데이터를 인덱스로 하나씩 읽느라 **시스템 장애 발생!**
```

### 3. 심층 동작 원리: 히스토그램과의 충돌
- **원리**: 옵티마이저는 컬럼의 값이 한쪽에 쏠려 있는지(Skew)를 히스토그램으로 알고 있음.
- **충돌**: 피킹된 값이 히스토그램의 어느 구간에 속하느냐에 따라 실행 계획이 'Index'와 'Full Scan' 사이를 극단적으로 오가게 됨.

### 4. 핵심 기술: 적응형 커서 공유 (ACS, Adaptive Cursor Sharing)
- 피킹의 한계를 극복하기 위해, 실행 시 실제 처리된 행 수를 관찰했다가 예상과 너무 다르면 새로운 실행 계획을 하나 더 만들어 관리하는 현대적 보완 기술.

- **📢 섹션 요약 비유**: ACS는 '손님의 입맛이 달라지면 게시판 레시피를 떼고 새로 적어주는 눈치 빠른 주방장'입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 리터럴 SQL vs 바인드 SQL vs 바인드 피킹

| 비교 항목 | Literal SQL (값 직접 입력) | Bind SQL (변수 사용) | Bind Peeking (피킹 적용) |
|:---:|:---|:---|:---|
| **파싱 방식** | 매번 Hard Parsing | 소프트 파싱 위주 | 최초 1회 하드 파싱 |
| **최적화** | **값마다 최적의 계획** | 평균적인 계획 수립 | **최초 값에 종속된 계획** |
| **CPU 부하** | **매우 높음** | 매우 낮음 | 낮음 |
| **성능 일관성** | 우수 (느려도 정확) | 보통 (안전빵) | **위험 (복불복)** |

### 2. 과목 융합 관점: 운영체제 및 확률론
- **운영체제(OS) 융합**: 라이브러리 캐시는 OS의 **Shared Memory** 개념이며, 여기서 발생하는 경합(Latch)은 프로세스 간의 임계 구역(Critical Section) 동기화 문제와 동일하다.
- **확률론 융합**: 옵티마이저는 바인드 변수의 '기대값'을 계산할 때 **균등 분포(Uniform Distribution)**를 가정하지만, 피킹은 특정 표본(Sample) 하나로 전체를 단정 짓는 **표본 편향(Sampling Bias)**의 위험을 안고 있다.

- **📢 섹션 요약 비유**: 리터럴은 '매일 새 옷을 맞춰 입는 것'이고, 바인드는 '단복 하나로 버티는 것'이며, 피킹은 '첫날 날씨에 맞춰 단복을 골라버리는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 배치 프로그램의 '실행 순서'에 따른 성능 편차
- **상황**: 새벽에 도는 배치 쿼리가 어떤 날은 1분, 어떤 날은 1시간이 걸림.
- **판단**: 배치가 처음 시작될 때 들어온 날짜나 구분값이 아주 특수한(데이터가 거의 없는) 경우였음.
- **결단**: **바인드 피킹 비활성화 또는 힌트 강제**. 특정 컬럼의 편중이 심할 경우 `_optim_peek_user_binds = FALSE` 옵션으로 피킹을 끄거나, 아예 `/*+ FULL(t) */` 힌트를 주어 변수 값에 상관없이 안전한 경로를 가게 함.

### 2. 안티패턴: 중요 검색 조건에 바인드 변수만 맹신
- **증상**: 분포가 극심하게 다른 컬럼(예: 공지사항 고정글 여부)에 바인드 변수를 쓰고 피킹이 알아서 해주길 바람.
- **결과**: 고정글 조회할 때는 빠른데, 일반글 조회할 때 시스템 전체 멈춤.
- **조치**: **분포도가 극단적인 핵심 컬럼은 리터럴(Literal) 사용 검토**. 하드 파싱 부하가 있더라도 정확한 계획이 수명보다 소중한 경우가 있음을 인지할 것.

### 📢 기술사적 결언
> "바인드 변수 피킹은 **'편의성과 정확성의 위험한 거래'**다. 시스템 전체의 CPU를 아끼기 위해 개별 쿼리의 최적화 확률을 희생하기 때문이다. 기술사는 피킹이 가져다주는 **소프트 파싱의 달콤함** 뒤에 숨은 **실행 계획 왜곡의 쓴맛**을 미리 예견할 수 있어야 한다. 데이터의 성격에 따라 리터럴, 바인드, 그리고 힌트라는 세 가지 카드를 적절히 섞어 쓰는 **'실행 계획 설계자'**가 되어야 한다."

- **📢 섹션 요약 비유**: 피킹은 '눈치'와 같아서, 눈치가 빠르면 절에서 새우젓(성능)을 얻어먹지만 눈치를 잘못 채면 뺨(장애)을 맞게 됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (피킹 및 ACS 관리 시)

| 지표 | 도입 전 | 도입 후 (관리) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **CPU 파싱 점유율** | 50% 이상 (Hard) | **5% 미만 (Soft)** | 서버 가용성 10배 확보 |
| **쿼리 성능 편차** | 예측 불가 | **안정적 / 적응형 대응** | 서비스 지연 시간 감소 |
| **애플리케이션 가독성** | 문자열 노가다 (Literal) | **깔끔한 변수 처리** | 코드 품질 향상 |

### 2. 미래 전망
- **AI-Managed Plan Baselines**: AI가 모든 바인드 변수의 분포를 미리 학습하여, 피킹 없이도 입력값에 따른 최적 계획 리스트를 미리 준비해두는 **지능형 쿼리 매핑**.
- **Real-time Feedback Tuning**: 쿼리 실행 직후 지연이 발생하면 즉시 피킹된 정보를 무효화하고 재최적화하는 **자가 치유형 옵티마이저** 기술의 일반화.

- **📢 섹션 요약 비유**: 미래의 피킹은 '몰래 훔쳐보는 것'이 아니라, '상대방의 마음을 미리 읽어 가장 잘 어울리는 옷을 골라주는' 배려 깊은 지능이 될 것입니다.

2. 만약 친구가 주머니에 사탕을 들고 있으면 "아, 사탕을 좋아하나 보다!" 하고 매번 사탕만 준비하게 되죠.
3. 하지만 친구가 나중에 초콜릿을 원할 수도 있어서, **처음 한 번 본 걸로만 판단하면 안 된다는 것**을 꼭 기억해야 한답니다!
