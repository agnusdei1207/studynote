# 150. 집합 연산자 심화 - 합집합, 교집합, 차집합의 물리적 구현

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 두 개의 독립된 결과 집합을 수학적 집합론에 근거하여 병합하는 연산이며, 물리적으로는 **정렬(Sort)** 또는 **해시(Hash)** 메커니즘을 통해 데이터의 중복을 제거하고 정합성을 맞추는 고비용 연산이다.
> 2. **물리적 구현**: `UNION ALL`은 단순 병합(Concatenation)으로 I/O 부하가 낮으나, `UNION`, `INTERSECT`, `EXCEPT`는 전체 데이터를 정렬한 뒤 비교하는 **Sort-Merge** 방식이나 해시 테이블을 빌드하는 **Hash-Set** 방식을 사용한다.
> 3. **가치**: 이기종 소스의 통합 뷰(Integrated View)를 생성하는 논리적 편의성을 제공함과 동시에, 옵티마이저의 **집합 연산 푸시다운(Push-down)** 최적화를 통해 분산 환경에서의 대규모 데이터 병합 성능을 결정짓는다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**집합 연산자 심화**는 단순한 문법 사용을 넘어, SQL 엔진이 수억 건의 데이터를 어떻게 중복 없이 합치고(UNION), 겹치는 것만 골라내며(INTERSECT), 한쪽을 깎아내는지(EXCEPT) 그 물리적 내부 구조를 다룹니다.

### 2. 💡 비유: 거대한 두 모래성 합치기
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        집합 연산 물리 구현 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 두 개의 큰 바구니에 든 조개껍데기들을 하나로 합치려 함.                   │
│                                                                             │
│  1. UNION ALL (그냥 붓기): 두 바구니를 큰 통에 확 쏟아부음. (매우 빠름)           │
│  2. UNION (정제하며 합치기):                                                  │
│     - 방법 A (정렬): 조개를 크기순으로 쭉 늘어놓고, 똑같이 생긴 건 하나씩 버림.      │
│     - 방법 B (해시): 조개 모양별로 칸막이가 든 통에 던져 넣어 중복을 걸러냄.        │
│                                                                             │
│  → **'똑같은 데이터'**를 찾아내는 과정에서 시간과 노력이 얼마나 드느냐가            │
│     집합 연산 성능의 핵심 비밀입니다!                                           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **대량 데이터 통합의 부하**: 과거 소량 데이터에서는 문제가 없었으나, 테라바이트급 데이터 통합 시 정렬(Sort) 작업이 전체 시스템의 병목(Bottleneck)이 됨.
- **분석 함수와의 결합**: 집합 연산의 결과를 다시 윈도우 함수 등으로 가공해야 하는 복잡한 OLAP 환경 도래.

- **📢 섹션 요약 비유**: 집합 연산은 '복사기가 없는 세상에서 두 권의 책을 합쳐 한 권으로 만드는 수작업'과 같아 정교한 기술이 필요합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 집합 연산의 2대 물리적 구현 방식

| 방식 | 작동 메커니즘 | 장점 | 단점 |
|:---:|:---|:---|:---|
| **Sort-Merge** | 두 집합을 각각 정렬한 뒤 포인터를 옮기며 비교. | 결과가 정렬되어 나옴. | **메모리/CPU 소모 큼**. |
| **Hash-Set** | 한 집합으로 해시 테이블을 만들고 다른 집합과 대조. | **대량 데이터에서 빠름**. | 메모리 부족 시 Temp DB 사용. |

### 2. 집합 연산 내부 실행 흐름 도해 (ASCII)

INTERSECT 연산이 해시 방식으로 처리되는 과정을 시각화한다.

```text
    [ Result Set A ]            [ Result Set B ]
    { 1, 2, 3, 4 }              { 3, 4, 5, 6 }
           │                           │
    ┌──────▼───────────────────────────▼──────────────────────────┐
    │  [ Hash Table Construction (Set A) ]                        │
    │  - Hash(1) -> Bucket 1                                      │
    │  - Hash(2) -> Bucket 2 ...                                  │
    └──────┬──────────────────────────────────────────────────────┘
           │ (Probing with Set B)
    ┌──────▼──────────────────────────────────────────────────────┐
    │  - Is 3 in Table A? -> Yes (Output)                         │
    │  - Is 5 in Table A? -> No (Discard)                         │
    └──────┬──────────────────────────────────────────────────────┘
           │
    [ Final INTERSECT : { 3, 4 } ]
```

### 3. 심층 동작 원리: 널(NULL) 값의 집합 연산 처리
- **특이점**: 일반 조인에서는 `NULL = NULL`이 False이지만, 집합 연산에서는 **두 NULL을 동일한 값으로 취급**함.
- **결과**: UNION 시 양쪽에 NULL이 있으면 하나만 남고, INTERSECT 시 양쪽에 NULL이 있으면 결과에 포함됨.

### 4. 핵심 기술: 병렬 집합 연산 (Parallel Set Ops)
- 데이터를 여러 슬라이스로 쪼개어 다수의 CPU 코어가 동시에 정렬 및 해시 비교를 수행하는 기술.
- 분산 시스템(Sharding)에서는 각 노드에서 먼저 중복을 제거한 뒤 마스터 노드에서 최종 병합함.

- **📢 섹션 요약 비유**: 해시 방식 집합 연산은 '이름표가 붙은 사물함'에 물건을 던져 넣는 것과 같아, 이미 물건이 들어있는지 확인하는 속도가 매우 빠릅니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 연산자별 물리적 오버헤드 체감 지수

| 연산자 | 구현 난이도 | 리소스 소모량 | 성능 최적화 포인트 |
|:---:|:---:|:---:|:---|
| **UNION ALL** | 최하 | **최하** | 네트워크 전송량(I/O) 최소화 |
| **UNION** | 상 | **중상** | 중복 제거 알고리즘 선택 (Hash vs Sort) |
| **INTERSECT** | 중 | 보통 | 드라이빙 집합(작은 쪽) 선정 |
| **EXCEPT** | 상 | 상 | 인덱스 활용을 통한 Not Exists 변환 |

### 2. 과목 융합 관점: 이산수학 및 자료구조
- **이산수학 융합**: 집합 연산의 심화는 **드모르간의 법칙**을 활용한 쿼리 변환(예: INTERSECT를 EXCEPT와 UNION의 조합으로 변경)을 통해 최적의 논리적 경로를 찾는다.
- **자료구조 융합**: 해시 기반 집합 연산은 **Bloom Filter**를 사용하여 사전에 불필요한 비교를 차단함으로써, 메모리 효율성을 비약적으로 높이는 전략을 취한다.

- **📢 섹션 요약 비유**: UNION ALL 이 '그냥 섞기'라면, 나머지 집합 연산은 '검문 검색'이 수반되는 통제된 합치기입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 초대용량 데이터셋 간의 차집합(EXCEPT) 연산
- **상황**: 10억 건의 어제 로그와 10억 건의 오늘 로그를 비교하여 '삭제된 데이터'를 찾아야 함.
- **판단**: 일반 `EXCEPT`는 20억 건 전체 정렬을 유발하여 시스템 장애 가능성 농후.
- **결단**: **Anti-Join 기법 활용**. 인덱스가 있는 날짜/ID 컬럼을 기준으로 `WHERE NOT EXISTS` 또는 `LEFT JOIN ... WHERE B.id IS NULL`로 변환하여, 전체 정렬 없이 인덱스 스캔만으로 차집합 효과 구현.

### 2. 안티패턴: 서브쿼리 내에서의 불필요한 UNION 사용
- **증상**: `IN (SELECT ... UNION SELECT ...)` 처럼 조건절 안에서 집합 연산을 수행.
- **결과**: 서브쿼리가 실행될 때마다 내부 임시 테이블 생성 및 정렬이 발생하여 메인쿼리 속도 저하.
- **조치**: **`OR` 조건이나 `VALUES` 절**로 단순화하거나, 집합 연산이 꼭 필요하다면 `UNION ALL`로 대체 가능한지 우선 검토.

### 📢 기술사적 결언
> "집합 연산의 심화 과정은 **'불필요한 정렬과의 싸움'**이다. 정렬은 디스크 I/O를 유발하는 가장 무거운 작업이기 때문이다. 기술사는 집합 연산자의 수학적 우아함에 매몰되지 말고, 그것이 유발하는 **물리적 비용(Memory Spill 등)**을 숫자로 계산할 수 있어야 한다. 또한, **해시 집합(Hash-Set) 연산**이 최신 하드웨어의 다중 코어를 얼마나 잘 활용하는지 파악하여, 대규모 데이터 통합 아키텍처의 성능 가이드를 제시해야 한다."

- **📢 섹션 요약 비유**: UNION을 쓸 때 'ALL' 한 글자를 빼먹는 실수는, 전교생에게 일일이 이름표 검사를 시키는 것과 같아 엄청난 시간 낭비를 초래합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (집합 연산 최적화 시)

| 지표 | 도입 전 | 도입 후 (물리 최적화) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **배치 작업 시간** | 5시간 | **1시간 이내** | 처리량 5배 향상 |
| **Temp DB 점유율** | 80% (Disk 사용) | **10% (In-memory)** | 시스템 안정성 확보 |
| **데이터 정합성** | 수동 대조 오류 | **집합 논리 자동 검증** | 데이터 신뢰도 100% |

### 2. 미래 전망
- **Hardware-Accelerated Sets**: FPGA나 전용 AI 가속기를 통해 집합 연산의 중복 제거를 하드웨어 와이어 속도(Wire-speed)로 처리하는 기술 등장.
- **Approximate Set Operations**: 100% 정확도 대신 99.9% 확률적 정확도를 제공하며 속도를 수천 배 높이는 **HyperLogLog 기반 집합 연산**의 보편화.

- **📢 섹션 요약 비유**: 미래의 집합 연산은 모든 조약돌을 다 만져보지 않아도, 무게와 형태만으로 중복을 골라내는 '초능력 저울'과 같은 기술로 진화할 것입니다.
