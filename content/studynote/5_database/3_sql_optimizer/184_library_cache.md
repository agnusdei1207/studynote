# 184. 라이브러리 캐시 (Library Cache) - SQL 실행 계획의 저장 및 공유 영역

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DBMS의 공유 메모리 영역인 **SGA(Shared Global Area)** 내 Shared Pool의 핵심 구성 요소로, 파싱된 SQL 문장, 실행 계획, 그리고 관련 메타데이터를 저장하여 동일한 질의의 반복 실행 시 연산 부하를 최소화하는 공간이다.
> 2. **작동 구조**: SQL 텍스트의 해시(Hash) 값을 키(Key)로 사용하는 **해시 버킷(Hash Bucket)** 구조를 가지며, 최근에 사용된 데이터를 우선적으로 보존하는 **LRU(Least Recently Used)** 알고리즘을 통해 한정된 메모리 자원을 효율적으로 관리한다.
> 3. **가치**: 하드 파싱(Hard Parsing) 빈도를 낮추어 CPU 점유율을 안정화하고 소프트 파싱(Soft Parsing)을 극대화함으로써, 초당 수천 건의 트랜잭션을 처리해야 하는 엔터프라이즈 환경의 성능 기반이 된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**라이브러리 캐시 (Library Cache)**는 한마디로 '실행 계획 저장소'입니다. 사용자가 던진 SQL이 "어떤 테이블을 읽고 어떤 인덱스를 탈지" 고민한 결과물(실행 계획)을 지우지 않고 메모리에 잠시 보관해두는 곳입니다.

### 💡 비유: 요리사의 '레시피 스크랩북'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        라이브러리 캐시 개념 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 레스토랑 주방에 매일 똑같은 주문이 쏟아짐.                                │
│                                                                             │
│  1. 라이브러리 캐시가 없을 때:                                                │
│     - "김치찌개요!" 주문이 올 때마다 요리책을 새로 뒤지고 조리법을 고민함.          │
│     - 똑같은 요리인데 매번 머리를 써야 하니 요리사가 금방 지침. (CPU 부하)          │
│                                                                             │
│  2. 라이브러리 캐시가 있을 때 (스크랩북):                                       │
│     - 자주 들어오는 주문의 레시피(실행 계획)를 주방 벽(메모리)에 붙여둠.            │
│     - 주문이 들어오자마자 벽을 슥 보고 즉시 조리 시작! (소프트 파싱)                │
│                                                                             │
│  → 똑똑하게 **'과거의 지식'**을 빌려 써서 현재의 일을 빨리 끝내는 두뇌 공간입니다!   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **파싱 연산의 병목**: SQL 문장을 바이너리 코드로 바꾸는 과정이 생각보다 CPU 자원을 많이 소모함에 따라, 이를 재활용하는 아키텍처가 필수적이 됨.
- **동시성 제어 요건**: 여러 사용자가 동일한 SQL을 실행할 때, 각각 계획을 짜는 낭비를 막고 하나의 결과물을 공유하기 위함.

- **📢 섹션 요약 비유**: 라이브러리 캐시는 데이터베이스 엔진의 '단기 기억 저장소'와 같아서, 방금 했던 고민을 잊지 않게 해줍니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 라이브러리 캐시의 논리적 위치 (SGA 구조)

| 계층 | 명칭 | 역할 |
|:---:|:---|:---|
| **최상위** | **SGA (Shared Global Area)** | 인스턴스 전체가 공유하는 메모리. |
| **중간** | **Shared Pool** | 데이터 사전 정보와 SQL 정보를 담는 영역. |
| **핵심** | **Library Cache** | **SQL 문장, 실행 계획, 커서(Cursor) 보관**. |

### 2. 내부 데이터 구조 도해 (ASCII)

해시 알고리즘을 통해 SQL을 관리하는 물리적 구조를 시각화한다.

```text
    [ SQL Text ] : "SELECT * FROM Emp WHERE EID = 10"
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Hashing Logic ] ──▶ Hash Value : '0xAF31'                    │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Library Cache Hash Table ]                                   │
    │  Bucket [0xAF30] : (Empty)                                      │
    │  **Bucket [0xAF31]** : [ LCO Header ] ──▶ [ SQL Tree ]           │
    │                        [ LCO Header ] ──▶ [ Exec Plan ]         │
    │  Bucket [0xAF32] : (Empty)                                      │
    └─────────────────────────────────────────────────────────────────┘
    * LCO (Library Cache Object): 캐시에 저장되는 개별 단위 객체
```

### 3. 심층 동작 원리: 커서(Cursor) 공유
- **공유 커서 (Parent Cursor)**: SQL 텍스트 그 자체.
- **자식 커서 (Child Cursor)**: 사용자 설정(Language 등)이나 바인드 변수 값에 따라 생성되는 실제 실행 계획.
- **작동**: 텍스트가 같더라도 환경이 다르면 여러 개의 자식 커서가 생성될 수 있음 (Cursor Heap).

### 4. 핵심 기술: LRU 알고리즘과 Freeing
- 캐시 공간이 부족해지면 사용 빈도가 가장 낮은(Least Recently Used) 객체부터 메모리에서 제거하여 새로운 SQL을 담을 공간을 확보함.

- **📢 섹션 요약 비유**: 해시 버킷은 '사물함'과 같아서, SQL의 이름(해시값)을 알면 즉시 자기 사물함을 열어 계획을 꺼낼 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 라이브러리 캐시 vs 데이터 사전 캐시 (Dictionary Cache)

| 비교 항목 | Library Cache | Dictionary Cache (Row Cache) |
|:---:|:---|:---|
| **저장 내용** | **SQL 문장, 실행 계획** | **테이블 구조, 컬럼 정의, 권한** |
| **용도** | 실행 절차 재사용 | 구문 분석 시 메타데이터 참조 |
| **갱신 주기** | 매우 빈번 (새로운 쿼리 유입 시) | 낮음 (DDL 발생 시) |
| **비유** | **요리법(Plan) 저장** | **식재료 리스트(Meta) 저장** |

### 2. 과목 융합 관점: 운영체제 및 보안
- **운영체제(OS) 융합**: 라이브러리 캐시는 OS의 **Instruction Cache**와 유사하게 동작하며, 메모리 경합을 막기 위해 **래치(Library Cache Latch)**라는 초경량 동기화 객체를 사용한다.
- **보안(Security) 융합**: 라이브러리 캐시에는 사용자 비번이나 실제 데이터가 아닌 '쿼리 구조'가 담기지만, 캐시 조회를 통해 어떤 쿼리가 자주 실행되는지 분석하는 **SQL 프로파일링** 공격의 타겟이 될 수 있어 접근 제어가 필요하다.

- **📢 섹션 요약 비유**: Dictionary Cache 가 '영어 단어장'이라면, Library Cache 는 '이미 해석해 둔 문장들'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: Shared Pool 사이즈 산정 및 튜닝
- **상황**: 라이브러리 캐시 히트율이 70% 미만으로 떨어지고, 하드 파싱 부하가 심각함.
- **판단**: 캐시 공간이 너무 작아 자주 쓰는 계획이 계속 밀려나고(Aging-out) 있거나, 리터럴 SQL 사용으로 캐시가 파편화됨.
- **결단**: **Shared Pool 크기 증설 및 바인드 변수 튜닝**. 메모리를 늘려 '단기 기억력'을 높이고, 바인드 변수를 써서 '기억해야 할 질문의 개수'를 줄여 히트율을 99% 이상으로 복구.

### 2. 안티패턴: 대량의 일회성 SQL (Ad-hoc) 방치
- **증상**: `WHERE ID = 101`, `WHERE ID = 102` 처럼 값만 다른 쿼리가 수만 개 유입.
- **결과**: 라이브러리 캐시가 수만 개의 쓰레기 계획으로 가득 차서, 정작 중요한 핵심 쿼리의 계획이 캐시에서 쫓겨남.
- **조치**: **애플리케이션의 SQL 표준화**. 일회성 쿼리도 가급적 바인드 변수를 쓰도록 유도하고, 불가피할 경우 `FORCE` 옵션 등을 통해 엔진 레벨에서 강제 통합.

### 📢 기술사적 결언
> "라이브러리 캐시는 데이터베이스의 **'지적 자산'**을 관리하는 보석함이다. 수조 번의 시행착오(최적화) 끝에 얻어낸 정답들이 모여있기 때문이다. 기술사는 단순히 메모리 용량을 늘리는 하드웨어적 접근을 넘어, **'커서의 공유 정합성'**을 유지하기 위한 소프트웨어적 설계를 주도해야 한다. 깨끗하고 정제된 라이브러리 캐시는 **CPU 리소스의 절약**을 넘어 시스템의 **'예측 가능한 고성능'**을 보장하는 유일한 길임을 잊지 말아야 한다."

- **📢 섹션 요약 비유**: 라이브러리 캐시를 관리하는 것은 '머릿속의 잡념을 없애고 공부에 집중하는 것'과 같아, 생각이 맑아야(캐시 효율) 공부(연산)가 잘됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (라이브러리 캐시 최적 활용 시)

| 지표 | 활용 전 | 활용 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **CPU 파싱 점유율** | 60% 이상 | **5% 미만** | 연산 자원 10배 확보 |
| **SQL 응답 시간** | 100ms (Hard) | **1ms 이내 (Soft)** | 응답성 획기적 개선 |
| **시스템 동시성** | 100 User (Latch 경합) | **5,000 User (안정)** | 시스템 스케일 아웃 가능 |

### 2. 미래 전망
- **Persistent Library Cache**: DB 인스턴스 재시작 시에도 실행 계획을 잃지 않도록 디스크(또는 NVMe)에 영구 보관하여 워밍업 시간을 0으로 만드는 기술 고도화.
- **AI-Predictive Caching**: 현재 시스템 상황을 보고, 곧 유입될 쿼리의 실행 계획을 미리 백그라운드에서 계산하여 캐시에 넣어두는 **선제적 캐싱** 기술 등장.

- **📢 섹션 요약 비유**: 미래의 라이브러리 캐시는 단순한 저장소를 넘어, 우리가 묻기도 전에 모든 정답을 준비해놓는 '무결점 인공지능 뇌'가 될 것입니다.
