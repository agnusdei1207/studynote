# 188. SQL 계획 베이스라인 (SQL Plan Baseline) - 실행 계획 고정과 안정성 관리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 옵티마이저가 수립한 실행 계획 중 성능이 검증된 것들만을 별도의 저장소에 등록하고, 향후 동일 쿼리 요청 시 **검증된 계획(Baseline) 내에서만 실행**되도록 강제하는 실행 계획 관리 체계다.
> 2. **작동 메커니즘**: 새로운 실행 계획이 발견되더라도 즉시 적용하지 않고 **'진화(Evolve)'** 단계를 통해 기존 베이스라인보다 성능이 우수함이 증명될 때만 교체하여, 갑작스러운 실행 계획 변경으로 인한 성능 퇴보(Regression)를 원천 차단한다.
> 3. **가치**: DB 버전 업그레이드, 통계 정보 갱신, 하드웨어 변경 등 환경 변화 속에서도 비즈니스 핵심 쿼리의 **성능 안정성**을 보장하며, 튜닝된 계획을 영구적으로 보존하는 아키텍처적 방어막 역할을 한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**SQL 계획 베이스라인 (SQL Plan Baseline)**은 옵티마이저의 판단을 기록하고 통제하는 '블랙박스'이자 '승인 리스트'입니다. 엔진이 멋대로 계획을 바꾸지 못하게, "내가 허락한 길로만 다녀라"라고 명령을 내리는 장치입니다.

### 💡 비유: 맛집의 '검증된 레시피' 관리
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        SQL 계획 베이스라인 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 손님들에게 극찬받는 '황금 레시피(Baseline)'가 있는 식당임.               │
│                                                                             │
│  1. 베이스라인이 없을 때:                                                     │
│     - 주방장(Optimizer)이 매일 새로운 요리법을 실험함.                           │
│     - 가끔은 더 맛있을 수도 있지만, 어느 날 폭망한 요리가 나와 손님들이 다 끊김.    │
│                                                                             │
│  2. 베이스라인이 있을 때 (레시피 고정):                                         │
│     - "검증된 황금 레시피 리스트"를 벽에 딱 붙여둠.                              │
│     - 주방장이 새로운 요리법(New Plan)을 개발해도, 시식단(Evolve)이 먹어보고       │
│       황금 레시피보다 맛있다고 인정할 때까지는 절대 손님에게 내놓지 않음!          │
│                                                                             │
│  → 환경이 바뀌어도 **'언제나 변함없는 최고의 맛(성능)'**을 보장하는 장치입니다!     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **실행 계획의 변덕(Instability)**: 통계 정보가 살짝 바뀌거나 인덱스가 추가되었을 때, 옵티마이저가 갑자기 멀쩡하던 쿼리를 Full Scan으로 바꿔버리는 사고 빈번.
- **업그레이드 리스크**: DB 버전을 올렸을 때 옵티마이저 알고리즘이 변하여 기존 쿼리들이 느려지는 현상을 막기 위한 보호책 필요.

- **📢 섹션 요약 비유**: 베이스라인은 '잘 닦인 고속도로'를 지도에 고정해두어, 옵티마이저가 갑자기 샛길(나쁜 계획)로 빠지지 않게 막는 가드레일입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. SQL 계획 관리 (SPM)의 3단계 라이프사이클

| 단계 | 명칭 | 수행 활동 | 비유 |
|:---:|:---|:---|:---|
| **1단계** | **Capture (수집)** | 실행되는 계획들을 베이스라인 후보로 등록. | 메뉴 개발 |
| **2단계** | **Selection (선택)** | 등록된 계획 중 **Accepted** 된 것만 실제 사용. | 메뉴 확정 |
| **3단계** | **Evolution (진화)** | 새로운 계획의 성능을 테스트하여 우수하면 승인. | 신메뉴 검증 |

### 2. 베이스라인 기반 쿼리 실행 프로세스 도해 (ASCII)

옵티마이저의 판단과 베이스라인의 대조 과정을 시각화한다.

```text
    [ SQL Request ] ──▶ [ Optimizer : Generate Plan X ]
                               │
    ┌──────────────────────────▼──────────────────────────────────────┐
    │  [ SQL Plan Baseline Check ]                                    │
    │  - "이 SQL에 대해 저장된 베이스라인이 있는가?"                     │
    └──────────────┬───────────────────────────┬──────────────────────┘
                   │ [ YES ]                   │ [ NO ]
                   ▼                           ▼
    ┌──────────────────────────────┐    ┌─────────────────────────────┐
    │  Plan X 가 **'Accepted'** 인가? │    │  Plan X를 즉시 실행하고      │
    ├──────────────┬───────────────┤    │  후보로 등록함. (Capture)    │
    │ [ YES ]      │ [ NO ]        │    └─────────────────────────────┘
    ▼              │               │
 [ Execute X ]     │               ▼
                   └─────▶ [ Search other Accepted Plans ]
                           - 베이스라인 중 가장 저렴한 계획 실행
                           - Plan X는 '비승인' 상태로 기록 (Evolution 대기)
```

### 3. 심층 동작 원리: 계획의 고정 (Fixing Plans)
- **원리**: 특정 계획에 `FIXED = YES` 속성을 부여하면, 옵티마이저는 베이스라인 내의 다른 계획들조차 무시하고 오직 그 계획만 사용함.
- **효과**: 힌트(`HINT`)를 코드에 직접 쓰지 않고도 시스템 운영 중에 특정 실행 경로를 강제할 수 있음.

### 4. 핵심 기술: 성능 비교 검증 (Evolve Task)
- 새로 발견된 계획이 기존 것보다 최소 1.5배 이상 성능이 좋아야만 승인해주는 등 수치 기반의 엄격한 교체 로직 적용.

- **📢 섹션 요약 비유**: 진화 단계는 '오디션'과 같아서, 기존 가수(계획)보다 압도적인 노래 실력을 보여줘야만 무대(실행)에 설 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. SQL Profile vs SQL Plan Baseline (SPB)

| 비교 항목 | SQL Profile | SQL Plan Baseline (SPB) |
|:---:|:---|:---|
| **작동 방식** | 통계 정보의 오류를 보정하는 힌트 셋. | **실행 계획 자체를 고정**하는 리스트. |
| **유연성** | 상대적으로 유연함 (경로 변동 가능). | 보수적이고 엄격함 (경로 고정). |
| **보수성** | 환경 변화에 민감함. | **환경 변화에 극도로 안정적**. |
| **비유** | 안경 도수 맞추기 (잘 보이게) | **목적지까지 가는 길 고정** (안 헤매게) |

### 2. 과목 융합 관점: 소프트웨어 공학 및 운영체제
- **소프트웨어 공학(SE) 융합**: 베이스라인은 **회귀 테스트(Regression Testing)**의 자동화 버전이며, 코드가 변하지 않아도 성능이 변하는 '성능 회귀'를 방지하는 형상 관리 기술이다.
- **운영체제(OS) 융합**: CPU 스케줄링의 **고정 우선순위(Fixed Priority)** 정책과 유사하게, 특정 작업에 최적화된 자원 할당 경로를 고정하여 전체 시스템의 결정론적(Deterministic) 성능을 보장한다.

- **📢 섹션 요약 비유**: SQL 프로필이 '근시 교정'이라면, 베이스라인은 '철길 건설'입니다. 기차(쿼리)는 정해진 철길로만 달려야 탈선(장애)하지 않습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 대규모 DB 업그레이드 전후 성능 보존
- **상황**: Oracle 12c에서 19c로 업그레이드 예정. 수만 개의 쿼리 성능이 변할까 봐 걱정됨.
- **판단**: 새로운 엔진의 옵티마이저가 과거와 다른 계획을 짤 위험이 큼.
- **결단**: **업그레이드 전 베이스라인 전수 캡처**. 구 버전에서 잘 돌아가던 계획들을 베이스라인으로 몽땅 저장한 뒤 신 버전으로 이관. 업그레이드 직후에는 과거의 안전한 계획으로만 돌리고, 안정화된 후에 하나씩 '진화'를 통해 신 버전의 최적화 기능을 적용. 무중단/무장애 업그레이드 실현.

### 2. 안티패턴: 모든 SQL에 대한 무차별 베이스라인 적용
- **증상**: "성능이 중요하니까" 전 시스템의 모든 SQL을 베이스라인으로 묶어버림.
- **결과**: 데이터가 10건에서 10억 건으로 늘어나도 과거의 '인덱스 스캔' 계획만 고집하여 시스템 전체 성능 침몰. 옵티마이저의 지능을 완전히 마비시키는 자충수.
- **조치**: **핵심 TOP 쿼리 위주로 적용**. 데이터 분포 변화에 민감하지 않은 쿼리나, 절대로 계획이 바뀌면 안 되는 '생명줄 쿼리'에만 전략적으로 활용할 것.

### 📢 기술사적 결언
> "SQL 계획 베이스라인은 데이터베이스의 **'성능 보험'**이다. 예상치 못한 변화로부터 서비스의 품질을 지켜주기 때문이다. 기술사는 옵티마이저의 판단력을 존중하되, 그 결과가 비즈니스에 미치는 파급력을 통제할 수 있는 **'최종 승인권자'**가 되어야 한다. 베이스라인은 고여있는 물이 아닌, **진화(Evolve)**를 통해 끊임없이 정답을 갱신해나가는 '성장하는 지식 체계'로 운영되어야 함을 명심해야 한다."

- **📢 섹션 요약 비유**: 베이스라인을 쓰는 것은 '보험을 드는 것'과 같아, 평소엔 귀찮아도 위기의 순간(업그레이드 등)에 시스템을 살려냅니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (베이스라인 관리 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **성능 퇴보(Regression) 건수** | 월 평균 10건 | **0건** | 장애 대응 비용 100% 절감 |
| **운영 가용성** | 환경 변화 시 불안정 | **극도의 안정성** | 서비스 신뢰도 최상 유지 |
| **튜닝 자동화 수준** | 매번 수동 분석 | **자가 진화 및 검증** | DBA 생산성 50% 향상 |

### 2. 미래 전망
- **Cloud-Cloud Plan Migration**: 온프레미스에서 튜닝된 베이스라인을 클라우드 DB로 즉시 이관하여, 클라우드 전환 첫날부터 최상의 성능을 내는 **이식형 베이스라인** 기술 보편화.
- **AI-Managed Baselines**: 사람이 일일이 'Accepted' 버튼을 누르지 않아도, AI가 비즈니스 영향도를 판단하여 스스로 계획을 교체하는 **자율 운영 성능 관리**.

- **📢 섹션 요약 비유**: 미래의 성능 관리는 우리가 지도를 그리지 않아도, 데이터베이스가 스스로 '가장 빠르고 안전한 길'을 찾아내어 영원히 기억하는 '불멸의 지능'이 될 것입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **옵티마이저 (Optimizer)**: 베이스라인의 재료(계획)를 공급하는 자.
- **실행 계획 (Execution Plan)**: 베이스라인에 저장되는 알맹이.
- **바인드 변수 피킹**: 베이스라인과 협력하여 변수 부작용을 막는 파트너.
- **AWR / STS**: 베이스라인을 대량으로 수집하는 창고.

### 👶 어린이를 위한 3줄 비유 설명
1. **SQL 계획 베이스라인**은 학교 가는 길을 잃어버리지 않게 **"딱 한 가지 안전한 길"**만 지도에 표시해두는 약속이에요.
2. 새로운 지름길이 생겨도, 그 길이 진짜 안전한지 **선생님이 미리 가보신 뒤에만(검증)** 그 길로 갈 수 있게 해주죠.
3. 이 약속 덕분에 눈이 오나 비가 오나(환경 변화) 우리는 **언제나 늦지 않고 학교에** 도착할 수 있답니다!
