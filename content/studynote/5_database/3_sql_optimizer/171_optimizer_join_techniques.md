# 171. 옵티마이저 조인 기법 - NL, Hash, Sort Merge Join 비교

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 두 개 이상의 테이블을 결합하는 논리적 SQL 명령을 처리하기 위해, DBMS 엔진이 선택하는 **물리적 실행 알고리즘**이며 데이터의 양, 인덱스 유무, 정렬 상태에 따라 최적의 기법이 결정된다.
> 2. **3대 기법**: 순차적 루프를 도는 **Nested Loop(NL)**, 메모리에 해시 테이블을 생성하는 **Hash**, 정렬 후 병합하는 **Sort Merge** 조인이 있으며, 각각 온라인 트랜잭션(OLTP)과 대량 분석(OLAP) 환경에서 차별화된 강점을 갖는다.
> 3. **가치**: 조인 기법의 선택은 쿼리 성능의 임계치를 결정하는 가장 중요한 요소이며, 실행 계획 분석을 통해 부적절한 조인 방식을 식별하고 교정하는 것이 SQL 튜닝의 핵심 역량이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**조인 기법 (Join Method)**은 옵티마이저가 흩어진 데이터를 합치기 위해 사용하는 '도구'입니다. 사용자가 `JOIN`이라고만 쓰면, 옵티마이저는 상황에 맞춰 망치(NL), 전기톱(Hash), 혹은 대형 절단기(Sort Merge)를 꺼내어 작업을 수행합니다.

### 2. 💡 비유: 두 벌의 카드 덱에서 같은 숫자 찾기
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        조인 기법별 비유                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. Nested Loop (한 장씩 대조):                                               │
│     - 왼쪽 덱에서 카드 1장을 뽑고, 오른쪽 덱 전체를 다 뒤져서 짝을 찾음.            │
│     - 오른쪽 덱이 가나다순으로 정리(Index)되어 있으면 훨씬 빨리 찾음!               │
│                                                                             │
│  2. Hash Join (바구니에 담기):                                                │
│     - 왼쪽 카드를 숫자별 바구니(Hash Table)에 몽땅 담아둠.                        │
│     - 오른쪽 카드를 한 장씩 꺼내서 해당 바구니만 슥 보면 끝! (매우 빠름)            │
│                                                                             │
│  3. Sort Merge (줄 세워 합치기):                                              │
│     - 두 덱을 모두 숫자 순서대로 쭉 나열함.                                      │
│     - 맨 앞부터 동시에 훑으면서 같은 숫자가 나오면 짝을 지어줌.                    │
│                                                                             │
│  → **'데이터가 얼마나 많은가'**와 **'정리(Index/Sort)가 되어있는가'**가 핵심!      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 규모의 가변성**: 수십 건의 조인과 수억 건의 조인은 물리적으로 전혀 다른 접근이 필요함.
- **자원 활용의 최적화**: CPU 연산을 더 쓸지(Hash), I/O를 더 쓸지(NL), 아니면 메모리 정렬을 더 쓸지(Sort Merge) 결정하기 위함.

- **📢 섹션 요약 비유**: 조인 기법은 '서로 다른 언어를 쓰는 두 집단을 대화시키는 통번역 알고리즘'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 3대 조인 기법 상세 비교 매트릭스

| 구분 | Nested Loop Join (NL) | Hash Join | Sort Merge Join (SMJ) |
|:---:|:---|:---|:---|
| **작동 원리** | 2중 For Loop 방식. | 해시 함수 기반 매칭. | 정렬 후 스캔/병합. |
| **인덱스 의존** | **매우 높음** (필수) | 없음 (있으면 좋음) | 보통 (정렬 생략 가능) |
| **주요 환경** | **OLTP (실시간 소량)** | **OLAP (대량 분석)** | 대량, 부등호 조건 |
| **메모리 사용** | 매우 적음 | **매우 높음** (해시 테이블) | 보통 (정렬 공간 필요) |
| **조인 조건** | 모든 연산자 가능 | **동등(=) 조건만 가능** | 모든 연산자 가능 |

### 2. 조인 기법별 작동 프로세스 도해 (ASCII)

데이터가 합쳐지는 경로의 차이를 시각화한다.

```text
    (1) NL Join : [Driving] ──▶ (Index) ──▶ [Driven]
        * 1건씩 순차적으로 찔러보는 방식
    
    (2) Hash Join : [Build Set] ──▶ {Hash Table} ◀── [Probe Set]
        * 메모리에 판을 깔아놓고 들이붓는 방식
    
    (3) Sort Merge : [Table A Sort] ──┐
                                      ├──▶ [Parallel Merge]
                     [Table B Sort] ──┘
        * 양쪽을 줄 세워 지퍼처럼 맞물리는 방식
```

### 3. 심층 동작 원리: 드라이빙 테이블(Driving Table) 선정
- **NL Join**: 먼저 읽는 테이블(Driving)이 작을수록 유리함.
- **Hash Join**: 해시 테이블을 만드는 쪽(Build Input)이 메모리에 다 들어갈 만큼 작을수록 유리함.

### 4. 핵심 기술: 부분 범위 처리 (Partial Range Scan)
- NL Join은 결과가 나오는 대로 즉시 사용자에게 던져줄 수 있지만, Hash나 Sort Merge는 전체 데이터를 다 읽거나 정렬해야 첫 번째 결과를 낼 수 있음.

- **📢 섹션 요약 비유**: NL Join은 '필요할 때마다 시장에 가는 것'이고, Hash Join은 '장을 몽땅 봐와서 냉장고를 채워두는 것'입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 튜닝 관점의 성능 결정 요인 (Critical Path)

| 기법 | 성능을 좌우하는 핵심 요소 |
|:---:|:---|
| **NL Join** | **인덱스 구성** 및 조인 순서. |
| **Hash Join** | **충분한 메모리(PGA)** 및 Build Input 크기. |
| **Sort Merge** | 정렬 작업의 효율성 및 인덱스 활용 여부. |

### 2. 과목 융합 관점: 알고리즘 및 운영체제
- **알고리즘 융합**: NL Join은 $O(N \times M)$의 단순 루프, Hash Join은 $O(N+M)$의 해시 매핑, Sort Merge는 $O(N \log N + M \log M)$의 정렬 복잡도를 가진다.
- **운영체제(OS) 융합**: Hash Join 시 메모리가 부족하면 디스크로 페이징이 발생하는 **Hash Spill** 현상이 나타나며, 이는 OS의 가상 메모리 관리 및 I/O 대기(Wait) 시간으로 전이된다.

- **📢 섹션 요약 비유**: NL Join은 '근면 성실한 일꾼'이고, Hash Join은 '장비빨 세우는 기술자'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 느린 쇼핑몰 주문 조회 쿼리 개선
- **상황**: 고객이 자기 주문을 보려는데 5초가 걸림. (고객 100만 명, 주문 1,000만 건)
- **판단**: 실행 계획을 보니 대량 분석용인 `HASH JOIN`이 돌고 있음. 고객은 한 명인데 굳이 해시 테이블을 만들 필요가 없음.
- **결단**: **인덱스 보강 후 NL Join 유도**. `주문` 테이블의 `고객ID`에 인덱스를 걸고, 힌트(`/*+ USE_NL(o) */`)를 통해 NL Join으로 전환. 응답 시간을 5초에서 0.05초로 단축.

### 2. 안티패턴: 인덱스 없는 테이블 간의 NL Join
- **증상**: 인덱스도 안 걸린 큰 테이블 두 개를 `WHERE` 절로 그냥 묶음.
- **결과**: 옵티마이저가 멍청하게 NL Join을 선택하면, 수조 번의 루프를 돌며 DB 서버 전체가 중단됨.
- **조치**: **반드시 대량 데이터 조인 시에는 Hash Join**이 선택되었는지 확인하고, 안 되어 있다면 통계 갱신이나 힌트로 교정할 것.

### 📢 기술사적 결언
> "조인 기법은 데이터베이스의 **'물리적 전술'**이다. 전술이 상황(데이터 양)에 맞지 않으면 전투(쿼리)에서 패배하기 때문이다. 기술사는 단순 문법을 넘어, **인덱스 전략과 가용 메모리**라는 두 축을 바탕으로 어떤 기법이 최소 비용을 산출할지 예측할 수 있어야 한다. 특히 온라인 서비스와 대형 배치가 공존하는 환경에서 **조인 기법의 조화로운 분배**가 전체 아키텍처의 안정성을 결정함을 잊지 말아야 한다."

- **📢 섹션 요약 비유**: 망치로 못을 박아야지(NL), 포크레인(Hash)을 가져오면 벽(성능)이 무너집니다. 도구의 크기와 용도를 아는 것이 튜닝입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (조인 기법 최적화 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **온라인 응답성** | 수 초 지연 | **즉시 응답 (ms)** | 고객 만족도 및 전환율 상승 |
| **배치 처리량** | 밤새 수행 | **수 시간 내 종료** | 데이터 가용성 시간 확보 |
| **서버 자원 효율** | CPU/Disk Full | **안정적 점유** | 인프라 수명 및 비용 최적화 |

### 2. 미래 전망
- **Hardware-Embedded Join**: CPU를 거치지 않고 저장 장치(SSD) 내에서 직접 해시 조인을 수행하는 **Computational Storage** 가속 기술 상용화.
- **Learned Join Optimizers**: 쿼리 패턴을 스스로 학습한 딥러닝 모델이, 고전적인 비용 계산보다 더 정확하게 조인 기법을 결정하는 **AI 기반 옵티마이저** 탑재.

- **📢 섹션 요약 비유**: 미래의 조인 기법은 우리가 고민하지 않아도, 데이터의 흐름을 스스로 감지하여 '물(Hash)'처럼 흐를지 '바늘(NL)'처럼 찌를지 결정하는 지능형 유체가 될 것입니다.

1. **조인 기법**은 흩어진 장난감 조각들을 합칠 때 쓰는 **"세 가지 마법"**이에요.
2. 한 조각씩 짝을 찾는 마법(NL), 같은 것끼리 바구니에 담는 마법(Hash), 순서대로 줄 세워 합치는 마법(Merge)이 있죠.
3. 이 마법들을 **장난감의 개수에 맞춰서** 잘 써야 지치지 않고 가장 빨리 장난감을 완성할 수 있답니다!
