# 170. 유니크 vs 비유니크 인덱스 - 중복 허용 여부와 성능 영향

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인덱스 키 값의 중복을 허용하지 않음으로써 데이터의 **유일성(Uniqueness)**을 물리적으로 강제하느냐, 아니면 동일한 값을 가진 여러 행의 저장을 허용하느냐의 차이다.
> 2. **성능적 차이**: 유니크 인덱스는 조회 시 원하는 값을 찾는 즉시 탐색을 종료하는 **Unique Scan**이 가능하여 이론적으로 가장 빠르며, 비유니크 인덱스는 중복 가능성 때문에 다음 레코드를 반드시 확인해야 하는 **Range Scan**을 수행한다.
> 3. **가치**: 성능 향상이라는 기능적 목적과 더불어 비즈니스 규칙(ID, 주민번호 등)의 **데이터 무결성**을 시스템 최하단 레이어에서 완벽하게 수호하는 최후의 보루 역할을 한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **유니크 인덱스 (Unique Index)**: 모든 인덱스 키 값이 유일해야 합니다. 중복된 값을 INSERT 하려 하면 '무결성 위반' 에러가 발생합니다.
- **비유니크 인덱스 (Non-Unique Index)**: 동일한 키 값을 가진 행이 여러 개 존재할 수 있습니다. 일반적인 속성에 대한 검색용으로 쓰입니다.

### 💡 비유: 1인 1실 호텔 vs 다인용 도미토리
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        유니크 vs 비유니크 인덱스 비유                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 유니크 인덱스 (1인 1실 호텔):                                              │
│     - "101호 손님 누구세요?" -> 한 명만 나오고 끝! (탐색 종료)                  │
│     - 다른 사람이 101호에 들어가려 하면 직원이 "안 됩니다!"라고 막음. (무결성)      │
│                                                                             │
│  2. 비유니크 인덱스 (다인용 도미토리):                                          │
│     - "305호 손님 누구세요?" -> 한 명이 대답해도, **'다음 침대에 또 있는지'**      │
│       반드시 고개를 돌려 확인해봐야 함.                                         │
│     - 같은 방에 여러 명이 들어와도 직원이 막지 않음.                             │
│                                                                             │
│  → **'한 번에 끝낼지'** 아니면 **'옆을 더 볼지'** 결정하는 효율의 차이입니다!       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **논리적 유일성 보장**: 기본 키(PK)나 대체 키(AK)의 성격을 가진 컬럼은 성능 이전에 데이터의 '중복 불가' 성질을 시스템적으로 보장받아야 함.
- **옵티마이저의 확신**: 값이 하나뿐이라는 사실을 엔진이 미리 알면, 실행 계획 수립 시 훨씬 공격적인 최적화(Unique Scan) 가능.

- **📢 섹션 요약 비유**: 유니크 인덱스는 '세상에 단 하나뿐인 지문'을 관리하는 것이고, 비유니크는 '동명이인을 관리하는 명단'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 유니크 vs 비유니크 작동 메커니즘 비교

| 비교 항목 | 유니크 인덱스 (Unique) | 비유니크 인덱스 (Non-Unique) |
|:---:|:---|:---|
| **스캔 방식** | **Index Unique Scan** | **Index Range Scan** |
| **탐색 종료 시점** | 일치하는 값을 **찾는 즉시** 종료. | 일치하는 값 이후 **다른 값이 나올 때까지**. |
| **INSERT 부하** | **중복 체크 연산** 추가 발생. | 중복 체크 없이 바로 저장. |
| **무결성 강제** | DBMS 레벨에서 차단. | 애플리케이션 레벨에서 관리 필요. |

### 2. 스캔 단계의 물리적 차이 도해 (ASCII)

리프 블록 내부에서 커서(Cursor)가 움직이는 범위를 시각화한다.

```text
    [ Index Leaf Block ]
    Key: [ 10 | 20 | 30 | 30 | 40 ]
    
    (1) SELECT * FROM T WHERE Key = 20 (Unique 인덱스일 때)
    - 20 찾음! ──▶ "아, 유니크니까 더 볼 필요 없지." ──▶ **즉시 종료**
    
    (2) SELECT * FROM T WHERE Key = 20 (Non-Unique 인덱스일 때)
    - 20 찾음! ──▶ "음, 20이 또 있을지 몰라." ──▶ **다음 칸(30) 확인** ──▶ 종료
```

### 3. 심층 동작 원리: 유니크 인덱스의 중복 체크 (DML)
- **메커니즘**: 데이터를 넣을 때 인덱스를 먼저 뒤져서 같은 값이 있는지 확인(Lookup)함.
- **Trade-off**: 조회는 빠르지만, 입력 시에는 이 확인 과정 때문에 비유니크 인덱스보다 미세하게 느릴 수 있음.

### 4. 핵심 기술: 결합 유니크 인덱스
- 여러 컬럼을 묶어 유니크로 지정하는 방식. 개별 컬럼은 중복되어도 되지만, **'조합'**은 유일해야 함을 보장.

- **📢 섹션 요약 비유**: 유니크 인덱스의 조회는 '정답을 맞히자마자 시험지를 내는 우등생'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 성능적 관점의 미세 차이 (Micro-performance)
- 사실 현대의 고성능 DBMS에서는 한 블록 안에서 다음 행을 확인하는 비용(CPU Cycle)이 매우 적어, 유니크와 비유니크의 **순수 조회 성능 차이는 미미**함.
- **진정한 차이**: 옵티마이저가 유니크임을 알고 수립하는 **실행 계획의 전체적인 견고함(Plan Stability)**에서 발생함.

### 2. 과목 융합 관점: 데이터 모델링 및 소프트웨어 공학
- **데이터 모델링 융합**: 유니크 인덱스는 **대체 키(Alternate Key)**의 물리적 실체이며, 정규화 과정에서 식별된 엔티티의 독립성을 수호하는 도구가 된다.
- **소프트웨어 공학(SE) 융합**: 애플리케이션에서 중복 체크 로직을 짜는 것(App-level)보다 DB 유니크 제약을 쓰는 것(DB-level)이 **레이스 컨디션(Race Condition)** 상황에서 훨씬 안전하고 확실한 동기화 방법이다.

- **📢 섹션 요약 비유**: 유니크 인덱스는 '최후의 방어선'입니다. 프로그램이 실수해도 DB가 중복을 막아주기 때문입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 사용자 중복 가입 방지 설계
- **상황**: '이메일'로 가입하는 사이트에서 동일 이메일 중복 가입을 막아야 함.
- **판단**: 프로그램에서 `SELECT`로 확인 후 `INSERT` 하는 방식은 동시 접속 시 뚫릴 수 있음.
- **결단**: **이메일 컬럼에 'Unique Index' 생성**. 프로그램 로직과 상관없이 DB 엔진이 물리적으로 중복 입력을 원천 봉쇄함. 성능과 정합성을 동시에 확보하는 가장 프로페셔널한 설계.

### 2. 안티패턴: 비유니크 인덱스에 의존한 유일성 관리
- **증상**: 유일해야 하는 컬럼(예: 사번)인데 중복 체크 부하가 겁난다고 비유니크 인덱스만 걸어둠.
- **결과**: 어느 날 데이터 입력 사고로 중복 사번이 발생하고, 월급 계산 쿼리가 2배로 계산되어 재무 사고 발생.
- **조치**: **업무적으로 유일해야 하는 컬럼은 무조건 'Unique' 속성 부여**. 중복 체크 부하보다 데이터 오염 복구 비용이 훨씬 크다는 점을 명심할 것.

### 📢 기술사적 결언
> "유니크 인덱스는 **'데이터의 신분증'**이다. 신분증이 중복되면 사회의 질서가 무너지듯, 유니크 속성이 깨진 DB는 비즈니스의 신뢰를 잃는다. 기술사는 성능상의 미세한 이득(Unique Scan)보다, **비즈니스 규칙을 물리적으로 강제**하여 예외 없는 정합성을 유지하는 데 이 도구를 사용해야 한다. 인덱스 하나를 정의할 때 '중복 가능성'을 냉정히 검토하는 습관이 **무결한 데이터 아키텍처**의 시작임을 잊지 말아야 한다."

- **📢 섹션 요약 비유**: 유니크 인덱스를 아끼는 것은 '도둑이 들까 봐 문을 잠그지 않는 것'과 같아, 나중에 훨씬 큰 손해를 볼 수 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (유니크 인덱스 활용 시)

| 지표 | 활용 전 | 활용 후 (Unique) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **조회 응답 시간** | 보통 (Range Scan) | **최상 (Unique Scan)** | 미세한 성능 향상 및 안정성 |
| **데이터 정합성 오류** | 간헐적 발생 | **0% (엔진 강제)** | 데이터 품질 신뢰도 극대화 |
| **실행 계획 안정성** | 변동 가능성 있음 | **인덱스 고정 유리** | 옵티마이저 판단 오류 예방 |

### 2. 미래 전망
- **Semantic Uniqueness**: 단순히 값이 같은지가 아니라, AI가 의미적으로 중복인 데이터(예: 주소가 미세하게 다른 동일인)를 파악하여 유니크성을 제안하는 **지능형 무결성 관리**.
- **Lock-free Unique Checking**: 분산 DB에서 합의 알고리즘(Paxos, Raft)을 통해 락 부하 없이 전역 유일성을 초고속으로 검증하는 기술의 발전.

- **📢 섹션 요약 비유**: 미래의 유니크 인덱스는 고정된 값을 넘어, 데이터의 '정체성'을 지키는 인공지능 수호자가 될 것입니다.
