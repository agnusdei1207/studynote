# 151. 뷰 (View) - 가상 테이블과 논리적 추상화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하나 이상의 기본 테이블(Base Table)로부터 유도된 **가상의 테이블(Virtual Table)**로, 물리적으로 데이터를 저장하지 않고 정의된 SELECT 문만을 데이터 사전에 보관하는 논리적 객체다.
> 2. **다차원적 기능**: 복잡한 쿼리를 단순화하여 사용자 편의성을 높이는 **추상화**, 특정 컬럼에 대한 접근을 제한하는 **보안**, 그리고 기본 테이블 구조 변경 시에도 사용자 관점을 유지하는 **논리적 데이터 독립성**을 제공한다.
> 3. **가치**: 데이터베이스 설계자와 사용자 사이의 완충 지대 역할을 수행하며, 시스템의 유연성과 유지보수성을 극대화하는 아키텍처적 장치다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**뷰 (View)**는 실제로 데이터를 가지지는 않지만, 사용자에게는 일반 테이블과 똑같이 보이는 창(Window)입니다. 뷰를 조회하면 저장된 SQL 문이 실행되어 기본 테이블의 데이터를 실시간으로 투영해 보여줍니다.

### 2. 💡 비유: 맛집의 '추천 메뉴판'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        뷰 (View) 개념 비유                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 주방(데이터베이스)에는 수백 가지 식재료(테이블)가 널브러져 있음.            │
│                                                                             │
│  1. 기본 테이블 (식재료): 고기, 채소, 소스 등 날것의 데이터.                      │
│  2. 뷰 (메뉴판):                                                              │
│     - "스테이크 세트"라는 메뉴판을 만듦.                                        │
│     - 손님은 주방에 가서 고기를 직접 구울 필요 없이 "세트 하나요!"라고 주문함.      │
│     - 메뉴판은 실제 음식이 아니며, 주문하는 순간 주방에서 요리(SQL 실행)해줌.      │
│                                                                             │
│  → 복잡한 주방 안을 보여주지 않고도, 손님이 원하는 **'완성된 요리'**만 보여주는      │
│     깔끔한 서비스 창구와 같습니다!                                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 보안 강화**: 사원 테이블에서 '급여' 컬럼은 숨기고 '이름, 부서'만 일부 사용자에게 보여줘야 할 필요성.
- **구조의 추상화**: 5개 테이블을 조인하는 복잡한 쿼리를 매번 짜기 힘들어서, 미리 합쳐진 형태(View)로 정의해 두고 싶어 함.

- **📢 섹션 요약 비유**: 뷰는 데이터베이스라는 거대한 성벽에 뚫린 '안전한 구멍'과 같아서, 허락된 풍경(데이터)만 볼 수 있게 해줍니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 뷰의 주요 특성 및 제약

| 특성 항목 | 상세 내용 | 비유 |
|:---:|:---|:---|
| **일시성** | 데이터가 저장되지 않고 조회 시점에만 생성됨. | 거울에 비친 모습 |
| **정적 정의** | `CREATE VIEW`로 정의된 SQL 문은 데이터 사전에 저장됨. | 요리 레시피 저장 |
| **수정 제한** | 집계 함수나 조인이 포함된 뷰는 `UPDATE`가 어려움. | 요리된 음식을 재료로 되돌리기 불가 |
| **독립성** | 기본 테이블이 바뀌어도 뷰 정의만 고치면 됨. | 간판은 그대로, 주인은 바뀜 |

### 2. 뷰의 논리적 아키텍처 도해 (ASCII)

사용자가 뷰를 통해 실제 데이터를 바라보는 계층 구조를 시각화한다.

```text
    [ User / Application ]
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ VIEW : V_Emp_Dept ]                                          │
    │  Logic: SELECT e.Name, d.Dname FROM Emp e, Dept d ...           │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (SQL Transformation)
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ RDBMS Engine ] : 뷰 쿼리와 사용자 쿼리를 하나로 병합           │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (Physical I/O)
    ┌──────▼──────────┐        ┌──────────▼──────┐
    │ [ Table: Emp ]  │        │ [ Table: Dept ] │
    │ (Real Data)     │        │ (Real Data)     │
    └─────────────────┘        └─────────────────┘
```

### 3. 심층 동작 원리: 뷰 병합 (View Merging)
- **작동**: 사용자가 `SELECT * FROM MyView WHERE id = 1`을 날리면, 엔진은 뷰 정의문에 `WHERE id = 1`을 합쳐서 최적화된 하나의 쿼리로 변환함.
- **효과**: 뷰를 사용한다고 해서 성능이 무조건 떨어지는 것이 아니며, 옵티마이저가 최적의 실행 경로를 다시 찾아냄.

### 4. 핵심 기술: WITH CHECK OPTION
- 뷰를 통해 데이터를 입력/수정할 때, 뷰의 `WHERE` 조건에 어긋나는 데이터가 들어오지 못하게 강제하는 무결성 제어 기술.

- **📢 섹션 요약 비유**: 뷰 병합은 '안경(View)을 쓰고 풍경(Data)을 보는 것'과 같아, 안경 도수(조건)가 풍경과 합쳐져 눈에 들어옵니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 뷰(View) vs 테이블(Table)

| 비교 항목 | 뷰 (View) | 테이블 (Table) |
|:---:|:---|:---|
| **저장 실체** | **질의 정의(SQL)**만 저장 | **실제 데이터** 저장 |
| **저장 공간** | 거의 소모 안 함 | 데이터 양에 비례하여 소모 |
| **영속성** | 기본 테이블 삭제 시 사용 불가 | 독립적으로 존재 |
| **인덱스** | 생성 불가 (일부 DBMS 제외) | **생성 가능 (성능 향상)** |
| **비유** | **TV 화면** | **방송국 스튜디오** |

### 2. 과목 융합 관점: 보안 및 소프트웨어 공학
- **보안(Security) 융합**: 뷰는 **RBAC(Role Based Access Control)**의 최소 단위로 활용되며, 사용자가 원본 테이블의 존재조차 모르게 하는 '데이터 은닉(Data Hiding)'의 핵심 수단이다.
- **소프트웨어 공학(SE) 융합**: 뷰는 디자인 패턴의 **파사드(Facade)** 패턴과 유사하게 하부의 복잡한 조인 관계를 단순한 인터페이스로 제공하여 개발 생산성을 높인다.

- **📢 섹션 요약 비유**: 테이블이 '금고'라면, 뷰는 금고 앞에 설치된 '감시 카메라 화면'입니다. 직접 만질 순 없어도 필요한 정보는 다 얻을 수 있죠.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 레거시 시스템 마이그레이션 대응
- **상황**: 성능 문제로 `Employee` 테이블을 `Emp_Base`와 `Emp_Detail`로 쪼갬.
- **판단**: 기존 수천 개의 프로그램이 `Employee`라는 이름을 쓰고 있어 수정이 불가능함.
- **결단**: **쪼개진 테이블들을 합친 이름으로 'View' 생성**. 기존 앱들은 테이블이 쪼개진 줄도 모르고 `Employee` 뷰를 바라보며 정상 작동함. '논리적 데이터 독립성'의 실무적 승리.

### 2. 안티패턴: 조인과 집계가 난무하는 뷰의 중첩
- **증상**: 뷰 위에 뷰를 5단계씩 쌓아서 만듦.
- **결과**: `SELECT * FROM TopView` 한 줄을 날렸을 뿐인데, 엔진 내부에서 수십 개의 테이블이 춤을 추며 서버가 뻗음.
- **조치**: **뷰의 깊이는 2단계 이내로 제한**하고, 복잡한 연산 결과가 자주 쓰인다면 **Materialized View**로 물리적 구체화 검토.

### 📢 기술사적 결언
> "뷰는 데이터베이스의 **'논리적 질서'**를 유지하는 방어선이다. 물리적 구조가 비즈니스의 변화를 따라가지 못할 때, 그 간극을 메워주는 것이 뷰의 진정한 가치이기 때문이다. 기술사는 단순히 편리함을 위해 뷰를 만들 것이 아니라, **보안 가이드라인**과 **성능 오버헤드** 사이의 접점을 정교하게 설계해야 한다. 뷰 정의 하나가 전체 시스템의 보안 정합성과 유지보수 비용을 결정짓는 핵심 지표임을 명심해야 한다."

- **📢 섹션 요약 비유**: 뷰를 남발하는 것은 '포스트잇을 너무 많이 붙여서 원래 책 내용을 못 보는 것'과 같아, 적절한 관리가 필요합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (뷰 도입 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **민감 정보 노출** | 원천 차단 불가 | **컬럼 단위 차단** | 데이터 보안성 100% 강화 |
| **SQL 코드 재사용성** | 복사/붙여넣기 | **뷰 이름 참조** | 개발 생산성 2배 향상 |
| **프로그램 수정 비용** | 스키마 변경 시 전수 수정 | **뷰 정의만 수정** | 유지보수 비용 80% 절감 |

### 2. 미래 전망
- **Virtual Private Database (VPD)**: 사용자의 신분(Context)에 따라 동일한 뷰가 서로 다른 데이터를 보여주는 **지능형 동적 뷰** 기술 고도화.
- **Data Virtualization Gateway**: 여러 클라우드에 흩어진 데이터들을 마치 하나의 로컬 뷰처럼 보이게 하는 **데이터 가상화 계층**의 핵심 컴포넌트로 뷰 사상이 확장됨.

- **📢 섹션 요약 비유**: 미래의 뷰는 단순한 창문을 넘어, 사용자의 생각에 따라 데이터의 모습을 바꾸어 보여주는 '홀로그램 인터페이스'가 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **뷰**는 엄청나게 복잡한 장난감 상자 위를 덮은 **"투명한 유리판"**과 같아요.
2. 유리판 위에 내가 보고 싶은 인형만 **동그라미 쳐두면**, 상자를 뒤지지 않아도 그 인형만 쏙 보이죠.
3. 실제 장난감을 꺼내는 게 아니라 **눈으로만 보는 것**이라서, 실수로 장난감을 망가뜨릴 걱정도 없답니다!
