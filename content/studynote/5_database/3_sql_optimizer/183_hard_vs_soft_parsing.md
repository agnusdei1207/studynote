# 183. 소프트 파싱 vs 하드 파싱 - 라이브러리 캐시 최적화 원리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SQL 문장이 실행될 때 DBMS가 구문 분석과 최적화 과정을 거쳐 실행 계획을 새로 만드느냐(**하드 파싱**), 아니면 메모리에 이미 저장된 계획을 재사용하느냐(**소프트 파싱**)의 차이다.
> 2. **성능적 파급력**: **하드 파싱**은 CPU와 메모리 자원을 극심하게 소모하며 공유 메모리에 대한 래치(Latch) 경합을 유발하는 고비용 작업인 반면, **소프트 파싱**은 연산 과정을 생략하여 즉각적인 응답을 가능케 한다.
> 3. **가치**: 바인드 변수(`:var`) 사용을 통해 동일한 SQL 구조를 유지함으로써 소프트 파싱 비율을 극대화하는 것이 대규모 동시 접속 환경의 시스템 안정성을 결정짓는 핵심 튜닝 요소다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **하드 파싱 (Hard Parsing)**: SQL이 처음 실행되거나 캐시에서 밀려났을 때 발생. 최적화(Optimization) 과정을 포함하여 실행 계획을 밑바닥부터 새로 생성함.
- **소프트 파싱 (Soft Parsing)**: SQL 텍스트의 해시(Hash) 값을 대조하여 라이브러리 캐시에 동일한 문장과 실행 계획이 있는지 확인하고, 있으면 그대로 가져다 씀.

### 💡 비유: 수학 문제 풀이 과정
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        하드 vs 소프트 파싱 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 선생님이 칠판에 수학 문제를 적어주심.                                     │
│                                                                             │
│  1. 하드 파싱 (처음 푸는 문제):                                               │
│     - 문제를 읽고(Parser), 공식(Index)을 고민하고, 풀이 방법(Plan)을 연습장에      │
│       직접 써가며 정답을 도출함.                                                │
│     - 머리를 많이 써야 하고 시간도 오래 걸림. (CPU 집중 연산)                     │
│                                                                             │
│  2. 소프트 파싱 (전에 푼 기억이 있는 문제):                                     │
│     - 문제를 보자마자 "아! 이거 아까 푼 거네?"라고 기억해냄.                      │
│     - 머리 쓸 필요 없이 머릿속에 저장된 정답지(Cache)를 보고 바로 대답함.           │
│                                                                             │
│  → 시스템의 성능은 **'얼마나 암기(Cache)'**를 잘해서 **'고민(Hard Parsing)'**을    │
│     줄이느냐에 달려 있습니다!                                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **SQL 파싱 부하의 임계치**: 수만 명이 동시에 접속하는 환경에서 매번 최적화 경로를 계산하는 것은 하드웨어적으로 불가능한 수준에 도달.
- **메모리 기술의 발전**: 자주 쓰이는 데이터를 넘어, '자주 쓰이는 지능(실행 계획)'까지 메모리에 상주시켜 재활용하는 아키텍처 확립.

- **📢 섹션 요약 비유**: 하드 파싱은 '매번 새 집을 짓는 것'이고, 소프트 파싱은 '모델하우스(캐시)에 이미 지어진 집을 보여주는 것'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. SQL 실행 단계별 파싱 분류

| 단계 | 수행 활동 | 하드 파싱 시 | 소프트 파싱 시 |
|:---:|:---|:---: |:---: |
| **1. Syntax Check** | 문법적 오류 검사. | 수행 | 수행 |
| **2. Semantic Check** | 테이블/컬럼 존재 및 권한 확인. | 수행 | 수행 |
| **3. Hash Lookup** | 라이브러리 캐시에서 동일 SQL 검색. | **실패 (Miss)** | **성공 (Hit)** |
| **4. Optimization** | **최적의 실행 계획 수립 (Cost 계산)**. | **수행 (Heavy)** | **생략** |
| **5. Row Source Gen** | 실행 가능한 바이너리 코드 생성. | **수행** | **생략** |

### 2. 라이브러리 캐시와 파싱 프로세스 도해 (ASCII)

메모리 내에서 실행 계획이 재사용되는 흐름을 시각화한다.

```text
    [ New SQL Request ]
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  1. [ Hashing ] : SQL 문자열을 해시 값으로 변환 (ex: '7a2b')      │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  2. [ Library Cache Lookup ] : '7a2b' 가 캐시에 있는가?          │
    └──────┬───────────────────────┬──────────────────────────────────┘
           │ [ NO: Cache Miss ]    │ [ YES: Cache Hit ]
           │                       │
    ┌──────▼──────────────┐        │      ┌───────────────────────────┐
    │ **[ Hard Parsing ]** │        └────▶ │ **[ Soft Parsing ]**      │
    │ - Optimization      │               │ - Reuse Existing Plan     │
    │ - Create Exec Plan  │               └──────────┬────────────────┘
    └──────┬──────────────┘                          │
           │ (Register to Cache)                     │
           └───────────────────────┬─────────────────┘
                                   ▼
                        [ Execution : Start ]
```

### 3. 심층 동작 원리: 대소문자 및 공백의 민감성
- **원리**: 소프트 파싱은 SQL 문자열의 **완전 일치**를 전제로 함.
- **결과**: `SELECT * FROM Emp`와 `select * from emp`는 사람에겐 같지만, DB 엔진에겐 다른 해시 값을 생성하여 각각 하드 파싱을 유발함.

### 4. 핵심 기술: 커서 공유 (Cursor Sharing)
- `CURSOR_SHARING = FORCE` 옵션을 통해, 리터럴 SQL을 엔진 레벨에서 강제로 바인드 변수로 변환하여 소프트 파싱을 유도하는 응급 최적화 기술.

- **📢 섹션 요약 비유**: 소프트 파싱은 '똑같이 생긴 열쇠'를 찾는 작업이라, 글자 하나라도 다르면(해시 불일치) 자물쇠가 열리지 않습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 하드 파싱 vs 소프트 파싱 리소스 소모 비교

| 비교 항목 | 하드 파싱 (Hard) | 소프트 파싱 (Soft) |
|:---:|:---|:---|
| **CPU 사용량** | **매우 높음** | 매우 낮음 |
| **메모리 경합 (Latch)** | 높음 (공유 메모리 쓰기) | 낮음 (읽기 전용) |
| **소요 시간** | 수십 ~ 수백 ms | **마이크로초 (us)** |
| **시스템 영향** | 전체 응답 속도 지연의 주범 | 안정적인 처리량 보장 |

### 2. 과목 융합 관점: 운영체제 및 컴파일러
- **운영체제(OS) 융합**: 하드 파싱 시 발생하는 Latch 경합은 OS의 **뮤텍스(Mutex)나 세마포어** 경쟁과 맥을 같이 하며, 프로세스 간 자원 동기화 실패 시 발생하는 **Context Switching** 부하로 이어진다.
- **컴파일러 융합**: 하드 파싱은 소스 코드를 구문 분석(Parsing)하고 중간 코드(Execution Plan)를 생성하는 전형적인 **Just-In-Time (JIT) 컴파일** 과정의 데이터베이스적 구현이다.

- **📢 섹션 요약 비유**: 하드 파싱은 '복잡한 조립 가구의 매뉴얼을 매번 새로 읽는 것'이고, 소프트 파싱은 '눈 감고도 조립하는 달인'의 경지입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 피크 타임 시의 CPU 100% 장애 대응
- **상황**: 동시 접속자가 몰리는 정오에 DB 서버 CPU가 100%를 찍으며 서비스 마비.
- **판단**: SQL Trace 분석 결과, 소프트 파싱 비율이 50% 미만이고 대량의 리터럴 SQL이 유입 중.
- **결단**: **애플리케이션 바인드 변수 적용 의무화**. 문자열 결합(`ID=' + val`) 대신 `ID=?` 형식을 강제하여 하드 파싱을 억제. 하드웨어 증설 없이 소프트웨어적 교정만으로 CPU 점유율을 20%로 낮춤.

### 2. 안티패턴: 파싱 부하를 무시한 복잡한 동적 SQL
- **증상**: 수천 개의 `IF` 문으로 SQL 문자열을 조각조각 기워 붙임.
- **결과**: 결과적으로 생성되는 SQL 종류가 무한대에 가까워져 라이브러리 캐시가 오염되고, 소프트 파싱 혜택을 전혀 받지 못함.
- **조치**: **SQL 구조의 정형화**. 검색 조건의 가지 수를 줄이고, 핵심 로직은 정적 SQL이나 저장 프로시저(Stored Procedure)로 캡슐화하여 캐시 히트율 제고.

### 📢 기술사적 결언
> "파싱 최적화는 데이터베이스 **'심장의 혈류'**를 조절하는 일이다. 하드 파싱이라는 거대한 노이즈가 시스템을 덮치지 않게 막아야 하기 때문이다. 기술사는 단순히 '빠른 쿼리'를 짜는 것을 넘어, **'공유 가능한 쿼리'**를 설계하는 미학을 가져야 한다. 바인드 변수 사용은 선택이 아닌 **엔터프라이즈 시스템의 생존 수칙**이며, 소프트 파싱 비율은 전체 DB 아키텍처의 건강 상태를 나타내는 가장 정직한 지표임을 잊지 말아야 한다."

- **📢 섹션 요약 비유**: 소프트 파싱 비율이 낮은 DB는 '치매에 걸린 천재'와 같아, 능력은 뛰어나도 매번 했던 일을 기억 못 해 고생만 하게 됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (소프트 파싱 극대화 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **파싱 소요 시간** | 100ms | **1ms 이내** | 응답 속도 100배 향상 |
| **서버 CPU 사용률** | 80~90% (과부하) | **10~20% (안정)** | HW 도입 비용 절감 |
| **동시 접속 수용량** | 100명 (한계) | **1,000명 이상** | 시스템 확장성 확보 |

### 2. 미래 전망
- **Plan-less SQL Execution**: 머신러닝 모델이 SQL 형태만 보고 실행 계획 수립 단계를 아예 건너뛰고 최적의 데이터를 즉시 인출하는 **AI-Bypass 파싱** 기술 등장.
- **Global Distributed Cache**: 클라우드 멀티 노드 환경에서 모든 서버가 실행 계획 캐시를 공유하여, 어느 서버에서 실행하든 하드 파싱을 0에 가깝게 유지하는 **분산 라이브러리 캐시** 아키텍처 진화.

- **📢 섹션 요약 비유**: 미래의 데이터베이스는 우리가 입을 떼기도 전에, 과거의 수조 건 경험을 바탕으로 완벽한 실행 계획을 준비해놓는 '지능형 조력자'가 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **하드 파싱**은 매번 어려운 문제를 풀 때마다 **"어떻게 풀지?"라고 처음부터 고민**하며 땀을 뻘뻘 흘리는 거예요.
2. **소프트 파싱**은 한 번 풀어본 문제를 기억해뒀다가, **"아! 이건 아까 본 거야!"** 하고 바로 답을 말하는 거고요.
3. 기억력(캐시)을 잘 사용하면 컴퓨터가 **힘들지 않고 공부(연산)**를 척척 끝낼 수 있답니다!
