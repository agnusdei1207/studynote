# 비용 기반 옵티마이저(CBO)와 실행 계획 - 통계 정보와 조인 최적화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비용 기반 옵티마이저(CBO, Cost-Based Optimizer)는 사용자가 던진 SQL 쿼리를 실행할 수 있는 수천 개의 '실행 계획(Execution Plan)' 후보군 중, **디스크 I/O와 CPU 비용이 가장 낮게 계산되는 최적의 경로를 수학적으로 도출**하는 데이터베이스의 두뇌다.
> 2. **가치**: 과거 인덱스가 있으면 무조건 인덱스를 타던 규칙 기반(RBO)의 멍청함을 극복하고, 테이블의 데이터 건수, 분포도(Histogram), 인덱스 깊이 등의 통계 정보(Statistics)를 바탕으로 "데이터가 너무 많으면 인덱스보다 풀 스캔이 빠르다"고 동적으로 판단한다.
> 3. **융합**: B+Tree의 클러스터링 팩터(Clustering Factor), 해시/NL 조인의 메모리 비용 등을 종합적으로 계산하며, 힌트(Hint) 기능과 융합되어 DBA와 개발자가 시스템의 한계를 극한으로 튜닝할 수 있는 통제권을 부여한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 옵티마이저는 SQL이라는 '선언적 언어(결과만 명시)'를 '절차적 실행 방법(How)'으로 번역하는 DB 커널 엔진이다. RBO(Rule-Based Optimizer)는 사전에 정의된 우선순위 규칙(예: PK 인덱스가 1순위)에 따라 무조건 실행하고, CBO는 현재 데이터 상태(통계)를 바탕으로 각 경로의 예상 비용(Cost)을 계산하여 결정한다.

- **💡 비유**: RBO가 "서울에서 부산 갈 때는 무조건 경부고속도로가 1순위야!"라고 고집부리는 옛날 내비게이션이라면, CBO는 "지금 경부선이 막히니까 국도로 빠졌다가 KTX로 갈아타는 게 30분 더 빠르다"고 실시간 교통량(통계 정보)을 계산해서 알려주는 최신 AI 내비게이션입니다.

- **등장 배경**:
  1. **RBO의 한계 노출**: 인덱스가 걸려 있는 성별(남/여) 컬럼으로 검색할 때, 100만 건 중 '남'이 50만 건인데도 RBO는 "인덱스가 있으니 무조건 탄다"며 50만 번의 랜덤 I/O를 발생시켜 DB를 마비시켰다. (이 경우 테이블 풀 스캔이 10배 빠름)
  2. **다양한 조인 기법의 등장**: 소트 머지 조인, 해시 조인 등 메모리를 사용하는 고비용 연산이 등장하면서, 단순한 규칙만으로는 이 중 어떤 것이 가장 빠른지 판단할 수 없게 되었다.
  3. **데이터 분포의 불균형**: 같은 테이블이라도 A값은 1건, B값은 10만 건 있는 '데이터 스큐(Data Skew)' 현상을 반영하기 위해 통계 기반의 비용 산정 모델이 필연적으로 요구되었다.

- **📢 섹션 요약 비유**: 요리사(옵티마이저)에게 "카레라이스 만들어줘"라고 주문했을 때, 무작정 마트로 뛰어가는 것(RBO)이 아니라 냉장고 재고(통계)와 예산(비용)을 먼저 계산한 뒤 최적의 동선으로 요리를 시작하는 것(CBO)입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. CBO의 쿼리 최적화 파이프라인

SQL이 입력되면 옵티마이저는 3단계를 거쳐 쿼리를 요리한다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 비용 기반 옵티마이저 (CBO) 동작 아키텍처               │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  1. 질의 변환기 (Query Transformer)                                │
  │     - 사용자의 SQL을 논리적으로 최적화. (예: IN 쿼리를 EXISTS로 변환,   │
  │       뷰 머징(View Merging), 조건 푸시 다운(Pushdown) 수행)         │
  │               │                                                  │
  │               ▼                                                  │
  │  2. 대안 계획 생성기 (Plan Generator)                              │
  │     - 테이블 조인 순서(A->B->C vs C->A->B)와 조인 방법(NL, Hash)의  │
  │       모든 가능한 조합(트리)을 수십~수백 개 생성.                       │
  │               │                                                  │
  │               ▼  ◀─── [ Data Dictionary (통계 정보) ]            │
  │                      - 테이블 행 수, 블록 수, 인덱스 트리 깊이(BLevel)  │
  │  3. 비용 예측기 (Cost Estimator)   - 히스토그램 (데이터 분포도)          │
  │     - 각 실행 계획이 소모할 디스크 I/O와 CPU 비용을 수치(Cost)로 환산. │
  │               │                                                  │
  │               ▼                                                  │
  │  [ 최저 비용의 '최종 실행 계획(Execution Plan)' 도출 및 실행 ]         │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CBO의 심장은 **'비용 예측기(Cost Estimator)'**와 **'통계 정보(Statistics)'**다. 아무리 최적화 알고리즘이 훌륭해도, 딕셔너리에 저장된 통계 정보가 과거의 것(예: 어제 100만 건을 지웠는데 통계에는 여전히 100만 건으로 남아 있음)이라면 CBO는 완전히 잘못된 길을 선택하게 된다(예: 작은 테이블인 줄 알고 풀 스캔을 때렸다가 시스템이 멈춤). 따라서 주기적으로 `GATHER_STATS` 작업을 수행하여 통계 정보를 최신화하는 것이 DBA의 가장 중요한 업무다.

---

### 2. 선택도 (Selectivity)와 카디널리티 (Cardinality)

CBO가 특정 조건(`WHERE`)을 실행했을 때 "몇 건이 추출될 것인가?"를 예측하는 수학적 기준이다.

- **선택도 (Selectivity)**: 전체 데이터 중 조건에 만족하는 행의 비율. (예: 남자/여자 = 0.5)
- **카디널리티 (Cardinality)**: 전체 행 수 $\times$ 선택도. (예: 100만 건 $\times$ 0.5 = 50만 건)
- **손익 분기점**: RDBMS에서 인덱스를 타는 랜덤 I/O 비용은 너무 비싸기 때문에, 일반적으로 카디널리티가 전체 데이터의 **10~15%를 넘어가면** CBO는 인덱스를 무시하고 테이블을 몽땅 읽는 풀 스캔(Table Full Scan)을 선택한다.

---

### 3. 클러스터링 팩터 (Clustering Factor, CF)

인덱스의 키값 정렬 순서와 실제 디스크에 저장된 테이블 데이터의 물리적 정렬 순서가 얼마나 일치하는지를 나타내는 지표다.

- **CF가 좋을 때 (수치가 낮음)**: 인덱스를 읽고 실제 데이터를 찾으러 디스크에 갈 때, 필요한 데이터들이 물리적으로 한 블록(페이지)에 옹기종기 모여 있어 I/O가 단 1번만 발생한다. (최고의 성능)
- **CF가 나쁠 때 (수치가 높음)**: 데이터가 디스크 여기저기 흩어져 있어, 인덱스 1건을 읽을 때마다 매번 새로운 디스크 블록을 퍼올려야 한다. (랜덤 I/O 폭발)
- **CBO의 판단**: CF가 너무 나쁘면 CBO는 인덱스 스캔을 포기하고 테이블 풀 스캔으로 돌아선다.

- **📢 섹션 요약 비유**: 클러스터링 팩터가 좋다는 것은 도서관 책들이 '역사'라는 인덱스 순서대로 실제 서가에도 쭉 꽂혀 있는 상태이고, CF가 나쁘다는 것은 인덱스는 가나다순인데 실제 책들은 서가 여기저기에 무작위로 흩어져 있어 사서가 하루 종일 뛰어다녀야 하는 상태입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### RBO (규칙 기반) vs CBO (비용 기반) 패러다임 비교

| 비교 항목 | RBO (Rule-Based Optimizer) | CBO (Cost-Based Optimizer) |
|:---|:---|:---|
| **판단 기준** | 사전에 1~15순위로 정해진 규칙 (예: ROWID > PK > Index) | 디스크 I/O와 CPU 연산의 추정 비용 (Cost) |
| **통계 정보** | 전혀 참조하지 않음 (눈감고 운전) | 테이블 건수, 인덱스 뎁스, 블록 수 필수 참조 |
| **데이터 스큐(Skew)**| "서울 10건, 부산 100만 건" 분포 무시 (항상 인덱스 탐) | **히스토그램(Histogram)**을 통해 분포 파악 후 유연하게 대응 |
| **튜닝 방식** | 쿼리 내의 문법 순서 변경, 인덱스 강제 생성 | 통계 정보 갱신, `/*+ HINT */`를 통한 비용 조작 |
| **현재 위상** | Oracle 10g 이후 폐기 (역사 속으로 사라짐) | **현대 모든 RDBMS의 기본 표준 두뇌** |

### 과목 융합 관점

- **운영체제 (OS)**: CBO가 계산하는 '비용(Cost)'의 90%는 바로 OS 레벨의 **디스크 블록 I/O 횟수**다. 최근에는 메모리 성능이 좋아짐에 따라, 디스크에서 읽어오는 비용뿐만 아니라 래치(Lock) 획득 및 CPU 사이클 연산량까지 코스트 계산식에 융합(System Statistics)하여 정밀도를 높였다.
- **자료구조 (히스토그램)**: 특정 컬럼 데이터의 치우침(Skew)을 파악하기 위해, 통계학의 히스토그램 기법(Frequency, Height-Balanced)을 도입하여 "이번 조건은 희귀값이니 인덱스를 타고, 저 조건은 흔한 값이니 풀 스캔을 타라"는 동적 결정을 내린다.

- **📢 섹션 요약 비유**: RBO가 "법(규칙)대로만 해!"를 외치는 융통성 없는 꼰대 판사라면, CBO는 피의자의 사정(통계 정보)과 현재 사회 분위기(시스템 부하)를 종합적으로 고려하여 합리적인 판결을 내리는 현명한 재판장입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 자정 배치 쿼리가 갑자기 5시간 걸려 서버가 터진 사태 (Plan Flip)

- **문제 상황**: 매일 밤 10분 만에 끝나던 '정산 배치 SQL'이 아무런 코드 변경도 없었는데 갑자기 5시간이 넘게 돌며 DB CPU가 100%를 치는 장애가 발생.
- **원인 분석**: 원인은 **'실행 계획 뒤집힘(Plan Flip)'** 현상. 어제 낮에 정산 대상 테이블에 대량의 데이터를 `DELETE` 했는데, DBA가 통계 정보(Statistics)를 갱신하지 않음. CBO는 여전히 "이 테이블은 1,000만 건짜리 거대 테이블이야"라고 착각하고 초고속인 'Nested Loop 인덱스 스캔' 대신 무거운 'Hash Join 및 Full Scan'을 선택해 버린 것.
- **의사결정 (튜닝 및 조치)**:
  1. 즉시 `DBMS_STATS.GATHER_TABLE_STATS`를 실행하여 딕셔너리 통계 정보를 최신화.
  2. SQL 쿼리에 `/*+ USE_NL(A B) INDEX(A idx_date) */` 힌트를 고정 삽입하여, 향후 통계가 틀어지더라도 옵티마이저가 무조건 빠르고 안전한 인덱스 경로를 타도록(Plan Stability) 강제 통제. (소요 시간 10분으로 복귀)

### 도입 체크리스트
- **통계 정보 갱신(Gather Stats) 정책**: 매일 새벽 트래픽이 없는 시간에 전체 테이블에 대한 샘플링 통계 수집 배치 Job(예: `auto_sample_size`)이 돌고 있는가? 통계 정보가 없는 CBO는 바보가 된다.
- **바인드 변수 (Bind Variable)와 히스토그램의 충돌**: 바인드 변수(`WHERE name = ?`)를 쓰면 쿼리 파싱 성능은 좋아지지만, CBO 입장에서는 안에 무슨 값이 들어올지 몰라 히스토그램(분포도)을 무시하고 무조건 '평균값'으로 실행 계획을 짠다. 데이터 불균형이 극심한 컬럼(예: 탈퇴 여부 Y/N)은 바인드 변수 대신 리터럴 상수를 쓰거나 힌트로 제어해야 한다(Peeking 문제).
- **Plan Baseline (실행 계획 고정)**: 쿼리 성능이 갑자기 나빠지는 것을 막기 위해, 한 번 성공적으로 수립된 실행 계획을 DB 내부에 영구 고정(SQL Plan Management)시키는 안전장치를 가동하고 있는가?

### 안티패턴: 강제 힌트(Hint)의 과도한 남용 (Hint Abuse)
- CBO를 믿지 못하고, 모든 SQL에 개발자가 `/*+ INDEX(A) USE_NL(A B) */` 같은 힌트를 도배하는 행위. 몇 년 뒤 데이터가 100배로 늘어나면 CBO는 똑똑하게 '해시 조인'으로 계획을 바꿔야 하지만, 과거에 박아둔 힌트 때문에 억지로 'NL 조인'을 돌다가 시스템이 파멸한다. 힌트는 옵티마이저가 멍청한 짓을 할 때만 제한적으로 개입하는 '최후의 비상 브레이크'로 써야 한다.

- **📢 섹션 요약 비유**: 뛰어난 자동 운전(CBO) 차량에 타 놓고, 불안하다며 매일 수동 기어(Hint)를 억지로 조작하면 결국 산길에서 엔진이 타버립니다. 차를 믿고, 오직 지도가 잘못되었을 때(통계 정보 오류)만 수동 개입하는 것이 튜닝의 정석입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 통계 부재 및 RBO 환경 | CBO 기반 통계 최적화 환경 | 개선 효과 |
|:---|:---|:---|:---|
| **조인 성능** | 수백만 건 조인 시 인덱스 스캔(랜덤 I/O) | 동적으로 Hash/Sort Join으로 자동 전환 | **대용량 집계 배치 처리 시간 수십 배 단축** |
| **운영 안정성** | 매번 다른 플랜을 타서 응답 시간 요동 | 통계 기반의 일관된 비용 예측 수립 | **쿼리 실행 속도(Latency)의 변동성 90% 억제** |
| **유지보수비용** | 데이터 건수가 늘 때마다 개발자가 SQL 수정 | CBO가 데이터 변화에 맞춰 알아서 플랜 진화 | **SQL 코드 튜닝 및 유지보수 공수 제로화** |

### 미래 전망
- **AI 기반 자율 최적화 (Autonomous Tuning)**: DBA가 수동으로 통계를 수집하고 힌트를 주던 시대를 지나, 현대의 클라우드 데이터베이스는 내장된 머신러닝 엔진이 수백만 개의 쿼리 패턴을 스스로 학습하여, 밤사이에 인덱스를 새로 만들거나 지우고 최적의 실행 계획을 영구 고정하는 '자율 주행 DB(Autonomous DB)' 시대로 완전히 접어들었다.
- **Adaptive Query Execution (적응형 실행)**: 최신 옵티마이저(Spark 3.0, Oracle 12c+)는 쿼리가 실행되기 '전'에 계획을 세우는 데 그치지 않고, 쿼리가 '실행되는 도중'에 데이터가 예상보다 많다고 판단되면 메모리 상에서 실시간으로 조인 방식(NL -> Hash)을 스스로 바꿔버리는 극강의 동적 최적화를 달성했다.

- **📢 섹션 요약 비유**: 비용 기반 옵티마이저는 단순히 길을 알려주는 지도를 넘어, 운전 중에 사고가 나면 실시간으로 우회로를 뚫고 목적지까지 가장 적은 기름(비용)으로 승객을 모시는 21세기 최고의 AI 운전기사입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 내가 심부름 로봇에게 "마트 가서 과자 사 오고, 도서관 가서 책 빌려와!"라고 시켰어요. (SQL 쿼리)
2. 멍청한 로봇(RBO)은 무조건 시킨 순서대로 마트부터 갔다가 도서관이 문을 닫아버렸죠.
3. 똑똑한 로봇(CBO)은 "지금 마트 세일 시간이라 사람이 많고 도서관은 곧 문 닫으니까(통계 정보), 도서관부터 갔다가 나중에 마트로 가는 게 시간(비용)이 제일 아껴지겠다!"라고 스스로 경로(실행 계획)를 짜서 완벽하게 심부름을 해낸답니다!
