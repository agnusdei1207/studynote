# 이벤트 소싱과 CQRS 패턴 - 분산 데이터 무결성 설계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트 소싱(Event Sourcing)은 데이터의 '최종 상태(State)'만을 덮어써서 저장하던 전통적인 CRUD 방식에서 탈피하여, 상태를 변경시킨 **'모든 이벤트(Event)의 순차적 흐름'을 불변(Immutable)의 로그로 영구 저장**하는 데이터 모델링 기법이다.
> 2. **가치**: 시스템 장애나 데이터 오염 발생 시 언제든 이벤트를 처음부터 재재생(Replay)하여 특정 시점의 완벽한 상태 복원이 가능하며, 모든 비즈니스 행위가 원천 기록되므로 완벽한 감사(Audit) 추적을 보장한다.
> 3. **융합**: 이벤트 소싱은 필연적으로 쿼리의 복잡성을 낳으므로, 쓰기(Command) 모델과 읽기(Query) 모델을 물리적으로 분리하는 **CQRS (Command and Query Responsibility Segregation)** 패턴과 완벽하게 융합되어 마이크로서비스(MSA) 아키텍처의 트래픽 병목을 해소한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **이벤트 소싱 (Event Sourcing)**: 상태를 업데이트(Update)하는 대신, 상태를 변화시키는 이벤트(예: '장바구니 추가', '장바구니 삭제')들을 순서대로 이벤트 스토어(Event Store)에 추가(Append)만 하는 방식이다.
  - **CQRS**: 상태를 변경하는 명령(Command) 시스템과 상태를 반환하는 조회(Query) 시스템의 책임을 분리하는 아키텍처 패턴이다.

- **💡 비유**: 전통적 DB(CRUD)가 현재 내 통장에 '10,000원'이 있다고 잔액(최종 상태)만 칠판에 썼다 지웠다 하는 것이라면, 이벤트 소싱은 은행 통장(입출금 내역)처럼 '5,000원 입금', '3,000원 출금' 내역을 지우지 않고 한 줄씩 계속 기록만 하는 것입니다. 현재 잔액을 알고 싶으면 처음부터 끝까지 더해보면(Replay) 됩니다.

- **등장 배경**:
  1. **CRUD의 상태 덮어쓰기 문제**: `UPDATE`를 수행하면 이전 데이터는 물리적으로 소멸된다. "고객이 왜 최종적으로 장바구니를 비웠는가?"라는 비즈니스 맥락(Context)과 의도가 완전히 사라져 버리는 한계가 있었다.
  2. **조회와 갱신의 트래픽 불균형**: 대부분의 서비스는 읽기(Read)와 쓰기(Write)의 비율이 100:1에 달한다. 이를 하나의 DB에서 처리하면 쓰기 시 발생하는 락(Lock)이 수많은 읽기 요청을 블로킹하는 성능 저하가 발생했다.
  3. **MSA의 분산 트랜잭션 한계 극복**: 여러 마이크로서비스 간에 2PC(2단계 커밋) 같은 강한 결합을 사용하면 시스템이 멈추기 때문에, 이벤트를 발행하고 구독하는 비동기적 통신 방식이 필요해졌다.

- **📢 섹션 요약 비유**: 사고가 났을 때 현재 망가진 자동차의 상태(CRUD)만 보는 것보다, 사고 나기 직전까지의 블랙박스 영상(이벤트 소싱)을 보는 것이 원인을 가장 완벽하게 찾아내는 방법입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 이벤트 소싱 (Event Sourcing)의 작동 원리

이벤트 소싱 아키텍처에서는 오직 `INSERT` 연산만 존재하며, `UPDATE`나 `DELETE`는 절대 발생하지 않는다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 전통적 CRUD vs 이벤트 소싱 (장바구니 예시)            │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 전통적 CRUD (RDBMS) ]                                            │
  │   - T1: INSERT 장바구니 (사과 1개)  -> DB: [사과 1]                    │
  │   - T2: UPDATE 장바구니 (사과 2개)  -> DB: [사과 2] (과거 데이터 소멸)     │
  │   - T3: DELETE 장바구니 (사과)     -> DB: [] (텅 빔, 왜 지웠는지 모름)   │
  │                                                                  │
  │  [ 이벤트 소싱 (Event Sourcing) ]                                   │
  │   (이벤트 스토어 - Append Only)                                     │
  │   - Seq 1: ItemAddedEvent { item: 사과, qty: 1 }                    │
  │   - Seq 2: ItemQuantityChangedEvent { item: 사과, qty: 2 }          │
  │   - Seq 3: ItemRemovedEvent { item: 사과 }                          │
  │                                                                  │
  │  * 현재 상태 도출 (Replay / Projection):                              │
  │    Seq 1 + Seq 2 + Seq 3 을 순차적으로 연산(Fold) -> 잔여 수량 0개 반환.  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이벤트 소싱은 상태 자체를 저장하지 않는다. 오직 일어난 '사건(Event)'을 과거형 동사(`ItemAdded`)로 기록한다. 이 이벤트들은 변경이 불가능(Immutable)하므로 DB 락(Lock) 경합이 발생하지 않아 쓰기 속도가 극한으로 빠르다. 하지만 현재 장바구니에 뭐가 들었는지 조회(Query)하려면 매번 수천 개의 이벤트를 처음부터 다시 계산(Replay)해야 하는 치명적인 단점이 있다. 이를 해결하기 위해 정기적으로 현재 상태를 찍어두는 **스냅샷(Snapshot)** 기법이 필수적이다.

### 2. CQRS (Command and Query Responsibility Segregation) 아키텍처

이벤트 소싱의 '조회(Query) 지연 문제'를 해결하기 위해 쿼리 전용 DB를 분리하는 아키텍처다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 Event Sourcing + CQRS 아키텍처                       │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │     [ Client ] ── (Command: 쓰기 요청) ──▶ [ Command API ]         │
  │         │                                        │               │
  │         │                                        ▼               │
  │         │                            ┌───────────────────────┐   │
  │         │                            │    Event Store (DB)   │   │
  │         │                            │ (Append Only 로그 저장) │   │
  │         │                            └───────────┬───────────┘   │
  │         │                                        │ (Message Bus/Kafka)│
  │         ▼                                        ▼               │
  │  (Query: 읽기 요청)                    ┌───────────────────────┐   │
  │         │                            │ Event Handler (프로젝션)│   │
  │         ▼                            └───────────┬───────────┘   │
  │  [ Query API ]                                   ▼               │
  │         │                            ┌───────────────────────┐   │
  │         └────────── (조회) ──────────│   Read Model (DB)     │   │
  │                                      │ (Redis, ES 등 조회 전용) │   │
  │                                      └───────────────────────┘   │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클라이언트가 상태를 변경하는 명령(Command)을 내리면, 이는 이벤트 스토어(RDBMS나 Cassandra)에 이벤트 형태로 순차 기록된다. 이 이벤트는 즉시 메시지 큐(Kafka)를 타고 이벤트 핸들러(Event Handler)로 전달된다. 핸들러는 이벤트를 해석하여 현재 상태를 미리 계산(Projection)한 뒤, 읽기 전용 DB(Read Model - 주로 Redis나 Elasticsearch)에 저장해 둔다. 클라이언트가 조회를 요청(Query)하면 무거운 이벤트 스토어를 뒤질 필요 없이, 이미 완성된 Read Model에서 1ms 이내에 데이터를 꺼내간다.

## Ⅲ. 융합 비교 및 다각도 분석

### 아키텍처 패턴 심층 비교

| 비교 항목 | 전통적 Monolithic DB | CQRS (Event Sourcing 포함) |
|:---|:---|:---|
| **데이터 소스** | 읽기와 쓰기가 동일한 RDBMS 테이블 공유 | **읽기 DB와 쓰기 DB가 물리적으로 분리됨** |
| **쓰기(Write) 최적화** | 인덱스 유지 및 락(Lock) 대기로 병목 발생 | **순차적 Append-only로 I/O 병목 없음** |
| **읽기(Read) 최적화** | 복잡한 조인(Join)으로 CPU/메모리 부하 | **비정규화된 캐시(View)에서 단순 조회 (O(1))** |
| **데이터 일관성** | 강한 일관성 (ACID) 즉시 보장 | **결과적 일관성 (Eventual Consistency) 수용** |
| **시스템 복잡도** | 단일 코드베이스로 관리 용이 | 매우 높음 (메시지 브로커, 동기화, 보상 트랜잭션 관리) |

### 과목 융합 관점

- **데이터베이스 (NoSQL)**: 읽기 전용 모델(Read Model)은 특정 쿼리 요구사항에 맞게 언제든지 갈아치울 수 있어야 한다(Polyglot Persistence). 따라서 검색이 필요하면 Elasticsearch, 단순 조회가 필요하면 Redis나 MongoDB를 Read Model로 융합시켜 사용한다.
- **소프트웨어 공학 (Saga 패턴)**: CQRS 환경에서 트랜잭션 실패 시 데이터 롤백을 수행하려면 RDBMS의 `ROLLBACK` 명령이 통하지 않는다. 대신 '주문 취소 이벤트(OrderCancelledEvent)'라는 **보상 트랜잭션(Compensating Transaction)**을 발행하여 수동으로 상태를 되돌리는 Saga 패턴이 필연적으로 결합된다.

- **📢 섹션 요약 비유**: CQRS는 '주방(Command)'과 '진열장(Query)'을 완벽히 분리한 빵집입니다. 주방에서는 빵을 만들기만 하고, 만들어진 빵은 컨베이어 벨트를 타고 즉시 진열장으로 이동합니다. 손님들은 진열장만 보고 빵을 순식간에 골라갈 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 금융 앱의 입출금 계좌 시스템 구축

- **문제 상황**: 고객의 입출금 계좌를 RDBMS의 `잔액` 컬럼으로 관리 중인데, 트래픽 폭주 시 데드락(Deadlock)이 발생하고, 가끔 100원이 비는데 원인을 추적할 로그가 부족함.
- **의사결정 (Event Sourcing 도입)**:
  1. 계좌의 `잔액` 컬럼을 없애고 `AccountEvent` 테이블만 생성하여 '입금 100원', '출금 50원' 이벤트를 Insert만 하도록 아키텍처 개편. (쓰기 락 완벽 해소)
  2. 고객이 앱을 켜면 화면에 표시될 잔액은 CQRS의 Read Model(Redis)에서 즉시 가져옴.
  3. 야간 배치를 통해 이벤트 스토어의 이벤트를 100개 단위로 압축하여 '현재 잔액 50원'이라는 스냅샷(Snapshot)을 생성, 다음번 Replay 연산 비용을 최적화함.

### 도입 체크리스트
- **결과적 일관성(Eventual Consistency) 수용 여부**: Command 단에 이벤트가 기록된 후, Read 단에 반영되기까지 네트워크 지연(약 0.1~1초)이 발생한다. 만약 고객이 "돈을 입금하고 0.1초 만에 새로고침했는데 잔액이 그대로인 상황"을 서비스 정책적으로 방어할 수 있는가? (UI에서 낙관적 업데이트 처리 등)
- **이벤트 스키마의 버저닝(Versioning)**: 비즈니스가 진화하면서 과거에 발생했던 이벤트의 구조가 바뀔 수 있다. (예: 주소 필드가 추가됨). 과거 이벤트를 Replay할 때 에러가 나지 않도록 스키마 버저닝(Upcaster 패턴 등) 전략이 수립되었는가?
- **인프라 운영 역량**: Kafka, Zookeeper, Event Store, Read DB(Redis) 등 인프라 아키텍처가 기하급수적으로 복잡해진다. 이를 관리할 DevOps 팀의 역량이 뒷받침되는가?

### 안티패턴: 모든 마이크로서비스에 무분별한 CQRS 적용 (Over-engineering)
- 블로그 게시판이나 단순한 관리자 페이지(Admin)처럼 읽기/쓰기 빈도가 1:1에 가깝고 데이터 복잡도가 낮은 시스템에까지 유행을 쫓아 이벤트 소싱과 CQRS를 도입하는 행위. 개발 리드타임이 5배 이상 길어지고, 버그 추적이 극도로 어려워지며 인프라 유지 비용만 낭비하는 최악의 과잉 설계다. 오직 비즈니스 도메인의 핵심 코어(Core Domain)에만 선택적으로 적용해야 한다.

- **📢 섹션 요약 비유**: 동네 떡볶이집(단순 로직)에서 주문을 받을 때 무전기(메시지 큐)를 쓰고 주방(Command)과 홀(Query)을 멀리 분리해 놓으면, 떡볶이 하나 나가는 데 직원들만 고생하고 손님은 떠나갑니다. 이 기술은 수백 명이 동시에 주문하는 대형 공장에 어울립니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 모놀리식 CRUD 구조 | Event Sourcing + CQRS 구조 | 개선 효과 |
|:---|:---|:---|:---|
| **트랜잭션 성능** | 쓰기/읽기 락 경합으로 CPU 병목 | Append-only 기록으로 락 0화 | **쓰기 처리량(TPS) 선형적 무한 확장** |
| **데이터 감사 (Audit)** | 최종 상태만 남아 과거 추적 불가 | 모든 비즈니스 인텐트(의도) 저장 | **데이터 유실 및 해킹 추적 능력 100% 확보** |
| **조회 성능 (Read)** | 무거운 조인 쿼리로 타임아웃 발생 | 캐시된 Read Model 직접 조회 | **API 응답 시간 밀리초(ms) 단위 단축** |

### 미래 전망
- **클라우드 네이티브 이벤트 스토어 (DBaaS)**: 과거에는 RDBMS를 억지로 개조해 이벤트 스토어로 썼으나, 현재는 EventStoreDB와 같이 이벤트 모델링에 완벽하게 최적화된 클라우드 관리형 인프라가 대세로 자리 잡고 있다.
- **스트림 처리 프레임워크와의 완벽한 융합**: 이벤트 소싱에서 발생하는 이벤트를 Apache Flink, Kafka Streams가 실시간으로 가로채어 복잡한 타임 윈도우(Time Window) 기반의 머신러닝 피처(Feature)로 가공하는 리얼타임 데이터 파이프라인의 핵심 근간이 될 것이다.

- **📢 섹션 요약 비유**: 이벤트 소싱과 CQRS는 시스템에 '타임머신(완벽한 과거 기록)'과 '순간이동(초고속 조회)' 능력을 동시에 부여하여, 예측 불가능한 미래의 트래픽 앞에서도 기업을 가장 안전하게 지켜주는 방패입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 일반 데이터베이스는 칠판에 적힌 점수판 같아요. 점수가 오르면 옛날 숫자를 쓱쓱 지우고 새 숫자를 덮어써 버리죠. 그래서 1시간 전에 몇 점이었는지는 아무도 몰라요. (CRUD)
2. **이벤트 소싱**은 비디오카메라를 켜고 시합 전체를 녹화하는 거예요. 언제 누가 골을 넣었는지 다 찍혀 있으니까, 나중에 비디오를 되감기(Replay) 해보면 모든 순간의 점수를 정확히 알 수 있답니다.
3. 하지만 점수 보려고 매번 비디오를 처음부터 볼 순 없잖아요? 그래서 다른 친구 한 명이 비디오를 보면서 점수만 큰 종이(조회 전용 DB, **CQRS**)에 빠르게 적어두어 사람들이 바로 볼 수 있게 돕는 거랍니다!
