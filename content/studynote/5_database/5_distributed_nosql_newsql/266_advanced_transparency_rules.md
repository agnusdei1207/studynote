# 266. 병행, 장애, 지역 사상 투명성 - 분산 운영의 삼중 보호막

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 병행, 장애, 지역 사상 투명성은 다수의 사용자가 동시에 접근하고, 노드 장애가 발생하며, 각기 다른 로컬 DB 환경이 섞인 복잡한 상황을 **사용자가 전혀 인지하지 못하게 은닉하는 고차원 투명성**이다.
> 2. **가치**: 데이터 충돌(Concurrency), 시스템 중단(Failure), 이기종 매핑(Local Mapping)의 문제를 DDBMS가 스스로 해결함으로써 **글로벌 비즈니스의 중단 없는 연속성**을 보장한다.
> 3. **융합**: 분산 락킹, 페일오버(Failover), 논리-물리 스키마 매핑 기술이 융합되어 분산 데이터베이스의 사용자 편의성을 완성한다.

+++

### Ⅰ. 3대 투명성의 정의 및 역할

1. **병행 투명성 (Concurrency Transparency)**:
    - 분산 노드에서 여러 트랜잭션이 동시에 실행되더라도 결과는 항상 일관됨을 보장합니다. 사용자는 다른 사람의 간섭을 느끼지 못합니다.
2. **장애 투명성 (Failure Transparency)**:
    - 특정 노드나 네트워크에 장애가 나도 시스템이 자동으로 우회(Failover)하여 서비스가 지속됩니다. 사용자는 시스템이 고장 났었는지조차 모릅니다.
3. **지역 사상 투명성 (Local Mapping Transparency)**:
    - 로컬 DB의 스키마 명칭과 전역 DB의 명칭이 달라도 상관없습니다. 사용자는 전역 이름만 사용하면 DDBMS가 알아서 로컬 명칭으로 변환해 줍니다.

+++

### Ⅱ. 장애 투명성 매커니즘 시각화 (ASCII Flow)

```text
[장애 투명성: Auto-Failover]

  (Client) "Reading Data..."
       │
  ┌────▼────┐
  │  Proxy  │ ──▶ [ Node A (Down! 💥) ] ── (Timeout)
  └────┬────┘          │
       │               │ (Detection)
       └──────▶ [ Node B (Backup) ] ──▶ ✅ SUCCESS!
       │
  (Result) "Data Delivered" (사용자는 장애가 있었는지 모름)
```

+++

### Ⅲ. 실무적 의의

- **무정지 운영**: 장애 투명성은 99.99% 이상의 고가용성 달성을 위해 필수적입니다.
- **이기종 DB 통합**: 지역 사상 투명성을 통해 Oracle, MySQL 등 서로 다른 DB를 하나의 분산 시스템으로 묶을 수 있습니다.

- **📢 섹션 요약 비유**: 이 삼중 보호막은 **'베테랑 집사'**와 같습니다. 집 안에서 손님들끼리 부딪히지 않게 조율하고(병행), 전구가 나가면 손님이 어둠을 느끼기 전에 바로 갈아 끼우며(장애), 손님이 외국어로 말해도 알아서 집안 규칙대로 통역(지역 사상)하여 일을 처리해 주는 것과 같습니다.
