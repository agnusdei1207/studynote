# 269. 수직 분할 (Vertical Fragmentation) - 속성 단위의 격리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 수직 분할(Vertical Fragmentation)은 하나의 논리적 테이블(Relation)을 구성하는 **애트리뷰트(Attribute, 속성) 집합을 사용 패턴(Access Pattern)이나 보안 등급에 따라 물리적으로 분리**하여 여러 노드에 분산 저장하는 데이터베이스 설계 기법이다.
> 2. **가치**: 관계 대수(Relational Algebra)의 **프로젝트(Project, $\pi$) 연산**에 기반하며, 불필요한 컬럼 조회를 억제하여 디스크 I/O 및 버퍼 캐시(Buffer Cache) 효율을 극대화하고, 민감 데이터의 격리를 통해 보안 컴플라이언스(Compliance)를 강화한다.
> 3. **융합**: 분산 환경에서 데이터의 무결성을 보장하기 위해 각 파편(Fragment)에 **기본키(Primary Key, PK)를 중복 포함**시켜야 하며, 이는 분산 질의 처리(Distributed Query Processing) 시 비용이 높은 조인(Join) 연산을 유발할 수 있는 트레이드오프(Trade-off)를 내포한다.

---

### Ⅰ. 개요 (Context & Background)

수직 분할은 관계형 데이터베이스(RDBMS)의 성능 최적화와 보안 강화를 위해 테이블의 **열(Column) 단위**로 데이터를 쪼개는 기술을 말합니다. 수학적으로는 관계 대수의 '프로젝션(Projection)' 개념에 해당하며, 테이블을 구성하는 속성들을 **'자주 참조되는 그룹'**과 **'드물게 참조되는 그룹'** 혹은 **'공개 가능 그룹'**과 **'기밀 그룹'**으로 분류합니다.

**1. 등장 배경 및 필요성**
과거의 OLTP(Online Transaction Processing) 환경에서는 테이블의 대부분 속성을 자주 접근했으나, 현대의 마이크로서비스 아키텍처(MSA)나 빅데이터 환경에서는 특정 도메인의 속성만 집중적으로 접근하는 패턴이 일반화되었습니다. 예를 들어, 사용자 프로필(이름, 프로필 사진)은 수백만 번 읽히지만, 결제 정보(CARD_INFO)는 거의 수정되지 않는 상황이 이에 해당합니다. 이때 모든 속성을 포함한 거대한 테이블을 조회하면, 불필요한 디스크 I/O가 발생하여 시스템 자원이 낭비됩니다.

**2. 기술적 원리**
수직 분할의 핵심은 **'저장소의 물리적 분리'**입니다. 논리적으로는 하나의 테이블처럼 보이지만, 물리적으로는 서로 다른 디스크 볼륨이나 서버에 저장됩니다. 이를 통해 특정 파편에 대한 부하(Load)가 다른 파편에 영향을 주지 않도록 격리(Isolation)합니다.

**3. 제약 조건: 기본키의 중복**
수직 분할된 파편들은 독립적인 존재가 되지만, 원본 데이터를 재구성(Reconstruction)해야 할 때가 있습니다. 이를 위해 **모든 파편은 원본 테이블의 기본키(Primary Key)를 반드시 포함**해야 합니다. 기본키가 없다면, 나중에 `NATURAL JOIN` 연산을 통해 원본을 복원할 때 레코드 간의 정합성을 보장할 수 없기 때문입니다.

> **📢 섹션 요약 비유**: 수직 분할은 **'도서관의 분류 시스템'**과 같습니다. 한 책(레코드)에 '목차', '본문', '참고문헌'이 모두 있지만, 자주 찾는 '목차'는 따로 복사하여 현관에 걸어두고, '참고문헌'은 서고 깊숙한 곳에 따로 보관하는 것입니다. 책 번호(PK)가 적혀 있기 때문에 나중에 두 위치의 자료를 합쳐서 원래의 책 정보를 완벽하게 복원할 수 있습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

수직 분할의 아키텍처는 단순히 테이블을 쪼개는 것을 넘어, **분산 트랜잭션 관리**와 **데이터 재구성 비용**을 고려한 설계가 필요합니다.

#### 1. 구성 요소 및 매트릭스

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 (Internal Behavior) | 필수 요건 (Constraint) |
|:---|:---|:---|:---|
| **Primary Key (PK)** | **연결 고리 (Connector)** | 모든 파편(Fragment)에 중복 저장되어 데이터 연결성을 보장함 | NULL 불가, 유일성 보장 |
| **Non-Key Attribute** | **분할 대상 (Target)** | 액세스 패턴에 따라 그룹화되어 분산됨 | 함수적 종속성 유지 |
| **Fragment Schema** | **정의서 (Definition)** | 분할된 테이블의 구조를 정의 (SQL DDL) | 속성 분리 및 타입 일치 |
| **Reconstruction Algorithm** | **복구 로직 (Merger)** | 사용자 질의 시 필요한 파편을 동적으로 조인(JOIN)함 | 분산 조인 최적화 |
| **Allocation Map** | **위치 정보 (Locator)** | 각 파편이 어느 노드(Node)에 있는지 추적 | 메타데이터 관리 |

#### 2. 수직 분할 프로세스 (ASCII Flow)

아래 다이어그램은 `EMPLOYEE` 테이블을 업무 용도에 따라 `PUBLIC_INFO`(일반 정보)와 `SENSITIVE_INFO`(민감 정보)로 분할하는 과정을 도식화한 것입니다.

```text
      [ 수직 분할 (Vertical Fragmentation) 아키텍처 ]

      1. 원본 테이블 (Source Table)
      ┌────────────────────────────────────────────────────────────┐
      │  EMPLOYEE (PK: E_ID)                                      │
      ├──────────┬────────────┬──────────────┬───────────────────┤
      │  E_ID    │  E_NAME    │  E_PHONE     │  E_SALARY         │  (PK)   (Cols)
      │ (PK)     │  (VARCHAR) │  (VARCHAR)   │  (CONFIDENTIAL)   │
      └──────────┴────────────┴──────────────┴───────────────────┘
                  │             │                │
                  │             ▼                ▼
      ┌───────────┴─────────────────┐  ┌──────────────────────────┐
      │ [Criteria: High Freq Query] │  │ [Criteria: Security]      │
      ▼                             ▼  ▼                            ▼
  ┌─────────────────┐          ┌─────────────────┐      ┌─────────────────┐
  │  PUBLIC_INFO    │          │  PUBLIC_INFO    │      │ SENSITIVE_INFO  │
  │  (Fragment 1)   │          │  (Fragment 1)   │      │  (Fragment 2)   │
  ├─────────┬───────┤          ├─────────┬───────┤      ├─────────┬───────┤
  │ E_ID    │ E_NAME│  +       │ E_ID    │E_PHONE │      │ E_ID    │SALARY │
  │ (PK)    │ ...   │          │ (PK)    │...     │      │ (PK)    │...    │
  └─────────┴───────┘          └─────────┴───────┘      └─────────┴───────┘
       ▲ Node A (App)                ▲ Node A                 ▲ Node B (Secure)
       │                             │                        │
       └───────────────┬─────────────┘                        │
                       │                                      │
                       │          [Query Execution]           │
                       │          ────────────────           │
                       │                 ▼                   │
                       │       ┌───────────────────┐          │
                       └──────►│   NATURAL JOIN    │◄─────────┘
                               │   (E_ID matched)   │
                               └─────────┬─────────┘
                                         │
                                         ▼
                               [ Reconstructed View ]
```

**3. 심층 해설**
위 다이어그램에서 볼 수 있듯이, `E_ID`(Primary Key)는 **공유 애트리뷰트(Shared Attribute)**로서 모든 파편에 존재해야 합니다. 이 설계의 핵심 메커니즘은 다음과 같습니다.

1.  **단편화 (Fragmentation)**:
    *   **단계 1**: 친밀도 분석(Affinity Analysis)을 통해 함께 호출되는 확률이 높은 속성들을 그룹화합니다. (예: Name, Phone → Group A)
    *   **단계 2**: 보안 레벨(Security Clearance)이 다른 속성을 분리합니다. (예: Salary → Group B)
    *   **단계 3**: 생성된 각 파편에 PK를 복제합니다.

2.  **데이터 재구성 (Reconstruction)**:
    *   분산된 파편들을 `SELECT * FROM EMPLOYEE`와 같은 질의로 조회할 때, 시스템 내부적으로는 각 파편을 스캔한 뒤 `E_ID`를 기준으로 **Natural Join** 수행합니다.
    *   **성능 저하 요인**: 만약 사용자가 항상 두 파편의 정보를 모두 필요로 한다면, 매번 네트워크 오버헤드가 발생하는 조인 연산이 수행되므로 성능이 저하될 수 있습니다.

**4. 핵심 알고리즘 및 코드**
데이터베이스 내부에서 수직 분할을 복원하는 관계 대수 수식은 다음과 같습니다.
$$ R = \pi_{K \cup A_1}(R_1) \bowtie \pi_{K \cup A_2}(R_2) \bowtie \dots \bowtie \pi_{K \cup A_n}(R_n) $$
(여기서 $K$는 기본키, $A_n$은 속성 집합, $\bowtie$는 조인 연산)

```sql
-- [실무 SQL 예시] 분할된 테이블의 재구성
-- 클라이언트는 하나의 뷰(View)를 통해 투명(Transparent)하게 접근함

CREATE VIEW V_EMPLOYEE AS
SELECT
    P.E_ID, P.E_NAME, P.E_PHONE, S.E_SALARY
FROM
    PUBLIC_INFO P  -- Node A
JOIN
    SENSITIVE_INFO S -- Node B
ON
    P.E_ID = S.E_ID; -- PK 기반 조인

-- 실행 계획(Execution Plan) 상에서는
-- 각 노드로의 분산 스캔(Scan) 후 네트워크 전송 -> 조인 과정이 거쳐짐
```

> **📢 섹션 요약 비유**: 수직 분할 아키텍처는 **'깁(Gib) 기술'**과 유사합니다. 팔다리(속성)가 부러졌을 때, 붕대(Primary Key)를 이용해 단단하게 고정하면 본래의 기능을 수행할 수 있듯이, 서로 떨어진 파편들은 PK라는 '연결 고리'를 통해 수학적으로 완벽한 하나의 집합으로 합쳐질 수 있습니다. 하지만 붕대를 감고 다니는(조인 비용) 불편함은 감수해야 합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

수직 분할은 다른 분할 기법인 수평 분할(Horizontal Fragmentation)이나 비즈니스 로직과의 융합 측면에서 다각도로 분석해야 합니다.

#### 1. 심층 기술 비교표: 수직 vs 수평

| 비교 항목 (Metric) | 수직 분할 (Vertical Fragmentation) | 수평 분할 (Horizontal Fragmentation) | 비고 (Note) |
|:---|:---|:---|:---|
| **분할 기준** | **속성 (Attribute/Column)** | **튜플 (Tuple/Row)** | Vertical=Projection, Horizontal=Selection |
| **대응 관계 대수** | $\pi$ (Project) | $\sigma$ (Select) | 수학적 기반 연산 차이 |
| **PK 중복** | **필수 (Duplicate)** | **불필요 (Unique)** | 수직 분할은 복구를 위해 PK가 모든 조각에 있어야 함 |
| **데이터 재구성** | **Join ($\bowtie$)** | **Union ($\cup$)** | Union은 Join보다 일반적으로 비용이 낮음 |
| **주요 용도** | **보안 강화, 컬럼 접근 최적화** | **규모 확장성(Scalability), 부하 분산** | 데이터 양보다 접근 패턴이 중요할 때 사용 |
| **I/O 감소 효과** | 불필요한 컬럼 로드 제거 | 스캔 범위(Row Count) 감소 | 목적은 다르지만 둘 다 I/O 최적화가 목표 |

#### 2. 과목 융합 관점

**1) 네트워크 (Network)와의 시너지와 오버헤드**
*   **시너지**: 대용량 BLOB(Binary Large Object) 데이터(이미지, 영상)를 텍스트 데이터와 분리하여 별도의 스토리지 서버에 둘 때, 메인 네트워크 대역폭을 절약할 수 있습니다.
*   **오버헤드**: 수직 분할된 데이터를 조인하기 위해 네트워크 트래픽이 발생합니다. 만약 Node A와 Node B가 물리적으로 멀리 떨어져 있으면, **Latency(지연 시간)**가 급격히 증가하여 전체 쿼리 성능이 저하될 수 있습니다. 따라서 수직 분할은 동일한 LAN 내에서의 파티셔닝에 적합합니다.

**2) 보안 (Security)와의 융합**
*   **RBAC (Role-Based Access Control)**: 데이터베이스 사용자 권한을 테이블 단위가 아닌 파편(Fragment) 단위로 부여할 수 있습니다. 예를 들어, 일반 직원 계정에는 `PUBLIC_INFO` 테이블에 대한 `SELECT` 권한만 부여하고, `SENSITIVE_INFO` 테이블에 대한 접근은 `REVOKE`하여 보안 정책을 물리적으로 구현합니다.

> **📢 섹션 요약 비유**: 수직 분할과 수평 분할의 관계는 **'도시 계획'**과 같습니다. 수평 분할은 인구가 너무 많아서 **'새로운 신도시를 건설'**하는 것(Row 증가 대응)이라면, 수직 분할은 도시 내에서 **'주거 지역과 상업 지역을 기능적으로 분리'**하여 도로 혼잡을 줄이는 것(Column 접근 최적화)과 같습니다. 둘은 함께 사용될 수도 있지만, 해결하고자 하는 문제(Congestion vs Scalability)가 다릅니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

기술사 및 실무 설계자로서 수직 분할을 도입할 때는 성능 이점과 복잡성 사이의 균형을 판단해
