# 269. 수직 분할 (Vertical Fragmentation) - 속성 단위의 격리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 수직 분할은 테이블의 **애트리뷰트(열, Column)들을 사용 빈도나 보안 중요도에 따라 나누어** 서로 다른 물리적 노드에 분산 저장하는 기법이다.
> 2. **가치**: 관계 대수의 **프로젝트(Project, $\pi$) 연산**과 대응되며, 자주 함께 쓰이는 열들을 모아 저장함으로써 디스크 I/O를 최적화하고 민감 정보(연봉 등)를 물리적으로 분리 보호한다.
> 3. **융합**: 각 파편이 원본 레코드를 식별할 수 있도록 **기본키(Primary Key)를 반드시 중복 포함**해야 한다는 제약 조건 하에 데이터 재구성의 무결성을 확보한다.

+++

### Ⅰ. 수직 분할의 특징

- **정의**: 테이블의 수직적 단면을 잘라 단편을 만듭니다. 각 단편은 서로 다른 속성 집합을 가지지만, 원본 테이블의 **모든 행(Row)** 정보를 유지합니다.
- **분할 기준**: 속성 간의 친밀도(Affinity), 즉 얼마나 자주 같은 쿼리에서 동시에 호출되는지를 기준으로 그룹화합니다.
- **복구**: 모든 단편을 **기본키를 기준으로 자연 조인(Natural Join)** 하면 원본 테이블이 완벽하게 복원됩니다.

+++

### Ⅱ. 수직 분할 시각화 (ASCII Flow)

```text
[수직 분할 (Vertical) 아키텍처]

  Table: [ Employee ] (ID, Name, Phone, Salary)
  ┌────┬──────┬────────┬───────┐
  │ ID │ Name │ Phone  │ Salary│
  └────┴──────┴────────┴───────┘
          │ (Divide by Usage/Security)
          ▼                     ▼
    [ Basic_Info ]        [ Secret_Info ]
    ┌────┬──────┬──────┐  ┌────┬───────┐
    │ ID │ Name │ Phone│  │ ID │ Salary│  <-- ID(PK)는 양쪽 모두 포함!
    └────┴──────┴──────┘  └────┴───────┘
```

+++

### Ⅲ. 수직 분할의 효용성

1. **보안 강화**: 'Salary' 같은 민감 데이터는 고도의 보안이 적용된 독립 노드에 격리 저장할 수 있습니다.
2. **I/O 효율화**: 이름과 전화번호만 조회하는 쿼리가 많을 때, 무거운 연봉 데이터를 함께 읽어올 필요가 없어 캐시 효율이 높아집니다.
3. **스키마 진화**: 특정 속성 그룹의 타입 변경이나 확장이 다른 파편에 영향을 주지 않습니다.

- **📢 섹션 요약 비유**: 수직 분할은 **'개인정보 열람 제한'**과 같습니다. 회사 명부에서 '이름과 부서'는 누구나 볼 수 있게 게시판에 붙여두고(Public Fragment), '주민번호와 급좌번호'는 인사팀 금고(Secure Fragment)에 따로 보관하는 것과 같습니다. 사번(PK)만 알면 나중에 두 정보를 합쳐서 한 사람의 정보를 완성할 수 있습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[기본키 중복]**: 수직 분할의 필수 생존 조건.
- **[친밀도 매트릭스]**: 분할 지점을 결정하기 위한 통계적 도구.
- **[Join]**: 쪼개진 열들을 다시 하나로 묶는 수학적 연산 (수평 분할의 Union과 대조).

📢 **마무리 요약**: **Vertical Fragmentation**은 조준 사격과 같습니다. 필요한 데이터에만 시스템 자원을 집중시켜 성능과 보안이라는 두 마리 토끼를 동시에 잡아내는 정밀한 설계 기법입니다.
