# NoSQL 아키텍처 - 분산 데이터 모델과 샤딩 (Sharding)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NoSQL (Not Only SQL)은 초고성능·초가용성 요구에 대응하기 위해 고정된 스키마를 탈피하고, 공유 아무것도 없음 (Shared Nothing) 아키텍처 기반의 수평적 확장 (Scale-out)을 핵심으로 하는 분산 데이터 관리 기술이다.
> 2. **가치**: 데이터 모델의 유연성 (Schemaless)과 샤딩 (Sharding) 기술을 통해 페타바이트 (Petabyte)급 데이터와 수백만 TPS (Transactions Per Second)를 수용하며, CAP 정리 (Consistency, Availability, Partition Tolerance)의 제약을 비즈니스 요구에 맞춰 트레이드오프 (Trade-off) 한다.
> 3. **융합**: 현대의 NoSQL은 단순 저장소를 넘어 서버리스 (Serverless) DB, NewSQL과의 하이브리드 진화, 그리고 AI 임베딩 (Embedding) 검색을 위한 벡터 데이터베이스 (Vector DB) 기술로 융합되어 클라우드 네이티브 (Cloud Native) 인프라의 중추 역할을 수행한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: NoSQL (Not Only SQL)은 관계형 데이터베이스 (RDBMS)의 엄격한 ACID (Atomicity, Consistency, Isolation, Durability) 트랜잭션과 고정된 테이블 스키마의 제약을 벗어나, 대규모 분산 환경에서 탄력적 확장성과 고가용성을 보장하기 위해 설계된 비관계형 데이터베이스 시스템의 총칭이다.

- **💡 비유**: NoSQL은 "대형 물류 창고의 모듈형 적재 시스템"과 같다. RDBMS가 칸막이가 고정된 아파트형 서랍장이라면, NoSQL은 필요에 따라 상자를 더 쌓거나(Scale-out) 규격이 다른 물건(Schema-less)을 자유롭게 넣을 수 있는 유연한 적재소에 비유할 수 있다.

- **등장 배경**:
  1. **빅데이터의 폭발적 증가**: 3V (Volume, Velocity, Variety) 특성을 지닌 웹 2.0 및 소셜 미디어 데이터는 RDBMS의 수직적 확장 (Scale-up) 한계를 초월했다.
  2. **스키마 유연성 요구**: 기획 단계에서 예측 불가능한 다양한 데이터 필드 (Semi-structured Data)를 실시간으로 수용해야 하는 애자일 (Agile) 개발 문화가 확산되었다.
  3. **고가용성 (High Availability) 우선**: 네트워크 장애 상황에서도 서비스 중단을 허용하지 않는 웹 스케일 (Web-scale) 서비스들의 등장으로 일시적인 일관성 (Eventual Consistency)을 수용하는 BASE (Basically Available, Soft-state, Eventually consistent) 패러다임이 부상했다.

- **📢 섹션 요약 비유**: 마치 정해진 규격의 컨테이너만 실을 수 있는 전용 항구에서, 어떤 형태의 화물이라도 즉시 수용하고 선단을 자유롭게 늘릴 수 있는 스마트 항구 시스템으로 진화한 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### NoSQL 4대 데이터 모델 구성 요소

| 모델명 | 특징 | 핵심 메커니즘 | 대표 솔루션 | 비유 |
|:---|:---|:---|:---|:---|
| **Key-Value Store** | 가장 단순하고 빠른 모델 | 고유 키에 불투명한 값 (Opaque Value) 매핑 | Redis, DynamoDB | 이름표가 붙은 개인 사물함 |
| **Document Store** | 트리 구조 데이터 저장 | JSON/BSON 포맷, 필드 단위 인덱싱 가능 | MongoDB, CouchDB | 내용물이 상세히 적힌 택배 상자 |
| **Wide-Column Store** | 행마다 다른 열(Column) 보유 | LSM-Tree (Log-Structured Merge-Tree) 기반 쓰기 최적화 | HBase, Cassandra | 가변 길이의 파일 색인 폴더 |
| **Graph Store** | 관계 중심의 그래프 구조 | 인접 리스트 (Adjacency List) 기반 포인터 탐색 | Neo4j, Neptune | 복잡하게 얽힌 소셜 인맥 지도 |

### 분산 아키텍처의 핵심: 샤딩 (Sharding)

샤딩은 거대한 하나의 데이터베이스를 여러 개의 작은 조각 (Shards)으로 나누어 물리적으로 다른 노드에 분산 저장하는 기술이다. 이는 수평적 확장 (Scale-out)의 물리적 기반이 되며, 데이터 부하 분산과 병렬 처리를 가능케 한다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 NoSQL 샤딩 (Sharding) 아키텍처                     │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │     [Client Application]                                         │
  │              │                                                   │
  │     ┌────────▼────────┐                                          │
  │     │  Router / Proxy │ ─── (Shard Key 분석: User_ID=123)         │
  │     └────────┬────────┘                                          │
  │              │                                                   │
  │   ┌──────────┴──────────┬──────────┐                             │
  │   ▼                     ▼          ▼                             │
  │ ┌────────────┐    ┌────────────┐    ┌────────────┐               │
  │ │  Shard A   │    │  Shard B   │    │  Shard C   │               │
  │ │ (ID: 1~100)│    │(ID:101~200)│    │(ID:201~300)│               │
  │ └────────────┘    └────────────┘    └────────────┘               │
  │        │                 │                 │                     │
  │    [Node 1]          [Node 2]          [Node 3]                  │
  │                                                                  │
  │  - Horizontal Partitioning: 테이블 단위 분할                        │
  │  - Shared Nothing: 각 노드는 독립적인 CPU/Mem/Disk 보유              │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클라이언트의 요청이 들어오면 라우터 (Router) 또는 설정 서버 (Config Server)가 요청에 포함된 **샤드 키 (Shard Key)**를 분석한다. 예를 들어 `User_ID`가 123이라면, 해당 범위 (101~200)를 관리하는 `Shard B`로 쿼리를 전달한다. 각 샤드는 물리적으로 독립된 서버 (Node)에 위치하며, 자원을 공유하지 않는 **공유 아무것도 없음 (Shared Nothing)** 구조를 가진다. 이를 통해 특정 노드에 부하가 집중되는 것을 방지하고, 성능이 부족할 경우 새로운 노드를 추가하여 무한히 확장할 수 있는 토대를 마련한다.

### 일관된 해싱 (Consistent Hashing) 알고리즘

분산 환경에서 노드의 추가나 삭제 (Scale-in/out)가 빈번할 때, 데이터의 대규모 이동을 최소화하기 위해 사용되는 기법이다. 일반적인 해시 함수 (Hash % N)가 노드 수 N의 변화에 취약한 것과 달리, 일관된 해싱은 링 (Ring) 구조를 사용하여 재배치 비용을 획기적으로 낮춘다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │              일관된 해싱 (Consistent Hashing) 원리                 │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │              [Hash Ring (0 ~ 2^32-1)]                            │
  │                                                                  │
  │                    Node 1 (S1)                                   │
  │                 .      |      .                                  │
  │          Data A @      |       @ Data B                          │
  │             .          |          .                              │
  │     Node 3 (S3) -------+------- Node 2 (S2)                      │
  │             .          |          .                              │
  │          Data D @      |       @ Data C                          │
  │                 .      |      .                                  │
  │                                                                  │
  │  - 데이터 재배치: Node 2 장애 발생 시, Data B/C만 Node 3로 이동      │
  │  - 오버헤드: 전체 데이터의 K/N (K:데이터수, N:노드수)만 이동          │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터(Object)와 서버(Node)를 모두 동일한 해시 값 범위 (0 ~ 2^32-1)의 원형 링 위에 배치한다. 데이터는 시계 방향으로 가장 가까운 곳에 위치한 서버에 저장된다. 만약 `Node 2`가 제거되면, 해당 노드에 할당되었던 `Data B`와 `Data C`만 시계 방향의 다음 노드인 `Node 3`로 이동하면 된다. 기존의 단순 해싱 방식이 노드 수 변경 시 모든 데이터를 재배치해야 했던 것과 비교하면, 시스템의 가용성과 안정성을 비약적으로 높여주는 핵심 기술이다. 가상 노드 (Virtual Nodes) 개념을 추가하여 데이터의 불균형 (Hotspot) 문제도 해결 가능하다.

### 심층 동작 원리: LSM-Tree (Log-Structured Merge-Tree)

대부분의 쓰기 최적화 NoSQL (Cassandra, HBase 등)은 디스크의 랜덤 I/O를 최소화하기 위해 LSM-Tree를 저장 엔진으로 사용한다.

1. **MemTable (In-memory)**: 모든 쓰기 요청은 메모리 상의 정렬된 트리 (Skip List 등)에 먼저 기록된다.
2. **Commit Log (WAL)**: 장애 복구를 위해 쓰기 내용을 디스크 로그에 순차적으로 기록한다 (Write-Ahead Logging).
3. **SSTable (Sorted String Table)**: MemTable이 꽉 차면 디스크에 불변 (Immutable) 형태의 정렬된 파일로 플러시 (Flush)한다.
4. **Compaction (Background)**: 배경 프로세스로 여러 SSTable을 병합하며 중복되거나 삭제된 데이터 (Tombstone)를 정리한다.

- **📢 섹션 요약 비유**: 마치 포스트잇에 할 일을 적어두었다가(MemTable), 퇴근할 때 이를 정리해서 일기장(SSTable)에 붙이고, 나중에 중복된 내용을 지우고 한 권으로 묶는(Compaction) 효율적인 업무 관리법과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### RDBMS vs NoSQL 심층 비교

| 비교 항목 | RDBMS (SQL) | NoSQL (Not Only SQL) |
|:---|:---|:---|
| **데이터 모델** | 고정 스키마, 관계 지향 (Table) | 유연한 스키마, 목적 지향 (KV, Doc 등) |
| **확장성** | 수직적 확장 (Scale-up) 위주 | 수평적 확장 (Scale-out) 기본 |
| **트랜잭션** | ACID (강한 일관성) | BASE (결과적 일관성) |
| **조인 (Join)** | 강력한 표준 Join 지원 | 애플리케이션 레벨 Join 또는 비정규화 |
| **복잡도** | 복잡한 쿼리, 정규화된 설계 | 단순 쿼리, 대용량 처리 성능 위주 |
| **주요 철학** | 데이터 무결성 최우선 | 가용성 및 성능 최우선 |

### 과목 융합 관점

- **네트워크 (Network)**: 분산 노드 간의 데이터 동기화를 위해 가십 프로토콜 (Gossip Protocol)이나 쿼럼 (Quorum) 합의 알고리즘이 사용된다. 네트워크 파티션 (Network Partition) 발생 시의 대응 전략이 데이터 일관성 모델을 결정한다.
- **운영체제 (OS)**: 디스크 I/O 최적화를 위해 다이렉트 I/O (Direct I/O)나 OS 페이지 캐시 활용 전략이 NoSQL 성능의 핵심 변수가 된다. 특히 가비지 컬렉션 (GC) 포즈 (Pause) 시간 제어가 실시간 응답성에 큰 영향을 미친다.

- **📢 섹션 요약 비유**: RDBMS가 법규를 엄격히 준수하는 관료 조직이라면, NoSQL은 법규보다는 현장의 빠른 대응과 확장을 우선하는 특수 작전 부대와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 샤드 키 (Shard Key) 선정 전략

- **문제 상황**: 인스타그램과 같은 대형 SNS에서 특정 연예인의 게시물에 댓글이 폭주할 때, 해당 연예인의 `User_ID`를 샤드 키로 설정하면 특정 샤드에만 부하가 집중되는 **핫스팟 (Hotspot)** 현상이 발생한다.
- **의사결정**:
  1. **레인지 샤딩 (Range Sharding)**: 특정 범위의 키를 묶으므로 범위 검색에 유리하지만, 최신 데이터가 특정 샤드에 몰리는 핫스팟에 취약함.
  2. **해시 샤딩 (Hash Sharding)**: 샤드 키를 해싱하여 균등 분산시키므로 핫스팟 해결에 탁월하지만, 범위 검색 성능이 저하됨.
  3. **복합 샤드 키 (Compound Shard Key)**: `User_ID` + `Timestamp`와 같이 카디널리티 (Cardinality)가 높은 필드를 조합하여 분산도와 검색 성능의 균형을 맞춤.

### 도입 체크리스트
- **데이터 일관성 수준**: 강력한 일관성 (Strong Consistency)이 필수적인 금융권 거래인가, 아니면 결과적 일관성 (Eventual Consistency)으로 충분한 조회 위주의 서비스인가?
- **데이터 접근 패턴**: 특정 Key 기반의 점 질의 (Point Query)가 많은가, 아니면 전체 데이터를 스캔하는 분석 작업이 많은가?
- **리밸런싱 (Rebalancing) 오버헤드**: 노드 추가 시 데이터 이동이 서비스에 미치는 영향도와 자동화 수준을 확인했는가?

### 안티패턴: NoSQL에서의 무분별한 조인 (Join)
- NoSQL은 물리적으로 데이터가 파편화되어 있어 네트워크를 넘나드는 분산 조인 (Distributed Join) 수행 시 성능이 기하급수적으로 하락한다. 이를 극복하기 위해 데이터 중복을 허용하는 **역정규화 (De-normalization)**와 **임베디드 문서 (Embedded Document)** 패턴을 적극 활용해야 한다.

- **📢 섹션 요약 비유**: 샤드 키를 잘못 고르는 것은 마치 고속도로의 특정 차선에만 모든 차를 몰아넣어 거대한 정체를 만드는 것과 같으므로, 통행량을 고루 나누는 스마트한 분산 설계가 필수적입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (RDBMS 한계) | 도입 후 (NoSQL 적용) | 개선 효과 |
|:---|:---|:---|:---|
| **확장성** | 수십 대 서버 이상의 확장 곤란 | 수천 대 이상의 서버로 무한 확장 | **Scale-out 자유도 극대화** |
| **비용** | 고가의 하이엔드 서버 (Scale-up) | 저가의 범용 서버 (Commodity Hardware) | **인프라 비용 효율화 (TCO 절감)** |
| **개발 생산성** | 복잡한 정규화 및 스키마 변경 오버헤드 | 자유로운 데이터 구조 변경 | **Time-to-Market 단축** |

### 미래 전망
- **NewSQL과의 융합**: NoSQL의 확장성과 RDBMS의 ACID를 동시에 제공하는 Google Spanner, CockroachDB 등의 NewSQL이 시장을 잠식하고 있다.
- **멀티 모델 (Multi-model) DB**: 단일 엔진에서 Document, Graph, KV를 모두 지원하는 ArangoDB, CosmosDB 등이 부상하며 데이터 관리의 복잡성을 낮추고 있다.
- **Serverless & AI**: 관리형 NoSQL 서비스가 AI 모델의 장기 기억 장치인 벡터 데이터베이스 (Vector DB) 기능을 내장하며 지능형 애플리케이션의 핵심 저장소로 진화 중이다.

- **📢 섹션 요약 비유**: 마치 상황에 따라 자유자재로 모양을 바꾸는 트랜스포머 로봇처럼, 미래의 데이터베이스는 어떤 데이터와 부하 상황에서도 스스로 최적화되는 지능형 플랫폼으로 진화할 것입니다.
