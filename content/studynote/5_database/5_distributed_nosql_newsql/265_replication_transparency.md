# 265. 복제 투명성 (Replication Transparency) - 데이터 중복의 은닉

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 복제 투명성은 동일한 데이터가 가용성과 성능을 위해 **여러 노드에 중복 저장(Replication)되어 있다는 사실을 사용자로부터 은닉**하는 성질이다.
> 2. **가치**: 사용자가 단일 사본을 수정하듯 명령을 내리면, DDBMS가 내부적으로 모든 복제본의 정합성을 자동으로 조율하여 **데이터의 일관성을 유지**한다.
> 3. **융합**: 고가용성(HA) 아키텍처와 읽기 분산(Read-only Replica) 기술의 근간이 되며, 장애 발생 시 사용자가 이를 인지하지 못하게 하는 장애 투명성과 긴밀히 연결된다.

+++

### Ⅰ. 복제 투명성의 작동 원리

- **정의**: 데이터의 중복성을 감추어, 사용자가 마치 하나의 데이터베이스만 있는 것처럼 작업할 수 있게 하는 능력입니다.
- **주요 기능**:
    - **자동 전파**: 마스터에서 데이터 변경 시 모든 슬레이브로 변경 내역 자동 전송.
    - **최적 사본 선택**: 조회 시 가장 가까운 노드나 부하가 적은 노드의 사본을 자동으로 선택하여 응답.

+++

### Ⅱ. 복제 투명성 시각화 (ASCII Flow)

```text
[복제 투명성을 통한 자동 동기화]

  (User) "UPDATE Point = 100 WHERE ID = 1;"
       │
  ┌────▼──────────────────────────┐
  │ DDBMS (Replication Manager)   │
  │ "OK, 3개 노드에 모두 반영하겠음" │
  └────┬──────────┬──────────┬────┘
       ▼          ▼          ▼
  [ Node A ]  [ Node B ]  [ Node C ]
  (Updated)   (Updated)   (Updated)
       │
  (Result) ◀─ "Update Success"
```

+++

### Ⅲ. 복제 투명성의 과제

- **일관성 딜레마**: 모든 복제본을 동시에 수정하려면 속도가 느려지고(Strong Consistency), 속도를 위해 나중에 수정하면(Eventual Consistency) 투명성이 일시적으로 깨질 수 있습니다.
- **오버헤드**: 복제본이 늘어날수록 네트워크 트래픽과 스토리지 비용이 증가하므로 최적의 복제 계수(Replication Factor) 선정이 중요합니다.

- **📢 섹션 요약 비유**: 복제 투명성은 **'아이돌 가수의 분신술'**과 같습니다. 전 세계 수천 명의 팬(사용자)이 동시에 사인을 요청할 때, 가수(데이터)는 여러 명의 분신(복제본)을 보내 응답합니다. 팬들은 자신이 진짜 가수와 만났다고 믿으며(투명성), 가수는 모두 똑같은 사인(일관성)을 해주는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[장애 투명성]**: 한 복제본이 죽어도 다른 복제본으로 서비스하는 능력.
- **[2단계 커밋]**: 복제본 간의 완벽한 일관성을 위한 도구.
- **[Read Replica]**: 복제 투명성을 이용해 성능을 높이는 실무적 예시.

📢 **마무리 요약**: **Replication Transparency**는 분산 시스템의 신뢰도를 결정합니다. 복잡한 동기화 과정을 사용자 뒤로 숨겨 편리함과 안전함을 동시에 선사합니다.
