# 315. 데이터 피딩 및 스트리밍 파이프라인 - 멈추지 않는 데이터의 혈류

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 피딩(Feeding) 및 스트리밍 파이프라인은 데이터 발생원으로부터 타겟 저장소까지 **데이터를 중단 없이 실시간으로 수집, 변환, 전송하는 자동화된 일련의 흐름**을 의미한다.
> 2. **가치**: 기존의 배치(Batch) 처리가 가진 시간적 지연을 극복하여, 데이터가 생성된 즉시 가치를 창출하는 **실시간 분석 및 대응 체계**를 구축한다.
> 3. **융합**: Kafka, Flink, Spark Streaming 등의 분산 처리 엔진과 융합되어, 대규모 빅데이터를 안전하게 유통하는 현대 클라우드 데이터 아키텍처의 신경망 역할을 한다.

+++

### Ⅰ. 스트리밍 파이프라인의 구성 요소

1. **Source (공급원)**: 데이터가 발생하는 원천. (앱 로그, DB 변경분, IoT 센서 등)
2. **Ingestion (수집)**: 발생한 데이터를 유실 없이 받아들이는 단계. (Kafka, Kinesis)
3. **Processing (처리)**: 흐르는 데이터에 대해 실시간 필터링, 집계, 변환 수행. (Flink, Spark)
4. **Sink (적재)**: 최종 가공된 데이터를 저장하거나 시각화 도구로 전달. (S3, ES, Dashboard)

+++

### Ⅱ. 파이프라인 아키텍처 시각화 (ASCII Flow)

```text
[ Real-time Data Pipeline ]

  (Data Source) ──▶ [ Ingestion Layer ] ──▶ [ Processing Layer ] ──▶ (Target)
  ┌───────────┐     ┌───────────────┐     ┌────────────────┐     ┌───────────┐
  │ DB Changes│ ──▶ │ Kafka Topic   │ ──▶ │ Apache Flink   │ ──▶ │ Search DB │
  │ (CDC)     │     │ (Buffer)      │     │ (Transformation)│     │ (Sink)    │
  └───────────┘     └───────────────┘     └────────────────┘     └───────────┘
```

+++

### Ⅲ. 스트리밍 파이프라인의 핵심 과제

- **Exactly-once 보장**: 중복이나 누락 없이 정확히 한 번만 처리되도록 하는 고도의 트랜잭션 기술이 필요합니다.
- **Backpressure 관리**: 공급되는 데이터가 너무 많을 때 시스템이 터지지 않도록 속도를 조절하는 메커니즘이 중요합니다.
- **Latency vs Throughput**: 얼마나 빨리 보낼 것인가와 얼마나 많이 보낼 것인가 사이의 최적점을 찾아야 합니다.

- **📢 섹션 요약 비유**: 스트리밍 파이프라인은 **'수돗물 공급 시스템'**과 같습니다. 강물(데이터 원천)을 정수장(수집/처리)으로 끊임없이 끌어올려, 깨끗하게 걸러낸 뒤 각 가정의 수도꼭지(타겟 시스템)까지 중단 없이 전달하는 것과 같습니다. 물이 중간에 새거나(데이터 유실) 고이면(지연) 큰 문제가 발생하므로 정교한 펌프와 필터 관리가 핵심입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[CDC]**: DB의 변화를 스트리밍의 원천으로 바꾸는 기술.
- **[Windowing]**: 흐르는 데이터를 시간 단위로 묶어 분석하는 기법.
- **[Schema Registry]**: 파이프라인을 흐르는 데이터의 형식을 관리하는 카탈로그.

📢 **마무리 요약**: **Streaming Pipeline**은 정적인 데이터를 살아 움직이게 합니다. 실시간 데이터 피딩을 통해 비즈니스의 반응 속도를 극한으로 끌어올리는 것이 현대 IT의 지향점입니다.
