# 281. 샤드 키 (Shard Key) - 분산 배치의 나침반

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 샤드 키(Shard Key)는 데이터베이스 샤딩 환경에서 **각 데이터(레코드)가 어느 샤드(물리적 노드)에 저장될지를 결정하는 기준이 되는 하나 이상의 컬럼**이다.
> 2. **가치**: 데이터의 균등한 분산(Uniform Distribution)을 유도하여 특정 노드에 부하가 집중되는 '핫스팟(Hotspot)' 현상을 방지하고, 쿼리 라우팅의 효율성을 결정짓는다.
> 3. **융합**: 비즈니스 쿼리 패턴과 데이터의 카디널리티(Cardinality) 분석이 융합되어 선정되어야 하며, 잘못된 선정은 시스템 전체의 재구축(Re-sharding)을 초래할 수 있는 치명적 설계 요소다.

+++

### Ⅰ. 샤드 키 선정의 3대 원칙

1. **높은 카디널리티 (High Cardinality)**: 값의 종류가 많아야 합니다. (예: '성별'은 2가지뿐이라 부적절, '사용자ID'는 수백만 가지라 적절)
2. **균등한 분산 (Even Distribution)**: 특정 값에 데이터가 몰리지 않아야 합니다. (예: '날짜' 키는 오늘 데이터만 특정 서버에 몰릴 위험)
3. **쿼리 최적화**: 가장 자주 사용되는 `WHERE` 절의 조건 컬럼을 샤드 키로 써야 라우팅이 빨라집니다.

+++

### Ⅱ. 샤드 키 매핑 구조 시각화 (ASCII Flow)

```text
[Shard Key 기반 라우팅 매커니즘]

  (Record) { UserID: 501, Name: "A" }
               │
               ▼
  [ Sharding Logic ] ──▶ Hash(501) % 3 = 0
               │
       ┌───────┴───────┬───────────────┐
       ▼               ▼               ▼
  [ Shard 0 ]     [ Shard 1 ]     [ Shard 2 ]
  (IDs mod 3=0)   (IDs mod 3=1)   (IDs mod 3=2)
  ┌───────────┐   ┌───────────┐   ┌───────────┐
  │ User 501  │   │           │   │           │
  └───────────┘   └───────────┘   └───────────┘
```

+++

### Ⅲ. 샤드 키 유형별 비교

| 유형 | 설명 | 장점 | 단점 |
|:---|:---|:---|:---|
| **Hashed** | 키 값을 해싱하여 분산 | **완벽한 균등 분산** | 범위 검색(Range Scan) 불가 |
| **Ranged** | 특정 범위별로 분산 | **범위 검색 최적화** | 특정 범위에 쓰기 집중(Hotspot) |
| **Compound** | 여러 컬럼을 묶어서 사용 | 세밀한 제어 가능 | 관리가 복잡함 |

- **📢 섹션 요약 비유**: 샤드 키는 **'대형 물류 센터의 분류 태그'**와 같습니다. 택배 상자(데이터)에 붙은 주소(샤드 키)를 보고 자동 분류기(Router)가 1번 트럭으로 보낼지 2번 트럭으로 보낼지 결정합니다. 주소를 잘못 적으면 한쪽 트럭에만 짐이 넘치거나(Hotspot), 엉뚱한 동네로 가서 찾기가 힘들어집니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Hotspot]**: 샤드 키 설계 실패의 상징.
- **[Partition Key]**: 카산드라나 다이나모DB에서 샤드 키를 부르는 명칭.
- **[Scatter-Gather]**: 샤드 키가 쿼리에 없어 모든 샤드를 다 뒤져야 하는 비효율적인 상황.

📢 **마무리 요약**: **Shard Key**는 분산 DB의 명줄을 쥐고 있는 핵심 설계 자산입니다. 한 번 정하면 바꾸기 매우 어렵기 때문에, 서비스 초기 단계에서 가장 신중하게 결정해야 합니다.
