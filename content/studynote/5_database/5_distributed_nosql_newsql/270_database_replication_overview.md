# 270. 데이터베이스 복제 (Replication) - 생존과 분산의 이중주

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 복제는 가용성과 성능 향상을 위해 **동일한 데이터를 물리적으로 분리된 여러 서버(Node)에 중복하여 저장하고 관리**하는 기술이다.
> 2. **가치**: 특정 노드 장애 시 백업 노드로 즉시 전환(Failover)하여 서비스 연속성을 확보하며, 다수의 읽기 전용 복제본(Read Replica)을 통해 조회 부하를 효율적으로 분산한다.
> 3. **융합**: 복제 시점(동기/비동기)과 마스터 구조(Single/Multi)에 따라 다양한 아키텍처로 진화하며, 분산 시스템의 CAP 정리와 정합성 모델의 실질적인 구현체가 된다.

+++

### Ⅰ. 복제 방식의 분류 (Timing)

1. **동기식 복제 (Synchronous)**:
    - 마스터에 데이터를 쓸 때 모든 슬레이브에 반영될 때까지 기다립니다.
    - **장점**: 데이터 일관성 완벽.
    - **단점**: 응답 속도 느림, 하나의 노드만 장애가 나도 전체 쓰기 중단.
2. **비동기식 복제 (Asynchronous)**:
    - 마스터에 먼저 쓰고 나중에 슬레이브로 데이터를 전송합니다.
    - **장점**: 매우 빠른 응답 속도.
    - **단점**: 복제 지연(Replication Lag) 발생 시 데이터 불일치 위험.

+++

### Ⅱ. 복제 아키텍처 시각화 (ASCII Flow)

```text
[Master-Slave (Primary-Replica) 구조]

      (Write Request)          (Read Request)
             │                       │
      ┌──────▼──────┐         ┌──────▼──────┐
      │   Master    │ ──────▶ │   Slave 1   │ (Read-only)
      │ (Read/Write)│         └─────────────┘
      └──────┬──────┘         ┌─────────────┐
             └──────────────▶ │   Slave 2   │ (Read-only)
             (Replication)    └─────────────┘
```

+++

### Ⅲ. 복제의 목적과 효용

- **고가용성 (High Availability)**: 마스터가 죽어도 슬레이브가 마스터로 승격하여 서비스 유지.
- **성능 확장 (Read Scale-out)**: 조회 트래픽이 많을 때 슬레이브 노드만 늘리면 무한 확장이 가능함.
- **데이터 보호**: 디스크 장애 등 물리적 재난으로부터 원본 데이터를 보호하는 실시간 백업 역할.

- **📢 섹션 요약 비유**: 복제는 **'가수의 백업 댄서'**와 같습니다. 메인 가수(마스터)가 노래를 부를 때 뒤에서 똑같은 춤을 추고 있죠. 만약 가수가 넘어지면(장애), 가장 잘 추는 댄서가 마이크를 이어받아 공연을 계속(Failover)하게 하여 관객(사용자)들이 환불을 요구하는 사태를 막는 것입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Multi-Master]**: 모든 노드에서 쓰기가 가능한 고난도 복제 방식.
- **[Replication Lag]**: 비동기 복제의 피할 수 없는 숙명.
- **[CDC (Change Data Capture)]**: 복제를 구현하는 핵심 기술 (변경 로그 추출).

📢 **마무리 요약**: **Replication**은 현대 데이터베이스 가용성의 '생명줄'입니다. 비즈니스의 지연 시간 허용치와 정합성 요구 수준에 맞는 최적의 복제 전략을 수립해야 합니다.
