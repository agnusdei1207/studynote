# 306. 셰어드 낫띵 (Shared Nothing) 아키텍처 - 무한 확장의 기반

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 셰어드 낫띵(Shared Nothing)은 분산 시스템 내의 각 노드가 **CPU, 메모리, 디스크 등의 하드웨어 자원을 독점적으로 소유**하며, 노드 간에는 오직 네트워크 메시지를 통해서만 데이터를 주고받는 독립형 아키텍처다.
> 2. **가치**: 자원 경합(Contention)이 발생하지 않아 서버를 추가할수록 전체 시스템 성능이 선형적으로 증가하는 **완벽한 수평적 확장성(Scale-out)**을 제공한다.
> 3. **융합**: Google BigTable, Amazon Dynamo, Cassandra 등 거의 모든 현대적 NoSQL 및 대규모 데이터 분산 시스템의 설계 표준으로 융합되어 사용된다.

+++

### Ⅰ. 셰어드 낫띵(Shared Nothing)의 특징

- **독립성**: 각 노드는 자신만의 '작은 데이터베이스'를 운영합니다. 다른 노드가 무엇을 하는지, 어떤 자원을 쓰는지 간섭받지 않습니다.
- **확장성**: 중앙 집중식 자원(공유 디스크 등)이 없으므로 수천 대의 서버로 클러스터를 구성해도 성능 병목이 발생하지 않습니다.
- **내결함성**: 한 노드의 장애가 다른 노드의 하드웨어 자원에 물리적 영향을 주지 않으므로 장애 격리가 용이합니다.

+++

### Ⅱ. 아키텍처 비교 시각화 (ASCII Model)

```text
[ Shared Disk (RAC 등) ]           [ Shared Nothing (NoSQL 등) ]

      [Node A]  [Node B]                 [Node A]      [Node B]
         │         │                    (CPU/RAM)     (CPU/RAM)
         ▼         ▼                       │             │
    ┌──────────────────┐                   ▼             ▼
    │  Shared Storage  │              [Disk A]      [Disk B]
    └──────────────────┘
    (저장소 병목 가능성)               (독립적 처리, 무한 확장)
```

+++

### Ⅲ. 장단점 및 도전 과제

- **장점**: 하드웨어 구축 비용이 저렴하며(범용 서버 활용), 대규모 빅데이터 처리에 최적화되어 있습니다.
- **단점**: 
    - **데이터 중복**: 가용성을 위해 데이터를 여러 노드에 복제해야 합니다.
    - **소프트웨어 복잡도**: 조인(Join)이나 분산 트랜잭션 관리를 소프트웨어 레이어에서 직접 구현해야 하므로 개발 난이도가 높습니다.

- **📢 섹션 요약 비유**: 셰어드 낫띵은 **'각자 자기 도시락을 싸 온 학생들'**과 같습니다. 친구 도시락(자원)을 뺏어 먹을 수 없으므로 다툴 일이 없고(경합 없음), 학생이 100명이든 1,000명이든 각자 자기 밥을 먹으므로 점심시간(처리 시간)이 늘어나지 않는 아주 자유롭고 평등한 방식입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Scale-out]**: 셰어드 낫띵 아키텍처의 최종 목적지.
- **[Partitioning]**: 데이터를 노드별로 찢어 나누는 핵심 기술.
- **[Shared Everything]**: 모든 자원을 공유하는 전통적 메인프레임 방식.

📢 **마무리 요약**: **Shared Nothing**은 현대 분산 데이터 시스템의 황금률입니다. 자원의 독점과 네트워크 소통이라는 단순한 원칙이 거대한 클라우드 세상을 지탱하는 힘이 됩니다.
