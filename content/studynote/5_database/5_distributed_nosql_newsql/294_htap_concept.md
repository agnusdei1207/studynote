# 294. HTAP (Hybrid Transactional/Analytical Processing) - 트랜잭션과 분석의 경계를 허물다

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: HTAP은 **동일한 데이터베이스 아키텍처 내에서 온라인 트랜잭션 처리(OLTP)와 온라인 분석 처리(OLAP)를 동시에 수행**할 수 있게 하는 하이브리드 기술 패러다임이다.
> 2. **가치**: 운영 DB에서 분석 DB로 데이터를 옮기는 복잡한 **ETL 과정을 제거**하여, "방금 발생한 거래 데이터"를 즉시 분석에 활용하는 **리얼타임 비즈니스 인텔리전스(Real-time BI)**를 실현한다.
> 3. **융합**: 행 기반(Row-store)과 열 기반(Column-store) 엔진의 병렬 운영, 또는 인메모리 압축 기술과 융합되어 현대 NewSQL 데이터베이스의 핵심 경쟁력이 되고 있다.

+++

### Ⅰ. HTAP의 배경 및 등장 이유

- **전통적 방식의 한계**: 
    - OLTP(운영)와 OLAP(분석)이 물리적으로 분리되어 있어 데이터 동기화에 수 시간에서 수 일의 지연(Latency)이 발생했습니다.
    - 데이터 복제로 인한 중복 저장 비용과 인프라 관리 복잡도가 증가했습니다.
- **HTAP의 해법**: 단일 사본 혹은 실시간 복제본을 활용하여 운영계 데이터에 분석 엔진이 직접 접근하게 합니다.

+++

### Ⅱ. HTAP 아키텍처 모델 (ASCII Diagram)

```text
[전통적 구조 (Separated)]           [HTAP 구조 (Unified)]
  
  [OLTP DB] ──(ETL)──▶ [DW]         [ Unified HTAP DB ]
      │                 │           ┌─────────┬─────────┐
      ▼                 ▼           │  Row    │  Col    │
  (Service)         (Reports)       │ Store   │  Store  │
                                    └────┬────┴────┬────┘
                                         ▼         ▼
                                     (Service) (Real-time Analysis)
```

+++

### Ⅲ. HTAP의 주요 구현 기술

1. **In-Memory Column Store**: 행 기반 엔진 옆에 열 기반 인메모리 포맷을 추가하여 분석 쿼리를 가속합니다. (예: Oracle Database In-Memory)
2. **Real-time Replication**: 운영 노드에서 분석 전용 노드로 데이터를 밀리초(ms) 단위로 동기화합니다. (예: TiDB TiFlash)
3. **Smart Pushdown**: 쿼리 엔진이 데이터의 성격을 판단하여 트랜잭션은 행 저장소로, 대량 집계는 열 저장소로 자동으로 라우팅합니다.

- **📢 섹션 요약 비유**: HTAP은 **'오픈 키친이 있는 레스토랑'**과 같습니다. 예전에는 주방(OLTP)에서 요리한 음식을 식당(OLAP)으로 옮겨서 손님에게 내놓는 식이었다면, HTAP은 손님이 요리되는 과정을 실시간으로 보면서 즉석에서 취향에 맞게 간을 맞추고 피드백을 주는 것과 같습니다. 기다림 없는 신선한 데이터 요리가 가능해집니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Freshness]**: 분석 데이터의 최신성을 의미하는 HTAP의 핵심 지표.
- **[ETL-less]**: HTAP 도입으로 사라지게 될 수고로운 과정.
- **[NewSQL]**: HTAP을 가장 완벽하게 구현하고 있는 차세대 DB 그룹.

📢 **마무리 요약**: **HTAP**은 데이터 활용의 패러다임을 바꿉니다. "과거를 돌아보는 분석"에서 "현재를 실시간으로 통제하는 분석"으로 나아가는 기술적 도약입니다.
