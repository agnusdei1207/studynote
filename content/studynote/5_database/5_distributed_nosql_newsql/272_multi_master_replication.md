# 272. 멀티 마스터(Multi-Master) 복제 - 무한 확장의 양날의 검

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 멀티 마스터 복제는 분산 시스템 내의 **모든 노드가 읽기(Read)와 쓰기(Write) 연산을 동시에 수행**할 수 있으며, 서로의 변경 사항을 양방향으로 동기화하는 대칭형 아키텍처다.
> 2. **가치**: 특정 지역(Region)이나 노드에 국한되지 않는 글로벌 쓰기 성능을 제공하며, 어느 한 노드가 죽어도 다른 노드에서 즉시 쓰기가 가능해 **최고 수준의 가용성**을 확보한다.
> 3. **융합**: 서로 다른 노드에서 동시에 동일 데이터를 수정할 때 발생하는 **데이터 충돌(Conflict)**을 해결하기 위해, 벡터 시계(Vector Clock)나 LWW(Last Writer Wins) 같은 정교한 합의 알고리즘과 결합된다.

+++

### Ⅰ. 멀티 마스터 복제의 작동 원리

- **정의**: '마스터-슬레이브'의 종속 관계를 깨고, 모든 노드를 대등한 권한을 가진 피어(Peer)로 구성합니다.
- **특징**:
    - **Active-Active**: 모든 서버가 실시간으로 트래픽을 처리합니다.
    - **양방향 동기화**: 내가 쓴 데이터는 남에게 보내고, 남이 쓴 데이터는 내가 받아서 합칩니다.

+++

### Ⅱ. 멀티 마스터 아키텍처 및 충돌 (ASCII Diagram)

```text
[Multi-Master 충돌 시나리오]

      (User 1)               (User 2)
         │                      │
  ┌──────▼──────┐        ┌──────▼──────┐
  │   Node A    │ ◀──X──▶│   Node B    │
  │ (A = 100)   │  단절  │ (A = 200)   │
  └──────┬──────┘        └──────┬──────┘
         │                      │
         └──────────┬───────────┘
                    ▼
         💥 [ CONFLICT!! ] : "A의 진짜 값은 무엇인가?"
```

+++

### Ⅲ. 충돌 해결 전략

1. **LWW (Last Writer Wins)**: 타임스탬프가 가장 늦은 노드의 데이터를 최종본으로 인정합니다. (가장 단순함)
2. **Quorum 기반 합의**: 과반수 이상의 노드가 동의한 값을 선택합니다.
3. **Application Merge**: 충돌이 나면 사용자에게 알려서 수동으로 합치게 하거나, 비즈니스 로직(예: 장바구니 합치기)으로 자동 해결합니다.

- **📢 섹션 요약 비유**: 멀티 마스터는 **'각자 따로 쓰는 공동 일기장'**과 같습니다. 친구와 내가 각자의 집에서 일기를 쓰고 나중에 합치는 방식이죠. 둘 다 같은 날짜에 다른 내용을 썼다면(충돌), 나중에 만나서 내용을 적절히 섞거나 한 명의 글을 선택해야 하는 숙제가 남지만, 일기를 쓰는 동안에는 서로 기다릴 필요가 없어 아주 자유롭습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Active-Active]**: 멀티 마스터의 또 다른 이름.
- **[Conflict Resolution]**: 멀티 마스터 설계의 꽃이자 가장 어려운 부분.
- **[글로벌 서비스]**: 대륙별로 쓰기 노드를 두어야 하는 대규모 인프라의 필수 선택.

📢 **마무리 요약**: **Multi-Master Replication**은 극강의 가용성을 선사하지만, 데이터 정합성 관리라는 무거운 짐을 수반합니다. 성능과 일관성 사이의 치밀한 설계가 요구되는 고난도 기술입니다.
