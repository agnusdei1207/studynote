# 267. 데이터 분할 기법 (Fragmentation) - 물리적 최적화의 기술

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 분할(Fragmentation)은 하나의 전역 릴레이션을 물리적으로 더 작은 단위인 단편(Fragment)으로 쪼개어, **네트워크 전송 비용을 줄이고 노드별 데이터 접근 효율을 높이는 기술**이다.
> 2. **가치**: 데이터 로컬리티(Data Locality)를 향상시켜 쿼리 응답 시간을 단축하며, 필요한 속성이나 행만 골라 저장함으로써 시스템 전반의 I/O 부하를 분산한다.
> 3. **융합**: 수평 분할(Select 기반)과 수직 분할(Project 기반)이 융합되어 사용자의 요구사항에 가장 최적화된 물리적 데이터 배치를 완성한다.

+++

### Ⅰ. 데이터 분할의 정의와 원칙

- **정의**: 데이터베이스 테이블을 행(Row) 또는 열(Column) 단위로 나누어 서로 다른 물리적 위치에 저장하는 기법입니다.
- **3대 설계 원칙**:
    1. **완전성 (Completeness)**: 모든 데이터는 최소한 하나의 단편에 반드시 포함되어야 합니다. (유실 금지)
    2. **재구성 (Reconstructibility)**: 분할된 단편들을 다시 합치면(Union 또는 Join) 원래의 테이블로 완벽히 복구되어야 합니다.
    3. **상호 배제 (Disjointness)**: 하나의 데이터는 원칙적으로 단 하나의 단편에만 존재해야 합니다. (중복 방지)

+++

### Ⅱ. 분할 기법의 유형 (ASCII Visualization)

```text
[데이터 분할의 논리적 모형]

  Original Table [ Employee ]
  ┌────┬──────┬────────┬───────┐
  │ ID │ Name │ Dept   │ Salary│
  ├────┼──────┼────────┼───────┤
  │ 1  │ Kim  │ Sales  │ 500   │
  │ 2  │ Lee  │ IT     │ 600   │
  └────┴──────┴────────┴───────┘

  1. 수평 분할 (Horizontal): 행 단위로 자름
     - Fragment 1: Sales 직원들 (ID 1)
     - Fragment 2: IT 직원들 (ID 2)

  2. 수직 분할 (Vertical): 열 단위로 자름
     - Fragment A: (ID, Name, Dept)
     - Fragment B: (ID, Salary)  <-- PK(ID)는 반드시 포함!
```

+++

### Ⅲ. 실무적 장점

- **성능**: 자주 조회하는 'Sales' 부서 데이터를 'Sales' 부서 서버에 두어 네트워크 지연을 없앱니다.
- **보안**: 민감한 '연봉(Salary)' 정보만 별도의 보안 노드에 수직 분할하여 저장할 수 있습니다.
- **가용성**: 특정 분할(파편)이 깨져도 다른 분할의 데이터는 정상적으로 서비스 가능합니다.

- **📢 섹션 요약 비유**: 데이터 분할은 **'대형 피자를 조각내는 것'**과 같습니다. 한 판을 통째로 들고 먹기 힘들 듯(네트워크 병목), 적당한 크기로 조각내어(분할) 여러 사람(노드)에게 나눠주면 훨씬 빠르고 편하게 식사(데이터 처리)를 마칠 수 있습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[수평 분할]**: SQL의 Select 연산과 대응.
- **[수직 분할]**: SQL의 Project 연산과 대응.
- **[혼합 분할]**: 수평과 수직을 섞어 복잡한 요구사항 해결.

📢 **마무리 요약**: **Fragmentation**은 분산 DB 설계의 첫 단추입니다. 비즈니스 접근 패턴을 완벽히 분석하여 최적의 단편을 만들어내는 것이 성능의 핵심입니다.
