# 305. 오라클 RAC (Real Application Clusters) - 공유 디스크의 결집

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 오라클 RAC는 여러 대의 서버(인스턴스)가 **하나의 공유 디스크(Shared Disk)에 저장된 데이터베이스를 공동으로 사용**하여 서비스하는 클러스터링 기술이다.
> 2. **가치**: 특정 서버 장애 시에도 다른 서버가 즉시 서비스를 대행하는 **고가용성(HA)**과, 서버 추가를 통해 시스템 용량을 확장하는 **수직/수평 결합형 확장성**을 동시에 제공한다.
> 3. **융합**: 캐시 퓨전(Cache Fusion) 기술을 통해 노드 간 메모리 데이터를 고속 전송하여, 물리적 디스크 I/O 없이도 데이터 일관성을 유지하는 엔터프라이즈 RDBMS의 정수다.

+++

### Ⅰ. 오라클 RAC의 핵심 아키텍처

- **Shared Disk 구조**: 데이터 파일은 모든 노드가 접근 가능한 중앙 저장소에 있습니다.
- **Instance**: 각 서버마다 독립적인 메모리 영역(SGA)과 백그라운드 프로세스가 돌아가며, 동일한 DB 파일을 바라봅니다.
- **Cache Fusion**: 노드 A가 가진 데이터를 노드 B가 필요로 할 때, 디스크를 거치지 않고 전용 전송망(Interconnect)을 통해 메모리 대 메모리로 데이터를 전달하는 핵심 기술입니다.

+++

### Ⅱ. RAC 클러스터 시각화 (ASCII Flow)

```text
[ Oracle RAC Architecture ]

      [ Instance 1 ] ◀──(Interconnect)──▶ [ Instance 2 ]
      (SGA / CPU)      (Cache Fusion)     (SGA / CPU)
            │                                   │
            └───────────────┬───────────────────┘
                            ▼
                ┌──────────────────────┐
                │  Shared Storage      │
                │  (Data Files / Logs) │
                └──────────────────────┘
```

+++

### Ⅲ. RAC의 주요 이점

1. **고가용성 (High Availability)**: 서버 한 대가 고장 나도 세션이 살아있는 다른 서버로 즉시 이동하여 사용자 체감 장애 시간을 제로화합니다.
2. **부하 분산 (Load Balancing)**: 여러 서버에 골고루 트랜잭션을 분산하여 처리량을 극대화합니다.
3. **단일 데이터 뷰**: 분산 DB(Shared Nothing)와 달리 데이터 파편화가 없어 SQL 작성이 매우 편리합니다.

- **📢 섹션 요약 비유**: 오라클 RAC는 **'하나의 냄비를 여러 명의 요리사가 함께 쓰는 주방'**과 같습니다. 요리사(인스턴스)는 여러 명이지만 요리 재료(데이터)는 하나의 큰 냉장고(공유 스토리지)에 들어있습니다. 한 요리사가 아프더라도(장애) 다른 요리사가 즉시 그 요리를 이어받아 완성할 수 있는 아주 튼튼한 시스템입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Cache Fusion]**: RAC 성능의 90%를 결정짓는 노드 간 메모리 공유 기술.
- **[Shared Nothing vs Shared Disk]**: 확장성 중심(NoSQL)과 정합성 중심(RAC)의 아키텍처 차이.
- **[VIP (Virtual IP)]**: 클라이언트가 서버 장애를 인지하지 못하게 돕는 가상 주소.

📢 **마무리 요약**: **Oracle RAC**는 대규모 기업용 시스템의 안정성을 책임지는 상징적인 기술입니다. 복잡한 클러스터링 제어를 통해 "절대 멈추지 않는 DB"를 구현합니다.
