# 분산 데이터 처리와 합의 알고리즘 - MapReduce, Raft, Paxos

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 시스템의 신뢰성과 성능을 확보하기 위해 대규모 데이터를 병렬로 처리하는 **맵리듀스 (MapReduce)** 모델과, 여러 노드 간의 데이터 일관성을 보장하기 위한 **합의 알고리즘 (Consensus Algorithm)**이 필수적이다.
> 2. **가치**: 맵리듀스는 테라바이트급 이상의 데이터를 저가의 범용 서버 클러스터에서 고속으로 분석하며, 팍소스 (Paxos)와 래프트 (Raft)는 일부 노드의 장애 상황에서도 시스템 전체의 결정 (State Machine Replication)을 가능케 하여 무정지 가용성을 제공한다.
> 3. **융합**: 이러한 분산 처리 및 합의 기술은 현대의 NoSQL, NewSQL, 분산 파일 시스템 (HDFS), 그리고 블록체인 (Blockchain)의 합의 메커니즘으로 확장되어 분산 컴퓨팅의 이론적·실천적 토대를 형성한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 분산 데이터 처리 (Distributed Data Processing)는 단일 서버의 한계를 넘어 대규모 클러스터에서 데이터를 병렬로 분석하는 기술이며, 분산 합의 (Distributed Consensus)는 네트워크 지연이나 장애가 존재하는 환경에서 다수의 노드가 동일한 값(또는 순서)에 대해 합의를 도출하는 프로세스다.

- **💡 비유**: 맵리듀스는 "수백 명의 요리사가 각자 재료를 썰고(Map), 종류별로 모아(Shuffle), 최종 요리를 완성하는(Reduce) 거대 주방"과 같다. 합의 알고리즘은 "여러 명의 선장이 있는 배에서 폭풍우(네트워크 장애) 속에서도 한 명의 리더를 정하고 항로를 만장일치로 결정하는 선상 회의"에 비유할 수 있다.

- **등장 배경**:
  1. **데이터 스케일의 거대화**: 전통적인 단일 서버 분석 도구로는 수천 대 노드에 흩어진 비정형 데이터를 효율적으로 처리할 수 없게 되었다 (Google의 MapReduce 탄생 배경).
  2. **분산 시스템의 불가피한 결함**: 분산 환경에서는 메시지 유실, 순서 바뀜, 노드 중단이 일상적이며, 이를 극복하고 단일 시스템처럼 동작(Single System Image)하게 할 강력한 일관성 프로토콜이 필요해졌다.
  3. **가용성과 정합성의 상충 해결**: CAP 정리의 제약 하에서, 최대한의 가용성을 유지하면서도 데이터의 최종 정합성을 보장하기 위한 알고리즘 경쟁이 시작되었다.

- **📢 섹션 요약 비유**: 마치 수많은 부품이 각자 움직이면서도 하나의 정교한 기계처럼 작동하기 위해, 중앙 통제 장치(합의)와 효율적인 컨베이어 벨트(맵리듀스)가 조화를 이루는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 맵리듀스 (MapReduce) 프레임워크

Google이 제안한 대규모 데이터 병렬 처리 모델로, 복잡한 분산 프로그래밍을 `Map`과 `Reduce`라는 두 가지 단순한 함수로 추상화했다.

```text
  ┌──────────────────────────────────────────────────────────────────────┐
  │                   MapReduce 데이터 흐름 및 단계                        │
  ├──────────────────────────────────────────────────────────────────────┤
  │                                                                      │
  │ [Input] ──▶ [Map Step] ──▶ [Shuffle & Sort] ──▶ [Reduce Step] ──▶ [Output] │
  │                                                                      │
  │ (Raw Data)  (Key, Value)     (Group by Key)      (Aggregate)   (Final Result)│
  │      │            │                │                  │              │     │
  │      │     "Apple" -> 1            │                  │              │     │
  │ [Block 1] ──▶ [Mapper] ───┐        │           ┌──▶ [Reducer] ──▶ [File 1]   │
  │                           ├──▶ [Partition] ──▶┤                              │
  │ [Block 2] ──▶ [Mapper] ───┘        │           └──▶ [Reducer] ──▶ [File 2]   │
  │                                    │                                         │
  │  - Map: 데이터를 읽어 필터링하거나 변환 (중간 결과 생성)                      │
  │  - Shuffle: 중간 결과 중 동일한 Key를 가진 데이터를 특정 Reducer로 모음         │
  │  - Reduce: 모인 데이터를 집계하여 최종 결과 도출                              │
  └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터는 먼저 여러 개의 블록(Split)으로 나뉘어 각 노드의 **매퍼 (Mapper)**에게 전달된다. 매퍼는 `(Key, Value)` 쌍을 생성하며, 이 중간 데이터들은 **셔플 (Shuffle)** 단계에서 네트워크를 통해 동일한 Key를 가진 데이터끼리 그룹화되어 특정 **리듀서 (Reducer)** 노드로 이동한다. 리듀서는 이 그룹화된 데이터를 합산하거나 통계 처리하여 최종 결과를 출력한다. 이 과정에서 개발자는 분산 처리의 복잡한 로직(데이터 분할, 장애 복구, 노드 통신)을 신경 쓰지 않고 비즈니스 로직(Map/Reduce 함수)에만 집중할 수 있다.

### 2. 분산 합의 알고리즘: 팍소스 (Paxos) vs 래프트 (Raft)

분산 시스템에서 여러 노드가 하나의 상태 머신 복제 (State Machine Replication)를 유지하기 위해 합의에 도달하는 알고리즘이다.

#### 팍소스 (Paxos): 고전적이며 복잡한 합의
- **역할**: 제안자 (Proposer), 수락자 (Acceptor), 학습자 (Learner)로 구성.
- **특징**: "한 번 합의된 값은 절대 변하지 않는다"는 안정성을 수학적으로 증명했으나, 구현이 매우 복잡하고 이해하기 어렵기로 유명하다.

#### 래프트 (Raft): 이해 가능성을 중시한 현대적 합의
래프트는 팍소스와 동일한 성능과 안전성을 제공하면서도 리더 선출, 로그 복제, 안전성의 세 가지 하위 문제로 분할하여 이해하기 쉽게 설계되었다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 Raft 합의 알고리즘 상태 전이                        │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │       [Follower] <──── (Hearbeat) ───────┐                       │
  │           │                              │                       │
  │    (Timeout 발생)                        │                       │
  │           ▼                              │                       │
  │       [Candidate] ─── (과반수 투표 획득) ──▶ [Leader]                │
  │           │                              │                       │
  │    (다른 리더 발견) ◀─────────────────────┘                       │
  │                                                                  │
  │  - Leader: 클라이언트 요청 수신, 로그 생성 및 복제 주도               │
  │  - Follower: 리더의 지시 수동적 수행                                │
  │  - Term: 논리적 시간 단위, 리더 선출 시마다 증가                      │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 래프트 클러스터의 모든 노드는 **Follower**, **Candidate**, **Leader** 중 하나의 상태를 가진다. 정상 상태에서는 단 한 명의 **Leader**가 모든 요청을 처리하고 주기적으로 하트비트 (Heartbeat)를 보내 권위를 유지한다. 만약 Follower가 일정 시간 동안 하트비트를 받지 못하면 **Candidate**로 전환하여 투표를 요청하고, 과반수의 찬성을 얻으면 새로운 리더가 된다. 리더는 모든 변경 사항을 로그로 기록하고 다른 노드들에 복제한 뒤, 과반수의 복제가 확인되면 비로소 해당 변경을 **커밋 (Commit)**한다.

### 3. 심층 동작 원리: 쿼럼 (Quorum) 기반 합의

분산 시스템에서 정족수(Quorum)는 의사결정을 위해 필요한 최소한의 찬성 노드 수다. 통상적으로 `N/2 + 1`을 사용하여 네트워크 단절 시 두 개의 리더가 생기는 **스플릿 브레인 (Split Brain)** 현상을 방지한다.

- **Write Quorum (W)**: 쓰기 성공을 인정받기 위해 복제해야 하는 노드 수.
- **Read Quorum (R)**: 가장 최신 데이터를 읽기 위해 응답받아야 하는 노드 수.
- **조건**: `W + R > N` (N: 전체 노드 수). 이 조건을 만족해야 항상 최신 데이터를 포함한 노드로부터 값을 읽을 수 있음이 보장된다.

- **📢 섹션 요약 비유**: 마치 국회에서 법안을 통과시키기 위해 과반수 찬성이 필요한 것처럼, 분산 시스템도 과반수의 동의를 통해 결정을 확정 지음으로써 혼란(데이터 불일치)을 막습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 팍소스 (Paxos) vs 래프트 (Raft) 심층 비교

| 비교 항목 | 팍소스 (Paxos) | 래프트 (Raft) |
|:---|:---|:---|
| **설계 목표** | 수학적 완결성 및 안정성 | 이해 가능성 (Understandability) |
| **리더 개념** | 리더가 필수는 아님 (Multi-Paxos에서 추가) | 리더 중심 (Strong Leader) 아키텍처 |
| **상태 전이** | Proposer, Acceptor, Learner | Follower, Candidate, Leader |
| **로그 복제** | 각 항목마다 개별 합의 가능 | 리더가 로그의 순차적 복제를 엄격히 강제 |
| **구현 난이도** | 매우 높음 (실제 구현 시 변형 필수) | 상대적으로 낮음 (오픈소스 라이브러리 풍부) |
| **주요 사용처** | Google Chubby, Spanner | Etcd (Kubernetes), Consul, CockroachDB |

### 과목 융합 관점

- **빅데이터 (Big Data)**: 맵리듀스는 하둡 (Hadoop)의 핵심 엔진으로 사용되며, 스파크 (Spark)와 같은 인메모리 처리 프레임워크로 진화하는 모태가 되었다.
- **가상화/컨테이너 (K8s)**: 쿠버네티스의 상태 저장소인 `etcd`는 래프트 알고리즘을 사용하여 클러스터 전체의 설정 정보와 상태에 대한 일관성을 유지한다.
- **보안/블록체인 (Blockchain)**: 합의 알고리즘은 블록체인의 PoW (Proof of Work), PoS (Proof of Stake) 등으로 확장된다. 특히 장애 노드가 악의적으로 동작할 수 있는 환경을 대비한 **비잔틴 장애 허용 (BFT, Byzantine Fault Tolerance)** 연구의 기초가 된다.

- **📢 섹션 요약 비유**: 맵리듀스가 데이터를 요리하는 주방이라면, 래프트는 그 주방의 위생 상태와 레시피가 모든 체인점에서 동일하도록 관리하는 본사의 운영 매뉴얼과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 맵리듀스 성능 병목 해결 (Data Skew)

- **문제 상황**: 특정 해시 키에 데이터가 몰리는 **데이터 스큐 (Data Skew)** 현상으로 인해 특정 리듀서만 작업이 끝나지 않아 전체 작업 시간이 늘어나는 현상이 발생.
- **의사결정**:
  1. **컴바이너 (Combiner) 활용**: 맵 단계에서 중간 결과를 미리 부분 집계하여 네트워크로 전송되는 데이터 양과 리듀서의 부하를 대폭 감소시킴.
  2. **커스텀 파티셔너 (Custom Partitioner)**: 데이터 분포를 분석하여 부하가 큰 Key를 여러 리듀서로 강제 분산시키는 로직 적용.
  3. **샘플링 기반 리밸런싱**: 전체 데이터의 일부를 샘플링하여 데이터 분포를 파악하고 샤드 범위를 동적으로 조정.

### 도입 체크리스트
- **합의 알고리즘 선택**: 시스템의 복잡도를 감수하고라도 이론적 완결성이 중요한가 (Paxos), 아니면 빠른 개발과 유지보수가 중요한가 (Raft)?
- **노드 수 산정**: 합의 알고리즘은 보통 홀수 대(3, 5, 7)로 구성한다. 짝수 대 구성 시 과반수 확보가 어려워져 가용성이 오히려 떨어질 수 있음에 유의해야 한다.
- **메시지 오버헤드**: 노드 수가 늘어날수록 합의를 위한 통신 횟수가 기하급수적으로 증가하므로, 클러스터 규모에 따른 적정 노드 수 설정이 필요하다.

### 안티패턴: 모든 작업에 맵리듀스 적용
- 맵리듀스는 배치 처리에 특화되어 있어 시작 오버헤드가 크다. 실시간 응답이 필요한 API 서버나 소량의 데이터 처리에는 오히려 RDBMS나 단순 인메모리 캐시보다 훨씬 느릴 수 있다. "모든 것을 망치로 보지 마라"는 격언이 적용되는 대표적 사례다.

- **📢 섹션 요약 비유**: 맵리듀스에 컴바이너를 쓰는 것은 마치 쓰레기를 버리기 전 미리 압축해서 쓰레기차(네트워크)의 횟수를 줄이는 지혜와 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (전통적 방식) | 도입 후 (분산 합의/처리) | 개선 효과 |
|:---|:---|:---|:---|
| **처리량** | 단일 노드 성능에 의존 | 노드 수에 비례한 선형적 확장 | **데이터 처리량 100배+ 향상** |
| **가용성** | 마스터 장애 시 수동 복구 필요 | 자동 리더 선출 및 페일오버 | **시스템 가동률 99.99%+ 달성** |
| **신뢰성** | 분산 노드 간 데이터 불일치 위험 | 수학적/논리적 합의 보장 | **강력한 일관성 (Strong Consistency) 확보** |

### 미래 전망
- **스트리밍 분석 (Streaming Analytics)**: 맵리듀스의 배치 처리 한계를 넘어 Apache Flink, Kafka Streams 등 실시간 스트리밍 처리가 주류로 자리 잡고 있다.
- **양자 내성 합의**: 양자 컴퓨터의 등장에 대비하여 현재의 암호화 기반 합의가 아닌 새로운 물리적/수학적 합의 구조 연구가 진행 중이다.
- **지능형 합의 (AI-driven Consensus)**: 네트워크 상태와 노드 부하를 AI가 실시간 분석하여 리더를 최적으로 선출하고 메시지 전송 경로를 최적화하는 기술이 논의되고 있다.

- **📢 섹션 요약 비유**: 과거의 분산 처리가 단순히 "많은 사람이 일하는 것"이었다면, 미래의 분산 처리는 "가장 똑똑한 협업 방식을 스스로 찾아내는 유기체"처럼 변해갈 것입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **맵리듀스**는 학교 급식실에서 수백 명의 친구들 비빔밥을 만드는 것과 같아요. 누구는 나물을 씻고(Map), 종류별로 모아서(Shuffle), 한꺼번에 슥슥 비비면(Reduce) 금방 맛있는 비빔밥이 완성되죠!
2. **합의 알고리즘**은 놀이터에서 어떤 게임을 할지 정하는 약속이에요. 친구 중 몇 명이 화장실에 가도(장애), 남은 친구들이 투표해서 한 명의 대장(Leader)을 정하고 다 같이 같은 규칙으로 놀 수 있게 도와줘요.
3. 이 기술들 덕분에 전 세계 사람들의 엄청나게 많은 정보를 컴퓨터 수천 대가 실수 없이, 아주 빠르게 처리할 수 있는 거랍니다!
