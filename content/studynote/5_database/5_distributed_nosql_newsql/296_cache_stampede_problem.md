# 296. 캐시 스탬피드 (Cache Stampede) - 분산 시스템의 압사 사고

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 캐시 스탬피드는 대규모 트래픽이 몰리는 환경에서 **특정 인기 데이터의 캐시가 만료되는 순간, 수많은 요청이 동시에 DB로 몰려들어 시스템 전체를 마비시키는 현상**이다.
> 2. **가치**: 단일 장애 지점(SPOF)이 된 데이터베이스의 과부하를 방어하기 위한 필수 고려사항이며, 서비스의 안정성을 결정짓는 결정적 장애 시나리오다.
> 3. **융합**: PER(Probabilistic Early Recomputation), Mutex Lock, 분산 락(Distributed Lock) 기술과 결합되어 대규모 조회 트래픽을 지능적으로 분산 관리한다.

+++

### Ⅰ. 캐시 스탬피드의 발생 매커니즘

- **상황**: 캐시 서버에 'Hot Key' (예: 실시간 검색어 1위 정보)가 저장되어 있습니다.
- **사건**: 해당 키의 TTL(만료 시간)이 종료되어 데이터가 사라집니다.
- **결과**: 그 찰나에 들어온 수만 건의 사용자 요청이 캐시 미스(Miss)를 내고, 일제히 원본 DB로 질의를 던집니다. (Thundering Herd 현상)

+++

### Ⅱ. 현상 시각화 (ASCII Flow)

```text
[캐시 스탬피드: DB 폭격 상황]

  (Users) ──▶ [ Cache Server ]
                 │ (TTL Expired! 💥)
          ┌──────┴──────┬───────────────┐
          ▼             ▼               ▼
  (Request 1)   (Request 2) ... (Request N)
          │             │               │
          └──────┬──────┴──────┬────────┘
                 ▼             ▼
          [ ⚡️ Database ⚡️ ] ◀── Overloaded!
          "수만 명의 쿼리를 한꺼번에 처리할 수 없어!!" ──▶ Down!
```

+++

### Ⅲ. 해결 방안 및 방어 전략

1. **Mutex / Distributed Lock**: 
    - 캐시 미스 발생 시, 딱 한 명의 요청만 DB에 접근할 수 있게 락을 걸고, 나머지는 대기하거나 이전 데이터를 보게 합니다.
2. **PER (Probabilistic Early Expiration)**: 
    - 캐시가 만료되기 직전에 확률적으로 미리 캐시를 갱신(Recompute)하여 '만료 시점' 자체를 없앱니다.
3. **만료 시간 무작위화 (Jitter)**: 
    - 여러 키의 만료 시간이 한꺼번에 겹치지 않도록 TTL에 약간의 난수를 더해 분산시킵니다.

- **📢 섹션 요약 비유**: 캐시 스탬피드는 **'유명 맛집의 문이 열리는 순간'**과 같습니다. 줄 서 있던 수천 명의 손님(요청)이 한꺼번에 좁은 입구(DB)로 달려들면 사고가 납니다. 번호표(Lock)를 나눠주어 한 명씩 입장시키거나, 재료가 떨어지기 전에 미리 주문을 받아두는(PER) 전략이 필요합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Thundering Herd]**: 여러 프로세스가 한 자원을 얻으려 동시에 깨어나는 현상.
- **[Hot Key]**: 스탬피드의 주된 대상이 되는 인기 데이터.
- **[Cache-Aside]**: 스탬피드에 가장 취약한 캐시 패턴.

📢 **마무리 요약**: **Cache Stampede**는 조용한 암살자입니다. 평소엔 드러나지 않다가 트래픽 피크 타임에 시스템을 무너뜨리므로, 선제적인 **Locking 및 갱신 전략** 수립이 필수입니다.
