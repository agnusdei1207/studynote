# 308. 폴리글랏 퍼시스턴스 (Polyglot Persistence) - 데이터 저장의 다각화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 폴리글랏 퍼시스턴스는 하나의 애플리케이션 내에서도 **데이터의 성격(정형, 비정형, 관계 등)에 따라 가장 적합한 여러 종류의 데이터베이스 엔진을 선택하여 혼용하는 설계 전략**이다.
> 2. **가치**: "하나의 망치(RDBMS)로 모든 못을 박으려 하지 마라"는 원칙을 실현하여, 정합성이 중요한 결제 데이터는 RDBMS에, 고속 조회가 필요한 세션은 Redis에 저장하는 등 각 엔진의 성능을 극대화한다.
> 3. **융합**: 마이크로서비스 아키텍처(MSA)의 각 서비스별 독립적 데이터 저장소 선택(Database-per-service) 사상과 융합되어 현대 엔터프라이즈 시스템의 표준 아키텍처로 자리 잡았다.

+++

### Ⅰ. 폴리글랏 퍼시스턴스의 필요성

- **과거**: 모든 데이터를 거대한 오라클이나 MySQL 하나에 몰아넣어 관리 (Monolithic DB).
- **문제**: 소셜 인맥 탐색은 RDBMS에서 너무 느리고, 단순 로그 기록은 RDBMS 비용이 너무 비쌉니다.
- **해결**: 데이터의 용도(캐시, 검색, 관계, 트랜잭션)별로 최적의 엔진을 매칭합니다.

+++

### Ⅱ. 폴리글랏 아키텍처 시각화 (ASCII Flow)

```text
[ Modern Service Data Flow ]

      [ Application Layer ]
       /      |       |      \
      /       |       |       \
  [RDBMS]  [NoSQL]  [Graph]  [Cache]
 (Oracle) (MongoDB) (Neo4j)  (Redis)
    │        │        │        │
  [결제/원장][카탈로그][친구추천][세션관리]
```

+++

### Ⅲ. 영역별 최적 엔진 매칭 사례

1. **RDBMS (PostgreSQL, Oracle)**: 금융 거래, 정산, 고객 정보 등 강력한 ACID가 필요한 데이터.
2. **Document DB (MongoDB)**: 상품 설명, 블로그 포스트 등 스키마가 자주 변하는 콘텐츠.
3. **Graph DB (Neo4j)**: 친구의 친구 추천, 사기 탐지 등 복잡한 관계망 탐색.
4. **Search Engine (Elasticsearch)**: 대량의 텍스트 전문 검색 및 분석.
5. **Key-Value (Redis)**: 실시간 랭킹, 로그인 세션, 짧은 유효 기간의 임시 데이터.

- **📢 섹션 요약 비유**: 폴리글랏 퍼시스턴스는 **'요리 용도에 맞는 칼 선택'**과 같습니다. 과일을 깎을 때는 작은 과도(Cache)를 쓰고, 고기를 썰 때는 큰 식칼(RDBMS)을 쓰며, 뼈를 추릴 때는 전용 골발도(Graph DB)를 쓰는 것과 같습니다. 하나의 칼로 모든 요리를 할 수는 있지만, 용도에 맞는 칼을 쓸 때 가장 빠르고 맛있는 요리가 완성됩니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Multi-model DB]**: 하나의 엔진에서 여러 모델을 지원하려는 시도 (폴리글랏의 대안).
- **[Data Silo]**: 여러 DB를 쓰면서 발생하는 데이터 파편화 및 관리 복잡도 문제.
- **[ETL]**: 폴리글랏 환경에서 시스템 간 데이터를 동기화하기 위한 필수 과정.

📢 **마무리 요약**: **Polyglot Persistence**는 기술의 전문화와 다양성을 상징합니다. 복잡한 현대 비즈니스를 단일 엔진으로 감당하기보다, 적재적소에 최적의 무기를 배치하는 것이 진정한 아키텍처의 미학입니다.
