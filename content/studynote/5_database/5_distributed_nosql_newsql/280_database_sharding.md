# 280. 샤딩 (Sharding) - 무한 확장의 아키텍처

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 샤딩은 거대한 하나의 데이터베이스를 여러 개의 작은 데이터 조각인 '샤드(Shard)'로 나누어, **물리적으로 완전히 분리된 여러 서버에 분산 저장하고 관리하는 수평적 확장(Scale-out) 기술**이다.
> 2. **가치**: 단일 서버의 하드웨어 한계(CPU, 메모리, I/O)를 돌파하여 데이터량과 트래픽 증가에 선형적으로 대응할 수 있게 하며, 부하를 분산시켜 병목 현상을 원천 제거한다.
> 3. **융합**: 샤드 키(Shard Key) 선정 전략과 데이터 재배치(Rebalancing) 기술이 융합되어 현대 대규모 분산 시스템과 클라우드 네이티브 데이터베이스의 핵심 확장 매커니즘을 형성한다.

+++

### Ⅰ. 샤딩의 정의와 원리

- **정의**: '조각'이라는 뜻의 Shard에서 유래하였으며, 데이터를 수평적으로 분할하여 물리적으로 다른 DB 서버에 나누어 담는 과정입니다.
- **작동 방식**:
    - 애플리케이션은 **샤드 키**를 기준으로 쿼리가 어느 서버로 가야 할지 결정합니다.
    - 각 서버(샤드)는 전체 데이터의 일부분만 소유하며, 서로 독립적으로 동작합니다. (Shared Nothing)

+++

### Ⅱ. 샤딩 아키텍처 시각화 (ASCII Flow)

```text
[데이터 샤딩 및 라우팅 구조]

      (Application) ──▶ [ Sharding Proxy / Middleware ]
                               │ (Check Shard Key)
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
    [ Shard 1 (DB) ]     [ Shard 2 (DB) ]     [ Shard 3 (DB) ]
    (ID: 1 ~ 1000)       (ID: 1001 ~ 2000)    (ID: 2001 ~ 3000)
    ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
    │ [Data Part A]│     │ [Data Part B]│     │ [Data Part C]│
    └──────────────┘     └──────────────┘     └──────────────┘
```

+++

### Ⅲ. 샤딩의 핵심 고려사항

1. **샤드 키 (Shard Key)**: 데이터를 나누는 기준. 고르게 분산되지 않으면 특정 서버에만 부하가 쏠리는 'Hotspot' 문제가 발생합니다.
2. **조인 연산의 어려움**: 서로 다른 샤드에 있는 테이블끼리 조인하는 것은 네트워크 비용이 매우 커서 가급적 피해야 합니다.
3. **복잡성 증가**: 데이터가 늘어나 서버를 추가할 때 데이터를 다시 나누는 '리밸런싱' 작업이 매우 까다롭습니다.

- **📢 섹션 요약 비유**: 샤딩은 **'거대 도서관의 서가 분리'**와 같습니다. 한 건물(서버)에 책이 꽉 차서 더 이상 넣을 수 없다면, '인문학 건물', '자연과학 건물' 식으로 주제별(샤드 키)로 건물을 따로 짓고 책을 나눠 보관하는 것과 같습니다. 찾는 사람은 주제만 알면 어느 건물로 가야 할지 바로 알 수 있어 혼잡함이 사라집니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Horizontal Partitioning]**: 샤딩의 또 다른 학술적 명칭.
- **[Shard Key Selection]**: 샤딩의 성패를 가르는 가장 중요한 설계 요소.
- **[Consistent Hashing]**: 리밸런싱 비용을 줄이기 위한 고도화된 샤딩 기법.

📢 **마무리 요약**: **Sharding**은 빅데이터 시대의 생존 전략입니다. 관리의 복잡성을 대가로 무한한 확장성을 얻는, 현대 아키텍처의 가장 강력한 무기 중 하나입니다.
