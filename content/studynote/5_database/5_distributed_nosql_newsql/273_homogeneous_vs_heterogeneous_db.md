# 273. 동종 vs 이종 분산 DB 통합 - 하이브리드 인프라의 조율

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동종(Homogeneous) 분산 DB는 동일한 DBMS (Database Management System) 환경을 기반으로 하는 투명한 투명성(Transparency)을 제공하는 반면, **이종(Heterogeneous) 분산 DB는 서로 다른 데이터 모델(RDBMS vs NoSQL), 스키마, 쿼리 언어를 가진 시스템들을 논리적으로 통합하여 단일 시스템처럼 보이게 하는 추상화 계층**이다.
> 2. **가치**: 이종 통합은 'Silos(사일로)'화된 레거시 데이터의 연동을 가능하게 하며, 각 도메인별 최적의 DB엔진을 선택하는 **Polyglot Persistence (폴리글랏 퍼시스턴스)** 전략을 지원하여 전체 시스템의 성능과 유연성을 극대화한다.
> 3. **융합**: 통합을 위해 **Gateway (게이트웨이)**, **Wrapper (래퍼)**, **Global Schema (글로벌 스키마)** 기술이 필수적으로 사용되며, 분산 처리 트랜잭션 관리를 위해 **2PC (Two-Phase Commit)** 프로토콜 등 OS/네트워크 계층의 기술과 깊게 융합된다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 정의**
분산 데이터베이스 시스템(Distributed Database System, DDBS)은 논리적으로는 하나의 시스템처럼 보이지만, 물리적으로는 네트워크로 연결된 여러 노드에 데이터가 분산 저장된 시스템이다. 이를 구성하는 노드들의 DBMS가 동일한지 여부에 따라 **동종(Homogeneous) 분산 DB**와 **이종(Heterogeneous) 분산 DB**로 나뉜다.

*   **동종 분산 DB**: 모든 사이트가 동일한 DBMS 소프트웨어(예: 전 구간 Oracle)를 사용하며, 동일한 스키마 구조를 가진다. 사용자는 데이터가 어디에 있는지 알 필요가 없는 **Location Transparency (위치 투명성)**를 완벽하게 제공받는다.
*   **이종 분산 DB**: 서로 다른 벤더의 DBMS(예: Oracle, MySQL, MongoDB 등) 혹은 서로 다른 데이터 모델(Relational, Document, Graph 등)를 사용하는 노드들이 네트워크로 연결된 상태를 말한다. 이 경우 단순한 통신이 아닌, 데이터 의미(Semantics)와 포맷을 통일하는 복잡한 변환 과정이 필요하다.

**2. 💡 비유**
*   **동종**: "동일한 모델의 자동차로 구성된 레이싱 팀." 모든 부품이 호환되며, 운영 방식이 동일하다.
*   **이종**: "가솔린 차, 전기차, 수소차가 섞인 물류 팀." 연료(데이터 포맷)와 엔진(처리 방식)이 완전히 다르므로, 이들을 하나의 목적지로 움직이게 하려면 각 차량에 맞는 특수한 연결 장치(어댑터)가 필요하다.

**3. 등장 배경**
과거의 단일 벤더(Mainframe 중심) 환경에서 벗어나, 클라우드(Cloud) 컴퓨팅과 오픈소스의 발전으로 기업 내부에 다양한 기술 스택이 혼재하게 되었다. 비즈니스 요구사항이 복잡해짐에 따라, 하나의 DBMS로 모든 요구를 충족시키기 어렵게 되자 '적재적소에 맞는 최적의 도구를 사용하되 이를 통합 관리하자'는 니즈가 대두되었다.

**4. 📢 섹션 요약 비유**
> 마치 **'인터넷(Internet)'**과 같습니다. 전 세계의 서로 다른 운영체제(윈도우, 리눅스 등)와 하드웨어를 가진 컴퓨터들이, TCP/IP라는 표준 프로토콜과 공통된 주소 체계(IP)를 통해 **하나의 거대한 네트워크**인 것처럼 동작하는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

이종 분산 DB 통합의 핵심은 **'다양성을 투명하게 숨기는 것'**이다. 이를 위해 **Federated Database (페더레이티드 데이터베이스)** 혹은 **Middleware (미들웨어)** 기반의 아키텍처가 주로 사용된다.

**1. 구성 요소 (상세 표)**

| 요소명 (Component) | 역할 (Role) | 내부 동작 (Internal Behavior) | 관련 프로토콜/기술 | 비유 (Analogy) |
|:---|:---|:---|:---|:---|
| **Global Schema** | 전체적인 논리적 뷰 제공 | 사용자의 쿼리를 해석하여 어떤 로컬 DB에 접근할지 결정 | XQuery, SQL | 통번역팀장 |
| **Wrapper / Gateway** | DBMS별 인터페이스 변환 | 각 로컬 DB의 고유 API를 표준 인터페이스로 변환하고, 데이터 타입을 매핑 | ODBC, JDBC, REST API | 현지 통역사 |
| **Local Schema** | 실제 데이터 저장소 | 실제 데이터가 물리적으로 저장되고 관리되는 곳 | Native SQL | 각국의 문서고 |
| **Transaction Manager** | 분산 트랜잭션 관리 | 노드 간의 데이터 정합성을 유지하기 위한 2PC(2단계 커밋) 제어 | XA Protocol, 2PC | 교통 통제소 |
| **Query Processor** | 분산 쿼리 최적화 | 글로벌 쿼리를 하위 쿼리로 분해하고, 실행 계획을 수립하며 중간 결과를 병합 | Cost-based Optimization | 물류 센터 컨트롤 타워 |

**2. 이종 분산 DB 통합 아키텍처 (Federated Architecture)**

아래 다이어그램은 사용자가 단일 인터페이스를 통해 이종 DB에 접근하는 과정을 도시화한 것이다. 중간의 **Mediation Layer (중재 계층)**가 핵심이다.

```text
[User Application]
      │
      │ 1. Query (Global SQL/API)
      ▼
+-------------------------+
|   Global Schema Layer   |  <--- 사용자는 하나의 통합된 스키마만 인식
|  (Logical Integration)  |
+-----------+-------------+
            │
            │ 2. Parsing & Decomposition
            ▼
+-------------------------------------------------------+
|              Middleware / Query Processor             |
|  [A] Sub-Query 1        [B] Sub-Query 2    [C] REST   |
+----+-------------------+-------------------+----------+
     │                   │                   │
     │ (Wrapper)         │ (Wrapper)         │ (Wrapper)
     ▼                   ▼                   ▼
+----------+        +----------+        +-------------+
|  Oracle  |        |  MongoDB |        |   SAP ERP   |
|  (RDBMS) |        |  (NoSQL) |        | (Proprietary)|
+----------+        +----------+        +-------------+
[Site A]            [Site B]            [Site C]
```

**3. 다이어그램 해설**
이 아키텍처는 **Schema Integration (스키마 통합)**의 3단계 과정을 보여준다.
① **사용자**는 `Global Schema`에 대해 질의를 수행한다. 데이터가 어디에 있는지 모른다.
② **미들웨어**는 질의를 분석(Parsing)하여, 이를 각 로컬 DB가 이해할 수 있는 하위 질의(Sub-query)로 분해(Decomposition)한다.
③ **래퍼(Wrapper)**는 각 DB의 고유 언어(Oracle은 SQL, MongoDB는 MQL, ERP는 RFC 등)로 변환하여 명령을 전달한다.
④ 각 **로컬 DB**로부터 반환된 결과셋은 다시 미들웨어로 전달되며, 서로 다른 데이터 포맷(예: Relational Row vs Document JSON)이 **Common Data Model (공통 데이터 모델)**로 통합되어 사용자에게 최종적으로 반환된다.

**4. 핵심 알고리즘: 스키마 매핑 (Schema Mapping)**
이기종 시스템 간의 데이터 충돌을 해결하는 기술이다.

*   **이름 충돌 (Naming Conflict)**: 동일한 이름의 필드가 다른 의미를 가질 때 혹은 다른 이름이 같은 의미를 가질 때 해결.
*   **도메인 충돌 (Domain Conflict)**: 서로 다른 데이터 타입 간의 변환 (예: `Integer` ↔ `String`).
*   **구조 충돌 (Structural Conflict)**: 테이블 구조가 다를 때의 매핑 (예: 정규화된 테이블 ↔ 비정규화된 JSON).

**📢 섹션 요약 비유**
> 마치 **'UN 회의실'**과 같습니다. 영어, 불어, 중국어 등 각기 다른 언어를 쓰는 대표들이 각자의 언어(로컬 DB)로 발표하지만, 중앙의 **동시통역기(Gateway/Middleware)** 덕분에 청중은 마치 한 언어로 대화하는 것처럼 이해할 수 있습니다. 이때 통역사가 언어의 뉘앙스(데이터 타입과 의미)를 정확히 파악해 번역하는 것이 가장 중요합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

이종 분산 DB는 단순한 데이터베이스 기술을 넘어, 운영체제(OS), 네트워크, 애플리케이션 아키텍처와 밀접하게 연결된다.

**1. 심층 기술 비교: 동종 vs 이종 vs 중간**

| 구분 | 동종 분산 DB | 이종 분산 DB (완전) | 이종 분산 DB (Federated/Multi-model) |
|:---|:---|:---|:---|
| **DBMS 종류** | 단일 종류 (예: All Oracle) | 완전히 상이한 벤더/모델 혼합 | 단일 DBMS 내에 여러 모델 지원 (예: PostgreSQL+JSON) |
| **투명성 (Transparency)** | ⭐⭐⭐⭐⭐ (매우 높음) | ⭐⭐ (낮음, 개발자가 드라이버 인식 필요) | ⭐⭐⭐⭐ (중간 ~ 높음) |
| **쿼리 복잡도** | 단일 SQL 방언 사용 | 복합 쿼리 처리 및 매핑 필요 | 벤더 확장 기능 사용 가능 |
| **성능 (Join)** | 로컬 네트워크 수준의 빠른 Join | **네트워크 병목 및 변환 오버헤드 심함** | 엔진 내부 최적화로 상대적 양호 |
| **주요 과제** | 데이터 분산 및 일관성 유지 | **글로벌 트랜잭션 처리, 데이터 타입 매핑** | 스토리지 엔진 최적화 |

**2. 타 과목 융합 관점**

*   **[운영체제 OS]와의 융합 - 트랜잭션 처리**:
    이종 DB에서의 트랜잭션은 분산 환경에서의 **ACID (Atomicity, Consistency, Isolation, Durability)** 원칙을 준수해야 한다. 이를 위해 OS 레벨의 **RPC (Remote Procedure Call)**나 소켓 통신을 기반으로 **2PC (Two-Phase Commit) 프로토콜**이 구현된다.
    *   *Phase 1 (Prepare)*: 모든 참여 노드에게 "커밋 가능한가?"를 묻는다.
    *   *Phase 2 (Commit)*: 모두가 "Yes"하면 최종 커밋 명령을 내린다.
    *   *단점*: 일부 노드가 실패하면 전체 트랜잭션이 롤백되거나, locking으로 인해 성능이 급격히 저하될 수 있다.

*   **[네트워크]와의 융합 - 병목 현상**:
    이종 통합은 대용량 데이터를 실시간으로 변환하므로 네트워크 대역폭을 많이 소모한다. 이를 해결하기 위해 **Data Replication (데이터 복제)** 기술을 함께 사용하여 읽기 연산을 분산시키거나, 쿼리 결과를 **Cache (캐싱)** 하는 전략이 요구된다.

**📢 섹션 요약 비유**
> 마치 **'자동차와 기차를 이용한 화물 운송'**과 같습니다. 도로(네트워크)에서 자동차(SQL DB)가 달리고, 철로에서 기차(NoSQL DB)가 달리지만, 이 둘을 연결하는 **컨테이너 터미널(미들웨어)**에서 화물(데이터)을 트럭에 싣고 내리는 작업(변환/매핑)이 발생합니다. 터미널 처리 속도(변환 오버헤드)가 전체 배송 시간을 결정하는 가장 큰 병목 구간이 되는 셈입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. 실무 시나리오 및 의사결정**

*   **시나리오 A: 금융권의 레거시 통합**
    *   *상황*: 30년 된 메인프레임(COBOL/VSAM)과 최신 클라우드 RDBMS, 그리고 고객 분석용 MongoDB가 존재함.
    *   *의사결정*: 기존 데이터를 마이그레이션하는 것은 리스크가 크므로, **Federated Database (페더레이티드 DB)** 아키텍처를 도입하여 ETL(Extract, Transform, Load) 배치보다는 실시간 혹은 준실시간 **Data Virtualization (데이터 가상화)**을 통해 통합 조회 서비스를 구축한다.
    *   *결과*: 마이그레이션 비용 50% 절감, 단일 조회 창구 구현으로 생산성 향상.

*   **시나리오 B: 폴리글랏 퍼시스턴스 (Polyglot Persistence) 도입**
    *   *상황*: 전자상거래몰. 상품 목록은 검색이 중요하므로 Elasticsearch, 주문/결제는 트랜잭션이 중요하므로 MySQL, 고객 리뷰는 스키마가 자주 변하므로 MongoDB를 쓰고 싶음.
    *   *의사결정*: 애플리케이션 레벨에서 각 DB에 접속하는 DAO(Data Access Object)를 분리하여 개발한다. 전역 통합 쿼리보다는 **API Gateway**를 통해 각 DB별로 최적화된 마이크로서비스를 제공하는 방식을 택한다.

**2. 도입 체크리스트**

| 구분 | 점검 항목 (Checklist) | 비고 |
