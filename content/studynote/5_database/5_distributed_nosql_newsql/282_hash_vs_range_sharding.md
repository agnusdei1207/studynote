# 282. 해시 샤딩 vs 레인지 샤딩 - 분산의 알고리즘

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해시 샤딩(Hash Sharding)은 키 값을 해시 함수로 변환하여 데이터를 **무작위로 고르게 분산**시키는 방식이며, 레인지 샤딩(Range Sharding)은 키 값의 **특정 범위에 따라 데이터를 그룹화**하여 저장하는 방식이다.
> 2. **가치**: 해시는 데이터 집중(Hotspot) 방지에 탁월하여 대규모 쓰기에 유리하고, 레인지는 물리적 순서가 보장되어 범위 검색(Scan) 쿼리의 효율성을 극대화한다.
> 3. **융합**: 현대 분산 DB(MongoDB, TiDB 등)는 이 두 방식의 장단점을 고려하여 키 설계에 따라 선택적으로 적용할 수 있는 유연한 아키텍처를 제공한다.

+++

### Ⅰ. 두 샤딩 기법의 상세 비교

| 비교 항목 | 해시 샤딩 (Hash Sharding) | 레인지 샤딩 (Range Sharding) |
|:---|:---|:---|
| **분산 원리** | `Hash(Key) % NodeCount` | `Key >= 1 AND Key < 100` |
| **데이터 균등성** | **매우 높음** (랜덤 분산) | 낮음 (특정 범위에 쏠림 가능) |
| **범위 검색** | 불리 (여러 노드 다 뒤져야 함) | **매우 유리** (특정 노드만 접근) |
| **확장 시 부하** | 높음 (전체 데이터 재배치 필요) | 낮음 (특정 범위만 쪼개면 됨) |

+++

### Ⅱ. 샤딩 알고리즘 시각화 (ASCII Model)

```text
[ 1. Hash Sharding ]              [ 2. Range Sharding ]
 Key: 101 -> Hash -> Node 2        Key: 101 -> Range(100~200) -> Node B
 Key: 102 -> Hash -> Node 0        Key: 150 -> Range(100~200) -> Node B
 ┌────────┬────────┬────────┐      ┌────────┬────────┬────────┐
 │ Node 0 │ Node 1 │ Node 2 │      │ Node A │ Node B │ Node C │
 │ [102]  │        │ [101]  │      │(0~100) │(100~200)│(200~300)│
 └────────┴────────┴────────┘      └────────┴────────┴────────┘
   (골고루 퍼짐, 순서 없음)           (순서대로 모임, 쏠림 위험)
```

+++

### Ⅲ. 실무적 선택 기준

- **해시 샤딩 선택**: 
    - 사용자 ID 기반의 랜덤 접근이 대부분인 경우.
    - 특정 시간에 데이터가 몰리는 서비스(이벤트 응모 등)의 부하 분산.
- **레인지 샤딩 선택**: 
    - 날짜별 매출 조회, 이름순 정렬 목록 조회 등 범위 쿼리가 핵심인 경우.
    - 데이터 증설 시 특정 범위의 데이터만 마이그레이션하고 싶을 때.

- **📢 섹션 요약 비유**: 해시 샤딩은 **'화카(카드 섞기)'**와 같아서 모든 자리에 카드가 골고루 돌아가게 하는 공평함이 특징입니다. 레인지 샤딩은 **'사전의 색인'**과 같아서 'ㄱ'으로 시작하는 단어는 1권에, 'ㄴ'은 2권에 모아두는 방식입니다. 찾기는 편하지만 특정 글자(데이터)가 너무 많으면 그 책만 두꺼워지는 단점이 있습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Consistent Hashing]**: 해시 샤딩의 확장 시 데이터 재배치 문제를 해결한 기술.
- **[Shard Split]**: 레인지 샤딩에서 핫스팟 발생 시 노드를 쪼개는 기법.
- **[Data Skew]**: 데이터가 특정 샤드에 몰리는 현상 (주로 레인지 샤딩의 고민).

📢 **마무리 요약**: **Hash vs Range**는 정답이 없는 선택의 문제입니다. 서비스의 쿼리 패턴을 분석하여 '분산의 균형'과 '조회의 속도' 중 무엇이 더 중요한지 판단해야 합니다.
