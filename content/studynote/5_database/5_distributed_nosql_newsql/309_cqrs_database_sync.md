# 309. CQRS 아키텍처와 DB 동기화 - 읽기와 쓰기의 완벽한 이별

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CQRS(Command Query Responsibility Segregation)는 시스템의 **상태를 변경하는 명령(Command) 모델과 상태를 조회하는 쿼리(Query) 모델을 물리적으로 분리**하여 최적화하는 아키텍처 패턴이다.
> 2. **가치**: 쓰기(Write) 부하와 읽기(Read) 부하를 독립적으로 확장할 수 있게 하며, 복잡한 조회 연산을 위해 미리 계산된 '읽기 전용 모델'을 운영함으로써 검색 성능을 극대화한다.
> 3. **융합**: 이벤트 주도 아키텍처(EDA) 및 CDC 기술과 융합되어, 쓰기 DB의 변경 사항을 비동기적으로 읽기 전용 DB(Elasticsearch 등)로 동기화하는 현대적 데이터 파이프라인을 형성한다.

+++

### Ⅰ. CQRS의 핵심 원리

- **Command (쓰기)**: 데이터의 생성, 수정, 삭제를 담당. 비즈니스 로직과 제약 조건 검증이 중요합니다. (보통 RDBMS 사용)
- **Query (읽기)**: 데이터의 단순 조회를 담당. 사용자에게 보여줄 최종 형태의 데이터를 고속으로 반환하는 것이 목표입니다. (보통 NoSQL이나 검색 엔진 사용)
- **분리 이유**: 하나의 모델이 쓰기와 읽기를 모두 감당하면, 성능 최적화가 상충(예: 인덱스가 많으면 읽기는 빠르나 쓰기는 느려짐)되기 때문입니다.

+++

### Ⅱ. CQRS 동기화 아키텍처 (ASCII Flow)

쓰기 DB와 읽기 DB 사이의 정합성을 맞추는 과정이 CQRS의 핵심입니다.

```text
[ CQRS & Data Synchronization ]

      (Client Request)
       /            \
  [ Command API ]  [ Query API ]
       │              │
  ┌────▼────┐    ┌────▼────┐
  │ Write DB│    │ Read DB │ (Elasticsearch, Redis 등)
  │ (RDBMS) │    └────▲────┘
  └────┬────┘         │
       │ (Event/CDC)  │ (Sync)
       └──────────────┘
```

+++

### Ⅲ. DB 동기화 전략

1. **이벤트 발행 (Event Sourcing)**: 쓰기 작업 완료 후 '변경 이벤트'를 메시지 큐(Kafka)로 던지고, 읽기 서비스가 이를 구독하여 자신의 DB를 갱신합니다.
2. **CDC (Change Data Capture)**: DB 로그를 직접 읽어 실시간으로 읽기 DB에 변경분을 반영합니다. 애플리케이션 코드를 수정하지 않아도 된다는 장점이 있습니다.
3. **결과적 일관성**: 읽기와 쓰기가 분리되었으므로, 아주 짧은 찰나에 두 DB의 값이 다를 수 있음을 인정하고 설계해야 합니다.

- **📢 섹션 요약 비유**: CQRS는 **'레스토랑의 주문서와 영수증'**과 같습니다. 주방(Command)에서는 복잡한 조리 과정과 재료 관리가 일어나지만, 손님에게 나가는 계산서(Query)는 이미 계산이 끝난 깔끔한 결과물만 담겨 있습니다. 주방장이 요리를 하나 끝낼 때마다 계산서에 자동으로 품목이 추가(동기화)되도록 하는 시스템입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Materialized View]**: 쿼리 모델을 미리 계산해 둔 상태.
- **[Eventual Consistency]**: CQRS가 받아들여야 할 정합성의 한계.
- **[Task-based UI]**: CQRS와 잘 어울리는 사용자 인터페이스 설계 방식.

📢 **마무리 요약**: **CQRS**는 복잡한 시스템을 위한 고도의 처방전입니다. 읽기와 쓰기를 분리함으로써 얻는 성능적 이득과 아키텍처적 유연성은 대규모 서비스 성공의 발판이 됩니다.
