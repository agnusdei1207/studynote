# 271. 마스터-슬레이브 (Master-Slave) 복제 - 읽기 분산의 정석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마스터-슬레이브 복제는 **하나의 마스터(Primary)가 모든 쓰기 연산을 담당하고, 하나 이상의 슬레이브(Secondary)가 데이터 복제본을 유지하며 읽기 전용 요청을 처리**하는 비대칭형 아키텍처다.
> 2. **가치**: 데이터 수정 권한을 일원화하여 충돌을 방지하면서도, 조회 트래픽이 몰리는 웹 서비스 환경에서 슬레이브 증설(Scale-out)을 통해 압도적인 조회 성능을 제공한다.
> 3. **융합**: 고가용성(HA) 구성을 위해 슬레이브가 마스터의 상태를 실시간 모니터링하다가, 마스터 장애 시 투표를 통해 새로운 마스터로 승격되는 '자동 페일오버' 기술과 결합된다.

+++

### Ⅰ. 마스터-슬레이브의 역할 분담

- **마스터 (Master / Primary)**:
    - `INSERT`, `UPDATE`, `DELETE` 등 모든 데이터 변경 작업 수행.
    - 변경 내역을 로그(Binary Log 등)에 기록하여 슬레이브로 전송.
- **슬레이브 (Slave / Replica)**:
    - 마스터로부터 받은 로그를 자신의 DB에 반영하여 최신 상태 유지.
    - 사용자의 `SELECT` 요청만 처리하여 마스터의 부하를 경감.

+++

### Ⅱ. 데이터 흐름 및 복제 시각화 (ASCII Flow)

```text
[Master-Slave 복제 매커니즘]

      [ Application ]
       /          \
  (Write Only)   (Read Only)
     /              \
  ┌─▼────────┐      ┌─▼────────┐
  │  Master  │ ──▶  │  Slave 1 │ (복제본 A)
  │  (Node)  │ ──▶  │  Slave 2 │ (복제본 A)
  └──────────┘ 로그 └──────────┘
               전송
```

+++

### Ⅲ. 실무적 한계와 해결책

- **복제 지연 (Replication Lag)**: 마스터에 쓴 내용이 슬레이브에 반영되기까지 아주 짧은 시간차가 발생합니다. "방금 쓴 글이 목록에 안 보여요!"라는 민원의 주원인입니다.
- **해결책**:
    - **Semi-synchronous**: 최소 하나의 슬레이브가 받았음을 확인하고 응답.
    - **중요 읽기 강제**: 방금 쓴 데이터 등 민감한 조회는 일시적으로 마스터에서 직접 수행.

- **📢 섹션 요약 비유**: 마스터-슬레이브는 **'유명 맛집의 본점과 분점'**과 같습니다. 신메뉴 개발과 레시피 결정(쓰기)은 본점에서만 하고, 분점(슬레이브)들은 본점에서 보내준 레시피대로 요리해서 손님들에게 서빙(읽기)만 하는 구조입니다. 손님들이 많아지면 분점만 계속 늘리면 됩니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Failover]**: 슬레이브가 새로운 마스터가 되는 과정.
- **[Binary Log]**: 복제의 원천이 되는 변경 기록 파일.
- **[Read-heavy Workload]**: 마스터-슬레이브 아키텍처가 가장 빛나는 환경.

📢 **마무리 요약**: **Master-Slave Replication**은 현대 대규모 웹 서비스의 표준입니다. 쓰기와 읽기를 분리하는 단순하면서도 강력한 논리로 시스템의 수평적 확장을 이끌어냅니다.
