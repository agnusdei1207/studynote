# 262. 분산 데이터베이스의 6대 투명성 규칙

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 데이터베이스의 투명성(Transparency)은 시스템의 물리적 분산 구조와 복잡한 내부 메커니즘을 사용자로부터 은닉하여, **사용자가 마치 단일 시스템을 사용하는 것처럼 느끼게 만드는 설계 원칙**이다.
> 2. **가치**: 데이터의 위치, 분할 방식, 복제 여부 등을 몰라도 쿼리 수행이 가능하게 함으로써 **애플리케이션 개발의 복잡도를 획기적으로 낮춘다.**
> 3. **융합**: 위치, 분할, 복제, 병행, 장애, 성능(또는 지역 사상) 투명성 등 6가지 요소가 융합되어 완벽한 '분산 추상화'를 완성한다.

+++

### Ⅰ. 분산 DB의 6대 투명성 (Six Transparency Rules)

| 투명성 항목 | 설명 |
|:---|:---|
| **1. 위치 (Location)** | 데이터의 물리적 저장 위치를 몰라도 논리적 명칭만으로 접근 가능 |
| **2. 분할 (Fragmentation)** | 하나의 테이블이 여러 조각으로 나뉘어 있다는 사실을 은닉 |
| **3. 복제 (Replication)** | 동일 데이터가 여러 노드에 복사되어 있음을 사용자가 몰라도 됨 |
| **4. 병행 (Concurrency)** | 여러 사용자가 동시에 접근해도 데이터 일관성이 유지됨 |
| **5. 장애 (Failure)** | 특정 노드에 장애가 발생해도 시스템 전체는 무중단으로 동작 |
| **6. 성능 / 지역 사상** | 질의 속도가 균일하게 유지되거나, 지역적 매핑 구조를 몰라도 됨 |

+++

### Ⅱ. 투명성 계층 구조 시각화 (ASCII)

```text
[분산 DB 사용자 경험 계층]

  (사용자 / 앱) : "SELECT * FROM Employee;" (단순 쿼리)
       │
  ┌────▼────┐
  │  DDBMS  │ ──▶ [ 투명성 레이어 ] ──▶ (위치, 분할, 복제 계산)
  └────┬────┘
       │
  ┌────┴────┬─────────┬─────────┐
  │ [서울]  │ [부산]   │ [미국]   │ (물리적 서버들)
  └─────────┴─────────┴─────────┘
```

+++

### Ⅲ. 투명성 확보의 실무적 난제

- **분할 투명성의 한계**: 조인(Join) 연산 시 데이터가 여러 노드에 파편화되어 있으면 네트워크 트래픽이 폭증하여 성능 투명성이 깨지기 쉽습니다.
- **복제 일관성**: 복제 투명성을 완벽히 지키려면 모든 노드에 쓰기가 완료될 때까지 기다려야 하므로(동기 복제), 가용성이 낮아지는 트레이드오프가 발생합니다.

- **📢 섹션 요약 비유**: 6대 투명성은 **'클라우드 스트리밍 서비스'**와 같습니다. 사용자는 영화 파일이 어느 나라 서버에 있는지(위치), 용량을 줄이려 잘게 쪼개져 있는지(분할), 백업본이 몇 개인지(복제) 전혀 몰라도 재생 버튼 하나만 누르면 끊김 없이(장애) 영화를 감상할 수 있는 것과 같습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[위치 투명성]**: 가장 기본이 되는 투명성.
- **[DDBMS]**: 투명성을 물리적으로 구현하는 소프트웨어 엔진.
- **[데이터 독립성]**: 투명성을 통해 얻게 되는 공학적 이점.

📢 **마무리 요약**: **6대 투명성**은 분산 데이터베이스의 '성적표'입니다. 이 규칙들을 얼마나 완벽히 지키느냐가 분산 시스템의 기술적 완성도를 결정합니다.
