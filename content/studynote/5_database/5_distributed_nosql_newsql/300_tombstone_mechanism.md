# 300. 툼스톤 (Tombstone) 메커니즘 - 분산 삭제의 비석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 툼스톤(Tombstone)은 분산 데이터베이스에서 데이터를 즉시 물리적으로 삭제하는 대신, **해당 레코드가 삭제되었음을 알리는 특수한 마커(Marker)를 남기는 지연 삭제 기법**이다.
> 2. **가치**: 노드 간 동기화 지연이나 장애 발생 시, 삭제된 데이터가 다른 노드로부터 '부활'하는 현상을 방지하여 **분산 환경의 데이터 정합성을 유지**한다.
> 3. **융합**: LSM-Tree 기반의 스토리지 엔진과 결합되어, 불변(Immutable) 파일 구조인 SSTable에서 삭제 연산을 '쓰기'의 일종으로 처리한 뒤 나중에 콤팩션(Compaction) 시점에 실제 데이터를 정리한다.

+++

### Ⅰ. 툼스톤이 필요한 이유: 데이터 부활 문제

- **상황**: 3개의 노드(A, B, C)에 데이터 X가 복제되어 있습니다.
- **문제**: 노드 C가 장애인 동안 노드 A에서 X를 삭제했습니다. 나중에 C가 살아나서 A에게 "나 X 가지고 있어!"라고 복구 요청을 하면, A는 X가 원래 없었던 건지 삭제된 건지 알 수 없어 X를 다시 살려버릴 위험이 있습니다.
- **해결**: "X는 삭제됨"이라는 **비석(Tombstone)**을 세워두면, C가 나타나도 "그건 죽은 데이터야!"라고 명확히 알려줄 수 있습니다.

+++

### Ⅱ. 툼스톤 생명주기 시각화 (ASCII Flow)

```text
[ Tombstone Lifecycle ]

  1. DELETE Request ──▶ [ Active Data X ]
                          │
  2. Write Tombstone ──▶ [ Tombstone X ] (Status: Deleted, Time: 10:00)
                          │
  3. Replication ──────▶ All Nodes receive "Tombstone X" ✅
                          │
  4. GC / Compaction ──▶ (After 10 days...) 
                          │ 툼스톤과 원본 데이터를 모두 물리적 삭제 💾
                          ▼
                       [ Empty Space ]
```

+++

### Ⅲ. 툼스톤의 부작용과 관리

- **저장 공간 낭비**: 삭제가 많아지면 툼스톤 마커 자체가 용량을 차지하여 성능을 저하시킬 수 있습니다.
- **Compaction**: 주기적으로 데이터 파일을 병합하며 유효 기간이 지난 툼스톤을 제거하는 과정이 필수적입니다.
- **GC Grace Seconds**: 툼스톤이 삭제되기 전 모든 노드에 전파될 충분한 시간을 보장하기 위한 유예 기간 설정이 중요합니다.

- **📢 섹션 요약 비유**: 툼스톤은 **'재개발 구역의 빨간 딱지'**와 같습니다. 건물을 바로 허물기(물리 삭제) 힘든 상황에서, "이 건물은 곧 허물어질 것"이라는 딱지(툼스톤)를 붙여두어 외부인들이 입주하려는 것(데이터 부활)을 막는 것입니다. 동네 정비(Compaction) 기간이 되면 비로소 딱지가 붙은 건물들을 한꺼번에 철거합니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[LSM-Tree]**: 툼스톤을 '삭제 로그'로 활용하는 스토리지 구조.
- **[Ghost Record]**: 툼스톤 처리가 제대로 안 되어 다시 나타난 좀비 데이터.
- **[Compaction]**: 비석을 치우고 땅을 고르는 정리 작업.

📢 **마무리 요약**: **Tombstone**은 분산 데이터의 무결성을 지키는 마지막 약속입니다. 눈에 보이지 않는 비석을 통해 시스템의 과거와 현재를 완벽하게 조율합니다.
