# 295. 메모리 캐싱(Caching) 기술 - 데이터베이스의 가속기

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메모리 캐싱은 느린 디스크 기반 데이터베이스의 접근 부하를 줄이기 위해, **자주 사용되는 데이터를 고속의 RAM 영역에 임시 저장하여 응답 속도를 획기적으로 개선하는 기술**이다.
> 2. **가치**: 데이터베이스 본체의 I/O 병목을 해소하여 전체 시스템의 처리량(Throughput)을 수십 배 높이며, 예측 가능한 저지연(Low Latency) 서비스를 가능케 한다.
> 3. **융합**: Look-aside, Write-through 등 다양한 캐싱 전략과 Redis, Memcached 등의 전문 엔진이 융합되어 현대 대규모 분산 웹 아키텍처의 필수 계층으로 자리 잡았다.

+++

### Ⅰ. 주요 캐싱 전략 패턴

#### 1. Look-aside (Lazy Loading)
- 앱이 먼저 캐시를 확인하고, 없으면 DB에서 가져와 캐시에 저장하는 방식.
- **특징**: 캐시 장애 시에도 DB가 살아있으면 서비스 가능. (가장 대중적)

#### 2. Write-through
- 데이터를 쓸 때 캐시와 DB에 동시에 기록하는 방식.
- **특징**: 캐시 데이터가 항상 최신 상태를 유지하지만, 쓰기 지연 시간이 늘어남.

#### 3. Write-back (Write-behind)
- 캐시에 먼저 쓰고, 일정 시간 후나 배치로 DB에 반영하는 방식.
- **특징**: 쓰기 성능이 매우 빠르나, 캐시 장애 시 데이터 유실 위험.

+++

### Ⅱ. 캐싱 아키텍처 시각화 (ASCII Flow)

```text
[ Look-aside Caching Workflow ]

      (Client)
         │
  1. GET Data? ──▶ [ Application ] ── 2. Check Cache ─▶ [ Cache (Redis) ]
                          │                                │ (HIT!)
          ┌───────────────┴─────────────────┐              ▼
          │ (MISS)                          │        3. Return Data ✅
          ▼                                 ▼
  4. Query DB ──▶ [ Main DB ] ──────── 5. Update Cache
```

+++

### Ⅲ. 캐시 만료 및 삭제 정책 (Eviction)

캐시 용량은 한정되어 있으므로 똑똑한 삭제 규칙이 필요합니다.
- **LRU (Least Recently Used)**: 가장 오랫동안 사용되지 않은 데이터를 삭제.
- **LFU (Least Frequently Used)**: 사용 빈도가 가장 낮은 데이터를 삭제.
- **TTL (Time To Live)**: 설정된 유효 기간이 지나면 자동으로 만료.

- **📢 섹션 요약 비유**: 캐싱은 **'책상의 서랍'**과 같습니다. 모든 정보가 담긴 거대한 도서관(DB)에 매번 가는 대신, 자주 보는 참고서(데이터)를 책상 서랍(캐시)에 넣어두고 바로 꺼내 보는 것입니다. 서랍이 꽉 차면 안 쓰는 책을 빼고(Eviction) 새 책을 넣어야 하죠.

+++

### Ⅳ. 개념 맵 및 요약

- **[Cache Hit Ratio]**: 캐시 효율을 나타내는 결정적 지표.
- **[Redis]**: 현대 캐싱 기술의 사실상 표준(De-facto standard).
- **[Data Inconsistency]**: 캐시와 DB의 데이터가 달라지는 위험 요소.

📢 **마무리 요약**: **Caching**은 단순한 성능 향상을 넘어 시스템의 생존 전략입니다. 서비스의 읽기/쓰기 비중에 맞는 최적의 캐시 패턴을 적용하는 것이 아키텍트의 실력입니다.
