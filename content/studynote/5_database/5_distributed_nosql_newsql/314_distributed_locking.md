# 314. 분산 락 (Distributed Lock) - 파편화된 노드들의 약속

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 락은 물리적으로 분리된 여러 서버(Node)에서 실행되는 프로세스들이 **공유 자원에 접근할 때 상호 배제(Mutual Exclusion)를 보장하기 위해 사용하는 전역 잠금 매커니즘**이다.
> 2. **가치**: 단일 서버의 `synchronized`나 `Mutex`로 해결할 수 없는 분산 환경의 경쟁 상태(Race Condition)를 해결하여 데이터의 중복 처리나 정합성 파괴를 방지한다.
> 3. **융합**: Redis(Redlock), ZooKeeper(Ephemeral Nodes), MySQL(GET_LOCK) 등 외부 저장소의 원자적 연산 기술과 융합되어 시스템 전체의 원자성을 수호한다.

+++

### Ⅰ. 분산 락의 필요성

- **상황**: 선착순 이벤트가 진행 중이며, 서버 3대가 동시에 재고 DB에 접근합니다.
- **문제**: 각 서버의 로컬 락은 자기 안의 스레드만 막을 뿐, 다른 서버에서 들어오는 요청은 막지 못합니다. (초과 발급 발생)
- **해결**: 모든 서버가 공통으로 바라보는 '전역 락 저장소'를 둡니다.

+++

### Ⅱ. 분산 락 아키텍처 시각화 (ASCII Flow)

Redis를 활용한 가장 일반적인 분산 락 구현 사례입니다.

```text
[ Redis 기반 분산 락 (SETNX) ]

  Server A           Server B           Redis (Lock Store)
     │                  │                  │
  1. 락 획득 시도 ──▶   │ ── SETNX lock_key ─▶ (Success! 🔓)
     │                  │                  │
     │               2. 락 획득 시도 ──▶   │ ── SETNX lock_key ─▶ (Fail! 🔒)
     │                  │                  │
  3. 작업 수행...       │ (Wait/Retry)     │
     │                  │                  │
  4. 락 해제 ───────▶   │ ── DEL lock_key ──▶ (Removed)
     │                  │                  │
     │               5. 락 획득 성공 ──▶   │ ── SETNX lock_key ─▶ (Success! 🔓)
```

+++

### Ⅲ. 분산 락 구현 시 주의사항

1. **Deadlock 방지 (Expiration)**: 락을 쥔 서버가 갑자기 죽어도 락이 영원히 남지 않도록 반드시 **TTL(Time To Live)**을 설정해야 합니다.
2. **원자성 (Atomicity)**: "락 존재 확인"과 "락 설정"이 한 번의 연산으로 일어나야 합니다. (Lua Script 활용)
3. **정확한 해제**: 자기가 건 락만 해제할 수 있도록 고유한 Value(UUID 등)를 체크해야 합니다.

- **📢 섹션 요약 비유**: 분산 락은 **'공용 화장실의 열쇠'**와 같습니다. 방(서버)은 여러 개지만 화장실(공유 자원)은 하나뿐일 때, 카운터(Redis)에서 단 하나뿐인 열쇠를 먼저 받아 간 사람만 화장실을 쓸 수 있게 규칙을 정하는 것입니다. 열쇠를 가진 사람이 볼일(작업)을 보고 돌려줘야 다음 사람이 쓸 수 있습니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Redlock]**: Redis 클러스터 환경에서 신뢰성을 높인 분산 락 알고리즘.
- **[Lease Time]**: 락을 빌려주는 유효 시간.
- **[Fencing Token]**: 락 만료 후 뒤늦게 온 요청을 거르기 위한 단조 증가 번호.

📢 **마무리 요약**: **Distributed Lock**은 분산 시스템의 질서를 유지하는 신호등입니다. 성능과 정합성 사이의 세밀한 튜닝을 통해 시스템의 안전을 담보해야 합니다.
