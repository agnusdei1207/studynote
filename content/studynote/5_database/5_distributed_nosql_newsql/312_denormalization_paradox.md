# 312. 비정규화 (Denormalization)의 역설 - 성능을 위한 무결성의 양보

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비정규화는 정규화된 데이터베이스 아키텍처에서 시스템의 **조회 성능(Read Performance)을 향상시키기 위해, 의도적으로 데이터 중복을 허용하거나 테이블을 통합하는 물리적 설계 기법**이다.
> 2. **가치**: 고비용의 조인(Join) 연산을 제거하여 응답 시간을 단축하며, 특히 읽기 비중이 압도적인 NoSQL이나 대용량 분석 환경에서 필수적인 전략으로 사용된다.
> 3. **융합**: 데이터 중복으로 인한 **정합성 유지 비용 증가**라는 역설적 문제를 해결하기 위해, 애플리케이션 레벨의 보상 트랜잭션이나 실시간 동기화 기술과 융합되어야 한다.

+++

### Ⅰ. 비정규화의 주요 기법

1. **테이블 통합**: 1:1, 1:N 관계의 테이블을 하나로 합쳐 조인을 원천 차단합니다.
2. **중복 컬럼 추가**: 빈번하게 참조되는 컬럼을 조회 대상 테이블에 미리 복사해 둡니다.
3. **파생 컬럼(Derived Column) 생성**: 총합, 평균 등 계산 결과를 미리 컬럼에 저장하여 런타임 연산을 줄입니다.

+++

### Ⅱ. 비정규화 전후 비교 (ASCII Visualization)

```text
[ 정규화 모델 (Join 필요) ]           [ 비정규화 모델 (Direct Read) ]

  [ Order ] ──▶ [ Member ]             [ Order_Denormalized ]
  - OrderID      - MemberID            - OrderID
  - MemberID     - Name                - MemberID
  - Amount                             - MemberName (중복!) ✅
                                       - Amount
  (쿼리 시 Join 발생 -> 느림)            (단일 테이블 조회 -> 빠름)
```

+++

### Ⅲ. 비정규화의 역설 (Paradox)

- **성능 vs 정합성**: 읽기 속도는 비약적으로 빨라지지만, 원본 데이터가 수정될 때 중복된 모든 곳을 찾아 고쳐야 하는 '갱신 이상' 위험이 커집니다.
- **저장 공간**: 중복 데이터로 인해 스토리지 소모량이 증가합니다.
- **결론**: 비정규화는 '정규화의 실패'가 아니라, **'현실적인 성능 타협'**입니다. 반드시 정규화를 마친 모델을 바탕으로, 성능 측정 결과에 따라 꼭 필요한 곳에만 전략적으로 적용해야 합니다.

- **📢 섹션 요약 비유**: 비정규화는 **'요리 재료를 미리 손질해 두는 것'**과 같습니다. 손님이 올 때마다 재료를 씻고 다듬는 것(Join)이 정규화라면, 미리 다 썰어서 통에 담아두는 것(비정규화)은 요리 시간을 획기적으로 줄여줍니다. 하지만 재료가 상했을 때 모든 통을 다 확인해서 버려야 하는(정합성 관리) 수고가 따릅니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Trade-off]**: 비정규화 결정의 핵심 판단 기준.
- **[Read-heavy Workload]**: 비정규화가 가장 효과적인 환경.
- **[Anomalies]**: 비정규화 시 반드시 관리해야 할 데이터 이상 현상.

📢 **마무리 요약**: **Denormalization**은 성능을 향상시키는 강력한 약이지만, 과용하면 무결성 파괴라는 부작용을 낳습니다. 철저한 관리 체계 하에 '이유 있는 중복'을 허용하는 지혜가 필요합니다.
