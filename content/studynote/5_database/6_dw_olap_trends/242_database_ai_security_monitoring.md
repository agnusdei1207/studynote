# 데이터베이스 이상 탐지 및 보안 - AI 기반 모니터링과 쿼리 검사

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 현대의 데이터베이스 보안은 외부 방화벽과 패스워드를 통한 단순 접근 통제를 넘어, 사용자(User)와 애플리케이션의 **실시간 쿼리 패턴과 행동 맥락을 머신러닝으로 분석**하여 은밀한 내부자 유출과 악성 행위를 탐지하는 능동적 방어 체계다.
> 2. **가치**: UBA(User Behavior Analytics) 기반의 AI 모니터링은 평소에 `SELECT 1건`만 하던 직원이 갑자기 새벽 3시에 `SELECT * 10만 건`을 덤프(Dump) 뜨는 행위나, 알려지지 않은 신종 SQL 인젝션 변종을 즉각적으로 격리하여 천문학적인 개인정보 대량 유출(Data Breach)을 사전에 차단한다.
> 3. **융합**: 이는 단일 DB의 범주를 넘어 클라우드 통합 보안 정보 이벤트 관리(SIEM) 및 SOAR(보안 오케스트레이션) 시스템과 융합되며, 아무도 믿지 않고 모든 쿼리를 검증하는 **데이터 중심의 제로 트러스트(Zero Trust) 아키텍처**를 완성한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: AI 기반 DB 모니터링 및 이상 탐지(Anomaly Detection)는 데이터베이스 커널이나 네트워크 앞단(Proxy/Sniffer)에서 쏟아지는 수만 건의 SQL 로그를 수집한 뒤, 비지도 학습(Unsupervised Learning) 알고리즘을 통해 '정상적인 기준선(Baseline)'을 학습하고 이에서 벗어나는 쿼리를 실시간 차단하는 기술이다.

- **💡 비유**: 기존 보안(방화벽/비밀번호)이 "사원증이 있으면 무조건 금고 출입을 허락하는 경비원"이라면, AI 이상 탐지는 사원증을 찍고 들어온 직원이 평소와 다르게 가방에 금괴를 100개씩 쓸어 담는 것을 CCTV로 지켜보다가 "이건 평소 행동이 아니야!"라며 셔터를 닫아버리는 '지능형 프로파일러'와 같습니다.

- **등장 배경**:
  1. **내부자 위협 (Insider Threats)의 극대화**: 해킹 기술이 발달하면서 방화벽을 뚫기보다 권한을 가진 내부 직원(DBA)의 계정을 탈취하거나 피싱(Phishing)하여, 정상적인 SQL로 고객 정보를 통째로 빼가는 사고가 전체 유출의 70%를 차지하게 되었다.
  2. **Rule 기반 방어(WAF)의 한계**: 웹 방화벽(WAF)에 `" ' OR 1=1 "` 같은 시그니처(Rule)를 등록해 막았으나, 해커들이 난독화와 우회 기법을 끊임없이 만들어 내면서 사람이 일일이 룰을 등록하는 방식이 무력화되었다.
  3. **빅데이터화된 보안 로그**: 하루에 발생하는 DB 접근 쿼리 로그만 테라바이트(TB) 급에 달하여, 인간 보안 관제사가 이를 눈으로 보고 악성 여부를 판단하는 것이 물리적으로 불가능해졌다.

- **📢 섹션 요약 비유**: 성문을 튼튼하게 잠그는 방어(접근 통제)의 시대를 지나, 성 안에 이미 들어온 사람들이 혹시 스파이가 아닌지 일거수일투족을 감시하고 분석하는 감시(모니터링)의 시대로 보안의 패러다임이 이동했습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. AI 이상 탐지 아키텍처 (UBA / UEBA)

User and Entity Behavior Analytics(사용자 및 엔티티 행동 분석)는 각 DB 접속 계정별로 고유한 프로파일을 생성한다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 AI 기반 DB 쿼리 이상 탐지 및 차단 파이프라인           │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 1. 학습 페이즈 (Baseline Profiling - 비지도 학습) ]              │
  │  정상 쿼리 로그 ──▶ 특성 추출(Feature Extraction) ──▶ Baseline 구축 │
  │   - DBA 계정: DDL(CREATE, ALTER) 빈번, 낮 시간에 활동.               │
  │   - WAS 계정: 초당 1,000건의 짧은 SELECT/UPDATE, 새벽에도 활동.       │
  │                                                                  │
  │  [ 2. 실시간 모니터링 페이즈 (Real-time Inference) ]                 │
  │  (의심 이벤트 발생)                                                  │
  │  WAS 계정이 갑자기 DDL(DROP TABLE) 명령을 시도하거나,                    │
  │  새벽 2시에 수만 건의 `SELECT *` 대용량 풀 스캔을 시도함.                │
  │          │                                                       │
  │          ▼                                                       │
  │  [ 3. 머신러닝 이상 탐지 엔진 (Isolation Forest, AutoEncoder 등) ] │
  │          │ (편차 점수: Anomaly Score 99점 산출!)                     │
  │          ▼                                                       │
  │  [ 4. 보안 오케스트레이션 (SOAR) / DB 방화벽 (DBFW) ]               │
  │   - 해당 세션 강제 킬(Kill) 및 IP/계정 즉시 차단 (Zero Trust 실행)      │
  │   - 보안팀 관제 모니터에 적색 알람 전송                                 │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** AI 보안 솔루션(예: Imperva, IBM Guardium)은 설치 후 약 2~4주간 내부망에서 오고 가는 모든 쿼리를 조용히 학습(Machine Learning)하여, 각 계정이 주로 어떤 테이블에, 어느 시간에, 얼마나 많은 데이터를 건드리는지 다차원 벡터로 프로파일링한다. 이후 이 범주(Baseline)를 벗어나는 쿼리(예: 영업팀 직원이 인사팀 테이블을 조인함, WAS가 평소 반환량의 100배인 1GB 데이터를 Fetch함)가 들어오면, 정해진 룰이 없더라도 편차값(Score)을 계산하여 자동으로 쿼리를 끊어(Session Terminate) 대형 유출을 막는다.

---

### 2. 구문 분석 기반(Syntax Tree) SQL 인젝션 방어

단순 문자열 매칭(Rule)을 우회하는 해커를 막기 위해, DB 파서(Parser) 원리를 역이용한 보안 기술이다.

- **원리**: 쿼리가 들어오면 실행하기 전에 AI 프록시가 먼저 쿼리를 컴파일하여 **추상 구문 트리(AST, Abstract Syntax Tree)** 구조로 분해한다.
- **방어 효과**: 앱에서 보낸 정상 쿼리 `SELECT * FROM User WHERE id = ?`의 구문 트리를 미리 기억해 둔다. 만약 해커가 `id` 입력칸에 `1 OR 1=1`을 넣어 쿼리의 논리적 깊이(Depth)나 `OR` 노드가 비정상적으로 추가되면, 트리 구조가 변형된 것을 감지하고 구조적 SQL 인젝션으로 즉시 판별하여 드랍(Drop)한다.

---

### 3. 클라우드 네이티브 감시 (Cloud Posture Management)

AWS Macie, Azure Defender for SQL 같은 클라우드 통합 보안 서비스다.

- **기능**: 데이터베이스 내부에 '주민번호', '여권번호' 같은 개인식별정보(PII)가 평문으로 저장되어 있는지 AI가 정규식과 자연어 처리(NLP)를 돌려 지속해서 스캔하고, 만약 해당 컬럼에 TDE(투명한 암호화) 설정이 꺼져 있다면 강제로 경고를 띄워 컴플라이언스(GDPR) 위반을 사전에 방지한다.

- **📢 섹션 요약 비유**: AI 쿼리 분석은 은행 창구 직원의 '말투와 동공의 떨림(트리 구조와 패턴)'까지 분석하여, 이 사람이 진짜 직원이 맞는지 아니면 직원 얼굴 가죽을 뒤집어쓴 도둑인지(탈취된 계정)를 0.1초 만에 파악하는 고도의 심리 탐지기입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### DB 보안 관제 패러다임 비교 (Rule 기반 vs AI 기반)

| 비교 항목 | 전통적 Rule 기반 (DB 방화벽, WAF) | AI 기반 모니터링 (UBA / 머신러닝) |
|:---|:---|:---|
| **탐지 원리** | 룰셋 시그니처 (예: `DROP`, `UNION` 금지) | 정상 행동 패턴(Baseline)과의 편차(Deviation) |
| **미탐 (False Negative)**| 해커가 룰을 우회(난독화)하면 100% 뚫림 | 알려지지 않은 제로데이(Zero-day) 공격도 탐지 가능 |
| **오탐 (False Positive)**| 관리자 업무 시에도 차단되어 예외 처리(White-list) 지옥 발생 | 맥락(Context)을 이해하므로 정교한 예외 판단 가능 |
| **내부자 위협 방어** | 합법적 권한을 가진 내부자의 정보 대량 유출 방어 불가 | **행동 패턴 변화 감지로 내부자 대량 유출 완벽 차단** |
| **유지보수비용** | 보안 담당자가 매일 새로운 해킹 룰 업데이트 | 스스로 학습하므로 수동 룰 관리 리소스 90% 이상 절감 |

### 과목 융합 관점

- **머신러닝과 AI 알고리즘**: 정상과 비정상을 구별하기 위해 클러스터링 알고리즘(K-Means, DBSCAN)이나 신경망 기반의 오토인코더(AutoEncoder)가 주로 사용된다. 정상 쿼리 구조를 압축했다가 복원할 때, 악성 쿼리는 구조가 복원되지 않는 성질(Reconstruction Error)을 이용하여 Anomaly Score를 도출한다.
- **분산 시스템과 스트림 처리**: 수만 대의 DB 서버에서 발생하는 초당 수백만 건의 Audit(감사) 로그를 지연 없이 분석하기 위해, Apache Kafka와 Flink 같은 실시간 스트림 처리 기술이 AI 모델 추론 파이프라인의 밑단을 강력하게 융합 지탱하고 있다.

- **📢 섹션 요약 비유**: Rule 기반 방어는 "마스크를 쓴 사람은 은행 출입 금지"라는 벽보를 붙여놓는 것이고, AI 방어는 "어제까지 오른손잡이던 VIP 고객이 갑자기 왼손으로 엄청난 돈을 찾으려 한다"는 수상한 행동 징후를 알아채고 지점장에게 콜을 넣는 방식입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 외주 개발자에 의한 고객 정보 대량 유출(Dump) 방어

- **문제 상황**: 권한을 부여받은 유지보수 외주 개발자가 새벽에 퇴사 전 사보타주 목적으로 `SELECT * FROM CUSTOMER_INFO` 쿼리를 날려 5천만 명의 고객 DB를 엑셀로 내려받기(Dump) 시도함. 기존 권한 제어(IAM)로는 막을 명분이 없어 시스템상 정상 처리됨.
- **의사결정 (AI 모니터링 연동 DBFW 도입)**:
  1. DB 앞단에 설치된 AI 에이전트가 해당 쿼리를 캡처함.
  2. "이 계정(외주업체)은 평소 낮 시간대에 `WHERE id=?` 형태의 1~2건 조회 쿼리만 날렸다. 그런데 현재 **새벽 2시**에 **조건 없이(Full Scan)** **5천만 건**을 리턴하려 한다"는 복합적인 비정상 컨텍스트(Context)를 감지.
  3. AI 엔진이 Anomaly Score 99점을 부여하고, DB 방화벽과 연동되어 쿼리 반환을 중간에 끊어버림(Session Drop). 결과적으로 개발자의 PC에는 데이터가 1건도 전송되지 않음.

### 도입 체크리스트
- **학습 데이터의 오염 여부**: 처음 도입하여 AI가 기준선(Baseline)을 학습하는 1달 동안, 이미 해커가 침투해 있거나 스파게티 쿼리가 난무하고 있다면 AI는 "이 엉망진창인 상태가 우리 회사의 정상 상태구나"라고 오학습(Poisoning)하게 된다. 깨끗한 환경에서의 학습 보장이 필수적이다.
- **오버헤드 (Performance Degradation)**: 쿼리가 DB 엔진에 들어가기 전, 프록시(Proxy) 단에서 구문 분석과 AI 추론을 거치면 필연적으로 응답 지연(Latency)이 1~5ms 추가된다. 주식 거래소처럼 0.1ms를 다투는 시스템의 경우, 인라인(In-line) 방식 대신 네트워크 미러링(Out-of-band 스니핑) 방식으로 아키텍처를 변경해야 한다.
- **경고 피로 (Alert Fatigue)**: AI가 지나치게 예민하게 설정되어 하루에 "비정상 쿼리 탐지" 알람을 1,000개씩 쏟아내면 보안 담당자는 이를 늑대 소년으로 취급하여 무시하게 된다. 탐지 임계치(Threshold)와 튜닝 지원 도구의 성숙도를 점검해야 한다.

### 안티패턴: Audit(감사) 기능만 켜놓고 "보안 끝"이라 착각하기
- RDBMS 엔진 내부의 기본 `AUDIT` 기능을 켜서 사용자가 날린 모든 SQL 쿼리를 텍스트 로그 파일로 쌓아두기만 하는 행위. 스토리지만 가득 차서 디스크 폭발 장애를 유발할 뿐, 아무도 그 로그 수천만 줄을 읽지 않는다. 이는 범인이 물건을 훔쳐간 뒤 한 달 뒤에 경찰이 수사할 때나 필요한 사후 증거용일 뿐, 공격을 실시간으로 차단하는 '보안 시스템'이 결코 아님을 명심해야 한다.

- **📢 섹션 요약 비유**: 블랙박스(단순 Audit 로그)는 내가 뺑소니를 당한 뒤에 범인을 잡을 순 있지만 내 다리가 부러지는 걸 막아주진 못합니다. AI 보안 모니터링은 차가 나를 덮치기 직전에 나를 밀쳐서 구해주는 자동 긴급 제동장치(AEB)입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 레거시 권한 제어 (ACL) + WAF | AI 이상 탐지 + DB 방화벽 | 개선 효과 |
|:---|:---|:---|:---|
| **위협 차단 범위** | 외부망에서의 알려진 공격만 방어 | **정상 권한을 탈취한 내부자 유출 시도 차단** | **내부 보안 위협 탐지율 99% 이상 극대화** |
| **사고 대응 시간** | 유출 발생 후 수일 뒤 로그 보고 인지 | 의심 징후 발생 즉시 세션 차단 및 경고 | **침해 사고 초기 대응(MTTR) 1초 이내 단축** |
| **운영 공수** | 지속적인 화이트리스트 및 룰 수동 관리 | 자율 학습을 통한 동적 Baseline 적응 | **보안 정책 유지보수 리소스의 획기적 절감** |

### 미래 전망
- **AI 대 AI의 전쟁**: 해커들 역시 생성형 AI(LLM)를 이용해 시스템의 보안 룰을 우회하는 극도로 정교한(사람이 친 것 같은) 다형성 악성 쿼리를 초 단위로 생성하여 공격하고 있다. 이를 막기 위해 방어하는 DB 보안 솔루션 역시 소형 LLM 에이전트를 내재화하여 쿼리의 의도를 논리적으로 간파하는 지능전 양상으로 치닫고 있다.
- **마이크로 세그멘테이션과 제로 트러스트**: "한 번 인증된 WAS 서버는 내부망이니까 안전하다"는 믿음이 붕괴됨에 따라, 모든 DB 커넥션과 쿼리 하나하나(마이크로 단위)에 대해 0.001초마다 신뢰 점수(Trust Score)를 평가하고 허가하는 완벽한 제로 트러스트 인프라가 2030년경 클라우드 DB의 표준이 될 것이다.

- **📢 섹션 요약 비유**: 데이터베이스 이상 탐지는 가장 깊숙한 보물창고에서 잠들지 않고 끝없이 감시하는 드래곤의 눈입니다. 그 어떤 정교한 도둑의 발소리(쿼리)도 이 빅데이터 기반의 AI 드래곤을 속일 수는 없습니다.

2. **AI 이상 탐지 모니터링**은 성안 곳곳에 숨겨진 '투명 마법사'예요. 경비원 옷을 입고 있더라도, "어? 원래 저 경비원은 밤에 안 돌아다니는데? 보물을 100개나 담는다고?"라며 행동을 감시하죠.
