# 369. 접근 제어 모델(RBAC, ABAC) - 권한 관리의 정석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RBAC(역할 기반)은 사용자의 **'직책이나 역할'**에 따라 권한을 묶어 관리하는 방식이며, ABAC(속성 기반)은 사용자, 자원, 환경의 **'다양한 속성 조합'**을 통해 동적으로 권한을 결정하는 방식이다.
> 2. **가치**: RBAC은 대규모 조직의 권한 관리를 단순화하고 효율화하며, ABAC은 "근무 시간 내, 특정 위치에서만 접근 허용"과 같은 매우 정교하고 유연한 보안 정책 구현을 가능케 한다.
> 3. **융합**: 현대 클라우드 IAM 및 데이터베이스 보안 시스템은 기본 골격은 RBAC으로 잡고, 세부 통제는 ABAC을 더하는 하이브리드 방식으로 융합되어 발전하고 있다.

+++

### Ⅰ. RBAC (Role-Based Access Control)

- **정의**: 개별 사용자에게 권한을 주는 대신, '역할(Role)'을 정의하고 그 역할에 권한을 할당한 뒤 사용자를 역할에 매핑합니다.
- **특징**:
    - 관리가 매우 쉽습니다 (퇴사/부서 이동 시 역할만 변경).
    - 대부분의 RDBMS(Oracle, MySQL 등)가 기본으로 지원하는 모델입니다.

+++

### Ⅱ. ABAC (Attribute-Based Access Control)

- **정의**: 속성(Attributes)들의 조합과 논리적 정책(Policy)을 사용하여 실시간으로 접근 허용 여부를 판단합니다.
- **고려 속성**:
    - **Subject**: 사용자 직급, 프로젝트 참여 여부.
    - **Resource**: 데이터 보안 등급, 생성 일자.
    - **Environment**: 접속 IP, 현재 시간, 기기 보안 상태.

+++

### Ⅲ. 두 모델의 비교 시각화 (ASCII Model)

```text
[ 1. RBAC (Static) ]
  (User) ──▶ [ Role: Manager ] ──▶ [ Data: Sales Report ] ✅
  (권한이 역할에 고정됨)

[ 2. ABAC (Dynamic) ]
  (User Attributes) + (Environment) + (Resource Meta)
         │                │                │
         └───────┬────────┴────────┬───────┘
                 ▼
        [ Policy Engine ] ──▶ "Is it working hours? YES" ──▶ ✅ Access
                              "Is IP from Korea? YES"
```

+++

### Ⅳ. 실무적 선택 기준

| 비교 항목 | RBAC | ABAC |
|:---|:---|:---|
| **관리 단위** | 역할 (Manager, Developer) | **속성 (Time, Location, Dept)** |
| **유연성** | 낮음 (정적) | **매우 높음 (동적)** |
| **복잡도** | 낮음 | 높음 (정책 설계 필요) |
| **추천 환경** | 일반 기업 조직 체계 | **클라우드, 원격 근무, 고도 보안망** |

- **📢 섹션 요약 비유**: RBAC은 **'아파트 출입 카드'**와 같습니다. '주민' 카드가 있으면 모든 공용 시설을 다 쓸 수 있는 식입니다. 반면 ABAC은 **'지능형 안면 인식기'**와 같습니다. 카드(역할)가 있어도 "지금 밤 12시가 넘었나?", "술에 취했나?", "등록된 가족이 동행했나?" 등을 따져서 문을 열어줄지 말지 매번 똑똑하게 판단하는 것과 같습니다.

+++

### Ⅴ. 개념 맵 및 요약

- **[MAC/DAC]**: RBAC/ABAC 이전에 존재하던 고전적 접근 제어 모델.
- **[XACML]**: ABAC 정책을 기술하기 위한 표준 언어.
- **[Zero Trust]**: ABAC을 핵심 기술로 사용하는 현대 보안 패러다임.

📢 **마무리 요약**: **RBAC & ABAC**은 데이터 성채의 문지기입니다. 조직의 규모와 보안 요구 수준에 맞춰 '관리의 편의성'과 '통제의 정밀함' 사이에서 최적의 모델을 설계해야 합니다.
