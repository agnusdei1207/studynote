# 370. 데이터베이스 감사(Audit) 로그 - 사후 추적의 기록

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DBMS (Database Management System) 감사 로그는 데이터베이스 내 발생하는 모든 활동(접속, 질의 실행, 권한 변경 등)을 '누가, 언제, 어디서, 무엇을, 어떻게' 했는지 비가역적으로 기록하는 **보안 증적 자료(Proof of Record)**이다.
> 2. **가치**: 보안 사고 발생 시 포렌식(Forensics) 조사의 핵심 근거가 되며, 내부자의 악의적인 행위를 심리적으로 억제하는 **Deterrence(억제력)** 효과를 제공하고 ISMS-P, GDPR 등 법적 컴플라이언스(Compliance) 준수를 위한 필수 요건이다.
> 3. **융합**: 단순 로그 저장을 넘어 SIEM (Security Information and Event Management) 및 행동 기반 이상 탐지(UEBA) 시스템과 연동하여, 수동적인 기록에서 능동적인 **Threat Intelligence(위협 정보)** 탐지의 핵심 데이터 소스(Data Source)로 진화하고 있다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 정의
**데이터베이스 감사(Audit)**란 데이터베이스 시스템의 보안 정책 및 무결성을 유지하기 위해, 데이터베이스에 수행된 모든 작업을 추적하고 기록하는 절차를 말한다. 이는 단순한 에러 로그(Error Log)나 트랜잭션 로그(Transaction Log)와는 달리, **'누가(User)', '언제(Time)', '무엇을(Action)', '어떤 객체에 대해(Object)', '어떤 결과를(Result)' 얻었는지**라는 보안 핵심 5요소(5W1H)를 명확히 식별 가능한 형태로 저장하는 것을 목적으로 한다.

#### 2. 💡 비유
편의점(CVS)의 **CCTV(폐쇄회로 텔레비전)**와 같다. CCTV는 가게 안에서 일어나는 모든 행동을 영상으로 기록하여, 도둑이 들었을 때 용의자를 특정하고 범행 시점과 과정을 입증한다. 마찬가지로 감사 로그는 데이터베이스라는 가게 안에서 데이터가 유출되거나 훼손되었을 때, 그 범인(쿼리를 날린 사용자)과 범행 도구(SQL)를 증명하는 디지털 CCTV 역할을 한다.

#### 3. 등장 배경
① **기존 한계**: 과거에는 DBMS 자체의 로그가 방대하고 분석이 어려워, 침해 사고 발생 후 "언제 탈취당했는지"조차 파악하기 어려운 **'블랙박스 상태'**였다.
② **혁신적 패러다임**: 데이터 자산의 가치가 급증함에 따라, 데이터 접근을 **실시간으로 모니터링**하고 무결성을 보장하는 **'감사 가능성(Accountability)'** 개념이 도입되었다.
③ **현재의 비즈니스 요구**: 금융, 공공 등의 분야에서 법적 규제(개인정보보호법, 신용정보법 등)에 의해 **6개월 이상(최대 5년 이상)**의 장기 보관 및 정기적 감사 보고가 의무화되었다.

#### 📢 섹션 요약 비유
데이터베이스 감사 시스템 구축은 **"은행 금고에 들어가는 모든 사람의 출입을 번호표 없이 입체적으로 녹화하는 보안 시스템"**을 도입하는 것과 같습니다. 단순히 출입 명부만 작성하는 것(로그인 로그)을 넘어, 금고 안에서 어떤 돈을 어떻게 집어 들었는지(쿼리 결과)까지 초고속 카메라로 찍는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 구성 요소 (상세 분석)
감사 시스템은 다음과 같은 주요 모듈로 구성되어 상호작용한다.

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 (Internal Behavior) | 관련 프로토콜/기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Audit Provider** | 감사 정책의 관리자 | "무엇을 감시할 것인가"에 대한 규칙(Rule)을 정의하고 활성화/비활성화 제어 | SQL Audit Spec | 감사관 |
| **Capture Engine** | 이벤트의 포획 모듈 | DB Engine 내부나 네트워크 패킷 레벨에서 SQL을 가로채어 바이너리 형태로 변환 | TDS Tabular Data / Packet Sniffing | 수색견 |
| **Queue Buffer** | 성능 저하 방지 장치 | 실시간 트래픽이 과부하 시 일시적으로 로그를 쌓아두는 임시 저장소(Asynch I/O) | Ring Buffer / In-Memory Queue | 하수관 |
| **Log Writer** | 영구 저장 모듈 | 버퍼의 내용을 파일시스템 또는 원격 서버로 안전하게 Flush | Journaling / fsync() | 기록자 |
| **Analyzer/Reporter** | 해석 및 경보 모듈 | 저장된 로그를 파싱하여 위험 징후(SQL Injection, 대량 추출 등) 탐지 | Regex / ML Model | 분석관 |

#### 2. 아키텍처 도해 (ASCII Diagram)
감사 로그 수집 방식은 크게 DBMS 내부 기능을 사용하는 **Native 방식**과 네트워크 트래픽을 분석하는 **Agent(Proxy) 방식**으로 나뉜다. 아래는 두 방식이 통합된 하이브리드 아키텍처이다.

```text
+-----------------------------------------------------------------------+
|                         [ Data Access Layer ]                          |
+-----------------------------------------------------------------------+
       |                                             ^
       v (SQL Execution)                            | (Return Result)
+------------------+                        +------------------+
|  [ User Client ] |                        | [ DBA/Admin Tool]|
+--------+---------+                        +--------+---------+
         |                                             |
    (1)  | Normal SQL Traffic            (2)          | Direct DB Control
         | (JDBC/ODBC)                                 |
    +----v--------------------------------------------v-----------------+
    |            [ Audit Proxy / Network TAP ]                        |
    |    (Layer 7 Inspection - Captures Traffic before it hits DB)     |
    +----+-----------------------------+----------------+-------------+
         |                             |                |
         v                             v                v
+----------------+          +----------------+  +----------------+
| [ Main DB ]    |          | [ Standby DB ] |  | [ Audit DB ]   |
|  (Production)  |          |  (Mirroring)   |  | (Dedicated)    |
+---+------------+          +----------------+  +--------+-------+
    | ^                                          ^        |
    | | (3) Async Log Shipping (Batch/Stream)    |        |
    | |                                          |        |
    v |                                          |        |
+-------v----------+                     +-------v--------v------+
| [ Log Archiver ] |                     | [ SIEM System ]      |
|  (Syslog/S3)     | <-------------------| (Correlation Engine) |
+------------------+                     +----------------------+
  (Immutable Storage)                     (Real-time Alerting)
```

**[다이어그램 해설]**
1. **Proxy Layer(프록시 레이어)**: 사용자의 쿼리가 DB에 도달하기 전에 네트워크 패킷을 감지하여 원본 DB에 부하를 주지 않고 감사 로그를 남긴다.
2. **DB Internal Trigger(내부 트리거)**: DBMS 자체의 감사 기능(Audit Trail)을 활용하여 `GRANT`, `DROP` 같은 DDL이나 시스템 변경 사항을 캡처한다.
3. **Log Shipping(로그 운반)**: 감사 로그는 데이터베이스의 트랜잭션 로그와 분리되어 별도의 **Write-Once-Read-Many (WORM)** 스토리지로 전송된다. 이는 해커가 관리자 권한을 탈취하더라도 감사 기록을 지울 수 없도록 하기 위함이다.
4. **SIEM Correlation(SIEM 상관분석)**: 단순 로그 파일 그대로 저장되는 것이 아니라, SIEM(보안 사고 및 이벤트 관리) 시스템으로 전송되어 다른 보안 장비(방화벽, OS 로그)의 정보와 결합된다. 예를 들어, '같은 IP에서 OS 로그인 실패 후 DB 관리자 계정 로그인 성공' 패턴을 감지하여 즉시 알람을 보낸다.

#### 3. 심층 동작 원리 (Process Flow)
감사 로그의 생성은 데이터베이스 성능과 직결되므로, 주로 **비동기(Asynchronous)** 방식으로 동작한다.

1. **SQL Parse & Execute (1단계)**: 사용자가 `SELECT * FROM users;`를 실행하면 DB 엔진은 구문 분석(Parsing)을 수행한다.
2. **Rule Check (2단계)**: 감사 엔진은 해당 쿼리가 감사 대상 테이블(`users`)에 접근하는지, 감사 대상 유저(`app_user`)인지를 필터 규칙과 비교한다.
3. **Event Capture (3단계)**: 조건에 부합하면, 실행 계획(Execution Plan)이 생성되기 전후의 Context값(시스템 시간, 세션 ID, 클라이언트 Hostname)을 캡처한다.
4. **Buffering (4단계)**: 즉시 디스크에 쓰는 것이 아니라 메모리 내의 **Ring Buffer(순환 버퍼)**에 임시 저장한다. 만약 버퍼가 가득 차면 오래된 로그부터 덮어쓰거나 디스크로 Flush한다.
5. **Write & Archive (5단계)**: 백그라운드 스레드가 주기적으로 버퍼 내용을 안전한 외부 파일 시스템이나 Syslog 서버로 전송한다.

#### 4. 핵심 기술 및 코드 스니펙 (Example)
실무에서는 단순 텍스트 로그보다는 분석이 용이한 **JSON** 형식이나 **CSV** 형식을 주로 사용하며, 중요도에 따라 레벨을 나눈다.

**[코드: Oracle Fine-Grained Audit (FGA) 설정 예시]**
```sql
-- DBMS_FGA 패키지를 사용하여 특정 테이블의 접근을 감사
BEGIN
  DBMS_FGA.ADD_POLICY (
    object_schema   => 'HR',
    object_name     => 'EMPLOYEES',
    policy_name     => 'audit_salaries',
    audit_column    => 'SALARY', -- 급여 컬럼에 접근할 때만 기록
    audit_condition => 'SYS_CONTEXT(''USERENV'', ''SESSION_USER'') != ''HR_ADMIN''',
    enable          => TRUE,
    statement_types => 'SELECT, UPDATE' -- SELECT와 UPDATE만 감시
  );
END;
/
```

#### 📢 섹션 요약 비유
감사 로그의 아키텍처는 **"고속도로의 하이패스 시스템과 블랙박스"**가 결합된 형태입니다. 차량(쿼리)은 하이패스(비동기 버퍼)를 통해 아주 빠르게 통과하며 지체가 없고, 뒤쪽에서는 데이터 센터(외부 스토리지)로 실시간 통행 기록이 전송되어 나중에 누가 얼마나 속도 위반을 했는지(비정상 쿼리)를 감시하는 시스템과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 심층 기술 비교: 감사(Audit) vs 로깅(Log)

| 비교 항목 | 일반 DB Error Log / Transaction Log | Database Audit Log |
|:---|:---|:---|
| **목적** | 시스템 장애 복구, 트랜잭션 롤백 | 보안 준수, 법적 증거 확보, 행위 추적 |
| **기록 대상** | 시스템 에러, 데이터 변경 사항(DML/DDL) | **Who(사용자)**, IP, 프로그램명, 성공/실패 여부 |
| **성능 영향** | 필수 과정이므로 동기식(Sync) 동작 (필수) | 선택 사항이며 비동기식(Async) 동작 권장 (오버헤드 최소화) |
| **저장 위치** | DB 내부 파일 시스템 (휘발성 위험) | **별도 분리된 보안 서버** (WORM Storage) |
| **가용성** | DB 관리자만 접근 가능 | 감사 담당자(Auditor) 별도 접근 |

#### 2. 타 과목 및 기술 융합 분석
① **OS (Operating System) 융합**: DB 감사 로그는 `syslog` 프로토콜을 통해 리눅스 서버의 `/var/log/audit` 경로로 전송되어 통합 관리된다. 이를 통해 해커가 OS 레벨에서 파일을 삭제하려 하더라도 **원격 로그 서버(Remote Syslog)**에 이미 백업되어 있어 무결성이 유지된다.
② **네트워크(Network) 융합**: DB 포트(예: 3306, 1521, 1433)를 흐르는 패킷을 **Mirroring(포트 미러링)** 하여, DB에 로그를 남기지 않고도 트래픽 패턴을 분석하는 **DB 활동 모니터링(DAM)** 솔루션과 연동된다.
③ **AI(Artificial Intelligence) 융합**: 전통적인 감사는 '나중에 확인'하는 방식이었으나, UEBA(User and Entity Behavior Analytics) 기술을 적용하여 평소 패턴과 다른 **이상 징후(Anomaly)**를 실시간으로 감지하고 쿼리 실행을 자동 차단하는 **동적 감사**로 발전 중이다.

#### 📢 섹션 요약 비유
감사 로그와 다른 시스템의 융합은 **"병원의 MRI, 혈액 검사, 심전도를 모아 종합 진단하는 의료 시스템"**과 같습니다. 데이터베이스라는 장기(臟器)에서 나온 신호(로그)만 보면 알 수 없는 병(보안 위협)도, 네트워크 흐름(혈관)과 OS 행동(신경계)을 종합하여 AI 의사(SIEM)가 분석하면 초기 암(내부자 위협)을 발견할 수 있습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 시나리오 및 의사결정

**상황 1: 성능 저하(Performance Degradation) 문제 발생**
- **문제**: 전체 감사(Full Audit)를 켜두었더니, 초당 1,000건 이상의 쿼리가 발생하는 OLTP 환경에서 디스크 I/O 병목이 발생하여 응답 속도가 5배 이상 늦어짐.
-
