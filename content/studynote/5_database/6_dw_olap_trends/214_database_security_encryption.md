# 데이터베이스 보안과 암호화 기술 - TDE, 마스킹, FPE, SQL 인젝션 방어

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 보안은 외부의 악의적인 쿼리 주입(SQL Injection)을 방어하는 **접근 통제(Access Control)**와, 디스크 탈취나 내부자 유출 시에도 데이터를 읽을 수 없게 만드는 **암호화(Encryption) 및 비식별화(Masking)** 기술의 총체적 결합이다.
> 2. **가치**: TDE(Transparent Data Encryption)는 애플리케이션의 쿼리 수정 없이 디스크 수준의 데이터 휴지 상태(Data at Rest) 암호화를 투명하게 제공하며, FPE(Format Preserving Encryption)는 데이터의 원래 형태(길이, 타입)를 유지하면서 암호화하여 기존 시스템과의 호환성을 극대화한다.
> 3. **융합**: 최신 DB 보안은 단순히 암호화를 넘어, 암호화된 상태 그대로 연산을 수행하는 동형 암호(Homomorphic Encryption) 및 프라이버시 보존형 데이터 공유 플랫폼(Clean Room) 기술로 융합 발전하고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 데이터베이스 보안(Database Security)은 기밀성(Confidentiality), 무결성(Integrity), 가용성(Availability)의 CIA 삼원칙을 DB 계층에서 달성하기 위한 물리적, 논리적, 관리적 통제 기법을 의미한다.

- **💡 비유**: 데이터베이스 보안은 거대한 은행의 금고 시스템과 같습니다. 외부 도둑이 속임수로 문을 열지 못하게 막는 경비원(SQL 인젝션 방어)이 있고, 만약 금고가 털리더라도 훔쳐간 지폐를 식별 불가능한 종이 조각으로 보이게 만드는 특수 잉크(TDE, 암호화)가 결합되어 있는 것입니다.

- **등장 배경**:
  1. **해킹 위협의 고도화**: 웹 애플리케이션 취약점을 노린 SQL 인젝션 공격이 기업 고객 정보 유출 사고의 1위를 차지하며, DB 단의 근본적 방어 체계가 시급해졌다.
  2. **강력한 규제 컴플라이언스**: GDPR, HIPAA, 한국의 개인정보보호법 등 법적 규제로 인해 주민번호, 카드번호 등의 민감 정보(PII)는 반드시 암호화된 상태로 저장해야 할 법적 의무가 생겼다.
  3. **내부자 위협 증가**: 외부 해킹뿐만 아니라, DB 서버에 직접 접근 권한이 있는 관리자(DBA)나 개발자가 디스크 파일 덤프를 떠서 유출하는 '내부자 위협'을 방어할 아키텍처가 요구되었다.

- **📢 섹션 요약 비유**: 성벽(방화벽)을 아무리 높게 쌓아도 성문을 여는 암구호(SQL)가 유출되거나 내통자(권한자)가 있으면 뚫릴 수 있으므로, 보물 자체를 풀 수 없는 자물쇠(암호화)로 한 번 더 잠가두는 다중 방어막입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. TDE (Transparent Data Encryption, 투명한 데이터 암호화)

애플리케이션(사용자) 관점에서는 데이터가 평문으로 보이지만, 디스크에 물리적으로 저장될 때(Data at Rest)는 암호화되어 저장되는 RDBMS의 엔진 레벨 기술이다. (Oracle, SQL Server 등 지원)

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 TDE (Transparent Data Encryption) 구조                │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [Application]  SELECT 주민번호 FROM User; (평문 반환)              │
  │        │                                                         │
  │  ──────▼───────────────────────────────────────────────────────  │
  │  [Database Engine (SGA/Buffer Cache)]                            │
  │   - 메모리 상에는 평문(Plaintext)으로 존재. 인덱스 검색 등 정상 수행.      │
  │  ──────┬───────────────────────────────────────────────────────  │
  │        │ (I/O Flush 시점)          (I/O Read 시점)                │
  │        ▼ 암호화(Encrypt)           ▲ 복호화(Decrypt)              │
  │  ┌───────────┐               ┌───────────┐                       │
  │  │ Master Key│ ──(해독)──▶ │ Table Key │  (KMS / Wallet 연동)    │
  │  └───────────┘               └─────┬─────┘                       │
  │                                    ▼                             │
  │  [Storage / Disk] (Data Files, Redo Logs, Backups)               │
  │   - 디스크에는 완벽히 암호화된 형태(Ciphertext)로 기록.                 │
  │   - 해커가 하드디스크를 통째로 훔쳐가도 복호화 키가 없어 읽을 수 없음.     │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TDE의 핵심은 '투명성(Transparency)'이다. 개발자는 SQL 코드를 단 한 줄도 수정할 필요가 없다. DB 엔진이 디스크에 데이터를 기록할 때 암호화(AES-256 등)를 수행하고, 디스크에서 메모리로 데이터를 읽어올 때 자동으로 복호화한다. 보안의 핵심은 데이터를 암호화한 '테이블 키(Table Key)'를 다시 암호화하는 **마스터 키(Master Key)**의 보관인데, 이를 DB 외부에 있는 보안 키 관리 시스템(KMS, Key Management System)이나 하드웨어 보안 모듈(HSM)에 보관하여 내부자(DBA)라 할지라도 디스크 파일 덤프로 데이터를 유출할 수 없게 막는다.

---

### 2. 형태 보존 암호화 (FPE, Format Preserving Encryption)

기존 암호화 방식(예: AES)은 '1234'라는 4자리 숫자를 넣으면 'A7x9Q...'와 같은 길고 복잡한 문자열로 반환하여 DB의 스키마(`VARCHAR(4)`)를 깨뜨리는 문제가 있었다. 이를 해결하는 기술이 FPE다.

- **원리**: 암호화 전의 데이터 길이와 데이터 타입(숫자는 숫자로, 문자는 문자로)을 완벽하게 동일하게 유지한다.
- **효과**: 주민번호 '123456-1234567'을 FPE로 암호화하면 '987654-9876543'과 같은 가짜 숫자 형태가 된다. 기존 DB의 컬럼 크기나 애플리케이션의 유효성 검사 로직(Validation)을 전혀 수정하지 않고도 보안을 적용할 수 있다.

---

### 3. 데이터 마스킹 (Data Masking)과 동적 마스킹 (DDM)

개인정보 일부를 별표(*)나 특정 문자로 가려서 비식별화하는 기법이다. (예: 홍길동 -> 홍*동)

- **정적 데이터 마스킹 (SDM)**: 운영 DB의 데이터를 개발/테스트 DB로 복제(Clone)할 때, 개인정보를 영구적으로 마스킹하여 복사한다. 개발자에게 실데이터가 유출되는 것을 원천 차단한다.
- **동적 데이터 마스킹 (DDM)**: 원본 데이터는 DB에 평문으로 저장되어 있으나, 사용자의 권한(Role)에 따라 조회(`SELECT`) 시점에 실시간으로 마스킹을 씌워서 보여주는 기술. (콜센터 직원은 전화번호 뒷자리 마스킹, 매니저는 전체 조회 가능 등)

## Ⅲ. 융합 비교 및 다각도 분석

### DB 암호화 기술 심층 비교 (API vs Plug-in vs TDE)

DB 암호화를 어느 계층에서 수행할 것인가에 따른 아키텍처 분류.

| 구분 | API 방식 (애플리케이션 계층) | Plug-in 방식 (DB 엔진 계층) | TDE 방식 (DB 커널/디스크 계층) |
|:---|:---|:---|:---|
| **암복호화 위치** | 애플리케이션 (WAS 서버) | DB 서버 내의 플러그인 프로세스 | DB 엔진 내부 (커널) |
| **장점** | DB 서버 부하 없음, 네트워크 전송 구간도 암호화 보장 | 애플리케이션 소스 코드 수정 불필요 | 완벽한 투명성, 인덱스 검색 완벽 지원 |
| **단점** | 앱 로직 수정 막대함 (최대 단점) | DB 서버에 암호화/복호화 CPU 부하 집중 | 메모리 상에서는 평문 노출, DB 벤더 종속적 |
| **권장 환경** | 금융권 중요 결제, 신규 시스템 | 기존 시스템의 빠른 암호화 적용 | 대용량 레거시 RDBMS 및 클라우드 DB |

### 과목 융합 관점

- **웹 보안 (Web Security)**: **SQL 인젝션(SQL Injection)**은 해커가 웹 폼 입력창에 `' OR 1=1 --` 와 같은 악성 쿼리를 주입하여 인증을 우회하거나 데이터를 탈취하는 공격이다. 이를 막기 위한 가장 완벽한 DB 융합 대응책은 애플리케이션 단에서 **바인드 파라미터 (Prepared Statement)**를 사용하여, 입력값을 '실행 가능한 코드'가 아닌 단순 '문자열 데이터'로만 취급하도록 쿼리 실행기(Parser)의 동작을 격리하는 것이다.
- **클라우드 아키텍처**: AWS RDS, Aurora 등은 원클릭으로 TDE를 지원하며, 암호화 마스터 키 관리를 위해 AWS KMS(Key Management Service)와 네이티브로 융합되어 키 롤테이션(Key Rotation) 자동화 등의 거버넌스를 제공한다.

- **📢 섹션 요약 비유**: API 방식은 물건을 보낼 때 집에서부터 비밀 금고에 넣어 택배를 보내는 것이고, TDE는 목적지 창고에 도착하면 알아서 투명 금고에 담아주는 시스템입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 의료 데이터 분석을 위한 클린 룸(Clean Room) 구축

- **문제 상황**: 제약 회사가 병원의 실제 환자 처방 데이터(RWD)를 AI 분석에 사용하고 싶으나, 의료법상 민감 정보 유출이 금지되어 원본 데이터 접근이 차단됨.
- **의사결정**:
  1. **가명 처리 (Pseudonymization)**: 병원 DB에서 주민번호, 이름은 해시(Hash) 처리하고 나이는 범위(20대, 30대)로 범주화하는 정적 마스킹(SDM) 적용.
  2. **동형 암호 (Homomorphic Encryption) 연구 도입**: 최신 보안 트렌드에 따라, 데이터를 복호화하지 않고 암호화된 텍스트(Ciphertext) 상태 그대로 연산(SUM, AVG)을 수행할 수 있는 동형 암호 DB 플러그인 도입. 제약 회사는 암호화된 상태로 AI를 학습시키고 결과만 도출함.
  3. **데이터 클린 룸 적용**: 물리적으로 격리된 샌드박스 환경(Snowflake Data Clean Room)에서 양측의 데이터가 섞이지 않고 쿼리 결과물만 교환하도록 구성.

### 도입 체크리스트
- **성능 오버헤드 측정**: 암호화를 적용하면 컬럼 단위 암호화(Plug-in)의 경우 평균 10~20%, 디스크 암호화(TDE)의 경우 2~5%의 CPU 성능 저하가 발생한다. 트랜잭션 피크 타임에 이 부하를 감당할 수 있는 여유 리소스가 있는가?
- **키 관리 (KMS)의 분리**: DB 서버 내부나 로컬 디스크에 암호화 키를 같이 저장해 두는 것은 '문 열쇠를 자물쇠 옆에 걸어두는 것'과 같다. 반드시 물리적으로 분리된 별도의 KMS 서버나 HSM 장비를 구축했는가?
- **인덱스 파괴 문제**: 암호화된 컬럼에 대해 기존의 B-Tree 인덱스(Range Scan)는 무용지물이 될 수 있다. 일치 검색(Exact Match)만 지원되므로, 검색 조건으로 자주 쓰이는 컬럼을 무리하게 암호화하면 풀 스캔(Full Scan)으로 인한 시스템 장애가 유발된다.

### 안티패턴: 모든 컬럼에 대한 과도한 암호화 적용
- "보안이 짱이다"라며 이름, 주민번호, 전화번호뿐만 아니라 주소, 가입일, 성별 등 모든 컬럼에 플러그인 방식의 암호화를 적용하는 행위. 암/복호화 CPU 부하가 폭증하고 모든 인덱스가 깨져 DB가 즉시 다운된다. 암호화 대상은 법적 규제 대상(PII)으로 최소화하고, 나머지는 접근 제어(DB 방화벽)로 방어하는 것이 원칙이다.

- **📢 섹션 요약 비유**: 외출할 때 귀중품(주민번호)만 금고에 넣어야지, 신발, 우산, 휴지통(일반 데이터)까지 전부 비밀 금고에 넣고 자물쇠를 채우면 외출(서비스) 자체가 불가능해지는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 보안 미적용 (평문) | DB 암호화/마스킹 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **데이터 유출 피해** | 디스크/백업 탈취 시 100% 유출 | 탈취 시 해독 불가능한 쓰레기값 | **기밀 유출 사고에 따른 천문학적 과징금 제로화** |
| **운영계/개발계 분리** | 운영 데이터 그대로 개발 테스트 | 동적/정적 마스킹을 통한 비식별화 | **내부 개발자/협력업체에 의한 정보 유출 원천 차단** |
| **성능 오버헤드** | 응답 지연 없음 | TDE 적용 시 2~5% 지연 | **최소한의 성능 희생으로 최고 수준 법적 요건 충족** |

### 미래 전망
- **양자 내성 암호 (PQC) 전환**: 다가오는 양자 컴퓨터 시대에 기존의 RSA 및 타원 곡선 암호(ECC) 체계가 붕괴될 것에 대비하여, 양자 알고리즘으로도 풀 수 없는 포스트 퀀텀 암호화 모듈이 DB 커널 레벨에 탑재되기 시작할 것이다.
- **완전 동형 암호 (FHE)의 상용화**: 현재는 속도가 너무 느려 제한적으로 쓰이는 동형 암호 체계가, 하드웨어 가속기(NPU)의 발달로 인해 실시간 암호화 쿼리 처리가 가능한 수준으로 상용화되어 클라우드 보안의 근본 패러다임을 바꿀 것이다.

- **📢 섹션 요약 비유**: 다가올 미래의 데이터베이스 보안은 두꺼운 장갑차(경계 보안)를 만드는 것을 넘어, 데이터 자체가 투명인간(동형 암호)이 되어 목적지에서 할 일만 몰래 하고 사라지는 완벽한 마법의 단계로 접어들 것입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 데이터베이스 보안은 나쁜 도둑(해커)으로부터 우리 집의 보물(개인정보)을 지키는 3중 잠금장치예요.
2. 첫째로 도둑이 이상한 암호(SQL 인젝션)를 대고 문을 열지 못하게 보디가드가 지키고 있고, 둘째로 보물을 밖으로 꺼내면 글자가 별표(*)로 변하는 요술 안경(마스킹)을 씌워요.
3. 셋째로 만약 도둑이 금고 통째로(디스크) 훔쳐가도, 마스터 키(열쇠)가 없으면 안의 내용물이 전부 외계어(TDE 암호화)로 보여서 절대 읽을 수 없게 만든답니다!
