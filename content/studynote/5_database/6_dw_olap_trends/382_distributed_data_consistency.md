# 382. 분산 데이터 정합성 보장 - 합의(Consensus)의 기술

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 데이터 정합성 보장은 여러 노드에 복제된 데이터가 **어떠한 장애 상황(네트워크 단절, 노드 다운) 속에서도 동일한 상태를 유지하도록 강제하는 분산 제어 기술**이다.
> 2. **가치**: 팍소스(Paxos)나 래프트(Raft)와 같은 **합의(Consensus) 알고리즘**을 통해 과반수(Quorum) 이상의 노드가 동의한 데이터만 확정함으로써, 전역적인 데이터 무결성을 실현한다.
> 3. **융합**: 분산 트랜잭션의 2PC(2단계 커밋) 한계를 보완하여 NewSQL과 분산 코디네이터(ZooKeeper, etcd)의 핵심 일관성 엔진으로 융합되어 사용된다.

+++

### Ⅰ. 분산 정합성 달성의 핵심 매커니즘

1. **과반수 합의 (Quorum)**: $N$개의 노드 중 $(N/2 + 1)$개 이상의 노드가 성공적으로 데이터를 기록하면 '성공'으로 간주합니다. (일부 노드 장애 수용)
2. **리더 기반 복제 (Leader-based)**: 하나의 리더가 모든 쓰기 권한을 독점하고, 리더의 로그 순서에 따라 모든 팔로워가 데이터를 똑같이 복제합니다.
3. **단조 증가 타임스탬프**: 연산의 선후 관계를 명확히 하기 위해 논리적 시계(Logical Clock)를 사용하여 데이터의 버전을 관리합니다.

+++

### Ⅱ. Raft 합의 알고리즘 시각화 (ASCII Flow)

```text
[ Raft Consensus Flow ]

  (Client) ──▶ [ Leader Node ] ───▶ (Log: Set A=10)
                    │
          ┌─────────┴─────────┐
          ▼                   ▼
  [ Follower B ]      [ Follower C ]
  (Append Log)        (Append Log)
          │                   │
          └─────────┬─────────┘
                    ▼
  [ Leader: 과반수(2/3) 확인 ] ──▶ [ COMMIT! ] ✅
                    │
          ┌─────────┴─────────┐
          ▼                   ▼
  (Apply to State)    (Apply to State)
```

### Ⅲ. 정합성 보장 방안 비교

- **강한 일관성 (Strong Consistency)**: 쿼리 시 무조건 최신 데이터를 보장. 성능보다는 정확성이 생명인 금융권에 적합 (Paxos/Raft 필수).
- **결과적 일관성 (Eventual Consistency)**: 당장은 달라도 결국 같아짐. 가용성이 중요한 소셜 서비스 등에 적합.
- **인과적 일관성 (Causal Consistency)**: 연관 있는 작업들의 순서만은 철저히 지키는 중간 단계의 정합성.

- **📢 섹션 요약 비유**: 분산 데이터 정합성은 **'합창단의 화음'**과 같습니다. 수십 명의 단원(노드)이 노래할 때, 지휘자(리더)의 지휘봉에 맞춰 동시에 입을 떼야 아름다운 화음(정합성)이 나옵니다. 한두 명이 박자를 놓쳐도(장애) 나머지 단원들이 과반수 이상 정확한 박자를 지키면 공연은 계속될 수 있는 민주적이고도 엄격한 규칙입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[CAP Theorem]**: 일관성(C)을 얻기 위해 가용성(A)을 얼마나 희생할 것인가의 문제.
- **[Split Brain]**: 정합성 보장 체계가 무너졌을 때 발생하는 최악의 재앙.
- **[Linearizability]**: 데이터가 마치 한 곳에만 있는 것처럼 순차적으로 변하는 최상위 정합성 수준.

📢 **마무리 요약**: **Distributed Consistency**는 분산 시스템의 성배입니다. 복잡한 네트워크 미로 속에서도 데이터의 '진실'을 하나로 모으는 합의의 기술이 분산 DB의 가치를 결정합니다.
