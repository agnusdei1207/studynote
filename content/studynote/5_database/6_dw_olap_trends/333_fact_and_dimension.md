# 333. 팩트(Fact)와 차원(Dimension) - 분석 데이터의 원자

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 팩트(Fact)는 비즈니스 프로세스에서 발생하는 수치적 측정값(숫자)이며, 차원(Dimension)은 그 수치를 설명하고 분류하는 맥락(텍스트/속성) 정보이다.
> 2. **가치**: "누가, 언제, 어디서, 무엇을(차원)" 했으며 "그 결과 수치가 얼마(팩트)"인지를 명확히 분리함으로써, 방대한 데이터를 다각도로 자유롭게 집계하고 분석할 수 있게 한다.
> 3. **융합**: 외래키(FK) 관계를 통해 팩트와 차원이 결합되어 스타 스키마를 형성하며, 이는 현대 모든 데이터 분석 시스템의 논리적 최소 단위가 된다.

+++

### Ⅰ. 팩트 테이블 (Fact Table)

- **정의**: 비즈니스의 개별 사건(이벤트)을 수치로 기록한 테이블입니다.
- **특징**:
    - **Measures**: 합계, 평균 등이 가능한 숫자 데이터 중심.
    - **Foreign Keys**: 모든 연관 차원 테이블의 ID를 가지고 있음.
    - **방대한 양**: 데이터베이스에서 가장 큰 부피를 차지함 (수억~수조 건).
- **예시**: 판매 금액, 재고 수량, 클릭 횟수, 온도 측정값.

+++

### Ⅱ. 차원 테이블 (Dimension Table)

- **정의**: 팩트 테이블의 데이터를 걸러내고(Filtering), 그룹화(Grouping)하는 기준이 되는 테이블입니다.
- **특징**:
    - **Attributes**: 이름, 주소, 카테고리 등 텍스트 데이터 중심.
    - **Primary Key**: 각 차원의 고유 ID 보유.
    - **상대적으로 작은 양**: 팩트에 비해 레코드 수가 적고 변화가 느림.
- **예시**: 고객 정보, 상품 마스터, 달력(시간), 매장 위치.

+++

### Ⅲ. 팩트와 차원의 조화 (ASCII Diagram)

```text
[ 분석의 관점: 차원(D) vs 측정: 팩트(F) ]

      (언제?) [ Dim_Time ] ─────┐
                               │
      (어디서?) [ Dim_Store ] ──┼──▶ [ Fact_Sales ] (F) ──▶ (얼마나?) 1,000,000원
                               │
      (무엇을?) [ Dim_Item ] ───┘
```

- **📢 섹션 요약 비유**: 팩트와 차원은 **'문장의 성분'**과 같습니다. "철수가(차원) 오늘(차원) 사과를(차원) **3개(팩트)** 샀다"에서, 팩트는 우리가 궁금한 '결과값'이고 차원은 그 결과를 더 자세히 설명해 주는 '주어, 목적어, 부사'와 같은 역할을 합니다. 주어(차원)를 바꿔가며 문장을 다시 써보는 과정이 바로 데이터 분석입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Granularity]**: 팩트 데이터의 최소 기록 단위 (그레인).
- **[Aggregatable]**: 팩트의 가장 큰 속성인 '연산 가능성'.
- **[Context]**: 차원이 팩트에게 부여하는 분석적 의미.

📢 **마무리 요약**: **Fact & Dimension**은 분석의 알파이자 오메가입니다. 숫자의 의미를 맥락으로 풀어내는 이 구조가 데이터 분석의 모든 가능성을 열어줍니다.
