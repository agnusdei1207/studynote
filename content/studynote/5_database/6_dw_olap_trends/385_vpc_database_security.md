# 385. VPC 내 DB 보안 구성 - 격리된 클라우드 요새

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: VPC(가상 사설 클라우드) 내 DB 보안 구성은 데이터베이스를 인터넷과 직접 연결되지 않는 **프라이빗 서브넷(Private Subnet)에 배치하여, 논리적/물리적 네트워크 수준에서 접근을 완벽히 격리하는 전략**이다.
> 2. **가치**: 보안 그룹(Security Group), ACL(Network Access Control List), 배스천 호스트(Bastion Host) 등 다중 계층 방어선을 구축하여 허가되지 않은 모든 침입 시도를 원천 차단한다.
> 3. **융합**: 엔드포인트(VPC Endpoint) 기술과 융합되어, 공인 인터넷망을 거치지 않고 내부망 전용 통신(Private Link)을 수행함으로써 클라우드 데이터 전송의 안전성과 성능을 동시에 확보한다.

+++

### Ⅰ. VPC 내 DB 보안 구성의 필수 요소

1. **Private Subnet 배정**: DB 서버는 공인 IP 없이 내부 IP만 할당받아 인터넷으로부터 직접 노출되지 않게 합니다.
2. **보안 그룹 (Security Group)**: 인스턴스 단위의 '상태 저장(Stateful)' 방화벽으로, 허용된 앱 서버의 IP와 포트(예: 3306, 5432)만 통신을 허가합니다.
3. **ACL (Access Control List)**: 서브넷 단위의 '상태 비저장(Stateless)' 방화벽으로 추가적인 접근 통제를 수행합니다.
4. **Bastion Host / VPN**: 관리자가 점검을 위해 접속할 때만 경유하는 안전한 통로를 지정합니다.

```text

  │  │ [ App Server ]   │ ──▶  │ [ Security Grp ] │  │ ✅

  └──────────────────────────────────────────────────┘
```

### Ⅲ. 클라우드 전용 보안 심화 기술

- **VPC Endpoint (PrivateLink)**: S3나 다른 관리형 서비스에 접근할 때 인터넷 게이트웨이를 거치지 않고 VPC 내부 경로로만 데이터를 주고받습니다.
- **Enhanced Monitoring**: VPC Flow Logs를 통해 DB로 들어오고 나가는 모든 트래픽의 흐름을 기록하고 이상 징후를 분석합니다.

- **📢 섹션 요약 비유**: VPC 내 DB 보안은 **'비밀 금고가 있는 성채'**와 같습니다. 성문(Public Subnet)에는 손님들이 드나들 수 있지만, 진짜 보물(DB)은 성안에서도 가장 깊숙한 지하실(Private Subnet)에 숨겨져 있습니다. 그 지하실은 오직 허락된 집사(App Server)나 특수 열쇠(VPN)를 가진 주인만 들어갈 수 있는 아주 폐쇄적이고 안전한 공간인 셈입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Inbound/Outbound Rules]**: 보안 그룹에서 정의하는 통신 방향성 규칙.
- **[NAT Gateway]**: 프라이빗 DB가 패치 업데이트 등을 위해 외부로만 나갈 수 있게 돕는 장치.
- **[Micro-segmentation]**: 제로 트러스트 사상을 반영하여 서브넷을 더 세밀하게 쪼개는 보안 기법.

📢 **마무리 요약**: **VPC DB Security**는 클라우드 설계의 기본 상식이자 핵심입니다. 물리적 망 분리보다 강력한 논리적 격리를 통해 전 세계 어디서든 안전한 데이터 서비스를 구축할 수 있습니다.
