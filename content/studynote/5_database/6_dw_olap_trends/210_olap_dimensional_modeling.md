# OLAP 연산과 다차원 모델링 - 스타 스키마와 스노우플레이크 스키마

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OLAP(On-Line Analytical Processing)는 사용자가 대용량 데이터를 다차원적(Multi-dimensional)으로 조작하며 대화식으로 분석할 수 있게 하는 기술이며, 이를 뒷받침하기 위해 **다차원 모델링 (팩트와 차원)**이 필수적으로 동반된다.
> 2. **가치**: 롤업(Roll-up), 드릴다운(Drill-down) 등의 직관적인 연산을 통해 기업의 성과 지표를 실시간으로 탐색하며, **스타 스키마(Star Schema)**의 단순성을 통해 복잡한 RDBMS 환경에서도 쿼리 성능을 극한으로 끌어올린다.
> 3. **융합**: 전통적인 MOLAP(다차원 큐브)과 ROLAP(관계형 테이블)의 장점을 결합한 HOLAP 엔진으로 진화하였으며, 현대 BI 툴(Tableau, PowerBI)의 시각화 밑단에서 핵심 연산 엔진으로 융합되어 사용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **OLAP**: 데이터 웨어하우스나 데이터 마트에 저장된 대규모 데이터를 사용자가 직접 다차원 큐브(Cube) 형태로 조작하며 분석하는 프로세스.
  - **다차원 모델링 (Dimensional Modeling)**: 랄프 킴볼(Ralph Kimball)이 제안한 기법으로, 복잡한 비즈니스 프로세스를 측정값인 '팩트(Fact)'와 분석 기준인 '차원(Dimension)'으로 나누어 모델링하는 방식.

- **💡 비유**: 루빅스 큐브 장난감처럼 요리조리 돌려가며 데이터를 보는 것과 같습니다. 한쪽 면을 보면 "연도별 매출"이 보이고, 큐브를 돌리면 "지역별 매출", 더 작게 쪼개보면 "지점별 일일 매출"을 자유롭게 볼 수 있는 매직 큐브입니다.

- **등장 배경**:
  1. **정규화 모델의 분석 한계**: RDBMS의 제3정규형(3NF) 모델은 수백 개의 테이블로 잘게 쪼개져 있어, 경영진이 "작년 3분기 서울 지역의 20대 여성 운동화 매출"을 보려면 수십 개의 조인(Join)이 발생하여 쿼리가 멈추는 사태가 발생했다.
  2. **직관적인 비즈니스 뷰 요구**: 경영진과 현업 분석가들은 IT 부서의 복잡한 SQL 작성 지원 없이도, 엑셀의 피벗 테이블(Pivot Table)처럼 직관적으로 데이터를 드래그 앤 드롭하여 분석하길 원했다.

- **📢 섹션 요약 비유**: 서랍을 일일이 열어 장부를 맞춰보던 복잡한 방식(정규화)에서, 모든 각도에서 속이 훤히 들여다보이는 투명한 다차원 유리 상자(OLAP)로 데이터를 옮겨 담은 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. OLAP의 5대 핵심 연산 (Operations)

사용자가 다차원 큐브를 조작하여 데이터를 탐색하는 방법이다.

| 연산명 | 동작 원리 | 예시 |
|:---|:---|:---|
| **Roll-up (롤업)** | 데이터의 요약(Aggregation), 계층을 타고 위로 올라감 | "월별 매출" -> "연도별 매출" 로 합산 |
| **Drill-down (드릴다운)** | 데이터의 구체화, 계층을 타고 아래로 내려감 (Roll-up의 반대) | "국가별 매출" -> "도시별 매출" 로 세분화 |
| **Slice (슬라이스)** | 다차원 큐브에서 특정 차원(Dimension) 하나를 잘라냄 | 전체 매출 중 "2026년(시간)"이라는 한 면만 잘라서 조회 |
| **Dice (다이스)** | 두 개 이상의 차원을 잘라내어 작은 큐브(Sub-cube)를 만듦 | "2026년(시간)" AND "서울(지역)" AND "가전(상품)" 추출 |
| **Pivot (피벗)** | 보고서의 축을 회전시켜 데이터를 보는 관점을 바꿈 | X축(지역)-Y축(상품) 보고서를 X축(상품)-Y축(지역)으로 전환 |

---

### 2. 다차원 모델링: 팩트(Fact)와 차원(Dimension)

- **팩트 테이블 (Fact Table)**: 비즈니스 프로세스에서 발생하는 수치적 사건(측정값)을 저장한다. (예: 매출액, 판매 수량, 클릭 수). 가장 크기가 크고 빠르게 증가하며, 모든 차원 테이블의 외래 키(FK) 조합으로 기본 키를 구성한다.
- **차원 테이블 (Dimension Table)**: 팩트 테이블의 데이터를 "어떻게" 분석할 것인지에 대한 기준 데이터를 저장한다. (예: 시간, 매장, 상품, 고객). 팩트에 비해 크기가 작고 변동성이 적으며, 상세한 텍스트 속성을 포함한다.

---

### 3. 스타 스키마 (Star Schema) vs 스노우플레이크 스키마 (Snowflake Schema)

데이터 마트를 설계하는 가장 대표적인 두 가지 물리적 구조다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 스타 스키마 vs 스노우플레이크 스키마                  │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 스타 스키마 (Star Schema) ]        [ 스노우플레이크 (Snowflake) ]    │
  │                                                                  │
  │   [시간 차원]     [매장 차원]          [시간 차원]      [도시 차원]     │
  │        ╲         ╱                         │             │          │
  │         ▼       ▼                          ▼             ▼          │
  │      [ 팩트: 매출 테이블 ]               [ 팩트: 매출 ] ◀─ [매장 차원]    │
  │         ▲       ▲                          ▲             ▲          │
  │        ╱         ╲                         │             │          │
  │   [고객 차원]     [상품 차원]          [고객 차원] ◀─[상품 차원]      │
  │                                                        │          │
  │  - 차원 정규화: 미적용 (중복 허용)       - 차원 정규화: 적용 (중복 제거)   │
  │  - 조인 횟수: 단 1단계 (초고속)         - 조인 횟수: 여러 단계 (다소 느림)  │
  │  - 유지보수: 데이터 중복으로 다소 불리   - 유지보수: 중복 제거로 관리 용이   │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 
**스타 스키마**는 중앙의 거대한 팩트 테이블 주변에 정규화되지 않은(중복을 허용한) 차원 테이블들이 별 모양으로 직접 연결된 구조다. 무조건 한 번의 조인(1-Depth)만으로 데이터를 가져올 수 있어 읽기(Read) 성능이 극한으로 높다.
반면, **스노우플레이크 스키마**는 차원 테이블 내의 데이터 중복을 막기 위해 차원 테이블을 한 번 더 쪼갠(정규화한) 눈송이 형태다. 예를 들어 매장 차원 테이블에서 '국가', '도시' 정보를 따로 빼서 하위 차원 테이블로 만든다. 저장 공간은 절약되지만 쿼리 시 조인(Join)이 2~3단계로 늘어나 OLAP 성능이 저하될 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### ROLAP, MOLAP, HOLAP 심층 비교

데이터를 물리적으로 어떻게 저장하고 연산하느냐에 따른 아키텍처 분류다.

| 분류 | 정의 및 원리 | 장점 | 단점 | 대표 제품 |
|:---|:---|:---|:---|:---|
| **ROLAP** (Relational) | RDBMS 내에 스타 스키마로 저장하고, 사용자가 요청 시 SQL로 실시간 집계 | 대용량 데이터 처리 가능, 확장성 우수 | 무거운 쿼리 시 응답 속도 저하 | Amazon Redshift, Snowflake |
| **MOLAP** (Multi-dimensional) | 데이터를 다차원 배열(Cube) 형태로 미리 계산(Pre-aggregation)하여 독자 포맷으로 저장 | 조회(Read) 속도 초고속, 직관적 뷰 | 큐브 생성/갱신(Build) 시간 길고 데이터 용량 한계 | Oracle Essbase, MS SSAS |
| **HOLAP** (Hybrid) | 상세 데이터는 ROLAP(관계형)에, 요약/집계 데이터는 MOLAP(큐브)에 하이브리드 저장 | 대용량과 빠른 응답 속도 동시 만족 | 아키텍처가 매우 복잡함 | SAP BW |

### 과목 융합 관점

- **SQL 옵티마이저**: 스타 스키마의 쿼리 성능을 최적화하기 위해, RDBMS 옵티마이저는 **스타 변환 (Star Transformation)** 기법을 사용한다. 큰 팩트 테이블을 스캔하기 전에, 작은 차원 테이블들의 조건(비트맵 인덱스)을 먼저 결합하여 팩트 테이블의 접근 범위를 극단적으로 줄이는 기술이다.
- **인메모리 DB**: 최근 하드웨어의 발전으로 데이터를 디스크에 큐브 형태로 굽는(MOLAP) 대신, 아예 ROLAP 형태의 데이터를 메모리에 모두 올려버려 실시간 집계를 초고속화하는 인메모리 OLAP(SAP HANA 등)가 대세가 되었다.

- **📢 섹션 요약 비유**: 요리(분석)를 할 때마다 재료를 썰어서 만드는 것(ROLAP)과, 미리 밀키트(다차원 큐브)를 만들어 냉장고에 넣어두고 전자레인지에 데우기만 하는 것(MOLAP)의 차이입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 유통 회사의 전사 대시보드 성능 튜닝

- **문제 상황**: Tableau 대시보드에서 '전국 매장 일일 판매 현황'을 로드하는 데 30초가 소요되어 현업의 불만이 접수됨. 기존 모델은 3NF로 정규화된 30개의 테이블을 조인하고 있었음.
- **의사결정**:
  1. **스타 스키마 설계**: '일일 판매 현황'이라는 주제를 중심으로 `팩트_매출` 테이블을 두고, 주변에 `차원_날짜`, `차원_매장`, `차원_상품` 3개의 테이블만 직접 연결(역정규화)하여 조인 횟수를 29번에서 3번으로 줄임.
  2. **구체화된 뷰 (Materialized View) / 큐브 활용**: 매일 자정에 어제까지의 데이터를 연도별, 월별, 카테고리별로 미리 그룹핑(Group By)하여 별도의 집계 테이블(Summary Table)을 만들어 둠. 
  3. **결과**: 대시보드 로딩 속도 30초 -> 1초 이내로 단축.

### 도입 체크리스트
- **비즈니스 요구의 명확성**: 팩트 테이블을 구성할 때 식별할 수 있는 가장 작은 정보 단위(Grain, 입도)를 어떻게 설정할 것인가? (예: 일별 매출인가, 영수증 1건당 매출인가?) 입도가 작을수록 유연하지만 데이터 양은 기하급수적으로 늘어난다.
- **천천히 변하는 차원 (SCD, Slowly Changing Dimension)**: 고객의 주소나 상품의 카테고리가 시간이 지나면서 변경되었을 때, 이를 어떻게 관리할 것인가? (과거 데이터는 옛날 주소로, 새 데이터는 새 주소로 집계하는 SCD Type 2 설계가 가장 널리 쓰인다.)

### 안티패턴: 데이터 마트에 무분별하게 스노우플레이크 적용하기
- 데이터 마트를 구축하면서 "저장 공간을 아낀다"거나 "정규화 원칙을 지켜야 한다"는 개발자의 관성 때문에 스노우플레이크 스키마를 고집하는 것은 최악의 안티패턴이다. 스토리지 비용은 매우 저렴해진 반면, 복잡한 조인으로 인한 CPU 연산 비용과 분석가의 쿼리 작성 난이도는 서비스 실패의 원인이 된다. DW 영역에서는 가독성과 읽기 성능(스타 스키마)이 최우선이다.

- **📢 섹션 요약 비유**: 도서관에서 책을 찾을 때 분야별, 연도별, 저자별 표지판(차원)을 큼직하게 달아두어야(스타 스키마) 누구나 쉽게 책을 찾지, 공간을 아끼겠다고 암호 기호로 분류해두면(스노우플레이크) 사서만 책을 찾을 수 있게 됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 3NF 정규화 모델 (OLTP) | 다차원 스타 스키마 (OLAP) | 개선 효과 |
|:---|:---|:---|:---|
| **쿼리 복잡도** | 조인 10개 이상, SQL 수백 줄 | 팩트와 차원의 1-Depth 조인 | **SQL 작성 시간 및 난이도 80% 감소** |
| **분석 쿼리 성능** | 인덱스 스캔 실패 시 타임아웃 | 차원 필터링 통한 대상 최소화 | **집계 응답 속도 10~100배 향상** |
| **현업의 자생력** | 매번 IT 부서에 데이터 추출 요청 | BI 툴 연동으로 직접 시각화 | **데이터 리터러시 강화 및 셀프 서비스 BI 실현** |

### 미래 전망
- **클라우드 데이터 웨어하우스의 추상화**: Snowflake나 BigQuery 같은 현대적 클라우드 DW는 강력한 분산 처리와 컬럼형 압축 스토리지(Columnar Storage)를 통해, 사실상 스타 스키마인지 정규화 모델인지를 따질 필요조차 없이 쿼리 속도를 보장하는 수준으로 하드웨어 스펙을 끌어올리고 있다.
- **AI 기반 자동 모델링**: 팩트와 차원의 설계, 큐브(Cube) 생성 주기를 DBA가 직접 설정하는 대신, 사용자의 쿼리 패턴을 AI가 학습하여 최적의 Materialized View와 인덱스를 백그라운드에서 자동 생성(Auto-tuning)하는 자율 주행 DB 시대로 진입하고 있다.

- **📢 섹션 요약 비유**: OLAP와 다차원 모델링은 단순한 데이터 정리를 넘어, 경영진에게 비즈니스의 현재를 보여주는 가장 강력한 '레이더 화면'이자 '나침반' 역할을 수행합니다.

- **[데이터 웨어하우스와 데이터 마트](./209_data_warehouse_data_mart.md)**: 다차원 모델링(OLAP)이 실제로 구축되고 저장되는 거시적 아키텍처 배경.
- **[구체화된 뷰 (Materialized View)](../3_sql_optimizer/153_materialized_view.md)**: MOLAP의 큐브(Pre-aggregation)와 동일한 원리로, RDBMS에서 집계 성능을 높이기 위해 결과를 물리적으로 저장하는 기술.
- **[컬럼 기반 저장소 (Columnar Store)](../1_db_basics_architecture/47_columnar_store.md)**: OLAP 쿼리의 압도적인 집계 성능을 뒷받침하는 물리적 스토리지 포맷 구조.
- **[정규화와 반정규화](../2_relational_model_normalization/112_denormalization.md)**: RDBMS의 정규화(3NF) 원칙이 OLAP 환경에서 어떻게 반정규화(스타 스키마)로 변형되는지에 대한 이론적 근거.

## 👶 어린이를 위한 3줄 비유 설명
1. 데이터베이스에 들어있는 엄청난 양의 장난감 판매 기록을 엑셀로 줄줄이 읽는 건 너무 지루하고 눈이 아파요.
2. **OLAP**는 이걸 요리조리 돌려볼 수 있는 마법의 큐브로 만드는 거예요. "지역별" 면을 보면 서울, 부산 매출이 보이고, 큐브를 돌려 "시간별" 면을 보면 봄, 여름 매출이 한눈에 들어오죠.
3. 이렇게 큐브를 만들기 쉽게 데이터를 '핵심 기록(팩트)'과 '분류 기준표(차원)'로 단순하게 별 모양처럼 짜놓는 것을 **스타 스키마**라고 부른답니다!
