# 데이터 웨어하우스와 데이터 마트 - Inmon vs Kimball, ODS

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 웨어하우스(Data Warehouse, DW)는 기업의 전략적 의사결정을 지원하기 위해 분산된 운영 시스템(OLTP)의 데이터를 **주제 중심적, 통합적, 시계열적, 비휘발성**으로 축적해 둔 전사적 중앙 저장소이다.
> 2. **가치**: Inmon의 'Top-down' 전사적 통합 모델과 Kimball의 'Bottom-up' 부서별 데이터 마트(Data Mart) 모델을 통해, 산재된 데이터를 정제(Cleansing)하여 신뢰할 수 있는 단일 진실 공급원(SSOT, Single Source of Truth)을 제공한다.
> 3. **융합**: 원천 시스템의 부하를 줄이기 위한 ODS(Operational Data Store) 및 ETL(Extract, Transform, Load) 파이프라인과 결합되어, 전통적인 비즈니스 인텔리전스(BI) 시스템의 근간을 완성한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 데이터 웨어하우스(DW)는 빌 인몬(Bill Inmon)이 정의한 바와 같이 "의사결정 지원을 목적으로 구성된 주제 지향적(Subject-oriented), 통합적(Integrated), 시계열적(Time-variant), 비휘발성(Non-volatile) 데이터의 집합"이다.

- **💡 비유**: 여러 마트(OLTP)에서 매일 팔린 수만 장의 지저분한 영수증들을 모아, 경영진이 한눈에 볼 수 있도록 품목별/월별로 깔끔하게 정리해 둔 '거대한 회계 장부 창고'와 같습니다.

- **등장 배경**:
  1. **OLTP의 분석 쿼리 한계**: 기존 운영 시스템(OLTP)에서 무거운 집계 쿼리(SUM, GROUP BY)를 돌리면 트랜잭션 락이 발생해 실제 비즈니스(예: 결제)가 멈추는 치명적 문제가 발생했다.
  2. **데이터 사일로 (Data Silo)**: 인사팀, 영업팀, 마케팅팀이 각자 다른 DB를 사용하여 전사적인 통합 분석(예: 마케팅 캠페인에 따른 영업 매출 변화)이 불가능했다.
  3. **과거 데이터 보존**: OLTP는 현재 상태만 유지(Update/Delete)하므로, "작년 3월의 고객 상태"를 추적하기 위한 시계열 저장 공간이 필요해졌다.

- **📢 섹션 요약 비유**: 나무를 베고 가구를 만드는 공장(OLTP) 한가운데서 장부를 정리하면 먼지가 날리고 시끄럽기 때문에, 조용하고 넓은 별도의 회의실(DW)을 만들어 과거의 모든 기록을 모아두는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. DW의 4가지 핵심 특징 (Inmon의 정의)

| 특징 | 설명 | RDBMS(OLTP)와의 차이 |
|:---|:---|:---|
| **주제 지향성 (Subject-oriented)** | 애플리케이션(급여, 주문) 중심이 아닌 비즈니스 주제(고객, 상품, 매출) 중심으로 구성 | 기능 중심 vs 데이터 중심 |
| **통합성 (Integrated)** | 여러 소스 시스템의 데이터 형식(단위, 성별 표기 등)을 일관되게 정제 및 변환 (M/F, 0/1 -> 남/여) | 사일로화된 데이터의 규격화 |
| **시계열성 (Time-variant)** | 데이터가 특정 시점의 스냅샷으로 저장되며, 수정되지 않고 과거 이력이 보존됨 | 현재 값만 유지 vs 과거 이력 유지 |
| **비휘발성 (Non-volatile)** | 한 번 적재(Load)된 데이터는 읽기 전용(Read-only)이며 수정이나 삭제(DML)가 발생하지 않음 | 잦은 C/U/D 발생 vs Batch Insert만 발생 |

---

### 2. Inmon vs Kimball 아키텍처 비교

DW를 구축하는 두 거장의 철학은 데이터 마트(Data Mart)의 생성 순서에서 극명하게 갈린다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 Inmon (Top-Down) vs Kimball (Bottom-Up)            │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [Inmon의 전사적 데이터 웨어하우스 (Top-Down)]                          │
  │                                                                  │
  │  Sources ──(ETL)──▶ [ Enterprise DW ] ──▶ [Data Mart A (영업)] │
  │                     (제3정규형, 원자적)     ──▶ [Data Mart B (마케팅)]│
  │  - 사상: 전사 데이터의 단일 통합(EDW)이 먼저. 이후 부서별 마트 생성.        │
  │                                                                  │
  │  [Kimball의 차원 모델링 (Bottom-Up)]                                 │
  │                                                                  │
  │  Sources ──(ETL)──▶ [Data Mart A (영업)] ──┐ Conformed        │
  │                   ▶ [Data Mart B (마케팅)] ─┼─▶ [ Enterprise DW ] │
  │                   ▶ [Data Mart C (재무)] ──┘  (논리적 결합)      │
  │  - 사상: 당장 필요한 부서별 데이터 마트(스타 스키마)부터 구축 후 통합.      │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
- **Inmon (탑다운)**: 정규화된 3NF 모델을 사용해 전사적 DW를 완벽하게 구축한 뒤, 이를 요약하여 부서별 데이터 마트를 만든다. 초기 구축 비용과 시간이 매우 크지만, 데이터 무결성과 전사적 일관성이 완벽히 보장된다.
- **Kimball (바텀업)**: 비즈니스 부서의 즉각적인 요구에 맞춰 다차원 모델링(스타 스키마) 기반의 데이터 마트를 먼저 빠르게 구축한다. 여러 마트를 결합할 때는 공통 차원(Conformed Dimension, 예: 시간/날짜 차원)을 사용하여 논리적인 전사 DW(Bus Architecture)를 완성한다. 애자일한 구축이 가능해 현대 BI에서 더 선호된다.

---

### 3. ODS (Operational Data Store)

ODS는 각 원천 시스템(Source DB)에서 DW로 데이터가 넘어가기 전, 데이터를 임시로 모아두는 '운영 데이터 저장소'다.

- **목적**: 원천 시스템의 부하를 줄이면서, DW보다는 최신의 데이터(준실시간)를 통합 조회하기 위함.
- **특징**: DW와 달리 데이터의 수정(Update)이 빈번하게 일어나며, 과거 이력보다는 '현재의 통합된 상태'를 유지한다. (예: 현재 고객의 총 포인트 잔액)

- **📢 섹션 요약 비유**: Inmon이 "완벽한 도시 계획을 세운 뒤 건물을 올리는 신도시 건설"이라면, Kimball은 "사람들이 많이 다니는 곳에 먼저 집을 짓고 나중에 도로를 연결하는 자연발생적 촌락 발전"과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 시스템 계층별 역할 비교 (OLTP vs ODS vs DW vs DM)

| 구분 | OLTP (운영 시스템) | ODS (운영 데이터 스토어) | DW (데이터 웨어하우스) | Data Mart (데이터 마트) |
|:---|:---|:---|:---|:---|
| **사용자** | 현업 실무자, 고객 | 시스템 관리자, 콜센터 | 데이터 분석가 | 특정 부서 (영업, 마케팅) |
| **초점** | 트랜잭션의 신속한 처리 | 단기적, 전사 통합적 운영 뷰 | 장기적, 전략적 트렌드 분석 | 부서의 전술적 의사결정 |
| **데이터 상태** | 100% 최신 상태 | 1~2시간 지연 (준실시간) | 하루~일주일 지연 (배치) | DW와 동일 지연 |
| **데이터 범위** | 개별 시스템 내로 한정 | 전사 시스템 통합 (단기 이력) | 전사 시스템 통합 (장기 이력) | 특정 주제 영역 (부서 한정) |

### 과목 융합 관점

- **데이터 공학 (ETL)**: Source -> ODS -> DW 로 이어지는 파이프라인에서 추출(Extract), 변환(Transform), 적재(Load) 기술이 결합된다. 특히 변환 과정에서의 데이터 품질(Data Quality) 확보가 DW 성공의 90%를 좌우한다.
- **경영 정보 시스템 (MIS)**: DW는 단순히 IT 부서의 시스템이 아니라, ERP(전사적 자원 관리)나 CRM(고객 관계 관리)에서 발생한 데이터를 의사결정 지표(KPI)로 바꾸는 경영 전략의 핵심 도구로 융합된다.

- **📢 섹션 요약 비유**: OLTP가 식당의 '주문 포스기'라면, ODS는 주방의 '실시간 요리 현황판', DW는 한 달 치 식자재와 매출을 분석하는 '창고 장부', 데이터 마트는 그중 음료수 판매량만 따로 뽑은 '음료 담당자용 요약본'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 통신사의 콜센터 화면 통합 프로젝트

- **문제 상황**: 고객이 콜센터에 전화하면, 상담원은 요금 DB, 가입 DB, 멤버십 DB 등 5개의 OLTP 화면을 각각 띄워놓고 고객 정보를 확인하느라 통화 시간이 길어지고 불만이 폭증함.
- **의사결정 (ODS 도입)**:
  1. 각 OLTP에서 실시간으로 변경되는 데이터를 CDC(Change Data Capture)를 통해 1분 주기로 **ODS**에 통합 적재.
  2. 상담원 화면은 무거운 집계가 필요한 DW가 아닌, '현재 상태'가 통합된 ODS를 바라보도록 아키텍처 개편.
  3. 야간 배치를 통해 ODS의 데이터를 장기 분석용 **DW**로 이관(ETL).

### 도입 체크리스트
- **데이터 적재 주기**: 비즈니스 부서가 요구하는 데이터의 최신성이 D-1(어제까지의 데이터) 배치인가, 아니면 준실시간인가? (배치라면 야간 ETL, 실시간이라면 CDC 및 ODS 필수)
- **Top-Down vs Bottom-Up**: 조직이 장기적인 IT 예산과 전사 데이터 거버넌스를 갖추고 있는가(Inmon)? 아니면 당장의 성과(Quick Win)가 필요한 애자일 조직인가(Kimball)?
- **ETL 병목 (Bottleneck)**: 데이터 양이 테라바이트급으로 커져 변환(Transform) 장비가 병목을 일으키는가? (이 경우 ELT로의 전환을 고려해야 함)

### 안티패턴: DW를 실시간 트랜잭션용으로 쿼리하기
- DW는 대량의 데이터를 배치로 한 번에 밀어 넣고, 무거운 그룹화 쿼리를 수행하기 위해 컬럼 스토어나 대형 해시 조인으로 최적화되어 있다. 이곳에 웹사이트의 로그인 세션이나 초당 수백 건의 단순 조회(Point Query)를 날리면, 시스템이 락에 걸리며 심각한 응답 지연이 발생한다.

- **📢 섹션 요약 비유**: 무거운 짐을 옮기기 위해 설계된 덤프트럭(DW)을 타고 좁은 골목길로 배달(실시간 트랜잭션)을 가려고 하면 사고가 날 수밖에 없습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (OLTP 집계) | 도입 후 (DW + Data Mart) | 개선 효과 |
|:---|:---|:---|:---|
| **운영 시스템 안정성** | 무거운 쿼리로 인한 결제 지연/장애 | 분석 부하 완벽 격리 | **운영 DB 장애율 90% 이상 감소** |
| **보고서 생성 시간** | 엑셀 수동 취합으로 며칠 소요 | 마트를 통한 즉시 조회 | **경영 지표 산출 시간 일 단위 -> 분 단위 단축** |
| **의사결정 신뢰도** | 부서마다 다른 매출 기준(사일로) | 단일 진실 공급원(SSOT) 기반 지표 | **데이터 정합성 보장 및 부서 간 이견 해소** |

### 미래 전망
- **클라우드 네이티브 DW**: 과거 수십억 원을 호가하던 온프레미스 DW 어플라이언스(Oracle Exadata, Teradata)는 저물고, 사용한 만큼만 컴퓨팅 비용을 지불하는 클라우드 DW (Snowflake, Amazon Redshift, Google BigQuery)가 완벽한 주류로 자리 잡았다.
- **데이터 레이크하우스(Lakehouse)로의 통합**: 정형 데이터만 다루던 전통적 DW는 한계를 맞이하여, 비정형 데이터(이미지, 로그)를 저장하는 데이터 레이크와 결합된 하이브리드 형태인 레이크하우스로 진화하고 있다.

- **📢 섹션 요약 비유**: 수백 명의 직원이 주판과 장부로 며칠을 계산하던 전통적 회계실이, 이제는 클라우드 위에서 수 초 만에 수조 건의 매출을 분석해 내는 AI 비서실로 탈바꿈하고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **데이터 웨어하우스(DW)**는 우리 가족의 모든 옛날 사진, 성적표, 영수증을 연도별로 아주 깔끔하게 모아둔 '대형 보물 창고'예요.
2. 하지만 창고가 너무 넓으면 내가 원하는 장난감을 찾기 힘들잖아요? 그래서 내 방에 따로 '내가 자주 노는 장난감 상자'를 두는데, 이게 바로 **데이터 마트(Data Mart)**랍니다.
3. 아빠(운영 시스템)가 새로 가져온 물건을 창고에 바로 넣기 전에 현관 앞에 잠깐 모아두고 먼지를 터는 곳이 있는데, 거기가 바로 **ODS**예요!
