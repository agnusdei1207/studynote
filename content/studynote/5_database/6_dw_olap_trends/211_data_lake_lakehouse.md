# 데이터 레이크와 데이터 레이크하우스 - ETL/ELT, 스키마 온 리드

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 레이크(Data Lake)는 구조화된 데이터뿐만 아니라 로그, 이미지, 영상 등 모든 비정형 데이터를 가공 없이 **원시 상태(Raw)**로 저장하는 무한 확장의 스토리지이며, 데이터 레이크하우스(Data Lakehouse)는 여기에 트랜잭션 처리(ACID) 능력을 결합한 최신 데이터 플랫폼이다.
> 2. **가치**: 사전 설계가 필수적인 ETL과 스키마 온 라이트(Schema-on-write)의 병목을 깨고, 일단 저장한 뒤 필요할 때 구조를 부여하는 **ELT**와 **스키마 온 리드(Schema-on-read)** 패러다임을 통해 데이터 적재 속도와 AI/ML 활용도를 극대화한다.
> 3. **융합**: 클라우드 객체 스토리지(S3, GCS)와 개방형 테이블 포맷(Apache Iceberg, Delta Lake, Hudi)이 융합되어 스토리지와 컴퓨팅 자원을 완벽히 분리(Separation of Storage and Compute)함으로써, 비용 효율적인 클라우드 네이티브 분석 환경을 완성한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **데이터 레이크**: 모든 종류의 데이터를 원래의 형태(Raw data) 그대로 중앙 집중식으로 저장하는 거대한 스토리지 풀.
  - **데이터 레이크하우스**: 데이터 레이크의 유연하고 저렴한 스토리지 위에 데이터 웨어하우스(DW)의 데이터 관리 기능(트랜잭션, 스키마, BI 지원)을 결합한 개방형 아키텍처.

- **💡 비유**: DW가 깨끗하게 정수되어 예쁜 페트병에 담긴 '생수(정형 데이터)'라면, 데이터 레이크는 물고기도 살고 흙탕물도 있는 거대한 '자연 호수(모든 원시 데이터)'입니다. 레이크하우스는 이 호수 위에 '첨단 정수 시설과 유람선 선착장(ACID, BI)'을 직접 띄워 놓은 것입니다.

- **등장 배경**:
  1. **DW의 한계와 비용 폭발**: 비정형 데이터(텍스트 로그, 음성, 센서 데이터)가 폭증했으나, 구조화된 RDBMS 기반의 DW는 이를 저장할 수 없었고 상용 DW 스토리지는 테라바이트당 유지 비용이 너무 비쌌다.
  2. **하둡(Hadoop) 생태계의 복잡성**: 초기 데이터 레이크는 Hadoop(HDFS) 기반으로 구축되었으나 관리 복잡도가 너무 높고 데이터가 방치되는 '데이터 늪(Data Swamp)' 현상이 발생했다.
  3. **AI/ML 파이프라인의 요구**: 데이터 사이언티스트들은 DW의 '이미 가공된 요약 데이터'가 아닌, 숨겨진 패턴을 찾을 수 있는 '가공되지 않은 날것(Raw)의 원천 데이터'를 원했다.

- **📢 섹션 요약 비유**: 요리사(분석가)가 마트에서 예쁘게 포장된 식재료(DW)만 사는 것에 만족하지 않고, 산과 바다(Data Lake)에 직접 가서 어떤 재료든 제한 없이 채집하여 요리(AI 모델링)를 시작하게 된 패러다임의 변화입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 스키마 온 라이트 (DW) vs 스키마 온 리드 (Data Lake)

데이터를 해석하는 '틀(Schema)'을 언제 적용하느냐에 따른 근본적인 패러다임 차이다.

| 구분 | Schema-on-Write (데이터 웨어하우스) | Schema-on-Read (데이터 레이크) |
|:---|:---|:---|
| **시점** | 데이터를 데이터베이스에 **쓸 때(Load)** 스키마 검증 | 데이터를 저장소에서 **읽을 때(Query)** 스키마 부여 |
| **적재 속도** | 느림 (형식에 맞지 않으면 에러 및 거부) | **매우 빠름** (일단 파일 형태로 무조건 저장) |
| **유연성** | 낮음 (사전 설계 및 ALTER TABLE 필요) | **높음** (하나의 데이터를 목적에 따라 다르게 해석 가능) |
| **주체** | 데이터베이스 관리자(DBA), 데이터 엔지니어 | 데이터 사이언티스트, 비즈니스 분석가 |

---

### 2. ETL vs ELT 아키텍처의 전환

클라우드 컴퓨팅 인프라의 막강한 파워 덕분에 데이터 변환(Transform)의 주체가 바뀌었다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 ETL 기반 DW vs ELT 기반 데이터 레이크               │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 전통적 ETL (Extract -> Transform -> Load) ]                     │
  │                                                                  │
  │  Sources ──▶ [ 별도 ETL 서버 ] ──▶ [ Data Warehouse ]              │
  │               (무거운 정제, 조인)        (구조화된 저장)                   │
  │               * 병목 발생 구역                                       │
  │                                                                  │
  │  [ 현대적 ELT (Extract -> Load -> Transform) ]                     │
  │                                                                  │
  │  Sources ──▶ [ Data Lake (S3) ] ──▶ [ Cloud DW / Lakehouse ]     │
  │              (원시 데이터 일단 저장)      (적재 후 강력한 컴퓨팅 파워로 변환)  │
  │                                                                  │
  │  - 장점: 중간 ETL 장비의 병목이 사라지고, 클라우드의 무한한 스케일 아웃 활용 │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 기존 **ETL**은 데이터를 추출한 뒤 별도의 ETL 도구(서버)에서 데이터를 정제하고 변환한 후 DW에 적재했다. 이 방식은 데이터 볼륨이 커질수록 ETL 장비가 병목(Bottleneck)이 되었다. 반면 **ELT**는 저렴한 객체 스토리지(Data Lake)에 원본 데이터를 그대로 쏟아부은(Load) 다음, Snowflake나 BigQuery 같은 막강한 클라우드 분산 엔진의 컴퓨팅 파워를 이용해 레이크 안에서 직접 변환(Transform) 쿼리를 돌리는 방식이다. 이를 통해 데이터 파이프라인의 민첩성이 극도로 향상되었다.

---

### 3. 데이터 레이크하우스 (Data Lakehouse) 아키텍처

데이터 레이크의 '저렴한 스토리지'와 DW의 'ACID 트랜잭션 보장'을 통합한 혁신적 아키텍처다.

1. **스토리지 계층 (Storage Layer)**: AWS S3, GCS 같은 클라우드 객체 스토리지에 Parquet, ORC, Avro 같은 개방형 열 기반(Columnar) 파일 포맷으로 저장.
2. **테이블/트랜잭션 계층 (Table Format Layer)**: **Apache Iceberg, Delta Lake, Apache Hudi**가 이 계층을 담당. 파일들의 모음을 하나의 '테이블'처럼 보이게 메타데이터를 관리하며, 여러 사람이 동시에 데이터를 읽고 써도 꼬이지 않도록 **ACID 트랜잭션**과 **타임 트래블(Time Travel, 과거 시점 복구)** 기능을 제공.
3. **컴퓨팅 계층 (Compute Layer)**: Spark, Presto, Trino 같은 분산 쿼리 엔진이 테이블 계층을 통해 스토리지를 병렬로 읽고 분석.

- **📢 섹션 요약 비유**: 넓은 공터(S3)에 파일들을 막 던져두던 레이크 시대에서, 공터 위에 정교한 도서관 카탈로그 시스템(Iceberg/Delta Lake)을 얹어 수백 명의 학생이 동시에 책을 대출하고 반납해도 절대 꼬이지 않는 레이크하우스 시대로 진화했습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### DW vs Data Lake vs Data Lakehouse 비교

| 비교 항목 | 데이터 웨어하우스 (DW) | 데이터 레이크 (Data Lake) | 데이터 레이크하우스 (Lakehouse) |
|:---|:---|:---|:---|
| **데이터 유형** | 정형 데이터 중심 | 정형, 반정형, 비정형 모두 수용 | 정형, 반정형, 비정형 모두 수용 |
| **트랜잭션 보장** | ACID 완벽 보장 | 파일 덮어쓰기로 인한 일관성 깨짐 위험 | **오픈 테이블 포맷을 통한 ACID 보장** |
| **분석 주체** | 비즈니스 인텔리전스 (BI) | 데이터 사이언스, 머신러닝 (ML) | **BI와 ML 모두 단일 플랫폼에서 지원** |
| **비용 구조** | 고비용 (스토리지/컴퓨팅 결합) | 저비용 (객체 스토리지) | **저비용 스토리지 + 사용량 기반 컴퓨팅** |
| **데이터 결함** | 사일로화된 마트로 분산 | 데이터 늪 (Data Swamp) 방치 위험 | 통합 메타데이터 거버넌스 제공 |

### 과목 융합 관점

- **운영체제 및 아키텍처**: 클라우드 환경에서 **스토리지와 컴퓨팅의 완전 분리(Separation of Storage and Compute)**가 레이크하우스를 가능케 한 핵심이다. CPU 자원(컴퓨팅)과 디스크 용량(스토리지)을 각각 독립적으로 오토스케일링할 수 있어 비용과 성능을 최적화한다.
- **소프트웨어 공학**: 데이터 메시(Data Mesh)와 같은 분산형 데이터 아키텍처 조직론에서, 각 도메인 팀이 자체 데이터를 소유하고 제공하는 물리적 인프라 베이스로 레이크하우스가 활용된다.

- **📢 섹션 요약 비유**: DW가 고급 레스토랑이고 데이터 레이크가 무질서한 식자재 도매상이라면, 데이터 레이크하우스는 모든 식자재를 보유하면서도 완벽한 위생 관리와 요리 시스템을 갖춘 최첨단 대형 복합 푸드코트입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 스트리밍 플랫폼의 시청 로그 통합 분석 시스템

- **문제 상황**: 일일 100억 건이 넘는 사용자의 영상 클릭/정지 로그(JSON)와 앱 에러 로그를 기존 DW에 넣으려니 스토리지 비용이 감당 불가능했고, 실시간 AI 추천 모델 학습을 위해 원시 데이터(Raw)가 필요했음.
- **의사결정 (레이크하우스 도입)**:
  1. **저장 (Data Lake)**: 모든 JSON 로그를 AWS Kinesis를 통해 S3 버킷에 원본 그대로 적재(Load)하여 스토리지 비용 극단적 절감.
  2. **포맷 변환 (Delta Lake/Iceberg)**: 압축률과 읽기 성능이 뛰어난 Parquet 포맷으로 변환하고, 개방형 테이블 포맷을 적용하여 데이터 변경/삭제 시의 트랜잭션 안전성 확보.
  3. **하이브리드 분석**: BI 팀은 데이터브릭스(Databricks)나 Athena를 통해 실시간 시청률 통계(SQL)를 뽑고, AI 팀은 Spark를 통해 동일한 S3 스토리지의 데이터를 읽어와 추천 알고리즘을 학습시킴.

### 도입 체크리스트
- **데이터 활용 목적**: 단순히 엑셀 스타일의 정형화된 경영 보고서(BI)만 필요하다면 여전히 DW가 유리할 수 있다. 레이크하우스는 AI/ML 파이프라인과 비정형 데이터 분석이 필수적일 때 빛을 발한다.
- **데이터 거버넌스(Governance) 역량**: 레이크에 쏟아지는 데이터에 카탈로그(AWS Glue, Hive Metastore)를 달고 접근 권한을 관리할 전담 데이터 엔지니어 조직이 있는가? 관리가 부재하면 즉시 데이터 늪(Swamp)이 된다.
- **개방형 포맷 종속성 방지**: 벤더(Vendor)에 종속되지 않기 위해 독자 포맷이 아닌 Apache Iceberg, Delta Lake 같은 오픈소스 테이블 포맷을 채택했는가?

### 안티패턴: '일단 무조건 다 붓고 보자'는 데이터 덤핑 (Data Dumping)
- 명확한 활용 목적이나 메타데이터 관리, 데이터 리니지(Lineage) 정책 없이 "저장 공간이 싸니까"라며 회사의 모든 로그를 데이터 레이크에 쏟아붓는 행위. 결국 아무도 그 데이터가 무슨 의미인지, 보안 등급이 무엇인지 몰라 쓰레기장(Data Swamp)이 되어 폐기 수순을 밟게 된다.

- **📢 섹션 요약 비유**: 거대한 호수(데이터 레이크)에 푯말이나 구역을 정하지 않고 물건을 마구 던져 넣으면, 나중에는 쓰레기를 건져 올리는 데 집어넣은 것보다 수십 배의 비용이 들게 됩니다. 철저한 카탈로그(지도) 관리가 필수입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기존 (DW + 사일로) | 도입 후 (데이터 레이크하우스) | 개선 효과 |
|:---|:---|:---|:---|
| **스토리지 비용** | 테라바이트당 고비용 인프라 종속 | 클라우드 객체 스토리지(S3 등) 활용 | **스토리지 비용 80~90% 절감** |
| **데이터 가용성** | 비정형 데이터는 버려짐 | 정형/비정형 모든 데이터 영구 보존 | **ML/AI 피처(Feature) 소스 무한 확보** |
| **파이프라인 속도** | 복잡한 ETL 프로세스 지연 | ELT 방식으로 적재 즉시 분석 가능 | **데이터 준비 시간(Time-to-Insight) 수 배 단축** |

### 미래 전망
- **스트리밍과 배치의 통합**: 과거 배치(Batch) 위주였던 레이크 생태계가 실시간 스트리밍(Kafka, Flink) 데이터를 Iceberg 같은 포맷을 통해 레이크하우스에 지연 없이 밀어 넣고 즉시 분석하는 방향으로 융합되고 있다.
- **모던 데이터 스택 (Modern Data Stack)의 표준화**: Fivetran(추출), dbt(적재), Snowflake/Databricks(스토리지/컴퓨팅), dbt(변환), Tableau(BI)로 이어지는 클라우드 기반의 파이프라인이 데이터 엔지니어링의 글로벌 업계 표준으로 굳어지고 있다.

- **📢 섹션 요약 비유**: 데이터 레이크하우스는 과거 부자들만의 전유물이었던 고급 식당(DW)을, 누구나 가장 신선한 재료로 마음껏 요리하고 실험할 수 있는 세계 최고의 '오픈 키친 플랫폼'으로 대중화시켰습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 옛날에는 과일이나 채소를 창고(DW)에 넣으려면 반드시 모양이 예쁘게 썰려 있고(정형 데이터), 예쁜 플라스틱 통(스키마)에 들어가 있어야만 했어요. (Schema-on-write)
2. **데이터 레이크**는 거대한 운동장이라서, 사과든 흙 묻은 감자든 포장도 안 뜯은 과자든 일단 다 던져놓고 보관할 수 있어 아주 싸고 편해요.
3. 하지만 운동장이 너무 어질러지니까, 최근에는 운동장 위에 스마트 정리 로봇을 가져다 두어서 물건이 섞이지 않게 관리하고 필요할 때 바로 꺼내 쓸 수 있게 만들었는데, 이게 바로 **데이터 레이크하우스**랍니다!
