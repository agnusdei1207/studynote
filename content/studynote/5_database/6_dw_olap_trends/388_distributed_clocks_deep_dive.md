# 388. 분산 클럭(Clocks) 심화 - 시간과 인과율의 조율

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 클럭은 물리적으로 떨어진 노드들 사이에서 **사건의 발생 순서(Ordering)를 일관되게 정의하기 위한 매커니즘**으로, 물리적 시계(Physical)와 논리적 시계(Logical/Vector)로 구분된다.
> 2. **가치**: 네트워크 지연이나 시계 오차(Clock Skew)가 불가피한 분산 환경에서, "누가 먼저 데이터를 수정했는가?"라는 질문에 수학적 해답을 제시하여 데이터 정합성을 유지한다.
> 3. **융합**: 구글 스패너의 TrueTime(물리+GPS) 기술이나 분산 NoSQL의 벡터 시계 기술과 융합되어, 분산 시스템의 전역적 일관성을 지탱하는 근간 아키텍처가 된다.

+++

### Ⅰ. 클럭의 종류와 특징

| 클럭 유형 | 작동 원리 | 주요 한계 / 특징 |
|:---|:---|:---|
| **물리적 클럭 (Physical)** | 하드웨어 수정 진동자 기반 | **Clock Skew (노드 간 오차)** 발생 필수적 |
| **람포트 클럭 (Logical)** | 사건 발생 시마다 카운터 증가 | 선후 관계는 알 수 있으나 동시성은 판별 불가 |
| **벡터 클럭 (Vector)** | 노드별 카운터 배열 유지 | **인과관계(Causality) 및 충돌 완벽 감지** |
| **하이브리드 (HLC)** | 물리 + 논리 결합 | 물리 시계의 직관성과 논리의 정확성 절충 |

+++

### Ⅱ. 논리적 클럭 매커니즘 시각화 (ASCII Model)

```text
[ Lamport vs Vector Clock ]

  1. Lamport Clock (Single Value)
     Node A: (1) ──▶ (2) ──▶ (Msg send: 3) ──▶ (4)
                                   │
     Node B: (1) ──▶ (Recv: Max(1,3)+1 = 4) ──▶ (5) ✅ "순서 정의"

  2. Vector Clock (Array [A, B])
     Node A: [1,0] ──▶ [2,0] (Send) ──▶ [3,0]
                                 │
     Node B: [0,1] ──▶ (Recv) ──▶ [2,2] ✅ "A의 2번 사건이 내게 왔다"
```

+++

### Ⅲ. 실무적 의의: 충돌 해결 (Conflict Resolution)

- **Causality (인과율)**: $A \to B$ 관계가 명확하면 덮어쓰기 가능.
- **Concurrency (동시성)**: 벡터 시계 비교 결과 엇갈림(둘 중 하나가 크지 않음)이 발생하면 "동시 수정"으로 판단하여 사용자에게 충돌 해결을 요청하거나 병합 로직을 수행합니다.
- **TrueTime (Spanner)**: 물리 시계의 오차 범위를 명시적으로 관리하여 "확실히 과거인 시간"이 지날 때까지 기다리는 방식으로 글로벌 정합성을 달성합니다.

- **📢 섹션 요약 비유**: 분산 클럭은 **'여러 도시에서 동시에 진행되는 퀴즈 대회의 타이머'**와 같습니다. 각 도시의 시계가 1초씩 다르다면 누가 먼저 버튼을 눌렀는지 알 수 없습니다. 이때 '서로 메시지를 주고받은 순서(논리 클럭)'나 '각 도시의 타이머를 합친 기록지(벡터 클럭)'를 보고 누가 진짜 1등인지, 혹은 동시에 눌렀는지를 판정하는 고도의 심판 시스템입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[Happened-before]**: 람포트가 정의한 사건의 인과 관계 기호 ($a \to b$).
- **[Clock Skew]**: 물리적 시계들이 서로 어긋나는 물리적 현상.
- **[NTP (Network Time Protocol)]**: 물리적 시계를 동기화하기 위한 표준 프로토콜.

📢 **마무리 요약**: **Distributed Clocks**는 분산 시스템의 나침반입니다. 시간의 상대성을 인정하고 논리적 질서를 구축함으로써, 거대한 분산 데이터의 세계에 명확한 선후 관계를 부여합니다.
