# 스트림 프로세싱 아키텍처 - Apache Kafka와 Flink 심층 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스트림 프로세싱(Stream Processing)은 멈춰있는 데이터(Data at Rest)를 야간에 모아서 처리하던 기존 배치의 한계를 깨고, 쏟아지는 무한한 이벤트 데이터(Data in Motion)를 **디스크에 영구 저장(Kafka)**함과 동시에 **밀리초 단위로 즉각 연산(Flink)**하는 실시간 데이터 파이프라인 아키텍처다.
> 2. **가치**: 관계형 DB의 락(Lock) 병목이나 폴링(Polling)의 부하 없이 초당 수백만 건의 이벤트를 순차적으로 흡수하며, **이벤트 타임(Event Time)** 처리와 **상태(State) 관리**를 통해 지연 도착한 데이터까지 오차 없이 완벽하게 집계해 낸다.
> 3. **융합**: 복잡한 람다(Lambda) 아키텍처를 단일 엔진으로 통일한 카파(Kappa) 아키텍처와 융합되어, 금융 이상 거래 탐지(FDS), 실시간 추천, CQRS의 읽기 뷰(Read Model) 동기화를 관장하는 현대 IT 시스템의 가장 강력한 혈관으로 작동한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **Apache Kafka (스토리지/버퍼)**: 데이터를 이벤트의 연속된 스트림으로 받아 디스크에 순차적(Append-only)으로 영구 보존하는 고성능 분산 메시지 브로커.
  - **Apache Flink (컴퓨팅/연산)**: Kafka에서 흘러나오는 무한한 스트림 데이터를 실시간으로 필터링, 조인(Join), 윈도우(Window) 집계 처리하는 상태 기반(Stateful) 분산 연산 엔진.

- **💡 비유**: Kafka는 끝없이 흘러나오는 수돗물을 멈추지 않고 담아내는 '거대한 저수지 파이프'입니다. Flink는 그 파이프 중간에 설치된 '초정밀 스마트 정수기'로, 물이 흘러가는 그 찰나의 순간에 오염 물질을 걸러내고 영양분을 즉시 섞어서 내보내는 역할을 합니다.

- **등장 배경**:
  1. **배치(Batch) 윈도우의 붕괴**: 과거 하둡(Hadoop) 기반의 MapReduce는 데이터 집계에 12시간이 걸렸다. 모바일 시대가 오며 "어제 가장 많이 팔린 상품"이 아닌 "지금 1초 동안 강남에서 가장 많이 클릭된 상품"을 요구하는 극단적 실시간성이 필요해졌다.
  2. **DB I/O 병목 해방**: 수만 대의 IoT 기기나 앱이 쏘는 로그를 직접 RDBMS에 INSERT하면 락(Lock) 경합으로 DB가 즉사했다. 이를 받아낼 압도적인 완충재(Buffer)가 필요했다 (Kafka의 탄생).
  3. **지연 데이터(Late Data)의 역설**: 스마트폰이 터널에 들어가 통신이 끊겼다가 1시간 뒤에 데이터가 도착했을 때, 이를 처리 서버 시간(Processing Time)이 아닌 실제 발생 시간(Event Time) 기준으로 완벽하게 소급 연산할 수학적 모델이 절실해졌다 (Flink의 탄생).

- **📢 섹션 요약 비유**: 물고기를 잡기 위해 매일 밤 그물을 치고 아침에 걷어 올리던(배치 처리) 어부가, 이제는 강물에 특수 센서망(스트림 프로세싱)을 깔아놓고 물고기가 지나가는 즉시 0.1초 만에 종류와 크기를 스캔해 내는 혁명입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 람다(Lambda) 아키텍처 vs 카파(Kappa) 아키텍처

빅데이터 생태계가 실시간성을 확보하기 위해 진화해 온 아키텍처의 패러다임 변화다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 Lambda 아키텍처 vs Kappa 아키텍처 비교               │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 람다 (Lambda) 아키텍처 ] - (과도기적 타협안)                        │
  │                               ┌──▶ (Batch Layer / Hadoop) ──┐    │
  │  Data Source ──▶ (Data Lake) ─┤                             ├──▶ │
  │                               └──▶ (Speed Layer / Storm) ───┘    │
  │  * 단점: 배치 코드(정확함)와 실시간 코드(부정확함) 두 벌을 유지보수해야 하는 지옥 발생.│
  │                                                                  │
  │  [ 카파 (Kappa) 아키텍처 ] - (현대적 표준)                             │
  │                                                                  │
  │  Data Source ──▶ (Kafka : 모든 데이터를 영구 스트림 로그로 저장)        │
  │                         │                                        │
  │                         ▼                                        │
  │                 (Stream Processing Layer / Flink)                │
  │                 실시간 연산 및 과거 데이터 재처리(Replay) 동시 수행        │
  │                                                                  │
  │  * 장점: 코드가 하나(스트림)로 통일됨. 배치 처리는 그저 '과거의 스트림'을 돌리는 것뿐.│
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 초창기 빅데이터는 정확도 100%를 보장하는 '느린 배치(Batch)'와 대충 빠르게 계산하는 '실시간(Speed)' 라인을 억지로 병렬 연결한 **람다 아키텍처**를 썼다. 하지만 개발자들은 두 벌의 로직을 일치시키는 데 피눈물을 흘렸다. 이를 혁신한 **카파 아키텍처**는 "모든 데이터는 결국 끝없는 이벤트의 연속(스트림)일 뿐이다. 정지된 데이터(배치)라는 건 환상이다"라는 철학 아래, Kafka에 데이터를 통째로 저장해 두고 Flink 하나로 실시간 처리와 과거 데이터 재처리(Replay)를 모두 커버하는 궁극의 단순화를 이룩했다.

### 2. Kafka의 압도적 성능 원리: Zero-Copy와 Sequential I/O

Kafka는 DB가 아니다. 하지만 디스크에 쓰는데도 메모리 DB급의 속도를 낸다.

- **순차 쓰기 (Sequential I/O)**: B-Tree처럼 디스크 헤드를 흔드는 랜덤 쓰기를 배제하고, 하드디스크의 맨 끝에 무조건 덧붙여 쓴다(Append Only). 이는 기계식 디스크 구조상 메모리 랜덤 쓰기보다 빠를 수 있다.
- **Zero-Copy 매커니즘**: 일반적으로 디스크의 데이터를 네트워크로 쏠 때 `디스크 -> 커널 버퍼 -> 앱 버퍼 -> 소켓 버퍼 -> 네트워크`의 4번 복사가 일어난다. Kafka는 OS의 `sendfile()` 시스템 콜을 악용하여, **디스크 커널 버퍼에서 냅다 바로 네트워크 소켓으로 꽂아버려(Zero-Copy)** CPU 개입을 0으로 만든다.

---

### 3. Flink의 윈도우(Window)와 워터마크(Watermark)

끝이 없는 스트림 데이터에서 "최근 5분간의 합계"를 구하기 위해 시간을 자르는 기술이다.

- **Event Time**: 데이터가 실제로 스마트폰에서 생성된 시간. (이걸 기준으로 집계해야 정확하다).
- **Processing Time**: 데이터가 Flink 서버에 도착한 시간.
- **워터마크 (Watermark)**: 터널을 지나느라 늦게 도착하는 지연 데이터(Late Event)를 얼마나 기다려줄 것인지에 대한 마지노선 타임스탬프다. "현재 워터마크가 10분이니까, 10분 이전 데이터는 다 도착했다고 가정하고 윈도우를 닫고 계산을 시작해!"라고 결단 내리는 시스템의 허용 오차 매커니즘이다.

- **📢 섹션 요약 비유**: 워터마크는 택배 마감 시간입니다. "오후 5시까지 모인 택배만 트럭에 싣고 출발한다(윈도우 닫음). 5시 1분에 도착한 택배(지연 데이터)는 다음 트럭에 싣거나 버린다!"라고 현실적인 통제선을 긋는 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 실시간 프로세싱 프레임워크 심층 비교

| 비교 항목 | Apache Spark Streaming | Apache Flink | Kafka Streams |
|:---|:---|:---|:---|
| **동작 패러다임** | **Micro-batch (마이크로 배치)** | **Native Stream (순수 스트림)** | Native Stream (경량 라이브러리) |
| **처리 지연(Latency)**| 초 단위 (1~5초) | **밀리초 단위 (10ms 이내)** | 밀리초 단위 (10ms 이내) |
| **상태(State) 관리** | 비교적 무거움 | **강력함 (RocksDB 내장, 스냅샷 체계)** | 강력함 (내부 상태 저장소 활용) |
| **클러스터 의존성** | 거대한 자체 마스터 클러스터 필수 | 자체 클러스터 또는 K8s 위 동작 | **별도 클러스터 불필요 (앱에 라이브러리로 내장)** |
| **적합한 유스케이스** | 기존 머신러닝/배치 로직 재활용 시 | **초저지연 금융 FDS, 완벽한 Exactly-once** | MSA 내에서 가볍게 Kafka 간 이벤트 변환 시 |

### 과목 융합 관점

- **데이터베이스 트랜잭션 (Exactly-once)**: 분산 환경에서 네트워크가 끊기면 메시지가 2번 전송되거나 누락되는 문제(At-least-once, At-most-once)가 필연적이다. Flink는 내부적으로 분산 스냅샷(Chandy-Lamport 알고리즘) 알고리즘을 융합하여, 시스템이 크래시 나더라도 결과 DB(Sink)에 데이터가 "정확히 한 번(Exactly-once)"만 저장되도록 트랜잭션 무결성을 수학적으로 보장한다.
- **분산 시스템 이론 (Stateful 처리)**: Flink가 스트림 연산을 하면서 임시 합계를 저장해야 할 때, 그 변수를 RAM에만 두면 죽었을 때 날아간다. 이를 막기 위해 노드 로컬에 **RocksDB (LSM-Tree 구조)**를 융합 내장하여 테라바이트급 상태(State)를 디스크에 실시간 백업하고 Checkpoint를 뜬다.

- **📢 섹션 요약 비유**: Spark Streaming이 "컨베이어 벨트에서 물건이 10개 쌓일 때마다 도장(Micro-batch)을 찍는 기계"라면, Flink는 "물건이 지나가는 찰나의 0.01초마다 움직이면서 도장(Native Stream)을 찍는 초정밀 레이저 기계"입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 카드사의 실시간 이상 거래(FDS) 탐지 시스템

- **문제 상황**: 기존에는 고객이 신용카드를 긁으면 1분 뒤에 이상 여부를 판별함. 도둑이 1분 안에 1,000만 원어치를 연속 결제하면 고스란히 털리는 사태 발생. 
- **의사결정 (Kafka + Flink 파이프라인 구축)**:
  1. 전 세계 포스기에서 발생하는 긁힘(Transaction) 이벤트를 **Kafka**로 0.05초 만에 중앙 수집. (수십만 건의 스파이크 트래픽을 완벽히 흡수하는 버퍼 역할).
  2. **Flink** 엔진이 Kafka를 구독하며 윈도우 연산 실행: "최근 1분 내에 (동일 카드 번호로) (서로 다른 국가에서) 결제 시도가 2번 이상 발생했는가?"라는 복합 State 매칭을 수행.
  3. Flink 내부에서 이상 탐지 시, 즉각 0.1초 만에 알람 토픽으로 이벤트를 쏴서 승인 서버가 결제를 **"승인 거절(Reject)"** 처리함. (배치 시대의 '수술'에서 스트림 시대의 '방패'로 완벽 진화)

### 도입 체크리스트
- **비즈니스의 실시간성 요구 (SLA)**: 데이터가 목적지에 도달하는 데 '5초' 걸리는 것을 비즈니스 부서가 못 참는가? 그렇다면 Kafka/Flink가 필수다. 하지만 '5분~10분 지연'도 괜찮다면, 굳이 난이도 극악의 스트림 프로세싱을 구축할 필요 없이 기존 RDBMS의 `Cron` 배치 스케줄러로 처리하는 것이 유지보수 비용을 1/10로 아끼는 길이다.
- **메시지 순서 보장 (Ordering)**: Kafka는 '하나의 파티션(Partition)' 내에서만 순서를 보장한다. 쇼핑몰 주문-취소 이벤트가 서로 다른 파티션에 들어가 역순으로 처리되면 "주문도 안 했는데 취소부터 되는" 파국이 일어난다. 도메인 키(예: Order_ID)를 파티션 키로 지정하여 멱등성과 순서를 보장하는 아키텍처가 코딩되어 있는가?
- **State 백업 및 Checkpoint 용량**: Flink가 과거의 데이터를 기억(State)하기 위해 주기적으로 남기는 스냅샷 용량이 너무 커지면, HDFS나 S3 네트워크 대역폭을 모두 잡아먹어 서비스가 멈춘다(Backpressure 현상). 증분 체크포인트(Incremental Checkpointing) 튜닝을 철저히 했는가?

### 안티패턴: Kafka를 단순한 장기 보관용 RDBMS처럼 사용하기
- Kafka의 성능에 심취한 나머지, 로그 삭제 기간(Retention)을 무한대(Infinite)로 설정하고 앱에서 Kafka의 오프셋(Offset)을 임의로 앞뒤로 조작해가며 `SELECT` 쿼리 날리듯 사용하는 행위. Kafka는 인덱스가 없는 '순차 테이프'이므로 특정 데이터를 건바이건으로 찾는 것은 최악의 비효율이다. 장기 보관과 복잡한 쿼리는 데이터 레이크(S3)나 RDBMS에 양보하고, Kafka는 철저히 파이프의 역할(통과 및 임시 버퍼)로만 써야 한다.

- **📢 섹션 요약 비유**: Kafka는 피가 온몸으로 막힘없이 돌게 해주는 '심장과 대동맥'이지, 에너지를 오랫동안 저장해 두는 '지방 덩어리(DB)'가 아닙니다. 혈관에 찌꺼기(무한 보존 데이터)가 쌓이면 시스템은 동맥경화로 즉사합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | RDBMS 폴링 (Polling) & 배치 | Kafka + Flink 스트리밍 | 개선 효과 |
|:---|:---|:---|:---|
| **이벤트 처리 지연 (Latency)**| 스케줄러(Cron) 주기에 종속 (최소 분 단위) | 데이터 발생 즉시 연속 처리 (ms 단위) | **실시간 비즈니스 반응 속도 1,000배 이상 극대화** |
| **시스템 스파이크 방어** | 트래픽 폭주 시 DB 커넥션 마비 및 OOM | Kafka가 버퍼 역할로 트래픽 1차 방어 | **피크타임 서버 생존율 100% 보장 (백프레셔 완화)** |
| **아키텍처 결합도** | WAS 간 직접 API 호출로 연쇄 장애 발생 | 브로커를 통한 Pub/Sub 비동기 격리 | **마이크로서비스(MSA) 간 물리적/논리적 완벽 디커플링** |

### 미래 전망
- **Streaming SQL의 대중화**: 과거에는 Java나 Scala로 복잡한 스트림 로직을 짜야 했으나, 현재는 **ksqlDB**나 **Flink SQL**이 상용화되어 데이터 분석가가 일반 SQL 쿼리(`SELECT * FROM stream GROUP BY TUMBLING WINDOW`)만 날리면 엔진이 알아서 실시간 스트림 파이프라인으로 번역해 주는 로우코드(Low-code) 시대로 혁신하고 있다.
- **실시간 Materialized View 융합**: 스트림 프로세싱 결과를 더 이상 외부 DB로 쏘지 않고, 처리 엔진 자체가 내부 메모리에 실시간 MView(구체화된 뷰)를 품고 외부 클라이언트의 REST API 조회를 직접 받아내는 스트리밍 데이터베이스(Materialize, RisingWave 등)가 차세대 유니콘으로 부상하고 있다.

- **📢 섹션 요약 비유**: 스트림 프로세싱은 비가 온 뒤에 고인 웅덩이(과거 데이터)를 퍼 담아 연구하는 것이 아니라, 떨어지는 빗방울을 공중에서 낚아채어 불순물을 거르고 목마른 자의 입에 즉시 떨어뜨리는 궁극의 '시간 정복' 기술입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 폭우가 쏟아질 때 양동이(일반 DB)로 빗물을 받으면 금방 꽉 차서 물이 다 넘쳐 흘러버려요. (서버 다운)
2. **카프카(Kafka)**는 아무리 물이 쏟아져도 멈추지 않고 흘려보내는 어마어마하게 크고 튼튼한 '대형 하수도 파이프'예요.
3. 파이프 끝에 **플링크(Flink)**라는 투시경 로봇을 매달아 두면, 물이 콸콸 지나가는 0.1초의 짧은 순간에도 "방금 지나간 물에 빨간 공이 3개 있었다!"라고 실시간으로 정확히 세어주는 천재랍니다.
