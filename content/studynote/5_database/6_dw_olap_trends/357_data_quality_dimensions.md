# 357. 데이터 품질(Data Quality) 6대 차원 - 정량적 평가의 기준

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 품질 6대 차원은 데이터가 비즈니스 목적에 부합하는지를 객관적이고 정량적으로 측정하기 위한 **정확성(Accuracy), 완전성(Completeness), 일관성(Consistency), 최신성(Timeliness), 유효성(Validity), 무결성(Integrity)**의 핵심 평가 기준이다.
> 2. **가치**: "우리 데이터는 믿을만한가?"라는 모호한 질문을 **DQ (Data Quality) 점수 및 지표**로 변환하여, 데이터 정화(Cleansing) 우선순위를 결정하고 데이터 자산의 가치를 유지하는 관리 체계를 제공한다.
> 3. **융합**: 데이터 거버넌스(Data Governance), ETL (Extract, Transform, Load) 파이프라인, 및 AI (Artificial Intelligence) 모델 학습 데이터셋 구축과 직결되며, "GIGO (Garbage In, Garbage Out)" 현상을 방지하는 최전선 기술이다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 정의
**데이터 품질(Data Quality)**이란 데이터가 특정 목적을 달성하기 위해 얼마나 적합한지를 나타내는 총체적인 수준을 의미한다. 단순히 데이터에 오류가 없는 상태를 넘어, 비즈니스 사용자가 의사결정을 내리거나 시스템이 운영될 때 **"신뢰할 수 있는 상태"**인지를 정의한다.
이를 평가하기 위해 **DAMA-DMBOK (Data Management Body of Knowledge)** 등의 국제 표준에서 정의한 6대 차원(Dimensions)이 널리 사용된다.

#### 2. 💡 비유
데이터 품질 관리는 **'정수 시스템'**과 같다. 원류(Source)에 흙탕물(불량 데이터)이 섞여 들어오더라도, 필터링(검증) 과정을 거쳐 최종 수도꼭지(비즈니스 애플리케이션)에서는 맑고 깨끗한 물(신뢰할 수 있는 정보)이 나와야 한다. 필터가 고장 나면 건강(비즈니스)이 망가진다.

#### 3. 등장 배경
① **기존 한계**: 과거에는 데이터의 양(Queue) 위주의 저장에 집중하여, 중복, 누락, 모순 등 데이터의 질 저하가 심각했고 이로 인한 잘못된 의사결정이 빈번했다.
② **혁신적 패러다임**: **Big Data** 및 **AI** 시대가 도래하며, 데이터 자체의 품질이 곧 알고리즘의 성능과 직결됨이 밝혀지면서, 정량적인 품질 관리가 필수불가결한 요소로 부상했다.
③ **현재의 비즈니스 요구**: 규제 준수(예: GDPR, 신용정보법)와 데이터 기반 경영(Data-Driven Decision Making)을 위해 고품질의 **Golden Record (단일 진실 공급원)** 확보가 절실해졌다.

#### 4. 📢 섹션 요약 비유
데이터 품질 6대 차원의 도입은 **'건강 검진의 6대 지표(혈압, 혈당, 체지방 등)'**를 정립하는 것과 같습니다. 어떤 하나가 문제가 있어도 전체 신체 시스템(기업 시스템)은 제 기능을 못 하므로, 이를 측정할 수표가 없으면 치료(개선)를 시작할 수조차 없습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 데이터 품질 6대 차원 상세 분석
데이터 품질의 6대 차원은 서로 독립적이면서도 상호 의존적인 관계를 가진다.

| 차원 (Dimension) | 핵심 정의 | 정량적 측정 지표 (Metric) | 실무적 확인 예시 |
|:---|:---|:---|:---|
| **1. 정확성 (Accuracy)** | 현실 세계의 **True Value**와 부합하는 정도 | 오류율 = (오류 건수 / 총 데이터 건수) × 100 | 고객의 주소지 DB와 우편물 배달 실패율 비교 |
| **2. 완전성 (Completeness)** | 필수 속성(Attribute)이 누락되지 않은 정도 | 누락율 = (NULL 건수 / 총 건수) × 100 | 주문 데이터에서 '배송지'가 비어있는 레코드 비율 |
| **3. 일관성 (Consistency)** | 이기종 시스템 간 데이터 값의 모순 없는 정도 | 위배율 = (불일치 건수 / 총 검사 건수) × 100 | ERP와 CRM의 동일 고객별 '등급' 값 불일치 건수 |
| **4. 최신성 (Timeliness)** | 비즈니스 유효 시점 내에 데이터 반영 여부 | 지연(Latency) = (현재 시간 - 데이터 생성 시간) | 어제 발생한 매출이 오늘 오전 9시 보고서에 반영됨? |
| **5. 유효성 (Validity)** | 정의된 도메인(Data Domain) 규칙 준수 여부 | 위배 건수 (COUNT) | 나이 컬럼에 '999'나 음수값이 들어있는가? |
| **6. 무결성 (Integrity)** | 데이터 간 참조 관계(Relationship)의 유지 여부 | 고아 레코드(Orphan) 수 | 주문 상세(Child)는 있는데 주문(Parent)가 없는 경우 |

#### 2. ASCII 구조 다이어그램: 데이터 품질 평가 프로세스
데이터 품질 관리는 일회성 사건이 아니라 지속적인 **PDCA (Plan-Do-Check-Act)** 사이클이다.

```text
      [Data Quality Management Lifecycle]

    +-----------------------------------------------+
    |  1. 정의 (Define)                              |
    |   - 비즈니스 목표 설정                          |
    |   - 측정 차원(6대) 선정                         |
    +---------------------------|-------------------+
                                v
    +---------------------------+-------------------+
    |  2. 측정 (Assess / Profiling)                 |
    |   - Data Profiling (통계적 분석)               |
    |   - Rules Engine (규칙 검증)                   |
    |   +---------------------------------------+   |
    |   | [Table: SAMPLE_TARGET]                  |   |
    |   | ID | Name |  Age | Reg_Date             |   |
    |   |----+------+-----+----------+            |   |
    |   | 01 | Kim  |  25 | 2024-01-| <= Null     |   |
    |   | 02 | Lee  | 999 | 2025-01-| <= Invalid  |   |
    |   +---------------------------------------+   |
    +---------------------------|-------------------+
                                v
    +---------------------------+-------------------+
    |  3. 개선 (Improve / Cleansing)                 |
    |   - 데이터 수정 (Correction)                   |
    |   - 결측치 처리 (Imputation)                   |
    |   - 중복 제거 (De-duplication)                 |
    +---------------------------|-------------------+
                                v
    +---------------------------+-------------------+
    |  4. 통제 (Control / Monitoring)                |
    |   - 대시보드 (Dashboard) 시각화                 |
    |   - 알림 (Alert) 발송                          |
    +-----------------------------------------------+
```

#### 3. 심층 동작 원리: 정확도(Accuracy)와 정밀도(Precision)
기술적으로는 **"True Positive(참 양성)"**를 얼마나 잘 맞추느냐가 핵심이다.
*   **정확도(Accuracy):** 전체 데이터 중 올바른 데이터의 비율.
*   **정밀도(Precision):** 검색/추출된 데이터 중 올바른 데이터의 비율.
예를 들어, `SELECT * FROM CUSTOMER WHERE AGE > 20;` 쿼리를 실행했을 때, **유효성(Validity)**은 SQL 문법이 맞는지이며, **정확성(Accuracy)**은 검색된 20살 이상 고객의 실제 나이가 진짜 20살이 맞는지(즉, DB에 저장된 값이 실제 값과 같은지)를 의미한다.

#### 4. 핵심 알고리즘 & SQL (품질 진단 코드)
데이터 품질 진단을 위해서는 **SQL (Structured Query Language)**을 활용한 프로파일링이 필수적이다.

```sql
-- [Example 1] 완전성(Completeness) 체크: NULL 값 비율 계산
SELECT 
    COUNT(*) AS total_records,
    COUNT(email) AS non_null_records,
    ROUND(100.0 * SUM(CASE WHEN email IS NULL THEN 1 ELSE 0 END) / COUNT(*), 2) AS null_rate_percentage
FROM users;

-- [Example 2] 무결성(Integrity) 체크: 고아 레코드(Orphan Record) 탐지
-- 참조키(FK) 위배: 주문(orders) 테이블의 user_id가 사용자(users) 테이블에 없는 경우
SELECT o.order_id, o.user_id
FROM orders o
LEFT JOIN users u ON o.user_id = u.user_id
WHERE u.user_id IS NULL;

-- [Example 3] 유효성(Validity) 체크: 이메일 형식 검증 (정규표현식 활용)
-- 대부분의 DBMS는 REGEXP 또는 LIKE를 사용하여 도메인 위배를 검출한다.
SELECT user_id, email
FROM users
WHERE email NOT LIKE '%_@__%.__%'; 
```

#### 5. 📢 섹션 요약 비유
6대 차원을 아키텍처로 구축하는 것은 자동차의 **'계기판과 경고 시스템'**을 만드는 것과 같습니다. 계기판(정확성/완전성 지표)을 통해 현재 속도와 연료 상태를 확인하고, 엔진 경고등(무결성/유효성)이 켜지면 즉시 정비(데이터 정제)를 해야 사고(시스템 다운)를 막을 수 있습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 심층 기술 비교: 차원별 Trade-off

| 구분 | **완전성 (Completeness)** | **정확성 (Accuracy)** | **최신성 (Timeliness)** |
|:---:|:---|:---|:---|
| **핵심 관심사** | "데이터가 다 있는가?" (양적 충실도) | "데이터가 맞는가?" (질적 진실성) | "데이터이 신선한가?" (시간적 효율) |
| **주요 병목 원인** | 시스템 통신 오류, 파싱 실패 | 입력 오류, 오타, 구형 코드값 | 배치(Batch) 지연, 네트워크 Latency |
| **해결 기술** | ETL 오류 처리 로직, Default 값 설정 | Master Data Management (MDM) | CDC (Change Data Capture), Real-time Pipeline |
| **Trade-off** | 강제 입력을 유도하면 가짜 데이터(정확성↓) 발생 가능 | 실제 검증에 비용이 많이 듦 (Latency↑) | 실시간 갱신을 강제하면 시스템 부하↑ |

#### 2. 과목 융합 관점
1.  **데이터베이스 (DB) & SQL**: DDL (Data Definition Language) 설계 단계에서 `NOT NULL` (완전성), `FOREIGN KEY` (무결성), `CHECK` (유효성) 제약조건을 선언하는 것이 품질 관리의 1차 방어선이다.
2.  **데이터 통신 & 네트워크**: 데이터 전송 중 패킷 손실이 발생하면 데이터의 **완전성**이 훼손된다. 이를 방지하기 위해 **TCP (Transmission Control Protocol)** 계층에서 재전송 요청 및 순서 보장 기능을 수행한다.
3.  **AI (Artificial Intelligence)**: 머신러닝 모델의 성능은 데이터 품질에 의해 결정된다. 훈련 데이터의 레이블이 잘못되면(정확성↓) **Garbage In, Garbage Out** 현상이 발생하여 모델의 신뢰도가 급격히 떨어진다. 따라서 AI 학습 전 **Data Preprocessing (전처리)**의 80%는 품질 보강에 쓰인다.

#### 3. 📢 섹션 요약 비유
데이터 품질 차원 간의 관계는 **'요리 재료의 관리'**와 유사합니다. 재료를 아주 신선하게 배송받았다(최신성) 하더라도, 양이 모자라면(완전성) 요리를 완성할 수 없고, 신선한 재료를 썩은 냉장고에 넣어두면(무결성/관리 실패) 망가집니다. 맛있는 요리(비즈니스 가치)를 위해 세 가지 조건을 모두 충족시켜야 합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 시나리오 및 의사결정 프로세스
**상황**: 이커머스 플랫폼의 마케팅 팀장이 "탈퇴 회원 대상 재가입 캠페인"을 기획했다.
**문제**: 캠페인 발송 후 오픈율이 0.1%로 역대 최저를 기록함.

**[기술사적 판단 프로세스]**
1.  **진단 (Diagnosis)**: 데이터 품질 리포트를 확인한다.
    *   *발견*: `CUSTOMER` 테이블의 `EMAIL` 컬럼에서 40%가 `test@test.com`, `a@a.com` 등 **더미 데이터(유효성 위배)**로 저장되어 있음.
2.  **원인 분석 (Root Cause)**: 회원가입 페이지 개발 시, 이메일 인증 절차를 생략하고 임의의 값이라도 넣어야 가입이 되도록(완전성 강조) 로직을 짰음. 개발자는 "가입률(Conversion Rate)을 높이기 위해서"라고 변명.
3.  **의사결정 (Decision)**:
    *   **[잘못된 판단]**: 대량의 이메일을 수동으로 수정하거나 캠페인을 포기한다. (비용 폭증 or 기회 손실)
    *   **[올바른 판단]**: 즉시 회원가입 로직에 **이메일 인증 API(Validaty Check)**를 도입하여 신규 데이터의 품질을 막고, 기존 데이터에 대해 **Email Hygiene Service**를 통해 정제(Cleansing)를 수행한다.
    *   **결과**: 초기 가입률은 소폭 하락하더라도, 유효 고객 데이터 확보로 인한 **LTV (Lifetime Value)** 증대 가능.

#### 2. 도입 체크리스트
*   **[
