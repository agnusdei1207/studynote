# 366. 투명한 데이터 암호화 (TDE, Transparent Data Encryption)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TDE(투명한 데이터 암호화)는 데이터베이스 엔진이 디스크에 저장되는 **데이터 파일(At-rest)을 실시간으로 자동 암호화하고, 메모리로 읽어올 때 자동으로 복호화**하는 보안 기술이다.
> 2. **가치**: 애플리케이션 소스 코드나 SQL 쿼리를 수정할 필요가 없어 **도입이 '투명(Transparent)'**하며, 물리적인 디스크 탈취나 백업 파일 유출로부터 데이터를 안전하게 보호한다.
> 3. **융합**: 하드웨어 보안 모듈(HSM) 및 외부 키 관리 시스템(KMS)과 결합되어, 암호화 키 자체의 보안성을 극대화하는 엔터프라이즈 보안의 표준으로 융합된다.

+++

### Ⅰ. TDE의 작동 매커니즘

- **암호화 시점**: 데이터가 메모리 버퍼 풀에서 디스크(데이터 파일, 로그 파일)로 쓰일 때 엔진이 즉시 암호화합니다.
- **복호화 시점**: 사용자가 쿼리를 던져 데이터를 디스크에서 메모리로 읽어올 때 즉시 복호화합니다.
- **애플리케이션 관점**: 사용자는 암호화 여부를 전혀 모른 채 평상시와 똑같이 SQL을 사용합니다. (복호화된 데이터를 받기 때문)

+++

### Ⅱ. TDE 아키텍처 및 키 계층 구조 (ASCII Flow)

TDE는 보안을 위해 2중 키 구조를 주로 사용합니다.

```text
[ TDE Key Hierarchy ]

  1. [ Master Key (MK) ] ──▶ 안전한 KMS/HSM에 보관
            │ (Encrypts)
            ▼
  2. [ Data Encryption Key (DEK) ] ──▶ DB 부팅 시 MK로 복호화되어 메모리 상주
            │ (Encrypts)
            ▼
  3. [ Actual Data Files (.dbf) ] ──▶ 디스크에 암호화된 상태로 저장 💾
```

+++

### Ⅲ. TDE의 장점과 고려사항

1. **도입 편의성**: 기존 시스템의 스키마 변경이나 앱 로직 수정이 전혀 필요 없습니다.
2. **범위 보호**: 테이블 단위, 혹은 데이터베이스 전체(Tablespace)를 통째로 보호할 수 있습니다.
3. **성능 부하**: 암복호화 과정에서 약 3~10% 내외의 CPU 오버헤드가 발생할 수 있습니다 (하드웨어 가속기 활용 시 최소화 가능).
4. **한계**: **데이터가 메모리에 올라와 있거나 네트워크로 전송되는 동안에는 복호화된 상태**이므로, 전송 중 암호화(TLS) 및 접근 제어(RBAC)와 반드시 병행해야 합니다.

- **📢 섹션 요약 비유**: TDE는 **'특수 금고가 있는 일기장'**과 같습니다. 내가 일기를 쓸 때는 평범하게 쓰지만(Transparent), 일기장을 덮고 금고(디스크)에 넣는 순간 자동으로 암호가 걸린 비밀 문서로 바뀝니다. 누군가 일기장을 통째로 훔쳐가도(디스크 탈취), 금고 열쇠(Master Key)가 없으면 안의 내용을 절대 읽을 수 없게 보호해 주는 든든한 금고입니다.

+++

### Ⅳ. 개념 맵 및 요약

- **[At-rest Encryption]**: 저장된 상태의 데이터 보호.
- **[KMS (Key Management Service)]**: 암호화의 핵심인 '열쇠'를 관리하는 서비스.
- **[Column-level Encryption]**: TDE와 달리 특정 열만 골라 암호화하는 방식 (보통 소스 수정 필요).

📢 **마무리 요약**: **TDE**는 가장 효율적이고 강력한 저장 데이터 보호 수단입니다. 물리적 보안의 빈틈을 메워주는 데이터베이스 보안의 '기본 사양'입니다.
