# 36. B-Tree (다진 탐색 트리) 원리 및 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이진 탐색 트리를 확장하여 하나의 노드에 다수의 키와 포인터를 가질 수 있게 설계된 **균형 잡힌 다진 검색 트리(Balanced Multi-way Search Tree)**다.
> 2. **효율성**: 모든 리프 노드가 같은 깊이에 존재하도록 자가 균형(Self-balancing)을 유지하며, 노드 당 키 개수를 늘려 트리의 높이를 낮춤으로써 디스크 I/O 횟수를 극적으로 절감한다.
> 3. **가치**: 삽입, 삭제, 검색 성능을 $O(\log n)$으로 일정하게 보장하여, 현대 RDBMS의 가장 보편적이고 핵심적인 **인덱스(Index) 표준 자료구조**로 활용된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**B-Tree (Balanced Tree)**는 데이터베이스와 파일 시스템에서 데이터를 효율적으로 저장하고 검색하기 위해 설계된 트리 기반 자료구조입니다. 1970년 루돌프 바이어(Rudolf Bayer)와 에드 매크라이트(Ed McCreight)에 의해 제안되었습니다.

### 2. 💡 비유: 다층 빌딩의 층별 안내도
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        B-Tree 구조 비유                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 루트 노드 (1층 로비): "100번대 매장은 왼쪽, 200번대는 오른쪽으로 가세요."      │
│  2. 브랜치 노드 (구역별 에스컬레이터): "110~150번은 저쪽, 160~190번은 이쪽입니다." │
│  3. 리프 노드 (실제 매장 입구): "여기가 120번 매장입니다!" (데이터 도착)          │
│                                                                             │
│  → 안내판(Key) 하나에 선택지가 여러 개(Multi-way)라, 계단을 조금만 올라도        │
│     수만 개의 매장 중 내가 원하는 곳을 금방 찾을 수 있음!                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **이진 트리의 한계**: 데이터가 많아지면 트리의 높이가 너무 깊어져, 디스크 읽기 횟수가 폭증함.
- **균형의 중요성**: 한쪽으로 치우친(Skewed) 트리는 최악의 경우 연결 리스트와 같은 $O(n)$ 성능으로 하락함.
- **디스크 블록 최적화**: 한 번의 디스크 블록 읽기($B$)를 통해 가급적 많은 키 정보를 가져와 트리 높이를 낮추려는 시도.

- **📢 섹션 요약 비유**: B-Tree는 '최단 거리로 목적지에 도착하게 해주는 입체 교차로'와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. B-Tree의 주요 특징 및 제약 조건 (Order $m$ 기준)

| 특징 | 상세 내용 |
|:---:|:---|
| **다진 검색 (Multi-way)** | 노드 하나가 최대 $m$개의 자식을 가질 수 있음. |
| **자가 균형 (Self-balancing)** | 모든 리프 노드는 항상 동일한 레벨(Level)에 위치함. |
| **노드 채움 제약** | 루트를 제외한 모든 노드는 최소 $\lceil m/2 \rceil$개의 자식을 가져야 함. (공간 활용도 보장) |
| **정렬 상태 유지** | 노드 내의 키 값들은 오름차순으로 정렬되어 저장됨. |

### 2. B-Tree 내부 구조 도해 (ASCII)

노드 내의 키(K)와 자식 포인터(P)의 배치를 시각화한다.

```text
    [ Root Node ]
    ┌────┬────┬────┬────┬────┐
    │ P0 │ K1 │ P1 │ K2 │ P2 │  (Key K1 < K2)
    └─┬──┴────┴─┬──┴────┴─┬──┘
      │         │         │
      │   ┌─────┴─────┐   │   ┌──────────────────────────┐
      │   │ [Branch]  │   │   │  - P0: K < K1 인 트리     │
      │   └─────┬─────┘   │   │  - P1: K1 < K < K2 인 트리│
      │         │         │   │  - P2: K > K2 인 트리     │
    ┌─▼──┐    ┌─▼──┐    ┌─▼──┐└──────────────────────────┘
    │Leaf│    │Leaf│    │Leaf│
    └────┘    └────┘    └────┘
```

### 3. 심층 동작 원리: 노드 분할 (Split) 및 합병 (Merge)
트리의 균형을 유지하기 위한 핵심 메커니즘입니다.
1. **삽입 (Insert)**: 리프 노드에 데이터를 넣다가 정해진 $m$개를 초과(Overflow)하면, 중간값(Median)을 부모로 올리고 노드를 반으로 **분할**함.
2. **삭제 (Delete)**: 데이터 삭제 후 노드의 키 개수가 최소 기준 미달(Underflow)이 되면, 형제 노드에서 빌려오거나 부모와 **합병**함.
3. **균형 유지**: 이 과정에서 트리의 높이가 위로 자라나거나 줄어들며, 항상 모든 리프의 높이를 일정하게 맞춤.

### 4. 핵심 알고리즘: 탐색 시간 복잡도
- **Time Complexity**: $O(\log_m n)$
- $m$ (차수)이 클수록 트리의 높이가 낮아져 성능이 비약적으로 향상됨. (보통 RDBMS는 수백 차수를 사용하여 3~4단계만으로 수억 건 검색 가능)

- **📢 섹션 요약 비유**: 노드 분할은 '상자가 꽉 차면 더 큰 상자로 옮기거나 상자를 두 개로 나누는 정리법'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. B-Tree vs Binary Search Tree (이진 탐색 트리)

| 비교 항목 | Binary Search Tree (BST) | B-Tree (Multi-way) |
|:---:|:---|:---|
| **자식 노드 수** | 최대 2개 | **최대 $m$개 (수백 개 가능)** |
| **트리 높이** | 매우 깊음 | **매우 낮음 (Flat)** |
| **디스크 I/O** | 많음 (높이만큼 발생) | **최소화 (Block 단위 읽기 최적화)** |
| **균형 여부** | 보장 안 됨 (최악 시 $O(n)$) | **항상 완벽한 균형 보장** |

### 2. 과목 융합 관점: 자료구조 및 OS
- **자료구조 융합**: B-Tree는 **B+Tree, B*Tree**로 진화하며 데이터베이스 인덱스의 근간이 된다.
- **운영체제(OS) 융합**: B-Tree의 한 노드 크기는 보통 OS의 **Page Size(4KB/8KB)**와 일치하도록 설정하여, 한 번의 Page Fault로 트리 노드 전체를 메모리에 적재한다.

- **📢 섹션 요약 비유**: 이진 트리가 '한 번에 한 계단씩 오르는 사다리'라면, B-Tree는 '한 번에 한 층씩 오르는 엘리베이터'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 인덱스 차수(Order)의 결정
- **상황**: 수억 건의 데이터를 가진 테이블에 인덱스를 설계하려 함.
- **판단**: 노드 크기가 너무 작으면 I/O가 늘어나고, 너무 크면 한 노드 내에서의 검색(CPU 연산) 부하가 커짐.
- **결단**: **디스크 블록 크기(보통 8KB)에 딱 맞게 차수 $m$을 계산**하여, 한 번의 I/O로 최대한 많은 키를 가져오게 설계함.

### 2. 안티패턴: 무분별한 인덱스 생성
- **증상**: 조회 속도를 높이려고 모든 컬럼에 B-Tree 인덱스를 생성함.
- **결과**: 데이터 삽입/삭제 시마다 수많은 B-Tree 노드의 분할과 합병이 발생하여 **쓰기 성능(DML)이 마비**됨.
- **조치**: 쿼리 패턴을 분석하여 **결합 인덱스(Composite Index)**를 활용하고, 인덱스 개수를 최소화하여 트리의 유지 비용을 관리해야 함.

### 📢 기술사적 결언
> "B-Tree는 **'디스크라는 거북이 위에 메모리라는 토끼를 태우는 기술'**이다. 하드웨어의 물리적 한계를 구조적 지혜로 극복해낸 알고리즘의 정수다. 기술사는 단순히 트리의 모양을 아는 것에 그치지 않고, **노드 채움 비율(Fill Factor)**이 인덱스 스캔 성능과 노드 분할 빈도에 미치는 영향까지 파악하여 실무에 적용해야 한다."

- **📢 섹션 요약 비유**: 인덱스 남발은 '모든 물건에 위치 추적기를 다는 것'과 같습니다. 배터리(시스템 자원)가 금방 방전되어 정작 물건을 쓸 수가 없습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (ROI)

| 지표 | 도입 전 (Full Scan) | 도입 후 (B-Tree Index) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **검색 시간** | $O(n)$ | **$O(\log n)$** | 데이터 증가에 따른 속도 저하 방어 |
| **디스크 I/O** | 수만 번 | **3~5번 이내** | 시스템 응답성 수천 배 향상 |
| **자원 효율** | CPU/Disk 전체 점유 | **최소 자원 사용** | 동시 접속자 수용량 증대 |

### 2. 미래 전망
- **B+Tree의 보편화**: 실제 상용 DB(Oracle, MySQL 등)는 범위 검색에 더 유리한 **B+Tree**를 기본으로 채택하고 있음.
- **Non-volatile Memory (NVM) 최적화**: SSD를 넘어 차세대 비휘발성 메모리 환경에 맞춘 **LSM-Tree**나 **BW-Tree** 등 변형된 트리 구조가 더욱 발전할 전망임.

- **📢 섹션 요약 비유**: 기술의 형태는 변해도 '가장 적은 노력으로 가장 확실하게 찾는' B-Tree의 철학은 미래 저장 기술의 중심축으로 남을 것입니다.

- **B+Tree (개량형 B-Tree)**: 리프 노드 간 연결 리스트가 추가된 형태.
