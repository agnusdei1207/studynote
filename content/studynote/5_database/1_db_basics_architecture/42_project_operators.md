# 42. 프로젝트 (Project) - 수직적 부분집합

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 관계 대수의 순수 관계 연산자 중 하나로, 릴레이션에서 사용자가 필요로 하는 **특정 속성(열)들만을 선택**하여 새로운 릴레이션을 생성하는 **수직적 연산**이다.
> 2. **수학적 특징**: 그리스 문자 파이(**$\pi$**)를 기호로 사용하며, 연산 결과물에서 발생할 수 있는 **중복된 튜플은 자동으로 제거**되어 집합의 성질을 유지한다. ($\pi_{attributes}(Relation)$)
> 3. **가치**: SQL의 `SELECT (Column List)` 절에 대응되며, 불필요한 데이터를 물리적/네트워크적으로 걸러내어 I/O 효율을 높이고 데이터 보안(민감 정보 제외)을 강화하는 핵심 도구다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**프로젝트 (Project)** 연산은 주어진 릴레이션에서 속성 리스트에 제시된 속성 값들만 추출하는 연산입니다. 릴레이션을 세로 방향으로 오려내는 것과 같아 **수직적 연산(Vertical Operation)**이라고도 불립니다.

### 2. 💡 비유: 사진 인화와 크롭 (Crop)
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        프로젝트 연산 비유                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 단체 사진(전체 데이터)에서 내 얼굴(특정 열)만 따로 뽑아내고 싶음.           │
│                                                                             │
│  1. 단체 사진: 릴레이션 (Relation)                                            │
│  2. 속성 리스트: 내가 보고 싶은 항목 (예: 이름, 연락처)                          │
│  3. 크롭된 사진: 프로젝트 결과 (Result Relation)                               │
│                                                                             │
│  → 사진 속 사람 수(행/튜플)는 그대로인 것 같지만, 배경이나 다른 사람의 몸(속성)은    │
│     모두 버리고 **'내가 정한 항목'**만 남기는 것이 바로 프로젝트!                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **데이터 미니멀리즘**: 테이블에는 수백 개의 컬럼이 있을 수 있으나, 특정 업무(예: 메일 발송)에는 '이메일' 주소 하나만 필요함.
- **집합론적 엄밀성**: 연산 결과에서 중복을 허용하지 않음으로써 관계형 모델이 수학적 '집합(Set)'임을 증명.

- **📢 섹션 요약 비유**: 프로젝트는 '복잡한 명세서에서 내가 읽고 싶은 항목에만 형광펜을 칠하는 것'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 프로젝트 연산의 주요 특성

| 특성 | 상세 내용 | 비유 |
|:---:|:---|:---|
| **기호** | $\pi$ (Pi) | 가위 |
| **연산 방향** | **수직적 (Vertical)** | 세로 자르기 |
| **결과 차수** | 입력 릴레이션의 차수보다 **작거나 같음** (열 개수 감소) | 다이어트 |
| **결과 카디널리티** | 중복 제거 시 입력보다 **줄어들 수 있음** | 인원 정리 |
| **중복 제거** | 결과 릴레이션 내 **중복 튜플은 하나만 남김** (Set Property) | 겹치기 금지 |

### 2. 프로젝트 동작 구조 (ASCII)

전체 테이블에서 특정 컬럼만 추출되고, 중복이 제거되는 과정을 시각화한다.

```text
    [ Relation: Student ]
    ┌────┬─────┬──────┬───────┐
    │ ID │ NAME│ GRADE│ CLASS │
    ├────┼─────┼──────┼───────┤
    │ S1 │ Kim │  A   │   1   │
    │ S2 │ Lee │  B   │   1   │
    │ S3 │ Park│  A   │   2   │
    │ S4 │ Choi│  B   │   1   │ 
    └────┴─────┴──────┴───────┘
               │
      [ π GRADE, CLASS (Student) ]
               ▼
    ┌──────┬───────┐
    │ GRADE│ CLASS │
    ├──────┼───────┤
    │  A   │   1   │ 
    │  B   │   1   │ <── (S2와 S4의 결과가 동일하므로 중복 제거)
    │  A   │   2   │
    └──────┴───────┘
    (차수는 4에서 2로 감소, 카디널리티는 4에서 3으로 감소)
```

### 3. 심층 동작 원리: 중복 제거의 물리적 비용
프로젝트 연산은 논리적으로 간단해 보이지만, 실제 DBMS 내부에서는 가장 무거운 작업 중 하나일 수 있습니다.
1. **추출**: 메모리 버퍼에서 지정된 컬럼만 읽어 들임.
2. **정렬 (Sorting)**: 중복을 찾기 위해 결과 데이터를 정렬함.
3. **비교 및 제거**: 인접한 데이터가 같으면 하나를 삭제.
4. **해싱 (Hashing)**: 정렬 대신 해시 테이블을 만들어 중복을 체크하기도 함.

### 4. 핵심 기술: Covering Index (커버링 인덱스)
- **개념**: 프로젝트 대상 속성들이 모두 인덱스에 포함되어 있는 경우.
- **효과**: 실제 데이터 페이지(Heap)를 읽지 않고 **인덱스 리프 노드만 읽어서** 프로젝트 연산을 완료함 (초고속 처리).

- **📢 섹션 요약 비유**: 중복 제거는 '초대장을 보낼 때, 이름만 적다 보니 같은 사람이 여러 번 나오면 한 통만 보내는 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 관계 대수 Project vs SQL SELECT

| 비교 항목 | 관계 대수 ($\pi$) | SQL SELECT |
|:---:|:---|:---|
| **중복 처리** | **기본적으로 제거 (Set)** | **기본적으로 유지 (Multiset/Bag)** |
| **명령어** | $\pi_{Name}(Student)$ | `SELECT Name FROM Student` |
| **SQL 일치 방법** | (없음) | `SELECT DISTINCT Name ...` 사용 시 동일 |

### 2. 과목 융합 관점: 보안 및 네트워크
- **보안(Security) 융합**: 특정 직무의 사용자에게 급여나 주민번호 컬럼을 제외하고 보여주는 **뷰(View)**를 생성하는 것은 프로젝트 연산의 보안적 활용이다.
- **네트워크(NW) 융합**: API 설계 시 클라이언트가 필요한 필드만 요청하는 **GraphQL**은 프로젝트 연산의 정신을 네트워크 프로토콜로 구현한 사례다.

- **📢 섹션 요약 비유**: 관계 대수가 '수학적 완벽주의자'라면, SQL은 '성능과 편의를 중시하는 실용주의자'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 네트워크 대역폭 최적화
- **상황**: 모바일 앱에서 사용자 목록을 불러오는데 속도가 너무 느림.
- **판단**: 백엔드에서 `SELECT *`로 모든 컬럼(소개글, 가입일 등)을 다 던져주고 있음.
- **결단**: **Project 연산을 강화**. 화면에 출력할 '이름'과 '프로필 사진 URL'만 추출하도록 쿼리를 수정하여 응답 데이터 크기를 90% 이상 축구함.

### 2. 안티패턴: 인덱스 스캔을 방해하는 무분별한 `DISTINCT`
- **증상**: 중복이 나올까 봐 습관적으로 `SELECT DISTINCT` 사용.
- **결과**: 이미 PK나 Unique 제약 조건 때문에 중복이 없는 컬럼임에도 불구하고, DBMS가 불필요한 **정렬/해싱 작업**을 수행하여 CPU 부하 가중.
- **조치**: 데이터의 무결성을 신뢰하고, 정말 필요한 경우에만 중복 제거 연산을 수행.

### 📢 기술사적 결언
> "프로젝트는 데이터의 **'수직적 압축'**이다. 우리가 다루는 데이터의 80%는 쓰이지 않는 '지방'과 같다. 기술사는 쿼리를 작성할 때 **필요 최소한의 속성**만을 정의함으로써, 디스크 I/O와 메모리 캐시, 네트워크 트래픽이라는 세 마리 토끼를 동시에 잡아야 한다. 또한, 물리적 구조인 **인덱스 설계** 단계에서 프로젝트 연산을 고려한 '커버링 인덱스' 전략을 수립하는 것이 진정한 전문가의 모습이다."

- **📢 섹션 요약 비유**: `SELECT *`는 여행 갈 때 집 안의 모든 물건을 트렁크에 넣는 것과 같습니다. 무겁고 느려지며 결국 정작 필요한 물건을 찾기도 힘듭니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (ROI)

| 지표 | 적용 전 (`SELECT *`) | 적용 후 (Project 명시) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **응답 페이로드 크기** | 100 KB | **5 KB** | 네트워크 비용 20배 절감 |
| **I/O 작업량** | Table Full Scan | **Index Only Scan** | 디스크 부하 1/10로 감소 |
| **보안 사고 위험** | 민감 정보 노출 위험 | **필요 정보만 노출** | 정보 유출 원천 차단 |

### 2. 미래 전망
- **Columnar Database**: 행 단위가 아닌 열 단위로 저장하는 DB(HBase, ClickHouse 등)에서는 프로젝트 연산이 물리적 읽기 효율의 극치를 보여줌.
- **Projection Push-down**: 데이터 레이크 환경에서 스토리지(S3 등)로부터 데이터를 읽어올 때, 아예 필요한 컬럼만 골라서 가져오는 기술이 더욱 고도화될 것임.

- **📢 섹션 요약 비유**: 미래의 데이터 처리는 '필요한 정보만 골라 담는 스마트한 핀셋'과 같은 프로젝트 연산이 주도할 것입니다.
