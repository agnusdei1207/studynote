# 47. 파일 매니저 (File Manager) - 데이터 파일 및 블록 관리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DBMS 커널의 하부 엔진인 **저장 관리자(Storage Manager)**의 핵심 모듈로, 데이터베이스의 논리적 데이터를 보조기억장치의 물리적 **파일 및 블록(Block/Page)** 구조로 할당하고 관리하는 인터페이스다.
> 2. **책임**: 운영체제(OS)의 파일 시스템과 직접 통신하거나 이를 우회(Raw Device)하여, 데이터의 영속성 저장, 공간 할당(Extent/Segment), 그리고 가용 공간(Free Space) 관리를 전담한다.
> 3. **가치**: 상위의 버퍼 매니저(Buffer Manager)에게 블록 단위의 I/O 서비스를 제공함으로써, 물리적 하드웨어의 복잡성을 은닉하고 전체 시스템의 **데이터 입출력 성능**을 결정짓는 토대가 된다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**파일 매니저 (File Manager)**는 데이터베이스 파일들을 생성, 삭제하고, 파일 내에서 블록을 읽거나 쓰는 작업을 총괄하는 소프트웨어 모듈입니다. 데이터베이스 전체를 구성하는 물리적 비트(Bit)와 바이트(Byte)의 집집 관리를 담당합니다.

### 2. 💡 비유: 거대한 창고의 구역 관리자
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        파일 매니저 비유                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 상위 매니저 (지배인): "A 상품(데이터) 좀 가져와주세요."                        │
│  2. 파일 매니저 (창고 관리자):                                                 │
│     - "A 상품은 '3번 창고(파일)', '5번 선반(블록)'에 있습니다." (위치 파악)        │
│     - "새 상품이 오면 '10번 구역(가용 공간)'에 넣겠습니다." (공간 할당)            │
│     - 창고 열쇠(OS 인터페이스)를 들고 직접 문을 열고 닫음. (I/O 수행)              │
│                                                                             │
│  → 복잡한 창고 바닥 구조는 몰라도, **'몇 번 파일의 몇 번 블록'**만 말하면           │
│     척척 해내는 든든한 일꾼!                                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **추상화의 필요성**: 쿼리 처리기나 인덱스 관리자가 물리적 디스크 주소를 일일이 계산하는 것은 비효율적임.
- **공간 관리의 효율성**: 데이터가 삭제되어 생기는 빈 공간(Hole)을 재사용하고, 데이터가 늘어날 때 디스크 공간을 동적으로 확보하기 위한 체계적인 관리 요구.

- **📢 섹션 요약 비유**: 파일 매니저는 데이터베이스라는 가상 세계와 디스크라는 물리 세계를 잇는 '차원의 문'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 파일 매니저의 주요 기능 및 역할

| 기능 요소 | 상세 설명 | 비유 |
|:---:|:---|:---|
| **파일 할당 (Allocation)** | 데이터베이스 생성 시 디스크 공간 점유 및 파일 생성. | 부지 확보 |
| **블록 관리 (Block Mgmt)** | 데이터를 읽고 쓰는 기본 단위인 블록(Page) 할당 및 반환. | 선반 번호표 |
| **가용 공간 관리 (FSM)** | 빈 블록이나 블록 내 남은 바이트를 추적하여 재사용 극대화. | 빈방 찾기 |
| **OS 인터페이스** | OS의 파일 시스템 API를 호출하거나 Raw I/O 직접 제어. | 출입 허가 |

### 2. 저장 관리자 내 파일 매니저 계층 구조 (ASCII)

상위 모듈과의 요청 흐름 및 하부 디스크와의 통신을 시각화한다.

```text
    [ Query Processor ] (SQL 실행)
           │
    [ Buffer Manager ] (메모리 캐싱)
           │ (Block Request: FileID, BlockNo)
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ File Manager ]                                               │
    │  ┌──────────────────────┐  ┌─────────────────────────────────┐  │
    │  │  File Catalog        │  │  Free Space Map (FSM)           │  │
    │  │  (파일 목록/크기 관리) │  │  (빈 블록 추적 및 할당)           │  │
    │  └──────────┬───────────┘  └────────────────┬────────────────┘  │
    │             │                               │                   │
    │  ┌──────────▼───────────────────────────────▼─────────────┐     │
    │  │  I/O Scheduler (I/O 요청 순서 최적화 및 시스템 호출)      │     │
    │  └──────────────────────────┬──────────────────────────────┘     │
    └─────────────────────────────┼────────────────────────────────────┘
                                  │ (System Call / Raw I/O)
    ┌─────────────────────────────▼────────────────────────────────────┐
    │ [ OS File System ] or [ Raw Storage Device ] (Data.dbf, Log.log) │
    └──────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: 블록 할당 전략 (Sequential vs Linked)
1. **연속 할당 (Sequential)**: 파일을 디스크의 인접한 블록들에 배치. Sequential I/O 속도가 매우 빠르나, 파일 확장 시 외부 단편화 발생.
2. **연결 할당 (Linked)**: 블록들을 포인터로 연결. 확장은 자유롭지만 무작위 접근(Random Access) 속도가 느림.
3. **인덱스 할당 (Indexed)**: 블록 주소들을 별도의 인덱스 블록에 모아 관리. 현대 DBMS의 일반적인 방식.

### 4. 핵심 기술: 데이터 파일 아키텍처 (Oracle 예시)
- **Datafile**: 물리적 OS 파일.
- **Extent**: 연속된 블록들의 묶음 (할당의 최소 단위).
- **Segment**: 특정 오브젝트(Table, Index)에 할당된 Extent들의 집합.

- **📢 섹션 요약 비유**: 할당 전략은 '아파트 층수를 미리 정해놓느냐(연속), 필요할 때마다 옥탑방을 올리느냐(연결)'의 차이와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. OS 파일 시스템 활용 vs Raw Device 직접 접근

| 비교 항목 | OS File System 기반 | Raw Device (Direct I/O) 기반 |
|:---:|:---|:---|
| **관리 편의성** | **매우 높음** (OS 툴 사용 가능) | 낮음 (DBMS가 모든 걸 책임짐) |
| **성능** | 보통 (OS 버퍼링 오버헤드) | **최상** (불필요한 복사 생략) |
| **백업/복구** | OS 명령어로 가능 | DBMS 전용 유틸리티 필수 |
| **주요 활용** | 일반적인 중소규모 DB | **금융, 대규모 고성능 OLTP** |

### 2. 과목 융합 관점: OS 및 스토리지 기술
- **운영체제(OS) 융합**: 파일 매니저는 OS의 **I/O 스케줄러(Elevator Algorithm 등)**와 경합하거나 협력하며, `mmap()`과 같은 메모리 매핑 기술을 활용하기도 한다.
- **스토리지 기술 융합**: SSD와 NVMe의 등장으로 파일 매니저는 디스크 헤드의 이동(Seek Time)보다는 **병렬 채널 활용(Queue Depth)**을 극대화하는 방향으로 진화하고 있다.

- **📢 섹션 요약 비유**: OS 파일 시스템이 '공용 주차장'이라면, Raw Device는 '나만 쓰는 전용 차고'와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 데이터 파일 자동 확장 (Auto-extend) 설정
- **상황**: 서비스 트래픽이 늘어나며 데이터 파일이 꽉 차서 DB가 멈춤.
- **판단**: 파일을 수동으로 늘리는 것은 운영 리스크가 큼.
- **결단**: **파일 매니저의 Auto-extend 기능을 활성화**하되, 한 번에 늘어나는 크기(Next Size)를 너무 작게 잡지 않음(단편화 방지). 디스크 임계치 알람(Monitoring)과 병행하여 안정성 확보.

### 2. 안티패턴: 너무 많은 데이터 파일 생성
- **증상**: 테이블마다 별도의 파일을 생성하여 관리.
- **결과**: OS의 **File Descriptor** 한계 도달, 파일 매니저가 관리해야 할 메타데이터 폭증으로 인한 성능 저하.
- **조치**: **테이블스페이스(Tablespace)** 개념을 활용하여 대형 파일 몇 개에 여러 테이블을 논리적으로 배치하여 관리 효율화.

### 📢 기술사적 결언
> "파일 매니저는 데이터베이스의 **'물리적 기초'**다. 아무리 화려한 쿼리라도 결국 디스크 블록 하나를 읽어오지 못하면 무용지물이다. 기술사는 소프트웨어적 논리 구조뿐만 아니라, 하부의 **파일 시스템 레이아웃, 스트라이핑(RAID), 블록 크기 불일치(Misalignment)**와 같은 물리적 이슈가 전체 성능에 미치는 파급력을 꿰뚫어 보아야 한다. 진정한 최적화는 파일 매니저가 디스크와 나누는 '대화'의 효율성에서 완성된다."

- **📢 섹션 요약 비유**: 수만 개의 파일을 만드는 것은 '물건 하나마다 금고를 따로 사는 것'과 같습니다. 금고 열쇠 관리하다가 하루가 다 갑니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (ROI)

| 지표 | 파일 매니저 최적화 전 | 파일 매니저 최적화 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **디스크 단편화** | 심함 (성능 저하) | **최소화 (Extent 단위 관리)** | I/O 효율 30% 향상 |
| **공간 재사용률** | 낮음 (낭비 발생) | **높음 (FSM 기반 재사용)** | 스토리지 비용 20% 절감 |
| **시스템 가용성** | 수동 확장 시 중단 | **자동 확장을 통한 무중단** | 비즈니스 연속성 확보 |

### 2. 미래 전망
- **Cloud-Native Storage**: EBS, S3와 같은 클라우드 스토리지를 직접 제어하며 지연 시간을 최적화하는 **Cloud-optimized File Manager**가 주류가 됨.
- **ZNS (Zoned Namespaces) SSD**: 데이터를 블록이 아닌 존(Zone) 단위로 묶어 관리하여 가비지 컬렉션 부하를 획기적으로 줄이는 차세대 파일 매니저 기술이 부상 중임.

- **📢 섹션 요약 비유**: 미래의 파일 매니저는 디스크라는 딱딱한 공간을 넘어, 클라우드라는 '무한한 바다'에서 데이터를 관리하는 항해사가 될 것입니다.
