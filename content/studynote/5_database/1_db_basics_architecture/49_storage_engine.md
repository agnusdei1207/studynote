# 49. 스토리지 엔진 (Storage Engine) 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DBMS의 상위 계층인 질의 처리기(SQL Layer)와 하위 계층인 물리적 저장소 사이에서 실제 데이터를 물리적 파일로 저장하거나 읽어오는 역할을 전담하는 **플러러블(Pluggable) 데이터 처리 엔진**이다.
> 2. **유연성**: MySQL과 같은 멀티 엔진 아키텍처에서는 사용자의 목적(트랜잭션 중심, 분석 중심 등)에 따라 **InnoDB, MyISAM, RocksDB** 등 서로 다른 물리적 구조를 가진 엔진을 선택적으로 장착할 수 있다.
> 3. **가치**: 인덱싱, 트랜잭션 잠금(Locking), 버퍼 관리, 데이터 회복(Recovery) 등 DB의 물리적 성능과 신뢰성을 결정짓는 핵심 로직이 집약된 **DBMS의 심장부** 역할을 수행한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**스토리지 엔진 (Storage Engine)**은 데이터베이스 엔진이 파일 시스템에 데이터를 생성, 조회, 업데이트, 삭제(CRUD)하기 위해 사용하는 기본 소프트웨어 컴포넌트입니다. 상위 SQL 레이어가 명령을 내리면, 스토리지 엔진은 이를 물리적 디스크 연산으로 변환합니다.

### 2. 💡 비유: 자동차의 가변 엔진 시스템
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        스토리지 엔진 아키텍처 비유                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. SQL 레이어 (운전대 및 대시보드): "시속 100km로 가자!" (명령 전달)            │
│  2. 스토리지 엔진 (엔진 본체): 명령을 받아 실제로 바퀴를 돌리는 물리 장치.         │
│     - InnoDB (고급 하이브리드 엔진): 힘이 좋고 사고 시 자동 복구됨. (안전)        │
│     - MyISAM (가벼운 가솔린 엔진): 복구는 어렵지만 단순히 달릴 땐 매우 빠름. (속도)│
│                                                                             │
│  → 껍데기(운전법)는 똑같아도, 본체(엔진)를 무엇으로 끼우느냐에 따라                │
│     차의 성격(트랜잭션 vs 조회 속도)이 완전히 달라지는 것과 같음!                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **다양한 워크로드**: 단순 웹 게시판(조회 위주)과 금융 시스템(정합성 위주)은 필요한 물리적 처리 방식이 상이함.
- **아키텍처의 분리**: SQL 파싱 로직과 데이터 저장 로직을 분리하여, 새로운 저장 기술(LSM-Tree, NVMe 등)이 등장했을 때 유연하게 교체할 수 있는 구조 지향.

- **📢 섹션 요약 비유**: 스토리지 엔진은 '데이터가 물리적으로 숨 쉬고 살아가는 공간'을 정의하는 규격입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 스토리지 엔진의 핵심 구성 요소

| 요소 | 상세 설명 | 비유 |
|:---:|:---|:---|
| **Handler API** | SQL 레이어와 소통하는 표준 인터페이스. (Open, Close, Scan 등) | 운전 인터페이스 |
| **Transaction Mgr** | ACID 준수를 위한 로킹 및 MVCC 관리. | 규칙 준수 감시 |
| **Buffer Manager** | 디스크 블록을 메모리에 캐싱하여 I/O 성능 가속. | 단기 기억력 |
| **Index Manager** | 데이터 위치를 빠르게 찾는 B-Tree 또는 Hash 관리. | 위치 내비게이션 |
| **Log Manager** | 장애 복구를 위한 WAL (Redo/Undo) 기록. | 블랙박스 |

### 2. DBMS 내 스토리지 엔진 계층 도해 (ASCII)

상위 레이어와의 통신 구조와 플러러블한 엔진 구성을 시각화한다.

```text
    [ Client Requests ]
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  DBMS SQL Layer (Parser, Optimizer, Authorization)              │
    └──────┬──────────────────────────────────────────────────────────┘
           │ (Standard Handler API Calls)
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  Pluggable Storage Engine Layer                                 │
    │  ┌──────────────────┐  ┌──────────────────┐  ┌───────────────┐  │
    │  │ [ InnoDB ]       │  │ [ MyISAM ]       │  │ [ Memory ]    │  │
    │  │ - Clustered Index│  │ - Table-level Lock│  │ - Hash Only   │  │
    │  │ - Row-level Lock │  │ - No Transaction │  │ - Volatile    │  │
    │  │ - Transactional  │  │ - Fast Select    │  │ - High Speed  │  │
    │  └────────┬─────────┘  └────────┬─────────┘  └───────┬───────┘  │
    └───────────┼─────────────────────┼────────────────────┼──────────┘
                │                     │                    │
    ┌───────────▼─────────────────────▼────────────────────▼──────────┐
    │  File System / Persistent Storage (OS, SAN, NAS, S3)            │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: 핸들러(Handler) API 통신
1. **명령 수신**: SQL 레이어에서 "조건 X인 데이터를 한 줄 읽어라"는 API 호출.
2. **물리 탐색**: 엔진은 자신의 인덱스(B+Tree 등)를 통해 데이터의 위치 파악.
3. **버퍼 조회**: 해당 블록이 메모리(Buffer Pool)에 있는지 확인 후, 없으면 디스크에서 로드.
4. **결과 반환**: 데이터 한 행을 SQL 레이어에 전달하고 포인터 이동.

### 4. 대표 엔진별 특징 비교 (MySQL 기준)
- **InnoDB**: 현대적 표준. 행 단위 락, MVCC, 외래 키 지원, 자동 복구(Crash Recovery).
- **MyISAM**: 과거 표준. 테이블 단위 락으로 쓰기 성능 제약, 복구 기능 부재, 단순 조회에 유리.
- **CSV Engine**: 텍스트 파일을 그대로 테이블처럼 사용.

- **📢 섹션 요약 비유**: 핸들러 API는 '만국 공용어'와 같아서, SQL 레이어는 상대가 누구든(어떤 엔진이든) 똑같은 방식으로 대화할 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 주요 엔진 간의 핵심 메커니즘 비교

| 비교 항목 | InnoDB | MyISAM | Memory |
|:---:|:---|:---|:---|
| **기본 구조** | **Clustered Index (B+Tree)** | Non-clustered Index | Hash / Skip List |
| **동시성 제어** | **행 단위 잠금 (MVCC)** | 테이블 단위 잠금 | 테이블 단위 잠금 |
| **영속성 보장** | Redo/Undo 로그 (완벽) | 없음 (부족) | 없음 (휘발성) |
| **메모리 활용** | Buffer Pool (데이터+인덱스) | Key Cache (인덱스만) | 전체 메모리 상주 |

### 2. 과목 융합 관점: OS 및 아키텍처
- **운영체제(OS) 융합**: 스토리지 엔진은 OS의 **Direct I/O**를 사용하여 OS 파일 시스템 캐시와의 중복을 피하고, 자신만의 정교한 버퍼 알고리즘을 사용한다.
- **컴퓨터 구조 융합**: 데이터 압축 및 암호화 연산 시 **하드웨어 가속기(AES-NI 등)**를 엔진 레벨에서 직접 호출하여 오버헤드를 줄인다.

- **📢 섹션 요약 비유**: 엔진을 고르는 것은 '집을 지을 때 철근 콘크리트(InnoDB)로 할지, 가벼운 목조(MyISAM)로 할지'를 정하는 설계의 첫 단계입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 트래픽 급증 시의 엔진 선택
- **상황**: 결제 트랜잭션이 많은 쇼핑몰 시스템.
- **판단**: MyISAM 사용 시 쓰기 작업 동안 테이블 전체가 잠겨 조회가 안 되는 현상 발생.
- **결단**: **InnoDB로 엔진 통일**. 행 단위 락을 통해 읽기/쓰기 동시성을 확보하고, 예기치 못한 서버 다운 시에도 트랜잭션 로그를 통해 데이터 손실을 방지함.

### 2. 안티패턴: 스토리지 엔진의 특성을 무시한 설정
- **증상**: InnoDB를 쓰면서 `innodb_buffer_pool_size`를 기본값(작은 값)으로 방치.
- **결과**: 고성능 엔진을 가져다 놓고도 매번 디스크를 뒤지는 성능 저하 발생.
- **조치**: 서버 가용 메모리의 70~80%를 해당 엔진의 버퍼로 할당하여 '엔진의 잠재력'을 해방함.

### 📢 기술사적 결언
> "스토리지 엔진은 데이터베이스의 **'물리적 유전자'**다. 유전자가 정해진 후에는 쿼리 튜닝으로 극복할 수 없는 구조적 한계가 존재한다. 기술사는 단순히 SQL 문법을 잘 쓰는 수준을 넘어, **B+Tree와 LSM-Tree**, **비관적 락과 낙관적 락** 등 각 엔진이 채택한 하부 알고리즘의 장단점을 꿰뚫고 비즈니스 요건에 최적인 엔진을 매칭할 수 있는 '시스템 인사이트'를 갖춰야 한다."

- **📢 섹션 요약 비유**: '엔진 설정'을 안 하는 것은 '스포츠카를 사놓고 1단 기어로만 달리는 것'과 같이 안타까운 일입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (엔진 최적 선택 시)

| 지표 | 엔진 부적합 시 | 최적 엔진 도입 시 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **동시 처리량** | 낮음 (Lock 경합) | **매우 높음 (Fine-grained)** | 5~10배 이상 처리량 증가 |
| **장애 복구 시간** | 수 시간 (체크/수동) | **수 분 (자동 로그 복구)** | 가용성(MTTR) 획기적 개선 |
| **데이터 압축률** | 낮음 | **높음 (엔진별 압축 기술)** | 저장 비용 50% 이상 절감 |

### 2. 미래 전망
- **LSM-Tree 엔진의 도약**: 쓰기 부하가 압도적인 빅데이터 시대를 맞아, B+Tree 대신 LSM-Tree(RocksDB 등)를 탑재한 스토리지 엔진이 주류로 부상 중임.
- **Cloud-Native Engines**: 로그 자체가 데이터가 되는 **Log-structured Storage** 엔진(예: Aurora)이 클라우드 환경의 새로운 표준으로 자리 잡음.

- **📢 섹션 요약 비유**: 미래의 엔진은 하드웨어의 종류(SSD, NVM, Cloud)에 따라 스스로를 최적화하는 '자율 주행 엔진'으로 진화할 것입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **InnoDB 엔진**: 가장 강력한 RDB 스토리지 엔진.
- **LSM-Tree 엔진**: 쓰기 최적화 엔진의 심장.
- **MVCC (다중 버전 병행 제어)**: 엔진의 동시성 제어 기술.
- **Buffer Pool**: 엔진이 데이터를 품고 있는 공간.
