# 데이터베이스 마이그레이션과 무중단 배포(Zero Downtime) 전략

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 24시간 365일 무정지 운영이 필수인 현대 IT 환경에서, 데이터베이스의 스키마를 변경(ALTER)하거나 이기종 DB로 교체할 때 **고객이 서비스 중단(Downtime)을 전혀 느끼지 못하게(Zero) 아키텍처를 스위칭하는 전환 전략**이다.
> 2. **가치**: 카나리(Canary) 및 블루/그린(Blue/Green) 배포와 결합하여 신/구 데이터베이스 간의 양방향 동기화(CDC)를 철저히 유지하다가, 전환 시점(Cut-over)에 L4/L7 로드밸런서의 트래픽을 단숨에 꺾음으로써 비즈니스 리스크와 장애 롤백(Rollback) 비용을 0으로 만든다.
> 3. **융합**: 소스 코드를 형상 관리(Git)하듯 DB 스키마의 변경 이력도 코드로 관리하는 Flyway, Liquibase 도구와 융합되어, 데이터베이스 영역마저 **CI/CD 파이프라인과 DataOps 데브옵스 철학의 완벽한 궤도**로 편입시킨다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **무중단 배포 (Zero Downtime Deployment)**: 서버의 버전 업그레이드나 DB 마이그레이션 도중에도 사용자 요청(트래픽)을 1초도 유실하지 않고 100% 처리하는 배포 기법.
  - **DB 스키마 마이그레이션**: 데이터베이스의 테이블 구조(컬럼 추가, 타입 변경 등)를 안전하게 다음 버전으로 이행하는 과정.

- **💡 비유**: 무중단 배포는 "달리는 자동차의 바퀴를 갈아 끼우는 기술"과 같습니다. 옛날에는 차를 갓길에 세우고(점검 시간) 타이어를 바꿨지만, 지금은 차가 시속 100km로 달리는 도중에 보조 바퀴(신규 DB)를 옆에 살짝 달아서 무게를 분산시킨 뒤 옛날 바퀴(구형 DB)를 떼어내어 승객이 전혀 눈치채지 못하게 하는 묘기입니다.

- **등장 배경**:
  1. **"야간 점검 시간"의 소멸**: 글로벌 서비스 시대가 열리며 한국의 새벽이 미국의 낮 시간이 되었다. "매주 목요일 새벽 2시~4시 DB 점검" 같은 공지는 글로벌 비즈니스의 사형 선고와 같아졌다.
  2. **ALTER TABLE의 락(Lock) 지옥**: 수억 건의 데이터가 있는 테이블에 컬럼을 하나 추가하려 치면 RDBMS가 수 시간 동안 테이블에 베타 락(X-Lock)을 걸어버려 서비스가 완전히 뻗어버리는 치명적 한계에 부딪혔다.
  3. **클라우드 이관(Migration)의 증가**: 온프레미스(물리 서버)의 오라클 DB를 AWS Aurora나 오픈소스 MySQL로 옮기기 위해, 수 테라바이트(TB)의 데이터를 수 주일에 걸쳐 이관하면서도 서비스는 계속 살려둬야 하는 고난도 미션이 일상화되었다.

- **📢 섹션 요약 비유**: 낡은 심장을 인공 심장으로 교체하는 수술에서, 뇌사에 빠지지 않기 위해 몸 밖의 인공 심폐기(CDC 동기화)로 피를 계속 돌리면서 안전하게 심장을 바꿔치기하는 생명 연장술의 극치입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. DB 무중단 마이그레이션 파이프라인 (CDC 동기화 기반)

Oracle(온프레미스)에서 MySQL(클라우드)로 이기종 DB를 무중단 이관할 때의 4단계 아키텍처다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 이기종 DB 무중단 마이그레이션 전환 (Cut-over) 흐름     │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ Phase 1: 초기 스냅샷 적재 (Initial Load) ]                      │
  │  기존 DB(Oracle) ──(1TB 데이터 통째로 복사)──▶ 신규 DB(MySQL)        │
  │  * 이때 웹 서비스는 기존 DB를 바라보며 평소처럼 트랜잭션을 계속 수행 중.      │
  │                                                                  │
  │  [ Phase 2: 실시간 CDC 변경분 동기화 (Replication) ]                 │
  │  (웹 서비스) ─▶ 기존 DB ──(Redo 로그/CDC)──▶ 신규 DB (격차 좁히기)     │
  │  * 복사하는 몇 시간 동안 새로 발생한 변경 데이터(델타)를 실시간으로 밀어 넣음. │
  │                                                                  │
  │  [ Phase 3: 양방향 동기화 및 카나리 검증 (Dual Write) ]               │
  │  - 사내 직원 트래픽 5%만 신규 DB로 라우팅하여 에러가 없는지(Canary) 확인. │
  │                                                                  │
  │  [ Phase 4: 컷오버 (Cut-over / Switch) ]                         │
  │  - 로드밸런서(DNS/Proxy) 방향을 1초 만에 '신규 DB'로 100% 전환!          │
  │  - 구형 DB는 폐기(Retire). 고객은 1초의 지연도 없이 이관 완료.             │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터가 너무 크면 복사하는 데만 며칠이 걸린다. 그 며칠 동안 사이트를 내릴 순 없으므로, 일단 과거 시점의 스냅샷 데이터를 신규 DB로 무식하게 붓는다(Initial Load). 이 복사 기간에 고객이 결제하여 새롭게 바뀐 데이터들은, Debezium(CDC)이나 AWS DMS 같은 마이그레이션 툴이 원본 DB의 로그를 추적하여 신규 DB로 끊임없이 쫓아가며 덮어쓴다(Catch-up). 양쪽 DB의 싱크로율이 99.99%에 달하는 바로 그 순간(Cut-over), 트래픽 라우터를 휙 꺾어버리면 서비스 중단율 제로(Zero Downtime)의 완벽한 엑소더스(대탈출)가 완성된다.

### 2. 스키마 무중단 변경 (Online DDL / pt-online-schema-change)

대용량 테이블에 컬럼을 추가할 때 락(Lock)이 걸리는 것을 방어하는 기법이다.

- **원리**: 
  1. 원본 테이블(A)과 구조가 똑같고 새로운 컬럼만 추가된 임시 그림자 테이블(A_new)을 껍데기만 만든다.
  2. 원본(A)에 트리거(Trigger)를 걸어, 새롭게 들어오는 INSERT/UPDATE 데이터를 A_new에도 똑같이 쓰도록(동기화) 세팅한다.
  3. 원본(A)의 기존 데이터를 백그라운드에서 천천히 A_new로 복사한다. (이때 서비스 락 없음).
  4. 복사가 다 끝나면 원본(A)의 이름을 A_old로 바꾸고, A_new의 이름을 A로 순간적으로 맞바꾼다(RENAME TABLE - 0.01초 소요).
- **도구**: MySQL의 `pt-online-schema-change`나 `gh-ost(GitHub)`가 이 알고리즘을 자동화해 주는 업계 표준 툴이다.

---

### 3. 데이터베이스 형상 관리 (Database Migration Tools)

DB 스키마 변경도 애플리케이션 소스 코드처럼 Git으로 버전을 관리하는 철학(Database as Code).

- **Flyway / Liquibase**: `V1__init_table.sql`, `V2__add_column.sql` 처럼 스크립트 파일에 버전을 매겨 저장해 두면, CI/CD 파이프라인에서 Spring Boot가 뜰 때 DB의 현재 버전을 확인하고 누락된 패치 스크립트(V2)를 자동으로 실행(Migration)해 주는 자동화 도구.
- **가치**: "개발 서버엔 컬럼이 있는데 운영 서버엔 컬럼을 안 만들어서 장애가 나는" 원시적인 휴먼 에러를 시스템 레벨에서 원천 봉쇄한다.

- **📢 섹션 요약 비유**: 코드는 최첨단 Git으로 관리하면서 DB 스키마는 여전히 엑셀과 수동 복붙(Ctrl+C/V)으로 관리하던 구시대를 청산하고, DB 설계도마저 완벽한 자동화 배포 파이프라인(CI/CD)의 톱니바퀴로 결합시키는 혁신입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 배포 전략 비교: 애플리케이션 vs 데이터베이스

WAS(앱)의 배포와 DB 배포는 차원이 다른 난이도를 가진다. DB는 '상태(State)'를 영구 보존해야 하기 때문이다.

| 비교 항목 | Blue/Green 배포 (Application) | 확장 계약 (Expand/Contract) 패턴 (Database) |
|:---|:---|:---|
| **동작 방식** | 신버전(Green) 띄우고, 로드밸런서 방향만 1초 만에 휙 돌리면 끝. | 컬럼 추가(Expand) -> 구/신 앱 모두 호환 구동 -> 데이터 동기화 -> 구버전 컬럼 삭제(Contract)의 수일이 걸리는 지난한 과정. |
| **롤백(Rollback)** | 라우터 방향만 다시 구버전(Blue)으로 돌리면 즉시 원상 복구 됨. | 한 번 데이터가 삽입/변형되면 롤백이 사실상 불가능함. (양방향 동기화 없이는 지옥). |
| **상태 유지** | Stateless 하므로 이전 인스턴스는 그냥 날려버려도 무방함. | Stateful 하므로 단 1건의 고객 데이터라도 날리면 법적 처벌 대상. |

### 과목 융합 관점

- **마이크로서비스 (MSA) 데이터 분리**: 거대한 모놀리식 DB를 5개의 마이크로서비스용 DB로 쪼개는 (Decompose) 이관 작업(Strangler Fig Pattern) 시, 이 무중단 CDC 기반 마이그레이션 아키텍처가 필수적으로 적용된다. 레거시 로직을 목조르는(Strangler) 동안 데이터의 양방향 일관성 유지가 프로젝트의 성패를 가른다.
- **클라우드 컴퓨팅 및 IaaS**: 클라우드가 제공하는 스토리지 블록 복제 기술(EBS Snapshot)과 결합하여, 이기종 DB 간 전환이 아닌 동일 벤더 버전 업그레이드(예: PostgreSQL 12 -> 15) 시에는 인프라 레벨의 순식간 볼륨 스위칭(Volume Switching)만으로 무중단 이관이 완료되는 눈부신 발전이 융합되었다.

- **📢 섹션 요약 비유**: 애플리케이션 배포가 단순히 '입고 있는 옷을 갈아입는 것(Stateless)'이라면, 데이터베이스 마이그레이션은 '내 머릿속의 모든 기억과 지식을 다른 뇌로 한 치의 오차 없이 복사해 넣는 것(Stateful)'입니다. 난이도와 철학이 완전히 다릅니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 채팅 앱의 DB 스키마 롤백 딜레마 방어

- **문제 상황**: 고객(User) 테이블의 `Name` 컬럼을 `First_Name`과 `Last_Name` 2개의 컬럼으로 분리하는 스키마 변경을 단행함. DB를 변경하고 새 애플리케이션을 배포했으나, 새 앱에 치명적 버그가 발견되어 옛날 앱으로 급하게 롤백(Rollback)하려 함. 그런데 **DB 스키마가 이미 분리되어버려 옛날 앱이 뻗어버리는 치명적 상황 발생**.
- **의사결정 (Expand-Contract 패턴 도입)**:
  1. **확장(Expand)**: 기존 `Name` 컬럼은 그대로 둔 채 `First`, `Last` 컬럼을 껍데기만 '추가'한다.
  2. **마이그레이션 (동기화)**: 구버전 앱과 신버전 앱을 동시 배포한다. 구버전 앱이 `Name`에 데이터를 쓰면, DB 트리거를 통해 `First/Last` 컬럼에도 동기화시킨다 (하위 호환성 100% 유지).
  3. **전환 (Transition)**: 신규 앱이 며칠간 완벽히 안정적으로 돌아가는 것을 확인한다.
  4. **수축 (Contract)**: 완벽히 검증이 끝난 후 비로소 기존 `Name` 컬럼과 트리거를 삭제(Drop)하여 스키마 마이그레이션을 마무리한다. (언제든 롤백 가능한 완벽한 방어막)

### 도입 체크리스트
- **CDC 지연 (Replication Lag)**: 구버전 DB에서 신버전 DB로 데이터를 동기화하는 도중(Phase 2), 네트워크 대역폭 한계나 로직 병목으로 인해 10초 이상의 지연(Lag)이 발생한다면? 이 상태로 컷오버(Cut-over)를 치면 10초 치의 고객 결제 데이터가 공중분해 된다. Lag이 0에 수렴하는 시점을 완벽히 모니터링하고 컷오버 스위치를 눌렀는가?
- **캐시(Redis) 인벨리데이션**: DB 마이그레이션이 끝났는데 앞단의 Redis 캐시가 여전히 옛날 DB의 키(Key)나 구조를 물고 있다면 대형 정합성 사고가 터진다. 컷오버 직후 전역 캐시를 플러시(Flush) 하거나 웜업(Warm-up)하는 파이프라인이 연계되어 있는가?
- **이기종 DB 함수(SQL) 번역 호환성**: 오라클에서 PostgreSQL로 넘어갈 때, 오라클의 `NVL()`, `SYSDATE` 같은 독자 함수는 당연히 작동하지 않는다. 애플리케이션의 모든 쿼리를 ANSI 표준 SQL이나 ORM(JPA) 계층으로 사전에 철저히 추상화(Abstraction)해 두었는가?

### 안티패턴: DB 이관 시 100% 무중단(Zero Downtime)에 대한 맹신과 과집착
- 트래픽이 거의 없는 새벽 3시 사내 ERP 시스템의 DB를 교체하면서도, "우린 무조건 무중단을 해야 해"라며 Kafka, Debezium, GoldenGate 등 수억 원의 솔루션과 수개월의 인력을 갈아 넣는 비효율. 정말 5분의 다운타임도 허용할 수 없는 코어 B2C 금융/커머스 서비스가 아니라면, 차라리 새벽에 "점검 창(Maintenance Window)" 배너를 1시간 띄우고 깔끔하게 데이터를 복사(Dump/Restore)하는 것이 비용과 정신 건강(ROI) 면에서 100배 이득이다.

- **📢 섹션 요약 비유**: 이사를 할 때(마이그레이션) TV와 인터넷이 1초도 안 끊기게 하려고 위성 중계기를 달면서 천문학적인 돈을 쓰는 짓을 할 필요가 없습니다. просто 10분 동안 플러그를 뽑았다가 새집에서 꽂는 것이 현명한 엔지니어링 타협입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 레거시 마이그레이션 (점검 시간 확보) | 무중단(Zero Downtime) 아키텍처 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **서비스 다운타임 (RTO)**| 작업 시간 동안 완전 셧다운 (수 시간 소요) | L4/DNS 스위칭만으로 전환 (1초 이내) | **글로벌 비즈니스 기회 손실 및 고객 이탈 0화** |
| **롤백 리스크 통제** | 롤백 실패 시 복구 불가능 (Data Corrupted) | 양방향 CDC 구축 시 즉각적 롤백 가능 | **대형 배포 시 심리적/물리적 리스크 완벽 제거** |
| **개발자 피로도** | 주말 새벽 출근 및 철야 작업(배포 대기) 빈발 | 주간 평일 업무 시간에 여유롭게 전환 | **DataOps 실현으로 데이터 팀 업무 생산성 극대화** |

### 미래 전망
- **DBaaS(Database as a Service)의 Blue/Green 네이티브 지원**: AWS RDS의 최근 버전들은 클릭 한 번으로 원본 DB의 완벽한 클론(Green)을 띄우고, 데이터 동기화를 유지하다가 개발자가 버튼을 누르면 내부 프록시(Proxy)에서 1초 만에 트래픽 방향을 전환해 주는 완전 관리형 블루/그린 배포 기능을 네이티브하게 제공하며 이관의 고통을 없애고 있다.
- **스키마 프리(Schema-Free) NoSQL로의 도피**: DDL(ALTER TABLE)의 고통을 아예 겪지 않기 위해, 고정된 테이블 열(Column) 구조를 버리고 Document DB(MongoDB)나 동적 열 추가를 지원하는 Wide-Column 스토어를 채택하여 마이그레이션이라는 개념 자체를 삭제해 버리는 아키텍처 진화도 가속화되고 있다.

- **📢 섹션 요약 비유**: 데이터베이스 마이그레이션과 무중단 배포는 비행 중인 항공기의 엔진을 교체하는 위험한 미션에서, 승객들은 영화를 보며 커피를 마시는 평온함을 누리게 만드는 현대 IT 인프라 마법의 결정체입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 낡은 냉장고(구형 DB)를 버리고 새 냉장고(신형 DB)를 샀어요. 원래는 냉장고 전원을 끄고 아이스크림을 다 옮길 때까지 한참을 기다려야 해서 아이스크림이 녹아버려요(서비스 중단).
2. **무중단 마이그레이션**은 일단 헌 냉장고 내용물을 새 냉장고에 싹 복사해 둔 뒤에, 투명한 로봇 파이프(CDC)를 연결해서 엄마가 헌 냉장고에 우유를 넣으면 0.1초 만에 새 냉장고에도 똑같이 복사되게 만들어요!
3. 두 냉장고 안의 내용물이 100% 똑같아진 순간! 아빠가 문짝을 확 바꿔치기(Cut-over)해서, 가족들은 냉장고가 바뀐 줄도 모르고 시원한 아이스크림을 계속 꺼내 먹을 수 있답니다!
