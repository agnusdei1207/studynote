# 시스템 카탈로그 (System Catalog) 및 데이터 사전 (Data Dictionary)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 카탈로그(또는 데이터 사전)는 데이터베이스에 저장된 모든 객체(테이블, 뷰, 인덱스 등)의 구조와 제약 조건에 대한 정보를 담고 있는 **'데이터에 관한 데이터(메타데이터)'** 저장소다.
> 2. **가치**: 사용자와 DBMS가 데이터의 위치와 형식을 즉시 파악하게 하여 쿼리 최적화 및 권한 제어를 가능케 하며, 시스템 스스로가 자신의 구조를 이해하게 하는 **자가 기술(Self-describing)** 능력을 부여한다.
> 3. **융합**: 일반 테이블과 동일한 구조로 관리되어 SQL로 조회가 가능하지만, 오직 DBMS만이 직접 수정할 수 있는 읽기 전용 구조를 통해 시스템 전체의 **무결성과 보안의 근간**을 유지한다.

+++

## Ⅰ. 시스템 카탈로그 (System Catalog)의 개요

### 1. 정의
- 데이터베이스 관리 시스템(DBMS)이 생성하고 유지하는 데이터베이스의 구성 요소들에 대한 명세서다.

### 2. 특징
- **메타데이터 (Metadata)**: 데이터베이스의 논리적/물리적 구조 정보 그 자체.
- **자가 기술성 (Self-describing)**: 데이터베이스가 스스로 무엇을 가지고 있는지 카탈로그를 통해 설명한다.
- **사용자 조회 가능**: 일반 사용자도 `SELECT` 문을 통해 시스템 정보를 조회할 수 있다. (예: `SELECT * FROM USER_TABLES`)
- **DBMS 전용 갱신**: 사용자가 `INSERT/UPDATE`로 직접 수정할 수 없으며, DDL(CREATE, ALTER 등) 실행 시 DBMS가 자동으로 갱신한다.

### 3. 💡 비유: 백화점의 안내 데스크와 층별 지도
- 백화점(DB)에 어떤 물건(데이터)이 어디에 있는지 일일이 돌아다니지 않아도, **안내 데스크(카탈로그)**에 가서 "신발 매장이 어디예요?"라고 물어보면 위치(주소)와 영업시간(제약 조건)을 알려주는 것과 같다. 이 안내 데스크의 지도는 백화점 주인(DBMS)만 고칠 수 있다.

## Ⅱ. 시스템 카탈로그의 구성 및 동작 아키텍처 (ASCII)

사용자의 요청이 실제 데이터에 도달하기 전 반드시 거쳐야 하는 '검문소' 역할을 한다.

```text
    [ 사용자 SQL 요청 ] (예: SELECT * FROM EMP)
              │
    +─────────▼──────────────────────────+
    |        [ DBMS Query Optimizer ]    |
    +─────────┬──────────────────────────+
              │ (1) 유효성 검사 및 위치 파악
    ┌─────────▼──────────────────────────┐
    │      [ 시스템 카탈로그 / 데이터 사전 ]   │
    │  - 테이블 명: EMP                  │
    │  - 컬럼: ID, NAME, SALARY          │ <--- 메타데이터 제공
    │  - 권한: 사용자 A (접근 가능)       │
    │  - 인덱스: ID_IDX (사용 가능)       │
    └─────────┬──────────────────────────┘
              │ (2) 실행 계획 수립 및 실행
    +─────────▼──────────────────────────+
    |        [ 실제 데이터 저장소 ]       | <--- 물리적 데이터 (0, 1)
    +------------------------------------+
```

### 1. 포함되는 주요 정보
- **스키마 정보**: 테이블, 뷰, 인덱스, 도메인 정의.
- **제약 조건**: 기본키, 외래키, CHECK 조건.
- **사용자 정보**: 사용자 계정, 비밀번호, 할당된 권한(Role).
- **통계 정보**: 데이터 양, 인덱스 깊이 등 (옵티마이저가 활용).

## Ⅲ. 데이터 사전 (Data Dictionary) vs 데이터 디렉터리 (Data Directory)

두 용어는 유사하지만 접근 권한 측면에서 구분된다.

| 비교 항목 | 데이터 사전 (Data Dictionary) | 데이터 디렉터리 (Data Directory) |
|:---|:---|:---|
| **내용** | 데이터베이스 객체에 대한 **논리적** 명세 | 메타데이터에 접근하기 위한 **물리적** 위치 |
| **가시성** | **사용자가 SQL로 조회 가능** | **시스템(DBMS)만 접근 가능** |
| **관리** | 논리적 관리 도구 | 물리적 관리 도구 |

- **📢 섹션 요약 비유**: 서점의 '베스트셀러 목록'은 누구나 볼 수 있는 데이터 사전이고, 점원이 창고에서 책을 꺼내오기 위해 보는 '창고 번호부'는 데이터 디렉터리입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. 성능 최적화와 카탈로그 통계
- **기술사적 결단**: 쿼리가 갑자기 느려졌을 때, 실제 데이터를 의심하기 전 카탈로그의 **통계 정보(Statistics)**가 최신인지 확인해야 한다. 
- **인사이트**: DBMS 옵티마이저는 카탈로그 정보를 바탕으로 최적의 길을 찾는데, 정보가 구식이면 엉뚱한 인덱스를 타게 된다. 따라서 실무에서는 주기적으로 `ANALYZE` 또는 `DBMS_STATS`를 실행하여 카탈로그를 현행화해야 한다.

### 2. 보안 감사(Audit)의 출발점
- **판단**: "누가 어떤 테이블에 접근 권한을 가지고 있는가?"에 대한 해답은 항상 시스템 카탈로그(`USER_TAB_PRIVS` 등)에 있다. 
- **결론**: 데이터 거버넌스 수립 시, 카탈로그 조회 권한을 엄격히 통제하여 스키마 구조 노출(Information Schema Leakage)을 방지해야 한다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **운영 효율성**: 복잡한 DB 구조를 수동으로 기록할 필요 없이 실시간 자동 문서화.
- **시스템 무결성**: 중앙 집중된 제약 조건 관리를 통해 데이터 일관성 원천 보장.

### 2. 미래 전망
최근에는 단일 DB를 넘어 기업 전체의 데이터 자산을 통합 관리하는 **데이터 카탈로그 (Data Catalog)** 기술로 진화하고 있다. AI가 수천 개의 시스템 카탈로그를 읽어 데이터 간의 관계를 스스로 추론하는 '데이터 리니지(Data Lineage)' 기술은, 현대 기업이 방대한 데이터의 바다에서 길을 잃지 않게 해주는 필수적인 지식 지도가 될 것이다.

- **📢 섹션 요약 비유**: 스스로를 들여다보는 거울이 있기에, 데이터베이스는 자신의 흐트러진 모습(무결성 파괴)을 바로잡고 성장할 수 있습니다.
