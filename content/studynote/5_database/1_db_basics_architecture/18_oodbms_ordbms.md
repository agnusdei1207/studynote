# 18. 객체지향 데이터 모델 (OODBMS) / 객체 관계형 데이터 모델 (ORDBMS)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 객체지향 프로그래밍(OOP)의 추상화, 캡슐화, 상속 개념을 데이터베이스(DB)에 도입하여, 복잡한 실세계 엔티티를 2차원 테이블이 아닌 객체(Object) 단위로 영속화(Persistence)하는 **고차원 데이터 관리 패러다임**이다.
> 2. **차별성**: **OODBMS (Object-Oriented Database Management System)**는 순수 객체 구조를 저장하여 임피던스 불일치(Impedance Mismatch)를 근본적으로 해결하며, **ORDBMS (Object-Relational Database Management System)**는 기존 관계형 DBMS(RDBMS)의 견고성과 SQL 표준을 유지하며 객체 기능(UDT, 상속 등)을 확장한 점진적 혁신 기술이다.
> 3. **가치**: 멀티미디어, CAD/CAM, GIS(Geographic Information System) 등 비정형 및 계층형 데이터를 효율적으로 처리하며, 데이터와 프로시저(행위)를 통합함으로써 애플리케이션 로직의 복잡도를 낮추고 데이터 무결성을 강화한다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 철학**
전통적인 **RDBMS (Relational Database Management System)**는 데이터를 행(Row)과 열(Column)로 구성된 테이블(Table)에 저장한다. 이는 데이터 간의 독립성과 무결성을 보장하는 데 탁월하지만, 현대 애플리케이션의 핵심인 **'객체(Object)'**의 구조와 데이터베이스의 **'릴레이션(Relation)'** 구조 사이에는 근본적인 모습 불일치가 존재한다. 이를 **임피던스 불일치(Impedance Mismatch)**라 하며, 개발자는 객체를 테이블로 변환하는 매퍼(Mapper) 코드를 짜야 하는 번거로움을 겪는다.
**객체지향 데이터 모델**은 이러한 한계를 극복하고자 데이터(Data)와 그 데이터를 조작하는 기능(Method)을 하나의 단위(객체)로 묶어 DBMS 차원에서 직접 지원하고 관리하는 기술적 철학이다. 단순한 값(Value)의 저장을 넘어, 추상화(Abstract Data Type)된 개체 자체를 보존한다.

**2. 등장 배경 및 필요성**
- **소프트웨어 위기와 복잡성**: 1990년대 이후 CAx(Computer-Aided x), 멀티미디어 시스템의 등장으로 텍스트가 아닌 이미지, 비디오, 복잡한 기하학적 좌표 데이터 등을 저장해야 했으나, 기존 RDBMS의 정형 스키마(Fixed Schema)로는 이를 효율적으로 표현하기 어려웠다.
- **생산성 갈망**: C++나 Java로 개발된 애플리케이션과 DB 간의 데이터 변환 오버헤드를 줄이고, 프로그래밍 언어의 모델과 DB 모델을 일치시켜 개발 생산성을 극대화하려는 수요가 폭발했다.
- **패러다임 전환**: 데이터만 저장하던 수동적인 DB에서, 데이터와 행위를 함께 저장하여 능동적으로 처리할 수 있는 '지능형 DB'로의 진화가 요구되었다.

**3. OODBMS와 ORDBMS의 분기**
시장의 요구는 두 갈래로 나뉘었다.
- **OODBMS**: "DB 자체를 객체지향 언어의 영구 저장 공간(Persistence)으로 만들자." (순수 객체 지향)
- **ORDBMS**: "검증된 RDB 위에 객체 기능을 덧씌워 사용하자." (관계형 확장)

> 기존 RDBMS가 '부품별로 분해해서 서랍에 넣는 방식'이라면, OODBMS는 '로봇을 완성된 상태 그대로 전시장에 보관하는 방식'이며, ORDBMS는 '기존 서랍장에 완성된 로봇의 부위(팔, 다리)를 통째로 넣을 수 있는 특수 칸을 추가한 하이브리드 서랍장'과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. 핵심 구성 요소 (Architecture Components)**
객체지향 데이터 모델은 데이터를 값(Value)이 아닌 능동적인 객체(Object)로 인식한다. 이를 지원하기 위한 5대 핵심 요소는 다음과 같다.

| 요소명 (Component) | 역할 (Role) | 내부 동작 (Internal Behavior) | 비유 (Analogy) |
|:---|:---|:---|:---|
| **OID (Object Identifier)** | 객체의 고유 식별자 | 테이블의 PK와 달리, 물리적 저장 위치가 변해도 불변하는 논리적 주소 값 사용. | 사람의 **주민등록번호** (집이 이사해도 번호는 변하지 않음) |
| **Encapsulation (캡슐화)** | 데이터 + 행위(메서드) 묶음 | 속성(Attribute)과 이를 조작하는 함수(Method)를 하나의 객체 내에 정의하여 외부 직접 접근을 제어. | **시계** (내부 톱니바퀴는 감추고, 버튼만으로 시간 조정) |
| **Inheritance (상속)** | 속성 및 행위의 재사용 | 상위 클래스(Superclass)의 구조를 하위 클래스(Subclass)가 물려받아 코드 중복을 최소화하고 IS-A 관계 표현. | **자동차 설계도** (기본 모델을 상속받아 SUV, 트럭 파생형 확장) |
| **Polymorphism (다형성)** | 인터페이스의 다양한 구현 | 동일한 메서드 호출이라도 객체마다 내부 로직이 다르게 동작하도록 지원 (오버라이딩). | **'그립' 명령** (볼펜은 잡고, 컵은 쥐는 식의 객체별 행위) |
| **Complex Objects (복합 객체)** | 중첩된 데이터 구조 | 객체 안에 다른 객체를 속성으로 포함(Nesting)하여, Collection이나 Array를 DB 스키마로 정의. | **러시아 인형** (인형 안에 또 다른 인형이 존재하는 구조) |

**3. 심층 동작 원리 (Deep Dive)**
- **객체 영속화(Persistence) 과정**: 
  1. 애플리케이션에서 `new Employee()`와 같이 객체 생성.
  2. OODBMS/OODBMS API는 이 객체에 **OID (Object Identifier)**를 부여하고, 메모리 상의 객체 상태(Transient)와 DB 상태(Persistent)를 동기화.
  3. 데이터를 저장할 때, 단순히 필드 값을 나열하는 것이 아니라, **VTBL (Virtual Table Pointer, 가상 함수 테이블 포인터)**까지 같이 저장하여 데이터를 꺼내 읽을 때 해당 객체의 행위(Method)까지 복원 가능하게 함.
- **네비게이션(Navigation) vs 조인(Join)**:
  - RDBMS는 데이터를 찾을 때 다른 테이블과 조인(Join)하여 조합하지만, 객체지향 모델은 객체 내부에 포함된 참조(Reference) 포인터를 따라 이동하는 **Navigation** 방식을 주로 사용한다. 이는 디스크 I/O를 획기적으로 줄일 수 있으나, 임의 쿼리(Ad-hoc Query)에는 불리할 수 있다.

**4. 핵심 알고리즘 및 코드**
> **SQL:1999 ORDBMS 예시 (사용자 정의 타입 및 메서드)**

```sql
-- 1. 추상 데이터 타입(ADT) 정의 (캡슐화)
CREATE TYPE Address AS (
    street VARCHAR(100),
    city VARCHAR(50),
    zip_code VARCHAR(10)
);

-- 2. 복잡한 객체 타입 정의 (중첩 구조)
CREATE TYPE Employee_T AS (
    name VARCHAR(50),
    addr Address,       -- 객체 참조 (중첩)
    salary NUMERIC,
    MEMBER FUNCTION get_annual_income() RETURN NUMERIC
);

-- 3. 메서드 구현 (행위 정의)
CREATE TYPE BODY Employee_T AS
    MEMBER FUNCTION get_annual_income() RETURN NUMERIC IS
    BEGIN
        RETURN SELF.salary * 12 + BONUS;
    END;
END;

-- 4. 테이블 생성 (객체 릴레이션)
CREATE TABLE Employees OF Employee_T;

-- 5. 객체 식별자(OID)를 통한 질의 및 메서드 호출
SELECT e.name, e.addr.city, e.get_annual_income()
FROM Employees e
WHERE e.addr.zip_code = '12345';
```

> **📢 섹션 요약 비유**:
> RDBMS가 "지도를 보고 길을 찾는 것(조인)"이라면, 객체 모델은 "친구의 손을 잡고 이동하는 것(네비게이션)"과 같습니다. 목적지가 명확할 때는 친구의 손(참조)을 잡고 가는 것이 훨씬 빠르고 직관적입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. OODBMS vs ORDBMS 심층 기술 비교**

| 비교 항목 | OODBMS (Pure Object Model) | ORDBMS (Object-Relational Model) |
|:---|:---|:---|
| **기반 철학** | DB 자체를 객체 저장소로 설계 | RDBMS 테이블 구조에 객체 기능을 확장 추가 (Extension) |
| **질의어** | **OQL (Object Query Language)**: SQL과 유사하나 객체 참조 반환 | **SQL:1999 / SQL3**: 기존 SQL 문법을 확장하여 객체 타입 지원 |
| **데이터 접근** | **Pointer Navigation**: OID를 통한 직접 메모리 접근 수준의 속도 | **SQL Join + Dereference**: 확장된 SQL 문으로
