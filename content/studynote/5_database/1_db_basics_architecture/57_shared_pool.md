# 57. 공유 풀 (Shared Pool) - Oracle 인스턴스 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Oracle DBMS의 시스템 글로벌 영역(SGA) 내에서 가장 복잡하고 핵심적인 메모리 영역으로, 실행된 SQL 문장과 실행 계획, 그리고 데이터베이스 메타데이터 정보를 모든 사용자가 공유할 수 있도록 캐싱하는 **중앙 집중식 정보 저장소**다.
> 2. **구성 및 역할**: SQL 실행 계획을 담는 **라이브러리 캐시(Library Cache)**와 스키마 정의를 담는 **데이터 딕셔너리 캐시(Row Cache)**로 구성되며, 동일한 쿼리를 재실행할 때 분석 과정을 생략하는 **소프트 파싱(Soft Parsing)**을 가능케 한다.
> 3. **가치**: 불필요한 CPU 연산(Hard Parsing)과 디스크 I/O를 최소화하여 수만 명의 동시 사용자가 지연 없이 시스템을 이용할 수 있도록 보장하는 현대 대규모 RDBMS 성능의 핵심 엔진이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**공유 풀 (Shared Pool)**은 Oracle 인스턴스의 SGA(System Global Area)를 구성하는 필수 메모리 컴포넌트입니다. 여러 사용자가 공통으로 사용하는 SQL 정보와 시스템 정보를 메모리에 유지하여 데이터베이스의 응답 속도를 높이는 역할을 합니다.

### 2. 💡 비유: 대형 식당의 '공용 레시피 북'과 '재료 위치도'
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        공유 풀(Shared Pool) 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 라이브러리 캐시 (공용 레시피 북):                                           │
│     - 이전에 손님이 시켰던 "A 요리 만드는 법(실행 계획)"을 적어둔 책.             │
│     - 다음 손님이 똑같은 걸 시키면, 연구할 필요 없이 이 책만 보고 요리함.          │
│                                                                             │
│  2. 데이터 딕셔너리 캐시 (재료 위치도):                                         │
│     - 소금, 설탕, 고기가 어디 있는지(메타데이터) 적어둔 지도.                     │
│     - 요리사가 창고 지도를 매번 그리지 않고 이 지도를 보고 재료를 바로 찾음.       │
│                                                                             │
│  → 이 두 정보가 담긴 **'공유 캐비닛'**이 바로 공유 풀! 주방 전체의 속도가 빨라짐.   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **파싱 오버헤드**: SQL을 기계어로 번역하고 최적의 길을 찾는 과정은 CPU를 매우 많이 소모함.
- **메모리 효율화**: 사용자마다 각자 실행 계획을 가지면 메모리가 낭비되므로, '똑같은 일은 한 번만 하자'는 철학 하에 통합 캐시 도입.

- **📢 섹션 요약 비유**: 공유 풀은 데이터베이스의 '지혜와 지식'을 모아둔 집단 지성의 보고입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 공유 풀의 핵심 구성 요소

| 요소 | 상세 설명 | 비유 |
|:---:|:---|:---|
| **Library Cache** | SQL 문장, 파스 트리, 실행 계획(Execution Plan) 보관. | 조리법 저장소 |
| **Dictionary Cache** | 테이블, 인덱스, 뷰, 컬럼 정보 및 권한 정보 보관. (Row Cache) | 재료 명부 |
| **Control Structures** | 락(Lock), 래치(Latch) 등 메모리 자원 관리용 제어 정보. | 주방 관리 열쇠 |
| **Reserved Pool** | 대규모 연속 메모리 할당 요청을 위해 비워둔 예약 공간. | 예약석 |

### 2. 공유 풀 내 데이터 흐름 아키텍처 (ASCII)

사용자 SQL이 공유 풀을 거쳐 실행되는 논리적 경로를 시각화한다.

```text
    [ Client SQL ] : "SELECT * FROM emp WHERE id = :1"
           │
    ┌──────▼────── [ Shared Pool ] ───────────────────────────────────┐
    │                                                                 │
    │  1. [ Library Cache Check ] (Hashing)                           │
    │     - "똑같은 SQL 문장이 있는가?"                                │
    │     - YES: 실행 계획 재사용 (Soft Parsing) ──▶ [ Execution ]     │
    │     - NO : 다음 단계로 진행 (Hard Parsing)                      │
    │                                                                 │
    │  2. [ Data Dictionary Cache Check ]                             │
    │     - "EMP 테이블이 있는가? 권한이 있는가?"                      │
    │     - 정보를 참조하여 Optimizer가 실행 계획 수립                 │
    │                                                                 │
    │  3. [ Store New Plan ]                                          │
    │     - 새로 만든 계획을 Library Cache에 저장하여 다음 공유를 대비    │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: 해싱(Hashing)을 통한 고속 매칭
- **동작**: 사용자 SQL 문장 전체를 해시 함수에 넣어 **해시 값(Hash Value)**을 생성.
- **비교**: 라이브러리 캐시에 동일한 해시 값을 가진 엔트리가 있는지 단숨에 확인.
- **주의**: 대소문자 하나, 공백 하나만 달라도 해시 값이 변하여 새로운 쿼리(Hard Parse)로 인식됨.

### 4. 핵심 기술: LRU (Least Recently Used) 관리
- 공유 풀은 유한한 공간이므로, **LRU 알고리즘**을 통해 자주 쓰이지 않는 실행 계획은 밀어내고 최신 정보를 유지함.
- 'Pinned' 상태의 오브젝트는 메모리에서 밀려나지 않게 보호 가능.

- **📢 섹션 요약 비유**: 해싱은 '지문 인식'과 같아서, 아무리 긴 쿼리라도 지문 한 번으로 똑같은 게 있는지 즉시 찾아냅니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 소프트 파싱 (Soft) vs 하드 파싱 (Hard)

| 비교 항목 | 소프트 파싱 (Soft Parsing) | 하드 파싱 (Hard Parsing) |
|:---:|:---|:---|
| **공유 풀 활용** | **기존 실행 계획 재사용 (Hit)** | **새로운 실행 계획 생성 (Miss)** |
| **CPU 소모** | 매우 낮음 | 매우 높음 (병목의 원인) |
| **수행 단계** | 문법 체크 -> 해시 매칭 -> 실행 | 문법 -> 시맨틱 -> 최적화 -> 계획생성 -> 실행 |
| **권장 사항** | **바인드 변수 사용 필수** | 지양해야 할 패턴 |

### 2. 과목 융합 관점: OS 및 시스템 최적화
- **운영체제(OS) 융합**: 공유 풀 내의 자원을 관리하기 위해 사용되는 **래치(Latch)**는 OS의 **스핀락(Spinlock)**과 유사한 초경량 동기화 객체다.
- **컴파일러 융합**: 라이브러리 캐시의 동작은 컴파일러의 **중간 코드 생성 및 최적화** 단계를 런타임 캐싱하는 고도의 인터프리터 전략과 닮아 있다.

- **📢 섹션 요약 비유**: 소프트 파싱은 '아는 길을 내비게이션 없이 가는 것'이고, 하드 파싱은 '매번 지도를 새로 그리며 가는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 공유 풀 단편화 (Fragmentation) 발생
- **상황**: SQL은 많은데 Shared Pool 공간이 부족하여 `ORA-04031` 에러 발생.
- **판단**: 바인드 변수를 안 써서 수만 개의 유사한 SQL이 공유 풀을 조각조각 차지하고 있음.
- **결단**: **바인드 변수 적용**으로 쿼리 통합. 일시적으로는 **`ALTER SYSTEM FLUSH SHARED_POOL`**을 수행하여 메모리를 정리하고 큰 덩어리의 연속 공간 확보.

### 2. 안티패턴: 리터럴 SQL (Literal SQL)의 남발
- **증상**: `SELECT * FROM T WHERE ID = 1`, `ID = 2` ... 등 값을 직접 넣어 쿼리 실행.
- **결과**: 라이브러리 캐시 미스 폭증, 라이브러리 캐시 래치 경합으로 시스템 전체 행(Hang) 상태 유발.
- **조치**: 반드시 **바인드 변수(`:id`)**를 사용하여 쿼리 원형을 하나로 유지.

### 📢 기술사적 결언
> "공유 풀은 데이터베이스의 **'지능형 필터'**다. 똑똑한 아키텍트는 공유 풀의 크기를 무작정 키우기보다, 그 안에서 **재사용(Reuse)**이 얼마나 일어나는지를 먼저 살핀다. 기술사는 바인드 변수 사용을 개발 표준으로 강제하여 하드 파싱을 원천 차단해야 하며, **Latch와 Lock**의 대기 이벤트를 분석하여 시스템의 동시성 한계를 예측할 수 있어야 한다. 결국 DB 튜닝의 절반은 공유 풀을 평온하게 유지하는 데서 시작된다."

- **📢 섹션 요약 비유**: 공유 풀 단편화는 '책장은 넓은데 낱장 종이들이 가득 차서 정작 두꺼운 책 한 권을 꽂을 자리가 없는 상태'와 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (ROI)

| 지표 | 최적화 전 | 최적화 후 (Shared Pool 활성) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **CPU Load** | 90% 이상 | **20~30%** | 서버 자원 4배 이상 여유 확보 |
| **SQL 응답 시간** | 500ms | **10ms 이하** | 사용자 체감 속도 50배 향상 |
| **동시 세션 수** | 100개 (경합 심함) | **1,000개 이상** | 시스템 수용량 10배 확장 |

### 2. 미래 전망
- **Automatic Memory Management (AMM)**: 수동 설정 대신 워크로드에 따라 공유 풀과 버퍼 캐시의 크기를 AI가 실시간 조정하는 기술이 고도화됨.
- **Cloud-optimized Plan Sharing**: 같은 클라우드 테넌트 내의 여러 인스턴스가 검증된 실행 계획을 중앙 서버에서 공유하는 **Global Plan Baseline** 서비스 등장.

- **📢 섹션 요약 비유**: 미래의 공유 풀은 단순한 캐시를 넘어, 전 세계 쿼리 데이터를 학습한 '슈퍼 브레인'으로 진화할 것입니다.
