# 59. 영구 저장소 (Persistent Storage) - 데이터 파일, 로그 파일, 제어 파일

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 인스턴스가 종료되거나 전원이 차단되어도 데이터가 소실되지 않도록 보조기억장치(HDD/SSD)에 영구적으로 보관되는 **물리적 파일들의 집합체**다.
> 2. **3대 핵심 파일**: 실제 데이터가 담긴 **데이터 파일(Data Files)**, 복구를 위한 변경 이력이 담긴 **로그 파일(Redo Log Files)**, 그리고 데이터베이스의 전체 구조 정보가 담긴 **제어 파일(Control Files)**로 구성된다.
> 3. **가치**: 데이터베이스의 '실체'이자 '영혼'이며, 이들 파일의 무결성과 가용성을 확보하는 것이 DBMS 운영의 궁극적 목표이자 재난 복구(DR) 설계의 출발점이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**영구 저장소 (Persistent Storage)**는 메모리(SGA)에 상주하는 휘발성 데이터와 대비되는 개념으로, 디스크에 기록되어 영구적으로 보존되는 물리적 계층을 의미합니다. DBMS 엔진은 이 파일들을 유기적으로 결합하여 하나의 논리적 데이터베이스로 인식합니다.

### 2. 💡 비유: 금고와 장부, 그리고 열쇠 꾸러미
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        영구 저장소 3대 파일 비유                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 데이터 파일 (금고 안의 보물): 우리가 진짜로 보관하려는 물건들.               │
│  2. 로그 파일 (입출고 기록 대장): 누가 언제 보물을 넣고 뺐는지 적어둔 공책.        │
│  3. 제어 파일 (금고 설계도 및 열쇠): 금고가 어디에 있고, 대장이 몇 권인지 적힌 쪽지.│
│                                                                             │
│  → 금고(데이터)만 있어선 안 되고, 기록(로그)과 설계도(제어)가 다 있어야            │
│     완벽한 보안 시스템(데이터베이스)이 완성됨!                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **비휘발성 보장**: RAM은 빠르지만 비싸고 전원이 꺼지면 지워짐. 대량의 데이터를 안전하게 저장할 장소 필요.
- **물리적 관리의 필요성**: 수 TB급의 데이터를 관리하기 위해 OS의 파일 시스템 규칙을 따르면서도 DBMS 고유의 효율적 배치가 요구됨.

- **📢 섹션 요약 비유**: 영구 저장소는 데이터베이스라는 성(Castle)을 지탱하는 '단단한 암반'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 영구 저장소 구성 파일 상세 분석

| 파일 종류 | 명칭 | 상세 역할 | 중요도 |
|:---:|:---|:---|:---|
| **Data Files** | 데이터 파일 | 사용자 테이블, 인덱스, 뷰 등 실제 데이터 블록이 저장되는 곳. | 최상 |
| **Redo Log Files** | 리두 로그 파일 | 트랜잭션의 모든 변경 사항을 순차 기록. 장애 시 복구(Redo)에 사용. | 최상 |
| **Control Files** | 제어 파일 | DB 이름, 생성 시간, 다른 파일들의 경로 정보 등을 담은 메타 파일. | 최상 |
| **Parameter File** | 파라미터 파일 | 인스턴스 기동 시 메모리 크기 등 설정 정보를 담은 파일. | 상 |
| **Archive Log** | 아카이브 로그 | 꽉 찬 리두 로그를 영구 보관용으로 복사해둔 파일. (타임머신 복구용) | 상 |

### 2. 물리적 파일 레이아웃 도해 (ASCII)

하나의 데이터베이스를 구성하는 파일들이 디스크 상에 어떻게 나뉘어 존재하는지 시각화한다.

```text
    [ Logical Database: PROD ]
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │  [ Control File ] : "데이터 파일은 A, B에 있고 로그는 C에 있다..."  │
    └──────┬──────────────────────────┬───────────────────────────────┘
           │                          │
    ┌──────▼──────┐            ┌──────▼────────┐            ┌────────▼──────┐
    │ [ Data File ]│            │ [ Redo Log ]  │            │ [ Archive Log ]│
    │  - Table A   │            │  - Log Group 1│            │  - Log #101    │
    │  - Index B   │            │  - Log Group 2│            │  - Log #102    │
    └──────┬───────┘            └──────┬────────┘            └────────┬───────┘
           │                           │                              │
    ┌──────▼───────────────────────────▼──────────────────────────────▼─────┐
    │  Persistent Storage (SSD / HDD / SAN / Cloud Block Storage)           │
    └───────────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: 제어 파일(Control File)의 역할
- **기동 단계**: 인스턴스 시동 시 `MOUNT` 단계에서 제어 파일을 읽어 데이터 파일과 로그 파일의 위치를 파악함.
- **동기화 확인**: 데이터 파일 헤더의 체크포인트 정보와 제어 파일의 정보를 비교하여 DB의 일관성 여부를 판단. 일치하지 않으면 즉시 복구(Recovery) 모드 진입.

### 4. 핵심 기술: 스트라이핑과 미러링 (RAID)
- **성능 (RAID 0)**: 데이터를 여러 파일에 쪼개어 동시 기록 (I/O 분산).
- **안정성 (RAID 1)**: 동일한 파일을 다른 디스크에 복제 (Mirroring).
- **조합 (RAID 1+0)**: 성능과 안정성을 모두 챙기는 영구 저장소의 표준 구성.

- **📢 섹션 요약 비유**: 제어 파일은 '가족 관계 증명서'와 같아서, 뿔뿔이 흩어진 파일들을 하나의 '데이터베이스 가족'으로 묶어주는 증거가 됩니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 데이터 파일(Data) vs 리두 로그(Log) 비교

| 비교 항목 | 데이터 파일 (Data File) | 리두 로그 파일 (Redo Log) |
|:---:|:---|:---|
| **쓰기 방식** | Random Write (페이지 단위) | **Sequential Write (순차 기록)** |
| **기록 시점** | 체크포인트 또는 버퍼 부족 시 | **커밋 시점 (반드시 먼저)** |
| **주요 목적** | 현재 시점의 데이터 보관 | **과거 변경 이력 및 복구 보장** |
| **물리적 성격** | 크기가 계속 커짐 (확장) | 순환 구조 (Overwrite 가능) |

### 2. 과목 융합 관점: OS 및 스토리지 가상화
- **운영체제(OS) 융합**: 데이터 파일은 OS의 **파일 시스템(XFS, EXT4 등)** 위에 존재하지만, 성능을 위해 OS 캐시를 우회하는 **Direct I/O**를 주로 사용한다.
- **클라우드 융합**: 현대의 영구 저장소는 물리 디스크를 넘어 클라우드의 **Block Storage(AWS EBS)**나 **Object Storage(S3)**로 확장되며, 네트워크 지연을 극복하기 위한 복제 기법이 필수적이다.

- **📢 섹션 요약 비유**: 데이터 파일이 '완성된 결과물'이라면, 로그 파일은 '그 결과물을 만들기까지의 모든 기록'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 디스크 I/O 분산 설계
- **상황**: 서비스 부하가 높은데 디스크 응답 시간이 너무 느림.
- **판단**: 데이터 파일과 로그 파일을 같은 디스크에 두어 I/O 경합(Contention) 발생.
- **결단**: **데이터 파일은 RAID 5/10 구성의 대용량 SSD로, 리두 로그 파일은 별도의 독립된 고속 채널로 물리적 분리**. 로그의 순차 쓰기 성능을 보장하여 시스템 전체의 Commit 속도를 개선함.

### 2. 안티패턴: 제어 파일(Control File)의 단일 구성
- **증상**: 제어 파일을 하나만 만들어둠.
- **결과**: 디스크 한 장이 고장 나자 DB 위치 정보를 잃어버려, 데이터가 멀쩡해도 DB를 켤 수 없는 상태 발생.
- **조치**: 제어 파일은 반드시 서로 다른 물리 디스크에 **다중화(Multiplexing)**하여 3개 이상 복사본을 유지해야 함.

### 📢 기술사적 결언
> "데이터베이스의 물리적 설계는 **'최악의 상황을 가정하는 것'**에서 시작한다. 메모리는 꺼질 수 있고, 디스크는 깨질 수 있다. 기술사는 영구 저장소의 3대 파일을 단순히 관리하는 것을 넘어, 이들이 장애 상황에서 어떻게 상호작용하여 **RTO(복구 시간)와 RPO(복구 시점)**를 만족시키는지 그 '물리적 정합성'을 설계해야 한다. 특히 현대의 SSD/NVMe 환경에 맞는 **블록 정렬(Alignment)**과 클라우드 스토리지의 특성을 고려한 아키텍처 수립이 필수적이다."

- **📢 섹션 요약 비유**: 제어 파일을 다중화하는 것은 '집 열쇠를 가족 모두가 하나씩 나누어 갖는 것'과 같습니다. 한 명이 잃어버려도 집에 들어갈 수 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (물리 설계 최적화 시)

| 지표 | 도입 전 | 도입 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **장애 시 데이터 유실** | 상존 | **0% (영속성 확보)** | 비즈니스 신뢰도 100% |
| **시스템 기동 성공률** | 불안정 | **안정적 (제어 파일 다중화)** | 인프라 운영 가용성 향상 |
| **쓰기 지연 시간** | 100ms 이상 | **10ms 이하** | 트랜잭션 처리 성능 10배 향상 |

### 2. 미래 전망
- **Software-Defined Storage (SDS)**: 하드웨어에 종속되지 않고 소프트웨어 계층에서 가상화된 영구 저장소를 관리하여 유연성 극대화.
- **All-Flash DB**: 모든 영구 저장소를 초고속 NVMe SSD로 구성하여, 디스크 헤드의 물리적 제약이 완전히 사라진 차세대 고성능 DB 아키텍처가 보편화될 것임.

- **📢 섹션 요약 비유**: 미래의 저장소는 '절대 깨지지 않는 투명한 다이아몬드'처럼, 속도와 안전함을 동시에 완벽하게 갖춘 형태가 될 것입니다.

- **로그 매니저 (Log Manager)**: 로그 파일을 다루는 프로세스.

- **체크포인트 (Checkpoint)**: 메모리와 저장소의 상태를 맞추는 이벤트.
- **RAID 기술**: 저장소의 안정성을 높이는 하부 기술.
