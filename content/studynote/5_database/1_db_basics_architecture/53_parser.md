# 53. 파서 (Parser) - SQL 구문 분석 및 파스 트리 생성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사용자가 입력한 텍스트 형태의 SQL 문장을 DBMS가 이해할 수 있는 내부 구조인 **파스 트리(Parse Tree)**로 변환하는 첫 번째 질의 처리 관문이다.
> 2. **검증**: 어휘 분석(Lexical Analysis)을 통해 단어의 유효성을 검사하고, 구문 분석(Syntactic Analysis)을 통해 문법적 오류를 찾아내어 시스템의 안정성을 일차적으로 방어한다.
> 3. **가치**: 인간의 언어에 가까운 선언적 SQL을 컴퓨터가 처리하기 쉬운 논리적 트리 구조로 정형화함으로써, 후속 단계인 옵티마이저가 최적화 전략을 수립할 수 있는 기반 데이터를 제공한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**파서 (Parser)**는 SQL 질의문을 분석하여 토큰(Token) 단위로 분리하고, 이들 간의 관계를 트리 구조로 재구성하는 모듈입니다. 컴파일러의 전단부(Front-end) 역할과 유사합니다.

### 2. 💡 비유: 외국어 문장 성분 분석
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        파서(Parser) 비유                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 입력 문장: "나는 학교에 간다."                                              │
│  2. 파서의 작업:                                                              │
│     - 단어 쪼개기: "나", "는", "학교", "에", "간다" (Lexical)                   │
│     - 역할 부여: 주어="나", 목적지="학교", 서술어="간다" (Syntactic)             │
│     - 문법 검사: "간다 학교에 나는" 처럼 순서가 틀리면 에러 발생!                 │
│                                                                             │
│  → 엉망진창인 말을 똑바로 정리해서, **'누가 무엇을 하는지'**를 주방장(DB 엔진)이    │
│     알아듣게 번역해주는 통역사와 같음!                                           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **SQL의 유연성**: SQL은 대소문자 구분이나 공백 처리가 비교적 자유로워, 이를 엄격한 컴퓨터 언어로 변환하는 정교한 분석기 필요.
- **오류 조기 발견**: 잘못된 쿼리가 엔진 깊숙이 들어가 시스템 자원을 낭비하기 전, 문법 단계에서 즉시 차단하기 위함.

- **📢 섹션 요약 비유**: 파서는 데이터베이스 입구에서 신분증(단어)과 복장(문법)을 검사하는 '깐깐한 안내원'과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 파싱의 2단계 핵심 공정

| 단계 | 명칭 | 수행 내용 | 비유 |
|:---:|:---|:---|:---|
| **1단계** | **어휘 분석 (Lexical)** | 텍스트를 최소 의미 단위인 **토큰(Token)**으로 분리. (SELECT, *, FROM 등) | 알파벳 나열에서 단어 찾기 |
| **2단계** | **구문 분석 (Syntactic)** | 토큰들을 문법 규칙(Grammar)에 맞춰 **파스 트리**로 조립. | 단어로 문장 만들기 |

### 2. 파싱 및 트리 생성 아키텍처 (ASCII)

입력된 SQL이 내부 트리 구조로 변환되는 과정을 시각화한다.

```text
    [ SQL String ] : "SELECT Name FROM Student WHERE ID = 10"
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 1. [ Lexical Analyzer (Scanner) ]                               │
    │    - Tokens: [SELECT], [Name], [FROM], [Student], [WHERE]...    │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 2. [ Syntax Analyzer (Parser) ]                                 │
    │    - Check against SQL Grammar (ISO/ANSI Standard)              │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    ┌──────▼─────── [ Parse Tree ] ───────────────────────────────────┐
    │                                                                 │
    │              < Query >                                          │
    │             /    |    \                                         │
    │      <Select> <From>  <Where>                                   │
    │         |       |        |                                      │
    │       "Name" "Student"  <Cond>                                  │
    │                        /  |  \                                  │
    │                      "ID" "=" 10                                │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: 문법(Grammar) 기반 분석
파서는 **BNF (Backus-Naur Form)**와 같은 형식 언어 표기법으로 정의된 SQL 문법 지도를 가지고 있습니다.
1. **재귀 하강 파싱 (Recursive Descent)**: 트리의 루트부터 리프까지 규칙을 따라 내려가며 분석.
2. **시맨틱 분석 (Semantic Analysis)**: 문법은 맞지만 의미가 통하는지 검사. (예: 테이블에 실제 'Name' 컬럼이 있는지, 권한이 있는지 등 - 주로 파서 이후 단계에서 수행)

### 4. 핵심 기술: 파싱 오버헤드와 캐싱 (Soft Parsing)
- **Hard Parsing**: 매번 파서가 전체 과정을 수행하는 것. CPU 소모가 큼.
- **Soft Parsing**: 이미 분석된 트리를 메모리(Shared Pool / Library Cache)에 저장해두고 재사용하는 기술. SQL 바인드 변수 사용이 권장되는 이유.

- **📢 섹션 요약 비유**: 소프트 파싱은 '매번 요리책을 보며 분석하는 대신, 자주 시키는 메뉴의 레시피를 외워두는 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 파서(Parser) vs 옵티마이저(Optimizer)

| 비교 항목 | 파서 (Parser) | 옵티마이저 (Optimizer) |
|:---:|:---|:---|
| **역할** | **이해 및 구조화** | **전략 수립 및 최적화** |
| **핵심 질문** | "이 문장이 올바른가?" | "이 데이터를 어떻게 빨리 가져올까?" |
| **산출물** | 파스 트리 (Parse Tree) | 실행 계획 (Execution Plan) |
| **수행 시점** | 질의 처리의 최전방 | 파싱 및 시맨틱 체크 이후 |

### 2. 과목 융합 관점: 오토마타 및 보안
- **오토마타 융합**: 파서의 렉서(Lexer)는 **유한 오토마타(Finite Automata)**를 기반으로 동작하며, 구문 분석기는 **푸시다운 오토마타(Pushdown Automata)**의 원리를 따른다.
- **보안(Security) 융합**: 파서는 **SQL Injection** 공격을 방어하는 일차적 방어선이다. 악의적인 특수문자나 비정상적인 구문을 필터링하고 차단하는 역할을 수행한다.

- **📢 섹션 요약 비유**: 파서가 '법전(문법)에 따라 죄를 묻는 판사'라면, 옵티마이저는 '최소 비용으로 형량을 집행하려는 행정가'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 바인드 변수(Bind Variable) 미사용에 따른 장애
- **상황**: `WHERE ID = 1`, `WHERE ID = 2` 처럼 값만 다른 쿼리가 초당 수만 건 들어옴.
- **판단**: 파서는 이를 매번 다른 쿼리로 인식하여 **Hard Parsing**을 수행, CPU 점유율이 100%에 도달함.
- **결단**: **바인드 변수(`WHERE ID = ?`)를 적용**. 파서가 쿼리 원형을 한 번만 분석하고 트리를 재사용하게 하여(Soft Parsing) CPU 부하를 즉각적으로 해결함.

### 2. 안티패턴: 너무 거대한 쿼리 (God Query)
- **증상**: 수천 줄에 달하는 SQL, 수십 개의 서브쿼리와 Union이 얽힌 쿼리.
- **결과**: 파서가 트리를 구성하는 데만 수 초가 걸리며, 메모리(Stack) 부족으로 분석 실패 가능성 발생.
- **조치**: 쿼리를 **모듈화(CTE 사용)**하거나, 비즈니스 로직을 쪼개어 파싱 부하를 분산.

### 📢 기술사적 결언
> "파서는 데이터베이스의 **'입'**이다. 입구에서 깨끗한 음식을 잘 씹어 넘겨야(Parsing) 위장(Engine)이 고생하지 않는다. 기술사는 단순히 SQL 문법 오류를 잡는 것을 넘어, **라이브러리 캐시(Library Cache)의 효율성**과 **하드 파싱의 비용**을 고려하여 애플리케이션의 쿼리 표준을 정립해야 한다. 특히 현대의 클라우드 환경에서는 파싱 비용조차 비용(Cost)이므로, 정교한 쿼리 설계가 곧 인프라 비용 절감으로 이어진다."

- **📢 섹션 요약 비유**: 바인드 변수를 안 쓰는 것은 '똑같은 노래를 매번 새로 작곡해서 부르는 것'과 같이 비효율적인 예술 행위입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (ROI)

| 지표 | 파서 최적화 전 (Hard) | 파서 최적화 후 (Soft) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **CPU 사용량** | 80% 이상 | **10% 미만** | 연산 자원 8배 절감 |
| **쿼리 응답 지연** | 수십 ms (파싱 지연) | **수 ms 이내** | 실시간성 확보 |
| **동시성 처리** | Latch 경합 발생 | **부드러운 처리** | 시스템 확장성 증대 |

### 2. 미래 전망
- **Schema-less Parsing**: JSON이나 XML 등 정해진 스키마가 없는 데이터를 실시간으로 파싱하여 정형화하는 **Lazy Parsing** 기술이 NoSQL과 Lakehouse에서 진화 중임.
- **Natural Language Parsing**: 자연어로 된 질문을 관계 대수 트리로 직접 변환하는 **LLM 기반 파서**가 차세대 인터페이스의 핵심이 될 전망임.

- **📢 섹션 요약 비유**: 미래의 파서는 우리가 '대충' 말해도 '찰떡같이' 트리로 만들어주는 지능형 통역가가 될 것입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **옵티마이저 (Optimizer)**: 파서의 결과물을 받는 직속 후속 공정.
- **시스템 카탈로그**: 파서가 테이블과 컬럼의 실체를 확인하는 장부.
- **Shared Pool / Library Cache**: 파싱된 결과물을 보관하는 메모리 공간.
- **바인드 변수 (Bind Variable)**: 소프트 파싱을 가능케 하는 핵심 기법.

### 👶 어린이를 위한 3줄 비유 설명
1. **파서**는 컴퓨터가 SQL이라는 어려운 문장을 읽을 때, **단어 하나하나를 쪼개서 이해하는 첫 번째 단계**예요.
2. "이 문법이 맞나?", "이 단어가 사전에 있나?"를 꼼꼼하게 따져서 **정리된 지도(파스 트리)**를 만들죠.
3. 이 지도가 잘 만들어져야 나중에 컴퓨터가 길을 잃지 않고 **우리가 원하는 데이터를 빨리** 찾아줄 수 있답니다!
