# 29. 파일 시스템 (File System)의 문제점: 종속성과 중복성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파일 시스템은 데이터를 별도의 파일 단위로 관리하며, 응용 프로그램이 파일의 물리적 구조에 직접 의존하는 **데이터 종속성**과 동일 정보가 여러 곳에 산재하는 **데이터 중복성**을 태생적으로 가진다.
> 2. **한계**: 중복 데이터 간 불일치로 인한 **무결성 파괴**, 파일 구조 변경 시 프로그램 전체를 수정해야 하는 **유지보수 재앙**, 동시 접근 제어 불가로 인한 데이터 오염 문제를 야기한다.
> 3. **가치**: 이러한 파일 시스템의 치명적 결함은 데이터와 프로그램을 분리하고 통합 관리하는 **DBMS(Database Management System) 탄생의 직접적인 배경**이 되었다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**파일 시스템 (File System)**은 운영체제(OS) 수준에서 데이터를 보조기억장치에 파일 형태로 저장하고 관리하는 방식입니다. DBMS가 등장하기 전, 모든 전산 시스템은 각 응용 프로그램이 자신만의 데이터 파일을 별도로 정의하고 관리하는 파일 시스템 기반이었습니다.

### 2. 💡 비유: 개인별 수첩 vs 공유 화이트보드
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        파일 시스템 vs 데이터베이스 비유                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 회사 직원들의 연락처를 관리함.                                          │
│                                                                             │
│  1. 파일 시스템 (개인 수첩):                                                  │
│     - 영업부도 수첩이 있고, 인사부도 수첩이 있음.                                 │
│     - 직원이 이사 가면 모든 부서 수첩을 일일이 고쳐야 함. (중복/불일치)             │
│     - 수첩 양식이 바뀌면(글자 크기 등) 쓰는 법을 다시 배워야 함. (종속성)           │
│                                                                             │
│  2. 데이터베이스 (공유 화이트보드):                                             │
│     - 중앙에 딱 하나만 적어둠. 누구나 와서 보고, 한 명만 고치면 됨.                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경 (DBMS가 왜 필요한가?)
- **과거 방식**: 1960년대 초반, 응용 프로그램마다 필요한 데이터를 파일로 직접 설계.
- **문제 발생**: 기업 규모가 커지면서 동일한 데이터(예: 고객 주소)가 수십 개의 파일에 중복 저장되어 정합성이 깨지기 시작함.
- **혁신적 전환**: 데이터의 논리적·물리적 구조를 응용 프로그램으로부터 독립시키기 위해 DBMS 패러다임 도입.

- **📢 섹션 요약 비유**: 파일 시스템은 각자 자기 방만 정리하는 '사일로(Silo)' 방식이며, 이로 인해 집 전체의 물건 관리가 엉망이 되는 구조적 한계를 가집니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 파일 시스템의 2대 핵심 문제점 상세 분석

| 문제점 | 상세 설명 | 파생되는 부작용 |
|:---:|:---|:---|
| **데이터 종속성 (Dependency)** | 응용 프로그램과 데이터 파일의 물리적 구조가 밀접하게 결합된 상태. | 파일 구조(예: 필드 추가) 변경 시 **프로그램 코드도 수정**해야 함. |
| **데이터 중복성 (Redundancy)** | 동일한 데이터가 서로 다른 파일에 중복되어 저장되는 현상. | **일관성 결여, 무결성 파괴**, 저장 공간 낭비, 경제성 저하. |

### 2. 파일 시스템의 데이터 관리 구조 (ASCII)

응용 프로그램과 데이터가 '1:1'로 결합되어 있어 유연성이 전혀 없는 상태를 시각화한다.

```text
    [ 응용 프로그램 계층 ]           [ 데이터 계층 (Files) ]
    ┌──────────────────┐           ┌──────────────────────┐
    │  급여 관리 프로그램  │ ◀──────▶ │  사원 급여 파일.dat   │ (물리 구조 의존)
    └──────────────────┘           └──────────────────────┘
                                              ▲
    ┌──────────────────┐           ┌──────────┴───────────┐
    │  인사 관리 프로그램  │ ◀──────▶ │  사원 기본 정보.txt   │ (동일 사원 정보 중복)
    └──────────────────┘           └──────────────────────┘
                                              ▼
    ┌──────────────────┐           ┌──────────────────────┐
    │  주소록 관리 앱    │ ◀──────▶ │  고객 연락처 파일.csv │ (프로그램 수정 시 파일도 수정)
    └──────────────────┘           └──────────────────────┘
```

### 3. 심층 동작 원리: 왜 '종속성'이 무서운가?
1. **물리적 구조 변경**: 파일 시스템에서 전화번호 필드를 10자리에서 11자리로 늘림.
2. **프로그램 파손**: 해당 파일을 읽는 모든 프로그램의 `read()` 로직이 오동작하거나 에러 발생.
3. **연쇄 수정**: 파일 하나를 고치기 위해 수천 줄의 코드를 분석하고 다시 컴파일해야 하는 '경직성' 발생.

### 4. 기타 주요 문제점
- **무결성 유지 어려움**: 데이터가 산재해 있어 제약 조건(예: 나이는 양수)을 일관되게 적용 불가.
- **동시 공유 불가능**: 한 프로그램이 파일을 열면 다른 프로그램은 접근 불가 (Concurrency 제어 부재).
- **보안성 취약**: 파일 단위의 권한 제어는 가능하나, 레코드나 필드 단위의 정교한 보안은 불가능.

- **📢 섹션 요약 비유**: 파일 시스템은 '톱니바퀴가 직접 맞물린 기계'와 같습니다. 톱니 하나만 크기가 변해도 전체 기계가 멈춰버리는 치명적 약점이 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 파일 시스템 vs 데이터베이스 시스템 (DBMS)

| 비교 항목 | 파일 시스템 (File System) | 데이터베이스 시스템 (DBMS) |
|:---:|:---|:---|
| **데이터 독립성** | 없음 (종속적) | **논리적·물리적 독립성 제공** |
| **데이터 중복** | 많음 (중복 저장이 기본) | **최소화 (통합 관리)** |
| **데이터 정합성** | 낮음 (일관성 유지 힘듦) | **매우 높음 (무결성 제약조건)** |
| **보안 및 권한** | 파일 단위 (단순함) | **필드/로우 단위 (정교함)** |
| **장애 복구** | 사용자 책임 (수동 백업) | **DBMS 책임 (로그/자동 복구)** |

### 2. 과목 융합 관점: 운영체제 및 소프트웨어 공학
- **운영체제(OS) 융합**: 파일 시스템은 OS의 핵심 모듈이지만, DBMS는 OS 위에서 동작하며 **버퍼 관리, 스케줄링** 등을 직접 수행하여 OS의 한계를 극복한다.
- **소프트웨어 공학(SE) 융합**: 파일 시스템은 **유지보수 비용(Technical Debt)**을 폭증시키는 주범이며, DBMS 도입은 시스템의 **응집도를 높이고 결합도를 낮추는** 최고의 리팩토링 전략이다.

- **📢 섹션 요약 비유**: 파일 시스템이 '각자 도생'하는 골목 시장이라면, DBMS는 모든 거래가 전산화되고 관리되는 '거대 쇼핑몰'과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 로그 파일의 직접 처리 위험성
- **상황**: 서비스 로그를 텍스트 파일로 쌓아두고, 직접 파이썬 스크립트로 통계를 뽑고 있음.
- **판단**: 데이터가 늘어날수록 탐색 속도가 느려지고, 로그 포맷 변경 시 스크립트가 모두 깨짐 (종속성).
- **결단**: **Elasticsearch나 RDBMS로 로그를 적재(Ingestion)**하여 데이터 독립성을 확보하고 표준화된 질의(Query) 수행.

### 2. 안티패턴: DBMS 내부의 파일 경로 저장
- **증상**: 이미지나 문서를 DB에 넣지 않고, 서버 파일 시스템 경로(Path)만 DB에 저장함.
- **결과**: 서버 이전 시 파일 경로가 깨지면 DB 데이터는 가비지(Garbage)가 됨. DB 백업과 파일 백업의 시점이 달라 데이터 불일치 발생.
- **조치**: 파일은 **Object Storage(S3 등)**에 보관하고 DB에는 식별자(ID)를 저장하거나, 정합성이 극도로 중요하다면 **BLOB(Binary Large Object)** 타입을 사용하여 DB 내에 통합 관리.

### 📢 기술사적 결언
> "파일 시스템은 데이터 관리의 '원시적 형태'이며, 그 한계인 **중복성과 종속성**은 정보 공학이 해결해야 할 첫 번째 과제였다. 현대 아키텍트는 비록 NoSQL이나 데이터 레이크 등 분산 환경을 다루더라도, 그 근저에는 **'데이터 무결성 확보'**라는 DBMS의 탄생 철학이 흐르고 있음을 잊지 말아야 한다. 편리함(파일 처리)을 위해 원칙(통합 관리)을 버리는 순간, 시스템은 다시금 파일 시스템의 혼돈으로 회귀할 것이다."

- **📢 섹션 요약 비유**: 안티패턴인 '경로 저장'은 보물지도는 가졌으나 보물 상자가 어디로 이사 갔는지 모르는 것과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (DBMS 도입 시)

| 지표 | 파일 시스템 대비 개선 효과 | 비즈니스 가치 |
|:---|:---:|:---|
| **데이터 정합성** | 불일치율 99% 감소 | 서비스 신뢰도 및 브랜드 가치 상승 |
| **유지보수 생산성** | 코드 수정량 70% 이상 절감 | 기술 부채 해소 및 개발 속도 향상 |
| **데이터 보안성** | 접근 제어의 세밀함 증가 | 컴플라이언스(GDPR 등) 준수 용이 |

### 2. 미래 전망
- **객체 저장소와의 융합**: 단순 파일 시스템을 넘어, 무한 확장 가능한 **Object Storage**와 DBMS가 결합된 형태가 클라우드 네이티브의 표준이 됨.
- **데이터 레이크의 도전**: 파일 시스템의 유연성(Schema-on-Read)과 DBMS의 관리성(Acid)을 합친 **Data Lakehouse** 기술이 파일 시스템의 고전적 문제점을 현대적으로 재해석하여 해결하고 있음.

- **📢 섹션 요약 비유**: 과거의 문제는 '중복'이었으나, 미래의 문제는 '파편화'입니다. 이를 하나로 묶는 통제력은 여전히 DBMS의 철학에서 나옵니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **파일 시스템**은 친구들마다 자기 방에 **따로따로 장난감 상자**를 둔 것과 같아요.
2. 똑같은 인형을 친구들마다 다 가지고 있어서 자리가 좁아지고(중복), 상자 모양이 바뀌면 인형을 넣기도 힘들어지죠(종속).
3. 그래서 모든 친구가 **커다란 마법 상자(DBMS)** 하나를 같이 쓰기로 약속했답니다!
