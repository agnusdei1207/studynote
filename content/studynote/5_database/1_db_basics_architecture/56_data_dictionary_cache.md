# 56. 데이터 딕셔너리 캐시 (Data Dictionary Cache)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 카탈로그(데이터 사전)에 저장된 테이블, 컬럼, 권한 등 메타데이터의 정보를 메모리에 상주시켜, 쿼리 파싱 시 발생하는 반복적인 디스크 I/O를 제거하는 **고속 메타데이터 캐시**다.
> 2. **메커니즘**: Oracle 아키텍처에서는 공유 풀(Shared Pool) 내에 존재하며, '행(Row)' 단위로 정보를 저장하여 관리하므로 **로우 캐시(Row Cache)**라고도 불린다.
> 3. **가치**: 모든 SQL 문장의 유효성 검증 및 옵티마이저의 실행 계획 수립에 필수적인 기초 정보를 즉각 제공하여, 시스템 전체의 **질의 처리 지연(Parsing Latency)**을 최소화한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**데이터 딕셔너리 캐시 (Data Dictionary Cache)**는 데이터베이스 스키마 정보(메타데이터)를 보관하는 메모리 영역입니다. 사용자가 테이블명을 입력하면, 해당 테이블이 존재하는지, 어떤 컬럼이 있는지, 사용자가 볼 권한이 있는지 등을 디스크가 아닌 이곳에서 확인합니다.

### 2. 💡 비유: 맛집 요리사의 암기 수첩
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        데이터 딕셔너리 캐시 비유                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 데이터 사전 (대형 요리 백과사전): 주방 깊숙한 창고(디스크)에 있는 두꺼운 책.    │
│  2. 딕셔너리 캐시 (요리사의 머릿속): 자주 만드는 메뉴의 재료와 위치를 외워둠.      │
│                                                                             │
│  - 손님이 "김치찌개(테이블)"를 주문함.                                         │
│  - 요리사가 창고까지 달려가서 백과사전을 뒤지지 않고,                          │
│  - **'머릿속(캐시)'**에서 "김치는 3번 냉장고에 있고, 소금은 1번 서랍에 있다"는      │
│    정보를 즉시 꺼내어 요리를 시작함!                                           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **파싱의 병목**: 하나의 SQL을 실행하려면 수십 번의 메타데이터 조회가 필요함. 이를 매번 디스크에서 읽으면 쿼리 실행보다 파싱이 더 오래 걸림.
- **공유의 필요성**: 여러 사용자가 동일한 테이블을 조회할 때, 공통된 스키마 정보를 한곳에 모아두어 자원 낭비를 방지해야 함.

- **📢 섹션 요약 비유**: 딕셔너리 캐시는 '자주 묻는 질문(FAQ)을 모아둔 고객센터 상담원의 메모'와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 딕셔너리 캐시의 주요 저장 정보

| 항목 | 상세 내용 | 비유 |
|:---:|:---|:---|
| **정의 정보** | 테이블명, 컬럼 정의, 데이터 타입, 제약조건. | 제품 카탈로그 |
| **권한 정보** | 어떤 사용자가 어떤 테이블에 접근 가능한지(GRANT 정보). | 보안 출입 명단 |
| **물리 정보** | 테이블 데이터가 저장된 데이터 파일의 위치와 범위(Extent). | 창고 주소록 |
| **기타 정보** | 뷰(View) 정의, 시퀀스(Sequence) 현재 값, 시노님 정보. | 단축번호 목록 |

### 2. 메모리 내 위치 및 구조 (ASCII)

Oracle의 SGA(System Global Area) 내 Shared Pool 구조를 중심으로 시각화한다.

```text
    [ System Global Area (SGA) ]
    ┌──────────────────────────────────────────────────────────────────┐
    │  [ Shared Pool ]                                                 │
    │  ┌────────────────────────┐  ┌────────────────────────────────┐  │
    │  │  Library Cache         │  │  Data Dictionary Cache         │  │
    │  │  (SQL 실행 계획 저장)   │  │  (Schema Metadata 저장)        │  │
    │  │                        │  │  - Table Definitions           │  │
    │  │                        │  │  - User Privileges             │  │
    │  └────────────────────────┘  │  - Column Info                 │  │
    │                              └────────────────┬───────────────┘  │
    └───────────────────────────────────────────────┼──────────────────┘
                                                    │ (Miss 발생 시)
    ┌───────────────────────────────────────────────▼──────────────────┐
    │  [ Persistent Storage ]                                          │
    │  - System Data Files (Tablespace: SYSTEM)                        │
    │  - System Catalog Tables (TAB$, COL$, etc.)                      │
    └──────────────────────────────────────────────────────────────────┘
```

### 3. 심층 동작 원리: 로우 캐시(Row Cache)인 이유
- **구조적 차이**: 일반적인 데이터 버퍼 캐시는 '블록(Block)' 단위로 데이터를 올리지만, 딕셔너리 캐시는 메타데이터의 **'행(Row)'** 단위로 정보를 관리함.
- **효율성**: 필요한 특정 컬럼의 정의 정보나 권한 정보 한 줄만 쏙 뽑아서 관리할 수 있어 메모리 효율이 극대화됨.

### 4. 핵심 기술: 캐시 히트(Hit)와 미스(Miss)
- **Cache Hit**: 파서가 요청한 정보가 메모리에 있어 즉시 반환.
- **Cache Miss**: 디스크의 SYSTEM 테이블스페이스를 읽어 메모리에 적재 후 반환 (성능 저하의 주범).

- **📢 섹션 요약 비유**: 딕셔너리 캐시는 '전화번호부 전체를 가져오는 게 아니라, 필요한 사람의 번호 하나만 쪽지에 적어둔 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 딕셔너리 캐시 vs 라이브러리 캐시 (Shared Pool 내 비교)

| 비교 항목 | 데이터 딕셔너리 캐시 | 라이브러리 캐시 (Library Cache) |
|:---:|:---|:---|
| **저장 대상** | **객체의 정의 (Metadata)** | **SQL 문장 및 실행 계획 (Plan)** |
| **관리 단위** | 로우 (Row) | 오브젝트 (LCO) |
| **역할** | "테이블이 있는가? 권한이 있는가?" | "이 쿼리를 어떻게 실행할 것인가?" |
| **의존 관계** | 라이브러리 캐시가 이를 참조함 | 딕셔너리 캐시 정보 기반으로 생성됨 |

### 2. 과목 융합 관점: OS 및 보안
- **운영체제(OS) 융합**: OS의 **아이노드 캐시(Inode Cache)**와 유사한 역할을 수행한다. 파일의 실제 내용이 아닌 파일의 속성과 위치를 빠르게 찾는 메커니즘을 공유한다.
- **보안(Security) 융합**: 사용자 권한 정보가 이곳에 캐싱되므로, **권한 탈취 방지 및 세밀한 접근 제어(RBAC)**를 실시간으로 수행하는 물리적 토대가 된다.

- **📢 섹션 요약 비유**: 딕셔너리 캐시가 '재료 설명서'라면, 라이브러리 캐시는 '조리법(레시피)'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 잦은 DDL 작업으로 인한 성능 저하
- **상황**: 운영 중에 `ALTER TABLE`이나 `CREATE INDEX`를 수천 번 반복함.
- **판단**: DDL은 카탈로그 정보를 수정하므로 딕셔너리 캐시의 빈번한 **무효화(Invalidation)**와 갱신을 초래함.
- **결단**: **운영 시간 외로 DDL 작업 통합**. 카탈로그 경합(Rowcache Lock)을 줄여 실시간 서비스의 파싱 지연을 방지함.

### 2. 안티패턴: 너무 작은 Shared Pool 설정
- **증상**: `Row Cache Objects` 래치 경합이 발생하고, 딕셔너리 캐시 미스율이 높음.
- **결과**: 모든 쿼리가 실행될 때마다 내부적으로 시스템 테이블을 조회하는 쿼리가 추가로 발생하여 시스템 전반이 느려짐.
- **조치**: **SGA_TARGET 또는 SHARED_POOL_SIZE를 증설**하고, `v$rowcache` 뷰를 통해 히트율을 모니터링하여 최적화.

### 📢 기술사적 결언
> "데이터 딕셔너리 캐시는 데이터베이스의 **'지식 기반'**이다. 지식이 부족한 엔진은 매번 사전을 찾아보느라 벙어리가 된다. 기술사는 쿼리 튜닝 시 실행 계획만 볼 것이 아니라, 그 계획이 만들어지기 전 단계인 **딕셔너리 캐시의 효율성**을 반드시 체크해야 한다. 특히 현대의 복잡한 마이크로서비스 환경에서는 수많은 스키마 객체를 관리해야 하므로, 카탈로그 관리의 정교함이 곧 시스템의 확장성으로 이어진다."

- **📢 섹션 요약 비유**: 딕셔너리 캐시가 부족한 것은 '공부할 때마다 모르는 단어가 나와서 매번 두꺼운 사전을 뒤적이는 수험생'과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (ROI)

| 지표 | 도입 전 (Hard Parsing) | 도입 후 (Soft Parsing) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **파싱 소요 시간** | 100ms 이상 | **1ms 이하** | 100배 이상의 파싱 속도 향상 |
| **시스템 테이블 I/O** | 매우 높음 | **거의 없음** | SYSTEM 테이블스페이스 부하 제거 |
| **동시 사용자 수** | 제한적 (래치 경합) | **대규모 수용 가능** | 동시성 처리 능력 극대화 |

### 2. 미래 전망
- **Persistent Dictionary**: 메모리뿐만 아니라 고속 NVMe에 카탈로그 전용 공간을 두어, 재부팅 후에도 캐시 예열(Warm-up) 없이 즉시 고성능을 내는 기술 발전 중.
- **Cloud-Native Catalog**: 여러 DB 인스턴스가 하나의 거대한 카탈로그 캐시를 공유하는 **Global Metadata Cache** 서비스가 클라우드 벤더에 의해 강화될 전망.

- **📢 섹션 요약 비유**: 미래의 딕셔너리 캐시는 단순한 메모리 조각을 넘어, 전 세계의 모든 데이터 정보를 기억하는 '거대한 공유 뇌'가 될 것입니다.
