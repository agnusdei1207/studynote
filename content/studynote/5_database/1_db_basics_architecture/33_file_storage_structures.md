# 33. 파일 저장 구조 (Heap, Sequential, Hash, Indexed)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 물리적 저장 계층에서 레코드를 보조기억장치의 블록에 배치하고 검색하는 논리적 조직 방식이다.
> 2. **다양성**: 빠른 삽입이 장점인 **Heap**, 순차 처리에 특화된 **Sequential**, 특정 값 검색에 최적화된 **Hash**, 그리고 인덱스를 통해 성능과 유연성을 모두 잡은 **Indexed** 구조로 분류된다.
> 3. **가치**: 데이터의 성격(조회 위주 vs 갱신 위주)에 맞는 최적의 저장 구조를 선택함으로써 디스크 I/O를 최소화하고 질의 응답 속도를 비약적으로 향상시킨다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**파일 저장 구조 (File Organization)**는 레코드들을 파일 내에 물리적으로 배치하는 방법과 이들에 접근하는 메커니즘을 정의합니다. 이는 DBMS의 물리적 설계(Physical Design) 단계에서 가장 중요한 의사결정 요소 중 하나입니다.

### 2. 💡 비유: 도서관의 책 정리법
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        파일 저장 구조 비유                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. Heap (난장판 상자): 책을 그냥 들어오는 대로 상자에 던져 넣음. 찾을 땐 다 뒤져야 함.│
│  2. Sequential (사전식): 가나다순으로 책을 꽂음. 순서대로 읽긴 좋지만 중간 삽입 힘듦.│
│  3. Hash (우편함): 책 이름에 따라 정해진 칸(번호)에 넣음. 이름만 알면 바로 찾음.    │
│  4. Indexed (색인 도서관): 책은 뒤죽박죽일 수 있지만, 앞단에 '찾아보기' 카드가 있음.  │
│                                                                             │
│  → 데이터의 **'사용 목적'**에 따라 책을 정리하는 전략이 달라지는 것과 같음!         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **물리적 제약**: 디스크 I/O는 메모리 연산보다 수십만 배 느리므로, 한 번의 디스크 접근으로 원하는 데이터를 찾아야 함.
- **워크로드 분화**: 단순 로그 적재(Write-heavy)와 실시간 검색(Read-heavy) 워크로드를 동시에 만족하기 위한 다양한 구조 연구.

- **📢 섹션 요약 비유**: 저장 구조는 '데이터가 잠드는 침대'를 설계하는 일이며, 숙면(성능)을 위해 체형(워크로드)에 딱 맞는 설계가 필요합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 4대 주요 파일 저장 구조 상세 분석

| 구조 | 특징 (Storage Strategy) | 장점 | 단점 | 주요 용도 |
|:---:|:---|:---|:---|:---|
| **Heap File** | 순서 없이 빈 공간에 삽입 | 삽입(Insert) 속도 최고 | 검색 시 전수 조사(Full Scan) | 로그 파일, 임시 테이블 |
| **Sequential** | 특정 키 값 순서로 정렬 저장 | 순차 처리(Batch) 효율적 | 삽입/삭제 시 레코드 이동 오버헤드 | 월말 결산, 백업 데이터 |
| **Hash File** | 해시 함수 결과를 주소로 사용 | 특정 키 검색 속도 최고 ($O(1)$) | 범위 검색(Range Query) 불가 | 사용자 계정, 세션 정보 |
| **Indexed** | 데이터와 별도로 색인 영역 관리 | 검색 및 순차 처리 균형 | 추가 저장 공간 및 유지비용 발생 | 대부분의 RDB 테이블 |

### 2. 파일 저장 구조 아키텍처 비교 (ASCII)

각 구조가 디스크 블록 상에 어떻게 배치되는지를 시각화한다.

```text
    [ 1. Heap File ]           [ 2. Sequential File ]       [ 3. Hash File ]
    +--------------+           +--------------------+       +--------------+
    | Rec 3 (New)  |           | Rec 1 (Key: 10)    |       | Bucket 1     |
    +--------------+           +--------------------+       | - Rec (H=1)  |
    | Rec 1 (Old)  |           | Rec 2 (Key: 20)    |       +--------------+
    +--------------+           +--------------------+       | Bucket 2     |
    | Rec 2 (Old)  |           | Rec 3 (Key: 30)    |       | - Rec (H=2)  |
    +--------------+           +--------------------+       +--------------+
    (빈 공간 삽입)              (키 순서 정렬)               (해시 함수 주소)

    [ 4. Indexed Sequential (ISAM/VSAM) ]
    ┌──────────────────┐       ┌────────────────────────────┐
    │  [ Index Area ]  │       │      [ Data Area ]         │
    │  Key 10 -> Blk 1 │ ────▶ │ Block 1: [Rec 10, Rec 15]  │
    │  Key 20 -> Blk 2 │ ────▶ │ Block 2: [Rec 20, Rec 25]  │
    └──────────────────┘       └────────────────────────────┘
```

### 3. 심층 동작 원리: 해시 파일의 충돌(Collision) 해결
해시 함수가 서로 다른 키에 대해 동일한 주소를 반환할 때의 처리 방식입니다.
1. **Chaining**: 동일한 해시 버킷에 연결 리스트(Linked List)로 레코드를 매달음.
2. **Open Addressing**: 충돌 시 다음 빈 버킷을 찾아 저장.
3. **Bucket Overflow**: 버킷이 꽉 차면 오버플로우 영역을 별도로 할당하여 관리.

### 4. 핵심 기술: ISAM (Indexed Sequential Access Method)
- **정적 인덱스**: 인덱스 구조가 고정되어 있어 데이터가 늘어나면 오버플로우 영역이 커지고 성능이 급격히 저하됨.
- **VSAM (Virtual Storage Access Method)**: ISAM의 단점을 보완한 동적 인덱스 구조로, B-Tree 계열의 인덱싱을 활용하여 성능 유지.

- **📢 섹션 요약 비유**: 해시 구조는 '내 전용 우편함'을 갖는 것이고, 인덱스 구조는 '도서관 입구의 검색 컴퓨터'를 활용하는 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 검색 성능 및 활용성 비교 매트릭스

| 비교 항목 | Heap | Sequential | Hash | Indexed |
|:---:|:---:|:---:|:---:|:---:|
| **단일 행 검색** | $O(N)$ | $O(\log N)$ | **$O(1)$** | $O(\log N)$ |
| **범위 검색** | 불량 | **우수** | 불가 | **매우 우수** |
| **삽입 성능** | **최상** | 불량 | 우수 | 보통 |
| **공간 효율성** | 우수 | 우수 | 보통 (공간 낭비) | 보통 (인덱스 저장) |

### 2. 과목 융합 관점: OS 및 자료구조
- **운영체제(OS) 융합**: 파일 저장 구조는 OS의 **File System Block** 관리와 직접 연동된다. DBMS는 OS의 버퍼 캐시를 우회하는 **Direct I/O**를 사용하여 저장 구조의 효율을 직접 제어하기도 한다.
- **자료구조 융합**: Hash File은 **Hash Table**, Indexed File은 **B-Tree / B+Tree** 자료구조의 물리적 구현체다.

- **📢 섹션 요약 비유**: 저장 구조를 고르는 것은 '가장 빠른 길을 찾는 지도'를 만드는 법을 결정하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 대규모 로그 수집 시스템의 파일 구조
- **상황**: 초당 수만 건의 접속 로그를 저장해야 함. 조회보다는 저장이 압도적임.
- **판단**: 정렬이나 인덱싱은 삽입 속도를 떨어뜨림.
- **결단**: **Heap File 구조(또는 Append-only)**를 선택하여 삽입 성능을 극대화하고, 조회가 필요한 시점에 별도의 인덱스를 생성하거나 대화형 분석 툴(ELK 등)로 이관함.

### 2. 안티패턴: 범위 검색이 잦은 컬럼에 해시 인덱스 적용
- **증상**: 날짜별 매출 조회가 많은데, 날짜 컬럼에 Hash 인덱스를 생성함.
- **결과**: `WHERE date BETWEEN '...' AND '...'` 쿼리가 인덱스를 타지 못하고 Full Table Scan 발생.
- **조치**: 범위 검색이 필요한 경우 반드시 **B+Tree 기반의 Indexed Sequential 구조**를 선택해야 함.

### 📢 기술사적 결언
> "물리적 저장 구조는 **'하드웨어의 한계를 소프트웨어의 지혜로 극복'**하는 지점이다. 데이터의 생명 주기(Life Cycle)를 분석하여, 생성 시점에는 Heap으로 성능을 잡고, 활용 시점에는 Indexed 구조로 효율을 잡는 **계층형 저장 전략**이 필요하다. 기술사는 디스크 암(Arm)의 이동을 최소화하려는 고전적 노력과, SSD/NVMe 환경에서의 병렬 처리 특성을 모두 이해하고 구조를 설계해야 한다."

- **📢 섹션 요약 비유**: 잘못된 인덱스 선택은 고속도로 입구에 과속 방지턱(해시 충돌/정렬 오버헤드)을 설치하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (ROI)

| 지표 | 구조 최적화 전 | 구조 최적화 후 | 개선 효과 |
|:---|:---:|:---:|:---:|
| **평균 응답 시간** | 수 초 단위 | **밀리초(ms) 단위** | 100배 이상의 성능 향상 |
| **디스크 I/O 횟수** | 수천 회 | **수 회 이내** | 하드웨어 수명 연장 및 비용 절감 |
| **동시 사용자 수** | 제한적 | **대규모 수용 가능** | 서비스 가용성 및 확장성 확보 |

### 2. 미래 전망
- **Columnar Storage**: 행 단위가 아닌 열 단위 저장 구조(Parquet, ORC)가 OLAP 환경의 대세가 되어 분석 성능을 혁신하고 있음.
- **In-Memory Optimization**: 디스크가 아닌 메모리 구조에 최적화된 **T-Tree**나 **LSM-Tree** 등의 저장 구조가 실시간 빅데이터 처리를 위해 확산 중임.

- **📢 섹션 요약 비유**: 미래의 침대는 사용자의 몸 상태에 따라 형태가 변하듯(Autonomous), 저장 구조도 워크로드에 따라 스스로 재구성되는 방향으로 진화할 것입니다.
