# 31. 클라이언트-서버 DBMS 아키텍처 (2-Tier, 3-Tier)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 연산 부하와 비즈니스 로직, 데이터 저장 기능을 물리적·논리적으로 분산하여 자원 활용을 극대화하는 계층형 아키텍처다.
> 2. **진화**: 응용 프로그램과 DB가 직접 통신하는 **2-Tier** 방식에서, 중간에 비즈니스 로직을 처리하는 미들웨어(WAS)를 두어 보안과 확장성을 강화한 **3-Tier** 방식으로 발전했다.
> 3. **가치**: 웹(Web)과 클라우드 환경의 표준인 3-Tier는 클라이언트의 부하를 줄이고(Thin Client), 서버의 유연한 확장을 가능케 하여 대규모 트래픽 처리를 위한 핵심 인프라 구조를 형성한다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**클라이언트-서버 DBMS 아키텍처**는 데이터베이스 기능을 수행하는 **서버(Server)**와 사용자 인터페이스 및 일부 로직을 담당하는 **클라이언트(Client)**로 역할을 분담하는 구조입니다.

### 2. 💡 비유: 식당의 서비스 구조
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        DBMS 아키텍처 비유                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 2-Tier (기사식당): 손님이 주방에 직접 말하고 바로 음식을 받음.               │
│     - 빠르지만 주방(DB)이 손님(Client) 응대로 바쁨. 보안 취약.                   │
│                                                                             │
│  2. 3-Tier (대형 레스토랑): 손님 - 홀 직원(WAS) - 주방(DB) 구조.                 │
│     - 홀 직원이 주문을 정리해서 전달하므로 주방은 요리에만 집중 가능.              │
│     - 손님은 주방에 함부로 들어갈 수 없음. (보안성/확장성 우수)                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **2-Tier의 한계**: 클라이언트 수가 늘어날수록 DB 연결(Connection) 관리가 어려워지고, 비즈니스 로직 수정 시 모든 클라이언트 앱을 재배포해야 하는 문제 발생.
- **혁신적 전환 (3-Tier)**: HTTP 기반 웹 서비스의 폭발적 성장과 함께 중앙 집중식 비즈니스 로직 관리의 필요성 증대.
- **현재의 비즈니스 요구**: 수백만 명의 유저가 동시 접속하는 환경에서 로드 밸런싱(Load Balancing)과 무중단 확장이 필수적임.

- **📢 섹션 요약 비유**: 아키텍처의 발전은 '개인용 도구'에서 '기업용 인프라'로 데이터 관리의 격을 높이는 과정입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 2-Tier vs 3-Tier 구성 요소 비교

| 구분 | 2-Tier (Client-Server) | 3-Tier (Web Architecture) | 비유 |
|:---:|:---|:---|:---|
| **제1계층** | **Client (App + Logic)** | **Client (Web Browser / Thin)** | 손님 |
| **제2계층** | **Server (DBMS)** | **Application Server (WAS / Middle)** | 홀 매니저 |
| **제3계층** | (없음) | **Database Server (DBMS)** | 주방장 |
| **연결 방식** | 전용 프로토콜 (Direct) | HTTP/HTTPS -> DB Protocol | 주문서 전달 |
| **로직 위치** | 클라이언트 내부 | 미들웨어(WAS) 내부 | 레시피 관리 |

### 2. 3-Tier 아키텍처 상세 도해 (ASCII)

사용자 요청이 각 계층을 거치며 처리되는 과정을 시각화한다.

```text
    [ Layer 1: Presentation ]    [ Layer 2: Application ]    [ Layer 3: Data ]
    
    ┌───────────────────┐        ┌───────────────────┐       ┌───────────────────┐
    │   Web Browser     │        │ Web Logic / WAS   │       │   DBMS (Oracle)   │
    │   (Thin Client)   │        │ (Middleware)      │       │ (Storage Engine)  │
    └─────────┬─────────┘        └─────────┬─────────┘       └─────────┬─────────┘
              │                            │                           │
    (1) Request  ───(HTTP/REST)──▶ (2) Logic Processing ──(SQL)──▶ (3) Query Execution
              │                            │                           │
    (6) Rendering ◀───(JSON/HTML)── (5) Data Formatting ◀──(Result)── (4) Data Fetch
              │                            │                           │
    └───────────────────┘        └───────────────────┘       └───────────────────┘
              ^                            ^                           ^
              │                            │                           │
        [ UI / UX ]                 [ Business Logic ]          [ Data Persistence ]
```

### 3. 심층 동작 원리: 미들웨어(Middleware)의 역할
3-Tier의 핵심인 미들웨어는 다음과 같은 고급 기능을 수행합니다.
1. **커넥션 풀링 (Connection Pooling)**: DB 연결을 미리 생성해두고 재사용하여 연결 오버헤드 감소.
2. **트랜잭션 관리 (Transaction Management)**: 여러 DB에 걸친 분산 트랜잭션 조율.
3. **부하 분산 (Load Balancing)**: 여러 대의 WAS 서버로 요청을 분산하여 성능 확장.
4. **보안 게이트웨이**: DB 서버를 내부망에 격리하고 오직 인증된 WAS의 요청만 허용.

### 4. 핵심 기술: Thin Client vs Fat Client
- **Thin Client**: 로직의 90% 이상을 서버에서 처리하고 클라이언트는 보여주기만 함 (현재의 웹/앱 추세).
- **Fat Client**: 클라이언트 PC가 데이터 처리와 비즈니스 연산을 모두 수행함 (과거 전산실 CS 프로그램).

- **📢 섹션 요약 비유**: 미들웨어는 수많은 주문(요청)을 정돈하여 주방(DB)이 패닉에 빠지지 않게 조절하는 '교통 관제 센터'와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 계층별 확장의 유연성 (Scalability)

| 아키텍처 | 확장성 (Scalability) | 유지보수성 (Maintainability) | 보안성 (Security) |
|:---:|:---|:---|:---|
| **2-Tier** | 낮음 (DB 노드만 확장 가능) | 낮음 (앱 배포 지옥) | 낮음 (DB 포트 노출) |
| **3-Tier** | **매우 높음 (WAS 무한 증설)** | **매우 높음 (중앙 관리)** | **높음 (다층 방어)** |

### 2. 과목 융합 관점: 네트워크 및 클라우드
- **네트워크(NW) 융합**: 3-Tier는 **L4/L7 스위치(Load Balancer)**와 결합하여 고가용성(HA)을 확보한다.
- **클라우드(Cloud) 융합**: AWS의 **Autoscaling** 기능은 주로 2계층(WAS)을 타겟으로 하여 트래픽에 따라 서버 대수를 유연하게 조절한다.

- **📢 섹션 요약 비유**: 2-Tier가 '단층 상가'라면, 3-Tier는 '수직적 확장이 용이한 마천루'와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 레거시 CS 프로그램의 웹 전환
- **상황**: 10년 전 만든 2-Tier 기반 인사 시스템이 인원 증가로 인해 너무 느려짐.
- **판단**: 클라이언트의 비즈니스 로직이 너무 복잡하여 DB 부하가 심각함.
- **결단**: **3-Tier 아키텍처로 전면 개편**. 비즈니스 로직을 WAS로 이전하고, DB는 데이터 정합성과 I/O에만 집중하게 하여 성능 개선.

### 2. 안티패턴: 과도한 DB 비즈니스 로직 (Stored Procedure 오남용)
- **증상**: 모든 계산 로직을 DB 내부의 프로시저(Procedure)에 몰아넣음.
- **결과**: DB 서버의 CPU가 100%에 도달하여 시스템 마비. WAS는 놀고 있는데 DB만 죽어나감.
- **조치**: 연산 부하가 큰 비즈니스 로직은 **WAS 계층으로 이관**하여 수평적 확장(Scale-out)이 가능하게 설계.

### 📢 기술사적 결언
> "아키텍처 설계의 정수는 **'책임의 분리(Separation of Concerns)'**에 있다. DB 서버를 단순히 '데이터 창고'로 쓸 것인지, '두뇌'로 쓸 것인지에 대한 전략적 선택이 시스템의 생존을 결정한다. 기술사는 3-Tier의 복잡함을 두려워하지 말고, 미들웨어가 제공하는 **추상화의 이점(보안, 확장, 관리)**을 극대화하여 비즈니스의 유연성을 확보해야 한다."

- **📢 섹션 요약 비유**: 프로시저 오남용은 요리사가 서빙과 계산까지 다 하는 1인 식당과 같습니다. 주문이 몰리면 금방 무너집니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (3-Tier 도입 시)

| 지표 | 개선 효과 | 비즈니스 임팩트 |
|:---|:---:|:---|
| **동시 접속자 수** | 10배 이상 증가 | 대규모 마케팅 및 프로모션 가능 |
| **배포 속도** | 중앙 배포로 즉각 반영 | 비즈니스 요구사항 대응 민첩성 향상 |
| **데이터 보안** | DB 직접 노출 차단 | 해킹 사고 및 데이터 유출 방어 |

### 2. 미래 전망
- **Microservices (MSA)**: 3-Tier의 WAS 계층이 수십 개의 작은 서비스로 쪼개지는 형태로 진화함.
- **Serverless**: 계층 자체가 추상화되어 인프라 고민 없이 코드만 올리는 방식으로 발전 중이나, 논리적인 3-Tier 계층 구조는 여전히 유효함.

- **📢 섹션 요약 비유**: 형태는 변해도 '인터페이스-로직-데이터'라는 3박자의 조화는 미래 아키텍처에서도 변하지 않는 기본 원칙이 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **2-Tier**는 내가 요리사 아저씨한테 바로 가서 떡볶이를 달라고 하는 거예요.
2. **3-Tier**는 서빙해주는 언니(미들웨어)에게 주문하면, 언니가 주방에 전달하고 음식을 가져다주는 거예요.
3. 서빙 언니가 있으면 요리사 아저씨는 요리에만 집중할 수 있어서 **음식이 훨씬 더 빨리 나오고 안전**하답니다!
