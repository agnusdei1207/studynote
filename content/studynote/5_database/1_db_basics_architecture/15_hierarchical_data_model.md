# 15. 계층형 데이터 모델 (Hierarchical Model)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터를 상하 종속 관계가 명확한 **트리(Tree) 구조**로 추상화하여, 물리적 포인터(Pointer)를 통해 부모-자식 간의 1:N 관계를 고속 연결하는 데이터베이스 관리 시스템(DBMS)의 시초가 되는 모델이다.
> 2. **가치**: 관계형 모델(RDBMS)의 복잡한 연산(Join) 없이, 디스크 상의 물리적 인접성을 바탕으로 **Path Navigation (경로 탐색)** 만으로 데이터를 조회하므로, 계층적 조회 성능이 극대화된다.
> 3. **융합**: 현대의 **JSON (JavaScript Object Notation)**, **XML (eXtensible Markup Language)**, **MongoDB**와 같은 Document-oriented DB의 논리적 구조 근간이며, 파일 시스템(NTFS, EXT4)의 디렉터리 구조나 레지스트리 등에 그 맥을 이어받고 있다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 기술적 배경: 구조적 한계와의 싸움
1960년대 중반, IBM (International Business Machines Corporation)은 아폴로 계획과 같은 거대 방위 사업의 방대한 Bill of Materials (BOM, 자재 명세서)를 관리하기 위해 **IMS (Information Management System)**를 개발했습니다. 당시 자기 디스크(Disk Drive)의 액세스 속도는 현저히 느렸기에, 데이터 논리 구조와 물리적 저장 위치를 일치시켜 "읽기 경로를 최소화"하는 것이 최우선 과제였습니다. 이는 데이터의 독립성을 희생하고서라도 조회 성능을 극대화하려는 당시의 엔지니어링 철학을 반영합니다.

#### 2. 핵심 정의 및 작동 원리
**계층형 데이터 모델(Hierarchical Data Model)**은 데이터를 레코드(Record)의 노드(Node)와 간선(Edge)으로 표현하며, 모든 자식 노드(Child)가 반드시 단 하나의 부모 노드(Parent)를 가지는 **Strict Tree (엄격한 트리)** 구조를 따릅니다. 여기서 데이터 접근은 항상 **Root Segment (루트 세그먼트)**부터 시작하여 하위로 내려가는 *Top-Down 방식*으로만 가능하며, 역방향 탐색이나 형제 간(Sibling) 이동은 제약적입니다.

#### 3. ASCII 다이어그램: 역사적 진화 흐름
```text
[파일 시스템 시대]          [IMS/DL1 시대]           [현대적 계승: NoSQL]
   +---------+              +---------+              +---------+
   |   File  |              |   IMS   |              |  JSON   |
   | Folder  |              |  DBMS   |              |  Document|
   +----+----+              +----+----+              +----+----+
        |                        |                        |
        v                        v                        v
   +---------+              +---------+              +---------+
   | 1:N    |              | 1:N    |              | 1:N    |
   | Hard   |              | Pointer |              | Nested |
   | Links  |              | Chains  |              | Object |
   +---------+              +---------+              +---------+
   (물리적 경로)           (논리적 세그먼트)      (반구조화 데이터)
```
*해설: 계층형 모델은 단순한 파일 시스템의 구조를 넘어, 포인터 기반의 논리적 연결을 구현했습니다. 이는 현재 JSON이나 XML 형태로 논리적 구조가 부활한 형태로 진화했습니다.*

#### 4. 📢 섹션 요약 비유
**마치 고전적인 가문의 족보처럼, 가장(루트)에서 시작하여 아들(부모), 손자(자식)로 이어지는 혈통이 단순 명료하여 대를 잇는 순서를 따라가는 데(조회)는 최적화되어 있지만, 여러 가문을 양자 입양(다중 부모)시키는 유연한 관계 표현에는 매우 엄격한 시스템입니다.**

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 상세 구성 요소 및 내부 동작 메커니즘
계층형 모델의 데이터베이스 스키마(Schema)는 여러 개의 **Segment Type (세그먼트 타입)**으로 구성되며, 이들의 집합을 **Database Record (DBD)**라고 합니다. 데이터는 세그먼트 단위로 저장되며, 세그먼트 간의 연결은 물리적 주소나 포인터(Pointer)를 통해 유지됩니다.

| 구성 요소 | 역할 및 정의 | 내부 동작 메커니즘 | 프로토콜/키 | 비유 |
|:---:|:---|:---|:---|:---|
| **Root Segment** | 트리의 시작점. 전체 DB Record에서 유일무이한 최상위 노드. | 모든 검색(Query)의 진입 지점이며, 단 하나만 존재해야 함. | Primary Key | 회사의 CEO |
| **Parent Segment** | 하나 이상의 Child를 소유하는 상위 세그먼트. | Child Segment의 포인터를 물리적 연속성을 통해 관리. | Logical Parent ID | 부모님 |
| **Child Segment** | 단 하나의 Parent에 종속적인 하위 세그먼트. | 부모가 삭제되면 **Cascade Delete(연쇄 삭제)** 됨. | Foreign Key (Internal) | 자식 |
| **Twin Segment (Sibling)** | 동일한 부모를 가진 형제 세그먼트. | 좌우 포인터(Left/Right Sibling Pointer)를 통해 순차 접근. | Sequence Field | 형제자매 |
| **Pointer (Forward)** | 부모가 첫 번째 자식을 가리키는 포인터. | Hierarchical Sequential Access Method (HSAM) 등에서 사용. | Physical Address | 길 안내표 |

#### 2. 물리적 저장 구조와 포인터 체인
계층형 DBMS의 핵심 성능 비결은 **Forward Pointer (전방 포인터)**와 **Sibling Pointer (형제 포인터)**의 물리적 연결입니다. 데이터를 찾을 때 B-Tree 인덱스를 거치는 것이 아니라, 포인터를 따라가는 방식이므로 경로가 정해져 있을 때 매우 빠릅니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    [물리적 저장 및 포인터 구조 ASCII]                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [ROOT: Dept]                                                               │
│  ┌───────────────┐                                                          │
│  │ Dept_ID: 'D1' │                                                          │
│  │ Next_Ptr: --->┼──────────────┐                                           │
│  └───────────────┘              ▼                                           │
│                         [Child 1: Employee]                                 │
│                         ┌───────────────┐                                   │
│                         │ Emp_ID: 'E1'  │                                   │
│                         │ Sib_Ptr: ----─┼─────┐ (형제 연결)                 │
│                         │ Child_Ptr: -->│     │                             │
│                         └───────────────┘     ▼                             │
│                                             [Child 2: Employee]             │
│                                             ┌───────────────┐               │
│                                             │ Emp_ID: 'E2'  │               │
│                                             │ Sib_Ptr: NULL │ (끝)          │
│                                             └───────────────┘               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
*해설: 위 다이어그램처럼 부모 세그먼트는 첫 번째 자식을 가리키고, 자식들은 형제 간(Sibling) 포인터로 서로 연결됩니다. 이를 통해 "부모를 찾으면 형제들까지 쭉 읽어내려가는" 순차 처리가 가능해집니다.*

#### 3. 데이터 조작 언어 (DL/I)
계층형 모델(특히 IBM IMS)은 SQL (Structured Query Language)이 아닌 **DL/I (Data Language/I)**라는 절차적 언어를 사용합니다. 개발자는 데이터를 직접 SELECT하는 것이 아니라, **SSA (Segment Search Arguments)**라고 불리는 조건절을 사용하여 현재 위치(Current Position)에서 세그먼트를 이동(Traversal)합니다.

* **GU (Get Unique)**: 루트부터 시작하여 특정 경로의 유일한 세그먼트 검색
* **GN (Get Next)**: 현재 위치에서 다음 순서의 세그먼트(형제 또는 부모 자식) 검색
* **GNP (Get Next within Parent)**: 동일한 부모 아래에서 자식 세그먼트들만 순차 검색
* **ISRT (Insert)**, **DLET (Delete)**, **REPL (Replace)**

#### 4. 📢 섹션 요약 비유
**마치 도서관의 책 분류표와 책장의 배치가 완벽히 일치하는 것과 같습니다. 분류표(논리적 구조)를 따라 들어가면 해당 책(물리적 데이터)이 그 자리에 정확히 꽂혀 있어, 관계형 데이터베이스처럼 색인을 찾아보고 헤맬 필요 없이 냉장고를 여는 것처럼 바로 데이터를 꺼낼 수 있습니다.**

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 데이터 모델 3세대 심층 비교 분석
| 비교 항목 | 계층형 모델 (Hierarchical) | 망형 모델 (Network) | 관계형 모델 (Relational) |
|:---:|:---|:---|:---|
| **표현 구조** | **Tree (트리)** | **Graph (그래프)** | **Table (테이블/Relation)** |
| **부모-자식** | **1:N (단일 부모만 허용)** | **M:N (다중 부모 허용)** | N:M (키 값으로 논리적 연결) |
| **접근 방식** | 포인터 추적 (Navigation) | 포인터 추적 (Navigation) | Non-Navigation (Set based) |
| **데이터 독립성** | 낮음 (물리적 순서 의존) | 낮음 (포인터 의존) | 높음 (논리적/물리적 독립) |
| **검색 유연성** | 제한적 (Path 필수) | 매우 유연함 (Set 이용) | 매우 높음 (Ad-hoc Query) |
| **표준 언어** | DL/I (Proprietary) | CODASYL (Proprietary) | **SQL (Standard)** |
| **현대적 계승** | XML, JSON, FileSystem | Neo4j, Graph DB | Oracle, MySQL, PostgreSQL |

#### 2. 관계형 모델(RDB)과의 융합: 계층형 데이터 처리 기술
현대의 RDBMS에서도 계층형 데이터의 성능을 끌어오기 위해 **Recursive CTE (Common Table Expression)**나 **Adjacency List Model**, **Nested Set Model** 등을 사용하여 트리 구조를 테이블로 시뮬레이션합니다. 이는 관계형 모델의 유연성과 계층형 모델의 구조적 이점을 융합한 사례입니다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│             [데이터 모델의 진화과정: 융합과 분석 ASCII]                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   1960s-70s                 1980s-Present              2010s-Future         │
│                                                                             │
│  [IMS/DB2]                 [Oracle/MySQL]             [MongoDB/Neo4j]       │
│      +                         +                            +               │
│      |   (Performance)         |   (Flexibility)            | (Scale)       │
│      v                         v                            v               │
│  Speed ⭐⭐⭐⭐⭐           Speed ⭐⭐⭐               Speed ⭐⭐⭐⭐        │
│  Flex  ⭐                     Flex  ⭐⭐⭐⭐⭐           Flex  ⭐⭐⭐⭐        │
│                                                                             │
│  Point: Pointer Based      Point: Value Based        Point: Hybrid         │
│         (Fast Path)                (Join Ops)                (Sharding)     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
*해설: 초기의 계층형 모델은 속도에 집중했으나 유연성이 부족했습니다. 관계형 모델이 등장하여 유연성을 해결했으나, 대용량 트리 데이터 처리에서 성능 이슈가 발생하자 다시 NoSQL(계층형/그래프)이 부활하는 융합 사이클을 보입니다.*

#### 3. 📢 섹션 요약 비유
**계층형 모델은 고속열차(新幹線)와 같습니다. 정해진 선로(트리 경로)를 벗어날 수 없고 역(자식 노드)을 추가하려면 선로를 깔아야 하지만, 일단 선로 위에만 있으면 중간에 멈추지 않고 엄청난 속도로 질주합니다. 반면 관계형 모델은 자동차와 같아서 어디든 갈 수 있지만(유연성), 교차로(Join)에서 막힐 수 있습니다.**

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 시나리오 및 도입 체크리스트
계층형 모델(또는 그 패턴을 사용하는 NoSQL)의 도입은 **'데이터의 본질적인 상하 관계가 변하지 않는가'**에 대한 판단이 선행되어야 합니다.

**Case A: 윈도우 레지스트리 (Windows Registry) 시스템 설계**
- **상황**: OS의 부팅 정보, 드라이버 설정, 사용자 프로파일 등을 저장.
- **의사결정**: 부팅 시 수만 개의 키(Key)를 읽어야 하므로 **Lookup Latency (탐색 지연 시간)**가 최소화되어야 함.
- **결과**: 계층형 구조(Hive)를 사용하여 `HKLM\Hardware\...` 경로를 통해 물리적 I/O를 최소화.
- **성능 지표**: RDBMS B-Tree 탐색 대비 약 10배 이상의 부팅 속도 향상 효과.

**Case B: 전자 상거래 시스템의 카테고리 설계**
- **상황**: 대형 > 중형 > 소형 카테고리를 관리. 카테고리는 자주 바뀌지 않음.
- **의사결정**: RDBMS의 **Adjacency List (인접 리스트)** 모델을 사용하여 논리적 계층을 구현하거나, 성능이 중요하면 별도의 NoSQL(JSON)로 분리.
- **Trade-off**: 카테고리 수정이 잦다면 RDB, 읽기가 압도적이라면 Cache/NoSQL(계층형) 고려.

#### 2. 안티패턴 (Anti-Pattern) 및 주의사항
- **비정형 데이터(Ad-hoc) 강제 금지**: "학생(Student)과 동아리(Club)의 관계"처럼 한 학생이 여러 동아리에 속하는 **M:N 관계**를 억지로 계층형으로
