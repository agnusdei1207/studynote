# 데이터 모델링의 3단계 - 개념적, 논리적, 물리적 설계와 ERD

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 모델링은 현실 세계의 복잡한 비즈니스 프로세스와 업무 규칙(Business Rule)을 추상화하여, **컴퓨터가 이해하고 저장할 수 있는 최적화된 데이터 구조(스키마)로 변환**하는 아키텍처 공학이다.
> 2. **가치**: 핵심 엔티티를 도출하는 개념적 모델링(ERD), 정규화를 통해 데이터 무결성을 확보하는 논리적 모델링, 그리고 디스크 구조와 인덱스를 반영해 성능을 극대화하는 물리적 모델링의 체계적인 3단계를 통해 시스템의 근본적인 실패를 예방한다.
> 3. **융합**: 이 과정은 관계형 데이터베이스의 정규화 이론과 결합될 뿐만 아니라, 현대의 도메인 주도 설계(DDD) 및 마이크로서비스(MSA)의 '바운디드 컨텍스트(Bounded Context)' 분리 전략과 융합되어 소프트웨어 설계 전체를 지배하는 뼈대가 된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 데이터 모델링은 현실 세계의 데이터를 단순화, 추상화, 명확화하는 과정이다. 업무 요구사항을 ERD(개체-관계 다이어그램)라는 그림으로 표현하고, 이를 바탕으로 데이터베이스의 최종 테이블 구조(DDL)를 만들어내는 일련의 설계 기법이다.

- **💡 비유**: 건축물을 지을 때, 고객의 요구사항을 듣고 러프하게 스케치하는 것(개념적), 이 스케치를 바탕으로 정확한 치수와 방의 용도가 들어간 설계도를 그리는 것(논리적), 마지막으로 벽돌의 재질과 배관의 위치까지 확정하여 실제 시공 도면을 그리는 것(물리적)과 같습니다.

- **등장 배경**:
  1. **소프트웨어 위기 (Software Crisis)**: 개발자들이 요구사항 분석 없이 무작정 코딩과 테이블 생성부터 시작하다 보니, 나중에 업무 규칙이 하나만 바뀌어도 시스템 전체를 폐기해야 하는 엄청난 비용 손실이 발생했다.
  2. **의사소통의 장벽**: "고객이 주문을 취소하면 쿠폰은 어떻게 되나요?"라는 현업(기획자)의 질문을 IT 개발자가 이해하고 검증할 수 있는 공통의 도면(ERD)이 필요해졌다.
  3. **데이터 무결성의 훼손 방지**: 데이터 중복(Anomaly)을 체계적으로 막기 위해, 설계 단계에서 미리 수학적인 잣대(정규화)를 들이밀어 구조적 결함을 사전 차단하는 공학적 접근이 요구되었다.

- **📢 섹션 요약 비유**: 건물을 다 짓고 나서 "어, 화장실이 안방 옆에 있네?"라며 벽을 부수는 것은 재앙입니다. 모델링은 코딩(시공) 전에 모형과 도면을 통해 치명적인 구조적 결함을 미리 발견하고 합의하는 최소한의 방어막입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 1단계: 개념적 데이터 모델링 (Conceptual Modeling)

비즈니스 요구사항을 추출하여 시스템의 가장 큰 뼈대인 '개체(Entity)'와 개체 간의 '관계(Relationship)'를 정의하는 과정이다. (특정 DB에 전혀 종속되지 않음)

- **주요 산출물**: E-R Diagram (개체-관계 다이어그램)
- **핵심 요소**:
  - **Entity (개체)**: 관리할 대상이 되는 명사. (예: `회원`, `주문`, `상품`)
  - **Attribute (속성)**: 개체가 갖는 성질. (예: `회원`의 `이름`, `주민번호`)
  - **Relationship (관계)**: 개체 간의 연관성. (예: 회원이 상품을 `주문한다` - 1:N 관계)

*(개념적 모델링 단계에서는 구체적인 데이터 타입이나 길이(VARCHAR 50 등)는 고민하지 않고, 오직 비즈니스 본질에만 집중한다.)*

---

### 2. 2단계: 논리적 데이터 모델링 (Logical Modeling)

개념 모델을 바탕으로 관계형 데이터베이스의 이론에 맞게 논리적 스키마를 구성하는 **가장 핵심적인 단계**다.

- **주요 작업**:
  - **식별자 (Primary Key, PK) 도출**: 각 개체를 유일하게 식별할 수 있는 키(Key) 확정.
  - **정규화 (Normalization)**: 1NF ~ 3NF(또는 BCNF) 과정을 거치며 데이터의 중복을 제거하고 이상 현상(Anomaly)을 방지하도록 엔티티를 쪼갬. (예: `주문` 릴레이션에서 `배송지 정보` 분리)
  - **M:N (다대다) 관계 해소**: RDBMS는 M:N 관계를 직접 표현할 수 없으므로, 중간에 '교차 엔티티(Mapping Table)'를 두어 1:N과 N:1 관계로 풀어냄.
- **특징**: 이 단계까지도 여전히 Oracle을 쓸지 MySQL을 쓸지 결정하지 않은 논리적 상태를 유지한다.

---

### 3. 3단계: 물리적 데이터 모델링 (Physical Modeling)

논리적 모델을 바탕으로, 실제 도입할 데이터베이스 벤더(DBMS)의 특성과 하드웨어 성능을 고려하여 물리적 스키마(Table)로 구현하는 단계다.

- **주요 작업**:
  - **데이터 타입 및 크기 지정**: `문자열`을 Oracle의 `VARCHAR2(50)` 또는 MySQL의 `VARCHAR(50)`으로 구체화.
  - **반정규화 (De-normalization)**: 정규화로 잘게 쪼개진 테이블들이 조회 성능(OLAP)을 저하시킬 것으로 예상될 경우, 의도적으로 조인을 줄이기 위해 테이블을 다시 합치거나 요약/파생 컬럼을 추가함.
  - **성능 튜닝 (Tuning)**: B-Tree 인덱스 설계, 파티셔닝(Partitioning), 테이블스페이스(디스크 할당) 지정 등.

- **📢 섹션 요약 비유**:
  - 개념적 모델링: "우리 집에 거실, 주방, 화장실이 필요해." (스케치)
  - 논리적 모델링: "거실은 5평, 화장실에는 변기와 세면대를 둬야 냄새가 안 섞여." (정규화된 설계도)
  - 물리적 모델링: "화장실 바닥은 미끄러지지 않게 삼성물산 타일을 깔고 배수관을 두껍게 하자." (시공 도면 및 성능 최적화)

## Ⅲ. 융합 비교 및 다각도 분석

### 관계형 모델링(RDBMS) vs 도메인 주도 설계(DDD / NoSQL) 비교

현대 아키텍처에서는 전통적인 RDBMS 모델링 기법과 NoSQL 시대의 모델링 기법이 크게 대조된다.

| 비교 항목 | 전통적 RDBMS 데이터 모델링 (정규화) | NoSQL / DDD 데이터 모델링 (Aggregate) |
|:---|:---|:---|
| **설계 철학** | 데이터 중복 배제 (Data-centric) | 애플리케이션의 쿼리/객체 중심 (App-centric) |
| **모델링 시작점** | 개체(Entity)와 명사 도출 | **쿼리 패턴 (Query Pattern) 및 액션 도출** |
| **조인 (Join)** | DB 내부 엔진의 강력한 Join 기능 신뢰 | Join 불가 전제, 데이터를 하나의 문서(JSON)에 **임베딩(Embedding)** |
| **정합성 경계** | 전사 통합 단일 진실 공급원 (SSOT) 지향 | **바운디드 컨텍스트(Bounded Context)** 단위로 데이터 분리 및 중복 허용 |

### 과목 융합 관점

- **소프트웨어 공학 (마이크로서비스)**: 과거에는 전사 시스템을 아우르는 거대한 통합 ERD 하나를 그리는 것이 미덕이었다. 그러나 MSA 시대에는 시스템 결합도를 낮추기 위해, "주문 서비스의 '상품' 엔티티와 배송 서비스의 '상품' 엔티티는 서로 다르게 생겨도(중복되어도) 좋다"는 철학으로 도메인 모델링이 융합 진화했다.
- **옵티마이저 및 인덱스 (튜닝)**: 물리적 모델링 단계에서 복합 키(Composite Key)의 순서를 어떻게 잡느냐(예: `[성별+생년월일]` vs `[생년월일+성별]`)에 따라, 나중에 B-Tree 인덱스의 스킵 스캔(Skip Scan) 효율이 100배 이상 차이 나는 치명적인 DB 튜닝 융합 지점이다.

- **📢 섹션 요약 비유**: 전통적 모델링이 '모든 레고 블록을 색깔별, 크기별로 완벽히 분해해서 서랍에 꽂아두는 꼼꼼함'이라면, 최신 NoSQL/DDD 모델링은 '어차피 매일 로봇을 만들 거니까, 로봇 팔과 다리 파츠를 굳이 분해하지 않고 한 통에 뭉쳐서 보관하는 실용주의'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '회원 탈퇴 및 이력 보존' 모델링 설계의 딜레마

- **문제 상황**: 회원이 탈퇴할 때 `User` 테이블에서 레코드를 `DELETE`하면, 기존에 이 회원이 작성한 `Order(주문)` 테이블의 행들이 부모를 잃고 고아 데이터(Orphan Record)가 되며 참조 무결성(FK) 제약 조건에 의해 에러가 터짐.
- **의사결정 (논리적 모델링 관점)**:
  1. **물리적 삭제 금지**: 실무 RDBMS에서는 `DELETE` 명령을 거의 사용하지 않는다. 대신 `status` 컬럼을 만들어 '탈퇴(Withdraw)' 상태로 `UPDATE`하는 **논리적 삭제(Soft Delete)** 패턴을 적용한다.
  2. **이력 테이블 분리**: 고객의 주소나 전화번호가 바뀔 때마다 덮어쓰면(Update) 과거 배송 이력을 추적할 수 없다. 이를 방지하기 위해 `User` 테이블 외에 `User_History` 테이블을 별도로 도출하여 점대점(시계열) 이력을 스냅샷으로 보존하는 모델링을 추가한다. (Slowly Changing Dimension 기법)

### 도입 체크리스트
- **데이터 명명 규칙 (Naming Rule)**: 논리적/물리적 모델링의 가장 흔한 실패 원인은 용어의 불일치다. A 개발자는 `cust_id`, B 개발자는 `user_no`, C 개발자는 `client_sn`으로 마음대로 테이블을 만들면 전사 조인이 불가능해진다. 전사 데이터 사전(Data Dictionary)이 존재하고 이를 강제하는가?
- **슈퍼타입/서브타입 모델 설계**: '개인 고객'과 '법인 고객'이 있을 때, 이를 하나의 `통합 고객` 테이블에 때려 넣을 것인가(Single Table), 아니면 공통 속성만 부모로 빼고 2개의 자식 테이블로 찢을 것인가(Joined Table)? 트랜잭션 패턴(공통 조회가 많은지, 개별 조회가 많은지)에 따라 신중히 물리 모델을 선택해야 한다.
- **대리 키 (Surrogate Key) 사용 여부**: 주민등록번호나 전화번호 같은 비즈니스 의미가 있는 자연 키(Natural Key)를 PK로 쓰면, 나중에 법이 바뀌거나 번호가 변경될 때 모든 연관 테이블의 FK를 다 뜯어고쳐야 하는 재앙이 발생한다. 반드시 의미 없는 일련번호(Auto_increment, UUID)를 대리 키로 사용하는가?

### 안티패턴: 개념/논리 모델링 건략 생략하고 바로 CREATE TABLE 치기 (Cowboy Coding)
- 일정이 바쁘다는 핑계로 ERD를 생략하고 즉흥적으로 DB 테이블부터 만들고 개발을 시작하는 행위. 초기 일주일은 빠를지 몰라도, 프로젝트 중반부에 1:N 관계가 M:N 관계임이 뒤늦게 밝혀지면 화면 로직, 백엔드 코드, 쿼리를 전부 갈아엎어야 하는 '소프트웨어 공학의 대재앙'을 초래한다.

- **📢 섹션 요약 비유**: 뼈대(모델링) 없이 시멘트(코드)부터 바르면 처음엔 건물이 빨리 올라가는 것처럼 보이지만, 3층만 올라가도 건물이 기우뚱해지며 결국 처음부터 폭파하고 다시 지어야 합니다. 기초 공사 시간은 절대 낭비가 아닙니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 주먹구구식 테이블 생성 | 3단계 데이터 모델링 거친 후 | 개선 효과 |
|:---|:---|:---|:---|
| **개발 재작업률** | 요구사항 누락으로 50% 이상 재작업 | ERD 검증을 통해 구조 결함 사전 차단 | **프로젝트 기한 지연 및 재작업 비용 80% 감소** |
| **데이터 무결성** | 곳곳에 NULL과 잉여 데이터 양산 | 정규화를 통한 완벽한 중복 제거 | **스토리지 낭비 방지 및 정합성 버그 0화** |
| **커뮤니케이션** | 기획자와 개발자가 서로 딴소리함 | 시각화된 ERD로 공통의 도면 확보 | **비즈니스 요구사항과 IT 시스템의 100% 일치** |

### 미래 전망
- **AI 기반 자동 모델링(Data-to-Model)**: 미래에는 데이터 엔지니어가 수동으로 ERD를 그리는 대신, LLM(거대 언어 모델)에게 "배달 앱 만들 건데 요구사항은 이래"라고 자연어로 입력하면, 즉시 개념적 ERD부터 3NF로 정규화된 물리적 DDL 스크립트까지 완벽하게 뽑아주는 생성형 AI 모델링 시대가 도래하고 있다.
- **스키마 리스(Schema-less)로의 유연성 확대**: 극단적인 초기 설계(빅 디자인 업프론트, BDUF)를 지양하는 애자일(Agile) 사상에 맞춰, 관계형 데이터베이스도 JSON 데이터 타입을 지원(PostgreSQL의 JSONB 등)하여 고정된 스키마와 유연한 스키마를 혼용하는 '하이브리드 모델링'이 대세로 자리 잡고 있다.

- **📢 섹션 요약 비유**: 데이터 모델링은 단순히 엑셀 표를 그리는 것이 아니라, 복잡한 현실 세계를 컴퓨터라는 0과 1의 도화지 위에 가장 논리적이고 아름답게 번역해 내는 위대한 추상화 예술입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 레고로 성을 만들 때, 아무 생각 없이 블록부터 꽂기 시작하면 나중에 문이 안 열리거나 지붕이 무너져요.
2. **개념적 모델링**은 "우리는 왕, 공주, 용이 사는 성이 필요해"라고 그림(ERD)을 그리는 거고, **논리적 모델링**은 "용의 방은 불타지 않는 블록으로, 공주 방은 예쁜 블록으로 분리하자(정규화)"라고 규칙을 정하는 거예요.
3. 마지막으로 **물리적 모델링**은 "우리 집 서랍에 있는 빨간 블록 10개, 파란 블록 5개를 꺼내서 실제로 조립하자(성능 튜닝)"라고 계획을 완성하는 완벽한 3단계 건축 방법이랍니다!
