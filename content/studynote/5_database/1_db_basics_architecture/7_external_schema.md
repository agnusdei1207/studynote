# 외부 스키마 (External Schema) 상세 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 외부 스키마(External Schema)는 전체 데이터베이스의 일부분으로서, 개별 사용자나 응용 프로그램의 관점에서 정의된 **데이터의 논리적 구조(View)**이다.
> 2. **가치**: 사용자마다 필요한 데이터만 선별하여 보여줌으로써 **사용자 편의성**을 높이고, 민감한 정보에 대한 접근을 제한하여 **강력한 보안 수준**을 유지하는 데이터베이스의 첫 번째 방어선이다.
> 3. **융합**: ANSI/SPARC 3단계 구조의 최상위 계층이며, 개념 스키마와의 사상(Mapping)을 통해 하부 구조의 변화로부터 사용자 프로그램을 보호하는 **논리적 데이터 독립성**의 실현 도구다.

+++

## Ⅰ. 외부 스키마 (External Schema)의 개요

### 1. 정의
- 특정 개인이나 특정 응용 분야에 한정된 논리적 데이터 구조로, '사용자 뷰(User View)' 또는 '서브 스키마(Sub-schema)'라고도 불린다.

### 2. 특징
- **다양성**: 하나의 데이터베이스 시스템에는 여러 개의 외부 스키마가 존재할 수 있다.
- **가변성**: 사용자의 요구에 따라 생성, 변경, 삭제가 비교적 자유롭다.
- **언어**: 주로 응용 프로그래머가 작성하며, SQL의 `VIEW` 명령어가 대표적인 구현 수단이다.

### 3. 💡 비유: 넷플릭스의 프로필
- 데이터베이스 전체는 넷플릭스가 가진 **'수만 편의 영화 리스트'**와 같다.
- **외부 스키마**는 내 계정의 **'나의 프로필'**과 같다. 내가 좋아하는 장르 위주로 화면이 구성되고, 어린용 프로필에서는 성인 영화가 보이지 않게 가려지는 것과 같다. 실제 영화 데이터는 하나지만, 누가 보느냐에 따라 화면 구성이 달라지는 원리다.

+++

## Ⅱ. 외부 스키마의 아키텍처 및 구현 (ASCII)

외부 스키마는 개념 스키마라는 '전체 지도' 위에서 필요한 부분만 오려낸 '개별 지도'다.

```text
    [ 인사팀 응용 프로그램 ]    [ 영업팀 응용 프로그램 ]    [ 일반 사원용 웹 ]
             │                         │                        │
    ┌────────▼────────┐       ┌────────▼────────┐      ┌────────▼────────┐
    │  외부 스키마 1  │       │  외부 스키마 2  │      │  외부 스키마 3  │
    │ (급여, 인사고과) │       │ (매출, 고객정보) │      │ (성명, 사내번호) │
    └────────┬────────┘       └────────┬────────┘      └────────┬────────┘
             │                         │                        │
    ─────────┼─────────────────────────┼────────────────────────┼──────────
             └─────────────┐           │           ┌────────────┘
                           ▼           ▼           ▼
                    ┌────────────────────────────────────┐
                    │          [ 개념 스키마 ]           │  <--- 조직 전체의
                    │ (사번, 성명, 번호, 급여, 고과 등)  │       통합 논리 구조
                    └────────────────────────────────────┘
```

### 1. 논리적 데이터 독립성 실현
- 개념 스키마에 새로운 필드가 추가되어도, 외부 스키마(응용 프로그램) 정의만 그대로 유지된다면 프로그램 소스 코드를 수정할 필요가 없다. 이 연결 고리를 DBMS가 '외부/개념 사상' 정보를 통해 관리한다.

+++

## Ⅲ. 외부 스키마 vs 개념 스키마 상세 비교

| 비교 항목 | 외부 스키마 (External) | 개념 스키마 (Conceptual) |
|:---|:---|:---|
| **관점** | 개별 사용자, 응용 프로그래머 | **조직 전체, DBA** |
| **개수** | **여러 개 (Multiple)** | **단 하나 (Unique)** |
| **범위** | 데이터베이스의 일부분 | 데이터베이스 전체의 논리적 모습 |
| **주요 목적** | 사용자 인터페이스, 보안 | 데이터 구조 정의, 무결성 유지 |
| **구현 예시** | SQL View, Sub-schema | Table, Entity, Relationship |

- **📢 섹션 요약 비유**: 수만 명의 관객이 각자 자기 자리에서 무대의 다른 각도를 보는 것이 외부 스키마라면, 하늘 위에서 무대 전체를 정석대로 내려다보는 것이 개념 스키마입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. 보안 전략으로서의 외부 스키마
- **기술사적 결단**: 실무에서 응용 프로그램이 실제 테이블(Base Table)에 직접 Query를 날리게 하는 것은 보안상 매우 위험하다. 반드시 **View(외부 스키마)**를 생성하고, 해당 View에 대해서만 `SELECT` 권한을 부여함으로써 데이터 노출을 최소화(Principle of Least Privilege)해야 한다.

### 2. 가상화 및 추상화 레이어 활용
- **인사이트**: 현대의 데이터 플랫폼 아키텍처에서는 외부 스키마의 개념을 확장하여 **데이터 가상화(Data Virtualization)** 레이어를 둔다. 여러 이기종 DB의 데이터를 모아 사용자에게는 마치 하나의 테이블인 것처럼 '외부 스키마'를 제공함으로써 복잡성을 은닉한다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **보안성 향상**: 민감한 컬럼(주민번호, 급여 등)을 원천적으로 가림.
- **개발 효율성**: 개발자는 복잡한 전체 스키마를 몰라도 자기 업무에 필요한 데이터만 보고 코딩 가능.

### 2. 미래 전망
데이터 주권(Data Sovereignty)과 개인정보보호 규제(GDPR 등)가 강화됨에 따라, 외부 스키마의 중요성은 더욱 커지고 있다. 이제는 단순한 '보기' 기능을 넘어, 사용자별로 데이터를 마스킹하거나 암호화하여 보여주는 **'지능형 외부 스키마'** 기술이 데이터 거버넌스의 핵심 기술로 자리 잡을 것이다.

- **📢 섹션 요약 비유**: 거울이 빛을 굴절시켜 내가 보고 싶은 곳만 비춰주듯, 외부 스키마는 데이터의 바다에서 내게 꼭 필요한 진실만을 비춰주는 창문입니다.

+++

### 📌 관련 개념 맵 (Knowledge Graph)
- **3단계 스키마 아키텍처**: 외부 스키마가 속한 상위 구조.
- **뷰 (View)**: 외부 스키마를 구현하는 실질적인 SQL 객체.
- **개념 스키마**: 외부 스키마의 모태가 되는 전체 구조.

+++

### 👶 어린이를 위한 3줄 비유 설명
1. 외부 스키마는 장난감 도서관의 **'나만의 장바구니'**와 같아요.
2. 수만 개의 장난감 중에서 내가 빌릴 수 있는, 그리고 내가 좋아하는 것들만 따로 모아놓은 목록이죠.
3. 다른 친구의 장바구니는 볼 수 없게 되어 있어서, 내 장난감들을 안전하고 편리하게 관리할 수 있답니다!
