# 3단계 스키마 아키텍처 (ANSI/SPARC Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ANSI/SPARC 3단계 아키텍처는 데이터베이스 시스템을 **외부(사용자), 개념(조직), 내부(저장)**의 세 가지 계층으로 분리하여 정의한 표준 구조 모델이다.
> 2. **가치**: 각 계층 사이의 독립성을 확보하여 하위 계층의 변경이 상위 계층에 영향을 주지 않도록 함으로써, 시스템의 **유지보수 효율성**과 데이터 관리의 유연성을 극대화한다.
> 3. **융합**: 현대 DBMS 설계의 근간이 되는 이론적 틀이며, 응용 프로그램과 물리적 저장 장치 사이의 복잡한 의존성을 제거하여 데이터의 공유와 보안을 동시에 실현한다.

---

## Ⅰ. 3단계 스키마 아키텍처의 개요

### 1. 정의
- 1975년 ANSI/X3/SPARC 위원회에서 제안한 데이터베이스 관리 시스템의 표준 모델로, 사용자로부터 물리적 저장 장치까지의 관점을 3단계로 추상화한 구조다.

### 2. 등장 배경: "종속성과의 전쟁"
- 과거에는 데이터 저장 방식(예: 파일 위치)이 바뀌면 모든 응용 프로그램을 새로 짜야 했다.
- 이를 해결하기 위해 "사용자는 논리적 모습만 보고, 기계는 물리적 모습만 신경 쓰게 하자"는 철학이 도입되었다.

### 3. 💡 비유: 레스토랑의 운영 구조
- **외부 단계 (손님)**: 메뉴판을 보고 음식을 고름. 주방이 어떻게 생겼는지 알 필요 없음.
- **개념 단계 (지배인)**: 식당 전체의 메뉴 구성, 재료 관리, 매출 관리를 총괄함.
- **내부 단계 (주방/창고)**: 냉장고의 위치, 식재료 보관 방식, 불 조절 장치 등을 관리함.
- 주방의 냉장고 위치가 바뀌어도(내부 변경), 손님이 보는 메뉴판(외부)은 변하지 않는 것과 같다.

## Ⅱ. 3단계 구조 상세 및 사상 (Mapping) (ASCII)

아키텍처의 핵심은 계층 간을 연결하는 **사상(Mapping)** 메커니즘이다.

```text
    [ 사용자 A ] [ 사용자 B ] [ 사용자 C ]  <--- 1. 외부 단계 (External Level)
    └─────┬────┘ └─────┬────┘ └─────┬────┘      (사용자 뷰 / 서브스키마)
          │            │            │
    ──────┼────────────┼────────────┼─────────── << 외부/개념 사상 >>
          └──────┬─────┴────────────┘           (논리적 데이터 독립성)
                 ▼
          ┌────────────────────────┐
          │     [ 개념 스키마 ]    │        <--- 2. 개념 단계 (Conceptual Level)
          └────────────┬───────────┘            (조직 전체 뷰 / 통합 뷰)
                       │
    ───────────────────┼─────────────────────── << 개념/내부 사상 >>
                       ▼                        (물리적 데이터 독립성)
          ┌────────────────────────┐
          │     [ 내부 스키마 ]    │        <--- 3. 내부 단계 (Internal Level)
          └────────────┬───────────┘            (저장 뷰 / 물리적 구조)
                       │
                       ▼
          ┌────────────────────────┐
          │    [ 물리적 데이터 ]   │        <--- 실제 저장 장치
          └────────────────────────┘
```

### 1. 외부 단계 (External Level)
- 개별 사용자나 응용 프로그램의 관점에서 본 스키마다. (View 개념)
- 동일한 데이터라도 부서별로 보는 항목과 권한이 다를 수 있다.

### 2. 개념 단계 (Conceptual Level)
- 모든 사용자의 관점을 통합한 조직 전체의 논리적 구조다.
- 개체 간의 관계, 제약 조건, 접근 권한 등을 정의하며 DBA가 설계한다.

### 3. 내부 단계 (Internal Level)
- 실제 물리적 저장 장치에서 데이터가 저장되는 방식이다.
- 레코드 구조, 필드 크기, 인덱스 활용 등을 정의한다.

---

## Ⅲ. 계층 간 사상(Mapping)과 독립성

| 사상 (Mapping) | 연결 계층 | 실현되는 가치 | 상세 설명 |
|:---|:---|:---|:---|
| **외부/개념 사상** | 외부 $\leftrightarrow$ 개념 | **논리적 독립성** | 개념 스키마가 변해도 외부 뷰는 유지되도록 연결 정보 수정 |
| **개념/내부 사상** | 개념 $\leftrightarrow$ 내부 | **물리적 독립성** | 내부 저장 구조(인덱스 등)가 변해도 전체 논리 구조는 유지 |

- **📢 섹션 요약 비유**: 두 계층 사이의 '번역기' 역할을 하는 사상(Mapping) 정보가 있기에, 한쪽이 바뀌어도 다른 쪽은 아무 일 없다는 듯이 자기 일을 계속할 수 있습니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. 데이터 독립성 확보를 위한 DBA의 역할
- **기술사적 결단**: 실무에서 성능을 위해 비정규화(Denormalization)를 하거나 저장 장치를 교체할 때, 응용 프로그램 소스 코드를 건드리지 않고 **DBMS의 사상 정보와 스키마 정의만 수정**하여 처리하는 것이 아키텍처를 준수하는 올바른 방향이다.

### 2. 뷰(View)를 통한 외부 스키마 구현
- **인사이트**: 현대 SQL 기반 RDBMS에서 외부 스키마는 주로 **View**나 **Stored Procedure**를 통해 구현된다. 이는 보안 강화와 더불어 논리적 독립성을 실무적으로 구현하는 가장 강력한 수단이다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **유지보수 비용 절감**: 하드웨어나 논리 구조 변경에 따른 영향도가 전파되지 않음.
- **데이터 공유 및 보안**: 하나의 물리 데이터 위에서 수많은 사용자 맞춤형 뷰 제공 가능.

### 2. 미래 전망
클라우드 DB(DBaaS) 시대로 가면서 사용자에게 '내부 단계'는 완전히 은닉되고 있다. 이제는 3단계 아키텍처를 넘어, 여러 클라우드에 분산된 데이터를 하나의 개념적 스키마로 묶는 **데이터 패브릭(Data Fabric)**이나 **데이터 가상화(Data Virtualization)** 기술이 3단계 스키마의 정신을 현대적으로 확장하고 있다.

- **📢 섹션 요약 비유**: 단단한 콘크리트 벽 대신, 언제든 옮길 수 있는 유연한 파티션으로 방을 나눈 스마트 오피스와 같은 구조입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 3단계 아키텍처는 **'장난감 상자를 꾸미는 3가지 규칙'**과 같아요.
2. 내가 노는 법(외부), 상자 속 칸막이(개념), 그리고 상자가 나무인지 플라스틱인지(내부)를 따로 정하는 거죠.
3. 이렇게 따로 정해두면, 상자 재료가 바뀌어도 내가 노는 법이나 칸막이 위치를 고칠 필요가 없어 아주 편하답니다!
