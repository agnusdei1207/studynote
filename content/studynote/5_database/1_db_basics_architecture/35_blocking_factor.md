# 35. 블로킹 팩터 (Blocking Factor)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하나의 물리적 디스크 블록(Block)에 저장할 수 있는 논리적 레코드(Record)의 개수를 의미하며, 보통 **$Bfr = \lfloor B / R \rfloor$** (B: 블록 크기, R: 레코드 크기)로 계산된다.
> 2. **성능 직결**: 블로킹 팩터가 클수록 한 번의 I/O로 더 많은 데이터를 읽어 들일 수 있어 전체적인 시스템의 처리량(Throughput)이 향상되고 디스크 접근 횟수가 감소한다.
> 3. **최적화**: 레코드 크기를 줄이거나 블록 크기를 워크로드에 맞게 조정하여 블로킹 팩터를 최적화하는 것은 대규모 데이터 검색 성능을 결정짓는 물리적 튜닝의 기초다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**블로킹 팩터 (Blocking Factor, Bfr)**는 보조기억장치와 메인 메모리 사이의 데이터 전송 단위인 **블록(Block)** 내에 몇 개의 **레코드(Record)**가 들어가는지를 나타내는 수치입니다.

### 2. 💡 비유: 이삿짐 트럭과 박스
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        블로킹 팩터 비유                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 레코드 (물건): 우리가 옮겨야 할 작은 물건들.                                 │
│  2. 블록 (이삿짐 박스): 물건을 담아 한 번에 옮기는 단위.                          │
│  3. 블로킹 팩터 (Bfr): **"박스 하나에 물건이 몇 개 들어가는가?"**                 │
│                                                                             │
│  → 박스가 크거나 물건이 작아서 한 박스에 많은 물건을 담을 수 있다면(Bfr이 높으면), │
│     왔다 갔다 하는 횟수(I/O)가 줄어들어 이사가 빨리 끝납니다!                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **I/O 병목**: CPU 속도에 비해 디스크 헤드가 움직여 데이터를 읽는 속도는 매우 느림.
- **전송 단위의 차이**: 운영체제나 하드웨어는 바이트(Byte) 단위가 아닌 블록(Sector/Page) 단위로 데이터를 전송함.
- **효율성 극대화**: 한 번 디스크를 읽을 때(Seek Time 발생) 가급적 많은 유효 데이터를 가져오기 위해 '블로킹' 개념 도입.

- **📢 섹션 요약 비유**: 블로킹 팩터는 '한 번의 발걸음으로 옮길 수 있는 짐의 양'이며, 발걸음 수를 줄이는 것이 성능의 핵심입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 블로킹 팩터 산출 공식 및 변수

| 변수 | 의미 | 상세 설명 |
|:---:|:---|:---|
| **$B$** | **Block Size** | 디스크 블록의 물리적 크기 (보통 4KB, 8KB 등) |
| **$R$** | **Record Size** | 하나의 데이터 행이 차지하는 바이트 수 |
| **$Bfr$** | **Blocking Factor** | $\lfloor B / R \rfloor$ (내림 처리, 소수점 이하 버림) |
| **$W$** | **Wasted Space** | $B - (Bfr \times R)$, 블록 내에 남는 빈 공간 |

### 2. 블로킹 구조 아키텍처 (ASCII)

하나의 블록 내에 여러 레코드가 배치되는 방식과 남는 공간(Unused Space)을 시각화한다.

```text
    [ Disk Block (Size B) ]
    ┌──────────────────────────────────────────────────────────────┐
    │ ┌────────────┐ ┌────────────┐ ┌────────────┐                │
    │ │  Record 1  │ │  Record 2  │ │  Record 3  │   (Empty)      │
    │ │  (Size R)  │ │  (Size R)  │ │  (Size R)  │                │
    │ └────────────┘ └────────────┘ └────────────┘                │
    └──────────────────────────────────────────────────────────────┘
    <─────────── Block Factor (Bfr = 3) ────────────> <── Wasted ──>

    * 레코드가 블록 경계에 걸쳐 저장되는 것을 허용하느냐(Spanned) 
      아니냐(Unspanned)에 따라 구현 방식이 달라짐.
```

### 3. 심층 동작 원리: Spanned vs Unspanned
- **Unspanned (비신장)**: 레코드가 블록 크기보다 작을 때, 블록 경계를 넘지 않게 저장. 남는 공간은 낭비되지만 관리가 단순함 (대부분의 RDB).
- **Spanned (신장)**: 레코드 크기가 블록보다 크거나, 공간 낭비를 줄이기 위해 하나의 레코드를 두 블록에 걸쳐 저장. 포인터 관리가 복잡하고 I/O가 늘어날 수 있음.

### 4. 성능 영향 분석
- **Total Blocks ($b$)**: 전체 레코드 수($r$)가 있을 때, 필요한 블록 수 $b = \lceil r / Bfr \rceil$.
- **Linear Scan**: 전체 데이터를 읽을 때 걸리는 시간은 $b$에 비례함. 따라서 $Bfr$을 높이면 $b$가 줄어들어 조회 성능이 선형적으로 향상됨.

- **📢 섹션 요약 비유**: Unspanned는 '박스에 물건을 꽉 채우지 못해도 뚜껑을 닫는 것'이고, Spanned는 '물건을 분해해서 두 박스에 나눠 담는 것'입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 블로킹 팩터 향상 전략 비교

| 전략 | 방법 | 장점 | 단점 |
|:---:|:---|:---|:---|
| **레코드 압축** | 데이터 압축 알고리즘 적용 | $R$이 줄어들어 $Bfr$ 증가 | CPU 연산 부하 증가 |
| **블록 크기 확대** | DB 생성 시 Block Size 증가 | $B$가 늘어나 $Bfr$ 증가 | 메모리 버퍼 풀 낭비 가능성 |
| **컬럼 분리** | 자주 쓰는 컬럼만 별도 저장 | 실질적인 $R$ 감소 | 조인(Join) 오버헤드 발생 |

### 2. 과목 융합 관점: OS 및 파일 시스템
- **운영체제(OS) 융합**: OS의 **Page Size**와 DBMS의 **Block Size**가 배수 관계(또는 일치)를 가져야 이중 버퍼링으로 인한 성능 저하를 막을 수 있다.
- **네트워크(NW) 융합**: 분산 DB 환경에서 노드 간 데이터를 보낼 때, 블로킹 팩터가 고려된 **MTU (Maximum Transmission Unit)** 최적화가 수반되어야 패킷 파편화를 방지할 수 있다.

- **📢 섹션 요약 비유**: 블록 크기를 키우는 것은 '큰 트럭을 빌리는 것'이고, 레코드를 압축하는 것은 '짐을 진공 포장하는 것'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: OLTP vs OLAP 환경의 블록 크기 결정
- **상황**: 새로운 데이터 웨어하우스(OLAP)를 구축하려 함. 대량의 데이터를 한꺼번에 읽는 작업이 많음.
- **판단**: 작은 블록 크기는 I/O 횟수를 폭증시킴.
- **결단**: **블록 크기를 16KB 또는 32KB로 크게 설정**하여 블로킹 팩터를 극대화하고, Sequential Read 성능을 확보함. (반대로 짧은 트랜잭션이 많은 OLTP는 4~8KB 선호)

### 2. 안티패턴: 너무 큰 레코드 정의 (Mega-Record)
- **증상**: 한 테이블에 수백 개의 컬럼을 정의하여 레코드 크기($R$)가 블록 크기($B$)에 육박함.
- **결과**: **$Bfr = 1$**이 되어, 데이터 한 행을 읽을 때마다 블록 하나를 통째로 읽어야 함. 인덱스 효율이 급감하고 버퍼 캐시 히트율 하락.
- **조치**: 테이블을 수직 분할(Normalization)하거나, 대용량 텍스트는 **LOB (Large Object)**로 분리하여 메인 블록의 $Bfr$을 유지.

### 📢 기술사적 결언
> "블로킹 팩터는 **'데이터의 밀도'**를 결정하는 물리적 지표다. 소프트웨어 아키텍처가 논리적 구조를 다룬다면, 블로킹 팩터는 물리적 한계를 다룬다. 기술사는 단순히 수식을 계산하는 것을 넘어, **캐시 라인 적중률, 디스크 탐색 지연, 메모리 페이지 폴트**라는 하부 계층의 파급 효과를 고려하여 최적의 블로킹 전략을 수립해야 한다."

- **📢 섹션 요약 비유**: 메가 레코드는 '박스 하나에 간신히 들어가는 거대한 소파'와 같습니다. 소파 하나 옮길 때마다 큰 박스가 필요하니 운반 효율이 최악입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (Bfr 최적화 시)

| 지표 | 개선 효과 | 비즈니스 가치 |
|:---|:---:|:---|
| **I/O Wait Time** | 30~50% 감소 | 시스템 응답 속도 체감 향상 |
| **Buffer Cache 효율** | 동일 메모리 대비 데이터 적재량 증가 | 하드웨어 추가 증설 비용 절감 |
| **Full Scan 속도** | $Bfr$ 증가 수치에 비례하여 향상 | 대규모 통계/분석 업무 마감 시간 단축 |

### 2. 미래 전망
- **Columnar Storage의 지배**: 행 기반 $Bfr$의 한계를 넘어, 동일 타입 데이터만 모아 $Bfr$을 수천 배로 높이는 컬럼 지향 저장소가 빅데이터의 표준이 됨.
- **Storage-Class Memory (SCM)**: 낸드 플래시와 D램 사이의 SCM(예: Optane) 도입으로 블록 단위 접근의 의미가 퇴색되고, 바이트 단위의 초고속 블로킹이 가능해질 전망임.

- **📢 섹션 요약 비유**: 기술이 발전해도 '한 번에 더 많이, 더 밀도 있게' 옮기려는 노력은 계속될 것이며, 블로킹 팩터는 그 효율을 측정하는 영원한 자대와 같습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **블로킹 팩터**는 커다란 박스 하나에 **장난감을 몇 개까지** 넣을 수 있는지 나타내는 숫자예요.
2. 장난감을 차곡차곡 잘 쌓아서 한 박스에 많이 담으면, 박스를 **조금만 옮겨도** 장난감을 다 가져올 수 있죠.
3. 박스를 왔다 갔다 옮기는 건 힘드니까, 한 번에 **많이 담을수록(Bfr이 클수록)** 일이 빨리 끝난답니다!
