# 16. 망형 데이터 모델 (Network Model)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 간의 관계를 단순한 트리(Tree)가 아닌 **그래프(Graph) 구조**로 추상화하여, 자식 노드(Child Record)가 **여러 부모 노드(Parent Record)**에 소속될 수 있는 유연함을 제공하는 데이터 모델맍 패러다임.
> 2. **가치**: **CODASYL (Conference on Data Systems Languages)**의 **DBTG (Data Base Task Group)** 보고서를 통해 표준화되었으며, 포인터(Pointer) 기반의 직접 접근으로 **계층형 모델(Hierarchical Model)**의 한계였던 M:N(다대다) 관계를 물리적으로 구현하여 데이터 중복을 최소화함.
> 3. **융합**: 높은 데이터 입출력 성능과 복잡한 관계 표현력을 갖추어 현대의 **그래프 데이터베이스 (Graph DB)** 및 **지식 그래프 (Knowledge Graph)**의 논리적·물리적 기반이 되었음.

---

## Ⅰ. 개요 (Context & Background)

망형 데이터 모델은 1960년대 후반, **IBM (International Business Machines)**의 **IMS (Information Management System)**로 대변되는 계층형 모델의 구조적 rigidness(경직성)을 극복하기 위해 등장했습니다. 계층형 모델에서는 하나의 자식 레코드가 반드시 하나의 부모 레코드 아래에만 존재해야 하므로, 예를 들어 '한 학생이 여러 과목을 수강하고, 한 과목에 여러 학생이 수강하는' **N:M (Many-to-Many)** 관계를 표현하려면 데이터를 중복 저장해야 하는 치명적인 비효율이 발생했습니다.

이를 해결하기 위해 **CODASYL** 산하 **DBTG**는 1971년 언어 정의 및 스키마 명세를 제정하여, 포인터(Pointer)를 사용한 레코드 간의 유연한 연결을 가능하게 하는 망형 모델을 표준화했습니다. 이 모델은 데이터를 논리적으로만 연결하는 것이 아니라, 물리적 주소를 통한 직접 연결을 시도함으로써 당대의 디스크 I/O 성능을 극대화하는 데 주력했습니다.

### 💡 비유: 복잡한 도시의 고속도로 나들목
계층형 모델이 "단방향 통행만 허용되는 직선 고속도로"라면, 망형 모델은 "여러 도시와 복잡하게 연결된 **하이패스(Hi-Pass) 네트워크**"와 같습니다. 어느 도시(부모)에서든 다른 도시(자식)로 가는 직통 연결 라인(포인터)이 있어, 굳이 중간에 다른 도시를 거치지 않고도 원하는目的地에 바로 도달할 수 있는 구조입니다.

```text
    [ 도시 A (부모) ] ──────┐
         │                 │
         └────> [ 교차로 ] <────┘
                   │
         ┌─────────┴─────────┐
         ▼                   ▼
    [ 도시 B (자식) ]    [ 도시 C (자식) ]
    
    ※ 계층형과 달리, 도시 B와 C는 A뿐만 아니라 다른 도시(D, E...)와도
       직접 연결될 수 있습니다. 이것이 망형(Network)의 핵심입니다.
```

**📢 섹션 요약 비유**
> 마치 단순한 일직선 트리구조보다 **고속도로 나들목이 복잡하게 얽힌 거미줄 같은 인터체인지 시스템**을 구축하여, 어디서든 어디로든 빠르게 진입할 수 있게 만드는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

망형 모델의 아키텍처는 **'Record(레코드)'**, **'Set(세트)'**, **'Area(영역)'**, 그리고 **'Schema(스키마)'**로 구성됩니다. 이 모델의 핵심은 데이터를 찾기 위해 인덱스를 탐색하는 것이 아니라, **포인터(Chain)**를 따라가는 **네비게이션(Navigation)** 방식을 사용한다는 점입니다.

### 1. 구성 요소 상세 분석

| 구성 요소 | 영문 명칭 (Abbreviation) | 역할 및 내부 동작 | 비유 |
|:---:|:---|:---|:---|
| **데이터 항목** | **Data Item** | 데이터의 최소 단위 (필드) | 상자 안의 물건 |
| **레코드** | **Record** | 데이터 항목의 집합. Node 역할. | 상자 |
| **세트** | **Set** | **Owner(소유자)**와 **Member(구성원)** 간의 1:N 관계를 정의하는 논리적 포인터 구조. | 가계도 또는 멤버십 |
| **포인터** | **Pointer** | 레코드 간의 물리적 주소 값. Next, Prior, Owner 포인터로 구성됨. | 다리(연결 고리) |
| **DBA 스키마** | **DDL (Data Definition Language)** | 데이터 논리 구조 정의 (Schema, Subschema) | 도시 설계 도면 |

### 2. 망형 모델의 논리적 구조 및 포인터 링크
망형 모델에서 관계는 '세트(Set)'라는 개념으로 표현됩니다. 아래 다이어그램은 한 명의 교수(Owner)가 여러 학생(Member)을 가지고, 동시에 한 명의 학생이 여러 교수(지도교수, 강의교수)로부터 연결을 받는 구조를 보여줍니다.

```text
      [ Ⅰ. 개요(Owner): 교수 A ]
                │
                ▼ (Pointer Chain)
        ┌───────────────────┐
        │  Student 1 (Member)│ ──────┐
        └───────────────────┘       │
                ▲                   │
                │ (Pointer Chain)    │ (Member of another Set)
        ┌───────────────────┐       ▼
        │  Student 2 (Member)│ ─── [ Ⅱ. 개요(Owner): 프로젝트 B ]
        └───────────────────┘
                
    [ Owner Record: 교수 ]
          │
          ├───> (Member) Student 1 ────┐
          │                            │
          ├───> (Member) Student 2 ────┼──> (Owner) Project
          │                            │
          └───> (Member) Student 3 ────┘
```

**해설**:
1.  **Set Type**: 관계의 명세. 예: `TEACHING-SET` (교수가 Owner, 학생이 Member).
2.  **Set Occurrence**: 실제 데이터 인스턴스. 교수 A가 학생 1, 2를 관리하는 구체적 집합.
3.  **Circular Link (환형 연결)**: Member 레코드들은 'Next', 'Prior' 포인터를 통해 서로 연결되며, 마지막 Member는 다시 Owner를 가리키는 **Owner Pointer**를 가집니다. 이중 연결 리스트(Doubly Linked List) 형태로 관리되어 삽입/삭제 시 포인터 재연결 오버헤드가 발생하지만, 탐색 시 빠른 접근이 가능합니다.

### 3. 핵심 동작 메커니즘: N:M 관계의 구현
관계형 데이터베이스(RDBMS)에서 N:M 관계를 표현하기 위해 **중개 테이블(Junction Table)**을 사용하는 것과 달리, 망형 모델은 **가상 레코드(Virtual Record)와 링크(Link)**를 통해 물리적으로 해결합니다.

```text
    [ Problem: Student(N) : Course(M) ]

    1. Student A ────────────────> Course 1
       (Set 1: Enrollment)      (Owner)

    2. Student A ────────────────> Course 2
       (Member)                 (Owner)
```

**📢 섹션 요약 비유**
> 복잡한 **'비행기 표 예매 시스템'**과 같습니다. 한 여객기(Owner)에는 여러 승객(Member)이 타지만, 승객들은 동시에 여러 항공편(Member of another Set)의 예약을 가질 수 있습니다. 각 표는 **포인터라는 바코드**를 통해 서로 다른 비행기에 연결되어 있어, 표만 보면 어느 비행기에 타야 할지 바로 알 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

망형 모델을 이해하기 위해서는 관계형 모델(Relational Model), 계층형 모델(Hierarchical Model)과의 기술적 차이를 정량적으로 분석해야 합니다.

### 1. 데이터 모델 상세 비교 분석표

| 비교 항목 | **계층형 모델** (Hierarchical) | **망형 모델** (Network) | **관계형 모델** (Relational) |
|:---:|:---|:---|:---|
| **데이터 구조** | **Tree (트리)** | **Graph (그래프)** | **Table (테이블)** |
| **관계 표현** | 부모 1 : 자식 N (1:N 강제) | **부모 M : 자식 N (M:N 허용)** | 외래 키(Foreign Key)로 무한 관계 |
| **접근 방식** | **Pre-order Traversal** (위계적 순회) | **Navigation** (포인터 추적) | **Non-procedural** (SQL 선언적) |
| **데이터 독립성** | 낮음 (물리적 경로에 의존) | **매우 낮음** (경로 변경 시 프로그램 수정 필수) | 높음 (논리적/물리적 독립성 보장) |
| **검색 성능** | 빠름 (직관적 경로) | **매우 빠름** (포인터 직접 참조) | 상대적으로 느림 (Join 연산 비용) |
| **구현 복잡도** | 단순함 | **매우 높음** (포인터 관리 어려움) | 낮음 (사용자 편의성 중심) |
| **중복성** | 높음 (M:N 표현 위해 중복 저장) | **낮음** (포인터로 공유) | 낮음 (정규화를 통한 제거) |

### 2. 타 영역과의 융합 및 시너지
*   **OS (Operating System)와의 연계**: 망형 모델의 포인터 연결 구조는 OS의 **메모리 관리(Memory Management)** 기법, 특히 **페이징(Paging)**이나 **세그먼테이션(Segmentation)**의 연결 리스트와 유사한 메커니즘을 가집니다. 또한 파일 시스템의 **Inode** 구조와도 유사성을 찾을 수 있습니다.
*   **AI 및 그래프 이론**: 현대의 **Graph Neural Network (GNN)**이나 소셜 네트워크 분석(SNA)은 망형 모델의 **Topology(위상학적 구조)**를 그대로 차용하고 있습니다. 노드와 엣지(Edge) 중심의 사고방식은 망형 모델에서 시작되었습니다.

```text
    [ Convergence Example ]
    DBMS (Network Model)  +  AI (Graph Algorithm)
           │                        │
           └────────> [ Knowledge Graph ]
                         (Google, SEO, Chatbots)
```

**📢 섹션 요약 비유**
> 계층형 모델이 **"계급 엄격한 군대 조직"**이라면, 망형 모델은 **"링크와 레퍼런스로 연결된 초거대 인터넷 웹(Web)"**과 같습니다. 웹브라우저가 하이퍼링크(포인터)를 통해 문서에서 문서로 이동하듯, 망형 모델도 포인터를 통해 데이터를 순회합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무 현장에서 망형 모델의 개념은 **'빠른 성능'**과 **'구조적 유연성'**이 모두 필요할 때 선택되지만, **'유지보수 비용'**과 **'스키마 변경의 어려움'**이라는 리스크를 반드시 고려해야 합니다.

### 1. 실무 시나리오: 제조업의 BOM (Bill of Materials) 관리
자동차 제조사의 부품 명세서(BOM)를 생각해 봅시다. 하나의 '나사(부품)'는 수천 대의 서로 다른 '자동차(완제품)'에 들어갑니다.
*   **상황**: RDBMS로 설계 시 `Car_BOM` 테이블에 수백만 건의 중복 데이터(나사 정보)가 들어가거나, 복잡한 조인(Join) 연산으로 인해 조회 속도가 느려집니다.
*   **기술사적 판단**: 망형 접근 방식(또는 Graph DB)을 도입하여 '나사' 레코드를 하나만 두고, 수천 개의 '완제품' 레코드에서 포인터로 이를 가리키게 구성합니다.
*   **결과**: 데이터 중복 최소화, **TPS (Transactions Per Second)** 향상, 저장 공간 절약.

### 2. 도입 체크리스트 및 안티패턴

| 구분 | 체크 항목 | 상세 설명 |
|:---:|:---|:---|
| **기술적** | **포인터 무결성** | 데이터 삭제 시 포인터가 끊어지는 **Dangling Reference** 방지 로직이 필수임. |
| **운영적** | **경로 의존성** | 데이터에 접근하기 위해 정확한 **Access Path**를 알아야 하므로, 개발자가 전체 구조를 숙지해야 함. |
| **보안적** | **Propagate Vulnerability** | 상위 레코드의 보안 권한이 하위 레코드로 **전파(Propagate)**되는지 검증 필요. |

**❌ 안티패턴 (Anti-Pattern)**: 망형 모델을 단순히 "복잡한 관계를 표현한다"는 이유로, 동적이고 빈번하게 스키마가 변경되는(Schema-less) 애자일 개발 환경에 적용하면, 포인터 재연결 로직으로 인해 시스템이 마비될 수 있습니다.

### 3. 심층 사례: IDMS (Integrated Database Management System)
1980년대 가장 대표적인 망형 DBMS였던 **IDMS**나 **Raima Database Manager**는银行业务, 항공 예약 시스템(SABRE) 등에서 사용되었습니다. 이 시스템들은 CPU 코어가 몇 개 안 되던 시절, 포인터 연산의 효율성을 통해 초당 수만 건 이상의 트랜잭션을 처리했습니다.

**📢 섹션 요약 비유**
> 고속도로를 건설할 때 *
