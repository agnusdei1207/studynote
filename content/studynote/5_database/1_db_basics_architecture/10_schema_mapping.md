# 스키마 사상 (Schema Mapping) 상세 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스키마 사상(Mapping)은 ANSI/SPARC 3단계 구조의 서로 다른 계층 간을 연결하는 **대응 규칙이자 번역 정보**로, 한 계층의 데이터 표현을 다른 계층의 표현으로 변환해주는 매커니즘이다.
> 2. **가치**: 계층 간의 직접적인 의존성을 제거하여 하부 스키마가 변경되어도 상위 스키마나 응용 프로그램을 수정할 필요가 없게 만드는 **데이터 독립성** 구현의 실질적인 기술적 핵심이다.
> 3. **융합**: DBMS의 카탈로그(데이터 사전)에 저장되어 실시간으로 쿼리를 변환하며, 논리적 독립성을 위한 '외부/개념 사상'과 물리적 독립성을 위한 '개념/내부 사상'으로 이원화되어 관리된다.

+++

## Ⅰ. 스키마 사상 (Schema Mapping)의 개요

### 1. 정의
- 데이터베이스 시스템에서 상위 단계 스키마의 각 정의가 하위 단계 스키마의 어떤 정의와 연관되는지를 명시한 기술(Description)이다.

### 2. 필요성: "변화의 파도를 막는 방파제"
- 데이터베이스의 세계에서 변화는 필연적이다. (사용자 요구 변경, 하드웨어 교체 등)
- 사상 정보는 이 변화가 시스템 상단으로 전파되지 않도록 중간에서 '주소와 이름'을 다시 연결해주는 역할을 한다.

### 3. 💡 비유: 별명(Nickname) 관리자
- 친구의 진짜 이름(개념 스키마)은 '홍길동'이지만, 우리 가족은 '길동이'라고 부르고(외부 스키마 1), 학교 친구들은 '홍군'이라고 부른다(외부 스키마 2).
- 만약 길동이가 이름을 '홍혁명'으로 개명해도(개념 변경), **'별명 관리자(Mapping)'**가 "이제부터 홍군과 길동이는 홍혁명을 가리킨다"라고 장부만 고쳐주면, 주변 사람들은 예전처럼 별명을 부르며 대화할 수 있는 것과 같다.

+++

## Ⅱ. 2대 핵심 사상 아키텍처 (ASCII)

사상은 계층 사이의 경계선에 위치하며 '변환기' 역할을 수행한다.

```text
    [ 응용 프로그램 1 ] [ 응용 프로그램 2 ]
             │                  │
    ─────────┴────────┬─────────┴─────────
             (1) [ 외부 / 개념 사상 ]         <--- 논리적 데이터 독립성 제공
    ──────────────────▼───────────────────
             ┌──────────────────┐
             │  [ 개념 스키마 ]  │
             └────────┬─────────┘
    ──────────────────▼───────────────────
             (2) [ 개념 / 내부 사상 ]         <--- 물리적 데이터 독립성 제공
    ──────────────┬─────────────┬─────────
                  ▼             ▼
             [ 인덱스 ]    [ 데이터 파일 ]
```

### 1. 외부/개념 사상 (External / Conceptual Mapping)
- **역할**: 개별 사용자의 뷰(View)와 전체 논리 구조(Table) 사이의 관계를 정의한다.
- **기여**: 테이블에 새로운 컬럼이 생겨도 외부 뷰 정의만 유지하면 프로그램은 안전하다. (**논리적 독립성**)

### 2. 개념/내부 사상 (Conceptual / Internal Mapping)
- **역할**: 전체 논리 구조와 실제 디스크 저장 구조(파일, 인덱스) 사이의 관계를 정의한다.
- **기여**: 성능을 위해 데이터 배치 방식을 바꿔도 테이블 명칭과 구조는 유지된다. (**물리적 독립성**)

+++

## Ⅲ. 사상(Mapping)과 독립성의 상관관계

| 사상 유형 | 대응 관계 | 변경 시 조치 | 실현 가치 |
|:---|:---|:---|:---|
| **외부/개념 사상** | View $\leftrightarrow$ Table | 사상 정의(View Definition) 수정 | **논리적 독립성** |
| **개념/내부 사상** | Table $\leftrightarrow$ File/Index | 사상 정의(Storage Mapping) 수정 | **물리적 독립성** |

- **📢 섹션 요약 비유**: 계층 사이를 잇는 '고무줄'과 같습니다. 하부 계층이 움직여도 고무줄(사상)이 늘어나거나 줄어들며 충격을 흡수해주기 때문에 상부 계층은 평온을 유지할 수 있습니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. 사상 오버헤드와 성능의 균형
- **기술사적 결단**: 사상은 추상화라는 큰 이득을 주지만, 쿼리 실행 시마다 계층을 넘나들며 변환하는 **런타임 오버헤드**를 발생시킨다. 
- **인사이트**: 따라서 극단적인 초고성능이 필요한 경우(예: High-Frequency Trading)에는 일부 독립성을 희생하더라도 사상 단계를 최소화하는 '수평적 아키텍처'를 고려해야 한다.

### 2. 현대적 변천: ORM(Object-Relational Mapping)
- **판단**: 이제 사상의 개념은 DBMS 내부를 넘어 애플리케이션 코드 영역까지 확장되었다. 
- **결론**: **Hibernate나 JPA** 같은 ORM 기술은 '객체 모델'과 '관계형 모델' 사이의 사상을 자동화하여 개발 생산성을 높이는 현대판 외부/개념 사상의 확장형으로 이해해야 한다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **유지보수 비용 극소화**: 변경 영향도(Blast Radius)를 해당 계층으로 한정.
- **데이터 투명성 확보**: 하부 구조를 몰라도 데이터를 다룰 수 있는 추상화 제공.

### 2. 미래 전망
최근의 가상화 DB나 데이터 레이크하우스(Data Lakehouse) 환경에서는 수많은 이기종 저장소 간의 사상을 동적으로 관리하는 **'지능형 매핑 인프라'**가 핵심이다. 메타데이터가 스스로 데이터의 위치와 형식을 찾아 연결해주는 'Active Metadata' 기술은 3단계 스키마 사상의 원리를 미래형으로 진화시키고 있다.

- **📢 섹션 요약 비유**: 복잡한 지하철 환승 통로(사상)가 잘 뚫려 있기에, 우리는 지상의 목적지만 보고도 복잡한 지하 세계를 이용해 어디든 갈 수 있는 것입니다.

+++

### 📌 관련 개념 맵 (Knowledge Graph)
- **3단계 스키마 아키텍처**: 사상이 존재하는 구조적 틀.
- **데이터 사전 (Data Dictionary)**: 사상 정의서가 보관되는 금고.
- **데이터 독립성**: 사상을 통해 얻고자 하는 궁극의 가치.

+++

### 👶 어린이를 위한 3줄 비유 설명
1. 스키마 사상은 **'번역기'**와 같아요.
2. 영어를 한국어로 바꿔주듯, 기계가 이해하는 어려운 방식(내부)을 우리가 보기 쉬운 예쁜 화면(외부)으로 중간에서 계속 통역해주는 거죠.
3. 이 번역기만 있으면 기계가 부품을 새 걸로 바꿔도(변경), 우리는 평소처럼 편하게 컴퓨터랑 대화할 수 있답니다!
