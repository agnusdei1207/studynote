# 외래 키 (Foreign Key)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 외래 키 (Foreign Key)는 다른 테이블의 기본 키를 참조하는 속성으로, 테이블 간의 관계 (Relationship)를 표현하고 참조 무결성을 보장한다.
> 2. **가치**: 외래 키는 데이터베이스에서 정규화의基礎이며, 관련 데이터 간의 일관성을 유지하고 불필요한 데이터 중복을防止する.
> 3. **융합**: 관계형 데이터베이스에서 ER 모델의 관계를 实现하며, InnoDB, PostgreSQL 등 대부분의 RDBMS에서 참조 무결성 enforcement 메커니즘으로 활용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

외래 키 (Foreign Key)는 **다른 테이블의 기본 키 (Primary Key)를 참조하는 속성**이다. 외래 키가 존재하는 테이블을从屬 테이블, 참조되는 테이블을기준 테이블이라 한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    외래 키의 구조                                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [기준 테이블: customers]                                          │
│   ┌────────┬────────┬────────┐                                    │
│   │ 고객ID* │ 이름    │ 도시   │                                    │
│   ├────────┼────────┼────────┤                                    │
│   │ C001   │ 김철수  │ 서울   │                                    │
│   │ C002   │ 이영희  │ 부산   │                                    │
│   │ C003   │ 박민수  │ 서울   │                                    │
│   └────────┴────────┴────────┘                                    │
│        * = 기본 키                                                    │
│                                                                     │
│   [從屬 테이블: orders]                                              │
│   ┌────────┬────────┬────────┬────────┐                          │
│   │ 주문ID* │ 고객ID† │ 상품명 │ 주문일 │                          │
│   ├────────┼────────┼────────┼────────┤                          │
│   │ O001   │ C001   │ TV     │ 2024-01-01│                       │
│   │ O002   │ C002   │ 냉장고 │ 2024-01-02│                       │
│   │ O003   │ C001   │ 세탁기 │ 2024-01-03│                       │
│   └────────┴────────┴────────┴────────┘                          │
│        * = 기본 키    † = 외래 키 (customers表的 기본 키 참조)        │
│                                                                     │
│   • orders表的 고객ID는 customers表的 고객ID를 참조                  │
│   • "주문" 관계는 "고객"과 "주문" 사이의 1:N 관계를 표현            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** orders表的의 고객ID는 customers表的의 기본 키인 고객ID를 참조하는 외래 키다. 이 외래 키를 통해 "주문"이 "어떤 고객"에 속하는지를 알 수 있다. 외래 키 제약조건에 의해 orders表中의 고객ID 값은 반드시 customers表에 존재하는 고객ID이어야 한다. 이것이 참조 무결성의 핵심이다.

### 비유

외래 키는 **도서관의 상호참조 카드**와 같다. A 열람실의서지카드는 "이 책은 B 열람실에 있습니다"라고 알려주고, B 열람실의서지카드도 역으로 참조한다.

- **📢 섹션 요약 비유:**호텔에서 객실카드 (외래 키)가フロントの客户名单 (기본 키)를 참조하면, "이 객실은 어떤 고객님께서 사용 중"인지를 알 수 있습니다.

## Ⅱ. 외래 키의 동작

### 참조 무결성 (Referential Integrity)

외래 키 제약조건은 다음을 보장한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    참조 무결성 보장 예시                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [정상 동작]                                                       │
│   INSERT INTO orders VALUES ('O004', 'C003', '에어컨', '2024-01-04'); │
│   → C003가 customers表에 존재 → 허용                               │
│                                                                     │
│   [참조 무결성 위반]                                                │
│   INSERT INTO orders VALUES ('O005', 'C999', '컴퓨터', '2024-01-05'); │
│   → C999가 customers表에 존재하지 않음 → 오류 발생!                  │
│                                                                     │
│   [삭제 시 동작]                                                    │
│   DELETE FROM customers WHERE 고객ID = 'C001';                       │
│   → C001를 참조하는 orders表의 행이 존재 → 오류 또는 CASCADE 삭제     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 참조 동작 (Referential Actions)

부모 테이블 (기준 테이블)의 행이 삭제되거나 기본 키가 변경될 때 취할 수 있는 동작이다.

| 동작 | 설명 |
|:---|:---|
| **NO ACTION** | 자식 테이블에 참조하는 행이 있으면 삭제/수정 거부 |
| **CASCADE** | 부모 삭제/수정 시, 참조하는 자식 행도 함께 삭제/수정 |
| **SET NULL** | 부모 삭제/수정 시, 외래 키 값을 NULL로 설정 |
| **SET DEFAULT** | 부모 삭제/수정 시, 외래 키 값을 기본값으로 설정 |
| **RESTRICT** | 자식 테이블에 참조하는 행이 있으면 부모 삭제/수정 거부 |

```
┌─────────────────────────────────────────────────────────────────────┐
│                    CASCADE 삭제 예시                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   부모 테이블 (customers):                                            │
│   DELETE FROM customers WHERE 고객ID = 'C001';                      │
│                                                                     │
│   자식 테이블 (orders) - CASCADE 적용 시:                             │
│   → C001를 참조하는 모든 주문 행이 함께 삭제됨                        │
│   O001 (C001), O003 (C001) → 함께 삭제                              │
│                                                                     │
│   장점: 고아 데이터 (참조 없는 데이터) 발생防止                        │
│   단점: 의도치 않은 대량 삭제 위험                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 온라인 쇼핑몰**: 주문 테이블이 고객 테이블을 참조하면, 존재하지 않는 고객으로 주문을 생성하는 것을防止할 수 있다.

2. **시나리오 —两级 삭제**: 게시글을 삭제할 때 댓��글이 있으면 CASCADE로 함께 삭제하거나, 댓��글이 있으면 삭제를 거부하도록 설정할 수 있다.

### 도입 체크리스트

- **기술적**: 외래 키 인덱스를 생성했는가 (참조 성능 향상)? 참조 동작 (ON DELETE CASCADE 등)을 적절히 설정했는가?
- **운영·보안적**: 외래 키 제약조건을暂时 비활성화해야 하는 상황 (대량 데이터 import 등)을 위한 절차를 마련했는가?

### 안티패턴

- **순환 외래 키**: A→B→C→A 같은 순환 참조는 데이터 마이그레이션과 복구를 어렵게 한다.
- **과도한 외래 키**: 모든 테이블에 외래 키를 설정하면, 대량 데이터 로드 시 오버헤드가 발생할 수 있다.

- **📢 섹션 요약 비유:**사서의 동의없이图书관의目録에서 도서를 제거하면, 이미 대출한 도서의 정보가 사라지는 것처럼, 외래 키는関連数据的整合性を保護합니다.

### 정량/정성 기대효과

| 구분 | 외래 키 미정의 | 외래 키 정의 | 개선 효과 |
|:---|:---|:---|:---|
| **정성** | 고아 데이터 발생 가능 | 참조 무결성 자동 보장 | 데이터 무결성 향상 |
| **정성** | 수동으로 관계 검증 | DBMS가 자동 검증 |管理 효율성 향상 |

### 미래 전망

- **NewSQL의 분산 외래 키**: Google Spanner와 같은 분산 RDBMS에서 全球規模의 참조 무결성 보장 방법 연구
- **이벤트 소싱과 외래 키**: 이벤트 소싱アーキテクチャ에서 Eventually Consistent 참조 무결성 모델 도입

- **📢 섹션 요약 비유:**호텔에서 고객이チェックアウト할 때, 아직清算되지 않은 비용이 있으면 퇴실을 막는 것과 같습니다. 외래 키도 데이터의"청산"이 완료되지 않으면删除를허용하지 않습니다.

## 👶 어린이를 위한 3줄 비유 설명

1. 외래 키는 **교실 출석부의 "반 번호"**와 같아요. 각 학생은 자신의 반 번호를 적고, 교실 출석부는 반 번호로 누가 어느 반인지 알 수 있어요.
2. 만약 존재하지 않는 반 번호를 적으면 안 되듯이, 데이터베이스에서도 외래 키는 반드시 존재하는 기본 키를 참조해야 해요.
3. 반을 삭제하려면 먼저 그 반 학생들을 옮기거나 없애야 하는 것처럼, 기본 키가 삭제되면 외래 키도 함께 처리해야 해요!
