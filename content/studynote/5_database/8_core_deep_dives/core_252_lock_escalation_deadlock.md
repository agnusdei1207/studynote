# 잠금 에스컬레이션과 교착 상태(Deadlock) 회피 알고리즘

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동시성 제어를 위해 데이터베이스가 부여하는 락(Lock)은 그 자체로 메모리 자원을 소모한다. DB 엔진은 메모리 고갈을 막기 위해 수만 개의 정밀한 행 락(Row Lock)을 뭉뚱그려 거대한 테이블 락(Table Lock)으로 승격시키는 **잠금 에스컬레이션(Lock Escalation)**을 자동 수행하며, 이 과정에서 트랜잭션의 병렬성이 급감한다.
> 2. **가치**: 락을 쥔 트랜잭션들이 서로 상대방의 락을 요구하며 영원히 멈추는 **교착 상태(Deadlock)**를 막기 위해, DB 커널은 OS의 개념을 차용하여 타임스탬프 기반의 선점(Wound-Wait)과 비선점(Wait-Die) 예방 기법을 적용하거나 주파수 탐지(Deadlock Detection)로 희생자(Victim)를 롤백시킨다.
> 3. **융합**: 이 무거운 락킹 메커니즘을 피하기 위해, 현대 마이크로서비스(MSA) 아키텍처에서는 DB의 물리적 락에 의존하지 않고 Kafka 같은 **이벤트 큐(Message Queue)**를 활용하여 트랜잭션을 싱글 스레드처럼 **직렬화(Serialization)**하는 애플리케이션 레벨의 회피 설계와 융합된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **잠금 에스컬레이션 (Lock Escalation)**: 세밀한 락(Row-level)이 너무 많이 발생하여 DB의 락 관리 메모리(Lock Table)가 부족해질 때, 시스템 스스로 여러 개의 행 락을 하나로 묶어 범위가 더 넓은 테이블 락(Table-level)으로 자동 격상시키는 내부 최적화 매커니즘. (MS SQL Server 등에서 두드러짐)
  - **교착 상태 (Deadlock)**: 두 개 이상의 트랜잭션이 각각 상대방이 쥐고 있는 자원(Lock)을 요청하며 무한 대기(Block) 상태에 빠진 논리적 마비 상태.

- **💡 비유**: 교착 상태(데드락)는 외나무다리에서 마주친 두 자동차가 서로 비키라고 경적만 울리며 꼼짝 못 하는 상황입니다. 잠금 에스컬레이션은 경찰(DB 엔진)이 그 다리의 한 대 한 대를 통제하기 너무 힘들자, 아예 다리 입구 전체에 바리케이드를 치고 "아무도 들어오지 마!(Table Lock)"라고 다리 전체를 잠가버리는 행정 편의주의적 조치입니다.

- **등장 배경**:
  1. **락(Lock)의 메모리 점유 비용**: 트랜잭션이 안전을 위해 "Row 1번에 락", "Row 2번에 락"을 걸다 보면, 이 락 정보들을 기록하는 메모리 해시 테이블 구조체가 기하급수적으로 커져 DB 메모리가 폭발하는 한계에 도달했다.
  2. **복잡한 비즈니스 트랜잭션의 증가**: 은행 송금 시 "A계좌 차감, B계좌 입금"을 하나의 트랜잭션으로 묶으면서, 다른 트랜잭션이 "B계좌 차감, A계좌 입금"을 시도할 때 교차(Cross) 지점이 발생해 데드락이 일상화되었다.
  3. **대규모 배치 처리의 병목**: 수십만 건을 한 번에 UPDATE 치는 배치 쿼리가 돌면 수십만 개의 락이 필요해 시스템이 마비되므로, 이를 시스템 차원에서 스스로 방어(Escalation)할 논리가 필요했다.

- **📢 섹션 요약 비유**: 락(Lock)은 도둑을 막는 훌륭한 자물쇠지만, 100만 개의 자물쇠를 사면 자물쇠 무게 때문에 집이 무너지고(에스컬레이션), 자물쇠 열쇠가 서로 얽히면 주인이 집에 못 들어가는 비극(데드락)이 발생합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 락의 입도(Granularity)와 잠금 에스컬레이션

데이터베이스는 락을 걸 수 있는 다양한 입도(크기 단위)를 제공한다.
`DB 전체 락 > Table 락 > Page(블록) 락 > Row(행) 락`

- **트레이드오프**: 
  - **Row Lock**: 병행성(동시성)은 최고지만, 메모리 자원(Lock Manager) 소모가 막대하다.
  - **Table Lock**: 락 관리는 1개만 하면 되니 편하지만, 다른 모든 사용자의 쓰기(Write)가 막혀 시스템이 멈춘다.
- **에스컬레이션 원리 (SQL Server 예시)**: 트랜잭션 하나가 단일 테이블에 대해 5,000개 이상의 행(Row)에 락을 획득하면, 옵티마이저는 메모리 보호를 위해 즉시 5,000개의 Row Lock을 해제하고 테이블 전체에 1개의 **Table Lock**을 걸어버린다.
- **치명적 부작용**: 개발자는 "나는 내 데이터만 건드렸는데 왜 옆 팀 직원의 INSERT까지 타임아웃으로 죽냐"며 황당해한다. 에스컬레이션이 발동하는 순간 그 테이블의 병행성은 0으로 수렴한다.

### 2. 교착 상태 (Deadlock) 발생 조건과 감지 (Detection) 매커니즘

데드락은 우연히 발생하는 것이 아니라, 다음의 4가지 조건(상호배제, 점유 대기, 비선점, 순환 대기)이 성립할 때 발생한다. 

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 Deadlock(교착 상태)의 순환 대기 그래프 (Wait-For)     │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │     [ 트랜잭션 1 (T1) ]                         [ 트랜잭션 2 (T2) ]     │
  │     1. Row A (계좌) 락 획득 ──┐               1. Row B (계좌) 락 획득 ──┐ │
  │                             │                                       │ │
  │     2. Row B 락 요청! (대기) ─┼──(충돌)──▶ 2. Row A 락 요청! (대기) ──┼─┘ │
  │        (T2가 놓을 때까지 무한정 대기)             (T1이 놓을 때까지 무한정 대기) │
  │                                                                  │
  │  [ 백그라운드 데드락 탐지 스레드 (Deadlock Detector) ]                   │
  │  - 동작: DB 엔진은 수 초(InnoDB 기본 1초 등)마다 내부의 Wait-For 그래프를 그림.│
  │  - 판별: 노드 간에 '사이클(Cycle)'이 형성된 것을 수학적으로 감지!               │
  │  - ⭐️ 조치 (Victim 선정): T1과 T2 중 '지금까지 수행한 연산(Undo)이 적어서 롤백│
  │    비용이 더 싼 놈'을 희생자(Victim)로 선정하여 강제로 에러를 뱉고 롤백시킴.      │
  │    -> 살아남은 다른 트랜잭션은 정상적으로 진행 완료.                          │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터베이스는 데드락을 '예방(Prevention)'하기보다, 일단 놔두었다가 터지면 '탐지 후 복구(Detection and Recovery)'하는 낙관적 정책을 주로 사용한다(예: MySQL InnoDB). 시스템 내부에 순환 대기 그래프(Wait-for Graph)를 그려서 꼬리물기가 감지되면, 무자비하게 한 명의 트랜잭션을 강제 종료(Deadlock Exception 1213)시킨다. 즉, 데드락 에러는 DB의 버그가 아니라 **시스템 전체의 붕괴를 막기 위한 DB 커널의 자위적 방어 기제**다.

+++

### 3. 분산 환경의 타임스탬프 기반 데드락 예방 기법

단일 DB가 아닌 여러 서버가 물린 분산 데이터베이스 환경에서는 중앙 통제 그래프(Wait-for)를 그리는 것 자체가 불가능하므로, OS에서 쓰이는 타임스탬프(늙은 놈/젊은 놈) 기반의 예방(Prevention) 기법을 쓴다.

- **Wait-Die (비선점, 기다리거나 죽거나)**:
  - 늙은 트랜잭션(먼저 시작함)이 젊은 놈이 쥔 락을 원하면 -> 존버(Wait)한다.
  - 젊은 트랜잭션(나중에 시작함)이 늙은 놈의 락을 원하면 -> 건방지다며 스스로 죽고(Die) 나중에 다시 시도한다.
- **Wound-Wait (선점, 부상 입히거나 기다리거나)**:
  - 늙은 트랜잭션이 젊은 놈의 락을 원하면 -> 젊은 놈을 찔러 죽여버리고(Wound, 선점) 락을 뺏는다.
  - 젊은 트랜잭션이 늙은 놈의 락을 원하면 -> 존버(Wait)하며 기다린다.

- **📢 섹션 요약 비유**: 은행에서 순번표(타임스탬프)를 뽑았을 때, 할아버지(늙은 트랜잭션)가 "내가 먼저 왔으니 네 자리 내놔"라며 젊은이를 쫓아내는 것이 'Wound-Wait'입니다. 이렇게 질서를 무조건 나이순으로 억지로 세우면 데드락 꼬리물기가 절대 생기지 않습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 락 최적화 및 회피 전략 비교

| 비교 항목 | 잠금 에스컬레이션 | 타임아웃 (Timeout) | 메시지 큐 (Kafka 등) |
|:---|:---|:---|:---|
| **설계 목적** | DB 메모리 한계 보호 | 데드락 무한 대기 탈출 | 락 경합의 아키텍처적 원천 차단 |
| **적용 계층** | DB 스토리지 엔진 자동 동작 | DB 커넥션 및 파라미터 설정 | 애플리케이션 / 인프라 아키텍처 |
| **부작용** | 동시성(Concurrency) 파멸적 저하 | 간헐적 사용자 에러 창 노출 | 시스템 복잡도 증가, 비동기 보장 |
| **근본 해결력** | 해결책이 아닌 차악의 방어 수단 | 사후 대응(수술) | **사전 예방 (백신), 현대적 표준** |

### 과목 융합 관점

- **운영체제 (OS)**: DB의 데드락은 운영체제에서 다루는 멀티 스레드의 식사하는 철학자 문제(Dining Philosophers Problem)를 트랜잭션 격리성(Isolation) 단위로 완벽하게 치환한 것이다. 해결책인 희생자(Victim) 선정 방식 역시 OS 프로세스 스케줄러의 기아 현상(Starvation) 방지 알고리즘과 융합되어 발전했다.
- **마이크로서비스 아키텍처 (MSA)**: MSA에서는 각 서비스별로 독립된 DB를 쓰기 때문에 '분산 데드락'이 터지면 원인 파악조차 안 된다. 이를 피하기 위해 애초에 **이벤트 큐(Kafka)**를 세워, 1만 명이 결제를 시도해도 락을 거는 대신 큐에 일렬로 세워서(Serialization) 순차적으로 폴링(Polling) 처리하는 락-프리(Lock-free) 아키텍처로 융합 진화했다.

- **📢 섹션 요약 비유**: 좁은 교차로(데이터)에서 차가 막히면 신호등(Lock)을 정교하게 바꾸거나 타임아웃을 거는 것보다, 아예 모든 차를 한 줄짜리 컨베이어 벨트(Kafka Queue)에 태워버려 알아서 1대씩 통과하게 만드는 것이 가장 완벽한 체증 해결법입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 이벤트 당첨금 지급 배치와 실시간 결제의 데드락 해소

- **문제 상황**: 고객(User) 테이블에 잔액을 업데이트할 때, "야간 이벤트 당첨금 지급 배치 프로그램"은 `User_ID` 1번부터 100만 번까지 오름차순(ASC)으로 UPDATE를 친다. 동시에 "실시간 고객 결제 환불 로직"은 무작위로 `User_ID`에 접근해 UPDATE를 친다. 이 두 로직이 교차하는 순간 매일 밤 수백 건의 데드락(Deadlock Victim) 에러가 발생하여 고객 항의가 폭주.
- **의사결정 (락 획득 순서의 전사적 동기화)**:
  1. 데드락의 근본 원인은 두 스레드가 서로 **'다른 순서'**로 자원을 요청하기 때문이다. (A->B, B->A).
  2. 애플리케이션(Java) 로직을 전면 수정하여, 여러 명의 유저 데이터를 한 번에 조작해야 할 때는 반드시 **`User_ID`를 미리 오름차순(Sort)으로 정렬한 뒤, 무조건 작은 번호부터 락(UPDATE)을 잡도록 강제 규약**을 만듦.
  3. 모든 트랜잭션이 동일한 방향(일방통행)으로만 락을 요청하게 되어 데드락 확률이 수학적으로 완벽히 0%가 됨.

### 도입 체크리스트
- **배치 작업 시 인덱스의 유무**: UPDATE 쿼리 조건절에 인덱스가 걸려 있지 않으면, DB는 특정 행(Row)만 잠글 수 없어 테이블 전체를 풀 스캔하며 지나가는 모든 행에 락(Table Lock)을 걸고 다닌다. 이로 인해 불필요한 데드락과 에스컬레이션이 빈발하므로, UPDATE 타겟 컬럼의 인덱스 설계는 SELECT 못지않게 치명적이다.
- **락 타임아웃 (Lock Timeout) 설정**: 무한정 대기하는 것을 막기 위해 `innodb_lock_wait_timeout` 같은 값을 50초가 아닌 3~5초로 짧게 튜닝했는가? 빠르게 실패(Fail-fast)를 던지고 앱 단에서 재시도(Retry)하는 로직이 서버 자원 고갈을 막는 최고의 방패다.
- **트랜잭션 범위 최소화**: `@Transactional` 어노테이션 안에 메일 발송, 외부 API 연동 등 느린 네트워크 작업이 포함되어 있는가? 그 느린 5초 동안 트랜잭션 락이 유지되면서 데드락과 에스컬레이션을 유발한다. 락을 쥐는 시간은 무조건 0.1초 이내로 DB 연산만 묶고, 무거운 로직은 트랜잭션 밖으로 빼내야 한다.

### 안티패턴: 데드락 발생 시 "에러 무시 (Try-Catch 빈 껍데기)" 처리
- 개발자가 데드락 Exception 로그를 보고 겁을 먹은 나머지, 원인을 찾아 인덱스나 로직 순서를 고치는 대신 Java 코드에 `try { DB 저장 } catch (DeadlockException e) { /* 로그만 남기고 무시 */ }` 처리를 해버리는 만행. 겉보기에 서버 에러는 안 나지만, 해당 고객의 결제나 회원가입 데이터가 디스크에 들어가지 않고 허공에 증발(Lost)해버려 나중에 수억 원 규모의 횡령 사고나 고객 보상 사태로 돌아온다. 데드락은 무조건 재시도(Retry)하거나 원인을 발본색원해야 하는 1급 경계 대상이다.

- **📢 섹션 요약 비유**: 수술실에서 혈관(데이터 흐름)이 막혀 경고음(데드락 에러)이 울리는데, 간호사가 시끄럽다고 스피커 전원(Try-Catch 묵살)만 꺼버리고 수술을 강행하는 것은 의료 사고를 넘어선 범죄 행위입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 락(Lock) 무관심 개발 환경 | 락 에스컬레이션/데드락 튜닝 환경 | 개선 효과 |
|:---|:---|:---|:---|
| **트랜잭션 동시성 (TPS)**| 빈번한 테이블 락 격상으로 DB 프리징 | 락 범위(Scope)와 순서 통제로 행 락 유지 | **동시 처리량(Concurrency) 10~50배 향상** |
| **시스템 장애율** | 피크 타임 시 데드락 체인으로 연쇄 마비 | 데드락 탐지기 및 일방통행 로직 설계 적용 | **Deadlock Exception 발생 빈도 0 수렴** |
| **사용자 체감 (UX)** | 쿼리 무한 대기(Spin)로 브라우저 무한 로딩 | 타임아웃과 Fail-fast 처리로 즉각 피드백 | **클라이언트(WAS) 자원(Thread) 고갈 방지 및 빠른 응답** |

### 미래 전망
- **낙관적 동시성 제어 (MVCC)의 궁극적 승리**: 물리적으로 남을 꼼짝 못 하게 묶어두는 락(Pessimistic Lock) 방식은 클라우드 분산 환경의 스케일 아웃을 가로막는 최대 적이다. 결국 데이터베이스 엔진은 락 없이 과거 스냅샷 버전을 관리하는 MVCC 구조로 전면 이행했으며, 충돌 제어의 책임은 앱 레벨의 버전(Version) 충돌 관리로 완전히 넘어오고 있다.
- **Event-Driven과 Lock-free Architecture**: 최신 고성능 도메인(주식 거래소 매칭 엔진, 대규모 티켓 예매)은 아예 RDBMS의 락킹을 신뢰하지 않는다. 모든 갱신 요청을 LMAX Disruptor 같은 인메모리 링 버퍼나 Kafka에 넣어 단일 스레드로 초당 수백만 건씩 락 없이 메모리에 순차 기록하는 아키텍처가 최정점의 표준으로 대두되었다.

- **📢 섹션 요약 비유**: 데이터베이스의 락(Lock) 관리는 인간 사회의 법과 규제와 같습니다. 억지로 통제하고 감옥에 가두는(물리적 락) 규제 만능주의에서 벗어나, 각자가 자유롭게 일하되 나중에 결과만 투명하게 검증하는(낙관적 동시성, 이벤트 기반) 방향으로 자유도가 진화하고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **동시성 제어 - 팬텀 읽기와 MVCC**: 락을 무식하게 걸 때 발생하는 데드락과 에스컬레이션을 회피하기 위해 탄생한, RDBMS 커널의 궁극적이고 혁신적인 대안 아키텍처.
- **이벤트 소싱과 CQRS 패턴**: 데드락을 유발하는 원인인 '데이터 덮어쓰기(UPDATE)' 행위 자체를 원천적으로 금지하고 오직 삽입(INSERT)만 허용하는 락-프리 아키텍처.
- **분산 트랜잭션 2PC의 한계**: 단일 DB의 데드락 문제를 넘어, 여러 대의 분산 서버 간에 락을 잡고 대기할 때 시스템 전체가 폭파하는 치명적 락킹 병목의 거시적 현상.

2. 데이터베이스 선생님은 이럴 때 가만히 두지 않고, 둘 중 한 명의 꿀밤을 때려서 쥐고 있던 펜을 강제로 뺏어버리는데(희생자 롤백), 덕분에 다른 한 명은 숙제를 끝낼 수 있답니다.
