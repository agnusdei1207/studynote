# 고가용성 분산 환경의 스플릿 브레인 방어와 Quorum 합의

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 시스템에서 마스터 노드 간의 네트워크가 단절되었을 때, 대기조(Standby) 노드가 상대방이 죽었다고 착각하여 **둘 다 자신이 마스터(Primary)라고 선언하는 치명적 뇌사 상태를 스플릿 브레인(Split-Brain)**이라 한다.
> 2. **가치**: 이를 방어하기 위해 클러스터 노드 수를 항상 '홀수'로 유지하고, 과반수의 표를 얻은 노드 묶음만이 마스터로 승격하여 트랜잭션을 처리하게 하는 **쿼럼(Quorum, 정족수) 합의 알고리즘**을 통해 데이터의 분기와 파괴를 막는다.
> 3. **융합**: 고장 났다고 판별된 구형 마스터가 뒤늦게 살아나 디스크(공유 스토리지)에 엉뚱한 데이터를 덮어쓰는 것을 막기 위해, 하드웨어 전원을 아예 강제로 차단해 버리는 **펜싱(STONITH, Shoot The Other Node In The Head)** 기법과 완벽히 융합된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **스플릿 브레인 (Split-Brain)**: 뇌가 두 개로 갈라진다는 뜻으로, HA(고가용성) 클러스터 환경에서 네트워크 단절(Network Partition)로 인해 두 개 이상의 노드가 동시에 'Active(마스터)' 상태로 작동하여 데이터가 양쪽으로 다르게 써지는 파국적 장애.
  - **쿼럼 (Quorum)**: 결정을 내리기 위해 필요한 '최소한의 과반수 노드' 투표 수.

- **💡 비유**: 스플릿 브레인은 한 배에 탄 두 명의 선장(마스터 DB)이 안개(네트워크 단절) 때문에 서로의 소리를 못 듣고, 각자 "내가 지금부터 진짜 선장이다!"라며 배의 앞머리와 뒷머리 노를 반대 방향으로 저어 배가 찢어지는 현상입니다. 쿼럼(다수결)은 이럴 때 선원(노드) 3명 중 2명의 표를 얻은 사람만 진짜 선장으로 인정하는 규칙입니다.

- **등장 배경**:
  1. **고가용성(HA)의 그림자**: 장애 발생 시 다운타임(Downtime)을 없애려고 Active-Standby 구조를 만들었으나, 서버가 죽은 게 아니라 '중간 통신 스위치'만 죽었을 때 Standby가 이를 '서버가 죽었다'고 오판하여 시스템이 분리되는 부작용이 생겼다.
  2. **복구 불가능한 데이터 파괴**: 양쪽 DB가 모두 쓰기(Write) 요청을 받아 데이터가 양갈래로 쪼개지면(Diverge), 네트워크가 복구된 후 이 두 데이터를 다시 하나로 합칠(Merge) 논리적 방법이 존재하지 않았다.
  3. **분산 합의 이론의 발전**: 노드가 늘어날수록 장애 확률도 늘어나므로, 절대 2개의 마스터가 탄생하지 않도록 하는 팩소스(Paxos), 래프트(Raft) 기반의 다수결 분산 합의 알고리즘이 엔터프라이즈의 표준으로 정착했다.

- **📢 섹션 요약 비유**: 두 명의 은행원이 한 고객의 계좌를 각각 자기 마음대로 관리하기 시작하면 은행은 파산합니다. 스플릿 브레인은 IT 인프라에서 발생하는 가장 두렵고 파괴적인 재앙입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 스플릿 브레인 (Split-Brain) 발생 시나리오

네트워크가 분리(Partition)되었을 때 왜 재앙이 일어나는지 보여준다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 네트워크 단절과 Split-Brain의 끔찍한 결과              │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 정상 상태 (Active-Standby) ]                                    │
  │  [ Node A (Master) ] ◀───(Heartbeat)───▶ [ Node B (Standby) ]    │
  │        ▲                                                         │
  │        │ (사용자 요청은 A로만 감)                                        │
  │                                                                  │
  │  [ 장애 발생 (Network Partition) ]                                 │
  │  [ Node A (여전히 Master) ] ◀── X (단절) X ──▶ [ Node B ]           │
  │                                                                  │
  │  1. Node B의 오판: "Heartbeat가 안 오네? A가 죽었군! 내가 Master로 승격한다!" │
  │  2. VIP(가상 IP) 충돌: A도 살아있고 B도 살아있음. (둘 다 자신이 Master라 주장) │
  │  3. 데이터 양방향 쓰기: 클라이언트 요청이 A와 B로 찢어져서 들어감.            │
  │     - A에는 "입금 100원", B에는 "출금 50원"이 각각 기록됨.               │
  │                                                                  │
  │  [ 네트워크 복구 후 대참사 ]                                           │
  │  - A와 B가 다시 연결되었으나, 서로 완전히 다른 데이터베이스가 되어버려 병합 불가! │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** HA 클러스터는 보통 서로 살아있는지 확인하기 위해 1초마다 심장 박동(Heartbeat) 패킷을 주고받는다. 이 케이블(또는 스위치)을 누군가 실수로 뽑아버리면, 양쪽 서버는 멀쩡히 살아있는데도 상대방이 죽었다고 생각한다. 결과적으로 하나의 시스템 안에 **두 명의 왕(Dual Master)**이 탄생하고, 고객의 결제 데이터가 양쪽으로 분산 저장되는 복구 불가능한 무결성 파괴가 발생한다.

+++

### 2. 방어 체계 1: 쿼럼 (Quorum)과 과반수 투표

스플릿 브레인을 방지하는 가장 완벽한 수학적 법칙은 **"클러스터 노드 수를 짝수가 아닌 '홀수'로 구성하고, 항상 $\frac{N}{2} + 1$ 이상의 표를 얻은 쪽만 살아남게 하는 것"**이다.

- **원리**: 노드가 3대(A, B, Tie-breaker C) 있다. 네트워크가 끊어져서 `[A]`와 `[B, C]` 그룹으로 나뉘었다고 치자.
- `A`는 혼자(1표)이므로 쿼럼(2표)을 채우지 못해 스스로 모든 데이터 쓰기를 차단하고 멈춘다(Read-only 또는 서비스 중단).
- `[B, C]` 그룹은 2표를 확보했으므로 자신들이 정당한 마스터 그룹임을 선언하고 서비스를 계속한다. 이로써 **마스터는 무조건 1개만 존재**함이 물리적으로 증명된다.
- 노드가 2개(짝수)라면 1:1로 표가 갈리기 때문에 누가 진짜인지 알 수 없다. 그래서 항상 3대, 5대의 홀수 노드 구성(ZooKeeper, etcd 등)이 필수적이다.

+++

### 3. 방어 체계 2: 펜싱 (Fencing / STONITH)

쿼럼을 잃은 구형 마스터가 미쳐 날뛰며 공유 디스크(SAN)에 데이터를 덮어쓰는(Data Corruption) 행위를 원천 차단하는 무자비한 하드웨어 격리 기술이다.

- **STONITH (Shoot The Other Node In The Head)**: "상대방 노드의 머리에 총을 쏴라"라는 뜻의 극단적인 펜싱 기법.
- **동작 방식**: 새로운 마스터(Node B)로 승격된 노드는, 기존 마스터(Node A)가 살아있을지도 모른다는 의심을 품는다. 그래서 Node B는 **서버실의 전원 제어 장치(PDU/IPMI)에 네트워크로 신호를 보내 Node A의 전원 플러그를 물리적으로 뽑아버린다(Reboot/Power Off)**. 죽은 자는 말이 없듯(Dead men tell no tales), 전원이 꺼진 A는 절대 디스크를 오염시킬 수 없다.

- **📢 섹션 요약 비유**: 쿼럼이 "3명 중 2명의 다수결로 진짜 왕을 뽑는 민주주의"라면, 펜싱(STONITH)은 "새로운 왕이 즉위하자마자 혹시 모를 반란의 싹을 자르기 위해 전임 왕의 목을 물리적으로 베어버리는 피의 숙청"입니다. 데이터 무결성을 위해서는 이 정도의 잔혹함이 필요합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Split-Brain 시나리오별 대응 아키텍처

| 해결/방어 아키텍처 | 동작 메커니즘 | 장점 | 단점 |
|:---|:---|:---|:---|
| **다수결 합의 (Quorum)** | 3대, 5대의 노드를 배치하고 (N/2)+1 대의 생존 그룹만 서비스 유지 | 스플릿 브레인을 논리적으로 완벽히 예방 | 최소 3대의 서버 운영 비용 (인프라 1.5배 증가) |
| **공유 디스크 락 (SCSI Reservation)**| 공유 스토리지 레벨에서 디스크의 소유권을 1개 노드에만 독점 부여 | 2대 노드만으로 구성 가능 | 클라우드 환경에서는 공유 블록 스토리지 사용 제약 많음 |
| **펜싱 (STONITH)** | 전원 관리 인터페이스(IPMI)로 구형 마스터의 전원 강제 차단 | 죽어가는 노드의 좀비화 및 디스크 오염 완벽 차단 | 펜싱 장비 고장 시 멀쩡한 노드를 서로 쏘아 죽이는 '자살 루프' 위험 |

### 과목 융합 관점

- **네트워크 (Network)**: 스플릿 브레인을 유발하는 것은 결국 Heartbeat 통신의 단절이다. 이를 막기 위해 실무에서는 서버 간의 크로스 케이블(직결)과 스위치를 거치는 케이블 등 '물리적 네트워크 경로를 2~3중으로 다원화(Redundancy)'하여 스위치 1대가 고장 나도 파티션이 발생하지 않도록 네트워크 계층을 융합 설계한다.
- **가상화와 클라우드 (K8s / AWS)**: 클라우드 환경에서는 물리적 전원 차단(STONITH)이 불가능하므로, AWS API를 호출하여 EC2 인스턴스를 강제로 `Stop` 시키거나(AWS Fencing Agent), 쿠버네티스의 파드(Pod) 축출(Eviction) 기법으로 융합 진화하였다.

- **📢 섹션 요약 비유**: 튼튼한 방탄조끼(네트워크 3중화)를 입는 것도 중요하지만, 조끼가 뚫렸을 때 피를 멈추는 지혈대(쿼럼)와 감염 부위를 즉시 잘라내는 메스(펜싱)가 있어야 환자(서비스)가 최종적으로 생존할 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 2-Node 클러스터에서의 아비터(Arbiter) 노드 도입

- **문제 상황**: 중소기업에서 비용을 아끼기 위해 메인 DB와 스탠바이 DB 딱 2대만으로 MySQL HA 클러스터(MHA)를 구성함. 어느 날 밤 네트워크 스위치가 1초간 재부팅되면서 두 DB가 서로 연결이 끊기고 둘 다 마스터로 승격해 버림(Split-Brain). 다음 날 아침 양쪽에 들어간 데이터 수만 건을 수동으로 짜깁기하느라 DBA가 철야함.
- **의사결정 (제3의 감시자 추가)**:
  1. 노드 3대를 띄우려면 비용이 비싸므로, 데이터를 저장하지 않는 아주 가벼운 컴퓨팅 노드인 **Arbiter(타이브레이커)**를 클라우드의 가장 싼 인스턴스로 1대 추가 띄움. (총 3대 완성)
  2. 투표권은 A, B, Arbiter가 각각 1표씩 가짐.
  3. 다시 A와 B 사이의 네트워크가 끊기더라도, A와 B는 Arbiter에게 "나 살아있어!"라고 신호를 보냄. Arbiter와 연결된 노드가 총 2표(과반수 쿼럼)를 얻어 합법적인 마스터로 승격함. 

### 도입 체크리스트
- **Tie-Breaker(아비터)의 독립적 배치**: 감시자 역할을 하는 3번째 노드(아비터/ZooKeeper)를 메인 DB와 같은 랙(Rack)이나 전원 콘센트에 꽂아두었는가? 전원이 나가는 순간 메인 DB와 감시자가 동시에 죽어버려 쿼럼이 붕괴된다. 감시자는 반드시 완전히 독립된 가용영역(AZ)이나 전원 체계에 분리 배치해야 한다.
- **스토리지 펜싱 (Storage Fencing) 설정 유무**: 전원 펜싱(STONITH)이 위험하다고 생각된다면, 최소한 디스크 레벨에서의 펜싱(예: PR, Persistent Reservation)을 설정하여 옛날 마스터가 공유 디스크(SAN)에 쓰기(Write) 명령을 내리는 순간 디스크가 "너 권한 없어!"라고 튕겨내도록 설정했는가?
- **가용성 vs 정합성 (가장 중요한 결정)**: 쿼럼이 무너진 상황(예: 3대 중 2대가 죽어서 1대만 남음)에서, 관리자 수동 개입 전까지 남은 1대의 DB를 읽기 전용(Read-only)으로라도 열어둘 것인가(가용성), 아니면 아예 고객에게 에러 창을 띄울 것인가(정합성)? 비즈니스 리더와의 협의가 필수다.

### 안티패턴: "알아서 죽겠지" (펜싱 없는 Failover 자동화)
- 펜싱 기법을 세팅하기 귀찮다는 이유로, 단순한 PING 스크립트만으로 상대방이 3초간 응답 없으면 무조건 VIP(가상 IP)를 뺏어오도록 자동화하는 스크립트. 구형 마스터 서버는 잠시 CPU 부하로 멈칫(Hang)했을 뿐 살아있는데, 신형 마스터가 VIP를 가져가면서 네트워크 상에 동일한 VIP 2개가 충돌하는 대재앙이 벌어진다. 죽인 것을 확실히 확인(펜싱)하지 않고 왕좌에 오르는 것은 쿠데타의 기본을 모르는 짓이다.

- **📢 섹션 요약 비유**: 영화에서 좀비를 쏴 죽였다고 생각하고 방심한 채 뒤돌아서면 꼭 다시 일어나서 물어뜯습니다. 반드시 머리(전원)를 완전히 박살 낸 것을 두 번 눈으로 확인(STONITH)한 뒤에 전진해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 2-Node 핑(Ping) 기반 Failover | 3-Node Quorum + STONITH 체계 | 개선 효과 |
|:---|:---|:---|:---|
| **데이터 정합성 파괴** | 스플릿 브레인 발생 시 데이터 양방향 유실 | 다수결 강제로 마스터 단일화 100% 보장 | **복구 불가능한 데이터 파편화 사고 제로화** |
| **장애 복구 자동화** | 좀비 노드 출현이 무서워 100% 수동 페일오버 | 전원 강제 차단으로 완벽한 자동 페일오버 | **새벽 시간대 DBA 개입 없이 10초 내 자동 복구(RTO)** |
| **운영 인프라 투명성**| 원인을 알 수 없는 간헐적 데이터 꼬임 | 쿼럼 붕괴 시 안전하게 스스로 시스템 락 | **예측 불가능한 뇌사 상태 방지 및 신뢰도 향상** |

### 미래 전망
- **합의 프로토콜(Raft)의 데이터베이스 내재화**: 과거에는 Pacemaker나 ZooKeeper 같은 외부 데몬 프로그램이 HA 클러스터를 밖에서 감시했다면, 최근의 분산 데이터베이스(CockroachDB, TiDB, MongoDB Replica Set)는 아예 DB 커널 코드 내부에 Raft 합의 알고리즘을 내장시켜 외부의 감시자 없이도 자체적으로 완벽한 쿼럼을 형성하고 페일오버를 수행하는 'Self-healing DB' 시대로 접어들었다.
- **무중단 서버리스 아키텍처**: 클라우드의 발전으로, 이제는 쿼럼이나 펜싱을 고객이 직접 세팅할 필요 없이, AWS나 Google Cloud의 스토리지 풀 자체가 백그라운드에서 수백 개의 복제본(Quorum Write)을 관리해 주는 완전 관리형(Managed) 인프라로 그 책임이 100% 이양되고 있다.

- **📢 섹션 요약 비유**: 스플릿 브레인 방어 기술은 컴퓨터 공학자들이 수십 년간 겪은 피눈물 나는 데이터 파괴 사고의 교훈이 집대성된 결과물입니다. 완벽한 합의(쿼럼)와 냉혹한 차단(펜싱)만이 시스템의 심장을 한 치의 오차 없이 뛰게 만듭니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **분산 트랜잭션의 난제 - Saga 패턴**: 분산된 서로 다른 서비스 간의 일관성을 맞추는 윗단(Application)의 기술이라면, 쿼럼은 동일한 DB 데이터를 복제할 때 발생하는 물리적 스토리지 단의 방어 기술이다.
- **데이터베이스 복제와 마스터-슬레이브**: 고가용성을 위해 마스터-슬레이브를 구축했을 때, 이들이 분리되었을 때 어떻게 진짜 마스터를 가려낼 것인지에 대한 해답(Quorum).
- **CAP 정리와 PACELC 정리**: 네트워크 단절(P) 시, 왜 DB가 스플릿 브레인의 딜레마에 빠지는지 증명한 시스템 공학의 절대 법칙.

## 👶 어린이를 위한 3줄 비유 설명
1. 배(데이터베이스)를 운전하는 선장(마스터)이 두 명이 되면, 서로 자기가 진짜 선장이라고 싸우다가 배가 찢어지는 사고를 **스플릿 브레인**이라고 해요.
2. 이걸 막기 위해 선원은 항상 홀수(3명, 5명)로 태워서, **다수결(쿼럼)**로 표를 많이 받은 한 사람만 진짜 선장으로 인정하는 완벽한 규칙을 만들었어요.
3. 그리고 만약 투표에서 진 가짜 선장이 고집을 피우고 조종대를 잡으려고 하면, 아예 배 밖으로 밧줄로 꽁꽁 묶어서 바다로 던져버리는 무시무시하지만 확실한 방법(**펜싱**)을 쓴답니다!
