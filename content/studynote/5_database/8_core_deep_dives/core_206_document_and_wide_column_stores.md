# 문서 저장소와 와이드 컬럼 스토어 - MongoDB, Cassandra, LSM-Tree

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 문서 저장소(Document Store)는 데이터의 계층적 유연성(JSON/BSON)에 집중하여 애자일한 개발을 돕고, 와이드 컬럼 스토어(Wide-Column Store)는 대규모 분산 쓰기 최적화(LSM-Tree)와 선형적 확장에 집중한 NoSQL 아키텍처다.
> 2. **가치**: MongoDB는 복잡한 조인(Join) 없이 한 문서 내에 연관 데이터를 임베딩(Embedding)하여 읽기 성능을 높이며, Cassandra는 마스터가 없는 링(Ring) 토폴로지와 가십 프로토콜(Gossip Protocol)을 통해 단일 장애점(SPOF)을 원천 제거한다.
> 3. **융합**: 이 두 모델은 현대 MSA 환경에서 각 서비스의 목적에 맞게 혼용되는 폴리글랏 퍼시스턴스(Polyglot Persistence)의 핵심 기둥이며, 최근 클라우드 네이티브 기반의 글로벌 분산 데이터베이스로 융합 발전하고 있다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **문서 저장소 (Document Store)**: JSON, XML, BSON과 같은 반정형(Semi-structured) 문서 형태로 데이터를 저장하며, 스키마리스(Schemaless)의 유연함과 풍부한 인덱싱을 지원하는 NoSQL이다.
  - **와이드 컬럼 스토어 (Wide-Column Store)**: 행(Row)마다 가질 수 있는 열(Column)의 개수와 이름이 모두 다를 수 있는 구조로, 대량의 데이터 쓰기와 압축에 극도로 최적화된 분산 NoSQL이다.

- **💡 비유**: 문서 저장소는 "내용물이 다 달라도 각자의 설명서가 붙어 있는 커스텀 택배 상자"와 같고, 와이드 컬럼 스토어는 "원하는 만큼 무한히 칸을 늘릴 수 있는 끝없는 엑셀 시트"와 같습니다.

- **등장 배경**:
  1. **객체-관계 임피던스 불일치 극복**: 개발자들은 코드 레벨의 객체(Object) 구조 그대로 DB에 저장하길 원했고, 이에 JSON/BSON 형태를 그대로 저장하는 문서 저장소가 각광받았다.
  2. **빅데이터의 초고속 기록(Write) 요구**: 시계열 센서나 서버 로그 등 초당 수십만 건씩 쏟아지는 데이터를 락(Lock) 없이 디스크에 기록하기 위해 기존 B-Tree 대신 쓰기에 특화된 아키텍처가 필요해졌다 (와이드 컬럼 스토어의 탄생).

- **📢 섹션 요약 비유**: RDBMS가 모든 옷을 정해진 사이즈로 맞춰 입어야 하는 기성복이라면, 문서 저장소는 내 몸에 맞게 자유롭게 변형되는 맞춤복이고, 와이드 컬럼 스토어는 수만 명이 동시에 옷을 보관해도 막히지 않는 거대한 코트룸과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. MongoDB (문서 저장소) 아키텍처

MongoDB는 BSON(Binary JSON) 포맷을 사용하며, 강력한 쿼리 기능과 레플리카 셋(Replica Set)을 통한 고가용성을 제공한다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 MongoDB 분산 아키텍처 (Sharded Cluster)            │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │     [Client App] ──▶  [ mongos (Query Router) ]                  │
  │                              │                                   │
  │        ┌─────────────────────┼─────────────────────┐             │
  │        ▼                     ▼                     ▼             │
  │  [Config Server]       [ Shard 1 ]           [ Shard 2 ]         │
  │  (메타데이터 저장)      (Replica Set)         (Replica Set)         │
  │                        - Primary             - Primary           │
  │                        - Secondary 1         - Secondary 1       │
  │                        - Secondary 2         - Secondary 2       │
  │                                                                  │
  │  - 데이터 모델링: RDBMS의 정규화(Normalization) 대신,                  │
  │    관련 데이터를 하나의 문서에 중첩(Embedding)하는 역정규화 우선.        │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** MongoDB 클러스터는 클라이언트의 쿼리를 적절한 샤드로 라우팅하는 **mongos**, 데이터의 파티션 정보를 관리하는 **Config Server**, 그리고 실제 데이터를 저장하는 여러 개의 **Shard**로 구성된다. 각 샤드는 단일 장애점 방지를 위해 **Primary-Secondary** 구조의 레플리카 셋으로 구성된다. 데이터를 저장할 때 테이블 나누어 조인(Join)하는 대신, 게시글 문서 안에 댓글 배열을 직접 집어넣는 **임베딩(Embedding) 패턴**을 사용하여 읽기 성능(디스크 I/O 1회)을 극대화한다.

+++

### 2. Cassandra (와이드 컬럼 스토어) 아키텍처

Cassandra는 마스터 노드가 없는(Masterless) 링(Ring) 아키텍처를 가지며, 노드 간 통신에 **가십 프로토콜(Gossip Protocol)**을 사용한다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │              Cassandra의 Masterless Ring 아키텍처                  │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │           [Node 1] ◀───── (Gossip) ─────▶ [Node 2]                 │
  │              │                              │                    │
  │          (Gossip)                       (Gossip)                 │
  │              │                              │                    │
  │           [Node 4] ◀───── (Gossip) ─────▶ [Node 3]                 │
  │                                                                  │
  │  - 모든 노드는 동등한 역할 (Peer-to-Peer)                           │
  │  - 파티셔너(Partitioner)가 해시 값을 통해 데이터를 링 상에 균등 분산      │
  │  - Tunable Consistency: Read/Write Quorum(정족수) 조절 가능         │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Cassandra 클러스터의 노드들은 P2P 방식으로 구성되며, 서로 상태 정보를 1초마다 무작위 노드와 교환하는 **가십 프로토콜**을 통해 전체 클러스터의 맵을 그린다. 마스터가 없으므로 병목이나 SPOF(Single Point of Failure)가 없다. 클라이언트의 요청은 아무 노드(Coordinator)나 받아서 해시 링(Hash Ring)을 계산해 해당 데이터가 있는 노드로 라우팅한다. 데이터는 설정된 복제 인자(Replication Factor)만큼 여러 노드에 동시 복제된다.

+++

### 3. 심층 동작 원리: LSM-Tree (Log-Structured Merge-Tree)

Cassandra의 압도적인 쓰기 성능의 비밀은 스토리지 엔진인 LSM-Tree에 있다. 이는 디스크의 랜덤 쓰기를 순차 쓰기로 변환하는 혁신적인 구조다.

1. **Commit Log (WAL)**: 쓰기 요청이 오면 복구를 위해 디스크 로그에 빠르게 순차 기록(Sequential Write)한다.
2. **MemTable**: 동시에 메모리 내의 구조체(MemTable)에 데이터를 업데이트한다. 여기서 쓰기 작업은 즉시 성공(ACK)을 반환한다.
3. **SSTable (Sorted String Table) Flush**: MemTable이 가득 차면, 변경 불가능한(Immutable) 파일인 SSTable 형태로 디스크에 통째로 쏟아낸다(Flush).
4. **Compaction**: 백그라운드에서 주기적으로 여러 개의 SSTable을 병합하여 중복 데이터와 삭제 마커(Tombstone)를 정리하고 디스크 용량을 최적화한다.

- **📢 섹션 요약 비유**: 문서 저장소가 서류철에 모든 문서를 예쁘게 클립으로 집어넣는 방식이라면, 와이드 컬럼 스토어는 일단 들어오는 서류를 메모장에 마구 갈겨쓴 뒤(MemTable), 퇴근할 때 한 번에 제본(SSTable)해 버리는 쓰기 최적화 방식입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### RDBMS vs Document vs Wide-Column 비교

| 비교 항목 | RDBMS | Document (MongoDB) | Wide-Column (Cassandra) |
|:---|:---|:---|:---|
| **저장 단위** | Row (고정 컬럼) | Document (JSON/BSON) | Row Key + 무한 확장의 Column |
| **스키마(Schema)** | 엄격함 (사전 정의 필수) | 유연함 (동적 추가 가능) | 유연함 (Column 단위 동적 추가) |
| **조인 (Join)** | 강력한 네이티브 Join 지원 | 제한적 지원 ($lookup), 임베딩 지향 | 미지원 (역정규화 필수) |
| **쓰기 성능** | 중간 (B-Tree 구조적 오버헤드) | 중간 (인덱스 업데이트 오버헤드) | **매우 높음 (LSM-Tree 순차 쓰기)** |
| **아키텍처** | Master-Slave 중심 | Master-Slave (Replica Set) | **Masterless (Peer-to-Peer)** |

### 과목 융합 관점

- **운영체제 (OS)**: LSM-Tree의 SSTable은 한 번 쓰이면 절대 수정되지 않는 **불변(Immutable)** 특성을 가지므로, OS 파일 시스템 레벨에서 캐싱 및 압축 효율이 극대화된다.
- **분산 시스템 이론**: Cassandra는 완벽한 분산 P2P 구조를 지향하며, CAP 이론에서 가용성(A)과 파티션 허용성(P)을 극대화한 전형적인 **AP 시스템**이다. 일관성(C)은 Quorum(정족수) 설정을 통해 개발자가 직접 튜닝(Tunable Consistency)할 수 있다.

- **📢 섹션 요약 비유**: RDBMS가 '수학의 정석'처럼 법칙이 엄격한 교과서라면, MongoDB는 그때그때 양식이 바뀌는 '자유 스케치북', Cassandra는 멈추지 않고 수만 개의 로그를 쏟아내는 '블랙박스 기록 장치'와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 메신저 시스템의 데이터베이스 분리

- **문제 상황**: 수억 명의 유저 프로필 정보와, 매일 수십억 건씩 발생하는 채팅 메시지 로그를 단일 RDBMS에서 처리하려다 트랜잭션 락과 I/O 지연으로 서비스가 마비됨.
- **의사결정**:
  1. **유저 프로필 (MongoDB)**: 유저마다 가진 속성(관심사, 기기 정보, 커스텀 세팅 등)이 제각각이므로, 스키마 유연성이 높은 Document Store에 저장하여 개발 생산성을 높임.
  2. **채팅 로그 (Cassandra)**: 쓰기 작업이 95% 이상을 차지하는 채팅 로그는 LSM-Tree 기반의 Wide-Column Store에 저장. 방 번호(Room_ID)를 파티션 키로, 시간(Time UUID)을 클러스터링 키로 설정하여 특정 방의 메시지를 시간순으로 초고속 조회할 수 있게 최적화.

### 도입 체크리스트
- **데이터 중첩도 (MongoDB)**: 연관된 데이터가 항상 같이 조회되는가? (그렇다면 임베딩 패턴이 적합). 배열의 크기가 무한히 커질 위험(Unbounded Array)은 없는가?
- **쿼리 패턴의 사전 정의 (Cassandra)**: Cassandra는 데이터 모델링 시 "어떤 쿼리를 던질 것인가"를 먼저 결정하고 그에 맞춰 테이블을 짜는 **쿼리 주도 모델링(Query-driven Modeling)**이 필수적이다. 조인이 불가능하므로 데이터 중복 저장을 두려워하면 안 된다.
- **Tombstone 관리 (Cassandra)**: 삭제가 빈번하게 발생하는가? LSM-Tree 특성상 삭제는 물리적 삭제가 아닌 '삭제 마커(Tombstone)' 기록이므로, 삭제가 너무 잦으면 읽기 성능이 급감한다.

### 안티패턴: MongoDB를 RDBMS처럼 1:N 정규화하여 설계하기
- MongoDB에서 문서를 RDBMS처럼 다루어 테이블 5개로 쪼개고 각각 저장한 뒤, 애플리케이션에서 이를 조합하려고 하면 성능이 심각하게 떨어진다. MongoDB의 철학은 '데이터가 함께 읽히면 함께 저장하라(Data that is accessed together, should be stored together)'다.

- **📢 섹션 요약 비유**: 가구 배치를 할 때, 책상과 의자를 항상 세트로 묶어두는 것(임베딩)이 밥 먹을 때마다 창고에서 의자를 꺼내오는 것(조인)보다 훨씬 편한 이치와 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | RDBMS 기반 설계 | NoSQL (Mongo/Cassandra) 분리 | 개선 효과 |
|:---|:---|:---|:---|
| **쓰기 처리량** | 초당 수천 건에서 I/O 병목 | 초당 수백만 건 지속 처리 | **Write Throughput 100배+ 향상** |
| **개발 리드타임** | DDL(ALTER TABLE) 승인 및 지연 | JSON 포맷 그대로 즉시 반영 | **데이터 모델 변경 공수 0에 수렴** |
| **가용성 유지** | 마스터 장애 시 수십 초 서비스 중단 | P2P 구조로 단일 장애점 없음 | **장애 인지 및 복구 시간 Zero화** |

### 미래 전망
- **NewSQL 및 HTAP 영향**: 최근 문서 저장소(MongoDB)에도 다중 문서 ACID 트랜잭션이 추가되고, 컬럼 스토어도 실시간 분석(OLAP) 기능이 강화되는 등 RDBMS의 장점을 흡수하며 경계가 허물어지고 있다.
- **클라우드 네이티브의 기본 인프라**: AWS DynamoDB(와이드 컬럼형), DocumentDB 등 완전 관리형 Serverless NoSQL 서비스들이 주류로 자리 잡으며, 개발자들은 인프라 관리 없이 데이터 모델링에만 집중하는 시대로 진입했다.

- **📢 섹션 요약 비유**: 문서 저장소와 와이드 컬럼 스토어는 과거 변방의 특수 무기에서 이제 현대 클라우드 IT 건축물의 가장 튼튼하고 유연한 기초 철골 구조로 완벽히 자리매김했습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **NoSQL 아키텍처 - 분산 데이터 모델**: Document와 Wide-Column 모델이 속한 상위 카테고리.
- **분산 합의 알고리즘 (Raft, Paxos)**: MongoDB 레플리카 셋의 리더 선출 과정에 적용되는 분산 이론.
- **CAP 정리와 PACELC 정리**: Cassandra의 가용성(AP) 중시 및 튜너블 일관성 설계의 사상적 배경.
- **트랜잭션 ACID 원칙**: NoSQL이 스케일 아웃을 위해 의도적으로 완화했던 RDBMS의 핵심 특성.

## 👶 어린이를 위한 3줄 비유 설명
1. **문서 저장소(MongoDB)**는 글과 사진, 스티커를 내 마음대로 붙일 수 있는 예쁜 '다이어리' 같아요. 양식이 정해져 있지 않아서 나만의 이야기를 아주 쉽고 빠르게 적을 수 있어요.
2. **와이드 컬럼 스토어(Cassandra)**는 전 세계 수천 명의 친구가 동시에 글을 써도 절대 찢어지지 않는 마법의 '초대형 칠판'이에요. 들어오는 글을 멈추지 않고 순식간에 받아 적는 데 선수랍니다.
3. 이 두 가지 요술 공책 덕분에, 카카오톡 메시지나 넷플릭스 영화 추천이 아무리 많은 사람이 몰려도 절대 느려지지 않고 우리 폰으로 날아오는 거예요!
