# 시계열 데이터베이스(TSDB) 심화 - 델타 압축과 다운샘플링

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시계열 데이터베이스(TSDB, Time Series Database)는 시간(Timestamp)을 절대적인 기본 인덱스(Primary Key)로 삼아, 센서나 서버에서 초당 수십만 건씩 쏟아지는 스트리밍 데이터를 락(Lock) 없이 순차 기록하는 데 특화된 저장 엔진이다.
> 2. **가치**: 기존 RDBMS의 극심한 삽입 부하와 엄청난 스토리지 낭비를 해결하기 위해, 페이스북의 Gorilla 논문에서 창안된 **델타-오브-델타(Delta-of-Delta) 인코딩과 XOR 압축**을 통해 데이터 용량을 1/10 수준으로 극한 압축한다.
> 3. **융합**: 보존 기간이 길어질수록 1초 단위 데이터를 1시간 단위의 평균값으로 뭉개어 디스크를 절약하는 **다운샘플링(Downsampling)** 기법과 결합하여, 마이크로서비스(MSA)와 쿠버네티스의 실시간 모니터링(Prometheus 등)을 관장하는 클라우드 네이티브의 심장으로 융합 작동한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: TSDB는 일정한 시간 간격으로 생성되는 데이터(메트릭, 주식 시세, IoT 센서 값)를 저장하고 분석하기 위해 처음부터 설계된 데이터베이스다. (대표 제품: Prometheus, InfluxDB, TimescaleDB)

- **💡 비유**: 일반 RDBMS가 백과사전에 글을 예쁘게 정렬해서 끼워 넣는(중간 삽입 발생) 방식이라면, TSDB는 두루마리 휴지에 펜을 대고 휴지가 풀리는 속도(시간 흐름)에 맞춰 무조건 끝에다가 멈추지 않고 선을 쭉쭉 그어 나가는(순차 쓰기) 심전도 기계와 같습니다.

- **등장 배경**:
  1. **데이터 쓰기(Insert) 폭발**: 수천 대의 서버 CPU 사용량이 1초마다 전송되면 초당 수백만 건의 트랜잭션이 발생한다. RDBMS의 B-Tree 인덱스는 데이터를 꽂을 때마다 페이지 분할(Split)과 디스크 랜덤 I/O를 일으켜 서버를 즉사시켰다.
  2. **업데이트(Update)의 부재**: 센서 데이터의 특성상 "과거 온도가 20도였다"는 사실을 나중에 25도로 수정(Update)하는 일은 절대 없다. 오직 과거에 데이터를 쏟아붓고 나중에 집계 분석(Read)만 하는 철저한 비대칭 워크로드 특성을 가졌다.
  3. **비용 한계**: 엄청난 양의 Float(8바이트) 숫자 데이터를 모두 원본으로 저장하려면 천문학적인 스토리지 비용이 발생하여, 극단적인 압축 알고리즘이 필수불가결해졌다.

- **📢 섹션 요약 비유**: 매일 물이 콸콸 쏟아지는 폭포수(시계열 데이터) 아래에서 일반 바가지(RDBMS)로 물을 담으면 넘쳐흐르지만, 바닥이 뚫려있어 물을 모으면서 동시에 증발(압축)시켜 소금(가치 있는 데이터)만 남기는 거대한 염전 시스템(TSDB)의 탄생입니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. TSDB의 극한 압축 원리 (Gorilla 압축 알고리즘)

TSDB의 꽃은 데이터 저장 용량을 90% 이상 깎아내는 무손실/손실 압축 수학이다. 페이스북(Meta)이 발표한 Gorilla 논문이 현대 TSDB 압축의 세계 표준이 되었다.

#### A. 타임스탬프 압축 (Delta-of-Delta Encoding)
- **원리**: 센서 데이터는 보통 1초, 10초 등 **일정한 간격**으로 들어온다.
- 시간 배열: `[T_0, T_10, T_20, T_30]` (절대 시간, 각 8Byte 소모)
- 1차 차이(Delta): `[T_0, +10, +10, +10]` (앞 시간과의 차이만 저장)
- **2차 차이(Delta of Delta)**: `[T_0, +10, 0, 0]` (차이의 차이를 저장). 대부분이 '0'이 되므로, '0'을 단 **1비트(Bit)**로 압축해 버린다.

#### B. 값(Value) 압축 (XOR Encoding)
- **원리**: 1초마다 변하는 CPU 온도(예: 25.1도 -> 25.2도)는 부동소수점(Float) 체계에서 상위 비트와 하위 비트 대부분이 변하지 않고 아주 미세한 끝부분 비트만 변한다.
- 이전 값과 현재 값을 2진수 비트 레벨에서 **XOR 연산**하면, 값이 같은 앞부분은 모두 '0'으로 날아가고 변한 뒷부분 비트만 남게 되어 엄청난 압축률을 달성한다. (숫자 1개를 약 1.37바이트로 압축)

+++

### 2. 다운샘플링 (Downsampling)과 데이터 보존(Retention) 정책

시간이 지난 시계열 데이터는 가치가 기하급수적으로 하락한다. "3달 전 수요일 오후 2시 1분 15초의 CPU 온도"는 알 필요가 없으며, "3달 전 수요일 평균 온도"만 알면 충분하다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 TSDB 다운샘플링 파이프라인 (Downsampling)             │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 최근 7일 ] (High Resolution)                                    │
  │   - 데이터: 초 단위 데이터 (초당 1건, 원본 100% 보존)                  │
  │   - 스토리지: 메모리(RAM) 및 초고속 NVMe SSD                         │
  │           │ (7일 경과 시 Background Batch 동작)                    │
  │           ▼                                                      │
  │  [ 7일 ~ 30일 ] (Medium Resolution)                                │
  │   - 데이터: 1분 단위 평균(AVG/MAX/MIN) 데이터로 압축 (용량 1/60 삭감)    │
  │   - 스토리지: 일반 SSD                                             │
  │           │ (30일 경과 시)                                         │
  │           ▼                                                      │
  │  [ 1개월 ~ 1년 ] (Low Resolution)                                  │
  │   - 데이터: 1시간 단위 평균 데이터로 압축 (용량 1/3600 삭감)             │
  │   - 스토리지: AWS S3 등 저비용 클라우드 객체 스토리지 (Cold Tier)         │
  │           │ (1년 경과 시)                                          │
  │           X (자동 영구 파기 - Retention Drop)                      │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TSDB 엔진은 백그라운드 프로세스(Continuous Query)를 통해 오래된 고해상도(High-Res) 데이터를 저해상도(Low-Res) 데이터로 요약/집계하여 다시 저장하고, 원본 데이터는 만료 정책(Retention Policy)에 의해 가차 없이 삭제(Drop)해 버린다. 이는 RDBMS의 무거운 `DELETE` 쿼리가 아니라, 파티션 파일을 OS 단위에서 통째로 날려버리는 방식이므로 I/O 부하가 전혀 없다.

+++

### 3. LSM-Tree 기반의 고속 쓰기 아키텍처

최신 TSDB(InfluxDB 3.0 등)는 디스크 I/O를 견디기 위해 B-Tree를 버리고 LSM-Tree(Log-Structured Merge-Tree) 계열의 스토리지 엔진 구조를 채택한다. 들어오는 데이터는 메모리 버퍼(MemTable)에 시간순으로 순차 정렬하며 쏟아붓고, 꽉 차면 불변의 파일(SSTable)로 디스크에 Flush 하는 극강의 쓰기(Write) 최적화 아키텍처다.

- **📢 섹션 요약 비유**: TSDB의 다운샘플링은 우리가 스마트폰 사진을 백업할 때, 최근 사진은 원본 화질(4K)로 폰에 두고, 몇 달 지난 사진은 화질을 낮춰서(압축) 구글 드라이브에 올린 뒤 폰에서는 지워버려 평생 용량 걱정 없이 사진을 찍게 해주는 자동화 시스템입니다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### RDBMS vs 일반 NoSQL vs 시계열 DB (TSDB) 심층 비교

| 비교 항목 | RDBMS (PostgreSQL) | NoSQL (MongoDB) | 시계열 DB (Prometheus, InfluxDB) |
|:---|:---|:---|:---|
| **기본 키(인덱스)** | 의미 있는 비즈니스 ID (사번 등) | Document Object ID | **절대적인 시간 (Timestamp)** |
| **워크로드 특성** | C/R/U/D 고른 분포 (OLTP) | Read 위주 또는 C/R/U/D | **99% Insert (Append Only), 대용량 범위 Read** |
| **압축 알고리즘** | 범용 압축 (LZO, Zlib) | 범용 압축 (Snappy, Zstandard)| **시계열 특화 압축 (Gorilla 델타/XOR)** |
| **과거 데이터 삭제**| 대규모 DELETE (락/Undo 폭발) | TTL 설정 통한 Background 삭제 | **Time 파티션(Chunk) 단위 통째로 1초 만에 파일 Drop** |
| **주 사용처** | 금융 거래, ERP 결제 원장 | 유연한 프로필, 콘텐츠 메타데이터 | **서버/네트워크 모니터링, IoT 센서, 주식 틱 데이터** |

### 과목 융합 관점

- **운영체제 및 인프라 (K8s 모니터링)**: TSDB는 컨테이너 오케스트레이션 환경(Kubernetes)과 융합되어 필수 불가결한 존재가 되었다. K8s의 수백 개 파드(Pod)가 1초마다 내뿜는 CPU/메모리 메트릭을 Prometheus(TSDB)가 스크랩핑(Scraping)하고, 수치가 치솟으면 이를 근거로 오토 스케일링(HPA)을 트리거하는 클라우드 생태계의 브레인 역할을 수행한다.
- **RDBMS와의 하이브리드 결합**: 기존 RDBMS의 장점(SQL 문법)을 버리지 않기 위해, PostgreSQL 위에 시계열 압축과 자동 파티셔닝(Hypertable) 모듈을 플러그인 형태로 이식한 **TimescaleDB**가 등장하여, 개발자들이 새로운 쿼리 언어(PromQL)를 배우지 않고도 TSDB의 성능을 누리게 하는 융합이 일어났다.

- **📢 섹션 요약 비유**: RDBMS가 '모든 물건을 완벽히 정리하는 서랍장'이라면, 일반 NoSQL은 '크기 상관없이 다 들어가는 대형 박스'이고, TSDB는 '끊임없이 밀려오는 컨베이어 벨트 위의 물건을 초고속으로 찍어내는 센서 기록 장치'입니다. 목적이 다르면 도구도 달라져야 합니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 자율주행 자동차 통합 관제 시스템의 텔레메트리 최적화

- **문제 상황**: 10만 대의 전기차에서 1초마다 100개의 센서 데이터(속도, 배터리, 온도)가 클라우드의 일반 NoSQL로 쏟아져 들어옴 (초당 1,000만 건 Write). NoSQL 클러스터가 디스크 I/O 경합으로 멈추고 스토리지 비용이 한 달에 수억 원 청구됨.
- **의사결정 (전용 TSDB 도입 및 압축)**:
  1. **InfluxDB 도입**: 쏟아지는 센서 데이터를 메모리에 먼저 쓰고 파일 단위로 밀어내는 LSM-Tree 기반 TSDB로 교체하여 초당 1,000만 건 쓰기를 서버 몇 대로 방어함.
  2. **Gorilla 압축 적용**: 속도가 시속 100km로 유지되는 동안의 데이터는 Delta 값이 '0'이므로 극단적 비트 압축이 일어나 스토리지 용량을 1/20로 절감.
  3. **Continuous Query (다운샘플링)**: 차량이 주행을 마친 하루 뒤에는 센서 데이터를 1분 단위 평균값으로 요약하여 보존(Retention)하고 원본 초 단위 데이터는 폐기하여 클라우드 비용을 최종적으로 95% 절감함.

### 도입 체크리스트
- **데이터의 불변성 (Immutability)**: 수집하는 데이터가 나중에 수정(Update)될 확률이 0.1%라도 있는가? (예: 어제 입력된 센서 값에 오류가 있어 오늘 덮어써야 함). TSDB는 구조상 Update 연산이 극악의 효율을 내거나 아예 지원하지 않는다. 수정이 잦다면 RDBMS나 일반 NoSQL을 써야 한다.
- **시계열 질의 지원 (Time-Window Query)**: "최근 1시간 동안의 5분 단위 이동 평균선(Moving Average)을 그려라" 같은 복잡한 시계열 쿼리를 네이티브 함수(`time_bucket`, `window`)로 지원하는가? TSDB는 이 연산을 C/Rust 커널 레벨에서 병렬로 처리해 주어 애플리케이션 단의 부하를 0으로 만든다.
- **카디널리티 폭발 (High Cardinality) 경계**: TSDB에 데이터를 넣을 때 메타데이터(Tag/Label)로 `사용자 IP`나 `난수 UUID`를 넣으면, TSDB 내부에 관리해야 할 시계열 인덱스 트리가 수억 개로 폭발(Cardinality Explosion)하여 램(RAM)이 터져버린다. Tag에는 반드시 종류가 적은 값(예: 서버명, 지역코드)만 넣어야 한다.

### 안티패턴: 파티셔닝을 수동으로 관리하는 RDBMS 억지 사용
- TSDB 도입을 꺼려서 기존 MySQL에 데이터를 쏟아붓고, 매일 자정마다 DBA가 `CREATE PARTITION` 스크립트를 짜서 내일짜 테이블 파티션을 수동으로 생성하고 오래된 파티션을 `DROP`하는 유지보수 지옥 패턴. 스크립트 에러로 파티션 생성이 실패하는 순간 다음 날 오전의 센서 데이터가 전부 허공으로 증발하는 끔찍한 인재(Human Error)를 낳는다. TSDB의 Hypertable 자동 파티셔닝 기능에 맡겨야 한다.

- **📢 섹션 요약 비유**: 100m 달리기 기록(시계열 데이터)을 잴 때, 공책(RDBMS)에 손으로 쓰는 것을 고집하다 보면 선수가 결승선을 지나가는 순간을 놓치게 됩니다. 스톱워치와 자동 기록기(TSDB)라는 전용 기계를 도입하는 것이 문명의 이기입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 일반 RDBMS (인덱스/파티셔닝) | 전용 TSDB (델타 압축/LSM 적용) | 개선 효과 |
|:---|:---|:---|:---|
| **쓰기 처리량 (Ingest Rate)** | 초당 수천~수만 건 한계 (인덱스 락 병목) | 초당 수십만~수백만 건 처리 보장 | **시계열 쓰기 성능(Write TPS) 10배~100배 극대화** |
| **스토리지 저장 용량** | 원본 데이터 8Byte 그대로 디스크 보관 | Gorilla 등 비트 압축으로 1.3Byte 수준 축소 | **클라우드 스토리지 유지 비용 80~90% 영구 절감** |
| **장기 보존 성능 관리** | 1년 치 테이블 조회 시 메모리 OOM 발생 | 1분/1시간 단위 다운샘플링 데이터 즉시 응답 | **장기 트렌드 분석 쿼리 응답 속도 밀리초(ms) 달성** |

### 미래 전망
- **객체 스토리지 완전 통합 (Serverless TSDB)**: InfluxDB 3.0 등 최신 아키텍처는 비싼 로컬 SSD 대신, 무한한 확장이 가능한 AWS S3 위에 데이터를 직접 Parquet 포맷으로 압축하여 저장하고 쿼리 엔진만 분리해서 띄우는 완전한 서버리스(Serverless) 분석 플랫폼으로 아키텍처를 뒤집고 있다.
- **AI 융합 시계열 예측 (AIOps)**: TSDB에 쌓인 매트릭을 단순 대시보드에 그리는 것을 넘어, 머신러닝 엔진이 TSDB에 직접 붙어 "현재 메모리 증가 추세를 볼 때 3시간 뒤에 서버가 다운될 확률 99%"라고 미래를 선제 타격(Predictive Maintenance)하는 AIOps의 핵심 코어가 될 것이다.

- **📢 섹션 요약 비유**: 시계열 데이터베이스(TSDB)는 과거의 데이터를 차곡차곡 쌓아두는 '창고'에서 진화하여, 끊임없이 박동하는 기업의 인프라와 비즈니스 심장 박동(Heartbeat)을 1밀리초 단위로 감시하고 심정지를 예측하는 가장 위대한 중환자실 모니터 기계입니다.

+++

## 📌 관련 개념 맵 (Knowledge Graph)
- **최신 데이터베이스 최적화 (LSM-Tree, Tiering)**: TSDB가 초당 수백만 건의 쓰기를 견딜 수 있도록 차용한 가장 근본적인 스토리지 엔진 내부의 쓰기 최적화 및 계층형 구조.
- **고급 확률적 자료구조 (Bloom Filter)**: 무수히 많은 시계열 파일 조각(Chunk) 중에서 원하는 시간대의 데이터를 찾을 때, 디스크 I/O를 막기 위해 TSDB 헤더에 심어두는 수학적 문지기.
- **데이터옵스와 실시간 스트림 처리**: IoT 센서에서 발생한 데이터가 Kafka(메시지 큐)와 Flink(스트림 처리)를 거쳐 최종적으로 불시착하는 영구적 종착지가 바로 TSDB다.

## 👶 어린이를 위한 3줄 비유 설명
1. 1초마다 변하는 온도계 숫자를 공책에 일일이 다 적으려면 손이 너무 아프고 공책도 금방 다 써버려요. (RDBMS의 한계)
2. **시계열 데이터베이스(TSDB)**는 똑똑해서, 처음 온도만 적고 그 다음부터는 "아까보다 +1 올랐음, 0 올랐음" 이렇게 차이점(델타 압축)만 아주 작게 적어서 공책 한 권을 10년 동안 쓸 수 있게 해줘요!
3. 그리고 1년이 지난 옛날 온도는 1초마다 알 필요 없으니까, "작년 여름 평균 온도는 30도"라고 뭉뚱그려(다운샘플링) 요약해 놓고 자잘한 기록은 싹 지워버리는 완벽한 정리의 달인이랍니다.
