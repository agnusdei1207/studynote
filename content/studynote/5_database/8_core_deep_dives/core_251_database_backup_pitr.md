# 데이터베이스 백업 및 복구 아키텍처 심화 - PITR과 이중화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 백업 및 복구 아키텍처는 하드웨어 파괴(Crash)나 운영자의 치명적 실수(Drop Table) 등 모든 재난 상황으로부터 데이터를 안전하게 보호하고, 원하는 특정 시점(Point)으로 데이터를 되살려내는 **비즈니스 연속성(BCP)**의 최종 방어선이다.
> 2. **가치**: 풀 백업(Full Backup)과 증분 백업(Incremental Backup)에 트랜잭션의 아카이브 로그(Archive Log)를 융합하여, 장애 발생 1초 전의 완벽한 상태로 되돌리는 **PITR (Point-In-Time Recovery)** 능력을 제공함으로써 데이터 유실률(RPO)을 0에 가깝게 수렴시킨다.
> 3. **융합**: 과거의 느린 테이프 물리 백업을 넘어, 현대 클라우드 인프라에서는 스토리지 블록 단위의 스냅샷(Snapshot)과 이중화(Replication) 기술이 융합되어 수 테라바이트(TB) 급 DB를 수 분 내에 복구(RTO 단축)해 내는 클라우드 네이티브 복원력(Resilience)으로 진화했다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **백업 (Backup)**: 데이터를 안전한 외부 스토리지(테이프, S3 등)에 정기적으로 복사해 두는 행위.
  - **복구 (Recovery)**: 장애 발생 시 백업 파일과 로그를 조합하여 데이터베이스를 정상 상태로 되돌리는 행위.
  - **PITR (Point-In-Time Recovery)**: 단순히 "어젯밤 12시"로 돌아가는 것이 아니라, "오늘 오후 2시 15분 33초(장애 1초 전)"와 같이 분/초 단위의 특정 시점으로 완벽하게 롤포워드(Roll-forward) 복원하는 최상위 복구 기법.

- **💡 비유**: 백업이 매일 밤 일기장을 통째로 복사본(Full)으로 떠두고 변경된 문장만 포스트잇(증분)으로 붙여두는 작업이라면, PITR 복구는 "어제 일기장 복사본을 가져온 뒤, 오늘 아침부터 내가 쓴 글(아카이브 로그)을 순서대로 똑같이 다시 적어나가다가 실수로 잉크를 쏟기 1초 전 시점에서 딱 펜을 멈춰 완벽한 일기장을 되살리는" 타임머신 기술입니다.

- **등장 배경**:
  1. **휴먼 에러 (Human Error)**: DBA나 개발자가 `WHERE` 조건 없이 `UPDATE`를 치거나 운영 DB의 테이블을 `DROP` 해버리는 치명적 실수는 하드웨어 이중화(HA)나 복제(Replication)로는 절대 막을 수 없다. (잘못된 명령도 0.1초 만에 똑같이 복제되어 모두 파괴되기 때문)
  2. **랜섬웨어 및 침해 사고**: 해커가 데이터를 암호화하고 인질극을 벌일 때, 망분리된 안전한 공간(오프사이트)에 보관된 '불변의 오프라인 백업본'만이 회사를 파산에서 구할 유일한 수단이 되었다.
  3. **RTO / RPO의 비즈니스적 요구**: "장애 시 1시간 이내에 서비스를 살려라(RTO)", "데이터 유실은 5분을 넘기지 마라(RPO)"라는 비즈니스 SLA 규정이 법제화되면서 고도화된 복구 아키텍처가 필수적이게 되었다.

- **📢 섹션 요약 비유**: 복제(Replication)가 칼에 베였을 때 즉시 피를 멈추는 '응급처치'라면, 백업과 복구(PITR)는 치명상을 입고 죽은 사람을 어제 멀쩡했던 상태로 되살려내는 '부활 마법'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 백업의 3대 방식 (Full, Differential, Cumulative)

데이터를 어떻게 백업 매체로 옮길 것인가에 대한 전략이다.

| 백업 방식 | 동작 원리 | 복구 속도 (RTO) | 스토리지 비용 및 백업 시간 |
|:---|:---|:---|:---|
| **Full Backup (전체 백업)** | 매번 데이터베이스 전체를 통째로 복사 | **가장 빠름** (파일 하나만 복원하면 끝) | **가장 큼** (매일 1TB 복사 시망) |
| **Differential Incremental<br>(차등 증분 백업)** | 가장 최근의 '어떤 백업(Full이든 증분이든)' 이후에 변경된 데이터만 백업 | 느림 (Full 복원 후, 월/화/수/목 증분 파일을 순서대로 덧붙여야 함) | **가장 작음** (변경분만 매일 저장) |
| **Cumulative Incremental<br>(누적 증분 백업)** | 가장 최근의 **'Full 백업'** 이후에 변경된 모든 데이터를 누적해서 백업 | 중간 (Full 복원 후, 가장 최근 누적 백업본 1개만 덧붙이면 됨) | 중간 (월요일보다 금요일 백업 파일이 더 커짐) |

+++

### 2. 완벽한 복원 매직: 아카이브 로그 (Archive Log)와 PITR

"매일 밤 12시"에 백업을 받는데, "오늘 낮 3시"에 데이터베이스가 파괴되었다. 어제 밤 12시로 되돌아가면 오늘 낮에 고객들이 주문한 결제 데이터가 날아간다. 이를 구원하는 것이 **아카이브 로그**다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 PITR (Point-In-Time Recovery) 동작 원리             │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 시간의 흐름과 백업 생성 ]                                           │
  │  (일) 00:00 ──▶ [ Full Backup ] 생성 완료                           │
  │  (월) 00:00 ──▶ [ Incremental Backup ] 생성 완료                    │
  │  (화) 00:00 ──▶ [ Incremental Backup ] 생성 완료                    │
  │  ※ 동시에 모든 트랜잭션 기록은 [Archive Log 1, 2, 3...] 파일로 연속 저장 중 │
  │                                                                  │
  │  [ (화) 오후 3:00 - 치명적 💥DROP TABLE💥 실수 발생! ]                 │
  │                                                                  │
  │  [ 복구 파이프라인 (RMAN / XtraBackup) ]                             │
  │  1. 현재 터져버린 DB 인스턴스 정지 및 데이터 파일 파기.                      │
  │  2. (일) 00:00 의 [ Full Backup ] 파일 복원.                        │
  │  3. (월), (화) 00:00 의 [ Incremental Backup ] 파일 덮어쓰기 (롤포워드).  │
  │     (이제 DB는 화요일 자정 상태로 돌아옴)                                │
  │  4. 화요일 자정부터 발생한 [ Archive Log ]들을 가져와 순차적으로 재실행(Redo).│
  │  5. ⭐️ 로그를 실행하다가 [오후 2:59:59] 시점에서 "STOP!" 명령을 내림.      │
  │  6. 복구 완료 후 DB 오픈. (고객 주문 데이터는 살리고, Drop Table은 취소됨!) │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터베이스가 작동하는 동안 메모리 버퍼의 변경 사항은 Redo Log(WAL)에 쓰인다. 이 Redo Log가 꽉 차면 덮어써지기 전에 다른 하드디스크나 S3로 복사해 두는데, 이것이 **아카이브 로그(Archive Log)**다. PITR은 과거의 베이스캠프(Full Backup)에서 출발하여 아카이브 로그라는 '발자국'을 그대로 따라 걷다가(Roll-forward), 절벽(장애 발생 시간) 바로 1초 전에서 딱 멈추는 고도의 복구 기술이다.

+++

### 3. 클라우드 스냅샷 (Storage-level Snapshot) 백업

전통적인 방식은 DB 엔진이 수 테라바이트의 파일을 일일이 읽어(Export) 네트워크로 쏘는 방식이라 디스크 I/O 부하가 극심하고 백업에만 10시간이 넘게 걸렸다.

- **스냅샷 원리**: 클라우드 벤더(AWS EBS, 스토리지 SAN)의 하드웨어/볼륨 컨트롤러 레벨에서 포인터 매핑만 복사하는 **COW (Copy-On-Write)** 방식이나 **ROW (Redirect-On-Write)** 방식을 사용하여, 10TB짜리 DB도 **단 1초 만에** 백업(스냅샷)이 완료된다.
- **장점**: 백업 윈도우(Backup Window, 야간 점검 시간)가 아예 필요 없어지며, DB CPU에 전혀 부하를 주지 않는다.

- **📢 섹션 요약 비유**: 과거의 백업이 100만 자짜리 소설을 밤새 손으로 펜을 들고 똑같이 필사하는 고된 노동이었다면, 스냅샷 백업은 그냥 소설책을 스마트폰 카메라로 '찰칵' 찍어두고 끝내는 최첨단 기술입니다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### 재해 복구 시스템: 복제(Replication) vs 백업(Backup)

이 둘을 혼동하는 것은 아키텍트의 가장 크고 흔한 실수다. 두 기술은 완벽히 상호 보완적으로 융합되어야 한다.

| 비교 항목 | 복제 및 HA (Replication, Master-Slave) | 백업 및 복구 (Backup, PITR) |
|:---|:---|:---|
| **주요 방어 대상**| 디스크 파손, 서버 다운, IDC 전원 차단 (물리적 장애) | 개발자의 DROP TABLE, 랜섬웨어 암호화 (논리적 장애) |
| **데이터 전파 속도**| 밀리초(ms) 단위의 실시간 즉시 복사 | 야간 1회 배치 또는 스냅샷 주기 |
| **약점 (위험성)** | 마스터의 '치명적 실수'도 슬레이브에 0.1초 만에 똑같이 적용되어 공멸함 | 복구하는 데 수 시간(RTO) 소요. 최신 1초 전 데이터는 유실됨 |
| **비즈니스 용어** | 고가용성 (High Availability) | 비즈니스 연속성 계획 (BCP / DR) |

### 과목 융합 관점

- **데이터베이스 로그 및 ARIES 복구** : 백업(PITR)의 뼈대가 되는 아카이브 로그는 결국 DB 커널이 트랜잭션의 영속성(Durability)을 지키기 위해 메모리에서 디스크로 내리는 WAL(Write-Ahead Logging) 파일의 물리적 보관소(Cold Storage) 버전이다. ARIES의 Redo 사상이 스토리지 백업 레이어로 확장된 것이다.
- **운영체제와 파일 시스템**: DB 복구 시 데이터를 디스크에 무작정 붓는다고 끝나는 것이 아니라, `fsync` 함수를 통한 OS 레벨의 캐시 동기화 보장과 볼륨 매니저(LVM)의 스냅샷 결속(Consistency) 상태가 완벽히 융합되어야만 '깨지지 않은 정합성 있는 백업본'이 생성된다.

- **📢 섹션 요약 비유**: 복제(Replication)는 스페어 타이어를 달고 달리는 것입니다. 펑크가 나면 바로 달릴 수 있죠. 하지만 차를 절벽 아래로 몰아버리는 미친 운전자(휴먼 에러)를 만나면 스페어 타이어도 같이 터집니다. 이때 시간을 되돌려 차를 다시 절벽 위로 올려놓는 마법이 백업(Backup)입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 랜섬웨어(Ransomware) 감염 시의 데이터 인질극 방어

- **문제 상황**: 회사 내부망이 랜섬웨어에 뚫려, 운영 DB 파일은 물론이고 같은 망 내에 엮여있던 백업 서버의 파일들마저 모조리 암호화되어 해커가 10억 원을 요구함.
- **의사결정 (오프사이트 및 불변 백업 아키텍처 도입)**:
  1. **3-2-1 백업 규칙 적용**: 3개의 복사본을, 2개의 서로 다른 매체(디스크, 테이프/S3)에 보관하고, 1개는 반드시 회사 망과 물리적/논리적으로 완전히 단절된 **오프사이트(Off-site, 예: AWS S3)**에 분리 보관.
  2. **객체 잠금 (Object Lock / WORM)**: S3에 백업 파일(Full 백업 + Archive Log)이 올라가면, AWS의 WORM(Write Once Read Many) 기능을 설정하여 AWS 최고 관리자 계정(Root)을 탈취하더라도 설정된 기간(예: 30일) 내에는 절대 파일을 수정하거나 삭제할 수 없게 물리적으로 락을 검.
  3. 해커가 랜섬웨어를 돌리거나 파일을 삭제하려 시도해도 스토리지 자체에서 튕겨내어 100% 무사히 데이터 복원 완료.

### 도입 체크리스트
- **RTO(복구 목표 시간)와 RPO(복구 목표 시점)의 수립**: 경영진과 합의된 명확한 수치가 있는가? "데이터는 1시간 분량(RPO=1H)까진 날아가도 되지만, 서버는 무조건 10분 내(RTO=10M)에 무조건 띄워라"라는 합의가 있어야만 백업 솔루션(테이프를 살지, 비싼 AWS 스냅샷을 쏠지)의 예산을 결정할 수 있다.
- **복구 모의훈련 (Fire Drill)**: 백업 스크립트를 짜놓고 'Success' 로그가 떨어진다고 안심하는 것은 최악의 안티패턴이다. 막상 장애가 터져서 백업 파일을 풀어보면 데이터 블록이 깨져있어(Corrupted) 복구 실패하는 사례가 허다하다. 분기에 1회 이상, **아무도 없는 가상 서버에 백업 파일을 실제로 Restore(복원) 해보는 자동화된 모의 훈련**이 필수적이다.
- **아카이브 로그 용량 폭발**: 아카이브 모드(Archive Log Mode)를 켜두면 DML(Insert/Update/Delete) 트랜잭션이 발생할 때마다 로그 파일이 디스크에 쌓인다. 연휴 기간 배치 작업으로 인해 아카이브 디스크가 100% 꽉 차면, **DB는 스스로 작동을 멈추고 뻗어버린다(Hang)**. 아카이브 삭제 스케줄러가 견고하게 세팅되었는가?

### 안티패턴: 논리적 백업(mysqldump / Data Pump)으로 수 테라바이트(TB) 백업하기
- 10GB 이내의 작은 DB에서나 쓰던 `mysqldump` (데이터를 SQL INSERT 문장 텍스트로 뽑아내는 방식)를 10TB짜리 메인 DB에 거는 행위. 텍스트로 데이터를 뽑아내는 동안 DB 엔진은 버퍼 풀에 데이터를 모조리 올리느라 메모리가 파탄 나고, 백업에만 3일이 걸린다. 만약 복구(Restore)를 하려면 이 수백억 줄의 INSERT 쿼리를 한 땀 한 땀 실행해야 하므로 RTO가 일주일 이상 걸린다. 대용량 환경에서는 무조건 RMAN, XtraBackup 같은 물리적(Physical) 블록 카피 기반의 백업을 써야 한다.

- **📢 섹션 요약 비유**: 이삿짐(대용량 데이터)을 쌀 때, 냉장고와 TV의 분자 구조(논리적 백업)를 수첩에 적어갔다가 새집에서 재조립하는 짓은 미친 짓입니다. 이삿짐센터 트럭을 불러 통째로 실어 나르는 것(물리적 백업)이 대용량 시스템의 유일한 상식입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 레거시 논리 백업 (Export/Import) | 물리적 증분 백업 및 PITR 융합 | 개선 효과 |
|:---|:---|:---|:---|
| **데이터 유실 (Data Loss)** | 어젯밤 자정 기준으로 롤백 (최대 24시간 유실) | 아카이브 로그 추적으로 장애 1초 전 복원 | **비즈니스 핵심 데이터 유실률(RPO) 0 수렴** |
| **복구 소요 시간 (RTO)**| 텍스트 SQL INSERT 구문 재실행으로 수일 소요 | 스냅샷 마운트 및 블록 레벨 복구로 즉시 기동 | **복구 시간(Downtime) 90% 이상 획기적 단축** |
| **DB 운영 성능 부하** | 백업 중 테이블 락 발생 및 버퍼 풀 오염 | 스토리지 레벨 분리(COW)로 무중단 백업 | **주간 백업 시에도 트랜잭션 성능(TPS) 100% 유지** |

### 미래 전망
- **CDP (Continuous Data Protection, 지속적 데이터 보호)**: 하루 1번, 1시간 1번 하던 스냅샷 백업 주기를 넘어, 모든 디스크 I/O 트랜잭션을 하이퍼바이저 레벨에서 밀리초(ms) 단위로 가로채어 다른 스토리지에 무한정 저널링(Journaling)하는 기술. 복구 시 유튜브 영상을 되감듯 원하는 0.001초의 스크롤 바로 정확히 되돌아갈 수 있는 초정밀 기술이 금융권을 중심으로 자리 잡고 있다.
- **서버리스 (Serverless) 데이터베이스의 백업 추상화**: AWS Aurora나 DynamoDB는 백업 인프라와 스크립트를 관리할 필요조차 없이, 사용자가 콘솔에서 "3일 전 오후 2시로 되돌린 새 DB 클러스터를 띄워줘(Clone)"라고 클릭 한 번만 하면 10초 만에 완벽히 복원된 인스턴스가 팝업되는 완전 관리형 복원 시스템을 제공한다.

- **📢 섹션 요약 비유**: 데이터베이스의 백업과 PITR 복구 기술은 인간이 만든 가장 든든한 '생명보험이자 타임머신'입니다. 평소에는 쓸 일이 없어 비용처럼 느껴지지만, 한 번의 치명적 절벽에서 기업의 목숨을 살려내는 것은 화려한 AI 모델이 아닌 구석에 조용히 쌓여있는 로그 백업 파일 하나입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **버퍼 풀과 WAL 로깅 (ARIES)**: PITR이 과거 상태를 1초 단위로 복원(Redo)할 수 있게 해주는 가장 근원적인 데이터베이스 커널 동작 원리(Archive Log의 기원).
- **데이터베이스 복제와 고가용성 (HA)**: 백업과는 역할이 다르지만(물리적 파괴 vs 논리적 파괴), 기업의 비즈니스 연속성(BCP)을 지키기 위해 백업 아키텍처와 반드시 1:1로 결합하여 설계되어야 하는 양대 산맥.
- **트랜잭션 락킹과 동시성 제어**: 과거 논리적 백업 시 테이블 전체에 락(Lock)이 걸리던 문제를, MVCC(다중 버전) 기술을 통해 락 없이 스냅샷 백업이 가능하도록 지원해 준 일등 공신.

## 👶 어린이를 위한 3줄 비유 설명
1. 게임을 하다가 갑자기 죽으면 처음부터 다시 해야 돼서 너무 화가 나죠? 그래서 '어제 완료한 스테이지(전체 백업)'에 세이브 파일을 만들어 둬요.
2. 하지만 오늘 아침부터 하루 종일 몬스터를 잡고 레벨업을 한 뒤에 죽었다면, 어제로 돌아가면 너무 억울하잖아요! 그래서 게임기가 1분마다 내가 움직인 발자국(아카이브 로그)을 몰래 자동 녹화해 둬요.
3. **PITR (특정 시점 복구)**은 이 녹화된 발자국을 빨리 감기로 틀어서(롤포워드), 내가 실수로 용암에 빠지기 딱 1초 전으로 완벽하게 시간을 되돌려(타임머신) 살려주는 궁극의 부활 마법이랍니다!
