# 인덱스 튜닝 심화 - 결합 인덱스와 인덱스 스킵 스캔

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 두 개 이상의 컬럼을 조합하여 생성하는 **결합 인덱스(Composite Index)**는 컬럼의 나열 순서에 따라 B-Tree의 물리적 정렬 기준이 완전히 달라지므로, 쿼리 조건절(`WHERE`)의 성격(점 검색 vs 범위 검색)에 따라 튜닝의 성패가 극명하게 갈린다.
> 2. **가치**: 선행 컬럼이 쿼리 조건에서 누락되면 인덱스가 무용지물이 되는 B-Tree의 구조적 한계를 극복하기 위해, 옵티마이저가 선행 컬럼의 고유 값(Distinct Value)들을 논리적으로 건너뛰며 탐색하는 **인덱스 스킵 스캔(Index Skip Scan)** 최적화가 작동하여 극적인 성능 구원을 이뤄낸다.
> 3. **융합**: 결합 인덱스 설계는 비용 기반 옵티마이저(CBO)의 실행 계획 산정 및 디스크 I/O를 원천 차단하는 커버링 인덱스(Covering Index) 아키텍처와 융합되어, 대규모 트래픽 환경의 데이터베이스 성능을 지배하는 가장 강력한 무기로 쓰인다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **결합 인덱스**: `CREATE INDEX idx_emp ON EMP(부서, 성별, 나이)` 처럼 여러 컬럼을 묶어 만든 인덱스. 첫 번째 컬럼(부서)으로 정렬된 상태에서, 같은 부서 내에서 두 번째 컬럼(성별)으로 정렬되는 **계층적 정렬 구조**를 갖는다.
  - **인덱스 스킵 스캔**: 결합 인덱스의 선행 컬럼(부서)이 `WHERE` 조건에 없더라도, DB 엔진이 선행 컬럼의 고유 값(종류)을 뽑아내어 마치 선행 컬럼이 조건에 있는 것처럼 우회 탐색하는 기법.

- **💡 비유**: 결합 인덱스는 '전화번호부'입니다. 먼저 '지역별(서울, 부산)'로 정렬되고, 그 안에서 '가나다순(이름)'으로 정렬됩니다. 만약 지역을 모르고 "홍길동"만 찾는다면 원래는 책 전체를 다 뒤져야 하지만, 인덱스 스킵 스캔은 똑똑하게 "서울 펴서 홍길동 찾고 -> 부산 펴서 홍길동 찾고" 식으로 대분류를 건너뛰며 스캔하는 기술입니다.

- **등장 배경**:
  1. **단일 인덱스의 한계**: `나이 > 20` 조건과 `성별 = '남'` 조건이 있을 때, 각각 단일 인덱스 2개를 만들면 옵티마이저가 하나만 타거나 두 인덱스를 병합(Index Merge)하느라 비효율이 발생했다.
  2. **B-Tree의 좌측 우선 정렬(Left-most Prefix) 약점**: 결합 인덱스는 반드시 선행 컬럼부터 조건에 주어져야만 트리를 탈 수 있다. 개발자가 선행 컬럼 조건을 빼먹으면 막대한 테이블 풀 스캔이 발생해 DB가 마비되었다.
  3. **인덱스 폭증 문제**: 화면 검색 조건이 다양해지자 모든 경우의 수마다 인덱스를 만들다 보니, INSERT 성능이 파멸적으로 느려졌다. 인덱스 개수를 최소화하면서도 다양한 조건을 커버할 '스킵 스캔' 등의 옵티마이저 고도화가 요구되었다.

- **📢 섹션 요약 비유**: 자물쇠 3개(A, B, C 컬럼)가 순서대로 채워진 문을 열 때, 1번 열쇠(A 조건)가 없으면 아예 문을 못 여는 것이 기존 인덱스의 맹점이었다면, 스킵 스캔은 마스터키로 1번 자물쇠를 무력화하고 2번부터 풀고 들어가는 첨단 도둑질(튜닝) 기술입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 결합 인덱스 컬럼 순서 결정의 절대 원칙

인덱스 설계 시 컬럼을 나열하는 순서는 성능의 99%를 좌우한다.

1. **1순위 (Equal 조건 = )**: 연산자가 `=` 이거나 `IN` 인 컬럼을 가장 앞에 둔다. (점을 찍어 스캔 범위를 극단적으로 줄임)
2. **2순위 (Range 조건 <, >, BETWEEN, LIKE)**: 범위 검색 컬럼은 뒷순위로 미룬다. **(범위 검색 조건이 나오는 순간, 그 뒤에 있는 인덱스 컬럼은 스캔 범위를 줄이는 데 아무런 역할을 하지 못하고 단순 필터용으로 전락한다.)**
3. **3순위 (카디널리티 / 선택도)**: `=` 조건이 여러 개라면, 데이터 종류가 많아 식별성이 높은 컬럼(예: 주민번호)을 성별(남/여)보다 앞에 두는 것이 B-Tree 분기 탐색 효율 상 유리하다.

+++

### 2. 인덱스 스킵 스캔 (Index Skip Scan)의 동작 원리

결합 인덱스 `[성별, 연령]`이 있을 때, `WHERE 연령 = 30` 이라는 조건(선행 컬럼 '성별' 누락)이 들어오면 발생하는 매커니즘.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 인덱스 스킵 스캔(Index Skip Scan) 논리 트리            │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ B-Tree 결합 인덱스: (성별, 연령) ]                                 │
  │                                                                  │
  │                       Root Node                                  │
  │                       /       \                                  │
  │            (성별=남) /         \ (성별=여)                        │
  │                    ▼           ▼                                 │
  │             Branch (남)     Branch (여)                          │
  │             /    |    \      /    |    \                         │
  │           10대  20대  30대  10대  20대  30대                       │
  │                                                                  │
  │  [ Query: WHERE 연령 = 30 ]                                      │
  │  1. 옵티마이저가 인덱스의 선행 컬럼 '성별'의 종류가 '남', '여' 2개뿐임을 파악. │
  │  2. 내부적으로 쿼리를 2개로 쪼갬 (IN 조건처럼 변환):                   │
  │     (WHERE 성별='남' AND 연령=30) OR (WHERE 성별='여' AND 연령=30)     │
  │  3. 남자의 10대, 20대 리프 노드는 쳐다보지도 않고 스킵(Skip) ──▶ 남자 30대 Hit │
  │  4. 여자의 10대, 20대 리프 노드도 스킵(Skip) ───────────────▶ 여자 30대 Hit │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 원래대로라면 선행 컬럼인 '성별'이 없어 인덱스를 사용할 수 없고 테이블 전체를 뒤져야(Table Full Scan) 하지만, 오라클(9i 이후)이나 MySQL(8.0)의 옵티마이저는 선행 컬럼의 Distinct Value(고유 값 종류)가 매우 적다는 것을 딕셔너리 통계로 눈치채고, 머릿속에서 강제로 '남'일 때 한 번, '여'일 때 한 번 인덱스 트리를 타는 마법을 부린다. 단, 선행 컬럼의 종류가 '고객명'처럼 수만 개라면 스킵 스캔은 작동하지 않는다.

+++

### 3. 인덱스 컨디션 푸시다운 (ICP, Index Condition Pushdown)

MySQL 5.6에 도입된 획기적인 최적화. 스토리지 엔진(InnoDB)과 옵티마이저(Server 엔진) 간의 통신 비용을 줄인다.

- **원리**: 인덱스 `(이름, 나이)`가 있을 때 `WHERE 이름 LIKE '김%' AND 나이 = 30` 쿼리를 날렸다. `LIKE` 조건 때문에 `나이` 인덱스는 탐색용(Range)으로 쓰이지 못한다.
- **과거 (ICP 끄기)**: 스토리지 엔진은 인덱스에서 '김'씨인 사람 1만 명의 ROWID를 찾아, 디스크 원본 데이터를 1만 번 읽어(랜덤 I/O) 서버 엔진으로 올려보낸다. 서버 엔진이 원본 데이터에서 `나이=30`인 사람을 걸러낸다. (최악의 디스크 I/O)
- **현재 (ICP 켜기)**: 서버 엔진이 스토리지 엔진에게 아예 `나이=30` 조건까지 같이 밀어내어 위임한다. 스토리지 엔진은 인덱스 트리에 어차피 '나이' 값도 같이 들어있으므로, 인덱스만 보고 `나이=30`이 아닌 사람들은 디스크를 읽기 전에 미리 버려버린다. 디스크 접근이 1만 번에서 10번으로 줄어든다.

- **📢 섹션 요약 비유**: 옛날엔 은행 창구(스토리지 엔진)에서 김씨 성을 가진 1만 명의 통장(디스크)을 무식하게 다 뽑아와서 팀장(옵티마이저)에게 주면 팀장이 나이가 30세인지 검사했다면, ICP는 팀장이 처음부터 "김씨 중에 통장 표지(인덱스)에 30세라고 적힌 것만 꺼내와!"라고 스마트하게 지시하는 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 다중 컬럼 조건 검색 시 접근 전략 비교

| 비교 항목 | 단일 인덱스 2개 병합 (Index Merge) | 결합 인덱스 (Composite Index) |
|:---|:---|:---|
| **생성 형태** | `IDX_A (A)`, `IDX_B (B)` 각각 생성 | `IDX_AB (A, B)` 하나 생성 |
| **동작 메커니즘**| A 인덱스 결과의 ROWID 집합과 B 인덱스의 ROWID 집합을 교집합(AND) 연산 후 테이블 접근 | A조건 타고 즉시 B조건 타서 단 1회의 테이블 접근 |
| **CPU/메모리 부하**| 매우 높음 (두 집합의 머지 및 정렬 오버헤드 발생) | **매우 낮음 (즉시 데이터 핀포인트 탐색)** |
| **추가 혜택** | 없음 | **커버링 인덱스(Covering Index)로 활용 가능성 열림** |
| **권장 환경** | 두 컬럼이 각자 독립적으로 빈번하게 검색될 때 | 두 컬럼이 항상 `AND` 조건으로 같이 다닐 때 (우선 권장) |

### 과목 융합 관점

- **자료구조 및 알고리즘**: 결합 인덱스는 본질적으로 데이터베이스 커널 C 코딩 레벨에서의 튜플 비트 비교(Tuple Bitwise Comparison)다. B-Tree 노드에 `[A값, B값, C값]`이 패킹(Packing)되어 있을 때, C++의 `memcmp` 함수를 통해 다중 컬럼의 대소 비교를 한 틱(Tick)에 처리해 내는 로우 레벨 융합 기술이 녹아 있다.
- **비용 기반 옵티마이저 (CBO)**: 스킵 스캔이나 ICP 같은 튜닝 기능은 개발자가 원한다고 항상 켜지는 것이 아니다. CBO가 계산한 논리적 I/O 비용이 "스킵 스캔을 하느니 차라리 풀 스캔을 하는 게 싸겠다"고 판단하면 작동하지 않으므로, 옵티마이저의 수학 모델과 힌트(`/*+ INDEX_SS(table idx) */`)를 이해하는 것이 선행되어야 한다.

- **📢 섹션 요약 비유**: 총을 두 자루(단일 인덱스 2개) 들고 쏜다고 표적(데이터)이 빨리 맞는 게 아닙니다. 영점이 완벽하게 잡힌 저격총 한 자루(결합 인덱스)로 정확히 급소를 쏘는 것이 RDBMS 튜닝의 진리입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 물류 시스템의 이력 조회 타임아웃 튜닝

- **문제 상황**: 사용자가 "특정 배송지(지역)에서 2026-03-01부터 03-10 사이에 출고 완료된 주문"을 조회(`WHERE 지역='서울' AND 상태='출고' AND 날짜 BETWEEN A AND B`)하는데, DB 응답이 10초 이상 걸림. 기존 인덱스는 `IDX_1 (날짜, 지역, 상태)`로 잡혀 있었음.
- **원인 분석**: 인덱스의 선행 컬럼인 `날짜`가 `BETWEEN` (범위 조건)으로 들어오는 바람에, 그 뒤에 있는 `지역`과 `상태` 컬럼은 인덱스의 정렬 구조를 타고 내려가는 탐색(Access) 역할을 상실하고 단순 필터(Filter)용으로 전락함. 결과적으로 10일 치 데이터를 전부 디스크에서 퍼올려 뒤지는 비효율 발생.
- **의사결정 (결합 인덱스 순서 재배치)**:
  1. 인덱스를 **`IDX_NEW (지역, 상태, 날짜)`** 순으로 재생성.
  2. 선행하는 `지역(=)`과 `상태(=)`가 1차적으로 데이터를 극단적으로 좁혀주고(Access), 마지막 `날짜`가 그 안에서 연속된 범위만 순차 스캔하게 됨.
  3. 10초 걸리던 악성 쿼리가 0.05초 만에 완료됨. (인덱스 컬럼 순서 튜닝의 위력)

### 도입 체크리스트
- **선행 컬럼의 선택성 (Cardinality)**: 결합 인덱스를 만들 때 가장 흔한 오해가 "고유값이 많은(카디널리티가 높은) 컬럼을 무조건 앞에 둬야 한다"는 속설이다. 하지만 `= (Equal)` 조건이라면 카디널리티가 낮든 높든 성능 차이가 거의 없다. 진짜 중요한 원칙은 **"항상 `=` 조건으로 조회되는 컬럼을 맨 앞에 두는 것"**이다.
- **커버링 인덱스로의 확장 가능성**: `SELECT 이름, 나이 FROM User WHERE 부서='영업' AND 성별='남'` 이라는 쿼리가 잦다면, 아예 인덱스를 `[부서, 성별, 이름, 나이]` 4개로 잡아버리는 것을 고려했는가? 이렇게 하면 디스크 테이블 블록에 아예 접근하지 않고 인덱스 트리의 리프 노드만 읽고 응답을 끝내는 궁극의 커버링 인덱스가 완성된다.
- **스킵 스캔의 한계 인지**: 스킵 스캔을 맹신하여 인덱스 설계를 대충 하면 안 된다. 선행 컬럼의 종류 수가 100개, 1,000개로 넘어가면 옵티마이저가 1,000번의 브랜치 점프 연산을 하느라 오히려 성능이 최악으로 치닫는다.

### 안티패턴: 쓸데없는 인덱스 병합(Index Merge) 유발 설계
- 테이블 하나에 A, B, C, D 컬럼별로 각각 단일 인덱스를 4개 만들어 두고, "옵티마이저가 알아서 인덱스를 합쳐서(Merge) 똑똑하게 쓰겠지"라고 방치하는 설계. Index Merge 연산은 각각의 인덱스에서 찾은 ROWID들을 메모리나 Temp 공간으로 끌어와 정렬하고 교집합(Intersection)을 구하는 어마어마한 오버헤드를 발생시킨다. 자주 같이 쓰이는 조건이라면 반드시 결합 인덱스 1개로 합쳐야 한다.

- **📢 섹션 요약 비유**: 열쇠 구멍 3개짜리 금고를 열 때, 각각의 열쇠를 따로 꽂아서(단일 인덱스 3개) 돌리는 건 엄청난 시간이 듭니다. 아예 돌기 3개가 한 번에 박혀 있는 만능열쇠(결합 인덱스) 하나를 깎아두는 것이 튜닝 마스터의 비법입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 단일 인덱스 남용 / 순서 오류 | 최적화된 결합 인덱스 설계 | 개선 효과 |
|:---|:---|:---|:---|
| **스캔 비용(Logical Reads)**| 인덱스 필터링 실패로 무의미한 블록 스캔 폭주 | Equal 선행 배치로 핀포인트 탐색 | **논리적 디스크 I/O 횟수 90~99% 극한 절감** |
| **디스크 I/O (물리 접근)**| ROWID 랜덤 액세스로 디스크 병목 | 커버링 인덱스(Covering Index) 융합 | **테이블 접근 완전 배제로 응답 지연 0화** |
| **쓰기 부하(Insert)** | 무분별한 10개의 단일 인덱스 업데이트 | 2~3개의 압축된 결합 인덱스만 갱신 | **DML 처리 속도 및 DB CPU 여유율 대폭 상승** |

### 미래 전망
- **인덱스 무용론과 컬럼 스토어의 진격**: Snowflake나 ClickHouse 같은 차세대 클라우드 데이터 웨어하우스에서는, 데이터를 컬럼(Column) 단위로 압축하고 각 블록의 최소/최대값 메타데이터(Zone Map, Min-Max Index)만으로 불필요한 디스크 스캔을 건너뛰는 아키텍처를 도입하여 복잡한 B-Tree 결합 인덱스 설계 자체를 불필요하게 만들고 있다.
- **머신러닝 기반 다차원 인덱스 (Learned Index)**: 향후에는 B-Tree의 구조를 완전히 버리고, 입력 데이터(x)를 넣으면 데이터가 위치한 디스크 블록(y)을 인공지능 신경망이 회귀 방정식(Regression)으로 예측하여 단 1회의 수학적 연산으로 찾아내는 Learned Index 모델이 주류 연구로 상용화를 앞두고 있다.

- **📢 섹션 요약 비유**: 결합 인덱스 튜닝은 도서관 사서가 수억 권의 책을 어떤 순서와 논리로 서가에 배치할지 결정하는 가장 고도화된 두뇌 게임입니다. 이 게임의 승리자만이 대규모 트래픽이라는 거대한 파도를 잠재울 수 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **B-Tree와 B+Tree 구조 심층 해부**: 결합 인덱스가 왜 좌측 컬럼부터 순차적으로 정렬될 수밖에 없는지, 그 물리적인 트리 제약을 설명하는 근간 아키텍처.
- **비용 기반 옵티마이저 (CBO)**: 인덱스 스킵 스캔(Skip Scan)을 할지, 인덱스 병합(Merge)을 할지 최종적으로 비용 함수를 돌려 판단하는 핵심 두뇌.
- **고급 SQL 튜닝 기법 (조인 순서)**: 조인 쿼리에서 '어느 쪽 테이블의 결합 인덱스를 타야(Driving Table) 조인 비용이 적어지는가'를 통제하는 연계 튜닝 기법.
