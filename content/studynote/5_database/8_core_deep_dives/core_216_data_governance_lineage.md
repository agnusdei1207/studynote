# 데이터 거버넌스와 리니지 - Dark Data, 메타데이터 관리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 거버넌스(Data Governance)는 전사적 데이터의 가용성, 가용성, 무결성, 보안을 유지하기 위한 정책, 프로세스 및 조직의 총체이며, **데이터 리니지(Data Lineage)**는 이 데이터가 어떤 경로로 흘러가며 변환되었는지를 추적하는 핵심 뼈대 기술이다.
> 2. **가치**: 버려져서 쓸모없고 리스크만 키우는 **다크 데이터(Dark Data)**를 지능형 메타데이터 카탈로그로 발굴하여 자산화하고, 대시보드 지표 오류 발생 시 리니지 그래프를 통해 근본 원인(Root Cause)을 즉각 도출한다.
> 3. **융합**: 폴리글랏 퍼시스턴스와 데이터 메시(Data Mesh)와 같은 고도로 분산된 아키텍처 위에서, 중앙화된 정책과 분산된 기술 통제(Federated Computational Governance)를 융합하는 논리적 지휘소 역할을 수행한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **데이터 거버넌스**: 데이터를 체계적으로 관리하고 활용하기 위한 원칙, 조직, 프로세스의 프레임워크.
  - **데이터 리니지 (Data Lineage)**: 데이터의 기원(Origin)부터 목적지(Destination)까지의 여정, 즉 데이터가 추출, 변환, 병합되는 모든 경로와 변경 이력을 시각적으로 추적하는 기술.

- **💡 비유**: 거버넌스가 "국가의 헌법과 교통 법규"라면, 데이터 리니지는 "농장에서 내 식탁까지 소고기가 어떻게 운송되고 가공되었는지 추적하는 이력제(바코드)"와 같습니다.

- **등장 배경**:
  1. **데이터 늪(Data Swamp) 현상**: 하둡(Hadoop)과 데이터 레이크에 데이터를 무작정 쏟아부었으나, 누가 언제 어떤 기준으로 넣은 데이터인지 아무도 모르는 쓰레기장(Dark Data)이 되어버렸다.
  2. **규제 준수 (Compliance)의 압박**: GDPR(유럽), CCPA(캘리포니아), 개인정보보호법 등 규제가 강화되면서, 기업이 고객 데이터를 어떻게 수집하고 폐기하는지 완벽하게 증명(추적)하지 못하면 천문학적 벌금을 맞게 되었다.
  3. **AI 파이프라인의 환각 및 편향**: AI 모델이 내린 결정이 어떤 원천 데이터를 학습하여 도출되었는지(설명 가능한 AI, XAI) 추적하기 위한 데이터의 역방향 파이프라인 증명이 필수가 되었다.

- **📢 섹션 요약 비유**: 과거에는 우물물(데이터)을 그냥 떠 마셔도 문제없었지만, 지금은 수만 개의 파이프가 얽힌 대도시에서 어느 파이프가 녹슬었는지(리니지)와 수질 기준(거버넌스)을 엄격히 감시하지 않으면 도시 전체가 전염병에 걸리게 됩니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 데이터 거버넌스의 3대 요소 (원칙, 조직, 프로세스)

거버넌스는 단순한 IT 툴 도입이 아니라 조직 문화의 정립이다.

| 구분 | 핵심 내용 | 비유 |
|:---|:---|:---|
| **원칙 (Principle)** | 전사 표준 단어 사전(Data Dictionary), 보안 정책, 품질 기준 수립. "고객"이란 단어는 오직 한 가지 의미로만 쓴다. | 헌법 제정 |
| **조직 (Organization)** | Chief Data Officer(CDO)를 필두로, 도메인별 데이터 소유자(Data Owner)와 관리자(Data Steward) 명확화. | 정부 부처와 공무원 할당 |
| **프로세스 (Process)** | 데이터의 생성부터 폐기, 마스킹 처리, 접근 권한 부여를 위한 승인 흐름과 자동화 워크플로우. | 행정 결재 및 실행 시스템 |

+++

### 2. 다크 데이터 (Dark Data)와 데이터 디스커버리 (Data Discovery)

- **다크 데이터**: 기업이 수집하고 저장하지만, 분석이나 비즈니스 의사결정에 전혀 사용되지 않아 가치를 창출하지 못하고 스토리지 비용과 보안 리스크만 발생시키는 데이터. (전체 기업 데이터의 50~80% 차지)
- **메타데이터 (Metadata)**: 데이터에 대한 데이터. (예: 생성자, 파일 크기, 컬럼 타입, 최종 수정일).
- **데이터 디스커버리 및 카탈로그 (Data Catalog)**: AI/ML을 이용해 흩어진 DB와 스토리지를 스캐닝하여 메타데이터를 수집하고, 검색 가능한 구글 스타일의 포털 사이트로 만들어 다크 데이터를 자산으로 발굴하는 플랫폼. (예: AWS Glue, Amundsen)

+++

### 3. 데이터 리니지 (Data Lineage) 추적 아키텍처

리니지는 방향성을 가진 그래프(DAG, Directed Acyclic Graph) 형태로 시각화된다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 데이터 리니지(Data Lineage) 추적 메커니즘              │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [역방향 추적 (Root Cause Analysis)]                              │
  │                                                                  │
  │  (Source)        (Transform/dbt)      (Target/BI)                │
  │  [MySQL:User] ──┐                                                │
  │                 ├──▶ [View: Daily_User] ──▶ [Tableau Dashboard]  │
  │  [S3:LogData] ──┘         │                     (에러 발생 💥)     │
  │                           │                                      │
  │  [Salesforce] ────────────┴────────────────▶ [CEO Report]        │
  │                                                                  │
  │  - 컬럼 레벨 리니지 (Column-level Lineage): 테이블을 넘어 'User.age'가 │
  │    대시보드의 '평균 연령' 지표로 어떻게 계산되어 들어갔는지까지 추적.           │
  │  - 정방향 추적 (Impact Analysis): [MySQL:User] 테이블의 스키마를 변경  │
  │    할 때, 장애가 발생할 하위 대시보드를 사전 경고(Alert)함.                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CEO가 보는 태블로(Tableau) 대시보드에서 '일일 활성 사용자 수' 지표가 갑자기 0으로 떨어지는 장애가 발생했다. **데이터 리니지 툴**을 켜면, 이 지표가 `Daily_User`라는 뷰(View)에서 왔고, 이 뷰는 다시 `MySQL`과 `S3`의 데이터를 `JOIN`하여 만들어졌다는 흐름이 완벽히 시각화된다. 역방향 추적(Trace-back)을 통해 어젯밤 S3 로그 적재 파이프라인이 멈췄음을 5분 만에 찾아낼 수 있다. 반대로, 개발자가 MySQL 스키마를 변경하려 할 때 정방향 분석(Impact Analysis)을 통해 "이 컬럼을 지우면 CEO 리포트가 망가집니다"라고 사전에 경고할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 마스터 데이터 관리(MDM) vs 메타데이터 관리 vs 데이터 카탈로그

| 구분 | 마스터 데이터 관리 (MDM) | 메타데이터 관리 | 데이터 카탈로그 플랫폼 |
|:---|:---|:---|:---|
| **관리 대상** | 실제 핵심 비즈니스 '값' (예: 전사 통합 고객 코드 1001번의 주소) | 데이터를 설명하는 '속성 정보' (예: 테이블 생성일자, 타입) | 메타데이터를 기반으로 한 '검색 포털' |
| **목적** | 부서 간 데이터 중복 제거 및 단일 진실 공급원(SSOT) 확보 | 데이터 구조와 흐름의 기술적 통제 | 누구나 쉽게 데이터를 쇼핑하고(Discovery) 이해하게 함 |
| **실행 주체** | 데이터 거버넌스 위원회 및 시스템 | DBA, 데이터 아키텍트 | 전사 직원 (데이터 분석가, 마케터 등) |

### 과목 융합 관점

- **소프트웨어 공학 (SE)**: 시스템 분석/설계에서 클래스 다이어그램이나 모듈 의존성 그래프를 그리는 것과 완벽히 대응된다. 소프트웨어 엔지니어링의 정적 분석(Static Analysis) 기술을 SQL 파서(Parser)에 융합하여 자동으로 리니지 그래프를 추출한다.
- **분산 시스템 아키텍처 (Data Mesh)**: 거버넌스는 데이터 메시를 지탱하는 핵심 기둥이다. 중앙 통제가 풀린 대신, **'연합된 전산적 거버넌스(Federated Computational Governance)'**를 통해 코드 레벨(Policy-as-Code)에서 규정 준수 여부를 자동 컴파일하고 배포를 차단하는 방식으로 융합된다.

- **📢 섹션 요약 비유**: MDM이 회사 전 직원의 '유일한 주민등록번호와 이름'을 정리하는 것이라면, 메타데이터는 직원의 '부서와 직급 명찰'이고, 데이터 카탈로그는 이 모든 정보를 검색할 수 있는 사내 '사조직 인트라넷 검색기'입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 개인정보 파기 요청(GDPR) 대응 파이프라인

- **문제 상황**: 유럽 고객이 자신의 모든 정보를 삭제해 달라고 요청함. 하지만 고객 데이터가 이커머스 DB(MySQL), 빅데이터 레이크(S3), 마케팅 캐시(Redis) 수백 곳에 복제되어 있어 어디를 지워야 할지 모름.
- **의사결정**:
  1. **개인정보 메타데이터 태깅**: 데이터 카탈로그(예: Collibra, Datahub)에 정규식을 이용해 주민번호, 이메일 형태를 띤 모든 컬럼에 'PII(Personally Identifiable Information)' 태그를 자동 부착.
  2. **리니지 그래프 스캔**: 고객 테이블의 `Email` 컬럼과 연결된 모든 하위 복제 파이프라인(S3, Redis, DW)을 리니지 그래프로 즉시 추출.
  3. **일괄 삭제 스크립트 트리거**: 추출된 엔드포인트에 대해 일괄적으로 Delete/Tombstone 작업을 수행하여 컴플라이언스(Compliance) 완벽 준수.

### 도입 체크리스트
- **컬럼 레벨 리니지 지원 여부**: 단순히 테이블 A가 테이블 B로 갔다는 '테이블 레벨 리니지'는 실무에서 반쪽짜리다. `User.age`와 `Log.duration`이 연산되어 `Report.score`가 되었다는 '컬럼 레벨 리니지(Column-level Lineage)'를 지원하는 솔루션인가?
- **메타데이터 자동 수집기 (Crawler)**: 수백 개의 RDBMS, NoSQL, BI 툴(Tableau)에 흩어진 메타데이터를 사람이 수동으로 적는 것은 불가능하다. 스케줄러 기반으로 자동 변경분을 감지하고 카탈로그를 갱신하는 커넥터(Connector)가 충분한가?
- **조직의 거버넌스 스폰서십**: C-Level(경영진)의 강력한 지지 없이 IT 부서 단독으로 추진하는 거버넌스는 100% 실패한다. 각 부서장이 데이터 품질에 책임(Data Owner)을 지도록 성과 평가(KPI)와 결합되어 있는가?

### 안티패턴: 형식적인 데이터 사전 엑셀(Excel) 관리
- 수백 개의 테이블과 컬럼 정의를 엑셀 파일(데이터 사전)에 수동으로 기록하고 사내 망에 올려두는 고전적 방식. DB 스키마는 매일 바뀌는데 엑셀은 업데이트되지 않아, 분석가들이 엑셀을 믿고 개발했다가 에러를 겪으며 결국 엑셀 문서 자체가 거대한 다크 데이터이자 기술 부채로 전락한다.

- **📢 섹션 요약 비유**: 수만 권의 장서가 있는 도서관에서 사서가 엑셀로 책 목록을 적어두는 것은 무의미합니다. 책이 반납되고 대출될 때마다 바코드로 중앙 시스템(카탈로그)이 실시간으로 갱신되어야만 완벽한 재고 관리가 가능합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 거버넌스 미비 체계 | 지능형 거버넌스 & 카탈로그 도입 | 개선 효과 |
|:---|:---|:---|:---|
| **데이터 탐색 시간** | 담당자 수소문 및 스키마 분석에 수주 소요 | 카탈로그 포털에서 검색 1분 컷 | **데이터 사이언티스트 생산성 50% 이상 극대화** |
| **장애 원인 파악 (RCA)** | 역방향 수동 쿼리 추적으로 며칠 소요 | 리니지 그래프 뷰를 통해 수 분 내 파악 | **MTTR(평균 복구 시간) 90% 이상 단축** |
| **다크 데이터 절감** | 안 쓰는 쓰레기 테이블 무한 저장 | 사용 빈도 분석을 통한 자동 폐기(Archiving) | **불필요한 클라우드 스토리지 비용 30%+ 절감** |

### 미래 전망
- **AI 기반 능동형 거버넌스 (Active Data Governance)**: 사람이 규칙을 정하는 수동적(Passive) 방식을 넘어, 머신러닝이 사람들의 쿼리 패턴과 데이터 분포를 학습하여 "이 데이터는 개인정보일 확률이 높으니 마스킹하겠습니다"라고 스스로 정책을 제안하고 실행하는 능동형 체계로 진화하고 있다.
- **데이터 컨트랙트 (Data Contract)**: API 통신 규약처럼, 데이터를 제공하는 팀과 소비하는 팀 사이에 스키마, 품질, 지연 시간(SLA)에 대한 소프트웨어적 계약(Contract)을 코드로 체결하여, 계약이 위반되면 CI/CD 파이프라인에서 배포를 차단하는 강력한 방어 기제가 표준화될 것이다.

- **📢 섹션 요약 비유**: 데이터 거버넌스와 리니지는 폭주하는 데이터라는 '야생마'에 올라타기 위해 씌우는 튼튼한 '안장과 고삐'입니다. 이 고삐가 완벽히 작동해야만 기업은 데이터라는 말을 통제하고 원하는 방향으로 질주할 수 있습니다.

+++

## 📌 관련 개념 맵 (Knowledge Graph)
- **데이터옵스와 실시간 처리**: 리니지 정보를 동적으로 생성하고 거버넌스 자동화 테스트를 실행(CI/CD)하는 배관망 파이프라인 프로세스.
- **차세대 거버넌스 - 데이터 메시**: 거버넌스 관할권을 중앙에서 부서별로 이양하면서(분산 오너십), 카탈로그를 통해 융합하는 최상위 아키텍처 철학.
- **데이터베이스 보안 및 마스킹**: 거버넌스 위원회가 식별한 민감 정보를 시스템 레벨에서 물리적으로 보호(TDE, DDM)하는 최하위 기술 계층.

## 👶 어린이를 위한 3줄 비유 설명
1. 서랍장 구석에 처박혀서 있는지도 몰랐던 장난감들을 **다크 데이터**라고 해요. 공간만 차지하고 쓸모가 없죠.
2. **데이터 카탈로그(메타데이터)**는 온 집안의 장난감 사진을 찍어서 태블릿에 '빨간색 자동차, 거실 로봇'이라고 예쁘게 정리해두고 1초 만에 찾게 해주는 마법의 앱이에요.
3. **데이터 리니지**는 이 장난감이 공장에서 어떻게 만들어져서 마트를 거쳐 내 손까지 왔는지, 중간에 망가진 곳은 없는지 길을 그려주는 투시경이랍니다!
