# 52. 회복 매니저 (Recovery Manager) - 백업 및 복구

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 장애(Crash), 미디어 결함, 혹은 사용자 실수로 인해 데이터베이스가 일관성을 잃었을 때, 로그와 백업 데이터를 활용하여 **장애 직전의 가장 최신 일관된 상태로 되돌리는** 복원 엔진이다.
> 2. **메커니즘**: 로그 매니저가 기록한 변경 이력을 추적하여, 커밋되지 않은 트랜잭션은 취소(**Undo**)하고, 커밋되었으나 디스크에 반영되지 않은 데이터는 재실행(**Redo**)하는 과정을 수행한다.
> 3. **가치**: 비즈니스 영속성을 보장하는 최후의 보루이며, 현대의 고가용성(HA) 환경에서는 복구 시간(RTO)과 데이터 유실 허용치(RPO)를 최소화하는 정교한 **ARIES 알고리즘** 등이 활용된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**회복 매니저 (Recovery Manager)**는 데이터베이스의 신뢰성을 보장하기 위해 장애 발생 시 데이터베이스를 정상 상태로 복구하는 기능을 담당합니다. ACID 원칙 중 **원자성(Atomicity)**과 **영속성(Durability)**을 실제로 구현하는 주체입니다.

### 2. 💡 비유: 영화 편집실과 자동 저장 기능
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        회복 매니저 비유                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 영화 촬영 (트랜잭션): 열심히 장면을 찍고 편집함.                             │
│  2. 정전 발생 (장애): 갑자기 컴퓨터가 꺼져서 편집 중이던 영상이 날아감.            │
│  3. 회복 매니저 (편집실 복구 전문가):                                          │
│     - "마지막으로 저장된 시점(Checkpoint)"을 찾음.                              │
│     - 저장 버튼을 누르기 전의 'NG 장면'은 삭제함. (Undo)                         │
│     - 저장 버튼을 눌렀지만 파일에 안 기록된 '완성 장면'은 다시 복원함. (Redo)       │
│                                                                             │
│  → 편집 전문가 덕분에 정전이 나도 처음부터 다시 찍을 필요 없이, **'딱 사고 직전'**   │
│     상태로 완벽하게 돌아가 다시 작업을 시작할 수 있음!                             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **다양한 장애 유형**: 전원 장애(System Crash), 디스크 손상(Media Failure), 소프트웨어 버그, 그리고 사람의 실수(Drop Table 등) 존재.
- **가용성 요구**: 24/365 서비스 환경에서 장애 후 복구에 걸리는 시간이 길어지면 막대한 경제적 손실 발생.

- **📢 섹션 요약 비유**: 회복 매니저는 데이터베이스의 '시간을 되돌리는 마법사'이자, 모든 사고로부터 데이터를 지켜내는 '데이터 보험사'와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 장애 유형별 회복 전략

| 장애 유형 | 상세 원인 | 회복 수단 |
|:---:|:---|:---|
| **트랜잭션 장애** | 내부 로직 오류, 데이터 입력 위반 | 로그를 이용한 해당 트랜잭션 **Undo** |
| **시스템 장애** | 하드웨어 정전, OS 커널 패닉 | 최근 체크포인트 이후 로그 기반 **Redo/Undo** |
| **미디어 장애** | 디스크 헤드 손상, 파일 파손 | **백업 데이터 로드** + 아카이브 로그 적용 |

### 2. 회복 프로세스 아키텍처 (ASCII)

장애 발생 시 회복 매니저가 로그를 분석하여 수행하는 3단계를 시각화한다.

```text
    [ 장애 발생 시점 ]
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 1. [ Analysis Phase ] : 로그 분석 (Dirty Page 및 활성 Tx 파악)     │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 2. [ Redo Phase ] : "이미 성공한 거래" 재현 (History 재구성)       │
    │    - 디스크에 반영 안 된 커밋 데이터를 로그 보고 강제 기록            │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────────────────────────────┐
    │ 3. [ Undo Phase ] : "진행 중이던 거래" 취소 (일관성 확보)          │
    │    - 장애 시점에 미완료된 트랜잭션의 변경 사항을 원복                │
    └──────┬──────────────────────────────────────────────────────────┘
           │
    [ DB 일관성 회복 완료 ]
```

### 3. 심층 동작 원리: ARIES 알고리즘 (Algorithm for Recovery and Isolation Exploiting Semantics)
현대 RDBMS(IBM DB2, MSSQL 등)의 표준 회복 기법입니다.
1. **LSN (Log Sequence Number) 활용**: 모든 로그와 데이터 페이지에 번호를 매겨 복구의 정확성 보장.
2. **Repeating History**: Redo 단계에서 장애 발생 전의 상태를 똑같이 재현하여 복구의 단순함 유지.
3. **CLR (Compensation Log Record)**: Undo 작업을 수행할 때도 로그를 남겨, 복구 도중 다시 장애가 나도 안전하게 재복구 가능.

### 4. 핵심 기술: 미디어 회복 (Media Recovery)
- **Full Backup**: 전체 데이터베이스 복사본.
- **Incremental Backup**: 마지막 백업 이후 변경된 부분만 저장.
- **Point-in-Time Recovery**: 특정 날짜, 특정 시(時)로 데이터를 완벽히 복원하는 기술.

- **📢 섹션 요약 비유**: ARIES는 '사고 현장의 블랙박스를 보고, 사고가 나기 전 1초 전까지의 모든 움직임을 재연해서 똑같이 맞춰놓는 기술'입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 지연 갱신(Deferred) vs 즉시 갱신(Immediate) 회복 기법

| 비교 항목 | 지연 갱신 (Deferred Update) | 즉시 갱신 (Immediate Update) |
|:---:|:---|:---|
| **갱신 시점** | 트랜잭션 커밋 전까지 메모리에만 유지 | 연산 수행 중 수시로 디스크 반영 |
| **로그 내용** | 새 값(New Value)만 기록 | **이전 값(Old)과 새 값(New)** 모두 기록 |
| **회복 방식** | **Redo**만 수행 | **Redo와 Undo** 모두 수행 |
| **안정성** | 복구 로직이 단순함 | 동시성과 가용성이 높음 (현대 표준) |

### 2. 과목 융합 관점: 소프트웨어 공학 및 보안
- **소프트웨어 공학(SE) 융합**: 회복 매니저는 시스템의 **복원력(Resiliency)**과 **내결함성(Fault Tolerance)** 아키텍처의 핵심이며, BCP(Business Continuity Planning)의 기술적 토대다.
- **보안(Security) 융합**: 랜섬웨어 공격 등으로 데이터가 암호화되었을 때, 회복 매니저의 **불변 백업(Immutable Backup)**과 시점 복구 기능은 최후의 방어 수단이 된다.

- **📢 섹션 요약 비유**: 지연 갱신은 '다 완성하고 한꺼번에 제출'하는 것이고, 즉시 갱신은 '만드는 족족 실시간으로 제출'하는 방식입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 실수로 날린 테이블 복구하기
- **상황**: 신입 개발자가 운영 DB에서 `DELETE FROM orders`를 조건 없이 실행하여 데이터가 다 날아감.
- **판단**: 이미 커밋이 완료되어 Undo 로그로는 복구 불가.
- **결단**: **회복 매니저의 Flashback(Oracle) 또는 PITR(Point-in-Time Recovery)** 기능을 가동. 어제 새벽 백업본을 로드한 뒤, 아카이브 로그를 사고 1초 전까지만 적용(Roll-forward)하여 비즈니스 피해 최소화.

### 2. 안티패턴: 백업 검증 없는 백업 수행
- **증상**: 매일 백업은 받는데, 한 번도 복구 테스트(Restore Test)를 해본 적 없음.
- **결과**: 실제 장애 발생 시 백업 파일이 깨져있거나 암호화 키를 분실하여 복구에 실패함.
- **조치**: **정기적인 복구 훈련(Drill)**을 통해 RTO를 측정하고 백업 파일의 무결성을 검증하는 프로세스 정립.

### 📢 기술사적 결언
> "회복은 기술이 아니라 **'철학'**이다. 완벽한 시스템은 없으며, 모든 시스템은 언젠가 실패한다. 기술사는 시스템 설계 시 '어떻게 빨리 처리할 것인가'보다 **'어떻게 안전하게 되돌릴 것인가'**에 더 큰 무게를 두어야 한다. 특히 클라우드 네이티브 환경에서는 스냅샷(Snapshot)과 로그 스트리밍을 결합하여 복구 시간을 획기적으로 단축하는 **서버리스 회복 아키텍처**를 적극 도입해야 한다."

- **📢 섹션 요약 비유**: 복구 테스트 없는 백업은 '한 번도 열어보지 않은 비상구'와 같습니다. 불이 났을 때 문이 안 열릴 수도 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (ROI)

| 지표 | 도입 전 | 도입 후 (회복 매니저 가동) | 개선 효과 |
|:---|:---:|:---:|:---:|
| **데이터 손실량 (RPO)** | 전체 유실 위험 | **수 초 이내 (로그 기반)** | 비즈니스 자산 보호 100% |
| **서비스 복구 시간 (RTO)** | 수일 소요 (재입력) | **수 분 ~ 수 시간** | 운영 효율성 수십 배 향상 |
| **관리자 심리적 안정** | 극심한 스트레스 | **시스템적 복구 신뢰** | 조직의 운영 성숙도 향상 |

### 2. 미래 전망
- **Autonomous Recovery**: 장애 징후를 감지한 AI가 스스로 대기 DB로 전환하고 복구를 진행하는 **자율 운전 데이터베이스**로 진화 중.
- **Cross-Region Serverless Backup**: 전 세계 클라우드 리전에 데이터를 실시간 복제하여 지리적 재해(지진 등)에도 즉각 회복 가능한 **Global Resiliency** 표준화.

- **📢 섹션 요약 비유**: 미래의 회복 매니저는 사고가 난 줄도 모르게 뒤에서 조용히 모든 것을 고쳐놓는 '투명한 수호천사'가 될 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **회복 매니저**는 장난감 성을 쌓다가 누가 실수로 무너뜨렸을 때, **다시 원래대로 예쁘게 고쳐주는** 해결사예요.
2. 성을 쌓는 과정을 **사진(로그)**으로 다 찍어두었기 때문에, 사진을 보면서 하나하나 똑같이 맞출 수 있죠.
3. 이 해결사 덕분에 우리는 실수로 망가뜨려도 **속상해하지 않고** 다시 즐겁게 놀 수 있답니다!
