# RDBMS 메모리 아키텍처 - SGA와 PGA, 파싱 오버헤드

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 고성능 RDBMS(특히 Oracle 아키텍처 기준)는 수백만 명의 동시 접속을 락(Lock) 경합 없이 처리하기 위해, 모든 사용자가 공유하는 거대한 전역 메모리 **SGA(System Global Area)**와 개별 세션이 독점적으로 사용하는 사적 메모리 **PGA(Program Global Area)**로 아키텍처를 완벽히 이원화했다.
> 2. **가치**: SGA 내의 공유 풀(Shared Pool)은 한 번 힘들게 계산된 SQL의 실행 계획(Plan)을 캐싱해 두어 치명적인 **하드 파싱(Hard Parsing) 오버헤드를 제거**하며, 데이터 버퍼 캐시는 디스크 I/O를 메모리 속도로 끌어올린다. 반면 PGA는 각 사용자의 대규모 정렬(Sort)과 해시 조인 공간을 안전하게 격리 보장한다.
> 3. **융합**: 이 구조는 애플리케이션의 **바인드 변수(Bind Variable)** 사용 여부와 극한으로 융합되며, OS 레벨의 멀티 프로세스/멀티 스레드 동기화(Latch, Mutex) 이론이 데이터베이스 커널에 가장 정교하게 응용된 컴퓨터 구조학의 축소판이다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **SGA (System Global Area)**: DB 인스턴스가 뜰 때 한 번 할당되며, DB에 접속하는 모든 프로세스(사용자)가 공통으로 읽고 쓸 수 있는 거대한 캐시 및 설정 정보 공간.
  - **PGA (Program Global Area)**: 사용자가 DB에 접속(커넥션)할 때마다 개별적으로 할당되는 프라이빗 메모리 영역. 남과 공유하지 않는 자신만의 작업(정렬, 변수 할당 등) 공간이다.

- **💡 비유**: SGA는 도서관 한가운데 있는 '거대한 공용 열람실 및 공용 책장(누구나 와서 책을 꺼내봄)'이고, PGA는 도서관에 들어온 학생 한 명 한 명에게 나누어주는 '개인용 독서실 책상(남의 간섭 없이 나만의 수학 문제를 푸는 공간)'과 같습니다.

- **등장 배경**:
  1. **디스크 병목 (Disk I/O Barrier)**: 데이터를 매번 디스크(하드 드라이브)에서 퍼올리면 초당 몇 백 건의 처리도 불가능했다. 전역 캐시(SGA)를 만들어 누군가 한 번 읽은 데이터는 메모리에 올려 모두가 쓰게 해야 했다.
  2. **동시성(Concurrency)과 격리성(Isolation)의 상충**: 캐시는 공유해야 좋지만, 각 사용자가 10만 건씩 정렬(Sort)하는 작업까지 공용 공간에서 하면 서로 자원을 뺏기 위해 락(Lock/Latch) 경합이 발생해 시스템이 죽는다. 따라서 작업 공간(PGA)의 엄격한 분리가 절실했다.
  3. **SQL 컴파일(Parsing) 비용의 폭발**: RBO(규칙)에서 CBO(비용 기반)로 옵티마이저가 진화하면서, 최적의 실행 계획을 짜는 연산 자체가 엄청난 CPU를 소모하게 되었다. 한 번 짠 계획을 재활용할 메모리 공간(Shared Pool)이 핵심으로 부상했다.

- **📢 섹션 요약 비유**: 훌륭한 주방(RDBMS)은 모든 요리사가 칼과 도마를 공유(SGA)하여 재료를 아끼면서도, 각자의 도마 위에서는 서로 방해받지 않고 자신만의 요리(PGA)를 완성할 수 있게 설계된 최적의 공간입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. SGA (공유 메모리)의 핵심 구성 요소

DB 인스턴스의 심장이며, 이곳의 히트율(Hit Ratio)이 시스템 성능의 99%를 결정한다.

| 구성 요소 | 역할 및 메커니즘 | 최적화 포인트 (Hit 목표) |
|:---|:---|:---|
| **Database Buffer Cache** | 디스크에서 읽은 데이터 블록(테이블/인덱스)을 메모리에 올려두어 다음 조회 시 I/O를 면제함. LRU 알고리즘 작동. | 90% 이상의 캐시 적중률 유지 필수. |
| **Shared Pool (공유 풀)** | **라이브러리 캐시(SQL 텍스트와 파싱된 실행 계획)**와 **딕셔너리 캐시(테이블 메타데이터, 권한 정보)** 보관. | 바인드 변수를 사용하여 쿼리 캐시 공유율 극대화. |
| **Redo Log Buffer** | 복구(Recovery)를 위해 수정된 내역(Log)을 디스크에 쓰기 전 임시 보관하는 매우 빠르고 작은 버퍼. | 커밋 시 순차 I/O로 즉각 비워짐 (Fast Flush). |

+++

### 2. PGA (사적 메모리)와 튜닝 포인트

PGA는 각 세션의 백그라운드 프로세스(Server Process)에 독립적으로 할당되는 공간이다.

- **Sort Area (정렬 공간)**: `ORDER BY`, `GROUP BY`를 수행할 때 데이터를 정렬하는 공간.
  - **위험성**: 이 공간이 꽉 차면 데이터베이스는 남은 데이터를 디스크의 **Temp Tablespace(임시 디스크 공간)**로 밀어내어(Swap/Spill) 정렬을 이어간다. 이는 메모리 속도에서 디스크 속도로 추락함을 의미하며, 쿼리가 100배 느려지는 치명적 원인이 된다.
- **Hash Area**: 대용량 `Hash Join` 시 선행 테이블로 해시 맵(Hash Map)을 구축하는 메모리.
- **Cursor State**: 현재 페치(Fetch) 중인 결과 집합의 상태와 포인터 정보 보유.

+++

### 3. 하드 파싱(Hard Parsing) vs 소프트 파싱(Soft Parsing)

사용자가 SQL을 던졌을 때, 옵티마이저가 쿼리 계획을 세우는 데 따르는 비용의 척도다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 SQL 쿼리 파싱(Parsing)의 동작 매커니즘                │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  Client ──▶ SELECT * FROM EMP WHERE id = 100;                    │
  │                                                                  │
  │  [ 1단계. 구문 분석 및 해시 함수 실행 ]                                  │
  │   - SQL 텍스트를 해시(Hash) 값으로 변환하여 SGA의 'Shared Pool' 검색      │
  │                                                                  │
  │  [ 분기점: Shared Pool에 동일한 SQL이 존재하는가? ]                      │
  │                                                                  │
  │  ▶ NO! (최초 실행)                                                  │
  │    [ Hard Parsing (하드 파싱) ]                                    │
  │    1. 문법 검사 (문법이 맞나?)                                        │
  │    2. 의미 검사 (테이블, 컬럼이 진짜 있고 권한은 있나?)                     │
  │    3. ⭐️ 최적화 (수천 개의 조인 경우의 수를 수학적으로 다 계산하여 최적 플랜 픽픽)│
  │    - 소모 시간: 0.1초~수 초 (CPU 과부하, 딕셔너리 락 발생)                │
  │    4. Shared Pool에 결과(실행 계획) 저장 후 실행                        │
  │                                                                  │
  │  ▶ YES! (누군가 이미 실행한 적 있음)                                     │
  │    [ Soft Parsing (소프트 파싱) ]                                    │
  │    - 최적화 등 무거운 작업 싹 다 건너뛰고, 메모리의 실행 계획 꺼내서 즉시 실행!   │
  │    - 소모 시간: 0.0001초 (거의 0에 수렴)                               │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터베이스에 있어 가장 무거운 노동은 데이터를 꺼내는 것보다 **"데이터를 꺼낼 최적의 길을 계산하는 것(최적화/Optimization)"**이다. 하드 파싱은 이 계산을 처음부터 쌩으로 다 하는 것이라 CPU를 극한으로 소모하고, 메모리에 락(Latch)을 걸어 다른 사람의 쿼리까지 지연시킨다. 소프트 파싱은 누군가 만들어둔 정답지를 그대로 베껴 쓰는 것이라 CPU 부하 없이 빛의 속도로 실행된다. 옵티마이저 아키텍처 튜닝의 목표는 **하드 파싱을 없애고 소프트 파싱을 극대화하는 것**이다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### 리터럴(Literal) 쿼리 vs 바인드(Bind) 쿼리 심층 비교

파싱 오버헤드를 결정짓는 애플리케이션 코딩 표준이다.

| 비교 항목 | 리터럴 쿼리 (상수 하드코딩) | 바인드 변수 쿼리 (Prepared Statement) |
|:---|:---|:---|
| **SQL 형태** | `SELECT * FROM A WHERE ID = 1;`<br>`SELECT * FROM A WHERE ID = 2;` | `SELECT * FROM A WHERE ID = ?;`<br>(`?`에 1, 2 값을 바인딩) |
| **Shared Pool 동작** | **숫자 1과 2를 다른 문자열로 인식**. 각기 다른 2개의 SQL로 간주함 | **완벽히 동일한 문자열로 인식** (하나의 SQL) |
| **파싱 결과** | 100만 명이 검색하면 **하드 파싱 100만 번 발생** (DB 즉사) | 최초 1번만 하드 파싱, 이후 99만 번은 **소프트 파싱** (초고속) |
| **보안 (Security)**| **SQL Injection 해킹에 100% 노출** | 파서가 구조를 먼저 고정하므로 SQL 해킹 원천 차단 |
| **예외적 단점** | 옵티마이저가 상수 값을 정확히 알아 최적의 인덱스를 섬세하게 고를 수 있음 | 값이 뭔지 모르므로 히스토그램(분포도)을 무시하고 대충 평균값으로 플랜을 짬 (Peeking 한계) |

### 과목 융합 관점

- **운영체제 (OS) Latch와 Mutex**: SGA는 공용 공간이므로 수천 개의 프로세스가 동시에 접근하면 메모리가 깨진다. 이를 막기 위해 DB 엔진은 OS 커널이 제공하는 아주 가볍고 빠른 메모리 락인 래치(Latch)와 뮤텍스(Mutex)를 수만 개씩 걸고 푼다. 하드 파싱이 잦아지면 이 Latch 경합(Spin) 현상이 극심해져서 CPU가 100%를 찍지만 실제 처리량은 0이 되는 '스래싱(Thrashing)'에 빠진다.
- **보안과 암호화**: Prepared Statement(바인드 변수)는 본래 '파싱 성능 최적화'를 위해 탄생한 DB 커널 기술이지만, 결과적으로 SQL 문장과 데이터 영역을 물리적으로 격리하는 효과를 가져와 현대 웹 보안의 제1원칙인 **SQL 인젝션 방어**의 가장 완벽한 융합 방패가 되었다.

- **📢 섹션 요약 비유**: 리터럴 쿼리는 손님이 올 때마다 "A 설계도로 지은 집", "B 설계도로 지은 집"을 매번 처음부터 도면을 새로 그리고 짓는(하드 파싱) 바보 짓입니다. 바인드 변수는 완벽한 모델하우스 도면(소프트 파싱)을 하나 픽스해 두고, 손님 얼굴(데이터)만 바꿔 끼우는 최고의 공장장입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 트래픽 폭주 시 "Library Cache Latch" 경합 장애 타개

- **문제 상황**: 이벤트 푸시 발송 직후 초당 1만 명의 고객이 접속하여 상품을 조회. DB CPU가 순식간에 100%를 치고 모든 시스템이 '대기 상태(Hang)'에 빠짐.
- **원인 분석**: 개발팀이 ORM(JPA)을 쓰지 않고 MyBatis에서 동적 쿼리를 짤 때, `<if>` 문 안에서 `${id}` (문자열 치환) 기법을 사용함. 이로 인해 `WHERE id=1`, `id=2`... 와 같이 초당 1만 개의 완전히 새로운 SQL 문자열이 DB로 날아감. DB는 1만 번의 하드 파싱을 시도하며 SGA의 라이브러리 캐시 래치(Library Cache Latch)를 획득하려다 상호 경합(교착)으로 질식사함.
- **의사결정 (바인드 변수 전환 및 튜닝)**:
  1. MyBatis의 쿼리 문법을 `${id}`(리터럴)에서 `#{id}`(바인드 변수)로 전면 수정 배포.
  2. SQL 문자열이 `WHERE id=?` 하나로 통일되면서, 최초 1건만 하드 파싱되고 나머지 9,999건은 래치 획득 없이 소프트 파싱되어 CPU 부하율이 100% -> 5%로 즉각 안정화됨.

### 도입 체크리스트
- **Sort/Hash Area (PGA) 사이징**: 통계 대시보드 쿼리(`GROUP BY`, `ORDER BY` 남용)가 많다면 공용 메모리인 SGA 버퍼를 줄이고, 각자의 개인 메모리인 PGA 타겟을 넉넉하게 잡아주어야 디스크 I/O (Temp 스왑) 없이 메모리 내 연산이 이루어진다. 시스템의 OLTP / OLAP 성향에 따라 메모리 파이 비율을 튜닝했는가?
- **Cursor Sharing (커서 공유) 강제**: 수정이 불가능한 외부 레거시 솔루션이 리터럴 쿼리를 미친 듯이 쏘아대어 하드 파싱이 폭주한다면? 오라클의 경우 `CURSOR_SHARING = FORCE`라는 파라미터를 통해 DB 커널이 들어오는 쿼리의 상수를 강제로 별표(`*`) 모양의 바인드 변수로 치환해 버리는 무식하지만 확실한 응급 처방을 사용할 수 있다.
- **OS Paging/Swapping 방지**: SGA 영역이 물리적 RAM 크기를 초과하거나 잘못 세팅되어, 가상 메모리(디스크 Swap 영역)로 페이징 아웃(Paging Out)되는 순간 메모리 DB의 이점은 완벽히 소멸한다. OS 세팅에서 `Lock SGA` 나 리눅스의 `HugePages` 옵션을 켜서 DB 메모리가 절대 디스크로 내려가지 않게 핀(Pin) 고정했는가?

### 안티패턴: 대용량 페이징 처리에 `OFFSET` 무지성 사용
- 화면의 10,000번째 페이지를 보여주기 위해 `SELECT * FROM Log ORDER BY date DESC LIMIT 10 OFFSET 100000` 을 던지는 행위. DB는 인덱스를 탔음에도 불구하고 100,010건의 데이터를 모조리 디스크에서 읽어 PGA에 올려놓고 정렬(Sort)한 뒤, 앞의 100,000건을 허무하게 휴지통에 버리고 딱 10건만 리턴한다. PGA 메모리 파괴와 CPU 낭비의 극치다. 
- 대용량 데이터는 반드시 이전 페이지의 마지막 값(`WHERE date < '마지막날짜' LIMIT 10`)을 들고 와서 10건만 딱 읽는 **No-Offset (커서 페이징)** 방식으로 개선해야 메모리를 구원할 수 있다.

- **📢 섹션 요약 비유**: 100만 번째 뒤에 있는 10명을 뽑기 위해 100만 명 전체를 운동장(PGA)에 다 불러모아 줄을 세운 뒤(정렬 부하), 100만 명을 다시 집에 돌려보내는 것은 최악의 비효율입니다. 그냥 100만 번째 서 있는 사람의 번호표를 기억했다가 그 뒤에서부터 10명만 부르는 게 현대 페이징의 상식입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 하드 파싱 및 메모리 미튜닝 | 바인드 변수 + SGA/PGA 최적화 | 개선 효과 |
|:---|:---|:---|:---|
| **SQL 처리량 (TPS)** | CPU 파싱 오버헤드로 인해 수백 건 병목 | 소프트 파싱 캐시 히트로 연산 0화 | **초당 처리량(TPS) 10배~100배 극대화** |

### 미래 전망
- **인메모리 DB(IMDB)와 비휘발성 램 (NVDIMM)**: 디스크를 캐싱하기 위해 존재했던 복잡한 SGA 버퍼 풀의 개념 자체가 사라지고 있다. 데이터 원본 자체가 전부 DRAM과 비휘발성 램에 상주하는 SAP HANA, Redis Enterprise 같은 구조에서는 튜닝의 초점이 디스크 I/O가 아닌 CPU 캐시 미스와 코어 간 통신 최소화로 완전히 이동하였다.
- **동적 자원 할당 (Auto Memory Management)**: DBA가 수동으로 SGA 60%, PGA 40%를 분할하던 시대를 벗어나, DB 내부의 머신러닝 엔진이 아침엔 OLTP(SGA) 위주로 자원을 밀어주고 야간엔 배치(PGA)용으로 메모리 벽을 허물어 자유롭게 크기를 늘렸다 줄이는 자율 주행 메모리 튜닝이 표준화되고 있다.

- **📢 섹션 요약 비유**: RDBMS 메모리 아키텍처 최적화는 단순히 RAM 용량을 늘리는 무식한 하드웨어 추가가 아니라, 요리사(CPU)들이 주방에서 한 치의 동선 낭비와 다툼(Latch) 없이 예술적으로 협력하게 만드는 궁극의 주방 설계 철학입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **비용 기반 옵티마이저 (CBO)와 실행 계획**: SGA의 공유 풀(Shared Pool)에서 하드 파싱이 일어날 때, 수천 개의 실행 계획 후보군 중 무엇을 골라 메모리에 남길지 결정하는 두뇌 메커니즘.
- **버퍼 풀과 WAL (Write-Ahead Logging)**: SGA 공간 중 가장 핵심인 '데이터 버퍼 캐시'와 '리두 로그 버퍼'가 전원이 꺼져도 날아가지 않도록 보장하는 디스크 I/O 융합 기술.
- **객체 관계 매핑 (ORM)과 MyBatis**: 바인드 변수 처리를 강제하여 하드 파싱과 SQL 인젝션 공격을 애플리케이션 프레임워크 단에서 손쉽게 막아주는 윗단의 아키텍처.

## 👶 어린이를 위한 3줄 비유 설명
1. **SGA(공용 메모리)**는 학교 도서관이에요. 철수가 힘들게 찾아놓은 정답지(실행 계획)를 도서관 책상에 올려두면(캐싱), 뒤에 온 영희는 고생 안 하고 0.1초 만에 베껴 쓸 수 있어서 반 전체 숙제 속도가 엄청 빨라져요! (소프트 파싱)
2. 하지만 **하드 파싱**은 매번 똑같은 숙제를 할 때마다 도서관에 정답이 있는지 안 찾고, 처음부터 끝까지 혼자 머리를 싸매며 새로 계산하는 아주 바보 같은 방법이랍니다.
3. **PGA(개인 메모리)**는 도서관 공용 책상이 아니라 나만 쓰는 내 방 책상이에요. 공용 책상에서 100만 개짜리 퍼즐(대규모 정렬)을 맞추면 친구들이 방해받고 싸움이 나니까, 크고 복잡한 퍼즐은 꼭 남의 간섭이 없는 내 방 책상(PGA)으로 가져와서 맞춰야 해요!
