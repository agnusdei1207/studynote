# B+Tree를 넘어서 - Fractal Tree, LSM-Tree, NVM 최적화 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 지난 50년간 RDBMS의 왕좌를 지켜온 B+Tree의 태생적 한계인 '중간 삽입 시 페이지 분할(Split)과 디스크 랜덤 I/O 병목'을 타파하기 위해, 쓰기(Write) 최적화에 모든 것을 건 **차세대 스토리지 엔진의 트리 구조 혁신**이다.
> 2. **가치**: **LSM-Tree**가 순차 쓰기(Append-only)로 쓰기 속도를 극대화했다면, **Fractal Tree(프랙탈 트리)**는 B-Tree의 구조를 유지하되 노드 내부에 '메시지 버퍼'를 두어 쓰기 속도를 100배 이상 높이고 데이터 파편화(Fragmentation)를 원천 차단하는 경이로운 아키텍처를 선보인다.
> 3. **융합**: 이러한 소프트웨어적 구조 혁신은, 전원이 꺼져도 날아가지 않고 디스크처럼 용량이 큰 **비휘발성 메모리(NVM, Non-Volatile Memory)**의 등장과 융합되어, 디스크와 메모리의 경계가 사라진 완전히 새로운 데이터베이스 아키텍처(예: Peloton DB)를 창조하고 있다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **LSM-Tree**: 디스크 랜덤 쓰기를 피하기 위해, 들어오는 데이터를 메모리에 쏟아붓고 꽉 차면 디스크에 순차적(Sequential)으로 합쳐버리는(Merge) 구조.
  - **Fractal Tree**: 수학의 프랙탈 기하학 원리를 트리 인덱스에 적용하여, 부모 노드에 임시 버퍼를 만들어 두고 자식 노드로 데이터를 아주 가끔씩만 '흘려보내는(Flush)' B-Tree의 진화형.

- **💡 비유**: B+Tree가 도서관 책장 사이에 억지로 새 책(Insert)을 끼워 넣느라 책들을 다 옆으로 밀어야 하는 고된 노동이라면, Fractal Tree는 책장 맨 위에 커다란 바구니(버퍼)를 두고 책을 던져넣다가, 바구니가 꽉 찰 때만 도서관 사서가 한 번에 뭉텅이로 책을 꽂아 넣는(배치 삽입) 똑똑한 요령입니다.

- **등장 배경**:
  1. **쓰기 절벽 (Write Cliff)**: B-Tree에 데이터가 1억 건 이상 쌓이면 트리의 깊이(Depth)가 깊어지고, 캐시 미스(Cache Miss)가 발생하여 초당 수만 건을 쓰던 DB가 갑자기 초당 수백 건 단위로 멈춰버리는 현상.
  2. **SSD의 수명(TBW) 문제**: SSD는 쓰기를 할 때마다 셀(Cell)이 마모된다. B-Tree의 덮어쓰기(In-place Update)는 하나의 값을 바꾸기 위해 8KB/16KB 블록 전체를 매번 지우고 다시 쓰는 쓰기 증폭(Write Amplification)을 유발해 비싼 SSD를 금방 망가뜨렸다.
  3. **빅데이터의 도래**: MySQL(InnoDB) 같은 범용 엔진만으로는 센서, 로그, 이벤트 등 초당 수백만 건씩 쏟아지는 데이터를 락(Lock) 없이 감당할 물리적 방법이 없었다.

- **📢 섹션 요약 비유**: 엔진이 감당할 수 없을 만큼 무거운 짐(빅데이터)을 실은 자동차가, 타이어(B-Tree)를 계속 갈아 끼우는 짓을 포기하고 아예 바퀴를 궤도형(LSM, Fractal) 무한궤도 탱크로 개조한 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. B-Tree 쓰기 증폭의 한계

- 10바이트짜리 데이터 하나를 `UPDATE` 하거나 `INSERT` 하기 위해, InnoDB는 그 데이터가 포함된 16KB짜리 전체 페이지(Page)를 메모리(Buffer Pool)로 퍼 올린다.
- 메모리에서 10바이트를 고친 뒤, 복구를 위해 Redo Log 16KB를 디스크에 쓰고, 나중에 원본 데이터 파일 16KB도 디스크에 통째로 덮어쓴다. 더블라이트 버퍼(Doublewrite Buffer)까지 쓰면 무려 32KB를 쓰게 된다.
- **결과**: 10바이트를 쓰기 위해 디스크를 32KB(약 3,000배 증폭)나 긁어먹는 **쓰기 증폭(Write Amplification)**의 끔찍한 비효율이 발생한다.

+++

### 2. Fractal Tree (프랙탈 트리)의 마법 (TokuDB)

B-Tree의 엄청난 읽기 성능을 유지하면서, LSM-Tree의 쓰기 성능을 결합하기 위해 MIT 공대에서 고안한 구조다. (MySQL의 TokuDB 엔진에 탑재)

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 Fractal Tree의 내부 버퍼(Message Buffer) 구조           │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │                     [ Root Node ]                                │
  │        [ 데이터 배열 (Keys) ] + [ ⭐️ Message Buffer (큐) ]           │
  │                  /                       \                       │
  │       [ Branch Node 1 ]              [ Branch Node 2 ]           │
  │  [ 데이터 배열 ] + [ Message Buffer ]   [ 데이터 배열 ] + [ 버퍼 ]     │
  │             /                                                    │
  │     [ Leaf Node (실제 데이터 블록) ]                                  │
  │                                                                  │
  │  [ 동작 원리 (Insert 99) ]                                          │
  │  1. 데이터 '99' 삽입 시, Leaf 노드까지 힘들게 디스크를 타고 내려가지 않음.    │
  │  2. 그냥 Root 노드의 [Message Buffer]에 "99 삽입" 메모만 꽂아두고 완료 응답!│
  │  3. 나중에 버퍼가 꽉 차면, 아래쪽 Branch Node 버퍼로 한 번에 주르륵 쏟아냄. │
  │     (마치 폭포수처럼 묶음 단위로 하달됨 -> 디스크 I/O 극단적 절약)           │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** B-Tree는 삽입 시 무조건 맨 아래쪽의 Leaf 노드를 찾아 디스크를 읽어야(Random I/O) 했다. 하지만 Fractal Tree는 각 노드 옆에 커다란 양동이(Message Buffer)를 매달아 두었다. 데이터가 들어오면 양동이에 대충 던져놓고 끝낸다. 양동이가 꽉 차면 그제야 밑에 있는 노드들의 양동이로 물을 쏟아붓듯(Cascade) 일괄 전송한다. 이 덕분에 디스크 접근 횟수가 수백 분의 일로 줄어들고, 데이터가 조각나지 않아 **LSM-Tree의 고질병인 콤팩션(Compaction) 부하가 발생하지 않는** 기적의 성능을 낸다.

+++

### 3. NVM (Non-Volatile Memory) 최적화 아키텍처

DRAM처럼 빠르고 바이트(Byte) 단위 접근이 가능하면서도, 전원이 꺼져도 데이터가 날아가지 않는 인텔 옵테인(Optane) 같은 비휘발성 메모리 시대의 새로운 DB 구조다.

- **이중화 철폐**: 전통 DB는 디스크가 느리니까 메모리(Buffer Pool)에 복사하고, 메모리는 정전되면 날아가니까 로그(WAL)를 디스크에 또 쓰는 끔찍한 복제 낭비를 겪었다.
- **NVM 아키텍처 (Peloton DB 등)**: 데이터 원본 자체를 NVM에 올려버린다. 디스크도 없고, 버퍼 풀도 없다. 그냥 NVM 자체가 유일한 원본이자 메모리다. **OS의 블록 I/O(4KB/8KB)를 거치지 않고, CPU가 직접 데이터베이스의 주소를 바이트 단위로 찌르는(Direct Access)** 신세계가 열린다.

- **📢 섹션 요약 비유**: NVM의 등장은 요리사(CPU)가 지하 창고(디스크)와 도마(메모리)를 번갈아 오가던 짓을 그만두고, 아예 '절대 썩지 않는 마법의 도마(NVM)' 위에서 재료를 평생 썰고 보관하는 혁명입니다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### 차세대 스토리지 엔진 트리 구조 비교

| 비교 항목 | B+Tree (InnoDB, Oracle) | LSM-Tree (RocksDB, Cassandra) | Fractal Tree (TokuDB) |
|:---|:---|:---|:---|
| **설계 철학** | 조회(Read) 성능 최적화 | 쓰기(Write) 성능 극대화 | Read와 Write의 완벽한 밸런스 |
| **삽입 (Insert)** | 무거운 Random I/O 발생 (느림) | 메모리 버퍼 후 순차 디스크 I/O (매우 빠름) | 노드 내 버퍼로 흡수 (매우 빠름) |
| **조회 (Read)** | O(log N)으로 가장 빠르고 안정적 | 여러 SSTable 파일 스캔 후 병합 (상대적으로 느림) | 버퍼에 있는 값까지 머지해서 리턴 (B-Tree와 유사) |
| **백그라운드 부하** | 잦은 페이지 분할(Split) 부하 | **치명적인 콤팩션(Compaction) CPU 부하** | 노드 간 버퍼 플러시 (부하 적음) |
| **SSD 수명(TBW)** | 쓰기 증폭 심함 (수명 단축) | 쓰기 증폭 중간 수준 | **쓰기 증폭 없음 (SSD 수명 대폭 연장)** |

### 과목 융합 관점

- **자료구조 (Data Structure)**: Fractal Tree는 본질적으로 '지연 업데이트를 허용하는 B-Tree(B-Tree with Buffered Updates)'이다. 트리 구조의 수학적 균형(Balance)은 유지하되 연산의 물리적 실행 시점만 지연(Lazy Evaluation)시키는 소프트웨어 공학 패턴의 절정이다.
- **클라우드 스토리지 (EBS/S3)**: LSM-Tree 구조는 디스크에 한 번 써진 파일(SSTable)을 절대 수정하지 않는 불변성(Immutability)을 가지므로, AWS S3와 같은 '수정 불가능한 객체 스토리지' 위에 DB 데이터를 그대로 올려버리는(Data Lakehouse) 현대 클라우드 네이티브 아키텍처와 100% 철학적으로 융합된다.

- **📢 섹션 요약 비유**: B+Tree는 튼튼하지만 너무 무거워 연비가 나쁜 옛날 자동차고, LSM-Tree는 직선 주로에서 빛의 속도로 달리는 로켓이라면, Fractal Tree는 두 마리 토끼를 다 잡은 도심형 하이브리드 스포츠카입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 로그 분석 플랫폼의 스토리지 한계 타파

- **문제 상황**: 하루 10TB씩 쏟아지는 웹 접근 로그를 MySQL(InnoDB)에 넣고 있었음. InnoDB는 100GB가 넘어가자 B-Tree 인덱스 깊이가 커져 쓰기 성능이 초당 100건으로 폭락했고, SSD는 엄청난 쓰기 증폭(Write Amplification) 때문에 6개월 만에 수명을 다해 교체 비용만 수억 원이 발생.
- **의사결정 (Fractal Tree/LSM 엔진으로 마이그레이션)**:
  1. 스토리지 엔진을 플러그인 형태로 MyRocks(LSM-Tree) 또는 TokuDB(Fractal Tree)로 교체. 애플리케이션의 SQL 소스 코드는 단 1줄도 수정할 필요 없음(MySQL의 Pluggable Storage Engine 장점).
  2. B-Tree 분할 시 발생하는 디스크 여유 공간(빈 공간)이 사라져, 압축 효율이 20배(10TB -> 500GB)로 극대화됨.
  3. SSD 쓰기 증폭이 사라져 하드웨어 수명이 5년 이상으로 연장되고, 초당 10만 건 이상의 INSERT 성능을 끝까지 무정지로 유지함.

### 도입 체크리스트
- **Read/Write 비율 (Workload Profiling)**: 시스템의 쿼리가 조회(SELECT) 90%, 수정(UPDATE) 10%의 전형적인 OLTP라면 B+Tree(InnoDB)를 절대 버려선 안 된다. LSM-Tree나 Fractal Tree는 쓰기 비율이 압도적으로 높거나(로그, 센서), 데이터 용량이 너무 커서 압축이 절실할 때만 꺼내 드는 비장의 카드다.
- **CPU 코어 자원의 여유**: LSM-Tree나 Fractal Tree는 디스크 I/O 병목을 해결한 대신, 데이터를 백그라운드에서 끊임없이 압축(Compression)하고 병합(Merge)하는 일에 CPU를 극도로 혹사시킨다. 클라우드 인스턴스의 CPU 스펙이 낮다면 디스크 대기보다 무서운 CPU 100% 프리징에 걸리게 된다.
- **Tombstone (삭제 마커) 감당 능력**: LSM-Tree는 `DELETE`를 하면 데이터가 바로 지워지지 않고 삭제되었다는 마커(Tombstone)만 남긴 채 며칠을 떠돈다. 이로 인해 쿼리를 돌렸을 때 지워진 데이터까지 디스크에서 퍼 올리느라 속도가 처참하게 느려진다. 잦은 업데이트나 삭제가 필요한 도메인에는 절대 금물이다.

### 안티패턴: "압축률 짱"이라며 핀테크 코어 원장에 LSM-Tree 도입
- 비트코인 거래소나 은행 코어 원장에서 디스크 비용을 아끼겠다고, 잔액이 1초에도 수십 번씩 업데이트되는 계좌 테이블을 RocksDB(LSM-Tree)에 올려버리는 미친 짓. 잔액을 업데이트할 때마다 파일이 계속 뒤에 덧붙여(Append) 생성되므로, "내 현재 잔액이 얼마야?"라고 물었을 때 디스크에 흩어진 수십 개의 최신 버전을 합쳐서 계산하느라 밀리초 단위의 핀테크 응답성을 완전히 상실한다.

- **📢 섹션 요약 비유**: 쓰기 최적화 엔진(LSM)은 한 번 그린 그림을 절대 수정하지 않는 '유화 캔버스'와 같습니다. 썼다 지웠다를 반복하는 '수학 문제 풀이 연습장(계좌 잔액)'에 유화 물감을 들이대면 아무것도 알아볼 수 없는 쓰레기가 됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | B+Tree (InnoDB) | Fractal Tree / LSM-Tree (RocksDB) | 개선 효과 |
|:---|:---|:---|:---|
| **저장 용량 압축률** | 빈 페이지로 인한 파편화 심각 (압축 한계) | 블록 빈 공간 없음, 극강의 Zlib 압축 | **디스크 스토리지 용량 10배~20배 압축 절감** |
| **대용량 데이터 쓰기** | 데이터 증가 시 Write 성능 수직 하락 절벽 | 1TB를 넘겨도 초기 Write 속도 100% 유지 | **초대용량(VLDB) 환경에서의 병목 없는 쓰기 100배 향상** |
| **SSD 수명 연장** | 사소한 변경에도 16KB 블록 전체 덮어쓰기 | 변경분만 순차적으로 추가 기록 (No in-place) | **플래시 메모리(SSD) 마모율 1/10 수준으로 억제** |

### 미래 전망
- **NVM(비휘발성 메모리) 네이티브 구조의 상용화**: Intel의 사업 철수로 NVM 시장이 주춤했지만, CXL(Compute Express Link) 인터페이스의 등장으로 RAM과 NVMe SSD의 경계가 무너지고 있다. 미래의 데이터베이스는 디스크 I/O 코드를 커널에서 모조리 삭제하고, 메모리 주소 체계만으로 데이터를 영속화하는 완전한 '제로 스토리지 엔진'으로 다시 태어날 것이다.
- **분산 블록체인(Blockchain) 생태계의 표준**: RocksDB(LSM-Tree) 등은 무결성과 Append-only(추가만 가능)를 생명으로 하는 비트코인 등 블록체인의 원장(Ledger) 스토리지 엔진으로 완벽하게 편입되어, 중앙 통제가 없는 탈중앙 데이터 아키텍처를 뒤에서 묵묵히 받쳐주고 있다.

- **📢 섹션 요약 비유**: B+Tree를 넘어서는 차세대 엔진들의 진화는, 인류가 말이 끄는 마차(디스크 기반)를 버리고 제트 엔진(메모리 중심)을 달아 우주(페타바이트 빅데이터)로 날아가기 위해 필연적으로 거쳐야 하는 발사체 혁명입니다.

- **특수 목적 인덱스 아키텍처**: Tree 구조 자체의 한계를 우회하기 위해 Hash나 비트 연산을 차용한 또 다른 형태의 스토리지 최적화 파생 가지.

## 👶 어린이를 위한 3줄 비유 설명
1. 빽빽한 책장(B-Tree)에 새 책을 꽂으려면 다른 책들을 다 옆으로 밀어야 해서 팔이 너무 아프고 오래 걸려요. (쓰기 지연)
2. 그래서 **LSM-Tree**는 새 책이 오면 책장에 안 꽂고 그냥 방바닥에 차곡차곡 쌓아만 둬서 엄청나게 빨리 책을 치울 수 있어요. (나중에 한 번에 정리함)
3. **프랙탈 트리(Fractal Tree)**는 책장 칸마다 '작은 바구니'를 둬서, 책을 바구니에 막 던져 넣다가 바구니가 꽉 차면 밑으로 와르르 쏟아내며 자동으로 정리되는 아주 똑똑한 마법 책장이랍니다!
