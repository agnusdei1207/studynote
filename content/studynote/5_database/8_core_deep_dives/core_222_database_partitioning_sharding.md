# 데이터베이스 파티셔닝과 샤딩의 심층 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파티셔닝(Partitioning)은 단일 데이터베이스 서버 내에서 거대한 논리적 테이블을 작고 관리하기 쉬운 여러 물리적 파일로 분할하는 기술이며, 샤딩(Sharding)은 데이터를 여러 대의 독립된 분산 노드(Server)로 쪼개어 배치하는 수평적 확장(Scale-out) 기술이다.
> 2. **가치**: 파티셔닝은 파티션 프루닝(Partition Pruning)을 통해 쿼리 스캔 범위를 극단적으로 줄이고 데이터 폐기(Drop Partition) 속도를 높이며, 샤딩은 특정 노드에 트래픽이 몰리는 핫스팟(Hotspot)을 분산시켜 트랜잭션 처리량(TPS)과 스토리지 용량을 무한히 확장한다.
> 3. **융합**: 이 두 기술은 분산 NoSQL(MongoDB, Cassandra) 아키텍처의 척추가 되며, 노드 증설 시 데이터 이동을 최소화하는 일관된 해싱(Consistent Hashing) 알고리즘과 융합되어 클라우드 네이티브의 탄력성(Elasticity)을 완성한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **파티셔닝 (Vertical/Horizontal Partitioning)**: 논리적으로는 하나의 테이블이지만, 디스크 상에는 여러 개의 파티션 공간에 분산 저장하여 가용성과 관리 용이성을 높이는 기법. (단일 DB 인스턴스 내)
  - **샤딩 (Sharding)**: 파티셔닝의 수평적 형태를 물리적으로 분리된 여러 데이터베이스 서버(Node)로 확장한 기법. (Shared-Nothing 아키텍처)

- **💡 비유**: 파티셔닝이 "하나의 거대한 도서관 건물을 층별로(역사, 과학, 소설) 나누어 책을 찾는 시간을 줄이는 것"이라면, 샤딩은 "도서관 건물 하나로는 더 이상 책을 수용할 수 없어, 서울, 부산, 대전 등 전국 각지에 도서관(노드)을 새로 짓고 책을 분산 배치하는 것"과 같습니다.

- **등장 배경**:
  1. **VLDB(Very Large Database)의 관리 한계**: 테이블 크기가 테라바이트(TB) 급으로 커지자, 인덱스를 재구성(Rebuild)하거나 오래된 데이터를 삭제(`DELETE`)할 때 락(Lock)이 걸리고 트랜잭션 로그(Undo/Redo)가 폭주하여 시스템이 마비되었다.
  2. **Scale-up(수직적 확장)의 비용 벽**: 트래픽을 감당하기 위해 DB 서버의 CPU와 RAM을 고급 기종으로 올리는 방식은 한계 체감 법칙에 부딪혔고 비용이 기하급수적으로 증가했다.
  3. **가용성과 장애 격리**: 단일 서버가 죽으면 모든 서비스가 멈추는 구조(SPOF)를 탈피하기 위해, 데이터베이스를 물리적으로 쪼개어 일부 노드가 죽어도 전체 서비스는 살아남게 할 아키텍처가 필요했다.

- **📢 섹션 요약 비유**: 커다란 통나무 통 하나에 물을 담으면 나중에 비우기도 힘들고 통이 깨지면 물이 다 쏟아지지만(단일 DB), 작은 생수통 100개에 물을 나눠 담으면(파티셔닝/샤딩) 하나를 버리기도 쉽고 한 개가 터져도 남은 물은 보존되는 이치입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 파티셔닝의 기준과 기법 (Partitioning Methods)

데이터를 어떤 기준으로 자를 것인가에 대한 전략이다.

| 분할 기법 | 동작 원리 | 장점 | 주 사용처 |
|:---|:---|:---|:---|
| **Range (범위)** | 날짜나 숫자 범위를 기준으로 분할 (예: 2026년 1월, 2월 파티션) | 이력 데이터의 삭제(Drop Partition)와 범위 검색에 압도적 유리 | 주문 이력, 로그, 시계열 데이터 |
| **Hash (해시)** | 파티션 키에 해시 함수를 적용해 결과값에 따라 분할 | 데이터가 균등하게 분산되어 특정 파티션에 병목(Hotspot) 집중 방지 | 고객 ID 기반의 균등 분산 |
| **List (리스트)** | 명시적인 목록의 값으로 분할 (예: 지역=서울, 지역=부산) | 특정 카테고리별로 물리적 보관 위치를 명확히 분리 가능 | 지역별, 지점별 매출 데이터 |
| **Composite (복합)** | Range로 먼저 나누고, 그 안에서 다시 Hash로 서브 파티셔닝(Sub-partition) 수행 | 이력 관리(Range)와 병목 방지(Hash)의 장점을 융합 결합 | 수십억 건 이상의 초대용량 VLDB |

### 2. 파티션 프루닝 (Partition Pruning)과 인덱스 전략

옵티마이저가 쿼리 조건(WHERE)을 분석하여, **결과가 절대 존재하지 않을 파티션은 아예 스캔 대상에서 제외(Pruning, 가지치기)**하는 핵심 성능 최적화 기술이다.

- **로컬 인덱스 (Local Index)**: 각 파티션 파일마다 독립적으로 생성된 인덱스. 파티션을 통째로 삭제(Drop)하거나 추가해도 다른 파티션의 인덱스에 전혀 영향을 주지 않아 유지보수성이 극대화된다.
- **글로벌 인덱스 (Global Index)**: 파티션 분할과 무관하게 전체 데이터를 아우르는 단일 통뼈 인덱스. 검색은 빠를 수 있으나, 파티션 하나를 삭제하면 전체 인덱스가 무효화(Unusable)되어 재생성 오버헤드가 발생한다.

+++

### 3. 샤딩 (Sharding) 아키텍처와 라우팅

샤딩은 어플리케이션이나 중간 라우터(Router)가 어떤 데이터가 어느 노드(Shard)에 있는지 알고 찾아가야 한다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 샤딩 환경에서의 쿼리 라우팅 (Router Architecture)      │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │     [Client App] ──▶ SELECT * FROM User WHERE User_ID = 852;       │
  │                                                                  │
  │        ┌─────────────▼─────────────┐                             │
  │        │   Query Router (Proxy)    │ ◀── 조회 (Config/Meta DB)     │
  │        │   (Shard Key = User_ID)   │     "ID 800~900은 Shard 3에 있음" │
  │        └─────────────┬─────────────┘                             │
  │                      │                                           │
  │      ┌───────────────┼──────────────────────────────┐            │
  │      ▼               ▼                              ▼            │
  │  [ Shard 1 ]      [ Shard 2 ]                  [ Shard 3 ]       │
  │ (User 1~300)     (User 301~600)               (User 601~1000)    │
  │                                                    │             │
  │                                                 [결과 반환]         │
  │                                                                  │
  │  - 샤드 키(Shard Key) 부재 쿼리 (예: WHERE name='홍길동'):               │
  │    모든 샤드에 쿼리를 뿌린(Scatter) 뒤 결과를 모으는(Gather) 최악의 오버헤드 발생.│
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 샤딩 시스템의 핵심은 **샤드 키(Shard Key, 파티션 키)**의 선정이다. 클라이언트가 쿼리에 샤드 키(`User_ID`)를 포함해 전송하면, 라우터는 메타데이터를 참조하여 오직 `Shard 3` 노드 하나에만 쿼리를 전달(Targeted Query)하여 1ms 이내에 응답을 받는다. 하지만 샤드 키가 없는 조건(`name='홍길동'`)으로 검색하면, 라우터는 모든 샤드(1, 2, 3)에 쿼리를 날리고(Scatter), 각 샤드의 응답을 기다렸다가 취합(Gather)해야 하므로 성능이 기하급수적으로 저하(Scatter-Gather 문제)된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 파티셔닝 vs 샤딩 심층 비교

| 비교 항목 | 파티셔닝 (Partitioning) | 샤딩 (Sharding) |
|:---|:---|:---|
| **저장 위치** | 단일 데이터베이스 (동일 머신 내 디스크 분할) | 물리적으로 완전히 분리된 여러 DB 서버 (노드 분할) |
| **목적** | 디스크 I/O 최적화, 관리/백업 용이성, 데이터 삭제 속도 | CPU/RAM 한계 극복, 글로벌 스케일 분산, 트래픽 부하 분산 |
| **조인 (Join)** | DBMS 내부 엔진이 알아서 완벽하게 Join 처리 | **서로 다른 샤드 간의 Join은 원칙적으로 불가능** (네트워크 비용 막대) |
| **트랜잭션** | 단일 노드 내 ACID 트랜잭션 완벽 보장 | 분산 트랜잭션(2PC) 필요로 성능 저하 혹은 결과적 일관성 타협 |
| **운영 주체** | DBA의 스키마 및 테이블스페이스 설정 | 애플리케이션 아키텍처 또는 분산 NoSQL 미들웨어 |

### 과목 융합 관점

- **운영체제 (OS) 및 스토리지**: 파티셔닝 시 각 파티션(예: 올해 데이터 파티션, 작년 파티션)을 서로 다른 디스크(테이블스페이스) 컨트롤러에 매핑하면, OS 레벨에서 디스크 헤드가 물리적으로 병렬 탐색을 수행하여 I/O 처리량이 배가되는 하드웨어적 최적화가 완성된다.
- **알고리즘 (Algorithm)**: 샤드 클러스터에 노드를 추가하거나 제거할 때, 일반 Hash 함수(`Hash(Key) % N`)를 쓰면 전체 데이터의 90%를 다른 서버로 옮겨야 하는 대재앙이 발생한다. 이를 막기 위해 **일관된 해싱 (Consistent Hashing)** 구조를 사용하여 이웃 노드로 K/N의 데이터만 이동시키도록 네트워크 오버헤드를 극소화한다.

- **📢 섹션 요약 비유**: 파티셔닝이 한 집 안에서 방을 여러 개로 쪼개어 물건을 찾는 시간을 줄이는 인테리어 공사라면, 샤딩은 아예 집을 여러 채 사서 물건을 각지에 분산 배치해두고 택배망으로 연결하는 부동산 증식과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 배달 앱의 샤딩 전략과 핫스팟 해결

- **문제 상황**: 서울 강남구의 금요일 저녁 배달 트래픽이 폭주하여, '지역코드(Region_ID)'를 샤드 키로 설정한 데이터베이스에서 강남구 샤드(Shard-A)만 CPU 100%를 치고 다운됨. 반면 제주도 샤드(Shard-B)는 놀고 있음 (데이터 핫스팟 문제).
- **의사결정 (복합 샤드 키 적용)**:
  1. **해시 샤딩 도입 검토**: 주문 ID(Order_ID) 해시값으로 샤딩하면 데이터는 100% 균등하게 찢어지지만, "강남구의 배달 기사 위치 목록"을 조회할 때 모든 샤드를 찔러야 하는 Scatter-Gather 지연 발생.
  2. **복합 키 (Compound Shard Key) 적용**: `[지역코드] + [가입일의 해시값]` 두 개를 결합하여 샤드 키로 지정. 같은 강남구 데이터라도 내부적으로 다시 여러 샤드에 균등하게 분산되게 만들면서도, 특정 지역 검색 시 최대 스캔할 샤드의 갯수를 제한(Bound)하여 핫스팟과 검색 지연을 동시에 잡음.

### 도입 체크리스트
- **비즈니스 성장성 (샤딩 도입 전)**: 샤딩은 한 번 적용하면 되돌리기 매우 어려운 "루비콘 강"이다. 읽기 전용 복제본(Read Replica) 추가나 인메모리 캐시(Redis), 쿼리 튜닝으로 현재의 트래픽을 해결할 수 없는지 끝까지 검증했는가?
- **파티션 Drop 전략 (파티셔닝)**: `DELETE FROM Logs WHERE date < '2025-01-01'` 쿼리는 수천만 건의 Undo 로그를 생성하며 DB를 죽인다. 대신 2025년 이전 파티션 파일을 통째로 날려버리는 `ALTER TABLE DROP PARTITION` 구문이 매월 1일 자정에 0.1초 만에 실행되도록 아키텍처가 짜여 있는가?
- **오토 인크리먼트 (Auto Increment) 배제**: 분산 샤드 환경에서는 서로 다른 노드가 동시에 PK 1번을 생성할 수 있다. 중앙화된 시퀀스 서버(Ticket Server)를 두거나, 트위터의 Snowflake ID(타임스탬프+워커ID 결합) 같은 유일무이한 식별자 체계를 도입했는가?

### 안티패턴: 샤딩된 환경에서의 분산 조인(Distributed Join) 강행
- 서로 다른 샤드 서버에 있는 `User` 테이블과 `Order` 테이블을 조인하는 쿼리를 애플리케이션에서 무리하게 던지는 행위. 이는 양쪽 DB의 모든 데이터를 네트워크로 끌어와 웹 서버(WAS) 메모리에서 조인해야 하므로 극악의 성능을 낸다. 샤딩 환경에서는 데이터를 중복 저장하더라도(역정규화) 한 샤드 안에서 조회(Local Query)가 끝나도록 데이터 모델링 자체를 갈아엎어야 한다.

- **📢 섹션 요약 비유**: 물건을 여러 창고(샤드)에 나눠 뒀으면 한 번 갈 때 한 창고에서 필요한 걸 다 가져오도록 미리 세트로 포장해 놔야지(역정규화), 조립할 때마다 A 창고 바퀴와 B 창고 핸들을 가져오라 시키면(분산 조인) 물류비로 회사가 파산하게 됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 단일 통짜 테이블 구조 | 파티셔닝 / 샤딩 구조 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **데이터 폐기 속도** | 대량 DELETE 쿼리로 수 시간 소요 및 Lock | 파티션 Drop(DDL)으로 0.1초 완료 | **유지보수 시간 99% 단축 및 무정지 운영** |
| **검색 범위 (I/O)** | 테라바이트급 테이블 Full Scan 발생 | 파티션 프루닝으로 타겟 파일만 스캔 | **조회 응답 속도 수백 배 향상** |
| **시스템 한계 극복**| Scale-up 비용 한계 봉착 (서버 1대 최대 스펙) | 저가형 Commodity 서버 N대 증설 (Scale-out) | **트래픽 확장성(Scalability) 선형적 무한 보장** |

### 미래 전망
- **NewSQL의 자동 샤딩 (Auto-Sharding)**: 구글 스패너(Spanner)나 TiDB 같은 현대 분산 관계형 DB는, 관리자가 샤드 키를 짜내고 앱 로직을 수정할 필요 없이 백그라운드에서 CPU 부하와 데이터 용량을 감지하여 데이터를 자동으로 분리(Split)하고 이웃 노드로 이동(Migration)시키는 자율 주행 분산 시스템으로 진화했다.
- **클라우드 네이티브 서버리스(Serverless) DB**: 샤드의 개념이 완전히 추상화되어, 개발자는 단순히 엔드포인트 URL 하나만 바라보고, 데이터베이스 엔진이 트래픽의 증감에 따라 수백 개의 컴퓨팅 노드를 초 단위로 늘렸다가 줄이는(Scale-to-Zero) 극한의 탄력성(Elasticity) 시대가 열렸다.

- **📢 섹션 요약 비유**: 파티셔닝과 샤딩 기술은 IT 인프라의 '세포 분열' 기술입니다. 하나의 거대한 단세포(단일 DB) 생물이 쪼개지고 확장되어 복잡하고 정교한 다세포 유기체(분산 클라우드 데이터베이스)로 진화하는 생명 연장의 비결입니다.
