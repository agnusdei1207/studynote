# 최신 데이터베이스 최적화 기술 - LSM-Tree 콤팩션, 티어링, 델타 인코딩

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 고성능 분산 데이터베이스는 디스크 I/O 병목과 스토리지 비용의 물리적 한계를 극복하기 위해, **쓰기 최적화(LSM-Tree), 무손실 극강 압축(Delta Encoding), 계층형 저장(Tiering)** 메커니즘을 커널 깊숙이 내재화하고 있다.
> 2. **가치**: 콤팩션(Compaction)은 데이터 파편화를 막아 읽기 속도를 유지하고, 델타 인코딩은 시계열 데이터의 용량을 수십 분의 일로 줄인다. 티어링(Tiering)은 고가의 RAM/SSD와 저렴한 클라우드 오브젝트 스토리지(S3)를 투명하게 연결하여 무한한 확장을 가능케 한다.
> 3. **융합**: 이 최적화 기술들은 NoSQL(Cassandra), 인메모리 DB(Redis), 시계열 DB(InfluxDB) 등 이기종 아키텍처 전반에 걸쳐 공통적으로 융합되어, 클라우드 네이티브 서버리스(Serverless) 데이터베이스의 비용 효율성을 극대화하는 핵심 원동력이 된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 데이터베이스 시스템에서 초당 수백만 건의 읽기/쓰기 성능을 보장하면서도 페타바이트(PB) 규모의 저장 비용을 최소화하기 위해 스토리지 엔진과 인덱스 레벨에 적용된 고도의 하드웨어/소프트웨어 통합 최적화 기법들이다.

- **💡 비유**: 최적화 기술은 초거대 물류 창고의 '공간 관리 마법'과 같습니다. 매일 들어오는 택배를 빈 공간에 대충 쌓지 않고 차곡차곡 압축해서 묶고(LSM-Tree & 콤팩션), 모양이 비슷한 물건은 포장지를 벗겨서 합치며(델타 인코딩), 자주 안 찾는 물건은 임대료가 싼 외곽 창고로 몰래 빼두는(스토리지 티어링) 완벽한 자동화 로봇입니다.

- **등장 배경**:
  1. **B-Tree의 쓰기 절벽(Write Cliff)**: B-Tree는 데이터가 삽입될 때마다 디스크 상의 노드를 분할(Split)하고 이동시켜야 하는 랜덤 I/O(Random I/O) 병목이 존재하여 빅데이터 환경의 엄청난 쓰기 트래픽을 감당할 수 없었다.
  2. **시계열/센서 데이터의 폭발**: 1초마다 0.1도씩 오르는 온도 센서 데이터를 매번 8Byte Float형으로 저장하는 것은 막대한 낭비였고, 데이터 특성에 맞는 특수 압축 알고리즘이 절실해졌다.
  3. **메모리(RAM)와 SSD 비용 한계**: 메모리 중심의 아키텍처가 대세가 되었지만, 수십 TB의 RAM 클러스터를 유지하는 것은 불가능에 가까운 비용을 초래하여 데이터의 온도(Hot/Cold)에 따른 물리적 격리가 필요해졌다.

- **📢 섹션 요약 비유**: 마치 스마트폰 용량이 꽉 차면 안 쓰는 사진을 클라우드로 알아서 올리고(Tiering), 화질 손상 없이 압축률을 높이는(Encoding) 것처럼, 데이터베이스도 생존을 위한 다이어트 기술을 스스로 갖추게 된 것입니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. LSM-Tree 콤팩션 (Compaction)과 툼스톤 (Tombstone)

LSM-Tree(Log-Structured Merge-Tree)는 디스크의 순차 쓰기(Sequential Write)를 활용해 압도적인 쓰기 성능을 내지만, 시간이 지나면 파편화되어 읽기 속도가 저하된다. 이를 해결하는 백그라운드 프로세스가 **콤팩션**이다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 LSM-Tree Compaction (병합 및 정리) 원리              │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ Level 0 (메모리 플러시 직후 SSTable) ]                            │
  │  SSTable 1: [A:1], [B:2], [C:3 (Tombstone)]                      │
  │  SSTable 2: [A:5], [D:4]                                         │
  │     │                                                            │
  │     ▼  (백그라운드 병합 프로세스 수행)                                 │
  │                                                                  │
  │  [ Level 1 (Compacted SSTable) ]                                 │
  │  Merged SSTable: [A:5 (최신)], [B:2], [D:4]                       │
  │                                                                  │
  │  - 메커니즘: 여러 개의 불변(Immutable) 파일을 읽어, 중복된 키(A:1)는 버리고  │
  │    최신 키(A:5)만 살리며, 삭제 마커(Tombstone C)를 가진 데이터는 물리적 폐기.│
  │  - 효과: 디스크 용량 확보 및 읽기 시 스캔해야 할 파일 수 최소화 (Read 최적화)│
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터베이스가 갱신(`UPDATE A=5`)이나 삭제(`DELETE C`) 요청을 받으면, 원본 데이터를 지우지 않고 무조건 새로운 파일(SSTable 2)에 덧붙여 쓴다(Append-only). 삭제의 경우에도 데이터를 지우는 대신 `C: 삭제됨(Tombstone)`이라는 특별한 꼬리표를 붙여 저장한다. 이후 **콤팩션(Compaction)** 프로세스가 백그라운드에서 실행되며 흩어진 여러 파일을 정렬된 하나의 거대 파일로 병합(Merge)한다. 이 과정에서 오래된 과거 버전이나 Tombstone이 붙은 데이터가 물리적으로 소멸되며, 결과적으로 읽기 성능(Read Latency)이 극적으로 회복된다.

### 2. 델타 인코딩 (Delta Encoding)과 Gorilla 알고리즘

주로 시계열 데이터베이스(TSDB)에서 극강의 압축률을 달성하기 위해 사용하는 기법이다. (Facebook의 Gorilla 백서에서 유래)

- **원리 (Delta of Delta)**: 타임스탬프나 센서 데이터는 일정한 주기로 증가하는 특성이 있다. 예를 들어 타임스탬프 `[1000, 1010, 1020, 1030]`이 있다면, 절대값을 저장하는 대신 그 차이(Delta)인 `[1000, +10, +10, +10]`을 저장한다. 나아가 차이의 차이(Delta of Delta)를 구하면 `[1000, 10, 0, 0]`이 되어, 대부분의 데이터가 '0'이 된다. '0'은 단 1비트만으로 압축할 수 있다.
- **결과**: 기존 16바이트(128비트)를 차지하던 타임스탬프와 센서 데이터 1쌍이, 평균 1.37바이트(약 10비트)로 줄어든다. 스토리지 비용이 **90% 이상 절감**된다.

+++

### 3. 스토리지 계층화 (Storage Tiering)

데이터의 '온도(접근 빈도)'에 따라 값비싼 저장 매체와 저렴한 저장 매체 사이를 데이터가 투명하게 오가는 아키텍처다.

1. **Hot Data**: 최근 7일 내 데이터. 가장 많이 검색되므로 초고가인 RAM(인메모리)이나 NVMe SSD에 배치.
2. **Warm Data**: 7일~30일 데이터. 가끔 검색되므로 범용 SSD(EBS)에 배치.
3. **Cold Data**: 30일 이상 지난 과거 로그. 보존 의무만 있으므로 가장 저렴한 클라우드 객체 스토리지(S3, Blob)에 배치.

- **핵심**: 사용자는 데이터가 S3에 있든 RAM에 있든 동일한 SQL을 던지면 된다. DB 엔진 내부의 가상화 계층이 S3에서 데이터를 스트리밍으로 끌어와(Fetch) 알아서 계산해 준다.

- **📢 섹션 요약 비유**: 티어링(Tiering)은 입는 겨울옷은 옷장(SSD)에 두고, 안 입는 여름옷은 지하 창고(S3)에 넣어두지만, 옷차림 앱(DB 쿼리)에는 모든 옷이 한 번에 검색되는 마법의 옷장 시스템입니다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### 인덱스 및 스토리지 구조 심층 비교

| 비교 항목 | B+Tree (전통적 RDBMS) | LSM-Tree (NoSQL / 최신 DB) |
|:---|:---|:---|
| **저장 방식** | In-place Update (해당 블록을 찾아 덮어쓰기) | Append-only (무조건 뒤에 덧붙여 쓰기) |
| **쓰기 속도** | 느림 (랜덤 I/O 및 노드 분할 오버헤드) | **초고속 (순차 I/O 및 메모리 버퍼 활용)** |
| **읽기 속도** | 초고속 (O(log N) 탐색, 깊이 일정함) | 상대적으로 느림 (여러 SSTable 스캔 및 병합 필요) |
| **공간 효율성** | 파편화로 인해 공간 낭비 발생 (Fill factor 60~70%) | 콤팩션 후 극도의 밀집도 (Fill factor 100%) |
| **주 사용처** | MySQL, Oracle (범용 트랜잭션) | Cassandra, RocksDB, LevelDB (빅데이터, 로깅) |

### 과목 융합 관점

- **운영체제 (OS)**: 델타 인코딩으로 극도로 압축된 데이터나 LSM-Tree의 순차 쓰기 방식은, OS의 페이지 캐시(Page Cache) 및 디스크 헤드 탐색(Seek Time) 원리를 100% 이해하고 이를 철저히 악용(?)하여 소프트웨어적으로 한계를 극복한 사례다.
- **클라우드 컴퓨팅**: Storage Tiering은 클라우드 환경에서 스토리지와 컴퓨팅을 완전히 분리(Separation of Storage and Compute)한 클라우드 네이티브 아키텍처의 산물이다. S3와 같은 객체 스토리지가 없다면 티어링 아키텍처 자체가 성립될 수 없다.

- **📢 섹션 요약 비유**: B+Tree가 잘 정리된 고급 백과사전(조회는 빠르나 새로 쓰기 어려움)이라면, LSM-Tree는 회의 시간에 속기사가 마구 휘갈겨 쓴 메모장(쓰기는 세계 최고이나, 나중에 다시 읽으려면 시간(콤팩션)이 필요함)입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 모바일 게임의 채팅 및 로그 데이터 보존

- **문제 상황**: 일간 수십억 건의 게임 로그(접속, 전투, 아이템 획득)를 MongoDB에 저장했으나, 3개월 치 데이터만으로 SSD 스토리지 비용이 수천만 원에 달함.
- **의사결정 (Tiering + Compaction 적용)**:
  1. 저장 엔진을 LSM-Tree 기반의 RocksDB나 Cassandra로 교체하여 SSD 쓰기 수명(TBW) 마모율을 줄임.
  2. Data Tiering을 설정하여 생성된 지 30일이 지난 게임 로그는 자동으로 AWS S3 (Cold Storage)로 마이그레이션(Flush)되도록 설정.
  3. 고객센터에서 과거 로그 조회 시, 쿼리 엔진이 S3에서 직접 데이터를 읽어오도록 구성하여 성능 저하 없이 스토리지 비용을 80% 이상 절감.

### 도입 체크리스트
- **Tombstone 회피 전략**: LSM-Tree 환경에서 `DELETE` 연산이 과도하게 많으면 콤팩션 전까지 쓰레기 데이터(Tombstone)가 계속 쌓여 읽기 속도가 파멸적으로 느려진다. 데이터 생명 주기(TTL)를 활용하여 자연 만료시키거나 삭제를 지양하는 설계가 가능한가?
- **콤팩션 부하(Spike) 통제**: 콤팩션 작업은 디스크 I/O와 CPU를 극도로 소모한다. 트래픽이 몰리는 피크 타임에 대규모 콤팩션이 돌지 않도록 스케줄링(Leveled vs Size-Tiered 방식 선택) 튜닝을 철저히 했는가?
- **티어링 조회 딜레이 감수**: S3로 내려간 Cold Data를 쿼리할 때는 SSD 대비 수십~수백 배의 지연 시간(Latency)이 발생한다. 현업 부서가 이 속도 저하를 인지하고 동의했는가?

### 안티패턴: 시계열 센서 데이터에 일반 RDBMS B-Tree 사용
- IoT 장비에서 초당 수만 건씩 들어오는 센서 데이터를 MySQL 같은 RDBMS에 꽂아 넣는 행위. 쓰기 작업(Insert)이 B-Tree 락과 랜덤 디스크 기록을 유발하며 전체 시스템이 마비된다. 또한 일반 Varchar/Float로 저장되어 엄청난 저장 공간을 낭비한다. 이런 워크로드에는 반드시 델타 인코딩과 LSM-Tree가 적용된 TSDB(InfluxDB 등)를 사용해야 한다.

- **📢 섹션 요약 비유**: 거대한 쓰레기 압축기(LSM-Tree)가 돌아가는 도중에 손을 집어넣어 물건을 찾으려 하면(피크타임 콤팩션 시 무거운 쿼리), 압축기도 멈추고 찾는 사람도 다칩니다. 콤팩션은 철저히 쉬는 시간에 돌아가게 해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 (전통적 DB 구조) | 최적화 후 (LSM/Delta/Tiering) | 개선 효과 |
|:---|:---|:---|:---|
| **저장 공간** | 원본 데이터 1:1 디스크 할당 | 델타 인코딩 기반의 극강 압축 | **물리적 스토리지 공간 90% 이상 절약** |
| **스토리지 비용** | 전체 데이터를 고가의 SSD에 유지 | 30일 초과 데이터 S3 티어링 적용 | **인프라 유지 비용 80% 절감** |
| **쓰기 처리량** | B-Tree 분할 시 I/O 병목 발생 | 메모리 Append 방식 순차 쓰기 | **초당 쓰기 성능(Write TPS) 100배 향상** |

### 미래 전망
- **NVM 및 지속성 메모리(PMEM) 통합**: 전원이 꺼져도 데이터가 날아가지 않는 비휘발성 메모리(NVDIMM)가 상용화되면서, LSM-Tree의 MemTable이 아예 디스크로 플러시할 필요조차 없어지는 새로운 커널 레벨의 스토리지 엔진이 연구되고 있다.
- **AI 기반 지능형 콤팩션**: DB가 스스로 쿼리 패턴과 I/O 대역폭을 머신러닝으로 학습하여, 관리자 개입 없이 가장 완벽한 타이밍에 콤팩션과 티어링을 수행하는 자율 주행(Autonomous) 최적화 시대로 진입하고 있다.

- **📢 섹션 요약 비유**: 현대 데이터베이스의 최적화 기술은 덩치만 키우던 공룡(RDBMS)에서 진화하여, 필요한 순간 깃털처럼 가볍게 날아오르고 뼈와 살을 고밀도로 압축하는 완벽한 생체 공학적 설계(클라우드 네이티브 엔진)의 정점입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **NoSQL 아키텍처 - 와이드 컬럼 스토어**: LSM-Tree 구조를 완벽하게 상용화하여 엄청난 쓰기 성능을 보여주는 대표적인 DB (Cassandra).
- **특수 목적 데이터베이스 (TSDB)**: 델타 인코딩(Delta Encoding) 압축 기법을 극한으로 활용하여 시계열 데이터를 관리하는 시스템.
- **데이터 레이크와 데이터 레이크하우스**: 스토리지 티어링(Tiering) 아키텍처가 종착역(Cold Tier)으로 의존하는 클라우드 객체 스토리지 생태계.

## 👶 어린이를 위한 3줄 비유 설명
1. **LSM-Tree 콤팩션**은 내 방에 어질러진 장난감을 일단 상자에 마구 던져 넣고(빠른 쓰기), 주말에 엄마 몰래 고장난 장난감은 버리고 예쁘게 다시 정리하는(병합 및 정리) 똑똑한 청소법이에요.
2. **델타 인코딩**은 100원, 101원, 102원이라고 길게 쓰는 대신, '처음 100원, 그 다음부턴 계속 +1'이라고 짧게 암호로 적어서 메모장 종이를 아끼는 기술이랍니다.
3. **스토리지 티어링**은 매일 가지고 노는 로봇은 내 책상 위(가장 빠르고 비싼 곳)에 두고, 작년에 가지고 놀던 레고는 지하실 창고(느리지만 넓고 싼 곳)에 알아서 옮겨주는 마법의 정리 시스템이에요!
