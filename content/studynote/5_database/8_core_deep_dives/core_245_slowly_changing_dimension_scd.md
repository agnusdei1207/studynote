# 차원 모델링 심화 - 서서히 변하는 차원(SCD)과 스냅샷 팩트

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 웨어하우스(DW)의 다차원 모델링 환경에서, 고객의 거주지나 상품의 부서처럼 "시간이 지나면서 값이 변경되는 차원 속성"의 이력을 어떻게 관리할 것인가에 대한 아키텍처 패턴이 바로 **서서히 변하는 차원(SCD, Slowly Changing Dimension)**이다.
> 2. **가치**: 가장 널리 쓰이는 **SCD Type 2 (이력 추가)** 패턴은 덮어쓰기(Type 1)의 치명적 결함인 과거 데이터의 왜곡을 방지하며, 작년에 발생한 매출 데이터가 현재의 변경된 주소가 아닌 '작년 주문 당시의 주소'를 완벽히 가리키도록 시계열적 무결성을 보존한다.
> 3. **융합**: 단순 트랜잭션 팩트 테이블만으로는 추출하기 어려운 "매월 말일 기준 전체 매장 재고 합계"와 같은 상태 지표를 극적으로 빠르게 연산하기 위해, 시스템 전체의 스냅샷을 주기적으로 구워버리는 **주기적 스냅샷 팩트(Periodic Snapshot Fact)** 기술과 융합되어 BI 대시보드의 성능을 완성한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **SCD (서서히 변하는 차원)**: 차원(Dimension) 테이블의 속성값이 변경될 때, 기존 OLTP처럼 단순 덮어쓰기(UPDATE)를 할 것인지, 아니면 과거 버전과 새 버전을 모두 유지할 것인지 결정하는 모델링 기법.
  - **스냅샷 팩트 (Snapshot Fact)**: 특정 이벤트 발생 시점(결제, 환불)에만 기록을 남기는 트랜잭션 팩트와 달리, 특정 시점(예: 매일 자정, 매월 말일)의 모든 데이터 상태를 사진 찍듯 통째로 기록하는 팩트 테이블.

- **💡 비유**: SCD는 주민등록등본의 '주소 이전 기록'과 같습니다. 이사했다고 옛날 주소를 지워버리면(Type 1) 과거에 어디 살았는지 증명할 수 없으므로, 줄을 그어 옛날 주소를 남겨두고 새 주소를 밑에 한 줄 더 적어 넣는(Type 2) 방식입니다.

- **등장 배경**:
  1. **OLAP 분석의 왜곡 문제**: 영업 사원 '홍길동'이 2024년에 서울에서 매출 100억을 냈고, 2025년에 부산으로 발령 났다. RDBMS처럼 주소를 '부산'으로 덮어써버리면, DW 통계에서 2024년 서울 매출 100억이 몽땅 2025년 부산 매출로 집계되어 버리는 심각한 통계 왜곡이 발생했다.
  2. **시점 기준(Point-in-time) 분석의 필요성**: 마케팅 팀에서 "3년 전 크리스마스 당시 VIP 등급이었던 고객들의 구매액"을 분석하려는데, 고객의 등급이 계속 바뀌다 보니 과거의 특정 시점(Time-variant) 상태를 고정해 줄 데이터 구조가 절실해졌다.
  3. **재고/잔고 집계의 쿼리 지옥 방어**: "현재 은행 잔고"를 구하기 위해 고객 가입일로부터 10년 치 입출금 내역 10만 건을 매일 SUM() 연산하다가 DB가 터지는 현상을 막기 위해, 매일 자정의 상태를 하드코딩해 두는 스냅샷 팩트 기법이 등장했다.

- **📢 섹션 요약 비유**: 데이터 웨어하우스(DW)는 현재의 모습만 비추는 '거울'이 아니라, 태어나서부터 지금까지의 모든 모습을 담은 '타임캡슐 비디오테이프'가 되어야 합니다. SCD와 스냅샷은 이 테이프의 시간을 자유자재로 되감기 위한 핵심 톱니바퀴입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 서서히 변하는 차원 (SCD)의 3가지 핵심 패턴

데이터 웨어하우스 거장 랄프 킴볼(Ralph Kimball)이 정의한 차원 이력 관리 기법이다.

| 패턴 종류 | 관리 방식 (주소 변경 시나리오) | 특징 및 장단점 |
|:---:|:---|:---|
| **SCD Type 1<br>(덮어쓰기)** | 기존 행의 주소 컬럼을 `서울`에서 `부산`으로 **그냥 덮어씀(UPDATE)**. | - 장점: 구현이 가장 쉽고 스토리지 공간 차지 안 함.<br>- 단점: **과거 이력이 영원히 소멸됨**. 서울에서 낸 과거 매출도 전부 부산 매출로 집계되는 치명적 통계 왜곡 발생. |
| **SCD Type 2<br>(새로운 행 추가)** | 기존 행은 놔두고, `부산`이라는 새로운 **행(Row)을 통째로 추가(INSERT)**. | - 장점: 특정 시점의 과거 사실을 완벽히 재현 가능(가장 권장됨).<br>- 단점: 데이터 용량이 커짐. 대체 키(Surrogate Key) 설계 필수. |
| **SCD Type 3<br>(컬럼 추가)** | 테이블에 `이전_주소`, `현재_주소` **컬럼을 만들어서 관리(ALTER)**. | - 장점: 현재와 직전 과거를 하나의 행에서 쉽게 비교 가능.<br>- 단점: 3번 이상 이사 가면 그 이전 기록은 사라짐(이력의 한계). |

+++

### 2. SCD Type 2의 물리적 구현 아키텍처 (대리 키와 유효 기간)

가장 이상적인 Type 2를 물리적 DB 테이블에 구현하기 위해 3가지 컬럼이 추가로 필요하다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 SCD Type 2 차원 테이블의 내부 구조                   │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 고객 차원 테이블 (Dim_Customer) ]                                 │
  │                                                                  │
  │  Surrogate_Key | Cust_ID(Natural) |  주소  | 시작일자  | 종료일자  | 최신여부│
  │  (DW용 대리키 PK)  (운영DB의 진짜 PK)                                │
  │  ──────────────┼──────────────────┼──────┼─────────┼─────────┼────│
  │       101      |      C-001       | 서울  | 24.1.1  | 25.5.5  |   N  │
  │       102      |      C-001       | 부산  | 25.5.6  | 9999.12 |   Y  │
  │                                                                  │
  │  [ 팩트 테이블 (Fact_Sales) 결합 시 ]                                │
  │   - 2024년 3월 매출이 발생하면 팩트 테이블은 대리키 '101'을 외래키(FK)로 잡음. │
  │   - 2025년 8월 매출이 발생하면 팩트 테이블은 대리키 '102'를 외래키(FK)로 잡음. │
  │   - 결과: 조인(Join) 한 번만으로 과거와 현재 매출이 정확한 지역에 매핑됨!     │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 고객 `C-001`이 이사를 할 때마다 테이블에는 새로운 행(Row)이 추가된다. 따라서 RDBMS의 `Cust_ID`를 PK로 쓸 수 없어, 데이터 웨어하우스(DW) 전용으로 아무 의미 없는 일련번호인 **대리 키(Surrogate Key, 101, 102)**를 PK로 부여한다. 그리고 이 정보가 언제부터 언제까지 유효했는지 타임스탬프(`시작일자`, `종료일자`)를 박아둔다. 나중에 팩트 테이블(매출)에 데이터가 쌓일 때, 결제 일자에 맞는 유효한 대리 키를 찾아 매핑하므로 "과거 매출의 주소"가 영원히 불변으로 보존되는 완벽한 시계열 구조가 완성된다.

+++

### 3. 주기적 스냅샷 팩트 (Periodic Snapshot Fact)

트랜잭션 팩트는 "결제가 일어났을 때만" 쌓이므로, "결제가 일어나지 않은 매장의 재고 수량"은 쿼리로 뽑아내기 극도로 어렵다. 이를 해결하는 팩트 모델링 기법이다.

- **원리**: 특정 주기(매일 자정, 매월 말일)마다 비즈니스의 **전체 상태를 강제로 사진 찍듯(Snapshot)** 하나의 팩트 행(Row)으로 구워서 적재한다.
- **활용**: 은행 계좌의 "일별 잔고", 물류 창고의 "일간 마감 재고", 구독형 서비스의 "일일 활성 구독자 수(DAU)" 등은 개별 트랜잭션을 1만 건 더해서 구하는 것보다, 자정에 스냅샷으로 딱 1건 적재해 둔 데이터를 읽는 것이 분석 성능을 수천 배 빠르게 만든다.

- **📢 섹션 요약 비유**: 트랜잭션 팩트가 "내가 하루 동안 쓴 카드 결제 영수증 10장"이라면, 스냅샷 팩트는 "매일 밤 12시에 가계부 맨 아래에 딱 한 번 적어두는 현재 남은 통장 총잔액"입니다. 결산할 때 영수증 1,000장을 다 뒤질 필요 없이 잔액만 보면 끝납니다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### 트랜잭션 팩트 vs 스냅샷 팩트 vs 누적 스냅샷 팩트

| 비교 항목 | 트랜잭션 팩트 (Transaction Fact) | 주기적 스냅샷 팩트 (Periodic Snapshot) | 누적 스냅샷 팩트 (Accumulating) |
|:---|:---|:---|:---|
| **발생 시점** | 이벤트(결제, 환불 등)가 발생할 때마다 | 정해진 주기(매일, 매월)마다 일괄 | 하나의 비즈니스 프로세스(주문~배송완료) 내에서 상태가 바뀔 때마다 |
| **저장 값의 성격**| 발생한 사건의 '측정값' (예: 구매 금액) | 특정 시점의 '누적 상태값' (예: 총 재고) | 전체 생명주기의 마일스톤 날짜 (예: 출고일, 도착일) |
| **데이터 변동(Update)**| 절대 없음 (Append Only) | 절대 없음 (Append Only) | **기존 행을 찾아서 날짜를 지속적으로 덮어씀 (Update 발생)** |
| **비즈니스 용도** | "오늘 총 얼마어치 팔렸나?" 집계 | "오늘 창고에 물건이 몇 개 남았나?" | "고객이 주문 후 배송까지 며칠 걸렸나?" (리드타임 분석) |

### 과목 융합 관점

- **이벤트 소싱 (Event Sourcing)**: 차원 이력을 관리하는 SCD Type 2의 철학은, 최신 데이터로 덮어쓰지 않고 변경 이벤트를 차곡차곡 쌓아나간다는 점에서 NoSQL/MSA 아키텍처의 이벤트 소싱 패턴과 100% 동일한 수학적 기저를 공유한다.
- **데이터베이스 인덱스 (Index)**: 대리 키(Surrogate Key)를 사용하게 되면 필연적으로 기존 운영 DB의 자연 키(Natural Key, 고객 ID)로 조인하는 업무에서 풀 스캔이 발생할 위험이 크다. 따라서 대리키(PK) 외에도 자연 키 컬럼에 대한 강력한 세컨더리(Secondary) 인덱스 및 파티셔닝 전략이 물리적 모델링 계층에서 반드시 융합 지원되어야 한다.

- **📢 섹션 요약 비유**: SCD Type 2와 스냅샷 팩트는 데이터 웨어하우스(DW)라는 거대한 시간 여행 기계의 핵심 부품입니다. 이 두 개가 조립되어야만, 경영자는 3년 전으로 시간 여행을 떠나 "그 당시 우리 회사의 정확한 모습"을 오차 없이 감상할 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: B2B 영업 사원 실적 랭킹 대시보드 오류 해결

- **문제 상황**: 김대리가 영업1팀에서 1~6월까지 50억, 7~12월에 영업2팀으로 전보되어 50억의 실적을 냈다. 연말 결산 시 경영진 대시보드(Type 1 덮어쓰기 적용)에 김대리의 소속이 모두 '영업2팀'으로 덮어써지는 바람에, 1팀의 상반기 실적은 50억이 증발하고 2팀의 실적이 뻥튀기되는 대형 정합성 사고 발생.
- **의사결정 (SCD Type 2 적용)**:
  1. `Dim_Employee (사원 차원)` 테이블을 SCD Type 2로 개편.
  2. 7월 발령 시, 김대리의 기존 영업1팀 레코드의 종료일자를 `24-06-30`으로 마감하고, 새로운 영업2팀 레코드(새로운 Surrogate Key 할당)를 `24-07-01` 시작일자로 INSERT.
  3. 과거에 일어난 1~6월 매출 팩트 데이터는 옛날 대리 키를 물고 있으므로, 대시보드에서 `GROUP BY 부서` 연산 시 정확하게 1팀(50억), 2팀(50억)으로 분산 집계되어 통계 오류 100% 해소.

### 도입 체크리스트
- **대리 키 (Surrogate Key) 생성 오버헤드**: SCD Type 2는 원본 DB의 PK(고객 ID)를 더 이상 유일키로 쓸 수 없게 만든다. ETL 과정에서 시퀀스(Sequence) 생성기나 해시 함수를 통해 1억 건의 데이터에 매일 새로운 대리 키를 발급하고 매핑(Lookup)하는 무거운 연산을 감당할 파이프라인(Apache Spark 등) 인프라가 갖춰져 있는가?
- **변화의 빈도(Volatility) 분석**: 이름에 '서서히(Slowly)'가 붙은 이유를 명심해야 한다. 주소나 부서는 가끔 변하지만, "고객의 몸무게"나 "주식 가격"처럼 1초마다 변하는 속성에 SCD Type 2를 걸면 차원 테이블의 데이터가 수백억 건으로 폭증하여(Rapidly Changing Dimension) 시스템이 파멸한다. 초단위 변경 속성은 아예 팩트(Fact) 테이블로 빼거나, 별도의 미니 디멘전(Mini-dimension)으로 분리해야 한다.

### 안티패턴: 모든 차원 테이블에 무조건 SCD Type 2 일괄 적용
- "이력 관리가 짱이다"라며 이름 맞춤법 수정, 고객의 오타 수정 같은 단순한(의미 없는) 변경 작업까지 모조리 Type 2로 행을 복사해 넣는 행위. 데이터 웨어하우스의 용량이 기하급수적으로 부풀어 오르고 쿼리 성능이 최악으로 치닫는다. 단순 오타 수정은 Type 1(덮어쓰기)로, 비즈니스 지표에 영향을 미치는 핵심 부서/주소 변경만 Type 2(이력 보존)로 적용하는 하이브리드(Hybrid SCD) 전략이 필수다.

- **📢 섹션 요약 비유**: 이력 관리(SCD Type 2)는 조선왕조실록을 쓰는 사관의 태도입니다. 왕이 반역을 진압한 사실(부서 이동)은 반드시 새 줄에 적어 남겨야 하지만, 왕이 기침을 하거나 오타를 낸 것(단순 맞춤법 변경)까지 매일 새 페이지를 써가며 기록하면 책장(디스크)이 터져버립니다.

### 정량/정성 기대효과

| 구분 | 레거시 DW (Type 1 덮어쓰기) | 차원 모델링 (SCD Type 2 + 스냅샷) | 개선 효과 |

### 미래 전망
- **데이터 레이크하우스(Lakehouse)로의 통합**: 전통적 RDBMS 환경에서 대리 키(Surrogate Key)를 발급하던 복잡한 ETL 과정이, 최근 Apache Iceberg나 Delta Lake 같은 개방형 테이블 포맷이 내장한 **타임 트래블(Time Travel)** 기술과 결합하여, 쿼리 단에서 `SELECT * FROM table TIMESTAMP AS OF '2024-01-01'` 만으로 SCD Type 2의 효과를 네이티브하게 제공하는 방향으로 진화하고 있다.
- **이벤트 스트리밍 융합**: 배치를 통한 하루 단위의 차원 이력 갱신이 아니라, Kafka를 타고 들어오는 CDC 이벤트를 실시간(Real-time)으로 감지하여 DW 내부의 차원 테이블을 1초 만에 최신화하는 스트리밍 차원 업데이트 기술이 표준화되고 있다.

- **📢 섹션 요약 비유**: 차원 모델링의 심화 기법(SCD)은 단순히 데이터를 저장하는 기술을 넘어, 기업의 과거와 현재를 단절 없이 하나로 묶어 미래를 예측하게 해주는 가장 정교하고 위대한 역사 기록 장치입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **OLAP 연산과 다차원 모델링 (스타 스키마)**: 팩트(Fact)와 차원(Dimension)이라는 뼈대를 처음 세우는 기초 모델링 기법. SCD는 이 뼈대 안의 '시간성(Time)'을 부여하는 심화 모듈.
- **데이터 웨어하우스와 ODS**: 실시간 트랜잭션(OLTP) 환경에서는 발생하지 않는, 오직 데이터를 길게 모아두는 데이터 웨어하우스 환경에서만 대두되는 아키텍처적 숙제.
- **구체화된 뷰 (Materialized View)**: 주기적 스냅샷 팩트(Snapshot Fact)를 물리적으로 가장 효율적으로 구현(Refresh)하기 위해 사용되는 데이터베이스 내부 캐싱 객체.

## 👶 어린이를 위한 3줄 비유 설명
1. 철수가 서울 살다가 부산으로 이사를 갔어요. 공책(DB)에 적힌 '서울'을 지우개로 빡빡 지우고 '부산'으로 덮어쓰면(SCD Type 1), 나중에 철수가 옛날 서울에서 산 물건을 어디로 보냈었는지 아무도 알 수 없게 되어버려요.
2. 그래서 똑똑한 사서(데이터 엔지니어)는 줄을 긋고 그 밑에 '24년부터 부산'이라고 새 줄을 하나 더 적어준답니다(SCD Type 2). 이러면 과거의 기록도 100% 안전하게 보호되죠!
3. **스냅샷 팩트**는 일기장에 "사탕 1개 삼, 초코 1개 삼" 이렇게 길게 쓰는 대신, 매일 밤 12시 땡! 치면 지갑을 사진으로 찰칵 찍어서 "남은 돈 5천 원"이라고 딱 한 줄만 깔끔하게 저장해 두는 마법의 앨범이에요.
