# 고급 SQL 튜닝 기법 - 뷰 머징, 조건 푸시 다운, 조인 순서 최적화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 고급 SQL 튜닝은 비용 기반 옵티마이저(CBO)가 최적의 실행 계획(Execution Plan)을 생성하도록 유도하기 위해, 데이터베이스 엔진 내부에서 일어나는 **쿼리 변환(Query Transformation)**의 수학적/논리적 원리를 이해하고 통제하는 과정이다.
> 2. **가치**: 서브쿼리 언네스팅(Subquery Unnesting)과 조인 조건 푸시 다운(Join Predicate Pushdown)을 통해, 불필요한 테이블 풀 스캔(Full Scan)과 메모리 고갈을 막고 수천만 건의 조인 쿼리를 밀리초(ms) 단위로 응답하게 만든다.
> 3. **융합**: RDBMS의 조인 최적화 기술은 데이터 웨어하우스(DW)의 다차원 모델링 환경과 만나 **스타 변환(Star Transformation)** 기법으로 융합되며, 대형 팩트 테이블과 다수의 차원 테이블 간의 분석 쿼리 성능을 비약적으로 끌어올린다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 사용자가 작성한 SQL은 "무엇(What)"을 가져올지만 명시하는 선언적 언어다. 이를 "어떻게(How)" 가져올지 결정하는 옵티마이저가, 사용자의 쿼리를 내부적으로 더 효율적인 형태로 재작성(Rewrite)하는 과정을 쿼리 변환이라고 하며, 이 원리를 이용해 힌트(Hint)나 인덱스를 설계하는 것이 SQL 튜닝이다.

- **💡 비유**: SQL 튜닝은 내비게이션(옵티마이저)에게 단순히 목적지(결과)만 입력하고 알아서 가라고 방치하는 대신, "A 톨게이트(인덱스)를 타고 B 도로(조인 순서)로 가라"고 경유지를 정확히 지정해 주어 꽉 막힌 퇴근길(병목 현상)을 뚫어내는 베테랑 택시 기사의 운전법과 같습니다.

- **등장 배경**:
  1. **복잡한 인라인 뷰의 남용**: 개발자들이 객체 지향 프로그래밍처럼 뷰(View) 안에 뷰를 중첩하여 재사용하는 방식(모듈화)을 선호하면서, 옵티마이저가 이를 통째로 해석하지 못해 각각의 뷰를 독립적으로 실행한 후 조인하는 끔찍한 병목이 발생했다.
  2. **데이터 볼륨의 폭발**: 수억 건의 테이블 간 조인 시, 조인 순서(Join Order)가 1단계만 잘못되어도 중간 결과 집합(Intermediate Result Set)이 테라바이트 급으로 불어나 디스크 스왑(PGA 고갈)을 유발했다.

- **📢 섹션 요약 비유**: 수백 명의 직원을 면접(조인) 볼 때, 문서를 밖에서 먼저 거르고(조건 푸시 다운) 면접관들을 한 방에 모아서 동시에 평가(뷰 머징)하는 것이 한 명씩 따로 면접을 보는 것보다 수십 배 빠른 것과 같은 원리입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 뷰 머징 (View Merging)과 서브쿼리 언네스팅 (Subquery Unnesting)

옵티마이저가 쿼리의 논리적 계층 구조를 해체하여 평면적(Flat)인 단일 조인 문으로 변환하는 과정이다.

| 변환 기법 | 동작 원리 | 효과 및 목적 |
|:---|:---|:---|
| **View Merging** | `FROM` 절에 있는 인라인 뷰나 독립된 뷰를 해체하여 메인 쿼리의 조인으로 병합시킴 | 옵티마이저가 뷰 내부의 테이블과 메인 쿼리 테이블 간의 조인 순서를 자유롭게 재배치할 수 있게 함 |
| **Subquery Unnesting** | `WHERE` 절에 둥지(Nest)를 튼 서브쿼리를 풀어내어 메인 쿼리와의 조인(Semi-Join 등)으로 변환시킴 | 메인 쿼리의 매 행마다 서브쿼리를 반복 실행(Filter)하는 오버헤드를 없애고 해시 조인 등을 가능케 함 |

*주의점: 뷰 안에 `GROUP BY`나 `DISTINCT`, 분석 함수가 포함되어 있으면 원칙적으로 뷰 머징이 불가능하다(Complex View).*

+++

### 2. 조건 푸시 다운 (Condition / Predicate Pushdown)

뷰 머징이 실패하여 뷰를 독립적으로 실행해야 할 때, 메인 쿼리에 있는 `WHERE` 조건을 뷰 내부로 밀어 넣는(Push-down) 최적화 기법이다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 조건 푸시 다운 (Predicate Pushdown) 원리             │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [변환 전 쿼리 (비효율적 실행)]                                        │
  │  SELECT *                                                        │
  │    FROM (SELECT dept_no, AVG(salary) FROM EMP GROUP BY dept_no) V│
  │   WHERE V.dept_no = 10;   ◀── 조건이 바깥에 있음                     │
  │                                                                  │
  │  [옵티마이저의 내부 동작 흐름]                                          │
  │  1. EMP 테이블 전체(10만 건)를 읽어 부서별 평균 급여를 계산. (무거운 집계)   │
  │  2. 계산이 끝난 임시 집합에서 dept_no=10 인 것 1건만 필터링. (비효율 발생)   │
  │                                                                  │
  │  [조건 푸시 다운 적용 후]                                              │
  │  SELECT *                                                        │
  │    FROM (SELECT dept_no, AVG(salary) FROM EMP                    │
  │           WHERE dept_no = 10    ◀── 메인 쿼리의 조건이 안으로 침투!   │
  │           GROUP BY dept_no) V;                                   │
  │                                                                  │
  │  [결과] EMP 테이블 중 10번 부서 직원(100건)만 읽어서 평균을 계산. 속도 1,000배 향상!│
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 개발자는 로직의 재사용성을 위해 통합된 뷰(부서별 평균 급여)를 만들어 두고 바깥에서 조건을 걸어 사용한다. 하지만 DB 엔진은 바깥의 조건(`dept_no=10`)을 뷰 안으로 억지로 밀어 넣음으로써, 뷰 내부에서 전체 테이블을 풀 스캔(Full Scan)하고 그룹핑(Group By)하는 막대한 연산 비용을 사전에 차단한다. 데이터베이스 튜닝의 제1원칙인 **"데이터 필터링을 최대한 빨리, 안쪽에서 수행하라"**를 옵티마이저가 자동으로 수행하는 것이다.

+++

### 3. 조인 순서 (Join Order) 최적화

수많은 테이블을 조인할 때, 어떤 테이블을 먼저 읽을지(Driving Table) 결정하는 것은 성능의 90%를 좌우한다.

- **원칙**: 조인 결과 집합(Intermediate Result)을 최소화할 수 있는, 즉 **가장 강력한 필터링 조건이 걸려 건수가 가장 적게 나오는 테이블을 드라이빙(선행) 테이블로 선택**해야 한다.
- **스타 변환 (Star Transformation)**:
  데이터 웨어하우스(DW)의 스타 스키마 환경에서 사용하는 특수 조인 튜닝이다. 무지막지하게 큰 팩트 테이블(매출)에 여러 개의 작은 차원 테이블(시간, 매장, 상품)을 조인할 때, 팩트 테이블을 무작정 스캔하는 대신 **각 차원 테이블들의 비트맵 인덱스(Bitmap Index)를 먼저 비트 연산(AND/OR)**하여 최종적으로 팩트 테이블에서 꺼내올 행의 위치(ROWID) 집합을 찾아낸 뒤, 단 한 번만 팩트 테이블에 접근하는 혁신적인 기법이다.

- **📢 섹션 요약 비유**: 맛있는 찌개를 끓일 때 무, 양파, 파(차원 테이블)를 먼저 작게 썰어 준비한 뒤 가장 큰 고기 덩어리(팩트 테이블)와 한 번에 끓여내는 것이 스타 변환 조인 순서입니다. 고기부터 무작정 삶으면서 채소를 다듬으면 시간이 배로 걸립니다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 조인 메소드 (Join Method) 심층 비교

| 비교 항목 | Nested Loop (NL) 조인 | Sort Merge 조인 | Hash 조인 |
|:---|:---|:---|:---|
| **작동 원리** | 2중 For 문 (선행 테이블 건수만큼 후행 테이블 인덱스 반복 탐색) | 양쪽 테이블을 정렬(Sort)한 후 한 번에 스캔하며 병합(Merge) | 선행 테이블로 메모리에 해시 맵(Hash Map) 생성 후 후행 테이블로 프로빙(Probe) |
| **최적 환경** | 소량 데이터 조인, 인덱스가 완벽할 때 (OLTP) | 대량 데이터이되 해시 조인이 불가능한 비동등 조인(>, <, BETWEEN) | 대량 데이터의 동등(=) 조인 (OLAP/DW) |
| **병목 지점** | 랜덤 I/O 다량 발생 | 메모리/Temp 영역에서 Sort 부하 | Hash 영역(PGA) 메모리 부족 시 디스크 스왑 |
| **조인 순서** | Driving Table의 크기가 성능을 절대적으로 결정 | 순서가 비교적 덜 중요함 | Hash Map을 올릴 수 있는 작은 테이블이 선행 |

### 과목 융합 관점

- **자료구조 (Data Structure)**: 해시 조인은 메모리 상에 `O(1)` 탐색 속도를 갖는 해시 테이블을 빌드(Build)하는 전형적인 자료구조의 응용이며, 소트 머지 조인은 병합 정렬(Merge Sort) 알고리즘의 원리를 디스크 I/O 레벨에 최적화하여 구현한 것이다.
- **분산 시스템 (Spark/Hadoop)**: 맵리듀스나 Spark 환경에서도 이 조인 원리는 그대로 융합된다. 큰 테이블과 작은 테이블을 조인할 때 셔플(Shuffle) 없이 작은 테이블을 모든 노드의 메모리에 복사(Broadcast)해 두고 해시 조인을 수행하는 **브로드캐스트 조인 (Broadcast Join)**으로 진화하였다.

- **📢 섹션 요약 비유**: NL 조인은 '전화번호부를 들고 한 명씩 일일이 전화를 거는 방식'이고, 소트 머지 조인은 '양쪽 반 학생들을 키순으로 세워놓고 짝을 맺어주는 방식', 해시 조인은 '이름표가 붙은 사물함 100개를 짜놓고 물건을 확 부어버리는 방식'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 복합 뷰(Complex View)로 인한 대형 서비스 장애 해결

- **문제 상황**: '월별 전체 부서 실적 뷰(View_A)'를 만들어 놓았는데, 영업팀 직원이 자기 부서의 실적만 보려고 `SELECT * FROM View_A WHERE dept = '영업'` 쿼리를 날렸으나 응답에 5분이 걸려 DB CPU가 100%를 침.
- **원인 분석**: View_A 내부에 `ROWNUM` 분석 함수와 복잡한 `GROUP BY`가 들어 있어, 옵티마이저가 바깥의 `dept='영업'` 조건을 안으로 밀어 넣지(Predicate Pushdown) 못함. 결국 DB는 '전체 1,000개 부서의 실적'을 모두 계산한 뒤 마지막에 1개 부서만 필터링하는 최악의 비효율을 저지름.
- **의사결정 (튜닝 방안)**:
  1. **뷰 해체 및 인라인 뷰 재작성**: 뷰 내부에 있던 분석 함수와 불필요한 그룹핑 로직을 제거하여 **뷰 머징(View Merging)**이 일어나게 유도.
  2. **Pushdown 강제**: 오라클의 경우 `/*+ PUSH_PRED */` 힌트를 사용하여 옵티마이저가 강제로 조인 조건을 뷰 안으로 밀어 넣도록 지시. (결과: 5분 -> 0.1초 만에 응답 반환)

### 도입 체크리스트
- **통계 정보 (Statistics) 최신화**: CBO(비용 기반 옵티마이저)는 테이블의 건수, 인덱스 깊이 등을 기반으로 변환을 결정한다. 통계 정보가 과거의 것이라면 옵티마이저가 엉뚱한 테이블을 드라이빙 테이블로 삼게 되므로, 통계 정보 갱신(Gather Stats) 스케줄이 정상 작동하는지 확인해야 한다.
- **바인드 변수 (Bind Variable) 사용 여부**: `WHERE age = 20`처럼 상수를 하드코딩(Literal)하면 파싱(Parsing) 비용이 매번 발생할 뿐만 아니라 커서(Cursor) 공유가 안 되어 힙 메모리가 고갈된다. 반드시 `WHERE age = :v_age` 형태의 바인드 변수를 사용하는가? (SQL 인젝션 방어와 성능 두 마리 토끼)
- **힌트(Hint)의 남용 경계**: 인덱스 구조가 바뀌었는데 쿼리에 박혀 있는 강제 힌트(`/*+ INDEX(...) */`) 때문에 더 좋은 실행 계획을 타지 못하는 기술 부채가 없는가?

### 안티패턴: 조인(Join)을 피하려고 애플리케이션에서 반복 쿼리 (N+1 문제)
- JPA나 Hibernate 같은 ORM 도구 사용 시, 데이터베이스 엔진의 강력한 해시 조인을 믿지 못하거나 무지하여, 부모 테이블을 1번 `SELECT`하고 그 결과로 나온 자식 ID 100개를 바탕으로 루프를 돌며 자식 테이블에 `SELECT`를 100번 날리는 전형적인 성능 파괴 패턴. 이는 네트워크 라운드트립(Round-trip) 지연을 100배 증가시키며 시스템을 붕괴시킨다.

- **📢 섹션 요약 비유**: 옵티마이저라는 천재 수학자에게 "더하기 100번"을 시키는 대신 "곱하기 1번"으로 수식을 바꿔주어 능력을 발휘하게 만드는 것이 바로 SQL 튜닝입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 튜닝 전 (옵티마이저의 오류/한계) | 튜닝 후 (뷰 머징/푸시 다운 적용) | 개선 효과 |
|:---|:---|:---|:---|
| **디스크 I/O (블록 읽기)** | 불필요한 Full Scan으로 100만 블록 읽음 | 조건 침투로 1,000 블록만 인덱스 스캔 | **논리적 읽기(Logical Read) 99% 감소** |
| **PGA / 메모리 사용** | 대규모 데이터 정렬(Sort) 시 디스크 스왑 발생 | 드라이빙 테이블 축소로 메모리 내 해시 조인 성공 | **CPU 사용량 및 메모리 고갈(OOM) 완벽 해소** |
| **시스템 전체 TPS** | 악성 쿼리 1개가 DB 전체의 자원 락(Lock) 점유 | 0.1초 내 반환으로 커넥션 즉시 반납 | **동시 접속자 수용량(Concurrency) 10배 이상 증가** |

### 미래 전망
- **AI/ML 기반 옵티마이저 (Autonomous Database)**: 과거에는 DBA가 실행 계획(Explain Plan)을 보고 수동으로 힌트를 주었으나, 현대의 클라우드 데이터베이스(Oracle Autonomous DB 등)는 AI가 쿼리 실행 이력을 실시간 학습하여 인덱스를 자동 생성하고 조인 순서를 스스로 튜닝하는 시대로 접어들었다.
- **분산 쿼리 엔진의 진화**: 단일 서버를 넘어 Trino, Presto와 같은 분산 쿼리 엔진이 데이터 패브릭(Data Fabric) 환경에서 '조인 푸시 다운(Join Push-down)'을 통해 서로 다른 물리적 서버 간의 네트워크 전송량을 획기적으로 줄이는 아키텍처로 튜닝 기술이 확장되고 있다.

- **📢 섹션 요약 비유**: SQL 튜닝은 무거운 짐(데이터)을 지고 가파른 산을 넘는 등산객에게, 지도를 꼼꼼히 분석하여 가장 평탄하고 짧은 '지름길(최적의 실행 계획)'을 찾아주는 필수적인 길잡이 역할입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **B-Tree 인덱스 구조**: 조건 푸시 다운(Pushdown)이 성공했을 때 실제로 데이터를 초고속으로 걸러내는 물리적 저장 매커니즘.
- **OLAP와 스타 스키마**: 스타 변환(Star Transformation) 튜닝 기법이 적용되는 베이스 데이터 모델링 아키텍처.
- **인메모리 데이터베이스**: 디스크 I/O 튜닝의 한계를 근본적으로 회피하기 위해 해시 맵을 메모리에 상주시키는 아키텍처의 연장선.
