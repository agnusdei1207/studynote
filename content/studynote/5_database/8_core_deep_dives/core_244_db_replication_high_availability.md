# 데이터베이스 복제와 고가용성 아키텍처 - 마스터-슬레이브, PACELC

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 복제(Replication)는 시스템의 단일 장애점(SPOF)을 제거하고 막대한 읽기(Read) 부하를 분산시키기 위해, 원본 데이터베이스의 트랜잭션 로그를 여러 대의 물리적 노드(Node)에 지속적으로 동기화하는 아키텍처 기술이다.
> 2. **가치**: 가장 범용적인 **마스터-슬레이브(Master-Slave)** 구조는 90%에 달하는 읽기 트래픽을 슬레이브로 분산하여 성능을 극대화하며, 마스터 서버 장애 시 슬레이브를 새로운 마스터로 자동 승격시키는 **페일오버(Failover)**를 통해 시스템의 고가용성(HA, High Availability)을 99.99% 보장한다.
> 3. **융합**: CAP 정리를 현실적으로 발전시킨 **PACELC 정리**와 융합되어, 네트워크 단절(P)이 없는 평상시 상황(E)에서도 '응답 지연 속도(Latency)'와 '데이터 일관성(Consistency)' 중 어느 것을 우선할지 비즈니스 목적에 따라 아키텍처 노선을 결정하는 절대적 나침반 역할을 한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **복제 (Replication)**: 한 대의 DB 서버(마스터)에서 일어난 데이터 변경 사항을 다른 여러 대의 DB 서버(슬레이브/레플리카)로 복사하여 데이터의 동일한 사본을 유지하는 프로세스.
  - **고가용성 (High Availability)**: 서버나 네트워크에 장애가 발생하더라도 서비스가 절대 멈추지 않고 지속적으로 사용자에게 응답할 수 있는 시스템의 능력.

- **💡 비유**: 복제와 고가용성은 유명 맛집의 주방 운영과 같습니다. 메인 셰프(마스터)가 요리 비법을 개발하면, 보조 셰프(슬레이브) 3명이 그 레시피를 실시간으로 똑같이 복사해서 외워둡니다. 만약 메인 셰프가 갑자기 쓰러져도(장애), 레시피를 외우고 있는 1번 보조 셰프가 즉시 메인 셰프 자리를 꿰차고(Failover) 요리를 계속 내놓아 손님들이 불만을 느끼지 못하게 하는 것입니다.

- **등장 배경**:
  1. **단일 장애점 (SPOF)의 공포**: 데이터베이스가 1대뿐일 때, 디스크가 깨지거나 전원이 나가는 순간 전체 웹 서비스가 즉시 셧다운(Shut-down)되는 치명적 약점을 가지고 있었다.
  2. **읽기/쓰기 부하의 불균형**: 웹 서비스는 특성상 게시글을 쓰는(Write) 횟수보다 조회하는(Read) 횟수가 압도적으로 많다(보통 1:10~1:100). 이를 한 대의 DB가 모두 감당하는 것은 비효율의 극치였다.
  3. **무중단 배포 및 백업의 필요성**: 마스터 DB가 서비스 중일 때 슬레이브 DB를 멈추고 안전하게 풀 백업(Full Backup)을 받거나, 엔진 버전을 업그레이드하기 위한 유연한 인프라가 필요해졌다.

- **📢 섹션 요약 비유**: 복제 기술은 비행기에 엔진을 하나만 다는 것이 아니라 양쪽 날개에 두 개를 달아, 한쪽 엔진이 불타더라도 남은 엔진 하나로 목적지까지 무사히 도착하게 만드는 항공 공학의 생존 법칙과 완벽히 동일합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 마스터-슬레이브 (Master-Slave / Primary-Replica) 아키텍처

가장 고전적이며 현대에도 90% 이상의 기업이 사용하는 표준 복제 모델이다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 마스터-슬레이브 복제 및 읽기 분산 아키텍처               │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │     [ Client / WAS ] ──(Write/INSERT)──▶ [ Master Node (쓰기 전용) ]│
  │            │                                  │ (Binlog 전송)      │
  │            │                                  ▼                   │
  │            │ (Read/SELECT 분산)      ┌──────────────────┐         │
  │            │                       │    Replication   │         │
  │            ├─────────────────▶   [ Slave 1 ]      [ Slave 2 ]     │
  │            └─────────────────▶  (읽기 전용 캐시)   (읽기 전용 캐시)   │
  │                                                                  │
  │  - 메커니즘: 마스터는 트랜잭션 완료 후 로그(Binlog/WAL)를 슬레이브로 전송. │
  │  - 슬레이브는 받은 로그를 자신의 DB에 재실행(Relay Log 실행)하여 동기화.   │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클라이언트는 데이터의 생성/수정/삭제(C/U/D) 명령을 무조건 **마스터 노드**에만 보낸다. 마스터는 이를 처리하고 그 내역(로그)을 여러 대의 **슬레이브 노드**로 뿌린다. 클라이언트의 단순 조회(SELECT) 쿼리는 로드밸런서(L4 스위치나 Proxy)를 통해 여러 대의 슬레이브 노드로 라우팅된다. 이로써 마스터는 락(Lock) 경합 없이 쓰기 작업에만 온전히 CPU를 집중할 수 있고, 슬레이브들은 수십만 건의 조회 트래픽을 거뜬히 방어한다.

+++

### 2. 동기식 (Synchronous) vs 비동기식 (Asynchronous) 복제

마스터가 슬레이브에게 데이터를 넘겨주고 "완료 응답"을 언제 받는가에 대한 결정이다.

| 복제 방식 | 동작 원리 (마스터의 행동) | 장점 | 단점 |
|:---|:---|:---|:---|
| **동기식 (Sync)** | 마스터가 쓴 뒤, 슬레이브도 완전히 썼다는 ACK를 받아야만 클라이언트에게 성공을 반환 | 슬레이브와 마스터의 데이터가 100% 일치 (Zero Data Loss) | 복제되는 시간 동안 마스터가 대기(Blocking)하므로 응답 지연 최악 |
| **비동기식 (Async)** | 마스터는 쓰자마자 클라이언트에게 성공 반환. 슬레이브 복제는 백그라운드로 대충 던져둠 | 응답 속도가 단일 DB 수준으로 가장 빠름 (표준) | 마스터가 죽는 찰나에 복제 안 된 1~2초 분량의 데이터 영구 유실(Loss) |
| **반동기식 (Semi-sync)** | 3대의 슬레이브 중 '단 1대'라도 받았다는 ACK를 주면 클라이언트에게 성공 반환 | 최소 1대의 복제본은 보장하므로 정합성과 성능의 완벽한 타협점 | 네트워크 지연 시 여전히 약간의 대기 발생 |

+++

### 3. 멀티 마스터 (Multi-Master) 아키텍처

마스터-슬레이브의 단점(쓰기 병목)을 극복하기 위해, **모든 노드가 쓰기(Write) 권한을 가지는 구조**다. (Cassandra, Amazon Aurora 등)
- **장점**: 전 세계 어디서든 가장 가까운 DB에 데이터를 쓸 수 있어 글로벌 서비스의 지연(Latency)이 획기적으로 짧아진다. 특정 마스터가 죽어도 다른 마스터가 즉시 쓰기를 받으므로 가용성이 우주 최강이다.
- **치명적 단점 (충돌)**: A노드와 B노드에서 동시에 같은 사용자 정보를 수정했을 때, 서로 동기화하는 과정에서 "누구의 데이터가 맞는가?"라는 **충돌(Conflict)**이 100% 발생한다. 이를 위해 벡터 시계(Vector Clock)나 CRDT 자료구조를 적용하거나 애플리케이션 레벨에서 복잡한 병합(Merge) 로직을 짜야 하는 지옥이 펼쳐진다.

## Ⅲ. 융합 비교 및 다각도 분석

### CAP 정리를 확장한 PACELC 정리

1998년 에릭 브루어가 주창한 CAP 정리(네트워크 단절-P 시, 일관성-C과 가용성-A 중 하나만 택해야 한다)는 완벽하지 않았다. "네트워크가 안 끊어진 평상시에는 뭘 선택해야 하는데?"라는 질문에 답하기 위해 2010년 대에 **PACELC 정리**가 탄생했다.

**PACELC 공식 해독**:
- **If P (네트워크 파티션 발생 시)**: **A**(가용성)를 택할래? **C**(일관성)를 택할래?
- **Else (평상시 정상 네트워크일 때)**: **L**(지연 시간/성능)을 택할래? **C**(일관성)를 택할래?

| 시스템 유형 | 철학 (PACELC 척도) | 대표적 데이터베이스 모델 |
|:---|:---|:---|
| **PA / EL** | 장애 시 무조건 응답(A). 평소엔 속도(L) 우선 | DynamoDB, Cassandra (결과적 일관성) |
| **PC / EC** | 장애 시 멈춤(C). 평소에도 동기화 대기(C) | 전통적 RDBMS 멀티 클러스터 (동기 복제) |
| **PA / EC** | 장애 시 일단 응답(A). 평소엔 완벽 동기화(C) | MongoDB (평소엔 Primary가 완벽 통제, 죽으면 대충 응답) |
| **PC / EL** | 장애 시 멈춤(C). 평소엔 속도(L) 우선 | PNUTS, Yahoo! (학술적 특수 모델) |

- **📢 섹션 요약 비유**: PACELC 정리는 식당 주인의 철학입니다. "직원이 아파서 안 나왔을 때(P), 맛이 없어도 대충 손님을 받을 것인가(A), 아니면 아예 문을 닫을 것인가(C)? 직원이 다 출근한 평상시(E)에는, 대충 빨리 요리를 낼 것인가(L), 아니면 레시피를 완벽히 지켜 늦게 낼 것인가(C)?"를 결정하는 설계 기준입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 이커머스의 Replica Lag (복제 지연) 장애 해결

- **문제 상황**: 사용자가 프로필 사진을 수정(Master Write)한 뒤 곧바로 마이페이지로 이동(Slave Read)했는데, 바뀐 사진이 안 나오고 예전 사진이 뜨는 오류 발생(고객 항의 폭주).
- **원인 분석**: 비동기(Async) 복제 방식의 태생적 한계인 **Replica Lag (복제 지연)** 현상. 마스터에 쓴 데이터가 슬레이브로 넘어가 적용되기까지 약 0.5초가 걸리는데, 웹 브라우저가 그보다 빠른 0.1초 만에 슬레이브를 조회했기 때문.
- **의사결정 (아키텍처 튜닝)**:
  1. **Write-After-Read (Read-Your-Own-Writes)** 정책 적용: 특정 고객이 데이터를 수정한 직후 5초 이내의 조회 요청은 로드밸런서가 슬레이브가 아닌 **마스터 DB로 강제 라우팅**하도록 세션/캐시 레이어에서 통제함.
  2. 일반적인 남의 프로필 조회 트래픽은 여전히 슬레이브를 태워 성능을 확보하고, '내가 쓴 글은 내가 바로 볼 수 있게' 하여 고객 불만을 100% 잠재움.

### 도입 체크리스트
- **스플릿 브레인 (Split Brain) 방어**: 네트워크 스위치 장애로 마스터(A)와 대기조(B)가 서로 "상대방이 죽었구나"라고 착각하여 둘 다 마스터로 승격해버리는 대참사. 고객 트래픽이 양쪽으로 나뉘어 쓰이게 되며 복구 불가능한 데이터 파괴가 일어난다. 이를 막기 위해 제3의 감시자(Tie-breaker/ZooKeeper)를 홀수로 배치하여 과반수(Quorum) 합의 시에만 승격하도록 설계했는가?
- **Auto-Failover (자동 복구)**: 마스터가 죽었을 때 DBA가 새벽에 일어나 수동으로 스크립트를 쳐서 슬레이브를 승격시키는가(MHA 등 미사용)? 이 경우 RTO(복구 시간)가 수십 분 소요된다. AWS RDS나 프록시SQL을 통해 VIP(Virtual IP) 또는 DNS가 10초 이내에 자동으로 슬레이브를 가리키도록 설정되었는가?
- **슬레이브의 쓰기 금지 (Read-Only)**: 슬레이브 DB 설정 파일에 반드시 `read_only = 1` 속성을 걸어두었는가? 개발자의 실수나 해킹으로 슬레이브에 직접 데이터가 삽입(INSERT)되면, 마스터로부터 내려오는 Binlog와 충돌하여 복제 파이프라인이 영원히 깨지는 크리티컬한 사고가 발생한다.

### 안티패턴: "복제(Replication)를 백업(Backup)으로 착각하기"
- 데이터가 3대의 슬레이브에 실시간으로 똑같이 복사되고 있으니, "안전하니까 야간 테이프/스냅샷 백업은 안 해도 된다"고 생각하는 가장 멍청하고 위험한 착각. 어느 날 신입 개발자가 실수로 `DELETE FROM User` (WHERE 없는 삭제)를 마스터에 날리면, 그 파괴적인 명령은 **0.1초 만에 모든 슬레이브로 복제되어 전사의 모든 DB가 동시에 삭제된다**. 복제는 고가용성(서비스 무중단)을 위한 것이며, 백업(과거 시점 복원)과는 목적이 180도 다른 기술이다.

- **📢 섹션 요약 비유**: 복제(Replication)는 거울입니다. 내 얼굴에 실수로 먹물이 튀면 거울 속 얼굴에도 0.1초 만에 먹물이 튀어 버립니다. 이 먹물을 지우려면 거울을 보는 게 아니라, 어제 찍어둔 진짜 옛날 사진(Backup)을 꺼내봐야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 단일 DB (Single Node) | 복제 아키텍처 (Master-Slave) | 개선 효과 |
|:---|:---|:---|:---|
| **읽기 성능 (Read TPS)**| 마스터에 모든 조회 부하 집중 (병목) | 수십 대의 슬레이브 분산 처리 (Scale-out) | **조회 트래픽 수용 능력 10~100배 수평적 확장** |
| **시스템 생존율 (HA)** | 마스터 하드웨어 고장 시 즉시 서비스 다운 | 슬레이브의 자동 마스터 승격 (Failover) | **연간 서비스 중단 시간 5분 이내 (99.99%) 달성** |
| **무중단 운영 지원** | DB 업그레이드/백업 시 서비스 정지 불가피 | 슬레이브 정지 후 백업 및 교차 패치 | **점검으로 인한 다운타임(Downtime) 완전 제로화** |

### 미래 전망
- **클라우드 네이티브 서버리스 클러스터 (Aurora)**: 과거의 마스터-슬레이브는 마스터가 죽고 슬레이브가 뜰 때 10~30초의 단절이 필수적이었다. 하지만 AWS Aurora 같은 현대 아키텍처는 컴퓨팅(CPU)과 스토리지(Disk)를 아예 분리하여, 디스크가 6개의 가용영역에 걸쳐 스스로 복제되므로 노드가 죽어도 단 1~2초 만에 다른 캐시 노드가 스토리지를 물고 올라오는 극강의 무중단 클러스터를 완성했다.
- **Active-Active 글로벌 분산 DB**: 마스터-슬레이브의 읽기 지연(Replica Lag)과 멀티 마스터의 데이터 충돌 지옥을 완벽히 해결하기 위해, 원자 시계(TrueTime)와 분산 락(Raft)을 내장한 Google Spanner, CockroachDB가 등장하여 '뉴욕에서 쓰고 런던에서 즉시 조회해도 일치하는' 신의 영역을 상용화하고 있다.

- **📢 섹션 요약 비유**: 데이터베이스 복제와 고가용성 설계는, 하나뿐인 심장(단일 DB)을 가진 인간의 한계를 극복하고 수십 개의 보조 심장을 몸속에 이식하여 결코 죽지 않는 불사신(무정지 시스템)을 창조하는 IT 인프라 공학의 정점입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 칠판(마스터 DB)에 선생님이 필기를 하면, 반장(슬레이브 DB) 3명이 똑같이 자기 노트에 베껴 적어요. 이걸 **데이터베이스 복제**라고 해요.
2. 학생들이 필기 내용을 볼 때는 칠판까지 안 가고 그냥 가까운 반장 노트를 빌려보면 되니까, 교실이 안 막히고(읽기 분산) 엄청 빠르게 공부할 수 있죠!
3. 만약 선생님이 갑자기 배가 아파서 양호실에 가도(마스터 장애), 필기를 다 외우고 있는 똑똑한 반장 한 명이 교탁으로 올라와서 선생님 대신(페일오버) 수업을 계속할 수 있는 완벽한 방어 시스템이랍니다!
