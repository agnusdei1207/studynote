# 분산 데이터베이스 데이터 할당 - 단편화, 복제, 투명성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 데이터베이스는 전 세계에 물리적으로 흩어져 있는 데이터를 네트워크로 연결하여, 사용자가 마치 **'단 하나의 지역 시스템'을 사용하는 것처럼 느끼게 하는 논리적 통합 아키텍처**다.
> 2. **가치**: 데이터를 최적의 위치에 쪼개어 배치하는 단편화(Fragmentation)와 할당(Allocation)을 통해 네트워크 전송 지연(Latency)을 최소화하고, 다중 복제(Replication)를 통해 단일 장애점(SPOF)을 제거하여 글로벌 무정지 가용성을 100% 확보한다.
> 3. **융합**: 이 과정에서 필수적으로 발생하는 분산 트랜잭션 병목은 2PC, Paxos/Raft와 같은 분산 합의 알고리즘으로 조율되며, 현대 클라우드의 글로벌 분산 NoSQL 및 NewSQL(Spanner) 아키텍처의 철학적 기반이 된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **분산 데이터베이스 (Distributed Database)**: 네트워크상에 분산된 물리적 DB들이 하나의 논리적 DB처럼 동작하도록 소프트웨어(DDBMS)를 통해 관리되는 시스템.
  - **투명성 (Transparency)**: 시스템의 복잡한 물리적 분산 구조를 사용자로부터 완벽히 숨겨, 사용자는 데이터가 어디에 어떻게 저장되어 있는지 알 필요 없이 쿼리만 던지게 하는 특성.

- **💡 비유**: 분산 데이터베이스의 투명성은 '글로벌 온라인 쇼핑몰'과 같습니다. 소비자는 마우스 하나 클릭(쿼리)할 뿐이지만, 시스템 내부에서는 이 물건이 중국 창고에 있는지 미국 창고에 있는지(위치 투명성), 여러 개로 나뉘어 있는지(단편화 투명성)를 알아서 계산해 내 집 앞까지 묶어서 배송해 줍니다.

- **등장 배경**:
  1. **로컬(Local) 자율성과 성능 보장 필요성**: 글로벌 기업의 경우, 한국 지사의 데이터는 한국 서버에 둬야 지연 속도(Latency) 없이 처리할 수 있다. 중앙 서버 한 곳에 전 세계 데이터가 모이면 병목과 마비가 발생했다.
  2. **장애 허용(Fault Tolerance) 요구**: 중앙 서버 하나가 지진이나 화재로 파괴되면 회사가 멈추는 리스크를 피하기 위해, 서로 다른 대륙에 데이터를 100% 동일하게 복사해 두는(Replication) 재해 복구(DR) 체계가 법제화되었다.
  3. **비용 효율적인 확장 (Scale-out)**: 수십억 원짜리 초고성능 메인프레임 1대(Scale-up)를 사는 대신, 값싼 범용 서버 100대(Scale-out)를 병렬 연결하여 슈퍼컴퓨터 이상의 성능을 내고자 했다.

- **📢 섹션 요약 비유**: 커다란 바위 하나를 혼자 짊어지고 가다 넘어지면 끝장나는 과거와 달리, 바위를 작게 부숴서 수백 명의 일꾼(노드)이 나눠 들고 무전기로 발을 맞춰 걷는 완벽한 군대 시스템입니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 분산 데이터베이스의 4대 투명성 (Transparency) 목표

분산 DB가 진정한 분산 DB로 불리기 위해 사용자로부터 은닉해야 하는 4가지 핵심 원칙이다.

| 투명성 | 의미와 효과 | 예시 (사용자의 관점) |
|:---|:---|:---|
| **위치 투명성 (Location)** | 데이터가 어느 서버나 어느 국가에 물리적으로 저장되어 있는지 알 필요 없음. | `SELECT * FROM EMP@NY_SERVER`가 아니라, 그냥 `SELECT * FROM EMP` 만 쓰면 DB가 알아서 뉴욕 서버로 찾아감. |
| **단편화 투명성 (Fragmentation)** | 테이블이 물리적으로 여러 개로 쪼개져 있다는 사실을 몰라도 됨. (가장 구현하기 어려움) | 서울 DB의 A파티션과 부산 DB의 B파티션을 합친 전체 결과를 하나의 테이블처럼 반환. |
| **복제 투명성 (Replication)** | 장애 방지를 위해 데이터가 여러 대의 서버에 복사본으로 존재한다는 사실을 몰라도 됨. | `UPDATE` 한 번 치면 DB가 알아서 복사본 3개 서버에 다 전파함. 사용자는 원본 1개만 수정했다고 느낌. |
| **병행 투명성 (Concurrency)** | 분산 시스템에서 여러 명이 동시에 접근해도 트랜잭션의 원자성과 일관성이 지켜짐. | 분산 락킹(2PL)과 2단계 커밋(2PC)이 백그라운드에서 동작하여 데이터 충돌 방어. |

+++

### 2. 데이터 단편화 (Fragmentation) 기법

단일 테이블을 여러 노드에 나누어 저장하기 위해 어떻게 자를 것인가에 대한 수학적 기법.

- **수평 단편화 (Horizontal)**: 테이블을 행(Row) 단위로 자름. (예: 아시아 고객은 한국 서버, 미주 고객은 미국 서버로 저장) -> *현대의 샤딩(Sharding)과 완벽히 동일한 개념.*
- **수직 단편화 (Vertical)**: 테이블을 열(Column) 단위로 자름. (예: 직원의 기본 인적사항은 공개 서버에, 연봉 등 민감 정보는 고도의 보안 서버에 별도 저장). 쪼갤 때 반드시 기본 키(PK)가 양쪽에 다 포함되어야 나중에 조인으로 원상 복구(무손실 분해) 가능.
- **혼합 단편화 (Hybrid)**: 수평 단편화 후 특정 열을 다시 수직 단편화하는 결합 방식.

+++

### 3. 데이터 할당 (Allocation) 및 복제 (Replication) 전략

쪼개진 파편들을 전 세계의 어느 서버에 배치하고, 몇 개나 복사해 둘 것인지 결정하는 전략.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 데이터 할당 및 복제의 3가지 설계 전략                 │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 1. 완전 복제 (Fully Replicated) ]                               │
  │   - 원본 데이터를 네트워크 내의 **모든 서버**에 똑같이 복사해 둠.           │
  │   - 👍 장점: Read 속도 최상, 어떤 서버가 죽어도 생존 (최고의 가용성).     │
  │   - 👎 단점: Write 속도 최악. 1건 수정 시 100대 서버 다 바꿔야 함 (2PC 병목).│
  │                                                                  │
  │  [ 2. 부분 복제 (Partially Replicated) ] - (가장 많이 쓰임)         │
  │   - 데이터의 특성에 따라 중요 데이터만 3대 정도의 주요 거점에 복사해 둠.    │
  │   - 👍 장점: Read/Write 성능과 가용성의 적절한 타협점 (Raft 등 Quorum 기반)│
  │                                                                  │
  │  [ 3. 분할 할당 (Partitioned / Non-replicated) ]                   │
  │   - 복제본 없이 오직 1개의 노드에만 각각 쪼개서(샤딩) 보관.               │
  │   - 👍 장점: Write 속도 최상, 스토리지 절약.                         │
  │   - 👎 단점: 해당 서버가 죽으면 그 지역 데이터는 영영 조회 불가능.          │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 분산 DB 아키텍트는 "이 데이터가 쓰기(Write) 위주인가, 읽기(Read) 위주인가?"에 따라 할당 전략을 달리한다. 은행의 마스터 계좌 정보는 가용성이 절대적이므로 **부분 복제(통상 3중 복제)**를 택하며 쿼럼(Quorum) 합의를 거친다. 반면, 단순히 흘러가는 수십억 건의 게임 접속 로그는 **분할 할당(Non-replicated 샤딩)**을 통해 복제 부하를 없애고 디스크 용량을 극단적으로 절약하는 것이 현대 클라우드의 표준 아키텍처다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### 분산 처리 패러다임: 중앙 집중형 vs 분산형 심층 비교

| 비교 항목 | 중앙 집중형 DB (Mainframe) | 분산 데이터베이스 (Distributed DB) |
|:---|:---|:---|
| **아키텍처** | 거대한 단일 서버 (Scale-up) | 네트워크로 묶인 여러 노드 (Scale-out) |
| **관리의 주체** | 단일 OS 및 DBMS 엔진 1개 | 전역(Global) 관리자 + 지역(Local) 관리자 계층 |
| **질의 처리 비용**| 디스크 I/O 최적화가 가장 중요 | **네트워크 통신 비용 최적화가 생명 (수만 배 느림)** |
| **장애 영향도** | 서버 1대 죽으면 시스템 전면 마비 | 일부 노드 죽어도 우회(Failover) 서비스 가능 |
| **복잡도** | 상대적으로 단순 (락킹만 신경 쓰면 됨) | 분산 교착 상태(Deadlock), 네트워크 파티션 등 지옥의 난이도 |

### 과목 융합 관점

- **운영체제 (OS)**: 분산 DB에서 가장 무서운 현상은 여러 서버가 서로 상대방의 데이터를 요청하며 영원히 멈추는 **분산 교착 상태 (Distributed Deadlock)**다. 이를 해결하기 위해 OS의 타임스탬프 기반 교착 상태 예방 알고리즘(Wait-Die, Wound-Wait)이 글로벌 트랜잭션 관리자(GTM)에 융합되어 탑재된다.
- **네트워크 보안**: 데이터가 인터넷 구간이나 WAN 망을 통해 노드 간 동기화되므로, 암호화되지 않은 복제(Replication) 패킷은 중간자 공격(MITM)에 극도로 취약하다. 이를 막기 위해 노드 간 통신 채널의 전면 TLS/SSL 암호화 융합이 필수적이다.

- **📢 섹션 요약 비유**: 중앙 집중형 DB는 왕(메인프레임) 한 명이 모든 결재 서류를 직접 도장 찍는 방식이고, 분산 DB는 각 도지사(Local DB)들이 알아서 도장을 찍고 매일 밤 왕에게 텔레파시(네트워크)로 보고하여 전국을 하나로 통치하는 지방 자치 분권 체계입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 OTT 서비스의 멀티 리전(Multi-Region) 복제 아키텍처

- **문제 상황**: 넷플릭스처럼 전 세계 시청자가 있는 서비스에서, 미국(US-East)에만 DB를 두었더니 한국과 유럽 시청자들의 영화 목록 로딩 속도가 300ms를 초과하며 타임아웃 발생 및 대거 이탈함.
- **의사결정 (부분 복제 및 투명성 보장 도입)**:
  1. **부분 복제 적용**: 영화 카탈로그 정보(Read Heavy)는 미국, 유럽, 한국의 3개 대륙 DB로 **완전 복제(Replication)**하여 로컬 캐시처럼 사용. 한국 유저는 한국 DB에서 10ms 만에 읽어옴. (위치 투명성 달성)
  2. **수평 단편화 (Sharding)**: 유저의 각자 시청 이력(Write Heavy)은 데이터가 너무 커서 복제하지 않고, 각 대륙별 유저는 자신의 대륙 DB에만 저장되게 파티셔닝함. 
  3. **분산 합의 (Paxos)**: 영화 평점이나 좋아요를 누를 경우, 전 세계 DB 중 과반수(Quorum)가 동의하면 커밋되도록 설정하여 분산 데이터의 강한 일관성(Strong Consistency) 유지.

### 도입 체크리스트
- **비즈니스적 네트워크 지연(Latency) 허용 한계**: 글로벌로 데이터를 복제하면 빛의 속도 한계(물리적 거리)로 인해 대륙 간 수백 ms의 지연이 발생한다. 금융 시스템처럼 이 지연(동기식 복제)을 무조건 감내할 것인가, 아니면 SNS처럼 늦게 복제되어도 무방한 비동기식(결과적 일관성) 복제를 허용할 것인가?
- **CAP 정리 기반 아키텍처 선택**: 네트워크가 끊어졌을 때(Partition Tolerance), 양쪽 대륙에서 서로 데이터 쓰기를 허용해 나중에 충돌을 고칠 것인가(Availability, NoSQL 방식), 아니면 아예 쓰기 버튼을 막아버릴 것인가(Consistency, RDBMS 방식)? 분산 DB 도입 전 가장 뼈아픈 결정이다.
- **글로벌 스키마 변경 (DDL) 위험성**: 분산된 100대의 서버에 한 번에 새로운 컬럼을 추가(ALTER TABLE)하는 작업 중 중간에 네트워크가 끊기면 시스템이 반쪽짜리로 폭발한다. 무중단 롤링 업데이트 방안이 준비되어 있는가?

### 안티패턴: 대용량 분산 DB에서 2PC(2단계 커밋) 남용
- 분산된 여러 DB에 트랜잭션을 동시에 보장(원자성)하겠다며 `XA Transaction`이나 2PC 프로토콜을 시스템 전역에 도배하는 행위. 1대의 노드만 잠깐 응답을 늦게 해도 나머지 99대의 노드가 락(Lock)을 쥔 채 무한 대기하게 되어, 클러스터 전체가 도미노처럼 붕괴된다. 분산 환경에서는 보상 트랜잭션(Saga 패턴)이나 결과적 일관성 기반의 이벤트 큐(Kafka)를 이용해 락킹 타임을 없애야 한다.

- **📢 섹션 요약 비유**: 여러 명의 친구들이 전화(네트워크)로 의논해 짜장면을 시킬 때, 무조건 전원이 "응!" 하고 동시에 대답할 때만(2PC) 주문하기로 하면 평생 밥을 못 먹습니다. 한두 명이 화장실 가 있더라도 대충 과반수가 동의하면 일단 시키는 것(Raft 쿼럼)이 분산 시스템이 살아남는 길입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 단일 중앙 서버 아키텍처 | 분산 DB (할당 및 복제 적용) | 개선 효과 |
|:---|:---|:---|:---|
| **응답 지연 속도 (Latency)**| 해외 유저 접속 시 수백 ms 소요 | 데이터 로컬리티(근접 할당) 활용 | **글로벌 응답 지연 90% 이상 단축 (10ms 이내)** |
| **시스템 고가용성 (HA)** | 메인프레임 다운 시 기업 마비 | 멀티 리전 장애 조치 (Failover) | **가동률 99.999% (연간 수 분 이내 장애) 달성** |
| **스케일 아웃 비용** | 수십억 원대 하이엔드 장비 교체 | 수백만 원대 x86 서버 N대 단순 병렬 추가 | **인프라 증설에 따른 TCO(총소유비용) 극단적 절감** |

### 미래 전망
- **NewSQL로의 패러다임 전환**: 과거의 분산 DB는 성능을 얻기 위해 개발자에게 투명성(SQL의 편리함)을 빼앗았다(NoSQL). 하지만 최근 구글 스패너(Spanner)나 CockroachDB 등 NewSQL이 등장하며, 완벽한 분산 쿼럼 복제와 SQL 투명성(ACID)을 동시에 제공하는 신의 경지로 진화하고 있다.
- **엣지 데이터베이스 (Edge Database)**: 스마트폰 기기 자체나 IoT 단말기(엣지)를 아예 분산 DB의 하나의 노드처럼 취급하여, 중앙 클라우드를 거치지 않고 사용자 기기 단에서 초실시간으로 데이터를 할당하고 동기화하는 로컬 퍼스트(Local-first) 분산 아키텍처가 급부상하고 있다.

- **📢 섹션 요약 비유**: 분산 데이터베이스 기술은 과거 중앙 통제형 '독재 국가' 아키텍처를, 전 세계 어디서든 똑같은 혜택과 법이 빠르고 완벽하게 작동하는 '고도화된 연방 클라우드 생태계'로 해방시킨 1등 공신입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **NoSQL 아키텍처와 샤딩 (Sharding)**: 수평 단편화(Horizontal Fragmentation)가 웹 스케일 환경에 맞게 발전하여 정착된 물리적 아키텍처.
- **CAP 정리와 PACELC 정리**: 분산 시스템에 데이터를 할당하고 복제할 때 겪을 수밖에 없는 물리적 한계와 트레이드오프 이론.
- **분산 합의 알고리즘 (Raft, Paxos)**: 복제된 데이터들 간에 완벽히 동일한 버전을 유지하기 위해 마스터 노드가 없는 환경에서도 다수결로 결과를 정하는 수학적 메커니즘.
