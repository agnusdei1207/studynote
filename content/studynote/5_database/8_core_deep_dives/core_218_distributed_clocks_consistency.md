# 분산 시스템 시계 및 데이터 일관성 - Lamport 시계, Vector Clock, TrueTime

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 전 세계에 흩어진 분산 시스템에서는 각 서버의 물리적 시계가 미세하게 어긋나는 **클럭 스큐(Clock Skew)** 현상이 필연적으로 발생하므로, 이벤트의 선후 관계를 명확히 판별하기 위한 **논리적 시계(Logical Clock)**의 도입이 필수적이다.
> 2. **가치**: 람포트 시계(Lamport Clock)와 벡터 시계(Vector Clock)는 절대적인 시간이 아닌 '인과 관계(Causality)'를 보장하여 데이터 충돌을 해결하며, 구글의 트루타임(TrueTime)은 하드웨어를 통해 물리적 시간의 오차 범위를 수학적으로 보장하는 혁신을 이루었다.
> 3. **융합**: 이러한 시계 동기화 알고리즘들은 NoSQL의 다중 마스터(Multi-master) 환경에서 최종 쓰기 승리(Last Writer Wins)의 오류를 막고, NewSQL(Google Spanner)의 전 지구적 트랜잭션 일관성(External Consistency)을 보장하는 핵심 기저 기술로 융합된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 분산 환경에서 여러 노드가 동시에 데이터를 갱신할 때, "어느 노드의 갱신이 먼저 일어났는가?"를 판별하는 것은 데이터 일관성 유지의 핵심이다. 이를 위해 이벤트를 시간순이 아닌 인과순으로 정렬하는 수학적/물리적 매커니즘이 분산 시스템 시계 알고리즘이다.

- **💡 비유**: 한국과 미국에서 동시에 주식 매수 버튼을 눌렀을 때, 각자의 손목시계(물리적 시계) 시간이 1~2초 달라서 늦게 누른 사람이 먼저 샀다고 처리되는 억울한 상황(클럭 스큐)을 막기 위해, 중앙 은행에서 발행한 '일련번호가 적힌 대기표(논리적 시계)'를 뽑게 만드는 원리와 같습니다.

- **등장 배경**:
  1. **NTP(Network Time Protocol)의 한계**: 서버들은 NTP 서버를 통해 시간을 동기화하지만, 네트워크 지연으로 인해 수 밀리초(ms)에서 수 초까지 시간 오차(Clock Skew/Drift)가 반드시 존재한다.
  2. **Last Writer Wins (LWW)의 치명적 결함**: 분산 DB(Cassandra 등)에서 타임스탬프가 가장 늦은 데이터를 최종 승리자로 덮어쓰는 LWW 방식을 썼으나, 시계가 느린 서버의 데이터가 엉뚱하게 옛날 데이터로 취급되어 덮어써지는(Lost Update) 사고가 발생했다.
  3. **인과율(Causality) 보장의 필요성**: "A가 댓글을 달았기 때문에, B가 대댓글을 달았다." 이 경우 A의 시간은 무조건 B보다 앞서야 한다는 '사건의 인과 관계(Happens-before)'를 증명할 수학적 모델이 필요했다.

- **📢 섹션 요약 비유**: 각자 자기 시계만 보고 달리는 달리기 경주에서 1등을 가리기 어렵자, 바통(메시지)을 넘겨받을 때마다 바통에 적힌 숫자를 +1씩 올리기로 약속하여 순서를 명확히 정하는 방식입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 람포트 시계 (Lamport Clock)

분산 시스템의 선구자 레슬리 람포트가 제안한 가장 기본적인 논리적 시계 알고리즘.

- **작동 원리**:
  1. 각 노드는 내부적으로 정수형 카운터(시계 $L$)를 0으로 초기화한다.
  2. 노드 내에서 이벤트가 발생하면 자신의 카운터를 1 증가시킨다. ($L = L + 1$)
  3. 다른 노드로 메시지를 보낼 때, 자신의 현재 카운터 값($L$)을 메시지에 동봉한다.
  4. 메시지를 받은 노드는 자신의 카운터와 수신된 카운터 중 **큰 값**을 선택한 후 1을 더한다. ($L_{new} = \max(L_{own}, L_{recv}) + 1$)
- **한계**: A가 B보다 먼저 일어났다면 $L(A) < L(B)$는 성립하지만, 그 역방향인 $L(A) < L(B)$라고 해서 반드시 A가 B의 원인인 것은 아니다. (동시에 일어난 독립적 사건일 수 있음)

+++

### 2. 벡터 시계 (Vector Clock)

람포트 시계의 한계를 극복하고, 두 사건이 인과관계가 있는지 아니면 완전히 독립적으로 동시(Concurrent)에 발생했는지를 정확히 판별하는 알고리즘. (DynamoDB, Riak 등에서 사용)

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 벡터 시계 (Vector Clock) 동작 및 충돌 감지             │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ Node A ]               [ Node B ]              [ Node C ]     │
  │  [A:0, B:0, C:0]          [A:0, B:0, C:0]         [A:0, B:0, C:0]│
  │     │                        │                       │           │
  │   (A에서 쓰기 발생)              │                       │           │
  │  [A:1, B:0, C:0] ──전송──▶ [A:1, B:1, C:0] (B에서 읽고 쓰기)         │
  │                              │                       │           │
  │     │                        └──전송─▶ [A:1, B:1, C:1] (C에서 읽고 쓰기)│
  │     │                                                │           │
  │  (네트워크 단절 시 독립적 쓰기 발생 시뮬레이션)                          │
  │  [A:2, B:0, C:0]          [A:1, B:2, C:0]                        │
  │     │                        │                                   │
  │     └─────────▶ [병합 요청] ◀─┘                                   │
  │          (충돌 발생! A의 카운터는 좌측이 크고, B의 카운터는 우측이 큼)          │
  │                                                                  │
  │  - 인과관계 판별: 모든 배열의 숫자가 크거나 같으면 후행 사건.                   │
  │  - 충돌(Concurrent): 어떤 숫자는 크고, 어떤 숫자는 작으면 '동시 발생(충돌)'을 의미.│
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 벡터 시계는 단일 정수가 아닌 **클러스터 내 모든 노드의 카운터 배열(Vector)**을 들고 다닌다. 만약 노드 A와 B가 네트워크 단절 상태에서 각자 장바구니 데이터를 업데이트했다면, 나중에 동기화될 때 두 벡터를 비교한다. A 벡터는 `[A:2, B:0]`, B 벡터는 `[A:1, B:2]`라면, 서로 자신이 더 최신이라고 주장하는 숫자가 엇갈리게 된다. 이를 **충돌(Conflict)**이라 부르며, 시스템은 임의로 덮어쓰지 않고 클라이언트 애플리케이션에 두 데이터를 모두 반환하여 "사용자가 직접 병합(Merge) 로직을 짜도록" 위임한다.

+++

### 3. 구글 트루타임 (Google TrueTime)

구글 스패너(Spanner)는 아예 논리적 시계를 버리고, **물리적 시계의 오차를 통제**하는 하드웨어적 접근법을 선택했다.

- **원리**: 각 데이터센터에 GPS 수신기와 원자 시계(Atomic Clock)를 설치했다.
- **API 반환값**: `TT.now()` 함수를 호출하면, 현재 시간이 특정 1개의 타임스탬프가 아니라 **`[earliest, latest]`라는 시간 구간**으로 반환된다. 이 구간의 크기(오차)는 보통 1~7밀리초 이내다.
- **커밋 대기 (Commit Wait)**: 트랜잭션을 커밋할 때, 이 오차 구간(예: 7ms)이 완전히 지나갈 때까지 의도적으로 멈춰서 기다린다. 이렇게 하면 전 세계 어떤 데이터센터에서 발생한 다음 트랜잭션이라도, 절대 이전 트랜잭션의 시간과 겹치지 않는다는 **완벽한 선후 관계(External Consistency)**가 보장된다.

- **📢 섹션 요약 비유**: 람포트는 '순서표'를 나눠주는 방식, 벡터 시계는 '각자가 본 모든 순서표의 목록'을 비교하는 방식이라면, 트루타임은 전 세계의 모든 시계가 0.007초 이상 틀리지 않도록 초정밀 원자 시계를 강제로 채우고 0.007초씩 무조건 쉬게 만드는 무식하지만 확실한 방법입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 시계열 일관성 알고리즘 심층 비교

| 비교 항목 | NTP (물리적 시계) | Lamport Clock | Vector Clock | Google TrueTime |
|:---|:---|:---|:---|:---|
| **기반 메커니즘** | OS 시간 동기화 | 논리적 단일 카운터 증분 | 노드별 다중 카운터 배열 | GPS + 원자시계 하드웨어 |
| **인과관계 증명** | 불가능 (클럭 스큐 존재) | 부분적 가능 (선후 관계만) | **완벽한 인과관계 및 충돌 감지** | **절대적 물리적 선후 관계 보장** |
| **오버헤드** | 없음 | 매우 낮음 | 노드 수에 비례하여 메타데이터 커짐 | 특수 하드웨어 인프라 및 대기 지연(ms) |
| **주요 사용처** | 일반적인 웹 로깅 | 제한적인 학술적 연구 / 기초 | DynamoDB, Riak, Cassandra(선택) | Google Spanner, CockroachDB |

### 과목 융합 관점

- **운영체제 (OS)**: 소프트웨어 레벨의 멀티스레드 동기화(Mutex, Semaphore)가 단일 머신 내의 메모리를 보호한다면, 람포트 시계와 트루타임은 이를 네트워크로 연결된 수천 대의 글로벌 노드 단위로 스케일업한 '분산 상호 배제(Distributed Mutual Exclusion)'의 근간이다.
- **네트워크 (Network)**: TCP/IP 프로토콜의 시퀀스 번호(Sequence Number)가 패킷의 순서를 맞추듯, 논리적 시계는 애플리케이션 레벨의 트랜잭션 순서를 맞추는 거대한 시퀀스 번호 체계로 이해할 수 있다.

- **📢 섹션 요약 비유**: 물리적 시계(NTP)가 고장 난 아날로그 시계라면, 벡터 시계는 서로의 일기장을 대조해 보는 탐정이고, 트루타임은 아예 시간을 통제하는 절대자의 스톱워치와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 SNS의 '좋아요' 및 '댓글' 일관성 유지

- **문제 상황**: 한국에 있는 A가 글을 올리고 1초 뒤에 미국에 있는 B가 댓글을 달았는데, 미국 서버의 물리적 시계(NTP)가 한국보다 2초 느려서, 시스템 상으로는 "댓글이 달린 후 글이 올라온" 물리법칙을 위반한 타임라인이 형성됨.
- **의사결정**:
  1. **벡터 시계 적용**: 글 작성 이벤트에 벡터 `[KR:1, US:0]`를 부여하고, 이를 수신한 미국 서버가 댓글을 달 때 `[KR:1, US:1]`로 업데이트하여 전송.
  2. **인과관계(Causality) 기반 정렬**: 시스템은 타임스탬프의 절대 시간이 아니라, 벡터 시계의 인과관계(`[KR:1, US:0]` -> `[KR:1, US:1]`)를 계산하여 무조건 글이 댓글보다 상단에 노출되도록 타임라인 UI를 렌더링함.

### 도입 체크리스트
- **LWW(Last Writer Wins)의 위험성 수용 여부**: Cassandra 등에서 기본값인 타임스탬프 기반 덮어쓰기(LWW)는 클럭 스큐로 인한 데이터 유실을 100% 방지할 수 없다. 유실되면 안 되는 장바구니 데이터인가, 덮어써져도 무방한 프로필 사진 업데이트인가?
- **형제(Siblings) 데이터 병합 능력**: 벡터 시계로 인해 충돌이 감지되었을 때(예: 장바구니 A에는 사과, 장바구니 B에는 바나나가 담김), 애플리케이션 단에서 이를 합쳐서 "사과, 바나나"로 만들어 줄 커스텀 병합(Merge) 로직을 개발할 여력이 있는가?
- **가비지 컬렉션 (GC)**: 벡터 시계는 클러스터 내 노드가 추가/삭제될 때마다 배열이 무한정 길어지는 단점이 있다. 이를 주기적으로 정리(Pruning)해주는 백그라운드 작업이 설정되어 있는가?

### 안티패턴: 분산 환경에서 단순 `AUTO_INCREMENT`나 `SYSDATE` 의존
- 분산 DB 환경에서 각 노드의 `SYSDATE`(현재 시간)를 기준으로 레코드의 최신 여부를 판단하거나, `AUTO_INCREMENT`로 PK를 부여하는 행위. 서버 간 시간이 1ms만 틀어져도 심각한 데이터 정합성 파괴가 일어나며, PK 충돌로 인해 분산 확장이 원천 차단된다. 분산 환경에서는 UUID/Snowflake ID나 논리적 시계를 반드시 사용해야 한다.

- **📢 섹션 요약 비유**: 서로 다른 나라의 사람들이 화상 회의를 할 때 각자의 벽시계(물리적 시계)를 보고 회의록 순서를 적는 것은 재앙입니다. 반드시 회의록에 "1번 발언, 2번 발언"이라는 순번(논리적 시계)을 명시해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 물리적 시계(NTP) 전존 | 논리적 시계 / 트루타임 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **데이터 정합성** | 시간 오차로 인한 업데이트 유실 발생 | 인과관계 추적 및 강제 대기로 충돌 배제 | **분산 환경의 갱신 손실(Lost Update) 0% 달성** |
| **개발 복잡도** | 앱에서 시간 차이를 보정하려 하드코딩 | DB가 인과성/선후관계 식별 정보 제공 | **분산 트랜잭션 관리 공수 대폭 감소** |
| **확장성 (Scalability)**| 시간 오차 두려워 단일 리더에 쓰기 집중 | 충돌 해결 믿고 멀티 마스터(Multi-Master) 구축 | **쓰기 처리량(Write TPS) 선형적 무한 확장** |

### 미래 전망
- **트루타임의 범용화 (NTP의 진화)**: 구글만의 전유물이었던 원자시계 인프라가 최근 AWS(Amazon Time Sync Service) 등 다른 퍼블릭 클라우드에서도 마이크로초 단위의 시계 동기화를 제공하는 인프라로 대중화되고 있다.
- **CRDT (Conflict-free Replicated Data Type)**: 벡터 시계의 "충돌 발생 시 앱에서 알아서 합쳐라"라는 단점을 개선하여, 수학적으로 덧셈/집합 연산이 교환법칙(A+B=B+A)이 성립하도록 설계해 병합 충돌을 원천적으로(Conflict-free) 없애버리는 궁극의 데이터 타입 연구가 Redis, Riak 등을 중심으로 상용화되고 있다.

- **📢 섹션 요약 비유**: 분산 시스템의 시계 기술은, 전 세계가 흩어져 살던 원시 시대에서 "그리니치 표준시(GMT)"라는 절대 기준을 세워 현대 문명과 글로벌 무역을 가능케 한 인류의 시간 혁명과 완벽히 궤를 같이합니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **CAP 정리와 PACELC 정리**: 네트워크 파티션 상황에서 가용성(A)을 택했을 때 발생하는 데이터 충돌을, 벡터 시계가 어떻게 해결하는지 설명하는 이론적 배경.
- **NewSQL과 분산 데이터베이스의 진화 (Spanner)**: 트루타임(TrueTime) 아키텍처를 도입하여 관계형 DB의 트랜잭션을 전 지구로 확장한 혁신적 DB.
- **다중 버전 동시성 제어 (MVCC)**: 단일 노드 내에서의 타임스탬프 순서화 기법. 이 타임스탬프가 분산 환경으로 확장될 때 논리적 시계가 필요해짐.
