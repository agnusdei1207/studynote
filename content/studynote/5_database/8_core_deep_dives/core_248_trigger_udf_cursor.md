# 트랜잭션 절차 제어 - 트리거(Trigger), UDF, 커서(Cursor)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 집합적이고 선언적인 SQL의 한계를 극복하기 위해, 데이터베이스 내부에 프로그래밍 제어 로직을 심어 **이벤트 시 자동 실행되게 하거나(Trigger)**, **사용자 정의 함수(UDF)로 캡슐화**하고, 집합 데이터를 **행 단위로 쪼개어 반복 처리(Cursor)**하는 서버 사이드 확장 기술이다.
> 2. **가치**: 트리거는 데이터 변경 시 감사(Audit) 로그를 남기거나 복잡한 무결성을 강제하는 데 유용하며, UDF는 복잡한 수식을 단순화한다. 그러나 커서(Cursor)는 관계형 DB의 가장 강력한 무기인 집합 연산(Set Operation)을 포기하고 절차적 반복문(Loop)을 강제하여 극심한 성능 저하를 유발한다.
> 3. **융합**: 과거 2-Tier 시절의 총아였으나, 현대 마이크로서비스(MSA)와 이벤트 주도 아키텍처(EDA) 패러다임이 도래함에 따라, 무거운 DB 트리거 대신 애플리케이션 레벨의 이벤트 퍼블리싱이나 **CDC(Change Data Capture)** 파이프라인으로 그 역할이 완전히 전이/융합되고 있다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **트리거 (Trigger)**: 테이블에 `INSERT/UPDATE/DELETE` 등의 이벤트가 발생할 때, 데이터베이스가 이를 감지하여 설정된 SQL 로직을 '자동'으로 실행하는 스토어드 프로그램.
  - **사용자 정의 함수 (UDF, User-Defined Function)**: 사용자가 직접 작성하여 DB에 등록한 함수. SELECT 절 등에서 내장 함수(`MAX()`, `UPPER()`)처럼 자유롭게 호출 가능.
  - **커서 (Cursor)**: `SELECT`로 가져온 수만 건의 결과 집합(Set)을 한 줄(Row)씩 순차적으로 가리키며 쪼개어 읽어 들일 수 있게 하는 포인터.

- **💡 비유**: 일반 SQL이 식당에 "비빔밥 10그릇 갖다주세요!"라고 한 번에 주문하는 것이라면, **커서**는 "비빔밥 한 그릇 주고, 다 먹으면 다음 그릇 주고..."를 10번 반복하는 것입니다. **트리거**는 "누가 문을 열고 들어오면 무조건 '어서 오세요'라고 외쳐라"라고 문에 달아둔 자동 센서 종입니다.

- **등장 배경**:
  1. **데이터 무결성의 강제**: 애플리케이션 개발자가 주문(Order) 테이블에 데이터를 넣을 때 재고(Stock) 테이블을 차감하는 로직을 빼먹는 실수를 막기 위해, 주문이 들어가면 무조건 재고가 차감되도록 DB 차원에서 방어막(Trigger)을 쳤다.
  2. **SQL의 비절차적 한계**: 집합(Set) 단위로만 동작하는 순수 SQL로는 "이전 행의 값과 다음 행의 값을 비교하여 조건부 루프(Loop)를 도는" 절차적 업무를 구현할 수 없었기에 커서가 도입되었다.
  3. **재사용성**: 세금 계산식처럼 복잡하고 자주 쓰이는 수학적 로직을 쿼리마다 수백 줄씩 복사/붙여넣기 하는 대신, UDF로 묶어 `Calculate_Tax(price)` 한 줄로 깔끔하게 호출하고자 했다.

- **📢 섹션 요약 비유**: 칼과 도마(SQL)만으로 요리하기 힘든 복잡한 상황을 위해, 자동으로 재료를 써는 기계(Trigger)와 전용 믹서기(UDF)를 주방 깊숙한 곳에 맞춤형으로 설치해 둔 것과 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 트리거 (Trigger)의 작동 원리와 마법의 가상 테이블

트리거는 이벤트 발생 전후(`BEFORE / AFTER`)와 적용 범위(`FOR EACH ROW / STATEMENT`)에 따라 나뉜다.

- **가상 테이블 (`NEW` 와 `OLD`)**: 트리거 내부에서는 데이터가 변경되기 전과 후의 상태를 비교할 수 있는 임시 메모리 공간이 제공된다.
  - `NEW`: `INSERT`나 `UPDATE`로 새롭게 들어올 데이터의 값을 담고 있다. (예: `NEW.salary`는 인상 후 월급)
  - `OLD`: `UPDATE`나 `DELETE`로 지워질 과거의 데이터를 담고 있다. (예: `OLD.salary`는 인상 전 월급)
- **활용**: `IF NEW.salary < OLD.salary THEN 에러 발생` 과 같이, 애플리케이션이 막지 못한 논리적 오류(월급 삭감 금지)를 DB 커널 단에서 최후의 보루로 완벽하게 튕겨낸다.

+++

### 2. 사용자 정의 함수 (UDF)와 컨텍스트 스위칭의 저주

UDF는 코드의 가독성을 극도로 높여주지만, 대용량 조회 성능을 파괴하는 주범이 되기도 한다.

- **동작 원리**: `SELECT name, Get_Dept_Name(dept_id) FROM Employee`
- **문제점**: Employee가 100만 명이라면, `Get_Dept_Name()` 함수가 100만 번 호출된다. 이때 DB의 메인 실행 엔진(SQL Engine)과 UDF를 실행하는 절차적 엔진(PL/SQL Engine) 간에 100만 번의 **컨텍스트 스위칭(Context Switching)**이 발생한다.
- 단순히 테이블을 일반 `JOIN`으로 묶으면 0.1초면 끝날 해시 조인 작업이, 스칼라 UDF를 남용하면 10분이 넘게 걸리는 재앙이 발생한다.

+++

### 3. 커서 (Cursor)의 구조와 행 단위 처리

커서는 집합(Set)의 덩어리를 쪼개어 절차 프로그래밍(For, While)을 가능케 한다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 명시적 커서 (Explicit Cursor)의 생명 주기           │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 메모리 영역 (Context Area) ]                                     │
  │   1. DECLARE: 커서의 이름과 실행할 SELECT 문을 정의.                   │
  │                                                                  │
  │   2. OPEN: 쿼리를 실행하여 결과 집합(Active Set)을 메모리에 퍼 올림.     │
  │                                                                  │
  │   3. FETCH ◀─┐ (루프 반복 / Loop)                                 │
  │      - 결과 집합에서 한 행(Row)씩 포인터를 이동하며 데이터를 변수로 꺼냄.   │
  │      - 꺼낸 데이터로 IF 문, UPDATE 등 절차적 비즈니스 로직 실행.         │
  │              │                                                   │
  │   4. CLOSE ──┘ (데이터가 끝나면 루프 탈출 후 메모리 반환)               │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터베이스는 태생적으로 집합 단위 연산(한 번의 조인, 한 번의 업데이트)에 최적화되어 있다. 그러나 커서를 사용하여 루프를 돌면서 한 줄씩 데이터를 빼내어(Fetch) 업데이트를 치면, DB 엔진은 매번 새로운 트랜잭션과 락(Lock)을 관리하느라 극도의 성능 저하(Row-by-Row Processing)를 겪는다. 이를 IT 업계에서는 **"Slow-by-Slow (행 단위 굼벵이 처리)"** 병목이라고 부른다.

## Ⅲ. 융합 비교 및 다각도 분석

### 트랜잭션 절차 제어 3대장 비교

| 비교 항목 | 트리거 (Trigger) | 사용자 정의 함수 (UDF) | 저장 프로시저 (Stored Procedure) |
|:---|:---|:---|:---|
| **실행 주체** | **자동 실행 (이벤트 감지 시 숨어서 작동)** | **수동 호출 (SELECT 등 쿼리 안에서)** | **수동 호출 (CALL 명령어로 단독 실행)** |
| **반환 값** | 없음 (백그라운드에서 로직만 수행) | 무조건 1개의 단일 값 반환 (Scalar) | 반환값이 없거나(OUT 파라미터), 결과 집합 반환 |
| **트랜잭션 제어**| 내부에서 COMMIT/ROLLBACK 불가 (원본에 종속) | 내부에서 트랜잭션 제어 불가 | **내부에서 COMMIT/ROLLBACK 자유롭게 제어 가능** |
| **가장 큰 부작용**| 연쇄 트리거 시 디버깅 지옥, 예상치 못한 락 | 컨텍스트 스위칭으로 인한 풀 스캔 오버헤드 | 비즈니스 로직의 DB 종속(Vendor Lock-in) |

### 과목 융합 관점

- **소프트웨어 공학 (결합도와 응집도)**: 트리거는 특정 테이블에 강하게 결합(Coupling)된 스파게티 로직의 원흉이다. 개발자가 소스 코드를 아무리 뒤져도 버그를 못 찾았는데, 알고 보니 "DB에 누군가 옛날에 몰래 달아둔 트리거"가 백그라운드에서 값을 몰래 바꾸고 있었던(Side-effect) 전설적인 괴담들이 실무에 즐비하다. 현대 소프트웨어 공학에서는 트리거 사용을 극도로 지양한다.
- **분산 데이터베이스 (Event Sourcing)**: 과거에는 "회원이 가입하면(INSERT), 트리거가 돌아서 쿠폰 테이블에 1000원을 넣어라"는 식으로 구현했다. 현대 MSA에서는 `INSERT` 후 **메시지 브로커(Kafka)**로 '가입 완료 이벤트'를 발행하여 쿠폰 마이크로서비스가 비동기로 이를 처리하게 하는(Event-Driven) 방식으로 완벽히 진화했다.

- **📢 섹션 요약 비유**: 트리거는 지뢰밭과 같습니다. 밟으면 터지도록 설정해 둔 사람은 편하지만, 나중에 그 길을 지나가야 하는 후임 개발자는 어디서 어떤 로직이 터질지 몰라 벌벌 떨어야 합니다. 명시적이고 투명한 코드가 최고의 코드입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 개인정보 조회/변경 이력 감사(Audit) 로그의 구현

- **문제 상황**: 보안 감사(Compliance) 대응을 위해 "누가 언제 고객의 비밀번호와 이메일을 수정했는지" 모든 이력을 `Audit_Log` 테이블에 남겨야 함. 이를 애플리케이션(Java)에서 구현하려니 수십 군데의 소스 코드를 모두 찾아 수정해야 하는 막막한 상황.
- **의사결정 (제한적인 트리거 도입)**:
  1. 비즈니스 로직(결제, 재고 차감 등)에는 절대 트리거를 쓰지 않는다는 사내 룰을 전제로, 오직 **'감사(Audit) 목적'**에 한해서만 트리거 도입을 허가함.
  2. 고객 테이블에 `AFTER UPDATE` 트리거를 생성하여, `IF OLD.password != NEW.password THEN` 일 경우에만 `Audit_Log` 테이블에 `[유저ID, 변경시각, 과거값, 새값]`을 INSERT 하도록 설정.
  3. 애플리케이션 코드를 단 1줄도 수정하지 않고 전사적인 감사 로직을 100% 누락 없이 완벽하게 적용 완료.

### 도입 체크리스트
- **Set-based(집합적) 사고로의 전환 (커서 타파)**: 로직에 `FOR` 문이나 `커서(Cursor)`가 보인다면 99%는 조인(Join), 서브쿼리, 윈도우 함수(Window Function)를 이용한 단일 쿼리로 재작성(Rewrite)할 수 있다. 커서 도입 전, "이것이 정말 순수 SQL 집합 연산으로 불가능한 수학적 한계인가?"를 처절하게 고민해야 한다.
- **돌연변이 테이블 (Mutating Table) 에러**: 트리거 내부에서, 그 트리거를 발생시킨 '자기 자신의 테이블'을 다시 `SELECT`하거나 `UPDATE`하려고 하면 오라클 등은 "테이블이 변이 중이다"라며 치명적 런타임 에러를 뱉는다. 이 제약 조건을 회피할 복합 트리거 설계 구조가 있는가?
- **성능 오버헤드 측정**: 트리거는 원본 트랜잭션의 실행 시간(Latency)에 트리거 로직 실행 시간을 고스란히 덧붙인다. 대량의 `Bulk Insert`가 일어나는 테이블에 `FOR EACH ROW` 트리거가 달려 있다면, 초당 수십만 건의 쓰기 성능이 초당 수백 건으로 파멸적으로 폭락한다.

### 안티패턴: 루프(Cursor) 안에서 쿼리 실행하기 (Slow-by-Slow)
- 10만 건의 데이터를 커서로 한 줄씩 Fetch 하면서, 루프 문 내부에서 또 다른 테이블을 향해 `SELECT`나 `UPDATE`를 10만 번 연사하는 최악의 행위. DB 엔진의 파싱, 락 획득, 네트워크 통신 비용이 10만 배로 증폭된다. **SQL은 한 번의 쿼리로 10만 건을 다루도록(Set Operation) 설계된 언어이지, C언어나 파이썬처럼 한 줄씩 다루라고 만든 언어가 아니다.**

- **📢 섹션 요약 비유**: 버스(집합 연산)에 50명을 한 번에 태우고 목적지에 가면 30분이면 끝날 일을, 굳이 커서(택시)를 불러서 1명씩 50번을 왕복하게 시키는 것은 DB 엔진에 대한 모독이자 극심한 자원 낭비입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 비즈니스 로직(WAS) 기반 처리 | 트리거 / 커서 기반 DB 처리 | 개선 효과 및 부작용 |
|:---|:---|:---|:---|
| **감사(Audit) 무결성** | 개발자 누락으로 이력 손실 확률 존재 | DB 커널 단에서 강제 이력 생성 | **보안 컴플라이언스(Compliance) 완벽 100% 충족** |
| **대량 데이터 연산** | 1회의 집합 연산(Set)으로 1초 내 조인 완료 | 행 단위 커서(Loop)로 1시간 소요 | (단점) **연산 성능 수천 배 폭락 (Slow-by-Slow)** |
| **아키텍처 확장성** | Stateless 로 무한 Scale-out 가능 | 로직이 DB에 묶여 서버 증설 불가 | (단점) **클라우드 및 MSA로의 마이그레이션 완전 차단** |

### 미래 전망
- **스트리밍 파이프라인(CDC)으로의 완벽한 대체**: DB 내부의 무거운 트리거는 결국 디스크 I/O와 락을 동반한다. 이에 따라 현대 IT는 트리거를 삭제하고, DB가 자연스럽게 뿜어내는 복구 로그(Binlog)를 외부의 Debezium(CDC)이 가볍게 훔쳐보아 이벤트 스트림으로 전달하는 '제로 임팩트(Zero-impact)' 아키텍처로 완전한 세대교체를 이룩했다.
- **인메모리 기반 프로시저 (Stored Procedure in IMDB)**: SAP HANA, Redis 등 초고속 인메모리 환경에서는 네트워크 지연을 없애기 위해 Lua 스크립트나 특수 프로시저를 커널 메모리 단에서 바로 실행시켜 버리는 기능(Redis Lua Scripting)이 트랜잭션 원자성을 보장하는 가장 가볍고 힙한 무기로 재조명받고 있다.

- **📢 섹션 요약 비유**: 트리거와 커서는 자동차(DB) 초창기에 운전자의 귀찮음을 덜어주기 위해 달아놓은 수많은 '잡다한 기계식 튜닝 버튼'들입니다. 지금은 자율 주행과 중앙 컴퓨터(클라우드/MSA)가 모든 것을 통제하는 시대이므로, 무겁고 고장 나기 쉬운 구시대의 튜닝 장치는 과감히 떼어내야 차가 가장 빠르게 달릴 수 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **저장 프로시저와 동적 SQL**: 트리거와 커서, UDF가 실제로 포장되어 컴파일되는 가장 거대한 껍데기이자 서버 사이드 프로그래밍의 총칭.
- **객체 관계 매핑(ORM/JPA)**: DB 내부에 로직을 숨기는 트리거/UDF 철학과 가장 극단적으로 대척점에 있는 현대의 '순수 애플리케이션 중심' 설계 철학.
- **데이터옵스와 CDC 기술**: 무겁고 시스템에 종속적인 트리거 기능을 성능 저하 0%의 무결점 이벤트 스트림으로 대체해 버린 진화의 결과물.

## 👶 어린이를 위한 3줄 비유 설명
1. **커서**는 레고 상자에 있는 1,000개의 블록을 "빨간색인지 확인해 줘"라고 할 때, 손으로 1개씩 꺼내서 눈으로 보고 다시 내려놓는 아주 느린 행동이에요. (SQL은 바구니째 엎어서 빨간색만 채로 거르는 게 제일 빠르거든요!)
2. **트리거**는 일기장 맨 뒷장에 달아둔 '자동 잉크 함정'이에요. 누군가 몰래 내 일기를 훔쳐보려고 표지를 넘기면(이벤트), 팍! 하고 자동으로 잉크가 터져서(자동 실행) 흔적을 남기게 만들죠.
3. **UDF(사용자 정의 함수)**는 나만의 '구구단 자판기'를 만든 거예요. 자판기에 5를 넣으면 25가 뿅 나오게 만들어 놓고, 아무 때나 버튼 하나로 편하게 계산을 맡기는 거랍니다!
