# 데이터베이스 동시성 제어 - 팬텀 읽기, 락킹, MVCC

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동시성 제어(Concurrency Control)는 다수의 사용자가 동일한 데이터에 동시에 접근하고 갱신할 때, **데이터의 일관성(Consistency)과 고립성(Isolation)**을 훼손하지 않으면서도 시스템의 처리량을 극대화하기 위한 트랜잭션 조율 메커니즘이다.
> 2. **가치**: 2단계 락킹(2PL) 기반의 엄격한 잠금 체계가 유발하던 심각한 성능 저하와 병목을 극복하기 위해, **MVCC(Multi-Version Concurrency Control)** 기술이 도입되어 "읽기 작업이 쓰기 작업을 막지 않는" 혁신적인 성능 향상을 이룩했다.
> 3. **융합**: 이상 현상(Dirty Read, Phantom Read 등)을 방어하기 위한 **트랜잭션 고립화 수준(Isolation Level)** 튜닝은 무결성과 성능(TPS) 간의 비즈니스적 타협을 결정하는 데이터베이스 아키텍처 설계의 가장 핵심적인 융합 지표로 작용한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 동시성 제어는 여러 트랜잭션(Transaction)이 직렬적(Serial)이 아닌 병행(Concurrent, 인터리빙)으로 실행될 때, 마치 혼자서 DB를 점유하여 사용하는 것과 같은 '직렬 가능성(Serializability)'을 수학적으로 보장해 주는 제어 기법이다.

- **💡 비유**: 한 권의 은행 장부에 여러 명의 은행원이 동시에 펜을 들고 잔액을 고쳐 적으려 할 때, 순서대로 펜(Lock)을 넘겨주거나, 각자 장부를 복사(MVCC 스냅샷)해서 먼저 적은 뒤 나중에 원본에 반영하여 잔액이 꼬이지 않게 관리하는 규칙과 같습니다.

- **등장 배경**:
  1. **이상 현상 (Anomaly) 발생**: 통제가 없을 경우 갱신 손실(Lost Update), 아직 확정 안 된 데이터를 읽는 오손 읽기(Dirty Read), 두 번 읽을 때 결과가 달라지는 비반복 읽기(Non-repeatable Read) 등의 치명적 데이터 파괴가 일어났다.
  2. **락(Lock)의 부작용**: 이를 막기 위해 데이터를 읽거나 쓸 때 무조건 락(Lock)을 걸면 일관성은 완벽해지지만, 다른 사용자들이 무한정 대기(Blocking)하거나 교착 상태(Deadlock)에 빠져 시스템 전체가 멈춰버렸다.
  3. **락 프리(Lock-Free) 아키텍처의 요구**: 고성능 인터넷 서비스 시대가 열리며 수만 명의 동시 접속을 락 없이 처리해야 했고, 이에 따라 과거 시점의 데이터를 뷰로 제공하는 MVCC 사상이 RDBMS의 표준으로 자리 잡게 되었다.

- **📢 섹션 요약 비유**: 수백 대의 차가 교차로에 동시에 진입할 때(동시성), 신호등(Lock)만으로 통제하면 엄청난 교통체증이 생기지만, 고가 도로와 지하차도(MVCC 다중 버전)를 뚫어 차들이 멈추지 않고 엇갈려 지나가게 만든 혁명입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 트랜잭션 병행 수행 시 발생하는 4대 이상 현상

데이터의 고립성(Isolation)이 뚫렸을 때 발생하는 치명적인 문제들이다.

| 이상 현상 (Anomaly) | 발생 시나리오 및 원리 | 비유 |
|:---|:---|:---|
| **갱신 손실 (Lost Update)** | A와 B가 동시에 같은 데이터를 읽은 뒤 각자 업데이트하면, 먼저 저장한 A의 값이 B의 덮어쓰기로 인해 허공으로 사라짐. | 하나의 문서 문단을 두 명이 복사해 가서 각자 수정 후 덮어쓰기 하는 꼴. |
| **오손 읽기 (Dirty Read)** | A가 데이터를 수정만 하고 아직 커밋(확정)하지 않았는데, B가 그 임시 데이터를 읽어감. A가 롤백(취소)하면 B는 존재하지 않는 데이터를 가지게 됨. | 식당에서 요리사가 버리려고 뺀 썩은 재료를 종업원이 몰래 손님상에 내간 꼴. |
| **비반복 읽기 (Non-Repeatable Read)** | A가 특정 `Row(행)`를 조회한 뒤 작업을 하는 도중, B가 그 행을 수정/커밋함. A가 다시 조회하면 값이 변해 있어 일관성이 깨짐. (Update 발생) | 방금 잔액을 확인하고 돌아서서 출금하려는데 누군가 잔액을 먼저 빼가서 출금이 안 됨. |
| **유령 읽기 (Phantom Read)** | A가 특정 `범위(Range, 예: 나이 20~30세)`를 조회한 후, B가 해당 범위에 새로운 `Row`를 삽입(Insert)함. A가 다시 조회하면 아까 없던 행이 유령처럼 나타남. | 영화관 빈자리를 확인하고 예매하려는데, 1초 새 그 자리에 누군가 앉아 있는(생겨난) 꼴. |

+++

### 2. 2단계 락킹 프로토콜 (2PL, Two-Phase Locking)

직렬 가능성(Serializability)을 보장하기 위해 트랜잭션이 락(Lock)을 요청하고 반납하는 순서를 엄격히 제한하는 가장 고전적인 규약이다.

- **확장 단계 (Growing Phase)**: 트랜잭션은 데이터에 대해 락(공유 락-S, 배타 락-X)을 **획득(Lock)만** 할 수 있으며, 절대 반납(Unlock)할 수 없다.
- **축소 단계 (Shrinking Phase)**: 한 번이라도 락을 반납하기 시작하면, 이후로는 락을 **반납(Unlock)만** 할 수 있으며, 새로운 락을 획득할 수 없다.
- **엄격한 2PL (Strict 2PL)**: 갱신 연쇄 복귀(Cascading Rollback)를 막기 위해, 쓰기 락(X-Lock)은 트랜잭션이 **커밋(Commit)되거나 롤백될 때까지 절대 반납하지 않고 유지**하는 방식. 현대 DB의 기본 락킹 체계다.

+++

### 3. 혁신적 패러다임: MVCC (Multi-Version Concurrency Control)

Oracle과 PostgreSQL, MySQL(InnoDB)의 핵심 엔진 기술로, 락(Lock) 없이 동시성을 획기적으로 올리는 "다중 버전 데이터 관리" 기술이다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 MVCC의 읽기 일관성 (Read Consistency) 원리           │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 현재 DB 원본 (데이터 블록) ]          [ Undo (롤백) 세그먼트 ]      │
  │  Row A: 잔액=100 (SCN: 100)                                       │
  │                                                                  │
  │  [ 시간 흐름 ]                                                     │
  │  1. TX 1 시작 (시점 SCN: 105)                                      │
  │     TX 1 -> SELECT Row A; (결과: 100)                             │
  │                                                                  │
  │  2. TX 2 시작 (시점 SCN: 110)                                      │
  │     TX 2 -> UPDATE Row A 잔액=200;                                 │
  │     (원본 블록 업데이트됨. 구버전 '100'은 Undo 세그먼트로 밀려남) ────────┐│
  │     TX 2 -> COMMIT; (SCN: 115)                                    ││
  │                                                                   ││
  │  3. TX 1 -> 다시 SELECT Row A;                                    ││
  │     * 옵티마이저 판단: "내 시작 시간(105)보다 현재 원본의 SCN(115)이 미래다!" ││
  │     * 조치: 원본을 읽지 않고, 포인터를 타고 Undo 영역의 구버전을 읽어옴. ◀──┘│
  │     (결과: 100 유지 -> Non-Repeatable Read 완벽 방어!)                │
  │                                                                  │
  │  - 핵심 요약: "쓰기 작업(TX 2)이 읽기 작업(TX 1)을 결코 블로킹하지 않는다."  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 전통적인 락킹 방식에서는 TX 2가 값을 수정(X-Lock)하는 동안 TX 1은 조회를 위해 무한정 대기(Blocking)해야 했다. 하지만 **MVCC**에서는 데이터를 수정할 때 이전 버전의 스냅샷 데이터를 임시 공간(Undo Segment)에 저장해 둔다. 읽기 트랜잭션(TX 1)은 다른 트랜잭션이 수정을 하든 말든 상관없이, 자신의 트랜잭션이 시작된 시점(SCN)을 기준으로 Undo 영역에 저장된 '과거 스냅샷'을 읽어들인다. 이를 통해 **읽기와 쓰기 간의 락 경합이 100% 제거**되며 트랜잭션 성능이 극단적으로 향상된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 트랜잭션 고립화 수준 (Isolation Level) 완벽 가이드

미국 표준 협회(ANSI/ISO) SQL 표준이 정의한 4단계 고립화 수준. 레벨이 높아질수록 데이터 정합성은 완벽해지나, 동시 처리 성능(TPS)은 수직 하락하는 치명적 트레이드오프를 갖는다.

| 레벨 (Level) | 명칭 | 방어하는 이상 현상 | 뚫리는 이상 현상 (발생 가능) | 특징 및 RDBMS 기본값 |
|:---|:---|:---|:---|:---|
| **Level 0** | **Read Uncommitted** | (없음) | Dirty Read, Non-Repeatable, Phantom | 커밋 안 된 데이터도 막 읽음 (거의 안 씀) |
| **Level 1** | **Read Committed** | Dirty Read | Non-Repeatable Read, Phantom Read | 커밋된 것만 읽음. **(Oracle, SQL Server 기본값)** |
| **Level 2** | **Repeatable Read** | Dirty, Non-Repeatable | Phantom Read (MVCC에선 대부분 방어됨) | 트랜잭션 내내 읽은 값 유지. **(MySQL InnoDB 기본값)** |
| **Level 3** | **Serializable** | **모두 완벽히 방어** | (없음, 완벽한 일관성) | 가장 엄격. 인덱스/테이블 전체에 Range 락을 걸어 병렬성 최악 |

### 과목 융합 관점

- **운영체제 (OS)**: 데이터베이스의 교착 상태(Deadlock, 트랜잭션 2개가 서로의 락을 물고 무한 대기) 처리 기법인 타임스탬프 기반 선점(Wound-Wait)과 비선점(Wait-Die) 기법은, 운영체제의 프로세스 자원 할당 데드락 예방 기법인 은행원 알고리즘(Banker's Algorithm)의 사상을 그대로 DB 계층에 구현한 것이다.
- **분산 시스템 (NoSQL)**: 관계형 DB가 MVCC와 락킹을 통해 단일 노드에서 ACID 일관성을 극도로 챙기는 반면, 분산 NoSQL(Cassandra 등)은 네트워크 단절 시나리오를 고려하여 아예 락을 포기하고 BASE 특성에 기반한 결과적 일관성(Eventual Consistency)으로 노선을 변경하였다.

- **📢 섹션 요약 비유**: 고립화 수준을 올리는 것은 자동차 에어백과 안전벨트 개수를 늘리는 것(무결성 증가)과 같아서, 완벽한 방탄 차(Serializable)를 타면 절대 죽지 않지만 차가 무거워져 목적지(응답 시간)에 너무 늦게 도착하는 타협의 예술입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 인기 콘서트 티켓 예매 시스템의 초과 예매(Overbooking) 방어

- **문제 상황**: 방탄소년단 콘서트 예매 오픈 시, 좌석 1개에 대해 100명이 동시에 `SELECT`로 남은 좌석 여부를 확인한 뒤, 동시에 100명이 `UPDATE`를 쳐버려 "1개 좌석이 100명에게 예매 완료"되는 갱신 손실(Lost Update) 사고 발생.
- **의사결정 (비관적 락 vs 낙관적 락 도입)**:
  1. **비관적 동시성 제어 (Pessimistic Lock)**: 예매를 시도하는 첫 `SELECT` 시점에 쿼리 끝에 `FOR UPDATE` 옵션을 붙여 해당 행에 즉시 배타 락(X-Lock)을 강제로 건다. 나머지 99명은 락이 풀릴 때까지 대기(Blocking)하다가 남은 좌석이 0임을 확인하고 튕겨 나간다. 데이터는 100% 안전하나 DB 스레드가 고갈되어 서버가 다운될 위험이 큼.
  2. **낙관적 동시성 제어 (Optimistic Lock) (채택)**: 락을 전혀 걸지 않는다. 대신 테이블에 `Version` 컬럼을 둔다. 앱에서 `UPDATE ... WHERE id=1 AND Version=1`로 쿼리를 날려, 100명 중 가장 먼저 도달해 버전을 2로 올린 사람만 성공(Commit)하고, 간발의 차로 늦은 99명은 "Version 불일치 에러"를 뱉으며 애플리케이션 레벨에서 깔끔하게 실패 처리됨. DB 부하가 전혀 없어 대용량 트래픽에 적합함.

### 도입 체크리스트
- **비즈니스 특성 파악**: 항공권 예매나 금융 계좌 이체처럼 돈과 직결되어 '조회 시점의 데이터'가 절대 변하면 안 되는가? (이 경우 `FOR UPDATE` 기반의 명시적 락이 필수적). 아니면 커뮤니티 게시글 조회수처럼 약간 오차가 생겨도 무방한가?
- **Undo 공간 (롤백 세그먼트) 모니터링**: MVCC 기반(Oracle, MySQL)에서 누군가 트랜잭션을 끝내지 않고(Commit 없이) 오랫동안 열어두면, DB는 이 사람에게 과거 스냅샷을 보여주기 위해 무한정 Undo 데이터를 쌓아두다가 디스크 풀(Snapshot Too Old 에러)로 폭발한다. 장기 실행 쿼리(Long-running Query)가 있는가?
- **데드락 (Deadlock) 회피 설계**: 애플리케이션에서 여러 테이블의 데이터를 갱신할 때, A->B 테이블 순서로 업데이트하는 로직과 B->A 테이블 순서로 업데이트하는 로직이 공존하면 100% 교착 상태가 발생한다. 항상 모든 로직이 동일한 순서(A->B)로 락을 획득하도록 코딩 컨벤션이 잡혀 있는가?

### 안티패턴: 기본 고립화 수준을 무시하고 무조건 Serializable로 올리기
- "데이터 꼬이는 건 절대 못 본다"며 DB 접속 세션의 Isolation Level을 최상위인 `Serializable`로 설정하는 행위. 이 수준에서는 단순 `SELECT` 범위 검색에도 락이 걸려 다른 사람이 데이터를 입력(INSERT)하는 것조차 막아버린다(Range Lock). 트랜잭션 동시성이 0에 수렴하여 웹 서비스 전체가 멈춰버리는 대참사를 낳는다. 현대 RDBMS는 Read Committed 또는 Repeatable Read + 앱 레벨 락 컨트롤로 풀어내는 것이 정석이다.

- **📢 섹션 요약 비유**: 누가 내 책을 훔쳐 갈까 봐 도서관 자체를 폐쇄하고 나 혼자 들어가는 것(Serializable)은 확실하지만 너무 이기적입니다. 책에 이름표(버전 컬럼)를 붙여두고 누가 건드리면 알람이 울리게 하는 것(낙관적 락)이 가장 현명한 분산 도서관의 지혜입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 튜닝 전 (2PL 위주의 단순 락킹) | 튜닝 후 (MVCC + 낙관적 락 제어) | 개선 효과 |
|:---|:---|:---|:---|
| **읽기/쓰기 대기 시간** | 쓰기 작업 시 수십~수백 개의 쿼리 블로킹 | 읽기와 쓰기가 서로의 락을 요구하지 않음 | **블로킹/타임아웃 에러 99% 근절** |
| **시스템 처리량 (TPS)** | 데드락 빈발 및 CPU 낭비율 극심 | 락 경합 없는 초고속 스냅샷 읽기 | **트랜잭션 동시 처리량(Concurrency) 10배+ 증대** |
| **데이터 정합성** | 갱신 손실(Lost Update) 사고로 재무 손실 | 버전 관리 및 명시적 락으로 원천 차단 | **데이터 오류 0화 및 완벽한 ACID 준수** |

### 미래 전망
- **인메모리 트랜잭션 제어 (STM)**: 기존 디스크 기반의 무거운 락킹 메커니즘을 탈피해, 메모리 상에서 CPU 캐시 일관성 프로토콜을 극한으로 활용하여 락 없이 초고속으로 트랜잭션을 조율하는 소프트웨어 트랜잭션 메모리(STM) 기술이 차세대 IMDB 엔진에 속속 도입되고 있다.
- **분산 트랜잭션과 HTAP 결합**: 단일 머신에서의 MVCC를 넘어, TiDB나 Google Spanner처럼 전 세계 수백 대의 분산 노드에 걸친 락킹과 스냅샷 격리를 완벽히 조율하는 기술이 발전하여, OLTP(트랜잭션)와 OLAP(분석)가 락 충돌 없이 동시에 실행되는 HTAP의 시대를 완성하고 있다.

- **📢 섹션 요약 비유**: 동시성 제어 기술은 좁은 골목길에 무작정 밀려드는 차들을 교통경찰(Lock)이 수동으로 통제하던 시대에서, 차들이 서로 허공을 관통하는 홀로그램(MVCC)이 되어 속도 저하 없이 완벽하게 교차하는 마법의 미래 도시로 진화한 것입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **트랜잭션 ACID 특성**: 동시성 제어가 완벽히 수행될 때 결과적으로 보장되는 트랜잭션의 4대 기본 법칙(원자성, 일관성, 고립성, 영속성).
- **데이터베이스 교착 상태 (Deadlock)**: 두 트랜잭션이 서로의 락(Lock)을 놓지 않아 무한 대기하는, 동시성 제어 실패 시 발생하는 최악의 현상.
- **분산 시스템 시계 및 일관성 (TrueTime)**: 단일 노드의 MVCC 타임스탬프 스냅샷 개념이, 수백 대의 분산 노드로 확장되었을 때 어떻게 시간을 동기화하는지 보여주는 상위 개념.

## 👶 어린이를 위한 3줄 비유 설명
1. **갱신 손실**은 두 친구가 같은 도화지에 동시에 그림을 그리다가, 나중에 그린 친구의 그림만 남고 먼저 그린 친구 그림은 통째로 물감에 덮여 사라져버리는 억울한 일이에요.
2. 이걸 막기 위해 처음엔 붓(Lock)을 딱 하나만 두고 한 명씩만 그리게(2단계 락킹) 했더니, 뒤에 기다리는 수백 명의 친구들이 하루 종일 줄을 서서 멈춰버렸죠.
3. 그래서 똑똑한 선생님이 **MVCC**라는 마법 도화지를 발명했어요! 누군가 그림을 그리고 있어도, 다른 친구는 그림을 그리기 직전의 도화지 '사진(스냅샷)'을 보면서 방해받지 않고 자기 일을 마저 할 수 있게 된 거랍니다!
