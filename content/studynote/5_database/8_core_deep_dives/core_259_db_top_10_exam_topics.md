# 데이터베이스 최빈출 10대 핵심 토픽 총정리 (기술사/실무 대비)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 공학은 '어떻게 데이터를 중복 없이 완벽하게 저장할 것인가(정규화)'와 '어떻게 원하는 데이터를 가장 빨리 꺼내올 것인가(인덱스/튜닝)'라는 두 가지 근본적인 물리적/수학적 숙제를 푸는 학문이다.
> 2. **가치**: 정보관리기술사, 컴퓨터응용시스템기술사 및 수석 엔지니어 면접에서 가장 빈번하게 다뤄지는 **동시성 제어(MVCC), 실행 계획(CBO), 트랜잭션 회복(ARIES), 분산 시스템(CAP/PACELC)** 등 10대 핵심 토픽을 완벽히 관통해야만 시스템의 한계를 설계할 수 있다.
> 3. **융합**: 각 토픽은 개별적으로 존재하지 않고, "B+Tree의 한계가 파티셔닝을 낳고, 파티셔닝이 샤딩을 낳으며, 샤딩이 2PC와 Saga 패턴을 강제한다"는 식의 **필연적인 인과관계와 진화의 융합 트리**를 구성한다.

+++

## Ⅰ. 관계형 모델과 설계의 근본 (Foundation & Design)

### 1. 정규화 (Normalization)와 반정규화 (De-normalization)
- **핵심**: 데이터 중복에 의한 이상 현상(Anomaly: 삽입, 수정, 삭제)을 막기 위해 함수적 종속성(FD)을 기준으로 테이블을 무손실 분해하는 과정.
- **키워드**: 제1정규형(원자성), 제2정규형(부분 종속 제거), 제3정규형(이행 종속 제거), BCNF(모든 결정자가 후보키), 반정규화(성능을 위한 의도적 중복/조인 회피).
- **실무/면접 포인트**: "OLTP 환경에서는 3NF/BCNF를 원칙으로 하되, 대량 조인이 발생하는 OLAP 환경(스타 스키마)이나 SNS 피드 화면에서는 조회 성능 극대화를 위해 반정규화가 필수적임을 설명할 수 있는가?"

### 2. 관계 대수 (Relational Algebra)와 SQL
- **핵심**: SQL이 겉으로는 '무엇을(What)'을 요구하는 선언적 언어지만, 그 밑바닥은 옵티마이저가 '어떻게(How)'를 수학적으로 계산하는 관계 대수 트리($\sigma$, $\pi$, $\bowtie$)로 번역됨을 이해하는 것.
- **키워드**: 순수 관계 연산(Select, Project, Join, Division), 카티션 곱(Cartesian Product), 뷰(View) 머징.

+++

## Ⅱ. 트랜잭션과 무결성 보장 (Transaction & Concurrency)

### 3. 트랜잭션 ACID 원칙과 ARIES 복구 알고리즘
- **핵심**: DB가 장애로 뻗어도 데이터를 절대 잃어버리지 않는(Durability) 철학.
- **키워드**: 원자성(Atomicity), 일관성(Consistency), 고립성(Isolation), 영속성(Durability). WAL(Write-Ahead Logging), ARIES (Analysis -> Redo -> Undo).
- **실무/면접 포인트**: "왜 데이터를 디스크 원본에 바로 쓰지 않고(No-Force), 메모리에서 로그(Redo Log)만 먼저 디스크로 내리는가? 디스크 I/O 병목(Random vs Sequential)을 정확히 이해하고 있는가?"

### 4. 동시성 제어 (Concurrency Control)와 고립화 수준 (Isolation Level)
- **핵심**: 수만 명이 동시에 데이터를 수정할 때 데이터가 꼬이는 것(Lost Update)을 막는 기술.
- **키워드**: 2단계 락킹(2PL), 교착 상태(Deadlock), **MVCC(다중 버전 동시성 제어)**, Read Committed, Serializable, 팬텀 읽기(Phantom Read).
- **실무/면접 포인트**: "락(Lock)을 걸면 시스템이 멈춘다. 읽기와 쓰기가 서로를 방해하지 않게 만들기 위해 Undo 영역(스냅샷)을 활용하는 MVCC의 혁명적 원리를 설명할 수 있는가?"

+++

## Ⅲ. 성능 최적화의 꽃 (Performance & Tuning)

### 5. 인덱스 (Index) 아키텍처
- **핵심**: 디스크를 전부 뒤지는 풀 스캔(Full Scan)의 악몽을 끝내고 $O(\log N)$으로 데이터를 찾아내는 마법.
- **키워드**: **B+Tree** (리프 노드 연결 리스트), 클러스터링 인덱스(데이터 물리 정렬), 세컨더리 인덱스, 커버링 인덱스(테이블 랜덤 액세스 제거), Bitmap/Hash 인덱스.
- **실무/면접 포인트**: "왜 범위 검색(BETWEEN)이 들어가면 결합 인덱스의 후행 컬럼은 인덱스 역할을 못 하는가? 인덱스 스킵 스캔(Skip Scan)과 클러스터링 팩터(CF)의 개념을 묻는 단골 질문."

### 6. 옵티마이저 (Optimizer)와 SQL 튜닝
- **핵심**: 수백 개의 조인 경우의 수 중 가장 싼 길을 찾는 DB의 두뇌.
- **키워드**: RBO(규칙 기반) vs **CBO(비용 기반)**, 딕셔너리 통계 정보, 하드 파싱(Hard Parsing) vs 소프트 파싱, 조건 푸시 다운(Pushdown), Nested Loop / Sort Merge / Hash Join.
- **실무/면접 포인트**: "변수를 리터럴(1, 2)로 하드코딩했을 때 왜 SGA 공유 풀(Shared Pool) 래치 경합이 발생하여 DB가 죽는지(하드 파싱 오버헤드), 바인드 변수(`?`)의 중요성을 방어할 수 있는가?"

+++

## Ⅳ. 대용량 분산 시스템 아키텍처 (Distributed & NoSQL)

### 7. 파티셔닝(Partitioning)과 샤딩(Sharding)
- **핵심**: 테라바이트급 통짜 테이블을 작게 쪼개어 디스크 I/O를 최적화하고(파티셔닝), 한 대의 서버 한계를 넘어 수십 대의 서버로 수평 확장(샤딩)하는 스케일 아웃 기법.
- **키워드**: Range/Hash 파티셔닝, 파티션 프루닝(Pruning), Shared-Nothing 아키텍처, 핫스팟(Hotspot) 방지, 일관된 해싱(Consistent Hashing).

### 8. CAP 정리와 PACELC, 그리고 NoSQL
- **핵심**: "분산 환경에서는 네트워크가 끊어지는 재앙(Partition)을 피할 수 없으므로, 완벽한 일치(Consistency)나 중단 없는 서비스(Availability) 중 하나는 무조건 포기해야 한다"는 물리학적 한계.
- **키워드**: CAP 이론, PACELC, 결과적 일관성(Eventual Consistency), Document Store(MongoDB), Wide-Column(Cassandra), Key-Value(Redis), LSM-Tree.
- **실무/면접 포인트**: "RDBMS의 강한 일관성(ACID)을 포기하면서까지 NoSQL(BASE)을 도입해야만 했던 웹 스케일 비즈니스의 배경과, 최근 이를 다시 극복하려는 NewSQL(Spanner)의 트루타임(TrueTime) 원리를 아는가?"

### 9. 분산 트랜잭션과 2PC, Saga 패턴
- **핵심**: 여러 대의 DB로 데이터가 쪼개졌을 때(MSA 환경), 양쪽 DB의 트랜잭션을 어떻게 묶어서(원자성) 처리할 것인가의 난제.
- **키워드**: 2단계 커밋(2PC, 동기식 락킹 병목), **Saga 패턴 (비동기 이벤트 + 보상 트랜잭션)**, 스플릿 브레인(Split Brain), 쿼럼(Quorum) 합의.

+++

## Ⅴ. 현대적 데이터 생태계 (Modern Data Ecosystem)

### 10. 데이터 레이크하우스와 데이터 메시 (Lakehouse & Data Mesh)
- **핵심**: 전통적 DW의 한계(비정형 데이터 수용 불가, 스토리지 고비용)를 무너뜨린 최신 클라우드 네이티브 아키텍처.
- **키워드**: ETL vs ELT, 스키마 온 리드(Schema-on-read), 데이터 레이크하우스(Iceberg, ACID 보장 객체 스토리지), **데이터 메시(도메인 주도 분산 오너십)**, 데이터 패브릭(가상화).
- **실무/면접 포인트**: "왜 중앙 데이터팀이 모든 파이프라인을 관리하는 것이 병목이 되었으며, 이를 조직적(Mesh) / 기술적(Fabric)으로 어떻게 풀어내고 있는가?"

## 📌 마스터 맵 (Master Connection)
데이터베이스의 학습은 이 10개의 꼭짓점을 서로 선으로 잇는 과정입니다.
- **[성능의 한계]**: `1.정규화`를 너무 완벽하게 하면 조인이 많아져 느려짐 $\rightarrow$ `5.인덱스`와 `6.조인 튜닝`으로 극복 시도 $\rightarrow$ 데이터가 너무 많아지면 결국 `7.파티셔닝/샤딩`으로 쪼갬.
- **[분산의 대가]**: `7.샤딩`으로 데이터를 쪼개면 단일 DB의 `3.ACID`가 깨짐 $\rightarrow$ `8.CAP 정리`의 딜레마에 빠짐 $\rightarrow$ 이를 `9.Saga 패턴`으로 비동기적 해결을 도모함.
- **[결론]**: 이러한 모든 트랜잭션의 결과물(Data)은 결국 `10.데이터 레이크하우스`로 흘러들어 기업의 궁극적인 AI/BI 자산이 됩니다.
