# 저장 프로시저(Stored Procedure)와 동적 SQL - 장단점과 보안 위협

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 저장 프로시저(Stored Procedure)는 일련의 SQL 쿼리와 절차적 제어 로직(IF, WHILE)을 데이터베이스 엔진 내부에 미리 컴파일하여 영구적으로 저장해 두고, 클라이언트가 함수처럼 호출(CALL)하여 사용하는 **서버 사이드(Server-side) 프로그램**이다.
> 2. **가치**: 복잡한 비즈니스 로직을 한 번의 네트워크 호출로 처리하여 네트워크 라운드트립(Round-trip) 지연을 극단적으로 줄이며, 테이블 접근 권한을 감추고 프로시저 실행 권한만 부여하는 등 캡슐화와 보안을 강화한다.
> 3. **융합**: 과거 2-Tier 아키텍처의 총아였으나, 현대의 클라우드 마이크로서비스(MSA)와 객체 지향 ORM(JPA) 패러다임과는 정면으로 충돌하여 '레거시(Legacy)'로 취급되기도 한다. 그러나 런타임에 쿼리가 조합되는 **동적 SQL(Dynamic SQL)**의 인젝션 방어 기법과 융합되어 대규모 일괄 배치(Batch) 처리 등 특정 영역에서는 여전히 압도적 우위를 점한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **저장 프로시저 (Stored Procedure, SP)**: DB 내부에 저장된 SQL 묶음. Oracle의 PL/SQL, SQL Server의 T-SQL 등 벤더(Vendor)별로 독립적인 절차적 언어를 사용한다.
  - **동적 SQL (Dynamic SQL)**: 개발 시점(Compile-time)에 쿼리의 구조가 고정되지 않고, 런타임(Run-time)에 사용자의 입력값이나 조건에 따라 SQL 문자열이 실시간으로 조립되어 실행되는 방식.

- **💡 비유**: 일반 SQL 쿼리가 '손님이 요리사에게 양파 썰고, 물 끓이고, 소스 넣으라고 매번 하나하나 지시하는 것(네트워크 지연)'이라면, 저장 프로시저는 요리사에게 '짜장면 레시피(SP)'를 아예 외우게 한 뒤 "짜장면 하나!"(CALL)라고 한마디만 하면 주방에서 알아서 요리가 튀어나오는 것과 같습니다.

- **등장 배경**:
  1. **네트워크 대역폭의 한계**: 과거에는 네트워크 속도가 매우 느려, 애플리케이션(WAS)과 DB 서버 간에 10개의 SQL 쿼리를 주고받는 과정 자체가 심각한 병목이었다.
  2. **SQL의 비절차적 한계 극복**: SQL은 집합적/선언적 언어라 반복문(Loop)이나 예외 처리(Try-Catch)가 불가능했기에, DB 내부에서 프로그래밍을 할 수 있는 강력한 절차적 확장이 요구되었다.
  3. **보안과 캡슐화 (Encapsulation)**: 평사원에게 '급여(Salary)' 테이블의 `SELECT/UPDATE` 권한을 직접 주지 않고, `Update_Salary()`라는 잘 정의된 프로시저 실행 권한만 부여하여 임의의 데이터 조작을 원천 차단하고자 했다.

- **📢 섹션 요약 비유**: 애플리케이션 서버(WAS)가 무거운 짐을 다 짊어지는 대신, 데이터가 가장 가까이 있는 데이터베이스 뱃속(엔진 내부)에 똑똑한 비서(프로시저)를 심어두고 힘든 계산을 떠넘기는 아키텍처입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 저장 프로시저의 실행 메커니즘과 성능의 비밀

일반적인 SQL이 실행될 때마다 매번 구문 분석(Parsing)을 거치는 것과 달리, SP는 한 번 생성(Create)될 때 파싱과 컴파일을 마쳐 캐시(Cache)에 실행 계획을 저장한다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 일반 SQL vs 저장 프로시저(SP) 실행 파이프라인           │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [ 일반 SQL 쿼리 (Hard Parsing 발생 시) ]                             │
  │  Client ──(Query)──▶ [ Syntax 검사 ] ──▶ [ Semantic 검사 ] ──┐      │
  │                                                              │      │
  │  [ 실행 (Execute) ] ◀── [ 실행 계획 생성 (Optimizer) ] ◀───────┘      │
  │   (매번 무거운 CPU 연산 오버헤드 발생)                                   │
  │                                                                  │
  │  [ 저장 프로시저 (Stored Procedure) 호출 시 ]                       │
  │  Client ──(CALL SP)──▶ [ 프로시저 캐시(Procedure Cache) 확인 ]       │
  │                                │                                 │
  │  [ 실행 (Execute) ] ◀── (Hit! 이미 컴파일된 실행 계획 즉시 재사용)        │
  │                                                                  │
  │  - 장점: 파싱 연산 회피, 네트워크 트래픽 급감 (수백 줄 SQL 대신 이름만 전송)│
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클라이언트가 `CALL Calculate_Bonus(2026)`를 날리면, 데이터베이스는 구문 분석이나 실행 계획 최적화를 생략하고 **프로시저 캐시**에 올라가 있는 바이트코드(실행 계획)를 즉각 꺼내어 실행한다. 또한, 부서원 1,000명의 보너스를 계산하기 위해 WAS와 DB가 1,000번 통신(Round-trip)해야 하는 로직을, 프로시저 내의 `FOR` 루프로 감싸버리면 통신 횟수가 단 1번으로 줄어들어 배치(Batch) 처리 시 성능이 폭발적으로 상승한다.

### 2. 동적 SQL (Dynamic SQL)의 명암

검색 조건(예: 검색어가 이름인지, 부서인지, 기간인지)이 동적으로 변하는 화면을 처리하기 위해, 프로그램이 런타임에 SQL 문자열을 `String sql = "SELECT * FROM EMP WHERE " + condition;` 형태로 조립하여 실행하는 기법.

- **장점**: 수많은 검색 콤비네이션(조합)을 단 하나의 코드로 유연하게 대응할 수 있다.
- **단점 (보안 위협)**: SQL 인젝션(Injection) 해킹의 1순위 타겟이다.
- **단점 (성능 저하)**: 쿼리 문자열이 매번 달라지므로 DB가 이를 매번 새로운 쿼리로 인식하여 **하드 파싱(Hard Parsing)**을 유발, Shared Pool(메모리)을 고갈시키고 CPU 100% 장애를 일으킨다.

+++

### 3. 바인드 변수 (Bind Variable)와 보안 융합

동적 SQL의 보안 및 성능 문제를 동시에 해결하는 가장 완벽하고 원초적인 방법이다.

- **원리**: 쿼리 문자열에 값을 직접 이어 붙이지 않고, `SELECT * FROM EMP WHERE name = ?` 처럼 자리표시자(Placeholder)를 둔 뒤 런타임에 파라미터 값만 바인딩(Bind)한다.
- **방어 효과**: 파서는 쿼리 구조를 미리 고정시키므로, 해커가 입력창에 `' OR 1=1 --` 와 같은 파괴적인 명령어를 넣어도 이를 단순한 '문자열 데이터'로만 취급하여 SQL 인젝션이 무력화된다.

- **📢 섹션 요약 비유**: 동적 SQL에 값을 그대로 이어 붙이는 건 서류에 도장(컴파일)을 찍기 전에 빈칸을 연필로 놔두는 것이라 누구나 몰래 내용을 바꿔치기(해킹) 할 수 있습니다. 바인드 변수는 서류에 도장을 다 찍고(구조 고정) 코팅까지 마친 뒤 포스트잇(데이터)만 붙였다 떼는 안전한 방법입니다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### 비즈니스 로직 배치: WAS(애플리케이션) vs DB(저장 프로시저)

현대 아키텍처의 가장 뜨거운 논쟁거리 중 하나다. 어디에 비즈니스 로직을 둘 것인가?

| 비교 항목 | WAS 중심 로직 (Spring/Node.js + ORM) | DB 중심 로직 (Stored Procedure) |
|:---|:---|:---|
| **스케일 아웃 (확장성)** | 매우 쉬움 (Stateless WAS 인스턴스 무한 복제) | **매우 어려움 (DB 장비는 수직 확장-Scale Up 위주)** |
| **코드 버전 관리/테스트** | Git, JUnit, CI/CD 파이프라인 완벽 연동 | **버전 관리 어려움, 디버깅/단위 테스트 환경 열악** |
| **벤더 종속성(Lock-in)** | JPA 방언(Dialect) 변경으로 DB 자유롭게 교체 | **PL/SQL, T-SQL 등 벤더에 완벽히 종속되어 이관 불가** |
| **대용량 배치 성능** | 대량 데이터 Fetch 시 네트워크 병목 발생 | **데이터가 있는 DB 커널에서 직접 연산하므로 초고속** |
| **현대 트렌드** | **90% 이상의 표준 아키텍처 (MSA에 적합)** | 데이터 이관, 정산 등 제한적 배치 업무로 축소 (레거시) |

### 과목 융합 관점

- **마이크로서비스 아키텍처 (MSA)**: MSA의 핵심 철학은 각 서비스가 독립적인 생명주기를 가지는 것이다. 만약 핵심 로직을 공유 DB의 프로시저에 넣어두면, 수십 개의 서비스가 얽혀버리는 '통합 DB 안티패턴(Shared Database Anti-pattern)'에 빠져 MSA의 유연성이 완전히 파괴된다.
- **클라우드 및 DevOps**: 프로시저 코드는 형상 관리(Git) 체계에 태우기 까다롭고 지속적 배포(CI/CD) 시 롤백 제어가 어려워, 클라우드 네이티브와 DataOps 트렌드에서는 철저히 배제당하는 추세다.

- **📢 섹션 요약 비유**: 저장 프로시저는 튼튼하고 빠르지만 부품을 바꾸기 불가능한 '용접된 커스텀 자동차'입니다. 반면 현대의 WAS+ORM 조합은 속도는 조금 느릴지 몰라도 언제든 부품을 갈아 끼우고 개조할 수 있는 '레고 블록 자동차'입니다. 변화의 속도가 빠른 현대에는 레고 블록이 정답입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 자정 결산(정산) 배치의 타임아웃 장애 개선

- **문제 상황**: 배달 플랫폼에서 매일 자정에 수백만 건의 주문을 읽어(WAS로 Fetch), 가맹점별 수수료를 계산(Loop)한 뒤 다시 DB에 업데이트(Update)하는 Java 기반의 Spring Batch 프로그램이 아침 6시까지 끝나지 않아 상점 주인들이 정산금을 받지 못함. (WAS-DB 간 네트워크 라운드트립 병목)
- **의사결정 (제한적 프로시저 도입)**:
  1. 객체 지향의 순수성을 일부 포기하더라도 비즈니스 연속성이 우선이므로, 수백만 건의 정산 로직을 Oracle **PL/SQL(저장 프로시저)**로 재작성함.
  2. Spring Batch는 단순히 야간에 프로시저를 호출(`CALL Calculate_Daily_Settle()`)하는 껍데기 역할(Job)만 수행.
  3. 모든 연산이 DB 디스크와 메모리(SGA) 안에서 다이렉트로 일어나 네트워크 비용이 0이 됨. (실행 시간 6시간 -> 10분 내 완료)

### 도입 체크리스트
- **벤더 락인 (Vendor Lock-in) 수용 여부**: 현재 회사가 값비싼 Oracle DB를 사용 중이다. 향후 오픈소스인 PostgreSQL이나 MySQL로 마이그레이션할 계획이 1%라도 있는가? 있다면 1만 줄짜리 PL/SQL 프로시저는 프로젝트를 무산시키는 최악의 족쇄가 되므로 절대 작성해선 안 된다.
- **스니핑 파라미터 (Parameter Sniffing) 문제**: 프로시저는 처음 실행될 때의 파라미터를 기준으로 실행 계획을 고정(캐싱)한다. 처음엔 '사장님(건수 작음)'으로 검색해 인덱스를 탔는데, 다음번엔 '일반회원(건수 수백만)'으로 검색하면 이미 고정된 인덱스 플랜을 그대로 타면서 풀 스캔보다 수십 배 느려지는 현상. 이를 방어할 RECOMPILE 옵션 등 튜닝 지식이 있는가?
- **형상 관리 (Version Control)**: 프로시저를 수정할 때 DBeaver나 Toad에서 직접 `CREATE OR REPLACE`를 치고 끝내는가? 반드시 `Flyway`나 `Liquibase` 같은 데이터베이스 마이그레이션 툴을 도입하여 스크립트를 Git 코드로 관리해야 한다.

### 안티패턴: 객체 지향 프레임워크(JPA) 환경에서 핵심 비즈니스 로직을 프로시저로 몰아넣기
- 최신 Spring Boot와 JPA를 도입해 놓고 정작 비즈니스 로직은 "DB가 빠르다"는 옛날 관성으로 수십 개의 프로시저에 다 박아둔 채, Java는 프로시저만 호출하는 깡통(Controller -> SP Call)으로 전락하는 경우. 테스트 주도 개발(TDD)이 불가능해지고, 에러 추적이 분절되며, 서버 Scale-out의 이점을 전혀 누리지 못하는 양 진영의 단점만 취한 최악의 설계다.

- **📢 섹션 요약 비유**: 최신식 스마트폰(JPA/MSA)을 사놓고 터치스크린 조작이 낯설다며 굳이 볼펜(저장 프로시저)으로 꾹꾹 눌러가며 쓰는 구세대적 아집을 경계해야 합니다. 기술은 철학과 패러다임이 일치할 때 써야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | WAS 기반 반복 연산 (Anti-pattern 시) | 저장 프로시저(SP) 오프로딩 | 개선 효과 |
|:---|:---|:---|:---|
| **배치 처리 성능** | 대량 데이터 왕복으로 네트워크 병목 | DB 내부 직접 연산 (네트워크 I/O = 0) | **대량 일괄 처리(Batch) 시간 수십 배 단축** |
| **데이터 보안** | DB 접속 계정이 테이블에 직접 노출 | SP 실행 권한만 부여하여 캡슐화 | **DBA 외의 직접적인 테이블 해킹 차단** |
| **유지보수 관점** | 소스 코드에 모든 로직 집중, 추적 용이 | 로직 파편화(Java + SP)로 디버깅 지옥 | (단점) **소프트웨어 공학적 유지보수성 극악 하락** |

### 미래 전망
- **클라우드 데이터베이스의 함수화 (UDF의 진화)**: 전통적인 독자 언어(PL/SQL)를 넘어서, 최근 Snowflake나 Google BigQuery는 클라우드 내부에 저장 프로시저(UDF)를 작성할 때 **Python이나 JavaScript**를 그대로 사용할 수 있게 허용하여 개발 친화성을 폭발적으로 끌어올리고 있다.
- **데이터 레이크하우스 중심의 전이**: 대용량 데이터를 처리하기 위해 기존 트랜잭션 RDBMS의 프로시저를 쓰던 관행에서 벗어나, 데이터는 S3에 두고 Spark나 DBT 같은 현대적 분산 엔진을 사용하여 SQL(ELT 방식)로 변환을 돌리는 방향으로 대형 배치의 주도권이 완전히 넘어가고 있다.

- **📢 섹션 요약 비유**: 저장 프로시저는 한때 모든 것을 호령하던 강력한 영주(전통적 아키텍처)였으나, 지금은 중앙 정부(클라우드와 MSA)에 권력을 내어주고 특수 임무(대규모 정산, 무거운 배치)에만 투입되는 정예 특수 부대로 그 역할이 변모하였습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **객체 관계 매핑(ORM)과 JPA**: 저장 프로시저의 'DB 중심' 철학과 완벽히 대척점에 있는, 현대의 '애플리케이션(객체) 중심' 데이터 조작 철학.
- **고급 SQL 튜닝 기법**: 프로시저 내부에서 발생하는 스니핑(Sniffing) 문제나 바인드 변수 처리 시 옵티마이저가 어떻게 실행 계획(Plan)을 고정하는지 이해하는 튜닝 기초.
- **데이터베이스 보안과 암호화 기술**: 동적 SQL의 취약점인 'SQL 인젝션(Injection)' 공격을 바인드 변수를 통해 방어하는 전사 보안 아키텍처 관점.
