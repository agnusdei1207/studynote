# 200. CAP 정리와 PACELC 정리 - 분산 데이터베이스의 근본적 한계와 트레이드오프

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CAP 정리는 분산 시스템에서 **일관성(Consistency)**, **가용성(Availability)**, **분할 내성(Partition Tolerance)**의 세 가지 속성 중 최대 두 가지만을 동시에 만족시킬 수 있다는 컴퓨터 공학의 절대 법칙이다.
> 2. **확장 (PACELC)**: 현실의 분산 시스템은 네트워크 분할(P) 상황뿐만 아니라 정상(E) 상태에서도 동작해야 하므로, PACELC 정리는 정상 상태에서 **지연 시간(Latency)**과 **일관성(Consistency)** 중 무엇을 선택할지에 대한 추가적인 트레이드오프를 제시한다.
> 3. **가치**: NoSQL과 분산 데이터베이스 아키텍처를 설계하거나 솔루션(Cassandra, MongoDB, DynamoDB 등)을 선택할 때, 비즈니스 요구사항에 맞는 타협점을 찾기 위한 가장 중요한 **나침반 역할**을 한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
**CAP 정리 (CAP Theorem)**는 2000년 에릭 브루어(Eric Brewer)가 제안한 이론으로, 분산 데이터 저장소는 일관성(C), 가용성(A), 분할 내성(P) 이 세 가지 특성을 동시에 완벽하게 충족할 수 없다는 이론입니다. 이후 2010년 다니엘 아바디(Daniel Abadi)가 이를 확장하여 정상 상태에서의 트레이드오프까지 포함한 **PACELC 정리**를 발표했습니다.

### 💡 비유: 맛집의 철학
식당을 운영할 때 "최고의 맛(일관성)", "24시간 365일 대기 없는 영업(가용성)", "전국 체인점 동시 오픈(분할 내성)"을 동시에 달성할 수는 없습니다. 체인점(분할)을 늘리면서 맛(일관성)을 100% 똑같이 유지하려면 주방장 점검 때문에 영업을 자주 쉬어야(가용성 하락) 하고, 24시간 문을 열면(가용성 보장) 지점마다 맛이 조금씩 달라질 수밖에 없는(일관성 하락) 이치와 같습니다.

### 2. 등장 배경
- **RDBMS의 스케일 아웃(Scale-out) 한계**: 기존의 단일 노드 RDBMS는 일관성(C)과 가용성(A)을 보장했지만, 인터넷 시대의 폭발적인 트래픽을 처리하기 위해 데이터를 여러 서버로 쪼개는 분산 환경(P)이 필수적이 되었습니다.
- **NoSQL의 탄생**: 완벽한 정합성(ACID)을 고집하다가는 글로벌 서비스를 만들 수 없다는 것을 깨달은 기업들(구글, 아마존 등)이, 일관성을 조금 희생하더라도 언제든 응답할 수 있는(Eventual Consistency) NoSQL을 개발하게 된 이론적 배경이 되었습니다.

- **📢 섹션 요약 비유**: CAP 정리는 분산 시스템이라는 우주에서 절대로 넘을 수 없는 '빛의 속도(물리적 제약)'와 같은 법칙입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. CAP 정리의 3요소

| 요소 | 정의 | 분산 DB 관점의 의미 |
|:---|:---|:---|
| **C (Consistency, 일관성)** | 모든 노드가 동일한 순간에 동일한 데이터를 보여줌 | 노드 A에 데이터를 쓰고 바로 노드 B에서 읽어도 최신 데이터가 반환되어야 함. |
| **A (Availability, 가용성)** | 모든 요청에 대해 (성공/실패 여부와 무관하게) 정상적인 응답을 줌 | 일부 노드가 죽거나 통신이 안 돼도 살아있는 노드는 무조건 사용자에게 응답을 줘야 함. |
| **P (Partition Tolerance, 분할 내성)** | 노드 간 네트워크 통신이 단절(Partition)되더라도 시스템은 계속 동작함 | 클라우드나 분산 환경에서는 네트워크 단절이 필연적이므로, P는 선택이 아닌 **필수 조건**임. |

### 2. 네트워크 분할(Partition) 시나리오와 시스템의 선택

클라우드/분산 환경에서는 네트워크 장애(P)가 언제든 일어날 수 있으므로, 설계자는 **CP 시스템**과 **AP 시스템** 중 하나를 선택해야 합니다. (CA 시스템은 네트워크 분할이 없는 단일 노드 RDBMS에서나 가능합니다.)

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                 [네트워크 분할(Partition) 발생 시 CP vs AP의 동작]                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  상황: 노드 A와 노드 B 간의 네트워크 케이블이 끊어짐 (Partition 발생)                 │
│                                                                             │
│  1. [CP 시스템] (일관성 선택, 가용성 포기)                                          │
│     - 사용자 1이 노드 A에 "X=2"로 업데이트 요청.                                  │
│     - 노드 A는 노드 B로 데이터를 동기화할 수 없음을 깨달음.                           │
│     - 결과: 노드 A는 "지금은 일관성을 맞출 수 없어 에러 반환!" 하며 쓰기 거부.           │
│     - (일관성은 지켜졌지만, 가용성은 실패함)                                       │
│                                                                             │
│  2. [AP 시스템] (가용성 선택, 일관성 포기)                                          │
│     - 사용자 1이 노드 A에 "X=2"로 업데이트 요청.                                  │
│     - 노드 A는 노드 B와 통신이 안 되지만 일단 "OK! 저장 완료" 응답함.                 │
│     - 사용자 2가 노드 B에서 X를 조회하면 이전 데이터인 "X=1"이 반환됨.                 │
│     - 결과: 양쪽 다 응답은 했으나, 서로 다른 데이터를 보여줌.                        │
│     - (가용성은 지켜졌지만, 일관성은 깨짐)                                        │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. PACELC 정리 (CAP의 현실적 확장)

CAP 정리는 "네트워크가 단절(Partition)되었을 때"의 극단적 상황만 설명합니다. 그러나 네트워크가 정상일 때도 시스템은 고민해야 합니다. 이를 수식화한 것이 **PACELC**입니다.

- **P (Partition 발생 시)**: **A**vailability(가용성) vs **C**onsistency(일관성) 중 선택
- **E (Else, 정상 상태 시)**: **L**atency(응답 지연시간) vs **C**onsistency(일관성) 중 선택

즉, 네트워크가 정상이어도 **"모든 노드에 완벽히 동기화(C)하고 응답할래?(속도 느림)"** 아니면 **"일단 응답(L)부터 빨리하고 동기화는 나중에 할래?(데이터 불일치 가능성)"**의 트레이드오프가 존재합니다.

- **📢 섹션 요약 비유**: PACELC는 "폭우가 내릴 때(P) 우산을 쓸래(C) 비옷을 입을래(A)?" 뿐만 아니라, "날씨가 맑을 때(E)도 더울까봐 얇게 입을래(L) 아니면 추울까봐 껴입을래(C)?"까지 묻는 더 정교한 질문입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 데이터베이스별 CAP & PACELC 포지셔닝

| 아키텍처 | 데이터베이스 | CAP 분류 | PACELC 분류 | 철학 (작동 방식) |
|:---|:---|:---:|:---:|:---|
| **전통적 RDBMS** | Oracle, MySQL | **CA** | **PC/EC** | 분할(P)을 고려하지 않음. 항상 강한 정합성을 유지하며, 이를 위해 속도(L)를 희생함. |
| **문서 / 키-값 DB** | MongoDB, HBase | **CP** | **PA/EC** | 단절 시 읽기/쓰기를 차단하여 정합성을 지킴. 마스터 노드 중심 설계. |
| **와이드 컬럼 DB** | Cassandra, DynamoDB | **AP** | **PA/EL** | 단절 시에도 무조건 쓰기를 받아줌(AP). 정상 시에도 속도(L)를 최우선하여 비동기 복제함. |
| **NewSQL** | Spanner, CockroachDB| **CP** | **PC/EC** | TrueTime 등 하드웨어 시계를 동원해 분산 환경에서도 100% C를 달성하려 노력함. |

### 2. 가용성을 택한 AP 시스템의 결과적 일관성 (Eventual Consistency)
AP 시스템은 당장의 불일치를 허용하지만 영원히 불일치하게 두지는 않습니다. 단절된 네트워크가 복구되면 백그라운드에서 데이터를 맞춥니다.
- **가십 프로토콜(Gossip Protocol)**: 노드들끼리 소문을 퍼뜨리듯 데이터를 동기화 (Cassandra).
- **벡터 시계 (Vector Clock)**: 데이터의 업데이트 순서와 출처를 버전으로 기록하여 충돌(Conflict) 발생 시 클라이언트에게 해결을 위임함.

- **📢 섹션 요약 비유**: CP 시스템은 결재 서류에 이사님 도장까지 완벽하게 찍혀야 통과되는 깐깐한 관료제이고, AP 시스템은 일단 구두로 승인하고 나중에 서류를 끼워 맞추는 유연한 스타트업과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 비즈니스 도메인별 실무 의사결정 (CP vs AP)

1. **시나리오: 페이스북(Facebook)의 "좋아요" 카운트와 피드**
   - **도메인 특성**: 내가 좋아요를 누른 글이 친구 화면에 0.5초 늦게 반영되어도 치명적이지 않습니다. 반면, 서버에 접속이 안 돼서 빈 화면이 뜨는 것은 이탈률을 높입니다.
   - **아키텍처 결정**: **AP / PA/EL (가용성, 지연시간 최우선)**
   - **사용 DB**: Cassandra, DynamoDB 등을 사용하여 무조건 응답부터 하고 백그라운드 복제.

2. **시나리오: 은행의 ATM 잔액 조회 및 이체**
   - **도메인 특성**: 통신 장애가 발생했다고 해서 내 잔액이 100만 원인데 200만 원으로 조회되거나 출금되면 회사에 막대한 손실이 발생합니다.
   - **아키텍처 결정**: **CP / PC/EC (일관성 최우선)**
   - **사용 DB**: 분산 환경이라면 MongoDB(과반수 쓰기)나 분산 트랜잭션이 완벽히 지원되는 RDBMS/NewSQL을 사용하여, 네트워크 이상 시 차라리 "시스템 점검 중" 에러를 내보내어 가용성을 포기함.

### 2. 튜너블 컨시스턴시 (Tunable Consistency)의 도입
현대의 NoSQL은 아키텍처가 AP로 고정되어 있지 않고, 실무자가 쿼리 단위로 C와 A의 비중을 조절할 수 있습니다.
- Cassandra의 **Quorum (정족수) 읽기/쓰기**: 복제본(Replica)이 3개일 때, `Write=2, Read=2` (총합 > 복제본 수) 로 설정하면 항상 최신 데이터를 읽는 강한 일관성(C)을 얻을 수 있고, `Write=1, Read=1` 로 설정하면 속도(L)와 가용성(A)을 극대화할 수 있습니다.

### 3. 안티패턴
- **비즈니스 분석 없이 유행 따라 NoSQL 도입**: 1원 단위의 정산이 맞아야 하는 결제 도메인에 수평 확장성(Scale-out)이 좋다는 이유만으로 Cassandra(AP)를 도입하면, 끊임없이 발생하는 결과적 일관성 충돌을 애플리케이션 레벨에서 맞추느라 개발 비용이 폭증합니다.

- **📢 섹션 요약 비유**: 스포츠카(AP)를 사서 오프로드 짐수레(결제 도메인)로 쓰면 차가 망가지고, 트랙터(CP)를 사서 레이싱 트랙(SNS 피드)에 올리면 경주에서 집니다. 용도에 맞는 엔진 선택이 핵심입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과

| 구분 | 도메인 불일치 (결제에 AP 적용) | 아키텍처 최적화 (결제 CP, 피드 AP 분리) | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 트래픽 폭주 시 결제 오차율 0.1% 발생 | 트래픽 무관 데이터 무결성 보장 | 금전적 사고율 **0% 달성** |
| **정량** | RDBMS 스케일업 비용 수억 원 | 무거운 읽기는 분산 NoSQL로 오프로딩 | 인프라 비용 **수십 분의 일로 절감** |
| **정성** | 야간 장애 시 수동 데이터 정합성 보정 | 튜너블 컨시스턴시를 통한 유연한 운영 | 시스템 확장성과 신뢰성의 밸런스 확보 |

### 2. 결론
CAP 정리와 PACELC 정리는 엔지니어에게 "세상에 공짜 점심은 없다"는 냉혹한 진리를 일깨워 줍니다. 무한대의 확장성을 원한다면 필연적으로 네트워크 분할(P) 상황을 마주하게 되고, 이때 서비스의 가용성(A)을 지킬 것인지 데이터의 완벽한 일관성(C)을 지킬 것인지 결정해야 합니다. 시스템 아키텍트는 하나의 만능 데이터베이스를 찾는 대신, **폴리글랏 퍼시스턴스(Polyglot Persistence)** 전략을 통해 도메인의 성격에 맞춰 CP 시스템과 AP 시스템을 조화롭게 배치하는 설계 역량을 갖추어야 합니다.

- **📢 섹션 요약 비유**: CAP와 PACELC는 완벽함을 포기하고 비즈니스의 현실적인 '최선'을 선택하게 해주는 타협의 저울입니다.

+++

## 📌 관련 개념 맵 (Knowledge Graph)
- **NoSQL 아키텍처**: CAP 정리가 낳은 결과물로, Key-Value, Document, Column-Family 등으로 나뉨.
- **분산 트랜잭션 (2PC / Saga)**: CP 시스템을 보장하기 위한 강압적 방법(2PC)과 AP 시스템을 보완하기 위한 비동기 방법(Saga).
- **NewSQL (Spanner)**: 분산 환경에서도 RDBMS의 트랜잭션(C)을 보장하기 위해 구글이 하드웨어 원자시계(TrueTime)를 동원해 만든 차세대 DBMS.

2. 인터넷 시스템(분산 DB)은 언제든 인터넷이 끊길 수 있어서 "잠(P)"은 무조건 자야 해요. 그래서 "공부(C)"를 선택할지 "놀기(A)"를 선택할지 정해야 한답니다.
3. 돈 계산처럼 한 치도 틀리면 안 되는 곳은 **공부(CP)**를 선택하고, 유튜브나 인스타처럼 끊기지 않고 계속 나오는 게 중요한 곳은 **놀기(AP)**를 선택해요!
