# NUMA (Non-Uniform Memory Access)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NUMA(불균일 메모리 접근)는 수십~수백 개의 코어가 단 하나의 메모리 버스에 몰리는 병목 현상을 막기 위해, **메인 메모리를 물리적으로 여러 조각으로 쪼개어 각 프로세서(노드) 옆에 분산 배치한 아키텍처**다.
> 2. **가치**: 소프트웨어(OS) 입장에서는 여전히 하나의 거대한 공유 메모리처럼 보이지만, 하드웨어적으로는 '내 옆에 있는 로컬 메모리'와 '남의 옆에 있는 원격 메모리' 간에 **접근 속도(Latency) 차이가 발생**하는 불균일성을 특징으로 한다.
> 3. **융합**: 멀티프로세서의 프로그래밍 편의성(Shared Memory)과 다중 컴퓨터의 무한한 확장성(Scalability)이라는 두 마리 토끼를 잡기 위해 융합된 궁극의 절충안으로, **현대 x86 및 ARM 기반 대형 서버 시장의 100% 절대 표준**이다.

+++

## Ⅰ. NUMA의 개요

### 1. 정의
- 다중 프로세서(MIMD) 환경에서 각 프로세서가 자신만의 로컬 메모리(Local Memory)를 가지며, 프로세서 간 상호 연결망(인터커넥트)을 통해 다른 프로세서의 원격 메모리(Remote Memory)에도 접근할 수 있는 물리적 분산-논리적 공유 메모리 구조다.

### 2. 등장 배경: 'UMA 버스의 처참한 붕괴'
- 코어가 8개일 때까지는 UMA(균일 메모리) 방식이 편했다. 하지만 코어가 64개로 늘어나자, 64명의 깡패가 하나의 메모리 버스와 하나의 메모리 컨트롤러를 잡겠다고 싸우면서 시스템이 얼어붙었다(Bus Contention). 아키텍트들은 "메모리를 64등분 해서 코어 하나당 메모리 하나씩 쥐여주자. 단, 프로그래머들이 귀찮아할 테니 OS 단에서는 몰래 1개의 램처럼 뻥을 쳐두자!"라고 결정하며 NUMA를 창조했다.

### 3. 💡 비유: '대형 뷔페의 분산된 음식 코너'
- UMA가 **'식당 한가운데에 밥솥을 하나만 둔 구조'**라면, NUMA는 **'100명이 앉은 연회장 테이블 구역마다 밥솥을 하나씩 놓아둔 구조'**다.
- 내 자리에 있는 밥솥(로컬 메모리)에서 밥을 풀 때는 1초면 되지만, 내 밥솥이 비어서 저 멀리 4번 구역 밥솥(원격 메모리)까지 걸어가서 밥을 푸려면 10초가 걸린다. 밥을 푸는 '시간(Access Time)'은 불균일(Non-Uniform)하지만, 어쨌든 식당 안의 모든 밥은 누구나 다 먹을 수 있다(공유 메모리).

- **📢 섹션 요약 비유**: 중앙 집중식 배급소의 미어터지는 줄을 없애기 위해 동네마다 배급소를 짓되, 재고가 모자라면 다른 동네 배급소에서도 물건을 타 갈 수 있게 허용한 유연한 물류망입니다.

## Ⅱ. 메커니즘 및 메모리 노드 (Deep Dive)

NUMA는 '노드(Node)'라는 개념을 중심으로 움직인다.

### 1. NUMA 노드 (Node)의 구조
- CPU 코어 1개(또는 코어 묶음) + 그 코어에 직결된 메모리 컨트롤러 + 직접 꽂혀있는 메인 메모리(RAM). 이 세트를 **1개의 NUMA 노드**라고 부른다.
- 노드와 노드 사이는 인텔의 QPI/UPI, AMD의 하이퍼트랜스포트(Infinity Fabric) 같은 **초고속 직렬 점대점 인터커넥트**로 연결된다.

### 2. 메모리 접근 속도의 불균일성 시각화 (ASCII)
```text
      [ Node 0 ]                            [ Node 1 ]
  ┌────────┬────────┐                   ┌────────┬────────┐
  │ CPU 0  │ RAM 0  │                   │ CPU 1  │ RAM 1  │
  └────────┴────────┘                   └────────┴────────┘
       ▲        ▲                             ▲        ▲
       │(매우 빠름)                            │(매우 빠름)
       └────────────────(조금 느림)────────────┘
```
- **Local Access (로컬 접근)**: CPU 0이 자기 옆에 있는 RAM 0을 읽는 것. 병목이 전혀 없어 빛처럼 빠르다 (예: 50ns).
- **Remote Access (원격 접근)**: CPU 0이 칩 외부의 UPI 버스를 타고 건너가 CPU 1 옆에 달린 RAM 1을 읽는 것. 길을 건너야 하므로 필연적으로 지연이 발생한다 (예: 100ns).

### 3. 심층 분석: ccNUMA (Cache-Coherent NUMA)
- 메모리가 쪼개져 있어도 결국 공유 메모리이기 때문에, 캐시 불일치 문제가 발생한다. 현대의 NUMA는 100% **ccNUMA** 구조다. 디렉토리 기반 프로토콜(Directory-based)을 사용하여, 노드 0의 캐시가 노드 1의 메모리 값을 고치면 전 시스템에 이 사실을 안전하고 렉 없이 전파(Snoop/Directory update)한다.

- **📢 섹션 요약 비유**: 각자 자기 방(노드)에 비밀 장부를 두고 쓰지만, 남의 장부를 빌려 쓸 때마다 중앙 상황판(디렉토리)에 반드시 신고를 남겨 회계 오류를 막는 투명한 분산 사무실입니다.

## Ⅲ. NUMA vs UMA 스펙 비교

| 비교 항목 | UMA (Uniform Memory Access) | NUMA (Non-Uniform Memory Access) |
|:---|:---:|:---:|
| **물리적 메모리 구조**| 단일 덩어리 | **물리적으로 쪼개져 CPU 곁에 분산** |
| **메모리 접근 시간** | 모든 주소가 **동일함** | 로컬은 빠르고, **원격은 느림** |
| **버스 병목 현상** | 코어가 늘면 심각하게 발생 | **거의 없음 (대역폭이 선형적으로 늘어남)** |
| **운영체제 스케줄러**| 아무 코어나 배정해도 속도 동일 | **코어와 램의 거리를 무조건 계산해서 배정해야 함** |
| **코어 확장성** | 최대 16~32 코어 언저리에서 한계 | **256 코어 이상 끝없이 확장 가능** |

- **📢 섹션 요약 비유**: 좁은 집에 살 때는 화장실이 1개라도 참을 만하지만(UMA), 100평짜리 대저택에 살 때는 방마다 화장실을 따로 뚫어야만(NUMA) 삶의 질이 유지됩니다.

+++

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: NUMA 팩터 (NUMA Factor) 튜닝
- **상황**: DB 서버(MySQL 등)를 64코어 2소켓(NUMA 노드 2개) 서버에 올렸는데 성능이 널뛰기를 한다.
- **원인**: 프로세스는 CPU 0(Node 0)에서 도는데, 이 프로세스가 쓰는 메모리는 OS가 바보같이 Node 1의 램에 할당해 버린 것이다. CPU 0은 연산을 할 때마다 느려터진 QPI 버스를 타고 Node 1의 램을 퍼오느라 성능이 반토막 났다.
- **판단 (해결책)**: 시니어 서버 엔지니어는 운영체제에 **NUMA Affinity(친화도)** 정책을 강제한다. 리눅스의 `numactl` 명령어나 DB 설정 파일에서 `innodb_numa_interleave` 옵션을 켜서, "CPU 0이 계산할 데이터는 무조건 CPU 0 옆의 로컬 램에만 저장해라!"라고 물리적 위치를 못 박아버린다. 하드웨어의 불평등을 이해한 이 소프트웨어 튜닝 하나로 DB 쿼리 속도가 30% 이상 폭증한다.

### 2. 기술사적 인사이트: 퍼스트 터치(First Touch) 정책
- **인사이트**: 리눅스의 기본 NUMA 메모리 할당 정책은 'First Touch'다. 어떤 변수를 `malloc`으로 선언할 때는 메모리 위치를 안 정하다가, CPU 0과 CPU 1 중 **그 변수에 처음으로 값을 쓰는(Touch) 코어가 있는 노드의 로컬 램에 메모리를 할당**해 버린다. 이는 대개 효율적이지만, 멀티스레딩 프로그래밍 시 메인 스레드 하나가 초기화(First Touch)를 다 해버리면 램이 특정 노드에 100% 몰빵되어 NUMA의 장점이 박살 나는 참사가 벌어진다. 시스템 아키텍트는 튜닝을 통해 페이지를 라운드 로빈으로 골고루 찢어 바르는(Interleave) 전략을 취하기도 한다.

- **📢 섹션 요약 비유**: 집을 지을 때 건축가(프로그래머)가 인부들(스레드)의 동선을 고려하지 않고 화장실을 한쪽에만 몰아 지어버리면, 인부들이 일은 안 하고 화장실 오가는 데 체력을 다 쓰는 끔찍한 사태가 벌어집니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **메모리 월(Memory Wall) 돌파**: 코어 수를 2배로 늘릴 때 메모리 컨트롤러도 2배로 늘려 대역폭을 2배로 뻥튀기시킴으로써, 대형 서버의 성능 스케일업(Scale-up) 한계를 물리적으로 부수었다.
- **서버 폼팩터의 지배**: 현재 세상에 존재하는 모든 2소켓 이상의 서버(Xeon, EPYC)와 거대 칩렛(AMD Threadripper) 모델은 예외 없이 100% NUMA 아키텍처로 구동되고 있다.

### 2. 미래 전망
최근 AMD의 칩렛(Chiplet) 구조는 하나의 CPU 칩 안에도 여러 개의 코어 덩어리(CCD)를 엮어 놓아, 단일 소켓의 데스크톱 CPU조차 내부적으로는 NUMA의 성격을 띠기 시작했다. 앞으로 코어가 수백 개로 조각날수록 로컬과 리모트의 경계는 더욱 복잡해질 것이다. 개발자가 코딩할 때 하드웨어 구조를 무시해도 되던 '공유 메모리의 낭만 시대'는 끝났다. 미래의 하이엔드 소프트웨어 개발자는 내가 선언한 변수가 실리콘 칩의 어느 쪽 구석에 물리적으로 처박히는지까지 고민해야 하는 극한의 하드웨어 친화적(Hardware-aware) 시대를 맞이하고 있다.

- **📢 섹션 요약 비유**: 세상이 너무 커져서 "우리는 모두 하나"라는 환상(UMA)은 깨졌습니다. 이제는 내가 서 있는 지역의 특성과 거리를 냉정하게 계산(NUMA)해야만 이 거대한 디지털 우주에서 살아남을 수 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **UMA (Uniform Memory Access)**: NUMA가 태어나기 전까지 세상을 지배했던 평등하고 따뜻했지만 느려터진 구형 아키텍처.
- **공유 메모리 시스템 (Shared Memory)**: NUMA가 불균일한 속도라는 단점을 감수하면서까지 포기할 수 없었던 코딩의 편리함, 그 정체성.
- **디렉토리 프로토콜 (Directory-based)**: 수십 개의 쪼개진 NUMA 노드들 사이에서 캐시 데이터가 꼬이지 않도록 중앙 장부를 적는 치밀한 동기화 사서.

## 👶 어린이를 위한 3줄 비유 설명
1. NUMA는 교실을 아주 넓게 합쳐서 100명의 학생이 쓰는데, **자기 책상 바로 옆에 전용 간식 바구니(로컬 메모리)를 따로 달아준 것**이에요.
2. 내 바구니에서 과자를 꺼내 먹을 땐 1초 만에 먹을 수 있어서 너무 빠르지만, 만약 짝꿍의 바구니에 있는 과자를 뺏어 먹으려면(원격 메모리) 자리에서 일어나야 하니까 시간이 좀 걸리죠.
3. 하지만 모두가 칠판 앞 하나의 큰 바구니(UMA)로 몰려가서 싸우는 것보다는 백만 배 빨리 쾌적하게 과자를 먹을 수 있는 최고의 교실 배치랍니다!
