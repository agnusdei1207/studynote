# 스레드 레벨 병렬성 (TLP, Thread-Level Parallelism)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스레드 레벨 병렬성(TLP)은 단일 프로그램의 명령어들을 하드웨어가 억지로 쥐어짜 내어 병렬로 돌리는 ILP(명령어 수준 병렬성)의 한계를 인정하고, 아예 프로그래머나 OS가 **논리적으로 독립된 큰 덩어리인 '스레드(Thread)'를 여러 개 만들어 동시에 실행하는 패러다임**이다.
> 2. **가치**: 클럭 속도나 파이프라인 깊이를 늘리는 대신, CPU 코어 개수를 가로로 늘리는(멀티코어) 방식과 맞물려, 서버와 PC의 성능 한계인 **전력 장벽(Power Wall)을 회피하며 무한한 처리량(Throughput)의 성장을 견인**했다.
> 3. **융합**: 물리적인 코어를 여러 개 박는 CMP(Chip Multi-Processor) 기술과, 하나의 코어에 두 개의 스레드를 욱여넣는 **동시 멀티스레딩(SMT, 하이퍼스레딩)** 기술이 융합되어 현대 CPU 아키텍처의 절대적인 성능 지표가 되었다.

---

## Ⅰ. 스레드 레벨 병렬성(TLP)의 개요

### 1. 정의
- 프로그램 실행의 논리적 흐름 단위인 '스레드(Thread)' 여러 개를, 운영체제의 통제하에 다중 프로세서(또는 멀티코어)에서 물리적/논리적으로 동시에 병렬 처리하여 시스템 전체의 작업 효율을 높이는 방식이다.

### 2. 등장 배경: '천재 한 명(ILP)의 과로사, 그리고 협업의 시작'
- 2000년대 중반까지 CPU 제조사들은 1개의 코어 안에서 명령어 여러 개를 동시에 구겨 넣는 ILP(슈퍼스칼라, 비순차 실행)에 몰빵했다. 하지만 ILP를 2배 올리려면 트랜지스터와 전력 소모가 4배로 뛰는 지독한 비효율(ILP Wall)에 직면했고, 칩이 불타올랐다. 아키텍트들은 항복을 선언했다. "코어 하나를 똑똑하게 만드는 건 포기! 그냥 적당히 똑똑한 코어를 2개, 4개 박아놓을 테니 프로그래머들이 알아서 스레드 찢어서 돌려라!" 하드웨어의 짐을 소프트웨어(프로그래머)에게 당당히 넘기며 TLP 시대가 개막했다.

### 3. 💡 비유: '분업화된 대규모 공장'
- ILP가 **'양손양발을 다 쓰며 혼자 미친 듯이 일하는 1명의 달인'**이라면, TLP는 **'각자 독립된 작업대에서 자기 할 일만 묵묵히 쳐내는 4명의 평범한 노동자'**다.
- 달인 1명을 더 빨리 일하게 채찍질(클럭 상승)하는 데는 한계가 있지만, 공장 공간만 넓혀서 평범한 노동자를 4명, 8명, 16명으로 계속 고용하면 공장 전체의 생산량(TLP)은 계속 늘어난다.

- **📢 섹션 요약 비유**: 슈퍼맨 1명(단일 코어 ILP)의 신체적 스펙에 의존하는 영웅물에서, 평범한 병사 100명(멀티코어 TLP)이 협력해서 전쟁을 이기는 군원물로 컴퓨팅 역사의 장르가 바뀐 것입니다.

---

## Ⅱ. 메커니즘 및 3대 아키텍처 구현 (Deep Dive)

TLP는 단순히 코어를 여러 개 박는 것 외에도 다양한 꼼수로 구현된다.

### 1. 칩 다중 프로세서 (CMP: Chip Multi-Processor)
- 가장 정직한 TLP 구현 방식. 실리콘 다이(Die) 하나에 완벽히 독립된 물리적 CPU 코어(L1 캐시, ALU 포함)를 2개 이상 박아 넣는 진짜 **'멀티코어'**다. 물리적으로 뇌가 여러 개이므로 완벽한 병렬 실행이 보장된다.

### 2. 세밀한/거친 멀티스레딩 (Fine/Coarse-grained Multithreading)
- 물리적 코어는 1개다. 그런데 A 스레드가 메모리에서 데이터를 가져오느라 100클럭을 멈춘다(캐시 미스).
- 이때 코어가 놀지 않고, 즉각 스위칭(Context Switch)하여 대기 중이던 B 스레드의 명령어를 실행한다. 
  - **Coarse-grained**: 스톨(Stall) 같은 큰 병목이 터졌을 때만 큼직하게 스레드를 교체.
  - **Fine-grained**: 아예 매 클럭 사이클마다 1번은 A 스레드, 다음 1번은 B 스레드를 번갈아 가며 실행.

### 3. 동시 멀티스레딩 (SMT: Simultaneous Multithreading) - **끝판왕**
- 인텔에서 **하이퍼스레딩(Hyper-Threading)**이라 부르는 마법. 코어는 1개지만, OS에게는 "나 코어 2개야!"라고 사기를 친다(논리 코어).
- 코어 안의 실행 유닛(ALU)이 6개(슈퍼스칼라)인데, A 스레드가 4개만 쓰고 2개가 놀고 있다면? 같은 클럭 사이클(동시에)에 노는 2개의 ALU에 B 스레드의 명령어를 쑤셔 넣는다. 즉, **단일 클럭 안에서 두 개의 스레드가 하드웨어 자원을 나눠 쓰며 100% 가동률을 짜내는 TLP와 ILP의 궁극적 융합체**다.

- **📢 섹션 요약 비유**: CMP가 '직원 2명을 고용한 것'이라면, SMT는 '직원 1명이 왼손으론 A 부서 서류를 쓰고 오른손으론 B 부서 서류를 도장 찍게 만드는 악덕(최고 효율) 사장의 용인술'입니다.

## Ⅲ. ILP (명령어) vs TLP (스레드) 비교

| 비교 항목 | ILP (Instruction-Level Parallelism) | TLP (Thread-Level Parallelism) |
|:---|:---|:---|
| **병렬화 대상** | 1개의 프로그램 안의 **개별 명령어들** | 완전히 독립된 **여러 개의 프로그램/함수** |
| **추출 주체** | **CPU 하드웨어 (디코더, OoO 엔진)** | **운영체제 스케줄러 & 프로그래머 (코드)** |
| **개발자 난이도** | 전혀 신경 안 써도 됨 (공짜 성능 향상) | **극악. 직접 스레드를 쪼개고 Lock을 걸어야 함** |
| **발전 한계** | 거의 한계 도달 (IPC 4~8 언저리) | 코어를 늘리는 만큼 무한히 증가 중 (수십~수백 코어) |

- **📢 섹션 요약 비유**: ILP가 하드웨어가 밥상을 다 차려서 입에 떠먹여 주는 '오마카세'라면, TLP는 재료와 조리 기구만 주고 프로그래머가 직접 요리해 먹어야 하는 '셀프 바비큐' 시스템입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: 멀티스레드 코딩과 암달의 법칙(Amdahl's Law)
- **상황**: 하드웨어가 64코어 TLP를 지원한다. 개발자는 이 64코어를 다 쓰기 위해 프로그램을 64개의 스레드로 쪼갰다.
- **판단**: TLP의 가장 큰 적은 동기화(Synchronization)다. 64개의 스레드가 1개의 공유 자원(데이터베이스 커넥션 등)을 건드릴 때마다 락(Lock)을 건다. 락이 걸린 구간에서는 63개의 스레드가 멈추고 1개만 실행된다(직렬화 구간). 암달의 법칙에 따르면, 프로그램에 직렬화 구간이 단 5%만 존재해도, 코어를 무한대로 늘려봤자 전체 성능은 20배(1/0.05) 이상 절대 빨라지지 않는다. 따라서 기술사는 단순히 스레드 개수만 늘리는 멍청한 짓을 멈추고, 스레드 간 락을 최소화하는 **락 프리(Lock-free) 알고리즘**을 짜는 데 모든 아키텍처 역량을 쏟아야 TLP의 진정한 스피드를 뽑아낼 수 있다.

### 2. 기술사적 인사이트: SMT(하이퍼스레딩)의 양날의 검
- **인사이트**: SMT를 켜면 OS는 코어가 2배가 된 줄 알고 스레드를 때려 넣는다. 평소에는 남는 ALU를 알뜰하게 써서 성능이 20~30% 오른다. 하지만 캐시 메모리(L1/L2)는 코어 1개 분량 그대로인데, 스레드 2개가 들어와서 서로 자기 데이터를 넣겠다고 싸우면서 **캐시 스래싱(Cache Thrashing)**이 터진다. 고성능 게임(단일 코어 캐시가 몹시 중요한 작업)을 할 때 컴덕들이 BIOS에 들어가서 일부러 하이퍼스레딩을 꺼버리는(Disable) 이유가 바로 이것이다. TLP는 항상 메모리 대역폭의 희생을 댓가로 지불한다.

- **📢 섹션 요약 비유**: 고속도로 차선(캐시)은 그대로인데, 억지로 소형차 두 대(스레드 두 개)를 한 차선에 욱여넣고 달리게 하면 평소엔 많이 가지만 조금만 차가 커져도 쾅 부딪혀 사고가 나는 딜레마입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **무어의 법칙 수명 연장**: 반도체 미세화로 늘어난 트랜지스터를 거대한 복잡성(ILP)에 낭비하지 않고, 똑같은 코어를 복사(Copy)하는 간단한 방법으로 활용해 수십 년간 칩 성능의 우상향을 보장.
- **서버/클라우드 시대의 지배자**: 수만 명의 사용자 접속을 각각의 독립된 스레드로 맵핑하여 렉 없이 동시 처리하는 최신 웹 서버(Nginx, Node.js, 멀티스레드 자바) 아키텍처의 가장 완벽한 물리적 토대.

### 2. 미래 전망
코어 개수가 128개, 256개로 늘어나며 TLP는 정점을 찍고 있다. 하지만 무작정 코어만 늘린다고 성능이 오르지 않는 병목(메모리 월) 때문에, 최근의 발전은 CPU 내부에 똑같은 코어를 늘리는 것에서 벗어났다. CPU, GPU, NPU 등 각기 잘하는 특기가 다른 코어들을 섞어 쓰는 **이기종(Heterogeneous) 멀티코어 TLP**로 진화했다. 프로그래머에게는 이 복잡한 이기종 코어들을 어떻게 배분하고 스레드를 태울 것인가 하는 지옥 같은 스케줄링 과제가 새로운 세대의 도전 과제로 던져졌다.

- **📢 섹션 요약 비유**: 단순히 삽질하는 일꾼 100명을 고용하던 노가다 시대(동종 멀티코어)를 넘어, 포클레인 기사, 설계사, 목수 등 전문 분야가 다른 100명을 지휘해 마천루를 짓는 초고도화된 종합 건설 현장의 시대가 열렸습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 스레드 레벨 병렬성(TLP)은 숙제를 할 때 **'나 혼자 수학, 영어, 국어를 번갈아 가며 푸는 게 아니라, 친구 3명을 불러서 각자 한 과목씩 동시에 풀어버리는 마법'**이에요.
2. 내 손이 아무리 빨라져도 한계가 있으니(ILP의 한계), 아예 똑똑한 친구(코어)들을 내 방에 여러 명 모아서 일을 맡기는 거죠.
3. 대신 친구들이 같은 지우개(공유 메모리)를 쓰겠다고 싸우지 않게 내가 옆에서 순서를 잘 정해줘야(프로그래머의 역할) 숙제가 엉망이 되지 않는답니다!
