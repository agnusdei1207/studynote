# MIMD (Multiple Instruction stream, Multiple Data stream)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MIMD(다중 명령어 다중 데이터)는 여러 개의 독립적인 프로세서(Core)가 **각자 자신만의 명령어(Instruction)를 읽어와 각기 다른 데이터(Data)를 동시에 씹고 뜯고 맛보는 완전한 자율적 병렬 처리 아키텍처**다.
> 2. **가치**: 한 명의 천재가 일하는 속도의 한계(클럭의 덫)를 돌파하기 위해 평범한 일꾼 10명을 고용하는 방식으로 전환하여, 현대의 듀얼코어 스마트폰부터 수만 개의 CPU를 엮은 클라우드 데이터센터까지 **오늘날 거의 모든 범용 컴퓨팅의 절대 표준**으로 등극했다.
> 3. **융합**: 독립된 코어들이 서로 메모리를 나눠 쓰는 방식에 따라 **공유 메모리(SMP, NUMA)** 시스템과, 각자 메모리를 챙겨서 네트워크로 통신하는 **분산 메모리(클러스터)** 시스템으로 분화되어 무한정 융합 확장 중이다.

---

## Ⅰ. MIMD의 개요

### 1. 정의
- 플린의 분류법에 따라 여러 개의 제어 유닛(CU)과 연산 유닛(PU)이 존재하며, 각 프로세서가 서로 다른 메모리 주소에서 전혀 상관없는 명령어를 가져와 완전히 별개의 데이터를 독립적으로 처리(비동기적 병렬 실행)하는 컴퓨터 구조다.

### 2. 등장 배경: '모두가 각자의 길을 가자'
- SIMD(벡터 프로세서)는 군대처럼 모두가 한 번에 똑같은 덧셈을 할 때만 좋았다. 하지만 "A코어는 워드 문서를 열고, B코어는 MP3 음악을 틀고, C코어는 카카오톡을 보내고 싶다"는 현대 멀티태스킹의 요구를 SIMD로는 도저히 흉내 낼 수 없었다. 아키텍트들은 "칩 하나에 아예 독립된 뇌(CPU)를 여러 개 박아 넣고, 각자 맘대로 일하게 풀어버려!"라는 결단(멀티코어 혁명)을 내리며 MIMD의 시대를 열었다.

### 3. 💡 비유: '대형 패밀리 레스토랑의 주방'
- MIMD는 **'4명의 요리사가 각자 다른 화구에서, 각자 다른 주문서를 보고, 다른 재료로 요리를 볶아내는 대형 식당 주방'**과 같다.
- 1번 요리사(코어 A)는 스테이크(데이터 A)를 굽고(명령어 A), 2번 요리사(코어 B)는 파스타(데이터 B)를 삶는다(명령어 B). 누가 먼저 끝날지도 모르고, 서로 하는 일도 완벽히 다르다(비동기성). 하지만 손님 입장에서는 수많은 요리가 동시에 쏟아져 나오는 기적을 맛본다.

- **📢 섹션 요약 비유**: 수만 명이 각자의 목적지를 향해 각자의 스마트폰 내비게이션을 켜고 제멋대로 달리지만, 도시 전체의 물류는 완벽하게 마비 없이 돌아가는 거대한 현대 메가시티 시스템입니다.

## Ⅱ. 메커니즘 및 메모리 공유 방식 (Deep Dive)

MIMD는 여러 뇌(Core)가 일하지만, **'도대체 데이터를 어떻게 나눠 가질 것인가?'**에 따라 아키텍처가 두 동강 난다.

### 1. 공유 메모리 모델 (Shared Memory System)
- **개념**: 여러 개의 CPU가 하나의 거대한 메인 메모리(RAM)를 찰떡같이 공유한다. 코어 A가 100번지에 편지를 써놓으면, 코어 B가 언제든 100번지를 열어서 편지를 읽을 수 있다.
- **장점**: 프로그래머가 코딩하기 엄청 쉽다(스레드 프로그래밍). 통신 속도가 칩 내부 버스를 타므로 빛처럼 빠르다.
- **단점**: 코어가 64개, 128개로 늘어나면 수많은 뇌가 하나의 메모리 문을 통과하려다 압사당하는 '버스 병목 현상'이 터진다. 확장에 한계가 뚜렷하다.
- **대표 모델**: 우리가 쓰는 PC(Intel Core, AMD Ryzen), SMP(대칭형 다중 프로세서).

### 2. 분산 메모리 모델 (Distributed Memory System)
- **개념**: 각 CPU가 아예 '자기만의 전용 RAM'을 꽉 끌어안고 있다. 남의 메모리를 직접 훔쳐볼 수 없다. 코어 A가 코어 B에게 데이터를 주려면, 굳이 랜선을 타고 네트워크 패킷(Message Passing)으로 편지를 보내야만 한다.
- **장점**: 메모리 문이 막힐 일이 없으므로, 컴퓨터를 1,000대, 10,000대 끝없이 무한정 이어 붙일 수 있다 (극강의 확장성).
- **단점**: 프로그래머가 MPI(Message Passing Interface) 같은 복잡한 언어를 써서 명시적으로 "데이터를 던져라, 받아라" 코딩을 해야 하므로 지옥을 맛본다.
- **대표 모델**: 슈퍼컴퓨터(HPC), 클라우드 컴퓨팅(AWS 클러스터).

### 3. MIMD 구조 시각화 (ASCII)
```text
  [ 공유 메모리 MIMD (우리가 쓰는 PC) ]       [ 분산 메모리 MIMD (구글 서버 클러스터) ]
  
   Core 1    Core 2    Core 3              [Core 1]─[RAM]   [Core 2]─[RAM]
     │         │         │                     │                │
     ▼         ▼         ▼                     ▼                ▼
  ════════ System Bus ════════               ═════ LAN (Ethernet/InfiniBand) ════
             │                                         │
             ▼                                         ▼
   [ 단 하나의 거대한 Main RAM ]               [Core 3]─[RAM]   [Core 4]─[RAM]
```

- **📢 섹션 요약 비유**: 한 테이블에 둘러앉아 반찬을 다 같이 나눠 먹는 대가족 식사(공유 메모리)와, 각자 자기 방에서 자기 도시락만 먹으면서 카톡으로만 대화하는 개인주의 자취생들(분산 메모리)의 차이입니다.

## Ⅲ. SIMD vs MIMD 철학적 대결

병렬 처리의 세계를 양분하는 두 괴물의 극명한 차이.

| 비교 항목 | SIMD (GPU, 벡터) | MIMD (CPU 멀티코어) |
|:---|:---|:---|
| **행동 통일성** | **무조건 똑같은 일**만 동시에 함 | **각자 완전히 다른 일**을 함 |
| **코어(ALU) 개수**| 수천 ~ 수만 개 (단순 노동자) | 4 ~ 64개 (고학력 엘리트) |
| **분기문(If-else)**| 분기 만나면 효율 박살 남 | **분기문을 밥 먹듯 씹어 먹음** |
| **스레드 동기화** | 동기화가 거의 필요 없음 (한 몸이니까) | 락(Lock)과 뮤텍스 동기화 지옥 발생 |
| **응용 분야** | 화면 픽셀 계산, 딥러닝 행렬 곱 | 운영체제 구동, 웹 서버, 게임 물리 엔진 |

- **📢 섹션 요약 비유**: 1만 명의 병사에게 동시에 "앞으로 갓!"을 시키는 스파르타 군대(SIMD)와, 8명의 장군이 방에 모여 각자 지도를 보고 독자적인 작전을 세우는 작전 회의실(MIMD)의 차이입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: SMP에서 NUMA로의 진화
- **상황**: 공유 메모리 MIMD(SMP) 방식에서 코어를 128개로 늘렸더니, 128명이 하나의 램을 쓰려다 트래픽 병목으로 서버가 죽었다. 분산 메모리로 가자니 윈도우 OS가 돌아가지 않는다.
- **판단**: 천재 아키텍트들은 공유와 분산을 섞어 **NUMA(Non-Uniform Memory Access)**라는 타협안을 만들었다. 물리적으로는 코어 32개씩 묶어서 자기들만의 램(Local RAM)을 따로 달아준다(분산 메모리 특성). 하지만 OS(소프트웨어) 눈에는 이 쪼개진 램들이 마치 하나의 거대한 공유 램인 것처럼 보이도록 하드웨어가 뻥을 친다(공유 메모리 특성). 단, 내 옆에 있는 램을 읽을 땐 엄청 빠르고, 남의 램을 읽으려면 케이블을 타야 해서 조금 느려지는 차별(Non-Uniform)이 생겼다. 현대 서버 아키텍처는 100% 이 NUMA 구조 위에서 돌아간다.

### 2. 실무 시나리오: 멀티스레드 코딩과 암달의 법칙 (Amdahl's Law)
- **인사이트**: 주니어 개발자가 "코어가 8개인 CPU(MIMD)를 샀으니 내 프로그램도 8배 빨라지겠지?"라고 생각한다. 하지만 2배밖에 안 빨라졌다. 왜일까?
- 프로그램 안에는 쪼개서 병렬로 실행할 수 있는 부분(예: 80%)도 있지만, **절대 쪼갤 수 없고 무조건 순서대로 실행해야 하는 부분(예: 20%)**이 존재한다(초기화, 파일 저장 등). 아무리 코어 10,000개를 때려 박아도 이 20%의 순차 코드 때문에 전체 속도는 5배(1/0.2) 이상 빨라지지 않는다. 이것이 병렬 처리의 차가운 진실인 **암달의 법칙**이다. 아키텍트는 맹목적인 코어 수 늘리기를 멈추고, 알고리즘 자체의 순차적 병목을 제거하는 데 시간을 쏟아야 한다.

- **📢 섹션 요약 비유**: 도로를 아무리 100차선으로 넓혀도(멀티코어), 중간에 다리가 1차선 톨게이트(순차적 코드)로 좁아지면 결국 차가 밀려서 목적지에 도착하는 시간은 똑같다는 냉혹한 물리 법칙입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **클럭 장벽의 붕괴**: 4GHz에서 더 이상 클럭을 올리지 못하는 전력/발열의 덫(Power Wall)을 회피하여, 코어를 가로로 복제하는 방식으로 프로세서의 무한 성능 업그레이드 시대를 열어젖힘.
- **클라우드 컴퓨팅의 모태**: 전 세계에 흩어진 수십만 대의 독립적인 서버(Node)들을 엮어 구글과 아마존을 탄생시킨 가장 거시적인 단위의 MIMD 아키텍처 실현.

### 2. 미래 전망
현재 인텔과 AMD의 최신 칩은 단순히 똑같은 코어를 여러 개 박는 동종(Homogeneous) MIMD를 벗어났다. 빡센 일을 하는 P-Core(성능 코어)와 백그라운드 잡일을 하는 전력 효율 E-Core를 섞어 쓰는 **이기종(Heterogeneous) 멀티코어 MIMD 아키텍처**인 빅리틀(big.LITTLE) 구조가 메인스트림이 되었다. 앞으로의 MIMD는 똑같은 일꾼의 숫자를 늘리는 무식한 방식에서, 다양한 전문가들을 적재적소에 배치하여 지휘하는 고도로 복잡한 '초정밀 오케스트라'로 진화하고 있다.

- **📢 섹션 요약 비유**: 단순히 힘센 장정 100명(과거 멀티코어)을 모으던 노가다판에서, 프로그래머, 디자이너, 마케터 등 각자 전문 분야(이기종 코어)가 다른 100명을 모아 세상을 바꾸는 스타트업으로 거듭나는 셈입니다.
