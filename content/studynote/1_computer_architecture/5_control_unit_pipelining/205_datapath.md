# 205. 데이터패스 (Datapath)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터패스 (Datapath)는 CPU 내부에서 연산이 수행되는 모든 물리적 구성 요소 (ALU, 레지스터 파일, 버스, 멀티플렉서)와 그들 간의 데이터 이동 경로를 의미한다.
> 2. **가치**: 명령어의 실행 속도 (사이클 시간)를 물리적으로 결정하는 핵심 하드웨어이며, 단일 사이클, 다중 사이클, 파이프라인 등 마이크로아키텍처 방식에 따라 경로의 복잡도가 크게 변화한다.
> 3. **융합**: 고성능 처리를 위해 제어 유닛 (Control Unit)과 협력하며, 명령어 간의 의존성 병목을 해결하기 위해 데이터 포워딩 (Data Forwarding) 같은 우회 경로 기술이 결합되어 성능을 끌어올린다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 데이터패스 (Datapath)는 레지스터 파일, 산술논리장치 (ALU), 멀티플렉서 (MUX), 시스템 버스 등 데이터의 실제 연산과 전송이 일어나는 물리적 소자들과 그들 간의 연결망을 의미한다. 명령어의 의미를 물리적인 비트 이동으로 실현하는 '컴퓨터의 육체'와 같다.

- **필요성**: 데이터패스는 프로세서의 **실행 지연 시간(Latency)과 주파수 한계를 결정짓는 물리적 뼈대**로서 반드시 필요하다. 전기가 수많은 게이트와 긴 배선을 통과하는 데 걸리는 시간(Propagation Delay)이 곧 클럭 주기를 결정하므로, 하드웨어 경로를 최적화하여 '임계 경로(Critical Path)'를 단축하는 것은 시스템 성능 향상의 절대적 전제 조건이기 때문이다. 또한, 파이프라인 단계별로 데이터를 격리하고 우회 경로(Forwarding)를 구축함으로써 **명령어 수준의 병렬성(ILP)을 하드웨어적으로 뒷받침**하며, 한정된 실리콘 다이 면적 내에서 데이터 처리 대역폭을 극대화하여 초고속 연산을 실현하는 물류 인프라 역할을 수행한다.
- **💡 비유**: 공장에서 제품을 조립하는 "컨베이어 벨트와 작업대"와 같다. 부품(데이터)이 창고(레지스터)에서 나와 조립 기계(ALU)를 거쳐 다시 창고로 들어가는 모든 물리적 동선이 바로 데이터패스다. 동선이 꼬이지 않고 짧을수록 생산성이 높아진다.

- **등장 배경**: 초기 프로세서는 복잡한 회로 없이 버스 하나에 여러 유닛이 매달려 순차적으로 데이터를 주고받았으나 (폰 노이만 병목), 점차 병렬 처리에 대한 요구가 높아졌다. 이에 따라 다중 버스, 레지스터 파일 분할 다중 포트, 전용 ALU를 포함하여 동시에 여러 데이터가 충돌 없이 흐를 수 있는 복잡한 데이터패스 설계가 필수적이 되었다.

```text
  [명령어 입력] ──▶ (Control Unit) ───[제어 신호]──▶ (Datapath 경로 제어)
                                                         │
  ┌──────────────────────────────────────────────────────▼───────┐
  │                        데이터패스 (Datapath)                 │
  │                                                              │
  │  [PC] ──▶ [명령어 메모리] ──▶ [레지스터 파일] ──▶ [ALU] ──▶ [데이터 메모리] │
  │                                │          │                  │
  │  ◀─────────────────────────────┴──────────┴───────────────────┘
  │                    (결과값 피드백 / Write Back)
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터패스는 프로그램 카운터 (PC)가 가리키는 주소에서 명령어를 인출하는 것을 시작으로, 레지스터에서 피연산자를 읽고, ALU에서 산술 연산을 수행하며, 결과를 다시 레지스터나 메모리에 저장하는 전체 경로를 포괄한다. 이 모든 구성 요소 사이의 데이터 흐름은 제어 유닛 (Control Unit)이 내리는 신호 (MUX 선택, 레지스터 쓰기 허용 등)에 의해 길목이 통제되며, 상태 플래그를 제어 유닛에 반환하여 조건 분기를 가능하게 한다.

- **📢 섹션 요약 비유**: 물탱크에서 출발해 복잡한 수도관을 따라 물이 흐르고 정수 필터를 거쳐 다시 저장조로 들어가는 '전체 배관 시스템'과 같습니다. 밸브(제어 유닛)를 열고 닫는 것에 따라 물(데이터)이 정해진 길(데이터패스)을 이동합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **레지스터 파일 (Register File)** | 프로세서 내 가장 빠른 임시 저장소 | 다수의 읽기/쓰기 포트를 통해 동시 접근 허용 | 작업자 바로 옆의 공구함 |
| **산술논리장치 (ALU)** | 산술 연산 및 논리 연산 수행 | 입력된 두 값과 제어 신호에 따라 결과 도출 | 핵심 조립 로봇 기계 |
| **멀티플렉서 (MUX)** | 여러 입력 중 하나를 선택해 출력 전달 | 제어 신호(Control Signal)에 따라 경로 스위칭 | 선로의 교차로 스위치 |
| **프로그램 카운터 (PC)** | 다음 실행할 명령어 주소 보관 | 순차 실행(+4) 또는 조건 분기 시 주소 갱신 | 작업 지시서의 페이지 번호 |
| **데이터 버스 (Bus)** | 구성 요소 간 데이터 전송 물리적 통로 | 병렬 전선(예: 32bit, 64bit)으로 다중 비트 동시 전송 | 부품을 나르는 컨베이어 벨트 |

### 단일 사이클 데이터패스의 데이터 흐름

MIPS 또는 RISC-V 아키텍처 기준의 R-타입 (레지스터 연산) 명령어 실행 시의 기본적인 데이터패스 흐름이다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │                 R-Type 명령어의 데이터패스 흐름                     │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │  [Instruction Memory]                                         │
  │         │  (rs, rt, rd 필드 해독)                               │
  │         ▼                                                     │
  │  ┌──────────────┐         [Read Data 1]                       │
  │  │              │ ────────────────────┐                       │
  │  │   Register   │                     ▼                       │
  │  │     File     │                ┌─────────┐                  │
  │  │              │ ────────────▶  │   ALU   │                  │
  │  └──────────────┘ [Read Data 2]  │         │                  │
  │         ▲                        └─────────┘                  │
  │         │                             │                       │
  │         │       [ALU Result]          │                       │
  │         └─────────────────────────────┘                       │
  │           (Write Data로 피드백/저장)                             │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** R-타입 명령어 (예: `ADD rd, rs, rt`)가 인출되면, 레지스터 파일은 `rs`와 `rt` 필드가 지정하는 주소에서 두 개의 피연산자 데이터를 읽어 ALU 입력으로 보낸다. ALU는 덧셈 연산을 수행하고 그 결과 (ALU Result)를 산출한다. 이 결과 데이터는 메모리로 가지 않고 바로 다시 레지스터 파일의 `rd` 포트로 피드백되어 저장 (Write Back)된다. 이 과정에서 각 분기점에 위치한 멀티플렉서는 제어 유닛의 신호를 받아 메모리 출력이 아닌 ALU 결과를 선택하도록 물리적 경로를 잇는다.

- **📢 섹션 요약 비유**: 부품 상자에서 나사와 바퀴를 꺼내(레지스터 읽기), 기계에 넣고 조립한 뒤(ALU 연산), 완성된 부품을 다시 상자에 넣는(레지스터 쓰기) 명확한 하나의 공정 라인과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 단일 사이클 vs 다중 사이클 데이터패스

| 비교 항목 | 단일 사이클 데이터패스 | 다중 사이클 데이터패스 |
|:---|:---|:---|
| **실행 시간** | 가장 긴 명령어 (예: Load) 시간에 맞춰 전체 사이클 고정 | 명령어를 단계별로 짧은 사이클로 분할 실행 |
| **자원 공유** | 불가능 (명령어/데이터 메모리 분리, 복수 연산기 필요) | 가능 (단일 메모리, 단일 ALU 시분할 재사용) |
| **하드웨어 비용** | 연산기와 메모리 중복 배치로 비용/면적 증가 | 상태 보존용 임시 레지스터 추가되나 주요 자원 절약 |
| **성능 효율성** | 짧은 명령어 (예: Add) 실행 시 여유 시간이 버려짐 | 명령어별 필요 사이클 수만 소모하여 효율적 클럭 운용 |

### 파이프라인 데이터패스와 전방 전달 (Forwarding)

파이프라이닝을 적용하면 데이터패스는 여러 단계 (IF, ID, EX, MEM, WB)로 나뉘고, 그 사이에 '파이프라인 래치(레지스터)'가 삽입된다. 하지만 이전 명령어의 결과를 다음 명령어가 바로 필요로 할 때 병목이 발생한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         파이프라인 데이터패스와 전방 전달 (Forwarding)           │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │ Inst 1: ADD $t0, $t1, $t2   [IF][ID][EX][MEM][WB]            │
  │                                       │ (ALU Result: $t0)    │
  │                                       ▼                      │
  │ Inst 2: SUB $t3, $t0, $t4       [IF][ID][EX][MEM][WB]        │
  │                                         ($t0 필요)             │
  │                                                              │
  │ ⚠ 문제 (Data Hazard):                                        │
  │ Inst 1이 WB 단계에서 $t0 레지스터에 쓰기 전에,                      │
  │ Inst 2가 ID 단계에서 그 값을 읽으려 하여 잘못된 옛날 값을 가져옴.       │
  │                                                              │
  │ ✅ 해결 (Forwarding Datapath):                                 │
  │ EX/MEM 파이프라인 레지스터에 있는 계산 직후의 값을 MUX를 통해 직접      │
  │ Inst 2의 ALU 입력으로 우회(Bypass) 연결하여 스톨(Stall) 없이 실행.   │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 흐름은 파이프라인 데이터패스 설계에서 필연적으로 발생하는 데이터 해저드 (Data Hazard) 상황과 그 물리적 해결책인 포워딩 (Forwarding, 우회 전달) 경로를 보여준다. 이전 명령어 (ADD)의 연산 결과가 최종 레지스터 파일에 기록되기 전에 다음 명령어 (SUB)가 그 값을 ALU 입력으로 필요로 할 때, 파이프라인 래치 (EX/MEM 단)에 임시 보관된 중간 값을 MUX를 통해 ALU 입력단으로 직접 피드백하는 추가적인 데이터패스를 뚫어준다. 이 배선을 통해 명령어는 파이프라인 지연(Stall) 없이 부드럽게 흘러간다.

- **📢 섹션 요약 비유**: 앞 사람이 만든 부품을 창고(레지스터)에 굳이 넣었다가 뒷사람이 다시 꺼내 가는 헛수고 대신, 컨베이어 벨트 중간에서 앞 사람이 뒷사람에게 직접 부품을 건네주는(포워딩 배선) 효율적인 요령과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **데이터센터 서버의 캐시 미스에 따른 데이터패스 대기 (Stall)**
   CPU 내부 데이터패스는 나노초(ns) 단위로 고속 동작하지만, 데이터가 캐시를 뚫고 메인 메모리(DRAM)에서 와야 할 때 데이터패스의 ALU와 레지스터는 수백 사이클 동안 할 일 없이 정지(Stall)한다. 아키텍트는 이 유휴 시간을 극복하기 위해 비순차적 실행 (Out-of-Order Execution) 데이터패스를 설계한다. 이는 데이터를 기다리는 동안 명령어를 임시 보관하는 예약역 (Reservation Station)과 결과 재배치 버퍼 (ROB)를 데이터패스에 추가하여, 독립적인 다른 명령어들을 여분의 ALU에서 먼저 연산하도록 길을 터주는 고비용·고효율 기술이다.

2. **DSP 및 AI 가속기의 데이터패스 확장 (SIMD/Tensor)**
   딥러닝 추론이나 신호 처리 시 방대한 양의 행렬 곱셈과 덧셈 (MAC 연산)이 반복된다. 일반 범용 CPU의 스칼라 데이터패스로는 한 사이클에 하나의 MAC 연산만 처리할 수 있어 비효율적이다. 따라서 아키텍트는 하나의 명령어로 여러 데이터를 동시에 병렬 처리하는 넓은 벡터 데이터패스 (SIMD: Single Instruction Multiple Data, 예: 256/512비트 버스)를 탑재하거나, 구글의 TPU처럼 2차원 배열 형태로 ALU가 엮인 시스톨릭 어레이 (Systolic Array) 데이터패스를 설계하여 처리량(Throughput)을 수백 배 높인다.

### 도입 체크리스트
- **기술적**: 데이터패스 내 임계 경로 (Critical Path)의 신호 지연 시간이 목표 클럭 주파수 상한선을 충족하는가? (일반적으로 레지스터 읽기 + ALU 연산 + 메모리 접근 시간이 결합된 경로가 가장 길다).
- **운영적**: 설계한 데이터패스의 버스 대역폭 (Bus Width)이 32비트인가 64비트인가? 타겟 애플리케이션의 메모리 요구량과 정확도(Precision)를 충당할 수 있으면서도 과도한 면적을 차지하지 않는 최적의 폭인가?

### 안티패턴
- **무조건 넓고 복잡한 데이터패스 채택**: 단순 제어용 임베디드 IoT 기기에서 데스크탑급 64비트 파이프라인 데이터패스를 채택하면, 실제 다루는 센서 데이터는 8/16비트 단위임에도 불구하고 넓은 버스 선로와 래치 회로로 인해 다이(Die) 면적과 누설 전류(Leakage)만 기하급수적으로 증가하는 낭비가 발생한다. 워크로드 성격에 맞춰 데이터패스를 경량화해야 한다.

- **📢 섹션 요약 비유**: 트랙터 1대만 간신히 다니는 시골길에 왕복 8차선 고속도로(64비트 데이터패스)를 뚫어버리면, 유지비(전력과 면적)만 막대하게 버려지는 것처럼, 목적에 맞는 적절한 배관(데이터패스) 크기 설정이 중요합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 범용 (스칼라) 데이터패스 | 특수 목적 (벡터/가속) 데이터패스 도입 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (성능)**| 1사이클에 1개 32bit 데이터 연산 | 256비트 SIMD 적용 시 8개 데이터 동시 연산 | 사이클당 연산량 **8배 증가** |
| **정량 (전력)**| 명령어 해독 및 제어 오버헤드 큼 | 제어 한 번에 묶음 연산으로 단위 전력 감소 | 동일 연산 대비 **전력 소모 대폭 절감** |

### 미래 전망
- **광학 (Optical) 데이터패스의 상용화 연구**: 반도체 공정이 미세화될수록 칩 내부의 구리선을 통한 전기적 데이터 전송 지연 (RC Delay)과 저항 발열이 커지고 있다. 이를 돌파하기 위해 미래의 데이터패스는 레이저와 실리콘 포토닉스 기술을 결합하여, 코어 간 데이터 전송 속도를 빛의 속도로 끌어올리는 O-NoC (Optical Network-on-Chip) 형태로 진화할 전망이다.
- **공간 아키텍처 (Spatial Architecture)로의 패러다임 전환**: 기존처럼 명령어(시간)에 따라 ALU로 데이터를 가져오는 폰 노이만 구조를 넘어, 알고리즘의 데이터 흐름도 자체를 하드웨어(수많은 연산기와 배선 네트워크)에 직접 매핑(Mapping)하여 데이터가 알아서 흘러가며 가공되는 CGRA (Coarse-Grained Reconfigurable Architecture) 방식이 AI 워크로드 데이터패스의 새로운 표준이 되고 있다.

- **📢 섹션 요약 비유**: 좁고 구불구불한 1차선 흙길(초기 폰 노이만 데이터패스)이 여러 개의 나들목과 고가도로가 복잡하게 얽힌 입체적인 스마트 하이웨이(현대 가속기 데이터패스)로 발전하여, 멈춤 없는 교통 흐름을 만들어내는 과정입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **제어 유닛 (Control Unit)** | 데이터패스 내의 멀티플렉서, 레지스터, ALU 등에 제어 신호를 보내 데이터 흐름의 방향을 물리적으로 스위칭한다. |
| **명령어 집합 구조 (ISA)** | 어떤 연산 종류와 어떤 크기의 레지스터가 필요한지 정의하여 데이터패스의 폭(Width)과 ALU 내부 논리 회로를 결정짓는다. |
| **ALU (Arithmetic Logic Unit)** | 데이터패스의 심장부로, 메모리나 레지스터로부터 전달받은 데이터의 실제 산술 및 논리 연산을 수행한다. |
| **파이프라이닝 (Pipelining)** | 데이터패스의 전체 긴 경로를 여러 스테이지로 분할하고 그 사이에 상태 레지스터를 삽입해 중첩 병렬 실행을 가능하게 한다. |
| **멀티플렉서 (MUX)** | 제어 유닛의 지시에 따라 여러 갈래의 데이터 경로 중 하나를 선택해, 한정된 자원의 데이터패스에 가변적 연결성을 제공한다. |
