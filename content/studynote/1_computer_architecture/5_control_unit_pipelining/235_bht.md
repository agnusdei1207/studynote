> **핵심 인사이트**
> 1. 분기 역사 표(BHT)는 동적 분기 예측(Dynamic Branch Prediction)의 두뇌 역할을 하는 하드웨어 테이블로, **각 분기 명령어의 과거 점프 이력(Taken / Not Taken)을 기록해 두는 전용 메모리 배열**이다.
> 2. 주로 **2비트 포화 카운터(2-bit Saturating Counter)** 상태 머신을 각 칸에 저장하여, 분기 예측이 한 번 틀렸다고 해서 성급하게 예측 방향을 바꾸지 않는 '관성'을 부여한다.
> 3. 목적지를 알려주는 BTB(Branch Target Buffer)와 함께 동작하여, 분기 명령어가 파이프라인에 진입하자마자 **미래의 점프 여부를 90% 이상의 확률로 정확히 추측**해 낸다.

---

## Ⅰ. BHT의 구조와 2비트 포화 카운터

BHT는 아주 단순한 1차원 배열(테이블) 형태의 캐시 메모리입니다.
명령어 주소(PC)의 하위 몇 비트를 인덱스로 사용하여 표의 특정 칸을 찌르면, 그 칸에 들어있는 '상태 비트'가 튀어나옵니다.

가장 대표적인 **2비트 포화 카운터(Bimodal Predictor)**는 4가지 상태를 가집니다.
- `11` (Strongly Taken): 강하게 뜀
- `10` (Weakly Taken): 약하게 뜀
- `01` (Weakly Not Taken): 약하게 안 뜀
- `00` (Strongly Not Taken): 강하게 안 뜀

**동작 원리:**
- 현재 상태가 `11`일 때 预测대로 뛰었다면(Taken), 여전히 `11`에 머뭅니다(포화 상태).
- 만약 예측이 틀려서 안 뛰었다면(Not Taken), 상태는 `10`(약하게 뜀)으로 한 칸 내려갑니다.
- **핵심:** 한 번 틀려도 다음번 예측은 여전히 "뜀(Taken)"입니다. 두 번 연속 틀려야만 비로소 `01`(약하게 안 뜀)로 넘어가며 예측 방향을 바꿉니다.

📢 **섹션 요약 비유:**
어떤 손님이 짬뽕을 매일 먹다가(11), 하루 실수로 짜장면을 시켰다고(10) 해서 내일 당장 짜장면을 대접하지는 않습니다. 하지만 이틀 연속 짜장면을 시키면(01) "아, 이 사람 입맛이 변했구나!" 하고 내일 짜장면을 준비하는 신중한 식당 주인입니다.

## Ⅱ. BHT의 한계와 해시 충돌 (Aliasing)

BHT는 완벽하지 않습니다. 하드웨어 용량의 한계 때문에 발생하는 치명적인 문제가 있습니다.

- **주소 충돌 (Aliasing):** BHT를 무한정 크게 만들 수 없으므로, 보통 명령어 주소(PC)의 하위 10~12비트만 잘라서 BHT의 인덱스로 씁니다 (즉, 표의 칸이 1024~4096개).
- 만약 주소 하위 비트가 우연히 똑같은 서로 다른 두 개의 분기 명령어(예: `0x1000`과 `0x5000`)가 존재한다면, 이 두 명령어는 **BHT의 같은 칸을 공유하며 서로의 과거 전적을 덮어써서 망쳐버리게 됩니다.**
- **해결책:** 이 충돌을 줄이기 위해 Gshare 같은 고급 예측기에서는 PC 값뿐만 아니라 '글로벌 분기 기록(Global History Register)'을 XOR 연산하여 해시(Hash) 값을 만들어 인덱스로 사용합니다.

📢 **섹션 요약 비유:**
전교생의 성향(BHT)을 기록해야 하는데 수첩이 100페이지밖에 없어서, '학번 끝자리'가 같은 학생들을 같은 페이지에 적다 보니 모범생과 날라리의 기록이 섞여서 예측을 망치는 현상입니다.

## Ⅲ. 상관관계 (Correlating) 예측과 2단계 BHT

`if (x == 2) { ... } if (x == 2) { ... }` 
이런 코드처럼, 두 번째 `if`문은 앞선 첫 번째 `if`문의 결과에 100% 종속됩니다. 하지만 기본 BHT는 각 `if`문을 독립적으로 보기 때문에 이 상관관계를 파악하지 못합니다.

- **Two-Level BHT (투 레벨 예측기):**
  1. 먼저 **GHR(Global History Register)**이라는 8비트 레지스터가 "최근 8번의 모든 분기문이 뛰었나 안 뛰었나"의 패턴(예: T-N-T-N...)을 저장합니다.
  2. 이 패턴 값을 바탕으로 BHT의 여러 층(Level) 중 하나를 고릅니다.
  3. 즉, "최근에 T-N-T-N 패턴으로 흘러왔을 때, 이 분기 명령어는 뛸 확률이 얼마나 되나?"를 BHT에서 읽어옵니다.
- 이를 통해 복잡한 `if-else`가 난무하는 객체지향 코드나 AI 알고리즘에서도 95% 이상의 예측 적중률을 끌어낼 수 있습니다.

📢 **섹션 요약 비유:**
단순히 "이 선수가 최근 폼이 좋냐?"(기본 BHT)만 보는 게 아니라, "비가 오고(GHR), 원정 경기일 때(GHR), 이 선수가 최근 폼이 어땠냐?"라는 입체적인 데이터 분석을 통해 승률을 찍어 맞히는 것입니다.

## Ⅳ. BTB와의 협업 프로세스

BHT는 단독으로 쓰이지 않고 항상 BTB(분기 목적지 버퍼)와 세트로 동작합니다.

1. **인출(Fetch) 단계:** PC 값이 메모리에서 명령어를 가져올 때, 동시에 BTB와 BHT를 찌릅니다.
2. **평가:** 
   - BHT 왈: "최근 기록을 보니 뛸 것 같다!(Taken)"
   - BTB 왈: "목적지는 0x8000이다!"
3. **액션:** 두 장치의 말이 모두 확인되면, CPU는 명령어 해독(Decode)이 채 끝나기도 전에 파이프라인의 다음 PC 값을 0x8000으로 강제 변경해버립니다.

📢 **섹션 요약 비유:**
BHT(전략가)가 "적은 북쪽으로 올 확률이 높다"고 판단하면, BTB(전술가)가 "북쪽 길의 매복 지점은 좌표 X다"라고 알려주어 군대가 한 치의 망설임 없이 미리 움직이는 완벽한 지휘 체계입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Dynamic Branch Prediction]`: BHT를 핵심으로 사용하는 동적 분기 예측 기술
- `[Branch Target Buffer (BTB)]`: BHT가 예측한 방향으로 뛰기 위해 목적지를 알려주는 짝꿍 하드웨어
- `[2-bit Saturating Counter]`: BHT의 각 칸에 들어가는, 관성을 가진 4상태 머신
- `[Control Hazard]`: BHT가 95% 확률로 막아내는 파이프라인의 치명적 충돌
- `[Speculative Execution]`: BHT의 예측을 믿고 다음 명령어를 선제적으로 실행해 버리는 최적화 기법
