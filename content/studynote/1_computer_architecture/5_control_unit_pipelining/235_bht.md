# 분기 이력 테이블 (BHT: Branch History Table)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분기 이력 테이블(BHT)은 동적 분기 예측을 구현하기 위해 CPU 내부에 마련된 고속 메모리 구조로, **과거 수천 개의 분기문들이 각각 점프했는지(Taken) 안 했는지(Not Taken)의 상태를 기록해 두는 전용 캐시 테이블**이다.
> 2. **가치**: 1비트 또는 2비트 포화 카운터(Saturating Counter) 배열을 저장하여, 프로그램이 실행되면서 시시각각 변하는 복잡한 if-else 패턴이나 루프 종료 시점을 95% 이상의 확률로 정확히 예측(학습)하게 해준다.
> 3. **융합**: 명령어의 하위 주소 비트를 인덱스로 사용하여 테이블에 접근하며, 단순한 BHT의 한계를 극복하기 위해 전역 이력 레지스터(Global History Register)와 융합된 형태의 예측기(예: GShare)로 발전하였다.

---

## Ⅰ. 분기 이력 테이블(BHT)의 개요

### 1. 정의
- 프로그램 카운터(PC) 값의 일부(주로 하위 비트)를 주소(인덱스)로 사용하여 접근하는 소규모 SRAM 해시 테이블이다. 각 항목(Entry)에는 해당 분기 명령어의 최신 '점프 성향(2비트 상태)'이 저장되어 있다.

### 2. 등장 배경: '기계에게 기억력을 부여하다'
- "이 if문은 뛸까, 안 뛸까?" 정적 예측처럼 고정된 규칙으로 찍으면 적중률이 80%를 넘지 못한다. 아키텍트들은 인간의 뇌처럼 '경험'을 기록할 공간이 필요함을 깨달았다. 그래서 명령어 주소마다 "최근 2번 다 점프했음" 같은 작은 도장을 찍어둘 메모리 게시판을 하드웨어 내부에 만들었는데, 이것이 BHT의 탄생이다.

### 3. 💡 비유: '야구 선수의 타격 방향 기록부'
- BHT는 **'야구팀 전력분석관이 들고 있는 타자들의 타격 방향 기록부'**와 같다.
- "100번 타자(PC 주소)는 최근 3번 연속 우측으로 쳤어(Taken). 그러니까 이번에도 우측 수비를 강화하자!"라고 예측한다. 만약 이번에 좌측으로 치면 기록부의 상태를 조금 수정(학습)해 두고 다음 타석에 반영한다.

- **📢 섹션 요약 비유**: 수만 번 반복되는 거대한 쳇바퀴(루프) 속에서 언제 쳇바퀴 밖으로 뛰어내려야 할지, 과거의 통계를 바탕으로 귀신같이 타이밍을 찾아내는 빅데이터 장부입니다.

---

## Ⅱ. 메커니즘 및 하드웨어 동작 (Deep Dive)

BHT는 명령어 인출(Fetch) 단계에서 PC 값을 이용해 0.1초도 안 되어 상태를 꺼내온다.

### 1. BHT의 맵핑 및 조회 구조
- 32비트 PC 주소가 `0x1000_1234`라면, 이 주소의 하위 특정 비트(예: 12비트)만을 인덱스로 사용해 BHT 배열(크기 $2^{12} = 4096$개)에 접근한다.
- 그 인덱스에 들어 있는 2비트 카운터 값(예: `11` = 강한 Taken)을 읽고, 파이프라인 제어 유닛에 "이 명령어는 무조건 점프할 거다!"라고 통보한다.

### 2. 하드웨어 데이터 흐름 시각화 (ASCII)
```text
    [ 현재 Fetch 중인 분기문 PC: 0x4010 ]
    
    PC의 하위 비트 (010000) ──┐
                              │ (인덱싱)
                              ▼
    ┌──────────────── BHT (Branch History Table) ────────────────┐
    │ Index(주소) │ 2-bit Counter (상태)   │ 예측 결과 (Predict) │
    ├─────────────┼────────────────────────┼─────────────────────┤
    │  ...        │ ...                    │ ...                 │
    │  001111     │ 01 (Weakly Not Taken)  │ 안 뛴다 (N.T)       │
    │▶ 010000 ───▶│ 11 (Strongly Taken)    │ 뛴다 (Taken)! ◀──── │ (이 예측을 바탕으로 실행)
    │  ...        │ ...                    │ ...                 │
    └────────────────────────────────────────────────────────────┘
                              │
    [ Execute 단계 후 ]       │ (실제 결과가 나옴)
    만약 실제로 '안 뛰었다면' 11 -> 10 (Weakly Taken)으로 상태를 하향 업데이트 (학습 완료)
```

### 3. 심층 분석: BHT의 치명적 약점 '알리아싱 (Aliasing)'
- BHT는 용량의 한계 때문에 PC 전체 주소(32비트)를 다 쓰지 않고 하위 10~12비트만 잘라서 쓴다. 만약 서로 완전히 다른 위치에 있는 두 개의 분기문이 하위 12비트 주소가 우연히 똑같다면? 두 분기문이 하나의 2비트 카운터를 공유하며 서로의 예측 기록을 엉망으로 훼손해 버린다. 이를 분기 예측기에서의 **알리아싱(충돌) 문제**라고 한다.

- **📢 섹션 요약 비유**: 동명이인이 너무 많아서 배달부가 1동 사는 김철수의 택배 기록을 3동 사는 김철수의 집 앞에 잘못 엎질러버리는 어처구니없는 통계의 혼선입니다.

## Ⅲ. BHT의 한계 극복: 연관 분기 예측 (Correlating Predictor)

단순한 BHT 하나만으로는 패턴이 복잡한 코드를 뚫어낼 수 없어 진화한 구조다.

| 예측기 종류 | 원리 | 장점 | 단점 |
|:---|:---|:---|:---|
| **기본 BHT (1D)** | PC 주소만으로 카운터 참조 | 구조 단순 | 패턴(홀짝홀짝 등) 학습 불가 |
| **Two-Level BHT** | 1차로 분기 패턴 레지스터(BHR)를 거쳐 2차 BHT 참조 | `1010` 같은 반복 패턴 완벽 예측 | 메모리가 이중으로 필요함 |
| **GShare 예측기**| PC 주소 $\oplus$(XOR) 전역 분기 이력(GHR) = BHT 인덱스 | 알리아싱 대폭 감소, 문맥 파악 가능 | 하드웨어 복잡도 증가 |

- **📢 섹션 요약 비유**: 단순히 "이 선수가 어제 홈런을 쳤나?"(기본 BHT)만 보는 게 아니라, "앞선 두 타자가 안타를 쳤을 때 이 선수가 어떻게 행동했나?"(전역 이력 조합)라는 흐름(Context)까지 파악하는 지능형 스카우터로의 진화입니다.

---

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: 테이블 크기와 성능의 트레이드오프
- **상황**: BHT 항목 수를 4,096개(4K)에서 16,384개(16K)로 4배 늘리자.
- **판단**: 테이블이 커지면 명령어 간 충돌(알리아싱)이 줄어 적중률이 90%에서 93%로 오른다. 하지만 BHT는 캐시의 일종이므로 너무 거대해지면 '읽어오는 데 걸리는 시간(Latency)' 자체가 1클럭을 넘어서 파이프라인 IF 단계를 늦춰버리는 대참사가 일어난다. 아키텍트는 칩 영역 예산(Area Budget)과 접근 지연(Access Latency)의 마지노선 내에서 BHT의 최적 크기를 결정하는 극한의 줄다리기를 해야 한다.

### 2. 실무 시나리오: 커널 코드와 BHT 캐시 오염
- **인사이트**: 운영체제 커널 코드가 실행되다 사용자 애플리케이션으로 컨텍스트 스위치(Context Switch)가 일어날 때, 커널이 마구 널뛰면서 BHT의 귀중한 2비트 카운터들을 모두 자신의 기록으로 덮어써(Pollution) 버린다. 애플리케이션으로 돌아오면 분기 예측률이 급전직하하여 성능 렉이 발생한다. 기술사는 이를 막기 위해 최신 CPU가 BHT에 권한 비트(User/Kernel)나 프로세스 ID 비트를 추가하여 문맥별로 예측 기록을 격리(Isolation)하는 아키텍처적 진보를 이루었음을 이해해야 한다.

- **📢 섹션 요약 비유**: 아빠가 쓰던 유튜브 알고리즘(BHT)을 아들이 이어받으면 한동안 엉뚱한 추천 영상(예측 실패)이 뜨며 버벅거리지만, 프로필을 분리하면 각자의 취향이 보호되는 원리입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **제어 해저드 방어의 핵심**: 동적 분기 예측의 뇌수 역할을 수행하며, 파이프라인 플러시로 낭비되는 클럭을 수십 배 감소시킴.
- **머신러닝의 하드웨어적 시초**: 과거의 데이터를 상태로 저장하여 미래의 방향을 확률적으로 결정하는, 반도체 칩 내부에서 가장 직관적인 학습(Learning) 메커니즘.

### 2. 미래 전망
BHT의 2비트 카운터 모델은 1980년대 짐 스미스(Jim Smith)가 발명한 이래 30년간 업계 표준이었다. 하지만 최근 신경망 가속 기술이 발달하면서, 거대한 카운터 테이블 대신 분기 히스토리를 신경망의 가중치(Weight) 연산기로 넘겨 예측하는 **타지(TAGE) 예측기**나 퍼셉트론 예측기가 하이엔드 CPU(x86, Apple Silicon)의 왕좌를 차지했다. BHT는 단순한 표(Table)를 넘어 인간의 직관을 흉내 내는 소형 신경망 칩으로 그 개념이 확장되고 있다.

- **📢 섹션 요약 비유**: 낡은 장부 책(BHT)에 바를 정(正)자를 그어가며 통계를 내던 주먹구구식 상회에서, 빅데이터 AI 알고리즘을 돌려 수요를 예측하는 최첨단 무역회사로 탈바꿈했습니다.
