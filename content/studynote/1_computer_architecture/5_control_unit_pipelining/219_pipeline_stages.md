> **핵심 인사이트**
> 1. 전통적인 RISC(예: MIPS, ARM)의 명령어 파이프라인은 명령어의 실행을 논리적, 물리적으로 분리 가능한 **5개의 기본 단계(IF, ID, EX, MEM, WB)**로 쪼개어 구성한다.
> 2. 각 단계는 서로 다른 전용 하드웨어 자원을 사용하며, 각 단계 사이에는 이전 단계의 결과물을 잠시 보관하는 **파이프라인 레지스터(Pipeline Register / Latch)**가 벽처럼 세워져 있다.
> 3. 모든 명령어가 5단계를 모두 거치는 것은 아니지만(예: ADD는 MEM을 사용하지 않음), 파이프라인의 동기화를 위해 **사용하지 않는 단계라도 반드시 1클럭의 시간을 소모하며 거쳐 가도록 강제**된다.

---

## Ⅰ. 5단 파이프라인의 구성과 역할

현대 CPU 설계의 교과서적인 5단계 파이프라인(MIPS 아키텍처 기준)의 역할은 다음과 같습니다.

### 1. IF (Instruction Fetch - 명령어 인출)
- **주요 동작:** 프로그램 카운터(PC)가 가리키는 주소의 명령어를 메모리(또는 L1 명령어 캐시)에서 읽어와 명령어 레지스터(IR)에 저장합니다.
- **부가 동작:** 다음 명령어를 위해 PC 값을 4(명령어 길이)만큼 자동 증가시킵니다.
- **핵심 하드웨어:** PC, 명령어 메모리(I-Cache), 주소 덧셈기.

### 2. ID (Instruction Decode & Register Read - 명령어 해독 및 레지스터 읽기)
- **주요 동작:** 제어 유닛(Control Unit)이 명령어의 Opcode를 해독하여 제어 신호를 만듭니다. 동시에 오퍼랜드 필드가 가리키는 레지스터 번호를 통해 범용 레지스터 파일에서 실제 데이터를 꺼내옵니다.
- **핵심 하드웨어:** 제어 유닛(디코더), 레지스터 파일.

### 3. EX (Execute / Address Calculation - 연산 및 주소 계산)
- **주요 동작:** ALU가 본격적으로 일을 하는 단계입니다. 산술/논리 연산을 수행하거나, 메모리 접근(Load/Store)을 위한 유효 주소(EA)를 덧셈으로 계산합니다.
- **핵심 하드웨어:** ALU(산술 논리 연산 장치).

### 4. MEM (Memory Access - 메모리 접근)
- **주요 동작:** EX 단계에서 계산한 유효 주소(EA)를 가지고 실제 데이터 메모리(D-Cache)에 접근합니다.
  - `Load`: 메모리에서 데이터를 읽어옵니다.
  - `Store`: 레지스터의 데이터를 메모리에 씁니다.
  - *주의:* 덧셈(`ADD`) 같은 산술 명령어는 메모리 접근이 필요 없으므로 이 단계에서 아무 일도 하지 않고(NOP) 1클럭을 대기합니다.
- **핵심 하드웨어:** 데이터 메모리(D-Cache).

### 5. WB (Write Back - 결과 저장)
- **주요 동작:** EX 단계에서 계산된 연산 결과나, MEM 단계에서 읽어온 메모리 데이터를 최종적으로 CPU 내부의 레지스터 파일에 덮어씁니다(Write).
- **핵심 하드웨어:** 레지스터 파일의 쓰기 포트(Write Port).

📢 **섹션 요약 비유:**
주문서 받기(IF) $\rightarrow$ 레시피 읽고 재료 꺼내기(ID) $\rightarrow$ 불 위에서 요리하기(EX) $\rightarrow$ 소스 창고 다녀오기(MEM) $\rightarrow$ 완성된 요리 접시에 담기(WB)라는 완벽한 5단계 분업 시스템입니다.

## Ⅱ. 파이프라인 레지스터 (Pipeline Register)

파이프라인이 섞이지 않고 돌아가기 위한 가장 중요한 하드웨어적 장치가 바로 단계와 단계 사이에 놓인 **파이프라인 레지스터(래치, Latch)**입니다.

- **구조:** `IF/ID`, `ID/EX`, `EX/MEM`, `MEM/WB` 라는 4개의 거대한 버퍼(Buffer)가 존재합니다.
- **역할:** 1클럭이 끝나는 순간, 각 단계에서 만들어진 결과물(데이터, 계산된 주소, 제어 신호 등)을 다음 단계로 넘겨주기 위해 이 레지스터에 임시로 저장합니다. 
- **독립성 보장:** 이 벽 덕분에 1번 명령어가 EX 단계에서 ALU를 쓰고 있을 때, 2번 명령어는 ID 단계에서 레지스터를 안심하고 읽을 수 있습니다.

📢 **섹션 요약 비유:**
컨베이어 벨트 작업자들 사이에 놓여있는 '바구니'입니다. 내가 나사를 다 조였으면 그 부품을 바구니(파이프라인 레지스터)에 담아두고 퇴근합니다. 다음 날(다음 클럭) 다음 작업자가 그 바구니에서 부품을 꺼내서 자기 작업을 시작합니다.

## Ⅲ. 단계의 일관성과 타임 낭비

파이프라인의 핵심 규칙은 **"모든 명령어는 무조건 이 5단계를 거쳐야 한다"**는 것입니다.

- `ADD` 명령어는 메모리 접근이 필요 없지만, 다른 명령어가 파이프라인의 리듬을 깨지 않도록 MEM 단계를 거치는 동안 1클럭을 그냥 흘려보냅니다.
- 만약 이 규칙을 깨고 `ADD`를 4단계 만에 끝내버리고 WB를 해버리면, 구조적 해저드(Structural Hazard)가 발생하거나 뒤따라오는 명령어와 타이밍이 엉켜 CPU가 붕괴합니다.
- **클럭 주파수의 결정:** 5개의 단계 중 **가장 시간이 오래 걸리는 단계(보통 MEM이나 EX)**의 시간에 맞춰 전체 시스템 클럭 속도가 결정됩니다.

📢 **섹션 요약 비유:**
5코스 요리를 먹을 때, 나는 샐러드(MEM)를 싫어하더라도 코스 순서를 깨고 바로 디저트(WB)를 달라고 할 수 없습니다. 샐러드 시간에는 그냥 멍하니 앉아 기다려야 전체 연회장(파이프라인)의 서빙 속도가 유지됩니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Instruction Pipelining]`: 5단계로 쪼개어 중첩 실행하는 전체 시스템 아키텍처
- `[Pipeline Stall (Bubble)]`: 5단계 중 특정 단계에서 문제가 생겨 파이프라인이 한 턴 쉬어가는 현상
- `[Harvard Architecture]`: IF 단계(I-Cache)와 MEM 단계(D-Cache)가 동시에 메모리에 접근할 수 있도록 분리한 메모리 구조
- `[Data Forwarding]`: EX 단계의 결과를 WB 단계까지 기다리지 않고 다음 명령어의 EX 단계로 바로 당겨 쓰는 최적화 기법
- `[Super-pipelining]`: 5단계를 넘어 10단계, 20단계로 더욱 잘게 쪼갠 파이프라인 방식

## 👶 어린이를 위한 3줄 비유 설명
1. 파이프라인 5단계는 자동차 공장의 5개 조립 구역이에요. 
2. 설명서 가져오기(IF) $\rightarrow$ 부품 찾기(ID) $\rightarrow$ 조립하기(EX) $\rightarrow$ 창고 다녀오기(MEM) $\rightarrow$ 포장하기(WB) 순서로 진행되죠.
3. 5명의 로봇이 자기 구역에서 딱 1분씩만 일하고 다음 구역으로 차를 넘기기 때문에, 1분마다 차가 한 대씩 완성되어 나온답니다!
