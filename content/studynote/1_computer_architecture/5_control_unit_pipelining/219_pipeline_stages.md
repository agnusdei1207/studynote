# 파이프라인 단계 (Pipeline Stages: IF, ID, EX, MEM, WB)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파이프라인 단계는 CPU가 하나의 명령어를 처리하는 전체 과정을 기능별로 독립된 하드웨어 유닛으로 분할한 **5개의 표준 물리적 실행 구역(IF, ID, EX, MEM, WB)**이다.
> 2. **가치**: 데이비드 패터슨(David Patterson) 등이 정립한 이 5단계 고전적 RISC 파이프라인 모델은, 메모리 접근과 ALU 연산을 완벽히 격리하여 **모든 단계가 정확히 1클럭 사이클 내에 동작하도록 균형(Balance)**을 맞춘 컴퓨터 구조의 마스터피스다.
> 3. **융합**: 각 단계 사이에는 래치(Latch)라 불리는 파이프라인 레지스터가 존재하여 이전 단계의 결과물을 다음 단계로 안전하게 넘겨주며, 컴파일러 최적화와 해저드(Hazard) 분석의 가장 기본적인 기준 단위가 된다.

+++

## Ⅰ. 파이프라인 5단계의 개요

### 1. 정의
- 명령어의 일생(Lifecycle)을 5개의 시간적/공간적 단계로 규격화한 것이다. (1) Instruction Fetch, (2) Instruction Decode, (3) Execute, (4) Memory Access, (5) Write Back으로 구성된다.

### 2. 등장 배경: '완벽한 분업화의 필요성'
- 파이프라인이 매끄럽게 돌아가려면 각 단계가 걸리는 시간이 엇비슷해야 한다. "메모리에 접근하는 시간"과 "덧셈을 하는 시간"을 한 단계에 몰아넣으면 그 단계만 병목이 생긴다. 아키텍트들은 "메모리는 MEM 단계에서만 건드리고, 연산은 EX 단계에서만 하자"는 철저한 역할 분담과 격리를 통해 가장 이상적인 5단 분리 모델을 고안해 냈다.

### 3. 💡 비유: '자동차 세차장 컨베이어'
- 5단계 파이프라인은 **'완벽하게 분업화된 자동 세차장'**과 같다.
- 1. 물 뿌리기(IF) $\to$ 2. 거품 내기(ID) $\to$ 3. 솔질하기(EX) $\to$ 4. 왁스 칠하기(MEM) $\to$ 5. 물기 닦기(WB). 각 구역에는 전담 직원이 있고, 차(명령어)는 멈추지 않고 1분마다 다음 구역으로 넘어간다.

- **📢 섹션 요약 비유**: 어떤 요리든 무조건 재료 준비, 다듬기, 불 조리, 식히기, 접시 담기의 5가지 공정으로 강제 통일시킨 완벽한 공장 라인입니다.

## Ⅱ. 5단계 세부 동작 및 메커니즘 (Deep Dive)

### 1. 각 단계별 핵심 임무
| 단계명 (약어) | 풀네임 | 핵심 수행 작업 | 관여하는 하드웨어 |
|:---:|:---|:---|:---|
| **IF** | **I**nstruction **F**etch | PC가 가리키는 주소에서 명령어 인출, PC 갱신 | PC, 명령어 캐시(L1i), Addr |
| **ID** | **I**nstruction **D**ecode| 명령어 해독, 제어 신호 생성, **레지스터 파일에서 데이터 읽기** | 디코더, 제어 유닛, 레지스터 파일 |
| **EX** | **EX**ecute | ALU를 통해 산술/논리 연산 수행, 또는 메모리 유효 주소 계산 | ALU, 분기 목적지 계산기 |
| **MEM** | **MEM**ory Access | 데이터 캐시(메모리)에서 데이터를 **읽거나(Load) 쓰는(Store)** 작업 수행 | 데이터 캐시(L1d) |
| **WB** | **W**rite **B**ack | 연산 결과나 메모리에서 읽은 데이터를 최종적으로 **레지스터에 저장** | 레지스터 파일 (Write Port) |

### 2. 데이터 흐름과 파이프라인 레지스터 (ASCII)
각 단계는 독립된 하드웨어이므로, 1클럭이 끝날 때마다 중간 결과물을 담아둘 바구니(버퍼)가 필요하다. 이를 파이프라인 레지스터(IF/ID, ID/EX, EX/MEM, MEM/WB)라고 한다.

```text
         [ IF ]       [ ID ]       [ EX ]       [ MEM ]       [ WB ]
          │    │       │    │       │    │       │     │       │    │
   PC ──▶ │ IF │ ────▶ │ ID │ ────▶ │ EX │ ────▶ │ MEM │ ────▶ │ WB │ ──▶ (Register)
   Cache  │ /  │ Regs  │ /  │ ALU   │ /  │ Cache │  /  │ Regs  │    │     (Write)
          │ ID │       │ EX │       │ MEM│       │ WB  │       │    │
          │    │       │    │       │    │       │     │       │    │
```

### 3. 심층 특징: 단계의 '휴식'
- 모든 명령어가 5단계를 다 쓰는 것은 아니다. 예를 들어 `ADD` 명령어는 메모리 접근이 필요 없으므로 **MEM 단계에서는 그냥 아무 일도 안 하고 쉰다(Idle)**. 하지만 파이프라인 리듬을 깨지 않기 위해 건너뛰지 않고 1클럭을 그냥 흘려보낸 채 WB 단계로 이동한다.

- **📢 섹션 요약 비유**: 빈 컵이 지나가더라도 컨베이어 벨트를 멈추거나 건너뛰지 않고, 원래 정해진 속도 그대로 무심하게 통과시키는 기계적 규칙성입니다.

## Ⅲ. 단계별 취약점과 해저드 발생 구간

각 단계는 특유의 취약점(Hazard)을 내포하고 있으며, 이를 극복하기 위한 설계가 필요하다.

| 파이프라인 단계 | 발생하기 쉬운 해저드 (Hazard) | 원인 및 시스템 영향 |
|:---:|:---|:---|
| **IF 단계** | **제어 해저드 (Control Hazard)** | 이전 JUMP 명령어가 EX 단계에서 목적지를 확정하기 전까지, IF가 엉뚱한 명령어를 가져올 수 있음 |
| **ID 단계** | **구조적 해저드 (Structural)** | WB 단계가 레지스터에 쓰는 것과 ID가 레지스터를 읽는 타이밍이 겹치면 충돌 (Half-clock 처리로 해결) |
| **EX 단계** | **데이터 해저드 (Data Hazard)** | 앞선 연산의 결과가 아직 WB 되지 않아 옛날 값을 읽고 계산하려 함 (Forwarding으로 해결) |
| **MEM 단계** | 캐시 미스 지연 (Stall) | 메모리에서 제때 데이터를 못 가져오면 전체 파이프라인 정지 |

- **📢 섹션 요약 비유**: 5개의 톱니바퀴 중 하나라도 뻑뻑해지거나 이물질(해저드)이 끼면 전체 시계의 움직임이 멈춰버리는 민감한 연결 구조입니다.

+++

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: 데이터 포워딩 (Data Forwarding / Bypassing)
- **상황**: `ADD R1, R2, R3` 직후에 `SUB R4, R1, R5`가 실행됨. SUB 명령어의 ID 단계는 R1을 원하지만, ADD 명령어는 아직 WB 단계에 도달하지 않아 R1은 옛날 값이다 (Data Hazard).
- **판단**: 파이프라인을 2클럭 멈추면 성능이 저하된다. 기술사는 하드웨어 설계 시, ADD 명령어의 **EX 단계 결과물(ALU 출력)**을 레지스터(WB)까지 가지 않고, 뒷 명령어의 **EX 단계(ALU 입력)로 다이렉트로 전선을 연결해서 쏴주는 '포워딩(전방 전달) 유닛'**을 추가해야 한다. 파이프라인 단계의 경계를 물리적으로 관통하는 이 배선이 현대 CPU 성능 유지의 핵심이다.

### 2. 하버드 아키텍처의 당위성
- **인사이트**: IF 단계는 명령어 메모리를 읽고, MEM 단계는 데이터 메모리를 읽는다. 만약 메모리가 하나(폰 노이만)라면 1클럭에 두 단계가 동시에 메모리에 접근하려다 충돌(구조적 해저드)이 난다. 이를 근본적으로 해결하기 위해 CPU 내부의 L1 캐시를 명령어용(L1i)과 데이터용(L1d)으로 분리하는 하버드 아키텍처가 5단 파이프라인의 필수 전제 조건이 되었다.

- **📢 섹션 요약 비유**: 앞 공정에서 막 만들어진 따끈따끈한 부품을 창고(레지스터)에 넣을 때까지 기다리지 않고, 뒷 공정 작업자에게 창문으로 휙 던져주는 센스(포워딩)입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **이상적인 1 IPC 달성**: 클럭당 1개의 명령어가 처리되는 RISC의 황금률 실현.
- **컴파일러의 예측성**: 5단계의 명확한 지연 시간(Latency)을 바탕으로 컴파일러가 소스 코드를 최적화(스케줄링)할 수 있는 완벽한 수학적 모델 제공.

### 2. 미래 전망
클래식 5단계는 MIPS나 초기 ARM에서 아름답게 동작했지만, 현대의 x86(Intel)이나 최신 ARM은 클럭 속도를 GHz 단위로 끌어올리기 위해 이 5단계를 14단계~20단계 이상으로 더욱 잘게 쪼갠 **슈퍼파이프라이닝(Superpipelining)** 구조를 갖게 되었다. 예를 들어 IF만 3단계, EX만 5단계로 나뉘는 식이다. 하지만 아무리 수십 단계로 쪼개지더라도 그 논리적 뼈대와 사상은 이 고전적인 'IF-ID-EX-MEM-WB' 5단계 모델을 벗어나지 않는다.

- **📢 섹션 요약 비유**: 5도막으로 나뉜 피자가 20도막으로 잘게 잘렸을 뿐, 피자를 구성하는 재료와 먹는 순서는 영원히 변하지 않는 것과 같습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **명령어 파이프라이닝**: 5단계 구분을 통해 달성하고자 하는 궁극적인 병렬 처리 개념.
- **데이터 해저드 (Data Hazard)**: EX와 WB 단계 간의 시간 차이 때문에 발생하는 고질적 문제.
- **MIPS 아키텍처**: 이 5단계 고전 파이프라인을 세상에서 가장 완벽하게 구현한 설계.

## 👶 어린이를 위한 3줄 비유 설명
1. 컴퓨터가 일하는 파이프라인은 정확히 **'5개의 방'**으로 나뉘어 있어요.
2. 1번 방(명령서 줍기), 2번 방(해석하기), 3번 방(진짜 계산하기), 4번 방(창고 다녀오기), 5번 방(수첩에 적기)이에요.
3. 데이터들은 한 방에서 딱 1초씩만 머물고 무조건 다음 방으로 밀려가기 때문에, 5개의 방 모두가 단 1초도 쉬지 않고 꽉 차서 돌아간답니다!
