# 230. 분기 지연 (Delayed Branch)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분기 지연 (Delayed Branch)은 파이프라인에서 조건 분기 명령어의 목적지가 아직 평가되지 않아 발생하는 제어 해저드(Stall)를 막기 위해, 분기문 바로 뒤에 오는 명령어(Delay Slot)를 '분기가 점프하든 안 하든 상관없이 무조건 실행되도록' 강제하는 초창기 하드웨어 최적화 기법이다.
> 2. **가치**: 값비싼 트랜지스터 면적을 차지하는 하드웨어 분기 예측기(Branch Predictor) 없이, 똑똑한 컴파일러(소프트웨어)가 코드 순서를 바꿔 유효한 명령어를 그 빈자리에 채워 넣음으로써 파이프라인의 멈춤을 공짜로 끌어올린 혁신적인 S/W-H/W 코디자인이다.
> 3. **융합**: 하지만 슈퍼스칼라 및 매우 깊어진 현대 파이프라인 환경에서는 채워야 할 지연 슬롯(Delay Slot)이 10여 개로 늘어나 컴파일러가 도저히 감당할 수 없게 되면서, 오늘날에는 하위 호환(Legacy)용으로만 남고 정교한 하드웨어 분기 예측기로 완전히 대체된 과거의 영광이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 분기 지연 (Delayed Branch)은 제어 해저드(Control Hazard)로 인한 파이프라인의 멈춤(Stall)이나 플러시(Flush)를 소프트웨어적으로 은닉하기 위한 기법이다. 분기 명령어 직후에 오는 1~2개의 명령어 공간(Delay Slot)을 점프 여부와 무관하게 무조건 순차 실행하도록 보장하는 아키텍처의 규칙이다.

- **필요성**: 파이프라인의 명령어 인출과 분기 경로 결정 간의 시간차로 발생하는 제어 해저드(Control Hazard)를 해소하여, 파이프라인 스톨(Stall)에 따른 CPI(Cycles Per Instruction) 증가 및 성능 저하를 최소화하기 위함이다. 하드웨어적인 분기 예측기가 부족하거나 단순한 아키텍처에서, 분기 명령어 다음의 빈 슬롯(Delay Slot)에 유효한 명령어를 배치하여 분기 처리 중에도 파이프라인이 멈추지 않고 연속적으로 명령어를 실행하도록 보장한다. 이를 통해 별도의 하드웨어 추가 비용 없이 컴파일러 최적화 기법만으로 파이프라인 효율성을 극대화하고 전체 시스템의 처리량(Throughput)을 획기적으로 개선할 수 있다.

- **💡 비유**: 길잡이가 "오른쪽으로 갈지 왼쪽으로 갈지 지도를 보고 3분 뒤에 알려주겠다"라고 했을 때, 뒤따르는 탐험대가 그 3분 동안 멍때리는 대신 "결과에 상관없이 일단 물 한 모금씩 마셔라!"라고 무조건 해야 할 유효한 행동을 미리 지시해 두는 꼼수와 같다.

- **등장 배경**: MIPS나 SPARC 같은 초기 1세대 RISC 아키텍처는 칩 내부의 하드웨어를 최대한 단순하게 유지하여 클럭 속도를 높이는 것이 지상 목표였다. 분기 예측기 같은 복잡하고 전기를 먹는 회로를 넣을 다이(Die) 면적이 부족했기 때문에, 하드웨어는 그냥 무식하게 순차적으로 명령어를 가져오고, 필연적으로 발생할 수밖에 없는 스톨(Stall)의 텅 빈 공간은 똑똑한 컴파일러(소프트웨어)가 억지로 메꾸도록 책임을 떠넘긴 철학의 산물이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 분기 지연의 동작 원리 (MIPS 5단 기준)

분기 명령어(`BEQ`)의 평가가 끝나는 EX 단계까지 필연적으로 1클럭의 지연이 발생한다. 이 1클럭을 어떻게 낭비 없이 때울까?

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         지연 분기 (Delayed Branch)의 파이프라인 동작 흐름             │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [기존 아키텍처 방식 - 멍청한 Stall 발생]                             │
  │ Inst 1 (BEQ) : [IF]-[ID]-[EX]-[ME]-[WB] (EX에서 비로소 분기 확정)  │
  │ Inst 2 (Stall)     [IF]-[Stall 1클럭 낭비 발생]                │
  │ Inst 3 (Target)          [IF]-[ID]-[EX] (새롭게 분기된 목적지)   │
  │                                                             │
  │ [지연 분기 적용 - Delay Slot 활용으로 무정지]                       │
  │ Inst 1 (BEQ) : [IF]-[ID]-[EX]-[ME]-[WB]                     │
  │ Inst 2 (Slot):      [IF]-[ID]-[EX]-[ME]-[WB] (무조건 실행 강행!!) │
  │ Inst 3 (Target):         [IF]-[ID]-[EX]-[ME]-[WB]           │
  │                                                             │
  │ * 작동 원리: Inst 1이 어디로 점프할지 EX 단계에서 끙끙 고민하고 있지만, │
  │             Inst 2(Delay Slot 위치)는 점프 여부와 아예 상관없이 무조건 │
  │             실행된다. 컴파일러가 이 Inst 2 자리에 유효한 명령어를 끌어 │
  │             와서 채워 넣는 것이 최적화의 핵심이다.                   │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 지연 분기 아키텍처에서 하드웨어 제어 유닛은 분기 명령어를 만나도 놀라지 않고 파이프라인을 절대 멈추거나 플러시(Flush)하지 않는다. 무조건 바로 밑에 있는 다음 주소(PC+4)의 명령어를 Fetch해서 실행해버린다. 이 특수한 한 칸의 자리를 **분기 지연 슬롯(Branch Delay Slot)**이라고 부른다. 만약 분기 조건이 만족되어 엉뚱한 목적지로 점프하게 되더라도, 이 슬롯에 들어온 명령어는 취소되지 않고 끝까지 정상 실행된다. (그래서 분기라는 행위 자체가 1명령어만큼 '지연'되어 실행된다고 부른다.)

- **📢 섹션 요약 비유**: 비행 중인 로켓트의 단 분리(점프) 스위치를 눌렀어도 즉시 분리되어 꺾이는 게 아니라, 관성(지연 슬롯) 때문에 1초 정도는 원래 궤도로 조금 더 맹렬히 날아간 뒤에야 비로소 궤도를 트는 물리적 법칙과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 지연 슬롯(Delay Slot) 채우기: 컴파일러의 3가지 스케줄링 마법

하드웨어가 빈 공간(지연 슬롯)을 만들어줬으니, 컴파일러는 어떻게든 이 빈칸을 아무 의미 없는 NOP가 아닌 유효한 명령어로 채워야 성능이 100% 나온다.

| 훔쳐 올 명령어 위치 | 컴파일러 최적화 동작 메커니즘 | 실행 조건 및 제약 | 스케줄링 성공 확률 |
|:---|:---|:---|:---|
| **1. 분기문 이전 (Before)** | 분기문보다 위에 있던 독립적인 명령어를 슬롯 위치로 끌어내림 | 그 명령어가 분기 조건에 영향을 주는 연산이면 절대 불가 | **가장 좋고 안전함 (Best)** |
| **2. 분기 목적지 (Target)** | 점프해서 도착할 목적지의 첫 명령어를 복사해서 당겨옴 | 점프를 안 할 경우(Fall-through) 치명적 부작용이 없어야 함 | 높음 (루프문에서 매우 효과적) |
| **3. 분기 실패지 (Fall-through)**| 점프 안 할 때 실행될 다음 순서 명령어를 끌어옴 | 점프를 할 경우(Taken) 레지스터 오염 등 부작용이 없어야 함 | 중간 |
| **4. NOP 강제 삽입** | 컴파일러가 아무 명령어도 못 찾으면 빈 깡통 삽입 | 컴파일러의 패배 선언 | 최악 (아무 성능 향상 없고 클럭만 날림) |

### 분기 예측 (Branch Prediction)과의 결정적 차이

| 비교 항목 | 지연 분기 (Delayed Branch) | 동적 분기 예측 (Branch Prediction) |
|:---|:---|:---|
| **해결 주체** | **컴파일러** (정교한 소프트웨어 스케줄링 최적화) | **하드웨어** (BHT, BTB 등 거대한 예측 회로) |
| **작동 시점** | 컴파일 타임 (프로그램이 기계어로 변환될 때 정적 결정) | 런타임 (프로그램이 CPU에서 실제 실행 중일 때 동적 갱신) |
| **하드웨어 비용** | 추가 회로 거의 없음 (초저비용 트랜지스터) | 수천만 개의 트랜지스터 다이 면적 요구 (초고비용) |
| **깊은 파이프라인**| 슬롯이 수십 개로 늘어나 다 못 채우고 성능 극락 (한계 명확) | 깊을수록 예측기 성능을 높여 스톨 완벽 방어 가능 (스케일업) |
| **적용 아키텍처** | MIPS, SPARC (초기 RISC 진영의 상징) | 현대 x86(인텔/AMD), ARM, 최신 RISC-V |

- **📢 섹션 요약 비유**: 지연 분기가 여행 출발 전에 컴파일러가 여행 가방의 빈틈을 테트리스처럼 꽉꽉 채워 넣는 꼼꼼하고 알뜰한 짐 싸기(소프트웨어)라면, 분기 예측은 여행 중 길이 막히면 하드웨어가 헬리콥터를 불러서 목적지로 그냥 날아가 버리는 무식하지만 강력한 자본주의적 해결책입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **초기 스마트폰 칩(ARM11)과 지연 분기의 황금기**
   2000년대 피처폰과 초기 스마트폰에 들어갔던 작고 단순한 RISC 프로세서들은 배터리를 아끼기 위해 극단적으로 칩 다이(Die) 크기와 누설 전류를 줄해야 했다. 트랜지스터 수만 개가 들어가는 고성능 분기 예측기는 사치였다. 대신 지연 분기를 채택하여 하드웨어 전력 소모를 사실상 0으로 유지하면서도, 똑똑한 컴파일러(GCC) 최적화를 통해 80% 이상의 지연 슬롯을 무사히 유효 명령어로 채워 넣어 전력 대 성능비(Perf/W)의 기적을 달성했다.

2. **레거시 짐 덩어리(Legacy Burden)가 된 MIPS의 몰락**
   시간이 흘러 클럭을 높이기 위해 파이프라인이 5단에서 10단, 15단으로 무자비하게 깊어졌다. 분기가 확정될 때까지의 딜레이가 1클럭에서 5클럭으로 늘어났다. 지연 분기 아키텍처는 이 5개의 텅 빈 딜레이 슬롯을 채워야 하지만, 컴파일러는 인간이 짠 코드의 좁은 스코프 안에서 도저히 완벽히 독립적인 5개의 유효 명령어를 찾을 수 없어 결국 깡통 NOP만 5개씩 가득 채우게 되었다. 결국 최신 MIPS 프로세서마저 하드웨어 분기 예측기를 도입했지만, 과거 바이너리와의 호환성(Legacy)을 지키기 위해 "명령어 1개는 무조건 지연 실행해야 한다"는 해괴한 구시대적 규칙을 평생 안고 가야 하는 저주받은 신세가 되었다.

### 도입 체크리스트
- **기술적**: 우리가 새롭게 바닥부터 설계하는 코어의 파이프라인이 7단 이하의 매우 얕은(Shallow) IoT 센서나 임베디드 전용인가? 그렇다면 면적과 전력 절감을 위해 하드웨어 예측기를 빼고 지연 분기를 고려할 수 있다.

### 안티패턴
- **지연 슬롯(Delay Slot) 내부에 분기문 또 쑤셔 넣기 (Nested Branch)**
  지연 분기가 무조건 실행되는 공간(Delay Slot) 안에 실수로, 혹은 컴파일러의 꼼수 버그로 또 다른 제어 명령어(점프나 함수 호출)를 겹쳐서 넣는 어셈블리 코드 작성. 프로세서의 하드웨어 상태 머신은 지연 슬롯 내부에서 PC가 다시 한번 튀어버리는 미친 상황을 전혀 고려하지 않고 단순하게 설계되었기 때문에, 이 코드가 실행되는 순간 CPU가 어느 주소로 복귀해야 할지 길을 잃고 영원히 다운되어버리는 끔찍한 언디파인드 비헤이비어(Undefined Behavior)가 발생한다.

- **📢 섹션 요약 비유**: 지연 분기는 가난했던 학생 시절 빈 지갑을 소프트웨어적인 꼼수로 메워준 고맙고 사랑스러운 친구였지만, 어른이 되어 거대한 시스템을 굴릴 때는 버리기 힘든 낡은 습관(레거시)이 되어 내 발목을 계속 잡는 옛 연인과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 파이프라인 최적화 상태 | 분기 해저드 (Stall) 발생 시 | 지연 분기 (Delay Slot 100% 채움) | 성능 개선 효과 |
|:---|:---|:---|:---|
| **정량 (CPI)** | 분기 만날 때마다 1~2클럭 낭비 (CPI 1.2 상승) | **0클럭 지연** (이상적 파이프라인) | 전체 파이프라인 IPC 20% 이상 공짜 상승 |
| **정량 (칩 면적/원가)** | 복잡한 Stall 제어 및 플러시 로직 필요 | 완벽한 무정지 순차 처리 유지로 하드웨어 간소화 | 하드웨어 설계 다이 면적(PPA) 극도 절감 |
| **정성 (소프트웨어)**| 컴파일러 개발 난이도 낮음 (그냥 기계어 번역) | 코드 의존성을 분석하는 컴파일러 설계 난이도 대폭 상승 | **H/W의 복잡한 부담을 S/W로 떠넘기는 RISC 철학의 완벽한 실현** |

### 미래 전망
- **지연 분기의 영구적 퇴출과 RISC-V의 단호한 결단**: 2010년대에 백지에서 새롭게 설계된 위대한 오픈소스 아키텍처인 RISC-V는, 과거 MIPS와 SPARC가 겪었던 '깊은 파이프라인에서의 지연 분기 레거시의 저주'를 뼈아픈 거울로 삼았다. 그 결과 RISC-V는 명령어 집합(ISA) 설계 첫 단계부터 지연 분기(Delayed Branch)라는 개념을 아예 명시적으로 금지하고 과감하게 퇴출시켜 버렸다. 현대 반도체 트랜지스터는 충분히 작고 싸기 때문에, 제어 해저드의 방어는 100% 정교한 하드웨어 분기 예측기(Branch Predictor)가 전담하는 방향으로 완전히 굳어졌다. 지연 분기는 컴퓨터 구조의 역사책에 '위대한 소프트웨어적 해결책'으로만 영광스럽게 남게 될 것이다.

- **📢 섹션 요약 비유**: 자동차 에어컨을 켜기 위해 엔진룸을 열고 손으로 벨트를 직접 돌려주던 낭만의 꼼수 시대(소프트웨어 개입)는 영원히 끝났습니다. 이제는 트랜지스터라는 넘쳐나는 자본을 쏟아부어 버튼 하나면 AI가 온도를 알아서 맞춰주는 스마트 시대(하드웨어 분기 예측)로 완전히 진화했습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **제어 해저드 (Control Hazard)** | 분기 지연이라는 기상천외한 소프트웨어적 해결책이 세상에 나오게 된 근본적인 문제(조건 분기로 인한 파이프라인 지연)다. |
| **명령어 스케줄링 (Instruction Scheduling)** | 텅 빈 1클럭의 지연 슬롯(Delay Slot)을 유효한 독립 명령어로 꽉꽉 채워 넣어 기포(Bubble)를 막아내는 컴파일러의 핵심 최적화 기술이다. |
| **분기 예측 (Branch Prediction)** | 지연 분기의 얕은 파이프라인 한계를 박살 내고 대체해버린, 현대 CPU의 필수적이고 거대한 하드웨어 통계적 예지 기술이다. |
| **NOP (No Operation)** | 똑똑한 컴파일러마저 지연 슬롯을 도저히 유효한 명령어로 채우지 못했을 때, 울며 겨자 먹기로 끼워 넣어야 하는 텅 빈 기포 명령어다. |
| **RISC (Reduced Instruction Set Computer)** | 하드웨어 물리적 회로를 최대한 단순하게 만들고, 그로 인한 패널티는 소프트웨어 컴파일러에 100% 떠넘기겠다는 철학으로, 지연 분기를 탄생시킨 아키텍처 사상이다. |

2. 그래서 똑똑한 코치님(컴파일러)이 "심판이 알려줄 때까지 기다리는 3초 동안, 제자리에 서서 발목 스트레칭이나 하고 있어라!"라고 명령을 내렸어요.
3. 어차피 점프하든 안 하든 경주 끝나기 전에 꼭 해야 할 스트레칭을 그 버려지는 시간에 미리 해버리니까, 1초도 낭비 없이 완벽하게 달릴 수 있게 된 거예요! 이걸 '분기 지연'이라고 한답니다.
