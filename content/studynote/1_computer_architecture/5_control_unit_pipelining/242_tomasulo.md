# 토마술로 알고리즘 (Tomasulo's Algorithm)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 토마술로 알고리즘은 1967년 IBM의 로버트 토마술로가 개발한 하드웨어 기반 동적 스케줄링 알고리즘으로, **예약역(Reservation Station), 레지스터 리네이밍, 공용 데이터 버스(CDB)**를 결합하여 비순차 실행을 구현한다.
> 2. **가치**: WAW와 WAR 해저드를 하드웨어적으로 제거하고, RAW 해저드의 지연을 최소화하여 명령어 수준 병렬성(ILP)을 극대화한다.
> 3. **융합**: 초기 IBM 360/91 메인프레임에서 시작되어 현대 x86, ARM 등 모든 고성능 CPU의 비순차 실행 엔진 설계에 지대한 영향을 미쳤다.

---

## Ⅰ. 토마술로 알고리즘의 개요

### 1. 정의
- 토마술로 알고리즘은 명령어들이 데이터 의존성을 만족하는 한, 원래 순서와 상관없이 실행될 수 있도록 동적으로 스케줄링하는 하드웨어 기법이다. 핵심 아이디어는 **'피연산자가 준비되면 즉시 실행하라'**는 것이다.

### 2. 등장 배경: 정적 스케줄링의 한계
- 초기 프로세서는 컴파일러가 정적으로 명령어 순서를 재배치했다. 하지만 캐시 미스 등 런타임 지연을 컴파일 타임에 예측하기 어렵다. 토마술로는 **런타임에 하드웨어가 스스로 최적의 실행 순서를 결정**하는 동적 스케줄링을 제안했다.

### 3. 💡 비유: '스마트한 식당 주방'
- 토마술로 알고리즘은 **'주방장이 주문서(명령어)를 받자마자 재료(피연산자)가 있는 것부터 바로 조리하는 식당'**과 같다.
- 정해진 순서대로 요리하면 3번 주문이 재료가 늦어질 때 4, 5번도 같이 기다려야 한다. 하지만 재료 준비된 4번부터 먼저 조리하면 전체 대기 시간이 줄어든다.

- **📢 섹션 요약 비유**: 모든 요리사가 공용 창고(CDB)를 통해 재료를 공유받고, 준비된 재료부터 즉시 요리하는 효율적인 주방 관리 시스템입니다.

---

## Ⅱ. 핵심 구성요소 및 동작 원리 (Deep Dive)

### 1. 핵심 구성요소
1. **예약역 (Reservation Station)**: 각 실행 유닛 앞에 위치한 버퍼. 피연산자가 준비될 때까지 명령어를 보관.
2. **공용 데이터 버스 (CDB, Common Data Bus)**: 모든 실행 결과를 방송하는 버스. 예약역과 레지스터 파일이 이를 모니터링.
3. **레지스터 리네이밍**: 아키텍처 레지스터를 물리 레지스터나 예약역 슬롯으로 매핑하여 가짜 의존성 제거.
4. **로드/스토어 버퍼**: 메모리 연산을 위한 별도의 예약역.

### 2. 동작 단계
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │              토마술로 알고리즘 동작 흐름                           │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  1. Issue (발급)                                                │
    │     - 명령어를 FIFO 큐에서 꺼내 예약역으로 보냄                   │
    │     - 피연산자가 레지스터에 있으면 값 복사, 없으면 태그 저장       │
    │                                                                 │
    │  2. Execute (실행)                                              │
    │     - 예약역에서 모든 피연산자가 준비되면 실행 유닛으로 발급       │
    │     - 여러 실행 유닛이 병렬로 동작 가능                          │
    │                                                                 │
    │  3. Write Result (결과 기록)                                     │
    │     - 실행 완료 후 결과를 CDB에 방송                             │
    │     - 대기 중인 예약역과 레지스터 파일이 결과를 캡처              │
    │                                                                 │
    │  * 핵심: CDB가 단일 포인트로 모든 결과를 전파                     │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. 해저드 해결 메커니즘
| 해저드 유형 | 설명 | 토마술로 해결 방법 |
|:---|:---|:---|
| **RAW** (Read After Write) | 진짜 의존성 | 피연산자 준비될 때까지 예약역에서 대기 |
| **WAR** (Write After Read) | 이름 의존성 | 레지스터 리네이밍으로 제거 |
| **WAW** (Write After Write) | 이름 의존성 | 레지스터 리네이밍으로 제거 |

- **📢 섹션 요약 비유**: 같은 이름을 가진 사람(레지스터)이 여러 명 있어, 착각하지 않고 정확히 각자에게 편지를 전달하는 시스템입니다.

## Ⅲ. 토마술로 vs 정적 스케줄링 비교

| 비교 항목 | 정적 스케줄링 (VLIW) | 토마술로 (동적 스케줄링) |
|:---|:---|:---|
| **스케줄링 시점** | 컴파일 타임 | 런타임 |
| **하드웨어 복잡도** | 낮음 | 높음 |
| **코드 밀도** | 낮음 (NOP 삽입) | 높음 |
| **이식성** | 낮음 (ISA 의존) | 높음 |
| **대표 아키텍처** | Itanium, DSP | x86, ARM Cortex-A |

---

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: CDB 병목 현상
- CDB는 모든 결과가 지나가는 단일 경로다. 실행 유닛이 많아지면 CDB 대역폭이 병목이 된다.
- **해결**: 다중 CDB 또는 결과 버스(Result Bus) 확장. 현대 CPU는 여러 개의 결과 포트를 가진다.

### 2. 현대 아키텍처로의 진화
- 순수 토마술로 알고리즘은 ROB(재주문 버퍼)가 없어 정밀 예외(Precise Exception) 처리가 어려웠다.
- 현대 CPU는 **토마술로의 예약역 + ROB + 물리 레지스터 파일**을 결합한 하이브리드 구조를 사용한다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **ILP 극대화**: 순차 실행 대비 2~4배 이상 명령어 처리량 향상.
- **투명성**: 프로그래머는 순차 실행 코드를 작성해도 하드웨어가 자동 최적화.

### 2. 미래 전망
- 토마술로의 핵심 아이디어(동적 스케줄링, 리네이밍)는 모든 고성능 CPU의 기본 원리가 되었다.
- AI 가속기(NPU)에서도 유사한 스케줄링 기법이 적용되고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. 토마술로 알고리즘은 **'각 반마다 다른 과제(명령어)를 하는 학교'** 같아요.
2. 선생님(하드웨어)이 숙제(피연산자)를 다 해온 친구부터 발표시켜서 수업(실행)이 막히지 않게 해요.
3. 덕분에 한 친구가 늦어도 다른 친구들은 계속 공부할 수 있어서 전체 성적이 올라간답니다!
