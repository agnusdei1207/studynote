# 실행 사이클 (Execute Cycle)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 실행 사이클 (Execute Cycle)은 명령어 사이클의 정점으로, **해독된 명령어의 연산 코드(Opcode)에 따라 실제 산술·논리 연산이 수행되거나 데이터가 이동하는 단계**이다.
> 2. **가치**: 명령어의 종류(산술, 논리, 데이터 전송, 제어 등)에 따라 데이터패스 내의 하드웨어 유닛들이 동적으로 활성화되며, 연산의 최종 결과를 레지스터나 메모리에 저장하여 프로그램의 실질적인 진보를 이끌어낸다.
> 3. **융합**: ALU(산술 논리 연산 장치)와 제어 유닛이 가장 긴밀하게 협력하는 단계이며, 실행 결과에 따라 상태 레지스터의 플래그가 갱신되어 다음 실행 경로에 영향을 미친다.

---

## Ⅰ. 실행 사이클의 정의 및 동작 유형

### 1. 정의
- CPU가 해독 단계에서 생성된 제어 신호를 바탕으로 데이터패스를 가동하여, 사용자가 의도한 실제 작업을 완수하는 주기이다.

### 2. 명령어 유형별 주요 동작
- **데이터 전송 (Load/Store/Move)**: 레지스터와 메모리, 혹은 레지스터 간에 데이터를 이동시킨다.
- **산술/논리 연산 (Add/Sub/And/Or)**: ALU를 가동하여 수치적 혹은 논리적 계산을 수행한다.
- **제어 흐름 (Jump/Branch/Call)**: 조건에 따라 프로그램 카운터(PC)의 값을 변경하여 실행 순서를 꺾는다.

### 3. 💡 비유: '요리사의 본격적인 조리'
- 실행 사이클은 **'해석된 조리법(해독 결과)에 따라 가스불을 켜고, 칼로 재료를 썰고, 냄비에 담아 요리를 완성하는 실제 행동 단계'**와 같다.
- 재료를 준비하고 메뉴를 이해하는 과정(인출/해독)이 아무리 훌륭해도, 이 조리 과정(실행)이 없으면 요리(결과)는 나오지 않는다.

---

## II. 실행 사이클의 단계별 마이크로 오퍼레이션 (Deep Dive)

명령어의 종류에 따라 소요 시간과 마이크로 오퍼레이션이 상이하다.

### 1. 예시: ADD R1, X (메모리 X의 값을 읽어 R1에 더함)
- **T3**: $MAR \leftarrow IR(Addr)$
  - 명령어 레지스터에 담긴 메모리 주소를 MAR로 보낸다.
- **T4**: $MBR \leftarrow M[MAR]$
  - 메모리에서 데이터를 읽어 MBR에 저장한다.
- **T5**: $R1 \leftarrow R1 + MBR$
  - R1의 기존 값과 MBR의 값을 ALU에서 더하여 다시 R1에 저장한다.

### 2. 예시: JMP X (X번지로 무조건 점프)
- **T3**: $PC \leftarrow IR(Addr)$
  - 명령어에 적힌 목표 주소를 프로그램 카운터에 즉시 로드한다. (이후 인출 사이클은 X번지부터 시작됨)

### 3. ASCII 구조 다이어그램
```text
    [ Control Unit ] ──(Opcode Signals)──▶ [ Data Path ]
                                              │
           ┌──────────────────────────────────┼──────────────────────────────────┐
           │                                  ▼                                  │
    ┌─────────────┐                  ┌──────────────────┐                  ┌─────────────┐
    │ Register File│ ──(Read Data)──▶│       ALU        │ ──(Write Res)──▶│ Memory / Reg│
    └─────────────┘                  │ (Math / Logic)   │                  └─────────────┘
                                     └────────┬─────────┘
                                              │
                                              ▼
                                     [ Status Flags ] ──▶ [ Z ] [ C ] [ V ] [ S ]
```

- **📢 섹션 요약 비유**: 지휘자의 손짓(제어 신호)에 따라 연주자들이 악기를 불고 두드려(하드웨어 가동) 실제 소리(연산 결과)를 만들어내는 오케스트라의 연주 실황과 같습니다.

## Ⅲ. 실행 사이클과 성능 쟁점

명령어의 복잡도에 따라 실행 사이클의 효율이 결정된다.

| 구분 | 특징 | 영향 |
|:---|:---|:---|
| **고정 주기 실행** | 모든 명령어가 동일한 클럭 소모 (RISC) | 파이프라이닝 설계가 단순하고 빠름 |
| **가변 주기 실행** | 명령어마다 클럭 소모 상이 (CISC) | 복잡한 연산을 한 번에 끝내지만 설계가 어려움 |
| **병렬 실행** | 여러 유닛을 동시에 가동 (Superscalar) | 한 사이클에 여러 명령어의 실행 단계 동시 완료 |

- **📢 섹션 요약 비유**: 요리마다 걸리는 시간이 다르면 배식(파이프라인)이 꼬이기 쉬우므로, 가급적 조리 시간을 일정하게 맞추거나(RISC) 여러 명의 요리사를 두는(병렬 실행) 전략이 필요합니다.

---

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 부동소수점 연산 최적화
- **상황**: 3D 그래픽 렌더링 중 부동소수점 곱셈 연산이 너무 많아 프레임 드랍 발생.
- **판단**: 일반 ALU에서의 실행 사이클은 너무 느리다. 전용 **FPU (Floating Point Unit)**를 가동하도록 명령어를 최적화하거나, 한 번의 실행 사이클에 여러 데이터를 처리하는 **SIMD** 명령어를 사용하여 실행 효율을 수십 배 높인다.

### 2. 기술사적 인사이트: 비순차 실행 (Out-of-Order Execution)
- 기술사라면 "실행 사이클은 반드시 순서대로 일어날 필요가 없다"는 현대적 아키텍처 관점을 제시해야 한다. 해독이 끝난 명령어들 중, 필요한 데이터가 준비된 것부터 먼저 실행(Data-flow 기반)하고 나중에 순서를 맞추는 **비순차 실행** 기술이 현대 CPU 성능의 핵심임을 강조해야 한다.

- **📢 섹션 요약 비조**: 재료가 먼저 준비된 요리부터 불에 올리는 영리한 주방 관리(비순차 실행)가, 주문 순서만 고집하는 것보다 훨씬 빠른 회전율을 보장합니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **목적의 달성**: 소프트웨어 지시 사항의 물리적 완성.
- **상태 변화의 누적**: 레지스터와 메모리 상태 갱신을 통한 프로그램 로직 진행.

### 2. 미래 전망
앞으로는 실행 사이클 자체가 하드웨어가 아닌 소프트웨어 정의 방식(FPGA 등)으로 변하거나, 데이터의 성격에 따라 연산 유닛을 실시간으로 최적화하는 **가변형 실행 엔진**이 보편화될 것이다. 실행 사이클은 컴퓨터가 존재 이유를 증명하는 가장 역동적이고 실질적인 단계이다.

- **📢 섹션 요약 비유**: 수많은 설계도와 계획 끝에 마침내 망치질을 시작하여 집을 짓는, 연산의 가장 뜨겁고 치열한 현장입니다.
