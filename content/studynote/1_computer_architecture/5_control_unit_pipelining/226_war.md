# 226. WAR (Write After Read)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 읽기 후 쓰기, 반의존성 (Anti-dependency)
> 2. **가치**: 파이프라인에서는 거의 발생 안 함, 비순차에서 문제
> 3. **융합**: 레지스터 리네이밍, 비순차 실행

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

WAR(Write After Read)는 **한 명령어가 데이터를 읽은 후, 다음 명령어가 그 위치에 쓰려 할 때 발생하는 반의존성**이다. 올바른 순서를 보장하기 위해 읽기가 완료된 후에 쓰기가 이루어져야 한다.

### 💡 비유: 신문 읽기

WAR는 **신문 읽기**와 같다. A가 신문을 읽고 있는데(Read) B가 신문을 가져가서 새 신문으로 바꾸면(Write) A는 잘린 기사를 읽게 된다. A가 다 읽을 때까지 B는 기다려야 한다.

### WAR 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                WAR 구조                                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【WAR 정의】                                                          │
│  ──────────────────                                                  │
│  명령어 i: ... ← R1        (Read from R1)                            │
│  명령어 j: R1 ← ...        (Write to R1)                             │
│  j가 R1에 쓰려는데 i가 아직 R1을 다 안 읽음                              │
│                                                                     │
│  【발생 예시】                                                         │
│  ──────────────────                                                  │
│  SUB R4, R1, R5   ; R4 ← R1 - R5  (Read R1)                         │
│  ADD R1, R2, R3   ; R1 ← R2 + R3  (Write R1)                        │
│                                                                     │
│  → ADD가 R1에 쓰기 전에 SUB가 R1을 읽어야 함                             │
│  → 잘못되면 SUB가 새로운 R1 값을 읽을 수 있음                             │
│                                                                     │
│  【파이프라인에서의 WAR】                                               │
│  ────────────────────────                                            │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  SUB R4, R1, R5   ; Read R1                                      │ │
│  │  ADD R1, R2, R3   ; Write R1                                     │ │
│  │                                                               │ │
│  │  클럭  IF    ID    EX    MEM   WB                                 │ │
│  │  ────  ────  ────  ────  ────  ────                               │ │
│  │   1    SUB                                                       │ │
│  │   2    ADD   SUB        ← SUB: ID에서 R1 읽기                      │ │
│  │   3          ADD   SUB   ← SUB: EX 실행                           │ │
│  │   4                ADD   SUB                                     │ │
│  │   5                      ADD   SUB  ← ADD: WB (R1 기록)            │ │
│  │                               ↑                                   │ │
│  │                         SUB WB (R4 기록)                          │ │
│  │                                                               │ │
│  │  → 파이프라인에서는 ID 단계에서 읽기가 일어남                          │ │
│  │  → ADD가 SUB보다 나중에 ID 진입                                     │ │
│  │  → SUB가 먼저 R1 읽음 → WAR 해저드 없음                              │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【왜 파이프라인에서는 WAR이 거의 없는가?】                               │
│  ───────────────────────────────────────────                         │
│  • ID 단계에서 레지스터 읽기가 일어남                                     │
│  • 파이프라인은 순차적으로 ID 진입                                       │
│  • 읽기 명령어가 항상 먼저 레지스터 읽음                                   │
│  • 쓰기는 WB 단계에서만 발생                                             │
│  → 순차적 파이프라인에서 WAR 불가능                                       │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  WAR이 문제가 되는 경우:                                               │ │
│  │  ──────────────────────                                              │ │
│  │  1. 비순차 실행 (Out-of-Order)                                       │ │
│  │     - 쓰기 명령어가 읽기 명령어보다 먼저 실행될 수 있음                   │ │
│  │  2. 레지스터 포트 경합                                                 │ │
│  │     - 읽기와 쓰기가 동시에 같은 포트 사용                               │ │
│  │  3. 메모리 일관성                                                     │ │
│  │     - 멀티코어에서 메모리 접근 순서                                    │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. WAR 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                WAR 상세                                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【반의존성 (Anti-dependency)】                                        │
│  ────────────────────────────────                                    │
│  • 이름 의존성 (Name Dependency)의 일종                                │
│  • 실제 데이터 의존성이 아님                                            │
│  • 레지스터 이름이 같아서 발생                                           │
│  • 레지스터 리네이밍으로 제거 가능                                        │
│                                                                     │
│  【비순차 실행에서의 WAR】                                              │
│  ────────────────────────────                                        │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  SUB R4, R1, R5   ; R1 읽기 (캐시 미스로 지연)                      │ │
│  │  ADD R1, R2, R3   ; R1 쓰기 (빠르게 실행 가능)                      │ │
│  │  MUL R6, R1, R7   ; R1 읽기 (새 R1 값 사용)                        │ │
│  │                                                               │ │
│  │  비순차 실행:                                                       │ │
│  │  1. ADD 먼저 실행 (R1 ← R2 + R3)                                   │ │
│  │  2. MUL 실행 (새 R1 사용)                                          │ │
│  │  3. SUB 실행 (원래 R1 읽어야 함!)                                   │ │
│  │                                                               │ │
│  │  → SUB가 ADD 후에 실행되면 잘못된 R1 읽음                            │ │
│  │  → WAR 위반                                                        │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【레지스터 리네이밍으로 해결】                                          │
│  ──────────────────────────                                          │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  원본 코드:                                                         │ │
│  │  SUB R4, R1, R5   ; R1 사용                                        │ │
│  │  ADD R1, R2, R3   ; R1 정의                                        │ │
│  │  MUL R6, R1, R7   ; 새 R1 사용                                     │ │
│  │                                                               │ │
│  │  리네이밍 후:                                                        │ │
│  │  SUB R4, R1, R5   ; R1 사용                                        │ │
│  │  ADD R1', R2, R3  ; R1' 정의 (새 레지스터)                          │ │
│  │  MUL R6, R1', R7  ; R1' 사용                                       │ │
│  │                                                               │ │
│  │  → WAR 제거: R1과 R1'은 다른 레지스터                                │ │
│  │  → ADD와 MUL이 SUB 전에 실행 가능                                   │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【Tomasulo 알고리즘】                                                  │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • 예약역 (Reservation Station) 사용                               │ │
│  │  • 레지스터 대신 태그 사용                                           │ │
│  │  • WAR과 WAW 모두 해결                                              │ │
│  │  • 비순차 실행 가능                                                  │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  WAR vs RAW vs WAW 비교:                                             │ │
│  │  ────────────────────────                                            │ │
│  │  종류   의존성       순차 파이프라인   비순차 실행   해결                 │ │
│  │  ────   ────────    ─────────────   ──────────   ────                 │ │
│  │  RAW   진의존성      발생             발생         포워딩               │ │
│  │  WAR   반의존성      거의 없음        발생         리네이밍              │ │
│  │  WAW   출력의존성    거의 없음        발생         리네이밍              │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

### 과목 융합 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                WAR × 타 과목 융합                                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【컴퓨터 구조】                                                      │

│  • 레지스터 리네이밍: WAR 해결                                          │

│  • 예약역: 의존성 추적                                                  │
│                                                                     │

│  • 레지스터 할당: WAR 최소화                                            │
│  • 코드 스케줄링: WAR 회피                                             │
│  • 정적 분석: 이름 의존성 제거                                           │
│                                                                     │
│  【병렬 프로그래밍】                                                     │
│  • 메모리 일관성: WAR 보장                                              │
│  • 동기화: 읽기-쓰기 순서                                               │
│  • 락: WAR 보호                                                       │
│                                                                     │
│  【데이터베이스】                                                        │
│  • 트랜잭션: 읽기-쓰기 충돌                                             │
│  • 동시성 제어: WAR 관리                                                │
│  • 직렬화 가능성: 순서 보장                                              │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  WAR 제거 기법:                                                       │ │
│  │  ──────────────────                                                  │ │
│  │  기법              설명                    적용                       │ │
│  │  ───────────────  ──────────────────     ───────                    │ │
│  │  레지스터 리네이밍   물리 레지스터 추가        비순차 실행                │ │
│  │  정적 리네이밍      컴파일러가 레지스터 변경    VLIW                     │ │
│  │  소프트웨어 파이프라닝 루프 의존성 제거       컴파일러                    │ │
│  │  루프 분할         의존성 없는 반복 분리       최적화                    │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### WAR 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                WAR 분석                                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【레지스터 리네이밍 확인】                                             │
│  ────────────────────────                                            │
│  $ cat /proc/cpuinfo | grep "register rename"                       │
│  → 현대 CPU는 레지스터 리네이밍 지원                                     │
│                                                                     │
│  【비순차 실행 분석】                                                   │
│  ────────────────────────                                            │
│  $ perf stat -e uops_issued.any,uops_executed.out_of_order ./prog  │
│  → 비순차 실행 통계 확인                                               │
│                                                                     │
│  【WAR 최소화 코드】                                                    │
│  ──────────────────                                                  │
│  # Before (WAR 발생 가능)                                             │
│  for (i = 0; i < n; i++) {                                          │
│      a = b[i];        ; a 읽기                                       │
│      b[i] = c[i];     ; b 쓰기 → WAR                                 │
│      c[i] = a;        ; a 사용                                       │
│  }                                                                  │
│                                                                     │
│  # After (WAR 제거)                                                   │
│  for (i = 0; i < n; i++) {                                          │
│      temp = b[i];     ; 임시 변수 사용                                │
│      b[i] = c[i];     ; WAR 없음                                     │
│      c[i] = temp;     ;                                              │
│  }                                                                  │
│                                                                     │
│  【컴파일러 최적화】                                                    │
│  ──────────────────                                                  │
│  # GCC 의존성 분석                                                    │
│  $ gcc -O3 -ftree-vectorize -fopt-info-vec ./prog.c                 │
│  → 의존성 분석 및 벡터화 정보                                           │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  WAR 감지 및 해결:                                                    │ │
│  │  ──────────────────                                                  │ │
│  │  1. 의존성 분석 도구 사용                                             │ │
│  │  2. 레지스터 리네이밍 확인                                            │ │
│  │  3. 임시 변수 사용                                                   │ │
│  │  4. 루프 분할 또는 융합                                               │ │
│  │  5. 컴파일러 최적화 활용                                              │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• WAR: Write After Read, 읽기 후 쓰기 반의존성
• 이름 의존성: 실제 데이터 의존성이 아님
• 파이프라인: 순차 실행 시 거의 발생 안 함
• 비순차 실행: WAR이 문제가 됨
• 해결: 레지스터 리네이밍 (물리 레지스터 분리)
• 현대 CPU: 하드웨어 리네이밍으로 자동 해결
• 컴파일러: 정적 리네이밍으로 최적화
```
