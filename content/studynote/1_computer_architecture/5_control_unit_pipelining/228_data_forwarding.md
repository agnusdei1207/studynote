# 228. 데이터 포워딩 (Data Forwarding / Bypassing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 포워딩 (Data Forwarding, 우회 전달)은 파이프라인에서 앞선 명령어의 연산 결과가 레지스터 파일에 최종 기록(WB)될 때까지 기다리지 않고, ALU 연산이 끝나는 즉시 다음 명령어의 ALU 입력단으로 직접 넘겨주는(Bypass) 하드웨어 우회 배선 기술이다.
> 2. **가치**: 파이프라인에서 가장 흔하게 발생하는 RAW(Read After Write) 데이터 해저드의 80% 이상을 파이프라인 정지(Stall) 없이 0클럭 지연으로 완벽하게 해결하여, 전체 시스템의 CPI를 이상적인 1.0에 가깝게 방어한다.
> 3. **융합**: 하지만 메모리 읽기(Load) 명령어 직후의 데이터 의존성은 물리적으로 포워딩할 수 없는 타이밍 한계(Load-Use Hazard)를 가지며, 이는 컴파일러의 명령어 스케줄링(Instruction Scheduling) 최적화와 결합되어 소프트웨어적으로 극복된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 데이터 포워딩 (Data Forwarding, 또는 Bypassing)은 파이프라인의 뒷단계 래치(EX/MEM, MEM/WB)에 임시 보관된 최신 연산 결과값을, 레지스터 파일에 기록(WB)될 때까지 기다리지 않고 다음 명령어의 연산기(ALU) 입력으로 즉시 우회 전달하는 하드웨어 메커니즘이다.

- **필요성**: 데이터 포워딩은 파이프라인에서 가장 빈번하게 발생하는 **RAW(Read After Write) 데이터 해저드로 인한 성능 저하를 원천 차단**하기 위해 필수적이다. 파이프라이닝의 본질은 중첩 실행이지만, 앞 명령어의 결과를 뒤 명령어가 즉시 필요로 하는 의존성 때문에 스톨(Stall)이 발생하면 처리량(Throughput)이 급격히 떨어진다. 포워딩은 하드웨어적인 '지름길 배선'을 통해 데이터가 생성되자마자 다음 명령어로 쏴줌으로써, **파이프라인 정지 없이 연속적인 연산을 보장**하며 시스템의 실질적인 연산 스루풋을 이상적인 수치(CPI=1)로 방어하는 핵심적인 가속 기술로 기능한다.
- **💡 비유**: 식당 주방에서 1번 요리사가 막 구워낸 고기 패티를 굳이 식당 공용 냉장고(레지스터)에 얌전히 넣었다가 2번 요리사가 나중에 다시 꺼내 오게 하는 바보 같은 짓을 그만두고, 구워지자마자 허공으로 던져 2번 요리사의 프라이팬(ALU)으로 직행시키는 환상적인 이어달리기와 같다. 지름길을 뚫어 낭비되는 시간을 없애는 것이다.

- **등장 배경**: 파이프라이닝을 도입하자 '데이터 해저드'가 빗발쳤다. `A = B + C` 다음 줄에 `D = A + E`가 오면, `A`가 레지스터에 완전히 기록될 때까지(클럭 5의 WB 단) 파이프라인 전체가 무조건 멈춰야(Stall) 했다. 엔지니어들은 "어차피 `A`의 덧셈 계산은 클럭 3(EX 단)에 벌써 끝났잖아? 왜 레지스터에 기록될 때까지 미련하게 기다려야 하지?"라는 근본적인 의문을 품었고, 결과물을 뒤에서 앞으로 다시 끌어오는 전방 전달 배선을 창안하게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 포워딩 회로의 하드웨어 구성 요소

| 요소명 | 역할 | 설명 |
|:---|:---|:---|
| **EX/MEM 래치 배선** | 1단계 직전(최신) 데이터 우회 | 앞 명령어가 EX(연산)를 막 끝내자마자 나온 따끈한 결과를 낚아채서 돌려보냄 |
| **MEM/WB 래치 배선** | 2단계 직전(과거) 데이터 우회 | 앞앞 명령어가 MEM 단계를 끝내고 나온 결과를 낚아채서 돌려보냄 |
| **포워딩 제어 유닛 (Forwarding Unit)**| 충돌(의존성) 감지 및 논리 판단 | 앞 명령어의 목적지 레지스터 번호(rd)와 뒤 명령어의 소스 레지스터 번호(rs, rt)가 일치하는지 비교 |
| **입력 멀티플렉서 (MUX)** | 데이터 출처 스위칭 조작 | ALU의 입력으로 "ID 단에서 온 원래 레지스터 값"을 받을지, "포워딩된 우회 값"을 받을지 최종 선택 |

### 포워딩 회로의 물리적 동작 (EX-to-EX Forwarding 예시)

```text
  ┌─────────────────────────────────────────────────────────────────────────────┐
  │         데이터 포워딩 (EX-to-EX Bypass) 배선도 및 작동 원리                 │
  ├─────────────────────────────────────────────────────────────────────────────┤
  │                                                                             │
  │ [Inst 1]: ADD $t0, $t1, $t2   [Inst 2]: SUB $t3, $t0, $t4                   │
  │                                                                             │
  │     [ID 단계]           [EX 단계]           [MEM 단계]                      │
  │                                                                             │
  │ 레지스터 파일 (Read) ──▶ [ ALU ] ──────┬──▶ 데이터 캐시                     │
  │ (Inst 2의 $t0 읽기)        ▲          │(Inst 1의 결과값 $t0)                │
  │ (오래된 쓰레기 값)         │          │                                     │
  │                          │          │                                       │
  │                          └─(포워딩 지름길 배선)┘                            │
  │                        (Bypass MUX 통과)                                    │
  │                                                                             │
  │ [작동 과정]                                                                 │
  │ 1. Inst 2는 ID 단계에서 일단 레지스터의 $t0 쓰레기 값을 순진하게 읽어온다.  │
  │ 2. 하지만 [포워딩 제어 유닛]이 "앗! 앞 명령어 목적지가 $t0이네!" 하고 판단. │
  │ 3. Inst 2가 EX 단계에 진입하는 순간, Inst 1이 방금 뱉어낸 EX/MEM 래치의     │
  │    진짜 최신 결과값을 전용 배선을 통해 끌어와 ALU 앞의 MUX 스위치를 튼다.   │
  │ 4. 결국 쓰레기 값은 덮어씌워져 버려지고, 최신 우회 값을 받아 정상 연산 수행!│
  └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 포워딩은 잃어버린 시간을 되돌리는 타임머신 마법이 아니라, 하드웨어 선(Wire)을 추가로 깔아 만든 '공간적 지름길 뚫기'다. ALU 앞단에는 원래 레지스터에서 온 데이터를 받을지, 아니면 뒤쪽 파이프라인(EX/MEM 래치 또는 MEM/WB 래치)에서 허공을 날아온 우회 데이터를 받을지 결정하는 3-way 멀티플렉서(MUX)가 필수적으로 추가된다. 이 추가 배선 덕분에 파이프라인은 단 1클럭의 지연(Stall)이나 기포 발생 없이 자연스럽게 명령어 2개를 연달아 처리한다.

- **📢 섹션 요약 비유**: 작업한 서류를 본사 창고(레지스터)에 정식으로 입고시켰다가 다시 출고하는 멍청한 헛수고를 없애고, 옆 부서 박 대리가 서류(데이터) 작성을 끝내자마자 복도(포워딩 배선)에서 바로 가로채서 내 작업(ALU)을 즉시 시작하는 극강의 스피드 업무 처리입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 포워딩 적용 전후의 파이프라인 타이밍 비교

RAW 해저드 발생 시 파이프라인의 엄청난 클럭 사이클 낭비가 어떻게 마법처럼 사라지는지 보여준다.

| 클럭 사이클 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 비고 |
|:---|:---|:---|:---|:---|:---|:---|:---|:---|
| **포워딩 없음 (ADD)** | IF | ID | EX | MEM | **WB** | | | 클럭 5 끝에서야 레지스터에 $t0 기록됨 |
| **포워딩 없음 (SUB)** | | IF | ID | *Stall* | *Stall* | EX | MEM | 꼼짝없이 **2클럭 기포(Bubble)** 낭비! |
| **포워딩 적용 (ADD)** | IF | ID | **EX** | MEM | WB | | | EX 끝난 시점(클럭 3)에 $t0 데이터 이미 획득 완료 |
| **포워딩 적용 (SUB)** | | IF | ID | **EX** | MEM | WB | | 포워딩 선을 타고 **클럭 4의 EX로 즉시 쏨 (Stall = 0)** |

### 포워딩의 물리적 한계: Load-Use Hazard

아무리 훌륭한 포워딩으로도 절대 막을 수 없는 최악의 데이터 해저드가 있다. 메모리에서 데이터를 가져오는 Load(`LW`) 명령어는 EX 단계가 아니라, 메모리를 다녀오는 **MEM 단계(클럭 4)가 완전히 끝나야만** 캐시에서 데이터가 튀어나온다.
만약 그 직후에 뒤따라오는 명령어가 클럭 4의 EX 단계에서 그 데이터를 당장 쓰려 한다면, 물리적으로 아직 데이터가 생성되기도 전(미래)이므로 포워딩 배선을 아무리 연결해 봤자 소용이 없다. 이때는 어쩔 수 없이 무조건 **1클럭의 스톨(Load-Use Stall)**이 발생한다.

- **📢 섹션 요약 비유**: 옆 부서 박 대리가 서류 작성을 마치는 건 1초 만에 복도에서 뺏어올 수(포워딩) 있지만, 저 멀리 우체국(메모리)에서 오토바이로 배달이 와야 하는 물건(Load)은 아무리 발버둥을 쳐도 우체부 아저씨가 도착할 때까지 무조건 기다릴(Stall) 수밖에 없는 명백한 물리적 한계가 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **깊은 슈퍼파이프라인에서의 포워딩 폭발 (Bypass Network Complexity)**
   최신 스마트폰의 AP나 데스크탑 CPU(예: 인텔 코어, ARM Cortex-X 시리즈)는 파이프라인 깊이가 14단이 훌쩍 넘는다. 이 말은 EX 단계 뒤로 결과값을 머금고 있는 래치(Latch)들이 수두룩하다는 뜻이다. EX1, EX2, EX3, MEM1, MEM2 등 수많은 단계에서 현재 실행하려는 명령어의 ALU 입력단으로 배선을 다 역방향으로 끌어와야 한다. 아키텍트는 칩 면적(Area)과 전력(Power)이 허용하는 선에서, 실질적으로 성능 향상에 도움이 되는 최근 2~3단계의 포워딩 경로만 남기고, 너무 먼 과거의 포워딩 경로는 과감히 쳐내는 뼈아픈 트레이드오프(PPA 최적화) 결정을 내려야 한다.

2. **컴파일러와의 융합 (Instruction Scheduling)**
   하드웨어 포워딩으로도 끝내 해결 못 하는 `Load-Use 해저드`의 1클럭 스톨을 없애기 위해, 실무 컴파일러(GCC, LLVM)는 어셈블리 코드를 고의로 뒤섞는다. Load 명령어와 그것을 사용하는 덧셈 명령어 사이에, 그 둘과 전혀 무관한 독립적인 명령어 1개를 강제로 끌어올려 샌드위치처럼 끼워 넣는다. 멍청한 하드웨어가 1클럭을 멍때리고 쉴 바에야, 그 시간에 남는 다른 연산을 처리하게 만드는 기적적인 하드웨어-소프트웨어 융합 최적화다.

### 도입 체크리스트
- **기술적**: 포워딩 제어 유닛(Forwarding Unit) 내부에 위치한 수십 개의 레지스터 주소 비교 회로(Comparator) 딜레이가 너무 길어져서, ALU의 임계 경로(Critical Path)를 초과해버려 오히려 칩 전체의 최고 클럭 주파수(Fmax) 상한선을 깎아먹고 있지 않은가?

### 안티패턴
- **단가 절감을 이유로 포워딩 회로 완전 배제 (소프트웨어에 짐 떠넘기기)**
  실리콘 면적과 배선 전력을 아끼겠다고 저가형 초소형 임베디드 칩에서 포워딩(Bypass) 배선을 아예 없애버리는 행위. 이렇게 되면 컴파일러가 데이터 의존성이 생길 때마다 앞뒤 명령어가 겹치지 않게, 빈 깡통 명령어인 NOP(No Operation)을 2~3개씩 무식하게 쑤셔 넣어야만 프로그램이 안 터진다. 이는 바이너리 사이즈(Code Bloat)를 끔찍하게 키우고 명령어 캐시(L1i) 적중률을 박살 내어 칩 전체를 쓰레기로 만드는 최악의 짠돌이 설계다. (현대엔 아무리 싼 칩이라도 포워딩 회로망은 무조건 넣는다).

- **📢 섹션 요약 비유**: 공사비(전선 면적) 조금 아끼겠다고 꽉 막히는 교차로에 고가도로(포워딩)를 안 깔아주면, 동네 사람들(컴파일러)이 매번 차가 막힐 때마다 동네를 빙빙 도는 헛수고(NOP 강제 삽입)를 수백 번씩 해야 해서 결국 도시 전체의 생산성이 죽어버립니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 기술 도입 | 파이프라인 스톨(Stall) 강제 방치 시 | 포워딩(Forwarding) 배선 도입 후 | 핵심 개선 효과 |
|:---|:---|:---|:---|
| **정량 (성능)** | RAW 발생 시마다 2~3클럭 패널티 누적 | **0클럭 지연**으로 쾌속 통과 | 파이프라인 IPC가 이상적 수치(1.0) 방어 |
| **정량 (전력)** | 기포(Bubble) 상태에서 클럭 낭비 | 멈춤 없는 파이프라인으로 낭비 제거 | 클럭 틱(Tick)당 유효 연산 전력 효율(Perf/W) 극대화 |
| **정성 (설계)** | 하드웨어 간단, 컴파일러 최적화 지옥 | 칩 배선 설계는 복잡해지나 S/W 투명성 보장 | 과거 레거시 코드도 수정 없이 하드웨어가 자동 고속화 |

### 미래 전망
- **거대 분산형 포워딩 네트워크 (Bypass Network)의 AI 라우팅**: 과거 초기 RISC 시절에는 단일 ALU를 위한 포워딩만 고민하면 됐다. 하지만 현대의 슈퍼스칼라 및 비순차적(OoO) 실행 엔진은 내부에 덧셈기, 곱셈기 등 연산기가 10개가 넘는다. 1번 덧셈기의 갓 구워진 결과를 4번 곱셈기로, 3번 로드 큐의 결과를 2번 분기 유닛으로 실시간으로 쏘아 넘겨야 하는 거대한 N x N 포워딩 그물망(Bypass Network)이 형성된다. 미래의 마이크로아키텍처 설계는 이 거대한 그물망의 라우팅 딜레이를 머신러닝으로 물리적으로 최적화하여 칩의 면적과 발열을 잡는 극한의 하드웨어 토폴로지 경쟁으로 고도화되고 있다.

- **📢 섹션 요약 비유**: 예전 구멍가게 시절에는 옆자리 동료에게만 서류를 손으로 건네주면 끝이었지만(단순 포워딩), 지금은 100명이 동시에 일하는 운동장 같은 거대 사무실에서 누구든 서류 작성이 끝나는 즉시 필요한 사람의 책상으로 드론을 띄워 정확히 쏴주는 거대한 택배 배송망 시스템(Bypass Network)으로 진화했습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **파이프라인 해저드 (Pipeline Hazard)** | 그중에서도 특히 데이터의 앞뒤 순서가 꼬여서 발생하는 RAW(Read After Write) 해저드를 하드웨어적으로 무효화시키는 기술이 바로 포워딩이다. |
| **명령어 스케줄링 (Instruction Scheduling)** | 포워딩 배선으로도 물리적 시간 역행이 불가해 막지 못하는 'Load-Use 해저드'의 1클럭 스톨을 메워버리는 컴파일러 차원의 소프트웨어 콤비 기술이다. |
| **데이터 해저드 (Data Hazard)** | 데이터 포워딩 기술이 세상에 탄생하게 된 근본적이고 필연적인 원인(질병)이다. |
| **ALU (Arithmetic Logic Unit)** | 포워딩된 최신의 따끈한 데이터가 최종적으로 다이렉트로 꽂혀서 곧바로 다음 연산을 수행하게 되는 물리적 목적지다. |
| **비순차적 실행 (Out-of-Order Execution)** | 슈퍼스칼라 환경에서 무수히 많은 연산기 사이를 거미줄처럼 잇는 포워딩 네트워크(Bypass Network)가 이 비순차 처리 기술의 핵심 혈관망이 된다. |
