# 데이터 포워딩 (Data Forwarding / Bypassing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 포워딩(Data Forwarding 또는 Bypassing)은 파이프라인에서 데이터 해저드가 발생했을 때, 앞 명령어의 결과를 레지스터에 기록(WB)될 때까지 기다리지 않고 **ALU 연산이 끝난 직후 곧바로 다음 명령어의 ALU 입력으로 다이렉트 전송하는 하드웨어 우회 회로**다.
> 2. **가치**: 2~3클럭에 달하는 치명적인 파이프라인 스톨(Stall/Bubble)을 순식간에 **0클럭으로 소거**하여, RISC의 이상향인 1클럭당 1명령어(IPC=1) 실행을 현실에서 가능케 한 구원 투수다.
> 3. **융합**: 멀티플렉서(MUX)와 비교기(Comparator)를 통한 하드웨어 제어 로직의 극치를 보여주며, 메모리에서 데이터를 가져오는 Load 명령어의 경우 포워딩만으로는 한계가 있어 컴파일러의 명령어 스케줄링과 반드시 결합되어야 한다.

+++

## Ⅰ. 데이터 포워딩의 개요

### 1. 정의
- CPU 파이프라인 내부의 EX(실행) 또는 MEM(메모리) 단계의 출력 포트에서 산출된 연산 결과를, 아직 레지스터 파일에 쓰여지기(WB) 전에, 뒤따르는 명령어의 ALU 입력 포트로 직접 연결하여 공급하는 하드웨어 패스(Path)다.

### 2. 등장 배경: '어차피 답은 나왔는데 왜 기다려야 해?'
- `ADD R1, R2, R3`를 하면 R1의 정답은 3번째 EX 단계에서 이미 완성된다. 하지만 정상적인 절차라면 5번째 WB 단계가 끝날 때까지 레지스터에 저장되지 않는다. 뒤따라오는 명령어는 이 정답이 필요해서 파이프라인을 멈추고(Stall) 레지스터에 저장되기만을 하염없이 기다렸다. 설계자들은 "어차피 ALU에서 답이 튀어나왔으니, 전선 하나만 새로 깔아서 바로 뒤 명령어의 입구에 던져주자!"는 발상의 전환을 이루어냈다.

### 3. 💡 비유: '컨베이어 벨트의 새치기'
- 데이터 포워딩은 **'공장에서 앞 작업자가 막 깎아낸 따끈따끈한 부품을, 창고(레지스터)에 넣고 전산 등록(WB)할 때까지 기다리지 않고 바로 뒷 작업자에게 창문으로 휙 던져주는 것'**과 같다.
- 공식적인 서류 절차(레지스터 저장)는 나중에 천천히 진행하더라도, 당장 작업이 급한 뒷사람이 멈추지 않고 계속 일할 수 있게 만드는 센스 넘치는 직거래다.

- **📢 섹션 요약 비유**: 서류를 우편함(레지스터)에 넣고 다음 날 찾아가게 하는 답답한 행정에서, 옆자리 동료에게 팩스로 바로 쏴주는 고속 우회(Bypass) 행정입니다.

## Ⅱ. 메커니즘 및 하드웨어 동작 (Deep Dive)

제어 유닛 내부의 '포워딩 유닛(Forwarding Unit)'이 레지스터 번호를 실시간으로 비교하여 MUX를 조작한다.

### 1. 포워딩이 일어나는 2가지 핵심 구간
1. **EX/MEM $\to$ EX 포워딩**: 직전 명령어의 ALU 결과가 다음 명령어의 ALU 입력으로 직행.
2. **MEM/WB $\to$ EX 포워딩**: 전전 명령어의 결과(또는 메모리 로드 값)가 다음 명령어의 ALU 입력으로 직행.

### 2. 하드웨어 데이터 흐름 시각화 (ASCII)
```text
    [ Data Hazard: ADD R1, R2, R3 실행 후 바로 SUB R4, R1, R5 실행 ]
    
    Instruction 1 (ADD): | IF | ID | EX | MEM | WB |
                                      │ (R1의 정답이 여기서 나옴)
          ┌───────────────────────────┘
          │ (Forwarding Path: 물리적인 전선과 MUX)
          ▼
    Instruction 2 (SUB):      | IF | ID | EX | MEM | WB |
                                         (R1 값을 레지스터에서 읽지 않고 바로 받아 씀!)
                                         
    * 결과: SUB 명령어는 아무런 Stall 없이 파이프라인을 완벽히 이어나감.
```

### 3. 심층 동작 원리: 포워딩 제어 로직의 판단
- 포워딩 유닛은 매 클럭마다 "앞 명령어의 Destination 레지스터 번호"와 "뒷 명령어의 Source 레지스터 번호"를 비교한다.
- `if (EX/MEM.RegisterRd == ID/EX.RegisterRs1)` 라면, 레지스터에서 읽은 옛날 값을 버리고, MUX를 틀어 **ALU에서 직배송된 따끈따끈한 새 값**을 ALU 입력으로 채택한다. 트랜지스터 몇 개로 이루어진 비교기가 시스템의 성능을 2배 이상 살려낸다.

- **📢 섹션 요약 비유**: 찰나의 순간에 "어? 네가 필요한 부품, 방금 내가 만들었어!"라고 외치며 컨베이어 벨트의 경로를 스위치 하나로 꺾어버리는 정밀한 제어 기술입니다.

## Ⅲ. 데이터 포워딩의 한계 (Load-Use Hazard)

전능해 보이는 포워딩도 뚫지 못하는 물리법칙의 한계가 있다.

| 일반적인 산술 해저드 (ADD -> SUB) | Load-Use 해저드 (LOAD -> ADD) |
|:---|:---|
| 결과가 **EX 단계**에서 즉시 생산됨 | 결과가 메모리를 읽는 **MEM 단계 끝에서** 생산됨 |
| 뒷 명령어의 EX 단계 시작 전 전달 가능 | 뒷 명령어의 EX 단계가 **이미 시작된 후**에야 답이 나옴 |
| 포워딩으로 완벽 회피 (0 Stall) | 포워딩 + **반드시 1클럭의 Stall(버블)** 필요 |

- **📢 섹션 요약 비유**: 아무리 직거래(포워딩)를 하려 해도, 택배(메모리)가 아직 도착하지 않았다면 시간 여행을 하지 않는 한 최소한의 기다림(Stall)은 피할 수 없습니다.

+++

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: 파이프라인이 깊어질수록 꼬이는 포워딩 배선
- **상황**: 5단 파이프라인에서는 포워딩 회로(MUX와 비교기)가 몇 개 안 필요했지만, 현대의 15단계 이상 슈퍼파이프라인 구조에서는 5~6개 전 명령어의 결과를 끌어다 써야 함.
- **판단**: 포워딩 배선이 기하급수적으로 얽히면서(Routing Complexity), 칩 면적을 잡아먹고 배선 자체가 새로운 크리티컬 패스(지연 시간)를 유발하게 된다. 아키텍트는 하드웨어 포워딩의 거미줄 같은 배선 비용과 성능 이득 사이에서 최적의 파이프라인 단수를 결정해야 한다. 이를 포기하고 스케줄링으로 넘기려는 철학도 있으나, 현재의 대세는 '돈(실리콘)을 쏟아부어 모든 경로에 포워딩 전선을 까는 것'이다.

### 2. 멀티코어 환경에서의 스눕 포워딩 (Snoop Forwarding)
- **인사이트**: 포워딩의 개념은 단일 코어를 넘어 멀티코어의 캐시 일관성(Cache Coherence) 문제로 스케일업 되었다. 코어 1이 쓴 데이터를 코어 2가 필요로 할 때, 메모리(RAM)까지 내려가서 읽어오면 너무 느리다. 따라서 코어 1의 L1 캐시에서 코어 2의 L1 캐시로 데이터를 직접 쏴주는 '캐시 간 포워딩(Cache-to-Cache Transfer)'이 현대 서버 칩의 핵심 통신망 기술로 적용되고 있다.

- **📢 섹션 요약 비유**: 공장 안에서의 직거래를 넘어, 전국에 흩어진 공장들끼리 중앙 물류센터를 거치지 않고 직통 퀵서비스로 부품을 쏴주는 초연결 물류망으로 진화했습니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **파이프라인 버블(Bubble) 90% 소거**: 데이터 의존성으로 인한 속도 저하를 하드웨어 레벨에서 거의 완벽하게 박멸.
- **소프트웨어 투명성**: 프로그래머나 컴파일러가 포워딩의 존재를 몰라도 하드웨어가 스스로 알아서 최적의 속도로 실행.

### 2. 미래 전망
최신 비순차 실행(Out-of-Order) 프로세서는 레지스터 리네이밍(Renaming) 기술을 통해 명령어의 실행 순서를 마구 뒤섞는데, 이때 포워딩 회로는 수십 개의 물리 레지스터 사이를 거미줄처럼 엮는 '결과 방송 버스(Common Data Bus, CDB)' 형태로 진화했다. 연산이 끝나는 즉시 태그를 달아 버스에 뿌리면, 그 값을 애타게 기다리던 모든 명령어가 동시에 낚아채서 실행을 재개하는 장관이 펼쳐진다. 데이터 포워딩은 폰 노이만 병목을 부수기 위한 인간의 가장 직관적이고 통쾌한 승리다.

- **📢 섹션 요약 비유**: 각자 수신호로 은밀하게 직거래하던 시대를 지나, 거대한 확성기로 정답을 동네방네 방송하여 필요한 사람들이 즉각 주워 먹는 주식 거래소 같은 아키텍처입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **데이터 해저드 (Data Hazard)**: 데이터 포워딩이 물리쳐야 할 영원한 숙적.
- **파이프라인 스톨 (Pipeline Stall)**: 포워딩이 없었다면 겪어야 했던 성능 저하의 벌칙.
- **명령어 스케줄링 (Instruction Scheduling)**: 포워딩으로도 막지 못하는 Load-Use 틈새를 메워주는 든든한 동업자.
