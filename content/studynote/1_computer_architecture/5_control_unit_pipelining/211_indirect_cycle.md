> **핵심 인사이트**
> 1. 간접 사이클(Indirect Cycle)은 해독(Decode) 사이클과 실행(Execute) 사이클 사이에 삽입되는 추가적인 단계로, **명령어의 오퍼랜드가 '데이터의 진짜 주소가 담긴 또 다른 주소'를 가리킬 때 발생**한다.
> 2. 진짜 유효 주소(Effective Address, EA)를 알아내기 위해 **반드시 메인 메모리(또는 캐시)에 한 번 더 접근(Memory Reference)**해야 하므로 명령어 처리 시간을 상당히 지연시킨다.
> 3. 제한된 명령어 비트 수로 광활한 전체 메모리 공간에 접근하기 위한 과거 CISC 아키텍처의 고육지책이었으며, 현대 RISC 구조에서는 포인터(레지스터 간접) 방식으로 대체되어 거의 사라졌다.

---

## Ⅰ. 간접 사이클의 발생 원인과 조건

폰 노이만 아키텍처에서 기본 명령어 사이클은 `Fetch $\rightarrow$ Decode $\rightarrow$ Execute`의 3단계로 이루어집니다. 하지만 명령어를 해독(Decode)해 보니, 오퍼랜드 필드의 모드 비트(Mode Bit)가 '간접 주소 지정(Indirect Addressing)'을 나타내는 경우 CPU는 실행 단계로 바로 넘어갈 수 없습니다.

- **원인:** 오퍼랜드에 적힌 주소(`Operand Address`)로 메모리를 찾아갔더니 실제 연산할 데이터가 있는 게 아니라, "진짜 데이터는 다른 곳에 있다"는 **유효 주소(EA)**가 들어있기 때문입니다.
- **조치:** CPU는 진짜 데이터의 주소(EA)를 얻기 위해 메모리를 한 번 더 읽어오는 별도의 사이클을 추가해야 합니다. 이것이 바로 간접 사이클입니다.

📢 **섹션 요약 비유:**
요리사가 창고 A칸(오퍼랜드)에 재료를 가지러 갔더니 재료는 없고 "진짜 재료는 C칸에 있다"라는 쪽지만 남겨져 있어서, 다시 C칸으로 걸음을 옮겨야 하는 추가적인 헛걸음(간접 사이클)입니다.

---

## Ⅱ. 간접 사이클의 물리적 마이크로 오퍼레이션

간접 사이클이 벌어지는 동안 CPU 내부의 레지스터들은 다음과 같이 움직입니다.

1. **`MAR $\leftarrow$ IR(Address)`**: 
   해독이 끝난 명령어 레지스터(IR)의 오퍼랜드(주소) 부분을 메모리 주소 레지스터(MAR)로 보냅니다.
2. **`MBR $\leftarrow$ M[MAR]`**: 
   메모리(M)에 읽기 신호를 보내어, 해당 주소 안에 들어있던 '진짜 주소(EA)'를 메모리 버퍼 레지스터(MBR)로 가져옵니다.
3. **`IR(Address) $\leftarrow$ MBR`**: 
   가져온 진짜 주소(EA)를 다시 명령어 레지스터(IR)의 오퍼랜드 필드에 덮어씁니다.

이제 IR에는 가짜 주소가 아닌 진짜 주소(EA)가 들어있으므로, 비로소 정상적인 실행(Execute) 사이클로 진입하여 진짜 데이터를 가져올 수 있게 됩니다.

📢 **섹션 요약 비유:**
명령서(IR)에 적힌 "사물함 1번"이라는 글씨를 지우고, 사물함 1번 안에 들어있던 진짜 위치인 "사물함 100번"이라는 글씨로 명령서를 수정 업데이트하는 과정입니다.

---

## Ⅲ. 아키텍처적 페널티와 딜레마 (Memory Wall)

간접 사이클은 컴퓨터 구조 역사상 가장 뼈아픈 트레이드오프(Trade-off) 중 하나입니다.

- **장점 (주소 공간 확장):** 16비트 오퍼랜드로도 32비트 전체 메모리 공간에 접근할 수 있게 해주는 유일한 마법이었습니다.
- **단점 (메모리 병목):** 현대 컴퓨터에서 CPU 연산은 1나노초(ns)면 끝나지만, 메모리를 한 번 다녀오려면 100나노초가 걸립니다. 간접 사이클은 이 느려터진 메모리 방문을 강제로 2번(간접 사이클 1번 + 실행 사이클 1번)이나 시키므로, 성능을 나락으로 떨어뜨립니다.
- 파이프라이닝 관점에서도, 간접 사이클이 발생하면 파이프라인의 타이밍이 완전히 꼬여버려 치명적인 지연(Stall)이 발생합니다.

📢 **섹션 요약 비유:**
작은 지갑(오퍼랜드)에 큰 물건(전체 주소)을 넣기 위해 어쩔 수 없이 보관증(간접 주소)을 썼지만, 물건을 찾을 때마다 창고를 두 번씩 왕복해야 해서 요리사(CPU)가 분통을 터뜨리는 상황입니다.

---

## Ⅳ. 현대 컴퓨터에서의 퇴장과 진화

이러한 치명적인 성능 저하 때문에, 현대의 범용 프로세서(특히 ARM, RISC-V 등 RISC 진영)에서는 하드웨어 차원의 순수 '메모리 간접 사이클'을 ISA에서 아예 삭제해 버렸습니다.

- **어떻게 극복했는가? (레지스터 간접 주소 지정):**
  메모리에 쪽지를 남기는 대신, CPU 내부에 있는 가장 빠른 저장소인 **범용 레지스터(GPR)** 안에 진짜 주소(포인터)를 담아둡니다.
- 이렇게 하면 간접 사이클이라는 별도의 단계 없이, 실행(Execute) 단계에서 레지스터 값을 바로 주소로 사용하여 메모리에 딱 1번만 접근(Load/Store)하면 되므로 속도 문제를 완벽히 해결할 수 있습니다.

📢 **섹션 요약 비유:**
창고에 쪽지를 남겨두는 옛날 방식(간접 사이클)을 폐지하고, 대신 요리사가 자기 주머니 수첩(레지스터)에 진짜 주소를 적어두고 바로 찾아가는 스마트한 방식(레지스터 간접)으로 진화했습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Instruction Cycle (명령어 사이클)]`: 간접 사이클을 품고 있는 CPU의 전체 동작 루틴
- `[Indirect Addressing (간접 주소 지정)]`: 간접 사이클을 강제로 유발하는 주소 지정 방식
- `[Memory Reference (메모리 참조)]`: 간접 사이클이 파이프라인 성능을 떨어뜨리는 핵심 원인
- `[Pointer (포인터)]`: 간접 사이클의 논리적 개념을 소프트웨어로 구현한 C언어의 문법
- `[Register Indirect Addressing]`: 느린 간접 사이클을 대체하기 위해 현대 CPU가 채택한 고속 포인터 지정 방식

## 👶 어린이를 위한 3줄 비유 설명
1. 간접 사이클은 심부름을 갔더니 "진짜 물건은 옆방에 있어!"라는 쪽지를 발견하고 옆방으로 한 번 더 걸어가는 시간이에요.
2. 한 번에 찾을 수 있는 걸 두 번이나 걸어가야 하니까 로봇이 엄청나게 힘들어하고 시간이 오래 걸리죠.
3. 그래서 요즘 똑똑한 로봇들은 이런 방식을 버리고, 아예 출발하기 전부터 진짜 주소를 수첩(레지스터)에 적어가는 방식을 쓴답니다!
