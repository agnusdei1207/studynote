# 222. 구조적 해저드 (Structural Hazard)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 구조적 해저드 (Structural Hazard)는 파이프라인의 서로 다른 단계(Stage)에 있는 두 개 이상의 명령어가 같은 클럭 사이클에 **동일한 물리적 하드웨어 자원(메모리, 버스, 레지스터 포트 등)**을 동시에 요구할 때 발생하는 하드웨어 충돌이다.
> 2. **가치**: 이 충돌은 프로그래머의 논리적인 코드 짜임새 문제가 아니라 순전히 칩 설계의 물리적/비용적 한계 때문에 발생하며, 이를 방어하기 위해 부품을 물리적으로 복제(Duplication)하는 방식이 프로세서 성능 발전을 이끌었다.
> 3. **융합**: 초창기 폰 노이만 구조의 뼈아픈 단일 메모리 버스 병목을 부수고, 명령어 캐시(L1i)와 데이터 캐시(L1d)를 내부적으로 분리하는 **하버드 아키텍처(Harvard Architecture)**를 현대 고성능 CPU의 표준으로 만든 결정적 원인이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 구조적 해저드 (Structural Hazard)는 파이프라인의 서로 다른 단계에 있는 두 개 이상의 명령어가 같은 클럭 사이클에 **동일한 물리적 하드웨어 자원(메모리, 버스, 연산기 등)을 동시에 점유**하려고 할 때 발생하는 물리적 충돌 현상이다.

- **필요성**: 구조적 해저드에 대한 분석과 해결은 파이프라인 아키텍처가 이론상의 최대 스루풋(CPI=1)을 실제로 달성하기 위해 반드시 필요하다. 아무리 논리적 의존성이 없는 코드라도 하드웨어 자원이 부족하면 물리적으로 병목(Stall)이 발생하여 클럭을 낭비하게 되기 때문이다. 따라서 구조적 해저드를 정밀하게 식별함으로써 **자원 복제(Duplication)나 기능 유닛의 다중화(Multi-porting)**와 같은 하드웨어 투자 우선순위를 결정하며, 특히 명령어 캐시와 데이터 캐시를 분리하는 **하버드 아키텍처** 도입의 결정적인 공학적 근거를 제공하여 시스템의 연산 밀도를 극대화하는 핵심 설계 이정표 역할을 수행한다.
- **💡 비유**: 공중화장실에 변기가 1개뿐(단일 하드웨어 자원)인데, 1번 손님과 4번 손님이 동시에 화장실을 급하게 쓰고 싶어 하면(구조적 해저드), 누군가 한 명은 문 밖에서 10분을 고통스럽게 기다려야(Stall) 하는 상황과 같다. 화장실을 하나 더 지어주면(자원 복제) 영구적으로 해결된다.

- **등장 배경**: 파이프라이닝이 처음 도입되었을 때, 설계자들은 이상적인 병렬 처리를 위해서는 모든 파이프라인 단계가 '독립적인 하드웨어'를 온전히 소유해야 한다는 점을 간과했다. 특히 폰 노이만 구조에서는 명령어(Code)와 데이터(Data)가 같은 메모리 버스에 물려 있었기 때문에, 파이프라인 앞단의 인출(IF) 단계와 뒷단의 메모리 접근(MEM) 단계가 필연적으로 충돌하며 최악의 파이프라인 정지(Stall)를 일으켰다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구조적 해저드의 대표적 발생 지점

| 발생 지점 | 충돌하는 단계 | 원인 | 하드웨어적 해결책 |
|:---|:---|:---|:---|
| **단일 메모리/버스** | **IF** vs **MEM** | 명령어를 읽는 작업(IF)과 데이터를 읽고 쓰는 작업(MEM)이 단일 포트 메모리를 동시에 요구함 | 내부 캐시를 L1i, L1d로 물리적 분리 (하버드 구조 적용) |
| **레지스터 파일 포트**| **ID** vs **WB** | ID에서 연산 데이터를 2개 읽으려 하고, 동시에 WB에서 결과를 1개 쓰려고 할 때 입출력 포트가 부족함 | 독립된 다중 포트(Multi-port: 2 Read, 1 Write) 레지스터 도입 |
| **단일 ALU 연산기** | **EX** vs **(특수 연산)**| 복잡한 부동소수점이나 나눗셈이 여러 클럭 동안 ALU를 점유할 때, 바로 뒤따르는 정수 연산 명령어가 꼼짝없이 막힘 | ALU 다수 복제 및 전용 연산기(FPU, 텐서 유닛) 완전 분리 |

### 가장 전형적인 메모리 버스 충돌 (폰 노이만 구조의 한계)

하나의 메모리와 하나의 버스를 공유할 때 벌어지는 참사다. 이 도식은 왜 현대 CPU가 L1 캐시를 굳이 명령어용과 데이터용 두 개로 쪼개어 만들었는지 보여주는 가장 완벽한 증거다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         단일 메모리 구조에서의 구조적 해저드 (Structural Hazard)     │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │        클럭:   1    2    3    4    5    6                   │
  │     ──────────────────────────────────────────              │
  │ Inst 1 (LOAD): [IF]-[ID]-[EX]-[MEM]-[WB]                    │
  │ Inst 2 (ADD) :      [IF]-[ID]-[EX]-[MEM]-[WB]               │
  │ Inst 3 (SUB) :           [IF]-[ID]-[EX]-[MEM]-[WB]          │
  │ Inst 4 (AND) :                [IF] <── 충돌 발생! (Stall)     │
  │                                                             │
  │ [문제 분석 (클럭 4의 치명적 상황)]                               │
  │ * Inst 1: 메모리(RAM)에서 연산용 '데이터'를 읽어오고 있다 (MEM 단계). │
  │ * Inst 4: 메모리(RAM)에서 다음 '명령어'를 읽어오고 싶어 한다 (IF 단계).│
  │ * 하드웨어 제약: CPU 밖으로 나가는 메모리 버스(도로)는 딱 1개뿐이다.   │
  │ * 제어 유닛의 결정: Inst 4는 Inst 1이 메모리 버스를 다 쓰고 반납할    │
  │                   때까지 무조건 1클럭 이상을 멈춰서 기다려야 한다.   │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클럭 4 시점에 파이프라인의 끝단(MEM)과 맨 앞단(IF)이 동시에 "나 메모리 버스 쓸래!"라고 손을 든다. 메모리가 단일 포트(한 번에 하나의 접근만 허용)라면 물리적으로 두 작업을 동시에 수행할 수 없다. 제어 유닛은 우선순위가 높은(이미 파이프라인 후반부에 진입하여 퇴근이 얼마 안 남은) Inst 1에게 메모리 사용권을 주고, Inst 4의 IF 단계는 강제로 스톨(Stall)시켜 파이프라인에 기포(Bubble)를 만든다. 이 설계라면 Load나 Store 명령어가 나올 때마다 파이프라인이 덜컹거리며 멈추는 끔찍한 성능 저하가 발생한다.

- **📢 섹션 요약 비유**: 왕복 1차선 좁은 흙길(단일 버스)밖에 없어서, 무거운 택배 트럭(데이터)이 지나가는 동안에는 오토바이를 탄 우편 배달부(명령어)가 길가에 서서 트럭이 다 지나가길 하염없이 기다려야 하는 꽉 막힌 병목 현상과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 하버드 아키텍처(Harvard Architecture)를 통한 궁극의 해결

구조적 해저드의 가장 완벽하고 통쾌한 해결책은 "싸우지 않게 물리적인 자원을 돈을 들여 하나 더 복사해 주는 것(Resource Duplication)"이다.

| 구분 | 폰 노이만 (Von Neumann) 아키텍처 | 하버드 (Harvard) 아키텍처 (현대 L1 캐시) |
|:---|:---|:---|
| **메모리 구조** | 통합 메모리 1개 덩어리 (명령어+데이터 혼재) | 명령어 전용 캐시(L1i), 데이터 전용 캐시(L1d) 분할 |
| **물리적 경로** | 단일 데이터/주소 버스 | 완벽히 독립된 2쌍의 데이터/주소 버스 (동시 병렬 접근) |
| **해저드 해결** | **IF와 MEM 단계의 치명적 병목 수시 발생** | **구조적 해저드 완벽 100% 소거** |
| **비용 (Area)** | 버스 1개, 포트 1개로 다이(Die) 면적은 절약됨 | 캐시 메모리 쪼개기와 수백 가닥의 전선 추가로 비용 증가 |

### 레지스터 파일의 Multi-port 확장 설계

메모리뿐만 아니라 레지스터 파일(Register File)에서도 ID(레지스터 읽기) 단계와 WB(결과 쓰기) 단계가 동시에 일어난다. 보통 MIPS나 ARM 같은 RISC 명령어는 레지스터를 2개 읽어 1개를 쓴다 (예: `ADD $t0, $t1, $t2` -> t1, t2 읽고 t0에 씀). 따라서 이 충돌을 막기 위해 현대 프로세서는 아예 레지스터 파일에 **물리적으로 읽기 포트(Read Port) 2개, 쓰기 포트(Write Port) 1개를 따로따로 뚫어놓는다.**
또한, 같은 클럭(예: 클럭 5) 안에서 읽기와 쓰기가 동시에 같은 레지스터 번호를 가리킬 때 발생하는 미세한 충돌을 막기 위해, 클럭 펄스의 전반부(Rising Edge)에는 무조건 WB(쓰기)를 먼저 수행하고, 후반부(Falling Edge)에는 ID(읽기)를 수행하도록 위상을 나누는 영리한 하드웨어 꼼수를 써서 완벽히 방어한다.

- **📢 섹션 요약 비유**: 장난감 하나를 두고 싸우는 형제에게 똑같은 장난감을 하나 더 사주거나(하버드 캐시 분리), 어쩔 수 없을 땐 "오전엔 형이 놀고 오후엔 동생이 놀아라"라고 시간대를 칼같이 나누는 것(레지스터 클럭 에지 분할)이 구조적 해저드를 없애는 부모님(하드웨어 설계자)의 지혜입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **슈퍼스칼라(Superscalar) 코어의 연산기 폭발적 복제 설계**
   최신 스마트폰의 ARM Cortex-X 코어나 데스크탑 CPU는 1클럭에 명령어를 4개~8개씩 동시에 빨아들여 실행하는 슈퍼스칼라 구조다. 만약 칩 안에 덧셈기(ALU)가 2개밖에 없는데, 4개의 덧셈 명령어가 동시에 EX 단계를 통과하려 한다면 즉시 ALU 부족으로 인한 거대한 구조적 해저드가 발생해 파이프라인이 멈춘다. 아키텍트는 이를 돈(실리콘 면적)으로 막기 위해 정수 연산 유닛(ALU) 6개, 부동소수점 유닛(FPU) 4개, 분기 전용 유닛 2개 식으로 어마어마한 양의 물리적 연산기 자원을 칩 안에 중복으로 꽉꽉 복제해(Duplication) 때려 넣는다.

2. **초저가형 마이크로컨트롤러(MCU)의 원가 절감 타협안**
   천원짜리 전기밥솥이나 단순한 장난감에 들어가는 ARM Cortex-M0 같은 저전력 MCU는 칩 다이(Die) 면적을 줄여 단가를 후려치는 게 지상 목표다. 아키텍트는 캐시를 L1i와 L1d로 굳이 나누거나 멀티포트를 마구 뚫는 '돈이 낭비되는' 짓을 과감히 포기한다. 폰 노이만 단일 버스 구조를 그대로 유지하여 파이프라인에서 발생하는 구조적 해저드(Stall)를 기꺼이 감수하되, 칩 사이즈와 누설 전류를 극한으로 줄이는(Area & Power Optimization) 경제적이고 합리적인 결정을 내린다.

### 도입 체크리스트
- **기술적**: 파이프라인의 최고 피크 상태(Peak IPC)에서 동시에 요구되는 특정 하드웨어 자원(예: 로드/스토어 큐, 연산기)의 최대 수를 시뮬레이션으로 계산하고, 그에 맞게 병목이 나지 않도록 물리적 포트와 유닛이 충분히 넉넉하게 할당(Over-provisioning)되어 있는가?

### 안티패턴
- **하드웨어 자원 복사 없이 무리하게 파이프라인 단수만 늘리기 (깡통 스펙)**
  마케팅을 위해 파이프라인을 20단, 30단으로 쪼개어 동시 실행되는 명령어 수는 잔뜩 늘려놓고, 정작 메모리 대역폭(Bus Width)이나 레지스터 입출력 포트의 갯수는 5단 시절 그대로 두는 인색한 설계. 이 경우 겉보기 클럭(GHz)을 아무리 높여봤자, 수많은 명령어가 좁은 골목(단일 자원)에서 서로 자기가 먼저 자원을 쓰겠다고 아우성치다 멈춰버리는 끔찍한 구조적 병목이 발생하여 실제 성능(IPC)은 폭락한다.

- **📢 섹션 요약 비유**: 식당에 장사 잘되라고 테이블을 100개로 빽빽하게 늘렸는데(깊은 파이프라인), 주방장은 여전히 1명이고 가스버너도 1구짜리 그대로(구조적 자원 방치)라면 손님들은 밥을 먹지 못하고 화만 내다가 굶고 나가버리게 됩니다. 규모 확장에 맞는 물리적 설비 투자가 필수입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 해결 방식 (Resource Duplication) | 자원 부족 (충돌 방치) 시 | 물리적 자원 분리/복제 도입 후 | 핵심 개선 효과 |
|:---|:---|:---|:---|
| **정량 (캐시 버스)**| 명령어/데이터 캐시 미분리로 매번 대기 | L1i, L1d 하버드 구조 채택 | 메모리 접근 충돌로 인한 **Stall 100% 제거** |
| **정량 (연산기)**| 단일 ALU 공유로 복잡 연산 시 마비 | 다중 ALU/FPU (슈퍼스칼라) 분산 배치 | 연산 병목 완벽 해소, **최대 IPC 한계선 돌파** |
| **정성 (설계 사상)**| 성능 저하와 스톨의 악순환 반복 | 실리콘 다이(Die) 면적은 증가하나 성능 극대화 | 돈(트랜지스터 갯수)으로 성능을 사는 현대 설계의 진수 |

### 미래 전망
- **다크 실리콘(Dark Silicon) 시대의 구조적 복제 한계**: 예전 마이크로아키텍트들은 구조적 해저드를 없애려고 부품(ALU, 캐시 포트)을 무한정 복제해서 칩에 쑤셔 박았다. 하지만 3nm, 2nm로 트랜지스터 밀도가 너무 높아진 지금은, 부품을 수백 개 넣어봤자 엄청난 발열(Thermal Design Power 초과) 때문에 전부 한 번에 전원을 켤 수가 없다. 칩의 절반은 식히기 위해 항상 꺼두어야 하는 '다크 실리콘' 현상이다. 미래에는 무작정 유닛을 복사해 넣는 무식한 투자를 멈추고, 워크로드의 특성에 따라 특정 유닛의 전원만 영리하게 켜서 충돌을 피하는 초지능형 전력/자원 라우팅 아키텍처가 메인스트림이 되고 있다.

- **📢 섹션 요약 비유**: 구조적 해저드는 컴파일러나 코딩을 잘못해서 생긴 소프트웨어적 병이 아니라 "단순히 기계 부품이 모자라서" 생기는 근본적인 물리적 병입니다. 돈을 넉넉히 들여 부품을 2개, 3개 사주면 가장 확실하고 깔끔하게 낫는 자본주의적 문제점이라 할 수 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **파이프라인 해저드 (Pipeline Hazard)** | 파이프라인의 이상적 성능(CPI=1)을 깎아먹는 3대 빌런 중 하나로, 유일하게 소프트웨어 스케줄링으로 극복 불가능한 '하드웨어적 결핍'을 뜻한다. |
| **하버드 아키텍처 (Harvard Architecture)** | 명령어와 데이터의 물리적 통로(버스와 메모리 공간)를 완벽히 2개로 분리하여 IF와 MEM 단계의 치명적인 구조적 해저드를 원천 차단한 구세주다. |
| **슈퍼스칼라 (Superscalar)** | 넘쳐나는 명령어 파이프라인에서 구조적 해저드를 피하기 위해 연산 유닛(ALU, FPU) 자체를 수량으로 밀어붙여 여러 개 복제해버린 물량 공세 아키텍처다. |
| **레지스터 파일 포트 (Register File Port)** | 파이프라인의 ID단과 WB단이 한 클럭 내에 동시에 읽고 쓸 수 있도록 다중 포트(Multi-port)로 배선 설계되어 병목 충돌을 원천 봉쇄한다. |
| **폰 노이만 병목 (Von Neumann Bottleneck)** | 명령어와 데이터를 단일 버스로 공유하는 폰 노이만 구조의 한계로 인해, 영원히 발생하는 근본적인 구조적 해저드의 뿌리다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 파이프라인 자동차 조립 공장이 1초도 안 쉬고 잘 돌아가려면, 각 단계의 작업자마다 자기만의 망치와 도구가 따로 있어야 해요.
2. 그런데 "1단계 설계도 읽는 아저씨(IF)"랑 "4단계 창고에서 물건 꺼내는 아저씨(MEM)"가 공장에 딱 1개뿐인 공용 망치(메모리 버스)를 쓰려고 서로 뺏고 싸우면 공장이 멈춰버려요! 이걸 '구조적 해저드'라고 해요.
3. 이 싸움을 말리는 가장 완벽한 방법은, 공장 사장님이 돈을 써서 똑같은 비싼 망치를 하나 더 사서 두 사람 손에 각각 전용으로 쥐여주는 거랍니다! (명령어/데이터 캐시 분리)
