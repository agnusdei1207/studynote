# 222. 구조적 해저드 (Structural Hazard)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하드웨어 리소스 부족으로 발생하는 충돌
> 2. **가치**: 파이프라인 스톨의 원인, 현대 CPU에서 거의 해결
> 3. **융합**: Harvard 구조, I-Cache, D-Cache, 포트 수

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

구조적 해저드(Structural Hazard)는 **두 명령어가 동시에 같은 하드웨어 리소스를 사용하려 할 때 발생**하는 충돌이다. 파이프라인에서 여러 명령어가 병렬로 실행될 때 리소스 부족으로 스톨이 발생한다.

### 💡 비유: 공장의 드릴 하나

구조적 해저드는 **공장에 드릴이 하나뿐인 상황**과 같다. 작업자 A가 드릴을 쓰면 작업자 B는 기다려야 한다. 드릴을 하나 더 사면(리소스 추가) 문제가 해결된다.

### 구조적 해저드 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                구조적 해저드 구조                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【발생 원인】                                                         │
│  ──────────────────                                                  │
│  1. 메모리 포트 하나: IF와 MEM 동시 접근 불가                          │
│  2. 레지스터 포트 하나: 읽기와 쓰기 동시 불가                            │
│  3. ALU 하나: 여러 명령어가 동시 연산 불가                               │
│  4. 캐시 포트 하나: 읽기와 쓰기 동시 불가                                │
│                                                                     │
│  【메모리 충돌 예시】                                                   │
│  ────────────────────────                                            │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  클럭  IF    ID    EX    MEM   WB                                 │ │
│  │  ────  ────  ────  ────  ────  ────                               │ │
│  │   1    I1    -     -     -     -                                  │ │
│  │   2    I2    I1    -     -     -                                  │ │
│  │   3    I3    I2    I1    -     -                                  │ │
│  │   4    I4    I3    I2    I1    -                                  │ │
│  │   5    -     I4    I3    I2    I1    ← I4(IF)와 I2(MEM) 충돌      │ │
│  │         ↑                       ↑                                 │ │
│  │      스톨                   메모리 접근                              │ │
│  │                                                               │ │
│  │  메모리가 하나면 IF(I4)와 MEM(I1, I2)이 동시 접근 불가              │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【해결 방안】                                                         │
│  ──────────────────                                                  │
│  1. Harvard 구조: 명령어 메모리 + 데이터 메모리 분리                     │
│  2. I-Cache + D-Cache: 분리된 캐시                                    │
│  3. 다중 포트 레지스터: 읽기 2포트 + 쓰기 1포트                          │
│  4. 파이프라인 분리: 연산 유형별 파이프라인                               │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  구조적 해저드 vs 현대 CPU:                                           │ │
│  │  ────────────────────────                                            │ │
│  │  • MIPS: 5단계, 단순 구조, 구조적 해저드 없음 설계                     │ │
│  │  • 현대 CPU: 다중 포트, 분리 캐시로 사실상 해결                        │ │
│  │  • 임베디드: 비용 절감으로 여전히 발생 가능                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 구조적 해저드 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                구조적 해저드 상세                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Harvard 구조로 해결】                                               │
│  ────────────────────────                                            │
│                                                                     │
│  Von Neumann (메모리 하나):                                           │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  ┌─────────────────────────────────┐                             │ │
│  │  │         통합 메모리               │                             │ │
│  │  │    (명령어 + 데이터)              │                             │ │
│  │  └─────────────────────────────────┘                             │ │
│  │           ↑         ↑                                            │ │
│  │          IF        MEM                                           │ │
│  │        (충돌)                                                     │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  Harvard (메모리 분리):                                               │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  ┌───────────────┐    ┌───────────────┐                         │ │
│  │  │  명령어 메모리  │    │  데이터 메모리  │                         │ │
│  │  │  (I-Memory)    │    │  (D-Memory)    │                         │ │
│  │  └───────────────┘    └───────────────┘                         │ │
│  │           ↑                   ↑                                  │ │
│  │          IF                  MEM                                 │ │
│  │        (충돌 없음)                                                 │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【다중 포트 레지스터 파일】                                           │
│  ─────────────────────────────                                       │
│                                                                     │
│  단일 포트 (구조적 해저드 발생):                                       │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  한 번에 하나만: 읽기 또는 쓰기                                      │ │
│  │  ID에서 읽기, WB에서 쓰기 동시 불가                                  │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  다중 포트 (해결):                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  2 읽기 포트 + 1 쓰기 포트                                          │ │
│  │  → ID에서 2개 읽기 + WB에서 1개 쓰기 동시 가능                       │ │
│  │  → MIPS 기본 설계                                                  │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【캐시 분리】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  L1 Cache 분리:                                                    │ │
│  │  ┌───────────────┐    ┌───────────────┐                         │ │
│  │  │   I-Cache     │    │   D-Cache     │                         │ │
│  │  │  (명령어)      │    │  (데이터)      │                         │ │
│  │  │   32KB        │    │   32KB        │                         │ │
│  │  └───────────────┘    └───────────────┘                         │ │
│  │  → IF와 MEM 동시 접근 가능                                          │ │
│  │  → L2, L3는 공유                                                   │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  구조적 해저드 발생 시나리오:                                          │ │
│  │  ────────────────────────                                            │ │
│  │  1. FP 연산 유닛 하나에 두 FP 명령어                                 │ │
│  │  2. 메모리 버스 하나에 두 메모리 접근                                 │ │
│  │  3. 레지스터 쓰기 포트 하나에 두 WB                                   │ │
│  │  4. 캐시 포트 하나에 읽기/쓰기 동시                                   │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 구조적 해저드 × 타 과목 융합

### 과목 융합 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                구조적 해저드 × 타 과목 융합                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【컴퓨터 구조】                                                      │
│  • Harvard vs Von Neumann: 메모리 구조                               │
│  • 캐시 계층: I-Cache, D-Cache 분리                                  │
│  • 레지스터 파일: 다중 포트 설계                                       │
│  • 슈퍼스칼라: 다중 실행 유닛으로 해결                                  │
│                                                                     │
│  【디지털 논리회로】                                                   │
│  • 다중 포트 메모리: 설계 복잡도                                       │
│  • 버스 구조: 대역폭과 포트 수                                          │
│  • MUX: 다중 입력 선택                                                │
│                                                                     │
│  【운영체제】                                                          │
│  • 메모리 보호: I/D 분리와 보안                                        │
│  • 캐시 관리: I-Cache 플러시 (자기 수정 코드)                           │
│  • 성능: 구조적 해저드가 CPI에 미치는 영향                               │
│                                                                     │
│  【성능 분석】                                                         │
│  • CPI = 1 + 구조적 스톨 비율                                          │
│  • 스톨 사이클 = (충돌 횟수) × (대기 사이클)                            │
│  • 해결 비용 vs 성능 향상                                              │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  구조적 해저드 해결 비용:                                              │ │
│  │  ──────────────────────                                              │ │
│  │  해결 방안          하드웨어 비용    성능 향상                         │ │
│  │  ────────────────  ────────────  ──────────                         │ │
│  │  I/D 캐시 분리      중간           높음                               │ │
│  │  다중 포트 레지스터  낮음           중간                               │ │
│  │  다중 실행 유닛     높음           높음                               │ │
│  │  파이프라인 분리    높음           높음                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### 구조적 해저드 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                구조적 해저드 분석                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【캐시 구조 확인】                                                    │
│  ──────────────────                                                  │
│  $ lscpu | grep -i cache                                            │
│  L1d cache: 32K       ← D-Cache                                     │
│  L1i cache: 32K       ← I-Cache                                     │
│  L2 cache: 256K                                                     │
│  L3 cache: 8192K                                                    │
│  → L1이 분리되어 구조적 해저드 없음                                     │
│                                                                     │
│  【메모리 대역폭 분석】                                                │
│  ────────────────────────                                            │
│  $ perf stat -e cache-references,cache-misses ./prog               │
│  cache-references: 100,000,000                                      │
│  cache-misses: 5,000,000                                            │
│  → 5% 캐시 미스율                                                     │
│                                                                     │
│  【포트 경합 분석】                                                    │
│  ──────────────────                                                  │
│  $ perf stat -e resource_stalls.any ./prog                          │
│  → 리소스 경합으로 인한 스톨 확인                                       │
│                                                                     │
│  【현대 CPU 설계】                                                      │
│  ──────────────────                                                  │
│  • Intel Skylake: L1 I-Cache 32KB, D-Cache 32KB                     │
│  • AMD Zen 3: L1 I-Cache 32KB, D-Cache 32KB                         │
│  • Apple M1: L1 I-Cache 128KB, D-Cache 128KB                        │
│  → 모두 Harvard 구조로 구조적 해저드 없음                               │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  구조적 해저드 회피 설계:                                              │ │
│  │  ──────────────────────                                              │ │
│  │  1. I-Cache와 D-Cache 분리                                           │ │
│  │  2. 다중 포트 레지스터 파일 (2R/1W)                                   │ │
│  │  3. 다중 실행 유닛 (슈퍼스칼라)                                        │ │
│  │  4. 분리된 주소/데이터 버스                                           │ │
│  │  5. 비순차 실행로 유닛 활용 최적화                                    │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• 구조적 해저드: 리소스 부족으로 발생하는 충돌
• 원인: 메모리/레지스터/ALU 동시 접근
• 해결: Harvard 구조 (I-Cache + D-Cache)
• 다중 포트: 읽기 2개 + 쓰기 1개
• 현대 CPU: L1 분리로 사실상 해결됨
• CPI 영향: 현대 CPU에서 0 (무시 가능)
• 슈퍼스칼라: 다중 실행 유닛으로 확장 해결
```
