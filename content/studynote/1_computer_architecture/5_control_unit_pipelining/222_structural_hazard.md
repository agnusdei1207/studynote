# 구조적 해저드 (Structural Hazard)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 구조적 해저드(Structural Hazard)는 파이프라인에서 동시에 실행 중인 두 개 이상의 명령어가 **동일한 하드웨어 자원(메모리, 레지스터 쓰기 포트, ALU 등)을 같은 클럭에 사용하려고 충돌**하는 현상이다.
> 2. **가치**: 이 문제를 방치하면 하드웨어 오작동이 발생하므로 파이프라인을 강제로 멈춰야(Stall) 하지만, 칩 면적을 조금 더 희생하여 하드웨어 유닛을 분리하거나 복제하면 완벽하게 해결할 수 있는 가성비 좋은 최적화 대상이다.
> 3. **융합**: 구조적 해저드를 근본적으로 없애기 위한 가장 위대한 하드웨어적 결단이 바로 명령어 캐시와 데이터 캐시를 물리적으로 분리한 **하버드 아키텍처(Harvard Architecture)**의 채택이다.

---

## Ⅰ. 구조적 해저드의 개요

### 1. 정의
- CPU 내부의 파이프라인을 타고 흐르는 명령어들이 겹칠 때, 특정 하드웨어 부품(Resource)이 모자라 병렬 처리가 불가능해지는 물리적인 자원 제약 상태다.

### 2. 등장 배경: '모두가 같은 우물을 원할 때'
- 파이프라인이 없던 시절에는 명령어 하나가 끝나야 다음 명령어가 들어왔으므로 자원이 겹칠 일이 없었다. 하지만 5개의 명령어가 동시에 파이프라인의 각 단계를 밟기 시작하자, 1번 명령어가 메모리에 결과를 쓸 때(MEM) 4번 명령어가 메모리에서 다음 코드를 가져오려(Fetch) 하면서 '외나무다리(하나의 메모리 버스)'에서 마주치는 병목 현상이 발생했다.

### 3. 💡 비유: '기숙사 화장실 쟁탈전'
- 구조적 해저드는 **'아침 출근 시간, 5명이 사는 기숙사에서 화장실(하드웨어 자원)이 하나밖에 없어 발생하는 대기 상황'**과 같다.
- 누군가 씻고 있다면, 뒷사람은 아무리 일찍 일어났어도 씻고 나올 때까지 복도에서 멍하니 기다려야(Stall) 한다.

- **📢 섹션 요약 비유**: 각자 맡은 일을 열심히 하려는데, 회사에 복사기(자원)가 한 대뿐이라 누군가 복사를 끝낼 때까지 일이 멈추는 오피스 병목입니다.

---

## Ⅱ. 주요 발생 원인 및 해결 메커니즘 (Deep Dive)

가장 빈번하게 발생하는 구조적 해저드는 메모리 접근과 레지스터 접근에서 일어난다.

### 1. 메모리 자원 충돌 (Memory Access Conflict)
- **현상**: 단일 폰 노이만 메모리 구조일 때 발생.
- **상황**:
  - `Instr 1`은 4번째 단계인 **MEM(데이터 읽기)** 수행 중.
  - `Instr 4`는 1번째 단계인 **IF(명령어 인출)** 수행 중.
  - 두 단계 모두 '메인 메모리'를 사용해야 하는데, 메모리로 가는 버스는 1개뿐이다.
- **해결책**:
  - **하버드 아키텍처 도입**: L1 캐시를 L1i(Instruction)와 L1d(Data)로 나누어, IF는 L1i에서, MEM은 L1d에서 동시에 데이터를 읽게 하여 물리적 경로를 완전히 분리한다.

### 2. 레지스터 포트 충돌 (Register File Conflict)
- **현상**: 레지스터 파일의 쓰기(Write) 포트가 모자랄 때 발생.
- **상황**:
  - `Instr 1`은 5번째 단계인 **WB(레지스터에 쓰기)** 수행 중.
  - `Instr 4`는 2번째 단계인 **ID(레지스터에서 읽기)** 수행 중.
- **해결책**:
  - **하프 클럭 동작(Half-clocking)**: 클럭의 전반부(Rising Edge)에는 무조건 쓰기(WB)를 먼저 완료하고, 후반부(Falling Edge)에는 읽기(ID)를 수행하게 하여 충돌을 시간적으로 회피한다.
  - **포트 증설**: 레지스터 파일에 하드웨어적으로 Read 포트와 Write 포트를 독립적으로 여러 개 뚫어준다.

### 3. 구조적 해저드로 인한 스톨(Stall) 시각화 (ASCII)
```text
    [ 폰 노이만 구조 (단일 메모리) 하의 해저드 발생 ]
    
    1. LOAD R1, X  | IF | ID | EX | MEM | WB |
    2. ADD R2, R3  |    | IF | ID | EX  | -- | WB |
    3. SUB R4, R5  |         | IF | ID  | EX | -- | WB |
    4. AND R6, R7  |              | IF* | ID | EX | -- | WB |
    
    * 문제 발생: 4클럭째에 LOAD 명령어가 'MEM'을 쓰고 있는데, 
      AND 명령어가 'IF'를 위해 메모리를 동시에 쓰려 함.
    * 결과: AND 명령어의 IF 단계는 LOAD의 MEM이 끝날 때까지 1클럭 대기(Stall)해야 함.
```

- **📢 섹션 요약 비유**: 싸움을 막으려면 화장실(캐시)을 두 개로 공사하거나, 시간표를 짜서(하프 클럭) 겹치지 않게 쓰도록 규칙을 정해야 합니다.

## Ⅲ. 구조적 해저드의 특징 비교

| 해저드 유형 | 발생 원인 | 해결 난이도 (비용) | 소프트웨어(컴파일러) 개입 |
|:---|:---|:---|:---:|
| **구조적 해저드** | **하드웨어 자원 부족** | **비용 투자로 완벽 해결 가능** | 불필요 |
| **데이터 해저드** | 명령어 간 연산 결과의존성 | 완벽한 제거 불가 (논리적 문제) | 필수적 (스케줄링) |
| **제어 해저드** | 분기(조건문)로 인한 PC 변경 | 해결 매우 어려움 (분기 예측 필요) | 필수적 (루프 언롤링) |

- **📢 섹션 요약 비유**: 구조적 해저드는 '돈(트랜지스터)'을 들여 기계를 더 사면 해결되는 가장 물리적이고 속 시원한 문제입니다.

---

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: 부동소수점(FPU) 연산기 부족
- **상황**: 저가형 임베디드 칩에서 FPU(부동소수점 연산기)를 단 1개만 탑재함. 연달아 3개의 부동소수점 곱셈 명령어가 들어옴.
- **판단**: 정수 ALU는 여러 개라 파이프라인이 멈추지 않지만, FPU는 1개뿐이므로 두 번째 `FMUL` 명령어는 앞의 `FMUL`이 FPU 자원을 내놓을 때까지 무조건 멈춰야(Stall) 한다. 기술사는 타겟 애플리케이션의 수학 연산 비율을 분석하여, 구조적 해저드를 막기 위해 칩 설계 시 FPU의 개수(Issue Width)를 몇 개로 늘릴지 비용 대비 성능(Cost/Performance)을 정밀하게 타진해야 한다.

### 2. 기술사적 인사이트: 슈퍼스칼라(Superscalar)의 자원 딜레마
- **인사이트**: 한 클럭에 6개의 명령어를 동시에 집어넣는 현대의 6-wide 슈퍼스칼라 구조에서는, 6개의 명령어가 동시에 레지스터에 결과를 쓰려 할 수 있다. 이를 위해 레지스터 파일에 Write 포트를 6개나 뚫으면 레지스터 크기와 발열이 기하급수적으로 커진다. 결국 '자원을 복제해서 구조적 해저드를 없애는 전략'은 칩의 면적(Area)과 전력(Power)이라는 거대한 물리적 장벽에 부딪히게 되며, 아키텍트들은 최소한의 자원만으로 충돌을 피하는 '스케줄러'의 알고리즘 튜닝에 목숨을 건다.

- **📢 섹션 요약 비유**: 차가 막힌다고 도로를 무한정 100차선으로 넓힐 수는 없습니다. 땅값(칩 면적)이 너무 비싸기 때문입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **이상적인 IPC 접근**: 하드웨어 분리(하버드 구조 등)를 통해 1클럭 1명령어(IPC=1)의 파이프라인 이상향 달성.
- **파이프라인 단순화**: 컴파일러가 "기계가 모자랄 것"을 걱정하지 않고 순수하게 데이터 로직에만 집중하여 코드 스케줄링 가능.

### 2. 미래 전망
구조적 해저드는 반도체 미세 공정이 발달하면서 트랜지스터가 남아돌게 된 현재, 가장 먼저 '돈으로 해결된' 해저드다. 하지만 최근 생성형 AI를 위한 거대 텐서(Tensor) 연산기나 HBM(고대역폭 메모리) 통신 버스 등 새로운 특수 하드웨어들이 등장하면서, 이 거대한 자원을 여러 스레드가 어떻게 나눠 쓸 것인가 하는 '다중 스레드(SMT) 구조적 해저드'라는 새로운 형태의 병목으로 부활하여 아키텍트들을 괴롭히고 있다.

- **📢 섹션 요약 비유**: 개인 화장실은 다 마련해 줬지만, 이제는 기숙사 전체의 인터넷망(대역폭)을 서로 쓰겠다고 싸우는 새로운 시대의 갈등입니다.
