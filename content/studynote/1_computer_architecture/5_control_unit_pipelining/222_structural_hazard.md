> **핵심 인사이트**
> 1. 구조적 해저드(Structural Hazard)는 파이프라인에서 동시에 실행 중인 두 개 이상의 명령어가 **동일한 하드웨어 자원(메모리, 레지스터 포트, ALU 등)을 같은 클럭 사이클에 사용하려고 할 때 발생하는 물리적 충돌**이다.
> 2. 주로 **단일 메모리(폰 노이만 구조)**를 사용하여 명령어 인출(IF)과 데이터 접근(MEM)이 겹칠 때 발생하며, 이 경우 뒤따라오는 명령어는 파이프라인 스톨(Stall)을 겪게 된다.
> 3. 이를 원천적으로 해결하기 위해 현대 컴퓨터는 캐시 메모리를 명령어용(I-Cache)과 데이터용(D-Cache)으로 물리적으로 분리하는 **하버드 아키텍처(Harvard Architecture)**를 채택하고 있다.

---

## Ⅰ. 구조적 해저드의 발생 원리

파이프라인의 핵심은 서로 다른 명령어가 각자 다른 구역(IF, ID, EX, MEM, WB)에서 동시에 작업하는 것입니다. 그런데 만약 두 구역이 같은 자재 창고를 써야 한다면 어떻게 될까요?

- **상황 가정:** 1번 명령어는 `LOAD` 명령어로, 현재 4단계인 MEM(메모리 접근) 단계를 지나고 있습니다. 이와 동시에, 4번 명령어는 파이프라인에 막 진입하여 1단계인 IF(명령어 인출) 단계를 지나려 합니다.
- **충돌 (Conflict):** 1번 명령어는 데이터를 가져오기 위해 메모리를 읽어야 하고, 4번 명령어는 명령어를 가져오기 위해 메모리를 읽어야 합니다. 만약 컴퓨터에 메모리와 연결된 입구(포트)가 1개뿐이라면 둘은 같은 문에 끼어버립니다.

📢 **섹션 요약 비유:**
요리사 A는 볶음밥에 넣을 참기름(데이터)을 가지러 창고에 가고, 요리사 B는 다음 요리 레시피(명령어)를 가지러 같은 창고에 동시에 뛰어갔다가 비좁은 창고 문에 둘이 끼어버린 상황입니다.

---

## Ⅱ. 구조적 해저드의 주요 발생 지점

메모리 외에도 하드웨어 자원이 부족하면 파이프라인 곳곳에서 이 해저드가 일어납니다.

1. **레지스터 파일 충돌:**
   - 한 명령어는 ID 단계에서 레지스터를 '읽으려' 하고, 다른 명령어는 WB 단계에서 레지스터에 '쓰려' 할 때 발생합니다.
   - *해결:* 레지스터 파일의 포트를 여러 개 뚫어서, 쓰기 전용 포트 1개와 읽기 전용 포트 2개를 동시에 사용할 수 있게 만듭니다(Multi-ported Register).
2. **연산기(ALU) 충돌:**
   - 만약 CPU 안에 덧셈기가 하나밖에 없는데, 분기 명령어의 주소를 계산하는 덧셈과 일반 산술 `ADD` 명령어가 겹친다면 ALU가 부족해집니다.
   - *해결:* 분기 주소 계산용 덧셈기나 PC 증가용 덧셈기를 메인 ALU와 별도로 여러 개 설치하여 하드웨어를 복제합니다.

📢 **섹션 요약 비유:**
화장실(레지스터)이 하나밖에 없어서 들어가는 사람과 나오는 사람이 겹치는 문제입니다. 돈(설계 비용)을 좀 더 들여서 화장실 문을 두 개 달거나 화장실을 여러 개 지으면 해결됩니다.

## Ⅲ. 근본적 해결책: 하버드 아키텍처 (Harvard Architecture)

명령어 인출(IF)과 데이터 접근(MEM)의 충돌은 너무나 빈번하게 일어나기 때문에, 이를 매번 스톨(Stall)로 막는 것은 엄청난 성능 저하를 부릅니다. 

- **폰 노이만 구조의 한계:** 데이터와 프로그램(명령어)이 하나의 메모리 덩어리와 하나의 버스를 공유하는 구조적 한계입니다.
- **분리형 캐시 (Split Cache):** 이를 극복하기 위해 CPU에 가장 가까운 L1 캐시를 물리적으로 두 동강 냅니다.
  - **L1 I-Cache (Instruction Cache):** 명령어만 담아두고 IF 단계에서만 접근합니다.
  - **L1 D-Cache (Data Cache):** 데이터만 담아두고 MEM 단계에서만 접근합니다.
- 이렇게 길을 아예 2개로 분리해 버리면, IF와 MEM이 완전히 독립적으로 돌아가 구조적 해저드가 100% 소멸합니다.

📢 **섹션 요약 비유:**
손님들이 들어오는 정문(I-Cache)과 식재료가 들어오는 뒷문(D-Cache)을 아예 따로 만들어서, 손님과 배달원이 문에서 마주쳐 부딪히는 일을 원천적으로 차단한 완벽한 식당 구조입니다.

---

## Ⅳ. 하드웨어 복제 vs 비용의 딜레마

구조적 해저드의 해결책은 아주 명확합니다. **"부족한 부품을 똑같이 하나 더 만들어서 복제해 두면 된다(Resource Duplication)."**

- 하지만 ALU, 캐시, 버스 등을 마구잡이로 늘리면 CPU의 칩 면적(Die Size)이 무지막지하게 커지고 전력 소모와 발열이 감당할 수 없게 됩니다.
- 따라서 설계자들은 "가장 자주 부딪히는 자원(L1 캐시 분리, 레지스터 다중 포트)만 돈을 들여 복제하고, 어쩌다 한 번 부딪히는 자원(예: FPU 곱셈기)은 그냥 스톨(Stall)을 발생시켜 1클럭 쉬게 하는 것"이 경제적이라는 트레이드오프 결정을 내립니다.

📢 **섹션 요약 비유:**
주방장이 바쁘다고 해서 비싼 오븐을 10대씩 사둘 수는 없습니다. 자주 쓰는 프라이팬은 10개 사두지만, 가끔 쓰는 오븐은 1대만 두고 차례대로 기다리게 하는 것이 식당 경영의 핵심입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Pipeline Hazards]`: 파이프라인의 멈춤을 유발하는 3대 충돌 중 물리적 자원의 부족 문제
- `[Von Neumann Bottleneck]`: 구조적 해저드의 가장 근본적인 원인 (단일 메모리 공유)
- `[Harvard Architecture]`: 메모리를 2개로 쪼개어 구조적 해저드를 박살 낸 아키텍처
- `[Split Cache (L1 I-Cache / D-Cache)]`: 하버드 아키텍처가 현대 CPU 내부에 적용된 실제 모습
- `[Resource Duplication]`: 구조적 해저드를 피하기 위해 하드웨어를 물리적으로 복제하는 설계 기법

## 👶 어린이를 위한 3줄 비유 설명
1. 구조적 해저드는 두 명의 요리사가 동시에 '하나밖에 없는 국자'를 쓰려고 싸우는 상황이에요.
2. 둘 중 한 명은 다른 요리사가 국자를 다 쓸 때까지 멍하니 기다려야(스톨) 하니까 요리가 늦어지죠.
3. 이걸 막기 위해 사장님은 돈을 조금 더 써서 '국자를 여러 개 사다 놓는 방법(하버드 아키텍처, 하드웨어 복제)'을 썼답니다!
