> **핵심 인사이트**
> 1. 예약역(Reservation Station, RS)은 토마술로 알고리즘(Tomasulo's Algorithm)의 핵심 자료구조로, **명령어들이 연산기(ALU)에 진입하기 직전에 대기하며 필요한 데이터를 끌어모으는 분산형 대기실**이다.
> 2. 명령어가 예약역에 들어온 순간부터, CPU는 메모리나 중앙 레지스터 파일을 쳐다보지 않고 오직 데이터 버스(CDB)를 감시(Snooping)하며 **자신에게 필요한 데이터가 흘러가면 즉시 낚아채어 연산을 시작**한다.
> 3. 이를 통해 데이터가 준비된 명령어부터 순서에 상관없이 연산기에 밀어 넣는 **진정한 의미의 데이터 주도형 비순차 실행(Data-driven Out-of-Order Execution)이 가능해진다.**

---

## Ⅰ. 예약역의 등장 배경: 중앙 집중의 한계

초기 컴퓨터는 데이터 해저드를 막기 위해 '스코어보딩(Scoreboarding)'이라는 중앙 집중식 통제 장치를 썼습니다.

- **문제점:** 중앙의 스코어보드가 모든 명령어의 상태를 1초에 수십억 번씩 체크하며 "너는 출발해!", "너는 기다려!"라고 지시하다 보니, 스코어보드 자체가 엄청난 병목(Bottleneck)이 되어버렸습니다.
- **토마술로의 철학:** "중앙에서 통제하지 말고, 각 덧셈기, 곱셈기 앞에 작은 대기실(예약역)을 하나씩 지어주자. 명령어들이 각자 자기 대기실에 앉아서 자기에게 필요한 데이터가 올 때까지만 알아서 기다렸다가, 다 모이면 스스로 연산기 안으로 뛰어들게 만들자!"

📢 **섹션 요약 비유:**
공항에서 한 명의 안내원이 100명의 탑승 수속을 혼자 다 지시하는 게 아니라, 게이트(연산기)마다 작은 대기실(예약역)을 따로 두고 승객들이 알아서 자기 비행기가 오면 타게 만드는 분산 시스템입니다.

## Ⅱ. 예약역(RS)의 구조와 동작

각각의 예약역은 여러 개의 필드(칸)로 구성된 하드웨어 테이블입니다.

**주요 필드:**
1. **Op:** 어떤 연산인가? (예: `ADD`)
2. **Qj, Qk:** 내가 기다리고 있는 데이터가 '어떤 연산기'에서 나올 결과인가? (예: "나는 곱셈기 1번의 결과가 필요해")
3. **Vj, Vk:** 데이터가 이미 준비되어 있다면, 그 진짜 값은 무엇인가? (예: "하나는 이미 숫자 5가 들어와 있어")
4. **A (Address):** 메모리 주소 계산을 위한 필드
5. **Busy:** 이 예약역 칸이 현재 사용 중인가?

**동작 과정:**
- 명령어가 디코드되면 즉시 빈 예약역 칸으로 들어갑니다(Issue).
- 필요한 데이터가 레지스터에 이미 있으면 `V` 필드에 적고, 아직 이전 명령어가 계산 중이라면 `Q` 필드에 "누구의 결과를 기다리는 중"이라고 적어둡니다.
- **CDB 스누핑:** 예약역은 공용 데이터 버스(CDB)를 뚫어지게 쳐다봅니다. 기다리던 데이터가 버스에 나타나면 낚아채서 `V` 칸에 채워 넣습니다.
- `Vj`, `Vk` 두 칸이 모두 채워지는 순간! 예약역은 해당 명령어를 즉시 ALU로 쏘아 보냅니다.

📢 **섹션 요약 비유:**
식당 테이블(예약역)에 앉아서 "나는 A세트를 기다리는 중"이라고 팻말(Q 필드)을 올려놓고, 서빙 로봇(CDB)이 지나갈 때 A세트가 보이면 잽싸게 낚아채서 바로 밥을 먹는(연산 시작) 셀프서비스 시스템입니다.

---

## Ⅲ. 공통 데이터 버스 (CDB, Common Data Bus)

예약역이 완벽하게 돌아가려면 **CDB**라는 고속도로가 반드시 필요합니다.

- ALU나 FPU에서 계산이 하나 끝날 때마다, 그 결과값은 레지스터 파일로 가지 않고 즉시 이 거대한 고속도로(CDB)에 뿌려집니다(Broadcast).
- 이때 결과값 옆에는 "나는 곱셈기 1번이 만든 데이터야"라는 꼬리표(Tag)가 붙어 있습니다.
- 모든 예약역은 이 CDB에 연결되어 있으며, 자기가 기다리던 꼬리표가 지나가면 마치 붕어빵을 낚아채듯 데이터를 가져갑니다. 덕분에 데이터 포워딩(Bypassing)이 극도로 효율적이고 우아하게 처리됩니다.

📢 **섹션 요약 비유:**
주방장이 요리를 완성해서 컨베이어 벨트(CDB)에 올리면서 "이건 3번 테이블 거!"라고 외치면, 3번 테이블(예약역) 손님이 그걸 듣고 바로 가져가는 완벽한 방송(Broadcast) 시스템입니다.

---

## Ⅳ. 레지스터 리네이밍의 물리적 실체

놀랍게도, 예약역 자체가 하드웨어적인 '레지스터 리네이밍(Register Renaming)'을 수행하는 숨겨진 마법사입니다.

- 명령어가 예약역에 들어갈 때, 목적지 레지스터(예: R1)의 값은 더 이상 중요하지 않아집니다. 뒤따라오는 명령어들은 "나는 R1을 기다린다"가 아니라, **"나는 방금 R1을 쓰겠다고 한 저 예약역 3번 방(RS3)의 결과물을 기다린다"**로 타겟을 바꿔버립니다.
- 즉, 프로그래머가 쓴 R1, R2 같은 이름은 사라지고, 오직 하드웨어 내부의 '예약역 번호'들끼리 꼬리를 물고 데이터를 주고받게 됩니다. 이것이 하드웨어 기반 리네이밍의 진짜 실체입니다.

📢 **섹션 요약 비유:**
"김철수(R1)한테 서류 받아와"가 아니라, "저기 파란 모자 쓴 사람(예약역 3번)한테 서류 받아와"로 이름을 아예 무시하고 사람 자체를 지목해버리는 현명한 방식입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Tomasulo's Algorithm]`: 예약역과 CDB를 고안하여 비순차 실행의 뼈대를 만든 알고리즘
- `[Out-of-Order Execution]`: 예약역 덕분에 실현되는, 데이터가 준비된 순서대로의 병렬 실행
- `[CDB (Common Data Bus)]`: 연산 결과를 모든 예약역에 동시에 방송하여 데이터 해저드를 없애는 통로
- `[Register Renaming]`: 예약역 번호를 레지스터 이름 대신 사용함으로써 달성되는 가짜 의존성(WAR, WAW) 해소 기술
- `[Reorder Buffer (ROB)]`: 예약역에서 제멋대로 끝난 결과들을 다시 정돈하기 위해 예약역 뒤에 붙어있는 거대한 버퍼

## 👶 어린이를 위한 3줄 비유 설명
1. 예약역은 놀이기구(연산기)를 타기 전에 친구들을 기다리는 '대기실'이에요.
2. 덧셈 놀이기구 앞 대기실에 앉아서, 필요한 숫자 블록(데이터)이 앞사람한테서 날아올 때까지 눈을 부릅뜨고 기다리죠.
