# 예약역 (Reservation Station)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 예약역(Reservation Station)은 비순차 실행(Out-of-Order Execution) 프로세서에서 명령어가 실제 실행 유닛(ALU, FPU 등)으로 발급되기 전에 피연산자가 도착할 때까지 대기하는 하드웨어 버퍼다.
> 2. **가치**: 데이터 의존성으로 인해 실행할 수 없는 명령어를 보류해 두고, 피연산자가 준비된 명령어부터 우선적으로 실행 유닛에 보내어 **명령어 수준 병렬성(ILP)을 극대화**한다.
> 3. **융합**: 레지스터 리네이밍, 재주문 버퍼(ROB)와 함께 토마술로 알고리즘의 핵심 구성요소로, 현대 고성능 CPU의 비순차 실행 엔진 필수 요소다.

+++

## Ⅰ. 예약역의 개요

### 1. 정의
- 예약역은 각 기능 유닛(정수 연산, 부동소수점 연산, 메모리 접근 등) 앞에 배치된 작은 버퍼로, 아직 피연산자가 준비되지 않은 명령어들을 임시로 저장하고 모니터링하는 하드웨어 구조다.

### 2. 등장 배경: 데이터 해저드의 극복
- 순차 실행 CPU에서는 앞선 명령어의 결과가 나올 때까지 다음 명령어가 기다려야 했다(Stall). 하지만 비순차 실행에서는 "피연산자가 준비 안 됐어? 그럼 옆에서 기다리고, 준비된 다른 명령어부터 실행하자"라는 전략이 필요하다. 이 '대기실'이 바로 예약역이다.

### 3. 💡 비유: '공항 라운지의 대기 게이트'
- 예약역은 **'비행기(실행 유닛)를 기다리는 승객(명령어)들이 탑승권(피연산자)을 받을 때까지 머무는 라운지'**와 같다.
- 승객이 여권(피연산자)이 없으면 비행기에 탈 수 없다. 라운지에서 여권이 도착할 때까지 기다리며, 여권이 준비된 승객부터 즉시 비행기에 탑승(Issue)한다.

- **📢 섹션 요약 비유**: 식당에서 주문한 요리(명령어)가 재료(피연산자)가 준비될 때까지 주방 입구에서 대기하는 대기표 시스템입니다.

+++

## Ⅱ. 메커니즘 및 동작 원리 (Deep Dive)

### 1. 예약역의 동작 사이클
1. **명령어 디스패치(Dispatch)**: ROB나 이슈 큐에서 명령어가 예약역으로 전달됨. 피연산자가 준비되지 않았다면 태그(Tag)만 저장.
2. **피연산자 대기(Wait)**: CDB(Common Data Bus)를 모니터링하며 필요한 피연산자가 도착하는지 감시.
3. **준비 완료(Ready)**: 모든 피연산자가 도착하면 Ready 상태가 됨.
4. **발급(Issue)**: 해당 실행 유닛이 비어 있으면 즉시 명령어를 실행 유닛으로 보냄.

### 2. 예약역의 내부 구조 (ASCII)
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │              정수 연산용 예약역 (Integer RS) 예시                  │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  ┌─────────┬──────────┬──────────┬───────────┬──────────┐     │
    │  │ Entry # │ Op       │ Vj       │ Vk        │ Qj/Qk    │     │
    │  ├─────────┼──────────┼──────────┼───────────┼──────────┤     │
    │  │   0     │ ADD      │  0x1000  │  0x2000   │   -      │ ← Ready!    │
    │  │   1     │ SUB      │    -     │  0x3000   │  ROB#5   │ ← Qj 대기 중│
    │  │   2     │ MUL      │  0x4000  │    -      │  ROB#8   │ ← Qk 대기 중│
    │  │   3     │ (비어있음)│    -     │    -      │    -     │             │
    │  └─────────┴──────────┴──────────┴───────────┴──────────┘     │
    │                                                                 │
    │  - Vj, Vk: 실제 값(Value)이 이미 있는 경우                      │
    │  - Qj, Qk: 값을 생산할 ROB 엔트리 번호 (아직 안 왔음)            │
    │  - CDB 감시: ROB#5에서 결과가 나오면 Entry #1의 Vj에 복사됨      │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. CDB(Common Data Bus)와의 연동
- **CDB**: 모든 실행 유닛의 결과가 한 곳으로 모여서 방송(Broadcast)되는 공용 데이터 버스.
- 예약역은 CDB를 계속 감시하다가, 자신이 기다리던 태그(ROB#5 등)의 결과가 나오면 즉시 캡처하여 피연산자 슬롯을 채운다.

- **📢 섹션 요약 비유**: 경매장에서 자신이 원하는 물건(피연산자) 번호가 호명되면 즉시 손을 들어 낙찰받는 시스템입니다.

+++

## Ⅲ. 예약역 유형별 비교 (Trade-off)

| 특성 | 통합 예약역 (Unified RS) | 분리 예약역 (Partitioned RS) |
|:---|:---|:---|
| **구조** | 모든 실행 유닛이 하나의 큰 RS 공유 | 각 실행 유닛마다 전용 RS 보유 |
| **유연성** | 높음 (어떤 명령어든 빈 슬롯 사용) | 낮음 (정수 명령어는 정수 RS만) |
| **복잡도** | 높음 (큰 CAM 구조 필요) | 낮음 (작은 RS 여러 개) |
| **전력 소모** | 높음 | 낮음 |
| **대표 사례** | Intel P6 아키텍처 | 현대 ARM, Apple Silicon |

### 심층 분석: 예약역 크기와 성능
- **작은 RS**: 쉽게 꽉 차서 명령어 발급이 막힘. IPC 저하.
- **큰 RS**: 더 많은 명령어를 동시에 추적 가능하지만, 면적과 전력 비용 증가.
- **현대 트렌드**: Apple M 시리즈는 유난히 큰 예약역을 가져 높은 IPC 달성.

+++

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: Wake-up & Select 로직
- 예약역의 핵심 회로는 **Wake-up** (피연산자 도착 감지)와 **Select** (준비된 명령어 중 무엇을 먼저 보낼지 결정)다.
- 클럭 주파수를 높이려면 이 두 로직의 지연 시간을 최소화해야 한다. 이를 위해 **스케줄링 윈도우(Scheduling Window)** 크기와 **발급 폭(Issue Width)**을 균형 있게 설계해야 한다.

### 2. 안티패턴: 예약역 부족으로 인한 병목
- 실행 유닛은 여유로운데 예약역이 꽉 차서 새 명령어를 받지 못하는 상황.
- 해결: 예약역 크기 확대 또는 백엔드 스케줄링 최적화.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **IPC 향상**: 피연산자 준비된 명령어부터 실행하여 스톨 최소화.
- **하드웨어 활용률 증대**: 실행 유닛이 유휴 상태로 낭비되는 것을 방지.

### 2. 미래 전망
- AI 워크로드 특성을 고려한 예약역 설계(예: 행렬 연산 최적화)가 연구되고 있다.
- 예측적 스케줄링(Predictive Scheduling)을 위해 머신러닝 기법이 예약역 관리에 도입될 가능성.

## 📌 관련 개념 맵 (Knowledge Graph)
- **토마술로 알고리즘 (Tomasulo's Algorithm)**: 예약역을 처음 제안한 알고리즘.
- **재주문 버퍼 (ROB)**: 예약역에서 실행 완료된 결과가 순서대로 커밋되도록 관리.
- **레지스터 리네이밍 (Register Renaming)**: 예약역과 협력하여 가짜 의존성 제거.
- **비순차 실행 (Out-of-Order Execution)**: 예약역이 구현하는 핵심 기법.
