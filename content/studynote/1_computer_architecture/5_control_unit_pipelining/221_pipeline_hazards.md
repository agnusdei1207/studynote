# 221. 파이프라인 해저드 (Pipeline Hazards)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파이프라인 실행을 방해하는 상황
> 2. **가치**: CPI 증가, 성능 저하의 원인
> 3. **융합**: 구조적/데이터/제어 해저드, 해결 기법

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

파이프라인 해저드(Pipeline Hazards)는 **파이프라인에서 다음 클럭에 명령어를 실행할 수 없는 상황**으로, CPI를 1보다 크게 만들어 성능을 저하시킨다.

### 💡 비유: 공장 라인의 문제

파이프라인 해저드는 **공장 조립 라인의 문제**와 같다. 구조적 해저드는 도구 부족(드릴이 하나), 데이터 해저드는 부품 대기(아직 도착 안 함), 제어 해저드는 작업 변경(고객 주문 변경)이다.

### 파이프라인 해저드 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                파이프라인 해저드 구조                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【해저드 3종류】                                                      │
│  ──────────────────                                                  │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 구조적 해저드 (Structural Hazard)                             │ │
│  │     • 원인: 하드웨어 리소스 부족                                    │ │
│  │     • 예: 메모리 하나에 IF와 MEM 동시 접근                         │ │
│  │     • 해결: 리소스 분리 (Harvard 구조, I-Cache + D-Cache)           │ │
│  │     • 현대 CPU: 거의 해결됨                                        │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  2. 데이터 해저드 (Data Hazard)                                    │ │
│  │     • 원인: 데이터 의존성                                          │ │
│  │     • 종류: RAW, WAR, WAW                                         │ │
│  │     • 해결: 포워딩, 스톨, 코드 재배치                                │ │
│  │     • 현대 CPU: 포워딩으로 대부분 해결                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  3. 제어 해저드 (Control Hazard)                                   │ │
│  │     • 원인: 분기로 인한 파이프라인 무효화                            │ │
│  │     • 예: BEQ 후 분기 시 이미 인출된 명령어 버려야                   │ │
│  │     • 해결: 분기 예측, 지연 분기, 스톨                                │ │
│  │     • 현대 CPU: 분기 예측으로 95%+ 정확도                           │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【해저드 영향】                                                       │
│  ──────────────────                                                  │
│                                                                     │
│  이상적 CPI = 1                                                       │
│  실제 CPI = 1 + 해저드 페널티                                          │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  해저드 페널티 계산:                                                  │ │
│  │  ──────────────────                                                  │ │
│  │  CPI = 1 + (구조적 페널티) + (데이터 페널티) + (제어 페널티)         │ │
│  │                                                               │ │
│  │  예:                                                             │ │
│  │  • 구조적: 0 (현대 CPU에서 거의 없음)                                │ │
│  │  • 데이터: 0.1 (포워딩으로 대부분 해결)                              │ │
│  │  • 제어: 0.2 (분기 예측 미스)                                        │ │
│  │  → CPI = 1 + 0 + 0.1 + 0.2 = 1.3                                  │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 파이프라인 해저드 상세

### 해저드 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                파이프라인 해저드 상세                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【구조적 해저드 상세】                                                │
│  ────────────────────────                                            │
│                                                                     │
│  문제 상황:                                                           │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  클럭  IF    ID    EX    MEM   WB                                 │ │
│  │  ────  ────  ────  ────  ────  ────                               │ │
│  │   1    I1                                                        │ │
│  │   2    I2    I1                                                  │ │
│  │   3    I3    I2    I1                                            │ │
│  │   4    I4    I3    I2    I1    ← I4(IF)와 I1(MEM) 메모리 충돌     │ │
│  │   5    I5    I4    I3    I2    I1                                │ │
│  │               ↑                                                   │ │
│  │            스톨 발생                                                │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  해결: Harvard 구조                                                   │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  ┌──────────┐    ┌──────────┐                                    │ │
│  │  │ I-Cache  │    │ D-Cache  │                                    │ │
│  │  │ (명령어)  │    │ (데이터)  │                                    │ │
│  │  └──────────┘    └──────────┘                                    │ │
│  │  → IF와 MEM이 서로 다른 캐시 사용 → 충돌 없음                      │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【데이터 해저드 상세】                                                │
│  ────────────────────────                                            │
│                                                                     │
│  RAW (Read After Write) - 가장 흔함:                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  ADD R1, R2, R3   ; R1 ← R2 + R3                                 │ │
│  │  SUB R4, R1, R5   ; R4 ← R1 - R5  (R1 아직 안 쓰여짐)             │ │
│  │                                                               │ │
│  │  클럭  IF    ID    EX    MEM   WB                                 │ │
│  │  ────  ────  ────  ────  ────  ────                               │ │
│  │   1    ADD                                                       │ │
│  │   2    SUB   ADD                                                 │ │
│  │   3          SUB   ADD    ← SUB가 R1 읽기 (아직 WB 전)             │ │
│  │   4                SUB   ADD                                     │ │
│  │                      ↑    ↓                                      │ │
│  │                   스톨   R1 기록                                   │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【제어 해저드 상세】                                                  │
│  ──────────────────                                                  │
│                                                                     │
│  분기 문제:                                                           │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  BEQ R1, R2, target  ; 분기                                       │ │
│  │  ADD R3, R4, R5      ; 분기 여부 모름                              │ │
│  │  SUB R6, R7, R8      ; 분기 여부 모름                              │ │
│  │                                                               │ │
│  │  클럭  IF    ID    EX    MEM   WB                                 │ │
│  │  ────  ────  ────  ────  ────  ────                               │ │
│  │   1    BEQ                                                       │ │
│  │   2    ADD   BEQ                                                 │ │
│  │   3    SUB   ADD   BEQ    ← EX에서 분기 결정                       │ │
│  │   4    ???   SUB   ADD   BEQ                                     │ │
│  │         ↑                                                        │ │
│  │   이미 인출된 ADD, SUB 무효 → 플러시                                │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  해저드 해결 기법 비교:                                               │ │
│  │  ──────────────────────                                              │ │
│  │  해저드     해결 기법          CPI 영향                              │ │
│  │  ───────  ────────────────  ──────────                               │ │
│  │  구조적    리소스 분리          0                                      │ │
│  │  데이터    포워딩             0~0.1                                   │ │
│  │  데이터    스톨               0.2~0.5                                 │ │
│  │  제어      분기 예측           0.05~0.2                               │ │
│  │  제어      스톨               0.2~0.5                                 │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 파이프라인 해저드 × 타 과목 융합

### 과목 융합 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                파이프라인 해저드 × 타 과목 융합                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【컴퓨터 구조】                                                      │
│  • CPI: 해저드로 CPI > 1                                              │
│  • Harvard 구조: 구조적 해저드 해결                                    │
│  • 포워딩: 데이터 해저드 해결                                          │
│  • 분기 예측: 제어 해저드 해결                                          │
│                                                                     │
│  【컴파일러】                                                          │
│  • 명령어 스케줄링: 데이터 해저드 회피                                  │
│  • 지연 슬롯: 제어 해저드 활용                                          │
│  • 루프 최적화: 분기 감소                                              │
│  • 의존성 분석: 데이터 해저드 파악                                      │
│                                                                     │
│  【운영체제】                                                          │
│  • 컨텍스트 스위치: 파이프라인 플러시                                   │
│  • 인터럽트: 제어 해저드 유발                                          │
│  • 스케줄링: 캐시/파이프라인 상태 고려                                  │
│                                                                     │
│  【알고리즘】                                                          │
│  • 분기 예측 친화적 코드                                               │
│  • 루프 구조 최적화                                                    │
│  • 데이터 지역성                                                       │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  해저드로 인한 성능 손실:                                              │ │
│  │  ──────────────────────                                              │ │
│  │  CPU Time = IC × CPI × CC                                           │ │
│  │                                                               │ │
│  │  CPI = 1 + StallCycles / InstructionCount                           │ │
│  │                                                               │ │
│  │  예: 20% 분기, 10% 미스, 3클럭 페널티                                 │ │
│  │  제어 페널티 = 0.2 × 0.1 × 3 = 0.06                                  │ │
│  │  CPI = 1 + 0.06 = 1.06                                              │ │
│  │  → 6% 성능 저하                                                      │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### 파이프라인 해저드 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                파이프라인 해저드 분석                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【해저드 분석 툴】                                                    │
│  ──────────────────                                                  │
│  $ perf stat -e stalls,cycles ./prog                                │
│  cycles: 10,000,000,000                                             │
│  stalls: 2,000,000,000                                              │
│  → 20% 스톨                                                           │
│                                                                     │
│  【분기 예측 분석】                                                    │
│  ──────────────────                                                  │
│  $ perf stat -e branches,branch-misses ./prog                       │
│  branches: 1,000,000,000                                            │
│  branch-misses: 20,000,000                                          │
│  → 2% 분기 미스율                                                     │
│                                                                     │
│  【데이터 해저드 분석】                                                │
│  ────────────────────────                                            │
│  $ perf stat -e uops_executed.stall_cycles ./prog                   │
│  → 포워딩으로 해결되지 않은 해저드 확인                                 │
│                                                                     │
│  【해저드 최적화】                                                      │
│  ──────────────────                                                  │
│  # 데이터 해저드 최소화                                                 │
│  ADD R1, R2, R3                                                      │
│  NOP                  ; 스톨 회피                                     │
│  NOP                                                                │
│  SUB R4, R1, R5    ; R1 사용                                         │
│                                                                     │
│  # 분기 예측 친화적 코드                                               │
│  if (likely(condition)) {  ; 분기 예측 힌트                           │
│      // 자주 실행되는 코드                                             │
│  }                                                                  │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  해저드 최소화 팁:                                                    │ │
│  │  ──────────────────                                                  │ │
│  │  1. 포워딩 활용 (대부분 CPU)                                         │ │
│  │  2. 분기 예측 친화적 코드 작성                                        │ │
│  │  3. 루프 언롤링 (분기 감소)                                          │ │
│  │  4. 함수 인라이닝 (분기 감소)                                        │ │
│  │  5. 의존성 없는 명령어 배치                                          │ │
│  │  6. 컴파일러 최적화 활용                                              │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• 파이프라인 해저드: 실행을 방해하는 상황
• 구조적 해저드: 리소스 부족 → Harvard 구조로 해결
• 데이터 해저드: 데이터 의존성 → 포워딩/스톨로 해결
• 제어 해저드: 분기 무효화 → 분기 예측으로 해결
• CPI = 1 + 해저드 페널티
• 현대 CPU: 구조적(0), 데이터(0~0.1), 제어(0.05~0.2)
```
