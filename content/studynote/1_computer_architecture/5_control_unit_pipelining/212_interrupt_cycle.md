# 212. 인터럽트 사이클 (Interrupt Cycle)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터럽트 사이클 (Interrupt Cycle)은 정상적인 명령어 실행 루프를 일시 중지하고, 외부 하드웨어 장치나 내부 예외에서 발생한 긴급 요청을 처리하기 위해 시스템 제어권을 인터럽트 서비스 루틴 (ISR)으로 넘기는 과정이다.
> 2. **가치**: 폰 노이만 구조의 단일 흐름(순차적 실행) 한계를 부수고, 하드웨어 타이머와 입출력(I/O) 기기의 비동기적 응답을 가능하게 하여 오늘날의 시분할 선점형 멀티태스킹 운영체제를 탄생시킨 근본적인 하드웨어 장치다.
> 3. **융합**: 문맥 교환 (Context Switching)을 위해 현재 레지스터 상태(PC, 상태 플래그 등)를 스택 메모리에 안전하게 백업하는 마이크로 오퍼레이션이 핵심이며, 깊은 파이프라인 아키텍처에서는 파이프라인 전체를 비우는 정밀한 예외 (Precise Exception) 처리 기술과 결합된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 인터럽트 사이클 (Interrupt Cycle)은 CPU가 현재 명령어의 실행(Execute) 단계를 마친 직후, 다음 명령어를 인출(Fetch)하러 가기 전에 외부 하드웨어의 긴급 신호(INTR)나 내부 예외 상황이 발생했는지 검사하고 대응하는 특별한 단계다.

- **필요성**: 인터럽트 사이클은 폰 노이만 구조의 단일 실행 흐름 한계를 부수고 **시스템의 비동기적 응답성과 멀티태스킹**을 실현하기 위해 반드시 필요하다. 입출력(I/O) 기기가 작업을 마칠 때까지 CPU가 무한 루프를 돌며 대기하는 '폴링(Polling)'의 자원 낭비를 원천 차단하고, 외부 이벤트 발생 시에만 즉각적으로 제어권을 넘겨 처리함으로써 **CPU 가동률을 극대화**하기 때문이다. 또한, 현재의 연산 문맥(Context)을 스택에 안전하게 보존하고 인터럽트 서비스 루틴(ISR)으로 점프하는 물리적 기작을 제공함으로써, 현대 운영체제가 수많은 프로세스를 시분할로 공평하게 관리할 수 있게 만드는 **하드웨어적 거버넌스의 핵심 안전장치** 역할을 수행한다.
- **💡 비유**: 요리사가 한창 파스타를 만들고 있다가 오븐에서 '타이머 알람(인터럽트)'이 울리면, 하던 요리를 잠깐 멈추고 파스타 레시피의 몇 페이지를 보던 중인지 포스트잇에 적어둔 뒤(상태 백업), 오븐으로 달려가 불을 끄고(ISR 실행) 돌아와 파스타를 마저 만드는 과정과 같다. 급한 불부터 끄고 다시 제자리로 돌아오는 하드웨어의 기민함이다.

- **등장 배경**: 과거 CPU는 I/O 장치가 작업을 마칠 때까지 계속 그 장치의 상태 레지스터를 물어보는 폴링 (Polling) 방식을 사용했으나, 이는 느린 I/O를 기다리며 엄청난 클럭 낭비를 초래했다. 이를 해결하기 위해 "장치가 작업을 마치면 CPU에게 전기 신호로 알려주자"는 인터럽트(Interrupt) 철학이 도입되었다. 이 비동기적 전기 신호를 CPU의 동기적 명령어 사이클 내에서 정식으로 받아주기 위해 만들어진 하드웨어 상태가 바로 인터럽트 사이클이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 |
|:---|:---|:---|
| **인터럽트 요청 선 (INTR / NMI)** | 외부 신호 수신 물리 채널 | 마우스, 키보드, 네트워크 카드 등이 CPU 외부 핀에 전기적 하이/로우 신호 인가 (NMI는 무시 불가능) |
| **인터럽트 플래그 (IF / IE)** | 인터럽트 허용 여부 제어 | CPU 상태 레지스터(PSW)의 특정 비트로, 0이면 INTR 신호를 의도적으로 무시(Masking)함 |
| **스택 포인터 (SP)** | 복귀 주소 백업 영역 관리 | 현재의 PC(되돌아올 주소)와 상태 값을 메모리에 안전하게 저장하기 위해 스택의 최상단 주소를 지목 |
| **인터럽트 벡터 테이블 (IVT)** | 핸들러 주소 매핑 참조 | 장치가 보낸 인터럽트 번호(Vector)를 기반으로 해당 ISR(함수)의 메모리 시작 주소를 찾아옴 |

### 인터럽트 사이클의 마이크로 오퍼레이션 흐름

실행 사이클 완료 직후, 펜딩(Pending)된 인터럽트가 있을 때 수행되는 동작이다. (되돌아올 주소인 PC를 메모리 스택에 PUSH하는 핵심 과정)

```text
  ┌─────────────────────────────────────────────────────────────┐
  │             인터럽트 사이클 (Interrupt Cycle)의 동작 흐름            │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [실행 사이클 완료] -> 인터럽트 플래그 검사 -> (인터럽트 발생 확인!)       │
  │                                                             │
  │ [T0 클럭] : MBR <- PC                                         │
  │    현재 PC(본래 다음 실행할 정상 명령어 주소)를 임시로 MBR에 피신시킨다.   │
  │                                                             │
  │ [T1 클럭] : MAR <- SP,  PC <- ISR의 시작 주소                    │
  │    메모리 주소 레지스터(MAR)에 스택 포인터(SP) 값을 넣는다. (스택 지목) │
  │    동시에 PC에는 인터럽트를 처리할 함수(ISR)의 주소를 덮어씌운다.        │
  │                                                             │
  │ [T2 클럭] : M[MAR] <- MBR,  SP <- SP - 1 (또는 +1)             │
  │    메모리(스택) 영역에 MBR에 든 원래 PC값을 기록하여 백업을 완료한다.     │
  │    스택 포인터를 갱신하여 다음 PUSH/POP 동작을 준비한다.               │
  │                                                             │
  │ [종료] -> 이제 PC가 ISR 주소를 가리키므로, 다음 [인출 사이클]은 ISR 시작!│
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 세 박자의 클럭 동작은 '하드웨어가 강제로 수행하는 서브루틴 호출'과 완벽히 일치한다. 일반적인 함수 호출은 프로그래머가 작성한 소프트웨어 명령어(CALL)에 의해 스택에 PC를 저장하지만, 인터럽트 사이클은 전기적 신호에 의해 하드웨어 제어 유닛이 자발적으로 PC를 스택(SP가 가리키는 곳)에 푸시(Push)하고, PC를 새로운 목적지(ISR)로 갈아 끼운다. 이 사이클이 무사히 끝나야만, 나중에 OS의 ISR 처리가 끝난 뒤 `IRET` (Interrupt Return) 명령어를 통해 얌전히 원래 하던 유저 프로그램으로 복귀할 수 있다.

- **📢 섹션 요약 비유**: 독서 중에 급한 전화가 오면, 지금 읽고 있던 책의 페이지 번호(PC)를 포스트잇에 적어서 서랍(스택)에 안전하게 붙여두고, 새로운 전화 대응 매뉴얼(ISR)을 펼치는 하드웨어의 생존 본능과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 폴링 (Polling) 방식 vs 인터럽트 (Interrupt) 방식

| 비교 항목 | 폴링 (Polling) | 인터럽트 (Interrupt) 방식 |
|:---|:---|:---|
| **동작 주체** | CPU가 계속해서 I/O 장치의 상태 레지스터를 읽어 검사 | 장치가 작업 완료 시 CPU에 전기적 신호로 비동기 통보 |
| **CPU 자원 낭비** | 매우 심함 (무한 루프로 헛돌며 사이클 낭비) | 없음 (통보 오기 전까지 다른 유효한 프로세스 작업 수행) |
| **반응 속도** | 코드가 검사하는 주기에 따라 가변적이고 느릴 수 있음 | 인터럽트 사이클을 통해 즉각적이고 확정적인 응답 보장 |
| **아키텍처 영향** | 단순함 (소프트웨어적 무한 루프 구현) | 하드웨어 제어선, IVT, 스택 상태 관리 등 매우 복잡해짐 |

### 파이프라인에서의 인터럽트 (정밀한 예외, Precise Exception)

수십 단계를 가지는 깊은 파이프라인 아키텍처에 인터럽트 사이클을 무작정 끼워 넣는 것은 칩 설계상 가장 큰 난제 중 하나다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         파이프라인 환경에서의 정밀한 예외(Precise Exception) 처리    │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │ Inst 1: [IF][ID][EX][MEM][WB] <- (메모리 접근 중 페이지 폴트 발생!)│
  │ Inst 2:     [IF][ID][EX][MEM][WB]                            │
  │ Inst 3:         [IF][ID][EX][MEM][WB]                        │
  │ Inst 4:             [IF][ID][EX][MEM][WB]                    │
  │                                                              │
  │ [문제 상황]                                                    │
  │ Inst 1이 파이프라인의 MEM 단계에서 에러(예외)가 발생했으나, 이미 뒤따르던 │
  │ Inst 2, 3은 비순차적으로 EX 단계를 거쳐 레지스터를 덮어썼을 수 있다.    │
  │                                                              │
  │ [해결: 정밀한 예외(Precise Exception) 메커니즘]                    │
  │ 1. Inst 1에서 발생한 예외 정보를 꼬리표처럼 명령어와 함께 뒤로 넘긴다.   │
  │ 2. 모든 명령어는 가장 마지막 'Retire(은퇴)' 단계에 도착해서야 예외를 터뜨린다.│
  │ 3. Inst 1이 예외를 터뜨리면, 뒤따라오던 Inst 2, 3, 4는 즉시 무효화     │
  │    (Flush)시키고 스택에 Inst 1의 주소를 저장하며 인터럽트 사이클을 가동. │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 파이프라인과 비순차적 실행(OoO) 환경에서는 수많은 명령어가 섞여 돌아가므로, 특정 명령어에서 인터럽트(예외)가 발생했을 때 파이프라인을 정확히 어느 선에서 자르고 멈출 것인지(어떤 PC 위치를 백업할 것인지) 판별하기가 매우 어렵다. 현대 아키텍처는 이를 해결하기 위해 '정밀한 예외'를 지원한다. 예외가 발견되더라도 즉시 파이프라인을 멈추지 않고, 재배치 버퍼(ROB)의 가장 끝단까지 기다렸다가 원래 프로그램 순서대로 은퇴(Retire)할 차례가 된 명령어에 예외 꼬리표가 묻어있을 때 비로소 파이프라인 전체를 비우고 인터럽트 사이클을 작동시킨다. 이를 통해 OS는 정확히 "Inst 1에서 문제가 생겼고, 그 이전까지의 상태는 완벽히 정상이다"라는 것을 보장받는다.

- **📢 섹션 요약 비유**: 공장 컨베이어 벨트 중간에서 불량품이 발견되어도 기계를 당장 거칠게 세우지 않고, 최종 검수대(Retire 단계)에 도착했을 때 비로소 라인을 멈추고 뒤따라오던 미완성 부품들을 다 쓸어버린 뒤, 정확히 그 불량품부터 수리를 시작하는 원리입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **OS 커널의 컨텍스트 스위칭 (Context Switching)**
   메인보드의 하드웨어 타이머 칩(PIT 또는 APIC)은 OS 커널이 설정한 시간(예: 1ms)마다 강제로 CPU에 인터럽트 신호를 보낸다. 정상적인 사이클을 돌던 CPU는 이 신호를 받고 인터럽트 사이클에 진입하여, 현재 실행 중인 유저 프로세스(예: 웹 브라우저)의 PC를 스택에 백업하고 커널의 스케줄러(ISR)로 진입한다. 스케줄러는 레지스터 전체를 저장한 뒤, 다음 차례인 프로세스(예: 음악 플레이어)의 레지스터와 PC를 복원하여 탈출한다. 이 타이머 기반의 인터럽트 사이클이 없었다면 1개의 CPU 코어에서 여러 프로그램이 동시에 도는 선점형(Preemptive) 멀티태스킹은 물리적으로 불가능했다.

2. **Top-Half와 Bottom-Half 패킷 처리 설계**
   고속 네트워크 카드(NIC)가 트래픽 패킷을 초당 수백만 개씩 밀어 넣으면 인터럽트 폭풍(Interrupt Storm)이 발생하여, CPU가 사용자 프로그램은 실행하지 못하고 인터럽트 사이클만 무한히 도는 '라이브락 (Livelock)'에 빠진다. 리눅스 커널 엔지니어는 이를 막기 위해, 인터럽트 핸들러(ISR)에서는 패킷 버퍼의 포인터만 빠르게 챙기고(Top-Half) 즉시 IRET로 복귀하는 구조를 짠다. 그리고 실제 패킷 까보기(TCP/IP 스택 처리)는 나중에 CPU가 여유 있을 때 소프트웨어 인터럽트나 커널 스레드로 처리하는 Bottom-Half(지연 처리) 구조로 분리하여 시스템 응답성을 방어한다.

### 도입 체크리스트
- **기술적**: 마스크 불가능 인터럽트 (NMI: Non-Maskable Interrupt) 핀이 전원 장애(Power Failure)나 하드웨어 메모리 오류(ECC Error) 같은 치명적 시스템 셧다운 방어 로직에 올바르게 연결되어 있는가? (어떤 상황에서도 NMI는 무시되지 않고 인터럽트 사이클을 발동해야 함).
- **운영적**: 인터럽트 사이클 발생 시 여러 번 중첩되어 스택 오버플로우가 발생하지 않도록, 커널 스택 또는 예외 전용 스택(Exception Stack) 영역이 안전하게 확보되어 있는가?

### 안티패턴
- **인터럽트 핸들러(ISR) 내에서의 무한 대기 (Blocking I/O)**
  인터럽트 사이클을 통해 진입한 핸들러 함수(ISR) 내부에서 다시 디스크 I/O를 요청하고 스레드를 잠재우거나(Sleep), 뮤텍스(Mutex) 락을 획득하려고 대기하는 행위. 이 기간 동안 CPU는 인터럽트 플래그를 꺼둔 상태이므로 다른 하드웨어의 인터럽트까지 모두 씹히게 되며, 시스템 전체가 락(Lock)에 걸려 커널 패닉이나 데드락(Deadlock)을 유발하는 가장 치명적인 디바이스 드라이버 설계 결함이다.

- **📢 섹션 요약 비유**: 응급실(인터럽트 사이클)에 실려 온 위급 환자는 무조건 지혈만 5분 안에 끝내고(Top-Half) 일반 병동으로 넘겨야지, 응급실 침대를 차지하고 몇 시간짜리 정밀 성형수술(Bottom-Half)을 하려고 들면 밖에서 기다리던 다른 응급 환자들이 다 죽어버리는 치명적 안티패턴이 됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 초기 폴링 (Polling) 구조 | 하드웨어 인터럽트 사이클 도입 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (가동률)** | I/O 대기 중 루프 돌며 CPU 가동률 100% 낭비 | I/O 중에는 다른 유효 프로세스로 전환 실행 | CPU 유효 가동률 **극대화** |
| **정량 (응답성)** | 소프트웨어 루프 길이에 따라 ms~초 단위 지연 | 전기적 하드웨어 신호로 **ns~μs 단위** 즉각 전이 | 실시간성(Real-time) 반응 속도 확보 |
| **정성 (아키텍처)**| 장치별로 타이밍 코드를 일일이 하드코딩해야 함 | IVT 기반의 통일된 예외 및 하드웨어 처리 구조 | **현대적 디바이스 드라이버 및 OS 생태계 탄생** |

### 미래 전망
- **메시지 신호 인터럽트 (MSI/MSI-X)의 표준화**: 과거에는 메인보드의 물리적인 구리선(INTR 핀, IRQ 라인)을 하드웨어적으로 당겨서 인터럽트를 발생시켰다. 하지만 선이 꼬이고 충돌(IRQ Conflict)하는 문제가 잦았다. 오늘날의 PCIe 기반 시스템에서는 물리적인 인터럽트 선을 없애고, 장치가 메모리의 특정 주소에 데이터를 쓰는(Write) 패킷을 보내면 코어 내의 APIC (Advanced Programmable Interrupt Controller)가 이를 가상의 인터럽트 신호로 해석해 인터럽트 사이클을 가동하는 MSI (Message Signaled Interrupts) 방식이 완전히 표준으로 자리 잡았다.

- **📢 섹션 요약 비유**: 옛날에는 식당 손님(하드웨어 장치)이 주방장(CPU) 옷자락을 직접 잡아당겨서(물리적인 인터럽트 핀) 요구사항을 말했다면, 이제는 테이블의 태블릿(PCIe)으로 주문을 넣으면 주방장의 스크린에 조용하고 정확하게 주문 알람(MSI)이 울리는 세련된 방식으로 진화한 것입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **명령어 사이클 (Instruction Cycle)** | 인터럽트 사이클은 기본 명령어 사이클(Fetch-Execute)의 무한 반복을 합법적으로 끊어낼 수 있는 유일한 하드웨어 탈출구다. |
| **문맥 교환 (Context Switching)** | 인터럽트 사이클이 스택에 PC를 백업해주는 물리적 동작이 OS의 멀티태스킹 문맥 교환의 하드웨어적 시작점이다. |
| **프로그램 카운터 (PC)** | 인터럽트 사이클 동안 기존의 실행 흐름 주소가 대피되고, 새로운 인터럽트 핸들러 주소로 강제 덮어씌워지는 핵심 레지스터다. |
| **폴링 (Polling)** | 인터럽트 사이클의 안티테제 개념으로, 하드웨어 통보 지원 없이 소프트웨어가 무한 루프를 돌며 상태 레지스터를 확인하는 방식이다. |
| **정밀한 예외 (Precise Exception)** | 파이프라인에서 인터럽트 사이클을 가동하기 전, 이전 명령어들의 실행 상태를 한 치의 오차 없이 깔끔하게 정리해두는 코어 설계 기법이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 요리사(CPU)가 파스타(프로그램)를 신나게 만들고 있는데, 갑자기 오븐에서 빵이 다 구워졌다고 삐빅! 알람(인터럽트)이 크게 울렸어요.
2. 그럼 요리사는 하던 파스타 레시피의 몇 페이지를 읽고 있었는지 포스트잇에 적어서 서랍에 쏙 넣어두고(상태 백업), 즉시 오븐으로 뛰어가서 빵을 꺼내요.
3. 급한 불을 끄면 다시 서랍에서 포스트잇을 꺼내 아까 만들던 파스타를 그대로 이어서 만드는 똑똑한 하드웨어의 생존 본능을 '인터럽트 사이클'이라고 한답니다!
