# 인터럽트 사이클 (Interrupt Cycle)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터럽트 사이클 (Interrupt Cycle)은 하나의 명령어 실행(Execute)이 끝난 직후, **외부 장치의 요청이나 내부 예외 상황을 확인하고 이를 처리하기 위해 현재 상태를 저장 및 분기하는 단계**이다.
> 2. **가치**: 실행 중인 프로그램의 진행 정보를 스택(Stack)에 안전하게 대피시켜 나중에 **정확한 지점으로 복귀**할 수 있게 하며, CPU가 주변 장치와 실시간으로 상호작용할 수 있는 논리적 기반을 제공한다.
> 3. **융합**: 인터럽트 벡터(Interrupt Vector)를 통해 해당 요청을 처리할 전용 루틴(ISR)으로 실행 흐름을 넘기며, 이는 현대 운영체제의 멀티태스킹과 이벤트 기반 처리를 가능케 하는 핵심 엔진이다.

---

## Ⅰ. 인터럽트 사이클의 정의 및 동작 조건

### 1. 정의
- 명령어 사이클의 마지막 단계에서 인터럽트 요청 신호를 검사하고, 요청이 있을 경우 현재 프로그램의 실행을 일시 중단한 뒤 인터럽트 처리 루틴으로 제어권을 넘기는 과정이다.

### 2. 동작 조건: '명령어 단위의 체크'
- CPU는 명령어를 실행하는 도중에는 인터럽트를 확인하지 않는다. 반드시 **실행 사이클(Execute Cycle)이 종료된 시점**에만 인터럽트 요청 라인을 확인하여 시스템의 정합성을 유지한다.

### 3. 💡 비유: '공부 중 벨소리 확인'
- 인터럽트 사이클은 **'숙제 한 문제 풀기를 끝낼 때마다(실행 종료), 현관 벨이 울리는지 확인(인터럽트 체크)하는 행동'**과 같다.
- 벨이 울리면 읽던 페이지를 접어두고(상태 저장) 문을 열어주러 가고(ISR 실행), 일이 끝나면 다시 접어둔 페이지를 펼쳐(복구) 숙제를 이어가는 것과 같다.

---

## II. 인터럽트 사이클의 단계별 마이크로 오퍼레이션 (Deep Dive)

인터럽트가 감지되었을 때 수행되는 하드웨어적 절차이다.

- **T3**: $MBR \leftarrow PC$
  - 다음에 실행할 명령어 주소(복귀 주소)를 MBR에 임시 저장한다.
- **T4**: $MAR \leftarrow Address\_of\_Stack$, $PC \leftarrow Routine\_Addr$
  - 복귀 주소를 저장할 스택 주소를 MAR에 넣고, 프로그램 카운터(PC)에는 인터럽트 서비스 루틴(ISR)의 시작 주소를 로드한다.
- **T5**: $M[MAR] \leftarrow MBR$
  - MBR에 보관했던 복귀 주소를 실제 메모리 스택에 기록한다. (이제 다음 사이클은 ISR에서 시작됨)

### ASCII 구조 다이어그램
```text
    [ Execute Cycle End ]
             │
             ▼
    [ Check Interrupt Req? ] ────(NO)────▶ [ Next Fetch Cycle ]
             │ (YES)
             ▼
    ┌──────────────────────────┐
    │    Interrupt Cycle       │
    │  1. Save PC to Stack     │ ◀── Context Saving
    │  2. Load ISR Addr to PC  │ ◀── Flow Redirection
    └──────────┬───────────────┘
               │
               ▼
    ┌──────────────────────────┐
    │  Interrupt Service       │ ──▶ [ I/O Task / Exception ]
    │  Routine (ISR)           │
    └──────────────────────────┘
```

- **📢 섹션 요약 비유**: 사고가 났을 때 현재 위치를 블랙박스(스택)에 저장하고, 즉시 구급차(ISR)를 호출하여 병원으로 이동하는 긴급 대응 프로세스와 같습니다.

## Ⅲ. 인터럽트 사이클의 주요 역할과 가치

| 역할 | 설명 | 효과 |
|:---|:---|:---|
| **문맥 보존** | PC, 상태 레지스터 등을 스택에 저장 | 인터럽트 종료 후 완벽한 복구 보장 |
| **제어권 전환** | 하드웨어적으로 ISR 시작 주소 로드 | 소프트웨어 개입 최소화로 신속 대응 |
| **실시간성 확보** | 외부 이벤트를 즉각 감지 및 처리 | 주변 장치와의 효율적 동기화 |

- **📢 섹션 요약 비유**: 하던 일을 잊어버리지 않게 책갈피를 끼워두는 꼼꼼함과, 위급 상황에 즉시 대처하는 순발력을 동시에 갖춘 단계입니다.

---

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 시스템 응답성 장애 조치
- **상황**: 마우스를 움직이는데 화면 커서가 뚝뚝 끊기며 반응함.
- **판단**: CPU가 **인터럽트 사이클**을 너무 자주 수행하고 있거나, ISR 실행 시간이 너무 길어 정상적인 명령어 사이클로 돌아오지 못하고 있는 'Interrupt Storm' 상태일 수 있다. 기술사라면 인터럽트 우선순위(Priority)를 재설정하거나, 불필요한 인터럽트 소스를 마스킹(Masking)하여 사이클의 균형을 찾아야 한다.

### 2. 기술사적 인사이트: 중첩 인터럽트 (Nested Interrupt)
- 기술사라면 인터럽트 사이클 중에 또 다른 인터럽트가 발생하는 상황을 논해야 한다. 이를 처리하기 위해 하드웨어가 인터럽트 금지(Disable) 플래그를 자동으로 설정하거나, 우선순위에 따라 스택에 중첩하여 상태를 저장하는 구조를 설명해야 한다. 이는 현대 운영체제의 커널 설계 역량과 직결되는 핵심 아키텍처 지식이다.

- **📢 섹션 요약 비조**: 불이 났을 때(인터럽트) 소방차가 출동하는 것도 중요하지만, 가는 길에 더 큰 사고가 났을 때 어떻게 대처할지(중첩 처리)까지 설계되어 있어야 진정한 고신뢰 시스템입니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **비동기 처리 완성**: 연산과 입출력의 병렬 운영 가능.
- **예외 처리 자동화**: 0으로 나누기, 잘못된 주소 접근 등 하드웨어 오류의 안전한 격리.

### 2. 미래 전망
앞으로는 멀티코어 환경에서 특정 코어에만 인터럽트 사이클 부하가 쏠리지 않게 하는 **분산 인터럽트 제어 (MSI-X 등)** 기술이 더욱 고도화될 것이다. 또한, 가상화 환경에서 가상 인터럽트 사이클의 지연을 줄이기 위한 전용 하드웨어 가속기가 보편화될 것이다. 인터럽트 사이클은 CPU가 세상의 변화를 놓치지 않고 반응하게 만드는 가장 기민한 '촉수'이다.

- **📢 섹션 요약 비유**: 묵묵히 제 갈 길을 가다가도, 누군가 부르면 즉시 돌아봐 주는 CPU의 '다정하고 유능한 배려'와 같은 단계입니다.
