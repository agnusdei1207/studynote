# 227. WAW (Write After Write)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: WAW (Write After Write, 출력 의존성)는 파이프라인에서 앞선 명령어가 특정 레지스터에 연산 결과를 '쓰기도(Write)' 전에, 뒤따라오는 명령어가 그 레지스터에 먼저 새로운 값을 '써버려(Write)' 최종적으로 남아있어야 할 값이 훼손되는 가짜 의존성(False Dependency)이다.
> 2. **가치**: 명령어의 실행 순서가 뒤집히는 비순차적 실행(OoO) 환경과, 파이프라인의 단계 수가 명령어마다 달라지는 복잡한 다중 파이프라인(Superscalar, FPU 등) 환경에서 주로 발생하는 고질적인 충돌이다.
> 3. **융합**: 이 문제 역시 WAR(반의존성)와 마찬가지로 데이터 자체의 인과관계가 아닌 "레지스터 이름(간판)의 재사용" 때문에 발생하므로, 하드웨어적 **레지스터 리네이밍 (Register Renaming)** 기법과 **ROB (Reorder Buffer)**를 통해 완벽하게 우회하고 병렬성을 극대화한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: WAW (Write After Write)는 명령어 A가 특정 레지스터에 결과값을 쓰려 하는데, 뒤따라온 명령어 B가 A보다 먼저 연산을 마치고 해당 레지스터에 자신의 결과값을 덮어써버리는 현상이다. 실행 속도 차이로 인해 최종 결과값이 뒤바뀌는 '출력 의존성(Output Dependency)'이다.

- **필요성**: WAW 해저드에 대한 처리는 다중 실행 유닛을 가진 **고성능 프로세서의 연산 무결성을 사수**하기 위해 반드시 필요하다. 정수 연산(1클럭)과 부동소수점 연산(수십 클럭)이 섞여 돌아가는 환경에서 쓰기 순서가 뒤집히는 것을 방치하면, 프로그램이 의도한 최신 데이터가 구형 데이터에 의해 파괴되는 치명적인 아키텍처 상태 불일치가 발생하기 때문이다. 따라서 WAW를 정밀하게 식별하고, **레지스터 리네이밍(Register Renaming)**을 통해 물리적 저장소를 분리하거나 **리오더 버퍼(ROB)**를 통해 퇴근 순서를 강제로 재정렬함으로써, 비순차 실행의 성능 이점을 누리면서도 데이터의 최종적인 논리적 순서를 완벽히 보존하는 핵심적인 아키텍처 관리 도구로 기능한다.
- **💡 비유**: 1번 페인트공이 벽에 빨간색을 칠하려고 페인트를 조색하는 동안(시간이 오래 걸림), 눈치 빠른 2번 페인트공이 먼저 와서 파란색을 칠해버렸는데, 뒤늦게 준비가 끝난 1번이 빨간색을 덧칠해버려 최종적으로 남아있어야 할 파란색이 완전히 지워지는 사고와 같다. 이를 막기 위해 페인트공마다 각자 다른 벽(물리 레지스터)을 주거나, 나중에 관리자가 순서를 다시 맞춰서 덧칠하게 해야 한다.

- **등장 배경**: 기본 5단 파이프라인에서는 모든 명령어가 맨 마지막 5번째 단계인 WB(쓰기)에서 철저히 순서대로 값을 쓰기 때문에 WAW가 발생할 일이 없다. 하지만 덧셈 정수 연산(1클럭)과 부동소수점 나눗셈(수십 클럭)처럼 파이프라인의 길이가 서로 다른 명령어들이 여러 개의 파이프라인에서 섞여 돌아가거나, 명령어 순서를 뒤바꾸는 비순차적 실행(OoO)이 도입되면서 '쓰기(Write)'의 순서가 역전되는 치명적인 상태 불일치가 발견되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 의존성의 방향 (WAW 메커니즘)

| 명령어 순서 | 어셈블리 예시 | 의존성 방향 | 설명 |
|:---|:---|:---|:---|
| **Inst 1 (선행, 느림)** | `DIV $t1, $t2, $t3` | `$t1` **생성 (느림)** | 연산이 매우 복잡해 `$t1`에 한참 뒤에 쓴다. |
| **Inst 2 (후행, 빠름)** | `ADD $t1, $t4, $t5` | `$t1` **생성 (빠름)** | 연산이 단순해 `$t1`에 순서를 앞질러 쓴다. |

### 파이프라인 길이 차이로 인한 WAW 발생 시각화

명령어 종류에 따라 WB(쓰기) 단계에 도달하는 물리적 시간이 다를 때 발생하는 명백한 모순이다.

```text
  ┌───────────────────────────────────────────────────────────────────────────┐
  │         실행 시간이 다른 명령어 간의 WAW (출력 의존성) 발생 원리          │
  ├───────────────────────────────────────────────────────────────────────────┤
  │                                                                           │
  │ [소프트웨어 프로그램의 원래 순서 (In-Order 논리적 기댓값)]                │
  │ 1. DIV $t1, $t2, $t3  (나눗셈이라 EX 단계가 10클럭 걸림)                  │
  │ 2. ADD $t1, $t4, $t5  (단순 덧셈이라 EX 단계가 1클럭이면 끝남)            │
  │ -> 논리적 기댓값: 1번이 $t1에 쓰고, 나중에 2번이 $t1에 덮어써서           │
  │                 최종적으로는 'ADD의 결과'가 $t1에 무조건 남아있어야 함.   │
  │                                                                           │
  │ [CPU 내부의 실제 하드웨어 실행 순서 (파이프라인 완료 시점 역전)]          │
  │                                                                           │
  │ 시간 (T+3) : Inst 2 (ADD) 실행 완료! -> $t1에 잽싸게 결과를 덮어씀.       │
  │ 시간 (T+12): Inst 1 (DIV) 뒤늦게 완료! -> 앗, 이제 내 차례네? 하고        │
  │              $t1에 나눗셈 결과를 덮어써버림.                              │
  │                                                                           │
  │ [물리적 모순 결과]: 프로그램 종료 후 $t1에는 최신 값인 ADD의 결과가       │
  │        남아있어야 하는데, 오래된 DIV의 쓰레기 결과가 영원히 남아버리는    │
  │        심각한 아키텍처 상태 불일치가 발생! (WAW 해저드)                   │
  └───────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** `$t1`이라는 레지스터 입장에서 보자. 프로그램 코드 순서상 무조건 나중에 불린 2번 명령어(ADD)의 결과가 최종적으로 남아야 정상이다. 하지만 1번 명령어가 너무 무거운 연산이라 늦게 퇴근(WB)하는 바람에, 일찍 퇴근해서 올바른 결과물을 놔둔 2번 명령어의 결과 위를 무자비하게 덮어버렸다. 이렇게 쓰기(Write)의 물리적 타이밍이 엇갈려 최종 산출물이 구형 데이터로 오염되는 현상이 출력 의존성의 본질이다.

- **📢 섹션 요약 비유**: 어제 출발한 일반 우편(DIV)과 오늘 출발한 특급 택배(ADD)가 있을 때, 특급 택배가 먼저 우편함(레지스터)에 무사히 도착했는데, 다음 날 도착한 일반 우편이 우편함의 특급 택배를 바닥으로 밀어내고 자리를 차지해 버려서 수취인(프로그램)이 옛날 물건을 받게 되는 꼴입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### WAR (반의존성) vs WAW (출력 의존성)

두 해저드 모두 레지스터 이름(간판)이 겹쳐서 발생하는 가짜 의존성(False Dependency)의 일종이다.

| 비교 항목 | WAR (Write After Read) | WAW (Write After Write) |
|:---|:---|:---|
| **한글 명칭** | 반의존성 (Anti-dependency) | 출력 의존성 (Output Dependency) |
| **순서 규칙** | 앞이 읽고 -> 뒤가 씀 | 앞이 쓰고 -> 뒤가 다시 덮어씀 |
| **타이밍 역전 시** | 뒤가 너무 일찍 써서 **앞이 잘못된 값을 읽음** | 뒤가 너무 일찍 써서 **앞이 최종 값을 덮어버림** |
| **보존 대상** | 앞 명령어의 **입력값** 보존이 중요 | 뒤 명령어의 **최종 출력값** 보존이 중요 |
| **해결 메커니즘** | 레지스터 리네이밍 | **레지스터 리네이밍 및 ROB 정렬** (동일) |

### 왜 WAW도 가짜 의존성인가?

위의 예시를 다시 보면, 앞의 `DIV`와 뒤의 `ADD`는 사실 서로 아무런 데이터를 주고받거나 참고하지 않는다. 단지 우연히 똑같이 `$t1`이라는 간판(이름표)을 가진 빈 상자가 출력용으로 필요했을 뿐이다. 만약 빈 상자가 넉넉해서 서로 다른 상자(예: `$t1`, `$t2`)를 썼다면, 누가 먼저 연산을 끝내든 늦게 끝내든 아무런 상관이 없는 완벽히 독립된 명령어들이다.

- **📢 섹션 요약 비유**: 회사에서 1번 부서와 2번 부서가 전혀 다른 일을 하는데, 우연히 똑같이 "공용 비품 창고(t1)"라는 공간을 쓰려고 해서 짐이 섞이는 것과 같습니다. 창고를 "1번 부서 전용 창고", "2번 부서 전용 창고"로 나눠주기만 하면 이 충돌은 즉시 100% 사라집니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **레지스터 리네이밍 (Register Renaming)을 통한 WAW 완벽 제압**
   최신 슈퍼스칼라 아키텍처(비순차 실행 엔진)는 WAR와 동일한 방식인 리네이밍으로 WAW를 제압한다. 칩 내부의 거대한 '물리 레지스터 파일(PRF)'을 이용해, 디코딩 단계에서 레지스터 이름표를 갈아 끼운다.
   - 변환 전: `DIV $t1, $t2, $t3`  /  `ADD $t1, $t4, $t5`
   - 변환 후: `DIV P5, P1, P2`      /  `ADD P10, P3, P4` (새로운 물리 레지스터 P10 연속 할당)
   이제 `DIV`는 어떤 일이 있어도 무조건 `P5`에 값을 쓰고, `ADD`는 무조건 `P10`에 값을 쓴다. 물리적 그릇이 다르니 누가 먼저 끝나든 서로의 결과값을 덮어쓸 일이 절대 없다. 
   
2. **리오더 버퍼 (ROB, Reorder Buffer)의 활약**
   이렇게 흩어진 물리 레지스터들 중에서 "진짜 최종적으로 남아야 할 `$t1`의 값은 무엇인가?"를 결정해야 한다. CPU 백엔드에는 ROB(Reorder Buffer)라는 퇴근 대기실이 있다. 명령어들이 비순차적으로 제멋대로 완료(Write)되더라도, ROB는 이들을 다시 원래 프로그램 순서대로 줄을 세워 정렬시킨다. 가장 나중에 배치된 `ADD(P10)` 명령어가 은퇴(Retire/Commit)할 때, CPU는 맵핑 테이블을 보고 "아, 프로그램이 원하는 진짜 최신 `$t1`은 `P10`이구나"라고 확정 짓고 이전의 `P5`는 쓰레기통에 버려(가비지 컬렉션) 아키텍처 상태를 완벽히 유지한다.

### 도입 체크리스트
- **기술적**: 명령어들이 꼬인 순서대로 무작정 레지스터에 기록되는 것을 막고, 원래의 프로그래밍 순서대로 정렬하여 상태를 확정하는 **리오더 버퍼(ROB)**와 **커밋(Commit) 단계**의 하드웨어 설계가 완벽하여, WAW 상황에서도 가장 마지막 명령어가 쓴 값만이 외부(Architectural State)로 확정 노출되는가?

### 안티패턴
- **인위적인 누적 변수(Accumulator) 사용 고집 코딩**
  `sum = sum + a; sum = sum + b;` 처럼 루프를 돌 때마다 하나의 좁은 변수 이름(`sum`)에 계속해서 값을 덮어쓰는 C/C++ 코딩. 이는 RAW와 WAW 해저드를 동시에 극심하게 유발한다. 현대 컴파일러가 이런 변수를 여러 개의 임시 변수(`sum1, sum2`)로 쪼개는 스칼라 대체(Scalar Replacement) 최적화를 해주지만, 복잡한 포인터 체인 환경에서는 이 최적화가 풀려버린다. 성능이 중요한 핫스팟(Hot-spot) 코드를 짤 때는, 개발자가 굳이 하나의 변수 이름표를 아끼려 들지 말고 명시적으로 여러 변수를 풀어쓰는 것이 파이프라인의 하드웨어 리네이밍을 적극 돕는 좋은 습관이다.

- **📢 섹션 요약 비유**: 커피숍에서 모든 손님의 주문을 "1번 진동벨" 하나로만 통일해서 주면 나중에 커피가 순서 섞여서 나올 때 누구 건지 다 엎어지고 섞여버립니다(WAW 충돌). 손님마다 고유한 개별 진동벨(리네이밍)을 넉넉하게 나눠주는 시스템이 꼭 필요합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 방어 기술 | WAW 방치 (비순차/다중 파이프 환경) | 리네이밍 + ROB 적용 후 | 비교 개선 효과 |
|:---|:---|:---|:---|
| **정량 (병렬성)** | 출력 충돌 훼손 우려로 파이프라인 정지 (Stall) | 이름 충돌 0% -> **명령어 병렬 실행 극대화** | 명령어 처리량(IPC) 100% 상승 |
| **정량 (파이프라인)**| 실행 시간이 긴 명령어(DIV 등) 뒤에는 무조건 막힘 | 무관한 명령어는 지연 없이 즉각 완료 통과 | 다중 파이프라인(FPU, 텐서 유닛) 효율 100% 가동 |
| **정성 (예외 처리)**| 순서가 꼬여 누가 최종값인지 몰라 예외 복구 불가능 | ROB를 통해 정확한 순차적 롤백(Roll-back) 가능 | 정밀한 예외(Precise Exception) 처리 완벽 보장 |

### 미래 전망
- **수백 개의 물리 레지스터(PRF) 자원 경쟁과 최적화**: 현대 슈퍼스칼라 CPU는 코어 하나에 200~300개에 달하는 물리 레지스터를 미친 듯이 때려 박아 가짜 의존성(WAW, WAR)을 막대한 자본과 실리콘 면적으로 박살 냈다. 하지만 이 물리 레지스터들은 SRAM 면적을 엄청나게 차지하며, 배선을 늘리고 전력 소모를 끔찍하게 높인다. 미래 아키텍처는 레지스터 파일의 크기를 무한정 늘리는 대신, 수명이 다한(더 이상 어떤 명령어도 읽지 않는 옛날 간판들) 레지스터를 단 1클럭 만에 귀신같이 찾아내어 즉시 수거(Garbage Collection)하고 재할당하는 초정밀 AI 기반 하드웨어 알고리즘 최적화 경쟁으로 넘어가고 있다.

- **📢 섹션 요약 비유**: WAW 해저드는 제한된 좁은 주차장(16개의 논리 레지스터)에 수천 대의 차가 들어오려다 주차선이 겹쳐서 박아버리는 사고였지만, 하드웨어가 몰래 수백 대 규모의 거대한 지하 주차장(PRF)을 파놓고 눈에 보이지 않게 발렛 파킹(리네이밍)을 해주는 환상적인 설계 덕분에 완전히 퇴치되었습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **데이터 해저드 (Data Hazard)** | RAW, WAR, WAW로 나뉘며 파이프라인을 멈추게 만드는 3가지 논리적/물리적 엉킴 현상이다. |
| **가짜 의존성 (False Dependency)** | WAR와 WAW의 본질로, 데이터가 오가는 게 아니라 '간판 이름(레지스터)'만 우연히 같아서 생긴 일시적 충돌이다. |
| **레지스터 리네이밍 (Register Renaming)** | 하드웨어가 런타임에 몰래 논리적 레지스터 번호를 고유한 물리 번호로 갈아 끼워 가짜 의존성을 완벽히 파괴하는 마술이다. |
| **ROB (Reorder Buffer)** | 비순차 실행으로 인해 순서가 뒤죽박죽으로 완료(Write)된 명령어들을, 퇴근할 때 다시 원래 줄대로 정렬시켜 WAW 사고를 막아주는 최종 수문장이다. |
| **비순차적 실행 (Out-of-Order Execution)** | 뒷명령어를 앞명령어보다 먼저 실행시키는 극단적 성능 최적화 기술로, 사실상 WAW 해저드를 세상에 탄생시킨 장본인이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 1번 친구가 아주 크고 복잡한 그림(느린 명령어)을 도화지에 끙끙대며 그리고 있었어요.
2. 성격 급한 2번 친구는 간단한 낙서(빠른 명령어)를 똑같은 도화지에 순식간에 휘리릭 그려버렸죠.
3. 근데 1번 친구가 뒤늦게 거대한 그림을 다 그리고 도화지를 덮어버려서, 나중에 남아야 할 2번 친구의 최신 낙서가 지워져 버렸어요! 이 억울한 순서 뒤바뀜 사고를 'WAW(쓰고 또 덮어쓰기) 해저드'라고 한답니다!
