# 227. WAW (Write After Write)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쓰기 후 쓰기, 출력 의존성 (Output Dependency)
> 2. **가치**: 파이프라인에서는 드묾, 비순차 실행에서 문제
> 3. **융합**: 레지스터 리네이밍, ROB, 비순차 실행

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

WAW(Write After Write)는 **두 명령어가 같은 위치에 쓰려 할 때 발생하는 출력 의존성**이다. 프로그램 순서대로 마지막 쓰기 결과가 유지되어야 하므로, 쓰기 순서가 보장되어야 한다.

### 💡 비유: 칠판에 글쓰기

WAW는 **칠판에 글쓰기**와 같다. A가 "Hello"라고 쓰고(Write), B가 "World"라고 쓴다(Write). 순서가 바뀌면 최종 결과가 달라진다. 반드시 순서대로 써야 한다.

### WAW 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                WAW 구조                                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【WAW 정의】                                                          │
│  ──────────────────                                                  │
│  명령어 i: R1 ← ...        (Write to R1)                             │
│  명령어 j: R1 ← ...        (Write to R1)                             │
│  i와 j가 같은 레지스터에 쓰려고 함                                       │
│  → 최종 결과는 j의 쓰기가 되어야 함                                      │
│                                                                     │
│  【발생 예시】                                                         │
│  ──────────────────                                                  │
│  ADD R1, R2, R3   ; R1 ← R2 + R3  (Write R1)                        │
│  SUB R1, R4, R5   ; R1 ← R4 - R5  (Write R1)                        │
│                                                                     │
│  → 두 명령어가 모두 R1에 씀                                             │
│  → SUB의 결과가 최종 R1이 되어야 함                                      │
│                                                                     │
│  【파이프라인에서의 WAW】                                               │
│  ────────────────────────                                            │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  ADD R1, R2, R3   ; Write R1                                     │ │
│  │  SUB R1, R4, R5   ; Write R1                                     │ │
│  │                                                               │ │
│  │  클럭  IF    ID    EX    MEM   WB                                 │ │
│  │  ────  ────  ────  ────  ────  ────                               │ │
│  │   1    ADD                                                       │ │
│  │   2    SUB   ADD                                                 │ │
│  │   3          SUB   ADD                                           │ │
│  │   4                SUB   ADD                                     │ │
│  │   5                      SUB   ADD  ← ADD: WB (R1 기록)            │ │
│  │   6                            SUB  ← SUB: WB (R1 기록)            │ │
│  │                                                               │ │
│  │  → ADD가 먼저 WB, SUB이 나중에 WB                                   │ │
│  │  → 순서대로 쓰기 → WAW 해저드 없음                                   │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【왜 파이프라인에서는 WAW가 거의 없는가?】                               │
│  ───────────────────────────────────────────                         │
│  • WB 단계는 순차적으로 진행                                             │
│  • 파이프라인은 프로그램 순서 보장                                         │
│  • 같은 사이클에 두 명령어가 WB 불가                                     │
│  → 순차 파이프라인에서 WAW 불가능                                         │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  WAW가 문제가 되는 경우:                                               │ │
│  │  ──────────────────────                                              │ │
│  │  비순차 실행에서:                                                     │ │
│  │  ┌─────────────────────────────────────────────────────────────┐    │ │
│  │  │  ADD R1, R2, R3   ; 느림 (캐시 미스)                            │    │ │
│  │  │  SUB R1, R4, R5   ; 빠름                                        │    │ │
│  │  │                                                               │    │ │
│  │  │  → SUB이 먼저 완료될 수 있음                                      │    │ │
│  │  │  → SUB(R1) → ADD(R1) 순서로 쓰기                                │    │ │
│  │  │  → 최종 R1 = ADD 결과 (잘못됨!)                                  │    │ │
│  │  │  → 원래는 SUB 결과여야 함                                        │    │ │
│  │  └─────────────────────────────────────────────────────────────┘    │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. WAW 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                WAW 상세                                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【비순차 실행에서의 WAW 문제】                                          │
│  ────────────────────────────                                        │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  프로그램 순서:                                                     │ │
│  │  ADD R1, R2, R3   ; 느린 명령어                                    │ │
│  │  SUB R1, R4, R5   ; 빠른 명령어                                    │ │
│  │  AND R6, R1, R7   ; R1 사용                                       │ │
│  │                                                               │ │
│  │  비순차 실행 시나리오:                                               │ │
│  │  1. ADD 대기 (R2 또는 R3 캐시 미스)                                 │ │
│  │  2. SUB 실행 완료 (R4, R5 준비됨)                                   │ │
│  │  3. SUB 결과를 R1에 씀                                              │ │
│  │  4. ADD 실행 완료                                                   │ │
│  │  5. ADD 결과를 R1에 씀                                              │ │
│  │  6. AND가 ADD 결과(R1) 사용                                         │ │
│  │                                                               │ │
│  │  문제: AND는 SUB 결과를 사용해야 함!                                  │ │
│  │  (프로그램 순서상 SUB이 마지막 R1 쓰기)                                │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【레지스터 리네이밍으로 해결】                                          │
│  ──────────────────────────                                          │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  논리 레지스터 → 물리 레지스터 매핑:                                   │ │
│  │                                                               │ │
│  │  ADD R1, R2, R3   → ADD P10, P2, P3  (R1 = P10)                 │ │
│  │  SUB R1, R4, R5   → SUB P11, P4, P5  (R1 = P11)                 │ │
│  │  AND R6, R1, R7   → AND P6, P11, P7  (R1 = P11 사용)             │ │
│  │                                                               │ │
│  │  → R1에 대한 두 쓰기가 다른 물리 레지스터 사용                        │ │
│  │  → WAW 의존성 제거                                                 │ │
│  │  → AND는 P11(SUB 결과) 사용                                         │ │
│  │  → 순서 상관없이 올바른 결과                                          │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【ROB (Reorder Buffer)로 해결】                                       │
│  ──────────────────────────                                          │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  ROB: 프로그램 순서대로 결과 기록                                     │ │
│  │                                                               │ │
│  │  ROB Entry:                                                      │ │
│  │  ┌─────────────────────────────────────────────────────────────┐│ │
│  │  │  Entry  |  명령어  |  결과  |  완료  |  목적지                 ││ │
│  │  ├─────────────────────────────────────────────────────────────┤│ │
│  │  │   1    |  ADD    |  100   |  Yes  |  R1                    ││ │
│  │  │   2    |  SUB    |  50    |  Yes  |  R1                    ││ │
│  │  │   3    |  AND    |   -    |  No   |  R6                    ││ │
│  │  └─────────────────────────────────────────────────────────────┘│ │
│  │                                                               │ │
│  │  • ROB가 순서대로 결과를 레지스터에 기록                              │ │
│  │  • Entry 2(SUB)가 나중에 기록 → R1 = 50                            │ │
│  │  • WAW 문제 해결                                                   │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  WAW vs WAR vs RAW 비교:                                             │ │
│  │  ────────────────────────                                            │ │
│  │  종류   의존성        발생 조건           해결                        │ │
│  │  ────   ────────     ────────────       ────                        │ │
│  │  RAW   데이터 의존성   j가 i 결과 사용      포워딩                     │ │
│  │  WAR   반의존성       j가 i 소스 덮어쓰기   리네이밍                   │ │
│  │  WAW   출력 의존성     둘 다 같은 곳 쓰기   리네이밍, ROB              │ │
│  │                                                               │ │
│  │  의존성 종류:                                                        │ │
│  │  • RAW: 진성 의존성 (True Dependency) - 제거 불가                   │ │
│  │  • WAR: 이름 의존성 (Name Dependency) - 리네이밍으로 제거 가능        │ │
│  │  • WAW: 이름 의존성 (Name Dependency) - 리네이밍으로 제거 가능        │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### WAW 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                WAW 분석                                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【레지스터 리네이밍 확인】                                             │
│  ────────────────────────                                            │
│  $ cat /proc/cpuinfo | grep -i "register"                           │
│  → 현대 CPU는 레지스터 리네이밍 지원                                     │
│                                                                     │
│  【물리 레지스터 수】                                                   │
│  ──────────────────                                                  │
│  • x86-64: 논리 16개 (R0-R15), 물리 100~300개                         │
│  • ARM64: 논리 31개 (X0-X30), 물리 100~200개                          │
│  • 충분한 물리 레지스터로 WAW 해결                                       │
│                                                                     │
│  【ROB 크기】                                                          │
│  ──────────────────                                                  │
│  • Intel Skylake: 224 entries                                       │
│  • AMD Zen 3: 256 entries                                           │
│  • Apple M1: 630+ entries                                           │
│  → 큰 ROB로 많은 비순차 명령어 처리                                      │
│                                                                     │
│  【WAW 최소화 코드】                                                    │
│  ──────────────────                                                  │
│  # Before (WAW 발생 가능)                                             │
│  for (i = 0; i < n; i++) {                                          │
│      a = b[i];        ; a 쓰기                                       │
│      a = c[i];        ; a 쓰기 → WAW                                 │
│      d[i] = a;        ; a 사용                                       │
│  }                                                                  │
│                                                                     │
│  # After (WAW 제거)                                                   │
│  for (i = 0; i < n; i++) {                                          │
│      d[i] = c[i];     ; 직접 사용                                     │
│  }                                                                  │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  WAW 감지 및 해결:                                                    │ │
│  │  ──────────────────                                                  │ │
│  │  1. 의존성 분석 도구 사용                                             │ │
│  │  2. 불필요한 중간 변수 제거                                           │ │
│  │  3. 레지스터 리네이밍 의존                                             │ │
│  │  4. 컴파일러 최적화 활용                                              │ │
│  │  5. 데드 코드 제거                                                   │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• WAW: Write After Write, 쓰기 후 쓰기 출력 의존성
• 이름 의존성: 실제 데이터 의존성이 아님
• 파이프라인: 순차 실행 시 거의 발생 안 함
• 비순차 실행: WAW가 문제가 됨
• 해결: 레지스터 리네이밍 (물리 레지스터 분리)
• ROB: 프로그램 순서대로 결과 기록
• 현대 CPU: 하드웨어 리네이밍으로 자동 해결
```
