# 223. 데이터 해저드 (Data Hazard)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 해저드 (Data Hazard)는 파이프라인에서 앞서 실행 중인 명령어의 연산 결과(데이터)가 아직 최종 레지스터에 기록되지 않았는데, 바로 뒤따라오는 명령어가 그 데이터를 읽어서 연산하려 할 때 발생하는 논리적 의존성 및 타이밍 문제다.
> 2. **가치**: 데이터를 기다리는 동안 발생하는 파이프라인 정지(Stall)를 최소화하기 위해, 연산이 끝나자마자 결과값을 다음 명령어의 입력단으로 직통 연결하는 전방 전달(Forwarding/Bypass) 기술이 현대 CPU의 핵심 성능 방어책으로 사용된다.
> 3. **융합**: 하드웨어적인 포워딩 배선으로도 해결이 불가능한 Load-Use 해저드(메모리 읽기 지연)는 똑똑한 컴파일러가 서로 무관한 명령어를 그 틈바구니에 끼워 넣는 소프트웨어 명령어 스케줄링(Instruction Scheduling)과 융합하여 완벽히 해결해낸다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 데이터 해저드 (Data Hazard)는 파이프라인 아키텍처에서 앞서 실행 중인 명령어의 연산 결과가 아직 최종 레지스터에 기록되지 않았음에도 불구하고, 바로 뒤따라오는 명령어가 해당 데이터를 읽어 연산하려 할 때 발생하는 논리적 의존성 충돌 현상이다.

- **필요성**: 데이터 해저드에 대한 정밀한 분석과 관리는 파이프라인의 **실행 연속성을 확보하고 연산 결과의 정합성을 수호**하기 위해 반드시 필요하다. 명령어들이 겹쳐서 실행되는 환경에서 데이터 의존성을 무시하고 연산을 강행하면 갱신되지 않은 과거의 '쓰레기 값'을 사용하여 치명적인 논리 오류를 낳기 때문이다. 따라서 데이터 해저드를 실시간으로 감지하고, **전방 전달(Forwarding)**을 통한 하드웨어적 우회나 **명령어 스케줄링**을 통한 소프트웨어적 최적화를 적용함으로써, 파이프라인 정지(Stall)를 최소화하고 CPU의 클럭당 명령어 처리량(IPC)을 극한으로 끌어올리는 핵심적인 성능 방어 전략 역할을 수행한다.
- **💡 비유**: 햄버거 가게에서 1번 직원이 '패티 굽기'를 담당하고 2번 직원이 그 패티를 받아 '조립'을 할 때, 패티가 다 구워지기도 전에 조립 직원이 빈 빵을 들고 서두르다 멈칫하는 상황과 같다. 패티가 나올 때까지 조립을 멈추거나(Stall), 아니면 패티가 구워지자마자 쟁반을 거치지 않고 직접 손으로 건네받아(Forwarding) 시간을 아껴야 한다.

- **등장 배경**: 과거 단일 사이클 아키텍처에서는 한 명령어가 1부터 100까지 완전히 다 끝난 뒤에야 다음 명령어가 시작되므로 데이터 해저드 자체가 존재할 수 없었다. 하지만 성능을 위해 파이프라인을 5단으로 겹쳐(Overlap) 실행하면서, 앞 명령어가 결과를 맨 마지막 'WB(쓰기)' 단계에서 저장하기도 전에, 뒤따라오던 명령어가 초반인 'ID(읽기)' 단계에서 갱신되지 않은 옛날 쓰레기 값을 읽어가는 치명적인 타이밍 역전 현상이 발견되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 데이터 의존성의 3가지 종류

명령어들이 같은 레지스터나 메모리를 다룰 때 발생하는 의존성은 세 가지로 나뉜다. 파이프라인과 비순차 실행 구조에 따라 겪는 해저드의 종류가 다르다.

| 의존성 유형 | 설명 (명령어 A 실행 후 B가 뒤따라 실행 시) | 비고 및 발생 환경 |
|:---|:---|:---|
| **RAW (Read After Write)** | **진성 의존성(True Dependency)**. A가 데이터를 레지스터에 확실히 쓰기도 전에 B가 성급히 읽으려 함. | 데이터 해저드의 가장 보편적이고 치명적인 원인 (모든 파이프라인 발생) |
| **WAR (Write After Read)** | **반의존성(Anti-Dependency)**. A가 채 읽기도 전에 뒤따라온 B가 값을 먼저 덮어쓰려 함. | 기본 5단에서는 안 생기나, 비순차적(OoO) 병렬 실행 환경에서 꼬임 발생 |
| **WAW (Write After Write)**| **출력 의존성(Output Dependency)**. A가 쓰기도 전에 빠른 B가 먼저 써버려 최종 상태가 덮어씌워짐. | 비순차 실행 환경에서 사용할 수 있는 물리 레지스터 이름이 부족해 충돌 시 발생 |

### 가장 대표적인 데이터 해저드 (RAW) 발생 시각화

`ADD $t0, $t1, $t2` (연산 결과를 t0 레지스터에 저장) 직후에 곧바로 `SUB $t3, $t0, $t4` (t0를 읽어와서 빼기 수행)가 파이프라인으로 들어올 때의 참사다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         데이터 해저드(RAW)로 인한 파이프라인 모순 및 스톨(Stall)         │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │        클럭:   1    2    3    4    5    6                   │
  │     ──────────────────────────────────────────              │
  │ ADD $t0,.. : [IF]-[ID]-[EX]-[ME]-[WB] (클럭 5 끝에서야 $t0에 저장)│
  │ SUB ..,$t0 :      [IF]-[ID]-[EX]-[ME]-[WB]                  │
  │                           ▲                                 │
  │                           │ 충돌 지점                         │
  │                                                             │
  │ [물리적 모순 발생]                                             │
  │ SUB 명령어는 클럭 3의 [ID] 단계에서 $t0의 값을 레지스터로부터 읽어간다.  │
  │ 하지만 ADD 명령어는 클럭 5의 [WB] 단계에 가서야 $t0에 진짜 결과값을 쓴다.│
  │ 즉, SUB는 $t0가 갱신되기 전의 '쓰레기 값'을 가져가서 연산을 망치게 된다. │
  │                                                             │
  │ [원시적인 하드웨어 해결책: 파이프라인 멈춤 (Stall)]                 │
  │ SUB ..,$t0 :      [IF]-[ID]-(wait)-(wait)-[EX]-[ME]-[WB]    │
  │ -> 제어 유닛이 이를 감지하고, ADD가 [WB]를 무사히 마칠 때까지          │
  │    SUB를 ID 단계에 가둬두고 강제로 2클럭을 멍때리게 만든다.             │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 레지스터 파일에 연산된 최신 값이 기록되는 것은 5단계 중 맨 마지막인 WB 단계다. 하지만 뒤따라오는 명령어가 데이터를 요구하여 읽어가는 것은 파이프라인 초반인 ID 단계다. 앞 명령어가 파이프라인을 완전히 빠져나가기도 전에 뒤 명령어가 앞 명령어의 산출물을 요구하면 물리적 시간 역행이 일어난다. 제어 유닛이 이를 방치하면 연산 결과가 모조리 틀려버리므로, 제어 유닛 내부의 '해저드 탐지 유닛(Hazard Detection Unit)'이 이 상황을 즉시 감지하고 뒷 명령어를 2클럭 동안 정지(Stall)시켜 기포(Bubble)를 만든다. 이는 엄청난 클럭 낭비다.

- **📢 섹션 요약 비유**: 앞사람이 열심히 작성 중인 보고서(데이터)를 다 쓰지도 않았는데, 뒷사람이 자기가 결재를 올리겠다고 책상에서 옛날 초안 서류를 마음대로 빼가는 꼴입니다. 그래서 뒷사람을 복도에 세워두고 보고서가 완성될 때까지 2시간을 무작정 기다리게 하는 것(Stall)입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 하드웨어적 궁극의 극복책: 전방 전달 (Data Forwarding / Bypassing)

데이터 해저드의 핵심 돌파구는 관점의 전환이다. 데이터가 최종적으로 클럭 5의 WB 단계에서 레지스터에 얌전히 기록되기는 하지만, **실제 연산 결과 자체는 클럭 3의 EX 단계가 끝난 직후 이미 ALU 밖으로 튀어나와 래치에 대기하고 있다.** 굳이 레지스터에 기록될 때까지 기다렸다가 다시 꺼내 올 필요가 없다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         전방 전달(Forwarding)을 통한 데이터 해저드 극복              │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │        클럭:   1    2    3    4    5    6                    │
  │     ──────────────────────────────────────────               │
  │ ADD $t0,.. : [IF]-[ID]-[EX]-[ME]-[WB]                        │
  │                          │                                   │
  │                          └──(전방 전달 배선으로 즉시 우회)──┐      │
  │                                                ▼             │
  │ SUB ..,$t0 :      [IF]-[ID]-[EX]-[ME]-[WB]                   │
  │                                                              │
  │ [포워딩 회로의 마법 같은 원리]                                   │
  │ ADD가 EX(연산) 단계를 마친 순간, 결과값은 EX/MEM 래치에 이미 담겨 있다.│
  │ 하드웨어는 이 래치의 값을 굳이 레지스터 파일로 보내어 보관하는 대신,      │
  │ 전용 배선(Bypass 경로)을 뚫어 다음 명령어인 SUB의 EX 단계 입력단(MUX) │
  │ 으로 즉각 다이렉트로 꽂아버린다.                                   │
  │                                                              │
  │ => 결과: 파이프라인을 단 1클럭도 멈추지 않고(Stall=0) 매끄럽게 진행!  │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 포워딩(또는 바이패싱) 회로는 데이터 해저드를 물리적으로 치료하는 마법의 하드웨어 배선이다. 파이프라인의 뒤쪽 래치(EX/MEM, MEM/WB)에 임시로 담긴 최신 핫한 데이터를 낚아채서, 현재 EX 단계로 진입하려는 뒷명령어의 ALU 입력단 멀티플렉서로 시간 지연 없이 다이렉트 연결해 준다. 레지스터 파일을 거쳐서 정직하게 읽고 쓰는 느린 정석 루트를 무시하고 하드웨어적으로 지름길(Short-cut)을 뚫은 것이다. 이를 통해 대부분의 데이터 해저드는 클럭 스톨(Stall) 없이 100% 무효화된다.

- **📢 섹션 요약 비유**: 1번 직원이 뜨겁게 구워낸 패티를 굳이 식당 창고(레지스터)에 얌전히 차곡차곡 넣었다가 2번 직원이 한참 뒤에 다시 꺼내 오는 바보 같은 짓을 당장 그만두고, 갓 구워진 패티를 곧바로 2번 직원의 손(포워딩 배선)에 휙 던져 쥐여주는 극강의 효율화입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **Load-Use 해저드와 컴파일러의 명령어 스케줄링의 결합**
   포워딩(전방 전달) 하드웨어 회로로도 절대 막지 못하는 유일하고 치명적인 데이터 해저드가 있다. 바로 메모리 캐시에서 값을 읽어오는 `LOAD` 명령어 바로 직후에 그 값을 쓰는 명령어(예: `ADD`)가 연달아 올 때다. LOAD는 뒷부분인 MEM 단계(클럭 4)가 끝나야만 캐시에서 데이터가 튀어나오므로, 그 직후의 ADD 명령어는 EX 단계(클럭 3)에서 값을 건네받을 방법이 물리적으로 존재하지 않는다 (시간을 거꾸로 역행할 순 없으므로). 따라서 무조건 1클럭의 파이프라인 멈춤(Load-Use Stall)이 발생한다.
   이를 극복하기 위해 C/C++ 컴파일러(GCC, LLVM) 개발자는 어셈블리어를 뽑아낼 때 **명령어 스케줄링(Instruction Scheduling)**이라는 소프트웨어적 최적화를 수행한다. LOAD 명령어와 그 값을 당장 쓰는 ADD 명령어 '사이'에, 이 작업과 전혀 무관한 다른 변수의 덧셈이나 대입 명령어를 억지로 끌어올려 끼워 넣는다. 하드웨어가 1클럭을 멍때릴 바에, 그 시간에 차라리 다른 독립적인 일을 하도록 소프트웨어 코드를 재배치해 주는 환상적인 시너지다.

### 도입 체크리스트
- **기술적**: 포워딩 유닛(Forwarding Unit) 내부의 의존성 비교 판단 회로(MUX Select Logic)의 지연 시간이 너무 뚱뚱하고 길어서, EX 단계의 ALU 연산 임계 경로(Critical Path)를 초과해버려 오히려 전체 칩의 최고 클럭(Fmax) 상한선을 깎아먹고 있지 않은가?
- **운영적**: 컴파일러 차원의 파이프라인 스케줄링 최적화(예: GCC의 `-fschedule-insns` 옵션)를 명시적으로 켜서, 하드웨어가 막지 못하는 Load-Use 패널티를 백그라운드에서 완벽히 상쇄하고 있는가?

### 안티패턴
- **원가 절감을 핑계로 포워딩 회로 누락 및 NOP 강제 삽입**
  하드웨어 설계 다이(Die) 면적(Area)과 트랜지스터 전력 소모를 쥐어짜서 줄이겠다고 포워딩(Bypass) 배선을 칩에서 아예 빼버리는 초저가형 악성 설계. 이렇게 되면 소프트웨어 컴파일러가 데이터 의존성이 있는 명령어 사이사이에 강제로 아무것도 안 하는 텅 빈 NOP(No Operation) 명령어를 2~3개씩 무식하게 쑤셔 넣어야만 프로그램이 꼬이지 않고 정상 동작한다. 이는 바이너리 코드 사이즈(Binary Size)를 폭증시키고 귀중한 명령어 캐시 적중률을 박살 내는 최악의 안티패턴이다.

- **📢 섹션 요약 비유**: 포워딩 배선(비상용 지름길)을 까는 공사비와 철근값이 아깝다고 아예 안 깔아버리면, 출퇴근하는 직원들이 막히는 길에서 매번 빈 공터(NOP)를 세 바퀴씩 뺑뺑 돌고 나서야 겨우 회사에 들어오게 만들어 전체 회사의 생산성을 나락으로 떨어뜨리는 꼴이 됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 해결 방식 | Stall 방치 시 | 전방 전달(Forwarding) + 컴파일러 최적화 | 비교 개선 효과 |
|:---|:---|:---|:---|
| **정량 (CPI)** | RAW 발생마다 매번 2클럭씩 성능 까먹음 | 0 ~ 1클럭 (Load-Use만 1클럭) 최소 지연 | 파이프라인 IPC가 이상적인 1.0에 턱밑까지 수렴 |
| **정량 (전력/효율)**| 파이프라인이 멈춘 유휴 상태(Bubble)에서도 누설 전류 발생 | 매 클럭 꽉 차게 유효 연산 수행으로 하드웨어 가동률 100% | 트랜지스터당 전력 대비 성능 효율(Perf/W) 극대화 |
| **정성 (소프트웨어)**| 컴파일러가 멍청한 하드웨어를 위해 NOP을 잔뜩 욱여넣어야 함 | 하드웨어가 알아서 우회 배선을 제공해줌 | 바이너리 코드 사이즈 절감 및 플랫폼 간 이식성 확보 |

### 미래 전망
- **비순차적 실행(OoO)과 레지스터 리네이밍(Register Renaming)의 결합**: 현대 스마트폰과 서버의 최고급 슈퍼스칼라 아키텍처는 전방 전달(포워딩) 선을 까는 것만으로 만족하지 않는다. WAR(반의존성)이나 WAW(출력 의존성) 같은 가짜 데이터 해저드를 원천적으로 없애기 위해, 하드웨어가 런타임에 눈속임으로 프로그래머의 레지스터 이름($t0, $t1)을 가상의 수백 개 물리 레지스터 이름(P1, P2)으로 몰래 바꿔치기(Renaming)하여 충돌을 물리적으로 회피한다. 그리고 진짜 지독한 의존성(RAW) 때문에 막혀버린 명령어는 대합실(Reservation Station)에 조용히 박아두고, 뒤에 줄 서 있는 무관한 명령어부터 먼저 확 당겨와서 빈틈없이 실행해버리는(Out-of-Order Execution) 극한의 병렬 최적화 엔진이 표준 사양이 되었다.

- **📢 섹션 요약 비유**: 앞차가 고장 나서 느리게 가거나(데이터 해저드) 차가 막힐 때, 단순히 갓길을 뚫어서 지나가게 해주고(포워딩), 아예 목적지가 다른 차선의 차들은 먼저 쌩쌩 지나가게 쭉쭉 빼주는(비순차적 실행) 지능형 교통 통제 시스템 덕분에 데이터 해저드라는 옛날의 병목은 역사 속으로 완전히 사라지고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **파이프라인 해저드 (Pipeline Hazard)** | 구조적, 제어 해저드와 더불어 파이프라인의 이상적인 속도 향상을 무참히 가로막는 3대 성능 저하 요인 중 가장 흔한 녀석이다. |
| **포워딩 / 바이패싱 (Forwarding)** | 데이터 해저드로 인한 파이프라인 스톨(Stall)을 하드웨어적으로 0클럭으로 둔갑시켜버리는 핵심 우회 배선 기술이다. |
| **명령어 스케줄링 (Instruction Scheduling)** | 포워딩 배선으로도 물리적으로 못 막는 Load-Use 해저드를 컴파일러가 무관한 코드를 얍삽하게 끼워 넣어 소프트웨어적으로 덮어버리는 기술이다. |
| **레지스터 리네이밍 (Register Renaming)** | 비순차적 실행 환경에서 발생하는 WAR, WAW 같은 가짜 데이터 해저드를 아예 레지스터 간판 이름을 바꿔치기해 해결하는 최신 꼼수 기법이다. |
| **스톨 / 기포 (Stall / Bubble)** | 해저드를 하드웨어나 컴파일러가 끝내 회피하지 못했을 때 파이프라인이 최후에 선택할 수밖에 없는, 클럭을 허공에 날리는 뼈아픈 휴식 상태다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 햄버거 공장에서 1번 요리사가 "고기 패티 다 구웠어!"라고 접시에 올려놓기도 전에, 2번 요리사가 "빨리 빵에 넣게 내놔!"라고 떼를 쓰면 일이 꼬이겠죠? 이걸 '데이터 해저드'라고 해요.
2. 옛날 바보 같은 공장장님은 2번 요리사를 밖에서 2시간 벌서게 하면서 무작정 기다리게 했는데(스톨/Stall), 이건 시간 낭비가 너무 심했어요.
3. 그래서 지금은 1번 요리사가 패티를 불에서 굽자마자 무거운 접시에 놓지도 않고, 곧바로 2번 요리사 빵 위로 휙! 던져주는 멋진 묘기(포워딩)를 써서 공장을 1초도 안 멈추게 만들었답니다!
