# 238. 비순차 실행 (Out-of-Order Execution)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 명령어 순서를 바꿔 의존성 없는 명령어 먼저 실행
> 2. **가치**: 파이프라인 활용률 증대, IPC 향상
> 3. **융합**: 레지스터 리네이밍, ROB와 연동

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

비순차 실행(OoO, Out-of-Order Execution)은 **프로그램 순서와 다르게 명령어를 실행하는 기술**이다. 데이터 의존성이 없는 명령어를 먼저 실행하여 파이프라인 스톨을 줄이고 IPC를 향상시킨다.

### 💡 비유: 식당 주문 처리

비순차 실행은 **식당 주문 처리**와 같다. 주문 순서대로 요리할 수도 있다. 하지만 파스타는 10분, 샐러드는 2분 걸린다면 샐러드를 먼저 만들어서 대기 시간을 줄일 수 있다. 나중에 순서대로 서빙하면 된다.

### 순차 vs 비순차 실행

```
┌─────────────────────────────────────────────────────────────────────┐
│                순차 vs 비순차 실행                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【순차 실행 (In-Order)】                                               │
│  ──────────────────                                                  │
│  코드:                                                               │
│    I1: LOAD  R1, [addr]    ; 캐시 미스, 100클럭 대기                   │
│    I2: ADD   R2, R1, R3    ; I1 대기 (RAW 의존성)                     │
│    I3: MUL   R4, R5, R6    ; I2 대기 (I1 때문에)                       │
│    I4: SUB   R7, R8, R9    ; I3 대기 (의존성 없음에도!)                 │
│                                                                     │
│  실행:                                                               │
│  클럭  I1    I2    I3    I4                                         │
│  ────  ────  ────  ────  ────                                       │
│  1     인출                                                          │
│  ...   ...   (캐시 미스 대기)                                          │
│  100   완료  인출                                                      │
│  101         디코드                                                   │
│  102         실행                                                     │
│  103               인출                                               │
│  ...                                                                     │
│  → 총 100+ 클럭 소요                                                   │
│                                                                     │
│  【비순차 실행 (Out-of-Order)】                                          │
│  ──────────────────                                                  │
│  코드:                                                               │
│    I1: LOAD  R1, [addr]    ; 캐시 미스, 100클럭 대기                   │
│    I2: ADD   R2, R1, R3    ; I1 대기 (RAW 의존성)                     │
│    I3: MUL   R4, R5, R6    ; ← 의존성 없음, 먼저 실행!                  │
│    I4: SUB   R7, R8, R9    ; ← 의존성 없음, 먼저 실행!                  │
│                                                                     │
│  실행:                                                               │
│  클럭  I1    I2    I3    I4                                         │
│  ────  ────  ────  ────  ────                                       │
│  1     인출  인출  인출  인출   ; 모두 인출                               │
│  2     디코드 디코드 디코드 디코드 ; 모두 디코드                           │
│  3     실행  대기  실행  실행   ; I3, I4 먼저 실행!                       │
│  ...   ...   ...                                                     │
│  100   완료  실행          ; I1 완료 후 I2 실행                          │
│  101         완료                                                     │
│  → 총 100 클럭 소요 (I3, I4는 이미 완료)                                │
│                                                                     │
│  【비순차 실행의 핵심 원칙】                                               │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 순차 인출 (In-Order Fetch): 프로그램 순서대로 인출              │ │
│  │  2. 비순차 실행 (Out-of-Order Execute): 의존성 없는 것 먼저 실행    │ │
│  │  3. 순차 완료 (In-Order Commit): 프로그램 순서대로 결과 반영        │ │
│  │                                                               │ │
│  │  → 프로그래머는 순차 실행처럼 보이지만, 내부는 비순차 실행           │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 비순차 실행 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                비순차 실행 상세                                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【비순차 실행 파이프라인】                                               │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │   ┌────────────────────────────────────────────────────────┐   │ │
│  │   │              순차 프론트엔드 (In-Order Front-end)          │   │ │
│  │   │  ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐              │   │ │
│  │   │  │ 인출 │ → │ 디코드│ → │ 리네이밍│ → │ 발급 │              │   │ │
│  │   │  └──────┘   └──────┘   └──────┘   └──────┘              │   │ │
│  │   └────────────────────────────────────────────────────────┘   │ │
│  │                              ↓                                 │ │
│  │   ┌────────────────────────────────────────────────────────┐   │ │
│  │   │              비순차 백엔드 (Out-of-Order Back-end)        │   │ │
│  │   │  ┌──────────┐   ┌──────────┐   ┌──────────┐            │   │ │
│  │   │  │예약 스테이션│ → │  실행 유닛  │ → │ 재주문 버퍼 │            │   │ │
│  │   │  └──────────┘   └──────────┘   └──────────┘            │   │ │
│  │   └────────────────────────────────────────────────────────┘   │ │
│  │                              ↓                                 │ │
│  │   ┌────────────────────────────────────────────────────────┐   │ │
│  │   │              순차 커밋 (In-Order Commit)                   │   │ │
│  │   │  ┌──────────┐   ┌──────────┐   ┌──────────┐            │   │ │
│  │   │  │  ROB  │ → │  리타이어  │ → │아키텍처 상태│            │   │ │
│  │   │  └──────────┘   └──────────┘   └──────────┘            │   │ │
│  │   └────────────────────────────────────────────────────────┘   │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【비순차 실행 윈도우】                                                  │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  CPU             윈도우 크기    설명                              │ │
│  │  ───────────────  ──────────  ──────────────────                │ │
│  │  Intel Skylake    224         딥 버퍼, 높은 OoO 능력             │ │
│  │  AMD Zen 3        256         큰 윈도우                          │ │
│  │  Apple M1         630+        매우 큰 윈도우                      │ │
│  │  ARM Cortex-A78   128         중간 크기                          │ │
│  │  → 윈도우가 클수록 더 많은 명령어를 재배치 가능                      │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【비순차 실행의 이점】                                                  │
│  ──────────────────                                                  │
│  • 캐시 미스 숨김: 긴 메모리 지연을 다른 작업으로 숨김                       │
│  • 파이프라인 활용: 스톨 시간을 유용한 작업으로 채움                         │
│  • IPC 향상: 의존성 없는 명령어 병렬 실행                                 │
│  • 성능: 일반적으로 20~40% IPC 향상                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• 비순차 실행: 프로그램 순서와 다르게 명령어 실행
• 목적: 파이프라인 스톨 감소, IPC 향상
• 원칙: 순차 인출 → 비순차 실행 → 순차 커밋
• 구조: 프론트엔드(순차) + 백엔드(비순차)
• 윈도우: 128~630+ 엔트리
• 이점: 20~40% IPC 향상
• 비용: 하드웨어 복잡도, 전력 증가
• 보안: Spectre 등 사이드 채널 공격 취약
```
