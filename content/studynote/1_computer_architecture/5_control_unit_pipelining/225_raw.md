# 225. RAW (Read After Write)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RAW (Read After Write)는 파이프라인에서 앞선 명령어가 특정 레지스터에 연산 결과를 '쓰기(Write)' 전에, 바로 뒤따라오는 명령어가 그 레지스터의 값을 먼저 '읽으려(Read)' 할 때 발생하는 가장 치명적인 진성 데이터 의존성(True Data Dependency)이다.
> 2. **가치**: 프로그램의 논리적 실행 결과가 정확히 유지되려면 이 순서(쓰고 나서 읽기)가 반드시 지켜져야 하므로, 소프트웨어적 꼼수로는 원천적인 회피가 불가능한 물리적 인과관계의 핵심이다.
> 3. **융합**: 이를 해결하기 위해 파이프라인 하드웨어는 포워딩(Forwarding) 회로망을 구축하여, 쓰기 단계(WB)까지 기다리지 않고 연산 직후(EX)의 값을 다음 명령어의 읽기 단계로 직접 쏴주는 바이패스(Bypass) 기술을 필수적으로 결합한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: RAW (Read After Write)는 파이프라인 아키텍처에서 명령어 A가 결과값을 특정 레지스터에 기록하기(Write) 전에, 뒤따르는 명령어 B가 해당 레지스터를 입력값으로 읽으려(Read) 할 때 발생하는 '진성 데이터 의존성(True Data Dependency)' 충돌 현상이다.

- **필요성**: RAW 해저드에 대한 정밀한 제어는 파이프라인의 **논리적 수치 무결성을 유지하고 연산 지연을 최소화**하기 위해 반드시 필요하다. 프로그램은 본래 순차적 실행을 전제로 작성되나 파이프라인은 이를 겹쳐서 실행하므로, 의존성을 무시하면 갱신되지 않은 과거의 '쓰레기 값'을 참조하는 치명적 오류를 범하기 때문이다. 따라서 RAW 상황을 하드웨어적으로 즉각 감지하고, **전방 전달(Forwarding)**을 통해 쓰기 단계(WB)를 기다리지 않고 연산 직후(EX)의 값을 다음 명령어로 직송하거나, 부득이한 경우 **스톨(Stall)**을 삽입함으로써 시스템의 연산 결과가 항상 프로그래머의 의도와 일치하도록 보장하는 핵심적인 실행 규칙 역할을 수행한다.
- **💡 비유**: 앞사람이 서류 결재란에 도장을 '찍기도(Write) 전에', 뒷사람이 급하다고 그 빈 서류를 휙 '가져가서(Read)' 다음 부서로 제출해 버리면 서류 자체가 무효가 되는 치명적인 순서 뒤바뀜 사고와 같다. 도장이 찍힐 때까지 뒷사람을 복도에 세워두거나(Stall), 도장을 찍자마자 비서가 가로채서 뒷사람에게 즉시 건네줘야(Forwarding) 사고를 막을 수 있다.

- **등장 배경**: 프로그래머가 짠 코드는 철저히 순차적이다. 파이프라이닝 이전의 단일 사이클 구조에서는 앞 명령어가 완전히 끝난 뒤에 다음 명령어가 시작되므로 데이터 해저드가 존재할 수 없었다. 하지만 성능을 위해 명령어를 겹쳐서 실행(Overlap)하면서, `ADD R1, R2, R3` (R1에 쓰기) 직후에 `SUB R4, R1, R5` (R1 읽기)가 오면, SUB가 ID(읽기) 단계를 지날 때 ADD는 아직 파이프라인 끝의 WB(쓰기) 단계에 도달하지 못해, SUB가 갱신되지 않은 옛날 쓰레기 값을 읽어가는 끔찍한 오류가 발견되면서 이 개념이 정립되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 의존성의 방향 (RAW 메커니즘)

| 명령어 순서 | 어셈블리 예시 | 의존성 방향 | 설명 |
|:---|:---|:---|:---|
| **Inst 1 (선행)** | `ADD $t0, $t1, $t2` | `$t0` **생성** | `$t1`과 `$t2`를 더해 **`$t0`에 쓴다 (Write)** |
| **Inst 2 (후행)** | `SUB $t3, $t0, $t4` | `$t0` **소비** | 방금 갱신되어야 할 그 **`$t0`를 읽는다 (Read)** |

### 파이프라인에서의 RAW 해저드 발생 타이밍

왜 문제가 되는지 5단 파이프라인 클럭 단위로 쪼개보면 그 물리적 모순이 명확해진다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │              RAW 의존성으로 인한 파이프라인 타이밍 충돌               │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │        클럭:   1    2    3    4    5                        │
  │     ──────────────────────────────────────────              │
  │ ADD $t0,.. : [IF]-[ID]-[EX]-[ME]-[WB] (여기서 $t0 쓰기 완료!)  │
  │ SUB ..,$t0 :      [IF]-[ID] (여기서 $t0 읽기를 시도함!)        │
  │                           ▲                                 │
  │                     시간 역행 불가                           │
  │                                                             │
  │ [물리적 모순 결과]                                             │
  │ 클럭 3 시점에 SUB는 $t0를 읽어야 하는데, ADD는 아직 연산(EX) 중이다. │
  │ 따라서 SUB는 올바른 최신 $t0 값을 절대로 얻을 수 없다. (RAW 해저드)    │
  │ 제어 유닛은 이를 감지하고 SUB를 강제로 2클럭 멈춰버린다(Stall).       │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 논리적으로는 ADD 연산이 완전히 끝난 뒤에 SUB가 그 값을 가져가야 하지만, 파이프라인은 시간을 압축한다. SUB의 데이터 '읽기(ID)' 타이밍(클럭 3)이 ADD의 데이터 '쓰기(WB)' 타이밍(클럭 5)보다 물리적으로 2클럭이나 먼저 도래해 버린다. 이 치명적인 2클럭의 간극이 바로 RAW 해저드의 본질이다.

- **📢 섹션 요약 비유**: 요리사(ADD)가 주방에서 고기를 한창 굽고 있는데, 성격 급한 웨이터(SUB)가 아직 익지도 않은 생고기를 접시째 뺏어 들고 손님 테이블에 서빙(ID)해버리는 끔찍한 타이밍 불일치와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 데이터 의존성 3대장 (RAW, WAR, WAW) 비교

RAW는 절대로 피할 수 없는 진성 의존성이지만, WAR와 WAW는 비순차 실행(OoO)에서 레지스터 이름이 겹쳐서 생기는 가짜 의존성이다.

| 비교 항목 | RAW (Read After Write) | WAR (Write After Read) | WAW (Write After Write) |
|:---|:---|:---|:---|
| **한글 명칭** | **진성 의존성 (True Dependency)** | 반의존성 (Anti-dependency) | 출력 의존성 (Output Dependency) |
| **순서 규칙** | 앞 명령어가 쓰고 -> 뒤가 읽음 | 앞이 읽고 -> 뒤가 씀 | 앞이 쓰고 -> 뒤가 다시 덮어씀 |
| **문제 상황** | 뒤가 **너무 일찍 읽어** 옛날 값을 가져감 | 뒤가 **너무 일찍 써서** 앞이 새 값을 읽어버림 | 뒤가 너무 일찍 써버려서 **최종 값이 옛날 값**으로 남음 |
| **해결책** | **포워딩 (Forwarding), 스톨 (Stall)** | 레지스터 리네이밍 (Register Renaming) | 레지스터 리네이밍 (Register Renaming) |

### RAW 방어의 물리적 한계: Load-Use 해저드

일반적인 산술 명령어끼리의 RAW는 전방 전달(Forwarding) 배선을 깔아서 100% 지연 없이 덮을 수 있지만, 메모리에서 값을 읽어오는 `Load` 명령어 직후의 RAW는 포워딩으로도 절대 지연을 막을 수 없다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         Load-Use RAW 해저드의 극복 불가능한 물리적 지연           │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │        클럭:   1    2    3    4    5                         │
  │     ──────────────────────────────────────────               │
  │ LW  $t0,.. : [IF]-[ID]-[EX]-[ME]-[WB]                        │
  │                              │ (캐시에서 데이터가 클럭 4 끝에 나옴) │
  │                              ▼                               │
  │ SUB ..,$t0 :      [IF]-[ID]-[EX]-[ME]-[WB]                   │
  │                              (클럭 3에 이미 데이터가 필요함)      │
  │                                                              │
  │ [결론]: 메모리에서 데이터를 가져오는(ME) 시간은 클럭 4인데,             │
  │        그걸 연산(EX)에 써야 하는 시간은 클럭 3이다. 타임머신이 없는 한,  │
  │        포워딩 전선을 어떻게 꼬아도 무조건 1클럭의 지연(Stall)은 발생한다.│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 산술 연산(ADD)은 클럭 3(EX 단)에 결과가 즉시 튀어나와서 다음 클럭 4의 EX 단계로 직통으로 넘겨줄 수 있다. 하지만 메모리 읽기(LW) 명령어는 메모리 캐시를 거치는 클럭 4(MEM 단)에야 결과가 나온다. 이 미래의 데이터를 과거인 클럭 3으로 전달할 수는 없으므로, 제어 유닛은 무조건 뒤따르는 SUB 명령어를 1클럭 뒤로 밀어버려야(Stall) 한다. 이것이 데이터 해저드 중 유일하게 뼈아픈 Load-Use RAW 해저드의 물리적 한계다.

- **📢 섹션 요약 비유**: 1층에서 2층으로 물건을 올려주는 건 엘리베이터(포워딩)를 타면 1초면 되지만, 내일 도착할 물건을 어제 날짜로 보내달라고 하는 건 타임머신이 없어서 불가능한 이치입니다. 어쩔 수 없이 내일까지 꼼짝없이 기다려야(Stall) 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **컴파일러의 기계어 스케줄링 (Instruction Scheduling)**
   실무에서 최신 컴파일러(GCC `-O2`나 LLVM)는 소스 코드를 기계어로 번역할 때, 하드웨어가 겪을 RAW 해저드를 피하기 위해 피눈물 나는 스케줄링을 한다. 특히 위의 'Load-Use 스톨 1클럭'을 없애기 위해, `LW $t0, ...` 명령어 바로 밑에 `$t0`와 전혀 상관없는 완전히 독립적인 다른 변수의 연산(`ADD $t5, $t6, $t7`) 코드를 살짝 끌어올려 끼워 넣는다. 이 단 1줄의 순서 바꿈 최적화 덕분에 하드웨어 CPU는 멍때리지 않고(Stall 회피) 알뜰하게 연산을 꽉꽉 채워 진행한다.

2. **슈퍼스칼라 코어와 파이프라인 깊이의 악몽**
   파이프라인을 깊게(20단 이상) 파면 래치가 너무 많아져서, RAW 포워딩 회로망이 극도로 복잡해진다. 만약 스마트폰 AP처럼 1클럭에 4개의 명령어를 뽑는 슈퍼스칼라 구조라면, 4개 명령어 간의 상호 RAW, WAR, WAW 의존성 여부를 매 클럭마다 $4 \times 4 = 16$가지 조합으로 검사해야 한다. 아키텍트는 이 막대한 비교 회로(Comparator)의 실리콘 면적과 전력 소모를 줄이기 위해, 레지스터 파일을 계층화하거나 분산 예약역(Distributed Reservation Station)을 채택하는 고도의 분산 설계를 결정한다.

### 도입 체크리스트
- **운영적/소프트웨어적**: 성능이 생명인 C/C++ 게임 엔진이나 임베디드 코딩 시, 좁은 루프 스코프 안에서 동일한 지역 변수(Register)에 짧게 쓰고 바로 읽는 행위를 빽빽하게 반복하여 극한의 RAW 의존성 병목을 스스로 유발하고 있지 않은가?

### 안티패턴
- **긴 종속성 체인 (Long Dependency Chain) 작성**
  `A = X + Y; B = A * 2; C = B - 3; D = C / 4;` 처럼 이전 라인의 결과를 다음 라인이 무조건 기다려야만 하는 극단적 순차적 코드 구조. 이런 코드는 최신 CPU의 가장 큰 자랑거리인 비순차적 실행(OoO)과 슈퍼스칼라 병렬 처리 능력을 완벽하게 죽여버린다. 이런 지독한 RAW 체인은 똑똑한 컴파일러조차 스케줄링으로 살려낼 구멍이 없으므로, 병렬로 찢어낼 수 있는 수식은 최대한 변수를 쪼개어 독립적으로 짜는 것이 소프트웨어 최적화의 기본이다.

- **📢 섹션 요약 비유**: 4명의 특급 요리사(슈퍼스칼라)가 있는데, 주방장이 요리 순서를 "1번이 썰면 그제야 2번이 볶고, 끝나면 3번이 담고..." 식으로 촘촘하게(종속성 체인) 짜놓으면, 결국 한 번에 한 명밖에 일을 못 해서 비싼 4명의 인건비를 완전히 허공에 날리게 됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 해결 기술 | RAW 방치 시 | 포워딩 + 컴파일러 최적화 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (성능)** | RAW 발생마다 매번 2~3클럭 멈춤 지연 | **0~1클럭** (완벽 은닉 가능) | 명령어 처리량 (IPC) 최대 수치 방어 |
| **정성 (구조)** | 하드웨어 설계는 매우 단순함 | 거미줄 같은 전방 전달 배선으로 칩 복잡도 증가 | 트랜지스터와 면적을 투자해 성능을 돈으로 사는 정석 |

### 미래 전망
- **데이터플로우 아키텍처 (Dataflow Architecture)로의 패러다임 전환**: 전통적인 폰 노이만 구조는 명령어의 '순서'가 중심이지만, RAW 의존성 병목을 근본적으로 깨기 위해 명령어 순서가 아닌 데이터의 '흐름' 자체를 중심으로 하는 데이터플로우나 공간(Spatial) 아키텍처(예: CGRA)가 AI 가속기(NPU)의 표준으로 부상하고 있다. 앞 연산의 데이터가 준비되면 그 데이터가 알아서 다음 연산 게이트를 켜버리는(Trigger) 구조로, 억지스러운 명령어 순서라는 개념 자체를 없애버려 RAW의 늪에서 완전히 탈출한다.

- **📢 섹션 요약 비유**: 무조건 줄을 서서 순서대로만 일을 처리해야 하는 융통성 없는 일직선 공장(폰 노이만 구조)에서 벗어나, 앞 부품이 도착하는 즉시 빈 일꾼들이 순서 상관없이 거미줄처럼 동시다발적으로 조립해버리는 초지능형 셀(Cell) 방식 공장(데이터플로우)으로 진화하고 있습니다.

| **파이프라인 해저드 (Pipeline Hazard)** | RAW는 파이프라인에서 가장 잦게 일어나고, 프로그램 논리상 피할 수도 없는 '데이터 해저드'의 대명사다. |
| **전방 전달 (Forwarding / Bypass)** | RAW로 인한 파이프라인 멈춤(Stall)을 하드웨어 배선을 통해 즉시 0클럭으로 치료하는 핵심 우회 기술이다. |
| **Load-Use 해저드** | 메모리 로드(LW) 명령어 직후에 발생하는 악질적인 RAW로, 포워딩으로도 막을 수 없어 무조건 1클럭의 스톨을 낳는다. |
| **명령어 스케줄링 (Instruction Scheduling)** | 이 Load-Use RAW의 스톨을 지우기 위해 컴파일러가 무관한 다른 명령어를 사이사이에 얄밉게 끼워 넣는 소프트웨어 최적화다. |
| **WAR / WAW** | RAW가 진짜 물리적 인과관계의 의존성이라면, 이 두 개는 비순차 실행 환경에서 레지스터 간판 이름이 겹쳐서 생기는 가짜 의존성이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 1번 친구가 공책에 정답을 다 쓰기도 전에, 성격 급한 2번 친구가 그 공책을 휙 뺏어서 읽으려고 하면 빈칸이나 오답을 보게 되겠죠?
2. 컴퓨터에서도 앞 명령어가 결과(정답)를 다 쓰기도 전에, 바로 뒤 명령어가 성급하게 옛날 값을 읽으려다 대형 사고가 나는 걸 'RAW(쓰고 나서 읽기) 해저드'라고 해요.
3. 이 사고를 막으려고 컴퓨터는 앞 친구가 정답을 계산하자마자, 공책에 굳이 적을 시간도 없이 2번 친구 귀에다 귓속말로 바로 정답을 쏴주는(포워딩) 엄청난 꼼수를 쓴답니다!
