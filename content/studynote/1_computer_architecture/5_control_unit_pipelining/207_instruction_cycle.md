> **핵심 인사이트**
> 1. 명령어 사이클(Instruction Cycle)은 CPU가 메모리에서 **하나의 명령어를 가져와서(Fetch), 해독하고(Decode), 실행한(Execute) 뒤, 결과를 저장(Write-back)하기까지의 전체 과정**을 의미한다.
> 2. 폰 노이만 아키텍처에서 컴퓨터가 켜져 있는 동안 이 사이클은 **무한히 반복(Fetch-Execute Cycle)**되며, 이것이 곧 프로그램이 실행되는 본질적인 메커니즘이다.
> 3. 과거에는 한 명령어가 전체 사이클을 모두 끝내야 다음 명령어가 시작되는 직렬(단일 사이클) 방식이었으나, 현대에는 이 사이클을 여러 단계로 쪼개어 중첩 실행하는 **파이프라이닝(Pipelining)**을 통해 성능을 극대화한다.

---

## Ⅰ. 명령어 사이클의 본질과 시스템 클럭(Clock)

CPU는 스스로 생각해서 움직이는 것이 아니라, 메인보드의 클럭 발생기(Clock Generator)가 주기적으로 보내는 전기적 진동(Tick)에 맞춰 기계적으로 움직입니다.

- **클럭 주기 (Clock Cycle):** CPU가 한 단계의 동작을 수행하는 최소 시간 단위. (예: 3GHz CPU는 1초에 30억 번 진동)
- **명령어 사이클:** 하나의 명령어를 완전히 끝마치는 데 필요한 전체 시간. 명령어의 종류(예: 덧셈 vs 메모리 읽기)에 따라 1클럭 만에 끝날 수도 있고, 수십 클럭이 걸릴 수도 있습니다.
- **기본 흐름:** `인출(Fetch) $\rightarrow$ 해독(Decode) $\rightarrow$ 실행(Execute)`의 3단계가 가장 기본적인 뼈대이며, 아키텍처에 따라 메모리 접근이나 인터럽트 처리 단계가 추가됩니다.

📢 **섹션 요약 비유:**
명령어 사이클은 심장 박동(클럭)에 맞춰 숨을 들이마시고(Fetch), 무슨 냄새인지 판단하고(Decode), 내뱉는(Execute) 하나의 완벽한 호흡 과정입니다. 컴퓨터가 켜져 있는 한 이 호흡은 멈추지 않습니다.

## Ⅱ. 명령어 사이클의 세부 단계 (기본 4단계)

현대의 RISC 아키텍처를 기준으로 명령어 사이클은 보통 4~5개의 세부 마이크로 오퍼레이션 단계로 나뉩니다.

1. **인출 사이클 (Fetch Cycle):**
   - 현재 PC(프로그램 카운터)가 가리키는 메모리 주소에서 명령어를 가져와 명령어 레지스터(IR)에 저장합니다.
   - 동시에 PC의 값을 다음 명령어 위치(PC + 4)로 증가시킵니다.
2. **해독 사이클 (Decode Cycle):**
   - 제어 유닛이 IR에 들어온 명령어의 Opcode를 분석하여 무슨 연산인지 파악합니다.
   - 동시에 연산에 필요한 데이터(오퍼랜드)를 레지스터 파일에서 읽어옵니다.
3. **실행 사이클 (Execute Cycle):**
   - ALU(산술 논리 연산 장치)가 실제로 덧셈, 뺄셈, 논리 연산을 수행하거나 메모리 접근을 위한 유효 주소(EA)를 계산합니다.
4. **저장 / 쓰기 사이클 (Store / Write-back Cycle):**
   - 연산된 최종 결과값을 CPU 내부의 레지스터나 메인 메모리에 덮어써서(저장하여) 상태를 갱신합니다.

📢 **섹션 요약 비유:**
우체통에서 편지를 꺼내고(인출), 편지 봉투를 뜯어 내용을 읽고(해독), 내용대로 심부름을 한 뒤(실행), 결과를 장부에 기록(저장)하는 업무 루틴입니다.

## Ⅲ. 특수 사이클: 간접(Indirect)과 인터럽트(Interrupt)

기본 사이클 외에 상황에 따라 사이클이 더 길어지고 복잡해질 수 있습니다. 이를 별도의 사이클로 정의합니다.

1. **간접 사이클 (Indirect Cycle):**
   - 명령어의 오퍼랜드가 '직접 주소'가 아닌 '간접 주소(주소를 가리키는 주소)'일 경우 발생합니다.
   - 해독 단계 이후, 진짜 유효 주소(EA)를 얻기 위해 메모리를 한 번 더 읽어와야 하는 추가적인 지연 단계입니다.
2. **인터럽트 사이클 (Interrupt Cycle):**
   - 하나의 명령어 실행(Execute)이 끝난 직후, CPU는 항상 "혹시 방금 들어온 긴급 요청(마우스 클릭, 타이머 등)이 있나?" 하고 확인합니다.
   - 인터럽트가 발생했다면, 원래 하던 일의 상태(PC 값 등)를 스택에 백업하고, 인터럽트 처리 루틴(ISR) 주소로 PC를 강제 변경하는 특수 사이클입니다.

📢 **섹션 요약 비유:**
간접 사이클은 심부름을 갔더니 "진짜 물건은 옆 창고에 있다"고 해서 한 번 더 걸음을 하는 것이고, 인터럽트 사이클은 업무 하나를 끝낼 때마다 혹시 사장님(OS)이 긴급 호출을 했는지 삐삐를 확인하는 과정입니다.

## Ⅳ. 사이클 상태도 (State Transition Diagram)

CPU의 제어 유닛은 내부적으로 상태 머신(State Machine)으로 동작하며, 클럭마다 다음 상태로 전이합니다.

```text
       [ 시스템 부팅 / 초기화 ]
                |
                v
+-----> [ 인출(Fetch) ] 
|               |
|               v
|       [ 해독(Decode) ] -----------------+
|               |                         |
|               v                         v
|    (간접 주소 방식인 경우) ---> [ 간접(Indirect) ]
|               |                         |
|               v                         |
|       [ 실행(Execute) ] <---------------+
|               |
|               v
|  (인터럽트 발생 시) ----------> [ 인터럽트(Interrupt) ]
|               |                         |
+---------------+-------------------------+
       (다음 명령어 인출로 무한 반복)
```

이 사이클을 물리적으로 어떻게 겹치느냐에 따라 파이프라이닝(Pipelining) 성능이 결정됩니다.

📢 **섹션 요약 비유:**
쳇바퀴처럼 돌아가는 뫼비우스의 띠입니다. 한 바퀴 도는 중에 간접 주소라는 샛길로 빠질 때도 있고, 인터럽트라는 비상구로 다녀올 때도 있지만, 결국 다시 처음(Fetch)으로 돌아와 다음 명령어를 처리합니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Program Counter (PC)]`: 인출 사이클의 시작점이자 명령어의 주소를 제공하는 레지스터
- `[Instruction Register (IR)]`: 인출된 명령어가 해독되기 위해 잠시 머무는 정거장
- `[Control Unit]`: 명령어 사이클의 각 단계마다 데이터패스의 MUX와 레지스터를 통제하는 유닛
- `[Pipelining]`: 한 명령어의 사이클이 다 끝나기 전에 다음 명령어의 인출 사이클을 겹쳐서 시작하는 최적화 기법
- `[Interrupt Handling]`: 실행 사이클 직후에 시스템의 긴급한 이벤트를 처리하기 위해 끼어드는 하드웨어 메커니즘
