# # [상태표 (State Table)]

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 상태표(State Table)는 유한 상태 머신(Finite State Machine, FSM)의 현재 상태(Present State), 입력(Input), 그에 따른 차기 상태(Next State) 및 출력(Output) 간의 모든 가능한 논리적 조합을 행렬(Matrix) 형태로 전수 나열한 **형식적 명세서(Formal Specification)**다.
> 2. **가치**: 시각적인 상태도(State Diagram)의 정보를 정량화하여, 불 대수(Boolean Algebra)를 이용한 논리 간소화 및 플립플롭(Flip-Flop)의 입력 식(Excitation Equation)을 도출하기 위한 **필수적인 중간 데이터 변환 계층**을 제공한다.
> 3. **융합**: 현대 EDA(Electronic Design Automation) 툴의 논리 합성(Logic Synthesis) 엔진은 설계자가 작성한 RTL 코드를 내부적으로 상태표로 파싱(Parsing)하여 최적의 하드웨어 게이트 레벨을 생성하며, 이는 컴파일러 최적화 기술과 맞닿아 있다.

---

### Ⅰ. 개요 (Context & Background)

**개념 (Definition)**
상태표는 디지털 시스템의 **순차 논리(Sequential Logic)** 동작을 표현하는 가장 기초이고도 강력한 수학적 모델입니다. 조합 논리(Combinational Logic) 회로가 입력만으로 출력이 결정되는 '무기억성' 시스템인 반면, 순차 논리는 **과거의 이력(현재 상태, $Q$)**과 **현재의 입력($x$)**이 결합하여 미래의 상태($Q^+$)와 출력($y$)을 결정하는 '유기체적' 시스템입니다. 상태표는 이러한 시간의 흐름에 따른 상태 변화를 $x, Q, Q^+, y$라는 변수들의 진리표 형태로 기술한 것입니다. 본질적으로 시스템의 행위를 기계가 해석 가능한 **상태 천이 행렬(State Transition Matrix)**으로 환원한 작업입니다.

**💡 비유 (Analogy)**
상태표는 복잡한 **'기차의 운행 시간표 및 경로도'**와 같습니다.
-   **현재 상태**: 기차가 현재 어느 역(서울역)에 있는가.
-   **입력**: 기사가 어떤 신호(직진/환승)를 받았는가.
-   **차기 상태**: 다음에 어느 역(부산역/광명역)으로 이동하는가.
-   **출력**: 그 과정에서 안내 방송을 켤 것인가, 문을 열 것인가.
이 모든 경우의 수를 표로 만들어두면, 기사(제어 장치)는 매번 상황을 판단하지 않고 표만 보고 안전하게 기차를 운행할 수 있습니다.

**등장 배경 (Background)**
1.  **기존 한계**: 초기 디지털 설계는 단순히 회로도(Schematic) 위주로 진행되었으나, 시스템이 복잡해짐에 따라 직관적인 회로도만으로는 '상태'의 흐름을 추적하고 최적화하기 어려워졌습니다.
2.  **혁신적 패러다임**: 수학자와 공학자들은 회로의 동작을 기호(Symbol)와 테이블(Table) 형식으로 분리하여 정의하면, 카르노 맵(Karnaugh Map)이나 퀸-맥클러스키(Quine-McCluskey) 알고리즘과 같은 수학적 도구를 통해 **최소한의 게이트**로 회로를 구성할 수 있음을 발견했습니다.
3.  **현재의 비즈니스 요구**: 오늘날 CPU의 파이프라인 제어 유닛(Control Unit)이나 통신 프로토콜의 핸드셰이크 로직처럼, 수십억 개의 트랜지스터가 움직이는 초고속 시스템에서 인간의 직관이 아닌 상태표 기반의 **정형 검증(Formal Verification)**이 필수적입니다.

**📢 섹션 요약 비유**
> 마치 복잡한 범죄 사건의 모든 증거와 시간대를 타임라인 표로 완벽하게 재구성하면, 범행의 동기(입력)와 결과(출력) 사이의 인과관계를 수학적으로 증명할 수 있듯이, 상태표는 시스템의 모든 동작 시나리오를 증거 테이블에 기록하여 설계의 오류를 원천 차단하는 법적 증거물과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

상태표의 구조는 크게 무어(Moore)형과 밀리(Mealy)형 머신에 따라 출력(Output) 열의 배치가 달라지며, 내부적으로는 현재 상태 저장을 위한 메모리 요소와 상태를 결정하는 조합 논리 요소로 분해됩니다.

#### 1. 핵심 구성 요소 (Component Table)

| 요소명 | 역할 (Role) | 내부 동작 (Internal Operation) | 비유 (Analogy) |
|:---|:---|:---|:---|
| **PS (Present State)** | 현재 상태 레지스터 | 클럭(Clock) 에지에 맞춰 이전 상태($Q$)를 유지함. 시스템의 단기 기억. | 사람의 현재 기분 상태 |
| **NS (Next State)** | 차기 상태 로직 | 입력($Inputs$)과 PS를 연산하여 다음에 저장될 값($D, T, J, K$)을 결정. | 다음 순간에 취할 행동 계획 |
| **Inputs** | 외부 신호 | 시스템의 상태를 바꾸는 트리거(Trigger). 단순 조합 논리 입력과 달리 시간적 개입. | 벨을 누르는 손가락 |
| **Outputs** | 시스템 반응 | PS(Mealy는 PS+Input)에 의해 결정됨. 상태에 따른 LED, 제어 신호 등. | 기분에 따른 표정/행동 |

#### 2. 아키텍처 다이어그램: 상태표의 하드웨어 매핑

아래 다이어그램은 추상적인 상태표가 어떻게 실제 하드웨어 블록과 매핑되는지를 보여줍니다.

```text
       ┌───────────────────────────────────────────────────────────────┐
       │                    FSM 하드웨어 모델 (Hardware Mapping)        │
       └───────────────────────────────────────────────────────────────┘
  
  외부 입력[x] ────┐
                  │
                  ▼
         ┌─────────────────┐
         │   Next State    │  [상태표의 "Next State" 열 구현부]
         │   Logic (Combi) │  ────> ① Q, x 조합 분석
         └────────┬────────┘
                  │
            D input │
                  ▼
         ┌─────────────────┐
         │   State Memory  │  [상태표의 "Present State" 구현부]
         │  (Flip-Flops)   │  ────> ② 클럭(CLK) 마다 상태 갱신 (Q -> Q+)
         └────────┬────────┘
                  │ Q (Current State)
                  ├───────────────────┐
                  │                   │
                  ▼                   ▼
         ┌─────────────────┐   ┌─────────────────┐
         │   Output Logic  │   │   Output Logic  │
         │   (Moore Type)  │   │  (Mealy Type)   │
         └─────────────────┘   └────────┬────────┘
                  │                   │
                  └───────┬───────────┘
                          ▼
                     시스템 출력 [y]
```

**[다이어그램 해설]**
이 다이어그램은 상태표가 단순한 종이 위의 표가 아니라, **'논리(Logic)'와 '기억(Memory)'의 순환 구조**를 물리적으로 구현한 설계도임을 보여줍니다.
1.  **상태 추출(State Extraction)**: 상태표의 각 행(Row)은 '현재 상태 $Q$'를 나타내며, 이는 실제 하드웨어에서 플립플롭의 출력 단자와 직결됩니다.
2.  **조합 논리 구현**: '차기 상태($Q^+$)' 열의 내용을 구현하기 위해, 입력 $x$와 현재 $Q$를 입력으로 받는 AND-OR 게이트 조합이 필요합니다. 상태표의 내용이 곧 게이트 레벨의 회로도가 되는 것입니다.
3.  **피드백 루프**: 출력 $Q^+$가 다시 클럭에 맞춰 입력 $Q$로 들어오는 피드백 구조가 상태표의 '시간적 흐름'을 하드웨어적으로 구현하는 핵심 메커니즘입니다.

#### 3. 심층 동작 원리 및 알고리즘

상태표를 실제 회로로 구현하는 과정은 다음과 같은 수학적 알고리즘을 따릅니다.

**Step 1: 상태 부호화 (State Encoding/Assignment)**
-   상태표의 기호 $S_0, S_1, S_2$를 이진수로 변환.
-   **Binary Code**: $00, 01, 10$ (Flip-Flop 2개 필요, 효율성 중시)
-   **One-Hot Code**: $01, 10, 100$ (Flip-Flop 3개 필요, 속도 중시)

**Step 2: 여기 테이블(Excitation Table) 변환**
-   사용할 플립플롭(예: D FF, JK FF)의 특성에 맞춰 상태표의 값($Q \to Q^+$)을 입력값($D$ 또는 $J, K$)으로 변환.
    -   **D FF**: $D = Q^+$ (그대로 대입)
    -   **JK FF**: $0\to0$ (J=0, K=X), $0\to1$ (J=1, K=X) 등으로 변환 필요.

**Step 3: 불 대수 최적화 및 회로 구현**
-   변환된 테이블을 바탕으로 각 플립플롭의 입력 함수 $D_1(Q, x), D_0(Q, x)$를 도출.
-   카르노 맵을 이용해 로직을 최소화.

```verilog
// [Code Snippet] 상태표를 기반으로한 Verilog RTL 설계 예시
// 상태표: S0(00) -> x=1 -> S1(01), Output=0; S1(01) -> x=1 -> S2(11), Output=1

module fsm_state_table (
    input clk, reset, x,
    output reg y
);
    // 1. 상태 변수 정의 (State Encoding: Binary)
    reg [1:0] state, next_state;
    localparam S0=2'b00, S1=2'b01, S2=2'b11; // 상태표의 행(Row) 정의

    // 2. 상태 레지스터 (Memory Section)
    always @(posedge clk or posedge reset) begin
        if (reset) state <= S0;
        else       state <= next_state; // 상태표의 차기 상태 적용
    end

    // 3. 차기 상태 로직 (Next State Logic - State Table 구현부)
    // state(S0)이고 x=1 이면 next_state = S1...
    always @(*) begin
        case (state)
            S0: next_state = (x) ? S1 : S0; 
            S1: next_state = (x) ? S2 : S0;
            S2: next_state = (x) ? S0 : S2;
            default: next_state = S0;
        endcase
    end

    // 4. 출력 로직 (Output Logic)
    always @(*) begin
        y = (state == S2) ? 1 : 0; // Moore Machine 출력 정의
    end
endmodule
```

**📢 섹션 요약 비유**
> 상태표는 마치 요리사의 **'레시피 공식책'**과 같습니다. 단순히 "맛있게 만드세요"(상태도)라고 말하는 대신, "토마토 2개(입력)와 현재 불 강도(상태)가 'High'일 때, 5분 후(차기 상태) 불을 'Low'로 줄이고 소금을 1g 넣으라(출력)"는 식으로 모든 조건과 행동을 수치화된 매트릭스로 제공하여, 누가 요리를 하더라도(H/W 구현) 완벽한 같은 맛을 낼 수 있게 하는 표준화된 조리법입니다.

### Ⅲ. 융합 비교 및 다각도 분석

#### 1. 진리표 vs 상태표 비교 (기술적 심층 분석)

| 구분 | 진리표 (Truth Table) | 상태표 (State Table) |
|:---|:---|:---|
| **저장소 유무** | **X (Memoryless)** | **O (Memory Element: FF)** |
| **시간 축** | 0차원 (순간적) | 1차원 (과거 $\to$ 현재 $\to$ 미래) |
| **입력 정의역** | $f(A, B, C \dots)$ | $f(Input(t), State(t))$ |
| **표현 능력** | 정적 논리만 표현 | **순차적 패턴, 타이밍, 제어 흐름 표현** |
| **주요 응용** | ALU, Encoder, Decoder | **CPU Control Unit, UART, Protocols** |
| **과목 융합** | 논리 회로(디지털) 기초 | **컴퓨터 구조(CPU Control)**와 직결 |

#### 2. 모델 간 비교: Moore vs Mealy Machine State Table

상태표 작성 시 출력(Output) 열을 어디에 두느냐에 따라 성능이 달라집니다.

| 비교 항목 | Moore Machine 상태표 | Mealy Machine 상태표 |
|:---|:---|:---|
| **출력 결정 요소** | 현재 상태($Q$)만 dependent | **현재 상태($Q$) + 입력($x$)** dependent |
| **상태표 구조** | 출력 열이 상태 열 하나에만 존재 | 출력 열이 입력 조합별로 분기되어 존재 |
| **응답 속도** | 느림 (CLK 1사이클 지연) | **빠름 (입력 즉시 반영)** |
| **설계 난이도** | 쉬움 (독립적) | 어려움 (Glitch/Noise 민감) |
| **실무 적용** | 신뢰성 중시 시스템 | **고속 통신 프로토콜 처리** |

#### 3. 과목 융합 분석: OS 스케줄링과의 연관성
컴퓨터 구조의 상태표는 OS(Operating System)의 **프로세스 상태 블록(PCB) 관리**와 논리적으로 동일합니다.
-   **Ready $\to$ Running**: OS 스케줄러(Context Switching)가 인터럽트(입력)를 받아 PCB(상태)를 업데이트하는 것은 거대한 상태표의 로직과 같습니다.
-   차이점이라면 HW 상태표는 나노초 단위의 클럭으로, OS는 밀리초 단위
