# # [감산기 (Subtractor)]

## #### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 감산기(Subtractor)는 두 이진수 간의 차(Difference)를 도출하고, 피감수가 감수보다 작을 경우 상위 비트로부터 빌림수(Borrow)를 생성하는 **조합 논리 회로(Combinational Logic Circuit)**의 핵심이다.
> 2. **가치**: ALU (Arithmetic Logic Unit) 내에서 산술 연산의 기초를 담당하며, 특히 뺄셈 연산을 $A + \bar{B} + 1$ 형태로 변환하여 하드웨어 자원을 최적화하는 **2의 보수(2's Complement)** 연산법과 밀접하게 연관되어 있다.
> 3. **융합**: 현대 프로세서 설계에서는 독립적인 감산기보다 가산기(Adder)와의 통합형 설계가 표준이며, 이는 제어 신호(Control Signal) 하나로 덧셈과 뺄셈을 모두 처리하는 고성능 저전력 아키텍처를 가능하게 한다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념 및 정의
**감산기(Subtractor)**는 디지털 회로에서 두 이진수 간의 뺄셈을 수행하는 회로이다. 기본적으로 입력으로 피감수($A$, Minuend)와 감수($B$, Subtrahend)를 받아, 연산의 결과인 **차(Difference, $D$)**와 연산 수행 중 필요한 **빌림수(Borrow, $B_{out}$)**를 출력한다.
뺄셈의 논리는 단순히 $A$에서 $B$를 빼는 것이 아니라, 해당 비트가 해결할 수 없는 상황($0-1$)에서 상위 비트에게 "1을 빌려왔음"을 알리는 신호를 생성해야 하므로, 가산기보다 상대적으로 복잡한 조건 분기가 필요하다.

#### 💡 비유
감산기는 마치 **"물품 정산을 담당하는 회계사"**와 같다.
- **차(Difference)**: 최종적으로 남은 잔액(결과값)이다.
- **빌림수(Borrow)**: 내 계좌(현재 비트)에 돈이 부족하여 상위 부서(상위 비트)에 "잠시 돈을 빌려쓰고 뒤에서 갚겠다"는 문서를 보내는 행위다. 상위 비트는 이 요청을 받아 자신의 값에서 1을 깎아준다.

#### 등장 배경: 연산 유닛의 진화
초기 컴퓨터 설계에서는 덧셈을 하는 가산기와 뺄셈을 하는 감산기가 물리적으로 분리되어 있었다. 그러나 이는 칩의 면적(Die Area)을 낭비하고 전력 소모를 증가시키는 주요 원인이었다.
수학적으로 "뺄셈($A - B$)"은 "$A$와 $B$의 2의 보수를 더하는 것($A + \bar{B} + 1$)"으로 완전히 대체 가능함이 증명되면서, 별도의 감산기보다는 **가산기의 입력단에 보수 로직(Inverter)을 결합**하여 뺄셈을 수행하는 방식으로 패러다임이 전환되었다.

#### 📢 섹션 요약 비유
감산기의 개념 정립은 마치 **'별도의 냉난방기'를 설치하던 것을 '에어컨의 흐름을 반전시켜 난방을 하는' 히트펌프 방식으로 진화**시킨 것과 같습니다. 독립된 기계가 아닌, 회로의 흐름 제어를 통해 두 가지 기능을 모두 해결하는 효율성을 추구한 것입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 구성 요소 상세 분석
감산기는 이전 비트에서 빌려온 수($B_{in}$)를 고려하느냐 마느냐에 따라 다음과 같이 구분된다.

| 요소명 | 역할 | 내부 동작 (Boolean Function) | 비고 |
|:---|:---|:---|:---|
| **반감산기<br>(Half Subtractor)** | 1비트 뺄셈 수행<br>(빌림수 입력 없음) | $D = A \oplus B$<br>$B_{out} = \bar{A}B$ | 하위 비트에서 올라온 빌림을 처리 못 하여 다단 연결 불가 |
| **전감산기<br>(Full Subtractor)** | 1비트 뺄셈 수행<br>(빌림수 입력 포함) | $D = A \oplus B \oplus B_{in}$<br>$B_{out} = \bar{A}B + \bar{A}B_{in} + BB_{in}$<br>= $\bar{A}(B + B_{in}) + BB_{in}$ | **리플(Ripple) 감산기** 구성의 기본 블록 |

#### 2. 논리 구조 및 다이어그램
아래는 반감산기와 전감산기의 내부 게이트 레벨 구조를 비교한 도식이다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│  [Subtractor Architecture: Half Subtractor vs Full Subtractor]              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ① 반감산기 (Half Subtractor)                           ② 전감산기 (Full Subtractor)  │
│                                                                             │
│      A ────┐                                       A ────┐                   │
│             ├─▶[ XOR ]───▶ Difference (D)                 │   ┌─────────┐      │
│      B ────┘      │                                 B ────┼──▶│   HS1   │───┐  │
│                    │                                    │   └─────────┘   │  │
│                    ▼                                    │                 │  │
│      A ──┬─▶[NOT]                                       ▼                 │  │
│           \     AND                                          ┌─────────┐  │  │
│            \    /                      Borrow_in (Bin) ─────┤   HS2   ├──┘  │
│             \  /                                       ───▶│(2nd HS) │     D
│              \ /                                       ───▶└─────────┘   (Diff)
│               ▼                                            ▲            │
│      B ────▶[ AND ]──────▶ Borrow (Bout)                     │            │
│                                                         HS1 Bout ────────┼──▶[ OR ]───▶ Borrow_out (Bout)
│                                                                         │
└─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 심층 해설]**
1.  **반감산기(HS)**: XOR 게이트가 차($D$)를 계산하고, NOT-AND 조합이 빌림($B_{out}$)을 생성한다. 핵심은 $A=0, B=1$일 때만 빌림이 발생한다는 점이다.
2.  **전감산기(FS)**: 두 개의 반감산기(HS1, HS2)와 하나의 OR 게이트로 구성된다.
    -   **1단계(HS1)**: 피감수($A$)와 감수($B$)를 먼저 계산하여 중간 차와 빌림을 생성한다.
    -   **2단계(HS2)**: 1단계의 결과와 하위 비트에서 넘어온 빌림수($B_{in}$)를 다시 연산한다.
    -   **Bout 로직**: HS1의 빌림 OR HS2의 빌림 중 하나라도 발생하면 상위 비트로 빌림을 전달한다. 이는 **"감수가 1이거나 하위 빌림이 있을 때, 피감수가 0이면 빌림이 발생한다"**는 논리를 구현한다.

#### 3. 실무 핵심 알고리즘: 2의 보수를 이용한 뺄셈 (Gate Level)
현대 하드웨어는 전감산기를 직렬로 연결하는 대신, 가산기(Adder)를 재사용한다. 이를 위한 회로 변환은 다음과 같다.

$$A - B = A + (\text{2's Complement of } B) = A + \bar{B} + 1$$

이를 구현하기 위해 가산기의 입력 $B$ 앞에 **XOR 게이트**를 배치하고, `Subtract Mode` 신호(1)를 인가하면 $B$를 반전(Invert)시키면서 동시에 `Carry Input(Cin)`에 1을 더해주는 방식을 사용한다.

```verilog
// Hardware Description Language (Verilog) Snippet
module Subtractor_Using_Adder (
    input wire [3:0] A, B,
    input wire Sub_Mode, // 1 for Subtraction, 0 for Addition
    output wire [3:0] Sum,
    output wire Cout
);
    wire [3:0] B_inv;

    // XOR acts as a controlled inverter
    // If Sub_Mode is 1, B becomes ~B (Inverted)
    genvar i;
    generate
        for (i = 0; i < 4; i = i + 1) begin
            xor inv_gate (B_inv[i], B[i], Sub_Mode);
        end
    endgenerate

    // Reuse the standard Adder
    // Carry_In (Cin) receives the Sub_Mode signal (acts as +1)
    Adder_4bit ALU_ADD (
        .A(A),
        .B(B_inv),
        .Cin(Sub_Mode), // Add 1 if subtraction mode
        .Sum(Sum),
        .Cout(Cout)
    );
endmodule
```

#### 📢 섹션 요약 비유
감산기의 내부 동작 원리는 마치 **"숫자가 큰 돈을 교환하는 과정"**과 같습니다. 만약 100원짜리(상위 비트)가 없으면 10원짜리 10개를 바꿔야 하듯이, 감산기는 하위 비트에서 "0-1"이 발생하면 상위 비트의 1을 뽑아 10(이진 2)으로 만들어 계산을 해결합니다. 가산기를 이용한 뺄셈은 **"마이너스 버튼을 누르면 기계가 계산식을 뒤집어서 더해주는"** 첨단 계산기의 로직과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

#### 비교 1: 구조적 특성 분석 (가산기 vs 감산기)

| 구분 | 가산기 (Adder) | 감산기 (Subtractor) |
|:---|:---|:---|
| **목적** | 이진수의 합($Sum$) 산출 | 이진수의 차($Difference$) 산출 |
| **주요 출력** | Sum (합), Carry (올림수) | Difference (차), Borrow (빌림수) |
| **상태 발생 조건** | Carry: 두 입력이 모두 1일 때 생성 ($A \cdot B$) | Borrow: 피감수가 0이고 감수가 1일 때 생성 ($\bar{A} \cdot B$) |
| **입력 조건** | $A+B+C_{in}$ | $A-B-B_{in}$ |
| **논리 복잡도** | 상대적으로 단순 (XOR + AND) | 반전 로직이 추가되어 약간 복잡함 (NOT + AND) |

#### 비교 2: 설계 패러다임 비교 (독립형 vs 통합형)

| 비교 항목 | 독립형 감산기 (Standalone) | 가산기 기반 통합형 (Integrated ALU) |
|:---|:---|:---|
| **하드웨어 복잡도** | 높음 (뺄셈을 위한 별도의 Gate Array 필요) | 낮음 (XOR Gate + Mux만 추가) |
| **연산 속도 (Latency)** | 빌림수(Ripple) 전파 지연 존재 | 가산기의 Carry Chain 지연에 의존 |
| **전력 소모 (Power)** | 높음 (유휴 회로 많음) | 낮음 (회로 공유로 인한 스위칭 감소) |
| **설계 유연성** | 낮음 (기능 고정) | 높음 (S/W 명령어로 Add/Sub 선택) |
| **주요 적용 분야** | 초보용 디지털 교육용 kit | 범용 CPU, GPU, MCU의 ALU |

#### 📢 섹션 요약 비유
가산기와 감산기의 관계는 **"직원용 엘리베이터와 화물용 엘리베이터"**를 따로 두느냐, **"하나의 엘리베이터에 모드 버튼(승객/화물)을 두느냐"**의 차이와 같습니다. 현대의 ALU 설계는 후자를 선택하여 공간 효율성을 극대화하고, 제어 신호(Control Signal) 하나로 운영 모드를 전환하여 자원을 효율적으로 재활용합니다.

### Ⅳ. 실무 적용 및 기술사적 판단

#### 실무 시나리오: CMP 명령어와 Zero Flag의 관계
**상황**: 32비트 CPU에서 `CMP EAX, EBX` (EAX와 EBX 비교) 명령어가 실행될 때의 내부 동작을 설계하라.

**1. 문제 정의**:
비교 연산은 결과값(차)을 메모리에 저장하지 않고, 단지 두 값의 관계(크다, 작다, 같다)만을 판단해야 한다.

**2. 아키텍처적 해결책**:
- **감산기(or Adder) 활용**: 내부적으로는 `SUB EAX, EBX` 연산을 수행한다.
- **Flag 레지스터 판단**:
    -   **Zero Flag (ZF)**: 연산 결과 $D$가 0이면 두 값이 같음 ($ZF=1$).
    -   **Carry Flag (CF)**: 뺄셈에서 빌림수($B_{out}$)가 발생했는지 확인. (부호 없는 수에서 $EAX < EBX$ 판단).
    -   **Sign Flag (SF)**: 연산 결과의 최상위 비트(MSB)가 1인지 확인 (부호 있는 수 판단).
- **결과 폐기**: 연산 결과 $D$는 버리고 플래그 레지스터만 갱신한다.

**3. 기술적 이슈 - 오버플로우(Overflow)**:
부호 있는 정수 뺄셈에서는 올바른 부호 비트가 뒤집히는 **Arithmetic Overflow**를 감지해야 한다. 이는 $B_{in}$과 $B_{out}$이 다르거나, 결과가 입력 범위를 벗어날 때 발생하며, 보호된 시스템(Protected Mode) OS에서는 이를 트랩(Trap)하여 예외 처리를 해야 한다.

#### 도입 체크리스트 (Subtractor/ALU 구현 시)
- [ ] **빌림수(Borrow) 처리 정확성**: 모든 비트에 대해 $0-1$ 상황에서 상위 비트로 정확히 1이 전파되는가?
- [ ] **2의 보수 범위 검증**: 최소값(Min)에 대한 뺄셈을 수행했을 때 올바른 양수로 복원되는가 (Overflow)?
- [ ] **지연 시간(Latency) 최적화**: 리픈 캐리(Ripple Carry) 방식이 아닌 **
