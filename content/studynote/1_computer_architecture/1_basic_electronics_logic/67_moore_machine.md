# # [67. 무어 머신 (Moore Machine)]

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 무어 머신(Moore Machine)은 유한 상태 머신(Finite State Machine, FSM)의 대표적인 유형으로, 시스템의 **출력(Output)이 오직 현재 상태(Present State)**에 의해서만 결정되는 동기식 순차 논리(Synchronous Sequential Logic) 회로이다.
> 2. **가치**: 출력 로직이 입력 경로와 완전히 분리되어 있어, 외부 입력의 노이즈나 간섭이 출력부로 직접 전파되지 않는다. 이는 **글리치(Glitch)가 없는(Clitch-free) 신호**를 보장하여 하드웨어 타이밍 클로저(Timing Closure)와 무결점(Fail-safe) 제어 설계에 필수적이다.
> 3. **융합**: CPU(Central Processing Unit)의 제어 유닛(Control Unit) 설계, 통신 프로토콜의 프레이머(Framer), 그리고 고속 인터페이스의 리타이밍(Retiming) 로직 등 신호 무결성이 생명인 시스템의 표준 아키텍처로 사용된다.

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의 및 철학
무어 머신은 1956년 에드워드 F. 무어(Edward F. Moore)가 제안한 모델로, **"시스템의 행동은 현재의 상태에 의해 완전히 정의된다"**는 원칙을 따른다. 수학적으로 $Output(t) = f(State(t))$로 정의되며, 출력 함수 $f$는 입력 $Input(t)$와 독립적이다.
이는 입력 변화가 발생하더라도 시스템이 내부 클럭(Clock) 에지(Edge)에 맞춰 상태를 갱신하기 전까지는 출력이 변하지 않음을 의미한다. 즉, **출력의 '원자성(Atomicity)'**을 보장하는 것이 핵심 철학이다.

### 2. 등장 배경: 비동기 문제에 대한 동기식 해법
초기 디지털 시스템에서는 입력 신호의 변화가 곧바로 출력으로 이어지는 조합 논리(Combinational Logic) 방식을 주로 사용했다. 그러나 회로가 복잡해지며 입력 신호의 전파 지연(Propagation Delay) 차이로 인해 일시적이고 허위인 펄스인 '허저드(Hazard)'나 '글리치'가 발생하여 시스템 오동작을 유발했다.
이를 해결하기 위해, 입력을 일단 '상태'라는 버퍼(Buffer)에 저장한 뒤, **전역 클럭(Global Clock)**에 동기화하여 출력을 내보내는 무어 모델이 등장했다. 이는 복잡한 디지털 시스템의 신뢰성을 담보하는 기반이 되었다.

### 3. 비유
- **💡 비유**: 무어 머신은 **"일렬로 서서 차례를 기다리는 톨게이트 시스템"**과 같다. 차량(입력)이 아무리 빠르게 접근해도, 톨게이트의 바리케이드(출력)는 오직 차단기가 "열림 상태"로 바뀌는 순간에만 움직인다. 차량이 멈춰 있든 달리든 상관없이, 바리케이드의 동작은 오직 시스템의 내부 상태(열림/닫힘)에만 의존한다.

> 📢 **섹션 요약 비유**: 입력이라는 교통 체증에 상관없이, 오직 정해진 '상태'라는 신호등에 따라만 운영되는 엄격한 일방통행 제어 시스템입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 하드웨어 구조 및 데이터 흐름
무어 머신의 하드웨어는 크게 '상태 기억 장치(State Memory)', '차기 상태 논리(Next State Logic)', '출력 논리(Output Logic)'의 3가지 블록으로 구성된다. 밀리 머신(Mealy Machine)과 결정적인 차이점은 출력 논리에 입력이 직접 연결되지 않는다는 점이다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                      Moore Machine Architecture                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   External Inputs (X)                                                   │
│         │                                                               │
│         ▼                                                               │
│  ┌──────────────────────┐                                               │
│  │ Next State Logic     │ (Present State + Input → Next State)         │
│  │ (Combinational Logic)│                                               │
│  └──────────┬───────────┘                                               │
│             │ Next State (D)                                            │
│             ▼                                                           │
│       ┌─────────────┐                                                   │
│  CLK──▶│    State    │──▶ Present State (Q) ────────┐                   │
│  (Edge) │  Memory    │    (Current State Code)      │                   │
│       │ (Flip-Flops) │                             │                   │
│       └─────────────┘                             │                   │
│             ▲                                     │                   │
│             │                                     │                   │
│             │ Feedback                            │                   │
│             │                                     ▼                   │
│             │                        ┌──────────────────────┐          │
│             └────────────────────────│ Output Logic         │          │
│                                     │ (Combinational Logic)│          │
│                                     │ Only uses (Q)        │          │
│                                     └──────────┬───────────┘          │
│                                                │                      │
│                                                ▼                      │
│                                         System Outputs (Y)             │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
위 다이어그램에서 가장 중요한 점은 **입력 신호(X)가 출력 로직 블록으로 직접 들어가지 않는다**는 것이다. 모든 입력은 '차기 상태 논리'를 거쳐 다음 상태를 계산하는 데에만 사용되며, 출력(Y)은 오직 현재 상태 레지스터(Q)의 값만을 참조한다. 이로 인해 입력 핀(Input Pin)에서 발생한 스파이크(Spike)나 노이즈가 출력 핀으로 즉시 전달되는 것을 물리적으로 차단한다.

### 2. 구성 요소 상세 분석표

| 요소명 (Component) | 역할 (Role) | 내부 동작 (Internal Operation) | 주요 프로토콜/특징 | 비유 |
|:---|:---|:---|:---|:---|
| **State Memory** | 현재 상태 유지 | 클럭 에지(Edge)에서 D 입력값을 Q에 래치(Latch)하여 저장 | 레벨 트리거(Level-triggered) 아닌 에지 트리거(Edge-triggered) FF 사용 | 정주소(현재 위치) |
| **Next State Logic** | 상태 천이 계산 | $f(Input, CurrentState)$를 연산하여 다음 상태 코드 생성 | 조합 논리(AND/OR/XOR 등) 게이트로 구성 | 내비게이션(경로 탐색) |
| **Output Logic** | 출력값 생성 | $g(CurrentState)$ 함수에 따라 상태에 매핑된 출력값 디코딩 | 입력 독립적, Decoding 속도가 Fmax에 영향 | 해당 위치의 표지판 |
| **Clock Generator** | 동기화 담보 | 모든 상태 변경의 시점(timing reference)을 제공 | 전역 클럭 트리(Global Clock Tree)를 통해 스큐(Skew) 최소화 | 지휘관의 호루라기 |

### 3. 핵심 알고리즘 및 타이밍 다이어그램

무어 머신의 출력은 입력이 변해도 바로 바뀌지 않고, 반드시 **한 클럭 사이클(Clock Cycle)의 지연(Latency)**을 갖는다. 이를 수식과 타이밍 다이어그램으로 확인한다.

**타이밍 로직:**
1. **T0**: 현재 상태 $S_0$ (출력 $Y_0$ 유지)
2. **T1 (Setup Time 前)**: 입력 $X$가 변화해도 $Y$는 변화 없음.
3. **T2 (Clock Edge)**: 상태 메모리가 갱신됨 ($S_0 \rightarrow S_1$).
4. **T3 (Clock Edge 후)**: $S_1$에 해당하는 새로운 출력 $Y_1$이 나타남.

```text
   Clock  ┐   ┌   ┐   ┐   ┐   ┐   ┐   ┐
          │   │   │   │   │   │   │   │
          └───┘   └───┘   └───┘   └───┘
               ▲       ▲       ▲
   Input  ─────┴───────┴───────┴──── (Noise/변화 발생)

   State  [S0]      [S1]      [S2]
            │         │         │
            ▼         ▼         ▼
   Output [Y0]      [Y1]      [Y2]  <-- 입력 변화에 관계없이 클럭에 동기됨
           ^          ^         ^
           |          |         |
           (입력 변화가 있어도 상태가 바뀔 때까지 출력 유지)
```

**[다이어그램 해설]**
위 타이밍도를 보면 입력 신호가 클럭 사이사이에 언제든지 변화할 수 있음에도, 출력 신호는 클럭의 상승 에지(Rising Edge) 이후에만 계단식으로 변한다. 이러한 특성은 **비동기 입력에 대한 동기화(Synchronization)** 역할을 자연스럽게 수행하며, 이로 인해 Metastability(준안정 상태)가 발생하더라도 그 영향이 다음 사이클로 한정되어 시스템이 무너지는 것을 방지한다.

> 📢 **섹션 요약 비유**: 입력과 출력 사이에 '플립플롭'이라는 무역창고를 두어, 언제 들어오든 상관없이 정기 배송 시간에만 물건을 내보내는 철저한 관세청 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 무어 머신 vs 밀리 머신 (Mealy Machine)
FSM 설계 시 가장 빈번하게 비교되는 두 모델을 정량적, 정성적으로 분석한다.

| 비교 항목 | 무어 머신 (Moore Machine) | 밀리 머신 (Mealy Machine) |
|:---|:---|:---|
| **출력 결정 함수** | $Y = f(State)$ | $Y = f(State, Input)$ |
| **반응 속도 (Latency)** | **느림 (1 클럭 지연)** | 빠름 (입력 즉시 반응) |
| **타이밍 안정성** | **매우 높음 (Glitch Free)** | 상대적으로 낮음 (입력 노이즈 전파) |
| **하드웨어 복잡도** | 상태 수가 많음 (출력 표현을 위해 상태 분리 필요) | 상태 수가 적음 (입력 조건으로 출력 구별 가능) |
| **설계 및 디버깅** | 용이함 (상태 중심 사고) | 복잡함 (경쟁 조건(Race Condition) 고려 필요) |
| **최대 동작 주파수** | 출력 경로가 짧아져 Fmax 유리 | 입력→출력 경로가 길어 타이밍 제약 발생 가능 |

### 2. 과목 융합 관점 분석

#### A. 운영체제(OS) 및 컴퓨터 구조와의 연계
- **CPU의 제어 유닛(Control Unit)**: 현대 CPU의 명령어 사이클(Instruction Cycle) 설계는 기본적으로 무어 머신 구조를 따른다.
    - 예: `FETCH` 상태일 때 무조건 `PC (Program Counter)` 증가 신호가 켜지고, `DECODE` 상태일 때 무조건 `IR (Instruction Register)` 로딩 신호가 켜진다.
    - 이는 명령어 디코딩 결과가 나오기 전에 제어 신호가 왔다 갔다 하여 발생할 수 있는 데이터패스(Data Path)의 혼선을 방지한다.

#### B. 네트워크 프로토콜과의 연계
- **링크 계층의 프레이머(Framer)**:
    - 데이터 스트림을 비트 단위로 읽어 프레임 시작 플래그를 찾는 과정은 비동기 입력(비트열)을 처리하는 무어 머신으로 설계된다.
    - `IDLE` 상태에서 `FLAG` 상태로 넘어가는 순간, 이전 비트의 값과 상관없이 일정한 제어 신호를 내보내어 CRC(Cyclic Redundancy Check) 로직을 초기화한다.

#### C. 전자 회로 및 신호 처리
- **동기식 리셋(Synchronous Reset) vs 비동기식 리셋(Asynchronous Reset)**:
    - 무어 머신은 동기식 설계를 전제로 하므로, 리셋 신호가 클럭에 동기화되어 처리되어야 한다. 이는 고속 회로에서 전원 켜짐 시(Power-up) 발생하는 불안정 신호를 거르는 데 핵심적인 역할을 한다.

> 📢 **섹션 요약 비유**: 밀리 머신은 '민첩한 번역가'라면(입력을 듣자마자 말함), 무어 머신은 '대변인'과 같습니다. 무어 머신은 상황(상태)을 파악하고 공식적인 입장(출력)을 발표하므로, 말의 실수(글리치)가 사회적 시스템(회로)을 망가뜨리지 않습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오: 고속 UART (Universal Asynchronous Receiver-Transmitter) 수신부 설계

**문제 상황**:
- 115,200 Baud로 들어오는 시리얼 데이터를 처리하는 UART 설계. 외부 라인에서 노이즈가 섞여 들어오는 환경.
- 데이터가 0에서 1로 바뀌는 'Start Bit'를 감지하고 그 후 8비트 데이터를 샘플링해야 함.

**의사결정 과정**:
1. **문제 정의**: 입력 데이터는 비동기(Asynchronous)이며, 타이밍이 불규칙할 수 있음.
2. **기술 선택**:
    - Mealy Machine 사용 시: Start Bit 감지 즉시 데이터를 읽으려 시도할 수 있으나, 노이즈로 인해 Start Bit로 착각(글리치)하는 순간 바로 잘못된 데이터를 출력할 위험 있음.
    - **Moore Machine 사용**: 'IDLE' -> 'START_DETECTED' -> 'RECEIVING' -> 'DONE' 상태를 거치며, 반드시 한 클럭 이상 신호가 유지될 때만 상태를 천이시킴.
3. **결과**: Start Bit가 확정된 상태(START_DETECTED)에서만 샘플링 타이머를 가동하므로, 순간적인 노이즈에 면역(Immunity)을 가지는 안정적인 수신기 구현 가능.
