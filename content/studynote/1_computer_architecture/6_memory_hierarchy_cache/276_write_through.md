> **핵심 인사이트**
> 1. Write-Through(동시 쓰기)는 CPU가 연산 결과를 캐시에 저장(Write)할 때, **그와 동시에 메인 메모리(DRAM)에도 똑같은 데이터를 덮어쓰는(Update) 가장 원초적이고 직관적인 캐시 쓰기 정책**이다.
> 2. 캐시와 메인 메모리의 데이터가 언제나 100% 동일하게 유지되므로, 시스템 오류가 발생하거나 멀티코어 환경에서 다른 장치가 메모리를 읽어갈 때 **데이터 일관성(Data Consistency)이 완벽하게 보장**된다.
> 3. 반면 데이터를 한 번 쓸 때마다 느려터진 메인 메모리에 강제로 다녀와야 하므로, 메모리 병목 현상(Memory Wall)을 악화시켜 **전체적인 CPU 쓰기 성능(Write Performance)을 치명적으로 떨어뜨리는 단점**이 있다.

---

## Ⅰ. Write-Through의 작동 메커니즘

CPU가 `A = 10`이라는 연산을 수행하여 변수 A의 값을 바꾼다고 가정해 봅시다.

1. **캐시 업데이트:** CPU는 L1 데이터 캐시(D-Cache)에 있는 변수 A의 값을 10으로 바꿉니다. (1~2클럭 소요)
2. **동시 메모리 업데이트:** 이와 완전히 동시에, 메모리 컨트롤러는 메인보드의 데이터 버스를 타고 저 멀리 있는 메인 메모리(DRAM)까지 달려가서 그곳의 A 값도 10으로 바꿉니다. (수백 클럭 소요)
3. **대기 (Stall):** CPU는 메인 메모리의 업데이트가 완벽히 끝났다는 신호(Acknowledge)를 받을 때까지 다음 명령어를 실행하지 못하고 파이프라인을 멈춘 채 기다립니다.

📢 **섹션 요약 비유:**
수업 시간에 필기를 할 때, 내 공책(캐시)에 한 글자를 적을 때마다 벌떡 일어나서 교실 뒤편에 있는 칠판(메인 메모리)에도 똑같이 한 글자를 적고 돌아와야 하는 아주 피곤한 규칙입니다.

---

## Ⅱ. 완벽한 일관성(Consistency)이 주는 안정성

이런 피곤한 짓을 왜 하는 걸까요? 바로 **'데이터 일관성'** 때문입니다.

- **DMA(Direct Memory Access)와의 충돌 방지:** 컴퓨터 안에는 CPU 말고도 메모리를 직접 읽고 쓰는 장치(네트워크 카드, 그래픽 카드 등)들이 많습니다. 이 장치들은 CPU의 캐시를 거치지 않고 메인 메모리를 직접 읽어갑니다. 만약 캐시만 업데이트되어 있고 메인 메모리가 과거의 쓰레기 데이터(Stale Data)를 들고 있다면, 그래픽 카드는 깨진 화면을 모니터에 뿌리게 됩니다.
- **오류 복구의 편리함:** 갑자기 정전이 되어 캐시(SRAM)의 데이터가 싹 날아가더라도, 메인 메모리에는 항상 최신 데이터가 백업되어 있으므로 데이터 유실 위험이 없습니다.

📢 **섹션 요약 비유:**
은행의 통장 잔고입니다. 지점장(CPU)이 고객의 잔고를 수정하자마자 무조건 본사 서버(메인 메모리)를 실시간으로 똑같이 고치기 때문에, 다른 지점에서 그 고객의 통장을 조회해도 절대 사고가 나지 않습니다.

---

## Ⅲ. Write Buffer를 통한 성능 저하의 극복

Write-Through의 치명적인 속도 저하를 막기 위해, 설계자들은 캐시와 메인 메모리 사이에 **쓰기 버퍼(Write Buffer)**라는 작은 징검다리를 하나 더 놓았습니다.

- **동작 원리:** CPU가 데이터를 캐시에 씀과 동시에, 메인 메모리까지 달려가지 않고 이 '쓰기 버퍼'라는 상자에 수정할 데이터들을 툭 던져놓고 곧바로 다음 명령어를 실행하러 돌아갑니다.
- **백그라운드 처리:** 쓰기 버퍼에 모인 데이터들은 CPU가 다른 일을 하는 동안, 메모리 컨트롤러가 백그라운드에서 천천히 알아서 메인 메모리로 운반해 줍니다.
- **한계점:** CPU가 너무 빠른 속도로 데이터를 계속 수정해서 쓰기 버퍼의 용량이 꽉 차게 되면(Buffer Full), 결국 CPU는 버퍼가 비워질 때까지 다시 멈춰 서서(Stall) 기다려야 합니다.

📢 **섹션 요약 비유:**
칠판에 직접 글씨를 적으러 가는 대신, 조수(Write Buffer)에게 "이거 칠판에 적어놔!"라고 쪽지를 던져주고 나는 내 할 일을 계속하는 것입니다. 단, 조수가 처리하는 속도보다 내가 쪽지를 던지는 속도가 빠르면 결국 일이 막히게 됩니다.

---

## Ⅳ. Write-Through의 현대적 위치

과거 단일 코어 시절이나 시스템 구조가 단순했던 임베디드 기기에서는 이 방식이 널리 쓰였습니다. 하지만 현대의 고성능 PC나 서버에서는 거의 쓰이지 않습니다.

- **이유:** 메모리 속도가 CPU를 따라가지 못하는 격차가 너무 심해져서, 아무리 쓰기 버퍼를 써도 병목을 막을 수 없기 때문입니다.
- **현재의 용도:** 아주 특수한 경우, 예를 들어 그래픽 메모리(VRAM)의 프레임 버퍼를 갱신할 때처럼 **"캐시에 남겨둘 필요 없이 무조건 메모리에 빨리 써서 화면에 띄워야 하는"** 데이터(Uncacheable Data)를 다룰 때 제한적으로 사용됩니다.

📢 **섹션 요약 비유:**
자전거를 타던 시절에는 우체국에 매번 들러도(Write-Through) 괜찮았지만, KTX를 타고 날아다니는 지금은 매번 우체국에 들렀다가는 스케줄이 완전히 망가지기 때문에 더 이상 쓰지 않는 구식 배달 방식입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Write Policy]`: Write-Through와 Write-Back으로 양분되는 캐시 데이터 갱신 규칙
- `[Write-Back]`: Write-Through의 속도 문제를 해결하기 위해 고안된 정반대의 캐시 정책
- `[Write Buffer]`: Write-Through의 끔찍한 지연 시간(Stall)을 숨기기 위해 도입된 하드웨어 큐
- `[Data Consistency (일관성)]`: Write-Through가 성능을 포기하면서까지 지켜내려 했던 절대 가치
- `[DMA (Direct Memory Access)]`: 캐시를 우회하여 메모리에 직접 접근하므로 일관성 문제를 터뜨리는 주범

---

## 👶 어린이를 위한 3줄 비유 설명
1. 동시 쓰기(Write-Through)는 일기장(캐시)에 글을 쓸 때, 복사지(먹지)를 대고 아래에 있는 진짜 공책(메모리)에도 동시에 똑같이 글이 써지게 하는 방법이에요.
2. 둘 다 똑같은 내용이 적혀 있으니까 나중에 헷갈릴 일이 없어서 아주 안전해요.
3. 하지만 글씨를 쓸 때마다 꾹꾹 눌러써야 해서 팔이 아프고 글씨 쓰는 속도가 엄청 느려진다는 단점이 있어요!
