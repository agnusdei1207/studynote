# Write-Through (연속 기록)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Write-Through(연속 기록)는 CPU가 캐시에 데이터를 기록할 때, **그 변경된 내용을 즉시 하위 메모리 계층(메인 메모리)에도 동시에 업데이트**하는 가장 정직하고 직관적인 캐시 쓰기 정책이다.
> 2. **가치**: 캐시와 메인 메모리의 데이터가 항상 100% 동일하게 유지되므로, 시스템이 갑자기 꺼지거나 다른 장치(DMA)가 메모리를 읽더라도 **절대적인 데이터 일관성(Data Consistency)과 무결성을 보장**한다.
> 3. **융합**: 매번 느린 메모리에 접근해야 하는 치명적인 속도 저하를 막기 위해, 하드웨어적으로 **쓰기 버퍼(Write Buffer)**와 결합하고 소프트웨어적으로는 **No-Write Allocate** 정책과 융합하여 단점을 완벽하게 우회한다.

---

## Ⅰ. Write-Through의 개요

### 1. 정의
- 캐시 메모리에 Write(쓰기) 연산이 발생할 때마다 캐시의 해당 블록을 갱신함과 동시에, 메인 메모리의 원본 주소에도 그 값을 즉각적으로 전송하여 덮어쓰는 방식이다.

### 2. 등장 배경: '거짓말을 허용하지 않는 시스템'
- 캐시가 도입된 초기, 가장 큰 공포는 "캐시에만 새 값이 있고 메모리에는 옛날 값이 있으면 컴퓨터가 미치지 않을까?"였다. 특히 입출력 장치가 메모리를 직접 긁어가는 DMA(Direct Memory Access) 환경에서는 메모리에 항상 최신값이 있어야 했다. 그래서 아키텍트들은 속도를 약간 희생하더라도 **"수정 사항이 생기면 무조건 실시간으로 동네방네 알리자"**는 가장 안전한 방어책을 1순위로 채택했다.

### 3. 💡 비유: '실시간 클라우드 동기화'
- Write-Through는 **'구글 닥스(Google Docs)나 노션에서 글자를 칠 때마다 1초 만에 클라우드 서버로 자동 저장되는 기능'**과 같다.
- 내 컴퓨터(캐시)에서 글자를 치면 동시에 서버(메인 메모리)에도 똑같이 타이핑된다. 갑자기 정전이 돼서 내 컴퓨터가 꺼져도 서버에는 방금 친 글자가 100% 안전하게 남아있다.

- **📢 섹션 요약 비유**: 장부에 잉크가 마르기도 전에 사장님 책상에 복사본을 팩스로 밀어 넣는, 융통성은 없지만 신뢰도 200%의 투명한 보고 체계입니다.

## Ⅱ. 메커니즘 및 하드웨어 동작 (Deep Dive)

안전함의 대가는 가혹한 메모리 버스 트래픽이다.

### 1. 명령어 흐름과 병목의 시각화 (ASCII)
```text
    [ CPU가 변수 A에 10을 쓰는 명령어 실행 (STORE A, 10) ]
    
        CPU ────────(1클럭)───────▶ [ L1 Cache ] (A = 10 갱신 완료!)
         │
         │  (하지만 CPU는 여기서 멈춰 서서 다음 작업을 못 함)
         ▼
      (수백 클럭) ─────────────────▶ [ Main Memory ] (A = 10 갱신 완료!)
      
    * 문제: CPU는 메모리에 쓰기가 완전히 끝날 때까지 100클럭을 제자리에 서서 기다려야(Stall) 함.
```

### 2. 심층 분석: Dirty Bit의 소멸
- Write-Back 정책에서는 캐시에 있는 데이터가 메모리와 다를 수 있어서 "나 메모리랑 달라요!"라고 외치는 표시표, 즉 **더티 비트(Dirty Bit)**가 필요했다. 하지만 Write-Through 환경에서는 캐시의 값이 무조건 메모리의 값과 똑같다. 따라서 캐시 라인이 꽉 차서 데이터를 쫓아낼 때(Eviction), 메모리에 다시 써줄 필요 없이 그냥 미련 없이 지워버리면(Drop) 끝이다. 하드웨어 관리가 극도로 깔끔해진다.

### 3. 짝꿍 정책: No-Write Allocate (쓰기 비할당)
- 데이터를 쓰려고 캐시를 봤더니 데이터가 없다(Write Miss). 어차피 Write-Through는 메모리에 무조건 쓰는 방식이다. 굳이 메모리에서 캐시로 블록을 낑낑대며 퍼오고 나서 메모리에 또 쓸 필요가 있을까? 그래서 "캐시에 없으면 걍 메모리에 바로 쓰고, 캐시에는 올리지 말자!"라는 **No-Write Allocate** 정책과 영혼의 단짝을 이룬다.

- **📢 섹션 요약 비유**: 어차피 본사에 직접 가서 결재를 받아야 하는 서류라면, 굳이 내 책상 서랍(캐시)을 거치지 않고 바로 본사(메모리) 우편함에 던져버리고 오는 게 이득인 논리입니다.

## Ⅲ. Write-Through의 한계 극복: 쓰기 버퍼 (Write Buffer)

메모리가 느리다고 CPU가 같이 느려지는 꼴을 아키텍트들은 참지 못했다.

### 1. 쓰기 버퍼(Write Buffer)의 마법
- CPU와 메인 메모리 사이에 아주 작은 크기(예: 4~8 엔트리)의 고속 FIFO 큐를 하나 달아놓는다.
- CPU는 캐시에 데이터를 씀과 동시에, 메모리까지 가지 않고 이 **쓰기 버퍼에 데이터를 휙 던져버리고 뒤도 안 돌아보고 다음 명령어를 실행하러 떠난다.**
- 쓰기 버퍼는 CPU가 떠난 뒤 백그라운드에서 낑낑대며 느린 메인 메모리에 데이터를 차곡차곡 옮겨 적는다.

### 2. 쓰기 버퍼가 꽉 찼을 때의 절망 (Buffer Full)
- 루프 안에서 연속으로 1,000번의 쓰기가 발생하면 4칸짜리 쓰기 버퍼는 1나노초 만에 꽉 차버린다. 이때부터는 버퍼가 비워질 때까지 CPU는 하염없이 멈춰서(Write Stall) 기다려야 한다. 이를 막기 위해 컴파일러는 코드 사이에 교묘하게 쓰기 없는 연산을 끼워 넣어 버퍼가 숨 쉴 시간을 벌어주는 스케줄링을 한다.

- **📢 섹션 요약 비유**: 우체국 수거함(쓰기 버퍼)에 편지를 휙 던져놓고 나는 바로 퇴근하면 되지만, 연말연시라 수거함이 꽉 차서 편지가 안 들어가면 꼼짝없이 수거함 옆에 서서 우체부 아저씨를 기다려야 하는 상황입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: 멀티코어 환경의 캐시 일관성
- **상황**: 멀티코어 CPU에서 코어 1과 코어 2가 동일한 변수 X를 쓴다.
- **판단**: 각 코어의 L1 캐시가 Write-Back을 쓰면 서로 값이 달라서 난장판이 된다. 하지만 **Write-Through**를 쓰면, 코어 1이 X를 바꿀 때마다 그 내용이 즉시 공용 버스를 타고 메모리로 향한다. 코어 2는 버스를 가만히 지켜보다가(Snooping) "어? X가 바뀌어 지나가네?" 하고 즉시 자신의 캐시에 있는 옛날 X를 무효화(Invalidation)시킬 수 있다. 이렇게 멀티코어 프로그래밍의 가장 큰 난제인 **캐시 일관성(Cache Coherence)**을 매우 단순하게 지켜낼 수 있기 때문에, 최신 CPU의 최상단 L1 캐시에서는 여전히 Write-Through 기반의 파생 정책들이 맹활약 중이다.

### 2. 실무 시나리오: 비디오 메모리 (VRAM) 제어
- **인사이트**: 운영체제 프로그래머가 프레임 버퍼(모니터에 띄울 화면 데이터)를 조작할 때, 캐시에만 그림을 그려놓고 메모리로 안 보내면(Write-Back) 화면은 계속 검은색으로 남아있다. 화면에 그림을 즉시 띄우려면 쓰는 즉시 디스플레이 메모리로 전송되어야 한다. 이 때문에 OS는 페이지 테이블의 속성을 조작하여 특정 메모리 영역(MMIO)을 **강제로 Write-Through 또는 Uncacheable 영역**으로 설정하는 아키텍처 레벨의 하드웨어 통제를 수행한다.

- **📢 섹션 요약 비유**: 나 혼자 쓰는 일기장은 나중에 몰아서 써도 되지만, 전광판에 띄울 공지사항은 쓰는 즉시 동기화(Write-Through) 버튼을 눌러야 세상 사람들이 볼 수 있습니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **복구 시간 제로 (Zero RTO)**: 시스템에 크래시가 나더라도 모든 데이터가 이미 램에 있기 때문에, 재부팅 시 데이터 손실률 0% 달성.
- **아키텍처의 직관성**: 더티 비트를 추적하고 쫓겨날 때 메모리에 쓰는 더러운(Dirty) 후처리 로직 없이 파이프라인의 맑은 흐름을 유지.

### 2. 미래 전망
거대한 L3 캐시(수십 MB) 전체를 Write-Through로 운영하면 메모리 버스가 터져서 시스템이 마비된다. 따라서 하위 레벨(L2, L3) 캐시는 무조건 Write-Back을 쓴다. 하지만 최근 CPU 내부에 1급 보안이 필요한 하드웨어 엔클레이브(Enclave)나 암호화 코어들이 생기면서, 해커가 전원을 뽑는 콜드 부트 공격(Cold Boot Attack)을 방어하기 위해 핵심 보안 키를 쓰는 즉시 암호화하여 메모리에 박아버리는 극단적인 보안 솔루션으로 Write-Through의 투명성이 재평가받고 있다.

- **📢 섹션 요약 비유**: 트래픽 낭비라는 죄목으로 왕좌에서 쫓겨났지만, 절대로 속이지 않는다는 그 고지식함 덕분에 가장 중요한 보안 요직에 다시 발탁된 셈입니다.
