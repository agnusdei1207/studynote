# 캐시 쓰기 정책 (Write Policy)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 캐시 쓰기 정책(Write Policy)은 CPU가 캐시 메모리에 있는 데이터를 수정(Write)했을 때, **그 변경된 내용을 하위 계층인 메인 메모리(RAM)에 언제, 어떻게 동기화할 것인지 결정하는 절대적인 규칙**이다.
> 2. **가치**: 캐시와 메인 메모리 간의 **데이터 불일치(Inconsistency)** 문제를 해결하는 핵심 메커니즘이며, 버스 트래픽(Bus Traffic)과 시스템 신뢰성 사이의 거대한 트레이드오프(Trade-off)를 조율한다.
> 3. **융합**: 즉시 기록하는 **Write-Through**와 나중에 몰아서 기록하는 **Write-Back**으로 나뉘며, 쓰기 미스(Write Miss)가 발생했을 때 캐시로 블록을 가져올지 말지를 결정하는 할당 정책(Write Allocate / No-Write Allocate)과 완벽하게 짝을 이룬다.

---

## Ⅰ. 캐시 쓰기 정책의 개요

### 1. 정의
- 캐시에 적재된 데이터 블록이 프로세서의 STORE 명령어 등에 의해 수정되었을 때, 이 변경 사항을 백킹 스토어(Main Memory)에 반영하는 시점과 방식을 정의한 하드웨어 설계 정책이다.

### 2. 등장 배경: '두 개의 장부, 하나의 진실'
- 캐시가 없던 시절에는 CPU가 메모리에 직접 썼으므로 데이터는 항상 100% 진실(최신 상태)이었다. 하지만 중간에 캐시라는 '임시 메모장'이 생기면서 문제가 터졌다. CPU가 캐시에만 쓱싹쓱싹 값을 고쳐 적고 메모리에는 알려주지 않으면, DMA(Direct Memory Access) 장치나 다른 코어가 메모리를 읽었을 때 **'과거의 쓰레기 값'**을 읽게 되는 치명적인 데이터 무결성 파괴 현상이 발생했다. 이를 막기 위한 하드웨어적인 보고 체계가 필요했다.

### 3. 💡 비유: '가계부와 통장 정리'
- 캐시 쓰기 정책은 **'집에서 쓰는 수기 가계부(캐시)와 실제 은행 통장(메인 메모리)의 잔고를 맞추는 방식'**과 같다.
- 돈을 쓸 때마다 매번 은행에 뛰어가서 통장 정리를 할 것인가(Write-Through)? 아니면 일단 가계부에만 연필로 쓱쓱 적어두고, 월말에 한 번만 은행에 가서 통장 정리를 몰아서 할 것인가(Write-Back)? 

- **📢 섹션 요약 비유**: 빠르지만 거짓말을 할 수 있는 비서(캐시)와, 느리지만 절대 거짓말을 하지 않는 금고(메인 메모리) 사이의 아슬아슬한 비밀 공유 규칙입니다.

---

## Ⅱ. 핵심 메커니즘: 쓰기 미스(Write Miss) 시의 할당 정책

데이터를 수정하려는데 캐시에 그 데이터가 아예 없다면(Write Miss) 어떻게 할 것인가?

### 1. Write Allocate (쓰기 할당)
- **동작**: 메모리에서 해당 데이터 블록을 캐시로 **일단 무조건 끌고 올라온 다음**, 캐시 위에서 데이터를 수정한다.
- **특징**: 이후에 해당 데이터가 다시 쓰일 확률이 높을 때(시간적 지역성) 유리하다. 주로 **Write-Back** 정책과 짝꿍을 이룬다.

### 2. No-Write Allocate (쓰기 비할당)
- **동작**: 캐시를 귀찮게 건드리지 않고, **메인 메모리로 바로 직행해서 그곳에만 데이터를 쓱 쓰고 만다**. (캐시에는 여전히 안 올라옴)
- **특징**: 한 번 쓰고 다시는 안 볼 데이터(스트리밍)에 유리하며, 캐시 공간을 절약한다. 주로 **Write-Through** 정책과 짝꿍을 이룬다.

### 3. 조합에 따른 파이프라인의 흐름 (ASCII)
```text
    [ CPU Write Request ]
          │
          ▼
    [ Cache 검사 ] ──(Miss!)──┐
          │                   ▼
        (Hit!)        [ Write Allocate? ]
          │           │                 │
          │         (Yes)              (No)
          ▼           ▼                 ▼
     [ 캐시 수정 ] ◀─ (메모리에서 로드)   [ 메모리에 바로 수정 ]
```

- **📢 섹션 요약 비유**: 서류를 고쳐야 하는데 내 책상(캐시)에 서류가 없으면, 굳이 캐비닛에서 내 책상으로 가져와서 고칠지(Allocate), 아니면 그냥 캐비닛에 서서 고치고 휙 돌아설지(No-Allocate)의 차이입니다.

## Ⅲ. 양대 쓰기 정책의 비교 (Write-Through vs Write-Back)

| 비교 항목 | Write-Through (연속 기록) | Write-Back (나중 기록) |
|:---|:---|:---|
| **업데이트 시점** | 캐시를 수정할 때 **항상 메모리도 같이 수정** | 캐시 블록이 쫓겨날 때(Eviction) **한 번만 수정** |
| **메모리 트래픽** | **매우 높음** (수정할 때마다 버스 점유) | **매우 낮음** (버스를 아낌) |
| **데이터 일관성** | **항상 100% 일치 (안전함)** | 일시적으로 불일치 발생 (위험함) |
| **Dirty Bit 필요**| 불필요 (항상 메모리와 같으니까) | **필수 (수정되었는지 표시해야 하므로)** |
| **복구 용이성** | 정전 시에도 데이터 안전 | 정전 시 캐시에만 있던 데이터 영구 증발 |

- **📢 섹션 요약 비유**: 안전을 위해 매 순간 클라우드에 자동 동기화를 켜두는 것(Through)과, 데이터 요금을 아끼기 위해 와이파이가 연결될 때만 수동으로 백업하는 것(Back)의 차이입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: Write Buffer의 도입 (Write-Through의 부활)
- **상황**: 초기 시스템에서는 안전을 위해 Write-Through를 썼지만, 매번 느려터진 메모리에 쓰느라 CPU가 멈추는 쓰기 스톨(Write Stall)이 발생해 성능이 박살 났다.
- **판단**: 엔지니어들은 CPU와 메인 메모리 사이에 **쓰기 버퍼 (Write Buffer)**라는 아주 작은 FIFO 큐를 달았다. CPU는 캐시와 쓰기 버퍼에 데이터를 던져넣고 "난 다 썼다!"고 착각한 채 자기 할 일을 하러(Execute) 떠난다. 그러면 쓰기 버퍼가 알아서 백그라운드에서 천천히 메모리로 데이터를 밀어 넣는다. 이 영리한 비동기적 완충 장치 덕분에 무식해 보였던 Write-Through도 최신 프로세서 L1 캐시 레벨에서 극강의 위력을 발휘하게 되었다.

### 2. 실무 시나리오: 다중 계층 캐시에서의 믹스 앤 매치
- **인사이트**: 현대의 멀티코어 CPU는 정책을 한 가지만 쓰지 않는다.
  - **L1 캐시**: Write-Through를 쓴다. (어차피 바로 뒤에 L2가 버티고 있어서 버스 트래픽 부담이 적고, 코어 간 일관성 유지가 편하기 때문)
  - **L3 캐시 (LLC)**: 절대적인 **Write-Back**을 쓴다. L3에서 메모리로 내려가는 트래픽은 시스템 전체의 목줄(DRAM 핀 대역폭)을 쥐고 있기 때문에, 메모리 쓰기를 극한으로 미루고 합쳐서(Merge) 한 방에 처리해야만 시스템이 살아남는다. 

- **📢 섹션 요약 비유**: 동네 지점(L1)에서는 본부(L2)로 실시간 보고를 올리지만, 총괄 본부장(L3)은 사장님(메모리)에게 귀찮게 하지 않고 월말에 결산 보고서 한 장만 딱 올리는 완벽한 계급 사회입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **트래픽 병목 해소**: Write-Back을 통해 동일한 변수에 1,000번 반복해서 쓰는 루프의 메모리 쓰기 횟수를 단 1회로 압축.
- **멀티코어 기반 마련**: 캐시 일관성(Coherence) 프로토콜들이 이 쓰기 정책의 Dirty 상태를 기준으로 동작하므로 SMP(대칭형 다중 프로세서) 발전의 초석이 됨.

### 2. 미래 전망
최근 NVDIMM이나 인텔 옵테인 같은 비휘발성 메모리(Non-Volatile Memory)가 메인 메모리로 들어오면서 쓰기 정책에 일대 혁명이 일어났다. 비휘발성 메모리는 전원이 꺼져도 데이터가 남지만 '쓰기 횟수'에 수명이 있고 쓰기 속도가 느리다. 따라서 아키텍트들은 캐시에서 메모리로 내려가는 쓰기를 더욱 극단적으로 줄이고 최적화하는 지능형 **NVM 전용 Write-Back 정책**과 쓰기 결합(Write Combining) 기술을 필사적으로 연구하고 있다.

- **📢 섹션 요약 비유**: 예전에는 종이(RAM)에 글을 막 썼다 지웠다 했지만, 이제는 비싼 대리석(NVM)에 조각을 해야 하니 캐시에서 완벽하게 퇴고를 끝낸 뒤 한 번만 칼을 대는 신중함이 요구됩니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 캐시 쓰기 정책은 공책(캐시)에 필기하다가 틀렸을 때, **'언제 교과서(메모리)의 원본 내용도 같이 고칠지 정하는 약속'**이에요.
2. 공책을 고칠 때마다 교과서도 매번 지우개로 지우고 같이 고치면 엄청 귀찮지만 절대 헷갈리지 않죠(Write-Through).
3. 반대로 일단 공책에만 잔뜩 고쳐두고, 학기 말에 공책을 버리기 직전에 한 번만 교과서에 옮겨 적으면 시간은 아끼지만 중간에 엄마가 교과서를 보면 깜짝 놀란답니다(Write-Back)!
