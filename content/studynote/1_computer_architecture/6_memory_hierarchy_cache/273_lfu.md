# 273. LFU (Least Frequently Used)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사용 빈도가 가장 낮은 항목 교체
> 2. **가치**: 장기적 인기도 반영, 안정적 성능
> 3. **융합**: 카운터, Aging, LFU-K, 하이브리드

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

LFU(Least Frequently Used)는 **사용 빈도가 가장 낮은 캐시 라인을 교체하는 알고리즘**이다. 과거의 접근 빈도를 기반으로 하며, 장기적으로 인기 있는 데이터를 캐시에 유지한다.

### 💡 비유: 도서관 인기 도서

LFU는 **도서관 인기 도서 선정**과 같다. 대출 횟수가 많은 책은 인기 책 코너에 둔다. 새 책은 대출 기록이 없어서 인기 코너에 못 들어간다. 오래된 책도 대출 많으면 계속 인기 코너에 있다.

### LFU 개요

```
┌─────────────────────────────────────────────────────────────────────┐
│                LFU 개요                                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【LFU 원리】                                                                  │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  가정: 자주 사용된 것은 계속 사용될 확률이 높다                           │ │
│  │                                                               │ │
│  │  동작:                                                            │ │
│  │  1. 각 항목별로 접근 횟수 카운터 유지                                   │ │
│  │  2. 접근할 때마다 해당 항목 카운터 증가                                  │ │
│  │  3. 교체 필요 시 카운터가 가장 낮은 항목 선택                              │ │
│  │  4. 선택된 항목 교체                                                │ │
│  │                                                               │ │
│  │  빈도 기반: 인기도가 높은 데이터 유지                                     │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【LFU 동작 예시】                                                              │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  4-way 캐시, 접근 순서: A A A B B C D E E E E E E                   │ │
│  │                                                               │ │
│  │  접근   캐시 상태 [데이터:카운터]        Hit/Miss   교체               │ │
│  │  ────   ──────────────────────        ────────    ────               │ │
│  │  A     [A:1 - - -]                   Miss        -                   │ │
│  │  A     [A:2 - - -]                   Hit         -                   │ │
│  │  A     [A:3 - - -]                   Hit         -                   │ │
│  │  B     [A:3 B:1 - -]                 Miss        -                   │ │
│  │  B     [A:3 B:2 - -]                 Hit         -                   │ │
│  │  C     [A:3 B:2 C:1 -]               Miss        -                   │ │
│  │  D     [A:3 B:2 C:1 D:1]             Miss        -                   │ │
│  │  E     [A:3 B:2 E:1 D:1]             Miss        C (min=1)           │ │
│  │  E     [A:3 B:2 E:2 D:1]             Hit         -                   │ │
│  │  E     [A:3 B:2 E:3 D:1]             Hit         -                   │ │
│  │  E     [A:3 B:2 E:4 D:1]             Hit         -                   │ │
│  │  E     [A:3 B:2 E:5 D:1]             Hit         -                   │ │
│  │  E     [A:3 B:2 E:6 D:1]             Hit         -                   │ │
│  │                                                               │ │
│  │  적중률: 6/13 = 46%                                                 │ │
│  │  → 자주 접근하는 A, E 유지                                            │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【LFU 특징】                                                                   │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  장점:                                                               │
│  │  • 장기적 인기도 반영                                                 │ │
│  │  • 스캔 패턴에 강함 (새 데이터가 바로 교체되지 않음)                        │ │
│  │  • 안정적인 성능                                                    │ │
│  │  • 악의적 패턴에 강함                                                 │ │
│  │                                                               │ │
│  │  단점:                                                               │ │
│  │  • 새 데이터에게 불리 (찬발적 캐시 문제)                                 │ │
│  │  • 과거 데이터가 영구히 유지될 수 있음                                   │ │
│  │  • 카운터 오버플로우 문제                                              │ │
│  │  • 구현 복잡도 높음                                                  │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. LFU 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                LFU 상세                                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【LFU 구현 방법】                                                               │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  1. 단순 카운터:                                                      │ │
│  │     • 각 라인에 접근 횟수 카운터 저장                                    │ │
│  │     • 카운터가 가장 낮은 것 교체                                        │ │
│  │     • 문제: 카운터 오버플로우, 구형 데이터 유지                             │ │
│  │                                                               │ │
│  │  2. 빈도 힙 (Frequency Heap):                                       │ │
│  │     • 최소 힙으로 빈도 관리                                            │ │
│  │     • O(log N) 갱신, O(1) 교체                                       │ │
│  │     • 복잡한 구현                                                    │ │
│  │                                                               │ │
│  │  3. 카운터 + Aging:                                                  │ │
│  │     • 주기적으로 모든 카운터 감소                                        │ │
│  │     • 오래된 데이터의 영향력 감소                                        │ │
│  │     • 시간과 빈도 모두 고려                                             │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【LFU 변형 알고리즘】                                                             │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. LFU with Aging:                                                 │ │
│  │     • 주기적 카운터 감소 (예: 2로 나눔)                                   │ │
│  │     • 과거 인기 데이터도 시간이 지나면 감소                                │ │
│  │     • 새 데이터에게 기회 부여                                           │ │
│  │                                                               │ │
│  │  2. LFU-K:                                                           │ │
│  │     • 최근 K번 접근만 고려                                             │ │
│  │     • K=1: LRU와 유사                                                │ │
│  │     • K=∞: 기본 LFU                                                 │ │
│  │     • K=2: DB 버퍼에서 사용 (2Q)                                       │ │
│  │                                                               │ │
│  │  3. In-Memory LFU:                                                   │ │
│  │     • 빈도별 리스트 유지                                                │ │
│  │     • O(1) 갱신, O(1) 교체                                            │ │
│  │     • 메모리 오버헤드                                                  │ │
│  │                                                               │ │
│  │  4. TinyLFU:                                                         │ │
│  │     • 블룸 필터로 빈도 추정                                             │ │
│  │     • 메모리 효율적                                                   │ │
│  │     • Caffeine 캐시에서 사용                                          │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【LFU vs LRU 비교】                                                             │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  특성           LRU             LFU                                │ │
│  │  ──────          ────             ────                                │ │
│  │  기준           시간              빈도                                │ │
│  │  스캔 패턴        취약             강함                                │ │
│  │  새 데이터       유리              불리                                │ │
│  │  구현 복잡도      중간              높음                                │ │
│  │  메모리          적음             많음                                │ │
│  │  워킹셋 변화      빠른 적응          느린 적응                           │ │
│  │  안정성          낮음              높음                                │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### LFU 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                LFU 분석                                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【LFU 구현 예시】                                                              │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  // Java Caffeine 캐시 (TinyLFU)                                   │ │
│  │  Cache<K, V> cache = Caffeine.newBuilder()                        │ │
│  │      .maximumSize(1000)                                           │ │
│  │      .build();                                                    │ │
│  │                                                               │ │
│  │  // Python LFU 캐시                                                │ │
│  │  from functools import lru_cache  # LRU지만 데코레이터 예시          │ │
│  │                                                               │ │
│  │  // 커스텀 LFU                                                      │ │
│  │  class LFUCache:                                                   │ │
│  │      def __init__(self, capacity):                                 │ │
│  │          self.capacity = capacity                                  │ │
│  │          self.freq = {}  # key -> frequency                        │ │
│  │          self.cache = {}  # key -> value                           │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【LFU 문제점과 해결】                                                             │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  문제 1: 찬발적 캐시 (Cache Pollution)                                │ │
│  │  • 새로 인기진 데이터가 기존 인기 데이터에 밀림                             │ │
│  │  • 해결: Aging, LFU-K, TinyLFU                                      │ │
│  │                                                               │ │
│  │  문제 2: 카운터 오버플로우                                              │ │
│  │  • 카운터가 최대값에 도달                                              │ │
│  │  • 해결: 주기적 감소, 로그 스케일                                        │ │
│  │                                                               │ │
│  │  문제 3: 메모리 오버헤드                                               │ │
│  │  • 카운터 저장 공간 필요                                               │ │
│  │  • 해결: TinyLFU, 근사 카운팅                                          │ │
│  │                                                               │ │
│  │  문제 4: 구현 복잡도                                                  │ │
│  │  • 빈도 관리 로직 복잡                                                │ │
│  │  • 해결: 힙 구조, 계층적 LFU                                           │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【LFU 최적화 팁】                                                               │
│  ──────────────────                                                  │
│  1. Aging 사용: 과거 데이터 감쇠                                          │
│  2. LFU-K: K값 조정으로 LRU/LFU 균형                                     │
│  3. TinyLFU: 메모리 효율적 구현                                          │
│  4. 하이브리드: LRU + LFU 조합                                           │
│  5. 워크로드 분석: 스캔 패턴 있으면 LFU 유리                                 │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• LFU: 사용 빈도가 가장 낮은 항목 교체
• 기반: 빈도 기반 (인기도)
• 구현: 카운터, 힙, Aging
• 장점: 스캔 저항, 안정적 성능, 장기적 인기 반영
• 단점: 새 데이터 불리, 구현 복잡, 메모리 오버헤드
• 변형: LFU with Aging, LFU-K, TinyLFU
• 비교: LRU는 시간, LFU는 빈도 기준
• 사용처: DB 버퍼, 웹 캐시, CDN
• 하이브리드: LRU + LFU 조합
• 최적화: Aging, LFU-K, TinyLFU
```
