> **핵심 인사이트**
> 1. Split Cache(분리형 캐시)는 CPU에 가장 가까운 L1 캐시를 명령어 전용(I-Cache)과 데이터 전용(D-Cache)으로 **물리적으로 완벽히 쪼개어 독립적으로 운영하는 하버드 아키텍처(Harvard Architecture)의 구현체**이다.
> 2. 명령어 인출(Fetch) 단계와 메모리 접근(MEM) 단계가 같은 클럭 사이클에 동시에 캐시에 접근하려 할 때 발생하는 **구조적 해저드(Structural Hazard)를 원천적으로 제거하여 파이프라인 성능을 극대화**한다.
> 3. 분리형 캐시는 서로의 영역을 침범하지 않아 캐시 충돌이 줄어들지만, 용량의 유연성이 떨어지므로 그 아래 계층인 L2, L3 캐시는 주로 두 개를 합친 통합 캐시(Unified Cache) 형태로 설계된다.

---

## Ⅰ. 단일 캐시(Unified Cache)의 병목과 한계

폰 노이만 아키텍처의 철학대로 명령어와 데이터를 하나의 거대한 L1 캐시(Unified Cache)에 때려 넣었다고 가정해 봅시다.

- **파이프라인 충돌:**
  - 파이프라인의 1번 단계(IF)에서는 5번 명령어를 가져오기 위해 캐시를 읽어야 합니다.
  - 동시에 4번 단계(MEM)에서는 2번 명령어가 쓸 변수 데이터를 가져오기 위해 똑같은 캐시를 읽어야 합니다.
- **구조적 해저드 발생:** 캐시의 입구(포트)가 하나뿐이라면, 이 둘은 문 앞에서 충돌합니다. 결국 하나는 파이프라인을 멈추고(Stall) 뒤에서 기다려야 하는 치명적인 성능 저하가 일어납니다.

📢 **섹션 요약 비유:**
요리사 두 명이 동시에 한 대의 냉장고를 향해 달려가서, 한 명은 레시피(명령어)를 꺼내려 하고 한 명은 돼지고기(데이터)를 꺼내려다 문이 하나라 부딪혀서 싸우는 상황입니다.

---

## Ⅱ. Split Cache의 마법: 하버드 아키텍처의 도입

하드웨어 설계자들은 이 문제를 해결하기 위해 L1 캐시 한정으로 폰 노이만 구조를 버리고 **하버드 아키텍처**를 도입했습니다.

- **I-Cache (Instruction Cache):** 명령어만 담는 전용 캐시입니다. 파이프라인의 IF 단계와만 직통으로 연결되어 있습니다. 명령어는 보통 읽기만 하므로(Read-Only) 구조가 아주 단순하고 빠릅니다.
- **D-Cache (Data Cache):** 데이터만 담는 전용 캐시입니다. 파이프라인의 MEM 단계와만 직통으로 연결됩니다. 읽고 쓰기가 빈번하므로 Write-Back 정책과 더티 비트 회로가 복잡하게 얽혀 있습니다.
- **효과:** 입구가 2개로 완전히 나뉘었기 때문에, IF 단계와 MEM 단계가 한 클럭(동시)에 각각의 캐시에 접근해도 단 1번의 충돌도 일어나지 않습니다.

📢 **섹션 요약 비유:**
냉장고를 아예 2대 사서, 왼쪽 냉장고에는 레시피(I-Cache)만 넣고 오른쪽 냉장고에는 식재료(D-Cache)만 넣어두어 요리사 두 명이 1초의 기다림도 없이 각자 물건을 빼갈 수 있게 만든 쾌적한 주방입니다.

---

## Ⅲ. Split Cache의 유연성 부족 (단점)

캐시를 쪼개면 무조건 좋을 것 같지만, 치명적인 딜레마(Trade-off)가 있습니다. 바로 **공간 활용의 낭비**입니다.

- **상황:** 총 64KB의 L1 캐시를 I-Cache 32KB, D-Cache 32KB로 쪼갰습니다.
- 어떤 프로그램은 명령어 코드는 아주 짧은데(10KB) 방대한 양의 데이터(50KB)를 다루는 프로그램이 있습니다.
- **낭비 발생:** I-Cache는 22KB나 텅텅 비어있는데, D-Cache는 32KB를 꽉 채우고 터져나가서 계속 캐시 미스(Capacity Miss)를 발생시킵니다. 빈 공간을 빌려 쓸 수가 없기 때문입니다.

📢 **섹션 요약 비유:**
식당에 고기 전용 냉장고와 야채 전용 냉장고를 나눠 놨더니, 오늘따라 고기 주문만 미친 듯이 들어와서 고기 냉장고는 터질 것 같은데 야채 냉장고는 텅텅 비어있는 비효율적인 상황입니다.

---

## Ⅳ. 현대 아키텍처의 타협 (L1 Split, L2 Unified)

현대 CPU(인텔, AMD, ARM 등)는 Split Cache의 장점과 Unified Cache의 장점을 완벽하게 섞어서 씁니다.

1. **L1 캐시 (Split Cache):** 파이프라인과 직접 맞닿아 있어 '충돌 회피(속도)'가 1순위이므로, 무조건 I-Cache와 D-Cache로 엄격하게 쪼갭니다.
2. **L2 및 L3 캐시 (Unified Cache):** 여기는 IF와 MEM 유닛이 직접 접근하지 않으므로 구조적 해저드의 위험이 적습니다. 따라서 '용량의 유연성'을 위해 명령어와 데이터를 구분하지 않고 섞어 담는 하나의 거대한 풀(Pool)로 설계합니다.

📢 **섹션 요약 비유:**
요리사 바로 옆에 있는 작은 바구니(L1)는 헷갈리지 않게 재료용/레시피용으로 나누지만, 주방 뒤편에 있는 거대한 업소용 냉장고(L2, L3)는 빈 공간이 아까우니 무조건 손에 잡히는 대로 다 쑤셔 넣는 효율적인 방식입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Harvard Architecture]`: 메모리를 명령어와 데이터용으로 물리적으로 분리한 아키텍처 (L1 Split Cache의 철학적 배경)
- `[Von Neumann Architecture]`: 명령어와 데이터를 하나의 메모리에 담는 아키텍처 (L2/L3 Unified Cache의 철학적 배경)
- `[Structural Hazard]`: 파이프라인에서 L1 캐시를 쪼개지 않았을 때 발생하는 치명적 충돌 현상
- `[Instruction Pipelining]`: Split Cache가 존재해야만 제 속도를 낼 수 있는 CPU 병렬 처리 구조
- `[L1 Cache]`: Split Cache 구조가 적용되는 유일한 캐시 계층
