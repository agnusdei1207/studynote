# 292. TLB 미스 (TLB Miss)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: TLB에 변환 정보가 없는 상황
> 2. **가치**: Page Walk 발생, 성능 저하 원인
> 3. **융합**: Page Table Walk, 하드웨어/소프트웨어 처리

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

TLB 미스(TLB Miss)는 **TLB에서 가상-물리 주소 변환 정보를 찾지 못한 상황**이다. TLB 미스 발생 시 Page Table Walk가 필요하며, 이는 성능 저하를 유발한다.

### 💡 비유: 기억나지 않는 책

TLB 미스는 **기억나지 않는 책**과 같다. 손님이 "과학-물리-001"을 요청했다. 하지만 기억력(TLB)에 없다. 위치록(페이지 테이블)을 찾아봐야 한다. 위치록을 뒤지는 동안 손님은 기다려야 한다. 기다리는 시간이 미스 페널티다.

### TLB 미스 개요

```
┌─────────────────────────────────────────────────────────────────────┐
│                TLB 미스 개요                                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【TLB Hit vs Miss】                                                            │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  TLB Hit:                                                            │ │
│  │  ┌───────────────┐                                                │ │
│  │  │ VPN = 0x7FFF  │                                                │ │
│  │  └───────┬───────┘                                                │ │
│  │          │                                                         │ │
│  │          ▼                                                         │ │
│  │  ┌─────────────────────────────────────┐                          │ │
│  │  │ TLB: VPN 0x7FFF → PFN 0x1234 (Hit!) │                          │ │
│  │  └─────────────────────────────────────┘                          │ │
│  │          │                                                         │ │
│  │          ▼                                                         │ │
│  │  PFN = 0x1234 즉시 반환 (1~2 클럭)                                   │ │
│  │                                                               │ │
│  │  TLB Miss:                                                           │ │
│  │  ┌───────────────┐                                                │ │
│  │  │ VPN = 0x8000  │                                                │ │
│  │  └───────┬───────┘                                                │ │
│  │          │                                                         │ │
│  │          ▼                                                         │ │
│  │  ┌─────────────────────────────────────┐                          │ │
│  │  │ TLB: VPN 0x8000 → ??? (Miss!)       │                          │ │
│  │  └─────────────────────────────────────┘                          │ │
│  │          │                                                         │ │
│  │          ▼                                                         │ │
│  │  Page Table Walk 시작                                              │ │
│  │  ┌─────────────────────────────────────┐                          │ │
│  │  │ 메모리에서 페이지 테이블 조회           │                          │ │
│  │  │ VPN 0x8000 → PFN 0x5678             │                          │ │
│  │  └─────────────────────────────────────┘                          │ │
│  │          │                                                         │ │
│  │          ▼                                                         │ │
│  │  TLB 갱신 후 PFN 반환 (10~100 클럭)                                  │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【TLB 미스 원인】                                                              │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. Cold Miss (Compulsory):                                         │ │
│  │     • 처음 접근하는 페이지                                            │ │
│  │     • TLB에 아직 로드되지 않음                                        │ │
│  │                                                               │ │
│  │  2. Capacity Miss:                                                  │ │
│  │     • TLB 엔트리보다 많은 페이지 접근                                   │ │
│  │     • Working Set > TLB 크기                                        │ │
│  │                                                               │ │
│  │  3. Conflict Miss:                                                  │ │
│  │     • TLB 교체 정책으로 인한 미스                                      │ │
│  │     • 완전 연관에서는 발생 안 함                                        │ │
│  │                                                               │ │
│  │  4. Context Switch Miss:                                            │ │
│  │     • 프로세스 전환 시 TLB 플러시                                       │ │
│  │     • ASID 없으면 발생                                                │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. TLB 미스 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                TLB 미스 상세                                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Page Table Walk 과정】                                                        │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  4단계 페이지 테이블에서의 TLB 미스 처리:                                 │ │
│  │                                                               │ │
│  │  1. CR3에서 PML4 베이스 주소 읽기                                     │ │
│  │     └→ 메모리 접근 #1                                               │ │
│  │                                                               │ │
│  │  2. PML4[PML4_인덱스]에서 PDPT 주소 읽기                               │ │
│  │     └→ 메모리 접근 #2                                               │ │
│  │                                                               │ │
│  │  3. PDPT[PDPT_인덱스]에서 PD 주소 읽기                                 │ │
│  │     └→ 메모리 접근 #3                                               │ │
│  │                                                               │ │
│  │  4. PD[PD_인덱스]에서 PT 주소 읽기                                     │ │
│  │     └→ 메모리 접근 #4                                               │ │
│  │                                                               │ │
│  │  5. PT[PT_인덱스]에서 PFN 읽기                                        │ │
│  │     └→ 메모리 접근 #5                                               │ │
│  │                                                               │ │
│  │  6. TLB에 VPN→PFN 엔트리 추가                                        │ │
│  │                                                               │ │
│  │  → 총 4~5번의 메모리 접근 (캐시 히트하면 더 빠름)                          │ │
│  │  → 각 메모리 접근: 10~100 클럭                                        │ │
│  │  → TLB 미스 페널티: 40~500 클럭                                       │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【TLB 미스 처리 방식】                                                           │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  1. 하드웨어 Page Walk (Hardware-Managed):                            │ │
│  │     • MMU가 자동으로 페이지 테이블 순회                                  │ │
│  │     • CPU는 대기 또는 다른 작업                                         │ │
│  │     • x86, ARM에서 사용                                               │ │
│  │     • 장점: 빠름, OS 개입 없음                                         │ │
│  │     • 단점: 하드웨어 복잡도                                             │ │
│  │                                                               │ │
│  │  2. 소프트웨어 Page Walk (Software-Managed):                           │ │
│  │     • TLB 미스 시 CPU 예외 발생                                        │ │
│  │     • OS가 페이지 테이블 순회                                           │ │
│  │     • MIPS, 일부 RISC-V에서 사용                                       │ │
│  │     • 장점: 유연함, OS 제어                                             │ │
│  │     • 단점: 느림, OS 오버헤드                                           │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【TLB 미스 비용】                                                              │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  시나리오                 클럭 수         설명                         │ │
│  │  ──────                  ──────         ────                         │ │
│  │  TLB Hit                 1~2            즉시 변환                      │ │
│  │  TLB Miss + L1 캐시 Hit   5~10           Page Walk 빠름               │ │
│  │  TLB Miss + L2 캐시 Hit   20~40          Page Walk 보통               │ │
│  │  TLB Miss + 메모리 접근    100~500        Page Walk 느림               │ │
│  │  TLB Miss + Page Fault    100,000+       디스크 접근 필요              │ │
│  │                                                               │ │
│  │  → TLB 미스는 성능에 큰 영향                                           │ │
│  │  → 작은 미스율도 전체 성능 저하                                         │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. TLB 미스 × 타 과목 융합

### 과목 융합 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                TLB 미스 × 타 과목 융합                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【컴퓨터 구조】                                                      │
│  • 파이프라인 스톨: TLB 미스 시 파이프라인 정지                             │
│  • MMU: Page Walk 수행                                               │
│  • 캐시: Page Walk 중 캐시 히트 여부 중요                                │
│  • 성능: TLB 미스율이 CPI 직접 영향                                      │
│                                                                     │
│  【운영체제】                                                          │
│  • TLB Shootdown: 다른 코어 TLB 무효화                                 │
│  • 페이지 교체: TLB 무효화 필요                                         │
│  • 소프트웨어 TLB: OS가 직접 관리                                        │
│  • 성능: TLB 미스 핸들러 최적화                                          │
│                                                                     │
│  【성능】                                                              │
│  • Working Set: TLB 크기 초과 시 미스 급증                               │
│  • 지역성: 낮으면 TLB 미스 증가                                          │
│  • Huge Page: TLB 미스 크게 감소                                       │
│  • 프로파일링: TLB 미스 핫스팟 찾기                                       │
│                                                                     │
│  【병렬 처리】                                                          │
│  • TLB Shootdown: 멀티코어에서 IPI 필요                                 │
│  • TLB 일관성: 코어 간 TLB 동기화                                        │
│  • Lock: TLB 미스 핸들링 중 락                                          │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  TLB 미스율과 성능 영향:                                                              │ │
│  │  ──────────────────────                                              │ │
│  │  TLB 미스율    Page Walk 비용    전체 성능 영향                         │ │
│  │  ──────────    ────────────    ────────────                            │ │
│  │  0.1%          50 클럭         약 5% 느려짐                             │ │
│  │  1%            50 클럭         약 50% 느려짐                            │ │
│  │  5%            50 클럭         약 250% 느려짐                           │ │
│  │  10%           50 클럭         약 500% 느려짐                           │ │
│  │                                                               │ │
│  │  → TLB 미스율이 성능에 지수적 영향                                       │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### TLB 미스 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                TLB 미스 분석                                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【TLB 미스 측정】                                                              │)
│  ──────────────────                                                  │
│  $ perf stat -e dTLB-loads,dTLB-load-misses,dTLB-store-misses ./prog│
│  dTLB-loads: 100,000,000                                            │
│  dTLB-load-misses: 500,000                                          │
│  dTLB-store-misses: 100,000                                         │
│  TLB 미스율: (500,000 + 100,000) / 100,000,000 = 0.6%                │
│                                                                     │
│  # iTLB 미스                                                        │
│  $ perf stat -e iTLB-loads,iTLB-misses ./prog                       │
│  iTLB-loads: 50,000,000                                             │
│  iTLB-misses: 100,000                                               │
│  iTLB 미스율: 0.2%                                                    │
│                                                                     │
│  【TLB 미스 줄이기】                                                             │)
│  ──────────────────                                                  │
│  1. Huge Page 사용:                                                 │
│     $ echo 1024 > /proc/sys/vm/nr_hugepages                         │
│     → TLB 커버리지 512배 증가 (4KB → 2MB)                               │
│                                                                     │
│  2. Working Set 관리:                                               │
│     • 자주 쓰는 데이터를 연속된 페이지에 배치                               │
│     • TLB 크기(64~128 엔트리)에 맞게                                    │
│                                                                     │
│  3. 지역성 향상:                                                      │
│     • 같은 페이지 반복 접근                                            │
│     • 배열 순차 접근                                                  │
│                                                                     │
│  【TLB 미스 핫스팟 찾기】                                                         │)
│  ──────────────────                                                  │
│  $ perf record -e dTLB-load-misses ./prog                           │
│  $ perf report                                                      │
│  # 미스가 많이 발생하는 함수/코드 위치 확인                                  │
│                                                                     │
│  【TLB 미스 최적화 팁】                                                           │)
│  ──────────────────                                                  │
│  1. Huge Page: 대용량 데이터에 필수                                     │
│  2. 데이터 구조: 페이지 단위로 접근 패턴 고려                               │
│  3. 루프 최적화: 같은 페이지 집중                                         │
│  4. 메모리 정렬: 페이지 경계 고려                                         │
│  5. prefetch: TLB 프리페치 (일부 CPU)                                   │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• TLB 미스: TLB에서 변환 정보를 찾지 못함
• 원인: Cold, Capacity, Conflict, Context Switch
• 결과: Page Table Walk 발생 (10~100 클럭)
• 하드웨어 처리: MMU가 자동 수행 (x86, ARM)
• 소프트웨어 처리: OS가 수행 (MIPS, 일부 RISC-V)
• 비용: TLB Hit 1~2클럭 vs TLB Miss 10~100클럭
• 성능 영향: 작은 미스율도 큰 성능 저하
• 최적화: Huge Page, 지역성, Working Set 관리
• 측정: perf로 dTLB-misses, iTLB-misses
• 분석: 핫스팟 찾아서 최적화
```
