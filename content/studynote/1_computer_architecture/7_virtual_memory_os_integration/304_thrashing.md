# 스래싱 (Thrashing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스래싱(Thrashing)은 컴퓨터의 물리적 메모리(RAM)가 꽉 찬 상태에서 다수의 프로그램이 동시에 실행될 때, CPU가 실제 연산은 하지 못하고 **메모리와 하드디스크(Swap) 사이에서 데이터를 넣고 빼는(Paging) 일에만 100%의 시간을 낭비하는 시스템 붕괴 상태**다.
> 2. **가치**: 다중 프로그래밍(Multi-programming) 정도를 무조건 높인다고 성능이 오르지 않으며, 어느 임계점을 넘는 순간 성능이 0으로 곤두박질치는 **컴퓨터 공학의 가장 유명한 임계점(Threshold) 현상**을 보여준다.
> 3. **융합**: 이 재앙을 막기 위해 운영체제는 각 프로세스가 자주 쓰는 메모리 덩어리를 묶어서 보장해 주는 **워킹셋 모델(Working Set Model)**이나 **페이지 부재 빈도(PFF)** 조절 알고리즘을 하드웨어 관리 기술과 융합하여 사용한다.

---

## Ⅰ. 스래싱(Thrashing)의 개요

### 1. 정의
- 가상 메모리 시스템에서 프로세스들이 워킹셋(자주 참조하는 페이지의 집합)을 물리 메모리에 다 올리지 못한 채, 끊임없이 페이지 부재(Page Fault)를 발생시키며 디스크 I/O만 반복하느라 시스템의 스루풋(Throughput)이 극단적으로 저하되는 현상.

### 2. 등장 배경: '과유불급의 저주'
- 운영체제는 CPU가 노는 꼴을 못 본다. CPU 사용률이 떨어지면 "아, 다른 프로그램을 더 띄워도 되겠네?" 하고 새로운 프로세스들을 램에 우겨넣는다(다중 프로그래밍 정도 증가). 그런데 프로그램이 너무 많아져서 각자에게 램이 1MB밖에 안 돌아가자, A 프로그램이 1장(Page)을 읽으려고 B의 1장을 디스크로 쫓아내고, 1초 뒤 B가 그걸 다시 읽으려고 A를 쫓아내는 지옥의 회전목마가 시작되었다.

### 3. 💡 비유: '가방이 너무 작은 고등학생'
- 스래싱은 **'책가방(RAM)에는 책이 3권밖에 안 들어가는데, 오늘 들어야 할 수업(프로세스)이 10개인 고등학생'**과 같다.
- 1교시 수학을 듣고 사물함(디스크)에 가서 수학책을 빼고 영어책을 넣어온다. 2교시 영어를 듣고 다시 사물함으로 뛰어간다. 공부(CPU 연산)를 하는 시간보다, 무거운 책을 들고 복도와 사물함을 오가는(디스크 I/O) 데 하루 체력과 시간을 다 써버려서 결국 시험을 망치게 된다.

- **📢 섹션 요약 비유**: 효율을 높이겠다고 알바생을 너무 많이 고용했더니, 주방(메모리)이 비좁아져서 요리는 안 하고 서로 비키라고 어깨빵만 치다가 하루가 끝나는 난장판 식당입니다.

## Ⅱ. 메커니즘 및 성능 붕괴 그래프 (Deep Dive)

스래싱은 서서히 느려지는 것이 아니라 낭떠러지로 추락하듯 일어난다.

### 1. 다중 프로그래밍 정도(Degree of Multi-programming)와 CPU 이용률
- 프로그램 1~5개를 띄울 때: CPU는 쉴 틈 없이 일하며 사용률이 80~90%까지 예쁘게 올라간다.
- 프로그램 20개를 띄울 때 (임계점 돌파): 램 용량이 한계에 달해 **페이지 부재(Page Fault)**가 폭발한다.
- **붕괴의 악순환 (Vicious Cycle)**:
  1. 페이지 부재 발생 $\to$ 디스크에서 퍼오느라 CPU가 쉼.
  2. OS: "어? CPU가 노네? 일을 덜 줬나? 프로세스 하나 더 띄워!"
  3. 램이 더 모자라짐 $\to$ 페이지 부재 더 발생 $\to$ CPU 더 쉼.
  4. **최종 결과**: 디스크(HDD)만 윙윙거리며 100% 돌아가고, 정작 CPU 사용률은 1%로 바닥을 치며 마우스 커서조차 안 움직임.

### 2. 스래싱의 시각적 그래프 (ASCII)
```text
  CPU 사용률 (%)
    100 │           /|  <-- 임계점 (Threshold)
        │         /  │ 
     80 │       /    │   ╲  <-- 스래싱(Thrashing) 발생 구간!
        │     /      │     ╲    (CPU 사용률 수직 낙하)
        │   /        │       ╲
        │ /          │         ╲
      0 └────────────────────────────
          1   5   10   20   30
         다중 프로그래밍 정도 (프로세스 수)
```

### 3. 심층 분석: 글로벌 교체(Global Replacement)의 배신
- 대부분의 OS는 메모리가 모자라면 '다른 프로세스'가 쓰고 있던 공간을 강제로 빼앗아버리는 글로벌 교체 정책을 쓴다. 내가 램이 모자라서 옆 프로그램의 램을 뺏어오면, 옆 프로그램은 자기가 쫓겨난 줄도 모르고 있다가 다음 줄을 실행할 때 "내 램 어디 갔어!" 하며 또 남의 것을 뺏어온다. 이것이 도미노처럼 번지며 시스템 전체가 마비되는 것이 스래싱의 근본 원인이다.

- **📢 섹션 요약 비유**: 남의 옷을 훔쳐 입어 추위를 면했더니, 옷을 뺏긴 사람이 또 다른 사람의 옷을 훔치면서 결국 온 마을 사람들이 하루 종일 옷만 뺏고 뺏기느라 일을 못 하는 대환장 파티입니다.

## Ⅲ. 스래싱 방지 및 해결 전략 (Strategy)

스래싱을 막으려면 결국 "안 될 것 같으면 아예 들여보내지 않는다"는 원칙이 필요하다.

### 1. 하드웨어적 궁극의 해결책
- **물리 메모리(RAM) 증설 (Scale-up)**: 가상 메모리가 아무리 마법을 부려도 물리 법칙은 이길 수 없다. 스래싱이 나면 가장 먼저 팀장님을 찾아가 "램 32GB 꽂아주세요"라고 말하는 것이 진리다.

### 2. 소프트웨어적 해결책: 워킹셋(Working Set) 모델
- 학자들은 프로그램이 '최근 집중적으로 사용하는 페이지들의 모임(워킹셋)'을 분석했다.
- OS는 각 프로그램의 워킹셋 크기(예: A는 5장, B는 3장)를 합산한다.
- **규칙**: "현재 램에 남아있는 빈 프레임이 워킹셋의 합보다 작다면, 아예 새로운 프로그램을 램에 올리지 마라!" (심지어 기존 프로그램을 통째로 메모리에서 쫓아내어 재우기도 한다).

### 3. 소프트웨어적 해결책: PFF (Page Fault Frequency) 알고리즘
- 페이지 부재가 터지는 **빈도(비율)**를 실시간으로 모니터링한다.
- A 프로그램이 너무 자주 Page Fault를 낸다? $\to$ "너 램이 좁구나? 상자 더 줄게."
- B 프로그램이 Page Fault를 한 번도 안 낸다? $\to$ "너 램 남아도는구나? 램 내놔!"
- 융통성 있게 램을 뺏고 할당하며 시스템 균형을 유지한다.

- **📢 섹션 요약 비유**: 수용 인원(RAM)이 100명인 클럽에 무작정 손님을 계속 받지 않고, 기도(OS)가 입구에서 철저하게 인원수를 세며 대기표를 끊어주는 입장 관리 시스템입니다.

## Ⅳ. 실무 적용 및 최신 아키텍처적 판단

### 1. 실무 시나리오: OOM (Out Of Memory) Killer의 암살
- **상황**: 리눅스 운영체제에서 도커 컨테이너들을 잔뜩 띄웠는데 스래싱 징후가 감지됨. 시스템 전체가 뻗어버릴 위기.
- **판단**: 스래싱은 놔두면 컴퓨터가 완전히 얼어버려 강제 재부팅(Hard Reset) 외엔 답이 없다. 리눅스는 이 끔찍한 상황을 막기 위해 **OOM Killer**라는 자객을 작동시킨다. 시스템을 스래싱의 늪으로 끌고 가는 주범(가장 메모리를 많이 먹으면서 덜 중요한 프로세스, 보통 자바 JVM이나 DB)을 찾아내 자비 없이 `Kill` 시켜버린다. 엔지니어 입장에서는 서버가 갑자기 죽어 황당하지만, OS 입장에서는 '머리를 잘라 몸통(시스템 전체)을 살린' 최선의 방어다.

### 2. 기술사적 인사이트: SSD의 등장과 스래싱 체감의 변화
- **인사이트**: 과거 HDD(하드디스크) 시절의 스래싱은 컴퓨터가 몇 시간 동안 마우스도 안 움직이는 말 그대로 '지옥'이었다. 하지만 최근 초당 7GB를 복사해 대는 NVMe M.2 SSD가 탑재되면서, 램과 디스크 간의 스왑(Swap) 속도가 비약적으로 빨라졌다. 요즘 컴퓨터는 메모리가 꽉 차서 스래싱이 나더라도 "어? 컴퓨터가 살짝 버벅거리네?" 수준으로 느끼고 넘어가게 되었다. 하드웨어의 극단적인 발전이 OS의 낡은 단점을 물리력으로 덮어버린 대표적 사례다.

- **📢 섹션 요약 비유**: 물을 푸는 속도(HDD)가 느릴 때는 배에 구멍(스래싱)이 나면 무조건 침몰했지만, 초고성능 모터 펌프(SSD)를 달아놓은 지금은 구멍이 나도 배가 가라앉지 않고 버티는 형국입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **다중 프로그래밍의 한계 인식**: 가상 메모리의 '무한한 확장성'이라는 환상을 깨고, 물리적 자원 한계에 대한 정확한 수학적 제약을 제시.
- **OS 스케줄링 고도화**: CPU 사용량뿐만 아니라 메모리 압박(Memory Pressure) 수치까지 고려하여 프로세스의 생사를 결정하는 진보된 커널 스케줄러 설계 촉진.

### 2. 미래 전망
스래싱은 근본적으로 '용량의 절대적 부족'에서 온다. 미래에는 클라우드 환경에서 내 컴퓨터의 램이 모자라면, 빛의 속도로 네트워크를 타고 옆 서버의 남아도는 램을 내 램처럼 빌려다 쓰는 **메모리 분리(Memory Disaggregation) 기술 (CXL 등)**이 상용화될 것이다. 이렇게 되면 내 디스크로 스왑 아웃(Swap-out) 하느라 생기던 스래싱의 고통은 '옆 동네 램 빌려오기'를 통해 영원히 역사의 뒤안길로 사라질지도 모른다.

- **📢 섹션 요약 비유**: 내 지갑(RAM)이 텅 비어 파산(스래싱)할 위기에 처했을 때, 고리대금업자(디스크)를 찾아가지 않고 클라우드로 연결된 친구 지갑에서 실시간으로 돈을 빼 쓰는 기적의 금융 네트워크가 다가오고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 스래싱은 책상이 너무 좁아서 **'스케치북을 올리면 크레파스가 떨어지고, 크레파스를 올리면 스케치북이 떨어지는 상황'**이에요.
2. 그림은 하나도 못 그리고, 바닥에 떨어진 물건 주워서 올리느라 헉헉대기만 하면서 하루가 다 가버리죠.
3. 이럴 땐 욕심부리지 말고 그림 그리기를 포기하거나, 아빠한테 당장 큰 책상(RAM 증설)을 사달라고 졸라야만 문제가 해결된답니다!
