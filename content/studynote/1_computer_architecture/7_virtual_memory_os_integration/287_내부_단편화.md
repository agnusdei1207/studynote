> **💡 Insight**
> - 내부 단편화(Internal Fragmentation)는 메모리를 일정한 고정 크기의 블록으로 할당할 때, 요청된 크기보다 할당된 블록이 더 커서 남게 되는 잉여 공간의 낭비를 의미합니다.
> - 페이징(Paging) 기법의 태생적 한계로, 프로세스의 마지막 페이지가 프레임(Frame)을 가득 채우지 못할 때 필연적으로 발생합니다.
> - 페이지 크기(Page Size)를 줄이면 내부 단편화를 최소화할 수 있으나, 반대로 페이지 테이블 관리 오버헤드(Overhead)가 증가하는 딜레마(Trade-off)가 존재합니다.

## Ⅰ. 내부 단편화 (Internal Fragmentation)의 개념

내부 단편화(Internal Fragmentation)는 운영체제(OS: Operating System)의 주 기억 장치(Main Memory) 관리 기법 중 고정 분할(Fixed Partitioning)이나 페이징(Paging) 시스템에서 발생하는 메모리 낭비 현상입니다.
프로세스가 필요로 하는 메모리 양이 할당된 파티션이나 프레임 단위의 정확한 배수가 되지 않을 경우, 할당된 구역 "내부"에 남겨진 빈 공간은 다른 프로세스가 사용할 수 없어 낭비됩니다.

📢 섹션 요약 비유: 피자를 무조건 한 판(8조각) 단위로만 파는 식당에서, 9조각을 먹고 싶어 두 판을 시켰을 때 남게 되는 7조각의 피자와 같습니다.

## Ⅱ. 페이징 시스템에서의 내부 단편화 메커니즘

프로세스의 크기가 72KB이고 페이지 크기가 10KB라면, 프로세스는 총 8개의 페이지가 필요합니다. 마지막 8번째 페이지에는 2KB의 데이터만 들어가고 8KB가 비어있게 됩니다.

```ascii
  Logical Process (72KB)               Physical Frames (10KB each)
  +------------------+                 +------------------+
  | Page 0 (10KB)    | --------------> | Frame X          |
  +------------------+                 +------------------+
  | ...              |                 | ...              |
  +------------------+                 +------------------+
  | Page 7 (2KB data)| ------+         | Frame Y (10KB)   |
  +------------------+       +-------> +------------------+
                                       | 2KB Used         |
                                       +..................+ <-- Internal
                                       | 8KB Wasted       |     Fragmentation
                                       +------------------+
```

📢 섹션 요약 비유: 10리터짜리 규격 통에 72리터의 물을 담으려면 8개의 통이 필요하고, 마지막 8번째 통에는 물 2리터만 담겨 8리터의 공간이 낭비되는 원리입니다.

## Ⅲ. 외부 단편화(External Fragmentation)와의 비교

- **내부 단편화 (Internal Fragmentation):** 고정된 크기(페이징)로 할당할 때 발생. 할당된 블록 **안쪽**의 남는 공간.
- **외부 단편화 (External Fragmentation):** 가변 크기(세그멘테이션)로 할당할 때 발생. 메모리 공간 **전체**에 잘게 쪼개진 빈 공간들의 합은 충분하지만, 연속되지 않아 할당할 수 없는 상태.

📢 섹션 요약 비유: 내부 단편화는 큰 박스 안에 작은 물건을 넣어 박스 속이 텅 비는 것이고, 외부 단편화는 창고에 남은 자투리 공간은 많지만 큰 물건 하나를 통째로 밀어넣을 넓은 자리가 없는 것입니다.

## Ⅳ. 내부 단편화의 해결 접근법 및 트레이드오프

내부 단편화를 줄이는 가장 직접적인 방법은 **페이지 크기를 줄이는 것**입니다. 평균적으로 프로세스 당 (페이지 크기 / 2) 만큼의 단편화가 발생하기 때문입니다.
그러나 페이지 크기를 줄이면 시스템 전체의 페이지 개수가 급증하여 **페이지 테이블(Page Table)**의 크기가 비대해지고, 디스크 I/O 효율성이 떨어지며 TLB(Translation Lookaside Buffer) 적중률이 낮아지는 부작용이 생깁니다.

📢 섹션 요약 비유: 낭비를 줄이려고 물건을 담는 규격 상자를 너무 작게 만들면, 상자 개수가 너무 많아져 상자 번호를 적어둔 장부(페이지 테이블)가 사전처럼 두꺼워집니다.

## Ⅴ. 현대 시스템의 실용적 타협

대부분의 현대 운영체제는 내부 단편화로 인한 약간의 메모리 낭비를 감수하더라도, 주소 변환의 고속화와 외부 단편화 제거의 이점이 훨씬 크기 때문에 4KB 크기의 페이징을 표준으로 채택하고 있습니다. 최근에는 대규모 서버를 위해 2MB 등의 Huge Page를 지원하여 오히려 단편화를 감수하고 성능을 극대화하기도 합니다.

📢 섹션 요약 비유: 약간의 피자가 남더라도 주문과 계산을 빠르고 단순하게 하기 위해, 무조건 세트 메뉴(표준 규격)로만 통일해서 파는 패스트푸드점의 전략입니다.
