# 페이지 테이블 (Page Table)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이지 테이블(Page Table)은 운영체제가 메인 메모리(RAM)에 생성해 두는 거대한 배열 형태의 자료구조로, **가상 페이지 번호(Virtual Page)를 실제 물리 프레임 번호(Physical Frame)로 연결해 주는 '주소록 장부'**다.
> 2. **가치**: 단순히 주소만 번역하는 것이 아니라, 각 페이지가 현재 램에 있는지 하드디스크로 쫓겨났는지(Valid Bit), 그리고 읽기 전용인지 쓰기가 가능한지(Protection Bit)를 낱낱이 기록하여 **메모리 보호와 요구 페이징(Demand Paging)의 심장 역할**을 한다.
> 3. **융합**: 가상 메모리 공간이 64비트로 거대해지면서 장부 자체의 크기가 램 용량을 압도하는 문제가 발생하자, 이를 해결하기 위해 **다단계(Multi-level) 페이지 테이블**과 역 페이지 테이블(Inverted Page Table)이라는 고도의 자료구조 최적화와 융합되었다.

---

## Ⅰ. 페이지 테이블의 개요

### 1. 정의
- 페이징(Paging) 기법을 사용하는 가상 메모리 시스템에서, 프로세스의 논리적 페이지가 물리 메모리의 어느 프레임(Frame)에 적재되어 있는지를 1:1로 매핑하여 기록해 둔 표(Table)다. 프로세스마다 1개씩 존재한다.

### 2. 등장 배경: '어디에 쑤셔 넣었는지 기억하라'
- 페이징 기법은 메모리를 효율적으로 쓰기 위해 4KB 단위의 블록들을 램의 빈 공간 아무 데나 비연속적으로 던져 넣었다. 그러자 CPU가 변수를 찾을 때 "도대체 내 코드가 램의 어느 구석에 박혀있지?" 하고 길을 잃어버렸다. OS는 "걱정 마, 내가 프로세스마다 가상 번호와 진짜 번호를 매칭한 주소록(페이지 테이블)을 만들어 줄게. MMU야, 넌 이 장부만 보고 찾아가!"라고 해결책을 던졌다.

### 3. 💡 비유: '호텔의 투숙객 명부'
- 페이지 테이블은 **'호텔 프런트 데스크에 있는 방 배정 장부'**와 같다.
- 손님(CPU)이 "철수(가상 페이지 번호) 만나러 왔는데요"라고 하면, 프런트(MMU)는 장부를 펼쳐보고 "철수 고객님은 502호(진짜 물리 프레임 번호)에 계십니다"라고 번역해 준다. 만약 철수가 외출 중(하드디스크로 Swap-out)이라면 "안 계십니다(Page Fault)"라고 알려준다.

- **📢 섹션 요약 비유**: 수백만 개의 흩어진 택배 상자(데이터)가 진짜 어느 창고 어느 선반에 꽂혀 있는지 엑셀로 꼼꼼히 정리해 둔 만능 데이터베이스입니다.

## Ⅱ. 메커니즘 및 내부 구조 (Deep Dive)

페이지 테이블의 한 줄 한 줄을 PTE(Page Table Entry)라고 부른다.

### 1. PTE (Page Table Entry)의 해부학
단순한 주소 변환 외에 엄청난 양의 메타데이터(통제 권한)가 숨어 있다.
| 비트 필드 | 역할 및 의미 |
|:---:|:---|
| **Frame Number** | 이 가상 페이지가 위치한 진짜 물리 메모리의 번지수 |
| **Valid / Invalid Bit** | 1: 현재 RAM에 있다. / 0: 하드디스크(Swap)에 있거나 아예 없는 주소다. (Page Fault 유발) |
| **Protection (R/W/X)** | 읽기, 쓰기, 실행 권한. 악성 코드가 권한 없이 건드리면 즉시 처단! |
| **Dirty (Modified) Bit** | RAM에 올라온 뒤 내용이 수정되었나? (쫓겨날 때 디스크에 저장해야 할지 결정) |
| **Reference Bit** | 최근에 CPU가 읽어본 적 있나? (LRU/Clock 캐시 교체 알고리즘의 기준이 됨) |

### 2. 다단계(Multi-Level) 페이지 테이블의 마법 (ASCII)
32비트 OS에서 페이지 테이블 하나의 크기는 4MB다. 크롬 탭을 100개 띄우면 장부 크기만 400MB가 되어 메모리가 터진다. 이를 막기 위해 장부를 '목차'와 '본문'으로 쪼갠다.
```text
    [ CPU 요청 주소 ] -> 앞쪽 비트: 목차 번호 / 뒤쪽 비트: 본문 번호
    
    [ 1단계: Page Directory (목차) ]  --> 항상 RAM에 존재 (4KB로 매우 작음)
       ├── 0번 목차: (본문 표 A의 주소) ──────────┐
       └── 1번 목차: (본문 표 B의 주소 - 미사용)  │ (미사용이면 본문 표 자체를 RAM에 안 만듦! 용량 대폭 절약)
                                                  ▼
                                      [ 2단계: Page Table (본문 표 A) ]
                                         ├── 0번 페이지: Frame 5
                                         └── 1번 페이지: Frame 12  --> 진짜 물리 주소 도출!
```

### 3. 심층 분석: PTBR (Page Table Base Register)
- 멀티태스킹 환경에서 프로그램 A가 돌다가 B로 바뀔 때(Context Switch), 페이지 테이블 장부 전체를 CPU로 퍼오는 건 불가능하다. 대신 OS는 각 프로세스의 **장부가 시작되는 램의 첫 주소표지판(PTBR)**만 CPU 내부의 레지스터에 딱! 덮어씌워 준다. CPU는 순간적으로 우주(주소 공간)가 바뀐 것처럼 새로운 프로그램의 메모리를 번역하기 시작한다.

- **📢 섹션 요약 비유**: 두꺼운 백과사전을 통째로 들고 다니지 않고, "ㄱ~ㄷ은 1권에, ㄹ~ㅂ은 2권에 있다"는 얇은 목차만 들고 다니며 필요할 때만 본문을 찾는 똑똑한 공간 절약술입니다.

## Ⅲ. 요구 페이징과 페이지 폴트 (Page Fault)

장부에 `Valid Bit = 0`으로 적혀 있을 때 일어나는 거대한 운영체제 이벤트다.

1. **CPU**: "나 10번 페이지 데이터 좀 줘." (MMU가 장부를 뒤져봄)
2. **MMU**: "헉! Valid Bit가 0이네. 램에 없고 하드디스크에 있어!" $\to$ **Page Fault 인터럽트 발생!**
3. **OS 개입**: CPU는 하던 일을 멈추고 OS로 제어권을 넘긴다.
4. **I/O 작업**: OS는 디스크(Swap)에 가서 잠들어 있는 10번 페이지를 느릿느릿 RAM의 빈 공간으로 퍼온다.
5. **장부 업데이트**: OS는 페이지 테이블의 Frame 번호를 적고 `Valid = 1`로 바꾼 뒤, CPU에게 "다시 해봐!"라고 명령어를 재실행시킨다.

- **📢 섹션 요약 비유**: 식당에서 메뉴를 시켰는데 냉장고(RAM)에 재료가 없어서, 셰프(CPU)가 요리를 멈추고 창고(디스크)에 조수를 보내 재료를 가져올 때까지 손님을 기다리게 하는 뼈아픈 시간입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: 역 페이지 테이블 (Inverted Page Table)
- **상황**: 64비트 시스템이 되면서 주소 공간이 너무 넓어져, 다단계(4단계, 5단계)를 써도 페이지 테이블 장부가 램을 다 갉아먹는 한계에 봉착했다.
- **판단**: 천재적인 발상의 전환. "가상 주소 1개당 장부를 1줄씩 쓰지 말고, **실제 램의 물리 프레임 1개당 장부를 1줄씩 쓰자!**" 램이 16GB라면 딱 그 개수만큼만 장부가 존재하게 된다. 대신 CPU가 찾는 가상 주소를 해시 함수(Hash)로 돌려 테이블을 뒤지는 방식을 쓴다. 메모리 낭비를 극단적으로 틀어막는 이 기술은 현재 고성능 서버 아키텍처(Itanium, PowerPC 등)에서 핵심 공간 절약 기법으로 활용된다.

### 2. 실무 시나리오: 메모리 맵 파일 (Memory-Mapped File)
- **인사이트**: 디스크에 있는 10GB짜리 거대 파일을 읽을 때, `read()` 함수로 버퍼에 일일이 퍼오면 복사(Copy) 때문에 시스템이 뻗는다. 리눅스 개발자는 `mmap()` 시스템 콜을 써서, 파일을 **현재 프로세스의 페이지 테이블에 직접 꽂아버린다**. 그러면 램에 데이터가 안 올라와 있어도 CPU가 포인터로 접근하는 순간 Page Fault가 터지면서 OS가 알아서 디스크의 파일 조각을 램으로 스무스하게 당겨온다. 대용량 DB(몽고DB 등)가 디스크를 마치 램처럼 다루며 미친 성능을 내는 비밀 병기다.

- **📢 섹션 요약 비유**: 물건을 보기 위해 택배를 내 방으로 일일이 배달시키는 대신, 물류 창고(디스크)에 직통으로 연결된 포탈(mmap)을 뚫어서 손만 뻗으면 바로 꺼내 쓸 수 있게 한 웜홀 마법입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **철통같은 보안 벽 (Isolation)**: 프로그램은 자신의 페이지 테이블에 적힌 곳 외에는 물리적으로 절대 접근할 수 없어, 크롬 브라우저 탭 하나가 랜섬웨어에 걸려도 다른 탭은 완벽히 안전함.
- **가상 메모리의 실체화**: 폰 노이만 구조의 한계(램 크기)를 소프트웨어적 장부 조작으로 부수어 버린 운영체제의 가장 위대한 발명품.

### 2. 미래 전망
최근 가상화(Virtualization) 환경에서는 클라우드 안의 OS(Guest)가 만든 페이지 테이블을 하이퍼바이저(Host)가 다시 한번 진짜 램으로 번역하는 이중 번역 지옥이 펼쳐진다. 이를 위해 인텔과 AMD는 하드웨어 내부에 **확장 페이지 테이블(EPT)**이라는 2차원 장부 구조를 아예 칩 속에 때려 박았다. 페이지 테이블은 단순한 주소록을 넘어, 클라우드와 가상화라는 수천억 달러짜리 시장을 지탱하는 가장 거대하고 복잡한 하드웨어-소프트웨어 융합 엔진으로 무한 증식하고 있다.

- **📢 섹션 요약 비유**: 옛날엔 동네 주민 명부(1단계)로 족했지만, 지금은 지구(가상 머신)의 주소를 우주(물리 서버)의 좌표로 두 번, 세 번 변환하는 은하계 통합 네비게이션 시스템이 되었습니다.
