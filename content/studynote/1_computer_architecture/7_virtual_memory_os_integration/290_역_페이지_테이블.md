> **💡 Insight**
> - 역 페이지 테이블(Inverted Page Table)은 논리적 페이지 단위가 아닌 물리적 프레임(Frame)을 기준으로 페이지 테이블을 구성하는 획기적인 메모리 관리 기법입니다.
> - 시스템 전체에 단 하나의 페이지만 존재하며, 테이블의 크기가 물리적 메모리의 크기에 비례하여 고정되므로 메모리 낭비를 극적으로 줄입니다.
> - 테이블 검색 시간이 길어질 수 있다는 단점이 있으나, 해시 테이블(Hash Table)과 연관 레지스터(TLB)를 결합하여 검색 성능을 보완합니다.

## Ⅰ. 역 페이지 테이블의 등장 배경

전통적인 페이징(Paging) 기법은 각 프로세스마다 고유한 페이지 테이블(Page Table)을 가집니다. 프로세스 수가 많아지거나 64비트 환경처럼 가상 주소 공간이 막대하게 커질 경우, 테이블 자체가 물리 메모리를 과도하게 점유하는 문제(Table Bloat)가 발생합니다.
이를 해결하기 위해 가상 주소가 아닌 물리 주소 관점에서 테이블을 하나로 통합한 역 페이지 테이블(Inverted Page Table)이 등장했습니다.

📢 섹션 요약 비유: 모든 학생에게 개별 시간표(기존 페이지 테이블)를 나눠주는 대신, 교실(물리 메모리)마다 어떤 학생(프로세스)이 들어오는지를 기록한 학교 전체 종합 시간표(역 페이지 테이블) 하나만 벽에 붙여두는 방식입니다.

## Ⅱ. 역 페이지 테이블의 구조 및 메커니즘

역 페이지 테이블의 각 항목(Entry)은 물리 메모리의 프레임 순서와 일치합니다. 예를 들어 i번째 테이블 항목은 i번째 물리 프레임에 대응합니다.

```ascii
   Logical Address (PID, p, d)
        |    |
        v    v
  +--------------+        +---------------------------+
  | CPU / MMU    |        | Inverted Page Table (IPT) |
  +--------------+        | Index = Frame Number      |
        | Search          |---------------------------|
        | (PID, p) ---->  | 0:  PID_A, Page_3         |
        v                 | 1:  PID_C, Page_1         |
  Hash Function (Opt)     | 2:  PID_B, Page_5         |
        |                 | ...                       |
        +-------------->  +---------------------------+
```
CPU가 (프로세스 ID, 페이지 번호 p)를 요청하면, IPT를 순차적(또는 해싱)으로 검색하여 일치하는 항목의 인덱스 i를 찾고, 이를 프레임 번호 f로 사용합니다.

📢 섹션 요약 비유: 내 이름(PID)과 학번(페이지 번호)이 적힌 자리를 찾기 위해 교실 1번부터 끝번까지 책상 이름표(IPT)를 쭉 훑어보는 과정입니다.

## Ⅲ. 역 페이지 테이블의 핵심 장점

가장 큰 장점은 메모리 절약입니다. 논리적 가상 공간이 아무리 넓어져도, 역 페이지 테이블의 크기는 오직 실제 설치된 물리적 주기억장치(Main Memory) 용량에 비례합니다.
시스템 전체에 오직 1개의 테이블만 유지보수하면 되므로, 다수의 프로세스가 생성되어도 페이지 테이블로 인한 메모리 오버헤드가 증가하지 않습니다.

📢 섹션 요약 비유: 아무리 전교생이 1만 명(가상 공간)으로 늘어나도, 실제 학교의 교실 수(물리 메모리)가 100개라면 벽에 붙이는 종합 시간표 목록은 딱 100줄만 있으면 되는 경제적인 방법입니다.

## Ⅳ. 한계점 및 검색 성능 저하 문제

테이블이 프레임 기준으로 정렬되어 있기 때문에, 특정 논리 주소를 찾기 위해 테이블 전체를 선형 검색(Linear Search)해야 할 수도 있어 주소 변환 시간이 끔찍하게 길어질 수 있습니다.
또한, 메모리 공유(Shared Memory) 구현이 까다롭습니다. 여러 프로세스의 가상 페이지가 하나의 물리 프레임을 가리켜야 하는데, IPT는 프레임당 하나의 항목만 가지므로 1:N 매핑을 직접 표현하기 어렵습니다.

📢 섹션 요약 비유: 내 자리를 찾으려고 1번 책상부터 100번 책상까지 일일이 확인해야 하니 시간이 오래 걸리고, 하나의 책상을 두 명이 같이 쓰게 허락(메모리 공유)하려면 장부에 적기가 매우 애매해지는 상황입니다.

## Ⅴ. 성능 개선 기법: 해시(Hash) 및 TLB 도입

검색 시간을 단축하기 위해 해시 테이블(Hash Table) 기반의 역 페이지 테이블(Hashed Page Table) 아키텍처가 널리 쓰입니다. (PID, Page)를 해싱하여 탐색 범위를 O(1)에 가깝게 좁힙니다.
동시에 TLB(Translation Lookaside Buffer) 적중률을 최대한 높여, 역 페이지 테이블까지 접근해야 하는 상황 자체를 최소화하는 하드웨어 지원이 필수적입니다.

📢 섹션 요약 비유: 책상을 1번부터 찾지 않고, 내 학번을 계산식(해시 함수)에 넣으면 "너는 대략 70번대 책상에 있을 거야"라고 알려주어 금방 찾게 해주는 내비게이션 시스템을 도입한 것입니다.

> **💡 지식 그래프 및 어린이 비유**
> - **Knowledge Graph:** Physical Memory -> Inverted Page Table -> Indexed by Frame Number -> Reduces Memory Overhead -> Slower Search -> Mitigated by Hash Table & TLB.
> - **Child Analogy:** 주차장 관리를 할 때, '각 자동차가 어디 주차했는지' 명부를 만드는 게 아니라, '1번 주차칸엔 A차, 2번 주차칸엔 B차'라고 주차칸 위주의 장부를 하나만 만드는 거예요!
