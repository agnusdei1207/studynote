> **💡 Insight**
> - 다단계 페이지 테이블(Multilevel Page Table)은 방대한 크기의 단일 페이지 테이블(Single-level Page Table)을 여러 단계로 쪼개어 연속된 메모리 할당의 부담을 줄이는 기법입니다.
> - 사용되지 않는 가상 메모리 공간에 대한 테이블 생성을 생략할 수 있어, 메모리 공간 낭비(Overhead)를 획기적으로 절약할 수 있습니다.
> - 주소 변환 과정에서 메모리 접근 횟수가 단계 수만큼 늘어나는 성능 저하가 발생하지만, TLB(Translation Lookaside Buffer)를 통해 이 단점을 상쇄합니다.

## Ⅰ. 다단계 페이지 테이블의 등장 배경

32비트 운영체제(OS)에서 페이지 크기가 4KB라면, 한 프로세스당 100만 개($2^{20}$)의 페이지 항목이 필요하며 이는 약 4MB의 연속된 물리 메모리를 요구합니다. 64비트 시스템에서는 이 크기가 기하급수적으로 커져 단일 테이블로는 감당할 수 없습니다.
이를 해결하기 위해 페이지 테이블 자체를 다시 페이징(Paging)하여 트리(Tree) 구조로 관리하는 다단계 페이지 테이블(Hierarchical Page Table) 구조가 도입되었습니다.

📢 섹션 요약 비유: 두꺼운 백과사전을 통째로 들고 다니기 너무 무거우니, 'ㄱ, ㄴ, ㄷ' 색인만 적힌 얇은 수첩을 만들고, 필요한 자음의 소책자만 그때그때 꺼내 보는 방식입니다.

## Ⅱ. 2단계 페이지 테이블의 구조 및 변환 메커니즘

논리 주소는 P1(외부 페이지 번호), P2(내부 페이지 번호), d(오프셋)으로 세분화됩니다.

```ascii
   Logical Address: [   P1   |   P2   |   d   ]
                        |        |        |
         +--------------+        |        |
         v                       |        |
   +-------------+               |        |
   | Page Dir    |               |        |
   | (Outer PT)  |----+          |        |
   +-------------+    |          v        |
                      |   +-------------+ |
                      +-->| Page Table  | |
                          | (Inner PT)  |-+
                          +-------------+ |
                                          v
                                    +-------------+
                                    | Frame in    |
                                    | Main Memory |
                                    +-------------+
```
외부 페이지 테이블(Page Directory)은 내부 페이지 테이블의 기준 주소를 가리키고, 내부 페이지 테이블은 실제 프레임(Frame) 주소를 가리킵니다.

📢 섹션 요약 비유: 도서관에서 책을 찾을 때, 1층 안내도(외부 테이블)를 보고 '문학 코너'로 간 뒤, 문학 코너 안의 세부 목록(내부 테이블)을 보고 정확한 책장의 위치(프레임)를 찾는 2단계 과정입니다.

## Ⅲ. 다단계 페이지 테이블의 장점 (메모리 절약)

대부분의 프로세스는 할당받은 가상 메모리 공간의 극히 일부만 사용합니다. 
다단계 구조에서는 사용하지 않는 주소 공간에 대해 내부 페이지 테이블을 아예 메모리에 적재하지 않으므로, 페이지 테이블이 차지하는 잉여 메모리 공간을 대폭 절약할 수 있습니다.

📢 섹션 요약 비유: 도서관 전체 목록을 다 인쇄하지 않고, 빈 서가가 있는 곳은 아예 세부 목록표조차 만들지 않아 종이를 아끼는 효과입니다.

## Ⅳ. 다단계 페이지 테이블의 단점 (메모리 접근 시간 증가)

단계가 깊어질수록 논리 주소를 물리 주소로 변환하기 위해 메모리를 여러 번 접근(Memory Access)해야 합니다.
예를 들어 2단계 구조는 실제 데이터를 읽기 전, 주소 변환을 위해서만 2번의 메모리 접근이 필요하여 캐시 페널티가 커집니다.

📢 섹션 요약 비유: 목적지에 가기 위해 안내원을 한 명만 거치면 될 것을, 안내원의 안내원을 거쳐야 하므로 길을 묻는 데만 시간이 배로 걸리는 셈입니다.

## Ⅴ. 성능 저하의 해결책: TLB의 적극적 활용

주소 변환의 지연(Latency)을 극복하기 위해 하드웨어 기반의 고속 캐시인 TLB(Translation Lookaside Buffer)가 필수적으로 사용됩니다.
TLB에 변환 정보가 캐싱되어 있다면(TLB Hit), 다단계 페이지 테이블을 검색할 필요 없이 즉시 프레임 주소를 획득하여 성능 저하를 방지합니다.

📢 섹션 요약 비유: 길을 매번 안내원 여러 명에게 묻지 않고, 자주 가는 목적지의 주소는 스마트폰(TLB)에 저장해두고 바로 찾아가는 지름길입니다.

> **💡 지식 그래프 및 어린이 비유**
> - **Knowledge Graph:** Large Virtual Address -> Multi-level Page Table -> Splits Table into Hierarchy -> Saves Memory by ignoring unused spaces -> Increases Memory Access Time -> Solved by TLB.
> - **Child Analogy:** 보물찾기 할 때 첫 번째 쪽지에는 '나무 밑으로 가시오'라고 적혀 있고, 나무 밑으로 가면 나오는 두 번째 쪽지에 비로소 '보물 상자' 위치가 적혀 있는 다단계 힌트 게임이에요!
