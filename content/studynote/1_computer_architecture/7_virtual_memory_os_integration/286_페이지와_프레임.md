> **💡 Insight**
> - 가상 메모리(Virtual Memory) 관리의 핵심은 논리적 공간인 페이지(Page)와 물리적 공간인 프레임(Frame)을 동일한 크기로 나누는 것입니다.
> - 페이지와 프레임의 크기를 일치시킴으로써 운영체제(OS: Operating System)는 메모리 할당의 복잡성을 크게 줄이고 외부 단편화(External Fragmentation)를 원천적으로 해결합니다.
> - 페이지 테이블(Page Table)을 통해 페이지 번호(Page Number)를 프레임 번호(Frame Number)로 변환(Mapping)하는 과정이 페이징(Paging)의 핵심입니다.

## Ⅰ. 페이지(Page)와 프레임(Frame)의 정의

가상 메모리 시스템(Virtual Memory System)에서 프로세스(Process)의 논리 주소 공간(Logical Address Space)은 고정된 크기의 블록으로 나뉘는데, 이를 **페이지(Page)**라고 부릅니다.
반면, 주기억장치(Main Memory)의 물리 주소 공간(Physical Address Space) 역시 페이지와 완전히 동일한 크기의 블록으로 나뉘며, 이를 **프레임(Frame)**이라고 부릅니다.

📢 섹션 요약 비유: 하나의 긴 소설을 복사할 때, 소설의 각 '장(Page)'을 똑같은 크기의 빈 '복사 용지(Frame)'에 찍어내는 것과 같습니다.

## Ⅱ. 주소 변환(Address Translation) 메커니즘

CPU가 생성하는 논리 주소는 페이지 번호(p)와 페이지 오프셋(d)으로 구성되며, MMU(Memory Management Unit)를 통해 프레임 번호(f)와 오프셋(d)으로 변환됩니다.

```ascii
  Logical Address (CPU)                   Physical Address (Memory)
  +-------+---------+                     +-------+---------+
  |   p   |    d    |                     |   f   |    d    |
  +-------+---------+                     +-------+---------+
      |         |                             ^         |
      v         |                             |         v
+-----------+   |                         +---------------+
| Page Table|   |                         |  Main Memory  |
| p -> f    |---+------------------------>|  Frame f      |
+-----------+                             +---------------+
```
오프셋(d, Displacement)은 페이지나 프레임 내에서의 상대적인 위치를 나타내므로 변환 과정에서 변경되지 않습니다.

📢 섹션 요약 비유: 소설의 '3장(Page) 5번째 줄(Offset)'이 복사본 서류철의 '7번째 용지(Frame) 5번째 줄(Offset)'로 맵핑되어도, 줄 번호는 변하지 않습니다.

## Ⅲ. 크기 설정의 중요성 (Page Size)

페이지 크기(Page Size)는 시스템 성능에 지대한 영향을 미칩니다. 일반적으로 4KB가 많이 쓰이나, 대용량 데이터를 다루는 현대 시스템에서는 Huge Page (2MB, 1GB 등)를 지원합니다.
- **작은 페이지:** 내부 단편화(Internal Fragmentation)가 감소하지만, 페이지 테이블의 크기가 커져 오버헤드(Overhead)가 증가합니다.
- **큰 페이지:** 페이지 테이블의 크기가 줄고 TLB(Translation Lookaside Buffer) 적중률이 상승하지만, 내부 단편화가 심해져 메모리 낭비가 발생할 수 있습니다.

📢 섹션 요약 비유: 물건을 담는 박스 크기를 작게 하면 박스 안의 빈 공간은 적지만 박스 관리가 힘들고, 크게 하면 관리는 편하지만 빈 공간 낭비가 심해지는 이치입니다.

## Ⅳ. 페이징 시스템의 한계: 내부 단편화

프레임과 페이지가 고정 크기이므로, 프로세스가 요구하는 메모리가 페이지 크기의 정수배가 아닐 경우 마지막 프레임에는 사용되지 않는 여백이 남게 됩니다. 이를 내부 단편화(Internal Fragmentation)라고 합니다. 외부 단편화(External Fragmentation)는 완전히 해결되지만 내부 단편화는 피할 수 없습니다.

📢 섹션 요약 비유: 10명 단위로만 탈 수 있는 버스에 11명이 타야 하면, 결국 버스 2대를 빌려야 하고 두 번째 버스에는 9개의 빈자리가 생기는 것과 같습니다.

## Ⅴ. 현대 OS의 페이지 및 프레임 관리

최신 운영체제는 요구 페이징(Demand Paging) 기법을 사용하여, 프로세스가 실제로 해당 페이지를 요구할 때만 빈 프레임을 할당합니다. 빈 프레임을 관리하기 위해 운영체제는 자유 프레임 리스트(Free Frame List)를 유지보수합니다.

📢 섹션 요약 비유: 렌터카 업체가 현재 주차장에 비어 있는 차량(자유 프레임) 목록을 가지고 있다가, 고객이 요청(요구 페이징)할 때 즉시 차 키를 내주는 방식입니다.

> **💡 지식 그래프 및 어린이 비유**
> - **Knowledge Graph:** Logical Memory -> divided into Pages -> Physical Memory -> divided into Frames -> Page Table maps Page to Frame.
> - **Child Analogy:** 페이지는 내 스케치북의 그림 한 장이고, 프레임은 그 그림을 끼워 넣을 수 있는 벽에 걸린 똑같은 크기의 액자예요!
