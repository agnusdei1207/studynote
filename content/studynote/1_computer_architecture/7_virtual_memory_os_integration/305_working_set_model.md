# 워킹 셋 (Working Set) 모델

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 작업 집합 모델(Working Set Model)은 프로세스가 일정 시간 동안 **집중적으로 참조하는 메모리 페이지들의 집합(Working Set)을 추적**하여, 이 집합만큼의 공간을 무조건 물리 메모리(RAM)에 보장해 주는 운영체제의 메모리 관리 기법이다.
> 2. **가치**: "자주 쓰는 놈들은 흩어지지 않게 한꺼번에 램에 묶어둔다"는 철학을 통해, 가상 메모리 시스템의 끔찍한 재앙인 **스래싱(Thrashing)을 원천적으로 방지하고 다중 프로그래밍의 효율을 극대화**한다.
> 3. **융합**: 과거의 참조 기록을 통해 미래를 예측하는 **지역성의 원리(Principle of Locality)**를 하드웨어(캐시)가 아닌 소프트웨어(OS 메모리 할당) 차원에서 완벽하게 융합 적용한 수학적이고 동적인 스케줄링 모델이다.

---

## Ⅰ. 작업 집합 모델의 개요

### 1. 정의
- 특정 시간 간격(Window, $\Delta$) 동안 프로세스가 실제로 참조한 모든 페이지들의 집합을 '작업 집합(Working Set)'으로 정의하고, 이 집합의 크기만큼 물리적 프레임(Frame)을 해당 프로세스에게 할당해 주는 동적 메모리 할당 알고리즘이다. (1968년 피터 데닝(Peter Denning)이 제안)

### 2. 등장 배경: '스래싱의 공포를 막아라'
- 초기 가상 메모리는 램이 꽉 차면 닥치는 대로 남의 페이지를 뺏었다. 그러다 보니 프로세스가 루프를 돌 때 꼭 필요한 A, B, C 페이지 중 C를 뺏겨버려 페이지 폴트(Page Fault)가 발생하고 시스템이 뻗는 스래싱이 밥 먹듯 발생했다. 피터 데닝은 "어떤 프로그램이 지금 당장 원활하게 돌아가려면 **최소한 이것만큼은 세트로 가지고 있어야 한다**"는 절대 영역을 찾아냈고, 이를 지켜주자는 모델을 발표했다.

### 3. 💡 비유: '요리사의 필수 조리 도구 세트'
- 작업 집합은 **'요리사가 오늘 볶음밥을 만들기 위해 내 책상(도마) 위에 반드시 올려두어야 하는 필수 도구 3개(칼, 도마, 프라이팬)'**와 같다.
- 주방(RAM)이 좁다고 누군가 내 칼을 창고(디스크)에 치워버리면 나는 요리를 할 수가 없다. 그래서 주방장(OS)은 나에게 "너 지금 볶음밥 하니? 그럼 적어도 세 자리(Working Set)는 너 전용으로 무조건 보장해 줄게!"라고 구역을 쳐주는 것이다.

- **📢 섹션 요약 비유**: 뿔뿔이 흩어지면 아무 일도 못 하는 레고 부품들을 하나의 지퍼백(Working Set)에 담아, 뺏길 땐 다 같이 뺏기고 줄 땐 다 같이 주는 완벽한 패키지 관리법입니다.

## Ⅱ. 메커니즘 및 동적 크기 조절 (Deep Dive)

작업 집합은 고정되어 있지 않고 프로그램의 흐름에 따라 유기적으로 변한다.

### 1. 수학적 모델과 윈도우 크기 ($\Delta$)
- $WS(t, \Delta)$: 시각 $t$에서 과거 $\Delta$(Delta) 시간 동안 참조된 페이지들의 집합.
- $\Delta$ (윈도우 크기)가 핵심이다. 
  - $\Delta$가 너무 작으면: 진짜 필요한 페이지도 작업 집합에서 빠져버려 스래싱이 난다.
  - $\Delta$가 너무 크면: 옛날에 쓰고 버린 페이지까지 껴안고 있어 귀중한 램이 낭비된다.
- **동적 변화**: 루프를 돌 때는 작업 집합이 몇 개로 딱 고정되지만, 함수를 호출하거나 다른 루프로 넘어갈 때(Transition) 작업 집합의 크기가 갑자기 확 커졌다가 다시 안정화된다.

### 2. 작업 집합의 변화 시각화 (ASCII)
```text
    시간(t): 1 2 3 4 5 6 7 8 9
    참조된 페이지: [ 2, 6, 1, 5, 7, 7, 7, 7, 5 ]
    
    만약 윈도우(Δ) 크기가 3 이라면?
    ▶ t=3 일 때 WS: {2, 6, 1} -> 3칸 보장
    ▶ t=5 일 때 WS: {1, 5, 7} -> 3칸 보장
    ▶ t=8 일 때 WS: {7, 7, 7} -> 중복 제거하고 오직 {7} 만 보장! (남는 RAM 반납)
    
    * OS는 이 WS 크기만큼 빈 공간이 있을 때만 프로그램을 RAM에 올린다.
```

### 3. 심층 분석: 스래싱 방지의 절대 원칙
- 운영체제는 램에 떠 있는 모든 프로세스의 작업 집합 크기를 더한다. $\sum WS_i$
- 만약 **$\sum WS_i > 물리 메모리 총 크기$** 라면? 당장 스래싱이 터질 위기다!
- 이때 OS는 프로세스 하나를 통째로 램에서 하드디스크로 쫓아내어 재워버린다(Swap-out). 그리고 남은 램을 다른 프로세스들에게 나눠주어 나머지라도 살려낸다. 이것이 워킹셋 모델의 무자비하지만 확실한 구원 방식이다.

- **📢 섹션 요약 비유**: 구명보트에 탈 수 있는 인원(RAM)이 10명인데, 각 가족(프로세스)을 찢어놓지 않으려면 차라리 한 가족 통째로 바다에 남겨두고(Swap) 나머지 가족만 안전하게 살리는 냉혹한 캡틴의 결단입니다.

## Ⅲ. 워킹셋 모델 vs PFF (Page Fault Frequency)

워킹셋 모델을 현실에서 구현하기 힘들어 나온 현실적 대안이 PFF다.

| 비교 항목 | Working Set Model (작업 집합) | PFF (Page Fault Frequency) |
|:---|:---:|:---:|
| **감시 대상** | **과거 델타($\Delta$) 시간 동안 참조된 페이지들** | **단순히 페이지 폴트가 터지는 횟수(비율)** |
| **구현 난이도** | 극한 (타이머 걸고 매번 페이지를 추적해야 함) | **매우 쉬움** (폴트 날 때 카운터만 올리면 됨) |
| **메모리 할당 방식**| 집합 크기를 정확히 계산하여 할당 | 폴트가 잦으면 방을 더 주고, 없으면 뺏음 |
| **현실적 적용** | 이론적 배경 및 이상향 제시 | 실제 리눅스 등 현대 OS의 타협적 구현 방식 |

- **📢 섹션 요약 비유**: 매일 직원의 업무 일지(참조 기록)를 샅샅이 뒤져 필요 인력을 배정하는 완벽주의 사장님(Working Set)과, "바쁘다고 소리 지르면(Page Fault) 알바 더 뽑아줄게"라고 하는 현실주의 사장님(PFF)의 차이입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: 자바 JVM GC (Garbage Collection) 튜닝
- **상황**: 백엔드 스프링(Spring) 서버에서 트래픽이 몰리면 응답 속도가 끔찍하게 느려짐.
- **판단**: 자바 애플리케이션의 워킹셋(자주 쓰는 객체) 용량이 JVM에 할당된 힙(Heap) 메모리의 물리적 한계를 아슬아슬하게 넘나들고 있다. 이 때문에 OS가 JVM의 페이지를 자꾸 스왑아웃시키고, 동시에 JVM 안에서는 GC가 남은 메모리를 쥐어짜려 미친 듯이 돌아가는 이중 스래싱이 터진 것이다. 시니어 엔지니어는 프로파일러를 통해 이 서버의 '워킹셋(Working Set)' 크기를 정확히 산출하고, `-Xms`, `-Xmx` 옵션을 통해 넉넉한 고정 메모리를 할당하거나 서버의 물리 램 사양(Scale-up) 자체를 올려주어야 한다.

### 2. 기술사적 인사이트: OOM (Out Of Memory)를 대하는 OS의 철학
- **인사이트**: 리눅스는 워킹셋이 램 용량을 넘어서면 멍청하게 같이 죽지 않는다. 워킹셋 이론에 따라 "하나를 확실히 죽여서 다수를 살린다"는 철학으로 **OOM Killer**를 발동한다. OOM Killer는 내부적으로 프로세스들이 차지한 페이지 수(워킹셋)와 중요도(oom_score_adj)를 계산하여 가장 뚱뚱하고 덜 중요한 놈의 명줄을 끊는다. 이는 워킹셋 모델이 시스템 안정성(Stability)을 유지하기 위해 OS 내부에 심어놓은 가장 폭력적이면서도 가장 위대한 생존 본능이다.

- **📢 섹션 요약 비유**: 나무에 달린 과일(RAM)보다 먹여 살릴 입(워킹셋)이 더 많아지면, 다 같이 굶어 죽기 전에 가장 많이 먹는 멧돼지 하나를 절벽으로 밀어버리는 생태계의 잔혹한 섭리입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **스래싱(Thrashing) 99% 차단**: 프로세스마다 최소한의 숨 쉴 공간(프레임)을 보장함으로써 디스크 I/O로 낭비되는 CPU 사이클을 획기적으로 구원.
- **동적 자원 분배의 최적화**: 런타임에 프로그램의 행태 변화(Phase Change)를 감지하여 램을 뺏고 돌려주는 유연한 공산주의적 분배 실현.

### 2. 미래 전망
최근 컨테이너(Docker, Kubernetes) 기반의 클라우드 인프라에서는 수천 개의 마이크로서비스가 돌아간다. 쿠버네티스는 각 파드(Pod)의 '워킹셋'을 실시간으로 분석하여, Limit(한계치)와 Request(보장치)라는 이름으로 메모리 자원을 스케줄링한다. 과거 단일 OS 안에서 돌아가던 워킹셋의 개념이, 이제는 전 세계 데이터센터를 엮는 거대한 오케스트레이션(Orchestration) 플랫폼의 핵심 클러스터 스케줄링 알고리즘으로 스케일업 되어 찬란하게 부활하고 있다.

- **📢 섹션 요약 비유**: 한 명의 사장님이 직원 책상 크기를 정해주던 시절을 지나, 이제는 전 세계의 위성 사무실(컨테이너) 크기를 AI가 실시간으로 늘렸다 줄였다 하는 글로벌 스마트 오피스 시대가 되었습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 워킹셋 모델은 레고를 조립할 때 **'지금 만들고 있는 성(Castle) 부분에 필요한 블록들만 작은 쟁반에 담아두는 것'**이에요.
2. 쟁반(RAM)이 너무 작아서 성문 블록을 다른 상자에 치워버리면, 성문을 조립할 때마다 상자를 뒤지느라(스래싱) 하루 종일 걸려요.
3. 그래서 엄마(운영체제)는 "네가 성을 다 만들 때까지 이 쟁반만큼은 절대 건드리지 않고 널 위해 보장해 줄게!"라고 약속해 주는 거랍니다!
