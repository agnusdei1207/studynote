# 306. 스래싱 (Thrashing)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 과도한 페이지 교체로 시스템이 마비되는 현상
> 2. **가치**: 메모리 부족 문제 이해, 성능 저하 원인
> 3. **융합**: 워킹셋, 페이지 폴트, CPU 이용률

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

스래싱(Thrashing)은 **프로세스의 워킹셋이 물리 메모리보다 커서 과도한 페이지 교체가 발생하고, CPU 이용률이 급격히 떨어지는 현상**이다. 시스템이 페이지 교체만 하느라 실제 작업을 거의 못 한다.

### 💡 비유: 너무 작은 책장

스래싱은 **너무 작은 책장**과 같다. 책장이 작아서 필요한 책을 꺼내려면 다른 책을 창고로 보내야 한다. 창고에서 책을 가져오면 또 다른 책을 보내야 한다. 책을 읽는 것보다 옮기는 데 시간을 다 쓴다. 결국 아무것도 못 읽는다.

### 스래싱 개요

```

│                스래싱 개요                                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【스래싱 발생 원인】                                                            │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  조건:                                                               │
│  │  • 워킹셋 > 할당된 물리 메모리                                          │ │
│  │  • 동시 실행 프로세스가 너무 많음                                        │ │
│  │  • 메모리가 부족하여 빈번한 페이지 교체                                    │ │
│  │                                                               │ │
│  │  과정:                                                               │ │
│  │  1. 메모리 부족 → 페이지 교체 빈번                                       │ │
│  │  2. 페이지 폴트 급증                                                   │ │
│  │  3. CPU가 I/O 대기 (디스크)                                            │ │
│  │  4. CPU 이용률 급감                                                    │ │
│  │  5. OS가 더 많은 프로세스 스케줄 (CPU 한가하다고 판단)                      │ │
│  │  6. 메모리 부족 악화                                                    │ │
│  │  7. 악순환 반복 → 시스템 마비                                           │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【CPU 이용률 vs 스래싱】                                                         │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  CPU 이용률                                                          │ │
│  │    100% ┤                    ┌─────────────────                │ │
│  │         │                   /│                                │ │
│  │     80% ┤          ┌───────/ │                                │ │
│  │         │         /         │← 스래싱 발생 지점                   │ │
│  │     60% ┤        /│         │\                               │ │
│  │         │       / │         │ \                              │ │
│  │     40% ┤      /  │         │  \    ← 스래싱: CPU 급감           │ │
│  │         │     /   │         │   \                             │ │
│  │     20% ┤    /    │         │    \                            │ │
│  │         │   /     │         │     \___                        │ │
│  │      0% └──┴──────┴─────────┴─────────┴───                     │ │
│  │            적정    최적점    과도한    스래싱                       │ │
│  │           멀티    (피크)    멀티프로그래밍                         │ │
│  │          프로그래밍                                                │ │
│  │                                                               │ │
│  │  → 프로세스 수가 늘어도 CPU 이용률이 떨어지면 스래싱                      │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【스래싱 증상】                                                                 │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • CPU 이용률 급감 (10% 이하)                                          │ │
│  │  • 디스크 I/O 급증                                                    │ │
│  │  • 시스템 응답 없음                                                    │ │
│  │  • 페이지 폴트율 급증                                                  │ │
│  │  • 교체 비율 높음 (swap in/out)                                        │ │
│  │  • 사용자 체감 성능 급저하                                              │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 스래싱 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                스래싱 상세                                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【스래싱 해결 방법】                                                            │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  1. 물리 메모리 증설:                                                  │
│  │     • 근본적 해결                                                     │ │
│  │     • 워킹셋을 모두 수용할 메모리                                        │ │
│  │                                                               │ │
│  │  2. 프로세스 수 제한:                                                   │
│  │     • 멀티프로그래밍 정도(Degree) 감소                                   │ │
│  │     • CPU 이용률 기반 조절                                              │ │
│  │                                                               │ │
│  │  3. 워킹셋 모델:                                                        │
│  │     • 각 프로세스 워킹셋만큼 프레임 보장                                  │ │
│  │     • 워킹셋 > 프레임이면 프로세스 중단                                   │ │
│  │                                                               │ │
│  │  4. Page Fault Frequency (PFF):                                       │ │
│  │     • 페이지 폴트율 기반 프레임 할당                                      │ │
│  │     • 폴트율 높으면 프레임 증가                                          │ │
│  │     • 폴트율 낮으면 프레임 감소                                          │ │
│  │                                                               │ │
│  │  5. OOM Killer:                                                        │
│  │     • 메모리 부족 시 프로세스 강제 종료                                   │ │
│  │     • Linux에서 사용                                                   │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【워킹셋과 스래싱】                                                              │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  워킹셋(Working Set): 자주 사용하는 페이지 집합                           │ │
│  │                                                               │ │
│  │  정상:                                                               │
│  │  ∑(모든 프로세스 워킹셋) ≤ 물리 메모리                                   │ │
│  │  → 페이지 교체 적음, CPU 이용률 높음                                     │ │
│  │                                                               │ │
│  │  스래싱:                                                               │ │
│  │  ∑(모든 프로세스 워킹셋) > 물리 메모리                                   │ │
│  │  → 페이지 교체 빈번, CPU 이용률 낮음                                     │ │
│  │                                                               │ │
│  │  예: 물리 메모리 100MB, 3개 프로세스 각 워킹셋 40MB                       │ │
│  │  총 워킹셋 = 120MB > 100MB → 스래싱 발생                                │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【로드 컨트롤 (Load Control)】                                                     │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  L=S 기준 (L=S Criterion):                                            │ │
│  │  • L: 페이지 폴트 간 평균 시간                                          │ │
│  │  • S: 페이지 교체 서비스 시간                                            │ │
│  │  • L ≈ S이면 최적 CPU 이용률                                           │ │
│  │  • L < S이면 스래싱                                                   │ │
│  │                                                               │ │
│  │  OS 조치:                                                              │ │
│  │  • L < S 감지 시 프로세스 스왑 아웃                                      │ │
│  │  • CPU 이용률 모니터링                                                 │ │
│  │  • 과부하 시 프로세스 일시 중단                                          │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### 스래싱 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                스래싱 분석                                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【스래싱 감지】                                                                │)
│  ──────────────────                                                  │
│  $ vmstat 1                                                         │
│  procs -----------memory---------- ---swap-- -----io----            │
│   r  b   swpd   free   buff  cache   si   so    bi    bo             │
│   1  15  1024   100    200   2000    500  500   1000  1000           │
│                                                                     │
│  # 스래싱 의심 징후:                                                   │
│  # - b (blocked) 높음 (I/O 대기)                                       │
│  # - si/so 높음 (스왑 in/out 활발)                                     │
│  # - free 매우 낮음                                                    │
│  # - bi/bo 높음 (디스크 I/O 활발)                                       │
│                                                                     │
│  【페이지 폴트율 확인】                                                            │)
│  ──────────────────                                                  │
│  $ sar -B 1 10                                                      │
│  pgpgin/s pgpgout/s fault/s  majflt/s                               │
│  100.00   100.00     5000.00   100.00                               │
│                                                                     │
│  # majflt/s 높으면 스래싱 의심 (메이저 폴트 = 디스크 I/O)                 │
│                                                                     │
│  【스래싱 방지 설정】                                                            │)
│  ──────────────────                                                  │
│  # swappiness 낮춤 (스왑 적극성 감소)                                   │
│  $ sysctl vm.swappiness=10                                          │
│                                                                     │
│  # 최소 여유 메모리 확보                                                │
│  $ sysctl vm.min_free_kbytes=524288  # 512MB                        │
│                                                                     │
│  # OOM 설정                                                          │
│  $ sysctl vm.overcommit_memory=2                                    │
│                                                                     │
│  【스래싱 발생 시 대응】                                                          │)
│  ──────────────────                                                  │
│  1. 즉시: 메모리 많이 쓰는 프로세스 종료                                  │
│  2. 단기: 메모리 증설                                                  │
│  3. 설정: swappiness 낮춤, min_free_kbytes 증가                        │
│  4. 앱: 메모리 사용 최적화                                              │
│  5. DB: 버퍼 풀 크기 조정                                               │
│                                                                     │
│  【스래싱 최적화 팁】                                                             │)
│  ──────────────────                                                  │
│  1. 메모리 충분: 워킹셋 > 메모리면 증설                                  │
│  2. swappiness: DB 서버는 0~10                                        │
│  3. 모니터링: si/so, majflt/s, CPU%                                  │
│  4. 프로세스 제한: 동시 실행 수 제어                                     │
│  5. SSD: 스왑 디스크를 SSD로                                           │
│  6. Huge Page: 페이지 수 감소로 스래싱 방지                              │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• 스래싱: 과도한 페이지 교체로 시스템 마비
• 원인: 워킹셋 > 물리 메모리
• 증상: CPU 이용률 급감, 디스크 I/O 급증
• 과정: 페이지 폴트 → I/O 대기 → CPU 한가 → 더 많은 프로세스 → 악순환
• 해결: 메모리 증설, 프로세스 제한, 워킹셋 모델, PFF
• 워킹셋: 자주 사용하는 페이지 집합
• PFF: 폴트율 기반 프레임 할당
• L=S 기준: 폴트 간 시간 ≈ 서비스 시간
• 감지: vmstat si/so, majflt/s
• 방지: 충분한 메모리, swappiness 조정
```
