# 메모리 보호 (Memory Protection)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메모리 보호(Memory Protection)는 실행 중인 프로그램(프로세스)이 **자신에게 할당되지 않은 남의 메모리 영역이나 운영체제 핵심 영역(커널)을 함부로 읽거나 쓰는 것을 하드웨어 레벨에서 원천 봉쇄하는 보안 메커니즘**이다.
> 2. **가치**: 하나의 프로그램에 발생한 버그(예: 무한 루프, 포인터 오류)나 악성 해킹 코드가 컴퓨터 전체를 블루스크린(Kernel Panic)으로 몰고 가는 것을 막아주는 현대 멀티태스킹 OS의 최후의 방패다.
> 3. **융합**: 가상 메모리를 관리하는 **MMU(메모리 관리 유닛)** 내부의 페이지 테이블 권한 비트(R/W/X)와 베이스/리미트 레지스터(Base/Limit Register)라는 하드웨어 회로가 운영체제의 권한 관리와 융합되어 치밀하게 작동한다.

---

## Ⅰ. 메모리 보호의 개요

### 1. 정의
- CPU가 메모리에 접근하려 할 때마다, 그 주소가 해당 프로세스에게 허락된 합법적 영역인지, 그리고 요청한 작업(읽기, 쓰기, 실행)이 권한에 부합하는지를 하드웨어(MMU)가 실시간으로 검사하여 불법 접근을 차단(Trap)하는 기능이다.

### 2. 등장 배경: '모두가 섞여 살던 무법지대의 종말'
- 초창기 MS-DOS 시절에는 메모리 보호 개념이 없었다. 게임을 하다가 포인터(주소)를 잘못 건드려서 엉뚱한 곳에 숫자 0을 썼는데, 하필 그곳이 마우스를 움직이게 하는 운영체제 코드가 있던 자리였다면? 마우스가 먹통이 되고 컴퓨터를 강제 종료해야 했다. 악성코드도 남의 비밀번호를 그냥 메모리에서 훔쳐봤다. "이대론 다 죽는다. 각자 자기 방(할당된 메모리) 안에서만 놀게 철창을 치자!" 이것이 메모리 보호의 시작이다.

### 3. 💡 비유: '아파트의 카드키 출입 시스템'
- 메모리 보호는 **'고급 아파트 엘리베이터의 카드키 시스템'**과 같다.
- 내 카드키(MMU 권한)를 찍으면 내 집이 있는 5층(할당된 메모리) 버튼만 눌린다. 남의 집인 10층이나, 관리사무소(운영체제 커널)인 1층은 아무리 버튼을 눌러도 불이 들어오지 않는다. 억지로 문을 따려 하면 경비원(Segmentation Fault)이 즉각 출동해 나를 아파트에서 쫓아낸다.

- **📢 섹션 요약 비유**: 남의 땅을 함부로 밟으면 지뢰가 터져 당사자만 죽고, 동네 전체의 평화는 완벽하게 유지되는 무자비하지만 정의로운 룰입니다.

## Ⅱ. 메커니즘 및 하드웨어 동작 (Deep Dive)

메모리 보호는 소프트웨어가 아닌 순수 **하드웨어(MMU)의 속도**로 1클럭 내에 처리된다.

### 1. 보호의 2대 축: 공간(Space) 방어와 권한(Permission) 방어
- **공간 방어 (Base & Limit Register)**
  - 세그먼테이션 방식에서 사용된다. CPU가 `150번지`를 요구하면, 하드웨어가 묻는다. "너에게 허락된 방 크기(Limit)가 100인데 왜 150번지를 열려고 해?" 그 즉시 CPU에 인터럽트를 걸어 프로그램을 사살(Kill)한다.
- **권한 방어 (Page Table Protection Bits)**
  - 현대 페이징 시스템에서는 페이지 테이블의 각 줄(PTE)마다 `R(읽기)`, `W(쓰기)`, `X(실행)` 비트가 달려 있다. 
  - 상수(Constant)나 코드(Code) 영역은 `R=1, W=0`으로 세팅되어 있다. 만약 악성코드가 이 코드 영역을 변조(Write)하려 하면, MMU가 `W=0`인 것을 보고 즉시 망치를 내려친다.

### 2. 하드웨어 보호 매커니즘 시각화 (ASCII)
```text
    [ CPU ] ──( 0x5000 주소에 '데이터 쓰기(Write)' 시도 )──▶ [ MMU ]
                                                              │
    ┌─────────────────────── Page Table ──────────────────────┐│
    │ 가상 주소  | 물리 프레임 | Valid | R (Read) | W (Write) ││
    │ 0x5000    | 0x12       |   1   |   1      |    0      ││ <- (쓰기 권한 없음!)
    └─────────────────────────────────────────────────────────┘│
                                                              │
                 ┌────────────────────────────────────────────┘
                 ▼ (MMU의 분노)
    [ 삐용삐용! Trap 발생! OS 개입! ]
    운영체제: "너 감히 읽기 전용 구역에 쓰기를 시도해? 너 사형(Segmentation Fault)!"
    -> 불법 프로그램 즉시 강제 종료 (Core Dumped).
```

### 3. 심층 분석: 커널 모드 (Ring 0) vs 유저 모드 (Ring 3)
- 메모리는 크게 운영체제가 사는 '커널 스페이스'와 일반 프로그램이 사는 '유저 스페이스'로 반갈죽 되어 있다.
- 일반 프로그램(Ring 3)은 죽었다 깨어나도 커널 스페이스의 주소를 부를 수 없다. 만약 디스크를 읽고 싶다면, 반드시 합법적인 절차인 **시스템 콜(System Call)**을 통해 운영체제에게 "저 대신 램 좀 읽어주세요 굽신굽신" 해야만 한다. 이 철벽같은 신분제가 메모리 보호의 근간이다.

- **📢 섹션 요약 비유**: 민간인(유저 모드)은 절대 군사 기지(커널 스페이스)에 들어갈 수 없고, 뭔가 필요하면 헌병(시스템 콜)에게 부탁해서 대신 받아와야 하는 철저한 통제 구역입니다.

## Ⅲ. 보호 실패의 역사: 멜트다운과 스펙터

하드웨어의 보호막이 뚫리면 세상이 망한다.

| 취약점 이름 | 공격 원리 (어떻게 뚫었나?) | 방어 대책 |
|:---|:---|:---|
| **멜트다운 (Meltdown)** | 유저 프로그램이 커널 메모리를 훔쳐보려 할 때, MMU가 막기 전 **찰나의 시간 동안(비순차 실행) CPU가 몰래 커널 데이터를 캐시에 올려버림**. 이 캐시의 흔적을 분석해 비밀번호를 탈취. | OS가 유저와 커널의 페이지 테이블을 아예 물리적으로 찢어버림 (KPTI 패치) $\to$ 성능 폭락. |
| **스펙터 (Spectre)** | **분기 예측기(Branch Predictor)**를 고의로 속여, 남의 메모리 구역을 투기적으로 실행(접근)하게 만든 뒤 캐시에 남은 잔상을 훔침. | 분기 예측 시 권한 검사 하드웨어 로직 강화 및 소프트웨어 방어벽 추가. |

- **📢 섹션 요약 비유**: 은행 금고(메모리) 문은 잠겨 있었지만, 도둑이 창문으로 카메라(비순차 실행)를 밀어 넣어 금고 비밀번호가 적힌 메모지(캐시)를 찰나의 순간에 찍어간 최악의 해킹 사건입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: W^X (Write XOR Execute) 보안 정책
- **상황**: 해커들이 램의 데이터 구역(버퍼)에 몰래 악성 기계어를 써넣은(Write) 뒤, 강제로 거기로 점프해서 그 코드를 실행(Execute)시키는 버퍼 오버플로우(Buffer Overflow) 공격을 남발함.
- **판단**: 천재 아키텍트들과 리눅스/윈도우 진영은 하드웨어 MMU에 **NX 비트(No-eXecute bit)**를 도입했다. 그리고 위대한 보안 철학인 **W^X (Write XOR Execute)**를 강제했다. "어떤 메모리 공간이든, 글씨를 쓸 수 있다면(Write) 절대 실행(Execute)되게 두지 말고, 실행될 수 있다면 절대 쓸 수 없게 만들어라!" 즉, 두 권한이 동시에 켜지는 것을 원천 봉쇄했다. 이로 인해 메모리에 악성 코드를 심고 실행시키는 고전적인 해킹은 지구상에서 거의 멸종했다.

### 2. 실무 시나리오: C/C++ `const` 키워드와 `Segfault`
- **인사이트**: C언어에서 전역 변수를 `const char* str = "hello";` 라고 선언했다 치자.
- 초보자는 "이건 C 문법이 쓰기를 막는 거야"라고 생각하지만, 실상은 **운영체제가 이 문자열을 메모리에 적재할 때 해당 페이지 테이블의 권한 비트를 Read-Only(읽기 전용)로 하드웨어 세팅해 버린다**. 그래서 포인터를 우회해서 억지로 값을 바꾸려 하면 C언어가 에러를 내는 게 아니라, CPU 하드웨어 MMU가 분노하여 `Segmentation Fault`를 띄우고 앱을 죽여버린다. `const`는 단순한 문법이 아니라 거대한 하드웨어 보호 체계와 직결된 맹세다.

- **📢 섹션 요약 비유**: 문서에 "수정 금지"라고 볼펜으로 쓴 것(소프트웨어 문법)이 아니라, 아예 특수 코팅(하드웨어 MMU)을 발라버려서 누가 펜을 대는 순간 종이가 폭발해버리는 무시무시한 결계입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **운영체제의 무결성 확보**: 어떤 악성 프로그램이나 버그가 터져도 운영체제 본연의 코드는 100% 안전하게 보호되어 시스템 블루스크린(Panic) 제로화 달성.
- **멀티태스킹의 신뢰성 보장**: A 프로그램이 멈춰도 B 프로그램은 아무 일 없었다는 듯이 돌아갈 수 있는 모던 OS의 샌드박싱(Sandboxing) 토대 완성.

### 2. 미래 전망
최근 ARM 아키텍처는 페이지 단위(4KB)의 거친 보호를 넘어, **MTE (Memory Tagging Extension)** 기술을 도입했다. 포인터 주소의 남는 비트에 4비트짜리 '컬러 태그'를 달고, 메모리에도 같은 색깔 태그를 칠해둔다. "빨간색 열쇠(포인터)로는 빨간색 메모리만 열 수 있다"는 룰이다. 페이지 내부에 있는 작은 변수 단위의 해킹(Use-After-Free 등)마저 하드웨어 레벨에서 0.1나노초 만에 완벽하게 차단해 버리는 이 놀라운 발전은, 메모리 보호가 투박한 철창에서 정밀한 레이저 보안망으로 진화하고 있음을 증명한다.

- **📢 섹션 요약 비유**: 아파트 정문만 지키던 경비 아저씨가, 이제는 집안의 서랍장 하나하나마다 자물쇠가 걸려있는지 1초에 수십억 번 감시하는 무적의 철통 보안 시스템으로 진화했습니다.
