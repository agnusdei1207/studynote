> **핵심 인사이트**
> 1. 인터럽트(Interrupt)는 CPU가 열심히 프로그램을 순차적으로 실행하고 있을 때, 키보드 입력이나 하드웨어 오류 같은 **예상치 못한 아주 급하고 중요한 이벤트가 발생하여 CPU의 하던 일을 강제로 멈춰 세우는 '비상벨' 시스템**이다.
> 2. 인터럽트가 발생하면 CPU는 하던 일(문맥, Context)을 스택(Stack) 메모리에 안전하게 백업해 두고, 지정된 인터럽트 처리기(ISR)로 점프하여 비상 사태를 해결한 뒤, 다시 원래 작업으로 돌아와 평온하게 일을 이어간다.
> 3. 과거 폴링(Polling) 방식처럼 CPU가 1초마다 "키보드 눌렀니?" 하고 멍청하게 물어보느라 귀한 클럭을 낭비하는 짓을 막고, **다중 작업(Multi-tasking)과 실시간 OS 아키텍처를 가능하게 만든 역사상 가장 위대한 발명**이다.

---

## Ⅰ. 무한 질문(Polling)의 늪에서 벗어나기
컴퓨터를 켰는데 내가 키보드로 'A'를 언제 누를지 CPU는 어떻게 알까요?
가장 멍청한 방법은 **폴링(Polling)**입니다. CPU가 0.001초마다 키보드한테 "너 눌렸어? 아니? 그럼 말고." "지금은 눌렸어? 아니?"를 무한히 반복하며 묻는 방식입니다. 내가 1시간 동안 아무것도 안 누르면 CPU는 허공에 1시간 동안 질문만 던지며 황금 같은 계산 능력을 완전히 낭비하게 됩니다.

이 바보 같은 짓을 때려치우기 위해 **인터럽트(비상벨)**를 발명했습니다.
CPU: "야 키보드, 나 딴 일(게임 연산) 하고 있을 테니까, 네가 꾹 눌릴 때만 나한테 와서 뒤통수를 빡 때려(인터럽트 선에 전기 1을 쏴)!" 
이렇게 하면 CPU는 1초의 낭비도 없이 자기 일을 풀가동하다가, 내가 키보드를 누르는 바로 그 찰나의 순간에만 잠깐 뒤를 돌아보고 'A' 글자를 화면에 띄운 뒤 다시 게임 연산으로 돌아갈 수 있습니다.

```ascii
[폴링(Polling) vs 인터럽트(Interrupt)의 CPU 효율성]

 [ 폴링 방식: CPU의 막대한 낭비 ]
 CPU "마우스 움직였어?" -> 안움직임 (대기)
 CPU "마우스 움직였어?" -> 안움직임 (대기)  => 딴일 못하고 무한 질문 중... (지옥)

 [ 인터럽트 방식: 비동기적 하드웨어 신호 ]
 CPU는 자기 할일(게임 로직 계산) 100% 팽팽 도는 중!
 -> (그때 손이 마우스를 확 움직임!) -> 마우스: "삐용삐용! 1번 비상벨 작동!"
 -> CPU "앗, 마우스 움직였네? 0.1초만 커서 옮기고 마저 하던 겜 해야지."
```

📢 **섹션 요약 비유**: 폴링은 라면 물 올려놓고 '3초마다 뚜껑 열어보고 아직 안 끓었네 닫기'를 반복하는 바보 짓입니다. 인터럽트는 그냥 불 올려놓고 내 방에 들어가서 유튜브를 보다가, '삐익!' 하고 물 끓는 냄비 소리(비상벨)가 날 때만 잠깐 주방에 뛰어나가 라면을 넣는 가장 똑똑한 멀티태스킹입니다.

## Ⅱ. 인터럽트 처리의 완벽한 5단계 마술 (Context Switching)
CPU가 하던 일을 멈추고 비상벨을 껐다고 해서 하던 일을 까먹으면 블루스크린이 뜹니다. 
그래서 CPU는 인터럽트를 맞았을 때 아주 정교한 대피 훈련 매뉴얼(하드웨어적 처리)을 수행합니다.

1. **인터럽트 요청 인지**: 명령어 Execute 단계가 끝날 쯤 핀(Pin)에 전기가 들어왔는지 쓱 확인합니다. "앗, 비상이다!"
2. **현재 문맥(Context) 백업**: 하던 일을 잊지 않기 위해, 현재 프로그램 카운터(PC, 다음 갈 주소)와 계산 중이던 레지스터 값(도마 위 재료)들을 안전한 **메모리 스택(Stack)** 창고 깊숙이 쑤셔 넣습니다(Push).
3. **인터럽트 벡터(Vector) 추적**: "어디서 울린 비상벨이지?" 마우스인지 하드디스크인지 고유 번호(벡터)를 확인합니다.
4. **ISR (인터럽트 서비스 루틴) 실행**: 번호에 맞는 구조대원 코드(ISR)가 있는 메모리 주소로 PC를 점프시켜, 119 출동(마우스 커서 이동 등) 업무를 싹 다 처리합니다.
5. **복귀 (Return from Interrupt)**: 불을 다 껐으면(RETI 명령어), 아까 스택에 숨겨뒀던 하던 일(PC와 레지스터 값)을 다시 꺼내와(Pop), 아무 일도 없었다는 듯이 게임을 마저 합니다.

```ascii
[인터럽트 발생 시 CPU 내부의 시공간적 흐름]

 (메인 프로그램 실행 중) -> 1번, 2번, 3번(여기서 비상벨 쾅!)
                             | 
  +--- (스택에 PC, 레지스터 백업) --+
  |                                 
  v                                 
 [ ISR: 마우스 클릭 처리 코드 ] -> 처리 완료! (RETI 명령어)
  |                                 
  +--- (스택에서 PC 꺼내서 복귀) --+
                             |
                             V
 (메인 프로그램 복귀 완료) -> 4번, 5번 자연스럽게 실행 지속...
```

📢 **섹션 요약 비유**: 인터럽트는 '책 읽다 택배 받기'입니다. 소설책을 150쪽까지 재밌게 읽고 있는데 딩동!(인터럽트) 합니다. 나는 무작정 나가는 게 아니라, 150쪽에 책갈피(PC 스택 백업)를 꽂아두고 나갑니다. 나가서 택배 상자를 까보고(ISR 실행) 다시 방에 돌아와서, 책갈피를 꽂아둔 150쪽부터 다시 읽기 시작(복귀)하면 흐름이 완벽히 이어집니다.

## Ⅲ. 아키텍처 관점: 내/외부 인터럽트와 우선순위(Priority) 전쟁
인터럽트는 단순히 마우스에서만 오는 게 아닙니다. CPU 내부에서도 지옥 같은 인터럽트(소프트웨어 예외, Exception)가 터집니다.
만약 게임 코드가 $1 \div 0$ 을 하라고 시키면, 덧셈기(ALU)는 계산을 못 하고 머리가 터져버립니다. 이때 ALU 자체가 "나 살려줘!"라며 **내부 인터럽트(Exception, Trap)**를 쾅 터뜨립니다. 그러면 OS(윈도우)가 즉각 개입하여 그 불량 게임 프로그램을 강제로 죽여버려(강제 종료) CPU 전체가 멈추는 걸 막습니다.

또한, 컴퓨터에는 비상벨이 너무 많아서 **'우선순위 인코더(Priority Encoder)'**라는 아키텍처 부품이 반드시 필요합니다.
마우스도 비상벨을 누르고, 프린터도 누르고, 시스템 정전 센서도 동시에 비상벨을 누르면 누굴 먼저 살려야 할까요?
당연히 가장 치명적인 **정전(Power Fail) 인터럽트**를 1순위로 받아들여 즉시 하드디스크에 데이터를 백업하고, 마우스 같은 건 좀 천천히 처리하는 식의 완벽한 우선순위 제어가 현대 컴퓨터가 다운되지 않는 숨은 비결입니다.

```ascii
[하드웨어의 인터럽트 우선순위 판별 (우선순위 인코더)]

 동시에 3개의 비상벨 발생!
 - 마우스 클릭 (우선순위 10)
 - 하드디스크 다운로드 완료 (우선순위 5)
 - 시스템 전원 차단 임박!!! (우선순위 0, 최강)

 [ 우선순위 인코더 회로 ] -> "10번, 5번 비켜!! 0번(전원차단) 인터럽트만 지금 당장 CPU로 쏴!!!"
```

📢 **섹션 요약 비유**: 인터럽트 우선순위는 병원 응급실의 '환자 분류(Triage)'입니다. 감기 환자(마우스)와 배탈 환자(프린터)가 와서 의사(CPU)를 불러도, 심장이 멈춘 환자(전원 차단)가 들어오는 순간 의사는 감기 환자 팽개치고 무조건 심장 마비 환자한테 먼저 달려가야만 병원 시스템(컴퓨터)이 무너지지 않습니다.

## 개념 맵 (Concept Map)
```ascii
[인터럽트 (Interrupt)]
   │
   ├── 정의: H/W나 S/W의 예기치 않은 비동기적 이벤트 발생 시, CPU의 정상 실행 흐름을 일시 중단하고 해당 루틴을 우선 처리하는 메커니즘
   │     └── 폴링(Polling) 방식의 막대한 CPU 클럭 낭비 한계를 극복하기 위해 탄생
   │
   ├── 인터럽트의 3대 분류
   │     ├── 하드웨어 (외부): I/O 입출력 완료, 타이머(Timer) 만료, 정전(Power Fail)
   │     ├── 소프트웨어 (내부 Exception/Trap): 0으로 나누기, 존재하지 않는 메모리 침범(Page Fault)
   │     └── 시스템 콜 (SVC): 사용자 프로그램이 OS의 특권 명령(파일 읽기 등)을 요구할 때 의도적 발생
   │
   └── 아키텍처 처리 사이클 (Context Switching)
         └── 비상벨 감지 -> 현재 PC/상태 레지스터 Stack에 백업 -> 우선순위 판별(Vector) -> ISR(처리 루틴) 점프 및 실행 -> 원상 복구(RETI)
```
