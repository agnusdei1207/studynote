## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 로드/스토어 아키텍처(Load/Store Architecture)는 명령어의 역할을 철저히 분리하여, **메모리에 접근하는 명령어는 오직 `LOAD`(읽기)와 `STORE`(쓰기)로만 제한**하는 설계 철학이다.
> 2. **가치**: 더하기(`ADD`), 곱하기(`MUL`) 같은 모든 연산 명령어는 오직 레지스터(Register) 내부에서만 수행되도록 강제함으로써, 연산 파이프라인에서 예측 불가능한 **메모리 지연(Memory Latency)을 완벽히 격리**시킨다.
> 3. **융합**: 이 엄격한 분업 시스템은 명령어 길이 고정(Fixed-length), 풍부한 레지스터 파일(Large Register File)과 함께 **RISC 아키텍처를 떠받치는 3대 기둥** 중 하나로 융합되어 현대 CPU 설계의 기본 규칙이 되었다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 로드/스토어 아키텍처(Load/Store Architecture)는 명령어의 역할을 철저히 분리하여, **메모리에 접근하는 명령어는 오직 로드(LOAD)와 스토어(STORE)로만 제한**하는 설계 철학이다. 산술 및 논리 연산 명령어는 오직 CPU 내부의 레지스터만을 연산 대상으로 삼도록 강제하는 구조다.

- **필요성**: 로드/스토어 아키텍처는 현대 고성능 CPU의 핵심 기술인 **'명령어 파이프라이닝(Pipelining)'의 효율을 극대화**하기 위해 반드시 필요하다. 과거 CISC처럼 연산 명령어가 메모리에 직접 접근하게 되면, 상대적으로 매우 느린 메모리 응답 속도(Latency)로 인해 연산 장치(ALU)가 멍하니 대기하는 병목 현상이 발생하여 전체 시스템 속도를 갉아먹기 때문이다. 로드/스토어 구조는 **연산과 메모리 접근 사이의 시간적·물리적 경계를 확립**함으로써, 컴파일러가 느린 메모리 명령어(LOAD)를 연산보다 훨씬 앞쪽에 미리 배치하여 지연 시간을 숨기는 **지연 은닉(Latency Hiding)** 최적화를 가능케 한다. 이는 GHz급 초고속 연산을 수행하는 RISC 계열(ARM, RISC-V) 프로세서가 단 1클럭의 정체 없이 데이터를 처리하게 만드는 핵심 설계 규율이다.
- **💡 비유**: 로드/스토어 아키텍처는 '수술실의 의사와 간호사'와 같다. 집도의(ALU)는 메스(데이터)를 찾으러 찬장(메모리)을 뒤지지 않는다. 간호사(LOAD 명령어)가 수술 도구를 쟁반(레지스터)에 올려주면, 의사는 쟁반 위의 도구만 가지고 가장 빠르고 정확하게 수술(연산)만 집도하는 완벽한 분업 시스템이다.
- **등장 배경**: 
  초창기 CISC 프로세서는 `ADD [메모리], [메모리]` 처럼 연산기가 직접 메모리를 건드렸다. 문제는 메모리가 CPU보다 수백 배 느리다는 것이었다. 덧셈을 하던 연산기가 메모리에서 데이터가 올 때까지 멍하니 서서 전체 파이프라인을 멈춰 세웠다. 버클리 대학의 RISC 연구진은 이 병목을 보고 분노하며 선언했다. **"연산기는 절대 메모리를 만져선 안 된다! 메모리 접근과 연산을 법으로 분리하라!"** 이렇게 로드/스토어 아키텍처가 탄생했다.

메모리 접근(느림)과 연산(빠름)이 철저하게 분리된 로드/스토어의 메커니즘을 시각화하면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────────────────┐
  │         로드/스토어 아키텍처(Load/Store)의 철저한 분업 체계               │
  ├───────────────────────────────────────────────────────────────────────────┤
  │                                                                           │
  │   [ 1단계: 재료 준비 (LOAD) ] - 메모리 전담반                             │
  │     LOAD R1, [1000]  ──(느림)──▶ 메모리 1000번지에서 값 10을 가져옴       │
  │     LOAD R2, [2000]  ──(느림)──▶ 메모리 2000번지에서 값 20을 가져옴       │
  │                                                                           │
  │   [ 2단계: 초고속 연산 (ALU) ] - 연산 전담반                              │
  │     ADD  R3, R1, R2  ──(빛의속도)─▶ R1(10) + R2(20) = R3(30)              │
  │     (이 순간 CPU는 메모리가 세상에 존재하는지조차 모른다!)                │
  │                                                                           │
  │   [ 3단계: 결과 보관 (STORE) ] - 메모리 전담반                            │
  │     STORE [3000], R3 ──(느림)──▶ R3의 값 30을 메모리 3000번지에 씀        │
  └───────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 로드/스토어 아키텍처의 위대함은 **'격리(Isolation)'**에 있다. 느린 작업(메모리 접근)과 빠른 작업(ALU 연산)을 명령어 레벨에서 찢어버렸다. 얼핏 보면 명령어 개수가 늘어나서 손해 같지만, 이렇게 분리해 놓으면 똑똑한 컴파일러가 LOAD 명령어를 연산보다 훨씬 앞쪽에 미리 배치(Instruction Scheduling)하여 메모리 지연 시간(Latency)을 완벽하게 숨길 수 있다.

- **📢 섹션 요약 비유**: 로드/스토어 방식은 '스타크래프트의 미네랄 캐기와 유닛 생산'의 분리와 같습니다. 일꾼(LOAD)은 미네랄(메모리)만 열심히 캐서 본진(레지스터)에 쌓아두고, 배럭(ALU)은 미네랄 밭에 갈 필요 없이 본진에 쌓인 자원만 써서 유닛(연산 결과)을 광속으로 뽑아내는 완벽한 분업입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (분업을 지탱하는 하드웨어 설계)

이 아키텍처가 성립하려면 물리적인 하드웨어가 뒷받침되어야 한다.

| 구성 요소 | 물리적 역할 | 아키텍처적 가치 | 비유 |
|:---|:---|:---|:---|
| **Large Register File** | 데이터를 임시 보관하는 넉넉한 공간 | **메모리 왕복 횟수를 줄이는 버퍼** | 아주 넓은 도마 |
| **Load/Store Unit (LSU)**| 메모리 주소 계산 및 버스 통신 전담| **ALU와 분리된 독립적인 메모리 접근기**| 배달 전담 기사 |
| **ALU (Arithmetic Logic)**| 레지스터 값만 읽고 쓰는 연산기 | **메모리 병목에서 해방된 1-Cycle 엔진** | 요리 전담 셰프 |
| **Instruction Scheduler**| LOAD와 ADD 사이의 거리를 벌림 | **메모리 레이턴시(지연 시간) 은닉** | 똑똑한 작업 반장 |

---

### 심층 동작 원리: "지연 은닉(Latency Hiding)의 컴파일러 마술"

왜 명령어를 분리해야만 속도를 올릴 수 있는지 보여주는 핵심 원리다.

```text
  ┌───────────────────────────────────────────────────────────────────────────┐
  │         로드/스토어 분리가 만들어낸 지연 은닉 (Latency Hiding)            │
  ├───────────────────────────────────────────────────────────────────────────┤
  │                                                                           │
  │   [ 멍청한 코드 (Stall 발생) ]                                            │
  │     1. LOAD R1, [MEM]                                                     │
  │     2. ADD  R2, R1, R1  ◀─ (R1이 도착할 때까지 파이프라인 멈춤!)          │
  │     3. SUB  R4, R5, R6                                                    │
  │                                                                           │
  │   [ 컴파일러의 재배치 (Instruction Scheduling 융합) ]                     │
  │     1. LOAD R1, [MEM]   ◀─ (데이터 주문해 놓고...)                        │
  │     2. SUB  R4, R5, R6  ◀─ (상관없는 SUB를 먼저 실행해버림!)              │
  │     3. ADD  R2, R1, R1  ◀─ (SUB 하는 동안 R1 도착! 멈춤 없이 덧셈!)       │
  │                                                                           │
  │ * 혁명: 명령어를 분리해 놓으니, 그 사이의 빈 시간에 다른 일을 할 수 있다! │
  └───────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 만약 CISC처럼 `ADD R2, [MEM]` 한 줄로 되어 있었다면, 이 명령어 전체가 꼼짝없이 메모리를 기다려야 한다. 중간에 다른 명령어를 끼워 넣을 틈이 없다. 하지만 로드/스토어 아키텍처는 이를 `LOAD`와 `ADD` 두 개의 레고 블록으로 쪼개 놓았다. 똑똑한 컴파일러는 이 두 블록 사이의 간격을 쫙 벌리고, 그 사이에 메모리 데이터가 필요 없는 다른 연산(SUB)들을 끼워 넣는다. 결과적으로 메모리를 기다리는 시간(지연)이 다른 연산을 하는 시간으로 완벽히 덮어지는(Hiding) 마술이 일어난다.

- **📢 섹션 요약 비유**: 지연 은닉은 '라면 끓이면서 계란 풀기'와 같습니다. 물을 끓이는(LOAD) 데는 3분이 걸리는데, 그 3분 동안 멍하니 서 있는 게 아니라, 상관없는 파를 썰고 계란을 푸는 작업(다른 연산)을 먼저 해버리면 버리는 시간 없이 완벽한 요리가 완성되는 이치입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 기술 비교: Register-Memory(CISC) vs Load-Store(RISC)

메모리를 다루는 철학의 근본적인 차이다.

| 비교 항목 | Register-Memory 구조 (CISC) | Load-Store 구조 (RISC) | 아키텍처 판단 포인트 |
|:---|:---|:---|:---|
| **메모리 연산 허용** | **가능 (`ADD R1, [M]`)** | **불가능 (문법 오류)** | ALU의 메모리 버스 직결 여부 |
| **명령어 종류** | 무수히 많음 (모든 연산마다 M 지원) | 아주 적음 (M 접근은 오직 2개뿐) | 디코더 설계의 단순화 |
| **메모리 주소 모드** | 연산 명령어 안에 복잡하게 얽힘 | 오직 LOAD/STORE 명령어에만 존재 | 주소 계산(AGU)의 독립성 |
| **파이프라인 충돌** | 예측 불가능한 메모리 지연으로 잦음 | 메모리 병목이 특정 단계(MEM)에 국한됨 | 해저드(Hazard) 통제력 |
| **비유** | 서재에서 바로 밥 먹기 | **식당(레지스터)으로 밥상 차려오기** | 공간의 용도 분리 |

### 과목 융합 관점

- **운영체제 (Memory-Mapped I/O, MMIO)**: 로드/스토어 아키텍처에서는 메모리를 읽고 쓰는 행위가 곧 외부 장치(I/O)와 통신하는 행위와 완전히 일치하도록 융합된다. OS는 비디오 카드나 랜카드의 버퍼를 특정 메모리 주소(예: 0x8000)에 매핑해 둔다. 그러면 CPU는 특별한 `IN/OUT` 명령어 없이, 그저 평범한 **`STORE R1, [0x8000]`** 명령어 하나로 화면에 점을 찍고 네트워크 패킷을 날릴 수 있다. 이 우아함이 현대 임베디드 제어의 근간이다.
- **캐시 메모리 (Cache Hierarchy)**: 로드/스토어 아키텍처는 캐시 메모리의 효율을 100% 믿고 설계된 구조다. 만약 매번 LOAD/STORE 할 때마다 메인 메모리(DRAM)까지 가야 한다면 이 구조는 망했을 것이다. 하지만 L1 Data Cache가 CPU 바로 옆에 융합되면서, LOAD 명령은 95% 확률로 1~2 Cycle 만에 캐시에서 데이터를 낚아채 오게 되었고, 로드/스토어는 가장 완벽한 설계로 증명되었다.

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │         마이크로아키텍처의 융합: 슈퍼스칼라(Superscalar)의 완성              │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │   [ 로드/스토어 분리가 가져온 병렬 처리의 축복 ]                             │
  │                                                                              │
  │   ALU 파이프라인 1 ─▶ ADD R3, R1, R2  (레지스터만 쓴다! 충돌 없음)           │
  │   ALU 파이프라인 2 ─▶ SUB R6, R4, R5  (레지스터만 쓴다! 충돌 없음)           │
  │   LSU 파이프라인   ─▶ LOAD R7, [MEM]  (메모리만 쓴다! 충돌 없음)             │
  │                                                                              │
  │ * 혁명: 역할이 완벽히 쪼개져 있어서, 한 클럭에 3개의 명령어를 무작정         │
  │   때려 넣어도 하드웨어가 꼬이지 않고 동시에 실행된다!                        │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 인텔 펜티엄 시대 이후 CPU 속도가 비약적으로 뛴 이유는 '여러 명령어를 동시에 실행(Superscalar)'했기 때문이다. CISC처럼 연산기가 메모리도 건드리고 덧셈도 하면, 한 클럭에 여러 명령어를 넣었을 때 서로 버스(Bus)를 차지하겠다고 싸우다 칩이 뻗어버린다. 로드/스토어 아키텍처는 '수학 계산팀(ALU)'과 '물건 운반팀(LSU, Load/Store Unit)'을 물리적으로 완전히 쪼개놨기 때문에, 각 팀에 동시에 명령을 내려도 아무런 충돌 없이 100%의 병렬 처리 효율을 뽑아낼 수 있다.

- **📢 섹션 요약 비유**: 부서의 완벽한 분리는 '분업화된 대형 병원'과 같습니다. 원무과(LSU)는 환자 접수(Load)만 미친 듯이 받고, 수술실(ALU)은 수술만 미친 듯이 합니다. 원무과 직원이 메스를 들거나 의사가 접수를 받지 않기 때문에, 병원 전체가 동시에 엄청난 수의 환자를 처리(슈퍼스칼라)할 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — GPU 셰이더(Shader) 프로그래밍의 텍스처 패치**: 
  **상황**: 3D 그래픽을 렌더링할 때 수백만 개의 픽셀이 동시에 이미지(텍스처) 메모리를 읽어와 색깔을 계산해야 함.
  **판단**: "극단적인 로드/스토어 분리와 벡터화(SIMD)"다. GPU 코어는 메모리(텍스처)에서 데이터를 읽어오는 `TEX` (Load 류) 명령어 그룹과, 그 색깔을 섞는 `FMA` (연산 류) 명령어를 완전히 분리한다. 그리고 `TEX` 명령어를 먼저 수천 개 던져놓고 메모리 지연(수백 클럭)이 발생하는 동안, 이미 데이터가 로드된 다른 픽셀들의 `FMA` 연산을 미친 듯이 덮어씌워 렌더링 지연을 0으로 만들어버린다.

2. **시나리오 — 임베디드 C 프로그래머의 `volatile` 키워드 딜레마**:
  **상황**: 스위치를 누를 때마다 메모리 특정 번지(0x100)의 값이 바뀌는데, `while (A == 0)` 루프를 돌려도 프로그램이 스위치 입력을 못 알아챔.
  **판단**: "로드/스토어 컴파일러 최적화의 역풍"이다. 컴파일러가 보기에 `A`의 값은 루프 안에서 변하지 않는다. 그래서 최초에 딱 한 번만 `LOAD A`를 해서 레지스터에 박아두고, 이후엔 메모리를 쳐다보지도 않고 레지스터 안에서만 무한 루프(연산)를 돈 것이다. 프로그래머는 변수 `A`에 **`volatile`**을 선언하여, "이 값은 레지스터에 캐싱하지 말고, 조건 검사할 때마다 무조건 밖으로 나가서 **`LOAD` 명령어를 다시 실행해라!**"라고 하드웨어에 직격탄을 날려야 한다.

```text
  ┌───────────────────────────────────────────────────────────────────────────┐
  │         마이크로아키텍처 합성: 메모리 배리어(Memory Barrier)의 도입       │
  ├───────────────────────────────────────────────────────────────────────────┤
  │                                                                           │
  │   [ Out-of-Order(비순차 실행) 환경에서 Load/Store가 꼬이는 문제 ]         │
  │                                                                           │
  │   명령어 1: STORE [A], 100  (메모리 A에 100 쓰기 - 엄청 느림)             │
  │   명령어 2: LOAD R1, [B]    (메모리 B에서 읽기 - 캐시 덕에 빠름)          │
  │                                                                           │
  │   CPU의 반란: "어? 둘이 주소가 다르네? 그럼 빠른 2번부터 먼저 실행할게!"  │
  │   결과: 멀티코어 환경에서 데이터 동기화가 완전히 박살 나버림!             │
  │                                                                           │
  │  최종 조치: 순서가 뒤집히면 안 되는 곳에 '메모리 배리어(DMB)'를 융합하라! │
  └───────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 로드/스토어 아키텍처가 비순차 실행(OoO)과 만났을 때 벌어지는 참사다. 성능에 미친 현대 CPU는 서로 주소가 다른 LOAD와 STORE를 만나면, 프로그래머가 쓴 순서를 개무시하고 캐시에 있는 빨리 끝나는 놈부터 먼저 실행해 버린다(Memory Reordering). 싱글 코어에선 문제가 없지만, 다른 코어가 이 메모리를 지켜보고 있는 멀티스레드 환경에서는 치명적인 버그가 발생한다. 이를 막기 위해 아키텍트들은 **`DMB` (Data Memory Barrier)** 같은 특수 명령어를 융합하여, "이 선을 넘기 전까지는 LOAD/STORE 순서를 절대 바꾸지 마라!"라고 CPU의 목줄을 강제로 죈다.

### 도입 체크리스트
- **Load-Use Delay 방어**: LOAD 직후에 그 데이터를 바로 쓰는 명령어(Use)가 올 때, 파이프라인 스톨을 막기 위해 데이터를 레지스터 파일에 쓰기도 전에 ALU 입력단으로 바로 쏴주는 **포워딩(Data Forwarding/Bypassing)** 회로가 완벽히 배선되어 있는가?
- **Store Buffer**: STORE 명령이 캐시나 메모리에 진짜로 써질 때까지 파이프라인이 기다리면 너무 느리다. 그래서 CPU 코어 바로 옆에 값과 주소를 임시로 던져놓고 파이프라인은 곧바로 다음 작업을 하러 떠날 수 있게 해주는 **스토어 버퍼(Store Buffer)**가 융합되었는가?

### 안티패턴
- **구조체 전체 복사(Struct Assignment)의 무분별한 남용**: C언어에서 큰 구조체를 `A = B;` 로 복사할 때, 겉으로는 한 줄이지만 하드웨어 레벨에서는 구조체 크기만큼의 수많은 `LOAD`와 `STORE` 짝꿍들이 미친 듯이 생성된다. 포인터를 넘기지 않고 값 자체를 복사하면, 연산(ALU)은 한 번도 안 하고 메모리 대역폭만 갉아먹는 최악의 '메모리 트래픽 잼' 안티패턴이 발생한다.

- **📢 섹션 요약 비유**: 구조체 전체 복사는 '이사할 때 집 통째로 옮기기'와 같습니다. 그냥 그 집 주소(포인터)만 적힌 종이 한 장을 친구한테 주면 끝날 일을, 굳이 트럭 수백 대(수많은 Load/Store)를 동원해서 벽돌 하나하나를 다 복사해서 옮겨주는 멍청하고 기운 빠지는 짓입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 레지스터-메모리 혼재 구조 | 철저한 로드/스토어 분리 구조 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 연산 시 메모리 접근 대기 (수십 Cycle) | **연산은 1 Cycle 무조건 보장** | CPU 클럭 주파수(MHz) **수 배 향상** |
| **정량** | 명령어 길이 다양, 해독기 복잡 | **LOAD/STORE 전용 포맷 고정** | 디코더 트랜지스터 면적 **대폭 절감** |
| **정성** | 컴파일러가 메모리 지연 예측 불가 | 메모리 명령어를 앞으로 당겨 최적화 | 파이프라인 **지연 은닉(Latency Hiding)** 완벽 달성 |

### 미래 전망
- **PIM (Processing-In-Memory)의 역습**: 로드/스토어는 "메모리가 느리니까 밖으로 빼자"는 철학이다. 하지만 AI 시대를 맞아 데이터가 기하급수적으로 커지자, CPU로 데이터를 가져오는(LOAD) 것 자체가 한계에 달했다. 결국 **메모리(DRAM) 칩 자체에 작은 연산기(ALU)를 박아 넣어** LOAD 없이 메모리 안에서 바로 계산해 버리는 PIM 기술이 차세대 슈퍼컴퓨터의 표준으로 융합되고 있다. (역사의 나선형 반복이다.)
- **Load/Store 명령어의 벡터화 (Scatter/Gather)**: 딥러닝 데이터를 위해 한 번의 LOAD 명령어로 흩어져 있는 메모리 수십 군데의 데이터를 긁어모아(Gather) 거대한 벡터 레지스터에 한 방에 꽂아 넣고, 연산 후 다시 여러 군데로 흩뿌리는(Scatter) 고차원 LSU(Load Store Unit)가 핵심 경쟁력이 되었다.

### 참고 표준
- **ARM / MIPS / SPARC / RISC-V ISA**: 메모리를 다루는 명령어는 오직 LDR(Load Register)과 STR(Store Register) 두 종류의 변형뿐이라는 원칙을 헌법처럼 지키는 순수 RISC 규격들.
- **C/C++ Pointer Dereferencing**: `*ptr` 연산자가 기계어로 번역될 때, 한 치의 오차도 없이 가장 순수한 형태의 `LOAD` 기계어 명령어 하나로 1:1 매핑되는 소프트웨어-하드웨어 간의 완벽한 조화.

"계산하는 자"와 "운반하는 자"의 철저한 계급 분리를 통해 파이프라인의 영원한 평화를 이룩한, 현대 아키텍처의 십계명 '로드/스토어 구조'의 진화 로드맵은 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │         철저한 분업: 로드/스토어(Load/Store) 진화 로드맵                                            │
  ├─────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │                                                                                                     │
  │   [1단계: 혼돈의 메모리 연산]   [2단계: 성역의 분리 (RISC)]  [3단계: 지연 은닉과 비순차 실행]       │
  │                                                                                                     │
  │   연산기가 메모리에 손을 댐 ──▶ 연산과 메모리 접근의 강제 분리 ──▶ OoO 융합 및 메모리 배리어        │
  │  (느린 놈 때문에 다 같이 죽음)  (넌 요리만 해! 난 배달만 할게)  (빨리 온 놈부터 순서 무시하고 먹어!)│
  │ "더하기 하는데 왜 이리 늦어!" "재료 올 때까지 딴 거 썰고 있어" "순서 꼬이면 죽는다, 배리어 쳐!"     │
  └─────────────────────────────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 로드/스토어는 컴퓨터 공학의 '포드 자동차 조립 라인' 혁명이다. 1단계: 장인이 혼자서 재료도 가져오고 부품도 깎고 조립도 다 하던 시절(CISC)은 느렸다. 2단계: 이를 재료 나르는 놈(LOAD/STORE)과 조립하는 놈(ALU)으로 분업화(RISC)하자 생산량이 폭발했다. 3단계: 현대에 이르러서는 나르는 놈들이 교통체증(Cache Miss)에 걸릴 것을 대비해, 조립하는 놈이 순서를 무시하고(Out-of-Order) 손에 잡히는 것부터 먼저 조립하게 만들었다. 이 과정에서 순서가 꼬이는 걸 막는 안전장치(Memory Barrier)까지 융합하며, 로드/스토어는 가장 완벽한 형태의 병렬 실행(Superscalar) 아키텍처로 진화했다.

- **📢 섹션 요약 비유**: 로드/스토어 아키텍처는 '수술실의 분업'과 같습니다. 옛날(CISC)엔 의사가 직접 약국(메모리)에 가서 약을 가져와 수술(연산)하느라 환자가 죽어 나갔습니다. 지금(RISC)은 간호사(LOAD)가 약을 카트(레지스터)에 미리 완벽하게 세팅해 두면, 의사는 절대 수술실 밖을 나가지 않고 카트 위의 약만 써서 수술을 빛의 속도로 끝내는 최고의 생명 연장 시스템입니다.

3. 이렇게 완벽하게 역할을 나누니까, 요리사는 재료 찾는 시간 낭비 없이 **도마 위에 있는 것만 썰면서 세상에서 제일 빨리 요리(계산)**를 끝낼 수 있답니다!
