# 3-주소 명령어 (3-Address Instruction)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 3-주소 명령어는 명령어 내에 **두 개의 입력(Source) 피연산자와 한 개의 출력(Destination) 피연산자 주소를 모두 명시**하는 가장 완전한 형태의 명령어 구조다.
> 2. **가치**: 연산 후에도 입력된 원본 데이터가 파괴되지 않는 **비파괴적 연산(Non-destructive Operation)**을 수행하여 컴파일러의 레지스터 최적화를 극대화한다.
> 3. **융합**: 명령어 길이가 일정하게 고정된 **RISC (Reduced Instruction Set Computer)** 아키텍처 및 로드/스토어(Load/Store) 구조와 완벽하게 결합하여 현대 모바일/서버 프로세서(ARM, RISC-V)의 글로벌 표준이 되었다.

+++

## Ⅰ. 3-주소 명령어의 개요

### 1. 정의
- `[ Opcode | Dest | Source 1 | Source 2 ]` 형태로 구성되며, Source 1과 Source 2를 연산한 결과를 온전히 독립된 Dest 주소에 저장하는 방식이다. (예: `ADD R1, R2, R3` $\to$ `R1 = R2 + R3`)

### 2. 등장 배경: '원본 데이터 보존의 열망'
- 2-주소 체계에서는 연산 결과가 원본 데이터 하나를 덮어써 버린다. 만약 그 원본 데이터가 다음 수식에서 또 필요하다면, 덮어쓰기 전에 다른 레지스터로 복사해두는(`MOV`) 낭비가 발생했다. 메모리 용량이 넉넉해지고 32비트 고정 길이 명령어가 표준이 되면서, "여유 공간에 목적지 주소까지 3개를 다 적어서 원본을 살리자"는 3-주소 방식이 탄생했다.

### 3. 💡 비유: '완벽한 분업 요리'
- 3-주소 명령어는 **'A컵(시럽)과 B컵(커피)의 내용물을 완전히 새로운 C컵(결과물)에 붓고 섞는 것'**과 같다.
- A컵과 B컵은 원래 모습을 그대로 유지하므로, 나중에 시럽이나 커피가 더 필요하면 언제든 다시 쓸 수 있다.

- **📢 섹션 요약 비유**: 원본 서류에 낙서하지 않고, 복사기(ALU)를 통해 깨끗한 새 종이(Dest)에 결과를 출력해 내는 가장 안전하고 스마트한 문서 작업입니다.

+++

## Ⅱ. 메커니즘 및 동작 원리 (Deep Dive)

모든 피연산자가 독립적으로 지정되므로 ALU의 구조와 완벽히 매핑된다.

### 1. 명령어 구조 및 기본 동작
- **명령어 형식**: `[ Opcode | Dest R | Src R1 | Src R2 ]`
- **산술 연산**: `ADD R1, R2, R3` $\to$ `R1 <- R2 + R3`
- **비파괴적 특성**: `R2`와 `R3`의 값은 연산 전후로 절대 변하지 않는다.

### 2. 동작 프로세스 시각화: $Y = (A + B) \times C$ 연산 (ASCII)
```text
    [ High-Level Code: Y = (A + B) * C ]
    [ 가정: A는 R1, B는 R2, C는 R3에 로드되어 있음 ]
    
    1. ADD R4, R1, R2     ( R4 <- R1 + R2 )
       * R1(A)와 R2(B)는 그대로 살아있음. 
       * 중간 결과는 새 레지스터 R4에 저장됨.
       
    2. MUL R5, R4, R3     ( R5 <- R4 * R3 )
       * R4(A+B)와 R3(C)를 곱해 최종 결과 R5에 저장.
       
    3. STORE Y, R5        ( M[Y] <- R5 )
    
    * 단 2개의 산술 명령어만으로 복잡한 수식이 깔끔하게 해결됨. (MOV 불필요)
```

### 3. 심층 분석: 명령어 길이(Code Density) 이슈
- 주소를 3개나 적어야 하므로, 만약 32개의 범용 레지스터를 쓴다면 `Opcode(8bit) + Dest(5bit) + Src1(5bit) + Src2(5bit) = 23bit`가 기본적으로 소모된다. 메모리를 직접 가리킨다면 명령어 길이가 64비트를 훌쩍 넘어가 버린다. 따라서 3-주소 명령어는 메모리를 직접 참조하지 못하게 막고 오직 레지스터 간의 연산만 허용하는 **로드/스토어 아키텍처**와 필연적으로 결합하게 되었다.

- **📢 섹션 요약 비유**: 주소를 3개나 적으려면 우편엽서 공간이 모자라니, 아주 짧은 동네 이름(레지스터 번호)만 쓰도록 규칙을 강력하게 제한한 셈입니다.

## Ⅲ. 2-주소 vs 3-주소 어셈블리 코드 비교

수식: `X = (A + B) * (A + C)`
(A, B, C가 각각 R1, R2, R3에 있다고 가정)

| 구분 | 2-주소 명령어 (CISC 스타일) | 3-주소 명령어 (RISC 스타일) |
|:---|:---|:---|
| **코드** | `MOV R4, R1` (R4에 A 복사)<br>`ADD R4, R2` (R4 = A+B)<br>`MOV R5, R1` (R5에 A 복사)<br>`ADD R5, R3` (R5 = A+C)<br>`MUL R4, R5` (R4 = R4 * R5) | `ADD R4, R1, R2` (R4 = A+B)<br>`ADD R5, R1, R3` (R5 = A+C)<br>`MUL R6, R4, R5` (R6 = R4 * R5) |
| **명령어 수** | **5 줄** (데이터 보존용 MOV 발생) | **3 줄** (수학적 논리와 완벽 일치) |
| **특징** | 원본 보존을 위해 프로그래머가 고생함 | 컴파일러가 코드를 생성하기 매우 쉬움 |

- **📢 섹션 요약 비유**: 2-주소는 공간이 좁아 퍼즐 맞추듯 짐을 옮겨야 하지만, 3-주소는 공간이 넓어 수식 그대로 직관적으로 풀어내는 마법입니다.

+++

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: ARM 아키텍처 기반 모바일 프로세서 컴파일
- **상황**: 스마트폰용 iOS/Android 앱의 C/C++ 코드 컴파일.
- **판단**: ARM은 철저한 3-주소 체계의 RISC 아키텍처다. LLVM/GCC 컴파일러는 수식을 파싱하여 추상 구문 트리(AST)를 만든 뒤, 이를 3-주소 중간 언어(Three-Address Code, TAC)로 변환한다. 이 TAC는 ARM의 기계어와 1:1로 매핑되므로, 레지스터 병렬 할당과 명령어 스케줄링(Instruction Scheduling) 최적화가 극도로 효율적으로 이루어져 모바일 기기의 배터리 타임을 아낀다.

### 2. 기술사적 인사이트: 파이프라인 데이터 해저드 (RAW) 회피
- **인사이트**: 3-주소 명령어는 매 연산마다 새로운 Destination 레지스터(R4, R5 등)를 쉽게 할당할 수 있다. 이는 명령어 간의 가짜 의존성(WAW, WAR)을 원천적으로 차단하고, 컴파일러가 루프 언롤링이나 소프트웨어 파이프라이닝을 통해 명령어들을 순서에 구애받지 않고 마구 뒤섞어(Out-of-Order) 하드웨어의 병렬 실행 유닛을 100% 혹사시킬 수 있게 하는 아키텍처적 승리다.

- **📢 섹션 요약 비유**: 각자 독립된 작업대(독립된 Dest 레지스터)를 배정받아, 다른 사람 눈치 보지 않고 내 작업만 최고 속도로 진행하는 완벽한 공장 라인입니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **소프트웨어 생산성 향상**: 컴파일러 제작이 쉬워지고 기계어 매핑 속도가 단축.
- **ILP(명령어 수준 병렬성) 극대화**: 비파괴적 연산으로 인해 최신 슈퍼스칼라 CPU의 다중 ALU 동시 가동에 최적.

### 2. 미래 전망
스마트폰(ARM), 슈퍼컴퓨터 및 데이터센터(AWS Graviton), 그리고 차세대 오픈소스 반도체인 RISC-V에 이르기까지, 현대의 모든 '새로운' 프로세서는 **고정 길이 3-주소 명령어 + 로드/스토어 아키텍처**라는 공식을 따르고 있다. 이것은 지난 반세기 동안 인류가 찾아낸 '실리콘 칩 위에서 수학을 가장 우아하고 효율적으로 계산하는 궁극의 해답'이며, 앞으로도 ISA 설계의 절대적인 교과서로 군림할 것이다.

- **📢 섹션 요약 비유**: 모래성을 쌓고 부수며 고민하던 인류가 찾아낸, 가장 견고하고 흔들리지 않는 레고 블록 조립의 정답입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **RISC (Reduced Instruction Set Computer)**: 3-주소 명령어를 표준으로 채택한 아키텍처 철학.
- **로드/스토어 아키텍처 (Load/Store)**: 3-주소 방식의 명령어 길이 폭발을 막기 위한 단짝 기술.
- **2-주소 명령어**: 원본을 덮어쓰는 구시대적(하지만 여전히 널리 쓰이는) 라이벌 아키텍처.
