# 직접 주소 지정 (Direct Addressing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 직접 주소 지정(Direct Addressing)은 명령어의 피연산자 필드에 **데이터가 저장된 실제 메모리 주소(절대 주소)**를 직접 명시하여 접근하는 가장 직관적인 방식이다.
> 2. **가치**: 추가적인 주소 연산 과정 없이 한 번의 메모리 참조만으로 데이터를 가져올 수 있어 로직이 단순하며, 전역 변수(Global Variable)나 고정된 장치 메모리 접근에 주로 사용된다.
> 3. **융합**: 명령어의 길이가 가변적인 CISC 아키텍처에서 넓은 주소 범위를 커버하기 위해 주로 활용되며, 고정 길이의 RISC에서는 주소 필드의 크기 제한으로 인해 제한적으로 사용된다.

---

## Ⅰ. 직접 주소 지정의 개요

### 1. 정의
- 명령어 안에 "메모리 $1000$번지에 있는 값을 가져와라"와 같이 실제 목적지 주소가 그대로 박혀 있는 방식이다.

### 2. 등장 배경: '명확한 목적지 설정'
- 초창기 프로그래밍에서는 데이터가 위치한 곳을 정확히 알고 접근하는 것이 가장 중요했다. 복잡한 계산 없이 주소 버스에 명령어 속 값을 그대로 실어 보내기만 하면 되는 '단순함' 덕분에 하드웨어 설계가 용이해졌고, 초기 컴퓨터 아키텍처의 표준적인 메모리 접근 모델이 되었다.

### 3. 💡 비유: '지정석 주차장'
- 직접 주소 지정은 **'주차장에 갈 때 내 차가 정확히 107번 구역에 있다는 것을 알고, 곧장 그 칸으로 찾아가는 것'**과 같다.
- 경비실에 물어보거나(간접), 내비게이션을 켤(레지스터) 필요 없이 주소표(명령어)에 적힌 번호 그대로 찾아가면 된다.

- **📢 섹션 요약 비유**: 지도를 뒤적일 필요 없이 "서울시 강남구..."라는 주소를 보고 바로 목적지로 달려가는 직구 방식입니다.

---

## Ⅱ. 메커니즘 및 동작 원리 (Deep Dive)

제어 장치가 명령어로부터 주소를 추출하여 즉시 MAR로 보낸다.

### 1. 명령어 구조 및 유효 주소 (EA)
- **명령어 형식**: `[ Opcode | Address A ]`
- **유효 주소 (EA)**: $EA = A$
- **동작**: `AC <- Memory[A]` (메모리 A번지의 내용을 누산기에 적재)

### 2. 동작 프로세스 시각화 (ASCII)
```text
    [ CPU Internal ]                     [ Main Memory ]
    ┌───────────────────────────┐        ┌───────────────────┐
    │ Instruction Reg (IR):     │        │                   │
    │ [ LOAD | 0x1000 ]         │        │ 0x0FFF: ....      │
    └─────────────┬─────────────┘        ├───────────────────┤
                  │                      │ 0x1000:  DATA     │◀─┐
                  ▼                      ├───────────────────┤  │
    ┌───────────────────────────┐        │ 0x1001: ....      │  │
    │ Address Bus Controller    │        └───────────────────┘  │
    │ (MAR <- 0x1000)           │───────────────┐               │
    └───────────────────────────┘               │               │
                                                ▼               │
                                         [ Memory Read ]        │
                                                │               │
                                                └───────────────┘
    
    * 메모리 접근 횟수: 1회 (Data Fetch 기준)
```

### 3. 결정적 한계: '주소 공간의 제약'
- 명령어 전체 길이가 32비트이고 Opcode가 8비트라면, 주소 필드는 24비트밖에 남지 않는다. 이 경우 직접 가리킬 수 있는 메모리는 $2^{24}$ (약 16MB)로 제한된다. 현대의 수십 GB 메모리 환경에서 명령어로 모든 주소를 직접 적는 것은 불가능에 가깝다.

- **📢 섹션 요약 비유**: 한 번만 움직이면 되니 편하지만, 주소 적는 칸이 좁아서 아주 먼 동네(넓은 메모리)는 가리키기 힘든 구조입니다.

## Ⅲ. 주소 지정 방식 비교 분석

| 비교 항목 | 직접 지정 (Direct) | 간접 지정 (Indirect) | 즉시 지정 (Immediate) |
|:---|:---:|:---:|:---:|
| **메모리 참조** | **1회** | 2회 이상 | 0회 |
| **유효 주소** | **명령어 내의 값** | 메모리에 저장된 값 | 없음 (값 자체가 데이터) |
| **장점** | 단순함, 직관적 | 유연함 (포인터 가능) | 가장 빠름 |
| **단점** | 가용 주소 범위 제한 | 속도 느림 | 상수 값만 가능 |

- **📢 섹션 요약 비유**: 한 번만 물어보면 답이 나오는 질문(직접)과, 답이 어디 있는지 알려주는 힌트를 받는 것(간접)의 차이입니다.

---

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: 하드웨어 레지스터 제어 (MMIO)
- **상황**: 하드웨어의 특정 상태를 읽기 위해 고정된 I/O 번지에 접근해야 함.
- **판단**: 이때는 주소가 변할 일이 없으므로 **직접 주소 지정**이 가장 효율적이다. 임베디드 프로그래밍에서 특정 센서의 데이터가 항상 `0x4000` 번지에 들어온다면, 어셈블러는 이를 직접 지정 방식으로 처리하여 한 치의 오차도 없이 고속 접근을 보장한다.

### 2. 기술사적 인사이트: 코드 재배치성 (Relocatability)
- **인사이트**: 직접 주소 지정은 프로그램의 '재배치'를 방해한다. 주소가 코드에 박혀 있기 때문에, 프로그램이 메모리의 다른 위치로 옮겨지면 모든 주소를 일일이 수정해야 한다. 기술사는 이를 해결하기 위해 직접 지정보다는 **상대 주소 지정(Relative)**이나 **베이스 레지스터 지정** 방식을 권장하여 프로그램의 유연성을 확보해야 한다.

- **📢 섹션 요약 비유**: 이정표를 "오른쪽으로 100m"라고 적으면 위치가 바뀌어도 쓸 수 있지만, "강남역 3번 출구"라고 박아버리면 이정표를 옮길 수 없는 것과 같습니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **접근 단순화**: 하드웨어 레벨의 주소 연산 장치(AGU) 부하 감소.
- **예측 가능성**: 실행 시간이 일정하여 실시간(Real-time) 시스템의 정밀한 타이밍 제어에 유리.

### 2. 미래 전망
광활한 주소 공간을 사용하는 현대의 64비트 아키텍처에서 순수 직접 주소 지정은 점점 사라지고 있다. 대신 주소의 일부 비트만 명령어에 담고 나머지는 특정 레지스터와 조합하는 '부분 직접 지정'이나 가상 메모리 매핑 기술이 그 자리를 대신하고 있다. 하지만 하드웨어의 가장 깊숙한 곳, 장치 제어와 초기 부팅 로직에서는 여전히 직접 주소 지정의 '단순한 진리'가 시스템을 지탱하고 있다.

- **📢 섹션 요약 비유**: GPS가 아무리 발달해도, 내 집 주소를 정확히 외우고 있는 것이 가장 확실한 길찾기 방법인 것과 같습니다.
