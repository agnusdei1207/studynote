## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SIMD(Single Instruction, Multiple Data)는 단 **하나의 명령어(Instruction)로 여러 개의 데이터(Data) 요소 배열을 동시에(병렬로) 처리**하는 플린 분류법(Flynn's Taxonomy) 기반의 벡터 연산 아키텍처다.
> 2. **가치**: 3D 그래픽 렌더링, 오디오/비디오 인코딩, 인공지능(AI) 행렬 곱셈처럼 동일한 연산을 수많은 데이터에 반복해야 하는 상황에서, 루프(Loop) 오버헤드를 없애고 **초당 처리량(Throughput)을 수십 배 이상 폭발**시킨다.
> 3. **융합**: 인텔의 **AVX(Advanced Vector Extensions)**와 ARM의 **NEON**이 대표적인 상용 표준이며, 최근에는 CPU 코어 면적의 절반 이상을 이 SIMD 레지스터와 연산기가 차지할 정도로 현대 마이크로아키텍처의 가장 중요한 융합 엔진으로 자리 잡았다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **概念**: SIMD는 컴퓨터의 '다중 파채 칼'이다. 파(데이터) 하나를 썰 때 식칼(기본 명령어)로 한 번씩 썰면 100번을 썰어야 하지만, 칼날이 10개 달린 다중 파채 칼(SIMD 명령어)을 쓰면 딱 10번만 긁어도 파채 100가닥이 동시에 쏟아져 나온다.
- **💡 비유**: SIMD 명령어 확장은 '학교 운동장의 단체 체조'와 같다. 선생님(CPU)이 학생 100명에게 일일이 다가가 "너 오른팔 들어"라고 100번 명령(SISD)하는 것이 아니라, 단상에서 확성기로 **"전원 오른팔 들어!(SIMD)"**라고 한 번만 외치면 100명의 팔(데이터)이 동시에 올라가는 경이로운 병렬 처리다.
- **등장 배경**: 
  1990년대 중반, 윈도우 95가 나오고 PC에서 동영상과 3D 게임이 돌아가기 시작했다. 화면의 픽셀 100만 개 밝기를 조절하려면 덧셈 명령어를 100만 번 실행해야 했다. 클럭 속도만으로는 이 엄청난 데이터의 파도를 감당할 수 없었다. 그때 인텔은 "100만 번 명령어를 내릴 게 아니라, 64비트짜리 긴 그릇에 8비트 픽셀 8개를 한꺼번에 우겨넣고 한 방에 더해버리자!"라는 아이디어를 냈다. 이것이 1996년 인텔 펜티엄 칩에 융합된 전설적인 **MMX(MultiMedia eXtension)**의 탄생이며, SIMD 혁명의 서막이었다.

루프를 돌며 하나씩 계산하던 과거를 부수고, 배열을 뭉텅이로 썰어버리는 SIMD의 메커니즘을 시각화하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │         스칼라 연산(SISD)과 벡터 연산(SIMD)의 압도적 차이               │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                                                                         │
  │   [ 스칼라 연산 (for 루프 4번 반복) ] - 4번의 명령어 실행 필요          │
  │     A[0] + B[0] = C[0]                                                  │
  │     A[1] + B[1] = C[1]                                                  │
  │     A[2] + B[2] = C[2]                                                  │
  │     A[3] + B[3] = C[3]                                                  │
  │                                                                         │
  │   [ SIMD 벡터 연산 (AVX 융합) ] - 단 1번의 명령어(`VADDPS`)로 끝!       │
  │     ┌──────┬──────┬──────┬──────┐                                       │
  │     │ A[3] │ A[2] │ A[1] │ A[0] │ ◀─ (256비트 YMM 레지스터)             │
  │     └──────┴──────┴──────┴──────┘                                       │
  │         +      +      +      +                                          │
  │     ┌──────┬──────┬──────┬──────┐                                       │
  │     │ B[3] │ B[2] │ B[1] │ B[0] │ ◀─ (256비트 YMM 레지스터)             │
  │     └──────┴──────┴──────┴──────┘                                       │
  │         =      =      =      =                                          │
  │     ┌──────┬──────┬──────┬──────┐                                       │
  │     │ C[3] │ C[2] │ C[1] │ C[0] │ ─▶ 4개의 결과가 동시에 탄생!          │
  │     └──────┴──────┴──────┴──────┘                                       │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SIMD의 본질은 '데이터의 포장(Packing)'이다. 32비트짜리 단정밀도 부동소수점(float) 8개를 묶어, 256비트라는 거대한 버스(Bus)에 태운다. 그리고 칩 내부에 병렬로 배치된 8개의 작은 ALU들이 이 버스를 덮쳐서 정확히 1 Cycle 만에 8개의 덧셈 결과를 뱉어낸다. 명령어 인출(Fetch)과 해독(Decode)에 드는 막대한 전력을 1/8로 깎아버리는 물리적 마법이다.

- **📢 섹션 요약 비유**: 스칼라 연산이 '한 명씩 타는 회전목마'라면, SIMD 벡터 연산은 '10명이 한 줄로 타는 바이킹'과 같습니다. 놀이기구(명령어)를 한 번만 작동시켜도 훨씬 많은 사람(데이터)이 비명을 지르며(연산 완료) 내려오는 극강의 회전율을 자랑합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (SIMD의 3대 물리적 토대)

명령어 하나로 병렬 처리를 하려면 하드웨어 뼈대 자체가 완전히 다르게 생겨야 한다.

| 구성 요소 | 물리적 역할 | 아키텍처적 가치 | 비유 |
|:---|:---|:---|:---|
| **Vector Registers** | 128/256/512비트의 초거대 레지스터 | **대량의 배열 데이터를 담는 물리적 그릇** | 10인분짜리 대형 접시 |
| **Vector ALU** | 레지스터 폭만큼 병렬 배치된 덧셈/곱셈기 | **1클럭에 N개의 연산을 동시 타격** | 10구짜리 가스레인지 |

### 심층 동작 원리: "SIMD 레인(Lane)의 독립성과 포화(Saturation) 연산"

일반 ALU와 SIMD ALU는 데이터를 대하는 태도 자체가 다르다.

```text
  ┌───────────────────────────────────────────────────────────────────────────┐
  │         포화 연산(Saturation Arithmetic): 픽셀이 깨지지 않는 이유         │
  ├───────────────────────────────────────────────────────────────────────────┤
  │                                                                           │
  │   [ 상황: 화면을 더 밝게 만들려고 픽셀값(8bit: 0~255)에 10을 더함 ]       │
  │                                                                           │
  │   [ 일반 ALU (Wraparound 현상 발생) ]                                     │
  │     250 + 10 = 260 ──▶ 8비트를 넘쳐서 오버플로우 발생!                    │
  │     결과: 0000 0100 (숫자 4) ──▶ 하얀색이 갑자기 까만색으로 변함! (버그)  │
  │                                                                           │
  │   [ SIMD ALU (Saturation 포화 연산 융합) ]                                │
  │     250 + 10 = 260 ──▶ 255를 넘으면 무조건 최댓값 255로 묶어버림!         │
  │     결과: 1111 1111 (숫자 255) ──▶ 아주 밝은 하얀색 유지! (정상)          │
  │                                                                           │
  │ * 혁명: 소프트웨어에서 if문으로 예외 처리할 필요 없이 하드웨어가 방어한다.│
  └───────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 초창기 SIMD(MMX)가 가장 환영받은 이유다. 이미지 처리에서는 값이 넘친다고 0으로 돌아가면 화면에 노이즈(깨진 픽셀)가 생긴다. 기존 아키텍처는 이를 막기 위해 덧셈을 한 뒤 `if (val > 255) val = 255;` 라는 분기문을 달아야 했는데, 이게 파이프라인을 다 박살 냈다. SIMD 아키텍트들은 벡터 ALU 내부에 **'포화(Saturation)'** 논리 게이트를 융합하여, 오버플로우가 나면 무조건 최댓값(255)으로 고정시켜버리는 특수 덧셈기(PADDS)를 만들었다. 이로 인해 분기문이 통째로 사라지는 기적이 일어났다.

- **📢 섹션 요약 비유**: 포화 연산은 '물탱크의 오버플로우 파이프'와 같습니다. 물을 계속 틀어놔서 넘치려고 하면(255 초과), 댐이 무너지거나(버그) 수위가 0으로 돌아가는 게 아니라, 넘치는 물을 밖으로 버려서 항상 찰랑찰랑한 100% 최고 수위(255)를 안전하게 유지해 주는 장치입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 기술 비교: 인텔 AVX 시리즈 vs ARM NEON

현대 PC와 모바일의 SIMD 패권을 쥔 두 아키텍처의 사상적 대립이다.

| 비교 항목 | Intel AVX / AVX-512 | ARM NEON / SVE | 아키텍처 판단 포인트 |
|:---|:---|:---|:---|
| **레지스터 폭(Width)**| 256비트 (YMM) / **512비트 (ZMM)** | 128비트 (NEON) / 가변 길이 (SVE)| 전력 소모 vs 최대 처리량 |
| **명령어 인코딩** | VEX/EVEX 접두사를 붙인 3-주소 포맷 융합| 처음부터 깔끔한 3-주소 RISC 포맷 | 디코더 설계의 더러움 |
| **클럭 스로틀링** | 발열이 미쳐서 칩 속도를 강제로 깎아냄 | 발열이 적어 클럭 하락이 거의 없음 | 실리콘 한계 (Thermal Wall) |
| **철학적 기조** | "하드웨어를 때려 박아 무조건 넓히자" | "벡터 길이는 하드웨어가 알아서 조절"| 가변 길이 벡터(VLA)의 수용 여부 |
| **비유** | 연료를 퍼먹고 달리는 16기통 엔진 | **차체에 맞춰 늘어나는 하이브리드 엔진**| 물리적 폭력성 vs 유연성 |

### 과목 융합 관점

- **딥러닝 / 인공지능 (Matrix Multiplication)**: AI의 본질은 무한한 행렬 곱셈이다($Y = W \times X + B$). 과거엔 CPU가 이걸 1개씩 하느라 GPU에게 AI 패권을 뺏겼다. 하지만 AVX-512와 ARM NEON에 **FMA(Fused Multiply-Add)** 명령어와 INT8/BF16 같은 딥러닝 특화 데이터 포맷이 융합되면서 상황이 달라졌다. 한 클럭에 수십 개의 곱셈과 덧셈을 끝내버리는 이 SIMD 가속기 덕분에, 이제 가벼운 AI 추론(Inference)은 비싼 GPU 없이 CPU만으로도 충분히 돌릴 수 있는 시대가 되었다.
- **컴파일러 (Auto-Vectorization)**: 프로그래머가 일일이 SIMD 어셈블리를 짜는 건 지옥이다. 현대 컴파일러(GCC/LLVM)는 일반적인 C언어 `for` 루프를 보면, 서로 의존성이 없는 데이터임을 수학적으로 증명(Data Dependency Analysis)한 뒤, 알아서 스칼라 명령어를 SIMD 명령어로 묶어버린다. 이를 **자동 벡터화(Auto-Vectorization)**라고 부르며, 현대 소프트웨어 최적화 기술의 최고봉이다.

```text
  ┌──────────────────────────────────────────────────────────────────────────┐
  │         아키텍처의 도약: 스캐터/개더 (Scatter / Gather) 융합             │
  ├──────────────────────────────────────────────────────────────────────────┤
  │                                                                          │
  │   [ 전통적 SIMD의 한계 ] : 배열이 연속적이어야만 읽을 수 있다.           │
  │     `VLOAD` ─▶ 메모리 100, 104, 108, 112 번지를 한 번에 쓱 긁어옴.       │
  │     (만약 구조체 안에서 특정 값만 뽑고 싶다면? 로드가 안 됨!)            │
  │                                                                          │
  │   [ 최신 AVX-512 / SVE의 마법: Gather 명령어 ]                           │
  │     `VGATHER` ─▶ 주소가 100, 500, 800처럼 사방에 흩어져 있어도,          │
  │                  하드웨어가 미친 듯이 버스를 긁어모아 512비트 레지스터에 │
  │                  하나의 연속된 배열처럼 예쁘게 포장해서 담아준다!        │
  │                                                                          │
  │ * 위력: 메모리 병목으로 터져나가던 비연속적 데이터 구조마저 SIMD로 정복! │
  └──────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 초창기 SIMD는 메모리에 데이터가 1열 종대로 예쁘게 늘어서 있을 때만 힘을 썼다. 하지만 객체지향 언어(C++, Java)를 쓰면 데이터가 힙(Heap) 메모리 사방팔방에 파편화된다. 이를 벡터 레지스터에 담으려면 일반 `LOAD`를 여러 번 써서 수동으로 조립해야 했는데, 이 오버헤드가 배보다 배꼽이 더 컸다. 인텔과 ARM 아키텍트들은 메모리 컨트롤러를 마개조하여 **Gather(모아오기)**와 **Scatter(흩뿌리기)** 명령어를 하드웨어에 융합했다. 메모리 지연 시간은 조금 늘어났지만, 소프트웨어 로직을 비약적으로 단순화시켜 데이터베이스 엔진과 그래프 분석 속도를 폭발시켰다.

- **📢 섹션 요약 비유**: 스캐터/개더 기능은 '심부름센터 직원의 한 번에 장보기'와 같습니다. 예전엔 마트 진열대에 나란히 있는 물건(연속된 배열)만 한 번에 카트에 담을 수 있었습니다. 지금(Gather)은 심부름 직원이 정육점, 생선가게, 과일가게(파편화된 메모리)를 오토바이로 쏜살같이 돌아다니며 내가 원하는 레시피 박스(벡터 레지스터) 하나로 완벽하게 포장해서 가져다주는 마법입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 웹 브라우저의 JPEG/WebP 이미지 고속 디코딩**: 
  **상황**: 사용자가 사진이 수백 장 있는 쇼핑몰 웹페이지를 스크롤할 때마다, 이미지를 압축 해제하느라 브라우저가 버벅거림(Jank).
  **판단**: "Wasm SIMD(웹어셈블리 SIMD)의 융합"이다. 과거엔 브라우저 자바스크립트가 SISD(1픽셀씩 연산)로 돌았다. 브라우저 엔진 개발자는 이 이미지 디코딩 라이브러리를 **WebAssembly 128-bit SIMD** 코드로 컴파일하여 배포한다. 브라우저의 JIT 컴파일러는 이 Wasm SIMD 코드를 클라이언트 PC의 AVX(인텔)나 NEON(모바일) 기계어로 실시간 번역한다. 렌더링 속도가 4배 이상 상승하며 스크롤이 버터처럼 부드러워진다.

2. **시나리오 — 데이터베이스(DBMS)의 인메모리 컬럼 스캔 최적화**:
  **상황**: 빅데이터 분석 엔진(Apache Arrow 기반)에서 특정 조건을 만족하는 데이터(예: 나이가 30 이상인 사람)를 수십억 건 뒤져야 함.
  **판단**: "SIMD 마스킹(Masking) 및 블렌딩(Blending) 연산의 융합"이다. `if`문으로 한 줄씩 검사하면 분기 예측 실패로 쿼리가 박살 난다. DB 엔지니어는 256비트(8개 데이터) 덩어리를 통째로 SIMD 레지스터에 올리고 한 번에 `VPCMPGTD`(비교) 명령어를 날린다. 결과로 나온 8비트 마스크(Mask)를 이용해 필요한 데이터만 고속으로 추출(Extract)해 낸다. 조건문(Branch)이 0개가 되어 DB 스캔 속도가 메모리 대역폭 한계까지 치솟는다.

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │         마이크로아키텍처 설계 시 SIMD 발열과 클럭 다운 딜레마              │
  ├────────────────────────────────────────────────────────────────────────────┤
  │                                                                            │
  │   [ 왜 서버 업체들은 인텔 AVX-512를 싫어했을까? ]                          │
  │                                                                            │
  │   명령어 A (일반 연산) ──▶ 칩 온도 평온함. 클럭 3.5GHz 유지                │
  │   명령어 B (AVX-512) ──▶ 발동 순간! 512비트 스위치가 동시에 켜짐!          │
  │                                                                            │
  │   [ 칩의 방어 기제 (Thermal Throttling) ]                                  │
  │    "온도가 너무 높다! 코어가 녹기 전에 전체 클럭을 2.5GHz로 깎아라!"       │
  │                                                                            │
  │  결과: AVX-512를 쓰면 벡터 연산은 빠르지만, 같은 코어에 물려 있던 다른     │
  │       스칼라 연산(웹서버 등)들까지 속도가 느려지는 치명적 부작용 발생!     │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ISA 확장이 낳은 가장 큰 물리적 부작용이다. AVX-512 유닛은 트랜지스터 밀도가 너무 높아서 가동되는 순간 엄청난 전류를 빨아들이고 열을 뿜어낸다(Voltage Droop). CPU는 스스로 타죽지 않기 위해 전체 클럭 속도를 강제로 떨어뜨린다. 문제는, AVX 명령어를 아주 잠깐만 써도 클럭이 떨어지고 회복되는 데 시간이 걸려 시스템 전체의 평균 반응 속도(Latency)가 하락한다는 점이다. 이 때문에 넷플릭스나 클라우드플레어 같은 거대 기업들은 아예 컴파일러 단에서 AVX-512를 끄고 AVX2(256비트)만 쓰도록 튜닝하는 웃지 못할 촌극이 벌어지기도 했다.

### 도입 체크리스트
- **Data Alignment (정렬)**: SIMD 명령어를 쓸 때, 메모리 주소가 32바이트(256bit) 또는 64바이트(512bit) 경계에 정확히 정렬(Aligned)되어 있는가? (과거 아키텍처에서는 정렬되지 않은 데이터를 로드할 때 페널티가 어마어마했다.)
- **SIMD Register Spilling**: 컴파일러가 SIMD 최적화를 너무 빡세게 한 나머지 벡터 레지스터(수십 KB 크기)가 모자라 스택에 쫓겨나는(Spill) 사태가 없는가? (벡터 데이터를 메모리로 쫓아내는 비용은 스칼라보다 수십 배 비싸다.)

### 안티패턴
- **AoS (Array of Structures) 데이터 구조 고집**: C언어에서 `struct { x, y, z }` 형태로 구조체를 만들고 그 배열을 선언하는 것(AoS). 이 구조를 SIMD 레지스터에 로드하면 `x, y, z`가 섞여서 들어가기 때문에 한 번에 `x`값들만 더할 수가 없다. 이를 억지로 분리(Shuffle)하느라 오히려 스칼라보다 느려지는 최악의 안티패턴이다. SIMD 성능을 뽑으려면 반드시 `x`배열, `y`배열을 따로 선언하는 **SoA (Structure of Arrays)** 구조로 소프트웨어 근본을 뜯어고쳐야 한다.

- **📢 섹션 요약 비유**: AoS 구조의 SIMD 처리는 '햄버거 통째로 여러 개 먹기'와 같습니다. 내 목표는 햄버거 속의 토마토(x값)만 빼먹는 건데, 햄버거 세트(AoS)를 통째로 여러 개 들고 오면 빵과 고기 속에서 토마토를 일일이 분리하느라 시간이 다 갑니다. 처음부터 토마토 통, 고기 통, 빵 통(SoA)을 따로 담아오면 토마토만 한 움큼(SIMD) 쥐어서 먹을 수 있습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 스칼라(SISD) 기반 처리 | SIMD (AVX/NEON) 융합 처리 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 32비트 데이터 1클럭당 1개 덧셈 | **256비트 버스로 1클럭당 8개 덧셈** | 배열 연산 속도 산술적으로 **8배 향상** |
| **정량** | 루프(Loop)를 위한 Branch 남발 | 벡터 길이에 맞춰 Branch 횟수 급감 | 제어 해저드 방지로 IPC **대폭 상승** |
| **정성** | 멀티미디어 처리 시 CPU 점유율 100% | CPU 10~20%만 쓰고도 부드러운 영상 | **현대 3D 그래픽 및 미디어 생태계** 창조 |

### 미래 전망
- **SVE (Scalable Vector Extension)의 표준화**: 기존 AVX나 NEON은 레지스터 크기(128, 256)가 명령어에 박혀 있었다. ARM SVE나 RISC-V V-Extension은 "벡터 길이를 하드웨어가 런타임에 결정한다(Vector Length Agnostic)"는 철학을 융합했다. 똑같은 바이너리가 스마트워치(128비트)에서도 돌고 슈퍼컴퓨터(2048비트)에서도 완벽히 최적화되어 도는 **궁극의 스케일링(Scaling)** 시대가 열렸다.
- **AMX (Advanced Matrix Extensions)**: 1차원 배열(Vector)을 넘어 아예 2차원 행렬(Matrix)의 타일(Tile) 전체를 하나의 레지스터에 담고, 단 1클럭 만에 행렬 곱셈을 박살 내버리는(Systolic Array 융합) AMX가 차세대 x86 서버의 메인 무기로 등판하며 AI 가속기 시장을 흔들고 있다.

### 참고 표준
- **Intel Intrinsics Guide**: 프로그래머가 C/C++ 안에서 어셈블리를 직접 짜지 않고도 컴파일러 함수(`_mm256_add_ps` 등)를 통해 AVX 레지스터를 직접 타격할 수 있게 해주는 업계 최고의 SIMD 프로그래밍 바이블.
- **ARM NEON**: 스마트폰의 카메라 필터부터 모바일 3D 게임 물리 엔진까지, 안드로이드와 iOS 앱 생태계의 성능을 책임지는 128비트 SIMD의 모바일 표준 규격.

"루프를 돌지 말고 배열을 통째로 씹어 먹어라"라는 병렬주의 철학으로 CPU 코어의 면적을 가장 극적으로 팽창시킨 'SIMD 명령어 확장'의 진화 로드맵은 다음과 같다.

```text
  ┌──────────────────────────────────────────────────────────────────────────────────────────────────┐
  │         데이터 폭식의 역사: SIMD 아키텍처 진화 로드맵                                            │
  ├──────────────────────────────────────────────────────────────────────────────────────────────────┤
  │                                                                                                  │
  │   [1단계: 멀티미디어의 등장]  [2단계: 버스의 확장과 부동소수점] [3단계: 가변 길이와 텐서의 시대] │
  │                                                                                                  │
  │   64비트 정수로 영상 좀 돌리자 ──▶ 256/512비트로 3D랑 과학 계산! ──▶ 행렬 곱셈과 크기 자율화!    │
  │  (인텔 MMX, 펜티엄의 신화)    (AVX의 등장, 클럭 다운의 시작) (ARM SVE, x86 AMX의 융합)           │
  │ "그림 8개 동시에 그릴게!"     "레지스터 터진다 클럭 좀 낮춰!"  "하드웨어 크기에 맞춰서 달린다"   │
  └──────────────────────────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SIMD의 역사는 '그릇 키우기'의 역사다. 1단계: 90년대 CD-ROM 동영상을 끊김 없이 보기 위해 64비트 그릇(MMX)을 만들었다. 2단계: 과학 연산과 3D 게임이 발전하자 128비트(SSE), 256비트(AVX), 512비트(AVX-512)로 그릇을 미친 듯이 키웠다. 하지만 그릇이 커질수록 칩이 불타올랐다. 3단계: 결국 현대 아키텍트는 하드코딩된 크기를 버리고 "소프트웨어는 모르게 하되, 하드웨어가 알아서 그릇 크기를 조절하는(SVE)" 유연한 아키텍처로 진화했으며, 배열을 넘어 2차원 행렬(Matrix) 자체를 씹어 삼키는 괴물로 진화하며 CPU의 생명 연장을 이끌고 있다.

- **📢 섹션 요약 비유**: SIMD 확장의 진화는 '포크레인의 적재함 진화'와 같습니다. 처음엔 삽으로 한 숟갈씩 퍼다가(SISD), 작은 포크레인(MMX)을 도입했고, 나중엔 아예 집채만 한 거대 포크레인(AVX-512)으로 산을 통째로 펐습니다. 하지만 기름을 너무 많이 먹고 엔진이 과열되니까, 이제는 흙의 양을 보고 삽의 크기가 스스로 변신하는 아주 똑똑한 스마트 포크레인(SVE)으로 진화한 것입니다.
