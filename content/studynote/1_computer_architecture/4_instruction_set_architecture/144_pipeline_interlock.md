> **핵심 인사이트**
> 1. 파이프라인 인터록(Interlock)은 파이프라인 내부에서 데이터 해저드나 제어 해저드가 감지되었을 때, **하드웨어 스스로 파이프라인의 톱니바퀴를 강제로 멈춰 세워(Stall) 에러를 방어하는 긴급 브레이크 시스템**이다.
> 2. 컴파일러가 NOP 명령어(거품)를 삽입하는 소프트웨어 방식과 달리, 칩 내부에 내장된 **해저드 감지 유닛(Hazard Detection Unit)**이 실시간으로 핀의 상태를 감시하다가 위기 시에 발동한다.
> 3. PC(프로그램 카운터)와 IF/ID 레지스터의 값을 얼려버려(Freeze) 명령어의 진행을 멈추고, 뒤쪽 파이프라인으로 **가짜 명령어(Bubble, 0으로 채워진 NOP)**를 흘려보내 빈 공간을 만든다.

---

## Ⅰ. 위기 감지: 해저드 감지 유닛 (Hazard Detection Unit)
앞서 배운 '전방 전달(Forwarding)'로도 막을 수 없는 최악의 데이터 해저드가 바로 `LOAD` 명령어 직후에 그 데이터를 쓰는 **Load-Use 해저드**입니다.
메모리에서 무를 뽑기도 전(M 단계)에, 덧셈기가 도마 위에서 무를 썰려고(E 단계) 기다리는 상황입니다.

이 위기를 감지하기 위해 CPU의 Decode(해독) 단계에는 **'해저드 감지 유닛'**이라는 cctv가 달려있습니다. 
이 유닛은 앞명령어가 `LOAD`인지, 그리고 앞명령어가 데이터를 넣을 목적지 레지스터(예: R1)가 내가 지금 썰려고 하는 재료(R1)와 똑같은지를 0.001초 만에 스캔합니다. 만약 100% 똑같다면, 유닛은 "위험해! 지금 덧셈기(E)로 넘어가면 쓰레기 숫자로 더하게 돼!"라며 **인터록(긴급 브레이크)**을 발동시킵니다.

```ascii
[해저드 감지 유닛의 감시 조건 (Load-Use 판단)]

 IF (앞 명령어의 Opcode == LOAD) AND 
    (앞 명령어의 목적지 레지스터 == 내 명령어의 재료 레지스터 1 OR 2)
 THEN
    -> 인터록(Stall) 발동!!
```

📢 **섹션 요약 비유**: 해저드 감지 유닛은 지하철의 '스크린도어 센서'입니다. 센서가 열차가 완전히 도착해서 문이 열렸는지(LOAD 완료)를 감시합니다. 열차가 안 왔는데 사람이 발을 내딛으려 하면(Use 시도), 센서가 삐용삐용 울리며 스크린도어를 꽉 잠가버려(인터록) 사람이 선로로 떨어지는 끔찍한 사고를 막아줍니다.

## Ⅱ. 인터록의 작동 방식: 얼음(Freeze)과 거품(Bubble)
인터록이 발동되면 파이프라인 공장에서는 두 가지 아주 극단적인 조치가 취해집니다.

1. **앞부분은 "얼음! (Freeze)"**: 해저드 감지 유닛은 PC(프로그램 카운터)와 IF/ID 레지스터에 신호를 보내서 **"클럭이 쳐도 절대 값을 다음으로 넘기지 말고 지금 상태 그대로 얼어붙어 있어라!"**라고 명령합니다. 이렇게 하면 문제가 된 덧셈 명령어는 Decode 방에 갇혀서 1클럭 동안 꼼짝도 못 합니다.
2. **뒷부분엔 "거품(Bubble) 쏘기"**: 덧셈 명령어가 갇혀버렸으니, 그 뒤에 있는 덧셈기(Execute 방)는 텅 비게 됩니다. 이때 해저드 유닛은 덧셈기 쪽으로 **'모든 제어 신호가 0으로 채워진 가짜 명령어(NOP, 아무것도 하지 마라)'**를 툭 밀어 넣습니다. 이 빈 공기방울(Bubble)이 파이프라인을 타고 뒤로 흘러가면서 1클럭의 시간을 완벽하게 벌어줍니다.

```ascii
[인터록 발동 시 파이프라인의 모습 (1클럭 Stall 발생)]

 (정상 흐름)
 1. LOAD R1 -> [F] [D] [E] [M (R1 도착!)] [W]

 (인터록 발동: 덧셈명령이 Decode 방에 갇힘)
 2. ADD R2, R1 ->  [F] [D(얼음!)] [Bubble(가짜명령)] [E(R1받고 덧셈!)] [M] [W]
                             ^^^^^^^^^^^^^^^^
                 (1클럭 쉬었기 때문에 M단계에서 날아온 R1을 낚아챌 수 있게 됨)
```

📢 **섹션 요약 비유**: 인터록의 작동은 '에스컬레이터 앞 멈춤'입니다. 내 앞사람이 에스컬레이터 끝에서 짐을 놓쳐서 넘어졌습니다(해저드). 나는 그 사람을 밟고 지나가면 안 되니까 그 자리에서 1초 동안 가만히 얼음(Freeze) 상태로 멈춰 섭니다. 그러면 나와 앞사람 사이에 1칸의 빈 계단(Bubble)이 생기며 안전거리가 확보되는 완벽한 회피 기술입니다.

## Ⅲ. 아키텍처 관점: H/W 인터록 vs S/W 스케줄링의 승리자
초기 MIPS 아키텍처 설계자들은 "CPU 칩(하드웨어) 안에 이런 복잡한 브레이크 회로(인터록)를 넣으면 칩이 무거워지고 전기를 많이 먹잖아? 그냥 똑똑한 **컴파일러(소프트웨어)**가 코드 사이에 알아서 NOP(쉬어라) 명령어를 끼워 넣게 만들자!"라고 주장했습니다. (소프트웨어적 해결)

하지만 세상은 하드웨어의 손을 들어주었습니다. 컴파일러가 아무리 코드를 잘 짜도, 프로그램이 실제로 실행될 때 메모리에서 지연이 터질지 안 터질지는 100% 알 수 없었습니다. 게다가 CPU 모델이 바뀌어서 파이프라인 깊이가 달라지면, 컴파일러가 NOP을 다시 끼워 넣기 위해 코드를 통째로 다시 번역해야 하는 재앙이 발생했습니다.
결국 현대의 모든 칩(ARM, 인텔, AMD)은 **CPU 내부에 막강한 해저드 감지 유닛과 하드웨어 인터록(Stall) 기능을 기본으로 박아넣는 방향**으로 진화했습니다. 컴파일러의 짐을 하드웨어가 완전히 짊어지게 된 것입니다.

```ascii
[해저드 방어 철학의 승자]

 [과거] S/W 의존 (컴파일러가 NOP 삽입)
 -> 하드웨어는 가볍지만 호환성 폭망, 예측 불가 지연에 취약함.

 [현대] H/W 인터록 (해저드 감지 유닛이 실시간 Stall)
 -> 칩은 복잡해졌지만, 어떤 구형 소프트웨어를 가져와도 하드웨어가 알아서
    브레이크를 잡아주어 절대 에러가 나지 않는 우주 방어 시스템 완성!
```

📢 **섹션 요약 비유**: 하드웨어 인터록은 자동차의 '자동 긴급 제동 시스템(AEB)'입니다. 옛날엔 운전자(컴파일러)가 눈을 부릅뜨고 앞차 거리를 재서 스스로 브레이크를 밟아야 했습니다. 하지만 사람이 실수해서 사고가 자꾸 나니까, 요즘 차들은 차 앞범퍼에 레이더 센서(해저드 감지 유닛)를 빡세게 달아놔서 운전자가 졸고 있어도 차가 알아서 끼익! 하고 강제 브레이크(인터록)를 밟아주는 것이 표준이 된 것입니다.

## 개념 맵 (Concept Map)
```ascii
[파이프라인 인터록 (Pipeline Interlock)]
   │
   ├── 정의: 데이터/제어 해저드 발생 시 하드웨어가 동적으로 파이프라인을 멈춰 세우는(Stall) 제어 메커니즘
   │
   ├── 작동 원리 (Hazard Detection Unit 주도)
   │     ├── Freeze: PC와 IF/ID 레지스터의 Write Enable을 0으로 만들어 명령어 진행 동결
   │     └── Bubble: EX 단계로 향하는 제어 신호를 모두 0(NOP)으로 변환하여 빈 파이프라인 공간 생성
   │
   └── 아키텍처 생태계의 결론
         └── S/W 컴파일러의 정적 스케줄링(NOP 삽입)에 의존하던 방식에서 벗어나, H/W 내부의 동적 인터록 기능이 현대 CPU 데이터패스의 필수 안전장치로 자리잡음
```

## 어린이 비유
파이프라인 인터록은 공장 레일 위의 **빨간색 긴급 정지 버튼**이에요! 앞 로봇이 장난감에 나사를 덜 조였는데, 뒤쪽 레일이 너무 빨리 굴러와서 장난감이 망가지게 생겼어요. 그때 천장에 달린 cctv 감시 카메라(해저드 감지 유닛)가 이걸 0.1초 만에 발견하고 삐용삐용! 하면서 레일을 "딱!" 멈춰 세워요(얼음!). 앞 로봇이 나사를 완벽하게 다 조이고 나면 그제야 다시 레일을 부드럽게 돌려주는 아주 든든한 안전장치랍니다!
