# 프로그램 카운터 (PC, Program Counter)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로그램 카운터 (PC, Program Counter)는 CPU 내부에서 **다음에 실행할 명령어의 메모리 주소**를 가리키는 핵심 특수 목적 레지스터로, x86 아키텍처에서는 명령어 포인터(IP, Instruction Pointer)라고도 불린다.
> 2. **가치**: 명령어 인출(Fetch) 시 메모리 주소 버스의 입력값이 되며, 인출 직후 자동으로 값이 증가하거나 분기 명령어(Jump, Branch) 실행 시 목표 주소로 갱신되어 프로그램의 실행 흐름을 제어한다.
> 3. **융합**: 운영체제의 멀티태스킹 환경에서 문맥 교환 시 가장 먼저 보존되어야 할 정보이며, 하드웨어 파이프라이닝 구조에서 분기 예측(Branch Prediction) 실패 시 성능 저하의 핵심 지점이 된다.

---

## Ⅰ. 프로그램 카운터(PC)의 정의 및 동작

### 1. 정의
- CPU가 메모리에 저장된 프로그램을 순차적으로 실행할 수 있도록, 다음에 읽어올 명령어의 위치를 항상 기억하고 있는 장치이다.

### 2. 기본 동작 원리
1. **인출 (Fetch)**: PC에 저장된 주소를 MAR(Memory Address Register)로 전송하여 메모리에서 명령어를 가져온다.
2. **증가 (Increment)**: 명령어를 성공적으로 인출하면, PC의 값은 자동으로 다음 명령어의 위치(현재 주소 + 명령어 크기)로 증가한다.
3. **갱신 (Update)**: 실행 단계에서 `JUMP`나 `CALL` 같은 제어문이 실행되면, 순차적인 증가 대신 새로운 대상 주소가 PC에 강제로 입력된다.

### 3. 💡 비유: '독서 중의 손가락'
- PC는 책을 읽을 때 **'다음에 읽어야 할 글자를 가리키고 있는 손가락'**과 같다.
- 한 글자를 읽으면 손가락은 자동으로 다음 글자로 이동(순차 실행)하지만, "3페이지로 가시오"라는 지시(분기 명령어)를 만나면 손가락을 해당 페이지로 즉시 옮겨야 한다.

---

## Ⅱ. PC의 세부 메커니즘 (Deep Dive)

### 1. 명령어 크기와 PC 증가분
- 고정 길이 명령어(예: 32비트)를 사용하는 RISC 환경에서는 PC가 항상 **+4**씩 일정하게 증가한다.
- 가변 길이 명령어를 사용하는 CISC 환경에서는 현재 명령어를 해독(Decode)하여 길이를 파악한 후 그만큼 증가시킨다.

### 2. 서브루틴 호출 (Function Call) 시의 PC
- 함수 호출 시 현재 PC 값(복귀 주소)을 **스택(Stack)**에 저장하고, 함수의 시작 주소를 PC에 로드한다. 함수가 끝나면 스택에서 이전 값을 꺼내와 PC를 원상복구 한다.

### 3. ASCII 구조 다이어그램
```text
    ┌──────────────────────────────────┐
    │        Program Counter (PC)      │ ◀───┐
    └───────────────┬──────────────────┘     │
                    │                        │ (Next Addr / Jump Addr)
                    ▼                        │
    ┌──────────────────────────────────┐     │
    │   Memory Address Register (MAR)  │     │
    └───────────────┬──────────────────┘     │
                    │                        │
                    ▼ (Address Bus)          │
    ┌──────────────────────────────────┐     │
    │          Main Memory             │     │
    │  [ Inst 1 ] [ Inst 2 ] [ Inst 3 ]│     │
    └───────────────┬──────────────────┘     │
                    │ (Data Bus)             │
                    ▼                        │
    ┌──────────────────────────────────┐     │
    │   Instruction Register (IR)      │ ────┘
    └──────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 다음에 가야 할 길을 미리 알고 있는 '내비게이션'과 같습니다. 경로 이탈(분기) 시에는 즉시 새로운 경로를 탐색하여 목적지(프로그램 종료)까지 안내합니다.

## Ⅲ. PC 관련 기술적 쟁점

| 상황 | PC의 변화 | 영향 |
|:---|:---|:---|
| **순차 실행** | PC = PC + 명령어 크기 | 일반적인 처리 흐름 |
| **분기 (Jump)** | PC = 목표 주소 | 실행 흐름의 비순차적 전환 |
| **인터럽트** | PC = ISR 시작 주소 | 긴급 상황 대처 (기존 값은 스택 보관) |
| **파이프라이닝** | PC가 미리 증가함 | 분기 예측 실패 시 파이프라인 플러시 발생 |

- **📢 섹션 요약 비유**: 평소에는 인도를 따라 걷다가(순차), 갈림길에서 방향을 틀거나(분기), 사고가 나면 병원으로 실려 가는(인터럽트) 과정의 모든 위치 정보가 PC에 담깁니다.

---

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 버퍼 오버플로우 공격 (BOF)
- **상황**: 해커가 스택에 저장된 복귀 주소를 자신의 악성 코드 주소로 덮어씌움.
- **판단**: 함수가 종료될 때 CPU는 스택에서 엉뚱한 주소를 읽어 **PC**에 로드하게 된다. 결과적으로 실행 흐름이 해커의 의도대로 꺾인다. 기술사라면 이를 방지하기 위한 **ASLR (주소 공간 무작위 배치)**이나 **Shadow Stack** 기술이 PC의 무결성을 어떻게 지키는지 설명할 수 있어야 한다.

### 2. 기술사적 인사이트: PC 상대 주소 지정 (PC-Relative)
- 기술사라면 명령어 내에 절대 주소를 적는 대신, **'현재 PC 값으로부터의 거리'**를 적는 **PC 상대 주소 지정 방식**의 가치를 논해야 한다. 이는 코드가 메모리의 어느 위치에 로드되더라도 수정 없이 실행될 수 있게 하는 **재배치 가능 코드 (Relocatable Code)**의 핵심 기반이 된다.

- **📢 섹션 요약 비조**: "강남역 100번지"라고 외우는 것보다 "지금 위치에서 오른쪽으로 300m"라고 외우는 것이, 내가 어디에 있든 길을 찾기 쉬운(유연한) 방법인 것과 같습니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **실행 자동화**: 사람의 개입 없이 명령어를 순차적으로 처리하는 연산의 자동화 구현.
- **프로그램 구조화**: 함수, 반복문, 조건문 등 고수준 논리를 하드웨어 수준에서 실현.

### 2. 미래 전망
앞으로는 보안이 강화된 **기밀 컴퓨팅 (Confidential Computing)** 환경에서 PC의 변조를 실시간으로 감지하고 차단하는 하드웨어 제어 흐름 무결성(CFI) 기술이 더욱 발전할 것이다. PC는 단순한 포인터를 넘어, 컴퓨터가 '다음에 무엇을 할지'를 결정하는 의사결정의 가장 낮은 단계를 지탱하는 등대가 될 것이다.

- **📢 섹션 요약 비유**: 어두운 밤길을 비추는 전등처럼, CPU가 가야 할 길을 한 걸음 앞서 비춰주는 보이지 않는 인도자입니다.
