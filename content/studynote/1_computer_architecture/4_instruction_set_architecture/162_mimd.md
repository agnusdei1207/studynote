> **핵심 인사이트**
> 1. MIMD는 플린의 분류(Flynn's Taxonomy) 중, **여러 개의 독립된 CPU(코어)가 각자 완전히 다른 명령어(프로그램)를 가져와서 서로 다른 데이터(Data)를 동시에 처리하는 궁극의 병렬 아키텍처**이다.
> 2. 우리가 흔히 아는 듀얼코어, 쿼드코어 스마트폰부터 전 세계 데이터 센터의 클라우드 서버, 수만 대의 컴퓨터가 묶인 거대 슈퍼컴퓨터까지 **현대 IT 인프라 생태계를 완벽하게 지배하고 있는 절대적인 표준 구조**이다.
> 3. 메모리를 어떻게 나눠 쓰느냐에 따라, 모든 코어가 하나의 큰 메모리를 공유하는 **공유 메모리(SMP) 방식**과, 각자 자기만의 메모리를 갖고 네트워크로 통신하는 **분산 메모리(클러스터) 방식**으로 나뉜다.

---

## Ⅰ. 자유와 혼돈의 극치: 각자도생(各自圖生) 아키텍처
앞서 배운 SIMD(GPU)는 군대였습니다. 장군이 "쏴라!" 하면 1만 명의 병사가 눈 감고 한 곳으로 활을 쐈습니다. 빠르긴 하지만 복잡한 일은 할 수 없었습니다.

**MIMD (다중 명령어 다중 데이터)**는 전혀 다릅니다. 이 구조는 독립적인 '두뇌'를 여러 개 모아놓은 형태입니다. 
코어가 4개(Quad-core) 있다면, 각 코어는 자기만의 프로그램 카운터(PC)를 가지고 있습니다.
* **코어 1 (명령어 A, 데이터 X)**: 크롬 브라우저를 켜서 텍스트 데이터를 읽어옵니다.
* **코어 2 (명령어 B, 데이터 Y)**: 유튜브 앱을 돌리며 동영상 데이터를 재생합니다.
* **코어 3 (명령어 C, 데이터 Z)**: 백그라운드에서 바이러스 검사를 합니다.

1, 2, 3번 코어는 서로 자기가 무슨 일을 하는지 남의 눈치를 볼 필요가 없습니다. 각자 완벽하게 독립적인 프로그램을 100% 동시에 돌리며 **스레드 레벨 병렬성(TLP)**의 진수를 보여줍니다. 이것이 현대 PC와 스마트폰이 수십 개의 앱을 동시에 띄워놔도 버벅대지 않는 이유입니다.

```ascii
[MIMD 구조의 완벽한 비동기 병렬 실행]

 (각자 다른 명령)           (각자 다른 코어)            (각자 다른 데이터)
 엑셀 코드 읽기 --------->  [ CPU 코어 1 ]  <--------- 엑셀 숫자 데이터
 롤(LoL) 코드 읽기 ------>  [ CPU 코어 2 ]  <--------- 마우스 클릭 데이터
 멜론 코드 읽기 --------->  [ CPU 코어 3 ]  <--------- MP3 음악 데이터

 -> 세 코어가 서로 남남처럼 1도 간섭 안하고 완전히 다른 작업을 동시에 폭풍 실행!
```

📢 **섹션 요약 비유**: MIMD는 '대형 로펌 사무실'입니다. 변호사 4명(코어)이 한 방에 모여 있습니다. 1번 변호사는 이혼 소송(명령어 A) 서류(데이터 A)를 보고 있고, 2번 변호사는 사기 사건(명령 B) 재판(데이터 B) 준비를 합니다. 4명이 완전히 다른 전문적인 일을 각자 완벽하게 수행하는 엘리트들의 모임입니다. (단, SIMD처럼 단순 노동을 시키기엔 변호사 월급이 너무 비쌉니다.)

## Ⅱ. 메모리 소유권 전쟁 1: 공유 메모리 (Shared Memory)
MIMD를 만들 때 아키텍트들의 가장 큰 골칫거리는 "이 많은 코어들에게 밥(메모리)을 어떻게 먹일까?"였습니다. 첫 번째 해답이 **공유 메모리(Shared Memory)** 모델입니다.

이것은 우리가 쓰는 노트북이나 폰의 일반적인 **멀티코어(SMP)** 방식입니다. 코어는 8개지만, 메인보드에 꽂혀 있는 램(DRAM)은 하나의 거대한 덩어리입니다. 
* **장점**: 프로그래머(운영체제)가 코딩하기 너무 편합니다. 1번 코어가 메모리에 써놓은 변수를, 2번 코어가 그냥 주소만 알면 바로 읽어올 수 있어 코어 간의 데이터 전달이 아주 쉽습니다.
* **단점**: 코어 8개가 메모리 통로(버스) 하나를 두고 멱살잡이를 하며 밥을 먹으려 하니 끔찍한 병목이 터지고(버스 경합), 각자의 캐시(L1)가 서로 꼬이는 캐시 일관성 문제(Snooping 오버헤드)가 발생해 코어 수를 64개 이상 늘리기가 물리적으로 매우 힘듭니다.

```ascii
[MIMD - 공유 메모리 (Shared Memory, SMP) 모델]

 [ 코어 1 ]   [ 코어 2 ]   [ 코어 3 ]   [ 코어 4 ]
     \            |            |            /
      \           |            |           /
       ======[ 시스템 버스 (교통체증!) ]======
                        |
              [ 거대한 단일 메인 메모리 ]
 (장점: 데이터 공유 쉬움 / 단점: 길이 막혀서 코어 무한 확장 불가)
```

📢 **섹션 요약 비유**: 공유 메모리는 '대가족의 큰 냄비 식사'입니다. 8명의 식구(코어)가 식탁 가운데 놓인 찌개 냄비(메모리) 하나를 같이 떠먹습니다. 반찬 나눠 먹긴 편하지만(데이터 공유 용이), 식구가 100명으로 늘어나면 냄비 하나에 숟가락 100개가 부딪혀서(버스 병목) 아무도 밥을 못 먹게 되는 한계가 명확합니다.

## Ⅲ. 메모리 소유권 전쟁 2: 분산 메모리 (Distributed Memory)
공유 메모리의 병목 한계(수십 개 코어)를 박살 내고, 코어를 수만 개까지 무한대로 늘리기 위해 등장한 것이 **분산 메모리(Distributed Memory / Message Passing)** 모델입니다.

이것은 구글, 아마존 데이터 센터나 슈퍼컴퓨터의 **클러스터(Cluster)** 방식입니다. 
"야! 거대한 메모리 하나 놔두고 싸우지 마. **각 코어마다 자기 전용 메모리를 따로 챙겨줄게! 남의 메모리는 절대 못 건드려!**"
이렇게 하면 버스 병목이 아예 사라지므로 코어를 10만 개, 100만 개까지 무한정 확장할 수 있습니다. 

단, 치명적인 단점이 있습니다. 1번 코어가 2번 코어의 데이터가 필요하면, 램을 직접 뒤지는 게 불가능하므로 반드시 랜선(이더넷 네트워크 등)을 통해 **"안녕, 나 1번인데 데이터 좀 보내줄래?" 하고 택배(Message Passing, MPI 등)를 보내야 합니다.** 이 네트워크 통신을 프로그래머가 일일이 코딩해 줘야 해서 S/W 개발 난이도가 지옥에 가깝고, 통신 지연 속도가 엄청나게 큽니다.

```ascii
[MIMD - 분산 메모리 (Cluster, Message Passing) 모델]

 [ 컴퓨터 1 ]            [ 컴퓨터 2 ]            [ 컴퓨터 3 ]
 [코어1]+[메모리A]       [코어2]+[메모리B]       [코어3]+[메모리C]
       |                      |                      |
       +------( 고속 이더넷/네트워크 통신망 )--------+
       
 (장점: 10만대까지 무한 확장 가능! / 단점: 데이터 공유하려면 택배(메시지) 보내야해서 느림)
```

📢 **섹션 요약 비유**: 분산 메모리는 '각자 도시락 싸오기'입니다. 학생 1,000명(코어)이 각자 자기 도시락(로컬 메모리)만 먹으니까 밥 먹다 부딪힐 일(병목)이 없어 10만 명이 한 번에 밥을 먹을 수 있습니다(무한 확장성). 그런데 1번 학생이 2번 학생의 계란말이를 먹고 싶으면 손으로 집어먹을 수 없고, 편지를 써서 "계란말이 좀 던져줘(메시지 패싱)"라고 부탁해야 하는 엄청난 불편함이 따릅니다.

## 개념 맵 (Concept Map)
```ascii
[MIMD (Multiple Instruction Multiple Data)]
         │
   ├── 정의: 플린의 분류 중, 다수의 독립된 프로세서(Core)가 각자 다른 명령어와 데이터 스트림을 병렬로 처리하는 가장 보편적이고 범용적인 현대 컴퓨팅 아키텍처
   │     └── 목표: 작업 스레드 간의 분산 처리(TLP) 및 독립적 멀티태스킹 달성
         │
   └── 메모리 아키텍처 결합 모델 (병목 극복의 갈래)
         ├── 공유 메모리 (Shared Memory / SMP)
         │     ├── 특징: 모든 코어가 단일 물리적 메모리 공간 공유 (코딩 난이도 낮음)
         │     └── 한계: System Bus Contention 및 캐시 Coherence 오버헤드로 인한 확장성(Scalability) 한계 (PC, 스마트폰)
         │
         └── 분산 메모리 (Distributed Memory / Cluster)
               ├── 특징: 각 노드(코어)가 독자적 로컬 메모리 소유, Message Passing(MPI)으로 통신
               └── 한계: S/W 통신 오버헤드는 높으나 하드웨어 병목이 없어 코어를 무한대(슈퍼컴퓨터, 클라우드 서버)로 Scale-out 가능
```

## 어린이 비유
MIMD는 미술 시간에 모인 **천재 화가 10명**이에요! 장군님이 지시하는 똑같은 색깔만 칠하는 멍청한 군대(SIMD)와 다르게, 1번 화가는 산을 그리고 2번 화가는 바다를 그리고 3번 화가는 사람을 그려요. 각자 머릿속(명령어)에 있는 완전히 다른 생각들을 각자 다른 물감(데이터)으로 아주 멋지게 동시에 그려내는 훌륭한 시스템이죠! 우리 집 컴퓨터 안에도 이런 똑똑한 화가들이 8명(옥타코어)이나 들어가 있어서 유튜브랑 게임을 동시에 돌릴 수 있는 거랍니다!
