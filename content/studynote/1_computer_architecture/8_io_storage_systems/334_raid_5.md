# 334. RAID 5 (분산 패리티)

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 최소 3개의 디스크에 데이터를 분산 저장하고, XOR 연산을 통해 생성한 패리티(Parity)를 디스크 전체에 순차적으로 분산(Distributed) 배치하여 복구성을 확보하는 블록 레벨 스토리지 기술.
> 2. **가치**: 전체 용량의 딱 1개 디스크 분량만 오버헤드로 사용하여(N-1의 효율), 경제적 가용성과 1개 디스크 장애 허용(Fault Tolerance)이라는 높은 가용성을 동시에 제공.
> 3. **융합**: OS의 파일 시스템(File System)과 밀접하게 연동되며, 쓰기 패널티(Write Penalty) 문제로 인해 NVRAM(Non-Volatile Random Access Memory) 캐시가 탑재된 HW RAID 컨트롤러 또는 ZFS와 같은 SW RAID 솔루션과의 조합이 필수적임.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 정의**
**RAID 5 (Redundant Array of Independent Disks 5)**는 블록 레벨 스트라이핑(Block-level Striping)과 분산 패리티(Distributed Parity)를 결합한 기술이다. 데이터를 여러 디스크에 나누어 저장(Striping)하여 읽기 성능을 높이는 동시에, **XOR (Exclusive OR)** 연산을 통해 생성한 패리티 정보를 특정 디스크에만 몰아두지 않고 전체 디스크에 균등하게 분산 저장한다. 이는 RAID 4의 패리티 병목 현상을 해결하여 쓰기 성능을 개선한 형태이다.

**2. 등장 배경 및 철학**
과거 스토리지 환경에서는 **RAID 0 (Striping)**의 빠른 속도와 **RAID 1 (Mirroring)**의 안전성 사이에서 비용과 성능의 트레이드오프가 존재했다. RAID 1은 50%의 용량 효율밖에 내지 못했고, RAID 0은 데이터 복구 능력이 전무했다. 따라서 "디스크 1개 분량의 비용을 희생하여, 전체 디스크 장애에 대비하면서도 용량 효율을 극대화하자"는 타협안으로 등장했다. 이는 특히 읽기 작업(Read Operation)이 많은 서버 환경에서 가장 사랑받는 구성이 되었다.

**3. 구조적 이해 - 디스크 배치도**

아래 다이어그램은 3개의 디스크(Disk 0, 1, 2)를 사용하여 데이터(A, B, C)와 패리티(P)가 분산되는 양상을 보여준다.

```text
+-------------------+          +-------------------+          +-------------------+
|      DISK 0        |          |      DISK 1        |          |      DISK 2        |
+-------------------+          +-------------------+          +-------------------+
| Chunk A (Data)     |  Stripe  | Chunk B (Data)     |  Stripe  | Parity P(A,B)      |
| [Binary 101]      |   #1     | [Binary 110]      |   #1     | [A xor B = 011]    |
+-------------------+          +-------------------+          +-------------------+
| Parity Q(C,D)     |          | Chunk C (Data)     |          | Chunk D (Data)     |
| [C xor D]         |  Stripe  | [Binary 100]      |  Stripe  | [Binary 001]      |
+-------------------+   #2     +-------------------+   #2     +-------------------+
| ...               |          | ...               |          | ...               |
+-------------------+          +-------------------+          +-------------------+
```

> **해설**: 위 구조에서 Stripe 1은 Disk 0, 1에 데이터를 쓰고 계산된 패리티를 Disk 2에 기록한다. 다음 Stripe 2에서는 패리티의 위치가 Disk 0으로 순환한다. 이렇게 패리티의 책임(Write Burden)이 특정 디스크 한 곳에 집중되지 않도록 분산시키는 것이 핵심 원리이다.

📢 **섹션 요약 비유**: 3명이 팀 프로젝트를 하는데, 각자 맡은 역할을 분담(Striping)하되, 팀원 한 명이 병가를 낼 경우를 대비해 서로의 '보고서 요약본(패리티)'을 조각내어 나누어 가지는 것과 같습니다. 이때 요약본 작성 책임을 한 명에게만 주지 않고 돌아가며 맡겨 업무 과부하를 방지합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. 구성 요소 상세 분석**

RAID 5 시스템을 구성하는 핵심 모듈과 그 역할은 다음과 같다.

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 (Internal Operation) | 관련 프로토콜/기술 |
|:---|:---|:---|:---|
| **RAID Controller** | 전체 스토리지 관리 및 I/O 스케줄링 | 호스트(Host)의 명령을 해석하여 데이터를 블록으로 분할하고 패리티를 계산하여 디스크에 전송 | SATA, SAS, NVMe |
| **XOR Engine** | 패리티 연산 가속기 | 하드웨어적(ASIC/FPGA) 또는 소프트웨어적으로 배타적 논리합(XOR) 연산을 수행하여 P 블록 생성 | Logical Operation |
| **Disk Array** | 물리적 저장 공간 | 최소 3개 이상의 디스크가 데이터와 패리티를 수납 | HDD, SSD |
| **Cache Memory** | 쓰기 성능 버퍼링 | **WB (Write Back)** 모드 시 데이터를 일시 저장 후 배치 작업(Batch Processing)으로 디스크 부하 감소 | DRAM, NVRAM |
| **Journaling Layer** | 데이터 무결성 보장 | 시스템 정전 시 복구를 위해 메타데이터와 패리티 업데이트 로그를 기록 | File System Journal |

**2. 핵심 동작 메커니즘: 패리티 생성과 복구**

RAID 5의 수학적 배경은 **XOR (Exclusive OR)** 연산에 기반한다.
- 특성: $A \oplus B = P$, $A \oplus P = B$ (덧셈/뺄셈이 동일한 성질)
- 따라서 데이터 일부가 손상되어도 나머지 데이터와 패리티를 XOR 연산하면 원본을 복원할 수 있다.

**데이터 쓰기(Write) 동작 과정 (Full Stripe Write vs Read-Modify-Write)**

1.  **Full Stripe Write (이상적)**: 새로운 스트라이프에 데이터를 쓸 때, 모든 디스크에 데이터를 쓰고 패리티를 한 번에 계산하여 기록한다. (성능 최대)
2.  **Read-Modify-Write (RMW, 현실적)**: 기존 데이터를 수정할 때 발생한다. 이는 **쓰기 페널티(Write Penalty)**의 주범이다.

아래는 '기존 블록 수정' 시 발생하는 쓰기 페널티의 4단계 I/O 과정을 도식화한 것이다.

```text
< State 1: 초기 상태 >
Disk 0: [ D0 ]  Disk 1: [ D1 ]  Disk 2: [ P = D0 xor D1 ]

< State 2: D0을 NewD0로 변경 요청 발생 >
Step 1 (Read):  [ D0 ]  읽기
Step 2 (Read):  [ P ]   읽기 (Old Parity)

< Memory Internal Calculation >
Step 3 (Calc):  New_P = (Old_P xor Old_D0) xor New_D0
                * 기존 패리티에서 기존 데이터를 제거(XOR)하고 새 데이터를 반영

Step 4 (Write): [ NewD0 ] 쓰기
Step 5 (Write): [ New_P ] 쓰기
```

> **해설**: 단순한 데이터 1회 쓰기 명령(Write I/O) 하나를 처리하기 위해 실제 디스크에는 **2회의 읽기(Read)**와 **2회의 쓰기(Write)**가 발생한다. 이를 **4 I/Os Operation**이라 하며, 쓰기 성능이 RAID 1(2 I/Os)에 비해 절반으로 떨어지는 원인이 된다. 이를 완화하기 위해 **BBU (Battery Backup Unit)** 또는 **NVRAM (Non-Volatile RAM)** 캐시를 사용하여 Write Back 정책을 쓴다.

**3. 핵심 알고리즘 코드 (Pythonic Pseudocode)**

```python
def calculate_parity(data_blocks):
    """
    여러 데이터 블록을 받아 XOR 패리티를 계산하는 함수
    """
    parity = 0  # 초기값 (Identity element of XOR)
    for block in data_blocks:
        parity ^= block  # 비트 단위 XOR 연산
    return parity

def rebuild_data(stripe_data, failed_disk_index):
    """
    장애가 발생한 디스크의 데이터를 복구하는 함수
    stripe_data: 현재 스트라이프의 데이터 및 패리티 리스트 (0이면 장애 상태 가정)
    """
    result = 0
    for i, data in enumerate(stripe_data):
        if i != failed_disk_index:
            result ^= data
    return result # 유실된 데이터 복원
```

📢 **섹션 요약 비유**: 책의 한 페이지를 고치는 작업을 할 때, 수정본과 **기존 본문, 그리고 페이지 바깥의 정답지(패리티)를 모두 읽어들여** 새로운 정답지를 계산한 뒤, 수정본과 새 정답지를 동시에 써야 하는 복잡한 행정 절차와 같습니다. 이로 인해 단순 수정이지만 4배의 업무량(쓰기 페널티)이 발생합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 심층 기술 비교 분석표**

RAID 5와 타 주요 RAID 레벨(RAID 10, RAID 6) 간의 정량적 비교는 시스템 아키텍처 결정에 중요한 근거가 된다. (비교 대상: 총 4개 디스크, 1TB each 기준)

| 비교 항목 | RAID 10 (Mirror+Stripe) | RAID 5 (Distributed Parity) | RAID 6 (Dual Parity) |
|:---|:---:|:---:|:---:|
| **총 용량** | 2 TB (50%) | **3 TB (75%)** | 2 TB (50%) |
| **결함 허용 (FT)** | 1개(그룹 내 1개) | **1개** | **2개** |
| **읽기 성능** | 매우 빠름 (N disks) | **빠름 (N-1 disks)** | 빠름 (N-2 disks) |
| **쓰기 성능** | 빠름 (2 I/Os) | 느림 (4 I/Os + Penalty) | 매우 느림 (6 I/Os) |
| **리빌드 시간** | 빠름 (복사만) | 느림 (재계산 필요) | 매우 느림 |
| **최소 디스크 수** | 2 (보통 짝수) | **3** | 4 |
| **비용 효율성** | 낮음 | **높음** | 중간 |

**2. OS 및 네트워크 융합 분석**
- **OS (Operating System) 관점**: Linux의 **MDADM (Multiple Device Administrator)** 드라이버를 통해 소프트웨어적으로 구현 시, CPU의 부하가 RAID 계산에 의해 증가한다. 반면 Windows Server의 Storage Spaces는 **ReFS (Resilient File System)**와 연동하여 블록 복제(Clone)와 계층화(Storage Tiers)를 제공한다.
- **Network Security 관점**: NAS(Network Attached Storage) 환경에서 RAID 5를 구성할 경우, 디스크 장애 발생 시 복구(Rebuild) 도중 네트워크 트래픽이 급증할 수 있으므로 네트워크 대역폭(Bandwidth)에 대한 모니터링이 필요하다. 또한, 데이터 보호가 물리적 장애 1개로 한정되므로 랜섬웨어(Ransomware) 같은 논리적 공격에 대해서는 RAID만으로는 방어가 불가능하며 별도의 백업 체계가 필수적이다.

**3. 장애 복구 시나리오 비교 (ASCII 다이어그램)**

RAID 5(단일 패리티)와 RAID 6(이중 패리티)의 신뢰성 차이를 시각화한다.

```text
[ Scenario: 1st Disk Failure Detected ]

   RAID 5 State:                RAID 6 State:
 +----------+                 +----------+
 | Disk 0   | FAIL!           | Disk 0   | FAIL!
 | (Data)   |                 | (Data)   |
 +----------+                 +----------+
   [ CRITICAL ]                 [ DEGRADED ]
   (Data reading OK)            (Data reading OK)
   (NO Safety Net)              (Parity Q Protects)


[ Scenario: URE Found During Rebuild ]

   RAID 5 Result:               RAID 6 Result:
 +------------------+          +------------------+
 | DATA LOST        |          | REBUILD SUCCESS  |
 | Volume Corrupted |          | Data Intact      |
 +------------------+          +------------------+
   [ FATAL ]                   [ HEALTHY ]
```

> **해설**: RAID 5는 리빌드 중 2번째 오류(URE 또는 디스크 장애)가 발생하면 즉시 볼륨이 파괴되는 '올 또는 낫(All or Nothing)' 구조를 가진다. 반면 RAID 6은 추가 패리티(Q) 덕분에 리빌드 중에도 한 번 더 버틸 수 있다. 대용량 디스크(10TB+) 시대에는 URE 발생 확률이 높아져 RAID 6이 필수적인 이유이다.

📢 **섹션 요약 비유**: 안전벨트를 두르는 것과 같습니다. RAID 5는 일반적인 안전벨트라면, RAID 10은 공간을 많이 차지하지만 튼튼한 2중 바디수트, RAID 6은 안전벨트 위에 에어백까지 추가한 형태입니다. 고속주행(대용량 리빌드) 중 사고가 나면 RAID 5는 단순 벨트만으로는 충격을 못 이겨 부서질 위험이 큽니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. 실무 시나리오 및 의사결정 매트릭스**

**Case A: 중소기업 문서 관리 서버**
- **상황**: 예산 제한이 있고, 읽기(Read) 위주의 정적 파일(PDF, 이미지)을 저장함. 디스크는 4TB x 4개.
- **결정**: **RAID 5 선정**.
- **이유**: 총 용량 12TB를 확보하여 가성비를 극대화하고, 문서 읽기 속도가 빠르므로 사용자 경험이 향상됨. 쓰기가
