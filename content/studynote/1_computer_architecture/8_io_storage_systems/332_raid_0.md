# # [332. RAID 0 (스트라이핑)]

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: RAID (Redundant Array of Independent Disks) 0은 패리티나 미러링 없이 순수한 **데이터 스트라이핑(Data Striping)** 만을 수행하여, $N$개의 디스크 대역폭을 합산하여 이론적인 성능을 $N$배로 끌어올리는 최적의 성능 아키텍처이다.
> 2. **가치**: 중복 저장 공간이 전혀 없어 **용량 효율성(Storage Efficiency)**이 100%이며, 순차 및 랜덤 I/O 대역폭을 극대화하여 대용량 멀티미디어 처리나 스크래치 디스크(Scratch Disk) 환경에서 필수적이다.
> 3. **융합**: OS (Operating System)의 파일 시스템과 하드웨어 컨트롤러 간의 인터페이스를 통해 구현되며, 단일 디스크 장애 시 전체 데이터 손실이라는 치명적인 RPO (Recovery Point Objective) 영향을 고려한 백업 전략이 반드시 수반되어야 한다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 정의**
RAID 0는 **RAID (Redundant Array of Independent Disks)** 레벨 중 가장 기초적이면서도 성능에만 집중한 구조입니다. 여러 개의 물리적 디스크(HDD, SSD 등)를 논리적으로 결합하여, 운영 체제(OS) 입장에서는 하나의 대용량 디스크로 인식되도록 만듭니다. 핵심은 **데이터 분할(Data Striping)** 기술로, 하나의 큰 데이터 블록을 '스트라이프(Stripe)' 단위로 쪼개어 물리적인 디스크들에 나누어 동시에 기록/읽기를 수행합니다.

**2. 등장 배경 및 필요성**
과거 단일 디스크의 회전 속도(RPM)나 인터페이스 대역폭(SATA/SAS 등)의 물리적 한계로 인해, 고화질 비디오 편집이나 대규모 과학 계산 등 대용량 데이터를 처리하는 데 병목이 발생했습니다. 이를 해결하기 위해 디스크 한 개당 처리량은 그대로 유지하면서, 디스크 개수를 늘려 총 처리량(Total Throughput)을 선형적으로 증가시키고자 하는 패러다임에서 등장했습니다.

**💡 비유**
한 명의 비서가 문서를 처리하는 것보다, 문서를 페이지별로 나누어 여러 명의 비서가 동시에 타이핑하여 처리 속도를 높이는 방식과 유사합니다.

| 구분 | 단일 디스크 (Single Disk) | RAID 0 (Striping) |
|:---:|:---:|:---:|
| **데이터 처리 방식** | 직렬 (Serial) | 병렬 (Parallel) |
| **성능 (이론적)** | 1x | N배 (디스크 수 N) |
| **용량 효율** | 100% | 100% (합산) |
| **안전성** | 단일 장애 시 일부 손실 | 단일 장애 시 **전체 파괴** |

**📢 섹션 요약 비유:** 마치 넓은 밭을 혼자 쟁기로 갈면 느리지만, 밭을 여러 구역으로 나누어 여러 명이 동시에 일을 진행하면 작업 시간이 획기적으로 단축되는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. 구성 요소 및 상세 동작**
RAID 0 컨트롤러(Hardware 혹은 Software Driver)는 호스트(Host)로부터 전달된 논리 블록 주소(LBA, Logical Block Address)를 수학적으로 계산하여 각 물리 디스크의 섹터(Sector)로 매핑합니다.

```text
   Host OS (Logical Volume)
         |
         v
+-------------------------------------------------------+
|              RAID 0 Controller (Driver/H/W)           |
|   - Stripe Size: 64KB (Configurable Parameter)        |
|   - Algorithm: Modulo N (Chunk Index % Disk Count)    |
+-------------------------------------------------------+
      |            |            |
      v            v            v
+-----------+-----------+-----------+-----------+
| Physical  | Physical  | Physical  | Physical  |
| Disk 0    | Disk 1    | Disk 2    | Disk 3    |
+-----------+-----------+-----------+-----------+
| Chunk 0   | Chunk 1   | Chunk 2   | Chunk 3   |
| Chunk 4   | Chunk 5   | Chunk 6   | Chunk 7   |
+-----------+-----------+-----------+-----------+
```

**[도해 해설]**
1.  **요청(Request):** 애플리케이션이 256KB 크기의 데이터 쓰기를 요청.
2.  **분할(Split):** 컨트롤러는 이를 64KB(Stripe Size) 단위로 쪼개어 4개의 청크(Chunk) 생성.
3.  **분산(Dispatch):** 청크 0→Disk 0, 청크 1→Disk 1, 청크 2→Disk 2, 청크 3→Disk 3으로 동시 전송.
4.  **완료(Completion):** 가장 늦린 디스크의 기록이 완료되는 시점이 전체 지연 시간(Latency)이 됨.

**2. 핵심 파라미터 분석**
-   **Stripe Size (Chunk Size):** 시스템 성능을 결정하는 가장 중요한 변수입니다.
    -   *작은 Stripe Size (예: 4KB):* 랜덤 I/O(Random I/O) 환경에서 유리. 여러 디스크에 작업이 균등하게 분산되어 부하 분산 효과가 크나, 디스크 헤드(Head)의 이동(Seek Time)이 잦아져 전체 대역폭이 낮아질 수 있음.
    -   *큰 Stripe Size (예: 256KB, 1MB):* 순차 I/O(Sequential I/O) 환경에서 유리. 대용량 파일 전송 시 각 디스크가 순차적으로 읽어 전체 대역폭(Throughput)을 극대화함.

**3. 내부 동작 메커니즘 (Pseudo Code)**
```python
# RAID 0 Address Mapping Algorithm
def get_physical_disk_and_offset(logical_block_address, num_disks, blocks_per_stripe):
    # Calculate Stripe Index
    stripe_index = logical_block_address // blocks_per_stripe
    
    # Calculate Disk Index (Round-Robin)
    disk_index = stripe_index % num_disks
    
    # Calculate Local Block Offset on target disk
    local_offset = (logical_block_address % blocks_per_stripe) + \
                   (stripe_index // num_disks) * blocks_per_stripe
    
    return disk_index, local_offset
```
> **해설:** 위 로직은 OS의 파일 시스템(File System)이 보내는 LBA(Logical Block Address)를 물리적 디스크의 좌표로 변환하는 과정입니다. 나머지 연산(Modulo)을 통해 디스크를 순차적으로 할당하는 단순무결한 구조이므로 연산 오버헤드(Overhead)가 거의 없습니다.

**📢 섹션 요약 비유:** 긴 빨대 한 개로 마시는 것보다, 빨대 여러 개를 모아서 빨대뭉치를 만들어 빨대 사이로 맹물이 동시에 통과하게 하면 물을 마시는 속도가 훨씬 빨라지는 원리와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. RAID 레벨 심층 기술 비교**
RAID 0는 다른 레벨과 비교할 때 명확한 성능 우위를 보이지만, 신뢰성 측면에서는 최약점을 가집니다.

| 비교 항목 | RAID 0 (Striping) | RAID 1 (Mirroring) | RAID 5 (Parity) | RAID 10 (1+0) |
|:---|:---|:---|:---|:---|
| **최소 디스크 수** | 1개 (기능상 2개 권장) | 2개 | 3개 | 4개 |
| **용량 효율** | **100%** (합산) | 50% (사이즈 중 작은 기준) | (N-1)/N | 50% |
| **순차 읽기 성능** | **최상 (N배)** | 상 (N배 혹은 단일) | 상 (N배) | **최상 (N배)** |
| **순차 쓰기 성능** | **최상 (N배)** | 중 (쓰기 1회, 복제 1회) | 하 (Parity 연산 병목) | 상 (미러링 동시) |
| **내결함성 (Fault Tolerance)** | **없음 (0)** | 있음 (1개) | 있음 (1개) | 있음 (1개/그룹) |
| **주요 용도** | 임시 파일, 캐시, 비디오 편집 | OS, 부팅 디스크, 중요 데이터 | 파일 서버, 일반 데이터 | 데이터베이스, 고성능/안정성 |

**2. 데이터베이스(DB) 및 OS 융합 관점**
-   **DBMS (Database Management System):** RAID 0는 데이터베이스의 **TempDB**나 정렬(Sort) 작업을 위한 임시 영역에 사용될 수 있습니다. 트랜잭션 로그(Transaction Log)는 순차 쓰기가 많으므로 RAID 0의 대역폭을 이용할 수 있으나, 단일 장애로 복구 불가능하므로 실무에서는 로그(Log) 저장소로 사용하지 않습니다.
-   **OS 메모리 관리:** 시스템의 **Swap Memory**(또는 Paging File) 영역으로 RAID 0를 사용하면, 메모리 부족(Swap Thrashing) 상황에서 디스크 I/O 병목을 줄여 시스템 다운(Downtime) 위험을 낮출 수 있습니다. 손실되어도 무방한 스왑 데이터의 특성과 RAID 0의 특성이 부합합니다.

**3. 정량적 성능 분석 (의사결정 매트릭스)**
-   **Read/Write Latency:** 디스크 1개의 평균 탐색 시간이 10ms라도, 4개 디스크로 구성된 RAID 0은 병렬 처리를 통해 유효 지연 시간(Effective Latency)을 대폭 감소시킴.
-   **Throughput (TPS):** 단일 SSD가 500MB/s를 낼 때, 4개의 SSD를 RAID 0로 구성 시 이론적으로는 2GB/s의 대역폭을 확보할 수 있어, 초당 트랜잭션 처리 수(TPS)가 I/O 바운드(IO Bound)인 시스템에서 급격한 성능 향상을 기대할 수 있음.

**📢 섹션 요약 비유:** 안전벨트와 에어백이 없는 레이싱 카트(RAID 0)를 타는 것과 같습니다. 최고의 속도를 낼 수 있지만, 사고(장애)가 발생하면 사람(데이터)을 보호할 장치가 전혀 없습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. 실무 도입 체크리스트 (Pre-flight Checklist)**
RAID 0 도입을 고려할 때 기술사(PE)는 다음의 항목을 엄격히 검토해야 합니다.

| 검토 구분 | 항목 | 세부 검토 내용 (Due Diligence) |
|:---|:---|:---|
| **기술적** | I/O 패턴 분석 | Random I/O가 주를 이루는가, Sequential I/O가 많은가? (Random 시 성능 이득이 떨어질 수 있음) |
| | 데이터 크기 | 블록 크기(Block Size)가 Stripe Size의 배수인가? (Misalignment 방지) |
| **운영적** | 백업 전략 | RPO (Recovery Point Objective)가 '0'에 수렴하는가? (실시간 백업 혹은 복제 구성 필수) |
| | 장애 탐지 | 디스크 온도나 S.M.A.R.T 정보 모니터링이 실시간으로 이루어지는가? |
| **비즈니스** | 데이터 평가 | 데이터 손실 시 재구축이 가능한가? (소스가 있는 렌더링 파일, 캐시 등인가?) |

**2. 실무 시나리오 및 의사결정 프로세스**

**[Case 1: 비디오 편집工作站(NLE Station) 설계]**
-   **문제:** 8K RAW 비디오 편집 중 버퍼링(Frame Drop) 발생.
-   **해결:** NVMe SSD 4개를 Hardware RAID 0 (8TB)로 구성.
-   **판단:** 대역폭 약 3GB/sec 확보. 작업 파일은 미디어 서버에 있고 로컬은 작업용(Scratch)이므로 장애 발생 시 재복사가 가능함. 따라서 RAID 0 도입이 타당함.

**[Case 2: 고성능 웹 서버의 세션 저장소]**
-   **문제:** 대규모 트래픽 처리 시 세션 파일 I/O 병목.
-   **해결:** SSD 2개를 Software RAID 0로 구성하여 `/tmp` 마운트.
-   **판단:** 세션 데이터는 일시적이며 장애 시 사용자 로그아웃으로 간주할 수 있는 수준의 임계점이므로, 속도 이득이 리스크(Risk)보다 큼. 단, 중요 트랜잭션 데이터는 절대 배제.

**3. 안티패턴 (Anti-Pattern) - 치명적 결함 사례**
-   **금지 사항:** 중요한 고객 데이터(Customer PII), 재무 장부(Ledger), 소스 코드 저장소(Git Repository)를 RAID 0에 저장하는 것.
-   **결과:** 단일 디스크의 **UNC 오류(Uncorrectable Error)** 혹은 컨트롤러 연결 끊김 즉시, 모든 데이터가 파편화되어 복구 불가능한 상태로 변질됨. 마치 단 하나의 못이 빠져도 다리가 무너지는 구조와 같음.

**📢 섹션 요약 비유:** 줄타기 곡예사가 안전망(백업)을 치지 않은 상태에서 톱니바퀴 위를 달리는 것과 같습니다. 성능은 극대화되지만, 실수를 할 수 있는 여지조차 허용되지 않는 극도의 위험성을 내포합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**1. 정량/정성 기대효과 (ROI)**
RAID 0 도입을 통해 기대할 수 있는 이익은 다음과 같습니다.

| 항목 | 도입 전 (Single Disk) | 도입 후 (RAID 0, N=4) | 비고 |
|:---|:---|:---|:---|
| **순차 쓰기 속도** | 500 MB/s | **2,000 MB/s** | 선형적 증가 |
| **용량 효율** | 4 TB (Disk 1개) | **16 TB** (4TB
