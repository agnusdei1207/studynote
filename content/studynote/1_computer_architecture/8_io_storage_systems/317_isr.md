# ISR (Interrupt Service Routine)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ISR(인터럽트 서비스 루틴, 또는 인터럽트 핸들러)은 인터럽트가 발생했을 때 하드웨어의 요청이나 오류를 **실질적으로 수습하고 처리하기 위해 운영체제(OS) 커널 내부에 작성된 소프트웨어 함수(코드)**다.
> 2. **가치**: CPU가 일상적인 유저 프로그램(엑셀, 게임 등)을 실행하다가도, ISR로 진입하는 순간 최고 권한(Ring 0)을 획득하여 마우스의 좌표를 계산하거나 디스크의 데이터를 메모리로 퍼 나르는 등 **하드웨어의 심박수를 유지**시킨다.
> 3. **융합**: ISR은 실행되는 동안 다른 인터럽트를 막아버려 시스템을 지연시킬 수 있으므로, 최소한의 일만 하고 빠지는 **Top Half**와 무거운 작업을 백그라운드로 넘기는 **Bottom Half** 아키텍처로 융합 설계된다.

---

## Ⅰ. ISR의 개요

### 1. 정의
- 하드웨어나 소프트웨어가 인터럽트(Interrupt)를 발생시켰을 때, 해당 인터럽트 벡터(Vector)가 가리키는 메모리 주소에 상주하며, 즉각적으로 호출되어 이벤트를 처리하고 CPU의 원래 상태를 복구해 주는 OS 커널의 핵심 루틴이다.

### 2. 등장 배경: '알람이 울렸으니 누군가는 불을 꺼야 한다'
- 하드웨어가 삐용삐용 알람(IRQ)을 울리는 건 훌륭하지만, 알람만 울린다고 밥이 지어지는 것은 아니다. 키보드 버튼이 눌렸다면, 그 눌린 알파벳이 'A'인지 'B'인지 칩에서 읽어와서 메모리 구석에 예쁘게 적어두는 구체적인 행동 대원이 필요했다. OS 개발자들은 장치마다 그 장치를 전담 마크하는 소방관(소프트웨어 코드)을 미리 메모리에 고용해 두었고, 그것이 바로 ISR이다.

### 3. 💡 비유: '119 구조대의 출동 매뉴얼'
- ISR은 **'화재 알람(인터럽트)이 울렸을 때 즉각 출동하여 불을 끄는 119 구조대원의 구체적인 행동 매뉴얼'**과 같다.
- 알람이 울리면, 하던 식사(기존 프로그램)를 멈추고 장비를 챙긴다(상태 백업). 그리고 화재 현장으로 뛰어가 매뉴얼대로 불을 끈 뒤(ISR 실행), 다시 소방서로 돌아와 식던 밥을 마저 먹는다(복귀).

- **📢 섹션 요약 비유**: 벨이 울린 건 하드웨어의 몫이지만, 문을 열고 나가서 택배를 받아 방 안에 예쁘게 놓아두는 모든 실질적 육체노동은 ISR이라는 소프트웨어의 몫입니다.

## Ⅱ. 메커니즘 및 ISR의 라이프사이클 (Deep Dive)

ISR은 일반 함수(Function)와 비슷해 보이지만, 진입과 복귀 과정이 매우 엄격하고 무겁다.

### 1. ISR 실행의 5단계 시퀀스
1. **문맥 저장 (Context Save)**: 인터럽트가 걸린 순간, CPU는 돌아갈 원래 프로그램의 주소(PC)와 상태 레지스터(Status Flags), 그리고 범용 레지스터 값들을 모조리 스택(Stack) 메모리에 푸시(Push)하여 백업한다.
2. **벡터 테이블 조회**: 찌른 장치의 인터럽트 벡터 번호를 보고 해당 ISR의 시작 주소로 PC를 점프시킨다.
3. **ISR 본문 실행**: 디바이스 컨트롤러의 버퍼에서 데이터를 읽어와 OS 큐(Queue)에 넣거나, 칩에 "잘 받았어(ACK)"라는 신호를 보내 하드웨어의 알람을 끈다.
4. **문맥 복구 (Context Restore)**: 스택에 저장해 두었던 레지스터 값들을 팝(Pop)하여 CPU 내부에 원래대로 꽉 채워 넣는다.
5. **인터럽트 복귀 (IRET)**: 특수한 복귀 명령어(`IRET`, Interrupt Return)를 실행하여 원래 멈췄던 프로그램의 바로 다음 줄 코드로 감쪽같이 되돌아간다.

### 2. 일반 함수 호출(CALL) vs ISR 호출(INT) 비교
| 비교 항목 | 일반 함수 호출 (`CALL`) | ISR 진입 (`INT` / 하드웨어 예외) |
|:---|:---|:---|
| **발생 시점** | 소스 코드에 적힌 **예측 가능한 시점 (동기적)** | **아무 때나 갑자기 튀어나옴 (비동기적)** |
| **백업 대상** | PC(돌아올 주소) 정도만 간단히 백업 | PC는 물론 상태 플래그 등 **모든 CPU 문맥 백업** |
| **실행 권한** | 일반 유저 권한 (Ring 3) 그대로 유지 | **막강한 커널 권한 (Ring 0)으로 수직 상승** |
| **종료 명령어** | `RET` (단순 복귀) | `IRET` (복귀 + 권한 강등 + 플래그 복구) |

- **📢 섹션 요약 비유**: 일반 함수 호출이 친구 집 문을 정중히 노크하고 들어가는 것이라면, ISR 진입은 경찰이 특수 권한(Ring 0)을 가지고 헬기를 타고 창문으로 난입해 상황을 정리하는 스케일이 다른 작전입니다.

## Ⅲ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무의 절대 원칙: "ISR은 무조건 짧고 빨라야 한다"
- **상황**: 초보 드라이버 개발자가 랜 카드 ISR 내부에서 패킷의 암호를 풀고 데이터베이스에 저장하는 수천 줄짜리 로직을 짰다.
- **판단 (재앙 발생)**: ISR이 실행되는 동안 CPU는 보통 다른 인터럽트를 **마스킹(무시)**해버린다. 랜 카드 ISR이 0.5초 동안 혼자 CPU를 꽉 잡고 있으면, 사용자가 마우스를 움직여도 마우스가 안 움직이고 키보드도 먹통이 된다(시스템 렉). 시니어 커널 개발자는 이 재앙을 막기 위해 **"Top Half / Bottom Half"** 패턴을 강제한다.
  - **Top Half (진짜 ISR)**: 장치에서 데이터만 빼서 메모리 큐에 던지고(1마이크로초 컷), 장치에 ACK만 날리고 즉시 ISR을 종료해 버린다. 다른 인터럽트가 들어오게 길을 비켜준다.
  - **Bottom Half (Tasklet/Workqueue)**: OS가 여유로울 때 큐에 쌓인 데이터를 백그라운드 스레드로 천천히 분석하고 암호를 푼다.

### 2. 기술사적 인사이트: 중첩 인터럽트 (Nested Interrupt) 처리
- **인사이트**: ISR이 실행 중인데, 더 중요하고 다급한 인터럽트(예: 시스템 전원 꺼짐 경고)가 들어오면 어떻게 할까? 
현대 시스템은 인터럽트에도 계급(우선순위, Priority)을 둔다. 우선순위가 낮은 키보드 ISR이 도는 도중에, 우선순위가 높은 하드디스크 ISR이 들어오면 하드웨어(PIC)가 과감하게 기존 ISR을 멈추고 새 ISR로 다시 점프하는 **중첩 인터럽트(Nested Interrupt)**를 허용한다. 마치 스택처럼 겹겹이 쌓이는 이 관리는 고도의 OS 스케줄링 예술이다.

- **📢 섹션 요약 비유**: 교통사고(일반 ISR) 수습 중인데, 뒤에서 산불(최고 중요도 인터럽트)이 번져오면 교통사고를 잠시 버려두고 산불부터 끄러 가는 완벽한 위기관리 매뉴얼입니다.

## Ⅳ. ISR과 디바이스 드라이버의 관계

### 1. 드라이버의 꽃, ISR 등록 (Request_irq)
- 새로운 그래픽카드나 마우스를 사서 끼우면, OS는 그 장치가 뭔지 모른다. 장치 제조사(NVIDIA, 로지텍)가 만든 **디바이스 드라이버(Device Driver)**를 설치해야 한다.
- 드라이버가 커널에 설치될 때 하는 가장 중요한 일이 바로 "운영체제님, 44번 인터럽트 들어오면 제가 짠 이 함수(ISR)를 실행시켜주세요!"라고 핑거 프린트를 남기는 **`request_irq()` 시스템 콜 등록 과정**이다.

### 2. 예외 처리기(Exception Handler)와의 구분
- 외부 장치가 찌르는 건 보통 ISR이라 부르고, CPU 내부에서 0으로 나누거나 페이지 폴트가 터졌을 때 불려 가는 코드는 **예외 처리기(Exception Handler)**라고 부른다. 본질적인 진입 메커니즘은 완벽히 동일하다.

- **📢 섹션 요약 비유**: 드라이버 설치는 새로운 용병을 고용한 뒤, 콜센터(OS)에게 "이런 민원이 들어오면 이 용병(ISR)에게 전화 넘기세요"라고 내선 번호를 등록하는 행정 절차입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **운영체제의 생명줄**: OS는 사실상 '인터럽트에 반응하여 ISR을 실행하는 거대한 이벤트 반응형 소프트웨어 덩어리'다. ISR이 없으면 OS는 영원히 잠들어 있을 것이다.
- **하드웨어 제어의 추상화**: 복잡한 기계적 제어를 깔끔한 소프트웨어 함수(C언어)로 감싸서, 애플리케이션 개발자가 마우스 포인터의 좌표를 쉽게 얻어 쓸 수 있게 해줌.

### 2. 미래 전망
과거에는 모든 짐을 CPU의 ISR이 감당했지만, 이제는 '지능형 네트워크 카드(SmartNIC)'나 GPU 내부에 아예 ARM 코어를 박아 넣고 있다. 외부 장치가 알아서 데이터를 지지고 볶은 뒤 완전히 요약된 결과물 한 줄만 CPU에 알려주는 시대다. CPU 커널 단의 ISR은 점점 더 가벼워지고 있으며, 무거운 인터럽트 처리 로직은 하드웨어 가속기(Offloading) 쪽으로 대거 이주하는 중이다.

- **📢 섹션 요약 비유**: 모든 민원(인터럽트)을 시장(CPU) 비서실이 일일이 처리하던 시대에서, 이제는 각 구청장(스마트 I/O)이 알아서 문제를 다 해결하고 월말 보고서만 딸랑 한 장 올리는 고도의 지방자치제로 발전하고 있습니다.
