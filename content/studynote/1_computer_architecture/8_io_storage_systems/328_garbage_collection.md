# 가비지 컬렉션 (SSD Garbage Collection)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSD 가비지 컬렉션(GC)은 NAND 플래시의 **'삭제 후 쓰기(Erase Before Write)' 특성**으로 인해 발생하는 유효/무효 데이터가 섞인 블록을 정리하여 **새로운 쓰기 공간을 확보**하는 백그라운드 작업이다.
> 2. **가치**: 무효 데이터가 점유한 공간을 회수하지 않으면 SSD는 새 데이터를 쓸 수 없어 성능이 급락하므로, **지속적인 쓰기 성능과 수명 유지**에 필수적이다.
> 3. **융합**: FTL(Flash Translation Layer)과 긴밀히 협력하며, **쓰기 증폭(Write Amplification)**, 마모 평준화(Wear Leveling), TRIM 명령과 연계하여 최적화된다.

+++

## Ⅰ. 가비지 컬렉션의 개요

### 1. 정의
- SSD 가비지 컬렉션은 NAND 플래시에서 삭제된(논리적으로 무효화된) 데이터가 차지한 공간을 물리적으로 회수하는 과정이다. 사용자 데이터 삭제 시 즉시 물리 삭제가 일어나지 않고, GC가 나중에 정리한다.

### 2. 등장 배경: NAND의 P/E 사이클 제약
- NAND 플래시은 **페이지 단위 쓰기, 블록 단위 삭제**만 가능하다.
- 한 블록에 유효 데이터와 무효 데이터가 섞여 있으면, 무효 데이터만 삭제할 수 없다.
- 따라서 **유효 데이터를 다른 곳으로 복사 후 블록 전체를 삭제**해야 한다.

### 3. 💡 비유: '정리가 필요한 지저분한 방'
- SSD는 **'물건을 가져다 놓기만 하고 치우지 않는 방'**과 같다.
- 사용자가 "이 물건 버려"라고 해도 물건은 그 자리에 있다. 청소부(GC)가 나중에 와서 진짜 쓰레기를 분리하고, 아직 쓰는 물건은 다른 곳으로 옮긴 뒤 방 전체를 청소한다.

- **📢 섹션 요약 비유**: 쓰레기통이 없는 방에서 쓰레기를 치우려면, 쓰레기와 살림을 분리해서 살림은 옆방으로 옮기고 방 전체를 청소해야 합니다.

## Ⅱ. 동작 원리 및 메커니즘 (Deep Dive)

### 1. 가비지 컬렉션의 단계
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │              SSD 가비지 컬렉션 동작 과정                          │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [1단계] 블록 선정 (Victim Block Selection)                     │
    │   - 무효 페이지 비율이 높은 블록 선택                            │
    │   - Wear Leveling 고려 (너무 많이 쓴 블록 회피)                 │
    │                                                                 │
    │  [2단계] 유효 데이터 복사 (Valid Data Copy)                      │
    │   - 선택된 블록의 유효 페이지를 새 블록으로 복사                 │
    │   - FTL 매핑 테이블 업데이트                                    │
    │                                                                 │
    │  [3단계] 블록 삭제 (Erase)                                       │
    │   - 기존 블록 전체 삭제 → Free Block 풀에 반환                  │
    │                                                                 │
    │  [비용]                                                          │
    │   - 읽기 + 쓰기 + 삭제 = 3배 I/O 발생                           │
    │   - 이것이 쓰기 증폭(Write Amplification)의 주원인              │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. 블록 상태 다이어그램
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │              블록 내 페이지 상태 (GC 전/후)                       │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [GC 전: Victim Block]           [GC 후: Free Block]            │
    │  ┌───┬───┬───┬───┐              ┌───┬───┬───┬───┐              │
    │  │ V │ I │ V │ I │   ──복사──▶  │   │   │   │   │  (삭제됨)    │
    │  ├───┼───┼───┼───┤              └───┴───┴───┴───┘              │
    │  │ I │ V │ I │ I │                                              │
    │  └───┴───┴───┴───┘              [새 블록으로 복사된 데이터]     │
    │   V=Valid, I=Invalid            ┌───┬───┐                       │
    │                                 │ V │ V │                       │
    │  * 8개 페이지 중 3개만 유효      └───┴───┘                       │
    │  * 5개는 쓰레기(Garbage)                                        │
    │  * 3개를 복사 후 전체 삭제                                       │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. 쓰기 증폭 (Write Amplification)
- **WA = 실제 NAND 쓰기 양 / 호스트 요청 쓰기 양**
- GC로 인해 호스트가 1MB 썼는데 NAND 내부에서는 2~3MB 쓰기가 발생할 수 있다.
- WA가 높을수록 SSD 수명 단축, 성능 저하.

- **📢 섹션 요약 비유**: 1페이지 쓰레기를 버리려고 3페이지 분량의 일을 해야 하는 셈입니다.

## Ⅲ. GC 유형 및 최적화 기법

### 1. GC 유형
| 유형 | 설명 | 장점 | 단점 |
|:---|:---|:---|:---|
| **Background GC** | SSD가 idle할 때 미리 수행 | 사용자 I/O 영향 없음 | Idle 시간 필요 |
| **Foreground GC** | 쓰기 요청 시 급히 수행 | 공간 부족 해결 | 사용자 I/O 지연 |
| **Idle GC** | 백그라운드 GC의 다른 이름 | - | - |

### 2. 최적화 기법
- **Over-provisioning (OP)**: 사용자에게 보이지 않는 여유 공간 확보 → GC 부담 감소.
- **TRIM 명령**: OS가 삭제된 블록 정보를 SSD에 알려줌 → GC가 무효 데이터를 미리 파악.
- **Dynamic/Static Wear Leveling**: 특정 블록 과다 사용 방지.

+++

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: 쓰기 집약적 워크로드와 GC
- 데이터베이스, 로그 서버 등 쓰기 집약적 환경에서는 GC가 병목이 된다.
- **대응**: 높은 OP 비율 SSD 사용, 쓰기 버퍼(Write Buffer) 활용, 읽기 중심 설계.

### 2. 안티패턴: SSD 꽉 채우기
- SSD 용량을 100% 사용하면 GC를 위한 Free Block이 부족해져 성능이 급락.
- **권장**: 최소 10~20% 여유 공간 유지.

### 3. TRIM과 GC의 관계
- TRIM이 없으면 SSD는 어떤 데이터가 삭제되었는지 알 수 없다.
- TRIM → FTL이 무효 페이지 표시 → GC가 효율적으로 블록 선정.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **지속적 쓰기 성능**: GC 없으면 SSD는 새 쓰기를 못함.
- **수명 연장**: 효율적 GC로 WA 감소 → P/E 사이클 절약.

### 2. 미래 전망
- **ZNS (Zoned Namespace) SSD**: 호스트가 쓰기 패턴을 관리하여 GC 오버헤드 최소화.
- **컴퓨테이셔널 스토리지**: GC를 스마트하게 수행하는 온디 스토리지 컴퓨팅.

## 📌 관련 개념 맵 (Knowledge Graph)
- **FTL (Flash Translation Layer)**: GC를 수행하는 주체.
- **마모 평준화 (Wear Leveling)**: GC와 연계된 수명 관리 기법.
- **SSD (Solid State Drive)**: GC가 동작하는 저장장치.
- **TRIM 명령**: GC 효율 향상을 위한 OS-SSD 통신.
