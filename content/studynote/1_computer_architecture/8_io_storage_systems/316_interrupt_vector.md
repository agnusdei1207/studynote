# 인터럽트 벡터 (Interrupt Vector)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터럽트 벡터(Interrupt Vector)는 키보드, 마우스, 디스크 등 수많은 장치에서 터져 나오는 인터럽트를 구분하기 위해 부여된 **고유한 번호(식별자)이자, 그 번호에 해당하는 처리 프로그램(ISR)의 시작 메모리 주소**를 의미한다.
> 2. **가치**: CPU가 어떤 장치에서 인터럽트가 발생했는지 하나하나 묻고 다니는(폴링) 비효율을 없애고, 벡터 테이블을 통해 **단 1클럭 만에 즉각적으로 해당 장치의 전담 처리 코드로 점프(분기)**할 수 있게 해주는 핵심 연결 고리다.
> 3. **융합**: 운영체제는 부팅 시 메모리의 특정 구역에 이 '인터럽트 벡터 테이블(IVT)'을 융합 구축해 두며, 하드웨어 컨트롤러(PIC)가 던져준 벡터 번호가 OS의 소프트웨어 루틴을 깨우는 브릿지 역할을 한다.

---

## Ⅰ. 인터럽트 벡터의 개요

### 1. 정의
- 특정 인터럽트 발생 시, 그 인터럽트를 처리할 수 있는 인터럽트 서비스 루틴(ISR, Interrupt Service Routine)이 저장된 메모리의 실제 시작 주소, 또는 그 주소를 가리키는 포인터(인덱스 번호)다.

### 2. 등장 배경: '누가 불렀는지 어떻게 알아?'
- CPU에 인터럽트 핀(Pin)은 하나뿐이다. 마우스, 키보드, 하드디스크가 동시에 이 핀에 전기를 쏴서 CPU를 깨웠다. CPU는 깨어났지만 "도대체 누가 날 부른 거지?" 알 길이 없었다. 처음에는 모든 장치에 일일이 "네가 불렀어?" 하고 물어보러 다녔다(Vectored 되지 않은 인터럽트). 이 바보짓을 멈추기 위해, 하드웨어 컨트롤러가 CPU를 찌를 때 **"난 3번(키보드)이야!"라고 자기 명찰 번호(Vector)를 함께 던져주도록** 규칙을 바꾼 것이 인터럽트 벡터의 시작이다.

### 3. 💡 비유: '호텔 로비의 내선 전화와 방 번호'
- 인터럽트 벡터는 **'프런트 데스크에 울리는 내선 전화의 객실 번호 표시'**와 같다.
- 101호(키보드), 205호(마우스) 등에서 동시에 룸서비스를 요청할 때, 프런트 직원은 직접 방을 찾아가서 물어볼 필요가 없다. 전화기 액정에 뜬 '205호(벡터 번호)'를 보고, 장부(벡터 테이블)에서 '205호는 청소 요청'이라는 대응 매뉴얼(ISR)을 바로 펼쳐서 즉시 룸서비스 직원을 파견하면 된다.

- **📢 섹션 요약 비유**: 신고자가 누군지 묻지 않고 일단 출동하는 게 아니라, 112 상황실에서 콜러의 위치와 신원(Vector)을 즉시 파악해 가장 적합한 부대(ISR)를 즉각 투입하는 최첨단 지휘 통제 시스템입니다.

## Ⅱ. 메커니즘 및 벡터 테이블 (Deep Dive)

CPU는 이 벡터 번호를 받아서 메모리의 'IVT'라는 표를 뒤진다.

### 1. IVT (Interrupt Vector Table) / IDT (Interrupt Descriptor Table)
- 메모리의 0번지(초기 x86)나 OS가 지정한 특정 주소 공간에 존재하는, 인터럽트 번호와 그에 매칭되는 ISR의 실제 주소를 차곡차곡 적어둔 1차원 배열(장부)이다.
- 예를 들어, x86 아키텍처에서는 총 256개의 벡터(0~255)를 관리한다.

### 2. 하드웨어 $\to$ 소프트웨어 연결 프로세스 시각화 (ASCII)
```text
    [ 1. 인터럽트 발생 ]
    키보드 컨트롤러가 CPU의 INTR 핀을 찌르고, 데이터 버스에 '0x21(33번)'이라는 벡터 번호를 실어 보냄.
         │
    [ 2. CPU의 IVT 인덱싱 ]
    CPU는 메모리의 IVT(예: 0x0000 시작)로 가서 33번째 칸을 열어봄.
         │
         ▼
    ┌─── Interrupt Vector Table (IVT) ───┐
    │ 0번 (Divide by 0) : 0x0100 0000    │
    │ 1번 (Debug)       : 0x0100 0500    │
    │ ...                                │
    │ 33번(Keyboard)    : ★ 0x0400 1200 │ ─┐ (여기다!)
    │ ...                                │  │
    └────────────────────────────────────┘  │
                                            ▼
                               [ 3. ISR로 JUMP (PC 갱신) ]
                               PC 레지스터에 0x0400 1200 을 덮어씌움!
                               -> 키보드 입력 처리 프로그램(ISR) 즉시 실행!
```

### 3. 심층 분석: PIC (Programmable Interrupt Controller)
- 키보드 자체가 "나는 33번이야"라고 말하는 것은 아니다. 수많은 장치는 8259A나 APIC 같은 인터럽트 전담 칩(PIC)에 연결되어 있다. 이 **PIC가 장치들의 우선순위를 정한 뒤, CPU에게 벡터 번호(예: IRQ1 = 벡터 33번)를 번역해서 던져준다**. 즉, 하드웨어 라인(IRQ)을 소프트웨어 식별자(Vector)로 매핑해 주는 중재자다.

- **📢 섹션 요약 비유**: 우편물(인터럽트)에 적힌 우편번호(Vector)만 보고, 수백 개의 동네 우체국(ISR) 중 어느 곳으로 보낼지 1초 만에 분류하는 중앙 집중식 자동 분류기입니다.

## Ⅲ. 인터럽트 분류와 벡터 할당 (x86 기준)

운영체제는 부팅될 때 이 256개의 벡터를 꼼꼼하게 채워 넣는다.

| 벡터 번호 | 분류 | 설명 및 예시 |
|:---:|:---|:---|
| **0 ~ 31** | **하드웨어 예외 (Exceptions)** | CPU 내부에서 연산 중 발생. `0번: 0으로 나누기`, `14번: 페이지 폴트(Page Fault)` |
| **32 ~ 47** | **외부 하드웨어 인터럽트 (IRQs)** | PIC가 관리. `32번: 타이머`, `33번: 키보드`, `44번: 마우스` |
| **128 (0x80)**| **소프트웨어 인터럽트 (Syscall)** | 리눅스에서 사용자 프로그램이 OS 커널에 서비스를 요청할 때 고의로 터뜨리는 인터럽트 (시스템 콜). |
| **기타** | 사용자 정의 / NMI | 프로그래머나 제조사가 자유롭게 쓸 수 있는 여유 공간. |

- **📢 섹션 요약 비유**: 0~31번은 내 몸속에서 나는 병(내과), 32~47번은 밖에서 날아온 돌멩이(외과), 0x80번은 내가 스스로 의사를 부르는 호출 벨(시스템 콜)로 병동을 완벽히 분리해 둔 종합병원입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: IDT (Interrupt Descriptor Table) 후킹 해킹
- **상황**: 안티 치트(Anti-Cheat) 프로그램이나 해커가 시스템의 모든 동작을 훔쳐보려 함.
- **판단**: 메모리에 있는 이 IVT(현대 OS에서는 IDT라 부름)는 시스템의 생살여탈권을 쥔 장부다. 만약 악성 코드가 권한을 뚫고 33번(키보드) 벡터에 적힌 '원래 OS 주소'를 지우고, '해커의 해킹 코드 주소(Keylogger)'로 바꿔치기(Hooking)해 버리면 어떻게 될까? 사용자가 키보드를 칠 때마다 OS가 아닌 해커의 코드가 먼저 실행되어 비밀번호가 다 털린다. 이를 막기 위해 최신 운영체제(Windows 10 이상)는 커널 패치 보호(PatchGuard)를 적용하여 IDT 메모리 영역을 수정 불가능한 강철 금고로 만들어버렸다.

### 2. 기술사적 인사이트: MSI (Message Signaled Interrupts)
- **인사이트**: 과거에는 인터럽트를 쏘기 위해 칩과 칩 사이에 물리적인 구리선(IRQ 핀)을 일일이 납땜해야 했다. 하지만 장치가 수십 개로 늘어나자 핀이 모자랐다. PCIe(PCI Express) 규격부터는 물리적인 핀을 싹 다 뽑아버리고, **MSI (Message Signaled Interrupts)** 방식을 쓴다. 장치가 CPU를 찌르고 싶으면, 그냥 메모리의 특정 주소에 "나 33번 벡터 인터럽트 발생!"이라는 **메시지(패킷)를 Write 해버린다**. 물리적 선이 논리적 데이터 패킷으로 완전히 진화한 아키텍처의 혁명이다.

- **📢 섹션 요약 비유**: 집마다 비상벨 전선을 따로 깔아주던 무식한 낡은 아파트에서, 비상벨 선을 다 뜯어내고 스마트폰 앱(네트워크 패킷)으로 경비실에 알림을 쏘는 스마트 아파트로의 진화입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **폴링 오버헤드 0%**: 수십 개의 장치가 동시에 소리를 지르는 상황에서도, CPU는 단 1클럭 만에 누구의 요청인지 식별하여 $O(1)$의 속도로 처리 루틴으로 진입 가능.
- **하드웨어 독립성 제공**: 장치가 바뀌어도 OS는 드라이버를 깔아 IVT의 주소 한 줄만 업데이트하면 되므로, 무한한 하드웨어 플러그 앤 플레이(PnP)의 소프트웨어적 기반 마련.

### 2. 미래 전망
최신 클라우드 가상 머신(VM) 환경에서는 이 인터럽트 벡터도 가상화된다. 물리적 하드웨어가 쏜 벡터 33번을, 하이퍼바이저가 가로채어 "가상 머신 1번에게 쏜 가짜 벡터 40번"으로 실시간 번역해 주는 복잡한 변환 계층이 생겼다. 인터럽트 벡터는 단순한 하드웨어 신호표를 넘어, 가상화 세상의 수많은 운영체제들을 헷갈리지 않게 통제하는 거대한 신호등 체계로 무한 확장 중이다.

- **📢 섹션 요약 비유**: 처음엔 단순히 내 번호를 알려주는 명찰이었지만, 이제는 내가 어느 세계(가상 머신)의 누구인지 증명하는 완벽한 다중 우주 주민등록번호로 거듭났습니다.

1. 인터럽트 벡터는 호텔 프런트 데스크에 걸려 온 전화기의 **'발신자 방 번호'**예요.
2. 전화벨이 울릴 때 일일이 방마다 찾아가서 "누가 전화했어요?"라고 묻지 않고, 화면에 뜬 '201호'라는 번호(벡터)를 1초 만에 딱 확인하죠.
3. 그리고 201호 손님이 원하는 게 뭔지 적혀있는 매뉴얼(벡터 테이블)을 바로 꺼내서 즉시 직원을 출동시키는 아주아주 빠르고 똑똑한 시스템이랍니다!
