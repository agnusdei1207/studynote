# 524. 스토리지 클래스 메모리 (SCM) 계층화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DRAM의 빠른 접근 속도(바이트 단위 접근)와 NAND Flash의 비휘발성(전원이 꺼져도 데이터 유지)이라는 두 매체의 장점을 결합한 새로운 계층의 메모리 아키텍처다.
> 2. **가치**: 메모리와 스토리지 사이의 거대한 성능 격차(Performance Gap)를 메워주며, 인메모리 데이터베이스의 복구 시간을 획기적으로 줄이고 시스템 전체의 I/O 병목을 해소한다.
> 3. **융합**: 인텔의 Optane(3D XPoint)으로 대표되며, 운영체제 레벨의 DAX(Direct Access) 지원 및 NVMe 프로토콜과 결합하여 'Persistent Memory(영구 메모리)'라는 새로운 소프트웨어 패러다임을 열었다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 스토리지 클래스 메모리(SCM, Storage Class Memory)는 RAM처럼 바이트(Byte) 단위로 접근할 수 있을 만큼 빠르면서도, SSD처럼 전원이 차단되어도 데이터가 지워지지 않는 특성을 가진 중간 계층의 기억장치다. SCM을 기존 계층에 편입시키는 구조적 방법을 SCM 계층화(Tiering)라 한다.
- **💡 비유**: DRAM이 뇌 속에 기억하는 '단기 암기(엄청 빠르지만 자고 일어나면 까먹음)'이고, SSD가 서재에 있는 '백과사전(영원히 남지만 책장까지 가야 해서 느림)'이라면, SCM은 내 책상 위에 항상 펼쳐져 있는 '다이어리'다. 손만 뻗으면 바로 읽을 수 있고(빠름), 퇴근 후 다음 날 와도 그대로 적혀 있다(비휘발성).
- **등장 배경 및 발전 과정**:
  1. **DRAM과 NAND의 극심한 성능 격차**: CPU의 속도는 기가헤르츠(GHz)인데, 메모리(DRAM)는 수십 나노초(ns) 안에 응답하지만 플래시 메모리(SSD)는 수십 마이크로초(µs)가 걸린다. 이 둘 사이에는 1,000배라는 엄청난 속도 절벽이 존재했다.
  2. **DRAM의 용량 한계와 가격**: 인메모리 DB(SAP HANA 등)가 인기를 끌면서 테라바이트급 메모리가 필요해졌으나, DRAM은 너무 비싸고 밀도를 높이기 어려웠다.
  3. **새로운 매질의 등장**: 2015년 인텔과 마이크론이 3D XPoint 기술(상표명 Optane)을 발표하며 상용화에 성공했다. PRAM(상변화 메모리), MRAM(자기 저항 메모리), ReRAM(저항 변화 메모리) 등 다양한 SCM 소재가 연구/상용화되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (SCM의 물리적/논리적 계층)

| 요소명 | 역할 | 내부 동작 / 특징 | 비유 |
|:---|:---|:---|:---|
| **SCM Media (매질)** | 데이터를 비휘발성으로 물리적 저장 | 상변화(열), 자기장, 저항 변화 등을 이용해 0과 1을 영구적으로 기록 | 지워지지 않는 특수 잉크 |
| **Memory Controller** | SCM을 메인 메모리(DRAM)의 연장선으로 인식하게 제어 | 일반 DDR 슬롯(DIMM)에 꽂혀 CPU가 직접 Load/Store 명령을 내리게 함 | 다이어리를 뇌와 연결하는 신경 |
| **NVDIMM (Non-Volatile DIMM)** | SCM을 메모리 슬롯에 꽂기 위한 폼팩터 | DDR4/DDR5 핀 규격과 호환되며 내부 컨트롤러를 탑재 | 다이어리의 규격화된 바인더 |
| **DAX (Direct Access)** | OS 커널의 파일시스템 우회 기술 | 애플리케이션이 페이지 캐시를 거치지 않고 SCM에 물리적으로 직접 접근(mmap) | 비서(OS) 없이 다이어리 직접 쓰기 |

### 메모리 계층 구조에서의 SCM (Storage Tiering)

SCM은 전통적인 메모리 피라미드의 중간에 삽입되어 새로운 계층을 형성한다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │              전통적 계층 구조 vs SCM 도입 후 계층 구조              │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 전통적 메모리 계층 (1,000배의 속도 절벽)                      │
  │      ┌─────┐                                                │
  │      │ CPU │  ~ 0.5 ns                                      │
  │      └──┬──┘                                                │
  │      ┌──┴──┐                                                │
  │      │ DRAM│  ~ 80 ns  (빠르지만 비싸고 용량 작음, 전원 끄면 날아감)│
  │      └──┬──┘                                                │
  │         │   ◀── [ 엄청난 성능 단층 (1,000배 느림) ] ──▶       │
  │      ┌──┴──┐                                                │
  │      │ SSD │  ~ 80,000 ns (싸고 대용량이지만 엄청 느림, 영구적) │
  │      └─────┘                                                │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] SCM 계층화 도입 (격차 완화 및 용량 극대화)                    │
  │      ┌─────┐                                                │
  │      │ CPU │                                                │
  │      └──┬──┘                                                │
  │      ┌──┴──┐ (Hot Data)                                     │
  │      │ DRAM│ ~ 80 ns                                        │
  │      └──┬──┘                                                │
  │      ┌──┴──┐ (Warm Data) ◀── [새로운 계층 등장!]              │
  │      │ SCM │ ~ 300 ns (DRAM보단 느리지만 SSD보다 200배 빠름!)   │
  │      │(PMEM)│          (용량은 TB급 확장 가능, 전원 꺼져도 유지됨) │
  │      └──┬──┘                                                │
  │      ┌──┴──┐ (Cold Data)                                    │
  │      │ SSD │ ~ 80,000 ns                                    │
  │      └─────┘                                                │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 기존 구조에서는 자주 쓰는 데이터(Hot)는 램에, 안 쓰는 데이터(Cold)는 SSD에 두었다. 하지만 램 용량이 부족해 SSD로 넘어가야 할 때(Swap), 속도가 1,000배나 느려지며 시스템이 멈추는 듯한 병목이 발생했다.
**SCM(Storage Class Memory)**을 이 사이에 끼워 넣으면 훌륭한 완충재가 된다. CPU 바로 옆(DDR 슬롯)에 꽂히는 SCM(예: 인텔 옵테인 DC Persistent Memory)은 램보다는 조금(약 3~4배) 느리지만, SSD보다는 무려 200배 이상 빠르다. 게다가 램보다 밀도가 높아 서버 1대에 수 테라바이트를 싸게 꽂을 수 있다. 이제 따뜻한 데이터(Warm Data)는 SCM에 머무르게 하여 시스템 전체의 가성비와 속도를 동시에 잡을 수 있게 되었다.

---

### 심층 활용 아키텍처: 2가지 동작 모드 (Memory Mode vs App Direct Mode)
SCM(특히 인텔 Optane PMEM)은 시스템에서 두 가지 완전히 다른 모드로 계층화될 수 있다.
1. **메모리 모드 (Memory Mode)**: SCM을 **"싸고 거대한 램(RAM)"**처럼 쓰는 방식. 물리적 DRAM은 SCM을 위한 거대한 L4 캐시(Cache)로 강등된다. OS와 애플리케이션 입장에서는 수 TB의 휘발성 RAM이 있는 것으로 보이며(비휘발성 특성은 잃음), 아무런 소스코드 수정 없이 인메모리 용량을 늘릴 수 있다.
2. **앱 다이렉트 모드 (App Direct Mode)**: SCM의 진짜 힘. SCM을 **"초고속 영구 스토리지"**로 쓴다. 리눅스의 DAX(Direct Access) 파일시스템으로 마운트한 뒤, 애플리케이션이 `mmap()`을 호출하면 커널 버퍼를 거치지 않고 CPU가 직접 바이트 단위로 영구 메모리에 Load/Store 명령을 내린다. 전원이 꺼져도 데이터가 유지되는 진정한 영구 메모리(Persistent Memory) 프로그래밍이 시작된다.

## Ⅲ. 융합 비교 및 다각도 분석

### SCM 매질(Material) 별 기술 비교

| 메모리 유형 | 데이터 기록 원리 | 속도 (Latency) | 내구성 (수명) | 비휘발성 |
|:---|:---|:---|:---|:---|
| **DRAM (비교군)** | 커패시터 전하 충/방전 | ~ 10 ns (최상) | 무한대 | 휘발성 |
| **PRAM (상변화)** | 열로 소재의 결정/비정질 상태 변환 | ~ 100 ns (우수) | 높음 (NAND 대비) | **비휘발성** (인텔 3D XPoint) |
| **MRAM (자기저항)**| 자성체의 자화 방향(스핀) 변경 | ~ 20 ns (거의 DRAM급) | 거의 무한대 | **비휘발성** (미세공정 캐시 대체용) |
| **NAND SSD (비교군)**| 플로팅 게이트에 전자 포획 | ~ 50,000 ns (최악) | 제한적 (P/E Cycle) | 비휘발성 |

### 과목 융합 관점
- **데이터베이스 아키텍처 (인메모리 DB 복구)**: Redis나 SAP HANA 같은 인메모리 DB는 평소엔 엄청 빠르지만, 서버 전원이 꺼지면 램에 있던 테라바이트급 데이터가 다 날아간다. 이를 막기 위해 디스크(SSD)에 스냅샷을 찍거나 로그(WAL)를 남기느라 성능이 깎인다. SCM(App Direct 모드)을 쓰면 데이터가 원래 영구적(Persistent)이므로, 정전 후 서버를 켜면 DB를 디스크에서 램으로 로딩할 필요 없이 **즉시 0초 만에 복구(Instant Recovery)**되어 서비스가 시작된다.
- **운영체제 (페이지 캐시의 소멸)**: 리눅스 커널은 느린 디스크를 보완하기 위해 항상 시스템 램의 일부를 페이지 캐시(Page Cache)로 쓴다. SCM+DAX 환경에서는 파일 시스템(예: ext4-dax)이 램을 중간에 끼지 않고 SCM 장비 자체에 CPU 레지스터를 직접 매핑해 버리므로, 수십 년간 이어온 운영체제의 페이지 캐시 아키텍처가 사실상 소멸하는 철학적 변화가 발생한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 클라우드 비용 최적화를 위한 티어링 (Tiering)**: 엄청난 메모리를 요구하는 가상머신(VM) 수백 대를 돌려야 하는 퍼블릭 클라우드 노드.
   - **아키텍처 적용**: 128GB짜리 비싼 DDR4 메모리로 2TB를 채우는 대신, 64GB DDR4(캐시 역할)와 512GB SCM 4개(Memory Mode)를 꽂아 서버를 구성한다. 90% 이상의 데이터 히트율이 보장되는 환경이라면 애플리케이션 성능 저하 없이 노드당 메모리 구축 비용(CapEx)을 절반 가까이 아끼면서 더 많은 VM을 밀어 넣을 수 있다.
2. **시나리오 — 메타데이터 서버의 병목 해소 (Ceph/Hadoop)**: 수천 대의 분산 스토리지 클러스터에서, 파일의 위치 정보를 기록하는 메타데이터 서버(MDS)가 초당 수십만 번의 4KB짜리 작은 로그(Journal)를 SSD에 쓸 때마다 전체 클러스터가 느려짐.
   - **해결책**: 메타데이터 서버의 저널링 드라이브만 PCIe NVMe 타입의 SCM(예: Optane SSD)으로 교체한다. SCM은 NAND 플래시와 달리 가비지 컬렉션(GC)이나 블록 지우기 페널티가 없어, 랜덤 쓰기가 아무리 작고 빈번하게 발생해도 지연 시간이 10µs(마이크로초) 이하로 칼같이 유지된다. 

### 도입 체크리스트
- **비즈니스 리스크 (인텔의 사업 철수)**: SCM의 대명사였던 인텔이 2022년 Optane(3D XPoint) 사업에서 전격 철수를 선언했다. 기술은 완벽했으나 메모리 시장의 규모의 경제를 이기지 못해 단가가 비쌌기 때문이다. 따라서 현재 NVDIMM 형태의 순수 SCM을 신규 시스템에 아키텍처로 박아 넣는 것은 치명적인 벤더 락인(Vendor Lock-in) 및 부품 수급 리스크를 안게 된다.
- **CXL 아키텍처로의 전환**: SCM이 몰락한 빈자리는 **CXL (Compute Express Link)**이 완벽히 대체하고 있다. CXL 인터페이스를 통해 저렴한 일반 DDR 메모리를 SCM처럼 확장하거나, 낸드 플래시 앞에 CXL 컨트롤러를 달아 SCM의 역할을 흉내 내는 방식(CXL Memory Pooling)으로 업계 트렌드가 급선회했음을 설계 시 반드시 고려해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 일반 DRAM + NAND 조합 | SCM (Persistent Memory) 도입 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (DB 복구 시간)** | TB 단위 메모리 로딩 시 수십 분 소요 | 재부팅 직후 포인터 맵핑만으로 즉시 복구 | 시스템 장애 시 RTO(목표 복구 시간) **0초에 수렴** |
| **정량 (Random Write 지연)** | SSD 내부의 복잡한 FTL로 100µs 이상 튐 | 매질 자체의 특성으로 10µs 이하 일정 | 꼬리 지연(Tail Latency)의 완벽한 억제 (QoS 보장) |
| **정성 (프로그래밍 모델)** | 파일 I/O API (open, read, write) 사용 필수 | 메모리 API (malloc, 포인터 연산)로 영구 저장 | SNIA NVM 프로그래밍 모델이라는 새로운 패러다임 탄생 |

### 미래 전망
- **CXL과 이기종 메모리 티어링(Tiering)**: SCM이라는 단일 매질의 꿈은 한풀 꺾였지만, 그 철학은 CXL 생태계로 흡수되었다. 미래의 서버는 CPU 캐시(SRAM) -> 메인 메모리(DDR) -> **CXL 확장 메모리(DDR+NAND 혼합 SCM 대용)** -> 네트워크 스토리지(NVMe-oF)로 이어지는 초정밀 다계층(Tiering) 시스템을 OS가 AI 스케줄러로 자동 이동시키는 구조가 될 것이다.
- **MRAM의 온칩(On-chip) 캐시 대체**: SCM 소자 중 하나인 MRAM(자기저항메모리)은 현재 CPU나 MCU 칩 내부의 임베디드 플래시(eFlash)나 SRAM을 대체하는 용도로 파운드리(TSMC, 삼성) 공정에서 활발히 상용화되고 있다. 전원이 꺼져도 유지되는 초저전력 IoT 엣지 칩셋의 핵심 부품으로 자리 잡고 있다.

### 참고 표준
- **SNIA NVM Programming Model**: 스토리지 네트워킹 산업 협회(SNIA)에서 제정한 영구 메모리 프로그래밍 표준. 개발자가 OS를 거치지 않고 DAX를 통해 영구 메모리에 안전하게 트랜잭션을 쏘아 넣기 위한 `pmem` 라이브러리 규격을 정의했다.
