> 1. x86 아키텍처의 Ring 보호 모드는 하드웨어 수준에서 권한을 4단계로 나누어 시스템의 핵심 자원을 사용자 프로그램으로부터 격리하는 보안 메커니즘입니다.
> 2. Ring 0은 커널, Ring 3는 사용자 애플리케이션에 할당되며, Ring 1과 2는 역사적으로 디바이스 드라이버용으로 설계되었으나 현대 OS에서는 거의 사용되지 않습니다.
> 3. 시스템 콜(System Call)을 통해 Ring 3에서 Ring 0으로 권한이 상승할 때 컨텍스트 스위칭 오버헤드가 발생하며, 이는 운영체제 성능의 주요 변수입니다.

## Ⅰ. x86 Ring 보호 모드의 탄생 배경 및 철학

초기 컴퓨터 시스템은 모든 프로그램이 하드웨어에 직접 접근할 수 있었기 때문에, 응용 프로그램의 작은 버그가 전체 시스템을 마비시키는 치명적인 문제(System Crash)를 야기했습니다. 이를 해결하기 위해 Intel은 80286 프로세서부터 보호 모드(Protected Mode)를 도입하였고, 그 핵심 보안 아키텍처가 바로 계층적 권한 링(Hierarchical Privilege Rings)입니다.

x86 아키텍처는 하드웨어 수준에서 권한 수준(Privilege Level)을 Ring 0부터 Ring 3까지 4단계로 분리합니다. 이 아키텍처의 철학은 **최소 권한의 원칙(Principle of Least Privilege)**에 입각하여, 신뢰할 수 있는 핵심 코드(운영 체제 커널)만을 가장 깊숙한 Ring 0에 배치하고, 신뢰할 수 없는 일반 애플리케이션은 가장 바깥쪽인 Ring 3로 밀어내어 메모리와 I/O 포트에 대한 직접 접근을 원천적으로 차단하는 것입니다.

> 📢 **섹션 요약 비유:** Ring 보호 모드는 옛날 왕궁의 방어 체계입니다. 왕(커널)은 가장 깊숙한 내성(Ring 0)에 살고, 일반 백성(애플리케이션)은 가장 바깥 성벽 밖(Ring 3)에 살게 하여 백성이 실수로 불을 내도 왕궁이 타지 않게 막는 것입니다.

## Ⅱ. 4단계 권한 계층과 보호 메커니즘

x86 프로세서는 현재 실행 중인 코드의 권한 수준을 CS(Code Segment) 레지스터의 하위 2비트인 CPL(Current Privilege Level)에 저장하여 관리합니다.

```ascii
      +---------------------------------------------------+
      |                 Ring 3 (CPL=3)                    |
      |          [ User Applications (Web, Game) ]        |
      |   +-------------------------------------------+   |
      |   |             Ring 2 (CPL=2)                |   |
      |   |       [ OS Services (Not used in OS) ]    |   |
      |   |   +-----------------------------------+   |   |
      |   |   |         Ring 1 (CPL=1)            |   |   |
      |   |   |    [ Device Drivers (Legacy) ]    |   |   |
      |   |   |   +---------------------------+   |   |   |
      |   |   |   |     Ring 0 (CPL=0)        |   |   |   |
      |   |   |   | [ OS Kernel / Core Code ] |   |   |   |
      |   |   |   +---------------------------+   |   |   |
      |   |   +-----------------------------------+   |   |
      |   +-------------------------------------------+   |
      +---------------------------------------------------+
             (System Call) |   ^ (IRET)
                           v   |
                   [ Hardware / CPU ]
```

1. **Ring 0 (Kernel Mode / Supervisor Mode):**
   가장 높은 권한을 가집니다. 메모리 페이징(Paging) 설정, GDT(Global Descriptor Table) 수정, 인터럽트 활성/비활성 등 프로세서의 모든 특권 명령어(Privileged Instructions)를 실행할 수 있습니다. 운영 체제(OS, Operating System)의 핵심 커널이 여기서 동작합니다.
2. **Ring 1 & Ring 2:**
   설계 초기에는 파일 시스템 서비스나 디바이스 드라이버(Device Driver) 등을 위해 존재했습니다. 하지만 다중 플랫폼 이식성을 중시하는 현대의 주류 운영 체제(Windows, Linux, macOS)들은 RISC 아키텍처(권한을 User와 Kernel 2단계로만 나눔)와의 호환성을 위해 이 두 링을 아예 사용하지 않습니다.
3. **Ring 3 (User Mode):**
   가장 낮은 권한입니다. 하드웨어를 직접 제어하는 명령어는 실행할 수 없으며, 시도할 경우 프로세서가 예외(General Protection Fault)를 발생시키고 해당 프로세스를 강제 종료시킵니다.

> 📢 **섹션 요약 비유:** Ring 0가 회사의 '대표이사(모든 결재권 가짐)'라면, Ring 3는 '일반 사원(금고 접근 불가)'입니다. Ring 1과 2는 중간 관리자인데, 요즘 회사는 결재 단계를 줄이기 위해 중간 관리자를 없애고 대표와 사원만 남겨둔 것과 같습니다.

## Ⅲ. Ring 간의 통신: 시스템 콜 (System Call)

Ring 3에서 실행되는 애플리케이션이 파일 읽기, 네트워크 전송, 메모리 할당 등 하드웨어 자원이 필요한 작업을 수행하려면 반드시 Ring 0의 커널에 요청을 보내야 합니다. 이를 시스템 콜(System Call)이라고 합니다.

과거에는 소프트웨어 인터럽트(Software Interrupt, `INT 0x80`)를 사용하여 강제로 예외를 발생시키고 Ring 0의 ISR(Interrupt Service Routine)로 점프했습니다. 하지만 이 방식은 CPU의 보호 메커니즘을 여러 단계 거쳐야 하므로 오버헤드(Overhead)가 컸습니다.
이를 해결하기 위해 최신 프로세서는 전용 고속 명령어인 `SYSENTER`/`SYSEXIT` (Intel) 또는 `SYSCALL`/`SYSRET` (AMD)를 제공합니다. 이 명령어들은 레지스터에 시스템 콜 번호를 넣고 호출하면, 권한 검사 과정을 대폭 생략하고 즉각적으로 Ring 0의 미리 정의된 안전한 진입점(Entry Point)으로 권한을 상승시켜 실행 속도를 극대화합니다.

> 📢 **섹션 요약 비유:** 일반 사원(Ring 3)이 회사 금고(하드웨어)에서 돈을 꺼내려면 맘대로 문을 부수면 안 되고, 반드시 정해진 양식(시스템 콜)을 작성해 은행 창구 직원(Ring 0 커널)에게 제출해서 안전하게 돈을 받아와야 합니다.

## Ⅳ. 메모리 보호: 페이징(Paging)과 권한 검사

Ring 아키텍처는 메모리 관리 유닛(MMU, Memory Management Unit)과 결합하여 물리적 메모리를 철저히 보호합니다.

페이징(Paging) 시스템에서 페이지 테이블 엔트리(PTE, Page Table Entry)에는 U/S (User/Supervisor) 비트가 존재합니다.
* **U/S 비트 = 0 (Supervisor):** Ring 0, 1, 2 권한(CPL <= 2)에서만 접근 가능한 메모리 페이지입니다. 주로 커널 데이터가 저장됩니다.
* **U/S 비트 = 1 (User):** Ring 3 권한(CPL = 3)에서도 접근 가능한 페이지입니다.

만약 Ring 3에서 동작하는 코드가 U/S 비트가 0인 커널 메모리 영역을 읽거나 쓰려고 시도하면, MMU는 즉각적으로 페이지 폴트(Page Fault)를 발생시켜 불법적인 접근을 하드웨어 레벨에서 차단합니다. 이를 통해 해커가 일반 애플리케이션을 해킹하더라도 OS 커널 영역을 변조하는 것을 방지합니다.

> 📢 **섹션 요약 비유:** 페이징의 U/S 비트는 기밀문서에 찍힌 '1급 기밀(Supervisor)'과 '일반 공개(User)' 도장과 같습니다. 일반 백성(Ring 3)이 1급 기밀 문서를 열어보려 하면 문서 보관소의 경비병(MMU)이 즉시 체포해버립니다.

## Ⅴ. 한계와 진화: 가상화 시대의 Ring -1 (VMX Root)

2000년대 들어 클라우드 컴퓨팅과 하드웨어 가상화(Hardware Virtualization) 시대가 도래하면서 기존의 Ring 0 ~ Ring 3 구조에 심각한 모순이 발생했습니다.
가상 머신(VM, Virtual Machine) 내의 Guest OS도 자신이 Ring 0에서 동작한다고 착각해야 하는데, 실제 물리적 제어권(진짜 Ring 0)은 호스트(Host)의 하이퍼바이저(Hypervisor)가 쥐고 있어야 했기 때문입니다. 소프트웨어적으로 이 권한을 가로채고 에뮬레이션하는 데에는 엄청난 성능 저하가 따랐습니다.

이 문제를 해결하기 위해 Intel과 AMD는 하드웨어 아키텍처를 확장하여, Ring 0보다 더 아래에 있는 새로운 특권 레벨인 이른바 **Ring -1 (VMX Root Mode / 하이퍼바이저 모드)**을 창설하게 됩니다. 이를 통해 가상화의 성능 패널티를 혁신적으로 제거했습니다.

> 📢 **섹션 요약 비유:** 왕(Guest OS)이 자기가 최고인 줄 알고 Ring 0에서 떵떵거리며 살고 있었는데, 알고 보니 그 왕국은 거대한 제국(클라우드)의 일부였고, 진짜 권력자인 황제(하이퍼바이저)가 구름 위(Ring -1)에서 모든 왕들을 내려다보며 통제하게 된 것입니다.
