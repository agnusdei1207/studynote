> **Insight**
> - 전통적인 블록(Block) 기반 스토리지 인터페이스의 한계를 극복하기 위해, 저장 공간을 '존(Zone)'이라는 큰 단위로 분할하여 관리하는 차세대 데이터 배치 아키텍처
> - 호스트 시스템이 순차 쓰기(Sequential Write) 원칙을 준수하도록 강제함으로써 스토리지 매체 내부의 가비지 컬렉션(GC)을 원천적으로 제거
> - QLC/PLC NAND, SMR HDD 등 고집적 스토리지 미디어의 수명(Endurance)과 성능 예측성(Predictable Performance)을 극대화하는 하이퍼스케일 데이터센터의 필수 기술

## Ⅰ. 존 스토리지 (Zoned Storage)의 개요

존 스토리지는 하드디스크(HDD)의 SMR(Shingled Magnetic Recording) 기술과 솔리드 스테이트 드라이브(SSD)의 NVMe ZNS(Zoned Namespace) 기술을 포괄하는 아키텍처 패러다임입니다. 기존 스토리지는 호스트가 512바이트 또는 4KB 크기의 논리적 블록(LBA)을 무작위(Random)로 덮어쓰는 것을 허용했습니다. 그러나 이 방식은 플래시 메모리나 기와식 자기 기록(SMR) 방식처럼 "물리적으로 덮어쓰기가 불가능한" 고밀도 매체 내부에서 엄청난 백그라운드 재배치 작업(Garbage Collection)을 유발합니다. 
존 스토리지는 매체를 수백 MB 단위의 여러 **'존(Zone)'**으로 쪼개고, **"각 존 내부에서는 반드시 순차적으로만 데이터를 써야 하며, 덮어쓰려면 존 전체를 초기화(Reset/Erase)해야 한다"**는 엄격한 규칙을 호스트 OS 및 애플리케이션에 강제합니다. 이를 통해 스토리지 내부의 비효율성을 완전히 제거합니다.

📢 **섹션 요약 비유**: 아무 곳에나 낙서를 할 수 있었던 연습장(기존 스토리지)을, "무조건 맨 윗줄부터 차례대로 적어야 하고, 틀리면 지우개로 한 줄만 지울 수 없고 페이지 전체를 찢어버려야 하는" 엄격한 규칙의 원고지(존 스토리지)로 바꾼 것과 같습니다.

## Ⅱ. 존 스토리지의 등장 배경과 아키텍처

존 스토리지는 오픈 채널 SSD(Open-Channel SSD)의 철학(호스트 제어 강화)을 계승하되, 물리적 칩의 복잡성을 숨겨 표준화에 성공한 절충형 아키텍처입니다.

```text
[Traditional SSD (Block Device)]        [Zoned Storage (ZNS SSD/SMR HDD)]
+------------------------------+        +------------------------------+
| Host App (Random Writes)     |        | Host App (Sequential Writes) |
+------------------------------+        +------------------------------+
              |                                        |
              v                                        v
+------------------------------+        +------------------------------+
| FTL (Flash Translation Layer)|        | Thin FTL (No Heavy GC)       |
| - Complex LBA->PBA Mapping   |        | - Simple Zone Mapping        |
| - Garbage Collection (GC)    |        | - Zone Reset Management      |
| - Over-Provisioning (OP)     |        +------------------------------+
+------------------------------+                       |
              |                                        v
              v                         [ Zone 0 ] [ Zone 1 ] [ Zone 2 ]
[ Physical NAND Blocks ]                [ Seq -> ] [ Seq -> ] [ Empty  ]
```

1. **SMR HDD (Host-Managed SMR):** 트랙을 겹쳐 써서 용량을 늘린 하드디스크입니다. 겹쳐진 트랙 특성상 중간 데이터를 수정하면 옆 트랙까지 망가지므로, 존 단위로 순차 기록을 해야만 데이터 무결성이 보장됩니다.
2. **NVMe ZNS (Zoned Namespace):** SSD 내부 FTL의 과도한 매핑 테이블 크기와 잦은 GC로 인한 성능 저하를 막기 위해, 낸드 플래시의 지우기 블록(Erase Block) 크기에 맞춰 존을 정의합니다. 

📢 **섹션 요약 비유**: 주차장에서 차들을 마구잡이로 빼고 넣느라 혼잡했던 상황(GC 발생)을 해결하기 위해, 아예 "주차 라인을 구역(Zone)별로 나누고, 들어온 순서대로 꽉 채워서 일렬 주차한 뒤, 차를 뺄 때는 구역 전체를 한 번에 비워야 한다"는 규칙을 만든 것입니다.

## Ⅲ. 핵심 이점: 수명 연장, 비용 절감, 예측 가능성

존 스토리지가 호스트 시스템에 불편한 규칙(순차 쓰기 강제)을 요구함에도 불구하고 클라우드 기업들이 도입하는 이유는 다음과 같은 막대한 이점 때문입니다.

1. **가비지 컬렉션(GC) 제거 및 성능 예측성:** 디바이스 내부에서 데이터를 옮겨 적는 GC가 발생하지 않으므로, 갑작스러운 응답 지연(Tail Latency Spikes) 현상이 완전히 사라집니다.
2. **쓰기 증폭 계수(WAF)를 1.0에 근접:** 호스트가 쓴 데이터의 양만큼만 실제 플래시 메모리에 기록되므로, SSD의 수명(Endurance)이 획기적으로 늘어납니다. 수명이 짧은 QLC(Quad Level Cell) 낸드의 약점을 완벽히 보완합니다.
3. **오버 프로비저닝(OP) 및 DRAM 절감:** 기존 SSD는 GC를 위해 전체 용량의 7~28%를 예비 공간(OP)으로 비워두고, 거대한 맵핑 테이블을 위해 고가의 DRAM을 탑재해야 했습니다. 존 스토리지는 OP를 거의 없애 사용 가능한 스토리지 용량을 극대화하고, 내부 DRAM 요구량을 획기적으로 줄여 비용을 절감합니다.

📢 **섹션 요약 비유**: 이사업체(FTL)가 물건을 임시 창고에 넣었다가 다시 옮기는 불필요한 작업(쓰기 증폭)을 없애고, 처음부터 목적지(Zone)에 차곡차곡 쌓음으로써 창고 공간(OP)도 아끼고 일하는 시간(성능)도 정확히 예측할 수 있게 된 것입니다.

## Ⅳ. 소프트웨어 생태계 및 적용 과제

존 스토리지 하드웨어를 사용하려면 호스트 운영체제와 애플리케이션이 존의 규칙(순차 쓰기, 존 리셋)을 이해하고 다룰 수 있도록 소프트웨어 스택이 전면 수정되어야 합니다.

* **Linux 커널 지원:** Linux 커널은 블록 레이어에 ZBD(Zoned Block Device) 서브시스템을 도입하여 SMR HDD와 ZNS SSD를 통합 지원합니다.
* **파일 시스템:** 전통적인 EXT4나 XFS는 랜덤 쓰기 기반이므로 ZBD와 호환되지 않습니다. 순차 쓰기 기반의 로그 구조 파일 시스템(Log-structured File System)인 **F2FS(Flash-Friendly File System)**나 **Btrfs**가 존 스토리지 환경에 맞게 개선되어 사용됩니다.
* **애플리케이션 직결:** 데이터베이스(RocksDB 등)나 객체 스토리지(Ceph) 시스템이 파일 시스템을 우회하여(raw device 접근) ZBD 인터페이스와 직접 통신하여 성능을 극대화하는 아키텍처가 주류를 이루고 있습니다.

📢 **섹션 요약 비유**: 새로운 특수 원고지(존 스토리지)를 쓰려면 평범한 연필(일반 파일 시스템)로는 글을 쓸 수 없고, 한 번에 쭉 이어 쓰는 특수 만년필(F2FS, RocksDB)을 새로 장만해야 하는 소프트웨어 생태계의 변화가 필요합니다.

## Ⅴ. 미래 전망 및 활용 분야

데이터 생성량이 폭발적으로 증가하는 AI 시대에, 콜드 데이터(자주 안 쓰이는 데이터) 보관용 저비용/고용량 스토리지 수요가 급증하고 있습니다. 존 스토리지는 TB당 단가를 낮추면서도 성능의 안정성을 보장할 수 있는 유일한 아키텍처적 해법으로 평가받습니다. 

특히 하이퍼스케일러(AWS, Meta, Alibaba 등)들은 독자적인 스토리지 인프라에 NVMe ZNS와 Host-Managed SMR 기술을 적극 채택하고 있으며, CXL(Compute Express Link) 메모리 기술과 결합하여 데이터센터의 메모리-스토리지 계층을 재구성하는 핵심 기반 기술로 자리 잡고 있습니다.

📢 **섹션 요약 비유**: 과거에는 책을 무작위로 꽂는 작은 동네 서점(일반 스토리지) 방식이 통했지만, 전 세계의 책을 다 모아두는 거대한 국립 도서관(하이퍼스케일 데이터센터)에서는 구역별로 분류를 엄격하게 지키는 존 스토리지 방식만이 건물을 효율적으로 관리하는 유일한 정답이 된 것입니다.
