# Huge Page (Large Page)

> 1. **본질**: 거대 페이지 (Huge Page)는 OS의 가상 메모리 관리 단위인 페이지(Page) 크기를 기본 4KB에서 2MB, 1GB 등으로 확장하여, TLB(Translation Lookaside Buffer) 적중률을 높이고 페이지 테이블 탐색(Page Table Walk) 오버헤드를 줄이는 기술이다.
> 2. **가치**: 수십 GB 이상의 거대 데이터를 다루는 데이터베이스(DB), 인메모리 컴퓨팅, 가상화 환경에서 메모리 주소 변환에 드는 CPU 사이클을 10~20% 이상 절감하여 애플리케이션의 실질 성능을 가속한다.
> 3. **융합**: TLB 아키텍처, 가상 메모리 시스템, THP (Transparent Huge Pages), 그리고 대용량 워크로드 최적화 기법과 융합되어 현대 서버 인프라의 필수 튜닝 요소로 자리 잡고 있다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 거대 페이지 (Huge Page)는 메모리를 관리하는 "포장 단위"를 키우는 기술이다. 보통은 4KB라는 작은 상자에 메모리를 담아 관리하지만, 거대 페이지는 2MB나 1GB짜리 대형 컨테이너에 데이터를 담아 한꺼번에 관리하는 방식이다.

- **💡 비유**: 거대 페이지는 "대형 마트의 묶음 판매"와 같다. 사과를 1개씩 낱개로 포장(4KB)하면 손님이 많을 때 계산대(TLB)가 금방 꽉 차고 번거롭지만, 사과를 100개들이 대형 박스(2MB)로 포장하면 계산 한 번으로 훨씬 많은 사과를 가져갈 수 있는 것과 같다.

- **등장 배경**: 
  1. **메모리 용량의 폭증**: 시스템 메모리가 테라바이트(TB) 단위로 커지면서, 4KB 단위로 관리하기에는 페이지 테이블 장부가 너무 두꺼워짐.
  2. **TLB Reach의 한계**: CPU 내부의 고속 주소 변환 캐시(TLB)는 크기가 제한적인데, 4KB 페이지로는 넓은 메모리 범위를 커버하기 역부족이 됨 (TLB Thrashing 발생).
  3. **가상화 오버헤드**: 하이퍼바이저와 게스트 OS가 중첩된 주소 변환(Nested Paging)을 수행할 때 발생하는 지연 시간을 줄여야 할 필요성 대두.

- **📢 섹션 요약 비유**: 장부의 페이지 수를 줄이기 위해 한 페이지에 더 많은 내용을 꽉 채워 넣는 '메모리 관리의 규모화' 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 관리 메커니즘

거대 페이지는 기존의 다단계 페이지 테이블 구조를 재활용하거나 단축하여 구현한다.

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **TLB (Translation Lookaside Buffer)** | 주소 변환 고속 캐시 | 거대 페이지 주소를 한 엔트리에 저장하여 Reach 확장 | Multi-level TLB | VIP 전용 계산대 |
| **Page Table Entry (PTE)** | 주소 매핑 정보 보관 | 하위 단계 테이블 대신 직접 물리 주소를 가리킴 | PDE (Page Directory Entry) | 단축 경로 이정표 |
| **Huge Page Pool** | 전용 메모리 영역 | 부팅 시 또는 런타임에 거대 페이지용 연속 메모리 확보 | Reserved Memory | 대형차 전용 주차장 |
| **THP (Transparent Huge Pages)** | 자동 거대 페이지 관리 | OS가 런타임에 작은 페이지들을 합쳐 거대 페이지로 변환 | Background Defragmentation | 자동 포장 기계 |
| **Memory Fragmentation** | 파편화 관리 | 연속된 물리 메모리 공간 확보를 위해 압축(Compaction) 수행 | Buddy System | 빈자리 메우기 |

+++

### 거대 페이지의 주소 변환 구조 (4KB vs 2MB)

페이지 크기가 커짐에 따라 페이지 테이블의 단계가 어떻게 생략되는지 시각화한다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │            거대 페이지 주소 변환 메커니즘 (Huge Page Walk)           │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │   [ Standard 4KB Page ]           [ Huge 2MB Page ]           │
 │   ┌───────────────┐               ┌───────────────┐           │
 │   │  PGD (Global) │               │  PGD (Global) │           │
 │   └───────┬───────┘               └───────┬───────┘           │
 │           ▼                               ▼                   │
 │   ┌───────────────┐               ┌───────────────┐           │
 │   │  PUD (Upper)  │               │  PUD (Upper)  │           │
 │   └───────┬───────┘               └───────┬───────┘           │
 │           ▼                               ▼                   │
 │   ┌───────────────┐               ┌───────────────┐           │
 │   │  PMD (Middle) │               │  PMD (Middle) │──▶ [Physical│
 │   └───────┬───────┘               └───────────────┘     2MB Blk]│
 │           ▼                                                   │
 │   ┌───────────────┐      (PTE 단계를 생략하고 PMD에서 직접 매핑!)    │
 │   │  PTE (Entry)  │──▶ [Physical 4KB Page]                    │
 │   └───────────────┘                                           │
 │                                                               │
 │   ※ 결과: TLB 엔트리 하나가 커버하는 메모리 영역이 512배 확장됨     │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 가상 주소에서 물리 주소를 찾아갈 때, 4KB 페이지는 총 4단계(x86_64 기준)의 장부를 거쳐야 한다. 하지만 2MB 거대 페이지를 쓰면 마지막 단계인 PTE(Page Table Entry)를 거치지 않고 PMD(Page Directory) 수준에서 바로 물리 주소의 '대형 블록'을 가리킬 수 있다. 이렇게 되면 CPU가 주소를 찾기 위해 메모리를 뒤지는 횟수(Memory Reference)가 줄어들어 속도가 빨라진다. 더 중요한 것은 TLB이다. TLB 엔트리가 100개라면, 4KB 페이지는 400KB만 커버하지만 2MB 페이지는 200MB라는 어마어마한 범위를 커버한다. 이를 통해 'TLB 미스'를 획기적으로 줄여 CPU가 주소 변환을 기다리느라 멈춰있는 시간을 최소화한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 일반 페이지(4KB) vs 거대 페이지(2MB/1GB)

| 비교 항목 | 일반 페이지 (4KB) | 거대 페이지 (2MB) |
|:---|:---|:---|
| **TLB 적중률** | 낮음 (잦은 교체 발생) | 매우 높음 (넓은 커버리지) |
| **메모리 낭비** | 적음 (정교한 할당) | 높음 (Internal Fragmentation) |
| **페이지 테이블 크기** | 큼 (장부가 두꺼움) | 작음 (장부가 얇음) |
| **OS 오버헤드** | 낮음 (표준 처리) | 높음 (연속 공간 확보 부하) |
| **주 사용처** | 일반 데스크톱, 소형 앱 | DB, 가상화, 빅데이터 분석 |

- **📢 섹션 요약 비유**: 포장지가 얇고 꼼꼼한 작은 택배(4KB)와, 포장지는 아깝지만 한 번에 많이 나르는 컨테이너(2MB)의 효율성 차이입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — Oracle/MySQL DB 성능 튜닝**: 데이터베이스 버퍼 풀(Buffer Pool)이 수십 GB인 경우 4KB 페이지는 TLB 병목을 유발한다. 기술사는 OS 레벨에서 **HugePages(Linux의 `vm.nr_hugepages`)**를 미리 할당하고 DB 설정에서 이를 사용하도록 하여 쿼리 응답 속도를 15% 이상 개선한다.
2. **시나리오 — Java 애플리케이션의 지연 시간(Jitter) 문제**: Transparent Huge Pages(THP)가 런타임에 메모리를 정리(Defragmentation)하느라 시스템이 순간적으로 멈추는 현상이 발생한다. 기술사는 실시간성이 중요한 서비스에서는 **THP를 끄고(Disable)** 수동으로 거대 페이지를 관리하는 의사결정을 내린다.

### 도입 체크리스트
- **Memory Footprint**: 애플리케이션이 사용하는 메모리 총량이 거대 페이지 단위를 채울 만큼 충분히 큰가?
- **Fragmentation Level**: 물리 메모리가 이미 잘게 쪼개져 있어 거대 페이지를 위한 연속 공간 확보가 가능한가?
- **Swap Strategy**: 거대 페이지는 보통 스왑(Swap)되지 않으므로 물리 메모리 부족 시 대응 방안이 있는가?

- **📢 섹션 요약 비유**: 컨테이너 트럭(Huge Page)을 들여오기 전에, 우리 집 마당(Memory)이 그 트럭이 들어올 만큼 넓은지 먼저 확인하는 과정입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (4KB Page) | 도입 후 (Huge Page) | 개선 효과 |
|:---|:---|:---|:---|
| **주소 변환 시간** | 전체 실행 시간의 15% | 3% 미만 | **CPU 오버헤드 80% 절감** |
| **TLB 미스율** | 5% (기준) | 0.1% 이하 | **캐시 효율성 50배 향상** |
| **시스템 처리량** | 100 (기준) | 115 ~ 125 | **전체 성능 15~25% 향상** |

### 미래 전망
향후 거대 페이지 기술은 **'가변 페이지 크기(Variable Page Size)'** 지원으로 발전할 것이다. 워크로드의 특성에 따라 4KB부터 1GB까지 하드웨어가 동적으로 가장 적합한 크기를 선택하는 지능형 관리가 핵심이 될 전망이다. 또한, CXL을 통한 원격 메모리 접근 시에도 지연 시간을 줄이기 위해 거대 페이지 단위의 데이터 전송이 표준이 될 것이며, 이는 초거대 AI 모델 서빙의 핵심 최적화 기술이 될 것이다.

- **📢 섹션 요약 비유**: 이제는 고정된 크기의 상자를 넘어, 물건의 크기에 맞춰 상자 크기가 자동으로 변하는 '카멜레온 포장 시스템'으로 진화하고 있습니다.

## 📌 관련 개념 맵
- **TLB (Translation Lookaside Buffer)**: 거대 페이지의 가장 큰 수혜를 입는 하드웨어 캐시.
- **IOMMU (Input-Output MMU)**: 장치 측면에서 거대 페이지 주소 변환을 지원하는 장치.
- **가상화 하드웨어 지원 (EPT/NPT)**: 이중 주소 변환 오버헤드를 줄이기 위해 거대 페이지를 필수적으로 사용하는 기술.
- **메모리 파편화 (Memory Fragmentation)**: 거대 페이지 도입 시 반드시 해결해야 할 부작용.
