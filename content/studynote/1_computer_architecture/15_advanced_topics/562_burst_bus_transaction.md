> **핵심 인사이트**
> 1. CPU가 메모리에서 데이터를 가져올 때, 가장 시간을 많이 잡아먹는 과정은 **데이터를 가져오는 것 자체가 아니라 "몇 번 주소를 달라"고 요청(주소 버스 사용)하고 응답을 기다리는 시간**이다.
> 2. **버스트 트랜잭션(Burst Transaction)**은 배열이나 캐시 라인처럼 연속된 공간에 있는 데이터를 가져올 때, **시작 주소 딱 한 번만 보내면 메모리가 알아서 그다음 주소들의 데이터를 클럭마다 연속으로 쏟아내게 만드는 기술**이다.
> 3. 이를 통해 주소 버스의 병목(오버헤드)을 제거하고 데이터 전송 대역폭(Bandwidth)을 이론적 최대치까지 끌어올릴 수 있어, 현대의 모든 고속 버스(DDR, PCIe, AXI)의 기본 통신 방식으로 사용된다.

<br>

## Ⅰ. 단일 트랜잭션의 비효율성 (Overhead)

과거의 단순한 버스(Single-beat)는 4바이트 데이터를 읽을 때마다 매번 주소를 보내야 했습니다.
예를 들어, 연속된 주소의 데이터 A, B, C, D를 읽고 싶다고 가정해 봅시다.

* **단일 트랜잭션 방식**:
  1. CPU: "100번 주소 줘!" $\rightarrow$ 메모리 준비 대기 $\rightarrow$ 데이터 A 도착
  2. CPU: "104번 주소 줘!" $\rightarrow$ 메모리 준비 대기 $\rightarrow$ 데이터 B 도착
  3. CPU: "108번 주소 줘!" $\rightarrow$ 메모리 준비 대기 $\rightarrow$ 데이터 C 도착
  4. CPU: "112번 주소 줘!" $\rightarrow$ 메모리 준비 대기 $\rightarrow$ 데이터 D 도착

데이터 4개를 얻기 위해 주소를 4번 묻고 4번 대기해야 합니다. 배보다 배꼽이 큰 이 오버헤드 때문에 버스는 항상 텅텅 비어있게 됩니다.

> 📢 **섹션 요약 비유**: 짜장면 4그릇을 시킬 때, 중국집에 전화를 4번 걸어서 "1그릇 갖다주세요"를 네 번 반복하는 멍청한 주문 방식입니다. 매번 전화를 걸고 끊는 시간(오버헤드)이 너무 낭비됩니다.

<br>

## Ⅱ. 버스트 트랜잭션의 원리와 마법

**버스트(Burst)**란 '폭발하다, 쏟아져 나오다'라는 뜻입니다.
CPU는 데이터가 100번부터 연속적으로 들어있다는 사실(공간적 지역성)을 알면, 다음과 같이 버스트 주문을 넣습니다.

* **버스트 트랜잭션 방식**:
  1. CPU: "100번 주소부터 시작해서 **총 4개를 버스트(Burst)로 연속해서 보내!**" (주소 전송 1번)
  2. 메모리 준비 대기 (초기 대기시간 1번)
  3. 메모리: 데이터 A 전송!
  4. 메모리: 데이터 B 전송! (주소 계산 하드웨어가 스스로 주소를 +4 증가시킴)
  5. 메모리: 데이터 C 전송!
  6. 메모리: 데이터 D 전송!

주소를 한 번만 보내면 1클럭마다 데이터가 쉴 새 없이 쏟아져 나옵니다. 

### 타이밍 다이어그램 (ASCII)

```text
 [ 단일 트랜잭션 ]
 주소 버스 : [A1] -대기- [A2] -대기- [A3] -대기- [A4]
 데이터 버스: ---- [D1] ---- [D2] ---- [D3] ---- [D4] (듬성듬성)

 [ 버스트 트랜잭션 (Burst Length = 4) ]
 주소 버스 : [A1(시작 주소) + 버스트 길이(4)] 
 데이터 버스: ---- [D1][D2][D3][D4] (빈틈없이 쏟아짐!)
```

> 📢 **섹션 요약 비유**: 전화를 딱 한 번만 걸어서 "짜장면 4그릇 한 번에 배달해 주세요"라고 말하는 것입니다. 전화 거는 시간(주소 전송)은 1번으로 줄고, 배달원 한 명이 4그릇을 한 번에 들고 옵니다.

## Ⅲ. 현대 아키텍처의 버스트 한계와 적용

현대 컴퓨터 구조에서 버스트 트랜잭션은 선택이 아닌 필수입니다.
* **캐시 라인 채우기 (Cache Line Fill)**: CPU가 L1 캐시 미스가 나서 메인 메모리에서 데이터를 가져올 때, 무조건 64바이트 덩어리 전체를 가져옵니다. 64비트 버스 기준으로 딱 **8번의 버스트(Burst Length = 8)**를 통해 64바이트를 한 번에 꽉 채웁니다.
* **AXI 버스 (ARM)**: 스마트폰 AP 내부의 통신 규격인 AXI 버스 역시, 주소와 제어 신호를 한 번만 날리면 최대 256번까지 연속으로 데이터를 쏠 수 있는 거대한 버스트 모드를 지원합니다.

하지만 주의할 점이 있습니다. 너무 긴 버스트(예: 1000번 연속 쏘기)를 허용하면, 다른 급한 장치가 버스를 쓰지 못하고 독점(Starvation)당할 수 있으므로 아키텍처마다 최대 버스트 길이를 엄격히 제한(보통 16~256)합니다.
