# 544. 안전한 컨텍스트 스위칭 (Secure Context Switching)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPU가 하나의 프로세스(또는 스레드/가상머신)에서 다른 프로세스로 제어권을 넘길 때 발생하는 레지스터와 메모리 상태의 교체 과정(Context Switch)에서, **이전 프로세스의 잔여 데이터(잔상)가 다음 프로세스로 유출되지 않도록 하드웨어적/소프트웨어적으로 완벽하게 소거(Scrubbing)하고 격리하는 보안 메커니즘**이다.
> 2. **가치**: 클라우드(퍼블릭 가상화)와 모바일 환경(TrustZone)에서 멀티 테넌트(Multi-tenant) 구조를 유지하기 위한 가장 핵심적인 방어벽이다. 이 과정이 불완전하면 A 고객의 암호키가 B 고객의 메모리 공간으로 유출되는 치명적인 데이터 누수가 발생한다.
> 3. **융합**: 운영체제 스케줄러의 소프트웨어적 통제뿐만 아니라, ARM TrustZone이나 인텔 SGX 같은 하드웨어 신뢰 실행 환경(TEE)과 결합하여 '일반 구역(Normal World)'과 '보안 구역(Secure World)'을 넘나드는 가장 엄격한 경계선(Boundary) 역할을 한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 안전한 컨텍스트 스위칭은 CPU가 문맥(Context)을 교환할 때, 단순히 A의 상태를 백업하고 B의 상태를 로드하는 것을 넘어, A가 남긴 민감한 레지스터 값이나 TLB, 분기 예측기(Branch Predictor)의 캐시 흔적까지 모두 파괴/초기화하여 '정보 누출 채널'을 원천 봉쇄하는 절차다.
- **💡 비유**: 호텔(서버)에서 투숙객 A가 체크아웃하고 투숙객 B가 들어올 때의 '청소(Housekeeping)' 과정이다. 일반적인 청소(일반 컨텍스트 스위치)는 침대 시트만 갈아끼우면 되지만, **안전한 청소(Secure Context Switch)**는 A가 쓰던 금고 비밀번호 흔적, 휴지통, 심지어 공기 중에 떠도는 체취(캐시 메모리 잔상)까지 완벽하게 소독하고 리셋하여 B가 A의 어떤 정보도 알아낼 수 없게 만드는 것이다.
- **등장 배경 및 발전 과정**:
  1. **초기 OS의 안일함**: 옛날 OS는 스위칭 속도를 높이기 위해 범용 레지스터만 저장/복구하고, FPU(부동소수점)나 SIMD 레지스터는 B가 실제로 쓸 때만 늦게 교체하는 '지연 복원(Lazy Restore)'을 썼다.
  2. **잔여 데이터 유출 (Data Leakage)**: 악의적인 프로세스 B가 CPU를 할당받자마자 이전에 A가 남겨둔 SIMD 레지스터(암호화 키 계산에 쓰였을 수 있음)를 몰래 읽어가는(Snoop) 해킹이 발생했다.
  3. **사이드 채널 및 TEE의 등장**: 스펙터(Spectre)/멜트다운 사태 이후, 마이크로아키텍처 수준의 캐시나 분기 예측기 상태까지 초기화해야 한다는 것이 밝혀지면서, 문맥 교환은 엄청난 보안 비용(Overhead)을 치르는 과정으로 진화했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (안전한 스위칭을 위한 초기화 대상)

단순한 상태 백업(`PUSH`/`POP`)을 넘어, 해커가 훔쳐볼 수 있는 모든 하드웨어 상태를 소거(Scrub)해야 한다.

| 요소명 | 유출 위험성 (Vulnerability) | 보안 컨텍스트 스위칭의 조치 | 비유 |
|:---|:---|:---|:---|
| **범용/SIMD 레지스터** | 이전 프로세스의 평문 데이터, 키(Key) 값 잔존 | OS가 레지스터를 백업한 직후, **명시적으로 0(Zero)으로 덮어씀 (Zeroing)** | 서랍 비우기 및 걸레질 |
| **TLB / Page Table** | 이전 프로세스의 메모리 주소 매핑 정보 잔존 | ASID(주소 공간 ID) 갱신 또는 **TLB 강제 플러시(Flush)** | 문패 이름표 떼어내기 |
| **Branch Predictor (분기 예측기)** | 이전 프로세스의 실행 흐름(Branch Target) 잔상 | **IBPB, STIBP 등 하드웨어 명령어로 예측기 내부 테이블 초기화** | 과거 투숙객의 동선 지우기 |
| **L1/L2 Cache** | 캐시 적중/미스 타이밍으로 이전 데이터 추론 (사이드 채널) | 캐시 라인 플러시 (단, 성능 저하가 극심하여 매우 민감한 TEE 환경에서만 수행) | 방 안의 공기 완벽 환기 |

### 일반 컨텍스트 스위칭 vs TEE(TrustZone) 안전한 컨텍스트 스위칭 비교

스마트폰 내부에서 일반 앱과 보안 앱(지문 인식 등) 사이의 전환(World Switch)을 예로 든다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │        일반 문맥 교환(OS) vs 안전한 문맥 교환(TrustZone SMC) 비교      │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 일반 OS의 문맥 교환 (A 프로세스 -> B 프로세스)                 │
  │                                                             │
  │   [Process A] ──(타이머 인터럽트)──▶ [OS Kernel 스케줄러]          │
  │                                     │                       │
  │                                 1. A 레지스터 백업 (PCB)        │
  │   (잔여 레지스터 값 그대로 방치)     2. B 레지스터 로드 (PCB)        │
  │                                     │                       │
  │   [Process B] ◀───────(실행)────────┘                       │
  │     * 취약점: B가 로드되지 않은 잉여 레지스터를 뒤져 A의 데이터를 훔쳐봄! │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] 하드웨어 TEE (ARM TrustZone)의 안전한 문맥 교환 (World Switch)│
  │                                                             │
  │  [Normal World (일반 안드로이드)]      [Secure World (지문 결제)]  │
  │                                                             │
  │  (일반 앱) "지문 결제 요청!"                                     │
  │     │                                                       │
  │     ▼ (SMC 명령어 발생)                                         │
  │  [ EL3 (Secure Monitor) - 절대 권한 하드웨어 통제 구역 ]            │
  │     │ 1. Normal World의 레지스터 상태를 안전한 메모리에 백업.       │
  │     │ 2. 일반 구역에서 썼던 레지스터를 **강제로 '0'으로 지워버림 (Zeroing)**│
  │     │ 3. 분기 예측기(Branch Predictor) 캐시를 **강제 Flush**        │
  │     │ 4. Secure World의 이전 레지스터 상태를 복원.                 │
  │     ▼                                                       │
  │  (보안 앱 시작) "지문 비교 로직 실행... 안전하게 처리 완료!"              │
  │     │                                                       │
  │     ▼ (결과 반환 후 다시 SMC 호출)                                 │
  │  [ EL3 (Secure Monitor) ]                                   │
  │     │ 1. Secure World의 민감한 레지스터(지문 데이터)를 백업.          │
  │     │ 2. **레지스터 100% Zeroing 및 하드웨어 캐시 초기화 (필수!)**     │
  │     │ 3. Normal World 상태 복원 후 복귀.                         │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 일반적인 OS 문맥 교환은 '성능'이 최우선이다. A가 쓰던 레지스터를 A의 메모리 공간(PCB)에 밀어 넣기만 할 뿐, CPU 내부에 남은 찌꺼기 값을 굳이 지우지는 않는다. 그래서 보안에 취약하다.
하단의 **TrustZone 스위칭(SMC, Secure Monitor Call)**은 '보안'이 최우선이다. 일반 세상(안드로이드)에서 보안 세상(지문, 삼성페이 등)으로 넘어갈 때는 가운데 있는 초법적 기관인 **EL3 (Secure Monitor)**를 무조건 거쳐야 한다. EL3는 스위칭을 할 때마다 결벽증 환자처럼 모든 레지스터와 하드웨어 예측기 캐시를 0으로 밀어버린다(Scrubbing). 만약 이 Zeroing 과정을 한 줄이라도 빼먹으면, 일반 안드로이드에 깔린 악성 앱이 지문 데이터나 암호키의 찌꺼기를 레지스터에서 주워 담을 수 있기 때문이다.

### 심층 동작 원리: 스펙터 방어를 위한 마이크로아키텍처 초기화
과거에는 레지스터만 '0'으로 만들면 안전한 줄 알았다. 하지만 스펙터(Spectre v2) 공격이 터지면서, **분기 예측기(Branch Target Buffer, BTB)**라는 보이지 않는 CPU 내부 하드웨어도 초기화해야 한다는 것이 밝혀졌다.
A 프로세스가 수만 번의 `if`문을 실행하여 분기 예측기를 특정 방향으로 훈련시켜 놓은 상태에서 문맥 교환이 일어나면, B 프로세스는 이 훈련된 예측기를 이용해 투기적 실행(Speculative Execution)을 유도하고 캐시 타이밍을 재어 A의 데이터를 훔칠 수 있다. 이를 막기 위해 최신 CPU는 문맥 교환 시 OS가 강제로 **IBPB(Indirect Branch Predictor Barrier)** 명령어를 호출하여 예측기를 백지상태로 리셋하도록 강제한다. 이로 인해 현대 OS의 컨텍스트 스위칭 비용이 폭등했다.

## Ⅲ. 융합 비교 및 다각도 분석

### 컨텍스트 스위칭 수준별 오버헤드와 보안성

| 스위칭 종류 | 대상 | 발생 비용 (Overhead) | 보안성 및 초기화 범위 |
|:---|:---|:---|:---|
| **Thread Switch (동일 프로세스)** | 스택 포인터, 기본 레지스터 | 낮음 (수천 클럭) | 메모리 공유로 인해 보안 격리 거의 없음 |
| **Process Switch (서로 다른 프로세스)**| 레지스터 + CR3(페이지 테이블) 갱신 | 중간 (TLB 갱신 등으로 지연) | 주소 공간 격리. 레지스터는 지우지 않음 (최근 취약함) |
| **VM Switch (가상 머신 간)** | VMCS 저장/복원, EPT 전환 | 높음 (VM Exit / Entry) | 하드웨어 수준의 물리적 격리 지원 |
| **Secure World Switch (TEE / SGX)**| 레지스터 Zeroing, 분기/캐시 Flush | **매우 높음 (수만~수십만 클럭)** | **최상. 사이드 채널까지 차단하는 완전한 상태 소거** |

### 과목 융합 관점
- **보안 및 클라우드 (멀티 테넌시 위협)**: 클라우드 서버(EC2 등)의 코어 1개에 고객 A의 VM과 고객 B의 VM이 번갈아 가며 스케줄링된다. AWS나 Azure의 커스텀 하이퍼바이저는 KVM을 튜닝하여, A에서 B로 문맥 교환이 일어날 때 L1 데이터 캐시(L1D)를 하드웨어 플러시(Flush) 명령어로 싹 비워버린다(L1D Flushing). 이 캐시 비우기 하나 때문에 클라우드 전체 성능이 5~10% 하락하지만, 고객 간의 사이드 채널 해킹을 막기 위해 억지로 감수하는 것이다.
- **운영체제 (KPTI, Kernel Page-Table Isolation)**: 과거 리눅스는 성능을 위해 커널의 주소 공간을 유저 프로세스와 공유했다(권한만 막아둠). 하지만 멜트다운(Meltdown) 공격으로 권한이 뚫리자, 리눅스는 문맥 교환 시 아예 **유저 페이지 테이블과 커널 페이지 테이블을 물리적으로 뜯어버리는 KPTI 아키텍처**를 도입했다. 커널로 진입할 때마다 페이지 테이블을 통째로 교체(CR3 스위칭)해야 하므로 시스템 콜을 부를 때 엄청난 성능 저하가 발생했다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 웹 어셈블리(WASM) 기반 서버리스(Serverless) 엣지 환경**: Cloudflare Workers나 AWS Lambda처럼 1개의 프로세스 안에서 1,000명의 유저 코드가 밀리초 단위로 생성되고 사라진다.
   - **보안 및 아키텍처 과제**: 무거운 VM 스위칭이나 OS 프로세스 스위칭을 쓰면 서버리스의 생명인 '콜드 스타트(Cold Start) 0초'를 달성할 수 없다. 따라서 V8 엔진 내부의 WASM 런타임 샌드박스로 문맥을 교환하는데, 하드웨어 보호가 없기 때문에 V8 엔진이 직접 스레드의 메모리를 Zeroing 하고 레지스터를 스크러빙(Scrubbing)하는 **소프트웨어적 안전한 컨텍스트 스위칭**을 구현해야 한다. 
2. **시나리오 — 클라우드 보안 노드 (Confidential Computing) 구축**: 병원의 환자 데이터를 AI로 분석해야 하는데, 데이터가 메모리에 올라가는 순간 클라우드 관리자나 하이퍼바이저가 이를 훔쳐볼까 두려움.
   - **아키텍처 적용**: 인텔 SGX(Software Guard Extensions)나 AMD SEV 기술을 쓴다. CPU 내부에 '엔클레이브(Enclave)'라는 암호화된 메모리 구역을 만든다. 일반 프로세스에서 엔클레이브로 들어갈 때(안전한 문맥 교환), 하드웨어는 EENTER 명령어를 통해 외부의 인터럽트를 모조리 차단하고 일반 레지스터를 격리한다. 빠져나올 때(EEXIT)는 **CPU가 알아서 엔클레이브 안에서 쓴 모든 레지스터를 하드웨어적으로 0으로 덮어버린(Synthetic State) 뒤에만** 밖으로 나갈 수 있게 강제한다.

### 도입 체크리스트
- **SMT (Simultaneous Multi-Threading / 하이퍼스레딩) 비활성화**: 아무리 문맥 교환 시 레지스터를 지우고 캐시를 비워도(Temporal Isolation), 하이퍼스레딩이 켜져 있으면 코어 1개를 2개의 스레드가 '동시에' 공유하므로(Spatial Sharing) 방어가 불가능하다(예: 포트 스매시 공격). 따라서 극도의 보안이 필요한 클라우드 인스턴스나 암호화 서버를 설계할 때는 아키텍트가 BIOS에서 강제로 **하이퍼스레딩(SMT)을 Disable 처리**하는 것이 사이버 보안의 불문율이다.

### 정량/정성 기대효과

| 구분 | 일반 문맥 교환 | 안전한 문맥 교환 (Secure Scrubbing 적용) | 개선 효과 |
|:---|:---|:---|:---|
| **정성 (데이터 유출 방어)** | 이전 테넌트의 암호키/평문 유출 위험 | 레지스터 0 초기화로 유출 원천 차단 | 메모리 및 레지스터 기반의 스누핑(Snooping) 완벽 차단 |
| **정량 (사이드 채널 방어)** | 분기 예측기 등을 통한 99% 확률 탈취 | BTB, L1D Flush로 마이크로 상태 제거 | 스펙터/멜트다운 계열 마이크로아키텍처 취약점 억제 |
| **정량 (성능 페널티)** | 매우 빠름 (마이크로초 이하) | **추가적인 하드웨어 Flush 발생 (지연 2~5배 증가)** | 완벽한 보안을 대가로 클라우드 노드의 전체적인 연산량 손실 |

### 미래 전망
- **하드웨어 파티셔닝 (Hardware Partitioning)으로 회귀**: 시간적으로 쪼개 쓰는 '문맥 교환' 방식 자체가 본질적인 사이드 채널 공격을 피할 수 없음을 깨닫고 있다. 미래의 클라우드 칩 아키텍처(예: Intel RDT 확장, CXL)는 캐시 메모리나 분기 예측기를 아예 하드웨어적으로 칸막이를 쳐서(Partitioning) 물리적으로 공간을 영구 분리해 버릴 것이다. 이렇게 되면 문맥 교환 시 비싸게 캐시를 비우거나 레지스터를 청소할 필요가 사라진다.
- **컨텍스트 스위치 하드웨어 오프로딩**: 수만 번 반복되는 레지스터 클리어와 캐시 플러시 작업을 OS 커널(소프트웨어)이 직접 명령을 내려 수행하는 대신, 최신 CPU들은 이를 아예 하나의 기계어 명령어 한 줄(`VMENTRY_SECURE` 등)로 만들어 마이크로코드 단에서 빛의 속도로 일괄 소거해 버리는 기능적 발전을 이루고 있다.

### 참고 표준
- **ARM TrustZone SMC (Secure Monitor Call) 규격**: ARM 아키텍처에서 Normal World와 Secure World 간의 컨텍스트를 스위칭할 때 반드시 지켜야 하는 레지스터 백업 및 초기화(Scrubbing) 절차를 명시한 하드웨어 및 펌웨어(TF-A) 보안 설계 표준.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[신뢰 실행 환경 (TEE, TrustZone / SGX)](./478_tee.md)**: 안전한 컨텍스트 스위칭이 가장 빡세고 엄격하게 일어나는 핵심 보안 아키텍처 구역.
- **[문맥 교환 (Context Switch)](../2_operating_system/2_process_thread/xx_context_switch.md)**: 운영체제가 멀티태스킹을 위해 수행하는 기본 동작. 이 과정이 무기화(Weaponized)되어 정보가 유출되는 것을 막는 것이 목표다.
- **[사이드 채널 공격 (Side-channel Attack)](./481_side_channel_attack.md)**: 레지스터만 지우면 안전할 줄 알았으나, 캐시와 분기 예측기라는 '숨겨진 문맥(Microarchitectural Context)'을 통해 데이터를 훔쳐 가는 해킹 기법.
- **[가상 머신 제어 구조 (VMCS)](./529_vmcs.md)**: 가상머신 간의 문맥 교환 시 레지스터 상태를 안전하게 담아두는 하드웨어 격리 보관소.

## 👶 어린이를 위한 3줄 비유 설명
1. 호텔 방(CPU)에 나쁜 도둑(프로세스 A)이 묵다가 나갔어요. 그다음에 착한 손님(프로세스 B)이 그 방에 들어올 차례예요.
2. 예전에는 청소부(OS)가 그냥 침대 시트(기본 레지스터)만 휙 갈고 끝냈어요. 그랬더니 도둑이 쓰레기통에 몰래 숨겨둔 도청 장치(잔여 데이터)로 손님의 비밀번호를 훔쳐 갔어요!
3. 그래서 이제는 손님이 바뀌기 전에 **방 안의 모든 가구를 소독하고, 쓰레기통을 텅 비우고, 공기까지 완벽하게 환기(안전한 컨텍스트 스위칭)**시켜서 도둑의 흔적을 0.001%도 안 남기는 완벽한 청소를 한답니다!
