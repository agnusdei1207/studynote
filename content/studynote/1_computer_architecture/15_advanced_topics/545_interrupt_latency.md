# 545. 인터럽트 지연 시간 (Interrupt Latency) 최소화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하드웨어 장치(센서, 타이머 등)가 CPU에 비상 신호(Interrupt)를 보낸 시점부터, CPU가 하던 일을 멈추고 해당 비상 상황을 처리하는 첫 번째 코드(ISR)를 실행하기까지 걸리는 **'응답 대기 시간'을 마이크로초(µs)나 나노초(ns) 단위로 극단적으로 줄이는 아키텍처 기술**이다.
> 2. **가치**: 자율주행차의 긴급 제동, 공장 로봇 팔 제어, 미사일 궤도 수정 등 '확정적 시간(Deterministic Timing)' 내에 반드시 응답해야만 하는 **실시간 운영체제(RTOS)**와 임베디드 시스템의 생사를 가르는 가장 중요한 성능 지표다.
> 3. **융합**: 하드웨어적인 '벡터형 인터럽트 컨트롤러(VIC)', '꼬리 물기(Tail-chaining)', '늦은 도착(Late-arriving)' 기술과 소프트웨어적인 '인터럽트 비활성화(CLI) 구간 최소화' 전략이 결합하여 지연 시간을 0에 수렴하게 만든다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 인터럽트 지연 시간(Interrupt Latency)은 "외부 장치에서 인터럽트 신호 발생 $\rightarrow$ CPU 파이프라인 정지 $\rightarrow$ 현재 상태(Context)를 스택에 저장 $\rightarrow$ 인터럽트 벡터 테이블 조회 $\rightarrow$ ISR(Interrupt Service Routine) 첫 줄 실행"까지 걸리는 총시간이다. 이 시간을 줄이는 최적화 과정이 지연 시간 최소화다.
- **💡 비유**: 요리사(CPU)가 볶음밥을 만들고 있는데, 갑자기 화재경보기(인터럽트 신호)가 울렸다. 요리사가 가스 불을 끄고, 프라이팬을 내려놓고, 앞치마를 벗고(상태 저장), 소화기가 어디 있는지 매뉴얼을 찾아본(벡터 테이블 조회) 뒤에야 소화기를 잡는다면 골든타임을 놓친다. **인터럽트 지연 최소화**는 화재경보기가 울리자마자 0.1초 만에 앞치마를 멘 채로 바로 소화기를 쏠 수 있게 훈련하는 것이다.
- **등장 배경 및 발전 과정**:
  1. **초기 CPU (순차적 처리)**: 과거에는 인터럽트가 들어오면 CPU가 하던 복잡한 명령어(예: 64비트 나눗셈)가 다 끝날 때까지 수십 클럭을 기다려야만 했다.
  2. **범용 OS의 딜레마**: Windows나 Linux 같은 범용 OS는 디스크 I/O를 빠르게 처리하기 위해 스스로 인터럽트를 꺼버리는(Interrupt Disable) 구간을 길게 가져간다. 이 구간에 마우스나 키보드 인터럽트가 들어오면 수 밀리초(ms)가 밀려버린다. (버벅거림 발생)
  3. **RTOS와 마이크로컨트롤러의 진화**: ARM Cortex-M 시리즈 등은 태생부터 지연 시간을 줄이기 위해 칩 내부에 하드웨어로 상태를 자동 저장하고 복원하는 영리한 컨트롤러(NVIC)를 박아 넣었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (인터럽트 처리 파이프라인의 오버헤드 요인)

인터럽트 지연 시간은 크게 하드웨어 지연과 소프트웨어 지연으로 나뉜다.

| 지연 요인 (Latency Source) | 주체 | 동작 및 특징 | 비유 |
|:---|:---|:---|:---|
| **Pipeline Flush (파이프라인 비우기)** | 하드웨어 | 현재 실행 중이던/예측했던 명령어들을 모두 취소하고 버림 | 요리하던 재료 쓰레기통에 버리기 |
| **Context Saving (문맥 저장)** | 하드웨어/SW | R0~R15 레지스터, PC, LR 등을 스택(메모리)에 PUSH 함 | 요리 도구 제자리에 내려놓기 |
| **Vector Fetch (벡터 주소 패치)** | 하드웨어 | 메모리에 있는 인터럽트 벡터 테이블에서 실행할 ISR 주소를 읽어옴 | 소화기 위치가 적힌 매뉴얼 찾기 |
| **Interrupt Disabled (인터럽트 무시 구간)** | 소프트웨어 | OS가 `CLI(Clear Interrupt)`로 인터럽트를 막아둔 '크리티컬 섹션' 구간 대기 | 요리사가 귀마개 쓰고 일하던 시간 |

### 하드웨어 기반 인터럽트 지연 최소화 아키텍처 (ARM NVIC 기준)

소프트웨어가 스택에 PUSH/POP을 하려면 코드가 길어지고 느리다. 현대 마이크로컨트롤러는 이를 하드웨어 회로가 가로채어 순식간에 처리한다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         일반 인터럽트 처리 vs 꼬리 물기 (Tail-chaining) 최적화         │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 일반적인 인터럽트 중첩 처리 (비효율적)                           │
  │                                                             │
  │  (메인 코드) ──▶ [IRQ 1 발생]                                  │
  │                   1. Context PUSH (수십 클럭 소요)             │
  │                   2. ISR 1 실행                              │
  │                   3. Context POP (복원 중... 앗! IRQ 2 발생)    │
  │  (메인 코드) ◀── (복귀)                                       │
  │               └──▶ 4. 다시 Context PUSH (또 수십 클럭 낭비!)    │
  │                   5. ISR 2 실행                              │
  │                   6. Context POP                             │
  │  (메인 코드) ◀── (최종 복귀)                                    │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] Tail-chaining (꼬리 물기) 하드웨어 아키텍처 - "오버헤드 소멸"       │
  │                                                             │
  │  (메인 코드) ──▶ [IRQ 1 발생]                                  │
  │                   1. 하드웨어가 자동 Context PUSH (12 클럭)     │
  │                   2. ISR 1 실행                              │
  │                        [IRQ 2 발생 (대기열에 등록)]             │
  │                   3. ISR 1 종료. (원래는 POP 해야 하지만...)      │
  │                   ▶ "어차피 IRQ 2 할 거잖아? POP 생략하고 바로 넘어가!" │
  │                   4. 하드웨어가 단 6클럭 만에 ISR 2로 점프!        │
  │                   5. ISR 2 실행                              │
  │                   6. 하드웨어가 자동 Context POP (최종 복원)     │
  │  (메인 코드) ◀── (최종 복귀)                                    │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 **일반 방식**은 인터럽트가 연속해서 들어올 때 최악이다. 첫 번째 ISR이 끝나고 원래 코드로 돌아가기 위해 레지스터를 다 복원(POP)했는데, 0.1초 뒤에 두 번째 인터럽트가 터지면 아까 복원했던 레지스터를 다시 스택에 집어넣는(PUSH) 뻘짓을 해야 한다.
하단의 **꼬리 물기(Tail-chaining)**는 ARM Cortex-M의 NVIC(Nested Vectored Interrupt Controller)가 자랑하는 궁극의 최적화다. 첫 번째 ISR이 끝났을 때 대기 중인 두 번째 인터럽트가 있다면, **POP과 PUSH 과정을 아예 생략(Bypass)**해버린다. 레지스터는 이미 스택에 안전하게 보관되어 있으므로, 단순히 PC(프로그램 카운터)만 ISR 2의 주소로 휙 돌려버린다. 문맥 교환 지연 시간이 1/4로 줄어드는 마법이 일어난다.

### 심층 최적화 기법: 늦은 도착 (Late-Arriving Interrupt)
하드웨어가 인터럽트 1번(우선순위 낮음)을 처리하려고 열심히 레지스터를 스택에 PUSH 하고 있는 도중에, 갑자기 인터럽트 2번(우선순위 높음, 예: 긴급 제동)이 터졌다.
기존 CPU는 일단 인터럽트 1번을 시작(ISR 진입)한 뒤에, 즉시 다시 2번으로 점프(중첩)하는 멍청한 짓을 했다. 하지만 최신 아키텍처의 **늦은 도착(Late-Arriving)** 기능은 PUSH를 다 끝낸 직후, "어? 더 급한 놈(2번)이 왔네?" 하고 원래 가려던 ISR 1번 주소를 버리고 즉시 **ISR 2번으로 새치기 점프**를 해버린다. 급한 인터럽트의 지연 시간을 극단적으로 0에 수렴하게 만든다.

## Ⅲ. 융합 비교 및 다각도 분석

### GPOS vs RTOS의 인터럽트 처리 철학 비교

범용 운영체제(Windows/Linux)와 실시간 운영체제(VxWorks/FreeRTOS)는 철학이 반대다.

| 비교 항목 | GPOS (General Purpose OS) | RTOS (Real-Time OS) |
|:---|:---|:---|
| **최우선 목표** | 전체 시스템의 처리량(Throughput) 극대화 | **인터럽트 응답의 확정성(Determinism)** |
| **인터럽트 무시(Disable) 구간**| 커널 락(Spinlock) 등에서 수 밀리초 이상 길게 유지 | **수 마이크로초(µs) 이하로 엄격히 제한** (금기시됨) |
| **ISR 구조** | Top Half(최소한) + Bottom Half(연기된 작업)로 분리 | ISR 내부에서 즉시 핵심 연산 완료 권장 |
| **지연 시간 (Latency)** | 상황에 따라 1ms ~ 100ms까지 널뜀 (Jitter 심함) | **무조건 10µs 이내 보장 (Hard Real-Time)** |

### 과목 융합 관점
- **운영체제 (선점형 커널, Preemptible Kernel)**: 리눅스 커널은 2.6 버전 이전에는 커널 공간에서 코드가 실행 중일 때 인터럽트가 들어와도 즉시 스위칭을 해주지 않았다(비선점). 이를 해결하기 위해 커널 공간에서도 언제든 멈추고 인터럽트를 처리할 수 있는 `PREEMPT_RT` 패치가 등장하여 범용 리눅스를 실시간 OS처럼 쓸 수 있게 만들었다.
- **소프트웨어 공학 (ISR 내부 코딩 규칙)**: 인터럽트 지연을 줄이는 가장 중요한 주체는 사실 프로그래머다. ISR 함수 내부에서는 절대 `printf()`(I/O 대기)나 `malloc()`(동적 할당 락), `sleep()` 같은 블로킹 함수를 쓰면 안 된다. ISR이 늦게 끝나면 그사이에 들어오는 더 낮은 우선순위의 인터럽트들이 전부 지연(Starvation)되기 때문이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 드론(드론) 비행 제어기(FC)의 자이로 센서 데이터 처리**: 드론은 초당 1,000번(1kHz) 자이로 센서의 값을 읽어 모터의 회전수를 실시간으로 보정해야 추락하지 않는다.
   - **아키텍처 적용**: 센서에서 데이터가 준비되었다는 핀(EXTI) 신호가 들어올 때마다 CPU에 인터럽트가 걸린다. 이때 인터럽트 지연 시간이 1ms를 넘어가면(Jitter 발생) 모터 제어 타이밍이 꼬여 드론이 흔들리거나 추락한다. 따라서 메인보드에 ARM Cortex-R(Real-time) 코어를 박아 넣고, 하드웨어 **FPU(부동소수점 연산기)의 레지스터 스태킹(Stacking)을 지연(Lazy)**시켜 인터럽트 반응 속도를 20나노초(ns) 이내로 묶어버린다.
2. **시나리오 — 고성능 네트워크 스위치 (DPDK와의 융합)**: 100Gbps 랜카드에서 1초에 1,400만 개의 패킷이 쏟아져 들어온다.
   - **엔지니어링 판단 (Interrupt Coalescing)**: 초저지연이 중요하다고 패킷 1개가 올 때마다 인터럽트를 1,400만 번 발생시키면, CPU는 Context Switch만 하다가 100% 뻗어버린다(Interrupt Storm). 이 경우에는 오히려 지연 시간 최소화를 포기한다. 칩셋에 **인터럽트 병합(Interrupt Coalescing)**을 걸어 패킷이 100개 쌓일 때까지 기다렸다가 한 번만 인터럽트를 걸게 하거나, 아예 인터럽트를 끄고 CPU가 무한 루프를 돌며 패킷을 퍼오는 **폴링(Polling, DPDK 기술)** 방식으로 전환하는 역발상이 필요하다.

### 도입 체크리스트
- **TCM (Tightly Coupled Memory) 사용 여부**: 인터럽트 벡터 테이블이나 ISR 코드가 메인 메모리(DRAM)에 있으면, 캐시 미스 발생 시 인터럽트 지연 시간이 10배 이상 튄다(Jitter). 확정적인(Deterministic) 실시간 응답을 원한다면, 반드시 칩 내부에 캐시가 아닌 **절대 지연이 발생하지 않는 전용 고속 SRAM인 TCM(밀결합 메모리)** 공간에 ISR 코드와 스택을 강제로 매핑(Linker Script 조작)시켜야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 소프트웨어 (GPOS) 인터럽트 처리 | 하드웨어 (RTOS/NVIC) 최적화 도입 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (지연 시간, Latency)**| 수십~수백 마이크로초(µs) 소요 | **10~20 클럭 (나노초 단위) 소요** | 긴급 제동 등 생명 직결 시스템의 골든타임 확보 |
| **정량 (Jitter, 지연의 널뜀)**| 캐시 상태에 따라 편차가 매우 심함 | TCM 및 NVIC 도입으로 항상 고정된 클럭 소요 | **확정성(Determinism) 100% 보장** |
| **정성 (시스템 안정성)** | 인터럽트 폭풍 시 전체 시스템 마비 | 우선순위 강제 할당 및 꼬리 물기로 깔끔한 정리 | 실시간 임베디드 기기의 완벽한 신뢰성 구축 |

### 미래 전망
- **SoC 내부의 분산 인터럽트 처리**: 코어가 128개씩 달린 서버 칩에서는 하나의 인터럽트 컨트롤러(APIC)가 모든 신호를 처리하면 병목이 생긴다. 차세대 아키텍처는 인터럽트를 메시지화(MSI, Message Signaled Interrupts)하여 PCIe 버스를 통해 메모리 쓰기처럼 특정 코어로 다이렉트로 쏴버리는 네트워크형 인터럽트 방식으로 진화하고 있다.
- **AI 칩(NPU)의 비동기 실행 (Asynchronous Execution)**: 미래의 시스템은 CPU가 지연 시간을 겪어가며 인터럽트를 직접 처리할 필요조차 없어진다. 스마트폰 카메라 센서가 켜지면, 센서가 CPU를 깨우는(인터럽트) 대신 센서 컨트롤러가 다이렉트로 NPU(인공지능 칩)를 깨우고 DMA로 데이터를 쏴버려 CPU는 꿀잠(Sleep)을 자도록 놔두는 **Zero-CPU Intervention** 구조로 설계가 바뀌고 있다.

### 참고 표준
- **ARM NVIC (Nested Vectored Interrupt Controller)**: ARM Cortex-M 시리즈에 탑재되는 하드웨어 모듈 스펙. Tail-chaining, Late-arriving 등 실시간 임베디드 시스템의 인터럽트 처리 표준을 정립한 전설적인 아키텍처 규격이다.

## 👶 어린이를 위한 3줄 비유 설명
1. 게임기(CPU)를 하다가 엄마가 "밥 먹어!" 하고 부르시면(인터럽트 발생), 하던 게임을 멈추고 세이브하고 숟가락을 들기까지 꽤 시간이 걸리죠? 이 시간을 **인터럽트 지연 시간**이라고 해요.
2. 예전에는 게임 세이브를 직접 해야 해서 5분이 걸렸다면, 최신 게임기(NVIC 하드웨어)는 엄마 목소리가 들리는 순간 자동으로 0.1초 만에 자동 저장(Auto-save)을 해줘요.
3. 심지어 밥 먹고 돌아왔을 때 양치질하라는 다른 심부름이 있으면, 번거롭게 게임을 켰다 다시 끄지 않고(꼬리 물기) 바로 양치질을 하러 가기 때문에 훨씬 빨리 모든 심부름을 끝낼 수 있답니다!
