# 539. 루프 타일링 (Loop Tiling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 거대한 배열이나 행렬을 다루는 중첩 반복문(Nested Loop)을 실행할 때, 전체 데이터를 한 번에 순회하지 않고 **CPU의 고속 L1/L2 캐시(Cache) 크기에 딱 맞는 작은 조각(Tile 또는 Block)으로 나누어 처리하는 컴파일러/알고리즘 최적화 기법**이다.
> 2. **가치**: 캐시에 올라온 데이터를 메모리로 쫓아내기 전에 최대한 뽕을 뽑아 재사용(Data Locality 극대화)함으로써, 폰 노이만 아키텍처의 가장 큰 약점인 메모리 병목(Memory Wall)을 타파하고 연산 속도를 수십 배 이상 향상시킨다.
> 3. **융합**: 딥러닝의 핵심인 거대 행렬 곱셈(Matrix Multiplication)과 영상 처리 알고리즘에서 절대 빠질 수 없는 필수 최적화 기술이며, 하드웨어의 캐시 크기와 소프트웨어의 접근 패턴을 융합하는 '캐시 인지(Cache-aware) 최적화'의 꽃이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 루프 타일링(Loop Tiling, 또는 루프 블로킹 Loop Blocking)은 큰 데이터셋을 처리하는 루프를 잘게 쪼개어, 각 조각(타일)이 CPU 내부의 작은 고속 캐시 메모리 안에 온전히 들어갈 수 있도록 루프의 순회 순서(Iteration Space)를 재배열하는 기술이다.
- **💡 비유**: 도서관(메인 메모리)에서 1만 권의 책을 찾아와야 한다. 한 번에 책 1권씩만 책상(캐시)에 가져와서 읽고 다시 반납하는 짓을 1만 번 반복하면 도서관을 왔다 갔다 하느라 하루가 다 간다. 대신, **책상 크기에 딱 맞게 한 번에 책 100권(타일)을 빌려와서 다 읽은 뒤 한꺼번에 반납**하는 것을 100번 반복하면 이동 시간이 1/100로 줄어든다.
- **등장 배경 및 발전 과정**:
  1. **Memory Wall의 도래**: CPU 연산 속도는 빛의 속도로 빨라지는데, 메인 메모리(DRAM)에서 데이터를 가져오는 속도는 거북이걸음이었다. 결국 CPU가 계산은 안 하고 데이터가 올 때까지 놀고 있는 캐시 미스(Cache Miss) 현상이 심각해졌다.
  2. **행렬 연산의 비효율 발견**: 특히 수학과 과학 시뮬레이션에서 10,000 x 10,000 크기의 행렬을 곱할 때, 첫 번째 행을 계산하고 두 번째 행을 계산하려니 이미 캐시가 꽉 차서 첫 번째 행 데이터가 다 지워져 버리는 참사가 발생했다.
  3. **알고리즘적 회피 기술 개발**: 이를 막기 위해 컴파일러와 수학 라이브러리(BLAS 등) 개발자들은 전체 행렬을 작게 잘라(예: 64x64 블록) 그 블록 안에서만 연산을 다 끝내고 다음 블록으로 넘어가는 '타일링' 알고리즘을 도입했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (타일링 최적화 요소)

| 요소명 | 역할 | 특징 | 비유 |
|:---|:---|:---|:---|
| **Iteration Space (반복 공간)** | 다중 루프가 탐색해야 할 전체 데이터의 영역 | 2중 루프면 2차원 평면, 3중 루프면 3차원 입체 공간 | 도서관의 전체 서고 |
| **Tile Size / Block Factor** | 한 번에 처리할 조각(타일)의 크기 (예: $B \times B$) | **타겟 CPU의 L1/L2 캐시 크기를 초과하지 않도록** 신중하게 계산되어야 함 | 한 번에 올릴 수 있는 책상의 크기 |
| **Temporal Locality (시간 지역성)** | 방금 쓴 데이터를 쫓겨나기 전에 다시 쓰는 성질 | 타일링의 주 목적. 타일 밖으로 나가기 전에 해당 타일의 데이터를 극한으로 재활용함 | 빌려온 100권을 완전히 마스터하기 |
| **Cache Miss Penalty (캐시 미스 지연)** | 캐시에 데이터가 없어 DRAM까지 다녀오는 시간 | 타일링을 하지 않으면 행렬 곱셈 시 이 페널티가 $O(N^3)$ 단위로 폭발함 | 도서관 왕복 시간 |

### 일반 행렬 곱셈 vs 타일링 행렬 곱셈 구조 비교

행렬 곱 연산 $C = A \times B$ 의 메모리 접근 패턴을 비교한다. (배열 크기 $N \times N$)

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         일반 중첩 루프 vs 루프 타일링(Blocking) 메모리 접근 패턴 비교   │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 일반 순차 방식 (Standard Nested Loop) - "캐시 대참사 유발"     │
  │                                                             │
  │   for (i=0; i<N; i++)                                       │
  │     for (j=0; j<N; j++)                                     │
  │       for (k=0; k<N; k++)                                   │
  │         C[i][j] += A[i][k] * B[k][j];                       │
  │                                                             │
  │   * 치명적 문제: 행렬 B는 세로(k축)로 접근함. C에서는 가로 배열이 기본인데,│
  │               세로로 징검다리 뛰듯 접근하면 한 칸 읽을 때마다 무조건  │
  │               캐시 미스가 발생함. (N이 크면 캐시 오염 극심)           │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] 루프 타일링 (Tiled/Blocked Loop) - "캐시 안에서 뽕 뽑기!"      │
  │                                                             │
  │   for (i=0; i<N; i+=B)        // 거대한 루프를 타일(B) 단위로 쪼갬 │
  │     for (j=0; j<N; j+=B)                                    │
  │       for (k=0; k<N; k+=B)                                  │
  │         // 이제부터는 [B x B] 크기의 작은 타일 안에서만 논다!       │
  │         for (ii=i; ii<i+B; ii++)                            │
  │           for (jj=j; j<j+B; jj++)                           │
  │             for (kk=k; k<k+B; kk++)                         │
  │               C[ii][jj] += A[ii][kk] * B[kk][jj];           │
  │                                                             │
  │   * 장점: [B x B] 크기의 A, B 타일 덩어리가 L1 캐시에 한 번 올라오면, │
  │           그 타일 계산이 다 끝날 때까지 데이터가 캐시 밖으로 쫓겨나지 │
  │           않고 계속 재활용됨. DRAM 왕복 횟수가 기적처럼 감소!       │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 **일반 방식**은 메모리가 무한대라고 착각하고 짠 코드다. 행렬이 10,000 x 10,000 크기라면, 한 줄(Row) 크기만 수십 MB다. L1 캐시(보통 32KB)는 몇 칸 읽기도 전에 꽉 차서 뒤의 데이터를 부르기 위해 앞의 데이터를 다 버린다. 방금 읽었던 데이터를 1초 뒤에 또 써야 하는데 이미 버려서 또 메인 메모리까지 가야 한다(Cache Thrashing).
하단의 **타일링 방식**은 큰 행렬을 64x64 사이즈(캐시에 쏙 들어가는 크기)의 색종이(Tile) 수천 장으로 잘라놓고, **"A 색종이 한 장과 B 색종이 한 장을 캐시에 가져왔으면, 이 색종이끼리 곱할 수 있는 건 다 곱할 때까지 절대 버리지 말자"**라는 철학을 적용한 것이다. 코드는 3중 루프에서 6중 루프로 복잡해지지만, 메모리 접근 속도가 수십 배 빠르므로 전체 계산 시간은 압도적으로 줄어든다.

---

### 심층 극복 과제: 캐시 라인(Cache Line) 충돌 피하기
타일 크기(B)를 무조건 L1 캐시 크기에 딱 맞춘다고 좋은 것이 아니다. 타일 사이즈가 2의 거듭제곱(예: 64, 128)일 때, 배열의 메모리 주소가 캐시의 특정 세트(Set)에만 집중적으로 매핑되어 다른 세트는 텅 비었는데 혼자 꽉 차서 쫓겨나는 **캐시 충돌(Conflict Miss)**이 발생할 수 있다. 고급 컴파일러 최적화 기법(Polyhedral Model)은 배열을 복사(Copy)하여 메모리에 연속적으로 재배치하거나 타일 사이즈를 일부러 소수(Prime number)나 비주기적인 크기로 잡아 이 충돌을 우회한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 루프 3대 최적화 기법 비교

컴파일러는 성능을 극대화하기 위해 이 세 가지 기술을 융합해서 쓴다.

| 최적화 기법 | 동작 방식 및 타겟 | 해결하려는 하드웨어 병목 | 부작용 / 한계 |
|:---|:---|:---|:---|
| **Loop Unrolling (언롤링)** | 루프를 복사해 세로로 길게 전개 | CPU 분기 페널티, 파이프라인 낭비 (ILP) | 코드 팽창 (Code Bloat) |
| **Auto-vectorization (벡터화)** | 스칼라 연산을 병렬 SIMD로 묶음 | ALU 연산기 낭비 방어 (DLP) | 복잡한 데이터 의존성 시 불가 |
| **Loop Tiling (타일링)** | 거대 루프를 캐시 크기로 블록 분할 | **캐시 미스로 인한 메모리 대기 시간 (Memory Wall)** | 2중/3중 다중 루프에만 유효. 코드가 몹시 복잡해짐 |

### 과목 융합 관점
- **아키텍처 (메모리 계층 구조)**: 폰 노이만 병목을 극복하는 하드웨어적 방법이 HBM(초광대역 메모리)을 다는 것이라면, 소프트웨어적인 궁극의 회피 방법이 루프 타일링이다. CPU 내부에 L1, L2, L3 캐시가 계층적으로 있듯이, 타일링도 L1용 타일, L2용 타일을 다단계(Multi-level Tiling)로 구성하여 극한의 지역성(Locality)을 짜낼 수 있다.
- **인공지능 가속 (GPU Tensor Core)**: AI 모델 학습의 90%는 `GEMM`(General Matrix Multiply) 연산이다. 엔비디아의 CUDA 코어나 cuBLAS 같은 최적화 라이브러리는 프로그래머가 짠 일반 코드를 GPU 내부의 공유 메모리(Shared Memory, 수동 제어 캐시) 크기에 맞춰 타일링하는 기법을 기본 탑재하고 있다. 이를 **공유 메모리 타일링(Shared Memory Tiling)**이라 부르며, GPU 성능을 결정짓는 핵심 요소다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 과학 시뮬레이션(CFD) HPC 서버 성능 최적화**: 기상청의 날씨 시뮬레이션 프로그램(Fortran/C++)이 128코어 서버에서 도는데 CPU 사용률은 20%밖에 안 되고 80%를 I/O 대기(Wait) 상태로 놀고 있다.
   - **아키텍처 튜닝**: `perf stat` 같은 프로파일링 도구로 찍어보니 **L1/L2 Cache Miss Rate가 40%**를 넘어가고 있었다. 3차원 공간의 유체 역학 배열을 계산할 때 루프 타일링이 적용되지 않은 것이다. 컴파일러에 `-O3`와 함께 `-floop-block` (GCC 타일링 옵션)을 주거나, 소스 코드를 OpenMP 라이브러리를 사용해 타일 단위로 명시적으로 쪼개어 캐시 미스율을 5% 미만으로 낮추자 CPU가 100% 풀로드를 찍으며 시뮬레이션 시간이 5일에서 1일로 단축되었다.
2. **시나리오 — 이미지 처리(필터링) 엣지 디바이스 설계**: 스마트폰 카메라가 4K 해상도(800만 픽셀) 이미지에 블러(Blur) 필터를 입히는 연산에서 배터리를 너무 많이 소모함.
   - **설계 적용**: 이미지 프로세싱도 결국 2차원 행렬 연산이다. 전체 사진의 첫 줄부터 끝줄까지 가로로 한 줄씩 계산하면 위아래 픽셀을 참조할 때마다 캐시 미스가 발생한다. 이미지를 32x32 픽셀 단위의 네모난 타일(Tile) 영역으로 잘라서 해당 타일 안의 필터링을 모두 끝낸 뒤 다음 타일로 넘어가는 **소프트웨어적 타일링 아키텍처**를 적용하면, 메모리 읽기/쓰기 횟수가 1/4로 줄어들어 모바일 AP의 배터리 수명을 극적으로 연장할 수 있다.

### 도입 체크리스트
- **하드웨어 종속성 주의**: 타일링의 마법(타일 크기 B 결정)은 대상 컴퓨터의 L1 캐시 사이즈에 절대적으로 의존한다. 내 노트북(캐시 32KB)에 맞춰 타일 크기를 최적화해 놓은 코드를, 클라우드 서버(캐시 1MB)나 다른 칩으로 가져가면 성능이 반토막 날 수 있다. 이를 막기 위해 컴파일할 때 `--march=native` 옵션을 주어 **"지금 이 코드를 컴파일하는 기계의 캐시 크기를 스스로 읽어와서 타일 사이즈를 결정해라"**라고 지시해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 타일링 미적용 (일반 중첩 루프) | 타일링 적용 (Blocked Loop) | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (캐시 미스율)**| N이 커질수록 100%에 수렴 (Thrashing) | 타일 내 데이터 재사용으로 **수 % 이내로 방어** | 행렬 연산 구간 성능 **10~50배 이상 폭증** |
| **정량 (메모리 대역폭)**| DRAM에서 계속 데이터를 퍼와야 함 | L1 캐시 내부에서만 연산 빙글빙글 돎 | 메모리 버스(FSB)의 트래픽 점유율 획기적 절감 |
| **정성 (컴파일러 복잡도)**| 개발자가 직관적으로 짠 3중 루프 | 컴파일러가 알아서 6중 루프로 뜯어고침 | 개발자는 비즈니스 로직에만 집중 가능 (투명성) |

### 미래 전망
- **머신러닝 컴파일러 (TVM / XLA)**: 딥러닝 모델(PyTorch 등)을 다양한 하드웨어(GPU, NPU, TPU)에 올릴 때, 예전에는 하드웨어 제조사가 제공하는 전용 수학 라이브러리(cuBLAS)를 썼다. 이제는 **Apache TVM** 같은 AI 전용 컴파일러가 모델의 행렬 수식을 통째로 받아들인 뒤, 타겟 하드웨어의 캐시 스펙에 맞춰 머신러닝(Auto-Tuning) 알고리즘으로 수만 번의 시뮬레이션을 돌려 **'가장 완벽한 타일 크기와 루프 모양'**을 스스로 찾아내 기계어로 굽는 시대로 접어들었다.
- **Cache-Oblivious Algorithms (캐시 인지 망각 알고리즘)**: 타일 크기를 하드웨어 스펙에 맞추는 짓 자체가 귀찮아서, 아예 "하드웨어 캐시가 몇 KB인지 몰라도, 알아서 모든 캐시 계층에 들어맞도록 분할 정복(Divide and Conquer) 방식으로 쪼개 들어가는 수학적 알고리즘"이 연구되고 있다. 하드웨어의 파편화가 심해지는 미래에는 이 방식이 대세가 될 것이다.

### 참고 표준
- **Polyhedral Compilation Model (다면체 컴파일 모델)**: 최신 컴파일러(GCC Graphite, LLVM Polly)가 루프 타일링과 병렬화를 수행하기 위해 사용하는 고급 수학 모델. 중첩 루프를 다차원 공간의 점(Point)들로 변환한 뒤 행렬 변환을 통해 캐시 효율성이 제일 높은 방문 순서를 수학적으로 찾아내는 기술 표준이다.

## 👶 어린이를 위한 3줄 비유 설명
1. 냉장고(메인 메모리)에 있는 재료로 요리를 1만 번 해야 하는데, 한 번 요리할 때마다 냉장고 문을 열었다 닫았다 하면 너무 힘들잖아요?
2. 그래서 도마(캐시 메모리) 크기에 딱 맞춰서, **냉장고에서 한 번에 요리할 재료 100개(타일)를 왕창 꺼내놓고** 100번 요리를 다 끝낸 뒤에 다시 냉장고에 가는 거예요.
3. 이렇게 작업할 덩어리(블록)를 도마 크기에 맞춰 영리하게 쪼개서 일하는 방식을 **루프 타일링**이라고 부른답니다! 요리 속도가 엄청나게 빨라지겠죠?
