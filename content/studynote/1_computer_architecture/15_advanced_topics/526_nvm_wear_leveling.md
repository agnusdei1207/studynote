# 526. 비휘발성 메모리 마모 평준화 (NVM Wear Leveling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 낸드 플래시(NAND Flash)나 PRAM 같은 비휘발성 메모리는 데이터를 지우고 쓸 수 있는 횟수(P/E Cycle)에 물리적 한계가 존재하므로, 특정 블록만 집중적으로 닳아서 칩이 조기 사망하는 것을 막기 위해 쓰기 작업을 칩 전체에 골고루 분산시키는 펌웨어 알고리즘이다.
> 2. **가치**: SSD와 같은 플래시 기반 저장장치의 **실질적인 수명(Endurance)**을 결정짓는 가장 중요한 소프트웨어 기술로, 이 알고리즘이 없으면 SSD는 운영체제가 부팅되는 며칠 만에 고장 나버린다.
> 3. **융합**: FTL(Flash Translation Layer)의 핵심 기능으로 가비지 컬렉션(Garbage Collection), 오버 프로비저닝(Over-Provisioning)과 삼위일체를 이루어 하드웨어의 물리적 수명 한계를 소프트웨어적으로 은닉(Hiding)한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 마모 평준화(Wear Leveling)는 데이터를 저장할 때 메모리 칩 내부의 각 블록(Block)들이 최대한 비슷한 횟수로 지워지고(Erase) 쓰이도록(Program) 논리적 주소와 물리적 주소의 맵핑을 동적으로 회전시키는 기법이다.
- **💡 비유**: 노트 필기를 할 때 보통 첫 페이지부터 쓴다. 만약 글씨를 틀리면 지우개로 빡빡 지우고 그 위에 다시 쓴다. 그런데 첫 페이지만 계속 1,000번을 쓰고 지우면 종이가 찢어져(배드 블록 발생) 노트를 버려야 한다. 이를 막기 위해 **'글씨를 틀릴 때마다 그 자리를 지우지 않고 새 페이지를 찾아 넘겨서 쓰는 것'**이 마모 평준화다. 노트의 모든 페이지를 골고루 다 쓰고 나서야 낡은 페이지들을 한 번에 지운다.
- **등장 배경 및 발전 과정**:
  1. **플로팅 게이트의 물리적 파괴**: 낸드 플래시에 데이터를 쓸 때는 고전압(약 20V)을 가해 전자를 절연막 너머로 억지로 밀어 넣는다(터널링). 이 짓을 수천 번 반복하면 절연막이 닳아 없어져 전자가 새어버리는 '수명(P/E Cycle)' 한계가 나타났다.
  2. **파일시스템의 무지 (FAT/NTFS)**: 기존 OS의 파일시스템은 디스크(HDD)를 무한정 덮어쓸 수 있다고 생각한다. FAT 테이블 같은 메타데이터 영역은 하루에도 수만 번 덮어쓰기가 발생하는데, 이를 플래시 메모리에 그대로 적용하면 해당 블록은 단 며칠 만에 수명을 다해 죽어버린다.
  3. **FTL (Flash Translation Layer)의 도입**: 칩 내부에 컨트롤러(FTL)를 달아, OS가 1번 논리 주소에 수만 번 덮어쓰기를 명령하더라도 실제로는 플래시의 1번, 2번, 3번 물리 블록으로 계속 자리를 옮겨가며 쓰도록 속이는(Mapping) 방어 기제가 탄생했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (Wear Leveling 알고리즘)

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **Erase Count Table** | 각 물리적 블록이 몇 번 지워졌는지 횟수를 추적 | FTL 내부 SRAM에 블록별 삭제 횟수(Erase Count)를 저장 및 갱신 | 각 타이어의 주행 마일리지 기록부 |
| **동적 마모 평준화 (Dynamic)** | 새로 들어오는 데이터(Write)를 가장 덜 지워진 빈 블록에 할당 | 핫 데이터(자주 변하는 데이터)를 닳지 않은 싱싱한 블록으로 보냄 | 짐을 실을 때 새 차에 싣기 |
| **정적 마모 평준화 (Static)** | 가만히 있는 콜드 데이터(OS 코어 등)를 억지로 이사시킴 | 삭제 횟수가 너무 적은 블록을 찾아 안의 데이터를 비우고(이사), 그 튼튼한 블록을 핫 데이터용으로 내어줌 | 차고에 방치된 새 차를 강제로 끌고 나와 일 시키기 |
| **배드 블록 관리 (BBM)** | 수명을 다해 에러가 나는 블록을 격리 | P/E Cycle을 초과해 쓰기 실패가 발생하면 해당 블록을 맵핑에서 영구 제외 | 펑크 난 타이어 폐기 |

---

### 마모 평준화 알고리즘 아키텍처 비교 (Dynamic vs Static)

SSD의 수명은 단순히 핫 데이터를 빈 곳에 쓰는 것만으로는 해결되지 않는다. 차가운 데이터(Cold Data)를 강제로 깨워야 한다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         마모 평준화(Wear Leveling)의 2가지 동작 방식 비교             │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 동적 마모 평준화 (Dynamic Wear Leveling) - "절반의 성공"          │
  │                                                             │
  │  (블록 상태)  [ OS 파일 (Cold) ]  [ 임시 파일 (Hot)이 계속 들어옴 ]      │
  │  물리 블록 0: ▓▓▓▓▓▓▓▓▓▓ (1회 씀, 이후 1년 내내 안 지워짐)              │
  │  물리 블록 1: ▒▒▒▒▒▒▒▒▒▒ (빈 공간) ◀─ (새 데이터는 무조건 빈 곳에 씀)      │
  │  물리 블록 2: ▒▒▒▒▒▒▒▒▒▒ (빈 공간) ◀─ (이곳들이 집중적으로 닳기 시작)      │
  │  * 문제점: 0번 블록은 1번만 지워지고 새것처럼 유지되지만,                   │
  │            1, 2번 블록만 계속 지웠다 써서(수천 번) 먼저 죽어버림.           │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] 정적 마모 평준화 (Static Wear Leveling) - "완벽한 평준화"           │
  │                                                             │
  │  * FTL의 판단: "0번 블록 혼자 수명이 너무 많이 남았잖아? 자리 바꿔!"         │
  │                                                             │
  │    (강제 이사)                                                │
  │    [ OS 파일 (Cold) ] ──────────┐                           │
  │  물리 블록 0: (완전 튼튼함) ◀──────┼── [ 임시 파일 (Hot) 강제 할당 ]  │
  │                                 │                           │
  │  물리 블록 1: (다 죽어감) ◀────────┘ (OS 파일을 낡은 블록에 넣어둠)    │
  │                                                             │
  │  * 결과: OS 파일은 어차피 수정(Erase)될 일이 없으므로, 다 죽어가는 블록에 │
  │          넣어둬도 더 이상 수명이 닳지 않음. 튼튼한 0번 블록을 핫 데이터용 │
  │          으로 쓰면서 디스크 전체의 수명(마일리지)이 완벽히 같아짐.          │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 **동적(Dynamic)** 방식은 디스크에 '새로 들어오는 데이터'만 가장 덜 닳은 블록에 골라서 써주는 방식이다. 이 방식의 치명적 단점은 윈도우 OS나 저장된 영화 파일(Cold Data)처럼 한 번 써놓고 절대 수정하지 않는 파일이 차지한 블록은 수명이 전혀 닳지 않은 채 방치된다는 점이다. 남은 빈 공간(블록)들만 계속 재활용되다 보니 디스크 절반이 멀쩡한데도 나머지 절반이 먼저 닳아 SSD 전체가 고장 난다.
하단의 **정적(Static)** 방식은 FTL이 백그라운드에서 전체 블록의 지우기 횟수(Erase Count)를 모니터링한다. 특정 블록이 너무 오랫동안 1회(수정 안 됨)에 머물러 있으면, 그 안의 콜드 데이터를 꺼내서 '수명이 간당간당한 낡은 블록'으로 이사(Migration)시킨다. 그리고 콜드 데이터가 차지하고 있던 튼튼하고 쌩쌩한 블록을 비워서, 매일 지웠다 써야 하는 핫 데이터(로그 파일 등)용으로 내어준다. 이 정적 마모 평준화가 지원되어야만 진정한 의미의 엔터프라이즈급 SSD라 할 수 있다.

---

### 심층 동작 원리: Write Amplification (쓰기 증폭)의 딜레마
정적 마모 평준화를 하려면 가만히 있는 콜드 데이터를 억지로 읽어서 다른 곳으로 쓰고, 원래 자리를 지우는 작업이 발생한다. 즉, 사용자는 쓰기 명령을 내리지 않았는데 하드웨어 스스로 수명을 맞추기 위해 '불필요한 쓰기 작업'을 추가로 하는 것이다. 
이를 **쓰기 증폭(Write Amplification, WA)** 현상이라고 한다. 마모를 맞추려다 오히려 디스크 전체 쓰기량이 늘어나 수명을 갉아먹는 역설이 발생하므로, "어느 정도의 지우기 횟수 격차(Delta)가 벌어졌을 때 정적 마모 평준화를 발동시킬 것인가"를 결정하는 임계값(Threshold) 알고리즘이 SSD 컨트롤러 제조사(삼성, 샌디스크 등)의 핵심 노하우(IP)다.

## Ⅲ. 융합 비교 및 다각도 분석

### 매질 별 수명(Endurance) 한계와 대처법

마모 평준화는 메모리 매질의 태생적 물리 한계(P/E Cycle)에 절대적으로 의존한다.

| 플래시 타입 | 저장 방식 (Cell당 비트) | 수명 (P/E Cycle) | 웨어 레벨링 요구도 | 사용처 |
|:---|:---|:---|:---|:---|
| **SLC (Single)** | 1 Bit (상태 2개) | 약 50,000 ~ 100,000회 | 상대적으로 낮음 | 과거 엔터프라이즈 캐시, 군사용 |
| **MLC (Multi)** | 2 Bit (상태 4개) | 약 3,000 ~ 10,000회 | 매우 높음 | 고성능 서버 부트 드라이브 |
| **TLC (Triple)** | 3 Bit (상태 8개) | 약 1,000 ~ 3,000회 | **필수적 (극도로 정교한 Static WL)** | 일반 PC, 모바일, 서버 데이터 |
| **QLC (Quad)** | 4 Bit (상태 16개) | 약 100 ~ 1,000회 미만 | **한계 봉착 (거대한 오버프로비저닝 필수)**| 아카이브(저장 전용) 스토리지 |

### 과목 융합 관점
- **파일 시스템 (TRIM 명령어)**: 과거 SSD는 파일 시스템(OS)이 특정 파일을 지워도(Delete), 논리적 매핑만 끊을 뿐 물리적으로는 데이터를 지우지 않았다. FTL 입장에서는 이 데이터가 유효한지 몰라 마모 평준화를 할 때 계속 같이 이사(Migration)시키며 쓰기 증폭을 유발했다. 이를 해결하기 위해 OS가 파일을 지울 때 "이 블록은 진짜 쓰레기니까 마모 평준화할 때 빼고 맘대로 지워도 돼!"라고 알려주는 하드웨어-소프트웨어 인터페이스가 바로 **TRIM 명령어**다.
- **분산 스토리지 아키텍처 (ZNS)**: 플래시 수명이 점점 짧아지자(QLC 시대), 컨트롤러가 웨어 레벨링을 하느라 고생하는 대신 클라우드 시스템 전체를 관장하는 OS(예: RocksDB)가 플래시의 물리적 블록 크기(Zone)를 인지하고, 아예 덮어쓰기를 하지 않고 **끝에서부터 순차적으로만 기록하는 ZNS(Zoned Namespace)** 아키텍처가 대두되고 있다. 웨어 레벨링의 주체가 하드웨어에서 소프트웨어로 다시 넘어가는 중이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 데이터베이스 서버의 SSD 조기 사망 사태**: 로그(WAL) 기록이 초당 수천 번 발생하는 카프카(Kafka) 또는 MySQL 서버에 일반 소비자용(Consumer) TLC SSD를 장착했다.
   - **문제 발생**: 일반 PC 환경을 가정하고 설계된 펌웨어는 정적 마모 평준화가 공격적이지 않다. 수백 기가바이트의 OS/콜드 데이터가 자리 잡은 공간은 방치되고, 빈 공간 20%에만 미친 듯이 로그가 덮어씌워지면서 6개월 만에 P/E Cycle 1,000회를 다 채우고 SSD가 죽어버렸다(Read-Only 상태로 잠김).
   - **해결책 (Over-Provisioning)**: 엔터프라이즈 환경에서는 디스크 파티션을 잡을 때 전체 용량의 20~30%를 일부러 할당하지 않고 비워둔다. 이를 오버 프로비저닝(OP) 공간이라고 하며, FTL이 이 숨겨진 거대한 여유 공간을 이용해 쓰기 증폭 없이 여유롭게 웨어 레벨링을 수행하게 만들어 SSD 수명을 2~3배 늘려야 한다.
2. **시나리오 — 보안 문서 완전 삭제(Secure Erase)의 어려움**: HDD 시대에는 기밀 문서 파일을 지우고 그 자리에 0을 덮어쓰면(Wipe) 데이터가 완전히 파기되었다.
   - **보안 취약점 (웨어 레벨링의 역효과)**: SSD에서는 특정 파일 위치(LBA 100)에 0을 덮어쓰라고 명령하면, 웨어 레벨링 알고리즘이 기존 물리 블록을 지우지 않고 다른 새 블록(PBA 500)에 0을 적은 뒤 논리 주소만 바꿔버린다. 즉, **원래 기밀문서 데이터는 플래시 칩 어딘가에 물리적으로 고스란히 남아 있게 된다**. 
   - **해결책**: 기밀을 완전히 파기하려면 소프트웨어적인 덮어쓰기가 아니라, SSD 컨트롤러에 ATA/NVMe 표준 명령어인 **Secure Erase**를 전송하여 컨트롤러가 모든 물리 블록(전체 칩)에 동시에 고전압을 가해 물리적으로 태워버리도록(Flush) 해야 한다.

### 도입 체크리스트
- **TBW (TeraBytes Written) 지표 확인**: 스토리지 아키텍트를 설계할 때, 제조사가 보증하는 TBW 또는 DWPD(Drive Writes Per Day, 하루에 디스크 전체를 몇 번 덮어쓸 수 있는가) 스펙을 확인해야 한다. 아무리 웨어 레벨링이 훌륭해도 플래시 메모리는 결국 소모품이므로, 일일 발생하는 로그/DB 쓰기량이 DWPD를 초과한다면 무조건 램디스크(RAMDisk)나 인메모리 버퍼(Redis)로 1차 쓰기를 받아주어야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 마모 평준화 부재 시 | 동적+정적 마모 평준화 적용 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (수명, Endurance)** | 특정 핫 블록에 집중되어 수 주일 내 파괴 | 칩 전체 블록으로 쓰기 분산 (TBW 충족) | SSD 실사용 수명 **수백 배 이상 극적 연장** |
| **정량 (쓰기 증폭, WA)** | WA = 1 (수명 단축) | 콜드 데이터 이사로 인한 추가 쓰기 발생 (WA > 1) | 수명 연장을 위해 컨트롤러의 백그라운드 I/O 부하 감수 |
| **정성 (신뢰성)** | 언제 배드 블록이 터질지 모르는 시한폭탄 | 모든 블록이 균일하게 늙어가며 수명 예측 가능 | S.M.A.R.T 값을 통한 계획적인 하드웨어 교체(Predictable Maint.) 가능 |

### 미래 전망
- **웨어 레벨링의 한계와 PRAM/MRAM의 부상**: 낸드 플래시가 QLC를 넘어 PLC(5비트)로 가면 수명이 수십 회 수준으로 떨어진다. 아무리 웨어 레벨링을 쥐어짜도 물리적 파괴를 막을 수 없는 한계에 달했다. 이를 해결하기 위해 P/E Cycle이 수백만 회에 달하는 PRAM(인텔 옵테인)이나 수명이 거의 무한대인 MRAM 같은 차세대 SCM(Storage Class Memory)이 쓰기 집약적인 환경을 대체할 것이다.
- **머신러닝 기반 펌웨어 예측**: 현재의 웨어 레벨링은 과거의 쓰기 기록(Erase Count)만 보고 움직인다. 미래의 컨트롤러는 칩 내부에 NPU를 달아, 파일이 생성되는 패턴을 AI가 실시간으로 분석하여 "이 파일은 10분 뒤에 지워질 핫 데이터군", "이건 1년간 안 지워질 콜드 데이터군"을 미리 예측해 처음부터 알맞은 수명의 블록에 배치하는 지능형(Predictive) 웨어 레벨링으로 진화하고 있다.

### 참고 표준
- **S.M.A.R.T (Self-Monitoring, Analysis and Reporting Technology)**: SSD 내부의 웨어 레벨링 상태(Erase Count 평준화 지수, 남은 예비 블록 수)를 호스트 OS에 알려주어 관리자가 디스크의 남은 수명(% 비율)을 파악할 수 있게 해주는 산업 표준.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[플래시 메모리 (Flash Memory)](../6_memory_hierarchy_cache/256_flash_memory.md)**: 덮어쓰기(Overwrite)가 불가능하고 반드시 지운 뒤(Erase) 써야 한다는 물리적 특징이 웨어 레벨링이라는 소프트웨어 괴물을 탄생시켰다.
- **[가비지 컬렉션 (Garbage Collection)](../8_io_storage_systems/328_gc.md)**: 무효화된 찌꺼기 파일이 있는 블록을 찾아 모아서 지워주는 작업. 웨어 레벨링은 가비지 컬렉션이 일어날 때 "기왕 지우고 새로 쓰는 김에 수명이 쌩쌩한 쪽으로 옮겨라"라고 지시하며 긴밀히 협력한다.
- **[NVMe (Non-Volatile Memory Express)](./509_nvme.md)**: 초고속으로 데이터를 쏟아붓는 인터페이스. NVMe의 속도를 버티려면 백그라운드에서 엄청난 속도로 웨어 레벨링을 쳐낼 수 있는 강력한 다중 코어 ARM 프로세서가 컨트롤러에 탑재되어야 한다.

## 👶 어린이를 위한 3줄 비유 설명
1. 지우개로 똑같은 자리만 계속 썼다 지웠다 하면, 결국 공책 그 자리가 찢어져서 버리게 되죠?
2. 컴퓨터의 플래시 디스크도 똑같아서, 똑똑한 인공지능(FTL)이 내가 틀린 글씨를 지우지 않고 **일부러 다른 깨끗한 새 페이지에 쓰도록 자리를 옮겨줘요.**
3. 심지어 한 번 쓰고 안 읽는 일기는 다 낡은 페이지에 옮겨놓고, 매일 쓰는 알림장용으로 깨끗한 페이지를 아껴두기까지 해요. 이렇게 해서 공책 전체를 구멍 없이 끝까지 다 쓸 수 있게 만드는 마법이 **마모 평준화**랍니다.
