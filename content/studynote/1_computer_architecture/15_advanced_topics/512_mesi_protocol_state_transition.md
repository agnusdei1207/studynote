# Mesh Protocol State Transition (MESI on Mesh Interconnect)

> 1. **본질**: 메시 프로토콜 상태 전이도 (Mesh Protocol State Transition)는 단일 버스가 아닌 그물망 형태의 메시 인터커넥트(Mesh Interconnect) 환경에서, 분산된 캐시들 간의 일관성을 유지하기 위해 MESI(Modified, Exclusive, Shared, Invalid) 등의 상태가 메시지 전달에 따라 어떻게 변화하는지를 정의한 복합 논리 모델이다.
> 2. **가치**: 중앙 집중식 버스 중재가 없는 분산 환경에서 메시지 전송 지연 및 비순차 도착 가능성을 고려한 안정적인 데이터 동기화를 보장하며, 현대 멀티코어 프로세서의 확장성을 결정짓는 핵심 설계도이다.
> 3. **융합**: NoC (Network-on-Chip) 아키텍처, 디렉터리 기반 일관성 모델, 그리고 라우팅 알고리즘과 융합되어 매니코어(Many-core) 시스템의 데이터 무결성을 유지한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 메시 프로토콜 상태 전이도 (Mesh Protocol State Transition)는 격자 모양의 통신망(Mesh) 위에서 돌아다니는 데이터들의 "신분 변화 규정"이다. 각 데이터는 자신이 '나만 가진 것(Exclusive)'인지, '남과 나눠 쓰는 것(Shared)'인지, '내가 수정한 것(Modified)'인지의 상태를 가지며, 네트워크를 통해 오는 요청 메시지에 따라 이 상태를 실시간으로 바꾼다.

- **💡 비유**: 메시 프로토콜 상태 전이는 "도시 전체의 실시간 공유 자전거 상태 관리"와 같다. 자전거(데이터)가 대여소(캐시)마다 흩어져 있을 때, 사용자가 자전거를 빌리거나 반납하거나 수리하는 행위(메시지)가 발생할 때마다 중앙 서버와 각 대여소의 상태판이 유기적으로 변하는 복잡한 규칙과 같다.

- **등장 배경**: 
  1. **단일 버스의 한계**: 코어가 수십 개가 되면서 하나의 선(Bus)으로는 트래픽을 감당할 수 없어, 바둑판 같은 메시 구조로 변경됨.
  2. **비동기 메시지 전달**: 버스 방식과 달리 메시 방식은 메시지가 도착하는 시간이 제각각이므로, 훨씬 정교한 상태 전이 로직이 필요함.
  3. **분산 제어의 필요성**: 중앙 관리자 없이 각 노드가 주변과 통신하며 스스로 상태를 결정해야 하는 시스템 요구.

- **📢 섹션 요약 비유**: 넓은 그물망 네트워크에서 데이터들이 서로 엉키지 않게 각자의 '신분증(상태)'을 엄격하게 갱신하며 소통하는 질서 유지 시스템입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Local Cache Controller** | 자신의 데이터 상태 관리 | 들어오는 메시지에 따라 MESI 상태 전이 수행 | FSM (Finite State Machine) | 지점 매니저 |
| **Mesh Router** | 메시지 전달 및 경로 최적화 | XY 라우팅 등을 통해 타 노드/디렉터리로 패킷 전달 | Packet Switching | 도로 위 이정표 |
| **Directory Entry** | 전체 노드의 상태 요약 보관 | 특정 블록을 가진 노드 목록과 마스터 상태 기록 | Presence Vector | 중앙 관제탑 장부 |
| **Control Messages** | 상태 전이 유발 신호 | Read-Req, Invalidate, Ack, Write-back 등의 패킷 | Protocol Data Unit | 업무 지시 공문 |
| **Wait State (Transient)** | 중간 전이 상태 관리 | 응답이 오기 전까지의 과도기적 상태(Pending) 처리 | Split-transaction | "확인 중" 푯말 |

### 메시 환경에서의 상태 전이 메커니즘

전형적인 MESI 상태가 메시 인터커넥트 상의 메시지 교환에 따라 어떻게 변화하는지 시각화한다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               메시 인터커넥트 기반 MESI 상태 전이도                 │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │       (Read Miss) / Send Read-Req     (Write) / Send Inv-Req  │
 │     ┌───────────────────────────┐   ┌────────────────────────┐│
 │     │                           ▼   │                        ▼│
 │  ┌───────┐      Read-Ack     ┌───────┐     Write-Ack      ┌───────┐
 │  │Invalid│ ────────────────▶ │ Shared│ ─────────────────▶ │Modified
 │  │  (I)  │ ◀──────────────── │  (S)  │ ◀───────────────── │  (M)  │
 │  └───────┘    Invalidate     └───────┘     Write-back     └───────┘
 │     ▲                                                        │
 │     └────────────────────────────────────────────────────────┘│
 │                   Invalidate (from other Core)                │
 │                                                               │
 │   [Mesh 특이 사항: 과도기 상태(Transient States)]                 │
 │   - I → S_pending: 데이터를 요청하고 기다리는 중                │
 │   - S → M_pending: 무효화 확인(Ack)을 기다리는 중                │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 기본 MESI(Modified, Exclusive, Shared, Invalid) 전이와 유사하지만, 메시 네트워크에서는 메시지 전송 지연이 발생하므로 `S_pending`과 같은 '기다림 상태'가 매우 중요하다. 예를 들어, Core 1이 Shared(S) 상태의 데이터를 Modified(M)로 바꾸려 할 때, 네트워크의 다른 코어들에게 무효화 메시지를 보내고 그들로부터 "지웠다"는 응답(Ack)을 모두 받을 때까지 `M_pending` 상태에 머물러야 한다. 만약 이 기다림 없이 바로 쓰기를 수행하면, 아직 메시지를 못 받은 다른 코어가 옛날 값을 읽어가는 일관성 붕괴가 발생할 수 있다. 따라서 메시 프로토콜 전이도는 물리적 거리(Hop count)에 따른 시간차를 완벽하게 극복하도록 설계된다.

+++

### 심층 동작 원리: 메시지 종류와 상태 변화

1. **GETS (Get Shared)**: 데이터를 읽기 위해 Shared 상태를 요청. 디렉터리가 데이터를 가진 노드에게 보라고 지시함.
2. **GETM (Get Modified)**: 데이터를 쓰기 위해 독점 권한을 요청. 다른 모든 Shared 복사본을 Invalid로 전이시킴.
3. **PUTM (Put Modified)**: 캐시에서 쫓겨나는 Modified 데이터를 메모리에 기록하고 Invalid로 전이.
4. **Recall / Fwd**: 디렉터리가 특정 코어에게 "데이터를 다른 코어에게 넘겨주거나 지워라"고 명령할 때 발생.

- **📢 섹션 요약 비유**: 물건을 빌려올 때(GETS), 내 것으로 만들 때(GETM), 반납할 때(PUTM)의 절차와 그사이에 서류 승인을 기다리는 과정(Pending)을 엄격히 정의한 절차서와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 버스 기반(Snooping) vs 메시 기반(Directory) 전이 로직

| 비교 항목 | 버스 기반 (MESI) | 메시 기반 (MESI on Mesh) |
|:---|:---|:---|
| **통신 매체** | 공유 버스 (Atomic) | 네트워크 패킷 (Non-atomic) |
| **전이 속도** | 즉시 전이 (방송 즉시 확인) | 단계적 전이 (메시지 왕복 필요) |
| **복잡도** | 낮음 (4개 상태면 충분) | 매우 높음 (수십 개의 중간 상태 존재) |
| **교착 상태** | 거의 없음 | 발생 가능 (메시지 엉킴 방지 필수) |
| **확장성** | 낮음 (8코어 한계) | 높음 (수백 코어 확장) |

메시 기반 전이는 메시지 순서가 뒤바뀔 수 있는(Out-of-Order Delivery) 네트워크 특성을 고려하여, 어떤 순서로 메시지가 와도 최종 상태가 올바르게 수렴되도록 보장하는 '프로토콜 무결성'이 핵심이다.

- **📢 섹션 요약 비유**: 한 명씩 말하는 회의실(버스)과 전국 각지에서 편지를 주고받는 우편 시스템(메시)의 소통 규칙 차이입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 매니코어(Many-core) AI 가속기 설계**: 256개의 코어를 격자형으로 배치할 때, 단일 디렉터리에 부하가 몰리는 것을 방지하기 위해 **분산 디렉터리(Distributed Directory)**와 이에 최적화된 상태 전이도를 설계하여 핫스팟(Hotspot)을 제거한다.
2. **시나리오 — 메시지 손실 및 타임아웃 처리**: 네트워크 혼잡으로 일관성 메시지가 유실되어 특정 코어가 무한정 대기(Pending)하는 상황. 기술사는 프로토콜에 **타임아웃 및 재전송(Retransmit)** 로직을 추가하여 시스템의 강인성(Resilience)을 확보한다.

### 도입 체크리스트
- **Deadlock Freedom**: 메시지 순환 의존성으로 인해 전체 시스템이 멈추지 않는가? (Virtual Channels 사용 여부)
- **State Compression**: 수많은 중간 상태를 저장하기 위한 하드웨어 비트 수가 과도하지 않은가?
- **Verification**: 형식 검증(Formal Verification) 도구를 통해 모든 가능한 메시지 조합에서도 일관성이 깨지지 않음을 증명했는가?

- **📢 섹션 요약 비유**: 메시지가 길을 잃거나 늦게 와도 전체 업무(연산)가 꼬이지 않는지, 비상시 대응 매뉴얼(타임아웃)이 있는지 확인하는 것이 필수적입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (Single Bus) | 도입 후 (Mesh Interconnect) | 개선 효과 |
|:---|:---|:---|:---|
| **최대 코어 수** | 8 ~ 16개 | 128 ~ 512개 이상 | **수십 배의 확장성 확보** |
| **네트워크 처리량** | 고정 (병목 지점 존재) | 네트워크 크기에 비례 | **데이터 소통 능력 극대화** |
| **시스템 안정성** | 단일 장애점(SPOF) 존재 | 경로 우회로 장애 극복 가능 | **고가용성 인프라 구축** |

### 미래 전망
앞으로 메시 프로토콜은 **'칩렛(Chiplet)'** 간의 연결을 지원하는 확장된 형태로 진화할 것이며, 특히 메모리 계층이 복잡해짐에 따라 **'CXL 기반의 글로벌 일관성'**을 유지하기 위한 초거대 전이 모델이 등장할 것이다. 또한 기계학습을 이용해 메시지 경로를 최적화하고 상태 전이 시간을 단축하는 '지능형 NoC' 기술이 핵심이 될 전망이다.

- **📢 섹션 요약 비유**: 도시의 교통 체계가 갈수록 정교해지듯, 컴퓨터 내부의 데이터 소통 규칙도 한 치의 오차도 허용하지 않는 완벽한 지능형 시스템으로 발전하고 있습니다.

## 📌 관련 개념 맵
- **MESI 프로토콜**: 메시 인터커넥트 상에서 돌아가는 기본 일관성 규약.
- **NoC (Network-on-Chip)**: 메시 프로토콜의 물리적 무대인 칩 내부 네트워크.
- **디렉터리 캐시 (Directory Cache)**: 메시지 전송의 방향을 결정하는 정보 저장소.
- **데드락 (Deadlock)**: 메시지 전송 시 반드시 피해야 할 설계 오류.
