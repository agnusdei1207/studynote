# 506. TLB 슛다운 (TLB Shootdown)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 멀티코어 환경에서 특정 코어가 페이지 테이블(Page Table)을 수정했을 때, 다른 모든 코어들이 가지고 있는 구형(Stale) TLB 캐시 항목을 일제히 무효화(Invalidate)하라고 강제하는 하드웨어-소프트웨어 동기화 메커니즘이다.
> 2. **가치**: 가상 메모리 시스템의 '일관성(Coherence)'을 유지하여 보안 오류나 프로세스 크래시를 막는 필수 안전장치이지만, 이 과정에서 모든 코어의 실행이 멈추는(Stall) 엄청난 성능 페널티를 동반한다.
> 3. **융합**: 가상화(Hypervisor), 멀티스레딩, 운영체제의 메모리 관리자가 만나는 교차점이며, 이 오버헤드를 줄이기 위해 ASID(Address Space ID) 도입과 같은 아키텍처적 개선이 지속되고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: TLB 슛다운(TLB Shootdown)은 운영체제가 멀티프로세서(SMP) 시스템에서 가상 메모리 맵핑을 변경(예: 메모리 해제, 권한 변경)할 때, 다른 프로세서들의 TLB(Translation Lookaside Buffer)에 남아있는 과거의 가상-물리 주소 변환 캐시를 강제로 지우기(Flush) 위해 발생시키는 프로세스 간 인터럽트(IPI) 절차다.
- **💡 비유**: 회사(OS)가 사무실 하나(물리 메모리)를 창고로 용도 변경했다. 이때 사장님이 각 부서 직원(CPU 코어들)이 가지고 있는 '구형 사내 안내도(TLB)'를 걷어서 파기하지 않으면, 직원이 창고 문을 열고 들어가서 사무 업무를 보려는 끔찍한 사고(페이지 폴트/보안 오류)가 발생한다. 그래서 사장님이 사내 방송(IPI 인터럽트)으로 "모두 하던 일 멈추고 안내도 수정해!"라고 소리쳐 일제히 고치게 만드는 과정이 슛다운이다.
- **등장 배경 및 발전 과정**:
  1. **단일 코어 시대의 평화**: 코어가 하나일 때는 운영체제가 페이지 테이블을 수정하면서 자기 자신의 TLB만 지우면 그만이었다. 아무 문제가 없었다.
  2. **멀티코어와 공유 메모리의 딜레마**: 스레드 A(코어 1)와 스레드 B(코어 2)가 같은 프로세스(메모리 공간)를 공유할 때, 코어 1이 특정 메모리를 해제(Free)해 버리면, 코어 2는 자신의 TLB에 남은 과거 주소를 믿고 접근했다가 시스템 전체를 패닉(커널 크래시)에 빠뜨렸다.
  3. **소프트웨어적 강제 동기화 (IPI 도입)**: 이를 막기 위해 1980년대 후반 Mach OS 등에서 '프로세서 간 인터럽트(IPI)'를 쏘아 다른 코어들의 실행을 강제로 멈추고 TLB를 지우도록 하는 슛다운 기법이 표준화되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **Initiator Core (개시 코어)** | 페이지 테이블을 수정한 원인 제공자 | 다른 코어들에게 IPI(인터럽트)를 전송하고 응답을 기다림 | 사내 방송을 켠 사장님 |
| **Responder Cores (수신 코어)** | IPI를 받고 자신의 TLB를 지워야 하는 코어들 | 하던 연산을 멈추고 ISR 진입 → 자신의 TLB 특정 항목 삭제 | 방송 듣고 하던 일 멈춘 직원들 |
| **IPI (Inter-Processor Interrupt)** | 코어 간 긴급 통신을 위한 하드웨어 인터럽트 | APIC(Advanced Programmable Interrupt Controller)를 통해 발송 | 긴급 사내 비상벨 |
| **Page Table Lock (PTL)** | 슛다운이 진행되는 동안 다른 메모리 변경을 막는 락 | 스핀락(Spinlock) 형태로 구현되어 동시 다발적 슛다운 방지 | 방송 중 아무도 말 못 하게 하기 |

### TLB Shootdown 동작 시퀀스 (동기화 과정)

TLB 슛다운은 본질적으로 엄청나게 무거운 **'글로벌 배리어(Global Barrier)'** 동작이다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │              TLB Shootdown 5단계 실행 시퀀스                  │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [Core 0: 개시 코어]                         [Core 1~N: 수신 코어들] │
  │                                                             │
  │ 1. 페이지 테이블 엔트리(PTE) 수정                             │
  │    (예: munmap 호출로 메모리 해제)                            │
  │            │                                                │
  │ 2. 다른 코어들에게 IPI (인터럽트) 발송 ───▶ (작업 중단 및 인터럽트 수신) │
  │            │                          │                     │
  │ 3. 대기 (Stall - Spinlock)              3. TLB Flush 실행 (개별)│
  │    (다른 코어가 지울 때까지 무한 대기)          (문제의 가상 주소 삭제)  │
  │            │                          │                     │
  │ 4. 대기 중... ◀────────── ACK 반환 ────── (Flush 완료 보고)    │
  │            │                                                │
  │ 5. 모든 ACK 수신 확인 후,                                      │
  │    자신의 작업(OS 커널 로직) 재개.                               │
  │                                                             │
  │ * 치명적 단점: Core 0은 Core 1~N이 작업을 멈추고 TLB를 비울 때까지   │
  │              아무것도 못하고 수천~수만 클럭을 낭비해야 한다 (Jitter 발생).│
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 코어 0에서 어떤 스레드가 `free()`나 `munmap()`을 호출하여 물리 메모리를 반환했다. 코어 0의 운영체제는 즉시 페이지 테이블을 지운다. 그다음 코어 1~N의 APIC를 향해 **IPI 인터럽트**를 발사한다. 코어 1~N은 사용자 애플리케이션을 신나게 돌리다가 갑자기 강제로 커널 모드로 진입(Context Switch 수준의 오버헤드)당하여, 자신의 TLB에 있는 해당 주소를 지운다. 지우고 나서 "다 지웠음!"(ACK) 신호를 보낸다. **이때 코어 0은 모든 코어의 ACK가 도착할 때까지 스핀락을 걸고 꼼짝 않고 기다린다(Stall).** 워크로드가 큰 서버(예: 64코어)일수록 이 대기 시간은 기하급수적으로 늘어나 시스템 전체가 순간적으로 얼어붙는(Freeze) 심각한 병목을 유발한다.

### 심층 동작 원리 (슛다운의 유발 조건)
어떤 상황에서 이 비싼 슛다운이 발동될까?
① **메모리 해제 및 권한 변경**: `munmap()`, `mprotect()` 호출 시.
② **스와핑 (Swapping)**: OS가 메모리가 부족해 특정 페이지를 디스크로 내릴 때, 그 주소가 담긴 모든 TLB를 지워야 한다.
③ **Copy-on-Write (CoW)**: `fork()` 시 부모/자식 프로세스가 메모리를 공유하다가 한쪽이 쓰기를 시도하면, 기존의 '읽기 전용' TLB 맵핑을 무효화하고 쓰기 가능한 새 페이지로 맵핑해야 하므로 슛다운이 발생한다.
④ **NUMA 마이그레이션**: 스레드가 다른 소켓으로 이동할 때, 메모리 페이지도 로컬 노드로 옮기면서 기존 TLB 맵핑을 무효화한다.

## Ⅲ. 융합 비교 및 다각도 분석

### Cache Coherence (캐시 일관성) vs TLB Coherence (슛다운)

둘 다 멀티코어 환경의 일관성 유지 기술이지만 구현 계층이 완전히 다르다.

| 비교 항목 | Cache Coherence (예: MESI) | TLB Coherence (TLB Shootdown) |
|:---|:---|:---|
| **대상** | L1/L2 데이터 및 명령어 캐시 | **가상-물리 주소 변환 캐시 (TLB)** |
| **제어 주체** | **순수 하드웨어 (스누핑/디렉토리)** | **운영체제 커널 (IPI 소프트웨어 방식)** |
| **발생 빈도** | 매 클럭/나노초 단위로 빈번히 발생 | 페이지 맵핑이 바뀔 때만 가끔 발생 (마이크로초) |
| **동기화 지연** | 매우 빠름 (하드웨어 버스 이용) | **매우 느림** (소프트웨어 인터럽트 및 대기 락) |
| **확장성 병목** | 코어 수 증가 시 버스 트래픽 폭증 | 코어 수 증가 시 IPI 대기 지연(Jitter) 극심화 |

### 과목 융합 관점
- **운영체제 (ASID, Address Space ID)**: 과거에는 문맥 교환(Context Switch)이 일어날 때마다 다른 프로세스의 메모리를 훔쳐보는 것을 막기 위해 아예 TLB 전체를 날려버렸다(Full Flush). 이를 막기 위해 현대 아키텍처는 TLB 엔트리 옆에 **ASID (프로세스 고유 ID)**를 함께 저장한다. ASID가 다르면 주소가 같아도 무시하므로, 프로세스가 바뀌어도 TLB를 지울 필요가 없어졌다. (슛다운은 주로 '같은' 프로세스 내의 멀티스레딩 환경에서 발생한다.)
- **가상화 (가상 머신과 EPT)**: 클라우드 환경에서는 게스트 OS의 가상 주소 -> 게스트 물리 주소 -> 호스트 물리 주소로 2번 변환된다(EPT, Extended Page Table). 만약 하이퍼바이저가 호스트 물리 메모리를 마이그레이션하면, 그 메모리를 쓰고 있는 모든 VM의 vCPU를 멈추고 2차원 TLB를 슛다운해야 하므로 오버헤드가 단일 OS보다 훨씬 치명적이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 인메모리 DB(Redis)의 극심한 Jitter 현상 분석**: 128코어 서버에서 Redis를 구동하는데, 평소에는 빠르다가 주기적인 백그라운드 스냅샷(BGSAVE) 작업 시 전체 서버의 응답 속도가 수십 밀리초(ms) 지연되는 마이크로 버스트(Micro-burst)가 발생함.
   - **원인 (TLB 슛다운 폭풍)**: `BGSAVE`는 `fork()`를 사용하여 거대한 Copy-on-Write(CoW)를 유발한다. 메인 프로세스가 메모리를 수정할 때마다 CoW가 터지며 읽기 전용 맵핑을 해제하기 위해 수많은 TLB 슛다운 IPI가 127개의 다른 코어로 쏟아져 들어간다. 이 인터럽트를 처리하느라 다른 코어들의 Redis 메인 로직이 멈춘(Jitter) 것이다.
   - **해결책**: 거대 페이지(Huge Pages, 2MB/1GB)를 적용하여 페이지 테이블의 개수 자체를 줄여 슛다운 빈도를 1/500로 극감시키거나, 앱단에서 `mmap` 기반의 잦은 할당/해제 대신 메모리 풀(Memory Pool) 구조를 사용하여 커널 개입을 차단한다.
2. **시나리오 — 보안 패치(KPTI) 이후의 성능 폭락**: 멜트다운 취약점 방어를 위해 KPTI(Kernel Page Table Isolation) 패치를 적용한 뒤, 데이터베이스 I/O 성능이 30% 폭락했다.
   - **원인**: KPTI는 유저 모드와 커널 모드의 페이지 테이블을 완전히 물리적으로 분리한다. 이로 인해 시스템 콜(System Call)을 할 때마다 유저 TLB 맵핑을 무효화(Flush)하고 커널 TLB를 로드해야 하는 막대한 오버헤드가 추가되었기 때문이다. (현대 칩은 PCID 기능으로 이를 하드웨어적으로 완화함).

### 도입 체크리스트
- **기술적**: `top`이나 `vmstat`에서 CPU 사용률 중 `hi(Hardware Interrupt)`나 `si(Software Interrupt)` 비율이 비정상적으로 높게 튄다면 TLB 슛다운 폭풍을 의심해봐야 한다. 리눅스의 `cat /proc/interrupts | grep TLB` 명령어로 슛다운 횟수(TLB shootdowns)를 직접 모니터링할 수 있다.
- **운영적**: 코어 수가 많은(NUMA) 서버에서 다중 스레드 애플리케이션을 돌릴 때, 잦은 `malloc()`과 `free()`는 독약이다. `tcmalloc`이나 `jemalloc` 같이 스레드별 로컬 캐시를 두어 페이지 반환(munmap) 빈도를 최소화하는 커스텀 메모리 할당자를 반드시 적용해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 슛다운 방치 (단편화된 4KB 페이지) | Huge Page (2MB) 적용 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (IPI 발생 횟수)** | 1GB 해제 시 262,144번의 PTE 수정/슛다운 | 1GB 해제 시 512번의 수정/슛다운 | TLB 슛다운 오버헤드 **99.8% 감소** |
| **정량 (TLB 적중률)** | TLB 미스 증가, 메모리 접근(Page Walk) 지연 | 단 1개의 TLB 엔트리로 2MB 커버 | TLB Hit Ratio 99% 이상 달성 |
| **정성 (시스템 안정성)** | 멀티코어 Jitter(버벅거림) 및 꼬리 지연(Tail Latency) 악화 | 인터럽트 감소로 응답 시간 일정 유지 | P99 Latency(상위 1% 지연 시간) 방어 성공 |

### 미래 전망
- **하드웨어 기반 TLB 일관성 (Hardware TLB Shootdown)**: 소프트웨어 IPI 방식의 한계를 극복하기 위해, 최근 연구에서는 L1 캐시 일관성 프로토콜(MESI)처럼 페이지 테이블 주소 자체를 하드웨어 버스에서 스누핑(Snooping)하여 OS 개입 없이 하드웨어 레벨에서 TLB를 자동 무효화하는 아키텍처(예: AMD의 INVLPGB 명령어 하드웨어 가속)가 확산되고 있다.
- **클라우드 스케일의 병목**: 코어 수가 256개, 512개로 늘어나는 매니코어(Many-core) 서버 시대에, 1개의 코어가 511개의 코어에게 IPI를 쏘고 ACK를 기다리는 O(N) 동기화 방식은 지속 불가능하다. 특정 클러스터나 코어 그룹(Domain) 단위로 슛다운 범위를 제한하는 타겟티드(Targeted) 슛다운 알고리즘이 리눅스 커널의 핵심 과제가 되고 있다.

### 참고 표준
- **x86 아키텍처 `INVLPG` 명령어**: 특정 가상 주소에 대한 TLB 엔트리 단 하나만 선택적으로 지울 수 있게 해주는 인텔의 필수 어셈블리 명령어. (과거에는 무식하게 `MOV CR3, CR3`로 전체 TLB를 날렸다.)

## 📌 관련 개념 맵 (Knowledge Graph)
- **[가상 메모리 (Virtual Memory)](../7_virtual_memory_os_integration/xx_virtual_memory.md)**: 프로세스마다 가짜(Virtual) 주소를 주고 물리 주소로 변환하는 기술. 이 변환표(페이지 테이블)가 바뀌기 때문에 슛다운이 필요하다.
- **[TLB (Translation Lookaside Buffer)](../7_virtual_memory_os_integration/xx_tlb.md)**: 느린 메모리에 있는 페이지 테이블을 매번 읽지 않기 위해, 최근 변환된 주소만 CPU 코어 안에 저장해 두는 초고속 변환 캐시.
- **[스핀락 (Spinlock)](../2_operating_system/4_concurrency_sync/xx_spinlock.md)**: 슛다운 시 개시 코어가 다른 코어들의 완료 응답(ACK)을 기다리며 무한 루프를 돌면서 멍하니 기다리는 커널 동기화 기법. 이 때문에 CPU 점유율은 높은데 실작업은 없는 현상이 발생한다.
- **[캐시 일관성 (Cache Coherence)](./402_cache_coherence.md)**: 데이터 캐시의 일관성을 유지하는 기술. 슛다운은 이 일관성 문제의 '주소 변환(TLB)' 버전이다.

## 👶 어린이를 위한 3줄 비유 설명
1. 학교에서 선생님(OS)이 각 반 반장들(CPU 코어)에게 "도서관 위치는 3층이야!"라고 적힌 쪽지(TLB)를 나눠줬어요.
2. 그런데 방학 때 도서관이 5층으로 이사를 갔어요. 만약 선생님이 반장들이 가진 '옛날 쪽지'를 빼앗지 않으면, 애들이 3층으로 가서 벽을 부수고 난리가 나겠죠? (오류 발생)
3. 그래서 선생님이 교내 방송(IPI 인터럽트)으로 "전교생 동작 그만! 지금부터 옛날 쪽지 찢어버려!"라고 강제로 지시하는 과정이 바로 **TLB 슛다운**이에요. 다들 하던 공부를 멈춰야 하니 시간이 많이 아깝겠죠!
