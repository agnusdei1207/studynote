> **핵심 인사이트**
> 1. **디렉터리 기반 캐시 일관성 (Directory-based Protocol)**은 수십~수백 개의 코어가 존재하는 시스템에서 스누핑(Snooping) 프로토콜의 병목을 해결하기 위해 고안된 방식이다.
> 2. 데이터를 수정할 때 전체 코어에 방송(Broadcast)하는 대신, 메모리 블록마다 '누가 이 데이터를 가지고 있는지'를 기록해 둔 **중앙 장부(Directory)**를 확인하여 해당 코어에게만 1:1(Unicast)로 수정/무효화 메시지를 보낸다.
> 3. 버스 트래픽을 획기적으로 줄여 코어의 확장성(Scalability)을 보장하지만, 디렉터리를 읽고 쓰는 데 추가적인 지연 시간(Latency)과 메모리 공간이 소모된다는 트레이드오프가 있다.



## Ⅰ. 디렉터리 프로토콜의 도입 배경

이전 챕터의 '스누핑 버스 병목'에서 보았듯, 코어가 16개 이상만 되어도 데이터를 쓸 때마다 전체 네트워크에 무효화(Invalidate) 방송을 뿌리는 것은 시스템을 마비시킵니다.

이 문제를 해결하려면 **"필요한 사람에게만 딱 집어서 알려주는"** 시스템이 필요합니다. 그러기 위해서는 '지금 누가 이 데이터를 캐시에 복사해 갔는지'를 어딘가에 기록해 두어야 하는데, 이 기록 장부가 바로 **디렉터리(Directory)**입니다.

> 📢 **섹션 요약 비유**: 전교생 1,000명에게 "철수야, 교무실로 와라!"라고 교내 전체 스피커(스누핑)로 매번 방송하면 너무 시끄럽습니다. 대신 반 배정표(디렉터리 장부)를 보고 철수가 3반에 있다는 걸 확인한 뒤, 3반 교실로만 조용히 전화를 거는(1:1 통신) 스마트한 방식입니다.



## Ⅱ. 디렉터리의 구조와 동작 원리

디렉터리는 보통 메인 메모리 컨트롤러나 L3 공유 캐시에 물리적으로 부착되어 있습니다. 메모리의 모든 데이터 블록은 디렉터리에 다음과 같은 자신의 '상태'를 기록해 둡니다.

* **상태 (State)**: 이 데이터가 아무도 안 씀(Uncached), 여러 명이 나눠 읽는 중(Shared), 누군가 혼자 쓰는 중(Modified)인지 표시합니다.
* **공유자 목록 (Sharers List)**: 현재 이 데이터를 가져간 CPU 코어의 번호를 비트맵(예: `1001`이면 0번, 3번 코어가 가짐)으로 기록합니다.

### 동작 시나리오 (데이터 수정 시)

1. **쓰기 요청**: CPU 0이 `주소 A`의 값을 수정하고 싶어 합니다.
2. **장부 확인**: CPU 0은 무작정 방송을 쏘는 대신, 디렉터리 컨트롤러에 "나 주소 A 수정할래"라고 요청합니다.
3. **타겟 지정**: 디렉터리는 장부를 뒤져봅니다. "주소 A는 현재 CPU 2와 CPU 5가 읽기 전용으로 가지고 있네?"
4. **1:1 무효화 전송**: 디렉터리는 CPU 2와 CPU 5에게만 콕 집어서 무효화 메시지(Invalidate)를 쏩니다. 나머지 코어들은 이 통신을 전혀 알지 못한 채 자기 연산에 집중합니다.
5. **장부 업데이트**: 디렉터리는 주소 A의 주인을 CPU 0(Modified 상태)으로 업데이트합니다.

### 통신 구조 (ASCII 다이어그램)

```text
 [ CPU 0 ] (수정 요청) ──▶ ┌───────────────┐
                          │   Directory   │  (장부 기록: CPU 2, 5가 소유 중)
 [ CPU 1 ]                └──────┬─┬───────┘
                                 │         │
 [ CPU 2 ] ◀──(1:1 무효화 메시지)─┘        │
                                           │
 [ CPU 3 ]                                 │
                                           │
 [ CPU 4 ]                                 │
                                           │
 [ CPU 5 ] ◀──(1:1 무효화 메시지)──┘
```

> 📢 **섹션 요약 비유**: 도서관에서 책(데이터)이 개정판으로 바뀌었을 때, 도서관장(디렉터리)은 방송을 켜는 대신 대출 장부를 펴서 이 책을 빌려 간 2명의 사람에게만 개별적으로 문자를 보내 "그 책 버려라"고 알려줍니다.



## Ⅲ. 디렉터리 방식의 한계 (트레이드오프)

완벽해 보이지만, 디렉터리 방식도 단점이 있습니다.
* **디렉터리 오버헤드**: 메모리 1블록마다 상태와 소유자를 기록해야 하므로, 시스템 전체 메모리의 2~5%가 오직 이 장부를 저장하는 데 낭비됩니다(디렉터리 오버헤드).
* **간접 통신에 의한 지연**: 스누핑은 CPU끼리 버스를 통해 다이렉트로 즉시 통신하지만, 디렉터리 방식은 반드시 중앙 장부(디렉터리)를 한 번 거쳤다가 가야 하므로 통신 속도(Latency)가 상대적으로 느립니다.

따라서 쿼드코어(4개) 스마트폰 칩은 여전히 빠르고 저렴한 스누핑 방식을 선호하며, 64코어 이상의 거대한 서버 프로세서(AMD EPYC, Intel Xeon)는 속도 지연을 감수하더라도 확장이 가능한 디렉터리 방식을 채택합니다.
