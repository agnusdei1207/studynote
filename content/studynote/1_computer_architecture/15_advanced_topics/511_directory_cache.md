# Directory Cache (Directory-based Coherence)

> 1. **본질**: 디렉터리 캐시 (Directory Cache)는 멀티코어 및 다중 프로세서 시스템에서 각 메모리 블록이 어느 캐시에 복사되어 있는지 중앙 집중식 또는 분산식 테이블로 관리함으로써, 캐시 일관성(Cache Coherency) 유지를 위한 방송 트래픽을 최소화하는 기술이다.
> 2. **가치**: 스누핑(Snooping) 방식의 한계인 버스 병목 현상을 해결하여 수백 개 이상의 코어를 가진 대규모 서버 시스템(Scalable Shared Memory)에서 효율적인 데이터 동기화를 가능케 한다.
> 3. **융합**: NUMA (Non-Uniform Memory Access) 아키텍처, 메시 인터커넥트, 그리고 MESI/MOESI 프로토콜과 융합되어 고성능 데이터센터 인프라의 근간을 형성한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 디렉터리 캐시 (Directory Cache)는 "누가 무엇을 가지고 있는지 기록한 중앙 장부"이다. 스누핑 방식이 모든 코어에 대고 소리를 지르는 방송 방식이라면, 디렉터리 방식은 장부를 확인하여 해당 데이터를 가진 코어에게만 1:1로 메시지를 보내는 점대점(Point-to-Point) 방식이다.

- **💡 비유**: 디렉터리 캐시는 "도서관의 도서 대출 목록"과 같다. 어떤 책(데이터)이 필요한 사람이 도서관 전체에 대고 "이 책 가진 사람!"이라고 소리 지르는(스누핑) 대신, 사서에게 가서 "이 책 누가 빌려 갔나요?"라고 물어보고 해당 사람(코어)에게만 연락하는 방식이다.

- **등장 배경**: 
  1. **스누핑 방식의 확장성 한계**: 코어 수가 늘어날수록 공유 버스의 트래픽이 폭증하여 시스템이 마비됨.
  2. **대규모 병렬 처리 요구**: 슈퍼컴퓨터나 대형 서버에서 수십~수백 개의 CPU를 연결해야 할 필요성 대두.
  3. **네트워크-온-칩 (NoC)의 발전**: 단일 버스 대신 복잡한 그물망(Mesh) 구조의 통신망이 도입되면서 방송보다는 개별 통신이 유리해짐.

- **📢 섹션 요약 비유**: 시끄러운 시장통 소통(스누핑) 대신, 체계적인 장부 관리와 개별 연락(디렉터리)을 통해 대규모 조직의 질서를 유지하는 기술입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Directory Table** | 데이터 복사본 위치 추적 | 메모리 블록당 상태 비트와 소유자 비트맵 저장 | Tag Array 확장 | 중앙 대출 장부 |
| **Presence Bits (Sharing Vector)** | 데이터를 가진 코어 식별 | 코어 개수만큼의 비트맵으로 복사본 위치 표시 | Bit-vector / Coarse Vector | 대출자 명단 |
| **State Bits** | 데이터 상태 관리 | Uncached, Shared, Modified 등 상태 표시 | MESI States | 대출 상태 (대여중/수정중) |
| **Home Node** | 특정 주소의 관리 주체 | 각 메모리 주소마다 관리 책임을 맡은 특정 코어/노드 | Hashing / Interleaving | 구역 담당 사서 |
| **Coherence Controller** | 요청 처리 및 메시지 발송 | 읽기/쓰기 요청에 따라 디렉터리 참조 및 메시지 제어 | Protocol Engine | 도서관 관리 시스템 |

### 디렉터리 기반 일관성 유지 프로세스

데이터를 수정(Write)하려 할 때 디렉터리가 어떻게 다른 코어들의 복사본을 무효화하는지 시각화한다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               디렉터리 기반 캐시 일관성 (Directory Flow)            │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │   1. [Core A] "I want to WRITE to Addr X" (Write Request)     │
 │       │                                                       │
 │   2. [Directory (Home Node)] Check Table for Addr X           │
 │       ├─ State: SHARED                                        │
 │       └─ Presence Bits: 0110 (Core B, Core C have copies)     │
 │       │                                                       │
 │   3. [Directory] Send INVALIDATE only to Core B and Core C    │
 │       │ (NOT broadcasting to Core D, E, F...)                 │
 │       │                                                       │
 │   4. [Core B, C] Invalidate copies & Send ACK to Core A       │
 │       │                                                       │
 │   5. [Directory] Update Table: State = MODIFIED, Owner = A    │
 │       │                                                       │
 │   6. [Core A] Proceed with Write                              │
 │                                                               │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 스누핑 방식이었다면 Core A가 쓰기를 시도할 때 시스템의 모든 코어(B, C, D, E, F...)에 무효화 신호를 보냈겠지만, 디렉터리 방식은 장부를 먼저 확인한다. 장부에는 Core B와 C만 데이터를 가지고 있다고 적혀 있으므로, 디렉터리는 오직 B와 C에게만 "데이터를 지우라"는 메시지를 보낸다. 이를 통해 관련 없는 Core D, E, F 등은 불필요한 통신 부하에서 자유로워진다. 코어 수가 1,000개일 때, 단 2개에게만 메시지를 보내는 것은 전체 네트워크 대역폭을 획기적으로 아끼는 결과를 낳는다. 이것이 디렉터리 방식이 대규모 시스템의 표준이 된 이유다.

+++

### 심층 동작 원리: 디렉터리 저장 방식의 최적화

코어 수가 많아지면 디렉터리 테이블 자체의 크기도 문제가 된다. 이를 해결하기 위한 기법들은 다음과 같다.

1. **Full Bit-vector**: 모든 코어에 대해 1비트씩 할당. 정확하지만 코어가 많아지면 디렉터리 메모리가 너무 커짐.
2. **Coarse Vector**: 여러 코어를 하나의 그룹으로 묶어 비트를 할당. 메모리를 아끼지만 불필요한 메시지가 조금 발생함.
3. **Limited Pointers**: 데이터를 가진 코어가 소수(예: 4개 이하)일 것이라 가정하고, 코어 번호(Pointer)만 몇 개 저장. 초과 시 방송으로 전환.
4. **Directory Caching**: 디렉터리 정보 자체도 캐시하여 자주 쓰이는 주소의 일관성 유지 속도를 높임.

- **📢 섹션 요약 비유**: 전교생 이름을 다 적기 힘들면 반별로 대표만 적거나(Coarse), 자주 빌려 가는 학생 명단만 따로 관리(Caching)하는 효율화 전략과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 스누핑 vs 디렉터리 상세 비교

| 비교 항목 | 스누핑 (Snooping) | 디렉터리 (Directory) |
|:---|:---|:---|
| **통신 인프라** | 공유 버스 (Shared Bus) | 메시/스위치 네트워크 (NoC) |
| **대역폭 소모** | 높음 (Broadcasting) | 낮음 (Unicast) |
| **응답 지연 (Latency)** | 낮음 (즉시 감시) | 중간 (디렉터리 참조 오버헤드) |
| **확장성** | 매우 낮음 | 매우 높음 |
| **제어 로직** | 단순함 | 복잡함 (메시지 프로토콜 설계) |

- **📢 섹션 요약 비유**: 확성기로 방송하기(스누핑)는 빠르지만 시끄럽고, 우편으로 연락하기(디렉터리)는 조금 느려도 훨씬 정숙하고 체계적인 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — HPC (High Performance Computing) 노드 설계**: 수천 개의 노드를 연결하는 클러스터 설계 시, 각 노드 내부에서는 스누핑을 쓰고 노드 간에는 디렉터리 방식을 쓰는 **계층적 일관성(Hierarchical Coherence)** 구조를 설계하여 성능과 확장성의 균형을 잡는다.
2. **시나리오 — 클라우드 서버 인스턴스 성능 저하**: 특정 메모리 영역에 대한 경합이 심해 디렉터리 업데이트 트래픽이 몰리는 상황(Directory Contention). 기술사는 데이터를 여러 노드로 분산(Data Striping)하여 특정 홈 노드(Home Node)에 부하가 쏠리지 않도록 튜닝한다.

### 도입 체크리스트
- **Memory Overhead**: 디렉터리 정보를 저장하기 위해 소모되는 메모리 용량이 전체의 10% 이내인가?
- **Network Topology**: 디렉터리 메시지가 지나갈 메시(Mesh) 네트워크의 홉(Hop) 수가 너무 길지는 않은가?
- **Protocol Race Condition**: 비순차 메시지 전달 시 발생할 수 있는 교착 상태(Deadlock)에 대한 하드웨어적 방지책이 있는가?

- **📢 섹션 요약 비유**: 장부를 보관할 공간이 충분한지, 우편배달부(네트워크)가 다니는 길이 너무 꼬여있지는 않은지 확인하는 것이 중요합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (Snooping-only) | 도입 후 (Directory) | 개선 효과 |
|:---|:---|:---|:---|
| **최대 코어 확장성** | 16코어 내외 | 수천 코어 가능 | **확장성 한계 돌파** |
| **네트워크 트래픽** | 100% (방송 위주) | 5% ~ 10% (선택 전송) | **대역폭 효율 10배 이상 향상** |
| **시스템 가용성** | 버스 장애 시 전체 마비 | 부분적 네트워크 장애에 강인 | **안정성(Resilience) 향상** |

### 미래 전망
최근에는 디렉터리 방식과 스누핑의 장점을 결합한 **'토큰 일관성(Token Coherence)'**이나 인공지능이 트래픽을 예측하여 디렉터리 참조를 생략하는 **'예측형 디렉터리'** 기술이 연구되고 있다. 또한 칩렛(Chiplet) 시대로 접어들며 서로 다른 다이(Die) 간의 일관성을 유지하기 위한 초고속 디렉터리 가속기 설계가 핵심 경쟁력이 될 전망이다.

- **📢 섹션 요약 비유**: 아날로그 장부에서 디지털 통합 관리 시스템으로 발전하듯, 컴퓨터의 기억 공유 방식도 더 지능적이고 거대한 네트워크 형태로 진화하고 있습니다.

## 📌 관련 개념 맵
- **캐시 일관성 (Cache Coherency)**: 디렉터리가 해결하려는 근본적인 문제.
- **NUMA (Non-Uniform Memory Access)**: 디렉터리 방식이 주로 적용되는 하드웨어 구조.
- **MESI 프로토콜**: 디렉터리 내부에서 관리하는 개별 데이터의 상태.
- **메시 인터커넥트 (Mesh Interconnect)**: 디렉터리 메시지가 이동하는 물리적 고속도로.

## 👶 어린이를 위한 3줄 비유 설명
1. 디렉터리 캐시는 **'학교 도서관의 대출 장부'**와 같아요.
2. 누가 어떤 책(데이터)을 가져갔는지 장부에 꼼꼼히 적어둬서, 책을 고칠 때 전교생에게 소리 지르지 않고 빌려 간 친구에게만 연락하는 방식이죠.
3. 덕분에 학교가 시끄럽지 않고(트래픽 감소), 아주 많은 학생이 있어도 질서 있게 공부할 수 있답니다!
