# 부채널 공격 - 캐시 타이밍 공격

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPU 캐시 접근 시간 차이를 측정하여 비밀 정보를 유출하는 사이드 채널 공격
> 2. **가치**: 보안 취약점 이해, 타이밍 분석, 완화 기법, 캐시 보안
> 3. **융합**: 스펙터, 멜트다운, Prime+Probe, Flush+Reload와 통합된 타이밍 공격

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

캐시 타이밍 공격은 CPU 캐시 접근 시간 차이를 측정하여 비밀 정보를 유출하는 사이드 채널 공격입니다. 캐시 히트(빠름)와 미스(느림)의 차이를 이용합니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    캐시 타이밍 공격                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              캐시 접근 시간 차이                               │ │
│   │                                                              │ │
│   │   시간 (사이클)                                                 │ │
│   │      ▲                                                        │ │
│   │   300 ─────────────────────┬──────────────────────           │ │
│   │      │                     │                                  │ │
│   │   200 ─────────────────────┼──────────────────────           │ │
│   │      │                     │  L2 Cache Miss                   │ │
│   │   100 ─────────────────────┼──────────────────────           │ │
│   │      │                     │  (느림, ~100-300 cycles)         │ │
│   │    10 ─────────────────────┼──────────────────────           │ │
│   │      │  L1 Cache Hit       │                                  │ │
│   │      │  (빠름, ~4-10 cycles)│                                  │ │
│   │      └─────────────────────┴──────────────────────→ 접근      │ │
│   │                                                              │ │
│   │   히트 vs 미스 시간 차이로 데이터 유추                         │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              공격 시나리오 (AES S-Box 예시)                     │ │
│   │                                                              │ │
│   │   1. 피해자: AES 암호화 수행                                  │ │
│   │      - 키 의존 S-Box 테이블 접근                              │ │
│   │      - 특정 키 바이트 → 특정 캐시 라인 접근                   │ │
│   │                                                              │ │
│   │   2. 공격자: 캐시 타이밍 측정                                 │ │
│   │      - 같은 캐시 라인 접근 시간 측정                          │ │
│   │      - 빠름(히트) → 피해자가 접근함 → 키 유추                 │ │
│   │      - 느림(미스) → 피해자가 접근 안 함                       │ │
│   │                                                              │ │
│   │   3. 결과: 키 바이트별 후보군 축소 → 키 복구                  │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> **해설**: 캐시 히트(4-10 cycles)와 미스(100-300+ cycles) 차이로 데이터를 유추합니다.

**💡 비유**: 캐시 타이밍 공격은 사무실 서랍을 확인하는 것과 같습니다. 누가 열었는지 흔적으로 압니다.

**📢 섹션 요약 비유**: 캐시 타이밍은 서랍 흔적 같아요. 누가 썼는지 알아요!

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **캐시 히트** | 데이터 존재 | 빠른 접근 | 서랍 열림 |
| **캐시 미스** | 데이터 없음 | 느린 접근 | 서랍 비어있음 |
| **타이밍 측정** | 시간 측정 | RDTSC 명령어 | 스톱워치 |
| **데이터 유추** | 정보 추론 | 통계 분석 | 추리 |

**주요 캐시 타이밍 공격 기법**

| 기법 | 원리 | 특징 |
|:---|:---|:---|
| **Prime+Probe** | 공격자가 캐시 채움 → 피해자 접근 → 공격자 재접근 | 공유 불필요 |
| **Flush+Reload** | 공유 메모리 → 플러시 → 피해자 접근 → 리로드 | 공유 필요 |
| **Evict+Time** | 캐시에서 제거 → 시간 측정 | 간단 |

**핵심 알고리즘**: 캐시 상태 조작 → 타이밍 측정 → 통계 분석 → 비밀 유출

**📢 섹션 요약 비유**: Prime+Probe는 내 캐시, Flush+Reload는 공유 캐시!

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**기술 비교: Prime+Probe vs Flush+Reload**

| 비교 항목 | Prime+Probe | Flush+Reload |
|:---|:---:|:---:|
| **공유 메모리** | 불필요 | 필요 |
| **정밀도** | 중간 | 높음 |
| **노이즈** | 높음 | 낮음 |
| **복잡도** | 중간 | 낮음 |

**📢 섹션 요약 비유**: Flush+Reload가 더 정밀해요!

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1: 암호화 라이브러리** → Constant-time 구현
**시나리오 2: 클라우드** → 캐시 파티셔닝, 하이퍼바이저 격리
**시나리오 3: 브라우저** → 사이트 격리, SharedArrayBuffer 제한

**도입 체크리스트**: Constant-time 코드, 캐시 플러시, 노이즈 추가

**📢 섹션 요약 비유**: 시간을 일정하게 하면 안전해요!
