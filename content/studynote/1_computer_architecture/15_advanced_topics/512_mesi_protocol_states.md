> **핵심 인사이트**
> 1. **MESI 프로토콜**은 멀티코어 환경에서 동일한 데이터가 여러 캐시에 존재할 때 발생할 수 있는 충돌을 막기 위해 고안된 '캐시 블록의 4가지 상태 규칙'이다.
> 2. 네 가지 상태는 **M(Modified, 수정됨), E(Exclusive, 독점), S(Shared, 공유됨), I(Invalid, 무효)**를 뜻한다.
> 3. CPU가 데이터를 읽거나 쓸 때, 버스를 통해 다른 CPU의 상태를 확인하고 자신의 캐시 상태를 끊임없이 변화(Transition)시키며 시스템 전체의 일관성을 강제한다.



## Ⅰ. MESI의 4가지 상태 (States) 정의

어떤 CPU의 캐시에 `데이터 A`가 들어있다면, 그 데이터 블록은 반드시 다음 4가지 상태 중 하나에 속해야 합니다.

1. **M (Modified, 수정됨)**
   * **나 혼자만** 이 데이터를 가지고 있으며, 메인 메모리의 값과 다르게 내가 **새롭게 수정했다(Dirty)**는 뜻입니다. 다른 CPU가 이 데이터를 읽으려 하면 메인 메모리가 아니라 내가 최신 값을 넘겨주어야 합니다.
2. **E (Exclusive, 독점)**
   * **나 혼자만** 이 데이터를 가지고 있으며, 그 값은 메인 메모리의 값과 **똑같다(Clean)**는 뜻입니다. 나 혼자 가지고 있으므로, 만약 내가 이 값을 수정하고 싶으면 남들 눈치 볼 것 없이 즉시 **M** 상태로 바꾸고 수정하면 됩니다.
3. **S (Shared, 공유됨)**
   * 이 데이터는 메인 메모리의 값과 똑같지만(Clean), **다른 CPU의 캐시에도 복사되어 있다**는 뜻입니다. 내가 이 데이터를 수정하고 싶다면, 반드시 버스에 방송(Invalidate)을 해서 남들이 가진 복사본을 지워버려야 합니다.
4. **I (Invalid, 무효)**
   * 이 캐시 라인에는 데이터가 없거나, 남이 수정하는 바람에 내 데이터가 쓰레기(구형)가 되어버려 **더 이상 읽을 수 없는 빈 방**이라는 뜻입니다.

> 📢 **섹션 요약 비유**: 하나의 문서를 가지고 작업할 때, 
> * **M**: "이 문서는 나 혼자 갖고 있고, 내가 빨간 펜으로 수정했어." 
> * **E**: "이 문서는 나 혼자 갖고 있는데, 아직 한 글자도 안 건드렸어."
> * **S**: "이 문서는 원본 그대로인데, 옆 부서 사람들도 복사본을 하나씩 들고 있어."
> * **I**: "내 문서 복사본은 구형이라 파쇄기에 버렸어. 없는 셈 쳐."



## Ⅱ. 상태 전이 (State Transition) 시나리오

CPU가 데이터를 읽거나 쓸 때, 이 4가지 상태가 어떻게 변하는지(전이도)가 MESI의 핵심입니다.

### 1. 첫 읽기 (Read Miss)
* **상황**: CPU 1이 주소 A를 처음 읽어옵니다. (상태 I $\rightarrow$ E)
* **동작**: 버스를 엿들어보니 아무도 주소 A를 안 가지고 있습니다. CPU 1은 데이터를 캐시에 넣고 **E(독점)** 상태를 부여합니다. "이건 나만 가졌군. 나중에 맘대로 고쳐야지."

### 2. 다른 CPU의 읽기 (Read by Another)
* **상황**: CPU 2도 주소 A를 읽고 싶어 합니다. (CPU 1의 상태 E $\rightarrow$ S)
* **동작**: CPU 2가 버스에 "A 읽을게!" 방송합니다. CPU 1은 "어, 나 그거 독점(E) 중인데. 너도 읽어? 알았어, 같이 쓰자"라며 자신의 상태를 **S(공유)**로 내립니다. CPU 2도 **S(공유)** 상태로 가져갑니다.

### 3. 공유 중인 데이터 수정 (Write Hit on S)
* **상황**: CPU 1이 S 상태인 A를 수정(Write)하려 합니다. (상태 S $\rightarrow$ M)
* **동작**: CPU 1은 버스에 무효화(Invalidate) 신호를 쏩니다. "나 이거 고친다! 너희들 거 다 버려!" CPU 2는 자신의 S 상태를 **I(무효)**로 바꿔버립니다. 혼자 남은 CPU 1은 안전하게 값을 수정하고 상태를 **M(수정됨)**으로 바꿉니다.

### 4. 남이 수정 중인 데이터 읽기 (Read Hit on M)
* **상황**: CPU 2가, CPU 1이 M 상태로 들고 있는 최신 데이터를 읽고 싶어 합니다.
* **동작**: CPU 1은 메모리보다 빨리 개입(Intervention)하여 최신 데이터를 버스에 쏴줍니다(동시에 메모리에도 업데이트). 그리고 혼자 쓰던 걸 나눴으니 CPU 1의 상태는 M $\rightarrow$ **S**로, CPU 2도 **S** 상태가 됩니다.

### 핵심 전이 요약 다이어그램 (ASCII)

```text
             (버리고 나감)
       ┌──────────────────────────┐
       │                       ▼
    [ E 독점 ]  ◀(혼자 읽음)─  [ I 무효 ]
       │  ▲                    │  ▲
(남이 읽음)(혼자 남음) (수정해서 남들 다 죽임)
       ▼  │                    │  │
    [ S 공유 ]  ──────(수정)───┼──┘
       ▲  │                       │
 (수정된 걸 남이 읽어감)          │
       │  ▼                    ▼
    [ M 수정됨 ] ◀──(E에서 바로 수정)
```

> 📢 **섹션 요약 비유**: 문서 하나를 두고 눈치 게임을 하는 것입니다. 나 혼자 볼 때(E)는 맘대로 고치고(M), 다른 사람이 와서 같이 보면(S) 고칠 때 남들에게 알려서 폐기시키며(I), 남이 고쳐서 낡은 문서가 되면 찢어버립니다(I).



## Ⅲ. MESI 프로토콜의 변형 (MOESI, MESIF)

MESI는 훌륭하지만, 캐시끼리 데이터를 주고받을 때 메인 메모리까지 꼭 거쳐야 하는(Write-back) 등 비효율적인 구석이 있어 현대 CPU들은 이를 살짝 변형해 사용합니다.

* **MOESI (AMD 선호)**: 'O (Owned, 소유)' 상태를 추가. 여러 명이 공유(S)할 때 "내가 이 그룹의 소유자(대표)야. 내가 최신값 들고 있으니 메모리 안 거치고 내가 뿌려줄게"라고 나서는 기능.
* **MESIF (Intel 선호)**: 'F (Forward, 전달자)' 상태를 추가. 10명이 공유할 때 똑같은 데이터를 다 같이 버스에 쏘면 혼란스러우니, 가장 최근에 데이터를 가져간 한 놈(F)만 대표로 다른 놈에게 데이터를 전달해주도록 하는 기능.
