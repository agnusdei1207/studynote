> **핵심 인사이트**
> 1. 앞 장의 캐시 타이밍 공격 중 하나인 **Prime+Probe**는 해커와 피해자가 메모리 주소를 아예 공유하지 않는(격리된) **클라우드(AWS, Azure)의 서로 다른 VM 환경**에서도 작동하는 무서운 기술이다.
> 2. 해커는 CPU의 공용 **L3 캐시(LLC)** 전체를 자기 데이터로 꽉꽉 채워 넣는다 (Prime 단계).
> 3. 피해자가 연산을 하면서 L3 캐시의 특정 방을 비우고 자기 데이터를 넣으면, 해커가 다시 전체 방을 확인(Probe)하여 **"피해자가 몇 번 방을 썼는지(캐시 미스가 나는 위치)"**를 알아내어 비밀 키를 유추한다.



## Ⅰ. L3 캐시(공용 공간)의 맹점

클라우드 서버에 해커(VM 1)와 은행(VM 2)이 같이 입주해 있습니다. 
당연히 VM 1은 VM 2의 메모리에 접근할 수 없습니다 (페이지 테이블 격리).

하지만, 서버의 메인 CPU에 박혀있는 **거대한 L3 캐시는 모든 코어와 모든 VM이 짬뽕으로 같이 씁니다(Shared Cache).** 이 공유 공간(L3 캐시)이 해커의 공격 통로가 됩니다.

> 📢 **섹션 요약 비유**: 해커와 은행은 서로 다른 집(VM)에 살지만, 아파트 1층의 공용 무인 택배함(L3 캐시)은 같이 씁니다. 해커는 이 공용 택배함의 변화를 훔쳐보며 은행의 비밀을 캡니다.



## Ⅱ. Prime+Probe의 3단계 동작

해커는 피해자의 암호화 연산이 시작되기 직전을 노립니다.

1. **Prime (미끼 깔기 / 채우기)**
   * 해커는 L3 캐시와 똑같은 크기(예: 10MB)의 무의미한 배열을 자신의 메모리에 만들고, 이를 무식하게 다 읽어들입니다.
   * 이렇게 하면 **L3 캐시의 모든 방(Cache Set)이 해커의 쓰레기 데이터로 100% 꽉 차게 됩니다.**
2. **Wait (피해자의 실행 대기)**
   * 은행(피해자)이 암호화 알고리즘(AES 등)을 실행합니다. 
   * 은행은 계산을 위해 자기 데이터를 캐시에 올려야 합니다. L3 캐시가 꽉 찼으니, 은행은 **해커의 쓰레기 데이터 중 일부(예: 3번 방)를 발로 걷어차고(Evict) 그 자리에 자기 데이터를 넣습니다.**
3. **Probe (찔러보기 / 검사)**
   * 은행의 연산이 끝난 직후, 해커는 아까 자기가 채워 넣은 배열을 다시 처음부터 끝까지 읽으며 스톱워치로 시간을 잽니다.
   * 대부분의 배열은 1ns(캐시 히트) 만에 읽힙니다. 그런데 아까 은행이 걷어찬 **3번 방**을 읽으려니 100ns(캐시 미스)가 걸립니다!
   * 해커: "아하! 은행이 방금 3번 방(특정 메모리 패턴)을 건드렸구나!" (비밀 키 유추 성공)

### 공격 다이어그램 (ASCII)

```text
 1. Prime (해커가 다 채움)        2. Victim 실행 (은행)           3. Probe (해커가 검사)
 ┌───────── L3 캐시 ─────────┐   ┌───────── L3 캐시 ─────────┐   ┌───────── L3 캐시 ──────────────────────────────────────┐
 │ 해커데이터 │ 해커데이터 │ 해커데이터 │   │ 해커데이터 │ 은행비밀 │ 해커데이터 │   │ 해커데이터 │ ◀(Miss!) │ 해커데이터 │
 └─────────┴─────────┴─────────┘   └─────────┴─────────┴─────────┘   └─────────┴─────────┴────────────────────────────────┘
   (전부 1ns 히트 예상)                (해커 데이터를 밀어냄)               시간 재보니 여기만 100ns 걸림!
```

> 📢 **섹션 요약 비유**: 해커가 공용 택배함 100칸에 자기 상자를 다 채워(Prime) 놓습니다. 은행 직원이 서류를 받기 위해 해커의 상자 1개를 빼서 버리고 자기 서류를 3번 칸에 넣습니다. 다음날 해커가 택배함을 쭉 열어보며(Probe) "어? 3번 칸에 내 상자가 사라졌네? 은행이 3번 칸을 썼구나!"라고 알아맞히는 마술입니다.



## Ⅲ. 방어의 어려움과 하드웨어 파티셔닝

Prime+Probe는 운영체제(OS)가 메모리를 공유하지 않아도(페이지 테이블 무관) 하드웨어 캐시 구조만으로 뚫어버리기 때문에 매우 무섭습니다.

클라우드 업체가 이를 막으려면 **인텔 CAT (Cache Allocation Technology)** 같은 하드웨어 파티셔닝 기술을 켜야 합니다.
"VM 1번(해커)은 L3 캐시의 1번~5번 방만 쓰고, VM 2번(은행)은 6번~10번 방만 써라!"라고 **하드웨어적으로 캐시 공간을 아예 칼같이 찢어버려(분리)** 해커가 은행의 방을 Prime/Probe 하지 못하게 차단해야 합니다.
