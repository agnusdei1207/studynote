# 520. PIM (Processing-In-Memory)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 연산을 담당하는 프로세서(CPU/GPU)와 데이터를 저장하는 메모리(DRAM)가 분리되어 있던 기존 구조를 파괴하고, **메모리 반도체 내부(또는 바로 옆)에 연산기(ALU)를 직접 집적**한 지능형 반도체 아키텍처다.
> 2. **가치**: 데이터가 프로세서로 이동할 때 발생하는 막대한 전력 소모와 지연 시간(폰 노이만 병목)을 원천 차단하여, 방대한 데이터를 다루는 거대 언어 모델(LLM)과 생성형 AI의 '메모리 월(Memory Wall)'을 허무는 궁극의 해결책이다.
> 3. **융합**: HBM 기반의 PIM(삼성 HBM-PIM, SK하이닉스 AiM), LPDDR 기반의 모바일 PIM, 그리고 차세대 CXL 인터페이스와 결합한 CXL-PIM 등 다양한 형태로 진화하며 메모리 제조사가 시스템 반도체의 영역을 넘보는 기폭제가 되고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: PIM(Processing-In-Memory, 또는 지능형 메모리 반도체)은 저장 기능만 하던 메모리 칩 내부에 연산 코어(Processor)를 심어, 데이터가 저장된 바로 그 위치에서 연산을 수행한 뒤 최종 '결과값'만 CPU/GPU로 보내는 새로운 패러다임의 반도체 기술이다.
- **💡 비유**: 옛날에는 도서관(메모리)에 있는 책 100권을 수레에 싣고 학자의 연구실(CPU)까지 낑낑대며 끌고 와서(데이터 이동) 중요한 문장 1개를 찾아냈다. 너무 힘들고 오래 걸린다. PIM은 **도서관 각 층마다 똑똑한 사서(연산기)**를 배치한 것이다. 학자가 "1층부터 10층까지 책에서 '사과'라는 단어 개수 좀 세워줘"라고 방송만 하면, 사서들이 각 층에서 알아서 찾은 뒤 "총 5개입니다"라는 결과 쪽지 1장만 학자에게 보내준다. 
- **등장 배경 및 발전 과정**:
  1. **데이터 이동 에너지의 역설**: 반도체 학계의 연구 결과, 64비트 실수 연산(MAC)을 할 때 쓰이는 에너지(0.1pJ)보다 그 데이터를 외부 DRAM에서 CPU로 가져오는 데 쓰이는 에너지(약 10pJ)가 무려 100배 이상 크다는 것이 밝혀졌다. 즉, 전기의 99%가 계산이 아니라 '배달'에 쓰이고 있었다.
  2. **1990년대의 실패**: 1990년대 후반 EXECUBE, IRAM 등 PIM 개념이 이미 나왔으나, 메모리 공정(DRAM)과 시스템 반도체 공정(로직)이 너무 달라서 하나로 합치면 수율이 바닥을 기고 단가가 비싸 상용화에 처참히 실패했다.
  3. **초거대 AI 시대의 부활 (2020년대)**: AI 모델 파라미터가 수천억 개 단위로 폭증하면서 '연산 속도'보다 '메모리 대역폭'이 전체 성능을 결정짓는(Memory-bound) 시대가 왔다. 더 이상 구리선을 늘릴 수 없자, 삼성전자와 SK하이닉스가 HBM과 TSV 기술을 무기로 PIM을 세계 최초로 상용화하며 부활시켰다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (HBM-PIM 기준)

| 요소명 | 역할 | 내부 동작 / 특징 | 비유 |
|:---|:---|:---|:---|
| **PCU (Programmable Computing Unit)** | 메모리 뱅크 안에 박혀있는 초소형 연산기 | 16비트 부동소수점(FP16) 행렬 곱셈/덧셈(MAC) 전담 수행 | 도서관 층마다 배치된 사서 |
| **메모리 뱅크 (Memory Bank)** | 실제 데이터가 저장되는 셀 어레이 | 일반 DRAM 뱅크와 동일하지만 PCU와 직접(Local) 연결됨 | 도서관의 책장들 |
| **Global Buffer** | 외부 프로세서나 다른 뱅크의 데이터를 공유 | 활성화 함수, 가중치 공유 등을 위한 임시 저장소 | 사서들이 모이는 중앙 데스크 |
| **명령어 디코더 (Instruction Decoder)** | 호스트(CPU)가 보낸 PIM 전용 명령어를 해석 | 일반 Read/Write 명령어가 아닌 PIM MAC 명령어를 수신 | 학자의 방송 지시를 번역 |
| **TSV (실리콘 관통 전극)** | 층간 데이터를 이동하거나 최종 결과를 호스트로 보냄 | 수직 데이터 통로로 3D 적층 시 대역폭 보장 | 층간 엘리베이터 |

### 폰 노이만 병목 vs PIM 아키텍처 데이터 흐름 비교

PIM의 본질은 무거운 데이터를 이동시키지 않고 가벼운 명령어를 이동시키는 것이다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │              기존 폰 노이만 구조 vs PIM 구조 데이터 흐름            │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 기존 구조 (Von Neumann Architecture)                      │
  │     - "데이터를 통째로 연산기로 가져온다"                        │
  │                                                             │
  │   [ 호스트 CPU / GPU ]                     [ 일반 DRAM ]      │
  │   ┌────────────────┐ ◀──(100GB 데이터)── ┌──────────────┐     │
  │   │   ALU / MAC    │ ──(1MB 결과값)───▶  │ Bank 0, 1, 2 │     │
  │   └────────────────┘                      └──────────────┘     │
  │   * 문제: 버스(Bus)가 터져나가고 배달(I/O) 전력을 엄청나게 소모함.     │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] PIM (Processing-in-Memory) 구조                         │
  │     - "연산 명령을 데이터가 있는 곳으로 보낸다"                    │
  │                                                             │
  │   [ 호스트 CPU / GPU ]                     [ PIM 메모리 칩 ]     │
  │   ┌────────────────┐  ──("행렬곱 해라!")─▶ ┌──────────────┐     │
  │   │   컨트롤러      │                      │ ┌──────────┐ │     │
  │   └────────────────┘  ◀──(1MB 결과값)─── │ │Bank0+ALU │ │     │
  │                                        │ ├──────────┤ │     │
  │                                        │ │Bank1+ALU │ │     │
  │                                        │ └──────────┘ │     │
  │                                        └──────────────┘     │
  │   * 장점: 100GB의 무거운 데이터는 PIM 칩 내부에서 1mm만 이동하여 계산됨.│
  │           CPU와 메모리 사이의 버스는 텅텅 비고 전력 소모는 1/10로 줆.│
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 레거시 구조에서 CPU가 1,000억 개의 숫자를 다 더하려면, 메모리에 1,000억 번의 Read 요청을 보내 그 데이터를 모조리 CPU 내부로 끌고 와서 더해야 한다. 이 과정에서 발생하는 PCIe 버스 지연과 전력 소모(발열)가 칩 성능의 한계(Memory Wall)를 만든다.
하단의 **PIM 구조**에서는 CPU가 "너희들끼리 다 더해서 최종 값만 나한테 줘"라는 아주 짧은 명령(Command) 1개만 메모리로 보낸다. 그러면 메모리 칩 내부에 있는 수십 개의 ALU(PCU)들이 각자 자기가 맡은 메모리 구역(Bank)의 숫자를 더한 뒤, 최종 답 1개만 버스를 태워 CPU로 돌려보낸다. 외부로 나가는 데이터 이동(Data Movement)이 완전히 삭제되는 마법이다.

---

### 심층 제조 공정: PIM의 3가지 진화 단계 (PNM → PIM → CIM)
① **PNM (Processing Near Memory)**: 연산기와 메모리를 완전히 합치지 못하고, 인터포저 위에서 연산기(GPU 칩렛)와 메모리(HBM)를 아주 가깝게 붙인 상태. (현재의 H100 같은 2.5D 패키징)
② **PIM (Processing In Memory)**: 메모리 다이(Die) 안쪽 비어있는 가장자리(Periphery)나 뱅크 사이에 조그만 연산 회로를 집적한 상태. (현재의 삼성 HBM-PIM, SK하이닉스 AiM)
③ **CIM (Computing In Memory)**: 궁극의 종착지. 아예 메모리 셀(커패시터, 멤리스터) 자체가 전기를 머금으며 데이터를 저장함과 동시에 아날로그 방식으로 행렬 곱셈을 0초 만에 수행해 버리는 뉴로모픽(Neuromorphic) 단계.

## Ⅲ. 융합 비교 및 다각도 분석

### PIM vs 일반 AI 가속기 (NPU)

| 비교 항목 | 전용 AI 가속기 (NPU, TPU) | PIM (HBM-PIM 등) |
|:---|:---|:---|
| **설계 주체** | 시스템 반도체 기업 (Google, Tesla) | **메모리 반도체 기업 (삼성, SK하이닉스)** |
| **코어(연산기) 파워** | 매우 막강함 (거대한 Systolic Array) | 작고 단순함 (간단한 MAC 회로) |
| **메모리 대역폭** | 여전히 외부 HBM/DRAM 버스 병목 존재 | **내부 대역폭 무한대 (Bank-level 병렬성)** |
| **전력 소모** | 연산기 자체 전력 소모 큼 (수백 W) | **데이터 이동이 없어 전성비 압도적** |
| **호환성** | 자체 SDK/컴파일러 툴체인 사용 | 기존 DRAM 컨트롤러 및 JEDEC 표준과 충돌 이슈 존재 |

### 과목 융합 관점
- **컴파일러와 소프트웨어 생태계**: PIM의 하드웨어는 완벽하지만 상용화가 더딘 이유는 '소프트웨어 생태계' 때문이다. 개발자가 짠 PyTorch 코드를 분석해서, CPU가 할 연산과 PIM 메모리가 할 연산을 분리(Offloading)하여 쪼개주는 아주 영리한 **PIM 전용 컴파일러 및 런타임 OS 스케줄러**가 필요하다. 엔비디아의 CUDA 생태계를 깨고 이를 독자 구축하는 것이 PIM 제조사들의 최대 과제다.
- **아키텍처 (CXL-PIM)**: 최신 CXL 기술은 메모리를 무한대로 꽂을 수 있게 해준다. 여기에 PIM을 결합하면 끔찍한(거대한) 괴물이 탄생한다. CPU가 수백 개의 CXL 슬롯에 분산된 CXL-PIM 모듈들에게 쿼리 하나만 던지면, 수백 개의 PIM 모듈이 1페타바이트(PB)의 데이터를 각자 동시에 뒤져서 결과만 CPU로 올려주는 궁극의 **스토리지/메모리 레벨 병렬 처리(In-Storage Computing)**가 실현된다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 거대 언어 모델(LLM) 텍스트 생성(디코더) 과정의 병목**: ChatGPT가 단어 하나를 뱉어낼 때마다(Auto-regressive Generation), 수백 기가바이트의 모델 가중치(Weight) 전체를 메모리에서 GPU로 한 번씩 다 끌어와야 한다. 연산은 덧셈 몇 번뿐인데(FLOPs 낮음), 데이터만 오가느라 H100 GPU조차 쩔쩔맨다(Memory-Bound).
   - **아키텍처 혁신 (PIM 도입)**: 삼성의 HBM-PIM을 GPU 옆에 단다. GPU는 가중치를 자기한테 가져오라고 하지 않고, "지금 사용자가 입력한 단어(토큰) 값(Vector)"만 HBM-PIM으로 휙 던져준다. HBM 내부의 PCU(연산기)들이 각자 자기가 가진 가중치와 토큰을 곱하고 더해서(Vector-Matrix Multiplication) 최종 값만 GPU로 돌려준다. 데이터 이동이 1/1000로 줄어들어 텍스트 생성 속도가 3배 이상 빨라진다.
2. **시나리오 — 모바일 기기의 온디바이스 AI 한계**: 스마트폰에서 실시간 번역을 하려고 NPU를 100% 가동했더니, LPDDR 메모리에서 데이터를 긁어오느라 스마트폰이 손난로가 되고 배터리가 광탈함.
   - **해결책 (LPDDR-PIM)**: 모바일 D램 안에 간단한 PIM 연산기를 넣은 LPDDR-PIM(또는 AiM)을 채택한다. 스마트폰 AP 내부의 NPU는 무거운 모델은 메모리(PIM)한테 계산을 맡기고 자신은 잠을 잔다. 데이터가 칩 밖(AP-DRAM 간 보드 배선)으로 나오지 않아 발열이 사라지고 배터리 타임이 비약적으로 늘어난다.

### 도입 체크리스트
- **호스트 프로세서 호환성**: PIM 메모리를 서버에 그냥 꽂는다고 동작하지 않는다. PIM 안에 있는 연산기(PCU)에게 명령을 내리려면, 호스트 CPU나 GPU의 메모리 컨트롤러가 일반적인 읽기/쓰기 신호(Read/Write)가 아니라 'PIM 연산 명령어'를 보낼 수 있도록 수정된 커스텀 메모리 컨트롤러(또는 소프트웨어 에뮬레이션 층)를 탑재하고 있어야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 레거시 메모리 (폰 노이만) | PIM 아키텍처 도입 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (AI 추론 성능)** | 외부 버스 대역폭(예: 2TB/s)의 한계에 막힘 | 내부 뱅크 병렬 연산으로 유효 대역폭 폭발 | 음성/언어 모델 추론 성능 **2.5배 ~ 3배 향상** |
| **정량 (에너지 효율)** | 데이터 이동(I/O)으로 전체 전력의 70% 소모 | 이동 최소화로 I/O 전력 소멸 | 시스템 전체 AI 연산 에너지 **70% 이상 극적 절감** |
| **정성 (반도체 생태계)** | 메모리 회사는 단순 하청(저장고) 역할 | 메모리 회사가 연산 능력까지 확보 | 메모리 벤더가 시스템 반도체 설계의 주도권 탈환 |

### 미래 전망
- **메모리(DRAM)와 로직(Logic) 공정의 융합 단층 극복**: D램 공정은 전하를 잘 가두는 데 특화되어 트랜지스터(연산기)를 만들면 속도가 느리고 면적을 많이 차지한다. 반면 파운드리 로직 공정은 연산은 빠르나 커패시터를 못 만든다. 미래의 PIM은 3D 하이브리드 본딩을 이용해 아래층에는 3nm 로직(연산기) 다이를, 바로 위층에는 DRAM 1znm 다이를 붙여 두 공정의 단점을 완벽히 상쇄하는 진정한 **SoIC (System-on-Integrated-Chips)** 형태로 완성될 것이다.
- **Processing-In-Storage (ISP)**: 메모리를 넘어 하드디스크(SSD) 레벨까지 PIM 사상이 퍼지고 있다. 삼성의 'SmartSSD'는 SSD 컨트롤러 안에 ARM 코어나 FPGA를 박아 넣어, 데이터베이스의 필터링이나 동영상 압축 해제를 디스크 내부에서 1차로 끝내버리는 형태(컴퓨테이셔널 스토리지)로 발전 중이다.

### 참고 표준
- 아직 초기 단계라 JEDEC 표준이 확립되지 않았다. 각 사(삼성, SK, 마이크론)가 자사만의 독자 아키텍처를 테스트 중이며, 이 파편화를 모아 산업 표준(PIM 표준 규격)을 제정하는 것이 상용화의 가장 시급한 과제다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[폰 노이만 병목 (Von Neumann Bottleneck)](./500_von_neumann_bottleneck.md)**: 데이터 저장(메모리)과 연산(CPU)이 떨어져 있어서 생기는 현대 컴퓨터구조의 영원한 적. PIM은 이 병목을 창조적으로 파괴한 구조다.
- **[HBM (High Bandwidth Memory)](./495_hbm.md)**: PIM이 주로 기생하는 숙주 메모리. HBM의 수직 적층 구조와 TSV 기술이 PIM 연산기의 집적을 가장 용이하게 만들어준다.
- **[도메인 특화 아키텍처 (DSA)](../3_architecture_basics_performance/156_dsa.md)**: 특정 작업에 몰빵하는 칩. PIM은 그중에서도 '메모리 병목이 심한 AI 추론'이라는 아주 좁은 도메인에 100% 최적화된 궁극의 DSA다.
- **[CXL (Compute Express Link)](./470_cxl.md)**: 차세대 메모리 풀링 기술. CXL 램 카드에 PIM 칩을 꽂으면, 서버는 단순히 메모리 용량만 늘어나는 게 아니라 '연산력'까지 함께 덤으로 얻게 되는 마법이 벌어진다.

## 👶 어린이를 위한 3줄 비유 설명
1. 옛날에는 도서관(메모리)에서 책을 잔뜩 빌려와서, 집에 있는 똑똑한 형(CPU)한테 "이 책들 다 읽고 요약해 줘!"라고 시켰어요. 무거운 책을 옮기느라 너무 힘들었죠.
2. 그래서 아예 **도서관 안에 똑똑한 사서 선생님(연산기)**을 엄청 많이 모셔다 놨어요! 이게 바로 **PIM**이에요.
3. 이제는 집에서 전화로 "도서관에 있는 요리책 다 읽고 김치찌개 레시피 하나만 알려주세요!"라고 하면, 사서들이 도서관 안에서 뚝딱 정리해서 레시피(정답) 딱 한 장만 보내준답니다. 힘을 엄청 아꼈죠!
