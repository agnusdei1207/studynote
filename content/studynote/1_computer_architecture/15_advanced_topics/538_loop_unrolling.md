> **핵심 인사이트**
> 1. 프로그램의 `for`나 `while` 같은 반복문은 한 번 돌 때마다 "루프가 끝났나?(조건 검사)" 확인하고, "루프의 처음으로 돌아가자(분기 점프)"라는 추가적인 **제어 오버헤드**를 발생시킨다.
> 2. **루프 언롤링 (Loop Unrolling)**은 루프 안의 알맹이 코드를 여러 개로 복사하여 늘어놓음으로써(Unroll), 루프가 도는 총 횟수를 줄이고 무의미한 조건 검사 횟수를 삭제하는 기법이다.
> 3. 이를 통해 분기 예측 실패(Misprediction) 페널티를 줄이고, 명령어들을 일렬로 쫙 늘어놓게 되어 CPU가 비순차 실행(OoOE)과 파이프라이닝을 극한으로 끌어올릴 수 있게 된다.

<br>

## Ⅰ. 루프 구조의 숨겨진 낭비 (Overhead)

아주 단순한 루프를 생각해 봅시다.
```c
for (int i = 0; i < 100; i++) {
    A[i] = 0;
}
```

이 코드는 단순히 100개의 배열에 0을 채우는 것처럼 보이지만, 컴퓨터(어셈블리어) 입장에서는 1번 돌 때마다 다음의 3가지 일을 해야 합니다.
1. 알맹이: `A[i] = 0` 실행
2. 카운터: `i`를 1 증가시킴 (`i++`)
3. **제어 점프: `i < 100`인지 비교하고, 맞으면 맨 위로 다시 점프!**

**결과**: 100번의 알맹이 실행을 위해, **100번의 점프 명령어와 100번의 조건 비교 연산**이 낭비됩니다. 점프 명령어가 많아질수록 분기 예측 실패의 위험성도 커집니다.

> 📢 **섹션 요약 비유**: 우편함 100개에 전단지를 꽂아야 합니다. 전단지 한 장 꽂고 "혹시 100번 다 꽂았나?" 수첩 확인하고, 다음 우편함 가고, 또 한 장 꽂고 수첩 확인하는 멍청하고 피곤한 방식입니다.

<br>

## Ⅱ. 루프 언롤링의 마법 (풀어헤치기)

컴파일러(예: GCC의 `-O2`, `-O3` 옵션)는 이 낭비를 막기 위해 코드를 인위적으로 **언롤링(풀어헤침)** 합니다.

루프를 4배(Unroll factor = 4)로 풀어헤치면 코드는 이렇게 변합니다.

```c
for (int i = 0; i < 100; i += 4) {
    A[i]   = 0;
    A[i+1] = 0;
    A[i+2] = 0;
    A[i+3] = 0;
}
```

**얻게 되는 이득:**
1. **점프 횟수 감소**: 루프가 100번에서 25번으로 줄었습니다. 무의미한 조건 비교와 분기 점프 횟수가 75%나 사라졌습니다!
2. **명령어 병렬성(ILP) 극대화**: CPU의 윈도우 안에 `A[i] = 0`부터 `A[i+3] = 0`까지 서로 아무 관련이 없는 덧셈 명령어 4개가 한 번에 들어옵니다. CPU는 이 4개를 각기 다른 ALU에 던져주어 1클럭에 동시에 처리해 버립니다.

> 📢 **섹션 요약 비유**: "수첩 좀 그만 확인하고, 일단 무지성으로 전단지 4장 연속으로 꽂은 다음에 한 번만 수첩 확인해!"라고 작업 방식을 바꾼 것입니다. 속도가 4배 이상 빨라집니다.

<br>

## Ⅲ. 무한정 언롤링할 수 없는 이유 (트레이드오프)

"그럼 아예 100번 다 풀어헤치지 왜 4번만 풀까?" 
루프 언롤링에는 명확한 부작용이 있습니다.

1. **코드 크기 비대화 (Code Bloat)**
   루프를 100번 풀어헤치면 프로그램 파일 크기가 커집니다.
2. **명령어 캐시(I-Cache) 미스 폭발**
   코드가 커지면 CPU 내부에 있는 작고 소중한 명령어 캐시(L1 I-Cache) 용량을 초과해 버립니다. 루프가 짧으면 캐시 안에 쏙 들어가서 초고속으로 뱅글뱅글 돌 수 있는데, 언롤링을 너무 많이 하면 코드가 캐시 밖으로 삐져나가 메모리에서 계속 코드를 읽어와야 하므로 오히려 성능이 박살납니다.

따라서 컴파일러는 캐시 크기와 레지스터 개수를 계산하여 '가장 효율적인 수준(보통 4배~8배)'까지만 루프를 언롤링하는 고도의 눈치 게임을 수행합니다.
