> **핵심 인사이트**
> 1. 가상 머신(VM)에서 메모리를 읽으려면 `VM의 가상 주소 $\rightarrow$ VM의 가짜 물리 주소 $\rightarrow$ 진짜 서버의 물리 주소`라는 끔찍한 2단계 번역을 거쳐야 한다.
> 2. CPU에 하드웨어 가상화 기능(EPT/NPT)이 없던 시절, 하이퍼바이저(VMware 등)는 이 병목을 피하기 위해 **`가상 주소 $\rightarrow$ 진짜 물리 주소`로 바로 직행하는 숏컷 장부인 '그림자 페이지 테이블(Shadow Page Table)'을 소프트웨어로 만들어 CPU(MMU)에 꽂아주었다.**
> 3. 속도는 그럭저럭 빠르지만, VM이 자신의 원래 장부를 수정할 때마다 하이퍼바이저가 몰래 따라가서 그림자 장부도 똑같이 수정해 주어야 하는 **극심한 동기화 오버헤드(Trap & Emulate)**가 발생했다.

<br>

## Ⅰ. 2단계 번역의 늪과 하이퍼바이저의 꼼수

우리가 영어를 한국어로 번역하고, 다시 한국어를 일본어로 번역해서 듣는다고 생각해 봅시다 (2단계 번역). 당연히 시간이 두 배로 걸립니다.
* 1단계: VM 안의 프로그램 주소 $\rightarrow$ VM 가짜 물리 주소 (VM OS가 관리)
* 2단계: VM 가짜 물리 주소 $\rightarrow$ 진짜 서버 물리 주소 (하이퍼바이저가 관리)

구형 CPU(MMU)는 사전을 딱 1개밖에 들 수 없어서 이 2단계를 혼자서 처리할 수가 없었습니다. 
그래서 하이퍼바이저가 나섰습니다.
"VM아, 네가 쓰는 1단계 사전은 네 방(메모리)에 고이 모셔둬. 내가 밤새워 번역해서 **영어를 일본어로 바로 번역하는 직통 사전(그림자 페이지 테이블)**을 몰래 하나 만들 테니, CPU 하드웨어한테는 이 직통 사전을 쥐여주자!"

> 📢 **섹션 요약 비유**: 번역 과정을 두 번 거치는 게 너무 느려서, 하이퍼바이저가 두 사전을 합친 '최종 직통 요약본(그림자 장부)'을 수작업으로 만들어서 CPU에게 건네준 것입니다.

<br>

## Ⅱ. 그림자 페이지 테이블의 치명적 부작용: 쓰기 권한 함정 (Write-Protect)

직통 사전(그림자 장부)을 만들었더니 CPU가 데이터를 읽는 속도는 물리 서버(Bare-metal)와 비슷할 정도로 빨라졌습니다. 하지만 **"장부 동기화"**라는 지옥이 열렸습니다.

VM 안의 운영체제(우분투)가 프로그램을 켜거나 끌 때마다 자기 방에 있는 원래 사전(1단계 사전)의 내용을 수정합니다.
문제는 우분투가 사전을 수정했다는 사실을 하이퍼바이저가 모르면, CPU가 들고 있는 직통 사전(그림자 장부)은 옛날 주소를 가리키는 엉터리가 되어버린다는 것입니다.

**해결책 (동기화 지옥):**
1. 하이퍼바이저는 VM이 자기 사전을 건드리지 못하게, VM의 1단계 사전에 **쓰기 금지(Write-Protect, 읽기 전용) 자물쇠**를 콱 채워버립니다.
2. 우분투(VM)가 사전을 수정하려고 펜을 대는 순간! "삐빅! 쓰기 금지 오류(Page Fault)!"가 터지며 CPU가 하이퍼바이저에게 긴급 콜(Trap)을 보냅니다.
3. 하이퍼바이저가 달려와서 "어? 너 사전 고치려고 했어? 내가 고쳐줄게"라며 VM 사전을 고쳐주고, **동시에 자기가 몰래 만든 그림자 장부(직통 사전)도 똑같이 고쳐서 싱크를 맞춥니다.**
4. 이 작업을 다 끝내고 나서야 우분투에게 다시 제어권을 돌려줍니다.

> 📢 **섹션 요약 비유**: 요약본(그림자)을 원본과 똑같이 유지하기 위해, 하이퍼바이저는 원본 노트에 유리 덮개(쓰기 금지)를 씌웠습니다. VM이 원본 노트에 글을 쓰려다 유리에 부딪혀 멈칫하면, 하이퍼바이저가 달려와서 원본과 요약본 양쪽에 글씨를 대신 써주고 퇴장하는 피곤한 방식입니다.

<br>

## Ⅲ. 그림자 장부의 종말과 EPT의 승리

이 방식은 치명적인 성능 저하(VM Exit 폭풍)를 불렀습니다. 
* 메모리 맵핑이 자주 바뀌는 작업(예: 자식 프로세스를 계속 생성하는 `fork()`, 대규모 DB 쿼리)을 돌리면, 1초에 수만 번씩 장부를 동기화하느라 하이퍼바이저의 CPU 점유율이 100%를 찍고 VM은 멈춰버렸습니다 (TLB Flush 오버헤드).

결국 소프트웨어의 눈물겨운 똥꼬쇼는 한계에 부딪혔고, 이를 불쌍하게 여긴 인텔과 AMD가 **"그냥 CPU 기계(MMU) 안에 사전을 두 개 넣고 하드웨어가 알아서 두 번 번역하게 하자!"**라며 EPT/NPT(하드웨어 보조 메모리 가상화)를 개발하면서, 그림자 페이지 테이블은 최신 클라우드 서버에서 역사의 뒤안길로 사라지게 되었습니다.
