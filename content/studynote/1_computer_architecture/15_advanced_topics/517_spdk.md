# 517. SPDK (Storage Performance Development Kit)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 초고속 NVMe 플래시 스토리지를 제어할 때 운영체제의 커널(Kernel) 모드를 완전히 우회(Bypass)하여, 사용자 공간(User Space)에서 애플리케이션이 직접 디스크 하드웨어를 구동하게 만드는 소프트웨어 라이브러리 집합이다.
> 2. **가치**: 기존 인터럽트(Interrupt) 기반의 블록 I/O 스택이 유발하는 막대한 문맥 교환(Context Switch) 오버헤드를 제거하고 폴링(Polling) 방식으로 전환하여, NVMe SSD의 수백만 IOPS 성능을 100% 온전히 끌어낸다.
> 3. **융합**: 고성능 네트워크 스택을 우회하는 DPDK(Data Plane Development Kit) 철학을 스토리지로 확장한 것이며, 최신 NVMe-oF(네트워크 스토리지)와 결합하여 클라우드 환경의 데이터센터 백엔드를 지배하고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: SPDK(Storage Performance Development Kit)는 인텔이 오픈소스로 공개한, 고성능 스토리지 애플리케이션 작성을 위한 도구 모음(C 라이브러리)이다. 스토리지 장비 드라이버를 커널 스페이스가 아닌 유저 스페이스에서 실행하게 해 준다.
- **💡 비유**: 일반적인 스토리지 접근(커널 스택)이 물건을 창고에서 꺼낼 때 '경비실(커널)에 허가증을 내고, 경비원이 대신 물건을 찾아와 나에게 줄 때까지 멍하니 기다리는 것(인터럽트 대기)'이라면, SPDK는 아예 경비실을 없애고 **'내가 창고의 VIP 마스터키를 받아 직접 물건을 미친 듯이 퍼 나르는 것(유저 스페이스 폴링)'**이다.
- **등장 배경 및 발전 과정**:
  1. **소프트웨어가 하드웨어보다 느린 모순**: 과거 기계식 하드디스크(HDD) 시절에는 디스크가 너무 느렸기 때문에 OS 커널이 I/O 스케줄링을 하며 시간을 끌어도 문제가 없었다. 그러나 NVMe SSD가 10µs(마이크로초) 만에 응답하기 시작하자, 리눅스 커널의 블록 I/O 스택을 통과하는 데만 20µs가 걸려 소프트웨어가 하드웨어의 발목을 잡는 'I/O 세금' 현상이 심화되었다.
  2. **DPDK의 성공 체험**: 네트워크 분야에서 이미 커널을 우회해 패킷 처리하는 DPDK가 대성공을 거두었다. 이를 본 인텔은 스토리지 분야에도 똑같은 폴링(Polling) 및 커널 바이패스 철학을 도입하여 2013년 SPDK 프로젝트를 시작했다.
  3. **NVMe-oF 시대의 촉매**: 네트워크 너머의 스토리지를 로컬처럼 쓰는 NVMe-oF 기술이 등장하면서, 로컬 NVMe 제어와 네트워크 제어를 모두 유저 모드에서 극초저지연으로 결합하는 SPDK가 데이터센터 스토리지 소프트웨어의 사실상 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 / 특징 | 비유 |
|:---|:---|:---|:---|
| **User Space NVMe Driver** | 커널 NVMe 드라이버를 대체하는 SPDK 핵심 엔진 | UI/앱과 하드웨어 큐(SQ/CQ)를 직접 매핑 (Kernel Bypass) | 경비실 거치지 않는 직통 마스터키 |
| **Polled-mode Execution** | 하드웨어 작업 완료를 확인하는 방식 | 인터럽트를 기다리지 않고, CPU 코어 하나가 큐를 계속 무한 루프로 감시(Polling) | 알람 울릴 때까지 안 자고 계속 시계 쳐다보기 |
| **Lock-free Data Structures** | 멀티스레드 환경에서 락(Lock) 경합 방지 | 각 스레드가 자신만의 전용 NVMe 큐 쌍(QP)을 할당받아 배타적 사용 | 각자 개인 전용 우편함 사용 |
| **Bdev (Block Device) Layer** | 추상화된 블록 스토리지 계층 제공 | NVMe뿐만 아니라 iSCSI, Ceph, RAMDisk 등을 동일한 API로 묶음 | 통합 물류 관리 시스템 |
| **Vhost Target** | 가상머신(QEMU/KVM)에 스토리지를 초고속으로 제공 | VM의 I/O 요청을 커널(vhost-net) 없이 SPDK 유저 스레드가 직접 낚아챔 | 호텔 룸서비스 전담 특급 버틀러 |

### Legacy Kernel I/O Stack vs SPDK Architecture 비교

왜 기존 방식은 느릴 수밖에 없었는가? 커널 모드 진입(Context Switch)과 인터럽트(Interrupt) 처리 때문이다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │                 리눅스 Kernel I/O 스택 vs SPDK 스택                  │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] Legacy Kernel I/O 스택 (무겁고 복잡함)                        │
  │                                                             │
  │   [User Space]       Application (DB, Web)                  │
  │                      │ ◀─ (1. 시스템 콜: Context Switch)     │
  │  ────────────────────┼──────────────────────────────────────  │
  │   [Kernel Space]     VFS (가상 파일 시스템)                    │
  │                      │                                      │
  │                      Block Layer (I/O 스케줄러, 엘리베이터)       │
  │                      │ ◀─ (2. Lock 경합, 데이터 구조 복사)     │
  │                      NVMe Driver                            │
  │  ────────────────────┼──────────────────────────────────────  │
  │   [Hardware]         NVMe SSD                               │
  │                        ◀─ (3. 인터럽트 발생: CPU 스톨 유발)   │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] SPDK 스택 (가볍고 초고속)                                    │
  │                                                             │
  │   [User Space]       Application (DB, SPDK Target)          │
  │                      │                                      │
  │                      SPDK NVMe Driver                       │
  │                      │ ◀─ (1. Lock-free, 시스템 콜 Zero)     │
  │                      │ ◀─ (2. 무한 폴링, 인터럽트 Zero)        │
  │  ────────────────────┼──────────────────────────────────────  │
  │   [Kernel Space]     │ (우회됨! 텅 빔!)                       │
  │  ────────────────────┼──────────────────────────────────────  │
  │   [Hardware]         NVMe SSD                               │
  │                        ◀─ (3. PCIe DMA 직결)                │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 레거시 스택에서 앱이 데이터를 읽으려면 `read()` 시스템 콜을 호출하여 유저 모드에서 커널 모드로 전환해야 한다. 이후 수십만 줄의 커널 코드(VFS, 파일 시스템, 블록 레이어)를 거친다. 드라이버가 SSD에 명령을 내린 뒤 CPU 코어는 다른 일을 하러 가고(Sleep), SSD가 일을 끝내면 **하드웨어 인터럽트**를 발생시켜 CPU를 강제로 깨운다(Wake-up). 이 전체 과정에서 낭비되는 사이클이 어마어마하다.
하단의 **SPDK**는 OS 커널을 완전히 무시한다. SPDK 라이브러리를 품은 애플리케이션은 사용자 공간(User Space)에서 PCIe 버스 레지스터를 직접 매핑하여 NVMe 큐에 명령어를 쓴다. 그리고 CPU를 재우지 않고 무한 루프(while 문)를 돌며 NVMe 큐에 결과가 도착했는지 계속 확인한다(**Polling**). 인터럽트도, 문맥 교환도 없으므로 NVMe SSD의 하드웨어 응답 속도(수 µs)가 곧 애플리케이션의 응답 속도가 된다.

---

### 심층 동작 원리: Polling의 역설과 CPU 100% 점유
SPDK는 성능을 위해 CPU 코어 중 일부(예: 1~2개 코어)를 오직 I/O 큐만 무한 감시(Polling)하도록 강제 할당한다. 
* **현상**: 이 때문에 SPDK를 돌리면 해당 CPU 코어의 점유율은 디스크 사용량이 적어도 항상 100%로 뜬다.
* **이유**: `while(true) { check_queue(); }` 상태이기 때문이다.
* **트레이드오프**: CPU 코어 1개를 완전히 희생(낭비)하는 것처럼 보이지만, 그 코어 1개가 혼자서 초당 500만 건의 I/O(5M IOPS)를 인터럽트 없이 쳐낸다. 기존 인터럽트 방식이었다면 500만 건의 I/O를 처리하기 위해 20개의 CPU 코어가 인터럽트 처리에 박살 나고 있었을 것이다. 즉, 폴링은 "코어 1개를 불태워 코어 19개를 살리는" 궁극의 효율 최적화다.

## Ⅲ. 융합 비교 및 다각도 분석

### 스토리지 엔진 설계 패러다임 비교

| 비교 항목 | Linux 커널 AIO / io_uring | SPDK (User Space Polling) |
|:---|:---|:---|
| **동작 위치** | OS 커널 내부 (Kernel Space) | **사용자 애플리케이션 내부 (User Space)** |
| **I/O 완료 알림** | 이벤트 기반 (Event / Interrupt) | **무한 폴링 (Polling, Busy-Wait)** |
| **장점** | 여러 프로세스가 디스크를 안전하게 공유 가능 | **절대적인 최고 성능, 극단적 초저지연 보장** |
| **단점** | 여전히 커널 스택을 통과해야 함 | 해당 디스크를 OS가 통제할 수 없음(독점), CPU 100% 낭비 |
| **사용처** | 일반적인 DB, 범용 애플리케이션 | **클라우드 벤더의 스토리지 백엔드, 초고성능 SAN 장비** |

### 과목 융합 관점
- **네트워크 (DPDK와의 결합)**: 네트워크 패킷 처리를 커널에서 빼내는 DPDK와 디스크 I/O를 커널에서 빼내는 SPDK는 형제 프로젝트다. 이 둘을 결합하면 데이터를 네트워크 랜카드(NIC)에서 받아 메모리에 올리고, 이를 바로 NVMe SSD로 내릴 때까지 **OS 커널이 단 한 번도 개입하지 않는** "완전 유저 스페이스 데이터 파이프라인"이 완성된다.
- **클라우드 스토리지 (NVMe-oF Target)**: AWS나 Azure의 블록 스토리지(EBS) 백엔드 서버는 SPDK를 기반으로 동작한다. 클라이언트 VM에서 네트워크(RoCEv2 등)를 통해 날아온 NVMe-oF 요청을 SPDK 타겟(Target)이 유저 스페이스에서 낚아채어 로컬 NVMe SSD에 써버리므로, 네트워크 스토리지임에도 로컬 디스크와 구별할 수 없는 속도를 낸다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 데이터베이스(ScyllaDB 등)의 극단적 레이턴시 최적화**: 밀리초(ms) 단위도 허용되지 않는 초실시간 광고 입찰(RTB) 시스템의 인메모리-디스크 영속성 데이터베이스 설계.
   - **아키텍처 적용**: DB 자체 엔진 코드를 수정하여 리눅스 파일 시스템(EXT4/XFS)을 거치지 않고, SPDK 라이브러리를 링크하여 DB 스레드가 원시 블록(Raw Device) 디바이스에 직접 읽기/쓰기를 하도록 만든다. VFS 계층과 페이지 캐시 계층을 우회하므로 99.9% 꼬리 지연(Tail Latency)이 획기적으로 개선된다.
2. **시나리오 — 하이퍼컨버지드 인프라(HCI)의 Vhost-user 가속**: KVM 기반의 가상머신(VM) 수십 대를 돌리는 노드에서, VM들이 발생시키는 디스크 I/O가 QEMU 에뮬레이터를 거치느라 엄청난 병목이 발생함.
   - **해결책**: SPDK의 **vhost-scsi** 또는 **vhost-nvme** 타겟을 호스트 OS에 띄운다. VM 게스트 OS 내의 Virtio 드라이버가 공유 메모리를 통해 I/O 링(Ring)에 데이터를 쓰면, 호스트의 SPDK 데몬이 폴링으로 이를 즉각 낚아채어 물리 SSD에 기록한다. QEMU나 커널의 개입을 완전히 배제하여 VM 스토리지 성능을 베어메탈(물리 서버)의 90% 수준까지 끌어올린다.

### 도입 체크리스트
- **독점적 하드웨어 권한**: SPDK를 사용하여 특정 NVMe SSD를 유저 스페이스에 바인딩(Binding, 예: VFIO/UIO 드라이버)하면, 그 순간부터 리눅스 커널(OS)은 해당 디스크가 사라진 것으로 간주한다. 따라서 그 디스크 위에서는 일반적인 파일(File) 생성이나 `ls`, `cat` 명령어를 쓸 수 없으며, 오직 SPDK 라이브러리로 짠 프로그램만 블록(LBA) 단위로 접근할 수 있다는 관리상의 제약을 반드시 숙지해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 리눅스 커널 기본 드라이버 | SPDK 도입 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (단일 코어 IOPS)** | 1개 코어로 약 300K ~ 500K IOPS 처리 | 1개 코어로 **5,000K (5백만) IOPS** 이상 처리 | 코어당 I/O 처리 효율 **10배 향상** |
| **정량 (I/O Latency)** | 소프트웨어 스택 오버헤드 약 10~20 µs | 오버헤드 **1 µs 미만** | 스토리지 병목 제거 및 P99 지연 시간 완벽 방어 |
| **정성 (자원 효율성)** | IOPS 증가 시 인터럽트로 인한 CPU 스톨 확산 | 폴링 전용 코어 격리로 타 앱의 성능 간섭 차단 | 코어 격리(CPU Pinning)를 통한 예측 가능한 성능 설계 |

### 미래 전망
- **스마트NIC / DPU로의 이식**: CPU 코어를 무한 폴링에 낭비하는 것이 아깝다는 의견에 따라, 최근에는 이 SPDK 소프트웨어 데몬 전체를 메인 CPU가 아닌 서버에 꽂힌 DPU(Data Processing Unit) 내부의 ARM 코어로 밀어내는(Offloading) 아키텍처가 트렌드다. 이렇게 되면 메인 x86 CPU는 스토리지 처리에 단 1%의 자원도 쓰지 않게 된다.
- **eBPF와 io_uring의 추격**: 커널의 반격도 거세다. 리눅스 커널은 시스템 콜 없이 링 버퍼를 공유하는 **io_uring**을 도입하여 SPDK 성능의 턱밑까지 쫓아왔고, 커널 내부에 유저 코드를 꽂아 넣는 **eBPF** 기술을 통해 커널 스택의 오버헤드를 대폭 줄이고 있다. 향후 독점적 통제권이 필요한 클라우드 백엔드는 SPDK가, 범용 고성능 앱은 io_uring이 양분할 것이다.

### 참고 표준
- **NVMe / NVMe-oF 규격**: SPDK는 사실상 이 두 하드웨어 규격을 소프트웨어적으로 가장 완벽하고 빠르게 구현하기 위한 레퍼런스 엔진에 해당한다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[NVMe (Non-Volatile Memory Express)](./509_nvme.md)**: HDD의 잔재를 털어내고 플래시 메모리의 병렬성을 극대화한 하드웨어 프로토콜. SPDK는 이 NVMe의 잠재력을 봉인 해제하는 열쇠다.
- **[문맥 교환 (Context Switch)](../2_operating_system/2_process_thread/xx_context_switch.md)**: 유저 모드와 커널 모드를 오가는 값비싼 행위. SPDK의 존재 목적 1순위가 바로 이 문맥 교환의 박멸이다.
- **[소프트웨어 정의 인프라 하드웨어 종속성 (SDI Hardware)](./499_sdi_hardware.md)**: 인프라 소프트웨어(가상 스위치, 스토리지 컨트롤러)의 오버헤드 한계를 설명하는 개념. 이 오버헤드를 극복하기 위해 탄생한 것이 SPDK, DPDK 다.
- **[인터럽트 (Interrupt)](../2_operating_system/1_overview_architecture/xx_interrupt.md)**: 장치가 CPU를 찔러서 깨우는 방식. 과거엔 효율적이었으나 10µs짜리 SSD 시대에는 오히려 족쇄가 되어 폴링(Polling) 방식으로 대체되었다.

## 👶 어린이를 위한 3줄 비유 설명
1. 보통 창고에서 과자를 꺼내려면 경비실 아저씨(운영체제)한테 부탁하고, 아저씨가 창고에 다녀와서 문을 두드릴 때(인터럽트)까지 방에서 기다려야 했어요.
2. 그런데 과자가 1초에 만 개씩 쏟아지는 엄청난 창고(NVMe SSD)가 생기자, 아저씨한테 부탁하는 시간조차 너무 아깝고 답답해졌어요.
3. 그래서 **SPDK**라는 마법의 황금 열쇠를 만들었어요. 이제 내가 직접 창고 문을 열어놓고 과자가 나오는 족족 내 주머니(유저 스페이스)에 바로 쓸어 담을 수 있게 됐답니다!
