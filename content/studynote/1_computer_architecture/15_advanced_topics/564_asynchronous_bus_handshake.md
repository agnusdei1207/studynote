> **핵심 인사이트**
> 1. CPU처럼 5GHz로 엄청나게 빠르게 도는 칩과, USB나 하드디스크처럼 느리게 도는 칩이 통신할 때 하나의 시계(클럭)로 속도를 맞추려 하면(동기식 버스) 가장 느린 장치에 속도가 하향 평준화되는 문제가 발생한다.
> 2. 이를 해결하기 위해 두 장치 간에 기준이 되는 시계(Clock)를 아예 없애버리고, 오직 **"나 데이터 보냈다(Req)", "잘 받았다(Ack)"라는 두 개의 신호선만으로 소통하는 방식이 비동기(Asynchronous) 버스**다.
> 3. 이 과정에서 한 치의 오차도 없이 데이터를 전달하기 위해 사용되는 **4단계 완전 핸드셰이크(4-Way Handshake)** 프로토콜이 비동기 설계의 핵심이다.

<br>

## Ⅰ. 동기식 버스의 한계 (클럭의 독재)

**동기식 버스(Synchronous Bus)**는 메인보드 중앙에 있는 시계(Clock Generator)의 딱딱거리는 템포에 맞춰 모든 장치가 일제히 행동하는 버스입니다. 
장점은 설계가 아주 쉽다는 것입니다. 하지만 심각한 단점이 있습니다.

* **속도 하향 평준화**: 1클럭에 데이터를 받아야 하는데, 세상에서 제일 느린 CD-ROM 드라이브를 버스에 달았다고 칩시다. CD-ROM이 1클럭 안에 데이터를 못 주면 에러가 나므로, 어쩔 수 없이 버스 전체의 클럭 속도를 CD-ROM 수준으로 팍 낮춰야 합니다.
* **물리적 한계 (클럭 스큐)**: 메인보드가 너무 커지면, 시계 신호가 1번 칩에 도달하는 시간과 10번 칩에 도달하는 시간이 미세하게 달라져(빛의 속도 한계) 타이밍이 꼬입니다.

> 📢 **섹션 요약 비유**: 동기식 버스는 100명이 2인 3각 경기를 하는 것과 같습니다. 메트로놈(클럭) 소리에 맞춰 다 같이 발을 굴러야 하므로, 무조건 제일 느린 사람(장치)의 걸음걸이에 템포를 맞춰야만 넘어지지 않습니다.

<br>

## Ⅱ. 비동기 핸드셰이크 프로토콜의 마법

"시계를 부숴버리자!"
**비동기식 버스**는 공통의 클럭을 없애고, 두 장치가 각자 자기 속도대로 놀게 둡니다. 대신 둘 사이에 **Req(요청)** 선과 **Ack(응답)** 선이라는 두 가닥의 신호선을 만들어, 대화하듯이 데이터를 넘깁니다. 이를 **핸드셰이크(악수)**라고 부릅니다.

### 4-Way Handshake 동작 과정 (완전 핸드셰이크)

송신자(빠른 CPU)와 수신자(느린 HDD)가 데이터를 주고받는 4단계 과정입니다.

1. **Req 올림 (송신자)**: CPU가 데이터 선에 데이터를 예쁘게 올려놓은 뒤, "나 데이터 놨다! 가져가라!"라며 `Req` 선의 전기를 켭니다(1).
2. **Ack 올림 (수신자)**: 느린 HDD가 자기 페이스대로 천천히 데이터를 읽어 들입니다. 다 읽고 나면 "응, 잘 받았어!"라며 `Ack` 선의 전기를 켭니다(1).
3. **Req 내림 (송신자)**: HDD가 잘 받았다는 `Ack`를 본 CPU는 그제야 안심하고 "오케이, 그럼 나 이제 데이터 치울게!"라며 `Req` 선을 끕니다(0).
4. **Ack 내림 (수신자)**: CPU가 `Req`를 끈 것을 본 HDD는 "알았어, 다음 데이터 올 때까지 대기할게"라며 `Ack` 선을 끕니다(0).

이 4단계를 거쳐야 비로소 **단 1개의 데이터**가 완벽하게 안전하게 전송됩니다.

### 핸드셰이크 신호도 (ASCII)

```text
 데이터 버스: XXXXXXXX[  안전한 유효 데이터 영역  ]XXXXXXXXXXXXX
             
 Req 신호   : ───┐   (1)             (3) ┌───────────────
 (송신자)         └───────────────────────┘
                     │               │
                     ▼ (기다림)      ▼ (기다림)
                     
 Ack 신호   : ───────┐ (2)           (4) ┌───────────────
 (수신자)             └───────────────────┘
```

> 📢 **섹션 요약 비유**: 물건(데이터)을 건넬 때, 시계가 없습니다. 
> 1. 내밀고("받아!") 
> 2. 상대가 잡으면("잡았어!") 
> 3. 내 손을 놓고("놓는다!") 
> 4. 상대도 완전히 가져가서 손을 뺍니다("가져갔어!"). 
> 속도가 아무리 느려도 절대 물건을 바닥에 떨어뜨릴(데이터 손실) 일이 없는 가장 안전한 전달 방식입니다.

<br>

## Ⅲ. 비동기 버스의 트레이드오프

* **장점**: 어떤 속도의 장치를 꽂아도 지들끼리 알아서 속도를 맞춰(Negotiation) 대화하므로, 시스템 확장성과 호환성이 미친 듯이 좋습니다.
* **단점 (오버헤드)**: 데이터 1개를 보낼 때마다 "줬어?" "어" "뺄게" "어"라는 4번의 확인 과정(시간 지연)이 낭비됩니다. 초고속이 생명인 캐시나 메모리 통신에 쓰기에는 너무 지루합니다.

따라서 현대 컴퓨터에서는 같은 칩 내부나 초고속 메모리(DRAM) 통신에는 빠르고 멍청한 **동기식 버스**를 쓰고, USB나 구형 주변기기, 칩과 칩을 멀리 연결하는 외부 포트 등에는 호환성 높은 **비동기 핸드셰이크 버스**를 혼용하는 설계(GALS: Globally Asynchronous Locally Synchronous)가 주류를 이룹니다.
