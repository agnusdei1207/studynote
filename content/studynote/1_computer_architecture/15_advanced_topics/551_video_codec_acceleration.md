# 551. 비디오 코덱 하드웨어 가속 (H.265 / AV1)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동영상 데이터를 압축(인코딩)하고 압축을 푸는(디코딩) 엄청난 연산량을 범용 CPU나 GPU 셰이더로 처리하는 대신, 이 수식만 전담해서 계산하도록 실리콘에 물리적으로 구워놓은(Hard-wired) 전용 ASIC 블록(가속기)이다.
> 2. **가치**: 4K/8K 초고화질 스트리밍 환경에서 CPU 점유율을 0%로 만들면서도 배터리 소모를 극적으로 줄이고 실시간성을 보장한다. 하드웨어 가속기가 없으면 스마트폰에서 넷플릭스를 볼 때 30분 만에 배터리가 방전되고 기기가 불타게 된다.
> 3. **융합**: H.264/H.265(HEVC)를 넘어 로열티가 없는 차세대 코덱인 AV1에 대한 하드웨어 디코딩/인코딩 지원 여부가 최근 모바일 AP(Apple A17, Snapdragon)와 그래픽카드(Nvidia RTX 40 등)의 핵심 스펙 경쟁 포인트가 되었다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 비디오 코덱 하드웨어 가속은 소프트웨어적으로 구현된 압축 알고리즘(코덱)을 하드웨어 논리 게이트 덩어리(VPU, Video Processing Unit)로 교체하여, 비디오의 픽셀 예측, 변환, 엔트로피 코딩 등을 초고속·저전력으로 수행하는 기술이다.
- **💡 비유**: CPU가 동영상을 디코딩하는 것은 '만능 요리사(CPU)'가 책을 보면서 수백만 개의 감자(픽셀)를 일일이 깎는 것과 같다. 요리사는 지치고 밥(다른 프로그램)도 못 만든다. 하드웨어 가속기는 아예 주방 한구석에 **'전자동 감자 깎는 기계(VPU)'**를 들여놓은 것이다. 요리사는 감자 포대만 기계에 넣어주고 다른 요리를 할 수 있다.
- **등장 배경 및 발전 과정**:
  1. **소프트웨어 디코딩의 딜레마**: 과거 MPEG-2 시절에는 비디오 압축률이 낮아 CPU만으로도 버틸 만했다. 그러나 H.264, H.265(HEVC)로 진화하며 수학 연산량이 기하급수적으로 폭발했다.
  2. **모바일 스트리밍 시대**: 배터리 용량이 제한된 스마트폰에서 유튜브 1080p를 소프트웨어로 돌리니 배터리가 버티지 못했다. 결국 퀄컴, 애플 등은 칩셋 안에 영상만 처리하는 미디어 엔진(하드웨어 코덱)을 필수적으로 박아 넣게 되었다.
  3. **AV1 코덱의 등장**: 구글, 넷플릭스 등이 주도하여 만든 AV1 코덱은 압축률이 H.265보다 30% 좋지만 연산량은 3~5배 더 무겁다. 이 무거운 AV1을 하드웨어로 가속하느냐 못 하느냐가 최신 디바이스의 세대교체 기준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (하드웨어 비디오 파이프라인)

비디오 압축/해제는 일련의 빡빡한 수학적 파이프라인으로 구성된다. 하드웨어 가속기는 이 파이프라인 단계를 물리적인 블록으로 구성한다.

| 요소명 (디코딩 기준) | 역할 | 특징 | 비유 |
|:---|:---|:---|:---|
| **Entropy Decoder** | 압축된 비트스트림(CABAC/CAVLC)을 풀어서 숫자로 만듦 | 데이터가 심하게 꼬여 있어 병렬화가 어려우며, 하드웨어 회로 설계가 가장 빡센 구간 | 압축된 암호문 해독 |
| **Inverse Quantization & IDCT** | 양자화된 데이터를 역양자화하고 주파수 영역을 다시 공간(픽셀) 영역으로 변환 | 행렬 곱셈 연산이 주를 이룸 (DSP 친화적) | 주파수 신호를 그림 스케치로 변환 |
| **Motion Compensation (움직임 보상)** | 이전/이후 프레임의 화면을 참조(Reference)하여 현재 프레임을 예측해 채워 넣음 | 메모리 대역폭(Bandwidth)을 가장 많이 잡아먹는 구간. 캐시 최적화 필수 | 이전 사진을 투명 종이에 대고 베끼기 |
| **Intra Prediction (인트라 예측)** | 한 프레임 내에서 주변 픽셀의 색깔을 보고 빈 공간을 유추 | 화면 내의 그라데이션 같은 연속적인 변화를 채움 | 주변 색연필 색깔로 빈칸 색칠하기 |
| **In-Loop Filter** | 픽셀 예측 시 생긴 깍두기 모양(블록 노이즈)을 부드럽게 뭉개줌 | 디블로킹(Deblocking) 필터, ALF 등 시각적 퀄리티를 결정함 | 붓질한 자국을 물로 부드럽게 펴 바르기 |

### 소프트웨어(CPU) vs 하드웨어(VPU) 비디오 처리 아키텍처

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         CPU 소프트웨어 디코딩 vs 하드웨어(VPU) 가속 디코딩 비교        │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 소프트웨어 디코딩 (CPU/GPU Shader 처리) - "전력 낭비 극심"         │
  │                                                             │
  │   [ 4K H.265 비트스트림 수신 ]                                │
  │        │                                                    │
  │        ▼ (수십 MB의 코드를 L1/L2 캐시에서 뺑뺑이 돌림)              │
  │   ┌────────────────────────────────────────────────┐        │
  │   │ [범용 CPU Core]                                │        │
  │   │ - 엔트로피 디코딩 연산 (분기문 엄청 많음 -> 분기 페널티) │        │
  │   │ - 움직임 보상 연산 (메모리 엑세스 폭발 -> 캐시 미스)   │        │
  │   │   ▶ 결과: CPU 코어 4개가 100% 풀로드. 온도 90도 돌파! │        │
  │   └────────────────────────────────────────────────┘        │
  │        │                                                    │
  │        ▼ [ YUV 픽셀 데이터 출력 ]                              │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] 전용 하드웨어 가속 디코딩 (VPU / Media Engine) - "전력 극소화"     │
  │                                                             │
  │   [ 4K H.265 비트스트림 수신 ]                                │
  │        │                                                    │
  │        ▼ (CPU는 명령만 내리고 꿀잠 Sleep 모드 진입)               │
  │   ┌────────────────────────────────────────────────┐        │
  │   │ [전용 VPU / ASIC 블록] (칩 한구석에 있는 특수 회로)    │        │
  │   │  [CABAC 회로] ─▶ [IDCT 회로] ─▶ [Motion 회로]     │        │
  │   │   ▶ 결과: 논리 게이트를 통과하며 1ms 만에 픽셀로 변환.  │        │
  │   │            소비 전력 1W 미만. 발열 없음.               │        │
  │   └────────────────────────────────────────────────┘        │
  │        │                                                    │
  │        ▼ [ YUV 픽셀 데이터 출력 ]                              │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** H.265(HEVC) 코덱은 압축 효율을 높이기 위해 화면을 64x64부터 4x4 크기까지 아주 잘게, 그것도 불규칙하게 쪼개서 수학 연산을 한다. 범용 CPU에게 이걸 시키면 "If (크기가 8x8이면) -> Jump" 같은 분기문(Branch)이 너무 많아서 성능이 반토막이 난다.
하지만 **전용 하드웨어 가속기(예: Nvidia NVDEC, Intel Quick Sync)**는 이 복잡한 H.265 표준 스펙 자체를 아예 실리콘 논리 회로(ASIC)로 굳혀놓았다. 데이터가 들어오면 파이프라인 컨베이어 벨트를 타듯 슉슉 통과하면서 곧바로 YUV 픽셀 데이터로 토해낸다. 하드웨어가 일하는 동안 비싼 CPU는 깊은 잠(Sleep)에 빠지기 때문에 스마트폰 배터리가 10시간 이상 버틸 수 있는 것이다.

---

### 심층 비교: AV1 코덱과 하드웨어 가속의 한계
AV1은 H.265보다 압축률이 우수해 동일 화질에서 데이터(트래픽)를 30%나 아껴주지만, 하드웨어 설계자들에게는 악몽이다. AV1은 예측 방향(Prediction mode)이 100개가 넘고, CDEF, LR 같은 엄청나게 복잡한 필터를 추가했다.
만약 구형 폰(AV1 가속기가 없는 칩)에서 유튜브 AV1 영상을 틀면, 시스템은 어쩔 수 없이 CPU를 100% 갈구는 **소프트웨어 폴백(Software Fallback)**을 시도하다가 화면이 뚝뚝 끊기고 폰이 뜨거워진다. 그래서 최근 모든 디바이스 교체의 가장 핵심적인 스펙 확인 사항이 "AV1 하드웨어 디코딩을 지원하는가?"가 되었다.

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 비디오 코덱 규격 및 하드웨어 지원 현황

코덱은 라이선스(돈)와 연산량의 싸움이다.

| 코덱 규격 | 주도 세력 | 특징 및 하드웨어 지원 현황 | 압축 효율 | 연산량(복잡도) |
|:---|:---|:---|:---|:---|
| **H.264 (AVC)** | MPEG, ITU-T | FHD 시대의 제왕. 세상 모든 기기에 하드웨어 가속기가 달려있음 | 기준 (1x) | 낮음 (가벼움) |
| **H.265 (HEVC)**| MPEG, ITU-T | 4K 시대의 표준. 애플이 밀어줌. 하지만 복잡하고 비싼 로열티 문제가 있음. 대부분 기기 가속 지원 | 2배 좋음 | 높음 |
| **VP9** | Google | 유튜브 4K의 핵심. H.265의 무료 대항마. 대부분 가속 지원 | 1.8배 좋음 | 중간 |
| **AV1** | AOMedia (Google, 넷플릭스 등) | **로열티 무료. 최강의 압축률.** 최신 칩(Intel 12세대 이상, RTX 40, M3 등)에서만 하드웨어 가속 지원 | **2.5배 좋음** | **매우 높음 (극악)** |

### 과목 융합 관점
- **클라우드 컴퓨팅 (스트리밍 인프라 트위치/유튜브)**: 넷플릭스나 유튜브는 매일 수만 시간의 영상을 서버에서 H.265나 AV1으로 압축(Encoding)해야 한다. 이 인코딩을 서버 CPU(x86)로 하면 전기세와 서버 구매비로 회사가 파산한다. 이를 위해 구글은 오직 유튜브 영상을 AV1으로 하드웨어 인코딩만 하는 커스텀 ASIC 칩인 **Argos(VCU)**를 자체 개발하여 데이터센터에 수만 장을 꽂아 넣었다.
- **네트워크 (WebRTC 및 초저지연 전송)**: 클라우드 게이밍(GeForce Now)이나 드론 조종 영상은 화질보다 지연(Latency)이 생명이다. 하드웨어 인코더/디코더가 없다면 1프레임을 압축하는 데 100ms가 걸린다. 하드웨어 가속기는 이를 5ms 이내로 줄여주어, 서버에서 그려진 게임 화면을 내 모니터에서 1프레임의 딜레이도 없이 보면서 게임을 할 수 있게 만들어주는 WebRTC망의 일등 공신이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 사내 영상 회의 시스템(WebRTC)의 대규모 배포**: 줌(Zoom)이나 구글 미트를 브라우저에서 사용할 때, 노트북 쿨링팬이 미친 듯이 도는 현상.
   - **아키텍처 분석**: 브라우저(WebRTC)가 상대방의 카메라 영상을 디코딩할 때, 내 그래픽카드(GPU)에 탑재된 하드웨어 가속 블록(VDPAU, VA-API 등)을 사용하지 못하고 CPU로 소프트웨어 디코딩을 하고 있을 확률이 높다. 코덱이 AV1으로 잡혀있는데 내 노트북이 AV1 가속을 미지원하거나, 리눅스에서 GPU 가속 드라이버가 꼬인 경우다.
   - **해결책**: 화질을 조금 포기하더라도 양측의 협상(SDP Negotiation) 코덱을 H.264나 VP8처럼 100% 하드웨어 가속이 보장되는 코덱으로 강제(Fallback) 세팅하면 CPU 점유율이 5% 밑으로 떨어지고 팬이 조용해진다.
2. **시나리오 — 방송국 송출용 인코딩 팜(Encoding Farm) 구축**: 방송국에서 실시간으로 들어오는 8K 카메라 영상을 시청자에게 스트리밍하기 위해 H.265로 실시간 압축(인코딩)해야 한다.
   - **엔지니어링 판단 (Nvidia NVENC vs x265)**: 하드웨어 가속기(Nvidia NVENC)는 미친 듯이 빠르고 전기를 안 먹지만, 압축 알고리즘이 칩에 박혀있어 화질 최적화의 자유도가 떨어진다. 반면 CPU 기반 소프트웨어 인코더(`x265`)는 느리지만 인간의 눈을 속이는 꼼수(심리음향 모델 등)를 극단적으로 적용해 같은 용량 대비 화질이 훨씬 좋다. 실시간 라이브 방송은 무조건 **하드웨어 가속(NVENC)**을 쓰고, VOD(넷플릭스 영화)용 서버 배포 영상은 시간을 며칠 들여서라도 **CPU 소프트웨어 인코딩**을 돌리는 투-트랙 아키텍처를 취해야 한다.

### 도입 체크리스트
- **크로스 플랫폼 미디어 프레임워크 (FFmpeg/GStreamer)**: 하드웨어 가속기를 쓰려면 운영체제와 GPU 벤더별로 파편화된 API를 알아야 한다 (Windows: DXVA, Linux: VA-API, Mac: VideoToolbox, Android: MediaCodec). 백엔드 개발자는 직접 이 API를 제어하지 않고, 이를 완벽히 추상화해 둔 `FFmpeg` 파이프라인에서 `-hwaccel` 옵션 한 줄로 하드웨어를 유연하게 호출하는 아키텍처를 짜야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 소프트웨어 디코딩/인코딩 (CPU) | 하드웨어 가속(VPU/NVENC) 도입 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (전력 소모)** | 4K 재생 시 CPU 100% 가동 (수십 W) | 전용 VPU 사용으로 CPU 유휴 상태 (수 W 미만) | 스마트폰/태블릿 등 모바일 기기의 배터리 수명 기적적 연장 |
| **정량 (처리 속도)** | 프레임 밀림, 실시간 처리 불가 | 4K/8K 영상을 초당 수백 프레임 처리 | 클라우드 게이밍 및 실시간 방송(Live Streaming) 딜레이 최소화 |
| **정성 (자원 해방)** | 백그라운드 작업 동시 수행 불가 | 미디어 처리를 100% 오프로딩 | CPU는 UI 렌더링이나 게임 물리 연산(AI) 등 본연의 임무에 100% 집중 |

### 미래 전망
- **AI 기반 비디오 코덱(AI Codec)**: 기존의 H.265나 AV1은 사람이 만든 복잡한 수학 공식(블록 분할, 코사인 변환 등)에 의존했다. 미래에는 수학 공식 대신 **딥러닝(Autoencoder) 모델**이 영상을 압축하고 복원하는 AI 코덱이 등장할 것이다. 이때는 영상을 압축하기 위해 VPU가 아니라, NPU(신경망 가속기)가 영상 데이터를 추론(Inference)하여 픽셀을 생성해 내는 방식으로 패러다임이 바뀔 것이다. (마치 딥페이크가 영상을 만들어내듯)
- **차세대 VVC (H.266)의 등장**: AV1의 대항마로 VVC(Versatile Video Coding)가 표준화되었다. 8K와 360도 VR 영상을 타겟으로 하며 AV1보다 더 극악의 연산량을 자랑한다. 향후 2~3년 내에 이 VVC를 하드웨어로 가속할 수 있는 실리콘 블록을 칩셋에 얼마나 작고 저렴하게 박아 넣느냐가 팹리스(Qualcomm, MediaTek, Apple) 기업들의 승부처가 될 것이다.

### 참고 표준
- **AOMedia Video 1 (AV1)**: 구글, 애플, 아마존, 넷플릭스 등 전 세계 빅테크가 MPEG 진영의 살인적인 로열티(H.265)에 빡쳐서 대동단결하여 만든 오픈소스/로열티 프리(Royalty-free) 비디오 코덱 표준. 인터넷 비디오 트래픽의 미래를 책임질 절대 표준이다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[텐서 코어 (NPU)](./515_tensor_core.md)**: VPU가 비디오 '압축'을 푸는 장인이라면, 해상도를 4K로 업스케일링(DLSS, FSR)하거나 노이즈를 AI로 제거하는 후처리 작업은 이 NPU가 넘겨받아 수행한다.
- **[시스템 온 칩 (SoC)](../3_architecture_basics_performance/xx_soc.md)**: 이 모든 하드웨어 비디오 가속기(VPU, Media Engine)는 스마트폰 AP나 PC 칩셋 구석에 물리적인 땅(Silicon Area)을 차지하고 박혀 있는 하드웨어다.
- **[QoS (Quality of Service)](./511_qos.md)**: 아무리 하드웨어 코덱이 빨리 압축을 풀어도, 네트워크에서 영상 패킷이 버려지면 화면이 깨진다. 동영상 패킷을 최우선으로 전송하는 QoS 네트워크 기술과 하드웨어 디코더는 한 쌍으로 움직인다.
