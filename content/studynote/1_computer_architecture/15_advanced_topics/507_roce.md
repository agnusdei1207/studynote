# 507. RoCE (RDMA over Converged Ethernet)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPU와 OS 커널을 거치지 않고 네트워크 카드가 다른 서버의 메모리에 직접 읽고 쓰는 RDMA(Remote Direct Memory Access) 기술을, 비싸고 폐쇄적인 인피니밴드(InfiniBand) 대신 범용 이더넷(Ethernet)망 위에서 구현한 프로토콜이다.
> 2. **가치**: 데이터센터 네트워크 인프라를 이더넷으로 단일화(Converged)하면서도, 마이크로초(µs) 단위의 초저지연과 제로 카피(Zero-Copy) 고대역폭 통신을 가능하게 하여 AI/HPC 클러스터 구축 비용을 혁신적으로 낮춘다.
> 3. **융합**: UDP/IP 스택을 활용하여 라우팅 확장이 가능한 RoCEv2가 현재의 표준이 되었으며, 데이터 무손실(Lossless)을 보장하기 위해 PFC, ECN 같은 DCB(Data Center Bridging) 스위치 기술과 강력하게 결합되어야만 제 성능을 낸다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: RoCE (로키, RDMA over Converged Ethernet)는 이더넷 네트워크를 통해 RDMA 통신을 수행할 수 있게 해주는 네트워크 프로토콜이다. 기존 이더넷의 TCP/IP 스택이 유발하는 소프트웨어 처리 부하(CPU 점유율, 메모리 복사 지연)를 하드웨어(NIC) 레벨에서 우회(Bypass)한다.
- **💡 비유**: 한국에서 미국으로 우편물(데이터)을 보낼 때, 일반 TCP/IP 방식은 우체국(OS 커널)에 가서 번호표를 뽑고 직원이 포장(패킷 헤더 생성)하고 분류하는 절차를 거친다(시간+비용 낭비). 반면 RoCE는 내 사무실 책상(로컬 메모리)에서 전용 직통 파이프(RDMA NIC)를 통해 상대방 사무실 책상(원격 메모리)으로 우편물을 그 즉시 쏴버리는 마법의 직배송 시스템이다. 단, 이 파이프는 중간에 우편물이 분실되면 복구할 방법이 없어 "절대 분실되지 않는 특수 파이프(무손실 이더넷)"가 필요하다.
- **등장 배경 및 발전 과정**:
  1. **InfiniBand의 고립**: 전통적으로 슈퍼컴퓨터와 고성능 클러스터(HPC)는 RDMA 전용망인 인피니밴드(InfiniBand)를 사용했다. 하지만 인피니밴드는 스위치와 케이블이 독자 규격이고 너무 비싸서 일반 데이터센터에 도입하기 어려웠다.
  2. **RoCEv1의 등장 (2010)**: 이더넷의 속도가 10G/40G로 발전하면서, 범용 이더넷망에서 RDMA를 쓰려는 시도가 시작되었다. RoCEv1은 이더넷 링크 계층 위에서 바로 RDMA 패킷을 날렸으나, IP 라우팅이 불가능해 동일 서브넷(L2) 안에서만 통신할 수 있는 치명적 한계가 있었다.
  3. **RoCEv2의 완성 (2014)**: RDMA 패킷을 표준 UDP/IPv4(또는 IPv6) 패킷으로 감싸서(Encapsulation) 전송하는 RoCEv2 규격이 발표되었다. 이로써 L3 라우팅을 타고 데이터센터 전역으로 통신할 수 있게 되면서, 현대 AI 클러스터(예: NVIDIA SuperPOD 백엔드)의 사실상 표준 기술로 등극했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 / 특징 | 비유 |
|:---|:---|:---|:---|
| **RNIC (RDMA-enabled NIC)** | RoCE 프로토콜을 하드웨어로 처리하는 랜카드 | CPU 개입 없이 메모리 DMA, 패킷 캡슐화 수행 | 전담 초고속 우편집배원 |
| **Queue Pair (QP)** | 두 노드 간의 통신 채널 (Send Queue / Receive Queue) | 응용 프로그램이 OS 전용 우체통 (송신함/수신함) |
| **Zero-Copy** | 메모리 복사 생략 | 커널 버퍼로의 데이터 복사 없이 앱 메모리에서 NIC로 직행 | 창고에서 트럭으로 바로 상차 |
| **Kernel Bypass** | OS 커널의 TCP/IP 스택 우회 | 컨텍스트 스위칭과 인터럽트 폭풍 제거 (초저지연) | 검문소/세관 무사통과 |
| **Lossless Ethernet (DCB)** | RoCE 패킷이 스위치 버퍼 부족으로 버려지는 것을 막는 네트워크 설정 | PFC(Priority Flow Control)로 트래픽 혼잡 시 일시 정지(Pause) 신호 발송 | 고속도로 전용차로 톨게이트 제어 |

### TCP/IP vs RoCEv2 통신 스택 아키텍처 비교

RoCE의 핵심은 운영체제(OS)를 데이터 전송 경로에서 완전히 배제하는 데 있다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │              전통적 TCP/IP 스택 vs RoCEv2 스택 비교               │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 전통적 TCP/IP 통신 (CPU 병목 발생)                          │
  │                                                             │
  │  [Application] ── (소켓 API, Context Switch) ──▶ (유저 공간) │
  │        ▼                                                    │
  │  [OS Kernel] : 데이터를 커널 버퍼로 복사 (Memcpy)               │
  │        ▼       TCP 헤더 붙이기, IP 라우팅, 인터럽트 처리          │
  │  [NIC (랜카드)] ── (이더넷 프레임 전송) ─────────────────▶       │
  │                                                             │
  │ * 문제점: 100Gbps 대역폭 시, 이 과정을 처리하느라 CPU 코어 여러 개가 │
  │           100% 점유되고, 복사(Copy) 때문에 수십 µs 지연 발생.      │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] RoCEv2 통신 (Kernel Bypass & Zero-Copy)                   │
  │                                                             │
  │  [Application] ──── (Verbs API, 직접 하드웨어 큐 접근) ────┐    │
  │                                                        │    │
  │  [OS Kernel] : 연결 초기화(Control Plane) 때만 관여함.     │    │
  │                데이터 전송(Data Plane) 시 완전 무시됨!      │    │
  │                                                        │    │
  │  [RNIC (RoCE 랜카드)]: ◀───────────────────────────────────┘    │
  │          앱 메모리에서 DMA로 직접 데이터 긁어옴 (Zero-Copy)      │
  │          하드웨어 칩(ASIC)이 UDP/IP/BTH 헤더 순식간에 생성     │
  │          ── (RoCEv2 패킷 전송) ──────────────────────────▶   │
  │                                                             │
  │ * 장점: CPU 점유율 0% 수렴. 1~2 µs 수준의 극한의 초저지연 달성.     │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 TCP/IP 방식에서는 데이터를 보낼 때 유저 공간(User Space)의 앱 데이터가 커널 공간(Kernel Space)의 소켓 버퍼로 복사(Memory Copy)되어야 한다. 이 과정에서 필연적으로 수많은 컨텍스트 스위치와 CPU 연산이 낭비된다. 대역폭이 넓어질수록 CPU가 통신 처리에 압사당한다. 
반면 하단의 **RoCEv2** 방식에서는 앱이 OS를 거치지 않고 RNIC의 Queue Pair(하드웨어 레지스터)에 직접 지시를 내린다(Kernel Bypass). 지시를 받은 RNIC는 OS 몰래 앱의 물리 메모리에 직접 접근(DMA)하여 데이터를 퍼 오고(Zero-Copy), 자신의 칩 내부에서 하드웨어적으로 UDP/IP 헤더를 씌워 전송한다. 데이터를 받는 쪽 RNIC도 마찬가지로 도착한 패킷을 헤더만 떼고 즉시 수신 측 앱 메모리의 정확한 번지에 꽂아 넣는다. 이 모든 과정에 CPU가 개입하지 않아 극한의 속도가 나온다.

---

### RoCEv2 패킷 캡슐화 구조
RoCEv2는 이더넷 장비(스위치/라우터)가 볼 때는 평범한 UDP 패킷처럼 생겨서 라우팅이 가능하다.
* **[ Ethernet Header ]** + **[ IP Header ]** + **[ UDP Header (포트 4791) ]** + **[ IB BTH (InfiniBand Base Transport Header) ]** + **[ 실제 Payload (데이터) ]** + **[ ICRC / FCS ]**
스위치는 앞단의 IP/UDP만 보고 목적지로 넘겨주며, 최종 목적지 RNIC만이 뒤에 숨은 IB BTH 헤더를 해석해 메모리 번지를 파악한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 3대 RDMA 프로토콜 생태계 비교

| 비교 항목 | InfiniBand (IB) | RoCEv2 (RDMA over Ethernet) | iWARP (Internet Wide Area RDMA) |
|:---|:---|:---|:---|
| **기반 네트워크** | 전용 IB 스위치 및 케이블 (폐쇄적) | **범용 Ethernet 스위치** (IP/UDP 라우팅) | 범용 Ethernet 스위치 (TCP/IP 기반) |
| **패킷 신뢰성 보장** | 링크 계층에서 완벽한 무손실(Lossless) | **DCB 스위치 설정(PFC)으로 무손실 강제** | TCP 자체가 재전송/흐름 제어를 지원 |
| **성능 (Latency)** | **1위 (가장 낮음, ~0.5µs)** | **2위 (거의 유사함, ~1µs)** | 3위 (TCP 오버헤드로 상대적 느림) |
| **비용 / 유연성** | 매우 비쌈 / 벤더 락인 심함 | **저렴함 / 기존 이더넷 인프라 100% 재활용** | 중간 (하드웨어 TCP 오프로드 엔진 필요) |
| **주요 진영** | NVIDIA(Mellanox), 슈퍼컴퓨터 | **AI 클러스터, 하이퍼스케일러(클라우드)** | Intel, 일부 스토리지 벤더 |

### 과목 융합 관점
- **네트워크 공학 (PFC와 Head-of-Line Blocking)**: RoCEv2는 기본적으로 패킷이 버려지면(Drop) 처음부터 끝까지 다시 보내야 하는 치명적 약점(Go-Back-N)이 있다. 이를 막기 위해 L2 스위치 단에서 버퍼가 차면 앞 노드에게 "잠깐 멈춰(Pause)!"라는 **PFC(Priority Flow Control)** 프레임을 보낸다. 하지만 이 PFC가 연쇄적으로 발생하면 네트워크 전체가 멈춰버리는 끔찍한 **PFC Storm (또는 Pause Storm)** 현상이 발생할 수 있어, 클라우드 엔지니어들에게 악명이 높다.
- **클라우드 스토리지 (NVMe-oF)**: 최신 올플래시(All-Flash) 스토리지 시스템은 로컬 PCIe 버스에 꽂힌 NVMe SSD와 똑같은 속도를 네트워크 너머에서도 제공하기 위해 **NVMe over Fabric (NVMe-oF)** 기술을 쓴다. 이때 하부 수송망(Transport)으로 압도적인 비율로 선택받는 프로토콜이 바로 RoCEv2다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — GPU 간 초고속 파라미터 동기화 (GPU Direct RDMA)**: 1,000대의 서버에 8,000개의 GPU가 꽂힌 AI 훈련 클러스터. 노드 A의 GPU 메모리에 있는 텐서(Tensor) 데이터를 노드 B의 GPU 메모리로 보내야 한다.
   - **기존 방식**: GPU A → CPU A 메모리 복사 → OS 커널 버퍼 → NIC A 송신 ──▶ NIC B 수신 → 커널 버퍼 → CPU B 메모리 → GPU B (수백 마이크로초 지연, GPU가 굶음)
   - **RoCE 적용 (NVIDIA GPUDirect RDMA)**: RoCE 기술을 활용해 NIC A가 GPU A의 VRAM에서 직접(DMA) 데이터를 퍼서, 네트워크를 타고 NIC B로 간 뒤, GPU B의 VRAM에 직접 꽂아버린다. 시스템 메모리와 CPU를 완벽히 패스(Bypass)하여 마이크로초 단위 동기화가 가능해져 AI 모델 훈련 시간이 절반으로 단축된다.
2. **시나리오 — RoCEv2 망의 PFC Storm으로 인한 클러스터 장애 (안티패턴)**: 일반적인 이더넷 스위치 환경에서 RoCEv2를 무작정 도입했다가 트래픽 폭주 시 데이터센터 전체 네트워크가 5초간 마비(Freeze)되는 사태 발생.
   - **해결책**: RoCEv2를 안전하게 쓰려면 **Lossless Ethernet** 설계가 필수다. 스위치에서 ECN(Explicit Congestion Notification) 임계치를 엄격하게 튜닝하여 큐(Queue)가 차기 전에 미리 송신자 측 RNIC가 전송 속도를 줄이도록(DCQCN 알고리즘) 만들어, 최후의 보루인 PFC(물리적 멈춤)가 터지는 상황 자체를 원천 차단하는 고급 네트워크 엔지니어링이 필요하다.

### 도입 체크리스트
- **기술적**: 사내에 도입된 Top-of-Rack(ToR) 및 Spine 스위치가 DCB(Data Center Bridging), PFC, ECN 기능을 완벽하게 지원하는 엔터프라이즈급 장비인가? (값싼 일반 스위치에서 RoCEv2를 돌리면 잦은 패킷 로스로 인해 TCP보다 오히려 성능이 떨어진다.)
- **운영적**: 애플리케이션(예: 분산 DB, MPI 앱)이 기존 BSD 소켓 API(`send()`, `recv()`)가 아닌, RDMA 전용 라이브러리인 **libibverbs** API 또는 통신 추상화 계층(UCX)을 사용하도록 코드가 개발/컴파일되어 있는가? (RoCE는 소켓 코드로는 혜택을 볼 수 없다.)

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 범용 100G 이더넷 (TCP/IP) | 100G RoCEv2 도입 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (지연 시간, Ping)** | 약 30 ~ 50 µs | 약 1 ~ 3 µs | 네트워크 지연 시간 **1/10 이하로 단축** |
| **정량 (CPU 점유율)** | 100Gbps 전송 시 CPU 10~15코어 점유 | 100Gbps 전송 시 CPU 0~1코어 미만 점유 | I/O Tax 0%로 호스트 연산 자원 100% 확보 |
| **정성 (인프라 비용)** | HPC용 InfiniBand 별도 구축 필요 (이중망) | 기존 Ethernet망 통합 (Converged) 관리 | 막대한 라이선스/스위치/케이블 비용 절감 |

### 미래 전망
- **Lossy 망에서의 RoCE 한계 극복 시도**: 깐깐한 PFC/ECN 설정(무손실 망 구성)에 지친 업계는, 패킷이 약간 버려지는 일반(Lossy) 이더넷 환경에서도 자체적인 초고속 재전송과 혼잡 제어(Congestion Control) 알고리즘을 강화하여 안정적인 성능을 내는 **차세대 차세대 RoCE 혼잡제어 알고리즘(HPCC 등)** 연구에 박차를 가하고 있다.
- **Ultra Ethernet Consortium (UEC)의 등장**: 인피니밴드 독점(NVIDIA)에 대항하기 위해 인텔, AMD, 메타, 마이크로소프트 등이 모여 RoCEv2의 단점을 보완하고 AI 워크로드에 최적화된 궁극의 이더넷 표준 규격인 **UEC(울트라 이더넷)** 제정을 선포했으며, 이는 향후 RoCE를 대체하거나 흡수할 강력한 표준이 될 것이다.

### 참고 표준
- **InfiniBand Trade Association (IBTA)**: 인피니밴드뿐만 아니라 RoCEv1 및 RoCEv2(Annex A17) 표준 규격을 제정하고 관리하는 글로벌 산업 기구.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[소프트웨어 정의 인프라 하드웨어 종속성 (SDI Hardware)](./499_sdi_hardware.md)**: CPU가 짊어지던 막대한 I/O(네트워크) 부하를 하드웨어로 떠넘기는 거대한 흐름. RNIC와 RoCE는 이 오프로딩(Offloading) 철학의 완벽한 실현이다.
- **[메모리 월 (Memory Wall)](./500_von_neumann_bottleneck.md)**: 폰 노이만 병목을 극복하기 위해 로컬 메모리뿐 아니라 네트워크 너머의 원격 메모리까지 내 것처럼 빠르게(Zero-Copy) 당겨오는 RoCE의 존재 목적.
- **[NVLink / NVSwitch](../12_accelerators_ai_hardware/444_nvlink.md)**: 노드(서버) 내부의 GPU 간 통신은 NVLink가 담당하고, 노드와 노드 사이(Rack-to-Rack)의 스케일 아웃(Scale-out) 통신은 RoCEv2(또는 인피니밴드)가 담당하는 역할 분담이 이루어지고 있다.
- **[DPU / SmartNIC](../12_accelerators_ai_hardware/436_dpu.md)**: RoCEv2 프로토콜 엔진을 하드웨어 칩(ASIC)으로 탑재하여 서버에 꽂히는 차세대 지능형 랜카드.

## 👶 어린이를 위한 3줄 비유 설명
1. 일반 택배(TCP/IP)로 친구에게 물건을 보내려면, 우체국에 가서 박스를 사고 주소를 적고 직원이 분류할 때까지 한참(CPU 일거리) 기다려야 해요.
2. 그런데 **RoCE**라는 마법의 파이프를 내 방에서 친구 방으로 연결하면, 내가 물건을 파이프에 쑥 넣자마자 1초 만에 친구 책상(메모리) 위에 물건이 뚝 떨어져요! 우체국을 안 가도 되죠. (Zero-Copy & Kernel Bypass)
3. 단, 이 파이프는 중간에 막히면 물건이 몽땅 날아가 버릴 수 있어서, 파이프 길목마다 교통정리를 기가 막히게 잘해주는 특수 신호등(DCB 스위치)이 꼭 필요하답니다.
