## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RPO는 재해 발생 시 허용 가능한 최대 데이터 유실 시점을 뜻하는 지표로, "어디 시점까지 데이터를 보존해야 하는가"를 숫자로 정의한다.
> 2. **가치**: RPO가 짧을수록 백업·복제 비용과 운영 복잡도는 커지지만, 장애 시 비즈니스 손실과 데이터 재입력 비용은 급격히 줄어든다.
> 3. **융합**: 스토리지 미러링, 데이터베이스 로그 전송, 애플리케이션 체크포인트, 네트워크 대역폭 설계가 모두 RPO 달성 가능성을 결정한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: RPO (Recovery Point Objective)는 시스템 장애나 재해가 발생했을 때 복구 가능한 데이터 기준 시점과 실제 장애 시점 사이의 허용 가능한 최대 간격이다. 즉, "최대 몇 분치 데이터를 잃어도 되는가"를 정하는 지표다.
- **💡 비유**: 10분마다 자동 저장되는 문서를 쓰다가 정전이 나면 최대 10분 전 상태까지만 복구되는 것과 같다.
- **등장 배경**:
  1. 백업은 했지만 실제로 어느 정도 데이터가 사라질 수 있는지 설명하지 못하는 문제가 있었다.
  2. 서비스마다 데이터 유실 허용치가 크게 달라, 복구 목표를 숫자로 계약화할 필요가 생겼다.
  3. 클라우드·분산 시스템 환경에서는 복제 지연이 항상 존재하므로 허용 오차 정의가 필수다.
- **📢 섹션 요약 비유**: 정전이 나더라도 숙제가 몇 분 전까지는 반드시 남아 있어야 하는지 정하는 안전 기준선과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Source Data** | 원본 업무 데이터 | 트랜잭션 발생 | DB, File System | 현재 작성 중인 원본 문서 |
| **Protection Layer** | 보존 수단 | 복제·백업·스냅샷 수행 | Sync/Async Mirror, Backup | 자동 저장 기능 |
| **Recovery Copy** | 복구 사본 | 특정 시점 상태 유지 | Snapshot, Replica | 저장된 복사본 |
| **Lag Monitor** | 시차 측정 | 복제 지연·백업 주기 감시 | Replication Lag | 시계 차이 측정기 |
| **Policy Engine** | 목표 관리 | 허용 RPO 초과 시 경보 | SLA, Monitoring | 규칙 관리자 |

```text
┌──────────────────────────────────────────────────────────────┐
│                 RPO 시간 개념                                │
├──────────────────────────────────────────────────────────────┤
│ Recovery Copy Time ------------------ Failure Time          │
│                <------ Data Gap ------>                      │
│                      = RPO Actual                           │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RPO는 장애 복구 시점과 장애 발생 시점 사이의 공백 길이다. 복구 사본이 최신에 가까울수록 데이터 공백은 작아진다. 동기식 미러링은 이 공백을 거의 0으로 줄이지만, 비동기식 복제나 주기 백업은 전송 간격 또는 백업 주기만큼 공백이 남는다.

```text
┌──────────────────────────────────────────────────────────────┐
│      보호 방식별 일반적 RPO 범위                             │
├──────────────────────────────────────────────────────────────┤
│ 동기식 미러링      : 0 ~ 수초 미만                            │
│ 비동기식 미러링    : 수초 ~ 수분                              │
│ 로그 전송/주기 스냅샷: 수분 ~ 수십분                          │
│ 일 배치 백업       : 수시간 ~ 1일                             │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RPO는 기술 선택의 결과물이다. 기술사는 데이터 중요도와 예산을 함께 보고 적정 목표를 정해야 한다. 결제 원장은 수초도 길 수 있지만, 사내 게시판은 수십 분 손실이 허용될 수 있다. 따라서 모든 시스템에 `RPO=0`을 요구하는 것은 과도한 설계일 수 있다.

1. 업무 중요도에 따라 허용 가능한 데이터 유실 시간을 정의한다.
2. 해당 목표를 만족하는 보호 기술(백업/복제/로그 전송)을 선택한다.
3. 실제 복제 지연과 백업 성공 여부를 지속 감시한다.
4. 장애 발생 시 복구본 시점을 확인해 실제 RPO를 측정한다.
5. 목표와 실제 값 차이가 크면 아키텍처를 재설계한다.

- **📢 섹션 요약 비유**: 자동 저장 주기를 짧게 할수록 잃어버릴 숙제 양이 줄지만, 저장 장치와 시스템은 더 바쁘게 돌아가는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

| 방식 | 일반적 RPO | 장점 | 한계 |
|:---|:---|:---|:---|
| **동기식 미러링** | 0에 근접 | 데이터 유실 최소화 | 지연·거리 제약 |
| **비동기식 미러링** | 수초~수분 | 장거리 DR 가능 | 복제 시차 존재 |
| **스냅샷** | 주기 단위 | 운영 부담 비교적 낮음 | 스냅샷 간 공백 |
| **백업** | 시간~일 단위 | 장기 보관, 비용 효율 | 복구 시점 오래됨 |

| 관점 | RPO | RTO |
|:---|:---|:---|
| **질문** | 얼마만큼의 데이터를 잃어도 되는가 | 얼마나 빨리 다시 살아나야 하는가 |
| **중심 자산** | 데이터 시점 | 서비스 가동 시간 |
| **주요 기술** | 복제, 백업, 로그 | 자동 전환, 오케스트레이션 |

- **과목 융합 관점**:
  1. **데이터베이스**: 트랜잭션 로그 전달 주기가 곧 DB 계층 RPO를 좌우한다.
  2. **IT 경영**: 업무 영향도 분석(BIA)이 수치화되지 않으면 RPO 값은 기술팀 추정에 머무른다.
- **📢 섹션 요약 비유**: 잃어버린 물건의 양을 정하는 기준과, 다시 일상으로 돌아오는 속도를 정하는 기준은 서로 다르다는 점을 분리해 봐야 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **시나리오 1: 인터넷 뱅킹**
  - 판단: 거래 데이터는 사실상 `RPO=0`이 요구되므로 동기 복제와 트랜잭션 로그 이중화가 필요하다.
- **시나리오 2: 그룹웨어**
  - 판단: 수분 단위 유실은 허용 가능해 비동기 복제와 주기 스냅샷 조합이 현실적이다.
- **시나리오 3: 분석 시스템**
  - 판단: 원본 데이터가 별도 저장돼 있다면 수시간 단위 배치 백업도 가능하다.

체크리스트:

1. 업무별 허용 데이터 유실량이 정의되었는가.
2. 실제 복제 지연이 목표 RPO 이내로 유지되는가.
3. 랜섬웨어와 논리 오류까지 고려한 복구본이 존재하는가.
4. 목표 위반 시 알림과 조치 절차가 있는가.

- **안티패턴**: 회선이 불안정한데도 문서상으로만 `RPO=0`을 선언하면, 실제 재해 때 목표 위반 사실만 확인하게 된다.
- **📢 섹션 요약 비유**: "숙제를 절대 잃지 않는다"라고 말하는 것보다, 실제로 몇 초마다 저장되고 있는지를 보는 것이 더 중요합니다.

| **복구 전략 명확화** | 시스템별 보호 수준 차등 설계 가능 |

  1. AI 기반 예측 복제가 복제 지연을 선제 제어할 가능성이 높다.
  2. 멀티클라우드 복제에서 서비스별 차등 RPO 정책이 세밀해진다.
