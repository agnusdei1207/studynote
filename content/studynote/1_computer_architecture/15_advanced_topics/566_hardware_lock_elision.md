> **핵심 인사이트**
> 1. 멀티스레딩의 필수품인 **락(Lock / Mutex)**은 공유 데이터를 보호하지만, 다른 스레드들을 무작정 대기하게 만들어 멀티코어 성능을 끔찍하게 저하시키는 가장 큰 병목이다.
> 2. **HLE (Hardware Lock Elision, 하드웨어 락 무시/생략)**는 인텔 TSX 기술의 핵심으로, 개발자가 짜놓은 평범한 락 코드를 CPU가 실행할 때 **"일단 락을 걸지 않은 척 무시하고" 다 같이 동시에 캐시에서 계산부터 해버리는 기법**이다.
> 3. 만약 기적적으로 데이터 충돌이 없으면 락 없이 고속 통과한 셈이고, 충돌이 발생하면 즉시 계산을 버리고(Rollback) 원래의 느린 락 방식으로 되돌아가는 하드웨어 기반의 낙관적 동시성 제어(Optimistic Concurrency)다.

<br>

## Ⅰ. 보수적인 소프트웨어 락(Lock)의 낭비

프로그래머들은 충돌이 두려워서 매우 보수적으로 락을 겁니다.
예를 들어, 수만 칸짜리 거대한 '해시 테이블(Hash Table)'에 스레드 A와 B가 데이터를 넣는다고 합시다.

```c
lock(hash_table_lock);    // 1. 자물쇠 잠금! B는 여기서 멈춰야 함.
insert(hash_table, data); // 2. A가 데이터를 넣음.
unlock(hash_table_lock);  // 3. 자물쇠 풂. 이제 B가 진입 가능.
```

스레드 A는 해시 테이블의 '1번 칸'을 건드리고, 스레드 B는 '9000번 칸'을 건드립니다. 서로 전혀 다른 곳을 건드리는데도 불구하고, '해시 테이블'이라는 커다란 전체 자물쇠를 하나로 묶어 잠갔기 때문에 B는 A가 끝날 때까지 멍청하게 기다려야 합니다. 이는 막대한 병렬 성능의 낭비입니다.

> 📢 **섹션 요약 비유**: 수만 권의 책이 있는 거대한 도서관(해시 테이블)입니다. 손님 A가 1층 역사 코너에서 책을 찾는다고, 도서관 정문 전체를 자물쇠로 잠가버립니다(Lock). 3층 과학 코너에 갈 손님 B는 아무 상관도 없는데 밖에서 벌벌 떨며 기다려야 하는 바보 같은 상황입니다.

## Ⅱ. HLE의 동작: "자물쇠를 투명하게 만들자"

CPU 하드웨어 설계자들은 이 낭비를 참을 수 없었습니다. 그래서 CPU 내부에 **HLE (Hardware Lock Elision)**라는 기능을 넣었습니다 (Intel TSX 등).

HLE 명령어 접두사(`XACQUIRE`, `XRELEASE`)를 붙인 락 코드를 만나면 CPU는 다음과 같이 미친 짓을 벌입니다.

1. **락 생략 (Lock Elision)**: CPU는 실제 메모리에 있는 자물쇠 변수(Mutex)를 `1(잠김)`로 바꾸지 않습니다. 그냥 읽기만 하고 지나가며 **자물쇠가 없는 척(Elision)**을 합니다.
2. **트랜잭션 시작 (낙관적 실행)**: 스레드 A와 B가 동시에 도서관 문을 부수고 들어갑니다. 각자 캐시(L1) 안에서 자기만의 계산(A는 1번 칸, B는 9000번 칸)을 미친 듯이 병렬로 수행합니다.
3. **하드웨어 충돌 감시**: MESI 캐시 프로토콜이 매의 눈으로 감시합니다. "A가 1번 칸 쓴다! B가 9000번 칸 쓴다! 오케이, 안 겹치네!"
4. **성공 (Commit)**: 안 겹쳤으므로 락 없이 완벽히 동시 처리에 성공했습니다! 계산 결과를 진짜 메모리에 조용히 밀어 넣고 끝냅니다. 성능은 2배가 됩니다.

### 만약 겹쳤다면? (Abort & Fallback)
운 나쁘게 A와 B 둘 다 '1번 칸'을 건드리려 했다면, 하드웨어가 "삐빅! 충돌 발생!"을 외칩니다.
즉시 캐시에서 하던 계산을 다 쓰레기통에 버리고(Rollback), 이번에는 **진짜로 보수적인 자물쇠(Lock)를 걸고 한 명씩 들어가게 원상 복구(Fallback)**시킵니다.

> 📢 **섹션 요약 비유**: 경비원(하드웨어 HLE)이 도서관 정문을 활짝 열어버립니다(락 생략). 수백 명의 손님이 일단 다 뛰어 들어가서 책을 찾습니다. 운 좋게 다들 다른 책을 찾으면 초고속으로 대출이 끝납니다. 하지만 두 명이 똑같은 책을 놓고 싸우려 들면, 경비원이 시간을 되돌려(Rollback) 멱살을 잡고 문밖으로 끌어낸 뒤 한 줄로 얌전히 세워서 한 명씩 들여보냅니다.

## Ⅲ. HLE의 가치: 레거시 코드의 구원

HLE의 가장 위대한 점은 **"개발자가 10년 전에 짜놓은 옛날 코드(Legacy Code)를 한 줄도 안 고치고도 성능을 2배, 3배 올려줄 수 있다"**는 점입니다.

개발자가 피눈물을 흘리며 세밀한 락(Fine-grained Lock)을 짤 필요가 없습니다. 그냥 옛날처럼 커다란 무식한 락(Coarse-grained Lock)을 하나 던져놔도, 최신 CPU가 그 코드를 읽어들일 때 하드웨어적으로 **자동으로 락을 무시하고 트랜잭션 메모리(HTM)로 변환해 버리기 때문**입니다. 소프트웨어의 게으름을 하드웨어의 압도적 지능으로 멱살 잡고 캐리하는 셈입니다.
