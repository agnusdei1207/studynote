# TMR (Triple Modular Redundancy, 삼중 모듈 중복)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**= **N-Modular Redundancy (NMR, 다중 모듈 중복)** 기법의 일종으로, 동일한 기능을 수행하는 3개의 독립된 하드웨어 모듈과 **다수결 투표기 (Voter, 과반수 투표자)**를 결합하여 단일 결함(Single Fault)을 실시간으로 마스킹(Masking)하는 하드웨어적 결함 허용 기술입니다.
> 2. **가치**= 소프트웨어 개입 없이 하드웨어 레벨에서 순간적으로 오류를 정정하므로, **RTO (Recovery Time Objective, 복구 시간 목표)**가 '0'에 수렴하는 극한의 신뢰성을 제공하며, 이는 SAF (Safety Availability Factor, 안전성 가용률) 99.999% 이상을 요구하는 시스템의 핵심 아키텍처입니다.
> 3. **융합**= 방사선 내성을 강화해야 하는 항공우주(Aerospace) 분야와 ML (Machine Learning, 기계학습) 추론 가속기 등 고성능 컴퓨팅에서의 SEU (Single Event Upset, 단일 사태 사건) 대응, 그리고 CLB (Configurable Logic Block) 설계 등 FPGA/ASIC 하드웨어 설계 전반에 필수적인 패러다임입니다.

### Ⅰ. 개요 (Context & Background)

**TMR (Triple Modular Redundancy)**는 임계 시스템(Critical System)의 신뢰성을 보장하기 위해 **하드웨어 중복성 (Hardware Redundancy)**을 극대화한 설계 기법입니다. 기존의 단일 시스템이나 이중화(Duplex) 시스템은 오류 발생 시 이를 탐지(Detect)하고 복구(Recover)하는 과정에서 필연적으로 지연(Latency)이 발생하거나 시스템 중단(Downtime)이 불가피합니다. 이를 해결하기 위해 TMR은 3개의 모듈이 동일한 연산을 수행하게 하고, **Voter (투표기)**가 그 결과를 실시간으로 비교하여 '다수결의 원칙(Majority Rule)'에 따라 최종 출력을 결정합니다. 이를 통해 시스템은 하나의 모듈에 고장(Fault)이 발생하더라도 외부적으로는 오류가 없었던 것처럼 지속적인 서비스를 제공할 수 있습니다. 이 기술은 특히 우주 방사선으로 인한 비트 반전(Bit Flip) 같은 소프트 에러(Soft Error)에 취약한 최신 반도체 공정에서 그 중요성이 더욱 부각되고 있습니다.

> **💡 개요를 위한 비유**
> 중요한 안건을 결정할 때, 단일 국왕(단일 모듈)의 결정은 그날의 컨디션에 따라 달라질 수 있고, 두 사람(이중화)의 의견이 충돌하면 판단이 늦어집니다. 하지만 세 명의 판사(TMR)가 동시에 판결을 내려 2명 이상이 동의한 안건만 집행한다면, 한 판사가 실수를 하거나 납득할 수 없는 의견을 내더라도 올바른 판결이 즉시 집행되어 업무가 중단되지 않습니다.

#### 등장 배경 및 필요성
1.  **기존 한계**: 초기 컴퓨팅 시스템은 ECC (Error Correction Code, 오류 정정 부호) 메모리나 예비 스위칭(Standby Switching) 방식을 주로 사용했으나, 이는 오류 발생 후 복구하는 '수동적' 방식으로 미세초(µs) 단위의 중단도 허용하지 않는 미션 크리티컬(Mission Critical) 분야에는 부족했습니다.
2.  **혁신적 패러다임**: TMR은 "예방이 치료보다 낫다"는 철학을 하드웨어에 적용하여, 오류가 발생하자마자(Zero Latency) 격리시키는 능동형 마스킹(Active Masking) 기술을 도입했습니다.
3.  **현재 비즈니스 요구**: 자율 주행, AI 추론 서버, 5G/6G 네트워크 장비 등 서비스 중단이 곧 생명 또는 막대한 금전적 손실로 이어지는 환경에서 **FIT (Failure in Time)**률을 획기적으로 낮추기 위한 필수 기술로 자리 잡았습니다.

> **📢 섹션 요약 비유**:
> 300km/h로 달리는 고속열차의 제어 시스템은 "브레이크를 잠시 놨다가 다시 잡는" 것을 허용하지 않습니다. 따라서 운전자 1명이 아닌 3명의 기관사가 동시에 브레이크를 보고, 2명 이상이 "멈춰라"라고 외칠 때만 제동이 걸리도록 설계하여, 한 명이 졸거나 판단을 착각해도 열차는 안전하게 정차할 수 있는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

TMR의 핵심은 **공통 모드 실패 (Common Mode Failure, 동시 다발 오류)**만 방지할 수 있다면, 독립적인 하드웨어 오류에 대해서는 100%에 가까운 내성을 가진다는 수학적 확신에 기초합니다.

#### 1. 구성 요소 상세 분석
TMR 시스템은 크게 **입력 분배부 (Input Splitter)**, **3개의 중복 모듈 (Replicated Modules)**, **다수결 투표기 (Majority Voter)**로 구성됩니다.

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 (Internal Operation) | 주요 프로토콜/회로 | 비유 (Analogy) |
|:---|:---|:---|:---|:---|
| **Input Splitter** | 신호 복제 | 단일 입력 소스(클럭, 데이터)를 3개의 모듈로 동기화하여 분배 (Clock Tree Splitting) | Clock Buffer, H-Tree | 시간 배달부 (편지 3통 복사) |
| **Functional Modules** | 독립 연산 수행 | 동일한 명령어(Instruction)를 인가받아 각자의 ALU/CPU로 연산 수행 | Isolated Power Domain | 3명의 독립적인 회계사 |
| **Majority Voter** | 결과 판정 및 출력 | $V = M_1M_2 + M_2M_3 + M_3M_1$ 논리식을 통해 2개 이상 일치 시 출력 | TTL/CMOS Logic, XOR Tree | 최종 재판장 (다수결 승인) |

#### 2. 아키텍처 및 데이터 흐름 (ASCII Diagram)

아래 다이어그램은 TMR의 동적 데이터 흐름과 특정 모듈(Module 2)에서 오류가 발생했을 때의 마스킹 과정을 도식화한 것입니다.

```ascii
       [Clock Source] & [Data Input]
              |  (Replicated Fan-out)
              V
    +-----------------------+
    |   Input Replicator    |
    +-----------------------+
         |          |          |
         V          V          V
  +----------+ +----------+ +----------+
  | Module 1 | | Module 2 | | Module 3 |
  | (Logic)  | | (Logic)  | | (Logic)  |
  +----------+ +----------+ +----------+
    | Out A     | Out B     | Out C
    | (Normal)  | (ERROR!)  | (Normal)
    |           |           |
    |  1        |  0        |  1   <-- 예: 단일 비트 반전 오류 발생
    |           |           |
    +-----+-----+-----+-----+-----+
                  |
           +------|------|------+
           |  +---|------|---+  |
           |  |  Majority     |  |
           |  |  Voter Logic  |  |
           |  |  (2/3 Logic)  |  |
           |  +-------+-------+  |
           |          |          |
           |          V          |
           |     [ Output A ]    |
           |     [ Output C ]    |
           |          |          |
           +----------+----------+
                      |
              Final Correct Output (1)
        (System tolerates Module 2 error)
```

**[다이어그램 해설]**
1.  **입력 복제 (Replication)**: 외부 입력 신호는 `Input Replicator`를 통해 3개 모듈에 완전히 동일한 타이밍으로 전달됩니다. 이때 스큐(Skew)를 최소화하는 것이 중요합니다.
2.  **독립적 오류 발생 (Fault Injection)**: 위 예시에서 `Module 2`가 방사선 노출 등으로 인해 잘못된 값 `0`을 출력했습니다. 이때 `Module 1`과 `Module 3`은 정상적인 값 `1`을 출력합니다.
3.  **다수결 투표 (Voting)**: Voter는 `(M1 && M2) || (M2 && M3) || (M1 && M3)` 로직을 수행합니다. 이 경우 `(1 && 0)`은 False, `(0 && 1)`은 False이지만, `(1 && 1)`은 True이므로 최종적으로 `1`을 출력합니다. 즉, Module 2의 오류는 즉시 마스킹(Masking)되어 시스템 출력에 영향을 주지 않습니다.

#### 3. 핵심 알고리즘 및 논리 회로
다수결 투표기는 디지털 논리 회로로 구현되며, 가장 효율적인 TMR Voter의 논리 식은 다음과 같습니다. FPGA의 LUT (Look-Up Table)나 ASIC의 표준 셀(Standard Cell)로 합성됩니다.

**논리 공식**:
$$ Output = (M_1 \cdot M_2) + (M_2 \cdot M_3) + (M_1 \cdot M_3) $$

```verilog
// Verilog Snippet: TMR Voter Module (Gate Level Description)
module tmr_voter (
    input wire m1,
    input wire m2,
    input wire m3,
    output wire valid_out
);
    // 3개 입력 중 2개 이상이 1이면 출력 1
    assign valid_out = (m1 & m2) | (m2 & m3) | (m1 & m3);
endmodule
```
위 코드는 하드웨어 기술 언어(HDL)로 구현된 가장 기본적인 Voter입니다. 실제 시스템에서는 이 Voter 회로 자체가 **SPOF (Single Point of Failure, 단일 장애점)**가 되지 않도록 Voter 또한 3중화(Triplicated Voter)하여 루프 형태로 연결하기도 합니다.

> **📢 섹션 요약 비유**:
> 3명의 사격수가 과녁을 맞히는 대회를 상상해보세요. 한 명이 실수로 배경을 맞히거나 총기가 고장 나서 오사(誤射)를 하더라도, 나머지 두 명이 정확히 과녁 중심을 맞혔다면, 심판 시스템(Voter)은 "과녁 적중"으로 판정하여 게임을 계속 진행시킵니다. 관중은 누가 실수했는지조차 모르고 경기를 즐기게 됩니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

TMR은 만능이 아닙니다. 여타 신뢰성 기술들과의 정량적 비교와 융합 관점을 분석합니다.

#### 1. 심층 기술 비교 분석표
TMR을 다른 결함 허용 기술과 비교하여, 엔지니어가 아키텍처를 선택할 때 활용할 수 있는 의사결정 매트릭스를 제공합니다.

| 비교 항목 | TMR (Triple Modular Redundancy) | Dual Modular Redundancy (DMR) | ERC (Error Correction Code) | Watchdog Timer |
|:---|:---|:---|:---|:---|
| **하드웨어 오버헤드** | 매우 큼 (약 3배 이상의 Area/Power) | 중간 (2배) | 작음 (약 5~15%의 Parity Bit) | 매우 작음 |
| **오류 정정 방식** | **마스킹 (Masking)** - 즉시 복구 | **탐지 (Detection)** - SW 복구 필요 | **복호 (Decoding)** - 비트 단위 복구 | **재시작 (Reset)** - 시스템 리부팅 |
| **Latency (지연시간)** | **0 (Zero Latency)** | High (복구 루틴 호출 시간) | Low (Decoder 지연) | Max (재부팅 시간) |
| **대처 가능 오류** | 단일 모듈의 완전한 고장 (Hard/Soft Error) | 단일 모듈 오류 탐지만 가능 | 정의된 비트 수 내의 오류 | 프로세서 Hang-up |
| **주요 적용 분야** | 항공, 원전, 자율주행, FPGA 핵심 로직 | 서버 RAID 1 (디스크 중복) | 메모리 시스템, SSD, 통신 | 임베디드 시스템 |

#### 2. 타 과목 융합 분석

**A. 운영체제(OS) 및 소프트웨어 관점**
TMR은 소프트웨어 계층(OS Kernel)에 투명(Transparent)하게 작동하는 경우가 많습니다. 하지만 **Software Implemented Fault Tolerance (SIFT)**와 결합하여, 하드웨어 TMR 내에서 복구 불가능한 2중 오류 발생 시 OS의 예외 핸들러(Exception Handler)가 이를 감지하고 안전하게 종료(Graceful Shutdown)하는 계층간 협력이 필요합니다.

**B. 네트워크 및 분산 시스템**
네트워크 패킷 스위칭 시스템에서 TMR은 스위칭 패브릭(Fabric) 카드에 적용됩니다. 하나의 라인 카드가 고장 나면 트래픽이 우회되는 방식이 아니라, 트래픽을 처리하는 **FPGA 내부 로직** 자체가 3중으로 돌아가고 있으므로, 네트워크 **RTO (Recovery Time Objective)**가 0에 수렴하게 되어 5G 통신망의 고신뢰성 요건을 충족합니다.

**C. 컴퓨터 구조 및 회로 설계**
최신 미세 공정(FinFET 등)에서는 **SER (Soft Error Rate, 연성 오류율)**가 급증합니다. 컴퓨터 구조 설계자는 파이프라인(Pipeline)의 각 스테이지에 TMR을 적용할지, 아니면 성능 저하를 감수하고 체크포인트(Checkpoint)를 할지를 결정해야 합니다. 주로 연산기(ALU)와 레지스터 파일(Register File)에 TMR을 적용하여 성능 저하를 최소화하면서 신뢰성을 확보합니다.

> **📢 섹션 요약 비유**:
> 암벽 등반(시스템 운영)을 할 때, 안전벨트(Watchdog)를 차는 것은 추락 후 병원에 가는 것(복구)이고, 요원 2명이 서로 확인하는 것(DMR)은 위험 상황을 알리는 것(탐지)입니다. 하지만 TMR은 로봇 팔
