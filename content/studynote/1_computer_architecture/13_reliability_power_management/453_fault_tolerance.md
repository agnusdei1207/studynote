## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 고장 허용 시스템 (Fault Tolerance, FT)은 하드웨어 또는 소프트웨어의 일부에 결함 (Fault)이 발생하더라도 시스템 전체의 서비스 중단이나 데이터 유실 없이 정상적인 기능을 지속적으로 수행할 수 있는 설계 능력을 의미한다.
> 2. **가치**: 미션 크리티컬 (Mission Critical) 및 라이프 크리티컬 (Life Critical) 환경에서 "실패는 옵션이 아니다"라는 전제하에, 오류를 감지하고 격리하며 중복 자원을 통해 즉각적으로 오류를 은폐 (Masking)함으로써 무정지 운영을 실현한다.
> 3. **융합**: 현대 아키텍처는 하드웨어 수준의 TMR (Triple Modular Redundancy)부터 소프트웨어 수준의 분산 합의 알고리즘 (Paxos, Raft)까지 결합하여, 개별 요소의 불완전함을 시스템 전체의 완전함으로 승화시키고 있다.

## Ⅰ. 개요 (Context & Background)

- **개념**: 고장 허용 시스템 (Fault Tolerance)은 시스템 내부의 구성 요소가 고장 나더라도 시스템 전체가 올바른 결과를 출력하도록 보장하는 기술이다. 단순히 고장을 빨리 고치는 것이 아니라, 고장 상황 자체를 시스템 내에서 '허용'하고 흡수하여 외부에서는 고장이 발생했는지조차 모르게 만드는 것이 핵심이다.

- **💡 비유**: FT는 "네 개의 엔진을 가진 대형 여객기"와 같다. 비행 중 엔진 하나가 꺼지더라도 남은 엔진들로 목적지까지 안전하게 비행을 계속할 수 있는 능력이 바로 고장 허용성이다.

- **등장 배경 및 발전 과정**:
  1. **초기 우주 탐사 (NASA)**: 아폴로 계획 당시, 우주의 가혹한 환경에서 컴퓨터 고장은 곧 승무원의 죽음을 의미했다. 이에 따라 하드웨어 이중화 및 다중화 기술이 비약적으로 발전했다.
  2. **금융 및 통신**: 연중무휴 거래가 이루어지는 은행 전산망과 중단 없는 통신망을 위해 무정지 시스템 (Non-stop System)이 상용화되었다.
  3. **클라우드 및 마이크로서비스**: 범용 하드웨어를 수천 대씩 사용하는 현대 데이터 센터에서는 고장이 '일상'이 되었고, 이를 소프트웨어적으로 해결하는 분산 고장 허용 모델이 표준이 되었다.

- **📢 섹션 요약 비유**: 전구 하나가 나가도 다른 전구들이 즉시 켜져서 방 안이 한 번도 어두워지지 않게 만드는 조명 시스템과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 고장 허용의 4단계 프로세스

FT 시스템은 오류가 발생했을 때 다음과 같은 단계로 대응한다.

| 단계 | 명칭 | 설명 | 비유 |
|:---|:---|:---|:---|
| **Fault Detection** | 고장 감지 | 시스템의 어느 부분에 문제가 생겼는지 즉각 파악 | 통증을 느낌 |
| **Fault Isolation** | 고장 격리 | 고장이 다른 정상 부품으로 전파되지 않도록 차단 | 상처 부위 지혈 |
| **Fault Reconfiguration** | 재구성 | 고장 난 부품을 배제하고 예비 부품을 투입 | 예비 타이어 교체 |
| **Fault Recovery** | 복구 | 시스템 상태를 고장 전의 정상 상태로 되돌림 | 새 살이 돋음 |

---

### 다중화 기법: TMR (Triple Modular Redundancy)

고장 허용의 가장 고전적이고 강력한 방법 중 하나는 동일한 모듈 3개를 돌려 다수결로 결과를 결정하는 것이다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               TMR (Triple Modular Redundancy) 구조              │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │               ┌──────────┐                                    │
 │        ┌─────▶│ 모듈 A    │─────┐                             │
 │        │      └──────────┘      │                             │
 │        │                        │     ┌───────────┐           │
 │ [입력] ┼─────▶┌──────────┐      ├────▶│ 다수결 회로 │──▶[출력]  │
 │        │      │ 모듈 B    │─────┤     │ (Voter)   │           │
 │        │      └──────────┘      │     └───────────┘           │
 │        │                        │           ▲                 │
 │        └─────▶┌──────────┐      │           │                 │
 │               │ 모듈 C    │─────┘     (2개 이상 일치 시        │
 │               └──────────┘             해당 값 출력)           │
 │                                                               │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TMR 구조에서는 입력 데이터를 세 개의 독립적인 모듈(A, B, C)이 동시에 처리한다. 만약 모듈 A에 하드웨어 결함이 발생하여 잘못된 결과를 내보내더라도, 다수결 회로 (Voter)는 정상인 B와 C의 결과가 일치함을 확인하고 올바른 최종 출력을 내보낸다. 이를 통해 오류를 외부로 노출하지 않고 즉각적으로 은폐 (Error Masking)한다. 이는 우주선이나 원자력 발전소 제어 시스템 등 극도의 신뢰성이 요구되는 곳에 쓰인다.

---

### 소프트웨어적 고장 허용: 체크포인트 (Checkpointing) & 롤백 (Rollback)

하드웨어 다중화 외에도 소프트웨어적으로 고장을 견디는 기법이 널리 쓰인다.

1. **Checkpointing**: 시스템의 정상 상태 (메모리, 레지스터 등)를 주기적으로 안정된 저장소에 저장한다.
2. **Rollback**: 장애 발생 시 처음부터 다시 시작하는 것이 아니라, 가장 최근의 체크포인트 지점으로 되돌아가서 작업을 재개한다.
3. **Rollforward**: 장애가 발생해도 멈추지 않고, 여분의 프로세서가 마지막 정상 상태부터 작업을 이어받아 처리한다.

- **📢 섹션 요약 비유**: 게임을 하다가 죽었을 때, 게임을 처음부터 다시 하는 것이 아니라 '세이브 포인트'에서 다시 시작하여 엔딩을 볼 수 있게 하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 고장 허용 (FT) vs 고가용성 (HA)

많은 사람들이 혼동하지만, FT와 HA는 보장 수준에서 큰 차이가 있다.

| 구분 | 고장 허용 (Fault Tolerance) | 고가용성 (High Availability) |
|:---|:---|:---|
| **장애 시 영향** | **전혀 없음 (No Downtime)** | 아주 짧은 중단 발생 (Failover 시간) |
| **비용** | 매우 높음 (자원 200~300% 소요) | 상대적으로 저렴 (자원 100%+ $\alpha$) |
| **데이터 보호** | 유실 가능성 0% | 극히 희박하지만 절체 중 유실 가능성 존재 |
| **주요 기술** | 하드웨어 다중화, TMR, 무정지 SW | 클러스터링, 로드밸런싱, 가상화 |

### 비교 2: Fail-Safe vs Fail-Operational

- **Fail-Safe**: 고장 시 시스템을 멈추더라도 안전한 상태 (Safe State)로 유도함. (예: 기차 고장 시 브레이크 자동 체결)
- **Fail-Operational**: 고장 시에도 기능을 멈추지 않고 계속 동작함. (예: 자동 비행 장치 고장 시 예비 장치 즉시 가동) -> **FT는 여기에 해당함.**

- **📢 섹션 요약 비유**: 브레이크가 고장 났을 때 차를 세우는 것(Fail-Safe)과, 브레이크가 고장 나도 예비 브레이크로 계속 운전할 수 있는 것(FT)의 차이와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 자율주행 자동차의 제어 유닛 설계

자율주행 아키텍트는 생명과 직결된 시스템을 위해 다음과 같은 FT 설계를 적용한다.
1. **이종 다중화 (Heterogeneous Redundancy)**: 똑같은 CPU 3개를 쓰면 공통의 버그(Common Mode Failure)에 취약하므로, 서로 다른 제조사의 CPU와 서로 다른 OS를 결합하여 다수결 시스템을 구축한다.
2. **Watchdog Timer**: 메인 프로세서의 동작을 감시하는 별도의 저사양 칩을 두어, 메인 칩이 응답하지 않으면 즉시 비상 제어 모드로 전환한다.
3. **Byzantine Fault Tolerance (BFT)**: 일부 노드가 단순히 죽는 것을 넘어, 악의적으로 잘못된 정보를 전달하는 상황에서도 전체 시스템이 합의를 이룰 수 있는 분산 알고리즘을 적용한다.

### 도입 체크리스트
- **비용 대 편익**: FT 도입 비용(H/W 3배 등)이 장애 시 발생하는 손실보다 정당화되는가?
- **Single Point of Failure (SPOF)**: 다수결 회로(Voter) 자체가 고장 나면 어떻게 할 것인가? (Voter 자체도 이중화 필요)
- **검증 가능성**: 장애 시나리오를 주입했을 때 실제로 시스템이 중단 없이 동작하는지 테스트했는가?

### 안티패턴
- **단일 지점 중복 (Partial FT)**: 서버는 이중화했지만 전원 공급 장치(PSU)는 하나만 써서, 전원 고장 시 전체가 멈추는 상황.
- **복구 시퀀스 누락**: 고장 상황은 잘 넘겼으나, 고장 난 부품을 교체한 후 다시 시스템에 편입시키는 과정에서 정합성이 깨지는 경우.

- **📢 섹션 요약 비유**: 앞문은 열쇠를 3개 달아놨는데 뒷문은 열어놓은 집(Partial FT)처럼, 한 군데라도 빈틈이 있으면 전체 FT가 무너진다는 것을 명심해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 설명 |
|:---|:---|:---|
| **정량적** | **제로 다운타임 (Zero Downtime)** | 계획되지 않은 장애 시간을 연간 0초에 가깝게 유지 |
| **정량적** | **데이터 무결성 보장** | 어떤 상황에서도 데이터 파손이나 손실 없이 완벽한 저장 보장 |
| **정성적** | **심리적 안전성 제공** | 미션 크리티컬 환경의 운영자가 시스템을 전적으로 신뢰하고 운영 가능 |
| **정성적** | **사회적 비용 감소** | 의료, 항공, 원자력 등 공공 안전 시스템의 사고 예방 |

### 미래 전망
향후 FT는 **"Software-Defined Fault Tolerance"**로 진화할 것이다. 과거에는 비싼 전용 하드웨어가 필요했지만, 이제는 가상화와 컨테이너 기술을 통해 일반 서버 위에서도 소프트웨어 계층이 고장을 실시간으로 감지하고 투명하게 복구하는 기술이 보편화될 것이다. 또한 양자 컴퓨팅 (Quantum Computing) 시대에는 양자 비트의 불안정성을 극복하기 위한 양자 오류 정정 (Quantum Error Correction) 기술이 FT의 새로운 핵심 과제가 될 것이다.

- **📢 섹션 요약 비유**: 미래의 시스템은 마치 어떤 충격을 받아도 금방 원래 모양으로 돌아오는 '형상기억합금'처럼, 고장의 상처가 남지 않는 매끄러운 서비스를 제공할 것입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **RAS (Reliability, Availability, Serviceability)**: FT가 지향하는 최종적인 시스템 특성.
- **가용성 (Availability)**: FT를 통해 달성하고자 하는 무정지 가용성.
- **Dual Redundancy**: FT의 가장 기초적인 이중화 기법.
- **Fail-safe/Fail-soft**: FT와 유사하지만 목표가 다른 안전 설계 기법.

## 👶 어린이를 위한 3줄 비유 설명
1. FT는 장난감 로봇에 **"예비 팔과 예비 다리"**가 숨겨져 있어서, 놀다가 팔이 하나 부러져도 바로 새 팔이 툭 튀어나와 계속 놀 수 있게 하는 마법 같은 능력이에요.
2. 비행기 엔진이 하나 꺼져도 다른 엔진으로 슝슝 날아가는 것처럼, 컴퓨터도 부품 하나가 아파도 다른 부품들이 힘을 합쳐서 우리를 도와주는 거예요.
3. 덕분에 우리가 유튜브를 보거나 게임을 할 때 컴퓨터 속 부품이 조금 아파도 우리는 전혀 모르고 즐겁게 놀 수 있답니다!
