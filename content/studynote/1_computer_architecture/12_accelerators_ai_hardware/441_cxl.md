# CXL (Compute Express Link)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CXL (Compute Express Link)은 CPU (Central Processing Unit)와 가속기(Accelerator) 간의 고속 통신을 위해 PCIe (Peripheral Component Interconnect Express) 물리 계층을 기반으로 캐시 일관성(Cache Coherency)을 보장하는 개방형 표준 인터커넥트 프로토콜이다.
> 2. **가치**: 메모리 풀링(Memory Pooling)과 컴포저블 인프라(Composable Infrastructure)를 통해 데이터 센터의 메모리 활용률을 40%에서 90% 이상으로 극대화하여, 하드웨어 구매 비용(CAPEX)과 운영 비용(OPEX)을 획기적으로 절감한다.
> 3. **융합**: HPC (High-Performance Computing) 및 AI (Artificial Intelligence) 워크로드의 '메모리 병목(Memory Wall)'을 해소하고, NUMA (Non-Uniform Memory Access) 아키텍처를 랙(Rack) 단위로 확장하는 핵심 패브릭 기술이다.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 정의**
CXL (Compute Express Link)은 PCIe의 전기적/물리적 인터페이스를 계승하면서도, 상위 계층에 새로운 프로토콜을 정의하여 CPU와 주변 장치(가속기, 메모리 확장 장치) 간의 메모리 공유와 캐시 일관성을 하드웨어 수준에서 보장하는 개방형 산업 표준이다. 기존 PCIe가 단순한 비연결형(Non-Coherent) 데이터 운반에 그쳤다면, CXL은 연결형(Coherent) 메모리 공간을 제공하여 '메모리 중심 컴퓨팅(Memory-Centric Computing)'을 실현한다.

**2. 등장 배경 (Why CXL?)**
① **기존 한계 (Memory Wall & Stranding)**:
전통적인 서버는 CPU가 소켓 내에 장착된 고정된 DRAM (Dynamic Random Access Memory)만 사용할 수 있었다. 이로 인해 특정 서버는 메모리가 부족해 작동을 멈추는 반면(Out of Memory), 다른 서버는 메모리가 남아도는 '메모리 스트랜딩(Memory Stranding)' 현상이 발생했다. 또한, AI 가속기가 연산을 위해 데이터를 주 메모리에서 복사해 오는 과정에서 PCIe의 대역폭과 지연 시간(Latency)이 병목이 되었다.
② **혁신적 패러다임 (Disaggregation)**:
스토리지가 SAN (Storage Area Network) 등을 통해 서버에서 분리된 것처럼, 메모리 자원 또한 서버에서 분리하여 공유 풀(Pool)로 구성하고 필요에 따라 동적으로 할당하는 '분리형 메모리(Disaggregated Memory)' 아키텍처가 필요해졌다.
③ **현재의 비즈니스 요구**:
LLM (Large Language Model)과 같은 거대 AI 모델 학습에는 단일 서버의 물리적 한계를 넘어서는 테라바이트(TB)~페타바이트(PB)급 메모리 공간이 요구되며, 이를 여러 서버의 메모리를 논리적으로 하나로 묶는 기술이 필수적이 되었다.

**3. 기술적 진화 과정**
CXL은 2019년 1.0 spec이 발표된 이후, PCIe Gen5의 속도 향상과 발맞추어 빠르게 진화하고 있다.
- **CXL 1.1/2.0**: 호스트-디바이스 간 1:1 연결 및 스위칭 기능 지원, 메모리 풀링 개념 도입.
- **CXL 3.0**: 패브릭(Fabric) 토폴로지 지원, P2P (Peer-to-Peer) 통신 강화, 멀티레벨 스위칭을 통한 수천 개 노드 연결 가능.

```text
   [Evolution of Interconnects]
   
   PCIe Gen1/2/3/4         CXL 1.1/2.0              CXL 3.0 Future
+-------------------+  +---------------------+  +-----------------------------+
| Non-Coherent I/O  |  | Coherent Memory     |  | Memory Fabrics /            |
| Point-to-Point    |-> | Memory Pooling      |-> | Disaggregated Data Center   |
| (Niche Bottleneck)|  | (Dynamic Allocation)|  | (Global Address Space)      |
+-------------------+  +---------------------+  +-----------------------------+
```

📢 **섹션 요약 비유**: 각 아파트(프로세서)마다 전용 창고(메모리)를 두어 빈방이 있어도 돌려쓰지 못하는 비효율을 없애고, 모든 아파트가 고속 도로(CXL)를 통해 거대한 공용 물류 센터(Memory Pool)를 자기 집 창고처럼 실시간으로 자유롭게 사용하는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. 구성 요소 및 하위 프로토콜**
CXL은 I/O, 메모리 확장, 캐시 일관성이라는 세 가지 요구사항을 만족시키기 위해 세 가지 동적 프로토콜을 하나의 링크에서 Time Division Multiplexing (TDM) 방식으로 multiplexing하여 사용한다.

| 요소명 | 역할 | 내부 동작 및 특징 | 프로토콜 레벨 | 비유 |
|:---:|:---|:---|:---:|:---:|
| **CXL.io** | 장치 발견, 링크 훈련, 구성 | PCIe 기반의 I/O semantics. 전통적인 부팅, DMA (Direct Memory Access), 구성 관리 담당. | Physical/Link | 건물의 **기본 상하수도/전기** 인프라 |
| **CXL.cache** | Device가 Host 메모리 캐싱 | Device(가속기)가 Host의 메모리를 읽고 쓸 때 캐시 일관성 유지. 호스트가 Home Node 역할 수행. | Coherency | 집 근처 **개인용 창고** (물건 비치해두고 씀) |
| **CXL.mem** | 메모리 확장 및 공유 | Host가 Device(메모리)에 있는 데이터를 직접 Load/Store. 물리적 메모리 주소 매핑. | Memory Protocol | 거대한 **공용 자전거 보관소** |
| **CXL.fabric** | 대규모 스위칭 (CXL 3.0) | 비연결형(Connection-less) 동작, 멀티캐스트, 스토리지 프로토콜 지원. | Switching Logic | **고속도로 인터체인지** 시스템 |

**2. CXL 링크 및 상태 전이 (State Machine)**
CXL 링크는 기존 PCIe와의 호환성을 위해 초기 부팅 시 PCIe Link Training을 수행한 후, CXL capability 교환을 통해 CXL 모드로 진입한다.

```text
+-------------------+          +-------------------+          +-------------------+
|     Host (CPU)    |          |   CXL Switch      |          |  Device (Mem/Acc) |
+-------------------+          +-------------------+          +-------------------+
|   Link Layer      | <------> |   Link Layer      | <------> |   Link Layer      |
| [CXL.mem / .cache]|          | [Routing Logic]   |          | [Mem Controller]  |
+-------------------+          +-------------------+          +-------------------+
        ^  |                           ^  |                          ^  |
        |  | (Coherent Mem Ops)       |  | (PASID, Snoop)           |  |
        |  +---------------------------+  +--------------------------+  |
        |                                                               |
        |          Virtual Hierarchy / Shared Memory Address Space     |
        +---------------------------------------------------------------+
        
[Link Initialization State Machine]
1. [PHY Reset]      : 전기적 신호 안정화
2. [Link Training]  : PCIe Gen5/Gen6 속도 협상 (Equalization)
3. [Caps Exchange]  : CXL capability 교환 (CXL.io/mem/cache 지원 여부 확인)
4. [Link Config]    : Flit Size, Path Enable 설정
5. [CXL Ready]      : 데이터 전송 가능 상태
```

**[해설]**
1. **PHY Layer**: PCIe 5.0(32 GT/s) 또는 6.0(64 GT/s PAM4)의 물리적 계층을 사용하여 초고속 대역폭을 확보한다. CXL 3.0에서는 PAM4 인코딩을 통해 대역폭을 두 배로 높인다.
2. **Link Layer**: 256B Flit(Flow Control Unit) 단위로 데이터를 전송하며, CRC(Cyclic Redundancy Check)와 순서 보장을 담당한다. CRC Error 발생 시 Retry mechanism이 작동한다.
3. **Transaction Layer**:
   - **CXL.cache**: Device가 "이 주소 데이터 줘(Request)"라고 하면, Host(Master)가 해당 데이터가 자신의 캐시에 있는지 확인(Snoop)하고 최신 데이터를 제공하거나 Invalidate 시그널을 보낸다. Write-Back, Write-Through 등의 정책을 지원한다.
   - **CXL.mem**: Host가 Device의 메모리 주소 공간을 자신의 물리 메모리 주소에 매핑(MMIO)하여 직접 접근한다. 이때 LATENCY는 로컬 DDR 대비 약 10~20ns 정도만 증가한다.
4. **Coherency Engine**: MESH(Manager, Home, Endpoint, Subordinate) 모델을 통해 여러 Host가 하나의 메모리를 읽고 쓸 때 데이터 최신성을 보장한다.

**3. 핵심 알고리즘 및 코드**
다음은 리눅스 커널 환경에서 CXL 메모리 영역을 NUMA 노드로 등록하여 페이지 할당자에 통합하는 개념적 코드이다.

```c
// Conceptual C Code for CXL Memory Integration
// Target: Linux Kernel Device Driver Context

struct cxl_mem_dev *cxl_dev;
unsigned long pfn_start, pfn_end;
int nid;

// 1. CXL Mem Device Enumeration (PCIe Scan)
cxl_dev = pci_get_device(PCI_VENDOR_ID_INTEL, CXL_MEM_DEVICE_ID, NULL);

// 2. Register Memory Resource
// CXL.mem Type-3 장치는 리소스로서 시스템에 등록됨
if (cxl_dev && cxl_is_enabled(cxl_dev)) {
    
    // 3. Acquire Physical Address Range
    // BAR0 (Base Address Register)를 통해 메모리 영역 획득
    resource_size_t phys_addr = pci_resource_start(cxl_dev, 0);
    resource_size_t size = pci_resource_len(cxl_dev, 0);

    // 4. Memory Hotplug operation
    // 이 물리 주소를 특정 NUMA 노드 ID에 연결 (e.g., Node 2)
    nid = numa_node_id(); 
    add_memory(nid, phys_addr, size);
    
    // 5. Online Memory
    // 온라인 상태로 변경하여 페이지 할당자(Buddy System)에 취급되게 함
    online_memory(phys_addr, size);
    
    printk(KERN_INFO "CXL Memory Registered at Node %d\n", nid);
}
```
*코드 설명: `add_memory` 함수는 새로운 메모리 영역을 커널의 메모리 관리 시스템에 추가하는 핵심 함수이다. 이 과정이 완료되면 `kmalloc()`이나 `malloc()` 호출 시 운영체제가 로컬 DRAM이 부족하면 자동으로 CXL 메모리 영역을 할당한다.*

📢 **섹션 요약 비유**: 복잡한 우편(Software) 절차 없이, 옆자리 동료의 책상 위에 있는 자료를 하드웨어적 스위치(Hardware Switch)를 통해 번개처럼 빌려 오는 초고속 물리적 협업 시스템과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 심층 기술 비교: CXL vs NVLink vs PCIe**

| 비교 항목 | PCIe 5.0 (Baseline) | NVLink 4.0 (NVIDIA) | CXL 3.0 (Target) |
|:---:|:---|:---|:---|
| **전송 속도** | 32GT/s (Raw) | ~70GT/s (Est.) | 64GT/s (Gen6 PAM4) |
| **캐시 일관성** | 지원 안 함 (Software) | 지원 (Proprietary) | **지원 (Open Standard)** |
| **프로토콜 스택** | Non-Coherent I/O | Coherent NVLink | **Coherent (CXL.mem/cache)** |
| **Topologies** | Root Complex only | Mesh/Strip | **Switched Fabric** |
| **주요 용도** | General Purpose I/O | GPU-to-GPU Clustering | **Memory Pooling & Composable** |
| **주요 장점** | 범용성, 호환성 | 최대 대역폭, 낮은 지연 | **표준화, 자원 분리/통합** |

*   **정량적 지표 분석**: CXL 메모리의 접근 지연 시간(Latency)은 로컬 DDR(약 80~100ns) 대비 약 2배(150~200ns) 수준이나, 기존 네트워크 기반 분산 메모리(RDMA, 1us+) 대비 훨씬 빠르며, 대역폭은 PCIe x16(약 64GB/s)을 Full Duplex로 활용 가능하다.

**2. 과목 융합 관점 (Inter-disciplinary Analysis)**

① **컴퓨터 구조 (Computer Architecture) - NUMA의 확장**:
기존의 UMA (Uniform Memory Access)나 소규모 NUMA에서 벗어나, CXL은 **"NUMA on a Fabric"**을 구현한다. OS의 메모리 관리자(MMU, Memory Management Unit)는 CXL 메모리를 '원격 노드(Remote Node)'로 인식한다. 이에 따라 페이지 부재(Page Fault) 시 데이터를 가져오거나, 핫 데이터(Hot Data)를 로컬 DRAM으로, 콜드 데이터(Cold Data)를 CXL 메모리로 이동시키는 **2-Tier Memory Placement** 알고리즘이 필수적이다.

② **운영체제 (OS) - 커널의 변화**:
리눅스 커널 5.15 이상에서는 CXL 포트를 통해 연결된 메모리를 `cxl_mem` 드라이버를 통해 관리하며, `dax` (Direct Access) 파일 시스템을 통해 파일 시스템 레벨에서도 영구 메모리(Persistent Memory, CXL.mem Type-3)로 활용할 수 있다. 이는 기존 스토리지 계층(BLOCK Layer)의 병목을 제거하여 `mmap()` 시스템 콜을 통해 스토리지에 직접 접근하는 듯한 성능을 제공한다.

③ **네트워크 (Network)와의 시너지**:
CXL 메모리가 부족한 서버는 CXL을 통해 이웃 서버의 메모리를 빌려쓰고(CXL.mem), 대용량 데이터 전송은 RDMA over Converged Ethernet (RoCE)와 병행하여 사용함으로써, 컴퓨팅 노드 간 데이터 이동 오버헤드를 최소화하는 하이브리드 인터커넥트 아키텍처가 등장하고 있다.

```text
   [System Architecture Integration]
   
+-------------------+      +--------------------+      +-------------------+
|  Compute Node 1   |      |   CXL Memory Pool  |      |  Compute Node 2   |
| (CPU + Local DRAM)| <==> | (Shared Volatile)  | <==> | (CPU + Local DRAM)|
+-------------------+      +--------------------+      +-------------------+
          |                           |                           ^
