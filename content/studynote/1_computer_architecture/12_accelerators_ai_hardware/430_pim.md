# PIM (Processing-In-Memory)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PIM (Processing-In-Memory)은 메모리 (DRAM, SRAM, MRAM 등) 내부나 바로 옆에 연산 유닛(ALU, MAC)을 집적하여, 데이터를 프로세서로 불러오지 않고 저장된 장소에서 직접 병렬 연산을 수행하는 차세대 메모리 아키텍처다.
> 2. **가치**: 프로세서와 메모리 간의 물리적 데이터 이동 (Data Movement)을 극단적으로 줄여, 폰 노이만 병목(메모리 월)으로 인한 극심한 지연(Latency)과 막대한 전력 소모 문제를 근본적으로 해결한다.
> 3. **융합**: 초거대 언어 모델(LLM)과 같은 메모리 대역폭 제한(Memory-bound) 워크로드의 폭발적 증가와 3D 스태킹(TSV) 패키징 기술의 발전이 맞물려, HBM-PIM, CXL-PIM 등 다양한 폼팩터로 진화하고 있다.

## Ⅰ. 개요 (Context & Background)

- **개념**: 기존 컴퓨터 시스템은 메모리(저장)와 프로세서(연산)가 분리된 폰 노이만 아키텍처를 따른다. 이와 달리 PIM (Processing-In-Memory)은 메모리 칩의 셀 어레이 주변부(Peripheral)나 메모리 다이(Die) 자체에 가벼운 로직 연산기를 내장하여 "메모리가 스스로 계산을 수행"하도록 만든 구조적 혁신이다.
- **💡 비유**: 요리사(CPU)가 식재료 창고(메모리)에서 재료를 도마(레지스터)까지 매번 수레로 실어와서 채소를 써는 대신, 창고 안에 간단한 채소 써는 기계(PIM)를 두고 "당근 좀 다져서 보내줘"라고 명령만 내리면, 다진 당근만 쏙 꺼내오는 완벽한 주방 시스템과 같습니다.
- **등장 배경**:
  1. **메모리 월 (Memory Wall)의 한계 봉착**: 무어의 법칙에 따라 프로세서 성능은 매년 비약적으로 증가했지만, DRAM의 접근 지연시간과 대역폭 발전 속도는 이를 따라가지 못했다. 아무리 연산기가 빨라도 데이터가 도착하지 않아 굶는(Starvation) 현상이 발생했다.
  2. **에너지 장벽 (Power Wall)과 데이터 이동 비용**: 7nm 공정 기준, 64-bit 부동소수점 연산 자체에 드는 에너지는 20pJ 수준이지만, 이를 오프칩 DRAM에서 가져오는 데는 1,000배가 넘는 수만 pJ의 에너지가 소모된다. AI 시대에 연산 에너지보다 "데이터 택배비"가 기형적으로 비싸진 것이다.
  3. **3D 패키징 및 공정 기술의 성숙**: 과거 로직 공정과 메모리 공정의 차이로 인해 PIM 칩 제조 단가가 너무 비쌌으나, TSV (Through Silicon Via)를 이용한 다이 스태킹(Die Stacking) 기술이 HBM (High Bandwidth Memory)을 통해 상용화되면서 로직 다이와 메모리 다이를 수직 결합하는 PIM 구현이 현실화되었다.
- **📢 섹션 요약 비유**: 왕복 2시간 걸리는 마트를 매번 오가는 대신, 마트 안에 조리실을 차려놓고 요리된 반조리 식품만 집으로 가져와 택배비와 시간을 모두 아끼는 혁신적인 유통 구조와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 위치 및 특징 | 비유 |
|:---|:---|:---|:---|:---|
| **Memory Cell Array** | 대규모 데이터 저장 (가중치 등) | 기존 DRAM/SRAM 셀과 동일하게 데이터 보존 | 메모리 뱅크 (Bank) | 책이 꽂혀 있는 거대한 서가 |
| **PIM Logic Unit (PCU)** | MAC(Multiply-Accumulate), 활성화 연산 등 수행 | 뱅크 내부에서 병렬로 활성화된 로우(Row) 데이터를 직접 가져와 연산 | 메모리 셀 주변부 (Peripheral) | 서가 각 층마다 배치된 요약 사서 |
| **Command Decoder** | 일반 읽기/쓰기 명령과 PIM 연산 명령 구분 | 컨트롤러에서 전송된 예약된 커맨드(예: PIM_OP) 해석 | 메모리 칩 제어부 | 일반 대출 vs 요약본 요청 분류 |
| **Internal Data Bus** | 뱅크와 PIM 유닛 간의 초고속 데이터 이동 | 칩 외부로 나가지 않으므로 수 TB/s 급의 거대한 내부 대역폭 제공 | 칩 내부 TSV 또는 메탈 배선 | 도서관 내부 전용 엘리베이터 |
| **Host Memory Controller** | CPU/GPU 측에서 PIM 동작 제어 및 스케줄링 | PIM 모드 전환, 가중치 사전 로드, 연산 시작/완료 동기화 | 프로세서 다이 (Host) | 메인 사서 (총괄 지휘자) |

### PIM (HBM-PIM 기반) 하드웨어 아키텍처

대표적인 PIM 구조인 HBM-PIM (High Bandwidth Memory 기반 PIM) 아키텍처는 메모리 뱅크 블록 사이에 소규모 연산기(PCU, Programmable Computing Unit)를 분산 배치하여, 메모리 내부의 방대한 대역폭을 연산에 직접 활용한다.

```text
 ┌───────────────────────────────────────────────────────────────────┐
 │               기존 폰 노이만 구조 vs PIM 아키텍처 비교                  │
 ├───────────────────────────────────────────────────────────────────┤
 │                                                                   │
 │  [기존 폰 노이만 구조 - Data Movement 병목]                         │
 │                                                                   │
 │  ┌─────────────────┐           외부 버스 (좁고 느림)   ┌───────────┐│
 │  │      Host       │             (수십 GB/s)          │   DRAM    ││
 │  │ ┌─────────────┐ │ ◀── 가중치, 입력 데이터 이동 ───▶ │ ┌───────┐ ││
 │  │ │ MAC Array   │ │                                  │ │ Bank0 │ ││
 │  │ └─────────────┘ │ ◀────────── 연산 결과 ──────────▶ │ └───────┘ ││
 │  └─────────────────┘                                  └───────────┘│
 │   ※ 연산은 빠르나 데이터를 가져오는 택배 기사가 부족하여 칩이 쉬고 있음   │
 │                                                                   │
 │                                                                   │
 │  [HBM-PIM 구조 - In-Memory Computing]                               │
 │                                                                   │
 │  ┌─────────────────┐           외부 버스 (제어/입력)   ┌───────────┐│
 │  │      Host       │            (명령어, 최종 결과만)  │ PIM-DRAM  ││
 │  │ ┌─────────────┐ │ ─── PIM 연산 커맨드(MAC) ────▶ │ ┌───────┐ ││
 │  │ │ Controller  │ │                                  │ │ Bank0 │ ││
 │  │ └─────────────┘ │ ◀──── 최종 부분합(Result) ───── │ │ ┌───┐ │ ││
 │  └─────────────────┘                                  │ │ │MAC│ │ ││
 │                                                       │ │ └───┘ │ ││
 │   ※ 수많은 뱅크 내부의 MAC 유닛들이 내부 초고속 대역폭   │ └───────┘ ││
 │      (수 TB/s)을 이용해 외부로 데이터를 내보내지 않고      └───────────┘│
 │      스스로 병렬 연산을 수행 후, 가벼운 결과값만 전송.                 │
 └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 기존 구조에서는 MAC 연산을 위해 거대한 가중치 행렬과 입력 벡터를 외부 인터페이스(핀)를 통해 칩 밖으로 끌어내야 했으며, 이 과정에서 심각한 병목(I/O Bottleneck)과 에너지 낭비가 발생한다. 반면 HBM-PIM 구조에서는 메모리 내부의 각 뱅크(Bank) 근처에 MAC 연산기를 삽입한다. 호스트(CPU/GPU)는 "A 주소의 가중치 행렬과 B 주소의 입력 벡터를 곱해라"라는 명령어(PIM 커맨드)만 메모리로 보낸다. 메모리 내부에서는 칩 외부로 데이터를 내보낼 필요 없이 내부의 광활한 핀 배선을 통해 수 테라바이트(TB/s)의 대역폭으로 데이터를 MAC 유닛에 쏟아부어 병렬 연산을 완료한 후, 축소된 결과값(스칼라 또는 작은 벡터)만 호스트로 반환한다.

### 심층 동작 원리 (PIM에서의 행렬-벡터 곱셈)

AI 추론, 특히 LLM 토큰 생성 과정에서 주로 발생하는 행렬-벡터 곱셈 (GEMV: General Matrix-Vector Multiplication) 시나리오:
1. **가중치 초기 로드**: 호스트 프로세서가 거대한 가중치 행렬(Weight Matrix)을 PIM-DRAM 뱅크들에 분산하여 미리 써둔다(Write).
2. **입력 벡터 브로드캐스트**: 추론 시, 호스트가 입력 벡터(Input Vector, 한 토큰의 임베딩)를 PIM-DRAM 전체에 브로드캐스트한다.
3. **In-Memory 연산**: PIM 커맨드 발동. 뱅크 내부의 PCU(PIM 연산기)들이 저장된 가중치를 로우(Row) 단위로 읽어 들여 입력 벡터와 MAC 연산을 병렬로 수행한다. 외부 대역폭은 전혀 소모되지 않는다.
4. **결과 취합**: 각 뱅크에서 계산된 부분 합(Partial Sum)들이 PIM 컨트롤러를 거쳐 최종 합산되어 호스트로 전달된다.

- **📢 섹션 요약 비유**: 광활한 농장(메모리)에서 수확한 과일을 모두 트럭에 싣고 도시의 공장(CPU)으로 보내 잼을 만드는 대신, 농장 구석구석에 미니 잼 공장(PIM)을 짓고 완성된 잼만 도시로 배송하여 도로 병목을 아예 없애버린 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 비교 1: 폰 노이만 (Von Neumann) vs PIM 아키텍처

| 구분 | 폰 노이만 (범용 CPU/GPU) | PIM (Processing-In-Memory) |
|:---|:---|:---|
| **데이터 이동** | 연산량 대비 데이터 이동이 압도적으로 많음 | 데이터가 저장된 곳에서 연산하므로 극소화 |
| **병목 지점** | 칩 간 인터페이스 대역폭 (Off-chip Bandwidth) | PIM 로직 면적으로 인한 메모리 용량 손실 |
| **에너지 소모** | 데이터 이동 비용 >>> 연산 비용 | 연산 비용 중심, 전체 소모 전력 70%+ 절감 |
| **주요 워크로드**| Compute-bound (CNN 등 연산 집약적) | **Memory-bound (LLM, RNN, 추천 시스템 등)** |
| **설계 복잡도** | 프로세서 최적화 집중 | 프로세서-메모리 코디자인(Co-design), 신규 명령어 필요 |

### 비교 2: PIM의 다양한 통합 방식 (수준별)

| 통합 수준 | 명칭 | 구조적 특징 | 대표적 한계점 |
|:---|:---|:---|:---|
| **Chip-Level (PNM)**| Processing Near Memory | 메모리 모듈/보드 단위에 가속기(FPGA 등) 부착 | PIM만큼의 극단적 대역폭 확보는 어려움 |
| **Package-Level** | 3D 스태킹 PIM (HBM-PIM) | 로직 다이와 DRAM 다이를 TSV로 수직 적층 | 발열 제어 (Thermal Issue) 및 공정 단가 상승 |
| **Bank-Level** | Bank-level PIM | DRAM 다이 내부 뱅크 주변에 연산기 집적 (상용화 진행 중) | DRAM 공정의 제약으로 고성능 연산기 구현 난해 |
| **Cell-Level (CIM)**| Computing In Memory | 아날로그 메모리 소자(ReRAM, SRAM) 자체에서 물리 법칙으로 연산 | 노이즈로 인한 정밀도 저하, 완전 디지털 변환 필요 |

### 과목 융합 관점
- **컴퓨터 구조 (CA)**: PIM은 기존 캐시 코히어런시 (Cache Coherency) 프로토콜에 심각한 도전을 제기한다. 프로세서 캐시에 있는 데이터와 PIM에서 연산 중인 데이터의 일관성을 맞추기 위해, 운영체제는 PIM이 접근하는 메모리 영역을 비캐시(Uncacheable) 영역으로 맵핑해야 한다.
- **반도체 공정**: 표준 DRAM 공정(커패시터 최적화, 누설 전류 최소화)과 로직 공정(고속 트랜지스터 최적화)은 특성이 극단적으로 다르다. 뱅크 레벨 PIM은 이 두 공정을 하나의 실리콘 다이에 타협하여 욱여넣어야 하는 수율과 설계 난이도 문제를 극복해야 한다.

- **📢 섹션 요약 비유**: 폰 노이만 구조가 "데이터를 프로세서로 배달시키는 콜드체인 물류"라면, PIM은 배달을 포기하고 "고객(연산기)이 산지(메모리)로 이사가서 직접 소비하는 로컬 푸드" 운동과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오
1. **시나리오 — 거대 언어 모델(LLM) 생성 단계(Decoding)의 극심한 병목 현상**: GPT와 같은 LLM은 첫 토큰 분석(Prefill) 시에는 병렬 연산이 잘 되지만, 단어를 하나씩 생성하는 Decoding 단계에서는 연산량(FLOPs)은 적은데 수십 GB의 가중치 행렬을 매번 메모리에서 읽어와야 하는 극단적인 Memory-bound 상태에 빠진다. GPU의 수천 개 코어가 데이터를 기다리며 놀게 된다.
   - **의사결정**: HBM-PIM 기술을 도입하여 가중치가 저장된 메모리 뱅크 내부에서 즉각적으로 벡터 행렬 내적을 수행하도록 아키텍처를 변경. 이를 통해 호스트로 데이터를 끌어오는 대역폭을 낭비하지 않고 시스템 성능을 2.5배 이상 개선하며 전력은 절반으로 줄일 수 있다.

2. **시나리오 — 클라우드 데이터센터의 추천 시스템 (Recommendation System) 지연**: 사용자 맞춤형 추천 알고리즘 (DLRM)은 임베딩 테이블 조회가 핵심인데, 이 테이블은 크기가 수백 GB에 달해 산발적인 메모리 읽기(Random Access)를 무수히 유발한다.
   - **의사결정**: CXL 기반의 메모리 확장 모듈에 연산 기능을 추가한 CXL-PNM/PIM (Processing-Near-Memory)을 도입. 호스트 CPU가 거대한 임베딩 테이블을 직접 조회하는 대신, "User ID 123의 임베딩 값을 합산해서 줘"라고 오프로드(Offload)하여 PCIe 버스의 트래픽을 기하급수적으로 줄인다.

### 도입 체크리스트
- **기술적**: 기존 소프트웨어 생태계와의 호환성을 위해 PIM 전용 명령어(Instruction Set) 컴파일이 프레임워크(PyTorch 등) 레벨에서 투명하게(Transparently) 지원되는가? PIM의 발열이 HBM 쿨링 시스템 설계 한계치 내에 들어오는가?
- **운영/보안**: PIM 메모리를 장착하기 위해 커스텀 메모리 컨트롤러가 탑재된 특정 벤더의 CPU/GPU 락인(Lock-in) 효과가 발생하지는 않는가?

### 안티패턴
- **Compute-Bound 작업의 무리한 PIM 할당**: CNN의 깊은 레이어 학습이나 빽빽한 행렬-행렬 곱셈(GEMM)처럼 데이터 재사용률이 압도적으로 높고 연산 강도(Arithmetic Intensity)가 높은 작업을 PIM으로 넘기면, PIM의 약한 연산 능력 탓에 오히려 기존 GPU 캐시/시스톨릭 어레이를 사용하는 것보다 성능이 수 배 하락한다. 워크로드 특성을 무시한 PIM 도입은 치명적 실패를 낳는다.

- **📢 섹션 요약 비유**: PIM은 짐이 아주 무겁고 부피가 큰 화물(Memory-bound)을 처리할 때 진가를 발휘하는 화물 열차입니다. 작고 가벼운 소포(Compute-bound)를 재빨리 배달해야 할 때 이 열차를 쓰면 오히려 오토바이(GPU 캐시)보다 늦어지는 역효과가 납니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량/정성 기대효과

| 구분 | 일반 GPU/HBM 시스템 | PIM 적용 시스템 (예: 삼성 HBM-PIM) | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (성능/속도)** | LLM 디코딩 시 대역폭 병목으로 초당 수십 토큰 | 메모리 내부 수 TB/s 대역폭 활용 | 시스템 성능 (Throughput) **2~3배 향상** |
| **정량 (에너지/전력)**| 데이터 I/O 에너지 비중이 전체의 70% | 외부 I/O 트래픽 제거 | 시스템 전체 에너지 소모 **70% 절감** |
| **정성 (발열/신뢰성)**| 칩 간 인터페이스의 극한 발열 | 로직 연산이 뱅크 단위로 분산 | 보드 레벨 발열 완화, 패키징 신뢰도 상승 |

### 미래 전망
- **PIM 컴파일러 생태계의 성숙**: 하드웨어는 준비되었으나 "어떤 연산을 PIM에 위임하고, 어떤 연산을 CPU/GPU가 할 것인가"를 자동으로 스케줄링하는 투명한(Transparent) 시스템 소프트웨어/컴파일러 기술이 향후 PIM 대중화의 성패를 가를 것이다.
- **아날로그 CIM (Computing-In-Memory)의 부상**: 디지털 로직을 추가하는 PIM을 넘어, 멤리스터(ReRAM)나 MRAM 자체의 저항 값을 이용해 키르히호프의 법칙(아날로그 전류 합산)으로 1사이클에 MAC을 수행하는 궁극의 에너지 효율 아키텍처가 차세대 에지(Edge) AI의 패러다임을 바꿀 것이다.

### 참고 표준
- **JEDEC PIM 표준화**: 메모리 표준 제정 기구인 JEDEC에서 DRAM에 연산 기능을 추가하기 위한 인터페이스, 프로토콜 및 핀맵 표준화 논의가 삼성전자, SK하이닉스 주도로 활발히 진행 중이다.
- **CXL (Compute Express Link)**: 메모리 확장과 PIM/PNM 디바이스를 호스트 캐시 코히어런트 영역에 연결하기 위한 차세대 개방형 인터커넥트 인터페이스.

- **📢 섹션 요약 비유**: 지금껏 "데이터의 보관소" 역할에 머물렀던 메모리가 스스로 생각하고 계산하는 "미니 두뇌"로 진화하면서, 반도체 산업은 보관과 연산의 경계가 허물어지는 거대한 패러다임 전환을 맞이하고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **메모리 월 (Memory Wall)**: 프로세서 속도와 메모리 속도의 극심한 격차로 인해 시스템 성능이 저하되는 현상. PIM의 가장 직접적인 등장 배경.
- **PNM (Processing-Near-Memory)**: PIM과 유사하나, 연산기가 칩 내부가 아닌 메모리 모듈이나 CXL 컨트롤러 근처에 위치하여 구현 난이도를 낮춘 기술.
- **CIM (Computing-In-Memory)**: 메모리 셀 자체가 아날로그 물리 법칙을 이용해 병렬 연산을 수행하는 PIM의 궁극적이고 극단적인 진화 형태.
- **LPU (Language Processing Unit)**: LLM 추론 가속기. LLM의 디코딩 병목을 해결하기 위해 PIM 기술이 적극적으로 검토/도입되고 있음.
