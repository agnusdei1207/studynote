# 스레드 블록 (Thread Block)과 워프 (Warp)

> 1. **본질**: 스레드 블록 (Thread Block)은 프로그래머가 정의하는 **논리적인 스레드 묶음**이며, 워프 (Warp)는 GPU 하드웨어가 실제로 한 번에 실행하는 **물리적인 스레드 실행 단위** (일반적으로 32개 스레드)다.
> 2. **가치**: 스레드 블록은 데이터 공유 (Shared Memory)와 동기화 (Barrier)의 경계를 제공하고, 워프는 하드웨어 스케줄링의 효율성을 극대화하여 GPU의 방대한 병렬 처리 능력을 실현하는 핵심 메커니즘이다.
> 3. **융합**: CUDA 프로그래밍 모델의 유연성과 SIMT (Single Instruction Multiple Threads) 하드웨어의 강제성이 결합된 지점으로, 성능 최적화의 80% 이상이 워프 단위의 정렬된 메모리 접근과 분기 제어에서 결정된다.

---

## Ⅰ. 개요 (Context & Background)

- **概念**: GPU 프로그래밍에서 '스레드 블록'은 작업을 나누는 추상적 단위다. 개발자는 수백 개의 스레드를 하나의 블록으로 묶어 특정 작업을 시킨다. 하지만 GPU 하드웨어는 이 블록을 한꺼번에 돌리지 않고, 내부적으로 32개씩 쪼개어 '워프'라는 단위로 번갈아 가며 실행한다.

- **💡 비유**: 스레드 블록은 **"한 반에 모인 128명의 학생들"**과 같고, 워프는 **"그중 한 분단인 32명의 학생들"**과 같다. 선생님(프로그래머)은 '반' 단위로 숙제(Block)를 내주지만, 급식소(하드웨어)는 한 번에 '한 분단(Warp)'씩만 입장시켜서 밥을 먹이는 것과 같다.

- **등장 배경**:
  1. **확장성 (Scalability) 확보**: 하드웨어마다 코어 수가 달라도 동일한 코드가 돌아가게 하려면, 작업을 독립적인 덩어리(Block)로 나누어 각 하드웨어가 가능한 만큼씩 가져가 처리하게 해야 했다.
  2. **하드웨어 단순화**: 수천 개의 코어에 각기 다른 명령어를 주는 것은 회로가 너무 복잡해진다. 따라서 32개 코어(Warp)에 하나의 명령어만 동시에 뿌려주는 방식을 택해 연산 밀도를 높였다.
  3. **지연 시간 은닉 (Latency Hiding)**: 한 워프가 데이터를 기다리느라 멈추면, 대기 중인 다른 워프를 즉시 교체 투입하여 하드웨어가 쉬지 않게 하려는 목적이 크다.

- **📢 섹션 요약 비유**: 논리적인 팀워크(Block)와 물리적인 실행 규칙(Warp)을 분리하여, 어떤 규모의 하드웨어에서도 최적의 병렬 성능을 낼 수 있도록 설계된 GPU의 작업 분배 시스템입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **스레드 (Thread)** | 최소 연산 수행 단위 | 자신의 레지스터와 로컬 메모리 보유 | Program Counter | 개별 작업자 |
| **스레드 블록 (Block)** | 스레드들의 논리적 집합 | 동일 SM에 할당되어 자원 공유 | Shared Memory, `__syncthreads()` | 협동 작업을 하는 부서 |
| **워프 (Warp)** | 하드웨어 실행의 최소 단위 | 32개 스레드가 하나의 명령어를 공유 | SIMT (Single Instruction Multiple Threads) | 한 목소리로 구호를 외치는 소대 |
| **워프 스케줄러 (Warp Scheduler)** | 실행할 워프를 선택 및 투입 | 사이클마다 준비된 워프를 ALU로 디스패치 | Zero-overhead Context Switch | 무대 위에 올릴 배우를 정하는 연출가 |
| **공유 메모리 (Shared Memory)** | 블록 내 스레드 간 데이터 교량 | 블록 단위로 할당되는 고속 SRAM | L1 Cache급 속도 | 부서원들이 함께 쓰는 공동 화이트보드 |

### 스레드 블록과 워프의 매핑 구조

스레드 블록은 프로그래머가 크기를 정할 수 있지만(예: 128, 256, 512 등), 하드웨어는 이를 무조건 32개 단위의 워프로 쪼개어 관리한다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               스레드 블록 (Block) 내 워프 (Warp) 분할 구조          │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │   Thread Block (Size: 128 Threads)                            │
 │   ┌───────────────────────────────────────────────────────┐   │
 │   │  ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐             │   │
 │   │  │Warp 0 │ │Warp 1 │ │Warp 2 │ │Warp 3 │             │   │
 │   │  │(0~31) │ │(32~63)│ │(64~95)│ │(96~127)             │   │   │
 │   │  └───────┘ └───────┘ └───────┘ └───────┘             │   │
 │   └───────┬─────────┬─────────┬─────────┬─────────────────┘   │
 │           │         │         │         │                     │
 │           ▼         ▼         ▼         ▼                     │
 │   ┌───────────────────────────────────────────────────────┐   │
 │   │           SM (Streaming Multiprocessor)               │   │
 │   │  ┌─────────────────────────────────────────────────┐  │   │
 │   │  │             Warp Scheduler (Active)             │  │   │
 │   │  └────────────────────────┬────────────────────────┘  │   │
 │   │             ┌─────────────┴─────────────┐             │   │
 │   │     [ Warp 0 ]      [ Warp 1 ]      [ Warp 2 ]        │   │
 │   │      (Running)       (Waiting)       (Ready)          │   │
 │   └───────────────────────────────────────────────────────┘   │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상기 구조는 128개의 스레드로 구성된 하나의 블록이 어떻게 하드웨어에서 처리되는지 보여준다. 블록은 4개의 워프 (128 / 32 = 4)로 나뉘어 SM 내부의 워프 스케줄러에 등록된다. 스케줄러는 이 중 실행 준비가 된(Ready) 워프를 골라 연산 유닛에 투입한다. 만약 Warp 0이 메모리에서 데이터를 가져오느라 기다려야 하면(Waiting), 스케줄러는 지체 없이 Warp 2를 실행(Running) 상태로 바꾼다. 이 과정을 통해 외부 메모리 접근 지연 시간(Latency)을 다른 워프의 연산 작업으로 완벽하게 숨긴다. 중요한 점은 블록 내 스레드들은 번호 순서대로 워프에 할당되며, 같은 워프에 속한 스레드들은 물리적으로 정확히 동시에 실행된다는 것이다.

---

### 심층 동작 원리: 워프 발산 (Warp Divergence)

워프 내 32개 스레드는 하나의 명령어를 공유하므로, `if (condition)` 문에서 스레드별로 조건이 갈리면 문제가 발생한다.
1. **Divergence 발생**: 조건이 참인 스레드들과 거짓인 스레드들이 나뉘면, 하드웨어는 두 경로를 순차적으로 실행한다.
2. **Masking**: 참인 경로를 실행할 때는 거짓인 스레드들의 연산 결과를 무시(Masking)하고, 반대의 경우도 마찬가지다.
3. **성능 하락**: 모든 스레드가 동일한 경로를 갈 때보다 실행 시간이 약 2배로 늘어난다.

- **📢 섹션 요약 비유**: 단체 무용을 하다가 반은 앉고 반은 서야 하는 동작이 나오면, 한 동작씩 따로 연습하고 합쳐야 하므로 시간이 두 배로 걸리는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 스레드 블록 vs 워프

| 비교 항목 | 스레드 블록 (Thread Block) | 워프 (Warp) |
|:---|:---|:---|
| **성격** | 논리적 단위 (Logical) | 물리적 단위 (Physical) |
| **크기 결정** | 프로그래머가 결정 (최대 1024) | 하드웨어가 결정 (고정 32) |
| **자원 공유** | 공유 메모리 (Shared Mem) 공유 가능 | 명령어 유닛 (Instruction) 공유 |
| **동기화** | `__syncthreads()`로 전체 동기화 | 암시적으로 모든 스레드 동기 실행 |
| **할당 단위** | SM (Streaming Multiprocessor) 단위 | Warp Scheduler 단위 |

### 비교 2: 블록 크기 설정에 따른 트레이드오프

| 블록 크기 (Threads) | 장점 | 단점 |
|:---|:---|:---|
| **작은 블록 (예: 64)** | SM 내 자원 조각화 감소, 세밀한 제어 | 워프 스케줄링 여유 부족, 하드웨어 활용 저하 |
| **큰 블록 (예: 512)** | 풍부한 워프로 메모리 지연 은닉 탁월 | 레지스터/공유메모리 부족으로 점유율 하락 |
| **권장 사항** | 128~256 사이의 32 배수 | 하드웨어 사양(Occupancy)에 따른 튜닝 필수 |

### 과목 융합 관점

- **컴퓨터 구조 (CA)**: 워프의 존재는 '동일 명령 다중 데이터 (SIMD)' 아키텍처를 '동일 명령 다중 스레드 (SIMT)'로 진화시킨 결과다. 이는 하드웨어의 단순성과 소프트웨어의 유연성 사이의 절묘한 균형점이다.
- **운영체제 (OS)**: 스레드 블록은 OS의 프로세스나 스레드와 달리 컨텍스트 스위칭 비용이 거의 없다. 이는 GPU 전용 하드웨어 스케줄러가 모든 상태(Register)를 미리 확보하고 있기 때문이다.

- **📢 섹션 요약 비유**: 큰 팀(Block)을 꾸려 협동하게 하되, 실제 훈련은 작은 조(Warp) 단위로 일사불란하게 진행하여 훈련장(SM)의 가동률을 극대화하는 군대식 운영 철학입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **워프 투표 (Warp Vote) 함수 활용**: 워프 내 모든 스레드가 특정 조건을 만족하는지 확인해야 할 때, 공유 메모리를 쓰는 대신 하드웨어 전용 함수(`__all_sync`, `__any_sync`)를 사용한다. 이는 블록 단위 동기화보다 훨씬 빠르며 워프의 물리적 특성을 이용한 고도화된 최적화 기법이다.
2. **블록 크기와 레지스터 사용량의 충돌**: 한 스레드가 레지스터를 너무 많이 쓰면 SM 하나에 올라갈 수 있는 블록 수가 제한된다. 실무에서는 블록 크기를 줄이거나 코드를 단순화하여 'Active Warps' 수를 늘림으로써 메모리 지연 시간을 효과적으로 숨긴다.
3. **워프 셔플 (Warp Shuffle)을 통한 리덕션 연산**: 합계(Sum)를 구할 때 공유 메모리 대신 레지스터끼리 데이터를 직접 교환하는 워프 셔플 지시어를 사용한다. 이는 메모리 접근 지연을 0으로 만들어 연산 속도를 극대화한다.

### 도입 체크리스트
- **32의 배수 설정**: 블록 크기가 32의 배수가 아닐 경우, 마지막 워프에 빈 스레드가 생겨 하드웨어 자원이 낭비되는가?
- **점유율 (Occupancy) 계산**: 현재 블록 설정에서 SM이 수용할 수 있는 최대 워프 수 대비 실제 활성 워프 수의 비율이 적절한가?
- **동기화 지점 최적화**: `__syncthreads()`를 남발하여 워프들이 서로를 기다리느라 연산 유닛이 놀고 있지는 않는가?

### 안티패턴
- **워프 내 조건 분기 무시**: `if (threadIdx.x % 2 == 0)` 처럼 워프 내에서 홀짝 스레드가 다른 일을 하게 짜면 성능이 50% 하락한다. 가능한 한 32개 단위로 묶어서 조건 분기가 일어나도록 유도해야 한다.

- **📢 섹션 요약 비유**: 32인용 버스(Warp)에 1명만 태우고 출발하거나, 버스 안의 절반은 왼쪽으로 절반은 오른쪽으로 가겠다고 싸우는 상황(Divergence)을 피하는 것이 최적화의 핵심입니다.

### 미래 전망
차세대 GPU 아키텍처에서는 **'독립적 스레드 스케줄링 (Independent Thread Scheduling)'** 도입을 통해 워프 발산 시에도 스레드들의 실행 순서를 하드웨어가 어느 정도 유연하게 관리할 수 있게 진화하고 있다. 또한 워프 크기 자체를 가변적으로 가져가거나(예: AMD의 Wavefront 32/64 선택), 블록 간의 직접적인 통신(Grid-level Sync)을 지원하는 방향으로 나아가고 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. 스레드 블록은 한 반에 모인 **"320명의 친구들"**이고, 워프는 그중 한 줄에 앉은 **"32명의 단짝들"**이에요.
2. 선생님이 한 반(Block) 전체에 숙제를 내주면, 로봇 팔(하드웨어)이 한 줄(Warp)씩 차례대로 빠르게 숙제를 도와줘요.
3. 한 줄 친구들이 숙제 자료를 찾느라 멈추면, 로봇 팔은 기다리지 않고 다음 줄 친구들을 바로 도와주기 때문에 엄청나게 빨리 일을 끝낼 수 있답니다!
