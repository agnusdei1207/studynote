# 126. 하버드 아키텍처 (Harvard Architecture)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 명령어와 데이터를 물리적으로 분리된 메모리와 버스로 처리하여 병렬 페치·실행 가능한 구조
> 2. **가치**: 명령어/데이터 동시 접근으로 높은 대역폭, 결정론적 타이밍으로 DSP/MCU에 최적
> 3. **융합**: L1 캐시 분리(I-Cache/D-Cache)로 현대 CPU에 부분 채택, 실시간 시스템의 기반

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

하버드 아키텍처(Harvard Architecture)는 명령어 메모리(Program Memory)와 데이터 메모리(Data Memory)를 물리적으로 분리하고, 각각 독립적인 버스로 접근하는 컴퓨터 구조다. 이를 통해 명령어 페치와 데이터 접근을 동시에 수행할 수 있다.

### 💡 비유: 양방향 도로 vs 일방통행 2개

폰 노이만은 **양방향 도로 하나**에 명령어 차와 데이터 차가 다니며 교통체증(병목)이 발생한다. 하버드는 **일방통행 도로 두 개**를 만들어 명령어 차선과 데이터 차선이 분리되어 동시에 빠르게 이동할 수 있다.

### 등장 배경

| 시기 | 배경 | 발전 |
|:---:|:---|:---|
| 1930s | Harvard Mark I | 릴레이 기반 계산기 |
| 1947 | 하버드 구조 정립 | 명령어/데이터 분리 |
| 1970s | DSP, MCU | 실시간 처리 요구 |
| 현재 | 수정 하버드 | L1 캐시 분리 |

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성요소

| 구성요소 | 폰 노이만 | 하버드 |
|:---|:---|:---|
| **메모리** | 통합 | 분리 (Program + Data) |
| **버스** | 공통 | 분리 (Instruction + Data) |
| **메모리 폭** | 동일 | 다를 수 있음 |
| **주소 공간** | 통합 | 분리 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오

**시나리오 1: 오디오 DSP 설계**
- 요구: 48kHz 샘플링, 32비트 처리
- 선택: 하버드 구조 DSP
- 이유: 명령어 fetch와 샘플 처리 병렬

**시나리오 2: IoT 센서 노드**
- 요구: 저전력, 실시간 응답
- 선택: Cortex-M (수정 하버드)
- 이유: Flash에서 실행, RAM은 데이터

**시나리오 3: 범용 서버 CPU**
- 요구: 유연성, 높은 성능
- 선택: 수정 하버드 (L1 분리)
- 이유: L1 하버드 + L2+ 통합
