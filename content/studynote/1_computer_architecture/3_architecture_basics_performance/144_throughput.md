# 144. 처리량 (Throughput)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 단위 시간당 시스템이 실제로 처리한 작업/데이터 양, TPS/QPS/RPS 단위
> 2. **가치**: 시스템 용량 계획·확장성 판단의 핵심, 높을수록 좋음
> 3. **융합**: 응답 시간과 트레이드오프, 병렬화·분산으로 증대

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

처리량(Throughput)은 시스템이 단위 시간 동안 실제로 처리하는 작업의 양을 의미한다. 대역폭(Bandwidth)이 이론적 최대치라면, 처리량은 실제 달성한 처리 성능이다.

### 💡 비유: 공장 생산량

처리량은 **공장이 한 시간에 생산하는 제품 수**다. 이론적 최대(대역폭)는 1000개/시간이어도, 실제(처리량)는 800개/시간일 수 있다. 기계 고장, 재료 부족 등이 원인이다.

└─────────────────────────────────────────────────────────────────────┘

### 처리량 vs 대역폭

```
┌─────────────────────────────────────────────────────────────────────┐
│                처리량 vs 대역폭                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【정의】                                                           │
│  ────────                                                           │
│  • 대역폭 (Bandwidth): 시스템의 최대 처리 용량 (이론적)             │
│  • 처리량 (Throughput): 실제 달성한 처리량                          │
│                                                                     │
│  【관계】                                                           │
│  ────────                                                           │
│  Throughput ≤ Bandwidth                                             │
│  Efficiency = Throughput / Bandwidth                                │
│                                                                     │
│  【처리량 감소 요인】                                               │
│  ──────────────────────                                             │
│  1. 프로토콜 오버헤드 (헤더, 체크섬)                                │
│  2. 재전송 (에러, 손실)                                             │
│  3. 혼잡 제어                                                       │
│  4. 처리 지연                                                       │
│  5. 병목 구간                                                       │
│  6. 경합 (Contention)                                               │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  예시: 1 Gbps 이더넷                                           │ │
│  │  ────────────────────                                          │ │
│  │  대역폭: 1 Gbps                                                 │ │
│  │  이더넷 오버헤드: -12%                                          │ │
│  │  IP/TCP 오버헤드: -5%                                           │ │
│  │  재전송/혼잡: -3%                                               │ │
│  │  처리량: 800 Mbps (80%)                                         │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 처리량과 Little's Law

```
┌─────────────────────────────────────────────────────────────────────┐
│                Little's Law                                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【공식】                                                           │
│  ────────                                                           │
│                                                                     │
│  L = λ × W                                                          │
│                                                                     │
│  • L: 시스템 내 평균 요청 수                                        │
│  • λ: 도착률 = 처리량 (요청/초)                                     │
│  • W: 평균 체류 시간 = 응답 시간                                    │
│                                                                     │
│  【변형】                                                           │
│  ────────                                                           │
│  λ = L / W                                                          │
│  처리량 = 동시 요청 수 / 응답 시간                                  │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  예시: 웹 서버                                                  │ │
│  │  ───────────────                                                │ │
│  │  • 동시 연결: 1000개                                            │ │
│  │  • 평균 응답 시간: 200ms                                        │ │
│  │  • 처리량 = 1000 / 0.2 = 5000 RPS                               │ │
│  │                                                               │ │
│  │  → 200ms 응답으로 1000 동시 연결 유지 시 5000 RPS              │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【의미】                                                           │
│  ────────                                                           │
│  • 처리량 증가 → 응답 시간 증가 or 동시 요청 증가                   │
│  • 동시 요청 제한 → 처리량 제한                                     │
│  • 응답 시간 단축 → 처리량 증가 (동일 동시 요청)                    │
└─────────────────────────────────────────────────────────────────────┘
```

### 처리량 증대 전략

```
┌─────────────────────────────────────────────────────────────────────┐
│                처리량 증대 전략                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【수직 확장 (Scale-Up)】                                           │
│  ────────────────────────                                           │
│  • 더 빠른 CPU, 더 많은 코어                                        │
│  • 더 많은 메모리                                                   │
│  • 더 빠른 스토리지                                                  │
│  • 장점: 단순, 일관성                                               │
│  • 단점: 비용, 한계                                                 │
│                                                                     │
│  【수평 확장 (Scale-Out)】                                          │
│  ──────────────────────────                                         │
│  • 서버 인스턴스 추가                                                │
│  • 로드 밸런싱                                                       │
│  • 샤딩, 파티셔닝                                                    │
│  • 장점: 무한 확장 가능                                             │
│  • 단점: 복잡성, 일관성                                             │
│                                                                     │
│  【최적화】                                                         │
│  ──────────────                                                     │
│  • 코드 최적화                                                       │
│  • 캐싱 (메모리, CDN)                                               │
│  • 비동기 처리                                                       │
│  • 배치 처리                                                        │
│                                                                     │
│  【공식】                                                           │
│  ────────                                                           │
│  Total Throughput = Σ(Instance Throughput) × Efficiency             │
│                                                                     │
│  → 완벽한 수평 확장: Efficiency = 1.0                               │
│  → 현실: Efficiency < 1.0 (오버헤드)                                │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 1: 블랙 프라이데이 트래픽**
- 일평균: 10K RPS
- 피크 예상: 100K RPS
- 현재 용량: 50K RPS
- 해결: 오토스케일링 3×, CDN, 캐싱

**시나리오 2: 데이터베이스 병목**
- 문제: 5K TPS 한계
- 요구: 20K TPS
- 해결: 읽기 복제 3대, 샤딩

**시나리오 3: 마이크로서비스 처리량**
- 서비스 A: 10K RPS
- 서비스 B: 5K RPS
- 메시지 큐: 50K msg/s
- 설계: 카프카 파티션 10개

**시나리오 4: 스토리지 IOPS**
- 요구: 100K IOPS
- 현재: HDD 200 IOPS
- 해결: NVMe SSD 4장 RAID

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 확장 전략 | 처리량 증가 | 비용 |
|:---|:---:|:---:|
| **Scale-Up 2×** | 1.5~2× | 2× |
| **Scale-Out 3×** | 2.5~3× | 3× |
| **최적화** | 1.5~5× | 개발 |

### 핵심 공식

```
처리량 = 동시 요청 수 / 응답 시간
처리량 = 대역폭 × 효율
총 처리량 = Σ(인스턴스 처리량) × 효율
```
