# CPI (Cycles Per Instruction)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPI (Cycles Per Instruction)는 컴퓨터가 **명령어 하나를 처리하는 데 평균적으로 소요되는 클럭 사이클 수**를 의미하며, CPU 아키텍처의 효율성을 측정하는 핵심 척도다.
> 2. **가치**: 명령어의 개수($N$)나 클럭 속도($f$)만으로는 알 수 없는 '실행의 질'을 나타내며, 파이프라인 해저드(Hazard)나 캐시 미스(Cache Miss) 등으로 인한 지연 시간을 수치화하여 성능 최적화의 방향을 제시한다.
> 3. **융합**: CISC 설계(높은 CPI, 복잡한 명령어)와 RISC 설계(낮은 CPI, 단순한 명령어)를 구분 짓는 결정적 지표이며, 현대의 슈퍼스칼라 구조에서는 명령어당 사이클 수를 1 미만으로 낮추는 IPC (Instructions Per Cycle) 경쟁으로 심화되고 있다.

+++

## Ⅰ. CPI의 개요

### 1. 정의
- 특정 프로그램을 실행하는 동안 소요된 총 클럭 사이클 수를 실행된 총 명령어 수로 나눈 값이다.
- **$CPI = \frac{\text{Total CPU Clock Cycles}}{\text{Instruction Count}}$**

### 2. 등장 배경: '클럭 속도의 함정 탈출'
- 과거에는 클럭 속도만 높으면 성능이 좋다고 믿었다. 하지만 클럭은 빨라도 명령어 하나를 처리하는 데 너무 많은 사이클이 소요되면 실제 실행 시간은 늘어날 수 있다. 따라서 '명령어 처리 효율'을 정밀하게 측정하기 위해 CPI 개념이 도입되었다.

### 3. 💡 비유: '한 문제를 푸는 데 걸리는 시간'
- CPI는 **'학생이 시험 문제 하나를 푸는 데 평균 몇 분이 걸리는가'**와 같다.
- 문제 수가 적어도 한 문제에 10분씩 걸리는 학생(높은 CPI)보다, 문제 수는 많아도 1분에 한 문제씩 척척 푸는 학생(낮은 CPI)이 결국 시험을 빨리 끝내게 되는 것과 같은 이치다.

+++

## Ⅱ. CPI의 산출 및 영향 요인 (Deep Dive)

### 1. 가중 평균 CPI 계산식
- 프로그램 내의 명령어 종류(산술, 로드/스토어, 분기 등)마다 필요한 사이클이 다르므로 가중 평균을 사용한다.
- **$CPI = \sum_{i=1}^{n} (CPI_i \times \frac{I_i}{I_c})$**
  - $CPI_i$: $i$번째 명령어 종류의 사이클 수
  - $I_i / I_c$: 전체 명령어 중 해당 종류가 차지하는 비율

### 2. CPI에 영향을 미치는 하드웨어적 요인
| 요인 | 영향 | 설명 |
|:---|:---:|:---|
| **파이프라이닝** | **감소** | 단일 명령 시간은 그대로지만, 평균 완료 시간은 $1$에 수렴하게 함 |
| **파이프라인 해저드** | **증가** | 데이터 의존성이나 분기 예측 실패로 인한 지연(Stall) 발생 |
| **메모리 계층 구조** | **증가** | 캐시 미스 발생 시 주 메모리에서 데이터를 가져오는 대기 시간 추가 |
| **명령어 집합 (ISA)** | **가변** | 명령어의 복잡도에 따라 기본 CPI 값이 결정됨 |

### 3. CPU 실행 시간 방정식에서의 위치 (ASCII)
```text
    [ CPU Performance Equation ]
    
    Execution Time =  Instruction Count  ×  CPI  ×  Clock Cycle Time
                      (Software/ISA)      (Arch)    (Hardware/Process)
    
    * CPI는 오직 '마이크로아키텍처'의 설계 수준에 의해 결정되는 값임.
```

- **📢 섹션 요약 비유**: 소프트웨어가 '해야 할 일'을 정하고 하드웨어가 '일의 속도'를 정한다면, CPI는 그 사이에서 '얼마나 요령 있게 일을 처리하느냐'를 결정하는 숙련도와 같습니다.

## Ⅲ. CISC vs RISC의 CPI 비교 분석

| 비교 항목 | CISC (Complex) | RISC (Reduced) |
|:---|:---:|:---:|
| **명령어당 사이클 (CPI)** | **높음 ($2 \sim 15$)** | **낮음 ($1 \sim 1.5$)** |
| **명령어의 복잡도** | 복잡함 (가변 길이) | 단순함 (고정 길이) |
| **코드의 총 개수** | 적음 | 많음 |
| **설계 철학** | "한 방에 해결하자" | "간단하게 여러 번 하자" |

- **📢 섹션 요약 비유**: 한 번 휘두르면 나무를 쓰러뜨리는 무거운 도끼(CISC)와, 가볍지만 여러 번 휘둘러야 하는 날카로운 칼(RISC)의 차이입니다.

## Ⅳ. 실무 적용 및 성능 튜닝 (Strategy)

### 1. 실무 시나리오: 임베디드 루프 최적화
- **상황**: 루프 내부에서 조건문(`if-else`)이 빈번하게 발생하여 실행 속도가 저하됨.
- **판단**: 조건문은 분기 예측 실패 시 파이프라인을 비워야 하므로 CPI를 급격히 높인다. 기술사는 **분기 제거 (Branch Elimination)** 기법이나 **루프 언롤링 (Loop Unrolling)**을 통해 명령어의 흐름을 직선화하여, Stall을 최소화하고 CPI를 이상적인 값인 $1$에 가깝게 낮추는 소프트웨어 최적화 가이드를 제시해야 한다.

### 2. 기술사적 인사이트: CPI 1.0의 벽과 그 너머
- **인사이트**: 전통적인 파이프라인의 한계는 CPI 1.0이다. 하지만 현대의 **슈퍼스칼라 (Superscalar)** 아키텍처는 한 사이클에 여러 명령어를 동시에 실행하여 CPI를 0.5, 0.25 등으로 낮춘다. 기술사는 더 이상 CPI 감소에만 매몰되지 말고, 역수인 **IPC (Instructions Per Cycle)**를 높이는 병렬성 확보 전략(TLP, ILP)을 시스템 설계의 핵심으로 삼아야 한다.

- **📢 섹션 요약 비유**: 한 명이 아무리 빨라져도 한계가 있으니, 이제는 여러 명의 일꾼을 동시에 투입(슈퍼스칼라)하여 효율을 극대화하는 시대입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **아키텍처 평가 기준**: CPU 설계 변경 시 성능 향상 정도를 객관적으로 측정하는 지표 제공.
- **소프트웨어 효율 검증**: 동일 하드웨어에서 컴파일러 최적화 옵션에 따른 코드 효율성 판별.

### 2. 미래 전망
최근의 이기종 컴퓨팅(Heterogeneous Computing) 환경에서는 CPU와 GPU의 CPI 성격이 완전히 다르다. CPU는 낮은 레이턴시(Low CPI)를 지향하고, GPU는 높은 처리량(High Throughput)을 지향한다. 인공지능 시대를 맞아 특정 도메인에 최적화된 **DSA (Domain Specific Architecture)**가 대두되면서, 해당 도메인의 핵심 연산에 대해 최소의 CPI를 보장하는 전용 프로세서 설계가 미래 경쟁력의 핵심이 될 것이다.

- **📢 섹션 요약 비유**: 모든 분야의 만능 선수보다는, 자기 전공 분야에서만큼은 0초 만에 답을 내놓는 전문 선수가 필요한 시대입니다.

- **IPC (Instructions Per Cycle)**: CPI의 역수이자 현대 CPU의 성능 지표.
- **파이프라인 해저드**: CPI를 증가시키는 주범.
- **슈퍼스칼라**: CPI를 1 미만으로 낮추기 위한 병렬 구조.
- **클럭 주기 (Clock Cycle Time)**: CPI와 곱해져 실제 실행 시간을 결정하는 변수.
