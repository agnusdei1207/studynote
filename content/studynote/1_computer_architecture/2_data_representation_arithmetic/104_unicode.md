# 유니코드 (Unicode)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 유니코드(Unicode)는 전 세계 모든 문자에 고유한 코드 포인트(Code Point, U+0000~U+10FFFF)를 할당한 국제 표준으로, 2024년 기준 149,813개 문자(이모지 포함)를 정의한다.
> 2. **가치**: 유니코드 이전에는 각 언어/지역별로 서로 다른 인코딩(ISO-8859, EUC-KR, Shift-JIS 등)을 사용해 "깨진 문자(Mojibake)" 문제가 심각했다. 유니코드는 하나의 문자 집합으로 모든 언어를 통합하여 이 문제를 근본적으로 해결했다.
> 3. **융합**: 유니코드는 "문자 집합(Character Set)"이며, UTF-8/UTF-16/UTF-32는 이를 인코딩하는 방식이다. 현대 시스템은 대부분 UTF-8을 사용하며, Java/C#은 내부적으로 UTF-16, Python 3는 유니코드 문자열을 기본으로 한다.

## Ⅰ. 개요 (Context & Background)

- **개념**: 유니코드(Unicode)는 전 세계 모든 문자에 고유한 숫자(코드 포인트, Code Point)를 할당한 국제 표준 문자 집합(Character Set)이다. 코드 포인트는 `U+` 접두사와 16진수로 표기하며(예: `U+AC00` = '가'), 범위는 `U+0000`~`U+10FFFF`(총 1,114,112개)다. 실제로 사용되는 것은 그중 일부로, 2024년 Unicode 15.1 기준 149,813개 문자가 할당되어 있다.

- **💡 비유**: 유니코드는 **'전 세계 모든 언어의 통합 사전'**과 같다. 영어 'A', 한글 '가', 한자 '中', 이모지 '😀' 모두 하나의 사전에서 고유한 페이지 번호(코드 포인트)를 갖는다. 이 사전만 있으면 어떤 언어든 같은 방식으로 찾을 수 있다.

- **등장 배경 및 발전 과정**:
  1. **인코딩 전쟁 (1970~80년대)**: 미국은 ASCII(7비트), 유럽은 ISO-8859 시리즈(8비트), 한국은 EUC-KR(KS C 5601), 일본은 Shift-JIS, 중국은 GB2312 등 서로 다른 인코딩을 사용했다. 한 파일이 시스템마다 다르게 표시되는 "깨진 문자(Mojibake)"가 빈번했다.
  2. **유니코드의 탄생 (1987~1991)**: Xerox의 Joe Becker, Apple의 Lee Collins, Mark Davis 등이 모여 "모든 문자를 하나의 코드로 통합"하는 프로젝트를 시작했다. 1991년 Unicode Consortium이 설립되고, Unicode 1.0이 발표되었다.
  3. **UTF-8의 등장 (1992)**: Ken Thompson과 Rob Pike가 UTF-8을 설계하여, ASCII와의 하위 호환성을 유지하면서 유니코드를 인코딩할 수 있게 했다. 이것이 유니코드 보급의 결정적 계기가 되었다.
  4. **현대의 표준**: 2024년 현재, 웹의 98%가 UTF-8을 사용하며, 거의 모든 OS(Windows, macOS, Linux, iOS, Android)와 언어(Java, Python 3, JavaScript, Go, Rust)가 유니코드를 기본으로 한다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 유니코드 구조: 평면(Planes)과 코드 포인트

유니코드는 17개의 평면(Plane 0~16)으로 구성되며, 각 평면은 65,536(2¹⁶)개의 코드 포인트를 가진다.

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │         유니코드 평면(Planes) 구조                                            │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │   전체 코드 포인트 범위: U+0000 ~ U+10FFFF (1,114,112개)                      │
  │                                                                              │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  Plane 0 (BMP, Basic Multilingual Plane): U+0000 ~ U+FFFF              ││
  │   │  ─────────────────────────────────────────────────────────────────────  ││
  │   │  • 대부분의 현대 언어 문자 (영어, 한글, 한자, 히라가나 등)                ││
  │   │  • ASCII (U+0000~U+007F), Latin-1 Supplement (U+0080~U+00FF)           ││
  │   │  • 한글 음절 (U+AC00~U+D7A3, 11,172자)                                 ││
  │   │  • CJK 통합 한자 (U+4E00~U+9FFF, ~20,992자)                            ││
  │   │  • 65,536개 중 ~55,000개 할당됨                                        ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  Plane 1 (SMP, Supplementary Multilingual Plane): U+10000 ~ U+1FFFF    ││
  │   │  ─────────────────────────────────────────────────────────────────────  ││
  │   │  • 고대 문자 (이집트 신성문자, 쐐기문자 등)                              ││
  │   │  • 이모지 (Emoji, U+1F600~U+1F64F 등)                                  ││
  │   │  • 수학 기호, 음악 기호                                                 ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  Plane 2 (SIP, Supplementary Ideographic Plane): U+20000 ~ U+2FFFF     ││
  │   │  ─────────────────────────────────────────────────────────────────────  ││
  │   │  • CJK 확장 한자 (희귀 한자)                                            ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  Plane 3~13: 할당되거나 예약됨                                          ││
  │   │  Plane 14 (SSP, Supplementary Special-purpose Plane): 태그, 변형       ││
  │   │  Plane 15~16 (PUA, Private Use Area): 사용자 정의 영역                  ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   ⚠ BMP(U+0000~U+FFFF)는 UTF-16에서 1코드 유닛(2바이트)로 표현 가능        │
  │      BMP 외(Supplementary)는 UTF-16에서 2코드 유닛(서로게이트 쌍) 필요      │
  │                                                                              │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 유니코드는 17개 평면(Plane 0~16)으로 구성된다. 각 평면은 65,536개의 코드 포인트를 가진다. Plane 0은 BMP(Basic Multilingual Plane)로, 대부분의 현대 언어 문자(ASCII, 한글, 한자 등)가 여기에 있다. Plane 1(SMP)에는 이모지와 고대 문자가 있다. Plane 2(SIP)에는 확장 한자가 있다. BMP는 UTF-16에서 2바이트로 표현되지만, BMP 외 문자는 서로게이트 쌍(Surrogate Pair)이라는 4바이트로 표현된다. 이것이 UTF-16의 복잡성의 원인이다.

### 한글의 유니코드 인코딩: 완성형 vs 조합형

한글은 유니코드에서 완성형(초성+중성+종성이 조합된 음절)으로 인코딩된다.

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │         한글 유니코드 인코딩 (완성형)                                         │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │   한글 음절 범위: U+AC00 ~ U+D7A3 (총 11,172자)                              │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  구조: (초성 인덱스 × 21 × 28) + (중성 인덱스 × 28) + 종성 인덱스     ││
  │   │                                                                         ││
  │   │  초성 19개: ㄱ, ㄲ, ㄴ, ㄷ, ㄸ, ㄹ, ㅁ, ㅂ, ㅃ, ㅅ, ㅆ,                ││
  │   │            ㅇ, ㅈ, ㅉ, ㅊ, ㅋ, ㅌ, ㅍ, ㅎ                              ││
  │   │  중성 21개: ㅏ, ㅐ, ㅑ, ㅒ, ㅓ, ㅔ, ㅕ, ㅖ, ㅗ, ㅘ, ㅙ,                ││
  │   │            ㅚ, ㅛ, ㅜ, ㅝ, ㅞ, ㅟ, ㅠ, ㅡ, ㅢ, ㅣ                      ││
  │   │  종성 28개: 없음, ㄱ, ㄲ, ㄳ, ㄴ, ㄵ, ㄶ, ㄷ, ㄹ, ㄺ, ㄻ,              ││
  │   │            ㄼ, ㄽ, ㄾ, ㄿ, ㅀ, ㅁ, ㅂ, ㅄ, ㅅ, ㅆ, ㅇ, ㅈ, ㅊ, ㅋ,    ││
  │   │            ㅌ, ㅍ, ㅎ                                                  ││
  │   │                                                                         ││
  │   │  총 조합: 19 × 21 × 28 = 11,172자                                     ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   코드 포인트 계산 예시:                                                     │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  '한' (초성 ㅎ=18, 중성 ㅏ=0, 종성 ㄴ=4)                               ││
  │   │  U+AC00 + (18 × 21 × 28) + (0 × 28) + 4                               ││
  │   │  = U+AC00 + 10584 + 0 + 4                                              ││
  │   │  = U+AC00 + 10588                                                      ││
  │   │  = U+D55C ✓                                                            ││
  │   │                                                                         ││
  │   │  '가' (초성 ㄱ=0, 중성 ㅏ=0, 종성 없음=0)                              ││
  │   │  U+AC00 + (0 × 21 × 28) + (0 × 28) + 0                                ││
  │   │  = U+AC00 ✓                                                            ││
  │   │                                                                         ││
  │   │  '힣' (초성 ㅎ=18, 중성 ㅣ=20, 종성 ㅎ=27)                             ││
  │   │  U+AC00 + (18 × 21 × 28) + (20 × 28) + 27                              ││
  │   │  = U+AC00 + 10584 + 560 + 27                                           ││
  │   │  = U+AC00 + 11171                                                      ││
  │   │  = U+D7A3 ✓ (마지막 한글 음절)                                         ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   ⚠ 유니코드 한글은 완성형만 인코딩 (조합형은 별도 처리)                   │
  │      초성/중성/종성 단독 문자도 별도 코드 포인트 존재 (U+3131~U+318E)       │
  │                                                                              │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 유니코드의 한글은 완성형으로, 초성 19개 × 중성 21개 × 종성 28개 = 11,172자가 U+AC00~U+D7A3에 연속으로 할당되어 있다. '가'는 U+AC00, '힣'은 U+D7A3이다. 코드 포인트 계산은 `(초성 × 21 × 28) + (중성 × 28) + 종성`을 U+AC00에 더하면 된다. 예를 들어 '한'은 초성 ㅎ(18), 중성 ㅏ(0), 종성 ㄴ(4)이므로 U+AC00 + 10588 = U+D55C다. 초성/중성/종성 단독 문자(ㄱ, ㅏ 등)는 U+3131~U+318E(호환용 자모)에 별도로 존재한다.

### 유니코드 vs 인코딩 (UTF-8/UTF-16/UTF-32)

유니코드는 "문자 집합"이고, UTF-8/UTF-16/UTF-32는 이를 "인코딩"하는 방식이다.

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │         유니코드 코드 포인트 ↔ UTF-8/UTF-16/UTF-32 인코딩                   │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │   예시: '한' (U+D55C) 인코딩                                                 │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  코드 포인트: U+D55C = 0xD55C = 1101010101011100                       ││
  │   │                                                                         ││
  │   │  UTF-8 (3바이트):                                                       ││
  │   │    0xD55C는 0x0800~0xFFFF 범위 → 3바이트 패턴                          ││
  │   │    1110xxxx 10xxxxxx 10xxxxxx                                          ││
  │   │    → 11101101 10010101 10011100                                        ││
  │   │    → 0xED 0x95 0x9C                                                    ││
  │   │                                                                         ││
  │   │  UTF-16 (2바이트):                                                      ││
  │   │    BMP 범위 → 그대로 2바이트                                            ││
  │   │    → 0xD55C                                                            ││
  │   │                                                                         ││
  │   │  UTF-32 (4바이트):                                                      ││
  │   │    코드 포인트 그대로 4바이트                                           ││
  │   │    → 0x0000D55C                                                        ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   예시: '😀' (U+1F600, BMP 외) 인코딩                                       │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  코드 포인트: U+1F600 = 0x1F600 = 11111011000000000                    ││
  │   │                                                                         ││
  │   │  UTF-8 (4바이트):                                                       ││
  │   │    0x10000~0x10FFFF 범위 → 4바이트 패턴                                ││
  │   │    11110xxx 10xxxxxx 10xxxxxx 10xxxxxx                                 ││
  │   │    → 11110000 10011111 10011000 10000000                               ││
  │   │    → 0xF0 0x9F 0x98 0x80                                               ││
  │   │                                                                         ││
  │   │  UTF-16 (4바이트, 서로게이트 쌍):                                       ││
  │   │    High Surrogate: (0x1F600 - 0x10000) >> 10 + 0xD800                  ││
  │   │                 = 0xF6 + 0xD800 = 0xD83E                               ││
  │   │    Low Surrogate: (0x1F600 - 0x10000) & 0x3FF + 0xDC00                 ││
  │   │                 = 0x000 + 0xDC00 = 0xDE00                              ││
  │   │    → 0xD83E 0xDE00                                                      ││
  │   │                                                                         ││
  │   │  UTF-32 (4바이트):                                                      ││
  │   │    → 0x0001F600                                                         ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   ⚠ UTF-8: ASCII는 1바이트, 한글은 3바이트, 이모지는 4바이트               │
  │      UTF-16: BMP는 2바이트, BMP 외는 4바이트 (서로게이트 쌍)                │
  │      UTF-32: 모든 문자가 4바이트 (공간 비효율)                             │
  │                                                                              │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 유니코드는 "무엇(코드 포인트)"을 정의하고, UTF-8/UTF-16/UTF-32는 "어떻게(인코딩)" 저장할지를 정의한다. '한'(U+D55C)은 UTF-8에서 3바이트(0xED 0x95 0x9C), UTF-16에서 2바이트(0xD55C), UTF-32에서 4바이트(0x0000D55C)로 인코딩된다. 이모지 '😀'(U+1F600)는 UTF-8에서 4바이트, UTF-16에서 서로게이트 쌍(0xD83E 0xDE00)으로 4바이트, UTF-32에서 4바이트다. UTF-8은 ASCII에서 1바이트로 효율적이고, UTF-16은 BMP에서 2바이트로 효율적이지만 서로게이트 쌍이 복잡하다. UTF-32는 모든 문자가 4바이트로 단순하지만 공간 비효율적이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: UTF-8 vs UTF-16 vs UTF-32

| 비교 항목 | UTF-8 | UTF-16 | UTF-32 |
|:---|:---|:---|:---|
| **바이트 수** | 1~4바이트 가변 | 2 또는 4바이트 | 4바이트 고정 |
| **ASCII** | 1바이트 (100% 호환) | 2바이트 (비효율) | 4바이트 (비효율) |
| **한글/한자** | 3바이트 | 2바이트 | 4바이트 |
| **이모지** | 4바이트 | 4바이트 (서로게이트) | 4바이트 |
| **인덱싱** | O(n) (가변 길이) | O(n) (서로게이트) | O(1) (고정 길이) |
| **주 사용처** | 웹(98%), 파일 | Java, C#, Windows | 내부 처리, 정렬 |

### 과목 융합 관점
- **프로그래밍 언어**: Python 3의 `str`은 유니코드 문자열이며, 내부적으로 UTF-8(PEP 393) 또는 UTF-16/UTF-32로 저장된다. Java의 `String`은 UTF-16이다. Go/Rust는 UTF-8을 기본으로 한다
- **데이터베이스(DB)**: MySQL의 `utf8`은 3바이트(이모지 미지원), `utf8mb4`는 4바이트(이모지 지원)다. PostgreSQL, Oracle은 UTF-8을 완전히 지원한다
- **웹(Web)**: HTML5는 UTF-8을 권장한다. HTTP 헤더 `Content-Type: text/html; charset=utf-8`로 지정한다. JSON은 기본적으로 UTF-8이다

## Ⅳ. 실무 적용 및 기술사적 판단
### 실무 시나리오
1. **시나리오 — MySQL utf8 vs utf8mb4**: 이모지가 포함된 사용자 댓글을 저장해야 하는데, MySQL의 `utf8` 컬럼에 저장 시 "Incorrect string value" 오류가 발생한다. 기술사는 `utf8mb4`로 컬럼을 변경하고, `character_set_server`도 `utf8mb4`로 설정한다
2. **시나리오 — Java의 서로게이트 쌍 처리**: Java `String.length()`가 이모지 '😀'에 대해 2를 반환한다(서로게이트 쌍이 2개의 `char`로 표현됨). 기술사는 `codePointCount()`를 사용하여 실제 문자 수를 계산한다
3. **시나리오 — 크로스 플랫폼 텍스트 파일**: Windows(UTF-16 LE BOM)에서 작성한 텍스트 파일을 Linux(UTF-8)에서 열면 깨진다. 기술사는 `iconv` 또는 Python으로 UTF-16→UTF-8 변환을 수행한다
### 도입 체크리스트
- **기술적**: 시스템이 UTF-8을 지원하는가? 레거시 시스템은 EUC-KR, Shift-JIS 등을 사용할 수 있다
- **운영·보안적**: 이모지, 4바이트 문자를 처리해야 하는가? MySQL `utf8`이 아닌 `utf8mb4`를 사용해야 한다
### 안티패턴
- **UTF-8로 가정하고 바이트 단위 처리**: UTF-8 문자열에서 `s[3]`으로 4번째 "문자"에 접근하면, 바이트 단위 접근으로 깨질 수 있다. 코드 포인트 단위 처리가 필요하다
- **MySQL utf8 사용**: 이모지, 희귀 한자가 저장되지 않는다. `utf8mb4`를 사용해야 한다

## 👶 어린이를 위한 3줄 비유 설명
1. 유니코드는 **'전 세계 모든 언어의 통합 사전'** 같아요. 영어 'A', 한글 '가', 이모지 '😀' 모두 하나의 사전에서 고유한 번호(코드 포인트)를 갖고 있어요
2. 유니코드 이전에는 나라마다 다른 사전을 써서, 한국에서 쓴 글이 일본에서 깨져 보였어요. 유니코드 덕분에 모두가 같은 사전을 써서 글이 안 깨져요!
3. UTF-8은 이 사전을 컴퓨터에 저장하는 방법이에요. 영어는 1칸, 한글은 3칸, 이모지는 4칸을 써요. 공간을 아끼면서 모든 글자를 저장할 수 있어요!
