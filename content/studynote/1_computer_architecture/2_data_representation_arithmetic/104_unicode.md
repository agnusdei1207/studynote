# 104. 유니코드 (Unicode)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 유니코드(Unicode)는 전 세계 모든 문자 시스템에 고유한 번호(Code Point, U+0000~U+10FFFF)를 부여하는 단일 문자집합(Character Set) 표준으로, 2024년 기준 149,813자 이상을 정의하여 "하나의 코드로 모든 언어를 다루는" 근본적인 해결책을 제시한다.
> 2. **가치**: 기존의 코드페이지(Code Page) 방식(ISO-8859, EUC-KR 등)에서 발생하던 "깨진 문자(Mojibake)" 문제를 근본적으로 해소하여, 글로벌 소프트웨어 배포 비용을 획기적으로 절감하고 국제화(i18n) 및 로컬라이제이션(l10n)을 단순화했다.
> 3. **융합**: 유니코드(추상적 문자 집합)와 UTF-8/UTF-16(물리적 인코딩 방식)의 철저한 구분을 요구하며, 현대 웹(98% 점유율)과 프로그래밍 언어(Python 3, Rust), 데이터베이스(MySQL utf8mb4)의 표준 인터페이스로 자리 잡았다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 및 정의
유니코드(Unicode)는 1980년대 후반부터 개발되기 시작한 국제 표준 문자 집합(Character Set)으로, 전 세계의 문자, 기호, 이모지 등에 고유한 식별자인 **코드 포인트(Code Point)**를 부여한다. 
- **코드 포인트(Code Point)**: `U+`로 시작하는 16진수로 표현되는 논리적 번호 (예: `U+0041`은 'A', `U+AC00`은 '가').
- **범위**: `U+0000`부터 `U+10FFFF`까지 총 1,114,112개의 공간을 가지며, 이를 17개의 평면(Plane)으로 분리管理한다.
- **목표**: 다국어 텍스트 처리를 위한 "하나의 문자 집합, 하나의 인코딩"을 실현하는 것(이상적인 목표는 실제로는 UTF-8, UTF-16 등 다양한 인코딩이 파생되었으나, 문자 집합 자체는 통일되었다).

### 2. 등장 배경: 인코딩의 혼란기
컴퓨팅 초창기인 1960~70년대에는 미국 중심의 **ASCII (American Standard Code for Information Interchange)**(7비트, 128자)로 충분했다. 하지만 컴퓨터가 전 세계로 보급되면서 문제가 발생했다.

1. **문제점: 더블 바이트 인코딩 및 충돌**
   - 1바이트(256자)로는 영어 외에 자국어를 표현하기 부족했다.
   - 각국은 2바이트를 사용하거나, 0x80~0xFF 영역을 자국어에 맞게 정의했다.
   - **한국**: 완성형(EUC-KR, KS C 5601), 조합형(Johab)
   - **일본**: Shift-JIS, EUC-JP
   - **유럽**: ISO-8859-1(Latin-1)
   - **결과**: 같은 바이트 스트림이 시스템(또는 언어 설정)에 따라 전혀 다른 문자로 해석되는 **Mojibake (문자 깨짐)** 현상이 일상화되었다.

2. **해결책: 유니코드의 등장**
   - 1991년 **Unicode Consortium** 설립. Apple, Microsoft, IBM 등이 참여.
   - 모든 문자를 하나의 거대한 코드 맵에 넣어, 어디서든 동일한 코드로 동일한 문자를 보게 만들었다.

### 3. 아키텍처적 핵심
유니코드는 단순히 "숫자를 문자에 매핑"하는 것을 넘어, 복잡한 언어적 특성(결합 문자, 바이디 텍스트, 정규화 등)을 처리하기 위한 복잡한 데이터베이스이자 알고리즘의 집합체이다.

> **📢 섹션 요약 비유**
> 유니코드는 세계 모든 언어의 전화번호부를 **'전 세계 단일 국제 번호부'**로 통합한 것과 같습니다. 과거에는 나라마다 전화번호부(인코딩)가 달라서 서로 전화를 걸 수 없거나(깨짐), 엉뚱한 사람에게 연결되는 문제가 있었지만, 유니코드는 모든 사람(문자)에게 유니크한 국제 번호(U+...)를 부여하여 어디서든 정확히 찾을 수 있게 해주었습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 구조적 설계: 17개 평면(Planes)
유니코드의 주소 공간은 17개의 그룹, 즉 **평면(Plane)**으로 나뉜다. 각 평면은 65,536(2¹⁶)개의 코드 포인트를 가진다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    유니코드 코드 공간 구조 (Unicode Planes)                  │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  Code Point 범위: U+0000 (0) ~ U+10FFFF (1,114,111)                         │
│                                                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐    │
│  │ Plane 0: BMP (Basic Multilingual Plane)                             │    │
│  │ 범위: U+0000 ~ U+FFFF                                               │    │
│  │ ─────────────────────────────────────────────────────────────────   │    │
│  │ • 역사: 1991년 유니코드 1.0 때부터 존재. 가장 중요한 평면.          │    │
│  │ • 내용: 세계 주요 언어 대부분(영어, 한글, 한자, 히라가나 등),        │    │
│  │         Latin-1 보조 문자, 기술 기호.                                │    │
│  │ • 특징: UTF-16 인코딩에서 단일 16비트 워드(2바이트)로 표현 가능.    │    │
│  └─────────────────────────────────────────────────────────────────────┘    │
│                                                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐    │
│  │ Plane 1: SMP (Supplementary Multilingual Plane)                     │    │
│  │ 범위: U+10000 ~ U+1FFFF                                            │    │
│  │ ─────────────────────────────────────────────────────────────────   │    │
│  │ • 내용: 고대 문자(이집트 신성문자, 설형문자), 역사적 문자,           │    │
│  │         음악 기호, 수학 기호, 그리고...                             │    │
│  │         **이모지(Emoji)**가 대부분 이곳에 위치 (예: U+1F600 😀).    │    │
│  │ • 특징: UTF-16에서 '서로게이트 쌍(Surrogate Pair)'(4바이트) 필요.   │    │
│  └─────────────────────────────────────────────────────────────────────┘    │
│                                                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐    │
│  │ Plane 2: SIP (Supplementary Ideographic Plane)                     │    │
│  │ 범위: U+20000 ~ U+2FFFF                                            │    │
│  │ ─────────────────────────────────────────────────────────────────   │    │
│  │ • 내용: CJK 통합 한자 확장 B(CJK Ext-B) 이상. 희귀 한자 및 옥편.    │    │
│  │ • 특징: BMP에 들어가지 못한 나머지 수만 개의 한자가 포함됨.         │    │
│  └─────────────────────────────────────────────────────────────────────┘    │
│                                                                              │
│  Plane 3~13  : 예약됨 (Reserved for future use)                             │
│  Plane 14    : SSP (Supplementary Special-purpose Plane) - 태그, 형식 지정 │
│  Plane 15~16 : PUA (Private Use Area) - 사용자 정의 영역                    │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
유니코드는 0x100000 개의 공간을 65536개 크기의 17개 평면으로 나눕니다. 가장 중요한 것은 **Plane 0 (BMP)**으로, 여기에 영어와 한글 등 우리가日常 사용하는 문자의 99%가 들어 있습니다. **Plane 1**에는 최근에 많이 사용하는 이모지와 고대 문자가 들어 있습니다. UTF-16 같은 인코딩을 사용할 때, Plane 0은 2바이트로 표현 가능하지만, Plane 1 이상은 2바이트로 표현 불가능하므로 4바이트(서로게이트 쌍)를 사용해야 하므로 처리 시 주의해야 합니다.

### 2. 핵심 알고리즘 및 내부 동작: 한글 인코딩의 수학
한글은 유니코드에서 **완성형(Precomposed Syllable)** 방식과 **자모 분해형(Jamo)** 방식을 모두 지원한다. 가장 널리 쓰이는 완성형(Hangul Syllables)은 수학적 계열을 이용해 코드를 할당한다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                한글 유니코드 자동 할당 알고리즘 (Hangul Algorithm)           │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  1. 기준점 및 파라미터                                                        │
│     • Base (Start): U+AC00 ('가')                                            │
│     • S (Initial Consonants, 초성): 19개 (ㄱ~ㅎ, Index 0~18)                 │
│     • M (Medial Vowels, 중성): 21개 (ㅏ~ㅣ, Index 0~20)                     │
│     • T (Final Consonants, 종성): 28개 (없음~ㅎ, Index 0~27)                │
│                                                                              │
│  2. 공식 (Code Point Formula)                                                │
│     Code_Point = U+AC00 + (S_Index × 21 × 28) + (M_Index × 28) + T_Index    │
│                                                                              │
│  3. 계산 예시: '한' (초성: ㅎ(18), 중성: ㅏ(0), 종성: ㄴ(4))                 │
│     • 초성 오프셋: 18 × 588 = 10584                                         │
│     • 중성 오프셋: 0 × 28 = 0                                               │
│     • 종성 오프셋: 4                                                        │
│     • 합계: 10584 + 0 + 4 = 10588                                           │
│     • 최종: U+AC00 (44032) + 10588 = U+D55C (54620)                         │
│                                                                              │
│  4. 내부 테이블 구조 (Index Mapping)                                          │
│     ┌──────┬──────┬──────┬──────┬──────┐                                   │
│     │ 초성 │ Index │ 중성 │ Index │ 종성 │ Index │                          │
│     ├──────┼──────┼──────┼──────┼──────┼──────┤                          │
│     │ ㄱ   │   0  │ ㅏ   │   0  │ 없음 │   0  │                          │
│     │ ...  │ ...  │ ...  │ ...  │ ㄱ   │   1  │                          │
│     │ ㅎ   │  18  │ ㅣ   │  20  │ ...  │ ...  │                          │
│     └──────┴──────┴──────┴──────┴──────┴──────┘                                   │
│                                                                              │
│  💡 UTF-8 인코딩 변환 과정                                                    │
│     '한' (U+D55C)는 3바이트 한글 범위에 해당                                 │
│     1. Binary: 1101 0101 0101 1100 (16비트)                                 │
│     2. UTF-8 패턴 적용 (1110xxxx 10xxxxxx 10xxxxxx)                           │
│     3. 결과: 11101101 10010101 10011100 -> 0xED 0x95 0x9C                    │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
한글 완성형 코드는 단순 나열식이 아니라 수학적 순서를 따릅니다. 모든 초성(19)에 대해 모든 중성(21)을 조합하고, 그 각각에 모든 종성(28)을 붙이는 **조합론적 접근(Cartesian Product)**을 사용하여 11,172자를 만듭니다. 이 덕분에 프로그래머는 '가'~'힣' 사이의 모든 글자를 반복문(Loop)으로 생성하거나, 한 글자를 초/중/종성으로 분해해내는 로직을 쉽게 구현할 수 있습니다. (예: 오타 변환, 초성 검색)

> **📢 섹션 요약 비유**
> 한글 유니코드 영역은 **'자동차 번호판 제조 시스템'**과 같습니다. 자동차 번호(문자)가 "지역코드(초성) + 숫자(중성) + 일련번호(종성)"의 조합으로 만들어지듯이, 한글 코드 포인트도 `(초성×588) + (중성×28) + 종성` 공식에 따라 순차적으로 배정됩니다. 덕분에 사전에 없는 글자는 없지만, 번호판 조합 규칙에 맞지 않는 조합은(예: 'ㄱㅏㅏ') 존재할 수 없게 됩니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

유니코드는 "무엇을" 정의할 뿐이며, 이를 메모리나 파일에 "어떻게" 저장할지는 인코딩(Encoding)이 결정한다. 기술사는 **문자 집합(CCS)**과 **문자 인코딩(CES)**을 명확히 구분해야 한다.
