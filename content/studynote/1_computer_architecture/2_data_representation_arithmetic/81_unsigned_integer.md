# 부호 없는 정수 (Unsigned Integer)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 부호 없는 정수(Unsigned Integer)는 이진 데이터의 모든 비트를 오로지 양수(0 포함)의 크기를 표현하는 데만 온전히 사용하는 데이터 자료형이다.
> 2. **가치**: 부호 비트(MSB)를 숫자 크기 계산에 양보함으로써, 동일한 메모리 크기(비트 수) 내에서 표현할 수 있는 최대 양수의 범위를 정확히 2배로 확장시킨다.
> 3. **융합**: 메모리 주소(포인터), 배열의 인덱스, 네트워크 패킷의 길이, 암호학의 모듈러 연산 등 논리적으로 '음수'가 존재할 수 없는 도메인에서 시스템 무결성을 지키는 핵심 방어선 역할을 한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 부호 없는 정수(Unsigned Integer)는 음수를 표현할 필요가 없는 상황에서 사용되는 컴퓨터의 정수 표현 방식이다. N비트가 주어졌을 때, 최상위 비트(MSB)를 부호 판별에 쓰지 않고 일반 비트와 동일하게 $2^{N-1}$의 가중치로 계산하므로, 표현 범위가 $0$부터 $2^N - 1$까지로 넓어진다.

- **💡 비유**: 부호 없는 정수는 자동차의 **'주행 거리계(Odometer)'**와 같다. 자동차는 후진을 하더라도 주행 거리는 마이너스(-)가 되지 않고 오직 0부터 위로만 계속 올라간다. 음수의 개념 자체가 없으며, 한계치(예: 999,999km)에 도달하면 다시 0으로 돌아가는(랩어라운드) 특성을 갖는다.

- **등장 배경 및 발전 과정**:
  1. **초기 메모리 제약**: 과거 8비트, 16비트 시절에는 메모리와 레지스터가 극도로 부족했다. 양수만 필요한 데이터(예: 나이, 픽셀 좌표)에 부호 있는 정수를 쓰면 절반의 공간(- 범역)을 낭비하게 되므로, 이를 아끼기 위해 모든 비트를 양수로 취급하는 방식이 필수적이었다.
  2. **C언어의 도입 (`unsigned` 키워드)**: 시스템 프로그래밍의 표준인 C언어가 `unsigned`라는 명시적 키워드를 도입하면서, 프로그래머는 컴파일러에게 "이 변수는 절대 음수가 될 수 없으니 가장 큰 비트까지 크기로 계산하라"고 명령할 수 있게 되었다.
  3. **메모리 어드레싱의 표준**: 가상 메모리 공간이 커지면서 메모리 주소를 표현하는 포인터는 필연적으로 부호 없는 정수(예: `uintptr_t`, `size_t`)로 발전하여 0 주소부터 끝 주소까지 전체 공간을 매핑하게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 표현 범위와 비트 가중치 (8비트 기준)

부호 있는(Signed) 정수와 부호 없는(Unsigned) 정수는 물리적인 비트 패턴이 동일하더라도, **CPU(ALU)가 이를 해석하는 가중치 맵핑 방식**이 다르다.

| 타입 | MSB (Bit 7)의 가중치 | 표현 범위 | 수식 |
|:---|:---|:---|:---|
| **Signed (2's Complement)** | $-128$ (음수 가중치) | $-128 \sim +127$ | $-2^7 \times d_7 + \sum (d_i \times 2^i)$ |
| **Unsigned Integer** | $+128$ (양수 가중치) | $0 \sim +255$ | $+2^7 \times d_7 + \sum (d_i \times 2^i)$ |

---

### Unsigned의 동작 메커니즘과 랩어라운드 (Wrap-around)

부호 없는 정수의 가장 중요한 하드웨어적 특징은 연산 한계 초과 시 부호가 바뀌는 '오버플로우(Overflow)'가 아니라, 다시 0으로 돌아가는 **'랩어라운드(Wrap-around)'**가 발생한다는 점이다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         Unsigned 정수의 모듈러 연산 (랩어라운드 원리)             │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  [8비트 Unsigned (최대치 255) 덧셈]                              │
  │       1111 1111  (255, 최댓값)                                │
  │    +  0000 0001  (  1)                                       │
  │    ────────────                                              │
  │     1 0000 0000  (256)                                       │
  │     ▲                                                        │
  │  Carry(올림수) 발생!                                            │
  │  하지만 레지스터는 8비트뿐이므로 맨 앞의 1은 버려짐 (Carry Flag 세팅)  │
  │                                                              │
  │  ▶ 최종 저장된 값: 0000 0000 (0)                             │
  │                                                              │
  │  [8비트 Unsigned (최소치 0) 뺄셈]                                │
  │       0000 0000  (  0)                                       │
  │    -  0000 0001  (  1)                                       │
  │    ────────────                                              │
  │       1111 1111  (255)                                       │
  │                                                              │
  │  ▶ 0에서 1을 빼면 마이너스가 되는 것이 아니라 최댓값으로 순환(Wrap)됨. │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 부호 없는 정수는 수학적으로 **모듈러(Modular) 산술**을 따른다. 8비트 시스템은 Modulo $2^8 = 256$ 시스템과 같다. 가장 큰 수인 255에서 1을 더하면 256이 되는 것이 아니라 시계바늘처럼 다시 0으로 돌아간다. 반대로 0에서 1을 빼면 가장 큰 수인 255로 돌아간다. 이 속성은 암호학(해시 함수 등)에서 값이 일정한 범위 내에서 무한히 순환해야 할 때 매우 유용하게 쓰이지만, 루프(Loop) 종료 조건을 `i >= 0`으로 잘못 설정하면 영원히 끝나지 않는 무한 루프 버그(언더플로우 랩어라운드)를 유발하는 양날의 검이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 비교: Unsigned vs Signed 연산 처리

CPU 내부의 ALU(산술논리장치)는 덧셈이나 뺄셈을 할 때 이것이 Signed인지 Unsigned인지 **모른다**. ALU는 그저 기계적으로 비트를 더할 뿐이며, 그 결과를 어떻게 해석할지는 전적으로 명령어(Opcode)와 컴파일러에 달려 있다.

| 비교 항목 | Unsigned 연산 | Signed 연산 |
|:---|:---|:---|
| **ALU 덧셈/뺄셈 회로** | **완전히 동일함** (회로 공유) | **완전히 동일함** (회로 공유) |
| **상태 플래그 (CPU Flag)** | **CF (Carry Flag)**를 확인하여 오차 판별 | **OF (Overflow Flag)**를 확인하여 오차 판별 |
| **크기 비교 분기 명령어**| `JA` (Jump if Above), `JB` (Jump if Below) | `JG` (Jump if Greater), `JL` (Jump if Less) |
| **우측 시프트 연산 (`>>`)**| 논리 시프트 (항상 0으로 채움) | 산술 시프트 (MSB 값으로 채워 부호 유지) |

### 과목 융합 관점

- **운영체제 (OS)**: 시스템 콜(예: 파일 읽기)에서 반환값을 설계할 때, 읽은 바이트 수(양수)와 에러 상태(음수, 예: `-1`)를 하나의 레지스터로 반환하기 위해 `ssize_t` (Signed Size)를 사용한다. 반면, 순수하게 메모리 블록의 크기만을 파악할 때는 절대 음수가 될 수 없는 `size_t` (Unsigned Size)를 사용하여 64비트 공간 전체를 활용한다.
- **컴퓨터 네트워크 (Network)**: TCP/IP 헤더의 포트 번호(Port Number)는 16비트 Unsigned Integer로 정의된다. 따라서 포트 번호는 $0$에서 $65,535$ ($2^{16}-1$)까지 존재하며 음수 포트는 존재하지 않는다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 무한 루프 버그 (역방향 순회)**: C/C++에서 배열을 뒤에서부터 앞으로 순회하기 위해 인덱스 변수를 `unsigned int`로 선언했다.
   ```cpp
   for (unsigned int i = 10; i >= 0; i--) {
       print(array[i]);
   }
   ```
   - **문제**: `i`가 0이 되어 0번째 요소를 처리한 후, `i--`가 실행되면 `i`는 -1이 되지 않고 랩어라운드가 발생하여 `4294967295` (32비트 최대값)가 된다. 이 값은 0보다 크므로 루프 조건(`i >= 0`)은 항상 참이 되어 시스템은 치명적인 무한 루프와 메모리 접근 위반(Segfault)에 빠진다.
   - **해결책**: 감소하는 루프의 인덱스는 반드시 Signed 타입(`int`)을 사용하거나, 루프의 종료 조건을 `i != -1` 과 같은 트릭 대신 `i < 11`처럼 역방향 인덱싱 논리를 재설계해야 한다.

2. **시나리오 — 타입 변환 공격 (Type Confusion / Integer Underflow)**: 웹 서버의 이미지 업로드 모듈에서 이미지의 너비를 검사하는 로직이 있다.
   ```cpp
   int width = get_user_input();
   if (width < 1000) {
       unsigned int buf_size = width * height;
       allocate_memory(buf_size);
   }
   ```
   - **문제**: 해커가 `width` 값으로 `-1`을 입력하면, 첫 번째 검사(`-1 < 1000`)를 통과한다. 하지만 이 값이 `unsigned int`로 암시적 형변환(Implicit Casting)되는 순간 `-1`은 `0xFFFFFFFF` (42억)로 둔갑한다. 결과적으로 시스템은 42억 바이트(4GB)의 메모리를 할당하려다 OOM(Out of Memory)으로 서버가 다운된다.
   - **해결책**: 입력값을 검증할 때는 시스템 경계에서부터 명시적으로 Unsigned 타입을 사용하여 음수 입력 자체를 원천 차단하거나, 비교 연산 전에 데이터 타입을 일치시켜 부호 변환으로 인한 크기 왜곡(Type Confusion)을 막아야 한다.

### 도입 체크리스트 (방어적 프로그래밍)
- **기술적**: `signed` 변수와 `unsigned` 변수를 비교(`==`, `<`)하는 코드가 있는가? (C/C++에서는 `signed`가 `unsigned`로 자동 형변환되므로, 음수가 거대한 양수로 취급되어 논리적 오류가 발생한다. 컴파일러 경고 플래그 `-Wsign-compare`를 활성화해야 한다).
- **운영적**: 시스템 시계나 타임스탬프(Unix Time)가 `unsigned int`(32비트)로 선언되어, 2106년에 랩어라운드가 발생하는 문제(Y2K38 문제의 연장선)에 노출되어 있지는 않은가?

### 안티패턴
- **배열 인덱스에 Signed 타입 사용**: Java는 언어 설계상 `unsigned` 타입을 지원하지 않아 배열 인덱스가 Signed 32비트(`int`)다. 이로 인해 이론상 자바 배열의 최대 크기는 40억 개가 아닌 21억 개($2^{31}-1$)로 제한된다. C/C++나 Rust처럼 하드웨어 통제가 중요한 언어에서는 배열 인덱싱에 무조건 `size_t` (Unsigned)를 사용하는 것이 표준이다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 적용 사례 | 기대효과 |
|:---|:---|:---|
| **정량** | 메모리 주소 및 길이를 나타내는 변수에 Unsigned 적용 | 동일 비트 수 대비 표현 한계 **2배 확장** |
| **정성** | 컴파일 타임에 형변환 경고(-Wall -Wextra) 강제화 | 정수 오버플로우/언더플로우 보안 취약점 **사전 차단** |
| **정성** | 하드웨어 레지스터 제어 변수를 `uint32_t`로 고정 | 최상위 비트(Bit 31) 조작 시 부호 오류에 의한 오작동 방지 |

### 미래 전망
현대 프로그래밍 언어들은 개발자의 실수를 줄이기 위해 아예 부호 없는 정수(Unsigned)의 사용을 제한하거나 금지하는 방향(예: Java, Kotlin의 일부 기본 타입 제약)과, 반대로 시스템 프로그래밍을 위해 더욱 세밀한 Unsigned 타입(`u8`, `u16`, `u32`, `u64`)을 강제하는 방향(예: Rust, Go)으로 양극화되고 있다. 향후 안전한 시스템 개발을 위해서는 언어가 제공하는 강타입(Strong Type) 시스템을 통해 Unsigned와 Signed 간의 암묵적 형변환을 원천 차단하는 컴파일러 기술이 대세가 될 것이다.

### 참고 표준
- **ISO/IEC 9899 (C Standard)**: 정수 승격(Integer Promotion) 및 일반 산술 변환 규칙에서, Signed와 Unsigned가 섞일 경우 Unsigned로 우선 변환됨을 엄격히 명시하고 있다.
- **MISRA C (자동차/항공 보안 코딩 표준)**: "부호 있는 정수와 부호 없는 정수를 혼합하여 연산하지 말 것"을 강제 규칙(Mandatory Rule)으로 규정한다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[MSB (Most Significant Bit)](./80_msb.md)**: Signed에서는 부호를 결정하지만, Unsigned에서는 가장 큰 양수 가중치($2^{N-1}$)를 가지는 비트.
- **[2의 보수 (2's Complement)](./85_2s_complement.md)**: Unsigned와 대척점에 있는 체계로, MSB를 부호로 희생하여 음수를 표현하는 수학적 기법.
- **[오버플로우 (Overflow)](./95_overflow.md)**: Signed에서 부호가 바뀌는 에러 현상. Unsigned에서는 이를 '랩어라운드(Wrap-around)' 또는 '캐리(Carry)' 발생이라고 부름.
- **[진법 변환 (Base Conversion)](./78_base_conversion.md)**: 부호 없는 정수는 수학적 위치 기수법의 공식을 가장 완벽하고 단순하게 따르는 형태임.
