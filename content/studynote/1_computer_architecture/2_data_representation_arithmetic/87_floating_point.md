# 부동소수점 (Floating Point)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 부동소수점(Floating Point)은 실수를 '유효숫자(가수)'와 '소수점의 위치(지수)'로 분리하여 표현하는 컴퓨터 과학의 과학적 기수법(Scientific Notation)이다.
> 2. **가치**: 한정된 비트(예: 32비트) 내에서 원자의 크기($10^{-38}$)부터 우주의 크기($10^{38}$)까지 극단적으로 넓은 범위(Dynamic Range)의 실수를 단일 포맷으로 다룰 수 있게 해 준다.
> 3. **융합**: 넓은 범위를 얻은 대가로 절대적인 '정밀도(Precision)'를 희생했기 때문에, 딥러닝과 3D 그래픽스 등 오차를 허용하는 분야에서는 최적이지만, 금융/결제 등 오차가 누적되면 안 되는 시스템에서는 재앙을 초래할 수 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 부동소수점은 '소수점이 떠다닌다(Float)'는 뜻이다. 실수를 저장할 때 소수점의 위치를 고정(Fixed)하지 않고, 숫자의 크기에 따라 소수점의 위치를 동적으로 이동시키며 저장하는 데이터 표현 방식이다. 수식으로는 $M \times 2^E$ ($M$: 가수, $E$: 지수) 형태로 표현된다.

- **💡 비유**: 부동소수점은 **'망원경과 현미경을 하나로 합친 렌즈'**와 같다. 별을 볼 때(아주 큰 수)는 렌즈(지수)를 크게 조절하여 수십 억 광년 밖을 보고, 세균을 볼 때(아주 작은 수)는 렌즈를 작게 조절하여 나노미터 단위를 본다. 단, 렌즈를 어떻게 조절하든 한 번에 볼 수 있는 시야의 선명도(유효숫자 개수)는 항상 일정하게 제한된다.

- **등장 배경 및 발전 과정**:
  1. **고정소수점의 한계**: 고정소수점은 빠르지만 표현 범위가 너무 좁았다. 천문학적 계산을 하려면 비트 수가 무한정 늘어나야만 했다.
  2. **과학적 기수법의 도입**: 인간이 큰 수를 적을 때 '3억'을 `3,0000,0000` 대신 `3.0 × 10^8`으로 압축해 적는 원리를 컴퓨터의 2진수 체계($M \times 2^E$)에 그대로 이식했다.
  3. **IEEE 754 표준화 (1985년)**: 초기에는 컴퓨터 제조사(IBM, 크레이 등)마다 부동소수점 표현 방식이 달라 프로그램의 이식성이 최악이었다. 이를 해결하기 위해 윌리엄 카한(William Kahan) 박사가 주도하여 제정한 IEEE 754가 전 세계 컴퓨터의 통일된 실수 표준으로 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 메모리 구조 (32비트 단정밀도 기준)

부동소수점은 32비트(또는 64비트)를 3개의 구역으로 철저히 나누어 사용한다.

| 구역 | 비트 수 | 역할 | 비유 |
|:---|:---:|:---|:---|
| **부호 (Sign)** | 1 비트 | 숫자가 양수(0)인지 음수(1)인지 결정 | 숫자의 방향 (+/-) |
| **지수 (Exponent)**| 8 비트 | 소수점이 어느 위치에 있는지를 결정 ($2^E$) | 망원경/현미경의 배율 |
| **가수 (Mantissa)**| 23 비트 | 실제 숫자의 정밀한 값을 나타냄 (유효숫자) | 렌즈를 통해 보이는 실제 모양 |

---

### 부동소수점 정규화(Normalization)와 저장 원리

컴퓨터는 부동소수점을 저장하기 전에 항상 **'정규화(Normalization)'**라는 과정을 거친다. 정규화란 이진수의 소수점 위치를 이동시켜 무조건 `1.xxxx` 형태로 만드는 작업이다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         10진수 13.625가 32비트 부동소수점으로 저장되는 과정       │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  [1단계: 이진수 변환]                                           │
  │   13.625 (10진수)  =  1101.101 (2진수)                        │
  │                                                              │
  │  [2단계: 정규화 (Normalization)]                               │
  │   1101.101을 소수점이 1 바로 뒤에 오도록 이동시킴 (왼쪽으로 3칸)       │
  │   => 1.101101 × 2^3                                          │
  │                                                              │
  │  [3단계: 각 구역별 비트 배치]                                     │
  │   1. 부호 (Sign): 양수이므로 [0]                               │
  │   2. 지수 (Exponent): 3칸 이동했으므로 3.                       │
  │      (단, 음수 지수를 피하기 위해 바이어스 127을 무조건 더함)         │
  │      => 3 + 127 = 130  => [1000 0010] (이진수)                 │
  │   3. 가수 (Mantissa): 1.xxxx 에서 무조건 존재하는 앞의 1은 버리고  │
  │      소수점 아래 '101101'만 저장. 남는 자리는 0으로 채움.           │
  │      => [101 1010 0000 0000 0000 0000]                       │
  │                                                              │
  │  [최종 메모리 레이아웃 (32비트 IEEE 754)]                         │
  │   S | Exponent | Mantissa (Fraction)                         │
  │   0 | 10000010 | 10110100000000000000000                     │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 부동소수점 아키텍처의 천재성은 가수부에 있다. 이진수를 정규화하면 맨 앞자리는 무조건 '1'이 된다(`1.xxxx`). 윌리엄 카한 박사는 "어차피 무조건 1이라면, 굳이 메모리에 1을 저장해서 1비트를 낭비할 필요가 있나?"라는 아이디어를 냈다. 이를 '숨겨진 비트(Hidden Bit)' 기술이라 부르며, 덕분에 23비트의 공간만으로 실제로는 24비트의 정밀도를 갖게 되는 마법을 부렸다. 또한, 지수부에는 2의 보수를 쓰지 않고 '바이어스(Bias, 편향치)'를 더하는 방식을 써서, 지수부가 그냥 일반적인 Unsigned 정수처럼 대소 비교가 가능하도록 하드웨어 비교기를 최적화했다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 비교: 부동소수점의 치명적 함정 (오차와 정밀도)

부동소수점은 **절대적인 정확도를 보장하지 않는다.** 크기가 커질수록 정밀도(해상도)가 떨어지는 치명적인 특성이 있다.

| 값의 크기 | 지수부 (배율) | 소수점 사이의 간격 (정밀도) | 발생 가능한 오차의 크기 |
|:---|:---|:---|:---|
| **0.0 ~ 1.0 근처** | 매우 작음 | 약 $10^{-7}$ 단위로 촘촘함 | 매우 정밀함 (거의 오차 없음) |
| **10만 단위** | 중간 | 약 0.01 단위로 벌어짐 | 소수점 3자리 이하는 잘려나감 |
| **1000만 단위 이상** | 큼 | 1 이상으로 벌어짐 (소수점 표현 불가) | 1을 더해도 값이 안 변하는 버그 발생 |

### 과목 융합 관점

- **알고리즘 기하학 (Algorithm/Graphics)**: 게임 엔진에서 월드 좌표계를 32비트 부동소수점(Float)으로 관리할 때, 플레이어가 중심(0,0)에서 수천 킬로미터 밖으로 멀어지면 부동소수점의 간격이 1cm 이상으로 듬성듬성해진다. 이로 인해 캐릭터가 덜덜 떨리면서 이동하는 현상(Floating Point Jitter)이 발생하며, 이를 막기 위해 중심 좌표를 주기적으로 0으로 리셋해 주는 'Origin Shifting' 기법이 필수적이다.
- **머신러닝 가속기 (NPU/TPU)**: 최근 AI 연산에서는 표준 32비트 대신 **bfloat16 (Brain Floating Point)**이라는 구글이 제안한 새로운 16비트 포맷이 대세다. bfloat16은 지수부는 8비트(표현 범위 유지)를 그대로 두고, 가수부를 7비트로 대폭 깎아버렸다. AI는 숫자의 디테일(정밀도)은 조금 떨어져도 대략적인 크기(범위)만 맞으면 학습이 잘 된다는 성질을 하드웨어 레벨에서 해킹한 결과다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 동등 비교 연산(==)의 실패**: 자바스크립트나 파이썬에서 금액을 계산하는 로직을 작성했다.
   ```javascript
   if (0.1 + 0.2 == 0.3) {
       console.log("계산 성공");
   } else {
       console.log("계산 실패"); // 이 코드가 실행됨!
   }
   ```
   - **문제**: 이진 부동소수점은 10진수 0.1을 무한소수로 인식하여 끝자리를 잘라버린다. 따라서 실제 메모리에는 `0.30000000000000004` 같은 값이 들어간다.
   - **해결책**: 부동소수점 변수는 절대로 `==` 로 비교하면 안 된다. 두 값의 차이의 절댓값이 매우 작은 무한소(Epsilon, $\epsilon$)보다 작은지를 확인하는 함수(예: `Math.abs(a - b) < 1e-9`)를 직접 구현하여 비교해야만 실무에서 런타임 버그를 막을 수 있다.

2. **시나리오 — 대소수 덧셈 흡수(Swamping) 현상**: 대기 과학 시뮬레이션에서 지구 전체의 질량(약 $5.97 \times 10^{24}$ kg)을 담은 변수에 운석 하나의 질량(약 $1000$ kg)을 계속해서 누적해서 더하고 있다.
   - **문제**: 1억 번을 더해도 총 질량 변수는 단 1g도 늘어나지 않았다.
   - **해결책**: 큰 수와 아주 작은 수를 부동소수점 체계에서 더하면, 작은 수의 소수점(지수)을 큰 수에 맞추기 위해 오른쪽으로 시프트하는 과정에서 유효숫자 범위를 벗어나 비트가 0으로 완전히 소멸해 버리는 **'정보 소실(Swamping/Absorption)'**이 발생한다. 이를 해결하려면 작은 수끼리 먼저 다 더해서 덩치를 키운 다음에 마지막에 큰 수와 더하거나(Kahan Summation Algorithm), 64비트(`double`) 이상으로 정밀도를 올려야 한다.

### 도입 체크리스트
- **기술적**: 실수 난수나 통계 합계를 구할 때, Kahan Summation 등의 정밀도 보정 알고리즘을 사용했는가? 아니면 루프를 돌릴 때마다 오차가 누적되어 최종 결괏값이 크게 왜곡되는가?
- **운영적**: 시스템 간 API 응답(JSON) 시 실수를 문자열이 아닌 숫자 타입으로 넘길 때, 양측 언어의 부동소수점 파서가 동일한 라운딩(Rounding) 룰을 적용하는지 테스트되었는가?

### 안티패턴
- **재화/포인트 컬럼에 Float/Double 사용**: 데이터베이스나 결제 백엔드 코드에서 화폐 단위에 부동소수점을 사용하는 것은 지구상 모든 시니어 엔지니어가 혐오하는 최악의 안티패턴이다. 돈 계산은 무조건 `정수(Integer)`로 변환하여 센트(Cent) 단위로 계산하거나, `BigDecimal` 같은 고정소수점 라이브러리를 사용해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 적용 관점 | 기대효과 |
|:---|:---|:---|
| **정량** | 과학/공학 시뮬레이션에서 Dynamic Range 극대화 | 오버플로우 한계를 $10^{38}$(단정밀) $\sim 10^{308}$(배정밀)로 확장 |
| **정성** | AI 가속기에 bfloat16, FP8 포맷 도입 채택 | 정밀도는 타협하되 칩 대역폭 및 속도 **최대 4배 증가** |
| **정성** | `==` 연산자 배제 및 Epsilon($\epsilon$) 검증 원칙 강제 | 물리 엔진 충돌 체크 미스 및 분기 오류 **원천 차단** |

### 미래 전망
최근 컴퓨팅 아키텍처의 부동소수점 혁신은 '극단적인 다이어트'에 있다. 과거에는 더 정밀한 64비트(Double), 128비트(Quad)로 진화해 왔으나, AI 딥러닝 트렌드가 폭발하면서 오히려 정밀도를 깎아내고 지수부만 유지하여 데이터 대역폭을 절약하는 **FP8 (8비트 부동소수점)**, **FP4** 형태의 마이크로 포맷들이 엔비디아(Nvidia)와 ARM 등 하드웨어 제조사들의 새로운 산업 표준으로 치열하게 경합 중이다.

### 참고 표준
- **IEEE 754**: 현재 사용되는 모든 CPU, GPU, FPU가 준수해야 하는 실수 표현과 연산 규칙, 예외(NaN, Infinity) 처리를 강제하는 바이블.
- **bfloat16 (Brain Floating Point)**: 구글이 TPU를 위해 IEEE 754의 단정밀도를 변형하여 만든 AI 특화 실수 포맷.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[IEEE 754 표준](./88_ieee_754.md)**: 부동소수점이 메모리에 배치되는 규칙(Sign, Exponent, Fraction)과 NaN, 무한대 등의 예외 상태를 정의한 국제 규약.
- **[고정소수점 (Fixed Point)](./86_fixed_point.md)**: 소수점 위치를 고정하여 오차를 없앤 정수 기반의 실수 연산법으로, 부동소수점의 완벽한 대척점에 있음.
- **[오버플로우/언더플로우 (Underflow)](./96_underflow.md)**: 부동소수점에서 숫자가 너무 작아져서(예: $10^{-40}$) 0으로 붕괴되어 버리는 치명적인 하드웨어 오류.
- **[ALU와 FPU](../3_architecture_performance/117_alu.md)**: 부동소수점은 구조가 너무 복잡해 CPU 내부의 일반 가산기(ALU)가 아닌 전용 코프로세서(FPU)를 통해 연산됨.
