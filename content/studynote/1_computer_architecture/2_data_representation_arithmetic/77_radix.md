# 기수 (Radix)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기수(Radix 또는 Base)는 특정 진법(Numeral System)에서 숫자를 표현하기 위해 사용되는 고유한 숫자 기호의 개수이자, 각 자릿수의 가중치를 결정하는 밑(Base)을 의미한다.
> 2. **가치**: 인간의 10진수(Radix-10) 직관을 하드웨어의 2진수(Radix-2) 물리적 한계에 맞게 변환하고, 이를 다시 메모리 주소 표현에 최적화된 16진수(Radix-16)로 추상화하는 모든 디지털 데이터 표현의 수학적 근간이다.
> 3. **융합**: 부동소수점 연산, 암호학의 모듈러 산술, 데이터 압축(Base64 인코딩) 등 시스템 전반에 걸쳐 정보의 밀도와 연산 복잡도를 조율하는 핵심 파라미터로 작용한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 기수(Radix)는 위치 기수법(Positional Numeral System)에서 자릿수의 가치(Place Value)를 결정하는 승수의 밑(Base)이다. 예를 들어 기수가 $r$인 시스템(r진법)은 $0$부터 $r-1$까지 총 $r$개의 독립적인 기호를 사용하여 모든 수를 표현하며, 각 자리는 $r^0, r^1, r^2...$ 의 가중치를 갖는다.

- **💡 비유**: 기수는 숫자를 담는 **'바구니의 크기'**와 같다. 10진수 바구니는 사과가 10개 찰 때마다 다음으로 큰 바구니로 넘기지만, 2진수 바구니는 사과가 단 2개만 차도 다음 바구니로 넘겨야 한다. 바구니가 작을수록(기수가 작을수록) 더 많은 수의 바구니(자릿수)가 필요해진다.

- **등장 배경 및 발전 과정**:
  1. **인간의 10진수 (Radix-10)**: 인간은 양손의 손가락 10개를 기반으로 자연스럽게 10진수를 채택했다. 그러나 이는 전자 회로로 구현하기에는 10개의 서로 다른 전압 상태를 구분해야 하므로 극도로 불안정했다.
  2. **하드웨어의 2진수 (Radix-2)**: 섀넌(Claude Shannon)과 폰 노이만 시대를 거치며, 스위치의 On/Off 특성에 완벽히 부합하는 2진수가 컴퓨터의 표준 기수로 확립되었다. 이는 노이즈에 대한 내성을 극대화했다.
  3. **표현의 추상화 (Radix-8, Radix-16)**: 2진수는 자릿수가 너무 길어져 인간이 읽기 어렵다. 2진수 3자리를 하나로 묶는 8진수(Octal)를 거쳐, 현재는 2진수 4자리(1니블)를 정확히 1자리로 맵핑하는 16진수(Hexadecimal)가 메모리 및 데이터 표현의 표준으로 자리 잡았다.

**📢 섹션 요약 비유**: 기수는 마치 숫자를 분류하는 '깔때기의 크기'와 같아서, 입구가 좁을수록(기수가 작을수록) 더 많은 단계의 깔때기를 거쳐야 하지만 그만큼 분류의 기준은 단순하고 명확해집니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 기수에 따른 데이터 표현 시스템 비교

| 기수 (Radix) | 명칭 | 사용 기호 | 시스템 내 주요 사용처 |
|:---|:---|:---|:---|
| **2 (Binary)** | 2진수 | `0, 1` | ALU (Arithmetic Logic Unit, 산술 논리 장치) 논리 연산, 트랜지스터 스위칭 상태, 기계어 |
| **8 (Octal)** | 8진수 | `0 ~ 7` | UNIX/Linux 파일 권한 (예: `chmod 755`), 과거 12/36비트 시스템 |
| **10 (Decimal)** | 10진수 | `0 ~ 9` | 사용자 인터페이스, BCD (Binary Coded Decimal, 2진 부호화 10진수) 연산, 금융 계산 |
| **16 (Hexadecimal)** | 16진수 | `0 ~ 9, A ~ F` | 메모리 주소 표기, MAC (Media Access Control) / IPv6 (Internet Protocol version 6) 주소, 색상 코드, 디버거 덤프 |
| **64 (Base64)** | 64진수 | `A-Z, a-z, 0-9, +, /` | MIME (Multipurpose Internet Mail Extensions), API 토큰 전송 시 바이너리 데이터를 텍스트로 인코딩 |

### 위치 기수법의 수학적 원리 (Positional Notation)

임의의 기수 $r$에 대한 숫자 $N$은 각 자리의 기호 $d_i$와 기수의 거듭제곱 $r^i$의 곱의 합으로 표현된다. 이 원리는 소수점 이하의 수에도 동일하게 적용되어 마이크로프로세서 내부의 부동소수점 연산기를 설계하는 기초가 된다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              위치 기수법(Positional Notation)의 수학적 원리      │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  숫자 N (r진법) = d_n d_{n-1} ... d_1 d_0 . d_{-1} d_{-2}        │
  │                                                              │
  │  N = d_n×r^n + d_{n-1}×r^{n-1} + ... + d_0×r^0                  │
  │      + d_{-1}×r^{-1} + d_{-2}×r^{-2} ...                      │
  │                                                              │
  │  [예시: 2진수 1101.101 (Radix = 2) 의 10진수 변환]                │
  │                                                              │
  │    자리:   3    2    1    0  .  -1   -2   -3                   │
  │    기호:   1    1    0    1  .   1    0    1                   │
  │  가중치:  2^3  2^2  2^1  2^0 . 2^-1 2^-2 2^-3                  │
  │          (8)  (4)  (2)  (1) . (0.5) (0.25) (0.125)           │
  │                                                              │
  │  계산: (1×8) + (1×4) + (0×2) + (1×1) + (1×0.5) + (0) + (1×0.125)│
  │      = 8 + 4 + 0 + 1 + 0.5 + 0 + 0.125 = 13.625 (10진수)       │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 컴퓨터가 메모리에 저장하는 모든 데이터는 본질적으로 $r=2$인 위치 기수법 수식의 계수배열($d_n, d_{n-1}...$)에 불과하다. 이 다이어그램은 2진수의 각 자리가 10진수 가중치로 어떻게 매핑되는지 명확히 보여준다. 특히 소수점 이하의 값($2^{-1}, 2^{-2}...$)은 0.5, 0.25, 0.125로 떨어지기 때문에, 10진수 소수 0.1과 같은 값은 2진수로 정확히 나누어 떨어지지 않아 순환소수가 된다. 이것이 실수 연산에서 정밀도 오류(Precision Error)가 발생하는 근본적인 수학적 이유다.

**📢 섹션 요약 비유**: 위치 기수법은 마치 자릿수마다 정해진 '무게추'가 있는 저울과 같아서, 어떤 기수를 쓰느냐에 따라 같은 숫자 기호라도 시스템이 느끼는 무게(값)가 천차만별로 달라집니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 비교: 10진수(Radix-10) vs 16진수(Radix-16)

프로그래머가 데이터를 해석할 때 10진수와 16진수 중 어느 것을 선택하느냐는 디버깅 효율성에 결정적인 영향을 미친다.

| 비교 항목 | 10진수 (Radix 10) | 16진수 (Radix 16) |
|:---|:---|:---|
| **물리적 비트 정렬** | 일치하지 않음 | **정확히 4비트(1니블)와 1자리 기호가 1:1 매칭됨** |
| **비트 패턴 유추** | 불가능 (예: `255` → `11111111` 직관적 변환 어려움) | 극히 용이 (예: `0xFF` → `1111 1111` 즉시 변환) |
| **자릿수 (문자열 길이)** | 가변적, 중간 정도의 길이 | 매우 짧고 고정적 크기 할당 용이 (예: 1바이트=2글자) |
| **주 사용 목적** | 금융 도메인, 최종 사용자 화면 표시 | 코어 덤프 분석, 메모리 주소 할당, 비트 마스킹 |

### 과목 융합 관점

- **네트워크 (Network)**: Base64 (Radix-64) 인코딩은 8비트 바이트 데이터 3개(24비트)를 6비트씩 4개로 쪼개어 출력 가능한 64개의 ASCII (American Standard Code for Information Interchange) 문자로 매핑하는 기법이다. 이메일 첨부파일 전송 시 이진 데이터를 안전한 텍스트로 변환하기 위해 기수를 변환하는 대표적인 사례다.
- **데이터베이스 (DB, Database)**: 일반적인 IEEE (Institute of Electrical and Electronics Engineers) 754 부동소수점(Radix-2)은 10진수 소수를 정확히 표현하지 못해 금융 시스템에서 1원 단위의 오차를 낸다. 이를 해결하기 위해 DB에서는 `DECIMAL`이나 `NUMERIC` 타입을 사용하여 하드웨어의 2진수 연산 대신 소프트웨어적으로 구현된 10진수(Radix-10) 기반의 팩드 BCD (Binary Coded Decimal) 연산을 수행한다.

**📢 섹션 요약 비유**: 10진수가 사람이 읽기 편한 '구어체'라면, 16진수는 기계와 사람 사이의 통역을 맡은 '전문 용어'와 같아서 데이터의 속을 들여다보는 데 가장 효율적인 도구가 됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 권한 관리(CHMOD)에서의 8진수 사용**: 리눅스 파일 시스템 권한을 설정할 때 `chmod 755 file.sh` 명령어를 사용한다. 여기서 755는 10진수가 아니라 8진수(Radix-8)다.
   - **원리**: 8진수 한 자리는 정확히 3비트와 매칭된다. 파일 권한은 `Read(4, 100)`, `Write(2, 010)`, `Execute(1, 001)`의 3비트 조합으로 구성되므로, 사용자/그룹/기타 권한을 각각 3비트씩 총 9비트로 표현하기에 8진수 3자리(`755`)가 완벽하게 들어맞는다. 이를 10진수로 쓰면 비트 패턴이 깨져 권한을 직관적으로 조합할 수 없다.

2. **시나리오 — 비트 마스킹 및 플래그 제어**: 시스템 프로그래밍에서 특정 하드웨어 레지스터의 특정 비트만 On/Off 해야 할 때, 10진수 숫자를 쓰는 것은 금기사항이다.
   - **해결책**: 반드시 16진수(Radix-16)를 사용해야 한다. 예를 들어 32비트 레지스터의 최상위 바이트를 마스킹하려면 `value & 4278190080` (10진수)라고 쓰는 대신, `value & 0xFF000000` (16진수)라고 작성해야 코드를 읽는 동료나 컴파일러가 비트의 정확한 위치(상위 8비트)를 직관적으로 파악할 수 있다.

### 도입 체크리스트
- **기술적**: 하드웨어 제어 및 네트워크 패킷 생성 코드에 10진수 매직 넘버(Magic Number)가 포함되어 있는가? (반드시 16진수 상수나 비트 시프트 연산으로 리팩토링해야 함).
- **보안적**: 난수 생성이나 암호화 키 생성 시, 로그 출력에 해당 값이 10진수 문자열로 저장되면서 버퍼 크기 부족(Truncation)으로 인한 데이터 유실이 발생하지 않았는가? (16진수 고정 길이 표기를 강제해야 함).

### 안티패턴
- **문자열-숫자 파싱 시 기수 누락**: 자바스크립트 등에서 `parseInt("08")`을 실행할 때 기수(Radix)를 명시하지 않으면, 구형 브라우저에서는 문자열이 "0"으로 시작한다는 이유로 이를 8진수로 해석하여 `8`이 아닌 오류나 예기치 않은 값을 반환하는 경우가 있었다. 항상 `parseInt("08", 10)`처럼 변환하고자 하는 대상의 기수를 명시해야 하는 것이 베스트 프랙티스다.

**📢 섹션 요약 비유**: 실무에서 기수를 잘못 선택하는 것은 마치 인치(Inch) 단위 설계도에 센티미터(CM) 부품을 끼워 넣는 것과 같아서, 눈에 보이지 않는 미세한 균열(버그)을 야기하게 됩니다.

|:---|:---|:---|
| **정량** | 메모리 덤프 출력을 2진수 대신 16진수로 압축 표기 | 화면 출력 문자 수 **75% 감소** (데이터 가독성 극대화) |

| **정성** | 금융권 `DECIMAL` 타입(Radix-10) 강제화 정책 적용 | 부동소수점 오차에 의한 정산 불일치 이슈 **원천 차단** |

### 미래 전망
디지털 시스템의 하드웨어는 영원히 기수 2(Radix-2)에 머물 것이지만, 데이터 표현과 압축의 층위에서는 기수를 동적으로 높여 정보의 밀도를 극대화하는 방식이 계속 발전하고 있다. 특히 블록체인에서 주소 길이를 줄이기 위해 영문 대소문자와 숫자를 조합하고 헷갈리는 문자를 뺀 Base58, Base56 같은 커스텀 기수 체계가 등장하는 등, 도메인 특화 기수법(Domain-Specific Radix)의 활용이 증가하고 있다.

### 참고 표준
- **IEEE (Institute of Electrical and Electronics Engineers) 754-2008**: 이진 부동소수점(Radix-2)뿐만 아니라, 십진 부동소수점(Radix-10) 연산 형식까지 표준에 포함시켜 금융 계산의 호환성을 보장한다.
- **RFC (Request for Comments) 4648 (Base64/Base32/Base16)**: 바이너리 데이터를 다양한 기수의 텍스트 표현으로 변환하기 위한 인터넷 표준.

**📢 섹션 요약 비유**: 기수는 디지털 문명이 숫자를 이해하고 저장하는 가장 깊은 곳의 '언어 규칙'이며, 이 규칙을 잘 조율하는 것이 시스템의 효율성과 정밀도를 결정짓는 열쇠입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **진법 변환 (Base Conversion)**: 기수 $r_1$로 표현된 숫자를 기수 $r_2$로 변환하는 수학적 알고리즘(나눗셈-나머지 기법 등).
- **2의 보수 (2's Complement)**: Radix-2 시스템에서 음수를 표현하기 위해 고안된 가장 완벽한 수학적 체계.
- **부동소수점 (Floating Point)**: 2진수 기수를 활용하여 매우 크거나 작은 실수를 과학적 표기법으로 메모리에 저장하는 방식.
- **BCD 코드 (Binary Coded Decimal)**: 인간의 기수(10진수)와 기계의 기수(2진수)의 격차를 해소하기 위해, 10진수 1자리를 2진수 4자리로 직역하여 저장하는 방식.

## 👶 어린이를 위한 3줄 비유 설명
1. 기수(Radix)는 숫자를 셀 때 사용하는 **'바구니의 크기'**예요.
2. 사람들은 손가락이 10개라서 사과 10개가 찰 때마다 다음 바구니로 넘기는 '10진수'를 쓰지만, 전기로 움직이는 컴퓨터는 켜짐/꺼짐 딱 2가지 상태만 있어서 사과 2개만 차도 다음 바구니로 넘기는 '2진수'를 써요.
3. 2진수는 너무 길어져서 사람이 읽기 힘드니까, 컴퓨터 일꾼들은 4개의 작은 바구니를 하나로 묶어 '16진수'라는 특별한 표지판을 달아서 쉽게 읽고 쓴답니다!
