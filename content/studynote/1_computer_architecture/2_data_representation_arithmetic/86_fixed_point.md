# 고정소수점 (Fixed Point)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 고정소수점(Fixed Point)은 한정된 비트 내에서 소수점의 위치를 고정(Fixed)해두고, 그 앞은 정수부, 그 뒤는 소수부로 나누어 실수를 표현하는 직관적인 데이터 규격이다.
> 2. **가치**: 값의 범위는 좁지만, 부동소수점 연산기(FPU) 없이도 정수 가산기(ALU)만으로 실수 연산을 초고속으로 처리할 수 있어 시스템의 복잡도와 전력 소모를 극적으로 낮춘다.
> 3. **융합**: 소수점 연산에서 1원의 오차도 허용되지 않는 금융(DB의 `DECIMAL` 타입) 도메인과, 전력이 극도로 제한된 IoT/임베디드 기기의 센서 데이터 처리에 필수불가결한 기술이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 고정소수점은 실수를 컴퓨터 메모리에 저장할 때, 전체 N비트 중 정수(Integer) 부분과 소수(Fraction) 부분의 비트 수를 미리 고정해 놓고 사용하는 방식이다. 예를 들어 16비트 시스템에서 '앞의 8비트는 정수, 뒤의 8비트는 소수'로 약속(Q 포맷)하여 데이터를 해석한다.

- **💡 비유**: 고정소수점은 **'모눈종이(Graph Paper)'**와 같다. 모눈종이의 칸 하나하나가 나타내는 크기(예: 0.01cm)가 이미 정해져 있다. 그래서 그릴 수 있는 도형의 최대 크기는 종이의 크기만큼으로 제한되지만, 정해진 칸 안에서는 자(Ruler) 없이도 칸 수만 세면 아주 정확하고 빠르게 길이를 계산할 수 있다.

- **등장 배경 및 발전 과정**:
  1. **초기 시스템의 한계**: 초기 마이크로프로세서나 마이크로컨트롤러(MCU)에는 복잡한 실수를 처리하는 하드웨어(FPU)가 없었다. 하지만 센서 값이나 통신을 위해 소수점 연산은 필요했다.
  2. **정수 연산의 재활용 (스케일링)**: 엔지니어들은 소수점에 $10^N$이나 $2^N$을 곱하여 정수로 만든 뒤 ALU에서 연산하고, 나중에 다시 나누는 방식을 고안했다. 이것이 고정소수점의 시초다.
  3. **금융권의 정밀도 요구**: 부동소수점(IEEE 754)이 표준이 된 이후에도, 이진 부동소수점이 10진수 소수를 정확히 표현하지 못해 생기는 1원의 오차가 금융권에서 큰 문제로 대두되었다. 이에 10진수 기반의 고정소수점 체계가 DB와 금융 소프트웨어의 핵심 표준으로 살아남게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 메모리 구조 및 Q 포맷 (Q Format)

고정소수점은 시스템에 따라 정수부와 소수부의 비율을 마음대로 정할 수 있는데, 이를 명시하기 위해 텍사스 인스트루먼트(TI) 등에서 고안한 **Q 포맷** 표기법을 주로 사용한다. `Qm.n`은 정수부가 m비트, 소수부가 n비트임을 뜻한다.

| 포맷 (16비트) | 부호 비트 | 정수부 (m) | 소수부 (n) | 표현 범위 (약) | 정밀도 (해상도) |
|:---|:---:|:---:|:---:|:---|:---|
| **Q15 (Q0.15)** | 1비트 | 0 비트 | 15 비트 | $-1.0 \sim +0.9999$ | $2^{-15}$ (최상 정밀도) |
| **Q7.8** | 1비트 | 7 비트 | 8 비트 | $-128 \sim +127.99$ | $2^{-8} = 0.00390625$ |
| **Q15.0** | 1비트 | 15 비트 | 0 비트 | $-32768 \sim +32767$ | $1$ (일반 정수와 동일) |

> **트레이드오프**: 전체 비트 수가 고정되어 있으므로, 정수부를 늘리면 표현할 수 있는 '최댓값'이 커지지만 소수점 아래 '정밀도'가 떨어지고, 소수부를 늘리면 정밀도는 높아지나 표현 범위가 극도로 좁아진다.

---

### 고정소수점 덧셈 원리와 10진/2진 매핑

컴퓨터 내부의 ALU는 고정소수점 데이터가 실수라는 사실을 모른다. 그저 일반적인 2의 보수 정수로 취급하여 덧셈을 수행할 뿐, 소수점의 위치는 오직 프로그래머의 머릿속(또는 컴파일러)에만 존재한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         Q3.4 포맷(총 8비트)의 고정소수점 연산 원리              │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  포맷: [부호 1비트] [정수 3비트] . [소수 4비트]                 │
  │  해상도(LSB 가중치): 2^-4 = 0.0625                             │
  │                                                              │
  │  [계산: 2.5 + 1.25]                                          │
  │                                                              │
  │  1. 2.5의 이진 표현 (2 + 0.5):                                │
  │     정수부: 0 010 (2)                                         │
  │     소수부: 1000 (0.5)                                        │
  │     비트: 0010.1000                                           │
  │                                                              │
  │  2. 1.25의 이진 표현 (1 + 0.25):                              │
  │     정수부: 0 001 (1)                                         │
  │     소수부: 0100 (0.25)                                       │
  │     비트: 0001.0100                                           │
  │                                                              │
  │  3. ALU 덧셈 (소수점은 무시하고 그냥 정수처럼 더함):                 │
  │       0010 1000                                              │
  │     + 0001 0100                                              │
  │     ────────────                                             │
  │       0011 1100  (정수로는 60)                                │
  │                                                              │
  │  4. 결과 해석:                                                │
  │     60 × 0.0625(해상도) = 3.75                                │
  │     또는 비트로 읽기: 0011(3) . 1100(0.5 + 0.25) = 3.75       │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 고정소수점의 가장 큰 장점은 FPU(부동소수점 유닛)가 필요 없다는 것이다. CPU의 ALU는 소수점의 존재를 전혀 모른 채 두 개의 8비트 정수를 더한다(결과 `60`). 이 결과를 화면에 출력하거나 의미를 부여할 때, 소프트웨어가 "이 숫자는 사실 끝에서 4번째 자리에 소수점이 있어"라고 해석(Scaling)해 주는 것이다. 하드웨어 관점에서는 100% 정수 연산이므로 속도가 부동소수점 대비 수십 배 빠르고 전력 소모가 극도로 적다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 비교: 고정소수점 vs 부동소수점

| 비교 항목 | 고정소수점 (Fixed Point) | 부동소수점 (Floating Point) |
|:---|:---|:---|
| **소수점의 위치** | 고정됨 (소프트웨어적 약속) | 데이터마다 유동적으로 변함 (지수부 활용) |
| **표현 범위 (Dynamic Range)**| 매우 좁음 (설정된 정수부 크기에 종속) | 천문학적으로 넓음 (예: $10^{-38} \sim 10^{38}$) |
| **정밀도 (Precision)** | 값의 크기와 무관하게 **항상 일정함** | 값이 커질수록 소수점 정밀도가 급격히 떨어짐 |
| **하드웨어 요구사항** | ALU (일반 정수 가산기) 만으로 충분 | FPU (전용 복잡한 회로) 필수 |
| **주요 사용처** | RDBMS `DECIMAL`, 임베디드 오디오 DSP | 3D 그래픽스, 과학 연산, 머신러닝 |

### 과목 융합 관점

- **데이터베이스 (DB)**: 오라클이나 MySQL에서 돈을 다룰 때는 `FLOAT`이나 `DOUBLE`을 쓰지 않고 `DECIMAL(10, 2)` (총 10자리 중 소수점 이하 2자리 고정) 타입을 쓴다. 부동소수점을 쓰면 `0.1 + 0.2`가 `0.30000000000000004`가 되어 정산이 틀어지기 때문이다. DB의 고정소수점은 내부적으로 숫자를 정수(예: 10.50달러 → 1050센트)로 스케일링하여 저장하므로 오차가 0%다.
- **머신러닝 가속기 (NPU)**: 과거 딥러닝 추론은 32비트 부동소수점(FP32)으로 수행되었으나, 최근 모바일 기기(Edge AI)에서는 모델의 가중치를 8비트 정수형 고정소수점(INT8)으로 변환하는 **양자화(Quantization)**가 대세다. 이를 통해 메모리 대역폭을 1/4로 줄이고 추론 속도를 비약적으로 끌어올린다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 금융 시스템의 이자 계산 오차**: 은행의 일일 이자 계산 배치 프로그램에서 잔액 변수에 `double` (64비트 부동소수점)을 사용했다.
   - **문제**: 수억 건의 계좌에 0.1%의 이자를 더하는 과정에서, 이진 부동소수점이 0.1을 정확히 표현하지 못해 계좌당 0.000001원씩 오차가 발생했고, 전체 정산 금액이 수만 원 어긋나 회계 감사가 실패했다.
   - **해결책**: 화폐는 절대 부동소수점으로 다루면 안 된다. Java의 `BigDecimal` 클래스나, C#의 `decimal` (내부적으로 128비트 고정소수점/스케일링 팩터 사용)을 사용하여 소수점의 정밀도를 고정하고 소프트웨어적(또는 BCD)으로 10진수 연산을 수행해야 한다. 속도는 느려지지만 무결성이 확보된다.

2. **시나리오 — 저전력 IoT 기기에서의 센서 필터링**: 배터리로 1년을 버텨야 하는 Cortex-M0 기반 IoT 온도 센서 노드에서 칼만 필터(Kalman Filter)를 돌려야 한다.
   - **문제**: Cortex-M0에는 하드웨어 FPU가 없다. C 코드로 `float` 변수를 사용하여 필터를 구현했더니, 컴파일러가 수천 바이트짜리 소프트웨어 부동소수점 에뮬레이션 라이브러리를 삽입하여 배터리가 1달 만에 방전되었다.
   - **해결책**: 필터 로직의 모든 `float` 변수를 Q15 포맷의 `int16_t` 고정소수점으로 리팩토링한다. 온도 데이터(예: 25.5도)에 256($2^8$)을 곱해 정수(6528)로 만든 뒤 필터를 통과시키고, 최종 출력 시 다시 256으로 나누는(우측 시프트 `>> 8`) 방식을 쓰면 연산 전력이 90% 이상 절감된다.

### 도입 체크리스트
- **기술적**: 고정소수점 곱셈(Multiplication) 시 발생하는 데이터 오버플로우를 고려했는가? (예: Q15 포맷 16비트 두 개를 곱하면 32비트 결과가 나오므로, 반드시 중간 변수를 32비트로 받아 시프트 연산으로 포맷을 맞춰주어야 한다).
- **운영적**: RDBMS 설계 시, 돈과 관련된 필드가 `DECIMAL/NUMERIC`으로 올바르게 설정되어 있으며, 백엔드 ORM 객체와 맵핑될 때 `Double`로 강제 변환되어 정밀도가 훼손되는 구간이 없는가?

### 안티패턴
- **부동소수점 루프 누적**: `float a = 0.0; for(int i=0; i<100; i++) a += 0.1;` 형태의 코드. 결과는 `10.0`이 아니라 `10.000002` 같은 쓰레기 값이 나온다. 반복 누적 연산이 필요하다면 반드시 고정소수점 기법을 차용하여 `int a = 0; for(...) a += 1;` 로 루프를 돌린 후 마지막에 `/ 10.0` 을 해주는 것이 시스템 아키텍처의 기본 상식이다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 적용 사례 | 기대효과 |
|:---|:---|:---|
| **정량** | FPU 없는 임베디드 환경에서 Float을 Q 포맷으로 변환 | 연산 사이클 **최대 50~100배 단축** |
| **정량** | AI 모델 추론 시 FP32 가중치를 INT8 고정소수점으로 양자화 | 메모리 사용량 **75% 감소** 및 캐시 히트율 급증 |
| **정성** | 결제/회계 시스템 아키텍처에 `DECIMAL` 타입 표준화 | 부동소수점 오차에 의한 정산 장애 **제로화** |

### 미래 전망
데스크톱과 서버 시장에서는 FPU의 성능이 압도적으로 좋아져 고정소수점을 직접 다룰 일이 거의 없어졌다. 하지만 세상의 패러다임이 '엣지 컴퓨팅(Edge Computing)'과 '온디바이스 AI(On-device AI)'로 넘어가면서, 초저전력 칩셋에서 신경망 연산을 수행하기 위해 다시 고정소수점(INT8, INT4 연산)이 르네상스를 맞이하고 있다. 즉, 하드웨어 자원이 극도로 제한된 프론티어 영역에서는 언제나 고정소수점이 핵심 무기가 된다.

### 참고 표준
- **ISO/IEC TR 18037 (C 언어 확장)**: 임베디드 시스템 프로그래밍을 위해 C 언어에 공식적으로 `_Accum`과 `_Fract` 라는 고정소수점 내장 데이터 타입을 추가한 기술 보고서.
- **SQL:2016 표준**: `DECIMAL(p, s)` 및 `NUMERIC(p, s)`를 정확한(Exact) 수치 데이터 타입으로 규정하여 금융 데이터 무결성을 보장.

## 👶 어린이를 위한 3줄 비유 설명
1. 고정소수점은 길이를 잴 때 눈금이 절대 변하지 않는 **'딱딱한 쇠자(Ruler)'**와 같아요.
2. 눈금이 고정되어 있어서 자보다 큰 물건은 잴 수 없지만, 눈금 하나하나가 아주 정확해서 절대 틀리는 법이 없어요.
3. 그래서 컴퓨터로 은행의 돈(1원 단위)을 계산하거나, 배터리를 아주 조금만 써야 하는 작은 시계 같은 기계에서는 이 방식이 최고랍니다!
