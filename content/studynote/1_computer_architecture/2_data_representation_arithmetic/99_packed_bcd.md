# 팩드 BCD (Packed BCD)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 팩드 BCD(Packed BCD)는 1바이트(8비트)에 두 개의 십진수 자릿수를 저장하는 BCD 변형으로, 상위 4비트에 십의 자리, 하위 4비트에 일의 자리를 저장한다.
> 2. **가치**: 언팩드 BCD(1바이트에 1자릿수) 대비 50% 메모리 절감을 달성하며, IBM 메인프레임과 COBOL 시스템에서 화폐·금융 데이터의 표준 포맷이다.
> 3. **융합**: 팩드 BCD는 z/Architecture, x86(레거시), IBM i에서 하드웨어 지원을 받으며, 금융권 레거시 시스템 마이그레이션 시 이해가 필수적이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 팩드 BCD(Packed BCD, 또는 Packed Decimal)는 1바이트(8비트)에 두 개의 십진수 자릿수를 "압축(Pack)"하여 저장하는 BCD 인코딩 방식이다. 상위 4비트(상위 니블)에 십의 자리, 하위 4비트(하위 니블)에 일의 자리를 저장한다. 예를 들어 십진수 59는 팩드 BCD로 `0101 1001`(= 0x59)가 된다. 부호를 표현하기 위해 마지막 바이트의 하위 니블을 부호 니블(Sign Nibble)로 사용하기도 한다.

- **💡 비유**: 팩드 BCD는 **'2인용 오토바이'**와 같다. 언팩드 BCD는 각 승객(자릿수)이 혼자 오토바이를 타지만, 팩드 BCD는 두 승객이 한 오토바이에 함께 탄다. 연료(메모리)는 절반으로 줄어든다.

- **등장 배경 및 발전 과정**:
  1. **메인프레임 시대의 효율화**: 1960년대 IBM System/360은 상업용 데이터 처리를 위해 Packed Decimal을 도입했다. 화폐 금액 등 십진수 데이터가 많은 COBOL 애플리케이션에서 메모리를 절약했다.
  2. **x86의 지원 (레거시)**: Intel 8086은 DAA(Decimal Adjust for Addition), DAS(Decimal Adjust for Subtraction) 등의 명령어로 팩드 BCD 연산을 지원했다. 64비트 모드에서는 제거되었다.
  3. **현대의 레거시**: 금융권, 보험사, 정부 기관의 메인프레임(z/OS, IBM i)에서 여전히 팩드 BCD가 활발히 사용된다. COBOL 프로그램의 데이터 포맷 표준이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 팩드 BCD 포맷 구조

팩드 BCD는 1바이트에 두 자릿수를 저장한다. 부호 있는 경우 마지막 바이트의 하위 니블을 부호로 사용한다.

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │         팩드 BCD 포맷 구조                                                    │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │   부호 없는 팩드 BCD (Unsigned Packed BCD):                                  │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  Byte n:   [Digit n+1] [Digit n]   → 두 자릿수를 1바이트에 저장        ││
  │   │            └──┬──┘    └──┬──┘                                           ││
  │   │            상위 4비트   하위 4비트                                        ││
  │   │                                                                         ││
  │   │  예: 십진수 59                                                         ││
  │   │  ┌─────────────────────────────────────────────────────────────────────┐││
  │   │  │  0 1 0 1   │   1 0 0 1   = 0x59                                    │││
  │   │  │    5       │      9       → "59"                                   │││
  │   │  └─────────────────────────────────────────────────────────────────────┘││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   부호 있는 팩드 BCD (Signed Packed BCD / zoned decimal):                    │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  Byte 0:  [Digit 2] [Digit 1]                                          ││
  │   │  Byte 1:  [Digit 4] [Digit 3]                                          ││
  │   │  ...                                                                     ││
  │   │  Byte n:  [Sign  ] [Digit m]  ← 마지막 바이트: 부호 + 마지막 자릿수      ││
  │   │                                                                         ││
  │   │  부호 니블(Sign Nibble):                                                ││
  │   │  ┌─────────────────────────────────────────────────────────────────────┐││
  │   │  │  0xC (1100) = 양수 (+)     0xD (1101) = 음수 (-)                    │││
  │   │  │  0xF (1111) = 양수 (+)     0xB (1011) = 음수 (-)  ← 시스템마다 다름 │││
  │   │  └─────────────────────────────────────────────────────────────────────┘││
  │   │                                                                         ││
  │   │  예: 십진수 +1234 (4자릿수)                                             ││
  │   │  ┌─────────────────────────────────────────────────────────────────────┐││
  │   │  │  Byte 0: 0001 0010 = 0x12  (1, 2)                                   │││
  │   │  │  Byte 1: 0011 1100 = 0x3C  (3, +)                                   │││
  │   │  │                      ↑                                              │││
  │   │  │                   0xC = 양수                                        │││
  │   │  │  → 2바이트로 4자릿수 + 부호 표현                                    │││
  │   │  └─────────────────────────────────────────────────────────────────────┘││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   ⚠ 홀수 자릿수인 경우: 상위 니블이 0으로 채워짐                              │
  │      예: +123 (3자릿수) → 0x01 0x23 → Byte 0: 0001 0010, Byte 1: 0011 1100 │
  │      → 실제로는 0x12 0x3C (2바이트)로 표현                                  │
  │                                                                              │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 팩드 BCD는 1바이트에 두 자릿수를 저장한다. 부호 없는 경우는 단순히 두 자릿수를 상/하위 니블에 저장한다. 부호 있는 경우는 마지막 바이트의 하위 니블을 부호로 사용한다(0xC = 양수, 0xD = 음수). 홀수 자릿수인 경우 상위 니블이 0으로 채워진다. 예를 들어 +123은 2바이트로 `0x12 0x3C`가 된다(1, 2 | 3, +).

### 언팩드 BCD vs 팩드 BCD 비교

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │         언팩드 BCD vs 팩드 BCD 메모리 비교                                    │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │   십진수 123456789를 표현:                                                   │
  │                                                                              │
  │   언팩드 BCD (Unpacked BCD):                                                 │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  1    2    3    4    5    6    7    8    9                            ││
  │   │  ↓    ↓    ↓    ↓    ↓    ↓    ↓    ↓    ↓                            ││
  │   │ 0x01 0x02 0x03 0x04 0x05 0x06 0x07 0x08 0x09                          ││
  │   │                                                                         ││
  │   │  총 9바이트 사용                                                        ││
  │   │  각 자릿수가 1바이트(상위 4비트 = 0, 하위 4비트 = 자릿수)                 ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   팩드 BCD (Packed BCD):                                                     │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  12   34   56   78   9C                                                ││
  │   │  ↓    ↓    ↓    ↓    ↓                                                 ││
  │   │ 0x12 0x34 0x56 0x78 0x9C                                              ││
  │   │                              ↑                                           ││
  │   │                         0xC = 양수 부호                                 ││
  │   │                                                                         ││
  │   │  총 5바이트 사용                                                         ││
  │   │  두 자릿수가 1바이트, 마지막 바이트는 자릿수 + 부호                      ││
  │   │                                                                         ││
  │   │  ⚠ 메모리 절감: 9바이트 → 5바이트 = 44% 절감                              ││
  │   │     일반적으로 홀수 자릿수인 경우 (n+1)/2바이트, 짝수 자릿수인 경우 n/2+1바이트 ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   일반 이진수:                                                               │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  123456789 = 0x75BCD15                                                 ││
  │   │  총 4바이트 사용 (32비트 정수)                                           ││
  │   │  ⚠ 가장 효율적이지만, 십진수 변환 필요                                   ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 9자릿수 십진수 123456789를 기준으로, 언팩드 BCD는 9바이트, 팩드 BCD는 5바이트(마지막 바이트는 부호 포함), 일반 이진수는 4바이트를 사용한다. 팩드 BCD는 언팩드 BCD 대비 약 44% 메모리를 절감한다. 일반 이진수가 가장 효율적이지만, 십진수 변환(이진수 ↔ 십진수)이 필요하다. 금융 시스템에서는 이 변환 오버헤드를 피하기 위해 BCD를 사용한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 팩드 BCD vs 언팩드 BCD vs 일반 이진수

| 비교 항목 | 팩드 BCD | 언팩드 BCD | 일반 이진수 |
|:---|:---|:---|:---|
| **메모리 (n자릿수)** | ⌈(n+1)/2⌉ 바이트 | n 바이트 | ⌈log₁₀(10ⁿ)/8⌉ 바이트 |
| **부호 표현** | 마지막 니블 사용 | 별도 바이트 사용 | 최상위 비트 사용 |
| **산술 연산** | 보정 필요, 복잡 | 보정 필요, 단순 | 직접 연산, 간단 |
| **디스플레이** | 변환 필요 | 거의 직접 가능 | 변환 필요 |
| **주 용도** | 메인프레임, COBOL | 7세그먼트 LED | 일반 연산 |
| **하드웨어 지원** | IBM z, x86(레거시) | 제한적 | 모든 CPU |

### 과목 융합 관점
- **컴퓨터구조(컴구)**: IBM z/Architecture는 팩드 BCD 연산을 위한 전용 명령어(AP, SP, MP, DP)를 제공한다. x86의 DAA/DAS는 레거시로 남아있다
- **운영체제(OS)**: z/OS, IBM i 운영체제는 COBOL과 PL/I의 팩드 BCD 데이터를 네이티브로 처리한다
- **데이터베이스(DB)**: IBM DB2 for z/OS는 DECIMAL(p, s) 타입을 팩드 BCD로 저장한다

## Ⅳ. 실무 적용 및 기술사적 판단
### 실무 시나리오
1. **시나리오 — COBOL 레거시 데이터 마이그레이션**: 은행 COBOL 시스템의 Packed Decimal 데이터를 Java로 마이그레이션해야 한다. 기술사는 ① 바이너리 포맷 이해(부호 니블 0xC/0xD), ② `BigDecimal` 변환, ③ 정확성 검증(동등성 테스트)을 수행해야 한다
2. **시나리오 — POS 단말기 펌웨어**: 소매점 POS 단말기가 금액을 팩드 BCD로 저장하여 영수증 출력 시 변환 없이 직접 사용한다. 기술사는 팩드 BCD를 사용하여 메모리를 절약하고, 7세그먼트 디스플레이와의 호환성을 확보한다
3. **시나리오 — 메인프레임 하드웨어 가속**: z/OS에서 대량의 팩드 BCD 연산을 수행해야 한다. 기술사는 IBM z16의 Packed Decimal 전용 명령어(AP, SP, MP, DP)를 활용하여 소프트웨어 에뮬레이션 대비 10~100배 성능 향상을 달성한다
### 도입 체크리스트
- **기술적**: 타겟 시스템이 팩드 BCD를 하드웨어로 지원하는가?(IBM z, x86 레거시)
- **운영·보안적**: 레거시 데이터와의 호환성이 요구되는가? 그렇다면 팩드 BCD가 필수적일 수 있다
### 안티패턴
- **팩드 BCD로 일반 연산**: 주소 계산, 루프 카운터 등에 팩드 BCD를 사용하면 성능이 심각히 저하된다
- **부호 니블 미검증**: 0xC/0xD 외의 값이 부호 니블에 있으면, 레거시 시스템에서 오류가 발생할 수 있다

## Ⅴ. 기대효과 및 결론
### 정량/정성 기대효과
| 구분 | 언팩드 BCD | 팩드 BCD | 효과 |
|:---|:---|:---|:---|
| **정량** | n자릿수 = n바이트 | n자릿수 ≈ n/2바이트 | 메모리 50% 절감 |
| **정량** | 변환 단순 | 변환 복잡 | 연산 효율 향상 |
| **정성** | 디스플레이 직접 | 메인프레임 표준 | 시스템 호환성 |
### 미래 전망
- **메인프레임 지속**: 금융권, 보험사, 정부 기관의 메인프레임은 향후 10~20년간 팩드 BCD를 계속 사용할 것이다
- **클라우드 마이그레이션**: 메인프레임에서 클라우드로의 마이그레이션 시, 팩드 BCD 데이터를 `BigDecimal` 또는 `DECIMAL` 타입으로 변환해야 한다
### 참고 표준
- **IBM z/Architecture Principles of Operation**: Packed Decimal 명령어 정의
- **COBOL ISO/IEC 1989:2023**: PACKED-DECIMAL 데이터 타입

## 👶 어린이를 위한 3줄 비유 설명
1. 팩드 BCD는 **'2인용 오토바이'** 같아요. 언팩드 BCD는 각 숫자가 혼자 오토바이를 타지만, 팩드 BCD는 두 숫자가 한 오토바이에 짝꿍으로 타요!
2. 그래서 연료(메모리)가 절반으로 줄어들어요. 10칸 짜리 주차장에 20개의 숫자를 주차할 수 있거든요
3. 단, 마지막 오토바이에는 한 명만 타고, 나머지 한 자리에는 "이 숫자는 양수야!"라는 표시판(부호)을 들고 있어요
