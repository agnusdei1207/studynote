# 팩드 BCD (Packed BCD)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 팩드 BCD (Packed Binary Coded Decimal)는 1바이트(8비트)에 두 개의 십진수 니블(Nibble, 4비트)을 저장하여 데이터 밀도를 높인 코딩 방식으로, 하위 4비트에 부호 정보(Sign Nibble)를 통합하여 저장한다.
> 2. **가치**: 언팩드 BCD(Unpacked BCD) 대비 최대 50%의 메모리 공간을 절약하며, IBM 메인프레임(Mainframe)의 z/Architecture와 COBOL(COmmon Business Oriented Language) 시스템에서 화폐 및 금융 데이터 처리의 표준으로 사용된다.
> 3. **융합**: 하드웨어적으로는 전용 명령어(예: `AP`, `SP`)를 통해 고정소수점(Fixed Point) 연산을 수행하고, 소프트웨어적으로는 Java의 `BigDecimal`이나 데이터베이스의 `DECIMAL` 타입과 연동되는 레거시 시스템의 핵심 기술이다.

## Ⅰ. 개요 (Context & Background) - [500자+]

- **개념**: 팩드 BCD(Packed BCD)는 10진수 한 자릿수를 표현하는 데 4비트(니블)를 사용하는 BCD(Binary Coded Decimal) 인코딩의 최적화된 형태다. 일반적인 BCD가 1바이트(8비트)에 1자릿수와 플래그 등을 저장하는 데 비해, 팩드 BCD는 1바이트 안에 두 자릿수(Digit)를 '압축(Pack)'하여 저장한다. 이때 상위 4비트(High Nibble)에는 십의 자리, 하위 4비트(Low Nibble)에는 일의 자리가 위치한다. 부호 있는 숫자의 경우, 가장 오른쪽(최하위) 바이트의 하위 4비트를 숫자 대신 부호(Sign)를 나타내는 데 사용한다(예: `1100`(C)는 양수, `1101`(D)는 음수).
- **💡 비유**: 언팩드 BCD가 각 손님(숫자)마다 4인석 테이블(바이트) 전체를 독차지하는 것이라면, 팩드 BCD는 **'2인용 좌석에 커플 손님을 배치하는 것'**과 같다. 레스토랑(메모리) 공간 효율이 2배로 좋아진다.

- **등장 배경**:
  1. **1960년대 상업용 컴퓨팅의 요구**: IBM System/360은 과학적 계산보다는 재무, 회계, 급여 관리 등 '10진수' 위주의 연산이 필요한 기업을 타겟으로 했다. 이진수(Binary) 변환 오버헤드와 10진수 표현의 비효율성을 해결하기 위해 팩드 BCD를 도입했다.
  2. **COBOL과의 결합**: COBOL의 `COMPUTATIONAL-3`(PACKED-DECIMAL) 타입은 메인프레임 하드웨어의 팩드 BCD 연산 능력을 그대로 활용하여, 0.01달러 단위까지 오차 없이 계산해야 하는 금융 로직의 필수 요소가 되었다.
  3. **현대의 레거시 통합**: x86 아키텍처 초기(8086 등)는 `DAA`(Decimal Adjust for Addition) 명령어로 이를 지원했으나, 64비트 시대로 넘어오면서 하드웨어 지원이 중단되었다. 그러나 금융권, 공공기관의 메인프레임 시스템(z/OS, VSE)에서는 여전히 핵심 데이터 포맷으로 사용된다.

- **📢 섹션 요약 비유**: 마치 1인용 책상에 1명을 앉히는 대신, **'칸막이로 나눈 2인용 책상'**에 2명을 앉혀서 사무실(메모리) 공간 효율을 극대화하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive) - [1,000자+]

### 1. 구성 요소 상세 분석

| 구성 요소 (Component) | 비트 범위 | 역할 (Role) | 내부 동작 (Internal Operation) | 비유 (Analogy) |
|:---|:---:|:---|:---|:---|
| **Digit Nibble** | 상위/하위 4비트 | 0~9의 숫자 데이터 저장 | `0000`(0) ~ `1001`(9)까지의 값을 저장. `1010`~`1111`은 미사용(Invalid) | 실제 짐가방 |
| **Sign Nibble** | 최하위 바이트의 하위 4비트 | 정수의 부호(+/-) 결정 | `1100`(C=Plus), `1101`(D=Minus), `1111`(F=Plus) 등 ISA에 따라 정의됨 | 짐가방에 붙은 파란색(양수)/빨간색(음수) 태그 |
| **Zone Nibble** | (Unpacked 시) 상위 4비트 | 문자셋 구분 (보통 0011) | EBCDIC 기준 `1111`, ASCII 기준 `0011` | 사람이 읽기 위한 껍데기 |
| **Padding (0000)** | 최상위 바이트의 상위 4비트 | 홀수 자릿수 정렬 | 데이터가 홀수 자릿수일 때, 상위 니블을 0으로 채워 4비트 경계를 맞춤 | 빈 좌석에 놓는 예약 표시 |
| **Decimal Point** | (논리적) 프로그램 카운터 | 소수점 위치 연산 | 하드웨어적으로 존재하지 않으며, COBOL `PIC` 절이나 소프트웨어적으로 위치를 가정 | 눈금자의 기준점 점 |

### 2. ASCII 구조 다이어그램: 팩드 BCD 메모리 레이아웃

이 다이어그램은 10진수 `+123456`이 메인프레임 메모리(Big-Endian)에 어떻게 저장되는지 보여준다. 부호는 마지막 바이트의 하위 니블에 위치한다.

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │         팩드 BCD (Packed BCD) 메모리 매핑 예시: +123456                      │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │  Target Data: + 1 2 3 4 5 6                                                 │
  │                                                                              │
  │  ┌───────────────────────────────────────────────────────────────────────┐  │
  │  │  Byte 0 (Addr 0)       │  Byte 1 (Addr 1)       │  Byte 2 (Addr 2)    │  │

  │  │  High Nibble │ Low Nibble │ High Nibble │ Low Nibble │ High │ Low    │  │
  │  │  (4bits)     │ (4bits)    │  (4bits)    │ (4bits)    │ Nibble│ Nibble │  │
  │  ├──────────────┼────────────┼─────────────┼─────────────┼───────┼────────┤  │
  │  │   0001       │   0010     │   0011      │   0100     │ 0101  │ 1100   │  │

  │  └──────────────┴────────────┴─────────────┴─────────────┴───────┴────────┘  │
  │       ▲            ▲            ▲             ▲             ▲      ▲        │
  │       └──────┬─────┴──────┬─────┴──────┬──────┴──────┬──────┘      │        │
  │              │            │            │             │             │        │
  │              ▼            ▼            ▼             ▼             │        │
  │           "1"          "2"          "3"          "4"          "5"     │        │
  │                                                                     │        │

  Hex Representation (Big-Endian): 0x12 0x34 0x5C
```

**[다이어그램 해설]**
위 다이어그램은 십진수 `+123456`이 메모리에 저장되는 과정을 도식화한 것이다.
1.  **압축(Packing) 과정**: 6개의 십진수 자릿수(1~6)는 3쌍의 니블로 구성된다. 첫 번째 바이트는 `1`과 `2`가, 두 번째 바이트는 `3`과 `4`가 결합한다.
2.  **부호 처리**: 마지막(최우측) 자릿수인 `6` 뒤에는 더 이상 숫자가 없으므로, 마지막 바이트의 하위 니블(Low Nibble)에 부호 정보(`C` = 1100, Positive)를 배치한다.
3.  **바이트 순서**: IBM 메인프레임은 Big-Endian 방식을 사용하므로 가장 왼쪽의 자릿수(1)가 가장 낮은 메모리 주소(Base Address)부터 저장된다. 결과적으로 3바이트 공간에 6자릿수와 부호를 모두 담을 수 있게 되어 언팩드 방식 대비 메모리를 절약한다.

### 3. 심층 동작 원리 및 연산 메커니즘

팩드 BCD의 연산은 내부적으로 **2진수 연산 → 보정(Adjustment)** 과정을 거친다.

**① 덧셈(Addition)의 경우:**
CPU는 먼저 두 팩드 BCD 피연산자에 대해 일반적인 이진수 덧셈을 수행한다.
- 예: `59` (0x59) + `23` (0x23) → `0101 1001` + `0010 0011` = `0111 1100` (0x7C)
- 이진수 결과 `0111 1100`은 BCD 관점에서 해석이 불가능하다(하위 니블 `1100`은 유효한 BCD 숫자가 아님).
- 이때 **`DAA` (Decimal Adjust for Addition)** 명령어가 실행된다. CPU는 각 니블을 검사하여 9보다 크거나 캐리(Carry)가 발생했는지 확인한다. 만약 그렇다면, 해당 니블에 `6`(`0110`)을 더하여 carry를 발생시키고 다음 니블로 전파시킨다.
- `0111 1100` (하위 니블 12) → `0110` 더함 → `0111 1100` + `0000 0110` = `1000 0010` (0x82). 다시 carry 처리 등을 거쳐 최종적으로 `0101 1001` (0x59) + `0010 0011` (0x23) = `1000 0010` (0x82, 즉 82)가 도출된다.

**② IBM z/Architecture 명령어:**
x86과 달리 IBM 메인프레임은 팩드 BCD 연산을 위한 전용 명령어를 가진다.
- **`AP` (Add Packed)**: 메모리 대 메모리 팩드 BCD 덧셈 수행. 레지스터 로딩 없이 메모리 간 직접 연산하여 성능 최적화.
- **`SP` (Subtract Packed)**: 팩드 BCD 뺄셈.
- **`MP` (Multiply Packed)**: 팩드 BCD 곱셈.
- **`DP` (Divide Packed)**: 팩드 BCD 나눗셈.

**③ 코드 스니펫 (레거시 시스템 제어 예시)**
```assembly
; x86 Assembly (Legacy) 예시: AL 레지스터에 있는 두 팩드 BCD 숫자 더하기
; 입력: AL = 0x59 (59), BL = 0x23 (23)
; 출력: AL = 0x82 (82)

MOV AL, 59h   ; 59 (Packed BCD)
MOV BL, 23h   ; 23 (Packed BCD)
ADD AL, BL    ; 이진수 덧셈 수행 (AL = 7Ch)
DAA           ; Decimal Adjust for Accumulator 실행
              ; 내부 로직: 하위 니블 보정 -> 상위 니블 보정
              ; 결과: AL = 82h (82)
```

- **📢 섹션 요약 비유**: 마치 **'낡은 화폐 계산기'**와 같습니다. 내부 기어(이진수 연산)는 돌아가지만, 우리가 눈으로 확인하는 숫자(BCD)가 맞게 나오도록 기어 사이에 복잡한 **'조정 장치(Adjustment Logic)'**가 달려 있어 항상 10진수 결과만을 내뱉습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy) - [비교표 2개+]

### 1. 심층 기술 비교: 팩드 BCD vs 언팩드 BCD vs 이진수

| 비교 항목 | 팩드 BCD (Packed) | 언팩드 BCD (Unpacked/Zoned) | 정수형 (Binary Integer) |
|:---|:---|:---|:---|
| **저장 방식** | 1 Byte = 2 Digits | 1 Byte = 1 Digit (남는 4비트는 Zone) | 1 Byte = 256 Values (2^8) |
| **공간 효율성** | ⭐⭐⭐⭐⭐ (매우 높음) | ⭐⭐ (낮음) | ⭐⭐⭐⭐⭐ (가장 높음) |
| **산술 연산 속도** | 느림 (보정 필요) | 매우 느림 (변환 후 연산) | 매우 빠름 (ALU 네이티브) |
| **입출력(I/O) 변환** | 변환 오버헤드 있음 | 거의 없음 (문자와 1:1 매핑) | 변환 오버헤드 큼 (2→10진 변환) |
| **정밀도 손실** | 없음 (Exact) | 없음 (Exact) | 있음 (Floating Point) |
| **주요 하드웨어** | IBM Mainframe (z/Arch), x86(Legacy) | IBM Mainframe, Microcontrollers | 모든 현대 CPU |
