# 106. UTF-16

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 유니코드 코드 포인트를 2바이트(기본 다국어 평면) 또는 4바이트(서로게이트 쌍)로 인코딩하는 가변 길이 방식으로, Windows·Java 내부 표현의 표준
> 2. **가치**: BMP 내 문자는 2바이트 고정으로 효율적이며, 한글·CJK 문자가 UTF-8(3바이트)보다 33% 공간 절약
> 3. **융합**: 윈도우 API·Java String·JavaScript의 내부 문자열 표현으로, 시스템 프로그래밍과 크로스플랫폼 개발의 핵심 인코딩

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

UTF-16(16-bit Unicode Transformation Format)은 유니코드 코드 포인트를 하나 또는 두 개의 16비트 코드 유닛으로 인코딩하는 방식이다. 기본 다국어 평면(BMP, U+0000~U+FFFF)은 2바이트, 보충 평면(U+10000~U+10FFFF)은 서로게이트 쌍을 사용하여 4바이트로 표현한다.

### 💡 비유: 2층짜리 주차장

UTF-16은 **2층짜리 주차장**과 같다. 1층(기본 평면)에는 대부분의 차(문자)가 주차하고, 2층(서로게이트)은 특별한 대형 차량(이모지, 고대 문자)만 사용한다. 1층은 항상 2칸이 필요하고, 2층 차량은 4칸이 필요하다. Windows와 Java는 이 주차장을 기본으로 사용한다.

### 등장 배경

| 시기 | 배경 | 해결책 |
|:---:|:---|:---|
| 1990대 초 | UCS-2는 2바이트 고정으로 BMP만 표현 | BMP 이상 확장 필요 |
| 1996 | Unicode 2.0 서로게이트 메커니즘 도입 | UTF-16 가변 길이화 |
| 2000대 | Windows XP, Java 내부 인코딩 채택 | OS/언어 표준화 |
| 현재 | 이모지 확산으로 서로게이트 활용 증가 | 4바이트 문자 빈번 |

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 범위 | 비고 |
|:---|:---|:---|:---|
| 코드 유닛 | UTF-16의 기본 단위 | 16비트 | 1~2개로 코드 포인트 구성 |
| BMP 코드 | 기본 평면 직접 표현 | U+0000~U+FFFF | U+D800~U+DFFF 제외 |
| High Surrogate | 서로게이트 쌍 시작 | U+D800~U+DBFF | 1024개 |
| Low Surrogate | 서로게이트 쌍 완성 | U+DC00~U+DFFF | 1024개 |
| BOM | 바이트 순서 표식 | 0xFEFF | LE: FF FE, BE: FE FF |

### UTF-16 인코딩/디코딩 코드

```c
#include <stdint.h>

// UTF-16 인코딩: 코드 포인트 → UTF-16 코드 유닛
int utf16_encode(uint32_t cp, uint16_t *out) {
    if (cp <= 0xFFFF) {
        // BMP: 직접 저장 (서로게이트 영역 제외 검증 필요)
        if (cp >= 0xD800 && cp <= 0xDFFF) return -1; // 무효
        out[0] = (uint16_t)cp;
        return 1;
    }
    else if (cp <= 0x10FFFF) {
        // 서로게이트 쌍 사용
        uint32_t cp_prime = cp - 0x10000;
        out[0] = 0xD800 + (uint16_t)(cp_prime >> 10);   // High
        out[1] = 0xDC00 + (uint16_t)(cp_prime & 0x3FF); // Low
        return 2;
    }
    return -1; // 무효 코드 포인트
}

// UTF-16 디코딩: UTF-16 코드 유닛 → 코드 포인트
int utf16_decode(const uint16_t *in, uint32_t *cp) {
    uint16_t unit = in[0];

    if (unit < 0xD800 || unit > 0xDFFF) {
        // BMP 문자
        *cp = unit;
        return 1;
    }
    else if (unit >= 0xD800 && unit <= 0xDBFF) {
        // High Surrogate → Low Surrogate 필요
        uint16_t high = unit;
        uint16_t low = in[1];

        if (low < 0xDC00 || low > 0xDFFF) return -1; // 잘못된 쌍

        uint32_t cp_prime = ((high - 0xD800) << 10) | (low - 0xDC00);
        *cp = cp_prime + 0x10000;
        return 2;
    }
    return -1; // 잘못된 시퀀스 (Low로 시작)
}
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### UTF-8 vs UTF-16 vs UTF-32 비교

| 특성 | UTF-8 | UTF-16 | UTF-32 |
|:---|:---:|:---:|:---:|
| **코드 유닛** | 8비트 | 16비트 | 32비트 |
| **BMP 문자** | 1~3바이트 | 2바이트 | 4바이트 |
| **서로게이트 문자** | 4바이트 | 4바이트 | 4바이트 |
| **영문 'A'** | 1바이트 | 2바이트 | 4바이트 |
| **한글 '가'** | 3바이트 | 2바이트 | 4바이트 |
| **이모지 '😀'** | 4바이트 | 4바이트 | 4바이트 |
| **ASCII 호환** | ✅ | ❌ | ❌ |
| **엔디안 영향** | ❌ | ✅ | ✅ |
| **랜덤 액세스** | ❌ | ❌ | ✅ |
| **주요 용도** | 웹, 파일 | Windows, Java | 내부 처리 |

### 과목 융합: 운영체제 관점

| 영역 | UTF-16 활용 | 비고 |
|:---|:---|:---|
| **Windows API** | 모든 유니코드 API가 UTF-16LE | `WideCharToMultiByte` 변환 함수 |
| **NTFS** | 파일명 UTF-16LE 저장 | 최대 255 UTF-16 코드 유닛 |

| **프로세스 통신** | COM/OLE 자동화 UTF-16 | 크로스 프로세스 문자열 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오

**시나리오 1: Windows 네이티브 앱 개발**
- 상황: Win32 API 기반 애플리케이션
- 결정: 내부 문자열 UTF-16LE 사용
- 이유: API 호출 시 변환 오버헤드 0%

**시나리오 2: 한글 중심 서비스**
- 상황: 한국 전용 서비스, 이모지 사용 빈도 낮음
- 결정: 저장소에 UTF-16 고려
- 이유: 한글 2바이트 → UTF-8(3바이트) 대비 33% 절감

**시나리오 3: 크로스 플랫폼 라이브러리**
- 상황: Windows/macOS/Linux 지원 라이브러리
- 결정: 내부 UTF-8, 경계에서 UTF-16 변환
- 이유: 네트워크/파일 호환성 확보

### 도입 체크리스트

| 기술적 | 운영적 |
|:---|:---|
| ☐ 엔디안 처리 로직 확인 | ☐ BOM 포함 여부 결정 |
| ☐ 서로게이트 쌍 처리 구현 | ☐ 기존 EUC-KR 마이그레이션 계획 |
| ☐ wchar_t 크기 플랫폼별 검증 | ☐ 문자열 길이 계산 로직 검토 |
| ☐ UTF-8 변환 함수 준비 | ☐ 이모지 처리 요구사항 확인 |

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량/정성 기대효과

| 항목 | UTF-8 사용 시 | UTF-16 사용 시 (한글) | 효과 |
|:---|:---:|:---:|:---|
| **한글 문서 크기** | 3바이트/글자 | 2바이트/글자 | 33% 절감 |
| **영문 혼합 문서** | 효율적 | 비효율 | 내용 비율 따라 결정 |
| **처리 속도** | 변환 필요 | API 직접 호출 | Windows에서 유리 |
| **이모지 처리** | 4바이트 | 서로게이트 쌍 | 동일 |

### 미래 전망

1. **UTF-8로의 수렴**: 웹 표준의 사실상 UTF-8 독점
2. **Windows 지속**: WinRT/UWP 여전히 UTF-16 기반
3. **서로게이트 빈도 증가**: 이모지 사용 확대로 4바이트 문자 증가

### 👶 어린이를 위한 3줄 비유 설명

**개념**: UTF-16은 글자를 담는 상자가 항상 2칸 크기인 방식이에요. 하지만 아주 특별한 이모지 같은 건 4칸이 필요해서 두 개의 상자를 붙여서 써요.

**원리**: 기본 글자(한글, 영어, 숫자)는 2칸 상자 하나면 되지만, 이모지는 '윗부분 상자'와 '아랫부분 상자'라는 두 개의 특별한 상자가 붙어야 하나가 돼요.

**효과**: 한글을 저장할 때 UTF-8보다 공간을 아낄 수 있어서, 윈도우 컴퓨터 내부에서는 UTF-16을 많이 사용해요!
