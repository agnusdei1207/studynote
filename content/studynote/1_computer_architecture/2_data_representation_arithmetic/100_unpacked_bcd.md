# 100. 언팩드 BCD (Unpacked BCD)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 언팩드 BCD(Unpacked BCD)는 각 십진수 자릿수(0~9)를 8비트(1바이트)의 하위 니블에 저장하고 상위 니블에 플래그(0000 또는 0011)를 저장하는 방식으로, 데이터 처리와 표현 간의 오버헤드를 최소화하는 인코딩 기법이다.
> 2. **가치**: 7세그먼트 디코더와 같은 하드웨어 인터페이스 및 ASCII 변환 로직에서 비트 연산(Bitwise Operation) 없이 직접 매핑 가능하여, 임베디드 시스템의 실시간성과 디버깅 효율을 비약적으로 향상시킨다.
> 3. **융합**: x86 아키텍처의 AAA(After Addition Adjust) 명령어와 COBOL의 Zoned Decimal 데이터 타입의 근간이 되며, 고성능 컴퓨팅이 아닌 신뢰성 있는 계측 및 제어 시스템에서 여전히 유효하다.

---

## Ⅰ. 개요 (Context & Background)

- **개념 및 정의**:
    언팩드 BCD(Unpacked BCD)는 10진수의 각 자릿수(Digit)를 독립된 1바이트(Byte, 8비트) 공간에 저장하는 이진화 십진 코드(Binary Coded Decimal)의 일종이다. 
    데이터는 하위 4비트(Lower Nibble)에 0000(0)부터 1001(9)까지의 값으로 저장되며, 상위 4비트(Higher Nibble)는 보통 0000(0x00) 또는 0011(0x03)으로 채워진다. 특히 상위 니블이 0011인 경우, 이는 ASCII 코드의 숫자 문자('0'~'9')와 바이너리 호환이 되며, 이를 **ASCII Zoned Decimal** 또는 단순히 **Zoned Format**이라고 부른다.

- **💡 비유**: 
    언팩드 BCD는 **'각 손님이 독립된 1인용 테이블을 사용하는 식당'**과 같다. 
    팩드 BCD(Packed BCD)가 두 명이 한 테이블(1바이트)을 비좁게 공유하는 것이라면, 언팩드 BCD는 한 명(숫자 1개)이 널널한 테이블(1바이트)을 독차지한다. 
    식당의 공간 효율(메모리)은 떨어지지만, 손님을 교체하거나 자리를 안내하는(입출력 변환) 절차가 훨씬 수월하고 직관적이다.

- **등장 배경 및 기술적 필요성**:
    1.  **초기 컴퓨팅 환경의 I/O 편의성**: 1960~70년대 컴퓨터는 10진수 입출력이 빈번했다. 2진수 연산 후 10진수로 변환하는 복잡한 수학적 연산(Division/Modulo)을 피하기 위해, 10진수 자릿수를 그대로 저장하면서도 ASCII 문자와 매핑이 쉬운 형식이 요구되었다.
    2.  **하드웨어 디코더와의 직접 연결**: 7세그먼트 디스플레이(7-Segment Display)를 구동하는 BCD-7세그먼트 디코더(예: 74LS47)는 4비트 BCD 입력을 받아들이도록 설계되었다. 언팩드 BCD는 상위 비트를 무시(Masking)하고 하위 4비트만 곧장 하드웨어로 전송하면 되므로 인터페이스 회로가 단순화된다.
    3.  **문자열 처리와의 친화성**: 현대의 네트워크 프로토콜이나 데이터베이스는 숫자를 문자열(String)로 다루는 경우가 많다. 언팩드 BCD(Zoned 형식)는 별도의 인코딩 과정 없이 메모리 덤프 시 바로 가독성 있는 텍스트로 확인된다는 장점이 있다.

- **📢 섹션 요약 비유**: 
    마치 복잡한 번역사(변환 소프트웨어) 없이도, 손님(데이터)이 입구에 들어서자마자 자신의 표(ASCII 코드)를 확인하고 바로 자리에 앉을 수 있도록, 자리(바이트)를 넉넉하게 배정한 초대장 시스템과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 데이터 구조 및 메모리 레이아웃

언팩드 BCD는 **순수 언팩드 BCD(Pure Unpacked BCD)**와 **ASCII Zoned Decimal** 두 가지 주요 변형이 있다. 
이 두 형식의 차이는 상위 4비트(High Nibble)의 값에 따라 결정되며, 이는 하드웨어 설계나 소프트웨어 파싱 전략에 결정적인 영향을 미친다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│         언팩드 BCD 메모리 매핑 구조도                                         │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  [Type A] Pure Unpacked BCD (순수 언팩드 BCD)                                │
│  ┌──────────────────────────────────────────────────────────────────────────┐│
│  │  MSB b7 b6 b5 b4 (High Nibble) │ LSB b3 b2 b1 b0 (Low Nibble)          ││
│  │            │   0   0   0   0  │          0 ~ 9 (BCD Value)             ││
│  │            └──────────────────┴─────────────────────────────────────────┘││
│  │                                                                              ││
│  │  예: 십진수 '5'  →  0000 0101 (0x05)                                        ││
│  │  예: 십진수 '0'  →  0000 0000 (0x00)                                        ││
│  └──────────────────────────────────────────────────────────────────────────┘│
│                                                                              │
│  [Type B] ASCII Zoned Decimal (IBM/COBOL Standard)                            │
│  ┌──────────────────────────────────────────────────────────────────────────┐│
│  │  MSB b7 b6 b5 b4 (Zone) │ LSB b3 b2 b1 b0 (Digit)                       ││
│  │            │   0   0   1   1  │          0 ~ 9 (BCD Value)             ││
│  │            │   (Hex 3)       │                                         ││
│  │            └──────────────────┴─────────────────────────────────────────┘││
│  │                                                                              ││
│  │  예: 십진수 '5'  →  0011 0101 (0x35) → ASCII 문자 '5'와 동일              ││
│  │  예: 십진수 '0'  →  0011 0000 (0x30) → ASCII 문자 '0'와 동일              ││
│  └──────────────────────────────────────────────────────────────────────────┘│
│                                                                              │
│  [Data Alignment Comparison]                                                  │
│  ┌──────────────────────────────────────────────────────────────────────────┐│
│  │  Decimal Value: 1 2 3                                                    ││
│  │                                                                           ││
│  │  [Pure Unpacked BCD]                                                      ││
│  │  Address 0: 0000 0001 (0x01)                                             ││
│  │  Address 1: 0000 0010 (0x02)                                             ││
│  │  Address 2: 0000 0011 (0x03)                                             ││
│  │                                                                           ││
│  │  [Packed BCD (참고용 비교)]                                               ││
│  │  Address 0: 0001 0010 (0x12)  <-- 2자리가 압축됨                          ││
│  │  Address 1: 0000 0011 (0x03)                                             ││
│  └──────────────────────────────────────────────────────────────────────────┘│
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

**[해설]**:
위 다이어그램과 같이 언팩드 BCD는 1바이트당 10진수 1자리를 저장한다. 
순수 언팩드 형식은 상위 니블이 0이므로 산술 논리 장치(ALU)가 상위 니블을 무시하고 연산할 때 유리하며, 주로 마이크로프로세서 내부 레지스터 연산에 사용된다. 
반면, Zoned Decimal 형식은 상위 니블이 3(0011)으로 고정되어 있어, 0x30을 더하거나 뺄 필요 없이 곧바로 ASCII 코드로 인식된다. 이는 `printf()`와 같은 표준 출력 함수나 터미널 I/O에서 즉시 문자열로 처리할 수 있음을 의미한다.

### 2. 언팩드 BCD 산술 연산 메커니즘 (x86 Arch)

언팩드 BCD는 메모리는 낭비하지만, 연산 후의 보정(Correction) 과정이 팩드 BCD보다 직관적일 수 있다. 
Intel 8086 및 x86 호환 프로세서는 이를 위해 전용 명령어를 제공했다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│         x86 언팩드 BCD 덧셈 연산 알고리즘 (AAA Instruction)                   │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  1. 초기 상태: AL 레지스터에 두 개의 언팩드 BCD 값이 더해짐                     │
│     예: 5 + 7                                                               │
│     MOV AL, 05h   ; AL = 0000 0101 (5)                                      │
│     ADD AL, 07h   ; AL = 0000 1100 (12 = 0x0C)                              │
│                  ; 문제 발생: 하위 4비트가 9를 초과함 (BCD 유효 범위 위반)      │
│                                                                              │
│  2. AAA (ASCII Adjust for Addition) 명령어 수행                               │
│     ┌──────────────────────────────────────────────────────────────────────┐│
│     │  if ( (AL & 0Fh) > 9 ) or (AF == 1) then                             ││
│     │       AL = AL + 6         ; 16진수 6를 더하여 캐리를 발생시킴          ││
│     │       AH = AH + 1         ; 다음 자릿수(10의 자리)에 올림(Carry)       ││
│     │       AF = 1, CF = 1     ; 보조 캐리, 캐리 플래그 설정                ││
│     │  end if                                                                 ││
│     │  AL = AL & 0Fh           ; 상위 니블을 0으로 만들어 언팩드 형태 유지    ││
│     └──────────────────────────────────────────────────────────────────────┘│
│                                                                              │
│  3. 수행 과정 시뮬레이션:                                                     │
│     현재 AL = 0x0C (1100)                                                     │
│     - 하위 4비트(1100, 12)이 9보다 크므로 조건 만족                           │
│     - AL = 0x0C + 6 = 0x12 (0001 0010)                                        │
│     - AH = AH + 1 (AH에 10의 자리 저장)                                        │
│     - AL = AL & 0x0F -> AL = 0x02 (결과의 1의 자리)                           │
│     - 최종 결과: AH=01, AL=02  -> 의미하는 값: 12                             │
│                                                                              │
│  ⚠ 주의: x86-64(64비트 모드)에서는 AAA, AAD, AAM, AAS 명령어가 제거됨.       │
│           현대에는 소프트웨어적으로 아래 로직을 구현해야 함.                   │
└──────────────────────────────────────────────────────────────────────────────┘
```

**[해설]**:
언팩드 BCD 연산의 핵심은 **니블(Nibble) 단위의 보정**이다. 4비트로 표현 가능한 최대값인 15(1111)와 10진수 최대값인 9의 차이인 6을 더함으로써, 결과값이 10~15 범위를 벗어나면 자연스럽게 다음 바이트로 캐리(Carry)가 발생하도록 유도한다. AAA 명령어는 이 하드웨어적 동작을 캡슐화한 것이나, 현대 RISC나 x86-64 환경에서는 컴파일러나 라이브러리 함수 내부에 위 알고리즘으로 구현된다.

### 3. 하드웨어 인터페이스 (7-Segment Display)

임베디드 시스템에서 언팩드 BCD가 가장 빛을 발하는 영역은 디스플레이 출력이다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│         언팩드 BCD와 7-Segment Display 인터페이스                             │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  MCU (Microcontroller)                                              │
│  ┌───────────────────────┐                                               │
│  │ Port A (Low Nibble)   │───[4bit Data]──┐                               │
│  │      0 0 0 1 (BCD 1)  │                │                               │
│  └───────────────────────┘                │                               │
│                                           ▼                               │
│                            ┌──────────────────────────────┐                │
│                            │  BCD to 7-Segment Decoder    │                │
│                            │  (e.g., 74LS47 / DM7447)     │                │
│                            │                              │                │
│                            │ Input: 0001 (1)              │                │
│                            │ Output: 0000110 (b, c ON)    │                │
│                            └──────────────────────────────┘                │
│                                           │                               │
│                                           ▼                               │
│                            ┌──────────────────────────────┐                │
│                            │      7-Segment LED           │                │
│                            │         ┌─┐                  │                │
│                            │        ││ │                  │                │
│                            │         └─┘  ──               │                │
│                            │        │  │                  │                │
│                            │         ──  ──               │                │
│                            │                              │                │
│                            └──────────────────────────────┘                │
│                                                                              │
│  📌 장점: 상위 4비트를 무시하므로, MCU의 Port 상위 4비트는 다른 용도        │
│          (Control 신호 등)로 사용 가능하여 하드웨어 자원 효율이 높음.        │
└──────────────────────────────────────────────────────────────────────────────┘
```
