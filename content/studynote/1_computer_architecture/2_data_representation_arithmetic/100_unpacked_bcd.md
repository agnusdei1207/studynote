# 언팩드 BCD (Unpacked BCD)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 언팩드 BCD(Unpacked BCD)는 각 십진수 자릿수를 1바이트(8비트)에 저장하는 BCD 방식으로, 하위 4비트에 자릿수(0~9)를, 상위 4비트에 0 또는 ASCII 영역 정보를 저장한다.
> 2. **가치**: 7세그먼트 디스플레이, ASCII 문자열과의 직접 변환, 디버깅 용이성 등에서 팩드 BCD보다 편리하며, 임베디드 시스템과 디지털 계측기의 디스플레이 인터페이스에 적합하다.
> 3. **융합**: 언팩드 BCD는 x86의 AAA/AAD 등 레거시 명령어에서 지원되며, z/Architecture의 Zoned Decimal 포맷과 연관된다. ASCII '0'~'9' 문자와의 변환이 단순하다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 언팩드 BCD(Unpacked BCD)는 각 십진수 자릿수(0~9)를 1바이트(8비트)에 독립적으로 저장하는 BCD 인코딩 방식이다. 하위 4비트(하위 니블)에 자릿수(0000~1001)를 저장하고, 상위 4비트(상위 니블)에는 0(0000) 또는 ASCII 영역 정보(0011 = '0'~'9'의 상위 니블)를 저장한다. 예를 들어 십진수 5는 언팩드 BCD로 `0000 0101`(0x05) 또는 ASCII로 `0011 0101`(0x35 = '5')로 저장된다.

- **💡 비유**: 언팩드 BCD는 **'각 승객이 개별 택시를 타는 것'**과 같다. 팩드 BCD가 두 승객이 한 오토바이에 함께 탄다면, 언팩드 BCD는 각 승객(자릿수)이 자신만의 택시(1바이트)를 탄다. 비용(메모리)은 더 들지만, 각자 독립적으로 행동할 수 있다.

- **등장 배경 및 발전 과정**:
  1. **ASCII와의 호환성**: 십진수를 ASCII 문자 '0'~'9'로 표현하면, 상위 4비트는 항상 0011이고 하위 4비트만 자릿수에 따라 다르다. 언팩드 BCD는 이 ASCII 숫자와 하위 니블이 동일하다.
  2. **x86의 지원**: Intel 8086은 AAA(ASCII Adjust After Addition), AAD(ASCII Adjust Before Division), AAM(ASCII Adjust After Multiply) 등 언팩드 BCD 연산 보정 명령어를 제공했다. 이는 "ASCII 산술"이라 불렸다.
  3. **현대의 제한적 사용**: 64비트 x86에서는 AAA 등이 제거되었지만, 임베디드 시스템, 디지털 계측기, 7세그먼트 디스플레이 인터페이스에서는 여전히 유용하다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 언팩드 BCD 포맷 구조

언팩드 BCD는 1바이트에 한 자릿수를 저장한다. 상위 니블의 내용에 따라 순수 언팩드 BCD와 ASCII Zoned Decimal로 구분된다.

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │         언팩드 BCD 포맷 구조                                                  │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │   순수 언팩드 BCD (Pure Unpacked BCD):                                       │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  1바이트 = [ 0000 ] [ 자릿수 0~9 ]                                      ││
  │   │             상위4비트   하위4비트                                        ││
  │   │              (항상 0)                                                   ││
  │   │                                                                         ││
  │   │  예: 십진수 5                                                          ││
  │   │  ┌─────────────────────────────────────────────────────────────────────┐││
  │   │  │  0000 0101 = 0x05                                                   │││
  │   │  │    ↑     ↑                                                          │││
  │   │  │  상위=0  하위=5                                                      │││
  │   │  └─────────────────────────────────────────────────────────────────────┘││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   ASCII Zoned Decimal (ASCII 호환 언팩드 BCD):                                │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  1바이트 = [ 0011 ] [ 자릿수 0~9 ]                                      ││
  │   │             상위4비트   하위4비트                                        ││
  │   │            (ASCII '0'~'9'의 상위 니블)                                   ││
  │   │                                                                         ││
  │   │  예: 십진수 5                                                          ││
  │   │  ┌─────────────────────────────────────────────────────────────────────┐││
  │   │  │  0011 0101 = 0x35 = ASCII '5'                                       │││
  │   │  │    ↑     ↑                                                          │││
  │   │  │  상위=3  하위=5  → ASCII '0'(0x30) + 5 = '5'                        │││
  │   │  └─────────────────────────────────────────────────────────────────────┘││
  │   │                                                                         ││
  │   │  ⚠ ASCII Zoned Decimal은 printf, puts 등으로 직접 출력 가능!          ││
  │   │     순수 언팩드 BCD는 0x30을 더해야 ASCII가 됨                          ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   십진수 123을 각 방식으로 표현:                                              │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  순수 언팩드 BCD:  0x01 0x02 0x03  (3바이트)                            ││
  │   │  ASCII Zoned:      0x31 0x32 0x33  (3바이트) = "123"                   ││
  │   │  팩드 BCD:          0x01 0x23       (2바이트, 부호 없는 경우)            ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 언팩드 BCD는 1바이트에 한 자릿수를 저장한다. 순수 언팩드 BCD는 상위 니블이 0이고, ASCII Zoned Decimal은 상위 니블이 0011(= 3)이다. ASCII Zoned Decimal은 0x30을 더하면 ASCII '0'~'9'가 되어, `printf`로 직접 출력할 수 있다. 예를 들어 십진수 123은 순수 언팩드 BCD로 `0x01 0x02 0x03`, ASCII Zoned로 `0x31 0x32 0x33` = "123"이 된다.

### 언팩드 BCD 덧셈 보정 (x86 AAA 명령어)

언팩드 BCD 덧셈은 일반 이진수 덧셈 후, 결과가 9를 초과하면 보정(Correction)한다. x86의 AAA(ASCII Adjust After Addition) 명령어가 이를 수행했다.

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │         언팩드 BCD 덧셈 보정 (AAA: ASCII Adjust After Addition)              │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │   알고리즘:                                                                  │
  │   1. AL = AL + other (일반 이진수 덧셈)                                      │
  │   2. AAA 보정:                                                              │
  │      if (AL & 0x0F > 9) OR (AF = 1) then                                    │
  │          AL = AL + 6                                                        │
  │          AH = AH + 1  (상위 자릿수로 캐리)                                    │
  │          AF = 1, CF = 1                                                     │
  │      AL = AL & 0x0F  (상위 니블 클리어)                                      │
  │                                                                              │
  │   예시: 5 + 7 = 12                                                          │
  │   ┌────────────────────────────────────────────────────────────────────────┐│
  │   │  MOV AL, 05h    ; AL = 0x05 (언팩드 BCD 5)                              ││
  │   │  ADD AL, 07h    ; AL = 0x05 + 0x07 = 0x0C (12, 하위 니블 > 9)          ││
  │   │  AAA            ; 보정:                                                 ││
  │   │                 ;   0x0C > 9 → AL = 0x0C + 6 = 0x12                    ││
  │   │                 ;   AH = AH + 1 (상위 자릿수)                           ││
  │   │                 ;   AL = AL & 0x0F = 0x02 (하위 자릿수)                 ││
  │   │  결과: AH = 1, AL = 2 → 십진수 12                                       ││
  │   │        (AH=1은 10의 자리, AL=2는 1의 자리)                              ││
  │   └────────────────────────────────────────────────────────────────────────┘│
  │                                                                              │
  │   ⚠ x86-64에서 AAA, AAD, AAM, AAS 명령어는 제거됨 (Invalid Opcode)         │
  │      대신 소프트웨어적으로 BCD 연산 구현 필요                               │
  │                                                                              │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 언팩드 BCD 덧셈의 보정은 팩드 BCD와 유사하다. 결과의 하위 니블이 9를 초과하면 6을 더하고, 상위 자릿수(AH 레지스터)를 증가시킨다. x86의 AAA 명령어가 이를 자동으로 수행했지만, x86-64에서는 제거되어 소프트웨어적으로 구현해야 한다. 결과는 AH에 10의 자리, AL에 1의 자리가 저장된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 언팩드 BCD vs 팩드 BCD

| 비교 항목 | 언팩드 BCD | 팩드 BCD |
|:---|:---|:---|
| **메모리 (n자릿수)** | n 바이트 | ⌈(n+1)/2⌉ 바이트 |
| **디스플레이** | 직접 또는 간단 변환 | 변환 필요 |
| **ASCII 변환** | 0x30 더하기만 하면 됨 | 각 니블 추출 후 0x30 더함 |
| **산술 연산** | AAA/AAD 등으로 보정 | DAA/DAS 등으로 보정 |
| **하드웨어 지원** | x86(레거시), 제한적 | IBM z, x86(레거시) |
| **주 용도** | 디스플레이, ASCII 문자열 | 메인프레임, COBOL |

### 과목 융합 관점
- **컴퓨터구조(컴구)**: x86의 AAA, AAD, AAM, AAS 명령어는 언팩드 BCD 연산을 지원했지만, x86-64에서 제거되었다. 대신 소프트웨어 라이브러리를 사용해야 한다
- **임베디드 시스템**: 7세그먼트 LED 디스플레이 컨트롤러는 언팩드 BCD를 직접 받아 각 자릿수를 표시할 수 있다. 하드웨어 BCD-7세그먼트 디코더(74LS47 등)가 간단하다
- **데이터 통신(NW)**: ASCII 숫자 문자열("12345")은 언팩드 BCD(Zoned Decimal)와 동일한 포맷이다. 네트워크 프로토콜에서 숫자를 문자열로 전송할 때 자연스럽게 사용된다

## Ⅳ. 실무 적용 및 기술사적 판단
### 실무 시나리오
1. **시나리오 — 디지털 온도계 디스플레이**: 센서에서 읽은 온도값을 언팩드 BCD로 변환하고, 7세그먼트 LED에 직접 출력한다. BCD-7세그먼트 디코더(74LS47)를 사용하여 각 자릿수를 표시한다. 기술사는 언팩드 BCD를 사용하여 디스플레이 인터페이스를 단순화한다
2. **시나리오 — ASCII 문자열에서 숫자 파싱**: 네트워크 프로토콜에서 "12345"와 같은 ASCII 문자열을 받아 내부 숫자로 변환한다. 언팩드 BCD(Zoned Decimal)에서 0x30만 빼면 바로 순수 BCD가 된다. 기술사는 언팩드 BCD 형식을 사용하여 파싱 오버헤드를 최소화한다
3. **시나리오 — 레거시 x86 코드 디버깅**: 16비트 x86 어셈블리 코드에서 AAA 명령어를 사용하는 언팩드 BCD 연산을 디버깅한다. 기술사는 AAA의 동작(보정, 캐리)을 이해하고, 현대 x86-64에서는 소프트웨어적으로 동등한 로직을 구현한다
### 도입 체크리스트
- **기술적**: 7세그먼트 디스플레이 또는 ASCII 문자열과 직접 인터페이스해야 하는가? 그렇다면 언팩드 BCD가 적합하다
- **운영·보안적**: 메모리 제약이 심한가? 그렇다면 팩드 BCD를 고려해야 한다
### 안티패턴
- **언팩드 BCD로 대량 데이터 저장**: 메모리 낭비가 심하다. 대량 데이터는 팩드 BCD 또는 일반 이진수를 사용해야 한다
- **x86-64에서 AAA 명령어 사용**: Invalid Opcode 예외가 발생한다. 소프트웨어적으로 구현해야 한다

## Ⅴ. 기대효과 및 결론
### 정량/정성 기대효과
| 구분 | 팩드 BCD | 언팩드 BCD | 효과 |
|:---|:---|:---|:---|
| **정량** | 메모리 효율적 | 메모리 2배 사용 | 저장 공간 증가 |
| **정량** | 디스플레이 변환 필요 | 직접 디스플레이 가능 | 인터페이스 단순화 |
| **정성** | 메인프레임 표준 | 임베디드/디스플레이 적합 | 용도에 따른 선택 |
### 미래 전망
- **임베디드에서의 지속**: IoT 센서, 스마트 미터, 디지털 계측기에서 언팩드 BCD는 여전히 유용하다. 디스플레이와의 직접 연결이 간단하기 때문이다
- **소프트웨어 라이브러리**: 현대 언어에서는 BCD 연산을 소프트웨어 라이브러리로 제공한다. Java `BigDecimal`, Python `Decimal` 등이 이에 해당한다
### 참고 표준

- **IBM z/Architecture**: Zoned Decimal 포맷 정의

## 👶 어린이를 위한 3줄 비유 설명
1. 언팩드 BCD는 **'각 숫자가 자기 방에서 자는 것'** 같아요. 5라는 숫자가 0101이라는 4비트 방에 들어가고, 나머지 4비트는 빈 방이에요
2. 그래서 "123"을 표현하려면 "1" 방, "2" 방, "3" 방 — 총 3개의 방(3바이트)이 필요해요. 각 방이 넓어서 편하지만, 방이 많이 필요해요!
3. 그런데 각 방에 "3"이라는 숫자를 써 붙이면(ASCII), 바로 "3"이라고 읽을 수 있어서 디스플레이에 보여주기가 아주 쉬워요!
