# 105. UTF-8

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 유니코드 코드 포인트를 1~4바이트 가변 길이로 인코딩하는 방식으로, ASCII와의 완벽한 하위 호환성을 제공하는 문자 인코딩 표준
> 2. **가치**: 전 세계 웹페이지의 98% 이상이 사용하는 사실상 표준으로, 영문 1바이트·한글 3바이트의 효율적 저장과 무결성 검증이 가능
> 3. **융합**: HTTP/JSON/XML의 기본 인코딩으로 네트워크 전송 표준이며, AI 토크나이저 설계의 근간이 되는 바이트 시퀀스 구조 제공

## Ⅰ. 개요 (Context & Background)

### 개념 정의

UTF-8(8-bit Unicode Transformation Format)은 유니코드 코드 포인트를 1바이트부터 최대 4바이트까지 가변 길이로 표현하는 문자 인코딩 방식이다. 1992년 Ken Thompson과 Rob Pike가 설계했으며, 핵심 설계 목표는 **ASCII 완전 호환성**과 **자기 동기화(Self-synchronizing)** 능력이다.

### 💡 비유: 다국어 레고 블록 상자

UTF-8은 마치 **레고 블록 조립**과 같다. 기본 블록(ASCII)은 1칸짜리로 간단하지만, 복잡한 구조(한글, 이모지)를 만들려면 3~4칸 블록을 연결해야 한다. 각 블록의 시작부에는 "이건 1칸짜리야" 또는 "뒤에 2칸 더 와"라는 돌기가 있어, 중간에서 잘라도 어디서부터 다시 읽어야 하는지 알 수 있다.

### 등장 배경

| 시기 | 기존 한계 | 혁신적 해결 |
|:---:|:---|:---|
| 1980s | 지역별 인코딩(EUC-KR, Shift-JIS) 혼재 | 유니코드로 문자 집합 통합 |
| 1990s 초 | UCS-2/UTF-16은 2바이트 고정으로 비효율 | 가변 길이로 공간 효율 확보 |
| 1992 | Plan 9 OS용 파일명 인코딩 필요 | Thompson-Pike가 UTF-8 고안 |
| 현재 | 다국어 혼용 문서의 저장/전송 표준 | 웹 98%, 이메일 80% UTF-8 사용 |

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 범위/비고 |
|:---|:---|:---|:---|
| 리드 바이트 | 시퀀스 시작 및 길이 표시 | 상위 비트 패턴으로 바이트 수 결정 | `0`, `110`, `1110`, `11110` |
| 컨티뉴에이션 바이트 | 코드 포인트 하위 비트 담당 | 상위 2비트 `10`으로 후속 바이트 식별 | `10xxxxxx` |
| 페이로드 비트 | 실제 코드 포인트 값 | 리드+컨티뉴에이션 비트 결합 | 7~21비트 |
| 오버롱 시퀀스 | 비최단 인코딩 방지 | 코드 포인트별 최소 길이 강제 | 보안 취약점 차단 |
| BOM (선택) | 바이트 순서 표식 | `EF BB BF` = U+FEFF | UTF-8에서는 권장 안 함 |

### UTF-8 인코딩/디코딩 알고리즘

```c
// UTF-8 인코딩: 코드 포인트 → 바이트 시퀀스
int utf8_encode(uint32_t cp, uint8_t *out) {
    if (cp <= 0x7F) {                    // 1바이트 (ASCII)
        out[0] = (uint8_t)cp;
        return 1;
    }
    else if (cp <= 0x7FF) {              // 2바이트
        out[0] = 0xC0 | (cp >> 6);       // 110xxxxx
        out[1] = 0x80 | (cp & 0x3F);     // 10xxxxxx
        return 2;
    }
    else if (cp <= 0xFFFF) {             // 3바이트
        out[0] = 0xE0 | (cp >> 12);      // 1110xxxx
        out[1] = 0x80 | ((cp >> 6) & 0x3F);
        out[2] = 0x80 | (cp & 0x3F);
        return 3;
    }
    else if (cp <= 0x10FFFF) {           // 4바이트
        out[0] = 0xF0 | (cp >> 18);      // 11110xxx
        out[1] = 0x80 | ((cp >> 12) & 0x3F);
        out[2] = 0x80 | ((cp >> 6) & 0x3F);
        out[3] = 0x80 | (cp & 0x3F);
        return 4;
    }
    return -1;  // 무효 코드 포인트
}

// UTF-8 디코딩: 바이트 시퀀스 → 코드 포인트
int utf8_decode(const uint8_t *in, uint32_t *cp) {
    if ((in[0] & 0x80) == 0) {           // 1바이트
        *cp = in[0];
        return 1;
    }
    else if ((in[0] & 0xE0) == 0xC0) {   // 2바이트
        *cp = ((in[0] & 0x1F) << 6) | (in[1] & 0x3F);
        return 2;
    }
    else if ((in[0] & 0xF0) == 0xE0) {   // 3바이트
        *cp = ((in[0] & 0x0F) << 12) | ((in[1] & 0x3F) << 6) | (in[2] & 0x3F);
        return 3;
    }
    else if ((in[0] & 0xF8) == 0xF0) {   // 4바이트
        *cp = ((in[0] & 0x07) << 18) | ((in[1] & 0x3F) << 12) |
              ((in[2] & 0x3F) << 6) | (in[3] & 0x3F);
        return 4;
    }
    return -1;  // 잘못된 시퀀스
}
```

| **랜덤 액세스** | 불가능 | 불가능 | 가능 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오별 UTF-8 활용

**시나리오 1: 글로벌 웹 서비스**
- 문제: 30개국 언어 지원 웹사이트 구축
- 해결: 모든 계층(TS → HTML → DB) UTF-8 통일
- 성과: 인코딩 변환 오버헤드 0%, 다국어 버그 100% 해결

**시나리오 2: 레거시 시스템 마이그레이션**
- 문제: EUC-KR로 된 20년 치 데이터를 UTF-8로 변환
- 해결: 점진적 마이그레이션 + 양쪽 인코딩 지원 기간 운영
- 주의: 한글 2바이트 → 3바이트로 DB 용량 50% 증가 대비

**시나리오 3: AI 토크나이저 최적화**
- 문제: 한글 토큰 수 과다로 컨텍스트 낭비
- 해결: UTF-8 바이트 레벨 BPE → 자모 분리 후 토크나이징
- 효과: 한글 토큰 수 40% 감소

### 도입 체크리스트

| 기술적 체크 | 운영·보안 체크 |
|:---|:---|
| ☐ 모든 소스 파일 UTF-8 저장 | ☐ HTTP 헤더 `charset=utf-8` 명시 |
| ☐ DB charset = utf8mb4 (MySQL) | ☐ HTML `<meta charset="utf-8">` |
| ☐ 에디터/IDE UTF-8 기본 설정 | ☐ BOM 제거 (EF BB BF) |
| ☐ JSON/XML 파서 UTF-8 지원 확인 | ☐ 입력 검증: 오버롱 필터링 |

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량/정성 기대효과

| 항목 | 도입 전 | 도입 후 | 효과 |
|:---|:---|:---|:---|
| **인코딩 변환 비용** | 언어별 변환 필요 | 없음 | CPU 15% 절감 |
| **다국어 버그** | 분기별 20건 | 0건 | 유지보수 비용 ↓ |
| **영문 문서 크기** | UTF-16 대비 100% | 50% | 스토리지 50% 절감 |
| **개발 생산성** | 인코딩 디버깅 2h/주 | 0h | 개발 집중도 향상 |

### 미래 전망

1. **UTF-8의 입지 강화**: JSON/HTTP/HTML 등 프로토콜 기본값으로 완전 정착
2. **새로운 이모지 확장**: 매년 새로운 이모지 추가 → 4바이트 활용 증가

### 👶 어린이를 위한 3줄 비유 설명

**개념**: UTF-8은 컴퓨터가 전 세계 모든 글자를 저장하는 방법이에요. 영어는 1칸, 한글은 3칸, 이모지는 4칸짜리 상자에 담아요.

**원리**: 상자 앞에 "이건 1칸이야" 또는 "뒤에 2칸 더 있어"라는 표시를 붙여서, 컴퓨터가 중간에서 읽어도 어디부터 시작인지 알 수 있어요.

**효과**: 같은 파일에 영어, 한글, 이모지를 섞어서 쓸 수 있고, 전 세계 누구와도 글자가 깨지지 않고 소통할 수 있어요!
