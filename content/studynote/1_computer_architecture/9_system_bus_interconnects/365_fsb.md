# 프론트 사이드 버스 (FSB, Front Side Bus)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: FSB는 과거 인텔 기반 컴퓨터 시스템에서 CPU 코어와 시스템 메모리 컨트롤러(노스브리지) 사이를 연결하는 메인 데이터 통신 병렬 버스이다.
> 2. **가치**: FSB의 클럭 속도와 버스 폭(Width)이 시스템 전체의 메모리 대역폭과 체감 성능을 결정짓는 핵심 지표였으나, 병렬 통신의 물리적 한계로 인해 아키텍처의 치명적 병목 지점이 되었다.
> 3. **융합**: 멀티코어 시대로 진입하며 대역폭 한계가 극에 달하자, 직렬 점대점(P2P) 방식인 인텔의 QPI(QuickPath Interconnect), UPI(Ultra Path Interconnect) 및 AMD의 HyperTransport로 진화하며 역사 속으로 사라졌다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: FSB(Front Side Bus)는 마이크로프로세서(CPU)가 외부의 칩셋(노스브리지)을 거쳐 메인 메모리, 그래픽 카드 및 기타 시스템 장치와 통신할 때 사용하는 유일한 주 통신로이다. 한국어로는 주로 '시스템 버스(System Bus)'로 혼용되어 불렸다.
- **💡 비유**: 도시 외곽(메모리)에서 생산된 식량(데이터)을 시청(CPU)으로 나르는 **"유일한 왕복 8차선 교량"**과 같다. 차선을 넓히고(데이터 폭) 제한속도를 높여도(클럭), 다리가 하나뿐이라 출퇴근 시간에 항상 교통 체증이 발생하는 병목 구간이었다.
- **등장 배경 및 한계**: 
  1. 초기 PC 구조에서는 CPU가 외부 칩셋(노스브리지)에 의존해 메모리를 읽어왔다. 이 연결 통로가 FSB였다.
  2. 펜티엄 4 시절에 클럭 펌핑(Quad-pumped) 기술을 도입하여 한 클럭 사이클에 4번의 데이터를 전송하며 성능을 높였다. (예: 200MHz 클럭 기반으로 800MT/s 대역폭 확보)
  3. 하지만 선들이 나란히 달리는 '병렬 버스(Parallel Bus)' 특성상 주파수를 극도로 높이면 선 간의 간섭(크로스토크)과 신호 도착 시간의 불일치(클럭 스큐) 문제가 발생하여 더 이상의 확장이 불가능해졌다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 대역폭 계산

| 요소명 | 역할 | 파라미터 / 특성 | 비유 |
|:---|:---|:---|:---|
| **기본 클럭 (Base Clock)** | FSB 동작의 기준이 되는 물리적 발진 주파수 | 수백 MHz 단위 (예: 200MHz, 333MHz) | 교량의 제한 속도 |
| **데이터 폭 (Data Width)** | 한 번에 전송할 수 있는 비트 수 | 인텔 기준 주로 64비트 (8바이트) | 다리의 차선 수 (8차선) |
| **전송 횟수 / 사이클** | 한 클럭 주기당 데이터를 전송하는 횟수 | QPB(Quad-Pumped Bus) 적용 시 4회 | 한 신호등 주기에 지나가는 차량 수 |
| **노스브리지 연결** | CPU 외부의 메모리 중재자와의 인터페이스 | 신호 동기화 및 핀 물리적 연결 | 톨게이트 시스템 |

*   **FSB 최대 대역폭 공식**:
    $$ \text{Bandwidth (MB/s)} = \text{Base Clock} \times \text{Data Width (Bytes)} \times \text{Transfers per Clock} $$
    *(예시: 333MHz 기본 클럭, 64-bit(8 Byte) 폭, Quad-pumped 적용 시 = $333 \times 8 \times 4 = 10,656$ MB/s ≒ 10.6 GB/s)*

### FSB 구조의 병목 (Bottleneck) 메커니즘

FSB가 왜 현대 멀티코어 환경에서 쇠퇴할 수밖에 없었는지를 보여주는 아키텍처 다이어그램이다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                [ 멀티코어 환경에서의 FSB 병목 현상 ]                   │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  ┌───────────────┐     ┌───────────────┐     ┌───────────────┐       │
 │  │ Core 1 (CPU)  │     │ Core 2 (CPU)  │     │ Core 3 (CPU)  │ ...   │
 │  └───────┬───────┘     └───────┬───────┘     └───────┬───────┘       │
 │          │                     │                     │               │
 │ ─────────┴─────────────────────┴─────────────────────┴─────────      │
 │  ▲                                                                   │
 │  │  프론트 사이드 버스 (FSB) — 공용 단일 버스 (Shared Bus)           │
 │  │                                                                   │
 │ ─┴─────────────────────────────────────────────────────────────      │
 │          │                                                           │
 │          ▼ (병목 지점 💥 교통 체증 발생)                               │
 │  ┌───────────────┐                                                   │
 │  │ Northbridge   │ ◀─────── (메모리 제어 집중)                       │
 │  │ (MCH)         │                                                   │
 │  └───────┬───────┘                                                   │
 │          │  Memory Bus                                               │
 │          ▼                                                           │
 │     [ RAM / Memory ]                                                 │
 │                                                                      │
 │  * 문제점: 코어가 늘어날수록 하나의 FSB를 두고 코어 간에               │
 │            메모리 접근 경쟁(Contention)이 극심해짐.                    │
 │  * 해결책: FSB 폐기 → 메모리 컨트롤러를 CPU 코어 내부에 이식 (IMC)     │
 │            → 코어 간 직접 연결 인터커넥트(QPI/UPI) 도입.               │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** FSB 아키텍처의 치명적인 약점은 "공유 버스(Shared Bus)"라는 점이다. 싱글 코어 시절에는 문제가 덜했지만, 듀얼 코어, 쿼드 코어로 넘어가면서 모든 코어가 시스템 메모리에 접근하기 위해 오직 하나의 FSB 통로를 공유해야 했다. 즉, 코어 1이 메모리를 읽고 있을 때 코어 2는 대기해야 하므로 심각한 병목 현상이 발생했다. 또한 버스의 속도가 메인보드에 부착된 구리선(Trace)의 물리적 길이에 제약을 받아 클럭을 기가헤르츠(GHz) 단위로 높이는 데 실패했다. 이 한계를 극복하기 위해 메모리 제어기(Northbridge)를 없애고 그 기능을 CPU 내부로 가져오는 혁신이 일어났다.

## Ⅲ. 융합 비교 및 다각도 분석

### 시스템 인터커넥트 진화 비교 (FSB vs QPI/UPI)

| 비교 항목 | FSB (Front Side Bus) | QPI / UPI (QuickPath / Ultra Path) |
|:---|:---|:---|
| **통신 방식** | **병렬 (Parallel) 통신**, 공유 버스 | **직렬 (Serial) 점대점 (Point-to-Point)** |
| **메모리 컨트롤러** | 메인보드 칩셋 (노스브리지)에 위치 | **CPU 내부에 통합 (IMC)** |
| **클럭 동기화** | 모든 장치가 동일한 시스템 클럭 공유 | 클럭 신호를 데이터에 임베딩하여 전송 |
| **확장성** | 코어 증가 시 심각한 병목 (확장성 낮음) | NUMA 아키텍처 완벽 지원 (멀티 소켓 확장) |
| **물리적 제약** | 선로 간 간섭(Skew/Crosstalk)으로 고주파수 불가 | 차동 신호(Differential Signaling)로 초고속 달성 |

### 과목 융합 관점
- **운영체제 (OS) 및 분산 처리**: FSB의 한계로 인해 UMA(Uniform Memory Access) 아키텍처가 쇠퇴하고, 메모리를 CPU 소켓마다 분산 배치하는 **NUMA(Non-Uniform Memory Access)** 아키텍처가 표준이 되었다. 이에 따라 OS의 스케줄러(NUMA-aware Scheduler)가 프로세스와 메모리를 동일한 노드에 할당하는 최적화 기술이 필수적으로 요구되었다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **과거 레거시 시스템 마이그레이션**: 오래된 산업용 제어 PC(Core 2 Duo 이전 세대)의 성능 이슈를 분석할 때, CPU 사용률이 낮음에도 애플리케이션 반응이 느리다면 전형적인 FSB 대역폭 포화(Memory Bound) 현상일 확률이 매우 높다. 이는 구조적 한계이므로 시스템 전체의 교체(Re-platforming)가 유일한 해결책이다.
2. **현대 하드웨어 성능 튜닝(오버클럭)**: 과거에는 FSB 베이스 클럭(BCLK)을 높이면 CPU, 메모리, 심지어 PCI 버스 클럭까지 동기화되어 강제로 올라가 시스템이 불안정해졌다. 현대에는 내부 직렬 인터커넥트와 버스 클럭이 분리되어 있으므로, CPU 배수(Multiplier)만 독립적으로 조정하여 안전하게 성능을 튜닝하는 전략을 채택한다.

### 도입/평가 체크리스트
- 레거시 아키텍처 이해도: 최신 시스템에 장애가 발생했을 때, 병목의 원인이 캐시 코히어런시 망(Mesh/Ring)인지 외부 PCIe 스위치인지 구분하기 위해서는 과거 FSB 기반 병목의 본질을 명확히 이해하고 있어야 한다.

### 안티패턴
- **병렬 버스에 대한 잘못된 직관**: "데이터 폭(선)을 64비트에서 128비트로 두 배 늘리면 속도도 두 배가 될 것이다"라는 착각. 초고주파수 대역에서는 선이 많아질수록 0.0001초의 타이밍 스큐(오차)와 전자기 간섭으로 인해 데이터가 엉키게 되므로, 넓은 병렬 통신보다 매우 빠른 좁은 직렬(Serial) 점대점 통신이 훨씬 우수하다.

## Ⅴ. 기대효과 및 결론

### 결론
FSB는 초기 개인용 컴퓨터 대중화 시대를 이끈 핵심 아키텍처였으나, 무어의 법칙에 따른 CPU 클럭의 비약적인 상승을 뒷받침하지 못하고 시스템의 '맥박'을 막는 아킬레스건이 되었다. FSB의 소멸은 컴퓨터 구조가 "공유 자원(Shared Bus)" 모델에서 "분산형 점대점 스위칭(P2P Interconnect)" 모델로 패러다임이 완전히 전환되었음을 상징하는 중요한 역사적 분기점이다.

### 미래 전망
FSB를 대체한 직렬 버스(QPI, PCIe) 또한 미래에는 한계에 봉착할 것이다. 이를 대비하여 전자를 이용한 구리선 통신을 넘어, 칩과 칩을 빛의 속도로 직접 연결하는 **실리콘 포토닉스(Silicon Photonics, 광학 인터커넥트)**와 **칩렛 기반 초고속 패키징(UCIe)** 기술이 차세대 시스템 버스의 역할을 대신할 것이다.
