# RoCE (RDMA over Converged Ethernet)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: RoCE(RDMA over Converged Ethernet)는 전용 인피니밴드(InfiniBand) 스위치 없이도 기존 이더넷 환경에서 RDMA(Remote Direct Memory Access)의 제로 카피(Zero-copy) 및 커널 바이패스(Kernel Bypass) 기능을 그대로 사용할 수 있도록 해주는 고성능 네트워크 프로토콜이다.
> 2. **가치**: CPU의 개입과 컨텍스트 스위칭 오버헤드를 완전히 제거함으로써 마이크로초(µs) 단위의 초저지연(Ultra-low Latency)과 높은 대역폭을 보장하며, 클라우드 데이터센터와 AI 클러스터의 인프라 TCO를 대폭 절감한다.
> 3. **융합**: NVMe-oF(NVMe over Fabrics) 스토리지 시스템, 분산 딥러닝 프레임워크(NCCL), 분산 데이터베이스(In-Memory DB) 등 고속 데이터 전송이 필수적인 최신 아키텍처의 기반 네트워크 계층으로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: RoCE는 분산 시스템 간 데이터를 주고받을 때 송신 측 메모리에서 수신 측 메모리로 데이터를 직접 전송(Direct Transfer)하는 RDMA 기술을 무손실 이더넷(Lossless Ethernet) 패브릭 위로 확장한 네트워크 표준이다. 현재 라우팅이 가능한 RoCEv2가 산업 표준으로 자리 잡았다.
- **💡 비유**: 일반적인 네트워크(TCP/IP)가 물건을 보낼 때 '택배사(운영체제 커널)'를 거쳐 포장, 검수, 재포장 단계를 거친다면, RoCE는 내 책상(메모리)에서 상대방 책상으로 '순간이동 포탈(네트워크 카드 직접 접근)'을 열어 물건을 던져 넣는 것과 같다.
- **등장 배경**: 
  1. **TCP/IP의 한계**: 기존 TCP/IP 스택은 인터럽트 처리, 버퍼 복사(커널↔유저), 컨텍스트 스위칭으로 인해 100Gbps 이상 대역폭 환경에서 CPU 자원의 30~50%를 낭비했다.
  2. **InfiniBand의 고비용**: RDMA를 위해 도입된 인피니밴드는 전용 스위치와 케이블이 필요해 구축 비용이 높고 기존 이더넷 생태계와의 호환성이 떨어졌다.
  3. **이더넷 기반 융합**: 데이터센터 표준인 이더넷 위에서 무손실(Lossless) 전송 기술(DCB: Data Center Bridging)이 발전함에 따라, 이더넷 트래픽에 RDMA를 캡슐화하는 RoCE 프로토콜이 혁신적 대안으로 부상했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 프로토콜 계층 | 비유 |
|:---|:---|:---|:---|:---|
| **RNIC (RDMA NIC)** | 하드웨어 기반 RDMA 처리 | 패킷 캡슐화, 메모리 DMA 직접 수행 | L1/L2 하드웨어 | 순간이동 장치 장착 |
| **QPs (Queue Pairs)** | 비동기 통신 채널 | 송신(SQ) / 수신(RQ) 큐로 구성되어 애플리케이션과 RNIC를 연결 | 전송 계층 (Transport) | 송수신 우편함 |
| **RoCEv2 캡슐화** | 이더넷 라우팅 지원 | InfiniBand 페이로드를 UDP/IP 헤더로 캡슐화 | 네트워크/전송 계층 | 범용 포장지 |
| **PFC (Priority Flow Control)** | 무손실 네트워크 보장 | 혼잡 발생 시 특정 트래픽 클래스에 PAUSE 프레임 전송 | L2 (Data Link) | 교차로 신호등 제어 |
| **Verbs API** | RDMA 애플리케이션 인터페이스 | OS 커널을 거치지 않고 RNIC의 Queue에 작업 요청 | 애플리케이션 계층 | 포탈 조종 리모컨 |

### 커널 바이패스 및 제로 카피 메커니즘

전통적인 TCP/IP 스택과 RoCE 환경에서의 데이터 전송 구조를 비교하면 성능 향상의 근본 원인을 명확히 파악할 수 있다.

```text
 ┌─────────────────────────────────────────────────────────────────┐
 │               [ TCP/IP vs RoCE 아키텍처 통신 비교 ]               │
 ├─────────────────────────────────────────────────────────────────┤
 │                                                                 │
 │   [ 기존 TCP/IP 구조 ]                [ RoCE 아키텍처 (RDMA) ]     │
 │                                                                 │
 │  ┌──────────────────┐               ┌──────────────────┐        │
 │  │ User Application │               │ User Application │        │
 │  │  (Data Buffer)   │               │  (Data Buffer)   │        │
 │  └───────┬──────────┘               └────────┬─────────┘        │
 │          │ (Context Switch & Copy)           │ (Kernel Bypass)  │
 │  ┌───────▼──────────┐               ┌────────▼─────────┐        │
 │  │   OS Kernel      │               │     OS Kernel    │        │
 │  │ (TCP/IP Stack)   │               │ (Control Path만)  │        │
 │  └───────┬──────────┘               └──────────────────┘        │
 │          │ (Copy to NIC)                     │                  │
 │  ┌───────▼──────────┐               ┌────────▼─────────┐        │
 │  │ Standard NIC     │               │   RDMA NIC (RNIC)│        │
 │  │ (Interrupts)     │               │   (QPs & Verbs)  │        │
 │  └───────┬──────────┘               └────────┬─────────┘        │
 │          ▼                                   ▼                  │
 │      Ethernet Network                  Lossless Ethernet        │
 │                                                                 │
 │  * 단점: 데이터가 복사(Copy)되며,     * 장점: 유저 공간에서 RNIC로      │
 │    CPU 오버헤드와 지연 발생.            직접 메모리 접근(Zero-Copy).  │
 └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TCP/IP 아키텍처에서는 사용자 애플리케이션의 데이터가 네트워크로 전송되기 위해 커널 소켓 버퍼로 복사되고, 다시 NIC 버퍼로 복사되는 'Multiple Copies' 과정이 일어난다. 이 과정에서 필연적으로 발생하는 인터럽트 처리와 컨텍스트 스위칭은 마이크로초 단위의 지연을 누적시킨다. 반면, RoCE는 Verbs API를 통해 유저 스페이스에서 직접 RNIC의 큐(Queue)에 전송 명령을 내린다. RNIC는 하드웨어적으로 DMA(Direct Memory Access)를 수행하여 애플리케이션 버퍼에서 데이터를 직접 읽어와(Zero-copy) 네트워크로 내보낸다. 커널은 연결 초기화(Control Path)에만 개입하므로 CPU 사용률이 획기적으로 낮아지고, 지연 시간은 수 마이크로초 수준으로 단축된다.

### RoCEv2 프로토콜 스택

RoCEv1은 L2 이더넷 위에서만 동작하여 라우터 너머(L3)로 통신이 불가능했다. 이를 극복한 RoCEv2는 패킷을 UDP/IP로 캡슐화하여 서브넷 간 라우팅을 지원한다.

```text
 ┌─────────────────────────────────────────────────────────────────┐
 │                     [ RoCEv2 패킷 구조 ]                        │
 ├─────────────┬─────────┬─────────┬──────────────┬────────┬───────┤
 │  Ethernet   │ IP Header│ UDP    │ InfiniBand   │ IB     │ FCS   │
 │  (L2 MAC)   │ (L3)    │ (L4)    │ Base Routing │ Payload│ (CRC) │
 │             │         │ Port 4791 Header       │        │       │
 └─────────────┴─────────┴─────────┴──────────────┴────────┴───────┘
```
**심층 동작 원리:**
① **메모리 등록 (Memory Registration)**: 애플리케이션이 사용할 메모리 영역을 커널에 등록하고 고정(Pinning)하여 스왑 아웃을 방지한다.
② **작업 요청 (Work Request)**: 애플리케이션이 Verbs API를 통해 전송/수신 작업을 Send Queue(SQ)에 넣는다 (Post Send).
③ **비동기 전송 (DMA)**: RNIC가 SQ에서 작업을 폴링하고 호스트 메모리에서 데이터를 DMA로 읽어들여 RoCEv2 패킷으로 캡슐화한다.
④ **무손실 라우팅**: 네트워크 스위치는 PFC(Priority Flow Control)와 ECN(Explicit Congestion Notification)을 활용해 패킷 드롭 없이 목적지까지 라우팅한다.
⑤ **수신 및 완료 (CQ)**: 수신 측 RNIC는 패킷을 까서 목적지 메모리에 직접 DMA Write를 수행하고, 완료 큐(CQ)에 상태를 기록한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 기술 비교: RoCE vs InfiniBand vs iWARP

| 비교 항목 | RoCEv2 | InfiniBand (IB) | iWARP (Internet Wide Area RDMA Protocol) |
|:---|:---|:---|:---|
| **기반 네트워크** | 무손실 이더넷 (Lossless Ethernet) | 전용 InfiniBand 패브릭 | 일반 이더넷 (Lossy 허용) |
| **전송 계층 프로토콜**| UDP/IP 캡슐화 | IB 전송 프로토콜 | TCP/IP 기반 |
| **비용 및 생태계** | 이더넷 인프라 재사용 (중간 비용) | 전용 스위치/케이블 필요 (고비용) | 이더넷 인프라 재사용 (저/중 비용) |
| **혼잡 제어/확장성**| PFC, DCQCN 필수 설정 필요 | 하드웨어 기반 크레딧 흐름 제어 (우수) | TCP 혼잡 제어 상속 (설정 용이) |
| **지연 시간 (Latency)**| 1~3 µs 수준 | 0.5~1 µs 수준 (최상) | 3~10 µs (RoCE 대비 약간 높음) |

### 과목 융합 관점
- **운영체제 (OS)**: 커널의 네트워크 스택을 우회하는 기술이므로, 소켓 프로그래밍(epoll, kqueue)의 문맥 교환 오버헤드를 어떻게 하드웨어로 오프로딩하는지에 대한 아키텍처적 진화를 보여준다.
- **분산 시스템 및 AI**: 분산 딥러닝 훈련 시 GPU 간 파라미터 동기화(All-Reduce 연산 등)를 위해 NVIDIA의 NCCL 라이브러리는 내부적으로 RoCE(또는 IB)를 적극 활용하여 통신 병목을 해소한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **NVMe-oF 스토리지 클러스터 구축**: 플래시 스토리지의 극단적인 성능(NVMe)을 네트워크로 확장하기 위해, SAN 환경을 RoCEv2 기반의 NVMe-oF로 설계한다. 이로써 로컬 NVMe SSD와 거의 동일한 레이턴시로 원격 스토리지에 접근한다.
2. **AI 데이터센터 랙 스케일 패브릭**: 수천 대의 GPU가 장착된 딥러닝 클러스터에서 노드 간 통신 대역폭을 확보하기 위해 400GbE 스위치와 멜라녹스(Mellanox/NVIDIA) ConnectX NIC를 활용하여 RoCEv2 환경을 구축한다. 이때 DCB와 PFC 설정 최적화가 필수적이다.

### 도입 체크리스트
- **기술적**: 이더넷 스위치가 무손실 네트워크(PFC, ECN)를 지원하는지, 스위치 홉(Hop) 수가 늘어날 때 Incast 혼잡(Congestion)이 관리되는지 검증해야 한다.
- **운영적**: 기존 소켓 기반 레거시 애플리케이션은 RDMA를 위해 소스 코드를 Verbs API로 재작성(혹은 미들웨어 사용)해야 하는 마이그레이션 비용을 고려해야 한다.

### 안티패턴
- **PFC 데드락 (PFC Deadlock)**: 무손실 네트워크 보장을 위해 사용하는 PFC가 링 토폴로지나 버그로 인해 상호 무한 대기(Pause) 상태에 빠지는 현상. 이를 방지하기 위해 워치독 매커니즘과 라우팅 루프 방지 설계가 필수적이다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 일반 100GbE (TCP/IP) | 100GbE RoCEv2 환경 | 향상 효과 |
|:---|:---|:---|:---|
| **정량 (Latency)** | 20~50 µs | 1~3 µs | 지연 시간 **최대 90% 이상 감소** |
| **정량 (CPU 사용률)** | ~40% (네트워크 처리에 낭비) | ~5% | 애플리케이션 가용 **CPU 자원 극대화** |
| **정성** | I/O 병목으로 인한 분산 DB 성능 저하 | 무정지, 초고속 데이터 동기화 | 클라우드 인프라의 TCO 절감 및 민첩성 향상 |

### 미래 전망
차세대 800GbE 및 1.6TbE 시대가 도래하면서, 호스트 CPU는 더 이상 네트워크 인터럽트를 감당할 수 없게 된다. RoCE는 SmartNIC 및 DPU(Data Processing Unit)와 결합하여 네트워크, 스토리지, 보안 기능을 완전히 오프로딩하는 클라우드 네이티브 인프라의 절대적인 기반 기술로 진화할 것이다.

### 참고 표준
- **IBTA (InfiniBand Trade Association)**: RoCEv1, RoCEv2 표준 규격 정의
- **IEEE 802.1Qbb (PFC)**, **IEEE 802.1Qaz (ETS)**: 무손실 이더넷을 위한 Data Center Bridging (DCB) 표준
