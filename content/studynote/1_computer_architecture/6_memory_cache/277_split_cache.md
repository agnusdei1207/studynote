# 277. 분리형 캐시 (Split Cache)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 명령어와 데이터를 별도 캐시에 저장
> 2. **가치**: 병렬 접근, 구조적 위험 해결
> 3. **융합**: Harvard vs Von Neumann, L1 구조

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

분리형 캐시(Split Cache)는 **명령어 캐시(I-Cache)와 데이터 캐시(D-Cache)를 분리한 구조**다. Harvard 아키텍처 기반이며, 명령어 인출과 데이터 접근을 동시에 수행할 수 있다.

### 💡 비유: 책장 분리

분리형 캐시는 **공부방 책장 분리**와 같다. 교과서 책장(명령어)과 참고서 책장(데이터)을 따로 둔다. 교과서를 보면서 참고서도 동시에 찾을 수 있다. 한 책장에 몰리지 않는다.

## Ⅱ. 분리형 캐시 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                분리형 캐시 상세                                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Harvard vs Von Neumann】                                                      │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  Von Neumann (통합 메모리):                                          │ │
│  │  • 명령어와 데이터가 같은 메모리                                        │ │
│  │  • 통합 캐시 사용                                                   │ │
│  │  • 단순하지만 병목 가능                                               │ │
│  │                                                               │ │
│  │  Harvard (분리 메모리):                                              │ │
│  │  • 명령어와 데이터가 별도 메모리                                        │ │
│  │  • 분리형 캐시 사용                                                   │ │
│  │  • 병렬 접근 가능                                                   │ │
│  │                                                               │ │
│  │  Modified Harvard (현대 CPU):                                       │ │
│  │  • L1: Split Cache (Harvard)                                        │ │
│  │  • L2/L3: Unified Cache (Von Neumann)                               │ │
│  │  • 두 방식의 장점 결합                                                │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【분리형 캐시와 파이프라인】                                                       │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  파이프라인 단계:                                                      │ │
│  │  IF (Instruction Fetch)    → I-Cache 접근                         │ │
│  │  ID (Instruction Decode)                                       │ │
│  │  EX (Execute)                                                   │ │
│  │  MEM (Memory Access)        → D-Cache 접근                        │ │
│  │  WB (Write Back)                                                │ │
│  │                                                               │ │
│  │  분리형 캐시의 이점:                                                   │ │
│  │  • IF와 MEM이 동시에 다른 캐시 접근                                     │ │
│  │  • 구조적 위험 없음                                                  │ │
│  │  • CPI 향상                                                      │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【I-Cache 특성】                                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  특징:                                                               │ │
│  │  • 읽기 전용 (코드는 실행 중 수정되지 않음)                               │ │
│  │  • 높은 공간 지역성 (순차적 명령어)                                      │ │
│  │  • 분기 예측과 연동                                                   │ │
│  │  • 더티 비트, 쓰기 정책 불필요                                          │ │
│  │                                                               │ │
│  │  최적화:                                                             │ │
│  │  • 코드 배치 최적화                                                  │ │
│  │  • 핫 코드 작게 유지                                                 │ │
│  │  • 함수 인라이닝 균형                                                │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【D-Cache 특성】                                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  특징:                                                               │ │
│  │  • 읽기/쓰기 모두 가능                                                │ │
│  │  • 다양한 접근 패턴                                                  │ │
│  │  • 더티 비트, 쓰기 정책 필요                                          │ │
│  │  • 데이터 지역성 활용                                                 │ │
│  │                                                               │ │
│  │  최적화:                                                             │ │
│  │  • 데이터 배치 최적화                                                 │ │
│  │  • 구조체 필드 배치                                                  │ │
│  │  • 배열 접근 패턴                                                   │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

```

│                                                                     │
│  【컴퓨터 구조】                                                      │

│  • L2/L3: Unified Cache                                             │

│  • 코드 배치: I-Cache 최적화                                           │
│  • 데이터 배치: D-Cache 최적화                                          │
│  • 프로파일링: I/D 캐시 핫스팟                                           │
│  • 인라이닝: I-Cache 크기 고려                                          │
│                                                                     │
│  【운영체제】                                                          │

│  • 컨텍스트 스위치: I/D 캐시 플러시                                        │
│  • Self-modifying code: I-Cache 무효화                               │
│                                                                     │
│  【성능 분석】                                                         │
│  • I-Cache 미스: 명령어 인출 지연                                        │
│  • D-Cache 미스: 데이터 접근 지연                                        │
│  • CPI: I/D 캐시 미스 모두 영향                                         │
│  • 벤치마크: I/D 패턴에 따라 다름                                         │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  현대 CPU 분리형 캐시 사례:                                                        │ │

│  │  CPU             I-Cache      D-Cache      총 L1                      │ │
│  │  ─────────────   ──────────   ──────────   ──────────                  │ │
│  │  Intel Skylake   32KB         32KB         64KB/코어                  │ │
│  │  AMD Zen 3       32KB         32KB         64KB/코어                  │ │

│  │  → 대부분 I/D 비슷한 크기                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• 분리형 캐시: I-Cache + D-Cache 분리
• 구조: Harvard 아키텍처 기반
• 장점: 병렬 접근, 구조적 위험 해결
• I-Cache: 읽기 전용, 순차적, 32~64KB
• D-Cache: 읽기/쓰기, 다양한 패턴, 32~64KB
• 현대 CPU: L1 Split, L2/L3 Unified
• 파이프라인: IF와 MEM 동시 접근 가능
• 최적화: I-Cache는 코드, D-Cache는 데이터
• 측정: perf로 I/D 미스 분리 측정
• Von Neumann vs Harvard: 통합 vs 분리
```
