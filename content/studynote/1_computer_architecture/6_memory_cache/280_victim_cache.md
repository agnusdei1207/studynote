# 280. Victim Cache

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 캐시에서 교체된 블록을 임시 저장하는 작은 캐시
> 2. **가치**: Conflict Miss 감소, 높은 적중률
> 3. **융합**: L1 + Victim, Fully Associative, 작은 크기

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

Victim Cache는 **주 캐시에서 교체된 블록을 저장하는 작은 완전 연관 캐시**다. Conflict Miss를 줄이고, 교체된 데이터에 대한 2차 기회를 제공한다.

### 💡 비유: 임시 보관함

Victim Cache는 **임시 보관함**과 같다. 책상(주 캐시)이 꽉 차서 책을 치워야 할 때, 바로 버리지 않고 임시 보관함에 둔다. 나중에 다시 필요하면 책상에 다시 올린다. 보관함이 꽉 차면 그때 버린다.

│  │  • 위치: L1과 L2 사이                                                │ │

│  │     - L1 교체 대상 블록을 Victim Cache로                              │ │

## Ⅱ. Victim Cache 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                Victim Cache 상세                                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Victim Cache 동작 예시】                                                       │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  L1: Direct Mapped, 4 엔트리 (인덱스 0~3)                            │ │
│  │  Victim: 2 엔트리, Fully Associative                                 │ │
│  │                                                               │ │
│  │  접근: A(0), B(4), C(0), B(4), A(0)                                 │ │
│  │  (괄호는 인덱스, A와 B는 같은 인덱스 충돌)                                │ │
│  │                                                               │ │
│  │  1. A(0): L1 Miss, L2 → L1[0]=A, Victim=[]                         │ │
│  │  2. B(4): L1 Miss, A→Victim, L1[0]=B, Victim=[A]                   │ │
│  │  3. C(0): L1 Miss, B→Victim, L1[0]=C, Victim=[A,B]                 │ │
│  │  4. B(4): L1 Miss, C→Victim, Victim Hit!                           │ │
│  │           L1[0]=B, Victim=[A,C]                                    │ │
│  │  5. A(0): L1 Miss, B→Victim, Victim Hit!                           │ │
│  │           L1[0]=A, Victim=[B,C]                                    │ │
│  │                                                               │ │
│  │  → Victim Cache로 A, B를 여전히 찾을 수 있음                           │ │
│  │  → Conflict Miss가 Victim Hit로 해결                                 │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【Victim Cache 효과】                                                             │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  워크로드: 충돌이 많은 패턴                                              │ │
│  │  L1: Direct Mapped, 64KB                                           │ │
│  │                                                               │ │
│  │  Victim Cache 크기   L1 적중률    전체 적중률    개선                    │ │
│  │  ──────────────      ──────      ──────        ──────                    │ │
│  │  없음                 85%         85%           기준                      │ │
│  │  4 엔트리             85%         92%           +7%                     │ │
│  │  8 엔트리             85%         95%           +10%                    │ │
│  │  16 엔트리            85%         97%           +12%                    │ │
│  │                                                               │ │
│  │  → 작은 Victim Cache로 큰 적중률 향상                                  │ │
│  │  → Conflict Miss가 많을수록 효과 큼                                    │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【Victim Cache vs L2 Cache】                                                      │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  특성           Victim Cache      L2 Cache                       │ │
│  │  ──────          ──────────         ──────                         │ │
│  │  크기            4~16 엔트리         256KB~1MB                      │ │
│  │  구조            완전 연관            집합 연관                       │ │
│  │  내용            L1 교체만            모든 데이터                     │ │
│  │  지연            1~2 클럭             5~10 클럭                      │ │
│  │  목적            Conflict Miss       Capacity Miss                  │ │
│  │                                                               │ │
│  │  → Victim는 Conflict Miss 특화                                       │ │
│  │  → L2는 일반적 미스 처리                                              │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. Victim Cache × 타 과목 융합

### 과목 융합 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                Victim Cache × 타 과목 융합                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【컴퓨터 구조】                                                      │
│  • L1 보조: Victim Cache로 L1 성능 향상                                │
│  • Direct Mapped: Victim과 조합으로 Conflict 감소                      │
│  • 파이프라인: Victim Hit으로 스톨 감소                                  │
│  • 전력: 작은 크기로 전력 오버헤드 적음                                   │
│                                                                     │
│  【캐시 계층】                                                        │
│  • L1 → Victim → L2 → L3 → DRAM                                     │
│  • 계층 간 미스 체인 완화                                               │
│  • AMAT 감소                                                         │
│  • 전체 성능 향상                                                      │
│                                                                     │
│  【워크로드】                                                          │
│  • 충돌 많음: Victim Cache 효과 큼                                     │
│  • 순차적: 효과 적음                                                   │
│  • 랜덤: 효과 중간                                                    │
│  • 스트라이드: 패턴에 따라 다름                                          │
│                                                                     │
│  【역사】                                                              │
│  • 1990년대: DEC Alpha, MIPS에서 사용                                  │
│  • 현대: 일부 ARM, low-power CPU에서 사용                               │
│  • 대안: 집합 연관 캐시가 표준화되면서 사용 감소                             │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  Victim Cache 사용 CPU:                                                        │ │
│  │  ──────────────────────                                              │ │
│  │  CPU             Victim Cache    특징                                 │ │
│  │  ─────────────   ──────────      ────                                   │ │
│  │  ARM Cortex-A53  있음            저전력                                  │ │
│  │  MIPS R10000     있음            고성능                                  │ │
│  │  DEC Alpha 21164 있음            연구용                                  │ │
│  │  Intel/AMD       거의 없음        큰 집합 연관 사용                        │ │
│  │  → 현대는 큰 집합 연관 캐시가 Victim 대체                                 │ │
│  │  → 하지만 임베디드/저전력에서는 여전히 유용                                 │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### Victim Cache 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                Victim Cache 분석                                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Victim Cache 구현 고려사항】                                                     │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 크기 결정:                                                        │ │
│  │     • 4~16 엔트리가 일반적                                             │ │
│  │     • 클수록 좋지만 수확 체감                                          │ │
│  │     • 캐시 라인 크기와 동일하게                                         │ │
│  │                                                               │ │
│  │  2. 구조 선택:                                                        │ │
│  │     • 완전 연관 (Fully Associative)                                   │ │
│  │     • 작은 크기라 구현 가능                                            │ │
│  │     • CAM으로 병렬 검색                                               │ │
│  │                                                               │ │
│  │  3. 교체 정책:                                                        │ │
│  │     • FIFO가 단순                                                    │ │
│  │     • LRU가 성능 좋음                                                 │ │
│  │     • Random도 가능                                                  │ │
│  │                                                               │ │
│  │  4. 배치:                                                            │ │
│  │     • L1 바로 옆                                                     │ │
│  │     • 빠른 접근 필요                                                  │ │
│  │     • 1~2 클럭 내 응답                                                │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【Victim Cache 효과 측정】                                                        │
│  ──────────────────                                                  │
│  # Victim Cache가 있는 시스템에서 측정                                  │
│  $ perf stat -e cache-references,cache-misses ./prog               │
│                                                                     │
│  # Victim Hit Rate이 별도로 측정되지 않는 경우                          │
│  # 시뮬레이터 사용 필요                                                │
│                                                                     │
│  【Victim Cache 대안】                                                              │
│  ──────────────────                                                  │
│  1. 더 큰 집합 연관도: 8-way, 16-way                                   │
│  2. 큰 L1 캐시: Conflict Miss 감소                                    │
│  3. 페이지 컬러링: OS 레벨 충돌 회피                                     │
│  4. 데이터 재배치: 컴파일러 레벨                                        │
│                                                                     │
│  【Victim Cache 최적화 팁】                                                         │
│  ──────────────────                                                  │
│  1. 충돌 패턴 분석: Victim 효과가 큰지 확인                              │
│  2. 크기 최적화: 4~16 엔트리 실험                                       │
│  3. 교체 정책: LRU 권장                                               │
│  4. 계층 통합: L1 + Victim을 하나로 고려                                │
│  5. 대안 비교: 집합 연관 확대와 비교                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• Victim Cache: L1 교체 블록을 저장하는 작은 캐시
• 크기: 4~16 엔트리 (매우 작음)
• 구조: 완전 연관 (Fully Associative)
• 목적: Conflict Miss 감소
• 동작: L1 미스 → Victim Hit 시 복구
• 효과: 작은 크기로 큰 적중률 향상
• 지연: 1~2 클럭 (L2보다 빠름)
• 위치: L1과 L2 사이
• 현대: 큰 집합 연관으로 대체되는 추세
• 사용처: 임베디드, 저전력, 일부 ARM
```
