# 412. 트랜잭셔널 메모리 (Transactional Memory)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메모리 연산을 원자적 트랜잭션으로 수행하는 동기화 패러다임
> 2. **가치**: 락 없는 동기화, 프로그래밍 단순화
> 3. **융합**: HTM, STM, 낙관적 동시성, 충돌 감지

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

트랜잭셔널 메모리(Transactional Memory, TM)는 **메모리 읽기/쓰기 연산을 하나의 원자적 트랜잭션으로 수행하고, 충돌 시 자동으로 롤백하여 재시도하는 동기화 메커니즘**이다. 락 기반 동기화의 문제점을 해결한다.

### 💡 비유: 데이터베이스 트랜잭션

트랜잭셔널 메모리는 **데이터베이스 트랜잭션**과 같다. 은행 이체다. A에서 출금, B에 입금. 둘 다 성공하거나 둘 다 실패. 중간에 오류? 롤백. 처음처럼. TM이 똑같다. 메모리 연산들, 원자적 실행, 충돌 시 롤백.

### 트랜잭셔널 메모리 개요

```
┌─────────────────────────────────────────────────────────────────────┐
│                트랜잭셔널 메모리 개요                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【TM 동작 원리】                                                               │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  transaction {                                                │ │
│  │      // 읽기/쓰기 연산들                                             │ │
│  │      x = data.read();                                         │ │
│  │      data.write(x + 1);                                       │ │
│  │  }                                                            │ │
│  │                                                               │ │
│  │  동작 순서:                                                                  │ │
│  │  1. 트랜잭션 시작                                                          │ │
│  │  2. 읽기/쓰기 로깅 (undo log)                                             │ │
│  │  3. 충돌 감지 (다른 트랜잭션과)                                            │ │
│  │  4. 충돌 없음 → 커밋 (변경 확정)                                          │ │
│  │  5. 충돌 있음 → 롤백 (변경 취소) 후 재시도                                 │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【락 vs TM】                                                                   │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  락 기반:                                                                   │ │
│  │  lock(A); lock(B);                                             │ │
│  │  // 작업                                                         │ │
│  │  unlock(B); unlock(A);                                          │ │
│  │  → 교착상태 위험, 순서 중요                                            │ │
│  │                                                               │ │
│  │  트랜잭셔널:                                                                  │ │
│  │  atomic {                                                                  │ │
│  │      // A와 B 작업                                                        │ │
│  │  }                                                                         │ │
│  │  → 교착상태 없음, 자동 롤백                                             │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【HTM vs STM】                                                                 │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  HTM (Hardware TM):                                                       │ │
│  │  • 하드웨어 지원 (Intel TSX, IBM POWER)                                 │ │
│  │  • 빠름                                                                  │ │
│  │  • 용량 제한                                                             │ │
│  │                                                               │ │
│  │  STM (Software TM):                                                        │ │
│  │  • 소프트웨어 구현                                                       │ │
│  │  • 유연함                                                                │ │
│  │  • 느림                                                                  │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 트랜잭셔널 메모리 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                트랜잭셔널 메모리 상세                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Intel TSX (Transactional Synchronization Extensions)】                         │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  RTM (Restricted Transactional Memory):                                   │ │
│  │  ┌────────────────────────────────────────────────────┐        │ │
│  │  │  if (_xbegin() == _XBEGIN_STARTED) {              │        │ │
│  │  │      // 트랜잭션 코드                               │        │ │
│  │  │      _xend();  // 커밋                              │        │ │
│  │  │  } else {                                          │        │ │
│  │  │      // 폴백 (락 기반)                               │        │ │
│  │  │  }                                                 │        │ │
│  │  └────────────────────────────────────────────────────┘        │ │
│  │                                                               │ │
│  │  중단 원인:                                                                  │ │
│  │  • 캐시 용량 초과                                                        │ │
│  │  • 다른 코어와 충돌                                                      │ │
│  │  • 인터럽트, 컨텍스트 전환                                               │ │
│  │  • 지원되지 않는 명령어                                                  │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【충돌 감지】                                                                   │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  낙관적 동시성:                                                                  │
│  │  • 일단 실행, 충돌 시 롤백                                              │ │
│  │  • 읽은 셋, 쓴 셋 추적                                                  │ │
│  │                                                               │ │
│  │  충돌 정의:                                                                    │
│  │  • 두 트랜잭션이 같은 위치에 쓰기                                       │ │
│  │  • 하나는 읽고 다른 하나는 쓰기                                          │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【장단점】                                                                      │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  장점:                                                                  │ │
│  │  • 교착상태 없음: 롤백으로 해결                                         │ │
│  │  • 단순성: atomic 블록으로 캡슐화                                        │ │
│  │  • 확장성: 낮은 경합 시 좋음                                             │ │
│  │                                                               │ │
│  │  단점:                                                                  │ │
│  │  • 용량 제한: HTM은 L1 캐시 크기                                         │ │
│  │  • 오버헤드: 롤백 비용                                                  │ │
│  │  • 라이브: 무한 롤백 가능 (기아)                                         │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### 트랜잭셔널 메모리 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                트랜잭셔널 메모리 분석                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Intel TSX 사용】                                                              │)
│  ──────────────────                                                  │
│  #include <immintrin.h>                                             │
│                                                                     │
│  // RTM (Restricted Transactional Memory)                           │
│  if (_xbegin() == _XBEGIN_STARTED) {                                │
│      // 트랜잭션 코드                                                 │
│      // 락 확인 (충돌 감지)                                           │
│      if (lock_is_held) _xabort(0xff);                               │
│      // 작업                                                         │
│      _xend();  // 커밋                                               │
│  } else {                                                           │
│      // 폴백: 일반 락                                                 │
│      lock();                                                        │
│      // 작업                                                         │
│      unlock();                                                      │
│  }                                                                  │
│                                                                     │
│  【TSX 지원 확인】                                                              │)
│  ──────────────────                                                  │
│  $ cat /proc/cpuinfo | grep rtm                                     │
│  # 또는                                                              │
│  $ lscpu | grep rtm                                                 │
│                                                                     │
│  【성능 측정】                                                                   │)
│  ──────────────────                                                  │
│  $ perf stat -e rtm_retired.start,rtm_retired.commit ./program     │
│  # 트랜잭션 시작/커밋 수                                               │
│                                                                     │
│  【TM 최적화 팁】                                                              │)
│  ──────────────────                                                  │
│  1. 작은 트랜잭션: 캐시 용량 내                                        │
│  2. 폴백: 항상 락 기반 대비책                                          │
│  3. 경합: 낮은 경합에 유리                                             │
│  4. 측정: 커밋율 모니터링                                              │
│  5. I/O: 트랜잭션 내 피하기                                           │
│  6. 재시도: 백오프 전략                                                │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• TM: 메모리 연산을 원자적 트랜잭션으로
• 장점: 교착상태 없음, 단순성
• 단점: 용량 제한, 롤백 오버헤드
• HTM: 하드웨어 TM (Intel TSX)
• STM: 소프트웨어 TM
• 충돌: 읽기/쓰기 셋 겹침
• 롤백: 자동 재시도
• 폴백: 락 기반 대비책 필수
• C++: atomic 확장
• 활용: 낮은 경합, 짧은 트랜잭션
```
