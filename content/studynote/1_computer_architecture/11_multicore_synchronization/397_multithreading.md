# 멀티스레딩 (Multithreading)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 멀티스레딩(Multithreading)은 하나의 무거운 프로그램(프로세스)을 **여러 개의 가볍고 독립적인 실행 흐름(스레드, Thread)으로 쪼개어, 메모리를 공유하며 동시에 병렬로 실행시키는 소프트웨어/하드웨어적 기법**이다.
> 2. **가치**: 스레드들은 프로세스의 힙(Heap)과 데이터 영역을 그대로 공유하므로 생성 및 통신 비용(Overhead)이 극도로 낮아, 멀티코어의 성능을 100% 쥐어짜 내어 **응답성(Responsiveness)과 처리량(Throughput)을 폭발적으로 높인다.**
> 3. **융합**: 소프트웨어 레벨의 멀티스레딩(OS 통제)이 한계에 부딪히자, 하드웨어 자체가 코어 하나에 여러 스레드를 억지로 쑤셔 넣는 **동시 멀티스레딩(SMT, 하이퍼스레딩)**과 융합되며 궁극의 자원 효율성을 완성했다.

---

## Ⅰ. 멀티스레딩의 개요

### 1. 정의
- 하나의 프로세스(실행 중인 프로그램) 내에서 여러 개의 실행 단위(스레드)를 생성하여, 다중 코어 또는 단일 코어의 시분할(Time-slicing)을 통해 여러 작업을 동시에(Concurrent or Parallel) 처리하는 아키텍처 및 프로그래밍 모델이다.

### 2. 등장 배경: '프로세스는 너무 무겁고 이기적이다'
- 1980년대 멀티태스킹은 "워드도 켜고, 음악도 튼다"는 **멀티프로세스** 방식이었다. 하지만 웹 서버가 수만 명의 접속을 받을 때, 접속자마다 1개씩 수만 개의 프로세스(복제 인간)를 찍어내면 메모리(RAM)가 펑 터져버렸다(C10K 문제). 프로세스는 각자 독립된 성(메모리)을 가져서 너무 무거웠다. 
- 해답은 간단했다. "하나의 성(프로세스) 안에 일꾼(스레드)만 여러 명 고용해서, 주방(공유 메모리)을 같이 쓰게 만들자!" 가볍고 빠른 경량 프로세스, 스레드의 시대가 열렸다.

### 3. 💡 비유: '분업화된 햄버거 주방'
- **멀티프로세스**는 **'햄버거 가게를 5개 차려서 각 가게마다 사장님이 혼자 햄버거를 만드는 것'**이다. 가게 월세(메모리)가 엄청나게 깨진다.
- **멀티스레딩**은 **'커다란 햄버거 가게 1개(프로세스)의 주방 안에서 5명의 요리사(스레드)가 도마(공유 메모리)를 같이 쓰며 빵 굽고, 고기 굽고, 포장을 나눠서 하는 것'**이다. 월세는 1번만 내면 되고 속도는 5배 빠르다.

- **📢 섹션 요약 비유**: 각자 무거운 자기 집을 이고 다니던 달팽이(프로세스) 무리에서, 넓은 거실 하나를 공유하며 잽싸게 움직이는 기숙사 룸메이트(스레드)들로 생태계가 진화한 것입니다.

## Ⅱ. 메커니즘 및 자원 공유 (Deep Dive)

스레드가 가벼운 이유는 '가진 게 없어서'다. 철저한 공유 경제를 실천한다.

### 1. 스레드가 공유하는 것 (같이 쓰는 재산)
- **코드(Code)**: 프로그램 명령어 자체
- **데이터(Data)**: 전역 변수(Global Variable), 정적 변수
- **힙(Heap)**: `malloc`이나 `new`로 동적 할당받은 거대한 메모리
- *결과*: 스레드 A가 전역 변수를 `1`로 바꾸면, 스레드 B가 즉시 `1`로 읽을 수 있어 통신이 빛의 속도다.

### 2. 스레드가 독립적으로 가지는 것 (나만의 프라이버시)
- **프로그램 카운터 (PC)**: 자기가 코드의 어디를 읽고 있는지 기억하는 책갈피.
- **레지스터 세트 (Registers)**: 연산 도중의 임시 숫자 보관함.
- **스택 (Stack)**: 자신만의 지역 변수(함수 안에서만 쓰는 변수)와 되돌아갈 함수 주소.
- *이유*: 이것마저 공유하면 5명이 똑같은 생각만 해야 하므로, 각자 딴생각(독립된 작업)을 하려면 최소한 뇌의 일부분(PC, 레지스터, 스택)은 무조건 따로 찢어 가져야 한다.

### 3. 문맥 교환 (Context Switch)의 압도적 비용 차이 (ASCII)
```text
  [ 멀티프로세스 교체 ] (무거운 이사)
  프로세스 A 정지 -> A의 페이지 테이블(메모리 장부) 백업 -> 캐시(TLB) 모조리 텅텅 비움 (Flush) 
  -> B의 페이지 테이블 덮어쓰기 -> B 실행 
  (비용: 수천~수만 클럭 낭비, 끔찍한 오버헤드)

  [ 멀티스레드 교체 ] (가벼운 교대)
  스레드 A 정지 -> A의 레지스터(PC, 상태)만 쏙 빼서 메모리에 저장 -> B의 레지스터 쏙 집어넣음 -> B 실행
  (비용: 수백 클럭 이내. 메모리 장부나 캐시는 그대로 재사용!)
```

- **📢 섹션 요약 비유**: 프로세스 교체가 집을 통째로 이사하는 것이라면, 스레드 교체는 같은 집에서 형이 컴퓨터 게임하다가 동생한테 의자만 살짝 비켜주는 초고속 자리교대입니다.

## Ⅲ. 하드웨어 멀티스레딩의 3가지 진화

소프트웨어가 스레드를 쪼개주면, 하드웨어(CPU)는 이를 어떻게 삼킬까?

| 종류 | 하드웨어 동작 방식 | 비유 |
|:---|:---|:---|
| **거친 멀티스레딩 (Coarse-grained)** | 스레드 A가 캐시 미스(긴 멈춤)로 뻗었을 때만 큼직하게 스레드 B로 스위칭 | 운전자가 졸음쉼터에 들를 때만 교대 운전 |
| **세밀한 멀티스레딩 (Fine-grained)** | 1클럭은 스레드 A, 다음 1클럭은 스레드 B 식으로 매 틱마다 촘촘하게 섞어서 스위칭 | 농구공 두 개를 양손으로 0.5초 단위로 번갈아 튀기는 묘기 |
| **동시 멀티스레딩 (SMT, 하이퍼스레딩)** | **단일 클럭 안에서 스레드 A와 B의 명령어를 동시에 ALU에 던져버림 (궁극의 병렬)** | 양손잡이가 왼손은 수학 풀고 오른손은 영어 푸는 미친 동시 작업 |

- **📢 섹션 요약 비유**: 놀고 있는 CPU 부품(ALU)을 단 0.001초도 가만히 두지 않기 위해, 스레드라는 땔감을 쉬지 않고 아궁이에 쑤셔 넣는 기계적 진화의 3단계입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: 동기화 지옥과 데드락 (Deadlock)
- **상황**: 멀티스레드는 힙(Heap)을 공유한다. 스레드 1만 개가 동시에 쇼핑몰 재고(공유 변수)를 `-1` 깎는 코드를 실행했다.
- **판단**: 스레드가 너무 가볍고 자유롭다 보니 참사가 일어난다. 두 스레드가 동시에 `재고 1개`를 읽고 각자 `-1`을 한 뒤 덮어쓰면, 재고는 0이 되어야 하는데 1명이 산 것으로 씹히는 **레이스 컨디션(Race Condition)**이 터진다. 개발자들은 이를 막기 위해 화장실 문을 걸어 잠그는 **뮤텍스(Mutex)**나 락(Lock)을 쓴다. 하지만 락을 너무 떡칠하면 스레드 A와 B가 서로 상대방이 화장실에서 나오기만을 평생 기다리며 멈춰버리는 **데드락(교착 상태)**의 늪에 빠져 서버가 재부팅 전까지 뻗어버리는 최악의 사태를 맞이한다.

### 2. 기술사적 인사이트: GIL (Global Interpreter Lock)의 파이썬의 비극
- **인사이트**: 파이썬(Python)은 언어 자체에 엄청난 족쇄가 있다. 파이썬 메모리 관리자가 스레드 꼬임(충돌)을 막기 위해, 코어 8개짜리 컴퓨터에서도 **파이썬 코드는 무조건 1번에 1개의 스레드만 락(GIL)을 쥐고 실행되게끔** 막아버렸다. 
- 결과적으로 멀티스레드 코드를 아무리 기가 막히게 짜도 결국 병렬이 아닌 순차(Context Switching만 할 뿐)로 돌아가서 C++이나 Java보다 10배 이상 느리다. (그래서 파이썬으로 딥러닝을 돌릴 땐, 파이썬이 안 돌고 파이썬이 호출한 밑바닥 C/C++ 라이브러리(NumPy)가 GIL 밖에서 멀티스레드로 미친 듯이 연산을 대신해 주는 것이다.)

- **📢 섹션 요약 비유**: 넓은 8차선 고속도로(멀티코어)를 깔아놨는데, 톨게이트 직원(GIL)이 바보라서 1차선만 열어두고 차를 한 대씩 보내는 바람에 도로 전체가 마비되는 억울한 현상입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **응답성 극대화**: 웹 브라우저가 파일 다운로드(스레드 A)를 받으면서도, 동시에 내 마우스 스크롤(스레드 B)에 버벅임 없이 즉각 반응할 수 있는 쾌적한 UX 환경 창조.
- **서버 인프라 경제성**: 1대의 물리 서버가 수만 명의 접속자를 수만 개의 깃털 같은 스레드로 받아내어, 기업의 클라우드 서버 호스팅 유지 비용을 수십 분의 일로 깎아버림.

### 2. 미래 전망
스레드도 가볍지만, 메모리가 너무 쪼개지면서 컨텍스트 스위칭 비용마저 아까운 극한의 시대가 왔다. 최근 백엔드(Go언어의 고루틴, Node.js, Kotlin의 코루틴) 생태계는 OS가 통제하는 진짜 스레드를 포기하고, 애플리케이션 자체가 수백만 개의 가짜 스레드(경량 스레드, User-level Thread)를 만들어 1개의 진짜 스레드 위에서 휙휙 스위칭하며 돌리는 **코루틴(Coroutine) / 비동기(Asynchronous) 프로그래밍**으로 패러다임이 전면 교체되었다. 멀티스레딩은 하드웨어를 넘어 소프트웨어의 궁극의 쪼개기 마술로 영원히 진화 중이다.

- **📢 섹션 요약 비유**: 칼로 무를 10등분(멀티스레드) 하던 시대를 넘어, 무를 아예 믹서기에 갈아 수백만 개의 즙(코루틴)으로 만들어 빨대로 순식간에 빨아먹는 극강의 유동성 시대로 넘어왔습니다.
