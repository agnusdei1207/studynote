# 디렉토리 기반 프로토콜 (Directory-based Protocol)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 디렉토리 기반 프로토콜은 "어떤 코어의 캐시가 어떤 메모리 블록을 가지고 있는지, 그 상태가 무엇인지"를 **중앙의 거대한 장부(Directory)에 기록해두고 일관성을 관리하는 아키텍처**다.
> 2. **가치**: 스누핑(Snooping) 프로토콜의 고질병인 방송(Broadcast) 버스 트래픽 폭주 문제를 원천 차단하여, 수십~수백 개의 코어가 장착된 대규모 서버 시스템에서 **무한에 가까운 확장성(Scalability)**을 제공한다.
> 3. **융합**: 거대한 메모리를 노드별로 쪼개어 갖는 **NUMA (Non-Uniform Memory Access)** 아키텍처와 필연적으로 결합되며, 장부 기록(Directory)이라는 소프트웨어적 아이디어를 하드웨어 SRAM으로 구현한 첨단 분산 처리의 뼈대다.

+++

## Ⅰ. 디렉토리 프로토콜의 개요

### 1. 정의
- 캐시 일관성을 유지하기 위해 공용 버스에 소리를 지르는(Broadcast) 대신, 메인 메모리 블록의 상태와 해당 블록을 복사해 간 코어들의 리스트(Presence Vector)를 저장하는 중앙/분산 '디렉토리(장부)'를 두고 점대점(Point-to-Point) 통신으로 제어하는 방식이다.

### 2. 등장 배경: '확성기의 한계와 중앙 통제의 필요성'
- 4코어, 8코어 시절에는 스누핑(확성기 방송)이 최고였다. 하지만 코어가 64개, 128개로 늘어나자 128명이 동시에 버스에 대고 "나 이거 수정한다!"라고 소리를 질러댔고, 데이터가 지나가야 할 버스는 '수정 알림' 신호로 꽉 막혀 시스템이 기절해 버렸다. 아키텍트들은 "소리치지 말고, 누가 뭘 가져갔는지 **관리 대장(Directory)**에 적어두고, 수정할 때 그 관리 대장한테만 조용히 가서 허락을 받자"는 관료제 시스템을 도입했다.

### 3. 💡 비유: '도서관 대출 장부 시스템'
- 디렉토리 프로토콜은 **'도서관 사서의 대출 기록 장부'**와 완벽히 일치한다.
- A학생이 책을 빌려 가고 싶으면, 온 학교 스피커에 대고 "이 책 가진 사람 있어?!"라고 소리칠 필요(스누핑)가 없다. 사서(Directory)에게 가서 물어보면, 사서는 장부를 보고 "B학생이 빌려 갔네"라고 확인한 뒤, 오직 B학생에게만 개별적으로 연락(Point-to-Point)해서 책을 반납하라고 지시한다.

- **📢 섹션 요약 비유**: 시장 바닥의 시끄러운 호객 행위를 금지하고, 모든 거래를 중앙 전산망 장부에 기록하여 타겟 마케팅만 수행하는 극강의 질서와 통제 시스템입니다.

## Ⅱ. 메커니즘 및 디렉토리 구조 (Deep Dive)

디렉토리는 메모리 컨트롤러 옆에 찰떡같이 붙어 있는 똑똑한 캐시다.

### 1. 디렉토리 엔트리(Entry)의 내부 구조
메모리의 각 캐시 라인(블록)마다 디렉토리 장부에 한 줄씩 기록이 존재한다.
| 필드 | 비트 수 | 역할 |
|:---:|:---:|:---|
| **상태 (State)** | 2 bits | 이 메모리 블록이 어떤 상태인가? (예: Uncached, Shared, Modified) |
| **공유자 목록 (Presence Vector)** | 코어 개수 (예: 64bits) | 만약 64코어 시스템이라면, **누가 이 블록을 자기 캐시에 복사해 갔는지** 비트로 표시. (예: 1번, 5번 코어가 가져갔으면 01000100... 로 1이 켜짐) |

### 2. 동작 프로세스 시각화 (ASCII)
```text
    [ 상황: 코어 0이 변수 X를 '수정(Write)' 하려 함 ]
    
    1. 코어 0 ──요청──▶ [ Directory (장부) ]
    
    2. 디렉토리가 장부 조회:
       "변수 X는 현재 Shared 상태이고, 코어 2와 코어 3이 복사해 갔군."
       (Presence Vector: 0011000...)
       
    3. 디렉토리의 타겟 무효화(Invalidate) 전송:
       [ Directory ] ──개별 귓속말──▶ [ 코어 2 ] "너 X 지워!"
       [ Directory ] ──개별 귓속말──▶ [ 코어 3 ] "너도 X 지워!"
       
    4. 코어 2, 3이 다 지웠다고 응답(Ack)하면, 
       디렉토리는 코어 0에게 "이제 너 혼자 써(Modified)!"라고 허락함.
```

### 3. 심층 분석: 디렉토리의 분산 (Distributed Directory)
- 코어 128개짜리 서버에서 거대한 디렉토리 1개만 두면, 모든 코어가 이 장부를 보려고 줄을 서는 또 다른 병목이 생긴다. 이를 막기 위해 디렉토리를 조각내서 **각 메모리 뱅크(또는 각 코어의 L3 캐시 슬라이스) 옆에 분산시켜 배치**한다. 자기 구역의 장부는 자기가 알아서 관리하는 NUMA 형태의 분산 통제 아키텍처다.

- **📢 섹션 요약 비유**: 중앙 구청 1개에서 100만 명의 등본을 떼주는 게 아니라, 50개의 동주민센터로 장부를 쪼개어 분산 처리하는 완벽한 행정망입니다.

## Ⅲ. 디렉토리 프로토콜 vs 스누핑 프로토콜 비교

| 비교 항목 | 스누핑 프로토콜 (Snooping) | 디렉토리 프로토콜 (Directory-based) |
|:---|:---:|:---:|
| **일관성 관리 주체**| 모든 코어의 캐시 컨트롤러들이 각자 | **중앙/분산된 디렉토리 하드웨어** |
| **통신 방식** | **브로드캐스트 (모두에게 방송)** | **유니캐스트 / 멀티캐스트 (타겟에게만)** |
| **장점** | 지연 시간(Latency)이 짧고 매우 빠름 | **트래픽이 적어 확장이 무한대로 가능** |
| **단점** | 코어 수가 많아지면 버스가 뻗어버림 | 장부를 읽고 쓰는 추가 지연(Overhead) 발생 및 디렉토리 SRAM 용량 증가 |
| **메인 타겟** | 2 ~ 16 코어 (데스크톱, 모바일) | **64 ~ 256+ 코어 (하이엔드 서버, 슈퍼컴퓨터)** |

- **📢 섹션 요약 비유**: 단톡방에서 떠드는 스누핑은 소수일 땐 재밌지만 100명이 넘어가면 지옥이 됩니다. 디렉토리는 꼭 필요한 사람에게만 개인톡을 쏴주는 매너 있는 시스템입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: NUMA (Non-Uniform Memory Access) 아키텍처의 심장
- **상황**: 2 소켓, 4 소켓 랙마이크 서버(예: 128코어 AMD EPYC 듀얼) 보드에는 거대한 CPU 칩이 여러 개 박혀 있고 메모리도 각 CPU 옆에 따로 달려 있다.
- **판단**: CPU 1번이 CPU 2번 옆에 있는 메모리 값을 읽어와 캐시에 올리고 수정하려 할 때, 스누핑으로 물리적으로 분리된 소켓 사이의 인터커넥트(UPI, Infinity Fabric)에 방송을 쏘면 엄청난 통신 폭풍이 인다. 이런 **NUMA 구조에서 멀티코어 캐시 일관성을 유지할 수 있는 유일한 마법**이 바로 각 메모리 노드마다 장부를 쥐고 있는 분산 디렉토리 프로토콜이다. 디렉토리가 없었다면 오늘날의 거대 클라우드 데이터센터 인프라는 존재조차 불가능했다.

### 2. 실무 시나리오: 디렉토리 용량 폭발(Directory Overhead) 제어
- **인사이트**: 1,024개의 코어가 있다면 공유자 목록(Presence Vector)도 1,024비트(128바이트)가 되어야 한다. 캐시 라인이 64바이트인데 장부 기록용으로 128바이트를 쓰면 배보다 배꼽이 커진다. 아키텍트들은 비트를 압축하는 **Coarse-grained Vector (코어 4개를 1비트로 묶기)**나 제한된 포인터(Limited Pointer) 방식을 써서 디렉토리의 하드웨어 비용(SRAM 면적)을 줄이는 극한의 설계를 단행한다. 트레이드오프의 예술이다.

- **📢 섹션 요약 비유**: 1,000명의 회원 이름을 다 적으려니 장부가 너무 두꺼워져서, "1동 사람들", "2동 사람들"로 뭉뚱그려 적어 장부 두께를 줄이는 현명한 꼼수입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **확장성(Scalability)의 절대적 해방**: 코어 개수가 수백 개, 수천 개로 늘어나더라도 상호 통신 오버헤드를 선형적(Linear) 수준으로 억제하여 멀티코어 성능의 우상향 그래프 보장.
- **전력 절감**: 브로드캐스트를 쏴서 모든 코어의 캐시 태그 비교기를 깨우는 쓸데없는 전기 낭비를 막고, 조용하게 타겟 칩만 깨워 전성비 극대화.

### 2. 미래 전망
거대 AI 모델 학습을 위해 CPU와 여러 대의 GPU가 얽혀서 메모리를 공유하는 시대가 왔다. NVIDIA의 NVLink나 차세대 CXL(Compute Express Link) 프로토콜은 기기 간의 메모리 캐시 일관성을 유지하기 위해 철저하게 **디렉토리 기반의 스눕 필터링 아키텍처**를 표준으로 삼고 있다. 디렉토리 프로토콜은 단일 실리콘 칩을 넘어, 거대한 데이터센터 랙(Rack) 전체를 하나의 거대한 유기체적 CPU로 묶어내는 마스터키로 군림할 것이다.

- **📢 섹션 요약 비유**: 구멍가게의 외상 장부(스누핑)에서 시작된 인류의 컴퓨팅이, 전 세계 은행을 실시간으로 연결하는 글로벌 금융 블록체인 원장(디렉토리 프로토콜)으로 진화한 위대한 서사시입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **캐시 일관성 (Cache Coherence)**: 디렉토리 프로토콜이 궁극적으로 달성하고자 하는 숭고한 목표.
- **스누핑 프로토콜 (Snooping)**: 소규모에서는 제왕이었으나 대규모에서 뻗어버린 1세대 선배 기술.
- **NUMA (Non-Uniform Memory Access)**: 디렉토리 프로토콜이 없으면 단 1초도 돌아갈 수 없는 거대 서버 아키텍처.

## 👶 어린이를 위한 3줄 비유 설명
1. 디렉토리 프로토콜은 학교 도서관의 **'사서 선생님이 가지고 있는 꼼꼼한 대출 장부'**예요.
2. 옛날엔 누가 내 책을 가져갔는지 몰라서 복도에서 "내 책 가진 사람!!" 하고 소리를 쳐야 했지만(스누핑), 이제는 사서 선생님 장부만 쓱 보면 누가 가져갔는지 1초 만에 알 수 있죠.
3. 소리 지르느라 목도 안 아프고, 수천 명이 있는 거대한 도서관에서도 조용하고 완벽하게 책(데이터)을 나눠 볼 수 있는 아주 훌륭한 시스템이랍니다!
