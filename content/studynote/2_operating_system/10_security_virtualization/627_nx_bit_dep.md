# NX Bit / DEP

## 핵심 인사이트

**NX(Non-Execute) Bit는 CPU 하드웨어 수준에서 메모리 페이지의 실행 여부를 제어하는 보안 기능이다.** 스택이나 힙 같은 데이터 영역에서 코드 실행을 금지하여 버퍼 오버플로우 공격 시 **쉸코드 실행을 방지**한다. 윈도우에서는 DEP(Data Execution Prevention), 리눅스에서는 XD(Execute Disable) 또는 NX Bit라고 부른다.

---

## Ⅰ. 개요 및 정의

### 1. 정의

| 용어 | 설명 |
|:---|:---|
| **NX Bit** | CPU의 실행 불가 비트, 페이지 단위 실행 제어 |
| **DEP** | 윈도우의 NX Bit 구현 명칭 |
| **XD** | Intel의 Execute Disable Bit |
| **W^X** | 메모리 페이지는 쓰기 또는 실행만 가능 (둘 다 ✗) |

### 2. 역사

```
┌─────────────────────────────────────────────────────────────┐
│               NX/DEP 발전 역사                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1990년대:                                                  │
│  • 스택에서 코드 실행 가능 (보안 문제!)                      │
│  • 버퍼 오버플로우 → 쉸코드 실행                            │
│                                                             │

│                                                             │

│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Ⅱ. 동작 원리

### 1. 페이지 테이블 비트

```
┌─────────────────────────────────────────────────────────────┐
│              페이지 테이블에서의 NX Bit                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  페이지 테이블 엔트리 (PTE/PDE)                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 63 62 ... 1 0                                       │   │
│  │ ┌─┬──────────────┬─┬───┐                             │   │
│  │ │NX│   주소      │R/W│P │                            │   │
│  │ └─┴──────────────┴─┴───┘                             │   │
│  │  │                                        │            │   │
│  │  └── 1 = 실행 불가, 0 = 실행 가능                │            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  NX Bit = 1인 페이지에서 코드 실행 시도 시:                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ CPU Exception: #PF (Page Fault)                     │   │
│  │ → SIGSEGV (리눅스)                                  │   │
│  │ → STATUS_ACCESS_VIOLATION (윈도우)                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

│  │   └─────────────────────────────────────────────┘   │   │

## Ⅲ. OS별 구현

### 1. 리눅스

```bash
# CPU NX 지원 확인
$ cat /proc/cpuinfo | grep flags
... nx ...  # NX 비트 확인

# PaX/Grsecurity 패치
# 페이지 실행 제어 선구자

# 커널 CONFIG 옵션
CONFIG_X86_64=y          # 64비트는 항상 NX
CONFIG_PAGE_TABLE_ISOLATION=y  # KPTI와 함께
```

### 2. 윈도우 (DEP)

```
┌─────────────────────────────────────────────────────────────┐
│                윈도우 DEP 모드                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. AlwaysOn (기본, Vista 이상)                            │
│     ├── 시스템 전체 DEP 활성화                               │
│     ├── OptIn 예외 허용 안 함                                │
│     └── 가장 안전한 모드                                    │
│                                                             │
│  2. OptIn (XP SP2 기본값)                                   │
│     ├── 필수 프로그램에만 DEP                               │
│     ├── 사용자 예외 추가 가능                                │
│     └── 호환성 우선                                         │
│                                                             │
│  3. OptOut                                                  │
│     ├── 기본 DEP 활성화                                     │
│     ├── 사용자 예외 삭제 가능                                │
│     └── 낮은 호환성                                         │
│                                                             │
│  4. AlwaysOff                                               │
│     ├── DEP 비활성화                                        │
│     └── 절대 사용 금지!                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```powershell
# DEP 상태 확인
# bcdedit /nx

# DEP 활성화
# bcdedit /set nx AlwaysOn

# DEP 비활성화 (권장하지 않음)
# bcdedit /set nx AlwaysOff
```

## Ⅳ. 우회 기법

### 1. ROP (Return-Oriented Programming)

```
┌─────────────────────────────────────────────────────────────┐
│         NX Bit 우회를 위한 ROP 기법                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  문제: NX Bit으로 쉸코드 실행 불가!                        │
│                                                             │
│  해결: 이미 존재하는 코드 조각(Gadget) 재조립               │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │  라이브러리/실행 파일에서 "쓸만한" 명령어 찾기      │   │
│  │                                                     │   │
│  │  [libc.so 또는 실행 파일]                           │   │
│  │  ┌─────────────────────────────────────────────┐   │   │
│  │  │ pop eax; ret  ← Gadget 1                     │   │   │
│  │  │ pop ebx; ret  ← Gadget 2                     │   │   │
│  │  │ xor ecx, ecx; ret ← Gadget 3                 │   │   │
│  │  │ ...                                           │   │   │
│  │  └─────────────────────────────────────────────┘   │   │
│  │                                                     │   │
│  │  스택에 Gadget 주소 연결:                           │   │
│  │  ┌─────────────────────────────────────────────┐   │   │
│  │  │ [Gadget1 주소][Gadget2 주소][...][system()] │   │   │
│  │  └─────────────────────────────────────────────┘   │   │
│  │                                                     │   │
│  │  실행 흐름:                                         │   │
│  │  ret → Gadget1 → ret → Gadget2 → ret → ...        │   │
│  │                                                     │   │
│  │  결과: 실행 가능 영역의 코드만 사용 → 우회!        │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. JIT Spraying

```
┌─────────────────────────────────────────────────────────────┐
│                JIT Spraying 공격                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  문제: NX로 실행 불가                                      │
│                                                             │
│  해결: JIT(Just-In-Time) 컴파일러 악용                      │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │  1. Flash/Java Script 등 JIT 환경                  │   │
│  │     ├── ActionScript/JavaScript 코드               │   │
│  │     └── JIT 컴파일 → 실행 가능한 기계어            │   │
│  │                                                     │   │
│  │  2. 악성 코드 패턴 생성                             │   │
│  │     예: "0x0c0c0c0c" 많이 반복                       │   │
│  │     → NOP 슬레드 효과                               │   │
│  │                                                     │   │
│  │  3. 버퍼 오버플로우로 JIT 영역으로 점프            │   │
│  │     → NX 우회                                       │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

│              Return-to-libc 공격                             │

## Ⅴ. 방어 강화

### 1. ASLR과의 결합

```
┌─────────────────────────────────────────────────────────────┐
│                NX + ASLR 조합 효과                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  NX만 사용:                                                 │
│  • 쉸코드 실행 ✗                                            │
│  • Return-to-libc 가능 (libc 주소 고정)                     │
│                                                             │
│  NX + ASLR 사용:                                            │
│  • 쉸코드 실행 ✗                                            │
│  • Return-to-libc 어려움 (libc 주소 무작위)                   │
│  • ROP도 어려움 (Gadget 주소 예측 불가)                     │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │   공격자 난이도                                      │   │
│  │                                                     │   │
│  │   없음  → NX  →  NX+ASLR  → NX+ASLR+Canary          │   │
│   ────────────────────────────────────────────────▶     │   │
│      쉬움       어려움    매우 어려움    거의 불가능     │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 추가 보안 조치

| 조치 | 설명 |
|:---|:---|
| **ASLR** | 주소 무작위화로 ROP 어려움 |
| **Canary** | 스택 변조 탐지 |
| **CFI** | Control Flow Integrity (제어 흐름 검증) |
| **CET** | Intel CET (Shadow Stack) |

## 어린이를 위한 비유

**NX Bit는 "이곳에선 달리지 마!" 표지판**과 같습니다.

상상해 보세요:
- **실행 가능한 곳**: 운동장 (코드 영역) - 여기선 달리기 OK!
- **실행 불가능한 곳**: 교실, 도서관 (데이터 영역) - 여기선 달리기 NO!

**NX Bit 없으면**:
- 나쁜 아이가 교실에 들어와서 달려도 괜찮아
- 교실에 "비밀번호 훔쳐!"라고 써두면
- 아무도 말릴 수 없어요

**NX Bit 있으면**:
- 교실 입구에 "여기선 달리기 금지!" 표지판
- 나쁜 아이가 교실에서 달리려고 하면
- 선생님(CPU)이 "야! 여기선 안 돼!"하고 막아요

**하지만 똑똑한 나쁜 아이들은**:
- 운동장에 있는 "달리기", "멈춰", "돌아" 같은 명령들을
- 엮어서 새로운 짓을 꾸며
- 이게 **ROP(Return-Oriented Programming)**예요

**더 막으려면**:
- **ASLR**: "운동장 위치를 매번 바꿔"
- **Canary**: "경비견을 세워두고"

NX Bit는 **"교실에서 못 뛰게"** 하는 첫 번째 방어선이에요!
