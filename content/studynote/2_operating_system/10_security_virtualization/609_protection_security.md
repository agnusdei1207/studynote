# 609. Protection vs Security (보호와 보안)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보호(Protection)는 OS 내부 자원에 대한 접근 제어 메커니즘이며, 보안(Security)는 외부 공격과 내부 위협으로부터 시스템을 방어하는 포괄적 체계이다.
> 2. **가치**: 보호는 프로세스 간 격리와 자원 무결성을 보장하고, 보안은 인증, 암호화, 감사를 통해 시스템 전체의 기밀성·무결성·가용성(CIA)을 확보한다.
> 3. **융합**: 현대 OS는 보호 도메인, 접근 제어 행렬, SELinux와 같은 강제적 접근 제어(MAC)를 통해 커널 레벨 보호와 애플리케이션 레벨 보안을 통합 관리한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**보호(Protection)**는 **운영체제가 관리하는 자원(프로세스, 메모리, 파일, 장치)에 대한 불법적·무단 접근을 방지하는 내부 메커니즘**이다. 접근 제어 행렬, 보호 도메인,_capability_ 시스템 등을 통해 주체(Subject)가 객체(Object)에 접근할 수 있는 권한을 명시적으로 제어한다. 주로 프로세스 간 격리, 메모리 보호, 파일 시스템 권한 등 OS 자원 관리의 핵심을 담당한다.

**보안(Security)**는 **시스템을 내·외부의 의도적·우발적 위협으로부터 방어하고 데이터의 기밀성(Confidentiality), 무결성(Integrity), 가용성(Availability)을 보장하는 포괄적 체계**이다. 인증(Authentication), 권한 부여(Authorization), 암호화(Encryption), 감사(Auditing), 취약점 관리 등을 포함하며, OS 보호 메커니즘을 기반으로 네트워크 방화벽, 침입 탐지, 보안 정책 enforcement까지 확장된다.

### 💡 비유: 집 안의 잠금 장치 vs 경비 시스템

보호는 **각 방마다 있는 문의 잠금 장치**와 같다. 가족 구성원 각자의 방에 침입하지 못하도록 내부에서 서로를 격리하고, 가재도구를 보호한다. 반면 보안은 **집 전체의 경비 시스템**과 같다. 외부 침입자를 막는 울타리, CCTV, 알람 시스템, 그리고 가족 구성원 확인을 위한 지문 인식, 출입 기록 등을 포함한다. 보호는 내부 구성원 간의 갈등을 막고, 보안은 외부 위협으로부터 집 전체를 지킨다.

### 등장 배경

1. **초기 시분할 시스템의 문제**: 1960년대 시분할 시스템(TSS)에서 다중 사용자가 단일 컴퓨터를 공유하면서, 사용자 A가 사용자 B의 프로세스나 메모리를 침해하는 문제가 발생했다. 이는 프로세스 간 격리와 자원 보호의 필요성을 부각했다.
2. **멀티프로그래밍 환경에서의 자원 경합**: 여러 프로세스가 동시에 실행되면서 파일, 메모리, 장치 등 공유 자원에 대한 접근 제어가 필수적이 되었다. 임계 구역 문제, 교착 상태, 경쟁 조건 등 병행성 문제를 해결하기 위한 보호 메커니즘이 요구되었다.
3. **네트워크와 인터넷의 등장**: 1970~80년대 ARPANET과 인터넷의 확산으로 원격 접속, 데이터 전송 보안, 악의적 공격 방어가 중요해졌다. 비밀번호 평문 전송, 인터럽트 남용, 버퍼 오버플로우 등 다양한 보안 취약점이 대두되었다.
4. **현대 클라우드와 컨테이너 환경**: 가상화, 컨테이너, 마이크로서비스 아키텍처에서는 전통적 프로세스 격리만으로 부족하다. 네임스페이스, cgroups, SELinux/AppArmor, TPM(Trusted Platform Module) 등 심화된 보호/보안 기술이 요구된다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **보호 도메인(Protection Domain)** | 프로세스의 접근 권한 집합 정의 | 도메인 전이 시 권한 동적 변경 | Ring 레벨(x86), Capability | 여권에 찍힌 입국 허가 |
| **접근 제어 행렬(Access Matrix)** | 주체-객체 간 권한 명시적 표현 | 행렬 압축하여 ACL/Capability로 구현 | DAC, MAC, RBAC | 건물 보안 출입 명단 |
| **참조 모니터(Reference Monitor)** | 모든 자원 접근을 중개하는 추상화 계층 | 시스템 콜/트랩을 통해 검증 수행 | 커널 레벨 검증, LSM 후크 | 보안 검색대 경비원 |
| **감사 로그(Audit Log)** | 보안 이벤트 기록 및 추적 | 시스템 콜, 접근 시도, 권한 변경 로깅 | auditd, syslog, SIEM | CCTV 보관용 녹화 |
| **보안 정책(Security Policy)** | 허용/거부 규칙 집합 정의 | 관리자가 설정한 규칙 기반 판단 | SELinux policy, firewall rule | 건물 보안 규정手册 |

### 보호 vs 보안: 핵심 차이

보호와 보안의 차이를 3가지 차원에서 비교하면 다음과 같다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                  보호(Protection) vs 보안(Security)                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【1. 관심 범위】                                                         │
│  ────────────                                                           │
│  • 보호: OS 내부 자원(프로세스, 메모리, 파일)의 접근 제어                       │
│  • 보안: 내·외부 위협으로부터 시스템 전체의 방어 (CIA triad)                 │
│                                                                         │
│  【2. 주요 메커니즘】                                                     │
│  ────────────────                                                       │
│  • 보호: 접근 제어 행렬, 보호 도메인, 모드 비트, 페이지 테이블 권한              │
│  • 보안: 인증, 암호화, 방화벽, IDS, 취약점 관리, 보안 정책                     │
│                                                                         │
│  【3. 구현 레벨】                                                        │
│  ────────────                                                           │
│  • 보호: 커널 수준, 하드웨어 지원(MMU, TLB, Ring)                           │
│  • 보안: 커널 + 애플리케이션 + 네트워크 + 물리적 장치까지                       │
│                                                                         │
│  【4. 예시】                                                             │
│  ────────────                                                           │
│  • 보호: 프로세스 A가 프로세스 B의 메모리를 읽지 못함 (mmu, page protection) │
│  • 보안: 외부 해커가 SSH 무차별 대입 공격 시도 차단 (fail2ban, key auth)     │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 보호는 OS 자원의 내부 격리에 집중한다. 예를 들어, MMU의 페이지 테이블에서 각 프로세스의 페이지가 서로 다른 물리 프레임을 가리키도록 매핑하여 프로세스 간 메모리 침해를 원천 차단한다. 반면 보안은 이러한 보호 메커니즘을 기반으로 네트워크 공격 방어, 악성코드 탐지, 데이터 암호화 등 더 넓은 범위를 다룬다. 보호가 "내부 자원의 접근 제어"라면 보안은 "내·외부 위협으로부터 시스템 방어"라고 이해할 수 있다.

### 접근 제어 모델

운영체제는 크게 세 가지 접근 제어 모델을 제공한다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                   접근 제어 모델 3가지                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【DAC (Discretionary Access Control) - 임의적 접근 제어】                  │
│  ────────────────────────────────────────────────                       │
│  • 소유자(Owner)가 자신의 자원에 대한 접근 권한을 자유롭게 지정                    │
│  • Unix 파일 권한(rwx), ACL(Access Control List)                            │
│  • 장점: 유연성, 사용자 편의성                                              │
│  • 단점: 소유자 실수로 권한 과도 부여 가능, 악의적 프로세스 권한 상속 위험         │
│  • 예: chmod 755 file.sh → 소유자에게 모든 권한, 타인에게 읽기/실행만 부여         │
│                                                                         │
│  【MAC (Mandatory Access Control) - 강제적 접근 제어】                      │
│  ────────────────────────────────────────────────                       │
│  • 시스템 보안 정책에 따라 모든 접근을 중앙 집중적으로 제어                           │
│  • 사용자는 권한을 변경할 수 없음                                             │
│  • SELinux, AppArmor, TrustedSolaris                                     │
│  • 장점: 강력한 보안, 권한 에스컬레이션 방지                                   │
│  • 단점: 구성 복잡, 오버헤드, 사용성 저하                                     │
│  • 예: SELinux policy로 웹 서버가 /etc/shadow에 접근 불가하게 강제              │
│                                                                         │
│  【RBAC (Role-Based Access Control) - 역할 기반 접근 제어】                 │
│  ────────────────────────────────────────────────────                    │
│  • 사용자가 아닌 역할(Role)에 권한 부여                                      │
│  • 사용자는 하나 이상의 역할을 맡을 수 있음                                   │
│  • POSIX.1e, Windows Active Directory                                     │
│  • 장점: 관리 효율성, 권한 분리가 용이                                         │
│  • 단점: 역할 정의 복잡, 역할 폭발(Role Explosion) 문제                       │
│  • 예: "DBA" 역할에 데이터베이스 모든 권한 부여, 사용자를 역할에 할당                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** DAC는 Unix/Linux의 기본 권한 모델로, 파일 소유자가 `chmod`, `chown` 명령으로 권한을 자유롭게 변경할 수 있다. 이는 사용자 편의성이 높지만, 악의적 사용자가 `/etc/passwd`를 world-writable로 변경하거나 setuid 스크립트를 통해 권한을 상속받는 보안 구멍이 있다. MAC는 이를 방지하기 위해 시스템 전체 정책으로 모든 접근을 제어한다. SELinux는 커널 레벨에서 모든 시스템 콜을 후킹(hook)하여 정책 위반 접근을 즉시 차단한다. RBAC는 기업 환경에서 사용자별 개별 권한 부여의 오버헤드를 줄이기 위해, 역할(예: "개발자", "관리자")에 권한을 모아서 할당한다.

### 리눅스 보호/보안 아키텍처

리눅스 커널의 보호와 보안 메커니즘을 계층별로 분석하면 다음과 같다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│               리눅스 커널 보호/보안 아키텍처                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【하드웨어 계층】                                                         │
│  ────────────                                                           │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │ • MMU(페이지 테이블): 프로세스별 주소 공간 격리                               │ │
│  │ • TLB ASID: 주소 공간 식별자로 TLB flush 최소화                             │ │
│  │ • Ring 레벨(0-3): 커널 모드(Ring 0)와 사용자 모드(Ring 3) 분리                 │ │
│  │ • NX Bit(DEP): 데이터 영역 실행 방지                                        │ │
│  │ • SMEP/SMAP: 커널 모드에서 사용자 공간 접근/실행 제어                           │ │
│  │ • TPM(Trusted Platform Module): 하드웨어 기반 키 저장, 부팅 무결성 검증          │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                ▲                                         │
│                                │ 하드웨어 보조                               │
│                                ▼                                         │
│  【커널 코어 계층】                                                       │
│  ────────────────                                                       │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │ • 모드 비트: 시스템 콜 시 커널 모드 전환                                     │ │
│  │ • 페이지 테이블 권한: 각 페이지별 R/W/X/U(Supervisor) 비트로 접근 제어            │ │
│  │ • VMA(Virtual Memory Area): 프로세스 주소 공간별 권한 관리                   │ │
│  │ • Capability: POSIX capability로 root 권한 분리(CAP_NET_BIND, etc.)   │ │
│  │ • Seccomp: 시스템 콜 필터링으로 공격 표면 최소화                               │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                ▲                                         │
│                                │ LSM(Linux Security Modules) 후크           │
│                                ▼                                         │
│  【LSM 계층】                                                           │
│  ────────────                                                           │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │ • SELinux: Type Enforcement으로 프로세스-자원 간 접근 제어                   │ │
│  │   - 도메인(프로세스)과 타입(파일, 포트 등) 간 접근 규칙 정의                     │ │
│  │   - 정책 위반 시 AV(Access Vector) 거부                                   │ │
│  │ • AppArmor: 경로 기반 프로필로 접근 제어                                    │ │
│  │   - /etc/passwd, /var/log/* 등 경로 패턴으로 접근 허용/거부                    │ │
│  │ • Smack: Simplified Mandatory Access Control                            │ │
│  │ • Tomoyo: 학습 모드로 정책 생성                                            │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                ▲                                         │
│                                │ 감사 로그                                 │
│                                ▼                                         │
│  【감사 및 시스템 콜 추적】                                                  │
│  ────────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │ • auditd: 커널 감사 하위 시스템을 통해 시스템 콜, 파일 접근, 권한 변경 로그       │ │
│  │ • eBPF: 안전한 커널 내 프로그래밍으로 실시간 이벤트 필터링                         │ │
│  │ • ftrace: 커널 함수 추적으로 보안 이벤트 분석                                 │ │
│  │ • SystemTap: 동적 커널 instrumentation                                   │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 리눅스는 하드웨어, 커널 코어, LSM, 감사 계층으로 보호/보안을 계층화했다. 하드웨어 계층에서 MMU와 페이지 테이블은 각 프로세스의 가상 주소를 다른 물리 주소로 매핑하여 격리한다. NX Bit는 데이터 영역(stack, heap)에서 코드 실행을 방지하여 버퍼 오버플로우 공격을 무력화한다. 커널 코어에서는 POSIX Capability로 전통적 root 권한을 세분화하여, 프로세스가 특정 작업(예: 낮은 포트 바인딩)에 필요한 최소 권한만 갖도록 한다. LSM(Linux Security Modules)은 커널의 핵심 코드를 수정하지 않고도 보안 모듈(SELinux, AppArmor 등)을 로드할 수 있는 후크 인터페이스를 제공한다. 예를 들어, SELinux는 모든 시스템 콜(허용, 거부)을 감사 로그에 기록하고, 정책 위반 시 즉시 차단한다.

### 보호 도메인과 도메인 전이

보호 도메인(Protection Domain)은 프로세스가 특정 시점에 접근할 수 있는 객체와 권한의 집합이다. 도메인 간 전이는 권한 변화를 동반한다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│              보호 도메인(Protection Domain) 구조                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【도메인 정의】                                                          │
│  ───────────                                                           │
│  • 도메인 D = {(객체 o, 권한 r) | 프로세스가 o에 대해 r을 가짐}                  │
│  • 예: D_shell = {(/etc/passwd, r), (/tmp/log, rw), (stdout, w)}           │
│                                                                         │
│  【도메인 전이(Domain Switching)】                                         │
│  ──────────────────────────────                                        │
│  • 시스템 콜, 파일 실행, 신호 수신 등으로 도메인 변경                                 │
│  • 전이 조건: 전이 권한이 있거나, 객체가 전이를 허용하는 경우                          │
│  • 예: 사용자가 `sudo /bin/bash` 실행하면 도메인이 user_domain → admin_domain으로 전이 │
│                                                                         │
│  【Unix 파일 실행에 따른 도메인 전이】                                        │
│  ──────────────────────────────────                                    │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  단계          도메인            권한 변화                            │ │
│  │  ────          ──────            ───────                            │ │
│  │  초기           사용자 도메인      자신의 파일만 접근                       │ │
│  │  execve()      setuid 실행파일    실행파일 소유자의 권한 획득                 │ │
│  │  sudo          관리자 도메인      root 권한 획득                         │ │
│  │  chmod         자신의 파일만      권한 변경 가능 (소유자만)                  │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  【Ring 레벨(x86 아키텍처)】                                             │
│  ────────────────────────────                                         │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  Ring 0 (Kernel Mode)      - 모든 명령어, 모든 메모리 접근 가능               │ │
│  │  Ring 1 (Device Drivers)   - 일부 I/O 명령어 제한                        │ │
│  │  Ring 2 (Device Drivers)   - 일부 I/O 명령어 제한                        │ │
│  │  Ring 3 (User Mode)        - 제한된 명령어, 자신의 주소 공간만 접근             │ │
│  │                                                                        │ │
│  │  전이: 시스템 콜(int 0x80, syscall) 시 Ring 3 → Ring 0                    │ │
│  │       iret 반환 시 Ring 0 → Ring 3                                       │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 보호 도메인은 프로세스의 권한 상태를 캡슐화한다. 예를 들어, 웹 브라우저 프로세스의 도메인은 사용자 홈 디렉터리의 파일만 읽을 수 있고, `/etc/shadow`에는 접근할 수 없다. 사용자가 `sudo`를 실행하면 인증 후 도메인이 관리자 도메인으로 전이하여 모든 자원에 접근할 수 있다. x86 Ring 레벨은 이를 하드웨어적으로 지원한다. 사용자 모드(Ring 3)에서는 특권 명령어(cli, sti, in/out 등)를 실행할 수 없어 하드웨어를 직접 조작할 수 없다. 시스템 콜이나 인터럽트 발생 시 CPU가 Ring 0로 전이하여 커널 코드가 실행된다. 이 전이 과정에서 스택을 사용자 스택에서 커널 스택으로 교체하고, 사용자 레지스터를 커널 스택에 저장하여 보호한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 보호 메커니즘 기법

| 기법 | 목적 | 구현 방식 | 장점 | 단점 | 사용 사례 |
|:---|:---|:---|:---|:---|:---|
| **모드 비트(Mode Bit)** | 커널/사용자 모드 분리 | CPU 레지스터의 1비트 플래그 | 하드웨어 지원, 빠름 | 2단계만 구현(단순) | 모든 범용 OS |
| **페이지 테이블 권한 비트** | 메모리 페이지별 R/W/X 제어 | MMU의 페이지 테이블 엔트리 비트 | 세분화된 제어, 하드웨어 지원 | TLB flush 오버헤드 | 프로세스 격리, Code segment 보호 |
| **세그먼트테이션** | 논리적 단위별 메모리 보호 | 세그먼트 테이블의 Base/Limit | 논리적 구조 반영 | 외부 단편화, 복잡함 | x86 리얼 모드(legacy) |
| **Capability** | 권한을 불변 토큰으로 캡슐화 | 포인터 + 권한 비트 조합 | 위조 불가, 권한 전파 용이 | 구현 복잡, 회복 어려움 | E 시스템, KEYKOS OS |
| **Access Control List** | 객체별 접근자 명시 | 객체 메타데이터에 권한 목록 | 관리 직관적 | 목록 크기 선형 증가 | Windows NTFS, POSIX ACL |
| **Ring 레벨** | 다단계 특권 계층 | x86의 4개 Ring | 계층적 보호 | 대부분 2개만 사용 | OS/2, Windows(일부) |
| **ASLR** | 주소 랜덤화로 공격 난이도 상승 | 메모리 영역 랜덤 배치 | 버퍼 오버플로우 방어 | 정보 누출로 우회 가능 | 모던 OS(Linux, Windows) |

### 비교 2: SELinux vs AppArmor

| 비교 항목 | SELinux (NSA) | AppArmor (SuSE, Canonical) |
|:---|:---|:---|
| **모델** | Type Enforcement(라벨 기반) | 경로(Path) 기반 프로필 |
| **구성 복잡도** | 매우 높음 (정책 언어 학습 필요) | 상대적으로 낮음 (프로필 간단) |
| **성능 영향** | 거의 없음 (LSM 후크, 인메모리 라벨) | 미미 (경로 매칛만 추가) |
| **유연성** | 높음 (세분화된 컨트롤) | 제한적 (파일 시스템 구조 의존) |
| **주요 사용처** | 보안 중요 시스템 (RHEL, CentOS) | 데스크톱, 컨테이너 (Ubuntu, openSUSE) |
| **이식성** | 커널 라벨이라 파일시스템 독립적 | 경로 기반이라 마운트 포인트 변경 시 문제 |

SELinux와 AppArmor의 동작 원리를 시각화하면 다음과 같다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│              SELinux vs AppArmor 동작 원리                               │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【SELinux: Type Enforcement】                                          │
│  ────────────────────────────                                          │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  1. 모든 객체(프로세스, 파일, 소켓 등)에 라벨 부여                             │ │
│  │     - 프로세스: httpd_t, user_t, init_t                             │ │
│  │     - 파일: httpd_sys_content_t, user_home_t, shadow_t               │ │
│  │                                                                        │ │
│  │  2. 정책(Allow/Deny Rules) 정의                                          │ │
│  │     - allow httpd_t httpd_sys_content_t:file { read open getattr };    │ │
│  │     - neverallow httpd_t shadow_t:file { read write };                │ │
│  │                                                                        │ │
│  │  3. 접근 시 라벨 검증                                                    │ │
│  │     - httpd_t 프로세스가 shadow_t 파일(/etc/shadow) 접근 시도             │ │
│  │     - AVC(Access Vector) 캐시에서 정책 검색                                │ │
│  │     - 정책 위반 시 즉시 거부 + auditd 로그 기록                             │ │
│  │                                                                        │ │
│  │  4. 장점: 파일 시스템 독립적, 세분화된 제어                                   │ │
│  │     단점: 복잡한 정책 언어, 학습 곡선 가파름                                 │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  【AppArmor: Path-based Profile】                                       │
│  ────────────────────────────────                                       │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  1. 프로필 파일 작성 (/etc/apparmor.d/profile.name)                    │ │
│  │     #include <tunables/global>                                        │ │
│  │     /usr/sbin/httpd {                                                 │ │
│  │       # 허용 규칙                                                       │ │
│  │       /var/www/** r,                                                  │ │
│  │       /etc/httpd/** r,                                                │ │
│  │       /var/log/httpd/* w,                                             │ │
│  │       # 거부 규칙                                                      │ │
│  │       deny /etc/shadow rwx,                                           │ │
│  │     }                                                                  │ │
│  │                                                                        │ │
│  │  2. 프로필 로드                                                         │ │
│  │     $ apparmor_parser -r /etc/apparmor.d/usr.sbin.httpd                │ │
│  │                                                                        │ │
│  │  3. 경로 매칭으로 접근 제어                                                │ │
│  │     - httpd 프로세스가 /etc/shadow 접근 시도                               │ │
│  │     - 프로필에서 deny /etc/shadow 규칙 검색                                │ │
│  │     - 규칙 위반 시 거부 + syslog 기록                                     │ │
│  │                                                                        │ │
│  │  4. 장점: 설정 간단, 직관적                                               │ │
│  │     단점: 경로 의존적, 마운트 포인트 변경 시 문제                               │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SELinux는 객체마다 라벨을 부여하고, 라벨 간 접근 규칙을 정의한다. 예를 들어, `httpd_t` 도메인(라벨을 가진 프로세스)이 `shadow_t` 타입(라벨을 가진 파일)에 접근할 수 없도록 정책에 명시하면, 파일 시스템 경로와 상관없이 강제로 차단된다. 반면 AppArmor는 파일 경로를 기반으로 한다. `/etc/shadow` 경로를 `deny` 규칙으로 명시하면, 해당 경로에 대한 접근을 차단한다. SELinux는 inode에 라벨을 저장하여 파일이 이동해도 라벨이 유지되지만, AppArmor는 경로가 바뀌면 규칙이 무력화될 수 있다. 따라서 SELinux는 보안이 중요한 서버에, AppArmor는 사용자 친화적인 데스크톱/컨테이너 환경에 적합하다.

### 과목 융합 관점

- **컴퓨터 구조**: MMU의 페이지 테이블, TLB, ASID는 가상 메모리 보호의 하드웨어적 기반이다. SMEP(Supervisor Mode Execution Prevention)와 SMAP(Supervisor Mode Access Prevention)은 커널 모드에서 사용자 공간 코드 실행/접근을 막아 ROP(Return-Oriented Programming) 공격을 방어한다.
- **네트워크**: IPsec, TLS/SSL은 통신 구간 보안을 제공한다. 운영체제는 IPsec 스택을 커널에 내장하여 패킷 암호화/복호화 오버헤드를 줄인다.
- **데이터베이스**: DBMS는 OS의 파일 시스템 권한과 독립적인 행(Row) 수준 접근 제어를 제공한다. 그러나 최종 데이터 파일은 OS의 보호(DAC/MAC)를 받는다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 다중 테넌트 웹 서버에서 사용자 간 파일 격리**: 공유 호스팅 서버에서 사용자 A가 PHP 스크립트 취약점을 악용하여 사용자 B의 `config.php`를 읽으려 시도하는 상황. 아키텍트는 open_basedir 제한, suEXEC/Suphp로 각 사용자의 CGI를 해당 사용자 권한으로 실행, 그리고 SELinux로 Apache 프로세스가 자신의 document root 외에는 접근 불가하게 정책 설정하여 격리를 강화해야 한다.

2. **시나리오 — 컨테이너 탈출 공격 방어**: Docker 컨테이너 내에서 권한 상위 공격(예: `--privileged` 플래그 남용)으로 호스트 시스템에 접근하려는 공격. 아키텍트는 (1) `--privileged` 사용 금지, (2) readOnly 루트 파일 시스템, (3) AppArmor/SELinux 프로필로 컨테이너 프로세스 제한, (4) 네임스페이스 격리(user, pid, net, mount, ipc, uts), (5) cgroups으로 자원(메모리, CPU) 제한, (6) rootless 컨테이너 모드 권장 등 다중 방어를 설계해야 한다.

3. **시나리오 — 제로데이 공격 대응**: 알려진 취약점 패치가 없는 제로데이 공격이 발생했을 때의 대응. 보안 팀은 (1) 영향 범위 분석, (2) 완화(mitigation) 조치(SELinux 정책 강화, 영향받는 바이너리 실행 금지), (3) 감사 로그 분석으로 침해 여부 확인, (4) 패치 배포 전 가상 패치(modsecurity WAF 규칙, ftrace로 공격 경로 후킹) 적용, (5) 보안 커뮤니티情报 공유(CERT, vendor)로 대응한다.

### 도입 체크리스트

- **기술적**: SELinux/AppArmor 정책이 최소 권한 원칙(least privilege)을 따르는가? 감사 로그(auditd)가 규정 준수(PCI-DSS, ISO 27001) 요건을 충족하는가? 정기적 취약점 스캔(OpenVAS, Nessus)이 자동화되어 있는가?
- **운영·보안적**: 관리자 계정의 2요소 인증(2FA), 패스워드 복잡도 정책이 설정되어 있는가? 일상 계정과 관리 계정이 분리(SoD: Separation of Duties)되어 있는가? 보안 사고 시 대응 절차(IR: Incident Response)가 문서화되어 있는가?

### 안티패턴

- **_root_ 계정으로 직접 로그인**: `ssh root@server` 허용은 무차별 대입 공격의 대상이 되며, 감사 로그에서 누가 무엇을 했는지 추적할 수 없다. sudo를 사용하고 감사 로그를 남겨야 한다.
- **_chmod 777_ 남용**: 모든 사용자에게 쓰기 권한 부여는 악의적 사용자가 시스템 바이너리를 교체할 수 있게 한다. 필요한 최소 권한만 부여해야 한다.
- **SELinux 비활성화**: 문제 해결이 귀찮다는 이유로 `setenforce 0`을 영구화하는 것은 보호/보안 핵심을 제거하는 것이다. 대신 문제가 되는 정책만 수정해야 한다.
- **평문 프로토콜 사용**: Telnet, FTP, HTTP 대신 SSH, SFTP, HTTPS를 사용하여 통신 구간 암호화를 해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 | 최적화 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 감사 로그 미수집, 보안 사고 시 원인 파악 불가 | auditd, SELinux logging 도입 | **보안 사고 조사 시간 90% 단축**, 규정 준수 |
| **정량** | 관리자 권한 남용, 실수로 시스템 장애 | RBAC, sudo 로깅, 권한 분리 | **권한 관련 인시던트 70% 감소** |
| **정성** | 보안 사고 시 피해 확산, 근본 원인 불명 | 다중 방어, 최소 권한, 감사 추적ability | **보안 태세 강화**, 거버넌스 확립 |

### 미래 전망

- **제로 트러스트(Zero Trust)**: "신뢰하지 않고 항상 검증(Verify, Never Trust)" 철학하에 모든 접근을 실시간 검증하고 최소 권한만 부여하는 아키텍처로 전환된다. BeyondProd, SPIFFE/SPIRE 등이 이를 구현한다.
- **eBPF 기반 동적 보안**: 커널 모듈 재컴파일 없이 안전한 샌드박스 내에서 방화벽, IDS, 네트워크 모니터링을 구현한다. Cilium, Falco가 대표적이다.
- **HW 기반 보안 enclave**: Intel SGX, ARM TrustZone은 애플리케이션 코드와 데이터를 OS/하이퍼바이저로부터도 격리하여, 커널이 손상되어도 보안을 보장한다.

### 참고 표준

- **POSIX.1e (formerly POSIX 6)**: 접근 제어 리스트(ACL), 감사, 정보 라벨링 등 보안 API 표준
- **FIPS 140-2**: 암호화 모듈 검증 표준(NIST)
- **ISO/IEC 27001**: 정보 보안 관리 시스템(ISMS) 국제 표준
- **Common Criteria (ISO 15408)**: IT 보안 평가 기준
- **NIST SP 800-53**: 연방 정보 시스템 보안 통제 가이드

### 👶 어린이를 위한 3줄 비유 설명

**개념**: 보호는 **내부의 방 잠금장치** 같고, 보안은 **경비원과 CCTV** 같아요! 보호는 우리 가족끼리 서로의 방에 못 들어가게 하고, 보안은 나쁜 사람이 집 밖에서 못 들어오게 감시해요.

**원리**: 각 방에는 문 잠금장치(보호)가 있어서 가족 구성원만 들어갈 수 있어요. 집 입구에는 경비원(보안)이 서서 낯선 사람을 확인하고, CCTV로 모든 출입을 기록해요.

**효과**: 보호와 보안이 함께 작동해야 우리 집이 안전해요. 잠금장치만 있으면 경비원이 모를 때 침입할 수 있고, 경비원만 있으면 내부에서 싸울 수 있으니 두 가지가 모두 필요해요!
