# 스푸핑

## 핵심 인사이트

**스푸핑(Spoofing)은 공격자가 자신의 신원을 위장하여 다른 사용자/시스템인 것처럼 가장하는 공격 기법이다.** IP 주소, MAC 주소, DNS, 이메일 등 다양한 계층에서 스푸핑이 가능하며, 주로 **다른 공격(DoS, 세션 하이재킹 등)의 전 단계**로 사용된다. 방어를 위해서는 **인증 강화, 암호화, 출처 확인**이 필수적이다.

---

## Ⅰ. 개요 및 정의

### 1. 정의

| 유형 | 설명 | OSI 계층 |
|:---|:---|:---:|
| **IP 스푸핑** | 발신 IP 주소 위조 | L3 (Network) |
| **MAC 스푸핑** | MAC 주소 위조 | L2 (Data Link) |
| **ARP 스푸핑** | ARP 메시지 위조 | L2/L3 |
| **DNS 스푸핑** | DNS 응답 위조 | L7 (Application) |
| **Email 스푸핑** | 발신자 위조 | L7 |
| **Web 스푸핑** | 가짜 웹사이트 | L7 |

### 2. 스푸핑의 목적

```
┌─────────────────────────────────────────────────────────────┐
│               스푸핑의 주요 목적                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 신분 위장                                               │
│     ├── 신뢰하는 호스트로 위장                             │
│     ├── 방화벽 우회                                       │
│     └── 로그 숨김                                          │
│                                                             │
│  2. 공격 은폐                                               │
│     ├── 실제 공격자 위치 숨기기                             │
│     ├── 흔적 남기지 않기                                   │
│     └── 탐지 회피                                          │
│                                                             │
│  3. 중간자 공격 (MitM)                                      │
│     ├── 통신 하이재킹                                     │
│     ├── 데이터 변조                                        │
│     └── 세션 하이재킹                                     │
│                                                             │
│  4. 거부 서비스 (DoS)                                       │
│     ├── 반사 공격 (Reflected Attack)                       │
│     ├── 증폭 공격 (Amplified Attack)                       │
│     └── 분산 공격 (DDoS)                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Ⅱ. IP 스푸핑

### 1. 원리

```
┌─────────────────────────────────────────────────────────────┐
│                  IP 스푸핑 공격 구조                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  정상 통신:                                                 │
│  ┌──────────┐              ┌──────────┐                    │
│  │ 공격자   │              │  타겟    │                    │
│  │203.0.113.5│              │10.0.0.1  │                    │
│  └─────┬────┘              └─────▲────┘                    │
│        │                         │                          │
│        │ 패킷 발송                │                          │
│        │ SRC=203.0.113.5          │                          │
│        └─────────────────────────►                          │
│                                                             │
│  IP 스푸핑:                                                 │
│  ┌──────────┐              ┌──────────┐                    │
│  │ 공격자   │              │  타겟    │                    │
│  │203.0.113.5│              │10.0.0.1  │                    │
│  └─────┬────┘              └─────▲────┘                    │
│        │                         │                          │
│        │ 패킷 발송                │                          │
│        │ SRC=198.51.100.0  <── 위조!                      │
│        │ (신뢰 서버)            │                          │
│        └─────────────────────────►                          │
│                                                             │
│  결과:                                                      │
│  • 타겟은 신뢰 서버(198.51.100.0)에서 온 것으로 착각       │
│  • 응답 패킷은 신뢰 서버로 전송 (무효)                      │
│  • 공격자는 연결 없이 일방 공격 가능                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. IP 스푸핑 코드 예시

```c
// raw socket을 사용한 IP 스푸핑
#include <netinet/ip.h>
#include <netinet/udp.h>
#include <sys/socket.h>

void send_spoofed_packet() {
    int sockfd = socket(AF_INET, SOCK_RAW, IPPROTO_RAW);

    struct sockaddr_in dest;
    dest.sin_family = AF_INET;
    dest.sin_port = htons(80);
    inet_pton(AF_INET, "10.0.0.1", &dest.sin_addr);

    // 스푸핑된 패킷 구성
    char packet[4096];
    struct ip *iph = (struct ip *)packet;

    iph->ip_src.s_addr = inet_addr("198.51.100.0");  // ★ 위조된 발신지
    iph->ip_dst.s_addr = inet_addr("10.0.0.1");     // 타겟
    iph->ip_ttl = 64;

    // 패킷 전송
    sendto(sockfd, packet, iph->ip_len, 0,
           (struct sockaddr *)&dest, sizeof(dest));
}
```

## Ⅲ. ARP 스푸핑

### 1. ARP 프로토콜

```
┌─────────────────────────────────────────────────────────────┐
│              ARP (Address Resolution Protocol)                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  목적: IP 주소 → MAC 주소 변환                              │
│                                                             │
│  1. ARP Request (브로드캐스트)                              │
│     ┌───────────────────────────────────────────────────┐  │
│     │ "192.168.1.1의 MAC 주소는?"                       │  │
│     │ 모두에게 물어봄                                     │  │
│     └───────────────────────────────────────────────────┘  │
│                                                             │
│  2. ARP Reply (유니캐스트)                                  │
│     ┌───────────────────────────────────────────────────┐  │
│     │ "내 MAC은 aa:bb:cc:dd:ee:ff"                      │  │
│     │ 192.168.1.1 소유자가 응답                          │  │
│     └───────────────────────────────────────────────────┘  │
│                                                             │
│  3. ARP Cache                                             │
│     ┌───────────────────────────────────────────────────┐  │
│     │ 192.168.1.1 → aa:bb:cc:dd:ee:ff                    │  │
│     │ (일정 시간 동안 캐싱)                              │  │
│     └───────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. ARP 스푸핑 공격

```
┌─────────────────────────────────────────────────────────────┐
│              ARP 포이즈닝(Poisoning) 공격                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  정상 상태:                                                 │
│  ┌──────────┐                 ┌──────────┐                  │
│  │  사용자   │ 192.168.1.2       │  라우터  │                  │
│  │          │                  │          │                  │
│  │ MAC: AA   │ ←─── 트래픽 ────→ │ MAC: RR   │ 192.168.1.1   │
│  └──────────┘                  └──────────┘                  │
│                                                             │
│  ARP 포이즈닝 후:                                           │
│  ┌──────────┐                 ┌──────────┐                  │
│  │  공격자   │ 192.168.1.3       │  라우터  │                  │
│  │          │                  │          │                  │
│  │ MAC: AA   │ ←─ 트래픽 ───→ │ MAC: RR   │ 192.168.1.1   │
│  │(위장)   │  (공격자가 모두 가로챰!)         │                  │
│  └──────────┘                  └──────────┘                  │
│                                                             │
│  공격 방법:                                                 │
│  1. "192.168.1.1의 MAC은 나야(공격자 MAC)!"라고 지속 알림  │
│  2. 사용자와 라우터의 ARP 캐시 오염                         │
│  3. 공격자가 중간자(MitM) 위치                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3. 방어: Static ARP + Dynamic ARP Inspection

```bash
# 1. 정적 ARP 테이블
# ARP 스푸핑 방지를 위해 고정 ARP 항목 설정
$ sudo arp -s 192.168.1.1 aa:bb:cc:dd:ee:ff

# 2. 스위치 Dynamic ARP Inspection (DAI)
# Cisco 스위치 설정
switch(config)# ip arp inspection vlan 1
switch(config)# interface gigabitethernet 1/0/1
switch(config-if)# ip arp inspection trust

# 3. ARP watcher
$ sudo arping -D -I eth0 -c 2 192.168.1.1
# 중복 ARP 응답 탐지
```

## Ⅳ. DNS 스푸핑

### 1. DNS 스푸핑 원리

```
┌─────────────────────────────────────────────────────────────┐
│              DNS 스푸핑 공격                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 사용자가 example.com 조회                              │
│     ┌───────────────────────────────────────────────────┐  │
│     │ DNS Query: example.com IP는?                     │  │
│     │ ────────────────────────────────────────────────▶  │  │
│     │           DNS 서버                                │  │
│     └───────────────────────────────────────────────────┘  │
│                                                             │
│  2. 공격자가 가짜 응답 전송 (타이밍 공격)                  │
│     ┌───────────────────────────────────────────────────┐  │
│     │ DNS Response (위조):                             │  │
│     │ example.com → 6.6.6.6 (악성 서버)                │  │
│     │ ◀───────────────────────────────────────────────┐  │  │
│     │           (사용자보다 먼저 도착)                 │  │
│     └───────────────────────────────────────────────────┘  │
│                                                             │
│  3. 사용자 캐시 오염                                      │
│     • 첫 번째 응답이 수락                                 │
│     • 정상 응답이 도착해도 무시                            │
│     • 이후 example.com = 6.6.6.6로 해석                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 방어: DNSSEC

```
┌─────────────────────────────────────────────────────────────┐
│            DNSSEC (DNS Security Extensions)                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  DNSSEC는 디지털 서명으로 DNS 레코드 보호                    │
│                                                             │
│  서명 체인:                                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ Root DNS Key                                         │   │
│  │      ↓ 서명 및 검증                                  │   │
│  │ TLD Key (.com)                                       │   │
│  │      ↓ 서명 및 검증                                  │   │
│  │ Zone Key (example.com)                              │   │
│  │      ↓ 서명 및 검증                                  │   │
│  │ RRset (www.example.com = 1.2.3.4)                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  DNS 응답 검증:                                             │
│  1. DNSKEY 레코드에서 공개키 조회                          │
│  2. RRSIG 레코드로 서명 검증                               │
│  3. 위조된 응답은 "Bogus"로 거부                            │
│                                                             │
│  확인:                                                     │
│  $ dig +dnssec www.example.com                             │
│  $ dig dnssec-failed.org                                  │
│  ; flags: qr rd ra ad; QUERY: 1, ANSWER: 0,                │
│  ; ERROR: no valid RRS                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Ⅴ. 이메일 스푸핑

### 1. 스푸핑 유형

```
┌─────────────────────────────────────────────────────────────┐
│              이메일 스푸핑 유형                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. From: 위조                                               │
│     From: "CEO <ceo@company.com>" <hacker@evil.com>       │
│     └── 보여지는 주소만 위조 가능                           │
│                                                             │
│  2. Reply-To: 위조                                         │
│     Reply-To: <hacker@evil.com>                           │
│     └── 응답이 공격자에게 전송                             │
│                                                             │
│  3. SMTP 서버 위장                                          │
│     └── 공격자의 SMTP 서버를 합법 서버로 위장             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 방어: SPF/DKIM/DMARC

```
┌─────────────────────────────────────────────────────────────┐
│          이메일 인증 3종 세트 (SPF + DKIM + DMARC)           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. SPF (Sender Policy Framework)                           │
│     ┌───────────────────────────────────────────────────┐  │
│     │ @company.com TXT SPF "v=spf1 ip4:1.2.3.4 -all"   │  │
│     │                                                    │  │
│     │ 의미: 1.2.3.4만 @company.com 메일 발송 허용       │  │
│     │ 수신: 1.2.3.4 외부에서 온 메일 거부                 │  │
│     └───────────────────────────────────────────────────┘  │
│                                                             │
│  2. DKIM (DomainKeys Identified Mail)                      │
│     ┌───────────────────────────────────────────────────┐  │
│     │ 발송 서버가 메일에 디지털 서명                     │  │
│     │ 수신 서버: 도메인 공개키로 검증                     │  │
│     │ 변조 탐지 가능                                      │  │
│     └───────────────────────────────────────────────────┘  │
│                                                             │
│  3. DMARC (Domain-based Message Authentication)              │
│     ┌───────────────────────────────────────────────────┐  │
│     │ @company.com TXT DMARC "v=DMARC1; p=reject; rua=.."│  │
│     │                                                    │  │
│     │ SPF 또는 DKIM 실패 시 정책                         │  │
│     │ p=none: 모니터링만                                  │  │
│     │ p=quarantine: 격리                                  │  │
│     │ p=reject: 거부 (권장)                               │  │
│     └───────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Ⅵ. 탐지 및 방어

### 1. 스푸핑 탐지

```bash
# IP 스푸핑 탐지
# 수신 패킷의 SRC이 로컬 네트워크에 없는지 확인

$ sudo arping -D -I eth0 -c 2 192.168.1.1

```

## 어린이를 위한 비유

**스푸핑은 "가면 쓰고 위장하기"**와 같습니다.

**IP 스푸핑**:
- "보내는 사람" 주소를 속이는 것
- 편지 봉투에 **가짜 이름** 쓰기
```
"내가 친구 철수야"라고 적지만
실제로는 낯선 사람이 보낸 편지
```

**ARP 스푸핑**:
- "우리 학교 가는 길"에 **가짜 안내판** 설치
- 친구가 너를 찾으면 **너 말고 공격한테** 가게

**DNS 스푸핑**:
- 전화번호부를 몰래 바꾸기
```
"엄마 전화번호: 010-XXXX-XXXX"
라고 적혀 있지만
실제로는 낯선 사람 전화
```

**이메일 스푸핑**:
- 편지 봉투에 **가짜 보내는 사람** 이름
- "교장선생님"이라고 적혀 있지만
실제로는 나쁜 사람이 보낸 편지

**방어하는 방법**:
- **IP 필터**: "학교 밖에서 온 편지는 확인!"
- **DNSSEC**: "전화번호부 진위 확인" (디지털 서명)
- **SPF/DKIM**: "보내는 사람 진위 확인"

스푸핑은 **"가면을 쓰고 내 집 열쇠 달라고 하고"** 다니는 것과 같아요!
