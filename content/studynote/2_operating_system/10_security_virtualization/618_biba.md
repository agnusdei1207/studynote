## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Biba 모델(1977)은 **"무결성(Integrity) 보장을 위한 다중 levels 보안(Multi-Level Security)"**의 수학적 모델로서, **"No Read Down, No Write Up"** 두 가지 핵심 규칙으로 신뢰할 수 없는 하위 등급이 상위 등급을 오염시키는 것을 방지한다.
> 2. **가치**: Bell-LaPadula가 기밀성을 보장하는 반면, Biba는 **"신뢰할 수 있는 정보가 신뢰할 수 없는 정보에 의해 오염되는 것"**을 방지한다. 금융 시스템, 의료 기록, 전자 투표 등 무결성이 중요한 시스템에 필수적이다.
> 3. **융합**: 현대 보안 시스템은 **"Bell-LaPadula(기밀성) + Biba(무결성)"**를 결합하여 완전한 격리를 구현한다. SELinux, AppArmor, TrustedSolaris는 두 모델의 하이브리드를 지원한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: Biba 모델은 Kenneth Biba가 1977년에 발표한 **"무결성(Integrity) 보장을 위한 상태 머신(State Machine) 모델"**이다. 시스템의 모든 주체(Subject)와 객체(Object)에 **신뢰 등급(Integrity Level)**을 부여하고, **"No Read Down, No Write Up"** 두 가지 규칙으로 접근을 제어한다. Bell-LaPadula가 **"정보의 상향 유출(하위가 상위를 읽기)을 방지"**하는 데 초점을 맞춘 반면, Biba는 **"정보의 하향 오염(상위가 하위에 쓰기)을 방지"**하는 데 초점을 맞춘다.

- **💡 비유**: Biba 모델은 **"연구실의 실험 기록과 데이터 무결성"**과 같다. 연구실에는 **"검증된 데이터(Trusted), 연구 중인 데이터(Research), 미검증 데이터(Unverified)"** 등급이 있고, 각 데이터 파일에도 등급이 부여된다. **"검증된 데이터베이스는 연구 중인 데이터를 참조할 수 없다(No Read Down)"**는 규칙으로, 잘못된 연구 결과가 신뢰하는 시스템을 오염시키는 것을 방지한다. 또한 **"미검증 데이터는 검증된 데이터베이스에 쓸 수 없다(No Write Up)"**는 규칙으로, 신뢰할 수 없는 정보가 신뢰하는 시스템을 오염시키는 것을 방지한다.

- **등장 배경**:
  1. **Bell-LaPadula의 무결성 미보장(1970s)**: Bell-LaPadula 모델은 기밀성을 보장하지만, **"Secret 등급 해커가 Top Secret 문서의 내용을 모르더라도 파일을 삭제하거나 수정"**할 수 있어 무결성이 보장되지 않았다. 이를 해결하기 위해 Biba 모델이 제안되었다.
  2. **금융/의료 시스템의 무결성 요구(1980s-90s)**: 금융 거래, 의료 기록, 전자 투표 등은 **"데이터가 위변조되지 않았음"**을 증명해야 하므로, Biba 모델의 **"신뢰 계층(Integrity Hierarchy)"**가 필수적이다.
  3. **Clark-Wilson 모델과 통합(1990s-현재)**: Clark-Wilson 모델의 **"역할 분리(Separation of Duty)"**와 결합하여, 전자 상거래, 전자 투표 시스템에 적용된다.

Biba 모델의 신뢰 격자(Lattice) 구조를 시각화하면 다음과 같다. 신뢰 등급은 Trusted(T) > High(H) > Medium(M) > Low(L) 순서로 상하 관계가 있다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                Biba 모델 — 신뢰 등급 격자 구조                         │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [격자(Lattice) — 부분 순서 집합]                                    │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │           Trusted (T) — 가장 신뢰 가능                      │  │
 │  │              │  ──────────────────────────────────────────┐      │  │
 │  │              │  상위 등급(Higher Trust Level)                  │  │
 │  │              ▼                                                 │  │
 │  │           High (H)                                            │  │
 │  │              │  ──────────────────────────────────────────┐      │  │
 │  │              │                                                 │  │
 │  │              ▼                                                 │  │
 │  │           Medium (M)                                          │  │
 │  │              │  ──────────────────────────────────────────┐      │  │
 │  │              │  하위 등급(Lower Trust Level)                   │  │
 │  │              ▼                                                 │  │
 │  │           Low (L) — 가장 신뢰 불가                            │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [핵심 규칙]                                                         │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ① No Read Down (Simple Integrity Property)                   │  │
 │  │     • 주체 S는 자신의 등급보다 낮은 등급의 객체를 읽을 수 없음│  │
 │  │     • 수학적 표현: I(S) ≤ I(O)이어야 read 허용                │  │
 │  │                                                                │  │
 │  │     예: High 등급 프로세스는 Low 등급 데이터를 읽을 수 없음 │  │
 │  │         (하위 등급의 오염 방지)                                │  │
 │  │                                                                │  │
 │  │  ② No Write Up (*-Property)                                   │  │
 │  │     • 주체 S는 자신의 등급보다 높은 등급의 객체에 쓸 수 없음 │  │
 │  │     • 수학적 표현: I(S) ≥ I(O)이어야 write 허용               │  │
 │  │                                                                │  │
 │  │     예: Low 등급 사용자는 High 파일에 쓸 수 없음            │  │
 │  │         (상위 등급 오염 방지)                                  │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [허용 가능한 정보 흐름]                                              │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   L  ────read──▶ L     (같은 등급 간 읽기 허용)               │  │
 │  │   ▲              ▲                                            │  │
 │  │   │              │                                            │  │
 │  │   │             M     (하위 → 상위 읽기 허용)                │  │
 │  │   │              ▲                                            │  │
 │  │   │             │                                            │  │
 │  │   │            T     (하위 → 상위 읽기 허용)                │  │
 │  │                                                                │  │
 │  │   T  ───write──▶ T     (같은 등급 간 쓰기 허용)               │  │
 │  │   ▼              ▼                                            │  │
 │  │   │             H     (상위 → 하위 쓰기 허용)                │  │
 │  │   │             ▼                                            │  │
 │  │   │             M                                             │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  ✅ Read: 하위 등급 → 상위 등급 흐름만 허용 (신뢰 역전)    │
 │  ✅ Write: 상위 등급 → 하위 등급 흐름만 허용                 │
 │  ❌ Read: 상위 → 하위 (예: H가 L을 읽기) = 금지               │
 │  ❌ Write: 하위 → 상위 (예: L가 T에 쓰기) = 금지              │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Biba 모델의 **"No Read Down"**은 신뢰할 수 없는 데이터(예: 인터넷 다운로드)가 신뢰하는 시스템을 오염시키는 것을 방지한다. 예를 들어 `/usr/bin`의 실행 파일들은 High 등급으로, 다운로드 파일은 Low 등급으로 레이블링하면, High 등급 프로세스가 Low 파일을 읽을 수 없게 된다. **"No Write Up"**은 Low 등급 사용자가 High 등급 시스템 파일을 수정하는 것을 방지한다(예: 일반 사용자가 `/bin/ls`를 백도어로 교체). Bell-LaPadula와 Biba를 동시에 적용하면 **"같은 등급 내에서만 읽기/쓰기 허용"**이라는 강력한 격리가 실현된다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **주체(Subject)** | 정보를 접근하는 능동 개체 | 사용자, 프로세스 | UID, 프로세스 ID | 사용자 |
| **객체(Object)** | 접근의 대상인 수동 개체 | 파일, 메모리, 장치 | inode, 메모리 페이지 | 데이터 |
| **신뢰 등급(Integrity Level)** | 주체/객체의 신뢰도 | Trustworthiness | Trusted, High, Medium, Low | 신뢰 등급 마크 |
| **격자(Lattice)** | 등급 간 상하 관계 정의 | 부분 순서 집합(Partial Order) | Poset(I, ≤) | 신뢰 계층 |
| **정책(Policy)** | 허용/거부 규칙 집합 | I(S) ≤ I(O) (read), I(S) ≥ I(O) (write) | Biba Policy | 보안 규정 |

### 수학적 정의와 시스템 모델

Biba 모델은 Bell-LaPadula와 유사한 상태 머신(State Machine)으로 정의되지만, 신뢰 등급(Integrity Level) 함수 I를 사용하며, 규칙이 반대다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │              Biba 모델 — 수학적 정의와 시스템 모델                   │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [시스템 구성 요소]                                                  │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  S = {s₁, s₂, ...}  ← 주체(Subject) 집합                   │  │
 │  │  O = {o₁, o₂, ...}  ← 객체(Object) 집합                     │  │
 │  │  I = {i₁, i₂, ...}  ← 신뢰 등급(Integrity Level) 집합       │  │
 │  │                                                                │  │
 │  │  부분 순서: ≤ 관계 (예: L ≤ M ≤ H ≤ T)                      │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [상태(State) 정의]                                                 │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  상태 Q = (S, O, I, F)                                         │  │
 │  │    • S: 현재 활성 주체 집합                                   │  │
 │  │    • O: 현재 접근 가능 객체 집합                               │  │
 │  │    • I: 신뢰 등급 함수                                         │  │
 │  │      - f_S(s): S → I  (주체 s의 등급)                        │  │
 │  │      - f_O(o): O → I  (객체 o의 등급)                        │  │
 │  │    • F: 현재 접근 권한 행렬                                  │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [보안 규칙(Security Rules)]                                        │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ① Simple Integrity Property (si-Property)                     │  │
 │  │     ∀s ∈ S, ∀o ∈ O:                                          │  │
 │  │       if read ∈ F(s, o) then f_S(s) ≤ f_O(o)                 │  │
 │  │     (읽기 권한이 있으면, 주체 등급 ≤ 객체 등급)              │  │
 │  │                                                                │  │
 │  │  ② *-Property (star Property)                                 │  │
 │  │     ∀s ∈ S, ∀o ∈ O:                                          │  │
 │  │       if write ∈ F(s, o) then f_S(s) ≥ f_O(o)                │  │
 │  │     (쓰기 권한이 있으면, 주체 등급 ≥ 객체 등급)             │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Biba 모델은 **"현재 상태에서 무결성 규칙을 위반하지 않는 다음 상태로만 전이"**하는 시스템을 정의한다. 예를 들어 High 등급 사용자 `s_H`가 Low 등급 데이터 `o_L`을 읽기(read) 요청하면, `f_S(s_H) = H > L = f_O(o_L)`이므로 **No Read Down 위반**으로 거부된다. 반대로 Low 등급 사용자 `s_L`가 High 등급 파일 `o_H`에 쓰기(write) 요청하면, `f_S(s_L) = L < H = f_O(o_H)`이므로 **No Write Up 위반**으로 거부된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: Bell-LaPadula vs Biba

| 비교 항목 | Bell-LaPadula | Biba |
|:---|:---|:---|
| **목적** | 기밀성(Confidentiality) | 무결성(Integrity) |
| **핵심 규칙** | No Read Up, No Write Down | No Read Down, No Write Up |
| **등급 체계** | TS > S > C > U (기밀성) | T > H > M > L (신뢰도) |
| **보호 대상** | 정보 유출 방지 | 정보 오염 방지 |
| **주 용도** | 군사 MLS 시스템 | 금융, 의료, 전자 투표 |
| **결합 시** | Biba와 결합 → 완전한 격리 | Bell-LaPadula와 결합 |

Bell-LaPadula는 **"정보의 상향 유출(하위가 상위를 읽기)을 방지"**하고, Biba는 **"정보의 하향 오염(상위가 하위에 쓰기)을 방지"**한다. 두 모델을 동시에 적용하면 **"같은 등급 내에서만 읽기/쓰기 허용"**이라는 강력한 격리가 실현된다. 이것이 현대 MLS(Multi-Level Security) 시스템의 표준 아키텍처다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 금융 시스템의 무결성 보장**: 은행의 핵심 장부 시스템은 무결성이 최우선이다. Biba 모델을 적용하여:
   - 신뢰하는 데이터베이스: Trusted 등급
   - 연구/개발 환경: High 등급
   - 인터넷 다운로드: Low 등급
   → 인터넷 다운로드 파일이 핵심 시스템에 영향을 주지 못하게 격리

2. **시나리오 — 전자 투표 시스템**: 전자 투표 시스템은 **"투표 데이터가 위변조되지 않았음"**을 증명해야 한다. Biba 모델로:
   - 투표 소프트웨어: Trusted 등급
   - 투표 데이터: High 등급
   - 검증된 유권자: Medium 등급
   - 미검증 사용자: Low 등급
   → 미검증 사용자의 입력이 투표 데이터에 영향을 주지 못하게 제어

### 도입 체크리스트
- **기술적**: 무결성 등급이 정의되어 있는가? Low 등급 프로세스가 High 등급 데이터를 읽지 못하도록 제어되는가?
- **운영·보안적**: 무결성 위반 시도록(audit trail)이 기록되는가? 정기적으로 무결성 검증을 수행하는가?

### 안티패턴
- **등급 미할당**: 중요한 데이터에 신뢰 등급을 부여하지 않으면, 기본적으로 최하위 등급으로 간주되어 오염 가능하다. 모든 중요 데이터는 명시적으로 레이블링해야 한다.
- **No Read Down 위반**: Low 등급 프로세스가 High 등급 데이터를 읽을 수 있게 허용하면, 신뢰할 수 없는 정보가 시스템을 오염시킬 수 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. Biba 모델은 **"연구실의 실험 데이터 무결성"**이에요. "검증된 데이터는 연구 중인 데이터를 참조할 수 없어요."
2. "검증된 데이터베이스(상위)는 인터넷 다운로드(하위)를 읽을 수 없고, 미검증 데이터는 검증된 데이터베이스에 쓸 수 없어요." 이렇게 하면 신뢰할 수 없는 정보가 시스템을 오염시키는 것을 막을 수 있어요.
