## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RBAC(Role-Based Access Control)는 **"사용자에게 직접 권한을 부여하지 않고, 중개자인 역할(Role)을 통해 간접적으로 부여"**하는 접근 제어 모델이다. 사용자 → 역할 → 권한의 3단계 계층으로 권한 관리의 복잡도를 O(N²)에서 O(N)으로 줄인다.
> 2. **가치**: 기업의 신규 입사자에게 개별 자원별 권한을 부여하면 수천 개의 ACL 엔트리가 필요하지만, RBAC로는 "Engineer 역할 부여" 한 줄로 해결된다. AWS IAM, Kubernetes RBAC, Azure RBAC는 모두 이 모델을 따른다.
> 3. **융합**: 최근 Zero Trust 보안과 결합하여 **"역할은 최소 권한으로 구성"**하고 **"역할 상승은 다중 인증(MFA)으로 강화"**하는 동적 RBAC(DRBAC)로 진화하고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: RBAC(Role-Based Access Control)는 사용자(User)가 **역할(Role)을 부여받고, 역할이 권한(Permission)을 보유**하는 **간접적 접근 제어** 모델이다. 수학적으로는 U × R × P의 삼중 관계(User-Role-Permission)로 정의되며, 사용자의 권한은 **UA(User Assignment) × PA(Permission Assignment)** 관계를 통해 결정된다. 이는 **"조직의 역할 구조를 보안 모델로 반영"**한 것이다.

- **💡 비유**: RBAC는 **"회사의 명찰과 출입카드 시스템"**과 같다. 각 직원은 "개발자", "매니저", "인턴" 같은 역할(Role)을 가지고 있으며, 역할별로 **출입 가능한 사무실과 사용 가능한 장비**가 미리 정의되어 있다. 신규 입사자는 인턴 명찰을 받으면 자동으로 인턴용 출입카드로 열리는 문만 통과할 수 있고, 개발자로 승진되면 명찰을 교체받아 개발자 구역에 접근 가능하다. 개별 문마다 "철수 통과, 영희 통과" 같은 명부를 관리할 필요 없이, **"역할별 출입 규칙"**만 있으면 된다.

- **등장 배경**:
  1. **DAC/MAC의 한계(1970s-80s)**: 기존의 DAC(소유자 중심)와 MAC(라벨 중심)는 사용자가 수백 명이 되면 관리가 불가능해진다. Ferraiolo & Kuhn(1992)이 RBAC 모델을 정립하여, **"사용자 대신 역할에 권한을 할당"**하면 관리 복잡도가 O(N²)에서 O(N)으로 줄어듦을 증명했다.
  2. **NIST RBAC 표준(2000)**: NIST SP 800-162에서 RBAC를 **4가지 수준(Flat, Hierarchical, Constrained, Symmetric)**으로 표준화하고, **"최소 권한, 책임 분리"** 등의 보안 원칙을 정의했다.
  3. **클라우드 시대의 RBAC(2010s-현재)**: AWS IAM, Kubernetes RBAC, Azure RBAC은 모두 NIST RBAC 표준을 구현한 것으로서, **"역할(Role), 정책(Policy), 바인딩(Binding)"**의 3요소로 클라우드 자원의 접근 제어를 관리한다.

RBAC와 기존 DAC의 대조를 시각화하면 다음과 같다. DAC는 사용자-자원 직접 매핑이지만, RBAC는 사용자-역할-권한의 3단계 추상화를 제공한다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                DAC vs RBAC: 권한 부여 구조 비교                      │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [DAC — 직접 사용자-권한 매핑]                                       │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ┌──────────┐                                                 │  │
 │  │  │  Alice   │───▶ File1: RW, File2: R, File3: —              │  │
 │  │  └──────────┘                                                 │  │
 │  │  ┌──────────┐                                                 │  │
 │  │  │   Bob    │───▶ File1: R,  File2: —, File3: R              │  │
 │  │  └──────────┘                                                 │  │
 │  │  ┌──────────┐                                                 │  │
 │  │  │ Charlie  │───▶ File1: —,  File2: R, File3: —              │  │
 │  │  └──────────┘                                                 │  │
 │  │                                                                │  │
 │  │  문제: 사용자 100명 × 파일 100개 = 10,000개의 매핑 관리       │  │
 │  │       신규 입사자: 개별 파일마다 권한 부여 필요               │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [RBAC — 사용자-역할-권한 3단계 매핑]                               │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ① 사용자 → 역할 할당 (User Assignment)                       │  │
 │  │     ┌──────────┐                                               │  │
 │  │     │  Alice   │───────────────────────┐                      │  │
 │  │     └──────────┘                       │                      │  │
 │  │     ┌──────────┐                       ▼                      │  │
 │  │     │   Bob    │───────────────▶ ┌─────────────┐              │  │
 │  │     └──────────┘               │             │              │  │
 │  │     ┌──────────┘               │    Roles    │              │  │
 │  │     │ Charlie  │───────────────▶ │             │              │  │
 │  │     └──────────┘               └─────────────┘              │  │
 │  │                                         │                     │  │
 │  │                                         │ ② 역할 → 권한 할당 │  │
 │  │                                         │    (Permission)     │  │
 │  │                                         ▼                     │  │
 │  │     ┌─────────────────────────────────────────────────┐      │  │
 │  │     │  Engineer: {File1: RW, File2: R, File3: —}       │      │  │
 │  │     │  Manager:   {File1: RW, File2: RW, File3: R}      │      │  │
 │  │     │  Intern:    {File1: R,  File2: —, File3: —}       │      │  │
 │  │     └─────────────────────────────────────────────────┘      │  │
 │  │                                                                │  │
 │  │  ✅ 관리 복잡도: 사용자 100 + 역할 3 + 권한 정의 3          │  │
 │  │     신규 입사자: "역할 부여" 한 줄로 모든 권한 해결         │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** DAC는 **"O(Users × Resources) 매핑"**을 직접 관리하므로, 사용자와 자원이 늘어나면 조합이 폭발적으로 증가한다. RBAC는 **"O(Users + Roles + Resources) 매핑"**으로 분리하므로, 신규 사용자에게는 적절한 역할을 부여하기만 하면 된다. 예를 들어 신규 개발자 입사 시 "Developer 역할 부여" 한 줄로, 수백 개의 개발 도구와 레포지토리에 대한 접근 권한을 일괄 부여할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **사용자(User)** | 시스템을 사용하는 주체 | UID, SID, ServiceAccount | IAM User, Kubernetes ServiceAccount | 직원 |
| **역할(Role)** | 권한의 논리적 묶음 | Job Function, Responsibility | AWS Role, K8s Role, Azure Role | 직책/직급 |
| **권한(Permission)** | 자원에 대한 연산 | {Resource, Action} 쌍 | `s3:GetObject`, `pods:exec` | 출입권+사용권 |
| **사용자 할당(UA)** | 사용자 → 역할 매핑 | 다대다(N:M) 관계 | User Group, RoleBinding | 배치발령 |
| **권한 할당(PA)** | 역할 → 권한 매핑 | 다대다(N:M) 관계 | Policy, Rule | 역할 설명서 |
| **세션(Session)** | 사용자의 활성 역할 집합 | 로그인 시 역할 활성화 | Login Session, Token | 근무 중인 역할 |
| **제약(Constraint)** | 역할 동시 부여 제한 | 상호 배역, 카디널리티 | SoD(책임 분리), Max 1 role | 겸직 금지 |

### NIST RBAC 모델의 4가지 수준

NIST SP 800-162는 RBAC의 기능을 4가지 수준으로 정의한다. 수준이 높을수록 더 강력한 제어가 가능하다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │              NIST RBAC 모델의 4가지 수준                              │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [RBAC0 — 최소 RBAC]                                                 │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  • 사용자 → 역할 할당 (UA)                                    │  │
 │  │  • 역할 → 권한 할당 (PA)                                      │  │
 │  │  • 세션에서 역할 활성화                                        │  │
 │  │                                                                │  │
 │  │  예: alice에게 "Engineer" 역할 부여, Engineer은 "code:read"   │  │
 │  │     권한 보유                                                  │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [RBAC1 — 계층적 RBAC (Hierarchical)]                                │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  • 역할 간 상속 관계(Role Hierarchy) 지원                     │  │
 │  │  • 상위 역할은 하위 역할의 모든 권한 상속                     │  │
 │  │                                                                │  │
 │  │         Director                                              │  │
 │  │           │                                                    │  │
 │  │     ┌─────┴─────┐                                              │  │
 │  │     Manager    Lead                                            │  │
 │  │       │         │                                              │  │
 │  │     Engineer ─┘  (Engineer 권한 상속)                          │  │
 │  │                                                                │  │
 │  │  예: Manager는 Engineer의 모든 권한 + 추가 관리 권한          │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [RBAC2 — 제약 RBAC (Constrained)]                                   │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  • 책임 분리(Separation of Duty, SoD) 제약                    │  │
 │  │  • 상호 배역(Mutually Exclusive Roles): 동시 부여 불가        │  │
 │  │  • 카디널리티 제약: 최대 N개 역할만 부여 가능                 │  │
 │  │                                                                │  │
 │  │  예: "Requester"와 "Approver" 역할은 동시에 부여 불가         │  │
 │  │      (자가 승인 방지)                                         │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [RBAC3 — 대칭 RBAC (Symmetric)]                                     │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  • 권한 검토(Permission Review) 지원                           │  │
 │  │  • "어떤 사용자가 이 권한을 가지는가?" 역질의 가능            │  │
 │  │                                                                │  │
 │  │  예: "s3:DeleteObject 권한을 가진 모든 사용자 조회"           │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  실무: 대부분의 클라우드 RBAC(AWS IAM, K8s)는 RBAC1 수준            │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RBAC0은 가장 기본적인 형태로서, 사용자-역할-권한의 3단계 할당만 지원한다. RBAC1은 **역할 계층(Role Hierarchy)**을 추가하여, "Manager는 Engineer의 권한을 모두 포함 + 추가 관리 권한" 같은 상속 관계를 정의할 수 있다. RBAC2는 **책임 분리(SoD)** 제약을 추가하여, "요청자와 승인자는 동일인이 될 수 없다" 같은 내부 통제를 강제할 수 있다. RBAC3는 **역질의 쿼리(Permission Review)**를 지원하여, 감사(Auditing) 시 "이 위험한 권한을 가진 모든 사용자를 조회"할 수 있다.

### AWS IAM — 클라우드 RBAC의 대표적 구현

AWS IAM(Identity and Access Management)은 RBAC를 **"User, Group, Policy, Role"** 4가지 요소로 구현한다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │               AWS IAM — RBAC 구조와 작동 원리                        │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [IAM 구성 요소]                                                     │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ① User / Group: 사용자와 그룹                                │  │
 │  │     ┌──────────────┐                                          │  │
 │  │     │ alice        │                                          │  │
 │  │     │ uid: AID...  │                                          │  │
 │  │     └──────────────┘                                          │  │
 │  │           │                                                  │  │
 │  │           ├── Group: Developers (그룹에 가입)                 │  │
 │  │           └── Group: Admins                                    │  │
 │  │                                                                │  │
 │  │  ② Policy: 권한 정의 (JSON 문서)                             │  │
 │  │     ┌──────────────────────────────────────────────────────┐   │  │
 │  │     │ {                                                   │   │  │
 │  │     │   "Version": "2012-10-17",                          │   │  │
 │  │     │   "Statement": [{                                    │   │  │
 │  │     │     "Effect": "Allow",                               │   │  │
 │  │     │     "Action": ["s3:GetObject", "s3:PutObject"],      │   │  │
 │  │     │     "Resource": "arn:aws:s3:::my-bucket/*"           │   │  │
 │  │     │   }]                                                  │   │  │
 │  │     │ }                                                    │   │  │
 │  │     └──────────────────────────────────────────────────────┘   │  │
 │  │                                                                │  │
 │  │  ③ Role: (User + Policy)의 조합                             │  │
 │  │     ┌──────────────────────────────────────────────────────┐   │  │
 │  │     │ Role: EC2-Read-Only                                  │   │  │
 │  │     │   ├── Policy: ReadOnlyAccess                         │   │  │
 │  │     │   └── Trust: ec2.amazonaws.com (EC2가 이 Role 가정)  │   │  │
 │  │     └──────────────────────────────────────────────────────┘   │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [권한 부여 흐름]                                                    │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   alice                                                     │  │
 │  │     │                                                        │  │
 │  │     ├── Group: Developers ──▶ Policy: DeveloperPolicy       │  │
 │  │     │                          └── s3:PutObject,             │  │
 │  │     │                             ec2:DescribeInstances     │  │
 │  │     │                                                        │  │
 │  │     └── Inline Policy: {"S3-Full-Admin"}                    │  │
 │  │                           └── s3:*                         │  │
 │  │                                                                │  │
 │  │   유효 권한 = DeveloperPolicy ∪ S3-Full-Admin                │  │
 │  │   (모든 Policy의 허용(Allow) 합집합)                         │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [역할 가정(Role Assumption) — 교차 계정 접근]                     │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   계정 A (Prod) ──────────────▶ 계정 B (Dev)                   │  │
 │  │                                │                              │  │
 │  │                         STS.AssumeRole                          │  │
 │  │                                │                              │  │
 │  │                         ┌───────────────┐                      │  │
 │  │                         │ Role: CrossAcct│                      │  │
 │  │                         │   Trust: A     │                      │  │
 │  │                         │   Policy: Read │                      │  │
 │  │                         └───────────────┘                      │  │
 │  │                                │                              │  │
 │  │                         임시 자격 증명                          │  │
 │  │                         (AccessKey + SessionToken)              │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** AWS IAM의 Policy는 **{Effect, Action, Resource}** 트리플로 정의된다. `Effect: Allow`는 허용, `Deny`는 명시적 거부(Allow보다 우선)를 의미한다. 사용자는 여러 Policy를 동시에 부여받을 수 있으며, 유효 권한은 **모든 Allow Policy의 합집합**에서 **모든 Deny Policy의 차집합**을 뺀 것이다. **Role Assumption**은 AWS RBAC의 강력한 기능으로서, 계정 A의 사용자가 계정 B의 Role을 **임시로 가정(Assume)**하여 교차 계정 접근을 가능하게 한다. 이는 **"최소 권한 + 일시적 권한 상승"**의 원칙을 구현한다.

### Kubernetes RBAC — Role, RoleBinding, ServiceAccount

쿠버네티스의 RBAC는 **{Role, ClusterRole, RoleBinding, ClusterRoleBinding}** 4가지 리소스로 구현된다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │            Kubernetes RBAC — Role, RoleBinding 구조                  │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [RBAC 리소스]                                                      │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ① Role / ClusterRole: 권한 정의                             │  │
 │  │     (Role=네임스페이스 범위, ClusterRole=클러스터 전체)      │  │
 │  │                                                                │  │
 │  │     ┌──────────────────────────────────────────────────────┐   │  │
 │  │     │ apiVersion: rbac.authorization.k8s.io/v1            │   │  │
 │  │     │ kind: Role                                           │   │  │
 │  │     │ metadata:                                            │   │  │
 │  │     │   namespace: default                                 │   │  │
 │  │     │ rules:                                               │   │  │
 │  │     │ - apiGroups: [""] (core)                             │   │  │
 │  │     │   resources: ["pods", "services"]                    │   │  │
 │  │     │   verbs: ["get", "list", "watch"]                    │   │  │
 │  │     │ - apiGroups: ["apps"]                                │   │  │
 │  │     │   resources: ["deployments"]                          │   │  │
 │  │     │   verbs: ["get", "list"]                             │   │  │
 │  │     └──────────────────────────────────────────────────────┘   │  │
 │  │                                                                │  │
 │  │  ② RoleBinding / ClusterRoleBinding: 사용자 → 역할 매핑     │  │
 │  │                                                                │  │
 │  │     ┌──────────────────────────────────────────────────────┐   │  │
 │  │     │ apiVersion: rbac.authorization.k8s.io/v1            │   │  │
 │  │     │ kind: RoleBinding                                    │   │  │
 │  │     │ subjects:                                            │   │  │
 │  │     │ - kind: User                                         │   │  │
 │  │     │   name: alice                                        │   │  │
 │  │     │ - kind: ServiceAccount                               │   │  │
 │  │     │   name: app-sa                                       │   │  │
 │  │     │ roleRef:                                             │   │  │
 │  │     │   kind: Role                                         │   │  │
 │  │     │   name: pod-reader                                   │   │  │
 │  │     └──────────────────────────────────────────────────────┘   │  │
 │  │                                                                │  │
 │  │  ③ ServiceAccount: 파드의 인증 주체                         │  │
 │  │                                                                │  │
 │  │     ┌──────────────────────────────────────────────────────┐   │  │
 │  │     │ apiVersion: v1                                        │   │  │
 │  │     │ kind: ServiceAccount                                 │   │  │
 │  │     │ metadata:                                            │   │  │
 │  │     │   name: app-sa                                        │   │  │
 │  │     │ secrets: [{name: app-sa-token-xxxxx}]                │   │  │
 │  │     └──────────────────────────────────────────────────────┘   │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [권한 검사 흐름]                                                    │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   alice 사용자가 kubectl get pods 요청                         │  │
 │  │                                                                │  │
 │  │   ① API Server는 인증 플러그인으로 alice의 신원 확인           │  │
 │  │      (Client Cert, OIDC Token, etc.)                         │  │
 │  │                                                                │  │
 │  │   ② Authorization 플로그인(RoleBindings) 순회                  │  │
 │  │      ┌──────────────────────────────────────────────────────┐ │  │
 │  │      │ RoleBinding #1: alice → pod-reader Role             │ │  │
 │  │      │ RoleBinding #2: alice → deployment-editor Role      │ │  │
 │  │      └──────────────────────────────────────────────────────┘ │  │
 │  │                                                                │  │
 │  │   ③ Role의 rules에서 "pods", "get" 확인                       │  │
 │  │      → 매칭: Allow!                                          │  │
 │  │                                                                │  │
 │  │   ④ API Server은 get pods 요청 수행                           │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 쿠버네티스 RBAC는 **"Namespace 범위의 Role/RoleBinding"**과 **"클러스터 전체의 ClusterRole/ClusterRoleBinding"**으로 분리된다. 예를 들어 `default` 네임스페이스의 `pod-reader` Role은 같은 네임스페이스의 Pod만 조회할 수 있지만, `cluster-admin` ClusterRole은 모든 네임스페이스의 모든 리소스에 무제한 접근이 가능하다. ServiceAccount는 **"파드의 인증 주체"**로서, 파드가 실행될 때 `/var/run/secrets/kubernetes.io/serviceaccount/token`에 JWT 토큰이 마운트되며, 이 토큰으로 API Server에 인증한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: RBAC vs ABAC (Attribute-Based Access Control)

| 비교 항목 | RBAC | ABAC |
|:---|:---|:---|
| **권한 기준** | 역할(Role) | 속성(Attribute) |
| **결정 방식** | 사전 정의된 역할에 매핑 | 동적 정책 평가 (if-then-else) |
| **확장성** | 중간(역할 추가 필요) | 높음(속성 조합으로 자동 확장) |
| **관리 복잡도** | 낮음(역할 중심) | 높음(정책 작성 복잡) |
| **성능** | 빠름(역할 검색) | 느림(정책 엔진 평가) |
| **대표 예시** | AWS IAM, K8s RBAC | XACML, AWS ABAC (태그 기반) |
| **사용 사례** | 정형화된 조직 구조 | 복잡한 컨텍스트 기반 제어 |

RBAC는 **"역할이라는 사전 정의된 틀"**에 사용자를 맞추는 방식이라서, 새로운 권한 요구가 생길 때마다 새 역할을 정의해야 한다. ABAC(Attribute-Based Access Control)는 **"사용자의 속성(부서, 직급, 시간, 위치) + 리소스의 속성(기밀 등급)"**을 동적으로 평가하여, "영업팀 + 정규직 근무 시간 + 본사 사무실" 같은 복잡한 조건을 처리할 수 있다. 그러나 ABAC는 정책 엔진이 모든 요청마다 복잡한 규칙을 평가해야 하므로 성능 저하가 발생할 수 있다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │            RBAC vs ABAC: 정책 표현 비교                              │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [RBAC 정책]                                                         │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  IF role == "Engineer" THEN                                    │  │
 │  │    ALLOW {code:read, code:write, deploy:execute}             │  │
 │  │  ELSE IF role == "Manager" THEN                                │  │
 │  │    ALLOW {code:read, approve:execute}                         │  │
 │  │                                                                │  │
 │  │  ✅ 단순 명료, 이해하기 쉬움                                   │  │
 │  │  ❌ "Engineer이지만 퇴사한 직원은?" → 별도 처리 필요         │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [ABAC 정책]                                                         │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  IF user.department == "Engineering" AND                      │  │
 │  │     user.status == "active" AND                               │  │
 │  │     resource.confidentiality <= "secret" AND                  │  │
 │  │     time.hour IN [9, 10, 11, 12, 13, 14, 15, 16, 17] AND      │  │
 │  │     user.location == "office"                                 │  │
 │  │  THEN                                                          │  │
 │  │    ALLOW {read, write}                                         │  │
 │  │                                                                │  │
 │  │  ✅ "Engineering + 재직 중 + 비밀 이하 + 근무 시간 + 본사"  │  │
 │  │     같은 복잡한 조건도 표현 가능                               │  │
 │  │  ❌ 정책이 너무 복잡하여 디버깅이 어려움                       │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 실무에서는 **"RBAC + ABAC 하이브리드"**를 많이 사용한다. AWS에서는 RBAC(IAM Policy)로 기본 권한을 부여하고, ABAC(태그 기반 정책)으로 세밀한 제어를 추가한다. 예를 들어 **"모든 Engineer는 S3를 읽을 수 있지만(Cost Center 태그가 있는 버킷은 제외)"** 같은 정책을, `aws:PrincipalTag`와 `s3:ResourceTag` 조건으로 구현할 수 있다.

### 비교 2: 계층적 RBAC(Hierarchical) vs 플랫 RBAC(Flat)

| 비교 항목 | 계층적 RBAC | 플랫 RBAC |
|:---|:---|:---|
| **역할 관계** | 상위 역할이 하위 권한 상속 | 역할 간 독립, 상속 없음 |
| **예시** | Manager ⊃ Engineer ⊃ Intern | Manager, Engineer, Engineer 간 독립 |
| **권한 계산** | 상속 트리 순회 | 단순 매핑 |
| **관리 용이성** | 중소規模에 적합 | 대규모에 적합 |
| **대표 구현** | AWS IAM Policy Group(변수), Solaris RBAC | Kubernetes RBAC, Azure RBAC |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — AWS IAM 역할 분리(AWS Account Separation)**: 기업에서 프로덕션과 개발 환경을 별도 AWS 계정으로 분리하되, 개발자가 **"개발 계정의 전체 권한 + 프로덕션 계정의 읽기 전용 권한"**을 갖도록 설계했다. 이를 위해 **"교차 계정 Role Assumption"**을 사용한다. ① 프로덕션 계정에 `ReadOnly` Role을 생성하고, Trust Relationship을 개발 계정의 AWS Account ID로 설정한다. ② 개발자는 `sts:AssumeRole`로 프로덕션 계정의 Role을 가정하여, 임시 자격 증명(AccessKey + SessionToken + Expiration)을 획득한다. ③ 이 임시 자격으로 프로덕션 리소스에 **읽기 전용**으로 접근한다.

2. **시나리오 — Kubernetes RBAC로 권한 최소화**: 쿠버네티스 클러스터에서 개발팀에게 **"자신네 네임스페이스의 파드는 전체 권한, 타 네임스페이스는 조회 불가"**라는 요구가 있었다. 이를 위해 ① 각 팀 네임스페이스(`team-a`, `team-b`) 생성 ② `team-admin` Role에 `pods, services, deployments`에 대한 모든 권한(`*`) 부여 ③ RoleBinding으로 `team-a` 네임스페이스의 `team-admin` Role을 팀 A ServiceAccount에만 바인딩했다. ④ ClusterRole `cluster-reader`를 생성하고 ClusterRoleBinding으로 모든 ServiceAccount에 조회 권한만 부여하여, **"타 네임스페이스는 전체 조회 가능, 자신 네임스페이스는 수정 가능"**이라는 제한을 구현했다.

3. **시나리오 — 책임 분리(SoD) 정책 적용**: 금융권 시스템에서 **"요청자와 승인자는 동일인이 될 수 없다"**는 SoD(책임 분리) 정책을 구현해야 했다. RBAC2의 **"상호 배역(Mutually Exclusive Roles)"** 제약을 사용하여, `Requester`와 `Approver` Role을 동시에 부여받은 사용자가 없도록 설정했다. 또한 Role 부여 요청 시 **"이미 Requester를 보유한 사용자에게 Approver 부여 불가"**라는 검증 로직을 추가하여, 자가 승인을 시스템 레벨에서 방지했다.

### 도입 체크리스트
- **기술적**: 모든 사용자에게 **"최소 권한의 역할"**만 부여되고 있는가? `aws iam list-attached-role-policies`나 `kubectl get rolebindings --all-namespaces`로 불필요한 권한을 점검하는가?
- **운영·보안적**: 역할 생성/변경/삭제 시 **"승인 프로세스와 검토(Review)"**가 있는가? 정기적으로(분기 1회) 미사용 역할을 식별하고 삭제하는가?

### 안티패턴
- **과도한 관리자 역할 부여**: `AdministratorAccess`(AWS), `cluster-admin`(K8s) 같은 최고 권한 역할을 너무 많은 사용자에게 부여하면, **"단일 실패점(Single Point of Failure)"**이 되어 계정 탈취 시 치명적이다. 정책에 따라 **"읽기 전용 관리자", "백업 전용 관리자"** 같은 세분화된 역할을 정의해야 한다.
- **역할 폭주(Role Explosion)**: 모든 권한 조합마다 별도 역할을 만들면(예: `S3-Read-RW-Only`, `S3-Read-Write-Delete`, ...), 수백 개의 역할이 생성되어 관리가 불가능해진다. **"공통 역할(Base Role) + 추가 정책(Inline Policy)"** 조합을 사용하여 역할 수를 통제해야 한다.

## 👶 어린이를 위한 3줄 비유 설명
1. RBAC는 **"학교의 반과 역할"**이에요. 반장은 교무실에 들어갈 수 있고, 도서위원은 도서관 책을 정리할 수 있어요.
2. 새로 전학 온 친구는 "1반 반장"이라는 역할을 받으면, 반장이 할 수 있는 모든 것을 할 수 있어요. 일일이 "너는 출석부를 쓸 수 있어, 너는 청소 도구를 쓸 수 있어"라고 설명할 필요가 없어요.
3. 그래서 학교에 수백 명의 학생이 있어도, "반장, 부반장, 위원장" 몇 가지 역할만 있으면 모두에게 알맞은 일을分配할 수 있어요!
