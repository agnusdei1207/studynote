## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 접근 제어 행렬은 시스템의 모든 주체(Subject)와 객체(Object) 사이의 허용된 연산(Operation)을 **행렬(Matrix) 형태**로 표현하는 보호 모델로서, "누가(Who) 무엇을(What) 어떻게(How) 할 수 있는가"를 수학적으로 정의한다.
> 2. **가치**: 행렬의 희소성(Sparsity) 문제를 해결하기 위해 ACL(객체 중심)과 Capability List(주체 중심) 두 가지 실용적 구현 방식이 탄생했으며, 현대 OS의 파일 권한(rwx), SELinux 정책, Kubernetes RBAC는 모두 이 행렬의 변형이다.
> 3. **융합**: 데이터베이스의 GRANT/REVOKE, 분산 시스템의 Kerberos 티켓, 블록체인의 스마트 컨트랙트 접근 제어는 모두 접근 제어 행렬의 다양한 구현체로서, 자원 보호의 보편적 언어가 되었다.

## Ⅰ. 개요 (Context & Background)

- **개념**: 접근 제어 행렬(Access Matrix)은 1971년 Butler Lampson이 제안한 보호 모델로서, **행(Row)에 주체(Subject: 프로세스, 사용자)를, 열(Column)에 객체(Object: 파일, 메모리, 장치)을 배치하고, 교차점 셀(Cell)에 허용된 연산(Operation)을 기록하는 직사각형 행렬**이다. 수학적으로는 함수 M: S × O → P(S는 주체 집합, O는 객체 집합, P는 권한(Power) 집합)으로 정의된다.

- **💡 비유**: 접근 제어 행렬은 **"학교의 교실 출입 허가표"**다. 행에는 학생(주체)이, 열에는 교실(객체)이 있으며, 각 칸에는 "📖읽기만, ✍️쓰기가능, 🚫출입금지" 같은 출입 권한이 표시되어 있다. 1반 학생은 "과학실 ✍️쓰기가능" 칸을 보고 과학실 실험 도구를 사용할 수 있지만, "교무실 🚫출입금지" 칸을 보고 선생님 허락 없이 교무실에 들어갈 수 없음을 알게 된다.

- **등장 배경**:
  1. **MULTICS 프로젝트(1960s-70s)**: MULTICS는 최초로 링 기반 보호와 접근 제어 행렬을 구현하여, 프로세스가 자신의 도메인 밖에 있는 자원에 접근하는 것을 체계적으로 차단했다.
  2. **UNIX의 단순화(1970s)**: UNIX는 행렬의 각 셀을 3비트(rwx)로 압축하고, 주체를 소유자/그룹/기타(Other) 3개 범주로 축소하여 실용적인 구현을 선택했다. 이것이 오늘날 `ls -l`에 표시되는 `-rwxr-xr--` 형태다.
  3. **세분화된 현대 시스템(1990s-현재)**: SELinux, AppArmor, Kubernetes RBAC은 다시 원래의 행렬 형태로 복귀하여, 주체와 객체의 레이블 조합으로 수천~수만 개의 셀을 정의한다.

접근 제어 행렬의 기본 구조를 시각화하면 다음과 같다. 주체(User A, B, C)가 객체(File1, File2, Printer)에 대해 어떤 연산(Read, Write, Execute, Own)을 수행할 수 있는지가 표현된다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                    접근 제어 행렬 (Access Matrix)                    │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │      ◀─ 주체 (Subject: 사용자, 프로세스)                            │
 │                                                                      │
 │      ┌──────────────────────────────────────────────────────────┐    │
 │      │              File1  │  File2  │ Printer │  MemSeg1 │     │    │
 │      ├──────────────────────────────────────────────────────────┤    │
 │ User A│  Own, R, W │   R     │    W    │    R, W  │     ▲     │    │
 │ User B│    R      │ Own, W  │    —    │    —    │     │     │    │
 │ User C│    R      │    R    │    W    │    —    │     │     │    │
 │      └──────────────────────────────────────────────────────────┘    │
 │                                                                      │
 │                        ▼                                            │
 │                  객체 (Object: 파일, 장치)                           │
 │                                                                      │
 │  연산(Operation)의 종류:                                             │
 │  • Own: 소유자 권한 (파일 삭제, 권한 변경 가능)                      │
 │  • R (Read): 읽기                                                   │
 │  • W (Write): 쓰기                                                  │
 │  • — (Empty): 접근 불가                                             │
 │                                                                      │
 │  예시:                                                               │
 │  • User A는 File1에 대해 Own, R, W를 가지므로 읽기, 쓰기, 삭제 가능  │
 │  • User B는 File2에만 쓸 수 있고, File1은 읽기만 가능                │
 │  • User C는 File1, File2을 읽을 수 있지만 쓸 수 없음                  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 행렬은 **"보호의 청사진"**이다. 어떤 보호 시스템이든 이 행렬의 **부분 집합**을 구현한 것에 불과하다. 예를 들어 UNIX는 각 파일 inode에 (Owner, Group, Other) × (r, w, x) = 9비트를 저장하여, 행렬의 각 열(Column)마다 3개의 행(Row)만을 명시적으로 저장한다. 반면 SELinux는 (Subject Domain, Object Class, Object Type, Permission)의 4중 조합으로 정책을 저장하여, 수천만 개의 셀을 표현할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **주체(Subject)** | 접근을 요청하는 능동 개체 | 프로세스, 스레드, 사용자 | UID, PID, SELinux Domain | 출입을 요청하는 사람 |
| **객체(Object)** | 접근의 대상이 되는 수동 개체 | 파일, 메모리, 장치, IPC 객체 | inode, 메모리 페이지, 장치 파일 | 건물/방/금고 |
| **연산(Operation)** | 객체에 대해 수행 가능한 동작 | Read, Write, Execute, Delete, Append | 시스템 콸번호, 파일 모드 비트 | 허용된 행동 목록 |
| **권한(Power)** | 주체-객체 쌍에 부여된 연산 집합 | 집합 P = {R, W, X, ...} | POSIX Capability, Permission 비트 | 출입증에 찍힌 허가 사항 |
| **전이 규칙(Transition Rule)** | 도메인 간 이동 시 권한 변경 규칙 | 접근 제어 행렬 확장(도메인도 객체) | 시스템 콜, Gate Descriptor | 승급/강극 절차 |

### 희소성 문제와 구현 방식: ACL vs Capability

현실적인 시스템에서는 주체(사용자 수천~수백만 명) × 객체(파일 수십억 개)의 행렬이 거의 비어있는 **희소 행렬(Sparse Matrix)**이 된다. 이를 효율적으로 저장하기 위해 두 가지 실용적 구현 방식이 탄생했다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │            희소성 문제와 구현 방식: ACL vs Capability                 │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [문제: 원본 행렬은 매우 희소(Sparse)]                               │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │          File1   File2   File3   File4   ...   FileN         │  │
 │  │  User1   R,W     —       —       —              —             │  │
 │  │  User2   R       R,W     —       —              —             │  │
 │  │  User3   —       —       R,W     —              —             │  │
 │  │  User4   —       —       —       R,W            —             │  │
 │  │  ...                                                      │  │
 │  │  UserN   —       —       —       —              R,W           │  │
 │  │                                                                │  │
 │  │  • N×N 행렬에서 O(N²) 저장 공간 필요                           │  │
 │  │  • 대부분의 셀이 비어있어 메모리 낭비                          │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [해결책 1: ACL (Access Control List) — 객체 중심]                   │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  각 객체가 "나를 접근할 수 있는 주체 리스트"를 보관            │  │
 │  │                                                                │  │
 │  │  File1: {(User1, R,W), (User2, R)}                            │  │
 │  │  File2: {(User2, R,W)}                                        │  │
 │  │  File3: {(User3, R,W)}                                        │  │
 │  │  File4: {(User4, R,W)}                                        │  │
 │  │  FileN: {(UserN, R,W)}                                        │  │
 │  │                                                                │  │
 │  │  • 저장: O(N) — 각 객체의 참조자만 저장                       │  │
 │  │  • 검색: "User1이 File1에 접근 가능?" → File1의 ACL 순회 O(K) │  │
 │  │  • 리눅스 파일 시스템 inode에 구현                             │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [해결책 2: Capability List — 주체 중심]                            │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  각 주체가 "내가 접근 가능한 객체 리스트"를 보관               │  │
 │  │                                                                │  │
 │  │  User1: {(File1, R,W)}                                        │  │
 │  │  User2: {(File1, R), (File2, R,W)}                            │  │
 │  │  User3: {(File3, R,W)}                                        │  │
 │  │  User4: {(File4, R,W)}                                        │  │
 │  │  UserN: {(FileN, R,W)}                                        │  │
 │  │                                                                │  │
 │  │  • 저장: O(N) — 각 주체의 권한만 저장                         │  │
 │  │  • 검색: "User1이 File1에 접근?" → User1의 C-List 순회 O(K)   │  │
 │  │  • Kerberos 티켓, POSIX Capabilities                          │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  ACL는 "방 벽에 붙은 출입명단", Capability는 "지갑 속 VIP 카드"    │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ACL(Access Control List)은 각 객체가 자신의 접근 제어 리스트를 보관한다. 리눅스 파일 시스템의 `inode->i_acl` 필드에 `{uid, perms}` 쌍의 연결 리스트가 저장된다. 검사할 때는 VFS가 "이 파일의 ACL을 순회하며 현재 프로세스의 UID가 있는지 확인"한다. Capability List는 각 주체가 자신의 권한 리스트(보통 커널 공간의 자료구조)를 보관한다. Kerberos 티켓이나 POSIX Capabilities가 이 모델을 따른다. "권한을 가진 자가 증명(Possession)"하기 때문에, 티켓을 탈취하면 권한을 도용당할 수 있다는 단점이 있다.

### 도메인 전이와 행렬의 확장

접근 제어 행렬을 도메인(보호 영역)까지 포함하도록 확장하면, **"주체가 도메인 간 전이(Switch)할 수 있는지"**까지도 행렬의 일부로 표현할 수 있다. 이때 도메인은 주체이자 객체가 되는 이중 성격을 갖는다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │               도메인 전이를 포함한 확장 접근 제어 행렬                │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │      ┌──────────────┬──────────────┬──────────────┬──────────────┐  │
 │      │  Domain D1   │  Domain D2   │   File F1    │   File F2    │  │
 │ ├──────────────┼──────────────┼──────────────┼──────────────┤  │
 │ │Domain D1│     —       │   switch     │     R        │     —        │  │
 │ ├──────────────┼──────────────┼──────────────┼──────────────┤  │
 │ │Domain D2│   switch     │     —       │     W        │   R, W      │  │
 │ ├──────────────┼──────────────┼──────────────┼──────────────┤  │
 │ │Domain D3│     —       │     —       │     —        │     —        │  │
 │ └──────────────┴──────────────┴──────────────┴──────────────┘  │
 │                                                                      │
 │  • D1에서 D2로 전이 가능(switch) — D1 프로세스가 시스템 콜로 D2 진입 │  │
 │  • D2에서 D1로도 전이 가능 — 양방향 전이                           │  │
 │  • D1 도메인에선 F1만 읽기(R) 가능                                 │  │
 │  • D2 도메인에선 F1에 쓰기(W), F2에 읽기/쓰기(R,W) 가능             │  │
 │  • D3는 어느 도메인으로도 전이 불가 — 격리된 샌드박스               │  │
 │                                                                      │
 │  예시: 웹 서버(D1) → 커널(D2) → 파일시스템(D2) → DB 소켓(D3)        │  │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  1. 웹 서버 프로세스는 D1(Ring 3)에서 시작                    │  │
 │  │  2. 시스템 콜 호출 → D2(Ring 0)로 switch                       │  │
 │  │  3. D2에서 F1(/var/www/html)에 쓰기 가능                       │  │
 │  │  4. D2에서 D3(DB 도메인)로 switch 불가 — DB는 별도 프로세스    │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 확장 행렬은 **"도메인 전이 제어"**까지 포함한다. 예를 들어 Ring 3(사용자 모드)에서 Ring 0(커널 모드)로 전이하려면, 행렬의 `(D1, D2)` 셀에 `switch` 권한이 있어야 한다. 이는 실제로는 CPU의 Call Gate나 시스템 콜 인터페이스로 구현된다. UNIX의 `setuid()` 시스템 콜은 "현재 도메인에서 target UID 도메인으로 전이"하는 것으로, 행렬의 `(현재_UID, target_UID)` 셀에 `CAP_SETUID` capability가 있어야 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: ACL vs Capability List

| 비교 항목 | ACL (Access Control List) | Capability List |
|:---|:---|:---|
| **저장 위치** | 객체의 메타데이터(inode) | 주체의 자격(Credentials) |
| **검사 주체** | 객체 소유자/커널이 관리 | 주체가 소지하고 증명 |
| **권한 취소(Revocation)** | 간단(ACL에서 제거) | 어려움(모든 티켓 찾아야 함) |
| **위변조 방지** | 커널이 보호 | 암호학적 서명 필요 |
| **전파(Delegation)** | 불가능(소유자만 변경) | 가능(티켓을 타인에게 양도) |
| **대표 예시** | 리눅스 파일 권한(rwx), NFS ACL | Kerberos 티켓, POSIX Capabilities |
| **검사 오버헤드** | 접근 시마다 ACL 순회 | 티켓 검증만으로 빠름 |

ACL은 객체 중심이므로 "이 파일을 누가 접근 가능한가?"를 확인하기 쉽고, 권한을 취소하려면 ACL에서 해당 항목을 삭제하면 된다. 반면 Capability는 주체 중심이므로 "나는 무엇을 할 수 있는가?"를 빠르게 확인할 수 있고, 티켓을 타인에게 양도(Delegation)할 수 있다. 그러나 권한을 취소하려면 발급된 모든 티켓을 찾아 무효화해야 하므로 복잡하다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                   ACL vs Capability: 검사 흐름 비교                    │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [ACL 기반 검사 — 객체 중심]                                         │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  프로세스(UserA)가 파일(File1)을 open() 시도                   │  │
 │  │                                                                │  │
 │  │  ① VFS: inode->i_acl 로드                                      │  │
 │  │  ┌──────────────────────────────────────────────────────────┐  │  │
 │  │  │  ACL Entry 1: {uid: UserA, perms: R}                     │  │  │
 │  │  │  ACL Entry 2: {uid: UserB, perms: R,W}                   │  │  │
 │  │  │  ACL Entry 3: {gid: Staff, perms: R}                     │  │  │
 │  │  └──────────────────────────────────────────────────────────┘  │  │
 │  │                                                                │  │
 │  │  ② 현재 프로세스의 UID(UserA)와 GID를 ACL과 비교              │  │
 │  │  ③ Entry 1 매칭 → perms에 W 없음 → -EACCES 반환              │  │
 │  │                                                                │  │
 │  │  ✅ 객체(파일)이 자신의 보호 규칙을 스스로 정의                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [Capability 기반 검사 — 주체 중심]                                 │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  프로세스(UserA)가 파일(File1)에 접근 시도                    │  │
 │  │                                                                │  │
 │  │  ① 프로세스의 Credential 로드                                 │  │
 │  │  ┌──────────────────────────────────────────────────────────┐  │  │
 │  │  │  current->cred:                                          │  │  │
 │  │  │    uid: UserA                                            │  │  │
 │  │  │    capabilities: {CAP_DAC_OVERRIDE, CAP_NET_BIND_SERVICE}│  │  │
 │  │  │    Capability List:                                      │  │  │
 │  │  │      [{obj: File1, perms: R}, {obj: File2, perms: R,W}]  │  │  │
 │  │  └──────────────────────────────────────────────────────────┘  │  │
 │  │                                                                │  │
 │  │  ② Capability List를 검색: File1에 대한 항목 확인             │  │
 │  │  ③ perms: R (쓰기 권한 없음) → -EACCES 반환                 │  │
 │  │                                                                │  │
 │  │  ✅ 주체(프로세스)가 자신의 권한护照를 지참                   │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ACL 기반 시스템에서는 커널이 객체의 ACL을 로드하여 현재 프로세스의 신원과 비교한다. Capability 기반 시스템에서는 프로세스가 자신의 Capability List(보통 커널 공간에 보호된 자료구조)를 제시하고, 커널은 이를 검증만 한다. **ACL는 "명부(Guest List)에 이름이 있는가?"** 방식이고, Capability는 **"티켓을 가지고 있는가?"** 방식이다.

### 비교 2: UNIX Permission vs SELinux Policy

| 비교 항목 | UNIX Permission (DAC) | SELinux Policy (MAC) |
|:---|:---|:---|
| **행렬 차원** | 3×3 (Owner/Group/Other × rwx) | 무한 (Domain×Type×Class×Perm) |
| **정의 위치** | inode의 mode_t 비트 | 커널 정책 DB(/etc/selinux/policy) |
| **권한 변경** | `chmod`, `chown`으로 소유자가 변경 | 보안 관리자만 정책 수정 |
| **검사 순서** | DAC만 수행 | DAC → MAC 순차 검사 |
| **트로이 목마 방어** | 불가(소유자 권한으로 동작) | 가능(도메인 강제로 차단) |
| **복잡도** | 단순(9비트) | 매우 복잡(수백만 규칙) |

UNIX permission은 접근 제어 행렬의 극도로 단순화된 형태로서, 각 객체(파일)마다 (Owner, Group, Other) × (Read, Write, Execute) = 9비트만 저장한다. 이는 효율적이지만 너무 조잡하여, "그룹에 속한 모든 사용자가 동일한 권한"이라는 제약이 있다. SELinux는 행렬의 원래 형태로 복귀하여, `(httpd_domain, httpd_content_type, file, read)` 같은 4중 조합으로 수백만 개의 셀을 정의할 수 있다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │          UNIX Permission vs SELinux: 행렬 표현 비교                  │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [UNIX Permission — 3×3 행렬의 압축]                                 │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  각 파일 inode에 16비트 mode_t 저장                             │  │
 │  │                                                                │  │
 │  │   15 12 11   8 7   4 3   0                                    │  │
 │  │   ┌──┬──┬──┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┐                      │  │
 │  │   │S│G│U│ │r│w│x│r│w│x│r│w│x│                           │  │  │
 │  │   └──┴──┴──┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┘                      │  │
 │  │     │  │  │  └─┬─┘ └─┬─┘ └─┬─┘                               │  │
 │  │     │  │  │    │     │     │                                 │  │
 │  │  SetGID│SetUID Other Group Owner                             │  │
 │  │                                                                │  │
 │  │  실제 행렬:                                                     │  │
 │  │         Owner   Group   Other                                 │  │
 │  │  File   rwx     r-x     r--                                   │  │
 │  │                                                                │  │
 │  │  ✅ 단순하지만 "Staff 그룹의 Alice와 Bob이 서로 다른 권한"    │  │
 │  │     같은 세분화 불가                                           │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [SELinux Policy — 무한 차원 행렬]                                 │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  (Subject Domain, Object Class, Object Type) → Permissions    │  │
 │  │                                                                │  │
 │  │  예시 규칙:                                                     │  │
 │  │  { httpd_t, file, httpd_sys_content_t } → { read, open }      │  │
 │  │  { httpd_t, file, shadow_t } → { } (빈 집합 = 모두 거부)       │  │
 │  │  { sysadm_t, file, etc_t } → { read, write, unlink }          │  │
 │  │                                                                │  │
 │  │  AVC(Access Vector Cache)로 빠른 검색:                         │  │
 │  │  ┌─────────────────────────────────────────────────────────┐  │  │
 │  │  │  Key: {httpd_t, file, shadow_t}                         │  │  │
 │  │  │  Value: DENY (캐시되어 있어 정책 DB 재검색 불필요)        │  │  │
 │  │  └─────────────────────────────────────────────────────────┘  │  │
 │  │                                                                │  │
 │  │  ✅ "httpd_t 도메인은 shadow_t 파일에 접근 불가" 같은 미세한   │  │
 │  │     제어 가능, 정책 수백만 개 규칙 가능                       │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** UNIX의 9비트 권한은 간단하지만, "Staff 그룹의 Alice는 파일 A에 쓰기, Bob은 읽기만" 같은 세분화된 제어가 불가능하다. SELinux는 **"주체의 보안 컨텍스트(Domain)"**과 **"객체의 보안 컨텍스트(Type)"**을 분리하여, 이 둘 사이의 허용 관계를 정책 데이터베이스에 저장한다. AVC(Access Vector Cache)는 이 정책 검색 결과를 캐싱하여, 매번 정책 DB를 조회하는 오버헤드를 줄인다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — NFS ACL와 POSIX ACL의 통합**: 리눅스 서버에서 `setfacl` 명령으로 `/data/project` 디렉터리에 "alice:rw-, bob:r--, group:staff:rwx" 같은 세밀한 ACL을 설정했다. 이를 NFS로 공유하려면 NFSv4 ACL로 변환해야 하는데, 두 ACL 형식의 미묘한 차이(ACE 순서, 상속 플래그)로 인해 권한 누수가 발생할 수 있다. 해결책은 `nfs4_setfacl`을 사용하여 NFSv4 ACL을 직접 설정하고, `getfacl --access`로 정확히 매핑되는지 검증하는 것이다.

2. **시나리오 — Kubernetes RBAC 설계**: 쿠버네티스 클러스터에서 개발자들에게 "자신네 네임스페이스의 파드만 조회/삭제" 권한을 부여해야 한다. Role 리소스에 `rules: [{apiGroups: [""], resources: ["pods"], verbs: ["get", "list", "delete"]}]`을 정의하고, RoleBinding으로 이를 개발자 그룹에 바인딩했다. 이것이 접근 제어 행렬의 (Subject, Resource, Verb) 관계를 선언적으로 정의한 것이다.

3. **시나리오 — 데이터베이스의 GRANT/REVOKE**: PostgreSQL에서 `GRANT SELECT ON orders TO analyst_role;`를 실행하면, pg_catalog의 pg_class 테이블에 ACL 필드(`{analyst_role=r/postgres}`)가 업데이트된다. 이것이 접근 제어 행렬의 (analyst_role, orders, SELECT) 셀에 값을 쓴 것과 동일하다. `REVOKE`는 ACL에서 해당 항목을 삭제한다.

### 도입 체크리스트
- **기술적**: 중요한 자원에 대해 ACL이 설정되어 있으며, 특히 `chmod 777`이나 `chmod 666` 같은 과도하게 열려 있는 권한이 없는가? SELinux/AppArmor가 enforcing 모드로 활성화되어 있는가?
- **운영·보안적**: 권한 부여는 최소 권한의 원칙(Principle of Least Privilege)을 따르는가? 정기적으로(반기 1회) 불필요한 권한을 제거(Revoke)하는 절차가 있는가?

### 안티패턴
- **ACL 폭주**: 수백 명의 사용자가 단일 파일에 ACL 엔트리를 갖게 되면, 매번 접근 시마다 ACL 전체를 순회해야 하여 성능이 저하된다. 이 경우에는 그룹을 활용하여 ACL 엔트리 수를 줄여야 한다.
- **Capability 누수**: Kerberos 티켓이나 POSIX Capabilities를 탈취당하면 공격자가 피해자의 권한을 그대로 획득한다. 티켓에 짧은 만료 시간(예: 8시간)을 설정하고, 다중 인증(MFA)으로 티켓 발급을 강화해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 | 최적화 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | `chmod 777`로 모든 파일이 공개 | ACL로 필요한 사용자만 접근 | **데이터 유출 경로 95%+ 차단** |
| **정량** | 전역 capability(`CAP_SYS_ADMIN`) 부여 | 개별 capability 분할 | **탈취 시 피해 범위 80%+ 감소** |
| **정성** | "누가 무엇을 할 수 있는지" 불명확 | RBAC로 계층적 권한 관리 | **규정 준거(Compliance) 달성** |

### 미래 전망
- **ReBAC (Relationship-Based Access Control)**: Google이 제안한 ReBAC는 접근 제어 행렬을 **"그래프"**로 확장하여, "프로젝트 P의 멤버이며 역할이 R인 사용자" 같은 복잡한 관계를 표현한다. Zanzibar(Google의 전역 권한 시스템)가 이를 구현하여, Google Drive, YouTube, Cloud의 수십억 개 자원에 대한 접근 제어를 밀리초 lateny로 처리한다.
- **영지식 증명(Zero-Knowledge Proof) 기반 Capability**: 권한을 증명할 때 자신의 신원이나 권한 내용을 노출하지 않고, "이 권한을 가지고 있다"는 것만 증명하는 암호학적 방식이 연구되고 있다.

### 참고 표준
- **POSIX 1003.1e**: POSIX ACL 표준 (Draft, 구현 간 상이함)
- **NFSv4 ACL**: 네트워크 파일 시스템을 위한 통합 ACL 형식
- **RFC 5530 (Kerberos)**: Capability 기반 인증 시스템
- **OAuth 2.0 / RFC 7519 (JWT)**: 웹에서의 Capability 토큰 표준
