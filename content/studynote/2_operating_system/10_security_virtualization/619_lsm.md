## 핵심 인사이트 (3줄 요약)
> 1. **본질**: LSM(Linux Security Modules)는 리눅스 커널에 **"보안 정책을 플러그인 모듈로 동적으로 로드할 수 있는 훅(Hook) 프레임워크"**를 제공하여, SELinux, AppArmor, Smack, TOMOYO 등 다양한 MAC(Mandatory Access Control) 구현체를 커널 수정 없이 사용할 수 있게 한다.
> 2. **가치**: LSM 이전에는 MAC 정책을 추가하려면 커널 소스를 직접 패치해야 했으나, LSM 도입 후에는 **"보안 정책을 커널 모듈로 개발하여 동적으로 로드"**할 수 있게 되어, 보안 벤더들이 독자적으로 MAC 솔루션을 제공할 수 있게 되었다.
> 3. **융합**: LSM은 **"VFS, 네트워킹, IPC, 시스템 콜"** 등 커널의 핵심 경로에 훅을 삽입하여, 보안 정책 모듈이 모든 객체 접근을 중계하여 검사할 수 있게 한다. eBPF LSM으로 사용자 공간에서도 보안 정책을 개발할 수 있게 확장되고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: LSM(Linux Security Modules)는 리눅스 커널 2.6에 통합된 **"보안 정책 플러그인 프레임워크"**로서, 커널의 주요 하위 시스템(VFS, 네트워킹, IPC, 프로그램 실행 등)에 **훅(Hook) 함수**를 삽입하여, 보안 정책 모듈(SELinux, AppArmor 등)이 **"모든 객체 접근(파일 열기, 네트워크 송신, 프로세스 생성 등)을 중계하여 허가/거부"**할 수 있게 한다.

- **💡 비유**: LSM은 **"공항의 보안 검색대"**와 같다. 모든 승객(시스템 콜)은 보안 검색대(LSM 훅)를 통과해야 하며, 검색대의 보안 요원(SELinux, AppArmor)이 여권과 비행 기록을 확인하여 통과 또는 거부를 결정한다. 보안 정책이 바뀌면 검색 요원만 교체하면 되고, 공항 시설(커널)을 개조할 필요가 없다.

- **등장 배경**:
  1. **LSM 이전(1990s)**: MAC 정책을 추가하려면 커널 소스를 직접 패치해야 했으나, 이는 유지보수가 어렵고 여러 MAC 시스템을 동시에 실행할 수 없었다.
  2. **LSM 도입(2001)**: Linus Torvalds가 LSM을 커널 2.5에 머지했고, 2.6에 정식 통합되었다. SELinux, AppArmor, Smack, TOMOYO 등 다양한 MAC 모듈이 LSM 위에서 구현되었다.
  3. **LSM 확장(2020s)**: eBPF LSM이 추가되어, 사용자 공간에서도 보안 정책을 개발할 수 있게 되었고, Landlock, KRSI 등 새로운 보안 모듈이 등장했다.

LSM의 아키텍처를 시각화하면 다음과 같다. LSM 훅은 커널의 핵심 경로에 삽입되어, 모든 객체 접근을 중계한다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                   LSM 아키텍처 — 훅(Hook) 구조                       │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [LSM 훅이 삽입되는 커널 경로]                                    │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ① VFS(가상 파일 시스템) 경로                                  │  │
 │  │     open() → inode_permission() ──▶ security_inode_permission() │  │
 │  │     unlink() → security_inode_unlink()                       │  │
 │  │                                                                │  │
 │  │  ② 네트워킹 경로                                              │  │
 │  │     socket() → security_socket_create()                       │  │
 │  │     connect() → security_socket_connect()                     │  │
 │  │     bind() → security_socket_bind()                           │  │
 │  │                                                                │  │
 │  │  ③ IPC(프로세스 간 통신) 경로                                 │  │
 │  │     msg_queue_create() → security_msg_queue_create()           │  │
 │  │     shmget() → security_shm_alloc()                            │  │
 │  │                                                                │  │
 │  │  ④ 프로그램 실행 경로                                          │  │
 │  │     execve() → security_bprm_check()                           │  │
 │  │                                                                │  │
 │  │  ⑤ 기타                                                      │  │
 │  │     task_create() → security_task_create()                    │  │
 │  │     kernel_module_request() → security_kernel_module_load()   │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [LSM 훅 동작 흐름 — inode_permission 예시]                          │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  사용자 공간 프로세스                                          │  │
 │  │  ┌──────────────────────────────────────────────────────┐    │  │
 │  │  │  open("/etc/shadow", O_RDWR)                          │    │  │
 │  │  └──────────────────────────────────────────────────────┘    │  │
 │  │           │                                                   │  │
 │  │           ▼                                                   │  │
 │  │  ┌─────────────────────────────────────────────────────┐    │  │
 │  │  │  VFS: do_sys_open()                                    │    │  │
 │  │  │         │                                               │    │  │
 │  │  │         ▼                                               │    │  │
 │  │  │  ┌───────────────────────────────────────────────┐  │    │  │
 │  │  │  │  inode_permission()                                 │  │    │  │
 │  │  │  │         │                                           │  │    │  │
 │  │  │  │         ▼                                           │  │    │  │
 │  │  │  │  ┌─────────────────────────────────────────────┐ │  │    │  │
 │  │  │  │  │  DAC 검사 (UNIX permission rwx)           │ │  │    │  │
 │  │  │  │  │  • root이면 통과                                 │ │  │    │  │
 │  │  │  │  │  • alice는 소유자 W 권한 있으면 통과      │ │  │    │  │
 │  │  │  │  └─────────────────────────────────────────────┘ │  │    │  │
 │  │  │  │         │                                           │  │    │  │
 │  │  │  │         ▼                                           │  │    │  │
 │  │  │  │  ┌─────────────────────────────────────────────┐ │  │    │  │
 │  │  │  │  │  LSM 훅: security_inode_permission()      ◀───┘    │  │
 │  │  │  │  │         │                                           │  │    │  │
 │  │  │  │  │         ▼                                           │  │    │  │
 │  │  │  │  │  ┌─────────────────────────────────────────┐ │  │    │  │
 │  │  │  │  │  │ SELinux: inode_has_perm()               │ │  │    │  │
 │  │  │  │  │  │   • 보안 컨텍스트 확인                     │ │  │    │  │
 │  │  │  │  │  │   • AVC 캐시 검색                          │ │  │    │  │
 │  │  │  │  │  │   → httpd_t가 shadow_t를 읽기? DENY!       │ │  │    │  │
 │  │  │  │  │  │   audit.log에 거부 기록                     │ │  │    │  │
 │  │  │  │  │  └─────────────────────────────────────────┘ │  │    │  │
 │  │  │  │  │         │                                           │  │    │  │
 │  │  │  │  │         ▼                                           │  │    │  │
 │  │  │  │  │    ┌───────────────────────────────────────┐    │  │    │  │
 │  │  │  │  │    │ AppArmor: apparmor_inode_perm()    │    │  │    │  │
 │  │  │  │  │    │   • 경로 기반 정책 검사               │    │  │    │  │
 │  │  │  │  │    │   → /etc/shadow 접근 불가? DENY!       │    │  │    │  │
 │  │  │  │  │    └───────────────────────────────────────┘    │  │    │  │
 │  │  │  │  │         │                                           │  │    │  │
 │  │  │  │  │         ▼                                           │  │    │  │
 │  │  │  │  │      return -EACCES (거부)                           │  │    │  │
 │  │  │  │  └───────────────────────────────────────────────┘ │  │    │  │
 │  │  └───────────────────────────────────────────────────────┘    │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  ✅ LSM 훅은 DAC 검사 "후"에 실행되어 MAC를 강제               │
 │  ✅ 여러 LSM 모듈이 등록 가능하며, 등록 순서대로 호출됨       │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** LSM 훅은 **"DAC 검사 후, MAC 강제 전"**에 실행된다. 예를 들어 `open()` 시스템 콜은 먼저 VFS의 `inode_permission()`으로 UNIX permission(rwx)을 검사하고, 통과하면 LSM의 `security_inode_permission()` 훅을 호출한다. LSM 훅은 등록된 모든 LSM 모듈(SELinux, AppArmor 등)을 순차적으로 호출하며, 하나라도 거부하면 최종적으로 거부된다. 이로써 **"DAC로는 허용되지만 MAC로는 거부"**되는 상황이 가능해진다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **LSM 훅(Hook)** | 커널 경로에 삽입된 콜백 함수 | 모든 객체 접근을 중계 | security_inode_permission() 등 | 보안 검색대 |
| **LSM 콜백(LSM Callback)** | 보안 모듈이 구현하는 함수 | 허가/거부 판단 | inode_permission, file_permission | 보안 요원 |
| **LSM API** | 보안 모듈 개발용 인터페이스 | 등록, 콜백 구현 | register_security(), selinux_ops | 보안 규정 |
| **보안 컨텍스트(Security Context)** | 주체/객체의 보안 레이블 | SELinux: user:role:type:level | struct cred, inode->i_security | 여권/여권 |
| **AVC(Access Vector Cache)** | 정책 검색 결과 캐시 | 빠른 허가/거부 판단 | selinux_avc.c | 검색 기록 |

### LSM 콜백( LSM Callbacks) 종류

LSM은 커널의 다양한 경로에 훅을 제공한다. 주요 훅은 다음과 같다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │               주요 LSM 콜백(LSM Callbacks) 종류                         │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [파일 시스템 관련]                                                │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  • security_inode_permission() — inode 접근 권한 검사      │  │
 │  │  • security_inode_unlink() — 파일 삭제 권한 검사           │  │
 │  │  • security_inode_create() — inode 생성 권한 검사         │  │
 │  │  • security_file_permission() — 파일 접근 권한 검사       │  │
 │  │  • security_path_mkdir() — 디렉터리 생성 권한 검사        │  │
 │  │  • security_path_rename() — 경로 이름 변경 권한 검사       │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [네트워킹 관련]                                                  │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  • security_socket_create() — 소켓 생성 권한 검사         │  │
 │  │  • security_socket_connect() — 연결 권한 검사             │  │
 │  │  • security_socket_bind() — 바인드 권한 검사               │  │
 │  │  • security_socket_listen() — listen 권한 검사             │  │
 │  │  • security_socket_accept() — 연결 수락 권한 검사        │  │
 │  │  • security_socket_sendmsg() — 송신 권한 검사             │  │
 │  │  • security_socket_recvmsg() — 수신 권한 검사             │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [IPC 관련]                                                        │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  • security_msg_queue_create() — 메시지 큐 생성 권한 검사   │  │
 │  │  • security_shm_alloc() — 공유 메모리 할당 권한 검사     │  │
 │  │  • security_sem_alloc() — 세마포어 할당 권한 검사        │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [프로세스 관련]                                                  │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  • security_bprm_check() — execve() 실행 권한 검사        │  │
 │  │  • security_task_create() — 프로세스 생성 권한 검사       │  │
 │  │  • security_task_free() — 프로세스 소멸 시 정리          │  │
 │  │  • security_task_setuid() — setuid() 권한 검사            │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [커널 모듈 관련]                                                │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  • security_kernel_module_request() — 커널 모듈 로드 검사  │  │
 │  │  • security_kernel_read_file() — /proc/kcore 읽기 검사     │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** LSM은 **"과도하게 제어적인 보안은 시스템의 유연성을 해친다"**는 철학에 기반하여, **"필수적인 훅만 제공"**한다. 예를 들어 `security_inode_permission()`은 파일 접근을 제어하는 데 필수적이지만, `security_file_permission()`은 성능상 이유로 제공되지 않는다. LSM 모듈 개발자는 이 훅을 구현하여 보안 정책을 적용한다.

### LSM 모듈 등록 및 호출 순서

LSM은 **"등록된 모듈들의 콜백을 순차적으로 호출"**하며, 하나라도 거부하면 최종적으로 거부된다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │             LSM 모듈 등록 및 호출 순서                                   │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [모듈 등록]                                                        │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ① 커널 부팅 시 LSM 프레임워크 초기화                         │  │
 │  │                                                                 │  │
 │  │  ② LSM 모듈 로드 (예: SELinux)                                │  │
 │  │     ┌──────────────────────────────────────────────────────┐  │  │
 │  │     │  register_security(&selinux_ops);                       │  │  │
 │  │     │  // selinux_ops는 LSM 콜백 함수 포인터 배열    │  │  │
 │  │     └──────────────────────────────────────────────────────┘  │  │
 │  │                                                                 │  │
 │  │  ③ security_add_hooks()로 LSM 콜백 등록                        │  │
 │  │                                                                 │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [호출 순서]                                                        │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  사용자 요청: open("/etc/shadow", O_RDWR)                      │  │
 │  │           │                                                   │  │
 │  │           ▼                                                   │  │
 │  │  ┌─────────────────────────────────────────────────────┐    │  │
 │  │  │  DAC 검사 (UNIX permission)                             │    │  │
 │  │  │  • root이면 통과                                       │    │  │
 │  │  └─────────────────────────────────────────────────────┘    │  │
 │  │           │                                                   │  │
 │  │           ▼                                                   │  │
 │  │  ┌─────────────────────────────────────────────────────┐    │  │
 │  │  │  LSM 훅: security_inode_permission()                   │    │  │
 │  │  │                                                         │    │  │
 │  │  │  등록된 LSM 모듈 순차 호출:                            │    │  │
 │  │  │  ┌─────────────────────────────────────────────┐    │  │  │
 │  │  │  │  1️⃣ Capability 모듈                            │    │  │  │
 │  │  │  │    → CAP_DAC_OVERRIDE 검사                     │    │  │  │
 │  │  │  │    → 허용? YES → 다음 모듈로                    │    │  │  │
 │  │  │  └─────────────────────────────────────────────┘    │  │  │
 │  │  │         │                                               │    │  │
 │  │  │         ▼                                               │    │  │
 │  │  │  ┌─────────────────────────────────────────────┐    │  │  │
 │  │  │  │  2️⃣ SELinux 모듈                               │    │  │  │
 │  │  │  │    → 보안 컨텍스트 확인                         │    │  │  │
 │  │  │  │    → httpd_t가 shadow_t를 읽기? DENY!         │    │  │  │
 │  │  │  │    → 거부하면 최종 거부, 더 이상 호출 안 함 │    │  │  │
 │  │  │  └─────────────────────────────────────────────┘    │  │  │
 │  │  │         │                                               │    │  │
 │  │  │         ▼                                               │    │  │
 │  │  │  ┌─────────────────────────────────────────────┐    │  │  │
 │  │  │  │  3️⃣ AppArmor 모듈 (등록되어 있다면)        │    │  │  │
 │  │  │  │    → 경로 기반 정책 검사                         │    │  │  │
 │  │  │  │    → /etc/shadow 접근 불가? DENY!              │    │  │  │
 │  │  │  │    → 거부                                             │    │  │  │
 │  │  │  └─────────────────────────────────────────────┘    │  │  │
 │  │  │                                                         │    │  │
 │  │  └─────────────────────────────────────────────────────┘    │  │
 │  │           │                                                   │  │
 │  │           ▼                                                   │  │
 │  │      return -EACCES (모든 모듈이 통과해야 허용)                  │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  ✅ LSM은 "AND 로직" — 모든 모듈이 허용해야 최종 허용           │
 │  ✅ 하나라도 DENY면 최종 거부                                         │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** LSM은 **"등록된 모듈들을 순차적으로 호출"**하며, **"하나라도 거부하면 최종 거부"**된다. 이는 **"보안은 보수적이어야 한다"**는 철학에 기반한다. 예를 들어 SELinux가 DENY하면, AppArmor는 호출조차 되지 않는다. Capability 모듈이 가장 먼저 호출되는 이유는, POSIX Capabilities는 **"DAC 우회"**를 제공하며, MAC 모듈보다 약한 보안 계층이므로 먼저 검사한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: LSM 기반 보안 모듈 비교

| 모듈 | 기반 모델 | 설정 방식 | 장점 | 단점 |
|:---|:---|:---|:---|:---|
| **SELinux** | Type Enforcement + MLS | `semanage`, `chcon` | 강력한 MAC, 세밀한 제어 | 학습 곡선 가파름 |
| **AppArmor** | 경로 기반(Path-based) | `aa-easyprof`, 프로필 파일 | 설정 간단, 사용자 친화적 | 경로 우회 취약 |
| **Smack** | Simplified MAC | `smackload`, `smackctl` | 매우 단순, 임베디드 시스템 | 세밀한 제어 불가 |
| **TOMOYO** | Path-based + JSON | `tomoyo-policy` | 일본어 지원, XML/JSON 정책 | 서구권 지원 한계 |
| **Landlock** | LandlockLock | `landlock` 시스템 콜 | 샌드박스 강화 | 파일 시스템만 지원 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — SELinux 활성화**: RHEL/CentOS에서 SELinux는 기본 활성화되어 있다. `sestatus`로 확인:
   ```bash
   $ sestatus
   SELinux status:                 enabled
   SELinuxfs mount:                /sys/fs/selinux
   Current mode:                   enforcing
   ```
   Permissive 모드에서는 거부 로그만 남기고 실제로 차단하지 않는다.

2. **시나리오 — AppArmor로 프로세스 격리**: Ubuntu에서 AppArmor로 프로세스를 격리한다.
   ```bash
   $ aa-status
   profiles are in enforce mode.
   /usr/sbin/tcpdump
   /usr/bin/pidgin
   ```
   각 프로세스별로 접근 가능한 경로를 제한한다.

### 도입 체크리스트
- **기술적**: LSM이 활성화되어 있는가?(`lsmod | grep selinux`, `lsmod | grep apparmor`)
- **운영·보안적**: AVC 거부 로그를 정기적으로 점검하는가?(`ausearch -m avc -ts recent`)

## 👶 어린이를 위한 3줄 비유 설명
1. LSM은 **"공항의 보안 검색대"**예요. 모든 승객(시스템 콜)은 검색대를 통과해야 하고, 보안 요원(SELinux, AppArmor)이 여권을 확인해요.
2. 보안 요원이 "통과"라고 하면 비행기를 타고, "거부"라고 하면 못 타요. 보안 요원은 여러 명이 있고, 한 명이라도 거부하면 최종 거부예요.
3. 그래서 보안 정책이 바뀌어도 공항 시설(커널)을 고칠 필요 없이, 보안 요원만 교체하면 돼요. LSM도 보안 모듈만 갈아끼면 되니까요?
