## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보호 도메인은 프로세스가 접근할 수 있는 자원(객체)과 허용된 연산의 집합으로서, **"프로세스가 가진 권한护照(Passport)"**와 같다. 도메인 간 경계를 넘을 때는 반드시 커널을 통한 검사가 이루어진다.
> 2. **가치**: 도메인 격리가 없으면 프로세스 A의 버그가 프로세스 B의 메모리를 파괴하여 시스템 전체가 붕괴될 수 있다. 웹 브라우저의 탭 프로세스가 서로 격리되지 않으면, 악성 스크립트가 다른 탭의 쿠키를 탈취할 수 있다.
> 3. **융합**: x86의 Ring 아키텍처(Ring 0-3), POSIX Capability, Docker 컨테이너, SELinux 도메인은 모두 보호 도메인의 구현체로서 계층별로 최소 권한의 원칙을 실현한다.

## Ⅰ. 개요 (Context & Background)

- **개념**: 보호 도메인(Protection Domain)은 운영체제가 정의하는 **접근 권한의 경계**로서, 특정 프로세스가 시스템의 어떤 자원(Object)에 대해 어떤 연산(Operation)을 수행할 수 있는지를 명시하는 집합이다. 수학적으로는 도메인 D = {(O₁, op₁), (O₂, op₂), ...} 형태의 쌍(Pair)으로 표현된다. 프로세스는 자신의 도메인 밖에 있는 자원에 직접 접근할 수 없으며, 도메인을 변경하려면 **반드시 커널을 통한 검사**를 거쳐야 한다.

- **💡 비유**: 보호 도메인은 **"여권에 찍힌 입국 허가 국가 리스트"**다. 여권 소지자는 자신의 여권에 비자가 찍힌 국가에만 입국할 수 있으며, 비자가 없는 국가에 들어가려면 입국 심사(커널 검사)를 받아 추가 비자를 발급받아야 한다. 마찬가지로 프로세스도 자신의 도메인(여권 비자)에 포함된 자원에만 접근할 수 있으며, 다른 자원에 접근하려면 시스템 콜(입국 심사)을 통해 커널의 허가를 받아야 한다.

- **등장 배경**:
  1. **MULTICS 프로젝트(1964)**: MIT, GE, Bell Labs가 개발한 MULTICS 시스템은 최초로 링(Ring) 기반의 보호 도메인을 도입했다. Ring 0은 가장 권한이 강한 커널 모드, Ring 3은 가장 제약이 많은 사용자 모드로, 숫자가 작을수록 권한이 크다.
  2. **UNIX의 UID/GID 모델(1970s)**: UNIX는 링 구조 대신 간단한 사용자 ID(UID)와 그룹 ID(GID)를 사용하여 도메인을 정의했다. `setuid(0)` 시스템 콜을 호출하여 도메인을 변경(권한 상승)할 수 있으며, 이는 링 전환(Trap)에 해당한다.
  3. **최신 OS의 Capability 기반 도메인(2000s-현재)**: POSIX Capabilities, SELinux 도메인, Docker 컨테이너는 더 세밀한 단위로 도메인을 정의한다. 예를 들어 웹 서버 프로세스는 `CAP_NET_BIND_SERVICE`만 보유하여 80번 포트에 바인딩할 수 있지만, `CAP_SYS_ADMIN`은 없어서 커널 모듈을 로드할 수 없다.

보호 도메인의 계층적 구조를 링 아키텍처와 함께 시각화하면 다음과 같다. 각 링은 자신보다 권한이 큰(숫자가 작은) 링으로 진입할 때만 제한적 접근이 가능하다.

```text
 ┌───────────────────────────────────────────────────────────────────┐
 │                  링(Ring) 기반 보호 도메인                        │
 ├───────────────────────────────────────────────────────────────────┤
 │                                                                   │
 │  권한: 크다 ▲                                                     │
 │        ┌─│                                                       │
 │        │ │                                                      │
 │  Ring 0│ │┌───────────────────────────────────────────────────┐  │
 │  (커널)│ ││  • 모든 메모리 접근 가능                          │  │
 │        │ ││  • 하드웨어 장치 직접 제어 (IN/OUT 명령어)        │  │
 │        │ ││  • 페이지 테이블 직접 수정                        │  │
 │        │ ││  • 인터럽트 활성화/비활성화 (CLI/STI)            │  │
 │        │ │└───────────────────────────────────────────────────┘  │
 │        ││                                                        │
 │        ├┤ Call Gate / Trap Gate를 통한 진입만 허용                 │
 │        ││                                                        │
 │  Ring 1│ │┌───────────────────────────────────────────────────┐  │
 │  (디바이스│││  • 특정 디바이스 드라이버 영역 접근               │  │
 │   드라이버)│││  • Ring 0으로 진입하는 시스템 콜 인터페이스    │  │
 │        │ │└───────────────────────────────────────────────────┘  │
 │        ││                                                        │
 │  Ring 2│ │┌───────────────────────────────────────────────────┐  │
 │        │ ││  • 그래픽 드라이버 등 I/O 특권 작업               │  │
 │        │ ││  (현대 x86-64에서는 거의 사용 안 함)              │  │
 │        │ │└───────────────────────────────────────────────────┘  │
 │        ││                                                        │
 │  Ring 3│ │┌───────────────────────────────────────────────────┐  │
 │  (사용자│││  • 자신의 가상 주소 공간만 접근 가능               │  │
 │   응용) │││  • 하드웨어 직접 접근 불가                        │  │
 │        │ ││  • 시스템 콜을 통해서만 Ring 0 기능 요청          │  │
 │        │ │└───────────────────────────────────────────────────┘  │
 │        │                                                        │
 │        ▼                                                        │
 │     권한: 작다                                                   │
 │                                                                   │
 │  도메인 전환 예: 시스템 콜 호출                                    │
 │  ┌─────────┐      시스템 콜          ┌─────────┐                 │
 │  │ 사용자   │ ─────────────────────▶ │  커널    │                 │
 │  │ 프로세스 │    (int $0x80 / syscall) │          │                 │
 │  │(Ring 3) │                        │ (Ring 0) │                 │
 │  └─────────┘                        └─────────┘                 │
 │                                                                   │
 └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** x86 프로세서는 CR4 레지스터의 CPL(Current Privilege Level) 필드로 현재 링을 추적하며, CS(Code Segment) 세그먼트 셀렉터의 하위 2비트에 저장된다. Ring 3(사용자 모드)에서 Ring 0(커널 모드)로 진입하려면 **소프트웨어 인터럽트(int 0x80, SYSENTER/SYSEXIT)** 또는 **시스템 콜 게이트(Call Gate)**를 통과해야 한다. 이 과정에서 CPU는 자동으로 CPL을 0으로 변경하고 커널 스택으로 전환한다. 반대로 커널에서 사용자 공간으로 복귀할 때는 `iret`(Interrupt Return) 명령어가 CPL을 3으로 복원하고 사용자 스택으로 전환한다. 이 링 전환 메커니즘이 하드웨어적으로 보호 도메인의 경계를 강제한다.

### 도메인 전환(Switching) 메커니즘

프로세스가 도메인을 변경하는 방법은 크게 ① **임시적 전환(Temporal Switch)** - 시스템 콜과 같은 단발성 검사와 ② **영구적 전환(Permanent Switch)** - `setuid()`, `execve()`와 같은 지속적 권한 변경으로 나뉜다.

```text
 ┌───────────────────────────────────────────────────────────────────┐
 │                 도메인 전환(Switching) 메커니즘                    │
 ├───────────────────────────────────────────────────────────────────┤
 │                                                                   │
 │  [임시적 전환 — 시스템 콜 예시]                                    │
 │  ┌────────────────────────────────────────────────────────────┐   │
 │  │  사용자 공간                                                │   │
 │  │  ┌──────────────────────────────────────────────────────┐   │   │
 │  │  │  write(fd, buf, count)  ─────── C 라이브러리        │   │   │
 │  │  │         │                                          │   │   │
 │  │  │         ▼                                          │   │   │
 │  │  │  ┌─────────────────────────────────────────────┐    │   │   │
 │  │  │  │  syscall(SYS_write, fd, buf, count)        │    │   │   │
 │  │  │  │         │                                │    │   │   │
 │  │  │  │         ▼                                │    │   │   │
 │  │  │  │  mov eax, 1    ; syscall number           │    │   │   │
 │  │  │  │  syscall        ; Ring 3→0 전환!         │    │   │   │
 │  │  │  └─────────────────────────────────────────────┘    │   │   │
 │  │  └──────────────────────────────────────────────────────┘   │   │
 │  └────────────────────────────────────────────────────────────┘   │
 │                              │ Trap                               │
 │                              ▼                                   │
 │  ┌────────────────────────────────────────────────────────────┐   │
 │  │  커널 공간 (entry_SYSCALL_64)                              │   │
 │  │                                                             │   │
 │  │   ① CPL 3 → 0 자동 변경 (CPU 하드웨어)                      │   │
 │  │   ② 사용자 스택 → 커널 스택 전환                           │   │
 │  │   ③ 시스템 콜 테이블(sys_call_table) 인덱싱               │   │
 │  │       ┌─────────────────────────────────────────────────┐  │   │
 │  │       │  [0] sys_read                                   │  │   │
 │  │       │  [1] sys_write  ◀────────── eax=1               │  │   │
 │  │       │  [2] sys_open                                   │  │   │
 │  │       │  ...                                            │  │   │
 │  │       └─────────────────────────────────────────────────┘  │   │
 │  │   ④ sys_write() 호출 → VFS 계층 권한 검사                  │   │
 │  │   ⑤ sysret으로 복귀 → CPL 0 → 3, 커널 → 사용자 스택       │   │
 │  │                                                             │   │
 │  └────────────────────────────────────────────────────────────┘   │
 │                                                                   │
 │  [영구적 전환 — setuid(0) 예시]                                   │
 │  ┌────────────────────────────────────────────────────────────┐   │
 │  │                                                             │   │
 │  │   프로세스 A (UID=1000)                                   │   │
 │  │     │                                                      │   │
 │  │     ├──▶ setuid(0) 시스템 콜                               │   │
 │  │     │                                                      │   │
 │  │     ▼                                                      │   │
 │  │   ┌──────────────────────────────────────────────────┐     │   │
 │  │   │  sys_setuid(uid)                                 │     │   │
 │  │   │    1. caller의 CAP_SETUID capability 확인        │     │   │
 │  │   │    2. current->cred->uid = 0으로 변경             │     │   │
 │  │   │    3. 이후 모든 시스템 콜에서 UID=0으로 검사      │     │   │
 │  │   └──────────────────────────────────────────────────┘     │   │
 │  │     │                                                      │   │
 │  │     ▼                                                      │   │
 │  │   프로세스 A (UID=0, 영구적으로 root 권한 획득)          │   │
 │  │                                                             │   │
 │  └────────────────────────────────────────────────────────────┘   │
 │                                                                   │
 └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 시스템 콜에 의한 임시적 전환은 프로세스의 영구적인 보호 도메인이 변경되지 않는다. 단지 **"커널이라는 특권 모드를 잠시 빌려 쓰는 것"**으로서, 커널이 사용자를 대신하여 권한 있는 작업을 수행한 뒤 다시 원래 도메인으로 복귀한다. 반면 `setuid(0)`는 프로세스의 자격(Credential) 자체를 영구적으로 변경하므로, 이후 해당 프로세스는 root 도메인에 지속적으로 속하게 된다. 이는 `sudo` 명령어가 작동하는 원리이기도 하다.

### POSIX Capabilities — 세분화된 도메인

리눅스 커널 2.2 이후부터는 "모두 혹은 무(all-or-nothing)"인 root 권한을 개별 capability 단위로 분할하여, 프로세스에 최소 권한만 부여할 수 있게 되었다. 예를 들어 웹 서버는 `CAP_NET_BIND_SERVICE`만 있어 80번 포트에 바인딩할 수 있지만, `CAP_SYS_ADMIN`이 없어서 `mount()`를 호출할 수는 없다.

```text
 ┌───────────────────────────────────────────────────────────────────┐
 │            POSIX Capabilities — 세분화된 보호 도메인               │
 ├───────────────────────────────────────────────────────────────────┤
 │                                                                   │
 │  [기존 모델: UID 기반 도메인]                                     │
 │  ┌─────────────────────────────────────────────────────────────┐  │
 │  │  if (uid == 0) {                                            │  │
 │  │      // 모든 권한 허용                                      │  │
 │  │      return ALLOW;                                          │  │
 │  │  } else {                                                   │  │
 │  │      // 모든 권한 거부                                      │  │
 │  │      return DENY;                                           │  │
 │  │  }                                                          │  │
 │  └─────────────────────────────────────────────────────────────┘  │
 │                                                                   │
 │  [Capability 기반 도메인]                                         │
 │  ┌─────────────────────────────────────────────────────────────┐  │
 │  │                                                             │  │
 │  │  프로세스의 capability 세트                                 │  │
 │  │  ┌─────────────────────────────────────────────────────┐    │  │
 │  │  │  Permitted (P):  { CAP_NET_BIND_SERVICE,           │    │  │
 │  │  │                     CAP_DAC_OVERRIDE,              │    │  │
 │  │  │                     CAP_SETUID }                   │    │  │
 │  │  │  Effective (E):   { CAP_NET_BIND_SERVICE }         │    │  │
 │  │  │  Inheritable (I): { }                              │    │
 │  │  └─────────────────────────────────────────────────────┘    │  │
 │  │                                                             │  │
 │  │  주요 capability 예시                                       │  │
 │  │  ┌─────────────────────┬─────────────────────────────────┐   │  │
 │  │  │ CAP_CHOWN           │ 파일 소유자 변경               │   │  │
 │  │  │ CAP_DAC_OVERRIDE    │ DAC(rwx) 검사 우회             │   │  │
 │  │  │ CAP_NET_BIND_SERVICE│ 1024번 이하 포트 바인딩         │   │  │
 │  │  │ CAP_SYS_ADMIN       │ 시스템 관리 작업(mount, etc)    │   │  │
 │  │  │ CAP_SYS_TIME        │ 시스템 시간 변경                │   │  │
 │  │  │ CAP_SYS_MODULE      │ 커널 모듈 로드/언로드            │   │  │
 │  │  │ CAP_NET_RAW         │ RAW 소켓 생성 (ICMP, etc)      │   │  │
 │  │  └─────────────────────┴─────────────────────────────────┘   │  │
 │  │                                                             │  │
 │  │  검사 예: bind(sockfd, port=80)                             │  │
 │  │  ┌─────────────────────────────────────────────────────┐    │  │
 │  │  │  if (port < 1024 && !capable(CAP_NET_BIND_SERVICE))│    │  │
 │  │  │      return -EACCES;                                │    │  │
 │  │  └─────────────────────────────────────────────────────┘    │  │
 │  └─────────────────────────────────────────────────────────────┘  │
 │                                                                   │
 └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 리눅스 커널은 `capable()` 함수로 capability를 검사한다. 예를 들어 `inet_bind()` 함수는 1024번 이하의 특권 포트에 바인딩하려 할 때 `capable(CAP_NET_BIND_SERVICE)`를 호출하여, 현재 프로세스의 **Effective(E) 세트**에 해당 capability가 있는지 확인한다. `capable()`은 `current->cred`에 저장된 capability 비트맵을 검사하며, 없으면 `-EPERM`을 반환한다. 이 모델에서는 웹 서버 프로세스가 `CAP_NET_BIND_SERVICE`만 보유하도록 설정하여, 웹 서버가 해킹당하더라도 공격자는 커널 모듈을 로드하거나(`CAP_SYS_ADMIN`) 파일 시스템을 마운트하는 등의 추가 피해를 입힐 수 없게 된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 링(Ring) vs Capability vs SELinux Domain

| 비교 항목 | 링 (Ring) | POSIX Capability | SELinux Domain |
|:---|:---|:---|:---|
| **권한 단위** | 4단계(Ring 0-3) | ~40개 개별 capability | 무한(레이블 조합) |
| **구현 위치** | 하드웨어(CPU CPL) | 커널(cred 구조체) | 커널(LSM 훅) |
| **전환 비용** | 낮음(소프트웨어 인터럽트) | 낮음(메모리 참조) | 높음(AVC 캐시 미스 시 정책 DB 검색) |
| **세분성** | 조잡(커널/드라이버/사용자) | 중간(개별 시스템 콜 단위) | 매우 세밀(타입/도메인/클래스) |
| **주 용도** | 기본적 하드웨어 보호 | root 권한 분할 | MAC 강제, 컨테이너 격리 |

링 아키텍처는 가장 기본적인 보호 도메인으로서, 하드웨어적으로 강제되지만 4단계로만 나뉘어 있어 세밀한 권한 분리가 불가능하다. Capability는 이를 보완하여 개별 시스템 콜 단위로 권한을 분할하지만, 여전히 "어떤 자원"에 접근하는지는 제어하지 못한다(예: `CAP_DAC_OVERRIDE`는 모든 파일의 rwx 권한을 우회한다). SELinux는 타입 강제(Type Enforcement)를 통해 "어떤 타입의 프로세스가 어떤 타입의 파일에 접근 가능한지"를 명시하여 가장 세밀한 격리를 제공한다.

### 비교 2: 도메인 격리 수준 비교

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                도메인 격리 수준 비교 (Isolation Depth)                 │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [레벨 1] 프로세스 단위 (UID/GID)                                    │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  프로세스 A (UID=1000)  │  프로세스 B (UID=1001)               │  │
 │  │  ┌─────────────────┐   │  ┌─────────────────┐                   │  │
 │  │  │ 메모리 공간      │   │  │ 메모리 공간      │                   │  │

 │  │  └─────────────────┘   │  └─────────────────┘                   │  │
 │  │                                                                │  │
 │  │  격리: 메모리는 분리되지만, /tmp 등 공유 자원은 상호 접근 가능  │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [레벨 2] 컨테이너 단위 (Linux Namespace + Cgroup)                   │
 │  ┌──────────────────────────────┬───────────────────────────────┐    │
 │  │  Container A                  │  Container B                 │    │
 │  │  ┌────────────────────────┐  │  ┌────────────────────────┐   │    │
 │  │  │ PID 네임스페이스 (1번만)│  │  │ PID 네임스페이스 (1번만)│   │    │
 │  │  │ NET 네임스페이스      │  │  │ NET 네임스페이스       │   │    │

 │  │  │ chroot 루트 디렉터리   │  │  │ chroot 루트 디렉터리    │   │    │
 │  │  └────────────────────────┘  │  └────────────────────────┘   │    │
 │  │                                                                │  │
 │  │  격리: 파일시스템, 네트워크, 프로세스 트리까지 완전히 분리      │  │
 │  └───────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [레벨 3] 가상머신 단위 (하이퍼바이저)                               │
 │  ┌───────────────────────────────┬────────────────────────────────┐  │
 │  │  VM A (하드웨어 완전 가상화)    │  VM B (하드웨어 완전 가상화)     │  │
 │  │  ┌─────────────────────────┐  │  ┌───────────────────────────┐ │  │
 │  │  │ 독립 커널               │  │  │ 독립 커널                 │ │ │  │
 │  │  │ 독립 하드웨어 (가상)    │  │  │ 독립 하드웨어 (가상)       │ │ │  │
 │  │  │ vCPU, vNIC, vDisk      │  │  │ vCPU, vNIC, vDisk          │ │ │  │
 │  │  └─────────────────────────┘  │  └───────────────────────────┘ │  │
 │  │                                                                │  │
 │  │  격리: 커널 레벨에서 완전히 분리, 보안 등급 가장 높음          │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  격리 비용(오버헤드): VM > Container > Process                       │

 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 프로세스 단위 격리는 MMU의 페이지 테이블로 메모리 공간만 분리하지만, `/tmp`, `/dev/shm` 같은 공유 자원은 여전히 상호 접근 가능하다. 컨테이너는 Linux Namespace를 사용하여 PID, 네트워크, 마운트 지점까지 분리하여, 각 컨테이너가 마치 독립 시스템인 것처럼 보이게 한다. 그러나 여전히 동일 커널을 공유하므로 커널 취약점으로 탈출 가능성이 있다. 가상머신은 하이퍼바이저가 완전히 독립된 하드웨어(vCPU, vNIC)를 에뮬레이션하므로, 게스트 OS끼리는 커널 레벨에서 격리되어 가장 강력한 보호를 제공한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 웹 브라우저의 사이트 격리(Site Isolation)**: Chrome 브라우저는 각 웹사이트 탭을 별도 프로세스로 실행하여, `site-a.com`의 악성 스크립트가 `site-b.com`의 쿠키를 탈취하는 것을 방지한다. 이것이 프로세스 단위 보호 도메인의 전형적 응용이다. 만약 단일 프로세스에서 모든 탭을 실행했다면, Same-Origin Policy(SOP)를 우회하는 Safari의 XS-Leak 취약점처럼 타 사이트의 데이터가 유출될 수 있었다.

2. **시나리오 — Kubernetes의 Pod Security Context**: 쿠버네티스 클러스터에서 컨테이너를 `runAsUser: 1000`, `readOnlyRootFilesystem: true`, `capabilities.drop: ["ALL"]`로 설정하여, 컨테이너가 root 권한으로 실행되는 것을 방지하고, 파일시스템 쓰기를 차단하며, 모든 capability를 제거했다. 이렇게 최소 권한 도메인으로 제한하면 컨테이너가 탈취되더라도 호스트 공격 벡터가 최소화된다.

3. **시나리오 — AWS Lambda의 보안 격리**: Lambda는 각 함수 실행을 별도 Firecracker 마이크로VM에서 실행하여, 함수 간 완전한 격리를 보장한다. Firecracker는 KVM 기반의 경량 하이퍼바이저로서, 일반적인 VM보다 오버헤드가 낮으면서도 커널 수준의 격리를 제공한다.

### 도입 체크리스트
- **기술적**: 모든 인터넷 노출 서비스가 비특권 사용자로 실행되고 있는가? `CAP_SYS_ADMIN` 등 위험한 capability는 제거되었는가?
- **운영·보안적**: 각 서비스별로 보호 도메인이 분리되어 있어, 한 서비스가 탈취되어도 다른 서비스로 전이(공격 이동)가 불가능한가?

### 안티패턴
- **과도한 권한 부여**: `docker run --privileged`는 모든 capability와 디바이스 접근을 허용하여, 컨테이너가 호스트와 거의 동등한 권한을 갖게 된다. 이는 보호 도메인의 원칙을 완전히 무시한 것으로, 절대 프로덕션에서 사용하면 안 된다.
- **UID 0으로 실행**: 대부분의 웹 서버는 처음에만 80번 포트에 바인딩하기 위해 root로 실행되고, 이후 `setuid()`로 www-data 사용자로 권한을 낮춘다. 만약 권한 하향에 실패하면 전체 서비스가 root로 계속 실행되어 심각한 보안 위험이 된다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 | 최적화 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 모든 컨테이너가 root로 실행 | 비특권 사용자 + Capability drop | **탈취 시 공격 영역 80%+ 축소** |
| **정량** | 단일 컨테이너에 모든 서비스 | 마이크로서비스별 파드 분리 | **보안 사고 영향 범위 90%+ 감소** |
| **정성** | 서비스 간 횡적 이동(Lateral Movement) 가능 | 네임스페이스 격리로 불가 | **제로데이 완화** |

### 미래 전망
- **gVisor — User-space Kernel**: Google이 개발한 gVisor는 완전히 사용자 공간에서 작동하는 커널을 구현하여, 애플리케이션과 호스트 커널 사이에 추가 보호 계층을 제공한다. 호스트 커널 취약점을 이용한 탈출을 방지한다.
- **다중tenant 웹 브라우저**: 웹 브라우저는 각 탭을 별도 프로세스, 별도 사이트 인스턴스(Site Instance), 심지어 별도 GPU 프로세스로 분리하여 보호 도메인을 극단적으로 세분화하고 있다.

### 참고 표준
- **POSIX 1003.1e**: Draft 표준으로, capability 기반 권한 모델을 정의
- **Docker Security Best Practices**: OCI 런타임 사양(runc)에 정의된 security-opt, cap-add/drop, user 옵션

## 👶 어린이를 위한 3줄 비유 설명
1. 보호 도메인은 **"학교에서 나눠준 반 팔찌"**예요. 1반 팔찰를 차면 1반 교실만 들어갈 수 있고, 2반은 못 가요.
2. 컴퓨터 프로그램도 자신이 속한 "반(도메인)"에 있는 것들만 사용할 수 있어요. 다른 반의 장난감을 가지고 싶으면 선생님께 허락(시스템 콜)을 받아야 해요.
3. 그래서 해커가 한 프로그램을 해킹해도, 다른 프로그램의 "반"은 다르므로 못 해쳐요. 우리 모두가 서로 다른 반에 속해 있어서 안전한 거예요!
