# 610. Access Matrix (접근 제어 행렬)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 접근 제어 행렬(Access Matrix)은 시스템의 모든 주체(Subject)가 객체(Object)에 대해 갖는 권한(Right)을 2차원 행렬로 표현한 보호 모델이다.
> 2. **가치**: 주체-객체 간 권한 관계를 명시적이고 직관적으로 표현하여, 보안 정책 설계와 검증의 수학적 기반을 제공한다.
> 3. **융합**: 실무로 구현 시 희소 행렬 압축하여 ACL(Access Control List)과 Capability List로 변환되며, Unix 파일 권한, Windows ACL, SELinux 정책 등 현대 OS 보호의 근간이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**접근 제어 행렬(Access Matrix)**은 **시스템의 주체(Subject: 프로세스, 사용자)와 객체(Object: 파일, 장치, 메모리) 사이의 모든 접근 권한을 행렬 형태로 표현한 모델**이다. 행(Row)은 주체, 열(Column)은 객체를 나타내며, 각 셀(cell)은 해당 주체가 객체에 대해 수행할 수 있는 연산의 집합(권한, Right)을 저장한다. 1971년 Butler Lampson이 제안한 이 모델은 보호 시스템의 형식화된 정의를 제공하며, 접근 제어의 기본 철학을 설명한다.

### 💡 비유: 건물 보안 출입 명단

접근 제어 행렬은 **건물의 보안 출입 명단**과 같다. 직원(주체)별로 각 방(객체)에 들어갈 수 있는 권한(출입, 읽기, 수정, 삭제)이 명시된 표를 생각하면 된다. 예를 들어, "김 과장"은 "회의실 A"에는 "입장+발표" 권한이 있지만, "금고"에는 권한이 없다. 이 행렬을 보관인이 보고, 출입 시마다 검사하면 보안이 유지된다.

### 등장 배경

1. **초기 시분할 시스템의 접근 제어 문제**: 1960년대 다중 사용자 시스템에서 프로세스 간 메모리 침해, 파일 무단 접근 문제가 발생했다. 이를 해결하기 위해 형식적 보호 모델이 필요했다.
2. **Honeywell Multics 프로젝트**: 초기 보안 OS 개발에서 접근 제어의 수학적 기반으로 Lampson의 접근 제어 행렬이 채택되었다.
3. **Capability 기반 시스템과의 대비**: 객체 중심의 ACL과 주체 중심의 Capability List 중 어떤 방식으로 구현할지 설계 트레이드오프를 제공했다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                   접근 제어 행렬의 발전 역사                               │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【1971: Butler Lampson - "Protection" 논문】                             │
│  ──────────────────────────────────────────                              │
│  • 접근 제어 행렬 모델 제안                                                  │
│  • 주체-객체-권한의 3요소 정의                                              │
│  • 보호 도메인(Protection Domain) 개념 도입                                 │
│                                                                         │
│  【1973: Graham-Denning 모델】                                            │
│  ────────────────────────────                                            │
│  • 접근 제어 행렬의 동작을 8가지 원시적 연산으로 정의                            │
│  • 객체 생성, 삭제, 권한 부여, 회수 등                                          │
│                                                                         │
│  【1976: Harrison-Ruzzo-Ullman(HRU) 정리】                                 │
│  ────────────────────────────────────────                                 │
│  • 접근 제어 행렬의 안전성(Safety Problem)이 결정 불가능(undecidable)임 증명      │
│  • 주어진 초기 상태와 연산 집합에서 특정 권한이 부여될 수 있는지 알 수 없음            │
│                                                                         │
│  【1980s-현재: 실무 적용】                                                  │
│  ──────────────────────                                                  │
│  • Unix: 간단한 3비트(rwx) 권한으로 축소 구현                                 │
│  • Windows NT: ACL(Access Control List)로 구현                             │
│  • SELinux: Type Enforcement으로 행렬의 라벨 기반 구현                       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 정의 | 예시 | 비유 |
|:---|:---|:---|:---|
| **주체(Subject)** | 능동적 엔티티, 접근을发起하는 쪽 | 프로세스, 사용자, 스레드 | 건물 출입자 |
| **객체(Object)** | 수동적 엔티티, 접근의 대상 | 파일, 디렉터리, 장치, 메모리 페이지 | 방, 금고, 서버실 |
| **권한(Right)** | 주체가 객체에 대해 수행 가능한 연산 | read, write, execute, delete | 출입, 읽기, 수정 |
| **보호 도메인(Protection Domain)** | 특정 시점에서 주체가 갖는 권한 집합 | 프로세스의 User ID, UID | 사용자의 출입증 |

### 접근 제어 행렬 구조

기본적인 접근 제어 행렬의 구조를 예시로 보자.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                    접근 제어 행렬 예시                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│         ┌────────────────────────────────────────────────────────┐       │
│         │                    객체 (Object)                        │       │
│         │  File A   File B   File C   Printer   Tape     ...    │       │
│         ├────────────────────────────────────────────────────────┤       │
│   ┌─────┼─────────┬─────────┬─────────┬─────────┬─────────┬─────────┤       │
│   │User1│  R,W    │         │  R      │         │  W      │         │       │
│   ├─────┼─────────┼─────────┼─────────┼─────────┼─────────┼─────────┤       │
│   │User2│         │  R,W,X  │  R,W    │   W     │         │         │       │
│   ├─────┼─────────┼─────────┼─────────┼─────────┼─────────┼─────────┤       │
│주체│User3│  R      │  R      │  R,W    │         │         │  R,W    │       │
│   │     ├─────────┼─────────┼─────────┼─────────┼─────────┼─────────┤       │
│   │Proc1│  R,W    │         │         │   W     │         │         │       │
│   ├─────┼─────────┼─────────┼─────────┼─────────┼─────────┼─────────┤       │
│   │Proc2│         │  R,W    │  R      │         │  R,W    │         │       │
│   └─────┴─────────┴─────────┴─────────┴─────────┴─────────┴─────────┘       │
│                                                                         │
│   R = Read, W = Write, X = eXecute                                      │
│   빈 셀 = 권한 없음                                                       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 행렬에서 User1은 File A에 읽기/쓰기 권한이 있고, File C에는 읽기만 가능하다. User2는 File B에 실행 권한까지 포함하여 가장 많은 권한을 갖는다. Proc1과 Proc2는 프로세스(주체)로서, User1이 실행한 프로세스는 User1의 권한을 상속받아 File A에 접근할 수 있다. 실제 시스템에서는 이 행렬을 그대로 저장하지 않고, 희소성(sparsity)을 고려하여 압축한다. 왜냐하면 대부분의 셀은 비어 있고(권한 없음), 행렬 전체를 메모리에 저장하면 공간 낭비가 심하기 때문이다.

### 행렬의 구현 방식

실무에서는 접근 제어 행렬을 두 가지 방식으로 압축하여 구현한다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│            접근 제어 행렬의 구현 방식: ACL vs Capability                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【방식 1: ACL (Access Control List) - 객체 중심】                          │
│  ────────────────────────────────────────────────────                    │
│                                                                         │
│    객체별로 "누가 어떤 권한을 갖는지" 리스트로 저장                             │
│                                                                         │
│    File A: [(User1, R,W), (User3, R), (Proc1, R,W)]                      │
│    File B: [(User2, R,W,X), (Proc2, R,W)]                               │
│    File C: [(User1, R), (User2, R,W), (User3, R,W)]                      │
│    Printer: [(User2, W), (Proc1, W)]                                     │
│                                                                         │
│  장점:                                                                   │
│    • 객체의 접근자를 빠르게 열거할 수 있음 (예: ls -l)                           │
│    • 권한 변경 시 해당 객체의 ACL만 수정                                     │
│    • 파일 시스템 메타데이터에 저장하기 용이                                    │
│                                                                         │
│  단점:                                                                   │
│    • 특정 주체의 모든 권한을 찾으려면 모든 객체의 ACL을 검색해야 함 (O(N))            │
│    • ACL이 커지면 검색 오버헤드                                              │
│                                                                         │
│  사용 사례: Unix/Linux inode, Windows NTFS ACL                             │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【방식 2: Capability List - 주체 중심】                                   │
│  ────────────────────────────────────────────────                        │
│                                                                         │
│    주체별로 "어떤 객체에 어떤 권한이 있는지" 리스트로 저장                        │
│                                                                         │
│    User1: [(File A, R,W), (File C, R), (Tape, W)]                       │
│    User2: [(File B, R,W,X), (File C, R,W), (Printer, W)]                │
│    User3: [(File A, R), (File B, R), (File C, R,W), (Tape, R,W)]        │
│                                                                         │
│  장점:                                                                   │
│    • 주체의 권한을 빠르게 열거할 수 있음                                     │
│    • Capability 토큰을 불변 객체로 구현하면 위조 불가                               │
│    • 분산 시스템에서 권한 위임(delegation) 용이                                 │
│                                                                         │
│  단점:                                                                   │
│    • 객체의 접근자를 찾으려면 모든 주체의 Capability를 검색해야 함                    │
│    • Capability 회수(revocation)이 어려움 (복사본 추적 필요)                     │
│                                                                         │
│  사용 사례: Kerberos 티켓, POSIX Capability, Capsicum (BSD)                 │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【비교: ACL vs Capability】                                             │
│  ──────────────────────────────                                         │
│                                                                         │
│   ┌────────────────────┬──────────────────┬──────────────────┐            │
│   │                    │       ACL        │   Capability     │            │
│   ├────────────────────┼──────────────────┼──────────────────┤            │
│   │저장 단위           │ 객체별           │ 주체별           │            │
│   │권한 검색           │ "누가 접근 가능?" │ "무엇을 할 수 있?" │            │
│   │권한 회수           │ 용이 (ACL 수정)   │ 어려움 (복사본 문제)│            │
│   │대표적 구현         │ Unix, Windows    │ Kerberos, POSIX  │            │
│   └────────────────────┴──────────────────┴──────────────────┘            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ACL은 파일 시스템의 inode 내에 권한 정보를 저장한다. `ls -l` 명령으로 볼 수 있는 `rwxr-xr--` 형태가 ACL의 간단한 구현이다. 반면 Capability는 Kerberos 티켓처럼 사용자가 가진 토큰의 형태로 권한을 증명한다. Capability는 주체가 소유하고 있으므로 분산 시스템에서 권한을 다른 노드로 전달(delegate)하기 쉽지만, 한 번 발급된 토큰을 회수(revoke)하기 어렵다는 근본적 단점이 있다. ACL은 객체 중심이므로 권한 회수가 용이하지만, 대규모 시스템에서는 모든 ACL을 검색해야 하므로 성능 오버헤드가 있다.

### Graham-Denning 모델 (1973)

Graham과 Denning은 접근 제어 행렬의 동작을 정의하는 8가지 원시적 연산(Primitive Operation)을 제안했다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│            Graham-Denning 모델: 8가지 원시적 연산                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【객체 관련 연산】                                                        │
│  ────────────                                                           │
│                                                                         │
│  1. Create Object(o)                                                    │
│     - 객체 o를 생성하고, 생성자(주체)에게 소유권(ownership) 부여                    │
│     - 초기 상태: 생성자만 o에 전체 권한(R, W, X 등)                              │
│                                                                         │
│  2. Delete Object(o)                                                    │
│     - 객체 o를 삭제하고, 행렬에서 o 열(column) 제거                             │
│     - o에 대한 모든 권한 소멸                                                   │
│                                                                         │
│  3. Enter Right into Matrix(S, o, r)                                    │
│     - 주체 S의 객체 o에 대한 권한 r을 행렬에 추가                                │
│     - 권한 부여(grant) 연산                                                │
│                                                                         │
│  4. Delete Right from Matrix(S, o, r)                                  │
│     - 주체 S의 객체 o에 대한 권한 r을 행렬에서 제거                              │
│     - 권한 회수(revoke) 연산                                               │
│                                                                         │
│  【주체 관련 연산】                                                        │
│  ────────────                                                           │
│                                                                         │
│  5. Create Subject(S)                                                   │
│     - 주체 S를 생성하고, 행렬에 새 행(row) 추가                                │
│     - 초기 상태: S는 아무 권한도 없음                                            │
│                                                                         │
│  6. Delete Subject(S)                                                  │
│     - 주체 S를 삭제하고, 행렬에서 S 행(row) 제거                               │
│     - S가 갖던 모든 권한 소멸                                                  │
│                                                                         │
│  7. Create Right(S, o, r)                                               │
│     - 주체 S가 객체 o에 대한 권한 r을 스스로 생성 가능한지 확인                        │
│     - 소유권(ownership) 기반 권한 생성                                         │
│                                                                         │
│  【보호 도메인 전이】                                                      │
│  ──────────────                                                      │
│                                                                         │
│  8. Transfer Right(S₁, S₂, o, r)                                        │
│     - 주체 S₁이 S₂에게 객체 o에 대한 권한 r를 이전(transfer)                      │
│     - S₁은 o에 대해 r 권한을 가지고 있어야 함                                   │
│     - "copy flag"로 r의 재이전 가능 여부 제어                                   │
│                                                                         │
│  【동작 예시】                                                            │
│  ───────────                                                            │
│                                                                         │
│  # User1이 File A를 생성                                                    │
│  Create Object(File A)    → User1은 File A의 소유자                          │
│                                                                         │
│  # User1이 User2에게 File A 읽기 권한 부여                                     │
│  Enter Right(User2, File A, R)                                          │
│                                                                         │
│  # User1이 User2의 File A 쓰기 권한 회수                                      │
│  Delete Right(User2, File A, W)                                          │
│                                                                         │
│  # User2가 자신의 File B를 생성                                               │
│  Create Object(File B)    → User2는 File B의 소유자                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Graham-Denning 모델은 접근 제어 행렬의 상태 변화를 정의한다. `Enter Right`는 권한 부여(grant)에 해당하며, `Delete Right`는 권한 회수(revoke)에 해당한다. 실제 시스템에서 이는 `chmod g+w file`(그룹 쓰기 권한 추가)이나 `chmod o-r file`(타인 읽기 권한 삭제)와 같은 명령으로 구현된다. `Transfer Right`는 권한 위임(delegation)을 허용하는지 여부를 제어한다. 예를 들어, 소유자는 파일을 다른 사용자에게 "공유"할 수 있지만, 그 사용자가 다시 제3자에게 권한을 줄 수 있는지는 "copy flag"로 결정한다.

### HRU 정리와 안전성 문제 (1976)

Harrison, Ruzzo, Ullman은 접근 제어 행렬의 **안전성 문제(Safety Problem)**이 결정 불가능(undecidable)임을 증명했다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│         HRU(Harrison-Ruzzo-Ullman) 정리                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【안전성 문제(Safety Problem) 정의】                                      │
│  ──────────────────────────────────                                      │
│                                                                         │
│  문제: "초기 행렬 상태 M₀와 연산 집합 O가 주어졌을 때,                          │
│        연산을 유한번 적용한 후 특정 권한 r이 어떤 주체에 부여될 수 있는가?"          │
│                                                                         │
│  형식화: ⟨M₀, O, S, o, r⟩가 주어졌을 때, O* 적용 후                          │
│          M'[S, o]가 r를 포함하는가?                                       │
│                                                                         │
│  【HRU 정리: 결정 불가능성 증명】                                          │
│  ────────────────────────────────                                       │
│                                                                         │
│  정리: 일반적인 연산 집합 O에 대해 안전성 문제는 결정 불가능(undecidable)이다.      │
│                                                                         │
│  증명 방법: 튜링 머신의 정지 문제(Halting Problem)로 환원                          │
│                                                                         │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  튜링 머신 TM                                                   │ │
│  │  ────────────                                                   │ │
│  │  • 상태 집합 Q = {q₀, q₁, ..., q_f}                                 │ │
│  │  • 테이프 기호 Σ = {□, a, b, ...}                                   │ │
│  │  • 전이 함수 δ: Q × Σ → Q × Σ × {L, R}                              │ │
│  │                                                                        │ │
│  │  접근 제어 행렬로 모델링:                                               │ │
│  │  • 각 상태 qᵢ를 주체로 모델링                                           │ │
│  │  • 테이프 셀을 객체로 모델링                                            │ │
│  │  • 전이 δ를 권한 연산으로 모델링                                         │ │
│  │                                                                        │ │
│  │  만약 안전성 문제가 결정 가능하다면 → 튜링 머신의 정지 문제도 해결 가능         │ │
│  │  하지만 정지 문제는 결정 불가능이므로 → 안전성 문제도 결정 불가능               │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  【실무적 의미】                                                          │
│  ────────────                                                          │
│                                                                         │
│  1. 일반적으로 "이 시스템은 안전한가?"를 자동으로 판단할 수 없음                     │
│  2. 제한된 연산 집합에 대해선 안전성이 결정 가능할 수 있음                          │
│     - 예: mono-conditional 연산만 허용하면 안전성은 O(n³) 시간에 결정 가능            │
│  3. 실무로는 모의 공격(penetration testing), 형식 검증(formal verification),   │
│     정책 검토(policy review)로 안전성을 평가                                 │
│                                                                         │
│  【안전성 평가 접근법】                                                     │
│  ────────────────                                                      │
│                                                                         │
│  접근법                     설명                  예시                      │
│  ──────                     ────                  ────                      │
│  정적 분석(Static Analysis)   소스 코드, 정책 검사    SELinux policy check,  │
│                                                     SPDX analysis         │
│  동적 테스트(Dynamic Test)    실행 중 시도         penetration testing,   │
│                                                     fuzzing              │
│  형식 검증(Formal Verif.)     수학적 증명           model checking,       │
│                                                     theorem proving      │
│  감사(Auditing)               로그 분석            access log analysis,   │
│                                                     SIEM                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** HRU 정리는 보안 엔지니어링에 근본적 한계를 지정한다. 아무리 정교한 접근 제어 정책이라도, 미래의 모든 가능한 상태를 예측하여 "이 시스템은 절대 root 권한 탈취가 불가능하다"고 증명할 수는 없다. 그러나 실무로는 제한적 환경에서는 안전성을 증명할 수 있다. 예를 들어, 권한 전이(Transfer Right)를 허용하지 않는 시스템에서는 권한 확산이 제한되어 안전성 분석이 단순해진다. 또한 모의 공격(pen-test)과 감사 로그 분석을 통해 실무적 안전성을 평가한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: ACL vs Capability vs RBAC

| 비교 항목 | ACL (Access Control List) | Capability | RBAC (Role-Based) |
|:---|:---|:---|:---|
| **저장 단위** | 객체별 | 주체별 | 역할(Role)별 |
| **권한 부여 방식** | 객체의 ACL에 주체 추가 | 주체의 Capability에 객체 추가 | 사용자를 역할에 할당 |
| **검색 복잡도** | 접근자 확인: O(1) <br>권한 열거: O(k) | 권한 확인: O(1) <br>접근자 찾기: O(n) | 역할-권한 매핑: O(r) |
| **권한 회수** | ACL에서 제거만 | 복사본 추적 필요 | 역할에서 제거 |
| **분산 시스템** | 중앙 집중적 ACL 서버 필요 | 토큰 전달 용이 | 디렉터리 서버 필요 |
| **대표적 구현** | Unix inode, NTFS ACL | Kerberos, POSIX cap | Active Directory, LDAP |

### 비교 2: Unix 파일 권한 vs Windows ACL

| 비교 항목 | Unix/Linux (DAC) | Windows NTFS (ACL) |
|:---|:---|:---|
| **권한 모델** | Owner/Group/Other 3그룹 | ACE(Access Control Entry) 리스트 |
| **권한 종류** | r, w, x, s, t 등 10비트 | Read, Write, Modify, Full Control 등 세분화 |
| **상속(Inheritance)** | 디렉터리의 setgid, umask로 제어 | 명시적 상속 플래그 |
| **기본 거부(Deny)** | 없음 (명시적 허용만) | ACE에서 Deny 우선 |
| **복잡도** | 단순, 관리 쉬움 | 복잡, 세분화된 제어 가능 |

```text
┌─────────────────────────────────────────────────────────────────────────┐
│           Unix 파일 권한 vs Windows ACL 비교                             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【Unix/Linux: 12비트 모드 (9비트 권한 + 3비트 특수)】                       │
│  ────────────────────────────────────────────────────                    │
│                                                                         │
│  $ ls -l file.txt                                                       │
│  -rwxr-xr--  1  user1  group1  1234  Jan 1 12:34  file.txt               │
│   ││││││││  │                                                            │
│   ││││││││  └─ Other(타인): r-- (읽기만)                                  │
│   ││││││└──── Group(그룹): r-x (읽기+실행)                                │
│   │││└─────── Owner(소유자): rwx (전체)                                  │
│   ││└───────── SetUID(4000), SetGID(2000), Sticky(1000)                  │
│   │            (rwsr-xr-x 처럼 s가 나타남)                                │
│   └─────────── 파일 타입(-: 일반, d: 디렉터리, l: 심볼릭 링크)                 │
│                                                                         │
│  # 권한 변경                                                            │
│  $ chmod 754 file.txt   # rwxr-xr-- = 754                               │
│  $ chmod u+x,g-w,o=r file.txt  # 소유자 실행 추가, 그룹 쓰기 삭제, 타인 읽기만   │
│                                                                         │
│  # ACL 확장 (getfacl, setfacl)                                           │
│  $ setfacl -m u:user2:rw file.txt  # user2에게 읽기/쓰기 추가               │
│  $ getfacl file.txt                                                     │
│  # file: file.txt                                                       │
│  # owner: user1                                                         │
│  # group: group1                                                        │
│  user::rwx                                                               │
│  user:user2:rw-                                                         │
│  group::r-x                                                              │
│  other::r--                                                              │
│                                                                         │
│  【Windows NTFS: ACL (Access Control List)】                            │
│  ──────────────────────────────────────────────                          │
│                                                                         │
│  icacls "C:\Data\file.txt"                                               │
│  C:\Data\file.txt BUILTIN\Administrators:(F)                              │
│                       BUILTIN\Users:(R)                                  │
│                       DOMAIN\User1:(W)                                   │
│                                                                         │
│  # 권한: Full Control(F), Read(R), Write(W), Modify(M), Delete(D), etc. │
│                                                                         │
│  # ACE(Access Control Entry) 구조                                        │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  ACE Type        │  Trustee        │  Permissions                  │ │
│  │  ─────────        │  ───────        │  ───────────                  │ │
│  │  Allow           │  User1          │  Read, Write                 │ │
│  │  Deny            │  User2          │  Write, Delete                │ │
│  │  Allow (Inherited)│  Group1         │  Read, Execute                │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  # ACL 수정                                                            │
│  icacls "C:\Data\file.txt" /grant User2:(W)  # User2에게 쓰기 권한 부여    │
│  icacls "C:\Data\file.txt" /deny User3:(W)   # User3의 쓰기 거부          │
│                                                                         │
│  # 상속(Inheritance) 제어                                                 │
│  icacls "C:\Data\dir\" /inheritance:d  # 상속 비활성화                     │
│  icacls "C:\Data\dir\" /grant *:(OI)(CI)F  # 자식 개체, 컨테이너 모두 Full    │
│  # OI = Object Inherit, CI = Container Inherit                           │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Unix는 3비트(rwx) × 3그룹(Owner/Group/Other)으로 단순하지만, 세분화된 제어가 어렵다. 예를 들어, "User1과 User2에게만 읽기 권한을 주고 싶다"는 요구사항을 만족시키려면 그룹을 생성하거나 ACL을 확장해야 한다. 반면 Windows ACL은 각 사용자마다 개별적으로 권한을 부여할 수 있어 유연하지만, 관리 복잡도가 높다. 특히 "Deny" ACE는 허용(Allow)보다 우선 순위가 높아, 사용자가 여러 그룹에 속해 있을 때 권한 계산이 복잡해질 수 있다.

### 과목 융합 관점

- **컴퓨터 구조**: CPU의 Ring 레벨과 MMU의 페이지 테이블 권한 비트는 접근 제어 행렬의 하드웨어적 구현이다. CPL(Current Privilege Level)에 따라 커널 영역 접근을 제어한다.
- **데이터베이스**: DBMS는 테이블, 행(Row), 열(Column) 수준에서 세분화된 접근 제어를 제공한다. GRANT/REVOKE 문은 접근 제어 행렬의 SQL 인터페이스다.
- **네트워크**: 방화벽 규칙도 일종의 접근 제어 행렬이다. 출발지 IP, 목적지 포트, 프로토콜을 주체/객체로 보고, 허용/거부를 권한으로 볼 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 파일 서버에서 디렉터리별 접근 제어 정책 설계**: 기업의 파일 서버에서 인사팀, 재무팀, 개발팀이 각자의 문서를 공유하되, 경영진만 모든 문서에 접근 가능해야 한다. 아키텍트는 (1) 그룹 생성(hr, finance, dev, management), (2) 각 팀 디렉터리에 그룹별 rwx 권한 부여, (3) 경영진 그룹은 모든 디렉터리에 rwx 부여, (4) setgid를 설정하여 그룹 파일 생성 시 그룹 소유권 유지, (5) ACL 확장으로 특정 사용자에게 예외적 권한 부여(medical_records 디렉터리에 의사만 접근 등)의 다층적 접근 제어를 설계해야 한다.

2. **시나리오 — 컨테이너 환경에서의 최소 권한 원칙(Least Privilege) 구현**: Docker 컨테이너로 웹 애플리케이션을 실행 시, 컨테이너가 호스트의 `/var/log`, `/etc/shadow`에 접근하지 못하게 해야 한다. 아키텍트는 (1) rootless 모드로 비root 사용자로 컨테이너 실행, (2) read-only 루트 파일 시스템 마운트, (3) capabilities 박탈(--cap-drop ALL), (4) 필요한 최소 capabilities만 추가(--cap-add NET_BIND_SERVICE), (5) AppArmor/SELinux 프로필로 허용 경로 제한, (6) 네임스페이스 격리(user, pid, net)로 호스트 자원 완전 분리 등의 계층적 보호를 적용해야 한다.

3. **시나리오 — IAM(Identity and Access Management) 설계**: 클라우드 환경(AWS/Azure/GCP)에서 다중 계정(AWS Organizations)으로 수백 개의 서비스 계정 관리. 아키텍트는 (1) 중앙 ID 공급자(IdP)와 SSO(Single Sign-On) 통합, (2) 역할(Role) 기반 접근 제어로 최소 권한 부여, (3) MFA(Multi-Factor Authentication) 강제, (4) 거버넌스 정책(Password rotation, Access review) 자동화, (5) 감사 로그(CloudTrail) SIEM 연동, (6) 임시 자격 증명(STS AssumeRole)으로 장기 키 사용 억제의 심화된 접근 제어 체계를 구축해야 한다.

### 도입 체크리스트

- **기술적**: 모든 객체에 명시적 ACL이 설정되어 있는가? "default deny"(명시적 허용 없으면 거부) 원칙이 적용되는가? 정기적 권한 리뷰(access review)가 자동화되어 있는가?
- **운영·보안적**: 관리자 계정이 일상 계정과 분리(SoD)되어 있는가? 권한 상승(privilege escalation) 시도가 감사 로그에 기록되는가? 비정상적 접근 시도(alert)가 실시간으로 모니터링되는가?

### 안티패턴

- **_chmod 777_ 남용**: 모든 사용자에게 쓰기 권한 부여는 악의적 사용자가 시스템 바이너리를 교체하거나 설정 파일을 변조할 수 있게 한다.
- **root 권한으로 CGI 실행**: 웹 서버가 CGI 스크립트를 root로 실행하면, 취약점이 있을 시 공격자가 시스템 전체를 장악할 수 있다. suEXEC/suPHP를 사용해야 한다.
- **감사 로그 미수집**: "권한 부여만 하고 사용 기록을 남기지 않으면" 침해 사고 시 원인 파악이 불가능하다. auditd, syslog를 활성화해야 한다.
- **DAC만 의존**: 소유자가 권한을 변경할 수 있는 DAC만으로는 랜섬웨어, 매크로 바이러스 등 악성코드로부터 보호할 수 없다. MAC(SELinux/AppArmor)를 함께 사용해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 | 최적화 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | chmod 777 남용, 권한 관리 불가 | ACL/RBAC 구축, 최소 권한 원칙 | **데이터 유출 사고 80% 감소** |
| **정량** | 감사 로그 미수집, 침해 시 원인 불명 | auditd, syslog, SIEM 도입 | **포렌식 분석 시간 95% 단축** |
| **정성** | 관리자 권한 남용, 내부자 위협 | SoD, MFA, 권한 리뷰 자동화 | **거버넌스 강화**, 규정 준수 |

### 미래 전망

- **제로 트러스트(Zero Trust)**: "신뢰할 네트워크는 없다"는 원칙하에 모든 접근을 실시간 검증한다. BeyondProd, SPIFFE/SPIRE는 서비스 간 인증에 mTLS와 x.509 인증서를 사용한다.
- **정책 기반 접근 제어(PBAC)**: ABAC(Attribute-Based Access Control)는 사용자, 환경, 리소스 속성(예: 시간대, 위치, 장치 유형)을 기반으로 동적으로 접근을 결정한다. XACML(OASIS 표준)이 이를 정의한다.
- **HW 기반 원격 증명(Remote Attestation)**: TPM(Trusted Platform Module)을 사용하여 시스템의 부팅 무결성(integrity)을 원격으로 증명한다. 클라우드에서 VM의 신뢰성을 검증하는 데 사용된다.

### 참고 표준

- **POSIX.1e (IEEE 1003.1e)**: ACL, 감사, 정보 라벨링 등 보안 API 표준 (취소되었으나 구현에 영향)
- **RFC 5533 (XACML 3.0)**: eXtensible Access Control Markup Language
- **NIST SP 800-162**: Guide to Attribute Based Access Control (ABAC) Model and Tensor Level of Abstraction
- **ISO/IEC 10181-3**: Security frameworks for open systems: Access control framework
