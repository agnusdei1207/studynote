# 614. Security Policy (보안 정책)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보안 정책(Security Policy)은 시스템의 자원 접근, 사용, 변경을 규제하는 규칙 집합으로, **"무엇이 허용되고 무엇이 금지되는가"**를 명시한 기준 문서이다.
> 2. **가치**: 기술적 보호 메커니즘(ACL, MAC)의 **목적**을 정의하며, 법적 준수(PCI-DSS, ISO 27001), 거버넌스, 위험 관리의 기반이 된다. 정책이 없으면 보안 도구는 방향을 잃는다.
> 3. **융합**: 조직의 보안 정책은 기술적(시스템 설정), 물리적(출입 통제), 행정적(사용자 계약) 3계층으로 구성되며, SELinux policy, 방화벽 규칙, AWS IAM policy 등이 기술적 구현에 해당한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**보안 정책(Security Policy)**은 **정보 자산을 보호하기 위해 조직이 수립한 명시적 규칙, 지침, 절차의 총체**다. 기술적으로는 "시스템이 어떤 상태에서 어떤 연산을 허용/거부할 것인가"를 형식화한 문서이다. 예를 들어, "모든 사용자는 8자 이상, 영문/숫자/특수문자 조합 비밀번호를 사용해야 한다", "외부에서의 SSH 접속은 키 인증만 허용하고 비밀번호 인증은 금지한다", "개인정보가 포함된 데이터는 암호화하여 저장해야 한다" 등이 보안 정책의 예시다. 보안 정책은 **기술적 통제(Technical Control)**, 물리적 통제(Physical Control), 행정적 통제(Administrative Control)** 3가지 계층으로 구성되며, 위험 평가, 비즈니스 요구, 법적 요구사항, 모범 사례(Best Practice)를 반영하여 수립된다.

### 💡 비유: 건물 보안 규칙

보안 정책은 **건물의 보안 규칙**과 같다. 건물에 "모든 출입구는 CCTV로 녹화해야 한다", "야간에는 출입문을 잠가야 한다", "방문객은 반드시 본인 확인 후 출입증을 발급받아야 한다"와 같은 규칙이 보안 정책이다. 이 규칙이 있어야 보안 경비원이 "문을 열어줄까?"라는 요청에 대해 "규칙에 따라 거절해야 한다"고 판단할 수 있다. 규칙이 없으면 경비원마다 다르게 판단하여 보안이 구멍이 된다. 마찬가지로, IT 시스템에도 "이 포트를 열어야 할까?", "이 사용자에게 권한을 줘야 할까?" 같은 질문에 대해 일관되게 답할 수 있는 기준이 보안 정책이다.

### 등장 배경

1. **1970-80대: 초기 컴퓨터 보안 정책**: 미국 국방부(DoD)이 Trusted Computer System Evaluation Criteria(TCSEC, Orange Book)를 발표하여, 보안 등급(D ~ A1)별 정책 요구사항을 정의했다.
2. **1990s: 인터넷 보안 정책의 필요성**: 인터넷 확산으로 바이러스, 해킹, 스팸이 급증하자 기업들이 보안 정책 수립이 필수적이 되었다. BS 7799(영국), ISO 17799(국제) 표준이 제정되었다.
3. **2000s: 규정 준수 중심의 정책**: Gramm-Leach-Bliley Act(GLBA), Health Insurance Portability and Accountability Act(HIPAA), PCI-DSS(카드 산업) 등 법적 요구사항이 보안 정책에 반영되었다.
4. **2010s-현재: 클라우드/DevSecOps 시대의 정책**: 클라우드 환경, DevOps, 마이크로서비스에서 보안을 "코드화(Policy as Code)"로 관리한다. OPA(Open Policy Agent), Sentinel, TerraformOPA 등이 등장했다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                  보안 정책의 발전 역사                                   │
├─────────────────────────────────────────────────────────────────────────┤

│                                                                         │

│  • 영국 표준(BS 7799) → 국제 표준(ISO/IEC 17799)                         │

│                                                                         │

│  ────────────────────────────────                                        │
│  • OPA(Open Policy Agent), Rego 언어                                      │
│  • Sentinel(Palo Alto), Pony                                             │

│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 정의 | 예시 | 비유 |
|:---|:---|:---|:---|
| **Strategic Policy(전략적)** | 최고 경영진 승인의 방향성 문서 | "고객 데이터는 기밀로 취급한다" | 회사의 가치관 |
| **Tactical Policy(전술적)** | 부서별 구체적 규칙 | "개인정보는 AES-256 암호화" | 부서 지침 |
| **Technical Policy(기술적)** | 시스템 설정 명세 | "SSH PasswordAuthentication no" | 방화벽 규칙 |
| **Procedure(절차)** | 단계별 수행 방법 | "비밀번호 변경 90일마다" | 매뉴얼 |
| **Guideline(지침)** | 권장 사례 | "비밀번호 관리자 사용 권장" | 가이드북 |

### 보안 정책 계층 구조

보안 정책은 계층적 구조를 가지며, 상위 정책이 하위 정책의 방향을 결정한다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│            보안 정책 계층 구조                                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【계층 1: 전략적 정책(Strategic Policy)】                                 │
│  ────────────────────────────────────                                   │
│  • 목적: 조직의 보안 목표, 방향성, 원칙 정의                                 │
│  • 대상: 최고 경영진, 이사회                                                │
│  • 예시:                                                              │
│    "본사는 고객의 개인정보 보호를 최우선 가치로 삼는다."                          │
│    "모든 직원은 정보 보안 의무를 준수해야 한다."                               │
│  • 형식: 1~2페이지의 선언문, 이사회 승인                                     │
│                                                                         │
│  【계층 2: 전술적 정책(Tactical Policy)】                                   │
│  ────────────────────────────────────                                    │
│  • 목적: 부서별 보안 기준, 책임, 위반 시 처벌                                   │
│  • 대상: 부서장, 보안 담당자                                               │
│  • 예시:                                                              │
│    "개인정보는 AES-256으로 암호화하여 저장해야 한다."                            │
│    "외부에서의 SSH 접속은 공개키 인증만 허용한다."                              │
│  • 형식: 5~20페이지의 문서, CISO(최고 보안 책임자) 승인                      │
│                                                                         │
│  【계층 3: 절차(Procedure)】                                               │
│  ──────────────────────────────                                          │
│  • 목적: 단계별 수행 방법, 체크리스트                                         │
│  • 대상: 일반 직원, 시스템 관리자                                           │
│  • 예시:                                                              │
│    "1. 사용자 생성 신청서 작성 → 2. 부서장 승인 → 3. HR 등록 → 4. IT 계정 생성"│
│  • 형식: 매뉴얼, 흐름도, 워크시트                                            │
│                                                                         │
│  【계층 4: 기술적 설정(Technical Configuration)】                             │
│  ────────────────────────────────────────                                  │
│  • 목적: 시스템별 명세, 파라미터                                             │
│  • 대상: 시스템 관리자, DevOps                                             │
│  • 예시:                                                              │
│    $ cat /etc/ssh/sshd_config                                            │
│    PasswordAuthentication no                                            │
│    PermitRootLogin no                                                   │
│    AllowGroups sshusers                                                │
│  • 형식: 설정 파일, 스크립트, Infrastructure as Code                        │
│                                                                         │
│  【계층 5: 감사 및 모니터링(Audit & Monitoring)】                              │
│  ────────────────────────────────────────                                  │
│  • 목적: 정책 준수 여부 확인, 위반 탐지                                       │
│  • 대상: 감사팀, SOC(Security Operations Center)                          │
│  • 예시:                                                              │
│    - SIEM 대시보드: "PasswordAuthentication no 위반 횟수"                   │
│    - 취약점 스캔: SSH 약한 버전 사용 탐지                                     │
│  • 형식: 대시보드, 로그, 알림                                               │
│                                                                         │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  계층                │  승인 권한         │  갱신 주기      │    │
│  │  ────                │  ────────         │  ────────      │    │
│  │  Strategic         │  이사회          │  연 1회        │    │
│  │  Tactical          │  CISO            │  분기 1회       │    │
│  │  Procedure         │  부서장         │  반기 1회       │    │
│  │  Technical         │  보안 팀         │  필요시         │    │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 보안 정책은 전략적(목표) → 전술적(규칙) → 절차(방법) → 기술적(설정) → 감사(검증)으로 계층화된다. 상위 정책이 하위 정책의 방향을 결정한다. 예를 들어, 전략적 정책이 "고객 데이터 기밀을 최우선으로 한다"고 선언하면, 전술적 정책은 "개인정보는 AES-256 암호화"로 구체화되고, 기술적 정책은 `openssl aes-256-cbc` 명령어로 설정된다. 감사 계층은 이 준수 여부를 SIEM, log 분석으로 확인한다. 각 계층은 서로 다른 주기로 갱신된다. 전략적 정책은 1년에 한 번(이사회), 전술적 정책은 분기별(CISO), 기술적 설정은 필요 시(취약점 발견 등) 즉시 갱신된다.

### 보안 정책 구성 요소

ISO/IEC 27001:2013은 정보 보안 경영 체계(ISMS)의 14개 관영역을 정의한다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│         ISO/IEC 27001:2013 정보 보안 관영체계 14개 영역                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  번호  │  영역                        │  정책 예시                  │ │
│  │  ────  │  ────                        │  ────────                  │ │
│  │  1.    │  보안 정책(Security Policy)  │  조직의 보안 방향 선언        │ │
│  │  2.    │  정보 보안 조직화              │  CISO, 보안팀 역할           │ │
│  │  3.    │  인적 보안(Human Security)    │  교육, 배경 조사, NDA        │ │
│  │  4.    │  자산 관리(Asset Management)   │  하드웨어/소프트웨어 목록      │ │
│  │  5.    │  접근 통제(Access Control)      │  ID/권한 부여 절차           │ │
│  │  6.    │  암호화(Cryptography)          │  암호화 알고리즘, 키 관리       │ │
│  │  7.    │  물리적 보안(Physical Security) │  출입 통제, CCTV            │ │
│  │  8.    │  운영 보안(Operational Security) │  백업, 로깅, 변경 관리      │ │
│  │  9.    │  통신 보안(Comms Security)      │  VPN, 방화벽, 전송 암호화    │ │
│  │  10.   │  시스템 취득/개발/유지         │  SDLC, 취약점 관리          │ │
│  │  11.   │  공급망 관리(Supplier Rel.)     │  보안 요구사항 계약 포함      │ │
│  │  12.   │  정보 보안 사고 대응          │  리포트 라인, 재난 복구      │ │
│  │  13.   │  업무 연속성 계획(BCP)        │  DRP, RPO/RTO             │ │
│  │  14.   │  준수(Compliance)             │  법적 요구사항 충족          │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ISO 27001은 정보 보안 관리 체계(ISMS)를 14개 영역으로 체계화한다. 각 영역은 **Statement of Applicability(SoA)** 문서로 "이 표준을 우리 조직에 어떻게 적용할 것인가"를 명시한다. 예를 들어, 접근 통제(영역 5)에서는 "모든 사용자는 고유한 ID를 가지며, 6개월마다 비밀번호를 변경해야 한다"는 정책을 수립하고, 기술적으로는 `/etc/login.defs`의 `PASS_MAX_DAYS`를 180으로 설정한다. 자산 관리(영역 4)에서는 "모든 자산(서버, DB, 문서)은 자산 등록부에 기록되어야 하며, 중요 자산은 분류(공용/내용/기바)하여 보안 수준을 달리해야 한다"는 정책을 수립한다.

### Policy as Code (PaC)

최근 보안 정책을 코드로 관리하는 **Policy as Code(PaC)**가 등장했다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│            Policy as Code (PaC) 개념과 구현                               │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【개념】                                                                │
│  ────────────                                                           │
│  • 보안 정책를 사람이 읽는 문서가 아니라, 머신이 검증 가능한 코드로 작성             │
│  • Git으로 버전 관리, Pull Request로 변경 승인                               │
│  • CI/CD 파이프라인에 통합하여 배포 전 정책 검증                                   │
│                                                                         │
│  【구현 도구】                                                            │
│  ────────────                                                           │
│                                                                         │
│  1. OPA (Open Policy Agent)                                             │
│     - Rego 언어로 정책 작성                                                 │
│     - Kubernetes, Terraform, APIs에 정책 적용                             │
│                                                                         │
│  2. Sentinel (Palo Alto Networks)                                        │
│     - 규칙 기반 보안 정책 플랫폼                                            │
│     - Cloud Infrastructure 정책 관리                                     │
│                                                                         │
│  3. Pony                                                               │
│     - AWS/ 리눅스용 오픈소스 policy-as-code 엔진                           │
│     - Cloud Custodian 기반                                               │
│                                                                         │
│  4. Conftest                                                           │
│     - Infrastructure as Code 검증                                        │
│     - Terraform, Kubernetes, ARM 템플릿 검증                             │
│                                                                         │
│  【OPA Rego 예시: Kubernetes Pod 정책】                                    │
│  ────────────────────────────────────────                                 │
│                                                                         │
│  package kubernetes.admission                                            │
│                                                                         │
│  # "privileged" 파드를 금지하는 정책                                         │
│  deny[msg] {                                                            │
│      input.kind == "Pod"                                               │
│      input.spec.containers[_].securityContext.privileged == true       │
│      msg := "Privileged pods are not allowed"                           │
│  }                                                                     │
│                                                                         │
│  # "latest" 태그 사용을 금지하는 정책                                          │
│  deny[msg] {                                                            │
│      input.kind == "Pod"                                               │
│      input.spec.containers[_].image endswith ":latest"                 │
│      msg := "Latest tag is not allowed"                                 │
│  }                                                                     │
│                                                                         │
│  【TerraformOPA 예시: AWS 보안 정책 검증】                                 │
│  ──────────────────────────────────────────                               │
│                                                                         │
│  # S3 버킷이 암호화되어 있는지 검증                                           │
│  terraform {                                                            │
│    deny[msg] {                                                          │
│      resource.aws_s3_bucket                                            │
│      not resource.server_side_encryption_configuration.enabled             │
│      msg := "S3 bucket must have encryption enabled"                     │
│    }                                                                    │
│  }                                                                     │
│                                                                         │
│  【Policy as Code의 이점】                                               │
│  ────────────────────────────                                           │
│  • 버전 관리: Git 커밋으로 정책 변경 이력 추적 가능                             │
│  • 자동화: CI/CD로 배포 전 자동 검증                                            │
│  • 일관성: 수동 검증의 인적 실수 예방                                          │
│  • 협업: Pull Request으로 다양한 팀이 정책 검토                                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Policy as Code는 보안 정책을 문서(Word/PDF)가 아니라 코드(OPA Rego, Sentinel Policy, YAML 등)로 작성하여, 머신이 자동으로 검증하게 한다. 예를 들어, "모든 S3 버킷은 암호화되어야 한다"는 정책을 문서로만 작성하면, 실제로 암호화되지 않은 버킷이 생성되어도 모를 수 없다. 하지만 TerraformOPA로 `aws_s3_bucket` 리소스에 `server_side_encryption_configuration.enabled = false`인 경우를 검증하는 코드를 작성하면, `terraform plan`/`apply` 시 자동으로 거부된다. OPA는 Kubernetes Admission Controller로 통합되어, `kubectl apply` 시 파드/서비스 생성을 허용/거부할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 보안 정책 표준

| 표준 | 목적 | 적용 범위 | 대표적 요구사항 |
|:---|:---|:---|:---|
| **ISO/IEC 27001** | 정보 보안 경영 체계(ISMS) | 모든 산업 | 114개 통제, 경영 프로세스 |
| **PCI-DSS** | 카드 결제 데이터 보안 | 금융, 결제 | 12 요구사항, 400+ sub-requirements |
| **NIST SP 800-53** | 연방 정보 시스템 보안 | 미국 정부/ contractors | 20개 가족, 수백 개 통제 |
| **SOC 2** | 서비스 조직 보안 | SaaS, 클라우드 | 5개 원칙(TSC), 70+ criteria |
| **HIPAA** | 의료 정보 보호 | 의료, 보험 | Privacy, Security Rules |

### 비교 2: 정책 유형별 특징

```text
┌─────────────────────────────────────────────────────────────────────────┐
│            정책 유형별 특징 비교                                        │

│                                                                         │
│  ┌───────────────────────────────────────────────────────────────────┐ │

│  │  (Regulatory) │  법적 요구사항                   │  변경해야 한다"      │ │
│  │                                                         │                     │ │
│  │  지침(Guideline)│  권장 사례, 강제력 없음             │  "비밀번호 관리자    │ │

│  │                                                         │                     │ │

│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 과목 융합 관점

- **네트워크**: 방화벽 규칙, NACL(Network ACL), Security Group도 일종의 보안 정책이다. AWS Security Group은 "포트 22에서 203.0.113.0/24로의 접속을 허용"하는 규칙의 집합이다.
- **운영체제**: SELinux policy, AppArmor profile은 OS 수준에서 접근 제어 정책을 강제한다. 예를 들어, `httpd_t` 도메인은 `httpd_sys_content_t` 파일만 읽을 수 있다는 규칙이 정책으로 코딩되어 있다.
- **클라우드**: AWS IAM Policy, Azure RBAC Role Definition은 JSON 형식의 보안 정책이다. `{"Effect": "Allow", "Action": "s3:Get*", "Resource": "*"}`는 특정 S3 작업을 허용하는 정책이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 스타트업 보안 정책 수립**: 직원 30명의 스타트업에서 보안 정책을 처음 수립. 아키텍트는 (1) 경영진과 인터뷰하여 비즈니스 목표 이해, (2) 자산 분류(공용/내용/기밀), (3) 위험 평가(Risk Assessment)로 중요 자산 식별, (4) 정책 문서 작성(전략/전술/절차/기술), (5) 직원 교육, (6) 정기적(분기) 검토 및 개선. 특히 (7) "BYOD(개인 장치 반입) 금지", "모든 코드는 Git으로 관리, PR 승인 후 merge", "생산 DB는 VPN 내에서만 접근" 같은 실용적 규칙을 포함한다.

2. **시나리오 — PCI-DSS 준수를 위한 정책 수립**: 온라인 쇼핑몰에서 카드 결제를 처리하는 기업. 아키텍트는 (1) PCI-DSS 12요구사항 분석, (2) 카드 데이터는 저장 및 전송 시 암호화(SSL/TLS, AES-256), (3) 카드 데이터를 로그에 남기지 않음, (4) 취약점 스캔을 분기별 수행, (5) 네트워크 구획(할당된 서브넷에 DB 격리), (6) 위반 시 심사(Audit) 로그 1년 보관, (7) QSA(Qualified Security Assessor) 심사 대응 절차 수립 등의 정책을 구축한다.

3. **시나리오 — Policy as Code로 EKS 클러스터 보안 강화**: Kubernetes(EKS) 클러스터에서 OPA/Gatekeeper를 통한 정책 enforcement. 아키텍트는 (1) Rego로 "privileged 파드 금지", "latest 태그 금지", "root 사용자 금지" 정책 작성, (2) Gatekeeper Webhook 배포, (3) `kubectl apply` 시 자동 검증, (4) 위반 시 거부 메시지 반환, (5) CI/CD 파이프라인에 `conftest` 통합, (6) GitOps로 정책 버전 관리, (7) 정기적 정책 리뷰(Pull Request)를 통해 팀 협업으로 정책 개선.

### 도입 체크리스트

- **기술적**: 보안 정책이 법적 요구사항(PCI-DSS, HIPAA, GDPR 등)을 충족하는가? 정책이 최신 취약점을 반영하고 있는가? 모든 중요 시스템에 정책이 적용되는가?
- **운영·보안적**: 정책 위반 시 조치 절차(Disciplinary Action)가 정의되어 있는가? 정기적 보안 교육이 이루어지는가? 정책 준수 여부를 감사(Audit)하는가?

### 안티패턴

- **"책장용 정책"**: 문서만 작성하고 실제로는 시행하지 않으면, 위반 시 처벌의 근거가 되지 않는다. 정책은 **기술적 설정과 연계**되어야 한다.
- **규정 준수(Compliance)에만 집중**: 보안은 규정 준수가 목적이 아니라, 리스크 관리가 목적이다. PCI-DSS를 준수해도 실제 보안이 취약하면 무의미하다.
- **정책 과잉(Over-policy)**: "모든 포트를 차단", "모든 관리자 행위를 승인" 같은 과도한 정책은 업무 효율을 떨어뜨리고, 직원들이 우회 경로를 찾게 만든다. **리스크 기반 접근**이 필요하다.
- **정책 업데이트 부재**: 보안 위협이 변하는데 정책은 2010년 그대로라면, 새로운 공격(랜섬웨어, API 공격)에 대응할 수 없다. 정기적(최소 연 1회) 검토가 필요하다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 | 최적화 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 보안 사고 시 대응 절차 미정의 | 문서화된 정책, 명확한 책임 소재 | **대응 시간 70% 단축** |
| **정량** | 규정 준수 불이행으로 벌금/과태료 risk | 정책 준수, 감사 로그 | **규정 준수 100% 달성** |
| **정성** | 보안 인식 부족, 내부자 위험 높음 | 정기 교육, 명확한 정책 | **보안 문화 강화**, 위험 감소 |

### 미래 전망

- **Automated Compliance Monitoring**: AWS Config, Azure Security Center, Google Cloud Security Command Center가 실시간으로 정책 준수 여부를 모니터링하고, 위반 시 자동으로 수정할 수 있다.
- **AI-based Policy Optimization**: 머신 러닝이 로그 데이터를 분석하여, 비정상적 접근 패턴을 탐지하고 정책을 제안한다. 예를 들어, "밤 10시 이후의 DB 접속은 이상하다"는 패턴을 학습하여 정책을 자동 생성한다.
- **Zero Trust Policy**: "신뢰할 사용자/장치는 없다"는 원칙하에, 모든 접근을 실시간 위험 평가(기기, 장치, 행동)로 동적으로 결정하는 정책이 확산되고 있다.

### 👶 어린이를 위한 3줄 비유 설명

**개념**: 보안 정책은 **학교 교칙** 같아요! "체육관 신발만 신고", "복도에서 뛰지 않기", "욕심 금지"처럼 우리 모두가 지켜야 할 규칙이에요.

**원리**: 교칙이 있어서 "이 친구는 체육관에 들어갈 수 있나?" 같은 질문에 대해 "교칙에 따라 들어갈 수 있어/없어"라고 대답할 수 있어요. 교칙이 없으면 선생님마다 다르게 대답해서 혼란이 생겨요.

**효과**: 교칙이 있어서 모두가 똑같이 기준을 따를 수 있어요! 친구가 규칙을 어겨도 선생님이 "교칙 위반이야"라고 제재할 수 있고, 친구도 "내가 잘못했구나"라고 알 수 있어요.
