# 낙관적 병행성 제어 (Optimistic Concurrency Control, OCC)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 낙관적 병행성 제어 (OCC)는 트랜잭션 수행 동안 락을 걸지 않고 자유롭게 작업을 진행한 후, 종료 시점에 다른 작업과의 충돌 여부를 확인(Validation)하여 문제가 없을 때만 반영하는 병행 제어 기법이다.
> 2. **가치**: 데이터 경합이 낮은 환경에서 락 획득/해제 오버헤드와 데드락 발생 가능성을 원천 제거하여 시스템 전체의 처리량(Throughput)을 비약적으로 향상시킨다.
> 3. **융합**: 데이터베이스 트랜잭션 격리, 분산 시스템의 MVCC (Multi-Version Concurrency Control), 소프트웨어 트랜잭셔널 메모리 (STM)의 근간 논리이며, 대규모 트래픽을 처리하는 현대 웹 아키텍처의 필수 요소다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 낙관적 병행성 제어 (OCC)는 "대부분의 트랜잭션은 서로 충돌하지 않을 것"이라는 낙관적인 가정을 기반으로 한다. 락을 사용하는 비관적 제어와 달리, 읽기/연산 단계에서는 제약이 없으며 오직 반영(Commit) 직전에만 검증을 수행한다.

- **💡 비유**: OCC는 **"수정 가능한 공유 문서 편집"**과 같다. 내가 문서를 고칠 때 다른 사람이 못 들어오게 암호를 거는 것이 아니라, 일단 내 컴퓨터에서 마음껏 고친다. 다 고친 후 저장 버튼을 누를 때, "그 사이에 다른 사람이 문서를 바꿨는지" 확인한다. 아무도 안 바꿨으면 내 내용이 저장되고, 누군가 바꿨다면 내 수정을 포기하고 다시 고쳐야 한다.

- **등장 배경**:
  1. **비관적 제어의 비용**: 락을 거는 행위 자체가 CPU 사이클을 소모하며, 특히 읽기 위주의 작업에서 락 경합은 심각한 성능 저하를 초래한다.
  2. **데드락 위험**: 여러 자원을 동시에 점유해야 하는 복잡한 로직에서 비관적 제어는 교착 상태 (Deadlock)를 유발할 확률이 높다.
  3. **분산 환경의 확장성**: 수천 대의 서버가 연결된 환경에서 전역 락(Global Lock)을 관리하는 것은 물리적으로 불가능에 가깝다.

- **📢 섹션 요약 비유**: "일단 해보고, 문제 있으면 그때 가서 사과(Rollback)하자"는 유연한 사고방식을 동기화에 적용한 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 3단계 과정

OCC는 모든 트랜잭션을 다음의 명확한 3단계로 분리하여 처리한다.

| 단계명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **1. 읽기 단계 (Read Phase)** | 데이터 조회 및 로컬 작업 | 원본 데이터의 복사본(Shadow Copy)에 연산 수행 | 로컬 버퍼 / 캐시 | 연습장에 답안 작성 |
| **2. 확인 단계 (Validation Phase)** | 충돌 여부 검사 | 내 작업 기간 동안 원본이 변경되었는지 확인 | Timestamp / Versioning | 제출 전 정답 대조 |
| **3. 쓰기 단계 (Write Phase)** | 결과의 영구 반영 | 검증 성공 시 원본에 덮어쓰기 수행 | 원자적 갱신 (Atomic Update) | 정답지에 옮겨 적기 |

+++

### OCC의 동작 메커니즘 (버전 관리 기반)

대부분의 현대적 OCC 구현은 데이터에 버전(Version)이나 타임스탬프를 부여하여 충돌을 감지한다.

```text
 ┌──────────────────────────────────────────────────────────────────┐
 │               낙관적 병행성 제어 (OCC) 프로세스                     │
 ├──────────────────────────────────────────────────────────────────┤
 │                                                                  │
 │   Shared Data [V.10]: Value = 100                                │
 │                                                                  │
 │   Transaction A:                     Transaction B:              │
 │   1. Read [V.10], Value=100          1. Read [V.10], Value=100   │
 │   2. Local Work: Value=110           2. Local Work: Value=90     │
 │                                                                  │
 │   3. Validate (A):                   3. Validate (B):            │
 │      Current V is 10? YES!              Current V is 10? NO!(11) │
 │   4. Write & V++:                    4. ABORT & RETRY            │
 │      Shared [V.11]: Value=110                                    │
 │                                                                  │
 └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 트랜잭션 A와 B가 동시에 작업을 시작했다. A가 먼저 검증을 통과하고 버전을 11로 올리며 데이터를 저장했다. 이후 B가 검증을 시도할 때, 자신이 읽었던 버전(10)과 현재 버전(11)이 다르다는 것을 발견한다. 이는 누군가 그 사이에 데이터를 바꿨음을 의미하므로, B는 자신의 작업을 무효화(Abort)하고 처음부터 다시 시작(Retry)해야 한다.

+++

### 검증 알고리즘 (Backward vs Forward)

1. **후향적 검증 (Backward Validation)**: 커밋하려는 트랜잭션이 이미 완료된 과거의 트랜잭션들과 충돌하는지 확인한다. (가장 널리 사용됨)
2. **전향적 검증 (Forward Validation)**: 현재 실행 중인 다른 트랜잭션들과 충돌할 가능성이 있는지 미리 확인한다.

```text
  [Backward Validation Condition]
  For every Transaction Tj that committed before Ti started:
  Check if WriteSet(Tj) intersects ReadSet(Ti)
```

**[다이어그램 해설]** 내가 읽고 있는 데이터가 내가 시작한 이후에 다른 사람에 의해 쓰여졌는지를 집합 연산으로 확인하는 과정이다. 이 검증 과정은 매우 빠르게 수행되어야 하며, 검증 단계 자체는 짧은 임계 구역(Critical Section)으로 보호되어야 한다.

- **📢 섹션 요약 비유**: 시험지 제출 직전에 "내가 문제를 풀고 있는 동안 정답이 바뀌지는 않았나"를 칠판(원본)과 대조해보는 과정과 같습니다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 비관적 제어 (PCC) vs 낙관적 제어 (OCC)

| 비교 항목 | 비관적 제어 (Pessimistic) | 낙관적 제어 (Optimistic) |
|:---|:---|:---|
| **기본 철학** | 충돌이 잦을 것이니 미리 막자 | 충돌이 드물 것이니 나중에 체크하자 |
| **동기화 도구** | 뮤텍스, 세마포어, 락 | 버전 번호, 타임스탬프, CAS |
| **데드락** | 발생 가능성 높음 | **원천적으로 발생 안 함** |
| **오버헤드** | 락 관리 비용 발생 | 충돌 시 재시도 (Retry) 비용 발생 |
| **최적 환경** | 쓰기 위주, 충돌 잦은 환경 | **읽기 위주, 충돌 적은 환경** |

### 과목 융합 관점

- **데이터베이스 (DB)**: MVCC (Multi-Version Concurrency Control)는 OCC의 확장판이다. 여러 버전을 유지하여 읽기 작업이 쓰기 작업에 의해 블로킹되지 않게 함으로써 고도의 병행성을 달성한다. (MySQL, PostgreSQL 필수 기술)
- **운영체제 (OS)**: 락-프리 (Lock-free) 자료구조의 핵심인 CAS (Compare-And-Swap) 연산은 단일 변수에 대한 가장 작은 단위의 OCC라고 볼 수 있다.

- **📢 섹션 요약 비유**: 비관적 제어가 "유료 주차장 입구에서 돈을 내는 것"이라면, 낙관적 제어는 "고속도로 하이패스를 통과하고 나중에 정산하는 것"과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 대규모 웹 커머스의 재고 관리**: 수만 명의 사용자가 상품 상세 페이지를 조회(Read)할 때는 락이 전혀 필요 없다. 하지만 구매 버튼을 누르는 순간(Write), 그 사이에 재고 수량이 변했는지 OCC로 확인한다. 대부분의 조회는 구매로 이어지지 않으므로 OCC가 훨씬 유리하다.
2. **시나리오 — 협업 툴 (Notion, Google Docs)**: 여러 사용자가 동시에 문서를 편집할 때, 에디터는 OCC 스타일로 변경 사항을 서버에 보내고 서버는 버전 충돌이 나면 병합(Merge)하거나 거절하는 방식으로 병행성을 관리한다.

### 도입 체크리스트
- **충돌 빈도**: 실시간 주식 거래 시스템처럼 동일 데이터에 초당 수천 번의 쓰기가 일어나는가? (이 경우 리트라이 비용 때문에 비관적 제어가 낫다)
- **트랜잭션 길이**: 한 트랜잭션의 작업 시간이 매우 긴가? (길수록 충돌 확률이 높아져 OCC 효율 저하)
- **데이터 일관성 수준**: 일시적인 불일치를 허용하는가, 아니면 엄격한 직렬화(Serializability)가 필요한가?

### 안티패턴
- **무한 리트라이**: 충돌이 잦은 환경에서 리트라이 로직에 지수 백오프 (Exponential Backoff)를 적용하지 않으면, 시스템 전체가 의미 없는 재작업으로 인해 자원이 고갈되는 라이브락 (Livelock)에 빠질 수 있다.

- **📢 섹션 요약 비유**: 충돌이 너무 잦은데도 "괜찮을 거야"라며 OCC를 고집하는 것은, 출퇴근 시간 붐비는 역에서 줄을 서지 않고 일단 밀고 들어가는 것과 같아 결국 더 큰 혼란을 야기합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (Locking) | 도입 후 (OCC) | 개선 효과 |
|:---|:---|:---|:---|
| **응답 시간** | 락 대기로 인한 지연 발생 | 대기 없는 즉시 처리 | 평균 레이턴시 (Latency) 감소 |
| **시스템 처리량** | 코어 수가 늘어도 락 경합으로 정체 | 코어 수에 비례한 성능 향상 | 확장성 (Scalability) 극대화 |
| **자원 효율성** | 컨텍스트 스위칭 오버헤드 | CPU 계산에만 집중 | 시스템 리소스 활용률 최적화 |

### 미래 전망
- **STM (Software Transactional Memory)**: 프로그래밍 언어 차원에서 OCC를 지원하여, 개발자가 직접 락을 관리하지 않고도 안전한 동시성 프로그래밍을 할 수 있는 환경이 점차 확대되고 있다.
- **분산 원장 기술**: 블록체인과 같은 분산 시스템에서도 트랜잭션의 병렬 처리를 위해 OCC 기반의 검증 모델이 성능 개선의 핵심으로 연구되고 있다.

- **📢 섹션 요약 비유**: 낙관적 제어는 서로를 믿고 자율적으로 행동하되 책임은 나중에 묻는, 성숙하고 고도화된 병렬 컴퓨팅의 상징과 같습니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **비관적 병행성 제어 (PCC)**: OCC와 대비되는 전통적 제어 방식.
- **CAS (Compare-And-Swap)**: OCC를 하드웨어적으로 구현하는 최소 단위 명령.
- **MVCC (Multi-Version Concurrency Control)**: 데이터베이스에서 OCC를 활용한 고성능 구현체.
- **라이브락 (Livelock)**: OCC 충돌이 극심할 때 발생하는 부작용.
