# 272. 락 웨이터 (Lock Waiter)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 락 획득을 위해 대기 중인 스레드
> 2. **가치**: 경합 측정, 공정성 분석의 핵심
> 3. **융합**: 락 홀더, 락 큐, 스케줄링과 연관

---

## Ⅰ. 개요

### 개념 정의

락 웨이터(Lock Waiter)는 **락을 획득하기 위해 대기 중인 스레드**를 의미한다. 경합 정도, 공정성, 스케줄링 분석에 핵심 정보다.

### 💡 비유: 줄서기
락 웨이터는 **줄서기**와 같다. 서비스를 기다리는 사람들. 줄의 길이를 보면 얼마나 기다려야 할지 알 수 있다.

### 락 웨이터 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                락 웨이터 구조                                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【대기 큐 구조】                                                       │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  락 홀더                대기 큐 (웨이터들)                           │ │
│  │  ┌─────┐               ┌─────┬─────┬─────┬─────┐                │ │
│  │  │ T1  │ ──────────►  │ T2  │ T3  │ T4  │ T5  │                │ │
│  │  │락 보유│              │대기 │대기 │대기 │대기 │                 │ │
│  │  └─────┘               └─────┴─────┴─────┴─────┘                │ │
│  │     ↑                       ↑       ↑       ↑       ↑            │ │
│  │     │                       │       │       │       │            │ │
│  │   진입 중                   T1이 unlock하면                        │ │
│  │                          T2가 다음 진입                             │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【대기 방식】                                                         │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 스핀 대기 (Spin Wait):                                          │ │
│  │     • busy loop                                                     │ │
│  │     • CPU 계속 사용                                                  │ │
│  │     • 짧은 대기에 적합                                                │ │
│  │                                                             │ │
│  │  2. 블록 대기 (Block Wait):                                         │ │
│  │     • 스케줄러에 의해 대기                                            │ │
│  │     • CPU 반환                                                       │ │
│  │     • 긴 대기에 적합                                                  │ │
│  │                                                             │ │
│  │  3. 하이브리드 (Adaptive):                                          │ │
│  │     • 짧으면 스핀                                                    │ │
│  │     • 길면 블록                                                      │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【웨이터 정보 활용】                                                   │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • 웨이터 수: 경합 정도                                               │ │
│  │  • 대기 시간: 공정성 측정                                              │ │
│  │  • 우선순위: 우선순위 스케줄링                                        │ │
│  │  • 스레드 ID: 디버깅                                                 │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                락 웨이터 상세                                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【구현별 대기 큐】                                                     │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  Linux mutex:                                                      │ │
│  │  struct mutex {                                                    │ │
│  │      atomic_long_t owner;                                          │ │
│  │      spinlock_t wait_lock;                                         │ │
│  │      struct list_head wait_list;  // 웨이터 리스트                  │ │
│  │  };                                                                │ │
│  │                                                             │ │
│  │  Linux rt_mutex (실시간):                                          │ │
│  │  // 우선순위 상속을 위한 웨이터 우선순위 추적                          │ │
│  │  struct rt_mutex_waiter {                                          │ │
│  │      struct rb_node tree_entry;                                    │ │
│  │      struct task_struct *task;                                     │ │
│  │      int prio;                                                     │ │
│  │  };                                                                │ │
│  │                                                             │ │
│  │  Windows SRWLock:                                                  │ │
│  │  // 웨이터 수만 추적, 큐는 커널 관리                                  │ │
│  │                                                             │ │
│  │  Java ReentrantLock:                                               │ │
│  │  // AQS (AbstractQueuedSynchronizer)                               │ │
│  │  // FIFO 큐로 웨이터 관리                                            │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【공정성 정책】                                                        │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. FIFO: 웨이터 큐에 들어온 순서대로                                   │ │
│  │     • 공정성 보장                                                    │ │
│  │     • ReentrantLock(fair=true)                                      │ │
│  │                                                             │ │
│  │  2. 불공정: 락 해제 시 누구나 획득 가능                                 │ │
│  │     • 처리량 향상                                                    │ │
│  │     • 기아 가능                                                      │ │
│  │     • ReentrantLock(fair=false) [기본]                              │ │
│  │                                                             │ │
│  │  3. 우선순위: 높은 우선순위 먼저                                       │ │
│  │     • 실시간 시스템                                                   │ │
│  │     • rt_mutex                                                       │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【성능 지표】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • 평균 웨이터 수: 경합 레벨                                           │ │
│  │  • 최대 웨이터 수: 피크 부하                                           │ │
│  │  • 평균 대기 시간: 지연 척도                                            │ │
│  │  • 최대 대기 시간: 최악의 경우                                          │ │
│  │  • 타임아웃 발생률: 설정 적절성                                          │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Java ReentrantLock 웨이터 확인】                                    │
│  ──────────────────                                                  │
│  import java.util.concurrent.locks.*;                               │
│                                                                     │
│  ReentrantLock lock = new ReentrantLock(true);  // fair             │
│                                                                     │
│  // 웨이터 수 확인                                                    │
│  int waiters = lock.getQueueLength();                               │
│  // 웨이터 존재 여부                                                   │
│  boolean hasWaiters = lock.hasQueuedThreads();                      │
│  // 특정 스레드 대기 여부                                               │
│  boolean waiting = lock.hasQueuedThread(someThread);                │
│                                                                     │
│  【Java ReentrantReadWriteLock】                                      │
│  ──────────────────                                                  │
│  ReentrantReadWriteLock rwlock = new ReentrantReadWriteLock();      │
│                                                                     │
│  int readWaiters = rwlock.getQueueLength();                          │
│  int writeWaiters = rwlock.getWaitingWriteCount();                   │
│                                                                     │
│  【Linux futex 웨이터】                                                │
│  ──────────────────                                                  │
│  // FUTEX_WAIT: 웨이터 등록                                            │
│  // FUTEX_WAKE: 웨이터 깨우기                                           │
│  // FUTEX_REQUEUE: 웨이터 이동                                          │
│                                                                     │
│  // 웨이터 수 확인                                                     │
│  cat /proc/<pid>/status | grep voluntary_ctxt_switches              │
│                                                                     │
│  【웨이터 덤프】                                                        │
│  ──────────────────                                                  │
│  // Linux: 모든 스레드 스택                                             │
│  echo t > /proc/sysrq-trigger                                       │
│                                                                     │
│  // Java: 스레드 덤프                                                  │
│  jstack <pid> | grep -A 20 "waiting for lock"                       │
│                                                                     │
│  // .NET: SOS 디버거                                                  │
│  // !syncblk                                                         │
│  // !threads                                                         │
│                                                                     │
│  【모니터링】                                                          │
│  ──────────────────                                                  │
│  // Java JMX                                                         │
│  ThreadMXBean threadBean = ManagementFactory.getThreadMXBean();     │
│  long[] threadIds = threadBean.getAllThreadIds();                   │
│  for (long id : threadIds) {                                        │
│      ThreadInfo info = threadBean.getThreadInfo(id);                │
│      if (info.getLockName() != null) {                              │
│          System.out.println("Thread " + id + " waiting for: " +     │
│              info.getLockName() + " owned by: " +                    │
│              info.getLockOwnerName());                               │
│      }                                                              │
│  }                                                                  │
│                                                                     │
│  【성능 분석】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  웨이터 수       처리량           평균 대기 시간                      │ │
│  │  ────────        ──────           ────────────                      │ │
│  │  0 (무경합)       100,000 ops      0 μs                             │ │
│  │  1-2             80,000 ops       10 μs                            │ │
│  │  3-5             60,000 ops       50 μs                            │ │
│  │  6-10            40,000 ops       200 μs                           │ │
│  │  >10             20,000 ops       1000+ μs                         │ │
│  │                                                             │ │
│  │  웨이터 증가 = 처리량 감소 + 지연 증가                                 │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【최적화 전략】                                                        │
│  ──────────────────                                                  │
│  • 웨이터 수 모니터링 → 락 세분성 조정                                  │
│  • 대기 시간 길면 → 락 타임아웃 설정                                    │
│  • 웨이터 많으면 → 락 스트라이핑 고려                                    │
│  • 불공정 락 사용 → 처리량 향상 (기아 모니터링)                            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```
