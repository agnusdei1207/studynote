# 조건 변수 (Condition Variable)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 조건 변수는 모니터 내부에서 스레드가 특정 조건이 만족될 때까지 실행을 멈추고 대기하거나, 조건이 만족되었음을 다른 스레드에게 알리는 데 사용하는 동기화 메커니즘이다.
> 2. **가치**: 락(Lock)만으로는 해결할 수 없는 '실행 순서'의 문제를 해결하며, 조건이 맞지 않을 때 락을 쥐고 기다리는 것이 아니라 락을 반납하고 잠들게 하여 시스템 교착을 방지한다.
> 3. **융합**: Pthreads의 `pthread_cond_t`, Java의 `wait/notify`, Python의 `threading.Condition` 등 거의 모든 현대적 병행성 라이브러리의 핵심 요소다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 상호 배제가 "한 번에 하나만"을 보장한다면, 조건 변수는 "A가 끝난 후에 B가"를 보장한다. 스레드가 공유 자원을 점유했더라도 원하는 조건(예: 버퍼에 데이터가 있음)이 아니면, 락을 해제하고 '조건 큐'에 들어가 대기한다. 나중에 조건이 충족되면 다른 스레드가 신호를 보내 깨워준다.

- **💡 비유**: 조건 변수는 "식당의 진동벨"과 같다. 손님(스레드)이 주문(락 획득)을 했지만 음식이 아직 안 나왔다면(조건 미충족), 카운터 앞에서 서성이지 않고 진동벨을 받고 구석 자리에 가서 쉰다(Wait). 음식이 나오면 직원이 벨을 울려(Signal) 손님을 호출한다.

- **📢 섹션 요약 비유**: 마치 택배 기사님이 벨을 누르기 전까지는 집 안에서 편하게 자기 일을 하다가, 벨 소리를 듣고서야 문을 열어 물건을 받는 효율적인 대기 방식과 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 조건 변수의 동작 시퀀스

조건 변수는 항상 뮤텍스(락)와 함께 쌍으로 작동해야 한다.

```text
  [ Thread A (Consumer) ]             [ Mutex & CV ]            [ Thread B (Producer) ]
            |                              |                             |
  1. Lock(m) ──────────────────▶ [ Mutex: Locked ]                       |
  2. While(!condition)                     |                             |
     Wait(cv, m) ───(Unlock & Sleep)──▶ [ CV Queue ]                     |
            |                              |                             |
            |                              |                    3. Lock(m)
            |                              |                    4. Make condition true
            |                              |                    5. Signal(cv)
            |                   [ Wake up Thread A ] ◀───────────────────┘
            |                              |                    6. Unlock(m)
  7. (Re-acquire Lock) ◀──────── [ Mutex: Locked ]                       |
  8. Do work & Unlock(m)                   |                             |
```

**[다이어그램 해설]** 스레드 A가 락을 잡고 조건을 확인했는데 거짓이면, `wait()`를 호출한다. 이때 중요한 점은 **들고 있던 락을 자동으로 놓고 잠든다**는 것이다. 그래야 스레드 B가 진입하여 조건을 바꿀 수 있기 때문이다. 스레드 B가 조건을 바꾸고 `signal()`을 보내면 스레드 A가 깨어나는데, 이때 **다시 락을 경쟁하여 획득**한 후에야 다음 코드를 실행할 수 있다.

### 주요 연산
- **wait()**: 호출 스레드를 대기 상태로 만들고 락을 반납한다.
- **signal() (또는 notify)**: 대기 중인 스레드 중 하나를 깨운다.
- **broadcast() (또는 notifyAll)**: 대기 중인 모든 스레드를 깨운다.

- **📢 섹션 요약 비유**: 자리가 날 때까지 가게 앞에서 무작정 기다리는 것이 아니라, 번호표를 받고 락을 반납한 뒤 용무를 보다가 호출을 받으면 다시 줄을 서는 합리적인 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 세마포어 vs 조건 변수

| 비교 항목 | 세마포어 (Semaphore) | 조건 변수 (Condition Variable) |
|:---|:---|:---|
| **상태 저장** | 정수 값으로 상태를 기억함 | 상태를 기억하지 않음 (Memoryless) |
| **Wait 동작** | 값이 0이면 즉시 대기 | 항상 큐에 들어가 대기 |
| **Signal 동작** | 대기자가 없어도 값이 증가함 | 대기자가 없으면 신호가 사라짐 |
| **뮤텍스 연동** | 단독 사용 가능 | 반드시 뮤텍스와 함께 사용 |

조건 변수는 '기억력'이 없기 때문에, 신호를 보냈을 때 기다리는 사람이 없다면 그 신호는 영영 사라진다. 따라서 조건을 확인하는 `while` 루프와 함께 사용하는 것이 필수적이다 (Spurious Wakeup 방지).

- **📢 섹션 요약 비유**: 세마포어가 기록이 남는 '포스트잇'이라면, 조건 변수는 그 순간에만 들리는 '확성기 소리'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 생산자-소비자 문제 해결
- **상황**: 공유 버퍼가 비어 있을 때 소비자가 접근하면 에러가 발생함.
- **해결책**: 소비자는 락을 획득한 후 `while(buffer.isEmpty()) cv.wait(lock);`를 실행한다. 생산자는 데이터를 넣은 후 `cv.signal();`을 호출한다. 이를 통해 소비자가 비어 있는 버퍼를 읽으려는 시도를 원천 차단한다.

### 도입 체크리스트
- **While vs If**: 조건을 `if`가 아닌 `while`로 검사하는가? (깨어났을 때 조건이 다시 변했을 가능성 대비)
- **Lock Acquisition**: `wait()`에서 깨어난 직후 락이 다시 획득되었음을 인지하고 있는가?

- **📢 섹션 요약 비유**: 잠에서 깼다고 바로 나가는 것이 아니라, 비가 그쳤는지 다시 한번 창밖을 확인(while)하고 나가는 신중함이 필요합니다.

## Ⅴ. 기대효과 및 결론

- **기대효과**: 조건 변수는 불필요한 CPU 소모(Busy Wait)를 없애고, 복잡한 스레드 간의 실행 순서를 우아하고 안전하게 조율한다.
- **미래 전망**: 함수형 프로그래밍의 영향으로 불변 객체와 메시지 패싱이 강조되고 있지만, 공유 메모리 모델을 사용하는 고성능 서버 프레임워크의 심장부에는 여전히 조건 변수가 핵심 메커니즘으로 자리 잡고 있다.

- **📢 섹션 요약 비유**: 오케스트라의 지휘자가 각 악기 팀의 연주 시점을 조율하듯, 조건 변수는 수많은 스레드가 조화롭게 협력하도록 만드는 지휘자와 같습니다.
