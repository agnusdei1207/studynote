# Lock Hierarchy (락 계층 구조 / 락 순서화)

> 1. **본질**: Lock Hierarchy는 멀티스레드 프로그래밍에서 여러 개의 락을 획득할 때 발생할 수 있는 순환 대기 (Circular Wait) 조건을 방지하기 위해, 모든 락에 고유한 우선순위나 순서를 부여하고 반드시 정해진 순서대로만 락을 획득하도록 강제하는 데드락 예방 전략이다.
> 2. **가치**: 데드락 (Deadlock)은 시스템을 중단시키는 치명적인 결함이며, 락 순서화는 복잡한 상호작용이 발생하는 대규모 커널이나 서버 소프트웨어에서 동기화의 안전성을 보장하는 가장 확실하고 실용적인 설계 가이드라인을 제공한다.
> 3. **융합**: 운영체제 커널의 락 검증 도구인 `lockdep` (Linux), 데이터베이스의 계층적 잠금 프로토콜, 그리고 세큐어 코딩 (Secure Coding)의 자원 관리 원칙과 깊이 통합된다.

+++

### Ⅰ. 개요 (Context & Background)

- **개념**: Lock Hierarchy는 시스템 내의 모든 락 자원 집합 $L = \{l_1, l_2, ..., l_n\}$에 대해 전체 순서 (Total Ordering) 관계를 정의하는 기법이다. 스레드가 락 $l_i$를 보유한 상태에서 $l_j$를 추가로 획득하려면, 반드시 $i < j$ (또는 정의된 계층 순서)를 만족해야 한다.

- **💡 비유**: 여러 개의 열쇠가 필요한 보관함을 열 때, 반드시 "1번 열쇠 → 2번 열쇠 → 3번 열쇠" 순서로만 사용해야 한다는 규칙과 같습니다. 만약 누구는 1번부터, 누구는 3번부터 열려고 한다면 서로 상대방의 열쇠가 나오길 기다리며 영원히 멈춰버리는 상황(데드락)이 발생할 수 있기 때문입니다.

- **등장 배경**:
  1. **순환 대기의 위험성**: 두 개 이상의 스레드가 서로 상대방이 가진 자원을 점유하려고 반대 방향으로 자원을 요청할 때 발생하는 순환 대기는 데드락의 4대 필요조건 중 하나다.
  2. **비결정적 실행의 공포**: 락 획득 순서가 일정하지 않으면, 수만 번의 실행 중 단 한 번의 타이밍 어긋남으로 시스템이 멈추는 비결정적 버그가 발생하며 이는 디버깅이 매우 어렵다.

- **📢 섹션 요약 비유**: 마치 일방통행 도로를 만들어 마주 오는 차들이 서로 비켜주지 못해 꽉 막히는 상황(데드락)을 원천 차단하는 교통 규칙과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Lock Hierarchy 설계의 핵심 요소

| 요소명 | 역할 | 설명 | 비유 |
|:---|:---|:---|:---|
| **락 레벨 (Lock Level)** | 순서 정의 | 각 락에 부여된 고유 번호 또는 계층적 위치 | 건물의 층수 |
| **전체 순서 (Total Order)** | 규칙 강제 | 모든 락 간의 획득 우선순위를 명확히 정의함 | 계단 오르는 순서 |
| **획득 규칙 (Acquisition Rule)** | 동작 가이드 | 현재 보유한 락보다 높은 레벨의 락만 요청 가능함 | 낮은 층에서 높은 층으로 이동 |
| **역순 해제 (Release Order)** | 안정성 확보 | (권장) 획득한 순서의 반대로 해제하여 스택 구조 유지 | 내려올 때의 순서 |
| **검증 도구 (Validator)** | 규칙 감시 | 런타임에 순서 위반을 감시하고 경고 발생 (ex: lockdep) | 교통경찰 |

### 순환 대기 제거 및 데드락 방지 메커니즘

데드락은 스레드 A가 락 1을 쥐고 락 2를 기다릴 때, 스레드 B가 락 2를 쥐고 락 1을 기다리는 교차 구조에서 발생한다. 락 순서화는 스레드 B가 락 2를 쥐기 전에 반드시 락 1을 먼저 획득하도록 강제하여 이 사이클을 파괴한다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │              [Before] 무질서한 락 획득 (Deadlock Risk)        │
  ├─────────────────────────────────────────────────────────────┤
  │   Thread A: lock(1) ──▶ lock(2) ──▶ [OK]                    │
  │   Thread B: lock(2) ──▶ lock(1) ──▶ [DEADLOCK!]             │
  │                                                             │
  │   * 상태: A는 1 보유/2 대기, B는 2 보유/1 대기 → 순환 발생    │
  └─────────────────────────────────────────────────────────────┘
                ▼ 락 순서화 (Lock Ordering) 적용 ▼
  ┌─────────────────────────────────────────────────────────────┐
  │              [After] 계층적 락 획득 (Deadlock Free)          │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │   규칙: 반드시 (1)번 락을 얻은 후 (2)번 락을 얻어야 함        │
  │                                                             │
  │   Thread A: lock(1) ──▶ lock(2) ──▶ [OK]                    │
  │   Thread B: lock(1) ──▶ lock(2) ──▶ [Waiting at (1)]        │
  │                                                             │
  │   * 상태: B는 1번을 얻으려다 대기하므로 2번을 쥘 수 없음      │
  │     => 순환 구조가 생성되지 않음                             │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 락 순서화를 적용하면 모든 스레드가 동일한 방향으로 자원을 요청하게 된다. Thread B가 2번 락을 필요로 하더라도, 규칙에 따라 1번 락을 먼저 시도해야 한다. 이때 1번은 Thread A가 이미 점유 중이므로 Thread B는 1번에서 대기하게 되고, 2번 락은 여전히 자유로운 상태로 남는다. 결국 Thread A는 2번을 획득하여 작업을 마치고 1번을 해제할 수 있으며, 이어서 Thread B가 진행된다.

### 리눅스 커널의 `lockdep` 검증 원리

리눅스 커널은 런타임에 락 획득 순서를 그래프로 그려서 순환이 발생하는지 감시한다.

1. **Lock Class 등록**: 각 락에 카테고리를 부여한다.
2. **Dependency Graph 구축**: A 락을 잡은 상태에서 B 락을 잡으면 $A \rightarrow B$ 간선을 생성한다.
3. **Cycle Detection**: 새로운 간선 추가 시 그래프에 사이클이 생기는지 확인한다.
4. **Warning**: 사이클(ex: $A \rightarrow B \rightarrow C \rightarrow A$) 감지 시 실제로 데드락이 발생하지 않았더라도 "잠재적 위반"으로 경고와 콜스택을 출력한다.

- **📢 섹션 요약 비유**: 도서관에서 책을 빌릴 때 반드시 "가나다" 순서로만 빌려야 한다고 정해두면, 서로 다른 사람이 책을 교차해서 들고 있어 생기는 갈등을 막을 수 있는 것과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 데드락 처리 전략 비교

| 전략 구분 | 기법명 | 특징 | 장점 | 단점 |
|:---|:---|:---|:---|:---|
| **예방 (Prevention)** | **Lock Hierarchy** | 순환 대기 조건의 구조적 제거 | 확실한 방지, 런타임 오버헤드 적음 | 설계 단계의 엄격함 필요 |
| **회피 (Avoidance)** | **은행원 알고리즘** | 자원 할당 전 안전 상태 확인 | 유연한 자원 할당 | 최대 자원량 사전 숙지 필요, 오버헤드 큼 |
| **탐지 (Detection)** | **Wait-for Graph** | 주기적으로 데드락 발생 여부 체크 | 제약 없는 설계 가능 | 탐지 및 복구(Rollback) 비용 발생 |
| **무시 (Ignorance)** | **타조 알고리즘** | 데드락이 드물다고 가정하고 방치 | 구현 극히 단순, 성능 최우선 | 시스템 중단 위험 상존 |

### 과목 융합 관점

- **데이터베이스 (DB)**: 다중 입도 잠금 (Multiple Granularity Locking)에서 DB → Table → Page → Row 순서로 락을 거는 것도 일종의 계층 구조다. 상위 레벨에 의도 잠금 (Intent Lock)을 걸어 하위 레벨과의 충돌을 예방한다.
- **네트워크 (Network)**: 분산 시스템에서의 메시지 순서 보장 (Ordering) 기법은 물리적 락은 없으나, 논리적인 자원 접근 순서를 정의하여 교착 상태를 방지한다는 점에서 락 계층 구조와 철학을 공유한다.

- **📢 섹션 요약 비유**: 범죄 예방을 위해 길목에 CCTV를 다는 것(계층 구조)이, 범죄가 일어난 후 범인을 잡으러 다니는 것(탐지 및 복구)보다 훨씬 효율적인 것과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 및 설계 가이드

1. **시나리오: 파일 시스템과 네트워크 드라이버의 연동**:
   - **문제**: 네트워크 파일 시스템(NFS)에서 파일 락을 잡은 채 네트워크 전송 락을 기다리고, 네트워크 드라이버는 전송 완료 처리를 위해 파일 메타데이터 락을 기다리는 데드락 발생.
   - **판단**: "파일 시스템 락 → 네트워크 락"의 계층을 엄격히 정의. 반대 순서가 필요한 경우, 기존 락을 해제(Release)하고 새 순서로 다시 획득하는 'Lock Reordering' 수행.

2. **시나리오: 객체 지향 프로그래밍에서의 계층 구조**:
   - **문제**: 부모 객체와 자식 객체 간의 상호 참조로 인한 락 경합.
   - **판단**: 항상 "부모 → 자식" 또는 "메모리 주소 순서"로 락을 획득하도록 코딩 컨벤션 강제.

### 락 계층 구조 설계 시 주의사항
- **락 순서 위반 (Order Violation)**: 제3의 라이브러리나 외부 모듈을 호출할 때 내부에서 어떤 락을 잡는지 모르면 순서가 꼬일 수 있다. (Opaque Call 문제)
- **성능 저하**: 순서를 지키기 위해 이미 잡을 수 있는 락을 해제했다가 다시 잡는 행위는 컨텍스트 스위칭과 경합을 유발할 수 있다.

- **📢 섹션 요약 비유**: 군대에서 지휘 계통(계층)이 명확해야 혼선이 없는 것처럼, 소프트웨어에서도 락의 계급을 정해주어야 질서가 유지됩니다.

| **검증 가능성** | 테스트로만 버그 발견 가능 | 정적/동적 분석으로 설계 검증 가능 | 소프트웨어 신뢰성 확보 |

### 미래 전망 및 결론
시스템이 복잡해질수록 수동으로 락 순서를 관리하는 것은 한계에 부딪힌다. 따라서 미래에는 **컴파일 타임 락 검증 (Rust 언어의 오너십 모델 등)**이나 **런타임 자동 락 재배치 시스템**이 락 계층 구조의 역할을 대신하게 될 것이다. 하지만 하위 레벨의 커널 설계나 임베디드 환경에서는 여전히 명확한 락 순서화가 설계의 바이블로 남을 것이다. 결론적으로 락 계층 구조는 "예방이 치료보다 낫다"는 격언을 컴퓨터 과학에서 가장 잘 실천하는 기법이다.

- **📢 섹션 요약 비유**: 아주 정교한 시계 태엽들이 서로 엉키지 않고 돌아가도록 설계 도면(계층 구조)을 미리 완벽하게 그리는 것과 같습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 락 계층 구조는 장난감 여러 개를 가지고 놀 때, 무조건 **"1번 로봇 → 2번 자동차 → 3번 비행기"** 순서로만 집어야 한다는 규칙이에요.
2. 만약 민수는 로봇을 잡고 자동차를 기다리는데, 철수는 자동차를 잡고 로봇을 기다리면 둘 다 아무것도 못 하게 되거든요.
3. 하지만 모두가 똑같이 1번부터 집기로 약속하면, 먼저 1번을 집은 사람이 나머지도 다 가지고 놀 수 있어서 싸울 일이 없어진답니다!
