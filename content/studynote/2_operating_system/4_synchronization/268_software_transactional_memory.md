# 소프트웨어 트랜잭셔널 메모리 (Software Transactional Memory, STM)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: STM은 메모리 접근을 데이터베이스 트랜잭션처럼 처리하여, 공유 자원 접근 시 명시적인 락(Lock) 없이도 원자성(Atomicity)과 일관성(Consistency)을 보장하는 소프트웨어 기반 동시성 제어 기술이다.
> 2. **가치**: 데드락(Deadlock) 위험을 원천 제거하고 코드의 합성성(Composability)을 높여, 복잡한 동시성 제어 로직을 선언적이고 안전하게 작성할 수 있게 한다.
> 3. **융합**: 함수형 프로그래밍(Haskell, Clojure)에서 시작되어 다중 코어 시대의 병렬 처리 병목을 해결하는 대안으로 각광받으며, 낙관적 병행성 제어(OCC)를 메모리 수준으로 끌어올린 사례다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: STM은 특정 코드 블록을 `atomic`으로 지정하면, 시스템이 그 안의 메모리 연산들을 트랜잭션으로 관리한다. 성공 시 일괄 반영하고, 충돌 시 자동으로 롤백 후 재시도한다.

- **💡 비유**: STM은 **"구글 독스(Google Docs)의 동시 편집"**과 같다. 내가 문서를 수정하는 동안 다른 사람이 못 들어오게 '문서 락'을 걸지 않는다. 각자 자유롭게 수정하고, 서버가 저장 시점에 충돌 여부를 판단하여 자동으로 병합하거나 재수정을 요청하는 것과 유사하다.

- **등장 배경**:
  1. **Lock-based Programming의 한계**: 락은 관리가 매우 어렵고, 우선순위 역전이나 데드락 같은 치명적 문제를 유발한다.
  2. **병렬성 확장성**: 코어 수가 늘어날수록 전역 락은 병목이 되며, 세밀한 락(Fine-grained lock)은 구현 난이도가 기하급수적으로 상승한다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### STM의 동작 메커니즘 (Read/Write Log 기반)

STM은 각 스레드마다 로컬 트랜잭션 로그를 유지하며 다음의 과정을 수행한다.

| 단계 | 동작 | 상세 내용 |
|:---|:---|:---|
| **1. 시작** | 트랜잭션 개시 | `atomic { ... }` 블록 진입 시 로그 초기화 |
| **2. 실행** | 로컬 연산 | 공유 메모리를 직접 수정하지 않고 로컬 로그(Write Log)에 기록 |
| **3. 검증** | 충돌 검사 | 내가 읽은 값이 그 사이에 다른 트랜잭션에 의해 바뀌었는지 확인 |
| **4. 커밋/롤백** | 결과 반영 | 충돌 없으면 메모리에 일괄 쓰기(Commit), 충돌 시 무효화 후 재시작(Retry) |

+++

### 핵심 구성 요소 다이어그램

```text
 [Thread A]                      [Shared Memory]               [Thread B]
      │                                 │                          │
      │ 1. Read Variable X  ──────────▶ [X: 10] ◀────────── 1. Read Variable X
      │ 2. Compute X + 5                │                   2. Compute X + 20
      │ 3. Store in Local Log           │                   3. Store in Local Log
      │                                 │                          │
      │ 4. Commit Request               │                          │
      │ 5. Validate (X is 10?) ───▶ [Success]                      │
      │ 6. Global Update ─────────▶ [X: 15]                        │
      │                                 │                          │
      │                                 │                   4. Commit Request
      │                                 │                   5. Validate (X is 10?)
      │                                 │                   6. [Fail!] X is 15 now.
      │                                 │                   7. Rollback & Retry
```

**[다이어그램 해설]** 스레드 A가 먼저 커밋에 성공하여 변수 X를 15로 바꾸면, 나중에 커밋을 시도하는 스레드 B는 자신이 처음에 읽었던 X(10)와 현재 상태(15)가 다르므로 검증에 실패하고 다시 처음부터 계산을 시작하게 된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 락 기반 동시성(Locking) vs STM 비교

| 비교 항목 | 락 기반 (Lock-based) | STM (Software Transactional Memory) |
|:---|:---|:---|
| **제어 방식** | 명시적 (Lock/Unlock) | 선언적 (Atomic Block) |
| **프로그래밍 난이도** | 높음 (오류 발생 쉬움) | 낮음 (추상화 수준 높음) |
| **데드락** | 발생 가능 | **발발 불가능** |
| **합성성 (Composability)** | 불가능 (락 뺏기 어려움) | **가능 (트랜잭션 중첩 가능)** |
| **성능 오버헤드** | 컨텍스트 스위칭 비용 | 로깅 및 검증 오버헤드 (소프트웨어 부하) |

- **📢 섹션 요약 비유**: 락 기반이 "열쇠 꾸러미를 직접 들고 다니며 문을 여닫는 것"이라면, STM은 "지문 인식기를 달아두고 시스템이 출입을 자동 제어하게 하는 것"입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
- **Haskell/Clojure**: 언어 차원에서 강력한 STM을 지원하여 불변 데이터 구조와 결합된 극강의 병렬성을 제공한다.
- **게임 엔진**: 복잡하게 얽힌 게임 오브젝트들의 상태 업데이트 시, 개발자가 수천 개의 락을 관리하는 대신 STM으로 논리적 원자성을 확보할 수 있다.

### 기술적 한계 및 주의점
1. **Side-Effect 관리**: 트랜잭션 내부에서 파일 출력이나 네트워크 송신 같은 되돌릴 수 없는 작업(I/O)을 수행하면 안 된다. (롤백 시 이미 보낸 패킷을 취소할 수 없기 때문)
2. **성능 이슈**: 소프트웨어적으로 모든 읽기/쓰기를 트랜잭션 로그에 기록하므로 순수 연산 속도는 락보다 느릴 수 있다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과
- **코드 안정성**: 레이스 컨디션과 데드락을 언어 수준에서 방어하여 버그 발생률을 획기적으로 낮춘다.
- **유지보수성**: 동시성 로직이 비즈니스 로직과 분리되어 가독성이 높아진다.

- **📢 섹션 요약 비유**: STM은 병렬 프로그래밍의 고통을 덜어주는 "동시성 관리 자동 비서"와 같습니다.

+++

### 📌 관련 개념 맵
- **낙관적 병행성 제어 (OCC)**: STM의 기반 이론.
- **하드웨어 트랜잭셔널 메모리 (HTM)**: STM을 CPU 하드웨어로 구현하여 성능을 극대화한 기술.
- **CAS (Compare-And-Swap)**: STM 구현의 최소 단위 원자적 연산.
