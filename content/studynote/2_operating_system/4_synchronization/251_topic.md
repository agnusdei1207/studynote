# 251. 트랜잭셔널 메모리 (Transactional Memory)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 원자적 트랜잭션으로 메모리 접근 동기화
> 2. **가치**: 락 없는 프로그래밍, 쉬운 병렬화
> 3. **융합**: 락 엘리션, CAS, 낙관적 락과 연관

---

## Ⅰ. 개요

### 개념 정의

트랜잭셔널 메모리(Transactional Memory, TM)는 **메모리 접근을 원자적 트랜잭션으로 수행하여 락 없이 동기화하는 프로그래밍 모델**이다. 성공 시 커밋, 충돌 시 롤백한다.

### 💡 비유: 은행 계좌 이체
트랜잭셔널 메모리는 **은행 계좌 이체**와 같다. A에서 100원 빼고 B에 100원 넣는다. 실패하면 원래대로 돌아간다. 모두 성공하거나 모두 실패한다.

### 트랜잭셔널 메모리 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                트랜잭셔널 메모리 구조                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【트랜잭션 의미론】                                                    │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  atomic {  // 트랜잭션 시작                                      │ │
│  │      x = x + 1;                                                   │ │
│  │      y = y - 1;                                                   │ │
│  │  }  // 커밋 또는 롤백                                              │ │
│  │                                                             │ │
│  │  특성:                                                             │ │
│  │  1. 원자성 (Atomicity): 전부 성공 또는 전부 실패                    │ │
│  │  2. 격리성 (Isolation): 다른 트랜잭션과 섞이지 않음                   │ │
│  │  3. 일관성 (Consistency): 항상 일관된 상태                          │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【동작 방식】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  1. 시작: Read/Write Set 초기화                                   │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │  Read Set: {x, y}                                        │ │ │
│  │  │  Write Set: {x, y}                                       │ │ │
│  │  │  Local Buffer: {x=1, y=9}                                │ │ │
│  │  └─────────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  │  2. 실행: 로컬 버퍼에서 읽기/쓰기                                  │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │  x = x + 1  // 로컬에서 x=2                                │ │ │
│  │  │  y = y - 1  // 로컬에서 y=8                                │ │ │
│  │  └─────────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  │  3. 검증: Read Set의 값이 변경되었는지 확인                        │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │  if (x == old_x && y == old_y) {                         │ │ │
│  │  │      // 커밋: Write Set을 실제 메모리에 반영                   │ │ │
│  │  │      x = 2; y = 8;                                        │ │ │
│  │  │  } else {                                                 │ │ │
│  │  │      // 롤백: 트랜잭션 재시도                                │ │ │
│  │  │  }                                                        │ │ │
│  │  └─────────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【HTM vs STM】                                                       │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  HTM (Hardware TM):                                               │ │
│  │  • CPU 캐시로 구현                                                 │ │
│  │  • 빠름, 제약 있음 (캐시 크기)                                      │ │
│  │  • Intel TSX, IBM POWER TM                                        │ │
│  │                                                             │ │
│  │  STM (Software TM):                                               │ │
│  │  • 소프트웨어로 구현                                                │ │
│  │  • 느림, 유연함                                                     │ │
│  │  • HyTM (하이브리드)                                                │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                트랜잭셔널 메모리 상세                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【충돌 감지】                                                         │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  낙관적 실행:                                                       │ │
│  │  • 일단 실행해 보고 충돌 감지                                       │ │
│  │  • Read Set에 있는 주소가 변경되면 충돌                              │ │
│  │  • Write Set와 다른 트랜잭션의 Read/Write Set 겹치면 충돌           │ │
│  │                                                             │ │
│  │  비관적 vs 낙관적:                                                   │ │
│  │  • 비관적 (락): 미리 락 획득, 충돌 방지                               │ │
│  │  • 낙관적 (TM): 일단 실행, 충돌 시 롤백                               │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【충돌 해결】                                                         │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 즉시 롤백: 충돌 감지 즉시 중단                                   │ │
│  │  2. 지연 롤백: 커밋 시점에 검증 후 중단                               │ │
│  │  3. 백오프: 재시도 전 대기 시간 증가                                  │ │
│  │  4. 폴백: 일정 횟수 실패 시 락 사용                                   │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【HTM 제약】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • 캐시 용량 제한: L1 캐시 크기                                      │ │
│  │    (Intel: ~수싱 KB)                                               │ │
│  │  • 중첩 제한: 트랜잭션 중첩 깊이                                     │ │
│  │  • 비결정적: 인터럽트, 컨텍스트 스위치로 중단 가능                      │ │
│  │  • I/O 불가: 입출력 작업 중단                                        │ │
│  │  • 버전 의존: CPU 마이크로아키텍처 의존                               │ │
│  │                                                             │ │
│  │  Intel TSX 버그:                                                   │ │
│  │  • Skylake: TSX 비활성화 (버그)                                     │ │
│  │  • Tiger Lake: 재활성화                                             │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【C++ TM TS (실험적)】                                               │
│  ──────────────────                                                  │
│  #include <experimental/atomic>                                     │
│                                                                     │
│  // atomic 블록                                                     │
│  void transfer(Account& from, Account& to, int amount) {            │
│      atomic {                                                        │
│          from.balance -= amount;                                     │
│          to.balance += amount;                                       │
│      }  // 커밋 또는 롤백                                              │
│  }                                                                   │
│                                                                     │
│  【Java (STM 라이브러리)】                                             │
│  ──────────────────                                                  │
│  import org.deuce.*;  // Deuce STM                                  │
│                                                                     │
│  @Atomic                                                             │
│  public void transfer(Account from, Account to, int amount) {        │
│      from.balance -= amount;                                         │
│      to.balance += amount;                                           │
│  }                                                                   │
│                                                                     │
│  【Haskell STM】                                                      │
│  ──────────────────                                                  │
│  import Control.Concurrent.STM                                       │
│                                                                     │
│  transfer :: TVar Int -> TVar Int -> Int -> STM ()                  │
│  transfer from to amount = do                                        │
│      fromBal <- readTVar from                                        │
│      toBal <- readTVar to                                            │
│      writeTVar from (fromBal - amount)                               │
│      writeTVar to (toBal + amount)                                   │
│                                                                     │
│  -- 실행                                                             │
│  atomically $ transfer fromVar toVar 100                             │
│                                                                     │
│  【Clojure STM】                                                      │
│  ──────────────────                                                  │
│  (dosync                                                             │
│    (alter from-acc - amount)                                         │
│    (alter to-acc + amount))                                          │
│                                                                     │
│  【Intel TSX (어셈블리)】                                              │
│  ──────────────────                                                  │
│  xbegin L1                         ; 트랜잭션 시작                     │
│  cmp eax, 0xFFFFFFFF                                                 │
│  je fallback                        ; 실패 시 폴백                     │
│  ; ... 임계 영역 ...                                                   │
│  xend                               ; 커밋                              │
│  jmp done                                                            │
│  fallback:                                                           │
│  ; 전통적 락                                                           │
│  lock cmpxchg [mutex], 0, 1                                          │
│  ; ... 임계 영역 ...                                                   │
│  mov [mutex], 0                                                      │
│  done:                                                               │
│                                                                     │
│  【언제 사용할까?】                                                    │
│  ──────────────────                                                  │
│  적합:                                                                 │
│  • 복잡한 원자적 연산                                                  │
│  • 여러 변수 동시 수정                                                 │
│  • 낮은 충돌률                                                         │
│  • 짧은 트랜잭션                                                       │
│                                                                     │
│  부적합:                                                               │
│  • 높은 충돌률                                                         │
│  • 긴 트랜잭션                                                         │
│  • I/O 포함                                                           │
│  • 큰 메모리 접근                                                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 원자적 트랜잭션으로 메모리 접근 동기화
• 특성: 원자성, 격리성, 일관성
• HTM: 하드웨어 구현, 빠름, 제약 있음
• STM: 소프트웨어 구현, 느림, 유연함
• 장점: 락 없는 프로그래밍, 교착 없음
• 단점: 오버헤드, 제약, 폴백 필요
• 활용: 복잡한 원자적 연산
```
