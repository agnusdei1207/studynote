# 249. 메모리 순서 (Memory Ordering)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPU와 컴파일러의 메모리 접근 순서 최적화
> 2. **가치**: 올바른 동기화를 위한 순서 보장
> 3. **융합**: 원자적 연산, happens-before, 캐시 일관성과 연관

---

## Ⅰ. 개요

### 개념 정의

메모리 순서(Memory Ordering)는 **CPU와 컴파일러가 메모리 접근(읽기/쓰기) 순서를 재배치하는 최적화 기법**이다. 올바른 동기화를 위해서는 적절한 순서 보장이 필요하다.

### 💡 비유: 요리 순서
메모리 순서는 **요리 순서**와 같다. 레시피는 1-2-3순이지만, 숙련된 요리사는 2-1-3으로 해도 맛은 수 있다. 하지만 특정 요리는 순서가 중요하다. 순서를 지켜야 한다.

### 메모리 순서 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                메모리 순서 구조                                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【순서 문제 시나리오】                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  소스 코드 순서:                                                  │ │
│  │  x = 1;         // 쓰기 1                                             │ │
│  │  ready = true;  // 쓰기 2                                             │ │
│  │                                                             │ │
│  │  컴파일러/CPU 재배치 가능:                                          │ │
│  │  ready = true;  // 쓰기 2가 먼저 실행                                │ │
│  │  x = 1;         // 쓰기 1이 나중에 실행                               │ │
│  │                                                             │ │
│  │  다른 스레드:                                                      │ │
│  │  if (ready) {   // true를 봄                                        │ │
│  │      use(x);    // x가 아직 1이 아닐 수 있음!                            │ │
│  │  }                                                               │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【순서 보장 계층】                                                      │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  강도 (강 → 약):                                                     │ │
│  │                                                             │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │  memory_order_seq_cst (가장 강함)                           │ │ │
│  │  │  • 전역 순서 보장                                          │ │ │
│  │  │  • 모든 스레드가 같은 순서 관찰                              │ │ │
│  │  │  • 가장 느림                                                 │ │ │
│  │  └─────────────────────────────────────────────────────────┘ │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │  memory_order_acquire/release                           │ │ │
│  │  │  • 획득/해제 의미론                                          │ │ │
│  │  │  • 효율적 동기화                                              │ │ │
│  │  │  • 일반적으로 사용                                             │ │ │
│  │  └─────────────────────────────────────────────────────────┘ │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │  memory_order_relaxed (가장 약함)                            │ │ │
│  │  │  • 순서 보장 없음                                             │ │ │
│  │  │  • 단순 카운터                                                │ │ │
│  │  │  • 가장 빠름                                                   │ │ │
│  │  └─────────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【아키텍처별 메모리 모델】                                             │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  Strongly Ordered (x86, x86-64):                                 │ │
│  │  • 대부분 순서 보장                                               │ │
│  │  • Store-Store 재배치만 허용                                       │ │
│  │  • MFENCE, SFENCE, LFENCE 명령어                                   │ │
│  │                                                             │ │
│  │  Weakly Ordered (ARM, POWER, MIPS):                               │ │
│  │  • 많은 재배치 허용                                                │ │
│  │  • DMB, ISB, SYNC 명령어                                          │ │
│  │  • 명시적 펜스 필요                                                 │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                메모리 순서 상세                                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【C++ Memory Order】                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. memory_order_relaxed                                          │ │
│  │     • 순서 보장 없음                                                │ │
│  │     • 원자성만 보장                                                 │ │
│  │     • 예: 단순 카운터                                               │ │
│  │     atomic_fetch_add(&counter, 1, relaxed);                       │ │
│  │                                                             │ │
│  │  2. memory_order_consume                                           │ │
│  │     • 데이터 의존성 보장                                             │ │
│  │     • 거의 사용 안 함 (컴파일러가 acquire로 승격)                    │ │
│  │                                                             │ │
│  │  3. memory_order_acquire                                           │ │
│  │     • 이후 읽기/쓰기가 이 지점 이후로 재배치 안 됨                     │ │
│  │     • "진입 장벽"                                                   │ │
│  │     • lock 획득 후 사용                                              │ │
│  │                                                             │ │
│  │  4. memory_order_release                                           │ │
│  │     • 이전 읽기/쓰기가 이 지점 이전으로 재배치 안 됨                     │ │
│  │     • "퇴출 장벽"                                                   │ │
│  │     • lock 해제 전 사용                                              │ │
│  │                                                             │ │
│  │  5. memory_order_acq_rel                                           │ │
│  │     • acquire + release                                             │ │
│  │     • 읽기-수정-쓰기 (RMW) 연산                                       │ │
│  │                                                             │ │
│  │  6. memory_order_seq_cst                                           │ │
│  │     • 전역 순서 보장                                                 │ │
│  │     • 가장 직관적                                                    │ │
│  │     • 성능 오버헤드 있음                                              │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【Acquire-Release 의미론】                                            │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  스레드 A:                           스레드 B:                        │ │
│  │  data = 42;                         while (!ready) {}            │ │
│  │  ready.store(true, release);        r = data.load(acquire);        │ │
│  │                                                             │ │
│  │  release:                           acquire:                       │ │
│  │  • data 쓰기가 완료                   • ready 읽기 이후             │ │
│  │  • ready 쓰기 전                     • data 읽기 전                │ │
│  │  • data → ready 순서                 • ready → data 순서           │ │
│  │                                                             │ │
│  │  결과: B가 ready=true를 보면 data=42도 반드시 봄                   │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【C++ 올바른 동기화】                                                 │
│  ──────────────────                                                  │
│  #include <atomic>                                                   │
│                                                                     │
│  std::atomic<bool> ready{false};                                     │
│  int data = 0;                                                       │
│                                                                     │
│  void producer() {                                                  │
│      data = 42;  // 일반 변수 쓰기                                   │
│      ready.store(true, std::memory_order_release);  // 펜스           │
│  }                                                                  │
│                                                                     │
│  void consumer() {                                                  │
│      while (!ready.load(std::memory_order_acquire)) {}  // 펜스       │
│      assert(data == 42);  // 반드시 42                               │
│  }                                                                  │
│                                                                     │
│  【Spinlock 구현】                                                    │
│  ──────────────────                                                  │
│  class Spinlock {                                                    │
│      std::atomic_flag flag = ATOMIC_FLAG_INIT;                      │
│  public:                                                            │
│      void lock() {                                                  │
│          while (flag.test_and_set(std::memory_order_acquire))       │
│              ;  // 스핀                                                │
│      }                                                              │
│      void unlock() {                                                │
│          flag.clear(std::memory_order_release);                     │
│      }                                                              │
│  };                                                                 │
│                                                                     │
│  【Double-Checked Locking】                                          │
│  ──────────────────                                                  │
│  std::atomic<Singleton*> instance{nullptr};                         │
│  std::mutex mtx;                                                     │
│                                                                     │
│  Singleton* get_instance() {                                        │
│      Singleton* p = instance.load(std::memory_order_acquire);       │
│      if (!p) {                                                       │
│          std::lock_guard lock(mtx);                                 │
│          p = instance.load(std::memory_order_relaxed);              │
│          if (!p) {                                                  │
│              p = new Singleton();                                   │
│              instance.store(p, std::memory_order_release);          │
│          }                                                          │
│      }                                                              │
│      return p;                                                      │
│  }                                                                  │
│                                                                     │
│  【성능 비교】                                                          │
│  ──────────────────                                                  │
│  x86-64 (Intel Core i7):                                            │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  Operation              Relaxed   Acquire   Seq_Cst              │ │
│  │  ─────────              ───────   ───────   ────────              │ │
│  │  atomic_load            1ns       1ns       1ns                   │ │
│  │  atomic_store           1ns       1ns       3ns                   │ │
│  │  atomic_fetch_add       5ns       5ns       10ns                  │ │
│  │                                                             │ │
│  │  ARM (Apple M1):                                               │ │
│  │  atomic_load            1ns       3ns       5ns                   │ │
│  │  atomic_store           1ns       3ns       10ns                  │ │
│  │  atomic_fetch_add       5ns       10ns      20ns                  │ │
│  │                                                             │ │
│  │  ARM에서 memory_order 차이가 더 큼                               │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: CPU/컴파일러 메모리 접근 순서 재배치
• 문제: 재배치로 인한 동기화 오류
• 계층: relaxed < acquire/release < seq_cst
• Acquire: 진입 장벽, 이후 연산 재배치 방지
• Release: 퇴출 장벽, 이전 연산 재배치 방지
• 아키텍처: x86 강함, ARM 약함
```
