# 264. 락 스래싱 (Lock Thrashing)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 과도한 락 경합으로 시스템 마비
> 2. **가치**: 성능 붕괴 임계점 식별
> 3. **융합**: 락 경합, 스핀락, 캐시 라인과 연관

---

## Ⅰ. 개요

### 개념 정의

락 스래싱(Lock Thrashing)은 **너무 많은 스레드가 락을 경쟁하여 실제 작업보다 락 관리에 더 많은 시간을 소비하는 상태**다. 시스템 처리량이 급격히 감소하거나 마비된다.

### 💡 비유: 회전문 붐비기
락 스래싱은 **회전문 붐비기**와 같다. 너무 많은 사람이 동시에 회전문을 통과하려 한다. 서로 부딪히고 막힌다. 실제로 나가는 사람보다 기다리는 사람이 더 많다.

### 락 스래싱 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                락 스래싱 구조                                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【정상 vs 스래싱】                                                     │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  처리량 그래프:                                                      │ │
│  │                                                             │ │
│  │  Throughput                                                         │ │
│  │     ↑    _______________                                           │ │
│  │     │   /                        ← 정상 (처리량 증가)              │ │
│  │     │  /                                                       │ │
│  │     │ /   _____________       ← 스래싱 시작 (처리량 감소)          │ │
│  │     │/   /                                                       │ │
│  │     └───/----/────────────                                       │ │
│  │         ↑    ↑                                                  │ │
│  │       최적점  스래싱                                              │ │
│  │         └────┴────┴────┴────→ 스레드 수                            │ │
│  │                                                             │ │
│  │  스레드 증가 = 처리량 증가? NO!                                      │ │
│  │  일정 수 이상에서 처리량 급감 = 스래싱                                  │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【스래싱 발생 요인】                                                   │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 스핀락 과다 사용:                                                 │ │
│  │     • 모든 코어가 스핀만 반복                                          │ │
│  │     • CPU 100% 사용, 실제 작업 없음                                   │ │
│  │                                                             │ │
│  │  2. 락 그래뉼러리티 부적절:                                            │ │
│  │     • 너무 굵은 락 = 높은 경합                                         │ │
│  │                                                             │ │
│  │  3. 캐시 라인 바운싱:                                                │ │
│  │     • 락 변수가 캐시 라인 간 이동                                      │ │
│  │     • MESI 프로토콜 트래픽 폭증                                        │ │
│  │                                                             │ │
│  │  4. 과도한 스레드 생성:                                               │ │
│  │     • CPU 코어보다 많은 스레드                                          │ │
│  │     • 컨텍스트 스위치 증가                                             │ │
│  │                                                             │ │
│  │  5. 캐시 라인 핫스팟:                                                 │ │
│  │     • 하나의 캐시 라인에 여러 락                                        │ │
│  │     • False sharing                                                  │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【증상】                                                              │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • CPU 사용률 100% (스핀락 시)                                        │ │
│  │  • 처리량 급감 또는 0                                                │ │
│  │  • 응답 시간 급증                                                     │ │
│  │  • 시스템 반응 없음                                                   │ │
│  │  • 락 대기 시간 폭증                                                  │ │
│  │  • 캐시 미스 증가                                                     │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                락 스래싱 상세                                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【스래싱 임계점】                                                      │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  임계 스레드 수 = (임계 영역 시간) / (전체 작업 시간)                   │ │
│  │                                                             │ │
│  │  예:                                                               │ │
│  │  • 임계 영역: 10%                                                   │ │
│  │  • 최적 스레드 수: ~10개                                             │ │
│  │  • 그 이상에서 스래싱 시작                                            │ │
│  │                                                             │ │
│  │  이상적으로:                                                         │ │
│  │  최적 스레드 수 ≈ 1 / (임계 영역 비율)                                 │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【캐시 라인 바운싱】                                                   │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  CPU 0    CPU 1    CPU 2    CPU 3                                  │ │
│  │    │        │        │        │                                   │ │
│  │    ▼        ▼        ▼        ▼                                   │ │
│  │  ┌──────────────────────────────────────┐                         │ │
│  │  │     Lock (캐시 라인 하나)              │                         │ │
│  │  └──────────────────────────────────────┘                         │ │
│  │    ↑        ↑        ↑        ↑                                   │ │
│  │    │        │        │        │                                   │ │
│  │   CAS      CAS      CAS      CAS                                  │ │
│  │                                                             │ │
│  │  모든 CPU가 같은 캐시 라인에 CAS →                                   │ │
│  │  MESI Invalid → Shared → Exclusive 트래픽 폭증                       │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【완화 전략】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 스레드 수 제한:                                                  │ │
│  │     • 스레드 풀 크기 = CPU 코어 수                                    │ │
│  │     • 과도한 스레드 생성 방지                                         │ │
│  │                                                             │ │
│  │  2. 락 세분성 향상:                                                  │ │
│  │     • 락 스트라이핑                                                  │ │
│  │     • 더 많은 락, 낮은 경합                                           │ │
│  │                                                             │ │
│  │  3. 백오프 전략:                                                     │ │
│  │     • 지수 백오프                                                    │ │
│  │     • 랜덤 지연                                                      │ │
│  │                                                             │ │
│  │  4. MCS 락 / CLH 락:                                               │ │
│  │     • 스핀락의 캐시 효율적 변형                                        │ │
│  │     • 각 스레드가 로컬 변수에서 스핀                                    │ │
│  │                                                             │ │
│  │  5. 락 프리 알고리즘:                                                │ │
│  │     • CAS 기반, 락 없음                                              │ │
│  │                                                             │ │
│  │  6. 적응적 락:                                                      │ │
│  │     • 스핀 → 블록 전환                                               │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【지수 백오프로 스래싱 완화】                                          │
│  ──────────────────                                                  │
│  void spinlock_with_backoff(atomic_flag* lock) {                    │
│      int backoff = 1;                                               │
│      while (true) {                                                  │
│          if (!atomic_flag_test_and_set(lock)) {                     │
│              return;  // 락 획득 성공                                  │
│          }                                                           │
│          // 백오프: 지수 증가                                          │
│          for (int i = 0; i < backoff; i++) {                         │
│              cpu_relax();  // PAUSE 명령어                             │
│          }                                                           │
│          backoff = min(backoff * 2, MAX_BACKOFF);                   │
│      }                                                              │
│  }                                                                  │
│                                                                     │
│  【MCS 락 (캐시 친화적 스핀락)】                                        │
│  ──────────────────                                                  │
│  struct mcs_node {                                                   │
│      struct mcs_node* next;                                          │
│      bool locked;                                                    │
│  };                                                                 │
│                                                                     │
│  void mcs_lock(mcs_node** lock, mcs_node* my_node) {                │
│      my_node->next = nullptr;                                        │
│      my_node->locked = true;                                         │
│      mcs_node* prev = atomic_exchange(lock, my_node);               │
│      if (prev != nullptr) {                                          │
│          prev->next = my_node;                                       │
│          while (my_node->locked) cpu_relax();  // 로컬에서 스핀       │
│      }                                                              │
│  }                                                                  │
│                                                                     │
│  // 캐시 라인 바운싱 없이 각 스레드가 로컬 변수에서 스핀                   │
│                                                                     │
│  【CLH 락】                                                           │
│  ──────────────────                                                  │
│  struct clh_node {                                                   │
│      atomic_bool locked;                                             │
│  };                                                                 │
│  // 각 스레드가 이전 노드의 locked 필드를 감시                           │
│  // 캐시 라인이 한 번만 이동                                             │
│                                                                     │
│  【Linux 커널 queued_spinlock】                                       │
│  ──────────────────                                                  │
│  // MCS 변형, NUMA 친화적                                              │
│  // /include/linux/qspinlock.h                                       │
│                                                                     │
│  【스레드 풀 크기 제한】                                                │
│  ──────────────────                                                  │
│  // CPU 코어 수 기반 스레드 풀                                          │
│  int num_threads = std::thread::hardware_concurrency();            │
│  ThreadPool pool(num_threads);  // 코어 수만큼                         │
│                                                                     │
│  // 과도한 스레드 생성 방지                                             │
│  // 예: 64코어에서 64 스레드만 생성                                      │
│                                                                     │
│  【락 스트라이핑】                                                      │
│  ──────────────────                                                  │
│  // 하나의 락 대신 여러 락                                               │
│  constexpr int STRIPES = 64;                                        │
│  std::array<std::mutex, STRIPES> locks;                            │
│                                                                     │
│  std::mutex& get_lock(int key) {                                    │
│      return locks[key % STRIPES];                                   │
│  }                                                                  │
│                                                                     │
│  // 스래싱 임계점 상향                                                   │
│                                                                     │
│  【성능 비교】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  64 코어 시스템, 높은 경합                                            │ │
│  │  ───────────────────────────────────────────────                 │ │
│  │  방식                    처리량 (ops/sec)                           │ │
│  │  ────                    ────────────                              │ │
│  │  단일 스핀락               10,000  (스래싱!)                          │ │
│  │  단일 스핀락 + 백오프        50,000                                   │ │
│  │  MCS 락                  200,000                                   │ │
│  │  락 스트라이핑 (64)        500,000                                   │ │
│  │  락 프리 (CAS)            800,000                                   │ │
│  │                                                             │ │
│  │  스래싱 완화로 수십배 성능 향상                                        │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【스래싱 감지】                                                       │
│  ──────────────────                                                  │
│  # perf top로 CPU 시간 확인                                           │
│  # 스핀락 함수가 상위에 있으면 스래싱 의심                                   │
│  perf top -p <pid>                                                  │
│                                                                     │
│  # CPU 100%인데 처리량 낮으면 스래싱                                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 과도한 락 경합으로 시스템 마비

• 감지: perf top, CPU 100%
```
