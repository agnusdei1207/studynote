# 독자-저자 문제 (Readers-Writers Problem)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 여러 명의 독자가 동시에 데이터를 읽는 것은 허용하되, 저자가 데이터를 쓰는 동안에는 독자와 다른 저자 모두의 접근을 차단하는 **공유/배타적 접근 제어** 문제다.
> 2. **가치**: "읽기는 병렬, 쓰기는 직렬"이라는 원칙을 통해 데이터 무결성을 유지하면서도, 뮤텍스만 사용했을 때보다 읽기 작업의 성능(Concurrency)을 극대화한다.
> 3. **융합**: 데이터베이스의 공유 락(Shared Lock)과 배타적 락(Exclusive Lock), 파일 시스템의 파일 점유 모델, 캐시 일관성 프로토콜의 기초 이론이다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 독자(Reader)는 데이터를 읽기만 하므로 여러 명이 동시에 들어와도 안전하다. 하지만 저자(Writer)는 데이터를 수정하므로 혼자서만 작업해야 한다. 이때 "누구에게 우선순위를 줄 것인가"에 따라 제1유형(독자 우선)과 제2유형(저자 우선)으로 나뉜다.

- **💡 비유**: 독자-저자 문제는 "칠판(공유 데이터)에 적힌 공지사항"과 같다. 학생들이 공지사항을 읽는 동안(Reader)은 수십 명이 한꺼번에 봐도 문제가 없다. 하지만 선생님이 공지사항을 수정하는 동안(Writer)은 아무도 봐서는 안 되고, 학생이 보고 있는 동안에도 선생님은 수정할 수 없다.

- **📢 섹션 요약 비유**: 마치 박물관의 전시품을 관람객들은 수십 명씩 동시에 볼 수 있지만, 큐레이터가 전시품을 교체할 때는 관람객의 접근을 모두 막아야 하는 운영 규칙과 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 제1유형: 독자 우선 (Readers Preference) 해법

독자가 한 명이라도 있는 동안에는 저자가 들어오지 못하게 막고, 새로운 독자는 계속 들어올 수 있게 한다.

```c
// 독자 로직
wait(mutex);         // readCount 보호
readCount++;
if (readCount == 1) 
    wait(rw_mutex);  // 첫 독자가 저자 진입 차단
signal(mutex);

... reading ...

wait(mutex);
readCount--;
if (readCount == 0)
    signal(rw_mutex); // 마지막 독자가 저자 진입 허용
signal(mutex);

// 저자 로직
wait(rw_mutex);       // 배타적 접근
... writing ...
signal(rw_mutex);
```

**[다이어그램 해설]** `readCount`는 현재 읽고 있는 독자의 수다. 첫 번째 독자가 대표로 `rw_mutex`를 잠가 저자를 막고, 이후 독자들은 카운트만 올리고 바로 들어온다. 마지막 독자가 나갈 때 대표로 잠금을 푼다. 이 방식은 **저자의 기아 상태(Starvation)**를 유발할 수 있다.

### 제2유형: 저자 우선 (Writers Preference)
저자가 대기 중이면 새로운 독자가 들어오지 못하게 하여, 저자가 너무 오래 기다리지 않게 하는 방식이다. 복잡하지만 공정성 측면에서 유리하다.

- **📢 섹션 요약 비유**: 독자 우선은 "손님이 한 명이라도 있으면 가게 문을 닫지 않는 것"이고, 저자 우선은 "주인이 수리하겠다고 공지하면 더 이상 손님을 받지 않는 것"과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Read-Write Lock (RW Lock)의 상태 매트릭스

| 요청 \ 현재 상태 | Reading (공유) | Writing (배타) |
|:---|:---:|:---:|
| **Read 요청** | **허용 (Success)** | 대기 (Wait) |
| **Write 요청** | 대기 (Wait) | 대기 (Wait) |

이 매트릭스는 현대 DBMS의 락 관리자(Lock Manager)가 트랜잭션을 제어하는 기본 로직이다. 읽기 비중이 압도적으로 높은 시스템(예: 웹 뉴스, 위키피디아)에서는 RW Lock 사용 시 성능이 수십 배 향상된다.

- **📢 섹션 요약 비유**: 여러 명의 시청자가 방송을 보는 것은 괜찮지만, 방송 송출 장비를 고칠 때는 시청자가 볼 수 없게 차단해야 하는 방송국 운영 시스템과 같습니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 캐시 서버 동기화
- **상황**: 수천 개의 요청이 캐시에서 데이터를 읽어가고, 1분에 한 번씩 백엔드에서 캐시를 갱신함.
- **해결책**: 일반 뮤텍스를 쓰면 읽기 요청끼리도 서로를 기다려야 하므로 성능이 급락한다. `pthread_rwlock_t`나 Java의 `ReentrantReadWriteLock`을 사용하여 읽기 스레드 간의 병렬성을 확보한다.

### 도입 체크리스트
- **Read/Write Ratio**: 읽기 비중이 쓰기보다 훨씬 높은가? (비슷하다면 일반 뮤텍스가 오버헤드가 적어 유리함)
- **Starvation Strategy**: 저자가 영원히 락을 얻지 못하는 현상을 방지하기 위한 타임아웃이나 우선순위 조정이 되어 있는가?

- **📢 섹션 요약 비유**: 대출자가 많은 베스트셀러(인기 데이터)일수록, 한 명씩 빌려주는 것보다 열람실(RW Lock)에 비치해 다 같이 보게 하는 것이 효율적인 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 독자-저자 해법은 시스템의 동시성(Concurrency)을 극대화하면서도 데이터의 일관성을 완벽하게 유지하는 스마트한 자원 관리 기법이다.
- **미래 전망**: 쓰기 작업이 읽기를 방해하지 않도록 하는 **MVCC (Multi-Version Concurrency Control)** 기술로 발전하여, 현대적인 고성능 DB(PostgreSQL, Oracle 등)의 핵심 엔진으로 자리 잡고 있다.

- **📢 섹션 요약 비유**: 규칙이 세밀해질수록 불편함은 줄고 효율은 높아지듯, 정교한 독자-저자 모델은 시스템의 흐름을 더 원활하게 만듭니다.
