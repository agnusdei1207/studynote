# RCU (Read-Copy-Update)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RCU는 읽기 작업에 대해서는 아무런 락도 걸지 않고(Lock-free), 쓰기 작업은 데이터를 복사하여 수정한 뒤 원본 포인터를 원자적으로 교체하는 고성능 동기화 기법이다.
> 2. **가치**: 읽기 성능이 압도적으로 중요한 커널 데이터 구조(라우팅 테이블 등)에서 읽기 스레드 간의 간섭을 완전히 제거하여 멀티코어 확장성을 극대화한다.
> 3. **융합**: 가비지 컬렉션(GC)과 유사한 **Grace Period(유예 기간)** 개념을 도입하여, 구버전 데이터를 참조하는 독자가 모두 사라진 시점에 안전하게 메모리를 해제한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 기존의 Read-Write Lock은 독자가 들어올 때도 카운터를 수정해야 하므로 캐시 라인 바운싱(Cache Line Bouncing) 성능 저하가 발생한다. RCU는 "읽는 사람은 그냥 읽게 두자"는 철학이다. 대신 쓰는 사람은 기존 데이터를 직접 고치지 않고, 복사본을 만들어 고친 후 포인터만 '탁' 바꾼다. 이전 데이터를 보고 있던 독자들이 다 나갈 때까지 기다렸다가 옛날 데이터를 삭제한다.

- **💡 비유**: RCU는 "게시판의 공지사항 교체"와 같다. 사람들이 공지사항을 읽고 있는 동안 종이를 뺏는 게 아니라, 옆에서 새 종이를 다 쓴 다음 기존 종이 위에 덧붙인다. 이전 종이를 읽기 시작한 사람들이 모두 다 읽고 떠나면(Grace Period), 그때 비로소 뒤에 숨겨진 헌 종이를 떼어 버리는 방식이다.

- **📢 섹션 요약 비유**: 독자에게는 아무런 제약을 주지 않으면서도, 조용히 세대교체를 준비하는 '그림자 동기화' 기법과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### RCU의 3단계 동작

1. **Read (rcu_read_lock)**: 단순히 선점만 금지하고 데이터를 읽는다. 오버헤드가 거의 0이다.
2. **Copy & Update**: 데이터를 복사하여 수정하고, 포인터를 새 위치로 바꾼다.
3. **Reclaim (Grace Period)**: 모든 CPU가 문맥 교환(Quiescent State)을 한 번씩 거칠 때까지 기다린 후, 구버전 메모리를 해제한다.

```text
  [ Reader 1 ] ──▶ [ Old Data ] ─────────────▶ [ Finish ]
                                       │
  [ Writer ]   ──▶ [ Copy & Update ] ──┼──▶ [ Point to New ] ──▶ [ Wait Grace Period ] ──▶ [ Free Old ]
                                       │
  [ Reader 2 ] ────────────────────────┼──▶ [ New Data ] ─────────────▶ [ Finish ]
```

**[다이어그램 해설]** 쓰기 작업이 일어난 시점에 이미 읽기를 시작한 Reader 1은 끝날 때까지 Old Data를 안전하게 본다. 쓰기 이후에 진입한 Reader 2는 New Data를 본다. 모든 CPU가 문맥 교환을 한 번씩 거치면(Grace Period 종료), Old Data를 참조하는 독자가 더 이상 없음을 보장할 수 있으므로 안전하게 메모리를 해제한다.

### 핵심 API
- `rcu_read_lock()` / `rcu_read_unlock()`: 읽기 임계 구역 표시.
- `rcu_assign_pointer()`: 새 데이터를 가리키도록 포인터 업데이트 (Memory Barrier 포함).
- `synchronize_rcu()`: Grace Period가 끝날 때까지 쓰기 스레드를 블로킹.

- **📢 섹션 요약 비유**: 새 길을 닦아놓고 표지판만 바꾸는 방식이라, 운전자(독자)는 멈출 필요가 없고 공사 인부(저자)만 조금 더 고생하는 시스템입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Read-Write Lock vs RCU

| 비교 항목 | Read-Write Lock | RCU |
|:---|:---|:---|
| **읽기 오버헤드** | 카운터 수정 (Atomic) 발생 | 거의 없음 (Pointer 참조) |
| **읽기 간 간섭** | 캐시 동기화 부하 발생 | 전혀 없음 |
| **쓰기 오버헤드** | 독자가 없을 때까지 대기 | 복사 비용 + Grace Period 대기 |
| **확장성** | 코어 수가 많아지면 급감 | 코어 수에 상관없이 선형적 |
| **주 사용처** | 읽기/쓰기 비중이 비슷할 때 | **읽기 비중이 압도적일 때** |

- **📢 섹션 요약 비유**: RW Lock은 "순번 대기표"이고, RCU는 "무제한 자유 열람"입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 커널 라우팅 테이블 업데이트
- **상황**: 네트워크 패킷은 초당 수백만 개가 들어오며 라우팅 테이블을 조회함. 라우팅 경로는 가끔씩만 변경됨.
- **해결책**: RCU를 적용한다. 패킷 처리 스레드들은 락 없이 초고속으로 테이블을 조회하고, 관리자 스레드는 RCU 방식으로 경로를 업데이트하여 패킷 처리에 지연(Jitter)을 주지 않는다.

### 도입 체크리스트
- **Read Intensive**: 읽기 비중이 90% 이상인가?
- **Memory Pressure**: 쓰기가 빈번하면 구버전 데이터가 메모리에 쌓일 수 있는데, 이를 감당할 수 있는가?
- **Grace Period Latency**: 쓰기 스레드가 유예 기간 동안 기다려도 되는 구조인가?

- **📢 섹션 요약 비유**: 명절의 고속도로 통행료 면제처럼, 대다수(독자)의 편의를 위해 시스템적인 양보(메모리 유지)를 선택하는 전략입니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: RCU는 멀티코어 하드웨어의 병렬성을 소프트웨어가 방해하지 않게 하는 최고의 수단이며, 리눅스 커널의 고성능화에 결정적인 기여를 했다.
- **미래 전망**: 유저 모드에서도 사용할 수 있는 User-space RCU 라이브러리가 발전하고 있으며, 실시간 시스템에서도 유예 기간을 단축한 형태의 RCU 연구가 활발히 진행 중이다.

- **📢 섹션 요약 비유**: 과거의 데이터를 소중히 다루며 미래의 데이터를 준비하는 온고지신의 정신이 담긴 동기화 기법입니다.
