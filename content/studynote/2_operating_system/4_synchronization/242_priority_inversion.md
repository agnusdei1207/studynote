# 우선순위 역전 (Priority Inversion)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 우선순위 역전은 낮은 우선순위의 프로세스가 점유한 자원을 높은 우선순위의 프로세스가 기다리는 동안, 중간 우선순위 프로세스가 CPU를 가로채어 높은 우선순위 프로세스의 실행이 지연되는 현상이다.
> 2. **가치**: 실시간 시스템(Real-time System)에서 마감 시간(Deadline)을 놓치게 만드는 치명적인 설계 결함으로, 1997년 화성 탐사선 패스파인더(Mars Pathfinder)의 잦은 리셋 원인으로 유명하다.
> 3. **융합**: 스케줄링 정책과 동기화 메커니즘이 충돌하는 지점으로, 우선순위 상속(Inheritance)이나 우선순위 천장(Ceiling) 프로토콜을 통해 해결한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 높은 우선순위(H)는 낮은 우선순위(L)보다 먼저 실행되어야 한다. 그러나 L이 락(Mutex)을 쥐고 있는 상태에서 H가 그 락을 기다리게 되면, L이 락을 풀 때까지 H는 멈춘다. 이때 중간 우선순위(M)가 나타나 L의 CPU를 뺏어버리면, L은 락을 풀 기회조차 잃고 결과적으로 H가 무기한 기다리게 된다. 이것이 우선순위가 뒤바뀌는 역전 현상이다.

- **💡 비유**: "VIP 손님(H)이 식당에 왔는데, 마침 단 하나뿐인 예약석을 일반 손님(L)이 쓰고 있다. VIP는 일반 손님이 나가기를 기다리는데, 갑자기 단체 예약 손님(M)이 들이닥쳐 일반 손님의 서빙을 가로채고 식당을 독점해 버린다. 결국 VIP는 일반 손님보다도, 심지어 단체 손님보다도 늦게 밥을 먹게 된다."

- **📢 섹션 요약 비유**: 마치 구급차(H)가 길을 지나가야 하는데, 앞에 가는 소형차(L)가 교차로에서 신호를 기다리다 멈췄고, 그 옆으로 수많은 일반 승용차(M)들이 지나가느라 소형차가 움직이지 못해 결국 구급차도 꼼짝 못 하는 상황과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 우선순위 역전의 3인 구조

이 현상은 반드시 세 가지 수준의 우선순위가 얽힐 때 발생한다.

```text
  [ High Prio (H) ]      [ Medium Prio (M) ]      [ Low Prio (L) ]
          |                      |                      |
          |                      |              1. Acquire Lock(S)
          |                      |                      |
  2. Try Lock(S)                 |                      |
     (Blocked by L)              |                      |
          |                      |              3. Preempted by M
          |              4. Running...                  |
          |              (L is waiting)                 |
          |              (H is waiting!)                |
```

**[다이어그램 해설]**
1. 최하위 L이 락 S를 획득한다.
2. 최상위 H가 실행되어 락 S를 요청하지만, L이 쥐고 있어 블록(Blocked)된다.
3. 중간 순위 M이 등장하여 L을 밀어내고(Preemption) CPU를 차지한다.
4. M은 락 S와 상관없는 작업이지만 CPU를 오래 써버린다. 결과적으로 H는 락과 상관없는 M이 끝날 때까지 기다려야 하는 황당한 상황이 발생한다.

### 실제 사례: Mars Pathfinder
1997년 화성 탐사선 패스파인더는 착륙 후 며칠 만에 계속해서 시스템 리셋이 발생했다. 분석 결과, 공유 메모리를 관리하는 낮은 우선순위 태스크가 락을 쥔 채 중간 우선순위 태스크들에 의해 밀려났고, 이를 기다리던 높은 우선순위의 정보 버스 태스크가 워치독(Watchdog) 타임아웃을 유발한 것이 원인이었다.

- **📢 섹션 요약 비유**: 주방장(L)이 요리 도구(Lock)를 들고 있는데, 홀 직원(M)이 계속 주방장을 불러 심부름을 시키는 바람에, 도구를 기다리던 메인 셰프(H)가 요리를 시작도 못 하고 주방이 마비되는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 우선순위 역전의 해결책 비교

| 기법 | 우선순위 상속 (Inheritance) | 우선순위 천장 (Ceiling) |
|:---|:---|:---|
| **핵심 원리** | 락을 쥔 L의 우선순위를 잠시 H로 높임 | 락 자체에 최고 우선순위를 부여함 |
| **적용 시점** | 경합이 발생한 순간 (동적) | 락을 획득하는 순간 (정적/동적) |
| **장점** | 구현이 비교적 직관적임 | 데드락까지 예방 가능함 |
| **단점** | 연쇄적인 상속 체인 발생 가능 | 모든 락의 천장 값을 미리 알아야 함 |

- **📢 섹션 요약 비유**: 상속은 "급한 사람이 자기 권한을 빌려주는 것"이고, 천장은 "특수 장비를 잡는 순간 무조건 대장급 대우를 해주는 것"입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: RTOS 기반 제어 시스템
- **상황**: 자동차 엔진 제어 장치(ECU)에서 엔진 점화 태스크(H)가 데이터 로깅 태스크(L)의 락을 기다리다가, 오디오 재생 태스크(M)에게 CPU를 뺏겨 점화 타이밍을 놓침.
- **해결책**: Pthreads의 `PTHREAD_PRIO_INHERIT` 속성을 뮤텍스에 적용한다. L이 락을 쥐고 있는 동안 H가 대기하면, L의 우선순위가 즉시 H 수준으로 격상되어 M이 끼어들지 못하게 막고, L이 락을 풀면 다시 원래대로 돌려놓는다.

### 도입 체크리스트
- **Lock Holding Time**: 낮은 우선순위 태스크가 락을 너무 오래 쥐고 있지는 않은가?
- **Nested Locks**: 여러 락이 중첩되어 상속 관계가 복잡해지면 데드락 위험이 없는가?

- **📢 섹션 요약 비유**: 중요한 서류를 결재받아야 할 때, 말단 사원이 결재판을 들고 있다면 잠시 그 사원을 '부장 대우'로 승격시켜 아무도 방해 못 하게 보호하는 전략과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 우선순위 역전 해결은 시스템의 예측 가능성(Predictability)을 확보하고, 실시간성 보장이 필요한 미션 크리티컬 시스템의 안정성을 완성한다.
- **미래 전망**: 자율주행, 로봇 제어 등 실시간 처리가 필수적인 분야가 늘어남에 따라, 커널 수준뿐만 아니라 미들웨어와 애플리케이션 프레임워크 수준에서도 이러한 방어적 스케줄링 설계가 기본 사양이 될 것이다.

- **📢 섹션 요약 비유**: 서열이 엄격한 사회에서도 공동의 목표를 위해 잠시 권한을 양보하고 협력하는 규칙이 전체의 조화를 만듭니다.
