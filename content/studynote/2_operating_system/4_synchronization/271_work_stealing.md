# 워크 스틸링 (Work Stealing)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 워크 스틸링은 멀티 코어 환경의 스레드 풀에서 자신의 큐에 작업이 떨어진 유휴(Idle) 프로세서가 다른 바쁜 프로세서의 큐 끝에서 작업을 "훔쳐와(Steal)" 대신 처리하는 동적 부하 분산 전략이다.
> 2. **가치**: 작업의 실행 시간이 불규칙한 환경에서도 모든 CPU 코어의 가동률을 평준화하여 시스템 전체의 처리량(Throughput)을 극대화하고 응답 시간을 단축한다.
> 3. **융합**: 자바의 Fork/Join Framework, Go 언어의 Goroutine 스케줄러, Intel TBB 등 현대적 병렬 런타임의 핵심 아키텍처이며, 락 경합을 최소화하기 위해 양방향 큐(Deque)를 활용한다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 중앙 집중식 스케줄러가 작업을 나누어주는 방식(Work Sharing)과 달리, 워크 스틸링은 각 코어가 분산된 큐를 가지며 스스로 작업을 찾아가는 분산형 스케줄링 방식이다.

- **💡 비유**: 워크 스틸링은 **"마트의 계산대 줄 이동"**과 같다. 내 계산대에 손님이 한 명도 없는데(유휴), 옆 계산대에는 줄이 길게 늘어서 있다면 내가 그 줄의 맨 뒤 손님을 데려와서 계산해주는 것과 같다.

- **등장 배경**:
  1. **정적 분산의 한계**: 작업을 미리 똑같이 나눠줘도, 어떤 작업은 빨리 끝나고 어떤 작업은 늦게 끝나 코어 간 불균형이 발생한다.
  2. **중앙 큐의 병목**: 모든 코어가 하나의 큐에서 작업을 꺼내려 하면 락 경합(Lock Contention)이 발생하여 성능이 저하된다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 덱(Deque, Double-Ended Queue) 기반의 메커니즘

워크 스틸링은 락 경합을 줄이기 위해 각 워커(Worker) 스레드마다 양방향 큐인 '덱'을 할당한다.

| 역할 | 동작 방향 | 상세 설명 |
|:---|:---|:---|
| **소유자 (Owner)** | **Top (LIFO)** | 자신의 작업을 처리할 때는 큐의 **앞쪽**에서 꺼냄 (캐시 지역성 활용) |
| **도둑 (Stealer)** | **Bottom (FIFO)** | 다른 워커의 작업을 훔칠 때는 큐의 **뒤쪽**에서 가져옴 (충돌 최소화) |

+++

### 워크 스틸링의 동작 알고리즘

```text
 [Worker A (Busy)]            [Worker B (Idle)]
  ┌──────────┐                 ┌──────────┐
  │ Task 1   │ <── Owner A     │          │
  ├──────────┤   (LIFO)        ├──────────┤
  │ Task 2   │                 │          │
  ├──────────┤                 ├──────────┤
  │ Task 3   │ <── Stealer B   │          │
  └──────────┘   (FIFO)        └──────────┘
        │                            │
        └─────── Steal Task 3 ───────┘
```

**[다이어그램 해설]** 워커 A는 자신이 넣은 최신 작업(Task 1)부터 처리한다. 반면, 할 일이 없어진 워커 B는 A의 큐에서 가장 오래된 작업(Task 3)을 훔쳐간다. 이렇게 앞과 뒤에서 서로 다른 포인터를 사용함으로써, A와 B가 동일한 작업에 접근하여 락 경합을 벌일 확률을 대폭 낮춘다.

+++

## Ⅲ. 융합 비교 및 다각도 분석

### 워크 쉐어링 (Sharing) vs 워크 스틸링 (Stealing)

| 비교 항목 | 워크 쉐어링 (Sharing) | 워크 스틸링 (Stealing) |
|:---|:---|:---|
| **제어 주체** | 중앙 스케줄러 (Push 방식) | 개별 워커 (Pull 방식) |
| **통신 비용** | 새 작업 생성 시마다 발생 | **자신의 큐가 빌 때만 발생** |
| **부하 균등화** | 예측 기반 (정적/동적) | **실제 상황 기반 (완전 동적)** |
| **구현 난이도** | 낮음 | 높음 (덱 및 원자 연산 필요) |
| **적합 환경** | 작업 크기가 균일할 때 | **작업 크기가 불균칙할 때** |

- **📢 섹션 요약 비유**: 워크 쉐어링은 "팀장이 업무를 배분해주는 것"이고, 워크 스틸링은 "할 일 끝난 팀원이 바쁜 팀원 일을 자발적으로 도와주는 것"입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
- **Java Fork/Join**: 재귀적으로 문제를 쪼개는 알고리즘(예: QuickSort)에서 분할된 태스크들을 워크 스틸링으로 처리하여 모든 코어를 활용한다.
- **Go Runtime**: 수백만 개의 고루틴을 수십 개의 OS 스레드에 매핑할 때, 특정 스레드에 고루틴이 몰리면 다른 유휴 스레드가 고루틴을 훔쳐와 실행 효율을 높인다.

### 성능 최적화 포인트 (주의점)
1. **스틸 비용**: 남의 작업을 훔쳐오는 과정 자체가 네트워크(분산 시스템)나 캐시 동기화(멀티 코어) 비용을 발생시킨다. 따라서 훔칠 작업이 충분히 클 때만 훔치는 것이 좋다.
2. **지역성 파괴**: 작업을 훔쳐오면 캐시에 로드된 데이터와 멀어지게 되어 캐시 미스(Cache Miss)가 증가할 수 있다.

+++

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과
- **코어 가동률 향상**: 99% 이상의 CPU 점유율을 유지하며 유휴 자원 낭비를 최소화한다.
- **확장성 (Scalability)**: 코어 수가 64개, 128개로 늘어나도 중앙 큐의 병목 없이 선형적인 성능 향상을 기대할 수 있다.

- **📢 섹션 요약 비유**: 워크 스틸링은 "누구 하나 놀지 않게 만드는 최고의 팀워크 시스템"과 같습니다.

+++

### 📌 관련 개념 맵
- **LIFO vs FIFO**: 워크 스틸링 덱의 동작 원리.
- **캐시 지역성 (Locality)**: 소유자가 앞쪽에서 작업하는 이유.
- **락 경합 (Lock Contention)**: 워커 간 작업 간섭 문제.

## 👶 어린이를 위한 3줄 비유 설명
1. 워크 스틸링은 친구들과 사탕을 포장할 때 **"내 바구니가 비면, 사탕이 많이 남은 친구의 바구니에서 가져오기"**와 같아요.
2. 바쁜 친구를 방해하지 않게 바구니의 맨 아래에 있는 사탕부터 몰래 가져온답니다.
3. 덕분에 누구는 놀고 누구는 고생하는 일 없이, 다 같이 아주 빨리 일을 끝낼 수 있어요!
