# 피터슨 알고리즘 (Peterson's Algorithm)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 피터슨 알고리즘은 두 개의 프로세스가 임계 구역(Critical Section)을 두고 다툴 때, **"내가 먼저 깃발(`flag`)을 들겠지만, 결정권(`turn`)은 무조건 상대방에게 먼저 양보한다"**는 역설적 철학을 단 4줄의 코드로 압축한 소프트웨어 동기화의 정수다.
> 2. **가치**: 데커 알고리즘의 복잡한 중첩 루프를 혁신적으로 단순화하면서도 상호 배제(Mutual Exclusion), 진행(Progress), 한정된 대기(Bounded Waiting)라는 데이크스트라의 3대 요건을 100% 수학적으로 증명해 낸 교과서의 바이블이다.
> 3. **융합**: 논리적으로는 완벽하나, 최신 CPU의 성능 최적화 기법인 **비순차 실행(Out-of-Order Execution)과 L1/L2 캐시 구조** 앞에서는 변수의 갱신 순서가 뒤틀리며 무참히 붕괴하므로, 현대 실무에서는 메모리 배리어(Memory Barrier)나 하드웨어 락(TAS, CAS)으로 완전히 대체되었다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 1981년 게리 피터슨(Gary L. Peterson)이 발표한, 두 프로세스 간의 임계 구역 문제를 해결하기 위한 순수 소프트웨어 락(Lock) 알고리즘이다. (이후 $N$개 프로세스로 확장된 버전도 존재하지만 핵심은 2개다.)
- **필요성**: 이전의 '데커 알고리즘'은 너무 복잡하고 코드가 길어 프로그래머들이 외워서 쓰거나 증명하기에 벅찼다. 직관적으로 락을 구현하려면 `if`와 `while`을 덜 써야 했다. 학자들은 "어떻게 하면 가장 적은 코드로 절대 뚫리지 않는 완벽한 논리적 자물쇠를 만들 수 있을까?"에 대한 해답(알고리즘의 최소화)이 필요했다.
- **💡 비유**: 좁은 골목길(임계 구역)에 두 차가 마주쳤을 때, 내가 먼저 상향등을 켜서(flag) 직진 의사를 밝히되, 반드시 창문을 열고 손짓으로 **"먼저 지나가시죠(turn)"라고 이타적인 양보를 하는 것**과 같다. 둘 다 상향등을 켜더라도, 늦게 손짓(양보)을 한 사람이 기다리게 되므로 절대 사고가 나지 않는다.
- **등장 배경**: 피터슨은 "충돌이 났을 때 양보를 묻는(데커 방식)" 기존의 복잡함을 버리고, "문 앞에 서자마자 일단 상대방에게 턴(Turn)을 넘겨버리고 시작하자"는 발상의 전환을 통해 20년 동안 풀지 못했던 데커의 복잡성을 단숨에 해결하며 컴퓨터 공학 교과서의 한 페이지를 장식했다.

```text
  [피터슨 알고리즘의 "선(先) 양보" 메커니즘]

  [ P0 (나) ]                              [ P1 (너) ]
  1. "나 화장실 갈래!" (flag[0] = true)
  2. "근데 너 먼저 가!" (turn = 1)          1. "나도 화장실 갈래!" (flag[1] = true)
                                         2. "근데 너 먼저 가!" (turn = 0)
  
  [ 운명의 3단계: 문 앞에서의 뺑뺑이 (while 대기) ]
  ▶ P0의 생각: "P1이 가고 싶어 하고(flag[1]==T), P1의 차례(turn==1)라면 난 기다릴게."
  ▶ P1의 생각: "P0이 가고 싶어 하고(flag[0]==T), P0의 차례(turn==0)라면 난 기다릴게."

  ✅ 최종 판결: `turn` 변수는 메모리에 하나뿐이므로, 
               가장 늦게 "너 먼저 가"라고 말한 놈(변수를 마지막에 덮어쓴 놈)이 
               상대방을 들여보내고 자기는 밖에서 기다리게 된다!
```
**[다이어그램 해설]** 피터슨 알고리즘의 천재성은 `turn` 변수를 대하는 태도에 있다. 데커가 `turn`을 "나의 권리"로 썼다면, 피터슨은 `turn`을 "상대방의 권리"로 썼다. 내가 문손잡이를 잡자마자 상대방에게 키를 던져준다. 상대방도 나에게 키를 던져주면, 결국 마지막에 공중에 떠 있는 키의 주인이 먼저 들어가게 되는 아름다운 상호 교차(Cross) 락킹 시스템이다.

- **📢 섹션 요약 비유**: 서로 밥값을 내겠다고 싸우는 상황입니다. P0이 카드를 내밀고(flag) P1에게 "네가 내(turn)"라고 합니다. P1도 카드를 내밀며 "네가 내"라고 합니다. 마지막에 계산원(메모리)에게 "네가 내"라고 말한 사람의 뜻대로 상대방이 계산(임계 구역 진입)하게 되는 유쾌하고 확실한 룰입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 소스 코드 (C언어 의사코드)와 3대 조건 증명

모든 전산학도가 숨을 멈추고 외워야 하는 그 4줄의 마법이다.
(전역 변수: `bool flag[2] = {false, false}; int turn;`)

```c
// 프로세스 i (0 또는 1)
// 프로세스 j 는 상대방 (1 - i)

void enter_critical_section(int i) {
    flag[i] = true;              // 1. 진입 의사 표시
    turn = j;                    // 2. 상대방에게 순서 양보 (핵심)
    
    // 3. 상대가 원하고, 상대 차례면 여기서 스핀락(대기)
    while (flag[j] && turn == j) { 
        /* do nothing (Busy Waiting) */ 
    }
}

void leave_critical_section(int i) {
    flag[i] = false;             // 4. 퇴장 의사 표시 (양보)
}
```

#### 수학적 100% 방어 증명 (Proof)
1. **상호 배제 (Mutual Exclusion)**
   - 두 프로세스가 동시에 임계 구역에 들어갔다고 가정해 보자. (귀류법)
   - 그 말은 P0도 `while`을 뚫고, P1도 `while`을 뚫었다는 뜻이다.
   - P0이 뚫으려면 `flag[1] == F` 이거나 `turn == 0` 이어야 한다. P1이 뚫으려면 `flag[0] == F` 이거나 `turn == 1` 이어야 한다.
   - 둘 다 `flag`를 `true`로 했으므로, 남은 조건은 `turn == 0` 이면서 동시에 `turn == 1` 이어야 한다. 하나의 변수가 두 값을 동시에 가질 수 없으므로(모순), 절대 두 놈이 동시에 들어갈 수 없다. (상호 배제 증명 완료)
2. **진행 (Progress)**
   - P1이 임계 구역 밖에서 놀고 있으면 `flag[1] = false`다.
   - P0은 `turn` 값이 뭐든 상관없이 `while(false && ...)`에 의해 즉각 뚫고 들어간다. 혼자일 땐 방해받지 않는다. (진행 증명 완료)
3. **한정된 대기 (Bounded Waiting)**
   - P0이 대기 중일 때, P1이 방에서 나오며 `flag[1] = false`로 바꾼다. 이때 갇혀있던 P0이 즉시 풀려나 들어간다.
   - P1이 얄밉게 바로 다시 들어오려 해도 `turn = 0`으로 P0에게 턴을 넘겨버리기 때문에 P0의 등 뒤에서 멈춰야 한다. 새치기 불가. (한정된 대기 증명 완료)

- **📢 섹션 요약 비유**: 이 코드는 완벽한 기계장치입니다. 톱니바퀴 2개(flag)와 걸쇠 1개(turn)가 맞물려 돌아가며, 절대로 두 개의 톱니바퀴가 동시에 하나의 구멍(임계 구역)에 빠지지 않도록 물리적인 잠금을 형성해 내는 소프트웨어 자물쇠의 최고봉입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 피터슨 알고리즘 vs 하드웨어 TAS (Test-And-Set)

피터슨이 논리적으로 완벽하는데도 현대 커널이 피터슨을 버리고 하드웨어 명령어(TAS/CAS)를 쓰는 이유를 비교한다.

| 비교 항목 | 피터슨 알고리즘 (소프트웨어) | Test-And-Set (하드웨어) |
|:---|:---|:---|
| **실행 원리** | 변수 2개(`flag`, `turn`)를 읽고 쓰며 논리적 유추 | 메모리 값을 읽고 1로 덮어쓰는 동작을 CPU 1클럭에 묶어버림 |
| **적용 프로세스 수**| 기본적으로 **2개** (N개용 빵집 알고리즘은 극도로 무거움) | **무한대 (1만 개)**가 `while` 하나에 붙어도 통제 가능 |
| **신뢰성 (현대 CPU)**| ❌ **비순차 실행 시 박살남** (Memory Barrier 떡칠 필요) | ✅ **100% 안전함** (실리콘 레벨에서 버스 락을 걸어버림) |
| **한정된 대기 (공평)**| ✅ **완벽 보장** (새치기 절대 불가) | ❌ **보장 안 됨** (운 나쁘면 영원히 TAS 실패하여 기아 발생) |

하드웨어 방식(TAS)은 피터슨보다 훨씬 무식하고 "한정된 대기"도 지켜주지 못한다. 그러나 피터슨 알고리즘을 N개의 스레드로 확장하려면 빵집 알고리즘처럼 수많은 배열과 루프를 만들어야 해서 오버헤드가 CPU를 찢어버린다. TAS는 속도가 미친 듯이 빠르고 스레드 수의 제약이 없기 때문에, 기아 상태 방지는 OS의 대기 큐(Wait Queue)나 뮤텍스 구조로 떠넘기는 융합을 택했다.

- **📢 섹션 요약 비유**: 피터슨은 정교한 "규칙과 예의"를 통해 2명의 아이가 싸우지 않게 통제하는 방법입니다(100명이면 규칙이 너무 복잡해짐). TAS는 그냥 방 문에 "무식하고 거대한 쇠사슬 자물쇠" 하나를 달아두는 겁니다. 1만 명이 와도 쇠사슬 앞에서는 한 명만 들어갈 수 있으니 통제가 너무 쉽습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오
1. **피터슨 알고리즘이 현대 CPU에서 붕괴하는 원리 (Out-of-Order Execution)**: 주니어 개발자가 "OS 수업에서 배운 피터슨을 C++로 짜봐야지!" 하고 멀티코어 환경에서 코드를 돌렸다.
   - **사건 발생**: 스레드 A와 B가 가끔 임계 구역에서 쾅 충돌한다. "증명된 알고리즘인데 왜 터지지?"
   - **아키텍트 분석**: 현대 컴파일러와 CPU 코어는 성능을 위해 코드를 '보이는 순서대로' 실행하지 않는다. 
     A코어가 `flag[0]=true`를 레지스터에 써놓고 메인 메모리에 내리기 전에(캐시 딜레이), CPU가 "어차피 `turn=1`이랑 상관없는 코드잖아?"라며 **`turn=1`을 먼저 메모리에 써버린다**. 
     이 틈을 타 B코어가 `flag[0]`를 읽으면 캐시 미반영으로 인해 `false`로 읽히고, B는 그냥 방으로 난입한다. 즉, **"논리는 완벽하나 물리 법칙(메모리 가시성)이 논리를 배신"**한 것이다.
2. **메모리 배리어 (Memory Barrier / Fence)를 통한 심폐소생술**: 실무에서 굳이 소프트웨어 락을 써야 한다면(혹은 Lock-free 자료구조를 짠다면) 이 물리 법칙을 제어해야 한다.
   - **실무 조치**: 코드 사이에 `mfence` (메모리 배리어) 어셈블리 명령어를 박아 넣는다. 
     `flag[0]=true;`
     **`Memory_Barrier();`** // "이 선을 넘기 전에 무조건 메모리에 쓰고 넘어가라! 순서 섞지 마!"
     `turn=1;`
   - 이렇게 배리어를 치면 피터슨은 현대 CPU에서도 정상 동작한다. 하지만 배리어 명령어 자체가 캐시 파이프라인을 멈추게 하므로 성능 오버헤드가 막대하여, 결국 실무에서는 `std::mutex` 나 `Atomic` 객체를 얌전히 쓰는 것이 절대 진리다.

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │     멀티코어 환경에서의 동기화 설계 실무 가이드라인 (Architect's Rule)│
  ├───────────────────────────────────────────────────────────────────────┤
  │                                                                       │
  │   [요구사항: 스레드 간 완벽한 동기화와 고성능(TPS)을 동시에 획득하라] │
  │                │                                                      │
  │                ▼ 1원칙: 바퀴(소프트웨어 락)를 재발명하지 마라.        │
  │   [ ❌ 피터슨 / 데커 등 수제(Hand-made) 소프트웨어 락 금지 ]          │
  │     - 이유: 캐시 일관성, 파이프라인 재배치 등 하드웨어 예외를         │
  │             개발자 레벨에서 100% 커버하는 것은 불가능에 가깝다.       │
  │                                                                       │
  │                ▼ 2원칙: 하드웨어가 빚은 가장 튼튼한 방패를 들어라.    │
  │   [ ✅ OS Mutex / Semaphore / Spinlock API 사용 ]                     │
  │     - 장점: OS 커널이 밑바닥에서 메모리 배리어와 TAS를 알아서         │
  │             다 세팅해 두었으므로 100% 안전하다.                       │
  │                                                                       │
  │                ▼ 3원칙: 방패가 무거우면 락-프리 검(Sword)을 쥐어라.   │
  │   [ 🚀 CAS(Compare-And-Swap) 기반 Atomic Variable 사용 ]              │
  │     - 장점: OS 개입(Sleep/Wakeup) 0. 궁극의 논블로킹 동기화 달성.     │
  └───────────────────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** "학교에서 배운 피터슨 알고리즘은 실무에 쓸 수 없다." 이것이 시스템 엔지니어링의 차가운 현실이다. 소프트웨어가 하드웨어의 최적화 본능을 이길 수 없기 때문이다. 피터슨은 프로그래머에게 "병행성이 꼬일 때 논리적으로 어떻게 풀어야 하는가"에 대한 두뇌 트레이닝 용도로만 존재하며, 실전은 무조건 OS와 CPU가 제공하는 최하단 도구(API)에 의존해야 한다.

- **📢 섹션 요약 비유**: 피터슨 알고리즘은 종이로 접은 완벽한 비율의 돛단배입니다. 잔잔한 호수(단일 코어 이론)에서는 예술적으로 뜨지만, 파도가 치는 바다(멀티코어 비순차 실행 환경)에 띄우는 순간 순식간에 젖어서 찢어집니다. 바다를 건너려면 강철로 만든 모터보트(하드웨어 TAS 락)를 타야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
피터슨 알고리즘을 체화하면 프로그래머는 두 개의 비동기적 흐름이 마주칠 때 발생할 수 있는 교착 상태(Deadlock)의 본질을 깨닫고, "의사 표현(`flag`)"과 "양보(`turn`)"라는 두 가지 상태 공간(State Space)을 결합하여 동시성 오류를 막아내는 락-프리(Lock-free) 알고리즘 설계의 기초 체력을 완성할 수 있다.

### 결론 및 미래 전망
피터슨 알고리즘은 1980년대 컴퓨터 과학이 낳은 "소프트웨어 락의 마스터피스"다. 비록 현재의 복잡한 CPU 캐시 아키텍처에 의해 실무적 생명은 다했지만, 이 알고리즘이 증명한 "동기화 3대 조건(상호배제, 진행, 한정된 대기)의 동시 충족 가능성"은 컴퓨터 과학을 학문의 반열로 올린 위대한 증명이다. 
미래의 개발자들은 피터슨을 코딩하진 않겠지만, 분산 시스템(MSA) 간의 분산 락(Distributed Lock)이나 분산 합의 알고리즘(Paxos, Raft)을 짤 때 "누가 먼저 양보하여 시스템의 데드락을 풀 것인가"라는 피터슨의 위대한 '양보(`turn`)' 철학을 계속해서 빌려 쓰게 될 것이다.

- **📢 섹션 요약 비유**: 피터슨 알고리즘은 고대 그리스의 피타고라스 정리와 같습니다. 지금 당장 집을 지을 때 우리가 종이에 삼각형을 그리며 피타고라스 공식을 풀진 않지만(CAD 프로그램이 다 해줌), 그 수학적 원리가 없다면 세상의 어떤 튼튼한 건축물(동기화 시스템)도 지어질 수 없는 영원불멸의 진리입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **데커의 알고리즘 (Dekker's)** | 피터슨의 아버지. 최초로 동기화 3조건을 만족했으나 코드가 더러워 피터슨에게 "단순함"으로 압살 당했다. |
| **비순차 실행 (Out-of-Order Execution)** | 피터슨 알고리즘의 숨통을 끊어버린 현대 CPU의 최적화 기술로, 변수 할당 순서를 맘대로 섞어버린다. |
| **메모리 배리어 (Memory Barrier)** | 비순차 실행 환경에서 피터슨 알고리즘이 돌아가게 멱살을 잡아주는 강제 코드 정렬 울타리다. |
| **Test-And-Set (TAS)** | 소프트웨어 변수 장난(피터슨)을 끝내고 CPU 하드웨어 단에서 락을 1클럭으로 잠가버린 무식하고 확실한 기술이다. |
| **상호 배제 (Mutual Exclusion)** | 피터슨이 `turn`과 `flag` 변수로 목숨 걸고 지켜내어, 두 놈이 동시에 화장실에 못 들어가게 막은 1번 조건이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 두 친구가 장난감을 놓고 싸우지 않게 아주 똑똑한 아저씨(피터슨)가 단 4줄짜리 말싸움 규칙을 만들었어요.
2. "나 장난감 놀고 싶어(깃발 들기)! 근데 네가 먼저 해(순서 양보)!" 이 이쁜 말 한마디만 먼저 건네면, 절대 싸우지도 않고(상호 배제) 둘 중 한 명은 무조건 놀게(진행) 되는 마법의 대화법이에요.
3. 하지만 요즘 컴퓨터들은 말을 너무 빨리해서 "네가 먼저 해"라는 말을 씹어먹고 서로 깃발만 들다가 부딪히는 바람에, 요즘은 이 대화법 대신 단단한 자물쇠(하드웨어 락)를 쓴답니다!
