# 257. 락 세분성 (Lock Granularity)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 락이 보호하는 데이터의 범위
> 2. **가치**: 동시성 vs 오버헤드 균형
> 3. **융합**: 락 스트라이핑, 파티셔닝, 락 에스컬레이션과 연관

---

## Ⅰ. 개요

### 개념 정의

락 세분성(Lock Granularity)은 **락이 보호하는 데이터의 범위로, 동시성과 오버헤드 사이의 트레이드오프**를 결정한다. 굵은 락은 관리 쉽지만 동시성이 낮고, 세밀한 락은 동시성이 높지만 관리가 복잡하다.

### 💡 비유: 도로 차단
락 세분성은 **도로 차단**과 같다. 전체 도로를 막으면(굵은 락) 관리는 쉽지만 교통 체증. 차선만 막으면(세밀한 락) 교통은 원활하지만 복잡하다.

### 락 세분성 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                락 세분성 구조                                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【세분성 스펙트럼】                                                    │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  굵은 락 (Coarse-grained)      ←────────────────→       세밀한 락 (Fine-grained)  │ │
│  │                                                             │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │  전체 데이터베이스 락                                        │ │ │
│  │  │         ↓                                                   │ │ │
│  │  │  테이블 락                                                   │ │ │
│  │  │         ↓                                                   │ │ │
│  │  │  페이지 락                                                   │ │ │
│  │  │         ↓                                                   │ │ │
│  │  │  행 락                                                       │ │ │
│  │  │         ↓                                                   │ │ │
│  │  │  컬럼 락                                                     │ │ │
│  │  └─────────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  │  동시성:    낮음 ─────────────────────────────── 높음           │ │
│  │  오버헤드:  낮음 ─────────────────────────────── 높음           │ │
│  │  복잡도:    단순 ─────────────────────────────── 복잡           │ │
│  │  교착상태:  드묾 ─────────────────────────────── 많음           │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【비교】                                                              │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  굵은 락 (Coarse-grained):                                         │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │  // 전체 해시맵에 하나의 락                                   │ │ │
│  │  │  lock(&hashmap_lock);                                     │ │ │
│  │  │  hashmap.insert(key, value);                              │ │ │
│  │  │  unlock(&hashmap_lock);                                   │ │ │
│  │  │                                                         │ │ │
│  │  │  장점: 단순, 교착상태 드묾                                    │ │ │
│  │  │  단점: 낮은 동시성, 병목                                    │ │ │
│  │  └─────────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  │  세밀한 락 (Fine-grained):                                          │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │  // 각 버킷에 별도 락                                        │ │ │
│  │  │  bucket = get_bucket(key);                                │ │ │
│  │  │  lock(&bucket_lock[bucket]);                              │ │ │
│  │  │  bucket.insert(key, value);                               │ │ │
│  │  │  unlock(&bucket_lock[bucket]);                            │ │ │
│  │  │                                                         │ │ │
│  │  │  장점: 높은 동시성                                         │ │ │
│  │  │  단점: 복잡, 많은 락 관리                                   │ │ │
│  │  └─────────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                락 세분성 상세                                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【데이터베이스 락 계층】                                              │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 데이터베이스 락 (가장 굵음)                                     │ │
│  │     • 전체 DB에 하나의 락                                          │ │
│  │     • 관리 단순, 동시성 매우 낮음                                   │ │
│  │                                                             │ │
│  │  2. 테이블 락                                                       │ │
│  │     • 테이블당 하나의 락                                             │ │
│  │     • MySQL MyISAM                                                  │ │
│  │                                                             │ │
│  │  3. 페이지 락                                                       │ │
│  │     • 페이지(8KB) 단위 락                                           │ │
│  │     • B-Tree 인덱스                                                 │ │
│  │                                                             │ │
│  │  4. 행 락 (InnoDB)                                                  │ │
│  │     • 행 단위 락                                                    │ │
│  │     • 높은 동시성                                                   │ │
│  │                                                             │ │
│  │  5. 컬럼 락 (가장 세밀)                                              │ │
│  │     • 컬럼 단위 락                                                   │ │
│  │     • 거의 사용 안 함                                                 │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【락 세분성 최적화 기법】                                             │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 락 스트라이핑 (Lock Striping)                                   │ │
│  │     • 데이터를 여러 파티션으로 분할                                   │ │
│  │     • 각 파티션에 별도 락                                            │ │
│  │     • ConcurrentHashMap (Java)                                     │ │
│  │                                                             │ │
│  │  2. 락 에스컬레이션 (Lock Escalation)                               │ │
│  │     • 세밀한 락이 많으면 굵은 락으로 전환                             │ │
│  │     • 오버헤드 감소                                                  │ │
│  │     • 동시성 감소                                                    │ │
│  │                                                             │ │
│  │  3. 적응적 락 (Adaptive Locking)                                    │ │
│  │     • 런타임에 락 세분성 조정                                        │ │
│  │     • Oracle: 행 락 → 테이블 락 자동 전환                            │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Java ConcurrentHashMap (Lock Striping)】                           │
│  ──────────────────                                                  │
│  import java.util.concurrent.*;                                     │
│                                                                     │
│  ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();│
│  // 내부적으로 16개 버킷, 각각 별도 락                                │
│  map.put("key", 1);  // 해당 버킷 락만 획득                         │
│                                                                     │
│  【C++ 세밀한 락 해시맵】                                             │
│  ──────────────────                                                  │
│  #include <mutex>                                                    │
│  #include <array>                                                    │
│  #include <list>                                                     │
│                                                                     │
│  template<typename K, typename V, size_t N = 16>                    │
│  class StripedHashMap {                                             │
│      struct Bucket {                                                │
│          std::mutex mutex;                                           │
│          std::list<std::pair<K, V>> entries;                        │
│      };                                                              │
│      std::array<Bucket, N> buckets;                                 │
│                                                                     │
│      size_t bucket_index(const K& key) {                            │
│          return std::hash<K>{}(key) % N;                            │
│      }                                                              │
│  public:                                                            │
│      void insert(const K& key, const V& value) {                    │
│          size_t idx = bucket_index(key);                            │
│          std::lock_guard<std::mutex> lock(buckets[idx].mutex);      │
│          buckets[idx].entries.push_back({key, value});              │
│      }                                                              │
│  };                                                                 │
│                                                                     │
│  【MySQL InnoDB 행 락】                                               │
│  ──────────────────                                                  │
│  -- 행 수준 락                                                        │
│  SELECT * FROM users WHERE id = 1 FOR UPDATE;                       │
│                                                                     │
│  -- 간격 락 (Gap Lock)                                               │
│  SELECT * FROM users WHERE id > 10 FOR UPDATE;                      │
│                                                                     │
│  -- 넥스트 키 락 (Next-Key Lock)                                     │
│  -- 행 + 간격                                                         │
│                                                                     │
│  【Oracle 락 에스컬레이션】                                           │
│  ──────────────────                                                  │
│  -- 행 락에서 시작                                                    │
│  UPDATE table SET col = val WHERE id = 1;                           │
│                                                                     │
│  -- 많은 행 수정 시 자동으로 테이블 락으로 에스컬레이션                 │
│  -- (MAXTRANS 설정에 따라)                                           │
│                                                                     │
│  【PostgreSQL 테이블 락 모드】                                         │
│  ──────────────────                                                  │
│  ACCESS SHARE       -- SELECT                                        │
│  ROW SHARE          -- SELECT FOR UPDATE                             │
│  ROW EXCLUSIVE      -- INSERT, UPDATE, DELETE                        │
│  SHARE UPDATE EXCLUSIVE -- VACUUM                                   │
│  SHARE              -- FK 생성                                        │
│  SHARE ROW EXCLUSIVE -- 없음                                         │
│  EXCLUSIVE          -- REFRESH MATERIALIZED VIEW                     │
│  ACCESS EXCLUSIVE   -- ALTER TABLE, DROP, TRUNCATE                   │
│                                                                     │
│  【선택 기준】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  상황                    권장 세분성                               │ │
│  │  ────                    ────────────                             │ │
│  │  읽기 위주               굵은 락                                   │ │
│  │  쓰기 빈번, 서로 다른 키   세밀한 락                                │ │
│  │  단순 애플리케이션        굵은 락                                   │ │
│  │  고성능 요구             세밀한 락                                  │ │
│  │  낮은 경합               굵은 락                                   │ │
│  │  높은 경합               세밀한 락                                  │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 락이 보호하는 데이터 범위
• 스펙트럼: 굵은 락 ↔ 세밀한 락
• 굵은 락: 단순, 낮은 동시성, 적은 오버헤드
• 세밀한 락: 복잡, 높은 동시성, 많은 오버헤드
• 기법: 스트라이핑, 에스컬레이션, 적응적
• 적용: DB 행/테이블 락, ConcurrentHashMap
```
