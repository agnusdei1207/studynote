# 윈도우 동기화 (Windows Synchronization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 윈도우 운영체제는 가벼운 프로세스 내 동기화를 위한 **유저 모드 객체(Critical Section)**와 시스템 전반의 동기화를 위한 **커널 모드 디스패처 객체(Dispatcher Object)**로 이원화된 동기화 아키텍처를 제공한다.
> 2. **가치**: 유저 모드 객체는 커널 진입 오버헤드를 줄여 고성능을 보장하며, 디스패처 객체는 뮤텍스, 세마포어, 이벤트 등 다양한 동기화 시나리오를 통일된 인터페이스(`WaitForSingleObject`)로 제어할 수 있게 한다.
> 3. **융합**: 윈도우의 비동기 I/O 모델인 IOCP(Input/Output Completion Port)와 결합하여 수만 개의 동시 연결을 효율적으로 처리하는 서버 아키텍처의 기반이 된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 윈도우 동기화는 성능 최적화에 초점을 맞춘다. 동일 프로세스 내 스레드들만 경쟁한다면 굳이 비싼 커널 모드로 전환할 필요가 없다. 이때 사용하는 것이 **Critical Section**이다. 반면 서로 다른 프로세스 간 동기화나 I/O 완료 신호를 기다려야 할 때는 커널이 관리하는 **Dispatcher Object**를 사용한다.

- **💡 비유**: Critical Section은 "우리 집 안에서 가족끼리 화장실 순서를 정하는 것"과 같아 절차가 간단하고 빠르다. Dispatcher Object는 "공공 도서관의 열람실 좌석을 예약하는 것"과 같아 신분증(Handle)을 지참하고 관리자(Kernel)의 승인을 받아야 하는 정식 절차가 필요하다.

- **📢 섹션 요약 비유**: 마치 사내 메신저(유저 모드)로 소통하는 것과 공식 공문(커널 모드)을 통해 부서 간 협조를 구하는 것의 효율성 차이와 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 유저 모드 vs 커널 모드 동기화 구조

윈도우는 하이브리드 접근 방식을 취한다.

```text
  [ User Mode Application ]
       |
  ┌────┴──────────────────────────┐
  │ Critical Section (CRITICAL_SECTION) │ ◀── No Context Switch if no contention
  └────┬──────────────────────────┘
       │ (Contention occurs)
       ▼
  [ Kernel Mode (Executive) ]
       |
  ┌────┴────────────────────────────────┐
  │ Dispatcher Objects (Event, Mutex, Sem) │ ◀── Signaled / Non-signaled states
  └─────────────────────────────────────┘
```

### 1. 크리티컬 섹션 (Critical Section)
- **특징**: 스핀락과 세마포어의 조합이다. 처음에는 잠시 스핀을 돌며 락을 기다리다가, 실패하면 비로소 커널 모드로 전환하여 대기한다.
- **장점**: 경합이 없을 때 시스템 콜 오버헤드가 0에 가깝다.

### 2. 디스패처 객체 (Dispatcher Object)
- **상태**: **Signaled** (자원 가용/이벤트 발생)와 **Non-signaled** (자원 점유 중/대기 중) 상태를 가진다.
- **Wait 함수**: `WaitForSingleObject()`나 `WaitForMultipleObjects()`를 통해 객체가 Signaled 상태가 될 때까지 스레드를 블로킹한다.

- **📢 섹션 요약 비유**: 처음엔 문을 두드려보고(Spin), 응답이 없으면 의자에 앉아 편히 기다리는(Blocking) 효율적인 대기 전략과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 유저 모드 객체 vs 커널 모드 객체

| 비교 항목 | 크리티컬 섹션 (Critical Section) | 뮤텍스 (Mutex - Kernel) |
|:---|:---|:---|
| **가시성** | 동일 프로세스 내에서만 보임 | 시스템 전체(모든 프로세스)에서 보임 |
| **성능** | 매우 빠름 | 상대적으로 느림 (Context Switch) |
| **안전성** | 프로세스 종료 시 자동 해제 안 됨 | 점유 스레드 종료 시 커널이 자동 해제 |
| **용도** | 일반적인 데이터 구조 보호 | 인스턴스 중복 실행 방지, IPC |

- **📢 섹션 요약 비유**: 크리티컬 섹션은 "가족 전용 열쇠"이고, 뮤텍스는 "마을 공용 창고의 마스터키"입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 애플리케이션 중복 실행 방지
- **상황**: 프로그램이 이미 실행 중이라면 추가로 실행되지 않게 막아야 함.
- **해결책**: 이름이 있는(Named) 커널 뮤텍스를 생성한다. `CreateMutex(NULL, FALSE, "MyUniqueAppName")`를 호출했을 때 `GetLastError()`가 `ERROR_ALREADY_EXISTS`를 반환하면 이미 실행 중인 것으로 판단하고 종료한다.

### 도입 체크리스트
- **Spin Count**: 크리티컬 섹션 생성 시 `InitializeCriticalSectionAndSpinCount`를 통해 적절한 스핀 횟수를 설정했는가?
- **Wait Timeout**: 커널 객체 대기 시 `INFINITE`를 지양하고 적절한 타임아웃을 설정했는가?

- **📢 섹션 요약 비유**: 이름표(Named Object)를 붙여두면 중복을 막을 수 있듯, 윈도우의 명명된 객체는 시스템 전반의 질서를 유지합니다.

## Ⅴ. 기대효과 및 결론

- **기대효과**: 윈도우의 이원화된 동기화 모델은 개발자에게 세밀한 성능 튜닝의 기회를 제공하며, 대규모 GUI 및 서버 애플리케이션의 반응성을 보장한다.
- **미래 전망**: 윈도우 커널의 지속적인 개선으로 유저 모드에서 동작하는 더 빠른 동기화 도구(SRW Lock 등)가 추가되고 있으며, 이는 멀티코어 환경의 확장성 문제를 해결하고 있다.

- **📢 섹션 요약 비유**: 상황에 맞는 도구를 골라 쓰는 목수가 좋은 집을 짓듯, 윈도우 동기화 API의 적재적소 활용이 고성능 윈도우 앱을 완성합니다.
