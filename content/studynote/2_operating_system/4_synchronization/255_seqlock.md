# SeqLock (순차 락 / Sequence Lock)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SeqLock은 일련번호(Sequence Counter)를 사용하여, 읽는 도중 쓰기가 발생했는지 여부를 사후에 검사함으로써 읽기 작업의 병렬성을 높이는 낙관적(Optimistic) 동기화 기법이다.
> 2. **가치**: 저자(Writer)가 독자(Reader)를 기다릴 필요가 없어 쓰기 성능이 보장되며, 독자 역시 락 획득 과정 없이 읽은 후 유효성만 체크하므로 오버헤드가 매우 적다.
> 3. **융합**: 리눅스 커널의 시스템 시간(`jiffies`), 경로명 캐시 등 데이터가 작고 일관성이 중요하며 읽기/쓰기가 잦은 곳에 주로 사용된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: SeqLock은 하나의 '순차 카운터'를 둔다. 저자는 쓰기 전후로 카운터를 1씩 올린다. 독자는 읽기 전의 카운터와 읽은 후의 카운터를 비교한다. 만약 카운터가 **홀수**거나(쓰기 중), 읽기 전후의 **값이 다르다면**(읽는 도중 쓰기 발생), 데이터가 오염된 것으로 간주하고 다시 읽는다.

- **💡 비유**: SeqLock은 "칠판에 계속 변하는 점수를 적는 것"과 같다. 점수를 적는 사람(저자)은 칠판 옆의 번호표(Counter)를 바꾸고 점수를 고친다. 점수를 보는 사람(독자)은 번호표를 확인하고 점수를 본 뒤, 다시 번호표를 확인한다. 번호표가 바뀌어 있다면 "아, 내가 보는 중에 점수가 바뀌었구나" 하고 다시 확인하는 식이다.

- **📢 섹션 요약 비유**: 일단 믿고 읽은 뒤에 나중에 확인하는 '선조치 후검증' 방식의 합리적인 신뢰 시스템과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SeqLock의 동작 알고리즘

```text
  [ Writer ]                         [ SeqLock Object ]                [ Reader ]
      |                                      |                             |
  1. Lock(mutex)                             |                     1. old_seq = counter
  2. counter++ (become odd)  ──▶ [ Counter: 1 ]                     2. Read Data...
  3. Write Data...                           |                             |
  4. counter++ (become even) ──▶ [ Counter: 2 ]                     3. new_seq = counter
  5. Unlock(mutex)                           |                     4. If (old != new) 
                                             |                           Goto Step 1
```

### 구현 코드 예시 (리눅스 커널 스타일)

```c
// Reader
unsigned int seq;
do {
    seq = read_seqbegin(&my_seqlock);
    // 데이터 읽기
    data = shared_data;
} while (read_seqretry(&my_seqlock, seq));

// Writer
write_seqlock(&my_seqlock);
// 데이터 수정
shared_data = new_value;
write_sequnlock(&my_seqlock);
```

### 주요 특징
- **저자 우선**: 저자는 절대 기다리지 않는다. (저자 간에는 뮤텍스로 상호 배제 필요)
- **독자 재시도**: 독자는 데이터 오염 감지 시 계속 재시도한다.
- **제약**: 데이터 구조 내에 포인터가 포함되어 있으면 안 된다. (독자가 유효하지 않은 포인터를 참조하여 Crash가 날 수 있기 때문)

- **📢 섹션 요약 비유**: 주필(저자)이 마감 시간에 쫓겨 기사를 수정하느라 독자를 기다려줄 여유가 없을 때, 독자가 알아서 최신 버전을 다시 확인하게 만드는 편집 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### RCU vs SeqLock

| 비교 항목 | RCU | SeqLock |
|:---|:---|:---|
| **독자 재시도** | 절대 없음 (구버전 보장) | 발생 가능 (데이터 오염 시) |
| **메모리 오버헤드** | 복사본 유지를 위해 높음 | 매우 낮음 (카운터 하나) |
| **포인터 사용** | 매우 적합 (포인터 원자 교체) | 위험함 (Crash 가능성) |
| **쓰기 오버헤드** | Grace Period 대기 필요 | 없음 (즉시 완료) |
| **데이터 크기** | 큰 구조체도 가능 | 작고 단순한 구조체에 적합 |

- **📢 섹션 요약 비유**: RCU는 "구버전 신문을 다 읽을 때까지 기다려주는 배려"이고, SeqLock은 "실시간 전광판을 다시 확인하게 하는 단호함"입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 시스템 시간 업데이트
- **상황**: OS 커널은 1ms마다 시간을 업데이트함. 수많은 애플리케이션이 이 시간을 수시로 읽어감.
- **해결책**: SeqLock을 사용한다. 시간 정보는 정수 몇 개로 이루어진 작은 데이터이므로 복사 비용이 적고, 독자가 아주 가끔 재시도하는 것이 락을 걸고 기다리는 것보다 훨씬 빠르기 때문이다.

### 도입 체크리스트
- **Data Size**: 읽어야 할 데이터가 한두 번의 CPU 사이클로 끝날 만큼 작고 단순한가?
- **Crash Safety**: 데이터에 포인터가 포함되어 있지 않은가? (가장 중요)

- **📢 섹션 요약 비유**: 시계의 분침이 돌아갈 때 시계를 멈추고 보는 것이 아니라, 보고 나서 "아, 지금 분침이 움직였나?" 하고 다시 보는 것이 더 자연스러운 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: SeqLock은 쓰기 성능을 희생하지 않으면서도 읽기 작업에 락 오버헤드를 전혀 주지 않는, 특정 시나리오에서의 최강의 솔루션이다.
- **미래 전망**: 64비트 원자적 연산이 발달하면서 정수 하나 정도는 락 없이 처리할 수 있게 되었지만, 여러 개의 연관된 값을 한꺼번에 보호해야 하는 영역에서는 SeqLock의 가치가 여전히 유효하다.

- **📢 섹션 요약 비유**: 확인의 책임은 독자에게 넘기고 진행의 속도는 저자에게 맡기는, 성능 중심의 합리적인 타협이 SeqLock의 미학입니다.
