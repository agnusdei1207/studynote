# 락-프리 자료구조 (Lock-free Data Structure)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 락-프리 자료구조는 뮤텍스나 세마포어 같은 전통적인 락을 사용하지 않고, 하드웨어의 원자적 연산(CAS 등)을 이용하여 여러 스레드가 동시에 안전하게 데이터를 조작할 수 있도록 설계된 자료구조다.
> 2. **가치**: 특정 스레드가 중단되어도 전체 시스템의 작업이 멈추지 않는 진행 보장(Progress Guarantee)을 제공하며, 락으로 인한 데드락, 우선순위 역전, 문맥 교환 오버헤드를 근본적으로 제거한다.
> 3. **융합**: 고성능 서버, 실시간 금융 거래 시스템, 고성능 임베디드 커널 등 지연 시간(Latency)에 극도로 민감한 환경에서 핵심 엔진으로 사용된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 락(Lock)은 안전하지만 한 번에 한 명만 일하게 한다. 락-프리는 모든 스레드가 동시에 달려들어 일을 하되, **Compare-and-Swap (CAS)** 같은 원자적 명령어로 마지막 '확정' 순간에만 충돌을 검사한다. 충돌이 나면 해당 스레드만 다시 시도하면 된다. 전체 시스템 관점에서는 누군가는 반드시 성공하므로 시스템은 계속 진행된다.

- **💡 비유**: 락-프리는 "자유로운 시장 경제"와 같다. 누구나 물건을 사려고 시도할 수 있고, 마지막 결제 순간에 재고가 있는지 확인(CAS)한다. 누군가 결제에 성공하면 재고가 줄어들고, 실패한 사람은 다시 쇼핑을 시작하면 된다. 시장 전체는 멈추지 않는다.

- **📢 섹션 요약 비유**: 한 명씩 줄 서서 들어가는 회전문 대신, 누구나 동시에 들어갈 수 있지만 문 통과 순간에만 부딪히지 않게 조율하는 똑똑한 출입 시스템과 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 락-프리의 핵심: CAS (Compare-and-Swap)
하드웨어가 제공하는 원자적 명령어다. `CAS(target, expected, new)`는 `target`의 값이 `expected`와 같을 때만 `new`로 바꾼다.

### 락-프리 스택 (Treiber Stack) 알고리즘

```text
  [ Thread A ]                   [ Shared Stack ]                   [ Thread B ]
       |                                |                                |
  1. Top = old_top               [ Top: Node 1 ]                 1. Top = old_top
  2. New_Node->next = Top               |                        2. New_Node->next = Top
       |                                |                                |
  3. CAS(&Top, old_top, New_A) ──▶ [ Success! ]                          |
       |                         [ Top: New_A ]                          |
       |                                |                        3. CAS(&Top, old_top, New_B)
       |                                |                        4. [ Failure! (Top changed) ]
       |                                |                        5. Retry from Step 1
```

**[다이어그램 해설]** 스레드 A와 B가 동시에 Push를 시도한다. 둘 다 현재의 Top 노드를 확인한다. A가 찰나의 순간 먼저 CAS에 성공하여 Top을 바꾼다. B는 자신이 알고 있던 'old_top'과 실제 'Top'이 달라졌으므로 CAS에 실패하고, 바뀐 Top을 기준으로 다시 작업을 시도한다.

### 락-프리의 조건
- **전체적인 진행 보장**: 시스템 전체를 봤을 때 무한한 시간 내에 적어도 하나의 스레드는 작업을 완료해야 한다.
- **아무도 멈추지 않음**: 한 스레드가 무한 루프에 빠지거나 종료되어도 다른 스레드는 영향을 받지 않는다.

- **📢 섹션 요약 비유**: 누군가 먼저 깃발을 꽂으면(CAS 성공), 뒤에 오던 사람은 깃발을 보고 경로를 수정(Retry)하여 계속 전진하는 원리와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 락 기반 (Lock-based) vs 락-프리 (Lock-free)

| 비교 항목 | 락 기반 (Lock-based) | 락-프리 (Lock-free) |
|:---|:---|:---|
| **동기화 기구** | Mutex, Semaphore | CAS, Atomic 연산 |
| **장점** | 구현이 쉽고 직관적임 | 데드락 없음, 높은 확장성, 낮은 지연 |
| **단점** | 데드락, 우선순위 역전 가능성 | 구현이 극도로 어렵고 ABA 문제 존재 |
| **부하 상황** | 스케줄링 오버헤드 증가 | CAS 재시도로 인한 CPU 소모 증가 |
| **사용처** | 일반적인 모든 애플리케이션 | 고성능 시스템 소트프웨어, 커널 |

- **📢 섹션 요약 비유**: 락은 "안전하지만 느린 기차"이고, 락-프리는 "빠르지만 사고 시 재주행이 필요한 슈퍼카"입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 고성능 로그 큐 (Log Queue)
- **상황**: 수천 개의 스레드가 동시에 로그를 남김. 일반 큐를 쓰면 락 경합으로 서비스 응답 속도가 느려짐.
- **해결책**: 락-프리 기반의 MPMC (Multi-Producer Multi-Consumer) 큐를 사용한다. 락으로 인한 문맥 교환 없이 모든 스레드가 각자 로그를 밀어넣을 수 있어 시스템 전체의 지연 시간이 획기적으로 줄어든다.

### 도입 체크리스트
- **ABA Problem**: 값이 A -> B -> A로 변했을 때 CAS가 성공해 버리는 논리 오류를 처리했는가?
- **Memory Reclamation**: 락-프리에서 노드를 삭제할 때, 다른 스레드가 여전히 그 노드를 참조하고 있을 위험은 없는가? (Hazard Pointers 등 필요)

- **📢 섹션 요약 비유**: 모든 스레드가 각자의 속도로 질주하되, 마지막 결승선에서만 부딪히지 않게 교통정리(CAS)를 해주는 지혜가 필요합니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 락-프리는 하드웨어 성능을 100% 이끌어내며, 특히 멀티코어 개수가 늘어날수록 락 기반 구조 대비 압도적인 성능 우위를 점한다.
- **미래 전망**: C++11, Java 8 이후 언어 차원에서 원자적 연산을 공식 지원하면서 락-프리 알고리즘의 적용 범위가 넓어지고 있으며, 더욱 강력한 진행 보장을 제공하는 **웨이트-프리 (Wait-free)** 연구로 이어지고 있다.

- **📢 섹션 요약 비유**: 규제가 없는 자유로운 경쟁이 경제를 활성화하듯, 락이 없는 자유로운 경쟁이 소프트웨어의 성능을 폭발시킵니다.
