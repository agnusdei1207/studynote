# 스케줄러 일드 (sched_yield / Scheduler Yield)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: `sched_yield()`는 현재 실행 중인 스레드가 남은 CPU 할당 시간(Time Slice)을 자발적으로 포기하고, 운영체제 스케줄러에게 다른 스레드를 실행할 기회를 넘겨주는 시스템 콜이다.
> 2. **가치**: 락 경합이 예상되거나 다른 스레드가 먼저 처리해야 할 작업이 있을 때, 무의미한 CPU 소모(Busy Waiting)를 줄이고 시스템의 전반적인 처리 효율을 높이는 양보 메커니즘이다.
> 3. **융합**: 사용자 수준 스레드 라이브러리(M:N 모델) 구현 시 스레드 간 협력적 스케줄링(Cooperative Scheduling)을 지원하는 기초 도구로 활용된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 스레드가 "나는 지금 당장 할 일이 없거나, 다른 누군가가 먼저 일을 해야 내 다음 일이 진행될 것 같아"라고 판단할 때 호출한다. 스케줄러는 이 스레드를 실행 큐의 맨 뒤로 보내고, 같은 우선순위의 다른 스레드에게 CPU를 넘긴다. 만약 실행 가능한 다른 스레드가 없다면 자기 자신이 다시 실행될 수도 있다.

- **💡 비유**: `sched_yield()`는 "편의점 계산대에서의 양보"와 같다. 내 물건을 계산하려는데 지갑을 차에 두고 왔다면, 뒤에 서 있는 손님에게 "먼저 계산하세요"라고 자리를 양보하고 나는 다시 줄의 맨 뒤로 가서 지갑을 챙겨오는 것과 같다.

- **📢 섹션 요약 비유**: 내가 지금 길을 막고 있는 것보다 남을 먼저 보내주는 것이 전체 교통 흐름(시스템 효율)에 도움이 된다는 이타적인 스케줄링 판단과 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### sched_yield()의 동작 메커니즘

운영체제 커널 내부에서 프로세스의 상태를 재조정한다.

```text
  [ Running State ] ──────▶ sched_yield() 호출
          │                         │
          │             1. Move to end of Ready Queue
          │             2. Trigger Context Switch
          ▼                         ▼
  [ Ready Queue ] ◀────────── [ Other Threads ]
     (T1, T2, T3)                (T2, T3, T1)
```

### 주요 특성
- **비차단 (Non-blocking)**: 스레드가 수면(Sleep) 상태로 들어가는 것이 아니라, 여전히 실행 가능한(Ready) 상태를 유지한다.
- **우선순위 존중**: 자기보다 우선순위가 낮은 스레드에게 양보하는 것이 아니라, **동일한 우선순위**를 가진 다른 스레드에게 기회를 준다.
- **이식성 문제**: 운영체제마다 일드(Yield) 이후 어느 위치로 가는지(맨 뒤인가, 중간인가)가 다를 수 있어 결과가 결정론적이지 않을 수 있다.

### 실전 코드 예시 (C)

```c
while (lock_is_busy) {
    // 락이 풀릴 때까지 CPU를 100% 쓰며 도는 대신,
    // 다른 스레드(아마도 락 소유자)가 실행될 수 있게 양보함.
    sched_yield(); 
}
```

- **📢 섹션 요약 비유**: 무작정 문을 두드리는 대신, 잠시 뒤로 물러나 다른 사람이 볼일을 보고 나오길 기다리는 매너 있는 행동과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### sleep() vs yield() vs block()

| 비교 항목 | sleep(n) | sched_yield() | block (Mutex wait) |
|:---|:---|:---|:---|
| **상태 전이** | Waiting (Timer) | Ready (Queue end) | Waiting (Resource) |
| **CPU 반납** | 확실함 (정해진 시간 동안) | 자발적 (타 스레드 존재 시) | 확실함 (자원 가용 시까지) |
| **복귀 시점** | 시간 경과 후 | 스케줄러의 선택 시 | 자원 해제(Signal) 시 |
| **용도** | 시간 지연 필요 시 | 협력적 양보 필요 시 | 자원 경합 해결 시 |

- **📢 섹션 요약 비유**: `sleep`은 "낮잠 자러 가기", `yield`는 "줄 양보하기", `block`은 "번호표 뽑고 기다리기"입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 사용자 정의 스핀락 최적화
- **상황**: 성능을 위해 사용자 정의 스핀락을 만들었는데, 경합이 심해지면 특정 코어가 100% 점유되며 다른 스레드가 락을 풀 기회조차 못 잡는 현상 발생.
- **해결책**: 스핀을 일정 횟수 이상 돌았는데도 실패하면 `sched_yield()`를 호출한다. 이를 통해 락 소유자가 CPU를 할당받아 락을 빨리 해제할 수 있는 확률을 높여준다.

### 도입 체크리스트
- **Priority Level**: 동일 우선순위에 다른 스레드가 많이 존재하는가? (없다면 일드는 아무 효과가 없음)
- **Starvation**: 일드를 너무 자주 호출하여 정작 자신의 작업 완료가 무한히 늦어지지는 않는가?

- **📢 섹션 요약 비유**: 내가 비켜줘도 갈 사람이 아무도 없다면(단독 스레드), 양보는 의미 없는 행동이 됩니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 적절한 `sched_yield()` 사용은 시스템의 응답성을 높이고, 락 경합으로 인한 불필요한 전력 소모와 열 발생을 억제한다.
- **미래 전망**: 현대 스케줄러(CFS 등)가 매우 영리해지면서 수동적인 일드의 필요성은 줄어들고 있지만, 사용자 수준 스레드(Fiber, Coroutine) 관리자에게는 여전히 핵심적인 제어 수단이다.

- **📢 섹션 요약 비유**: 먼저 가라고 손짓하는 작은 양보가 도시 전체의 교통 정체를 막듯, 일드 시스템 콜은 시스템의 원활한 흐름을 만드는 윤활유입니다.
