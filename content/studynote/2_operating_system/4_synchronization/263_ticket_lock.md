# 티켓 락 (Ticket Lock)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 티켓 락 (Ticket Lock)은 은행의 번호표 시스템을 동기화에 도입하여, 스레드들이 도착한 순서대로 락을 획득하게 함으로써 기아 상태 (Starvation)를 방지하는 공정한 스핀락 (Spinlock) 메커니즘이다.
> 2. **가치**: 일반 스핀락의 무작위성(Unfairness) 문제를 해결하여 실행 시간의 예측 가능성을 높이고, 특정 스레드가 무한히 대기하는 리스크를 아키텍처적으로 차단한다.
> 3. **융합**: 리눅스 커널(2.6.25~3.14 버전) 등에서 표준 락으로 사용되었으며, 현대의 큐잉 스핀락 (MCS Lock / qspinlock)으로 발전하는 과도기적이지만 매우 중요한 알고리즘이다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 티켓 락 (Ticket Lock)은 두 개의 정수 변수(`ticket`과 `turn`)를 사용하여 구현된다. 락을 요청하는 스레드는 번호표(`ticket`)를 뽑고, 현재 호출 중인 번호(`turn`)가 자신의 번호와 일치할 때까지 기다리는 방식이다.

- **💡 비유**: 은행 창구의 번호표 기계와 같다. 고객(스레드)이 오면 번호표를 뽑는다(ticket++). 은행원(CPU)은 "10번 손님 오세요!"라고 부른다(turn). 11번 손님은 10번 손님의 업무가 끝날 때까지 앉아서 전광판만 바라보며 기다린다. 10번이 끝나고 11번이 호출되는 순간, 그 고객이 창구를 점유한다.

- **등장 배경**:
  1. **일반 스핀락의 불공정성**: 기존 스핀락은 락이 해제되는 순간 "먼저 낚아채는 놈이 임자"인 구조다. 운이 없는 스레드는 계속해서 다른 스레드에게 밀려 기아 상태 (Starvation)에 빠질 수 있다.
  2. **성능의 일관성**: 실시간 시스템이나 중요 서버 환경에서는 모든 작업이 일정 시간 내에 완료된다는 보장(Bounded Waiting)이 필수적이다.
  3. **캐시 일관성 문제의 인지**: 비록 티켓 락도 캐시 바운싱 문제가 있지만, 단순 스핀락보다는 "누가 다음에 들어올지"가 명확하여 분석이 용이하다.

- **📢 섹션 요약 비유**: 줄 서기 문화가 없는 정류장에서 먼저 타려고 싸우는 대신, 번호표를 나눠주어 온 순서대로 버스에 타게 하는 문명화된 규칙과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Next Ticket (Ticket)** | 다음에 부여될 번호표 | 스레드 접근 시 fetch-and-add 연산 | Atomic Increment | 번호표 발행기 |
| **Current Turn (Turn)** | 현재 락 소유자의 번호 | 락 해제 시 increment 수행 | 원자적 변수 (Atomic Var) | 전광판 번호 |
| **My Ticket** | 스레드 개별 보유 번호 | 로컬 레지스터/스택에 저장 | Local Variable | 내가 쥔 종이 쪽지 |
| **Spin Loop** | 번호 대기 과정 | `while (my_ticket != turn)` 반복 | 바쁜 대기 (Busy Wait) | 전광판 주시 |

### 티켓 락의 동작 시퀀스

티켓 락은 원자적 명령어인 `fetch-and-add` (또는 `xadd`)를 사용하여 번호표의 유일성을 보장한다.

```text
 ┌──────────────────────────────────────────────────────────────────┐
 │                    티켓 락 (Ticket Lock) 작동 흐름                  │
 ├──────────────────────────────────────────────────────────────────┤
 │                                                                  │
 │   Shared Memory: [Ticket: 12]  [Turn: 10]                        │
 │                                                                  │
 │   Thread A (Request):                                            │
 │     1. my_ticket = fetch_and_add(&Ticket) -> my_ticket=12, Ticket=13 │
 │     2. while (my_ticket != Turn) -> (12 != 10) -> SPIN           │
 │                                                                  │
 │   Thread B (Owner of Turn 10):                                   │
 │     1. Critical Section 작업 완료                                │
 │     2. Turn++ (10 -> 11)                                         │
 │                                                                  │
 │   Thread C (Waiting with Ticket 11):                             │
 │     1. Spin 중 (11 == Turn) 발견 -> 락 획득!                     │
 └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 번호표 발행 과정(`fetch_and_add`)은 원자적으로 일어나므로, 여러 스레드가 동시에 요청해도 겹치지 않는 연속된 번호를 받는다. 대기 중인 모든 스레드는 공유 변수인 `Turn`을 계속해서 읽으며(Spinning) 자신의 번호와 대조한다. 락 소유자가 작업을 마치고 `Turn`을 1 증가시키면, 정확히 다음 번호를 가진 스레드 한 명만 루프를 탈출하게 된다.

+++

### 하드웨어 레벨의 원자성 보장

```c
typedef struct {
    atomic_t ticket; // 다음에 뽑을 번호
    atomic_t turn;   // 지금 들어올 번호
} ticket_lock_t;

void lock(ticket_lock_t *lock) {
    // 번호표를 뽑고 다음 번호를 위해 증가시킴 (원자적 연산)
    int my_ticket = atomic_fetch_and_add(&lock->ticket, 1);
    
    // 내 차례가 올 때까지 대기 (Spin)
    while (my_ticket != atomic_read(&lock->turn)) {
        cpu_relax(); // 전력 소모 감소 및 파이프라인 최적화
    }
}

void unlock(ticket_lock_t *lock) {
    // 다음 번호 손님 호출
    atomic_inc(&lock->turn);
}
```

**[다이어그램 해설]** `cpu_relax()`는 x86의 `PAUSE` 명령어 등으로 변환되어, 스핀 루프 중 발생하는 CPU의 과도한 전력 소모를 줄이고 하이퍼스레딩 환경에서 다른 스레드에게 실행 유닛을 양보하는 역할을 한다. `unlock` 과정은 단순히 `turn`을 증가시키는 것이며, 이 역시 다른 CPU들의 캐시 라인을 무효화(Invalidate)하여 대기 중인 스레드들이 변경된 값을 인지하게 한다.

- **📢 섹션 요약 비유**: 번호표 기계가 고장 나지 않게(원자성) 관리하고, 차례가 된 사람을 큰 소리로 부르는(Unlock) 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 테스트-앤-셋 락 vs 티켓 락

| 비교 항목 | Test-and-Set (TAS) Lock | Ticket Lock |
|:---|:---|:---|
| **공정성 (Fairness)** | 불공정 (랜덤 획득) | **공정 (FIFO)** |
| **기아 상태** | 발생 가능 | **구조적 예방** |
| **대기 오버헤드** | 높음 (전체 스레드 경합) | 높음 (전체 스레드 경합) |
| **캐시 일관성** | 나쁨 (Write-Heavy) | 나쁨 (모든 스레드가 Turn 주시) |
| **구현 복잡도** | 매우 낮음 | 낮음 |

### 한계점: 캐시 라인 핑퐁 (Cache Line Ping-Pong)

티켓 락의 결정적인 단점은 모든 대기 스레드가 하나의 공유 변수(`turn`)를 읽고 있다는 점이다. 락 소유자가 `turn`을 업데이트하면, 대기 중인 수십 개의 CPU 코어에 있는 해당 캐시 라인이 모두 무효화된다. 수많은 코어가 동시에 다시 읽기를 시도하면서 버스 트래픽이 폭증하는데, 이를 **캐시 라인 핑퐁** 현상이라 한다. 이는 코어 수가 많아질수록 성능이 급격히 저하되는 원인이 된다.

- **📢 섹션 요약 비유**: 전광판(Turn)이 바뀔 때마다 수백 명의 대기자가 동시에 고개를 들어 전광판을 확인하느라 목이 아프고(트래픽) 복도가 붐비는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 중소규모 멀티코어 서버 최적화**: 코어 수가 8~16개 정도인 시스템에서는 티켓 락이 매우 효율적이다. 구현이 단순하면서도 공정성을 보장하기 때문이다. 리눅스 커널도 과거 이 범위의 서버들을 위해 티켓 락을 표준으로 채택했었다.
2. **시나리오 — 실시간 운영체제 (RTOS)**: 작업의 마감 시간(Deadline)이 중요한 RTOS에서 락 획득 순서가 보장되지 않으면 최악의 실행 시간(WCET) 산출이 불가능하다. 티켓 락은 대기 순번이 명확하여 예측 가능성을 제공한다.

### 도입 체크리스트
- **코어 수 확장성**: 시스템의 코어 수가 32개 이상으로 늘어날 계획이 있는가? (이 경우 MCS Lock 고려)
- **임계 구역 길이**: 락을 쥐고 있는 시간이 매우 짧은가? (스핀락 계열인 티켓 락은 짧은 작업에 적합)
- **메모리 배리어**: `turn` 증가 시 다른 메모리 연산이 완료되었음을 보장하는 메모리 장벽 (Memory Barrier)이 적절히 배치되었는가?

### 안티패턴
- **티켓 락 내부의 블로킹**: 티켓 락을 쥔 상태에서 I/O 작업을 하거나 잠들면(Sleep), 뒤에 줄 서 있는 모든 스레드가 CPU를 점유한 채 바쁜 대기를 하게 되어 시스템 전체가 멈춘다.

- **📢 섹션 요약 비유**: 번호표를 받고 기다리는 사람들을 두고 창구 직원이 점심 먹으러 나가는 것과 같아서, 뒤에 선 사람들의 시간만 낭비하게 됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 일반 스핀락 (TAS) | 티켓 락 (Ticket) | 개선 효과 |
|:---|:---|:---|:---|
| **기아 발생율** | 높음 (확률적) | **0% (FIFO)** | 시스템 안정성 및 공정성 확보 |
| **대기 시간 편차** | 큼 (예측 불가) | **작음 (일정)** | 실시간성 (Determinism) 향상 |
| **처리량 (Throughput)** | 경합 시 저하 | 경합 시 저하 (유사) | 공정성 대가로 성능 유지 |

### 미래 전망
- **큐잉 스핀락으로의 진화**: 티켓 락의 캐시 핑퐁 문제를 해결하기 위해, 각 스레드가 자신만의 로컬 변수를 바라보며 기다리는 MCS 락 (Mellor-Crummey and Scott Lock)이나 qspinlock이 현대 대규모 서버 CPU의 표준이 되었다.
- **하드웨어 지원**: 최신 CPU 아키텍처는 티켓 락의 로직을 하드웨어적으로 최적화하거나, 락 엘리전 (Lock Elision) 기술을 통해 실제 경합이 없을 경우 락 자체를 생략하는 방향으로 진화하고 있다.

- **📢 섹션 요약 비유**: 티켓 락은 "모두가 평등하게 줄을 서는 사회"를 만든 위대한 첫걸음이었으며, 이제는 "자기 자리에서 편하게 알림을 받는" 더 스마트한 시스템으로 발전하고 있습니다.
