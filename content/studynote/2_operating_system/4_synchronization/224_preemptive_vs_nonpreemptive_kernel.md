# 선점형 커널 (Preemptive Kernel) vs 비선점형 커널

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 커널 모드에서 실행 중인 프로세스가 다른 프로세스에 의해 **CPU 제어권을 강제로 뺏길 수 있는지**에 따라 선점형(Preemptive)과 비선점형(Non-preemptive) 커널로 구분된다.
> 2. **가치**: 선점형 커널은 실시간 응답성이 뛰어나지만 복잡한 동기화 메커니즘이 필수적이며, 비선점형 커널은 단순하고 경쟁 조건(Race Condition) 위험이 적지만 응답성이 떨어진다.
> 3. **융합**: 현대 범용 운영체제(Linux, Windows 등)는 사용자 경험과 실시간성 향상을 위해 고도로 최적화된 선점형 커널 구조를 채택하고 있다.

+++

## Ⅰ. 선점형 vs 비선점형 커널의 개요

### 1. 비선점형 커널 (Non-preemptive Kernel)
- **정의**: 프로세스가 커널 모드에 진입하면, 작업을 마치고 스스로 CPU를 반납(Exit)하거나 I/O 대기를 위해 블록(Block)될 때까지 CPU를 계속 점유하는 방식.
- **특징**: 커널 내부 데이터 구조에 대한 경쟁 조건이 발생하지 않아 설계가 단순하다.

### 2. 선점형 커널 (Preemptive Kernel)
- **정의**: 커널 모드에서 실행 중이더라도 더 높은 우선순위의 프로세스가 나타나거나 타임 슬라이스가 만료되면 CPU 제어권을 즉시 뺏길 수 있는 방식.
- **특징**: 빠른 응답성(Responsiveness)을 제공하지만, 공유 데이터 보호를 위한 정교한 동기화(Locking)가 필수적이다.

### 3. 💡 비유: '회의실 사용 규칙'
- **비선점형**: 회의 중인 팀이 "저희 다 끝났습니다"라고 말하고 나올 때까지 밖에서 무조건 기다려야 하는 규칙이다.
- **선점형**: 사장님이 긴급 회의를 위해 들어오시면, 기존에 회의하던 팀은 즉시 짐을 싸서 나오고 나중에 다시 회의를 이어가는 규칙과 같다.

- **📢 섹션 요약 비유**: 자기 할 일이 끝날 때까지 기다려주는 예의 바른 사회(비선점)와, 급한 일이 생기면 바로 끼어드는 효율 중심 사회(선점)의 차이입니다.

+++

## Ⅱ. 커널 모드 동기화 메커니즘 (Deep Dive)

선점형 커널이 커널 데이터를 보호하는 방식을 분석한다.

### 1. 선점형 커널의 도전 과제
- 커널 모드에서 프로세스 A가 리스트 구조를 수정하던 중 선점되어 프로세스 B가 같은 리스트를 읽으려 할 때, 데이터 불일치나 커널 패닉이 발생할 수 있다.

### 2. ASCII 구조 및 제어 흐름
```text
    [ Process A (Kernel Mode) ]      [ Interrupt / Scheduler ]      [ Process B (Kernel Mode) ]
    ┌─────────────────────────┐      ┌───────────────────────┐      ┌─────────────────────────┐
    │  Modify Shared Data     │      │                       │      │                         │
    │  (Lock Acquired)        │      │  High Priority Task   │      │                         │
    │          ───────────────┼─────▶│      Arrives!         │      │                         │
    │          ◀──────────────┼─────┤  (Preemption Attempt) │      │                         │
    │  (Wait for Lock release)│      │                       │──▶   │  Wants Shared Data...   │
    │          ───────────────┼─────▶│                       │◀──   │  (Blocked until Lock)   │
    └─────────────────────────┘      └───────────────────────┘      └─────────────────────────┘
```

### 3. 주요 해결 기법
- **Spinlocks**: 아주 짧은 시간 동안 락을 기다리며 루프를 도는 방식 (멀티코어 환경).
- **Mutex/Semaphore**: 대기 큐에 프로세스를 넣고 잠들게 하는 방식.
- **Interrupt Disabling**: 특정 구간 동안 인터럽트를 아예 막아 선점 자체를 차단하는 방식.

- **📢 섹션 요약 비유**: 중간에 불려 나갈 것에 대비해, 중요한 서류(데이터)는 항상 금고(Lock)에 넣고 작업하는 꼼꼼함이 필요합니다.

+++

## Ⅲ. 두 커널 방식의 상세 비교

| 비교 항목 | 비선점형 커널 | 선점형 커널 |
|:---|:---|:---|
| **응답성** | 낮음 (커널 작업 완료까지 대기) | **높음 (즉각적인 문맥 교환)** |
| **복잡도** | 낮음 (단순 설계) | **높음 (동기화 이슈 해결 필요)** |
| **경쟁 조건** | 거의 없음 (커널 내 안전) | **항상 존재 (철저한 관리 필요)** |
| **실시간성** | 부적합 | **적합 (RTOS의 필수 조건)** |
| **대표 사례** | Windows 95, 구버전 Unix | **Modern Linux, Windows 10/11** |

- **📢 섹션 요약 비유**: 단순하지만 답답한 완행열차(비선점)와, 복잡하지만 빠른 고속열차(선점)의 차이입니다.

+++

## Ⅳ. 실무 적용 및 설계 인사이트 (Strategy)

### 1. 실무 시나리오: 임베디드 실시간 제어 (RTOS)
- **상황**: 자동차 에어백 제어 시스템.
- **판단**: 사고 감지 시 커널이 다른 작업을 하고 있다고 해서 기다려줄 여유가 없다.
- **해결**: 반드시 **선점형 커널**을 사용해야 하며, '최대 선점 지연 시간(Maximum Preemption Latency)'을 보장할 수 있도록 커널을 설계해야 한다.

### 2. 기술사적 인사이트: Preemption Count
- 리눅스 커널 등에서는 `preempt_count`라는 변수를 사용한다. 프로세스가 락을 획득하면 이 값을 증가시키고, 0이 아닐 때는 선점을 일시적으로 금지한다. 이를 통해 선점형 커널의 장점을 살리면서도, 임계 구역 내에서의 안전성을 하이브리드 방식으로 보장한다.

- **📢 섹션 요약 비유**: 응급환자가 오면 일반 진료를 멈춰야 하지만, 수술 중인 의사를 억지로 끌어낼 순 없듯(Locking), 적절한 예외 상황 관리가 핵심입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **사용자 경험 향상**: 마우스 움직임이나 키보드 입력에 대한 지연 시간(Latency) 최소화.
- **시스템 처리 효율**: 우선순위가 높은 작업을 먼저 처리함으로써 전체 시스템의 가치 극대화.

### 2. 미래 전망
코어 수가 비약적으로 증가함에 따라, 선점형 커널에서의 락 경합(Lock Contention) 문제가 심화되고 있다. 이를 해결하기 위해 **Read-Copy-Update (RCU)**와 같이 읽기 작업에는 락을 걸지 않는 혁신적인 동기화 기법들이 선점형 커널에 적극 도입되고 있으며, 커널의 크기를 최소화하여 선점 가능 영역을 넓히는 마이크로커널(Microkernel) 구조도 다시 주목받고 있다.

- **📢 섹션 요약 비유**: 찰나의 순간을 다투는 현대 컴퓨팅에서, 선점형 커널은 멈추지 않는 엔진을 만드는 고도의 설계 기술입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **임계 구역 (Critical Section)**: 선점형 커널에서 철저히 보호해야 할 영역.
- **문맥 교환 (Context Switch)**: 선점이 일어날 때 발생하는 실제 오퍼레이션.
- **실시간 스케줄링**: 선점형 커널의 존재 이유 중 하나.
