# 큐잉 스핀락 (MCS Lock / qspinlock)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: MCS 락 (Mellor-Crummey and Scott Lock)은 대기 중인 각 스레드가 자신만의 로컬 메모리 위치(노드)에서 스핀하도록 설계하여, 공유 변수 수정 시 발생하는 캐시 일관성 트래픽을 최소화한 큐 기반 스핀락이다.
> 2. **가치**: 코어 수가 수십~수백 개에 달하는 NUMA (Non-Uniform Memory Access) 아키텍처에서 락 경합 시 발생하는 버스 병목을 제거하여 압도적인 확장성(Scalability)을 제공한다.
> 3. **융합**: 현대 리눅스 커널의 `qspinlock` 아키텍처의 근간이 되며, 분산 시스템의 락 매니저 설계 및 고성능 유저 레벨 동시성 라이브러리의 핵심 알고리즘으로 자리 잡고 있다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: MCS 락은 1991년 John Mellor-Crummey와 Michael Scott이 제안한 알고리즘이다. 락을 기다리는 스레드들을 논리적인 연결 리스트(Queue)로 세우고, 각 스레드는 앞사람이 자신의 로컬 변수(`waiting` 플래그)를 바꿔줄 때까지 자기 집 앞(로컬 캐시)만 바라보며 기다린다.

- **💡 비유**: 티켓 락이 "공용 전광판"을 모든 사람이 쳐다보는 방식이라면, MCS 락은 **"릴레이 경주"**와 같다. 대기자들은 각자 자기 구간에 서서 뒷사람을 보지 않고, 오직 앞 주자가 와서 내 손에 바통(신호)을 쥐여줄 때까지만 기다린다. 바통 터치는 오직 두 사람 사이에서만 일어나므로 경기장 전체가 소란스러워질 일이 없다.

- **등장 배경**:
  1. **티켓 락의 한계 (캐시 핑퐁)**: 티켓 락은 코어가 늘어날수록 `turn` 변수 하나를 확인하려는 읽기 요청과 수정 시 발생하는 무효화(Invalidation) 트래픽이 지수적으로 증가하여 성능이 폭락한다.
  2. **NUMA 아키텍처의 확산**: CPU 소켓 간 메모리 접근 비용이 다른 환경에서는 원격 메모리를 계속 읽는(Spinning) 행위가 시스템 전체의 성능 저하를 초래한다.
  3. **확장성 요구**: 클라우드 서버와 같이 수백 개의 스레드가 동시에 돌아가는 환경에서 락 경합이 성능의 발목을 잡지 않아야 한다.

- **📢 섹션 요약 비유**: 수백 명이 한꺼번에 전광판을 보는 대신, 각자 자기 휴대폰으로 "다음은 당신 차례입니다"라는 개인 메시지를 받는 알림 시스템으로 진화한 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **MCS 노드 (Node)** | 스레드별 대기 정보를 담은 객체 | `next` 포인터와 `waiting` 플래그 보유 | 구조체 (Struct) | 대기 중인 선수 |
| **Tail 포인터** | 큐의 마지막 위치를 가리킴 | 새로운 스레드 진입 시 원자적 교체 | CAS (Compare-And-Swap) | 줄의 끝부분 |
| **Waiting 플래그** | 스레드가 스핀할 로컬 변수 | 초기값 True, 앞사람이 False로 변경 | 로컬 캐시 스핀 | "기다려" 표지판 |
| **Next 포인터** | 뒷사람의 노드를 가리킴 | 내가 끝난 후 깨워줄 대상 식별 | 포인터 연결 | 다음 주자 지목 |

### MCS 락의 동작 시퀀스

MCS 락은 연결 리스트 구조를 사용하여 대기 주체들을 정렬한다.

```text
 ┌──────────────────────────────────────────────────────────────────┐
 │                    MCS 락 (MCS Lock) 작동 아키텍처                  │
 ├──────────────────────────────────────────────────────────────────┤
 │                                                                  │
 │   Global Tail ──────┐                                            │
 │                     ▼                                            │
 │   [Node A: Owner] ──▶ [Node B: Wait] ──▶ [Node C: Wait]          │
 │       (Spinning)         (Spinning)         (Spinning)           │
 │       (on next)        (on B.waiting)     (on C.waiting)         │
 │                                                                  │
 │   1. Acquire:                                                    │
 │      - 내 노드를 큐의 끝(Tail)에 붙임 (원자적 연산)               │
 │      - 앞사람이 있다면, 내 노드의 waiting을 True로 하고 스핀       │
 │   2. Release:                                                    │
 │      - 내 노드의 next가 가리키는 뒷사람의 waiting을 False로 변경   │
 │      - 뒷사람은 자신의 로컬 변수가 변했음을 보고 락 획득           │
 └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 가장 중요한 차이점은 스레드 B는 `Node B.waiting`만 보고, 스레드 C는 `Node C.waiting`만 본다는 점이다. 스레드 A가 락을 해제할 때 오직 `Node B.waiting` 하나만 수정한다. 이 수정은 스레드 C의 캐시에는 아무런 영향을 주지 않는다. 따라서 코어 수가 아무리 많아져도 락 해제 시 발생하는 캐시 무효화는 오직 다음 차례인 스레드 하나에만 국한된다.

+++

### 원자적 연산을 이용한 구현 (Pseudo Code)

```c
void lock(mcs_lock_t *tail, mcs_node_t *my_node) {
    my_node->next = NULL;
    my_node->waiting = true;
    
    // 큐의 마지막 노드를 내 노드로 교체하고 이전 마지막 노드를 반환
    mcs_node_t *prev = atomic_exchange(tail, my_node);
    
    if (prev != NULL) {
        prev->next = my_node; // 앞사람과 연결
        while (my_node->waiting) {
            cpu_relax(); // 내 로컬 변수만 보며 대기
        }
    }
}

void unlock(mcs_lock_t *tail, mcs_node_t *my_node) {
    if (my_node->next == NULL) {
        // 내가 마지막이면 Tail을 NULL로 원자적 복구
        if (atomic_cas(tail, my_node, NULL)) return;
        // 실패했다면 그 사이 누군가 들어온 것이므로 기다려줌
        while (my_node->next == NULL);
    }
    // 뒷사람의 대기 상태를 풀어줌
    my_node->next->waiting = false;
}
```

**[다이어그램 해설]** `atomic_exchange`와 `atomic_cas`를 통해 큐에 안전하게 진입하고 탈출한다. `unlock` 과정에서 뒷사람이 들어오는 찰나의 순간을 처리하기 위한 루프가 존재하지만, 이는 매우 짧은 시간이다. 핵심은 `my_node->next->waiting = false` 연산이 단 한 번의 캐시 미스만을 유발한다는 경제성이다.

- **📢 섹션 요약 비유**: 뒷사람의 어깨를 툭 치며 "자, 이제 네 차례야"라고 알려주는 릴레이 경주 방식과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 티켓 락 vs MCS 락

| 비교 항목 | 티켓 락 (Ticket Lock) | MCS 락 (MCS Lock) |
|:---|:---|:---|
| **스핀 위치** | 하나의 공유 변수 (`turn`) | 스레드별 **로컬 변수** (`waiting`) |
| **캐시 트래픽** | 코어 수에 비례하여 폭증 ($O(N)$) | **코어 수와 무관하게 일정 ($O(1)$)** |
| **메모리 요구량** | 고정 (정수 2개) | 스레드당 노드 1개 필요 (동적/스택) |
| **최적 환경** | 소규모 멀티코어 (UP/SMP) | 대규모 멀티코어 (NUMA) |
| **복잡도** | 낮음 | 높음 (연결 리스트 관리) |

### qspinlock (Queued Spinlock)의 등장

MCS 락은 스레드마다 노드 객체를 전달해야 하므로 API 사용이 번거롭고 메모리 오버헤드가 있다는 단점이 있다. 이를 해결하기 위해 리눅스 커널은 **qspinlock**을 도입했다. qspinlock은 락 객체의 크기를 티켓 락처럼 작게(4바이트) 유지하면서도, 내부적으로는 CPU별로 미리 할당된 MCS 노드 배열을 사용하여 MCS 락의 성능적 이점을 취하는 하이브리드 방식이다.

- **📢 섹션 요약 비유**: MCS 락이 개인 비서를 고용하는 고급 방식이라면, qspinlock은 공용 비서실(CPU별 노드)을 효율적으로 운영하는 방식과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 128코어 서버의 스케줄러 락 경합**: 수많은 코어가 런큐(Run-queue)에 접근하려 할 때 일반 락을 사용하면 버스 트래픽 때문에 시스템 전체가 멎을 수 있다. 이때 MCS 기반 큐잉 락을 사용하면 경합이 심해도 다른 코어들의 실행 속도가 저하되지 않는다.
2. **시나리오 — 고성능 유저 레벨 스레드 라이브러리**: 커널의 도움 없이 유저 모드에서만 동작하는 런타임 환경(예: Go, Rust)에서 자원 경합을 처리할 때, 확장성을 위해 MCS 락 아키텍처를 직접 구현하여 사용한다.

### 도입 체크리스트
- **확장성 목표**: 향후 하드웨어 업그레이드 시 코어 수가 대폭 늘어날 가능성이 있는가?
- **메모리 제약**: 스레드별로 노드 공간을 할당할 수 있는 메모리 여유가 있는가?
- **NUMA 인지**: 메모리 할당 시 현재 코어와 물리적으로 가까운 노드 공간을 사용하도록 설계했는가?

### 안티패턴
- **재진입 시도**: MCS 락은 기본적으로 재진입(Reentrancy)을 지원하지 않는다. 동일 스레드가 락을 쥔 상태에서 다시 요청하면 큐에 자기 자신을 이어 붙이려다 데드락에 빠진다.

- **📢 섹션 요약 비유**: 복잡한 미로(대규모 시스템)에서 모든 사람이 입구만 쳐다보는 것이 아니라, 각자 자기 앞에 놓인 신호등만 보고 움직이는 질서가 필요한 시점입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (Ticket Lock) | 도입 후 (MCS Lock) | 개선 효과 |
|:---|:---|:---|:---|
| **버스 트래픽** | 지수적 증가 | **상수 (Constant) 유지** | 대규모 시스템 안정성 확보 |
| **락 획득 시간** | 경합 시 예측 불가 | **선형적/일정함** | 시스템 응답 시간 (Latency) 안정화 |
| **CPU 확장성** | 32코어 이상에서 성능 저하 | **1024코어 이상 확장 가능** | 하드웨어 투자 효율 극대화 |

### 미래 전망
- **비휘발성 메모리 (NVM) 최적화**: 쓰기 횟수에 민감한 차세대 메모리 환경에서, 쓰기 발생 위치를 로컬로 격리하는 MCS 락의 특성은 하드웨어 수명 연장에도 기여할 수 있다.
- **분산 환경으로의 확장**: 네트워크 지연이 큰 분산 락 관리자(DLM) 설계 시, 중앙 집중형이 아닌 큐 기반의 노티피케이션 방식인 MCS 철학이 널리 채택될 것이다.

- **📢 섹션 요약 비유**: 모든 사람이 똑같은 정보를 보려 애쓰는 시대에서, 필요한 사람에게 정확히 필요한 시점에 정보를 전달하는 "개인화된 동기화"로의 패러다임 전환입니다.
