# 경쟁 조건 (Race Condition)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 경쟁 조건 (Race Condition)은 둘 이상의 프로세스나 스레드가 **공유 자원(Shared Resource)**에 동시에 접근하여 조작할 때, 접근 순서에 따라 실행 결과가 달라지는 현상을 말한다.
> 2. **가치**: 데이터의 무결성(Integrity)과 일관성(Consistency)을 파괴하는 주범이며, 간헐적으로 발생하기 때문에 디버깅이 매우 어려운 소프트웨어 결함 중 하나다.
> 3. **융합**: 이를 방지하기 위해 임계 구역(Critical Section) 설정 및 상호 배제(Mutual Exclusion)와 같은 동기화(Synchronization) 메커니즘이 필수적으로 요구된다.

+++

## Ⅰ. 경쟁 조건의 개요

### 1. 정의
- 여러 프로세스/스레드가 공유 데이터에 동시에 접근하여 쓰기(Write) 작업을 수행할 때, 최종 결과가 마지막에 작업을 마친 프로세스의 영향을 받는 불안정한 상태이다.

### 2. 발생 원인
- **공유 자원 존재**: 전역 변수, 파일, 데이터베이스 등 여러 주체가 함께 사용하는 자원이 있을 때.
- **비원자적 연산**: 한 줄의 코드(`count++`)가 실제 CPU 수준에서는 여러 단계(Read-Modify-Write)로 나뉘어 실행될 때.
- **스케줄링의 불확실성**: 프로세스가 실행 도중 언제든 인터럽트되어 제어권을 뺏길 수 있을 때.

### 3. 💡 비유: '공동 은행 계좌의 동시 인출'
- 부부가 하나의 통장에서 동시에 10만 원을 인출한다고 가정하자.
- 남편이 잔액 100만 원을 확인(Read)하고 인출 버튼을 누르기 직전, 아내가 100만 원을 확인하고 인출해버리면, 남편의 화면에는 여전히 100만 원이 찍혀 있어 실제 잔액보다 더 많은 돈이 인출될 수 있는 위험한 상황과 같다.

- **📢 섹션 요약 비유**: 결승선에 누가 먼저 들어오느냐에 따라 결과가 뒤바뀌는 예측 불가능한 달리기 경주와 같습니다.

+++

## Ⅱ. 경쟁 조건의 메커니즘 (Deep Dive)

CPU 수준에서 `count++` 연산이 어떻게 경쟁 조건을 유발하는지 분석한다.

### 1. 비원자적 실행 단계 (Read-Modify-Write)
1. **Load**: 메모리의 `count` 값을 CPU 레지스터로 읽어옴.
2. **Add**: 레지스터 값을 1 증가시킴.
3. **Store**: 증가된 값을 다시 메모리의 `count`에 저장함.

### 2. ASCII 시나리오 다이어그램
```text
    Time    Process A (count++)          Process B (count++)          Memory (count)
     │      ───────────────────          ───────────────────          ──────────────
     t1     1. Load (count=5)                                              5
     t2     2. Add  (reg=6)              1. Load (count=5)                 5
     t3                                  2. Add  (reg=6)                 5
     t4     3. Store (count=6)                                             6
     t5                                  3. Store (count=6)                6 (Error!)
     ▼
    [결과] 두 번 증가해야 하는데, 최종 값은 6이 됨 (데이터 손실 발생)
```

### 3. 커널 수준의 경쟁 조건
- **인터럽트 처리**: 커널 변수를 수정하던 중 인터럽트가 발생하여 핸들러가 같은 변수를 건드릴 때.
- **시스템 콜**: 두 프로세스가 서로 다른 시스템 콜을 통해 동일한 커널 데이터 구조를 변경할 때.

- **📢 섹션 요약 비유**: 요리사 둘이 레시피 메모를 동시에 고치다가, 한 명의 수정 사항이 덮어씌워져 요리를 망치는 과정입니다.

## Ⅲ. 경쟁 조건의 유형 및 영향

### 1. 주요 유형
| 유형 | 설명 |
|:---|:---|
| **Check-then-Act** | 조건 확인 후 행동하기 직전에 데이터가 바뀌어버리는 경우 (Null Check 후 접근 등) |
| **Read-Modify-Write** | 값을 읽고 수정해서 저장하는 사이 다른 프로세스가 개입하는 경우 |

### 2. 시스템에 미치는 영향
- **데이터 불일치**: DB 잔고 오류, 재고 수량 미일치 등.
- **시스템 크래시**: 잘못된 포인터 참조나 널 참조로 인한 커널 패닉.
- **보안 취약점**: 권한 확인 로직을 우회하는 Race Condition 공격 (TOCTOU: Time-of-Check to Time-of-Use).

- **📢 섹션 요약 비유**: 신호등이 초록불인 것을 확인하고 발을 뗐는데, 그 찰나에 신호가 바뀌어 사고가 나는 상황과 같습니다.

## Ⅳ. 실무 적용 및 해결 전략 (Strategy)

### 1. 실무 시나리오: 웹 서버의 조회수 카운터
- **상황**: 초당 수만 건의 요청이 들어오는 뉴스 사이트의 기사 조회수.
- **판단**: 단순히 변수를 증가시키면 경쟁 조건으로 인해 실제보다 적게 집계된다.
- **해결**: Redis의 `INCR` 명령과 같은 **원자적 연산(Atomic Operation)**을 사용하거나, DB 수준에서 `UPDATE SET count = count + 1`과 같이 락(Lock)을 활용한 동기화를 적용해야 한다.

### 2. 기술사적 인사이트: 디버깅의 어려움 (Heisenbug)
- 경쟁 조건은 재현이 어렵기로 유명하다. 특정 타이밍에만 발생하기 때문에 개발 장비에서는 괜찮다가 운영 서버에서만 터지곤 한다. 이를 해결하기 위해 **정적 분석 도구(Static Analysis)**나 **Thread Sanitizer** 같은 동기화 오류 검출 도구를 도입하여 설계 단계부터 방어해야 한다.

- **📢 섹션 요약 비유**: 가끔씩만 오작동하는 기계는 고치기 힘들 듯, 근본적인 설계(동기화)를 통해 고장을 원천 차단해야 합니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **데이터 무결성 보장**: 시스템 전체의 신뢰도를 높이고 금전적/기능적 손실 방지.
- **시스템 안정성 확보**: 예측 불가능한 오류를 제거하여 업타임(Uptime) 증대.

### 2. 미래 전망
멀티코어를 넘어 수백 개의 코어를 사용하는 매니코어(Many-core) 시대에는 경쟁 조건 방지를 위한 전통적인 락(Lock) 방식이 성능 병목을 일으킨다. 따라서 락을 사용하지 않는 **Lock-free 알고리즘**이나 **불변 객체(Immutable Object)** 지향 프로그래밍이 더욱 중요해질 것이며, 하드웨어 차원의 트랜잭션 메모리(Transactional Memory) 지원도 확대될 전망이다.

- **📢 섹션 요약 비유**: 복잡한 도로에서 신호등과 차선이 없다면 대혼란이 오듯, 경쟁 조건 통제는 소프트웨어 세계의 질서를 유지하는 기초 공사입니다.

+++

### 📌 관련 개념 맵 (Knowledge Graph)
- **임계 구역 (Critical Section)**: 경쟁 조건이 발생하는 코드 영역.
- **상호 배제 (Mutual Exclusion)**: 경쟁 조건을 막는 가장 기본적인 원칙.
- **원자적 연산 (Atomic Operation)**: 중단되지 않는 최소 단위 연산.

### 👶 어린이를 위한 3줄 비유 설명
1. 경쟁 조건은 두 친구가 도화지에 **'동시에 그림을 그리려고 해서 그림이 엉망이 되는 것'**과 같아요.
2. 한 명이 사과를 그리는데 다른 친구가 그 위에 포도를 그려버리면, 사과도 포도도 아니게 되죠.
3. 그래서 "내가 다 그릴 때까지 기다려!"라고 순서를 정하는 약속이 꼭 필요하답니다.
