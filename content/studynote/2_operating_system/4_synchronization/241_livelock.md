# 라이브락 (Livelock)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 라이브락은 두 개 이상의 프로세스가 서로의 상태 변화에 반응하여 자신의 상태를 계속 변경하지만, 정작 실제 작업(Progress)은 전혀 진전되지 않는 동시성 결함 상태다.
> 2. **가치**: 데드락(Deadlock)이 '멈춤'이라면 라이브락은 '무한 루프'의 일종으로, CPU 자원을 소모하면서도 아무런 결과물을 내지 못한다는 점에서 시스템 자원에 더 치명적일 수 있다.
> 3. **융합**: 네트워크 충돌 회피 알고리즘(Ethernet CSMA/CD)이나 복구 시퀀스 설계 시 발생할 수 있으며, 지수 백오프(Exponential Backoff)와 같은 무작위성 도입이 주요 해결책이다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 데드락은 프로세스들이 자원을 기다리며 완전히 멈춰버린 상태인 반면, 라이브락은 프로세스들이 실행은 계속하고 있으나(Active) 서로 양보하거나 상태를 맞추느라 정작 중요한 일은 수행하지 못하는 상태다. 마치 예의 바른 두 사람이 좁은 복도에서 마주쳐 동시에 같은 방향으로 비켜주는 동작을 무한히 반복하는 것과 같다.

- **💡 비유**: 라이브락은 "복도에서 마주친 두 사람이 서로 길을 비켜주기 위해 왼쪽으로 갔다가, 다시 오른쪽으로 갔다가 하는 행동을 끝없이 반복하는 상황"과 같다. 두 사람 모두 움직이고는 있지만, 복도를 통과(Progress)하지는 못한다.

- **📢 섹션 요약 비유**: 마치 두 대의 로봇이 충돌을 피하기 위해 똑같은 회피 알고리즘을 수행하며 서로의 움직임에 맞춰 똑같이 옆으로만 움직이는 '동기화된 춤'과 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 라이브락의 발생 메커니즘

라이브락은 주로 '실패 시 재시도' 로직이 너무 정교하거나 대칭적일 때 발생한다.

```text
  [ Process A ]                       [ Process B ]
       |                                     |
  1. Try acquire Resource 1           1. Try acquire Resource 2
       |                                     |
  2. See Resource 2 is busy           2. See Resource 1 is busy
       |                                     |
  3. "Let me release R1 to avoid      3. "Let me release R2 to avoid
      Deadlock for B"                     Deadlock for A"
       |                                     |
  4. Wait a moment                    4. Wait a moment
       |                                     |
  5. Repeat step 1                    5. Repeat step 1
```

**[다이어그램 해설]** 프로세스 A와 B는 데드락을 피하기 위해 자원을 선점하지 않고 상대방에게 양보하는 로직을 가지고 있다. 하지만 두 프로세스가 **정확히 같은 타이밍**에 양보하고 다시 요청하기를 반복하면, 시스템은 어떤 작업도 끝내지 못하고 양보 로직만 수행하게 된다.

### 코드 레벨의 라이브락 예시

```c
void process_A() {
    while (1) {
        get_lock(L1);
        if (try_lock(L2)) {
            // Success!
            do_work();
            break;
        }
        // L2 실패 시 L1도 내려놓고 다시 시도 (Deadlock 회피 로직)
        release_lock(L1); 
        // 여기서 정확히 B와 같은 시간을 쉰다면 Livelock 발생
    }
}
```

- **📢 섹션 요약 비유**: 마치 두 전화기가 통화 연결이 안 되자마자 동시에 다시 전화를 걸어 서로 '통화 중' 신호만 무한히 듣게 되는 상황과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 데드락 (Deadlock) vs 라이브락 (Livelock)

| 비교 항목 | 데드락 (Deadlock) | 라이브락 (Livelock) |
|:---|:---|:---|
| **프로세스 상태** | Waiting (Blocked) | Active (Running) |
| **CPU 사용** | 거의 없음 (잠들어 있음) | 매우 높음 (루프 실행 중) |
| **증상** | 시스템이 얼어붙음 (Freeze) | 시스템은 바쁘지만 응답이 없음 |
| **해결책** | 자원 할당 순서 강제, 타임아웃 | **무작위 대기 시간(Random Backoff)** 도입 |

- **📢 섹션 요약 비유**: 데드락은 엔진이 꺼진 차가 길을 막고 있는 것이고, 라이브락은 엔진이 켜진 차 두 대가 좁은 길에서 서로 비켜주려다 계속 엉키는 것입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 네트워크 충돌 (CSMA/CD)
- **상황**: 이더넷 네트워크에서 두 노드가 동시에 데이터를 전송하여 충돌 발생. 두 노드가 충돌을 감지하고 즉시 다시 전송하면 또 충돌이 발생함.
- **해결책**: **지수 백오프 (Exponential Backoff)** 알고리즘을 사용한다. 충돌 시 대기하는 시간을 무작위로 설정하고, 충돌이 반복될수록 대기 시간의 범위를 2배씩 늘려 두 노드가 다시 충돌할 확률을 수학적으로 낮춘다.

### 도입 체크리스트
- **Randomness**: 재시도 로직에 무작위성(Jitter)이 포함되어 있는가?
- **Symmetry Breaking**: 모든 프로세스가 동일한 가중치와 타이밍으로 동작하여 대칭성을 형성하지 않는가?

- **📢 섹션 요약 비유**: 가위바위보에서 계속 비길 때, 한 사람이 아주 미세하게 늦게 내거나 다른 패턴을 섞어야 승부가 나는 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 라이브락을 방지하는 설계는 시스템의 가용성(Availability)을 높이고, 불필요한 CPU 자원 낭비를 막아 전체 성능을 최적화한다.
- **미래 전망**: 복잡한 분산 마이크로서비스 아키텍처에서 서비스 간의 재시도 정책(Retry Policy)이 라이브락을 유발하는 경우가 많아지고 있으며, 이를 감지하고 해결하는 서비스 메시(Service Mesh) 기술이 중요해지고 있다.

- **📢 섹션 요약 비유**: 무작정 열심히 움직이는 것보다, 가끔은 멈추고 타이밍을 조절하는 것이 목적지에 더 빨리 도착하는 비결입니다.
