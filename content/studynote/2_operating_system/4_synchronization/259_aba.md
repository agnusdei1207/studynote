# 259. 락 에스컬레이션 (Lock Escalation)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 세밀한 락에서 굵은 락으로 자동 전환
> 2. **가치**: 높은 경합 시 오버헤드 감소
> 3. **융합**: 락 세분성, 락 스트라이핑, 적응적 락과 연관

---

## Ⅰ. 개요

### 개념 정의

락 에스컬레이션(Lock Escalation)은 **세밀한 락(행 락)이 너무 많이 사용되면, 데이터베이스가 자동으로 굵은 락(테이블 락)으로 전환하는 최적화 기법**이다. 오버헤드를 줄이지만 동시성은 감소한다.

### 💡 비유: 도로 공사
락 에스컬레이션은 **도로 공사**와 같다. 여러 구간에서 공사(세밀한 락)를 하면 전체 도로를 막는 게 낫다(굵은 락). 관리는 쉽지만 교통은 막힌다.

### 락 에스컬레이션 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                락 에스컬레이션 구조                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【발생 시나리오】                                                      │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  초기: 행 락 사용                                                   │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │  SELECT * FROM table WHERE id = 1;  -- 행 락               │ │ │
│  │  │  SELECT * FROM table WHERE id = 2;  -- 행 락               │ │ │
│  │  │  SELECT * FROM table WHERE id = 3;  -- 행 락               │ │ │
│  │  │  ...                                                       │ │ │
│  │  │  SELECT * FROM table WHERE id = N;  -- 행 락                │ │ │
│  │  └─────────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  │  임계값 도달: 에스컬레이션 발생                                        │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │  // 임계값 (예: 5000행) 초과                                  │ │ │
│  │  │  → 행 락 해제                                                │ │ │
│  │  │  → 테이블 락 획득                                            │ │ │
│  │  │  → 전체 테이블 잠금                                           │ │ │
│  │  └─────────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【트리거 조건】                                                       │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 락 임계값 초과                                                  │ │
│  │     • SQL Server: 기본 5000행                                      │ │
│  │     • Oracle: 메모리/시간 기반                                       │ │
│  │                                                             │ │
│  │  2. 메모리 부족                                                     │ │
│  │     • 락 관리자 메모리 한계                                          │ │
│  │     • 임계값 초과 시 테이블 락으로 전환                               │ │
│  │                                                             │ │
│  │  3. 인덱스 없는 스캔                                                 │ │
│  │     • 전체 스캔 필요 시                                              │ │
│  │     • 행 락보다 테이블 락이 효율적                                     │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【장단점】                                                            │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  장점:                                                              │ │
│  │  • 락 관리 오버헤드 감소                                             │ │
│  │  • 메모리 사용 감소                                                  │ │
│  │  • 교착상태 가능성 감소                                               │ │
│  │                                                             │ │
│  │  단점:                                                              │ │
│  │  • 동시성 급감                                                       │ │
│  │  • 다른 쿼리 차단                                                    │ │
│  │  • 예기치 않은 동작                                                   │ │
│  │  • 성능 저하                                                         │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                락 에스컬레이션 상세                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【DBMS별 동작】                                                       │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  SQL Server:                                                       │ │
│  │  • 락 에스컬레이션 명시적 설정 가능                                  │ │
│  │  • sp_configure 'locks' 옵션                                       │ │
│  │  • MAXDOP 힌트로 제어                                               │ │
│  │                                                             │ │
│  │  Oracle:                                                           │ │
│  │  • 자동 에스컬레이션                                                 │ │
│  │  • ENQUEUE_RESOURCES 파라미터                                       │ │
│  │  • 힌트로 제어 가능                                                  │ │
│  │                                                             │ │
│  │  MySQL (InnoDB):                                                   │ │
│  │  • 행 락 → 테이블 락 전환                                            │ │
│  │  • innodb_lock_scale 파라미터                                       │ │
│  │                                                             │ │
│  │  PostgreSQL:                                                       │ │
│  │  • MVCC로 락 에스컬레이션 최소화                                     │ │
│  │  • 명시적 LOCK TABLE 사용                                            │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【에스컬레이션 단계】                                                  │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  행 락 (Row Lock)                                                   │ │
│  │       ↓                                                             │ │
│  │  페이지 락 (Page Lock)                                               │ │
│  │       ↓                                                             │ │
│  │  파티션 락 (Partition Lock)                                         │ │
│  │       ↓                                                             │ │
│  │  테이블 락 (Table Lock)                                              │ │
│  │       ↓                                                             │ │
│  │  데이터베이스 락 (Database Lock)                                     │ │
│  │                                                             │ │
│  │  일반적으로 행 → 페이지 → 테이블 순서                                   │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【방지 방법】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 적절한 인덱스 생성                                               │ │
│  │     → 범위 스캔 대신 인덱스 탐색                                       │ │
│  │                                                             │ │
│  │  2. 배치 크기 제한                                                   │ │
│  │     → 한 번에 처리하는 행 수 줄이기                                     │ │
│  │                                                             │ │
│  │  3. 명시적 테이블 힌트                                                │ │
│  │     → WITH (ROWLOCK) 등                                             │ │
│  │                                                             │ │
│  │  4. 쿼리 최적화                                                       │ │
│  │     → 불필요한 행 읽기 회피                                            │ │
│  │                                                             │ │
│  │  5. 적절한 트랜잭션 크기                                               │ │
│  │     → 긴 트랜잭션 분할                                                │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【SQL Server 힌트로 제어】                                           │
│  ──────────────────                                                  │
│  -- 행 락 강제 (에스컬레이션 방지)                                      │
│  SELECT * FROM table WITH (ROWLOCK) WHERE id > 1000;               │
│                                                                     │
│  -- 테이블 락 명시 (의도적 에스컬레이션)                                  │
│  SELECT * FROM table WITH (TABLOCKX);                               │
│                                                                     │
│  -- 페이지 락 사용                                                     │
│  SELECT * FROM table WITH (PAGLOCK);                                │
│                                                                     │
│  【Oracle 힌트】                                                      │
│  ──────────────────                                                  │
│  -- 행 락 유지                                                         │
│  SELECT /*+ INDEX(table idx) */ * FROM table WHERE id > 1000;      │
│                                                                     │
│  【MySQL InnoDB】                                                    │
│  ──────────────────                                                  │
│  -- 명시적 테이블 락 (필요한 경우만)                                     │
│  LOCK TABLES table WRITE;                                           │
│  -- 작업...                                                          │
│  UNLOCK TABLES;                                                     │
│                                                                     │
│  【에스컬레이션 모니터링】                                               │
│  ──────────────────                                                  │
│  -- SQL Server:                                                     │
│  SELECT * FROM sys.dm_tran_locks                                    │
│  WHERE resource_type = 'OBJECT'                                     │
│  AND request_mode = 'X';  -- 배타적 테이블 락                           │
│                                                                     │
│  -- Oracle:                                                         │
│  SELECT * FROM v$lock WHERE type = 'TM';  -- 테이블 락                │
│                                                                     │
│  【성능 비교】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  시나리오                  행 락 유지    에스컬레이션                │ │
│  │  ────                      ────────        ────────                │ │
│  │  소량 행 (100)             좋음           좋음                      │ │
│  │  중간 행 (1000)            좋음           좋음                      │ │
│  │  다량 행 (10000)           느림           빠름                      │ │
│  │  전체 스캔                  매우 느림       빠름                      │ │
│  │  동시 업데이트               좋음           차단                      │ │
│  │                                                             │ │
│  │  트레이드오프: 락 오버헤드 vs 동시성                                    │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【적응적 락 전략】                                                     │
│  ──────────────────                                                  │
│  시작: 행 락 → 락 수 모니터링 → 임계값 근접 시 선택:                      │
│  1. 에스컬레이션 허용 (오버헤드 감소, 동시성 감소)                        │
│  2. 행 락 유지 (오버헤드 유지, 동시성 유지)                              │
│  3. 배치 크기 줄이기 (중간 지점)                                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 세밀한 락 → 굵은 락 자동 전환
• 트리거: 락 수 임계값, 메모리 부족, 전체 스캔
• 단계: 행 → 페이지 → 파티션 → 테이블
• 장점: 오버헤드 감소, 메모리 절약
• 단점: 동시성 급감, 차단 발생
• 방지: 인덱스, 배치 제한, 힌트 사용
```
