# 이벤트 객체 (Event Object)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트 객체는 특정 사건이 발생했음을 다른 스레드에게 알리기 위해 사용하는 동기화 신호로, 상태 정보(Signaled/Non-signaled)를 유지하는 동기화 프리미티브다.
> 2. **가치**: 뮤텍스나 세마포어처럼 자원 점유가 목적이 아니라, "준비 완료"나 "작업 끝"과 같은 **상태의 변화를 전파**하는 데 특화되어 있다.
> 3. **융합**: 윈도우 프로그래밍의 비동기 통지 메커니즘의 핵심이며, 생산자-소비자 패턴의 통지부나 스레드 풀의 작업 할당 신호로 널리 사용된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 이벤트 객체는 두 가지 상태를 가진다. 'Signaled'는 신호가 온 상태, 'Non-signaled'는 신호가 없는 상태다. 스레드는 `Wait` 함수를 통해 신호가 올 때까지 기다린다. 이벤트는 다시 **자동 리셋(Auto-reset)**과 **수동 리셋(Manual-reset)**으로 나뉜다.

- **💡 비유**: 이벤트 객체는 "교실의 종소리"와 같다. 수동 리셋 이벤트는 "점심시간 종"과 같아서 한 번 울리면(Signaled) 선생님이 끄기(Reset) 전까지 모든 학생이 밥을 먹으러 나갈 수 있다. 자동 리셋 이벤트는 "출석 부르기"와 같아서 한 명의 대답을 들으면 신호가 즉시 사라져 다음 사람을 기다리게 된다.

- **📢 섹션 요약 비유**: 마치 경주 시작을 알리는 총성(Event)이 울리면 모든 주자가 동시에 출발하는 것과 같은 신호 전달 체계입니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 이벤트 리셋 모드의 차이

| 유형 | 신호 발생 시 동작 (SetEvent) | 리셋 시점 | 비유 |
|:---|:---|:---|:---|
| **Manual-Reset** | 대기 중인 **모든 스레드**를 깨움 | 프로그래머가 직접 `ResetEvent()` 호출 | 댐의 수문 개방 |
| **Auto-Reset** | 대기 중인 스레드 중 **단 하나**만 깨움 | 스레드가 깨어나는 즉시 **커널이 자동 리셋** | 지하철 개찰구 |

### 이벤트 동작 흐름

```text
  [ Producer Thread ]              [ Event Object ]               [ Consumer Thread ]
          |                               |                              |
  1. Prepare Data...              [ Non-signaled ]               1. WaitForSingleObject()
          |                               |                          (Blocked)
  2. SetEvent(hEvent) ──────────▶ [ Signaled ]                           |
                                          |                      2. Wake up!
                                          |                              |
                                  [ (Auto-reset) ]               3. Process Data...
                                  [ Non-signaled ] ◀──────────────┘
```

- **SetEvent()**: 객체를 Signaled 상태로 만든다.
- **ResetEvent()**: 객체를 Non-signaled 상태로 만든다.
- **PulseEvent()**: 신호를 아주 잠깐 주어 현재 대기 중인 스레드들만 깨우고 즉시 리셋한다 (사용 권장 안 됨).

- **📢 섹션 요약 비유**: "자, 다들 시작해!"라고 외치는 것은 수동 리셋이고, "다음 사람 들어오세요"라고 한 명씩 부르는 것은 자동 리셋입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 조건 변수 (Condition Variable) vs 이벤트 객체 (Event)

| 비교 항목 | 조건 변수 (Pthreads/Java) | 이벤트 객체 (Windows) |
|:---|:---|:---|
| **기억력 (State)** | 없음 (신호 발생 시 대기자 없으면 소멸) | 있음 (Set 된 상태가 유지됨) |
| **뮤텍스 연동** | 필수 (락 해제와 수면이 원자적) | 선택 사항 (독립적 객체임) |
| **사용성** | 논리적 조건 결합에 유리 | 단순한 상태 알림에 유리 |

윈도우 이벤트는 '기억력'이 있기 때문에 신호가 먼저 오고 나중에 `Wait`를 해도 즉시 통과된다. 이는 논리적 오류를 줄여주지만, 의도치 않은 중복 통과를 유발할 수도 있다.

- **📢 섹션 요약 비유**: 조건 변수는 그 순간에만 들리는 '외침'이고, 이벤트 객체는 나중에 온 사람도 볼 수 있는 '벽보'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 일괄 작업 시작 (Manual-Reset 활용)
- **상황**: 10개의 워커 스레드가 데이터를 분담해서 처리해야 하는데, 모든 데이터 로딩이 끝난 후에 동시에 시작해야 함.
- **해결책**: Manual-Reset 이벤트를 생성하고, 메인 스레드가 데이터 로딩 완료 후 `SetEvent()`를 한 번만 호출한다. 그러면 대기 중이던 10개의 스레드가 동시에 깨어나 작업을 시작한다.

### 도입 체크리스트
- **Auto vs Manual**: 신호 한 번에 스레드 하나가 가야 하는가, 아니면 모두가 가야 하는가?
- **Missed Signal**: 수동 리셋 사용 시 제때 `ResetEvent()`를 호출하여 다음 신호가 섞이지 않게 관리하고 있는가?

- **📢 섹션 요약 비유**: 출발 신호를 보냈으면 다음 경주를 위해 깃발을 다시 내려놓는(Reset) 과정이 반드시 필요합니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 이벤트 객체는 스레드 간의 복잡한 의존 관계를 신호 기반으로 단순화하여, 비동기 프로그래밍의 직관성을 높여준다.
- **미래 전망**: 윈도우의 차세대 API인 WinRT 등에서도 이벤트 기반의 비동기 처리(Awaitable)는 핵심 패턴으로 유지되고 있으며, 분산 환경의 이벤트 버스(Event Bus) 개념으로 확장되고 있다.

- **📢 섹션 요약 비유**: 신호등이 도로의 소통을 돕듯, 이벤트 객체는 스레드들의 흐름을 조율하는 시스템의 신호등입니다.
