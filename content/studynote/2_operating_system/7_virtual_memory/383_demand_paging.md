# 요구 페이징 (Demand Paging)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구 페이징 (Demand Paging)은 프로세스의 모든 페이지를 시작 시 메모리에 올리지 않고, 실제로 접근이 요구 (demand)될 때만 디스크에서 해당 페이지를 메모리로 적재하는 Lazy Loading 메커니즘이다.
> 2. **가치**: 프로그램의 국소성 원리 (Locality of Reference)를 활용하여 실제 사용되는 페이지만 메모리에 유지하면, 물리 메모리보다 훨씬 큰 가상 주소 공간을 가진 프로세스를 실행할 수 있고, 프로세스 시작 시간도 단축된다.
> 3. **융합**: 요구 페이징은 가상 메모리 (Virtual Memory)의 핵심 구현 기법으로, 페이지 교체 알고리즘 (LRU, FIFO, Clock), 스왑 공간, COW (Copy-on-Write), 선행 페이징 (Prepaging)과 결합하여 현대 OS 메모리 관리의 완성된 구조를 이룬다.

---

## Ⅰ. 개요 및 필요성

1GB 프로그램을 실행할 때 실제로 실행 중에 사용하는 코드와 데이터는 전체의 극히 일부다 (국소성 원리). 모두를 메모리에 올리는 것은 낭비다.

요구 페이징은 "필요한 것만, 필요할 때" 적재하는 원칙이다.

**💡 비유**: 도서관에서 모든 책(페이지)을 책상에 쌓지 않고, 필요한 책이 생기면 서가(디스크)에서 꺼내 오는 것 — 페이지 부재(Page Fault)가 서가에서 책 찾기 요청.

```text
┌──────────────────────────────────────────────────────────────┐
│         요구 페이징 동작 흐름                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  CPU가 논리 주소 접근                                       │
│         │                                                    │
│         ▼                                                    │
│  페이지 테이블 엔트리 확인                                  │
│         │                                                    │
│    ┌────┴──────┐                                             │
│  Valid=1     Valid=0 (미적재)                                │
│  (TLB 히트)   │                                             │
│    │           ▼                                             │
│  물리 접근  Page Fault 인터럽트 발생!                       │
│             │                                               │
│             ▼                                               │
│         OS가 폴트 처리 (6단계):                             │
│         1. 페이지 테이블에서 디스크 위치 확인              │
│         2. 빈 프레임 탐색 (없으면 교체 알고리즘으로 회수)  │
│         3. 디스크 I/O로 페이지 로드 (수백μs~수ms)          │
│         4. 페이지 테이블 갱신 (Valid=1, 프레임 번호 설정)  │
│         5. TLB 갱신                                         │
│         6. 실패한 명령어 재실행                             │
│                                                              │
│  EAT = (1-p) × tm + p × (ts + tm)                          │
│  p: 페이지 부재율, ts: 디스크 서비스 시간 (수 ms)          │
│  p=0.001: EAT ≈ 100ns + 0.001 × 10ms = 10.1ms (100배 느림)│
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: 요구 페이징은 필요한 책만 꺼내는 OnDemand 도서관 — 모든 책을 미리 책상에 올려두지 않아서 책상(메모리)이 훨씬 깔끔합니다.

---

## Ⅱ. 아키텍처 및 핵심 원리

### 페이지 부재율 (p)과 성능

페이지 부재가 너무 자주 발생하면 스래싱(Thrashing) — 디스크 I/O가 CPU 연산보다 많아져 처리량이 붕괴된다. 이를 막는 핵심이 **워킹 셋 (Working Set)** 모델이다.

```text
┌──────────────────────────────────────────────────────────────┐
│         순수 요구 페이징 vs 선행 페이징 (Prepaging)         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  순수 요구 페이징 (Pure Demand Paging):                     │
│  ● 시작 시 메모리에 아무것도 올리지 않음                   │
│  ● 첫 번째 명령어 실행부터 Page Fault 발생                 │
│  ● 초기에 많은 Page Fault → 시작 오버헤드                  │
│                                                              │
│  선행 페이징 (Prepaging):                                   │
│  ● 곧 쓸 것으로 예상되는 페이지를 미리 올림                │
│  ● Page Fault 총 횟수는 감소                                │
│  ● 예측 실패 시 불필요한 I/O 낭비                          │
│  ● Linux: readahead (순차 접근 예측 프리패치)              │
│                                                              │
│  실제: 대부분의 OS는 혼합 방식 사용                         │
└──────────────────────────────────────────────────────────────┘
```

### 유효-무효 비트 (Valid-Invalid Bit)

페이지 테이블 엔트리의 `V` 비트:
- `V=1`: 페이지가 물리 메모리에 적재됨 → 정상 변환
- `V=0`: 미적재 → Page Fault 트랩 발생

**📢 섹션 요약 비유**: V 비트는 도서관 책 대출 현황 태그 — '대출 중(V=0)'이면 서가에서 찾아야(디스크 I/O) 하고, '제자리(V=1)'이면 바로 읽을 수 있어요.

## Ⅲ. 융합 비교 및 다각도 분석

| 항목 | 요구 페이징 | 선행 페이징 | 완전 적재 |
|:---|:---|:---|:---|
| 초기 메모리 | 최소 | 중간 | 최대 |
| 시작 시간 | 빠름 | 중간 | 느림 |
| Page Fault 빈도 | 높음 (초기) | 낮음 | 없음 |
| 메모리 효율 | 최고 | 중간 | 낮음 |

**📢 섹션 요약 비유**: 완전 적재는 여행 전 짐을 모두 꾸리는 것, 요구 페이징은 필요할 때마다 짐을 추가하는 것 — 가볍게 출발하고 필요한 건 나중에 챙깁니다.

## Ⅳ. 실무 적용 및 기술사적 판단

**실무 시나리오**:
1. **대용량 Java 애플리케이션**: JVM 힙이 물리 메모리보다 크게 설정돼도 요구 페이징으로 실제 사용 메모리만 할당. `mmap` 기반 파일 매핑도 요구 페이징 활용.
2. **컨테이너 빠른 시작**: Docker 이미지 레이어를 OverlayFS + 요구 페이징으로 마운트 → 실제 컨텐츠 접근 시점에만 디스크 읽기. 컨테이너 시작 시간 수 초 → 수백ms.

**안티패턴**:
- **NUMA 노드 원격 메모리 Page Fault**: NUMA 서버에서 멀리 있는 노드의 메모리에 Page Fault → 원격 메모리 접근 레이턴시 ~2배. `numactl`로 로컬 할당 강제.

**📢 섹션 요약 비유**: NUMA 원격 Page Fault는 다른 도시 도서관에서 책을 빌려오는 것 — 같은 책이지만 배달 받는 데 훨씬 오래 걸려요.

## Ⅴ. 기대효과 및 결론

| 구분 | 완전 적재 | 요구 페이징 |
|:---|:---|:---|
| 메모리 효율 | 낮음 (전체 적재) | 높음 (실사용만) |
| 프로세스 시작 | 느림 (전체 I/O) | 빠름 (최소 적재) |
| Page Fault 비용 | 없음 | 있음 (수 ms) |
| 가상 메모리 지원 | 불가 | 가능 |

요구 페이징은 현대 OS 가상 메모리의 근간이다. 물리 메모리 한계를 초월하고 프로세스 시작을 가속하며, COW/선행 페이징과 함께 현대 컴퓨팅의 메모리 효율을 극대화한다.

## 👶 어린이를 위한 3줄 비유 설명
1. 요구 페이징은 도서관 OnDemand 배달 — 책이 필요해질 때만 가져오니까 책상(메모리)이 넓게 쓰여요.
2. 책을 꺼내오는 중엔 잠깐 기다려야 해요 (Page Fault) — 하지만 딱 필요한 책만 가져오니까 전체적으로 훨씬 효율적이에요.
3. 요구 페이징 덕분에 책상(16GB RAM)보다 훨씬 큰 프로그램(100GB 가상 메모리)도 실행할 수 있어요!
