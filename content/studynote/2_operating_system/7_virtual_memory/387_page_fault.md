# 페이지 부재 (Page Fault)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이지 부재 (Page Fault)는 CPU가 접근하려는 가상 페이지가 현재 물리 메모리(DRAM)에 없을 때 MMU가 발생시키는 하드웨어 트랩으로, OS가 가로채어 해당 페이지를 디스크에서 메모리로 적재한 후 실패한 명령어를 투명하게 재실행하는 요구 페이징의 핵심 이벤트다.
> 2. **가치**: Page Fault가 비정상적으로 높으면 스래싱(Thrashing)으로 처리량이 붕괴되지만, 정상적인 범위의 Page Fault는 요구 페이징의 필연적 결과이며, 페이지 교체 알고리즘의 품질로 그 빈도를 최소화할 수 있다.
> 3. **융합**: OOM (Out Of Memory) 상황, 성능 프로파일링 (`perf stat page-faults`), mmap 파일 매핑, 마이너 폴트(Minor Fault, 디스크 I/O 없음) vs 메이저 폴트(Major Fault, 디스크 I/O 필요)의 구분이 실무 메모리 성능 튜닝의 핵심이다.

---

## Ⅰ. 개요 및 필요성

가상 메모리 시스템에서 각 프로세스는 거대한 가상 주소 공간을 갖지만, 실제 DRAM의 물리 프레임은 유한하다. 일부 가상 페이지는 DRAM에 없고 디스크(스왑 공간)에만 있을 수 있다.

CPU가 이런 페이지에 접근하면 MMU는 페이지 테이블 엔트리의 `V=0` (Valid=0)을 감지하고 Page Fault 트랩을 발생시킨다.

**💡 비유**: 책상에 없는 책을 찾을 때 서가에서 가져오는 것 — Page Fault는 "이 책이 없어요!"라는 신호이고, OS가 디스크 서가에서 꺼내와 책상(RAM)에 올려주는 것.

```text
┌──────────────────────────────────────────────────────────────┐
│         Page Fault 처리 6단계                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. CPU → MMU: 가상 주소 접근                               │
│  2. MMU: PTE Valid=0 감지 → Page Fault 인터럽트 발생        │
│  3. OS 폴트 핸들러 진입:                                    │
│     ① 레지스터·PCB 저장                                    │
│     ② 페이지 테이블에서 디스크(스왑) 위치 확인            │
│  4. 빈 프레임 확보:                                         │
│     ● 빈 프레임 있으면 바로 사용                           │
│     ● 없으면 페이지 교체 알고리즘으로 희생 페이지 선택    │
│     ● Dirty 비트=1이면 스왑 아웃(디스크 쓰기) 먼저         │
│  5. 디스크 I/O: 페이지를 프레임으로 로드 (수백μs~수ms)    │
│  6. PTE 갱신 (Valid=1, 프레임 번호)  →  PC 복원 → 재실행  │
│                                                              │
│  [Minor Fault]:  페이지가 커널 버퍼에 이미 있음            │
│                  → 디스크 I/O ZERO, 수십μs                 │
│  [Major Fault]:  디스크 실제 읽기 필요                      │
│                  → 수백μs~수ms (훨씬 비쌈)                 │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: Page Fault는 도서관 사서가 받는 '서가 요청 슬립' — 슬립을 받으면 창고(스왑)에서 책(페이지)을 꺼내 책상(물리 메모리)에 올려주고, 독자(CPU)는 잠시 대기합니다.

## Ⅱ. 아키텍처 및 핵심 원리

### Major vs Minor Page Fault

```text
┌──────────────────────────────────────────────────────────────┐
│         Major Fault vs Minor Fault 비교                     │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Major Fault (메이저 폴트):                                 │
│  ● 페이지가 디스크 스왑 또는 파일시스템에 있음              │
│  ● 실제 디스크 I/O 발생 (HDD: 수ms, SSD: 수십~수백μs)     │
│  ● 높은 레이턴시 → 문맥 교환 후 I/O 대기                   │
│                                                              │
│  Minor Fault (마이너 폴트):                                 │
│  ● 페이지가 이미 메모리 어딘가에 있음 (페이지 캐시 등)    │
│  ● 디스크 I/O 없이 매핑만 수정                             │
│  ● 매우 빠름 (수십μs 미만)                                  │
│                                                              │
│  발생 예:                                                    │
│  fork() 후 COW → 최초 쓰기 시 Minor Fault (메모리 복사)   │
│  mmap 공유 라이브러리 첫 접근 → Minor Fault (페이지 캐시) │
│  스왑된 페이지 접근 → Major Fault (디스크 읽기)            │
└──────────────────────────────────────────────────────────────┘
```

### Page Fault율 측정

```bash
# 프로세스의 Major/Minor Fault 카운트
ps -o pid,min_flt,maj_flt <PID>

# perf로 Page Fault 이벤트 측정
perf stat -e page-faults ./my_program

# /proc/PID/status
grep -i fault /proc/<PID>/status
```

**📢 섹션 요약 비유**: Major Fault는 도서관에서 직접 서가에 가서 책 찾기(느림), Minor Fault는 사서실 옆 반납대에서 책 바로 꺼내기(빠름) — 어디에 있느냐로 속도가 천지차이에요.

## Ⅲ. 융합 비교 및 다각도 분석

### 스래싱 (Thrashing) — 과도한 Page Fault의 결말

다중 프로그래밍 정도가 너무 높으면 모든 프로세스의 워킹 셋(Working Set)을 DRAM에 올릴 공간이 부족 → 지속적 Page Fault → CPU가 스케줄링보다 스왑에 더 많은 시간 소비 → 처리량 붕괴.

**해결**: 워킹 셋 모델 적용 → 프로세스가 워킹 셋 만큼의 프레임을 보장받지 못하면 일시 중단.

**📢 섹션 요약 비유**: 스래싱은 책상이 좁아서 책을 서가에서 꺼내 올리고, 올리자마자 내려야 하는 악순환 — 정작 공부(연산)는 못 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

**실무 시나리오**:
1. **ML 학습 데이터 로딩**: 대규모 데이터셋을 mmap으로 매핑 → 에포크별 무작위 접근 시 Major Fault 다발 → SSD 기반 스토리지가 필수. NVMe SSD: Minor Fault 전환 확률 높음.
2. **Node.js 서버 콜드 스타트**: 첫 요청 시 모든 require() 모듈 페이지가 Major Fault 발생 → 워밍업 기간 레이턴시 급등. Container 이미지로 표준화하면 페이지 캐시 재사용 가능.

**안티패턴**:
- **스왑 파티션 없는 시스템에서 OOM**: 물리 메모리 소진 시 스왑 없어 OOM Killer 즉시 발동 → 중요 프로세스 종료. 4~8GB 스왑 파티션으로 완충 필요.

**📢 섹션 요약 비유**: 스왑 없이 OOM은 창고(스왑)도 없이 책상이 꽉 찬 상황 — 도서관장이 학생(프로세스)을 강제 퇴실시켜야 새 자리가 생겨요.

## Ⅴ. 기대효과 및 결론

| 구분 | Major Fault | Minor Fault |
|:---|:---|:---|
| 디스크 I/O | 있음 | 없음 |
| 레이턴시 | 수백μs~수ms | 수십μs 미만 |
| 성능 영향 | 큼 | 작음 |

Page Fault는 요구 페이징의 핵심 이벤트이며, Minor/Major 구분과 Fault 빈도를 모니터링하는 것이 메모리 성능 튜닝의 첫 단계다. 스래싱을 방지하기 위한 워킹 셋 모델과 적절한 스왑 설계가 안정적인 메모리 관리의 기반이다.

## 👶 어린이를 위한 3줄 비유 설명
1. Page Fault는 "이 책 지금 책상(RAM)에 없어요!"라는 경고 — 도서관(OS)이 창고(디스크)에서 꺼내와 책상에 올려줘요.
2. 창고까지 갔다 와야 하면 느리고(Major Fault), 옆 반납대에서 꺼내면 빠릅니다(Minor Fault).
3. 책상이 너무 작아 책을 계속 올리고 내리기만 반복하면 정작 공부(연산)를 못 해요 — 그게 스래싱이에요!
