# 429. 메모리 과할당 (Memory Overcommit)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 실제 메모리보다 많이 할당 약속
> 2. **가치**: 메모리 효율성 증대
> 3. **융합**: OOM 킬러, 스와핑과 연관

---

## Ⅰ. 개요

### 개념 정의

메모리 과할당(Memory Overcommit)은 **실제 물리 메모리보다 더 많은 메모리를 프로세스에 할당 약속하는 기법**이다. 실제로는 다 쓰지 않는다는 가정을 한다.

### 💡 비유: 항공사 오버부킹
메모리 과할당은 **항공사 오버부킹**과 같다. 좌석보다 많이 티켓을 판다. 다 타지 않을 것이라 가정한다.

### 메모리 과할당 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                메모리 과할당 구조                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【기본 개념】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  물리 메모리: 16GB                                                      │ │
│  │                                                             │ │
│  │  과할당 없음:                                                            │
│  │  프로세스 A: 4GB 할당                                                   │ │
│  │  프로세스 B: 4GB 할당                                                   │ │
│  │  → 최대 16GB까지만 할당 가능                                             │ │
│  │                                                             │ │
│  │  과할당:                                                                 │ │
│  │  프로세스 A: 4GB 요청, 2GB 실제 사용                                     │ │
│  │  프로세스 B: 4GB 요청, 1GB 실제 사용                                     │ │
│  │  프로세스 C: 4GB 요청, 1GB 실제 사용                                     │ │
│  │  → 총 12GB 요청, 4GB 실제 사용 → 16GB로 충분!                           │ │
│  │                                                             │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【과할당이 가능한 이유】                                                │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  1. Copy-on-Write:                                                     │
│  │     fork() 시 부모와 자식이 페이지 공유                                   │ │
│  │     쓰기 시에만 복사                                                     │ │
│  │                                                             │ │
│  │  2. Demand Paging:                                                      │ │
│  │     malloc() 해도 즉시 물리 페이지 할당 안 됨                             │ │
│  │     실제 접근 시 페이지 할당                                              │ │
│  │                                                             │ │
│  │  3. Sparse Allocation:                                                  │
│  │     대용량 배열을 할당해도 일부만 사용                                    │ │
│  │                                                             │ │
│  │  4. 공유 메모리:                                                         │ │
│  │     여러 프로세스가 같은 페이지 공유                                       │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【과할당 문제 발생】                                                    │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  모든 프로세스가 할당받은 메모리를 실제로 사용하면:                        │ │
│  │                                                             │ │
│  │  → 물리 메모리 부족 (OOM: Out of Memory)                                │ │
│  │  → OOM Killer가 프로세스 종료                                           │ │
│  │  → 또는 스와핑으로 인한 성능 저하                                         │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                메모리 과할당 상세                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Linux 과할당 모드】                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  /proc/sys/vm/overcommit_memory:                                       │ │
│  │                                                             │ │
│  │  0 (Heuristic):                                                         │ │
│  │  • 기본값                                                               │ │
│  │  • 과할당 허용하되 "명백히 불가능한" 요청은 거부                          │ │
│  │  • 휴리스틱 알고리즘으로 판단                                             │ │
│  │                                                             │ │
│  │  1 (Always):                                                            │ │
│  │  • 항상 과할당 허용                                                      │ │
│  │  • "malloc(1TB)"도 성공                                                 │ │
│  │  • 위험하지만 일부 워크로드에 유용                                        │ │
│  │                                                             │ │
│  │  2 (Never):                                                             │ │
│  │  • 과할당 금지                                                          │ │
│  │  • CommitLimit까지만 할당                                               │ │
│  │  • 안정성 중요 시 사용                                                   │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【CommitLimit 계산】                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  CommitLimit = (물리메모리 + 스왑) × overcommit_ratio / 100             │ │
│  │                                                             │ │
│  │  예: 물리메모리 16GB, 스왑 4GB, overcommit_ratio=50                     │ │
│  │  CommitLimit = (16 + 4) × 50 / 100 = 10GB                             │ │
│  │                                                             │ │
│  │  Committed_AS: 현재 할당된 총 메모리                                     │ │
│  │  Committed_AS > CommitLimit → OOM 가능                                 │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【과할당 vs 비과할당】                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  과할당 장점:                                                            │ │
│  │  • 메모리 효율성 향상                                                    │ │
│  │  • fork() 빠름                                                          │ │
│  │  • sparse allocation 지원                                               │ │
│  │                                                             │ │
│  │  과할당 단점:                                                            │ │
│  │  • OOM Killer 위험                                                     │ │
│  │  • 예측 불가능한 프로세스 종료                                           │ │
│  │  • 데이터 손실 가능                                                     │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【과할당 설정 확인】                                                  │
│  ──────────────────                                                  │
│  // 현재 과할당 모드                                                    │
│  $ cat /proc/sys/vm/overcommit_memory                               │
│  0  // 0=heuristic, 1=always, 2=never                                │
│                                                                     │
│  // 과할당 비율                                                         │
│  $ cat /proc/sys/vm/overcommit_ratio                                │
│  50  // 물리메모리의 50% 추가로 과할당 허용                               │
│                                                                     │
│  // 커밋 한계 확인                                                      │
│  $ cat /proc/meminfo | grep -E "CommitLimit|Committed_AS"           │
│  CommitLimit:     10240000 kB                                       │
│  Committed_AS:     8192000 kB  // 현재 할당량                         │
│                                                                     │
│  【과할당 설정 변경】                                                  │
│  ──────────────────                                                  │
│  // 과할당 비활성화 (안정성 우선)                                        │
│  $ sysctl vm.overcommit_memory=2                                    │
│  $ sysctl vm.overcommit_ratio=100                                   │
│                                                                     │
│  // 과할당 활성화 (성능 우선)                                           │
│  $ sysctl vm.overcommit_memory=1                                    │
│                                                                     │
│  // 영구 설정                                                          │
│  $ echo "vm.overcommit_memory=2" >> /etc/sysctl.conf                │
│  $ sysctl -p                                                         │
│                                                                     │
│  【OOM 발생 시 대응】                                                  │
│  ──────────────────                                                  │
│  // OOM 로그 확인                                                      │
│  $ dmesg | grep -i "out of memory"                                  │
│  Out of memory: Killed process 1234 (bigprogram)                    │
│                                                                     │
│  // OOM 점수 확인                                                      │
│  $ cat /proc/<pid>/oom_score                                        │
│  123  // 높을수록 종료 우선순위                                          │
│                                                                     │
│  // 특정 프로세스 OOM 보호                                               │
│  $ echo -1000 > /proc/<pid>/oom_score_adj                           │
│  // -1000 = 절대 종료 안 함                                             │
│                                                                     │
│  【프로그래밍에서 주의점】                                              │
│  ──────────────────                                                  │
│  // 큰 메모리 할당                                                      │
│  void *ptr = malloc(1024L * 1024 * 1024 * 10);  // 10GB             │
│  // overcommit_memory=1이면 성공할 수 있음                              │
│  // 실제 사용 시 OOM 발생 가능                                          │
│                                                                     │
│  // 안전한 할당                                                         │
│  #include <sys/mman.h>                                               │
│  void *ptr = mmap(NULL, size, PROT_READ|PROT_WRITE,                 │
│                   MAP_PRIVATE|MAP_ANONYMOUS|MAP_NORESERVE, -1, 0);   │
│  // MAP_NORESERVE: 스왑 공간 예약 안 함                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 실제 메모리보다 많이 할당 약속
• 이유: 다 쓰지 않음, COW, demand paging
• 모드: 0(heuristic), 1(always), 2(never)
• 위험: OOM Killer로 프로세스 종료
• 설정: overcommit_memory, overcommit_ratio
• 확인: CommitLimit, Committed_AS
• 권장: 서버는 2(never), 데스크톱은 0
```
