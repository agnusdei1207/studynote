## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 순수 요구 페이징(PDP, Pure Demand Paging)은 프로세스 실행 시작 시 단 하나의 페이지도 메모리에 올리지 않고, CPU가 첫 명령어를 실행하는 순간 발생하는 페이지 부재를 통해 필요한 페이지를 하나씩 채워가는 극단적인 지연 로딩 방식이다.
> 2. **가치**: 메모리 낭비를 제로(Zero)에 가깝게 유지하며 프로세스를 즉각적으로 '논리적 실행 상태'로 만들 수 있으나, 초기 실행 시 잦은 페이지 부재로 인한 지연(Latency)이 발생한다.
> 3. **특징**: "메모리에 페이지가 단 하나도 없는 상태"에서 시작한다는 점이 일반적인 요구 페이징과 구별되는 핵심 포인트다.

---

### Ⅰ. 개요 (Context & Background)

- **概念**: **순수 요구 페이징 (PDP, Pure Demand Paging)**은 요구 페이징의 가장 순수한 형태다. 프로세스가 생성될 때 OS는 메모리에 어떤 페이지도 적재하지 않는다. CPU가 프로세스의 첫 번째 명령어를 실행하려고 시도하면 즉시 페이지 부재가 발생하며, 이때 비로소 첫 페이지가 로드된다.

- **💡 비유**: 순수 요구 페이징은 **"빈 가방으로 등교하기"**와 같다. 학교에 갈 때 책을 한 권도 챙기지 않고 일단 교실(CPU)에 앉은 뒤, 1교시 수학 시간이 되면 그제서야 퀵서비스(Page Fault Handler)로 수학책을 배달받아 공부를 시작하는 것과 같다.

- **등장 배경**:
  1. **극단적 자원 최적화**: 어떤 코드가 처음에 쓰일지 전혀 예측할 수 없는 상황에서 불필요한 I/O를 완전히 배제하기 위함이다.
  2. **이론적 완결성**: 가상 메모리 시스템이 "필요할 때만 가져온다"는 원칙을 어디까지 밀어붙일 수 있는지 보여주는 모델이다.
  3. **메모리 부족 상황**: 가용 RAM이 극히 적은 환경에서 여러 프로세스를 일단 띄워놓고 봐야 할 때 유용하다.

- **📢 섹션 요약 비유**: 밑바닥부터 하나씩 벽돌을 쌓아 올리는 것처럼, 아무것도 없는 백지상태에서 시작하는 메모리 적재 방식입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 순수 요구 페이징의 실행 단계 (표)

| 단계 | 상태 | 주요 동작 | 결과 | 비유 |
|:---|:---|:---|:---|:---|
| **Step 1** | 초기화 | 프로세스 제어 블록(PCB) 생성, 메모리 적재량 0 | 실행 준비 완료 | 빈 책상 |
| **Step 2** | 첫 실행 시도 | CPU가 Entry Point 주소 참조 | **Page Fault 발생** | "책이 없네?" |
| **Step 3** | 부재 처리 | Disk에서 첫 명령어 페이지 적재 | 실행 환경 구성 시작 | 책 배달 완료 |
| **Step 4** | 반복 | 다음 명령어/데이터 참조 시 부재 발생 | 페이지 점진적 확충 | 한 페이지씩 읽기 |

#### 순수 요구 페이징의 초기 부재 곡선 (ASCII Diagram)

```text
    Page Fault Count
      ▲
      │  * (Start)
      │  *
      │   *
      │    *
      │     *
      │       *
      │          * * * * *  (Steady State)
      └──────────────────────────────────▶ Time
         (Initial Burst)
```

**[다이어그램 해설]** 순수 요구 페이징은 실행 초기에 페이지 부재가 폭발적으로 발생한다(Initial Burst). 메모리에 아무것도 없기 때문에 거의 모든 참조가 부재로 이어지기 때문이다. 하지만 시간이 흐르며 프로세스 실행에 꼭 필요한 '지역성(Locality)' 기반의 페이지들이 메모리에 채워지면, 부재 빈도는 급격히 낮아져 안정 상태(Steady State)에 도달한다.

- **📢 섹션 요약 비유**: 처음 이사 왔을 때는 모든 물건을 박스에서 꺼내느라 정신없지만(부재 폭발), 며칠 지나면 자주 쓰는 물건은 이미 다 꺼내져 있어 편해지는 것(안정화)과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

#### 비교: 일반 요구 페이징 vs 순수 요구 페이징

| 비교 항목 | 일반 요구 페이징 (Demand) | 순수 요구 페이징 (Pure Demand) |
|:---|:---|:---|
| **시작 적재량** | 최소한의 코드/데이터 (Pre-load) | **Zero (0)** |
| **초기 응답성** | 중간 (약간의 로딩 대기) | 최상 (즉시 시작되나 바로 멈춤) |
| **초기 I/O 부하** | 중간 (분산 적재) | 매우 높음 (연쇄 부재 발생) |
| **구현 난이도** | 표준적 | 예외 처리 로직의 정교함 요구 |

- **📢 섹션 요약 비유**: 편의점에서 삼각김밥 몇 개 사들고 소풍 가는 것(일반)과, 일단 빈손으로 가서 배고플 때마다 편의점을 찾는 것(순수)의 차이입니다.

### Ⅳ. 실무 적용 및 기술사적 판단

#### 실무 시나리오: 마이크로서비스의 빠른 Scale-out
수천 개의 컨테이너 인스턴스를 순식간에 늘려야 하는 클라우드 환경에서, 모든 데이터를 다 로드하고 실행하려면 전체 클러스터에 과부하가 걸릴 수 있다. 이때 순수 요구 페이징과 유사한 'Lazy 로딩' 기법을 사용하면, 일단 인스턴스를 가동 상태(Running)로 빠르게 전환한 뒤, 실제 트래픽이 들어오는 부분부터 점진적으로 데이터를 채워 넣어 전체 시스템의 급격한 I/O 피크를 방지할 수 있다.

- **📢 섹션 요약 비유**: 손님이 오기도 전에 요리를 다 해놓는 것이 아니라, 손님이 문을 열고 들어오는 순간부터 가스불을 켜는 초절전 운영 방식입니다.

### Ⅴ. 기대효과 및 결론

#### 순수 요구 페이징의 정성적 기대효과
1. **메모리 점유 제로화**: 실행되지 않는 프로세스는 물리 메모리를 단 1바이트도 차지하지 않게 할 수 있다.
2. **이론적 지표 제공**: 페이지 부재 처리 성능이 시스템 전체 성능에 미치는 영향을 가장 극명하게 관찰할 수 있는 기준점이 된다.
3. **지역성 원리 증명**: 시간이 지나면 결국 소수의 페이지만으로도 원활한 실행이 가능해진다는 '참조 지역성'을 몸소 보여주는 기법이다.

- **📢 섹션 요약 비유**: 비록 시작은 느리고 고통스러울 수 있지만, 가장 필요한 것만 골라 담는 '미니멀리즘'의 극치를 보여주는 메모리 관리 기술입니다.

### 📌 관련 개념 맵
- **참조 지역성 (Locality of Reference)**: 초기 부재 폭발 이후 안정화되는 근거.
- **페이지 부재 (Page Fault)**: 순수 요구 페이징의 유일한 적재 동력.
- **Lazy Swapper**: 페이지 단위로 필요한 것만 실어 나르는 주체.

### 👶 어린이를 위한 3줄 비유 설명
1. 순수 요구 페이징은 장난감을 하나도 안 가지고 놀이터에 가는 거예요.
2. 가서 "시소 타고 싶어!" 하면 엄마가 시소를 가져오고, "모래놀이 하고 싶어!" 하면 그때 모래 삽을 가져다주는 방식이죠.
3. 처음엔 기다리는 시간이 많지만, 나중엔 내가 좋아하는 장난감이 다 옆에 모이게 된답니다!
