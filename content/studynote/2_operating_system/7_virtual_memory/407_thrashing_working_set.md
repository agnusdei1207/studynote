# 스래싱 (Thrashing) & 워킹 셋 (Working Set)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스래싱 (Thrashing)은 다중 프로그래밍 정도가 임계치를 초과하여, 물리 메모리가 모든 프로세스의 워킹 셋(Working Set)을 수용하지 못할 때 CPU가 연산보다 페이지 스와핑에 시간을 대부분 소비하며 처리량이 붕괴되는 현상이다.
> 2. **가치**: 워킹 셋 모델 (Working Set Model)은 각 프로세스가 현재 참조하는 최소 페이지 집합(워킹 셋)에 해당하는 프레임을 보장하여 스래싱을 사전에 예방하는 이론적·실무적 프레임 할당의 기준이다.
> 3. **융합**: 페이지 부재 빈도 (PFF, Page Fault Frequency) 기반으로 프레임 수를 동적 조정하고, 다중 프로그래밍 정도를 제한하며, Linux의 OOM Killer와 swappiness 파라미터가 스래싱 상황에서 OS 수준의 완충 역할을 한다.

---

## Ⅰ. 개요 및 필요성

요구 페이징으로 동시에 여러 프로세스를 실행하면 CPU 이용률이 높아진다 — 어느 한 프로세스가 I/O 대기 중일 때 다른 프로세스가 CPU를 사용할 수 있기 때문이다.

그러나 프로세스 수가 너무 많아지면 역효과가 발생한다:
1. 모든 프레임이 사용 중
2. 새 Page Fault 발생 → 교체 대상이 다른 프로세스의 워킹 셋 페이지
3. 교체된 프로세스가 곧 그 페이지를 요구 → 또 Page Fault
4. CPU는 Page Fault 처리와 스왑 I/O만 반복 → 처리량 0에 가까움

**💡 비유**: 공부방에 너무 많은 학생이 있어서 책상(메모리)이 꽉 차고, 모두가 계속 책을 꺼냈다 넣었다만 반복하느라 정작 공부(연산)를 못 하는 상황.

```text
┌──────────────────────────────────────────────────────────────┐
│         스래싱 발생 곡선 — 다중 프로그래밍 정도 vs CPU 이용률│
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  CPU 이용률                                                  │
│  ▲                    ●                                      │
│  │                 ●     ●  ← 최적점                         │
│  │              ●           ●                                │
│  │           ●                ●                              │
│  │         ●                    ●                            │
│  │        ●                       ●  ← 스래싱 시작!          │
│  │       ●                           ●                       │
│  └──────────────────────────────────────────▶                │
│    다중 프로그래밍 정도 (프로세스 수)                        │
│                                                              │
│  최적점 이후: 프로세스 증가 → CPU 이용률 급감                │
│  이유: 스왑 I/O가 CPU 시간을 독점                            │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: 스래싱은 혼잡한 식당에서 서빙 직원(CPU)이 주문(연산) 대신 접시 나르기(스왑)에만 정신없는 상황 — 주문이 없는 게 아닌데 처리가 안 돼요.

## Ⅱ. 아키텍처 및 핵심 원리

### 워킹 셋 모델 (Working Set Model)

```text
┌──────────────────────────────────────────────────────────────┐
│         워킹 셋 (Working Set) 개념                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Working Set W(Δ, t): 시간 t 기준 최근 Δ회의 참조에서        │
│                        나타난 고유 페이지 집합               │
│                                                              │
│  예: Δ=5 (최근 5회 참조), 참조 문자열: 1 2 3 2 1 5 4 5 4 5   │
│  t=9: W = {1, 4, 5} (최근 5회: 1,5,4,5,4)                    │
│                                                              │
│  프레임 할당 원칙:                                           │
│  각 프로세스의 |WS(Δ, t)| 만큼의 프레임 보장                 │
│                                                              │
│  총 워킹 셋 크기:  D = Σ |WS_i(Δ)|                           │
│  D > 전체 프레임 수 → 스래싱 발생 → 프로세스 중단 필요       │
│                                                              │
│  D ≤ 전체 프레임 수 → 안전, 새 프로세스 추가 가능            │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Δ (델타)는 워킹 셋 창(Window)—크면 워킹 셋이 커져 보수적이고, 작으면 빠르게 변화를 반영한다. 국소성 원리 덕분에 시간 t의 워킹 셋은 프로그램 실행 중 특정 단계에서 자주 참조되는 일부 페이지만 포함한다.

### PFF (Page Fault Frequency) 기반 동적 조정

```text
┌──────────────────────────────────────────────────────────────┐
│         PFF 기반 프레임 동적 조정                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  PFF 상한(Upper Bound) 초과:                                 │
│  → Page Fault 너무 많음 → 프레임 추가 할당                   │
│  → 프레임 여유 없으면 → 다중 프로그래밍 정도 감소            │
│     (프로세스 중단 후 스왑 아웃)                             │
│                                                              │
│  PFF 하한(Lower Bound) 미달:                                 │
│  → Page Fault 너무 적음 → 프레임 과다 할당                   │
│  → 프레임 회수 → 다른 프로세스에 재분배                      │
│                                                              │
│  PFF 범위 내:                                                │
│  → 프레임 할당 유지 → 안정적 동작                            │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: PFF 조정은 온도 조절기 — 너무 춥다(PFF 낮음=프레임 과다)면 히터를 줄이고, 너무 덥다(PFF 높음=프레임 부족)면 히터를 올리는 자동 균형 시스템입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Linux swappiness 파라미터

```bash
# 현재값 확인 (기본 60)
cat /proc/sys/vm/swappiness

# 즉시 변경 (0: 스왑 최소화, 100: 스왑 적극)
sysctl vm.swappiness=10
```

- `swappiness=0`: 물리 메모리 최대 사용, 스왑 최소 (DB 서버 권장)
- `swappiness=60`: 기본값 (범용)
- `swappiness=100`: 적극적 스왑 (메모리 비용 절감 환경)

**📢 섹션 요약 비유**: swappiness는 물건 정리 성향 — 0은 책상을 꽉 채워야 직성이 풀리는 사람, 100은 조금만 쌓여도 서랍에 넣는 사람이에요.

## Ⅳ. 실무 적용 및 기술사적 판단

**실무 시나리오**:
1. **MySQL InnoDB 스래싱**: 서버 RAM 32GB, InnoDB 버퍼 풀 30GB + OS 페이지 캐시 경쟁 → 스왑 발생 → 쿼리 레이턴시 100배 급증. `vm.swappiness=1`로 설정, 더 적은 버퍼 풀 크기(26GB)로 조정 해결.
2. **JVM 대규모 힙 스래싱**: 64GB 힙 JVM 서버에서 GC 후 힙 페이지들이 스왑에서 복구되는 과정에서 수 초 스래싱 → `-Xmx`를 물리 메모리의 50%로 제한.

**안티패턴**:
- **스래싱 무시 운영**: 서버 CPU 이용률이 90%+ 인데 실제 처리량은 낮은 경우 스래싱 의심 → `vmstat 1`로 si/so (swap in/out) 확인.

**📢 섹션 요약 비유**: CPU가 바쁜데 일처리가 안 되면 스래싱 의심 — 직원(CPU)이 서류(데이터) 찾느라 바쁜 거지 정작 업무(연산)는 못 하는 상황이에요.

## Ⅴ. 기대효과 및 결론

| 구분 | 스래싱 방치 | 워킹 셋·PFF 기반 관리 |
|:---|:---|:---|
| CPU 이용률 | 겉보기 높음, 실효 0 | 실효 높음 |
| 처리량 | 붕괴 | 정상 |
| Page Fault | 극단적으로 증가 | 관리 범위 내 |
| 해결 방법 | 없음 (자동 재귀) | 프로세스 중단/프레임 재분배 |

스래싱은 메모리 관리의 가장 위험한 성능 함정이다. 워킹 셋 모델로 프레임을 보장하고, PFF 기반으로 동적 조정하며, swappiness와 OOM Killer를 적절히 설정하여 시스템 안정성을 확보해야 한다.

## 👶 어린이를 위한 3줄 비유 설명
1. 스래싱은 공부방 대혼란 — 너무 많은 학생(프로세스)이 책(페이지)을 꺼냈다 넣었다만 반복해서 정작 공부(연산)를 못 해요.
2. 워킹 셋은 각 학생에게 최소한의 책상 자리(프레임)를 보장하는 것 — 자기 책을 펼쳐둘 공간이 있어야 공부할 수 있어요.
3. 자리가 부족하면 공부 잠시 쉬게(프로세스 중단) 하는 게 오히려 모두가 더 빨리 끝내는 방법이에요!
