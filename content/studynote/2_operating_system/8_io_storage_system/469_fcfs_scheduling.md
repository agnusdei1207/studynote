## 핵심 인사이트 (3줄 요약)
> 1. **본질**: FCFS(First-Come, First-Served) 스케줄링은 디스크 I/O 요청이 들어온 순서대로 처리하는 가장 단순하고 직관적인 디스크 스케줄링 알고리즘이다.
> 2. **가치**: 스케줄링 오버헤드가 거의 없고 모든 요청에 대해 절대적인 공정성(Fairness)을 보장하며, 기아(Starvation) 현상이 발생하지 않는다는 장점이 있다.
> 3. **융합**: 요청된 트랙의 위치가 무작위일 경우 헤드가 디스크 양끝을 무분별하게 오가게 되어, 현대의 고부하 시스템에서는 성능 최적화보다는 기준점(Baseline) 비교용으로 주로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: **FCFS 스케줄링**은 선입선출(First-In, First-Out) 원칙에 따라 입출력 큐에 먼저 도착한 요청을 가장 먼저 서비스하는 방식이다.

- **💡 비유**: FCFS는 "은행의 번호표 시스템"과 같다. 내가 처리해야 할 업무가 무엇이든, 어디에 있든 상관없이 오직 번호표를 뽑은 순서대로만 직원이 업무를 처리해주는 방식이다.

- **주요 특징**:
  1. **비선점형(Non-preemptive)**: 한 번 서비스를 시작하면 끝날 때까지 중단되지 않음.
  2. **단순성**: 복잡한 계산이나 최적화 로직이 필요 없어 구현이 매우 쉬움.
  3. **예측 가능성**: 요청 순서가 실행 순서이므로 실행 흐름을 파악하기 좋음.

- **📢 섹션 요약 비유**: 복잡하게 머리 쓰지 않고 순서대로만 일하는 "고지식한 일꾼" 시스템입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### FCFS 작동 예시와 헤드 이동 경로

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 FCFS Scheduling 헤드 트랙 이동 예시                 │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ 요청 큐 순서 ]: 98, 183, 37, 122, 14, 124, 65, 67             │
  │   [ 현재 헤드 위치 ]: 53                                          │
  │                                                                  │
  │   헤드 이동 경로:                                                 │
  │   53 → 98 → 183 → 37 → 122 → 14 → 124 → 65 → 67                  │
  │                                                                  │
  │   【성능 분석】                                                   │
  │   - 총 헤드 이동 거리: (98-53) + (183-98) + (183-37) + (122-37)   │
  │                     + (122-14) + (124-14) + (124-65) + (67-65)   │
  │                     = 45 + 85 + 146 + 85 + 108 + 110 + 59 + 2    │
  │                     = 640 Cylinders                              │
  │   - 특징: 헤드가 좌우로 매우 크게 요동침 (지그재그 현상)           │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** FCFS는 트랙 간의 거리를 전혀 고려하지 않는다. 위 예시처럼 183번 트랙을 들렀다가 바로 다음에 37번 트랙으로 가는 식의 비효율적인 동선이 빈번하게 발생한다. 이는 물리적인 헤드 이동 시간(Seek Time)을 극단적으로 낭비하게 만드는 원인이 된다.

## Ⅲ. 융합 비교 및 다각도 분석

### FCFS vs SSTF (최단 탐색 시간 우선)

| 비교 항목 | FCFS 스케줄링 | SSTF 스케줄링 |
|:---|:---|:---|
| **결정 기준** | 도착 시간 (Arrival Time) | 현재 헤드와의 거리 (Distance) |
| **공정성** | **매우 높음** (차별 없음) | 낮음 (먼 트랙은 소외됨) |
| **성능 (처리량)** | 낮음 (헤드 이동 큼) | **높음** (헤드 이동 최소화) |
| **기아 현상** | 없음 | 발생 가능 |

---

### 과목 융합 관점

- **운영체제 (OS)**: 부하가 적은 시스템에서는 FCFS가 충분히 효율적일 수 있다. 스케줄링 알고리즘 자체의 CPU 점유율이 낮기 때문이다. 하지만 I/O 요청이 밀리는 고부하 상황에서는 시스템 전체의 병목을 유발한다.
- **알고리즘 (Algorithm)**: 큐(Queue) 자료구조를 사용하는 가장 기본적인 형태의 알고리즘이다. 시간 복잡도는 요청 선택 시 O(1)로 가장 빠르다.

- **📢 섹션 요약 비유**: 줄 선 순서대로 밥을 주는 건 공평하지만(FCFS), 가까운 곳부터 배달 가는 게 훨씬 효율적(SSTF)입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 초기 학습용 운영체제**: 운영체제의 원리를 처음 배우거나, 아주 단순한 임베디드 시스템(요청이 거의 발생하지 않는)에서는 리소스를 아끼기 위해 FCFS를 기본으로 채택할 수 있다.
2. **시나리오 — 순차적 데이터 처리**: 테이프 드라이브나 스트리밍 데이터를 순서대로 읽어야만 하는 특수 환경에서는 요청 순서가 곧 물리적 위치 순서인 경우가 많아 FCFS가 자연스럽게 적용된다.

### 도입 체크리스트
- **요청 빈도**: 디스크 요청이 드물게 발생하는가?
- **공정성 중시**: 성능보다 모든 사용자의 요청이 반드시 순서대로 처리되어야 하는가?
- **비교 분석**: 최적화 알고리즘 도입 전, 시스템의 성능 하한선(Lower Bound)을 측정할 데이터가 필요한가?

### 안티패턴
- **랜덤 읽기가 많은 데이터베이스**: DB 인덱스 스캔처럼 여러 트랙을 무작위로 오가는 작업에 FCFS를 쓰면, 디스크 헤드가 춤을 추는 소리를 들으며 무한 대기를 경험하게 된다.

- **📢 섹션 요약 비유**: 서울 들렀다 부산 갔다가 다시 경기도로 오는 동선은 여행 가이드(스케줄러)가 짤 수 있는 최악의 동선입니다.

## Ⅴ. 기대효과 및 결론

### 기대효과

- **안정성**: 알고리즘이 단순하여 버그가 발생할 여지가 적고 시스템이 견고함.
- **공정성 보장**: 먼저 요청한 프로그램이 반드시 먼저 결과를 받으므로 응답 순서가 뒤바뀌지 않음.

### 미래 전망
- **SSD에서의 재발견**: SSD는 헤드 이동 거리에 따른 페널티가 없다. 따라서 복잡한 SCAN 알고리즘 대신, 먼저 들어온 요청을 먼저 처리하는 FCFS(또는 Noop) 방식이 오히려 CPU 오버헤드를 줄여 더 좋은 성능을 낼 수도 있다.
- **Legacy 시스템**: 오래된 메인프레임이나 구형 하드웨어 컨트롤러에서는 여전히 기본 동작 방식으로 남아 하위 호환성을 지탱하고 있다.

- **📢 섹션 요약 비유**: "가장 기본이 가장 강력하다"는 말은 SSD 시대에 이르러 FCFS에게 다시 유효해지고 있습니다.
