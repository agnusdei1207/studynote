## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RAID 4는 데이터를 블록 단위로 여러 디스크에 스트라이핑하되, 오류 복구를 위한 패리티(Parity) 정보를 별도의 전용 디스크 하나에 몰아서 저장하는 방식이다.
> 2. **특징**: RAID 3(바이트 단위)와 달리 블록 단위 접근을 지원하여 독립적인 읽기 성능이 우수하지만, 모든 쓰기 작업 시 패리티 디스크에 부하가 집중되는 병목 현상이 발생한다.
> 3. **가치**: RAID 5로 넘어가는 과도기적 모델로 평가받으며, 쓰기 작업보다 읽기 작업이 압도적으로 많은 특정 워크로드(NetApp의 WAFL 등)에서 제한적으로 사용된다.

---

## Ⅰ. 개요 (Context & Background)

- **정의**: 블록 수준의 스트라이핑과 전용 패리티 드라이브를 결합하여 성능과 데이터 보호를 동시에 꾀한 RAID 레벨이다.
- **배경**: RAID 1의 낮은 공간 효율성(50%)을 극복하고, RAID 3의 단점인 '모든 디스크 동시 회전(병렬 처리 제약)'을 해결하기 위해 블록 단위 처리를 도입했다.
- **💡 비유**: **"전담 서기 시스템"**과 같다. 여러 명의 일꾼(데이터 디스크)이 각자 업무 일지를 쓰는데, 내용을 검증하고 요약하는 일(패리티 생성)은 단 한 명의 전담 서기(패리티 디스크)가 도맡아 하는 것과 같다. 일꾼이 많아질수록 서기 한 명의 업무량이 폭주하게 된다.

- **📢 섹션 요약 비유**: "분업은 잘 되어 있지만, 결재권자가 한 명이라 결재 대기 줄이 길게 늘어선 사무실"과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### RAID 4 블록 스트라이핑 및 전용 패리티 (ASCII Diagram)

```text
  [ 호스트 쓰기 요청 ]
           │
  ┌────────▼────────┐
  │  RAID 컨트롤러  │ (XOR 연산 수행)
  └────────┬────────┘
           │
  ┌────────┴───────────────────────────────────────────┐
  │      [ Disk 1 ]      [ Disk 2 ]      [ Parity Disk ]  │
  │      ┌──────┐        ┌──────┐        ┌─────────┐      │
  │      │ 블록 A │        │ 블록 B │        │ P(A⊕B)  │      │
  │      ├──────┤        ├──────┤        ├─────────┤      │
  │      │ 블록 C │        │ 블록 D │        │ P(C⊕D)  │      │
  │      └──────┘        └──────┘        └─────────┘      │
  │                                                       │
  │  * 읽기: D1, D2 독립적 수행 가능                         │
  │  * 쓰기: 어떤 블록을 쓰든 Parity Disk는 항상 가동        │
  └───────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터는 Disk 1, 2 등에 블록 단위로 나누어 저장된다. 이때 각 가로줄(Stripe)에 대한 XOR 연산 결과인 패리티 값은 무조건 마지막 '전용 패리티 디스크'에 기록된다. 데이터 디스크가 고장 나면 나머지 데이터와 패리티를 XOR 연산하여 소실된 데이터를 복구한다.

- **📢 섹션 요약 비유**: "각자 자기 집을 짓는데, 설계도 원본은 동사무소(패리티 디스크)에만 보관하는 방식"과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### RAID 4의 장단점 분석

| 항목 | 특성 | 비고 |
|:---|:---|:---|
| **최소 디스크 수** | 3개 | |
| **읽기 성능** | 우수 | 여러 디스크에서 블록 단위로 독립적 읽기 가능 |
| **쓰기 성능** | **나쁨 (병목 발생)** | 모든 쓰기 시 패리티 디스크를 갱신해야 함 |
| **공간 효율성** | **(n-1) / n** | 디스크 1개 분량만 희생 (RAID 5와 동일) |
| **장애 허용** | 1개 가능 | 데이터 디스크 또는 패리티 디스크 중 1개 |

### 패리티 병목 (Parity Bottleneck)
- RAID 4의 최대 아킬레스건이다. 디스크가 10개라고 해도, 9개의 데이터 디스크에 쓰기가 발생할 때마다 1개의 패리티 디스크는 9번의 쓰기를 감당해야 한다. 결국 패리티 디스크의 수명이 가장 먼저 다하거나 성능의 한계치가 된다.

- **📢 섹션 요약 비유**: "식당에서 요리사는 여럿인데, 계산대(패리티)가 하나뿐이라 손님이 나갈 때마다 줄을 서야 하는 상황"입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **NetApp 스토리지 (WAFL)**: NetApp은 RAID 4의 병목 문제를 소프트웨어적인 쓰기 최적화(Write Anywhere File Layout)로 극복하여 사용해왔다. 여러 쓰기를 모아 한꺼번에 패리티 디스크에 기록하는 방식으로 병목을 완화했다.
2. **읽기 전용 아카이브**: 일단 한 번 써두면 거의 읽기만 하는 대용량 데이터 저장소에서는 패리티 병목이 큰 문제가 되지 않으므로 적용 가능하다.

### 도입 체크리스트
- **쓰기 빈도**: 시스템의 Read:Write 비율이 9:1 이상인가?
- **패리티 디스크 내구성**: 패리티 디스크만 혹사당하므로, 해당 디스크만 고성능/고내구성(SSD 등) 제품을 쓰는 '비대칭 구성'이 가능한가?

### 안티패턴
- **일반 PC/서버용 RAID**: 특별한 최적화 알고리즘 없이 일반적인 환경에서 RAID 4를 쓰는 것은 RAID 5에 비해 아무런 이점이 없다. 패리티를 분산하는 RAID 5를 쓰는 것이 훨씬 현명하다.

- **📢 섹션 요약 비유**: "특별한 관리 요령이 없다면, 짐을 한 명에게 몰아주는 무모한 팀워크는 피해야 합니다."

## Ⅴ. 기대효과 및 결론

### 기대효과
- **경제적 데이터 보호**: RAID 1보다 훨씬 적은 비용(디스크 1개)으로 대용량 공간과 데이터 보호 기능을 동시에 확보.

### 미래 전망
- **RAID 5로의 진화**: 패리티 병목 문제를 해결하기 위해 패리티를 모든 디스크에 골고루 분산하는 RAID 5가 표준이 되면서, 순수 RAID 4는 시장에서 거의 자취를 감추었다. 다만, 일부 하이엔드 스토리지 솔루션의 내부 알고리즘으로 그 원리가 계승되고 있다.

- **📢 섹션 요약 비유**: "누군가 한 명의 희생으로 평화를 유지하던 시대에서, 모두가 책임을 분담하는 민주적인 시대(RAID 5)로 넘어가는 징검다리"입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **RAID 3 (바이트 단위 스트라이핑)**: RAID 4의 조상 격으로, 데이터 쪼개기 단위가 더 작음.
- **RAID 5 (분산 패리티)**: RAID 4의 패리티 병목을 해결한 완성형 모델.
- **XOR 연산**: 패리티 생성과 복구의 핵심 수학적 원리.

## 👶 어린이를 위한 3줄 비유 설명
1. **RAID 4**는 여러 명의 친구가 일기를 쓰는데, 내용을 요약해서 기록하는 '비밀 노트'를 한 친구에게만 맡기는 거예요.
2. 친구들이 일기를 쓸 때마다 비밀 노트를 가진 친구도 같이 써야 해서 그 친구만 엄청 바쁘고 힘들어요.
3. 하지만 나중에 일기장 하나를 잃어버려도 비밀 노트를 보면 내용을 다시 알아낼 수 있는 신기한 방법이랍니다!
