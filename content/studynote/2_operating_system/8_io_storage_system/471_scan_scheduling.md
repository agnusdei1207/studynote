## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SCAN 스케줄링은 디스크 헤드가 한쪽 끝에서 반대쪽 끝으로 이동하며 경로상의 모든 요청을 처리하고, 끝에 도달하면 방향을 바꾸어 다시 처리하는 방식이다.
> 2. **가치**: 엘리베이터의 동작 원리와 유사하여 '엘리베이터 알고리즘'이라 불리며, SSTF의 고질적인 문제인 기아(Starvation) 현상을 방지하고 공정성을 확보한다.
> 3. **융합**: 헤드의 이동 방향에 따라 응답 시간의 편차가 발생할 수 있으나, 전체적인 처리량(Throughput)과 응답 시간의 분산(Variance) 측면에서 균형 잡힌 성능을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: **SCAN 스케줄링**은 헤드가 현재 진행 방향으로 디스크의 끝까지 이동하면서 마주치는 모든 요청을 서비스하는 기법이다. 끝(0번 트랙 또는 최외곽 트랙)에 도달해서야 방향을 전환한다.

- **💡 비유**: 고층 빌딩의 **엘리베이터**와 같다. 엘리베이터가 올라가는 동안은 올라가는 방향의 층 버튼만 누른 사람들을 다 태우고 꼭대기까지 간 뒤에야 내려오는 것과 동일한 논리다.

- **주요 특징**:
  1. **기아 현상 해결**: 헤드가 계속 왕복하므로 어떤 요청도 무한정 대기하지 않는다.
  2. **상대적 공정성**: SSTF보다 응답 시간의 편차가 적다.
  3. **끝점 도달 강제**: 요청이 없더라도 일단 해당 방향의 마지막 트랙까지 이동해야 한다는 비효율성이 존재한다.

- **📢 섹션 요약 비유**: "한 방향으로 끝까지 가야 다음 길을 찾는 정직한 순찰차"와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SCAN 작동 메커니즘 (ASCII Diagram)

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 SCAN Scheduling 헤드 이동 시뮬레이션                 │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ 요청 큐 ]: 98, 183, 37, 122, 14, 124, 65, 67                  │
  │   [ 현재 위치 ]: 53 / [ 이동 방향 ]: 0번 트랙 방향 (감소 방향)        │
  │                                                                  │
  │   이동 경로:                                                      │
  │   53 (시작) → 37 → 14 → 0 (끝점 도달 및 방향 전환)                 │
  │   → 65 → 67 → 98 → 122 → 124 → 183                               │
  │                                                                  │
  │   【SCAN의 특징적 구간】                                           │
  │   - 14번 처리 후 요청이 없어도 '0번'까지 끝까지 이동함 (중요!)        │
  │   - 방향 전환 후 65, 67 등을 순차적으로 처리                      │
  │                                                                  │
  │   【성능 지표 계산】                                               │
  │   - 총 이동 거리: (53 - 0) + (183 - 0) = 236 Cylinders           │
  │   - SSTF와 거리는 비슷할 수 있으나, 공정성은 훨씬 높음             │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SCAN의 핵심은 '방향성'과 '끝점(Boundary)'이다. 14번 트랙 이후에 요청이 없어도 0번 트랙까지 찍고 돌아오는 것이 특징이다. 이 과정에서 헤드가 지나간 자리에 바로 새로 들어온 요청은 반대편 끝을 찍고 올 때까지 기다려야 하므로, 트랙 위치에 따른 대기 시간 불평등이 존재한다.

- **📢 섹션 요약 비유**: 중간에 손님이 없어도 종점까지 무조건 운행하는 "노선 버스"의 철학입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SSTF vs SCAN vs C-SCAN

| 비교 항목 | SSTF (Shortest Seek) | SCAN (Elevator) | C-SCAN (Circular) |
|:---|:---|:---|:---|
| **기아 현상** | 발생 가능 (심각) | 거의 없음 | 없음 |
| **이동 효율** | 최고 (탐욕적) | 좋음 | 보통 (복귀 시 이동 발생) |
| **공정성** | 낮음 | 중간 (중앙 트랙 유리) | 높음 (균등한 대기 시간) |
| **끝점 이동** | 안 함 | **수행함** | 수행함 |

### 응답 시간의 불균등성 (Wait Time Variance)
- **중앙 트랙**: 헤드가 왕복할 때 자주 지나치게 되므로 응답 시간이 빠르다.
- **양 끝 트랙**: 헤드가 한 번 지나가면 반대쪽 끝을 찍고 돌아올 때까지 가장 오래 기다려야 한다.

- **📢 섹션 요약 비유**: 길 중간에 사는 사람은 버스를 자주 보지만, 종점 근처 사는 사람은 버스 구경하기 힘든 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **대규모 파일 서버**: 수많은 사용자가 불규칙하게 I/O를 요청하는 환경에서 특정 사용자의 무한 대기를 방지하기 위해 SCAN 계열이 선호된다.
2. **범용 OS (Legacy)**: 과거 HDD가 주력일 때 많은 유닉스(Unix) 시스템의 기본 스케줄러 원리로 채택되었다.

### 도입 체크리스트
- **Boundary 이동 비효율**: 요청이 없는 끝단까지 이동하는 시간을 허용할 수 있는가? (허용 불가 시 LOOK 알고리즘 검토)
- **응답 시간 균일성**: 모든 실린더 위치에서 균일한 응답이 필요한가? (필요 시 C-SCAN 검토)

### 안티패턴
- **실시간 시스템 (RTOS)**: 마감 시간이 중요한 요청이 헤드 진행 방향 반대편에 생기면 마감 시간을 놓칠 수 있으므로, SCAN보다는 우선순위 기반 스케줄링이 필요하다.

- **📢 섹션 요약 비유**: 모든 학생을 한 번씩은 다 상담해주겠다는 "공평한 선생님"의 전략이지만, 상담 순서가 밀린 학생은 지루할 수 있습니다.

## Ⅴ. 기대효과 및 결론

### 기대효과
- **시스템 안정성**: 기아 현상을 원천적으로 배제하여 시스템의 예측 가능성을 높임.
- **오버헤드 관리**: 알고리즘이 단순하여 스케줄링을 위한 CPU 부하가 적음.

### 미래 전망
- **HDD에서 SSD로**: 물리적 헤드가 없는 SSD에서는 SCAN의 의미가 퇴색되었으나, 논리적 주소 공간에서의 요청 병합(Merging) 및 순서 최적화 원리로 여전히 응답성 향상에 기여한다.
- **V-SCAN으로의 진화**: SCAN의 공정성과 SSTF의 효율성을 동적으로 결합한 알고리즘들이 연구되고 있다.

- **📢 섹션 요약 비유**: 효율과 공정함 사이에서 절묘하게 줄타기를 하는, 운영체제의 "품격 있는 균형추"입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **SCAN**은 엘리베이터가 1층부터 꼭대기까지 올라가면서 타려는 사람들을 다 태우는 것과 같아요.
2. 꼭대기에 도착하면 다시 1층으로 내려오면서 사람들을 태우죠.
3. 아무도 안 태우고 그냥 지나치는 층이 없으니까, 모두가 언젠가는 엘리베이터를 탈 수 있게 된답니다!
