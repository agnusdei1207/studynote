## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RAID 5는 데이터를 블록 단위로 스트라이핑하되, 오류 복구를 위한 패리티(Parity) 정보를 모든 디스크에 골고루 분산하여 저장하는 방식이다.
> 2. **해결**: RAID 4의 치명적 약점이었던 '전용 패리티 디스크의 병목 현상'을 모든 디스크가 책임을 분담함으로써 해결하여 쓰기 성능을 개선하였다.
> 3. **가치**: 디스크 1개의 용량만 희생하면서도 1개의 디스크 장애를 허용하므로, 성능·용량·안정성 사이의 가장 이상적인 균형점으로 평가받아 가장 널리 사용된다.

---

## Ⅰ. 개요 (Context & Background)

- **정의**: 분산 패리티(Distributed Parity) 메커니즘을 사용하여 가용성과 성능을 동시에 확보한 RAID 레벨이다.
- **배경**: RAID 4에서 패리티 디스크 하나에만 쓰기 작업이 몰려 수명이 단축되고 시스템 전체 속도가 저하되는 문제를 극복하기 위해 제안되었다.
- **💡 비유**: **"당번 정하기"**와 같다. 학급 일지를 한 명만 계속 쓰는 대신(RAID 4), 월요일은 A가, 화요일은 B가 돌아가면서 쓰는 것과 같다. 누구 한 명만 유독 힘들지 않고, 누구나 하루씩만 고생하면 학급 전체의 기록(데이터 보호)이 안전하게 유지되는 원리다.

- **📢 섹션 요약 비유**: "모두가 조금씩 짐을 나눠 들어 전체 팀의 행군 속도를 높인 민주적인 방식"입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### RAID 5 분산 패리티 구조 (ASCII Diagram)

```text
  [ 호스트 데이터 저장 구조 ]
           │
  ┌────────▼──────────────────────────────────────────────────┐
  │       [ Disk 1 ]      [ Disk 2 ]      [ Disk 3 ]      │
  │      ┌──────────┐    ┌──────────┐    ┌──────────┐     │
  │ Stripe 1: │  데이터 A1  │    데이터 A2  │  패리티 P(A) │     │ (Parity on D3)
  │      ├──────────┤    ├──────────┤    ├──────────┤     │
  │ Stripe 2: │  데이터 B1  │  패리티 P(B) │  데이터 B2  │     │ (Parity on D2)
  │      ├──────────┤    ├──────────┤    ├──────────┤     │
  │ Stripe 3: │  패리티 P(C) │  데이터 C1  │  데이터 C2  │     │ (Parity on D1)
  │      └──────────┘    └──────────┘    └──────────┘     │
  │                                                           │
  │  * 특징: 패리티 위치가 스트라이프마다 순환하며 변경됨          │
  └───────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터 블록과 패리티 블록이 모든 디스크에 교차하여 배치된다. 특정 디스크 하나가 고장 나더라도, 해당 디스크에 있던 데이터는 나머지 디스크들의 '데이터+패리티' 조합을 XOR 연산하여 즉석에서 복구(On-the-fly)할 수 있다.

- **📢 섹션 요약 비유**: "누가 빠져도 남은 사람들이 퍼즐 조각을 맞춰 원래 그림을 그려낼 수 있는 완벽한 협동 시스템"과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### RAID 5의 특징 상세

| 항목 | 특성 | 비고 |
|:---|:---|:---|
| **최소 디스크 수** | 3개 | |
| **읽기 성능** | 매우 우수 | RAID 0에 근접 (n개 디스크 병렬 읽기) |
| **쓰기 성능** | 중간 | 패리티 연산 및 기록 오버헤드 발생 |
| **공간 효율성** | **(n-1) / n** | 디스크가 많아질수록 효율 증가 |
| **장애 허용** | 1개 가능 | 2개 이상 동시 장애 시 데이터 소실 |

### 쓰기 페널티 (Write Penalty)
- RAID 5에서 데이터를 하나 쓰기 위해서는 '기존 데이터 읽기 -> 기존 패리티 읽기 -> 새 패리티 계산 -> 새 데이터 쓰기 -> 새 패리티 쓰기'라는 복잡한 과정을 거친다. 이를 '4-way I/O'라고 하며, 이 때문에 단일 쓰기 속도는 스트라이핑(RAID 0)보다 느리다.

- **📢 섹션 요약 비유**: "하나를 적을 때마다 힌트(패리티)를 새로 만드느라 펜을 더 많이 움직여야 하는 수고로움"입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **일반 기업용 파일 서버**: 문서, 이미지 등 공유 데이터를 저장할 때 비용 대비 가장 안전하고 빠른 환경을 제공한다.
2. **웹 서버 클러스터**: 웹 콘텐츠 읽기 작업이 많은 환경에서 최고의 가성비를 보여준다.

### 도입 체크리스트
- **리빌딩 시간**: 고용량 HDD(10TB 이상)를 쓸 경우, 장애 복구 시 수일이 걸릴 수 있으며 그 사이 다른 디스크가 고장 나면 파멸적이다. (이 경우 RAID 6 권장)
- **RAID 컨트롤러 캐시**: 쓰기 페널티를 줄이기 위해 하드웨어 RAID 카드의 'Write-back' 캐시와 배터리 백업(BBU)이 필수적이다.

### 안티패턴
- **쓰기 집약적 DB**: 초당 수만 건의 쓰기가 발생하는 로그성 DB를 RAID 5에 두면 쓰기 페널티로 인해 시스템 전체 지연 시간이 증가한다. 이럴 때는 RAID 10이 정답이다.

- **📢 섹션 요약 비유**: "평소에는 훌륭한 팀워크를 자랑하지만, 한 명이 다쳤을 때 나머지 팀원들이 간호하느라 업무 효율이 뚝 떨어지는 상황을 대비해야 합니다."

## Ⅴ. 기대효과 및 결론

### 기대효과
- **TCO 절감**: RAID 1처럼 용량을 절반이나 버리지 않고도 기업용 수준의 데이터 안정성 확보.
- **확장성**: 디스크를 추가할수록 용량과 읽기 성능이 비례하여 증가.

### 미래 전망
- **RAID 6로의 이동**: 디스크 용량이 거대해짐에 따라 리빌딩 중 추가 장애 확률이 높아지면서, 엔터프라이즈 환경에서는 디스크 2개 장애를 버티는 RAID 6가 RAID 5의 자리를 점차 대체하고 있다.
- **Erasure Coding**: 클라우드 저장소에서는 RAID 5의 개념을 논리적으로 확장하여 더 높은 유연성을 가진 오류 정정 코드를 사용한다.

- **📢 섹션 요약 비유**: "스토리지 역사상 가장 화려한 전성기를 누린, 실용주의 공학의 정점"입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **RAID 4 (전용 패리티)**: RAID 5의 직계 조상이자 병목의 원인.
- **RAID 6 (이중 패리티)**: RAID 5보다 안정성을 한 단계 더 높인 모델.
- **Write Back Cache**: RAID 5의 쓰기 페널티를 완화하는 필수 기술.
