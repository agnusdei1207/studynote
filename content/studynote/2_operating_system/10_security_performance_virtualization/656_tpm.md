# TPM (Trusted Platform Module)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TPM (Trusted Platform Module)은 **암호화 코프로세서(Cryptographic Coprocessor)가 내장된 하드웨어 칩**으로, 시스템 부팅 단계부터 시작되는 **신뢰의 뿌리(RoT, Root of Trust)**를 제공하여 보안 키 생성, 저장, 디지털 서명, 플랫폼 무결성 측정을 수행한다.
> 2. **가치**: 소프트웨어 기반 보안의 취약점을 극복하여, **키 추출 불가(Non-exportable)** 구조로 해킹 시도 차단; **BitLocker (Windows)**, **LUKS (Linux)** 등의 **FDE (Full Disk Encryption)** 솔루션과 결합하여 OS 레벨의 데이터 보안을 완성한다.
> 3. **융합**: TPM 1.2 대비 향상된 TPM 2.0 표준은 **ECC (Elliptic Curve Cryptography)**와 **SHA-256 (Secure Hash Algorithm-256)**을 지원하며, **Intel PTT (Platform Trust Technology)**, **AMD fTPM (Firmware TPM)**等形式으로 CPU에 통합되어 하드웨어 가상화와 시스템 관리의 효율성을 높인다.

+++

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 정의**
TPM (Trusted Platform Module)은 **TCG (Trusted Computing Group)**가 주도하는 국제 표준 **ISO/IEC 11889**를 따르는 **보안 부표(Secure Cryptoprocessor)**이다. 이는 마더보드나 CPU에 내장되어, 일반 운영체제(OS)나 응용 프로그램의 영향을 받지 않는 독립적인 **보안 실행 환경(Trusted Execution Environment)**을 제공한다. TPM의 핵심 역할은 시스템의 **무결성(Integrity)**을 보장하고, 암호화 키를 하드웨어적으로 격리하여 보관하는 것이다.

**2. 등장 배경과 필요성**
과거의 보안 모델은 소프트웨어 방식의 방화벽이나 백신에 의존하였으나, 이는 OS 자체가 타격을 입으면 무력화된다는 근본적인 한계가 있다. 해커가 OS 루트 권한을 탈취하면 메모리상의 모든 키를 읽어낼 수 있기 때문이다. 이를 해결하기 위해 **"소프트웨어가 아무리 손상되어도 건드릴 수 없는 하드웨어적 금고"**라는 패러다임이 등장했다. 현재는 **Microsoft Secured-Core PC**, **Zero Trust (영위도 신뢰 모델)** 보안의 필수 요소로 자리 잡았다.

**💡 비유: 일상생활**
TPM은 마치 **"자물쇠가 내장된 불가항력의 금고"**와 같다. 컴퓨터라는 집(프로그램들)이 도둑(바이러스)에게 침입당해 내부를 엉망으로 만들더라도, 바닥에 단단히 박힌 이 금고(TPM) 안에 든 비밀 키(열쇠)만은 절대 꺼내갈 수 없다.

> **📢 섹션 요약 비유:**
> 부실한 초인종(소프트웨어 보안)을 고치는 것이 아니라, 문 구조 자체를 바꿔서 문이 열려도 집 안의 금고(TPM)는 열리지 않게 만드는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. TPM 내부 구성 요소**
TPM은 단순한 저장소가 아니라 독립적인 연산 능력을 가진 마이크로컨트롤러이다. 주요 구성 요소는 다음과 같다.

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 (Internal Operation) | 프로토콜/규격 (Spec) | 비유 (Analogy) |
|:---|:---|:---|:---|:---|
| **ECC (Endorsement Key)** | 칩 고유 ID | 출하 시 생성되어 변경 불가, PKI 인증서 사용 | RSA 2048-bit | 주민등록번호 |
| **SRK (Storage Root Key)** | 키 계층 최상위 루트 | 사용자 키들을 암호화하여 저장(Hierarchical Key Storage) | TPM Storage Hierarchy | 마스터 키 |
| **PCR (Platform Configuration Register)** | 무결성 측정 저장소 | 해시(Hash) 체인으로 부팅 과정 상태 기록 (SHA-256) | CRTM (Core Root of Trust for Measurement) | 보안 일지 |
| **HMAC Engine** | 데이터 인증 | 메시지 무결성 검증 및 승인 | HMAC-SHA256 | 서명 확인 |
| **RNG (Random Number Generator)** | 엔트로피 공급 | 하드웨어 노이즈 기반 진성 난수 생성 | FIPS 140-2 | 주사위 |

**2. TPM 아키텍처 다이어그램**
아키텍처는 크게 **호스트(CPU/OS)**와 **TPM 칩** 간의 통신, 그리고 내부적인 **키 계층 구조**로 나뉜다.

```text
      [ Host Computer System ]
      +----------------------+
      |      CPU / OS        |
      |  (Untrusted State)   |
      +---------+------------+
                |
                | TPM Interface (SPI / LPC / I2C)
                | v
      +---------v------------+      +--------------------------+
      |   TPM Driver (TSS)   | <--> |   TPM Chip Hardware      |
      +----------------------+      | +----------------------+ |
                                   | | Crypto Engine (RSA/ECC)|
                                   | | +--------------------+ |
                                   | | | RNG (Entropy)      | |
                                   | | +--------------------+ |
                                   | | +--------------------+ |
                                   | | | Key Storage (NVRAM) | |
                                   | | | - SRK (Root)        | |
                                   | | | - AIK (Identity)    | |
                                   | | | - Derived Keys      | |
                                   | | +--------------------+ |
                                   | | +--------------------+ |
                                   | | | PCR (0~23)         | |
                                   | | |[SHA-256 Registers] | |
                                   | | +--------------------+ |
                                   | +----------------------+ |
                                   +--------------------------+
```
*(도해 해설: CPU는 TPM Command를 전송하고 TPM은 이를 처리하여 Response를 돌려준다. 키 저장소와 PCR은 물리적 분리되어 있어 외부 메모리 덤프로 유출되지 않는다.)*

**3. 심층 동작 원리: 측정(Measurement)과 인증(Attestation)**
TPM의 가장 강력한 기능은 **Remote Attestation (원격 증명)**이다. 이는 **CRTM (Core Root of Trust for Measurement)**이 BIOS가 로드되기 직전에 해시 값을 계산하며 시작된다.

1.  **Power On**: 전원이 들어오면 BIOS 코드의 Hash를 **PCR[0]**에 저장.
2.  **Boot Loader**: 부트로더가 로드되면 그 해시 값을 PCR[0] 값과 연산(Extend)하여 **PCR[0]**을 갱신.
3.  **OS Kernel**: 커널 이미지가 로드되면 **PCR[1]** 등에 기록.
4.  **Attestation**: 외부 검증자(Verifier)가 "현재 PCR 값이 안전한 상태인가?"를 물으면, TPM은 자신의 **AIK (Attestation Identity Key)**로 PCR 값들에 대해 디지털 서명을 하여 증명.
    *   만약 악성 코드가 BIOS에 침투했다면 PCR 값이 변하므로, 안전한 서명을 생성할 수 없거나(Seal) 비밀 해제가 불가능해진다(Unseal Fail).

**4. 핵심 알고리즘 및 코드 예시**
TPM 2.0에서는 **ECC (Elliptic Curve Cryptography)**를 적극 활용하여 키 크기 대비 높은 보안성을 제공한다.
아래는 리눅스 환경에서 TPM을 제어하는 `tpm2-tools`를 활용한 키 생성 및 봉인(Sealing) 과정의 실무 예시이다.

```bash
# 1. TPM 소유자 권한 초기화 (Hierarchy Authorization)
tpm2_takeownership -c

# 2. ECC 타입의 기본 키(Primary Key) 생성 (부모 키)
tpm2_createprimary -c primary.ctx

# 3. 사용자 키 생성 (Sealing용 자식 키)
# 데이터를 특정 PCR 상태에서만 해제되도록 설정하여 'Sealing'
tpm2_create -C primary.ctx -p passwd -u key.pub -r key.priv \
  -L "sha256:0,1,2+sha256:3"  # PCR 0,1,2는 결합, 3은 단독 지정 예시

# 4. TPM의 NVRAM에 키 로드
tpm2_load -C primary.ctx -u key.pub -r key.priv -c key.ctx

# 5. 데이터 봉인 (Seal): my_secret.txt를 TPM 키로 암호화
echo "Top Secret Data" | tpm2 seal -c key.ctx -o sealed.dat

# 6. 데이터 해제 (Unseal): PCR 상태가 일치할 때만 복호화 성공
tpm2 unseal -c key.ctx -i sealed.dat -p passwd
```

> **📢 섹션 요약 비유:**
> 마치 고속도로 톨게이트의 **"블랙박스"**와 같습니다. 어떤 차량(프로그램)이 지나갔는지, 어디서 멈췄는지, 속도위반은 없었는지를 계산된 해시값(일지)으로 남겨서, 나중에 이 차량이 사고(해킹)를 냈는지 정확히 판단할 수 있게 해줍니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 기술 표준 비교: TPM 1.2 vs TPM 2.0**
TPM 2.0은 단순한 버전업이 아닌, 알고리즘 유연성을 극대화한 구조적 혁신이다.

| 비교 항목 (Criteria) | TPM 1.2 (Legacy) | TPM 2.0 (Current Standard) | 비고 (Note) |
|:---|:---|:---|:---|
| **지원 알고리즘** | RSA (SHALL), SHA-1 (취약) | RSA, **ECC (MUST)**, SHA-256, Camellia | ECC는 작은 키 크기로 높은 안전성 제공 |
| **키 계층 구조** | 단일 계층 (TCS용) | **Hierarchical (Platform/Endorsement/Owner)** | 다중 테넌시 및 다중 사용자 지원 |
| **인증 방식** | OIAP/OSAP (복잡함) | 속성 기반 인증 등 단순화된 세션 관리 | |
| **국제 암호 표준** | 미국(CNSA) 중심 | **국제 표준(ISO/IEC)** 준수 (중국/러시아 알고리즘 지원) | 글로벌 호환성 확보 |

**2. OS/하드웨어 융합 및 시너지**
TPM은 단독으로 존재하지 않으며 OS 및 하드웨어 보안 기술과 시너지를 낸다.

*   **Windows & BitLocker**: **BitLocker Drive Encryption (BDE)**는 TPM을 **VK (Volume Master Key)**의 보안 저장소로 사용한다. 시스템이 부팅될 때 TPM은 PCR 값을 확인하고, 정상 부팅임이 확인되면 VMK를 해제하여 디스크를 잠금 해제한다. 이 과정에서 사용자 개입 없이 자동으로 보안 부팅이 이루어진다.
*   **Linux & LUKS**: LUKS (Linux Unified Key Setup)는 `tpm2-tools`와 연동하여 키 material을 TPM 내부에 봉인(Seal)할 수 있다. `/etc/crypttab` 설정을 통해 `tpm2-pkcs11` 모듈을 호출하는 방식이 주로 사용된다.
*   **Intel PTT & AMD fTPM**: 별도의 칩을 사용하는 dTPM (Discrete TPM)과 달리, CPU 내의 **ME (Management Engine)** 또는 **PSP (Platform Security Processor)** 영역을 사용하는 SPI Flash 기반의 펌웨어 TPM이 존재한다. 비용 절감 효과가 있으나, CPU 취약점(예: Spectre/Meltdown)과 연계될 가능성이 있는 트레이드오프가 있다.

> **📢 섹션 요약 비유:**
> TPM 2.0은 **"국제 공인이 가능한 다중 언어 통역기"**가 된 것과 같습니다. 1.2는 내부 사정만 통하는 소규모 부서였다면, 2.0은 전 세계(알고리즘)와 호환되고 권한 등급(계층)을 세분화한 글로벌 보안 센터로 발전했습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**1. 실무 시나리오 및 의사결정**
*   **시나리오 1: 직원 노트북 분실 도난 사고**
    *   **대응**: TPM이 없는 경우 하드디스크를 탈착하여 데이터를 읽을 수 있다. 하지만 TPM + BitLocker가 적용된 경우, 하드디스크만 분리해서는 **VMK(키)**가 TPM 칩 내부에 봉인되어 있으므로 데이터 복호화가 불가능하다.
    *   **결정**: 기업급 자산 반드시 TPM 2.0 + FDE(전체 디스크 암호화) 활성화 기본 정책 적용.

*   **시나리오 2: DevOps 파이프라인 무결성 보장 (Cloud/Server)**
    *   **대응**: 클라우드 환경(AWS Nitro Enclaves, Azure Trusted Launch 등)에서 부팅 시 악성코드가 삽입되지 않았음을 보장해야 할 때, PCR 값을 로그에 남겨 감사(audit) 자료로 활용한다.
    *   **결정**: 서버 부팅 스크립트 내 PCR 값 검증 로직 추가.

*   **시나리오 3: OTP/인증서 저장소 활용**
    *   **대응**: 은행권 보안 인증서(T-Bridge 등)를 하드디스크가 아닌 TPM에 저장하여, 악성코드가 키를 복사해가더라도 다른 기기에서는 사용 불가능하게 만든다.
    *   **결정**: 공인인증서/고가용 키는 TPM 저장 권장.
