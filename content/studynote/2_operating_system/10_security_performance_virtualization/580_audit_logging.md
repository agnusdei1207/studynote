# 감사 및 로깅 (Audit & Logging)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 감사(Audit)와 로깅(Logging)은 시스템 이벤트를 **기록하고 검토하여 무슨 일이 일어났는지, 언제, 누가, 어떻게** 추적하는 보안의 필수 요소다.
> 2. **가치**: 보안 사고 발생 시 **원인 규명, 피해 범위 파악, 법적 증거 확보**가 가능하며, 사전에 비정상 행동을 탐지하는 **모니터링**의 기반이 된다.
> 3. **융합**: Syslog, ELK Stack, Splunk, SIEM(Security Information and Event Management)과 결합하여 **실시간 모니터링 및 포렌식**을 실현한다.

+++

## Ⅰ. 감사 및 로깅의 개요

### 1. 정의
- **로깅(Logging)**: 시스템 이벤트, 오류, 상태 변화를 기록하는 활동
- **감사(Auditing)**: 기록된 로그를 분석하여 정책 준수 여부, 보안 위협을 확인하는 활동

### 2. 등장 배경: "무슨 일이 있었지?"
- "시스템이 왜 죽었지?", "누가 이 파일에 접근했지?", "언제 해킹당했지?"
- 이런 질문에 답하려면 이벤트 기록 필수

### 3. 💡 비유: 'CCTV와 기록부'
- 로깅은 **'CCTV로 영상 녹화'**와 같고, 감사는 **'기록부를 검토'**하는 것과 같다.
- 사건이 발생했을 때 누가, 언제, 무엇을 했는지 확인 가능.

- **📢 섹션 요약 비유**: 매장의 CCTV(로깅)는 계속 녹화하고, 점장(감사)은 나중에 영상을 보고 도둑을 잡습니다. 둘 다 있어야 범죄를 예방하고 해결할 수 있어요.

## Ⅱ. 로그의 구조 및 종류 (Deep Dive)

### 1. 로그 메시지 구조
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │                   표준 로그 메시지 구조                          │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [Timestamp] 시간                                               │
    │   → 언제 발생했는가? (2026-03-16T14:23:01.123Z)                 │
    │                                                                 │
    │  [Hostname/Source] 출처                                         │
    │   → 어디서 발생했는가? (server01.example.com)                   │
    │                                                                 │
    │  [Severity] 심각도                                              │
    │   → 얼마나 심각한가? (DEBUG/INFO/NOTICE/WARN/ERROR/CRIT)        │
    │                                                                 │
    │  [Process/Service] 프로세스                                     │
    │   → 무엇이 발생했는가? (sshd[1234])                            │
    │                                                                 │
    │  [Message] 메시지                                                │
    │   → 무슨 내용인가? (Failed password for root from 192.168.1.100)│
    │                                                                 │
    │  [Structured Fields] 구조화 필드 (선택)                          │
    │   → JSON/Key-Value 형태의 추가 정보                             │
    │   → user=admin, src_ip=10.0.0.1, action=file_delete             │
    │                                                                 │
    │  예시: Mar 16 14:23:01 server01 sshd[1234]: Failed password     │
    │       for root from 192.168.1.100 port 22 ssh2                 │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. 로그의 종류
| 로그 유형 | 설명 | 예시 |
|:---|:---|:---|
| **시스템 로그** | 커널, 시스템 서비스 이벤트 | /var/log/messages, syslog |
| **인증 로그** | 로그인, 인증 시도 | /var/log/auth.log, wtmp |
| **애플리케이션 로그** | 애플리케이션 동작, 오류 | Apache access.log, error.log |
| **감사 로그** | 보안 관련 이벤트 | Linux Auditd, Windows Event Log |
| **네트워크 로그** | 방화벽, 라우터 이벤트 | iptables logs, NetFlow |
| **성능 로그** | 리소스 사용량 | top, vmstat, sar |

### 3. Linux 주요 로그 파일
```bash
/var/log/syslog       # 일반 시스템 로그 (Debian/Ubuntu)
/var/log/messages     # 일반 시스템 로그 (RHEL/CentOS)
/var/log/auth.log     # 인증 관련 (Debian/Ubuntu)
/var/log/secure       # 인증 관련 (RHEL/CentOS)
/var/log/kern.log     # 커널 로그
/var/log/boot.log     # 부팅 로그
/var/log/dmesg        # 부팅 시 커널 메시지
```

- **📢 섹션 요약 비유**: 병원에 차트(환자 기록), 경찰에 사건 일지, 학교에 출석부가 있듯이, 시스템에도 각각의 기록장(로그 파일)이 있습니다.

## Ⅲ. 감사 시스템 (Audit System)

### 1. Linux Auditd
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │                Linux Auditd 구조                                 │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │   [Userspace]           [Kernel Space]                          │
    │                                                                 │
    │   auditd ──────▶  /var/log/audit/audit.log                      │
    │      │                                                         │
    │      │ 설정                                                   │
    │      ▼                                                         │
    │   audispd ──▶  Dispatcher (외부 시스템으로 전송)               │
    │                                                                 │
    │   ausearch   ──▶  로그 검색                                    │
    │   aureport   ──▶  보고서 생성                                  │
    │                                                                 │
    │              ┌──────────────────┐                              │
    │              │   Audit Rules    │                              │
    │              │   (auditctl)     │                              │
    │              │  -w /etc/passwd  │                              │
    │              │  -p wa           │                              │
    │              └──────────────────┘                              │
    │                       │                                        │
    │                       ▼                                        │
    │              [Kernel Audit Subsystem]                           │
    │              - 시스템 콜 감지                                   │
    │              - 파일 접근 감시                                   │
    │              - 규칙 매칭                                       │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. Auditd 예시
```bash
# /etc/passwd 감시 (쓰기, 속성 변경)
auditctl -w /etc/passwd -p wa -k passwd_changes

# 규칙 확인
auditctl -l

# 로그 검색
ausearch -k passwd_changes

# 보고서
aureport --auth
```

### 3. 감사 추적 (Audit Trail)
- **Chain of Custody**: 로그의 무결성 보장
- **Tamper-evident**: 로그 변경 시 감지 가능
- **Non-repudiation**: 행위자가 행동을 부인 못 함

## Ⅳ. 로그 수집 및 분석

### 1. 중앙 집중식 로깅
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │                 중앙 집중식 로깅 아키텍처                         │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [로그 생성자]                                                  │
    │   ┌───────┐  ┌───────┐  ┌───────┐  ┌───────┐                  │
    │   │Server1│  │Server2│  │Server3│  │Server4│                  │
    │   └───┬───┘  └───┬───┘  └───┬───┘  └───┬───┘                  │
    │       │         │         │         │                         │
    │       └─────────┴─────────┴─────────┘                         │
    │                     │ Syslog (514/UDP, 6514/TCP)               │
    │                     ▼                                          │
    │  [로그 수집기]                                                  │
    │   ┌──────────────────────────────────────────┐                │
    │   │          rsyslog / syslog-ng             │                │
    │   └───────────────┬──────────────────────────┘                │
    │                   │                                            │
    │                   ▼                                            │
    │  [로그 저장 및 분석]                                            │
    │   ┌──────────────────────────────────────────┐                │
    │   │       ELK Stack / Splunk / Graylog      │                │
    │   │   - Elasticsearch (저장)                 │                │
    │   │   - Logstash (파이프라인)                │                │
    │   │   - Kibana (시각화)                      │                │
    │   └──────────────────────────────────────────┘                │
    │                                                                 │
    │  [SIEM]                                                         │
    │   - 상관 분석 (Correlation)                                     │
    │   - 알림 (Alerting)                                            │
    │   - 대시보드 (Dashboard)                                       │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. SIEM (Security Information and Event Management)
- **기능**: 로그 수집, 상관 분석, 위협 탐지, 대시보드, 보고서
- **예시**: Splunk, QRadar, LogRhythm, Wazuh(오픈소스)
- **Rule Engine**: 특정 패턴 감지 시 알림
  - 예: "1분 내 5회 실패 로그인" → 계정 잠금

## Ⅴ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 로그 관리 모범 사례
1. **충분한 정보 기록**: WHO, WHAT, WHEN, WHERE, WHY
2. **구조화된 로그**: JSON, Key-Value 형태
3. **중앙 집중**: 분산된 로그를 한 곳으로
4. **보호**: 로그 무결성(서명, WORM), 접근 제어
5. **보존 기간**: 규정 준수 (일반적으로 1~7년)
6. **정기 검토**: 로그 리뷰, 이상 징후 조사

### 2. 안티패턴
- **"로그 안 남김"**: 사고 시 조사 불가
- **"로그 지우기"**: 파기 정책 있어야 함, 무단 삭제는 위법
- **"과도한 로그"**: 성능 저하, 저장소 낭비
- **"로그 무시"**: 수집만 하고 분석 안 함
- **"시간 동기화 안 함"**: NTP 설정 필수

### 3. Compliance 요구사항
| 규정 | 로그 요구사항 |
|:---|:---|
| **PCI-DSS** | 6개월 보관, 감사 추적 |
| **HIPAA** | 6년 보안, 접근 로그 |
| **GDPR** | 동의/거부 기록, 데이터 처리 로그 |
| **SOX** | 재무 관련 시스템 감사 |
