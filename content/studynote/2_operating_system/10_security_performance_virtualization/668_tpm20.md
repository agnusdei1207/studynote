# TPM 2.0 (Trusted Platform Module 2.0)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: **TPM 2.0 (Trusted Platform Module 2.0)**은 기존 TPM 1.2의 알고리즘 편향성(RSA/SHA-1)과 구조적 경직성을 극복하기 위해 **TCC (Trusted Computing Group)**가 정의한 차세대 하드웨어 신뢰 루트(Root of Trust) 표준이다.
> 2. **가치**: 암호화 알고리즘의 모듈화와 **ECC (Elliptic Curve Cryptography)** 지원을 통해 동일한 보안 강도에서 키 크기와 연산 오버헤드를 획기적으로 줄였으며, 이는 IoT 및 모바일 환경의 **Low Latency** 보안 통신을 가능하게 한다.
> 3. **융합**: OS 부팅 단계에서의 무결성 검증(Measured Boot), **BitLocker** 같은 FDE (Full Disk Encryption) 키 저장, 그리고 **fTPM (firmware TPM)** 형태로 SoC (System on Chip)에 통합되어 가상화 보안의 기반이 된다.

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 정의
**TPM (Trusted Platform Module)**은 메인보드나 CPU에 내장된 보안 코프로세서(Secure Cryptoprocessor)로, 시스템의 부팅 과정부터 실행 권한, 데이터 암호화 키까지를 하드웨어적으로 보호하는 **신뢰 실행 환경(Trusted Execution Environment)**의 물리적 근거이다. TPM 2.0은 2014년에 발표된 표준으로, 암호화 알고리즘의 독점성을 깨고 범용성을 확보한 것이 가장 큰 특징이다.

#### 2. 💡 비유: '교차 검증이 가능한 스마트 금고'
기존 TPM 1.2가 "특정 제조사의 열쇠로만 열리는 강철 금고"라면, TPM 2.0은 **"열쇠의 형태를 마음대로 바꿀 수 있고, 누가 언제 열었는지 블록체인처럼 기록하는 스마트 금고"**와 같다.

#### 3. 등장 배경 및 기술적 패러다임 변화
- **기존 한계 (TPM 1.2)**: 오직 RSA와 SHA-1 알고리즘에만 의존하여, 양자 컴퓨팅 시대의 보안 취약성과 미국 수출 규제(이미 RSA 제한) 문제가 있었음. 또한 명령어 구조가 복잡하여 임베이드 시스템에 적합하지 않았음.
- **혁신적 패러다임**: **알고리즘 독립성(Algorithm Independence)**. 암호화 알고리즘을 소프트웨어적으로 교체 가능한 모듈 구조로 변경하여, 향후 신규 알고리즘(Lattice-based 등)의 대응이 용이하도록 설계됨.
- **현재 비즈니스 요구**: **Windows 11**의 출시와 함께 TPM 2.0은 필수 하드웨어 요구사양(Spec)이 되었으며, Zero Trust 네트워크 환경에서 디바이스 신원 증명(Device Attestation)의 핵심 요소로 자리 잡음.

#### 4. 하드웨어 형태별 분류 (ASCII 다이어그램)
TPM 2.0은 구현 형태에 따라 성능과 비용, 보안성의 트레이드오프가 발생한다.

```text
+------------------+       +------------------+       +------------------+
|  Discrete TPM    |       |    fTPM / SOC    |       |   vTPM (Virtual) |
|  (H/W Chip)      |       |  (Firmware based)|       |  (Hypervisor)    |
+------------------+       +------------------+       +------------------+
| ▶ Physical Isolation     | ▶ CPU Integrated (e.g. PTT) | ▶ Software Emulated |
| ▶ Highest Security       | ▶ Low Cost, Power Efficient | ▶ Multi-tenant Support |
| ▶ Slower (SPI/LPC)       | ▶ Shared Memory Risk       | ▶ Dependent on Hypervisor |
+------------------+       +------------------+       +------------------+
        ^                           ^                           ^
        | (Server/High Security)    | (Laptop/IoT/Modern PC)    | (Cloud VM Instance)
```
*(해설)*
1.  **Discrete TPM**: 독립 칩 형태로 물리적 격리가 가장 완벽하나, 통신 버스(LPC/SPI) 지연으로 인해 속도가 상대적으로 느리며 비용이 높음.
2.  **fTPM (firmware TPM)**: CPU의 일부(예: Intel PTT, AMD fTPM)로 펌웨어화된 형태. 저전력/저비용이나 DMA(Direct Memory Access) 공격 등 메모리 취약점에 대한 방어가 필요함.
3.  **vTPM (Virtual TPM)**: 가상화 환경에서 각 VM(Virtual Machine)에 할당되는 논리적 TPM. Hypervisor가 자원을 분리하여 관리하며 클라우드 환경의 필수 요소임.

#### 5. 섹션 요약 비유
> 📢 **섹션 요약 비유**: 단순히 단단한 문짝 하나를 달아놓은 것(RSA)이 아니라, 상황에 따라 지문인식, 홍채인식, 패턴 등 다양한 잠금장치(알고리즘)를 교체해서 사용할 수 있는 만능 보안 시스템을 건물 설계 단계부터 심어놓은 것입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 구성 요소 (표)
TPM 2.0은 내부적으로 히어러키(Hierarchy) 구조를 통해 키를 분리 관리한다.

| 요소명 (Component) | 역할 (Role) | 내부 동작 (Internal Behavior) | 주요 프로토콜/알고리즘 | 비유 (Analogy) |
|:---|:---|:---|:---|:---|
| **EK (Endorsement Key)** | 디바이스 고유 ID | 제조사(CA)가 싸인한 불변의 인증서. TPM 소유권 증명에 사용. | RSA-2048 / ECC-P256 | 디바이스의 '주민등록번호' |
| **SRK (Storage Root Key)** | 저장소 계층의 루트 | 사용자 소유의 상위 키. 디스크 암호화 키를 생성/보호하는 부모 키. | TPM 생성 계층 | '자물쇠 공장의 장인' |
| **AK (Attestation Key)** | 신원 증명 키 | 플랫폼의 무결성 상태를 증명할 때 사용되는 일시적 키. | AIK 신원 증명 ID | '일회용 출입 카드' |
| **PCR (Platform Configuration Register)** | 무결성 측정 | 부팅 각 단계(UEFI, OS, Driver)의 해시값을 누적 저장. | SHA-256 Extend | **'부팅 로그의 블록체인'** |
| **NV Storage** | 비휘발성 메모리 | 키와 인증서를 영구 저장하는 영역. 카운터나 비밀 데이터 보관. | NVRAM | '비밀 금고 창고' |

#### 2. TPM 2.0 레지스터 및 히에라르키 구조 (ASCII)
아래 다이어그램은 TPM 2.0의 키 관리가 계층적으로 이루어지며, 각 영역(Platform, Owner, Endorsement)이 격리되어 있음을 보여준다.

```text
     [ 하드웨어 영역 : TPM Chip (SoC / Discrete) ]
     ═══════════════════════════════════════════════════
                    ┌───────────────────────┐
                    │  EK (Endorsement Key)  │ ◄─── [제조사 할당, 불변]
                    │   (Root of Trust ID)   │
                    └───────────┬───────────┘
                                │ 1. 증명(Attestation)
                                ▼
      ┌─────────────────────────────────────────────────────┐
      │              TPM 2.0 Engine (Cryptographic)         │
      │  ┌──────────────────┐      ┌───────────────────┐   │
      │  │   RSA Engine     │      │   ECC Engine      │   │
      │  │ (Sign/Decrypt)   │      │ (Sign/Key Agree)  │   │
      │  └──────────────────┘      └───────────────────┘   │
      └─────────────────────────────────────────────────────┘
                                │ 2. 래핑(Wrapping)
                                ▼
                    ┌───────────────────────┐
                    │  SRK (Storage Root)   │ ◄─── [사용자 생성, Owner 권한]
                    └───────────┬───────────┘
                                │
            ┌───────────────────┼───────────────────┐
            ▼                   ▼                   ▼
       ┌─────────┐         ┌─────────┐         ┌─────────┐
       │ Data Key│         │  AIK    │         │ Sealed  │
       │ (BitLocker)       │ (Identity)│       │  Data   │
       └─────────┘         └─────────┘         └─────────┘
       
     ▶ [PCR Registers] : SHA-256 Hash Chain
        [PCR-0]: CRT/UEFI Code     -> Extend(Hash)
        [PCR-1]: Host Platform Config
        [PCR-2]: Option ROMs
        ...
        [PCR-7]: OS Loader
     ═══════════════════════════════════════════════════
```
*(해설)*
1.  **루트 분리**: EK(제조사 영역)와 SRK(사용자 영역)가 논리적으로 분리되어 있다. 이는 프라이버시 보호를 위해 사용자가 EK를 직접 사용하는 것을 방지하는 구조다.
2.  **알고리즘 이중화**: TPM 2.0은 기존 방식인 RSA 엔진과 함께 키 사이즈는 작지만 보안 강도가 높은 ECC 엔진을 내장하여, 모바일/IoT 환경에서의 **전력 소모량**을 최적화한다.
3.  **PCR의 확장**: 부팅 시각(Boot Time)마다 코드의 해시 값을 PCR에 `Extend(append)` 연산하여 저장하므로, 해커가 부트 로더를 변조하더라도 PCR 값이 달라져 시스템이 잠기게 된다.

#### 3. 심층 동작 원리 (Remote Attestation)
TPM 2.0의 가장 강력한 기능인 원격 증명(Remote Attestation)은 4단계를 거쳐 신뢰를 확보한다.

1.  **Challenge**: 검증자(Verifier)가 디바이스에 무결성 증명을 요청하는 Nonce(난수)를 전송.
2.  **Quote**: TPM이 현재 PCR 값들을 **AK (Attestation Identity Key)**로 서명(Sign). 이 서명은 EK 체인을 통해 신뢰성을 보장받음.
3.  **Verification**: 검증자는 서명을 EK 인증서로 검증하고, PCR 값이 기준값(Golden Value)과 일치하는지 확인.
4.  **Sealing/Unsealing**: 특정 PCR 값이 일치할 때만 데이터를 복호화할 수 있는 기능으로, 랜섬웨어 방지에 활용됨.

#### 4. 핵심 알고리즘 변화 (코드/수식)
TPM 2.0의 가장 큰 수학적 변화는 RSA 기반에서 벗어나 타원곡선 기반 암호(ECC)를 도입한 것이다.

**RSA vs ECC (Security Equivalence)**
```text
Key Size (bits)  |  Security Strength (bits) |  Algorithm Type
-----------------|---------------------------|------------------
1024             |  80                       |  RSA (Legacy, Deprecated)
2048             |  112                      |  RSA (TPM 1.2 Standard)
3072             |  128                      |  RSA (High Security)
256              |  128                      |  ECDSA/ECDH (TPM 2.0 Standard)
```
*   **분석**: TPM 2.0은 **ECC (Elliptic Curve Cryptography)**를 통해 2048비트 RSA와 동일한 보안 수준을 단 256비트 키로 달성한다. 이는 **TPM Command**의 패킷 크기를 줄이고, 암호화 연산 시 **CPU Cycles**를 획기적으로 절감하여 배터리 수명을 연장한다.

#### 5. 섹션 요약 비유
> 📢 **섹션 요약 비유**: TPM 2.0의 아키텍처는 거대한 '수퍼마켓'의 관리 체계와 같습니다. **EK**는 상점 고유의 '사업자등록번호'이고, **SRK**는 점장이 관리하는 '마스터 키', **PCR**은 매일매일 손님들이 입장하고 물건이 팔리는 기록이 쌓이는 '자동 기록지(CCTV)'와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. TPM 1.2 vs TPM 2.0 심층 기술 비교

| 비교 항목 | TPM 1.2 (Legacy) | TPM 2.0 (Current) | 비고 (Note) |
|:---|:---|:---|:---|
| **알고리즘 지원** | RSA *only*, SHA-1 | RSA, **ECC**, **SHA-256**, Camellia, HMAC | 국내 암호법(양암호) 호환 가능 |
| **키 구조** | 단일 계층 (Rigid) | **다중 계층 (Hierarchical)** <br> (Platform/Owner/Endorsement) | 권한 관리의 세분화 |
| **국제 호환성** | 미국/서구 위주 | **ISO/IEC 11889** 표준 채택 (국제화) | 각국 수출 규제 문제 해결 |
| **데이터 크기** | 큰 패킷 (Key size에 따름) | **최적화된 패킷** (ECC 사용 시) | 네트워크 오버헤드 감소 |
| **가용성** | OS별 드라이버 의존적 | **OS 독립적 플랫폼** (TBS) | Windows, Linux, macOS 동일 API |
| **신원 증명** | Privacy CA 복잡 | **Direct Anonymous Attestation (DAA)** 지원 강화 | 프라이버시 보호 강화 |
