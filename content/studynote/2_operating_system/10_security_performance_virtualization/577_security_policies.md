# 보안 정책 (Security Policies)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보안 정책(Security Policy)은 시스템이 **보호해야 할 자산과 허용할 동작을 명시한 규칙 집합**이며, 보안 모델은 이를 수학적으로 형식화한 것이다.
> 2. **가치**: "무엇이 보안 동작인가?"에 대한 기준을 제공하여 **보안 메커니즘 설계, 구현, 검증**의 방향성을 제시하고 규정 준수의 기준이 된다.
> 3. **융합**: 참조 모니터(Reference Monitor) 개념, TCB(Trusted Computing Base), 보안 커널 등 하드웨어/소프트웨어 계층에서 정책을 강제한다.

+++

## Ⅰ. 보안 정책의 개요

### 1. 정의
- 보안 정책은 시스템의 리소스에 대한 접근 규칙, 사용자 의무, 보안 요구사항을 정의한 문서/규칙이다.
- **보안 모델**: 보안 정책을 수학적으로 형식화하여 검증 가능한 모델

### 2. 등장 배경: 보호 vs 보안의 간극
- "보호 메커니즘을 구현했지만, 정확히 구현했는지 어떻게 확인?"
- 보안 정책을 먼저 정의하고, 이를 만족하는지 검증 필요

### 3. 💡 비유: '학교 교칙과 학생회 규정'
- 보안 정책은 **'학교 교칙(무엇을 해야 한다/안 한다)'**과 같다.
- 보안 모델은 **'교칙을 구체적으로 해석한 학생회 규정'**이며, 보안 메커니즘은 **'교사가 교칙을 지키게 하는 방법'**이다.

- **📢 섹션 요약 비유**: 건물을 지울 때 "어떤 자재를 쓰고, 어떤 설계를 따를지" 정하는 설계도와 같습니다.

+++

## Ⅱ. 보안 정책의 구성 요소 (Deep Dive)

### 1. 보안 정책의 주요 구성
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │                  보안 정책의 구성 요소                           │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [1] 보호된 개체 (Protected Objects)                            │
    │   - 어떤 리소스를 보호할 것인가?                               │
    │   - 예: 파일, 메모리, 프로세스, 네트워크 포트                    │
    │                                                                 │
    │  [2] 허용된 동작 (Permitted Operations)                         │
    │   - 어떤 동작을 허용/거부할 것인가?                            │
    │   - 예: 읽기(OK), 쓰기(거부), 실행(OK)                         │
    │                                                                 │
    │  [3] 주체 (Subjects)                                            │
    │   - 누가 접근할 수 있는가?                                     │
    │   - 예: 사용자, 프로세스, 역할                                  │
    │                                                                 │
    │  [4] 제어 조건 (Control Conditions)                             │
    │   - 언제/어디서/어떤 상황에서?                                  │
    │   - 예: 업무 시간만, 사무실 내에서만                            │
    │                                                                 │
    │  [5] 위반 시 조치 (Violation Actions)                           │
    │   - 규칙 위반 시 어떻게 할 것인가?                             │
    │   - 예: 로그, 차단, 관리자 통보                                 │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. 정책 예시
| 정책 유형 | 규칙 예시 |
|:---|:---|
| **신원 정책** | "모든 사용자는 고유 ID로 식별되어야 한다" |
| **접근 정책** | "급여 파일은 HR 부서만 읽기 가능" |
| **기밀성 정책** | "기밀 문서는 암호화되어야 한다" |
| **책임성 정책** | "모든 접근은 로그에 기록된다" |

## Ⅲ. 대표적인 보안 모델

### 1. Bell-LaPadula 모델 (기밀성 중심)
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │             Bell-LaPadula 모델 (1973, DoD)                      │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [목적] 기밀성(Confidentiality) 보호                             │
    │                                                                 │
    │  [보안 레벨 (Security Level)]                                   │
    │   Top Secret(4) > Secret(3) > Confidential(2) > Unclassified(1) │
    │                                                                 │
    │  [핵심 규칙]                                                    │
    │   ① No Read Up (Simple Security Property)                      │
    │      - 하위 레벨 주체는 상위 레벨 객체를 읽을 수 없다            │
    │      - Secret 사용자 → Top Secret 문서 읽기 ✗                   │
    │                                                                 │
    │   ② No Write Down (*-Property, Star Property)                  │
    │      - 상위 레벨 주체는 하위 레벨 객체에 쓸 수 없다              │
    │      - Top Secret 사용자 → Secret 문서 쓰기 ✗                  │
    │      (정보가 흘러내리지 않게)                                   │
    │                                                                 │
    │  [문제점]                                                      │
    │   - 무결성 보장 X                                               │
    │   - 하향 정보 흐름 제한으로 비효율                              │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. Biba 모델 (무결성 중심)
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │                  Biba 모델 (1977)                               │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [목적] 무결성(Integrity) 보호                                   │
    │                                                                 │
    │  [무결성 레벨]                                                  │
    │   Critical(4) > Important(3) > Normal(2) > Untrusted(1)         │
    │                                                                 │
    │  [핵심 규칙]                                                    │
    │   ① No Read Down (Simple Integrity Property)                   │
    │      - 상위 레벨 주체는 하위 레벨 객체를 읽을 수 없다            │
    │      - 하위 데이터는 신뢰할 수 없음                              │
    │                                                                 │
    │   ② No Write Up (Integrity *-Property)                         │
    │      - 하위 레벨 주체는 상위 레벨 객체에 쓸 수 없다              │
    │      - 신뢰할 수 없는 데이터가 상위 오염 방지                    │
    │                                                                 │
    │  [용도]                                                        │
    │   - 데이터베이스 무결성, 시스템 프로그램 보호                     │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. Clark-Wilson 모델 (상업용)
- 실무 중심: **분리된 의무(SoD)**, **검증된 변환(Well-Formed Transaction)**
- 사용자는 데이터를 직접 수정 불가, 검증된 프로그램 통해서만 수정

### 4. Chinese Wall 모델 (이해충돌 방지)
- 금융, 법률 등 이해충돌 방지
- A 회사 데이터를 본 사람은 B 회사(경쟁사) 데이터 접근 불가

- **📢 섹션 요약 비유**: Bell-LaPadula는 "비밀 등급이 낮은 사람은 높은 문서 못 본다", Biba는 "신뢰 등급이 낮은 사람은 높은 문서 못 쓴다"는 규칙입니다.

## Ⅳ. 참조 모니터 (Reference Monitor)

### 1. 참조 모니터 개념
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │               참조 모니터 (Reference Monitor)                    │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │   Subject ──[요청]──→ ┌──────────────────┐ ──[결과]──→ Object   │
    │                       │                  │                      │
    │                       │  Reference       │                      │
    │                       │  Monitor         │                      │
    │                       │                  │                      │
    │                       │  - 매 접근 시    │                      │
    │                       │    규칙 검증      │                      │
    │                       │  - 거부/허용     │                      │
    │                       │                  │                      │
    │                       └──────────────────┘                      │
    │                                  │                             │
    │                                  ▼                             │
    │                           Security Policy                      │
    │                           (보안 정책)                           │
    │                                                                 │
    │  [필수 조건]                                                    │
    │   ① 보호된 실행: 수정 불가                                     │
    │   ② 항상 실행: 우회 불가                                       │
    │   ③ 충분히 작음: 성능 저하 최소                                │
    │                                                                 │
    │  [참조 모니터의 구현: 보안 커널]                               │
    │   - 하드웨어 + 커널 모드에서 동작                              │
    │   - TCB(Trusted Computing Base)의 핵심                         │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. TCB (Trusted Computing Base)
- 보안 정책을 올바르게 강제하는 시스템의 모든 부분
- 하드웨어(MMU, CPU Rings), 커널, 보안 소프트웨어 포함
- **TCB가 작을수록 보안 검증이 쉽다** (Security Kernel)

## Ⅴ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기업 보안 정책 수립 절차
1. **자산 식별**: 보호할 데이터, 시스템 파악
2. **위협 분석**: 어떤 위협이 있는지
3. **요구사항 정의**: 규정, 비즈니스 요구사항
4. **정책 작성**: 문서화
5. **구현 및 강제**: 기술적/관리적 통제
6. **감사 및 수정**: 주기적 검토

### 2. 안티패턴
- **"정책만 있고 구현 없음"** (Paper Policy)
- **"너무 엄격해서 업무 불가"**
- **"일관성 없는 정책"**

+++

## Ⅵ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **명확한 기준**: 보안 요구사항 명확화
- **검증 가능성**: 모델 기반 수학적 검증

### 2. 미래 전망
- **Policy as Code**: 정책을 코드로 관리 (OPA, Rego)
- **Zero Trust**: "신뢰 없음, 항상 검증"
