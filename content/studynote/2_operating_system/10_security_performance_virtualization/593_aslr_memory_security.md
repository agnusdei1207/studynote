### 💡 핵심 인사이트 (Insight)
1. **공격 난이도의 비약적 상승**: ASLR은 프로세스의 중요 메모리 영역(스택, 힙, 라이브러리 등) 주소를 실행 시마다 무작위로 배치하여, 공격자가 특정 코드나 데이터의 위치를 예측하기 어렵게 만드는 확률적 방어 기법입니다.
2. **취약점 완화 (Mitigation) 기술**: 메모리 오버플로우와 같은 근본적인 버그를 수정하는 대신, 버그를 이용한 공격(Exploit) 단계에서 필요한 정보(주소 값)를 차단함으로써 공격의 성공 가능성을 현저히 낮춥니다.
3. **계층적 방어의 필수 요소**: ASLR은 단독으로 완벽하지 않으며, DEP(Data Execution Prevention)와 결합될 때 현대 운영체제 보안의 강력한 근간을 형성합니다.

---

## Ⅰ. ASLR의 정의 및 필요성
### 1. 정의
ASLR (Address Space Layout Randomization)은 운영체제가 프로세스를 메모리에 로드할 때 실행 파일 이미지, 시스템 라이브러리, 사용자 스택, 힙 영역의 베이스 주소를 랜덤화하는 기술입니다.

### 2. 필요성 (Before ASLR)
- 과거에는 실행 파일과 라이브러리가 항상 고정된 주소에 로드되었습니다.
- 공격자는 `libc`의 `system()` 함수 주소 등을 미리 알아내어 버퍼 오버플로우 공격 시 해당 주소로 점프(Return-to-libc)하기 쉬웠습니다.

📢 **섹션 요약 비유**: ASLR은 '중요한 보물을 매일 다른 방에 숨겨두는 것'과 같습니다. 도둑이 어제 보물이 있던 방을 알아냈어도 오늘 다시 오면 헛수고를 하게 됩니다.

## Ⅱ. 기술적 메커니즘 분석
### 1. 메모리 레이아웃 변화 (ASCII Diagram)
```text
[Standard Layout]                    [ASLR Layout (Run 1)]          [ASLR Layout (Run 2)]
------------------                   ---------------------          ---------------------
|     Stack      | (Fixed 0x7FFF)    |     [Random 1]    |          |     [Random 2]    |
|       |        |                   |       Stack       |          |       Stack       |
|       V        |                   |         |         |          |         |         |
|      ...       |                   |         V         |          |         V         |
|   Shared Libs  | (Fixed 0x4000)    |   Shared Libs     | (0x5A1F) |   Shared Libs     | (0xC2B4)
|      ...       |                   |      ...          |          |      ...          |
|      Heap      | (Fixed 0x1000)    |      Heap         | (0x23D9) |      Heap         | (0x8F51)
|       A        |                   |       A           |          |       A           |
|       |        |                   |       |           |          |       |           |
|  Executable    | (Fixed 0x0804)    |  Executable       | (PIE)    |  Executable       | (PIE)
------------------                   ---------------------          ---------------------
```

### 2. 핵심 대상 영역
- **Stack & Heap**: 프로그램 데이터와 제어 흐름이 동적으로 변하는 영역.
- **Shared Libraries (DLL/SO)**: 공격자가 자주 타겟팅하는 표준 함수들이 포함된 영역.
- **PIE (Position Independent Executable)**: 실행 코드 자체도 무작위 주소에 배치될 수 있도록 컴파일된 형태.

📢 **섹션 요약 비유**: ASLR은 '퍼즐 조각들의 위치를 매번 섞어버리는 기술'입니다. 완성된 그림(정상 실행)은 똑같지만, 특정 조각(메모리 주소)을 찾으려는 사람은 매번 다시 찾아야 합니다.

## Ⅲ. 엔트로피(Entropy)와 보안 수준
### 1. 엔트로피의 개념
무작위성의 정도를 의미하며, 비트(Bit) 수로 표현됩니다. 주소 공간이 넓고 랜덤화에 사용되는 비트가 많을수록 공격자가 주소를 맞출 확률은 낮아집니다.

### 2. 32비트 vs 64비트 환경
- **32비트**: 주소 공간이 좁아 엔트로피가 낮습니다. 공격자가 무작위로 주소를 대입하는 무차별 대입(Brute-force) 공격에 취약할 수 있습니다.
- **64비트**: 광활한 주소 공간 덕분에 엔트로피가 매우 높으며, 사실상 무차별 대입 공격이 불가능에 가깝습니다.

📢 **섹션 요약 비유**: 엔트로피는 '비밀번호의 길이'와 같습니다. 4자리(32비트)는 금방 풀릴 수 있지만, 20자리(64비트)는 평생 걸려도 풀기 힘든 것과 같습니다.

## Ⅳ. ASLR 우회 기법 및 한계
### 1. 메모리 누수 (Memory Leak / Information Leak)
공격자가 취약점을 이용해 메모리의 특정 주소 하나만 알아낼 수 있다면, 라이브러리의 상대적 오프셋(Offset)은 고정되어 있으므로 전체 레이아웃을 계산해낼 수 있습니다.

### 2. 무차별 대입 공격 (Brute-force)
프로세스가 충돌 후 즉시 다시 시작되는 환경(예: 일부 서버 서비스)에서는 계속해서 다른 주소로 찔러보다가 운 좋게 맞는 경우를 노릴 수 있습니다.

### 3. JIT Spraying
JIT(Just-In-Time) 컴파일러를 이용해 메모리에 다량의 쉘코드(Shellcode)를 뿌려놓아, 랜덤 주소 어디로 점프해도 공격 코드가 실행될 확률을 높이는 기법입니다.

📢 **섹션 요약 비유**: 아무리 보물방 위치를 바꿔도 '바닥에 흘린 지도 조각(메모리 누수)'이 있거나 '수만 번 문을 두드려보는 도둑(Brute-force)'은 막기 힘들 수도 있습니다.

---

## Ⅴ. 운영체제별 구현 및 설정
### 1. Linux
- `/proc/sys/kernel/randomize_va_space` 설정을 통해 제어합니다. (0: Off, 1: Stack/Lib, 2: Stack/Lib/Heap)

### 2. Windows
- Vista 이후 기본 도입되었으며, EMET나 Windows Defender Exploit Guard를 통해 강제 적용 수준을 높일 수 있습니다.

### 3. 모바일 (iOS/Android)
- 초기부터 보안 강화를 위해 PIE와 고수준의 ASLR을 필수적으로 적용하고 있습니다.

📢 **섹션 요약 비유**: ASLR 설정은 '우리 동네 전체에 보안 카메라를 설치하는 규칙'과 같습니다. 모든 프로그램이 이 규칙을 따를 때 동네가 안전해집니다.
