# 인증 및 권한 부여 (Authentication & Authorization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: **인증(Authentication)**은 **"누구인지 확인(N=${who})"**하는 것이며, **권한 부여(Authorization)**는 **"무엇을 할 수 있는지(N=${what})"**를 결정하는 별개의 절차다.
> 2. **가치**: 인증은 ID 도용 방지로 신원을 보장하고, 권한 부여는 최소 권한 원칙으로 무단 접근을 방지하며, **계정(Accounting)**과 함께 AAA 프레임워크를 형성한다.
> 3. **융합**: 다중 인증(MFA), SSO(Single Sign-On), OAuth 2.0, Kerberos 등 현대 인증 시스템은 보안성과 사용성의 균형을 추구한다.

+++

## Ⅰ. 인증(Authentication)의 개요

### 1. 정의
- 인증은 사용자, 장치, 프로세스의 신원을 검증하는 과정이다.
- **Question**: "당신이 말하는 그 사람이 맞습니까?"

### 2. 등장 배경: 신원 확인의 필요성
- 멀티유저 시스템에서 각 사용자를 식별하여 자원에 접근
- "내 계정을 탈취하지 못하게", "가짜 사용자를 막아라"

### 3. 💡 비유: '공항 보안 검색'
- 인증은 **'여권과 얼굴을 대조하는 입국 심사'**와 같다.
- 여권(ID)과 얼굴(Biometric)이 일치해야 입국(접근) 허용.

- **📢 섹션 요약 비유**: 문을 열기 전에 "당신 누구세요?"라고 묻는 출입문 인터폰입니다.

+++

## Ⅱ. 인증 요소 및 방식 (Deep Dive)

### 1. 인증의 3요소 (MFA)
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │                  인증의 3요소 (다중 인증)                        │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [1] 지식 (Knowledge) - 알고 있는 것                             │
    │   - 비밀번호, PIN, 보안 질문 답변                                │
    │   * 가장 널리 사용되지만 유출/도용 위험                           │
    │                                                                 │
    │  [2] 소유 (Possession) - 가지고 있는 것                          │
    │   - OTP 토큰, 스마트카드, 보안 키(YubiKey), 핸드폰               │
    │   * 분실 도난 위험 있지만 지식 기반보다 안전                      │
    │                                                                 │
    │  [3] 본질 (Inherence) - 본인의 신체 특징                         │
    │   - 지문, 홍채, 안면 인식, 정맥, 서명                             │
    │   * 가장 안전하지만 비용 높고 위조 가능성 존재                    │
    │                                                                 │
    │  [MFA (Multi-Factor Authentication)]                            │
    │   - 2개 이상 요소 조합: "비밀번호 + OTP"                          │
    │   - 보안 강도: 지식 < 소유 < 본질 < 조합                          │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. 인증 방식 비교
| 방식 | 설명 | 장점 | 단점 | 예시 |
|:---|:---|:---|:---|:---|
| **Password** | 문자열 기반 | 구현 간단 | 유출/추측 위험 | Unix/Linux |
| **Public Key** | 공개키 암호화 | 강력한 보안 | 키 관리 복잡 | SSH, TLS |
| **Kerberos** | 티켓 기반 | SSO 가능 | KDC 단일 실패점 | Active Directory |
| **Biometric** | 생체 인식 | 편리함 | 위조, 프라이버시 | TouchID, FaceID |
| **TOTP** | 시간 기반 OTP | 일회용 | 시간 동기화 필요 | Google Authenticator |

### 3. 비밀번호 정책
- **최소 길이**: 8~12자 이상
- **복잡성**: 대소문자, 숫자, 특수문자 혼합
- **만료**: 주기적 변경 (단, 보안 연구에서는 긴 비밀번호 권장)
- **솔트(Salt) & 해시**: 저장 시 무조건 적용

- **📢 섹션 요약 비유**: 비밀번호는 집 열쇠, OTP는 일회용 입장권, 생체인식은 내 얼굴입니다. 이 중 2개를 합치면 보안이 훨씬 강해져요.

## Ⅲ. 권한 부여(Authorization)의 개요

### 1. 정의
- 권한 부여는 인증된 주체가 수행할 수 있는 작업을 결정하는 과정이다.
- **Question**: "당신은 이 리소스에 대해 무엇을 할 수 있습니까?"

### 2. AAA 프레임워크
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │                    AAA 보안 프레임워크                           │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [Authentication] 인증                                           │
    │   → "Who are you?" (신원 확인)                                  │
    │   → 예: 비밀번호, 생체인식, 인증서                               │
    │                                                                 │
    │  [Authorization] 권한 부여                                      │
    │   → "What can you do?" (권한 검증)                              │
    │   → 예: ACL, RBAC, MAC                                          │
    │                                                                 │
    │  [Accounting] 계정 감사                                         │
    │   → "What did you do?" (행동 기록)                              │
    │   → 예: 로그, 감사 추적, 통계                                   │
    │                                                                 │
    │  * 순서: 인증 → 권한 부여 → 접근 → 감사                         │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. 인증 vs 권한 부여
| 비교 항목 | 인증(Authentication) | 권한 부여(Authorization) |
|:---|:---|:---|
| **질문** | 누구인가? | 무엇을 할 수 있는가? |
| **목적** | 신원 확인 | 권한 검증 |
| **수행 시점** | 먼저 | 나중에 |
| **데이터** | 자격증명(Credential) | 정책(Policy) |
| **실패 시** | "아는 사람 아님" | "권한 없음" |
| **예시** | 비밀번호 확인 | 파일 읽기 허용 |

- **📢 섹션 요약 비유**: 인증은 "회사 직원 확인(사원증)", 권한 부여는 "사무실 출입 권한(카드 등급)"입니다. 둘 다 있어야 출입이 가능합니다.

## Ⅳ. 현대 인증 시스템

### 1. SSO (Single Sign-On)
- 한 번의 인증으로 여러 서비스 이용
- **구현**: CAS, SAML, OAuth 2.0, OpenID Connect
- **장점**: 사용 편의성, 비밀번호 관리 단순화
- **위험**: SSO 탈취 시 전체 서비스 위험

### 2. OAuth 2.0 & OpenID Connect
- **OAuth 2.0**: 위임된 권한 부여 프레임워크
- **OIDC**: OAuth 2.0 기반 인증 계층
- **용어**:
  - Resource Owner: 사용자
  - Client: 접근 요청 앱
  - Authorization Server: 인증 서버
  - Resource Server: 데이터 서버

### 3. Zero Trust Architecture
- "신뢰하지 않고 항상 검증"
- 네트워크 내부/외부 구분 없이 모든 접근을 인증/권한 검증

## Ⅴ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. Linux 인증 시스템
- **/etc/passwd, /etc/shadow**: 사용자 정보
- **PAM (Pluggable Authentication Modules)**: 인증 플러그인 시스템
- **sudo**: 권한 상승 도구

### 2. 안티패턴
- **"1234" 같은 취약 비밀번호**
- **비밀번호 공유/재사용**
- **관리자 권한으로 일반 작업 수행**
- **로그인 세션 무기한 유지**

+++

## Ⅵ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **보안 강화**: MFA로 계정 탈취 방지
- **감사 가능**: Accounting으로 보안 사고 추적

### 2. 미래 전망
- **Passwordless**: 생체인식, 하드웨어 키 기반
- **Decentralized Identity**: 블록체인 기반 자기 주권 신원

## 👶 어린이를 위한 3줄 비유 설명
1. 인증은 **'학교에서 학생증을 보여주는 것'**과 같아요.
2. 권한 부여는 **'교실, 화장실, 교무실 등 어디까지 갈 수 있는지'** 정하는 거예요.
3. 학생증(인증)이 있어도, 1학년은 6학년 교실에 못 가는 것처럼, 인증과 권한 부여는 다른 단계랍니다!
