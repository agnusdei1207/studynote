# 성능 튜닝 (Performance Tuning)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 성능 튜닝은 **시스템 병목(Bottleneck)을 식별하고 제거**하여 처리량(Throughput)을 높이고 지연(Latency)을 줄이는 체계적 최적화 과정이다.
> 2. **가치**: "80/20 법칙"에 따라 **20%의 병목을 제거하면 80%의 성능 개선** 효과를 얻을 수 있으며, 하드웨어 업그레이드 없이 소프트웨어만으로 성능을 높일 수 있다.
> 3. **융합**: 프로파일링 → 병목 식별 → 튜닝 → 검증의 **반복적 사이클**이며, CPU, 메모리, I/O, 네트워크 각 계층에서 다른 접근이 필요하다.

+++

## Ⅰ. 성능 튜닝의 개요

### 1. 정의
- 성능 튜닝은 시스템의 **처리량, 응답 시간, 자원 사용량**을 최적화하는 활동이다.

### 2. 등장 배경
- "하드웨어만으로는 해결 안 돼", "소프트웨어 튜닝이 필요"

### 3. 💡 비유: '도로 정체 해소'
- 성능 튜닝은 **'도로 정체를 해소하는 작업'**과 같다.
- 교차로 신호 최적화(CPU), 우회로 건설(I/O), 차량 제어(알고리즘) 등 다각도 접근이 필요하다.

- **📢 섹션 요약 비유**: 수도관에서 물이 잘 안 나올 때, 어느 구간이 막혔는지 찾아내고 뚫어주는 작업입니다. 가장 좁은 곳(병목)을 찾는 게 핵심이죠.

+++

## Ⅱ. 튜닝 접근법 (Deep Dive)

### 1. 튜닝 프로세스
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │                   성능 튜닝 사이클                               │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [1단계] 측정 (Measure)                                         │
    │   - 베이스라인 성능 측정                                        │
    │   - 프로파일링 도구 실행                                        │
    │                                                                 │
    │  [2단계] 분석 (Analyze)                                         │
    │   - 병목 지점(Bottleneck) 식별                                 │
    │   - "CPU 100%인가? I/O 대기인가?"                              │
    │                                                                 │
    │  [3단계] 튜닝 (Tune)                                            │
    │   - 가장 큰 병목부터 제거                                       │
    │   - 한 번에 하나씩 변경                                        │
    │                                                                 │
    │  [4단계] 검증 (Verify)                                          │
    │   - 변경 후 성능 측정                                           │
    │   - 개선 확인 or 다시 2단계                                     │
    │                                                                 │
    │  [반복]                                                         │
    │   - "충분히 좋아"까지 계속                                     │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. 병목의 종류
| 병목 유형 | 증상 | 해결 방향 |
|:---|:---|:---|
| **CPU Bound** | CPU 사용률 100% | 알고리즘 개선, 병렬화 |
| **Memory Bound** | Swap 발생, Page Fault | 메모리 증설, 캐시 |
| **I/O Bound** | I/O 대기 시간 길음 | SSD, RAID, 비동기 I/O |
| **Network Bound** | 대역폭 포화, 지연 | 네트워크 업그레이드, 압축 |
| **Lock Contention** | 대기 시간, 스레드 경쟁 | 락 프리, 세분화 |

## Ⅲ. CPU 튜닝

### 1. CPU 병목 확인
```bash
# CPU 사용륩 확인
top -bn1 | grep "Cpu(s)"

# 프로세스별 CPU
ps aux --sort=-%cpu | head -10

# perf로 프로파일링
perf top
```

### 2. CPU 튜닝 기법
- **알고리즘 개선**: O(n²) → O(n log n)
- **캐시 친화적**: 캐시 히트율 높이기 (배열 접근 순서 등)
- **병렬화**: 멀티스레드, 멀티프로세스 활용
- **컴파일러 최적화**: `-O2`, `-O3`

- **📢 섹션 요약 비유**: 셰프(CPU)가 일을 빨리 처리하려면, 레시피(알고리즘)를 개선하고, 조리사(쓰레드)를 늘리고, 재료를 미리 준비(캐시)해야 합니다.

+++

## Ⅳ. 메모리 튜닝

### 1. 메모리 병목 확인
```bash
# 메모리 사용량
free -h

# 프로세스별 메모리
ps aux --sort=-%mem | head -10

# 페이지 폴트
vmstat 1
```

### 2. 메모리 튜닝 기법
- **불필요한 로드 제거**: 사용하지 않는 모듈 언로드
- **메모리 풀**: 미리 할당하여 할당 비용 절감
- **데이터 구조 최적화**: compact structure
- **Swap 최소화**: swappiness 설정

```bash
# swappiness 줄이기 (0~100)
sysctl vm.swappiness=10
```

+++

## Ⅴ. I/O 튜닝

### 1. I/O 병목 확인
```bash
# I/O 통계
iostat -x 1

# 디스크 사용량
df -h

# slow query 로그 (DB)
```

### 2. I/O 튜닝 기법
- **SSD 사용**: HDD보다 100배 빠른 IOPS
- **RAID**: 0(성능), 1(안전), 10(둘 다)
- **I/O 스케줄러**: CFQ, deadline, noop
- **비동기 I/O**: Blocking 없는 I/O

+++

## Ⅵ. 네트워크 튜닝

### 1. 네트워크 병목 확인
```bash
# 대역폭
iftop

# 연결 수
ss -s

# 지연 시간
ping -c 10 example.com
```

### 2. 네트워크 튜닝 기법
- **Keep-alive**: 연결 재사용
- **압축**: gzip, brotli
- **HTTP/2**: Multiplexing
- **CDN**: 콘텐츠 캐싱

+++

## Ⅶ. 애플리케이션 튜닝

### 1. 데이터베이스 튜닝
```sql
-- 인덱스 추가
CREATE INDEX idx_user_email ON users(email);

-- 쿼리 최적화
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'test@example.com';
```

### 2. 캐싱 전략
- **계층형 캐시**: L1(코드) → L2(Redis) → L3(DB)
- **Cache Aside**: 코드에서 캐시 관리
- **Write-Through**: 캐시에 먼저 쓰기

+++

## Ⅷ. 실무 적용

### 1. 안티패턴
- **"미리 최적화 (Premature Optimization)"**: 측정 없이 예상으로 튜닝
- **"동시에 여러 변경"**: 무엇이 효과 있었는지 모름
- **"마이크로 최적화"**: 큰 병목 무시하고 사소한 것에 집착
- **"하드웨어 먼저"**: 소프트웨어 튜닝 후 하드웨어 고려

### 2. 성능 테스트 도구
- **Load Testing**: JMeter, Locust, k6
- **Benchmarking**: sysbench, fio

+++

## Ⅸ. 기대효과 및 결론

### 1. 정량/정성 기대효가
- **처리량 증가**: 더 많은 요청 처리
- **지연 감소**: 빠른 응답 시간
- **비용 절감**: 하드웨어 업그레이드 연기
