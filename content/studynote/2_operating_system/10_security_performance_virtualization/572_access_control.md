# 접근 제어 (Access Control)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 접근 제어(Access Control)는 **누가(Subject) 무엇(Object)에 어떤 작업(Operation)을 수행할 수 있는지**를 정의하고 강제하는 보호 메커니즘이다.
> 2. **가치**: 시스템 리소스에 대한 무단 접근을 방지하여 **최소 권한의 원칙(Principle of Least Privilege)**을 구현하고, 데이터 유출과 시스템 장애를 예방한다.
> 3. **융합**: DAC(임의적), MAC(강제적), RBAC(역할 기반) 등 다양한 모델이 존재하며, 현대 OS는 이를 계층화하여 Defense in Depth를 실현한다.

+++

## Ⅰ. 접근 제어의 개요

### 1. 정의
- 접근 제어는 주체(Subject)가 객체(Object)에 접근할 때 권한을 검증하여 승인된 작업만 허용하는 메커니즘이다.
- **Subject**: 사용자, 프로세스, 스레드 등 접근을 요청하는 개체
- **Object**: 파일, 메모리, 디바이스, 소켓 등 접근의 대상

### 2. 등장 배경: 멀티유저 시스템의 필요성
- 시분할 시스템에서 다수 사용자가 리소스를 공유하면서 각자의 데이터 보호 필요
- "내 파일을 남이 읽지 못하게", "시스템 파일은 아무나 수정 못하게"

### 3. 💡 비유: '호텔 키카드 시스템'
- 접근 제어는 **'호텔 키카드 시스템'**과 같다.
- 각 키카드는 특정 객실만 열 수 있고, 청소부 카드는 모든 객실, 마스터 키는 건물 전체에 접근 가능.

- **📢 섹션 요약 비유**: 아파트 각 세대의 출입키는 그 집 사람만 줍니다. 경비실은 전체 관리 권한을 갖습니다.

+++

## Ⅱ. 접근 제어 모델 (Deep Dive)

### 1. DAC (Discretionary Access Control) - 임의적 접근 제어
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │              DAC (임의적 접근 제어) 특징                         │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [정의] 객체 소유자(Owner)가 접근 권한을 직접 결정               │
    │                                                                 │
    │  [구현] Access Control List (ACL)                               │
    │   - 파일마다 권한 목록 부착                                      │
    │   - User1: rwx, User2: r--, Group: rw-, Others: r--            │
    │                                                                 │
    │  [장점]                                                         │
    │   - 유연성: 소유자가 자유롭게 권한 변경                          │
    │   - 사용 편의성                                                 │
    │                                                                 │
    │  [단점]                                                         │
    │   - Malware가 소유자 권한을 탈취하면 전체 접근 가능              │
    │   - 권한 확산(Policy Creep)                                     │
    │                                                                 │
    │  [예시] Unix/Linux 파일 권한, Windows NTFS                      │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. MAC (Mandatory Access Control) - 강제적 접근 제어
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │             MAC (강제적 접근 제어) 특징                          │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [정의] 시스템 정책에 의해 접근 권한이 강제적으로 결정            │
    │                                                                 │
    │  [구현] Security Label                                          │
    │   - Subject와 Object에 보안 라벨 부착                           │
    │   - 라벨 비교로 접근 허용/거부                                  │
    │                                                                 │
    │  [Bell-LaPadula 모델] (기밀성 중심)                             │
    │   - No Read Up: 하위 등급은 상위 문서 읽기 불가                  │
    │   - No Write Down: 상위 등급은 하위에 쓰기 불가                  │
    │                                                                 │
    │  [Biba 모델] (무결성 중심)                                      │
    │   - No Read Down: 상위는 하위 읽기 불가                         │
    │   - No Write Up: 하위는 상위에 쓰기 불가                         │
    │                                                                 │
    │  [장점]                                                         │
    │   - 강력한 보안: 소유자가 권한 변경 불가                         │
    │   - 데이터 유출 방지                                             │
    │                                                                 │
    │  [단점]                                                         │
    │   - 관리 복잡성                                                 │
    │   - 사용 불편                                                   │
    │                                                                 │
    │  [예시] SELinux, AppArmor, SMACK                                │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. RBAC (Role-Based Access Control) - 역할 기반 접근 제어
- 사용자에게 직접 권한을 부여하지 않고 **역할(Role)**을 부여
- 역할에 권한 묶음 연결
- **장점**: 권한 관리 단순화, 조직 구조 반영 용이
- **예시**: Oracle, SQL Server, 기업용 응용 프로그램

- **📢 섹션 요약 비유**: 회사에서 직책(역할)에 따라 출입카드를 주는 방식입니다. 대리는 2층까지, 과장은 3층까지, 사장은 전층 접근 가능합니다.

## Ⅲ. 접근 제어 매트릭스와 구현

### 1. 접근 제어 매트릭스
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │                 Access Matrix 구조                              │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │              Object 1   Object 2   Object 3   Object 4         │
    │           ┌──────────┬──────────┬──────────┬──────────┐        │
    │ Subject 1 │   R W    │    R     │          │    R     │        │
    │           ├──────────┼──────────┼──────────┼──────────┤        │
    │ Subject 2 │    R     │  R W X   │    W     │          │        │
    │           ├──────────┼──────────┼──────────┼──────────┤        │
    │ Subject 3 │          │    R     │  R W     │    X     │        │
    │           └──────────┴──────────┴──────────┴──────────┘        │
    │                                                                 │
    │   * R=Read, W=Write, X=Execute, 공백=권한 없음                   │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. 구현 방식
| 방식 | 설명 | 장점 | 단점 |
|:---|:---|:---|:---|
| **Global Table** | 전체 매트릭스 저장 | 간단 | 공간 낭비 심함 |
| **ACL** | 객체별 열 저장 | 객체 중심 접근 용이 | 사용자별 권한 확인 어려움 |
| **Capability List** | 사용자별 행 저장 | 사용자 중심 접근 용이 | 권한 취소 어려움 |
| **Lock-Key** | Lock(객체), Key(사용자) | 유연함 | 복잡함 |

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. Linux 접근 제어 구현
- **DAC 기본**: Unix Permission (rwxrwxrwx)
- **MAC 추가**: SELinux, AppArmor
- **ACL 확장**: POSIX ACL (getfacl, setfacl)

### 2. SELinux (Security-Enhanced Linux)
- **Type Enforcement**: 도메인(프로세스)과 타입(파일) 간 접근 규칙
- **모드**: Enforcing(강제), Permissive(로그만), Disabled(비활성)
- **ls -Z**: SELinux 컨텍스트 확인

### 3. 안티패턴: 과도한 권한 부여
- **777 권한 남용**: 모든 사용자에게 모든 권한 부여
- **Root로 실행**: 최소 권한 원칙 위배
- **권한 누적**: 사용자가 퇴사해도 권한 회수 안 됨

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **제로 트러스트**: 모든 접근을 검증하여 보안 강화
- **책임 추적**: 감사 로그로 접근 이력 추적 가능

### 2. 미래 전망
- **ABAC (Attribute-Based)**: 속성 기반 동적 접근 제어
- **Policy as Code**: 접근 제어 정책을 코드로 관리
