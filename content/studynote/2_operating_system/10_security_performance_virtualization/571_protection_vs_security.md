# 보호 vs 보안 (Protection vs Security)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: **보호(Protection)**는 시스템 내부에서 리소스 간의 **상호 간섭을 방지**하는 메커니즘이며, **보안(Security)**은 외부/내부의 **악의적 공격으로부터 시스템을 방어**하는 포괄적 개념이다.
> 2. **가치**: 보호는 OS가 커널 모드와 사용자 모드를 분리하고 프로세스 간 메모리를 격리하여 **시스템 안정성**을 보장하며, 보안은 인증, 암호화, 감사를 통해 **기밀성, 무결성, 가용성**을 확보한다.
> 3. **융합**: 현대 OS는 보호 메커니즘(Ring, MMU, ACL)을 기반으로 보안 기술(방화벽, SELinux, TPM)을 계층화하여 **Defense in Depth** 전략을 구현한다.

+++

## Ⅰ. 보호(Protection)의 개요

### 1. 정의
- 보호는 컴퓨터 시스템에서 프로그램, 프로세스, 사용자가 시스템 리소스에 접근하는 것을 제어하는 메커니즘이다.
- 주어진 규칙에 따라 **승인된 접근만 허용**하고, 그 외의 접근을 거부한다.

### 2. 등장 배경: 멀티프로그래밍과 리소스 공유
- 단일 사용자 시스템에서는 불필요했지만, 시분할 시스템과 멀티프로그래밍에서 필수적이다.
- 프로세스 A가 프로세스 B의 메모리를 침범하거나, 일반 사용자가 시스템 콜을 직접 호출하는 것을 방지해야 한다.

### 3. 💡 비유: '아파트의 각 세대와 출입문 보안'
- 보호는 **'각 세대마다 자기 방으로만 들어갈 수 있게 문 잠금장치를 설치한 구조'**와 같다.
- 101호 거주자는 102호에 들어갈 수 없고, 옥탑(커널)에는 관리자만 접근할 수 있다.

- **📢 섹션 요약 비유**: 아파트 경비실이 입주민의 사생활을 보호하지만, 도둑이 침입하면 경비실만으로는 부족합니다. 여기에 CCTV와 보안문(보안)이 더해져야 합니다.

+++

## Ⅱ. 보호 메커니즘 (Deep Dive)

### 1. 보호의 핵심 요소
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │                    보호 시스템의 구성 요소                        │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [1] 보호 도메인 (Protection Domain)                            │
    │   - 프로세스가 접근 가능한 리소스의 집합                         │
    │   - Domain = {자원1: 읽기, 자원2: 쓰기, 자원3: 실행}             │
    │                                                                 │
    │  [2] 객체 (Object)                                              │
    │   - 보호의 대상: 파일, 메모리, 디바이스, 프로세스 등             │
    │                                                                 │
    │  [3] 접근 권한 (Access Rights)                                  │
    │   - Read, Write, Execute, Append, Delete                        │
    │                                                                 │
    │  [4] 접근 제어 매트릭스 (Access Matrix)                         │
    │   ┌─────────┬───────┬───────┬───────┬───────┐                   │
    │   │ Domain  │ File1 │ File2 │ Printer│ Mem   │                   │
    │   ├─────────┼───────┼───────┼───────┼───────┤                   │
    │   │ User    │  R/W  │   R   │   W   │  R/W  │                   │
    │   │ Kernel  │  ALL  │  ALL  │  ALL  │  ALL  │                   │
    │   │ Admin   │  ALL  │  ALL  │  ALL  │  ALL  │                   │
    │   └─────────┴───────┴───────┴───────┴───────┘                   │
    │                                                                 │
    │  [5] 구현 방법                                                   │
    │   - Access Control List (ACL): 객체별 권한 목록                  │
    │   - Capabilities: 도메인별 접근 토큰                             │
    └─────────────────────────────────────────────────────────────────┘
```

### 2. 하드웨어 지원 보호
- **CPU Rings (x86)**: Ring 0(커널) ~ Ring 3(사용자)
- **MMU (Memory Management Unit)**: 페이지 테이블 기반 메모리 격리
- **TLB**: 주소 변환 캐시로 접근 권한 검증

### 3. 모드 전환 (Mode Switch)
- 사용자 모드 → 커널 모드: 시스템 콜, 인터럽트, 예외 발생 시
- 커널 모드에서만 특권 명령 실행 가능

- **📢 섹션 요약 비유**: 일반 직원(사용자 모드)은 경비실을 통해서만 사무실에 들어갈 수 있고, 관리자(커널 모드)는 모든 방에 자유롭게 출입할 수 있습니다.

+++

## Ⅲ. 보안(Security)의 개요

### 1. 정의
- 보안은 시스템의 자산을 **외부/내부의 위협으로부터 보호**하는 포괄적 활동이다.
- CIA 트리아드: 기밀성(Confidentiality), 무결성(Integrity), 가용성(Availability)

### 2. 보안 위협 유형
| 위협 유형 | 설명 | 예시 |
|:---|:---|:---|
| **무단 접근** | 권한 없는 사용자의 접근 | 비밀번호 탈취, 백도어 |
| **무단 삽입** | 악의적 코드/데이터 주입 | 바이러스, SQL Injection |
| **무단 삭제** | 데이터 파괴 | 랜섬웨어 |
| **도청** | 데이터 유출 | 패킷 스니핑 |
| **서비스 거부** | 가용성 방해 | DDoS |

### 3. 보안 메커니즘
- **인증(Authentication)**: 사용자 확인 (비밀번호, 생체인식)
- **권한 부여(Authorization)**: 접근 권한 검증
- **암호화**: 데이터 기밀성 확보
- **감사(Auditing)**: 보안 로그 및 추적
- **방화벽/IDS**: 네트워크 보안

- **📢 섹션 요약 비유**: 보호는 "출입증 확인"이고, 보안은 "출입증 + CCTV + 금고 + 보안요원" 전체 시스템입니다.

+++

## Ⅳ. 보호 vs 보안 비교

| 비교 항목 | 보호 (Protection) | 보안 (Security) |
|:---|:---|:---|
| **목표** | 리소스 간 간섭 방지 | 위협으로부터 시스템 방어 |
| **대상** | 시스템 내부 프로세스/사용자 | 내부/외부 공격자 |
| **주요 메커니즘** | 모드 비트, MMU, ACL | 암호화, 방화벽, 인증 |
| **필수성** | OS 기본 기능 | 추가적 보호 계층 |
| **실패 시** | 시스템 충돌, 데이터 손상 | 정보 유출, 시스템 장악 |
| **예시** | 프로세스 메모리 격리 | 비밀번호 정책, SELinux |

+++

## Ⅴ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: Defense in Depth
- 단일 보안 계층은 취약하다.
- **계층별 보호**: 하드웨어(MMU) → 커널(모드 비트) → 파일시스템(권한) → 네트워크(방화벽)

### 2. Linux 보호/보안 구현
- **Protection**: User/Kernel Mode, SELinux, AppArmor
- **Security**: PAM(인증), iptables/nftables(방화벽), TPM(트러스트)

### 3. 안티패턴: 보호와 보안의 혼동
- "루트 권한이 있으니 안전하다" → 오류, 보호만 확보된 상태
- "방화벽만 설치하면 된다" → 내부 위협 방어 불가

+++

## Ⅵ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **안정성**: 보호로 프로세스 격리 → 시스템 크래시 방지
- **신뢰성**: 보안으로 데이터 유출 방지 → 컴플라이언스 준수

### 2. 미래 전망
- **Zero Trust Architecture**: "신뢰하지 않고 항상 검증"
- **Hardware Root of Trust**: TPM, SGX, TrustZone

- **접근 제어 (Access Control)**: 보호의 핵심 구현

- **SELinux**: 강제적 접근 제어(MAC) 구현
- **가상화 보안**: VM 격리 및 보안

2. 보안은 **'낯선 사람이 들어오지 못하게 문단속하고, CCTV로 감시하는 것'**이랍니다.
