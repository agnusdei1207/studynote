# 601. 성능 측정 지표 - CPU 이용률, 처리량, 지연 시간

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 성능은 단일 척도가 아닌 **자원 효율성(Utilization)**, **시스템 용량(Throughput)**, **응답성(Latency)**의 상충 관계(Trade-off)로 정의되며, 이는 큐잉 이론(Queuing Theory)에 기반한다.
> 2. **가치**: 리틀의 법칙(Little's Law, $L=\lambda W$)을 통해 부하(Load) 증가에 따른 지연 시간의 비선형적 급증(Saturation)을 예측하고, 시스템 한계점(Capacity Planning)을 정량적으로 관리할 수 있다.
> 3. **융합**: CPU 스케줄링(OS), 네트워크 대역폭(Net), 디스크 IOPS(Storage)의 융합적 병목 지점을 분석하여 클라우드 오토 스케일링(Auto-scaling) 정책 및 SLA (Service Level Agreement) 수립의 근거가 된다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 정의
성능 측정 지표는 컴퓨팅 시스템이 얼마나 효율적으로 작동하는지를 정량화한 척도입니다. 단순히 "빠르다"는 주관적 느낌을 배제하고, **CPU (Central Processing Unit)**의 점유율, 단위 시간당 처리량, 그리고 요청에 대한 반응 속도라는 세 가지 축으로 분석합니다. 이는 컴퓨터 시스템을 하나의 '블랙박스(Black Box)'로 보고, 입력(요청)에 대한 출력(응답)을 수학적으로 모델링하는 큐잉 이론에서 출발합니다.

#### 2. 등장 배경 및 필요성
초기 컴퓨팅 환경인 배치(Batch) 처리 시스템에서는 '처리량(Throughput)'이 곧 성공의 척도였습니다. 하지만 시간이 지남에 따라 대화형 시스템(TSS) 및 웹 서비스가 등장하면서, 사용자가 기다리는 시간인 '지연 시간(Latency)'이 중요해졌습니다. 현대의 클라우드 환경에서는 무제한 자원을 투입할 수 없으므로, 제한된 자원(CPU, Memory) 내에서 '이용률(Utilization)'을 높이면서도 '지연 시간'이 급격히 늘어나는 현상을 막는 고도로 정교한 튜닝이 요구됩니다.

#### 3. 핵심 지표 상관관계
```text
       [ 시스템 자원 (Capacity) ]
              ▲
              │
      ┌───────┴───────┐
      │   Processor   │
      │   (Service)   │
      └───────┬───────┘
              │
      ┌───────┴─────────────────────┐
      │      Queue (Buffer)         │
      │  Waiting Requests (Jobs)    │
      └───────┬─────────────────────┘
              │
      [Arrivals (Requests) ]
```
- **해설**: 위의 다이어그램과 같이 요청은 큐(Queue)에 쌓인 뒤 서비스(처리)됩니다. 
  - **이용률($\rho$)**: 서버가 바쁜 시간의 비율 ($\rho = \lambda / \mu$, 도착률/서비스률).
  - **처리량**: 단위 시간당 서비스를 완료한 작업 수 ($\lambda$, 도착률과 같아짐).
  - **지연 시간**: 큐에서 대기하는 시간($W_q$) + 실제 처리 시간($S$).
  이 세 가지는 서로 밀접하게 연결되어 있어, 하나를 조정하면 나머지 두 지표에 반드시 영향을 미치게 됩니다.

> **📢 섹션 요약 비유**: 시스템 성능 측정은 마치 **'고속도로 톨게이트'**의 운영 효율을 분석하는 것과 같습니다. 
> 1. **이용률**은 톨게이트 부스가 얼마나 바쁘게 가동되는지이고, 
> 2. **처리량**은 한 시간 동안 통과시킬 수 있는 차량의 대수이며, 
> 3. **지연 시간**은 운전자가 진입부터 통과까지 걸린 시간입니다. 
> 부스를 하나만 열면(이용률 100%) 차량이 줄을 서서 지연 시간이 길어지지만, 운영 비용은 절감되는 상충 관계가 발생합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 지표 상세 분석 및 구성 요소

| 구분 | 지표 명칭 (Full Name) | 정의 및 수식 | 단위 (Unit) | 실무적 관점 |
|:---|:---|:---|:---|:---|
| **자원 효율** | **CPU Utilization**<br>(Central Processing Unit Utilization) | $U = \frac{\text{Busy Time}}{\text{Total Time}}$<br>전체 시간 중 CPU가 작업을 수행한 비율 | % (Percent) | **70-80% 법칙**: 80% 이상 지속 시 병목 가능성 높음. 클라우드 과금 지표와 직결됨. |
| **시스템 용량** | **Throughput**<br>(Transactions Per Second: TPS) | $X = \frac{N}{T}$<br>단위 시간당 완료된 트랜잭션/작업의 개수 | TPS, RPS, IOPS | 시스템의 '부패력'을 나타냄. 스케일 아웃(Scaling Out) 시 선형 증가 여부 확인 필수. |
| **응답 속도** | **Latency**<br>(Response Time / Turnaround Time) | $R = W_q + S$<br>(대기 시간 + 서비스 시간) | ms (Millisecond), s | 사용자 경험(UX)을 좌우. 평균보다 P99(99번째 백분위수)가 중요함. |

#### 2. 리틀의 법칙 (Little's Law)과 동역학
큐잉 이론의 가장 강력한 도구인 리틀의 법칙은 안정적인 시스템에서 다음 성립합니다.
$$ L = \lambda W $$
- **L (Length)**: 시스템 내 평균 작업 수 (큐에 있는 요청 + 처리 중인 요청)
- **$\lambda$ (Lambda)**: 평균 도착률 (Arrival Rate, TPS)
- **W (Wait)**: 평균 체류 시간 (Response Time)

이 법칙은 시스템의 내부 구조(스케줄링 알고리즘 등)에 상관없이 성립하므로, 복잡한 시스템의 성능을 예측하는 데 필수적입니다.

#### 3. ASCII 다이어그램: 이용률에 따른 지연 시간의 변이 (Response Time Curve)
이용률이 증가함에 따라 지연 시간이 어떻게 비선형적으로 증가하는지 시각화합니다.

```text
   [ 응답 시간 (Response Time) ]
       ^
       |                    /
       |                   /
       |                  /
       |                 / 
       |                / 
       |               /
       |              /
       |             /
       |____________/ 
       +---------------------------> [ 이용률 (Utilization) ]
       0%                        70%   100%
            (Safety Zone)     (Knee)
```
- **① 도입 서술**: 그래프의 가로축은 이용률, 세로축은 지연 시간입니다. 이용률이 낮을 때는 지연 시간이 일정하지만, 특정 임계점(Knee, 보통 70~80%)을 넘어가면 지연 시간이 기하급수적으로 폭발합니다.
- **③ 해설**:
  1. **안정 구간 (0~70%)**: 요청이 오면 즉시 처리되거나 짧게 대기합니다. 지연 시간은 처리 시간(Service Time)과 유사합니다.
  2. **무릎 구간 (Knee, 70~80%)**: 큐(Queue)가 형성되기 시작합니다. 작업이 밀리기 시작하지만 아직 견딜 만한 수준입니다. 이 구간을 **'최적의 성능 구간'**으로 봅니다.
  3. **포화 구간 (80%+)**: 큐가 순식간에 찹니다. 새로운 요청이 들어올 때마다 대기열이 길어져, 지연 시간이 무한대로 수렴합니다(Thrashing 현상). 시스템이 반응하지 않는 것과 같습니다.

#### 4. 코드 및 메트릭 분석 (Linux Kernel Scenarios)
```bash
# 실무에서 CPU 이용률과 성능을 확인하는 vmstat 샘플 (Linux)
# vmstat 1
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs  us sy id wa
 1  0      0 100000  20000 300000    0    0     5     2   10   15  50 10 35  5
```
- **us (User)**: 사용자 레벨 애플리케이션 CPU 사용량.
- **sy (System)**: 커널 레벨 시스템 콜 호출 비율.
- **id (Idle)**: CPU 유휴 시간.
- **wa (I/O Wait)**: 입출력을 기다리며 CPU가 놀고 있는 시간 (디스크 병목 지표).

> **📢 섹션 요약 비유**: 시스템의 원리는 마치 **'피자 가게의 주문 처리'**와 같습니다. 
> **L(리틀의 법칙)**은 가게 안에 있는 주문서의 총개수이고, **$\lambda$(도착률)**은 시간당 전화 주문 건수, **W(체류 시간)**은 주문 후 배달까지 걸리는 시간입니다. 
> 요리사(CPU)가 너무 바쁘면(이용률 90%), 주문서는 가득 차고(L), 신규 주문은 배달될 때까지 한참을 기다려야(W) 합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 심층 기술 비교: 처리량 vs 지연 시간 (Latency vs Throughput)

| 비교 항목 | 처리량 (Throughput) 중심 시스템 | 지연 시간 (Latency) 중심 시스템 |
|:---|:---|:---|
| **대표 시스템** | **HPC (High Performance Computing)**, 배치 처리, 렌더링 팜 | **HFT (High Frequency Trading)**, 웹 서버, 게임 서버 |
| **최적화 목표** | 단위 시간당 처리량 극대화 (Work越多越好) | 개별 요청의 대기 시간 최소화 (Speed越多越好) |
| **OS 설정** | 배치 큐(Batch Queue) 크게, 타임 슬라이스 길게 | 선점형(Preemptive) 스케줄링, 타임 슬라이스 짧게, CPU Affinity 적용 |
| **네트워크** | 대역폭(Bandwidth) 최우선, Jumbo Frame 사용 | RTT (Round Trip Time) 최소화 우선 |
| **융합 포인트** | 처리량이 높아야 지연 시간이 짧아질 수 있음 (선형 구간) | 지연 시간을 줄이려면 Context Switching 오버헤드가 발생하여 처리량 감소 가능성 있음 |

#### 2. ASCII 다이어그램: P99 Latency vs Average Latency
평균 값이 실무 성능을 대변하지 못하는 '롱 테일(Long Tail)' 현상을 시각화합니다.

```text
   [ 확률 밀도 함수 (Probability Density) ]
       ^
       |   __A__           _P99_
       |  /      \        /
       | /        \      /
       |/          \____/
       |                \________
       +---------------------------> [ 응답 시간 (Response Time) ]
       ^                ^
    Average          Tail
```
- **해설**: 
  - **Average (평균)**: 대부분의 요청은 빠르게 처리(왼쪽)되지만, 일부 느린 요청(오른쪽 꼬리)이 평균을 왜곡할 수 있습니다.
  - **P99 (99th Percentile)**: 100명 중 99번째 사용자의 경험. 실제 **SLA (Service Level Agreement)**를 만족하는지 결정하는 핵심 지표입니다. 
  - 융합 관점: 미세한 **Gc Pause (Garbage Collection)**나 디스크 **Seek Time**이 전체 평균을 망치지 않지만, P99 사용자에게는 치명적입니다.

#### 3. 타 과목 융합 분석
- **CPU 스케줄링 (OS)**: **MLFQ (Multi-Level Feedback Queue)**와 같은 스케줄러는 CPU Burst를 예측하여 대화형 프로세스(Latency 중요)를 우선순위에 둡니다.
- **네트워크 (Network)**: **TCP (Transmission Control Protocol)**의 혼잡 제어(Congestion Control)는 네트워크 스위치의 '처리량'이 포화되면 '지연 시간'이 증가함을 감지하여 윈도우 크기를 줄입니다.

> **📢 섹션 요약 비유**: 이는 **'신호등과 교통 흐름'**을 조율하는 것과 같습니다. 
> 교통청(CPU 스케줄러)은 구급차(지연 시간 민감 작업)를 위해 신호를 바꾸지만, 그러다 보면 일반 승용차의 전체 통과 대수(처리량)가 줄어들 수 있습니다. 
> 따라서 도시 전체가 마비되지 않도록, **P99(재수 없는 한 명의 운전자)**가 너무 오래 걸리지 않는 선에서 전체 효율을 극대화해야 합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 시나리오: 웹 서비스 성능 저하 진단
**상황**: 이커머스 사이트의 세일 기간 중, 모니터링 대시보드에서 CPU 이용률은 60%로 정상이나, 사용자 클레임("페이지가 너무 느려요")이 쇄도함.

**의사결정 과정**:
1. **데이터 분석**: CPU 이용률은 낮으나, **Load Average**와 **I/O Wait**가 높음.
