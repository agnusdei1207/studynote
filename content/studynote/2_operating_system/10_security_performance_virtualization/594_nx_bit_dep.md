### 💡 핵심 인사이트 (Insight)
1. **데이터와 코드의 엄격한 분리**: DEP/NX bit는 메모리 영역을 '데이터 저장용'과 '코드 실행용'으로 명확히 구분하여, 데이터 영역(스택, 힙)에 주입된 악성 코드가 실행되는 것을 하드웨어 레벨에서 차단합니다.
2. **하드웨어와 OS의 협업**: CPU의 NX bit(AMD) 또는 XD bit(Intel) 기술을 운영체제가 활용하여 페이지 단위로 실행 권한을 제어하는 보안 메커니즘입니다.
3. **공격 기법의 변화 유도**: 이 기술의 도입으로 인해 직접적인 쉘코드(Shellcode) 실행이 어려워졌으며, 결과적으로 공격자들이 기존에 존재하는 코드를 재조합하는 ROP(Return-Oriented Programming) 기법을 사용하게 만드는 계기가 되었습니다.

---

## Ⅰ. NX bit와 DEP의 정의
### 1. NX bit (No-Execute bit)
CPU의 페이지 테이블 엔트리(PTE)에 포함된 1비트의 플래그입니다. 이 비트가 설정된 페이지에서는 CPU가 명령어를 인출(Fetch)하여 실행하려 할 때 예외(Exception)를 발생시킵니다.
- **AMD**: NX (No-Execute)
- **Intel**: XD (eXecute Disable)
- **ARM**: XN (eXecute Never)

### 2. DEP (Data Execution Prevention)
운영체제가 NX bit 하드웨어 기능을 활용하여 구현한 보안 기능입니다. 프로세스의 데이터 영역(Stack, Heap)을 실행 불가능으로 표시하여 버퍼 오버플로우를 통한 코드 실행 공격을 방지합니다.

📢 **섹션 요약 비유**: NX bit는 '도서관의 책(데이터)을 읽을 수는 있지만, 그 책에 적힌 지시대로 행동(실행)하는 것은 금지하는 규칙'과 같습니다.

---

## Ⅱ. 기술적 메커니즘 분석
### 1. 페이지 테이블 권한 제어 (ASCII Diagram)
```text
[Memory Page Table Entry]
+-------+-------+-------+-------+---+---+---+
| Addr  | Other | Write | Read  |NX | P | U |
+-------+-------+-------+-------+---+---+---+
                                  ^
                                  |
[CPU Execution Check]             |
1. Instruction Pointer (IP) points to Address X
2. Check NX bit for Page containing X
3. IF NX == 1:
      GENERATE [Access Violation Exception]
      TERMINATE [Process]
   ELSE:
      Execute Instruction
```

### 2. 하드웨어 지원 필수성
과거 소프트웨어 방식의 DEP는 오버헤드가 컸으나, 현대의 하드웨어 DEP는 CPU가 주소 번역 단계에서 직접 권한을 체크하므로 성능 저하가 거의 없습니다.

📢 **섹션 요약 비유**: NX bit는 '무대(실행 영역)와 객석(데이터 영역)을 구분하는 펜스'와 같습니다. 객석에 있는 사람이 갑자기 무대로 뛰어들어 공연(코드 실행)을 하려 하면 즉시 보안 요원(CPU)이 제지합니다.

## Ⅲ. DEP의 종류와 설정
### 1. 하드웨어 DEP vs 소프트웨어 DEP
- **하드웨어 DEP**: NX bit 기능을 지원하는 CPU에서 동작. 모든 종류의 실행 시도를 차단.
- **소프트웨어 DEP**: 하드웨어 지원이 없을 때 예외 처리 메커니즘(SafeSEH 등)을 이용하여 제한적으로 방어.

### 2. 운영체제별 설정 정책 (Windows 예시)
- **OptIn**: 운영체제 핵심 파일 및 사용자가 지정한 프로그램에만 적용.
- **OptOut**: 기본적으로 모든 프로그램에 적용하되, 제외 목록을 관리.
- **AlwaysOn**: 시스템 전체에 강제 적용 (가장 안전).

📢 **섹션 요약 비유**: DEP 설정은 '학교 전체에 엄격한 규칙을 적용할지(AlwaysOn), 말썽꾸러기 학생들(특정 앱)에게만 적용할지(OptIn) 결정하는 교칙'과 같습니다.

## Ⅳ. DEP 우회 기법: ROP (Return-Oriented Programming)
### 1. 한계점
DEP는 '새로운 코드'를 실행하는 것은 막지만, 메모리에 '이미 존재하는 코드'를 사용하는 것은 막지 못합니다.

### 2. ROP의 원리
공격자는 프로그램 내부에 이미 존재하는 유효한 코드 조각들(Gadgets)의 주소를 스택에 나열합니다. 각 조각 끝에 있는 `ret` 명령을 이용하여 이 조각들을 체인(Chain)처럼 엮어서 원하는 기능을 수행합니다.
- **Gadget**: `pop eax; ret`, `add esp, 8; ret` 등 짧은 코드 조각.

📢 **섹션 요약 비유**: 직접 연설문(악성 코드)을 써서 낭독하는 것은 금지되자, 도서관에 있는 수천 권의 책에서 한 단어씩 오려 붙여(Gadget 조합) 협박 편지(공격)를 완성하는 것과 같습니다.

---

## Ⅴ. 상호 보완적 방어 체계
### 1. DEP + ASLR의 시너지
- **DEP 단독**: 공격자가 ROP 가젯의 주소를 알면 뚫립니다.
- **ASLR 단독**: 공격자가 운 좋게 주소를 맞추면 직접 쉘코드를 실행할 수 있습니다.
- **결합**: 가젯의 주소도 모르고(ASLR), 직접 실행도 안 되는(DEP) 강력한 방벽이 구축됩니다.

### 2. W^X (Write XOR Execute)
메모리 페이지는 '쓰기 가능(Write)'하거나 '실행 가능(Execute)'할 수 있지만, 동시에 두 권한을 가질 수는 없다는 원칙입니다. 현대 보안 운영체제의 핵심 원칙입니다.

📢 **섹션 요약 비유**: '방 문을 잠그는 것(DEP)'과 '방 번호를 계속 바꾸는 것(ASLR)'을 동시에 하면 도둑이 절대 들어올 수 없는 것과 같습니다.
