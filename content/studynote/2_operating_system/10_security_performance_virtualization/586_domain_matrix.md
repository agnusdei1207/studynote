### 💡 핵심 인사이트 (Insight)
1. **추상화의 정점**: 보호 도메인(Protection Domain)과 접근 제어 행렬(Access Control Matrix)은 시스템의 모든 보안 규칙을 주체(Subject), 객체(Object), 권한(Access Right)이라는 세 가지 요소로 추상화한 모델입니다.
2. **이론적 근거**: 접근 제어 행렬은 모든 접근 제어 모델(DAC, MAC 등)을 설명할 수 있는 이론적 기반을 제공하며, 시스템 보안 상태를 수학적으로 명확하게 정의합니다.
3. **구현의 효율성**: 행렬은 대규모 시스템에서 희소성(Sparsity) 문제를 가지므로, 이를 해결하기 위해 ACL(Access Control List)이나 Capability List와 같은 실질적인 데이터 구조로 변환하여 구현됩니다.

---

## Ⅰ. 보호 도메인 (Protection Domain)
### 1. 정의 및 개념
보호 도메인(Protection Domain)은 하나의 프로세스가 접근할 수 있는 객체들의 집합과 그 객체에 대해 허용된 연산(권한)을 정의한 논리적인 영역입니다.

### 2. 특징
- **동적 변화**: 프로세스는 실행 중에 한 도메인에서 다른 도메인으로 전환(Domain Switching)될 수 있습니다.
- **최소 권한 보장**: 도메인을 세분화할수록 시스템 오류나 공격 발생 시 피해 범위를 해당 도메인 내로 국한할 수 있습니다.

📢 **섹션 요약 비유**: 보호 도메인은 '호텔 방 카드키'와 같습니다. 내가 가진 카드키가 열 수 있는 방들과 그 방에서 할 수 있는 행동들이 바로 내 도메인입니다.

---

## Ⅱ. 접근 제어 행렬 (Access Control Matrix)
### 1. 정의 및 구조
접근 제어 행렬은 주체(행)와 객체(열)의 교차점에 해당 주체가 객체에 대해 수행할 수 있는 권한을 명시한 2차원 테이블입니다.

### 2. 행렬 모델 (ASCII Diagram)
```text
          |  File A  |  File B  |  Printer  |  Domain Switch
----------+----------+----------+-----------+---------------
 User 1   |  Read    |  Write   |  Print    |  To Domain 2
 User 2   |  None    |  Read    |  None     |  None
 Admin    |  R/W/O   |  R/W/O   |  Manage   |  All
```

### 3. 주요 요소
- **주체 (Subject)**: 사용자, 프로세스, 도메인 등 접근을 시도하는 능동적 실체.
- **객체 (Object)**: 파일, 메모리 영역, 하드웨어 등 보호받아야 할 수동적 실체.
- **권한 (Right)**: Read, Write, Execute, Owner, Control 등.

📢 **섹션 요약 비유**: 접근 제어 행렬은 '학교의 시간표와 담당 구역표'를 합친 것과 같습니다. 어느 반(주체)이 어느 교실(객체)에 들어가서 무엇을 할 수 있는지 한눈에 보여주는 표입니다.

---

## Ⅲ. 행렬의 실제 구현 방식 (Implementation)
### 1. 전역 테이블 (Global Table)
행렬 전체를 하나의 거대한 테이블로 저장하는 방식입니다. 관리는 편하지만, 행렬이 비어있는(None) 경우가 많아 메모리 낭비가 심합니다.

### 2. 접근 제어 목록 (ACL, Access Control List)
- **객체 중심**: 행렬의 **열(Column)**을 기준으로 리스트를 만듭니다.
- **특징**: 특정 파일에 누가 접근 가능한지 확인하기 좋습니다. (대부분의 파일 시스템에서 사용)

### 3. 자격 증명 리스트 (C-List, Capability List)
- **주체 중심**: 행렬의 **행(Row)**을 기준으로 리스트를 만듭니다.
- **특징**: 특정 사용자가 가진 모든 권한(Ticket)을 한꺼번에 관리하기 좋습니다.

📢 **섹션 요약 비유**: 거대한 표를 그대로 들고 다니기 힘드니까, '방문에 명단 붙이기(ACL)'나 '개인에게 티켓 나눠주기(C-List)' 방식으로 효율화하는 것입니다.

---

## Ⅳ. 도메인 전환 (Domain Switching)
프로세스가 더 높은 권한이나 다른 환경의 권한이 필요할 때, 현재 도메인에서 목표 도메인으로 안전하게 전이하는 과정입니다.

- **예시**: UNIX의 `SetUID` 비트 설정 파일 실행 시, 일반 사용자 도메인에서 파일 소유자(Root 등) 도메인으로 일시적으로 전환됩니다.
- **제어**: 도메인 전환 자체도 하나의 권한(Switch Right)으로 행렬 내에서 관리되어야 합니다.

📢 **섹션 요약 비유**: 도메인 전환은 '비행기 일등석으로 업그레이드받는 것'과 같습니다. 특정 자격(티켓)이 있을 때만 일시적으로 더 좋은 자리(권한)로 옮겨갈 수 있습니다.

---

## Ⅴ. 행렬 기반 보안의 한계 및 대안
행렬 모델은 정적(Static)인 성격이 강해, 사용자나 객체가 수만 명에 이르는 현대의 대규모 클라우드 환경에서는 관리가 매우 복잡해집니다. 이를 보완하기 위해 권한을 묶음으로 관리하는 **RBAC(역할 기반)**이나 속성을 따지는 **ABAC(속성 기반)**으로 발전하고 있습니다.

📢 **섹션 요약 비유**: 명단이 수백만 명이 되면 표 하나로 관리하기 힘듭니다. 그래서 '팀별 묶음(RBAC)'이나 '조건별 필터링(ABAC)' 같은 스마트한 방법이 필요합니다.
