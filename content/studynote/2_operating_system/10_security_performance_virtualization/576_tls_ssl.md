# TLS/SSL 보안 프로토콜

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TLS(Transport Layer Security)와 SSL(Secure Sockets Layer)은 **전송 계층에서 통신을 암호화**하여 도청, 변조, 스푸핑을 방지하는 보안 프로토콜이다.
> 2. **가치**: HTTPS, SMTPS, FTPS 등 다양한 애플리케이션에 적용되어 **인터넷 보안의 표준**이 되었으며, 비밀번호, 신용카드 정보 등 민감 데이터를 보호한다.
> 3. **융합**: 공개키 암호화로 키 교환 후 대칭키로 데이터 암호화(하이브리드)하는 **Handshake 프로토콜**을 통해 보안성과 성능의 균형을 실현한다.

+++

## Ⅰ. TLS/SSL의 개요

### 1. 정의
- TLS(SSL)는 TCP/IP 위에서 동작하는 보안 프로토콜로, 애플리케이션 계층(HTTP, SMTP, FTP 등) 데이터를 암호화한다.
- **SSL 1.0~3.0**: 넷스케이프 개발 (현재 모두 폐기, 취약)
- **TLS 1.0~1.3**: IETF 표준화 (SSL의 후속)

### 2. 등장 배경: 평문 통신의 위험
- HTTP는 평문이라 네트워크 상에서 모든 내용 노출
- 공용 와이파이, 네트워크 스니핑, 중간자 공격 위험

### 3. 💡 비유: '봉투에 넣어서 보내는 편지'
- TLS는 **'내용을 볼 수 없는 불투명한 봉투에 넣어 편지를 보내는 것'**과 같다.
- 우편 집배원(중간자)이 봉투를 전달하지만 내용은 볼 수 없다.

- **📢 섹션 요약 비유**: 엽서(평문 HTTP) 대비 밀봉된 편지봉투(TLS)를 사용하면 배달过程中内容이 노출되지 않습니다.

## Ⅱ. TLS 핸드셰이크 과정 (Deep Dive)

### 1. TLS 1.2 핸드셰이크
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │               TLS 1.2 핸드셰이크 과정                            │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │   Client                                            Server      │
    │      │                                                   │       │
    │      │ ① ClientHello                                       │       │
    │      │ - 지원 Cipher Suite                                 │       │
    │      │ - Random (Client)                                   │       │
    │      │ - Session ID                                        │       │
    │      ├───────────────────────────────────────────────────▶│       │
    │      │                                                   │       │
    │      │                                                   │ ② ServerHello
    │      │                                                   │ - 선택 Cipher Suite
    │      │                                                   │ - Random (Server)
    │      │                                                   │ - Session ID
    │      │◀───────────────────────────────────────────────────┤       │
    │      │                                                   │       │
    │      │◀───────────────────────────────────────────────────┤ ③ Certificate
    │      │ - 서버 인증서 (공개키 포함)                          │       │
    │      │                                                   │       │
    │      │◀───────────────────────────────────────────────────┤ ④ ServerKeyExchange
    │      │◀───────────────────────────────────────────────────┤ ⑤ ServerHelloDone
    │      │                                                   │       │
    │      │ ⑥ Certificate Verify (선택)                       │       │
    │      │ - 인증서 검증 (CA 체인, 유효기간, CRL)               │       │
    │      │                                                   │       │
    │      │ ⑦ ClientKeyExchange                               │       │
    │      ├───────────────────────────────────────────────────▶│ - Pre-Master Secret
    │      │ - 서버 공개키로 암호화                               │       │       │
    │      │                                                   │       │
    │      │ ⑧ ChangeCipherSpec                                │       │
    │      ├───────────────────────────────────────────────────▶│       │
    │      │ ⑨ Finished (Handshake 완료)                        │       │
    │      ├───────────────────────────────────────────────────▶│       │
    │      │                                                   │       │
    │      │◀───────────────────────────────────────────────────┤ ⑩ ChangeCipherSpec
    │      │◀───────────────────────────────────────────────────┤ ⑪ Finished
    │      │                                                   │       │
    │      │ ⑫ Application Data (암호화된 통신)                 │       │
    │      │◀───────────────────────────────────────────────────┤       │
    │                                                                 │
    │  * Master Secret = PRF(Pre-Master Secret, Client Random, Server Random)
    └─────────────────────────────────────────────────────────────────┘
```

### 2. TLS 1.3 개선 사항
```text
    ┌─────────────────────────────────────────────────────────────────┐
    │              TLS 1.3의 주요 개선 사항                            │
    ├─────────────────────────────────────────────────────────────────┤
    │                                                                 │
    │  [1단계] 핸드셰이크 단축 (1-RTT)                                │
    │   - ClientHello에 Key Share 포함                                │
    │   - Server가 즉시 응답 가능                                    │
    │   * 1.2는 2-RTT, 1.3은 1-RTT (지연 시간 단축)                    │
    │                                                                 │
    │  [2단계] 취약한 Cipher Suite 제거                               │
    │   - RSA 키 교환 제거 (forward secrecy 미보장)                   │
    │   - static RSA, DH export 제거                                  │
    │   - (E)DHE, (E)CDH만 허용                                      │
    │                                                                 │
    │  [3단계] 암호화된 Server Hello                                 │
    │   - 핸드셰이크 메시지 암호화                                    │
    │   - 패킷 분석 어려움                                           │
    │                                                                 │
    │  [4단계] 0-RTT Resume (재연결 빠름)                            │
    │   - 이전 세션으로 즉시 데이터 전송 가능                         │
    │   * Replay 공격 위험 있어 주의 필요                             │
    └─────────────────────────────────────────────────────────────────┘
```

### 3. Cipher Suite 구조
```
TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
  │     │    │    │    │    │   │     │
  │     │    │    │    │    │   │     └ 해시 함수 (무결성)
  │     │    │    │    │    │   └────── 모드 (GCM: AEAD)
  │     │    │    │    │    └────────── 대칭키 알고리즘
  │     │    │    │    └─────────────── 키 교환 인증 알고리즘
  │     │    │    └──────────────────── 키 교환 방식
  │     │    └───────────────────────── ECDHE (임시 키)
  │     └────────────────────────────── 프로토콜
  └───────────────────────────────────── TLS
```

- **📢 섹션 요약 비유**: 핸드셰이크는 "비밀 언어를 사용할 약속을 정하는 과정"입니다. 암호책(키)을 어떻게 교환할지, 어떤 언어(알고리즘)를 쓸지 미리 합의하는 거죠.

## Ⅲ. TLS/SSL 버전별 특징

| 버전 | 상태 | 주요 특징 | 보안성 |
|:---|:---|:---|:---|
| **SSL 1.0** | ❌ 폐기 | 넷스케이프 첫 버전 (미공개) | 매우 취약 |
| **SSL 2.0** | ❌ 폐기 | 공개 버전 (1995) | 취약 (DROWN) |
| **SSL 3.0** | ❌ 폐기 | RFC 6101 (1996) | 취약 (POODLE) |
| **TLS 1.0** | ⚠️ 단계적 폐기 | RFC 2246 (1999) | 취약 (BEAST) |
| **TLS 1.1** | ⚠️ 단계적 폐기 | RFC 4346 (2006) | 취약 |
| **TLS 1.2** | ✅ 권장 | RFC 5246 (2008) | 안전 |
| **TLS 1.3** | ✅ 최신 | RFC 8446 (2018) | 가장 안전 |

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 웹 서버 TLS 설정 (Apache/Nginx)
```nginx
# Nginx 예시
server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_certificate /etc/ssl/certs/example.com.crt;
    ssl_certificate_key /etc/ssl/private/example.com.key;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers 'ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384';
    ssl_prefer_server_ciphers on;

    # HSTS (HTTP Strict Transport Security)
    add_header Strict-Transport-Security "max-age=31536000" always;
}
```

### 2. 안티패턴
- **"SSL 3.0/TLS 1.0 허용"**
- **"자서명 인증서 사용(내부 시스템 제외)"**
- **"만료된 인증서 방치"**
- **"약한 Cipher Suite 허용"**
- **"HTTP 리다이렉션 누락"**

### 3. 보안 헤더
- **HSTS**: HTTP 접속을 강제로 HTTPS로
- **HPKP**: 공개키 고정 (폐지 권장)
- **CSP**: 콘텐츠 보안 정책

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **데이터 기밀성**: 네트워크 스니핑 방지
- **데이터 무결성**: 중간자 변조 방지
- **서버 인증**: 피싱 사이트 방지

### 2. 미래 전망
- **TLS 1.3 기본**: 빠른 핸드셰이크, 강력한 암호화
- **ECH (Encrypted Client Hello)**: SNI(서버 이름) 암호화
- **QUIC Protocol**: UDP 기반 TLS 1.3
