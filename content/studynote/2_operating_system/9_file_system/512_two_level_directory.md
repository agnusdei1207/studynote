## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 2단계 디렉터리 (Two-level Directory)는 시스템 전체의 마스터 디렉터리 (MFD)와 각 사용자별 사용자 파일 디렉터리 (UFD)로 계층을 분리한 구조다.
> 2. **가치**: 서로 다른 사용자가 동일한 파일 이름을 사용할 수 있게 함으로써 1단계 디렉터리의 고질적인 이름 충돌 (Naming Conflict) 문제를 해결한다.
> 3. **한계**: 사용자 간의 파일 공유 (Sharing)가 어렵고, 파일들을 논리적으로 그룹화 (Grouping)하는 유연성이 여전히 부족하다.

---

## Ⅰ. 개요 (Context & Background)

- **정의**: 시스템이 관리하는 마스터 파일 디렉터리 (MFD, Master File Directory) 아래에, 각 사용자 전용의 사용자 파일 디렉터리 (UFD, User File Directory)를 배치하는 2단계 계층 구조다.
- **배경**: 시분할 시스템 (Time-sharing System)의 등장으로 다중 사용자 환경이 일반화되면서, 사용자 간의 프라이버시를 보호하고 이름 선택의 자유를 보장하기 위해 고안되었다.
- **핵심 용어**:
  - **MFD (Master File Directory)**: 시스템에 로그인한 사용자들의 목록과 각 사용자의 UFD 위치 정보를 담고 있는 최상위 디렉터리.
  - **UFD (User File Directory)**: 특정 사용자가 소유한 파일들의 이름, 주소, 속성 정보를 담고 있는 하위 디렉터리.

📢 **섹션 요약 비유**: 2단계 디렉터리는 "호텔의 프런트 데스크와 개별 객실"과 같다. 프런트(MFD)에서 손님 명단을 확인하고 각자 방(UFD)으로 안내하면, 각 방 안에서는 물건 이름을 마음대로 지어도 옆 방과 섞이지 않는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 계층 구조 시각화 (ASCII Diagram)

```text
 [ Master File Directory (MFD) ]
 ┌──────────┬──────────┬──────────┬──────────┐
 │ User A   │ User B   │ User C   │ ...      │
 └────┬─────┴────┬─────┴────┬─────┴──────────┘
      │          │          │
      ▼          ▼          ▼
 [ UFD A ]    [ UFD B ]    [ UFD C ]
 ┌───────┐    ┌───────┐    ┌───────┐
 │ test.c│    │ data  │    │ test.c│  <-- 서로 다른 사용자간
 │ a.out │    │ doc.tx│    │ main.c│      같은 이름 사용 가능
 └───────┘    └───────┘    └───────┘
```

### 2. 검색 및 접근 메커니즘
- **경로명 (Pathname)**: 파일을 식별하기 위해 `사용자명/파일명` 형태의 경로 형식을 사용하기 시작했다.
- **접근 권한**: 사용자는 자신의 UFD 내 파일만 자유롭게 접근할 수 있으며, 타 사용자의 UFD에 접근하려면 운영체제의 명시적인 허가가 필요하다.
- **검색 효율**: 시스템 전체 파일을 뒤지는 대신, 로그인한 사용자의 UFD만 검색하면 되므로 $O(N)$ 검색에서도 $N$이 획기적으로 줄어든다.

📢 **섹션 요약 비유**: 마치 "아파트의 동·호수 체계"와 같다. '101동 101호'와 '102동 101호'는 같은 번호를 쓰지만 다른 집이듯이, 사용자라는 '동'으로 먼저 범위를 좁히는 방식이다.

---

## Ⅲ. 주요 특징 및 제약 (Features & Constraints)

- **독립성 (Isolation)**: 사용자 A와 사용자 B가 각각 `test.c`라는 파일을 만들어도 UFD가 다르므로 충돌이 발생하지 않는다.
- **공유의 어려움 (Sharing Issue)**: 각 사용자의 디렉터리가 격리되어 있어, 공동 프로젝트를 수행할 때 파일을 서로 공유하거나 링크를 거는 과정이 매우 복잡하다.
- **분류의 한계 (Grouping Problem)**: 사용자 한 명이 수천 개의 파일을 가질 경우, UFD 내부에 하위 폴더를 만들 수 없으므로 여전히 1단계 디렉터리와 같은 관리 효율 저하 문제에 직면한다.

📢 **섹션 요약 비유**: "가족들이 각자의 방을 가진 것"과 같다. 내 물건 정리는 쉬워졌지만, 거실(공유 공간)이 없어서 물건을 빌려주기가 까다로운 상태다.

## Ⅳ. 융합 비교 및 분석 (Comparison)

| 비교 항목 | 2단계 디렉터리 | 트리 구조 디렉터리 |
|:---|:---|:---|
| **계층 깊이** | 고정 2단계 | 가변 깊이 (N단계) |
| **격리 수준** | 사용자 단위 강력 격리 | 논리적 구조에 따른 유연한 격리 |
| **공유 편의성** | 낮음 (OS 수준 제어 필요) | 높음 (공유 디렉터리 생성 가능) |
| **시스템 복잡도** | 낮음 | 높음 (재귀적 탐색 필요) |

📢 **섹션 요약 비유**: 2단계가 "단층 빌라들의 모임"이라면, 트리 구조는 "수백 층 높이의 초고층 빌딩"과 같이 무한한 확장성을 가진다.

## Ⅴ. 기술적 구현 및 사례 (Implementation)

- **로그인 시퀀스**: 운영체제는 사용자가 로그인하면 MFD를 참조하여 해당 사용자의 UFD 주소를 메모리의 **현재 디렉터리 (Current Directory)** 포인터로 설정한다.
- **특수 사용자**: 시스템 관리자나 시스템 전용 프로그램(Compiler 등)을 위한 특수 UFD를 두어 공용 유틸리티를 관리한다.
- **현대적 자취**: 다중 사용자 리눅스 시스템에서 `/home/[username]` 디렉터리 구조는 2단계 디렉터리의 철학을 계층적 트리 구조로 녹여낸 형태라고 볼 수 있다.

📢 **섹션 요약 비유**: 이제는 단독으로 쓰이지 않지만, 모든 현대 운영체제의 "사용자별 홈 디렉터리"라는 철학의 뿌리가 되었다.
