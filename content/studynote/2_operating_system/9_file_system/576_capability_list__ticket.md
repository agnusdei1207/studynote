# # 자격 증명 리스트 (Capability List / Ticket)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 자격 증명 리스트(Capability List)는 **"주체가 소지한 권한(passport)"**으로서, 시스템의 객체(Object)가 아닌 주체(Subject) 측에 "내가 접근 가능한 자원과 허용된 연산"을 리스트 형태로 보관하는 접근 제어 모델이다.
> 2. **가치**: ACL (Access Control List)은 접근 시마다 객체의 명부를 검색해야 하지만, Capability는 **"권한을 가진 자가 증명(Possession)"** 방식이므로 검증만으로 빠르게 접근 허가 여부를 결정할 수 있어 대규모 분산 시스템(Kerberos, OAuth, Microservices)에 최적화되어 있다.
> 3. **융합**: POSIX Capabilities, Kerberos 티켓, OAuth 2.0 Access Token, 웹 쿠키, 쿠버네티스 ServiceAccount 토큰은 모두 Capability의 변형으로서, 현대 클라우드와 웹의 Zero Trust 아키텍처 기반을 이룬다.

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**
자격 증명(Capability)은 컴퓨터 보안에서 **"특정 자원(Object)에 대해 특정 연산(Operation)을 수행할 수 있는 권한을 캡슐화한 불투명한 토큰(Token)"**을 의미한다. 접근 제어 행렬(Access Matrix)의 관점에서 볼 때, ACL (Access Control List)이 객체(Object) 중심으로 "누가 나를 접근할 수 있는가"를 관리하는 열(Column) 기반 저장소라면, **Capability List는 주체(Subject) 중심으로 "내가 무엇을 접근할 수 있는가"를 관리하는 행(Row) 기반 저장소**이다.

수학적으로는 주체 $S$에 대해 다음과 같이 표현된다.
$$C(S) = \{ (O_1, rights_1), (O_2, rights_2), \dots, (O_k, rights_k) \}$$
여기서 핵심은 **"소지(Possession)"**다. 누군가 Capability $C(S)$를 획득했다는 것은, 해당 권한에 대한 증명을 이미 완료했음을 의미하며, 시스템은 이를 다시 검증(Verify)만 하면 된다. 이는 **"무상태성(Stateless)"**을 지향하는 현대 네트워크 아키텍처와 완벽하게 부합한다.

**💡 비유**
Capability는 **"VIP 입장권 티켓"**과 같다.
*   **티켓(Capability)을 가진 사람**: 입구 직원(시스템)에게 "내가 누구인지" 증명할 필요 없이, 티켓만 제시하면 VIP 라운지(자원)에 입장할 수 있다.
*   **ACL(명부) 방식**: 입장 직원이 "VIP 게스트 명부"를 확인하고 "당신이 명부에 있군요, 들어가세요"라고 확인해야 한다. 느리고 복잡하다.
*   단, 티켓을 분실하거나 도난당하면, 그 티켓을 가진任何人(공격자)이 VIP 대우를 받게 된다는 보안 리스크가 존재한다.

**등장 배경 및 진화**
1.  **MULTICS 시스템 (1960s-70s)**: 세그먼트 디스크립터(Segment Descriptor)에 Capability 개념을 최초로 도입. 메모리 주소와 접근 권한(R/W/E)을 하나의 구조체로 묶어 하드웨어적으로 보호하려는 시도.
2.  **CAP 컴퓨터 (1970s)**: 하드웨어적으로 Capability를 위조할 수 없도록 **태그 비트(Tag Bit)**를 메모리 워드에 추가하는 Tagged Architecture 구현. 사용자 프로세스가 Capability를 위조하는 것을 원천 차단.
3.  **현대 분산 시스템 (1990s-현재)**: Kerberos의 **TGT (Ticket Granting Ticket)**, 웹의 **세션 쿠키/세션 ID**, OAuth 2.0의 **Access Token (JWT)** 등으로 진화. 단일 시스템을 넘어 네트워크 전체에서 권한을 증명하는 표준으로 자리 잡음.

ACL과 Capability의 구조적 대조를 시각화하면 다음과 같다. 접근 제어의 "관점(Perspective)"이 어떻게 다른지 확인할 수 있다.

```text
     ┌─────────────────────────────────────────────────────────────────────┐
     │              ACL vs Capability: 접근 제어 관점 비교                  │
     ├─────────────────────────────────────────────────────────────────────┤
     │                                                                     │
     │  [ACL — 객체(Object) 중심 관점]                                     │
     │  ┌───────────────────────────────────────────────────────────────┐  │
     │  │  🔍 Object: File_A (이 파일에게 질문: "너를 누가 열 수 있니?") │  │
     │  │  ────────────────────────────────────────────────────────────│  │
     │  │  • User_A: {Read, Write}                                     │  │
     │  │  • User_B: {Read}                                            │  │
     │  │  • Group_C: {Read}                                           │  │
     │  └───────────────────────────────────────────────────────────────┘  │
     │                                                                     │
     │  [Capability — 주체(Subject) 중심 관점]                             │
     │  ┌───────────────────────────────────────────────────────────────┐  │
     │  │  🔑 Subject: User_A (나에게 질문: "너는 무엇을 할 수 있니?")  │  │
     │  │  ────────────────────────────────────────────────────────────│  │
     │  │  • File_A: {Read, Write}  ← Ticket #1                       │  │
     │  │  • File_B: {Read}         ← Ticket #2                       │  │
     │  │  • Printer_X: {Print}    ← Ticket #3                       │  │
     │  └───────────────────────────────────────────────────────────────┘  │
     │                                                                     │
     └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
ACL은 파일 시스템과 같이 **"자원이 소유자 역할"**을 하는 환경에 유리하다. 파일이 이동하거나 이름이 변경되어도 권한은 파일에 붙어 다니기 때문이다. 반면, Capability는 **"사용자가 권한을 휴대"**하는 형태이므로, 분산 환경에서 사용자가 서버 A에서 서버 B로 이동하며 자원에 접근해야 할 때 매우 유리하다. 사용자는 자신의 능력(Capability)을 가지고 다니며, 어디서든 "나는 이것을 할 수 있는 티켓이 있다"고 증명하면 되기 때문이다.

📢 **섹션 요약 비유**: 마치 관공서 민원실에서 **"발급받은 허가증(증명서)"**을 가지고 창구를 방문하면, 직원이 내 신분을 조회할 필요 없이 증명서만 확인하고 바로 처리해주는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 구성 요소 상세 분석

Capability 시스템은 단순한 리스트가 아니라, 보안 무결성을 보장하기 위한 복잡한 하위 시스템으로 구성된다.

| 요소명 (Component) | 역할 (Role) | 내부 동작 (Internal Operation) | 관련 기술/표준 (Tech/Std) | 비유 (Analogy) |
|:---|:---|:---|:---|:---|
| **Capability Token** | 권한의 실체 | Object Identifier(객체 ID) + Rights(권한 비트) + Tag/Signature(무결성) | POSIX `cap_t`, JWT, Kerberos Ticket | **입장권(Ticket)** |
| **C-List (Capability List)** | 주체의 지갑 | 주체(프로세스)가 보유한 Capability들의 연결 리스트(Linked List) 또는 배열 | Linux Kernel `task_struct->cred` | **티켓 지갑(Wallet)** |
| **Tag Bit / Tagging** | 위조 방지(진품 여부) | 메모리 워드 또는 객체에 특정 비트를 할당하여, Capability와 일반 데이터를 구분 | **CAP Computer**, Sparc Secure Memory, Language-based Tagging | **전자 칩(Hologram)** |
| **Verifier / Guard** | 검사관 | 시스템 콜 진입 시, C-List에 해당 Capability가 있는지, 유효한지 검증 | System Call Handler, API Gateway | **매표원/경비원** |
| **Revocation List** | 무효화 관리 | 취소된 Capability의 ID를 관리(Access 시 검사), 또는 짧은 만료 기간 적용 | CRL (Certificate Revocation List), Token Blacklist | **분실 신고 부(Blacklist)** |

#### 2. Capability의 보호 메커니즘 (Protection Mechanism)

Capability의 치명적인 약점은 **"위조(Forgery)"**다. 공격자가 `{obj: /etc/shadow, perm: RW}`와 같은 Capability를 만들어낼 수 있다면 시스템은 붕괴한다. 이를 막기 위해 3가지 접근법이 존재한다.

```text
     ┌─────────────────────────────────────────────────────────────────────┐
     │           Capability 무결성 보호 메커니즘 (3 Layers)                │
     ├─────────────────────────────────────────────────────────────────────┤
     │                                                                     │
     │  [Layer 1: 하드웨어적 태깅 (Hardware Tagging)]                       │
     │  ┌───────────────────────────────────────────────────────────────┐  │
     │  │  메모리의 각 워드(Word)마다 'Tag Bit'를 추가                  │  │
     │  │  User Mode에서는 Tag Bit를 수정할 수 없도록 하드웨어 Lock      │  │
     │  │                                                               │  │
     │  │  Data: [0xFFFFFFFF | TAG=0]  ← 일반 데이터                   │  │
     │  │  Cap:  [0xABCD1234 | TAG=1]  ← Capability (O)                │  │
     │  │                                                               │  │
     │  │  ⚙️ 예: CAP Computer, B5000, Monitors                         │  │
     │  └───────────────────────────────────────────────────────────────┘  │
     │                           ↕                                        │
     │  [Layer 2: 커널 공간 격리 (Kernel Protection)]                      │
     │  ┌───────────────────────────────────────────────────────────────┐  │
     │  │  Capability 자체는 커널 메모리에 저장                         │  │
     │  │  프로세스는 'Capability Handle(정수 ID)'만 갖음               │  │
     │  │  User Process가 임의의 ID로 syscall 시도 → Kernel이 검증      │  │
     │  │                                                               │  │
     │  │  int fd = open(...);  // fd는 Capability에 대한 핸들          │  │
     │  │  write(fd, ...);     // Kernel가 fd가 유효한지 확인           │  │
     │  │                                                               │  │
     │  │  ⚙️ 예: Linux File Descriptors, POSIX Caps                    │  │
     │  └───────────────────────────────────────────────────────────────┘  │
     │                           ↕                                        │
     │  [Layer 3: 암호학적 서명 (Cryptographic Protection)]                 │
     │  ┌───────────────────────────────────────────────────────────────┐  │
     │  │  Capability = {ObjID, Rights, Expiry, Sig}                    │  │
     │  │  Sig = HMAC(Key, ObjID | Rights | Expiry)                     │  │
     │  │                                                               │  │
     │  │  검증자는 Secret Key(또는 공개키)로 서명만 확인               │  │
     │  │                                                               │  │
     │  │  ⚙️ 예: JWT, Kerberos Ticket, X.509 Certificate              │  │
     │  └───────────────────────────────────────────────────────────────┘  │
     │                                                                     │
     └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1.  **하드웨어적 태깅**: 가장 강력하지만 구현 비용이 비싸다. 데이터와 포인터를 하드웨어 수준에서 분리하여, 사용자가 Capability를 위조할 수 없게 한다.
2.  **커널 격리**: 리눅스(Linux)가 사용하는 방식이다. Capability 자체는 커널이 보관하고, 사용자에게는 그것을 가리키는 `fd`(File Descriptor)나 핸들만 준다. 이는 **간접 참조**를 통해 보안을 유지한다.
3.  **암호학적 서명**: 웹과 분산 시스템의 표준이다. 토큰의 내용을 특정 비밀키로 서명하여, 키를 가진 주체만이 유효한 토큰을 생성할 수 있게 한다. 서버는 키를 공유/공개함으로써 토큰의 진위 여부를 빠르게 확인할 수 있다.

#### 3. Linux POSIX Capabilities 실제 구현

리눅스 커널은 전통적인 **root(Binary: All or Nothing)** 권한을 세분화하여, **"과도한 권한 최소화(Least Privilege)"**를 구현했다.

```text
     ┌─────────────────────────────────────────────────────────────────────┐
     │              Linux Kernel Capabilities (5.x Standard)                │
     ├─────────────────────────────────────────────────────────────────────┤
     │                                                                     │
     │  [Process Capability Sets]                                          │
     │  ┌───────────────────────────────────────────────────────────────┐  │
     │  │                                                               │  │
     │  │  ① Permitted (P):    프로세스가 가질 수 있는 최대 권한 집합   │  │
     │  │                        (보유 가능한 상한선)                    │  │
     │  │                                                               │  │
     │  │  ② Inheritable (I):  execve() 호출 시 자식에게 상속 가능한 집합│  │
     │  │                        (상속 허용 목록)                        │  │
     │  │                                                               │  │
     │  │  ③ Effective (E):    커널이 실제 권한 검사를 수행하는 때 사용  │  │
     │  │                        (현재 활성화된 권한)                    │  │
     │  │                                                               │  │
     │  │  ④ Bounding (B):     이 세트를 넘어서 권한을 획득할 수 없는    │  │
     │  │                        영구 상한선 (하락만 가능)               │  │
     │  │                                                               │  │
     │  │  ⑤ Ambient (A):      Inheritable을 보완, execve 후에도        │  │
     │  │                        유지되는 권한 (Modern Linux)            │  │
     │  └───────────────────────────────────────────────────────────────┘  │
     │                                                                     │
     │  [代表性 Capabilities Example]                                       │
     │  ┌─────────────────────┬──────────────────────────────────────────┐ │
     │  │ CAP_NET_BIND_SERVICE│ 1024번 이하 포트(Port) 바인딩 허용      │ │
     │  │ CAP
