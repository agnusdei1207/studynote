## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 자격 증명 리스트(Capability List)는 **"주체가 소지한 권한(passport)"**으로서, 객체가 아닌 주체 측에 "내가 접근 가능한 자원과 허용된 연산"을 리스트 형태로 보관하는 접근 제어 모델이다.
> 2. **가치**: ACL은 접근 시마다 객체의 명부를 검색해야 하지만, Capability는 **"권한을 가진 자가 증명(Possession)"** 방식이므로 검증만으로 빠르게 접근 허가 여부를 결정할 수 있어 분산 시스템(Kerberos, OAuth)에 적합하다.
> 3. **융합**: POSIX Capabilities, Kerberos 티켓, OAuth 2.0 Access Token, 웹 쿠키는 모두 Capability의 변형으로서, 현대 웹과 클라우드의 인증·인가 기반을 이룬다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 자격 증명 리스트(Capability List)는 접근 제어 행렬(Access Matrix)의 **주체 중심 구현**으로서, 각 주체(사용자, 프로세스)가 자신이 접근 가능한 객체(Object)와 허용된 연산(Operation)을 리스트 형태로 보관하는 모델이다. 수학적으로는 주체 S에 대해 C(S) = {(O₁, p₁), (O₂, p₂), ...} 형태의 쌍(Pair) 집합으로 표현된다. 여기서 **"Capability를 가진 자가 권한을 증명(Possession)"**하므로, Capability를 탈취당하면 공격자는 피해자의 모든 권한을 획득하게 된다.

- **💡 비유**: Capability는 **"VIP 입장권 티켓"**과 같다. 티켓을 가진 사람은 입구에서 자신의 신원을 증명할 필요 없이, 티켓만 제시하면 VIP 라운지에 입장할 수 있다. 반면 ACL은 **"VIP 게스트 명부"**와 같아서, 입구 직원이 명부를 확인하고 "당신이 명부에 있군요, 들어가세요"라고 확인해야 한다. Capability는 **"티켓을 가진 사람이 입장권"**이고, ACL은 **"명부에 이름이 있는 사람이 입장권"**이다.

- **등장 배경**:
  1. **MULTICS 시스템(1960s-70s)**: MULTICS는 Capability라는 용어를 최초로 사용하여, 각 프로세스의 **세그먼트 디스크립터(Segment Descriptor)**에 Capability를 저장했다. 이 디스크립터는 메모리 세그먼트에 대한 포인터와 접근 권한(R/W/E)을 포함했다.
  2. **CAP 컴퓨터(1970s)**: Cambridge CAP 시스템은 Capability를 하드웨어적으로 보호된 자료구조로 구현하여, 사용자 공간 프로세스가 Capability를 위조할 수 없게 만들었다.
  3. **현대 분산 시스템(1990s-현재)**: Kerberos 티켓, OAuth 2.0 Access Token, JWT(JSON Web Token)은 모두 Capability의 개념을 분산 환경으로 확장한 것이다. 클라이언트는 서버에 매번 인증할 필요 없이, 티켓(Capability)을 제시하고 서버는 이를 검증만 하면 된다.

ACL과 Capability의 대조를 시각화하면 다음과 같다. ACL은 객체(파일) 중심이고, Capability는 주체(프로세스) 중심이다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │              ACL vs Capability: 접근 제어 비교                        │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [ACL — 객체 중심 (Access Control List)]                             │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  각 객체(파일)이 "나를 접근 가능한 주체 리스트"를 보관         │  │
 │  │                                                                │  │
 │  │  File1:                                                        │  │
 │  │  ┌──────────────────────────────────────────────────────────┐  │  │
 │  │  │  ACL Entry 1: {uid: alice, perms: R,W}                  │  │  │
 │  │  │  ACL Entry 2: {uid: bob, perms: R}                      │  │  │
 │  │  │  ACL Entry 3: {gid: staff, perms: R}                    │  │  │
 │  │  └──────────────────────────────────────────────────────────┘  │  │
 │  │                                                                │  │
 │  │  검사: alice가 File1에 쓰기 요청                               │  │
 │  │   → File1의 ACL을 로드하여 alice의 권한 확인                  │  │
 │  │   → W 권한 있음 → 허용                                        │  │
 │  │                                                                │  │
 │  │  ✅ "이 파일에 접근 가능한 사람?" — 파일이 답함              │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [Capability — 주체 중심]                                           │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  각 주체(프로세스)가 "내가 접근 가능한 객체 리스트"를 보관     │  │
 │  │                                                                │  │
 │  │  Process alice:                                                │  │
 │  │  ┌──────────────────────────────────────────────────────────┐  │  │
 │  │  │  Capability 1: {object: File1, perms: R,W, token: 0xABCD}│  │  │
 │  │  │  Capability 2: {object: File2, perms: R, token: 0x1234}  │  │  │
 │  │  │  Capability 3: {object: Socket5, perms: RW, token: ...} │  │  │
 │  │  └──────────────────────────────────────────────────────────┘  │  │
 │  │                                                                │  │
 │  │  검사: alice가 File1에 쓰기 요청                               │  │
 │  │   → alice의 Capability List에서 File1 검색                    │  │
 │  │   → Capability 1 발견, W 권한 있음 → 허용                      │  │
 │  │                                                                │  │
 │  │  ✅ "내가 접근 가능한 파일?" — 프로세스가 답을 알고 있음      │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ACL은 **"파일이 자신의 경비원을 고용"**하고, Capability는 **"사용자가 VIP 티켓을 지참"**하는 모델이다. ACL은 접근 시마다 객체의 ACL을 로드하고 검색해야 하므로 오버헤드가 있지만, 권한을 취소(Revoke)하려면 ACL에서 해당 항목을 삭제하면 된다. Capability는 검증만으로 빠르게 접근을 허가할 수 있지만, 권한을 취소하려면 발급된 모든 Capability를 무효화해야 하므로 복잡하다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Capability 토큰** | 권한을 증명하는 불투명 자료구조 | 객체 포인터 + 권한 비트 + 서명/태그 | POSIX cap_t, Kerberos 티켓, JWT | VIP 입장권 |
| **C-List (Capability List)** | 주체가 보유한 Capability의 집합 | 커널 공간 또는 사용자 공간에 연결 리스트로 저장 | current->cred, 프로세스 자격 구조체 | 지갑 속 티켓 모음 |
| **태그 비트(Tag Bit)** | Capability 위조 방지 하드웨어 보호 | 메모리의 각 워드에 태그 비트를 두어, Capability인지 일반 데이터인지 구분 | CAP 컴퓨터, 그 자바(JVM)의 태깅 | 티켓의 진품 여부 마크 |
| **검증자(Verifier)** | Capability의 유효성 검증 | 서명 검증, 만료 시간 확인, 태그 확인 | Kerberos KDC, OAuth 인증 서버 | 입구 티켓 검사원 |
| **무효화(Revocation) 리스트** | 취소된 Capability 추적 | CRL(Revocation List), 만료 시간 기반 취소 | X.509 CRL, 토큰 블랙리스트 | 분실 신고된 티켓 번호부 |

### Capability의 구조와 보호

Capability는 단순한 포인터가 아니라, **"보호된 포인터(Protected Pointer)"**로서, 다음 세 가지 요소를 반드시 포함해야 한다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                    Capability의 구조와 보호                           │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [기본 Capability 구조]                                              │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                 │  │
 │  │  ┌────────────────────────────────────────────────────────┐    │  │
 │  │  │  Object Pointer  │  Permissions  │  Tag/Signature     │    │  │
 │  │  │  (64-bit)        │  (R, W, X)    │  (Crypto/Tag)      │    │  │
 │  │  └────────────────────────────────────────────────────────┘    │  │
 │  │         │                  │                  │               │  │
 │  │         ▼                  ▼                  ▼               │  │
 │  │   • 접근할 객체의    • 이 Capability로    • 위조 방지        │  │
 │  │     주소(또는 ID)      가능한 연산       • 태그 비트 또는     │  │
 │  │                         • Read: 0x01      암호학적 서명      │  │
 │  │                         • Write: 0x02                        │  │
 │  │                         • Execute: 0x04                      │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [위조 방지 메커니즘]                                                │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                 │  │
 │  │  ① 하드웨어적 태그 비트 (Tagged Architecture)                  │  │
 │  │  ┌──────────────────────────────────────────────────────────┐  │  │
 │  │  │  메모리의 각 워드(32/64비트)에 1~3비트 태그 추가          │  │  │
 │  │  │                                                          │  │  │
 │  │  │  데이터: [0xFFFFFFFF│TAG=0]  ← 일반 데이터               │  │  │
 │  │  │  Capability: [0xABCD1234│TAG=1] ← Capability!            │  │  │
 │  │  │                                                          │  │  │
 │  │  │  • TAG=1인 워드는 Capability로만 간주                    │  │  │
 │  │  │  • 사용자 공간에서 TAG 비트 수정 불가 (하드웨어 보호)    │  │  │
 │  │  │  • CAP 컴퓨터, IBM System/38, 그 자바(JVM)의 태깅       │  │  │
 │  │  └──────────────────────────────────────────────────────────┘  │  │
 │  │                                                                 │  │
 │  │  ② 암호학적 서명 (Software Protection)                         │  │
 │  │  ┌──────────────────────────────────────────────────────────┐  │  │
 │  │  │  Capability = {obj_id, perms, expire, signature}        │  │  │
 │  │  │                                                          │  │  │
 │  │  │  • signature = HMAC(obj_id ‖ perms ‖ expire, K_secret)  │  │  │
 │  │  │  • 검증자는 K_secret으로 서명 재계산 후 비교             │  │  │
 │  │  │  • Kerberos 티켓, OAuth JWT, AWS STS Token              │  │  │
 │  │  └──────────────────────────────────────────────────────────┘  │  │
 │  │                                                                 │  │
 │  │  ③ 커널 공간 격리 (Kernel Protection)                         │  │
 │  │  ┌──────────────────────────────────────────────────────────┐  │  │
 │  │  │  POSIX Capabilities, Linux File Descriptor              │  │  │
 │  │  │                                                          │  │  │
 │  │  │  • Capability은 커널 공간에만 저장 (사용자가 위조 불가) │  │  │
 │  │  │  • 프로세스는 "Capability 핸들(정수)"만 보유             │  │  │
 │  │  │  • 시스템 콜 시 커널이 핸들을 실제 Capability로 변환     │  │  │
 │  │  └──────────────────────────────────────────────────────────┘  │  │
 │  │                                                                 │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  위조 방지가 없으면 공격자가 임의의 Capability를 위조하여             │
 │  모든 자원에 접근 가능 → 완전한 시스템 장악 가능                     │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Capability의 가장 큰 보안 문제는 **"위조"**다. 공격자가 `{object: /etc/shadow, perms: RW}` 같은 Capability를 위조하면, 시스템의 모든 보호가 붕괴된다. 이를 방지하는 세 가지 방법이 있다:
1. **하드웨어적 태그 비트**: 메모리의 각 워드마다 태그 비트를 추가하여, TAG=1인 워드만 Capability로 간주한다. 사용자 공간에서는 TAG 비트를 수정할 수 없으므로 위조가 불가능하다.
2. **암호학적 서명**: Kerberos 티켓처럼 KDC(키 분배 센터)의 비밀키로 서명된 토큰을 사용한다. 공격자는 비밀키 없이 위조된 서명을 생성할 수 없다.
3. **커널 공간 격리**: POSIX Capabilities처럼 Capability 자체를 커널 공간에 보관하고, 사용자 공간에는 핸들(정수)만 노출한다. 리눅스 파일 디스크립터(fd)가 이 모델을 따른다.

### POSIX Capabilities — 리눅스의 Capability 구현

리눅스 커널은 POSIX 1003.1e 표준의 Capabilities를 구현하여, root 권한을 개별 capability 단위로 분할했다. 이는 **"전능(All-powerful)한 root"를 최소 권한 집합으로 분해**한 것이다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │            POSIX Capabilities — 리눅스 구현                          │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [전통적 모델: UID 기반]                                             │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  if (uid == 0) { // root                                       │  │
 │  │      return ALLOW; // 모든 것 허용                             │  │
 │  │  } else {                                                       │  │
 │  │      return DENY;  // 모든 것 거부                             │  │
 │  │  }                                                              │  │
 │  │                                                                │  │
 │  │  문제: root 하나만 탈취되어도 시스템 전체 장악                  │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [Capability 기반 모델]                                             │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  주요 Capabilities (Linux 5.x 기준 ~40개)                     │  │
 │  │  ┌───────────────────────┬──────────────────────────────────┐  │  │
 │  │  │ CAP_CHOWN            │ 파일 소유자 변경                │  │  │
 │  │  │ CAP_DAC_OVERRIDE     │ DAC(rwx) 검사 우회               │  │  │
 │  │  │ CAP_DAC_READ_SEARCH  │ 파일 읽기/디렉터리 검색 우회      │  │  │
 │  │  │ CAP_FOWNER           │ 파일에 대한 ID 기반 검사 우회     │  │  │
 │  │  │ CAP_FSETID           │ set-user-ID/set-group-ID 비트 설정│  │  │
 │  │  │ CAP_KILL             │ 임의 프로세스 시그널 전송        │  │  │
 │  │  │ CAP_SETGID           │ GID 조작 (setgid, setgroups)     │  │  │
 │  │  │ CAP_SETUID           │ UID 조작 (setuid, setfsuid)      │  │  │
 │  │  │ CAP_SETPCAP          │ Capability 세트 조작             │  │  │
 │  │  │ CAP_LINUX_IMMUTABLE  │ 불변(immutable) 파일 속성 변경    │  │  │
 │  │  │ CAP_NET_BIND_SERVICE │ 1024번 이하 포트 바인딩          │  │  │
 │  │  │ CAP_NET_BROADCAST    │ 브로드캐스트 소켓 생성          │  │  │
 │  │  │ CAP_NET_ADMIN        │ 네트워크 관리 (방화벽, 라우팅)   │  │  │
 │  │  │ CAP_NET_RAW          │ RAW 소켓 생성 (ICMP, etc)       │  │  │
 │  │  │ CAP_SYS_MODULE       │ 커널 모듈 로드/언로드            │  │  │
 │  │  │ CAP_SYS_ADMIN        │ 시스템 관리 (mount, swap, etc)  │  │  │
 │  │  │ CAP_SYS_BOOT         │ 시스템 재부팅 (reboot)          │  │  │
 │  │  │ CAP_SYS_CHROOT       │ chroot(2) 수행                  │  │  │
 │  │  │ CAP_SYS_TIME         │ 시스템 시간 변경                │  │  │
 │  │  │ CAP_SYS_TTY_CONFIG   │ tty 장치 설정                   │  │  │
 │  │  │ CAP_MKNOD            │ 특수 파일(s/device) 생성       │  │  │
 │  │  └───────────────────────┴──────────────────────────────────┘  │  │
 │  │                                                                │  │
 │  │  프로세스의 Capability 세트                                    │  │
 │  │  ┌──────────────────────────────────────────────────────────┐  │  │
 │  │  │  Permitted (P):  보유 가능한 최대 Capability 집합        │  │  │
 │  │  │  Effective (E):   현재 활성화된 Capability (실제 검사용)   │  │  │
 │  │  │  Inheritable (I): 자식 프로세스에게 상속 가능한 집합     │  │  │
 │  │  │  Bounding (B):   이 이상으로 상승 불가능한 상한선        │  │  │
 │  │  │  Ambient (A):    execve(2) 후에도 보존되는 Capability    │  │  │
 │  │  └──────────────────────────────────────────────────────────┘  │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  예: 웹 서버는 CAP_NET_BIND_SERVICE만 보유하여 80번 포트에 바인딩    │
 │      가능하지만, CAP_SYS_ADMIN이 없어서 mount() 시스템 콜 차단      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 리눅스 커널의 `capable()` 함수는 `current->cred->cap_effective` 비트맵을 검사하여, 요청된 capability가 있는지 확인한다. 예를 들어 `inet_bind()` 함수는 1024번 이하 포트에 바인딩할 때 `capable(CAP_NET_BIND_SERVICE)`를 호출한다. 웹 서버를 실행할 때 `setcap cap_net_bind_service+ep /usr/sbin/nginx`를 실행하면, nginx 프로세스는 `CAP_NET_BIND_SERVICE`만 보유하게 되어, 웹 서버가 해킹당하더라도 `mount()`나 `insmod()` 같은 위험한 시스템 콜을 호출할 수 없게 된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: ACL vs Capability

| 비교 항목 | ACL (Access Control List) | Capability List |
|:---|:---|:---|
| **저장 위치** | 객체의 메타데이터(inode) | 주체의 자격(Credentials) |
| **검사 주체** | 객체 소유자/커널이 관리 | 주체가 소지하고 증명 |
| **권한 취소(Revocation)** | 간단(ACL에서 제거) | 어려움(모든 티켓 찾아야 함) |
| **위변조 방지** | 커널이 보호 | 태그 비트 또는 암호학적 서명 필요 |
| **전파(Delegation)** | 불가능(소유자만 변경) | 가능(티켓을 타인에게 양도) |
| **검사 오버헤드** | 접근 시마다 ACL 순회 | 티켓 검증만으로 빠름 |
| **분산 환경 적합성** | 중앙 집중형 ACL DB 필요 | 티켓으로 분산 검증 가능 |
| **대표 예시** | 리눅스 파일 권한(rwx), NFS ACL | Kerberos 티켓, OAuth Token, POSIX Cap |

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │            ACL vs Capability: 검사 흐름 비교                          │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [ACL 검사 — 파일 시스템 예시]                                       │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  alice가 File1에 쓰기 요청                                     │  │
 │  │                                                                │  │
 │  │  ┌──────────────────────────────────────────────────────────┐  │  │
 │  │  │  VFS: inode_permission() 호출                           │  │  │
 │  │  │         │                                                │  │  │
 │  │  │         ▼                                                │  │  │
 │  │  │  ┌────────────────────────────────────────────────────┐  │  │  │
 │  │  │  │  inode->i_acl 순회                                  │  │  │  │
 │  │  │  │  Entry 1: {uid: alice, perms: R,W}  ← 매칭!         │  │  │  │
 │  │  │  │  Entry 2: {uid: bob, perms: R}                     │  │  │  │
 │  │  │  │  ...                                               │  │  │  │
 │  │  │  └────────────────────────────────────────────────────┘  │  │  │
 │  │  │         │                                                │  │  │

 │  │  └──────────────────────────────────────────────────────────┘  │  │
 │  │                                                                │  │
 │  │  ⏱ O(K) 시간 (K = ACL 엔트리 개수)                           │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [Capability 검사 — Kerberos 티켓 예시]                             │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  alice가 File Server에 접근 요청                              │  │
 │  │                                                                │  │
 │  │  ┌──────────────────────────────────────────────────────────┐  │  │
 │  │  │  1. alice가 KDC에서 TGT 획득 (비밀키로 인증)             │  │  │
 │  │  │  2. alice가 TGT로 File Server용 티켓(TGS) 요청          │  │  │
 │  │  │  3. KDC는 alice의 ID 확인 후 Service Ticket 발급          │  │  │
 │  │  │     Ticket = {client: alice, server: filesrv,            │  │  │
 │  │  │                perms: RW, expire: 8h, sig: ...}          │  │  │
 │  │  │  4. alice가 File Server에 Ticket 제시                     │  │  │
 │  │  │  5. File Server는 서명을 검증하고 perms: RW 확인 → 허용  │  │  │
 │  │  └──────────────────────────────────────────────────────────┘  │  │
 │  │                                                                │  │
 │  │  ⏱ O(1) 시간 (서명 검증만으로 허가 여부 결정)                  │  │
 │  │  ✅ File Server은 KDC와 통신하지 않아도 됨 (Stateless)       │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ACL은 접근 시마다 객체의 ACL을 순회하므로 O(K) 시간이 걸린다. Capability는 **"티켓을 한 번 검증하면 유효"**하므로 O(1) 시간에 접근을 허가할 수 있다. 특히 **분산 환경에서의 Stateless 검증**이 Capability의 가장 큰 장점이다. Kerberos 티켓은 서버가 KDC와 통신하지 않고도 독립적으로 검증할 수 있으므로, 수천 대의 파일 서버가 있어도 각 서버가 독립적으로 티켓을 검증할 수 있다.

### 비교 2: Kerberos vs OAuth 2.0

| 비교 항목 | Kerberos | OAuth 2.0 |
|:---|:---|:---|
| **Capability 형태** | 티켓(Ticket) - 암호화된 바이너리 | Access Token - JWT 문자열 |
| **발급자** | KDC (Key Distribution Center) | Authorization Server |
| **검증 방식** | KDC 비밀키로 서명 검증 | RS256/ECDSA 공개키로 서명 검증 |
| **사용 예시** | Active Directory, MIT Kerberos | Google Login, GitHub OAuth |
| **용도** | 기업 내부 SSO (Single Sign-On) | 웹/모바일 타사 인증 |
| **만료 처리** | 티켓 갱신(Renewable) 가능 | Refresh Token으로 재발급 |
| **보안 모델** | Symmetric Key (KDC와 공유) | Asymmetric Key (공개키로 검증) |

Kerberos는 1980년대 MIT에서 개발된 네트워크 인증 프로토콜로서, **티켓(Ticket)** 이라는 Capability를 사용한다. KDC(Key Distribution Center)는 사용자를 인증한 후 서비스 티켓을 발급하며, 이 티켓은 KDC의 비밀키로 암호화되어 있다. 서버는 티켓을 복호화하여 사용자의 신원과 권한을 확인한다. OAuth 2.0은 이 패턴을 웹으로 확장한 것으로서, Authorization Server가 발급한 **Access Token(JWT)** 을 클라이언트가 제시하면, Resource Server는 토큰의 서명을 공개키로 검증하여 접근을 허가한다.

OAuth 2.0은 **Asymmetric Key(공개키)** 기반이라서, Authorization Server가 발급한 JWT 토큰은 자신의 개인키로 서명한다. Resource Server(Google Drive, GitHub API 등)는 Authorization Server의 공개키를 가지고 있으므로, 독립적으로 토큰의 서명을 검증할 수 있다. 이것이 **"Capability의 분산 검증"** 장점이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 웹 서버의 Capability 제한**: 웹 서버 nginx를 root가 아닌 `www-data` 사용자로 실행하되, `setcap cap_net_bind_service+ep /usr/sbin/nginx`를 실행하여 `CAP_NET_BIND_SERVICE`만 부여했다. 이제 nginx는 80번 포트에 바인딩할 수 있지만, `mount()`, `insmod()`, `chmod()` 등 다른 특권 작업은 불가능하다. 웹 서버가 해킹당하더라도 공격자는 커널 모듈을 로드하거나 파일 시스템을 마운트하는 등의 추가 피해를 입힐 수 없게 된다.

2. **시나리오 — Kubernetes Service Account 토큰**: 쿠버네티스의 Pod는 자동으로 Service Account용 JWT 토큰을 `/var/run/secrets/kubernetes.io/serviceaccount/token`에 마운트받는다. 이 토큰은 API Server의 공개키로 서명되어 있어, Pod 내부의 애플리케이션은 이 토큰을 HTTP Bearer 헤더에 실어 API Server에 요청을 보낼 수 있다. API Server는 토큰의 서명을 검증하여 **"이 Pod는 어떤 네임스페이스의 어�은 Service Account인가?"** 를 확인하고 RBAC 규칙에 따라 접근을 허가/거부한다.

3. **시나리오 — OAuth 2.0 / OpenID Connect 연동**: SaaS 웹 애플리케이션이 "Google로 로그인" 기능을 구현했다. 사용자가 Google 로그인 버튼을 클릭하면, Google의 Authorization Server로 리다이렉트되어 로그인을 수행하고, Authorization Code를 발급받는다. 백엔드는 이 코드를 Access Token으로 교환하고, Access Token을 사용하여 Google Drive API를 호출한다. Google Drive API는 토큰의 서명을 Google의 공개키로 검증하여, **"이 토큰은 정상적인 사용자의 것"**임을 확인하고 API를 실행한다.

### 도입 체크리스트
- **기술적**: root로 실행되는 모든 데몬을 식별하고, 개별 Capability로 분할 가능한지 검토했는가? `getpcaps <pid>`로 프로세스의 Capability를 확인했는가?
- **운영·보안적**: Kerberos 티켓/OAuth 토큰의 만료 시간을 8시간 이하로 설정했는가? Refresh Token은 안전하게 저장(HTTP-Only 쿠키)하고 있는가?

### 안티패턴
- **티켓 누수(Token Leakage)**: OAuth Access Token을 URL 쿼리 파라미터(`?access_token=xxx`)로 전달하면, 브라우저 기록, 프록시 로그, Referer 헤더에 토큰이 노출된다. 반드시 Authorization Bearer 헤더로 전달해야 한다.
- **Capability 만료 없음**: Kerberos 티켓이나 OAuth 토큰에 만료 시간을 설정하지 않으면, 티켓이 탈취당했을 때 **무기한 유효**하게 되어 영구적인 권한 노출이 발생한다. 반드시 짧은 만료 시간(예: 1~8시간)을 설정하고, Refresh Token으로 갱신하게 해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 | 최적화 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | root UID로 모든 권한 보유 | 개별 Capabilities 분할 | **탈취 시 권한 범위 90%+ 축소** |
| **정량** | 중앙 인증 서버에 매번 요청 | 티켓/토큰으로 분산 검증 | **인증 서버 부하 70%+ 감소** |
| **정성** | 권한 취소가 어려움 | 만료 시간 기반 자동 취소 | **보안 관리 자동화** |

### 미래 전망
- **DPoP (Demonstrating Proof-of-Possession)**: OAuth 2.0의 새로운 표준 RFC 9449로서, Access Token에 공개키를 바인딩하여, 토큰 탈취 시에도 개인키가 없으면 사용 불가능하게 만든다.
- **Selective Disclosure Credential**: 영지식 증명(ZK-proof) 기반의 Capability로서, "나는 성인입니다"라는 사실만 증명하고 생년월일은 노출하지 않는 등, **최소 정보 공개**를 가능하게 한다.

### 참고 표준
- **RFC 4120 (Kerberos v5)**: 네트워크 인증 프로토콜 표준
- **RFC 6749 / RFC 6750 (OAuth 2.0)**: 위임된 권한 부여를 위한 프레임워크
- **RFC 7519 (JWT - JSON Web Token)**: 토큰 형식 표준
- **POSIX 1003.1e**: Capabilities 표준 (Draft)

## 👶 어린이를 위한 3줄 비유 설명
1. Capability는 **"놀이공원 VIP 입장권"**이에요. 입장권을 가진 아이는 줄을 서지 않고 바로 입장할 수 있어요.
2. 입구 직원은 입장권의 진짜 여부만 확인하면 되고, 아이가 누구인지 물어볼 필요가 없어요. 입장권만 있으면 무조건 통과!
3. 하지만 입장권을 잃어버리면 누구든지 그 입장권을 주운 사람이 VIP 대우를 받게 돼요. 그래서 입장권을 아주 조심하게 다뤄야 해요.
