## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서버리스(Serverless) 컴퓨팅은 본래 무상태(Stateless)를 지향하지만, 복잡한 로직 처리를 위해 외부 파일 시스템(EFS 등)을 마운트하여 영구적인 상태(State)를 유지하는 기술적 진화를 이루었다.
> 2. **해결 과제**: 임시 파일 공간인 `/tmp`는 실행이 끝나면 휘발되는 한계가 있으며, 대용량 데이터 처리나 라이브러리 공유를 위해 네트워크 기반 파일 시스템과의 고속 연결이 필수적이다.
> 3. **가치**: 서버리스의 확장성(Scalability)과 파일 시스템의 지속성(Persistence)을 결합하여, 머신러닝 추론, 대규모 이미지 처리 등 무거운 워크로드를 서버 관리 없이 수행할 수 있게 한다.

---

## Ⅰ. 서버리스의 무상태성과 파일 시스템의 필요성 (Context)

- **Stateless Principle**: 서버리스 함수(Lambda 등)는 호출 시 생성되고 완료 시 소멸되므로, 내부 메모리나 디스크에 저장된 정보는 보존되지 않는다.
- **상태 유지가 필요한 경우**:
  1. **대용량 파일 처리**: 메모리 한계를 초과하는 수 GB 단위의 데이터를 처리할 때.
  2. **체크포인팅**: 긴 연산 과정 중 중간 결과를 저장하고 이어서 작업할 때.
  3. **공유 라이브러리**: 여러 함수가 공통으로 사용하는 거대한 모델 파일이나 바이너리를 공유할 때.

> **📢 섹션 요약 비유**: 서버리스는 "주문할 때마다 새로 생기는 임시 주방"과 같습니다. 요리가 끝나면 주방이 사라지기 때문에, 남은 식재료나 조리법을 보관하려면 "주방 밖에 있는 커다란 공용 창고(파일 시스템)"가 필요합니다.

---

## Ⅱ. 서버리스 스토리지 계층 구조 (Technical Layers)

### 1. 스토리지 옵션 비교 ASCII 다이어그램
```text
[ Serverless Function ]
      |
      +-- [ Memory ] (Volatile, Fastest, MB unit)
      |
      +-- [ /tmp Storage ] (Ephemeral, Fast, GB unit)
      |
      +-- [ Mounted FS (EFS) ] (Persistent, Shared, TB unit) <-- Shared State
```

### 2. 기술적 특징
- **Ephemeral Storage (/tmp)**:
  - 함수 인스턴스 로컬 디스크 활용.
  - 빠르지만 함수 호출 간 공유가 불가능하고 용량이 제한적임.
- **Network File System (EFS 마운트)**:
  - VPC를 통해 함수와 파일 시스템을 연결.
  - 여러 인스턴스가 동시에 같은 파일을 읽고 쓸 수 있어 강력한 상태 공유 가능.

> **📢 섹션 요약 비유**: `/tmp`는 "요리사의 개인 앞치마 주머니"이고, `Mounted FS`는 "주방 식구들 모두가 같이 쓰는 커다란 냉장고"입니다.

## Ⅲ. 성능 최적화와 콜드 스타트 (Performance & Cold Start)

- **Cold Start 영향**: 파일 시스템 마운트 과정이 추가되면 함수 실행 지연(Latency)이 발생할 수 있다.
- **최적화 전략**:
  1. **Provisioned Concurrency**: 인스턴스를 미리 띄워 마운트 상태를 유지.
  2. **Keep-alive**: 연결을 재사용하여 TCP 핸드쉐이크 및 마운트 오버헤드 감소.
  3. **Sparse Loading**: 필요한 파일 부분만 읽어 들이는 기법 사용.

> **📢 섹션 요약 비유**: 성능 최적화는 "요리를 시작하기 전에 미리 냉장고 문을 열어두고 재료 위치를 파악해 두는 것"과 같습니다.

---

## Ⅳ. 동시성 제어와 일관성 모델 (Concurrency & Consistency)

- **Write Conflict**: 수천 개의 서버리스 함수가 동시에 같은 파일에 쓰기를 시도할 때 발생하는 문제.
- **해결책**:
  - **Advisory Locking**: 파일 잠금을 통해 쓰기 권한 제어.
  - **Append-only Design**: 파일을 수정하는 대신 새로운 내용을 추가하는 방식 선호.
  - **Event-driven Trigger**: 파일 생성이 완료된 후 다음 함수를 호출하는 워크플로우 설계.

> **📢 섹션 요약 비유**: 동시성 제어는 "하나의 도화지에 여러 명이 동시에 그림을 그릴 때 서로 손이 부딪히지 않도록 순서를 정해주거나 구역을 나누는 것"과 같습니다.

---

## Ⅴ. 서버리스 데이터 처리의 진화 (Future)

- **서버리스 머신러닝 (Serverless ML)**: 수백 MB의 모델 파일을 파일 시스템에 두고, 초 단위로 호출되는 함수들이 이를 로드하여 즉시 추론 수행.
- **데이터 공유 파이프라인**: 단계별 서버리스 함수들이 파일 시스템을 거점으로 데이터를 주고받는 복합 파이프라인 구축.
- **Global Serverless Storage**: 전 세계 여러 리전에 흩어진 서버리스 함수들이 하나의 일관된 파일 시스템을 바라보는 기술 연구 중.

> **📢 섹션 요약 비유**: 미래에는 "어디서나 나타나는 요리사(서버리스)가 전 세계 어디에나 있는 창고(파일 시스템)를 자기 주방처럼 편하게 쓰는 시대"가 올 것입니다.
