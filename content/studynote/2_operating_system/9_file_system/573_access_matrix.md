# # [접근 제어 행렬 (Access Matrix)]

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: **접근 제어 행렬 (Access Matrix)**는 컴퓨터 보안의 기초가 되는 수학적 모델로, **주체 (Subject)**와 **객체 (Object)**의 2차원 교차점에 **권한 (Access Right)**을 배치하여 시스템의 보안 상태를 완벽하게 정의하는 형식 모델(Formal Model)이다.
> 2. **가치**: 직관적인 구조를 제공하지만, 실제 시스템에서 발생하는 **희소 행렬 (Sparse Matrix)** 문제(대부분의 셀이 비어있음)를 해결하기 위해 **ACL (Access Control List)**과 **CL (Capability List)**이라는 두 가지 상호 보완적인 구현 전략을 제시하며, 이는 현대 **OS (Operating System)** 및 **DBMS (Database Management System)** 보안 아키텍처의 표준이 되었다.
> 3. **융합**: 단순한 파일 시스템 보안을 넘어, **분산 시스템(Kerberos)**, **컨테이너 오케스트레이션(Kubernetes RBAC)**, **마이크로서비스 권한 관리** 등 자원 보호가 필요한 모든 영역에서 "누가 무엇을 할 수 있는가"를 정의하는 보편적인 언어(Universal Language)로 작동한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 접근 제어 행렬은 1971년 Butler Lampson이 제안한 추상화된 보호 모델이다. 수학적으로는 행렬 $M$은 $M: S \times O \rightarrow 2^P$로 정의된다. 여기서 $S$는 주체(행, Row)의 집합, $O$는 객체(열, Column)의 집합, $P$는 권한의 집합(e.g., Read, Write, Execute)이다. 이 모델의 핵심은 **보안 상태(Security State)**를 시스템 전체의 한 곳에서 집중적으로 관리하려는 시도에 있다. 시스템의 모든 보안 정책은 이 행렬의 상태 변화(State Transition)로 요약된다.

- **💡 비유**: 이는 **"거대한 건물 복합체의 중앙 보안 통제실"**과 같다. 건물의 모든 방(객체)과 모든 출입권을 가진 사람(주체)에 대한 정보가 하나의 거대한 전자 게시판(행렬)에 real-time으로 표시된다. 경비원(커널)은 누군가 문을 열려고 하면 이 게시판을 보고 "허가" 또는 "거부"를 결정한다.

- **등장 배경**:
  1. **자원 공유와 격리의 딜레마 (1960s-70s)**: 초기 **시분할 시스템 (Time-sharing System)**에서는 다수의 사용자가 메인프레임 자원(CPU, 메모리, 파일)을 공유하면서도, 서로의 민감한 데이터를 침해하지 못하도록 하는 **격리 (Isolation)** 메커니즘이 절실했다.
  2. **접근 제어의 추상화 필요성**: 프로그래머가 보안 로직을 매번 "If user == A then allow..."와 같이 하드코딩하는 것은 오류 prone했다. 이를 행렬이라는 수학적 구조로 분리하여, 보안 정책을 비즈니스 로직과 독립적으로 설계하고 검증할 수 있는 이론적 기반이 필요했다.
  3. **UNIX의 실용적 구현과 압축 (1970s)**: 완벽한 행렬을 저장하기엔 당대 메모리가 비싸고 접근 속도가 느렸기에, UNIX는 이를 **Owner/Group/Other** 3개의 그룹과 **rwx** 9비트로 압축하는 실용적 타협안을 선택했다. 이는 접근 제어 행렬의 메모리 오버헤드를 최소화한 첫 번째 성공 사례였다.

```text
 ┌─────────────────────────────────────────────────────────────────────────┐
 │                     접근 제어 행렬 (Access Matrix) 개념도               │
 ├─────────────────────────────────────────────────────────────────────────┤
 │                                                                         │
 │   Subject(행) : Process A, Process B, User C                           │
 │   Object(열)  : File 1, File 2, Printer, Memory Seg                    │
 │                                                                         │
 │      ┌──────────────────────────────────────────────────────────────┐   │
 │      │      File 1       File 2      Printer    Memory Seg 1        │   │
 │      ├──────────────────────────────────────────────────────────────┤   │
 │ Proc A│ Own, R, W       R            W             R, W             │   │
 │ Proc B│    R           Own, W        —             —               │   │
 │ User C│    R              R           W             Execute         │   │
 │      └──────────────────────────────────────────────────────────────┘   │
 │                                                                         │
 │   [Legend]                                                              │
 │   • Own  : 소유자 권한 (권한 부여/삭제 가능)                            │
 │   • R/W  : 읽기/쓰기 권한                                              │
 │   • —    : 권한 없음 (No Access)                                        │
 │                                                                         │
 └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
이 다이어그램은 보호 모델의 가장 기초적인 형태를 보여준다.
1. **차원의 의미**: 행(Row)은 '능동적인 개체(Actor)'를, 열(Column)은 '수동적인 개체(Target)'를 상징하며, 이 둘의 교차점(Cross-point)이 바로 보안의 핵심 지점이다.
2. **권한의 집합**: 셀(Cell)에 있는 값은 단순한 불리언(Boolean)이 아니라 연산의 집합(Set)이다. 예를 들어 `Proc A`는 `File 1`에 대해 `{Own, Read, Write}`라는 권한 집합을 가지므로, 파일을 읽을 뿐만 아니라 다른 사용자에게 권한을 줄 수 있는 제어권까지 갖는다.
3. **희소성 (Sparsity)**: 실제 엔터프라이즈 시스템에서는 사용자 수와 자원 수가 수만 단위로 늘어난다. 이때 대부분의 사용자는 대부분의 자원에 접근할 필요가 없으므로, 행렬의 99% 이상이 비어있거나(—) 기본값만 가지게 된다. 이러한 **희소 행렬(Sparse Matrix)** 문제를 해결하기 위해 메모리를 효율적으로 쓰는 방법(ACL, Capability)이 고안되었다.

**📢 섹션 요약 비유**: 접근 제어 행렬은 마치 **"모든 교실 출입문에 부착된 전자 출입명단과 모든 학생이 소지한 지문 인식 카드 정보를 통합 관리하는 학교 보안 시스템의 중앙 서버"**와 같다. 이 하나의 행렬만 보면 학교 전체의 보안 현황을 단 번에 파악할 수 있는 전지전능한 지도(Map)를 제공한다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

접근 제어 행렬의 단순함은 강력하지만, 실제 컴퓨터 시스템의 메모리 구조 및 검증 알고리즘으로 구현할 때는 심각한 성능 저하 문제를 야기할 수 있다. 본 섹션에서는 이를 해결하는 두 가지 주요 아키텍처와 동적 권한 변경 메커니즘을 심층 분석한다.

#### 1. 구성 요소 및 내부 동작

| 요소명 | 역할 | 내부 동작 및 데이터 구조 | 프로토콜/표준 | 비유 |
|:---|:---|:---|:---|:---|
| **주체 (Subject)** | 접근을 요청하는 능동적 개체 | UID(User ID), PID(Process ID), Security Token | POSIX UID, Windows SID | 출입을 시도하는 사람 |
| **객체 (Object)** | 접근의 대상이 되는 수동적 개체 | inode 번호, 메모리 주소, Socket FD, Object ID | File, IPC, Device | 건물의 특정 방 |
| **권한 (Access Right)** | 수행 가능한 연산의 집합 | 비트 마스크(Bitmask), ACE(Access Control Entry), Tuple | rwx, ACL Flags, DACL/SACL | 출입 허가 종류(키, 출입도장) |
| **참조 모니터 (Reference Monitor)** | 보안 정책의 강제 실행체 | 시스템 콜 인터셉트, 행렬 조회, 큐바(Kube) 판정 | Kernel VFS, LSM(Linux Security Modules) | 출입구의 경비원(물리적 잠금장치) |

#### 2. 구현 전략: ACL vs Capability

행렬 전체를 저장하는 것은 메모리 낭비가 심하다. 따라서 행(주체) 중심으로 저장하거나 열(객체) 중심으로 저장하는 방식으로 나뉘며, 이는 각각 **Capability**와 **ACL**로 발전했다.

```text
 ┌─────────────────────────────────────────────────────────────────────────┐
 │           구현 방식 비교: ACL (열 중심) vs CL (행 중심)                  │
 ├─────────────────────────────────────────────────────────────────────────┤
 │                                                                         │
 │  [저장 관점의 변환 (Transforming the Matrix)]                           │
 │                                                                         │
 │  ┌─────────────────────────────────────────────────────────────────┐   │
 │  │  Matrix (이상적)          ACL (객체 중심)      CL (주체 중심)   │   │
 │  │ ┌──────────────────┐    ┌──────────────────┐  ┌──────────────────┐ │   │
 │  │ │F1  F2  F3  F4  │    │File 1: {A:rw, B:r}│  │User A: {F1, F2, W}│ │   │
 │  │ │A: rw  r   —   — │ => │File 2: {A:w}     │  │User B: {F1, R}    │ │   │
 │  │ │B:  r   w   —   — │    │File 3: {C:rwx}   │  │User C: {F3, RWX}  │ │   │
 │  │ │C:  —   —  rwx  — │    │...               │  │...               │ │   │
 │  │ └──────────────────┘    └──────────────────┘  └──────────────────┘ │   │
 │  │   (비효율적)              (각 파일이 소유)     (각 사용자가 소유)    │   │
 │  └─────────────────────────────────────────────────────────────────┘   │
 │                                                                         │
 │  [검증(Validation) 성능 및 특징]                                         │
 │                                                                         │
 │  ┌───────────────────────────────────────┐   ┌───────────────────────────┐│
 │  │  1. ACL (Access Control List)         │   │  2. CL (Capability List) ││
 │  │                                      │   │                           ││
 │  │  • 데이터 저장 위치: Object Header   │   │  • 데이터 저장 위치: User  ││
 │  │    (e.g., inode, File Metadata)      │   │    Space (Process Token)  ││
 │  │                                      │   │                           ││
 │  │  • 접근 검사: "이 파일에 A가 있나?"   │   │  • 접근 검사: "이 티켓이   ││
 │  │    -> 리스트 스캔(Scan) 필요        │   │    유효한가?" (Hash Check) ││
 │  │    -> O(N) 복잡도 (N=ACL 엔트리 수) │   │    -> O(1) 상수 시간      ││
 │  │                                      │   │                           ││
 │  │  • 강점: 관리 편의성 (Revoke easy)   │   │  • 강점: 접근 속도 (Fast)  ││
 │  │    파일 삭제 시 ACL 자동 소멸       │   │  • 약점: 회수(Revocation)   ││
 │  │                                      │   │    어려움 (분산 env에서    ││
 │  │                                      │   │    티켓 회수가 복잡함)    ││
 │  └───────────────────────────────────────┘   └───────────────────────────┘│
 │                                                                         │
 └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **ACL (Access Control List)의 구조**: 객체가 자신을 지키는 방식이다. 리눅스 EXT4 파일 시스템의 `inode` 구조 내부에는 사용자/그룹별 권한을 저장하는 공간이 있다. 이 방식의 가장 큰 장점은 **권한 회수(Revocation)**가 용이하다는 점이다. 예를 들어 직원이 퇴사하면, 그 사용자의 ID만 시스템에서 제거하면(혹은 그룹에서 배제하면) 모든 파일에 대한 접근이 차단된다. 단점은 특정 사용자가 '전체 시스템에서 내가 접근 가능한 모든 파일'을 찾으려면 모든 파일의 ACL을 순회해야 하므로 매우 비효율적이다.
2. **Capability List의 구조**: 주체가 자신의 권한을 증명하는 '티켓'을 가지는 방식이다. 예를 들어 **Kerberos**의 티켓이나 **Capsicum**의 Capability가 이에 해당한다. 접근 검사는 단순히 "나에게 이 파일을 열 유효한 티켓(Capability)이 있는가?"를 확인하는 것이므로 매우 빠르다(보통 해시 테이블 이용). 하지만 사용자가 퇴사할 때, 그 사용자가 이미 발급받아 가지고 있는 모든 티켓을 찾아 파기하는 것은 물리적으로 불가능에 가까우므로, 유효기간(Expiration) 짧은 티켓을 발행하거나 별도의 검증 서버를 두어 해결해야 한다.

#### 3. 도메인 전이 (Domain Transition)와 행렬 확장

보안 모델에서 주체(프로세스)는 자신의 권한을 변경할 수 있어야 한다(예: 일반 사용자 -> 관리자). 이를 행렬로 표현하면 **주체가 곧 객체가 되는** 재귀적 구조가 필요하다.

```text
 ┌─────────────────────────────────────────────────────────────────────────┐
 │                 도메인 전이를 포함한 확장 행렬 모델                      │
 ├─────────────────────────────────────────────────────────────────────────┤
 │                                                                         │
 │   [수식적 정의]                                                          │
 │   주체 집합 S = {User Process, Kernel Daemon}                           │
 │   객체 집합 O = {Files, S}  ★ 주체 S가 객체 O에 포함됨 ★               │
 │                                                                         │
 │   ┌──────────────────────────────────────────────────────────────────┐  │
 │   │         Object (Columns)                                         │  │
 │   │  ┌─────────────────────┬─────────────────────┬──────────────┐   │  │
 │   │  │      File A         │      File B         │   Kernel     │
