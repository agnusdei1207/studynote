## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MAC(Mandatory Access Control)는 **"시스템 보안 관리자가 정의한 보안 정책에 따라, 주체와 객체의 보안 레이블(Security Label)을 비교하여 강제적으로 접근을 제어"**하는 모델이다. 소유자의 의지와 무관하게 **"No Read Up, No Write Down"** 같은 보안 규칙이 시스템 레벨에서 강제된다.
> 2. **가치**: DAC의 **"트로이 목마 취약성"**을 근복적으로 해결한다. 웹 서버 프로세스가 탈취되더라도, MAC 정책에 의해 `/etc/shadow`나 시스템 설정 파일에 접근하는 것을 원천 봉쇄할 수 있다. 멀티테넌트 클라우드, 군사 시스템, 금융 시스템에서 필수적이다.
> 3. **융합**: SELinux, AppArmor, TrustedSolaris, grsecurity는 현대 OS의 MAC 구현체로서, **"Type Enforcement", "Role-Based MAC", "Multi-Level Security"** 등 다양한 기법을 지원한다.

## Ⅰ. 개요 (Context & Background)

- **개념**: MAC(Mandatory Access Control)는 **"보안 등급(Security Clearance)과 라벨(Security Label)"**을 기반으로 접근을 제어하는 보안 모델이다. **"강제적(Mandatory)"**이란 **"사용자(소유자)의 재량으로 변경할 수 없고, 시스템 보안 정책에 의해 강제됨"**을 의미한다. 수학적으로는 주체 S의 보안 등급 L(S)과 객체 O의 보안 등급 L(O)을 비교하여, **L(S) ≥ L(O)**이면 읽기 허용, **L(S) ≤ L(O)**이면 쓰기 허용하는 식으로 정의된다. Bell-LaPadula(기밀성)과 Biba(무결성) 모델이 MAC의 이론적 기반이다.

- **💡 비유**: MAC는 **"군사 기지의 보안 등급과 출입 통제"**와 같다. 기지에는 **"Top Secret(1등급), Secret(2등급), Confidential(3등급), Unclassified(4등급)"** 보안 등급이 있고, 각 문서와 병사도 등급으로 분류된다. **"1등급 보유자는 모든 등급의 문서를 읽을 수 있지만, 4등급 문서에 쓸 수는 없다(정보 하향 유출 방지)"**는 규칙이 시스템(보안 규정)에 의해 강제된다. 설령 장군(소유자)이 "이 1등급 문서를 훈련병에게 공개하겠다"고 명령하더라도, **보안 규정 위반이므로 시스템은 이를 거부**한다.

- **등장 배경**:
  1. **다중 levels 보안(Multi-Level Security, MLS) 요구(1960s-70s)**: 미 국방부는 **"기밀 문서가 일반 문서로 유출되는 것을 방지"**할 MLS 시스템을 필요로 했다. Bell-LaPadula 모델(1973)이 이 이론적 기반이 되었다.
  2. **Trusted UNIX(1980s-90s)**: DoD의 Trusted Computer System Evaluation Criteria(TCSEC, Orange Book) **B2/B3 등급** 인증을 받기 위해, **"DAC + MAC"** 하이브리드를 구현한 Trusted Solaris, Trusted UNIX가 개발되었다.
  3. **오픈 소스 MAC(2000s-현재)**: SELinux(2000년 NSA 주도), AppArmor(2002년 Immunix), grsecurity(2001년) 등 오픈 소스 MAC 시스템이 등장하여, 메인스트림 리눅스 배포판(RHEL, Ubuntu)에 기본 포함되었다.

DAC와 MAC의 대조를 시각화하면 다음과 같다. DAC는 소유자가 권한을 통제하지만, MAC는 시스템이 보안 레이블을 기반으로 강제한다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                  DAC vs MAC: 권한 제어 철학 비교                      │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [DAC — 사용자 재량]                                                 │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   alice(소유자)                                                │  │
 │  │     │                                                         │  │
 │  │     ├── chmod 777 file.txt  ← 자유롭게 권한 변경             │  │
 │  │     │                                                        │  │
 │  │     └── chown bob file.txt  ← 소유권 양도                   │  │
 │  │                                                                │  │
 │  │   철학: "내 파일이니 내 마음대로"                             │  │
 │  │   문제: alice가 속으면 공격자도 모든 권한 획득               │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [MAC — 시스템 강제]                                                 │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   alice(소유자)                                                │  │
 │  │     │                                                         │  │
 │  │     ├── chcon system_u:object_r:shadow_t:s0 file.txt          │  │
 │  │     │   ← 보안 레이블 변경(관리자만 가능)                    │  │
 │  │     │                                                        │  │
 │  │     └── chown bob file.txt  ← 소유권은 변경 가능하지만       │  │
 │  │                                bob도 접근 불가                │  │
 │  │                                                                │  │
 │  │   보안 정책(Security Policy):                                 │  │
 │  │   {httpd_t, shadow_t, file, read} → DENY                      │  │
 │  │   (웹 서버 프로세스는 /etc/shadow를 절대 읽을 수 없음)       │  │
 │  │                                                                │  │
 │  │   철학: "보안 레이블 간 흐름만 허용"                           │  │
 │  │   장점: 소유자가 속아도 보안 흐름은 유지                     │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [MLS(Multi-Level Security) — Bell-LaPadula 모델]                   │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   보안 등급: TS > S > C > U                                   │  │
 │  │   (Top Secret > Secret > Confidential > Unclassified)          │  │
 │  │                                                                │  │
 │  │   No Read Up: 하위 등급은 상위 등급을 읽을 수 없음           │  │
 │  │     • S 등급 사용자는 TS 문서를 읽을 수 없음                  │  │
 │  │     (정보 상향 유출 방지)                                      │  │
 │  │                                                                │  │
 │  │   No Write Down: 상위 등급은 하위 등급에 쓸 수 없음         │  │
 │  │     • TS 등급 사용자는 U 문서에 쓸 수 없음                   │  │
 │  │     (정보 하향 유출 방지)                                      │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** DAC에서는 소유자가 `chmod 777`를 실행하면 모든 사용자가 파일에 접근할 수 있게 된다. 그러나 MAC에서는 **"주체의 보안 컨텍스트(Domain/Type)"**와 **"객체의 보안 컨텍스트(Type)"**이 일치하는지 검사하므로, 소유자가 root라 하더라도 웹 서버 프로세스는 `/etc/shadow`에 접근할 수 없다. Bell-LaPadula 모델은 기밀성을 보장하기 위해 **"No Read Up(하위가 상위를 읽기 금지), No Write Down(상위가 하위에 쓰기 금지)"** 규칙을 강제한다.

### Bell-LaPadula 모델 — 기밀성 위주 MAC

Bell-LaPadula 모델(1973)은 **"다중 levels 보안(Multi-Level Security, MLS)"** 시스템의 기밀성을 보장하기 위해 설계되었다. **"No Read Up, No Write Down"** 두 가지 핵심 규칙이 있다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │               Bell-LaPadula 모델 — 기밀성 보장 MAC                    │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [보안 등급(Lattice)]                                                │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │        Top Secret (TS) — 가장 높음                           │  │
 │  │              │                                                │  │
 │  │              ▼                                                │  │
 │  │        Secret (S)                                             │  │
 │  │              │                                                │  │
 │  │              ▼                                                │  │
 │  │        Confidential (C)                                       │  │
 │  │              │                                                │  │
 │  │              ▼                                                │  │
 │  │        Unclassified (U) — 가장 낮음                          │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [핵심 규칙]                                                         │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ① No Read Up (Simple Security Property)                     │  │
 │  │     • 주체 S는 자신의 등급보다 높은 등급의 객체를 읽을 수 없음│  │
 │  │     • L(S) ≥ L(O)이어야 read 허용                             │  │
 │  │                                                                │  │
 │  │     예: Secret 등급 사용자는 Top Secret 문서를 읽을 수 없음   │  │
 │  │         (정보 상향 유출 방지)                                  │  │
 │  │                                                                │  │
 │  │  ② No Write Down (*-Property)                                 │  │
 │  │     • 주체 S는 자신의 등급보다 낮은 등급의 객체에 쓸 수 없음 │  │
 │  │     • L(S) ≤ L(O)이어야 write 허용                             │  │
 │  │                                                                │  │
 │  │     예: Top Secret 등급 사용자는 Unclassified 파일에 쓸 수 없음│  │
 │  │         (정보 하향 유출 방지)                                  │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [허용 가능한 흐름]                                                  │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   TS    ─────▶  TS    (같은 등급 간 읽기/쓰기 허용)          │  │
 │  │   │             │                                            │  │
 │  │   │             ▼                                            │  │
 │  │   │            TS    (상위 → 하위 읽기 허용)                  │  │
 │  │   │             │                                            │  │
 │  │   ▼             ▼                                            │  │
 │  │   S    ─────▶  S     (같은 등급 간)                          │  │
 │  │   │             │                                            │  │
 │  │   │             ▼                                            │  │
 │  │   │            S     (상위 → 하위 읽기 허용)                  │  │
 │  │   │             │                                            │  │
 │  │   ▼             ▼                                            │  │
 │  │   C    ─────▶  C                                             │  │
 │  │   │             │                                            │  │
 │  │   │             ▼                                            │  │
 │  │   │            C                                             │  │
 │  │   │             │                                            │  │
 │  │   ▼             ▼                                            │  │
 │  │   U    ─────▶  U                                             │  │
 │  │                                                                │  │
 │  │   ✅ Read: 상위 → 하위 흐름만 허용                            │  │
 │  │   ✅ Write: 같은 등급, 또는 하위 → 상위 흐름만 허용          │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Bell-LaPadula 모델의 **"No Read Up"**은 정보의 상향 유출(예: 일반 병사가 기밀 문서를 읽고 적에게 공개)을 방지한다. **"No Write Down"**은 정보의 하향 유출(예: 기밀 요원이 기밀 정보를 일반 문서에 숨겨서 적에게 공개)을 방지한다. 이 두 규칙을 조합하면, **"정보는 같은 등급 내에서만 흐르거나, 하위에서 상위로만 흐를 수 있다"**는 특성이 생겨, 기밀성이 보장된다. 그러나 이 모델은 무결성을 보장하지 못한다(예: 하위 등급 사용자가 상위 등급 문서를 수정할 수 있음).

### Biba 모델 — 무결성 위주 MAC

Biba 모델(1977)은 Bell-LaPadula와 반대로 **"무결성(Integrity)"**을 보장하기 위해 설계되었다. **"No Read Down, No Write Up"** 규칙으로, 신뢰할 수 없는 하위 등급이 상위 등급을 오염시키는 것을 방지한다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                 Biba 모델 — 무결성 보장 MAC                          │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [무결성 등급(Integrity Lattice)]                                    │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │        Trusted (T) — 가장 신뢰 가능                          │  │
 │  │              │                                                │  │
 │  │              ▼                                                │  │
 │  │        High (H)                                               │  │
 │  │              │                                                │  │
 │  │              ▼                                                │  │
 │  │        Medium (M)                                             │  │
 │  │              │                                                │  │
 │  │              ▼                                                │  │
 │  │        Low (L) — 가장 신뢰 불가                              │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [핵심 규칙]                                                         │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ① No Read Down (Simple Integrity Property)                  │  │
 │  │     • 주체 S는 자신의 등급보다 낮은 등급의 객체를 읽을 수 없음│  │
 │  │     • I(S) ≤ I(O)이어야 read 허용                             │  │
 │  │                                                                │  │
 │  │     예: High 등급 프로세스는 Low 등급 데이터를 읽을 수 없음  │  │
 │  │         (하위 등급의 오염 방지)                                │  │
 │  │                                                                │  │
 │  │  ② No Write Up (*-Property)                                   │  │
 │  │     • 주체 S는 자신의 등급보다 높은 등급의 객체에 쓸 수 없음 │  │
 │  │     • I(S) ≥ I(O)이어야 write 허용                            │  │
 │  │                                                                │  │
 │  │     예: Low 등급 사용자는 High 파일에 쓸 수 없음               │  │
 │  │         (상위 등급 오염 방지)                                  │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [허용 가능한 흐름]                                                  │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   L    ─────▶  L     (같은 등급 간 읽기/쓰기 허용)          │  │
 │  │   ▲             ▲                                            │  │
 │  │   │             │                                            │  │
 │  │   │            M     (하위 → 상위 읽기 허용)                  │  │
 │  │   │             ▲                                            │  │
 │  │   │             │                                            │  │
 │  │   │            T     (하위 → 상위 읽기 허용)                  │  │
 │  │                                                                │  │
 │  │   ✅ Read: 하위 → 상위 흐름만 허용 (신뢰 역전)               │  │
 │  │   ✅ Write: 같은 등급, 또는 상위 → 하위 흐름만 허용          │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  Bell-LaPadula(기밀성)과 Biba(무결성)를 결합하면                   │
 │  → "같은 등급 내에서만 읽기/쓰기 허용"이라는 강력한 격리가 됨     │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Biba 모델의 **"No Read Down"**은 신뢰할 수 없는 데이터(예: 인터넷 다운로드)가 신뢰하는 시스템을 오염시키는 것을 방지한다. 예를 들어 `/usr/bin`의 실행 파일들은 High 등급으로, 다운로드 파일은 Low 등급으로 레이블링하면, High 등급 프로세스가 Low 파일을 읽을 수 없게 된다. **"No Write Up"**은 Low 등급 사용자가 High 등급 시스템 파일을 수정하는 것을 방지한다(예: 일반 사용자가 `/bin/ls`를 백도어로 교체). Bell-LaPadula와 Biba를 동시에 적용하면 **"같은 등급 내에서만 읽기/쓰기 허용"**이라는 강력한 격리가 실현된다.

### SELinux — 리눅스의 MAC 구현체

SELinux(Security-Enhanced Linux)는 NSA가 주도하여 2000년에 발표한 리눅스 MAC 시스템으로서, **Type Enforcement(TE)**, **Role-Based Access Control(RBAC)**, **Multi-Level Security(MLS)**를 통합 지원한다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                   SELinux — 보안 컨텍스트 구조                        │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [보안 컨텍스트(Security Context) 형식]                             │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                 │  │
 │  │   user:role:type:level[:category]                              │  │
 │  │                                                                 │  │
 │  │   예: system_u:system_r:httpd_t:s0-s0:c0.c255                  │  │
 │  │       └──┬──┘ └──┬──┘ └──┬──┘ └────┬────┘                    │  │
 │  │          │      │       │         │                           │  │
 │  │       User   Role   Type   Level + Category                   │  │
 │  │                                                                 │  │
 │  │  • User: SELinux 사용자 (system_u, user_u, staff_u)          │  │
 │  │  • Role: 역할 (system_r, staff_r, object_r)                   │  │
 │  │  • Type: 타입 (httpd_t, shadow_t, httpd_sys_content_t)         │  │
 │  │  • Level: MLS 등급 (s0~s15)                                   │  │
 │  │  • Category: 카테고리 (c0.c1023)                              │  │
 │  │                                                                 │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [Type Enforcement — 도메인-타입 간 접근 제어]                        │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                 │  │
 │  │   주체(도메인)               객체(타입)                       │  │
 │  │   ┌──────────────┐         ┌──────────────┐                   │  │
 │  │   │  httpd_t     │         │  httpd_     │                   │  │
 │  │   │  (웹 서버    │         │  sys_       │                   │  │
 │  │   │   프로세스)   │         │  content_t  │                   │  │
 │  │   └──────────────┘         │  (웹 콘텐츠) │                   │  │
 │  │          │                └──────────────┘                   │  │
 │  │          │                         │                         │  │
 │  │          │                         ▼                         │  │
 │  │          │                ┌──────────────┐                   │  │
 │  │          │    정책:        │  shadow_t    │                   │  │
 │  │          │    {httpd_t,     │  (/etc/shadow)│                   │  │
 │  │          │     shadow_t,    └──────────────┘                   │  │
 │  │          │     file, read}  ← DENY!                          │  │
 │  │          │    → DENY        (httpd_t는                        │  │
 │  │          │                   shadow_t를                    │  │
 │  │          │                   읽을 수 없음)                  │  │
 │  │          │                                                 │  │
 │  │          └─────────────▶ ┌──────────────┐                   │  │
 │  │                          │  httpd_      │                   │  │
 │  │            정책:          │  sys_rw_     │                   │  │
 │  │            {httpd_t,       │  content_t   │                   │  │
 │  │             httpd_         │  (쓰기 허용)  │                   │  │
 │  │             sys_rw_t,      └──────────────┘                   │  │
 │  │             file, write}   ← ALLOW!                          │  │
 │  │            → ALLOW                                            │  │
 │  │                                                                 │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [AVC(Access Vector Cache) — 빠른 정책 검색]                         │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                 │  │
 │  │   Key: {httpd_t, shadow_t, file, read}                         │  │
 │  │   Value: DENY (캐시되어 있어 정책 DB 재검색 불필요)             │  │
 │  │                                                                 │  │
 │  │   접근 요청 → AVC 캐시 검색 → Hit: 바로 결과 반환               │  │
 │  │               → Miss: 정책 DB 검색 → AVC에 저장 → 반환         │  │
 │  │                                                                 │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SELinux는 **"도메인(Domain)"**과 **"타입(Type)"**을 분리하여, 프로세스(도메인)와 파일(타입) 사이의 허용 관계를 정책 데이터베이스에 정의한다. `httpd_t` 도메인의 프로세스가 `shadow_t` 타입의 파일에 접근하려면, 정책에 **{httpd_t, shadow_t, file, read}** 허용 규칙이 있어야 한다. 없으면 AVC(Access Vector Cache)가 **DENY**를 반환하고, `audit.log`에 거부 기록을 남긴다. AVC는 정책 검색 결과를 캐싱하므로, 두 번째 요청부터는 정책 DB를 재검색하지 않아 빠르다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: SELinux vs AppArmor

| 비교 항목 | SELinux | AppArmor |
|:---|:---|:---|
| **기반 모델** | 레이블 기반(Label-based) | 경로 기반(Path-based) |
| **정책 언어** | M4 macro, 복잡한 TE 언어 | 간단한 allow/deny 규칙 |
| **학습 곡선** | 가파름 | 완만함 |
| **성능** | AVC 캐시로 빠름 | 경로 검색으로 약간 느림 |
| **대상 파일** | 모든 객체(파일, 소켓, IPC) | 주로 파일 시스템 |
| **주 용도** | 정부/군사, 대규모 서버 | 데스크톱, 소규모 서버 |
| **설정 난이도** | 복잡(`semanage`, `checkpolicy`) | 간단(`aa-easyprof`) |

SELinux는 **"모든 객체에 레이블을 부착"**해야 하므로, 초기 설정이 복잡하지만 일관된 보안을 제공한다. AppArmor는 **"경로 기반"**이라서, `/etc/shadow` 파일에 대한 접근 제어를 `/etc/shadow { }` 블록으로 간단히 정의할 수 있다. 그러나 경로가 변경되거나 심볼릭 링크를 사용하면 우회 가능하다.

### 비교 2: MAC 모델별 특성

| 모델 | 목적 | 핵심 규칙 | 주 용도 |
|:---|:---|:---|
| **Bell-LaPadula** | 기밀성(Confidentiality) | No Read Up, No Write Down | 군사 MLS 시스템 |
| **Biba** | 무결성(Integrity) | No Read Down, No Write Up | 금융, 의료 시스템 |
| **Clark-Wilson** | 무결성(거래 중심) | Well-formed Transactions, Separation of Duty | 전자 상거래 |
| **Chinese Wall** | 이해 상충 방지 | No Conflict of Interest | 금융, 법률 firms |
| **Type Enforcement** | 도메인 격리 | Domain-Type 간 접근 제어 | SELinux, AppArmor |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — SELinux 활성화 및 웹 서버 보안**: Apache httpd를 실행하는데, DAC만으로는 부족하여 SELinux를 활성화했다.
   ① `setenforce 1`로 enforcing 모드 활성화
   ② `semanage permissive -a httpd_t`로 초기에는 permissive 모드(위반 시 로그만 남기고 차단하지 않음)
   ③ 웹 애플리케이션을 테스트하여 AVC 거부 로그(`ausearch -m avc -ts recent`) 확인
   ④ 필요한 허가 규칙을 추가(`allow httpd_t httpd_sys_content_t:file { read open }`)
   ⑤ `semanage permissive -d httpd_t`로 permissive 모드 해제 후 enforcing 전환

2. **시나리오 — AppArmor로 컨테이너 격리**: Docker 컨테이너에서 실행되는 앱을 AppArmor로 격리했다.
   ① `aa-easyprof docker-default`로 기본 프로필 생성
   ② `/etc/apparmor.d/docker-default`에 파일 접근 규칙 추가:
       ```
       /var/www/** rw,
       deny /etc/shadow rw,
       deny /sys/** rw,
       ```
   ③ `aa-enforce /etc/apparmor.d/docker-default`로 강제 모드 활성화
   → 컨테이너가 탈취되더라도 호스트의 `/etc/shadow`나 `/sys`에 접근 불가

3. **시나리오 — MLS(Multi-Level Security) 구성**: 기관에서 **"일반 직원과 비밀 직원의 파일 격리"**가 필요했다.
   ① SELinux MLS를 활성화(`setenforce 1`, MLS policy 로드)
   ② 일반 직원: `s0`(Unclassified), 비밀 직원: `s1`(Confidential)
   ③ `/confidential/` 디렉터리를 `s1`로 레이블링: `chcon -l s1 /confidential/`
   ④ 일반 직원 프로세스(`s0`)는 `/confidential/` 파일을 읽을 수 없음(No Read Up)

### 도입 체크리스트
- **기술적**: SELinux/AppArmor가 enforcing 모드인가(`sestatus`, `aa-status`)? AVC 거부 로그를 정기적으로 점검하는가?
- **운영·보안적**: 새로운 애플리케이션 배포 시 MAC 정책을 먼저 테스트(permissive 모드)하는가? 불필요한 정책은 제거하는가?

### 안티패턴
- **SELinux 비활성화**: 보안상 문제가 발생하자 `setenforce 0`으로 비활성화하는 것은 임시 방편이다. 대신 `ausearch`로 거부 로그를 분석하고 `audit2allow`로 필요한 허가 규칙을 추가해야 한다.
- **과도한 permissive 모드**: permissive 모드는 MAC 차단을 하지 않으므로, 실제 공격에 대한 방어력이 없다. 초기 테스트 후에는 반드시 enforcing 모드로 전환해야 한다.

### 미래 전망
- **eBPF LSM**: 커널 모듈 없이 eBPF 프로그램으로 MAC 정책를 동적으로 로드할 수 있어, **"사용자 공간 MAC 정책 엔진"**이 가능해진다.
- **Confidential Computing**: AMD SEV-SNP, Intel TDX와 MAC를 결합하여, **"메모리 암호화 + MAC 격리"**로 가상머신 간 완전한 격리를 제공한다.

## 👶 어린이를 위한 3줄 비유 설명
1. MAC는 **"군사 기지의 출입 통제 시스템"**이에요. "1등급 비밀 구역에는 1등급 자격증이 있는 사람만 들어갈 수 있어요."
2. 설령 장군이 "훈련병에게 1등급 구역을 보여주겠다"고 명령해도, 보안 규정에 따라 시스템이 거부해요. "규정이니까 못 들어가죠!"라고 차단해요.
3. 그래서 해커가 웹 서버를 해킹해도, MAC가 "웹 서버는 사용자 비밀번호 파일에 접근 금지!"라고 강제해서, 비밀번호가 유출되는 것을 막을 수 있어요.
