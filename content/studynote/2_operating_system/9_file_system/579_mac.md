# 579. 강제적 접근 제어 (MAC, Mandatory Access Control)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MAC (Mandatory Access Control, 강제적 접근 제어)는 시스템의 최고 보안 관리자가 중앙 집중식으로 정의한 **보안 정책(Security Policy)**에 따라, **주체(Subject)**와 **객체(Object)**의 보안 라벨(Security Label)을 비교하여 접근을 강제적으로 통제하는 모델이다. 사용자(소유자)의 재량에 의한 권한 변경이 원천적으로 불가능하며, **"No Read Up, No Write Down"**과 같은 정보 흐름 보안 규칙을 시스템 커널 레벨에서 시행한다.
> 2. **가치**: DAC (Discretionary Access Control, 임의적 접근 제어)의 근본적 취약점인 **"트로이 목마(Trojan Horse)"** 공격을 방어한다. 예를 들어, 웹 서버 프로세스가 해킹당하더라도, MAC 정책에 의해 `/etc/shadow`나 시스템 설정 파일에 접근하는 것이 강제로 차단되므로 피해 확산을 방지할 수 있다. 이는 멀티테넌트 클라우드, 금융권, 국방 시스템 등에서 요구하는 필수적인 보안 계층이다.
> 3. **융합**: SELinux (Security-Enhanced Linux), AppArmor, TrustedBSD, Windows Mandatory Integrity Control 등 현대 OS의 핵심 보안 모듈로 구현된다. **Type Enforcement (TE)**, **Role-Based Access Control (RBAC)**, **Multi-Level Security (MLS)** 기술과 융합하여 **제로 트러스트(Zero Trust)** 아키텍처의 기반을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념 정의**: MAC는 **"보안 등급(Security Clearance)"**과 **"분류 등급(Classification)"**을 기반으로 접근을 제어하는 보안 모델이다. **"강제적(Mandatory)"**이라는 용어는 **"자원의 소유자가 아닌, 시스템의 보안 정책(Security Policy)에 의해 접근 권한이 결정되며 변경할 수 없음"**을 의미한다. 수학적으로는 주체 $S$의 보안 등급 $L(S)$과 객체 $O$의 보안 등급 $L(O)$을 비교하여, 기밀성 모델인 Bell-LaPadula에서는 $L(S) \ge L(O)$일 때 읽기(Read)를 허용하고, $L(S) \le L(O)$일 때 쓰기(Write)를 허용하는 식으로 정보의 흐름을 제어한다.
  
- **💡 비유**: MAC는 **"고도로 통제된 군사 기지의 출입 통제 시스템"**과 같다. 모든 인원(주체)과 문서(객체)에는 **"Top Secret(일등급), Secret(이등급), Confidential(삼등급), Unclassified(사등급)"**과 같은 보안 라벨이 붙어 있다. 설령 장군(시스템 관리자가 아닌 소유자)이 "이 훈련병에게 일등급 문서를 보여주라"고 명령하더라도, **시스템은 보안 규정(MAC Policy)에 위배됨을 확인하고 자동으로 출입을 거부**한다. 즉, 소유자의 의지보다 시스템의 규칙이 우선한다.

- **등장 배경 및 역사**:
  1. **다중 levels 보안(Multi-Level Security, MLS)의 요구 (1960s-70s)**: 미 국방부는 서로 다른 기밀 등급의 정보를 한 컴퓨터에서 처리하면서, 고등급 정보가 저등급 구역으로 유출되는 것을 막기 위해 **TCSEC (Trusted Computer System Evaluation Criteria, 신뢰 컴퓨터 시스템 평가 기준, Orange Book)**을 제정했다. 이에 따라 **Bell-LaPadula 모델**이 제안되었다.
  2. **상용 UNIX 및 보안 커널의 발전 (1980s-90s)**: DAC만으로는 충분하지 않음이 밝혀지며, Trusted Solaris, Trusted UNIX와 같이 커널 레벨에서 MAC를 구현한 OS가 등장했다.
  3. **오픈 소스 리눅스로의 확산 (2000s-현재)**: 미 NSA (National Security Agency)가 주도하여 개발한 **SELinux**가 리눅스 커널에 통합되었고, 이후 AppArmor, Smack 등 다양한 MAC 구현체가 등장하여 일반 서버 환경에서도 표준처럼 사용되게 되었다.

DAC(임의적)와 MAC(강제적)의 결정적인 차이는 **"누가 접근 통제 권한을 가지는가?"**에 있다. 아래 다이어그램은 이 두 모델의 통제 철학을 시각적으로 비교한 것이다.

```text
 ┌──────────────────────────────────────────────────────────────────────────┐
 │                 DAC vs MAC: 접근 제어 권한의 비교                         │
 ├──────────────────────────────────────────────────────────────────────────┤
 │                                                                          │
 │  [DAC (Discretionary Access Control)]                                    │
 │   ┌─────────────────────────────────────────────────────────────────┐    │
 │  │                                                                  │   │
 │  │   👤 alice (File Owner)                                          │   │
 │  │     │                                                            │   │
 │  │     ├── "내 파일이니 마음대로 조작해!"                            │   │
 │  │     │                                                            │   │
 │  │     ├── chmod 777 secret.txt   ← 소유자의 재량으로 권한 부여     │   │
 │  │     │   (모두에게 읽기/쓰기 허용)                                │   │
 │  │     │                                                            │   │
 │  │     └── 📂 secret.txt                                            │   │
 │  │         👻 hacker (alice의 권한을 획득 시 파일 접근 가능)          │   │
 │  │                                                                  │   │
 │  │   ⚠️ 취약점: 소유자의 실수(예: 777 권한 부여)나                 │   │
 │  │            탈취된 계정으로 인한 보안 사고 발생 가능               │   │
 │  │                                                                  │   │
 │  └─────────────────────────────────────────────────────────────────┘    │
 │                                                                          │
 │  [MAC (Mandatory Access Control)]                                        │
 │   ┌─────────────────────────────────────────────────────────────────┐    │
 │  │                                                                  │   │
 │  │   👤 alice (File Owner)                                          │   │
 │  │     │                                                            │   │
 │  │     ├── "이 파일을 공유하고 싶어!"                                │   │
 │  │     │                                                            │   │
 │  │     ├── chcon unconfined_u:object_r:secret_data_t:s0 secret.txt  │   │
 │  │     │   ← 명령은 가능하지만, 실제 접근은 보안 정책이 결정         │   │
 │  │     │                                                            │   │
 │  │     └── 🛡️ Security Policy Kernel (Reference Monitor)            │   │
 │  │           │                                                       │   │
 │  │           ├── "alice의 도메인(alice_t)과 파일의 타입(secret_t)을   │   │
 │  │           │    비교한 결과, 기본 규칙上 다른 주체의 접근을 차단"    │   │
 │  │           │                                                       │   │
 │  │     └── 📂 secret.txt (Label: secret_t)                          │   │
 │  │         👻 hacker (해커의 프로세스 도메인이 secret_t 접근 불가)    │   │
 │  │                                                                  │   │
 │  │   ✅ 보안: 소유자가 권한을 열어주려 해도, 시스템 정책에 의해       │   │
 │  │            허용되지 않으면 무조건 차단됨 (강제성)                  │   │
 │  │                                                                  │   │
 │  └─────────────────────────────────────────────────────────────────┘    │
 │                                                                          │
 └──────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
위 그림에서 DAC는 사용자가 자신의 파일(`secret.txt`)에 대해 `chmod` 명령어를 통해 자유롭게 권한을 변경할 수 있음을 보여준다. 이는 유연하지만, 사용자 계정이 탈취되거나 악성코드에 감염되면 공격자가 해당 권한을 그대로 사용하여 데이터를 유출할 수 있는 **트로이 목마 취약점**을 내재하고 있다. 반면, MAC는 사용자가 `chcon`과 같은 명령으로 라벨을 변경하려 시도하더라도, 실제 파일 시스템 접근은 **커널 내부의 보안 정책(Security Policy)**에 의해 엄격하게 제한된다. 즉, 사용자의 의지(Intent)와 무관하게 시스템이 정의한 보안 룰셋(Rule Set)이 우선하므로, 웹 서버 프로세스가 해킹당하더라도 시스템의 핵심 파일(`/etc/passwd` 등)로의 **불필요한 정보 흐름**을 시스템 차원에서 물리적으로 차단할 수 있다.

> **📢 섹션 요약 비유**: DAC는 "내 집 자물쇠는 내가 열어주고 싶은 사람에게 열쇠를 주는 식"이라면, MAC는 "금고가 설치된 방의 출입문이 본인 확인 여부와 상관없이 보안 등급에 따라 자동으로 개폐되는 고속도로 톨게이트"와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. MAC 아키텍처의 핵심 구성 요소

| 요소명 | 역할 | 내부 동작 메커니즘 | 관련 기술/프로토콜 | 비유 |
|:---|:---|:---|:---|:---|
| **보안 라벨 (Security Label)** | 주체와 객체의 보안 등급 식별 | 파일 시스템 확장 속성(xattr)에 저장 | SELinux Context, SMACK Label | 명찰, 보안 등급 마크 |
| **참조 모니터 (Reference Monitor)** | 모든 접근 요청을 중계하고 검사 | 커널의 시스템 콜 후킹(Hooking)을 통해 검증 | LSM (Linux Security Modules) | 공항 보안 검색대 |
| **보안 정책 데이터베이스 (Security Policy DB)** | 허용/거부 규칙을 저장 및 관리 | TE(Type Enforcement) 규칙, MLS 규칙 저장 | SELinux Policy Binary, AppArmor Profile | 보안 규정집 |
| **결정 모듈 (Decision Module)** | 라벨과 정책을 비교하여 최종 판정 | `allow domain_t type_t:class permission` 규칙 로딩 | AVC (Access Vector Cache) | 판사, 심판관 |
| **감사 및 로그 (Audit Logging)** | 거부된 접근 시도를 기록 | `/var/log/audit/audit.log`에 AVC Deny 로그 저장 | auditd, rsyslog | CCTV, 블랙박스 |

### 2. 정보 흐름 보안 모델: Bell-LaPadula vs Biba

MAC의 동작 원리를 이해하기 위해서는 **정보의 흐름을 어떻게 제어하는지**를 이해해야 한다. 대표적인 두 가지 모델인 **Bell-LaPadula (기밀성 중심)**와 **Biba (무결성 중심)**를 비교 분석한다.

```text
 ┌──────────────────────────────────────────────────────────────────────────┐
 │        [Bell-LaPadula 모델: 기밀성(Confidentiality) 보장]                │
 │                                                                          │
 │  🛡️ 목적: 기밀 정보가 하위 등급으로 유출되는 것을 방지                    │
 │                                                                          │
 │     [보안 등급 계층 구조]                                                 │
 │        Top Secret (TS)   ────  Level 4 (최고 기밀)                       │
 │              │                                                          │
 │              ▼                                                          │
 │        Secret (S)        ────  Level 3 (비밀)                            │
 │              │                                                          │
 │              ▼                                                          │
 │        Confidential (C)  ────  Level 2 (대외비)                          │
 │              │                                                          │
 │              ▼                                                          │
 │        Unclassified (U)  ────  Level 1 (일반 공개)                       │
 │                                                                          │
 │  ┌────────────────────────────────────────────────────────────────────┐ │
 │  │ ⚠️ 핵심 규칙 (Security Properties)                                │ │
 │  │                                                                    │ │
 │  │  1️⃣ No Read Up (Simple Security Property)                        │ │
 │  │     • 하위 등급 주체는 상위 등급 객체를 읽을 수 없음                │ │
 │  │     • 예: 등급 S(Secret) 사용자는 등급 TS(Top Secret) 문서를       │ │
 │  │            읽을 수 없음. (정보 상향 유출 방지)                      │ │
 │  │                                                                    │ │
 │  │  2️⃣ No Write Down (* Property - Star Property)                    │ │
 │  │     • 상위 등급 주체는 하위 등급 객체에 쓸 수 없음                  │ │
 │  │     • 예: 등급 TS 사용자는 등급 U(Unclassified) 파일에            │ │
 │  │            내용을 쓸 수 없음. (정보 하향 유출 방지)                 │ │
 │  │                                                                    │ │
 │  └────────────────────────────────────────────────────────────────────┘ │
 │                                                                          │
 ├──────────────────────────────────────────────────────────────────────────┤
 │        [Biba 모델: 무결성(Integrity) 보장]                              │
 │                                                                          │
 │  🛡️ 목적: 신뢰할 수 없는 정보가 상위 시스템을 오염시키는 것을 방지        │
 │                                                                          │
 │     [신뢰 등급 계층 구조]                                                 │
 │        Trusted (T)        ────  Level 4 (매우 신뢰함)                    │
 │              ▲                                                          │
 │              │                                                          │
 │        High (H)           ────  Level 3 (신뢰함)                         │
 │              ▲                                                          │
 │              │                                                          │
 │        Medium (M)         ────  Level 2 (보통)                           │
 │              ▲                                                          │
 │              │                                                          │
 │        Low (L)            ────  Level 1 (신뢰 안 함)                     │
 │                                                                          │
 │  ┌────────────────────────────────────────────────────────────────────┐ │
 │  │ ⚠️ 핵심 규칙 (Integrity Properties)                               │ │
 │  │                                                                    │ │
 │  │  1️⃣ No Read Down (Simple Integrity Property)                     │ │
 │  │     • 상위 등급 주체는 하위 등급 객체를 읽을 수 없음                │ │
 │  │     • 예: Trusted 프로세스는 인터넷(Low) 데이터를 바로 읽지 않음.   │ │
 │  │            (하위 등급 데이터로 인한 오염 방지)                      │ │
 │  │                                                                    │ │
 │  │  2️⃣ No Write Up (* Property - Star Property)                      │ │
 │  │     • 하위 등급 주체는 상위 등급 객체에 쓸 수 없음                   │ │
 │  │     • 예: Low 사용자
