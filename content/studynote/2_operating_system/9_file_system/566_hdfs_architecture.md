## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하둡 분산 파일 시스템(HDFS, Hadoop Distributed File System)은 범용 하드웨어를 사용하여 대규모 데이터를 저장하고 처리하기 위해 설계된 분산형 블록 기반 파일 시스템이다.
> 2. **가치**: 데이터 처리량(Throughput) 극대화에 초점을 맞추어 수 페타바이트(PB) 이상의 초거대 파일을 지원하며, 하드웨어 장애를 예외가 아닌 일상적인 현상으로 간주하여 높은 결함 허용(Fault Tolerance)을 제공한다.
> 3. **철학**: "데이터가 연산 장치로 이동하는 대신, 연산이 데이터가 있는 곳으로 이동한다(Moving Computation is Cheaper than Moving Data)"는 데이터 지역성(Data Locality) 원칙을 실현한다.

---

## Ⅰ. HDFS의 정의와 설계 목표 (Design Goals)

- **정의**: Apache Hadoop 프로젝트의 핵심 구성 요소로, 저가형 서버 수천 대를 묶어 거대한 가상 저장소를 만드는 시스템이다.
- **핵심 목표**:
    - **장애 대응**: 자동 복구 기능을 통해 상용 하드웨어의 빈번한 고장 문제를 해결한다.
    - **초대형 데이터셋**: 수백 기가바이트(GB)에서 테라바이트(TB) 크기의 파일을 관리한다.
    - **단순한 일관성 모델**: '한 번 쓰고 여러 번 읽기(Write-once-read-many)' 방식을 채택하여 동시성 제어 복잡성을 줄인다.

📢 **섹션 요약 비유**: HDFS는 "거대한 화물선"과 같다. 아주 무거운 짐(빅데이터)을 한 번에 많이 싣고 나르는 데 최적화되어 있으며, 배의 엔진이 한두 개 고장 나도 목적지까지 안전하게 운항할 수 있도록 설계되었다.

---

## Ⅱ. HDFS 아키텍처 구조 (Architecture & Data Flow)

HDFS는 메타데이터를 관리하는 마스터(NameNode)와 실제 데이터를 저장하는 슬레이브(DataNode)로 구성된다.

### 1. 시스템 계층도 (ASCII Diagram)

```text
      [ HDFS Client ]
      ┌───────┐
      │ Read  │◀─────┐ (3. Actual Data Flow)
      │ Write │      │
      └───┬───┘      │          [ NameNode (Master) ]
          │          │          ┌───────────────────┐
 (1. Get Meta)       └─────────▶│ Metadata (Namespace,│
          │                     │ Block Locations)  │
          ▼                     └─────────┬─────────┘
 [ HDFS Namespace ]                       │ (2. Data Mapping)
          │                               ▼
          ▼           [ DataNodes (Slaves / Workers) ]
 ┌────────────────┐   ┌────────────────┐   ┌────────────────┐
 │   DataNode A   │   │   DataNode B   │   │   DataNode C   │
 │ [B1][B2][B3]   │◀─▶│ [B1][B4][B5]   │◀─▶│ [B2][B3][B5]   │
 └────────────────┘   └────────────────┘   └────────────────┘
                (4. Block Replication & Heartbeat)
```

📢 **섹션 요약 비유**: HDFS 아키텍처는 "대형 물류 센터"와 같다. 센터장(NameNode)은 어느 창고에 어떤 물건이 있는지 장부만 관리하고, 실제 물건(Data)은 수많은 창고 직원(DataNode)들이 나누어 보관한다.

---

## Ⅲ. 마스터-슬레이브 모델 (Master-Slave Model)

- **네임노드 (NameNode)**:
    - **역할**: 파일 시스템의 트리 구조와 파일 내 블록들의 위치 정보를 메모리(RAM)에서 관리한다.
    - **부하**: 메타데이터가 많아질수록 네임노드의 메모리 부담이 커지므로 확장성의 병목이 될 수 있다.
- **데이터노드 (DataNode)**:
    - **역할**: 클라이언트의 읽기/쓰기 요청을 직접 처리하며, 블록을 생성, 삭제, 복제한다.
    - **보고**: 주기적으로 네임노드에 자신이 가진 블록 목록을 보고(Block Report)한다.

📢 **섹션 요약 비유**: 네임노드는 "도서관의 검색 컴퓨터"이고, 데이터노드는 "수천 개의 책꽂이"다. 검색 컴퓨터가 고장 나면 책이 어디 있는지 찾을 수 없으므로 네임노드는 매우 소중하게 관리되어야 한다.

---

## Ⅳ. 블록 기반 저장 방식 (Block-based Storage)

- **블록 크기**: 일반 OS(4KB~64KB)와 달리 HDFS는 기본 64MB 또는 128MB의 매우 큰 블록 단위를 사용한다.
- **장점**:
    - **탐색 비용 절감**: 큰 단위로 읽으므로 디스크 헤더의 이동 시간을 최소화한다.
    - **메타데이터 효율성**: 블록 수가 줄어들어 네임노드가 관리해야 할 정보량이 줄어든다.
- **분산 저장**: 하나의 파일은 여러 개의 블록으로 쪼개져 서로 다른 데이터노드에 흩어져 저장된다.

📢 **섹션 요약 비유**: 블록 기반 저장은 "피자를 조각내어 여러 접시에 담는 것"과 같다. 피자(파일)가 너무 커서 한 접시에 담을 수 없으므로, 조각(블록)을 내어 친구들(데이터노드)에게 나누어 주는 방식이다.

---

## Ⅴ. 스트리밍 데이터 접근 (Streaming Data Access)

- **특징**: HDFS는 임의 접근(Random Access)보다는 전체 데이터를 순차적으로 읽는 스트리밍 방식에 최적화되어 있다.
- **Write-once-read-many**: 파일이 생성되어 닫히면 수정할 수 없으며(Append만 가능), 이는 배치 작업(MapReduce 등)의 효율성을 극대화한다.
- **고가용성 메커니즘**: 모든 블록은 여러 노드에 복제(기본 3배)되어 일부 노드 장애 시에도 데이터 유실 없이 서비스를 지속한다.

📢 **섹션 요약 비유**: 스트리밍 접근은 "영화 스트리밍"과 같다. 중간의 특정 장면을 1초씩 끊어서 보기보다, 처음부터 끝까지 끊김 없이 데이터를 쭉 빨아들이는 방식이다.

## 👶 아이를 위한 비유 (Child Analogy)
> "HDFS는 아주 커다란 **'퍼즐 보관소'**야. 퍼즐 조각(블록)이 너무 많아서 한 박스에 다 못 넣으니까, 여러 친구네 집(데이터노드)에 나누어서 보관하는 거지. 대신 대장 친구(네임노드)는 어느 조각이 누구네 집에 있는지 다 적어두고 있어. 조각 하나를 잃어버려도 미리 복사해서 다른 친구네 집에도 맡겨뒀기 때문에 걱정할 필요가 없단다!"
