## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NFS (Network File System)는 네트워크상의 다른 컴퓨터에 저장된 파일을 마치 로컬 디스크에 있는 것처럼 투명하게 접근할 수 있게 해주는 클라이언트-서버 기반 프로토콜이다.
> 2. **가치**: 데이터의 중앙 집중 관리를 가능하게 하며, 원격 프로시저 호출 (RPC, Remote Procedure Call)을 통해 복잡한 네트워크 통신을 은폐하고 추상화한다.
> 3. **특징**: 상태 비저장 (Stateless) 프로토콜 설계로 서버 장애 복구가 쉽지만, 클라이언트 캐싱으로 인한 일관성 (Consistency) 관리 문제가 핵심 과제다.

---

## Ⅰ. 개요 (Context & Background)

- **정의**: 썬 마이크로시스템즈에서 개발한 분산 파일 시스템 (DFS, Distributed File System) 표준 프로토콜로, 네트워크를 통해 이기종 운영체제 간 파일을 공유하는 데 널리 쓰인다.
- **배경**: 워크스테이션 환경에서 모든 사용자가 각자의 하드디스크를 갖는 대신, 중앙 서버의 고성능 스토리지 자원을 공유하여 비용을 절감하고 관리 효율을 높이기 위해 탄생했다.
- **핵심 기술**:
  - **RPC (Remote Procedure Call)**: 원격지의 함수를 로컬 함수처럼 호출하는 기술.
  - **XDR (External Data Representation)**: 서로 다른 기종 간에 데이터 형식을 통일하는 기술.

📢 **섹션 요약 비유**: NFS는 "공용 사물함"과 같다. 내 책상에는 사물함이 없지만, 복도(네트워크)에 있는 커다란 사물함(서버)을 내 책상 서랍인 것처럼 쓰는 방식이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. NFS 동작 구조 시각화 (ASCII Diagram)

```text
 [ NFS Client ]                     [ NFS Server ]
 ┌────────────────┐                 ┌────────────────┐
 │ Application    │                 │ File System    │
 ├────────────────┤                 ├────────────────┤
 │ VFS Layer      │                 │ VFS Layer      │
 ├────────────────┤                 ├────────────────┤
 │ NFS Client Stub│ <─── RPC/XDR ───▶ NFS Server Stub│
 └────────────────┘   (Network)     └────────────────┘
        │                                  │
 [ Local Mount Point ]              [ Exported Disk ]
```

### 2. 주요 통신 과정
1. **마운트 요청**: 클라이언트가 서버의 특정 경로를 자신의 마운트 지점에 연결한다.
2. **파일 핸들 (File Handle)**: 서버는 특정 파일을 식별할 수 있는 고유한 '파일 핸들'을 클라이언트에 전달한다.
3. **요청 전달**: 클라이언트에서 `read()`가 발생하면 VFS가 이를 NFS 클라이언트로 전달하고, 이는 다시 RPC를 통해 서버의 NFS 데몬(nfsd)으로 전달된다.
4. **응답**: 서버가 디스크에서 데이터를 읽어 RPC 응답으로 보내면, 클라이언트는 이를 로컬 메모리에 캐싱하고 애플리케이션에 전달한다.

📢 **섹션 요약 비유**: NFS는 "전화 주문 배달"과 같다. 내가 전화(RPC)로 메뉴(파일명)를 말하면, 주방(서버)에서 요리를 해서 집(클라이언트)까지 배달해주는 과정과 같다.

## Ⅲ. 상태 비저장 설계와 일관성 (Stateless & Consistency)

- **상태 비저장 (Stateless)**: 서버는 클라이언트가 어떤 파일을 열어 놓았는지(Open state) 기억하지 않는다. 모든 요청은 그 자체로 독립적이며, 서버가 재부팅되어도 클라이언트는 이전 상태와 무관하게 요청을 재시도할 수 있다.
- **캐시 일관성 (Cache Consistency)**:
  - 성능을 위해 클라이언트는 서버의 데이터를 로컬 메모리에 캐싱한다.
  - **Close-to-Open 일관성**: 파일을 닫을 때만 변경 사항을 서버에 쓰고, 파일을 열 때만 서버의 최신 데이터를 확인하는 느슨한 일관성 모델을 사용한다.

📢 **섹션 요약 비유**: 상태 비저장은 "자판기"와 같다. 자판기는 앞 손님이 누구였는지 기억하지 않아도, 돈과 번호(파일 핸들)만 맞으면 물건을 내어주는 것과 같다.

## Ⅳ. 융합 비교 및 분석 (Comparison)

| 비교 항목 | 로컬 파일 시스템 | NFS (Network File System) | 클라우드 스토리지 (S3 등) |
|:---|:---|:---|:---|
| **물리적 위치** | 로컬 버스 (SATA/NVMe) | 로컬 네트워크 (LAN) | 인터넷 (WAN) |
| **인터페이스** | 블록 기반 / POSIX | RPC 기반 / POSIX | HTTP API 기반 (REST) |
| **장애 복구** | 파일 시스템 체크 필수 | 재시도 (Idempotent)로 자동 복구 | 고도화된 분산 복제 |
| **성능** | 매우 높음 | 네트워크 속도에 의존 | 레이턴시 높음 (Batch 위주) |

📢 **섹션 요약 비유**: 로컬 시스템이 "내 집 냉장고"라면, NFS는 "동네 마트", 클라우드 스토리지는 "해외 직구 매장"과 같다.

## Ⅴ. 기술적 구현 및 성능 튜닝 (Implementation)

- **수출 (Export)**: 서버 측에서는 `/etc/exports` 파일을 통해 어떤 디렉터리를 어떤 클라이언트에게 공유할지 설정한다.
- **성능 튜닝**:
  - `rsize`, `wsize`: 한 번에 주고받는 데이터 덩어리의 크기를 조절한다.
  - `async`: 서버가 디스크에 쓰기 전에 미리 응답을 보내 속도를 높이지만, 정전 시 데이터 손실 위험이 있다.
- **보안**: 초기 NFS는 보안에 취약했으나, 최신 NFS v4는 Kerberos 인증과 강한 접근 제어를 지원한다.

📢 **섹션 요약 비유**: 성능 튜닝은 "배달용 오토바이 상자 크기를 키우는 것"과 같다. 한 번에 많이 나르면(size 증가) 효율은 좋지만, 사고가 나면 한꺼번에 잃을 수도 있는 것과 같다.
