## 핵심 인사이트 (3줄 요약)
> 1. **이원적 구조**: HDFS는 메타데이터를 통합 관리하는 관리자(NameNode)와 실제 데이터 블록을 물리적으로 저장하는 작업자(DataNode)로 역할을 철저히 분리한다.
> 2. **상태 동기화**: 데이터노드는 '하트비트(Heartbeat)'와 '블록 리포트(Block Report)'를 통해 자신의 생존 여부와 데이터 상태를 실시간으로 네임노드에 보고한다.
> 3. **안정성의 핵심**: 네임노드의 메타데이터는 시스템 전체의 지도로서, 장애 시 복구를 위해 Edit Log와 FSImage라는 이중 영속화 장치를 사용한다.

---

## Ⅰ. 네임노드 (NameNode)의 상세 역할

- **정의**: HDFS 클러스터의 마스터 서버로, 전체 파일 시스템의 네임스페이스(Namespace)를 관리한다.
- **주요 기능**:
    - **디렉터리 구조 관리**: 파일 및 디렉터리의 생성, 삭제, 이동 기록.
    - **블록 매핑**: 특정 파일이 어떤 블록들로 구성되며, 그 블록들이 어느 데이터노드에 있는지 정보를 유지한다.
    - **복제 관리**: 블록의 복제본 개수를 모니터링하고 부족 시 추가 복제를 지시한다.
- **특징**: 메타데이터를 빠른 접근을 위해 메모리(RAM)에 상주시키며, 이로 인해 전체 클러스터의 파일 개수가 네임노드의 메모리 용량에 제한을 받는다.

📢 **섹션 요약 비유**: 네임노드는 "은행의 중앙 전산망"과 같다. 내 돈(데이터)이 어느 지점에 보관되어 있는지 장부를 관리하지만, 실제 돈 뭉치를 금고에 들고 있지는 않다.

---

## Ⅱ. 데이터노드 (DataNode)의 동작 및 상호작용

- **정의**: 실제 파일 데이터를 블록 단위로 로컬 디스크에 저장하는 슬레이브 서버다.
- **데이터 흐름도 (ASCII Diagram)**:

```text
    [ NameNode ] <──────────┐ (Heartbeat: "I'm alive")
          │                 │ (Block Report: "I have blocks A, B, C")
          │                 │
    (Commands)              │
 [Replicate Block X]        │
 [Delete Block Y]           │
          │                 │
          ▼                 │
    [ DataNode ] ───────────┘
    ┌───────────────────────┐
    │ Local Disk (FS)       │
    │ ┌──────┐ ┌──────┐     │
    │ │blk_01│ │blk_02│     │
    │ └──────┘ └──────┘     │
    └───────────────────────┘
```

📢 **섹션 요약 비유**: 데이터노드는 "창고 직원"과 같다. 주기적으로 관리자에게 "저 여기 살아있어요(하트비트)"라고 알리고, "지금 창고에 이런 물건들이 있어요(블록 리포트)"라고 보고한다.

---

## Ⅲ. 메타데이터 영속성: FSImage와 Edit Log

네임노드가 갑자기 꺼졌을 때 메모리에 있던 지도를 복구하기 위한 장치다.

- **FSImage**: 특정 시점의 파일 시스템 스냅샷(Snapshot)이다. 전체 구조가 파일로 저장되어 있다.
- **Edit Log**: FSImage 생성 이후에 발생한 모든 변경 사항(파일 생성, 삭제 등)을 기록한 트랜잭션 로그다.
- **복구 과정**: 네임노드 시작 시 FSImage를 읽어 뼈대를 만들고, Edit Log를 순차적으로 실행(Replay)하여 최신 상태를 복원한다.

📢 **섹션 요약 비유**: FSImage는 "어제까지의 가계부 결산"이고, Edit Log는 "오늘 쓴 영수증 뭉치"다. 영수증을 하나씩 가계부에 옮겨 적으면 현재 잔액을 정확히 알 수 있는 것과 같다.

---

## Ⅳ. 세컨더리 네임노드 (Secondary NameNode)

- **오해**: 이름 때문에 네임노드의 백업(Standby)으로 착각하기 쉽지만, 실시간 백업 서버가 아니다.
- **역할 (Checkpointing)**:
    - 네임노드로부터 FSImage와 Edit Log를 주기적으로 가져온다.
    - 둘을 병합(Merge)하여 새로운 FSImage를 생성한다.
    - 결과물을 네임노드에 돌려주어 Edit Log의 크기가 무한정 커지는 것을 방지한다.
- **가치**: 네임노드 재시작 시 Edit Log를 적용하는 시간을 획기적으로 단축시킨다.

📢 **섹션 요약 비유**: 세컨더리 네임노드는 "회계 대행업체"와 같다. 사장님(네임노드)이 바쁘니까 영수증 뭉치를 가져가서 밤새 가계부 정리를 대신 해주는 역할을 한다.

---

## Ⅴ. 노드 간 통신 프로토콜 (Interaction Protocol)

- **하트비트 (Heartbeat)**: 3초(기본값)마다 데이터노드가 네임노드에 보내는 신호. 10분간 신호가 없으면 네임노드는 해당 데이터노드가 죽은 것으로 간주한다.
- **블록 보고 (Block Report)**: 6시간(기본값)마다 데이터노드가 자신이 보관 중인 모든 블록 정보를 상세히 보고한다.
- **클라이언트 요청**: 클라이언트는 네임노드에서 블록 위치를 먼저 알아낸 뒤, 실제 데이터 읽기/쓰기는 해당 데이터노드와 직접 통신한다.

📢 **섹션 요약 비유**: 이들의 대화는 "군대 점호"와 같다. 매일 인원 보고를 하고(하트비트), 가끔은 모든 보급품 수량을 전수 조사(블록 보고)하여 사령부(네임노드)의 장부와 맞추는 과정이다.

## 👶 아이를 위한 비유 (Child Analogy)
> "네임노드는 **'똑똑한 비서'**고, 데이터노드는 **'힘센 일꾼'**이야. 비서는 책상 앞에 앉아서 장부에 '사과 상자는 1번 창고에, 바나나 상자는 2번 창고에 있어요'라고 적기만 해. 일꾼은 창고에서 무거운 상자를 직접 들고 있지. 비서가 일꾼에게 '바나나 상자 하나가 없어졌으니 새로 가져오세요'라고 시키면 일꾼은 그대로 행동한단다!"
