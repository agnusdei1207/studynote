## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파일 제어 블록(FCB, File Control Block)은 파일의 실제 내용(Data)을 제외한 모든 메타데이터(Metadata)를 저장하는 운영체제(OS) 내부의 고유한 데이터 구조체이다.
> 2. **가치**: 유닉스 계열에서는 **아이노드(inode, Index Node)**라고 불리며, 파일명 대신 고유 번호를 통해 파일을 식별함으로써 데이터 관리의 일관성과 효율적인 접근 포인터를 제공한다.
> 3. **융합**: 디렉터리 구조와 데이터 블록 사이를 연결하는 가교 역할을 하며, 파일 권한, 소유주, 생성 시간 및 물리적 블록 주소 등을 통합 관리한다.

---

## Ⅰ. 파일 제어 블록(FCB)의 정의와 중요성 (Definition)

- **정의**: 개별 파일마다 하나씩 존재하는 제어 정보의 집합체다. 파일이 생성될 때 할당되고, 삭제될 때 해제된다.
- **포함 정보**:
    - **파일 권한**: 누가 읽고, 쓰고, 실행할 수 있는가?
    - **파일 크기**: 파일의 현재 바이트 수.
    - **날짜 정보**: 생성(Creation), 수정(Modification), 접근(Access) 시간.
    - **데이터 블록 포인터**: 실제 데이터가 디스크의 어느 블록들에 흩어져 있는가?

📢 **섹션 요약 비유**: FCB는 사람으로 치면 "주민등록등본"과 같다. 그 사람의 이름(파일명은 디렉터리가 관리하지만)을 뺀 본적지(위치), 생년월일(시간), 직업(권한) 등이 상세히 기록되어 있다.

---

## Ⅱ. 유닉스의 아이노드 (inode, Index Node) 구조 (Architecture)

유닉스(Unix) 및 리눅스(Linux) 시스템에서 FCB를 구현하는 방식이 바로 inode다.

### 1. inode의 특징
- **고유 번호 (inode number)**: 파일 이름은 사용자 편의를 위한 별명일 뿐이며, 시스템 내부에서는 숫자로만 관리한다.
- **직접/간접 블록 포인터**: 작은 파일은 직접 블록 주소를 가리키고, 큰 파일은 간접 포인터를 통해 계층적으로 주소를 찾아간다.

### 2. inode 구조 개념도 (ASCII Diagram)

```text
 [ Inode Structure ]               [ Data Blocks in Disk ]
 ┌───────────────────────┐        ┌──────────────────────┐
 │ File Mode (Permissions)│        │ Block #10 (Direct)   │◀──┐
 ├───────────────────────┤        ├──────────────────────┤   │
 │ Owner (UID/GID)       │        │ Block #11 (Direct)   │◀──┤ (10개 내외)
 ├───────────────────────┤        ├──────────────────────┤   │
 │ Size, Time Stamps     │        │ Block #12 (Direct)   │◀──┘
 ├───────────────────────┤        ├──────────────────────┤
 │ Direct Blocks [10]    │───────▶│ ...                  │
 ├───────────────────────┤        ├──────────────────────┤
 │ Single Indirect Block │───┐    │ Single Indirect Area │
 ├───────────────────────┤   │    │ ┌──────────────────┐ │
 │ Double Indirect Block │   └────▶ │ Block #100       │─┼───▶ [Data]
 ├───────────────────────┤        │ │ Block #101       │─┼───▶ [Data]
 │ Triple Indirect Block │        │ └──────────────────┘ │
 └───────────────────────┘        └──────────────────────┘
```

📢 **섹션 요약 비유**: inode의 포인터 방식은 "포스트잇 메모"와 같다. 중요한 메모(데이터)가 많아지면, 다른 종이(간접 블록)에 메모가 어디 있는지 리스트를 적어두는 식이다.

## Ⅲ. FCB의 생명주기와 인메모리 관리 (Lifecycle)

- **디스크 상의 FCB**: 파일이 사용되지 않을 때는 디스크의 특정 구역(inode table 등)에 영구 저장된다.
- **인메모리 FCB**: 파일이 'Open'되면, 운영체제는 디스크의 FCB를 메모리로 복사하여 활성화한다. 이를 통해 파일 접근 시마다 디스크를 읽는 성능 저하를 방지한다.
- **오픈 파일 테이블 연동**: 시스템 전체 오픈 파일 테이블에는 이 메모리상의 FCB 주소가 기록되어 여러 프로세스가 공유한다.

📢 **섹션 요약 비유**: FCB의 메모리 로드는 "도서관 대출증 작성"과 같다. 평소엔 서고 구석에 보관되다가, 누군가 책을 읽으려 하면 데스크(메모리)로 가져와 상태를 체크하는 것이다.

---

## Ⅳ. 파일 이름과 FCB의 분리 (Decoupling)

운영체제 설계에서 가장 중요한 개념 중 하나는 **파일명과 FCB를 분리**한 것이다.

- **디렉터리 엔트리**: "파일명 ──▶ inode 번호"의 매핑만 가지고 있다.
- **하드 링크 (Hard Link)**: 하나의 inode 번호를 가리키는 서로 다른 파일명을 여러 개 만들 수 있다.
- **장점**: 파일 내용을 바꾸지 않고도 이름만 바꾸거나, 여러 위치에서 동일한 파일에 접근하기 용이하다.

📢 **섹션 요약 비유**: 파일명은 "별명"이고 FCB는 "실체"다. '홍길동'을 '의적'이라고 부르든 '호부호형 못하는 자'라고 부르든, 그 사람의 신분(FCB)은 하나뿐인 것과 같다.

---

## Ⅴ. FCB의 한계와 현대적 발전 (Modern Trends)

- **inode 고갈**: 파일 시스템 생성 시 inode 개수가 고정되는 경우, 디스크 용량이 남아도 파일을 더 이상 만들 수 없는 문제가 발생할 수 있다.
- **동적 할당**: 최신 파일 시스템(ZFS, Btrfs 등)은 inode를 동적으로 생성하여 이 한계를 극복한다.
- **확장 속성 (Extended Attributes)**: 기본 FCB 정보 외에 체크섬(Checksum), 압축 여부 등 더 많은 정보를 담기 위한 구조로 진화하고 있다.

📢 **섹션 요약 비유**: 현대의 FCB는 "스마트 신분증"으로 변하고 있다. 단순히 이름과 주소만 적는 게 아니라, 건강 상태나 출입 기록 등 훨씬 많은 정보를 유연하게 담는 추세다.
