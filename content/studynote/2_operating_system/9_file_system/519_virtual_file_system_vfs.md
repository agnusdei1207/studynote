## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 가상 파일 시스템 (VFS, Virtual File System)은 커널 내부에서 다양한 물리적 파일 시스템 (EXT4, NTFS, NFS 등)을 추상화하여 사용자에게 일관된 인터페이스를 제공하는 소프트웨어 계층이다.
> 2. **가치**: 응용 프로그램은 파일이 로컬 디스크에 있는지, USB에 있는지, 혹은 네트워크 너머에 있는지 알 필요 없이 동일한 시스템 호출 (open, read, write 등)을 사용할 수 있다.
> 3. **구조**: `vnode` (Virtual Inode)와 함수 포인터 배열을 활용하여 객체 지향적 다형성을 구현하며, 성능 향상을 위해 dentry 및 inode 캐시를 관리한다.

---

## Ⅰ. 개요 (Context & Background)

- **정의**: 서로 다른 파일 시스템의 구체적인 구현 방식을 숨기고, 상위 계층(시스템 콜 계층)에 단일한 추상화 모델을 제공하는 커널 인터페이스다.
- **배경**: 초기에는 한 OS당 하나의 파일 시스템만 사용했으나, 저장 매체와 통신 기술의 발달로 이기종 파일 시스템을 동시에 지원해야 할 필요성이 생겼다.
- **핵심 특징**:
  - **다형성 (Polymorphism)**: 동일한 `read()` 호출이 하부 파일 시스템 종류에 따라 다른 함수(Ext4_read, NFS_read 등)를 호출한다.
  - **네임스페이스 통합**: 여러 장치의 파일 시스템을 하나의 루트(/) 트리로 묶어 관리한다.

📢 **섹션 요약 비유**: VFS는 "만능 변환 젠더"와 같다. 어떤 모양의 플러그(파일 시스템)가 꽂히더라도, 반대쪽 콘센트(사용자 API)는 항상 동일한 모양으로 유지해주는 장치다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 계층적 구조 시각화 (ASCII Diagram)

```text
 [ Application ]
       │  (System Calls: open, read, write, close)
       ▼
 [ Virtual File System (VFS) Layer ] ─── [ VFS Cache: dentry, inode ]
       │  (Generic Interface, vnode)
       ▼
 ┌──────────────┬──────────────┬──────────────┐
 [ EXT4 FS ]    [ NTFS FS ]    [ NFS Client ] [ ... ]
       │              │              │
       ▼              ▼              ▼
 [ Hard Drive ]  [ External ]   [ Network ]
```

### 2. VFS의 핵심 데이터 구조
- **vnode (Virtual Inode)**: 모든 파일 시스템의 파일 정보를 공통된 형식으로 저장하는 가상 inode다.
- **file structure**: 열린 파일의 상태(현재 위치 등)를 기록한다.
- **dentry (Directory Entry)**: 경로 이름과 vnode 사이의 매핑을 메모리에 캐싱하여 탐색 속도를 높인다.
- **Superblock**: 마운트된 각 파일 시스템의 전체 정보를 담고 있다.

📢 **섹션 요약 비유**: VFS는 "표준 규격서"와 같다. 각 파일 시스템 제조사들은 이 규격서(인터페이스)에 맞춰서 기능을 구현해야만 운영체제라는 거대한 기계에 조립될 수 있다.

---

## Ⅲ. 주요 동작 및 함수 포인터 (Operations)

VFS는 객체 지향의 '인터페이스'와 유사하게 동작한다. 각 파일 시스템은 자신의 기능을 등록한 **Operation Vector** (함수 포인터 배열)를 VFS에 제공한다.

```c
/* VFS의 파일 연산 인터페이스 예시 (C 언어) */
struct file_operations {
    ssize_t (*read) (struct file *, char *, size_t, loff_t *);
    ssize_t (*write) (struct file *, const char *, size_t, loff_t *);
    int (*open) (struct inode *, struct file *);
    /* ... 기타 연산들 ... */
};
```

사용자가 `read()`를 호출하면, VFS는 현재 열린 파일 객체의 `file_operations`를 참조하여 해당 물리 파일 시스템의 실제 `read` 함수를 호출한다.

📢 **섹션 요약 비유**: VFS는 "리모컨 버튼"과 같다. 내가 '채널 올림' 버튼을 누르면, TV 브랜드마다 내부 회로는 다르지만 어쨌든 채널이 올라가는 결과는 같은 것과 같다.

## Ⅳ. 융합 비교 및 분석 (Comparison)

| 비교 항목 | 물리 파일 시스템 (PFS) | 가상 파일 시스템 (VFS) |
|:---|:---|:---|
| **실체** | 디스크 상의 데이터 레이아웃 | 커널 메모리 상의 추상화 계층 |
| **관심사** | 블록 할당, 단편화, 저널링 | 시스템 콜 처리, 인터페이스 통일 |
| **개수** | 장치마다 1개씩 존재 | 시스템 내 단 하나 존재 |
| **데이터 단위** | 섹터, 블록 | 파일, 디렉터리, 객체 |

📢 **섹션 요약 비유**: 물리 파일 시스템이 "각 나라의 고유한 언어"라면, VFS는 "UN 통역 본부"와 같다. 모든 말을 공용어(VFS API)로 번역하여 전달하는 역할을 한다.

## Ⅴ. 기술적 구현 및 성능 최적화 (Implementation)

- **VFS 캐시**:
  - **dentry 캐시**: 파일 경로를 inode로 변환하는 반복적인 작업을 줄인다.
  - **inode 캐시**: 자주 사용하는 파일의 속성 정보를 메모리에 상주시킨다.
- **다양한 마운팅**: 로컬 디스크뿐만 아니라 메모리 기반 파일 시스템(tmpfs), 프로세스 정보를 담은 가상 시스템(procfs) 등도 VFS를 통해 동일한 방식으로 접근된다.
- **추상화의 비용**: 약간의 함수 호출 오버헤드가 발생하지만, 개발 생산성과 시스템 확장성 면에서의 이득이 훨씬 크다.

📢 **섹션 요약 비유**: VFS는 "도시의 도로망"과 같다. 차종(파일 시스템)이 무엇이든 도로(VFS) 위를 달리는 규칙만 지키면 어디든 갈 수 있게 해주는 인프라다.
