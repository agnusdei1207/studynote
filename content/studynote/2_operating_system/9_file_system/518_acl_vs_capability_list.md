## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 접근 제어 리스트 (ACL, Access Control List)는 자원 중심의 권한 관리 방식이고, 능력 리스트 (Capability List)는 사용자 중심의 권한 관리 방식이다.
> 2. **가치**: ACL은 특정 자원에 누가 접근할 수 있는지 확인하기 쉽고, Capability List는 특정 사용자가 어떤 권한들을 가졌는지 파악하고 전파하는 데 유리하다.
> 3. **선택**: 현대 운영체제는 구현의 편의성과 보안성 사이의 균형을 위해 두 방식을 혼합하여 사용하거나, 객체 중심의 ACL을 기본으로 채택하는 경우가 많다.

---

## Ⅰ. 개요 (Context & Background)

- **정의**: 시스템 내의 객체(파일, 디렉터리 등)와 주체(사용자, 프로세스) 사이의 접근 권한을 정의한 '접근 제어 행렬 (Access Matrix)'을 물리적으로 구현하는 두 가지 상반된 접근법이다.
- **배경**: 단순한 rwx 비트만으로는 복잡한 다중 사용자 환경의 미세한 권한 제어가 불가능해지면서, 더 정교한 관리 체계가 필요하게 되었다.
- **핵심 차이**: 행렬을 '열(Column)' 단위로 쪼개느냐, '행(Row)' 단위로 쪼개느냐의 차이다.

📢 **섹션 요약 비유**: ACL은 "방 문 앞에 붙은 명단"과 같고, Capability List는 "사람들이 들고 다니는 마스터키"와 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 두 방식의 구조 비교 (ASCII Diagram)

```text
 [ Access Matrix ]           [ ACL (Object-centric) ]
          File A  File B     File A: { (User1, RW), (User2, R) }
 User 1     RW      R    ──▶ File B: { (User1, R) }
 User 2     R       -        (파일의 메타데이터에 명단 저장)

                             [ Capability List (Subject-centric) ]
                             User 1: { (FileA, RW), (FileB, R) }
                         ──▶ User 2: { (FileA, R) }
                             (사용자의 프로세스 제어 블록에 티켓 저장)
```

### 2. 주요 메커니즘
- **ACL (Access Control List)**:
  - 각 파일은 자신에게 접근 가능한 사용자들의 리스트를 가지고 있다.
  - 파일 접근 시, 커널이 리스트를 검색하여 요청한 사용자가 있는지 확인한다.
- **Capability List (Ticket)**:
  - 각 사용자는 자신이 접근할 수 있는 객체들의 목록과 권한이 적힌 '티켓' 혹은 '포인터'를 가진다.
  - 이 리스트는 사용자가 직접 수정할 수 없도록 커널 영역에서 관리된다.

📢 **섹션 요약 비유**: ACL은 "클럽 입구의 게스트 명단" 확인 절차와 같고, Capability List는 "클럽 입장 팔찌"를 보여주고 바로 들어가는 것과 같다.

---

## Ⅲ. 비교 및 장단점 분석 (Trade-offs)

| 비교 항목 | ACL (접근 제어 리스트) | Capability List (능력 리스트) |
|:---|:---|:---|
| **관리 중심** | 객체 (Object) 중심 | 주체 (Subject) 중심 |
| **권한 확인** | 파일 접근 시마다 리스트 검색 | 티켓만 제시하면 되어 빠름 |
| **권한 회수** | 매우 쉬움 (파일 명단에서 삭제) | 어려움 (사용자의 티켓을 회수해야 함) |
| **저장 위치** | 파일 시스템 메타데이터 (inode) | 프로세스 제어 블록 (PCB) |
| **시스템 전체 파악** | 특정 파일의 공유 현황 파악 용이 | 특정 사용자의 전체 권한 파악 용이 |

📢 **섹션 요약 비유**: ACL은 "블랙리스트 관리"에 유리하고, Capability List는 "VIP 프리패스권 발행"에 유리하다.

---

## Ⅳ. 융합 및 실무 적용 (Implementation)

- **현대 OS의 방식**:
  - 대부분의 파일 시스템(Windows NTFS, Linux ext4 ACL)은 **ACL**을 기본으로 사용한다.
  - 하지만 성능 최적화를 위해 파일을 처음 열 때(Open)만 ACL을 확인하고, 이후에는 **File Descriptor (능력 리스트의 일종)**를 통해 빠르게 접근하는 방식을 혼용한다.
- **능력 기반 보안 (Capability-based Security)**:
  - 객체 지향 운영체제나 마이크로커널(L4, seL4)에서는 모든 자원 접근을 Capability로 관리하여 세밀한 샌드박싱을 구현한다.

📢 **섹션 요약 비유**: 실무에서는 "처음 입장할 때만 명단을 확인하고(ACL), 안에서는 번호표(Capability)를 들고 다니게 하는" 하이브리드 방식을 선호한다.

---

## Ⅴ. 보안 이슈와 해결책 (Issues & Solutions)

- **권한 전파 (Propagation)**: Capability 방식에서는 티켓을 다른 사용자에게 빌려줄 수 있는 권한까지 포함될 수 있어 관리가 복잡하다.
- **권한 회수 (Revocation)**: ACL은 파일에서 이름만 지우면 끝이지만, Capability는 시스템에 흩어진 티켓을 모두 무효화하기 위해 간접 참조(Indirection) 테이블을 사용하기도 한다.
- **결론**: 일반적인 파일 관리에는 ACL이 직관적이고 관리가 편하지만, 초고보안 시스템이나 분산 환경에서는 Capability 방식이 성능과 유연성 면에서 장점을 가진다.

📢 **섹션 요약 비유**: "잃어버린 신용카드를 정지시키는 것(Capability 회수)"이 "가게 주인이 특정 손님을 못 오게 하는 것(ACL)"보다 훨씬 힘든 일인 것과 같은 이치다.
