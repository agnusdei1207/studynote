## 핵심 인사이트 (3줄 요약)
> 1. **복제의 목적**: HDFS는 데이터 손실을 방지하고 가용성을 높이기 위해 블록을 여러 노드에 복제(Default 3회)하며, 이를 위해 '랙 인식(Rack Awareness)'이라는 지능적 배치 전략을 사용한다.
> 2. **SPOF 극복**: 단일 네임노드의 장애가 전체 시스템 중단으로 이어지는 것을 막기 위해 '고가용성(HA, High Availability)' 기능을 통해 Active/Standby 이중화 구조를 유지한다.
> 3. **데이터 일관성**: 공유 저장소(Shared Storage)나 쿼럼 저널 매니저(QJM, Quorum Journal Manager)를 활용하여 두 네임노드 간의 메타데이터를 실시간으로 동기화한다.

---

## Ⅰ. 데이터 복제 전략: 3-way Replication

- **기본 정책**: 모든 데이터 블록은 최소 3개의 복제본을 유지한다.
- **배치 방식 (Rack Awareness)**:
    - **1번째 복제본**: 데이터를 작성하는 클라이언트와 같은 노드에 저장한다.
    - **2번째 복제본**: 다른 랙(Rack)에 있는 임의의 노드에 저장한다.
    - **3번째 복제본**: 2번째 복제본과 같은 랙의 다른 노드에 저장한다.
- **기대 효과**: 단일 노드 장애는 물론, 전체 랙의 전원이 나가는 대규모 장애 시에도 데이터를 보호할 수 있다.

📢 **섹션 요약 비유**: 복제 전략은 "중요 문서를 3장 복사해서 하나는 내 가방에, 하나는 친구네 집에, 마지막 하나는 친구네 옆집에 맡겨두는 것"과 같다.

---

## Ⅱ. 랙 인식 (Rack Awareness) 정책

클러스터의 물리적 구조를 고려하여 데이터의 생존 가능성을 극대화한다.

### 1. 랙 배치 구조도 (ASCII Diagram)

```text
       [ Rack 1 ]                  [ Rack 2 ]
    ┌───┴───┴───┐               ┌───┴───┴───┐
    │  Node 1   │               │  Node 3   │
    │ [Block A] │ (Replica 1)   │ [Block A] │ (Replica 2)
    └───────────┘               └───────────┘
    ┌───────────┐               ┌───────────┐
    │  Node 2   │               │  Node 4   │
    │           │               │ [Block A] │ (Replica 3)
    └───────────┘               └───────────┘
```

📢 **섹션 요약 비유**: 랙 인식은 "계란을 여러 바구니(랙)에 나누어 담는 것"이다. 한 바구니를 떨어뜨려도 다른 바구니에 담긴 계란은 무사한 것과 같은 원리다.

## Ⅲ. 네임노드 고가용성 (High Availability)

HDFS 2.0부터 도입된 기능으로, 네임노드의 단일 장애점(SPOF) 문제를 해결한다.

- **Active NameNode**: 현재 클라이언트 요청을 처리하는 실제 마스터다.
- **Standby NameNode**: Active 노드와 동일한 메타데이터를 유지하며 대기하다가, 장애 시 즉시 Active 역할을 승계한다.
- **데이터 동기화**: Edit Log를 공용 저장소(QJM 등)에 기록하여 실시간으로 공유한다.

📢 **섹션 요약 비유**: 고가용성은 "비행기의 보조 조종사"와 같다. 기장(Active)이 조종을 못 하게 되면 부기장(Standby)이 즉시 조종간을 넘겨받아 안전하게 비행하는 시스템이다.

---

## Ⅳ. 주/부 전환 및 페일오버 (Failover)

장애가 발생했을 때 역할이 교체되는 과정이다.

- **주키퍼 (ZooKeeper)**: 하둡 클러스터의 상태를 감시하고 페일오버를 결정하는 조정자 역할을 한다.
- **ZKFC (ZK Failover Controller)**: 각 네임노드에 상주하며 주키퍼와 통신하여 리더 선출을 돕는다.
- **자동 페일오버**: Active 노드의 하트비트가 끊기면 주키퍼가 감지하여 Standby를 Active로 승격시킨다.

📢 **섹션 요약 비유**: 페일오버는 "계주 경기에서 바통 터치"와 같다. 앞 주자(Active)가 넘어지면 뒤에 대기하던 선수(Standby)가 바통을 이어받아 경기를 계속하는 것이다.

---

## Ⅴ. 쿼럼 저널 매니저 (QJM)와 펜싱 (Fencing)

두 마스터가 동시에 왕이 되려는 상황을 방지하는 안전장치다.

- **QJM (Quorum Journal Manager)**: 다수결(Quorum) 원칙에 따라 메타데이터 변경 내역을 기록하는 분산 서비스다. (예: 3대 중 2대 찬성 시 기록)
- **Split-brain 방지**: 네트워크 단절로 두 네임노드가 모두 자신이 Active라고 믿는 위험한 상황을 막아야 한다.
- **펜싱 (Fencing)**: 새로운 Active가 선출되면 이전 Active가 데이터를 더 이상 고치지 못하게 강제로 차단(I/O 차단 또는 노드 재시작)하는 기법이다.

📢 **섹션 요약 비유**: 펜싱은 "옛날 왕의 도장을 회수하는 것"이다. 새로운 왕이 등극하면 이전 왕이 함부로 명령을 내리지 못하게 권한을 뺏고 차단하는 과정이다.
