## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마운팅 (Mounting)은 서로 다른 저장 장치에 존재하는 독립적인 파일 시스템을 하나의 통합된 디렉터리 트리 계층 내 특정 위치 (Mount Point)에 결합하는 프로세스다.
> 2. **가치**: 사용자에게 물리적 하드웨어 구성을 은폐하고, 논리적으로 단일한 네임스페이스 (Unified Namespace)를 제공하여 일관된 파일 접근 인터페이스를 보장한다.
> 3. **제어**: 마운트 테이블 (Mount Table)을 통해 각 마운트 지점의 옵션 (Read-only, No-exec 등)을 관리하며, 장치 제거 전 파일 시스템의 일관성을 위해 언마운트 (Unmount) 과정이 필수적이다.

---

## Ⅰ. 개요 (Context & Background)

- **정의**: 특정 저장 장치(하드디스크, USB, 네트워크 드라이브 등) 내의 파일 시스템 구조를 운영체제가 인식하고, 기존 디렉터리 구조의 한 분기점으로 편입시키는 작업이다.
- **배경**: 초기에는 단일 디스크만 사용했으나, 저장 장치가 다양해지고 네트워크 저장소가 등장하면서 이들을 하나의 논리적 트리로 묶어 관리할 필요가 생겼다.
- **핵심 용어**:
  - **마운트 지점 (Mount Point)**: 외조 파일 시스템이 결합될 대상 디렉터리. (보통 빈 디렉터리 사용)
  - **마운트 테이블 (Mount Table)**: 현재 시스템에 마운트된 모든 장치와 경로 정보를 담은 커널 내부 데이터 구조.

📢 **섹션 요약 비유**: 마운팅은 "나무에 다른 나무 가지를 접붙이는 것"과 같다. 원래 있던 나무(루트 파일 시스템)의 특정 가지 끝에 새로운 가지(USB 등)를 이어 붙여, 마치 처음부터 하나의 나무였던 것처럼 자라게(관리되게) 하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 마운트 전후 구조 시각화 (ASCII Diagram)

```text
 [ 기존 시스템 트리 ]          [ 외부 장치 (USB) ]
      / (Root)                     / (USB Root)
      ├─ [ bin ]                   ├─ photo.jpg
      ├─ [ mnt ] <── Mount!        └─ data.zip
      └─ [ etc ]

             ▼ [ 마운트 완료 후 ] ▼

               [ Root (/) ]
                    │
          ┌─────────┼─────────┐
       [ bin ]   [ mnt ]   [ etc ]
                    │
          ┌─────────┴─────────┐
     photo.jpg            data.zip  <── USB의 내용이 mnt 아래에 나타남
```

### 2. 마운트 과정 (Mounting Process)
1. **장치 인식**: 운영체제가 새 저장 장치의 연결을 감지하고 디바이스 드라이버를 로드한다.
2. **슈퍼블록 판독**: 장치의 첫 섹터 부근에 있는 슈퍼블록 (Superblock)을 읽어 파일 시스템의 유형, 크기, 상태를 확인한다.
3. **무결성 검사**: 파일 시스템이 정상인지 확인하고, 마운트 지점 디렉터리의 inode를 외부 장치의 루트 inode와 연결한다.
4. **테이블 갱신**: 커널의 마운트 테이블에 해당 정보를 등록하여 이후의 모든 경로 탐색이 외부 장치로 리다이렉트되도록 한다.

📢 **섹션 요약 비유**: 마운팅은 "새로운 역을 지하철 노선도에 추가하는 것"과 같다. 선로(물리 연결)가 깔려 있어도 노선도(디렉터리 트리)에 이름을 올려야 승객(사용자)이 찾아갈 수 있는 것과 같다.

## Ⅲ. 마운트 옵션과 보안 (Options & Security)

- **읽기 전용 (ro, Read-only)**: 중요 시스템 데이터나 백업 장치를 보호하기 위해 쓰기 권한을 차단한 상태로 마운트한다.
- **실행 방지 (noexec)**: 보안상 위험한 바이너리 파일이 실행되지 않도록 설정한다. (주로 `/tmp`나 USB에 적용)
- **자동 마운트 (Automount)**: 부팅 시 자동으로 마운트하거나 (`/etc/fstab`), 장치 접근 시점에 실시간으로 마운트하는 방식이다.

📢 **섹션 요약 비유**: 마운트 옵션은 "방문에 붙이는 안내문"과 같다. '눈으로만 보세요(ro)'나 '장난 금지(noexec)' 같은 규칙을 정해 방 안의 자원을 안전하게 지키는 것이다.

## Ⅳ. 융합 비교 및 분석 (Comparison)

| 비교 항목 | 유닉스/리눅스 방식 | 윈도우(Windows) 방식 |
|:---|:---|:---|
| **네임스페이스** | 단일 트리 (Unified) | 드라이브 문자 (C:, D:) 분리 |
| **마운트 위치** | 임의의 디렉터리 (Mount Point) | 루트 레벨 (Physical Drive) |
| **유연성** | 매우 높음 (디렉터리 깊숙이 가능) | 낮음 |
| **사용자 인지** | 물리적 위치를 몰라도 사용 가능 | 장치가 바뀌면 경로가 바뀜 |

📢 **섹션 요약 비유**: 유닉스 방식이 "모든 주소가 '서울시' 아래에 있는 것"이라면, 윈도우 방식은 "서울, 부산, 광주가 각각 독립된 국가인 것"처럼 동작한다.

## Ⅴ. 기술적 구현 및 주의사항 (Implementation)

- **언마운트 (Unmount)**: 장치를 제거하기 전, 버퍼에 남은 데이터를 디스크에 쓰고 연결을 끊는 과정이다. `umount` 명령을 사용한다.
- **Device Busy 오류**: 특정 프로세스가 마운트된 디렉터리의 파일을 사용 중이거나, `cd` 명령으로 그 안에 들어가 있으면 언마운트가 실패한다. (`lsof`나 `fuser`로 확인 필요)
- **VFS와의 관계**: 가상 파일 시스템 (VFS) 계층이 마운트된 서로 다른 파일 시스템들 사이의 가교 역할을 수행하여, 서로 다른 포맷(EXT4, NTFS) 간의 데이터 이동을 가능하게 한다.

📢 **섹션 요약 비유**: 언마운트는 "안전하게 가전제품 코드를 뽑는 것"과 같다. 전원이 켜진 채로(사용 중일 때) 무턱대고 뽑으면 기기(데이터)가 고장 날 수 있으니 주의해야 한다.
