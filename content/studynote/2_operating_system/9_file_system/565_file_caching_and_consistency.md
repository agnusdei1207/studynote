## 핵심 인사이트 (3줄 요약)
> 1. **캐싱의 본질**: 파일 캐싱(File Caching)은 네트워크 지연을 극복하기 위해 서버의 데이터를 클라이언트 로컬 메모리에 복사해 두는 기법이다.
> 2. **정책의 충돌**: 즉각적인 서버 반영으로 데이터 안전성을 우선하는 '통과 쓰기(Write-through)'와 로컬 작업을 모았다가 한꺼번에 반영하여 성능을 극대화하는 '나중 쓰기(Write-back)' 사이의 선택이 핵심이다.
> 3. **일관성의 대가**: 캐싱 효율이 높아질수록 여러 노드 간의 데이터 불일치(Inconsistency) 위험이 커지며, 이를 해결하기 위한 무효화(Invalidation) 프로토콜이 필수적이다.

---

## Ⅰ. 파일 캐싱 (File Caching)의 원리

- **정의**: 서버에 있는 파일의 일부분을 클라이언트 노드의 메인 메모리나 로컬 디스크에 임시로 저장하여 재사용하는 기술이다.
- **장점**:
    - **응답 시간 단축**: 네트워크 전송 없이 즉시 데이터 접근 가능.
    - **서버 부하 감소**: 서버로 가는 요청 횟수를 획기적으로 줄임.
- **단위**: 보통 블록(Block) 단위로 캐싱하며, 최근에는 파일 전체를 캐싱하는 방식(AFS 등)도 사용된다.

📢 **섹션 요약 비유**: 캐싱은 "주머니 속의 비상금"과 같다. 멀리 있는 은행(서버)에 매번 가지 않아도 당장 필요한 돈(데이터)을 주머니(캐시)에서 꺼내 쓸 수 있는 것과 같다.

---

## Ⅱ. 쓰기 정책 (Write Policy) 비교 (ASCII Diagram)

데이터가 수정되었을 때 서버에 반영되는 시점의 차이를 나타낸다.

```text
 1. Write-through (Safety First)
 [Client Cache]      [Network]       [Server Storage]
     Update A ───────▶ 전송 ─────────▶ Save A (Sync)
 (Cache and Server are always same)

 2. Write-back (Performance First)
 [Client Cache]      [Network]       [Server Storage]
     Update B (Local)
     Update C (Local)
     ...
     [Close/Flush] ──▶ 전송 ─────────▶ Save B, C (Async)
 (Server is updated later in bulk)
```

📢 **섹션 요약 비유**: 통과 쓰기는 "매 글자마다 보내는 문자 메시지"이고, 나중 쓰기는 "다 써서 한꺼번에 보내는 편지"다.

---

## Ⅲ. 통과 쓰기 (Write-through)

- **정의**: 클라이언트가 캐시의 데이터를 수정하는 즉시 서버의 원본 데이터도 함께 업데이트하는 방식이다.
- **특징**:
    - **높은 신뢰성**: 클라이언트 장애가 발생해도 서버 데이터는 최신 상태다.
    - **일관성 유지 용이**: 서버가 항상 최신본을 유지하므로 관리가 쉽다.
- **단점**: 매번 쓰기 연산 시 네트워크 지연이 발생하여 쓰기 성능이 저하된다.

📢 **섹션 요약 비유**: 통과 쓰기는 "실시간 공유 문서"와 같다. 내가 한 글자만 고쳐도 서버에 바로 저장되어 안전하지만, 인터넷이 느리면 글자가 늦게 써지는 불편함이 있다.

---

## Ⅳ. 나중 쓰기 (Write-back / Delayed Write)

- **정의**: 클라이언트가 캐시를 수정한 후 즉시 서버에 알리지 않고, 나중에 특정 조건(캐시 교체, 파일 닫기 등)이 충족될 때 한꺼번에 서버에 쓰는 방식이다.
- **특징**:
    - **고성능**: 네트워크 오버헤드가 작아 매우 빠른 쓰기가 가능하다.
    - **쓰기 통합**: 여러 번의 수정을 하나의 네트워크 패킷으로 묶어 보낼 수 있다.
- **단점**: 서버 반영 전에 클라이언트가 다운되면 수정 내용이 소실될 위험이 있다.

📢 **섹션 요약 비유**: 나중 쓰기는 "오프라인 일기장"과 같다. 일기장에 내용을 잔뜩 써두었다가 나중에 한꺼번에 블로그에 올리는 것과 같다. 글은 빨리 쓸 수 있지만, 일기장을 잃어버리면 블로그에 올리지 못한 내용은 다 사라진다.

---

## Ⅴ. 캐시 일관성 문제와 해결 (Consistency Issues)

여러 클라이언트가 동일한 파일을 캐싱하고 있을 때 한 명이 수정하면 다른 캐시들은 구형(Stale) 데이터가 된다.

- **서버 개입형 (Server-initiated)**: 서버가 데이터 변경을 감지하고 모든 클라이언트에게 캐시를 무효화(Invalidation)하라고 명령한다.
- **클라이언트 검증형 (Client-initiated)**: 클라이언트가 데이터를 읽기 전에 서버에 "내 캐시가 아직 유효한가?"라고 매번 확인한다.
- **유효 시간(TTL) 방식**: 일정 시간이 지나면 캐시가 자동으로 만료되게 하여 느슨한 일관성을 유지한다.

📢 **섹션 요약 비유**: 일관성 유지는 "뉴스 속보"와 같다. 상황이 바뀌면 방송국(서버)에서 자막(무효화 신호)을 띄워 시청자(클라이언트)가 잘못된 정보(캐시)를 믿지 않게 하는 것이다.

## 👶 아이를 위한 비유 (Child Analogy)
> "**통과 쓰기**는 네가 장난감을 가지고 놀면서 하나를 정리할 때마다 엄마한테 보고하는 거야. 엄마는 네가 뭘 정리했는지 바로 알 수 있지. **나중 쓰기**는 장난감을 다 가지고 놀고 나서 마지막에 한꺼번에 정리하고 보고하는 거야. 더 빨리 놀 수 있지만, 중간에 엄마가 물어보면 아직 정리가 덜 끝났을 수도 있단다!"
