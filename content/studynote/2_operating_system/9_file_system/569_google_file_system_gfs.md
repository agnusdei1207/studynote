## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 구글 파일 시스템(GFS, Google File System)은 구글의 검색 엔진과 대규모 데이터 처리를 지원하기 위해 설계된 확장성 높은 분산 파일 시스템이다.
> 2. **혁신**: 저가형 PC 수천 대를 연결하여 시스템을 구축하면서도, 하드웨어 장애를 상시적인 현상으로 받아들여 소프트웨어 차원의 자동 복구와 데이터 무결성을 실현했다.
> 3. **특화**: 주로 거대 파일의 순차적 읽기와 추가(Append) 연산에 최적화되어 있으며, 이는 이후 하둡 분산 파일 시스템(HDFS)의 설계 모델이 되었다.

---

## Ⅰ. GFS의 설계 배경 및 목표 (Philosophy)

- **상시 장애(Component Failures)**: 수천 개의 노드 중 일부가 고장 나는 것은 예외가 아니라 '일상'이다. 시스템은 이를 자동으로 감지하고 복구해야 한다.
- **거대 파일(Huge Files)**: 수 GB에서 수 TB 규모의 파일을 효율적으로 관리하는 데 초점을 맞춘다.
- **연산 패턴**: 임의의 쓰기(Random Write)보다는 대량의 순차적 읽기와 데이터 추가(Record Append)가 주를 이룬다.
- **높은 처리량(High Throughput)**: 지연 시간(Latency) 최소화보다는 대역폭(Bandwidth) 극대화에 설계 비중을 둔다.

📢 **섹션 요약 비유**: GFS는 "거대한 컨테이너 터미널"과 같다. 아주 큰 짐들을 싣고 내리는 데 최적화되어 있으며, 크레인 하나가 고장 나도 터미널 전체 운영에는 지장이 없도록 설계되었다.

---

## Ⅱ. GFS 시스템 아키텍처 (Architecture)

단일 마스터(Master)와 여러 청크 서버(Chunkserver)로 구성된 구조다.

### 1. 상호작용 구조도 (ASCII Diagram)

```text
       [ GFS Client ]                 [ GFS Master ]
       ┌────────────┐                 ┌─────────────────────┐
       │ 1. Ask Loc ├────────────────▶│ Metadata (Namespc,  │
       │            │◀────────────────┤ Mapping, Leases)    │
       └─────┬──────┘                 └──────────┬──────────┘
             │ (2. Get Loc)                      │ (Background)
             ▼                                   ▼
      [ Chunkservers ]                  [ Chunkservers ]
      ┌──────────────┐                  ┌──────────────┐
      │ Primary C-S  │◀────────────────▶│ Secondary C-S │
      │ [Chunk A]    │ (3. Actual Data) │ [Chunk A]    │
      └──────────────┘                  └──────────────┘
```

📢 **섹션 요약 비유**: GFS 아키텍처는 "대형 서점의 안내소"와 같다. 안내소(Master)에서 책의 위치(Chunk Location)만 물어보고, 실제 책은 서가(Chunkserver)로 직접 가서 꺼내오는 방식이다.

---

## Ⅲ. 주요 구성 요소의 역할

- **마스터 (Master)**:
    - **메타데이터 관리**: 네임스페이스, 접근 제어 정보, 파일-청크 매핑 정보를 메모리에서 관리한다.
    - **청크 관리**: 청크 서버와의 통신(Heartbeat)을 통해 청크의 상태와 위치를 파악한다.
- **청크 서버 (Chunkserver)**:
    - **물리적 저장**: 청크를 로컬 디스크에 Linux 파일 형태로 저장한다.
    - **청크 (Chunk)**: 파일을 64MB 고정 크기로 나눈 단위다. 각 청크는 고유한 64비트 핸들(Handle)을 가진다.

📢 **섹션 요약 비유**: 마스터는 "도서관 관장"이고 청크 서버는 "서가 담당 직원"이다. 관장은 도서 목록만 쥐고 있고, 실제 책 관리는 담당 직원들이 각 구역에서 수행한다.

---

## Ⅳ. GFS의 특징적 메커니즘

- **청크 리스 (Chunk Lease)**: 마스터가 특정 청크 서버에 쓰기 권한(Lease)을 부여하여 'Primary'로 지정함으로써, 여러 복제본 간의 수정 순서를 결정하게 한다.
- **스냅샷 (Snapshot)**: 복사-쓰기(Copy-on-write) 기법을 사용하여 파일이나 디렉터리 트리의 복사본을 순식간에 생성한다.
- **가비지 컬렉션 (Garbage Collection)**: 파일 삭제 시 즉시 지우지 않고 숨긴 뒤, 나중에 마스터가 정기적인 검사를 통해 청크 서버에서 실제 데이터를 회수한다.

📢 **섹션 요약 비유**: 청크 리스는 "작업 반장 임명"과 같다. 여러 일꾼 중 한 명에게 완장을 채워 작업 순서를 정하게 하는 것이다. 가비지 컬렉션은 "휴지통 비우기"와 같아서, 버린 물건을 바로 소각하지 않고 모아두었다가 한꺼번에 처리하는 방식이다.

## Ⅴ. GFS vs HDFS 비교 분석

| 비교 항목 | GFS (Google File System) | HDFS (Hadoop DFS) |
| :--- | :--- | :--- |
| **마스터 명칭** | Master | NameNode |
| **작업자 명칭** | Chunkserver | DataNode |
| **데이터 단위** | Chunk (64MB) | Block (64MB / 128MB) |
| **변경 방식** | Record Append 중심 | Append 가능 (1.0 이후) |
| **로그 관리** | Master가 Operation Log 관리 | NameNode가 Edit Log 관리 |

📢 **섹션 요약 비유**: GFS와 HDFS는 "쌍둥이 형제"와 같다. 형인 GFS가 설계 원칙을 세웠고, 동생인 HDFS가 그 원칙을 따라 만들어진 오픈소스 버전이기 때문에 근본적인 유전자는 거의 동일하다.

## 👶 아이를 위한 비유 (Child Analogy)
> "구글 파일 시스템은 **'거대한 창고 관리 로봇'**이야. 너무 큰 물건들이 매일 쏟아져 들어오니까, 로봇 대장(Master)이 물건을 적당한 크기(Chunk)로 잘라서 여러 작은 창고(Chunkserver)에 나누어 넣어둬. 만약 창고 한두 군데가 무너져도 대장 로봇은 똑같은 물건을 다른 창고에 미리 복사해 뒀기 때문에 금방 찾아낼 수 있단다!"
