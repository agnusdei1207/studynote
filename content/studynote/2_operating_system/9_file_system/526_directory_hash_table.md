# 526. 디렉터리 구현 - 해시 테이블 (Hash Table)

---

### Ⅰ. 개요 (Context & Background)

**개념 및 정의**
해시 테이블(Hash Table) 디렉터리는 파일 시스템(File System)에서 파일명과 같은 키(Key) 값을 수학적인 해시 함수(Hash Function)에 입력하여, 해당 데이터의 메타데이터가 저장된 위치(Inode Pointer)를 가리키는 버킷(Bucket)의 주소를 즉시 계산해내는 자료 구조이다. 이는 기존의 선형 탐색(Linear Search) 방식이 가진 순차적 접근의 한계를 극복하고, '계산에 의한 접근(Calculative Access)'을 통해 데이터 검색 속도를 획기적으로 향상시킨다.

**등장 배경**
1.  **기존 한계**: 초기 파일 시스템은 연결 리스트(Linked List)나 배열을 기반으로 하여, 파일을 찾기 위해 디렉터리 엔트리를 처음부터 순회해야 했다. 이는 파일 수(N)가 증가함에 따라 검색 시간이 선형적으로(O(N)) 증가하는 성능 병목을 야기했다.
2.  **혁신적 패러다임**: '비교(Comparison)' 기반 탐색에서 '주소 계산(Address Arithmetic)' 기반 탐색으로의 패러다임 전환이 이루어졌다. 해시 함수를 통해 키를 인덱스로 변환함으로써, 비교 연산 없이 직접 접근이 가능해졌다.
3.  **현재의 비즈니스 요구**: 현대의 클라우드 및 빅데이터 환경에서는 단일 디렉터리에 수십만 개 이상의 파일이 존재하는 것이 일반적이다. 이러한 환경에서 마이크로초(µs) 단위의 짧은 지연 시간(Latency)을 보장하기 위해 해시 테이블은 선택이 아닌 필수 요소가 되었다.

**💡 비유**
도서관에서 책을 찾을 때, 분류기호가 적힌 라벨을 하나하나 확인하며 책장을 훑는 방식(선형 리스트) 대신, 사서에게 책 제목을 말하자마자 해당 책이 놓인 정확한 좌표(층, 열, 번호)가 적힌 포스트잇을 즉시 받아보는 방식과 같다.

**📢 섹션 요약 비유**: 해시 테이블 디렉터리는 거대한 창고에 **"순간 이동 장치(Teleporter)"**를 설치한 것과 같습니다. 물건의 이름(파일명)만 입력하면 비효율적인 탐색 과정 없이, 시스템이 계산한 좌표값(해시 주소)을 통해 단숨에 목적지 버킷으로 이동하는 원리입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

해시 테이블 디렉터리의 성능은 해시 함수의 분포도와 충돌(Collision) 처리 기법에 의해 결정된다. 구조적으로 버킷 배열과 해시 함수, 그리고 충돌 해결을 위한 체이닝(Chaining) 또는 개방 주소법(Open Addressing)으로 구성된다.

**구성 요소 상세 분석**

| 구성 요소 (Component) | 역할 (Role) | 내부 동작 (Internal Operation) | 프로토콜/메커니즘 | 비유 |
|:---|:---|:---|:---|:---|
| **Key (File Name)** | 검색 입력값 | 사용자가 요청한 고유 문자열 (e.g., "report.pdf") | Input Vector | 도서 제목 |
| **Hash Function** | 주소 변환기 | 키를 정수형 Index로 변환 (Modulo 연산, Bit-shift) | $h(k) = k \pmod M$ | 좌표 계산기 |
| **Bucket Array** | 저장소 | 포인터(Inode)를 저장하는 고정 크기 슬롯의 집합 | Direct Access Table | 사물함 |
| **Collision Handler** | 충돌 조정기 | 중복 주소 발생 시 데이터를 연결(Chaining)하거나 재 Probe | Chaining / Probing | 분류대 |
| **Load Factor (α)** | 밀도 지표 | (저장된 항목 수 / 버킷 총 개수) | 성능 척도 (0 < α < 1) | 혼잡도 |

**ASCII 구조 다이어그램: 해시 테이블 디렉터리의 데이터 흐름**

아래 다이어그램은 파일명 "kernel.img"를 검색하는 과정을 도식화한 것이다. 해시 함수를 통해 인덱스를 계산하고, 충돌이 발생했을 때 체이닝(Chaining)을 통해 실제 Inode를 찾아내는 과정을 보여준다.

```text
   [ 1. User Request ]
   Search for "kernel.img"
          │
          ▼
┌──────────────────────────────────────────┐
│  2. Hash Function (h)                    │
│  Input : "kernel.img" (ASCII String)     │
│  Logic : Sum(ASCII) % Bucket_Size(16)    │
├──────────────────────────────────────────┤
│  Result Index : 5 (Hex 0x05)             │
└──────────────────────────────────────────┘
          │
          ▼
┌───────────────────────────────────────────────────────────┐
│  3. Directory Entry Table (Hash Table in Memory)          │
│  ┌───┬───┬───┬───┬───┬───┬───┬───┐                        │
│  │ 0 │ 1 │ 2 │ 3 │ 4 │[5]│ 6 │...│  <-- Index             │
│  ├───┼───┼───┼───┼───┼───┼───┼───┤                        │
│  │   │   │   │   │   │ ● │   │   │  <-- Pointer to Head   │
│  └───┴───┴───┴───┴───┴┬─┴───┴───┴───┘                        │
│                      │                                     │
│           (Collision Resolution Area)                      │
│                      ▼                                     │
│      ┌──────────────────────────────────────┐              │
│      │ Linked List (Chain)                  │              │
│      │ 1. [ name: "config.sys", Inode: #88 ]│              │
│      │ 2. [ name: "kernel.img", Inode: #12 ]│ ◄── Match!   │
│      └──────────────────────────────────────┘              │
└─────────────────────────────────────────────────────────────┘
          │
          ▼
┌──────────────────────────────────────────┐
│  4. Inode Table (Disk Metadata)          │
│  Inode #12                               │
│  - Permissions: -rwxr-xr-x               │
│  - Location Block: [1001, 1002, 1005]    │
└──────────────────────────────────────────┘
```

**다이어그램 해설**
1.  **해싱(Hashing)**: 파일명 "kernel.img"가 해시 함수에 입력되어 고유한 인덱스 값 '5'를 반환받는다. 이 과정은 O(1) 시간 복잡도를 가진다.
2.  **버킷 접근(Access)**: 계산된 인덱스 5번 버킷으로 즉시 이동한다. 버킷은 연결 리스트의 헤드(Head) 포인터를 가지고 있다.
3.  **충돌 해결(Collision Handling)**: 충돌로 인해 여러 엔트리가 연결된 상태이다. 시스템은 리스트를 순회하며 문자열 비교(String Comparison)를 수행하여 "kernel.img"와 일치하는 Inode #12를 찾는다.
4.  **메타데이터 획득**: 최종적으로 찾아낸 Inode 포인터를 통해 디스크 상의 실제 데이터 블록 위치를 알아낸다.

**핵심 알고리즘 및 수식**
해시 테이블의 성능은 **부하 계수(Load Factor, $\alpha$)**에 비례하여 충돌 확률이 높아진다.
$$ \alpha = \frac{n}{k} $$
*   $n$: 저장된 항목(Key-Value 쌍)의 수
*   $k$: 해시 테이블의 버킷(Bucket) 총 개수

체이닝(Chaining) 방식을 사용할 경우, 검색 성공 시 평균 비교 횟수(기대값) $E$는 다음과 같다.
$$ E \approx 1 + \frac{\alpha}{2} $$
따라서 성능을 O(1)에 가깝게 유지하기 위해서는 $\alpha$를 1 미만으로 관리하거나, 동적 해싱(Dynamic Hashing)을 통해 $k$를 확장(Rehashing)하는 전략이 필수적이다.

**📢 섹션 요약 비유**: 해시 테이블의 작동은 **"고속도로 하이패스 시스템"**과 유사합니다. 차량(데이터)이 요금소에 도착하여 일일이 정차하고 계산하는 대신(선형 탐색), 차량 번호(키)를 미리 인식하여 미리 지정된 차로(버킷)로 빠르게 유도합니다. 만약 지정된 차로가 막혀있다면(충돌), 바로 옆 차로로 우회하는(체이닝) 방식으로 원활한 흐름을 보장합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

디렉터리 구현 방식을 기술적, 정량적 관점에서 비교하고, 타 시스템 영역과의 융합성을 분석한다.

**심층 기술 비교 분석표**

| 비교 항목 | 선형 리스트 (Linear List) | 해시 테이블 (Hash Table) | B-Tree (Balanced Tree) |
|:---|:---|:---|:---|
| **검색 복잡도 (Time Complexity)** | $O(N)$ (최악) | **$O(1)$ (평균) / $O(N)$ (최악)** | $O(\log N)$ |
| **삽입/삭제 복잡도** | $O(1)$ (위치 알 시) / $O(N)$ (탐색 포함) | **$O(1)$** | $O(\log N)$ (노드 분할/합병 포함) |
| **공간 효율성** | 낮음 (포인터 오버헤드) | **중간** (버킷 낭비 가능성) | 높음 (포인터 다수 사용) |
| **순차적 접근 (Sorted Access)** | 용이 (삽입 순) | **불가능 (Unordered)** | **용이 (In-order Traversal)** |
| **구현 난이도** | 매우 낮음 | 높음 (해시 함수/충돌 처리) | 매우 높음 (회전 로직) |
| **주요 적용 분야** | 소규모 임베디드 시스템 | **OS 디렉터리, 캐시 시스템** | 데이터베이스 인덱스, 파일 시스템(Ext4 HTree) |

**과목 융합 관점**
1.  **운영체제 (OS) 메모리 관리**: 해시 테이블은 디렉터리 관리뿐만 아니라, **페이지 테이블(Page Table)**을 위한 **TLB (Translation Lookaside Buffer)** 구현에도 핵심적으로 사용된다. 가상 주소(Virtual Address)를 물리 주소(Physical Address)로 매핑할 때, 선형 검색 대신 해싱을 사용하여 Context Switching 오버헤드를 줄인다.
2.  **네트워크 (Switching)**: 고성능 라우터나 L3 스위치의 라우팅 테이블(Routing Table) 검색에 활용된다. 수만 개의 경로 정보에서 목적지 IP를 찾을 때 해시 테이블을 사용하여 패킷 처리량(PPS: Packet Per Second)을 극대화한다.

**정량적 성능 메트릭스 (Latency 비교)**
*   **Linear List**: 파일 10,000개 존재 시, 평균 5,000번의 문자열 비교 필요. 약 **5ms** 소요.
*   **Hash Table ($\alpha \approx 0.75$)**: 1번의 해시 연산 + 약 1.3번의 비교. 약 **0.005ms** 소요.
*   **성능 향상률**: 약 **1,000배** 이상의 성능 차이를 보이며, 데이터 규모가 커질수록 효율은 기하급수적으로 증가한다.

**📢 섹션 요약 비유**: 해시 테이블은 **"비상 연락망(Speed Dial)"**과 같습니다. 전화번호부를 쭉 넘겨가며 친구를 찾는 것(리스트)보다, 이미 친구 번호를 '즐겨찾기'에 등록해두고(해시) 단축키로 바로 전화를 거는 것이 훨씬 빠르듯이, 데이터가 많을수록 해시 테이블의 효율은 압도적입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

해시 테이블 디렉터리 도입 시 발생할 수 있는 문제 상황과 이에 대한 기술사적 의사결정 프로세스를 다룬다.

**실무 시나리오 및 의사결정 과정**
1.  **문제 상황 - 대용량 로그 파일 병목**: 매알 5만 개 이상의 로그 파일이 생성되는 금융 서버에서, 파일 생성 시 `open()` 시스템 콜 내부의 디렉터리 검색(중복 파일명 확인) 과정에서 지연이 발생하여 전체 처리량(Throughput)이 저하됨.
2.  **의사결정**: 선형 리스트 기반의 디렉터리 구조를 포기하고, 확장형 해시 테이블(Extendible Hashing) 기반의 파일 시스템(예: Linux xfs, ext4의 HTree)으로의 전환을 결정.
3.  **기술적 고려사항**:
    *   **해시 함수 선택**: 파일명의 분포가 고르지 않을 경우(예: `log_001.txt`, `log_002.txt`)를 대비해 비트 믹싱이 강력한 `MurmurHash` 또는 `xxHash` 사용을 검토.
    *   **메모리 오버헤드**: 해시 테이블 크기에 따른 메모리 점유율을 모니터링하고, 동적 리사이징(Rehashing) 시 발생하는 스파이크(Spike) 현상을 완화하는 이중 해싱(Double Hashing) 전략 수립.

**도입 체크리스트**
-   [ ] **해시 함수 품질**: 입력 데이터의 사소한 변경이 해시 값을 크게 변화시키는 **Avalanche 특성**을 만족하는가?
-   [ ] **충돌 해결책**: 체이닝(Chaining) 방식 사용 시, 메모리 파편화를 방지하기 위해 **Slab Allocator** 등과 연계했는가?
-   [ ] **DoS 방어**: 악의적인 사용자가 의도적으로 충돌을 유발하여 O(N) 성능을 강요하는 **Hash DoS 공격**에 대비하여, 비대칭형 암호화 해시 함수
