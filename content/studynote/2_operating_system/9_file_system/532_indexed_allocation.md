# [532. 인덱스 할당 (Indexed Allocation)]

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파일 시스템에서 데이터 블록들의 물리적 위치와 무관하게, 파일별로 전담하는 **인덱스 블록 (Index Block)**을 두어 포인터 집합을 관리하는 직접-간접 매핑 방식.
> 2. **가치**: 연속 할당(Continuous Allocation)의 **외부 단편화(External Fragmentation)**를 근본적으로 해소하고, 연결 할당(Linked Allocation)의 순차 접근(Sequential Access) 한계를 극복하여 **O(1)** 수준의 **임의 접근(Random Access)** 성능을 제공함.
> 3. **융합**: 이 구조는 현대 **OS (Operating System)**인 UNIX/Linux의 **inode (Index Node)** 기반 파일 시스템(ext4, xfs 등)의 이론적 토대이며, **DBMS (Database Management System)**의 인덱스 페이지 관리 기법과도 맞닿아 있음.

---

### Ⅰ. 개요 (Context & Background)

**1. 개념 및 정의**
**인덱스 할당(Indexed Allocation)**은 디스크 상의 비연속적인 공간에 흩어진 데이터 블록들을 효율적으로 관리하기 위해, 각 파일마다 하나의 전용 블록(인덱스 블록)을 할당하고, 여기에 해당 파일의 모든 데이터 블록 주소를 포인터 배열 형태로 저장하는 방식이다. 파일 시스템의 디렉터리 엔트리에는 데이터 블록 주소 대신 이 **인덱스 블록의 주소**가 저장된다.

**2. 등장 배경 및 철학**
초기 파일 시스템은 **연속 할당(Continuous Allocation)** 방식을 사용하여 속도는 빠르르나 공간 관리에 어려움을 겪었다. 이를 개선하고자 **연결 할당(Linked Allocation)**이 도입되었으나, 파일의 중간 데이터를 읽기 위해 모든 이전 블록을 순회해야 하는 **순차 접근(Sequential Access)**의 비효율성과 포인터가 저장된 블록 손상 시 복구가 불가능하다는 취약점이 있었다.
인덱스 할당은 이러한 trade-off를 해결하기 위해 **"데이터의 위치는 자유롭게, 접근 경로는 직관적으로"**라는 철학으로 등장했다. 이는 곧 메모리 관리의 **페이지 테이블(Page Table)** 개념과 유사한 논리-물리 주소 매핑을 디스크 관리에 도입하는 것으로, **OS (Operating System)**의 설계 패러다임을 파일 시스템에 확장한 결정체이다.

**3. 비유 및 구조적 이해**
인덱스 블록은 파일의 '위치 안내서' 역할을 수행한다. 디스크라는 거대한 도시에 파일의 집(데이터 블록)들이 여기저기 흩어져 지어져 있다고 가정할 때, 인덱스 블록은 각 집의 주소가 적힌 **'마스터 키 보관함'**과 같다. 우리는 어느 집에 살든 상관없이, 보관함만 열면 모든 방의 위치를 즉시 알 수 있다.

```text
+-----------------------------------------------------------------------+
| [ Concept: Logical Independence ]                                     |
|                                                                       |
|  File "A.txt"                                                          |
|  ┌───────────────────┐                                                |
|  │ Directory Entry   │                                                |
|  │ ─────────────────  │                                                |
|  │ Location: Ptr to ────┐                                             |
|  │      [Index Block]    │                                             |
|  └───────────────────┘   │                                             |
|                         │                                              |
|                         ▼                                              |
|                  +--------------+      +-----------------------------+ |
|                  | Index Block  |      |    Physical Disk Space       | |
|                  +--------------+      | (Non-contiguous & Scattered) | |
|                  | [0] -> 204  |─────▶│ ┌───┐ ┌───┐ ┌───┐ ┌───┐     | |
|                  | [1] -> 901  |─────▶│ │204│ │901│ │55  │ │...│     | |
|                  | [2] -> 55   |─────▶│ └───┘ └───┘ └───┘ └───┘     | |
|                  | [...]        │      └-----------------------------+ |
|                  +--------------+                                       |
+-----------------------------------------------------------------------+
```
*<그림 1. 인덱스 할당의 기본 개념도: 논리적 순서(0, 1, 2)와 물리적 위치(204, 901, 55)의 완전한 분리를 보여준다>*

**해설**: 위 다이어그램은 파일 시스템이 데이터를 관리하는 논리적 관점과 물리적 저장소의 분리를 시각화한 것이다. 디렉터리 엔트리는 더 이상 데이터의 헤드를 가리키지 않고, '주소록'인 인덱스 블록을 가리킨다. 이를 통해 물리적인 데이터 블록(204, 901, 55)이 디스크 곳곳에 흩어져 있더라도(Scattered), 파일 시스템은 논리적인 순서(0, 1, 2)를 유지하며 데이터를 재구성할 수 있다. 이는 메모리 관리 기법에서 페이지 테이블이 물리 프레임의 분리를 추상화하는 것과 동일한 추상화 메커니즘이다.

📢 **섹션 요약 비유**: 인덱스 할당은 책의 **'찾아보기(Index)'**와 같습니다. 책의 내용(데이터)이 페이지 순서대로 연속되어 있지 않더라도, 찾아보기 페이지(인덱스 블록)에 모든 주제의 위치가 정리되어 있으므로, 독자는 책을 처음부터 필기하지 않고도 원하는 내용을 즉시 찾아갈 수 있습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**1. 구성 요소 상세 분석**
인덱스 할당 시스템은 크게 메타데이터 영역과 데이터 영역으로 나뉘며, 그 핵심은 **인덱스 블록**의 설계에 있다.

| 요소명 (Component) | 역할 (Role) | 내부 동작 및 프로토콜 | 비유 (Analogy) |
|:---:|:---|:---|:---|
| **Directory Entry** | 파일 식별 및 진입점 | 파일명, 소유자, 권한 등과 **인덱스 블록 번호** 저장 | 회사의 '사원 명부' (사무실 위치 기재) |
| **Index Block** | 주소 매핑 테이블 | **Pointer Array**로 구성. $i$번째 슬롯은 $i$번째 논리 블록의 **Physical Block Number(PBN)** 저장 | '열쇠 보관함' (모든 방의 키/주소 보유) |
| **Data Block** | 실제 데이터 저장 | 사용자 파일의 실제 내용(Contents) 저장. 디스크 어디든 위치 가능 | '각각의 사무실' (건물 위치 무관) |
| **Free Space Mgmt** | 공간 관리 | Bit Map 또는 Linked List 방식 사용하여 가용 블록 관리 | '부동산 중개 업체' (빈 집 관리) |

**2. 동작 메커니즘 및 알고리즘**
파일 시스템이 특정 블록에 접근하는 과정은 **메모리 관리의 페이징(Paging)** 기법과 수학적으로 동일하다.
*   **논리 주소(Logical Address)**: $L$ (파일 내에서의 offset)
*   **변환 과정**:
    1.  $Block\_Size$를 기준으로 $Logical\_Block\_Number (LBN)$과 $Offset$을 계산.
    2.  디렉터리에서 해당 파일의 $Index\_Block\_Addr$을 획득.
    3.  디스크 I/O: $Index\_Block\_Addr$을 로드.
    4.  인덱스 블록의 $LBN$번째 엔트리 값을 읽어 $Physical\_Block\_Number (PBN)$ 획득.
    5.  디스크 I/O: $PBN$에 접근하여 데이터 획득.

```text
+-----------------------------------------------------------------------+
| [ Architecture: Deep Dive Structure ]                                 |
|                                                                       |
|  User Request: Read File "data.bin", Offset=4096 (LBA=1)             |
|       │                                                              |
|       ▼                                                              |
|  [FILE SYSTEM LAYER]                                                 |
|  ┌───────────────────────────────────────────────────────────────┐   |
|  │ 1. Metadata Check                                             │   |
|  │    File: data.bin -> Points to Index Block #102               │   |
|  │                                                               │   |
|  │ 2. Calculation                                                │   |
|  │    Block Size = 4KB                                           │   |
|  │    LBN (Logical Block Number) = 4096 / 4096 = 1               │   |
|  │    Offset = 0                                                 │   |
|  └───────────────────────────────────────────────────────────────┘   |
|       │                                                              |
|       ▼                                                              |
|  [DISK I/O OPERATION]                                               |
|  ┌──────────────────────┐       ┌───────────────────────────────┐   |
|  │  Step A: Read Index  │       │  Step B: Read Data            │   |
|  │  Block (Addr: #102)  │       │  Block (Addr: 2048)           │   |
|  │                      │       │                               │   |
|  │  +------------------+│       │  +---------------------------+ │   |
|  │  | [0] -> 900       ││───────│──│ Data from LBN=1            │ │   |
|  │  | [1] -> 2048   <───┘       │  | (Target for Offset=4096)  │ │   |
|  │  | [2] -> 55       │        │  +---------------------------+ │   |
|  │  | [...]           │        └─────────────────────────────────┘   |
|  │  +------------------+│                                               |
|  └──────────────────────┘                                                |
|                                                                       |
|  * 포인터 크기가 4Byte(32bit)이고 블록이 4KB일 때,               |
|    하나의 인덱스 블록은 1024개의 포인터를 저장하여 4MB 파일을 커버.  |
+-----------------------------------------------------------------------+
```
*<그림 2. 인덱스 할당의 상세 동작 아키텍처: 논리적 오프셋에서 물리적 블록으로의 매핑 과정을 두 단계(Directory->Index->Data)로 나타낸다>*

**해설**: 파일 시스템은 사용자의 요청(Offset 4096)을 받으면, 이를 블록 단위(LBN 1)로 변환한다. 연결 할당 방식이라면 0번 블록을 거쳐 1번으로 가야 하지만, 인덱스 할당은 인덱스 블록(#102)을 직접 읽어(Figure 2 Step A), 해당 엔트리([1])가 가리키는 물리적 주소(2048)로 즉시 이동한다(Figure 2 Step B). 이 과정에서 추가적인 디스크 탐색(Seek)이 발생하지만, 순차적인 읽기 연산을 수행할 필요가 없어 **임의 접근(Random Access)**이 가능해진다.

**3. 핵심 코드 및 수식**
인덱스 할당의 공간 효율성은 포인터 크기($P$)와 블록 크기($B$)에 의해 결정된다. 하나의 인덱스 블록으로 관리 가능한 최대 파일 크기($MaxFile$)는 다음과 같다.

$$ MaxFile = \frac{B}{P} \times B $$

*   예시: $B=4096$ bytes, $P=4$ bytes
*   단일 인덱스 블록 엔트리 수 = $1024$개
*   최대 파일 크기 = $1024 \times 4KB = 4MB$

이때 문제는 파일 크기가 4MB를 초과할 때 발생한다. 이를 해결하기 위해 **Multi-level Index** 기법이 사용된다.

📢 **섹션 요약 비유**: 인덱스 할당은 **'지하철 노선도'**와 같습니다. 승객(데이터)은 역(블록)마다 흩어져 있지만, 노선도(인덱스 블록)만 보면 어디에 누가 있는지, 어디로 가야 할지 한눈에 알 수 있습니다. 역 간의 거리가 멀든(데이터가 떨어져 있든) 상관없이 연결선만 있으면 즉시 이동이 가능합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**1. 심층 기술 비교: 할당 기법 3대장**

| 구분 (Criteria) | 연속 할당 (Continuous) | 연결 할당 (Linked) | 인덱스 할당 (Indexed) |
|:---:|:---:|:---:|:---:|
| **접근 방식** | **Random Access** (직접) | **Sequential Access** (순차) | **Random Access** (간접) |
| **외부 단편화** | 발생함 (심각) | 없음 | 없음 (데이터 블록 간) |
| **내부 단편화** | 파일 끝부분에 발생 가능 | 없음 | 인덱스 블록 내 미사용 공간 발생 가능 |
| **접근 속도** | 가장 빠름 (Seek 최소화) | 느림 (순회 필요) | 빠름 (인덱스 1회 읽기 후 접근) |
| **신뢰성** | 중간 블록 손상 시 뒤쪽 손실 | 포인터 끊어지면 전부 손실 | 인덱스 블록 손상 시 전부 손상 (복제/미러링 필요) |
| **구현 복잡도** | 단순 | 단순 | 복잡 (블록 관리 로직 필요) |
| **주요 용도** | CD-ROM, WORM | 초기 FAT (FAT12/16) | 현대 OS (UNIX ext, NTFS MFT) |

**2. OS 및 네트워크 융합 관점**
*   **메모리 관리와의 시너지**: 인덱스 할당의 구조는 가상 메모리의 **페이지 테이블(Page Table)**과 거의 동일하다. 다만 페이지 테이블은 메모리(RAM) 상에 존재하여 MMU가 참조하지만, 인덱스 블록은 디스크(Disk) 상에 존재하여 OS가 참조한다는 차이가 있다.
*   **데이터베이스 인덱싱**: **DBMS (Database Management System)**의 B-Tree 인덱스 구조 또한 인덱스 할당의 확장된 형태로 볼 수 있다. 데이터 페이지가 디스크 곳곳에 흩어져 있어도, 루트 페이지에서 리프 페이지까지의 포인터를 따라가며 데이터를 빠르게 찾는 메커니즘은 인덱스 할당의 철학을 그대로 따르고 있다.
*   **네트워크 패킷 라우팅**: 패킷의 목적지가 연속적이지 않더라도, **라우팅 테이블(Routing Table)**이라는 '인덱스'를 통해 다음 홉(Next Hop)을 결정하는 방식
