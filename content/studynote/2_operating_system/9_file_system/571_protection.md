## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보호(Protection)는 OS 커널이 제공하는 시스템 자원 접근 통제 메커니즘이며, 보안(Security)은 이를 포함한 전체 시스템의 기밀성·무결성·가용성을 보장하는 포괄적 철학이다.
> 2. **가치**: 보호 메커니즘이 결여된 시스템은 프로세스 간 격리가 불가능하여任意 코드 실행에 의한 메모리 파괴가 1ms 내에 발생하며, 이는 클라우드 멀티테넌트 환경에서 치명적 데이터 유출로 이어진다.
> 3. **융합**: 최근 CPU의 가상화 확장(Intel VT-x/AMD-V)과 TrustZone, SGX 등 하드웨어 보안 enclave는 보호/보안 경계를 커널 레벨에서 하드웨어 루트 오브 트러스트로 확장하고 있다.

## Ⅰ. 개요 (Context & Background)

- **개념**: 보호(Protection)는 운영체제가 제공하는 **자원 접근 제어 메커니즘**으로서, 프로세스·파일·메모리·장치 등 시스템 객체(Object)에 대해 허용된 연산(Operation)만을 수행하도록 강제하는 기술적 장치다. 반면 보안(Security)은 이러한 보호 메커니즘을 포함하여, **기밀성(Confidentiality)·무결성(Integrity)·가용성(Availability)**의 CIA 삼각을 보장하는 더 포괄적인 목표다.

- **💡 비유**: 보호는 "집의 문 잠금 장치"다. 현관문, 방문, 창문마다 자물쇠(Protection Mechanism)를 설치하여 허락된 사람만이 특정 방에 들어갈 수 있게 한다. 보안은 "집안 전체의 안전 시스템"이다. 자물쇠뿐만 아니라 CCTV, 경비원, 화재 경보기, 재난 복구 계획까지 포함된 전략이다. 자물쇠가 없으면 도둑이 마음대로 드나들고, 보안 시스템이 없으면 자물쇠를 뚫는 강도를 막을 수 없다.

- **등장 배경**:
  1. **초기 일괄 처리 시스템(1950s-60s)의 신뢰 기반**: 초기 컴퓨터는 한 번에 하나의 프로그램만 실행했고, 사용자가 물리적으로 시스템을 독점했으므로 별도의 보호가 필요 없었다. 프로그래머의 실수가 시스템 전체를 충돌시킬 수 있었지만, 사용자 집단이 좁았기에 사회적 신뢰로 운영되었다.
  2. **시분할 시스템(TSS, 1960s)과 다중 프로그래밍의 등장**: CTSS, MULTICS 프로젝트에서 여러 사용자가 동일한 시스템을 동시에 사용하기 시작하면서, 프로세스 A가 프로세스 B의 메모리를 침범하거나 CPU 시간을 독점하는 문제가 발생했다. 이에 따라 **베이스/한계 레지스터(Base & Limit Register)** 를 도입하여 각 프로세스의 메모리 영역을 하드웨어적으로 격리하는 보호 메커니즘이 탄생했다.
  3. **네트워크와 클라우드 시대의 보안 요구(1990s-현재)**: 인터넷 보급과 함께 익명의 공격자가 원격에서 시스템을 침투할 수 있게 되었다. 단순한 접근 통제로는 부족하게 되었고, 암호화, 인증, 감사, 침입 탐지 등 계층적 보안(Security in Depth)이 필수가 되었다.

보호와 보안의 계층적 관계를 시각화하면 다음과 같다. 보호가 "자동차의 브레이크"라면, 보안은 "브레이크 + 에어백 + 차체 강성 + 운전자 보험"을 포함한 종합 안전 시스템이다.

```text
 ┌─────────────────────────────────────────────────────────────────┐
 │              보호(Protection) vs 보안(Security) 관계               │
 ├─────────────────────────────────────────────────────────────────┤
 │                                                                 │
 │  [보안 (Security) — 포괄적 목표]                                 │
 │  ┌─────────────────────────────────────────────────────────┐    │
 │  │  기밀성(C)  │  무결성(I)  │  가용성(A)                   │    │
 │  │  데이터 은폐  │  데이터 위변조 방지  │  서비스 지속성      │    │
 │  └─────────────────────────────────────────────────────────┘    │
 │                              │                                  │
 │                 ┌────────────┴────────────┐                     │
 │                 ▼                         ▼                     │
 │  [보호 (Protection) — 기술적 수단]     [기타 보안 통제]           │
 │  ┌─────────────────────────────┐   ┌──────────────────────┐    │
 │  │ • 접근 제어 행렬 / ACL        │   │ • 암호화             │    │
 │  │ • Capability / Ticket       │   │ • 인증(Auth)          │    │
 │  │ • 메모리 보호 (페이지 테이블)  │   │ • 감사(Auditing)     │    │
 │  │ • 파일 시스템 권한 (rwx)      │   │ • 백업/복구          │    │
 │  │ • 시스템 콜 인터페이스 검증  │   │ • 보안 정책           │    │
 │  └─────────────────────────────┘   └──────────────────────┘    │
 │                                                                 │
 │  보호는 "기술적 강제력(Technical Enforcement)"                 │
 │  보안은 "관리적·물리적·기술적 통제의 총체"                     │
 └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 보안(Security)은 CIA 트라이어드로 대표되는 최종 목표다. 이 목표를 달성하기 위한 수단으로서 보호(Protection)가 존재한다. 보호는 **"시스템이 스스로 강제하는 기술적 규칙"**으로서, ACL, 메모리 보호, 시스템 콜 검증 등 커널 레벨에서 자동으로 위반을 차단한다. 반면 암호화, 인증, 감사 등은 **"사람이나 프로세스가 능동적으로 수행하는 보안 활동"**으로서, 보호 메커니즘을 보완한다. 예를 들어 암호화는 데이터가 유출되더라도 내용을 읽을 수 없게 하지만, 데이터 자체의 접근을 막지는 못한다. 이를 막는 것은 보호 메커니즘(파일 권한, 메모리 격리)이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **보호 도메인(Protection Domain)** | 프로세스의 접근 권한 집합을 정의하는 경계 | 자원 식별자 + 권한 비트맵으로 표현, 도메인 간 전이 시 커널 모드 진입 | Ring 아키텍처(Ring 0-3), POSIX Capabilities | 여권에 찍힌 입국 허가국 리스트 |
| **참조 모니터(Reference Monitor)** | 모든 접근 요청을 중계하여 허가 여부를 판단하는 추상 개체 | 시스템 콜 후킹, 접근 제어 리스트 검사, 감사 로그 기록 | Kernel LSM(Linux Security Modules) 훅 | 공항 보안 검색대 (모든 여객 검사) |
| **감사 추적(Audit Trail)** | 보안 위반 사건의 재구성을 위한 기록 | Syslog, Auditd, EDR이 모든 접근 시도를 시간 stamped 기록 | Linux Audit Subsystem, Windows Event Log | CCTV 블랙박스 영상 |
| **보안 커널(Security Kernel)** | 참조 모니터를 구현하는 하드웨어/소프트웨어 | TCB(Trusted Computing Base)의 일부, 검증 가능한 최소 크기 유지 | seL4 microkernel (Formal Verification) | 금고의 견고한 벽면 |
| **인증 주체(Principal)** | 권한을 부여받는 엔티티(사용자, 프로세스, 디바이스) | UID/GID, SID, JWT 토큰으로 식별, 신원 확인 후 보호 도메인 할당 | PAM(Pluggable Authentication Modules) | 여권 소지자 본인 |

### 보호 메커니즘의 계층별 구현

현대 OS는 하드웨어, 커널, 파일시스템, 네트워크 계층에 걸쳐 다단계 보호 메커니즘을 배치하여 Defense in Depth를 실현한다. 각 계층이 실패하더라도 다음 계층이 공격을 차단하는 구조다.

```text
 ┌───────────────────────────────────────────────────────────────────┐
 │            다층적 보호 메커니즘 (Defense in Depth)                 │
 ├───────────────────────────────────────────────────────────────────┤
 │                                                                   │
 │  [사용자 공간 응용]                                               │
 │  ┌──────────────────────────────────────────────────────────┐     │
 │  │  애플리케이션 → 자체 권한 검사, 사용자 인증              │     │
 │  └──────────────────────────────────────────────────────────┘     │
 │                              │ 시스템 콜                          │
 │                              ▼                                   │
 │  [커널 공간 - OS 보호 레이어]                                     │
 │  ┌──────────────────────────────────────────────────────────┐     │
 │  │  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐  │     │
 │  │  │   VFS 계층     │  │  프로세스 관리자│  │  네트워크 스택 │  │     │
 │  │  │   ACL 검사    │  │  PID/UID 검사 │  │  방화벽/SELinux│  │     │
 │  │  │   rwx, DAC    │  │  Signal 제어  │  │  포트 필터링  │  │     │
 │  │  └───────────────┘  └───────────────┘  └───────────────┘  │     │
 │  │                            │                               │     │
 │  │                   ┌────────┴────────┐                      │     │
 │  │                   ▼                 ▼                      │     │
 │  │           ┌─────────────┐   ┌─────────────┐                │     │
 │  │           │ LSM 훅      │   │  Capability │                │     │
 │  │           │ (SELinux,   │   │  체크       │                │     │
 │  │           │  AppArmor)  │   │             │                │     │
 │  │           └─────────────┘   └─────────────┘                │     │
 │  └──────────────────────────────────────────────────────────┘     │
 │                              │                                   │
 │                              ▼                                   │
 │  [하드웨어 보호 레이어]                                           │
 │  ┌──────────────────────────────────────────────────────────┐     │
 │  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐       │     │
 │  │  │ MMU/페이지   │  │  CPU Ring   │  │  SMP Lock   │       │     │
 │  │  │  테이블      │  │  (Ring 0-3) │  │  (스핀락)   │       │     │
 │  │  │  메모리 격리 │  │  명령어 제한 │  │  원자적 연산 │       │     │
 │  │  └─────────────┘  └─────────────┘  └─────────────┘       │     │
 │  │                                                           │     │
 │  │  • Supervisor/User 모드 분리                               │     │
 │  │  • I/O 포트 접근 제한 (IN/OUT 명령어 보호)                  │     │
 │  │  • 인터럽트 게이트를 통한 커널 진입                         │     │
 │  └──────────────────────────────────────────────────────────┘     │
 │                                                                   │
 │  각 계층은 "Fail Safe" — 상위 계층이 뚫려도 하위가 방어           │
 └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 최상위 응용 계층에서부터 하드웨어까지 계층별 보호 메커니즘이 배치된다. 예를 들어 웹 서버가 `/etc/shadow` 파일을 읽으려 시도하면:
1. **응용 계층**: 웹 서버 자체가 "권한 없음"을 인지하고 요청을 거부할 수 있다(이상적이지만 신뢰할 수 없음)
2. **VFS 계층**: 커널의 VFS(가상 파일 시스템)가 시스템 콜(`open()`)을 가로채어, 파일의 inode에 저장된 ACL(Web 서버 프로세스의 UID=www-data, 파일 UID=0(root))를 검사하고 **EPERM**을 반환한다
3. **LSM 훅**: SELinux/AppArmor가 추가로 보안 컨텍스트(`system_u:object_r:shadow_t:s0`)를 검사하여 레이블 기반 정책을 적용한다
4. **하드웨어**: 만약 공격자가 커널 모드로 스케일업(Ring 3→0)을 시도하면 CPU가 **General Protection Fault**를 발생시켜 프로세스를 즉시 종료한다

### 심층 동작 원리: 참조 모니터(Reference Monitor)

1972년 James P. Anderson이 제안한 참조 모니터는 보호 메커니즘의 이론적 기반으로, **① 항상 활성화(Tamper-proof)** **② 수시 검증(Always invoked)** **③ 충분히 작음(Small enough to verify)** 의 세 가지 속성을 만족해야 한다. 리눅스 커널의 LSM(Linux Security Modules) 훅은 이를 구현한 현대적 사례다.

```text
 ┌───────────────────────────────────────────────────────────────────┐
 │          참조 모니터(Reference Monitor) 구현 - LSM 훅             │
 ├───────────────────────────────────────────────────────────────────┤
 │                                                                   │
 │  사용자 공간 프로세스                                             │
 │  ┌──────────────────────────────┐                                 │
 │  │  open("/etc/passwd", O_RDWR) │───── 시스템 콜 요청 ────┐       │
 │  └──────────────────────────────┘                             │    │
 │                                                               │    │
 │  ┌─────────────────────────────────────────────────────────┐  │    │
 │  │              커널 공간 (VFS 계층)                         │  │    │
 │  │                                                          │  │    │
 │  │   ① SYSCALL_DEFINE3(open, ...) — 시스템 콜 엔트리        │  │    │
 │  │          │                                               │  │    │
 │  │          ▼                                               │  │    │
 │  │   ② do_sys_open(AT_FDCWD, filename, flags, mode)         │  │    │
 │  │          │                                               │  │    │
 │  │          ▼                                               │  │    │
 │  │   ③ vfs_open()                                          │  │    │
 │  │          │                                               │  │    │
 │  │          ├──▶ inode_lock(inode)                         │  │    │
 │  │          │                                               │  │    │
 │  │          ├──▶ security_inode_permission() ──── LSM 훅! ─┼──┘    │
 │  │          │                              │                   │    │
 │  │          │                              ▼                   │    │
 │  │          │              ┌─────────────────────────┐          │    │
 │  │          │              │  SELinux / AppArmor      │          │    │
 │  │          │              │  ├─ 보안 컨텍스트 검사   │          │    │
 │  │          │              │  ├─ AVC(Access Vector)   │          │    │
 │  │          │              │  └─ 결정: PERMIT/DENY    │          │    │
 │  │          │              └─────────────────────────┘          │    │
 │  │          │                              │                   │    │    │
 │  │          │                              ▼                   │    │    │
 │  │          ├──▶ may_open() — DAC(rwx) 검사                     │    │    │
 │  │          │                                                    │    │    │
 │  │          └──▶ dentry_open() — 파일 객체 생성                 │    │    │
 │  │                                                          │  │    │
 │  └─────────────────────────────────────────────────────────┘  │    │
 │                                                               │    │
 │  ┌─────────────────────────────────────────────────────────┐  │    │
 │  │               감사(Audit) 기록                           │  │    │
 │  │   type=AVC msg=audit(1678XXXXXX.XXX:XXX):               │  │    │
 │  │     avc: denied { read } for pid=1234 comm="httpd"       │  │    │
 │  │     path="/etc/passwd" scontext=... tcontext=...         │  │    │
 │  └─────────────────────────────────────────────────────────┘  │    │
 │                                                               │    │
 │  반환값: -EACCES (Permission denied) ◀─────────────────────────┘    │
 │                                                                   │
 │  LSM 훅는 "모든" 객체 접근 경로에 삽입되어 Always Invoked 보장    │
 └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 사용자 프로세스가 `open()` 시스템 콜을 호출하면 커널의 VFS 계층으로 진입한다. 기존 리눅스 커널은 `inode->i_op->permission`으로 DAC(Discretionary Access Control, 즉 rwx 권한 비트)만 검사했지만, LSM 프레임워크는 **security_inode_permission()** 훅을 호출하여 SELinux, AppArmor, Smack 등 모듈에 보안 정책 검사 기회를 부여한다. 이 훅은 VFS가 파일 객체를 반환하기 **반드시 거쳐야 하는 경로**에 배치되어 있어 **Always Invoked** 속성을 만족한다. 또한 LSM 모듈 자체는 커널 모듈로 로드되지만, 핵심 훅은 커널 코어에 내장되어 있어 TAMPER-PROOF하다. 마지막으로 보안 정책 검증 로직은 비교적 단순(주체/객체 레이블 비교, 접근 벡터 캐시 참조)하여 **Small enough to verify** 속성에도 부합한다.

① 프로세스가 시스템 콜 호출 → ② 커널 모드 진입, VFS 계층 도달 → ③ LSM 훅이 보안 모듈에 정책 검사 위임 → ④ SELinux/AppArmor가 AVC(Access Vector Cache)에서 빠르게 허가/거부 결정 → ⑤ DAC(rwx) 2차 검사 → ⑥ 감사 로그 기록 → ⑦ 결과 반환(성공 시 파일 디스크립터, 실패 시 -EACCES)

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 보호 vs 보안 속성 대조

| 속성 | 보호 (Protection) | 보안 (Security) |
|:---|:---|:---|
| **관심사** | 시스템 자원의 무결성과 격리 | 기밀성, 무결성, 가용성 전체 |
| **구현 주체** | 운영체제 커널, 하드웨어 | OS + 네트워크 + 물리적 통제 + 정책 |
| **강제 방식** | 하드웨어적(MMU), 커널적(시스템 콜) 자동 차단 | 정책, 법규, 교육, 기술적 장치 혼합 |
| **위반 시 결과** | Segmentation Fault, EPERM | 데이터 유출, 서비스 마비, 재정적 손실 |
| **측정 가능성** | 이진적(허용/거부)로 명확히 측정 가능 | 정량적 위험(Risk) = Threat × Vulnerability × Impact |
| **예시** | 메모리 보호, 파일 권한, UID/GID | 방화벽, 암호화, 침입 탐지, 백업 |

보호는 "어떤 자원에 누가 무엇을 할 수 있는가(Who can do What on Which object)"를 기술적으로 제한하는 반면, 보안은 "조직의 자산을 어떻게 보전할 것인가"라는 경영적·기술적 질문에 답한다. 보호가 없으면 보안은 불가능하지만, 보호만 있어서는 보안이 불가능하다.

### 비교 2: DAC vs MAC

| 비교 항목 | DAC (임의적 접근 제어) | MAC (강제적 접근 제어) |
|:---|:---|:---|
| **권한 위임** | 소유자가 자유롭게 권한 위임 가능 | 시스템 보안 관리자만 정책 설정 |
| **대표 예시** | 리눅스 rwx, ACL | SELinux, AppArmor |
| **정책 모델** | 자유(Freedom) 위주 | 보안(Security) 위주 |
| **결정 기준** | 사용자 신원(UID/GID) + 자원 소유권 | 보안 레이블(Security Label) + 분류(Classification) |
| **트로이 목마 취약성** | 소유자가 속으면 전파됨 | 정책에 의해 강제 차단 가능 |
| **주 용도** | 일반 업무용 시스템 | 군사, 금융, 멀티테넌트 클라우드 |

DAC는 사용자가 자신이 소유한 파일의 권한을 자유롭게 변경할 수 있다. `chmod 777`으로 모든 사용자에게 쓰기 권한을 부여할 수 있는데, 이는 편리하지만 트로이 목마에 취약하다. 사용자가 악의적인 프로그램을 실행하면 그 프로그램은 사용자의 권한을 모두 물려받아 사용자가 접근 가능한 모든 자원을 침해할 수 있다. MAC는 이를 방지하기 위해 소유자의 의지와 무관하게 시스템 수준에서 정책을 강제한다. SELinux는 `httpd_t` 도메인의 프로세스가 `shadow_t` 유형의 파일에 접근하는 것을 소유자가 root라 하더라도 원천 봉쇄한다.

DAC와 MAC의 정책 결정 흐름을 비교하면 다음과 같다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                  DAC vs MAC 정책 결정 비교                           │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [DAC: Discretionary Access Control]                                 │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  요청: UID=1000(user)가 /home/user/data.txt에 쓰기 시도         │  │
 │  │                                                                 │  │
 │  │  ① 파일 inode 로드                                               │  │
 │  │  ② inode->i_uid == 1000?  ───Yes──▶  소유자 확인                │  │
 │  │  ③ inode->i_mode & 0200 (write bit) 체크  ───Yes──▶  허용!      │  │
 │  │                                                                 │  │
 │  │  ✅ user가 chmod 000으로 권한을 제거하면 root도 접근 불가         │  │
 │  │     (root는 DAC 우회 가능하지만 일반 사용자는 완전히 차단)         │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [MAC: Mandatory Access Control]                                     │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  요청: SELinux 도메인=httpd_t가 /var/www/html/index.html 읽기   │  │
 │  │                                                                 │  │
 │  │  ① 주체 컨텍스트: system_u:system_r:httpd_t:s0                 │  │
 │  │ ② 객체 컨텍스트: system_u:object_r:httpd_sys_content_t:s0       │  │
 │  │                                                                 │  │
 │  │  ③ AVC(Access Vector Cache) 검색:                               │  │
 │  │     Key = {httpd_t, httpd_sys_content_t, file, read}            │  │
 │  │     Value = ALLOW / DENY (정책 데이터베이스에서 사전 결정)        │  │
 │  │                                                                 │  │
 │  │  ④ 결과: DENY인 경우 root 권한으로 실행해도 차단!               │  │
 │  │     audit.log: "avc: denied { read }..."                        │  │
 │  │                                                                 │  │
 │  │  ✅ 정책은 보안 관리자만 설정, 소유자는 변경 불가                 │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  DAC는 "소유자의 재량"에, MAC는 "시스템 강제"에 의존                 │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** DAC에서는 파일 소유자가 `chmod` 명령으로 권한을 변경할 수 있다. 사용자가 실수로 `chmod 777 ~/private.txt`를 실행하면, 모든 사용자가 해당 파일을 읽을 수 있게 되어 기밀성이 파괴된다. 반면 MAC에서는 파일에 보안 레이블(SELinux에서는 `user:role:type:level`)이 부착되며, 이 레이블 간의 허용/거부 관계는 보안 관리자가 설정한 정책 파일(`/etc/selinux/policy/policy.*`)에 의해 결정된다. `httpd_t` 타입의 프로세스가 `shadow_t` 타입의 파일에 접근하는 것은 정책에 명시적으로 허용되지 않는 한 불가능하며, 이는 웹 서버 프로세스가 하이재킹되더라도 `/etc/shadow`가 유출되는 것을 방지한다.

### 과목 융합 관점

- **컴퓨터구조**: MMU의 페이지 테이블 엔트리에 User/Supervisor 비트와 Read/Write 비트를 두어, 프로세스가 자신의 가상 주소 공간 밖이나 커널 영역(0xC0000000+)에 접근하려 하면 **Page Fault**를 발생시킨다. 이것이 가장 원초적인 보호 메커니즘이다.
- **네트워크**: 방화벽, IDS/IPS는 네트워크 패킷을 검사하여 공격 트래픽을 필터링하지만, OS 자체의 보호 메커니즘이 뚫리면 방화벽은 무용지물이 된다. 예를 들어 RCE(Remote Code Execution) 취약점으로 원격 공격자가 root 권한을 획득하면, 방화벽은 내부에서 나가는 공격 트래픽(exfiltration)을 막지 못한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 멀티테넌트 클라우드에서의 컨테이너 격리 실패**: 공유 커널 환경의 Docker 컨테이너가 루트 권한으로 실행되어, 호스트의 `/var/lib/docker` 디렉터리에 접근하여 다른 테넌트의 이미지를 탈취하는 사고가 발생했다. 원인은 DAC만으로는 컨테이너 간 격리가 불가능했기 때문이다. 해결책은 `--security-opt selinux:enabled`와 `--security-opt apparmor=docker-default`를 활성화하여 MAC 정책으로 컨테이너를 제한하고, `user namespaces`를 사용하여 컨테이너 내부 root를 호스트에서는 비특권 사용자로 매핑하는 것이다.

2. **시나리오 — 금융권 시스템의 제3자 공급업체 보안 사고**: 신용평가사의 제휴 업체 개발자가 실수로 운영 서버의 디버그 포트를 인터넷에 노출하여, 해커가 이를 통해 DB에 접근하는 사고가 발생했다. 기술적 보호(방화벽, 침입 탐지)는 존재했으나, 보안 정책(포트 관리 규정), 보안 교육, 정기 감사 등이 누락되어 있었다. 이는 **"보호는 있었지만 보안이 부재"**한 전형적 사례다. 해결책은 ① Nginx 등 리버스 프록시 뒤에 애플리케이션을 숨기고 ② Bastion Host를 통한 관리자 접속만 허용하며 ③ 모든 관리 작업을 Auditd로 기록하고 ④ 정기적 보안 교육과 포트 스캔을 수행하는 것이다.

3. **시나리오 — IoT 디바이스의 하드웨어 보안 enclave 활용**: 의료용 IoT 기기가 환자의 민감한 데이터를 수집하는데, 소프트웨어만으로는 해커가 펌웨어를 수정하여 데이터를 유출하는 것을 막을 수 없다. ARM TrustZone 기반의 TEE(Trusted Execution Environment)를 활용하여, 데이터 수집 → 암호화 → 전송 파이프라인을 Secure World에서만 실행하도록 설계했다. 일반 OS(Normal World)가 감염되더라도 TEE 내부의 코드와 데이터는 하드웨어적으로 격리된다.

### 도입 체크리스트
- **기술적**: 모든 중요 자원에 대해 DAC(rwx)와 MAC(SELinux/AppArmor) 정책이 중복 적용되었는가? 감사 로그(Auditd)가 정상적으로 기록되고 있으며, 로그를 원격 syslog 서버로 전송하여 Tamper-proof하게 보관하는가?
- **운영·보안적**: 보안 정책(Security Policy) 문서가 존재하며, 정기적으로(최소 1년 1회) 검토되고 있는가? 사고 발생 시 대응 절차(Incident Response)가 문서화되고 훈련되어 있는가?

### 안티패턴
- **과도한 보안 정책(Policy Bloat)**: SELinux를 활성화하고 디폴트 정책(`targeted`) 대신 `strict` 정책을 사용하면, 허용되지 않은 모든 동작이 차단되어 시스템이 정상적으로 동작하지 않는다. 초기에는 `permissive` 모드에서 로그만 수집하고, 차단할 규칙을 확인한 뒤 `enforcing` 모드로 전환해야 한다.
- **감사 로그의 무단 삭제**: `ausearch`로 특정 사용자의 접근 기록을 삭제하려는 공격자를 막기 위해, 감사 로그는 `/var/log/audit/audit.log`에 쓰이는 즉시 원격 서버로 전송하거나 WORM(Write Once Read Many) 스토리지에 기록해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 | 최적화 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | DAC만 사용 | SELinux MAC + DAC 추가 | **제로데이 공격 벡터 90%+ 감소** |
| **정량** | 감사 없음 | Auditd 전면 도입 | **보안 사고 포렌식 시간 80% 단축** |
| **정성** | 보안 사고 시 원인 규명 불가 | 감사 로그로 재구성 가능 | **규정 준거(Compliance) 달성** |

### 미래 전망
- **eBPF(extended Berkeley Packet Filter)**: 커널 모듈 없이 안전하게 커널 훅을 확장 가능하여, 사용자 공간에서 보안 정책을 동적으로 로드할 수 있다. 기존 LSM 훅은 커널 모듈로 작성해야 했으므로 개발이 어려웠지만, eBPF LSM 프로그램은 JIT 컴파일되어 실행되며 검증기(Verifier)가 무한 루프나 비안전한 메모리 접근을 사전에 차단한다.
- **Confidential Computing**: AMD SEV-SNP, Intel TDX 등 가상화 기반의 암호화된 VM 메모리(Encrypted VM)이 출시되어, 클라우드 제공자조차도 가상머신 메모리를 염탐할 수 없게 되었다. 이것은 보호 경계를 가상머신 경계로 확장한 것으로, 하이퍼바이저가 손상되더라도 게스트 OS의 메모리는 보호된다.

### 참고 표준
- **TCSEC (Trusted Computer System Evaluation Criteria, 1985)**: "Orange Book"으로도 불리며, 보호 메커니즘의 엄격성에 따라 D~A1 등급을 부여하는 최초의 보안 평가 기준
- **Common Criteria (ISO/IEC 15408)**: TCSEC를 계승한 국제 표준. EAL1~EAL7 등급으로 보안 Target의 보증 수준을 평가
- **NIST SP 800-53**: 미 연방 정보 시스템의 보안 통제 기술적·관리적 권고안

## 👶 어린이를 위한 3줄 비유 설명
1. 보호는 **"집의 자물쇠"**예요. 현관문, 방문, 창문에 자물쇠가 있어서 허락받지 않은 사람은 못 들어오게 해요.
2. 보안은 **"집안의 안전 시스템 전체"**예요. 자물쇠뿐만 아니라 CCTV, 경비원, 화재 경보기까지 모두 합쳐서 우리 가족을 안전하게 지켜요.
3. 자물쇠(보호)가 없으면 도둑이 마음대로 드나들고, CCTV(보안)가 없으면 도둑이 누구인지 몰라요. 둘 다 있어야 집이 안전해요!
