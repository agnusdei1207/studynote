## 핵심 인사이트 (3줄 요약)
> 1. **투명성의 정의**: 위치 투명성(Location Transparency)은 사용자가 파일에 접근할 때 해당 파일이 저장된 서버의 물리적 주소를 알 필요가 없게 하는 분산 시스템의 핵심 성질이다.
> 2. **독립성의 진화**: 위치 독립성(Location Independence)은 파일이 시스템 내 다른 위치로 이동하더라도 파일 이름이나 경로를 변경하지 않고 그대로 접근할 수 있는 고차원적인 추상화다.
> 3. **설계의 가치**: 이 두 개념은 시스템의 유연성을 극대화하며, 관리자가 사용자의 서비스 중단 없이 데이터 마이그레이션이나 서버 증설을 수행할 수 있는 기반이 된다.

---

## Ⅰ. 위치 투명성 (Location Transparency)

- **정의**: 파일의 이름(Name)이 그 파일이 위치한 물리적 서버나 저장 장치의 식별 정보를 포함하지 않는 상태를 의미한다.
- **특징**:
    - **논리적 접근**: 사용자는 서버 IP나 호스트명 대신 `/usr/data/file.txt`와 같은 논리적 경로만 사용한다.
    - **네이밍 스킴(Naming Scheme)**: 파일 이름에서 위치 정보가 분리되어야 한다.
- **예시**: NFS에서 리모트 디렉토리를 로컬 디렉토리에 마운트(Mount)하면, 사용자는 그것이 원격지에 있는지 로컬에 있는지 구분하지 못한다.

📢 **섹션 요약 비유**: 위치 투명성은 "집 주소"와 같다. 우체부는 집의 위도와 경도(물리적 주소)를 몰라도 "서울시 강남구..."라는 주소(논리적 이름)만 보고 편지를 배달할 수 있다.

---

## Ⅱ. 위치 독립성 (Location Independence)

- **정의**: 파일이 한 물리적 노드에서 다른 노드로 이동하더라도 그 파일의 논리적 이름이 변하지 않는 동적인 바인딩(Dynamic Binding) 능력을 말한다.
- **매핑 구조도 (ASCII Diagram)**:

```text
    [ User View ]           [ Mapping Layer ]           [ Physical View ]
 ┌─────────────────┐      ┌───────────────────┐      ┌─────────────────────┐
 │ /global/docs/A  │────▶ │ Global Name       │────▶ │ Server_1: /disk2/A  │
 └─────────────────┘      │ Service (Catalog) │      └─────────────────────┘
                          └─────────┬─────────┘
          ▲                         │ (Migration)
          │                         ▼
 ┌─────────────────┐      ┌───────────────────┐      ┌─────────────────────┐
 │ /global/docs/A  │────▶ │ Updated Mapping   │────▶ │ Server_3: /vol/A    │
 └─────────────────┘      └───────────────────┘      └─────────────────────┘
 (Same Path!)             (Automatic Update)         (New Physical Location)
```

- **차이점**: 위치 투명성은 "현재 위치를 몰라도 됨"을 의미하고, 위치 독립성은 "위치가 바뀌어도 이름이 안 바뀜"을 의미한다. 독립성이 투명성보다 더 구현하기 어렵고 강력한 개념이다.

📢 **섹션 요약 비유**: 위치 독립성은 "휴대폰 번호"와 같다. 내가 서울에 있든 부산에 있든(물리적 이동), 내 번호(논리적 이름)는 그대로 유지되어 사람들이 나를 찾을 수 있는 것과 같다.

---

## Ⅲ. 네이밍 서비스와 해싱 (Naming Service & Hashing)

위치 투명성과 독립성을 구현하기 위해 사용되는 주요 기술들이다.

- **네이밍 서비스 (Naming Service)**: 논리적 이름을 물리적 주소로 변환해주는 전산망의 '전화번호부' 역할을 한다.
- **정적 매핑 (Static Mapping)**: 위치가 고정되어 투명성만 제공하며, 변경 시 관리가 힘들다.
- **동적 매핑 (Dynamic Mapping)**: 위치 정보를 런타임에 결정하여 독립성까지 제공한다.
- **일관된 해싱 (Consistent Hashing)**: 파일 이름을 해시값으로 변환하여 노드에 할당함으로써 노드 추가/삭제 시 이동되는 데이터 양을 최소화한다.

📢 **섹션 요약 비유**: 네이밍 서비스는 "부동산 중개인"과 같다. 세입자(사용자)는 집주인이 바뀌거나 집의 등기부 정보가 변해도 중개인(시스템)을 통해 문제없이 집(파일)을 이용할 수 있다.

---

## Ⅳ. 투명성의 단계별 비교 (Levels of Transparency)

| 단계 | 명칭 | 설명 |
| :--- | :--- | :--- |
| **Lv.1** | **호스트 명시** | `ServerA:/path/file` 형태. 투명성 없음. |
| **Lv.2** | **마운트 기반** | 특정 서버를 로컬 경로에 마운트. 위치 투명성 제공. |
| **Lv.3** | **전역 네임스페이스** | 모든 노드가 동일한 루트 경로 공유. 높은 위치 투명성. |
| **Lv.4** | **위치 독립성** | 데이터 이동 시에도 전역 경로 유지. 최고 수준의 추상화. |

📢 **섹션 요약 비유**: 투명성의 단계는 "해외 여행"과 같다. 현지 가이드를 직접 섭외해야 하는 단계(Lv.1)부터, 여행사가 모든 걸 알아서 해줘서 나는 비행기만 타면 되는 단계(Lv.4)까지의 발전 과정이다.

---

## Ⅴ. 설계 시 고려사항 및 장점 (Design Considerations)

- **유연한 자원 관리**: 사용자의 개입 없이 서버 간 부하 분산(Load Balancing)을 위해 데이터를 자유롭게 이동시킬 수 있다.
- **데이터 복제 통합**: 동일한 논리적 이름을 여러 개의 물리적 복제본에 매핑하여 가용성을 높일 수 있다.
- **복잡성 증가**: 네이밍 서버 자체가 장애점(SPOF)이 될 수 있으며, 매핑 정보를 동기화하는 데 오버헤드가 발생한다.

📢 **섹션 요약 비유**: 투명성 설계는 "자동 변속기"와 같다. 운전자는 기어 단수(물리적 동작)를 신경 쓰지 않아도 되지만, 내부 장치는 훨씬 복잡하고 정교하게 만들어져야 한다.
