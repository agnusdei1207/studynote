# 529. 연결 할당 (Linked Allocation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 연결 할당(Linked Allocation)은 파일 시스템에서 디스크 공간의 효율성을 극대화하기 위해, 물리적으로 분산된 블록들을 포인터(Pointer)로 연결하는 **Singly Linked List (단일 연결 리스트)** 형태의 관리 기법이다.
> 2. **가치**: 연속 할당(Contiguous Allocation)의 치명적인 단점이었던 **External Fragmentation (외부 단편화)**을 근본적으로 해소하며, 파일 크기의 동적 변경(Dynamic Resizing)이 가능하여 현대 대용량 스토리지의 기초가 된다.
> 3. **융합**: 이 구조는 **FAT (File Allocation Table)** 파일 시스템의 이론적 토대이며, 포인터 관리 오버헤드를 해결하기 위해 인덱싱(Indexing) 기법과 융합되어 **EXT4**나 **NTFS**의 Extent 기술로 진화하는 과정의 시발점이다.

## Ⅰ. 개요 (Context & Background)

### 개념 및 철학
연결 할당은 파일을 물리적으로 연속된 디스크 블록에 저장하는 것을 포기하고, 논리적인 순서만 보장하는 방식이다. 각 디스크 블록의 일부 공간을 '다음 블록의 주소'를 저장하는 포인터로 활용한다. 이는 고정된 크기의 "슬롯"에 데이터를 넣으려는 연속 할당 방식의 강박을 버리고, "어떤 빈 공간이든 활용 가능"이라는 유연성을 채택한 것이다.

### 등장 배경 (Background)
1.  **기존 한계**: 연속 할당 방식에서는 파일이 커질 때마다 인접한 빈 공간이 없으면 파일 전체를 이동(Copying)해야 했으며, 파일 생성 시 최종 크기를 예측해야만 했다.
2.  **혁신적 패러다임**: 모든 블록을 동등한 자원으로 간주하여, 물리적 위치와 무관하게 **Chaining (체이닝)** 기법으로 연결한다. 이로 인해 공간 효율성이 극대화되었다.
3.  **현재 비즈니스 요구**: 로그 파일이나 데이터베이스 트랜잭션 로그처럼 크기가 수시로 증감하는 데이터를 관리하는 데 필수적인 메커니즘이다.

### ASCII 다이어그램: 연속 할당 vs 연결 할당

연속 할당은 데이터가 뭉쳐있지만 공간 낭비가 발생하고, 연결 할당은 흩어져 있지만 공간을 100% 활용함을 보여준다.

```text
 [ 연속 할당 (Contiguous Allocation) - 외부 단편화 발생 ]
  Disk: [ File A ] [ File A ] [  Free  ] [ File A ] [ File B ]
                              (이 공간보다 파일 A가 더 커버리면 사용 불가)
                               (Total Size > Free Space Slot)

 [ 연결 할당 (Linked Allocation) - 외부 단편화 없음 ]
  Disk: [ File A-1 ] [ File B-1 ] [ File A-2 ] [ File A-3 ] [ File B-2 ]
          Pointer: 2            Pointer: 3    Pointer: EOF
          (중간의 File B 블록을 건너뛰어 A를 논리적으로 연결)
```

📢 **섹션 요약 비유**: 연결 할당의 도입은 "주차장 관리 방식의 혁신"과 같습니다. 연속 할당은 '5번 연석에 10대를 연달아 세워야 한다'는 규칙 때문에, 5번 구역 중간에 차 한 대가 비어도 주차를 못 해 옆으로 이동해야 하는 불편함이 있습니다. 연결 할당은 "주차장 어디든 빈 곳에 세우고, 차에 '다음 차는 102번에 있어'라는 쪽지를 붙여라"는 방식이므로, 구석에 남은 빈 공간 하나도 낭비하지 않고 모두 채울 수 있습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 상세 분석 (Table)

| 요소명 (Component) | 역할 (Role) | 내부 동작 (Internal Operation) | 프로토콜/구조 | 비유 |
|:---|:---|:---|:---|:---|
| **Directory Entry** | 파일의 시작점 식별 | 파일 생성 시 첫 번째 블록의 주소(물리적 블록 번호) 저장 | `<StartAddr, Len>` or `<StartAddr, EndAddr>` | 보물찾기 시작 위치 |
| **Data Block** | 실제 데이터 저장 | 사용자 데이터 저장 + 다음 블록 포인터(4Byte) 저장 | `<Payload, NextPtr>` | 사탕 + 다음 사탕 위치 |
| **Pointer** | 블록 간 연결 | 현재 블록에서 다음 블록의 물리적 주소를 가리킴 | Block ID (Integer) | 줄 |
| **Free Block List** | 공간 관리 | 비어있는 블록들의 주소를 연결 리스트로 관리하여 할당 시 제공 | Singly Linked List | 빈 주차구역 목록 |
| **EOF Marker** | 종료 식별 | 마지막 블록의 포인터 영역에 `NULL` 또는 `-1` 저장 | `-1` (NULL Pointer) | 도착지 표지판 |

### ASCII 다이어그램: 연결 할당의 메모리 레이아웃 및 동작 과정

파일 `jeep`가 디스크 상에 분산되어 저장된 상태를 보여준다. 디렉터리는 시작 블록(9)만 알고 있으며, 9번 블록은 16번을, 16번은 1번을 가리킨다.

```text
  [ Directory Table ]
  ┌──────────┬───────────┐
  │ Filename │ Start Ptr │
  ├──────────┼───────────┤
  │ jeep     │    9      │  <-- 시작점만 알고 있음
  └──────────┴───────────┘

         │
         ▼
  [ Physical Disk Layout (Logical Block Address) ]
  
  ┌───────┐    ┌───────┐    ┌───────┐    ┌───────┐    ┌───────┐
  │   9   │───▶│  16   │───▶│   1   │───▶│  25   │───▶│   33  │
  ├───────┤    ├───────┤    ├───────┤    ├───────┤    ├───────┤
  │ Data  │    │ Data  │    │ Data  │    │ Data  │    │ Data  │
  │ 0~3KB │    │ 0~3KB │    │ 0~3KB │    │ 0~3KB │    │ 0~3KB │
  ├───────┤    ├───────┤    ├───────┤    ├───────┤    ├───────┤
  │ Ptr:  │    │ Ptr:  │    │ Ptr:  │    │ Ptr:  │    │ Ptr:  │
  │  16   │    │   1   │    │  25   │    │  33   │    │  -1   │ (EOF)
  └───────┘    └───────┘    └───────┘    └───────┘    └───────┘
```

**해설 (Deep Dive)**: 위 다이어그램에서 볼 수 있듯이, 디렉터리 엔트리는 오직 첫 번째 블록인 '9'번만 가리키고 있습니다. 실제 파일 시스템 커널은 9번 블록을 읽어들이고, 그 끝에 붙은 포인터 값을 확인하여 다음 헤드(HDD Head)를 16번으로 이동시킵니다. 이 과정이 반복되면서 논리적으로는 연속된 데이터 스트림처럼 보이게 됩니다. 각 블록 내부의 포인터는 일반적으로 4바이트(32비트 시스템 기준)를 차지하며, 이는 파일 시스템의 오버헤드로 작용합니다.

### 심층 동작 원리 (Deep Dive Mechanics)

1.  **탐색 (Traversal)**: 파일의 N번째 블록에 접근하기 위해서는 반드시 0번째 블록부터 순차대로 포인터를 따라가야 한다. `O(N)`의 시간 복잡도를 가진다.
2.  **블록 할당 (Allocation)**:
    *   파일 생성 시, Free List에서 헤드(Head) 블록을 하나 떼어내어 디렉터리의 Start Pointer에 연결한다.
    *   데이터 추가 시, 새로운 Free Block을 할당하고, 파일의 현재 마지막 블록(End)의 포인터를 새 블록으로 수정한다.
3.  **블록 해제 (Deallocation)**: 파일 삭제 시, 연결된 모든 블록을 순회하며 포인터 연결을 끊고 Free List로 되돌린다. 이 때 연결이 끊어지지 않으면 **Dangling Block**이 발생하여 디스크 공간이 유실된다.

### 핵심 수식 및 코드 (C 스타일 의사코드)

포인터 공간으로 인한 낭비율을 계산하는 수식과 읽기 동작의 코드이다.

```c
// 포인터 오버헤드 계산
// Block Size: 4KB (4096 Bytes), Pointer Size: 4 Bytes
// 유효 데이터 공간 = Block Size - Pointer Size

#define BLOCK_SIZE 4096
#define PTR_SIZE 4
#define PAYLOAD_SIZE (BLOCK_SIZE - PTR_SIZE) // 4092 Bytes

// 파일 시스템 효율성 (지름길 탐색 불가능한 구조)
void read_linked_file(int start_block, int offset) {
    int current = start_block;
    for (int i = 0; i < offset; i++) {
        // 디스크 I/O 발생: 매번 블록을 읽어 포인터를 확인해야 함
        current = read_pointer_from_disk(current); 
        if (current == -1) return; // EOF
    }
    read_data(current);
}
```

📢 **섹션 요약 비유**: 이 구조는 "길게 늘어선 기차 칸 연결"과 같습니다. 객차(블록)는 서로 달랑그랑 연결(포인터)되어 있어서, 기관사가 승객을 100번째 칸에 내려주려면 1번째 칸부터 시작해 칸마다 문을 열고 지나가야 합니다(순차 접근). 하지만 기차를 더 길게 만들고 싶다면, 역에 대기 중인 새 객차를 맨 뒤에 연결하기만 하면 되므로(동적 확장), 기차를 전체 이동시킬 필요가 없어 매우 편리합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 심층 기술 비교: 연속 할당 vs 연결 할당

| 비교 항목 (Metric) | 연속 할당 (Contiguous) | 연결 할당 (Linked) | 비고 |
|:---|:---:|:---:|:---|
| **접근 방식** | **Direct Access (직접 접근)** | **Sequential Access (순차 접근)** | 연속 할당은 수식으로 위치 계산 가능 (Latency 낮음) |
| **외부 단편화** | 발생함 (심각) | **없음 (0)** | Linked는 공간 효율성 최우선 |
| **파일 확장** | 어려움 (이동 비용 발생) | **매우 쉬움** | 파일 크기 예측 불필요 |
| **데이터 신뢰성** | 높음 (일부 손상 시 다른 부분 무난) | **취약** | 포인터 하나 끊어지면 전체 파일 유실 |
| **포인터 오버헤드** | 없음 | **있음** | 블록 내 일부 공간 손실 |

### 과목 융합 분석 (Convergence)

1.  **데이터베이스 (DB)와의 충돌**: RDBMS의 **B-Tree Index** 구조는 연결 할당과 상극이다. 인덱스는 `O(log N)`의 빠른 검색을 위해 데이터가 연속적이거나 최소한 특정 구간에 모여있기를 선호한다. 연결 할당은 디스크 헤드가 트랙을 왔다 갔다 하는 **Thrashing (쓰래싱)**을 유발하여 DB 쿼리 성능을 저하시킬 수 있다.
2.  **운영체제 (OS) 메모리 관리**: 연결 할당은 OS의 가상 메모리 페이지 관리와 유사한 점이 있으나, 메모리는 하드웨어(MMU)의 지원을 받아 페이지 테이블을 통해 논리 주소를 물리 주소로 매핑한다. 즉, 소프트웨어적으로 포인터를 따라가는 연결 할당보다 하드웨어적으로 변환하는 방식이 훨씬 빠르다.

### 비교 다이어그램: 접근 시간(Access Time) 비교

```text
  [ Target: 3rd Block ]
  
  [ Contiguous Allocation ]
  Start ────────────▶ Block 0, Block 1, [Block 2]
  (Math: Start + 2 * BlockSize) => Fast!

  [ Linked Allocation ]
  Start ──▶ Read Ptr ──▶ Read Ptr ──▶ [Block 2]
  (Seek: Head moves to 9, then 16, then 1...) => Slow!
```

📢 **섹션 요약 비유**: "사전 찾기"와 "신문 읽기"의 차이와 같습니다. 연속 할당은 책 페이지 번호를 보고 펄썩 넘겨서 원하는 내용을 찾는 것이고(사전, DB Index), 연결 할당은 1면부터 끝까지 스크롤을 내려서 읽어야 하는 신문과 같습니다. 그래서 뉴스(순차적 데이터)를 볼 때는 좋지만, 단어 하나를 찾을 때(랜덤 접근)는 매우 비효율적입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오 및 의사결정 (Scenarios)

1.  **[시나리오] 임베디드 시스템 로그 저장 장치 설계**
    *   **상황**: 마이크로컨트롤러(MCU)에서 발생하는 디버그 로그를 **NAND Flash**에 저장해야 한다. 전원이 갑자기 끊겨도 데이터가 유실되면 안 된다.
    *   **결정**: 연결 할당 방식이 적합하다. 로그는 순차적으로만 쓰이므로 직접 접근이 필요 없으며, Flash 메모리의 Bad Block(불량 블록)이 발생해도 포인터만 우회시키면 연속적으로 데이터를 기록할 수 있기 때문이다.
2.  **[시나리오] 고화질 비디오 편집 서버**
    *   **상황**: 4K 영상을 실시간으로 렌더링해야 한다. 디스크 I/O 병목이 발생한다.
    *   **결정**: 연결 할당은 부적합하다. 영상 렌더링은 특
