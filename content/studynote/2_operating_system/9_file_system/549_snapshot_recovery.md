## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스냅샷(Snapshot)은 특정 시점(Point-in-time)의 파일 시스템 상태를 사진을 찍듯 저장하여, 원본 데이터의 수정 없이 과거 상태로 즉각 복구할 수 있게 하는 가상화 기술이다.
> 2. **가치**: 전체 데이터를 복제하지 않고 변경된 부분만 관리하는 CoW(Copy-on-Write) 기법을 통해 저장 공간을 절약하면서도 강력한 데이터 보호와 백업 기능을 제공한다.
> 3. **융합**: 현대의 엔터프라이즈 스토리지(ZFS, Btrfs, Cloud Storage)에서 시스템 업데이트 전 보험이나 랜섬웨어 피해 복구를 위한 핵심 기술로 활용된다.

---

## Ⅰ. 스냅샷의 정의와 필요성 (Definition & Necessity)

- **정의**: 스냅샷은 파일 시스템이나 볼륨의 특정 순간을 포착한 읽기 전용(혹은 쓰기 가능) 복사본이다.
- **필요성**: 
    - **실수 복구**: 중요한 파일을 잘못 지웠을 때 몇 분 전 상태로 되돌림.
    - **온라인 백업**: 서비스 중단 없이 현재 상태를 고정하여 백업 장치로 전송.
    - **테스트**: 시스템 설정 변경 전 스냅샷을 찍고, 실패 시 즉시 롤백(Rollback).

📢 **섹션 요약 비유**: 스냅샷은 "게임의 세이브 포인트"와 같다. 보스와 싸우다 죽어도 저장한 시점에서 다시 시작할 수 있는 것과 같다.

---

## Ⅱ. 스냅샷 구현 기술: CoW (Copy-on-Write)

스냅샷은 데이터를 실제로 다 복사하지 않는다.

### 1. 작동 원리
- 스냅샷 생성 시에는 메타데이터(포인터 정보)만 복사한다 (거의 즉시 완료).
- 원본 데이터가 수정될 때, 수정 전 데이터를 다른 곳에 보관하고 원본 위치에는 새 데이터를 쓴다.

### 2. 구조도 (ASCII Diagram)

```text
 [ Original State ]           [ After Modify (Snapshot) ]
 ┌───────────────┐            ┌───────────────┐
 │ Block A       │◀───┐       │ Block A (Old) │◀─── [Snapshot Reference]
 ├───────────────┤    │       ├───────────────┤
 │ Block B       │◀───┤       │ Block B' (New)│◀─── [Original Reference]
 └───────────────┘    │       └───────────────┘
                      │
 [ Snapshot Ref ] ────┘
```

📢 **섹션 요약 비유**: 스냅샷은 "투명 종이를 원본 그림 위에 대고 그리는 것"과 같다. 원본은 그대로 두고 바뀐 부분만 투명 종이에 표시하여 두 장을 겹쳐 보면 최신 상태가 된다.

---

## Ⅲ. 스냅샷의 주요 유형 (Snapshot Types)

- **Read-only Snapshot**: 과거 상태를 보기만 할 수 있으며, 복구 소스로 활용된다.
- **Writable Snapshot (Clone)**: 스냅샷에서 가지를 쳐서 새로운 작업을 수행할 수 있는 독립된 볼륨처럼 작동한다.
- **Differential Snapshot**: 마지막 스냅샷 이후 변경된 블록만 추적하여 저장 효율을 높인다.

📢 **섹션 요약 비유**: Read-only는 "박물관에 전시된 사진"이고, Writable은 "사진을 복사해서 내 마음대로 낙서할 수 있는 종이"다.

---

## Ⅳ. 복구 기술: 롤백 및 파일 단위 복원 (Recovery)

- **전체 롤백 (Rollback)**: 파일 시스템 전체를 스냅샷 시점으로 즉시 되돌린다. 대규모 시스템 장애 시 유용하다.
- **개별 파일 복원**: 스냅샷 영역을 특정 디렉터리(예: `.zfs/snapshot`)로 마운트하여 필요한 파일만 골라서 복사해온다.
- **데이터 일관성**: 데이터베이스처럼 메모리에 데이터가 남아있는 경우, 스냅샷 직전 버퍼를 비우는(Flush) 과정이 필요하다.

📢 **섹션 요약 비유**: 롤백은 "타임머신을 타고 과거로 가는 것"이고, 개별 복원은 "과거의 나에게 가서 잃어버린 물건 하나만 빌려오는 것"이다.

---

## Ⅴ. 한계점 및 관리 주의사항 (Limitations)

- **공간 누적**: 스냅샷을 너무 오래 유지하면 변경된 블록이 많아져 실제 데이터보다 스냅샷이 차지하는 용량이 더 커질 수 있다.
- **성능 저하**: CoW 방식은 데이터를 수정할 때마다 추가적인 쓰기 작업이 발생하므로 미세한 성능 하락이 있을 수 있다.
- **백업과의 차이**: 스냅샷은 동일 디스크 내의 논리적 보호이므로, 디스크 자체가 고장 나면 스냅샷도 사라진다 (물리적 백업 필수).

📢 **섹션 요약 비유**: "세이브 파일이 아무리 많아도 게임기 자체가 부서지면 소용없는 것"과 같다. 별도의 저장 장치에 복사해두는 백업은 여전히 중요하다.
