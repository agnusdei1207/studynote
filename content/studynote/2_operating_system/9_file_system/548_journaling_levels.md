## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 저널링 수준(Journaling Levels)은 성능과 데이터 보호 강도 사이의 절충안을 제공하기 위해 메타데이터와 실제 데이터의 기록 순서를 정의하는 설정이다.
> 2. **가치**: 사용자는 시스템의 용도에 따라 가장 안전한 'Journal', 균형 잡힌 'Ordered', 성능 중심의 'Writeback' 모드 중 최적의 방식을 선택할 수 있다.
> 3. **융합**: 리눅스의 EXT4와 같은 현대 파일 시스템은 성능 저하를 최소화하면서도 파일 시스템 파손을 막을 수 있는 'Ordered' 모드를 기본값으로 채택한다.

---

## Ⅰ. 저널링 수준의 개요 (Overview)

저널링을 수행할 때 무엇을 기록할 것인가에 따라 세 가지 모드로 나뉜다.

- **메타데이터 (Metadata)**: 파일 이름, 크기, 위치 등 관리 정보.
- **사용자 데이터 (User Data)**: 파일의 실제 내용.
- **핵심 딜레마**: 모든 것을 기록하면 안전하지만 느리고, 메타데이터만 기록하면 빠르지만 실제 데이터가 오염될 수 있다.

📢 **섹션 요약 비유**: 저널링 수준은 "보험의 보장 범위"와 같다. 보장 범위가 넓을수록 안심되지만 보험료(성능 비용)가 비싸지는 것과 같다.

---

## Ⅱ. 3단계 모드 상세 분석 (Three Modes)

### 1. Journal 모드 (가장 안전)
- **방식**: 메타데이터와 사용자 데이터를 모두 저널에 기록한다.
- **특징**: 모든 변경 사항을 이중으로 쓰기 때문에 가장 느리지만, 완벽한 복구를 보장한다.

### 2. Ordered 모드 (기본/균형)
- **방식**: 메타데이터만 저널에 기록하되, 저널을 확정(Commit)하기 전에 반드시 실제 데이터를 먼저 디스크에 쓴다.
- **특징**: 메타데이터와 실제 데이터 사이의 순서를 보장하여, 복구 후 파일 내용이 엉뚱한 값으로 채워지는 현상을 방지한다.

### 3. Writeback 모드 (최고 성능)
- **방식**: 메타데이터만 저널에 기록하며, 실제 데이터가 기록되는 순서는 보장하지 않는다.
- **특징**: 쓰기 성능은 가장 좋지만, 장애 복구 후 파일에 이전 작업의 '쓰레기 데이터'가 남아있을 수 있다.

### 비교 구조도 (ASCII Diagram)

```text
 [ Journal Mode ]         [ Ordered Mode ]        [ Writeback Mode ]
 ┌──────────────┐         ┌──────────────┐        ┌──────────────┐
 │ Log: Meta    │         │ Data Write   │        │ Log: Meta    │
 │ Log: Data    │────▶    └──────┬───────┘  ────▶ └──────────────┘
 └──────┬───────┘         ┌──────▼───────┐        ┌──────────────┐
 ┌──────▼───────┐         │ Log: Meta    │        │ Data Write   │
 │ Commit       │         └──────┬───────┘        └──────────────┘
 └──────────────┘         ┌──────▼───────┐           (No Order)
                          │ Commit       │
                          └──────────────┘
```

📢 **섹션 요약 비유**: Journal은 "편지 내용과 주소를 모두 복사해두는 것", Ordered는 "주소를 적기 전에 편지를 먼저 부치는 것", Writeback은 "주소만 적고 편지는 내키는 대로 부치는 것"이다.

## Ⅲ. 성능 및 안전성 트레이드오프 (Trade-off)

| 모드 | 데이터 안전성 | 파일 시스템 무결성 | 성능 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **Journal** | 최고 | 최고 | 최저 | DB 로그 등 특수 용도 |
| **Ordered** | 높음 | 최고 | 중간 | 일반적인 리눅스 기본값 |
| **Writeback** | 낮음 | 최고 | 최고 | 임시 파일, 고속 작업 |

*※ 모든 모드에서 '파일 시스템 구조(무결성)' 자체는 보호된다. 차이는 '사용자 데이터'의 보호 여부다.*

📢 **섹션 요약 비유**: "튼튼한 상자(무결성)는 모두 주지만, 상자 안의 물건(데이터)까지 뽁뽁이로 감쌀지 말지를 고르는 것"이다.

---

## Ⅳ. 장애 시 발생 상황 시나리오 (Failure Scenarios)

- **Journal 모드 시**: 재부팅 후 저널을 다시 읽어 데이터와 메타데이터 모두 사고 직전으로 완벽 복구.
- **Ordered 모드 시**: 메타데이터는 복구되지만, 데이터 기록 중 사고 시 파일 내용 일부가 사라질 수 있음 (하지만 쓰레기 데이터는 없음).
- **Writeback 모드 시**: 메타데이터는 복구되어 파일은 열리지만, 안의 내용은 사고 전 다른 파일이 쓰던 데이터가 섞여 나올 수 있음.

📢 **섹션 요약 비유**: Writeback 모드의 사고는 "새로 산 공책을 펼쳤는데, 예전 사람이 쓰다 만 낙서가 있는 것"과 같은 황당한 상황을 초래할 수 있다.

---

## Ⅴ. 선택 가이드 및 실무 적용 (Practical Guide)

- **서버용**: 안전이 최우선이라면 `data=journal`을 고려할 수 있으나 부하가 큼.
- **데스크탑용**: 성능과 안전의 균형인 `data=ordered`가 최적.
- **데이터베이스**: DB 자체가 로그를 관리하므로 파일 시스템은 `data=writeback`으로 설정하여 중복 오버헤드를 줄이기도 함.

📢 **섹션 요약 비유**: "고속도로(Writeback)를 달릴지, 일반 도로(Ordered)를 달릴지, 안전 점검을 다 받으며 천천히 갈지(Journal) 목적에 맞게 고르는 것"이다.
