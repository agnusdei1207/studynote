# # [임의적 접근 제어 (DAC, Discretionary Access Control)]

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: **DAC (Discretionary Access Control)**는 **"자원의 소유자(Owner)가 자신의 재량(Discretion)에 따라 접근 권한을 제어"**하는 보안 모델이다. "내 파일이니 내가 허가하는 사람에게만 접근을 허용"한다는 철학에 기반하며, **UNIX (Unix-like System)**의 rwx 권한 비트, **ACL (Access Control List)**, **NTFS (New Technology File System)**의 보안 설명자가 대표적인 구현체이다.
> 2. **가치**: DAC는 **"사용자 자율성과 유연성"**을 극대화하여 협업 환경에서의 생산성을 보장하지만, 소유자가 속아서 악성코드를 실행할 경우 공격자도 소유자의 모든 권한을 상속받는 **"트로이 목마(Trojan Horse) 취약성"**이 내재되어 있다. 이는 `setuid` (Set User ID) 권한이 있는 바이너리가 해킹당했을 때 치명적으로 작용하며, **신뢰의 전이(Transitive Trust)** 문제를 야기한다.
> 3. **융합**: 현대 OS는 **"DAC + MAC (Mandatory Access Control) 하이브리드"** 모델을 채택하여 이러한 취약점을 상쇄한다. 일반 사용자 편의성은 DAC로 제공하되, 시스템의 핵심 자원은 **SELinux (Security-Enhanced Linux)**나 **AppArmor** 같은 MAC 계층으로 감싸 트로이 목마 공격을 방어하는 심층 방어 전략(Defense in Depth)을 사용한다.

### Ⅰ. 개요 (Context & Background)

#### 개념 및 철학
**DAC (Discretionary Access Control)**는 정보 보안의 가장 기본이 되는 접근 제어 모델로서, **"자원에 대한 접근 권한을 자원의 소유자(Owner)가 결정하고 통제"**하는 방식이다. 여기서 '임의적(Discretionary)'이라는 용어는 시스템 관리자(System Admin)나 보안 정책이 아닌, 개별 사용자의 자율적 판단(Discretion)에 따라 권한이 위임(Delegation)되거나 철회(Revocation)될 수 있음을 의미한다.

수학적으로는 **Access Matrix** 모델로 표현 가능하며, 구현 시 **ACL (Access Control List)**이나 **Capability**의 형태를 띤다. 객체(Object)인 파일마다 주체(Subject)인 사용자별로 접근 권한(RWX)을 매핑한다. 이는 사용자 간의 데이터 공유를 위해 설계되었기에 유연성(Flexibility)이 뛰어나지만, 반대로 보안 정책의 일관성을 유지하기 어렵고 권한 관리가 복잡해진다는 단점이 있다.

#### 💡 비유
DAC는 **"개인 사무실의 출입키와 방문객 명부"**와 같다. 사무실 주인은 자신의 판단에 따라 직원, 협력사, 손님에게 키를 복제해서 줄 수 있다. 주인이 "이 사람은 믿을 수 있으니 들어오라"고 열어주면 시스템(건물 관리실)은 이를 막지 않는다. 하지만 주인이 사칭 도둑에게 문을 열어주거나, 열쇠를 떨어뜨리면 그 즉시 보안이 뚫리는 구조다.

#### 등장 배경 및 진화
1.  **초기 분산 시스템 (1960s-70s)**: 초기 컴퓨팅은 중앙 집중식이었으나, 시분할 시스템(**TSS, Time-sharing System**)이 도입되며 다중 사용자가 파일을 공유해야 할 필요가 생겼다. 이에 **MULTICS (Multiplexed Information and Computing Service)** 프로젝트에서 최초로 ACL 개념이 도입되었다.
2.  **UNIX의 단순화 (1970s)**: MULTICS의 복잡한 ACL을 단순화하여, **UNIX**는 **"Owner/Group/Other"**라는 세 가지 카테고리와 **"Read/Write/Execute (rwx)"** 세 가지 권한의 조합으로 디스크 공간(Inode)을 절약하면서도 효율적인 보안을 제공하는 9비트 모델을 정착시켰다.
3.  **현대의 확장 (1990s-Present)**: 웹과 클라우드 시대가 오면서 단순한 rwx로는 세밀한 권한 제어가 부족해졌다. 이에 **POSIX (Portable Operating System Interface)** ACL, **NFSv4 (Network File System version 4)** ACL, **AWS IAM (Identity and Access Management)** 정책 등으로 발전하여 더욱 세분화된 DAC를 구현하고 있다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                   DAC vs MAC: 보안 통제권의 비교                     │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [DAC — 임의적 접근 제어 (사용자 자율)]                              │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   Subject (alice)                                              │  │
 │  │     │                                                         │  │
 │  │     ├── "chmod 644 my_file.txt"   ──▶ 소유자가 권한 변경     │  │
 │  │     │   (나는 읽고 쓰고, 너희는 읽기만)                       │  │
 │  │     │                                                        │  │
 │  │     └── "chown bob my_file.txt"    ──▶ 소유권 양도           │  │
 │  │                                                                │  │
 │  │   ◈ 핵심: "내 파일이니 내 마음대로 설정"                       │  │
 │  │   ◈ 위험: alice가 속아서 악성코드 실행 시,                    │  │
 │  │           악성코드도 alice의 권한으로 시스템 전복 가능        │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [MAC — 강제적 접근 제어 (시스템 강제)]                              │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   Subject (alice)                                              │  │
 │  │     │                                                         │  │
 │  │     ├── chown ... (성공)                                       │  │
 │  │     │                                                        │  │
 │  │     └── cat /etc/shadow (실패)                                │  │
 │  │         │                                                     │  │
 │  │         ▼                                                     │  │
 │  │   ◀─ Security Kernel (SELinux/AppArmor)                       │  │
 │  │                                                                │  │
 │  │   ◈ 핵심: "보안 정책 레이블(Security Label)에 따라 강제 통제" │  │
 │  │   ◈ 안전: 소유자이더라도 시스템 정책상 불가는 절대 불가       │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
상단 도표는 보안의 결정권(Dominion)이 누구에게 있는지를 시각화한 것이다. **DAC** 하에서는 사용자(Alice)가 파일 소유자라면 `chmod`(Change Mode)나 `chown`(Change Owner) 명령을 통해 권한을 자유롭게 변경하거나 위임할 수 있다. 이는 유연하지만, 만약 Alice의 계정이 탈취당하거나 Alice가 악성 스크립트를 실행하면, 공격자는 Alice의 권한을 그대로 물려받아 시스템 전체를 위협한다. 반면 **MAC**는 소유자가 누구든 간에, 시스템 전체에 적용된 보안 정책(예: SELinux 커널)이 최종적인 접근 허용 여부를 결정한다. 즉, DAC는 "주인의 허락", MAC는 "국가의 법령"과 같은 개념이다.

> **📢 섹션 요약 비유**
> DAC는 마치 자신의 집 열쇠를 친구에게 맘대로 복제해서 줄 수 있는 **'자율적인 방문객 관리 제도'**와 같습니다. 주인이 열어주면 누구든 들어올 수 있지만, 주인이 현혹되어 도둑을 들여보내도 시스템은 이를 막지 못하는 치명적인 단점이 있습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 구성 요소 상세 분석
DAC의 구현은 Subject(주체), Object(객체), Permission(권한)의 삼각관계로 정의되며, 이를 실제 시스템에서 관리하는 메커니즘이 포함된다.

| 요소명 | 역할 | 내부 동작 및 프로토콜/구조 | 비유 |
|:---|:---|:---|:---|
| **Owner (소유자)** | 객체에 대한 통제권 보유 | **UID (User ID)**로 식별. `chown` 명령으로 변경 가능. 자원의 생성자이거나 권한을 위임받은 자. | 집주인 |
| **Subject (주체)** | 접근을 요청하는 프로세스/사용자 | **RUID (Real UID)**(로그인 사용자)와 **EUID (Effective UID)**(권한 판정 사용자)를 기반으로 검사. | 방문객 |
| **Object (객체)** | 접근의 대상 (파일, IPC 등) | **Inode** 또는 파일 시스템 메타데이터에 **Mode Bits**와 **ACL**을 저장. | 방/금고 |
| **Permission (권한)** | 수행 가능한 연산 종류 | **R (Read)**: 데이터 조회(4), **W (Write)**: 데이터 수정(2), **X (Execute)**: 실행(1). | 출입카드 권한 |
| **ACL (Access Control List)** | 세분화된 권한 명단 | 객체에 부착된 연결 리스트. `{UID, GID, Permissions}` 쌍을 저장하여 9비트 모델의 한계 해결. | VIP 방문객 명단 |
| **ACE (Access Control Entry)** | ACL의 개별 항목 | 각 항목은 Allow(허용) 또는 Deny(거부) 플래그를 가짐. Windows의 **DACL (Discretionary ACL)**에 해당. | 명단의 한 줄 |
| **Identity Token** | 주체의 신분 증명 | **JWT (JSON Web Token)**이나 **Kerberos Ticket** 등 현대적 환경에서의 소유자 증명 수단. | 신분증 |

#### 핵심 동작 메커니즘: UNIX Kernel Permission & setuid
리눅스/유닉스 시스템의 커널(VFS, Virtual File System)은 프로세스가 파일에 접근할 때 다음과 같은 알고리즘을 수행한다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │          UNIX DAC Kernel Algorithm (VFS permission_check)            │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [Context]                                                            │
 │   Process:  EUID=1000 (alice), GIDs=[1000(staff), 1001(dev)]        │
 │   Target File: "secret.txt"                                          │
 │                                                                      │
 │  1. Fetch Inode Metadata of "secret.txt"                             │
 │     ├─ i_uid (Owner): 1000                                          │
 │     ├─ i_gid (Group): 1001 (dev)                                    │
 │     └─ i_mode (Permission): 0rw-rw---- (640)                        │
 │                                                                      │
 │  2. Bitwise Check Logic (순차적 우선순위: ▶ Owner ▶ Group ▶ Other) │
 │     │                                                                │
 │     ├─ Step 1: Owner Match?                                         │
 │     │   if (Process.EUID == Inode.i_uid) {                          │
 │     │       Granted = (Inode.i_mode >> 6) & 0x7;  // Owner Mask     │
 │     │   }                                                           │
 │     │   ⇒ Match! EUID(1000) == i_uid(1000)                          │
 │     │   ⇒ Check (RW- & 0x7). Result: ALLOW (Read/Write)             │
 │     │   → RETURN GRANTED                                            │
 │     │                                                                │
 │     ├─ Step 2: Group Match? (Only if Owner check fails)             │
 │     │   if (Process.GIDs ⊆ {Inode.i_gid}) {                         │
 │     │       Granted = (Inode.i_mode >> 3) & 0x7;  // Group Mask     │
 │     │   }                                                           │
 │     │                                                                │
 │     └─ Step 3: Others Match? (Fallback)                             │
 │         Granted = Inode.i_mode & 0x7;   // Others Mask              │
 │                                                                      │
 │  ┌───────────────────────────────────────────────────────────────┐   │
 │  │  [Special Mechanism: Privilege Escalation via setuid]         │   │
 │  │                                                               │   │
 │  │  File: /usr/bin/passwd  (mode: -rwsr-xr-x, 4755)             │   │
 │  │   │                                                         │   │
 │  │   │  [Normal User Execution]                                │   │
 │  │   │    user$ ./my_app   → EUID = user (1000)                │   │
 │  │   │                                                         │   │
 │  │   │  [setuid Binary Execution]                              │   │
 │  │   │    user$ /usr/bin/passwd                                │   │
 │  │   │    → Kernel detects 's' bit in owner permission (chmod 4755)│   │
 │  │   │    → Process.EUID = File.Owner_UID (root, 0)             │   │
 │  │   │    → Process temporarily gains ROOT PRIVILEGES          │   │
 │  │   │                                                         │   │
 │  │   ▼  [Security Implication]                                 │   │
 │  │   If /usr/bin/passwd has a buffer overflow (BOF) bug:       │   │
 │  │   Attacker can spawn shell(/bin/sh) with EUID=0 (ROOT).     │   │
 │  │   ⇒ Total System Compromise                                 │   │
 │  └───────────────────────────────────────────────────────────────┘   │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```
