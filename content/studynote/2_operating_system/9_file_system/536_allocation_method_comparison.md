# 536. 성능 비교: 연속 vs 연결 vs 인덱스 할당

## # [파일 시스템 할당 기법 성능 비교]

### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 디스크 공간 할당은 `외부 단편화 (External Fragmentation)`를 허용하여 접근 속도를 극대화할 것인가(연속), 속도를 희생하여 공간 효율성과 동적 확장성을 확보할 것인가(연결), 혹은 `인덱스 블록 (Index Block)`이라는 메타데이터 오버헤드를 감수하고 둘을 절충할 것인가(인덱스)에 대한 트레이드오프(Trade-off) 설계 문제이다.
> 2. **가치**: 현대 파일 시스템의 성능, 특히 `TPS (Transactions Per Second)`와 `Latency (지연 시간)`는 선택한 할당 기법에 따라 결정된다. HDD(Hard Disk Drive) 환경에서는 `Seek Time (탐색 시간)` 최소화가 연속 할당의 우위를 점하게 하며, SSD(Solid State Drive) 환경에서는 인덱스 할당의 논리적 관리 효율이 중요해진다.
> 3. **융합**: 단일 기법의 한계를 극복하기 위해 UNIX의 `inode (Information Node)` 기반 다단계 인덱싱과 현대 파일 시스템(Ext4, NTFS)의 `Extents (익스텐트)` 기법이 결합되어, 대용량 파일 처리와 높은 동시성을 지원하는 복잡한 아키텍처로 진화하고 있다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 철학
운영체제(OS)가 파일(File)을 보조기억장치에 저장할 때, 논리적인 파일과 물리적인 블록(Block)을 매핑하는 방식은 파일 시스템의 성능과 신뢰성을 지배합니다. 크게 `연속 할당 (Contiguous Allocation)`, `연결 할당 (Linked Allocation)`, `인덱스 할당 (Indexed Allocation)` 세 가지 패러다임으로 나뉩니다.
이는 마치 도시 계획에서 토지를 구획하는 방식과 같습니다. 연속 할당은 대규모 단지를 연필로 그어 한 번에 조성하는 방식이고, 연결 할당은 여기저기 흩어진 빈 땅을 사서 다리(포인터)로 연결하는 방식이며, 인덱스 할당은 중앙 통제실에서 각 지역의 위치 정보를 담은 목록을 관리하는 방식에 비유할 수 있습니다.

#### 2. 등장 배경 및 진화
1.  **초기의 성능 최적화 (연속 할당)**: 초기 컴퓨터 시스템에서는 처리해야 할 파일의 크기가 고정되어 있거나 변경이 드물었습니다. 당시 가장 큰 병목은 `Seek Time`과 `Rotational Latency`였으므로, 데이터를 물리적으로 인접한 곳에 모아두는 연속 할당이 가장 효율적이었습니다.
2.  **공간 효율성의 요구 (연결 할당)**: 시간이 지나며 파일의 생성과 삭제가 빈번해지면서, 디스크 곳곳에 사용 불가능한 작은 빈 공간들이 널브러지는 `외부 단편화 (External Fragmentation)` 문제가 심각해졌습니다. 이를 해결하기 위해 물리적으로 떨어진 빈 공간들을 논리적으로 연결하여 활용하는 연결 할당이 고안되었습니다.
3.  **현대의 절충안 (인덱스 할당)**: 연결 할당은 임의 접근(`Random Access`)이 불가능하여 성능 저하가 심했습니다. 이를 해결하기 위해 파일의 모든 블록 주소를 별도의 블록(`Index Block`)에 모아두는 인덱스 할당이 등장했습니다. 이는 공간 효율과 접근 속도라는 두 마리 토끼를 잡기 위한 현대 파일 시스템의 기초가 되었습니다.

#### 💡 비유
연속 할당은 "기차가 갈 수 있는 직선적인 선로를 미어 째기 위해 자리를 통으로 비워두는 것"과 같습니다. 연결 할당은 "각 역마다 다음 역의 위치를 적은 쪽지를 남겨두고, 기차가 그 쪽지를 보고 여기저기 기동하는 것"입니다. 인덱스 할당은 "기관사가 중앙 통제실의 전체 노선도를 보고 바로 목적지로 향하는 것"과 같습니다.

#### 📢 섹션 요약 비유
파일 시스템의 할당 기법 진화는 "도시 정비 시스템의 발전"과 같습니다. 연속 할당은 거대한 땅을 통으로 매입하여 정리하는 것(공간 낭비 심함), 연결 할당은 빈 틈틈이 주차장을 만들되 다음 주차장 위치를 표시해두는 것(이동 시간 오래 걸림), 인덱스 할당은 스마트 주차 앱을 통해 실시간으로 빈자리를 찾아 바로 이동하는 것(정보 관리 비용 발생)에 비유할 수 있습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 구성 요소 및 메커니즘 상세 비교
각 할당 기법은 `Metadata` 구조와 `Block` 관리 방식에서 결정적인 차이를 보입니다.

| 구성 요소 | 연속 할당 (Contiguous) | 연결 할당 (Linked) | 인덱스 할당 (Indexed) |
|:---|:---|:---|:---|
| **주소 매핑 구조** | 시작 블록 번호 <br> 파일 길이 (`Length`) | 각 블록의 헤더에 <br> 다음 블록 주소(`Pointer`) 포함 | 전용 인덱스 블록에 <br> 전체 데이터 블록 주소 배열(`Array`) 저장 |
| **접근 유형** | `Direct Access` (직접 접근) <br> `Sequential Access` (순차 접근) | `Sequential Access`만 가능 <br> (임의 접근 시 O(N) 탐색) | `Direct Access` 가능 <br> (O(1) ~ O(log N) 연산) |
| **핵심 파라미터** | `Start Address`, `File Size` | `Next Pointer` (보통 4B/8B) | `Index Block Pointer` (i-node number) |
| **대표 시스템** | IBM OS/360, CD-ROM <br> (ISO 9660) | MS-DOS `FAT` (File Allocation Table) <br> (FAT는 연결 할당의 변형) | UNIX `UFS`, Linux `ext4`, NTFS |

#### 2. ASCII 구조 다이어그램: 디스크 레이아웃 물리적 관점
아래 다이어그램은 세 가지 방식이 물리적 디스크 상에서 어떻게 데이터를 배치하고 관리하는지 시각화한 것입니다.

```text
+--------------------- DISK PHYSICAL LAYOUT VIEW ---------------------+
|                                                                      |

  (1) Contiguous Allocation         (2) Linked Allocation
  [  File A (Start: 10, Len: 4)  ]    [ File A ]              [ File A ]
  +----+----+----+----+----+         +--+--+--+               +--+--+--+
  | 10 | 11 | 12 | 13 | 14 |         | 10 | 11 |               | 20 | 21 |
  | D1 | D2 | D3 | D4 |    |         | D1 |--▶| D2 |--▶ ... --▶| D4 |--▶ NIL
  +----+----+----+----+----+         +-----+--+               +-----+--+
      ▲        ▲
      |        |                           (Pointer embedded in data block)
   Start     End

  (3) Indexed Allocation
  [ Index Block (No. 50) ]           Data Blocks are randomly scattered...
  +-----+-----+-----+-----+           +--+--+          +--+--+
  | 103 | 207 | 405 | NIL |           |D1 |  |          |D3 |  |
  +-----+-----+-----+-----+           +--+--+          +--+--+
    ▲     ▲     ▲                       ▲               ▲
    |     |     |                       |               |
    +-----+-----+-----------------------+---------------+
              (Index Block holds direct pointers to data blocks)
+----------------------------------------------------------------------+
```

**해설**:
- **연속 할당**: 파일이 10번 블록에서 시작해 4개의 블록(10, 11, 12, 13)을 물리적으로 연속하여 점유합니다. `Disk Head`가 한 번 이동하면 나머지 데이터를 회전 지연(Rotational Latency) 없이 읽을 수 있어 순차 읽기 성능이 가장 뛰어납니다.
- **연결 할당**: 데이터 블록 내부에 데이터뿐만 아니라 다음 블록을 가리키는 포인터가 함께 저장됩니다. 이로 인해 실제 데이터 저장 공간이 줄어들며, 임의 위치의 데이터를 읽으려면 첫 번째 블록부터 순차적으로 따라가야 합니다.
- **인덱스 할당**: 50번 블록은 데이터가 아닌, 데이터가 위치한 주소(103, 207, 405)만을 담고 있습니다. 이를 통해 OS는 인덱스 블록만 읽으면 즉시 원하는 데이터 블록으로 이동(`Seek`)할 수 있습니다.

#### 3. 심층 동작 원리: 임의 접근(Random Access) 및 파일 확장
파일의 중간에 위치한 데이터를 읽거나 쓸 때 각 시스템의 내부 동작 메커니즘과 연산 복잡도는 다음과 같습니다.

**A. 연속 할당: O(1) 수학적 연산**
- **메커니즘**: 파일 시스템은 `Start Address`와 `Offset`을 이용해 물리적 주소를 즉시 계산합니다.
- **공식**: `Physical Block = Start Block + Floor(Offset / Block_Size)`
- **장점**: 추가적인 탐색 없이 즉시 접근합니다. 순차 읽기(`Prefetch`)에 최적화되어 있습니다.
- **단점**: 파일이 커질 경우 인접 공간이 이미 점유되어 있다면, 전체 파일을 새로운 위치로 이동(`Migration`)시켜야 하므로 막대한 오버헤드가 발생합니다.

**B. 연결 할당: O(N) 선형 탐색**
- **메커니즘**: N번째 블록에 접근하기 위해 첫 번째 블록부터 시작하여 포인터를 N번 따라갑니다.
- **오버헤드**: 디스크의 `Seek Time`이 N번 발생하며, 이는 HDD의 물리적 한계 때문에 성능을 저하시키는 주된 원인이 됩니다.
- **FAT (File Allocation Table)의 변형**: MS-DOS의 경우 포인터를 디스크 내부가 아닌 메모리에 캐싱된 테이블에 관리하여 성능을 개선했으나, 여전히 큰 파일의 관리 비용이 높습니다.

**C. 인덱스 할당: O(1) ~ O(log N) 트리 접근**
- **메커니즘**: 파일의 메타데이터 영역(`inode` 등)에 있는 인덱스 블록을 읽고, 인덱스 배열의 해당 인덱스(Index)를 참조하여 데이터 블록 주소를 얻습니다.
- **확장성**: 파일이 커서 인덱스 블록이 부족해지면, 인덱스 블록이 다른 인덱스 블록을 가리키는 `Multilevel Index` 구조로 확장됩니다. 이는 메모리 관리의 페이징(Paging) 기법과 논리적으로 유사합니다.

#### 4. 핵심 알고리즘 코드 및 수식
인덱스 할당에서의 주소 계산 로직은 다음과 같이 구현될 수 있습니다.

    # 1. Calculate the index within the file
    index = file_offset // block_size

    if index < 12: # Usually 12 direct pointers in ext4
        return inode.direct[index]

    index -= 12

        indirect_block = read_disk(inode.single_indirect)
        return indirect_block[index]

#### 📢 섹션 요약 비유
연속 할당은 "고속철도가 정차하지 않고 달리는 직선 선로"와 같아서 빠르지만 경로 변경이 불가능합니다. 연결 할당은 "각 집마다 다음 집 주소를 물어보면서 찾아가는 구석진 골목길"이라서 목적지까지 가는 데 시간이 오래 걸립니다. 인덱스 할당은 "목적지별로 좌석 번호가 적힌 공연장 예약 시스템"과 같아서, 목록(인덱스)만 보면 바로 그 자리로 찾아갈 수 있습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 심층 기술 비교: 정량적 메트릭스 분석표
파일 시스템 설계 시 고려해야 할 핵심 지표들을 정량화하여 비교합니다. (HDD 기준, Seek Time ≈ 5ms 가정)

| 비교 항목 | 연속 할당 (Contiguous) | 연결 할당 (Linked) | 인덱스 할당 (Indexed) |
|:---|:---|:---|:---|
| **임의 접근 성능**<br>(Random Access) | **최상 (O(1))**<br>연산 1회, Seek 1회 | **최악 (O(N))**<br>Seek N회 (100번째 블록 시 500ms 소요) | **우수 (O(1)~O(log N))**<br>Seek 2회 (인덱스 + 데이터) |
| **공간 효율성**<br>(Space Efficiency) | **취약**<br>`External Fragmentation` 심각<br>(Compaction 필요) | **우수**<br>단편화 없음<br>단, 블록 내 `Pointer` 공간 낭비 (약 12.5% @ 4KB/4B) | **우수**<br>단편화 없음<br>`Index Block` 자체 공간 필요, 파일 내 빈 공간(`Internal Fragmentation`) 발생 가능 |
| **동적 확장성**<br>(Dynamic Growth) | **매우 어려움**<br>인접 공간 부족 시 파일 전체 이동 발생 | **매우 쉬움**<br>빈 블록이면 어디든 연결 가능 | **쉬움**<br>인덱스 배열에 빈 슬롯만 있으면 됨 |
| **신뢰성/복구**<br>(Reliability) | **높음**<br>섹터 오류 시 해당 부분만 손상 | **취약함**<br>하나의 포인터 파손 시 연결 끊어짐 (`Chain Corruption`) | **중간**<br>Index Block 손상 시 전체 파일 소실<br>-> `
