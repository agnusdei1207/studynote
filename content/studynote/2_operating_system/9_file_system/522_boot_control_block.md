## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 부트 제어 블록(BCB, Boot Control Block)은 시스템을 부팅(Booting)하기 위해 필요한 최소한의 로더(Loader)와 구성 정보를 담고 있는 저장 장치의 첫 번째 섹터 또는 블록이다.
> 2. **가치**: 운영체제(OS, Operating System)가 아직 메모리에 로드되지 않은 상태에서, 하드웨어가 스스로 OS를 찾아 실행할 수 있게 해주는 마법의 '시동 키' 역할을 한다.
> 3. **융합**: BIOS/UEFI 펌웨어와 연계되어 MBR(Master Boot Record) 또는 VBR(Volume Boot Record) 형태로 존재하며, 시스템의 무결성 검증과 멀티 부팅 관리의 시작점이 된다.

---

## Ⅰ. 부트 제어 블록의 정의와 역할 (Definition & Role)

- **정의**: 파티션(또는 볼륨)마다 존재할 수 있으며, 해당 파티션에서 운영체제를 부팅하는 데 필요한 정보를 포함하는 블록이다.
- **주요 역할**:
    - **부트 로더(Boot Loader) 저장**: 커널을 메모리로 읽어 들이는 작은 프로그램을 보관한다.
    - **파티션 상태 확인**: 부팅 가능한(Active) 파티션인지 여부를 확인한다.
    - **시동 시퀀스(Startup Sequence)**: BIOS/UEFI가 제어권을 넘겨주는 지점이다.

📢 **섹션 요약 비유**: 부트 제어 블록은 자동차의 "시동 버튼"과 같다. 버튼을 누르면 엔진(커널)이 돌아가기 시작하듯, 시스템이 깨어나는 첫 번째 물리적 접점이다.

---

## Ⅱ. 부트 제어 블록의 물리적 배치 (Physical Layout)

부트 제어 블록은 대개 디스크의 가장 앞에 위치하며, 고정된 규격을 따른다.

### 1. MBR (Master Boot Record) vs VBR (Volume Boot Record)
- **MBR**: 디스크 전체의 첫 번째 섹터. 파티션 테이블과 주 부트 로더를 포함한다.
- **VBR**: 각 파티션의 첫 번째 블록. 특정 OS를 부팅하기 위한 부트 코드를 포함한다.

### 2. 내부 구조 및 흐름 (ASCII Diagram)

```text
 [ Physical Disk Sector 0 ]
 ┌────────────────────────────────────────────────────────┐
 │ 1. Boot Code (부트 로더의 일부: 446 Bytes)               │
 ├────────────────────────────────────────────────────────┤
 │ 2. Partition Table (파티션 정보: 64 Bytes)               │
 │    - Partition 1 (Active/Bootable)                     │
 │    - Partition 2                                       │
 ├────────────────────────────────────────────────────────┤
 │ 3. Signature (유효성 확인: 0x55AA)                      │
 └────────────────────────────────────────────────────────┘
            │
            ▼ (제어권 이전)
 ┌────────────────────────────────────────────────────────┐
 │ Volume Boot Record (VBR) - Partition 1의 첫 블록          │
 │ ┌────────────────────────────────────────────────────┐ │
 │ │ OS-specific Boot Loader (커널 로드 프로그램)         │ │
 │ └────────────────────────────────────────────────────┘ │
 └────────────────────────────────────────────────────────┘
            │
            ▼ (메모리로 로드)
      [ Operating System Kernel ]
```

📢 **섹션 요약 비유**: 부트 제어 블록의 구조는 "보물지도의 첫 단서"와 같다. 지도(MBR)의 첫 머리를 읽어야 진짜 보물(OS)이 있는 위치(VBR/Kernel)를 알 수 있다.

---

## Ⅲ. 부팅 프로세스와 BCB의 상호작용 (Booting Process)

1. **Power On Self Test (POST)**: 하드웨어 점검.
2. **Firmware Execution**: BIOS 또는 UEFI가 실행되어 부팅 장치를 찾는다.
3. **Load BCB/MBR**: 부팅 장치의 첫 섹터를 읽어 메모리에 올리고 제어권을 넘긴다.
4. **Boot Loader Execution**: BCB 내의 부트 코드가 파티션 테이블을 확인하여 실행할 운영체제 커널을 찾는다.
5. **Kernel Loading**: 최종적으로 커널이 메모리에 로드되어 운영체제가 제어권을 갖는다.

📢 **섹션 요약 비유**: 부팅 프로세스는 "릴레이 경주"와 같다. 하드웨어(BIOS)가 바통(제어권)을 BCB에게 넘겨주면, BCB가 이를 최종 주자(커널)에게 전달한다.

---

## Ⅳ. 파일 시스템별 BCB 명칭 및 차이 (Naming)

운영체제와 파일 시스템에 따라 부트 제어 블록을 부르는 명칭이 다를 수 있다.

- **UFS (Unix File System)**: Boot Block이라 부른다.
- **NTFS (New Technology File System)**: Partition Boot Sector라고 칭한다.
- **FAT (File Allocation Table)**: BIOS Parameter Block (BPB) 영역이 부트 섹터의 일부로 포함된다.
- **특징**: 만약 해당 파티션에 OS가 없다면, 이 블록은 비어 있거나 단순히 "No OS found"라는 메시지만 출력하는 코드를 담는다.

📢 **섹션 요약 비유**: BCB의 명칭은 다르지만 기능은 같다. 전 세계 어디를 가든 "입구"라는 표지판의 언어는 달라도(명칭) 들어가는 곳(기능)이라는 점은 변하지 않는 것과 같다.

---

## Ⅴ. 부트 제어 블록의 손상과 복구 (Damage & Recovery)

BCB는 물리적으로 매우 중요한 위치에 있기 때문에 손상 시 시스템 부팅이 불가능해진다.

- **손상 원인**: 물리적 배드 섹터, 마스터 부트 레코드 바이러스, 파티션 테이블 오류 등.
- **복구 방법**: 
    - **FixMBR / FixBoot**: 윈도우 복구 모드에서 부트 레코드를 다시 쓴다.
    - **Grub Reinstall**: 리눅스에서 부트 로더(GRUB)를 다시 설치하여 BCB를 갱신한다.
    - **Backup**: 현대적인 파일 시스템은 BCB의 복사본을 볼륨의 끝부분에 백업해 두기도 한다.

📢 **섹션 요약 비유**: BCB 손상은 "열쇠가 부러진 것"과 같다. 집(시스템)은 멀쩡하지만 들어갈 방법이 없으므로, 열쇠공(복구 도구)을 불러 새 열쇠(BCB 재설치)를 맞춰야 한다.
