## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 1단계 디렉터리 (Single-level Directory)는 시스템 내의 모든 파일이 단 하나의 중앙 디렉터리에만 위치하는 가장 단순한 논리적 구조다.
> 2. **가치**: 구현이 매우 간단하고 오버헤드가 적어 초창기 운영체제나 임베디드 시스템의 단순 저장 장치 관리용으로 적합하다.
> 3. **한계**: 파일 수가 증가함에 따라 파일 이름 중복(Naming Conflict) 문제와 검색 성능 저하가 발생하며, 다중 사용자 환경에서의 그룹화(Grouping)가 불가능하다.

---

## Ⅰ. 개요 (Context & Background)

- **정의**: 1단계 디렉터리 (Single-level Directory)는 모든 사용자의 모든 파일이 하나의 디렉터리 테이블에 저장되는 선형적인 구조를 의미한다.
- **배경**: 컴퓨터 자원이 극도로 제한적이었던 초기 컴퓨팅 환경에서, 복잡한 계층 구조 없이 파일 이름과 물리적 주소를 매핑하는 최소한의 기능만을 제공하기 위해 탄생했다.
- **핵심 특징**:
  - **평면 구조**: 폴더나 하위 디렉터리라는 개념이 존재하지 않는다.
  - **전역 고유성**: 시스템 내의 모든 파일 이름은 유일해야 한다.

📢 **섹션 요약 비유**: 1단계 디렉터리는 "칸막이가 없는 커다란 상자"와 같다. 모든 물건을 한 곳에 쏟아부어 관리하므로 넣기는 쉽지만, 물건이 많아지면 찾기 힘들고 이름표가 겹치면 큰 혼란이 생긴다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 논리적 구조 시각화 (ASCII Diagram)

```text
 [ Single-level Directory Table ]
 ┌─────────────┬─────────────┬─────────────┬─────────────┐
 │ File Name   │ Attributes  │ Address     │ Owner       │
 ├─────────────┼─────────────┼─────────────┼─────────────┤
 │ cat.jpg     │ rw-r--r--   │ Block 120   │ User_A      │
 │ data.csv    │ rwxr-xr-x   │ Block 450   │ User_B      │
 │ notes.txt   │ rw-------   │ Block 098   │ User_A      │
 │ test.py     │ rwxr-xr-x   │ Block 211   │ User_C      │
 └─────────────┴─────────────┴─────────────┴─────────────┘
        │             │             │
        └─────────────┴──────┬──────┴─────────────┐
                             ▼
              [ Physical Storage Blocks ]
              ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
              │120│ │450│ │098│ │211│ │...│
              └───┘ └───┘ └───┘ └───┘ └───┘
```

### 2. 주요 동작 메커니즘
- **파일 생성 (Create)**: 디렉터리 테이블을 처음부터 끝까지 검색하여 동일한 이름이 있는지 확인한 후, 비어있는 슬롯에 새 항목을 추가한다.
- **파일 검색 (Search)**: 사용자가 입력한 이름을 키(Key)로 하여 선형 검색 (Linear Search)을 수행한다. 파일 수가 $N$개일 때 시간 복잡도는 $O(N)$이다.
- **파일 삭제 (Delete)**: 해당 항목을 찾아 삭제하고, 해당 항목이 점유하던 디렉터리 공간과 데이터 블록을 회수한다.

📢 **섹션 요약 비유**: 마치 "전교생 이름을 한 장의 명렬표에 적어둔 것"과 같다. 이름만 대면 바로 찾을 수 있지만, 동명이인이 생기면 누구인지 구분할 방법이 없는 것과 같은 원리다.

---

## Ⅲ. 기술적 제약과 문제점 (Constraints & Issues)

- **이름 충돌 (Naming Problem)**: 서로 다른 사용자가 같은 이름의 파일을 만들려고 할 때 충돌이 발생한다. 다중 사용자 환경에서는 치명적인 결함이다.
- **그룹화 부족 (Grouping Problem)**: 파일의 성격(사진, 문서, 코드 등)에 따라 분류할 수 없어, 파일 수가 수천 개를 넘어가면 사용자가 자신의 파일을 기억하기 매우 어렵다.
- **확장성 (Scalability)**: 검색 성능이 파일 수에 비례하여 선형적으로 떨어지므로 대용량 파일 시스템에는 부적합하다.

📢 **섹션 요약 비유**: "수만 장의 서류를 번호 순서대로만 쌓아둔 서류 더미"와 같다. 특정 서류를 찾으려면 처음부터 끝까지 뒤져야 하며, 분류가 안 되어 있어 관리 효율이 극도로 떨어진다.

## Ⅳ. 융합 및 비교 분석 (Comparison)

| 비교 항목 | 1단계 디렉터리 | 2단계 디렉터리 | 트리 구조 디렉터리 |
|:---|:---|:---|:---|
| **복잡도** | 매우 낮음 | 중간 | 높음 |
| **이름 중복** | 불가능 (Global Unique) | 사용자 내 중복 불가 | 경로가 다르면 허용 |
| **그룹화** | 불가능 | 사용자별 그룹화 | 무한 계층 그룹화 |
| **검색 속도** | 느림 ($O(N)$) | 빠름 ($O(N/User)$) | 매우 빠름 (Index 활용) |
| **주 사용처** | 초기 OS, 마이크로컨트롤러 | 초기 다중 사용자 시스템 | 현대 범용 OS (Windows, Linux) |

📢 **섹션 요약 비유**: 1단계가 "거대한 원룸"이라면, 2단계는 "방이 여러 개인 아파트", 트리 구조는 "도시 전체의 번지수 체계"로 진화한 것과 같다.

## Ⅴ. 구현 및 미래 전망 (Implementation & Outlook)

- **물리적 구현**: 보통 고정 크기 레코드의 배열 (Array of Fixed-size Records)로 구현된다. 파일 시스템의 특정 섹터에 디렉터리 영역을 할당한다.
- **현대적 관점**: 범용 OS에서는 사라졌으나, 아두이노(Arduino)와 같은 초소형 임베디드 장치의 SD 카드 라이브러리 초기 버전이나, FAT12의 루트 디렉터리 구조 등에서 그 흔적을 볼 수 있다.
- **최적화**: 검색 속도를 높이기 위해 해시 테이블 (Hash Table)을 적용하기도 하지만, 구조적 한계(분류 불가)는 해결되지 않는다.

📢 **섹션 요약 비유**: 이제는 골동품이 되었지만, "복잡한 도구 없이 단순하게 물건을 보관해야 하는 임시 창고"에서는 여전히 유효한 개념이다.
