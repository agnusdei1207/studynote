## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 일관성 체크(Consistency Checking)는 갑작스러운 시스템 장애로 인해 발생한 파일 시스템 메타데이터의 불일치(Inconsistency)를 탐지하고 복구하는 사후 관리 프로세스다.
> 2. **가치**: 데이터 블록 할당 오류, 연결 유실(Orphaned files), 참조 횟수 불일치 등을 바로잡아 파일 시스템의 무결성(Integrity)을 유지하고 시스템 붕괴를 막는다.
> 3. **융합**: 현대 시스템은 `fsck`의 긴 소요 시간을 단축하기 위해 저널링(Journaling) 기법을 병행하며, 부팅 시 신속한 무결성 검증을 수행한다.

---

## Ⅰ. 불일치 발생의 원인 (Causes of Inconsistency)

파일 시스템은 여러 개의 메타데이터를 순차적으로 업데이트한다.

- **상황**: 파일 생성 시 1) inode 할당, 2) 디렉터리 엔트리 추가, 3) 데이터 블록 할당 순으로 작업이 진행된다.
- **장애**: 이 과정 중간에 전원이 차단(Power Failure)되거나 커널 패닉이 발생하면, 어떤 정보는 기록되고 어떤 정보는 기록되지 않아 논리적 모순이 생긴다.
- **결과**: 파일은 존재하나 이름이 없거나, 사용 중인 블록이 자유 블록 리스트에도 포함되는 등의 문제가 발생한다.

📢 **섹션 요약 비유**: 일관성 불일치는 "이체 중 돈은 빠져나갔는데 상대방 계좌엔 입금이 안 된 상황"과 같이 장부의 앞뒤가 맞지 않는 상태다.

---

## Ⅱ. fsck의 작동 메커니즘 (How fsck Works)

`fsck` (File System Consistency Check)는 일반적으로 다음 5단계를 거쳐 검사를 수행한다.

### 1. 주요 검사 단계
- **Pass 1 (블록 및 크기 검사)**: 모든 inode를 훑으며 구조적 오류를 찾는다.
- **Pass 2 (디렉터리 검사)**: 디렉터리 구조와 파일 이름의 연결을 확인한다.
- **Pass 3 (연결 검사)**: 모든 디렉터리가 루트로부터 연결되어 있는지 확인한다.
- **Pass 4 (참조 횟수 검사)**: 파일의 실제 연결 수와 inode의 링크 카운트를 대조한다.
- **Pass 5 (자유 블록 검사)**: 비트맵(Bitmap) 정보가 실제 할당 상태와 일치하는지 확인한다.

### 2. 검사 개념도 (ASCII Diagram)

```text
 [ Metadata Table ]         [ Scan Process ]         [ Actions ]
 ┌───────────────┐        ┌────────────────┐       ┌─────────────────┐
 │ inode #10:    │        │ 1. Is Used?    │──Yes─▶│ Check Links     │
 │ Size: 5KB     │◀───────┤ 2. Blocks OK?  │       ├─────────────────┤
 ├───────────────┤        │ 3. Valid Path? │──No──▶│ Move to /lost+found
 │ Block Bitmap: │        └────────────────┘       └─────────────────┘
 │ 1011001...    │◀──────────────┘
 └───────────────┘
```

📢 **섹션 요약 비유**: `fsck`는 "도서관의 모든 책을 일일이 꺼내어 대출 장부와 맞는지 전수조사하는 사서"와 같다.

---

## Ⅲ. 주요 복구 항목 (Common Repair Items)

- **lost+found**: 이름은 잃어버렸지만 데이터 블록은 살아있는 파일들을 이 디렉터리로 모아 사용자가 수동으로 복구하게 한다.
- **중복 할당 블록**: 두 파일이 같은 데이터 블록을 가리키는 경우, 데이터를 복사하여 분리한다.
- **잘못된 링크 카운트**: 실제 파일은 지워졌는데 링크 수가 0이 아니어서 용량만 차지하는 경우 이를 0으로 수정하여 공간을 해제한다.

📢 **섹션 요약 비유**: `lost+found`는 "이름표가 떨어진 분실물을 모아두는 학교 복도의 분실물 보관함"과 같다.

---

## Ⅳ. fsck의 한계와 저널링의 등장 (Limitations)

- **시간 소모**: 디스크 크기가 커질수록 모든 inode를 검사하는 시간이 기하급수적으로 늘어난다 (테라바이트 단위에서 수 시간이 걸릴 수 있음).
- **오프라인 검사**: 대부분의 경우 파일 시스템을 언마운트(Unmount)한 상태에서 실행해야 하므로 시스템 가용성이 떨어진다.
- **대안**: 현대 OS는 **저널링(Journaling)**을 통해 사고 직전의 로그만 확인하여 수 초 내에 복구를 완료한다.

📢 **섹션 요약 비유**: 전수조사(`fsck`)가 너무 오래 걸리자, "매일 작성한 가계부(저널링)만 확인해서 틀린 부분만 고치는 방식"으로 발전한 것이다.

---

## Ⅴ. 관리자 가이드 (Best Practices)

- **강제 실행 금지**: 마운트된 상태에서 `fsck`를 강제로 실행하면 데이터 파손의 위험이 크다.
- **정기 검사**: 저널링이 있더라도 가끔은 정밀 검사를 수행하는 것이 안전하다.
- **백업 우선**: 심각한 오류 시 `fsck`가 데이터를 삭제하며 복구할 수 있으므로, 실행 전 디스크 이미지를 백업하는 것이 권장된다.

📢 **섹션 요약 비유**: "수술(fsck)을 하기 전에 반드시 동의서(백업)를 받고 환자의 상태(언마운트)를 확인해야 안전한 것"과 같다.
