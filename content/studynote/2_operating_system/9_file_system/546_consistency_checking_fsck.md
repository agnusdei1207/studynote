# # [파일 시스템의 일관성 체크 (Consistency Checking) - fsck]

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: `fsck` (File System Consistency Check)는 시스템 장애(Crash) 후 불일치해진 메타데이터(Metadata)의 논리적 모순을 해소하여 파일 시스템의 무결성(Integrity)을 보장하는 복구 메커니즘이다.
> 2. **가치**: 데이터 블록의 중복 할당이나 연결 유실(Orphaned Inode)과 같은 치명적 오류를 수정하여 데이터 영구 손실(Permanent Data Loss)을 방지하고 스토리지 자원의 누수를 막는다.
> 3. **융합**: 대용량화에 따른 `fsck`의 장시간 복구 단점을 극복하기 위해 `저널링 파일 시스템(Journaling File System)`이나 `코피온라이트(Copy-on-Write)` 기반의 `Btrfs`/`ZFS` 등으로 진화하고 있다.

---

### Ⅰ. 개요 (Context & Background)

파일 시스템은 사용자의 데이터를 저장하기 위해 디스크 블록에 `Inode`(Index Node), `Superblock`, `Bitmap` 등의 메타데이터를 기록한다. 이 메타데이터 갱신은 원자적(Atomic)이지 않으며, 여러 디스크 쓰기 작업으로 분리된다. 만약 이 과정에서 정전(Power Failure)이나 시스템 패닉(System Panic)이 발생하면, "파일은 생성되었는데 디렉터리 엔트리는 없다"거나 "블록은 사용 중인데 할당 리스트는 비어 있다"는 식의 논리적 모순이 발생한다. 이를 **메타데이터 불일치(Metadata Inconsistency)**라 하며, 이를 수정하는 과정이 바로 `fsck`다.

#### 💡 비유
이는 회계 장부에서 '전표'와 '잔고'가 맞지 않는 상황과 유사하다. 돈은 출금됐는데 기록만 안 남았거나, 반대의 경우가 생길 수 있다.

#### 등장 배경
1.  **기존 한계**: 초기 `Unix` 시스템은 갑작스러운 정전 후 복구 시 수동으로 데이터 블록을 검사해야 했으며, 이는 막대한 시간 소모를 요했다.
2.  **혁신적 패러다임**: 자동화된 복구 도구인 `fsck`가 도입되어 부팅 과정(Boot Process) 중 자동으로 무결성을 검증하고 수정하게 되었다.
3.  **현재의 비즈니스 요구**: 테라바이트(TB)급 이상의 대용량 스토리지 환경에서는 기존 방식의 수 시간 대의 복구 시간(Downtime)은 용인될 수 없으므로, 더 빠른 무결성 검증 기술이 요구되었다.

> **📢 섹션 요약 비유**: 파일 시스템 불일치 상태는 "이체는 완료됐는데 은행 장부엔 아직 돈이 남아있어 이중으로 지급될 뻔한 '기장 오류' 상태"와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

`fsck`는 파일 시스템의 구조적 무결성을 위협하는 요소를 찾아내어 수정한다. 주로 `Inode` 테이블, 블록 비트맵(Bitmap), 디렉터리 구조를 분석한다.

#### 1. 구성 요소 및 검사 모듈 (Components)

| 요소 (Module) | 역할 (Role) | 내부 동작 (Internal Logic) | 대상 데이터 |
|:---|:---|:---|:---|
| **Superblock Checker** | 마스터 정보 검증 | 총 블록 수, `Inode` 수 등의 주요 파라미터가 물리적 용량과 일치하는지 확인 | `Superblock` |
| **Inode Scanner** | 파일 노드 무결성 | 형식 오류, 잘못된 링크 카운트(Link Count), 미할당 블록 참조 등을 탐지 | `Inode Table` |
| **Block Bitmap Walker** | 블록 할당 검증 | `Inode`가 사용 중이라고 표시한 블록이 비트맵에서도 '사용 중'인지 확인 (이중 할당 검출) | `Block Bitmap` |
| **Directory Validator** | 디렉터리 계층 구조 검증 | 디렉터리 엔트리의 `.`(현재), `..`(상위) 링크가 올바른지, 파일 이름이 `Inode`와 연결되는지 확인 | Directory Blocks |
| **Connectivity Checker** | 연결성 검증 | 루트(Root)부터 도달 가능한지 확인하고, 고립된 `Inode`(Orphan)를 발견하면 `lost+found`로 이동 | Directory Tree |

#### 2. fsck 수행 단계 다이어그램 (Pass 1~5)

아래는 전통적인 `Unix fsck`의 동작 흐름을 나타낸다.

```text
 Phase 1: Check Blocks & Inodes          Phase 2: Check Directory Structure
 ┌──────────────────────┐                ┌──────────────────────────────┐
 │ 1. Read Inode Table  │                │ 1. Scan Directory Entries    │
 │ 2. Verify Types      │                │ 2. Validate '..' and '.'    │
 │ 3. Check Block Lists │                │ 3. Build Inode -> Path Map  │
 └──────────┬───────────┘                └──────────────┬───────────────┘
            │                                          │
            ▼                                          ▼
 Phase 3: Check Connectivity          Phase 4: Check Reference Counts
 ┌──────────────────────┐                ┌──────────────────────────────┐
 │ 1. Build Parent Map  │                │ 1. Compare Link Count(Table) │
 │ 2. Find Unreferenced │                │    vs Actual Links(Dir)     │
 │    Inodes (Orphans)  │                │ 2. Correct Mismatches        │
 └──────────┬───────────┘                └──────────────┬───────────────┘
            │                                          │
            └────────────────┬─────────────────────────┘
                             ▼
                   Phase 5: Check Block & Inode Bitmaps
                   ┌──────────────────────────────────────┐
                   │ 1. Mark Used Blocks based on Inodes   │
                   │ 2. Compare against Bitmap             │
                   │ 3. Allocate Free Blocks if Mismatched │
                   └──────────────────────────────────────┘
```

#### 3. 심층 동작 원리 및 복구 알고리즘

1.  **블록 및 Inode 스캔 (Pass 1)**: 모든 `Inode`를 순회하며 잘못된 형식(Form)을 가진 노드를 제거하고, 데이터 블록 포인터가 파일 시스템 범위를 벗어났는지 확인한다. 만약 `Inode`가 '할당됨'으로 표시되어 있으나 데이터 블록이 없는 경우(Corrupted), 이를 '삭제' 처리하거나 크기를 0으로 수정한다.
2.  **경로명 복원 및 고립 파일 처리 (Pass 2~3)**: 디렉터리 트리를 순회하며 부모-자식 관계를 검증한다. 어떤 `Inode`는 데이터는 존재하지만 디렉터리 엔트리(파일 이름)에서 참조되지 않는 경우가 있다. 이를 **고립 파일(Orphaned Inode)**이라 하며, 복구를 위해 `/lost+found` 디렉터리 아래에 `Inode` 번호로 이름을 생성하여 이동시킨다.
3.  **참조 횟수 불일치 수정 (Pass 4)**: `Inode`에 저장된 링크 카운트(Link Count)와 실제 디렉터리 엔트리의 개수가 다른 경우, 실제 개수에 맞춰 카운트를 수정한다. (실제가 0인데 카운트가 1 이상이면 삭제, 실제가 2인데 1이면 증가)
4.  **비트맵 동기화 (Pass 5)**: `Inode`가 가리키는 블록을 모두 집계하여 실제 사용 중인 블록 맵을 만들고, 이를 `Block Bitmap`과 비교한다. `Bitmap`은 0(비어있음)인데 실제로는 어떤 파일이 사용하고 있다면 `Bitmap`을 1(사용중)로 수정한다.

> **📢 섹션 요약 비유**: `fsck`의 작동은 "도서관이 폐관 후 서가와 대출 장부를 일일이 대조하여, 장부엔 있는데 서가에 없는 책을 찾고, 서가에 있는데 장부에 없는 책을 새로 등록하는 전수 조사 작업"과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

`fsck`는 필수적이지만 비용이 큰 작업이다. 이를 극복하기 위한 기술과 비교 분석이 필요하다.

#### 1. 심층 기술 비교: fsck vs 저널링 (Journaling)

| 구분 | Consistency Check (fsck) | Journaling File System (JFS, Ext4, XFS) |
|:---|:---|:---|
| **복구 방식** | **사후 검증 (Post-hoc Verification)**: 전체 메타데이터를 스캔하여 오류를 찾음 | **로그 재실행 (Log Replay)**: 트랜잭션 로그(Journal)만 확인하여 변경사항만 재반영 |
| **복구 소요 시간** | 디스크 용량에 비례 (TB 단위에서 수 시간) | 로그 크기에 비례 (수 초~수 분, 일정함) |
| **일관성 모델** | `fsck` 실행 완료 시점까지 파일 시스템 불안정 상태일 수 있음 | 시스템 붕괴 직전의 상태로 신속히 복원 |
| **성능 오버헤드** | 부팅 시에만 발생하지만, 다운타임(Downtime)이 큼 | 쓰기 작업 시 메타데이터를 로그에 2번 기록하므로 쓰기 성능 저하 발생 |
| **안전성** | 복구 과정에서 실수로 데이터를 삭제할 위험(Miscorrection) 존재 | 로그가 손상되지 않았다면 데이터 보장이 확실함 |

#### 2. 과목 융합 관점 (OS & Database)

*   **OS (운영체제) - LRU & Buffer Cache**: `fsck`의 성능은 디스크 I/O에 의존한다. 많은 OS는 메타데이터를 캐싱하기 위해 `LRU (Least Recently Used)` 알고리즘을 사용하는 버퍼 캐시를 활용하지만, 정전 시 캐시 데이터는 사라지기 때문에 불일치가 발생한다. `fsck`는 이런 캐싱 전략의 휘발성을 보완하는 안전장치다.
*   **Database (데이터베이스) - WAL (Write-Ahead Logging)**: 파일 시스템의 저널링과 `fsck`의 관계는 DB에서의 ` WAL(선작성 로그)`와 백업/복구의 관계와 유사하다. DB는 트랜잭션 로그를 통해 커밋되지 않은 트랜잭션을 롤백하거나, 커밋된 내용을 다시 실행(Redo)하여 데이터 무결성을 유지한다. `fsck`는 로그가 없는 환경에서의 '일관성 점검' 담당이다.

> **📢 섹션 요약 비유**: 저널링 도입은 "매일 장부를 다시 검산하는 대신, 거래가 발생할 때마다 카드 결제 영수증(로그)을 보관해두었다가 사고 시 영수증만 확인하여 장부를 수정하는 방식"으로 변화한 것과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무 환경에서 `fsck` 실행은 신중해야 하며, 잘못된 판단은 데이터 손실로 이어진다.

#### 1. 실무 시나리오 및 의사결정 매트릭스

| 상황 (Scenario) | 증상 | 의사결정 (Decision) |
|:---|:---|:---|
| **비정상 종료 후 부팅** | OS가 "Filesystem contains errors, check forced" 메시지 출력 | **자동 수행 허용**: 단일 사용자 모드(Single User Mode)로 진입하여 자동 복구 진행 |
| **서버 운용 중 데이터 접근 불가** | 특정 파일 접근 시 `I/O Error`, 시스템 느려짐 | **즉시 Umount 후 수동 실행**: 데이터를 더 망가뜨리기 전에 파일 시스템을 마운트 해제하고 읽기 전용(Read-only) 모드로 진단 |
| **fsck 도중 "Fix?" 질의** | 수동 모드 시 시스템이 수정 여부를 묻는 경우 | **사전 데이터 백업 확인**: 가능하면 이미지(Image)를 백업 받고 진행. 응답은 보통 'Yes'하나, 위험도 판단 필요 |

#### 2. 도입 체크리스트 (Checklist)

*   **기술적 사항**:
    *   [ ] 파일 시스템이 `umount` 되었는지 확인? (마운트된 상태에서 실행은 데이터 파손의 주원인)
    *   [ ] `Read-only` 모드로 먼저 실행하여 오류 범위를 파악했는가? (`fsck -n`)
    *   [ ] 복구에 필요한 메모리(RAM)가 충분한가? (대규모 파일 시스템 검사 시 메모리 부족으로 프로세스가 죽을 수 있음)
*   **운영·보안적 사항**:
    *   [ ] 작업 전 `dd` 등을 이용한 디스크 이미지 백업을 생성했는가?
    *   [ ] 복구 과정에서 발생하는 로그를 별도로 저장했는가? (`fsck -C` 옵션 등 활용)
    *   [ ] 복구 후 시스템 무결성을 재검증했는가?

#### 3. 안티패턴 (Anti-Pattern)

*   **마운트된 볼륨의 강제 fsck**: 리눅스 커널은 디스크 캐시를 관리하므로, 마운트된 상태에서 `fsck`를 실행하면 캐시되지 않은 데이터와 `fsck`가 읽은 데이터 사이에 충돌이 발생해 **파일 시스템이 완전히 망가질 수 있다.**
*   **부팅 직후 무작정 수리(RAID) 중지**: RAID 배열의 복구(Rebuild) 중 `fsck`를 실행하면 성능 저하뿐만 아니라 복구 불가 상태로 만들 수 있다. RAID 복구가 끝난 후 `fsck`를 실행해야 한다.

> **📢 섹션 요약 비유**: 운영 중인 파일 시스템에 `fsck`를 강제로 실행하는 것은 "자동차가 달리는 중에 엔진 오일을 교체하려고 후드를 여는 행위"와 같아서, 반드시 시동을 끄고(Umount) 들어가야 차량(데이터)이 망가지지 않습니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)
