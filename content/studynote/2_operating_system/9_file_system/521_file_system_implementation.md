## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파일 시스템 구현(File-System Implementation)은 저장 장치의 물리적 블록(Block)을 논리적인 파일과 디렉터리 구조로 매핑하고 관리하기 위한 하드웨어와 소프트웨어의 계층적 결합이다.
> 2. **가치**: 효율적인 데이터 배치, 신속한 접근 속도, 안정적인 메타데이터 관리 및 디스크 공간의 최적 활용을 통해 운영체제(OS, Operating System)의 저장 서비스 품질을 결정한다.
> 3. **융합**: 현대 운영체제는 온디스크(On-disk) 구조와 인메모리(In-memory) 구조를 병행하여 성능을 극대화하며, 계층 구조를 통해 상위 애플리케이션에 일관된 인터페이스를 제공한다.

---

## Ⅰ. 파일 시스템의 계층적 구조 (Layered File System)

운영체제는 복잡성을 줄이기 위해 파일 시스템을 여러 계층(Layer)으로 나눈다.

1. **응용 프로그램 (Application Programs)**: 파일 접근 요청(열기, 읽기 등)을 시작한다.
2. **논리적 파일 시스템 (Logical File System)**: 메타데이터(Metadata) 및 파일 제어 블록(FCB, File Control Block)을 관리하며, 파일 이름을 파일 번호로 변환한다.
3. **파일 구성 모듈 (File-Organization Module)**: 파일의 논리적 블록 번호를 물리적 디스크 블록 번호로 변환(LBA, Logical Block Address)하고 자유 공간 관리를 수행한다.
4. **기본 파일 시스템 (Basic File System)**: 드라이버에게 특정 물리적 블록을 읽고 쓰도록 명령을 내린다.
5. **입출력 제어 (I/O Control)**: 장치 드라이버(Device Driver)와 인터럽트 핸들러를 포함하며, 메모리와 디스크 간의 실제 전송을 담당한다.

📢 **섹션 요약 비유**: 파일 시스템의 계층 구조는 "회사 조직도"와 같다. 사장님(응용 프로그램)이 지시를 내리면 각 부서(계층)를 거쳐 실제 현장 직원(장치 드라이버)이 물건(데이터)을 옮기는 것과 동일하다.

## Ⅱ. 온디스크 및 인메모리 구조 (On-disk & In-memory Structures)

파일 시스템 구현을 위해 디스크와 메모리 모두에 특정 데이터 구조가 유지되어야 한다.

### 1. 온디스크 구조 (On-disk Structures)
- **부트 제어 블록 (Boot Control Block)**: 시스템 부팅에 필요한 정보가 담긴 블록.
- **볼륨 제어 블록 (Volume Control Block)**: 파티션 정보, 블록 수, 자유 블록 포인터 등을 포함 (UFS의 수퍼 블록).
- **디렉터리 구조 (Directory Structure)**: 파일 이름을 식별자(inode 등)로 연결하는 구조.
- **파일 제어 블록 (FCB, File Control Block)**: 개별 파일의 상세 정보(권한, 소유주, 크기, 위치 등)를 저장.

### 2. 인메모리 구조 (In-memory Structures)
- **인메모리 마운트 테이블 (In-memory Mount Table)**: 마운트된 파티션 정보 관리.
- **인메모리 디렉터리 구조**: 최근 접근한 디렉터리 정보 캐싱.
- **시스템 전체 오픈 파일 테이블 (System-wide Open-file Table)**: 모든 프로세스가 연 파일의 FCB 복사본 유지.
- **프로세스별 오픈 파일 테이블 (Per-process Open-file Table)**: 프로세스가 연 파일의 포인터 및 상태 정보 관리.

### 3. 파일 시스템 동작 개념도 (ASCII Diagram)

```text
 [ User Space ]           [ Kernel Memory ]             [ Storage Device ]
 ┌────────────┐        ┌────────────────────────┐      ┌────────────────────┐
 │ Process A  │        │ Per-process File Table │      │ Boot Control Block │
 │ fd: 3 ─────┼───────▶│ [fd 3] ──▶ System Table│      ├────────────────────┤
 └────────────┘        └───────────┬────────────┘      │ Volume Ctl (Super) │
                                   │                   ├────────────────────┤
                                   ▼                   │  Directory Blocks  │
                       ┌────────────────────────┐      ├────────────────────┤
                       │ System-wide File Table │      │     inode Table    │
                       │ [Entry] ──▶ FCB/inode  │──────┤ (FCB structures)   │
                       └────────────────────────┘      ├────────────────────┤
                                                       │    Data Blocks     │
                                                       └────────────────────┘
```

📢 **섹션 요약 비유**: 온디스크 구조가 "도서관의 실제 서가와 책"이라면, 인메모리 구조는 사서가 들고 있는 "대출 현황판"과 같아서 빠른 처리를 돕는다.

## Ⅲ. 가상 파일 시스템 (VFS, Virtual File System)

현대 OS는 다양한 파일 시스템(NTFS, ext4, NFS 등)을 동시에 지원하기 위해 VFS 계층을 사용한다.

- **추상화 계층**: 서로 다른 파일 시스템의 상세 구현을 숨기고 응용 프로그램에 표준 `read()`, `write()` 인터페이스를 제공한다.
- **데이터 구조**:
    - **vnode**: 모든 파일 시스템 객체를 나타내는 객체.
    - **inode**: 개별 디스크 상의 파일 정보를 나타냄.
- **역할**: 로컬 저장소뿐만 아니라 네트워크 파일 시스템(NFS)까지 일관된 방식으로 다룰 수 있게 한다.

📢 **섹션 요약 비유**: VFS는 "통번역기"와 같다. 상대방이 어떤 언어(파일 시스템 종류)를 쓰든 상관없이 나는 한국어(표준 API)로 대화할 수 있게 해준다.

## Ⅳ. 디렉터리 구현의 핵심 요소 (Directory Implementation)

디렉터리 구현 시 성능과 효율성을 위해 다음 사항이 고려된다.

- **선형 리스트 (Linear List)**: 파일 이름과 데이터 블록 포인터의 단순 목록. 구현이 쉽지만 검색 속도가 O(N)으로 느리다.
- **해시 테이블 (Hash Table)**: 해시 함수를 이용해 파일 이름에서 포인터를 직접 계산. 검색 속도가 O(1)로 매우 빠르지만 해시 충돌(Collision) 관리가 필요하다.
- **B-Tree/B+Tree**: 현대적인 대규모 파일 시스템(XFS, ZFS 등)에서 빠른 검색과 삽입을 위해 사용된다.

📢 **섹션 요약 비유**: 디렉터리 구현은 "주소록"과 같다. 처음부터 끝까지 넘겨보는 것(선형)과 'ㄱㄴㄷ' 색인을 이용하는 것(해시/트리)의 차이다.

## Ⅴ. 파일 시스템 구현의 성능 최적화 (Performance)

- **디스크 캐싱 (Disk Caching)**: 자주 사용되는 블록을 메인 메모리에 유지하여 I/O 횟수를 줄인다.
- **비동기 쓰기 (Asynchronous Writing)**: 즉시 디스크에 쓰지 않고 메모리에 모았다가 한꺼번에 저장(Lazy Write).
- **미리 읽기 (Read-ahead)**: 순차 접근이 예상될 때 다음 블록을 미리 메모리에 로드한다.
- **저널링 (Journaling)**: 파일 시스템 변경 사항을 로그(Log)에 먼저 기록하여 시스템 장애 시 복구 속도를 높인다.

📢 **섹션 요약 비유**: 성능 최적화는 "요리사의 미리 준비하기"와 같다. 손님이 주문할 때마다 재료를 사러 가는 대신, 미리 손질해서 냉장고(캐시)에 넣어두는 것이다.
