# 521. 파일 시스템 구현 (File-System Implementation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파일 시스템 구현은 추상적인 논리 파일(Logical File)을 물리적 저장 매체의 섹터(Sector)나 블록(Block)에 매핑하고, 계층적인 소프트웨어 아키텍처를 통해 관리하는 운영체제의 핵심 메커니즘이다.
> 2. **가치**: 계층화(Layering)와 캐싱(Caching), 버퍼링(Buffering)을 통해 디스크 I/O (Input/Output) 병목을 최소화하여 데이터 무결성을 보장하면서도 극대화된 처리량(Throughput)과 낮은 지연 시간(Latency)을 제공한다.
> 3. **융합**: 가상 파일 시스템(VFS, Virtual File System)을 통해 이기종 저장소(Local Disk, Network Storage)를 통합 관리하며, 저널링(Journaling) 기술과 결합하여 데이터베이스(DBMS) 및 트랜잭션 시스템의 ACID 성능을 지원한다.

---

## Ⅰ. 개요 (Context & Background)

파일 시스템(File System)은 운영체제(OS, Operating System)가 저장 장치(Storage Device)의 데이터를 효율적으로 관리하고 접근할 수 있도록 하는 추상화 계층(Abstraction Layer)이다. 단순히 데이터를 0과 1로 저장하는 것을 넘어, 사용자에게 논리적인 관점(Naming, Protection, Access Method)을 제공하여 하드웨어(HDD, SSD)의 물리적 특성(섹터 크기, 회전 지연, 회전 대기 시간 등)을 은폐한다.

초기의 파일 시스템은 단순한 연속 할당(Contiguous Allocation) 위주였으나, 저장 매체의 용량이 급증하고 다중 사용자 및 다중 태스킹 환경(Multi-user/Multi-tasking Environment)이 도입됨에 따라 외부 단편화(External Fragmentation) 문제를 해결하고 동시성 제어(Concurrency Control)를 지원하는 복잡한 구조로 진화했다. 현대의 파일 시스템은 온디스크(On-disk) 영속성 구조와 인메모리(In-memory) 성능 최적화 구조가 밀접하게 결합된 하이브리드 형태를 띠며, 데이터 손실을 방지하기 위한 로깅(Logging) 기술이 필수적인 구성 요소가 되었다.

📢 **섹션 요약 비유**: 파일 시스템 구현은 "도서관의 분류 및 관리 시스템"과 같다. 독자(사용자/프로세스)는 분류 번호(파일 경로)를 통해 책을 요청하면, 사서(OS)는 책이 실제로 어느 서가(디스크 블록)에 꽂혀 있는지 찾아주고, 대출 테이블(메모리 캐시)을 통해 다음 독자를 위해 바로 꺼낼 수 있도록 정리해두는 시스템이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

파일 시스템 구현은 복잡성을 관리하기 위해 **계층적 설계(Layered Design)**를 따른다. 상위 계층은 하위 계층의 세부 사항(물리적 주소, 디바이스 드라이버 로직 등)을 알 필요 없이 인터페이스를 호출하여 관심사를 분리(Separation of Concerns)한다.

### 1. 구성 요소 및 계층 (Table)

| 계층 (Layer) | 모듈 명칭 (Module) | 핵심 역할 | 내부 동작 및 프로토콜 | 비유 |
|:---:|:---|:---|:---|:---|
| **L5** | **응용 프로그램 (Application)** | 파일 사용자 | `open()`, `read()`, `write()` 시스템 콜(System Call) 호출 | 주문 고객 |
| **L4** | **논리 파일 시스템 (Logical File System)** | 메타데이터 관리 | 파일명 → FCB(File Control Block) 매핑, 권한 검사(Permission Check) | 도서 검색 시스템 |
| **L3** | **파일 구성 모듈 (File-Organization Module)** | 주소 변환 (Address Translation) | 논리적 블록(Logical Block) → 물리적 블록(Physical Block) 변환 (연결/인덱스) | 서가 번호 계산 |
| **L2** | **기본 파일 시스템 (Basic File System)** | 명령 전달 | 디스크 상의 블록 번호(Physical Block Number)로 I/O 요청 전송 | 책꽂이 로봇 팔 |
| **L1** | **I/O 제어 (I/O Control)** | 장치 제어 | 디바이스 드라이버(Device Driver), 인터럽트 처리(Interrupt Handling), DMA 제어 | 컨베이어 벨트 모터 |

### 2. 계층별 데이터 흐름 및 아키텍처 (ASCII)

파일을 읽을 때의 요청 흐름은 다음과 같이 상위에서 하위로 전달되며, 각 계층을 거치며 데이터가 가공되고 주소가 변환된다.

```text
+-----------------------------------------------------------------------+
|                    User Space (Ring 3)                                 |
|  +------------------+       ▲ 3. User Buffer (논리적 데이터 전달)      |
|  | Application      |───────┤                                           |
|  +------------------+       │                                           |
|           |                 │                                           |
|           | read("A.txt")   │                                           |
|           ▼                 │                                           |
|=======================================================================|
|                    Kernel Space (Ring 0)                               |
|  +------------------+       │                                           |
|  | Logical File Sys |───┐   │  (Metadata Lookup, Permission Check)      |
|  +------------------+   │   │                                           |
|           |             │   │                                           |
|           | (Name→FCB)   │   │                                           |
|           ▼             │   │                                           |
|  +------------------+   │   │                                           |
|  | File-Organ. Mod. |───┘   │  (Logical → Physical Block Mapping)       |
|  | (inode 처리)     │       │                                           |
|  +------------------+       │                                           |
|           |                 │                                           |
|           | (Calc PBN)      │                                           |
|           ▼                 │                                           |
|  +------------------+       │                                           |
|  | Basic File Sys.  |───┐   │  (Issue Read Command to Driver)           |
|  +------------------+   │   │                                           |
|           |             │   │                                           |
|           | (Cmd/Args)   │   │                                           |
|           ▼             │   │                                           |
|  +------------------+   │   │                                           |
|  | I/O Control      |───┘   │  (Device Registers, IRQ, DMA Setup)      |
|  | (Driver)         │       │                                           |
|  +------------------+       │                                           |
|           |                 │                                           |
|           | (DMA Transfer)  ▼                                           |
|           ▼        +----------------------+                             |
|  +----------------------+      | Physical Disk (HDD/SSD) |             |
|  | Disk Blocks (Data)   |◀─────| [Sector 0...N]        |             |
|  +----------------------+      +----------------------+             |
+-----------------------------------------------------------------------+
```

**[해설]**
1.  **응용 계층**: 사용자가 `read()` 시스템 콜을 호출하면 트랩(Trap)에 의해 커널 모드(Kernel Mode)로 전환된다.
2.  **논리 파일 시스템(LFS)**: 요청받은 파일명으로 메타데이터(Metadata)를 검색한다. 이때 FCB(File Control Block)나 유닉스 계열의 **inode (Index Node)**를 메모리로 로드하여 파일의 소유자, 크기, 권한을 검증한다.
3.  **파일 구성 모듈**: 파일의 현재 오프셋(Offset, 예: 1024 bytes)을 바탕으로 논리 블록 번호(LBN, 예: Block #2)를 계산한다. 이후 할당 방식(연속, 연결, 인덱스)에 따라 LBN을 물리적 블록 번호(PBN)나 **LBA (Logical Block Addressing)** 값으로 변환한다.
4.  **기본 I/O 및 드라이버**: 변환된 주소 정보를 디스크 컨트롤러에 명령어로 전송하고, **DMA (Direct Memory Access)**를 설정하여 CPU 개입 없이 디스크 데이터를 메인 메모리의 커널 버퍼로 직접 전송한다.

### 3. 온디스크(In-disk) vs 인메모리(In-memory) 구조

파일 시스템의 성능 핵심은 느리지만 영구적인 **디스크 영속성(Durability)**과 빠르지만 휘발성인 **메모리 속도(Volatility)**의 조화에 있다.

#### A. 온디스크 구조 (Persistent Storage)
파일 시스템은 파티션(Partition) 단위로 포맷되며, 다음과 같은 블록들로 구성된다.
*   **부트 제어 블록 (Boot Control Block)**: 시스템 시작 시 부팅에 필요한 정보를 포함하며, 파티션의 첫 번째 섹터에 위치한다.
*   **파티션 제어 블록 (Partition Control Block)**: 파티션의 세부 정보(블록 크기, 총 블록 수, 자유 공간 포인터)를 포함한다. ext4의 Superblock, NTFS의 Master File Table이 이에 해당한다.
*   **파일 시스템 구조**:
    *   디렉터리 구조: 파일명과 **inode** 번호의 맵핑 테이블.
    *   FCB (File Control Block): 파일의 소유자, 크기, 권한, 데이터 블록 위치 등을 포함. Unix에서는 inode와 구분하여 사용하기도 함.

#### B. 인메모리 구조 (Volatile Cache)
자주 접근하는 메타데이터와 데이터 블록을 DRAM에 캐싱(Caching)하여 디스크 접근 횟수를 획기적으로 줄인다.
*   **마운트 테이블 (Mount Table)**: 현재 마운트된 파일 시스템들의 정보와 루트 디렉터리 위치를 저장.
*   **시스템 전역 오픈 파일 테이블 (System-wide Open-File Table)**: 열려 있는 파일들의 inode 복사본과 현재 오프셋을 관리. 여러 프로세스가 파일을 공유할 때 동기화(Synchronization)를 위해 사용됨.
*   **프로세스별 오픈 파일 테이블 (Per-Process Open-File Table)**: 시스템 테이블을 가리키는 포인터와 접근 모드(Read/Write) 등을 저장.
*   **버퍼 캐시 (Buffer Cache)**: 디스크 블록의 내용을 메모리에 저장해 두는 공간으로, 페이지 캐시(Page Cache)와 통합되어 운영되는 경우가 많다.

**[해설]**
프로세스 A가 파일을 읽을 때, 커널은 우선 시스템 전역 오픈 파일 테이블을 확인하여 파일이 이미 열려 있는지 확인한다. 해당 파일의 inode가 메모리에 없다면 디스크에서 로드하여 Inode Cache에 올린다. 실제 데이터가 위치한 블록 번호를 알아낸 후, Buffer Cache에 해당 블록이 있는지 확인한다. 캐시 미스(Cache Miss) 발생 시에만 디스크 I/O를 발생시켜 데이터를 로드한다. 이 과정은 메모리 관리의 **페이지 폴트(Page Fault)** 처리와 유사한 메커니즘을 가진다.

📢 **섹션 요약 비유**: 파일 시스템 구조는 "대형 쇼핑몰의 재고 관리 시스템"과 같다. **온디스크 구조**는 실제 넓은 창고에 쌓여 있는 상품 박스들(Physical Blocks)이다. **인메모리 구조**는 계산대 근처에 두어 바로 계산(처리)할 수 있도록 미리 꺼내둔 바구니(Buffer Cache)이자, 현재 어떤 고객이 어떤 상품을 보고 있는지 실시간으로 추적하는 **POS 시스템(Open File Table)**이다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

파일 시스템의 성능과 기능은 단순히 운영체제의 영역에 국한되지 않고, 하드웨어(컴퓨터 구조)와 데이터베이스, 네트워크 프로토콜과
