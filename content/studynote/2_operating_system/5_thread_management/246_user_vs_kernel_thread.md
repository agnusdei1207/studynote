# 사용자 수준 스레드 (User-level) vs 커널 수준 스레드 (Kernel-level)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 두 개념은 **'스레드를 생성하고 관리(스케줄링)하는 주체가 누구인가?'**에 따른 분류다. 사용자 수준 스레드는 OS 모르게 유저 라이브러리가 몰래 관리하고, 커널 수준 스레드는 운영체제(OS)가 그 존재를 직접 인지하고 통제한다.
> 2. **가치**: 사용자 수준 스레드는 모드 전환(Mode Switch)이 없어 **문맥 교환 속도가 압도적으로 빠르지만**, 하나의 스레드가 I/O 대기(Block)에 빠지면 프로세스 전체가 멈추는 치명적 약점이 있다. 반면 커널 수준 스레드는 무겁지만 완벽한 **멀티코어 병렬성(Parallelism)**과 독립적 블로킹 처리를 보장한다.
> 3. **융합**: 현대 백엔드 아키텍처(Go 코루틴, Java 21 가상 스레드)는 이 두 모델의 장점만을 뽑아내어, 소수의 커널 스레드(SCS) 위에서 수십만 개의 사용자 스레드(PCS)를 유연하게 매핑하는 **'M:N 하이브리드 모델'**로 진화하며 C10K 문제를 돌파했다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 
  - **사용자 수준 스레드 (User-level Thread, ULT)**: 사용자 공간(User Space)에 떠 있는 런타임 라이브러리(예: Pthreads 라이브러리 초창기, Green Threads)가 스레드의 생성, 종료, 스케줄링을 모두 전담한다. OS 커널은 이들의 존재를 모른다.
  - **커널 수준 스레드 (Kernel-level Thread, KLT)**: 운영체제의 커널(Kernel Space)이 직접 TCB를 만들고 스케줄러 큐에 올려 관리한다. (예: Windows Thread, Linux NPTL).
- **필요성**: 멀티스레딩이 처음 등장했을 때, 운영체제 커널을 뜯어고쳐 스레드 기능을 넣는 것은 너무 위험하고 벅찬 일이었다. 그래서 학자들은 "커널은 건드리지 말고, 유저 앱 단에서 라이브러리로 스레드를 흉내 내보자"며 ULT를 발명했다. 하지만 멀티코어가 대중화되자, 커널이 스레드를 모르면 여러 코어에 스레드를 분산시킬 수 없다는 한계에 부딪혀 결국 커널이 스레드를 직접 쥐락펴락하는 KLT 시대로 넘어오게 되었다.
- **💡 비유**: ULT가 **'회사 내에서 부장(라이브러리)이 팀원들(스레드)에게 몰래 업무를 쪼개주는 사내 스터디'**라면, KLT는 **'국가(OS)에 정식으로 사업자 등록을 하고 4대 보험(커널 통제)을 받으며 일하는 공식 직원'**과 같다.
- **등장 배경**: 1990년대 Java 1.1은 OS 독립성을 위해 'Green Threads'라는 완벽한 ULT를 썼다. 하지만 이 방식은 I/O 블로킹 시 전체 JVM이 멈추는 극악의 버그를 낳았다. 결국 Sun Microsystems는 OS의 Native Thread(KLT)를 1:1로 매핑하는 방식으로 백기를 들었고, 이것이 수십 년간 멀티스레딩의 표준으로 굳어지게 된 역사적 배경이 있다.

```text
  [사용자 수준 스레드(ULT)와 커널 수준 스레드(KLT)의 아키텍처 가시성]

  [ 1. 사용자 수준 스레드 (ULT) - N:1 모델 ]
  (User Space)   [ T1 ] [ T2 ] [ T3 ]  ◀─ 라이브러리가 셋을 빠르게 교체(Switch)
                   │      │      │
  ─────────────────┼──────┼──────┼────────────────── (모드 전환 경계선)
                   ▼      ▼      ▼
  (Kernel Space)      [ 커널 프로세스 P1 ]  ◀─ 커널은 그냥 P1 1개만 돌고 있는 줄 안다.


  [ 2. 커널 수준 스레드 (KLT) - 1:1 모델 ]
  (User Space)   [ T1 ]         [ T2 ]         [ T3 ] 
                   │              │              │
  ─────────────────┼──────────────┼──────────────┼──── (모드 전환 경계선)
                   ▼              ▼              ▼
  (Kernel Space) [ K-Thread 1 ] [ K-Thread 2 ] [ K-Thread 3 ]
                 ◀─ 커널이 3명의 존재를 모두 알고, 각자 다른 코어에 던져줄 수 있다!
```
**[다이어그램 해설]** 가시성(Visibility)의 차이가 모든 것을 결정한다. ULT 환경에서 커널의 눈에는 오직 1개의 거대한 프로세스만 보인다. 안에서 100만 명의 유저 스레드가 피 터지게 싸워도, 커널은 그저 1명분(예: 10ms)의 타임 슬라이스만 던져준다. 반면 KLT 환경에서는 커널이 3명을 정확히 인식하므로, 각각 10ms씩 총 30ms의 타임 슬라이스를 공평하게 챙겨준다.

- **📢 섹션 요약 비유**: 영화관에서 단체석 1표(프로세스)만 끊고 들어가서 10명이 하나의 의자에 낑겨 앉아 번갈아 영화를 보는 게 ULT입니다. KLT는 10명이 각자 1장씩 정식 티켓(TCB)을 끊고 당당하게 10개의 의자(멀티코어)에 앉아 영화를 즐기는 합법적 방식입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 사용자 수준 스레드 (ULT, User-Level Thread)
- **스위칭 원리**: 스레드 A에서 B로 넘어갈 때, 시스템 콜(System Call)을 부르지 않는다. `yield()` 같은 유저 함수가 불리면, 라이브러리가 유저 메모리(Heap)에 있는 A의 레지스터를 백업하고 B를 복원한다.
- **장점**: 모드 전환(Mode Switch: 유저 ─▶ 커널 ─▶ 유저)이 아예 발생하지 않으므로, 스위칭 속도가 KLT 대비 **10배~100배 이상 미친 듯이 빠르다** (수십 나노초 컷). 또한 OS 종류(Windows, Linux)에 상관없이 돌아가는 미친 이식성을 가진다.
- **치명적 약점 (Blocking Problem)**: T1이 디스크 읽기(I/O)를 하려고 시스템 콜을 불렀다 치자. 커널은 "아, 이 프로세스가 I/O를 하는구나" 하고 프로세스 전체를 Block 상태로 잠재워 버린다. 남은 T2, T3는 연산만 하면 되는데도 T1 때문에 억울하게 같이 뻗어버리는 **'연대 책임'**을 져야 한다.

### 2. 커널 수준 스레드 (KLT, Kernel-Level Thread)
- **스위칭 원리**: 스레드가 바뀔 때마다 하드웨어 타이머 인터럽트가 걸리고 커널 모드로 진입하여 OS 스케줄러가 개입한다.
- **장점**: T1이 I/O 블로킹에 걸려도 커널은 "T1만 자고 있네. 그럼 T2, T3는 계속 실행시켜야지"라며 완벽히 분리 통제한다. 또한 멀티코어 환경에서 4개의 코어에 T1, T2, T3, T4를 뿌려주는 진정한 물리적 병렬 처리(Parallel Execution)를 지원한다.
- **단점**: 너무 무겁다. 스레드를 1만 개 만들면 커널 안에 TCB가 1만 개 생기고, 이들이 모드 스위칭을 하며 싸우느라 커널이 폭발한다 (C10K 문제).

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         스레드 블로킹(Blocking) 시나리오에 따른 시스템 생존 여부 비교 │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [ 상황: 1번 스레드가 네트워크 소켓 Read 대기 중 (1초 지연) ]        │
  │                                                             │
  │  [ ❌ 사용자 수준 스레드 (ULT) 의 비극 ]                       │
  │   - 커널 입장: "이 프로세스 전체가 Read 한다고 멈췄네. 재워라!"        │
  │   - 결과: 2, 3, 4번 스레드도 모두 얼어붙어 앱 전체가 무응답(Hang).   │
  │                                                             │
  │  [ ✅ 커널 수준 스레드 (KLT) 의 분리 방어 ]                     │
  │   - 커널 입장: "1번 스레드만 Read 하네. 1번만 재우자."              │
  │   - 결과: 2, 3, 4번 스레드는 다른 CPU 코어에서 계속 정상 서비스!      │
  └─────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: ULT는 "4인 3각 달리기"입니다. 한 명이 넘어지면 나머지 3명도 질질 끌려가며 다 같이 넘어집니다. KLT는 "400m 계주"입니다. 각자 독립적으로 뛰기 때문에 한 명이 넘어지더라도 다음 주자에게 바통을 넘기는 데는 지장이 있지만 최소한 다 같이 자빠지진 않습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 스레드 매핑 모델 (Mapping Models)의 3가지 진화

ULT와 KLT를 어떻게 연결하느냐에 따라 아키텍처의 패러다임이 결정된다.

| 모델 명칭 | 매핑 구조 | 설명 및 현대적 위상 |
|:---|:---|:---|
| **다대일 모델 (Many-to-One)** | N개의 ULT ─▶ 1개의 KLT | 초창기 스레드 (Green Thread). I/O 블로킹 시 전체 마비. 멀티코어 활용 불가. **(현재 폐기됨)** |
| **일대일 모델 (One-to-One)** | 1개의 ULT ─▶ 1개의 KLT | 생성하는 모든 스레드가 곧 커널 스레드. 병렬성 완벽. 단 스레드를 무한정 생성하면 커널 뻗음. **(현대 Java, Linux NPTL, Windows 표준)** |
| **다대다 모델 (Many-to-Many)** | N개의 ULT ─▶ M개의 KLT | 가장 완벽한 융합. 커널 스레드를 코어 수(M)만큼만 띄우고, 그 위에서 수십만 개의 유저 스레드(N)를 쪼개 돌림. **(Go 코루틴, Java 21 가상 스레드)** |

### 다대다(M:N) 모델의 압도적 효율성 (하이브리드)
현대의 일대일(1:1) 모델은 무겁다. 스레드를 10만 개 띄우면 10만 번의 커널 모드 스위칭이 일어난다.
그래서 다시 ULT의 장점을 가져온 M:N 모델이 부활했다.
- 커널 스레드(M)는 CPU 물리 코어 개수(예: 8개)만큼만 띄운다. (커널 오버헤드 최저).
- 유저 스페이스에 10만 개의 경량 스레드(N)를 만들고 런타임 환경이 이를 8개의 커널 스레드 위에 휙휙 매핑한다 (초고속).
- 만약 특정 경량 스레드가 I/O 블로킹에 걸려 커널 스레드 1개가 잠들면? 런타임이 재빨리 다른 7개의 커널 스레드로 남은 경량 스레드들을 이사시켜(Work Stealing) 시스템 정지를 막아낸다.

- **📢 섹션 요약 비유**: 1:1 모델은 10만 명의 시민에게 10만 대의 자동차(커널 스레드)를 다 사주는 겁니다(도로 마비). M:N 모델은 거대한 고속버스 8대(커널 스레드)를 굴리면서, 시민 10만 명(유저 스레드)이 정류장에서 0.1초 만에 타고 내리게 하는 궁극의 대중교통 시스템입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오
1. **Java Spring Boot의 스레드 풀 한계 (1:1 KLT의 한계)**: Spring WebMVC를 쓰면 내장 톰캣이 디폴트로 200개의 스레드 풀을 만든다.
   - **이유**: 자바의 `Thread` 객체는 OS의 KLT와 1:1로 매핑되기 때문에 너무 무겁다. 1만 개를 띄우면 메모리 10GB(1개당 1MB 스택)가 증발하고 커널 스래싱이 터진다.
   - **아키텍트의 극복 (WebFlux 도입)**: 이 KLT의 한계를 부수기 위해, 아키텍트들은 스레드를 늘리는 대신 Netty 기반의 **싱글 스레드 이벤트 루프 (WebFlux, Node.js 방식)**로 도망갔다. 무거운 커널 스레드 개입을 없애버린 것이다.
2. **Java 21의 Virtual Thread (다대다 M:N 모델의 완벽한 부활)**: 콜백 지옥(WebFlux)에 지친 자바 진영이 2023년 꺼내든 궁극의 카드다.
   - **실무 작동**: 개발자는 예전처럼 `new Thread()`를 100만 번 때린다 (동기식 코딩).
   - **아키텍처 혁명**: 하지만 이 스레드는 무거운 KLT가 아니라, JVM 유저 스페이스 힙(Heap)에 생성되는 2KB짜리 초경량 **가상 스레드 (Virtual Thread, 즉 ULT)**다. 이 100만 개의 가상 스레드는 뒤편에서 소수의 ForkJoinPool(KLT) 위를 널뛰기하며(M:N) 완벽한 I/O 논블로킹을 수행한다. KLT의 무거움과 이벤트 루프의 코드 더러움을 동시에 파괴한 21세기 아키텍처의 승리다.

```text
  ┌────────────────────────────────────────────────────────┐
  │     백엔드 동시성 아키텍처 선택 의사결정 트리 (2026년 기준 실무) │
  ├────────────────────────────────────────────────────────┤
  │                                                        │
  │   [요구사항: 초당 100만 유저가 접속하는 블록체인 거래소 백엔드]    │
  │                │                                       │
  │                ▼ 기존 1:1 커널 스레드(Java/C++)로 가능한가?  │
  │          ├─ [예] ─▶ 무조건 OOM 사망 (서버 대당 1만 접속 한계)│
  │          │                                             │
  │          └─ [아니오 (현대적 M:N 아키텍처 도입 필수)]         │
  │                 │                                      │
  │                 ▼ 언어 및 프레임워크 선택                  │
  │          ├─ [ Go 언어 ] ─▶ Goroutine 채택 (Built-in M:N)│
  │          │             (가장 성숙하고 빠르며 문법이 쉬움)     │
  │          │                                             │
  │          ├─ [ Java 21+ ] ─▶ Virtual Thread 적용         │
  │          │             (기존 레거시 코드를 그대로 쓰며 성능 폭발)│
  │          │                                             │
  │          └─ [ Rust ] ─▶ Tokio 비동기 런타임 사용         │
  │             (러닝 커브는 지옥이나, 메모리 안전성과 속도의 끝판왕) │
  └────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** 유저 스레드(ULT) 기술은 90년대에 실패한 뒤 20년간 죽어있었다가, 클라우드 C10K 문제(초대형 동시 접속)라는 벽을 만나며 코루틴(Coroutine), 고루틴, 가상 스레드라는 세련된 이름표를 달고 백엔드 생태계를 완전히 집어삼켰다. 최상위 아키텍트는 "커널 스레드는 너무 비싸서 쓰면 안 된다"는 패러다임 전환을 뼈저리게 인식해야 한다.

- **📢 섹션 요약 비유**: 옛날엔 편지(작업)를 보낼 때마다 무조건 국가 공무원(커널 스레드)을 1명씩 배정해서 보냈습니다(1:1 모델, 엄청난 세금 낭비). 지금은 택배 회사(언어 런타임)를 하나 차려서 민간 배달원(가상 스레드) 100만 명을 고용해 싸고 빠르게(M:N 모델) 편지를 돌리는 민영화의 시대가 도래했습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
사용자 수준 스레드의 가벼움(문맥 교환 오버헤드 Zero)과 커널 수준 스레드의 강력함(I/O 논블로킹 및 멀티코어 병렬성)을 결합한 M:N 모델을 아키텍처에 채택하면, 기존의 무거운 스레드 풀 튜닝이나 끔찍한 비동기 콜백 코드 없이도 서버 1대당 **수백만 건의 동시 연결(Concurrency)**을 가뿐히 처리해 내는 압도적인 인프라 비용 절감 효과를 거둘 수 있다.

### 결론 및 미래 전망
사용자 스레드(ULT)와 커널 스레드(KLT)의 대결은 "누가 더 우월한가"의 싸움이 아니라, "시스템의 어느 계층(Layer)에서 스케줄링을 책임질 것인가"에 대한 역할 분담의 역사다.
운영체제(Linux)는 철저하게 KLT(1:1 모델)를 유지하여 시스템의 하드웨어(CPU 코어) 통제력과 안정성을 극한으로 방어하는 기조를 영원히 유지할 것이다. 반면, 그 위에서 도는 애플리케이션 프레임워크(Go, JVM, Node.js)는 커널의 멱살잡이(System Call)를 피해 도망치며, 자신만의 거대한 ULT 생태계를 구축하여 유저 스페이스 내에서 극강의 가벼운 동시성을 창조해 내는 **'스케줄러의 이원화(Two-Tier Scheduling)'** 체계로 미래 컴퓨터 공학의 표준이 완전히 굳어졌다.

- **📢 섹션 요약 비유**: 국방과 외교(커널 스레드)는 무겁고 깐깐한 중앙 정부(OS)가 1:1로 확실하게 통제합니다. 하지만 동네 쓰레기 수거나 골목길 교통정리(유저 스레드)는 중앙 정부에 결재를 올리지 않고, 동네 이장님(언어 런타임)이 알아서 0.1초 만에 휙휙 해결하는 완벽한 지방자치제도의 완성입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **문맥 교환 (Context Switch)** | 유저 스레드가 커널 스레드를 이기는 단 하나의 압도적 무기. (Mode Switch 여부의 차이) |
| **I/O 블로킹 (Blocking I/O)** | 초창기 유저 스레드(다대일) 아키텍처의 숨통을 단숨에 끊어버린 치명적 독약이다. |
| **고루틴 (Goroutine)** | Go 언어가 유저 스레드(M:N) 모델을 현대적으로 완벽하게 부활시켜 클라우드를 제패한 1등 공신이다. |
| **시스템 콜 (System Call)** | 유저 영역에서 커널 영역으로 넘어가는 무거운 관문으로, 유저 스레드는 이 관문을 통과하는 비용을 증오한다. |
| **다대다 모델 (Many-to-Many)** | ULT와 KLT의 장점만 융합하여, 적은 수의 커널 스레드 위에 수만 개의 유저 스레드를 매핑하는 궁극의 현대 아키텍처다. |
