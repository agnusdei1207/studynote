# 스레드 풀 (Thread Pool)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스레드 풀 (Thread Pool)은 프로그램 실행 중에 스레드를 무지성으로 1개씩 생성하고 파괴하는 비용을 없애기 위해, **미리 정해진 개수의 스레드를 생성해 웅덩이(Pool)에 담아두고 재사용하는 멀티스레딩 아키텍처 패턴**이다.
> 2. **가치**: 스레드 생성/소멸로 인한 시스템 콜(System Call) 오버헤드를 제로화하고, 동시에 실행되는 스레드의 최대 개수(Max Threads)를 제한하여 **문맥 교환(Context Switch) 폭주로 인한 서버의 OOM(Out of Memory) 사망과 스래싱(Thrashing)을 원천 차단**한다.
> 3. **융합**: 들어오는 작업(Task)들은 스레드 풀 앞단의 큐(Queue, BlockingQueue)에 적재되며, 노는 스레드가 이 큐에서 작업을 꺼내어 실행하는 **생산자-소비자(Producer-Consumer) 패턴**과 완벽히 융합되어 현대 웹 서버(Tomcat)의 기본 엔진으로 동작한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 스레드라는 비싼 자원을 필요할 때마다 사서 쓰고 버리는 일회용 젓가락처럼 쓰지 않고, 식당의 은수저처럼 미리 100벌을 사두고 **'세척해서 계속 돌려쓰는(Reuse)'** 디자인 패턴이다.
- **필요성**: 웹 서버에 10만 명의 사용자가 접속할 때, 접속자 1명당 스레드 1개(`new Thread()`)를 띄우면 2가지 끔찍한 일이 발생한다. 
  1) 스레드 1개당 스택 메모리가 1MB씩 할당되므로 즉시 100GB 메모리가 증발한다 (OOM). 
  2) OS가 10만 개의 스레드를 스케줄링하느라 1초에 1,000만 번의 문맥 교환을 일으키며 CPU가 터져버린다. 시스템이 버틸 수 있는 '물리적 한계'를 강제할 방파제가 필요했다.
- **💡 비유**: 놀이공원의 **'범퍼카 개수 제한'**과 같다. 손님이 1만 명 왔다고 범퍼카를 1만 대 풀어버리면 차들이 엉켜서 1cm도 못 움직인다. 범퍼카를 딱 50대(스레드 풀)만 풀어두고, 차가 다 차면 나머지 손님은 대기줄(작업 큐)에 서게 해야만 50명이 신나게 놀고 빨리빨리 교대할 수 있다.
- **등장 배경**: 초기 멀티스레드 서버 모델인 'Thread-per-request(요청당 스레드 1개)' 모델이 C10K(1만 동시 접속) 문제 앞에서 처참하게 박살 나면서, 스레드의 무한 증식을 막고 커널 자원을 보호하기 위해 자바(Java 5의 `ExecutorService`) 등 엔터프라이즈 프레임워크의 코어 스펙으로 등장했다.

```text
  [Thread-per-request의 파국 vs Thread Pool의 질서]

  [ ❌ 과거: Thread-per-request (무한 생성) ]
  요청 1 ─▶ `new Thread()` ─▶ 작업 후 파괴
  요청 2 ─▶ `new Thread()` ─▶ 작업 후 파괴
  ...
  요청 10,000 ─▶ 🚨 스레드 1만 개 동시 생성! ─▶ 메모리 고갈(OOM) & OS 스케줄러 마비!

  [ ✅ 현대: Thread Pool (재사용 및 제어) ]
  [ 대기 큐 (Task Queue) ]         [ Thread Pool (Max 5개 제한) ]
  요청 1 ┐                           ┌─▶ [스레드 1] (요청 1 처리 중)
  요청 2 ├─▶ (일단 큐에 쌓음) ────▶ ├─▶ [스레드 2] (요청 2 처리 중)
  요청 6 ┘                           └─▶ [스레드 5] (요청 5 처리 중)
  
  ▶ 요청 6은 스레드가 반납될 때까지 큐에서 얌전히 대기(Throttling)하므로 서버가 죽지 않음!
```
**[다이어그램 해설]** 스레드 풀의 본질은 "빠르게 처리하기 위함"이 아니라 **"서버가 죽는 것을 막기 위함(방어)"**이다. 1만 개의 요청을 받아서 다 같이 느려지며 터지는 것보다, 5개씩 2,000번을 돌려서 서버를 살려내는 것이 진정한 엔지니어링이다.

- **📢 섹션 요약 비유**: 택시 회사에 전화가 100통 왔다고 무턱대고 택시 100대를 새로 사서(new Thread) 기사를 고용하면 회사는 파산합니다. 원래 있던 택시 10대(Pool)를 24시간 풀가동해서 10번씩 왕복(재사용)시키는 것이 자본주의(시스템 자원 관리)의 원리입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 스레드 풀의 3대 핵심 구성 요소

스레드 풀은 단순히 스레드만 모아놓은 게 아니라, 3가지 부품이 맞물려 돌아가는 기계다.

1. **작업 큐 (Task Queue / BlockingQueue)**
   - 외부에서 들어온 일거리(Runnable, Callable)를 담아두는 버퍼다.
   - 크기를 무한대(Unbounded)로 설정하면 큐가 터져서(OOM) 죽고, 제한(Bounded)을 두면 꽉 찼을 때 요청이 튕겨나간다(Reject).
2. **워커 스레드 집합 (Worker Threads)**
   - 큐를 감시하며 `while(true)` 루프를 도는 좀비 스레드들이다.
   - 큐에 일이 들어오면 낚아채서 처리하고, 일이 끝나면 스레드를 종료(`return`)하지 않고 **다시 `while`문 처음으로 돌아가 큐를 쳐다보며 대기(Wait)**한다.
3. **거절 정책 (Rejection Policy)**
   - 워커 스레드도 다 일하고 있고, 작업 큐마저 꽉 찼을 때 101번째로 들어온 요청을 어떻게 할 것인가?
   - 그냥 버릴지(Drop), 에러를 던질지(Abort), 아니면 요청한 놈이 직접 일하게 만들지(Caller Runs) 결정하는 최후의 방어선이다.

### 워커 스레드의 진짜 모습 (내부 코드)
스레드 풀에 담긴 스레드는 일반 스레드와 달리 **절대 죽지 않는 무한 루프**를 품고 있다.

```java
  // 스레드 풀 내부의 워커 스레드 의사 코드
  public void run() {
      while (!isShutDown) {
          try {
              // 1. 큐에서 일을 꺼낸다. (일이 없으면 여기서 Sleep 상태로 대기함!)
              Runnable task = taskQueue.take(); 
              
              // 2. 일을 실행한다.
              task.run(); 
              
              // 3. 일이 끝났지만 스레드는 종료되지 않고 다시 while 문 위로 올라감! (재사용)
          } catch (InterruptedException e) {
              // 종료 신호 시 루프 탈출
          }
      }
  }
```
**[천재성 분석]** 개발자가 스레드 풀에 던지는 건 `Thread` 객체가 아니라 `Runnable`(그냥 함수 덩어리)이다. 풀 안의 워커 스레드는 OS 커널 스레드(KLT)와 1:1로 묶인 진짜 비싼 자원이고, 이 녀석이 죽지 않고 남이 던진 `Runnable.run()`만 쏙쏙 빼서 실행해 주므로 커널 모드 스위칭 비용이 완전히 제로화된다.

- **📢 섹션 요약 비유**: 일반 스레드는 일용직입니다. 일 하나 주면 끝내고 집에 가버립니다. 워커 스레드는 정규직입니다. 아침에 출근해서(생성), 컨베이어 벨트(작업 큐) 앞에 서서 벨트가 멈출 때까지 하루 종일 지나가는 부품(Runnable)을 조립(run)하는 든든한 일꾼입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 스레드 개수(Pool Size) 설정의 마법 공식

가장 많이 받는 질문: "서버 코어가 8개인데, 스레드 풀 크기는 몇 개로 해야 할까요?"
이 질문에 대답하려면 태스크의 성격을 완벽히 파악해야 한다.

| 워크로드 성격 | 적정 스레드 풀 크기 공식 | 이유 (아키텍트의 관점) |
|:---|:---|:---|
| **CPU Bound (연산 집약적)**<br>예: 암호화, 동영상 인코딩 | **코어 수 (N) + 1** | 스레드를 코어 수 이상 늘려봐야, 남는 코어가 없어서 문맥 교환만 발생하고 속도는 오히려 박살 남. (+1은 페이지 폴트 대비용) |
| **I/O Bound (입출력 집약적)**<br>예: 웹 서버, DB 조회, API 호출 | **코어 수 (N) × (1 + I/O 대기시간 / CPU 연산시간)** | 스레드가 DB 응답을 기다리며(Sleep) 코어를 놔주기 때문에, 코어가 놀지 않게 하려면 스레드를 코어 수의 수십 배(예: 200개)로 꽉꽉 채워 넣어야 함. |

### 스레드 풀의 함정: 데드락 (Thread Pool Starvation)
스레드 풀 크기를 10개로 잡았다. 들어온 작업 A가 내부에서 또 다른 작업 B를 스레드 풀에 던지고 그 결과를 기다린다(`get()`).
- **참사**: 작업 A가 10개 동시에 들어오면, 풀의 스레드 10개가 전부 A를 잡고 B를 기다리며 Block된다. 하지만 풀에 남은 스레드가 0개라서 B는 영원히 실행되지 못한다.
- 이를 **스레드 풀 기아 상태 (Thread Pool Starvation Deadlock)**라 부르며, 부모 작업과 자식 작업을 같은 풀에서 섞어 쓸 때 터지는 최악의 안티패턴이다.

- **📢 섹션 요약 비유**: 요리사(스레드 풀)가 10명입니다. 10명 모두가 프라이팬(작업 A)을 잡고 "누가 파 좀 썰어줘(작업 B)"라고 기다립니다. 하지만 남은 요리사가 없어서 파를 썰 사람이 영원히 오지 않습니다. 메인 요리사 풀과 보조 요리사 풀을 아예 물리적으로 분리(Bulkhead)해야 이 멍청한 파업을 막을 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오
1. **Spring Boot (Tomcat)의 기본 스레드 풀 튜닝**: 자바 스프링으로 짠 서버가 부하 테스트 중 멈췄다. 
   - **원인**: 톰캣의 기본 `max-threads`는 200이다. 외부 결제 API가 5초 동안 지연되자, 스레드 200개가 순식간에 결제 API를 부르고 5초간 Sleep 상태에 빠졌다. 201번째 유저부터는 큐에 쌓이다가 서버가 터졌다.
   - **아키텍트 조치**: 단순히 스레드를 1,000개로 늘리는 건 바보짓이다. 문맥 교환에 서버가 찢어진다. 진짜 조치는 타임아웃을 1초로 줄이거나(Circuit Breaker), 결제 전용 스레드 풀을 따로 파서(Bulkhead 패턴) 결제망이 터져도 일반 상품 조회 스레드 풀은 멀쩡히 살아서 돌아가게 만드는 '격리(Isolation)' 아키텍처를 도입하는 것이다.
2. **Java의 ForkJoinPool (작업 훔치기, Work Stealing)**: 일반 스레드 풀은 큐가 1개라서 스레드 100개가 큐 1개에 접근하며 락(Lock) 경합이 터진다.
   - **혁신**: 자바 7부터 도입된 `ForkJoinPool`은 **스레드 100개마다 각자 개인용 큐 100개**를 쥐여준다. 내 큐에 있는 일은 락 없이 초고속으로 빼서 쓴다.
   - **작업 훔치기**: 만약 내 큐가 비어서 내가 놀게 되면? 잽싸게 다른 바쁜 스레드의 큐 뒤쪽으로 몰래 가서 일을 훔쳐(Steal) 온다. 이는 멀티코어 최적화의 끝판왕이자 자바 8 병렬 스트림(Parallel Stream)의 코어 엔진이다.

```text
  ┌────────────────────────────────────────────────────────┐
  │     실무 백엔드 스레드 풀(Thread Pool) OOM 방어 파라미터 튜닝 가이드│
  ├────────────────────────────────────────────────────────┤
  │                                                        │
  │   [ ❌ 주니어의 파멸적 세팅 (Executors.newCachedThreadPool) ]  │
  │     - Core: 0, Max: Integer.MAX_VALUE, Queue: 동기화 큐    │
  │     ▶ 결과: 트래픽 오면 스레드를 21억 개까지 무한 생성함 ─▶ OOM 즉사│
  │                                                        │
  │   [ ❌ 주니어의 2차 파멸 (Executors.newFixedThreadPool) ]      │
  │     - Core: 100, Max: 100, Queue: LinkedBlockingQueue(무한대)│
  │     ▶ 결과: 스레드는 100개로 방어했으나, 트래픽이 큐에 10억 개 쌓임.│
  │             결국 메모리에 객체가 꽉 차서 OOM 즉사.            │
  │                                                        │
  │   [ ✅ 시니어 아키텍트의 정답 (Custom ThreadPoolExecutor) ]   │
  │     - Core: 50, Max: 200, Queue: ArrayBlockingQueue(1000)  │
  │     - RejectionPolicy: CallerRunsPolicy (거절 시 호출자가 직접 돎)│
  │     ▶ 결과: 최대 1200개의 요청만 쥐고, 그 이상은 가차 없이 튕겨내어│
  │             서버가 어떤 폭격에도 절대 죽지 않는 텐션(Resilience) 확보!│
  └────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** 스레드 풀의 본질은 "우아하게 실패하는 법(Graceful Degradation)"을 설계하는 것이다. 큐의 크기를 무한대(Unbounded)로 두는 것은 인프라 아키텍처에서 가장 미친 짓이다. 큐가 꽉 찼을 때(1000개), OS가 터지기 전에 미리 HTTP 429(Too Many Requests) 에러를 뱉고 쳐내는 것이 트래픽 폭주 시나리오의 유일한 정답이다.

- **📢 섹션 요약 비유**: 식당에 대기 줄(Queue)을 끝없이 세우면 손님들이 도로까지 점령해서 동네 교통(서버 메모리)이 마비됩니다. 훌륭한 식당은 "대기석 딱 10명! 11명째부터는 재료 소진으로 안 받습니다!"라고 단호하게 쳐내어 식당의 퀄리티와 동네의 평화를 지킵니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
스레드 풀 아키텍처를 정확히 설계하면, 스레드 생성/소멸에 드는 막대한 커널 시스템 콜(System Call) 비용이 0%로 수렴하며, 악의적인 트래픽 폭주(DDoS) 상황에서도 서버의 CPU와 메모리 한계치를 완벽히 방어(Bounded)하여 무중단 서비스를 담보할 수 있다.

### 결론 및 미래 전망
스레드 풀은 2000년대 멀티코어 웹 서버 시대를 지탱한 가장 위대한 발명품이다. 하지만 시대가 변했다. 하나의 요청에 1개의 KLT(Kernel Thread)를 풀에서 할당해 주는 방식은 초당 100만 건의 I/O가 터지는 C10M 문제 앞에서는 문맥 교환 오버헤드 때문에 한계를 맞이했다.
미래의 패러다임은 이 무거운 커널 스레드 풀을 코어 수(예: 8개)만큼만 최소한으로 띄워놓고, 그 위에서 수백만 개의 **가상 스레드(Virtual Thread), 코루틴(Coroutine), 넷티 이벤트 루프(Netty Event Loop)**를 유저 공간(User Space)에서 직접 스위칭하는 '비동기/경량 스레드 풀' 모델로 운영체제의 역할을 완전히 추상화시키며 진화하고 있다.

- **📢 섹션 요약 비유**: 스레드 풀은 거대한 트럭(커널 스레드) 100대를 사서 돌려 쓰는 훌륭한 물류 회사였습니다. 하지만 이제는 트럭 100대를 굴리는 기름값(오버헤드)도 아까워서, 트럭은 8대만 유지하고 그 안에 수만 개의 드론(가상 스레드)을 싣고 다니며 배달하는 최첨단 무인 물류 시대로 패러다임이 바뀌고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **문맥 교환 (Context Switch)** | 스레드를 너무 많이 띄우면 폭증하는 비용. 이 비용을 물리적으로 제한하기 위해 스레드 풀(Max 제한)이 존재한다. |
| **생산자-소비자 패턴 (Producer-Consumer)** | 톰캣이 요청(생성자)을 큐에 던지고, 풀 안의 스레드(소비자)가 꺼내 먹는 스레드 풀의 본질적 작동 원리다. |
| **작업 훔치기 (Work Stealing)** | 하나의 중앙 큐에서 발생하는 락(Lock) 경합 병목을 깨기 위해, 스레드마다 큐를 찢고 서로의 일을 훔쳐 오게 만든 ForkJoinPool의 비기다. |
| **가상 스레드 (Virtual Thread)** | 커널 스레드 1MB의 무거움을 깨부수기 위해, JVM 힙에 2KB짜리 스레드를 수백만 개 띄우는 스레드 풀의 진화형 미래다. |
| **쓰로틀링 (Throttling)** | 큐와 스레드가 꽉 찼을 때, 시스템 사망을 막기 위해 101번째 요청을 가차 없이 쳐내는(Reject) 훌륭한 방패막이다. |
