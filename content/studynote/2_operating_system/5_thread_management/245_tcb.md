# TCB (Thread Control Block)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TCB (Thread Control Block)는 운영체제 커널이 프로세스 내의 개별 스레드(Thread)를 스케줄링하고 관리하기 위해 생성하는 **"스레드 전용 주민등록증이자 상태 보관소"**다.
> 2. **가치**: PCB(Process Control Block)가 가상 메모리(CR3)와 열린 파일 목록 등 무겁고 전역적인 자원을 관리한다면, TCB는 오직 **PC(Program Counter), 스택 포인터(SP), CPU 레지스터** 같은 가장 가볍고 독립적인 실행 문맥(Execution Context)만을 담고 있어 스레드 문맥 교환(Context Switch) 비용을 극단적으로 낮춘다.
> 3. **융합**: 멀티스레딩 환경에서 1개의 PCB는 여러 개의 TCB를 포인터(Linked List)로 거느리게 되며, 현대의 1:1 스레딩 모델(Linux NPTL 등)에서는 커널 스케줄러가 PCB가 아닌 이 TCB를 직접 큐(Runqueue)에 올리고 내리며 디스패치(Dispatch)를 수행한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 스레드 제어 블록(TCB)은 하나의 스레드가 생성될 때부터 종료될 때까지의 생애 주기 동안, OS 커널이 이 스레드의 현재 상태(어디까지 실행했는지, 스택은 어디 있는지)를 추적하고 백업하기 위해 커널 메모리 영역에 할당하는 데이터 구조체(C언어 struct)다.
- **필요성**: 싱글 스레드 시절에는 PCB 하나로 모든 게 해결되었다. 하지만 한 프로세스 안에서 10개의 스레드가 각자 다른 함수를 실행하려면, 10개의 각기 다른 Program Counter와 10개의 독립적인 호출 스택(Call Stack)이 필요하다. PCB 하나에 이 10개분의 데이터를 다 쑤셔 넣을 수는 없으므로, 실행 흐름(Execution Flow) 데이터만 따로 똑 떼어내어 가벼운 TCB라는 파생 구조체를 만들 필요가 생겼다.
- **💡 비유**: PCB가 10명의 직원이 함께 일하는 **'부서 전체의 예산안과 사무실 임대 계약서(무거운 공유 자원)'**라면, TCB는 각 직원이 퇴근할 때 서랍에 넣어두고 가는 **'개인용 업무 수첩과 필기구(가벼운 독립 자원)'**와 같다.
- **등장 배경**: 1980년대 후반 Mach 운영체제가 프로세스(자원 할당 단위)와 스레드(실행 단위)를 학술적으로 완벽히 분리하면서, 커널의 자료구조 역시 기존의 거대한 PCB 1개에서 "1개의 PCB + N개의 TCB" 계층 구조로 진화(Refactoring)하게 되었다.

```text
  [프로세스(PCB)와 스레드(TCB)의 계층적 포함 관계도]

  ┌───────────────── [ PCB (Process Control Block) ] ─────────────────┐
  │ - PID (프로세스 ID), 프로세스 상태                                  │
  │ - Memory Management Info (페이지 테이블 포인터, CR3)              │ ─▶ 무겁고 
  │ - Accounting Info (누적 CPU 사용량, 할당량)                        │    공유되는
  │ - Open File Table (열린 파일 식별자 목록)                          │    전역 자원
  │                                                               │
  │   ┌─── 포인터 (Linked List) ───┐                                │
  │   ▼                          ▼                                │
  │ [ TCB 1 ]                  [ TCB 2 ]                          │
  │ - TID (스레드 ID)          - TID (스레드 ID)                   │ ─▶ 가볍고
  │ - PC (명령어 주소)         - PC (명령어 주소)                   │    독립적인
  │ - Registers (문맥)         - Registers (문맥)                  │    개별 자원
  │ - Stack Pointer (SP)       - Stack Pointer (SP)               │
  │ - Thread State (Ready)     - Thread State (Running)           │
  └───────────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** TCB는 PCB에 기생하는 구조다. TCB 안에는 항상 자기를 낳아준 부모 PCB를 가리키는 포인터가 있다. 스레드 1(TCB 1)에서 스레드 2(TCB 2)로 문맥 교환이 일어날 때, OS는 무거운 PCB 정보(메모리 테이블, 열린 파일)는 건드리지 않고, 오직 TCB에 있는 레지스터 값만 슥싹 바꾼다. 이 '선택적 백업'이 스레딩 속도의 핵심이다.

- **📢 섹션 요약 비유**: 회사(PCB)가 망하면 그 회사에 소속된 직원(TCB)들은 책상이 다 빠지니까 당연히 다 같이 해고됩니다. 하지만 직원 1명(TCB)이 퇴사한다고 해서 회사(PCB) 건물이나 다른 직원들의 책상이 사라지지는 않습니다. TCB는 철저히 PCB의 우산 아래 존재합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### TCB의 핵심 구성 요소 (Inside the Struct)
운영체제마다 필드 이름은 다르지만, TCB가 필연적으로 가져야만 하는 4가지 핵심 구역이 있다.

1. **식별자 영역 (Identifiers)**
   - **TID (Thread ID)**: 시스템 내에서 스레드를 유일하게 식별하는 번호.
   - **부모 포인터**: 자신이 속한 프로세스(PCB)의 메모리 주소.
2. **CPU 문맥 영역 (CPU Context) - 가장 중요 ⭐**
   - **PC (Program Counter)**: 이 스레드가 멈췄을 때 실행하고 있던 코드의 주소. 다음에 재개할 때 여기부터 시작함.
   - **GPR (General Purpose Registers)**: AX, BX, CX 등 연산 중이던 데이터.
   - **SP (Stack Pointer)**: 이 스레드만의 독자적인 지역 변수/함수 호출 기록이 담긴 스택 메모리의 꼭대기 주소.
3. **스케줄링 영역 (Scheduling Info)**
   - **State**: New, Ready, Running, Blocked, Terminated 중 하나.
   - **Priority**: 스레드의 우선순위 (실시간 RT 스레드인지 일반 스레드인지).
   - **CPU Time**: 이 스레드가 혼자 CPU를 소모한 누적 시간.
4. **포인터 영역 (Linkers)**
   - 대기 큐(Wait Queue)나 레디 큐(Ready Queue)에 줄을 설 때 다음 TCB를 가리키는 `next` 포인터.

### 스레드 문맥 교환 (Thread Context Switch)의 물리학
인터럽트가 터지면 커널은 현재 Running 중인 스레드의 TCB를 열어젖힌다.
1. 현재 물리적 CPU의 모든 레지스터 값을 TCB 1번의 `Context` 영역에 쓴다(Save).
2. TCB 1번의 상태를 `Ready`로 바꾸고 큐의 맨 뒤로 보낸다.
3. 큐 맨 앞의 TCB 2번을 뽑아 상태를 `Running`으로 바꾼다.
4. TCB 2번에 적혀있던 `Context` 값을 물리적 CPU 레지스터에 밀어 넣는다(Restore).
5. TCB 2번의 `PC` 값이 가리키는 주소로 `JUMP` 한다. 끝! (TLB 플러시 따윈 없다).

- **📢 섹션 요약 비유**: 게임을 하다가 동생에게 패드를 넘겨줄 때, 게임기 전원을 끄고 팩을 갈아 끼우는 것(PCB 교체)이 아니라, 그냥 일시정지(Save) 누르고 세이브 파일 1번(TCB 1)을 저장한 뒤, 세이브 파일 2번(TCB 2)을 불러와서(Restore) 바로 게임을 이어하는 초고속 교대 시스템입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### PCB (프로세스) vs TCB (스레드) 문맥 교환 비용의 정량 비교

왜 TCB를 쓰는 게 그토록 빠르다고 찬양할까? 메모리 아키텍처 관점에서 파고들어 보자.

| 백업/복원 항목 | 프로세스 문맥 교환 (PCB Switch) | 스레드 문맥 교환 (TCB Switch) |
|:---|:---|:---|
| **CPU 레지스터 저장/복원** | ⭕ (당연히 수행) | ⭕ (당연히 수행) |
| **스택 포인터 (SP) 교체** | ⭕ | ⭕ |
| **명령어 포인터 (PC) 교체** | ⭕ | ⭕ |
| **페이지 테이블 포인터(CR3) 교체**| 🚨 **반드시 수행** (가상 메모리 영토가 바뀜) | ✅ **생략** (같은 영토를 쓰므로) |
| **TLB 캐시 비우기 (Flush)** | 🚨 **발생** (기존 캐시 주소가 쓰레기가 됨) | ✅ **발생 안 함** (기존 캐시 그대로 사용) |
| **디스패치 체감 속도** | 약 2~10 마이크로초 (무거움) | **약 100 나노초 단위 (깃털 같음)** |

**[핵심 분석]** 레지스터 몇 개 백업하는 건 CPU에게 일도 아니다(수 나노초). 진짜 오버헤드는 "메모리 지도를 통째로 갈아 끼우고(CR3), 그 지도를 바탕으로 외워뒀던 지름길(TLB)을 싹 다 지워버리는 짓"이다. TCB 간의 스위칭은 이 끔찍한 '지도 갈아 끼우기' 작업을 완벽히 스킵(Skip)하기 때문에 압도적으로 빠르다.

### 유저 레벨 스레드(TCB) vs 커널 레벨 스레드(TCB)
- **유저 레벨 (Go 코루틴, 초창기 Java)**: TCB를 OS 커널이 아니라, 프로세스 내부의 유저 메모리(Heap 등)에 라이브러리가 몰래 만들어 둔다. 커널은 이 TCB의 존재를 모른다. (교체 속도가 미친 듯이 빠르지만 I/O 블로킹에 취약함).
- **커널 레벨 (현대 Linux, Windows)**: OS가 커널 공간에 직접 TCB를 만들고 관리한다. 커널 모드로 진입(System Call)해야 하므로 살짝 무겁지만, 멀티코어에 스레드를 찢어발기기(병렬성) 위해선 필수적이다.

- **📢 섹션 요약 비유**: PCB 교체는 이사 갈 때 "집 문서, 등기부등본, 전입신고"를 다 갈아엎는 행정 처리(무거움)입니다. TCB 교체는 같은 집 안에서 "형 방에 있던 일기장을 동생 방으로 옮기는 것"입니다. 주소(집)가 똑같기 때문에 주민센터(페이지 테이블)에 신고할 필요 없이 1초 만에 끝납니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오
1. **리눅스 커널의 task_struct (PCB와 TCB의 융합)**: 운영체제 교과서에서는 PCB와 TCB가 완벽히 쪼개져 있다고 가르친다. 하지만 천재 해커 리누스 토발즈(Linus Torvalds)는 리눅스를 만들 때 이 둘을 나누는 걸 거부했다.
   - **실무 아키텍처**: 리눅스 커널 소스(`sched.h`)에는 TCB라는 구조체가 없다! 오직 `task_struct`라는 하나의 만능 구조체만 존재한다. 
   - **클론(Clone)의 마법**: 리눅스는 스레드를 만들 때 `clone()` 시스템 콜을 쓴다. 이때 옵션 플래그(`CLONE_VM`, `CLONE_FILES`)를 주면, 새로 만든 `task_struct`가 부모의 메모리와 파일 테이블을 복사하지 않고 포인터만 공유(Share)하게 만든다. 
   - **결론**: 리눅스는 "메모리를 공유하는 얇은 프로세스(task_struct)"를 곧 스레드(TCB)로 취급한다. 교과서의 TCB 이론을 코드로 가장 변태적이고 효율적으로 구현한 실무적 승리다.
2. **JVM(Java Virtual Machine)의 스레드 덤프(Thread Dump) 분석**: 새벽 2시에 서버가 멈췄을 때 백엔드 개발자가 가장 먼저 하는 일이 서버에 `jstack` 명령을 날려 스레드 덤프를 뜨는 것이다.
   - **이 행위의 본질**: 이 명령은 JVM이 관리하는 모든 자바 스레드의 **유저 레벨 TCB 정보(특히 PC와 Call Stack)**를 텍스트 파일로 쭉 뽑아주는 것이다. 
   - **아키텍트 분석**: 덤프를 열어봤더니 스레드 100개의 TCB 상태(State)가 모조리 `BLOCKED`이고, 스택 트레이스(PC 위치)가 특정 DB 커넥션 `lock()` 함수에 머물러 있다? 100% 데드락이나 커넥션 풀 고갈 장애다. TCB가 없었다면 이런 내부 디버깅은 영원히 불가능했다.

```text
  ┌────────────────────────────────────────────────────────┐
  │     멀티스레드 서버 설계 시 TCB 메모리 고갈 (OOM) 방어 트리      │
  ├────────────────────────────────────────────────────────┤
  │                                                        │
  │   [요구사항: 초당 10만 접속을 처리하는 톰캣(Tomcat) 서버 세팅]   │
  │                                                        │
  │   [ 1. 스레드 1개당 소비되는 TCB 및 커널 자원 계산 ]          │
  │     ▶ TCB 구조체 자체는 작음 (수백 바이트)                    │
  │     ▶ 🚨 하지만 스레드가 생길 때마다 딸려오는 **Thread Stack**이│
  │        디폴트로 1MB (리눅스 기준) 할당됨!                    │
  │                                                        │
  │   [ 2. 무지성 스레드 생성의 파국 (Thread per Request) ]    │
  │     ▶ 1만 개 동시 접속 = 스레드 1만 개 띄움                   │
  │     ▶ 1만 개 * 1MB = 10GB! 순수 스레드 유지에만 메모리 10GB 증발.│
  │     ▶ 서버 즉각 OOM (Out Of Memory) 사망.                 │
  │                                                        │
  │   [ 3. 아키텍트의 성능 방어 로직 (Thread Pool 튜닝) ]        │
  │     ▶ max-threads 옵션을 200~500개로 캡(Cap)을 씌움.       │
  │     ▶ TCB와 Stack 메모리 점유를 500MB 선에서 완벽히 방어.       │
  │     ▶ 남는 요청은 큐(Queue)에 넣어 대기시켜 서버의 생명줄 유지. │
  └────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** "스레드는 프로세스보다 가볍다"는 교과서 문장을 맹신하는 주니어들이 가장 많이 터뜨리는 장애다. 스레드가 가벼운 건 '문맥 교환'과 '주소 공간'이지, 스택 메모리(1MB)마저 0바이트인 건 아니다. TCB 하나가 생성될 때 1MB의 스택이 함께 쪼개진다는 물리적 감각이 있어야 진정한 서버 엔지니어가 될 수 있다.

- **📢 섹션 요약 비유**: 직원(스레드)을 고용하면 회사 건물(프로세스 메모리)을 새로 지을 필요는 없지만, 최소한 그 직원이 앉을 개인 책상(TCB)과 서랍장(1MB 스택)은 사줘야 합니다. 직원을 만 명 뽑으면 건물은 멀쩡해도 책상 둘 공간이 없어서 회사가 터집니다(OOM). 무분별한 스레드 생성은 절대 금물입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
TCB 구조를 도입함으로써 운영체제는 무거운 프로세스 스위칭 비용을 최소화하고, 하나의 앱 안에서 UI 렌더링(스레드 1), 백그라운드 다운로드(스레드 2), I/O 대기(스레드 3)를 수 나노초 만에 넘나들며 처리하는 극강의 **다중 코어 병렬성(Parallelism)과 비동기 응답성(Responsiveness)**을 획득했다.

### 결론 및 미래 전망
TCB는 현대 멀티코어 운영체제(Windows, Linux, MacOS) 스케줄러가 가장 사랑하고, 가장 빈번하게 꺼내보는 커널 자료구조다. 사실상 오늘날 OS가 "스케줄링한다"는 말은 "TCB를 줄 세운다"는 말과 100% 동의어다.
미래에는 이 OS 커널 레벨의 TCB마저도 "문맥 교환 시 커널 모드로 진입하는 비용(System Call)이 아깝다"는 이유로 유저 공간(User Space)으로 격하되고 있다. Go 언어의 Goroutine이나 Java 21의 Virtual Thread는 OS의 TCB 대신, 애플리케이션 힙(Heap) 메모리에 불과 2KB짜리 초경량 가상 TCB를 수백만 개 띄워놓고 유저 레벨에서 1나노초 만에 휙휙 스위칭해 버리는 **'가상 스레드(Virtual Thread) 혁명'**으로 동시성의 한계를 또 한 번 찢어발기고 있다.

- **📢 섹션 요약 비유**: 옛날엔 신분증(TCB)을 갱신하려면 무조건 동사무소(OS 커널)에 가서 번호표를 뽑고 줄을 서야(System Call) 했습니다. 지금은 스마트폰(Virtual Thread)으로 집구석(User Space)에서 1초 만에 모바일 신분증을 갱신하고 스위칭하는 초고속 비대면 행정의 시대로 진화했습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **PCB (Process Control Block)** | TCB들의 부모. 메모리(CR3)와 파일 목록 등 무거운 전역 자원을 들고 있어 TCB가 가벼워지게 해 준 든든한 백팩이다. |
| **문맥 교환 (Context Switch)** | 인터럽트가 터졌을 때 OS가 현재 CPU의 레지스터를 TCB에 우겨넣고(Save), 다른 TCB의 데이터를 꺼내오는(Restore) 행위다. |
| **스레드 (Thread)** | TCB라는 주민등록증을 발급받아 CPU 위에서 실제로 춤을 추는 실행 단위(Entity) 그 자체다. |
| **프로그램 카운터 (PC)** | TCB 안에서 가장 중요한 변수로, 스레드가 기절했다가 깰 때 "아, 나 53번 줄 읽고 있었지!"를 기억하게 해 주는 책갈피다. |
| **가상 스레드 (Virtual Thread)** | 1MB씩 쳐먹는 무거운 OS 커널 TCB를 버리고, JVM 힙에 2KB짜리 초경량 TCB를 만들어 C10K 문제를 뚫어버린 최신 기술이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 컴퓨터라는 학교(OS)에 1반(프로세스)이 있어요. 1반에는 10명의 학생(스레드)이 같이 공부하고 있죠.
2. 1반의 "교실 열쇠나 청소 도구함(공유 자원)"은 **PCB**라는 큰 장부에 묶여서 다 같이 써요.
3. 하지만 학생 한 명 한 명이 "나는 어제 수학 30페이지까지 풀었어!"라고 개인적으로 기억하는 작은 알림장 수첩을 **TCB**라고 해요. 이 수첩 덕분에 학생들은 헷갈리지 않고 자기 공부를 착착 해나갈 수 있답니다!
