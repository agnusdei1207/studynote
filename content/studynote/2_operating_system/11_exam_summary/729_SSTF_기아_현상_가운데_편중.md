# SSTF 기아 현상 (Shortest Seek Time First Starvation)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSTF (Shortest Seek Time First)는 현재 헤드 위치에서 탐색 거리 (Seek Distance)가 가장 짧은 요청을 우선 처리하여 평균 탐색 시간을 극소화하는 그리디 (Greedy) 기반 디스크 스케줄링 알고리즘이다.
> 2. **가치**: 입출력 요청이 집중된 환경에서 단위 시간당 처리량 (Throughput)을 최대화할 수 있으나, 새로운 요청이 특정 영역에 지속적으로 유입될 경우 외곽 트랙의 요청이 무한정 대기하게 되는 기아 현상 (Starvation)과 중앙 편중 (Locality Bias) 문제를 야기한다.
> 3. **융합**: CPU 스케줄링의 SJF (Shortest Job First)와 궤를 같이하는 "최단 우선" 철학의 물리적 구현체이며, 이를 극복하기 위해 에이징 (Aging) 기법이나 SCAN 계열 알고리즘으로의 보완적 설계가 필수적이다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: SSTF 스케줄링은 매 순간 현재 헤드 위치를 기준으로 가장 가까운 트랙 번호를 가진 요청을 선택하는 방식이다. 이는 디스크 접근 시간의 80% 이상을 차지하는 탐색 시간 (Seek Time)을 줄이는 데 최적화되어 있다.

- **💡 비유**: 이는 여러 곳에서 심부름 요청이 들어왔을 때, 요청이 들어온 순서와 상관없이 무조건 "내 몸에서 가장 가까운 곳"부터 들르는 배달원과 같다. 내가 5층에 있는데 6층 손님과 20층 손님이 동시에 불렀다면 당연히 6층으로 가고, 6층 일을 마칠 때쯤 7층에서 또 부르면 20층 손님은 무시하고 7층으로 가는 식이다.

- **등장 배경**:
  1. **FCFS의 높은 오버헤드**: 단순히 선착순으로 처리하면 헤드가 디스크의 왼쪽 끝과 오른쪽 끝을 무의미하게 가로지르는 '스윙(Swing)' 현상이 심해 성능이 저하된다.
  2. **처리량 (Throughput) 극대화 요구**: 대량의 I/O가 발생하는 데이터베이스나 파일 서버에서 헤드 이동 시간을 1ms라도 더 줄여야 하는 성능 지상주의적 요구가 있었다.
  3. **그리디 알고리즘의 적용**: 전체적인 최적해를 구하기는 어려우니, 당장 눈앞의 최적(로컬 최적해)을 선택하여 전체 효율을 높이려는 시도로 도입되었다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              SSTF 스케줄링의 기아 현상 (Starvation) 메커니즘         │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [상황] 현재 헤드: 50 , 외곽 요청: 190 (매우 멀음)                │
  │                                                                 │
  │   Time T1: 요청 {48, 52, 190}  → 52 서비스 (거리 2)               │
  │   Time T2: 요청 {48, 55, 190}  → 55 서비스 (거리 3)               │
  │   Time T3: 요청 {48, 51, 190}  → 51 서비스 (거리 4)               │
  │   ... (중앙 부근에 계속 새로운 요청 유입) ...                      │
  │                                                                 │
  │   [결과]                                                        │
  │   - 헤드는 50번 근처(중앙)에서만 맴돌며 춤을 춤 (Sticky Head)      │
  │   - 190번 요청은 거리가 너무 멀어 절대 선택되지 못함               │
  │   - 이것이 전형적인 기아 현상 (Starvation)                       │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SSTF는 항상 '최단 거리'만 탐색한다. 만약 디스크 중앙 부근에 요청이 끊임없이 들어온다면, 헤드는 그 좁은 영역 안에서만 바쁘게 움직이게 된다. 반면 디스크의 양 끝단(0번이나 199번 근처)에 위치한 요청은 중앙 트랙의 요청이 모두 소진될 때까지 영원히 기다려야 하며, 이는 응답 시간의 표준 편차를 무한대로 발산시킨다.

- **📢 섹션 요약 비유**: "가까운 친구들과만 놀다 보니, 멀리 사는 친구는 영영 만날 기회가 사라지는 외톨이 현상"과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SSTF의 성능 및 문제점 상세 분석

SSTF는 '효율성'과 '공정성'이라는 두 가치가 극단적으로 충돌하는 지점에 있다.

| 분석 항목 | 상세 내용 및 메커니즘 | 기술적 영향 | 비유 |
|:---|:---|:---|:---|
| **평균 탐색 시간** | 이론적 하한치에 근접 (일부 조건 제외) | 전체 시스템 처리량(TPS) 대폭 상승 | 빠른 지름길 탐색 |
| **기아 현상** | 특정 요청이 무한히 대기하는 현상 | 응답 시간의 예측 불가능성(Jitter) 증대 | 잊혀진 손님 |
| **중앙 편중** | 디스크 통계적으로 중앙 트랙의 접근성이 높음 | 외곽 트랙의 데이터 가용성 저하 | 도심지만 우대 |
| **그리디 선택** | 현재 위치에서의 최적(Local Optima)만 추구 | 전체 이동 경로가 최적이 아닐 수 있음 | 앞만 보고 달리기 |

---

### 기아 현상의 수리적 모델: 응답 시간 분산

SSTF의 가장 큰 결함은 응답 시간의 **분산 (Variance)**이 너무 크다는 것이다.

1. **평균 대기 시간 ($E[W]$)**: 낮음. (전체적으로 빨리 처리하니까)
2. **대기 시간의 분산 ($Var[W]$)**: 매우 높음.
   - 중앙 트랙: $W \approx 0$
   - 외곽 트랙: $W \to \infty$ (기아 발생 시)
3. **SLA (Service Level Agreement) 위반**: 기업 환경에서 특정 요청이 10초 이상 지연되면 평균 속도가 아무리 빨라도 장애로 간주된다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 SSTF vs SCAN 대기 시간 확률 분포                   │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   Wait Time Probability                                          │
  │      ▲                                                           │
  │      │   SSTF (양극화)                                            │
  │      │    |                                                      │
  │      │    | (중앙 요청)                                           │
  │      │    |_________..............__________| (외곽 요청)         │
  │      │                                                           │
  │      │   SCAN (완만한 분포)                                       │
  │      │          _______                                          │
  │      │        /         \                                        │
  │      └───────/───────────\─────────────────────────▶             │
  │                                           Wait Time              │
  │                                                                  │
  │   [결론]                                                         │
  │   - SSTF는 운 좋은 요청과 불운한 요청의 차이가 극단적임.            │
  │   - SCAN은 기아를 방지하여 모든 요청에 상한선을 제공함.             │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SSTF의 대기 시간 분포는 매우 불균형하다. 대다수의 요청은 빠르게 처리되어 왼쪽에 쏠려 있지만, 일부 요청은 기아 현상으로 인해 그래프의 오른쪽 끝(무한대)까지 길게 늘어진다 (Long Tail). 이는 시스템의 안정성을 해치는 가장 큰 요인이다.

- **📢 섹션 요약 비유**: "평균 연봉은 높지만 빈부격차가 극심하여 사회적 불만(기아)이 팽배한 불평등 도시"와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: SSTF vs FCFS vs SCAN (스케줄링의 3대 패러다임)

| 비교 항목 | FCFS (선착순) | SSTF (최단거리) | SCAN (엘리베이터) |
|:---|:---|:---|:---|
| **철학** | "먼저 온 사람이 임자" | "가까운 곳부터 빨리" | "가는 길에 다 태우자" |
| **처리량** | 낮음 | **최고** | 높음 |
| **공평성** | **완벽** | 최악 (기아 발생) | 좋음 |
| **이동 패턴** | 무질서함 (Zig-Zag) | 국부 집중 (Sticky) | 질서 있는 왕복 |
| **적합 환경** | 부하가 거의 없는 시스템 | 단일 사용자, 처리량 우선 | **일반적 서버 환경 표준** |

### 비교 2: CPU 스케줄링과의 상관관계

- **SSTF ≒ SJF (Shortest Job First)**: 둘 다 실행/이동 시간이 짧은 것을 우선하여 평균 대기 시간을 줄이지만, 실행 시간이 긴 프로세스나 거리가 먼 트랙 요청은 기아 상태에 빠진다는 공통된 한계를 가진다.
- **해결책의 유사성**: SJF의 기아를 해결하기 위해 '에이징 (Aging)'을 쓰듯, 디스크 스케줄링에서도 일정 시간 대기한 요청의 우선순위를 강제로 높이거나 SCAN 방식으로 전환하는 하이브리드 전략을 사용한다.

- **과목 융합 관점 (알고리즘)**: SSTF는 전형적인 **최단 경로 (Shortest Path)** 탐색 문제의 변형이다. 하지만 다익스트라 (Dijkstra) 알고리즘이 전체 그래프의 최적을 찾는 것과 달리, SSTF는 다음 단계만 보는 그리디 방식이기에 '전역 최적해'를 보장하지는 못한다.

- **📢 섹션 요약 비유**: "순서만 지키면 느려지고(FCFS), 욕심만 부리면 소외자가 생기니(SSTF), 규칙을 가지고 골고루 살피는 것(SCAN)이 최선"이라는 중용의 미학입니다.

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 개인용 PC 디스크 최적화**: 부팅 시 수천 개의 작은 파일들을 읽어야 하는 상황. 이때는 공평성보다 속도가 우선이므로 SSTF나 그 변형인 **LOOK**이 유리할 수 있다. 하지만 현대의 SSD 환경에서는 이마저도 무의미해져서 순수 FCFS(None)로 회귀하는 추세다.

2. **시나리오 — 엔터프라이즈 DB 서버의 'Hot Spot' 문제**: 특정 인덱스 페이지가 디스크 중앙 트랙에 몰려 있어 헤드가 그곳만 무한 반복 접근하는 상황. 외곽에 저장된 로그 기록 요청이 기아 상태에 빠져 트랜잭션 커밋이 지연됨. 해결책으로 SSTF 대신 **Deadline** 스케줄러를 적용하여, 일정 시간(예: 500ms) 이상 대기한 요청은 거리와 상관없이 우선 처리하도록 강제함.

3. **시나리오 — 클라우드 가상 디스크 레이턴시 튐 (Spike)**: VM에서 가끔 I/O 레이턴시가 수 초 이상 튀는 현상 발생. 분석 결과 하이퍼바이저의 스케줄러가 SSTF 성향을 띠어 외곽 블록 요청이 기아에 빠졌던 사례. 이를 방지하기 위해 호스트 OS의 스케줄러를 **C-SCAN**이나 **BFQ (Budget Fair Queuing)**로 변경.

### 도입 체크리스트
- **기아 현상 허용 여부**: 시스템이 실시간성을 요구하거나 특정 요청 지연에 민감한가?
- **부하의 밀집도**: I/O 요청이 특정 영역에 몰리는 경향(Locality)이 강한가?
- **평균 vs 최대**: 평균 응답 시간이 중요한가, 아니면 응답 시간의 최대 편차(Variance) 제어가 중요한가?

### 안티패턴
- **서버 환경에서의 무분별한 SSTF 채택**: 처리량 수치만 보고 SSTF를 썼다가, 특정 사용자의 요청이 무한 대기하며 "먹통" 현상을 일으키는 것.
- **에이징 (Aging) 없는 최단 우선 알고리즘**: 거리가 멀다는 이유로 특정 요청을 영원히 후순위로 미루는 설계.

- **📢 섹션 요약 비유**: "단거리 육상 선수가 아무리 빨라도 마라톤(시스템 운영)에서는 페이스를 조절하지 못하면 완주할 수 없다"는 점을 명심해야 합니다.

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (SSTF의 양면성)

| 구분 | 긍정적 기대효과 (성능) | 부정적 위험요소 (안정성) | 개선 방안 |
|:---|:---|:---|:---|
| **처리량** | 탐색 시간 감소로 I/O 효율 40% 이상 향상 | 특정 상황에서 처리량 0으로 수렴 (Stall) | 부하에 따른 동적 스케줄링 |
| **사용자 경험** | 대다수 요청에 대해 즉각적인 반응 제공 | '운 나쁜' 사용자의 이탈 및 불만 폭주 | 최대 대기 시간 (Timeout) 설정 |
| **하드웨어** | 헤드의 불필요한 이동 거리 최소화 | 특정 지점 반복 이동으로 인한 국부적 마모 | 웨어 레벨링 (S/W 레벨) |

### 미래 전망
- **I/O 결정론 (Deterministic I/O)**: 기아 현상을 수학적으로 0으로 수렴시키기 위해, 요청마다 '수명(TTL)'을 부여하고 SSTF 거리와 수명을 가중치로 합산하여 다음 목표를 정하는 AI 기반 스케줄러가 연구 중임.
- **ZNS (Zoned Namespaces) SSD**: SSD에서도 논리적인 영역을 나누어 관리하므로, 각 존(Zone) 내에서의 SSTF와 존 간의 공평한 스케줄링을 결합한 새로운 계층적 아키텍처가 부상함.

### 참고 표준
- **NVMe 1.4+ / Host Managed SMR**: 호스트가 직접 헤드(또는 쓰기 포인터)의 위치를 제어하고 스케줄링할 수 있게 돕는 표준 인터페이스.
- **Linux I/O Scheduler Framework**: `mq-deadline`, `kyber` 등 기아 현상을 방지하면서도 SSTF의 효율을 뽑아내는 현대적 프레임워크.

- **📢 섹션 요약 비유**: SSTF는 효율성이라는 강력한 유혹이지만, 기아 현상이라는 독배를 품고 있으므로 "에이징"이라는 해독제와 함께 사용해야 하는 알고리즘입니다.

- **에이징 (Aging)**: 기아 현상을 해결하기 위한 공학적 표준 치료법.
- **디스크 스케줄링 SCAN**: SSTF의 기아 현상을 구조적으로 해결한 대안.
- **탐색 시간 (Seek Time)**: SSTF가 줄이고자 하는 유일한 목표 변수.

### 👶 어린이를 위한 3줄 비유 설명
1. SSTF는 **"가까운 친구랑만 놀기"** 규칙이에요. 내 바로 옆에 있는 친구랑 놀면 이동하는 시간이 짧아서 더 많이 놀 수 있거든요.
2. 그런데 옆에 친구들이 계속 새로 오면, **저 멀리서 나를 기다리는 친구**는 영영 나랑 놀지 못하고 계속 기다리다가 울어버려요. 이게 바로 "기아 현상"이에요.
3. 그래서 똑똑한 엘리베이터(SCAN)처럼, 때로는 멀리 있는 친구를 위해서라도 **한 번쯤은 끝까지 쭉 다녀오는 것**이 모두가 행복해지는 방법이랍니다!
