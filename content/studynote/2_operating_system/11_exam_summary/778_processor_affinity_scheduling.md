# 프로세서 친화성 (Processor Affinity) 스케줄링

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스가 이전에 실행되었던 **동일한 CPU 코어에서 계속 실행**되도록 우선순위를 부여하여, 해당 코어의 로컬 캐시(L1/L2)에 남아있는 데이터를 최대한 활용하게 하는 스케줄링 기법.
> 2. **가치**: 문맥 교환 시 발생하는 **캐시 미스(Cache Miss)**와 메인 메모리 재로딩 오버헤드를 획기적으로 줄여, 멀티코어 시스템의 실질 연산 성능을 극대화한다.
> 3. **융합**: 운영체제의 스케줄러 알고리즘과 하드웨어의 캐시 계층 구조가 결합된 최적화 전략이며, 고성능 연산(HPC) 및 실시간 시스템의 성능 핵심 지표가 된다.

---

### Ⅰ. 프로세서 친화성의 분류

스케줄러가 친화성을 얼마나 엄격하게 지키느냐에 따라 두 가지로 나뉜다.

| 분류 | 설명 | 특징 |
|:---|:---|:---|
| **소프트 친화성 (Soft)** | 가능한 한 같은 CPU에 할당하려 노력하지만, 부하 분산(Load Balancing)을 위해 다른 CPU로 옮길 수 있음. | 대부분의 범용 OS(Linux, Windows) 채택. |
| **하드 친화성 (Hard)** | 사용자가 특정 프로세스를 특정 CPU 코어에 **박제(Pinning)**함. 절대로 다른 코어로 이동하지 않음. | 실시간 제어, DB 성능 튜닝, 격리된 워크로드. |

---

### Ⅱ. 캐시 워밍과 친화성 메커니즘 (ASCII)

프로세스가 이동할 때 발생하는 캐시 효율성 차이다.

```ascii
    < With Affinity: High Performance >
    [ CPU Core 0 ]  <-- (Process P1 returns)
    +--------------+
    | [ L1 Cache ] | <--- "Warm" Cache: P1's data already here!
    +--------------+
    | [ L2 Cache ] | <--- No need to go to RAM
    +--------------+
    
    < No Affinity: Migration Overhead >
    [ CPU Core 1 ]  <-- (Process P1 migrated)
    +--------------+
    | [ L1 Cache ] | <--- "Cold" Cache: Empty or other app's data
    +--------------+
    | [ L2 Cache ] | <--- Cache Miss!
    +--------------+
           | (Fetch from RAM - Slow)
           v
    [ Main Memory ]
```

---

### Ⅲ. 친화성 스케줄링의 Trade-off: 부하 균등화

친화성을 너무 고집하면 특정 코어는 바쁘고 다른 코어는 노는 불균형이 발생한다.

- **Push Migration**: 특정 CPU의 대기 큐가 너무 길면 유휴 CPU가 프로세스를 강제로 뺏어옴. (친화성 파괴)
- **Pull Migration**: 쉬고 있는 CPU가 바쁜 CPU의 작업을 가져옴. (친화성 파괴)
- **절충점**: 현대 스케줄러는 캐시가 완전히 '식기 전(데이터 유실 전)'에만 친화성을 유지하고, 그 이후에는 과감히 이주시킨다.

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. CPU 피닝 (CPU Pinning)의 활용
- **현상**: 주식 트레이딩 시스템에서 미세한 지연 시간(Jitter)이 수익에 직결됨.
- **기술사적 결단**: 
  - 핵심 연산 스레드를 특정 코어(예: Core 7)에 **Hard Affinity**로 고정한다.
  - 해당 코어에는 다른 인터럽트나 OS 작업이 침범하지 못하도록 **CPU Isolation** 설정을 병행한다.

#### 2. 기술사적 인사이트: NUMA 아키텍처 연계
- 멀티 소켓 서버(NUMA)에서는 프로세서 친화성이 캐시뿐만 아니라 **로컬 메모리 접근** 속도와도 직결된다. 프로세스가 다른 소켓의 CPU로 옮겨지면 멀리 있는 메모리를 읽어야 하므로 성능이 2~3배 급락할 수 있음을 유의해야 한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **애플리케이션 처리량 향상**: 캐시 히트율 상승을 통한 유효 명령어 처리 속도 증가.
- **예측 가능한 응답성**: 특정 코어 점유를 통한 실행 시간 변동폭(Jitter) 최소화.

#### 2. 미래 전망
앞으로의 친화성 스케줄링은 단순히 코어 번호를 맞추는 것을 넘어, **캐시 계층의 토폴로지**를 완벽히 이해하는 방향으로 진화할 것이다. 특히 L3 캐시를 공유하는 코어 그룹 내에서는 이주를 허용하고, 캐시 경계를 넘는 이주는 철저히 통제하는 **토폴로지 인지형 (Topology-aware)** 스케줄링이 클라우드 하이퍼바이저의 핵심 경쟁력이 될 것이다.
