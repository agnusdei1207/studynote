# 다단계 페이지 테이블 (사이즈 줄이기)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이지 테이블 크기를 줄이기 위해 계층적 구조를 사용하여 필요한 부분만 메모리에 유지
> 2. **가치**: 메모리 절약, 대용량 주소 공간 지원, 희소 주소 공간 효율적 관리
> 3. **융합**: 페이징, TLB, 가상 메모리와 통합된 페이지 테이블 관리

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

다단계 페이지 테이블(Hierarchical Page Table)은 페이지 테이블 크기를 줄이기 위해 계층적 구조를 사용하는 기법입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    다단계 페이지 테이블                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              문제: 단일 페이지 테이블 크기                     │ │
│   │                                                              │ │
│   │   32비트 주소, 4KB 페이지:                                    │ │
│   │   • 페이지 수 = 2^32 / 4KB = 2^20 = 1,048,576 페이지         │ │
│   │   • PT 항목 = 4바이트                                         │ │
│   │   • PT 크기 = 1,048,576 × 4 = 4MB (프로세스당!)               │ │
│   │                                                              │ │
│   │   64비트 주소:                                                │ │
│   │   • PT 크기 = 수천 TB (현실적으로 불가능)                     │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              해결: 2단계 페이지 테이블                         │ │
│   │                                                              │ │
│   │   논리 주소 (32비트):                                          │ │
│   │   ┌────────────┬────────────┬────────────┐                  │ │
│   │   │  외부 페이지  │  내부 페이지  │  오프셋    │                  │ │
│   │   │  번호 (10비트) │  번호 (10비트) │ (12비트)  │                  │ │
│   │   └────────────┴────────────┴────────────┘                  │ │
│   │         │              │                                     │ │
│   │         ↓              ↓                                     │ │
│   │   ┌─────────┐     ┌─────────┐                              │ │
│   │   │외부 PT   │ ──→ │내부 PT   │ ──→ 프레임 번호               │ │ │
│   │   │(1개)    │     │(필요한 것만)│                             │ │
│   │   └─────────┘     └─────────┘                              │ │
│   │                                                              │ │
│   │   장점: 사용하는 주소 영역만 inner PT 할당                    │ │
│   │   크기: 4KB (outer) + 4KB × 사용 영역 수                      │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              4단계 페이지 테이블 (x86-64)                      │ │
│   │                                                              │ │
│   │   논리 주소 → PML4 → PDPT → PD → PT → 물리 주소               │ │
│   │              (4KB)  (4KB) (4KB) (4KB)                         │ │
│   │                                                              │ │
│   │   각 단계: 512개 항목 (9비트 × 4 + 12비트 = 48비트 가상 주소)   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> **해설**: 다단계 PT는 실제 사용하는 주소 영역만 메모리에 할당합니다.

**💡 비유**: 다단계 페이지 테이블은 폴더와 파일로 정리된 파일 시스템과 같습니다.

**📢 섹션 요약 비유**: 폴더처럼 계층으로 정리해요!

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **Outer PT** | 최상위 | 큰 그룹 | 폴더 |
| **Inner PT** | 하위 | 세부 항목 | 파일 |
| **PML4** | 4단계 최상위 | x86-64 | 루트 |
| **PDPT/PD/PT** | 하위 단계 | 순차 조회 | 서브폴더 |

**단계별 비용**

| 단계 | PT 수 | 메모리 접근 |
|:---|:---:|:---:|
| **1단계** | 1개 큰 PT | 1회 |
| **2단계** | 작은 PT 여러 개 | 2회 |
| **4단계** | 매우 작은 PT | 4회 |

**핵심 알고리즘**: 주소 분할 → 단계별 PT 조회 → 최종 프레임

**📢 섹션 요약 비유**: 단계가 많으면 느려요!

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**기술 비교: PT 크기 vs 접근 비용**

| 구조 | 메모리 | 접근 | 용도 |
|:---|:---:|:---:|:---|
| **1단계** | 많음 | 빠름 | 작은 주소 |
| **2단계** | 적음 | 중간 | 32비트 |
| **4단계** | 최소 | 느림 | 64비트 |

**📢 섹션 요약 비유**: TLB가 있어서 괜찮아요!

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1: 32비트 시스템** → 2단계 PT
**시나리오 2: 64비트 시스템** → 4단계 PT
**시나리오 3: Huge Pages** → 단계 축소

**도입 체크리스트**: 주소 크기, 메모리 제약, TLB 활용

**📢 섹션 요약 비유**: 64비트엔 4단계가 필요해요!

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**정량/정성 기대효과**: 메모리 절약, 대용량 주소 지원

**미래 전망**: 5단계 PT, 가상화 확장

**참고 표준**: x86-64 PT, ARM PT
