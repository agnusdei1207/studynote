# 인터럽트 처리의 상반부(Top Half)와 하반부(Bottom Half) 분리

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터럽트 발생 시 즉각 실행해야 하는 최소한의 작업(**상반부**)과, 시간이 오래 걸려 나중에 처리해도 되는 나머지 작업(**하반부**)으로 인터럽트 핸들러를 이원화하는 기술.
> 2. **가치**: 인터럽트 금지 시간(Interrupt Latency)을 최소화하여 시스템의 응답성을 높이고, 다른 중요한 인터럽트가 차단되는 병목 현상을 방지한다.
> 3. **융합**: 하드웨어적인 즉각 반응성과 소프트웨어적인 스케줄링 유연성을 결합한 구조이며, 리눅스 커널의 **Softirq, Tasklet, Workqueue** 등이 하반부 구현의 핵심 도구다.

---

### Ⅰ. 상반부와 하반부의 역할 분담 (Division of Labor)

| 구분 | 상반부 (Top Half / ISR) | 하반부 (Bottom Half / Deferred) |
|:---|:---|:---|
| **실행 시점** | 인터럽트 발생 즉시 | 상반부 종료 후 운영체제가 적절히 판단 |
| **작업 내용** | 하드웨어 레지스터 읽기, 승인(ACK), 큐잉 | 데이터 패킷 처리, 체크섬 계산, 파일 쓰기 |
| **인터럽트 상태** | **차단됨 (Disabled)** | **허용됨 (Enabled)** |
| **제약 사항** | 매우 짧아야 함, Sleep 불가 | 상대적으로 길어도 됨, 스케줄링 가능 |

---

### Ⅱ. 인터럽트 지연 처리 아키텍처 (ASCII)

```ascii
    [ Hardware Interrupt ]
           |
           v
    +------------------------------------------+
    | [ Top Half (ISR) ]                       |
    | 1. Mask Interrupts                       |
    | 2. ACK Hardware                          |
    | 3. Schedule Bottom Half Work             |
    | 4. Unmask Interrupts                     |
    +-------------------|----------------------+
                        | (Return from IRQ)
                        v
    ============================================
    [ Context Switch to Softirq / Tasklet ]
    ============================================
                        |
                        v
    +------------------------------------------+
    | [ Bottom Half (Deferred Work) ]          |
    | - Process large data buffer              |
    | - Notify user-space apps                 |
    | - Free temporary memory                  |
    +------------------------------------------+
```

---

### Ⅲ. 리눅스 하반부 구현 기법 (Mechanisms)

| 기법 | 특성 | 실행 문맥 | 용도 |
|:---|:---|:---|:---|
| **Softirq** | 정적으로 정의된 고속 지연 처리. | 하드웨어 인터럽트 직후 | 네트워킹, 블록 장치 (고성능) |
| **Tasklet** | Softirq 기반이나 동적 생성 및 순서 보장. | Softirq 문맥 | 일반적인 장치 드라이버 |
| **Workqueue** | **커널 스레드**에 의해 실행됨. | 프로세스 문맥 | **I/O 대기(Sleep)가 필요한 작업** |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 성능 병목 진단: ksoftirqd 부하
- **현상**: CPU 이용률 중 `si`(Softirq) 수치가 매우 높고 네트워크 패킷 유실이 발생함.
- **기술사적 결단**: 
  - 이는 하반부 처리가 상반부의 발생 속도를 따라가지 못하는 상황이다.
  - 인터럽트 병합(Interrupt Coalescing)을 통해 상반부 발생 횟수 자체를 줄이거나, 멀티코어에 하반부 부하를 분산하는 **RSS (Receive Side Scaling)** 기술을 적용해야 한다.

#### 2. 기술사적 인사이트: 원자성 유지
- 상반부는 인터럽트가 차단된 원자적(Atomic) 상태이므로 절대 잠들거나(Sleep) 블로킹 함수를 호출해서는 안 된다. 반면 하반부는 일반 프로세스와 경쟁하며 실행되므로 자유롭게 락을 쓰거나 대기할 수 있다. 이 경계를 명확히 하는 것이 시스템 안정성의 핵심이다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **시스템 처리량 증대**: 인터럽트 대기 시간 감소를 통한 연산 효율 극대화.
- **실시간성 확보**: 긴급한 하드웨어 신호를 놓치지 않는 견고한 인터페이스 구축.

#### 2. 미래 전망
최근에는 하드웨어가 직접 하반부의 일부 기능을 처리하는 **SmartNIC**이나 **DPU** 오프로딩 기술이 발전하고 있다. 운영체제 레벨에서도 eBPF를 사용하여 ISR 단계에서 즉시 패킷을 필터링하거나 처리함으로써, 전통적인 상/하반부 전환 오버헤드조차 줄이려는 시도가 계속되고 있다.
