# iOS 앱 샌드박싱 구조 (App Sandboxing in iOS)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: iOS 앱 샌드박싱은 운영체제 수준에서 각 애플리케이션을 고유한 컨테이너(Home Directory)와 엄격하게 제한된 권한 내에 가두어, 인가되지 않은 시스템 자원이나 타 앱의 데이터에 대한 접근을 하드웨어 및 커널 레벨에서 원천 차단하는 보안 아키텍처다.
> 2. **가치**: 악성 앱에 의한 시스템 핵심 설정 변조나 사용자 개인정보(연락처, 사진 등) 무단 탈취를 방어하며, 앱이 침해당하더라도 그 피해가 샌드박스 외부로 확산되지 않도록 하는 **"폭쇄 영역(Blast Radius)"** 최소화 전략의 정수다.
> 3. **융합**: 하부 커널의 **TrustedBSD** 정책, 바이너리 무결성을 검증하는 **AMFI (Apple Mobile File Integrity)**, 그리고 정교한 권한 명세인 **Entitlements**와 유기적으로 결합되어 애플 에코시스템의 신뢰 기반(Trust Base)을 형성한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: iOS 샌드박스는 모든 앱에게 각자의 '독방'을 주는 것과 같다. 앱은 자신의 방 안에서는 자유롭게 파일을 만들고 읽을 수 있지만, 방문을 열고 복도로 나가거나 다른 앱의 방을 들여다보는 행위는 운영체제라는 간수(Sandbox Daemon)에 의해 철저히 통제된다.

- **💡 비유**: 이는 "초호화 아파트의 개별 호실"과 같다. 각 입주민(앱)은 자신의 집(Documents 폴더) 안에서는 마음대로 생활할 수 있다. 하지만 옆집에 들어가려면 반드시 주인의 허락(사용자 동의)과 관리사무소의 특별 마스터키(Entitlements)가 있어야만 한다. 복도에 있는 공용 시설(카메라, 마이크)을 쓰려 할 때도 매번 관리사무소에 신고해야 한다.

- **등장 배경**:
  1. **모바일 개인정보의 가치 상승**: 스마트폰이 금융, 건강, 위치 정보를 담는 핵심 기기가 됨에 따라 기존 PC 방식의 개방형 접근 모델은 보안적 재앙이 되었다.
  2. **앱스토어 모델의 안전성 확보**: 검증되지 않은 수백만 개의 써드파티 앱을 안전하게 실행하기 위해선, 앱 개발자를 믿는 대신 시스템 구조(Sandbox)를 믿는 아키텍처가 필요했다.
  3. **권한 상승 공격 차단**: 특정 앱의 버그를 이용해 루트(Root) 권한을 획득하려는 익스플로잇 시도를 샌드박스 경계에서 차단하고자 했다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              iOS 앱 컨테이너 및 샌드박스 내부 구조 (FS View)        │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ App Sandbox Container ]                                     │
  │   ┌─────────────────────────────────────────────────────────┐   │
  │   │  Bundle Container : App.app (Code, Assets) - [ReadOnly] │   │
  │   ├─────────────────────────────────────────────────────────┤   │
  │   │  Data Container   :                                     │   │
  │   │  ├─ Documents/    : 유저 생성 데이터 (iCloud 백업 대상)    │   │
  │   │  ├─ Library/      : 앱 설정, 캐시, 스냅샷 (비공개)         │   │
  │   │  └─ tmp/          : 임시 파일 (재부팅 시 삭제 가능)        │   │
  │   └─────────────────────────────────────────────────────────┘   │
  │          │                                                      │
  │   ───────┼────────────────────────────────── (Sandbox Wall) ──  │
  │          ▼                                                      │
  │   [ Kernel / Hardware Resources ] (Camera, Location, Contacts)  │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 모든 iOS 앱은 설치 시 자신만의 고유한 UUID 기반 경로를 할당받는다. 앱 실행 파일이 들어있는 'Bundle' 영역은 위변조 방지를 위해 읽기 전용으로 보호되며, 데이터를 쓸 수 있는 곳은 오직 자신에게 할당된 'Data' 영역뿐이다. 이 벽을 넘으려는 모든 시스템 콜은 커널의 샌드박스 프로필에 의해 거부된다.

- **📢 섹션 요약 비유**: "자신의 방 안(Data Container)에서만 놀 수 있게 하고, 밖으로 나가는 모든 통로를 검문소(Kernel Policy)로 막아버린 철통 격리 시스템"입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 샌드박스 시스템의 3대 가디언

iOS 보안은 단순히 파일 경로만 막는 것이 아니라, 다층적인 검증 단계를 거친다.

| 구성 요소 | 기술적 역할 | 상세 메커니즘 | 비유 |
|:---|:---|:---|:---|
| **AMFI** | 바이너리 무결성 검증 | Apple Mobile File Integrity. 실행 전 코드 서명이 유효한지 확인 | 정품 인증 홀로그램 |
| **Entitlements** | 권한 명세서 | 앱이 사용할 수 있는 특권(iCloud, Push 등)을 담은 XML 프로필 | 특수 임무 수행권 |
| **TCC (Transparency)** | 사용자 개인정보 보호 | 가시적 권한(카메라, 마이크 등)에 대해 유저의 동적 승인 획득 | 당사자 동의 확인 |

---

### 샌드박스 프로필과 TrustedBSD 정책

iOS 커널은 **TrustedBSD** 프레임워크를 사용하여 접근 제어 정책을 강제한다.

1. **Mandatory Access Control (MAC)**: 프로세스의 주인이 누구든 상관없이 시스템 정책이 우선함.
2. **Sandbox Profile**: 앱 실행 시 커널에 로드되는 규칙 모음. (예: `(deny file-write* (subpath "/etc"))`)
3. **Containerization**: 앱마다 서로 다른 데이터 디렉터리를 강제 매핑하여 타 앱 데이터 접근 불가.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 앱 간 데이터 공유의 예외 경로 (App Groups)          │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ App A ] ──▶ [ Group Container ] ◀── [ App B ]                │
  │                    (Shared Path)                                 │
  │                                                                  │
  │   [조건] 1. 동일 개발자가 만든 앱들일 것                           │
  │          2. Entitlements에 동일한 Group ID가 명시될 것             │
  │                                                                  │
  │   [가치] 완전 격리의 불편함을 해소하기 위한 공식적 통로 제공          │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 샌드박스는 기본적으로 '제로 쉐어링'을 지향하지만, 같은 회사의 앱끼리(예: 네이버 블로그와 네이버 카페) 데이터를 공유해야 할 때는 'App Groups'라는 특수 통로를 열어준다. 이는 엄격한 규칙 아래 허용된 '합법적 탈출구'다.

- **📢 섹션 요약 비유**: "성벽을 높게 쌓되(격리), 가족(동일 개발자)끼리는 소통할 수 있는 작은 비밀 통로(App Groups)를 만들어둔 계획도시"와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: iOS 샌드박스 vs 안드로이드 샌드박스

| 비교 항목 | iOS (Closed/Strict) | Android (Semi-open) |
|:---|:---|:---|
| **기반 기술** | **TrustedBSD / AMFI** | **SELinux / UID isolation** |
| **격리 단위** | 앱 컨테이너 기반 | 사용자 ID (UID) 기반 |
| **상호 작용** | URL Scheme, Share Sheet (매우 제한) | Intent, Content Provider (비교적 자유) |
| **보안 강도** | **최상** (탈옥 없이 우회 거의 불가) | 우수 (단, 권한 허용 시 위험 노출) |

### 비교 2: 샌드박싱과 앱 생명주기(Lifecycle) 연동

- **Suspended State**: 샌드박싱은 앱이 백그라운드로 갔을 때의 행위도 제한한다. 마음대로 위치 정보를 수집하거나 네트워크를 쓰는 행위를 샌드박스 정책으로 억제하여 배터리와 프라이버시를 동시에 보호한다.
- **Data Protection**: 폰이 잠금 상태일 때 샌드박스 내의 파일을 암호화하여, 물리적 탈취 상황에서도 데이터를 보호한다.

- **과목 융합 관점 (네트워크)**: 샌드박스는 네트워크 통신에서도 작동한다. **App Transport Security (ATS)** 정책은 앱이 보안되지 않은 HTTP 통신을 시도할 때 샌드박스 레벨에서 이를 차단하고 HTTPS 사용을 강제한다.

- **📢 섹션 요약 비유**: "안드로이드가 집집마다 담장을 세운 마을이라면, iOS는 모든 집이 중앙 통제실의 감시를 받는 최첨단 보안 빌딩과 같습니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 사내 보안 앱 개발**: 임직원용 앱이 사내 내부망에만 접속해야 하고 사진첩에는 접근하면 안 됨. 기술사적 판단으로 **Entitlements 최적화**를 수행한다. 사진첩 접근 권한(`NSPhotoLibraryUsageDescription`)을 아예 제거하고, 네트워크는 `com.apple.developer.networking.vpn.api` 특권을 통해 회사 VPN을 통해서만 통신하게 강제하여 샌드박스 내에서 보안 거버넌스를 실현한다.

2. **시나리오 — 탈옥(Jailbreak) 탐지 및 대응**: 샌드박스 벽이 뚫린 기기에서 금융 앱이 실행됨. 해결책으로 앱 내부에서 **Sandbox Escape 체크**를 수행한다. 자신의 샌드박스 외부에 파일을 쓰거나(`/private/jailbreak.txt`), `fork()`와 같은 금지된 시스템 콜을 시도하여 성공한다면 시스템 무결성이 깨진 것으로 간주하고 앱 실행을 즉시 중단한다.

3. **시나리오 — 대용량 미디어 편집 앱의 저장 공간 부족**: 4K 영상을 편집하는데 샌드박스 용량 제한이 걸림. 기술사적 판단으로 **`Library/Caches`** 영역의 활용과 `isExcludedFromBackup` 플래그 설정을 가이드한다. 중요 유저 데이터와 재생성 가능한 작업 데이터를 구분하여 배치함으로써, 아이클라우드 백업 효율과 샌드박스 내 가용 공간을 동시에 확보한다.

### 도입 체크리스트 (Security Audit)
- **최소 권한 원칙**: `Info.plist`에 정의된 권한 설명문이 실제 앱 동작과 일치하며, 과도한 권한을 요구하고 있지 않은가?
- **코드 서명 유효성**: 배포용 프로비저닝 프로파일이 최신 상태이며, AMFI에 의해 거부될 소지는 없는가?
- **공유 데이터 무결성**: App Groups를 통한 공유 시, 데이터 레이스(Race Condition)를 막기 위한 파일 락(Lock) 처리가 되어 있는가?

### 안티패턴
- **샌드박스 우회 시도**: 비공식 API(Private API)를 사용하여 샌드박스 밖의 시스템 로그를 읽으려 하거나 파일 시스템 루트를 탐색하는 행위. 앱스토어 거절 1순위이며 향후 OS 업데이트 시 앱이 즉시 크래시됨.
- **불필요한 Entitlements 신청**: 사용하지 않는 기능을 권한 리스트에 넣어두는 것. 공격자가 앱을 장악했을 때 활용할 수 있는 통로만 늘려주는 꼴임.

- **📢 섹션 요약 비유**: "울타리(Sandbox)는 앱을 가두는 감옥이 아니라, 외부의 폭풍(공격)으로부터 앱의 소중한 데이터를 보호하는 안락한 울타리임을 이해해야 합니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (샌드박싱 아키텍처 도입 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **사용자 프라이버시** | 명시적 승인 없는 데이터 접근 0% 실현 | 고객 신뢰도 확보 |
| **시스템 가용성** | 앱 오류나 악성 행위가 시스템 전체로 전파되는 것 방지 | 안정성 99% 유지 |
| **멀웨어 전파 억제** | 앱 간 데이터 전송 경로 통제로 웜/바이러스 확산 차단 | 클린 에코시스템 유지 |
| **복구 용이성** | 앱 삭제 시 해당 샌드박스 컨테이너만 지우면 찌꺼기 없이 제거 | 클린 언인스톨 지원 |

### 미래 전망
- **User-space Partitioning**: 하드웨어 가상화 기술을 모바일 칩셋에 적용하여, 샌드박스보다 더 강력한 하드웨어 격리 수준의 앱 실행 환경(Memory Tagging 등) 보편화.
- **Privacy-preserving Computing**: 샌드박스 내에서 수집된 데이터를 외부로 보내지 않고 기기 안에서만 연산(On-device AI)하여 결과를 내는 '프라이버시 강화 컴퓨팅'의 표준 기술화.

### 참고 표준
- **Apple Platform Security Guide**: iOS 보안 아키텍처의 공식 표준 문서.
- **Common Criteria (CC)**: iOS 샌드박스의 격리 수준 평가에 사용되는 국제 보안 표준.

- **📢 섹션 요약 비유**: iOS 샌드박싱은 "자유로운 앱 생태계"라는 바다 위에 떠 있는 수많은 "독립된 섬"이며, 이 섬들이 서로 연결되는 다리(API)가 견고할 때 가장 안전한 항해가 가능합니다.

3. 이 텐트 덕분에 한 친구가 장난감을 엉망으로 어질러도 내 텐트는 **언제나 깨끗하고 안전하게** 유지될 수 있는 거랍니다!
