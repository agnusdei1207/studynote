# 안드로이드 LMK (Low Memory Killer) 및 메모리 관리 전략

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: LMK (Low Memory Killer)는 안드로이드 시스템에서 가용 메모리가 부족해질 때 시스템 전체의 패닉(Global OOM)을 방지하기 위해, 프레임워크가 부여한 앱의 가시성과 중요도(**oom_adj**)에 따라 프로세스를 단계적으로 강제 종료하는 동적 자원 관리 메커니즘이다.
> 2. **가치**: 사용자가 현재 상호작용 중인 포그라운드(Foreground) 앱과 인지 가능한 핵심 서비스(음악 재생 등)를 최우선으로 보호하며, 제한된 모바일 자원 환경에서도 부드러운 사용자 경험(UX)과 시스템 안정성을 동시에 확보한다.
> 3. **융합**: 리눅스 커널의 OOM 킬러 아키텍처를 기반으로 하되, 안드로이드 **ActivityManagerService (AMS)**의 앱 상태 추적 로직과 **cgroups**의 메모리 통계, 그리고 현대적인 **PSI (Pressure Stall Information)** 기술이 유기적으로 결합된 하이브리드 시스템이다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: LMK는 안드로이드의 "선제적 메모리 정지 시스템"이다. 시스템에 램이 부족해서 숨이 넘어갈 때까지 기다리는 것이 아니라, 메모리 잔량이 위험 수위에 도달할 때마다 덜 중요한 앱부터 순서대로 정리하여 숨통을 틔워주는 역할을 한다.

- **💡 비유**: 이는 "침몰하는 배의 짐 던지기"와 같다. 배(시스템)가 가라앉으려 할 때, 가장 소중한 사람(포그라운드 앱)을 살리기 위해 가장 먼저 버려도 되는 쓰레기(캐시된 앱)부터 바다로 던지고, 그래도 부족하면 가구(백그라운드 서비스)를 던지는 단계적 비상조치다.

- **등장 배경**:
  1. **모바일 환경의 특수성**: PC와 달리 스마트폰 사용자는 여러 앱을 켜두고 수시로 전환하며, 각 앱은 가비지 컬렉션(GC) 시점까지 대량의 램을 점유한다.
  2. **가용 메모리 예측 불가능성**: 새 앱을 띄울 때 필요한 메모리가 부족하면 사용자 반응 속도(Response Time)가 급격히 떨어지므로, 미리 여유 공간을 확보해둘 필요가 있었다.
  3. **사용자 경험(UX) 최우선**: 사용자가 지금 보고 있는 앱이 갑자기 꺼지는 것을 막는 것이 보안보다 더 중요한 사용성 지표가 되었다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              안드로이드 메모리 계층 및 LMK 작동 임계치 (Example)     │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ Free Memory Level ]        [ Action: Kill Process Group ]   │
  │                                                                 │
  │   1. 512 MB ──────────────▶ [ Empty Apps (ADJ 15) ]             │
  │   2. 256 MB ──────────────▶ [ Content Providers (ADJ 10) ]      │
  │   3. 128 MB ──────────────▶ [ Background Apps (ADJ 7) ]         │
  │   4.  64 MB ──────────────▶ [ Visible Apps (ADJ 1) ]            │
  │   5.  32 MB ──────────────▶ [ Foreground App (ADJ 0) ]          │
  │                                                                 │
  │   [!] 임계치(minfree)에 도달할 때마다 해당 ADJ 이상의 앱을 숙정함     │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** LMK는 여러 개의 '방어선'을 가지고 있다. 메모리가 조금 부족할 때는 한참 전에 썼던 캐시 앱들만 지우지만, 메모리가 심각하게 부족해지면 지금 바로 앞에 떠 있는 앱(Visible)까지도 건드리기 시작한다. 이 임계값들은 `/sys/module/lowmemorykiller/parameters/minfree`에 정의되어 있다.

- **📢 섹션 요약 비유**: "배가 가라앉기 전에 미리미리 짐을 하나씩 버려두어, 절대 배가 뒤집히는 일(Kernel Panic)이 없게 만드는 치밀한 안전 설계"입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 앱 중요도의 척도: oom_adj (Adjust)

안드로이드 프레임워크는 앱의 상태를 실시간으로 감시하여 점수를 매긴다.

| ADJ 범위 | 상태 명칭 | 상세 설명 | 비유 |
|:---|:---|:---|:---|
| **0** | **Foreground** | 현재 화면에 떠 있고 유저와 소통 중인 앱 | 주인공 |
| **1** | **Visible** | 화면 일부가 보이거나 투명한 창 뒤에 있는 앱 | 조연 배우 |
| **2** | **Perceptible** | 화면엔 없지만 소리가 나거나 알림을 띄운 앱 | 보이지 않는 목소리 |
| **7 ~ 9** | **Previous / Home** | 바로 이전에 썼던 앱이나 런처 화면 | 어제 입은 옷 |
| **15** | **Empty / Cached** | 실행은 끝났으나 나중에 빨리 띄우려고 램에 남겨둔 앱 | 다 본 잡지 |

### LMK의 기술적 진화: Kernel LMK vs lmkd

1. **Kernel LMK (Legacy)**: 리눅스 커널 드라이버로 구현됨. 속도는 빠르나 안드로이드 유저 영역(Framework)의 복잡한 상태를 반영하기 힘들었음.
2. **lmkd (Userspace Daemon)**: 현대 안드로이드의 방식. 유저 영역에서 돌아가는 데몬이 커널의 **PSI (Pressure Stall Info)**를 감시하다가, 메모리 압박이 감지되면 직접 프로세스를 종료시킴.
3. **PSI (Pressure Stall Information)**: 단순 '남은 램 용량'이 아니라, "프로세스들이 메모리를 기다리느라 얼마나 굶고 있는가(Stall)"를 측정하는 더 과학적인 지표 활용.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 lmkd와 커널 PSI의 상호작용 흐름                    │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ App Processes ] ────▶ [ Request RAM ] ────▶ [ Linux Kernel ] │
  │                                                    │             │
  │   ┌──────────────┐         (Trigger)               │             │
  │   │    lmkd      │ ◀────────────────────────── [ PSI Monitor ]   │
  │   │   (Daemon)   │                                 │             │
  │   └──────┬───────┘                                 ▼             │
  │          │ (Send SIGKILL)                   [ Memory Reclaim ]   │
  │          ▼                                                       │
  │   [ Select & Kill ADJ 15 ] ──▶ [ Success! ] ──▶ [ Release RAM ]  │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** `lmkd`는 커널로부터 "메모리 부족 경보"를 받으면, 자신의 장부를 뒤져 현재 `oom_adj`가 가장 높은 앱을 찾아내어 `kill` 시그널을 보낸다. 유저 영역에서 결정하므로 정책을 유연하게 바꿀 수 있고, 안드로이드 버전별로 최적화가 쉽다는 장점이 있다.

- **📢 섹션 요약 비유**: "현장의 아우성(PSI)을 듣고 본부의 지령(lmkd)을 받아, 가장 비효율적인 부서부터 정리해나가는 기업 구조조정의 논리"와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 일반 Linux OOM Killer vs Android LMK

| 비교 항목 | Linux OOM Killer | Android LMK (lmkd) |
|:---|:---|:---|
| **트리거 시점** | 메모리가 **0**이 되어 패닉 직전일 때 | 메모리가 **부족할 징조**가 보일 때 (선제적) |
| **종료 대상** | `oom_score`가 높은 놈 (주로 대용량 점유) | `oom_adj`가 높은 놈 (주로 안 쓰는 앱) |
| **목적** | 시스템 공멸 방지 | **포그라운드 앱의 부드러움(UX) 유지** |
| **관리 주체** | 커널 (Kernel) | 안드로이드 유저 데몬 (lmkd) |

### 비교 2: LMK와 가비지 컬렉션 (GC)의 협력

- **GC**: 프로세스 **내부**의 쓰레기 메모리를 정리함. (청소)
- **LMK**: 프로세스 **자체**를 종료하여 메모리를 통째로 회수함. (철거)
- **융합**: 안드로이드는 램이 부족하면 먼저 각 앱에게 `onTrimMemory()` 콜백을 보내 "너네 GC 좀 해라"라고 경고를 준다. 그래도 부족하면 LMK가 출동하여 앱을 철거한다.

- **과목 융합 관점 (임베디드)**: LMK는 임베디드 시스템의 **Watchdog**과 유사한 철학을 가진다. 자원 임계치를 감시하다가 비정상 상태로 가기 전 강제 리셋(종료)을 수행하여 전체 시스템의 가용성을 지키는 행위가 본질적으로 동일하기 때문이다.

- **📢 섹션 요약 비유**: "방 안의 쓰레기를 치우는 것(GC)이 먼저지만, 집이 무너지려 하면 방 하나를 아예 폐쇄(LMK)해야 전체 건물을 살릴 수 있습니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 모바일 게임 멀티태스킹 이슈**: 고사양 게임을 하다가 잠깐 카톡을 보냈는데, 다시 게임으로 돌아오니 로딩 화면이 뜸. 원인은 카톡 실행 시 메모리가 모자라 LMK가 게임 프로세스를 죽였기 때문. 기술사적 판단으로 게임 앱의 **`onSaveInstanceState()`** 구현을 확인하고, 게임사의 경우 '포그라운드 서비스'를 활용하여 `oom_adj` 점수를 낮춰 LMK의 칼날을 피하도록 가이드한다.

2. **시나리오 — 저사양 폰(Android Go) 성능 최적화**: 램이 2GB인 폰에서 자꾸 시스템이 버벅임. 해결책으로 **LMK minfree 임계값을 상향 조정**한다. 메모리가 조금만 부족해도 더 빨리 앱을 죽여서 가용 램을 항상 넉넉히(약 300MB 이상) 유지하게 함으로써 시스템 UI 응답성을 확보한다.

3. **시나리오 — 커스텀 커널의 LMK 오작동**: 커널 패치 후 LMK가 너무 공격적으로 변해 필요한 서비스까지 죽임. 분석 결과 PSI 임계치가 너무 낮게 잡혀 있었음. 해결책으로 **`lmkd` 설정 파일의 PSI 윈도우**를 조절하여, 실제 사용자가 버벅임을 느끼는 순간에만 킬러가 작동하도록 정밀하게 튜닝한다.

### 도입 체크리스트 (Optimization Audit)
- **ADJ 가중치 확인**: 우리 서비스의 백그라운드 작업이 중요한가? (중요하다면 `Foreground Service` 선언 확인)
- **Memory Leak 모니터링**: LMK가 특정 앱만 반복적으로 죽이고 있다면 해당 앱의 메모리 누수 가능성 조사.
- **zRAM 연동**: LMK가 출동하기 전, zRAM(압축 메모리)이 먼저 작동하여 메모리 부족 상황을 완충하고 있는가?

### 안티패턴
- **LMK 기능을 끄려는 시도**: "메모리 부족하다고 죽이지 마!"라며 커널 드라이버를 비활성화하는 행위. 곧바로 전체 시스템이 멈추고 강제 재부팅되는 대참사 발생.
- **모든 앱을 포그라운드 서비스로 등록**: 권한 남용으로 인해 LMK가 죽일 앱이 없어지면, 결국 시스템 핵심 부품(SurfaceFlinger 등)이 죽어 폰이 벽돌이 됨.

- **📢 섹션 요약 비유**: "모두가 VIP(ADJ 0)가 되려 하면, 결국 파티장(시스템)은 무너집니다. 적절한 위계 질서가 시스템의 평화를 지킵니다."

| **자원 효율** | 안 쓰는 앱의 램을 능동적으로 회수하여 실제 필요 작업에 배분 | 램 활용 밀도 극대화 |
| **배터리 보호** | 불필요한 백그라운드 프로세스 제거를 통한 전력 소모 절감 | 대기 시간 15% 연장 |

### 미래 전망
- **AI-Based Predict-Kill**: 유저의 앱 사용 패턴을 학습하여, "이 앱은 1시간 동안 안 켤 것 같다"고 판단되면 메모리가 부족해지기 전에 미리 죽여두는 지능형 LMK 등장.
- **Hardware-Integrated Memory Management**: CPU 하드웨어가 직접 PSI 메트릭을 분석하고 하드웨어 수준에서 프로세스 일시 정지/재개를 수행하는 초저지연 자원 통제 기술 발전.

### 참고 표준
- **Android Open Source Project (AOSP) - lmkd**: 안드로이드 공식 저메모리 킬러 데몬 표준 소스.
- **Linux Kernel PSI (Pressure Stall Information)**: 현대적 메모리 부족 탐지의 표준 커널 인터페이스.

- **📢 섹션 요약 비유**: LMK는 안드로이드라는 거대한 도시의 "교통 정리 경찰"이며, 흐름을 방해하는 차량을 과감히 견인함으로써 도시 전체의 소통을 유지하는 필수적인 공권력입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **OOM 킬러**: LMK가 실패했을 때 작동하는 시스템의 최후 심판자.
- **cgroups (제어 그룹)**: LMK가 프로세스들의 램 사용량을 감시하기 위해 사용하는 기초 인프라.
- **슬랩 (Slab) 할당기**: LMK가 확보한 빈 페이지들을 쪼개어 다시 사용하는 실제 메모리 소비처.
- **프로세스 친화성 (Affinity)**: 특정 CPU에 고정된 프로세스가 메모리까지 폭점할 때 LMK의 감시 대상이 됨.
