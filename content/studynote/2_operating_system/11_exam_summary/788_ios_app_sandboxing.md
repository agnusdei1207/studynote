# iOS 앱 샌드박싱 (App Sandboxing) 구조

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 앱마다 독립된 파일 시스템 영역(Container)을 할당하고, 시스템 자원(카메라, 연락처 등) 접근을 커널 레벨에서 엄격히 통제하여 **앱 간의 데이터 접근을 원천 차단**하는 보안 아키텍처.
> 2. **가치**: 악성 앱이 설치되더라도 시스템 전체나 다른 앱의 정보(금융 데이터, 비밀번호 등)를 탈취할 수 없게 격리하며, 사용자 승인 없이는 어떠한 사생활 정보도 수집하지 못하게 강제한다.
> 3. **융합**: 운영체제의 파일 권한(ACL), 코드 서명(Code Signing), 그리고 선언적 권한 명세인 **엔타이틀먼트(Entitlements)** 기술이 집약된 폐쇄형 생태계의 핵심 방어막이다.

---

### Ⅰ. iOS 샌드박스의 구성 요소 (Structure)

앱은 오직 자신에게 허용된 '박스' 안에서만 놀 수 있다.

1. **App Bundle**: 앱 바이너리와 리소스가 담긴 읽기 전용 영역.
2. **Data Container**: 앱이 실제 데이터를 쓰고 저장하는 영역.
   - **Documents**: 사용자 생성 데이터 (백업 대상).
   - **Library**: 앱 설정, 캐시, 스냅샷 등.
   - **tmp**: 임시 파일 (시스템이 수시로 삭제).
3. **System Resources**: 카메라, 마이크, 위치 정보 등 (Entitlements를 통해 접근 허가).

---

### Ⅱ. 샌드박싱 격리 아키텍처 (ASCII)

운영체제 커널(XNU)이 모든 요청을 감시하고 거부하는 구조다.

```ascii
    [ App A ]          [ App B ]          [ App C ]
    +-------+          +-------+          +-------+
    | Box A |          | Box B |          | Box C |
    +---|---+          +---|---+          +---|---+
        |                  |                  |
    ====|==================|==================|===== (Access Denied)
        v                  v                  v
    +----------------------------------------------+
    | [ iOS Security Framework / Kernel Hooks ]    |
    | - Sandbox Profiles (Trusted Code Only)       |
    | - Entitlement Verification                   |
    +----------------------|-----------------------+
                           v
    +----------------------------------------------+
    | [ Shared Resources: Photos, Contacts, HW ]   |
    +----------------------------------------------+
```

---

### Ⅲ. 핵심 보안 매커니즘: Entitlements

- **개념**: 앱이 사용할 권한을 명시한 XML 형태의 디지털 서명 파일.
- **동작**: 앱이 카메라를 켜려고 하면, 커널은 해당 앱의 **엔타이틀먼트** 목록에 `com.apple.security.device.camera`가 있는지 확인한다. 없으면 즉시 차단한다.
- **TCC (Transparency, Consent, and Control)**: 권한이 있더라도 실제 접근 시 사용자에게 "카메라를 허용하시겠습니까?"라는 팝업을 띄워 최종 승인을 받는다.

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 앱 간 데이터 공유 (Inter-app Communication)
- **현상**: 샌드박싱 때문에 앱끼리 데이터를 주고받기가 매우 어렵다.
- **기술사적 결단**: 
  - **App Groups**: 동일 개발자가 만든 앱끼리는 공유 컨테이너 영역을 설정하여 데이터를 교환하게 한다.
  - **Custom URL Schemes / AirDrop**: 특정 규격화된 통로를 통해서만 데이터 전달을 허용하여 보안과 편의성의 균형을 맞춘다.

#### 2. 기술사적 인사이트: 탈옥 (Jailbreak)의 원리
- 탈옥은 사실상 이 **샌드박싱 커널 훅을 무력화**하여 모든 프로세스가 root 권한으로 파일 시스템 전체에 접근할 수 있게 만드는 행위다. 따라서 탈옥된 기기는 샌드박싱이 보장하는 모든 보안 가치가 상실된 상태임을 인지해야 한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **프라이버시 강화**: 사용자 허락 없는 개인정보 유출 방지.
- **시스템 무결성**: 한 앱의 크래시나 감염이 OS 전체로 번지지 않음.

#### 2. 미래 전망
최근 macOS에도 iOS 수준의 샌드박싱이 도입되는 등 '데스크탑의 모바일화'가 진행되고 있다. 향후에는 하드웨어 보안 칩(Apple Silicon의 Secure Enclave)과 연동하여, 샌드박스 내부의 데이터를 하드웨어 레벨에서 암호화하고 격리하는 **하드웨어 가속 샌드박싱**이 보안의 표준이 될 것이다.

### 👶 어린이를 위한 3줄 비유 설명
1. **샌드박싱**은 유치원에서 친구들마다 자기만의 **'모래 놀이상자'**를 하나씩 주는 것과 같아요.
2. 내 상자 안에서만 모래성을 쌓을 수 있고, 친구 상자에 있는 모래를 몰래 가져오거나 망가뜨릴 수 없게 선생님이 꽉 지키고 있죠.
3. 덕분에 친구가 실수로 모래를 쏟아도 내 상자는 항상 깨끗하고 안전하게 유지될 수 있답니다!
