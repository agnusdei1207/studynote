# 보호 도메인 및 최소 권한의 원칙 (Protection Domain & Least Privilege)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보호 도메인 (Protection Domain)은 특정 시점에 프로세스가 접근할 수 있는 자원(Objects)과 허용된 권한(Rights)의 집합이며, 최소 권한의 원칙 (POLP, Principle of Least Privilege)은 이 도메인을 업무 수행에 필요한 "최소한"으로 제한하는 정보 보안의 핵심 설계 철학이다.
> 2. **가치**: 시스템의 특정 개체가 침해당하거나 오동작하더라도 그 피해가 도메인 외부로 확산되는 것을 방지하는 "폭쇄 영역 (Blast Radius)" 제한 기제이며, 내부자 위협과 권한 오남용을 원천적으로 차단하여 시스템 견고성을 확보한다.
> 3. **융합**: 운영체제의 커널/유저 모드 전환, 가상화 환경의 샌드박싱, 마이크로서비스 아키텍처 (MSA)의 API 접근 제어, 그리고 현대 보안 패러다임인 제로 트러스트 (Zero Trust)의 이론적 근간으로 작용한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 보호 도메인은 '누가(Subject)', '무엇에(Object)', '어떻게(Right)' 접근할 수 있는지를 규정한 논리적 경계다. 최소 권한의 원칙은 이 경계를 최대한 좁게 설정하여, 주체가 가진 권한이 그 주체가 수행해야 할 합법적인 목적을 절대 초과하지 않도록 관리하는 것이다.

- **💡 비유**: 이는 호텔 객실 키와 같다. 투숙객(프로세스)은 자신이 예약한 방(자원)만 열 수 있는 카드키(보호 도메인)를 받는다. 만약 투숙객에게 호텔 전체 마스터키를 준다면(과도한 권한), 그 손님이 나쁜 마음을 먹거나 키를 잃어버렸을 때 호텔 전체가 위험에 빠진다. 따라서 딱 필요한 문만 열리는 키를 주는 것(최소 권한)이 가장 안전하다.

- **등장 배경**:
  1. **슈퍼유저 (Root)의 위험성**: 과거 시스템은 관리자 권한 하나로 모든 것을 제어했으나, 관리자 계정 탈취 시 시스템 전체가 붕괴되는 파멸적 결과를 초래했다.
  2. **복잡한 소프트웨어 공급망**: 외부 라이브러리나 써드파티 앱이 늘어나면서, 이들이 시스템 깊숙한 곳까지 들여다보지 못하게 가둘 필요가 생겼다.
  3. **결함 감수성 (Fault Tolerance)**: 버그가 있는 프로그램이 자신의 영역을 넘어 커널이나 다른 프로세스의 데이터를 오염시키는 것을 막아야 했다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              보호 도메인 (Protection Domain)의 논리 구조           │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ Subject: Process P1 ]           [ Objects: System Resources ]│
  │          │                                     ┌──────────┐     │
  │          │        Access Matrix                │  File A  │     │
  │          │    ┌────────────────────┐           └──────────┘     │
  │          ▼    │ Object │ Rights    │                 ▲          │
  │   ┌──────────┐├────────┼───────────┤                 │          │
  │   │ Protection││ File A │ Read      │ ───────────────┘          │
  │   │ Domain D1 ││ Port 80│ Write     │ ───────────────┐          │
  │   └──────────┘├────────┼───────────┤                 │          │
  │               │ SysInfo│ None      │                 ▼          │
  │               └────────┴───────────┘           ┌──────────┐     │
  │                                                │ Port 80  │     │
  │                                                └──────────┘     │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 프로세스 P1은 보호 도메인 D1 내에서 실행된다. 이 도메인은 'File A'에 대해서는 오직 '읽기'만, 'Port 80'에 대해서는 '쓰기'만 허용한다. 만약 P1이 시스템 정보(SysInfo)를 읽으려 한다면, D1의 권한 밖이므로 하드웨어 트랩이 발생하여 접근이 차단된다.

- **📢 섹션 요약 비유**: "자신이 맡은 구역의 열쇠만 가지고 있는 경비원이, 사고가 났을 때 가장 피해를 적게 입히는 유능한 경비원인 것"과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 보호 도메인 구현 방식 및 정적/동적 전이

권한의 집합인 도메인은 고정되어 있을 수도 있고, 상황에 따라 바뀔 수도 있다.

1. **정적 도메인 (Static Domain)**: 프로세스 수명 주기 동안 권한이 변하지 않음. (단순하나 유연성 부족)
2. **동적 도메인 (Dynamic Domain)**: 작업 단계에 따라 권한이 확장되거나 축소됨.
   - **도메인 전이 (Domain Switching)**: 유저 모드 프로세스가 시스템 콜을 통해 일시적으로 커널 모드(더 넓은 도메인)로 권한을 높이는 행위.
3. **SetUID 메커니즘**: 유닉스에서 실행 파일 소유자의 권한으로 잠시 갈아타는 전형적인 도메인 전이 사례. (남용 시 보안 취약점)

### 접근 제어 행렬 (Access Matrix)과 최적화 기법

운영체제는 도메인과 자원 간의 권한을 행렬 형태로 개념화한다.

| 구현 방식 | 설명 및 장단점 | 실무 사례 | 비유 |
|:---|:---|:---|:---|
| **Global Table** | `<Domain, Object, Rights>`의 모든 조합 나열. 크기가 너무 커져서 구현 불가능. | 이론적 개념 | 전체 명부 |
| **ACL (Access Control List)** | **객체** 중심. 파일마다 접근 가능한 도메인 리스트를 붙임. | **ext4, NTFS** | 문 앞의 출입 명부 |
| **Capability List** | **주체** 중심. 프로세스가 가질 수 있는 자원 티켓들을 보유. | **OAuth, Kerberos** | 주머니 속의 열쇠 |
| **Lock-Key 기법** | 자원에 자물쇠(Bit)를 걸고 도메인에 열쇠를 줌. 값이 맞아야 접근. | 하드웨어 메모리 보호 | 암호화 통신 |

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 최소 권한 원칙 (POLP) 적용 전후 비교                 │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ Case A: Root 권한 실행 ]        [ Case B: POLP 적용 실행 ]      │
  │   - 프로세스: 웹 서버               - 프로세스: 웹 서버             │
  │   - 권한: 모든 파일 R/W/X           - 권한: /var/www (R), Log (W)   │
  │                                                                  │
  │   [해킹 사고 발생 시]               [해킹 사고 발생 시]             │
  │   ▶ 해커가 /etc/passwd 탈취         ▶ 해커가 /etc/passwd 접근 시도  │
  │   ▶ 시스템 전체 장악 (Root Shell)   ▶ 권한 거부 (Permission Denied) │
  │   ▶ 파멸적 데이터 증발              ▶ 웹 데이터 외에 추가 피해 없음   │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** POLP의 진가는 사고가 났을 때 발휘된다. Case A는 권한의 경계가 없어 한 점의 뚫림이 곧 전체의 붕괴로 이어지지만, Case B는 보호 도메인이 철저히 격리되어 있어 해커가 웹 서버를 장악해도 그 안에서만 맴돌 뿐 시스템 핵심부로 진입할 수 없다.

- **📢 섹션 요약 비유**: "만능열차 패스보다는 목적지별 편도 승차권이, 부정 승차 사고 시 철도청(시스템)의 손해를 줄여주는 것"과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: DAC vs MAC vs RBAC (접근 제어 모델)

| 모델 | 권한 결정 주체 | 특징 | 보호 도메인 성격 |
|:---|:---|:---|:---|
| **임의적 (DAC)** | 자원 소유자 | `chmod` 등 사용자 재량 | 유연하지만 전파 통제 어려움 |
| **강제적 (MAC)** | 시스템 (커널) 정책 | 사용자도 권한 변경 불가 | 아주 좁고 고정된 격리 도메인 |
| **역할 기반 (RBAC)** | 조직 내 직무(Role) | 인사 명령과 연동 | 직무 단위의 표준화된 도메인 |

### 비교 2: 보안 모델 간의 철학적 차이

- **전통적 보안 (Perimeter Security)**: "성벽(방화벽)만 잘 쌓으면 내부자는 다 믿는다." -> 내부 이동(Lateral Movement)에 취약.
- **제로 트러스트 (Zero Trust)**: "아무도 믿지 마라. 내부자라도 매 순간 권한을 검증하고 최소화하라." -> 현대적인 보호 도메인 및 POLP의 극단적 적용.

- **과목 융합 관점 (네트워크/보안)**: 네트워크의 **마이크로 세그먼테이션 (Micro-segmentation)**이 보호 도메인의 네트워크 버전이다. 서버 한 대가 털려도 같은 서브넷의 다른 서버로 못 가게 막는 것이 최소 권한 원칙의 물리적 구현이다.

- **📢 섹션 요약 비유**: "누가 권한을 주느냐(관리자 vs 소유자)보다, 그 권한이 꼭 필요한 것인가를 따지는 '의심의 미학'이 보안의 본질입니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 웹 서버 (Nginx/Apache) 운영**: 루트 계정(Port 80 바인딩용)으로 실행하되, 실행 직후 즉시 낮은 권한의 계정(www-data)으로 도메인을 전환(Switching)한다. 이는 초기 부팅 시에만 필요한 '특권'을 반납하고 평시 운영에는 '최소 권한'만 유지하는 고전적이고 강력한 기법이다.

2. **시나리오 — 스마트폰 앱 권한 승인**: 안드로이드/iOS에서 앱 설치 시 "카메라 접근", "연락처 접근" 등을 일일이 묻는 것 자체가 보호 도메인을 유저가 직접 정의하는 과정이다. 사진 편집 앱이 주소록 권한을 요구한다면 POLP 위반이므로 거부해야 한다.

3. **시나리오 — 컨테이너 기반 CI/CD 파이프라인**: 빌드 스크립트가 실행되는 컨테이너에 호스트의 `/var/run/docker.sock`을 마운트해주는 행위는 빌드 도구에 호스트 전체 제어권을 주는 것과 같다. 이는 POLP를 정면으로 위반하는 안티패턴이며, 대안으로 권한이 없는(Rootless) 컨테이너 기술을 사용해야 한다.

### 도입 체크리스트 (Security Audit)
- **과도한 특권 확인**: `sudo` 권한이나 `SetUID` 비트가 설정된 바이너리가 불필요하게 많지는 않은가?
- **계정 분리**: 로그 수집, 백업, DB 실행 계정이 모두 별도로 분리되어 각자의 도메인 내에 있는가?
- **임시 권한 폐기**: 특정 작업을 위해 부여된 임시 권한이 작업 종료 후 자동으로 회수되는가? (JIT, Just-In-Time Access)

### 안티패턴
- **귀찮음으로 인한 `chmod 777`**: "권한 오류 나니까 일단 다 풀어!"라는 태도. 보호 도메인 자체를 파괴하고 시스템을 무방비 상태로 노출하는 행위.
- **Root 계정으로 서비스 실행**: 편의를 위해 모든 데몬을 루트로 돌리는 것. 해커에게 고속도로 하이패스 카드를 쥐여주는 것과 같음.

- **📢 섹션 요약 비유**: "불편함(제약)은 보안의 대가가 아니라, 시스템의 생명을 연장하는 정기적인 건강검진과 같습니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (POLP 도입의 가치)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **Blast Radius 축소** | 사고 발생 시 피해 범위를 해당 도메인 내로 한정 | 데이터 유출 80% 차단 |
| **침해 사고 조기 발견** | 비정상적인 권한 요청 시 하드웨어 트랩/로그 발생 | 침투 탐지 시간(MTTD) 단축 |
| **시스템 투명성** | 누가 어떤 자원을 쓰는지 명확히 정의되어 관리 효율 증가 | 거버넌스 체계 확립 |
| **컴플라이언스** | 금융, 공공 등 엄격한 보안 규제 대응 가능 | ISMS 인증 핵심 항목 |

### 미래 전망
- **Self-Healing Protection Domain**: AI가 프로세스의 동작을 실시간 학습하여, 평소와 다른 자원 접근 시 즉시 도메인을 폐쇄하고 자가 복구에 들어가는 지능형 보안 커널.
- **Confidential Computing (TEE)**: 하드웨어 수준에서 보호 도메인을 암호화하여, 운영체제 관리자조차 프로세스 내부 데이터를 볼 수 없게 만드는 극강의 격리 기술 보편화.

### 참고 표준
- **Saltzer & Schroeder (1975)**: "The Protection of Information in Computer Systems" - 최소 권한 원칙의 기원이 된 고전적 논문.
- **Common Criteria (CC)**: 보안 제품 인증 시 보호 도메인의 분리 강도를 주요 평가 항목으로 규정.

- **📢 섹션 요약 비유**: 최소 권한 원칙은 복잡한 IT 세상에서 시스템을 지탱하는 "가장 작지만 가장 단단한 자물쇠"이며, 이 원칙을 지키는 것이 엔지니어의 첫 번째 사명입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 최소 권한 원칙은 **"딱 필요한 열쇠만 나눠주기"** 약속이에요. 요리사에게는 주방 열쇠만 주고, 운전사에게는 자동차 열쇠만 주는 거죠.
2. 만약 요리사가 자동차 열쇠까지 가지고 있다가 나쁜 마음을 먹으면 차를 훔쳐 갈 수도 있잖아요? 그래서 **남의 물건은 건드리지 못하게** 하는 거예요.
3. 이 약속을 잘 지키면, 누군가 열쇠를 하나 잃어버려도 우리 집 전체가 위험해지지 않고 **안전하게 지낼 수** 있답니다!
