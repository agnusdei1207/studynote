# 인터럽트 벡터 테이블 (Interrupt Vector Table) 구조화 및 하드웨어 매핑

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하드웨어 이벤트(IRQ)가 발생했을 때 CPU가 실행해야 할 **인터럽트 서비스 루틴(ISR)의 메모리 주소**를 인덱스 번호별로 저장해 놓은 특수 배열 구조.
> 2. **가치**: 발생한 인터럽트의 종류를 소프트웨어적으로 하나씩 검사(Polling)하지 않고, 하드웨어가 직접 해당 주소로 점프하게 함으로써 **응답 지연(Latency)**을 극단적으로 단축한다.
> 3. **융합**: CPU의 아키텍처(x86의 IDT, ARM의 Vector Table)에 따라 구현 방식이 다르며, 커널 초기화 과정에서 하드웨어와 운영체제 소프트웨어가 처음으로 신뢰 관계를 맺는 지점이다.

---

### Ⅰ. 개요 (Context & Background)

- **정의**: 인터럽트 벡터 테이블(IVT)은 다양한 인터럽트 소스(타이머, 키보드, 디스크 등)에 고유 번호(Vector)를 부여하고, 그 번호에 대응하는 처리 함수(ISR)의 포인터를 담고 있는 테이블이다.
- **💡 비유**: **아파트의 인터폰 단자함**과 같다. 101호에서 호출이 오면 단자함 내부의 101번 선을 따라 경비실의 특정 화면이 켜지듯, 인터럽트 번호만 알면 어떤 코드를 실행해야 할지 즉시 알 수 있는 지도이다.
- **등장 배경**: 초기 컴퓨터는 인터럽트가 발생하면 모든 장치를 다 찔러보며 누가 불렀는지 확인해야 했다. 장치가 많아질수록 이 시간이 길어지자, 각 장치에 고유 번호를 주고 바로 찾아가기 위해 탄생했다.

---

### Ⅱ. 아키텍처 및 내부 구조 (Deep Dive)

#### 1. 인터럽트 처리 흐름 (Hardware to Software)

```ascii
    [ Hardware Device ]        [ CPU Control Unit ]        [ Memory (IVT/IDT) ]
    +-----------------+        +------------------+        +------------------+
    |  Disk Controller|        | 1. Detect Signal |        | Index | Handler  |
    |  (Sends IRQ 14) |------->| 2. Finish Instr. |        |-------|----------|
    +-----------------+        | 3. Get Vector 14 |------->|  14   | 0x804000 |---+
                               | 4. Push Context  |        +------------------+   |
                               +------------------+                               |
                                        ^                                         |
                                        | (Jump to Handler Address)               |
                                        +-----------------------------------------+
                                                              |
                                                   [ Interrupt Service Routine ]
                                                   | void disk_handler() { ... } |
```

#### 2. x86 아키텍처의 IDT (Interrupt Descriptor Table)
현대 x86 시스템에서는 단순 주소 배열을 넘어, 게이트(Gate)라는 기술을 통해 보호 모드 기능을 강화한 IDT를 사용한다.
- **Task Gate**: 작업 전환 유도.
- **Interrupt Gate**: 인터럽트 발생 시 다른 인터럽트를 차단(CLI).
- **Trap Gate**: 다른 인터럽트를 차단하지 않음 (예외 처리용).

### Ⅲ. 구조화 및 매핑 기술 (Technical Details)

#### 1. 벡터 번호 할당 체계 (Standard Mapping)

| 벡터 범위 | 용도 | 예시 |
|:---|:---|:---|
| **0 ~ 31** | **시스템 예외 (Exceptions)** | Zero Division (0), Page Fault (14) |
| **32 ~ 255** | **사용자 정의 / 하드웨어** | Timer (32), Keyboard (33), System Call (128/0x80) |

#### 2. 하드웨어 매핑 방식
- **고정 매핑**: CPU 설계 시 특정 번호가 특정 장치로 고정됨 (주로 임베디드).
- **가변 매핑 (PIC/APIC)**: 인터럽트 컨트롤러를 프로그래밍하여 장치와 벡터 번호 간의 관계를 운영체제가 동적으로 설정함.

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 시나리오: 인터럽트 폭풍 (Interrupt Storm) 대응
- **문제**: 특정 장치 오작동으로 초당 수만 번의 인터럽트가 발생하여 시스템이 마비됨.
- **기술사적 결단**:
  - IVT/IDT에서 해당 벡터의 핸들러를 일시적으로 'Null' 처리하거나 마스킹(Masking)하여 CPU 자원을 보호한다.
  - 근본적으로는 **Interrupt Throttling** 기술을 드라이버 레벨에서 적용하여 단위 시간당 처리 횟수를 제한한다.

#### 2. 기술사적 인사이트
- **보안성 (Rootkit)**: 해커가 IDT의 특정 주소를 자신의 악성 코드 주소로 바꿔치기(IDT Hooking)하면, 모든 시스템 호출을 가로챌 수 있다. 현대 OS는 IDT 영역을 Read-only로 보호한다.
- **실시간성**: RTOS 설계 시 IVT 점프 이후 ISR의 첫 번째 명령이 실행되기까지의 시간(**Interrupt Latency**)을 최소화하는 것이 가장 큰 과제다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **빠른 문맥 전환**: 복잡한 조건문 없이 즉시 분기하여 하드웨어 응답성 확보.
- **모듈화된 개발**: 장치 드라이버 개발자가 커널 전체를 알 필요 없이 특정 벡터에 함수만 등록하면 됨.

#### 2. 미래 전망
최신 하드웨어에서는 인터럽트 핀을 줄이기 위해 **MSI/MSI-X (Message Signaled Interrupts)** 방식을 사용한다. 이는 전기 신호 대신 특정 메모리 주소에 '메시지'를 쓰는 방식인데, 운영체제는 이를 다시 논리적인 벡터로 변환하여 기존 IVT 구조와 하위 호환성을 유지하며 발전하고 있다.
