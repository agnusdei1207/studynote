# 실시간 스케줄링 (마감 시간 / Deadline)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 각 작업에 데드라인을 보장하는 스케줄링 알고리즘
> 2. **가치**: 하드 실시간 보장, 우주항공, 산업 제어
> 3. **융합**: EDF, RM, 우선순위, 선점 스케줄링과 통합

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

실시간 스케줄링(Real-time Scheduling)은 각 작업에 데드라인(deadline)을 보장하는 스케줄링 방식입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    실시간 스케줄링 (Deadline)                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              실시간 시스템 분류                                 │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  Hard Real-time:                                     │   │ │
│   │   │  • 데드라인 미준수 시 치명적 결과                     │   │ │
│   │   │  • 예: 항공우주, 원자력, 의료기기                     │   │ │
│   │   │  • 보장 필수 (Guaranteed)                            │   │ │
│   │   │                                                      │   │ │
│   │   │  Soft Real-time:                                     │   │ │
│   │   │  • 데드라인 미준수 시 품질 저하                        │   │ │
│   │   │  • 예: 비디오 스트리밍, 게임                         │   │ │
│   │   │  • 최선 노력 (Best-effort)                           │   │ │
│   │   │                                                      │   │ │
│   │   │  Firm Real-time:                                     │   │ │
│   │   │  • 데드라인 미준수 시 결과 폐기                        │   │ │
│   │   │  • 예: 멀티미디어 통신                               │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              주요 알고리즘                                     │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  EDF (Earliest Deadline First):                      │   │ │
│   │   │  • 데드라인이 가장 가까운 작업 우선                   │   │ │
│   │   │  • 동적 우선순위                                     │   │ │
│   │   │  • 이론적으로 최적 (100% 활용 가능)                  │   │ │
│   │   │                                                      │   │ │
│   │   │  RM (Rate Monotonic):                                │   │ │
│   │   │  • 주기가 짧은 작업이 높은 우선순위                   │   │ │
│   │   │  • 정적 우선순위                                     │   │ │
│   │   │  • 간단하지만 69% 활용률 제한                        │   │ │
│   │   │                                                      │   │ │
│   │   │  Deadline Scheduling (Linux SCHED_DEADLINE):         │   │ │
│   │   │  • 런타임, 데드라인, 주기 지정                        │   │ │
│   │   │  • CBS (Constant Bandwidth Server) 구현              │   │ │
│   │   │  • 하드 실시간 보장                                  │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> **해설**: EDF는 이론적으로 최적이지만 오버헤드가 있습니다.

**💡 비유**: 실시간 스케줄링은 약속 시간을 반드시 지켜야 하는 배달과 같습니다.

**📢 섹션 요약 비유**: EDF가 최적!

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **Deadline** | 마감 시간 | 절대 시간 | 약속 |
| **Period** | 주기 | 반복 간격 | 배차 |
| **WCET** | 최악 실행 시간 | 보장 필요 | 시간 |
| **Utilization** | 활용률 | sum(Ci/Pi) | 부하 |

**스케줄링 정책**

| 정책 | Linux | 설명 |
|:---|:---|:---|
| **SCHED_FIFO** | RT | FIFO |
| **SCHED_RR** | RT | 라운드 로빈 |
| **SCHED_DEADLINE** | RT | EDF 기반 |

**핵심 알고리즘**: 매 tick → 데드라인 비교 → 최소 선택

**📢 섹션 요약 비유**: SCHED_DEADLINE이 진짜 RT!

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**기술 비교: 알고리즘**

| 알고리즘 | 활용률 | 복잡도 |
|:---|:---:|:---:|
| **RM** | 69% | 낮음 |
| **EDF** | 100% | 높음 |
| **DM** | 69% | 낮음 |

**📢 섹션 요약 비유**: RM이 구현 쉬워요!

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1: 산업 제어** → RTOS + RM
**시나리오 2: 오디오** → SCHED_DEADLINE
**시나리오 3: 우주항공** → 하드 RTOS

**도입 체크리스트**: WCET 분석, 활용률 계산

**📢 섹션 요약 비유**: PREEMPT_RT 패치!

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**정량/정성 기대효과**: 데드라인 100% 보장

**미래 전망**: AUTOSAR, ROS 2 실시간

**참고 표준**: POSIX RT, AUTOSAR

**📢 섹션 요약 비유**: 실시간은 보장이 핵심!
