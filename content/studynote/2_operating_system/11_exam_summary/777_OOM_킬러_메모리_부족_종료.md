# OOM 킬러 (Out-Of-Memory Killer) 및 시스템 보호 메커니즘

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: OOM 킬러는 운영체제의 가용 메모리가 고갈되어 시스템 전체의 패닉(Panic)이나 마비가 우려될 때, 특정 프로세스를 강제 종료(SIGKILL)하여 즉각적인 가용 공간을 확보하는 리눅스 커널의 최후 자가 보호 메커니즘이다.
> 2. **가치**: 메모리 부족 상황에서도 커널과 핵심 서비스의 생존을 보장하며, 정교한 점수 산정 방식(`oom_score`)과 사용자 정의 조정값(`oom_score_adj`)을 통해 서비스의 중요도에 따른 차별화된 희생자 선정 정책을 수립할 수 있게 한다.
> 3. **융합**: 가상 메모리의 **오버커밋 (Overcommit)** 정책, 자원 제어 그룹인 **Cgroups**, 그리고 클라우드 및 컨테이너 환경의 오토 스케일링 전략과 유기적으로 결합되어 인프라의 안정성을 지탱한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: OOM 킬러는 "배가 가라앉기 직전에 가장 무거운 짐을 바다로 던지는 선장"과 같다. 시스템에 램이 부족해지면 운영체제는 더 이상 일을 할 수 없게 되는데, 이때 가장 '나쁜' 프로세스(메모리를 많이 쓰면서 중요도는 낮은 것)를 골라 강제로 죽임으로써 나머지 시스템을 살리는 기술이다.

- **💡 비유**: 이는 산소가 부족한 잠수함의 비상조치와 같다. 모두가 숨을 쉬기에 산소가 모자라면, 잠수함 시스템(커널)은 생존을 위해 가장 산소를 많이 소모하는 불필요한 장치나 인원의 활동(프로세스)을 강제로 중단시켜 핵심 제어 장치가 멈추지 않게 방어하는 것과 같다.

- **등장 배경**:
  1. **오버커밋 (Overcommit)의 부작용**: 리눅스는 효율성을 위해 실제 램보다 더 많은 메모리를 프로세스에게 약속(할당)하는 경향이 있는데, 모든 프로세스가 동시에 그 약속된 메모리를 다 쓰려 할 때 충돌이 발생한다.
  2. **가용성 보장**: 메모리 부족 시 시스템이 그냥 멈춰버리는(Freeze) 것보다, 비핵심 프로세스 하나를 희생시켜서라도 SSH 접속이나 커널 루틴을 유지하는 것이 관리 측면에서 훨씬 유리했다.
  3. **비정상적 메모리 누수 방지**: 버그가 있는 특정 앱이 메모리를 무한정 빨아들여 시스템 전체를 공멸로 몰고 가는 상황을 차단해야 했다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              OOM 킬러의 트리거 및 실행 워크플로우 (Logic)           │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ Memory Request ] ──▶ [ Page Fault ] ──▶ [ No Free Page ]    │
  │                                                │                │
  │          ┌─────────────────────────────────────┘                │
  │          ▼                                                      │
  │   [ 1. Reclaim Attempt ] : 캐시 비우기, 스왑 활용 (실패 시)        │
  │          │                                                      │
  │          ▼                                                      │
  │   [ 2. Select Victim ] : oom_score가 가장 높은 프로세스 선정      │
  │          │                                                      │
  │          ▼                                                      │
  │   [ 3. Send SIGKILL ] : 프로세스 강제 종료 및 메모리 즉시 회수     │
  │          │                                                      │
  │          ▼                                                      │
  │   [ 4. Recovery ] : 시스템 안정화 및 로그 기록 (dmesg)           │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** OOM 킬러는 아무 때나 작동하지 않는다. 운영체제가 할 수 있는 모든 수단(캐시 삭제, 스왑 아웃 등)을 동원해도 메모리 확보가 안 될 때 비로소 '살생부'를 작성한다. 이때 `dmesg` 로그에는 어떤 프로세스가 왜 죽었는지에 대한 상세한 'OOM 리포트'가 남게 된다.

- **📢 섹션 요약 비유**: "모두를 살리기 위해 하나를 버려야만 하는 운영체제의 비정한, 그러나 필수적인 생존 결단"입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 희생자 선정을 위한 점수 산정: oom_score

커널은 `/proc/[PID]/oom_score`에 실시간으로 점수를 매긴다. 점수가 높을수록 죽을 확률이 높다.

| 산정 요소 | 가중치 로직 | 기술적 이유 | 비유 |
|:---|:---|:---|:---|
| **메모리 점유율** | 전체 메모리 대비 사용 비율 | 많이 뺏어와야 시스템이 빨리 사니까 | 가장 큰 짐가방 |
| **프로세스 수명** | 오래된 프로세스 우대 | 오랫동안 잘 돌아간 프로세스는 중요할 확률이 높음 | 짬밥 대우 |
| **특권 계정** | Root 소유 프로세스 우대 | 시스템 관리용일 가능성이 큼 | 관리직 보호 |
| **직접적 원인** | 메모리 부족을 직접 유발한 놈 | 범인 검거 원칙 | 사고 친 당사자 |
| **oom_score_adj** | **사용자 수동 조정값** | 관리자가 서비스 중요도를 직접 지정 가능 | VIP 권한 |

### oom_score_adj의 정교한 튜닝

관리자는 특정 프로세스를 OOM으로부터 보호하거나, 반대로 1순위 희생양으로 만들 수 있다.

1. **범위**: -1000 ~ +1000 사이의 정수.
2. **-1000 (OOM_SCORE_ADJ_MIN)**: "절대 죽이지 마라." (예: sshd, 핵심 DB)
3. **+1000 (OOM_SCORE_ADJ_MAX)**: "메모리 부족하면 이 녀석부터 죽여라."
4. **0 (Default)**: 커널의 기본 알고리즘에 맡김.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 메모리 오버커밋(Overcommit)과 OOM의 관계            │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   vm.overcommit_memory 설정값:                                   │
  │   - 0 (Heuristic): 커널이 대충 봐서 적당히 할당 (기본값)           │
  │   - 1 (Always): 묻지도 따지지도 않고 다 빌려줌 -> OOM 위험 높음    │
  │   - 2 (Never): 실제 램+스왑 한도 내에서만 할당 -> OOM 방지         │
  │                                                                  │
  │   [가치] 0번이나 1번을 쓰면 램 활용도는 좋지만, OOM 킬러와 친해짐. │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 리눅스가 "부도날 확률이 있는 어음(Overcommit)"을 발행하는 이유는, 실제 프로그램들이 메모리를 신청만 해놓고 다 안 쓰는 경우가 많기 때문이다. 하지만 모든 프로세스가 어음을 현금으로 바꾸려 하면(실제 메모리 사용), OOM 킬러라는 파산 절차가 가동된다.

- **📢 섹션 요약 비유**: "평소에는 넉넉하게 대출(오버커밋)을 해주다가, 은행 잔고가 바닥나면 가장 등급이 낮은 고객(oom_score 고점자)의 계좌부터 동결하는 냉혹한 금융 시스템"과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: OOM Killer vs Swap (메모리 부족 대응책)

| 비교 항목 | 스왑 (Swap Space) | OOM 킬러 (OOM Killer) |
|:---|:---|:---|
| **성격** | 완만하고 평화로운 대응 | **급진적이고 파괴적인 대응** |
| **수단** | 디스크의 일부를 메모리로 사용 | 프로세스 강제 종료 |
| **성능** | 시스템이 매우 느려짐 (Thrashing 위험) | 죽은 프로세스 외에는 성능 회복 |
| **데이터** | 보존됨 | **유실됨 (비정상 종료)** |
| **비유** | 대출 받아서 버티기 | 사업장 하나 폐쇄하기 |

### 비교 2: Cgroups 기반의 OOM (Container OOM)

- **Global OOM**: 시스템 전체 램이 부족할 때 발생.
- **Cgroup OOM**: 특정 컨테이너 그룹에 할당된 메모리 제한(`memory.limit_in_bytes`)을 넘었을 때 발생. 시스템 전체는 여유가 있어도 해당 컨테이너 내의 프로세스만 골라 죽임. (도커의 일반적인 상황)

- **과목 융합 관점 (데이터베이스)**: DB 엔진은 OOM 킬러의 1순위 타겟이다. 메모리를 대량으로 점유하기 때문이다. 이를 방지하기 위해 DBA는 DB 프로세스의 `oom_score_adj`를 낮추고, 대신 DB 내부의 **`Memory Manager`**가 쿼리를 스스로 취소하게 하거나 캐시를 비우도록 이중 방어 체계를 구축한다.

- **📢 섹션 요약 비유**: "집 전체가 무너지는 것(Global OOM)을 막기 위해 방 하나만 철거하는 것(Cgroup OOM)은 클라우드 시대의 필수적인 국소 방어 전략입니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — SSH 접속 불가 장애**: 서버가 응답은 하는데 SSH 접속이 안 됨. 원인은 메모리 고갈 시 OOM 킬러가 `sshd` 프로세스를 희생자로 골라 죽였기 때문. 기술사적 판단으로 **sshd의 `oom_score_adj`를 -1000으로 고정**하는 설정을 적용하여, 관리 통로만큼은 어떤 극한 상황에서도 열려 있도록 조치한다.

2. **시나리오 — 쿠버네티스 포드(Pod) 무한 재시작**: 특정 앱이 5분마다 죽고 다시 살아남. 로그 확인 시 `Reason: OOMKilled` 메시지 발견. 해결책으로 컨테이너의 **Memory Request/Limit**을 실제 사용량에 맞춰 상향 조정하고, 앱 내부의 메모리 누수(Leak) 지점을 프로파일링하여 근본 원인을 제거한다.

3. **시나리오 — 자바(JVM) 힙 메모리 설정 오류**: `-Xmx`를 물리 램보다 크게 설정함. JVM 내부 GC는 돌지도 않았는데 OS가 자바 프로세스를 죽임. 원인은 JVM이 OS로부터 실제 페이지를 할당받는 과정에서 OOM 트리거. 해결책으로 **`Xmx` 값을 물리 램의 70~80% 수준으로 제한**하여 OS와 JVM 간의 메모리 주도권 충돌을 해결한다.

### 도입 체크리스트 (System Hardening)
- **중요 서비스 보호**: DB, 가상화 엔진, 원격 관리 툴의 `oom_score_adj`가 적절히 낮게 설정되었는가?
- **모니터링 알람**: `dmesg`나 `/var/log/syslog`에서 "Out of memory" 키워드를 실시간 감시하고 있는가?
- **Panic 설정**: 메모리 부족 시 프로세스를 죽이는 대신 시스템을 재부팅할 것인가? (`vm.panic_on_oom` 설정 확인)

### 안티패턴
- **무조건적인 스왑 확장**: 램이 부족하다고 스왑을 수백 GB 늘리는 행위. 서버가 죽지는 않지만 서비스 응답이 수 초 이상 걸리는 '산송장' 상태가 되어 사용자 이탈을 초래함. (차라리 빨리 죽고 재시작되는 게 나을 수 있음)
- **OOM 킬러 비활성화 시도**: 억지로 킬러를 끄려다 커널 패닉으로 전체 서비스가 다운되는 파국을 맞는 행위.

- **📢 섹션 요약 비유**: "나쁜 소식(OOM Kill)을 전하는 전령을 죽이지 말고, 왜 전령이 오게 되었는지(Resource Leak) 원인을 파악하는 것이 진정한 엔지니어링입니다."

| **운영 가시성** | OOM 리포트를 통한 비정상 앱 식별 및 개선 근거 확보 | 장애 원인 분석 시간 단축 |

### 미래 전망
- **AI-Driven OOM Prevention**: 커널이 앱의 메모리 증가 곡선을 학습하여, OOM이 발생하기 수 분 전에 미리 관리자에게 경고하거나 자원을 동적으로 늘리는 지능형 관리.
- **Memory Tiering (CXL)**: 램이 부족할 때 프로세스를 죽이는 대신, 네트워크 너머의 공유 메모리 풀에서 즉시 메모리를 빌려와 OOM 자체를 지연시키는 기술 표준화.

### 참고 표준
- **Linux Kernel `oom_kill.c`**: 리눅스 커널의 공식 OOM 구현 표준.
- **Kubernetes Quality of Service (QoS) Class**: 포드의 중요도에 따라 OOM 점수를 차등 부여하는 클라우드 표준 정책 (Guaranteed > Burstable > BestEffort).

- **📢 섹션 요약 비유**: OOM 킬러는 "시스템이라는 생태계가 무너지지 않도록 개체 수를 조절하는 자연의 섭리"와 같으며, 이를 잘 이해하고 다스리는 것이 안정적인 시스템 운영의 핵심입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **슬랩 (Slab) 할당기**: 커널 메모리 부족 시 OOM 킬러가 보호해야 할 커널의 핵심 자산.
- **cgroups (제어 그룹)**: OOM 킬러의 사격 범위를 컨테이너 단위로 좁혀주는 기술.
- **클라우드 가상화**: 수많은 가상 자원이 엉켜 OOM 위험이 상존하는 거대 생태계.
- **프로세스 친화성 (Affinity)**: 특정 CPU에 갇힌 프로세스가 메모리까지 독점하려 할 때 OOM과 연계됨.

### 👶 어린이를 위한 3줄 비유 설명
