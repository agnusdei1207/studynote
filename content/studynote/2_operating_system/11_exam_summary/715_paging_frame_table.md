# 페이징 시스템의 프레임 테이블 (Frame Table)과 역 페이지 테이블

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 물리 메모리의 각 **프레임(Frame)**이 현재 어떤 프로세스의 어떤 페이지에 할당되어 있는지, 혹은 비어 있는지를 추적하는 운영체제 커널의 관리용 자료구조.
> 2. **가치**: 물리 자원의 실시간 상태를 파악하여 새로운 메모리 할당 요청을 처리하고, 시스템 전체의 가용 물리 공간을 효율적으로 관리한다.
> 3. **융합**: 프로세스별로 존재하는 일반 페이지 테이블과 달리 **시스템 전체에 단 하나만 존재**하며, 메모리 낭비를 줄이기 위한 '역 페이지 테이블(Inverted Page Table)' 설계의 기초가 된다.

---

### Ⅰ. 프레임 테이블 (Frame Table)의 역할

- **개념**: 물리 메모리를 고정 크기로 나눈 단위를 '프레임'이라 하며, 이 프레임들의 상태를 기록한 장부가 프레임 테이블이다.
- **기록 정보**:
  - 프레임의 사용 여부 (Free / Allocated)
  - 할당된 경우: 어느 프로세스(PID) 소유인지
  - 해당 프로세스의 몇 번째 페이지(Page Number)인지

---

### Ⅱ. 역 페이지 테이블 (Inverted Page Table) 아키텍처 (ASCII)

일반적인 페이지 테이블은 프로세스마다 존재하여 메모리를 많이 차지한다. 이를 해결하기 위해 물리 메모리 프레임 기준으로 테이블을 구성한 것이 역 페이지 테이블이다.

```ascii
    [ Logical Address ]
    ( PID, Page Number, Offset )
           |
           v
    [ Inverted Page Table Search ]
    +-------+-------+------------------+
    | Index |  PID  |  Page Number     |
    +-------+-------+------------------+
    |   0   |  101  |      5           |
    |   1   |  205  |      2           | <-- Match Found at Index 1
    |  ...  |  ...  |     ...          |
    +-------+-------+------------------+
           |
           v (Index 1 becomes Frame Number)
    [ Physical Memory ]
    +-----------------+
    | Frame 0 (P101)  |
    | Frame 1 (P205)  | <--- Actual Data Access
    +-----------------+
```

**[특징]**
- **장점**: 페이지 테이블을 위한 메모리 공간을 획기적으로 절약함 (물리 메모리 크기에 비례).
- **단점**: 주소 변환 시 테이블 전체를 탐색해야 하므로 속도가 느림. (해시 테이블로 보완 필수)

---

### Ⅲ. 일반 페이지 테이블 vs 역 페이지 테이블

| 구분 | 일반 페이지 테이블 (Forward) | 역 페이지 테이블 (Inverted) |
|:---|:---|:---|
| **개수** | 프로세스당 1개 | **시스템당 딱 1개** |
| **인덱스** | 페이지 번호 (Page Number) | **프레임 번호 (Frame Number)** |
| **메모리 소모** | 높음 (프로세스 많을수록 증가) | **낮음** (물리 RAM 크기에만 의존) |
| **검색 속도** | 빠름 (직접 인덱싱) | 느림 (순차 검색, 해시 필요) |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 대용량 주소 공간 (64비트)에서의 선택
- **문제**: 64비트 시스템에서 일반적인 계층형 페이지 테이블은 그 자체로 수 기가바이트의 RAM을 소모할 수 있다.
- **기술사적 결단**: 
  - 물리 메모리 용량이 상대적으로 적고 프로세스가 매우 많은 경우 **역 페이지 테이블**이나 **해시 페이지 테이블** 도입을 검토한다.
  - 현대의 범용 데스크톱/서버는 성능을 위해 여전히 다단계 페이지 테이블과 TLB를 주력으로 사용한다.

#### 2. 기술사적 인사이트
- **공유 메모리 구현의 어려움**: 역 페이지 테이블은 물리 프레임 하나당 하나의 (PID, Page)만 매핑하므로, 여러 프로세스가 같은 프레임을 공유하는 '공유 메모리' 구현이 매우 복잡해지는 설계적 Trade-off가 있음을 인지해야 한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **시스템 가용 메모리 증대**: 커널 자료구조 공간을 아껴 사용자 앱에 더 많은 메모리 할당 가능.
- **확장성 확보**: 수만 개의 프로세스가 실행되는 환경에서도 안정적인 메모리 관리 오버헤드 유지.

#### 2. 미래 전망
메모리 용량이 테라바이트(TB) 단위로 늘어남에 따라, 주소 변환 정보를 효율적으로 압축하여 관리하는 역 페이지 테이블의 변형 기법들이 재조명받고 있다. 또한, 비휘발성 메모리(NVDIMM)와 같이 물리 주소 공간이 극도로 넓어지는 차세대 아키텍처에서 프레임 관리의 중요성은 더욱 커질 것이다.
