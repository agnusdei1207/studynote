# 데드락 희생자 선정 및 롤백 복구 (Deadlock Victim & Rollback)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데드락 희생자 선정 및 롤백은 교착 상태(Deadlock) 탐지 시, 시스템을 정상 상태로 복구하기 위해 특정 프로세스를 희생자(Victim)로 선정하여 점유 자원을 강제 회수하고, 프로세스의 상태를 안전한 이전 시점(Checkpoint)으로 되돌리는 최후의 수단이다.
> 2. **가치**: 데드락 예방이나 회피가 불가능한 복잡한 시스템 환경에서 시스템 가용성을 최종적으로 확보하며, 비용 기반(Cost-based) 희생자 선정 알고리즘을 통해 복구 시 발생하는 자원 낭비와 시간 손실을 최소화한다.
> 3. **융합**: 데이터베이스의 트랜잭션 복구 모델(ARIES 등), 고신뢰 시스템의 체크포인팅(Checkpointing) 기술, 그리고 분산 시스템의 분산 커밋 및 타임아웃 기반 복구 전략과 기술적 철학을 공유한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 데드락 복구는 "누가 양보할 것인가"를 결정하는 과정이다. 여러 프로세스가 서로 자원을 꽉 쥐고 놓지 않을 때, 운영체제가 개입하여 가장 만만한 녀석(희생자)을 골라 그가 가진 자원을 뺏고, 그를 과거의 안전했던 시점(롤백)으로 타임머신을 태워 보내는 작업이다.

- **💡 비유**: 이는 좁은 골목길에서 대치 중인 차들 중 한 대를 고르는 것과 같다. 뒤로 빼기 가장 쉬운 경차(희생자)를 골라, 근처 넓은 공터(체크포인트)까지 후진하게 한 뒤 다른 차들을 먼저 지나가게 하는 것과 같은 원리다.

- **등장 배경**:
  1. **예방/회피의 고비용**: 모든 자원 요청을 미리 검사(은행가 알고리즘)하는 것은 시스템 성능을 심각하게 저하시키므로, 일단 데드락을 허용하고 나중에 고치는 '탐지 및 복구' 전략이 실무적으로 선호되었다.
  2. **트랜잭션의 원자성 요구**: 데이터베이스 등에서 작업 도중 오류가 나면 '없었던 일'로 만들어야 하는 원자성(Atomicity) 보장 요구가 파일 시스템과 운영체제 전반으로 확산되었다.
  3. **기아 현상(Starvation) 방지**: 복구 과정에서 특정 프로세스만 반복적으로 희생되는 불평등을 막기 위한 정교한 정책이 필요해졌다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              데드락 탐지 및 희생자 선정 프로세스 (Logic Flow)       │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   1. [ Detection ] : 자원 할당 그래프에서 사이클 탐지              │
  │          │                                                      │
  │          ▼                                                      │
  │   2. [ Victim Selection ] : 복구 비용이 가장 적은 프로세스 선택     │
  │      - 우선순위가 낮은가?                                        │
  │      - 지금까지 수행한 시간이 짧은가?                             │
  │      - 점유한 자원이 많은가? (뺏었을 때 이득이 큼)                 │
  │          │                                                      │
  │          ▼                                                      │
  │   3. [ Resource Preemption ] : 희생자의 자원을 회수하여 할당       │
  │          │                                                      │
  │          ▼                                                      │
  │   4. [ Rollback ] : 희생자를 체크포인트 시점으로 복구/재시작        │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 단순히 프로세스를 죽이는 것(`Kill`)은 가장 원시적인 복구다. 세련된 시스템은 롤백을 선택한다. 롤백은 과거의 세이브 포인트(Checkpoint)로 돌아가서 다시 시작하게 함으로써, 이미 수행했던 작업의 일부를 보존하고 데이터 무결성을 유지한다.

- **📢 섹션 요약 비유**: "모두가 멈춰버린 교차로에서, 교통경찰(OS)이 가장 후진하기 편한 차를 골라 뒤로 빼게 함으로써 소통을 재개시키는 과정"입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 희생자 선정을 위한 비용 함수 (Cost Function)

운영체제는 주관이 아닌 객체화된 수치를 바탕으로 희생자를 고른다.

| 고려 요소 | 상세 기준 | 선택 가중치 | 비유 |
|:---|:---|:---|:---|
| **우선순위 (Priority)** | 시스템 중요도 및 사용자 등급 | 낮을수록 우선 희생 | 직급이 낮은 사원 |
| **수행 시간 (Time Spent)** | 지금까지 CPU를 얼마나 썼는가? | 짧을수록 우선 희생 (버려도 아깝지 않음) | 신입 사원 |
| **잔여 시간 (Time to finish)** | 완료까지 남은 시간 | 길수록 우선 희생 | 갈 길이 먼 사람 |
| **점유 자원 수** | 뺏어올 수 있는 자원의 양 | 많을수록 우선 희생 (데드락 해결 확률↑) | 짐이 많은 사람 |
| **회수 횟수 (Rollback Count)** | 이전에 희생된 적이 있는가? | 적을수록 우선 희생 (**기아 방지**) | 처음 걸린 사람 |

---

### 롤백(Rollback)의 3단계 수준

프로세스의 상태를 어디까지 되돌릴 것인가에 따라 구현 난이도가 달라진다.

1. **Total Rollback (완전 롤백)**: 프로세스를 아예 종료하고 처음부터 다시 시작함. 가장 간단하지만 비용이 큼.
2. **Partial Rollback (부분 롤백)**: 주기적으로 저장해둔 **체크포인트 (Checkpoint)** 중 가장 최근으로 복구. 효율적이지만 체크포인팅 오버헤드 발생.
3. **Instruction Level (명령어 단위)**: 하드웨어가 지원하는 트랜잭셔널 메모리를 사용하여 명령 한 줄 단위로 복구. 극강의 성능이나 구현이 매우 복잡.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 체크포인팅과 롤백의 상관관계                         │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   Process State:  [S1] ───▶ [S2] ───▶ [S3] ──▶ [Deadlock!]       │
  │                    │         │         │                         │
  │   Checkpoints:   CP 1      CP 2      CP 3                        │
  │                                        │                         │
  │   Rollback Path:  ◀────────────────────┘                         │
  │                                                                  │
  │   [분석] CP 3으로 돌아가면 CP 3 ~ Deadlock 구간의 작업만 손실됨.     │
  │          CP가 촘촘할수록 손실은 적으나 저장 오버헤드는 큼.            │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 롤백 복구망의 핵심 인프라는 '체크포인트'다. 시스템이 살아있을 때 건강 검진을 하듯 상태를 미리 찍어두어야(Snapshots), 데드락이라는 중병에 걸렸을 때 과거의 건강한 모습으로 되돌릴 수 있다.

- **📢 섹션 요약 비유**: "게임에서 보스 몬스터(데드락)를 만나 죽더라도, 마지막 세이브 포인트(체크포인트)에서 다시 시작할 수 있는 기회를 주는 것"과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 프로세스 종료 (Abort) vs 자원 선점 (Preemption)

| 비교 항목 | 프로세스 종료 (Abort) | 자원 선점 (Preemption) |
|:---|:---|:---|
| **복구 방식** | 관련된 모든 프로세스 또는 일부를 강제 종료 | 자원만 뺏어서 다른 프로세스에 할당 |
| **데이터 손실** | 매우 큼 (처음부터 다시) | 보통 (롤백 지점 이후만 손실) |
| **구현 난이도** | 매우 쉬움 (`kill -9`) | 매우 어려움 (상태 저장/복원 필요) |
| **적합성** | 일반 범용 OS (Windows, Linux) | **고신뢰성 DBMS, 분산 컴퓨팅** |

### 비교 2: 자원 회수 시의 기아(Starvation) 문제

- **문제**: 비용 함수에만 의존하면 복구 비용이 가장 적은 특정 프로세스가 '영원히' 희생자로 선정되어 절대 일을 마치지 못할 수 있음.
- **해결 (Aging)**: 희생자로 선정될 때마다 'Rollback Count'를 증가시키고, 이 수치가 높을수록 다음 선정 시 가중치를 낮추어(우선순위 보호) 언젠가는 일을 끝낼 수 있게 보장함.

- **과목 융합 관점 (데이터베이스)**: 데드락 복구는 DB의 **회복 (Recovery)** 기법과 일맥상통한다. 특히 로그를 이용한 **Undo/Redo** 연산은 롤백의 구체적인 구현체다. 시스템은 로그(WAL)를 통해 이전 상태로 되돌리고(Undo), 다시 안전하게 실행(Redo)하는 과정을 반복한다.

- **📢 섹션 요약 비유**: "단순히 전원을 껐다 켜는 것(Abort)이 해결책이 될 수 있지만, 똑똑한 관리자는 어디까지 했는지 장부(Log)를 보고 그 부분만 다시 하는 법(Preemption & Rollback)입니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 엔터프라이즈 DBMS (Oracle/PostgreSQL) 데드락**: 두 트랜잭션이 서로 락을 잡고 대치함. DB 엔진은 수 초 내에 이를 감지하고, **'Undo 비용'**이 가장 적은 쿼리를 희생자로 골라 `Deadlock Detected` 에러와 함께 롤백시킴. 기술사적 판단으로 애플리케이션 레벨에서 이 에러 발생 시 자동 재시도(Retry) 로직을 갖추도록 아키텍처를 설계한다.

2. **시나리오 — 항공우주 제어 소프트웨어**: 실시간 데이터 처리 중 자원 경합 발생. 작업의 연속성이 생명이므로 프로세스를 죽일 수 없음. 해결책으로 **체크포인팅 기반의 부분 롤백**을 적용하여, 100ms 이전의 안전한 센서 데이터 상태로 복귀시킨 뒤 재실행함으로써 미션의 연속성을 유지함.

3. **시나리오 — 분산 마이크로서비스 (Saga Pattern)**: 여러 서비스에 걸친 분산 트랜잭션 중 데드락과 유사한 타임아웃 발생. 원자적 롤백이 불가능하므로 **보상 트랜잭션 (Compensating Transaction)**을 실행하여 이미 수행된 작업을 논리적으로 취소(롤백)하는 복구망을 구축함.

### 도입 체크리스트 (System Resiliency)
- **희생자 선정 정책**: 기아 현상을 방지하기 위한 '희생 횟수 가중치'가 포함되어 있는가?
- **체크포인트 주기**: 저장 비용(I/O)과 복구 시간 사이의 최적 균형점이 설정되었는가?
- **자원 원자성**: 자원을 회수한 후에도 시스템의 다른 부분이 오염되지 않도록 완벽히 분리(Isolation)되어 있는가?

### 안티패턴
- **무조건적인 `First-come Victim`**: 사이클 내에서 가장 먼저 발견된 프로세스를 죽이는 것. 시스템의 핵심 프로세스가 희생되어 전체 시스템이 가동 불능(Panic)에 빠질 수 있음.
- **로그 없는 롤백 시도**: 상태 저장 없이 롤백을 하려다 메모리 정합성이 깨져 '알 수 없는 오류'를 양산하는 행위.

- **📢 섹션 요약 비유**: "누구를 희생시킬지 결정하는 것은 비정한 일이 아니라, 나머지 모두를 살리기 위한 가장 숭고한 엔지니어링적 결단입니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (복구망 구축 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **시스템 가용성** | 데드락 발생 시 수 초 내 자가 치유(Self-healing) | 다운타임 99% 감소 |
| **데이터 무결성** | 롤백을 통한 비정상 데이터 상태 원천 차단 | 정합성 100% 보장 |
| **운영 효율** | 관리자의 수동 개입(Restart) 없는 자동 복구 체계 | OPEX 절감 |
| **성능 예측성** | 탐지-복구 주기를 관리하여 일정한 처리량 유지 | 서비스 수준(SLA) 확보 |

### 미래 전망
- **AI 기반 선제적 희생자 예측**: 데드락이 발생하기 직전의 징후를 머신러닝으로 감지하여, 실제 데드락 전에 가장 저비용의 프로세스를 미리 정지(Pause)시키는 지능형 스케줄러 등장.
- **Hardware Transactional Memory (HTM) 복구**: CPU 하드웨어가 메모리 롤백을 나노초 단위로 처리하여, 소프트웨어 오버헤드 없는 데드락 복구망 실현.

### 참고 표준
- **SQL-92 Standard**: 트랜잭션 롤백 및 데드락 에러 핸들링 표준.
- **IEEE 754 / POSIX**: 시스템 예외 처리 및 프로세스 제어 표준 인터페이스.

- **📢 섹션 요약 비유**: 데드락 희생자와 롤백은 "넘어진 곳에서 다시 일어날 수 있게 해주는 시스템의 회복 탄력성 (Resilience)"이며, 이는 완벽하지 않은 세상에서 완벽한 서비스를 만드는 유일한 방법입니다.

- **[트랜잭션 ACID](../../5_database/xx_transaction_acid.md)**: 롤백 기술이 궁극적으로 보호하고자 하는 데이터의 신성한 가치.

### 👶 어린이를 위한 3줄 비유 설명
1. 데드락 복구는 친구들이 장난감을 서로 안 놓아주려고 싸울 때, 선생님이 나타나서 **"잠시 한 명만 뒤로 물러나자"**라고 정해주는 거예요.
2. 이때 선생님은 가장 뒤로 가기 쉬운 친구(희생자)를 골라, 아까 놀던 **"안전한 곳(체크포인트)"**으로 데려다준답니다.
3. 그러면 나머지 친구들이 먼저 놀고, 나중에 물러났던 친구도 다시 와서 즐겁게 놀 수 있어서 **아무도 울지 않게** 되는 거랍니다!
