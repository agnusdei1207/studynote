# CPU 프로세스 친화성 (CPU Affinity) 및 성능 최적화

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPU 친화성 (Affinity)은 특정 프로세스나 스레드가 실행될 CPU 코어를 명시적으로 지정하여, 운영체제 스케줄러가 작업을 다른 코어로 임의 이동(Migration)시키는 것을 방지하는 하드웨어-소프트웨어 바인딩 기술이다.
> 2. **가치**: CPU 로컬 캐시(L1/L2)의 데이터 지역성(Locality)을 극대화하여 캐시 미스를 줄이고, NUMA 노드 간의 원격 메모리 접근 지연을 원천 차단함으로써 시스템의 확정적 응답성과 최고 성능을 인출한다.
> 3. **융합**: 멀티코어 스케줄링 알고리즘, 고성능 네트워크 패킷 처리(DPDK, RSS), 그리고 실시간 운영체제(RTOS)의 태스크 격리 전략과 유기적으로 결합되어 미션 크리티컬 시스템의 안정성을 보장한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: "전담 CPU 지정제"다. 일반적인 상황에서는 운영체제가 노는 CPU 코어에 프로세스를 아무렇게나 던져주지만, 친화성을 설정하면 "너는 0번 코어에서만 일해!"라고 못 박아두는 것이다. 이로 인해 프로세스는 이사(Migration)를 가지 않고 한 코어에 정착하게 된다.

- **💡 비유**: 이는 도서관에서 매번 자리를 바꾸는 것(Default)과 전용 지정석(Affinity)을 갖는 것의 차이다. 매일 자리가 바뀌면 어제 보던 책과 노트를 매번 옮겨야 하지만(Cache Pollution), 지정석이 있으면 어제 펼쳐둔 책(Cache Data)이 그대로 있어 자리에 앉자마자 즉시 공부(연산)를 시작할 수 있는 것과 같다.

- **등장 배경**:
  1. **캐시 오염 (Cache Thrashing)**: 프로세스가 코어를 옮겨 다닐 때마다 이전 코어의 L1/L2 캐시에 저장된 데이터가 무용지물이 되어 성능이 급락하는 현상이 발생했다.
  2. **NUMA 아키텍처의 복잡성**: 서버의 램(RAM)이 CPU별로 나뉘어 있어, 0번 CPU가 1번 CPU 근처의 램을 읽을 때 발생하는 지연 시간(Latency)이 무시 못 할 수준이 되었다.
  3. **실시간성 확보**: 특정 코어를 오직 하나의 중요한 작업(예: 자율주행 제어)에만 할당하여, 다른 작업의 간섭을 원천 배제해야 하는 요구가 커졌다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              CPU 친화성 적용 전 vs 적용 후 실행 흐름 비교           │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ 1. No Affinity ]               [ 2. With Affinity ]         │
  │   (Scheduler's Choice)             (Fixed Core Binding)         │
  │                                                                 │
  │   Time T1: Run on Core 0           Time T1: Run on Core 0       │
  │   Time T2: Move to Core 2 ◀─[!]    Time T2: Run on Core 0       │
  │   Time T3: Move to Core 1          Time T3: Run on Core 0       │
  │                                                                 │
  │   [부작용]                         [이점]                       │
  │   - Cache Flush 발생               - L1/L2 Cache Warm 유지      │
  │   - TLB Miss 증가                  - Context Switch 비용 감소    │
  │   - NUMA 원격 접근 발생             - 지터(Jitter) 제거          │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 친화성이 없으면 운영체제는 부하 균등화(Load Balancing)를 위해 프로세스를 이 코어 저 코어로 마구 옮긴다. 이 '이사' 과정에서 CPU 내부의 귀중한 캐시 데이터가 모두 폐기된다. 친화성을 설정하면 프로세스가 한 코어에 상주하며 캐시를 '따뜻하게(Warm)' 유지하여 연산 속도를 비약적으로 높인다.

- **📢 섹션 요약 비유**: "자리를 자주 옮기면 짐을 싸고 푸는 데 시간을 다 쓰지만, 한자리에 진득하게 앉아 있으면 오직 일에만 집중할 수 있는 원리"입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CPU 친화성의 두 가지 형태

| 구분 | 소프트 친화성 (Soft Affinity) | 하드 친화성 (Hard Affinity) |
|:---|:---|:---|
| **동작 원리** | 스케줄러가 "가급적" 같은 코어에 배정하려 노력함 | 프로세스가 특정 코어에서만 실행되도록 **강제**함 |
| **유연성** | 높음 (코어 과부하 시 다른 곳 이동) | 낮음 (지정 코어 바쁘면 대기) |
| **제어 방법** | OS 내부 알고리즘 (사용자 개입 불가) | `taskset`, `sched_setaffinity` 시스템 콜 |
| **비유** | "되도록 단골집 가기" | "무조건 지정된 지점만 가기" |

---

### 핵심 구현체: CPU 비트마스크 (CPU Mask)

커널은 어떤 프로세스가 어떤 CPU를 쓸 수 있는지 '비트맵'으로 관리한다.

1. **8비트 마스크 예시**: `00000001` (0번 코어만), `00000011` (0, 1번 코어), `11111111` (전체 코어).
2. **동적 변경**: 프로세스 실행 도중에도 관리자는 이 비트마스크를 수정하여 실시간으로 실행 코어를 옮기거나 고정할 수 있다.
3. **스케줄러 연동**: 커널 스케줄러는 준비 큐(Ready Queue)에서 프로세스를 꺼낼 때, 해당 프로세스의 CPU 마스크를 확인하여 허용된 CPU의 큐에만 집어넣는다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 NUMA 노드와 CPU Affinity의 시너지                  │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ NUMA Node 0 ]                  [ NUMA Node 1 ]               │
  │   ┌────────┐   ┌────────┐          ┌────────┐   ┌────────┐       │
  │   │ CPU 0  │   │ CPU 1  │          │ CPU 2  │   │ CPU 3  │       │
  │   └────┬───┘   └────┬───┘          └────┬───┘   └────┬───┘       │
  │        ▼            ▼                   ▼            ▼           │
  │   [ Local Memory 0 ]               [ Local Memory 1 ]            │
  │                                                                  │
  │   [최적화] CPU 0에 실행되는 프로세스는 Memory 0에 데이터를 배치하고 │
  │           Affinity를 통해 CPU 0/1 밖으로 나가지 않게 격리함.       │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 현대 서버는 여러 개의 NUMA 노드로 구성된다. CPU 0이 옆방인 Memory 1의 데이터를 읽어오려면 멀리 돌아가야 해서 속도가 떨어진다. 친화성을 통해 프로세스를 CPU 0에 가두고, 메모리 할당도 Node 0에서만 받게 하면(Memory Affinity), 마치 거대한 서버를 작고 빠른 여러 대의 독립 컴퓨터처럼 쓸 수 있다.

- **📢 섹션 요약 비유**: "자기 방에 냉장고(메모리)를 둔 사람이 거실 건너 주방까지 갈 필요가 없도록 동선을 최적화하는 방 가두기 전략"과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: CPU Affinity vs 프로세스 우선순위 (Priority)

| 비교 항목 | CPU 친화성 (Affinity) | 우선순위 (Priority/Nice) |
|:---|:---|:---|
| **제어 대상** | **실행 장소** (어디서?) | **실행 순서** (언제?) |
| **주요 목적** | 캐시 효율, NUMA 최적화 | 긴급도 처리, 공평성 조절 |
| **성능 영향** | 캐시 적중률 향상 (물리적 가속) | 대기 시간 감소 (논리적 가속) |
| **비유** | 전용 작업대 배정 | 새치기 허용권 |

### 비교 2: `taskset` vs `cpuset (cgroups)`

- **`taskset`**: 개별 프로세스 단위로 즉각적인 친화성을 부여할 때 사용. (간편함)
- **`cpuset`**: 컨테이너나 대규모 프로세스 그룹에 자원을 할당할 때 사용. 메모리 노드 제한과 병행 가능. (강력함)

- **과목 융합 관점 (네트워크)**: 고속 네트워크 카드의 **RSS (Receive Side Scaling)** 기술은 인터럽트 친화성을 활용한다. 0번 랜카드로 들어오는 패킷은 0번 CPU 코어가, 1번 랜카드는 1번 코어가 전담 처리하게 '친화성'을 맺어줌으로써 소프트웨어 인터럽트 부하를 분산시킨다.

- **📢 섹션 요약 비유**: "누가 먼저 하느냐(우선순위)보다, 어디서 하느냐(친화성)가 데이터 집약적 작업에서는 훨씬 더 중요할 수 있습니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 고성능 DB 서버 (Redis/Memcached) 튜닝**: 단일 스레드로 동작하는 레디스 서버. 여러 코어를 왔다 갔다 하면 캐시 미스 때문에 성능이 널뛰기함. 기술사적 판단으로 **전용 코어 고정 (Core Pinning)**을 수행한다. 0번 코어는 OS 전용, 1번 코어는 레디스 전용으로 `taskset -c 1 redis-server` 명령을 통해 격리하여 응답 지연(P99 Latency)을 30% 이상 개선한다.

2. **시나리오 — 가상화 환경의 'Noisy Neighbor' 해결**: 특정 VM이 CPU를 폭점하여 옆집 VM의 성능을 갉아먹음. 해결책으로 하이퍼바이저 수준에서 **vCPU Pinning**을 적용한다. 각 VM이 사용할 물리 코어를 겹치지 않게 딱딱 정해주어, 물리적 수준의 완벽한 자원 격리를 실현한다.

3. **시나리오 — 자율주행 차량의 제어 루프**: 센서 데이터를 처리하고 조향 신호를 보내는 핵심 루프가 일반 백그라운드 작업(업데이트 등) 때문에 밀리면 대형 사고. 해결책으로 커널 부팅 파라미터 **`isolcpus`**를 사용하여 특정 코어를 커널 스케줄링에서 아예 제외하고, 그 코어에는 오직 제어 프로세스만 친화성을 주어 실행함으로써 무중단 실시간성을 확보한다.

### 도입 체크리스트 (Optimization Checklist)
- **코어 개수**: 하이퍼스레딩(HT) 환경인가? (물리 코어와 논리 코어를 구분해서 핀닝해야 함)
- **부하 균등화**: 너무 많은 프로세스를 한 코어에 몰아넣어 특정 코어만 100%를 찍고 있지는 않은가?
- **IRQ Affinity**: 소프트웨어(프로세스)만 옮기지 말고, 해당 데이터가 들어오는 하드웨어 인터럽트(IRQ)도 같은 코어로 옮겼는가? (병목 해결의 핵심)

### 안티패턴
- **모든 프로세스에 수동 Affinity 부여**: 운영체제 스케줄러의 지능적인 부하 분산 기능을 완전히 무시하는 행위. 특정 코어는 놀고 특정 코어는 터져나가는 비효율 초래.
- **캐시 계층 무시**: L3 캐시를 공유하지 않는 서로 다른 물리 소켓(Socket) 간에 Affinity를 줘서 코어 이동 시 데이터 전송 지연을 유발하는 것.

- **📢 섹션 요약 비유**: "최적화의 정수는 스케줄러를 믿는 것이지만, 스케줄러보다 내가 상황을 더 잘 안다고 확신할 때만 비장의 카드(Affinity)를 꺼내야 합니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (CPU Affinity 도입 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **연산 속도** | L1/L2 캐시 히트율 상승으로 처리량 20~40% 향상 | 데이터 집약적 작업 효과 극대화 |
| **지연 시간** | 컨텍스트 스위칭 및 이사(Migration) 지연 제거 | 실시간 응답성(Jitter-free) 확보 |
| **전력 효율** | 불필요한 코어 간 데이터 이동 트래픽 감소 | 전체 시스템 전력 소모 최적화 |
| **관리 가시성** | 어떤 업무가 어떤 물리 자원을 쓰는지 명확히 파악 | 자원 거버넌스 체계 확립 |

### 미래 전망
- **Heterogeneous Affinity (Big.LITTLE)**: 고성능 코어(P-core)와 고효율 코어(E-core)가 섞인 환경에서, 업무 성격에 맞춰 최적의 코어 타입으로 친화성을 자동 조절하는 AI 스케줄러 보편화.
- **Dynamic Resource Partitioning**: 서버 부하에 따라 리얼타임으로 `cpuset`을 재구성하여, 서비스 수준(SLA)에 맞춰 물리 자원을 유동적으로 격리/통합하는 자율형 인프라 진화.

### 참고 표준
- **POSIX `sched_setaffinity`**: 유닉스/리눅스 표준 인터페이스.
- **Windows `SetProcessAffinityMask`**: 윈도우 환경의 친화성 제어 API.

- **📢 섹션 요약 비유**: CPU 친화성은 "자유로운 스케줄링"이라는 광야에서 "확정된 성능"이라는 울타리를 치는 작업이며, 복잡한 멀티코어 시대를 다스리는 가장 정교한 통치 수단입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. CPU 친화성은 공부할 때 **"나만의 전용 책상"**을 만드는 거예요. 매일 자리가 바뀌면 가방을 싸고 풀어야 해서 공부할 시간이 줄어들거든요.
2. 내 자리에 내 책과 필통을 그대로 두고 다니면, **자리에 앉자마자 바로** 공부를 시작할 수 있어서 훨씬 빨리 숙제를 끝낼 수 있답니다.
3. 하지만 친구들이 한 책상에만 너무 많이 몰리면 복잡해지니까, **누가 어떤 책상을 쓸지** 선생님(운영체제)이 잘 나누어 주는 것이 중요해요!
