# 워킹 셋 (Working Set) 메모리

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 워킹 셋 (Working Set)은 프로세스가 특정 시점 ($t$)에서 일정 시간 간격 ($\Delta$, 윈도우) 동안 집중적으로 참조한 페이지들의 집합으로, 프로세스가 페이지 폴트 없이 원활하게 실행되기 위해 메모리에 반드시 상주해야 하는 "최소 생존 프레임 세트"를 의미한다.
> 2. **가치**: 스래싱 (Thrashing) 현상을 방지하기 위한 동적 프레임 할당의 수리적 모델을 제공하며, 모든 실행 중인 프로세스의 워킹 셋 크기 합이 물리 메모리 용량을 초과할 경우 일부 프로세스를 중단 (Swap-out)시키는 자원 관리의 의사결정 기준이 된다.
> 3. **융합**: 참조의 국부성 (Locality of Reference) 원리를 정형화한 피터 데닝 (Peter Denning)의 모델로, 현대 OS의 가상 메모리 관리뿐만 아니라 데이터베이스 버퍼 풀 관리, 웹 브라우저의 탭 격리 메모리 최적화, 클라우드 네이티브 환경의 자동 자원 할당 정책에 광범위하게 적용된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 워킹 셋 (Working Set) 모델은 프로세스가 실행되는 동안 보일 '지역성 (Locality)'을 예측하여, 빈번하게 사용되는 페이지들을 메모리에 고정 (Resident)시킴으로써 페이지 폴트 발생을 최소화하고 시스템 처리량을 극대화하는 동적 메모리 관리 기법이다.

- **💡 비유**: 이는 수험생이 공부할 때 책상 위에 올려두는 "지금 당장 보고 있는 참고서와 공책"의 묶음과 같다. 가방(디스크)에는 수백 권의 책이 있지만, 책상(메인 메모리)에는 지금 풀고 있는 수학 문제집과 개념서(워킹 셋)만큼은 반드시 펼쳐져 있어야 공부(실행) 속도가 나지 않는 것과 같다.

- **등장 배경**:
  1. **고정 할당의 한계**: 프로세스마다 필요한 메모리 양이 실행 시점마다 다른데, 이를 고정적으로 할당하면 자원 낭비나 스래싱이 발생한다.
  2. **국부성 모델 (Locality Model)의 실체화**: 1960년대 피터 데닝은 프로세스가 특정 시간에 특정 영역의 페이지만을 집중적으로 참조한다는 사실을 발견하고, 이를 정량적으로 제어할 모델을 제시했다.
  3. **스래싱 (Thrashing) 예방**: 무분별한 다중 프로그래밍으로 인해 모든 프로세스가 '헐떡이는' 상태를 막기 위해, 최소한의 자원 보장이라는 개념이 필요해졌다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                 워킹 셋 (Working Set) 정의 및 예시                │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [수식] W(t, Δ) = { p | 페이지 p가 시간 (t-Δ, t) 사이에 참조됨 }  │
  │                                                                 │
  │   참조 스트림: 2 6 1 5 7 7 7 7 5 1 6 2 3                        │
  │               └─────────┘ (Δ=5)                                 │
  │                                                                 │
  │   Time t1 시점 (참조 5회째):                                      │
  │   최근 5회 참조: {2, 6, 1, 5, 7}                                 │
  │   Working Set = {1, 2, 5, 6, 7} , Size (WSS) = 5                │
  │                                                                 │
  │   Time t2 시점 (참조 10회째):                                     │
  │   최근 5회 참조: {7, 7, 7, 5, 1}                                 │
  │   Working Set = {1, 5, 7}       , Size (WSS) = 3                │
  │                                                                 │
  │   [관찰]                                                        │
  │   - 프로세스의 집중도에 따라 WSS는 수시로 변함                     │
  │   - Δ가 너무 크면 과거 정보가 남고, 너무 작으면 지역성을 못 담음     │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 워킹 셋의 크기(WSS)는 고정된 값이 아니다. 프로그램이 루프를 돌거나 특정 함수 내에서 집중적으로 연산할 때는 WSS가 작아지고 안정되지만, 새로운 함수를 호출하거나 데이터 구조를 대대적으로 변경하는 '전이 단계 (Transition Phase)'에서는 WSS가 급격히 커진다. OS는 이 WSS의 변화를 모니터링하여 프레임 할당량을 조절한다.

- **📢 섹션 요약 비유**: "지금 당장 필요한 연장통의 크기"를 실시간으로 재어서, 연장이 부족해 작업이 중단되지 않도록 미리 챙겨주는 배려와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 워킹 셋 모델의 자원 관리 알고리즘

운영체제는 시스템 전체의 가용성을 유지하기 위해 다음과 같은 수리적 모델을 적용한다.

1. **Working Set Size (WSS_i) 계산**: 각 프로세스 $i$에 대해 최근 $\Delta$ 시간 동안의 고유 페이지 수를 계산한다.
2. **Total Demand (D) 산출**: $D = \sum WSS_i$ (시스템 내 모든 활성 프로세스의 워킹 셋 합).
3. **Admission Control (입입 제어)**:
   - **$D \le m$ (전체 프레임 수)**: 모든 프로세스에게 필요한 만큼 프레임을 줄 수 있는 안전 상태. 새로운 프로세스 추가 가능.
   - **$D > m$**: 자원 부족 상태. 이대로 두면 스래싱 발생. OS는 일부 프로세스를 선택하여 실행을 중단(Suspend)시키고 해당 프레임을 다른 프로세스에게 몰아줌.

---

### 윈도우 크기 ($\Delta$)의 결정과 영향

$\Delta$는 워킹 셋 모델의 성능을 결정하는 가장 중요한 파라미터(Hyper-parameter)다.

| $\Delta$ 설정 | 영향 및 결과 | 기술적 의미 | 비유 |
|:---|:---|:---|:---|
| **$\Delta \to 0$** | 워킹 셋이 현재 참조하는 페이지 1개로 수렴 | 지역성을 전혀 반영하지 못함, 폴트 폭발 | 바늘구멍으로 세상 보기 |
| **$\Delta \to \infty$** | 프로세스가 평생 참조한 모든 페이지가 워킹 셋에 포함 | 메모리 낭비 극심, 다중 프로그래밍 정도 하락 | 평생 쓴 물건 다 쌓아두기 |
| **적정 $\Delta$** | 현재의 지역성(Locality)을 온전히 포함 | 페이지 폴트 최소화 및 메모리 효율 균형 | 필요한 물건만 챙긴 배낭 |

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 워킹 셋 크기와 페이지 부재의 관계                   │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   WSS / Page Faults                                              │
  │      ▲                                                           │
  │      │          Page Fault Rate (급격히 감소 후 안정)             │
  │      │  *                                                        │
  │      │   *                                                       │
  │      │    *      Working Set Size (계속 증가)                    │
  │      │     *    /                                                │
  │      │      *  /                                                 │
  │      │       *----------------- (안정 구간 - 지역성 확보)          │
  │      │      /  *                                                 │
  │      └─────/─────*────────────────────────────────▶              │
  │           /        *                         Δ (Window Size)     │
  │                                                                  │
  │   [교훈]                                                         │
  │   - 폴트율이 급격히 떨어지는 지점의 Δ를 찾는 것이 최적화의 핵심       │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 윈도우 크기($\Delta$)를 늘릴수록 더 많은 과거 데이터를 기억하므로 페이지 폴트율은 떨어진다. 하지만 어느 지점을 넘어서면 폴트 감소 효과는 미미한데 반해 워킹 셋 크기(WSS)만 불필요하게 커져 메모리를 낭비하게 된다. 이 "무릎(Knee)" 지점을 찾아내는 것이 효율적인 OS 설계의 정수다.

- **📢 섹션 요약 비유**: "너무 짧게 생각하면 실수를 하고, 너무 오래 고민하면 행동이 느려지는 것"처럼, 과거를 기억하는 적절한 시간 범위를 찾는 것이 중요합니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 워킹 셋 모델 vs 페이지 부재 빈도 (PFF)

| 비교 항목 | 워킹 셋 모델 (Working Set) | 페이지 부재 빈도 (PFF, Page Fault Frequency) |
|:---|:---|:---|
| **제어 철학** | **원인** 제어 (지역성 보장) | **결과** 제어 (폴트 발생에 대응) |
| **동작 방식** | 주기적으로 참조 비트 확인 및 집합 계산 | 폴트 시 인터럽트 핸들러 내에서 조정 |
| **구현 난이도** | 높음 (주기적인 참조 이력 관리 필요) | 낮음 (폴트 발생 간격만 측정하면 됨) |
| **오버헤드** | 높음 (모든 참조/주기적 스캔) | 낮음 (폴트 발생 시에만 동작) |
| **실무적 위치** | 이론적 배경 및 고성능 서버 정책 | 리눅스 등 범용 OS의 실제 구현 방식 |

### 비교 2: 가상화 및 클라우드에서의 응용

- **VMware / KVM의 메모리 관리**: 가상 머신(VM)이 할당받은 메모리를 다 쓰지 않을 때, 하이퍼바이저는 해당 VM의 워킹 셋을 추정한다. 워킹 셋 이외의 페이지는 '벌루닝(Ballooning)' 기법으로 회수하여 다른 VM에 할당함으로써 전체 클라우드 자원 효율을 높인다.
- **컨테이너 (Docker/K8s) 제한**: 컨테이너의 `memory limit`은 사실상 고정된 워킹 셋 크기를 강제하는 것과 같다. 워킹 셋이 이 제한을 넘어서면 OOM (Out Of Memory) 킬러가 작동하거나 심각한 성능 저하가 발생한다.

- **과목 융합 관점 (알고리즘)**: 워킹 셋은 데이터 스트림 알고리즘의 **슬라이딩 윈도우 (Sliding Window)** 기법과 동일한 수학적 기반을 가진다. 최근 $N$개의 데이터 중 유니크한 요소의 개수를 세는 문제는 현대적인 카운트-미니 스케치 (Count-Min Sketch) 등 확률적 자료구조를 통해 오버헤드 없이 근사적으로 계산되기도 한다.

- **📢 섹션 요약 비유**: 사고가 나기 전에 차간 거리를 유지하는 것(워킹 셋)과 사고가 날 것 같을 때(PFF) 급브레이크를 밟는 것의 차이입니다.

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — DB 서버의 버퍼 캐시 튜닝**: 대규모 쿼리가 실행될 때 인덱스 페이지들이 워킹 셋에 포함되지 못해 디스크 I/O가 폭증하는 상황. DBA는 워킹 셋 모델의 원리를 적용하여, 자주 쓰이는 인덱스를 'Keep Pool'에 고정(Pinning)함으로써 강제적으로 워킹 셋을 메모리에 유지시킨다.

2. **시나리오 — 고성능 웹 서버의 동시 접속자 제한**: 아파치나 엔진엑스(Nginx) 서버에서 `MaxClients` 설정은 사실상 워킹 셋의 합이 전체 물리 메모리를 넘지 않게 하려는 Admission Control이다. 이 값을 너무 크게 잡으면 스래싱이 발생하여 서비스가 완전히 마비된다.

3. **시나리오 — 모바일 앱의 백그라운드 캐시**: 안드로이드나 iOS는 사용자가 최근에 쓴 앱들의 워킹 셋을 메모리에 유지한다. 새로운 앱(워킹 셋)을 띄울 때 메모리가 부족하면, 가장 오래전에 쓰인 앱의 워킹 셋 전체를 날려버림(Purge)으로써 현재 앱의 실행 속도를 보장한다.

### 도입 체크리스트 (OS 및 미들웨어 설계 시)
- **참조 이력 수집 비용**: 페이지 참조 비트를 스캔하는 주기가 CPU 성능에 영향을 주지 않는가?
- **윈도우 크기 가변성**: 워크로드의 성격(Batch vs Interactive)에 따라 $\Delta$를 동적으로 바꿀 수 있는가?
- **페이지 교체 알고리즘과의 조화**: LRU 알고리즘이 워킹 셋 모델의 통계를 활용하고 있는가?

### 안티패턴
- **$\Delta$의 고정 불변**: 모든 프로세스에 동일한 윈도우 크기를 적용하는 것. 텍스트 에디터와 비디오 인코더는 지역성의 크기가 완전히 다르다.
- **예측 없는 프로세스 추가**: CPU 이용률이 낮다는 이유만으로 워킹 셋의 합을 고려하지 않고 새 프로세스를 투입하는 행위. (스래싱의 지름길)

- **📢 섹션 요약 비유**: "책상이 좁으면 공부를 잠시 쉬게 하더라도(Suspend), 공부하는 동안에는 필요한 책을 다 펼쳐둘 수 있게 해줘야 한다"는 철학이 실무의 핵심입니다.

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (워킹 셋 모델 도입 시)

| 구분 | 도입 전 (단순 할당) | 도입 후 (워킹 셋 기반) | 개선 효과 |
|:---|:---|:---|:---|
| **시스템 안정성** | 부하 증가 시 예측 불가능한 붕괴 | 스래싱 발생 전 선제적 방어 가능 | **장애 발생률 80% 감소** |
| **메모리 활용도** | 프로세스마다 넉넉히/부족하게 할당 | 실제 필요량에 기반한 정밀 배분 | **메모리 효율 30~50% 향상** |
| **사용자 경험** | 잦은 페이지 폴트로 인한 버벅임 | 유효 연산 시간 극대화로 매끄러운 동작 | **응답 속도 편차 최소화** |

### 미래 전망
- **AI 기반의 지능형 윈도우 조절**: 과거의 실행 이력을 학습하여, 특정 앱이 실행될 때 최적의 $\Delta$를 자동으로 설정해주는 머신러닝 기반 메모리 관리자가 주류가 될 것임.
- **분산 환경의 원격 워킹 셋**: CXL (Compute Express Link) 등을 통해 다른 서버의 메모리를 내 워킹 셋의 확장 공간으로 사용하는 '메모리 풀링' 기술이 상용화되어, 물리적 메모리 한계를 극복할 것임.

### 👶 어린이를 위한 3줄 비유 설명
1. 워킹 셋은 **"지금 당장 가지고 놀고 있는 장난감 바구니"**예요. 방 안에는 수천 개의 장난감이 있지만, 지금 내가 로봇 놀이를 하려면 로봇과 비행기 몇 개는 반드시 내 옆에 있어야 하죠.
2. 엄마(운영체제)는 내가 놀이를 멈추지 않도록, 내가 지금 푹 빠져 있는 장난감들을 치우지 않고 잘 챙겨줘요.
3. 하지만 동생들이 너무 많이 들어와서 거실이 좁아지면, 엄마는 "지금은 형아가 놀 차례니까 동생들은 잠시 방에 가서 기다려~"라고 말하며 우리가 모두 즐겁게 놀 수 있게 도와준답니다!
