# 커널 모드 진입 메커니즘 및 하드웨어 특권 수준 (Protection Rings)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPU가 실행할 수 있는 명령어의 범위를 제한하기 위해 도입된 **하드웨어 기반 계층적 권한 체계(Ring Structure)**와 그 경계를 넘나드는 제어 로직.
> 2. **가치**: 응용 프로그램이 실수나 고의로 하드웨어를 직접 조작하거나 다른 프로세스의 메모리를 침범하는 것을 물리적으로 방지하여 시스템의 **가용성과 기밀성**을 보장한다.
> 3. **융합**: 현대 운영체제는 효율성을 위해 대개 2단계(Ring 0, Ring 3)만 사용하며, 가상화 환경에서는 하이퍼바이저를 위한 **Ring -1** 개념이 추가되어 하드웨어-OS-가상화가 유기적으로 맞물린다.

---

### Ⅰ. 하드웨어 특권 수준: Protection Rings

x86 아키텍처를 기준으로 한 권한 계층 구조다. 숫자가 낮을수록 권한이 높다.

| 계층 (Level) | 명칭 | 실행 가능 범위 | 비유 |
|:---|:---|:---|:---|
| **Ring 0** | **커널 모드** | 모든 CPU 명령어 및 하드웨어 직접 제어 | 국왕 (모든 법 집행 가능) |
| **Ring 1, 2** | (OS 서비스) | 디바이스 드라이버 등 (현대 범용 OS는 거의 안 씀) | 귀족 / 관리 |
| **Ring 3** | **사용자 모드** | 제한된 명령어, 자기 자신의 메모리 영역만 접근 | 일반 백성 (정해진 일만 함) |

---

### Ⅱ. 커널 모드 진입 및 복귀 아키텍처 (ASCII)

사용자 프로세스가 어떻게 높은 권한의 성벽(Ring 0) 안으로 안전하게 들어가는지 보여준다.

```ascii
    [ Ring 3: User Space ]             [ Ring 0: Kernel Space ]
    +--------------------+             +----------------------+
    | User Application   |             | OS Kernel Services   |
    | (Low Privilege)    |             | (High Privilege)     |
    +---------|----------+             +----------^-----------+
              |                                   |
    ----------|-----------------------------------|------------
    (CPU Instruction Level Boundary - Protection Gate)
    ----------|-----------------------------------|------------
              |                                   |
      (1) SYSCALL / INT 0x80              (3) SYSRET / IRET
              |                                   |
    +---------v-----------------------------------|-----------+
    | CPU Hardware Transition:                    |           |
    | - Change CPL (Current Privilege Level) to 0 |           |
    | - Switch to Kernel Stack                    |           |
    | - Jump to predefined Entry Point (Gate)     |           |
    +----------------------|----------------------+           |
                           |                                  |
                           v (2) Kernel Handler               |
                    +----------------------+                  |
                    | Validate Arguments   |                  |
                    | Execute Action       |------------------+
                    +----------------------+
```

**[진입 프로세스 상세]**
1. **이벤트 트리거**: `syscall` 명령어가 실행되면 CPU는 하드웨어적으로 현재의 권한 레벨(CPL)을 0으로 강제 변경한다.
2. **관문 통제**: CPU는 사용자가 지정한 주소로 점프하지 않고, 커널이 미리 하드웨어 레지스터(MSR 등)에 등록해 놓은 **단 하나의 엔트리 포인트**로만 점프한다. (강력한 보안 관문)
3. **특권 실행**: 커널 모드에 진입한 후에는 I/O 포트 제어, 페이지 테이블 수정 등 모든 **특권 명령어(Privileged Instructions)**를 실행할 수 있다.

---

### Ⅲ. 특권 명령과 보호 위반 (Mechanism)

#### 1. 특권 명령어의 예
- 하드웨어 I/O (IN, OUT)
- CPU 상태 레지스터(CR0, CR3 등) 변경
- 인터럽트 허용/차단 (CLI, STI)
- 시스템 종료 및 재부팅

#### 2. 보호 모드 위반 (General Protection Fault)
만약 Ring 3(사용자 모드)에서 위와 같은 명령어를 직접 실행하려고 시도하면, CPU 하드웨어가 즉시 실행을 거부하고 커널에게 알린다. 커널은 해당 프로세스를 즉시 종료(Segmentation Fault)시킨다.

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 가상화 환경의 확장: Ring -1
- **현상**: 가상 머신(VM) 내부의 OS(게스트)도 자기가 Ring 0인 줄 알고 특권 명령을 내린다.
- **기술사적 결단**: 하드웨어 가속 가상화(Intel VT-x)는 실제 커널 모드보다 더 높은 **VMM Mode (Ring -1)**를 제공한다. 게스트 OS가 Ring 0 명령을 내리면 하이퍼바이저로 트랩(VM-Exit)이 발생하여 중재하는 구조다.

#### 2. 기술사적 인사이트
- **Meltdown 취약점**: CPU가 Ring 0 권한 체크를 '나중에' 하고 데이터를 '미리' 가져오는 비순차 실행(Out-of-Order) 허점을 이용해 Ring 3에서 Ring 0 메모리를 훔쳐본 사건이다. 이를 위해 **KPTI**와 같은 강력한 페이지 테이블 격리가 도입되었다.
- **성능 vs 보안**: 잦은 모드 진입은 파이프라인 플러시와 TLB 오버헤드를 유발한다. 최근 DPDK 등은 보안을 일부 포기하더라도 성능을 위해 사용자 모드에서 직접 하드웨어를 제어하기도 한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **시스템 가용성**: 개별 앱의 오류가 전체 하드웨어 중단으로 이어지는 것을 방지.
- **다중 사용자 보호**: 사용자 간의 데이터 격리를 보장하는 물리적 근거 제공.

#### 2. 미래 전망
최근에는 하드웨어가 더 세밀한 권한 관리를 지원하는 방향으로 가고 있다. Intel의 **CET (Control-flow Enforcement Technology)**나 ARM의 **PAC (Pointer Authentication)** 등은 커널 모드 진입 시 호출 스택의 무결성까지 하드웨어가 검증하여, 제어 흐름 탈취 공격을 원천 차단하는 수준에 이르고 있다.
