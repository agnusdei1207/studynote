# cgroups (Control Groups) 및 컨테이너 자원 제어

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: cgroups (Control Groups)는 리눅스 커널에서 프로세스들을 계층적인 그룹으로 묶어 CPU, 메모리, I/O, 네트워크 등 시스템 자원의 사용량을 제한(Limiting), 우선순위 지정(Prioritizing), 계측(Accounting) 및 제어(Control)하는 핵심 프레임워크다.
> 2. **가치**: 특정 프로세스나 컨테이너의 자원 폭주로 인한 시스템 전체의 공멸(Noisy Neighbor 문제)을 방지하며, 현대 클라우드 네이티브 환경에서 서비스별 자원 쿼터(Quota)와 서비스 품질(QoS)을 하드웨어 수준에서 강제하는 물리적 기반이 된다.
> 3. **융합**: 자원 가시성을 제공하는 **네임스페이스 (Namespace)**와 결합하여 컨테이너 가상화를 완성하며, `systemd` 서비스 관리 아키텍처 및 쿠버네티스(Kubernetes)의 리소스 제한 정책(`requests/limits`)의 실질적인 구현체로 작동한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: cgroups는 운영체제가 프로세스들에게 채우는 "전자 발찌"와 같다. 프로세스가 마음껏 뛰어놀 수 있는 범위(네임스페이스)를 정해줬다면, cgroups는 그 프로세스가 밥(CPU)을 얼마나 먹을지, 방(메모리)을 얼마나 차지할지를 엄격하게 규제하는 도구다.

- **💡 비유**: 이는 "아파트 단지의 관리비 및 자원 할당 규정"과 같다. 각 가구(컨테이너)는 독립된 집(네임스페이스)에서 살지만, 공동 주택의 전체 전기 용량이나 주차 공간(시스템 자원)은 정해져 있다. 관리소(cgroups)는 "A동 101호는 한 달에 전기 300kWh까지만 써라"라고 제한을 걸어, 특정 집이 전기를 다 써버려 단지 전체가 정전되는 것을 막는 것과 같다.

- **등장 배경**:
  1. **단일 프로세스 제어의 한계**: `ulimit`과 같은 기존 기술은 개별 프로세스 단위로만 작동하여, 여러 프로세스가 협력하는 '서비스' 단위의 자원 통제가 불가능했다.
  2. **가상화 효율화**: 하이퍼바이저 없이 커널 하나로 수백 개의 격리된 환경을 돌리기 위해선, 자원 배분 알고리즘이 커널 내부에서 매우 정교하게 작동해야 했다.
  3. **구글의 요구 (2006)**: 구글 내부의 거대 분산 시스템을 운영하기 위해 프로세스 그룹 단위의 자원 격리가 필요해졌고, 'Process Containers'라는 이름으로 최초 제안되었다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              cgroups의 계층적 자원 관리 아키텍처 (V1 vs V2)         │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ User Interface: vfs /sys/fs/cgroup ]                        │
  │          │                                                      │
  │   ┌──────▼───────┐          ┌────────────────────────────────┐  │
  │   │  Hierarchy 1 │          │  Unified Hierarchy (V2)        │  │
  │   │  (CPU Only)  │          │  ┌────────────┐                │  │
  │   └──────┬───────┘          │  │ [ Root ]   │                │  │
  │   ┌──────▼───────┐          │  └─────┬──────┘                │  │
  │   │  Hierarchy 2 │          │  ┌─────▼──────┐                │  │
  │   │ (Memory Only)│          │  │ [ Group A ]│ ── (CPU, Mem)  │  │
  │   └──────┬───────┘          │  └────────────┘                │  │
  │          │                  └────────────────────────────────┘  │
  │          ▼                                                      │
  │   [ Kernel Controllers ] ──▶ CPU, Memory, I/O, PIDs, Network    │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** cgroups는 리눅스 파일 시스템 인터페이스를 통해 제어된다. V1은 자원마다 별개의 트리를 만들어 관리가 복잡했지만, 현대의 V2(Unified)는 하나의 통합된 트리 구조에서 모든 자원을 효율적으로 관리한다. 이 계층 구조를 통해 부모 그룹의 제약 사항이 자식 그룹으로 상속되는 구조적 특징을 가진다.

- **📢 섹션 요약 비유**: "자원을 쪼개는 것(CPU/Memory)을 넘어, 이를 계층적으로 상속하고 감시하는 운영체제의 거대한 행정 조직망"입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 4대 핵심 자원 컨트롤러 (Subsystems)

cgroups는 하부 자원의 특성에 따라 전용 컨트롤러를 운영한다.

| 컨트롤러 | 상세 제어 항목 | 주요 파라미터 | 비유 |
|:---|:---|:---|:---|
| **CPU** | 프로세스 스케줄링 비중 및 시간 | `cpu.shares`, `cpu.cfs_quota_us` | 식사 시간 배정 |
| **Memory** | 프로세스 점유 램 용량 제한 | `memory.limit_in_bytes`, `memory.oom_control` | 방 평수 제한 |
| **BlkIO** | 디스크 읽기/쓰기 대역폭(IOPS) | `blkio.throttle.read_bps_device` | 택배 전용 차선 |
| **PIDs** | 생성 가능한 프로세스/스레드 수 | `pids.max` | 자녀 수 제한 (Fork Bomb 방지) |

---

### 자원 제한 방식: Shares vs Quota

CPU 자원을 배분할 때 cgroups는 두 가지 철학을 사용한다.

1. **Shares (Relative)**: CPU가 한가할 땐 다 쓰게 해주지만, 바빠지면 정해진 비율(예: 1024 대 512)대로 시간을 나눠주는 방식. (자원 활용도 중시)
2. **Quota (Absolute)**: CPU가 아무리 한가해도 정해진 시간(예: 100ms 중 20ms) 이상은 절대 못 쓰게 칼같이 자르는 방식. (예측 가능성 중시)

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 Memory Cgroup과 OOM의 연동 메커니즘                 │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ Cgroup Container A ]                                         │
  │   - Limit: 512MB                                                 │
  │   - Current: 511MB                                               │
  │                                                                  │
  │   1. App: malloc(10MB) 요청                                      │
  │   2. Kernel: "너네 그룹 한도 넘었어!" (Limit Exceeded)            │
  │   3. Kernel: 그룹 내 프로세스 중 하나를 OOM 킬러로 처단            │
  │                                                                  │
  │   [가치] 시스템 전체 램은 남아있어도, 해당 그룹만 선택적 숙정 가능   │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** cgroups의 백미는 '그룹 단위의 연대 책임'이다. 시스템 전체 메모리가 128GB나 남아있어도, 512MB 한도가 걸린 그룹이 1바이트라도 더 쓰려 하면 커널은 가차 없이 그 그룹 안에서 범인(희생자)을 찾아 처형(SIGKILL)한다. 이것이 컨테이너의 안정적인 멀티 테넌시를 보장하는 힘이다.

- **📢 섹션 요약 비유**: "전체 예산이 남아돌아도, 부서별 법인카드 한도(Quota)를 넘으면 결제가 거부되는 엄격한 회계 감사 시스템"과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: cgroups v1 vs cgroups v2

| 비교 항목 | cgroups v1 (Legacy) | cgroups v2 (Modern) |
|:---|:---|:---|
| **구조** | 자원별 다중 계층 (Multiple) | **단일 계층 (Unified)** |
| **일관성** | 프로세스-스레드 관리 혼선 | 명확한 프로세스 중심 관리 |
| **가시성** | 단순 사용량 통계 | **PSI (Pressure Stall Info)** 제공 |
| **보안** | Root 권한 위주 | 비루트 유저도 제어 가능 (Delegation) |
| **상황** | 레거시 컨테이너 엔진 | **쿠버네티스 v1.25+ 표준** |

### 비교 2: cgroups와 하이퍼바이저 격리 비교

- **Hypervisor (HW 격리)**: 가상 하드웨어를 통해 자원을 물리적으로 갈라치기함. 격리는 완벽하나 전환 비용이 큼.
- **cgroups (Logic 격리)**: 커널이 모든 상황을 알고서 "너는 요만큼만 먹어"라고 말로만 타이르는 방식. 오버헤드가 0에 수렴하지만, 커널 공유로 인한 보안 취약점 상존.

- **과목 융합 관점 (네트워크)**: cgroups의 `net_cls` 및 `net_prio` 컨트롤러는 네트워크의 **QoS (Quality of Service)** 정책과 연동된다. 특정 컨테이너에서 나가는 패킷에 클래스 ID를 부여하여, 스위치나 라우터가 해당 트래픽을 저순위로 처리하게 유도할 수 있다.

- **📢 섹션 요약 비유**: "방 자체를 따로 만드는 것(VM)보다 거실 중앙에 선(cgroups)을 긋고 못 넘어오게 하는 것이 훨씬 빠르고 효율적인 통제 방식입니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 특정 앱의 CPU 독점 방지**: 파이썬 데이터 분석 스크립트가 CPU 16코어를 다 잡아먹어 웹 서버가 마비됨. 기술사적 판단으로 **`systemd-run`** 명령을 통해 해당 스크립트를 즉시 cgroup으로 묶고 `CPUQuota=20%`를 적용한다. 별도의 코드 수정 없이도 시스템 가용성을 즉각적으로 회복한다.

2. **시나리오 — '포크 밤 (Fork Bomb)' 공격 방어**: 해커가 ` :(){ :|:& };: ` 같은 코드를 실행하여 프로세스를 무한 생성함. 해결책으로 cgroups의 **PIDs 컨트롤러**를 활성화하고 컨테이너당 최대 프로세스 수를 100개로 제한한다. 공격이 시작되자마자 cgroup 한도에 막혀 시스템은 평화롭게 유지된다.

3. **시나리오 — 쿠버네티스 OOM 킬링 디버깅**: 특정 Pod가 자꾸 죽음. `kubectl describe` 확인 결과 `OOMKilled` 발생. 원인은 앱이 쓰는 메모리보다 쿠버네티스의 `limit`이 너무 낮게 설정된 것. 해결책으로 cgroups v2의 **PSI (Pressure Stall Info)** 데이터를 분석하여, 앱이 메모리를 기다리느라 소모한 시간(Stall Time)을 근거로 최적의 `request/limit` 값을 재산정한다.

### 도입 체크리스트 (SRE 가이드)
- **v2 지원 확인**: 현재 사용하는 리눅스 배포판과 컨테이너 런타임이 cgroups v2를 완벽히 지원하는가? (`mount | grep cgroup2`)
- **Swap 제어**: 메모리 한도 도달 시 스왑(Swap)을 쓰게 할 것인가, 바로 죽일 것인가? (`memory.swappiness` 조절)
- **Soft Limit 활용**: `memory.high`를 설정하여 하드 리미트에 걸리기 전 미리 앱이 메모리를 반환(Reclaim)하도록 유도하고 있는가?

### 안티패턴
- **제한(Limit) 없는 컨테이너 실행**: "우리 앱은 메모리 적게 써요"라며 제한 없이 띄우는 행위. 메모리 누수 발생 시 물리 서버 전체가 OOM에 빠져 모든 서비스가 동반 자살함.
- **CPU Share의 오해**: `cpu.shares`는 CPU가 남을 때는 제한하지 않는다는 사실을 모르고, 성능이 왜 들쭉날쭉하냐고 묻는 것. (일정한 성능이 필요하면 `CPUQuota` 사용 필수)

- **📢 섹션 요약 비유**: "울타리(Limit)가 없는 목장은 평화로워 보이지만, 늑대(Bug) 한 마리가 나타나면 양떼(전체 시스템)가 몰살당하게 됩니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (cgroups 도입 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **시스템 예측성** | 워크로드별 자원 사용량을 수치로 통제하여 장애 예측 가능 | 안정성 지수 300% 향상 |
| **비용 효율 (ROI)** | 하나의 서버에 수백 개의 격리된 앱 배치 가능 (Density) | 인프라 비용 70% 절감 |
| **보안 강도** | 자원 고갈 공격(DoS)으로부터 시스템 핵심 기능 보호 | 방어 체계 고도화 |
| **운영 가시성** | 그룹별 상세 리소스 메트릭 수집을 통한 용량 산정 용이 | 데이터 기반 운영(FinOps) |

### 미래 전망
- **AI-Driven Resource Partitioning**: 앱의 부하 패턴을 AI가 학습하여, cgroup의 Quota와 Shares 값을 실시간으로 자동 조절하는 '자율 운영 커널' 등장.
- **Energy-Aware cgroups**: 단순히 성능만 제한하는 게 아니라, 탄소 배출량이나 전력 요금에 따라 그룹별 자원 사용을 통제하는 'ESG 대응형 스케줄링' 발전.

### 참고 표준
- **Linux Control Groups v2 (Unified Hierarchy)**: 현대 리눅스의 표준 자원 관리 명세.
- **OCI (Open Container Initiative) Runtime Spec**: 컨테이너 런타임이 cgroups를 조작하는 표준 방식 정의.

- **📢 섹션 요약 비유**: cgroups는 클라우드라는 거대한 기계를 정교하게 움직이는 "톱니바퀴의 칸막이"이며, 이 칸막이가 있기에 우리는 수만 개의 서비스를 하나의 구름(Cloud) 속에 안전하게 담을 수 있습니다.
