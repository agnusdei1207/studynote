# 멀티프로세서 부하 균등화 (Load Balancing)와 큐 이주

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 멀티코어 환경에서 특정 CPU 코어에만 프로세스가 몰리는 불균형을 해소하기 위해, **대기 중인 프로세스를 바쁜 코어에서 한가한 코어로 이동(Migration)**시키는 자원 최적화 기법.
> 2. **가치**: 전체 프로세서의 활용도를 균일하게 유지하여 시스템의 **총 처리량(Throughput)**을 극대화하고, 특정 코어의 과부하로 인한 지연 시간을 방지한다.
> 3. **융합**: 운영체제의 다중 처리기 스케줄링 전략의 핵심이며, 프로세서 친화성(Affinity)과 상충 관계(Trade-off)를 이루며 동적으로 작동한다.

---

### Ⅰ. 부하 균등화 (Load Balancing)의 필요성

- **SMP의 한계**: 각 코어가 독립적인 실행 큐(Run Queue)를 가질 때, 스케줄링 운에 따라 어떤 코어는 10개의 작업이 쌓이고 어떤 코어는 놀고 있는 상황이 발생할 수 있다.
- **목표**: 모든 코어가 가능한 한 비슷한 수의 프로세스를 처리하도록 보장하는 것.

---

### Ⅱ. 큐 이주 (Queue Migration)의 2대 방식 (ASCII)

프로세스를 옮기는 주체에 따라 두 가지 메커니즘으로 나뉜다.

#### 1. Push Migration
- **원리**: 감시 스레드가 주기적으로 각 코어의 부하를 체크하여, 균형이 깨지면 바쁜 CPU의 작업을 한가한 CPU로 강제로 밀어넣음.

#### 2. Pull Migration
- **원리**: 유휴 상태가 된 CPU가 스스로 다른 바쁜 CPU의 대기 큐를 들여다보고 작업을 뺏어옴 (Work Stealing).

```ascii
    [ High Load CPU 0 ]                [ Idle CPU 1 ]
    +-----------------+                +-----------------+
    | P1 | P2 | P3 | P4|                | (Empty)         |
    +---|-------------+                +-----^-----------+
        |                                    |
        +---- (Push/Pull Migration) ---------+
        |
        v
    [ CPU 0 ]                          [ CPU 1 ]
    +-----------+                      +-----------+
    | P1 | P2 | |                      | P3 | P4 | |
    +-----------+                      +-----------+
    ( Balanced State )
```

---

### Ⅲ. 부하 균등화 vs 프로세서 친화성 (Trade-off)

| 구분 | 부하 균등화 (Load Balancing) | 프로세서 친화성 (Affinity) |
|:---|:---|:---|
| **목표** | 모든 CPU 코어를 바쁘게 만들기 | 동일 코어의 **캐시 재사용** 극대화 |
| **장점** | 처리량 향상, 대기 시간 평준화 | 실행 속도 향상, 메모리 오버헤드 감소 |
| **단점** | 이주 시 캐시 미스 발생 (Cold Cache) | 특정 코어 과부하 시 자원 낭비 발생 |
| **결합** | 부하 불균형이 **임계치**를 넘을 때만 이주 수행. |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 리눅스 커널의 부하 분산 (Scheduling Domains)
- **현상**: 멀티 소켓, 멀티 코어 서버에서 무분별한 이주는 성능을 떨어뜨린다.
- **기술사적 결단**: 
  - **Scheduling Domains** 계층 구조를 활용한다. 
  - 하이퍼스레딩 코어 간의 이주(비용 낮음)는 자주 수행하고, 물리 소켓 간의 이주(비용 높음)는 극심한 불균형 시에만 수행하도록 정책을 차등화한다.

#### 2. 기술사적 인사이트: 전력 관리와의 조화
- 현대 OS는 부하가 적을 때 균등하게 나누기보다, 오히려 **몇 개의 코어에 작업을 몰아넣고** 나머지 코어를 깊은 잠(C-state)에 빠뜨려 전력을 아끼는 '부하 집중' 전략을 병행하기도 함을 인지해야 한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **SLA 보장**: 작업별 응답 시간의 편차(Variance) 최소화.
- **하드웨어 ROI 극대화**: 비싼 서버 리소스를 골고루 사용하여 투입 대비 성능 인출.

#### 2. 미래 전망
앞으로 수백~수천 개의 코어가 탑재되는 **매니코어(Many-core)** 환경에서는 중앙 스케줄러가 모든 큐를 감시하는 방식이 병목이 될 것이다. 이에 따라 코어들이 서로 자율적으로 작업을 주고받는 **분산 부하 분산(Distributed Work Stealing)** 알고리즘이 더욱 정교해질 것이며, AI가 프로세스의 작업 패턴을 예측하여 미리 이주시시키는 지능형 스케줄링이 보편화될 것이다.
