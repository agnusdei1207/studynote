# 페이지 교체 LRU 원리 (Least Recently Used Page Replacement)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: LRU (Least Recently Used) 알고리즘은 "과거에 가장 오랫동안 사용되지 않은 페이지가 미래에도 사용될 가능성이 가장 낮다"는 시간적 국부성 (Temporal Locality)의 가설에 기반하여, 가장 오래 전 참조된 페이지를 교체 대상으로 선정하는 최적 근사 알고리즘이다.
> 2. **가치**: 구현 복잡도는 존재하나 최적 교체 알고리즘 (OPT)의 성능에 가장 근접하며, 프레임 수가 증가할 때 페이지 폴트가 늘어나는 벨라디의 모순 (Belady's Anomaly)이 발생하지 않는 스택 알고리즘 (Stack Algorithm)의 특성을 가져 안정적인 성능을 보장한다.
> 3. **융합**: 하드웨어 오버헤드로 인해 실무에서는 참조 비트 (Reference Bit)를 활용한 2차 기회 (Second Chance) 또는 시계 (Clock) 알고리즘으로 변형되어 Linux 커널의 활성/비활성 리스트 (Active/Inactive List) 관리 등에 핵심적으로 활용된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: LRU (Least Recently Used)는 가상 메모리 관리에서 물리 프레임이 부족할 때, 메모리에 적재된 페이지 중 마지막 참조 시점 (Last Reference Time)이 가장 과거인 페이지를 찾아내어 디스크로 스왑 아웃(Swap-out)시키는 페이지 교체 전략이다.

- **💡 비유**: 이는 책상(메인 메모리)이 좁을 때, 가장 오랫동안 손대지 않은 책을 골라 책꽂이(디스크)로 돌려보내는 것과 같다. 최근에 읽은 책은 다시 읽을 확률이 높으므로 책상에 남겨두는 합리적 정리 방식이다.

- **등장 배경**:
  1. **FIFO의 한계**: 단순히 먼저 들어온 페이지를 내보내는 FIFO (First-In First-Out) 방식은 자주 쓰이는 페이지(예: 루프 코드)를 내보내 성능을 저하시키고, 벨라디의 모순이라는 기현상을 초래했다.
  2. **OPT의 실현 불가능성**: 미래의 참조를 예측하는 최적 알고리즘(OPT)은 이론적으로만 존재할 뿐, 실제 시스템에서 미래를 알 수 없으므로 과거 데이터를 바탕으로 한 최선의 추론이 필요해졌다.
  3. **시간적 국부성 (Temporal Locality)**: "한 번 참조된 데이터는 가까운 미래에 다시 참조될 가능성이 높다"는 컴퓨터 공학의 대원칙을 실제 교체 로직에 반영하고자 도입되었다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              LRU 페이지 교체 시뮬레이션 (프레임 3개)               │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   참조 스트림: 7 → 0 → 1 → 2 → 0 → 3 → 0 → 4 → 2                │
  │                                                                 │
  │   Time:  T1  T2  T3  T4  T5  T6  T7  T8  T9                     │
  │   Ref :  7   0   1   2   0   3   0   4   2                      │
  │         ┌───┬───┬───┬───┬───┬───┬───┬───┬───┐                   │
  │   F1    │ 7 │ 7 │ 7 │ 2 │ 2 │ 2 │ 2 │ 4 │ 4 │                   │
  │         ├───┼───┼───┼───┼───┼───┼───┼───┼───┤                   │
  │   F2    │   │ 0 │ 0 │ 0 │ 0 │ 0 │ 0 │ 0 │ 0 │                   │
  │         ├───┼───┼───┼───┼───┼───┼───┼───┼───┤                   │
  │   F3    │   │   │ 1 │ 1 │ 1 │ 3 │ 3 │ 3 │ 2 │                   │
  │         └───┴───┴───┴───┴───┴───┴───┴───┴───┘                   │
  │   Fault: P   P   P   P   H   P   H   P   P                      │
  │                                                                 │
  │   [분석]                                                        │
  │   T4 시점: 2번 참조, 가장 오래된 7번(F1) 교체                   │
  │   T5 시점: 0번 참조, Hit! (LRU 순서가 0번을 최신으로 갱신)        │
  │   T6 시점: 3번 참조, 가장 오래된 1번(F3) 교체                   │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 프레임이 가득 찬 상태에서 새로운 페이지(2번)가 들어올 때, 현재 메모리에 있는 {7, 0, 1} 중 참조 시점을 비교한다. 7번은 T1, 0번은 T2, 1번은 T3에 참조되었으므로 가장 오래된 7번이 희생자(Victim)가 된다. 중요한 점은 T5에서 0번을 다시 참조(Hit)하면 0번의 '최근성'이 갱신되어, 다음에 새로운 페이지가 올 때 0번은 보호받게 된다는 것이다.

- **📢 섹션 요약 비유**: "가장 오래된 친구를 먼저 보내는 것"이 아니라, "가장 오랫동안 연락이 없던 친구를 주소록 뒷순위로 밀어내는 것"과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### LRU의 두 가지 고전적 구현 기법

정확한 LRU를 구현하기 위해서는 매 메모리 참조마다 시간 정보를 갱신해야 하므로 하드웨어의 지원이 필수적이다.

| 구현 방식 | 동작 메커니즘 | 장점 | 단점 | 비유 |
|:---|:---|:---|:---|:---|
| **카운터 (Counter)** | 각 PTE에 '사용 시간' 레지스터 추가, 참조 시 CPU 클럭 기록 | 논리적으로 가장 명확함 | 매 참조마다 메모리 쓰기 발생, 오버헤드 큼 | 도서관 대출증에 시간 도장 찍기 |
| **스택 (Stack)** | 사용된 페이지 번호를 스택의 Top으로 이동, Bottom이 LRU 페이지 | 검색 없이 즉시 희생자 선택 가능 | 포인터 갱신 부담, 하드웨어 구현 복잡 | 책상 위에 방금 읽은 책을 계속 쌓기 |

---

### 실무적 대안: 근사 LRU (LRU Approximation)

순수 LRU는 매 나노초(ns) 단위의 메모리 참조마다 하드웨어 레지스터를 갱신해야 하므로 매우 비싸다. 따라서 현대 OS는 참조 비트 (Reference Bit)를 활용한 근사 기법을 사용한다.

1. **참조 비트 (Reference Bit)**: 초기값 0, 페이지 참조 시 하드웨어가 1로 세팅.
2. **부가적 참조 비트 알고리즘 (Aging)**: 일정한 주기마다 참조 비트를 오른쪽으로 시프트(Shift)하여 8비트 정수값으로 관리. 값이 클수록 최신임.
3. **2차 기회 알고리즘 (Second Chance)**: FIFO 기반이되, 참조 비트가 1이면 0으로 바꾸고 기회를 한 번 더 줌.
4. **시계 알고리즘 (Clock Algorithm)**: 환형 큐(Circular Queue)를 순회하며 참조 비트가 0인 페이지를 즉시 교체.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 시계 알고리즘 (Clock Algorithm) 작동 원리          │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │      ┌───────┐           [규칙]                                   │
  │   ┌──▶ Page A │──┐        1. 시계 바늘이 돌며 Ref Bit 확인         │
  │   │  (Ref=1)  │  │        2. 1이면 0으로 바꾸고 PASS (2차 기회)     │
  │   │  └───────┘  │        3. 0이면 즉시 교체 (Victim 선정)          │
  │ ┌──────┐      ┌──────┐                                           │
  │ │Page D│  ↖   │Page B│                                           │
  │ │(Ref=0)│  바늘 │(Ref=1)│    [현재 상황]                          │
  │ └──────┘      └──────┘    Page B(1) → 0으로 변경 후 이동          │
  │   │  ┌───────┐  │        Page A(1) → 0으로 변경 후 이동          │
  │   └──│ Page C │──┘        Page D(0) → 발견! 교체 및 로딩           │
  │      │(Ref=0) │                                                  │
  │      └───────┘                                                  │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Clock 알고리즘은 LRU의 시간적 순서를 완벽히 지키지는 않지만, "최근에 사용된 페이지는 적어도 한 번은 봐준다"는 원칙을 통해 LRU와 유사한 효율을 낸다. 바늘이 한 바퀴 도는 동안 한 번도 참조되지 않은(Ref=0) 페이지는 가장 과거에 쓰인 것으로 간주되어 교체된다.

- **📢 섹션 요약 비유**: 시험장에서 답을 모르는 학생에게 "한 번 더 생각할 시간(2차 기회)"을 주고, 그래도 답을 못 쓰면 퇴실시키는 관대한 감독관과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: FIFO vs LRU vs OPT

| 비교 항목 | FIFO (First-In First-Out) | LRU (Least Recently Used) | OPT (Optimal) |
|:---|:---|:---|:---|
| **기준 시점** | 적재된 시점 (In) | 마지막 사용 시점 (Use) | 미래 사용 시점 (Future) |
| **벨라디 모순** | 발생 (프레임↑ 폴트↑ 가능) | 발생 안 함 (스택 알고리즘) | 발생 안 함 |
| **성능** | 낮음 (자주 쓰는 페이지 교체 위험) | 높음 (지역성 기반 최적화) | 최고 (이론적 상한선) |
| **구현 난이도** | 매우 쉬움 (Queue) | 어려움 (하드웨어/스택/카운터) | 불가능 (미래 예측 불가) |

### 비교 2: LRU vs LFU (Least Frequently Used)

| 구분 | LRU (최근성 중심) | LFU (빈도성 중심) |
|:---|:---|:---|
| **핵심 논리** | 얼마나 최근에 썼는가? | 얼마나 많이 썼는가? |
| **강점** | 갑작스러운 작업 전환에 민감하게 반응 | 오랫동안 꾸준히 쓰인 데이터 보호 |
| **약점** | 과거에 엄청나게 쓰였어도 최근 안 쓰면 교체 | 처음에만 반짝 쓰인 데이터가 계속 상주 |
| **실무** | 일반적인 OS 페이지 교체 표준 | 웹 캐시(CDN), 특정 DB 인덱스 관리 |

- **과목 융합 관점 (데이터베이스)**: DB 버퍼 캐시 관리에서도 LRU는 핵심이다. 하지만 DB는 전체 테이블 스캔(Full Scan) 시 LRU 릴(Reel)이 오염되는 문제가 있어, 한 번만 읽힌 데이터는 LRU 리스트의 중간에 삽입하거나 일정 횟수 이상 참조되어야만 상위 리스트로 올리는 **LRU-K** 또는 **Touch-Count 기반 LRU**를 변형하여 사용한다.

- **📢 섹션 요약 비유**: "누가 제일 먼저 들어왔나(FIFO)"보다 "누가 제일 마지막까지 일했나(LRU)"를 따지는 것이 팀의 생산성을 높이는 합리적인 인사 관리와 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 리눅스 커널의 페이지 교체**: 리눅스는 순수 LRU 대신 **2-List 전략 (Active/Inactive List)**을 사용한다. 페이지가 처음 들어오면 Inactive 리스트에 두고, 두 번 이상 참조되면 Active 리스트로 승격시킨다. 메모리가 부족하면 Inactive 리스트의 꼬리(LRU)부터 교체한다. 이는 한 번만 읽히고 마는 '대량 I/O 데이터'가 자주 쓰이는 '실행 코드 페이지'를 밀어내는 것을 방지하기 위함이다.

2. **시나리오 — Redis/Memcached 캐시 교체**: In-memory DB인 Redis는 메모리 한계 도달 시 교체 정책으로 `allkeys-lru`를 제공한다. Redis는 성능을 위해 정확한 LRU 대신 5개(기본값)의 샘플 키를 무작위로 뽑아 그중 가장 오래된 것을 지우는 **근사 LRU (Approximated LRU)**를 사용하여 오버헤드를 극단적으로 줄인다.

3. **시나리오 — 성능 디버깅 중 '벨라디의 모순' 오해**: 개발자가 FIFO를 사용하다 프레임을 늘렸는데 성능이 떨어지는 것을 보고 당황하는 경우. 해결책으로 LRU 계열(스택 알고리즘)로 교체 정책을 변경하여 메모리 증설이 성능 향상으로 직결되도록 아키텍처를 개선함.

### 도입 체크리스트
- **하드웨어 지원 여부**: 사용 중인 CPU 아키텍처가 PTE의 Reference Bit를 자동으로 세팅해주는가? (x86은 지원, 일부 RISC는 소프트웨어 핸들링 필요)
- **교체 오버헤드 vs 적중률**: 초고성능 시스템인가? (정밀한 LRU보다 빠른 Clock 선호)
- **워크로드 특성**: Sequential Scan이 많은가? (LRU-K 또는 MRU 선호)

### 안티패턴
- **단순 LRU의 오염 (LRU Pollution)**: 100GB 영화 파일을 한 번 읽는 작업이 8GB 메모리에 있는 중요 시스템 페이지를 모두 밀어내게 방치하는 것. (해결: 비선형 접근 감지 및 리스트 분리)
- **하드웨어 구현 집착**: 나노초 단위의 정밀함을 위해 너무 복잡한 하드웨어 로직을 넣으면 오히려 메모리 접근 레이턴시 자체가 늘어남. (해결: 소프트웨어 근사 알고리즘 활용)

- **📢 섹션 요약 비유**: 모든 사람을 똑같이 대우하는 것이 공정해 보이지만, 실제로는 오랫동안 묵묵히 기여해온 멤버(Active List)를 특별 관리하는 것이 조직의 안정성 유지에 유리합니다.

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (FIFO 대비)

| 구분 | FIFO 방식 | LRU 방식 | 개선 효과 |
|:---|:---|:---|:---|
| **페이지 폴트 횟수** | 국부성을 무시하여 잦은 발생 | 지역성 활용으로 폴트 최소화 | **평균 20~30% 감소** |
| **시스템 처리량 (TPS)** | I/O 대기로 인한 CPU 유휴 발생 | 캐시 적중률 상승으로 CPU 가동율↑ | **처리량 15~25% 향상** |
| **안정성 (Reliability)** | 벨라디 모순으로 인한 예측 불허 | 선형적인 성능 예측 가능 | **운영 가시성 확보** |

### 미래 전망
- **AI/ML 기반 예측 교체**: 단순 과거 기록이 아닌, 프로세스의 동작 패턴을 LSTM(Long Short-Term Memory) 등 신경망으로 학습하여 다음에 안 쓰일 페이지를 예측 교체하는 기술이 실험적으로 도입 중임.
- **계층형 메모리 (Tiered Memory)**: DRAM과 CXL 기반 확장 메모리, NVMe가 섞인 환경에서 LRU는 단순히 '삭제'가 아닌 '하위 계층으로의 이동(Demotion)'을 결정하는 정책으로 진화하고 있음.

1. LRU는 **"가장 오랫동안 안 만진 장난감 버리기"** 대작전이에요. 방금 가지고 놀았던 인형은 나중에 또 놀 확률이 높으니까 보물 상자에 남겨둬요.

3. 이 규칙을 지키면, 상자 안에 항상 내가 제일 좋아하는 인기 장난감들만 가득하게 유지할 수 있답니다!
