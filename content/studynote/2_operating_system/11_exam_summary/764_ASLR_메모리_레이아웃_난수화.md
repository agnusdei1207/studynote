# ASLR (Address Space Layout Randomization) 메모리 레이아웃 난수화

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: ASLR (Address Space Layout Randomization)은 프로세스가 실행될 때마다 스택(Stack), 힙(Heap), 공유 라이브러리, 그리고 실행 코드의 시작 주소를 가상 주소 공간 내에서 무작위로 배치하여 메모리 주소의 예측 가능성을 제거하는 운영체제의 보안 메커니즘이다.
> 2. **가치**: 버퍼 오버플로우 공격자가 악성 쉘코드(Shellcode)의 위치나 `system()` 같은 라이브러리 함수의 주소를 특정하지 못하게 함으로써 익스플로잇(Exploit)의 성공률을 낮추고, ROP (Return Oriented Programming)와 같은 고도화된 공격을 무력화한다.
> 3. **융합**: 데이터 실행 방지 (**DEP/NX**), 위치 독립 실행 파일 (**PIE**, Position Independent Executable) 기술과 유기적으로 결합되어 현대 운영체제의 다층적 메모리 보호 아키텍처를 구성한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: ASLR은 메모리의 '지도'를 매번 바꾸는 보안 기술이다. 똑같은 프로그램을 실행해도 어떤 날은 스택이 7번지에 있고, 어떤 날은 99번지에 있게 하여 공격자가 "7번지를 덮어쓰면 쉘이 실행되겠지?"라는 기대를 무너뜨린다.

- **💡 비유**: 이는 "매일 방 번호가 바뀌는 호텔"과 같다. 도둑(공격자)이 어제 101호에 금고가 있는 것을 확인하고 오늘 털러 왔는데, 오늘은 금고가 505호로 옮겨져 있고 101호에는 빈방만 있는 상황이다. 도둑은 금고 위치를 알기 위해 수만 번 시도해야 하며, 그 과정에서 보안 요원(IDS)에게 들킬 확률이 높아진다.

- **등장 배경**:
  1. **고정 주소의 취약성**: 초기 운영체제는 메모리 배치가 고정되어 있어, 한 번 개발된 공격 코드가 전 세계 모든 PC에서 동일하게 작동하는 대량 감염 사고가 잦았다.
  2. **DEP 우회 공격 (Ret-to-libc)**: 메모리에 실행 권한을 없애도(DEP), 이미 실행 권한이 있는 라이브러리 함수로 점프하는 공격이 나오자 함수 주소 자체를 숨길 필요가 생겼다.
  3. **브루트 포스 (Brute Force) 지연**: 공격자가 주소를 때려 맞히는 데 걸리는 시간을 기하급수적으로 늘려 공격의 경제성을 떨어뜨리고자 도입되었다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              ASLR 적용 전 vs 적용 후 메모리 레이아웃 비교           │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ No ASLR (Static) ]             [ With ASLR (Random) ]       │
  │   ┌────────────────┐               ┌────────────────┐           │
  │   │ Stack (0xBFFF) │ ◀─ 항상 동일   │ Stack (0x7FFE) │ ◀─ 매번   │
  │   ├────────────────┤               ├────────────────┤     변함  │
  │   │ Libraries      │               │ Libraries      │           │
  │   ├────────────────┤               ├────────────────┤           │
  │   │ Heap  (0x0804) │               │ Heap  (0x55AD) │           │
  │   ├────────────────┤               ├────────────────┤           │
  │   │ Code  (PIE)    │               │ Code  (Random) │           │
  │   └────────────────┘               └────────────────┘           │
  │                                                                 │
  │   [결과] 공격자가 return address를 특정 주소로 고정할 수 없음        │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ASLR이 적용되면 실행 시마다 각 세그먼트의 베이스 주소에 랜덤한 '오프셋(Offset)'이 더해진다. 공격자는 프로그램이 현재 어디에 로드되어 있는지 알 수 없으므로, 하드코딩된 주소값을 이용한 공격은 즉시 `Segmentation Fault`를 유발하며 실패하게 된다.

- **📢 섹션 요약 비유**: "보물의 위치를 매일 밤 몰래 바꾸어, 어제의 지도로는 보물을 절대 찾을 수 없게 만드는 보안관의 지혜"입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### ASLR의 3대 난수화 대상 영역

현대 OS는 메모리의 거의 모든 영역을 난수화하여 빈틈을 없앤다.

| 대상 영역 | 기술적 명칭 | 상세 메커니즘 | 비유 |
|:---|:---|:---|:---|
| **스택 (Stack)** | Stack Randomization | `RSP` 레지스터 시작점에 랜덤 오프셋 부여 | 가변적인 계단 높이 |
| **힙 (Heap)** | Heap Randomization | `brk` 또는 `mmap` 할당 시 베이스 주소 난수화 | 매번 바뀌는 주차 칸 |
| **라이브러리** | Shared Lib ASLR | `libc.so` 등 공용 라이브러리의 로드 위치 분산 | 비밀 장소에 숨긴 도구함 |
| **실행 코드** | **PIE** (Pos. Indep. Exec) | 메인 프로그램의 코드 세그먼트 자체를 난수화 | 건물 위치의 무작위 이동 |

---

### 리눅스 ASLR 제어 및 설정 (Kernel Parameter)

리눅스 커널은 `/proc/sys/kernel/randomize_va_space` 파일을 통해 난수화 수준을 결정한다.

1. **Level 0**: ASLR 비활성화. 모든 주소가 고정됨. (디버깅용)
2. **Level 1**: 스택, 가상 메모리 매핑(mmap), VDSO 영역만 난수화.
3. **Level 2 (Default)**: Level 1에 더해 **데이터(Heap) 영역**까지 완전히 난수화.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 ASLR의 한계: 엔트로피(Entropy)와 정보 유출           │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ 한계 1: Entropy ]                                            │
  │   32비트 시스템은 난수화 공간이 좁아(약 16비트), 브루트 포스로      │
  │   수 분 내에 주소를 맞출 수 있음. (64비트는 사실상 불가능)         │
  │                                                                  │
  │   [ 한계 2: Info Leak ]                                          │
  │   공격자가 포인터 하나만 유출(Format String Bug 등)시키면,        │
  │   상대적 거리를 계산하여 전체 메모리 지도를 복원 가능함.           │
  │                                                                  │
  │   [결론] ASLR은 만능이 아니며 DEP, Canary와 함께 써야 함           │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ASLR의 보호 강도는 얼마나 많은 주소 경우의 수(엔트로피)를 제공하느냐에 달려 있다. 64비트 환경에서는 주소 공간이 워낙 넓어 공격 성공률이 0에 수렴하지만, 메모리 정보를 살짝 흘리는 취약점이 있다면 ASLR은 무용지물이 될 수 있다.

- **📢 섹션 요약 비유**: "비밀번호가 아무리 길어도(64비트 ASLR), 누군가 힌트(Info Leak)를 주면 금방 뚫릴 수 있는 보안의 상대성"과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: ASLR vs DEP vs Stack Canary (3대 메모리 가디언)

| 보안 기술 | 핵심 목표 | 방어 방식 | 비유 |
|:---|:---|:---|:---|
| **ASLR** | 주소 예측 방지 | 메모리 배치 난수화 | **은신술** |
| **DEP (NX)** | 코드 실행 차단 | 스택/힙의 실행 권한 제거 | **방전복** |
| **Canary** | 오버플로우 탐지 | 복귀 주소 앞의 비밀값 확인 | **경보 장치** |

### 비교 2: 정적 컴파일 vs PIE (Position Independent Executable)

- **정적 주소 실행 파일**: 코드 내 주소가 고정되어 ASLR 환경에서도 메인 코드는 항상 같은 자리에 위치함. (보안 취약)
- **PIE 지원 실행 파일**: 컴파일 시 `-fPIE -pie` 옵션을 사용하여, 실행 시점에 어디로 옮겨져도 상대 주소(Relative)로 동작하게 만듦. ASLR의 효과를 극대화함.

- **과목 융합 관점 (컴퓨터 구조)**: ASLR은 하드웨어의 **MMU (Memory Management Unit)**를 이용한 가상 메모리 기법이 있었기에 가능해졌다. 물리 주소를 숨기고 가상 주소만 유저에게 보여주는 추상화 계층 덕분에, 실제 램의 위치와 상관없이 논리적 지도를 마음껏 섞을 수 있게 된 것이다.

- **📢 섹션 요약 비유**: "보호 장비(DEP)를 갖추고 경보 기기(Canary)를 설치해도, 내 위치를 들키지 않는 것(ASLR)이 생존율을 높이는 가장 영리한 방법입니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 서버 취약점 점검**: 외부로 노출된 웹 서버의 보안 설정을 점검함. `checksec` 도구를 사용하여 바이너리들을 검사해보니, 핵심 엔진이 PIE 없이 빌드되어 주소가 고정된 상태임. 기술사적 판단으로 **PIE 옵션을 포함한 재빌드**를 강력 권고하며, 커널 파라미터 `randomize_va_space`가 2로 설정되어 있는지 확인한다.

2. **시나리오 — 32비트 임베디드 장치 보안**: 주소 공간이 좁아 ASLR 엔트로피가 낮음. 공격자가 수백 번의 시도 끝에 주소를 맞추는 사고 발생. 해결책으로 ASLR 강화보다는 **Stack Canary와 무결성 검증**을 강화하고, 비정상적인 프로세스 재시작(Crash 반복)을 감지하여 계정을 잠그는 '행위 기반 차단' 정책을 병행한다.

3. **시나리오 — ROP (Return Oriented Programming) 방어**: 공격자가 DEP를 뚫기 위해 라이브러리 내의 코드 조각(Gadget)들을 엮어서 공격함. ASLR 덕분에 라이브러리 위치가 숨겨져 있어 가젯의 주소를 찾지 못하게 방해함으로써 공격 체인(Exploit Chain)을 끊어낸다.

### 도입 체크리스트 (Security Audit)
- **PIE 활성화**: 운영 중인 모든 핵심 서비스 바이너리가 PIE(Position Independent Executable) 모드로 컴파일되었는가?
- **커널 설정**: `sysctl kernel.randomize_va_space` 값이 2인가?
- **Library Patch**: 시스템 내 `libc` 등 공용 라이브러리가 ASLR을 지원하는 동적 공유 객체(DSO) 형태인가?

### 안티패턴
- **디버깅을 위해 ASLR 영구 비활성화**: "주소가 바뀌면 분석하기 힘들다"는 이유로 서버의 ASLR을 꺼두는 행위. 해커에게 레드카펫을 깔아주는 것과 같음. (해결: 분석 시에만 로컬에서 끄고 운영 서버는 상시 가동)
- **ASLR 단독 맹신**: "주소를 섞었으니 절대 못 뚫는다"는 자만. Info Leak 한 방에 무력화될 수 있음을 명심해야 함.

- **📢 섹션 요약 비유**: "숨바꼭질에서 가장 잘 숨는 방법(ASLR)은 단순히 구석에 숨는 것이 아니라, 숨는 장소를 매번 바꾸는 정성입니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (ASLR 아키텍처 도입 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **공격 비용 증가** | 공격자가 성공하기 위해 수만 배의 리소스와 시간 투입 유도 | 공격의 가성비 하락 |
| **범용 공격 차단** | 한 번 제작된 익스플로잇 코드가 다른 시스템에 재사용되는 것 방지 | 웜/바이러스 확산 억제 |
| **시스템 복원력** | 공격 시도가 높은 확률로 충돌(Crash)을 유발하여 탐지 기회 제공 | 침투 탐지 시간(MTTD) 단축 |
| **플랫폼 안전성** | 64비트 광대한 주소 공간과 결합하여 난공불락의 요새 구축 | 현대 컴퓨팅의 기본 방어선 |

### 미래 전망
- **Fine-grained ASLR**: 현재의 '세그먼트 단위' 난수화를 넘어, 함수 순서나 명령어 순서까지 실행 시점에 섞어버리는 '미세 난수화' 기술이 표준화될 것임.
- **Hardware-based ASLR (ARM MTE)**: 메모리 태깅 기술을 통해 하드웨어가 주소의 정당성을 나노초 단위로 검증하여, ASLR의 한계인 정보 유출(Info Leak)까지 원천 차단하는 시대 도래.

### 참고 표준
- **PaX Project**: ASLR 개념을 최초로 리눅스에 구현한 선구적 프로젝트.
- **NIST SP 800-123**: 가상화 및 운영체제 보안 가이드라인 내 메모리 보호 표준.

- **📢 섹션 요약 비유**: ASLR은 "무질서 속의 질서"를 통해 시스템을 보호하는 현대 운영체제의 가장 세련된 방패입니다.
