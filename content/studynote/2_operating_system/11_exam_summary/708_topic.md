# 318. 타조 알고리즘 (Ostrich Algorithm)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 교착상태를 무시하는 실용적 접근법
> 2. **가치**: 비용 대비 효과적인 선택
> 3. **융합**: 교착상태 무시, 교착상태 예방과 연관

---

## Ⅰ. 개요

### 개념 정의

타조 알고리즘(Ostrich Algorithm)은 **교착상태가 발생하더라도 무시하는 실용적 접근법**이다. 타조가 위험을 느끼면 모래에 머리를 묻고 보지 않는다는 전설에서 유래했다.

### 💡 비유: 타조의 모래
타조 알고리즘은 **머리를 모래에 묻는 타조**와 같다. 위험이 있어도 보지 않으면 없는 것과 같다. 문제가 커지면 재부팅한다.

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【운영체제에서의 적용】                                                 │
│  ──────────────────                                                  │
│  // Linux 커널: 타조 알고리즘 사용                                       │ │
│  // 교착상태 예방/회피 메커니즘 없음                                       │ │
│                                                                     │
│  // 뮤텍스 획득: 무한 대기 가능                                          │ │
│  void __sched mutex_lock(struct mutex *lock) {                      │
│      might_sleep();                                                  │
│      if (__mutex_trylock_fast(lock))                                │
│          return;                                                     │
│      __mutex_lock_slowpath(lock);  // 대기, 교착상태 가능             │ │
│  }                                                                  │
│  // 교착상태 발생 시: 사용자가 프로세스 종료 또는 재부팅                   │ │
│                                                                     │
│  【응용 프로그램에서의 적용】                                             │
│  ──────────────────                                                  │
│  // Java: 교착상태 처리 없이 락 사용                                     │ │
│  public class BankAccount {                                         │
│      private final Object lock1 = new Object();                     │
│      private final Object lock2 = new Object();                     │
│                                                                     │
│      public void transfer(Account a, Account b, int amount) {       │
│          synchronized (lock1) {  // 순서 다르면 교착상태 가능           │ │
│              synchronized (lock2) {                                 │
│                  a.withdraw(amount);                                │
│                  b.deposit(amount);                                 │
│              }                                                      │
│          }                                                          │
│          // 교착상태 예방/회피 없음 → 타조 알고리즘                        │ │
│      }                                                              │
│  }                                                                  │
│                                                                     │
│  【타조 알고리즘의 인정】                                                 │
│  ──────────────────                                                  │
│  // 사용자 매뉴얼에 명시                                                 │ │
│  /*                                                                  │
│   * Known Limitations:                                               │
│   * - 드물게 응답 없음 현상이 발생할 수 있음                               │ │
│   * - 이 경우 작업 관리자에서 강제 종료 후 재시작                          │ │
│   * - 이 현상은 데이터 손실을 유발할 수 있음                               │ │
│   * - 자주 발생하면 고객 지원에 문의                                       │ │
│   */                                                                 │
│                                                                     │
│  【간접적 대응: Watchdog】                                              │
│  ──────────────────                                                  │
│  // 타조 알고리즘이지만 Watchdog로 최소한의 대응                           │ │
│  class SimpleWatchdog {                                             │
│      Thread watchedThread;                                          │
│      long lastHeartbeat;                                            │
│      long timeout = 60000;  // 60초                                  │ │
│                                                                     │
│      void start() {                                                 │
│          Thread watchdog = new Thread(() -> {                       │
│              while (true) {                                         │
│                  try {                                               │
│                      Thread.sleep(10000);  // 10초마다 체크           │ │
│                      if (System.currentTimeMillis() - lastHeartbeat > timeout) {│
│                          log.error("Thread appears deadlocked!");   │
│                          // 로그만 남기고 복구는 사용자에게 맡김           │ │
│                      }                                              │
│                  } catch (InterruptedException e) {                 │
│                      break;                                         │
│                  }                                                  │
│              }                                                      │
│          });                                                         │
│          watchdog.setDaemon(true);                                  │
│          watchdog.start();                                          │
│      }                                                              │
│  }                                                                  │
│                                                                     │
│  【자동 재시작 설정】                                                    │
│  ──────────────────                                                  │
│  // systemd로 교착상태 시 자동 재시작                                    │ │
│  [Unit]                                                              │
│  Description=My Application (with Ostrich Algorithm)                │
│                                                                     │
│  [Service]                                                           │
│  ExecStart=/usr/bin/myapp                                           │
│  Restart=always                                                      │
│  RestartSec=10s                                                      │
│  # 교착상태로 멈추면 watchog이나 사용자가 kill                             │ │
│  # systemd가 자동으로 재시작                                            │ │
│                                                                     │
│  【모니터링】                                                            │
│  ──────────────────                                                  │
│  // 교착상태 발생 추적 (처리는 안 함)                                     │ │
│  class DeadlockTracker {                                            │
│      AtomicLong suspectedDeadlocks = new AtomicLong();              │
│                                                                     │
│      void reportSuspicion(Thread t) {                               │
│          suspectedDeadlocks.incrementAndGet();                      │
│          log.warn("Suspected deadlock in thread: {}", t.getName()); │
│          // 복구는 안 함, 로깅만                                         │ │
│      }                                                              │
│                                                                     │
│      long getSuspectedDeadlockCount() {                             │
│          return suspectedDeadlocks.get();                           │
│      }                                                              │
│  }                                                                  │
│                                                                     │
│  【타조 알고리즘 선택 이유 문서화】                                        │
│  ──────────────────                                                  │
│  /*                                                                  │
│   * Design Decision: Ostrich Algorithm                              │
│   *                                                             │ │
│   * Why we ignore deadlocks:                                         │
│   * 1. Historical data shows < 0.01% deadlock rate                  │
│   * 2. Prevention would add 15% latency to all lock operations      │
│   * 3. Users accept occasional restarts                             │
│   * 4. Implementation complexity of prevention is not justified      │
│   *                                                             │ │
│   * Trade-off: Occasional crash vs. constant performance penalty    │
│   */                                                                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 교착상태를 무시하는 실용적 접근법
• 유래: 타조가 모래에 머리를 묻는 전설
• 철학: 처리 비용 > 발생 비용이면 무시
• 장점: 구현 없음, 오버헤드 없음
• 단점: 발생 시 멈춤, 사용자 개입
• 적합: 일반 PC, 비실시간
• 부적합: 실시간, 미션 크리티컬
```
