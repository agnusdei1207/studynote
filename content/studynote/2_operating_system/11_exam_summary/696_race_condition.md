# 경쟁 조건 (Race Condition) 발생 원인 및 해결 방안

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 여러 프로세스나 스레드가 공유 자원에 동시에 접근하여 읽고 쓰는 과정에서, **접근 순서나 실행 타이밍에 따라 결과값이 달라지는** 비결정론적 상태.
> 2. **가치**: 경쟁 조건은 소프트웨어 버그 중에서도 재현이 가장 어렵고 치명적이며, 이를 방어하는 것이 **병렬 프로그래밍**의 핵심 품질 척도가 된다.
> 3. **융합**: 하드웨어의 인터럽트 발생, CPU 스케줄러의 선점, 그리고 컴파일러의 명령어 재배치 등이 복합적으로 작용하여 발생하며, 상호 배제(Mutual Exclusion)를 통해 해결한다.

---

### Ⅰ. 경쟁 조건 발생의 3대 요소 (Conditions)

경쟁 조건은 아래 세 가지 상황이 동시에 충족될 때 발생한다.

1. **공유 자원 (Shared Resource)**: 전역 변수, 파일, DB 등 여러 엔티티가 같이 쓰는 자원.
2. **수정 (Modification)**: 단순히 읽기만 하는 것이 아니라, 자원의 상태를 바꾸는 쓰기 작업 포함.
3. **비동기적 접근 (Asynchrony)**: 인터럽트나 스케줄링에 의해 실행 흐름이 언제든 바뀔 수 있는 환경.

---

### Ⅱ. 경쟁 조건 사례 분석 (ASCII)

가장 흔한 '은행 계좌 잔액 업데이트' 사례다.

```ascii
    [ Initial Balance: 100 ]
    
    Thread A (Deposit 50)              Thread B (Deposit 30)
    ---------------------              ---------------------
    1. Read Balance (100)              |
    |                                  1. Read Balance (100)
    2. Calc (100 + 50 = 150)           |
    |                                  2. Calc (100 + 30 = 130)
    3. Write Balance (150)             |
    |                                  3. Write Balance (130)
    
    [ Result: 130 ]  <-- WRONG! Should be 180.
```

**[분석]**
- Thread A의 작업이 완료(Write)되기 전에 Thread B가 예전 데이터를 읽어버려, A의 기여분(50)이 증발해버린 상황이다.

---

### Ⅲ. 해결 방안: 동기화와 원자성 (Solutions)

#### 1. 상호 배제 (Mutual Exclusion)
- **임계 구역 (Critical Section)**을 설정하고, 한 번에 하나의 스레드만 진입하도록 뮤텍스(Mutex)나 세마포어(Semaphore)를 사용한다.

#### 2. 원자적 연산 (Atomic Operation)
- `Read-Modify-Write` 과정을 하드웨어 수준에서 중단되지 않는 하나의 명령어로 처리한다. (예: `atomic_add`)

#### 3. 소프트웨어적 기법
- **Lock-free 알고리즘**: CAS(Compare-And-Swap) 명령어를 사용하여 락 없이도 데이터 일관성을 유지한다.
- **불변성 (Immutability)**: 공유 자원 자체를 수정 불가능하게 만들고, 변경 시 새 객체를 생성한다.

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 디버깅의 어려움 (Heisenbug)
- 경쟁 조건은 테스트 시에는 나타나지 않다가 운영 환경(실제 부하 상황)에서만 간헐적으로 나타나는 특성이 있다.
- **기술사적 결단**: 
  - 코드를 수동으로 검사하는 데 한계가 있으므로, **ThreadSanitizer**나 **Valgrind**와 같은 정적/동적 분석 도구를 빌드 파이프라인에 통합해야 한다.
  - 근본적으로는 공유 자원을 최소화하는 아키텍처(Actor Model 등) 설계를 권고한다.

#### 2. 기술사적 인사이트
- **데이터 레이스(Data Race)**와 경쟁 조건은 다르다. 데이터 레이스는 락 없는 동시 접근 자체를 말하며, 경쟁 조건은 그로 인해 시스템의 논리적 결함(결과 오류)이 생기는 상위 개념이다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **시스템 신뢰도 확보**: 예측 가능한 비즈니스 로직 수행 보장.
- **보안 사고 예방**: 경쟁 조건을 악용한 권한 상승(Race Condition Exploit) 공격 차단.

#### 2. 미래 전망
최근 **Rust** 언어와 같이 컴파일 타임에 소유권(Ownership)을 체크하여 경쟁 조건을 원천 봉쇄하는 언어들이 시스템 프로그래밍의 표준으로 부상하고 있다. 하드웨어 측면에서는 트랜잭셔널 메모리(HTM)가 보편화되어 복잡한 동기화 코드 없이도 고성능 동시 처리가 가능한 시대가 올 것이다.
