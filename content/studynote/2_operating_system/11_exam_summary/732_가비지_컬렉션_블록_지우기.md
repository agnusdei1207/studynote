# 가비지 컬렉션 (SSD Garbage Collection)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 가비지 컬렉션 (GC)은 SSD 내부에서 더 이상 유효하지 않은 데이터(Invalid Page)가 포함된 물리적 블록을 선택하여, 유효한 데이터(Valid Page)만을 새로운 빈 블록으로 이주(Migration)시킨 후 기존 블록을 통째로 소거(Erase)하여 가용 쓰기 공간을 확보하는 청소 프로세스다.
> 2. **가치**: 플래시 메모리의 "덮어쓰기 불가" 및 "블록 단위 소거" 제약을 극복하여 시스템의 지속적인 쓰기 성능을 유지하게 하지만, 이 과정에서 발생하는 추가적인 데이터 이동으로 인해 쓰기 증폭 (WAF, Write Amplification Factor) 현상을 유발하여 칩 수명에 영향을 준다.
> 3. **융합**: 운영체제의 TRIM 명령어 지원 여부에 따라 효율이 극명하게 갈리며, FTL (Flash Translation Layer)의 주소 매핑 및 마모 평준화 (Wear Leveling) 알고리즘과 유기적으로 결합되어 현대 고성능 스토리지의 응답 일관성 (QoS)을 결정하는 핵심 기제로 작용한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 가상 메모리의 가비지 컬렉션이 메모리 누수를 막는 소프트웨어적 작업이라면, SSD의 가비지 컬렉션은 "지우지 않으면 쓸 수 없는" 낸드 플래시의 물리적 한계를 극복하기 위해 컨트롤러가 수행하는 하드웨어 관리 작업이다.

- **💡 비유**: 이는 "포스트잇이 가득 붙은 칠판 정리"와 같다. 칠판(블록)의 일부분만 지우는 것은 불가능하여, 아직 쓸모 있는 메모(유효 페이지)들만 옆의 새 칠판으로 옮겨 적은 뒤, 기존 칠판을 걸레로 싹 지워(Erase) 다시 깨끗한 상태로 만드는 과정이다.

- **등장 배경**:
  1. **Out-of-place Update의 산물**: 데이터를 수정할 때 기존 위치를 두고 새 위치에 기록하는 FTL의 특성상, 물리 계층에는 '무효화된 쓰레기 데이터'가 필연적으로 쌓이게 된다.
  2. **쓰기 지연 방지**: 빈 공간이 없는데 쓰기 요청이 들어오면 그제야 지우기 작업을 수행해야 하므로(Foreground GC), 이를 방지하기 위해 미리미리 공간을 치워둘 필요가 생겼다.
  3. **데이터 단편화 해결**: 흩어져 있는 유효 데이터들을 한데 모음으로써 큰 단위의 연속된 빈 블록을 확보해야 효율적인 쓰기가 가능해진다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              SSD 가비지 컬렉션 (GC) 단계별 동작 흐름                │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [단계 1] 대상 블록 선정 (Victim Selection)                      │
  │   - 무효 페이지(Invalid)가 가장 많은 블록을 우선 선택              │
  │                                                                 │
  │   [단계 2] 유효 데이터 이주 (Valid Data Migration)                │
  │   - 블록 A의 유효 페이지(P1, P3)를 깨끗한 블록 B로 복사            │
  │                                                                 │
  │   [단계 3] 기존 블록 소거 (Block Erase)                           │
  │   - 블록 A 전체를 물리적으로 소거하여 'Free Block' 상태로 전환     │
  │                                                                 │
  │   [단계 4] 주소 장부 갱신 (Mapping Update)                       │
  │   - FTL 매핑 테이블의 물리 주소를 블록 B의 새 위치로 업데이트      │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** GC의 핵심은 '모으고 지우기'다. 무효 데이터(쓰레기)가 섞여 있는 블록에서 보석(유효 데이터)만 골라내어 새 집으로 이사시키고, 헌 집은 폭파(Erase)하여 대지를 확보하는 과정이다. 이 과정이 효율적일수록 SSD는 지치지 않고(성능 저하 없이) 일할 수 있다.

- **📢 섹션 요약 비유**: "재건축을 위해 주민들을 새 아파트로 이주시킨 뒤, 낡은 단지 전체를 한꺼번에 철거하여 새 부지를 만드는 도시 계획"과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 가비지 컬렉션의 효율을 결정하는 3대 요소

| 요소명 | 기술적 역할 | 성능 영향도 | 비유 |
|:---|:---|:---|:---|
| **TRIM 명령어** | OS가 삭제한 파일 정보를 FTL에 전달 | GC 대상 페이지를 미리 'Invalid'로 표시하여 복사 비용 절감 | 사전에 버릴 물건 표시 |
| **오버 프로비저닝 (OP)** | 사용자가 못 쓰는 예약된 여유 공간 | GC를 수행할 수 있는 '작업장' 면적 확보, WAF 감소 효과 | 작업용 빈 책상 |
| **빅팀 선정 알고리즘** | 어떤 블록을 먼저 지울지 결정 | Greedy(가장 쓰레기 많은 것) 또는 Cost-benefit(지운 횟수 고려) 적용 | 분리수거 우선순위 |

### 쓰기 증폭 (WAF, Write Amplification Factor)

GC의 가장 뼈아픈 부작용은 호스트가 요청한 것보다 더 많은 물리적 쓰기가 발생하는 **쓰기 증폭** 현상이다.

1. **정의**: $WAF = \frac{\text{실제 플래시 미디어에 기록된 데이터량}}{\text{호스트 운영체제가 요청한 데이터량}}$
2. **이상적인 값**: 1.0 (호스트가 4KB 쓰면 물리적으로도 4KB만 쓰는 것)
3. **GC의 영향**: 유효 페이지를 옮기기 위해 추가적인 쓰기가 발생하므로 WAF는 보통 1.5 ~ 3.0 이상으로 치솟는다.
4. **결과**: WAF가 높을수록 SSD의 수명은 단축되고 쓰기 속도는 느려진다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 쓰기 증폭 (WAF) 시각화 예시                        │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   OS: "4KB 한 페이지만 써줘!" (Host Write)                       │
  │                                                                  │
  │   SSD Controller: "알았어. 그런데 빈 공간을 만들려고 보니          │
  │                    옆집 데이터 12KB도 같이 옮겨야겠네?"           │
  │                                                                  │
  │   실제 쓰기: (새 데이터 4KB) + (이동 데이터 12KB) = 16KB           │
  │                                                                  │
  │   => WAF = 16KB / 4KB = 4.0                                      │
  │                                                                  │
  │   [결론] GC 부하가 클수록 WAF가 상승하여 성능과 수명을 갉아먹음      │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 사용자는 작게 요청했지만, SSD 내부에서는 대청소를 위해 큰 데이터를 옮겨 적는 비효율이 발생한다. 이를 최소화하는 것이 FTL 설계의 정수이며, TRIM과 OP(Over-provisioning)가 이 비효율을 줄여주는 구세주 역할을 한다.

- **📢 섹션 요약 비유**: "책 한 권을 넣으려고 책장 전체를 다시 정리하다가 팔 근육(수명)을 더 많이 쓰게 되는 억울한 상황"이 바로 쓰기 증폭입니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: Foreground GC vs Background GC

| 비교 항목 | Foreground GC (전경 처리) | Background GC (배경 처리) |
|:---|:---|:---|
| **실행 시점** | 가용 블록이 완전히 고갈되었을 때 | 시스템이 한가할 때 (Idle Time) |
| **사용자 체감** | 갑작스러운 성능 급락 (Stuttering) | 성능 저하 거의 없음 |
| **위험성** | I/O 요청이 들어와도 GC 끝날 때까지 응답 불가 | GC 도중 갑자기 요청 오면 중단 오버헤드 발생 |
| **전력 소모** | 단기간 집중 소모 | 저전력 모드 진입 방해 가능 |

### 비교 2: 파일 시스템 삭제 vs SSD 내부 삭제 (TRIM의 역할)

- **TRIM 미지원 시**: 파일 시스템에서 파일을 지워도 SSD는 해당 위치가 유효한 줄 알고 GC 때마다 계속 복사하며 보존한다. (비효율의 극치)
- **TRIM 지원 시**: OS가 삭제 즉시 "이 주소는 이제 쓰레기야"라고 알려주면, FTL은 복사 목록에서 즉시 제외하여 GC 속도를 비약적으로 높인다.

- **과목 융합 관점 (데이터베이스)**: DB 서버에서 로그 파일 기록 방식(Append-only)은 SSD의 GC 부하를 줄이는 데 유리하다. 순차적으로 데이터를 쓰면 블록 내부 페이지들이 비슷한 시점에 무효화되어, GC가 옮길 유효 데이터 자체가 적어지기 때문이다. 반면 무작위 수정이 잦은 워크로드는 파편화를 유발하여 GC 지옥을 만든다.

- **📢 섹션 요약 비유**: "버릴 물건에 미리 빨간 딱지(TRIM)를 붙여두면, 이삿짐센터(GC) 직원이 헛수고를 하지 않아 이사 비용(WAF)이 줄어드는 원리"입니다.

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — SSD 성능 유지 (Aging)**: 새 SSD는 빨랐는데 1년 쓰니 느려짐. 원인은 '더티 스테이트 (Dirty State)' 진입에 따른 Foreground GC 발생. 해결책으로 전체 용량의 10%를 파티션 잡지 않고 비워두는 **'수동 오버 프로비저닝'**을 실시하거나, 리눅스의 `fstrim` 서비스를 주기적으로 실행하여 수동으로 TRIM 신호를 보냄.

2. **시나리오 — 실시간 임베디드 시스템의 지연**: 드론이나 자율주행 장치에서 영상 기록 중 가끔 1~2초간 멈춤 현상 발생. 분석 결과 GC가 I/O 요청을 막고 있었음. 해결책으로 **Determinism 지원 SSD**를 사용하여 GC 수행 시간을 호스트가 제어하거나, 쓰기 요청이 없는 시간에만 GC를 수행하도록 펌웨어 파라미터를 조정함.

3. **시나리오 — 데이터 포렌식과 GC**: 범죄자가 파일을 지웠는데 포렌식 팀이 복구하려 함. 하지만 SSD의 Background GC가 이미 작동하여 해당 블록을 소거했다면 복구는 물리적으로 불가능함. HDD와 SSD의 증거 보존 방식이 근본적으로 달라지는 지점임.

### 도입 체크리스트 (스토리지 선정 시)
- **WAF 최적화 수준**: 컨트롤러 제조사의 GC 알고리즘 신뢰도 확인.
- **DRAM 유무**: DRAM-less 모델은 매핑 테이블 업데이트와 GC 속도가 현저히 느림.
- **OP 비율**: 데이터 센터용 엔터프라이즈 SSD는 보통 20~30%의 높은 OP를 기본 탑재함.

### 안티패턴
- **TRIM 기능 비활성화**: "보안을 위해 덮어쓰기 정보를 안 주겠다"며 TRIM을 끄는 행위. SSD는 곧 느려지고 수명은 반토막 난다.
- **가득 채워 쓰기 (Filling up)**: 용량의 99%를 채우면 GC가 유효 데이터를 옮길 공간이 없어 '무한 복사' 루프에 빠지고 시스템이 프리징됨.

- **📢 섹션 요약 비유**: "청소할 공간도 없이 짐을 꽉 채워 넣는 것은, 집을 쓰레기장으로 만들어 스스로 쫓겨나게 만드는 자폭 행위"입니다.

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (지능형 GC 도입 시)

| 구분 | 도입 전 (비효율 GC) | 도입 후 (지능형 GC + TRIM) | 개선 효과 |
|:---|:---|:---|:---|
| **쓰기 수명 (TBW)** | 300 TBW | 600 TBW 이상 | **수명 2배 연장** |
| **쓰기 레이턴시** | 들쭉날쭉함 (Jitter) | 일관된 성능 유지 | **사용자 체감 성능 안정화** |
| **WAF 수치** | 5.0 이상 | 1.2 미만 | **기계적 부하 극소화** |

### 미래 전망
- **ZNS (Zoned Namespaces)**: 아예 GC 자체가 필요 없도록, 호스트가 데이터의 '수명'이 비슷한 것끼리 구역을 나누어 담게 하는 혁신적 기술이 표준화 중임.
- **Host-managed GC**: 하이퍼바이저가 서버 내 수많은 SSD의 GC 타이밍을 스케줄링하여, 특정 시점에 모든 디스크가 동시에 GC에 빠지는 'I/O 블랙아웃'을 방지하는 상위 레벨 오케스트레이션 발전.

### 참고 표준
- **T13 ATA/ATAPI Command Set (Data Set Management - TRIM)**: TRIM 명령어 표준 정의.
- **NVMe Management Interface (MI)**: GC 상태 모니터링 및 성능 통계 수집 표준 규격.

- **📢 섹션 요약 비유**: 가비지 컬렉션은 SSD라는 유기체가 숨을 쉬기 위한 "노폐물 배설" 과정이며, 이 과정이 원활해야 시스템이라는 신체가 건강하게 유지됩니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **FTL (Flash Translation Layer)**: GC를 직접 실행하고 관리하는 SSD의 뇌.
- **마모 평준화 (Wear Leveling)**: GC 과정에서 데이터 위치를 전략적으로 배치하는 기술.
- **쓰기 증폭 (WAF)**: GC의 효율성을 보여주는 가장 정직한 성적표.
- **낸드 플래시 소거 단위 (Block)**: 왜 페이지 단위로 못 지우고 GC를 해야 하는지에 대한 물리적 근거.

### 👶 어린이를 위한 3줄 비유 설명
1. 가비지 컬렉션은 SSD라는 방 안의 **"장난감 정리 시간"**이에요. 가지고 놀다가 흩어놓은 장난감(쓰레기)들 사이에서, 아직 아끼는 인형(유효 데이터)만 골라내어 새 상자에 담는 거죠.
2. 장난감을 다 치운 낡은 상자는 텅 비게 되어, 나중에 새로운 선물을 담을 수 있는 **깨끗한 상자**가 된답니다.
3. 이 정리를 매일 잘해줘야지, 방 안이 쓰레기로 가득 차서 발 디딜 틈도 없게 되는 불상사를 막을 수 있어요!
