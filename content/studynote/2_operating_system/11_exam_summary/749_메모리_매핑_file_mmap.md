# 메모리 매핑 파일 (mmap, Memory-mapped Files)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메모리 매핑 파일 (mmap)은 프로세스의 가상 주소 공간의 일부를 특정 파일의 데이터 영역과 1:1로 매핑하여, 파일 입출력을 `read`/`write` 시스템 콜 대신 직접적인 메모리 접근 명령어(`load`/`store`)로 수행할 수 있게 하는 운영체제의 메모리 관리 기법이다.
> 2. **가치**: 커널 공간과 사용자 공간 사이의 불필요한 데이터 복사(Double Buffering) 오버헤드를 제거하여 입출력 성능을 극대화하며, 동일한 물리 메모리 영역을 여러 프로세스의 주소 공간에 매핑함으로써 고속의 공유 메모리 IPC(Inter-Process Communication) 수단을 제공한다.
> 3. **융합**: 가상 메모리의 요구 페이징(Demand Paging) 메커니즘을 파일 시스템과 결합한 형태로, 데이터베이스 엔진(LMDB, RocksDB), 고성능 로그 시스템, 그리고 운영체제의 실행 파일(ELF) 및 라이브러리 로딩의 표준 방식으로 활용된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: mmap은 파일을 "메모리처럼" 취급하는 기술이다. 파일을 열고 포인터를 하나 받으면, 그 포인터를 통해 데이터를 읽고 쓰는 행위가 곧 파일에 대한 입출력으로 직결된다. 운영체제는 매핑된 영역에 접근할 때만 실제 파일 데이터를 메모리로 로드(Demand Paging)한다.

- **💡 비유**: 이는 도서관에서 책을 한 장씩 복사해서 자리에 가져오는 것(Standard I/O)이 아니라, 책상 자체를 도서관 서가 옆으로 옮겨서 책을 펼쳐두고 직접 읽는 것(mmap)과 같다. 복사하는 수고가 없고, 내가 책에 밑줄을 그으면 서가에 있는 원본 책에도 즉시 반영되는 식이다.

- **등장 배경**:
  1. **I/O 시스템 콜의 오버헤드**: `read()`와 `write()`는 호출할 때마다 커널 모드 전환과 데이터 복사가 발생하여 대용량 파일 처리 시 성능 병목이 심각했다.
  2. **메모리 활용의 효율화**: 가상 메모리 관리 기법(Paging)이 고도화됨에 따라, 디스크 블록을 페이지 프레임으로 직접 관리하는 것이 논리적으로 더 깔끔해졌다.
  3. **프로세스 간 데이터 공유**: 서로 다른 프로세스가 거대한 데이터셋을 복사 없이 초고속으로 공유해야 하는 실시간 통신 요구가 커졌다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              Standard I/O vs mmap 입출력 아키텍처 비교              │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ Standard I/O ]                 [ Memory Mapped I/O ]        │
  │                                                                 │
  │   ┌──────────────┐                 ┌──────────────┐             │
  │   │  User Buffer │                 │ User Address │             │
  │   └──────▲───────┘                 │    Space     │             │
  │          │ (Copy 2)                │ (Mapped Reg) │             │
  │   ┌──────▼───────┐                 └──────▲───────┘             │
  │   │  Page Cache  │                        │                     │
  │   └──────▲───────┘                 ┌──────▼───────┐             │
  │          │ (Copy 1)                │  Physical    │             │
  │   ┌──────▼───────┐                 │   Memory     │             │
  │   │     Disk     │                 └──────▲───────┘             │
  │   └──────────────┘                        │ (Direct Paging)     │
  │                                    ┌──────▼───────┐             │
  │                                    │     Disk     │             │
  │                                    └──────────────┘             │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 일반 I/O는 디스크에서 커널 캐시로, 다시 유저 버퍼로 총 2번의 복사가 일어난다. 반면 mmap은 파일이 물리 메모리의 페이지 프레임과 직접 연결되며, 유저 주소 공간은 이 프레임을 직접 가리킨다. '중간 복사'가 사라지는 이 획기적인 절차가 mmap 고속화의 핵심이다.

- **📢 섹션 요약 비유**: "물건을 전달할 때 전달자를 거치지 않고, 창고 문을 직접 열어 고객이 바로 가져가게 하는 직거래 시스템"과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### mmap의 내부 동작 메커니즘 (Page Fault 기반)

mmap은 호출 즉시 데이터를 로드하지 않는다. 실제 접근이 일어날 때까지 로딩을 미루는 '게으른 로딩(Lazy Loading)' 방식을 취한다.

1. **Mapping**: `mmap()` 호출 시 커널은 프로세스의 `vm_area_struct`에 새로운 영역을 등록하고 파일 객체와 연결만 해둔다.
2. **First Access**: 프로세스가 매핑된 주소를 읽으려 하면 하드웨어(MMU)는 해당 주소에 대응하는 물리 프레임이 없음을 감지하고 **Page Fault**를 발생시킨다.
3. **Paging In**: 커널의 페이지 폴트 핸들러가 발생 위치를 확인하고, 연결된 파일의 해당 오프셋 데이터를 물리 메모리로 읽어 들인다.
4. **PTE Update**: 페이지 테이블을 갱신하여 가상 주소와 물리 프레임을 연결하고, 명령어를 재실행한다.

### mmap의 주요 파라미터 및 플래그

| 항목 | 의미 및 역할 | 상세 설명 | 비유 |
|:---|:---|:---|:---|
| **prot (Protection)** | 접근 권한 설정 | `PROT_READ`, `PROT_WRITE`, `PROT_EXEC` 조합 | 출입 허가증 종류 |
| **MAP_SHARED** | 공유 매핑 | 메모리 수정 시 즉시 파일에 반영, 타 프로세스와 공유 | 공용 칠판 |
| **MAP_PRIVATE** | 사적 매핑 | 메모리 수정 시 복사본 생성(COW), 원본 파일 미수정 | 개인 연습장 |
| **MAP_ANONYMOUS** | 익명 매핑 | 특정 파일 없이 순수 메모리만 할당 (`malloc` 대용) | 빈 캔버스 |
| **msync()** | 동기화 | 메모리 변경 내용을 강제로 디스크에 기록 | 중간 저장 버튼 |

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 mmap을 이용한 프로세스 간 데이터 공유                │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   Process A's VA          Physical RAM          Process B's VA   │
  │   ┌───────────┐          ┌───────────┐          ┌───────────┐    │
  │   │  Mapped   │─────────▶│ File Page │◀─────────│  Mapped   │    │
  │   │  Region   │          │ (Shared)  │          │  Region   │    │
  │   └───────────┘          └─────▲─────┘          └───────────┘    │
  │                                │                                 │
  │                          ┌─────▼─────┐                           │
  │                          │ File Data │ (on Disk)                 │
  │                          └───────────┘                           │
  │                                                                  │
  │   [가치] 메시지 복사 없이 메모리 쓰기만으로 즉시 데이터 교환 완료      │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 두 프로세스가 같은 파일을 `MAP_SHARED`로 열면, 각자의 가상 주소는 다르지만 결국 동일한 물리 메모리 프레임을 가리키게 된다. A가 메모리에 값을 쓰면 B가 즉시 그 값을 읽을 수 있는 가장 강력한 IPC 수단이 탄생한다.

- **📢 섹션 요약 비유**: "각자의 방에서 벽을 헐어 하나의 거실을 공유함으로써, 물건을 주고받는 수고 없이 바로 옆 사람의 물건을 함께 쓰는 공동 주거 방식"과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: mmap vs read/write (대결)

| 비교 항목 | mmap 입출력 | Standard I/O (read/write) |
|:---|:---|:---|
| **데이터 복사** | 0~1회 (커널-유저 복사 없음) | 최소 2회 (커널 캐시 ↔ 유저 버퍼) |
| **시스템 콜 횟수** | 시작 시 1회 (이후 메모리 접근) | 데이터 단위마다 반복 호출 |
| **주소 공간 소모** | 가상 주소 공간 대량 점유 | 고정된 버퍼 크기만 소모 |
| **적합한 상황** | 대용량 파일, 랜덤 액세스, IPC | 소량 파일, 순차 읽기, 파이프/소켓 |
| **무결성** | 크래시 시 손상 위험 높음 | `fsync` 등으로 제어 용이 |

### 비교 2: malloc vs mmap (메모리 할당 관점)

- **`malloc()`**: 내부적으로 작은 크기는 `brk()` 시스템 콜을, 큰 크기(보통 128KB 이상)는 `mmap()` 익명 매핑을 사용한다.
- **`mmap()`**: 힙(Heap) 영역의 한계를 넘어 가상 주소 공간 어디든 거대 메모리 블록을 할당할 수 있으며, 사용 후 `munmap()`으로 즉시 OS에 자원을 반납할 수 있어 메모리 파편화 관리에 유리하다.

- **과목 융합 관점 (컴퓨터 구조)**: mmap은 하드웨어의 **DMA (Direct Memory Access)** 기술과 환상의 궁합을 이룬다. DMA가 디스크 데이터를 메모리로 쏴주면, mmap된 프로세스는 CPU 개입 없이 즉시 그 데이터를 읽을 수 있다. 이는 고성능 컴퓨팅(HPC) 아키텍처의 기본 원리다.

- **📢 섹션 요약 비유**: "택배를 매번 주문해서 받는 것(Standard I/O)과, 아예 가게에 상주하며 물건을 집는 것(mmap)의 성능 차이는 비교할 수 없을 만큼 큽니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 고성능 DB 저장 엔진 (SQLite, LMDB)**: 수백만 건의 레코드를 조회해야 할 때. DB 파일 전체를 mmap으로 매핑한다. OS의 가상 메모리 관리자가 '자주 쓰이는 페이지'를 알아서 캐싱(Page Cache)해주므로, DB 개발자가 복잡한 버퍼 관리 로직을 직접 짤 필요 없이 OS 수준의 최적화 성능을 그대로 누릴 수 있다.

2. **시나리오 — 실행 파일 로더 (Dynamic Linker)**: 프로그램 실행 시 `.exe`나 `.so` 파일을 메모리에 올리는 과정. 커널은 파일의 섹션들을 주소 공간에 mmap한다. 덕분에 프로그램의 모든 코드를 부팅 시점에 다 읽지 않고, 실행되는 부분만 그때그때 읽어와 실행 속도를 높이고 메모리를 절약한다.

3. **시나리오 — 대규모 로그 분석 툴**: 100GB 로그 파일에서 특정 키워드를 검색해야 함. `read()`로 읽으면 메모리가 부족해지거나 CPU 점유율이 폭증함. mmap을 쓰면 파일 크기와 상관없이 가상 주소에 띄워두고 필요한 페이지만 '슥' 훑고 지나갈 수 있어 매우 효율적이다.

### 도입 체크리스트
- **가상 주소 공간**: 32비트 시스템인가? (32비트에서는 mmap 가능 용량이 2~3GB로 제한되어 대용량 파일 매핑 불가)
- **파일 수정 빈도**: 빈번한 작은 쓰기가 발생하는가? (매번 Dirty 페이지 체크와 Flush 오버헤드가 발생할 수 있음)
- **공유 모드**: 데이터 일관성이 중요한가? (여러 프로세스가 쓸 경우 `Lock` 메커니즘 필수)

### 안티패턴
- **네트워크 파일 시스템(NFS) 매핑**: 네트워크 상의 파일을 mmap 하면, 네트워크 지연이나 끊김 발생 시 프로세스가 `Bus Error`로 죽거나 알 수 없는 지연(I/O stall)에 빠짐.
- **파일 크기 변경**: mmap 도중 외부에서 파일 크기를 줄여버리면, 매핑된 주소 접근 시 `SIGBUS` 시그널이 발생하여 프로그램이 비정상 종료됨.

- **📢 섹션 요약 비유**: "mmap은 가장 빠른 길이지만, 지도가 바뀌거나(파일 축소) 길이 끊기면(네트워크 장애) 대형 사고가 날 수 있는 고속도로와 같습니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (mmap 도입 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **I/O 처리량** | Standard I/O 대비 2배~10배 이상 향상 | 복사 및 시스템 콜 제거 |
| **메모리 효율** | 커널-유저 중복 캐싱 제거로 실제 램 사용량 50% 절감 | Single Copy 특성 |
| **개발 생산성** | 복잡한 버퍼 관리 및 캐싱 알고리즘 구현 생략 | OS 최적화 로직 활용 |
| **프로세스 통신** | 메모리 접근 속도의 초고속 데이터 공유 실현 | 제로 카피(Zero-copy) IPC |

### 미래 전망
- **Persistent Memory (NVMe/Optane) 시대**: 디스크 자체가 메모리 버스에 달리는 시대가 오면, 모든 I/O는 mmap으로 통일될 것임. 파일과 메모리의 경계가 완전히 사라지는 '단일 레벨 저장소(Single-level Store)'의 핵심 기술이 될 전망.
- **Io_uring과의 결합**: 리눅스의 최신 비동기 I/O 인터페이스인 io_uring과 mmap을 결합하여, 유저 공간과 커널 공간의 완전한 Zero-copy 통신 환경이 보편화될 것임.

### 참고 표준
- **IEEE 1003.1 (POSIX)**: `mmap()`, `munmap()`, `mprotect()`, `msync()` 함수 명세 표준.
- **Linux Kernel `mmap_lock`**: 멀티스레드 환경의 주소 공간 경합 관리 표준.

- **📢 섹션 요약 비유**: mmap은 "가장 단순한 인터페이스로 가장 강력한 성능을 내는 운영체제의 치트키"이며, 이를 잘 다루는 것이 시니어 엔지니어의 척도입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **요구 페이징 (Demand Paging)**: mmap이 실제로 데이터를 메모리에 올리는 물리적 기제.
- **쓰기 시 복사 (COW)**: `MAP_PRIVATE` 설정 시 작동하는 핵심 최적화 기술.
- **버퍼 캐시 (Page Cache)**: mmap이 데이터를 빌려오는 커널의 실제 저장소.
- **페이지 폴트 (Page Fault)**: mmap 영역 접근 시 발생하는 하드웨어 예외 상황.

### 👶 어린이를 위한 3줄 비유 설명
1. mmap은 두꺼운 백과사전(파일)을 한 페이지씩 넘기며 공부하는 게 아니라, **"책상 위에 책을 쫙 펼쳐놓는 것"**과 같아요.
2. 내가 보고 싶은 페이지만 슥 쳐다보면 마법처럼 그 내용이 나타나고, 책에 낙서(쓰기)를 하면 **원본 책에도 그대로** 낙서가 남는답니다.
3. 이 마법 덕분에 컴퓨터는 무거운 데이터를 옮기느라 땀 흘리지 않고, **눈으로 슥 훑는 것만으로** 엄청나게 빠른 일을 할 수 있어요!
