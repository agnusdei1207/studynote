# 접근 제어 목록 (ACL, Access Control List)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 접근 제어 목록 (ACL)은 파일, 디렉터리, 네트워크 객체 등 시스템 자원에 대하여 누가(User/Group) 어떤 동작(Read/Write/Execute)을 할 수 있는지를 명시한 데이터베이스 기반의 상세 권한 명세서다.
> 2. **가치**: 전통적인 유닉스 권한 모델(UGO: User, Group, Others)의 경직성을 극복하여 복수의 특정 사용자나 그룹별로 상이한 권한을 부여할 수 있는 세밀한(Granular) 보안 통제를 가능하게 하며, '최소 권한의 원칙'을 실무적으로 구현하는 핵심 도구다.
> 3. **융합**: 운영체제 파일 시스템(ext4, NTFS)뿐만 아니라 네트워크 프로토콜(NFSv4), 방화벽 룰셋, 클라우드 IAM(Identity and Access Management) 등 현대 IT 인프라 전반의 권한 설계 표준으로 활용된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: ACL은 객체에 접근하려는 주체(Subject)의 권한을 객체 측면에서 관리하는 목록이다. 특정 파일에 대해 "철수는 읽기만 가능하고, 영희는 읽기와 쓰기가 모두 가능하다"는 식의 개별적인 규칙을 무한히 추가할 수 있다.

- **💡 비유**: 이는 아파트 공동 현관의 "출입 명부"와 같다. 전통적인 방식이 "입주민은 들어오고 외부인은 못 들어온다"는 흑백논리였다면, ACL 방식은 명부에 "배달원은 오전 10시~12시만 가능", "청소 업체는 월/수/금만 가능", "친척인 길동이는 언제든 가능"처럼 구체적이고 다양한 조건을 적어두고 관리하는 것과 같다.

- **등장 배경**:
  1. **조직 구조의 복잡화**: 한 파일에 대해 서로 다른 부서의 여러 사용자가 각기 다른 권한을 가져야 하는 협업 환경이 보편화되었다.
  2. **전통적 UGO 모델의 한계**: 유닉스 표준 권한(755, 644 등)으로는 '두 번째 특정 사용자'나 '세 번째 그룹'에게 권한을 줄 방법이 없어 보안 구멍이 발생했다.
  3. **보안 감사(Audit) 요구**: 기업의 컴플라이언스 준수를 위해 누가 어떤 자원에 접근할 수 있는지에 대한 명확하고 정교한 가시성이 필요해졌다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              유닉스 기본 권한(UGO) vs ACL 구조 비교                 │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ Legacy UGO Model ]             [ Modern ACL Model ]         │
  │   - 파일당 딱 3개의 엔트리만 존재     - 필요한 만큼 엔트리 추가 가능    │
  │                                                                 │
  │   Owner : alice (rwx)              Owner : alice (rwx)          │
  │   Group : dev   (r-x)              Group : dev   (r-x)          │
  │   Other :       (---)              Other :       (---)          │
  │                                    ---------------------------  │
  │                                    User  : bob   (rw-)  ◀─ 추가 │
  │                                    Group : qa    (r--)  ◀─ 추가 │
  │                                    Mask  :       (rw-)  ◀─ 한도 │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 전통적인 방식에서는 bob이라는 특정 개인에게만 권한을 주려면 bob을 위한 새로운 그룹을 만들거나 bob에게 소유권을 넘겨야 하는 등 관리적 오버헤드가 컸다. ACL은 i-node의 메타데이터 영역(Extended Attributes)을 활용하여 bob에 대한 개별 규칙을 즉시 덧붙임으로써 이 문제를 우아하게 해결한다.

- **📢 섹션 요약 비유**: "단 세 칸짜리 고정된 서랍(UGO)을 버리고, 필요한 만큼 이름을 써서 붙일 수 있는 포스트잇 게시판(ACL)으로 바꾼 것"과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### ACL의 주요 구성 요소 및 제어 파라미터

| 구성 요소 | 기술적 역할 | 상세 동작 | 비유 |
|:---|:---|:---|:---|
| **Access ACL** | 현재 권한 제어 | 파일/디렉터리에 대한 실시간 접근 가부 결정 | 현재 출입증 확인 |
| **Default ACL** | 권한 상속 관리 | 디렉터리 내에서 새로 생성되는 파일이 물려받을 권한 정의 | 부모의 성씨 물려받기 |
| **ACL Entry** | 개별 규칙 단위 | `Tag:Qualifier:Permissions` 형식으로 구성 (예: `user:bob:rw-`) | 명부의 한 줄 |
| **Effective Rights** | 실제 유효 권한 | **Mask** 비트와 개별 권한의 AND 연산을 통해 최종 결정되는 값 | 최종 승인 도장 |
| **Mask** | 권한 상한선 | 개별 사용자/그룹이 가질 수 있는 최대 권한을 일괄 제한 | 건물 전체 최고 보안 등급 |

---

### 접근 권한 확인 알고리즘 (Permission Check Sequence)

운영체제는 성능과 보안을 위해 엄격한 순서로 ACL을 검사한다.

1. **Owner Check**: 요청자가 파일 소유자이면 소유자 권한 적용 후 종료.
2. **Named User Check**: 요청자 ID와 일치하는 '개별 사용자 ACL'이 있는지 확인. 있으면 적용 후 종료.
3. **Group Check**: 요청자가 속한 그룹들과 '그룹 ACL'들을 대조. (복수 그룹 가능, 권한의 합집합 적용)
4. **Other Check**: 위 조건에 모두 해당하지 않으면 Others 권한 적용.
5. **Mask Filtering**: 모든 과정에서 얻은 권한은 반드시 Mask 값과 비교되어, Mask에서 허용하지 않는 권한은 자동으로 소거됨.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 ACL Mask와 유효 권한(Effective)의 산출              │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [상황] 사용자가 'user:bob:rwx' 권한을 받았으나,                  │
  │          관리자가 시스템 보호를 위해 'mask::rw-'를 설정함.          │
  │                                                                  │
  │   1. bob's ACL :  r  w  x  (1  1  1)                             │
  │   2. System Mask: r  w  -  (1  1  0)                             │
  │   ---------------------------------- (AND 연산)                  │
  │   3. Effective :  r  w  -  (1  1  0)                             │
  │                                                                  │
  │   [결과] bob은 x(실행) 권한을 명시적으로 받았음에도 실행 불가능함.      │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Mask는 개별 권한 설정의 실수를 방지하는 2중 안전장치다. 특정 디렉터리 내의 모든 파일이 절대 실행되지 않게 하려면, 수천 개의 파일 ACL을 고칠 필요 없이 Mask 하나만 'rw-'로 조여주면 된다.

- **📢 섹션 요약 비유**: "아무리 개인 초대권(ACL Entry)이 있어도, 파티장 전체 금지 수칙(Mask)을 어길 수는 없는 법"입니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: ACL vs Capability (자원 중심 vs 주체 중심)

| 비교 항목 | 접근 제어 목록 (ACL) | 능력 리스트 (Capability List) |
|:---|:---|:---|
| **관리 중심** | **객체(Object)** 중심 (파일에 명부 부착) | **주체(Subject)** 중심 (사용자가 티켓 보유) |
| **탐색 방식** | 자원 접근 시 "누가 왔니?" 확인 | 주체가 자원 제시하며 "나 이거 있다!" 증명 |
| **권한 취소** | 쉬움 (파일의 명부에서 한 줄 지우기) | 어려움 (사용자가 가진 티켓을 뺏어야 함) |
| **비유** | 아파트 출입 명부 | 아파트 현관 열쇠 |
| **현대 구현** | POSIX ACL, Windows NTFS | OAuth Token, Kerberos Ticket |

### 비교 2: 임의적 접근 제어(DAC) 내에서의 ACL 위치

- **기본 DAC**: 소유자가 마음대로 `chmod` 하는 수준.
- **ACL 확장**: DAC의 틀 안에서 소유자가 더 정교하게 '공유 대상'을 지정할 수 있는 고도화된 DAC 기법임.
- **강제적 접근 제어(MAC)**: ACL조차 무시하고 커널 보안 정책(SELinux 등)이 강제로 막는 상위 계층 보안.

- **과목 융합 관점 (데이터베이스)**: DB의 `GRANT/REVOKE` 명령어는 SQL 표준 ACL이다. 테이블이라는 객체에 대해 사용자별로 `SELECT`, `UPDATE` 권한을 주는 방식은 운영체제의 파일 ACL 철학과 완벽히 일치한다.

- **📢 섹션 요약 비유**: "누구에게 무엇을 허용할 것인가를 적어둔 '블랙리스트/화이트리스트'의 집합체가 곧 ACL입니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 부서 간 프로젝트 협업**: 개발팀(dev)과 기획팀(plan)이 `/project` 폴더를 공유함. 개발팀은 쓰기 가능, 기획팀은 읽기만 가능해야 함. 또한 외부 감사를 위해 감사관(auditor) 1인에게만 임시 읽기 권한을 부여해야 함. 해결책으로 `setfacl -m g:dev:rwx,g:plan:r-x,u:auditor:r-- /project` 명령을 통해 세밀하게 통제함.

2. **시나리오 — 웹 서비스 정적 파일 보안**: 웹 서버(nginx)가 이미지 파일을 읽어야 하지만, 해킹 사고를 대비해 nginx 실행 계정이 소유자가 되어서는 안 됨. 해결책으로 파일 소유권은 관리자에게 두고, ACL을 통해 `user:nginx:r--` 권한만 추가하여 최소 권한을 유지함.

3. **시나리오 — 자동 권한 상속 문제**: 새로 생기는 로그 파일마다 일일이 권한을 주는 것이 번거로움. 해결책으로 디렉터리에 **Default ACL**을 설정하여, 누가 파일을 만들든 자동으로 `group:log_viewer:r--` 권한이 붙도록 아키텍처를 자동화함.

### 도입 체크리스트
- **파일 시스템 마운트 옵션**: 리눅스 구형 커널의 경우 `/etc/fstab`에 `acl` 옵션이 켜져 있는지 확인.
- **백업 소프트웨어 호환성**: 백업 툴이 파일의 확장 속성(Extended Attributes)인 ACL 정보를 함께 복구할 수 있는가? (지원 안 하면 복구 후 권한 다 깨짐)
- **성능 영향**: 너무 많은 ACL 엔트리(수천 개 이상)는 파일 접근 시 탐색 오버헤드를 유발할 수 있음.

### 안티패턴
- **복잡한 ACL 남발**: 그룹 관리로 충분한데 모든 사용자를 개별 ACL로 등록하는 행위. 관리 주체가 퇴사하거나 이동할 때 권한 관리가 불가능해짐. (해결: 가급적 그룹 ACL 권장)
- **Mask 이해 부족**: 개별 권한은 줬는데 Mask가 막고 있어서 왜 안 되는지 삽질하는 경우. `ls -l` 끝에 `+` 기호가 있다면 반드시 `getfacl`로 Mask를 확인해야 함.

- **📢 섹션 요약 비유**: "도구(ACL)가 정교할수록 관리는 어려워지므로, 정석(그룹 관리)을 지키되 예외 상황에만 정밀 도구(개별 ACL)를 꺼내 쓰는 것이 고수의 비결입니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (ACL 도입의 가치)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **보안 정밀도** | 1인/1그룹 단위의 타겟팅된 권한 제어 가능 | 보안 사고 60% 감소 |
| **비즈니스 유연성** | 조직 개편이나 프로젝트 팀 구성 변경에 즉각 대응 | 관리 리드타임 단축 |
| **컴플라이언스** | '누가' 접근 가능한지에 대한 명확한 근거 제시 | ISMS/ISO27001 인증 필수 |
| **자원 공유** | 불필요한 파일 복사 없이 권한 추가만으로 협업 가능 | 스토리지 절약 |

### 미래 전망
- **지능형 동적 ACL**: 시간, 장소(IP), 접속 기기 상태에 따라 ACL 규칙이 실시간으로 변하는 'Zero Trust' 기반 접근 제어로 진화 중.
- **블록체인 기반 권한 증명**: ACL 장부 자체를 위변조 불가능한 분산 원장에 기록하여, 권한 오남용을 원천 차단하는 신뢰 아키텍처 연구.

### 참고 표준
- **POSIX.1e (Withdrawn but standard)**: 유닉스 ACL의 사실상 표준 명세.
- **NFSv4 ACL**: 네트워크 환경에서의 고도화된 ACL 표준 (Windows ACL과 유사도가 높음).

- **📢 섹션 요약 비유**: ACL은 "열린 문(Open System)"과 "지켜야 할 보물(Data Privacy)" 사이에서 가장 정교한 경계선을 긋는 현대 운영체제의 방어 전략입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **최소 권한 원칙 (Least Privilege)**: ACL을 사용해야 하는 근본적인 보안 철학.
- **VFS 가상 파일 시스템**: 다양한 파일 시스템의 서로 다른 ACL 구현을 하나로 묶어주는 계층.
- **i-node 직접/간접 포인터**: ACL 데이터가 실제로 저장되는 위치(Extended Attributes).
- **DAC vs MAC**: ACL이 속한 접근 제어 모델의 거대 담론.

### 👶 어린이를 위한 3줄 비유 설명
1. ACL은 우리 교실 사물함에 붙은 **"특별 허락 명단"**이에요. 원래는 주인만 열 수 있지만, 선생님이 명단에 "철수도 빌려줄 수 있음"이라고 적어주면 철수도 열 수 있게 돼요.
2. 친구들 이름을 하나하나 다 적을 수 있어서, **내가 친한 친구만 콕 집어서** 내 장난감을 보여줄 수 있는 아주 편리한 규칙이랍니다.
3. 하지만 선생님이 사물함 전체에 **"오늘은 아무도 열지 마!(Mask)"**라고 하시면, 명단에 이름이 있어도 열 수 없는 무서운 규칙도 숨어있답니다!
