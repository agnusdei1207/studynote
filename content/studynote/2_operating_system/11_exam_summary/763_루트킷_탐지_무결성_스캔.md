# 루트킷 탐지 및 시스템 무결성 스캔 (Rootkit Detection & Integrity Scan)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 루트킷(Rootkit)은 운영체제 커널의 실행 경로를 탈취하거나 시스템 장부(파일 목록, 프로세스 리스트 등)를 은닉하여 자신의 존재를 숨기는 지능형 악성코드이며, 무결성 스캔(Integrity Scan)은 시스템의 정적 파일 및 동적 커널 상태가 인가된 원본과 일치하는지 검증하는 보안 기술이다.
> 2. **가치**: 보이지 않는 침입자를 식별하여 훼손된 시스템의 신뢰 체계(Chain of Trust)를 회복하며, APT(지능형 지속 위협) 공격의 Persistence(지속성 확보) 단계에서 핵심적인 탐지 보루 역할을 수행한다.
> 3. **융합**: 하드웨어 기반의 신뢰 루트(Root of Trust)인 **TPM**, 리눅스 커널의 무결성 측정 아키텍처인 **IMA/EVM**, 그리고 실시간 행위 분석을 위한 **eBPF** 기술과 유기적으로 결합되어 현대 엔드포인트 보안(EDR)의 중추가 된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 루트킷은 단순한 바이러스가 아니라 운영체제의 '눈'을 가리는 도구다. 공격자가 시스템에 들어온 뒤, 관리자가 `ls`나 `ps`를 쳤을 때 자신의 파일과 프로세스가 안 보이도록 커널을 속이는 기법이다. 무결성 스캔은 주기적으로 시스템의 '지문(Hash)'을 찍어 원본과 대조함으로써 이 속임수를 찾아낸다.

- **💡 비유**: 이는 은행 강도가 은행의 CCTV 화면을 미리 찍어둔 평화로운 영상으로 바꿔치기(루트킷)하는 것과 같다. 무결성 스캔은 보안 요원이 직접 현장에 가서 실제 금고의 위치와 상태가 장부(정상 해시)와 일치하는지 꼼꼼히 대조해보는 과정이다.

- **등장 배경**:
  1. **탐지 기술과의 군비 경쟁**: 백신이 악성코드를 잘 찾아내자, 아예 운영체제 깊숙이 숨어 백신에게 "아무 문제 없음"이라고 거짓말을 하는 고도화된 기술이 필요해졌다.
  2. **시스템 관리 권한의 오남용**: 'Root' 권한을 획득한 공격자가 시스템 설정을 마음대로 바꾸고 그 흔적을 지우는 행위를 원천적으로 차단하기 위한 장치가 요구되었다.
  3. **공급망 공격 (Supply Chain Attack)**: 펌웨어나 OS 설치 단계부터 변조된 코드가 들어오는 위협에 대응하기 위해 부팅 단계부터의 무결성 검증이 필수적이 되었다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              루트킷의 은닉 메커니즘 및 무결성 스캔 원리             │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ User Request ] ──▶ [ ps / ls command ]                      │
  │                                │                                │
  │   ─────── (Trap to Kernel) ────┼──────────────────────────────  │
  │                                ▼                                │
  │   [ Kernel Layer ]     [ System Call Table ]                    │
  │                        ┌───────────┐                            │
  │                        │ Open  ───▶│───┐ (Rootkit Hooking)      │
  │                        │ Read  ───▶│◀──┘ [ Malicious Code ]     │
  │                        │ Write ───▶│     - Filter Results       │
  │                        └───────────┘     - Hide Files/Procs     │
  │                                                                 │
  │   [ Integrity Scan ] ──▶ Compare(Current Hash, Known-Good Hash) │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 루트킷은 시스템 콜 테이블(System Call Table)의 주소값을 살짝 바꿔서 자기 코드로 먼저 오게 만든다. 자기 코드는 시스템의 실제 정보를 다 읽어온 뒤, 공격자의 정보만 쏙 빼고 유저에게 전달한다. 무결성 스캔은 이 테이블의 주소가 원본과 달라졌는지, 혹은 실행 파일의 바이너리 해시가 바뀌었는지를 감시한다.

- **📢 섹션 요약 비유**: "거짓말을 하는 앵무새(루트킷)를 믿지 않고, 직접 기록부(무결성 DB)를 열어 실제 상황을 대조하는 교차 검증 시스템"입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 루트킷의 침투 및 은닉 계층 분류

어디에 숨느냐에 따라 탐지의 난이도와 위험도가 달라진다.

| 분류 | 침투 위치 | 주요 기술 및 특징 | 탐지 방법 |
|:---|:---|:---|:---|
| **User-mode** | 시스템 라이브러리 (libc 등) | LD_PRELOAD를 통한 API 후킹, 바이너리 교체 | Tripwire, AIDE (파일 스캔) |
| **Kernel-mode** | 커널 영역 (LKM) | **시스템 콜 테이블 변조**, DKOM (데이터 조작) | 커널 메모리 스캔, rkhunter |
| **Hypervisor** | VMM (Type-1/2) | 운영체제 아래에서 하드웨어 트랩 제어 (Blue Pill) | 하드웨어 기반 검증 (TPM) |
| **Firmware** | BIOS / UEFI | 부팅 시작점 탈취, OS 재설치해도 생존 | Secure Boot, Measured Boot |

---

### 무결성 스캔의 2대 축: 정적 vs 동적 검사

1. **정적 무결성 검사 (Static Scan)**:
   - **대상**: 디스크에 저장된 실행 파일, 라이브러리, 설정 파일.
   - **방식**: `SHA-256` 등의 해시 함수로 파일의 지문을 생성하여 깨끗한 상태의 DB와 비교.
   - **도구**: **AIDE**, **Tripwire**, **Samhain**.
2. **동적 무결성 검사 (Runtime Scan)**:
   - **대상**: 현재 메모리에 로드된 커널 코드, 시스템 콜 테이블, 인터럽트 핸들러.
   - **방식**: 메모리 내의 특정 코드 영역이 실행 중 수정되었는지 감시.
   - **기술**: **IMA (Integrity Measurement Architecture)**, **EVM (Extended Verification Module)**.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 Measured Boot과 Chain of Trust 아키텍처            │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ Hardware TPM ] ──▶ [ BIOS/UEFI ] ──▶ [ Bootloader ] ──▶ [ OS ]│
  │          │                  │                  │             │   │
  │          └────(Measure)─────┴────(Measure)─────┴────(Measure)┘   │
  │                                                                  │
  │   [원리] 각 단계는 다음 단계를 실행하기 전 해시를 측정(Measure)하여  │
  │          TPM 내부의 PCR 레지스터에 기록(Extend)함.                │
  │                                                                  │
  │   [결론] 중간에 하나라도 변조되면 최종 부팅 결과값이 달라져 탐지됨    │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** '측정된 부팅(Measured Boot)'은 하드웨어 수준에서 무결성을 보장하는 가장 강력한 방법이다. 칩(TPM)이 부팅 전 과정을 지켜보며 기록하고, 이 기록이 원본과 다르면 부팅을 중단하거나 비밀키를 내주지 않음으로써 루트킷의 시작을 원천 봉쇄한다.

- **📢 섹션 요약 비유**: "문을 열 때마다 지장을 찍게 하고(정적), 건물 안에서 돌아다니는 동안에도 계속 신분증을 확인(동적)하는 철저한 경비 체계"와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 안티바이러스 (AV) vs 무결성 스캔 (Integrity)

| 비교 항목 | 안티바이러스 (AV) | 무결성 스캔 (Integrity) |
|:---|:---|:---|
| **판단 기준** | "이 코드가 나쁜 짓을 하는가?" (Blacklist) | **"이 코드가 원본과 같은가?"** (Whitelist) |
| **대응 범위** | 알려진 악성코드 (Signature) | **알려지지 않은 제로데이 공격**까지 탐지 |
| **오버헤드** | 실시간 감시로 인해 상대적으로 높음 | 정기적 스캔이나 부팅 시에만 수행 (낮음) |
| **한계** | 신종 루트킷 은닉 기술에 취약 | 정상적인 업데이트와 변조를 구분해야 함 |

### 비교 2: 루트킷의 은닉 기술 (Hooking vs DKOM)

- **Hooking**: 시스템 콜 테이블의 포인터를 가로챔. 비교적 흔한 방식이며, 테이블의 주소값만 체크해도 탐지 가능.
- **DKOM (Direct Kernel Object Manipulation)**: 테이블은 건드리지 않고 커널의 내부 데이터(예: 프로세스 연결 리스트)를 직접 수정하여 특정 노드를 건너뛰게 함. 탐지가 매우 어렵고 정교한 커널 메모리 분석이 필요함.

- **과목 융합 관점 (네트워크)**: 루트킷의 네트워크 은닉은 **Promiscuous Mode** 감지나 **Raw Socket** 모니터링을 통해 우회 탐지된다. 시스템에서는 보이지 않는 트래픽이 네트워크 장비(IDS)에서는 보이는 '불일치'를 찾아내는 것이 융합 보안의 핵심이다.

- **📢 섹션 요약 비유**: "나쁜 짓을 하는 놈을 찾는 것보다, 내 얼굴이 조금이라도 변했는지 거울(무결성 DB)을 보고 확인하는 것이 훨씬 확실한 보안책입니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 엔터프라이즈 서버 침해 사고 대응**: 서버 성능이 급격히 저하되고 알 수 없는 네트워크 통신이 발생하지만 `top`이나 `netstat`에는 깨끗함. 기술사적 판단으로 **LKM 기반 루트킷** 감염을 의심하고, `chkrootkit`과 `rkhunter`를 실행함과 동시에 커널 외부에서 메모리 덤프를 떠서 분석(Forensics)을 수행한다.

2. **시나리오 — 안전한 클라우드 부팅 (Trusted Cloud)**: 고객이 빌린 VM이 하이퍼바이저 수준에서 조작되지 않았음을 증명해야 함. 하드웨어 **vTPM**을 활성화하고 **Secure Boot**를 강제하여, 하드웨어부터 운영체제까지의 무결성 체인(Chain of Trust)을 리포트로 제공함으로써 신뢰를 확보한다.

3. **시나리오 — 웹 변조 방지 시스템**: 해커가 웹 소스 코드를 살짝 수정하여 개인정보를 빼돌림. 해결책으로 **AIDE**를 Cron에 등록하여 1시간마다 `/var/www` 이하의 모든 파일 해시를 대조하고, 변경 발생 시 관리자에게 즉시 경보를 울리도록 자동화한다.

### 도입 체크리스트 (Security Governance)
- **Known-Good Base**: 시스템 설치 직후의 깨끗한 해시 정보를 안전한 오프라인 저장소에 보관하고 있는가?
- **Update Management**: 정기 패치 시 무결성 DB를 자동으로 갱신하는 워크플로우가 갖춰져 있는가? (오탐 방지)
- **Privilege Separation**: 무결성 스캔 도구 자체가 루트킷에 의해 오염되지 않도록 도구와 DB를 읽기 전용 매체(CD-ROM 등)나 원격 서버에 두었는가?

### 안티패턴
- **로컬 서버 내에 무결성 DB 보관**: 해커가 루트킷을 깔면서 무결성 DB(원본 해시값)까지 조작해버리면 스캔 도구는 영원히 "정상"만 출력함. (해결: DB의 원격 관리 필수)
- **무분별한 실시간 전체 스캔**: 시스템 부하를 고려하지 않고 전체 디스크를 실시간 해싱하는 행위. 서비스 불능 상태(DoS)를 스스로 유발함.

- **📢 섹션 요약 비유**: "금고 비밀번호를 금고 바로 옆에 적어두는 것(로컬 DB)은 해커에게 금고 문을 열어주는 것과 다름없습니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (무결성 체계 구축 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **탐지 정확도** | 알려지지 않은 변종 루트킷에 대해 99% 탐지 가능 | 지능형 위협 대응 |
| **복구 신뢰성** | 어떤 파일이 변조되었는지 명확히 파악하여 정밀 복구 지원 | 장애 복구 시간(MTTR) 단축 |
| **규제 준수** | ISMS, PCI-DSS 등 주요 보안 인증의 필수 요건 충족 | 법적/사회적 신뢰 확보 |
| **공급망 안전** | 하드웨어-OS-소프트웨어 전 구간의 신뢰 체인 확보 | 인프라 무결성 보장 |

### 미래 전망
- **eBPF 기반 실시간 무결성 모니터링**: 커널 실행 경로를 나노초 단위로 추적하여, 비정상적인 점프나 코드 수정을 즉시 차단하는 자율 방어 커널의 등장.
- **AI-Augmented Scan**: 정상적인 시스템 변경 패턴을 AI가 학습하여, 사람의 개입 없이도 '업데이트'와 '공격'을 구분하여 무결성 DB를 스스로 갱신하는 기술 발전.

### 참고 표준
- **TCG (Trusted Computing Group) Specs**: TPM 및 신뢰 컴퓨팅의 글로벌 표준.
- **NIST SP 800-147**: BIOS 무결성 보호 가이드라인.
- **ISO/IEC 15408 (Common Criteria)**: 보안 제품 무결성 평가 기준.

- **📢 섹션 요약 비유**: 루트킷 탐지는 "진실을 찾는 과정"이며, 무결성 스캔은 그 진실을 증명하는 "가장 정직한 잣대"입니다.
