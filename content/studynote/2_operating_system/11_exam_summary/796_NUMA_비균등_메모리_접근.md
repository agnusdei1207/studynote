# NUMA (Non-Uniform Memory Access) 및 메모리 배치 최적화

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: NUMA (Non-Uniform Memory Access)는 멀티프로세서 시스템에서 CPU 코어와 물리 메모리 사이의 물리적 거리에 따라 데이터 접근 지연 시간(Latency)과 대역폭(Bandwidth)이 상이하게 나타나는 비대칭형 메모리 아키텍처다.
> 2. **가치**: 단일 공유 버스의 병목 현상을 해결하여 수백 개 이상의 코어를 가진 대규모 서버의 확장성(Scalability)을 보장하며, 프로세스가 자신과 가까운 로컬 메모리를 우선 사용하도록 유도하여 시스템 전체의 연산 성능을 극대화한다.
> 3. **융합**: 운영체제의 **CPU 친화성 (Affinity)** 정책, 커널의 **AutoNUMA** 밸런싱 알고리즘, 그리고 차세대 서버 상호 연결 표준인 **CXL (Compute Express Link)** 기술과 결합되어 현대 데이터센터 인프라의 핵심 성능 설계 기준으로 작용한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: "메모리에도 거리가 있다"는 원리다. 모든 CPU가 하나의 램(RAM)을 공유하던 옛날 방식(UMA)과 달리, NUMA는 각 CPU 코어 뭉치마다 전용 램(Local Memory)을 옆에 붙여준다. 내 옆의 램은 아주 빠르지만, 옆 동네 CPU의 램을 읽으려면 네트워크(Interconnect)를 거쳐야 해서 느려지는 구조다.

- **💡 비유**: 이는 "내 방 냉장고"와 "주방 냉장고"의 차이다. 시원한 물(데이터)을 마시고 싶을 때 내 방에 냉장고(Local Node)가 있으면 즉시 마실 수 있다. 하지만 내 방 냉장고가 비어서 주방(Remote Node)까지 가야 한다면 거실 복도(QPI/UPI 버스)를 지나는 만큼 시간이 더 걸리는 것과 같다.

- **등장 배경**:
  1. **버스 경합 (Bus Contention)**: 코어가 많아지자 하나의 메모리 버스에 수십 개의 코어가 몰려 통신 정체가 심각해졌다.
  2. **확장성의 한계**: 물리적으로 램 슬롯을 늘리는 데 한계가 있어, CPU와 램을 한 세트(Node)로 묶어 모듈식으로 늘리는 방식이 효율적이었다.
  3. **고성능 서버 요구**: 테라바이트급 메모리를 사용하는 데이터베이스나 가상화 서버에서 메모리 대역폭을 코어 수만큼 확보해야 했다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              UMA (균등) vs NUMA (비균등) 아키텍처 비교             │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ 1. UMA: Uniform ]              [ 2. NUMA: Non-Uniform ]     │
  │                                                                 │
  │   CPU 0 ──┬── CPU 1                [ Node 0 ]       [ Node 1 ]  │
  │           │                        ┌────────┐       ┌────────┐  │
  │   ────────┴──────── (Bus)          │ CPU 0  │ ◀───▶ │ CPU 1  │  │
  │           │                        │ LocalM │ (UPI) │ LocalM │  │
  │      [ Shared RAM ]                └────────┘       └────────┘  │
  │                                                                 │
  │   - 병목 현상 심함                  - 노드별 독립 대역폭 확보      │
  │   - 소규모 시스템 적합              - 대형 서버/클라우드 표준      │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** NUMA 시스템에서 각 '노드'는 CPU와 그에 직접 연결된 로컬 메모리의 집합이다. 노드 간에는 고속 상호 연결(Intel UPI, AMD Infinity Fabric)로 소통한다. 로컬 접근은 보통 100ns 이하인 반면, 원격 노드 접근은 홉(Hop) 수에 따라 150~300ns 이상으로 지연 시간이 늘어난다.

- **📢 섹션 요약 비유**: "모두가 한 개의 화장실을 공유하며 줄을 서는 것(UMA)보다, 방마다 전용 화장실을 두고(NUMA) 급할 때만 공용을 쓰는 것이 훨씬 쾌적한 생활 방식입니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### NUMA의 성능 결정 3요소

| 요소명 | 기술적 의미 | 상세 메커니즘 | 비유 |
|:---|:---|:---|:---|
| **NUMA Node** | 관리 최소 단위 | 하나의 물리 CPU 소켓과 그에 딸린 메모리 뱅크들의 묶음 | 하나의 독립된 팀 |
| **NUMA Distance** | 접근 비용 지표 | 로컬 대비 원격 메모리 접근이 얼마나 느린지 나타내는 상대적 수치 | 이동에 걸리는 시간 |
| **Locality** | 지역성 원리 | 데이터가 실제 연산을 수행하는 CPU의 로컬 노드에 있을 확률 | 자기 자리 소지품 배치 |

### 운영체제의 메모리 할당 정책 (Memory Policy)

리눅스 커널은 NUMA 효율을 높이기 위해 다양한 `mempolicy`를 제공한다.

1. **Local Alloc (Default)**: 프로세스가 현재 실행 중인 CPU의 노드에서 메모리를 우선 할당함.
2. **Interleave**: 데이터를 모든 노드에 골고루 쪼개서 뿌림. 대규모 스트리밍 등 특정 노드 편중을 막을 때 사용.
3. **Bind**: 특정 노드에서만 메모리를 받도록 강제함. (성능 격리)
4. **Preferred**: 가급적 특정 노드에서 받되, 부족하면 다른 곳으로 가도 됨.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 AutoNUMA의 동적 페이지 이동 (Migration)            │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ T1: Poor Placement ]           [ T2: After AutoNUMA ]        │
  │                                                                  │
  │   CPU 0 (Node 0)                   CPU 0 (Node 0)                │
  │      │ (Remote Access)                │ (Local Access)           │
  │      ▼                                ▼                          │
  │   Memory (Node 1)  ──(Migration)──▶ Memory (Node 0)              │
  │                                                                  │
  │   [가치] 커널이 런타임에 힌트를 읽어 페이지를 가까운 곳으로 옮겨줌    │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 가끔 프로세스가 0번 노드에서 실행되는데 데이터는 1번 노드에 있을 수 있다. 현대 리눅스 커널의 'AutoNUMA' 기능은 이런 불일치를 감지하여, 백그라운드에서 데이터를 CPU가 있는 쪽 노드로 슬그머니 이사(Migration)시켜 성능을 회복시킨다.

- **📢 섹션 요약 비유**: "물건을 둔 곳과 사람이 일하는 곳이 멀어지면, 퇴근 후나 쉬는 시간에 물건을 작업자 옆으로 옮겨주는 배려 깊은 커널 관리자"와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: UMA vs NUMA vs CC-NUMA

| 비교 항목 | UMA (Uniform) | NUMA (Non-Uniform) | CC-NUMA (Cache Coherent) |
|:---|:---|:---|:---|
| **데이터 일관성** | 공유 버스로 해결 | 소프트웨어 처리 필요 가능성 | **하드웨어 수준에서 자동 보장** |
| **확장성** | 낮음 | 매우 높음 | **매우 높음** |
| **대표 기술** | 저가형 PC, 임베디드 | 초기 워크스테이션 | **현대 대형 서버 (Xeon, EPYC)** |
| **특징** | 구조 단순 | 데이터 꼬임 위험 | 복잡하지만 프로그래밍 편함 |

### 비교 2: CPU Affinity와 NUMA의 시너지

- **CPU Affinity**: 프로세스를 특정 코어에 가둠.
- **NUMA Local**: 메모리를 해당 코어 옆에 배치함.
- **결과**: 이 둘이 결합되면 시스템 콜이나 스케줄링 간섭 없이 '초고속 데이터 처리 차선'이 완성된다.

- **과목 융합 관점 (가상화)**: 하이퍼바이저의 **vNUMA** 기술은 호스트의 물리적 NUMA 토폴로지를 게스트 VM에게도 그대로 노출한다. 게스트 OS가 호스트의 하드웨어 구조를 알아야만 최적의 메모리 할당 정책을 펼칠 수 있기 때문이다.

- **📢 섹션 요약 비유**: "지도가 없어도 길을 찾을 수 있게(UMA) 하는 것보다, 정확한 지도(NUMA Topology)를 주고 최적의 경로를 찾게 유도하는 것이 대도시(대형 서버) 운영의 정석입니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 데이터베이스 (SAP HANA, Oracle) 성능 최적화**: 인메모리 DB의 응답 속도가 간헐적으로 늦어짐. `numastat`으로 확인 결과 `remote_node` 접근 비율이 높음. 기술사적 판단으로 **NUMA 고정 기법 (`numactl`)**을 적용한다. DB 인스턴스별로 특정 CPU 소켓과 그에 대응하는 로컬 메모리 노드를 1:1로 바인딩하여 메모리 병목을 40% 이상 개선한다.

2. **시나리오 — 쿠버네티스 고성능 워커 노드 설계**: 통신사 5G NFV 장비에서 패킷 드랍 발생. 해결책으로 **Topology Manager**를 활성화한다. CPU 매니저와 Device 매니저를 연동하여, 네트워크 카드(NIC)가 달린 물리 위치와 동일한 NUMA 노드의 CPU/메모리만 앱에 할당하는 '물리 밀착형 격리'를 수행한다.

3. **시나리오 — 클라우드 비용 절감 (Memory Tiering)**: 램이 부족하여 고가의 고사양 인스턴스를 써야 함. 해결책으로 **CXL 기반의 확장 메모리**를 도입한다. 메인 보드의 램은 아니지만 PCIe를 통해 연결된 '원격 NUMA 노드'처럼 인식시켜, 성능은 조금 낮아도 대용량 메모리를 저렴하게 구축하는 Tiered Memory 전략을 수립한다.

### 도입 체크리스트 (Performance Audit)
- **Node Balance**: 특정 노드의 램은 꽉 차고 다른 노드는 텅 비어있는 '노드 불균형' 상태인가? (OOM 위험 체크)
- **Interconnect Load**: CPU 간 소통하는 UPI/Infinity Fabric 대역폭이 포화 상태인가? (Remote Access 과다 확인)
- **BIOS 설정**: 'Node Interleaving'이 꺼져 있고 NUMA 모드가 활성화되어 있는가? (일부 서버는 기본값이 UMA 에뮬레이션일 수 있음)

### 안티패턴
- **NUMA 무시한 대형 VM 생성**: 2소켓 서버(각 소켓 128GB)에서 200GB짜리 VM 하나를 띄우는 것. VM 내부에서 원격 메모리 접근이 필연적으로 발생하여 성능이 처참해짐. (해결: VM을 노드 크기에 맞춰 쪼개거나 vNUMA 설정)
- **무조건적인 `numactl --interleave`**: 메모리를 섞으면 대역폭은 늘지만 지연 시간(Latency)은 '평균치'로 나빠짐. 짧은 응답이 중요한 작업에는 독이 됨.

- **📢 섹션 요약 비유**: "전체 몸집을 키우는 것보다 각 부위(Node)가 제 기능을 다 하도록 혈류(메모리 대역폭)를 분산시키는 것이 거인(Server)이 오래 달리는 비결입니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (NUMA 최적화 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **메모리 대역폭** | 노드별 병렬 전송으로 총 대역폭 $N$배 향상 | 스케일 아웃 효과 |
| **응답성 (Latency)** | 로컬 메모리 우선 처리로 지연 시간 30% 이상 단축 | 실시간 처리 강화 |
| **가용성** | 특정 메모리 노드 장애 시 해당 노드만 격리 가능 | 장애 전파 차단 |
| **집적도** | 단일 서버 내 수천 개 컨테이너의 독립적 성능 보장 | 클라우드 경제성 확보 |

### 미래 전망
- **CXL (Compute Express Link)**: 램과 SSD, GPU의 경계가 무너지고 전 서버의 자원이 하나의 '거대 가상 NUMA 풀'로 통합되는 **Composable Infrastructure**의 시대 개막.
- **Heterogeneous NUMA**: 일반 DRAM 노드와 고속 HBM 노드, 저속 Persistent Memory 노드가 섞인 환경에서 데이터 수명에 따라 위치를 바꾸는 '지능형 계층화 스토리지' 진화.

### 참고 표준
- **ACPI (Advanced Configuration and Power Interface)**: 하드웨어가 운영체제에 NUMA 구조(SRAT 테이블)를 보고하는 표준.
- **Linux Kernel `libnuma`**: 애플리케이션 개발자를 위한 표준 NUMA 제어 API.

- **📢 섹션 요약 비유**: NUMA는 "물리적 한계를 영리한 배치로 극복하는 지혜"이며, 한 대의 서버가 거대한 우주(클라우드)를 담을 수 있게 해주는 기초 설계 도면입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. NUMA는 컴퓨터 방 안에 **"내 전용 책상과 내 전용 서랍"**이 있는 것과 같아요. 내 옆에 있는 서랍은 아주 빠르게 열 수 있죠.
2. 거실 건너편에 있는 친구 서랍을 열려면 **한참을 걸어가야 해서** 조금 늦게 물건을 가져오게 된답니다.
3. 그래서 똑똑한 컴퓨터는 자주 쓰는 장난감을 **항상 내 옆 서랍(로컬 노드)에** 넣어두고 신나게 노는 거랍니다!
