# 디스크 스케줄링 SCAN (SCAN/Elevator Scheduling)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: SCAN 알고리즘은 디스크 헤드가 실린더의 한쪽 끝에서 반대쪽 끝으로 이동하며 경로상의 모든 요청을 처리하고, 끝단에 도달해야만 방향을 전환하여 역방향 요청을 처리하는 엘리베이터 (Elevator) 방식의 스케줄링 기법이다.
> 2. **가치**: SSTF (Shortest Seek Time First)의 치명적 결함인 기아 현상 (Starvation)을 방지하여 모든 트랙 요청에 대한 응답 시간의 상한을 보장하며, 헤드의 급격한 방향 전환을 억제하여 기계적 마모 감소와 예측 가능한 처리량 (Throughput)을 제공한다.
> 3. **융합**: 고전적인 HDD (Hard Disk Drive)의 탐색 시간 (Seek Time) 최적화뿐만 아니라, 엘리베이터 제어 알고리즘, 물류 자동화 시스템의 피킹 경로 최적화 등 선형 서비스 경로 설계의 표준 모델로 활용된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: SCAN 스케줄링은 현재 헤드의 진행 방향에 있는 요청 중 가장 가까운 것을 순차적으로 처리하며 끝까지 이동하는 방식으로, 마치 엘리베이터가 상승/하강하며 층별 버튼을 처리하는 것과 유사하여 엘리베이터 알고리즘 (Elevator Algorithm)이라고도 불린다.

- **💡 비유**: 이는 아파트 엘리베이터가 1층에서 20층까지 올라가면서, 중간에 버튼이 눌린 모든 층에 서서 사람을 태우고 내리는 것과 같다. 10층에서 버튼을 눌렀는데 엘리베이터가 5층에서 위로 올라오고 있다면, 10층 승객은 곧 탈 수 있지만, 이미 지나친 3층 승객은 엘리베이터가 20층 끝까지 갔다가 다시 내려올 때까지 기다려야 한다.

- **등장 배경**:
  1. **FCFS의 비효율성**: 요청 순서대로만 헤드를 움직이면 (예: 1번 트랙 → 199번 트랙 → 2번 트랙), 헤드가 디스크 전체를 미친 듯이 왕복하여 탐색 시간 (Seek Time)이 극대화된다.
  2. **SSTF의 기아 현상 (Starvation)**: 현재 위치에서 가까운 것만 처리하면, 바깥쪽 트랙 요청은 안쪽 트랙에 요청이 계속 들어오는 한 영원히 서비스받지 못하는 불평등이 발생한다.
  3. **기계적 안정성 요구**: 헤드의 잦은 방향 전환은 액추에이터 암 (Actuator Arm)에 물리적 무리를 주므로, 한 방향으로 끝까지 이동하는 부드러운 제어가 필요해졌다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              SCAN 스케줄링 헤드 이동 경로 (0 ~ 199 트랙)            │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   요청 대기열: 98, 183, 37, 122, 14, 124, 65, 67                │
  │   현재 헤드 위치: 53 , 이전 방향: 0 (안쪽) → 53 (바깥쪽 방향)      │
  │                                                                 │
  │   Track: 0   14   37      53  65  67      98     122 124     183 199│
  │          │    │    │       │   │   │       │       │   │       │  │ │
  │   Start: └─────────────────●                                      │ │
  │   Scan:                    └─▶ 65 ──▶ 67 ──▶ 98 ──▶ 122 ──▶ 124 ──▶ 183 ──▶ 199 │
  │   Reverse: ◀── 37 ◀── 14 ◀───────────────────────────────────────┘ │
  │                                                                 │
  │   [이동 순서]                                                    │
  │   53 → 65 → 67 → 98 → 122 → 124 → 183 → 199(끝) → 37 → 14      │
  │                                                                 │
  │   [특징]                                                        │
  │   - 199번 트랙(끝)을 반드시 찍고 돌아옴                            │
  │   - 37번과 14번은 헤드가 반환할 때까지 기다려야 함                  │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 헤드가 53번에서 증가 방향(바깥쪽)으로 이동을 시작하면, 경로상에 있는 65, 67... 183번까지 모두 처리한다. 중요한 점은 183번 요청을 처리한 후에도 트랙의 물리적 끝인 199번까지 이동한다는 것이다. 이후 방향을 틀어 내려오면서 비로소 37번과 14번 요청을 처리한다.

- **📢 섹션 요약 비유**: "가는 길에 있는 손님은 다 태우고, 종점까지 갔다가 다시 회차하여 반대편 손님을 태우는 노선버스"와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SCAN 알고리즘의 주요 구성 요소

| 요소명 | 역할 | 상세 동작 메커니즘 | 비유 |
|:---|:---|:---|:---|
| **현재 위치 (Current Track)** | 서비스 기준점 | 현재 헤드가 위치한 실린더 번호 | 엘리베이터 현재 층 |
| **진행 방향 (Direction)** | 다음 서비스 결정 | 한 번 결정된 방향은 끝단에 도달할 때까지 유지 | 상행/하행 전광판 |
| **요청 큐 (Request Queue)** | 대기열 관리 | 트랙 번호별로 정렬되어 효율적 스캔 지원 | 층별 호출 버튼 |
| **바운더리 (Boundary)** | 회차 지점 | 트랙 0 또는 최대 트랙(199 등) 도달 시 방향 반전 | 최상층/최하층 |
| **탐색 시간 (Seek Time)** | 성능 지표 | 헤드가 트랙 사이를 이동하는 물리적 시간의 합 | 엘리베이터 이동 시간 |

### SCAN의 파생 및 변형 알고리즘

SCAN의 단점을 보완하기 위해 실무에서는 여러 변형 기법을 사용한다.

1. **C-SCAN (Circular-SCAN)**:
   - **원리**: 한 방향으로만 요청을 처리하고, 끝에 도달하면 반대편 끝으로 '즉시 복귀'하여 다시 같은 방향으로 처리함.
   - **가치**: 트랙의 양 끝단과 중간 부분의 대기 시간 편차를 줄여 서비스 균등성 (Uniformity)을 높임.
2. **LOOK 스케줄링**:
   - **원리**: 진행 방향에 더 이상의 요청이 없으면 끝(199)까지 가지 않고 그 자리에서 즉시 방향을 전환함.
   - **가치**: 불필요한 헤드 이동을 제거하여 응답 시간을 단축함. (SCAN의 실무적 최적화)
3. **C-LOOK**:
   - **원리**: LOOK의 효율성과 C-SCAN의 균등성을 결합. 진행 방향 마지막 요청 처리 후 반대편 가장 먼 요청으로 즉시 복귀.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                    SCAN vs LOOK 비교 (이동 거리 차이)              │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [상황] 마지막 요청이 183번, 최대 트랙이 199번일 때                │
  │                                                                  │
  │   1. SCAN 방식:                                                  │
  │      ... → 183 ──▶ 199 (끝까지 이동) ──▶ 37 (반전)                 │
  │      (불필요한 16트랙 이동 발생)                                    │
  │                                                                  │
  │   2. LOOK 방식:                                                  │
  │      ... → 183 (더 이상 요청 없음) ──▶ 37 (즉시 반전)               │
  │      (16트랙 절약, 더 빠른 응답)                                    │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SCAN은 "규칙(끝까지 가기)"을 중시하고, LOOK은 "상황(요청 유무)"을 중시한다. 실무적으로는 디스크의 끝단에 요청이 없는 경우가 많으므로 LOOK 방식이 훨씬 효율적이며, 대부분의 현대적 스케줄러는 LOOK 기반으로 구현된다.

- **📢 섹션 요약 비유**: "손님이 없어도 꼭 종점까지 가서 유턴하는 버스(SCAN)"와 "마지막 손님을 내린 곳에서 바로 회차하는 택시(LOOK)"의 차이입니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 디스크 스케줄링 알고리즘 종합 비교

| 알고리즘 | 결정 기준 | 장점 | 단점 | 평가 |
|:---|:---|:---|:---|:---|
| **FCFS** | 도착 순서 | 공평성 완벽, 구현 단순 | 호위 효과(Convoy), 낮은 처리량 | 가장 기본 |
| **SSTF** | 최단 거리 | 탐색 시간 최소화 | **기아 현상(Starvation)** 발생 | 성능 위주 |
| **SCAN** | 방향+거리 | **기아 방지**, 높은 처리량 | 양 끝단 대기 시간 불평등 | 안정적 |
| **C-SCAN** | 단방향 스캔 | **대기 시간의 균등성** 최고 | 이동 거리 자체는 SCAN보다 김 | 공정성 위주 |
| **LOOK** | 방향+요청 유무 | SCAN보다 이동 거리 단축 | 방향 판단 로직 추가 필요 | 실무적 최적 |

### 비교 2: 서비스 균등성 (Service Uniformity) 분석

- **SCAN의 불평등**: 헤드가 가운데를 지날 때 요청을 처리하면, 다음번에 가운데를 지날 확률이 양 끝단보다 높다. 따라서 가운데 트랙의 평균 대기 시간이 끝단 트랙보다 짧은 "중간 편중 현상"이 발생한다.
- **C-SCAN의 해결**: 한 방향으로만 순환하므로, 어떤 트랙이든 헤드가 한 바퀴 돌고 돌아오는 시간이 일정해진다. 대기 시간의 표준 편차가 낮아져 예측 가능한 응답 속도를 제공한다.

- **과목 융합 관점 (네트워크)**: 네트워크의 토큰 패싱 (Token Passing) 방식이나 라운드 로빈 (Round Robin) 스케줄링이 C-SCAN의 철학과 맞닿아 있다. 모든 노드(트랙)에게 공평한 전송 기회를 부여하려는 설계 의도가 동일하기 때문이다.

- **📢 섹션 요약 비유**: "누가 제일 급한가(SSTF)"만 따지면 소외되는 사람이 생기므로, "순서대로 다 훑고 지나가기(SCAN)"를 통해 최소한의 생존권을 보장하는 사회 복지와 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 서버 HDD 스토리지 부하**: 데이터베이스 로그 파일 기록과 백업 작업이 동시에 발생하여 디스크 헤드가 요동치는 상황. 이때 FCFS를 쓰면 I/O 큐가 폭증한다. 리눅스의 **CFQ (Completely Fair Queuing)**나 **Deadline** 스케줄러는 내부적으로 SCAN/LOOK의 원리를 적용하여 헤드 이동을 정렬(Merging)함으로써 I/O 대기 시간을 제어한다.

2. **시나리오 — 비디오 스트리밍 서버**: 여러 사용자가 디스크의 서로 다른 위치에 있는 영상을 요청할 때, 응답 시간의 급격한 변화(Jitter)는 영상 끊김을 유발한다. C-SCAN의 균등한 응답 속도 특성을 활용하여 지터(Jitter)를 최소화하는 멀티미디어 전용 스케줄러를 설계한다.

3. **시나리오 — SSD 환경에서의 무용론**: SSD (Solid State Drive)는 물리적으로 이동하는 헤드가 없으므로 탐색 시간 (Seek Time)이 거의 0에 가깝다. 따라서 SCAN 같은 복잡한 스케줄링은 오히려 CPU 오버헤드만 발생시킨다. 현대 OS는 SSD 감지 시 스케줄러를 **NOOP (No-Operation)** 또는 **None**으로 설정하여 요청이 들어온 순서대로 처리하게 한다.

### 도입 체크리스트 (스토리지 설계 시)
- **미디어 타입**: 물리적 헤드가 있는 HDD인가, 반도체 기반의 SSD인가?
- **부하 특성**: 무작위 읽기(Random Read)가 많은가, 순차적 기록(Sequential Write)이 많은가?
- **지연 한계 (Deadline)**: 특정 요청이 일정 시간 내에 반드시 처리되어야 하는 실시간성 요구가 있는가?

### 안티패턴
- **SSD에 SCAN/C-SCAN 적용**: 헤드 이동을 고려한 정렬 로직이 SSD의 내부 FTL (Flash Translation Layer) 알고리즘과 충돌하여 오히려 쓰기 수명을 단축시키고 성능을 저하시킴.
- **실시간 시스템에 무조건적 SSTF 적용**: 처리량은 좋으나, 중요한 시스템 제어 요청이 기아 상태에 빠져 전체 시스템이 다운될 수 있음. (해결: SCAN 또는 Deadline 스케줄러 사용)

- **📢 섹션 요약 비유**: "발이 빠른 사람(SSD)에게 보폭을 맞춰 걷는 법(SCAN)을 가르치는 것은 시간 낭비"인 것처럼, 매체의 물리적 특성에 맞는 규칙 적용이 필요합니다.

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (SSTF 대비)

| 구분 | SSTF 방식 | SCAN 방식 | 개선 효과 |
|:---|:---|:---|:---|
| **기아 현상 발생율** | 높음 (외곽 트랙 소외) | **0% (완전 제거)** | **서비스 신뢰도 극대화** |
| **평균 응답 시간** | 가장 짧음 (최적 상태 시) | SSTF보다 약간 길지만 안정적 | **예측 가능한 응답성 확보** |
| **물리적 내구성** | 헤드의 잦은 반전으로 마모 발생 | 부드러운 왕복 운동 | **HDD 기대 수명 연장** |

### 미래 전망
- **컴퓨테이셔널 스토리지 (Computational Storage)**: 디스크 컨트롤러 자체에서 I/O 요청을 지능적으로 스캔하고 처리하는 기능이 강화되어, OS 커널의 부담이 줄어들 것임.
- **HDD의 대용량화와 헬륨 충전**: 트랙 밀도가 높아짐에 따라 아주 미세한 헤드 제어가 필요해지며, SCAN의 정밀도와 LOOK의 효율성을 결합한 더욱 정교한 펌웨어 알고리즘이 발전 중임.

### 참고 표준
- **POSIX `fadvise(POSIX_FADV_SEQUENTIAL)`**: 파일 접근 패턴을 OS에 알려주어 스케줄러가 스캔 최적화를 할 수 있게 돕는 표준.
- **S.M.A.R.T. (Self-Monitoring, Analysis and Reporting Technology)**: 디스크 스케줄링 효율 및 헤드 건강 상태를 모니터링하는 표준.

- **📢 섹션 요약 비유**: SCAN은 물리적 한계(헤드 이동) 속에서도 공평과 효율이라는 두 마리 토끼를 잡기 위해 탄생한 "운영체제의 중용(中庸)의 도"입니다.

- **C-SCAN 단방향 회전**: SCAN의 응답 시간 불균형을 해결한 순환 스캔 방식.
- **SSTF 기아 현상**: SCAN이 해결하고자 한 근본적인 문제점.
- **탐색 시간 (Seek Time)**: 디스크 스케줄링이 줄이고자 하는 가장 핵심적인 물리 지연.
- **I/O 스케줄러 (NOOP, Deadline, CFQ)**: SCAN 원리가 실제 구현된 운영체제의 모듈들.

### 👶 어린이를 위한 3줄 비유 설명
1. SCAN은 **"부지런한 엘리베이터"**와 같아요. 1층부터 꼭대기 층까지 쭉 올라가면서 타려는 사람을 다 태우고, 다시 내려오면서 또 태워주는 거예요.
2. 중간에 급한 사람이 있다고 해서 갑자기 방향을 바꾸지 않고, 가던 길을 마저 간 다음에 돌아오기 때문에 아무리 멀리 있는 사람도 **결국에는 차례가 온답니다.**
3. 이 방식 덕분에 디스크 안의 "헤드"라는 부품이 이리저리 방황하지 않고 **차분하게 순서대로** 일을 할 수 있게 된답니다!
