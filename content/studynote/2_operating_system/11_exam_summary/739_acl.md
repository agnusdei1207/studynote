## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ACL(Access Control List)은 접근 제어 행렬(Access Matrix)의 **객체 중심 구현**으로서, 각 객체(파일, 디렉터리, 네트워크 자원)가 **"나를 접근 가능한 주체 리스트와 그 권한"**을 자신의 메타데이터에 보관하는 방식이다.
> 2. **가치**: UNIX의 9비트 rwx 권한(Owner/Group/Other)은 너무 조잡하여 "Staff 그룹의 Alice는 쓰기, Bob은 읽기만" 같은 세분화된 제어가 불가능하다. ACL은 개별 사용자/그룹 단위로 권한을 부여하여, **"Staff 그룹 내에서도 개별적으로 권한 차등"**을 가능하게 한다.
> 3. **융합**: NFSv4 ACL, Windows NTFS ACL, AWS S3 Bucket Policy, Kubernetes NetworkPolicy는 모두 ACL의 변형으로서, 현대 파일시스템과 클라우드 보안의 기반을 이룬다.

## Ⅰ. 개요 (Context & Background)

- **개념**: ACL(Access Control List)은 각 객체(Object)에 연결된 **"접근 허가 명부(Guest List)"**로서, **"어떤 주체(Subject)가 어떤 연산(Operation)을 수행할 수 있는가"**를 ACE(Access Control Entry)의 리스트 형태로 명시한다. 수학적으로는 객체 O에 대해 ACL(O) = {(S₁, p₁), (S₂, p₂), ...} 형태의 쌍(Pair) 집합으로 표현된다. 여기서 S는 주체(UID/GID), p는 권한(R/W/X/...)이다.

- **💡 비유**: ACL은 **"VIP 라운지 입구의 게스트 명부"**와 같다. 입구에는 "Alice: 입장 가능, Bob: 입장 가능, Charlie: 입장 금지"라고 적힌 명부가 붙어 있고, 경비원은 이 명부를 보고 "Alice는 들어가세요, Charlie는 못 들어가요"라고 판단한다. 명부는 라운지(객체) 측에서 관리하므로, 새로운 손님을 추가하려면 라운지 관리자가 명부에 이름을 적어야 한다.

- **등장 배경**:
  1. **MULTICS 시스템(1960s-70s)**: MULTICS는 최초로 ACL을 구현하여, 각 파일 세그먼트에 접근 제어 리스트를 연결했다. 이는 UNIX의 단순한 rwx 모델보다 훨씬 세밀한 제어를 가능하게 했다.
  2. **UNIX POSIX ACL(1990s-2000s)**: 리눅스는 POSIX 1003.1e 표준을 구현하여, 기존의 9비트 rwx 모델을 확장한 ACL을 `setfacl`, `getfacl` 명령으로 제공한다. 이는 "Owner/Group/Other" 3개 범주의 한계를 넘어서, **"특정 사용자 alice만 쓰기 허용"** 같은 세밀한 제어를 가능하게 한다.
  3. **네트워크 장비 ACL(1980s-현재)**: 라우터, 방화벽의 ACL은 **"IP 주소/포트별 접근 제어"**를 정의한다. 예를 들어 `permit tcp any host 10.0.0.1 eq 22`는 "모든 IP에서 10.0.0.1의 22번 포트로의 TCP 접속을 허가"한다.

ACL과 기존 UNIX 권한의 대조를 시각화하면 다음과 같다. UNIX 권한은 9비트로 고정되어 있지만, ACL은 무한히 확장 가능하다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │            UNIX rwx vs ACL: 권한 표현 비교                           │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [UNIX Permission — 3×3 행렬의 압축]                                 │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  $ ls -l file.txt                                             │  │
 │  │  -rw-rwxr--  1 alice  staff  1024 Mar 10 12:00 file.txt       │  │
 │  │                                                                │  │
 │  │   └─┬─┘ └─┬─┘ └─┬─┘                                            │  │
 │  │     │     │     │                                             │  │
 │  │   Owner Group Other                                           │  │
 │  │   (R,W) (R,W,X) (---)                                          │  │
 │  │                                                                │  │
 │  │  • Owner(alice): 읽기, 쓰기                                   │  │
 │  │  • Group(staff): 읽기, 쓰기, 실행                             │  │
 │  │  • Other: 아무 권한 없음                                       │  │
 │  │                                                                │  │
 │  │  ⚠️ 문제: "Staff 그룹의 Bob은 읽기만, Alice는 쓰기도"        │  │
 │  │           같은 세분화 불가능!                                 │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [POSIX ACL — 무한 확장 가능]                                        │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  $ getfacl file.txt                                           │  │
 │  │  # file: file.txt                                            │  │
 │  │  # owner: alice                                               │  │
 │  │  # group: staff                                               │  │
 │  │  user::rw-                     ← Owner(alice)의 기본 권한    │  │
 │  │  user:bob:r--                  ← Bob은 읽기만!               │  │
 │  │  user:charlie:---               ← Charlie는 금지              │  │
 │  │  group::rwx                    ← Group(staff)의 기본 권한    │  │
 │  │  group:devteam:r-x              ← devteam 그룹은 읽기+실행    │  │
 │  │  mask::rwx                     ← 최대 허용 권한 마스크         │  │
 │  │  other::---                    ← Other의 기본 권한            │  │
 │  │                                                                │  │
 │  │  ✅ "Staff 그룹 내에서도 Bob과 Charlie의 권한을 다르게!"    │  │
 │  │     "devteam 그룹은 읽기+실행만 허용!"                        │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** UNIX permission은 **"3개의 구획(Owner/Group/Other)"**으로 고정되어 있어서, 같은 그룹 내에서도 개별 사용자의 권한을 차등 부여할 수 없다. ACL은 이 제약을 제거하여, **"user:bob:r--"** 같은 개별 사용자별 ACE(Access Control Entry)를 추가하여 세분화된 제어를 가능하게 한다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **ACE (Access Control Entry)** | ACL을 구성하는 개별 항목 | {주체, 권한} 쌍 | POSIX ACE, Windows ACE | 명부의 한 줄 |
| **ACL 헤더** | ACL의 메타데이터(버전, 개수) | ACL 전체의 형식 정보 | NFSv4 ACL 헤더 | 명부의 표지 |
| **기본 ACE** | Owner/Group/Other의 기본 권한 | UNIX rwx와 호환 | `user::`, `group::`, `other::` | 기본 회원권 |
| **확장 ACE** | 추가 사용자/그룹별 권한 | `user:bob:r--` 형식 | POSIX ACL, NFSv4 ACL | 특별 초대장 |
| **마스크(Mask)** | 그룹 ACL의 최대 권한 제한 | `mask::rwx`가 실제 유효 권한 | POSIX ACL mask | 상한선 |
| **상속(Inheritance) 플래그** | 부모→자식으로 ACL 전파 | 디렉터리의 파일에 자동 적용 | NFSv4, NTFS | 자녀에게 물려주기 |

### POSIX ACL 구조와 검사 알고리즘

리눅스의 POSIX ACL은 **{Tag Type, Qualifier, Permissions}** 트리플로 ACE를 정의한다. Tag Type은 **"이 ACE가 누구를 위한 것인가"**를 나타내며, Qualifier는 실제 UID/GID 값이다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │                  POSIX ACL의 구조와 검사 알고리즘                    │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [ACE 구조 (Access Control Entry)]                                   │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  ┌───────────┬──────────────┬──────────────┐                  │  │
 │  │  │ Tag Type │  Qualifier   │ Permissions  │                  │  │
 │  │  └───────────┴──────────────┴──────────────┘                  │  │
 │  │                                                                │  │
 │  │  Tag Type 종류:                                               │  │
 │  │  ┌──────────────────────────────────────────────────────────┐ │  │
 │  │  │  ACL_USER_OBJ (0x01)  → Owner(alice)의 기본 권한         │ │  │
 │  │  │  ACL_USER      (0x02)  → user:bob (Qualifier=UID)        │ │  │
 │  │  │  ACL_GROUP_OBJ (0x04)  → Group(staff)의 기본 권한        │ │  │
 │  │  │  ACL_GROUP     (0x08)  → group:devteam (GID)            │ │  │
 │  │  │  ACL_MASK      (0x10)  → 그룹 권한의 상한선              │  │
 │  │  │  ACL_OTHER     (0x20)  → Other의 기본 권한              │  │
 │  │  └──────────────────────────────────────────────────────────┘ │  │
 │  │                                                                │  │
 │  │  Permissions: rwx (Read=4, Write=2, Execute=1)                │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [ACL 검사 알고리즘 — check_permission()]                            │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  입력: current->uid, current->gid, requested_perm             │  │
 │  │                                                                │  │
 │  │  ① Owner 확인                                                 │  │
 │  │     if (uid == file->uid) {                                   │  │
 │  │         perm = ACL_USER_OBJ.permissions;                      │  │
 │  │         return (requested_perm ⊆ perm) ? ALLOW : DENY;        │  │
 │  │     }                                                         │  │
 │  │                                                                │  │
 │  │  ② 사용자별 ACE 확인                                           │  │
 │  │     for each ACE in ACL where Tag == ACL_USER {               │  │
 │  │         if (ACE.Qualifier == uid) {                           │  │
 │  │             perm = ACE.permissions & ACL_MASK.permissions;    │  │
 │  │             return (requested_perm ⊆ perm) ? ALLOW : DENY;    │  │
 │  │         }                                                     │  │
 │  │     }                                                         │  │
 │  │                                                                │  │
 │  │  ③ 그룹 확인 (사용자가 속한 모든 그룹)                       │  │
 │  │     for each g in current->groups {                           │  │
 │  │         if (g == file->gid) {                                 │  │
 │  │             perm = ACL_GROUP_OBJ.permissions;                 │  │
 │  │         }                                                     │  │
 │  │         for each ACE in ACL where Tag == ACL_GROUP {          │  │
 │  │             if (ACE.Qualifier == g) {                         │  │
 │  │                 perm = ACE.permissions;                       │  │
 │  │             }                                                 │  │
 │  │         }                                                     │  │
 │  │         perm = perm & ACL_MASK.permissions;  ← 마스크 적용    │  │
 │  │         if (requested_perm ⊆ perm) return ALLOW;              │  │
 │  │     }                                                         │  │
 │  │                                                                │  │
 │  │  ④ Other 확인                                                 │  │
 │  │     perm = ACL_OTHER.permissions;                             │  │
 │  │     return (requested_perm ⊆ perm) ? ALLOW : DENY;            │  │
 │  │                                                                │  │
 │  │  ✅ ACL은 위에서 아래로 순차 검색하며, 매칭되는 즉시 반환      │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  예: alice(UID=1000, GID=100)이 file.txt(W 권한) 요청              │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  file.txt의 ACL:                                              │  │
 │  │    user::rw-          (alice가 Owner라면 매칭! W 없음)        │  │
 │  │    user:bob:r--       (bob 전용, alice 무관)                  │  │
 │  │    group::rwx         (alice가 staff(100) 소속이라면 매칭!)    │  │
 │  │    mask::rwx          (W 허용됨)                              │  │
 │  │    other::---                                                 │  │
 │  │                                                                │  │
 │  │  결과: alice가 Owner이지만 ACL_USER_OBJ에 W 없음 → DENY      │  │
 │  │       (ACL은 UNIX permission보다 우선!)                       │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** POSIX ACL 검사는 **"Owner → Named User → Group → Other"** 순서로 진행된다. 이때 **마스크(Mask)**가 중요한 역할을 한다. 마스크는 **"그룹 관련 ACE의 최대 허용 권한"**으로서, `group::rwx`라도 `mask::r--`이면 실제 유효 권한은 `r--`이다. 이는 의도치 않은 권한 확대를 방지하는 안전장치다. 예를 들어 관리자가 `chmod g+w file.txt`로 그룹에 쓰기 권한을 추가하면, 커널은 자동으로 `mask::rw+`로 확장한다.

### 네트워크 장비 ACL — 라우터/방화벽

네트워크 장비에서 ACL은 **"패킷 필터링 규칙"**으로 사용된다. Cisco, Juniper 라우터는 IP 주소, 포트, 프로토콜별로 접속을 허가/차단한다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │              네트워크 장비 ACL — 패킷 필터링                         │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [Cisco ACL 예시]                                                    │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  ! SSH 관리자(10.0.0.100)만 접속 허용                          │  │
 │  │  access-list 100 permit tcp host 10.0.0.100 any eq 22         │  │
 │  │  ! 내부 네트워크(192.168.0.0/16)에서의 모든 접속 허용          │  │
 │  │  access-list 100 permit ip 192.168.0.0 0.0.255.255 any        │  │
 │  │  ! 기타 모든 접속 차단                                         │  │
 │  │  access-list 100 deny ip any any                              │  │
 │  │                                                                │  │
 │  │  interface GigabitEthernet0/0                                  │  │
 │  │    ip access-group 100 in  ← 인바운드 패킷에 ACL 적용        │  │
 │  │                                                                │  │
 │  │  ✅ ACL은 "위에서 아래로" 순차 평가하며, 매칭되면 즉시 반환    │  │
 │  │     (첫 번째 permit/deny에 일치하는 규칙만 적용)               │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [ACL 평가 흐름]                                                     │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   패킷 도착: src=192.168.1.50, dst=10.0.0.5, dport=22         │  │
 │  │                                                                │  │
 │  │   ① Rule 1: permit tcp host 10.0.0.100 any eq 22            │  │
 │  │      192.168.1.50 ≠ 10.0.0.100 → 불일치, 다음 규칙으로       │  │
 │  │                                                                │  │
 │  │   ② Rule 2: permit ip 192.168.0.0 0.0.255.255 any           │  │
 │  │      192.168.1.50 ⊆ 192.168.0.0/16 → 일치!                 │  │
 │  │      → PERMIT, 패킷 통과 (더 이상 규칙 검사 안 함)            │  │
 │  │                                                                │  │
 │  │   ③ Rule 3: 도달하지 않음                                     │  │
 │  │                                                                │  │
 │  │  ⚠️ 규칙 순서가 중요! "deny ip any any"를 맨 위에 쓰면        │  │
 │  │      모든 패킷이 차단됨                                        │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [확장 ACL — Stateful Inspection]                                  │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  ! Established 연결에 대한 응답 패킷 자동 허용               │  │
 │  │  access-list 100 permit tcp any any established              │  │
 │  │                                                                │  │
 │  │  • 내부→외부 연결 시스템이 SYN 패킷을 보내면                 │  │
 │  │    방화벽은 연결 상태를 추적(State Table에 저장)             │  │
 │  │  • 외부→내부의 SYN-ACK, ACK, FIN 등은 "established"로        │  │
 │  │    자동 허용                                                  │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 네트워크 ACL은 **"상태 비저장(Stateless)"** 필터링이다. 각 패킷은 독립적으로 규칙에 맞는지 검사되며, 이전 패킷의 기록은 참조하지 않는다. 그러나 **`established` 키워드**는 예외적으로 연결 추적(State Tracking)을 수행하여, 이미 확립된 TCP 연결의 응답 패킷을 자동으로 허용한다. 현대 방화벽은 이를 확장하여 완전한 Stateful Packet Inspection(SPI)을 수행한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: POSIX ACL vs NFSv4 ACL vs NTFS ACL

| 비교 항목 | POSIX ACL | NFSv4 ACL | NTFS ACL |
|:---|:---|:---|:---|
| **ACE 순서 의존성** | 없음 (명시적 Tag) | 있음 (Deny가 우선) | 있음 (Deny가 우선) |
| **상속(Inheritance)** | `default:` ACL로 디렉터리만 | 파일/디렉터리별 플래그 | OBJECT_INHERIT, CONTAINER_INHERIT |
| **ACE 유형** | Allow만 | Allow + Deny | Allow + Deny |
| **권한 종류** | rwx만 | rwx + delete, readattr 등 20+ | GENERIC_READ, GENERIC_WRITE 등 |
| **권한 부여 대상** | UID/GID | principal@realm, group@realm | SID (Security Identifier) |
| **표준화** | POSIX 1003.1e (Draft) | RFC 5661 (NFSv4) | Microsoft 독자 |
| **구현 OS** | 리눅스, FreeBSD | NAS, NetApp | Windows |

POSIX ACL은 **"Deny 명시 불가"**라는 단순함이 있지만, NFSv4 ACL과 NTFS ACL은 **"Allow + Deny"**를 모두 지원한다. Deny가 Allow보다 우선하므로 규칙 순서가 매우 중요하다. 예를 들어 NTFS ACL에서 "Allow: Everyone Full Control" 다음에 "Deny: Bob Write"가 있으면, Bob은 모든 권한 중 쓰기만 차단된다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │         POSIX ACL vs NFSv4 ACL: 구조 차이 비교                       │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [POSIX ACL]                                                         │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  # file: project/data.txt                                    │  │
 │  │  user::rw-                                                    │  │
 │  │  user:bob:r--                                                 │  │
 │  │  user:alice:rw-                                               │  │
 │  │  group::rwx                                                   │  │
 │  │  group:devteam:r-x                                            │  │
 │  │  mask::rwx                                                    │  │
 │  │  other::---                                                   │  │
 │  │                                                                │  │
 │  │  특징:                                                         │  │
 │  │  • Allow만 존재 (Deny 없음)                                   │  │
 │  │  • 순서 무관 (Tag Type으로 식별)                              │  │
 │  │  • rwx 3비트만 지원                                            │  │
 │  │  • UID/GID 기반                                               │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [NFSv4 ACL]                                                         │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  A::OWNER@DOMAIN:ALLOWED:rwattnyCY/                           │  │
 │  │  A:g:GROUP@DOMAIN:ALLOWED:rxtncy/                            │  │
 │  │  D: bob@DOMAIN:ALLOWED:wo/                    ← Deny Bob Write│  │
 │  │  A:g:devteam@DOMAIN:ALLOWED:rx/                              │  │
 │  │  A@:EVERYONE@DOMAIN:ALLOWED:r/                                │  │
 │  │                                                                │  │
 │  │  특징:                                                         │  │
 │  │  • A(Allow) + D(Deny)                                         │  │
 │  │  • 순서 중요 (Deny가 우선 적용)                               │  │
 │  │  • rwattnyCy 등 14개 권한 비트                               │  │
 │  │  • principal@realm 기반 (Kerberos)                            │  │
 │  │  • 상속 플래그 (f, d, fi, etc.)                               │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [NTFS ACL (Security Descriptor)]                                   │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │  DACL:                                                        │  │
 │  │    → ACE: Allow SID:S-1-5-21-...-1001 (Alice) GENERIC_ALL    │  │
 │  │    → ACE: Allow SID:S-1-5-21-...-1003 (Staff) GENERIC_READ   │  │
 │  │    → ACE: Deny  SID:S-1-5-21-...-1002 (Bob) GENERIC_WRITE    │  │
 │  │                                                                │  │
 │  │  특징:                                                         │  │
 │  │  • SID (Security Identifier) 기반                            │  │
 │  │  • GENERIC_ALL, GENERIC_READ 등 표준 권한                    │  │
 │  │  • 상속 플래그 (OBJECT_INHERIT, CONTAINER_INHERIT)            │  │
 │  │  • SACL (System ACL)로 감사(Auditing)도 지원                  │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** POSIX ACL은 단순하지만 "Bob만 쓰기 금지" 같은 부정적 제어가 불가능하다. NFSv4/NTFS ACL은 Deny를 지원하여 **"모두 허용 + Bob만 제외"** 같은 정책을 표현할 수 있다. 그러나 Deny 규칙의 순서가 잘못되면 의도치 않은 권한 부여가 발생할 수 있어 관리가 복잡하다.

### 비교 2: 파일시스템 ACL vs 네트워크 ACL

| 비교 항목 | 파일시스템 ACL | 네트워크 ACL |
|:---|:---|:---|
| **제어 대상** | 파일, 디렉터리, IPC 객체 | 패킷(IP/Port/Protocol) |
| **검사 위치** | VFS 계층, LSM 훅 | NIC 드라이버, 방화벽 |
| **검사 빈도** | open(), read(), write() 시 | 모든 패킷 수신/송신 시 |
| **성능 영향** | 크지 않음 (캐싱 가능) | 매우 큼 (와이어 스피드에 영향) |
| **상태 유지** | Stateless (요청마다 검사) | Stateful (연결 추적) |
| **대표 예시** | POSIX ACL, NTFS ACL | Cisco ACL, iptables, nftables |
| **구현 계층** | OS 커널 | L3/L4 네트워크 계층 |

파일시스템 ACL은 **"시스템 콜 단위"**로 검사되며, 커널의 VFS(Virtual File System) 계층에서 이루어진다. 한 번 열린 파일 디스크립터는 ACL 재검사 없이 반복 사용할 수 있어 성능 영향이 적다. 반면 네트워크 ACL은 **"모든 패킷마다"** 검사되어, CPU 집약적이며 와이어 스피드(예: 10Gbps)에 근접한 처리를 위해서는 하드웨어 가속(TCAM, FPGA)이 필요하다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 다중 사용자 프로젝트 디렉터리**: 개발팀의 `/data/project` 디렉터리에서 "alice는 모든 권한, bob은 읽기만, charlie는 금지"라는 요구가 있었다. `chmod 750`로는 Staff 그룹 내에서 권한 차등을 둘 수 없으므로, POSIX ACL을 적용했다:
   ```bash
   setfacl -m u:alice:rwxA,u:bob:rX,u:charlie:- /data/project
   setfacl -d -m u:alice:rwxA,u:bob:rX,u:charlie:- /data/project  # default ACL
   ```
   `-d` 옵션으로 기본(Default) ACL을 설정하면, 이후 생성되는 파일에도 자동으로 상속된다.

2. **시나리오 — NFS 공유와 ACL 변환**: 리눅스 서버의 `/data` 디렉터리를 NFSv4로 공유하려 했는데, POSIX ACL과 NFSv4 ACL의 미묘한 차이로 권한 누수가 발생했다. 해결책은 `nfs4_getfacl` / `nfs4_setfacl`을 사용하여 NFSv4 형식의 ACL을 직접 설정하고, `nfs4_edit` 유틸리티로 POSIX → NFSv4 변환을 검증하는 것이다. 특히 NFSv4는 `mode` bits를 ACL로 통합하므로, `chmod`와 `nfs4_setfacl`을 혼용하면 충돌이 발생할 수 있다.

3. **시나리오 — 방화벽 ACL 최적화**: 대규모 데이터센터의 방화벽에 10,000개 이상의 ACL 규칙이 있어, 패킷 처리 지연이 발생했다. 해결책은:
   ① 자주 일치하는 규칙을 상단으로 이동
   ② 중복/모순 규칙 제거
   ③ 객체 그룹(네트워크 오브젝트 그룹, 서비스 그룹)으로 규칙 통합
   ④ 하드웨어 가속(Cisco ACE, TCAM) 활성화

### 도입 체크리스트
- **기술적**: 파일시스템이 ACL을 지원하는가(`getfacl` 명령 확인)? ACL이 마운트 옵션으로 활성화되어 있는가(`mount -o acl` 또는 현대 파일시스템은 기본 지원)?
- **운영·보안적**: 불필요하게 널린 권한(ACL 엔트리 수십 개)은 없는가? 정기적으로(분기 1회) `find / -perm -0002` 명령으로 "Everyone 쓰기" 파일을 검색하는가?

### 안티패턴
- **ACL 폭주**: 특정 파일에 수백 개의 ACE를 추가하면, 접근 시마다 전체 ACL을 순회해야 하여 성능이 저하된다. 이 경우에는 그룹을 활용하여 ACE 수를 줄여야 한다.
- **Deny 규칙 남용**: NFSv4/NTFS ACL에서 "Deny Bob" 대신 "Allow Except Bob"으로 표현할 수 있다면, Deny를 피하는 것이 좋다. Deny는 Allow보다 우선하므로, 예상치 못한 차단이 발생할 수 있어 디버깅이 어렵다.

### 정량/정성 기대효과

| 구분 | 최적화 전 | 최적화 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | `chmod 777`로 과도하게 열림 | ACL로 최소 권한 부여 | **데이터 유출 경로 95%+ 차단** |
| **정량** | 방화벽 ACL 10,000개 | 그룹화/최적화로 1,000개 축소 | **패킷 처리 지연 70%+ 개선** |
| **정성** | 그룹 내 개별 권한 차등 불가 | ACE로 세분화된 제어 | **규정 준거(Compliance) 달성** |

### 미래 전망
- **Rich ACL (Linux 5.x)**: 리눅스 커널 4.x 이후부터 NFSv4와 유사한 Rich ACL이 실험적으로 지원되며, "delete, readattr, writeattr" 등 세밀한 권한 비트를 제공한다.
- **RBAC와의 통합**: Kubernetes의 NetworkPolicy, AWS NACL 같은 최근 시스템은 ACL의 세밀한 제어와 RBAC의 역할 기반 관리를 통합하여, **"역할별 ACL 템플릿"**을 적용하는 추세다.

## 👶 어린이를 위한 3줄 비유 설명
1. ACL은 **"교실 입구에 붙은 출석부"**예요. "철수 ✅입장, 영희 ✅입장, 민수 🚫금지"라고 적혀 있어요.
2. 선생님은 이 명부를 보고 "철수는 들어가고, 민수는 못 들어가"라고 해요. 명부는 교실마다 다르게 적혀 있어요.
3. 그래서 1반 교실은 1반 선생님이, 2반 교실은 2반 선생님이 각자 명부를 관리해요. 컴퓨터도 파일마다 이런 명부(ACL)가 있어서 안전하게 지켜준답니다!
