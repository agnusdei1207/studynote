# 분산 락 (Distributed Lock) 및 분산 환경의 상호 배제

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 락 (Distributed Lock)은 여러 대의 독립적인 서버 노드로 구성된 분산 환경에서 특정 공유 자원에 대해 오직 하나의 주체만 접근할 수 있도록 보장하는 중앙 집중식 또는 합의 기반의 상호 배제(Mutual Exclusion) 메커니즘이다.
> 2. **가치**: 마이크로서비스 아키텍처 (MSA)에서 데이터의 최종 정합성(Consistency)을 유지하고, 중복 결제나 재고 오차와 같은 비즈니스 치명적 오류를 방지하며, 분산 시스템의 리더 선출(Leader Election)을 위한 핵심 인프라로 기능한다.
> 3. **융합**: Redis의 **Redlock** 알고리즘, ZooKeeper의 **Ephemeral Node**, etcd의 **Lease/Revision** 기술과 결합되며, CAP 이론 중 일관성(Consistency)과 분할 내성(Partition Tolerance)을 중시하는 **CP 모델**을 지향한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: "전 지구적 자물쇠"다. 단일 컴퓨터 안에서는 뮤텍스(Mutex)로 동기화가 가능하지만, 서버가 100대라면 각자의 뮤텍스로는 서로를 막을 수 없다. 따라서 모든 서버가 공통으로 바라보는 외부의 제3자(코디네이터)에게 "지금 이 자원 내가 써도 되니?"라고 물어보고 허락을 받는 방식이다.

- **💡 비유**: 이는 "공용 화장실의 열쇠 꾸러미"와 같다. 여러 명의 사람들(서버)이 각자의 방(로컬 환경)에 화장실 열쇠를 가지고 있는 게 아니라, 관리사무소(Redis/ZK)에 단 하나의 열쇠만 맡겨둔다. 화장실을 쓰려면 반드시 관리소에 가서 열쇠를 받아와야 하며, 열쇠를 가져간 사람이 반납하기 전까지는 아무도 화장실에 들어갈 수 없다.

- **등장 배경**:
  1. **단일 서버의 한계**: 서비스 규모가 커지며 서버를 수평 확장(Scale-out)하게 되었고, 이로 인해 로컬 동기화 도구들이 무력화되었다.
  2. **비즈니스 정합성**: 쿠폰 선착순 발급, 한정판 재고 관리 등 "동시에 단 한 명만 성공해야 하는" 로직의 비중이 커졌다.
  3. **클라우드 네이티브의 복잡성**: 서버가 언제든 죽고 새로 생기는 동적 환경에서, 락을 가진 주체가 갑자기 사라졌을 때의 안전한 락 해제(Cleanup) 요구가 발생했다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              분산 락의 기본 아키텍처 및 제어 흐름 (Distributed)     │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ Client A ]         [ Client B ]         [ Client C ]        │
  │        │                    │                    │              │
  │   1. Acquire Lock ──▶ ┌──────────────┐ ◀── 2. Acquire Lock      │
  │      (Success)        │ Distributed  │        (Fail/Wait)       │
  │                       │  Lock Store  │                          │
  │   3. [Do Work]        │ (Redis/etcd) │        4. [Waiting...]   │
  │                       └──────┬───────┘                          │
  │        │                     │                                  │
  │   5. Release Lock ───────────┘                                  │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 분산 락의 핵심은 모든 클라이언트가 동의하는 '공통의 상태 저장소'가 있다는 점이다. 1번 클라이언트가 락을 획득하면 저장소에 "Key=ResourceX, Owner=A"라는 기록이 남는다. 2번 클라이언트가 요청하면 저장소는 이미 임자가 있음을 알리고 거절하거나 대기시킨다.

- **📢 섹션 요약 비유**: "각자 자기 집 빗장을 거는 것이 아니라, 마을 광장의 커다란 종(중앙 저장소)을 먼저 치는 사람이 대장 노릇을 하는 규칙"입니다.

### 대표적인 분산 락 구현 기술 3종 비교

사용하는 도구의 철학(CP vs AP)에 따라 성능과 신뢰성이 갈린다.

| 기술 도구 | 구현 메커니즘 | 특징 및 장점 | 단점 및 리스크 | 비유 |
|:---|:---|:---|:---|:---|
| **Redis** | `SET NX PX` (Redlock) | **압도적 성능**, 인메모리 방식의 빠른 처리 | 분산 환경에서 일관성 결여 가능성 (fencing 토큰 필요) | 빛의 속도로 움직이는 비서 |

### 분산 락의 3대 안전 수칙 (Safety Guarantees)

분산 락은 네트워크 장애라는 가혹한 환경을 견뎌야 한다.

1. **상호 배제 (Mutual Exclusion)**: 어떤 순간에도 단 하나의 클라이언트만 락을 보유해야 함.
2. **교착 상태 방지 (Deadlock Free)**: 락을 잡은 클라이언트가 사고로 죽더라도, 타임아웃(TTL)을 통해 결국 락이 풀려야 함.
3. **결함 허용 (Fault Tolerance)**: 락 저장소 자체가 일부 죽어도 시스템은 전체적으로 작동해야 함. (Quorum 기반)

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 락 타임아웃과 비즈니스 로직의 경합 (Race)           │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   Client A: [Get Lock (10s)] ──▶ [Heavy DB Job (12s...)]         │
  │                                         │                        │
  │   Lock Store: [ ⚡ TTL Expired! ] ──▶ [ Release Lock ]           │
  │                                         │                        │
  │   Client B: [Get Lock (New)] ───────────┘ ──▶ [Do Same Job!]     │
  │                                                                  │
  │   [위험] A가 아직 안 끝났는데 락이 풀려 B와 충돌 발생!              │
  │   [해결] Fencing Token (증가하는 버전 번호) 대조 필수               │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 분산 락의 가장 무서운 적은 '지연'이다. 내 일이 안 끝났는데 락의 유효시간이 먼저 끝나버리면 다른 사람이 들어와 데이터를 망가뜨릴 수 있다. 이를 막기 위해 락을 줄 때마다 증가하는 번호(Fencing Token)를 부여하고, 자원(DB 등)은 "나보다 작은 번호를 가진 놈의 요청은 거절한다"는 규칙을 지켜야 한다.

- **📢 섹션 요약 비유**: "빌려간 열쇠의 시간이 다 되면 자동으로 소멸되게 하되, 이전 사람이 아직 화장실에 있는지 확인하는 추가 확인 절차(Token)가 반드시 필요합니다."

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 데이터베이스 락(DB Lock) vs 분산 락 (Distributed Lock)

| 비교 항목 | Database Lock (Pessimistic) | Distributed Lock (Redis/ZK) |
|:---|:---|:---|
| **위치** | 데이터 보관 장소 자체 (MySQL 등) | 별도의 코디네이터 계층 |
| **성능** | 낮음 (DB 트랜잭션 오버헤드) | **매우 높음** |
| **유연성** | 낮음 (DB 내 자원만 보호) | **높음 (파일, 네트워크, 로직 보호)** |
| **신뢰성** | 최고 (ACID 보장) | 보통 (네트워크 장애 시 위험 존재) |

### 비교 2: 분산 합의 알고리즘과 락의 관계

- **Paxos / Raft**: 락 저장소가 "누가 락을 가졌는지"에 대해 서버들끼리 투표를 거쳐 결정하는 기반 기술.
- **Redlock**: 과반수(Majority) 이상의 노드에서 락 획득에 성공해야 비로소 락을 가진 것으로 간주하는 알고리즘.

- **과목 융합 관점 (네트워크)**: 분산 락은 **시계 동기화 (Clock Skew)** 문제와 밀접하다. 서버 A의 1초와 서버 B의 1초가 다를 수 있으므로, 락의 만료 시간을 계산할 때 단순 로컬 시간이 아닌 논리적 시계나 보수적인 여유 시간을 두어야 한다.

- **📢 섹션 요약 비유**: "내부 규정(DB Lock)으로 해결할 것인가, 공신력 있는 외부 기관(Distributed Lock)의 공증을 받을 것인가의 전략적 판단"입니다.

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 커머스 서비스 선착순 쿠폰 발급**: 수백만 명이 동시에 클릭함. DB로만 처리하면 DB가 폭발함. 기술사적 판단으로 **Redis 기반의 분산 락**을 제안한다. Redis의 초고속 성능을 활용하여 입구에서 락을 통해 1초에 10만 건 이상의 요청을 정렬하고, 락을 획득한 성공자만 DB에 접근하게 하여 시스템 전체의 부하를 격리한다.

2. **시나리오 — 중복 결제 방지 (Idempotency)**: 네트워크 재시도로 동일 결제 요청이 두 번 들어옴. 해결책으로 **UUID 기반 분산 락**을 적용한다. 결제 요청마다 고유 ID를 락 키로 사용하고, 첫 요청이 처리 중일 때 들어온 두 번째 요청은 "이미 처리 중" 에러를 리턴하여 사용자 통고와 정합성을 동시에 잡는다.

3. **시나리오 — 배치 서버 리더 선출 (Scheduler)**: 3대의 서버가 동일한 주기로 배치를 돌려야 하는데, 한 번에 한 대만 돌아야 함. **ZooKeeper의 Ephemeral Node**를 활용한다. 3대 모두 `/master` 노드 생성을 시도하고, 성공한 한 대만 일을 한다. 만약 마스터 서버가 죽으면 주키퍼가 노드를 자동 삭제하고, 남은 2대 중 하나가 즉시 새로운 마스터가 되어 업무 연속성을 유지한다.

### 도입 체크리스트 (Architecture Audit)
- **Time-to-Live (TTL)**: 비즈니스 로직의 최대 실행 시간보다 락 만료 시간이 넉넉한가?
- **Retry Policy**: 락 획득 실패 시 얼마나 기다리고 몇 번 다시 시도할 것인가? (Exponential Backoff 적용 확인)
- **Cleanup**: 락 획득 후 로직에서 예외 발생 시, `finally` 블록에서 확실하게 락을 반납(Delete)하고 있는가?

### 안티패턴
- **단일 Redis 노드에 락 의존**: 해당 Redis 서버가 죽으면 전사 시스템의 모든 비즈니스 로직이 멈춤 (SPOF). 반드시 클러스터나 쿼럼 기반(Redlock) 구성을 해야 함.
- **너무 넓은 락 범위**: "불안하니까 전체를 다 잠그자"는 태도. 분산 환경의 최대 장점인 병렬성을 스스로 죽이고 성능을 1990년대로 되돌리는 행위.

- **📢 섹션 요약 비유**: "자물쇠를 채우는 것보다 더 중요한 것은, 열쇠를 가진 사람이 사고를 당했을 때 어떻게 문을 열 것인지에 대한 대비책(Timeout/Quorum)입니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (분산 락 도입 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **데이터 정합성** | 멀티 노드 환경에서 데이터 꼬임 현상 0% 달성 | 비즈니스 신뢰도 확보 |
| **시스템 확장성** | 노드 수와 상관없이 일관된 동기화 성능 제공 | 가용성 증대 |
| **운영 유연성** | 장애 노드 발생 시 자동 락 회수 및 서비스 지속 | 무중단 서비스 인프라 |
| **개발 생산성** | 복잡한 분산 동기화 로직을 라이브러리(Redisson 등)로 단순화 | 개발 속도 향상 |

### 미래 전망
- **Hardware-offloaded Locking**: 네트워크 카드(SmartNIC) 내부에서 RDMA 기술을 이용해 하드웨어 수준에서 분산 락을 처리하여 소프트웨어 레이턴시를 제로화하는 기술.
- **Byzantine Fault Tolerant (BFT) Locking**: 일부 노드가 악의적으로 거짓 정보를 보내도 락의 정당성을 보장하는 블록체인 기반의 초고신뢰 분산 락 연구.

### 참고 표준
- **Redis Redlock Algorithm**: 안티레즈(Antirez)가 제안한 분산 락의 사실상 표준 알고리즘.
- **Raft Consensus Algorithm**: etcd 등에서 사용하는 분산 합의 및 락 관리의 이론적 표준.

- **📢 섹션 요약 비유**: 분산 락은 "혼돈의 도가니인 네트워크 세상을 질서의 정원으로 바꾸는 보이지 않는 울타리"이며, 분산 아키텍처를 완성하는 마지막 퍼즐 조각입니다.
