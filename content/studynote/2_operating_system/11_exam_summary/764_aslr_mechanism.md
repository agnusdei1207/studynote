# ASLR (Address Space Layout Randomization)의 원리와 보호 매커니즘

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스가 실행될 때마다 스택(Stack), 힙(Heap), 라이브러리(Library) 등의 **주요 메모리 영역의 시작 주소를 무작위로 배치**하여 예측 불가능하게 만드는 보안 기술.
> 2. **가치**: 공격자가 특정 함수의 주소를 미리 알아내어 실행 제어권을 탈취하는 **버퍼 오버플로우** 및 **ROP(Return Oriented Programming)** 공격의 성공 확률을 획기적으로 낮춘다.
> 3. **융합**: 운영체제의 로더(Loader) 기능과 하드웨어의 MMU 주소 변환 기술이 결합되어 성능 저하를 최소화하면서도 강력한 방어막을 형성한다.

---

### Ⅰ. ASLR의 동작 원리

- **정적 배치 (과거)**: 모든 프로세스의 스택이 항상 `0x7fff0000`에서 시작한다면, 해커는 이 주소를 목표로 악성 코드를 심을 수 있다.
- **ASLR 적용 (현재)**: 실행할 때마다 커널이 임의의 오프셋(Offset)을 생성하여 메모리 맵을 비튼다.

---

### Ⅱ. 메모리 레이아웃 변화 비교 (ASCII)

```ascii
    < Run 1 >                          < Run 2 >
    +-----------------------+          +-----------------------+
    | Stack [0x7fff1234]    |          | Stack [0x7fff9876]    | <-- Position Changed!
    +-----------------------+          +-----------------------+
    | (Empty Space)         |          | (Empty Space)         |
    +-----------------------+          +-----------------------+
    | Shared Libs           |          | Shared Libs           |
    +-----------------------+          +-----------------------+
    | Heap [0x00a1b2c3]     |          | Heap [0x00f9e8d7]     | <-- Position Changed!
    +-----------------------+          +-----------------------+
    | Code (PIE)            |          | Code (PIE)            |
    +-----------------------+          +-----------------------+
```

**[핵심 기술: PIE (Position Independent Executable)]**
- 코드 영역까지 난수화하려면, 코드가 메모리의 어느 위치에 있든 상관없이 실행 가능하도록 컴파일된 **PIE** 바이너리가 필요하다.

---

### Ⅲ. ASLR의 한계 및 우회 공격

| 우회 기법 | 설명 | 대응 방안 |
|:---|:---|:---|
| **Information Leak** | 프로그램의 취약점을 통해 메모리 주소 하나를 알아내어 전체 맵을 유추. | 정교한 에러 메시지 통제 |
| **Brute-forcing** | 32비트 환경처럼 난수 범위가 좁을 때 반복적으로 실행하여 주소를 맞힘. | **64비트 시스템**으로 전환 (난수 범위 확장) |
| **Heap Spraying** | 힙 영역에 악성 코드를 도배하여 어디를 찔러도 걸리게 함. | 힙 보호 강화 (Metadata Integrity) |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. ASLR 활성화 확인 및 튜닝 (Linux)
- **명령어**: `cat /proc/sys/kernel/randomize_va_space`
- **설정값**: 
  - **0**: 꺼짐.
  - **1**: 스택, 데이터, 라이브러리 난수화.
  - **2**: **(권장)** 힙 영역까지 포함하여 전체 난수화.
- **기술사적 결단**: 성능에 민감한 시스템이라도 보안 사고의 비용이 훨씬 크므로, 반드시 레벨 2 설정을 유지해야 한다.

#### 2. 기술사적 인사이트
- **DEP와 ASLR의 협업**: ASLR은 주소를 숨기는 것이고, DEP(Data Execution Prevention)는 해당 영역의 실행을 막는 것이다. 이 둘이 동시에 적용될 때(Defense in Depth) 비로소 현대적인 메모리 보호 체계가 완성된다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **익스플로잇 비용 상승**: 해커가 공격 코드를 작성하는 난이도와 시간을 기하급수적으로 높임.
- **자동화된 공격 방어**: 주소를 고정해서 공격하는 대부분의 자동화 도구를 무력화.

#### 2. 미래 전망
최근에는 부채널 공격(Spectre 등)을 통해 ASLR의 무작위 주소를 알아내려는 시도가 늘고 있다. 이에 대응하여 커널과 유저 주소 공간을 더 엄격히 격리하는 **KPTI**나, 하드웨어가 포인터의 유효성을 직접 검증하는 **Pointer Authentication (PAC)** 기술이 ASLR의 든든한 후원군으로 등장하고 있다.
