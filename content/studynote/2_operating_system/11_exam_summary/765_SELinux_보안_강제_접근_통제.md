# SELinux (Security-Enhanced Linux) 강제 접근 통제

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: SELinux는 리눅스 커널의 보안 모듈(LSM)을 통해 구현된 **강제적 접근 제어 (MAC, Mandatory Access Control)** 시스템으로, 모든 주체(프로세스)와 객체(파일, 포트 등)에 보안 컨텍스트(레이블)를 부여하고 사전에 정의된 보안 정책에 따라 최소 권한의 실행을 강제한다.
> 2. **가치**: 루트 권한(Root)이 탈취당하더라도 프로세스가 정책에서 허용한 행위 외의 자원을 건드리지 못하게 함으로써 보안 사고의 "폭쇄 영역 (Blast Radius)"을 해당 도메인으로 가두며, 제로 데이 취약점 공격으로부터 시스템 전체의 붕괴를 방어한다.
> 3. **융합**: **Type Enforcement (TE)** 기법을 중심으로 RBAC, MLS(다단계 보안) 모델을 제공하며, 현대의 안드로이드 OS 보안(SEAndroid), 클라우드 컨테이너 격리 및 고신뢰성 서버 보안 아키텍처의 필수 구성 요소로 기능한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: SELinux는 "주인이 허락하면 다 된다"는 전통적인 방식(DAC) 대신, "나라(시스템 정책)가 허락한 일만 할 수 있다"는 원칙(MAC)을 따르는 보안 운영체제 확장 기술이다. 아무리 높은 직급의 사람이라도 정해진 출입 규칙을 어길 수 없는 보안 구역 관리와 같다.

- **💡 비유**: 일반 리눅스(DAC)가 "내 가방은 내 마음대로 열 수 있고, 내가 남에게 열어줄 수도 있는 방식"이라면, SELinux는 "가방에 들어가는 모든 물건에 바코드를 붙이고, 가방을 열 때마다 보안 요원(SELinux)이 '이 사람이 이 바코드가 붙은 물건을 꺼낼 자격이 있는가'를 일일이 대조하여 허가하는 방식"이다.

- **등장 배경**:
  1. **DAC의 구조적 한계**: 소유자가 권한을 실수로 `777`로 주거나, 루트 권한을 가진 데몬 하나가 해킹당하면 시스템 전체가 무너지는 취약점이 있었다.
  2. **미 국방성(NSA)의 요구**: 국가 보안 시설 등에서 사용할 수 있는, 관리자조차 마음대로 바꿀 수 없는 강력하고 세밀한 보안 모델이 필요해졌다.
  3. **내부자 위협 방어**: 시스템에 합법적으로 접속한 사용자나 앱이 인가되지 않은 다른 영역으로 이동(Lateral Movement)하는 것을 막아야 했다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              DAC (전통 방식) vs MAC (SELinux) 제어 흐름 비교       │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ 1. DAC: Discretionary ]        [ 2. MAC: Mandatory ]        │
  │                                                                 │
  │   User (Root) ──▶ File A           Subject ──▶ [ Security ] ──▶ Object │
  │      │              │              (Label)     [  Policy  ]     (Label)│
  │      └─(UID=0)─▶ [ OK ]            (httpd_t)   [ Allow? ]       (etc_t)│
  │                                                   │                    │
  │   [특징] 소유자 마음대로             [특징] 정책이 강제로 결정           │
  │   - 루트는 무적임                   - 루트도 정책을 위반할 수 없음       │
  │   - 전파 통제 불가                  - 최소 권한 원칙의 완벽한 구현       │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** DAC에서는 루트 사용자가 모든 파일을 건드릴 수 있지만, SELinux(MAC) 환경에서는 '웹 서버 프로세스(httpd_t)'라는 레이블을 가진 주체가 '비밀번호 파일(shadow_t)'이라는 레이블을 가진 객체에 접근하려 할 때, 비록 그 프로세스가 루트 권한이라 할지라도 정책에 허용되지 않았다면 커널 수준에서 즉시 차단한다.

- **📢 섹션 요약 비유**: "누구인지(UID)보다 어떤 역할을 맡았는지(Security Context)를 따져서 출입을 제한하는 철저한 업무 중심 보안관"입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SELinux 3대 작동 모드

관리자는 상황에 따라 SELinux의 개입 수준을 조절할 수 있다.

| 모드명 | 상태 및 특징 | 실무적 의미 | 비유 |
|:---|:---|:---|:---|
| **Enforcing** | 정책을 강제하고 위반 시 차단 | **실제 보안 운영 상태**. 최고 보안 수준 유지 | 경찰이 현장에서 체포함 |
| **Permissive** | 정책 위반을 허용하되 로그만 남김 | 정책 튜닝 및 디버깅용. 앱 오작동 원인 분석 | 위반 딱지만 끊고 보냄 |
| **Disabled** | SELinux 기능을 완전히 끔 | 보안 기능 부재. 재부팅 필요 | 경찰서가 문을 닫음 |

---

### 핵심 아키텍처 요소: 컨텍스트와 TE (Type Enforcement)

SELinux는 **`user : role : type : level`** 형식의 레이블을 사용하여 모든 것을 관리한다. 이 중 가장 핵심은 **Type**이다.

1. **Type Enforcement (TE)**: "이 타입의 프로세스는 저 타입의 파일에 접근할 수 있다"는 규칙을 정의함. (예: `allow httpd_t user_home_t:file read;`)
2. **AVC (Access Vector Cache)**: 성능 향상을 위해 최근의 권한 결정 사항을 메모리에 캐싱함.
3. **Security Server**: 커널 내부에 위치하며, 실제 정책을 해석하고 결정을 내리는 두뇌 역할.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 SELinux의 결정 과정 (Decision Making)              │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   Subject (P1) ──▶ [ Request: Read File A ] ──▶ Object (File A)  │
  │          │                                         │             │
  │          ▼                                         ▼             │
  │   Label: system_u:system_r:httpd_t      Label: var_log_t         │
  │                                                                  │
  │          └───────────────▶ [ AVC Cache ] ───────────────┐        │
  │                               │ (Miss?)                 │        │
  │                               ▼                         ▼        │
  │                        [ Security Policy ] ──▶ [ Decision: YES ] │
  │                                                                  │
  │   [결론] 주체와 객체의 타입을 대조하여 허용 리스트에 있는지 확인      │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 모든 I/O 요청은 하단 커널로 내려가기 전 SELinux 보안 서버를 통과해야 한다. 먼저 속도가 빠른 AVC(캐시)를 보고, 기록이 없으면 전체 정책 DB를 뒤진다. 이 과정은 하드웨어적으로 최적화되어 있어 시스템 성능에 미치는 영향은 2~5% 이내로 매우 적다.

- **📢 섹션 요약 비유**: "모든 사람의 가슴에 명찰(타입)을 달게 하고, 명찰의 색깔에 따라 들어갈 수 있는 방을 미리 정해놓은 거대한 규칙의 성"과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: SELinux vs AppArmor (라이벌 보안 모듈)

| 비교 항목 | SELinux (RHEL, Fedora) | AppArmor (Ubuntu, Debian) |
|:---|:---|:---|
| **기반 원칙** | **타입(Label)** 기반 제어 | **경로(Path)** 기반 제어 |
| **복잡도** | 매우 높음 (학습 곡선 가파름) | 낮음 (직관적인 프로필 작성) |
| **보안 강도** | **최강** (전방위적 제어) | 보통 (정해진 경로만 보호) |
| **유연성** | 좋음 (객체 중심의 세밀한 정의) | 낮음 (파일명 바뀌면 무력화 가능) |

### 비교 2: SELinux와 컨테이너 보안 (Docker/K8s)

- **sVirt**: 하이퍼바이저가 각 VM에 고유한 SELinux 카테고리를 부여하여, VM 간의 침범을 막는 기술.
- **Container Isolation**: 도커 실행 시 `--security-opt label:type:container_t` 등의 옵션을 주어, 컨테이너가 호스트의 중요 파일이나 다른 컨테이너의 데이터를 볼 수 없게 물리적으로 격리함.

- **과목 융합 관점 (데이터베이스)**: SELinux의 MLS(Multi-Level Security)는 군사 시스템의 **다단계 보안 모델 (Bell-LaPadula)**을 그대로 옮겨온 것이다. "낮은 등급은 높은 등급을 읽을 수 없고(No Read Up), 높은 등급은 낮은 등급에 쓸 수 없다(No Write Down)"는 원칙을 커널 수준에서 강제한다.

- **📢 섹션 요약 비유**: "전신 갑옷을 입는 것(SELinux)이 움직이기는 불편해도 공격에는 완벽히 안전하며, 가벼운 방탄조끼(AppArmor)는 편하지만 틈새가 있는 것과 같습니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 웹 서버 포트 변경 후 동작 불능**: Apache 서버의 포트를 80에서 8080으로 바꿨는데 실행이 안 됨. 로그에는 `Permission Denied` 발생. 기술사적 판단으로 **SELinux 포트 정책 위반**임을 진단한다. `semanage port -a -t http_port_t -p tcp 8080` 명령을 통해 새 포트를 화이트리스트에 등록하여 문제를 해결한다.

2. **시나리오 — 랜섬웨어 감염 시 피해 최소화**: 파일 서버가 해킹당해 랜섬웨어가 실행됨. 하지만 SELinux 정책에 따라 파일 서버 프로세스는 오직 `/data` 영역만 수정 가능하고 시스템 설정(`/etc`)이나 백업 폴더는 건드릴 수 없음. 결과적으로 중요한 시스템 파괴를 막아내고 데이터 복구의 골든타임을 확보함.

3. **시나리오 — 신규 애플리케이션 정책 생성**: 새로운 보안 앱을 개발했는데 SELinux 때문에 자꾸 막힘. 해결책으로 잠시 **Permissive 모드**로 돌려 모든 거부 로그(`audit.log`)를 수집한 뒤, **`audit2allow`** 도구를 사용하여 해당 앱에 딱 맞는 최소 권한 정책을 자동으로 생성하여 배포한다.

### 도입 체크리스트 (Security Hardening)
- **모드 확인**: `getenforce` 실행 시 반드시 `Enforcing` 상태인가?
- **Context 무결성**: 파일 복사나 이동 시 `-p` 옵션 등을 써서 보안 컨텍스트가 유지되고 있는가?
- **Booleans 관리**: `getsebool -a`를 통해 웹 서버의 DB 접속 허용(`httpd_can_network_connect_db`) 등 세부 스위치가 비즈니스 요건에 맞게 설정되었는가?

### 안티패턴
- **귀찮다고 `setenforce 0` (Disabled)**: 보안 오류가 날 때마다 SELinux를 꺼버리는 행위. 이는 성벽을 통째로 허물고 적을 맞이하는 것과 같음. (해결: 로그를 보고 정책을 수정하는 것이 정석)
- **잘못된 컨텍스트 복원**: 설정 파일을 직접 수정하고 `restorecon`을 하지 않아 시스템 재부팅 시 권한 오류가 재발하는 상황.

- **📢 섹션 요약 비유**: "불편함(제약)은 안전의 대가이며, 진정한 고수는 방패(SELinux)를 버리는 것이 아니라 내 몸에 맞게 조절해서 쓰는 사람입니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (SELinux 도입 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **침해 방지율** | 알려지지 않은 공격에 대한 시스템 장악 방어율 90% 이상 | 제로데이 방어의 핵심 |
| **권한 상승 차단** | Root 계정 탈취 후의 Lateral Movement 완벽 봉쇄 | 피해 범위 최소화 |
| **컴플라이언스** | 국방, 금융 등 고신뢰성 보안 표준(CC 인증) 충족 | 대외 신뢰도 향상 |
| **운영 가시성** | 모든 접근 시도에 대한 상세 감사 로그(Audit Log) 제공 | 사후 포렌식 용이 |

### 미래 전망
- **Self-Tuning SELinux**: AI가 시스템의 정상적인 동작 패턴을 학습하여, 새로운 앱이 설치될 때 필요한 정책을 스스로 제안하고 적용하는 지능형 MAC 연구.
- **Micro-segmentation with SELinux**: 클라우드 네이티브 환경에서 컨테이너 단위의 세밀한 정책 적용을 자동화하는 서비스 메시(Service Mesh) 보안과의 융합 가속화.

### 참고 표준
- **LSM (Linux Security Modules)**: SELinux가 플러그인되는 커널 표준 프레임워크.
- **NIST SP 800-192**: 접근 제어 시스템의 평가 및 보안 가이드라인.

- **📢 섹션 요약 비유**: SELinux는 리눅스라는 자유로운 대지에 세워진 "법과 질서"이며, 이 질서가 확립될 때 비로소 거대한 엔터프라이즈 시스템이 안전하게 지탱될 수 있습니다.
