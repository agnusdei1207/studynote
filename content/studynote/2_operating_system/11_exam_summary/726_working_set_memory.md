# 워킹 셋 (Working Set) 모델을 활용한 스래싱 방지

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스가 일정 시간($\Delta$) 동안 자주 참조하는 페이지들의 집합을 정의하고, 이를 **메모리에 상주하도록 보장**하여 페이지 폴트를 최소화하는 동적 메모리 관리 모델.
> 2. **가치**: 프로그램의 국부성(Locality)을 수치화하여, 시스템의 가용 프레임이 프로세스들의 워킹 셋 합계보다 작을 경우 일부 프로세스를 중단시킴으로써 **스래싱(Thrashing)**을 원천 차단한다.
> 3. **융합**: 참조 국부성 이론과 운영체제의 중기 스케줄러(Medium-term Scheduler) 기능이 결합된 형태이며, 현대의 윈도우/리눅스 메모리 관리자의 기초 원리로 작용한다.

---

### Ⅰ. 워킹 셋 (Working Set)의 정의 및 개념

- **정의**: 시간 $t$에서 윈도우 크기 $\Delta$ 동안 참조된 페이지의 집합 $WSS_i(t, \Delta)$.
- **Working Set Window ($\Delta$)**: 프로세스의 국부성을 파악하기 위해 관찰하는 과거의 시간 범위.
- **WSS (Working Set Size)**: 워킹 셋에 포함된 고유한 페이지의 개수.

---

### Ⅱ. 워킹 셋 모델의 동작 아키텍처 (ASCII)

시간 흐름에 따라 워킹 셋의 구성이 변하는 모습이다.

```ascii
    [ Page Reference Stream ]
    ... 2 6 1 5 7 7 7 7 5 1 6 2 3 4 1 2 3 4 4 4 3 4 3 4 ...
    
    Time t1: (Delta = 10)
    Reference: [ 2 6 1 5 7 7 7 7 5 1 ]
    Working Set: { 1, 2, 5, 6, 7 } -> WSS = 5
    
    Time t2: (Delta = 10)
    Reference: [ 3 4 1 2 3 4 4 4 3 4 ]
    Working Set: { 1, 2, 3, 4 } -> WSS = 4
```

- **관리 규칙**:
  - $D = \sum WSS_i$ (모든 프로세스의 워킹 셋 크기 합)
  - $m$ = 총 가용 물리 프레임 수
  - **If $D > m$**: 스래싱 위험! 운영체제는 한 프로세스를 선택하여 Swap-out 시키고 프레임을 회수한다.

---

### Ⅲ. 워킹 셋 윈도우 ($\Delta$) 크기의 중요성

$\Delta$의 설정에 따라 메모리 관리의 정밀도가 달라진다.

| $\Delta$ 크기 | 결과 | 성능 영향 |
|:---|:---|:---|
| **너무 작을 때** | 전체 국부성을 포함하지 못함. | 빈번한 페이지 폴트 발생 |
| **너무 클 때** | 서로 다른 국부성 영역까지 포함함. | 메모리 낭비 (Over-allocation) |
| **무한대 ($\infty$)** | 프로그램 시작부터 지금까지의 모든 페이지 포함. | 다중 프로그래밍 불가능 |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 스래싱 탐지 및 대응 (PFF 연계)
- **현상**: 시스템이 버벅거리며 I/O 사용률만 치솟는 경우.
- **기술사적 결단**: 워킹 셋 모델을 직접 구현하기는 복잡하므로, 실무적으로는 **페이지 폴트 빈도 (PFF, Page Fault Frequency)**를 모니터링하여 상한선을 넘으면 프레임을 더 주고, 하한선 아래면 뺏는 상호보완적 전략을 취한다.

#### 2. 기술사적 인사이트
- **프로세스 중단 (Swapping)**: 워킹 셋을 보장해주지 못할 바에는, 해당 프로세스를 통째로 디스크로 내보내는 것이 시스템 전체의 가용성을 지키는 최선의 선택이다. (중기 스케줄러의 존재 이유)
- **현대적 메모리 압축**: 최근에는 Swap-out 대신 메모리 내에서 데이터를 압축(zRAM)하여 워킹 셋을 유지하려는 기술도 널리 쓰인다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **CPU 이용률 안정화**: 스래싱 구역 진입 방지를 통한 처리량 극대화.
- **응답성 보장**: 프로세스가 필요한 자원을 항상 쥐고 있게 하여 지연 최소화.

#### 2. 미래 전망
최근의 서버리스(Serverless) 컴퓨팅 환경에서는 함수의 실행 시간이 매우 짧아 워킹 셋을 학습할 시간이 부족하다. 이에 따라 과거 실행 이력을 DB화하여 실행 즉시 워킹 셋을 미리 로드하는 **Warm-start 최적화** 기술로 진화하고 있으며, 이는 고전적인 워킹 셋 모델의 데이터 기반 확장판이라 할 수 있다.
