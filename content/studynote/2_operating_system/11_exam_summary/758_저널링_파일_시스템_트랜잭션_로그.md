# 저널링 파일 시스템 (Journaling File System / WAL)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 저널링 (Journaling)은 파일 시스템의 변경 사항을 실제 데이터 영역에 반영하기 전, 전용 로그 영역(Journal)에 먼저 기록하는 **선행 기록 로그 (WAL, Write-Ahead Logging)** 기법을 통해 시스템 크래시 시 데이터 일관성을 보장하는 기술이다.
> 2. **가치**: 비정상 종료 후 재부팅 시 디스크 전체를 전수 조사해야 하는 고전적인 `fsck` (File System Check)의 거대한 시간 오버헤드를 제거하여 시스템 가용성(High Availability)을 혁신적으로 높인다.
> 3. **융합**: 데이터베이스의 트랜잭션 처리 원리를 파일 시스템 계층으로 확장한 아키텍처로, ext4, XFS, NTFS 등 현대 표준 운영체제 파일 시스템의 무결성 보장을 위한 핵심 메커니즘으로 자리 잡았다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 저널링은 파일 시스템 작업을 하나의 '트랜잭션'으로 묶어 장부에 먼저 적고 실제 공사를 시작하는 방식이다. 공사 도중 사고가 나더라도 장부(저널)를 확인하여 어디까지 진행되었는지, 무엇을 되돌려야 하는지 즉시 판단할 수 있다.

- **💡 비유**: 이는 은행의 송금 과정과 같다. 통장 잔액을 바꾸기 전, 은행 전산 로그에 "A가 B에게 100원을 보냄"이라고 먼저 적어둔다. 로그를 적는 도중 정전이 나면 송금은 없었던 일이 되고, 로그를 다 적은 뒤 실제 이체 중 정전이 나면 재부팅 후 로그를 보고 이체 작업을 마저 끝내는 식이다.

- **등장 배경**:
  1. **데이터 불일치 (Inconsistency)**: 메타데이터(아이노드)는 수정했는데 실제 데이터 블록을 쓰기 전 정전이 나면 파일 시스템이 '누더기'가 되는 문제가 빈번했다.
  2. **fsck의 한계**: 테라바이트급 하드디스크가 등장하면서, 부팅 시 전체 블록을 스캔하여 오류를 잡는 `fsck` 작업이 수 시간씩 걸려 서버 가동률을 심각하게 갉아먹었다.
  3. **대규모 I/O 환경**: 고성능 서버에서 지연 쓰기(Delayed Write)를 쓰면서도 데이터 안전을 담보할 장치가 필요해졌다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              저널링 파일 시스템의 기본 동작 구조 (ext4 기준)         │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ Memory ]           [ Journal Area ]        [ Data Area ]    │
  │   ┌────────┐           ┌──────────────┐        ┌─────────────┐  │
  │   │ Buffer │ ──(1)──▶  │ Log Entry 1  │        │             │  │
  │   │ Cache  │           │ Log Entry 2  │        │             │  │
  │   └────────┘           │ [Commit Mark]│──(2)─▶ │ Real Block  │  │
  │          │             └──────────────┘        │   Update    │  │
  │          │                    │                │             │  │
  │          └────────────────────┴─────(3)──────▶ │             │  │
  │                                                └─────────────┘  │
  │                                                                 │
  │   (1) Journal Write: 변경 사항을 로그에 기록 (빠른 순차 쓰기)       │
  │   (2) Checkpointing: 실제 데이터 영역에 반영 (느린 랜덤 쓰기)       │
  │   (3) Commit: 로그 삭제 또는 완료 표시                             │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 핵심은 '순서'다. 실제 위치에 데이터를 쓰기 전, 헤드 이동이 적은 전용 영역에 로그를 한꺼번에 쏟아붓는다(순차 쓰기). 로그에 'Commit' 마크가 찍히는 순간 데이터는 안전한 것으로 간주되며, 이후 실제 작업 중 사고가 나도 로그를 다시 읽어(Replay) 복구할 수 있다.

- **📢 섹션 요약 비유**: "실제 집을 짓기 전, 설계도(로그)에 확정 도장을 먼저 찍음으로써 태풍(크래시)이 와도 집을 다시 지을 수 있게 하는 안전 장치"입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 저널링 모드에 따른 3단계 전략 분석 (ext4 사례)

모든 것을 저널링하면 안전하지만 시스템이 매우 느려진다. 따라서 운영체제는 3가지 선택지를 제공한다.

| 모드명 | 저널링 대상 | 장점 (Pros) | 단점 (Cons) | 비유 |
|:---|:---|:---|:---|:---|
| **data=journal** | 메타데이터 + **데이터** | 최고의 안전성. 데이터 유실 0% 추구. | 모든 데이터를 두 번 써야 하므로 매우 느림. | 복사본까지 일일이 기록 |
| **data=ordered** | **메타데이터** | 성능과 안전의 균형. 메타데이터 기록 전 데이터 쓰기 보장. | 파일 내용 일부가 낡은 상태일 수 있음. | 작업 목록만 기록 (리눅스 기본값) |
| **data=writeback** | 메타데이터 | 최고의 성능. 메타데이터만 보호. | 크래시 후 파일 내부에 쓰레기 값이 들어갈 수 있음. | 목차만 기록 |

---

### 크래시 복구 프로세스 (Recovery Mechanism)

재부팅 시 파일 시스템은 가장 먼저 저널 영역을 스캔한다.

1. **Scan Journal**: 저널 영역에 완료되지 않은(Commit 마크가 없는) 트랜잭션이 있는지 확인.
2. **Replay (Redo)**: 저널에 기록은 끝났지만 실제 영역에 반영되지 않은 작업(Checkpoint 누락)을 찾아 실제 영역에 다시 기록.
3. **Discard**: 기록조차 끝나지 않은 로그는 무효화하고 버림.
4. **Consistency Achieved**: 파일 시스템은 일관된 상태로 마운트됨. (수 초 이내 완료)

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 fsck vs Journaling 복구 시간 비교                  │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   Recovery Time                                                  │
  │      ▲                                                           │
  │      │      fsck (Linear to Disk Size)                           │
  │      │       /                                                   │
  │      │      /                                                    │
  │      │     /                                                     │
  │      │    /      Journaling (Constant / Log Size)                │
  │      │   /      _________________________________                │
  │      └──/──────────────────────────────────────────▶             │
  │                                           Disk Capacity (TB)     │
  │                                                                  │
  │   [결론] 저널링은 디스크가 아무리 커져도 복구 시간이 일정함.          │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 저널링의 최대 강점은 '예측 가능성'이다. 1TB든 100TB든, 단 몇 MB 크기의 저널 로그만 훑으면 복구가 끝나기 때문이다. 반면 저널링이 없는 파일 시스템은 디스크가 커질수록 복구 시간도 기하급수적으로 늘어나 엔터프라이즈 환경에서 퇴출당했다.

- **📢 섹션 요약 비유**: "창고 전체 물건을 세어보는 것(fsck)보다, 오늘 들어온 장부 한 장(Journal)을 확인하는 것이 훨씬 빠르고 정확한 재고 관리법"입니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 고전적 FS (ext2) vs 저널링 FS (ext3/4) vs 차세대 FS (ZFS)

| 비교 항목 | ext2 (No Journal) | ext4 (Journaling) | ZFS (COW-based) |
|:---|:---|:---|:---|
| **복구 방식** | 전수 조사 (`fsck`) | 로그 재현 (`Journal Replay`) | 항상 일관성 유지 (COW) |
| **복구 시간** | 매우 김 (수십 분~시간) | 매우 짧음 (수 초) | **거의 즉시** |
| **쓰기 오버헤드** | 없음 (빠름) | 보통 (로그 쓰기 추가) | 보통 (메타데이터 트리 갱신) |
| **데이터 보호** | 낮음 | 중간 | **최상 (Checksum 포함)** |

### 비교 2: 파일 시스템 저널링 vs 데이터베이스 WAL

- **철학적 유사성**: 둘 다 "안전하게 저장했다는 확신이 들기 전까지는 원본을 건드리지 않는다"는 트랜잭션 원리를 공유함.
- **차이점**: DB WAL은 애플리케이션 레벨에서 데이터 정합성을 관리하고, FS 저널링은 커널 레벨에서 블록 장치의 물리적 정합성을 관리함. 현대 시스템은 DB WAL과 FS 저널링이 겹쳐서 작동하는 **이중 로깅** 구조를 가짐.

- **과목 융합 관점 (컴퓨터 구조)**: 저널링은 비휘발성 메모리(NVRAM)가 도입될 경우 아키텍처가 변한다. 로그 쓰기 자체를 초고속 NVRAM에 수행하면 `data=journal` 모드에서도 오버헤드가 0에 수렴하게 되어, 극강의 안전성과 성능을 동시에 확보할 수 있다.

- **📢 섹션 요약 비유**: "저널링이 '적어두는 기술'이라면, COW는 '새로 만드는 기술'이며, 둘 다 사고로부터 데이터를 구하려는 인류의 지혜입니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 고성능 웹 서버 포맷 결정**: 이미지 썸네일을 대량 생산하는 서버. 데이터 유실보다 응답 속도가 중요함. 기술사적 판단으로 **ext4의 `data=writeback` 모드** 마운트를 권고한다. 메타데이터 구조만 저널링하여 파일 시스템 붕괴는 막되, 개별 데이터 기록 오버헤드를 최소화하여 I/O 성능을 30% 이상 끌어올린다.

2. **시나리오 — 금융 트랜잭션 서버 구축**: 절대 데이터 유실이 없어야 함. **`data=journal`** 모드와 함께 하드웨어 RAID (Battery Backed Cache)를 결합한다. 저널 로그를 쓰는 중간에 전원이 나가더라도 BBU(Battery Backup Unit)가 메모리 데이터를 보호하여, 최악의 경우에도 100% 복구를 보장한다.

3. **시나리오 — SSD 수명 및 저널링**: SSD 환경에서 저널 영역은 특정 블록에 쓰기가 집중되어 마모가 심할 수 있다. 해결책으로 **로그 구조 파일 시스템 (LFS)**이나 저널 영역을 SSD의 여러 곳으로 분산시키는 FTL 최적화를 확인하고 적용한다.

### 도입 체크리스트 (Storage Tuning)
- **저널 위치 분리**: 성능을 위해 저널 로그만 속도가 빠른 별도 SSD에 저장할 수 있는가? (ext4의 External Journal 기능)
- **배리어 (Barrier) 설정**: 디스크 컨트롤러의 쓰기 캐시가 저널링 순서를 어기지 않도록 `barrier=1` 옵션이 켜져 있는가?
- **용량 설계**: 저널 영역 크기가 워크로드의 초당 트랜잭션 양을 수용하기에 충분한가?

### 안티패턴
- **저널링을 믿고 백업 소홀**: 저널링은 '파일 시스템의 논리적 구조'를 지켜주는 것이지, 디스크 고장(Physical Failure)이나 실수로 지운 파일(Human Error)을 복구해주지 않음.
- **오래된 fsck의 맹신**: 저널링 시스템에서 강제로 `fsck`를 돌리는 행위. 정상적인 저널 리플레이를 방해하여 오히려 데이터를 오염시킬 위험이 있음.

- **📢 섹션 요약 비유**: "보험(저널링)을 들었다고 해서 불장난(백업 미비)을 해도 된다는 뜻은 아닙니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (저널링 도입 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **가용성 (Uptime)** | 크래시 후 부팅 즉시 서비스 재개 가능 | MTTR 90% 이상 단축 |
| **데이터 정합성** | 파일 시스템 꼬임(Cross-linked files) 현상 제거 | OS 안정성 혁명 |
| **운영 자동화** | 수동 fsck 개입 없이 자가 치유(Self-healing) 구현 | 관리 비용 절감 |
| **사용자 신뢰** | 갑작스러운 종료에도 작업 결과 보존 확률 증대 | 서비스 신뢰도 상승 |

### 미래 전망
- **Meta-journaling in NVMe**: NVMe 표준 내부의 원자적 쓰기(Atomic Write) 기능을 활용하여, 소프트웨어 저널링 오버헤드를 하드웨어가 대신 짊어지는 기술 보편화.
- **Log-structured Merge Tree (LSM) 기반 파일 시스템**: 파일 시스템 자체가 하나의 거대한 로그가 되어, 쓰기 증폭을 줄이고 저널링 성능을 극대화하는 신개념 FS 연구 활발.

### 참고 표준
- **POSIX `write()`**: 저널링 계층과 상호작용하는 표준 I/O 함수.
- **Linux Kernel JBD2 (Journaling Block Device)**: 리눅스 커널의 공용 저널링 엔진 표준.

- **📢 섹션 요약 비유**: 저널링은 컴퓨터가 "기억"을 잃지 않기 위해 매 순간 쓰는 "디지털 일기장"이며, 이 일기장이 있기에 우리는 안심하고 거대한 데이터를 다룰 수 있습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 저널링은 장난감을 정리하기 전에 **"어디에 무엇을 넣을지 수첩에 미리 적어두는 것"**과 같아요.
2. 장난감을 치우다가 갑자기 엄마가 불러서 나가게 되어도, 나중에 수첩만 보면 **어디까지 치웠는지 금방 알 수** 있죠.
3. 이 수첩 덕분에 우리는 방이 어질러져 있어도 **금방 깨끗하게** 다시 정리할 수 있는 거랍니다!
