# 스래싱 (Thrashing, CPU 이용률 저하)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스래싱 (Thrashing)은 프로세스가 실제 유효한 연산을 수행하는 시간보다 페이지 부재 (Page Fault)로 인한 페이지 교체 작업에 더 많은 CPU 자원을 소모하여, 시스템 전체의 처리량 (Throughput)이 급격히 떨어지는 성능 붕괴 현상이다.
> 2. **가치**: 다중 프로그래밍의 정도 (Degree of Multiprogramming)와 시스템 가용성 사이의 임계점 (Threshold)을 정의하며, 이를 방지하기 위해 워킹 셋 (Working Set) 보장 및 페이지 부재 빈도 (PFF, Page Fault Frequency) 제어 등 동적인 자원 할당 전략 수립의 근거가 된다.
> 3. **융합**: 가상 메모리의 스왑 (Swap) 병목, 디스크 I/O 서브시스템의 지연 시간 (Latency), 그리고 현대 클라우드 환경의 오토 스케일링 (Auto Scaling) 및 컨테이너 자원 제약 (Resource Limit) 정책과 밀접하게 연계되어 시스템 안정성의 핵심 지표로 활용된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 스래싱 (Thrashing)은 가상 메모리 시스템에서 프로세스들이 필요로 하는 최소한의 페이지 프레임 (Frame) 세트를 확보하지 못해, 빈번한 페이지 폴트와 교체 작업이 반복되면서 CPU 이용률이 바닥으로 추락하는 현상이다.

- **💡 비유**: 이는 주방에 요리사(프로세스)가 너무 많아서 요리 기구(프레임)가 부족해지자, 요리사들이 실제 요리는 안 하고 서로 기구를 뺏고 돌려주느라 시간만 다 보내고, 정작 음식은 하나도 안 나오는 상황과 같다. 홀(사용자)은 주문이 안 나와서 난리가 났는데, 주방장(OS)은 "요리사들이 왜 이렇게 조용해? 더 뽑아야겠군!"이라며 사람을 더 투입하는 악순환에 빠진 셈이다.

- **발생 배경**:
  1. **다중 프로그래밍의 과욕**: CPU 이용률을 높이기 위해 프로세스 수를 무분별하게 늘리다 보면, 각 프로세스에 할당되는 프레임 수가 줄어든다.
  2. **지역성 (Locality)의 붕괴**: 할당된 프레임 수가 프로세스의 현재 작업 집합인 워킹 셋 (Working Set)보다 작아지면, 페이지를 올리자마자 바로 쫓겨나야 하는 상황이 발생한다.
  3. **OS의 오판과 악순환**: CPU 이용률이 떨어지면 운영체제의 장기 스케줄러 (Long-term Scheduler)는 시스템이 한가하다고 판단하여 프로세스를 더 추가하게 되고, 이는 메모리 부족을 더욱 심화시킨다.

```text

  │                 스래싱 발생 곡선 (CPU Utilization Curve)           │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   CPU Utilization (%)                                            │
  │      ▲                                                           │
  │      │             (A) 최적 지점                                  │
  │      │                  ●                                        │
  │      │                ╱   ╲                                      │
  │      │              ╱       ╲                                    │
  │      │            ╱           ╲  (B) 스래싱 발생 구간               │

  │      │        ╱                   ╲                              │
  │      │      ╱                       ╲                            │

  │   - (A) 지점: CPU 활용도가 극대화되는 임계치                       │
  │   - (B) 지점: 프로세스가 추가될수록 CPU 이용률이 급감하는 역설        │
```

**[다이어그램 해설]** 가로축은 다중 프로그래밍의 정도(동시 실행 프로세스 수)를 의미한다. 초기에는 프로세스가 많아질수록 CPU가 쉬지 않고 일하므로 이용률이 상승한다. 그러나 메모리 한계치에 도달하여 각 프로세스가 '워킹 셋' 이하의 프레임을 받게 되면, 모든 프로세스가 페이지 폴트로 인한 I/O 대기 상태에 빠지며 CPU 이용률은 급격히 수직 낙하한다.

- **📢 섹션 요약 비유**: "배고픈 아이들이 많아질수록 밥그릇 싸움이 치열해져서, 정작 밥은 아무도 못 먹게 되는 굶주림의 역설"과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 스래싱의 악순환 메커니즘 (Vicious Cycle)

스래싱은 단순한 메모리 부족을 넘어 시스템 전체의 피드백 루프가 고장 나는 과정이다.

1. **Step 1: 메모리 포화**: 다수의 프로세스가 실행되며 가용 프레임이 고갈됨.
2. **Step 2: 빈번한 페이지 폴트**: 프로세스들이 자신의 페이지를 계속 뺏고 뺏기는 과정 발생.
3. **Step 3: I/O 대기 행렬 증가**: 페이지를 읽어오기 위해 디스크 컨트롤러 대기 큐에 프로세스들이 줄을 섬 (Blocked State).
4. **Step 4: CPU Idle 시간 증가**: 연산을 수행할 프로세스가 준비 큐 (Ready Queue)에 없으므로 CPU 이용률 급감.
5. **Step 5: 스케줄러의 착각**: OS가 "CPU가 노니까 일을 더 시켜야겠다"며 새로운 프로세스 적재.
6. **Step 6: 완전 마비**: 메모리 부족 가속화 및 시스템 중단 수준의 지연 발생.

### 스래싱 방지 및 제어 전략

현대 OS는 스래싱을 방지하기 위해 각 프로세스가 필요로 하는 최소한의 프레임을 동적으로 계산하고 보장한다.

| 전략 기법 | 핵심 메커니즘 | 특징 | 비유 |
|:---|:---|:---|:---|
| **워킹 셋 모델 (Working Set Model)** | 일정 시간($\Delta$) 동안 참조된 페이지 집합을 메모리에 상주 보장 | 지역성 기반, 스래싱을 예방하는 가장 강력한 수단 | 필수 생존 자금 확보 |
| **페이지 부재 빈도 (PFF)** | 폴트율의 상한/하한을 정해 프레임을 늘리거나 줄임 | 직접적인 폴트 모니터링 기반 동적 조절 | 성적에 따른 용돈 조절 |
| **지역 교체 정책 (Local Replacement)** | 한 프로세스의 폴트가 다른 프로세스의 프레임을 뺏지 못하게 격리 | 스래싱의 확산(전염)을 막으나 시스템 전체 효율 저하 위험 | 각자도생(격리) |
| **강제 중단 (Process Swapping)** | 스래싱 감지 시 일부 프로세스를 통째로 스왑 아웃하여 프레임 확보 | 긴급 처방, 중단된 프로세스는 나중에 재개 | 일시적 휴업 선고 |

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │              워킹 셋 (Working Set) 기반 프레임 할당 관리           │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │    Time (t) ───────▶                                             │
  │    [ 참조 스트림: 2 6 1 5 7 7 7 7 5 1 ]                          │
  │                                                                  │
  │    Δ (Window Size) = 5 일 때:                                    │
  │    t1: {2, 6, 1, 5, 7}  → Working Set Size (WSS) = 5             │
  │    t2: {7, 7, 7, 5, 1}  → Working Set Size (WSS) = 3             │
  │                                                                  │
  │    [정책]                                                        │
  │    Σ WSS_i (모든 프로세스의 워킹 셋 합) > 물리 메모리 크기         │
  │    => 이 시점에 스래싱 위험! 일부 프로세스 실행 일시 중단 필요.      │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 워킹 셋 모델은 시간의 흐름에 따라 변하는 프로세스의 '지역성'을 추적한다. 윈도우 크기($\Delta$) 내에 등장한 서로 다른 페이지의 개수가 해당 프로세스가 지금 당장 죽지 않고 일하기 위해 필요한 최소한의 프레임 수(WSS)다. 시스템의 모든 WSS 합이 전체 메모리보다 크면 OS는 스래싱이 올 것임을 직감하고 일부 프로세스를 희생시킨다.

- **📢 섹션 요약 비유**: "아무리 배가 고파도 각자에게 최소한의 밥 한 공기(워킹 셋)는 보장해줘야지, 조금씩 나눠주다가는 모두가 굶어 죽는다"는 관리 원칙입니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 워킹 셋 모델 vs 페이지 부재 빈도 (PFF)

| 비교 항목 | 워킹 셋 모델 (Working Set) | 페이지 부재 빈도 (PFF) |
|:---|:---|:---|
| **제어 관점** | 참조 패턴 (과거 이력) 기반 | 실제 결과 (페이지 폴트 율) 기반 |
| **장점** | 스래싱 예방 효과가 탁월함 | 구현이 비교적 단순하고 오버헤드가 적음 |
| **단점** | 윈도우 크기($\Delta$) 설정이 어렵고 추적 비용 발생 | 폴트가 이미 발생한 후의 사후 대응 성격 |
| **동작 방식** | $D = \sum WSS_i > m$ 이면 중단 | 상한선 초과 시 프레임 추가, 하한선 미달 시 회수 |

### 비교 2: 하드웨어 성능과 스래싱

- **SSD vs HDD**: 과거 HDD 시절 스래싱은 시스템의 완전한 멈춤(Hard Lock)을 의미했다. 디스크 헤드가 미친 듯이 움직이는 소리가 들릴 정도였다. 하지만 SSD와 NVMe의 보편화로 페이지 폴트 레이턴시가 획기적으로 줄어들면서, 현대 시스템에서는 스래싱이 발생해도 '멈춤'보다는 '느려짐'으로 체감되는 경우가 많다.
- **ZRAM/ZSWAP**: 메모리 내의 페이지를 압축하여 보관하는 기술로, 스래싱이 발생하기 직전의 '임계 영역'을 뒤로 미루는 완충 역할을 한다.

- **과목 융합 관점 (컴퓨터 구조)**: CPU의 캐시 미스 (Cache Miss)가 잦아져 성능이 떨어지는 것과 가상 메모리의 스래싱은 구조적으로 유사하다. 하지만 스래싱은 소프트웨어인 OS의 스케줄러와 하드웨어인 디스크 I/O가 복합적으로 얽혀 '악순환 피드백 루프'를 만든다는 점에서 훨씬 치명적이다.

- **📢 섹션 요약 비유**: 사고가 나기 전에 차간 거리를 유지하는 것(워킹 셋)과 사고가 날 뻔할 때(폴트 폭증) 브레이크를 밟는 것(PFF)의 차이입니다.

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 자바 어플리케이션 서버 (WAS) 장애**: 특정 시간대에 사용자가 몰리며 JVM의 Heap 영역에서 빈번한 Full GC와 함께 페이지 폴트가 급증함. CPU 이용률은 낮은데 서비스 응답이 안 옴. 원인은 물리 메모리 부족으로 인한 OS 레벨의 스래싱이었음. 해결책으로 인스턴스 당 메모리 할당량을 늘리고 동시 처리량(Thread Pool)을 제한하여 다중 프로그래밍 정도를 강제 조절함.

2. **시나리오 — 클라우드 환경의 OOM Killer**: 리눅스 서버에서 메모리가 고갈되어 스래싱 징후가 보이면, OS는 시스템 전체의 붕괴를 막기 위해 가장 메모리를 많이 쓰고 '나쁜' 프로세스를 골라 강제 종료시킴 (OOM Killer). 이는 최악의 스래싱을 막기 위한 최후의 방어 수단임.

3. **시나리오 — 가상화 솔루션의 메모리 벌루닝 (Ballooning)**: 하이퍼바이저가 게스트 VM의 메모리를 회수하려 할 때, 게스트 내부에서 워킹 셋을 건드리면 게스트 OS가 스래싱에 빠질 수 있음. 이를 방지하기 위해 하이퍼바이저는 게스트의 페이징 통계를 실시간 모니터링함.

### 도입 체크리스트 (성능 모니터링 시)
- **Major Page Fault 수치**: 초당 발생하는 메이저 폴트가 급증하고 있는가?
- **CPU I/O Wait**: CPU가 연산이 아닌 I/O 대기에 보내는 시간이 30%를 넘는가?
- **Swap In/Out 활동**: 스왑 영역의 읽기/쓰기가 지속적으로 발생하는가? (일시적인 것은 정상)

### 안티패턴
- **무조건적인 오토 스케일링**: CPU 이용률이 낮다고 해서 무조건 서버 인스턴스를 줄이거나, 반대로 이용률이 낮으니 프로세스를 더 띄우는 것. 스래싱 상황에서 CPU 이용률 저하는 '한가함'이 아니라 '고통'의 신호임.
- **스왑 영역의 과도한 확장**: 메모리가 부족하다고 스왑 공간을 무한정 늘리면, 시스템은 죽지 않고 스래싱 상태로 영원히 버티게 되어 사용자에게는 '무응답'보다 나쁜 경험을 줌. (해결: 적절한 시점에 프로세스 종료/재시작)

- **📢 섹션 요약 비유**: 환자가 의식이 없다고(CPU Idle) 억지로 깨우는(프로세스 추가) 것이 아니라, 영양제(메모리)를 주거나 휴식(프로세스 중단)을 시켜야 회복될 수 있습니다.

- **CXL (Compute Express Link) 기반 메모리 풀링**: 단일 서버의 메모리 한계를 넘어, 네트워크를 통해 공유 메모리 풀에서 프레임을 빌려옴으로써 물리적 스래싱 상황을 원천적으로 회피함.

### 👶 어린이를 위한 3줄 비유 설명
1. 스래싱은 **"장난감 정리하느라 놀 시간이 없는 상태"**예요. 친구들을 너무 많이 초대해서 장난감이 부족해지자, 서로 장난감을 뺏고 돌려주느라 정작 놀지는 못하는 거죠.
2. 밖에서 보면 모두 바빠 보이지만, 실제로는 아무도 재미있게 놀지 못하고 시간만 낭비하고 있는 거예요.
3. 이럴 때는 **"친구들 중 몇 명은 잠시 다른 방에 가서 쉬게 하고"**, 남은 친구들이 장난감을 충분히 가지고 놀 수 있게 해줘야 다시 즐거운 놀이 시간이 시작된답니다!
