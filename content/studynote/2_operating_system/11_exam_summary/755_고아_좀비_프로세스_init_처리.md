# 고아 및 좀비 프로세스와 init 처리 (Orphan, Zombie & init)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 고아(Orphan) 프로세스는 실행 중 부모 프로세스가 먼저 종료되어 보호자를 잃은 프로세스이며, 좀비(Zombie) 프로세스는 실행을 마쳤으나 부모가 종료 상태를 수거(`wait`)하지 않아 시스템 장부(PCB)에만 유령처럼 남은 상태다.
> 2. **가치**: 한정된 시스템 자원인 프로세스 식별자(PID) 고갈을 방지하고 프로세스 간의 엄격한 계층적 생명주기 관리를 통해 운영체제의 안정성과 자원 정합성을 보장한다.
> 3. **융합**: 유닉스 시스템의 근간인 `fork-wait-exit` 메커니즘의 산물이며, 시스템의 뿌리인 `init`(PID 1) 프로세스의 '입양(Adoption)' 정책 및 `SIGCHLD` 시그널 핸들링 기술과 밀접하게 연계된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 프로세스는 혼자 태어나지 않는다. 항상 부모가 자식을 낳는 계층 구조를 가진다. 이때 부모가 자식을 책임지지 못하거나(좀비), 부모가 먼저 세상을 떠나는(고아) 비정상적인 가족 관계가 발생할 수 있는데, 이를 운영체제가 어떻게 중재하느냐가 핵심이다.

- **💡 비유**: 좀비는 "다 쓴 물건을 버리지 않고 영수증만 챙겨두는 게으른 주인" 때문에 서랍(PID 리스트) 공간만 차지하고 있는 상태다. 고아는 "부모님이 갑자기 출장을 가셔서 혼자 남겨진 아이"와 같은데, 이때 "동네 이장님(init)"이 나타나서 아이를 자기 집으로 데려가 안전하게 보살펴주는 것과 같다.

- **등장 배경**:
  1. **비동기적 종료**: 부모와 자식은 서로 다른 속도로 일을 하므로, 누가 먼저 끝날지 예측할 수 없는 전산학적 특성이 존재한다.
  2. **자원 회수의 책임**: 자식이 끝났을 때 그 자식이 쓴 메모리는 OS가 바로 치우지만, "왜 끝났는지(Exit Code)"는 부모가 읽어갈 때까지 OS가 대신 보관해주어야 한다.
  3. **PID 유한성**: 컴퓨터가 발급할 수 있는 번호표(PID)는 한계가 있으므로, 죽은 프로세스의 번호를 빨리 회수하여 재사용해야 한다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              정상 종료 vs 좀비 vs 고아 프로세스 비교                 │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ 1. Normal ]      [ 2. Zombie ]          [ 3. Orphan ]       │
  │                                                                 │
  │   Parent (wait)      Parent (Busy/Ignore)   Parent (Exit first) │
  │      │                  │                      │ (Dead)         │
  │      ▼ (exit)           ▼ (exit)               ▼                │
  │   Child (Finish)     Child (Zombie)         Child (Orphan)      │
  │      │                  │                      │                │
  │      └─▶ Cleaned Up     └─▶ <defunct>          └─▶ Adopted by   │
  │                                                    init (PID 1) │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 정상적인 상황에서는 자식이 죽으면 부모가 `wait()`를 통해 즉시 치운다. 하지만 부모가 너무 바쁘거나 코딩 실수로 자식을 무시하면 '좀비'가 된다. 부모가 자식보다 먼저 죽으면 자식은 즉시 시스템의 최고 조상인 `init` 프로세스에게 입양되어 '고아' 상태를 탈출한다.

- **📢 섹션 요약 비유**: "끝난 일을 정리하지 않으면 쓰레기(좀비)가 쌓이고, 주인이 없는 일(고아)은 국가(init)가 관리해야 시스템이 유지됩니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 좀비 프로세스 (Zombie Process): "죽었지만 죽지 않은 자"

좀비 프로세스는 실제 실행 코드나 메모리는 모두 해제된 상태다. 오직 **프로세스 제어 블록 (PCB)**만 시스템에 남아있다.

| 항목 | 상세 내용 | 시스템 영향 |
|:---|:---|:---|
| **발생 원인** | 자식이 `exit()` 호출 후 부모가 `wait()` 시스템 콜을 호출하지 않음 | 불필요한 커널 메모리 점유 |
| **상태 표시** | `ps` 명령어 실행 시 `<defunct>` 또는 상태 코드 `Z` | 관리적 혼란 유발 |
| **위험성** | 좀비가 많아지면 사용 가능한 **PID가 고갈**되어 새 프로세스 생성 불가 | 시스템 마비 (DoS 유발) |
| **해결책** | 부모 프로세스에 `SIGCHLD` 시그널을 보내 `wait()`를 강제하거나 부모 종료 | 프로세스 재기동 필요 |

---

### 고아 프로세스 (Orphan Process): "새로운 보호자 init"

고아 프로세스는 시스템에 해를 끼치지는 않지만, 종료 시 수거할 주체가 없다는 잠재적 위험이 있다.

1. **Detection**: 자식 프로세스가 살아있는데 부모의 PID가 사라짐을 커널이 감지.
2. **Adoption (입양)**: 커널은 해당 고아의 부모 PID를 **1번 (init 또는 systemd)**으로 강제 변경.
3. **Reaping**: 나중에 고아가 일을 마치고 종료하면, 새 부모인 `init`은 24시간 `wait()` 대기 중이므로 즉시 자원을 수거함.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 init 프로세스의 수거 메커니즘 (Reaping)             │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ init Process ]  <───  (Wait Loop) ────────────────┐          │
  │      (PID 1)                                          │          │
  │          │                                            │          │
  │          └─▶ [ Adopted Orphan 1 ] ──▶ (Finish) ──▶ [SIGCHLD] ──┘  │
  │          └─▶ [ Adopted Orphan 2 ] ──▶ (Finish) ──▶ [SIGCHLD] ──┘  │
  │                                                                  │
  │   [결론] init는 모든 고아 자식들의 장례식을 대신 치러주는 수호자임      │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** `init` 프로세스는 시스템 부팅 시 가장 먼저 실행되어 무한 루프를 돌며 자식들의 종료 시그널을 기다린다. 어떠한 이유로든 부모를 잃은 프로세스들은 `init`의 품으로 들어가 안정적으로 생을 마감할 수 있게 된다.

- **📢 섹션 요약 비유**: "버려진 쓰레기(좀비)는 청소부(부모)가 직접 치워야 하지만, 길 잃은 아이(고아)는 파출소(init)가 안전하게 보호하다가 나중에 집(자원 회수)으로 보내줍니다."

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 좀비 vs 고아 (명확한 대조)

| 비교 항목 | 좀비 프로세스 (Zombie) | 고아 프로세스 (Orphan) |
|:---|:---|:---|
| **프로세스 상태** | 종료됨 (Terminated) | **실행 중 (Running)** |
| **자원 점유** | PCB (Process ID)만 점유 | CPU, 메모리 등 모든 자원 사용 |
| **부모의 상태** | 살아있으나 무관심함 | **이미 종료됨 (사망)** |
| **시스템 유해성** | **높음** (PID 고갈 위험) | 낮음 (정상 프로세스와 동일) |
| **처리 방법** | 부모에게 시그널 전송 or 부모 종료 | `init` 프로세스가 자동 관리 |

### 비교 2: `wait()`와 `waitpid()`의 전략적 선택

- **`wait()`**: 자식 중 아무나 하나 죽을 때까지 부모를 잠재움 (Blocking).
- **`waitpid()`**: 특정 자식을 지목하거나, `WNOHANG` 옵션을 통해 자식이 안 죽었으면 기다리지 않고 자기 일을 계속함 (Non-blocking). 현대적인 서버 프로그램은 대부분 `waitpid()`를 선호함.

- **과목 융합 관점 (네트워크)**: 웹 서버(Apache 등)는 클라이언트 요청마다 자식 프로세스를 만든다. 이때 자식들이 좀비가 되면 서버는 금방 뻗어버린다. 따라서 웹 서버 개발 시 **`SIGCHLD` 핸들러** 내에서 비블로킹 `waitpid`를 무한 루프로 돌려 모든 좀비를 즉시 소탕하는 기법이 필수적이다.

- **📢 섹션 요약 비유**: "죽어서도 자리를 차지하는 귀신(좀비)은 굿(wait)을 해서 보내야 하고, 갈 곳 없는 방랑자(고아)는 숙소(init)를 마련해줘야 합니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 대규모 배치 작업 중 PID 고갈**: 수천 개의 이미지 프로세싱 작업을 `fork`로 실행했는데, 어느 순간부터 `fork: Resource temporarily unavailable` 에러가 발생함. `top`으로 확인하니 좀비(`defunct`)가 수천 개임. 해결책으로 부모 코드에 **`signal(SIGCHLD, SIG_IGN)`**을 추가하여 커널이 자식 종료 시 자동으로 자원을 회수하게 하거나, 적절한 `wait` 로직을 삽입하여 시스템을 정상화함.

2. **시나리오 — 데몬(Daemon) 프로세스 생성**: 백그라운드에서 영원히 돌아가는 서비스를 만들고 싶음. 기술사적 판단으로 **'Double Fork'** 기법을 권고한다. 첫 번째 자식을 낳고 부모가 즉시 죽으면 자식은 고아가 되어 `init`에 입양된다. 다시 이 자식이 손자를 낳고 죽으면 손자는 완전한 고아가 되어 터미널 제어에서 벗어난 진정한 백그라운드 데몬이 된다.

3. **시나리오 — 좀비 킬러 (Zombie Killer)**: 부모 프로세스가 루프에 빠져 좀비를 안 치우고 있음. `kill -9 <좀비PID>`를 날려도 좀비는 죽지 않음 (이미 죽었으니까). 해결책으로 **부모 프로세스를 강제 종료**시킨다. 그러면 좀비들이 '고아'가 되어 `init`에게 넘어가고, `init`이 즉시 그들을 수거하여 소멸시킨다.

### 도입 체크리스트 (Developer Checklist)
- **Signal Handling**: `SIGCHLD` 시그널에 대해 무시(`SIG_IGN`)하거나 핸들러를 등록했는가?
- **Error Path**: 함수 실패 시 `exit()` 전후로 자식 관리가 누락되지는 않았는가?
- **Init System**: 현대의 `systemd` 환경에서 `KillMode` 설정이 고아 프로세스 처리에 어떤 영향을 주는지 이해하고 있는가?

### 안티패턴
- **`wait()` 없이 무한 루프**: 자식을 만들고 무조건 `sleep()`만 하는 부모. 좀비 제조기의 전형.
- **좀비에 `kill -9` 남발**: 좀비는 명령어를 듣지 않는다. 원인은 부모에게 있음을 간과하는 행위.

- **📢 섹션 요약 비유**: "자식을 낳는 것(fork)보다 더 중요한 것은 자식의 뒷마무리(wait)를 책임지는 부모의 도리(Clean-up)입니다."

| **리소스 효율** | 불필요한 커널 메모리(PCB) 반환으로 최적화 유지 | 미세 오버헤드 제거 |
| **운영 가시성** | 깨끗한 프로세스 목록 유지로 모니터링 정확도 향상 | 디버깅 생산성 증대 |
| **보안성** | 고아 프로세스를 이용한 비인가 백그라운드 실행 방지 | 보안 무결성 강화 |

### 미래 전망
- **Subreaper (PR_SET_CHILD_SUBREAPER)**: 모든 고아가 `init`으로 가는 대신, 특정 중간 부모가 고아들을 거두도록 지정하는 기능이 컨테이너 환경에서 널리 쓰일 것임.
- **Serverless & Lightweight Isolation**: 프로세스 개념이 아닌 WASM이나 극소형 컨테이너 단위에서는 커널이 생명주기를 강제 관리하여 좀비/고아 개념 자체가 사라지는 아키텍처로 진화 중.

### 👶 어린이를 위한 3줄 비유 설명
1. 좀비는 장난감을 다 놀고 상자에 넣어야 하는데, **"상자 뚜껑을 안 닫아서"** 거실 한복판에 굴러다니는 장난감과 같아요. 다른 장난감을 놓을 자리를 뺏고 있죠.
2. 고아는 같이 놀던 형아가 갑자기 집에 가버려서 **"혼자 남겨진 동생"**이에요. 울고 있으면 안 되니까 **"착한 경찰관 아저씨(init)"**가 오셔서 보호해주신답니다.
3. 경찰관 아저씨는 동생이 다 놀 때까지 기다렸다가, **안전하게 집으로(자원 회수)** 보내주시는 고마운 분이랍니다!
