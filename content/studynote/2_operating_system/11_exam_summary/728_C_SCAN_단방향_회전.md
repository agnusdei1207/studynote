# C-SCAN (Circular-SCAN) 단방향 회전

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: C-SCAN (Circular SCAN)은 디스크 헤드가 한 방향으로만 이동하며 경로상의 모든 요청을 처리하고, 끝에 도달하면 서비스를 중단한 채 반대편 끝(시작점)으로 즉시 복귀하여 다시 같은 방향으로 서비스를 시작하는 순환형 스케줄링 기법이다.
> 2. **가치**: 일반 SCAN의 고질적 문제인 "가운데 트랙 편중 현상"을 해결하여, 디스크 전체 영역에 걸쳐 대기 시간의 표준 편차를 최소화하고 통계적으로 균등한 응답 시간 (Uniform Wait Time)을 보장한다.
> 3. **융합**: 고가용성 데이터베이스 서버나 멀티미디어 스트리밍 시스템과 같이 일관된 데이터 전송률과 예측 가능한 I/O 레이턴시가 요구되는 엔터프라이즈 환경의 스토리지 제어 로직에 필수적으로 활용된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: C-SCAN은 디스크 트랙을 하나의 거대한 고리(Circular Queue)처럼 간주하여, 헤드가 한 방향으로만 순환하며 요청을 처리하는 알고리즘이다. 역방향 이동 시에는 요청을 처리하지 않고 오직 헤드의 위치를 리셋하는 데에만 집중한다.

- **💡 비유**: 이는 "편도 노선만 운행하는 셔틀버스"와 같다. 기점에서 종점까지 가면서 정류장마다 사람을 태우지만, 종점에 도착하면 승객을 태우지 않고 고속도로를 타고 즉시 기점으로 돌아와 다시 운행을 시작하는 방식이다. 이 덕분에 기점 근처에 있는 사람도 종점 근처 사람과 비슷한 주기로 버스를 탈 수 있다.

- **등장 배경**:
  1. **SCAN의 불공평성**: SCAN은 왕복 운행을 하므로, 가운데 트랙은 왕복할 때마다 두 번씩 지나가게 되어 양 끝단 트랙보다 서비스 기회가 2배 더 많다. (중간 편중)
  2. **응답 시간의 불확실성 제거**: 특정 영역의 대기 시간이 유독 길어지는 현상을 방지하여 시스템의 성능 예측 가능성을 높여야 했다.
  3. **대규모 데이터 순차 처리**: 로그 파일이나 거대 바이너리 데이터를 일정한 속도로 읽어들여야 하는 워크로드의 최적화 요구가 커졌다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │             C-SCAN 스케줄링 헤드 이동 경로 (0 ~ 199 트랙)           │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   요청 대기열: 98, 183, 37, 122, 14, 124, 65, 67                │
  │   현재 헤드 위치: 53 , 진행 방향: 증가 방향 (오른쪽)               │
  │                                                                 │
  │   Track: 0   14   37      53  65  67      98     122 124     183 199│
  │          │    │    │       │   │   │       │       │   │       │  │ │
  │   Start: └─────────────────●                                      │ │
  │   Service:                 └─▶ 65 ──▶ 67 ──▶ 98 ──▶ 122 ──▶ 124 ──▶ 183 ──▶ 199 │
  │   Return : 0 ◀═══════════════════════════════════════════════════┘ │
  │            (복귀 시 서비스 안 함)                                   │
  │   Repeat : └─▶ 14 ──▶ 37 ──▶ (다음 요청으로...)                    │
  │                                                                 │
  │   [이동 순서]                                                    │
  │   53 → 65 → 67 → 98 → 122 → 124 → 183 → 199(끝) → 0(복귀) → 14 → 37  │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 헤드가 53번에서 증가하여 183번을 지나 끝(199)까지 가는 동안 모든 요청을 처리한다. 이후 방향을 틀어 내려올 때 37, 14번을 처리하는 SCAN과 달리, C-SCAN은 아무 서비스 없이 0번으로 점프(Seek)한다. 다시 0번부터 올라가며 비로소 14번과 37번을 순서대로 서비스한다.

- **📢 섹션 요약 비유**: "뒤로 걸을 때는 주변을 보지 않고 앞만 보고 걷는 일방통행의 정석"과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### C-SCAN의 성능 결정 요인

C-SCAN의 핵심은 '공정성'과 '복귀 속도'의 트레이드오프에 있다.

| 분석 항목 | 상세 메커니즘 | 기술적 영향 | 비유 |
|:---|:---|:---|:---|
| **서비스 구간** | 한 방향(보통 저트랙→고트랙) 이동 | 트랙 간 이동 간격이 일정하게 유지됨 | 정기 열차 운행 |
| **즉시 복귀 (Fast Return)** | 끝단 도달 시 최대 탐색 거리를 한 번에 이동 | 실제 데이터 전송 없이 물리적 헤드 이동만 발생 | 빈 차로 회차 |
| **대기 시간 분산** | 모든 트랙의 최대 대기 시간이 $(T_{service} + T_{return})$으로 수렴 | 응답 시간의 표준 편차가 매우 낮음 | 균등한 배차 간격 |
| **처리량 (Throughput)** | SCAN 대비 복귀 오버헤드로 인해 약간 낮음 | 부하가 많을수록 SCAN보다 안정적인 처리량 제공 | 안정적인 운송량 |

### 서비스 공정성 (Service Fairness)의 수리적 모델

SCAN과 C-SCAN의 대기 시간을 비교 분석하면 C-SCAN이 왜 공정한지 알 수 있다.

1. **SCAN의 대기 시간**:
   - 끝단 트랙: 헤드가 반대편 끝까지 갔다가 돌아와야 하므로 대기 시간 최대 $2 \times (\text{최대 트랙 수})$.
   - 중앙 트랙: 왕복 중 어느 쪽에서든 헤드를 만날 확률이 높아 대기 시간 기대값 낮음.
2. **C-SCAN의 대기 시간**:
   - 모든 트랙: 헤드가 한 바퀴 돌고 오는 시간 $\approx (\text{최대 트랙 이동 시간} + \text{복귀 시간})$으로 거의 일정함.
   - 결과: 특정 트랙이 소외되거나 특별 대우를 받는 일이 사라짐.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 알고리즘별 대기 시간 분포 (Variance)                │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   Wait Time Probability                                          │
  │      ▲                                                           │
  │      │          C-SCAN (좁고 높은 분포 - 예측 가능)                │
  │      │             _|_                                           │
  │      │            /   \                                          │
  │      │           |     |                                         │
  │      │      SCAN (넓고 낮은 분포 - 편차 큼)                         │
  │      │       _____/     \_____                                   │
  │      │      /                 \                                  │
  │      └─────/───────────────────\──────────────────▶              │
  │                                           Response Time          │
  │                                                                  │
  │   [결론]                                                         │
  │   - C-SCAN은 '최고의 속도'보다 '일관된 응답성'을 중시하는 모델       │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SCAN은 운이 좋으면 아주 빠르고, 운이 나쁘면 아주 느리다. 반면 C-SCAN은 대부분의 요청이 비슷한 시간 내에 처리된다. 엔지니어링 관점에서 '예측 불가능한 고성능'보다 '예측 가능한 적정 성능'이 시스템 설계에 훨씬 유리하며, 이것이 C-SCAN이 사랑받는 이유다.

- **📢 섹션 요약 비유**: "누구는 1분, 누구는 1시간 기다리는 로또 배차보다, 모두가 10분씩 기다리는 정기 배차가 평화로운 사회를 만든다"는 원리입니다.

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: SCAN vs C-SCAN vs LOOK vs C-LOOK

| 비교 항목 | SCAN | C-SCAN | LOOK | C-LOOK |
|:---|:---|:---|:---|:---|
| **이동 범위** | 전체 트랙 끝까지 | 전체 트랙 끝까지 | 마지막 요청까지만 | 마지막 요청까지만 |
| **서비스 방향** | 양방향 | **단방향** | 양방향 | **단방향** |
| **공정성** | 보통 (가운데 편중) | **최고 (전 영역 균등)** | 보통 | 좋음 |
| **효율성** | 보통 | 낮음 (복귀 비용) | 높음 | 보통 |
| **실무 적용** | 기본 스케줄러 | **공정성 우선 시스템** | 실무적 표준 | **고효율+공정성** |

### 비교 2: 하드웨어적 관점의 복귀 (Full-stroke Seek)

- **Seek Time 특성**: 헤드가 1트랙 이동하는 시간보다 100트랙 한 번에 이동하는 시간이 상대적으로 짧다. (가속도 때문)
- **C-SCAN의 복귀 전략**: 끝에서 끝으로 한 번에 점프하는 복귀 시간은 생각보다 크지 않으며, 이를 통해 얻는 서비스 일관성의 이득이 훨씬 크다.

- **과목 융합 관점 (자료구조)**: C-SCAN은 자료구조의 **환형 연결 리스트 (Circular Linked List)**를 순회하는 포인터의 움직임과 정확히 일치한다. 마지막 노드의 `next`가 헤드를 가리키듯, 디스크의 마지막 실린더가 다시 0번 실린더와 논리적으로 연결된 구조다.

- **📢 섹션 요약 비유**: "끝까지 가서 멈추는 것(SCAN)"보다 "끝에서 다시 처음으로 점프하는 것(C-SCAN)"이 논리적으로는 훨씬 완벽한 원형을 이룹니다.

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 대용량 로그 수집 서버**: 수천 개의 서비스에서 발생하는 로그를 디스크에 순차적으로 기록해야 함. 특정 시점의 지연(Jitter)이 발생하면 로그 유실 위험이 있음. C-SCAN 기반 스케줄러를 사용하여 어떤 로그 파일이든 일정한 쓰기 레이턴시를 보장함으로써 버퍼 오버플로우를 예방함.

2. **시나리오 — 클라우드 블록 스토리지 (EBS 등) 설계**: 여러 테넌트(Tenant)가 물리 디스크를 공유하는 환경에서, 특정 사용자의 데이터가 디스크 안쪽에 있다고 해서 더 빠른 서비스를 받으면 불공평함. C-SCAN 정책을 적용하여 모든 사용자에게 물리적 위치와 상관없는 '균등한 IOPS'를 제공하는 과금 모델을 수립함.

3. **시나리오 — 현대적 LOOK 기반 최적화**: 실무에서는 C-SCAN 자체보다 **C-LOOK**을 더 많이 쓴다. 0번이나 199번 끝까지 갈 필요 없이, 진행 방향의 마지막 요청만 처리하고 즉시 반대편의 가장 먼 요청으로 건너뛰어 C-SCAN의 철학을 유지하면서도 탐색 거리를 20~30% 절약한다.

### 도입 체크리스트
- **Latency-Sensitive 여부**: 응답 시간의 '평균'이 중요한가, 아니면 '최대치(P99)'가 중요한가? (P99 제어에는 C-SCAN 유리)
- **워크로드의 편향성**: 요청이 특정 트랙 영역에 몰려 있는가, 아니면 골고루 분산되어 있는가? (분산 시 C-SCAN 효과 극대화)
- **스토리지 계층**: 하이퍼바이저 수준인가, 게스트 OS 수준인가, 디스크 펌웨어 수준인가?

### 안티패턴
- **매우 낮은 부하에서의 C-SCAN**: 요청이 1~2개뿐인데 굳이 단방향 스캔과 복귀를 반복하는 것은 불필요한 Seek만 유발함. (해결: 부하에 따라 FCFS와 동적으로 스위칭)
- **복귀 시간 과소평가**: 하드웨어 특성에 따라 복귀 시간이 너무 길면(예: 구형 테이프 드라이브), C-SCAN은 최악의 선택이 됨. (해결: 매체 특성 사전 파악)

- **📢 섹션 요약 비유**: "모두에게 평등한 혜택을 주는 시스템이 때로는 전체 평균을 깎아 먹을 수 있지만, 사회(시스템)의 안정성을 위해서는 반드시 필요하다"는 판단이 엔지니어의 몫입니다.

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (SCAN 대비)

| 구분 | SCAN 방식 | C-SCAN 방식 | 개선 효과 |
|:---|:---|:---|:---|
| **응답 시간 편차** | 높음 (위치별 불공평) | **매우 낮음 (균등)** | **서비스 안정성 150% 향상** |
| **최대 대기 시간** | 약 $2 \times T_{seek}$ | 약 $1 \times T_{seek}$ (복귀 제외) | **Worst-case 레이턴시 감소** |
| **시스템 예측성** | 낮음 | **매우 높음** | **SLA (Service Level Agreement) 준수 용이** |

### 미래 전망
- **SMR (Shingled Magnetic Recording) 드라이브 최적화**: 트랙이 겹쳐서 기록되는 SMR 방식은 한 방향으로만 기록/읽기를 수행하는 것이 유리하므로, C-SCAN의 원리가 더욱 중요해짐.
- **소프트웨어 정의 스토리지 (SDS)**: 소프트웨어 계층에서 가상 트랙 주소를 재배치(Remapping)하여, C-SCAN이 물리적 헤드 이동과 가장 완벽하게 일치하도록 유도하는 지능형 스케줄러가 발전 중임.

### 참고 표준
- **SCSI-3 / SAS Standards**: 디스크 드라이브 명령 집합에서 I/O 요청의 정렬 및 태그 큐잉(Tagged Command Queuing) 정의.
- **Linux Kernel `blk-mq`**: 멀티 큐 환경에서도 C-SCAN의 순차성을 유지하며 병렬 처리를 지원하는 표준 아키텍처.

- **📢 섹션 요약 비유**: C-SCAN은 가장 바쁜 곳(가운데)에만 시선을 뺏기지 않고, 소외된 곳(양 끝단)까지 골고루 살피는 "민주적인 자원 관리의 표본"입니다.

- **[CFQ (Completely Fair Queuing)](../8_io_storage_system/474_리눅스_IO_스케줄러.md)**: C-SCAN의 정신을 다중 큐 환경에서 구현한 리눅스 스케줄러.

### 👶 어린이를 위한 3줄 비유 설명
1. C-SCAN은 **"일방통행 셔틀버스"**와 같아요. 정거장을 하나씩 들르며 사람을 태우고 종점까지 간 다음, 돌아올 때는 아무도 안 태우고 쌩~ 하니 출발점으로 달려가요.
2. 왜 힘들게 그냥 돌아오냐고요? 그래야 버스 정류장 어디에 서 있는 사람이라도 **버스를 기다리는 시간이 똑같아지기 때문**이에요.
3. 이 규칙 덕분에 디스크 안의 "헤드" 아저씨는 길을 잃지 않고 **동그라미를 그리듯 뱅글뱅글 돌며** 부지런히 일을 할 수 있답니다!
