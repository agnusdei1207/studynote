# 인터럽트 처리 (상프/하프 메커니즘)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터럽트를 Top Half(즉시)와 Bottom Half(지연)로 나누어 처리
> 2. **가치**: 인터럽트 지연 최소화, 시스템 응답성 향상
> 3. **융합**: ISR, SoftIRQ, Tasklet, Workqueue와 통합

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

인터럽트 처리 상프/하프(Top Half/Bottom Half) 메커니즘은 인터럽트를 즉시 처리하는 부분과 지연 처리하는 부분으로 나누는 방식입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    인터럽트 처리 (Top Half / Bottom Half)            │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              Top Half (ISR)                                   │ │
│   │                                                              │ │
│   │   특징:                                                       │ │
│   │   • 인터럽트 컨텍스트에서 실행                                │ │
│   │   • 다른 인터럽트 블록 가능                                   │ │
│   │   • 최소한의 작업만 수행                                      │ │
│   │   • 빠르게 완료해야 함                                        │ │
│   │                                                              │ │
│   │   작업 예시:                                                  │ │
│   │   • 하드웨어 상태 확인                                        │ │
│   │   • 데이터 읽기/쓰기 (최소)                                   │ │
│   │   • Bottom Half 스케줄                                        │ │
│   │   • ACK 전송                                                  │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  ISR (Interrupt Service Routine)                     │   │ │
│   │   │  1. 인터럽트 원인 확인                                │   │ │
│   │   │  2. 최소 데이터 복사                                  │   │ │
│   │   │  3. SoftIRQ/Tasklet 스케줄                           │   │ │
│   │   │  4. 빠른 반환 (10-100 µs)                            │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                              │                                      │
│                              ↓ 스케줄                               │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              Bottom Half (지연 처리)                          │ │
│   │                                                              │ │
│   │   메커니즘 (빠름 → 유연함):                                   │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  SoftIRQ:                                             │   │ │
│   │   │  • 가장 빠름, 가장 제약적                             │   │ │
│   │   │  • 네트워크, 블록 I/O 등 핵심 용도                    │   │ │
│   │   │  • 여러 CPU에서 병렬 실행 가능                        │   │ │
│   │   │                                                      │   │ │
│   │   │  Tasklet:                                             │   │ │
│   │   │  • SoftIRQ 기반, 더 간편                              │   │ │
│   │   │  • 동일 유형은 한 CPU에서만 실행                      │   │ │
│   │   │  • 대부분의 드라이버에서 사용                         │   │ │
│   │   │                                                      │   │ │
│   │   │  Workqueue:                                           │   │ │
│   │   │  • 프로세스 컨텍스트에서 실행                         │   │ │
│   │   │  • 슬립 가능, 사용자 공간 접근 가능                   │   │ │
│   │   │  • 가장 유연하지만 느림                               │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   │   처리 예시:                                                  │ │
│   │   • 패킷 처리 (프로토콜 스택)                                │ │
│   │   • 디스크 I/O 완료 처리                                     │ │
│   │   • 복잡한 데이터 처리                                       │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> **해설**: Top Half는 최소화, Bottom Half가 실제 작업을 처리합니다.

**💡 비유**: Top Half는 응급 처치, Bottom Half는 병원 후속 치료입니다.

**📢 섹션 요약 비유**: ISR은 짧게!

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **ISR** | Top Half | 인터럽트 컨텍스트 | 응급 |
| **SoftIRQ** | 빠른 BH | 벡터 기반 | 구급차 |
| **Tasklet** | 간편 BH | SoftIRQ 기반 | 담당자 |
| **Workqueue** | 유연 BH | 프로세스 컨텍스트 | 병원 |

**비교 표**

| 메커니즘 | 컨텍스트 | 슬립 | 병렬 |
|:---|:---|:---:|:---:|
| **SoftIRQ** | 인터럽트 | X | O |
| **Tasklet** | 인터럽트 | X | 제한 |
| **Workqueue** | 프로세스 | O | O |

**핵심 알고리즘**: ISR → 스케줄 → BH 실행

**📢 섹션 요약 비유**: Workqueue가 가장 유연!

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**기술 비교: BH 선택**

| 용도 | 추천 메커니즘 |
|:---|:---|
| **네트워크** | SoftIRQ (NET_RX) |
| **일반 드라이버** | Tasklet |
| **복잡한 작업** | Workqueue |
| **타이머** | Timer SoftIRQ |

**📢 섹션 요약 비유**: 용도에 따라 선택!

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1: NIC 드라이버** → NAPI + SoftIRQ
**시나리오 2: USB** → Tasklet
**시나리오 3: 복잡한 프로토콜** → Workqueue

**도입 체크리스트**: ISR 길이, 슬립 필요 여부

**📢 섹션 요약 비유**: ISR은 100µs 이내!

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**정량/정성 기대효과**: 인터럽트 지연 90% 감소

**미래 전망**: threaded IRQ, eBPF

**참고 표준**: Linux Interrupt Handling

**📢 섹션 요약 비유**: Top/Bottom Half는 인터럽트 최적화의 핵심!
