# 다단계 페이지 테이블 (Multi-level Page Table) 구조 및 메모리 절감 원리

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 거대한 선형 페이지 테이블을 한꺼번에 메모리에 올리는 대신, 테이블 자체를 다시 페이징하여 **계층적 트리 구조**로 관리하는 방식.
> 2. **가치**: 실제로 사용되지 않는 가상 주소 공간에 대한 하위 페이지 테이블을 생성하지 않음으로써, **페이지 테이블이 차지하는 메모리 공간을 획기적으로 절감**한다.
> 3. **융합**: 현대의 64비트 시스템(4단계 이상)의 표준 메모리 관리 기법이며, 공간 효율성을 얻는 대신 주소 변환 단계가 늘어나 성능 지연이 발생하므로 **TLB**와의 병행이 필수적이다.

---

### Ⅰ. 다단계 페이지 테이블의 등장 배경

- **문제점 (Linear Page Table)**: 32비트 주소 공간(4GB)에서 페이지 크기가 4KB일 때, 페이지 테이블 엔트리(PTE)는 100만 개($2^{20}$)가 필요하다. 1개당 4B라면 프로세스마다 **4MB**의 연속된 메모리가 페이지 테이블용으로 낭비된다.
- **비효율**: 대부분의 프로세스는 4GB 공간 중 아주 일부분만 사용하므로, 빈 공간을 위한 테이블 엔트리까지 메모리에 유지하는 것은 비경제적이다.

---

### Ⅱ. 2단계 페이지 테이블 아키텍처 (ASCII)

가장 기본적인 형태로, 페이지 번호를 '페이지 디렉터리'와 '페이지 테이블' 두 부분으로 나눈다.

```ascii
    [ Logical Address (32-bit) ]
    +----------------+----------------+----------------+
    | Directory (10) |  Table (10)    |  Offset (12)   |
    +-------|--------+-------|--------+-------|--------+
            |                |                |
            v                v                |
    [ Page Directory ]  [ Page Table ]        |
    +--------------+    +--------------+      |
    | Outer Entry  |--->| Inner Entry  |      |
    +--------------+    +--------------+      |
            |                |                |
            +----------------(----------------+
                             v
                    [ Physical Memory ]
```

**[공간 절약 원리]**
- **Sparse Address Space**: 프로세스가 사용하지 않는 거대한 주소 범위에 대해서는 **Page Directory의 항목을 비워두기만 하면 된다.** 
- 그 하위의 상세한 Page Table(4KB)을 아예 만들지 않아도 되므로, 실제 사용하는 영역에 대해서만 메모리를 소모하게 된다.

---

### Ⅲ. 단계별 성능 및 공간 Trade-off

| 구분 | 선형 페이지 테이블 (1단계) | 다단계 페이지 테이블 (Multi-level) |
|:---|:---|:---|
| **메모리 점유** | 항상 최대 (비효율) | **실제 사용량에 비례 (효율)** |
| **주소 변환 속도** | 빠름 (1회 참조) | 느림 (단계별 메모리 참조 발생) |
| **연속 공간 요구** | 큰 연속 공간 필요 | 작은 조각으로 분산 저장 가능 |
| **적합성** | 작은 주소 공간 | **현대적 대용량 주소 공간 (64비트)** |

---

### ④. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 64비트 시스템의 확장
- **현상**: 64비트는 공간이 너무 넓어 2단계로도 부족하다.
- **기술사적 결단**: 현대 인텔 x86-64 아키텍처는 **4단계 (PML4, PDPT, PD, PT)** 혹은 최신 5단계 페이지 테이블을 사용한다. 단계가 깊어질수록 메모리 접근 횟수가 늘어나므로, **TLB Hit Ratio**를 극대화하는 것이 OS 성능 최적화의 제1과제가 된다.

#### 2. 기술사적 인사이트
- **페이지 테이블의 페이징**: 다단계 구조의 핵심은 "페이지 테이블조차 페이지 단위로 쪼개어 가상 메모리 관리 대상에 넣는다"는 발상의 전환이다. 이를 통해 커널 메모리 단편화 문제까지 동시에 해결한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **커널 메모리 오버헤드 감소**: 수천 개의 프로세스가 돌아가는 서버 환경에서 안정성 확보.
- **유연한 주소 할당**: 힙(Heap)과 스택(Stack) 사이의 거대한 빈 공간을 비용 없이 유지 가능.

#### 2. 미래 전망
최근에는 단계가 너무 깊어지는 문제를 해결하기 위해, 가변 크기 페이지를 지원하거나 하드웨어가 직접 트리 탐색을 수행하는 기능을 강화하고 있다. 또한 가상화 환경에서의 중첩된 다단계 테이블(Nested Page Tables)로 인한 성능 저하를 막기 위해, 페이지 테이블 자체를 캐싱하는 기술이 지속적으로 연구되고 있다.
