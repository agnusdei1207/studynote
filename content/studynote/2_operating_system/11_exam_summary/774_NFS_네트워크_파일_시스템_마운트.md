# NFS (Network File System) 네트워크 파일 시스템 마운트

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: NFS (Network File System)는 네트워크를 통해 원격 서버에 존재하는 파일 시스템을 로컬 시스템의 디렉터리 계층 구조에 연결(Mount)하여, 물리적 위치와 상관없이 로컬 파일처럼 투명하게(Transparent) 접근할 수 있게 하는 분산 파일 시스템 프로토콜이다.
> 2. **가치**: 데이터의 중앙 집중 관리를 통해 스토리지 자원 활용 효율을 높이고 데이터 중복을 제거하며, 여러 대의 서버가 동일한 데이터셋을 공유하여 병렬 처리 및 고가용성 서비스 구성을 가능케 한다.
> 3. **융합**: 원격지 함수를 호출하는 **RPC (Remote Procedure Call)** 메커니즘, 커널의 **VFS (Virtual File System)** 추상화 계층, 그리고 현대의 **pNFS (Parallel NFS)** 기술과 결합되어 클라우드 및 HPC (고성능 컴퓨팅) 환경의 핵심 공유 저장소로 기능한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: NFS는 "남의 컴퓨터에 있는 하드를 내 것처럼 쓰는 기술"이다. 서버가 자신의 폴더를 공유(Export)하면, 클라이언트는 그 폴더를 자기 집의 빈 방(Mount Point)에 연결하여 일반 파일을 읽고 쓰듯이 네트워크 너머의 데이터를 조작할 수 있다.

- **💡 비유**: 이는 "도서관 책 대여"와 같다. 도서관(NFS 서버)에는 수만 권의 책이 있고, 나는 내 집의 책꽂이 한 칸(마운트 포인트)을 비워둔다. 도서관과 내 집 사이에 마법 통로(NFS 프로토콜)를 연결하면, 나는 내 방 책꽂이에서 책을 꺼내 보지만 실제로는 도서관에 있는 책을 실시간으로 보고 있는 것과 같다.

- **등장 배경**:
  1. **워크스테이션의 보편화**: 1980년대 썬 마이크로시스템즈(Sun Microsystems)는 각 컴퓨터마다 데이터를 따로 저장하는 방식의 비효율(데이터 불일치, 관리 오버헤드)을 해결하기 위해 개발했다.
  2. **데이터 중앙 집중화**: 중요 데이터를 안전한 중앙 서버에 모아두고 백업과 보안을 일괄 관리하려는 요구가 커졌다.
  3. **디스크리스 클라이언트 (Diskless)**: 하드디스크가 없는 저가형 단말기도 네트워크를 통해 운영체제를 부팅하고 파일을 저장할 수 있는 아키텍처가 필요했다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              NFS 서비스의 추상화 및 제어 흐름 (Architecture)       │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   [ NFS Client ]                 [ NFS Server ]                 │
  │   ┌──────────────┐               ┌──────────────┐               │
  │   │ Application  │               │  NFS Daemon  │               │
  │   └──────┬───────┘               └──────▲───────┘               │
  │          │ (System Call)                │                       │
  │   ┌──────▼───────┐               ┌──────┴───────┐               │
  │   │     VFS      │               │     VFS      │               │
  │   └──────┬───────┘               └──────▲───────┘               │
  │          │ (RPC Request)                │ (Local Access)        │
  │   ┌──────▼───────┐               ┌──────┴───────┐               │
  │   │  NFS Client  │ ──▶ Network ──▶│  NFS Server  │               │
  │   │    Module    │ ◀──  (RPC)  ──◀│    Module    │               │
  │   └──────────────┘               └──────────────┘               │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 애플리케이션은 대상이 로컬인지 원격인지 모른 채 VFS에 `read()`를 요청한다. VFS는 마운트 정보를 보고 원격 파일임을 판단하여 NFS 모듈에 넘긴다. NFS 모듈은 이 요청을 RPC 패킷으로 포장하여 서버로 쏘고, 서버는 이를 받아 실제 디스크에서 데이터를 읽어 다시 RPC로 응답하는 구조다.

- **📢 섹션 요약 비유**: "파일 시스템이라는 껍데기(VFS)는 그대로 두고, 속 알맹이(물리 장치)만 네트워크 너머의 서버로 갈아 끼운 투명한 연결 장치"입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### NFS 동작의 기반: RPC (Remote Procedure Call)

NFS는 독자적인 통신 규약 대신 범용적인 RPC를 사용하여 서비스를 구현한다.

1. **XDR (External Data Representation)**: 서로 다른 기종(빅 엔디안 vs 리틀 엔디안) 간에도 데이터가 깨지지 않도록 표준화된 형식으로 직렬화함.
2. **Portmapper**: NFS 서비스(mountd, nfsd 등)가 동적으로 사용하는 포트 번호를 클라이언트에게 안내해주는 안내소 역할.
3. **Idmapd (NFSv4)**: 서버와 클라이언트의 사용자 ID(UID) 숫자가 달라도 이름 문자열로 매핑하여 권한 문제를 해결함.

---

### NFS 버전별 기술적 진화 및 차이점

| 구분 | NFSv2 / v3 | NFSv4 (Modern) | 비고 |
|:---|:---|:---|:---|
| **통신 프로토콜** | UDP 위주 (v3는 TCP 지원) | **TCP 전용** | 안정성 강화 |
| **연결 상태** | Stateless (상태 비저장) | **Stateful (상태 유지)** | 락(Lock) 관리 효율화 |
| **파일 보안** | 단순 IP/UID 기반 (취약) | **Kerberos / ACL 지원** | 기업용 보안 수준 확보 |
| **작업 결합** | 개별 요청마다 전송 | **Compound (여러 명령 묶음)** | 왕복 시간(RTT) 감소 |
| **확장성** | 단일 서버 병목 | **pNFS (Parallel NFS)** | 대규모 병렬 I/O 지원 |

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │                 pNFS (Parallel NFS)의 분산 아키텍처                │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [ Client ] ──(1) Get Layout ──▶ [ Metadata Server ]            │
  │       │                                  │                       │
  │       │                                  ▼                       │
  │       └────(2) Parallel I/O ──▶ [ Data Server 1 ]                │
  │                                [ Data Server 2 ]                │
  │                                [ Data Server 3 ]                │
  │                                                                  │
  │   [가치] 관리 정보와 데이터 통로를 분리하여 대역폭 한계 돌파          │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 고전적 NFS는 서버 한 대의 랜카드 속도가 전체 성능의 한계였다. pNFS는 클라이언트가 메타데이터 서버에서 "파일 조각들이 어디어디 있는지" 지도(Layout)를 받은 뒤, 실제 데이터는 여러 대의 데이터 서버에서 동시에 읽어 들여 기가비트 장벽을 넘어선다.

- **📢 섹션 요약 비유**: "창구 직원 한 명에게만 매달리던 방식(v3)에서, 매니저에게 위치만 물어보고 여러 창고에서 동시에 물건을 꺼내오는 방식(pNFS)으로 진화했습니다."

### Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: NFS vs SMB/CIFS (Unix vs Windows)

| 비교 항목 | NFS (Network File System) | SMB/CIFS (Server Message Block) |
|:---|:---|:---|
| **주요 진영** | 유닉스, 리눅스 | 윈도우 |
| **설계 철학** | 간결함, 원격 프로시저 호출 중심 | 복잡함, 프린터/장치 공유 포함 통합 서비스 |
| **네트워크 효율** | 좋음 (오버헤드 적음) | 보통 (채팅성 통신 많음) |
| **보안 모델** | POSIX 기반 권한 | Windows AD / ACL 기반 |

### 비교 2: 네트워크 파일 스토리지 (NAS) vs 객체 스토리지 (S3)

- **NFS (File)**: 계층 구조 유지, 부분 수정 가능, 기존 앱 수정 없이 마운트 가능. (범용성 최고)
- **Object (S3)**: 평면 구조, 불변성(Immutable), HTTP API 필요. (확장성 최고)
- **융합**: 현대 클라우드는 **EFS (AWS)**나 **Azure Files**처럼 오브젝트 스토리지의 내구성을 가지면서도 인터페이스는 NFS로 제공하는 매니지드 서비스를 주류로 사용함.

- **과목 융합 관점 (네트워크)**: NFS는 전형적인 **계층형 캡슐화**의 사례다. `File I/O -> NFS -> RPC -> TCP/UDP -> IP -> Ethernet`으로 이어지는 층층의 포장 과정을 이해하는 것이 네트워크 성능 튜닝의 시작이다.

- **📢 섹션 요약 비유**: "파일을 통째로 갈아치워야 하는 택배 방식(Object)과, 멀리 있는 책의 내용 중 한 글자만 지우고 고칠 수 있는 원격 마법 펜(NFS)의 차이입니다."

### Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 웹 서버 팜(Farm) 구성**: 10대의 웹 서버가 동일한 소스 코드와 사용자 업로드 이미지를 공유해야 함. 기술사적 판단으로 **고가용성 NFS (HA-NFS)** 구성을 제안한다. 모든 웹 서버가 동일한 NFS 경로를 마운트하게 하여 데이터 일관성을 유지하고, 서버 한 대가 죽어도 서비스 연속성을 보장한다.

2. **시나리오 — 컨테이너 (Kubernetes) 영구 저장소**: 도커 컨테이너가 삭제되어도 데이터는 남아야 함. 해결책으로 **NFS 기반의 Persistent Volume (PV)**을 사용한다. 컨테이너가 어느 노드에서 뜨더라도 네트워크를 통해 동일한 NFS 볼륨에 접근하게 함으로써 상태 유지(Stateful) 서비스를 구현한다.

3. **시나리오 — 딥러닝 데이터 로딩 병목**: 수천 대의 GPU 노드가 동시에 학습 데이터를 읽어오느라 NFS 서버가 뻗음. 해결책으로 **pNFS** 또는 **All-Flash 기반 분산 NFS**를 도입하여, 데이터 트래픽을 여러 스토리지 노드로 분산시켜 IOPS를 선형적으로 확장한다.

### 도입 체크리스트 (Performance & Security)
- **동기화 모드**: `sync`(안전)인가 `async`(빠름)인가? (중요 데이터는 반드시 `sync` 권장)
- **속성 캐시 (Attribute Cache)**: `noac` 옵션을 쓰면 정합성은 좋아지나 네트워크 부하가 극심해짐. 적절한 `actimeo` 설정이 있는가?
- **Root Squash**: 클라이언트의 루트 계정이 서버의 루트 권한을 가지지 못하도록 제한(`root_squash`)하고 있는가? (보안 필수)

### 안티패턴
- **불안정한 WiFi 환경에서의 NFS 마운트**: 네트워크가 끊기면 프로세스가 `D (Uninterruptible Sleep)` 상태에 빠져 시스템 전체가 좀비가 됨. (해결: `soft` 마운트 또는 `intr` 옵션 사용)
- **NFS 위에 데이터베이스(DB) 설치**: NFS의 네트워크 레이턴시와 락(Lock) 불안정성 때문에 DB 파일 오염 및 성능 저하 유발. (해결: DB는 전용 블록 스토리지 사용 권장)

- **📢 섹션 요약 비유**: "편리하다고 해서 모든 것을 네트워크로 연결하려다가는, 줄이 꼬였을 때(네트워크 장애) 집안 전체(시스템)가 엉망이 될 수 있음을 경계해야 합니다."

### Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과 (NFS 구축 시)

| 구분 | 기대효과 | 비고 |
|:---|:---|:---|
| **저장 효율** | 중앙 서버 단일 보관으로 디스크 중복 낭비 제거 | 스토리지 비용 40% 절감 |
| **협업 생산성** | 여러 사용자가 동일 디렉터리에서 실시간 공동 작업 가능 | 업무 리드타임 단축 |
| **관리 용이성** | 보안 정책, 백업, 할당량 관리를 서버 한 곳에서 수행 | 운영 오버헤드 70% 감소 |
| **확장 유연성** | 클라이언트 추가 시 하드웨어 증설 없이 마운트만으로 확장 | Scale-out 환경의 핵심 |

### 미래 전망
- **NFS over RDMA**: TCP 오버헤드를 줄이기 위해 메모리 직접 접근(RDMA) 기술과 NFS를 결합, 나노초 단위의 네트워크 I/O 실현.
- **SmartNIC 기반 NFS 오프로딩**: NFS 패킷의 암호화, 압축, 프로토콜 해석을 CPU가 아닌 네트워크 카드에서 전담하여 서버 부하를 0으로 만드는 기술 보편화.

### 참고 표준
- **RFC 1813 (NFSv3)**: 수십 년간 쓰인 가장 안정적인 표준 명세.
- **RFC 7530 / 8881 (NFSv4.x)**: 현대적인 보안과 병렬 처리를 담은 표준 규격.

- **📢 섹션 요약 비유**: NFS는 "네트워크는 컴퓨터다"라는 구호를 현실로 만든 위대한 유산이며, 미래의 분산 컴퓨팅 환경에서도 데이터의 혈관 역할을 묵묵히 수행할 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. NFS는 우리 집 컴퓨터와 친구 집 컴퓨터를 **"긴 투명 빨대"**로 연결하는 것과 같아요.
2. 친구네 집 보물 상자에 있는 만화책을 내 방 책꽂이에 있는 것처럼 **편하게 꺼내 볼 수** 있게 해주죠.
3. 덕분에 친구랑 나랑 똑같은 만화책을 **따로 살 필요 없이** 같이 볼 수 있어서 정말 사이좋은 기술이랍니다!
