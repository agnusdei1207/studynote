# 커널 덤프 (Kdump) 시스템 크래시 원인 분석 커널 구조

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 커널 패닉(Kernel Panic) 등 치명적 시스템 장애 발생 시, 현재 메모리 상태를 안전하게 덤프 파일(`vmcore`)로 추출하기 위해 **미리 예약된 독립적인 캡처 커널(Capture Kernel)**로 제어권을 넘기는 메커니즘.
> 2. **가치**: 장애 당시의 메모리(레지스터, 프로세스 상태, 락 정보 등)를 보존하여 사후 분석(Post-mortem Analysis)을 가능하게 함으로써, 간헐적이고 치명적인 버그의 근본 원인(Root Cause)을 정확히 파악할 수 있다.
> 3. **융합**: `kexec` 시스템 콜을 기반으로 재부팅 없는 커널 교체 기술을 활용하며, 분산 환경의 가시성 도구(Observability) 및 디버거(Crash, GDB) 툴체인과 결합하여 무중단/고가용성 시스템 운영의 최후 보루 역할을 수행한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: Kdump는 리눅스 환경에서 커널 패닉이나 심각한 하드웨어 예외 발생 시 시스템 메모리의 내용을 파일 형태로 저장하는 신뢰성 높은 크래시 덤프 생성 도구이다.
- **💡 비유**: 비행기가 추락(커널 패닉)할 때, 추락 직전까지의 모든 비행 데이터(메모리)를 기록하는 **블랙박스(Black Box)**와 같다. 메인 엔진이 망가져도 보조 배터리로 작동하는 작은 녹음기(캡처 커널)가 데이터를 온전히 저장해 준다.
- **등장 배경**:
  - ① **기존 한계**: 과거에는 크래시 덤프를 생성하는 코드가 장애가 난 원래 커널 내에 존재했기 때문에, 커널이 오염(Corruption)된 상태에서는 덤프 과정 자체가 실패하는 문제가 잦았다.
  - ② **혁신적 패러다임**: 시스템이 살아있을 때 아예 완전히 분리된 작은 메모리 공간에 '두 번째 커널(Capture Kernel)'을 미리 올려두고, 장애 시 오염된 커널을 버리고 곧바로 두 번째 커널로 부팅(`kexec`)하여 메모리를 추출하는 구조를 도입했다.
  - ③ **현재의 비즈니스 요구**: 클라우드 인프라와 대규모 서버 팜에서는 장애 원인을 모른 채 단순히 재부팅하는 것만으로는 동일한 장애가 재발할 위험이 크므로, 완벽한 사후 분석을 위한 덤프 아키텍처가 필수가 되었다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 핵심 구성 요소 (Module & Role)

| 구성 요소 | 역할 | 내부 동작 메커니즘 | 비유 |
|:---|:---|:---|:---|
| **Primary Kernel** | 평상시 운영체제 환경 | 시스템의 일반적인 서비스를 제공하며, 예약된 메모리 영역을 캡처 커널을 위해 보호함. | 비행기 메인 엔진 |
| **Capture Kernel** | 덤프 추출용 보조 커널 | 크래시 발생 시 `kexec`를 통해 즉각 부팅되어 기존 메모리 영역(vmcore)을 추출 및 파일로 저장. | 블랙박스 보조 녹음기 |
| **kexec** | 커널 교체 메커니즘 | 하드웨어 초기화(BIOS/UEFI)를 생략하고 현재 커널에서 바로 다음 커널을 메모리로 로드하고 점프함. | 운전석 즉시 교대 |
| **kdump/kexec-tools** | 유저스페이스 도구 | 부팅 시 캡처 커널을 메모리에 미리 로드(예약)하고 패닉 시 커널 간 컨텍스트를 구성함. | 덤프 스크립트 세팅 팀 |
| `/proc/vmcore` | 메모리 덤프 인터페이스 | 캡처 커널로 부팅된 후, 이전 커널(Primary)의 메모리 상태를 ELF 파일 형식으로 가상화하여 제공. | 블랙박스 데이터 포트 |

#### 2. Kdump 아키텍처 및 실행 흐름 다이어그램
Kdump는 장애가 발생한 원래의 커널에 의존하지 않고 안전한 환경에서 덤프를 수행하기 위해 메모리를 분할한다.

```ascii
  [System Physical Memory (RAM)]
  +-------------------------------------------------------------+
  |                                                             |
  |             Primary Kernel (Production OS)                  |
  |                                                             |
  |  - Kernel Code / Data                                       |
  |  - User Processes                                           |
  |  - Caches / Buffers                                         |
  |                                                             |
  |    <-- (Crash Occurs! Kernel Panic / NMI / Oops)            |
  |           |                                                 |
  +-----------|-------------------------------------------------+
              | (kexec triggered, HW init skipped)
              v
  +-------------------------------------------------------------+
  | Crashkernel Reservation Area (e.g., 128MB ~ 256MB)          |
  |-------------------------------------------------------------|
  |                                                             |
  |             Capture Kernel (Kdump Kernel)                   |
  |  - Clean, Uncorrupted state                                 |
  |  - Reads original memory via /proc/vmcore                   |
  |  - Writes to Disk / Network (NFS, SSH)                      |
  |                                                             |
  +-------------------------------------------------------------+
              | (Dumping /proc/vmcore to storage)
              v
       [ Local Disk / Network Storage (vmcore file) ]
```

**[다이어그램 해설]**
1. 부팅 시 시스템 메모리 중 일부(예: `crashkernel=256M`)를 Capture Kernel을 위해 예약한다. 이 영역은 Primary Kernel이 절대 건드리지 않는다.
2. 운영 중 Primary Kernel에서 치명적인 오류(Null Pointer Dereference, Deadlock 등)가 발생하여 패닉 상태가 되면 커널 내 `crash_kexec()` 함수가 호출된다.
3. BIOS 단계 없이 즉시 예약된 메모리에 있던 Capture Kernel로 실행 흐름이 점프(`kexec`)한다.
4. Capture Kernel은 자신이 부팅된 후, 기존 Primary Kernel이 사용하던 메모리 영역 전체를 `/proc/vmcore`라는 특수 인터페이스를 통해 읽어들인다.
5. 이를 로컬 디스크의 `/var/crash/`나 외부 네트워크 서버로 전송하여 `vmcore` (크래시 덤프 파일)를 생성하고 재부팅한다.

#### 3. 심층 동작 메커니즘 (Step-by-Step)
- **Phase 1: 초기화 및 로딩** - 시스템 부팅 시 `kexec-tools` 서비스가 시작되며 `kexec -p` 명령어를 통해 캡처 커널 이미지와 initramfs를 미리 확보된 커널 메모리 영역에 로드한다.
- **Phase 2: 장애 발생** - `panic()`, NMI(Non-Maskable Interrupt), SysRq 등에 의해 장애가 탐지되면, 중단 프로세스를 최소화하고 모든 CPU 코어를 멈춘다(Halt).
- **Phase 3: 컨텍스트 전환** - CPU 레지스터 상태를 보존하고, MMU(메모리 관리 장치) 캐시를 무효화한 뒤 제어권을 캡처 커널의 엔트리 포인트로 넘긴다.
- **Phase 4: 덤프 파일 생성 (makedumpfile)** - 캡처 커널은 매우 제한된 자원으로 동작하며, `makedumpfile` 유틸리티를 사용하여 불필요한 메모리(빈 페이지, 캐시 페이지)를 압축 및 제외(Filtering)한 후 실제 덤프 파일을 디스크에 기록한다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 크래시 덤프 기술 비교: Kdump vs Traditional Dump (Diskdump 등)

| 비교 항목 | Kdump (현재 표준) | Traditional Kernel Dump |
|:---|:---|:---|
| **의존성** | 장애 커널과 분리 (독립적 Capture 커널) | 장애 커널의 함수/드라이버 재활용 |
| **안정성** | **매우 높음** (오염된 데이터나 락에 영향 안 받음) | 낮음 (커널 내부 자료구조 오염 시 덤프 실패) |
| **메모리 오버헤드** | 상시 예약 메모리 소모 (약 128~512MB) | 상시 메모리 소모 없음 |
| **부팅 속도 (장애 시)** | kexec로 인해 하드웨어 포스팅 불필요 (빠름) | 보통 |
| **디버깅 포맷** | ELF 형식 (`vmcore`) | 독자 형식 또는 Raw Memory |

#### 2. 과목 융합 관점 (Synergy)
- **가상화 (Hypervisor)**: 가상 머신(VM) 환경에서는 게스트 OS의 Kdump 외에도 하이퍼바이저 레벨에서 전체 머신의 상태를 캡처하는 `virsh dump` 기능이 병행 사용된다.
- **네트워크 (Network Storage)**: 로컬 디스크 용량이 부족하거나 스토리지 컨트롤러 장애 시, Kdump는 덤프 파일을 NFS나 SSH를 통해 외부 분석 서버로 바로 전송(Netdump)할 수 있다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 장애 분석 시나리오 (Post-mortem Analysis)
- **증상**: 프로덕션 서버가 특정 부하 상황에서 이유 없이 먹통이 되거나 재부팅됨. Syslog에는 패닉 직전의 로그조차 남지 않음.
- **대응 (Kdump + Crash 도구)**:
  1. 시스템에 Kdump가 설정되어 있는지 확인하고, 재부팅 후 `/var/crash/`에 `vmcore` 파일이 생성되었는지 확인.
  2. `crash /usr/lib/debug/lib/modules/.../vmlinux /var/crash/vmcore` 명령어로 분석 환경 진입.
  3. `bt` (Backtrace) 명령으로 패닉을 유발한 스레드의 함수 호출 스택 확인.
  4. `ps`, `kmem`, `struct` 명령어를 사용하여 커널 내 뮤텍스 경합이나 불법적인 메모리 접근(Null Pointer)을 식별.

#### 2. 도입 체크리스트 및 주의사항
- **예약 메모리 확보**: 물리적 램 크기에 비례하여 적절한 `crashkernel` 파라미터 크기를 할당해야 한다. (예: `crashkernel=1G-4G:192M,4G-64G:256M`). 너무 작으면 캡처 커널 구동 중 OOM이 발생하여 덤프가 실패한다.
- **OOM Killer와의 구분**: 프로세스 레벨의 메모리 부족(OOM)은 커널 패닉이 아니므로 기본적으로 Kdump가 동작하지 않는다. 패닉으로 유도하려면 `sysctl vm.panic_on_oom=1` 설정이 필요하나, 고가용성 측면에서는 신중해야 한다.

#### 3. 안티패턴 (치명적 결함 사례)
- **덤프 저장소 용량 미확보**: 로컬 스토리지의 `/var` 파티션 용량이 물리 램 크기보다 훨씬 작을 때, 덤프 쓰기가 실패한다. (해결책: `makedumpfile` 압축 레벨 증가 또는 원격지 덤프 구성).
- **무분별한 덤프 유지**: 덤프 파일(`vmcore`)은 수십 기가바이트에 달할 수 있다. 장애 조치 후 기존 덤프를 삭제하지 않으면 디스크 풀(Full) 장애로 이어질 수 있다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과

| 구분 | 도입 전 | Kdump 도입 후 (ROI) |
|:---|:---|:---|
| **MTTR (복구 시간)** | 동일 장애 재발 시마다 추측성 패치 반복 (수 일 소요) | 첫 장애 시 덤프 분석을 통해 근본 원인 도출 (수 시간 내 픽스) |
| **서비스 가용성** | 재현 불가능한 버그를 방치하여 잠재적 리스크 안고 운영 | 명확한 장애 원인 증명 및 벤더(Red Hat 등) 지원 티켓 발행 가능 |
| **운영 투명성** | "왜 죽었는지 모름" | 덤프 스택 트레이스 기반 구체적 분석 보고서 작성 가능 |

#### 2. 미래 전망
Kdump는 클라우드 네이티브 환경에서 더 정교하게 진화하고 있다. 향후에는 AI를 활용하여 생성된 `vmcore` 덤프를 자동으로 분석하고 알려진 커널 버그(CVE) 데이터베이스와 매핑하여, 장애 원인 리포트를 사람의 개입 없이 자동으로 생성해주는 **AIOps 기반의 무인 크래시 분석 시스템**으로 발전할 것이다.
