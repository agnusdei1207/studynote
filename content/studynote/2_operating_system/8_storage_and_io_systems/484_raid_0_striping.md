# RAID 0 (스트라이핑, Striping)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RAID 0 (Zero)은 여러 개의 물리적 디스크를 묶어, 데이터를 특정 크기의 블록(스트라이프, Stripe) 단위로 쪼개어 각 디스크에 순차적·교차적으로 분산 기록하는 스트라이핑 (Striping) 기술 체계다.
> 2. **가치**: 데이터를 읽고 쓸 때 묶여 있는 모든 디스크가 동시에 I/O(Input/Output)를 수행하므로(병렬 처리), 이론적으로 디스크 개수(N)만큼의 **선형적인 I/O 속도 증폭(대역폭 극대화)** 이라는 최고의 성능을 달성한다.
> 3. **한계**: 패리티(Parity) 연산이나 미러링(Mirroring) 같은 데이터 보호 장치가 단 1바이트도 존재하지 않으므로, 어레이에 속한 **단 하나의 디스크만 고장 나도 전체 볼륨의 데이터가 100% 영구 유실(Fault Tolerance 0)** 되는 '극한의 성능 모험' 설계다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: RAID의 원조격 기법이자 가장 공격적인 형태인 레벨 0은 사실 "여분의 중복성(Redundant)"이라는 RAID 본래의 개념표에는 위배된다. 복원 데이터(Redundancy)가 "0"이라는 의미에서 RAID 0으로 명명되었다. 오직 속도(Performance)와 용량(Capacity)의 최대 활용이라는 한 우물만 파는 방식이다.
- **필요성**: 고해상도 비디오 렌더링, 3D 모델 실시간 텍스처 스트리밍, 대규모 과학 연산의 임시 캐시 영역 등 데이터의 '영구 보존'보다는 당장 **막대한 대역폭(Bandwidth)을 통해 짧은 시간 내에 처리해내는 성능**이 절대적으로 필요한 특수 환경이 존재한다. 가장 비싼 단일 규격의 디스크를 샀음에도 I/O 속도가 CPU의 연산 속도를 도저히 따라가지 못할 때, 엔지니어는 저렴한 디스크 4개를 엮어 초고속 고속도로 4차선을 뚫어버리는 결단(RAID 0)을 내리게 된다. 

- **💡 비유**: RAID 0은 "1시간짜리 강연 녹취록"을 1명의 타자수가 치면 1시간이 걸리지만, 녹음 파일을 5분 단위로 12명(디스크 개수 N)의 타자수에게 찢어서 동시에 치라고 지시하면(스트라이핑), 전체 문서를 5분 만에 완성해버리는(성능 N배 상승) 극강의 효율 분업입니다. 하지만 12명 중 1명이라도 화장실에 가서 타이핑을 안 치면(디스크 H/W 에러), 전체 문서가 이가 빠져 아무 의미 없는 쓰레기가 되어버리는 리스크(신뢰성 제로)와 똑같습니다.

- **RAID 0 (스트라이핑)의 데이터 분산 물리 매핑 구조**:
블록 단위의 데이터가 순차적으로 여러 스핀들(디스크)에 어떻게 쪼개어져 뿌려지는지를 ASCII 다이어그램으로 시각화하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                 RAID 0 (Striping) 어레이의 데이터 분산 매핑             │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                                                                         │
  │   운영체제가 바라보는 단일 거대 가상 볼륨 (예: 1TB x 4개 = 4TB 볼륨)    │
  │   [ 블록 A | 블록 B | 블록 C | 블록 D | 블록 E | 블록 F | G... ]        │
  │                                                                         │
  │          │      (데이터 스트라이핑 쓰기 명령어 하달 시)                 │
  │          ▼         동시에 병렬(Parallel)로 기록 팍!        ▼            │
  │                                                                         │
  │   ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐                  │
  │   │ 디스크 1│    │ 디스크 2│    │ 디스크 3│    │ 디스크 4│   디스크 2   │
  │   │        │    │        │    │        │    │        │ 💥 파괴 시!      │
  │   │ [ A ]  │    │ [ B ]  │    │ [ C ]  │    │ [ D ]  │    │             │
  │   │ [ E ]  │    │ [ F ]  │    │ [ G ]  │    │ [ H ]  │    ▼             │
  │   │ [ I ]  │    │ [ J ]  │    │ [ K ]  │    │ [ L ]  │   전체 배열      │
  │   │  ...   │    │  ...   │    │  ...   │    │  ...   │  100% 붕괴!      │
  │   └────────┘    └────────┘    └────────┘    └────────┘                  │
  │                                                                         │
  │   * 청크(Chunk Size/Stripe Size): A, B, C 등 하나의 조각 크기           │
  │     (일반적으로 64KB, 128KB, 256KB 등으로 컨트롤러에서 튜닝)            │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RAID 0 컨트롤러는 호스트로부터 들어오는 연속된 일련의 데이터를 미리 설정된 청크 크기(예: 64KB)로 깍둑썰기(Strip)한다. 그리고 첫 64KB(A)는 디스크 1에, 다음 64KB(B)는 디스크 2에, 그다음 64KB(C)는 디스크 3에, D는 디스크 4에 쓰는 식으로 라운드 로빈(Round-Robin) 방식으로 동시에 뿌려버린다. 4개의 독립된 헤드(Head)와 채널(Channel)이 동시에 기록을 진행하므로 이론적으로 쓰기 및 읽기 처리량이 단일 디스크 대비 약 4배 폭증한다 (마치 수도관 4개를 모아 하나의 파이프를 만든 것과 같다). 만일 디스크가 100개라면 대역폭도 산술적으로 100배(오버헤드 제외)에 도달할 수 있다. 오직 성능 향상의 목표만을 철저히 달성한 구조다.

- **📢 섹션 요약 비유**: 아주 넓은 왕복 10차선 고속도로(RAID 0)를 뚫어 차들이 쌩쌩 달릴 수 있게 만들었지만, 중간의 다리 교각 단 1개 (개별 디스크 1곳)라도 무너지면 그 차선뿐만 아니라 고속도로 전체가 셧다운(데이터 복구 0%)되어 폐쇄되는 극단적인 설계입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 대역폭 뻥튀기(Multiplier)의 구조와 스트라이프 사이즈 (Stripe Size) 튜닝

RAID 0의 성능 튜닝의 핵심은 **청크 크기(Chunk Size = Stripe Size)** 의 설정값에 모든 것이 달려 있다. 이것은 어플리케이션의 I/O 특성(랜덤 vs 순차, 파일의 크기)과 완벽하게 매칭되어야만 제 성능을 발휘한다.

| 워크로드 (Workload) | 적정 Stripe Size | 튜닝 원점 매커니즘 | 실무 예제 | 비유 |
|:---|:---|:---|:---|:---|
| **거대 파일 순차(Sequential) I/O** | **매우 큼 (256KB ~ 2MB)** | 큰 파일을 다수의 디스크에 한 방에 길게 분산시켜, 디스크 헤드의 이동(Seek)을 최소화하고 스트리밍 스루풋 점유 극대화 | 4K/8K 동영상 편집 원본 소스 읽기, 거대 백업 이미지 전송 | 피자 반죽을 길고 큼지막하게 썰어서 오븐 레일 태우기 |
| **작은 파일 무작위(Random) I/O** | **매우 작음 (4KB ~ 16KB)** | 파일 사이즈가 작으므로 쪼갤 의미가 적다. 각 디스크가 서로 다른 수많은 작은 파일 요청을 동시다발적(병렬 IOPS)으로 처리하게끔 유도 (디스크당 1파일 온전 전담) | 초당 수천 개의 로그 기록 Web/DB 템프 캐시 영역, 소스코드 컴파일 | 깨알 같은 쿠키를 한 명씩 전담해서 수백 명 접객하기 |

> **용어 주의**: '가장 넓은 청크가 무조건 좋다'는 편견은 심각한 병목을 초래한다. 만약 1MB짜리 거대 청크를 세팅해놓고 OS가 4KB짜리 워드 파일 여러 개를 저장한다면, 스트라이핑 교차(분산)가 일어나기는 커녕 첫 번째 디스크 1개만 계속 끄적끄적 읽고 쓰기를 반복하여 단일 디스크 대비 성능 이득이 단 1%도 없는 상황이 발생한다.

### 결함 허용도 (Fault Tolerance)가 수학적으로 '0'인 이유

RAID 0 배열에서 디스크 숫자가 늘어날수록, 시스템 전체가 다운될 가용성 위험 확률은 무서운 속도로 지수 배 증폭된다. (용량 대비 위험도 폭등). 디스크 1개의 생존 확률(Reliability)을 $R$ (예: 95% = 0.95)이라 가정할 때, RAID 0 어레이(N개)의 생존 확률은 $R^N$ 이 된다.

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │                 RAID 0 볼륨의 위험도(디스크 파손 무결점 파괴) 지수 계산      │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │  전제 조건: 디스크 1개 당, 5년 내 고장(Fail)이 날 확률 5% (R = 0.95)         │
  │                                                                              │
  │  [ 단일 디스크 1대 ]                                                         │
  │     가동 확률: 0.95 (95.0%) ─▶ 고장 날 확률 5.0%                             │
  │                                                                              │
  │  [ RAID 0 (2 디스크) 구성 시 ] (용량 2배, 속도 2배!)                         │
  │     가동 확률: 0.95 × 0.95 = 0.902 (90.2%)                                   │
  │     ─▶ 전체 고장 날 확률 9.8% (위험도 거의 2배 증가)                         │
  │                                                                              │
  │  [ RAID 0 (4 디스크) 구성 시 ] (속도 4배 우왕!)                              │
  │     가동 확률: (0.95)^4 = 0.814 (81.4%)                                      │
  │     ─▶ 전체 고장 날 확률 18.6% (위험도 3.7배 폭발 증가)                      │
  │                                                                              │
  │  [ RAID 0 (10 디스크) 풀 파밍 구성 시 ]                                      │
  │     가동 확률: (0.95)^10 = 0.598 (59.8%)                                     │
  │     ─▶ 전체 시스템 돌연사/데이터 증발할 확률 무려 40.2% 돌파!!!              │
  │                                                                              │
  │   최종 결론: 속도를 위해 N차선을 늘릴수록, 교통사고 발생 리스크는 기하급수적!│
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RAID 0의 치명적 설계 한계는 서로의 영역 데이터를 한 치도 알지 못한다는 점에 있다 (교차 패리티 없음). 단 하나의 64KB 블록 덩어리(C)를 들고 있던 디스크 3가 기계 결함으로 죽어버리면, 남은 디스크 1, 2, 4에 살아있는 수천 개의 파일 파편 조각 (A, B, D, E...) 역시 '연결의 끈(무결성)'을 완전히 상실하여 휴지 조각(Corrupted File)이 되어버린다. 데이터 포렌식(Carving)으로도 사실상 복원이 거의 불가능한 수준으로 파편화된 채 파괴된다. 디스크의 개수(N)를 늘려 수백 Gbps를 달성할수록, 고장 나는 디스크가 발생할 사건 확률은 $1-(R)^N$ 그래프를 따라 폭발적으로 상승한다. 보안과 백업 시스템이 갖춰지지 않았다면 시한폭탄을 안고 코딩/서비스를 운영하는 것과 같다.

- **📢 섹션 요약 비유**: 배(디스크)를 여러 척 묶어서 노(I/O 스루풋)를 다 같이 저으면 초대형 유람선 속도로 질주(성능 증폭)할 수 있지만, 묶여 있는 뗏목 중 단 하나라도 밧줄 끊겨 가라앉으면(SPOF), 모아놨던 모든 짐(스트라이핑 된 파편)과 선원들이 다 바닷속으로 연쇄 수몰(100% 데이터 파괴)되는 삼국지 조조의 연환계 묶인 배와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 다른 RAID 레벨과의 트레이드오프 스펙 비교

RAID 0은 미지원하는 신뢰성 옵션을 모조리 포기하는 대가로, **'가용 가능한 용량(Usable Capacity)' 효율 100%와 속도 효율 100%** 라는 타의 추종을 불허하는 지표를 거머쥐었다.

| 비교 특성 | RAID 0 (스트라이핑) | RAID 1 (미러링) | RAID 5 (패리티) |
|:---|:---|:---|:---|
| **사용 가능 용량 효율** | **100% (총합: $N \times 용량$)** | 50% ($1/2$ 용량만 사용 가능) | 약 66~90% ($(N-1) \times 용량$) |
| **읽기 성능 (Read)** | **N배 (최고조 극대화)** | N배 (전체 동시 병렬 읽기 시) | N-1배 (상당히 우수함) |
| **쓰기 성능 (Write)** | **N배 (오버헤드 제로 최고 속도)** | 1배 수준 (양쪽을 똑같이 써야만 끝남) | 쓰기 페널티발생(패리티 XOR 연산 오버헤드병목) 🔻 |
| **장애 내성 (Fault Tol.)** | **0개 드라이브 허용 (사망 직결)** | 최소 1개 드라이브 (상대에 복제본 있음)| 1개 드라이브 고장 허용 (역산 복구) |

### Non-RAID 볼륨 조인 (JBOD/Spanning) 구조와의 확연한 차이
물리 디스크를 엮어서 용량을 합친다는 점에서 JBOD(Spanning 선형 볼륨)와 자주 혼동되나 동작의 궤는 180도 다르다. JBOD는 디스크 1이 다 찰 때까지 주구장창 쓰다가, 다 차면 디스크 2의 남은 공간을 순차적으로 이어서 쓰는 "꼬리 잇기(Append)" 방식이다. I/O 속도는 단일 디스크 1대 속도일 뿐 성능 향상은 0이다. 대신 앞의 디스크가 망가져도 뒤쪽 디스크의 온전한 스팬 파일은 운 좋게 살려낼 여지가 생긴다. RAID 0은 처음부터 끝까지 무조건 N개 디스크에 피자를 썰듯 분무기처럼 뿌리므로 데이터 조각이 운 좋게 살아남을 여지조차 소거된다.

- **📢 섹션 요약 비유**: JBOD가 큰 가방 여러 개를 배낭 끈 묶어서 하나 다 찰 때까지 채우고 다음 가방 여는 일차원적 증축이라면, RAID 0는 모든 가방 지퍼를 열어놓고 양손으로 수십 개의 짐 조각들을 동시에 마구 꽂아 넣는 엄청나게 빠른 양손 동시 분배 포장법입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 전략 모델

1. **시나리오 — HPC (High Peformance Computing) 환경의 머신 러닝 캐시용 스왑 디스크 (Scratch Space)**: 딥러닝(AI) 모델에서 테라바이트급 데이터셋 에포크(Epoch)를 주기적으로 로딩할 때 메인 시스템의 NVMe 1장으론 대역폭 I/O 병목이 와서 GPU 코어가 데이터를 기다리느라 놀고 있다. 이 데이터 셋은 원본 오브젝트 스토리지(S3 등 백엔드 아카이빙)에 안전히 보관되어 있으므로 컴퓨트 노드 안에서는 깨져도 다시 퍼오면(Pull) 끝이다. 따라서 가장 싼 NVMe 4~8장을 **RAID 0 (S/W mdadm)** 으로 묶고 블록 스트라이핑으로 대역폭 20GB/s 급의 거대 임시 스크래치(Scratch) 파티션을 내어주고 ML 엔지니어의 에포크 병목 시간을 완전히 찢는 아키텍처 결단이 핵심이다.

2. **시나리오 — 초대형 로그 스트리밍 버퍼 시스템 (Kafka / ELK Stack)**: 웹서버 수천 대가 초당 기가바이트의 분산 로그 이벤트를 던진다. 로컬에 쌓이는 원본 로그 파일은 Kafka나 Logstash가 다음 파이프라인으로 퍼가는 즉시 버려지는 임시 데이터이며(하루 단위 등), 약간의 누락(Log drop)은 서비스의 치명상(금전결제 등)보단 트래픽 모니터링 분석 누수 정도의 오차만 만든다. 이런 인프라에서 RAID 5 같이 느려터진 I/O 오버헤드로 인해 로그 Drop 병목이 나게 두느니, 노드들을 **RAID 0**으로 묶어 쓰기 병목 자체를 초전박살내 소화 통로를 뚫는 것이 기술 아키텍트의 타당한 선택이다.

위 무결성 0의 시한폭탄 환경 속에서도 굳이 도입해야만 하는 성능 특화형 RAID 0 채택 여부의 판단 트리는 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────────────────────┐
  │         RAID 0 채택 여부 판단을 위한 아키텍처 리스크 필터 플로우              │
  ├───────────────────────────────────────────────────────────────────────────────┤
  │                                                                               │
  │   [ 데이터 영속성 (Durability) 및 복구 타임라인 파악 ]                        │
  │                │                                                              │
  │                ▼                                                              │
  │      스토리지(볼륨)가 지금 당장 파괴(Crash)되면 생기는 비즈니스 임팩트는?     │
  │          ├─ [치명적: 원본 훼손, 고객 DB, 복구 불가 영구 보존 급]              │
  │          │    └─▶ 🚨 절대로 RAID 0 도입 금지! (즉시 해고 사유급 장애)         │
  │          │                                                                    │
  │          └─ [견딜만함: 백업/원본이 따로 존재 (단순 임시 캐싱/작업장)]         │
  │                │                                                              │
  │                ▼                                                              │
  │      해당 볼륨에서 I/O 대역폭(성능/Performance) 병목이 발생 중인가?           │
  │          ├─ [아니오: CPU 위주 연산이라 디스크 남음/성능 충분함]               │
  │          │    └─▶ 굳이 위험 안고 갈 이유 없음 (단일 스토리지 유지)            │
  │          │                                                                    │
  │          └─ [예: 렌더링, 수천 개의 쓰기 파워로 시스템 응답이 마비됨!!]        │
  │                │                                                              │
  │                ▼                                                              │
  │   ✅ 초고속 스크래치/캐시 통로 목적으로 S/W RAID 0 극단 처방 결재 승인!       │
  │      (※ 단, 반드시 '이 볼륨에는 중요한 자료 임시 폴더화 금지' 사내 정책 하달) │
  └───────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RAID 0의 도입 기준은 오직 한 문장으로 압축된다. **"지금 여기 저장되어 있는 데이터 쓰레기통이 1초 뒤에 싹 폭파되어도 어제 백업으로 다시 구우면 상관없는 잡동사니인가?"** 이다. 기업 실무에서 RAID 0 단일 구성으로 OS나 DB 엔진을 구동하는 미친 짓(?)은 절대 일어날 수 없지만, 중간 임시 연산, 압축 및 영상 트랜스코딩 작업 폴더 등, 병목으로 인한 I/O 낭비 비용이 드라이브 고장 확률 교체 비용을 압도하는 순간에는 이보다 값싸고 성능 낭비 없이 100% 용량을 구사하는 완벽한 선택지도 없다. 무결성은 포기하되 성능 한계를 파괴한 '버서커 모드'의 튜닝이다.

### 도입 체크리스트 (재앙적 사고 예방)
- **운영적**: RAID 0 볼륨 내에 OS(운영체제)의 부팅 매니저나 시스템의 중요한 Swap 파티션이 무의식적으로 혼입되지 않았는가?
- **용량 및 IO 매칭**: 드라이브 묶음 시, 반드시 구성하는 N개의 디스크 용량과 스펙(RPM 혹은 플래시 타입)을 동일한 기종으로 맞추었는가? (작은 것에 맞춰 하향 평준화 스트라이프가 커팅되므로 낭비 방지 필요).

- **📢 섹션 요약 비유**: 이 강력한 스위치(RAID 0)는 마치 카레이싱 경주차의 안전벨트, 에어백, 범퍼, 문의 쇳덩이를 모조리 뜯어내 무게(유지비/오버헤드)를 0으로 만들고 속도에만 극단적으로 몰빵을 친 극한의 F1 머신 개조 튜닝(안 부딪힌다는 전제!)과 놀랍도록 똑같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (단일 디스크 혹은 다른 RAID 옵션) | RAID 0 스트라이핑 단독 결합 아키텍처 성립 시 | 얻어지는 폭발적 개선 효과 |
|:---|:---|:---|:---|
| **정량 (I/O량)** | 1 Gbps SATA 전송 속도 한계 | 8Gps 이상 (디스크 8개 장착 시 논리적 수치 기준) | 스토리지 백플레인 레인(Lane) 효율 **한계까지 선형 돌파(Linear Growth)** |
| **정량 (용량액)** | RAID 1 선택 등으로 50%의 자원 버진/낭비 | 용량 효율(Usable Capacity) **타협 없이 100% 획득** | 구매 예산 TCO (결함 내성 제외 시) 극초 최적화 달성 완료 |
| **정성 (운영)** | 성능 병목 구간(영상 인코딩, 대용량 아카이브 밀어넣기) 적체 | 쓰기(Write) 오버헤드 0% 달성 및 즉각적인 I/O 버퍼 비움 처리 | 하드웨어 I/O 로 인한 어플리케이션 지연 체증(Bottleneck) 급소 타파 |

### 미래 전망
- **RAID 10 (1+0) 및 다중화 계층의 서포터(Under-layer) 이주**: 순수한 RAID 0 볼륨은 그 치명적인 시한폭탄성 한계(FT=0) 때문에 이제 메인스트림 볼륨에서는 찾아보기 힘들다. 하지만 스트라이핑의 **압도적인 성능** 마법만큼은 절대 포기할 수 없기 때문에, 미래에도 성능과 안정성(복제 백업)의 기치를 모두 쥐여주는 가장 위대한 하이브리드 어레이 조합인 **RAID 10 (Striping over Mirroring 조립 모델)의 '핵심 성능 뼈대'** 로서 엔진 역할의 수명을 다할 것이다.
- **클라우드 Ephemeral Storage (임시 휘발성 볼륨)**: AWS 인스턴스 스토어, GCP의 로컬 NVMe 스토리지 등 클라우드 VDC 환경에서 '어차피 인스턴스 끄면 다 날아가는 임시 고속 블록 저장소'의 토대를 묶을 때, S/W RAID 0은 백업의 개념을 초월하여 지금 당장의 I/O를 빨아들이는 호스트 레지스터 개념의 연장선 클라우드 휘발 스택으로 계속 사용되고 있다.

### 참고 표준
- **SNIA (Storage Networking Industry Association)**: RAID 레벨 분류 0(Data Striping WITHOUT Parity) 코어 워드 규약 정의.
- **Linux 커널 `mdadm` 매뉴얼**: S/W 계층 스트라이프 어레이(레벨 0) 블록 사이즈 청크 콤바인 최적화(Tuning) 규약서. ( `mdadm --create --level=0` )

결론적으로 RAID 0은 RAID의 창시 목적인 "데이터 무중단 (Redundancy)" 이라는 신성한 강령을 완벽하게 배신한 철저한 이단아적 존재이지만, 역설적으로 그 덕분에 "속도와 성능" 이라는 한 줄기의 신계(神界) 영역의 지표 효율성을 발명해낸 천재적 아이디어의 결정체다. 흠결(SPOF)은 다른 백업이나 미러링 기법(RAID 1+0 등)의 어깨 위에 올라타 극복할 수 있기에 성능 돌파의 영원한 초석이 된다.

- **📢 섹션 요약 비유**: 이 기술(RAID 0)은 수레를 끌기 위해 말이 다치면 멈출지언정 장갑을 아예 벗어던지고 8마리 말들의 안대까지 다 풀어 가장 가볍고 미친 속도로 달리게 돌진시키는, 가장 리스크가 크지만 그 어떤 짐승 스펙도 추월해버리는 광전사(Berserker) 돌격 전술과 같습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **스트라이프 사이즈 (Stripe Size) 최적화** | 데이터를 갈가리 썰어내는 도마의 구획 크기. 어플리케이션의 Random/Sequential IO 성격에 따라 잘게(16k) 혹은 무지막지하게 길게(2m) 써야 0 레벨의 영혼(속도)이 성립하는 핵심 인자. |
| **SPOF (단일 지점 장애, Single Point Of Failure)** | RAID 0을 두려워해야 하는 유일무이한 개념. 디스크가 100개건 1000개건 그중 딱 1개 기계의 나사만 빠져도, 볼륨全体の 100%가 죽음의 길로 들어서는 끔찍한 연쇄 실패 아킬레스건. |
| **JBOD (Just a Bunch Of Disks)** | 레이드 0이 분무기처럼 와라락 데이터를 병풍 뿌리기라면, 제이봇은 디스크 하나가 목 끝까지 다 찰 때까지 멍하니 넘기다가 다음 방판(Span)으로 넘사벽 줄 긋는 옛날식 확장법이라 대역폭 상승이 제로. |
| **RAID 10 (1+0 스트라이프 미러 결합)** | 이 극악의 유리대포급 생존 생명력을 구원하기 위해, RAID 1(거울 복제술)의 튼튼한 복제본 그룹들끼리 다시 RAID 0(초고속 스트라이핑 분산) 끈으로 엮어 완성한 엔터프라이즈 데이터베이스 끝판왕 융합 등급. |
| **다이렉트 I/O (MDADM 오버헤드 무시)** | 소프트웨어 모듈로 호스트 디스크 영역을 0으로 배포할 땐 캐시 연산이나 복잡성 찌꺼기가 없어 CPU 점유율 패널티도 거의 스쳐 진다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 100조각 퍼즐이 있는데 이걸 나 혼자서 다 맞추면 1시간이 꼬박 걸리잖아? 너무 지루해서 옆에 있는 친구들 3명을 더 불러서 각각 25조각씩 나눠서 한 번에 확 달려들어 맞춰버렸어! 
2. 우와, 그랬더니 15분 만에 번개처럼 엄청난 속도(RAID 0 스트라이핑)로 그림 완성이라는 엄청난 파워 효율성이 생겼어. 성능이 대박이야.
3. 그런데 아뿔싸! 친구 중 한 명이 갑자기 배가 아파서 25조각을 들고 집에 도망가버리면? 가운데 그림 이빨이 통째로 빠져버려서 사진 전체가 망가지고 100% 빵점(신뢰성 제로, 백업 없음) 처리가 되는 아찔한 도박 팀플레이야.
