# 디스크 스케줄링 (Disk Scheduling) 목적

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 디스크 스케줄링(Disk Scheduling)은 수십 개의 프로세스가 동시에 하드디스크의 각기 다른 위치(블록)를 긁어달라고 I/O 요청을 마구 쏟아낼 때, **운영체제가 이 요청들을 큐(Queue)에 모은 뒤 하드웨어 바늘(Head)의 물리적 이동 동선을 1mm라도 아낄 수 있도록 순서를 재배치(Sorting)하는 커널의 교통정리 기술**이다.
> 2. **가치**: 가장 무겁고 끔찍한 기계적 렉인 **'탐색 시간(Seek Time)'을 최소화**하고, 버려지는 시간 없이 데이터를 쏙쏙 퍼 올려 **디스크의 전체 처리량(Throughput)과 시스템 가동률을 극대화**하는 것이 지상 최대의 목표다.
> 3. **융합(한계)**: 단순히 동선만 아끼려다 구석에 있는 요청이 평생 무시당하는 기아 현상(Starvation)이 발생하므로, 최신 OS는 효율성(동선 최적화)과 공평성(Fairness) 사이에서 줄타기를 하는 **C-SCAN이나 CFQ 같은 하이브리드 알고리즘으로 융합 진화**했다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 운영체제의 블록 I/O 레이어 안에는 `I/O 스케줄러`라는 심판관이 서 있다. 앱 3개가 `블록 10번`, `블록 1000번`, `블록 20번`을 읽어달라고 큐에 던졌다. 바보(디스크)는 들어온 순서대로 10번 -> 1000번 -> 20번 순으로 바늘을 널뛰기하며 읽는다. 스케줄러는 이 목록을 가로채서 `10번 -> 20번 -> 1000번` 순서로 쓱 재정렬해 준다. 바늘이 한 방향으로 스무스하게 직진하며 짐을 줍게 만드는 최적화 로직이다.
- **필요성**: CPU는 빛의 속도로 계산을 마치고 I/O 요청 수만 개를 1초 만에 큐에 들이붓는다. 하지만 하드디스크는 바늘을 한 번 징~ 하고 움직이는데 무려 8밀리초가 걸린다. 만약 바늘이 1번 트랙과 1만 번 트랙을 탁구 치듯 왔다 갔다 한다면? 바늘 움직이는 데만 시간이 다 가고, 정작 데이터를 읽어 올리는 시간은 1%도 안 되는 **'디스크 스래싱(Disk Thrashing)'** 지옥이 펼쳐진다. "어떻게든 저 멍청하고 무거운 쇳덩어리(바늘)가 헛걸음하지 않게 동선을 완벽하게 짜주자!"라는 절박함이 스케줄링의 존재 이유다.
- **💡 비유**: 디스크 스케줄링은 **택배 기사님의 완벽한 배송 루트 짜기**와 같다. 강남구, 노원구, 송파구에서 택배 콜이 무작위로 100개 들어왔다. 초보 기사(스케줄링 없음)는 콜이 들어온 시간 순서대로 강남 갔다가 노원 갔다가 다시 강남 가는 바보 짓을 하느라 하루에 10건 배달하고 뻗어버린다(탐색 시간 낭비). 베테랑 기사(I/O 스케줄러)는 아침에 콜 100개를 지도로 쫙 펴놓고, 강남 -> 송파 -> 노원으로 한 번에 쭉 훑고 지나가는 일직선 동선(엘리베이터 알고리즘)을 짠다. 동선이 짧아지니 기름값(오버헤드)이 아껴지고 하루에 100건(Throughput)을 다 쳐낼 수 있다.

- **등장 배경 및 OS의 오지랖**:
  1. **FCFS의 참사**: 먼저 요청한 놈 먼저 처리해 줬더니 디스크 바늘이 갈지(之) 자로 춤을 추며 서버가 죽어버림.
  2. **물리 법칙의 한계**: 디스크 모터 속도(RPM)를 올리는 데는 발열과 소음이라는 한계가 명확했다. 하드웨어 스펙업이 막힘.
  3. **소프트웨어의 멱살 캐리**: "하드웨어가 못 빨라지면, 소프트웨어가 길을 똑똑하게 알려줘서 낭비를 줄이자"는 최적화 사상이 커널에 이식됨.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│        스케줄링 유무에 따른 디스크 바늘(Head) 동선의 끔찍한 차이 시각화 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│ [ 상황: 현재 바늘 위치 50번 / 요청 큐: 90번, 10번, 80번, 20번 ]         │
│                                                                         │
│ ▶ 1. 스케줄링 없음 (들어온 순서대로 FCFS)                               │
│  동선: 50 ──(이동량 40)──▶ 90 ──(이동량 80)──▶ 10                       │
│        ──(이동량 70)──▶ 80 ──(이동량 60)──▶ 20                          │
│  💥 총 바늘 이동 거리 = 40 + 80 + 70 + 60 = 250 트랙 이동!              │
│  (바늘이 이리저리 와이퍼처럼 미친 듯이 널뛰며 쇳소리를 냄)              │
│                                                                         │
│ ▶ 2. 똑똑한 디스크 스케줄링 (거리순 SSTF / 방향순 SCAN)                 │
│  큐 재정렬 발동! -> 가까운 곳부터 훑고 가자!                            │
│  동선: 50 ──(이동량 30)──▶ 80 ──(이동량 10)──▶ 90                       │
│        ──(이동량 70)──▶ 20 ──(이동량 10)──▶ 10                          │
│  ✅ 총 바늘 이동 거리 = 30 + 10 + 70 + 10 = 120 트랙 이동!              │
│  (이동 거리가 절반 이상 줄어들며 디스크 처리 속도가 2배 빨라짐 🚀)      │
└─────────────────────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** 이 단순한 산수가 디스크 스케줄링의 위대함을 100% 증명한다. 바늘 이동 거리가 250에서 120으로 줄었다는 것은, 응답 시간(Latency)이 2배 빨라지고 단위 시간당 처리량(Throughput)이 2배 늘어났다는 뜻이다. OS는 단지 메모리 배열의 순서를 0.001초 만에 바꿨을 뿐인데, 기계 장치의 물리적 효율은 기가 막히게 치솟았다.

- **📢 섹션 요약 비유**: 마트에서 장을 볼 때, 아내가 적어준 구매 리스트(큐) 순서대로 "우유 샀다가 고기 샀다가 다시 우유 코너 옆 치즈 사러" 카트를 이리저리 몰고 다니는 건 하수입니다. 고수는 카트 밀기 전에 리스트를 싹 보고 "야채 코너 쫙 돌고, 정육 코너 쫙 돌자"며 카트 동선(스케줄링)을 짜서 10분 만에 장보기를 끝내는 법입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 디스크 스케줄링의 2대 절대 목표

운영체제 스케줄러는 항상 모순되는 두 마리 토끼를 쫓는다. 

1. **탐색 시간(Seek Time) 최소화 = 전체 처리량(Throughput) 극대화**
   - **원리**: 기왕 바늘을 움직일 거, 지금 내 바늘 위치에서 제일 가까운 놈부터 쓱싹 처리하는 게 우주 최고로 효율적이다. 이동 동선 낭비가 아예 없어진다.
   - **부작용**: 가까운 놈만 챙기다 보면, 100만 번 트랙 구석에 있는 요청은 아무리 오래 기다려도 평생 바늘이 오지 않는다. 시스템 효율은 쩔지만 한 놈이 피눈물을 흘린다.

2. **공평성 (Fairness) 보장 = 기아 상태(Starvation) 방지**
   - **원리**: 구석에 박혀있든 뭐하든, 너무 오래 기다린 놈은 불쌍하니까 무조건 가서 처리해 줘야 한다. 응답 시간의 기복(Variance)을 줄이는 것이다.
   - **부작용**: 저 멀리 있는 놈 하나 처리해주겠다고 바늘을 덜그럭 끝까지 옮겨야 한다. 바늘이 이동하는 시간 동안 디스크는 아무 데이터도 못 퍼 나르므로 전체 시스템 효율(Throughput)이 뚝 떨어진다.

---

### 이율배반의 줄타기 (Trade-off)

모든 디스크 스케줄링 알고리즘(FCFS, SSTF, SCAN 등)의 역사적 발전은 이 '효율성'과 '공평성' 사이에서 줄을 타다 엎어지고 다시 일어난 기록이다.
- 무작정 효율(가까운 곳)만 추구하면 구석의 데이터가 굶어 죽고(SSTF의 실패).
- 무작정 공평하게 순서대로 하면 바늘이 너무 헛돌아 서버가 느려 터진다(FCFS의 실패).
- 결국 "엘리베이터처럼 한 방향으로 쭉 훑고 지나가며 공평하게 처리하되, 방향을 안 꺾어서 효율도 챙기자!"는 예술적 타협안(SCAN 알고리즘)이 등장하며 세상을 지배하게 된다. (다음 키워드들에서 상세 증명).

- **📢 섹션 요약 비유**: 중국집 배달원(디스크)이 '가까운 집 10곳'만 먼저 배달하면 배달량(처리량)은 최고지만, 멀리서 1시간 전에 시킨 1집은 면이 다 불어 터져 클레임(기아 상태)이 걸립니다. 그렇다고 무조건 '주문 들어온 순서'대로 1시간 전 집 1곳 갔다가 다시 동네로 오면 그사이 10집이 다 불어 터집니다. 이 동선과 시간 사이의 최적의 밸런스를 찾는 배달 앱 AI가 바로 디스크 스케줄러입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### CPU 스케줄링 vs 디스크 I/O 스케줄링

운영체제에는 두 명의 위대한 스케줄러가 있다. 하지만 일하는 철학은 정반대다.

| 비교 척도 | CPU 스케줄링 (단기 스케줄러) | 디스크 I/O 스케줄러 (블록 레이어) |
|:---|:---|:---|
| **관리 대상** | 논리적 0/1 소프트웨어 스레드 (순식간에 교체 가능) | **무겁고 느린 기계식 물리 바늘(Head)** |
| **최우선 목표** | 유저 응답성(Response Time)과 공평성 (타임 퀀텀) | **기계적 동선 낭비 억제 (Seek Time 최소화)** |
| **선점(Preemption)** | 하던 일 뺏고 남의 일로 0.001초 만에 교체 가능 | 한 번 바늘 움직이기 시작하면 **중도 취소 불가능** |
| **주요 알고리즘** | Round-Robin, MLFQ | SCAN (엘리베이터), CFQ (완전 공평 큐잉) |

### Page Cache (버퍼)와의 기막힌 공조 작전
디스크 스케줄러 혼자서는 아무리 용을 써도 큐(Queue)에 요청이 몇 개 없으면 동선 최적화를 할 수가 없다.
- 큐에 10번, 5000번 두 개밖에 없으면 무조건 널뛰기를 해야 한다.
- **버퍼 캐시의 합류**: OS의 '페이지 캐시(Page Cache)'가 램에서 데이터를 꽉꽉 모아둔다(지연 쓰기). 
- 5초 정도 꾹 참고 모았더니 버퍼에 쓰기 요청이 1만 개나 쌓였다!
- 버퍼 캐시가 이 1만 개의 요청을 한 방에 디스크 스케줄러 큐로 쏟아붓는다(Flush). 
- 스케줄러는 이 1만 개의 방대한 주소 풀(Pool)을 가지고 완벽하게 촘촘한 엘리베이터 동선을 짜서, 바늘이 1mm 움직일 때마다 데이터를 폭격하듯 쏟아내는 압도적인 I/O 스루풋을 달성한다. I/O 스케줄링은 램 캐시의 버퍼링 뭉치기(Batching) 없이는 성립할 수 없는 예술이다.

```text
┌──────────┬────────────┬────────────┬────────────────────────────────┐
│ 요청 환경  │ 큐(Queue) 상태 │ 스케줄러의 기분 │ I/O 스루풋 결과     │
├──────────┼────────────┼────────────┼────────────────────────────────┤
│ 동기 찔끔찔끔│ 1~2개 텅 빔  │ 정렬할 게 없음 ☠️│ 최악 (바늘 헛돎)   │
│ 캐시 한방덤프│ 1만개 꽉 참  │ 테트리스 대잔치 🚀│ 최상 (바늘 안 쉼) │
└──────────┴────────────┴────────────┴────────────────────────────────┘
```
**[매트릭스 해설]** "물이 들어와야 노를 젓는다." I/O 스케줄러의 효율은 큐 안에 쌓인 대기표의 개수에 완벽하게 비례한다. 선택지가 많아야 최단 경로를 우아하게 그릴 수 있기 때문이다. 이것이 바로 DB 서버들이 자잘하게 글을 안 쓰고 로그를 몇 MB 단위로 뚱뚱하게 뭉쳐서(Bulk Insert) OS에 던져주는 핵심 이유다.

- **📢 섹션 요약 비유**: 택배 기사(디스크 스케줄러)가 물건 배달 동선을 짜려는데, 본사에서 물건을 1개씩만 내려주면 최단 경로고 나발이고 그냥 갈지 자로 뛸 수밖에 없습니다. 본사(버퍼 캐시)가 택배 1만 개를 터미널에 한 번에 확 내려줘야, 기사가 동네별로 묶어서 가장 완벽한 1바퀴 동선을 짜서 퇴근 시간을 3시간 당길 수 있는 이치입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: SSD 시대의 디스크 스케줄링 무용론 (Noop의 승리)
컴퓨터 역사상 가장 위대한 "알고리즘의 자살" 사건이다.
1. **과거의 영광**: 2010년 이전 하드디스크(HDD) 시절, 엔지니어들은 이 I/O 동선(Seek Time)을 줄이기 위해 C-SCAN, Deadline, CFQ 등 수십 종의 복잡한 수학적 스케줄링 알고리즘을 만들어 냈다. 이 알고리즘 하나가 서버 성능을 2배씩 올렸다.
2. **SSD의 파괴적 등장**:
   - 바늘(Head)도 없고 원판도 없는 반도체 SSD가 튀어나왔다. 
   - 1번지나 100만 번지나 전기 쏘는 속도가 0.001ms로 100% 똑같다 (Seek Time = 0).
3. **스케줄러의 자해 공갈**:
   - 리눅스 OS가 옛날 버릇 못 버리고 SSD 앞에서도 `CFQ(엘리베이터)` 스케줄러를 돌렸다. 
   - 들어온 요청을 번호순으로 예쁘게 쫙 정렬(Sorting)하느라 소중한 CPU 코어를 파먹고 0.1초 렉을 만들었다.
   - 그런데 정작 SSD는 주소가 정렬되든 뒤죽박죽이든 속도가 똑같았다! 스케줄러의 정렬 노가다가 완벽한 **헛수고(Overhead)**가 되어버렸다.
4. **엔지니어의 철퇴 (`noop / none`)**:
   - 실무자들은 SSD가 달린 서버에서 OS 블록 스케줄러를 **`noop` (No Operation, 아무 짓도 안 하고 들어온 대로 쑤셔 넣기)**으로 바꿔버렸다.
   - 정렬 연산을 안 하니 CPU가 널널해지고, 수십만 IOPS가 그대로 SSD에 꽂히며 렉이 소멸했다.
   - 물리적 기계 팔이 사라진 현대 서버 환경에서 고전적 디스크 스케줄링의 '목적'은 이미 그 수명을 다하고 역사 속으로 퇴장 중이다.

- **📢 섹션 요약 비유**: 은행 창구(HDD)에선 번호표 뽑고 순서를 지켜야(스케줄링) 직원이 덜 피곤하고 빠릅니다. 하지만 인터넷 뱅킹 앱(SSD)이 나왔는데도, 굳이 유저 1만 명을 스마트폰 밖에서 번호순으로 줄을 세운 뒤(스케줄러 오버헤드) 1명씩 앱 버튼을 누르게 한다면 미친 짓이죠. 앱은 1만 명이 동시에 누르는(noop) 게 무조건 이득입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량/정성 기대효과

| 구분 | 내용 |
|:---|:---|
| **하드웨어 수명(Wear) 연장** | 무작위로 튀는 바늘의 급발진/급정거를 막고 한 방향으로 부드럽게 훑게 만들어, 하드디스크 액추에이터 모터의 물리적 마모율을 극적으로 감소 |
| **처리량(Throughput) 극대화**| 디스크가 가장 아파하는 Seek Time(8ms) 낭비를 정렬을 통해 1ms 이하로 압축하여 초당 읽기/쓰기 용량을 수배 폭증시킴 |
| **QoS (Quality of Service) 보장**| 공평성을 가미한 데드라인(Deadline) 알고리즘을 통해, 구석에 박힌 파일이라도 1초 안에는 무조건 응답을 받게 하여 시스템 멈춤 방어 |

### 결론 및 미래 전망

디스크 스케줄링 (Disk Scheduling)은 차갑고 둔탁한 쇳덩어리(기계)의 한계를 부수기 위해, 소프트웨어가 인간의 직관(엘리베이터 움직임 등)을 코드로 구현하여 덮어씌운 눈물겨운 땜질의 역사다. "가까운 곳 먼저", "가는 길에 줍기" 같은 지극히 상식적인 행동 패턴이 커널의 큐(Queue) 구조로 변환되었을 때, 컴퓨터 시스템 전체의 스루풋이 마법처럼 튀어 오르는 공학적 카타르시스를 증명했다. 비록 낸드 플래시(SSD)의 시대가 도래하여 "바늘 이동 최적화"라는 본래의 목적은 박물관의 유물로 사라지고 `none/noop`이 그 자리를 꿰찼지만, "수많은 비동기 요청이 쏟아질 때 그것을 버퍼에 모아 병합(Merge)하고 최적의 순서로 정렬(Sort)하여 하위에 던진다"는 I/O 스케줄링의 절대 철학은 분산 네트워크 큐 라우팅이나 클라우드 스토리지 API 로드밸런싱의 코어 로직으로 영원히 환생하여 살아갈 것이다.

- **📢 섹션 요약 비유**: 교통순경(I/O 스케줄러)이 사거리에서 수신호로 꼬인 차(데이터)들을 예쁘게 정리하여 교통 체증을 풀어주는 위대한 직업이었습니다. 훗날 하늘을 나는 플라잉 카(SSD) 시대가 와서 교차로 자체가 없어지고 순경은 실직했지만, 수많은 궤도를 꼬이지 않게 통제하던 그 '교통 통제의 철학'은 우주 항공 관제탑(네트워크 스케줄러)의 기본 매뉴얼로 영원히 남게 된 것과 같습니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- 탐색 시간 (Seek Time) | 디스크 스케줄링이 목숨을 걸고 죽이려고 하는, 바늘이 트랙을 찾아 징징거리는 끔찍한 물리적 딜레이
- 스래싱 (Thrashing) | 스케줄링을 개판으로 해서 바늘이 널뛰기하느라 디스크 I/O가 100%를 찍고 CPU가 기절하는 병목 상태
- FCFS / SSTF / SCAN | 동선 최적화와 공평성 사이에서 줄타기를 하며 발전해 온 디스크 스케줄링의 고전적 알고리즘 형제들
- noop / none 스케줄러 | 쇳덩어리 바늘이 없는 SSD 환경에서, OS의 쓸데없는 정렬 오버헤드를 끄고 즉시 통과시키는 현대 최고의 튜닝값
- 버퍼 캐시 (Buffer Cache) | 스케줄러가 동선(테트리스)을 예쁘게 짤 수 있도록 램에 대량의 데이터를 장시간 묶어두는 든든한 저수지

### 👶 어린이를 위한 3줄 비유 설명
1. **디스크 스케줄링이 뭔가요?** 심부름꾼(디스크 바늘)이 1층에서 우유, 5층에서 사과, 2층에서 빵을 가져와야 할 때, 엄마(OS)가 쪽지에 동선을 예쁘게 다시 적어주는 거예요.
2. **어떻게 적어주나요?** 그냥 부른 순서(1->5->2층)대로 왔다 갔다 하면 다리가 부러지니까, 엄마가 "1층 우유 -> 2층 빵 -> 5층 사과" 순서로 엘리베이터 타고 한 번에 올라가면서 싹 집어오게(스케줄링) 순서를 싹 고쳐줘요!
3. **그럼 뭐가 좋나요?** 심부름꾼 다리(기계)도 안 아프고, 내가 빵이랑 우유를 먹기까지 기다리는 시간(렉)이 엄청나게 짧아지는 똑똑한 방법이랍니다!
