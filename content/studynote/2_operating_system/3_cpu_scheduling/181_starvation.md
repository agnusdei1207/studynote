# 기아 상태 (Starvation / Indefinite Blocking)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기아 현상 (Starvation)은 특정 프로세스가 우선순위가 낮거나 실행 시간이 길다는 이유로 **CPU 등 시스템 자원을 영구적으로 할당받지 못하고** 준비 큐(Ready Queue)에서 무한정 대기하는 현상이다.
> 2. **가치**: 효율성만을 강조하는 스케줄링 알고리즘(SJF, 우선순위 스케줄링 등)의 구조적 결함을 드러내는 지표이며, 시스템의 **공정성 (Fairness)**을 해치는 주범이다.
> 3. **융합**: 교착 상태(Deadlock)와 유사해 보이지만, 기아 현상은 자원은 존재하나 '분배'의 문제라는 차이가 있으며, 이를 해결하기 위해 대기 시간에 비례해 점수를 높여주는 **에이징 (Aging)** 기법이 필수적으로 수반된다.

+++

## Ⅰ. 기아 현상의 개요

### 1. 정의
- 특정 프로세스가 자원 할당 순위에서 계속 밀려나서 실행이 불가능해지는 상태를 의미한다. '무한 대기(Indefinite Blocking)'라고도 한다.

### 2. 발생 배경: '효율의 역설'
- 운영체제가 시스템의 평균 대기 시간을 줄이기 위해 짧은 작업이나 높은 우선순위 작업만 계속 먼저 처리하다 보면, 상대적으로 불리한 조건의 작업들은 찬밥 신세가 된다.

### 3. 💡 비유: '인기 맛집의 2인석 우선 정책'
- 기아 현상은 **'유명 식당에서 회전율을 높이려고 2인 손님을 8인 단체 손님보다 계속 먼저 들여보내서, 단체 손님은 3시간째 밖에서 떨고 있는 상황'**과 같다.
- 식당은 계속 만석(CPU 가동)이지만, 단체 손님(긴 작업) 입장에서는 음식을 영영 못 먹을 수도 있는 위기에 처한 것이다.

+++

## Ⅱ. 기아 현상의 메커니즘 (Deep Dive)

### 1. 주요 발생 원인 알고리즘
- **SJF (Shortest Job First)**: 짧은 작업이 계속 들어오면 긴 작업은 영원히 뒤로 밀림.
- **우선순위 스케줄링**: 높은 우선순위 작업이 끊임없이 도착하면 낮은 우선순위 작업은 굶주림.
- **SRTF**: 남은 시간이 짧은 새로운 작업이 계속 선점하면 기존 작업은 멈춤.

### 2. 기아 현상 vs 교착 상태 (Deadlock)
| 구분 | 기아 현상 (Starvation) | 교착 상태 (Deadlock) |
|:---|:---|:---|
| **상태** | 작업이 실행될 '가능성'은 있음 | **절대로 실행 불가 (상호 잠김)** |
| **자원** | 자원을 다른 누군가 쓰고 있음 | 서로가 가진 자원을 서로 기다림 |
| **해결** | 우선순위 조정 (에이징) | 자원 회수, 프로세스 강제 종료 |

### 3. ASCII 구조 다이어그램
```text
    [ Ready Queue ]             [ CPU Scheduler ]           [ CPU Status ]
    ┌─────────────┐             ┌───────────────┐           ┌────────────┐
    │ High Prio 1 │ ──(Go)────▶ │ Always Picks  │ ──▶       │ Running HP │
    ├─────────────┤             │ High Priority │           └────────────┘
    │ High Prio 2 │ ──(Go)────▶ │               │
    ├─────────────┤             └───────────────┘
    │ ... (New HP)│
    ├─────────────┤             [ Sad Process ]
    │ Low Prio X  │ ◀────────── (Waiting Forever)
    └─────────────┘
```

- **📢 섹션 요약 비유**: VIP 고객 전용 입구(높은 우선순위)만 계속 열려 있고 일반 고객 입구는 닫혀 있어, 일반 고객들이 문 밖에서 굶주리는 상황과 같습니다.

## Ⅲ. 기아 현상 해결 전략

운영체제 설계자는 시스템의 공정성을 위해 다음과 같은 장치를 마련한다.

| 해결 기법 | 설명 | 효과 |
|:---|:---|:---|
| **에이징 (Aging)** | 기다린 시간에 비례하여 우선순위를 점진적으로 높임 | **가장 확실한 해결책** |
| **라운드 로빈 (RR)** | 모든 프로세스에게 공평하게 시간 할당 | 구조적으로 기아 방지 |
| **피드백 큐 (MLFQ)** | 작업의 성격에 따라 큐를 이동시키며 기회 부여 | 동적 균형 달성 |

- **📢 섹션 요약 비유**: 줄을 오래 선 손님에게는 '우대권(에이징)'을 주어 강제로 먼저 입장시키거나, 모든 테이블에 10분이라는 시간 제한(RR)을 두어 공평하게 순환시키는 전략입니다.

+++

## Ⅳ. 실무 적용 및 운영 인사이트 (Strategy)

### 1. 실무 시나리오: 백그라운드 배치 작업의 고사 방지
- **상황**: 낮 시간에는 사용자의 웹 요청(High Prio)이 너무 많아 데이터 통계 작업(Low Prio)이 아예 시작도 못 함.
- **판단**: 스케줄러에 **에이징 가중치**를 설정한다. 통계 작업이 대기 큐에서 1시간을 보낼 때마다 우선순위를 한 단계씩 올리게 하면, 아무리 바쁜 낮 시간이라도 최소한의 연산 기회는 얻을 수 있게 된다.

### 2. 기술사적 인사이트: 공정성과 효율의 균형점
- 기술사라면 기아 현상이 단순한 버그가 아니라 **'최적화의 부작용'**임을 강조해야 한다. 무조건적인 공정성(RR)은 전체 처리량을 떨어뜨릴 수 있으므로, 응답성이 중요한 작업에는 높은 우선순위를 주되 에이징을 통해 최하위 작업의 **'최소 생존권 (Lower Bound)'**을 보장하는 하이브리드 정책이 최고의 설계임을 제시해야 한다.

- **📢 섹션 요약 비유**: 성적이 좋은 학생(고성능 앱)에게 장학금을 더 주되, 형편이 어려운 학생(저우선순위 앱)에게도 최소한의 급식(CPU 자원)은 제공해야 학교(시스템)가 건강하게 유지됩니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **신뢰성 확보**: 모든 사용자 요청이 언젠가는 반드시 처리된다는 보장 제공.
- **시스템 가동률 유지**: 버려지는 프로세스 없이 모든 자원을 골고루 순환.

### 2. 미래 전망
최근의 클라우드 환경에서는 기아 현상이 개별 프로세스를 넘어 특정 사용자나 조직의 **리소스 고갈** 문제로 확대되고 있다. 이를 위해 쿼터(Quota) 관리와 페어 쉐어(Fair Share) 스케줄링이 고도화되고 있으며, 운영체제는 이제 단순한 순번 정하기를 넘어 사회적 합의(Policy)를 기술적으로 완벽히 구현하는 '자원 정의관'으로 발전할 것이다.

- **📢 섹션 요약 비유**: 소외되는 이 없이 모두가 행복한 컴퓨팅 세상을 만드는 운영체제의 따뜻한 복지 정책입니다.
