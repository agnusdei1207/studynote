# 178. 타임 퀀텀 (Time Quantum)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 라운드 로빈에서 각 프로세스에 할당되는 CPU 시간
> 2. **가치**: 응답성과 오버헤드의 균형점 결정
> 3. **융합**: 라운드 로빈, 문맥 교환, 타이머 인터럽트와 연관

---

## Ⅰ. 개요

### 개념 정의

타임 퀀텀(Time Quantum) 또는 타임 슬라이스(Time Slice)는 **라운드 로빈 스케줄링에서 각 프로세스에 연속적으로 할당되는 CPU 시간의 최대량**이다.

### 💡 비유: 게임 턴 시간
타임 퀀텀은 **게임에서 각 플레이어의 턴 시간**과 같다. 턴 시간이 너무 짧으면 게임이 끊기고, 너무 길면 다른 플레이어가 지루해한다.

### 타임 퀀텀 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                타임 퀀텀 구조                                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【타임 퀀텀의 역할】                                                    │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  타임 퀀텀 = q                                              │ │
│  │                                                             │ │
│  │  프로세스 실행:                                              │ │
│  │  ┌─────────────────────────────────────────────────────┐   │ │
│  │  │ P1 실행 (최대 q)                                     │   │ │
│  │  │ ├──────────────────────────┤                        │   │ │
│  │  │ 0                          q                        │   │ │
│  │  │                                                     │   │ │
│  │  │ 완료 안 됨 → 큐 뒤로 이동                             │   │ │
│  │  │ 완료됨 → 종료                                        │   │ │
│  │  └─────────────────────────────────────────────────────┘   │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【타임 퀀텀 크기별 효과】                                               │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  q → 0 (매우 작음):                                            │ │
│  │  ├─┤├─┤├─┤├─┤├─┤├─┤├─┤├─┤                                    │ │
│  │  P1 P2 P3 P1 P2 P3 P1 P2                                     │ │
│  │  → 문맥 교환만 반복, 실제 작업 거의 없음 (프로세서 공유)          │ │
│  │                                                             │ │
│  │  q → ∞ (매우 큼):                                              │ │
│  │  ├───────────────────────────────────────────────────────┤   │ │
│  │  P1                                                       │   │
│  │  → FCFS와 동일, 호위 효과 발생                                 │ │
│  │                                                             │ │
│  │  q 적절:                                                        │ │
│  │  ├───┤├───┤├───┤├───┤├───┤├───┤                              │ │
│  │  P1   P2   P3   P1   P2   P3                                 │ │
│  │  → 균형잡힌 응답성과 처리량                                     │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【타임 퀀텀 선택 기준】                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  고려사항                                                      │ │
│  │  ────                                                          │ │
│  │  • 문맥 교환 시간 (보통 1-10 µs)                                │ │
│  │  • 평균 CPU 버스트 시간                                         │ │
│  │  • 시스템 부하                                                  │ │
│  │  • 응답성 요구사항                                              │ │
│  │  • CPU 개수                                                    │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                타임 퀀텀 상세                                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【타임 퀀텀 크기 결정】                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  경험적 공식:                                                   │
│  │                                                             │ │
│  │  q = T / n                                                    │ │
│  │                                                             │ │
│  │  T: 목표 응답 시간                                              │ │
│  │  n: 실행 가능한 프로세스 수                                      │ │
│  │                                                             │ │
│  │  예: T = 100ms, n = 10                                        │ │
│  │  q = 100 / 10 = 10ms                                          │ │
│  │                                                             │ │
│  │  일반적 범위: 10ms ~ 100ms                                      │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【타임 퀀텀과 문맥 교환 비율】                                           │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  문맥 교환 오버헤드 비율 = 문맥 교환 시간 / 타임 퀀텀             │ │
│  │                                                             │ │
│  │  문맥 교환: 10 µs = 0.01 ms                                    │ │
│  │                                                             │ │
│  │  q = 1ms  → 오버헤드 = 1%                                      │ │
│  │  q = 10ms → 오버헤드 = 0.1%                                    │ │
│  │  q = 100ms → 오버헤드 = 0.01%                                  │ │
│  │                                                             │ │
│  │  → 타임 퀀텀이 클수록 오버헤드 비율 감소                          │ │
│  │  → 하지만 응답성 저하                                           │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【시스템별 권장 타임 퀀텀】                                              │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  시스템              타임 퀀텀                                  │ │
│  │  ────                        ────                                │ │
│  │  전통적 유닉스         100ms                                    │ │
│  │  Linux (CFS)          동적 (1-100ms)                           │ │
│  │  실시간 Linux          1-10ms                                   │ │
│  │  Windows              15.6ms (가변)                            │ │
│  │  macOS                10ms                                     │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 설정 방법
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Linux 타임 퀀텀 관련 설정】                                           │
│  ──────────────────                                                  │
│  # 최소 스케줄링 단위                                                  │
│  $ cat /proc/sys/kernel/sched_min_granularity_ns                     │
│  2250000    # 2.25ms                                                 │
│                                                                     │
│  # wakeup 그랜규래리티                                                │
│  $ cat /proc/sys/kernel/sched_wakeup_granularity_ns                  │
│                                                                     │
│  # RR 타임 퀀텀 (실시간)                                               │
│  $ cat /proc/sys/kernel/sched_rr_timeslice_ms                        │
│  100    # 100ms                                                      │
│                                                                     │
│  # 수정                                                               │
│  $ sudo sysctl -w kernel.sched_min_granularity_ns=10000000           │
│                                                                     │
│  【타임 퀀텀 튜닝 가이드】                                               │
│  ──────────────────                                                  │
│  # 대화형 워크로드                                                    │
│  $ sudo sysctl -w kernel.sched_min_granularity_ns=1000000  # 1ms     │
│                                                                     │
│  # 배치/계산 집약적                                                   │
│  $ sudo sysctl -w kernel.sched_min_granularity_ns=10000000 # 10ms    │
│                                                                     │
│  # 실시간                                                              │
│  $ sudo sysctl -w kernel.sched_rr_timeslice_ms=10          # 10ms    │
│                                                                     │
│  【프로그래밍에서의 타임 퀀텀】                                           │
│  ──────────────────                                                  │
│  #include <time.h>                                                   │
│  #include <sched.h>                                                  │
│                                                                     │
│  // RR 스케줄링으로 설정                                               │
│  void set_rr_scheduling(int priority) {                              │
│      struct sched_param param;                                       │
│      param.sched_priority = priority;                                │
│      sched_setscheduler(0, SCHED_RR, &param);                        │
│  }                                                                   │
│                                                                     │
│  // 타임 퀀텀 확인                                                     │
│  int get_time_quantum() {                                            │
│      FILE *f = fopen("/proc/sys/kernel/sched_rr_timeslice_ms", "r"); │
│      int quantum;                                                    │
│      fscanf(f, "%d", &quantum);                                      │
│      fclose(f);                                                      │
│      return quantum;                                                 │
│  }                                                                   │
│                                                                     │
│  【문맥 교환 비용 측정】                                                 │
│  ──────────────────                                                  │
│  # 문맥 교환당 평균 시간                                                │
│  $ perf stat -e task-clock,context-switches ./program               │
│                                                                     │
│  # 타임 퀀텀 대비 문맥 교환                                             │
│  $ perf record -e sched:sched_switch ./program                      │
│  $ perf report                                                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 각 프로세스에 할당되는 최대 CPU 시간
• 역할: 라운드 로빈의 핵심 매개변수
• 범위: 일반적으로 10-100ms
• 작으면: 문맥 교환 오버헤드 증가
• 크면: FCFS와 유사, 응답성 저하
• 결정: 응답성 vs 오버헤드 균형
```
