# 라운드 로빈 (Round Robin, RR) 스케줄링 - 시분할 시스템, 선점형

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 라운드 로빈 (Round Robin, RR)은 각 프로세스에게 동일한 크기의 **시간 할당량 (Time Quantum)**을 부여하고, 그 시간이 지나면 CPU를 강제로 회수하여 다음 프로세스에게 넘기는 가장 대표적인 선점형(Preemptive) 스케줄링 알고리즘이다.
> 2. **가치**: 모든 프로세스가 주기적으로 CPU를 할당받을 수 있어 **응답 시간 (Response Time)**이 짧고, 특정 프로세스의 CPU 독점 및 기아 현상(Starvation)을 구조적으로 방지하여 대화형 시스템에 최적화되어 있다.
> 3. **융합**: FCFS의 공정성과 하드웨어 타이머 인터럽트 기반의 선점 기술이 결합된 형태이며, 시간 할당량의 크기에 따라 FCFS처럼 동작하거나 문맥 교환 오버헤드가 극대화되는 성능의 변곡점을 가진다.

---

## Ⅰ. 라운드 로빈(RR) 스케줄링의 개요

### 1. 정의
- 준비 큐(Ready Queue)를 원형 큐(Circular Queue)로 간주하고, 스케줄러가 큐를 돌면서 각 프로세스에게 한 번에 한 번의 시간 할당량만큼 CPU를 할당하는 방식이다.

### 2. 특징: '공평한 순환과 제한 시간'
- **선점형**: 프로세스가 작업을 다 마치지 못했더라도 시간이 다 되면 강제로 중단된다.
- **평등성**: 우선순위 없이 오직 도착 순서와 시간 할당량에 의해서만 결정된다.
- **응답성 특화**: n개의 프로세스가 있고 할당량이 q라면, 어떤 프로세스도 $(n-1)q$ 시간 이상 기다리지 않음을 보장한다.

### 3. 💡 비유: '회전 초밥 식당'
- 라운드 로빈은 **'한 접시씩만 먹고 다음 사람에게 순서를 넘겨야 하는 회전 초밥 식당'**과 같다.
- 내가 배가 아주 고파도(긴 작업) 한 번에 한 접시(시간 할당량)만 먹을 수 있으며, 다 먹으면 다시 줄의 끝으로 가서 내 차례가 오기를 기다려야 한다. 덕분에 늦게 온 손님(짧은 작업)도 오래 기다리지 않고 초밥을 맛볼 수 있다.

## Ⅱ. 동작 메커니즘 및 성능 분석 (Deep Dive)

### 1. 실행 과정
1. 스케줄러가 준비 큐의 첫 번째 프로세스를 선택한다.
2. 타이머를 시간 할당량(Time Quantum)으로 설정하고 실행을 시작한다.
3. **상황 A (작업 완료)**: 할당량 전에 작업이 끝나면 즉시 자원 반납 후 다음 프로세스 실행.
4. **상황 B (시간 만료)**: 타이머 인터럽트가 발생하면 현재 상태를 PCB에 저장하고 준비 큐의 맨 뒤로 이동한다.

### 2. 시간 할당량(q)에 따른 성능 변화
- **q가 매우 큰 경우 ($q \to \infty$)**: 프로세스가 시간 내에 작업을 다 끝내게 되므로 **FCFS**와 동일해진다.
- **q가 매우 작은 경우 ($q \to 0$)**: 문맥 교환이 너무 자주 일어나 실제 연산보다 상태 저장/복구에 시간을 다 쓴다. (오버헤드 폭발)
- **최적의 설계**: 문맥 교환 시간($C$)보다 $q$가 충분히 커야 한다 ($q \gg C$). 보통 $q$는 10ms ~ 100ms 사이에서 결정된다.

### 3. ASCII 구조 다이어그램
```text
    [ Ready Queue (Circular) ]          [ CPU ]
    ┌────┐  ┌────┐  ┌────┐              ┌───────────────┐
    │ P3 │  │ P2 │  │ P1 │ ──(Assign)─▶ │ P0 (Current)  │ ◀── Timer Start
    └────┘  └────┘  └────┘              │ [ Time Left ] │
      ▲                                 └──────┬────────┘
      │                                        │ (Time Quantum Over!)
      └──────────────────(Back to Tail)────────┘
```

- **📢 섹션 요약 비유**: 여러 명의 아이들이 공 하나를 가지고 놀 때, 선생님이 1분마다 호각을 불어 다음 아이에게 공을 넘겨주게 하는 질서 정연한 놀이 규칙과 같습니다.

## Ⅲ. 성능 지표 비교 (SJF vs RR)

| 지표 | SJF (Shortest Job First) | Round Robin (RR) |
|:---|:---|:---|
| **평균 대기 시간** | **최상 (Optimal)** | 보통 |
| **평균 응답 시간** | 보통 | **최상 (Very Good)** |
| **공정성** | 낮음 (기아 현상 발생) | **높음 (모두에게 기회)** |
| **오버헤드** | 적음 | 높음 (잦은 문맥 교환) |

- **📢 섹션 요약 비유**: SJF가 '빨리 먹는 사람부터 먹이는 효율 중심'이라면, RR은 '한 입씩 골고루 먹이는 복지 중심'의 정책입니다.

## Ⅳ. 실무 적용 및 스케줄링 튜닝 (Strategy)

### 1. 실무 시나리오: 현대적인 운영체제의 기본값
- **상황**: 윈도우나 리눅스 데스크톱 환경에서 여러 창을 띄워놓고 작업할 때.
- **판단**: 사용자는 마우스 클릭이나 키보드 입력에 즉각적인 반응을 원한다. 따라서 현대 OS는 기본적으로 **선점형 라운드 로빈** 기반의 스케줄링을 수행한다. 다만, 무조건 똑같이 나누는 것이 아니라 여기에 우선순위를 가미한 다단계 피드백 큐(MLFQ) 형태로 발전시켜 사용한다.

### 2. 기술사적 인사이트: 타임 슬라이스와 캐시 적중률
- 기술사라면 단순히 시간 분배를 넘어 **캐시 지역성 (Cache Locality)** 문제를 짚어야 한다. 타임 슬라이스가 너무 짧으면 한 프로세스가 CPU 캐시에 데이터를 다 올리기도 전에 쫓겨나게 되어, 전체적인 연산 효율이 급감한다. 따라서 시스템의 L1/L2 캐시 크기와 메모리 대역폭을 고려하여 타임 슬라이스의 최소 길이를 설계하는 하드웨어 친화적 스케줄링 전략이 고득점의 포인트이다.

- **📢 섹션 요약 비조**: 요리사가 도마를 닦고 칼을 가는 시간(문맥 교환)보다 요리하는 시간(실행 시간)이 짧으면 식당은 망합니다. 도구를 준비한 보람이 있을 만큼의 최소 시간은 보장해줘야 합니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **상호작용성 극대화**: 사용자가 시스템을 실시간으로 제어하고 있다는 환상 유지.
- **예측 가능한 대기**: 최악의 상황에서도 내 차례가 언제 올지 계산 가능.

### 2. 미래 전망
최근에는 멀티코어를 넘어 수천 개의 가상 코어를 관리하는 클라우드 하이퍼바이저에서 라운드 로빈의 변형인 **드래프트 (Draft)** 스케줄링이나 **가중치 기반 라운드 로빈 (WRR)** 기술이 트래픽 제어에 활발히 쓰이고 있다. 라운드 로빈은 단순하지만 '민주적 자원 배분'이라는 가장 강력한 가치를 지닌, 운영체제 역사상 가장 성공한 알고리즘이다.

- **📢 섹션 요약 비유**: 강자가 독식하지 못하게 하고 모두에게 햇살(CPU)을 골고루 나누어 주는, 운영체제의 공평한 태양과 같은 기술입니다.
