# 태스크 스케줄링의 캐시 일관성 (Cache Coherence)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다중 처리기(Multiprocessor) 환경에서 동일한 데이터를 참조하는 여러 태스크가 서로 다른 CPU 코어의 로컬 캐시에 복사본을 가질 때, 데이터의 변경 사항이 모든 코어에 즉시 반영되어 모순이 발생하지 않도록 보장하는 메커니즘이다.
> 2. **가치**: 스케줄러가 태스크를 다른 코어로 이동시킬 때 발생하는 캐시 무효화(Invalidation) 오버헤드와 데이터 불일치 문제를 방지하여, 병렬 프로그래밍의 논리적 정확성과 성능 최적화를 동시에 달성한다.
> 3. **융합**: MESI 프로토콜 등 하드웨어적 일관성 유지 기법과 운영체제의 프로세서 친화성 (Processor Affinity) 스케줄링 전략이 긴밀하게 협력하는 지점이다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 멀티코어 시스템에서 각 코어는 성능을 위해 독자적인 L1/L2 캐시를 가진다. 만약 스케줄러가 코어 A에서 실행되던 태스크를 코어 B로 옮기면, 코어 A 캐시에 있던 최신 데이터가 코어 B에서는 보이지 않거나 예전 데이터를 읽게 되는 문제가 발생한다. 이를 해결하기 위한 하드웨어와 소프트웨어의 총체적 노력이 캐시 일관성 관리다.

- **💡 비유**: 캐시 일관성은 "여러 명의 비서(코어)가 공유하는 하나의 일정표(메인 메모리)"와 같다. 한 비서가 자기 수첩(로컬 캐시)에 일정을 수정했다면, 다른 비서들도 그 수정 내용을 즉시 알 수 있어야 회의가 꼬이지 않는 것과 같다.

- **📢 섹션 요약 비유**: 각자의 요약 노트를 가진 팀원들이 정보가 바뀔 때마다 서로의 노트를 최신화해주어, 누구에게 물어봐도 똑같은 정답을 얻게 만드는 정보 동기화 시스템과 같습니다.

+++

## II. 아키텍처 및 핵심 원리 (Deep Dive)

### MESI 프로토콜 (하드웨어적 해결책)

캐시 라인의 상태를 4가지로 정의하여 일관성을 유지한다.

1. **Modified (M)**: 이 캐시에만 최신 데이터가 있음 (메모리와 다름).
2. **Exclusive (E)**: 이 캐시에만 데이터가 있음 (메모리와 같음).
3. **Shared (S)**: 여러 캐시에 데이터가 공유됨 (메모리와 같음).
4. **Invalid (I)**: 이 캐시의 데이터는 무효함 (다른 놈이 고쳤음).

```text
  [ Core 0 ]          [ Core 1 ]          [ Main Memory ]
  Cache: [A:M]        Cache: [A:I]        Data A: (Old)
      |                   |                     |
      |---(Write A)------>| (Bus Snooping)      |
      |                   |                     |
      |                   |---(A is Invalid!)---|
```

**[다이어그램 해설]** 코어 0이 데이터 A를 수정하면 'M' 상태가 된다. 이때 하드웨어는 버스 스누핑 (Bus Snooping)을 통해 다른 코어(코어 1)에게 "너네가 가진 A는 이제 쓰레기야!"라고 알려 'I' 상태로 만든다. 나중에 코어 1이 A를 읽으려 하면 'I' 상태임을 확인하고 메인 메모리나 코어 0으로부터 최신 데이터를 다시 가져온다.

### 스케줄러의 역할: 프로세서 친화성 (Affinity)
하드웨어가 일관성을 맞춰주긴 하지만, 코어를 옮겨 다니는 것 자체가 'I' 상태를 양산하여 성능을 떨어뜨린다. 따라서 스케줄러는 가급적 태스크를 이전에 실행되던 코어에 계속 머물게 하려는 **Soft Affinity** 정책을 사용한다.

- **📢 섹션 요약 비유**: 비서들끼리 수첩을 고쳐주는 것도 다 일(오버헤드)이기 때문에, 웬만하면 한 비서가 전담해서 일정을 관리하게 하는 것이 효율적인 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 캐시 일관성 (Coherence) vs 메모리 일관성 (Consistency)

| 비교 항목 | 캐시 일관성 (Coherence) | 메모리 일관성 (Consistency) |
|:---|:---|:---|
| **대상 범위** | 동일한 메모리 주소(변수) 하나 | 여러 메모리 주소 간의 연산 순서 |
| **관점** | "누가 읽어도 같은 값인가?" | "쓰기 순서가 지켜져 보이는가?" |
| **주요 메커니즘** | MESI 프로토콜, 스누핑, 디렉토리 | 메모리 장벽 (Memory Barrier), Strong/Weak Model |
| **실패 시 현상** | 데이터 손상 (Corrupted Data) | 인과 관계 오류 (Out-of-order Error) |

- **📢 섹션 요약 비유**: 일관성(Coherence)은 '정답이 하나인가'의 문제이고, 모델(Consistency)은 '사건의 순서가 맞는가'의 문제입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 거짓 공유 (False Sharing)로 인한 성능 저하
- **상황**: 멀티스레드 앱에서 서로 다른 변수를 건드리는데도 불구하고, 두 변수가 동일한 캐시 라인(보통 64바이트)에 묶여 있어 한 스레드가 고칠 때마다 옆 코어의 캐시가 무효화(Invalid)되어 성능이 폭락함.
- **판단**: 이는 하드웨어 일관성 프로토콜의 부작용이다. 개발자는 변수 사이에 `padding`을 넣어 캐시 라인을 분리하거나, 언어에서 제공하는 `alignas(64)` 같은 지시어를 사용하여 독립적인 캐시 라인을 점유하도록 설계해야 한다.

- **📢 섹션 요약 비유**: 한 장의 종이에 두 명의 일기를 쓰게 했더니, 한 명이 지우개질을 할 때마다 종이가 흔들려 옆 사람도 글을 못 쓰게 되는 상황이니 종이를 따로 주어야 합니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 복잡한 멀티코어 환경에서도 프로그래머가 데이터 모순 걱정 없이 병렬 알고리즘을 설계할 수 있는 신뢰 기반을 제공한다.
- **미래 전망**: 칩렛 (Chiplet) 아키텍처와 CXL (Compute Express Link) 기술이 발전함에 따라, 여러 개의 칩을 넘어 외부 가속기까지 포함하는 '범시스템적 통합 캐시 일관성' 기술이 컴퓨팅의 핵심 고지가 될 것이다.

- **📢 섹션 요약 비유**: 모두가 하나의 진실(Data)을 공유하게 만드는 정교한 기술이 멀티코어 시대를 지탱하는 보이지 않는 힘입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **MESI 프로토콜**: 캐시 일관성을 보장하는 가장 대표적인 하드웨어 약속
- **프로세서 친화성 (Affinity)**: 캐시 효율을 위해 코어 이동을 제한하는 스케줄링 전략
- **메모리 장벽 (Memory Barrier)**: 소프트웨어 레벨에서 일관성 순서를 강제하는 명령어
