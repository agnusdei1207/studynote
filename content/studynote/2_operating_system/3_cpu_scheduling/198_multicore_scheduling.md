# 멀티코어 스케줄링 (Multicore Scheduling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 단일 칩 내에 여러 개의 물리적 연산 코어가 존재하는 멀티코어 프로세서 환경에서, 메모리 접근 지연(Memory Stall) 시간을 활용하여 하드웨어 스레드를 교차 실행함으로써 전체적인 연산 효율을 극대화하는 스케줄링 기술이다.
> 2. **가치**: CPU 연산 속도와 메모리 접근 속도 간의 격차(Memory Wall) 문제를 하드웨어적 다중 스레딩 (Multithreading)으로 보완하여, 프로세서 유휴 시간을 줄이고 처리량 (Throughput)을 향상시킨다.
> 3. **융합**: 하이퍼스레딩 (Hyper-threading), SMT (Simultaneous Multithreading), 그리고 공유 캐시 구조에서의 데이터 간섭 최소화 전략과 밀접하게 연계된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 멀티코어 스케줄링은 하드웨어가 제공하는 다중 스레드 기능을 운영체제가 인식하고, 메모리 스톨 (Memory Stall) 발생 시 다른 스레드로 즉시 전환하거나, 여러 스레드를 동시에 실행하여 CPU 파이프라인의 빈 공간을 채우는 복합적인 전략이다.

- **💡 비유**: 멀티코어 스케줄링은 "요리사(코어)가 물이 끓기를 기다리는 동안(Memory Stall) 다른 재료를 손질하는 것"과 같다. 화력(CPU 연산)은 좋은데 재료가 오는 시간(메모리 접근)이 길어서 생기는 낭비를 줄이기 위해 여러 요리를 동시에 준비하는 전략이다.

- **📢 섹션 요약 비유**: 기차(데이터)가 오기를 기다리는 동안 역무원(코어)이 다른 서류 업무를 처리하여 시간을 1분 1초도 낭비하지 않게 만드는 지능적인 시간 관리와 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 메모리 스톨 (Memory Stall)과 하드웨어 스레딩

최신 CPU는 연산 속도에 비해 메모리 접근 속도가 매우 느리다. 데이터를 기다리는 동안 CPU는 아무것도 하지 못하는 '메모리 스톨' 상태에 빠진다.

```text
  [ 단일 스레드 실행 ]
  연산 -- 연산 -- [ Memory Stall (기다림) ] -- 연산 -- 연산
  
  [ 멀티코어 하드웨어 스레딩 ]
  Thread 0: 연산 -- 연산 -- [ Stall ] -------- 연산 -- 연산
  Thread 1: -------- [ 연산 -- 연산 ] -- [ Stall ] --------
  
  (결과) CPU Utilization ↑, Idle Time ↓
```

**[다이어그램 해설]** 하나의 물리적 코어 내에 두 개 이상의 레지스터 세트(하드웨어 스레드)를 두어, 한 스레드가 메모리 데이터를 기다리는 동안 다른 스레드가 즉시 코어를 점유하여 연산을 수행한다. 운영체제는 이 하드웨어 스레드들을 각각의 논리적 CPU로 인식하여 스케줄링한다.

### 두 가지 하드웨어 스레딩 방식

1. **거친 수준 (Coarse-grained) 다중 스레딩**: 메모리 스톨 같은 긴 지연 시간이 발생할 때만 스레드를 교체한다. 교체 시 파이프라인을 비워야 하는 오버헤드가 있다.
2. **세밀한 수준 (Fine-grained) 다중 스레딩**: 명령어 수준에서 라운드 로빈 방식으로 스레드를 교차 실행한다. 설계가 복잡하지만 지연 시간 은폐 효과가 크다.

- **📢 섹션 요약 비유**: Coarse-grained는 요리가 완성될 때까지 다른 일을 하는 것이고, Fine-grained는 양파 썰고 고기 볶는 것을 아주 빠르게 번갈아 가며 하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 멀티코어 vs 멀티프로세서 스케줄링

| 구분 | 멀티코어 (Multicore) | 멀티프로세서 (Multiprocessor) |
|:---|:---|:---|
| **물리적 구조** | 단일 칩 내 다수 코어 | 별개의 여러 칩(소켓) |
| **자원 공유** | L2/L3 캐시, 메모리 버스 공유 심함 | 독립적인 버스 및 캐시 (NUMA 가능성) |
| **통신 속도** | 매우 빠름 (내부 버스) | 상대적으로 느림 (QPI/UPI 등) |
| **스케줄링 고려** | 공유 캐시 경합 최소화 | 부하 균등화 및 메모리 거리 |

- **📢 섹션 요약 비유**: 멀티코어는 한 주방 안의 여러 요리사이고, 멀티프로세서는 서로 다른 층에 있는 여러 주방의 요리사들입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 공유 캐시 스래싱 (Shared Cache Thrashing)
- **상황**: 4코어 프로세서에서 두 개의 메모리 집약적인 스레드를 동일한 L2 캐시를 공유하는 코어 쌍에 배치했더니, 서로 캐시 데이터를 쫓아내며 성능이 급격히 저하됨.
- **판단**: 운영체제 스케줄러는 **캐시 인식 스케줄링 (Cache-aware Scheduling)**을 수행해야 한다. 메모리 집약적 스레드들은 가급적 캐시를 공유하지 않는 물리적으로 떨어진 코어에 배치하고, 데이터를 자주 주고받는 스레드들은 캐시를 공유하는 인접 코어에 배치하여 통신 비용을 줄여야 한다.

- **📢 섹션 요약 비유**: 덩치가 큰 사람 둘(메모리 집약 스레드)을 좁은 벤치(공유 캐시)에 앉히면 서로 밀려나기 쉬우니, 따로 앉히는 것이 상책입니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 하드웨어 리소스의 낭비를 최소화하여 동일 전력 대비 시스템 성능 (Performance per Watt)을 극대화한다.
- **미래 전망**: 수백 개의 코어가 집적되는 Many-core 시대에는 소프트웨어가 하드웨어의 위계 구조(Topology)를 얼마나 정확히 이해하고 스케줄링하느냐가 운영체제의 핵심 경쟁력이 될 것이다.

- **📢 섹션 요약 비유**: 빈 틈 없이 꽉 찬 시간표가 우등생을 만들듯, 빈 틈 없는 스케줄링이 고성능 컴퓨터를 만듭니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **하이퍼스레딩 (Hyper-threading)**: 멀티코어 스케줄링의 대표적 구현 기술
- **캐시 일관성 (Cache Coherence)**: 멀티코어 환경에서 스케줄링이 해결해야 할 숙제
- **메모리 벽 (Memory Wall)**: 멀티코어 스케줄링이 등장하게 된 근본 배경
