# 이기종 다중 처리기 스케줄링 (HMP: Heterogeneous Multiprocessing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동일한 명령어 집합 (ISA)을 공유하지만 연산 성능과 전력 소모 특성이 다른 여러 종류의 코어(예: 고성능 Big 코어, 저전력 Little 코어)를 조합하여, 작업의 특성에 따라 최적의 코어를 동적으로 선택하는 스케줄링 기술이다.
> 2. **가치**: 모바일 기기의 배터리 수명을 획기적으로 연장하면서도 게임이나 영상 편집 같은 고성능 작업 시에는 폭발적인 연산력을 제공하는 성능-전력 효율 (PPA)의 최적 균형을 달성한다.
> 3. **융합**: ARM의 big.LITTLE / DynamIQ 아키텍처, 인텔의 하이브리드 아키텍처 (Alder Lake 이후), 그리고 운영체제의 EAS (Energy-Aware Scheduling) 엔진과 밀접하게 연계된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: HMP는 모든 코어가 동일한 대칭 다중 처리 (SMP)와 달리, 서로 다른 성능 프로파일을 가진 코어 집합을 운영체제가 하나의 시스템으로 관리하는 방식이다. 스케줄러는 각 작업의 부하 (Load)를 실시간으로 분석하여 어떤 코어에 배치할지 결정한다.

- **💡 비유**: HMP 스케줄링은 "택배 회사가 짐의 크기에 따라 자전거(Little)와 트럭(Big)을 골라 쓰는 것"과 같다. 가벼운 서류(백그라운드 앱)는 연료비가 적은 자전거로 보내고, 무거운 가구(고성능 게임)는 강력한 엔진의 트럭으로 보내 효율을 극대화하는 전략이다.

- **📢 섹션 요약 비유**: 마라톤 선수(Big 코어)와 일반인(Little 코어)이 섞여 있는 팀에서, 가파른 언덕길(무거운 작업)은 선수에게 맡기고 평지는 일반인에게 맡겨 팀 전체의 체력을 안배하는 지혜와 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### HMP 아키텍처 모델: big.LITTLE 구조

일반적으로 고성능을 지향하는 'Big' 클러스터와 저전력/고효율을 지향하는 'Little' 클러스터로 구성된다.

```text
  [ Operating System Scheduler ]
               |
  (Load Analysis & Core Selection)
        /              \
  [ Big Cluster ]      [ Little Cluster ]
  - High Clock         - Low Clock
  - Large Cache        - Small Cache
  - Out-of-Order       - In-Order
  (Performance)        (Power Efficiency)
```

**[다이어그램 해설]** 스케줄러는 개별 프로세스나 스레드의 CPU 점유율 역사를 추적한다. 부하가 특정 임계치 (Threshold)를 넘으면 해당 스레드를 Little 코어에서 Big 코어로 즉시 마이그레이션 (Up-migration)시킨다. 반대로 작업이 가벼워지면 다시 Little 코어로 내려보내 (Down-migration) 전력을 아낀다.

### 스케줄링 진화 단계
1. **Cluster Switching**: 클러스터 단위로 통째로 껐다 켰다 하던 초기 방식.
2. **In-kernel Switcher (IKS)**: Big-Little 코어를 쌍으로 묶어 하나처럼 보이게 관리.
3. **Global Task Scheduling (GTS)**: 모든 코어를 독립적으로 인식하고 작업별로 최적의 코어를 할당 (진정한 HMP).

- **📢 섹션 요약 비유**: 초기에는 팀 전체가 교체되었다면, 이제는 개별 선수들이 실시간으로 코트에 들어오고 나가는 자유로운 교체 시스템으로 진화한 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SMP (Symmetric) vs HMP (Heterogeneous)

| 비교 항목 | SMP (대칭형) | HMP (이기종/비대칭형) |
|:---|:---|:---|
| **코어 구성** | 모든 코어의 성능/구조가 동일 | 성능/전력 특성이 다른 코어 혼합 |
| **스케줄링 복잡도** | 낮음 (부하 균등화 위주) | 높음 (작업 특성 분석 및 매핑 필수) |
| **주요 목표** | 최대 병렬 처리량 | 전력 효율 (Battery Life) 및 최대 성능 |
| **적용 분야** | 서버, 데스크탑 PC | 스마트폰, 최신 랩탑, 임베디드 |

- **📢 섹션 요약 비유**: SMP는 똑같은 실력의 쌍둥이 팀이고, HMP는 각기 다른 특기를 가진 어벤져스 팀입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 모바일 게임의 프레임 드랍 (Frame Drop)
- **상황**: 고성능 게임 실행 중 발열이 심해지자 OS가 모든 작업을 Little 코어로 강제 이동시켜 프레임이 뚝 떨어지는 현상 발생.
- **판단**: 스케줄러와 써멀 매니지먼트 (Thermal Management)의 협업이 필요하다. 단순히 코어를 끄는 것이 아니라, 전압과 주파수를 미세하게 조정하는 DVFS를 활용하고, 사용자 입력이 집중되는 전경 (Foreground) 작업은 가급적 Big 코어의 최저 클럭이라도 유지하도록 우선순위를 부여해야 한다.

- **📢 섹션 요약 비유**: 트럭(Big)이 과열되었다고 해서 짐을 전부 자전거(Little)에 옮기면 배달 사고가 나니, 트럭의 속도를 조금 줄이더라도 끝까지 운반하게 해야 합니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 모바일 기기 기준 대기 전력을 50% 이상 절감하면서도, 최고 성능은 기존 SMP 대비 동등 이상을 유지한다.
- **미래 전망**: 인공지능 연산을 위한 NPU, 그래픽을 위한 GPU가 CPU와 긴밀하게 결합된 시스템 온 칩 (SoC) 환경에서, CPU를 넘어 전용 가속기까지 포함하는 '범시스템적 이기종 스케줄링'으로 영역이 확장될 것이다.

- **📢 섹션 요약 비유**: 적재적소에 알맞은 일꾼을 배치하는 안목이 전체 시스템의 생명력을 결정합니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **EAS (Energy-Aware Scheduling)**: HMP를 위한 리눅스 커널의 핵심 에너지 최적화 엔진
- **DVFS (Dynamic Voltage & Frequency Scaling)**: 코어 선택과 병행되는 전력 관리 기술
- **암달의 법칙 (Amdahl's Law)**: 이기종 구성 시 성능 향상의 이론적 한계 분석

## 👶 어린이를 위한 3줄 비유 설명
1. 컴퓨터 안에 힘센 거인(Big) 일꾼과 작고 부지런한 꼬마(Little) 일꾼이 같이 살고 있어요.
2. 무거운 짐을 옮길 때는 거인이 출동하고, 간단한 청소를 할 때는 꼬마가 일을 해서 에너지를 아껴요.
3. 이렇게 하면 컴퓨터 배터리도 오래가고, 게임도 아주 빠르게 잘 돌아간답니다!
