# 무중단 라이브 마이그레이션 스케줄링 (Live Migration Scheduling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 실행 중인 가상 머신 (VM)이나 컨테이너를 서비스 중단 없이 물리적 호스트 간에 이동시킬 때, 메모리 복사 상태와 CPU 실행 문맥을 동기화하여 서비스 가용성을 유지하는 고급 스케줄링 기술이다.
> 2. **가치**: 데이터센터의 부하 균등화, 하드웨어 점검, 에너지 절감을 위한 서버 통합 시 사용자에게 끊김 없는 경험(Zero-downtime)을 제공하며 비즈니스 연속성을 보장한다.
> 3. **융합**: 가상화 하이퍼바이저 (KVM, Xen), 네트워크 터널링 기술, 그리고 더티 페이지 (Dirty Page) 추적 및 메모리 압축 기술이 집약된 클라우드 운영의 정점이다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 라이브 마이그레이션은 물리 서버 A에서 동작하던 VM을 서버 B로 옮기는 과정이다. 핵심은 VM이 이동 중이라는 사실을 스스로도, 외부 사용자도 모르게 하는 것이다. 이를 위해 '반복적 메모리 복사'와 '최종 순간의 문맥 전환' 스케줄링이 필요하다.

- **💡 비유**: 라이브 마이그레이션은 "달리는 기차의 엔진을 통째로 다른 선로의 기차로 옮기는 것"과 같다. 승객(사용자)이 커피를 마시는 동안 기차는 멈추지 않고, 어느 순간 다른 엔진이 기차를 끌고 있게 만드는 마술 같은 과정이다.

- **📢 섹션 요약 비유**: 이삿짐을 다 옮기고 나서 집을 옮기는 게 아니라, 살고 있는 상태에서 벽과 가구(메모리)를 하나씩 옮기다가 마지막 순간에 몸만 휙 건너가는 초고속 이사 작전과 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 라이브 마이그레이션의 3단계 스케줄링 프로세스 (Pre-copy 방식)

1. **사전 준비 (Preparation)**: 대상 호스트를 선정하고 자원을 예약한다.
2. **반복적 전송 (Iterative Pre-copy)**:
   - 실행 중인 VM의 메모리 전체를 복사한다.
   - 복사하는 동안 VM이 내용을 고친 메모리(Dirty Pages)를 별도로 기록한다.
   - 고쳐진 부분만 다시 복사한다. 이 과정을 데이터가 아주 작아질 때까지 반복한다.
3. **정지 및 재개 (Stop-and-Copy)**:
   - 아주 짧은 순간(보통 100ms 이내) VM을 일시 정지한다.
   - 남은 소량의 더티 페이지와 CPU 레지스터 상태를 전송한다.
   - 새 호스트에서 즉시 실행을 재개(Resume)하고 네트워크 경로를 업데이트한다.

```text
  [ Source Host ]                      [ Destination Host ]
  
  (1) Memory All Pages --------------> (1) Receive All
  (2) Dirty Pages (Changed) ---------> (2) Receive Only Changes
  (3) Final Stop & Context State ----> (3) Resume Execution
           |                                    |
           +---------- (Network ARP Update) ----+
```

**[다이어그램 해설]** 반복 전송 횟수가 많아질수록 정지 시간(Downtime)은 짧아지지만 네트워크 대역폭 소모가 커진다. 스케줄러는 메모리 쓰기 속도와 네트워크 전송 속도를 비교하여, 전송이 끝날 기미가 보이지 않으면 (쓰기가 너무 빠르면) VM의 실행 속도를 강제로 늦추는(Throttling) 스케줄링을 수행하기도 한다.

### 포스트-카피 (Post-copy) 방식
일단 VM을 먼저 보내서 실행시킨 뒤, VM이 메모리를 참조할 때마다 원본 호스트에서 가져오는 방식이다. 다운타임은 거의 0에 가깝지만, 네트워크 장애 시 VM이 즉사하는 위험이 있다.

- **📢 섹션 요약 비유**: Pre-copy는 짐을 거의 다 옮기고 사람을 부르는 것이고, Post-copy는 사람을 먼저 보내고 필요한 옷을 택배로 보내주는 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Pre-copy vs Post-copy

| 비교 항목 | Pre-copy (선 복사) | Post-copy (후 복사) |
|:---|:---|:---|
| **다운타임 (정지 시간)** | 수십~수백 ms 발생 가능 | 거의 0 (Zero) |
| **안정성** | 높음 (실패 시 원본 유지) | 낮음 (네트워크 끊기면 VM 붕괴) |
| **메모리 전송 효율** | 낮음 (더티 페이지 중복 전송) | 높음 (딱 한 번씩만 보냄) |
| **주요 용도** | 일반적인 클라우드 마이그레이션 | 메모리 쓰기가 극도로 빈번한 앱 |

- **📢 섹션 요약 비유**: 돌다리도 두드려보고 건너는 건 Pre-copy, 일단 뛰고 보는 건 Post-copy입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: DB 서버 마이그레이션 실패 (Convergence 문제)
- **상황**: 메모리 쓰기가 엄청나게 잦은 메인 DB 서버를 라이브 마이그레이션하려 했으나, 1시간째 전송이 끝나지 않고 맴도는 현상 발생.
- **판단**: 메모리 오염 속도(Dirty rate)가 네트워크 전송 속도보다 빠르기 때문이다. 이를 '수렴(Convergence) 실패'라고 한다. 해결을 위해 하이퍼바이저의 **Auto-converge** 기능을 켜서 DB 서버의 CPU 사용률을 강제로 20% 정도 깎아 쓰기 속도를 늦추거나, 10Gbps 이상의 전용 마이그레이션 망을 확보해야 한다. 최후의 수단으로는 데이터 위험을 감수하고 Post-copy 방식으로 전환해야 한다.

- **📢 섹션 요약 비유**: 물을 퍼내는 속도보다 비가 들이치는 속도가 빠르면 배가 가라앉으니, 천막(Throttling)을 치거나 펌프(Network)를 늘려야 하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

- **기대효과**: 인프라의 유연성을 극대화하여 무중단 패치, 무중단 하드웨어 교체 등을 가능케 함으로써 서비스 가용성 99.999% 달성에 기여한다.
- **미래 전망**: 컨테이너 기반의 라이브 마이그레이션(CRIU 기술 등)이 성숙해짐에 따라, VM보다 훨씬 가벼운 단위의 초고속 이동 기술이 엣지 컴퓨팅(사용자를 따라다니는 서비스)의 핵심이 될 것이다.

- **📢 섹션 요약 비유**: 물리적 한계를 넘어 데이터센터 전체를 하나의 유기적인 거대 컴퓨터로 만드는 기술의 꽃입니다.
