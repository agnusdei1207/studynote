# 부하 균등화 (Load Balancing) - Push vs Pull Migration

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다중 처리기 (Multiprocessor) 시스템에서 각 CPU 코어의 작업 부하(Workload)를 고르게 분산시켜 특정 코어의 유휴(Idle) 상태와 다른 코어의 과부하(Overload) 상태가 공존하는 불균형을 해결하는 기술이다.
> 2. **가치**: 전체 시스템의 처리량 (Throughput)을 최적화하고 응답 시간을 단축하며, 특정 하드웨어 자원의 편중된 사용으로 인한 발열 및 수명 저하를 방지한다.
> 3. **융합**: 가상화 환경의 VM 마이그레이션, 클라우드 오케스트레이션의 스케줄링 정책, 그리고 NUMA 아키텍처의 메모리 접근 지연 최적화와 밀접하게 연계된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 다중 처리기 스케줄링에서 각 프로세서가 자신만의 준비 큐 (Private Ready Queue)를 가질 때 발생하는 부하 불균형을 해소하기 위해, 프로세스나 스레드를 한 프로세서에서 다른 프로세서로 이동시키는 기법이다.

- **💡 비유**: 부하 균등화는 "마트에서 계산대 줄을 맞춰주는 것"과 같다. 어떤 줄(CPU 코어)은 손님이 10명인데 어떤 줄은 텅 비어 있다면, 마트 관리자가 손님을 이동시키거나(Push) 비어 있는 계산원이 손님을 데려와서(Pull) 전체 계산 속도를 높이는 과정이다.

- **📢 섹션 요약 비유**: 마치 여러 개의 컵에 물(작업)을 따를 때, 한 컵만 넘치지 않게 모든 컵의 수위(부하)를 비슷하게 맞추는 평형 유지 작업과 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 부하 균등화의 두 가지 메커니즘

부하 균등화는 능동적인 감시를 통한 **Push Migration**과 유휴 코어의 자발적 요청인 **Pull Migration**으로 구분된다.

```text
  [ Core 0 ]          [ Core 1 ]          [ Core 2 ]
  (Overload)          (Normal)            (Idle)
  
      |                   |                   |
      |---(Push)--------->|                   |  <-- 특정 감시 프로세스가 
      |                   |                   |      부하를 밀어냄
      |                   |                   |
      |<--------------------------------------|  <-- 유휴 코어가 작업을
                                    (Pull)           가져옴
```

**[다이어그램 해설]** 
1. **Push Migration**: 특정 시스템 태스크가 주기적으로 각 코어의 부하를 체크하여, 임계치를 넘은 코어의 작업을 부하가 적은 코어로 강제로 이동시킨다.
2. **Pull Migration**: 특정 코어가 자신의 준비 큐가 비게 되면 (Idle), 다른 코어의 큐에서 대기 중인 작업을 스스로 가져와 실행한다.

### 주요 고려 사항: 프로세서 친화성 (Processor Affinity)의 충돌
부하 균등화는 성능 향상을 위해 수행되지만, 프로세스를 다른 코어로 이동시키면 해당 프로세스가 이전 코어의 캐시에 쌓아둔 데이터(Warm Cache)를 사용할 수 없게 되어 일시적인 성능 저하가 발생한다. 따라서 '부하 균등'과 '캐시 효율' 사이의 트레이드오프가 존재한다.

- **📢 섹션 요약 비유**: 요리사(CPU)가 옆 주방으로 이동할 때, 자기가 쓰던 도구와 재료(캐시)를 두고 가야 해서 처음에는 조금 서툴 수밖에 없는 것과 비슷합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Push Migration vs Pull Migration

| 비교 항목 | Push Migration | Pull Migration |
|:---|:---|:---|
| **동작 방식** | 외부 감시자(OS 태스크)에 의한 능동적 이동 | 유휴 프로세서에 의한 수동적/자발적 이동 |
| **발생 시점** | 주기적 체크 또는 임계치 도달 시 | 자신의 큐가 비었을 때 즉시 |
| **장점** | 시스템 전체의 균형을 강제할 수 있음 | 유휴 상태를 즉각적으로 해소함 |
| **단점** | 감시 프로세스의 오버헤드 발생 | 바쁜 코어의 큐를 뒤져야 하는 경합 발생 |

- **📢 섹션 요약 비유**: Push는 선생님이 학생들을 빈 자리로 배정하는 것이고, Pull은 빈 자리에 앉은 학생이 친구를 자기 옆으로 부르는 것과 같습니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: NUMA 아키텍처에서의 부하 균등화
- **문제**: NUMA (Non-Uniform Memory Access) 시스템에서 부하 균등화를 위해 프로세스를 다른 노드로 이동시켰더니, 메모리 접근 지연 (Memory Latency)이 급증하여 오히려 성능이 저하됨.
- **판단**: 단순 CPU 부하뿐만 아니라 '메모리 거리'를 고려해야 한다. 리눅스 커널의 **Scheduling Domains**를 사용하여 동일한 메모리 노드 내의 코어 간에 우선적으로 부하를 분산하고, 노드 간 이동은 신중하게 결정하도록 설정해야 한다.

- **📢 섹션 요약 비유**: 옆 동네 가게(다른 노드)가 한가하다고 해서 배달원(프로세스)을 보냈더니, 물건(메모리)을 가지러 우리 동네까지 다시 오느라 시간이 더 걸리는 상황을 막아야 합니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 다중 코어 시스템에서 자원 활용률을 90% 이상으로 끌어올리며, 특정 코어에만 작업이 몰려 발생하는 스로틀링 (Throttling) 현상을 방지한다.
- **미래 전망**: AI 가속기(NPU)나 이기종 멀티코어 (big.LITTLE) 환경이 일반화되면서, 각 코어의 성능 특성까지 고려한 '이기종 부하 균등화' 기술이 핵심이 될 것이다.

- **📢 섹션 요약 비유**: 모든 바퀴가 골고루 힘을 받아야 자동차가 똑바로 빨리 가듯, 모든 코어가 제 몫을 다하게 만드는 것이 부하 균등화의 마침표입니다.

2. 그래서 노는 친구가 바쁜 친구의 숙제를 도와주거나, 선생님이 숙제를 나눠주는 것을 '부하 균등화'라고 해요.
