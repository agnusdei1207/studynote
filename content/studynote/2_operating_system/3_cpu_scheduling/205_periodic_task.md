# 주기적 태스크 (Periodic Task) - P, D, T 모델

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 일정한 시간 간격 (Period)마다 반복적으로 발생하여 특정 연산을 수행하고 정해진 마감 시간 (Deadline) 내에 완료되어야 하는 실시간 시스템의 가장 기본적인 작업 모델이다.
> 2. **가치**: 시스템의 예측 가능성을 수학적으로 모델링할 수 있게 하며, RMS(Rate Monotonic)나 EDF(Earliest Deadline First) 같은 실시간 스케줄링 알고리즘의 분석 토대를 제공한다.
> 3. **융합**: 센서 데이터 수집, 제어 신호 출력, 화면 갱신 주율 등 하드웨어 루프와 운영체제 타이머 관리 기술이 결합된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 주기적 태스크는 고정된 주기(p)마다 실행 요청이 발생하며, 각 실행(Job)은 실행 시간(t)만큼 CPU를 사용하고, 발생 시점으로부터 마감 시간(d) 이전에 종료되어야 하는 특성을 갖는다.

- **💡 비유**: 주기적 태스크는 "심장 박동"과 같다. 일정한 간격(주기)으로 펌프질을 해야 하며, 다음 박동이 오기 전(마감 시간)에 피를 다 보내야(실행) 몸이 정상적으로 유지되는 것과 같다.

- **📢 섹션 요약 비유**: 정해진 시간표에 맞춰 10분마다 도착하고, 다음 차가 오기 전에 승객을 태워 떠나야 하는 지하철 운행 시스템과 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 주기적 태스크의 3대 파라미터 모델 (P, D, T)

실시간 스케줄링 분석을 위해 각 태스크는 다음 세 가지 수치로 정의된다.

```text
  실행 (Job) 1           실행 (Job) 2           실행 (Job) 3
  |<---- p ---->|       |<---- p ---->|       |<---- p ---->|
  
  [==t==]               [==t==]               [==t==]
  |<- d ->|             |<- d ->|             |<- d ->|
  
  p (Period): 작업 발생 간격 (주기)
  t (Execution Time): 실제 연산 소요 시간
  d (Deadline): 작업이 반드시 완료되어야 하는 시점 (보통 d <= p)
```

**[다이어그램 해설]**
- **주기 (p)**: 태스크가 '준비 상태'가 되는 빈도. 주기가 짧을수록 더 자주 실행되어야 하므로 시스템 부하가 높다.
- **실행 시간 (t)**: CPU를 점유하는 순수 시간. 최악 실행 시간 (WCET)을 기준으로 계산한다.
- **마감 시간 (d)**: 작업 완료의 시한폭탄. 많은 모델에서 주기와 마감 시간을 동일하게 (d = p) 설정한다.

### 이용률 (Utilization, U)과 스케줄 가능성
시스템에 n개의 주기적 태스크가 있을 때, 전체 이용률 $U = \sum (t_i / p_i)$ 이다.
- 만약 $U > 1$이면, 어떤 알고리즘으로도 모든 마감 시간을 지킬 수 없다 (Overload).
- RMS 알고리즘의 경우 $U \leq n(2^{1/n} - 1)$ 이면 모든 마감 시간 준수가 보장된다.

- **📢 섹션 요약 비유**: 1시간 동안 처리해야 할 일들의 총 시간(이용률)이 1시간을 넘으면, 아무리 순서를 잘 짜도 결국 일을 다 못 끝내는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 주기적 (Periodic) vs 비주기적 (Aperiodic) vs 간헐적 (Sporadic) 태스크

| 구분 | 주기적 (Periodic) | 비주기적 (Aperiodic) | 간헐적 (Sporadic) |
|:---|:---|:---|:---|
| **발생 시점** | 일정 간격 (예측 가능) | 불규칙적 (예측 불가) | 불규칙하지만 최소 간격 존재 |
| **마감 시간** | 엄격함 | 없음 (Best-effort) | 엄격함 |
| **분석 용이성** | 매우 높음 (수학적 증명) | 낮음 (통계적 분석) | 높음 (최악 상황 분석) |
| **비유** | 매일 아침 알람 | 갑자기 오는 문자 메시지 | 간격 두고 터지는 초인종 |

- **📢 섹션 요약 비유**: 주기적 태스크는 정기 간행물 잡지이고, 비주기적 태스크는 속보 뉴스입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 드론 비행 제어 시스템 설계
- **상황**: 드론의 자세 제어 태스크(p=10ms, t=2ms)와 GPS 위치 수신 태스크(p=100ms, t=10ms)가 동시에 실행될 때, 시스템이 안전한지 판단해야 함.
- **판단**: 이용률 $U = (2/10) + (10/100) = 0.2 + 0.1 = 0.3$. 전체 이용률이 30%로 낮으므로 RMS나 EDF 알고리즘을 사용하면 아주 안전하게 스케줄링 가능하다. 단, 자세 제어의 주기가 더 짧으므로 더 높은 우선순위를 부여해야 한다.

- **📢 섹션 요약 비유**: 10초마다 확인해야 하는 나침반과 1분마다 확인해도 되는 지도가 있다면, 당연히 나침반을 더 자주, 먼저 봐야 길을 잃지 않습니다.

## Ⅴ. 기대효과 및 결론

- **기대효과**: 시스템의 동작을 결정론적 (Deterministic)으로 설계하여, 운영 중 발생할 수 있는 타이밍 오류를 설계 단계에서 사전에 차단한다.
- **미래 전망**: 복잡한 자율주행 알고리즘이 추가됨에 따라, 수많은 주기적 태스크들 사이의 의존성 (Dependency)과 통신 지연까지 고려한 통합 스케줄링 모델링 기술이 더욱 중요해질 것이다.

- **📢 섹션 요약 비유**: 정교하게 맞물려 돌아가는 시계태엽처럼, 각자의 주기(p)를 지키는 태스크들이 모여 완벽한 시스템을 만듭니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **RMS (Rate Monotonic Scheduling)**: 주기에 반비례하여 우선순위를 주는 정적 알고리즘
- **EDF (Earliest Deadline First)**: 마감 시간에 따라 우선순위를 주는 동적 알고리즘
- **최악 실행 시간 (WCET)**: 주기적 태스크 모델의 't'를 결정하는 핵심 데이터
