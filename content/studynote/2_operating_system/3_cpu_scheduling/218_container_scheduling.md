# 컨테이너 스케줄링 (Container Scheduling via cgroups)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 리눅스 커널의 제어 그룹 (cgroups) 기능을 활용하여, 컨테이너 단위로 CPU 점유 시간(Quota)과 상대적 가중치(Shares)를 할당함으로써 멀티 테넌트 환경의 자원 격리를 실현하는 스케줄링 기술이다.
> 2. **가치**: 특정 컨테이너가 CPU 자원을 독점하여 다른 서비스에 영향을 주는 '시끄러운 이웃 (Noisy Neighbor)' 문제를 방지하고, 약속된 서비스 품질 (QoS)을 하드웨어 수준에서 강제한다.
> 3. **융합**: 도커(Docker), 쿠버네티스(Kubernetes)의 리소스 제한 설정, 리눅스 CFS의 대역폭 제어(Bandwidth Control) 메커니즘이 통합된 클라우드 네이티브의 핵심 기술이다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 컨테이너 스케줄링은 "어떤 컨테이너가 얼마나 많은 CPU를 쓸 수 있는가"를 결정한다. 리눅스 커널은 각 컨테이너를 하나의 cgroup으로 묶고, CFS 스케줄러를 통해 이 그룹에 할당된 예산(Quota)만큼만 CPU를 사용하도록 제한한다.

- **💡 비유**: 컨테이너 스케줄링은 "공유 주방의 가스레인지 사용 시간 배분"과 같다. 여러 요리사(컨테이너)가 주방을 같이 쓰는데, 관리자(cgroups)가 'A 요리사는 1시간에 10분만 가스불을 켜세요' 혹은 'B 요리사는 C 요리사보다 2배 더 자주 불을 쓸 수 있습니다'라고 규칙을 정해두는 것과 같다.

- **📢 섹션 요약 비유**: 하나의 거대한 케이크(CPU 자원)를 여러 조각으로 미리 나누고, 각자 자기 접시에 담긴 몫(cgroups 제약)만 먹게 하는 공정한 배급 시스템과 같습니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### cgroups CPU 제어의 두 가지 방식

컨테이너의 리소스 제한은 보통 두 가지 파라미터로 조절된다.

1. **CPU Shares (상대적 가중치)**:
   - CPU가 100% 사용 중일 때, 그룹 간의 상대적인 배분 비율을 결정한다.
   - 예: A그룹 Shares=1024, B그룹 Shares=512 -> A가 B보다 2배 더 많이 쓴다.
   - CPU가 한가할 때는 제한 없이 더 쓸 수 있는 '유연한' 방식이다.

2. **CPU Quota/Period (절대적 제한)**:
   - 특정 주기(Period) 동안 사용할 수 있는 최대 시간(Quota)을 명시한다.
   - 예: Period=100ms, Quota=20ms -> 이 컨테이너는 어떤 경우에도 CPU 코어 0.2개분 이상의 성능을 낼 수 없다.

```text
  [ CFS Scheduler ]
         |
  +------+------+------- (Bandwidth Enforcement)
  |             |
  V             V
[cgroup A]    [cgroup B]
(Shares: 1024) (Quota: 50ms/100ms)
  |             |
  +-- [T1, T2]  +-- [T3]
```

**[다이어그램 해설]** 스케줄러는 각 cgroup이 사용한 총 시간을 추적한다. 만약 cgroup B가 이번 주기(100ms)에 50ms를 다 썼다면, 다음 주기가 시작될 때까지 cgroup B에 속한 모든 스레드(T3)의 실행을 강제로 중단(Throttling)시킨다.

- **📢 섹션 요약 비유**: Shares는 '형님이 동생보다 2배 더 먹어라'는 권고이고, Quota는 '너는 딱 한 공기만 먹고 숟가락 내려놔라'는 엄격한 통제입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### CPU Shares vs CPU Quota

| 비교 항목 | CPU Shares (Relative) | CPU Quota (Absolute) |
|:---|:---|:---|
| **제한 성격** | Soft Limit (최소 보장 지향) | Hard Limit (최대 한도 강제) |
| **자원 활용** | 유휴 자원을 빌려 쓸 수 있음 | 유휴 자원이 있어도 못 씀 |
| **예측 가능성** | 낮음 (다른 그룹 부하에 영향받음) | 높음 (언제나 일정 성능 유지) |
| **쿠버네티스 매핑** | `cpu.requests` | `cpu.limits` |

- **📢 섹션 요약 비유**: Shares는 '여유 있으면 더 타도 되는 자유 이용권'이고, Quota는 '딱 10분만 타야 하는 시간 제한 입장권'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 자바(Java) 앱의 컨테이너 Throttling 미스터리
- **상황**: 컨테이너 CPU 사용률은 50%인데, 자바 애플리케이션의 응답 지연이 심해지고 커널 로그에 `cpu.stat`의 `nr_throttled` 수치가 급증함.
- **판단**: 자바 VM은 실행 시 호스트의 전체 코어 수를 보고 많은 수의 가비지 컬렉션(GC) 스레드를 생성한다. 짧은 시간(Period) 동안 수많은 스레드가 동시에 CPU를 사용하면 순식간에 Quota를 다 써버려 스케줄러에 의해 강제로 멈춰버리는 현상이다. 해결을 위해 `-XX:ActiveProcessorCount` 옵션으로 컨테이너 할당량에 맞게 스레드 수를 제한하거나, 스케줄링 주기(Period)를 늘려 '짧고 굵은' 폭주를 수용해야 한다.

- **📢 섹션 요약 비유**: 한 달 용돈(Quota)을 하루 만에 다 써버려서 남은 한 달을 굶고 있는 상황이니, 하루 일당을 정해주거나 용돈 주기를 바꿔야 합니다.

## Ⅴ. 기대효과 및 결론

- **기대효과**: 하나의 물리 서버에 수백 개의 독립적인 서비스를 안전하게 격리 수용하여 클라우드 인프라의 집적도와 경제성을 극대화한다.
- **미래 전망**: 서버리스(Serverless) 컴퓨팅이 확산됨에 따라, 밀리초(ms)보다 더 정교한 마이크로초(us) 단위의 초미세 리소스 스케줄링 기술이 cgroups v2를 중심으로 발전할 것이다.

- **📢 섹션 요약 비유**: 보이지 않는 울타리(cgroups)가 빽빽한 아파트(클라우드) 생활을 평화롭게 만드는 마법의 규칙입니다.
