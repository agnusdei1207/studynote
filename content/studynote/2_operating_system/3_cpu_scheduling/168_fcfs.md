# FCFS (First-Come, First-Served) 스케줄링

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: FCFS (First-Come, First-Served) 스케줄링은 준비 큐(Ready Queue)에 **도착한 순서대로 CPU를 할당**하는 가장 단순한 형태의 비선점형(Non-preemptive) 스케줄링 알고리즘이다.
> 2. **가치**: 구현이 매우 쉽고 스케줄링 오버헤드가 거의 없으며, 프로세스의 실행 순서가 도착 시간과 동일하여 기아 현상(Starvation)이 발생하지 않는다는 공정성을 가진다.
> 3. **융합**: 시분할 시스템에서는 단독으로 쓰이기 힘들지만, 선점형 스케줄링 알고리즘(예: 라운드 로빈)의 내부적인 대기열 관리 기법으로 널리 활용된다.

---

## Ⅰ. FCFS 스케줄링의 개요

### 1. 정의
- CPU를 요청한 프로세스 순서대로 CPU를 할당하는 방식이다. FIFO(First-In-First-Out) 큐를 사용하여 관리한다.

### 2. 특징: '기다린 만큼 보상받는 클래식'
- **비선점형**: 한 번 CPU를 잡으면 작업이 끝날 때까지 뺏기지 않는다.
- **공정성**: 먼저 온 프로세스가 무조건 먼저 실행되므로 무한정 기다리는 프로세스가 없다.
- **효율성 가변성**: 프로세스들이 도착하는 순서에 따라 평균 대기 시간이 크게 달라진다.

### 3. 💡 비유: '은행 창구 줄 서기'
- FCFS는 **'번호표 없이 그냥 온 순서대로 줄을 서서 업무를 보는 은행 창구'**와 같다.
- 내 앞에 아주 오래 걸리는 대출 상담 고객(CPU Bound)이 있다면, 나는 간단한 통장 정리(I/O Bound)만 하면 되는데도 그 사람이 끝날 때까지 뒤에서 하염없이 기다려야 한다.

---

## Ⅱ. FCFS의 성능 분석 (Deep Dive)

### 1. 평균 대기 시간의 가변성 (Gantt Chart 분석)
- **Case 1 (긴 작업 먼저)**: P1(24ms), P2(3ms), P3(3ms) 순으로 도착.
  - 대기 시간: P1(0), P2(24), P3(27) ──▶ 평균 대기 시간: **17ms**
- **Case 2 (짧은 작업 먼저)**: P2(3ms), P3(3ms), P1(24ms) 순으로 도착.
  - 대기 시간: P2(0), P3(3), P1(6) ──▶ 평균 대기 시간: **3ms**
- **결론**: 도착 순서에 따라 성능 차이가 5배 이상 날 수 있다.

### 2. 호위 효과 (Convoy Effect)
- CPU 사용 시간이 긴 하나의 프로세스로 인해 CPU 사용 시간이 짧은 여러 프로세스가 오랫동안 기다리게 되어, 결과적으로 **CPU와 장치 이용률이 저하되는 현상**이다.

### 3. ASCII 구조 다이어그램
```text
    [ Ready Queue (FIFO) ]          [ CPU ]
    ┌───────┐  ┌───────┐            ┌───────────────┐
    │ P3(3) │  │ P2(3) │ ──(Next)─▶ │ P1 (24ms)     │ ◀── 현재 실행 중
    └───────┘  └───────┘            │ (Running)     │
                                    └──────┬────────┘
                                           │ (Finish)
                                           ▼
                                    [ Next: P2 ]
```

- **📢 섹션 요약 비유**: 좁은 1차선 도로(CPU)에서 앞에 느린 트럭(긴 프로세스)이 가고 있으면, 뒤에 빠른 스포츠카(짧은 프로세스)들이 줄줄이 밀려 속도를 내지 못하는 것과 같습니다.

## Ⅲ. 장점과 단점 요약

| 구분 | 장점 (Pros) | 단점 (Cons) |
|:---|:---|:---|
| **복잡도** | 최하 (큐 하나면 구현 완료) | **응답 시간**이 매우 길어질 수 있음 |
| **오버헤드** | 거의 없음 (전환이 적음) | **호위 효과**로 인한 자원 낭비 |
| **안정성** | 기아 현상(Starvation) 없음 | 대화형 시스템에 부적합 |

- **📢 섹션 요약 비유**: 규칙이 단순해서 싸울 일은 없지만(장점), 효율성 면에서는 융통성이 전혀 없는 융통성 제로의 관리 방식입니다.

---

## Ⅳ. 실무 적용 및 현대적 가치 (Strategy)

### 1. 실무 시나리오: 일괄 처리 시스템 (Batch Processing)
- **상황**: 급여 계산, 영수증 발행 등 대량의 정적 데이터를 처리하는 시스템.
- **판단**: 사용자와의 인터랙션이 없고, 모든 작업을 순차적으로 끝내는 것이 목적이다. 이때 **FCFS**는 문맥 교환 오버헤드를 최소화하여 순수 연산 성능을 높이는 가장 효율적인 선택이 된다.

### 2. 기술사적 인사이트: 알고리즘의 기초 체력
- FCFS는 그 자체로 쓰이기보다 다른 알고리즘의 **부분적 구성 요소**로 의미가 크다. 기술사라면 라운드 로빈(RR)의 각 타임 슬라이스 내에서 프로세스를 선택하는 기본 질서가 FCFS임을 이해하고, 복잡한 현대적 스케줄러(CFS 등)의 밑바닥에 흐르는 '도착 순서에 대한 존중'이라는 공정성 철학을 언급할 수 있어야 한다.

- **📢 섹션 요약 비유**: 가장 기초적인 근력 운동(FCFS)이 되어 있어야 화려한 기술(복잡한 알고리즘)도 빛을 발하는 것과 같습니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **시스템 단순화**: 하드웨어 리소스가 극도로 적은 임베디드 환경에서 안정적 구동.
- **공정성 보장**: 예측 가능한 서비스 순서 제공.

### 2. 미래 전망
최근 하이퍼스케일 컴퓨팅에서는 특정 노드의 작업을 아주 단순하게 처리하기 위해 다시금 FCFS 방식의 경량 스케줄러가 연구되기도 한다. 비록 시분할 OS의 메인 알고리즘 자리는 내주었지만, 선입선출이라는 대원칙은 데이터 구조와 운영체제 설계의 영원한 '근본'으로 남을 것이다.

- **📢 섹션 요약 비유**: 최첨단 내비게이션(현대 알고리즘)이 나와도, 지도의 기본(도착 순서)은 변하지 않는 것과 같습니다.
