# 코-스케줄링 (Co-scheduling / Gang Scheduling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서로 밀접하게 통신하거나 자원을 공유하는 스레드 그룹(Gang)을 여러 개의 CPU 코어에 동시에(Simultaneously) 할당하여 함께 실행시키는 스케줄링 기법이다.
> 2. **가치**: 스레드 간의 빈번한 동기화(Lock, Barrier) 시 발생하는 대기 시간을 줄여 병렬 컴퓨팅 효율을 극대화하며, 최근에는 하이퍼스레딩 보안 취약점 대응을 위한 'Core Scheduling'으로 재조명받고 있다.
> 3. **융합**: 병렬 처리 알고리즘, 고성능 컴퓨팅 (HPC), 클라우드 가상화의 vCPU 관리, 그리고 시스템 보안 격리 기술이 융합된 영역이다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **Gang Scheduling**: 하나의 작업(Job)에 속한 모든 스레드를 동시에 실행시켜, 스레드 A가 스레드 B를 기다리느라 CPU를 낭비하는 일을 막는다.
  - **Core Scheduling**: 동일한 물리 코어 내의 하이퍼스레드들이 서로 신뢰할 수 있는 그룹에 속할 때만 동시에 실행되도록 하여 사이드 채널 공격을 차단한다.

- **💡 비유**: 코-스케줄링은 "2인 3각 경기"와 같다. 두 사람(스레드)이 발이 묶여 있는데, 한 사람만 먼저 출발(스케줄링)하면 다른 사람은 끌려가거나 넘어지게 된다. 두 사람이 동시에 신호에 맞춰 출발해야 제대로 경기를 할 수 있는 것과 같다.

- **📢 섹션 요약 비유**: 뿔뿔이 흩어져서 일하는 게 아니라, 손발을 맞춰야 하는 팀원들을 한자리에 모아 동시에 업무를 시작하게 하는 '팀 단위 소집 시스템'입니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 병렬 처리에서의 코-스케줄링 메커니즘

서로 동기화가 잦은 4개의 스레드가 4개의 코어에 배치되는 상황을 가정한다.

```text
  [ Time Slot ]    [ Core 0 ]    [ Core 1 ]    [ Core 2 ]    [ Core 3 ]
  
  T1 (Sync) ---->  [ Thread A ]  [ Thread B ]  [ Thread C ]  [ Thread D ]
                   (함께 실행되므로 Barrier 도달 시 즉시 통과)
  
  T2 (Switch) ---> [ Other X ]   [ Other Y ]   [ Other Z ]   [ Other W ]
                   (팀 전체가 한꺼번에 물러남)
```

**[다이어그램 해설]** 
1. **동시성 보장**: 스레드 A가 B로부터 데이터를 받아야 할 때, B가 다른 코어에서 실행 중이지 않다면 A는 B가 스케줄링될 때까지 하염없이 기다려야 한다 (Spin-wait 낭비). 코-스케줄링은 이를 원천 차단한다.
2. **단점 (Fragmentation)**: 만약 코어가 4개인데 팀원이 3명뿐이라면, 남는 1개의 코어는 효율적으로 쓰기 어려워지는 '공간적 단편화'가 발생할 수 있다.

### 보안을 위한 코-스케줄링 (Core Scheduling)
최근 리눅스 커널에 도입된 기술로, 하이퍼스레딩(SMT) 환경에서 동일 물리 코어를 공유하는 두 논리 CPU가 서로 다른 보안 도메인(예: 사용자 앱과 커널, 혹은 다른 VM)의 작업을 수행하지 못하게 강제한다. 오직 '같은 팀'일 때만 코어 공유를 허용한다.

- **📢 섹션 요약 비유**: 같이 밥을 먹어야 할 사람들은 한 식탁에 앉히고(병렬 효율), 모르는 사람끼리는 한 방을 쓰지 못하게 칸막이를 치는(보안 격리) 이중적인 관리 기법입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 코-스케줄링 vs 독립 스케줄링

| 비교 항목 | 코-스케줄링 (Co-scheduling) | 독립 스케줄링 (Independent) |
|:---|:---|:---|
| **스케줄링 단위** | 스레드 그룹 (Gang) | 개별 스레드 |
| **통신 오버헤드** | 매우 낮음 (동기화 즉시 완료) | 높음 (상대방 스케줄링 대기) |
| **자원 활용도** | 낮음 (빈 공간 발생 가능) | 높음 (빈틈없이 채움) |
| **주요 목적** | 고성능 병렬 처리, 보안 격리 | 전체 시스템 처리량(Throughput) 극대화 |

- **📢 섹션 요약 비유**: 코-스케줄링은 '단체 전세 버스'이고, 독립 스케줄링은 '각자 타는 지하철'입니다. 단체 버스는 한 명이라도 늦으면 다 같이 늦지만, 도착해서는 한꺼번에 이동할 수 있어 편리합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 클라우드 SMT 보안 패치
- **상황**: 인텔 CPU의 L1TF 보안 취약점으로 인해 동일 코어 내 데이터 유출 위험 발생. 클라우드 사업자는 SMT를 끄자니 성능이 30% 하락하고, 켜두자니 보안이 걱정되는 딜레마에 빠짐.
- **판단**: 리눅스 커널의 **Core Scheduling** 기능을 활성화해야 한다. 이를 통해 서로 다른 사용자의 VM 스레드가 동일한 물리 코어에서 동시에 돌아가는 것을 원천 봉쇄하면서도, 같은 사용자의 스레드끼리는 코어를 공유하게 하여 SMT의 성능 이점을 어느 정도 유지(약 5-10% 성능 하락으로 방어)하는 절충안을 선택해야 한다.

- **📢 섹션 요약 비유**: 모르는 사람과는 합석 금지! 하지만 가족끼리는 합석 가능! 이라고 써 붙여서 식당 운영 효율과 방역(보안)을 동시에 잡는 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 슈퍼컴퓨터(HPC) 급의 대규모 연산에서 스레드 간 동기화 병목을 제거하여 성능을 수십 배 향상시키며, 현대적인 클라우드 인프라의 보안 수준을 한 단계 끌어올린다.
- **미래 전망**: 엣지 컴퓨팅에서 실시간성(RT)이 필요한 작업 그룹을 묶어서 처리하는 '실시간 코-스케줄링' 기술이 자율주행차 등의 제어 분야에서 핵심이 될 것이다.

- **📢 섹션 요약 비유**: 혼자 가면 빨리 가지만, 함께 가면 멀리(성능과 보안) 갈 수 있음을 보여주는 지혜로운 스케줄링입니다.
