# 186. 스케줄링 지연 (Scheduling Latency)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 태스크가 실행 가능해진 후 실제 실행까지의 대기 시간
> 2. **가치**: 실시간 및 대화형 시스템의 핵심 성능 지표
> 3. **융합**: 디스패치 지연, 문맥 교환, 선점과 연관

---

## Ⅰ. 개요

### 개념 정의

스케줄링 지연(Scheduling Latency)은 **프로세스가 실행 가능 상태가 된 시점부터 실제 CPU를 할당받아 실행되기까지의 시간**이다. 스케줄러 오버헤드와 시스템 부하의 척도다.

### 💡 비유: 식당 주문 대기
스케줄링 지연은 **식당에서 주문 후 음식이 나올 때까지의 대기 시간**과 같다. 주문(실행 요청) 후 서빙(실제 실행)까지 걸리는 시간이다.

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                스케줄링 지연 상세                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【실시간 시스템 요구사항】                                              │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  시스템 유형              최대 허용 지연                          │ │
│  │  ────                        ────                                 │ │
│  │  경성 실시간 (Hard RT)     ~10-100 µs                            │ │
│  │  연성 실시간 (Soft RT)     ~100-1000 µs                          │ │
│  │  대화형                   ~10-50 ms                               │ │
│  │  일반                     ~100+ ms                                │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【지연 최소화 기법】                                                    │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 선점형 커널 (PREEMPT_RT)                                    │ │
│  │     - 커널 실행 중에도 선점 가능                                 │ │
│  │     - 지연 크게 감소                                            │ │
│  │                                                               │ │
│  │  2. 높은 우선순위                                               │ │
│  │     - 실시간 스케줄링 (SCHED_FIFO, SCHED_RR)                    │ │
│  │     - 일반 태스크보다 먼저 실행                                  │ │
│  │                                                               │ │
│  │  3. CPU 고정 (Affinity)                                         │ │
│  │     - 캐시 지역성 유지                                          │ │
│  │     - 마이그레이션 지연 방지                                     │ │
│  │                                                               │ │
│  │  4. 인터럽트 스레드화                                           │ │
│  │     - 인터럽트 핸들러를 스레드로                                  │ │
│  │     - 우선순위 기반 스케줄링 가능                                │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【Linux PREEMPT_RT】                                                   │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  CONFIG_PREEMPT_NONE     : 서버용, 높은 처리량                   │ │
│  │  CONFIG_PREEMPT_VOLUNTARY: 일반용, 균형                         │ │
│  │  CONFIG_PREEMPT          : 데스크탑, 낮은 지연                   │ │
│  │  CONFIG_PREEMPT_RT       : 실시간, 최저 지연                     │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 측정 방법
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【스케줄링 지연 측정 도구】                                             │
│  ──────────────────                                                  │
│  # cyclictest - 실시간 지연 측정                                      │
│  $ sudo cyclictest -t1 -p 80 -n -i 10000 -l 10000                   │
│  # -t: 스레드 수, -p: 우선순위, -n: clock_nanosleep                   │
│  # -i: 간격(µs), -l: 루프 횟수                                        │
│                                                                     │
│  # 결과 예시                                                         │
│  # Min: 1 µs, Act: 3 µs, Avg: 4 µs, Max: 50 µs                     │
│                                                                     │
│  # ftrace로 스케줄링 이벤트 추적                                       │
│  $ trace-cmd record -e sched:sched_switch -e sched:sched_wakeup \   │
│      ./program                                                      │
│  $ trace-cmd report                                                 │
│                                                                     │
│  【커널 설정 확인】                                                      │
│  ──────────────────                                                  │
│  # 선점 모드 확인                                                     │
│  $ cat /boot/config-$(uname -r) | grep PREEMPT                      │
│  CONFIG_PREEMPT=y                                                   │
│  # 또는                                                              │
│  $ uname -v                                                         │
│  # ... PREEMPT RT ...                                               │
│                                                                     │
│  # 실시간 그룹 스케줄링                                                │
│  $ cat /proc/sys/kernel/sched_rt_runtime_us                         │
│  $ cat /proc/sys/kernel/sched_rt_period_us                          │
│                                                                     │
│  【실시간 태스크로 실행】                                                │
│  ──────────────────                                                  │
│  # SCHED_FIFO로 실행                                                 │
│  $ sudo chrt -f 80 ./realtime_task                                  │
│                                                                     │
│  # CPU 고정                                                          │
│  $ sudo taskset -c 0 chrt -f 80 ./realtime_task                     │
│                                                                     │
│  【지연 분석 코드】                                                      │
│  ──────────────────                                                  │
│  #include <time.h>                                                   │
│  #include <sched.h>                                                  │
│                                                                     │
│  void measure_scheduling_latency() {                                 │
│      struct timespec start, end;                                     │
│      struct sched_param param = { .sched_priority = 80 };            │
│      sched_setscheduler(0, SCHED_FIFO, &param);                      │
│                                                                     │
│      for (int i = 0; i < 1000; i++) {                                │
│          clock_gettime(CLOCK_MONOTONIC, &start);                     │
│          sched_yield();  // CPU 양보 후 다시 스케줄                   │
│          clock_gettime(CLOCK_MONOTONIC, &end);                       │
│                                                                     │
│          long latency = (end.tv_sec - start.tv_sec) * 1000000000 +   │
│                         (end.tv_nsec - start.tv_nsec);               │
│          printf("Latency: %ld ns\n", latency);                       │
│      }                                                               │
│  }                                                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 실행 가능 → 실제 실행까지의 대기 시간
• 구성: 인터럽트 + 웨이크업 + 스케줄링 + 문맥 교환
• 일반: 5-40 µs
• 실시간: 1-10 µs 목표
• 영향: 부하, 알고리즘, 우선순위
• 최소화: 선점형 커널, 높은 우선순위, CPU 고정
```
