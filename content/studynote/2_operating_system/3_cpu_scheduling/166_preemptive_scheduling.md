# 선점형 스케줄링 (Preemptive Scheduling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 선점형 스케줄링 (Preemptive Scheduling)은 운영체제가 프로세스로부터 **CPU 제어권을 강제로 회수**하여 다른 프로세스에게 할당할 수 있는 권한을 가진 동적인 스케줄링 방식이다.
> 2. **가치**: 특정 프로세스의 CPU 독점을 방지하여 모든 프로세스에게 공평한 실행 기회를 제공하며, 대화형 시스템(Interactive)에서 높은 응답성과 실시간성을 보장한다.
> 3. **융합**: 하드웨어 타이머 인터럽트와 밀접하게 연동되며, 문맥 교환(Context Switch) 오버헤드와 데이터 일관성 유지라는 기술적 과제를 수반하는 현대 범용 운영체제의 표준 방식이다.

+++

## Ⅰ. 선점형 스케줄링의 개요

### 1. 정의
- 현재 실행 중인 프로세스보다 우선순위가 높은 프로세스가 나타나거나, 할당된 시간(Time Slice)이 만료되었을 때 운영체제가 강제로 실행을 중단시키고 CPU를 다른 프로세스에게 넘겨주는 방식이다.

### 2. 특징: '커널의 강력한 통제'
- **강제성**: 프로세스의 의사와 상관없이 CPU 제어권이 박탈될 수 있다.
- **높은 응답성**: 빠른 전환을 통해 사용자에게 여러 프로그램이 동시에 실행되는 듯한 환경을 제공한다.
- **오버헤드 존재**: 빈번한 문맥 교환(Context Switch)으로 인해 시스템 자원이 소모된다.

### 3. 💡 비유: '제한 시간이 있는 놀이기구'
- 선점형 스케줄링은 **'타이머를 맞춰놓고 정해진 시간이 되면 강제로 내려야 하는 놀이기구'**와 같다.
- 내가 더 타고 싶어도 뒤에 줄을 선 친구들(준비 큐)을 위해 관리자(운영체제)가 호각(타이머 인터럽트)을 불어 나를 내리게 하고 다음 친구를 태우는 방식이다.

+++

## Ⅱ. 선점형 스케줄링의 주요 알고리즘 (Deep Dive)

### 1. 라운드 로빈 (Round Robin, RR)
- 각 프로세스에게 동일한 시간 할당량(Time Quantum)을 부여하고 순서대로 실행한다. 가장 대표적인 선점형 방식이다.

### 2. SRTF (Shortest Remaining Time First)
- SJF의 선점형 버전이다. 현재 실행 중인 작업의 남은 시간보다 더 짧은 작업이 들어오면 CPU를 즉시 교체한다.

### 3. 다단계 피드백 큐 (MLFQ)
- 프로세스의 행동 패턴(I/O 바운드 vs CPU 바운드)에 따라 우선순위를 동적으로 조정하며 선점한다.

### 4. ASCII 구조 다이어그램
```text
    [ Ready Queue ]             [ CPU ]                 [ Execution Flow ]
    ┌───────────┐               ┌───────────┐           P1 (Start)
    │ P2 (Wait) │               │           │           ■■■ (Time Out!)
    ├───────────┤ ──(Assign)──▶ │    P1     │           P1 (Saved to Queue)
    │ P3 (Wait) │ ◀─(Preempt)── │ (Running) │           P2 (Dispatch)
    └───────────┘               └───────────┘           ■■■ (Time Out!)
                                                        P3 (Dispatch)...
```

- **📢 섹션 요약 비유**: 빵 하나를 한 사람이 다 먹게 두지 않고, 한 입씩 번갈아 먹게 하여 모두가 굶지 않게 관리하는 공평한 배급 시스템과 같습니다.

+++

## Ⅲ. 선점형 vs 비선점형 기술적 쟁점

| 비교 항목 | 선점형 (Preemptive) | 비선점형 (Non-preemptive) |
|:---|:---|:---|
| **응답 시간** | **빠름 (시분할 적합)** | 느림 (일괄 처리 적합) |
| **시스템 복잡도** | **매우 높음 (동기화 필요)** | 낮음 |
| **자원 활용** | CPU 유휴 시간 최소화 | 문맥 교환 오버헤드 최소화 |
| **데이터 정합성** | **공유 자원 보호(Lock) 필수** | 상대적으로 안전함 |

- **📢 섹션 요약 비유**: 일을 하다가 수시로 전화를 받아야 하는 상황(선점형)에서는 하던 일의 위치를 계속 표시(Context Save)해야 하고 흐름이 끊기지만, 급한 용무를 즉시 처리할 수 있는 장점이 있습니다.

+++

## Ⅳ. 실무 적용 및 설계상의 도전 (Strategy)

### 1. 실무 시나리오: 커널 수준의 선점 (Preemptive Kernel)
- **상황**: 시스템 콜을 수행 중인 커널 모드에서도 긴급한 인터럽트를 즉시 처리해야 함.
- **판단**: 과거 리눅스 커널은 비선점형이었으나, 현대 리눅스는 **선점형 커널 (CONFIG_PREEMPT)** 옵션을 제공한다. 커널 코드가 실행 중이라도 안전한 지점(Safe Point)이라면 즉시 다른 프로세스로 전환할 수 있게 설계하여 실시간 응답성을 극대화한다.

### 2. 기술사적 인사이트: 공유 데이터와 Race Condition
- 선점형 스케줄링의 가장 큰 기술적 과제는 **데이터 무결성**이다. 어떤 프로세스가 공유 변수를 수정하는 도중에 선점당하고 다른 프로세스가 그 변수를 읽으면 데이터가 깨진다. 기술사라면 이를 해결하기 위해 뮤텍스(Mutex), 스핀락(Spinlock) 등의 **상호 배제 (Mutual Exclusion)** 기법이 선점형 OS 설계의 필수 요소임을 역설해야 한다.

- **📢 섹션 요약 비유**: 서류를 작성하다가 갑자기 자리를 비워야 할 때, 서류 내용이 바뀌지 않도록 금고에 넣고 잠그는(동기화) 절차가 반드시 따라와야 합니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **공정성 증대**: 모든 프로세스에게 실행 기회를 균등하게 배분.
- **사용자 만족도 향상**: 멀티미디어 재생, 웹 서핑 등 실시간 반응이 중요한 작업 최적화.

### 2. 미래 전망
최근에는 멀티코어와 가속기가 복잡하게 얽힌 환경에서 성능과 전력 사이의 균형을 맞추기 위해 **지능형 선점 (Intelligent Preemption)** 기술이 발전하고 있다. 단순히 시간을 다 써서 뺏는 것이 아니라, 전력 효율이 더 좋은 코어로 옮기기 위해 선점하거나 캐시 적중률이 높은 시점까지 기다려주는 정교한 선점 전략이 운영체제의 새로운 경쟁력이 될 것이다.

- **📢 섹션 요약 비유**: 억지로 뺏는 것이 아니라, 모두가 윈-윈(Win-win)할 수 있도록 최적의 타이밍에 바톤을 넘기는 스마트한 계주 경주와 같습니다.
