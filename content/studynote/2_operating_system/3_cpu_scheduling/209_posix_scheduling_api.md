# POSIX 스케줄링 API (SCHED_FIFO, SCHED_RR, SCHED_OTHER)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 운영체제 간의 호환성을 위해 POSIX (Portable Operating System Interface) 표준에서 정의한 스케줄링 정책 및 제어 인터페이스로, 실시간 및 일반 태스크의 우선순위와 실행 방식을 프로그래밍 언어 수준에서 제어할 수 있게 한다.
> 2. **가치**: 개발자가 특정 운영체제에 종속되지 않고 실시간 응답성이 필요한 애플리케이션을 구현할 수 있도록 표준화된 도구를 제공하며, 시스템의 결정성 (Determinism)을 제어하는 핵심 수단이다.
> 3. **융합**: Pthreads (POSIX Threads) 라이브러리, 실시간 운영체제 (RTOS)의 스케줄링 구조, 그리고 임베디드 시스템의 성능 최적화 전략과 밀접하게 연계된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: POSIX.1b 표준은 실시간 확장을 위해 세 가지 주요 스케줄링 정책을 정의한다. 비실시간 기본 정책인 `SCHED_OTHER`와 실시간 정책인 `SCHED_FIFO`, `SCHED_RR`이다. 이를 통해 프로세스나 스레드의 우선순위를 동적으로 설정하고 스케줄링 동작을 변경할 수 있다.

- **💡 비유**: POSIX 스케줄링 API는 "식당의 손님 관리 매뉴얼"과 같다. 일반 손님(OTHER), 선착순 VIP(FIFO), 시간제 VIP(RR) 등 손님을 어떻게 대접할지에 대한 전 세계 공통의 규칙을 정해둔 것이다.

- **📢 섹션 요약 비유**: 어떤 기종의 비행기를 타더라도 조종석의 조작 버튼(API)이 표준화되어 있어 조종사가 안정적으로 운행할 수 있게 돕는 표준 가이드와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### POSIX의 3대 스케줄링 정책

운영체제는 각 정책별로 별도의 준비 큐를 유지하거나 우선순위 범위 내에서 다르게 처리한다.

```text
  [ Real-time Range (1-99) ]          [ Normal Range (0) ]
  
  SCHED_FIFO: [ T1 ] -> [ T2 ]        SCHED_OTHER: [ T5 ]
  (먼저 온 놈이 끝날 때까지)              (커널이 알아서 공평하게)
  
  SCHED_RR:   [ T3 ] -> [ T4 ]
  (시간 할당량만큼씩 번갈아)
```

**[다이어그램 해설]** 
1. **SCHED_FIFO (First-In, First-Out)**: 가장 높은 우선순위의 프로세스가 CPU를 잡으면, 본인이 스스로 양보(`yield`)하거나 I/O 블로킹이 발생하기 전까지는 CPU를 절대 놓지 않는다.
2. **SCHED_RR (Round Robin)**: FIFO와 비슷하지만, 각 프로세스에 시간 할당량 (Time Quantum)이 주어지며 시간이 다 되면 동일 우선순위의 다른 프로세스에게 차례를 넘긴다.
3. **SCHED_OTHER**: 기본 시분할 정책. 실시간성이 보장되지 않으며 커널의 동적 우선순위 알고리즘에 의해 관리된다.

### 주요 API 함수
- `sched_setscheduler()`: 프로세스의 정책과 우선순위를 설정한다.
- `sched_setparam()`: 이미 실행 중인 프로세스의 우선순위만 변경한다.
- `sched_get_priority_max/min()`: 해당 정책에서 사용 가능한 우선순위 범위를 확인한다.
- `sched_yield()`: 현재 CPU 점유를 포기하고 다른 프로세스에게 양보한다.

- **📢 섹션 요약 비유**: FIFO는 한 명씩 끝까지 진료하는 완고한 의사이고, RR은 환자마다 5분씩만 상담하고 다시 줄 세우는 바쁜 의사입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SCHED_FIFO vs SCHED_RR

| 비교 항목 | SCHED_FIFO | SCHED_RR |
|:---|:---|:---|
| **중단 시점** | 자발적 종료, I/O 대기, 더 높은 우선순위 등장 | FIFO 조건 + 시간 할당량 (Time Slice) 만료 |
| **적합한 작업** | 짧고 빠른 응답이 필요한 긴급 작업 | 동일 중요도의 실시간 작업이 여러 개일 때 |
| **위험 요소** | 무한 루프 시 시스템 전체가 먹통이 됨 | 문맥 교환 오버헤드 발생 가능 |
| **공정성** | 낮음 (독점 가능) | 중간 (동일 순위 내 공정) |

- **📢 섹션 요약 비유**: FIFO는 1등이 다 먹는 승자독식 구조이고, RR은 1등 그룹 내에서 사이좋게 나눠 먹는 구조입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 제어 루프 시스템의 먹통 (Hang) 현상
- **상황**: 제어 알고리즘을 `SCHED_FIFO` 최고 우선순위로 설정했는데, 알고리즘 내부에 조건문 오류로 무한 루프가 발생하여 키보드, 마우스 등 시스템의 모든 입력이 마비됨.
- **판단**: 실시간 프로세스는 시스템의 모든 자원을 독점할 수 있으므로 매우 주의해서 설계해야 한다. 개발 단계에서는 `SCHED_RR`을 사용하거나, 실시간 프로세스가 CPU를 너무 오래 점유하면 강제로 중단시키는 `RLIMIT_RTTIME` 설정을 적용하여 시스템 붕괴를 방지하는 안정 장치를 두어야 한다.

- **📢 섹션 요약 비유**: 사이렌을 울리며 달리는 구급차(실시간)는 빨간불도 지나갈 수 있지만, 만약 구급차가 길 한복판에 멈춰 서버리면 전체 교통이 마비되는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

- **기대효과**: 운영체제 수준의 강력한 우선순위 제어를 통해 응답 시간을 확정적으로 관리할 수 있으며, 코드의 이식성 (Portability)을 높여 개발 비용을 절감한다.
- **미래 전망**: 리눅스 커널 5.x 이후 도입된 `SCHED_DEADLINE` 정책과 결합하여, 단순히 우선순위만 따지는 것을 넘어 '주기(p)'와 '마감 시간(d)'을 직접 API로 제어하는 정교한 실시간 제어 시대로 나아가고 있다.

- **📢 섹션 요약 비유**: 표준화된 매뉴얼이 사고를 막고 효율을 높이듯, POSIX API는 프로그래머에게 시스템을 다루는 정교한 핸들을 제공합니다.
