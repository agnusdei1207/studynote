# 공정 공유 스케줄링 (Fair Share Scheduling, FSS)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 공정 공유 스케줄링 (Fair Share Scheduling, FSS)은 개별 프로세스 단위가 아닌, **사용자(User) 또는 프로세스 그룹(Group) 단위**로 CPU 자원의 일정 비율을 할당하고 보장하는 스케줄링 기법이다.
> 2. **가치**: 특정 사용자가 수많은 프로세스를 생성하여 시스템 자원을 독점하는 것을 방지하며, 다중 사용자 환경에서 각 사용자에게 부여된 권한(지분)에 따라 공평하게 자원을 분배한다.
> 3. **융합**: 유닉스 계열의 전통적인 우선순위 기반 스케줄링에 '그룹 가중치' 개념을 결합한 형태이며, 현대 클라우드 컴퓨팅의 멀티테넌시(Multi-tenancy) 자원 격리 기술의 근간이 된다.

---

## Ⅰ. FSS의 개요

### 1. 정의
- 시스템의 자원을 사용자나 그룹별로 할당된 지분(Share)에 따라 분배하는 방식이다. 프로세스의 개수와 상관없이 특정 사용자 그룹이 사용할 수 있는 CPU 총량을 제한한다.

### 2. 등장 배경: '쪽수 대결의 방지'
- 일반적인 스케줄러는 프로세스 하나당 기회를 준다. 만약 사용자 A는 프로세스 1개를 돌리고, 사용자 B는 99개를 돌린다면, B가 자원의 99%를 가져가게 된다. FSS는 이러한 불공평을 해결하기 위해 "사용자당 50%씩"과 같이 큰 틀의 제한을 둔다.

### 3. 💡 비유: '가족별 피자 배분'
- **일반 스케줄링**: "사람 수대로 피자를 한 조각씩 준다." 식구가 많은 집이 피자를 다 먹는다.
- **FSS**: "집집마다 피자를 반 판씩 준다." 식구가 한 명이든 열 명이든 그 집은 반 판(지분)만 먹을 수 있다. 식구가 많은 집은 자기들끼리 그 반 판을 다시 잘게 쪼개 먹어야 한다.

---

## II. FSS의 동작 메커니즘 (Deep Dive)

### 1. 계층적 스케줄링 구조
- **Level 1 (Group)**: 시스템 전체 CPU 자원을 사용자/그룹별 지분에 따라 나눈다.
- **Level 2 (Process)**: 각 그룹 내부에 할당된 자원을 해당 그룹 소속 프로세스들끼리 다시 나눈다.

### 2. 우선순위 계산 방식
- FSS는 기존 우선순위 공식에 '그룹 사용량' 항을 추가한다.
- **우선순위 = $Base + (Process\_Usage / K) + (Group\_Usage / L)$**
- 특정 그룹의 프로세스들이 CPU를 많이 쓸수록, 해당 그룹 전체의 우선순위가 낮아져 다른 그룹에게 기회가 돌아가게 설계된다.

### 3. ASCII 구조 다이어그램
```text
    [ Total CPU Resource (100%) ]
           │
    ┌──────┴───────────────────────────┐
    │      Fair Share Scheduler        │ (Divide by User/Group Shares)
    └──────┬──────────────┬────────────┘
           ▼ (50%)        ▼ (50%)
    ┌──────────────┐  ┌──────────────┐
    │   User A     │  │   User B     │ ◀── (User B creates 10 processes)
    │ [ 1 Process ]│  │ [ 10 Procs ] │ ◀── (But still gets only 50%)
    └──────────────┘  └──────────────┘
```

- **📢 섹션 요약 비유**: 각 부서에 예산(CPU 지분)을 먼저 배정하고, 부서원들이 그 예산 안에서만 활동하게 관리하는 기업의 예산 통제 시스템과 같습니다.

---

## Ⅲ. FSS의 주요 특징 및 분석

| 비교 항목 | 일반 프로세스 스케줄링 | 공정 공유 스케줄링 (FSS) |
|:---|:---|:---|
| **관리 단위** | 개별 프로세스 (PID) | **사용자 / 그룹 (UID / GID)** |
| **자원 독점** | 프로세스 많이 띄우면 가능 | **구조적으로 불가능** |
| **주 용도** | 개인용 PC, 단일 사용자 | **서버, 메인프레임, 클라우드** |
| **장점** | 단순하고 직관적임 | 다중 사용자 간 형평성 보장 |

- **📢 섹션 요약 비유**: 개인 플레이를 존중하는 것보다 팀 단위의 성과와 배분을 중시하는 공동체 운영 원칙과 같습니다.

---

## Ⅳ. 실무 적용 및 클라우드 전략 (Strategy)

### 1. 실무 시나리오: 공용 개발 서버 운영
- **상황**: 50명의 개발자가 한 대의 고성능 서버를 공유하여 컴파일 작업을 수행.
- **판단**: 특정 개발자가 무거운 빌드 스크립트를 여러 개 돌려도 다른 개발자의 터미널 응답이 느려지지 않아야 한다. **FSS**를 적용하여 인당 2%의 CPU 지분을 강제 할당한다. 이제 누가 수백 개의 프로세스를 띄우더라도 다른 사람들의 2% 영역은 침범받지 않아 쾌적한 개발 환경이 유지된다.

### 2. 기술사적 인사이트: 컨테이너 오케스트레이션과 FSS
- 현대의 쿠버네티스(K8s)나 도커의 **CPU Shares** 설정이 바로 FSS의 현대적 구현이다. 기술사라면 커널의 **Cgroups**를 활용해 그룹별 CPU 가중치를 설정하는 메커니즘을 FSS의 관점에서 설명할 수 있어야 한다. 이는 단순한 알고리즘을 넘어 클라우드 서비스의 과금 모델(SLA)을 뒷받침하는 핵심 기술임을 강조해야 한다.

- **📢 섹션 요약 비조**: "낸 만큼 쓴다"는 자본주의의 원리가 운영체제 스케줄링에 가장 잘 녹아있는 형태가 바로 FSS입니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **사용자 간 간섭 차단**: 한 사용자의 폭주가 시스템 전체의 마비로 이어지는 현상 방지.
- **예측 가능한 서비스**: 각 사용자에게 약속된 최소 성능 지표 보장.

### 2. 미래 전망
앞으로는 에지 컴퓨팅 환경에서 수많은 기기가 하나의 게이트웨이 자원을 공유할 때, 기기별 중요도에 따라 자원을 나누는 **디바이스 기반 FSS**가 중요해질 것이다. 또한, 탄소 배출권과 연동하여 그룹별 에너지 사용량을 조절하는 '그린 FSS' 기술도 연구되고 있다. FSS는 집단 지성과 공공의 질서를 유지하는 운영체제의 '사회적 조율사' 역할을 계속 수행할 것이다.

- **📢 섹션 요약 비유**: 모두가 똑같은 양의 공기(자원)를 마실 수 있게 보장하여, 거대한 도시(시스템)가 평화롭게 숨 쉬게 만드는 마법의 환경 관리자입니다.
