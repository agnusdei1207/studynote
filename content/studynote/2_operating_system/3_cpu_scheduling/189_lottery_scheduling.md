# 복권 스케줄링 (Lottery Scheduling) - 확률적 스케줄링

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 추첨 스케줄링 (Lottery Scheduling)은 각 프로세스에게 자원 지분을 나타내는 **복권 (Ticket)**을 할당하고, 무작위 추첨을 통해 당첨된 복권을 가진 프로세스에게 CPU를 배분하는 **확률 기반 비례 공유 스케줄링** 방식이다.
> 2. **가치**: 복권의 개수를 통해 자원 할당 비율을 매우 직관적이고 세밀하게 조절할 수 있으며, 엄격한 우선순위 방식에서 발생하는 기아 현상(Starvation)을 확률적으로 방지하는 유연성을 제공한다.
> 3. **융합**: 티켓 양도(Transfer), 티켓 인플레이션(Inflation) 등의 기법을 통해 프로세스 간의 협력을 지원하며, 현대의 클라우드 자원 관리나 가상화 환경의 지분 기반 할당 논리의 기초가 된다.

---

## Ⅰ. 추첨 스케줄링의 개요

### 1. 정의
- 시스템 자원의 일정 비율을 보장하기 위해 고안된 알고리즘으로, 프로세스가 가진 티켓의 개수가 곧 그 프로세스가 자원을 할당받을 확률이 되는 방식이다.

### 2. 특징: '무작위의 공정성'
- **비례 공유**: 티켓 60장을 가진 A와 40장을 가진 B가 있다면, 장기적으로 A는 CPU의 60%, B는 40%를 점유하게 된다.
- **유연성**: 새로운 프로세스가 추가되거나 제거되어도 티켓 총량만 조절하면 즉시 비례 배분이 수정된다.
- **반응성**: 아주 짧은 실행 시간 동안에는 오차가 있을 수 있으나, 시간이 흐를수록 설정된 확률에 수렴한다.

### 3. 💡 비유: '경품 추첨 행사'
- 추첨 스케줄링은 **'응모권을 많이 넣을수록 당첨 확률이 높아지는 경품 추첨'**과 같다.
- 돈을 많이 낸 손님(중요 프로세스)에게는 응모권 100장을 주고, 적게 낸 손님에게는 1장을 준다. 추첨함에서 무작위로 번호를 뽑을 때(스케줄링), 응모권이 많은 손님이 불릴 확률이 압도적으로 높지만, 1장만 가진 손님도 언젠가는 반드시 한 번은 당첨(실행)될 수 있다.

---

## Ⅱ. 추첨 스케줄링의 주요 메커니즘 (Deep Dive)

### 1. 티켓 화폐 (Ticket Currencies)
- 사용자별로 자신만의 '화폐 단위'로 티켓을 발행하고, 이를 시스템 전체의 '글로벌 티켓'으로 환산하는 방식이다. 사용자 내부의 자원 관리를 자율화한다.

### 2. 티켓 양도 (Ticket Transfer)
- 한 프로세스가 다른 프로세스에게 자신의 티켓을 잠시 빌려주는 기능이다. 특히 클라이언트가 서버에게 요청을 보냈을 때, 서버가 빨리 일을 끝내도록 자신의 지분을 넘겨주는 상황(RPC 등)에서 유용하다.

### 3. 티켓 인플레이션 (Ticket Inflation)
- 프로세스가 서로 신뢰하는 관계일 때, 필요에 따라 스스로 티켓 수를 늘려 더 많은 자원을 일시적으로 점유하는 방식이다.

### 4. ASCII 구조 다이어그램
```text
    [ Ready Queue ]             [ Scheduler (Lottery) ]         [ Result ]
    ┌─────────────┐             ┌─────────────────────┐         ┌────────────┐
    │ P1: [1-60]  │             │ Generate Random     │         │            │
    ├─────────────┤             │ Number (1~100)      │ ──▶     │ P1 Wins!   │
    │ P2: [61-90] │             │                     │         │ (CPU Go)   │
    ├─────────────┤             │ Winner: 42          │         └────────────┘
    │ P3: [91-100]│             └─────────────────────┘
    └─────────────┘
```

- **📢 섹션 요약 비유**: 주머니 속에 각자의 이름이 적힌 구슬을 비율대로 넣고, 눈을 감고 하나를 꺼내어 다음 순서를 정하는 마법의 주머니와 같습니다.

## Ⅲ. 추첨 스케줄링 vs 우선순위 스케줄링

| 비교 항목 | 우선순위 스케줄링 (Priority) | 추첨 스케줄링 (Lottery) |
|:---|:---|:---|
| **결정 방식** | 절대적 (계급 기반) | **상대적 (확률 기반)** |
| **기아 현상** | 발생 가능 (에이징 필요) | **확률적으로 매우 낮음** |
| **정밀도** | 정수 단위의 딱딱한 구분 | **지분 단위의 미세 조절 가능** |
| **복잡도** | 낮음 | 중간 (난수 생성 오버헤드) |

- **📢 섹션 요약 비유**: "1등부터 나와!"라고 소리치는 엄격한 교관(우선순위)과, "제비뽑기로 정하자!"라고 말하는 유연한 레크리에이션 강사(추첨)의 차이입니다.

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 가상 호스팅 서비스의 자원 분배
- **상황**: 고객들에게 서버 자원을 7:3 비율로 판매함.
- **판단**: 고객 A의 가상 머신에 티켓 700장을, 고객 B에게 300장을 부여한다. 각 VM 내부에서 프로세스가 몇 개가 생기든 상관없이, 하이퍼바이저 수준의 **추첨 스케줄러**는 전체 물리 CPU의 70%를 정확히 A에게 전달한다. 고객 B는 프로세스를 1,000개 띄워도 A의 70% 지분을 뺏어올 수 없다.

### 2. 기술사적 인사이트: 보폭 스케줄링 (Stride Scheduling)으로의 진화
- 추첨 스케줄링은 '무작위성' 때문에 짧은 시간 측정 시 오차가 발생한다. 기술사라면 이를 보완하기 위해 무작위성을 제거하고 결정론적으로 지분을 보장하는 **보폭 스케줄링 (Stride Scheduling)**을 대안으로 언급해야 한다. 보폭 스케줄링은 티켓 수에 반비례하는 '보폭'을 설정하여 가장 적게 걸어간 프로세스를 선택함으로써 추첨의 철학을 더욱 정밀하게 완성한다.

- **📢 섹션 요약 비조**: 행운(확률)에 맡기는 것도 좋지만, 결과의 정확성이 생명인 시스템에서는 보폭(Stride)을 재어가며 걷는 정밀함이 필요합니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **자원 할당의 유연성**: 시스템 중단 없이 티켓 수만 변경하여 자원 비중 즉시 조정.
- **구조적 공정성**: 낮은 지분이라도 통계적으로 반드시 자원을 할당받는 '최소 보장' 실현.

### 2. 미래 전망
추첨 스케줄링의 아이디어는 현대 클라우드 오케스트레이션의 **Weight-based 스케줄링**이나 네트워크 트래픽 제어(QoS) 기술에 깊이 녹아들어 있다. 앞으로는 전력 소모량이나 탄소 배출량을 티켓으로 환산하여 자원을 배분하는 '에너지 지분 스케줄링' 등 사회적 가치와 연동된 지분 관리 기술로 그 영역이 확장될 것이다.

- **📢 섹션 요약 비유**: 누구나 당첨의 꿈을 꿀 수 있고, 기여한 만큼 더 많은 기회를 얻는 '기회의 평등'을 구현한 스케줄링의 민주주의입니다.
