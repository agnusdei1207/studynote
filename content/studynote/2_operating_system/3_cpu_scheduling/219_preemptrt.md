# 219. 워크로드 보존 스케줄링 (Workload-Conserving Scheduling)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 유휴 CPU 없이 항상 작업을 실행하는 정책
> 2. **가치**: 높은 CPU 활용률, 최대 처리량
> 3. **융합**: 배치 처리, 서버 워크로드, CFS와 연관

---

## Ⅰ. 개요

### 개념 정의

워크로드 보존 스케줄링(Workload-Conserving Scheduling)은 **실행 가능한 태스크가 있는 한 어떤 CPU도 유휴 상태가 되지 않도록 하는 스케줄링 정책**이다. 최대 CPU 활용률과 처리량을 목표로 한다.

### 💡 비유: 효율적 공장
워크로드 보존 스케줄링은 **쉬지 않는 공장**과 같다. 주문이 있으면 기계가 쉬지 않고 계속 돈다. 생산성을 극대화한다.

## Ⅲ. 실무 적용

### 설정 방법
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【워크로드 보존 활성화】                                              │
│  ──────────────────                                                  │
│  # idle 밸런싱 활성화 (기본)                                         │
│  $ cat /proc/sys/kernel/sched_smt_capacity                         │
│                                                                     │
│  # 로드 밸런싱 활성화                                                 │
│  $ sysctl -w kernel.sched_load_balance=1                           │
│                                                                     │
│  【비보존 설정 (전력 절약)】                                           │
│  ──────────────────                                                  │
│  # CPU 오프라인                                                      │
│  $ echo 0 > /sys/devices/system/cpu/cpu3/online                    │
│                                                                     │
│  # 에너지 인식 스케줄링                                               │
│  $ echo 1 > /proc/sys/kernel/sched_energy_aware                    │
│                                                                     │
│  # CPU 주파수 제한                                                   │
│  $ echo powersave > /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor
│                                                                     │
│  【모니터링】                                                          │
│  ──────────────────                                                  │
│  # CPU 활용률                                                        │
│  $ mpstat -P ALL 1                                                  │
│                                                                     │
│  # 실행 중인 태스크 수                                                │
│  $ vmstat 1                                                         │
│                                                                     │
│  # runqueue 길이                                                     │
│  $ cat /proc/stat | grepprocs_running                              │
│                                                                     │
│  # idle 밸런싱 통계                                                   │
│  $ perf stat -e sched:sched_idle_balance ./program                 │
│                                                                     │
│  【워크로드별 설정】                                                   │
│  ──────────────────                                                  │
│  # 배치 작업 (보존)                                                   │
│  $ echo performance > /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor
│  $ sysctl -w kernel.sched_min_granularity_ns=10000000              │
│                                                                     │
│  # 대화형 (균형)                                                      │
│  $ echo schedutil > /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor
│  $ sysctl -w kernel.sched_min_granularity_ns=1000000               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 실행 가능한 태스크가 있으면 CPU 유휴 없음
• 목표: 최대 CPU 활용률, 높은 처리량
• CFS: 기본적으로 워크로드 보존
• 비보존: 에너지 절약, 격리, 선호도
• 적용: HPC, 배치, 렌더링
• 트레이드오프: 처리량 vs 전력/응답성
```
