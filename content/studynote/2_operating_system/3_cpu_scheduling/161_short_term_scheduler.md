# 단기 스케줄러 (Short-term Scheduler) / CPU 스케줄러

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 단기 스케줄러 (Short-term Scheduler)는 준비 큐(Ready Queue)에 있는 프로세스 중 **다음에 CPU를 할당받을 프로세스를 선택**하는 커널의 핵심 모듈로, 흔히 CPU 스케줄러라 불린다.
> 2. **가치**: 밀리초(ms) 단위로 매우 빈번하게 실행되어 시스템의 응답성과 처리량을 직접적으로 결정하며, 시분할 시스템에서 여러 프로그램이 동시에 돌아가는 것처럼 보이게 하는 실질적인 주체이다.
> 3. **융합**: 하드웨어 타이머 인터럽트와 결합하여 실행 중인 프로세스를 선점(Preemption)하고, 디스패처(Dispatcher)에게 제어권을 넘겨 문맥 교환(Context Switch)을 유도하는 트리거 역할을 한다.

+++

## Ⅰ. 단기 스케줄러의 개요

### 1. 정의
- 메모리에 로드된 프로세스들 중 실행 준비가 된(Ready) 프로세스를 골라 CPU를 할당하는 스케줄러이다.

### 2. 특징: '찰나의 선택'
- **실행 빈도**: 매우 빈번하다 (보통 100ms에 한 번 이상).
- **속도 중요성**: 스케줄러 자체가 CPU를 쓰기 때문에, 선택 로직이 너무 복잡하면 실제 앱이 돌아갈 시간을 뺏게 된다. 따라서 극도로 빨라야 한다.
- **결정 시점**: 타이머 인터럽트 발생 시, I/O 요청 시, 프로세스 종료 시 등.

### 3. 💡 비유: '은행 창구의 번호표 호출기'
- 단기 스케줄러는 **'대기석(준비 큐)에 앉아 있는 손님 중 다음 차례를 불러 창구(CPU)로 안내하는 자동 호출기'**와 같다.
- 호출기는 1초에도 수십 번씩 번호를 바꿔가며 손님을 부르며, 이 호출기가 얼마나 빨리, 그리고 공정하게 번호를 부르느냐에 따라 은행(시스템)의 서비스 품질이 결정된다.

+++

## Ⅱ. 단기 스케줄러의 동작 조건 (Deep Dive)

스케줄러는 프로세스의 상태가 다음과 같이 변할 때 호출된다.

### 1. 주요 스케줄링 시점
1. **Running ──▶ Waiting**: 프로세스가 I/O를 요청하거나 자원을 기다릴 때. (자발적 양보)
2. **Running ──▶ Ready**: 타임 슬라이스가 만료되거나 인터럽트가 발생했을 때. (강제 선점)
3. **Waiting ──▶ Ready**: 대기하던 I/O가 완료되어 다시 실행 자격을 얻었을 때.
4. **Running ──▶ Terminated**: 프로세스가 실행을 마쳤을 때.

### 2. 스케줄링 방식의 구분
- **비선점형 (Non-preemptive)**: 1번과 4번 상황에서만 스케줄링이 일어남. 프로세스가 스스로 물러날 때까지 기다림.
- **선점형 (Preemptive)**: 모든 상황에서 스케줄링 가능. 현대 운영체제의 표준.

### 3. ASCII 구조 다이어그램
```text
    [ Ready Queue ]             [ CPU Scheduler ]           [ CPU / Core ]
    ┌─────────────┐             ┌───────────────┐           ┌────────────┐
    │ PCB (P1)    │             │  Scheduling   │           │            │
    ├─────────────┤ ──(Select)─▶│  Algorithm    │ ──(Assign)▶│  Running   │
    │ PCB (P2)    │             │  (FCFS, RR..) │           │  Process   │
    ├─────────────┤             └───────┬───────┘           └────────────┘
    │ PCB (P3)    │                     │
    └─────────────┘                     ▼
                                 [ Dispatcher ] ◀── (실제 문맥 교환 수행)
```

- **📢 섹션 요약 비유**: 대기자 명단을 보고 다음 주자를 뽑는 '눈(스케줄러)'과, 실제로 바톤을 넘겨주는 '손(디스패처)'의 긴밀한 협업 과정입니다.

## Ⅲ. 스케줄러의 3계층 비교

단기 스케줄러의 위치를 파악하기 위한 비교 분석이다.

| 구분 | 단기 (Short-term) | 중기 (Medium-term) | 장기 (Long-term) |
|:---|:---|:---|:---|
| **별칭** | **CPU 스케줄러** | **스와퍼 (Swapper)** | 작업 스케줄러 |
| **대상** | 준비 큐의 프로세스 | 메모리의 프로세스 | 디스크의 작업 |
| **빈도** | **매우 높음 (ms)** | 중간 | 낮음 (초/분) |
| **목적** | **응답성/처리량** | **메모리 여유 확보** | 다중 프로그래밍 정도 조절 |

- **📢 섹션 요약 비유**: 단기 스케줄러는 1분 1초가 급한 '현장 관리자', 중기 스케줄러는 창고 공간을 조절하는 '재고 관리자', 장기 스케줄러는 신입 사원을 뽑는 '인사 담당자'와 같습니다.

+++

## Ⅳ. 실무 적용 및 알고리즘 인사이트 (Strategy)

### 1. 실무 시나리오: 리눅스 CFS (Completely Fair Scheduler)
- **상황**: 수많은 웹 서비스와 백그라운드 작업이 동시에 돌아가는 리눅스 서버.
- **판단**: 단기 스케줄러로 **CFS**를 사용한다. 모든 프로세스에게 공평한 '가상 실행 시간(vruntime)'을 부여하고, 가장 적게 실행된 프로세스를 레드-블랙 트리(Red-Black Tree)에서 찾아 우선적으로 CPU를 준다. 이를 통해 특정 프로세스의 독점을 막고 시스템 전반의 공정성을 유지한다.

### 2. 기술사적 인사이트: 선택의 비용 (Context Switch Cost)
- 단기 스케줄러가 아무리 좋은 프로세스를 뽑아도, 전환 과정(Context Switch)이 너무 잦으면 배보다 배꼽이 더 커진다. 기술사라면 **타임 퀀텀(Time Quantum)**의 크기를 부하에 따라 동적으로 조절하거나, 캐시 오염(Cache Locality)을 고려하여 이전에 실행되었던 코어에 다시 배정하는 **친화성(Affinity)** 스케줄링 전략을 제시해야 한다.

- **📢 섹션 요약 비유**: 최고의 타자를 기용하는 것도 좋지만, 대타를 너무 자주 바꾸면 경기 흐름이 끊기듯(오버헤드), 적절한 교체 타이밍을 잡는 것이 감독(운영체제)의 실력입니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **시스템 가동률 극대화**: CPU가 한순간도 쉬지 않게 프로세스를 끊임없이 공급.
- **체감 응답성 향상**: 짧은 주기의 전환을 통해 사용자에게 실시간 멀티태스킹 경험 제공.

### 2. 미래 전망
최근에는 단기 스케줄러가 하드웨어의 전력 상태(P-states)와 연동되어, 고성능 코어와 저전력 코어에 프로세스를 지능적으로 배분하는 **이기종 스케줄링 (Heterogeneous Scheduling)**으로 진화하고 있다. 또한, 인공지능 워크로드를 위해 GPU 가속기 점유 시간까지 조율하는 방향으로 영역이 확장되면서, 단기 스케줄러는 현대 컴퓨팅의 '지능형 엔진'으로 자리매김하고 있다.

- **📢 섹션 요약 비유**: 찰나의 순간에 최선의 선택을 내려 전체의 조화를 완성하는, 운영체제라는 오케스트라의 보이지 않는 지휘자입니다.
