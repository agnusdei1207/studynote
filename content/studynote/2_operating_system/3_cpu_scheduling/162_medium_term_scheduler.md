# 중기 스케줄러 (Medium-term Scheduler) / 스와퍼(Swapper)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 중기 스케줄러 (Medium-term Scheduler)는 메모리에 너무 많은 프로세스가 올라와 여유 공간이 부족할 때, 일부 프로세스를 하드디스크의 스왑(Swap) 영역으로 내쫓는 **스와핑 (Swapping)**을 결정하는 모듈이다.
> 2. **가치**: 다중 프로그래밍의 정도(Degree of Multiprogramming)를 동적으로 조절하여 시스템이 느려지는 스래싱(Thrashing) 현상을 방지하고, 메모리 자원의 효율성을 극대화한다.
> 3. **융합**: 프로세스 상태를 'Suspended'로 변화시키는 주체이며, 현대의 가상 메모리 시스템에서는 페이지 교체(Page Replacement) 알고리즘과 긴밀히 협력하여 동작한다.

---

## Ⅰ. 중기 스케줄러의 개요

### 1. 정의
- 가용 메모리 공간을 확보하기 위해 실행 중인 프로세스 중 일부를 선택하여 메모리에서 제거하고, 나중에 다시 불러오는 역할을 하는 스케줄러이다.

### 2. 등장 배경: '메모리는 한정되어 있다'
- 장기 스케줄러가 너무 많은 프로세스를 메모리에 올렸거나, 현재 프로세스들이 갑자기 많은 메모리를 요구할 때 시스템 전체가 마비될 수 있다. 이때 응급조치로 특정 프로세스를 잠시 디스크로 대피시키는 기술이 필요해졌다.

### 3. 💡 비유: '식당의 웨이팅 관리'
- 중기 스케줄러는 **'식당 테이블이 꽉 찼을 때, 밥을 다 먹고 수다만 떠는 손님(유휴 프로세스)에게 잠시 밖의 대기 의자(디스크)로 나가 달라고 정중히 부탁하는 매니저'**와 같다.
- 덕분에 정말 배고픈 새 손님(중요 프로세스)이 자리에 앉아 밥을 먹을 수 있게 된다.

---

## Ⅱ. 스와핑 (Swapping) 메커니즘 (Deep Dive)

중기 스케줄러의 핵심 작업인 스와핑은 두 단계로 나뉜다.

### 1. 스왑 아웃 (Swap-out)
- 메모리 부족 시, 우선순위가 낮거나 오랫동안 CPU를 쓰지 않은 프로세스를 선택하여 그 상태(Context)를 통째로 디스크의 스왑 영역에 저장하고 메모리를 해제한다.
- 프로세스 상태는 **Ready/Waiting ──▶ Suspended**로 변한다.

### 2. 스왑 인 (Swap-in)
- 메모리에 다시 여유가 생기면, 디스크에 있던 프로세스의 상태를 다시 메모리로 불러와 실행을 재개시킨다.

### 3. ASCII 구조 다이어그램
```text
    ┌──────────────────────────────────┐
    │        Main Memory (RAM)         │
    │  ┌───────────┐    ┌───────────┐  │
    │  │ Process A │    │ Process B │  │
    │  └─────┬─────┘    └─────▲─────┘  │
    └────────│────────────────│────────┘
             │ (Swap-out)     │ (Swap-in)
             ▼                │
    ┌─────────────────────────┴────────┐
    │        Disk (Swap Space)         │
    │  ┌───────────┐                   │
    │  │ Process A │                   │
    │  └───────────┘                   │
    └──────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 좁은 책상(메모리) 위에 책을 다 펼칠 수 없을 때, 지금 안 보는 책을 잠시 바닥(디스크)에 내려놓았다가 필요할 때 다시 집어 올리는 동작과 같습니다.

## Ⅲ. 스케줄러 간 역할 비교 (Context)

| 구분 | 장기 스케줄러 | 중기 스케줄러 | 단기 스케줄러 |
|:---|:---|:---|:---|
| **역할** | 작업 선택 (Job) | **프로세스 대피 (Swap)** | 실행 프로세스 선택 (CPU) |
| **결정권** | 메모리에 누가 들어올까? | **누구를 쫓아낼까?** | 누구에게 CPU를 줄까? |
| **목적** | 다중 프로그래밍 시동 | **메모리 과부하 해결** | 시스템 응답성 확보 |

- **📢 섹션 요약 비유**: 장기 스케줄러가 입학 사정관이라면, 중기 스케줄러는 휴학생(내보내기)과 복학생(불러오기)을 관리하는 학사 관리자이고, 단기 스케줄러는 수업 시간에 발표자를 지목하는 교수님입니다.

---

## Ⅳ. 실무 적용 및 운영 인사이트 (Strategy)

### 1. 실무 시나리오: 리눅스의 스왑 텐디니스 (Swappiness)
- **상황**: 서버의 램이 90% 이상 차서 시스템이 버벅거리기 시작함.
- **판단**: 커널의 **중기 스케줄러**가 작동하여 일부 프로세스를 디스크로 옮기기 시작한다. 이때 `vm.swappiness` 설정값을 높이면 더 공격적으로 스와핑을 수행하고, 낮추면 물리 메모리를 끝까지 버티며 사용한다. 기술사라면 데이터베이스 서버처럼 지연 시간에 민감한 장비에서는 스와핑을 최소화하도록 설정하고, 일반 웹 서버에서는 메모리 여유 확보를 위해 적절한 스와핑을 허용하는 전략을 취해야 한다.

### 2. 기술사적 인사이트: 스래싱 (Thrashing) 방지
- 너무 잦은 스와핑은 CPU 연산보다 디스크 I/O에 더 많은 시간을 쓰게 만들어 시스템 성능을 0으로 만든다. 기술사라면 이를 방지하기 위해 프로세스마다 필요한 최소 메모리 세트인 **Working Set**을 보장하거나, 페이지 부재 빈도(PFF)를 모니터링하여 중기 스케줄러가 과도하게 작동하지 않도록 제어해야 한다.

- **📢 섹션 요약 비유**: 손님을 너무 자주 밖으로 내보내고 들여보내면 서빙할 시간(CPU 연산)이 없어지는 것과 같으므로, 적당한 선에서 손님 수(다중 프로그래밍 정도)를 조절하는 지혜가 필요합니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **가상적 확장**: 물리적 메모리 한계를 넘어 더 많은 프로세스 수용 가능.
- **시스템 안정성**: 메모리 고갈로 인한 커널 패닉 방지.

### 2. 미래 전망
최근에는 하드디스크보다 훨씬 빠른 NVMe SSD와 **zSwap(압축 스왑)** 기술이 발전하면서 스와핑의 오버헤드가 크게 줄어들고 있다. 하지만 모바일 OS(Android, iOS)에서는 배터리와 수명 문제로 디스크 스와핑 대신 프로세스를 강제로 죽이고 상태만 저장하는 **Low Memory Killer** 방식으로 진화하고 있다. 중기 스케줄러는 플랫폼의 특성에 맞게 '누가 메모리를 점유할 것인가'를 결정하는 가장 강력한 권력자로 남을 것이다.

- **📢 섹션 요약 비유**: 한정된 자원을 가장 영리하게 나누어 쓰는, 운영체제의 노련한 '살림꾼'입니다.
