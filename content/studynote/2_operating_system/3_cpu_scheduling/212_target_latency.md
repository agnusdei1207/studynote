# 대상 지연 시간 (Target Latency) 및 최소 입자 (Minimum Granularity)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 리눅스 CFS 스케줄러가 모든 실행 가능한 태스크가 적어도 한 번씩 CPU를 점유하도록 보장하는 시간 주기(Target Latency)와, 문맥 교환 오버헤드를 막기 위해 태스크가 보장받는 최소 실행 시간(Min Granularity)이다.
> 2. **가치**: 시스템 응답성(Latency)과 전체 처리량(Throughput) 사이의 트레이드오프를 정교하게 조절하는 제어판 역할을 하며, 시스템 환경(데스크탑 vs 서버)에 따른 최적 튜닝 지점을 제공한다.
> 3. **융합**: 시분할 시스템의 타임 슬라이스 (Time Slice) 개념의 현대적 진화이며, 태스크 수 증가에 따른 '스케줄링 폭풍' 방지 메커니즘과 연계된다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **대상 지연 시간 (Target Latency)**: "모든 태스크가 늦어도 이 시간 안에는 한 번씩 실행되어야 한다"는 약속된 시간대다.
  - **최소 입자 (Minimum Granularity)**: "한 번 CPU를 잡으면 적어도 이 시간만큼은 방해받지 않고 실행되어야 한다"는 최소 점유 시간이다.

- **💡 비유**: 대상 지연 시간은 "수업 시간 50분"과 같고, 최소 입자는 "발표자당 최소 발언 시간 3분"과 같다. 50분 안에 모든 학생이 발표하게 하려고 노력하지만, 학생이 너무 많아지면 3분씩만 말해도 50분을 훌쩍 넘길 수밖에 없는 상황을 관리하는 규칙이다.

- **📢 섹션 요약 비유**: 모든 손님에게 공평한 기회를 주려는 '총 순환 시간'과, 너무 잦은 교체로 인한 혼란을 막기 위한 '최소 식사 시간'의 균형 잡기입니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CFS의 시간 할당 계산 방식

CFS는 고정된 타임 슬라이스 대신, 현재 실행 가능한 태스크 수($N$)에 따라 동적으로 시간을 나눈다.

```text
  [ Case 1: 태스크 수가 적을 때 (N < Latency / Min_Gran) ]
  Time_Slice = Target_Latency / N
  (예: 20ms / 4 = 5ms씩 공평하게 실행)
  
  [ Case 2: 태스크 수가 너무 많을 때 (N > Latency / Min_Gran) ]
  Time_Slice = Minimum_Granularity
  (예: 20ms / 100 = 0.2ms? 안돼! 너무 짧아! -> 최소 1ms는 보장)
  -> 실제 지연 시간은 N * Min_Gran 만큼 늘어남
```

**[다이어그램 해설]** 
1. **Target Latency (sched_latency_ns)**: 기본값은 보통 6ms~20ms다. 태스크가 적을 때는 이 시간을 정확히 나눠 쓴다.
2. **Min Granularity (sched_min_granularity_ns)**: 기본값은 보통 0.75ms~3ms다. 태스크가 급증하여 `Latency/N`이 이 값보다 작아지면, 공정성을 조금 포기하더라도 문맥 교환 오버헤드를 줄이기 위해 `Min Granularity`를 타임 슬라이스로 선택한다.

### 트레이드오프 결정 요소
- **대상 지연 시간 ↓**: 응답성이 좋아져서 마우스 움직임 등이 매끄럽지만, 문맥 교환이 잦아져 CPU가 실제 일하는 시간보다 전환하는 데 쓰는 시간이 많아진다 (Throughput 저하).
- **최소 입자 ↑**: 한 놈이 오래 일하므로 캐시 효율과 처리량이 좋아지지만, 다른 놈들이 기다리는 시간이 길어져 시스템이 버벅거리는 느낌을 준다 (Latency 증가).

- **📢 섹션 요약 비유**: 회의 시간을 짧게 쪼개면(낮은 지연 시간) 모두가 발언할 수 있지만 인사만 하다가 끝날 수 있고, 길게 잡으면(높은 입자) 깊이 있는 논의는 가능하지만 내 차례가 영영 안 올 수 있는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 데스크탑 vs 서버 프로파일 튜닝

| 항목 | 데스크탑 (Interactive) | 서버 (Throughput) |
|:---|:---|:---|
| **Target Latency** | 낮게 설정 (예: 6ms) | 높게 설정 (예: 24ms) |
| **Min Granularity** | 낮게 설정 (예: 0.75ms) | 높게 설정 (예: 4ms) |
| **최우선 목표** | 마우스/키보드 반응성 | 일괄 처리(Batch) 성능 및 캐시 적중률 |
| **체감 효과** | 부드러운 화면 전환 | 대용량 데이터 처리 속도 향상 |

- **📢 섹션 요약 비유**: 데스크탑은 '속전속결 퀴즈쇼' 같아야 하고, 서버는 '심층 토론회' 같아야 제맛입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 고부하 웹 서버의 응답 지연 해결
- **상황**: 수천 개의 요청을 처리하는 웹 서버가 CPU 사용률은 높지 않은데 이상하게 사용자 응답이 늦어짐. 분석 결과 너무 많은 스레드가 경합하며 문맥 교환 오버헤드(Context Switch)가 1초에 수십만 번 발생 중.
- **판단**: `sched_min_granularity_ns`를 높여서 각 스레드가 한 번 CPU를 잡았을 때 충분히 요청을 처리하고 내려오게 해야 한다. 또한 `sched_latency_ns`를 적절히 늘려 '스케줄링 폭풍'을 잠재우고 캐시 로컬리티를 확보하여 전체적인 처리 지연을 개선해야 한다.

- **📢 섹션 요약 비유**: 창구 직원이 손님 한 명당 10초만 상담하고 다음 사람을 부르면 서류 정리만 하다가 하루가 가버리니, 최소 5분은 집중해서 상담하게 규칙을 바꾸는 것입니다.

## Ⅴ. 기대효과 및 결론

- **기대효과**: 하드웨어 자원을 사용자나 관리자의 의도에 맞게 최적으로 배분하여 시스템의 가치를 극대화한다.
- **미래 전망**: 클라우드 가상화 기술이 정교해짐에 따라, 호스트 OS와 게스트 OS 간의 스케줄링 주기 동기화 기술이 CPU 지연 시간 문제를 해결하는 핵심 기술이 될 것이다.

- **📢 섹션 요약 비유**: 정답은 고정된 숫자에 있는 것이 아니라, 상황에 맞춰 저울추(Latency vs Granularity)를 옮기는 유연함에 있습니다.
