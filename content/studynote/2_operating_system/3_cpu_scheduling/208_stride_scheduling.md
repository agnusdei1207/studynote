# 비례 배분 스케줄링 (Proportionate Share Scheduling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보폭 스케줄링 (Stride Scheduling)은 추첨 스케줄링의 확률적 오차를 해결하기 위해 고안된 **결정론적 비례 공유 스케줄링** 방식으로, 프로세스마다 티켓 수에 반비례하는 **보폭 (Stride)**을 부여하고 가장 적게 이동한 프로세스를 선택한다.
> 2. **가치**: 난수에 의존하지 않고 각 프로세스가 자신의 지분(티켓 수)에 정확히 비례하는 CPU 시간을 점유하도록 보장하며, 아주 짧은 실행 시간 동안에도 자원 배분의 정밀도를 완벽히 유지한다.
> 3. **융합**: 'Pass'라 불리는 누적 이동 거리 개념을 통해 스케줄링 결정을 내리며, 현대의 정밀한 대역폭 제어나 클라우드 자원 할당 로직의 고도화된 모델로 활용된다.

+++

## Ⅰ. 보폭 스케줄링의 개요

### 1. 정의
- 각 프로세스에게 티켓 수에 반비례하는 보폭(Stride)을 할당하고, 매 스케줄링 시점마다 현재까지 가장 적은 거리(Pass)를 이동한 프로세스를 선택하여 한 보폭만큼 전진시키는 방식이다.

### 2. 등장 배경: '추첨의 불확실성 제거'
- 추첨 스케줄링은 장기적으로는 공정하지만, 단기적으로는 운 나쁜 프로세스가 한참 동안 선택되지 않을 수 있다. 보폭 스케줄링은 이러한 '행운의 요소'를 제거하고 산술적으로 정확한 순번을 보장하기 위해 탄생했다.

### 3. 💡 비유: '보폭이 다른 사람들의 경주'
- 보폭 스케줄링은 **'다리가 아주 긴 거인(보폭이 큰 프로세스)과 다리가 짧은 꼬마(보폭이 작은 프로세스)의 경주'**와 같다.
- 규칙은 간단하다. **"항상 가장 뒤처진 사람(Pass가 작은 사람)이 한 발자국 전진한다."**
- 다리가 짧은 꼬마(우선순위가 높은 프로세스)는 거인의 한 걸음 거리를 따라잡기 위해 여러 번 발을 내디뎌야 하므로, 결과적으로 더 자주 선택되어 CPU를 많이 쓰게 된다.

+++

## Ⅱ. 핵심 메커니즘: Stride와 Pass (Deep Dive)

### 1. 주요 변수 및 공식
- **티켓 (Ticket)**: 프로세스가 가진 자원 지분.
- **보폭 (Stride)**: $L / 티켓$ (L은 매우 큰 상수). 티켓이 많을수록 보폭은 작아진다.
- **패스 (Pass)**: 프로세스가 지금까지 이동한 누적 거리. 초기값은 0이다.

### 2. 스케줄링 과정
1. 모든 프로세스 중 **Pass 값이 가장 작은** 프로세스를 선택한다.
2. 선택된 프로세스를 CPU에서 실행시킨다.
3. 실행이 끝나면, 해당 프로세스의 Pass 값에 자신의 Stride 값을 더한다. ($Pass = Pass + Stride$)
4. 다시 1번으로 돌아가 반복한다.

### 3. ASCII 구조 다이어그램
```text
    [ Stride Settings: L=10,000 ]
    P1 (100 Tickets) -> Stride: 100
    P2 (50 Tickets)  -> Stride: 200
    
    [ Step-by-Step Execution ]
    Step 1: P1(Pass:0), P2(Pass:0) -> P1 Select! -> P1(Pass:100)
    Step 2: P1(Pass:100), P2(Pass:0) -> P2 Select! -> P2(Pass:200)
    Step 3: P1(Pass:100), P2(Pass:200) -> P1 Select! -> P1(Pass:200)
    Step 4: P1(Pass:200), P2(Pass:200) -> P1 Select! -> P1(Pass:300)
    
    (P1이 P2보다 보폭이 좁으므로 더 자주 선택됨)
```

- **📢 섹션 요약 비유**: 보폭이 좁을수록(우선순위 높음) 같은 거리를 가기 위해 더 많은 발걸음(CPU 할당)을 해야 하는 원리를 이용한 정밀한 공정성 실현 방식입니다.

## Ⅲ. 추첨 스케줄링 vs 보폭 스케줄링

| 비교 항목 | 추첨 스케줄링 (Lottery) | 보폭 스케줄링 (Stride) |
|:---|:---|:---|
| **결정론** | 확률적 (Non-deterministic) | **확정적 (Deterministic)** |
| **정밀도** | 단기적 오차 발생 가능 | **항상 정확한 지분 보장** |
| **구현 부하** | 난수 생성 오버헤드 | 큐 정렬(최소값 찾기) 오버헤드 |
| **상태 관리** | 티켓 수만 관리 | **누적 Pass 값 관리 필요** |

- **📢 섹션 요약 비유**: 제비뽑기로 순서를 정하는 축제(추첨)와, 줄자로 거리를 재어가며 가장 뒤에 있는 사람을 챙기는 마라톤 대회(보폭)의 차이입니다.

+++

## Ⅳ. 실무 적용 및 스케줄링 인사이트 (Strategy)

### 1. 실무 시나리오: 비디오 스트리밍 대역폭 제어
- **상황**: 100명의 사용자에게 유료 등급에 따라 대역폭을 정확히 5:3:2로 배분해야 함.
- **판단**: 추첨 방식을 쓰면 순간적으로 영상이 끊기는 사용자가 생길 수 있다. **보폭 스케줄링**을 적용하여 전송한 데이터양을 'Pass'로 관리한다. 가장 적게 전송한 사용자(가장 뒤처진 사용자)를 실시간으로 선택하여 데이터를 보내줌으로써, 지연 시간의 편차(Jitter)가 없는 완벽한 비례 배분 서비스를 제공한다.

### 2. 기술사적 인사이트: 새로운 프로세스의 진입 문제
- 보폭 스케줄링에서 가장 큰 고민은 **'새로 들어온 프로세스의 초기 Pass 값을 얼마로 줄 것인가'**이다. 0으로 주면 기존 프로세스들을 다 제치고 CPU를 독점하게 된다. 기술사라면 이를 해결하기 위해 현재 시스템의 **평균 Pass 값**이나 **최소 Pass 값**을 부여하는 지능형 진입 정책을 제시해야 한다.

- **📢 섹션 요약 비조**: 뒤늦게 레이스에 참가한 선수에게 출발선을 어디로 정해주느냐가 시스템의 공정성과 안정성을 결정하는 핵심 설계 포인트입니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **완벽한 예측 가능성**: 특정 시간 동안 프로세스가 얻을 자원량을 수학적으로 확정.
- **자원 활용의 고도화**: 지분 기반의 미세한 자원 할당으로 수익 모델(Cloud Billing) 최적화.

### 2. 미래 전망
최근 하드웨어 가속기(GPU, NPU)의 자원 공유 기술에서는 보폭 스케줄링의 원리를 응용한 **Credit-based 스케줄링**이 표준이 되고 있다. 또한, 네트워크 기능 가상화(NFV)에서 패킷 처리 순서를 정할 때 보폭 스케줄링은 가장 강력한 알고리즘으로 대우받는다. 보폭 스케줄링은 '정의로운 배분'이 무엇인지 기술적으로 증명한 운영체제의 가장 정교한 저울이다.

- **📢 섹션 요약 비유**: 한 치의 오차도 허용하지 않는 운영체제의 정밀한 측정 기술이자, 모든 프로세스에게 약속된 미래를 보장하는 신뢰의 기술입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 보폭 스케줄링은 친구들과 경주를 할 때 **'가장 뒤에 처진 친구부터 한 발자국씩 걷게 하는 것'**과 같아요.
2. 다리가 짧은 친구(중요한 친구)는 한 발자국이 아주 작아서, 다른 친구들보다 훨씬 더 많이 발을 움직여야 하죠.
3. 이렇게 계속 뒤에 있는 친구를 챙겨주니까, 누구 한 명도 뒤처지지 않고 모두가 골고루 앞으로 나갈 수 있답니다!
