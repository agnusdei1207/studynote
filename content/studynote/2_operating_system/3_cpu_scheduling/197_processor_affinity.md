# 프로세서 친화성 (Processor Affinity) - 캐시 최적화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세서 친화성 (Processor Affinity)은 멀티프로세서 환경에서 특정 프로세스가 이전에 실행되었던 **동일한 CPU 코어에서 계속 실행되도록** 유도하여 캐시 로컬리티(Cache Locality)를 극대화하는 스케줄링 정책이다.
> 2. **가치**: 프로세스가 코어를 옮길 때 발생하는 캐시 무효화(Invalidation)와 리필(Refill) 비용을 절감하여 연산 성능을 비약적으로 높이며, Soft 방식(권장)과 Hard 방식(강제)으로 나뉘어 유연성과 성능 사이의 균형을 맞춘다.
> 3. **융합**: 부하 분산(Load Balancing)과 상충하는 관계에 있으며, 현대 운영체제는 기본적으로 Soft Affinity를 유지하되 시스템 전체의 균형이 깨질 때만 프로세스를 다른 코어로 이주(Migration)시키는 지능형 조율을 수행한다.

+++

## Ⅰ. 프로세서 친화성의 개요

### 1. 정의
- 프로세스가 마지막에 실행된 처리기(Processor)에서 다시 실행되려는 경향을 의미한다.

### 2. 등장 배경: '따뜻해진 캐시 (Warm Cache)'
- 프로세스가 실행되면 CPU의 L1/L2 캐시에는 해당 프로세스의 데이터와 명령어가 가득 채워진다. 만약 프로세스가 다른 코어로 옮겨가면 기존 캐시는 버려지고(Cold Cache), 새 코어에서 데이터를 다시 불러오는 긴 지연 시간이 발생한다. 이를 방지하기 위해 '친화성' 개념이 도입되었다.

### 3. 💡 비유: '단골 식당의 지정석'
- 프로세서 친화성은 **'내가 자주 가는 식당에서 항상 앉던 자리에 앉는 것'**과 같다.
- 그 자리에는 내 취향에 맞는 양념과 도구가 이미 세팅되어 있어(캐시 데이터) 자리에 앉자마자 바로 식사를 시작할 수 있다. 만약 억지로 다른 테이블로 옮겨지면, 처음부터 다시 세팅하느라 밥 먹는 시간이 늦어지는 것과 같다.

+++

## Ⅱ. 친화성의 분류: Soft vs Hard (Deep Dive)

### 1. 소프트 친화성 (Soft Affinity)
- **방식**: 운영체제 스케줄러가 가급적 프로세스를 동일 코어에 배정하려고 노력하지만, **강제하지는 않는다**.
- **유연성**: 특정 코어에 부하가 너무 몰리면 공정성과 부하 분산을 위해 프로세스를 다른 한가한 코어로 옮길 수 있다. 현대 범용 OS의 기본값이다.

### 2. 하드 친화성 (Hard Affinity)
- **방식**: 시스템 호출(예: Linux의 `sched_setaffinity`)을 통해 특정 프로세스가 **지정된 코어 집합에서만** 실행되도록 못 박는 방식이다.
- **강제성**: 운영체제는 어떤 상황에서도 해당 프로세스를 지정되지 않은 다른 코어로 옮길 수 없다.

### 3. ASCII 구조 다이어그램
```text
    [ Soft Affinity ]                   [ Hard Affinity ]
    ┌──────────┐  ┌──────────┐          ┌──────────┐  ┌──────────┐
    │  Core 0  │  │  Core 1  │          │  Core 0  │  │  Core 1  │
    └────┬─────┘  └────┬─────┘          └────┬─────┘  └────┬─────┘
         │ (Try same)  │                     │ (Locked)    │ (Forbidden)
    ┌────▼─────┐       │                ┌────▼─────┐       │
    │ Process A│ ◀─────┘                │ Process B│ ◀─────┘
    └──────────┘  (If Load Balanced)    └──────────┘  (Never migrates)
```

- **📢 섹션 요약 비유**: 소프트 친화성은 "가급적 원래 쓰던 방을 써라"는 권고 사항이고, 하드 친화성은 "너는 101호 밖으로 절대 나가지 마라"는 엄격한 감금 규칙과 같습니다.

## Ⅲ. 친화성 유지와 부하 분산의 갈등

| 비교 항목 | 프로세서 친화성 (Affinity) | 부하 분산 (Load Balancing) |
|:---|:---|:---|
| **최우선 가치** | 개별 프로세스의 **연산 속도** | 시스템 전체의 **자원 효율** |
| **핵심 기술** | 캐시 데이터 보존 | 프로세스 이주 (Migration) |
| **성능 영향** | 캐시 미스 감소 | CPU 유휴 시간 감소 |
| **충돌 상황** | 특정 코어만 바쁘고 나머지는 놈 | 코어를 옮기느라 캐시가 다 날아감 |

- **📢 섹션 요약 비유**: 한 명의 요리사가 자기 주방 도구에 익숙해지게 두는 것(친화성)이 효율적이지만, 주문이 너무 밀리면 옆의 놀고 있는 요리사에게 일을 나눠줘야 하는(부하 분산) 딜레마를 해결하는 과정입니다.

## Ⅳ. 실무 적용 및 최적화 기법 (Strategy)

### 1. 실무 시나리오: 고성능 패킷 처리 (DPDK 등)
- **상황**: 초당 수천만 개의 패킷을 처리해야 하는 보안 장비 개발.
- **판단**: 스케줄러의 개입으로 코어가 바뀌면 패킷 처리 지연이 발생한다. **Hard Affinity**를 사용하여 네트워크 카드의 인터럽트를 처리하는 스레드를 특정 코어에 박아버린다(Core Pinning). 이제 해당 코어는 오직 패킷 처리 데이터만 캐시에 담게 되어 극강의 처리 속도를 확보한다.

### 2. 기술사적 인사이트: NUMA 친화성 (NUMA Affinity)
- 기술사라면 단순한 코어 친화성을 넘어 **메모리 노드 친화성**까지 고려해야 한다. 프로세스가 실행되는 코어와 해당 프로세스의 데이터가 담긴 물리 메모리(NUMA Node)가 멀어지면 성능이 수십 % 하락한다. 따라서 스케줄러는 코어 친화성을 지키면서도 메모리 위치를 고려한 **Local Memory 할당**을 병행해야 함을 강조해야 한다.

- **📢 섹션 요약 비조**: 내 손에 익은 도구(캐시)만큼이나, 재료 창고(메모리)가 내 주방에서 얼마나 가까운지가 요리(연산) 속도를 결정하는 결정적 요인입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **애플리케이션 가속**: 메모리 접근 지연 시간 단축으로 전체 실행 시간 10~20% 감소.
- **예측 가능한 성능**: 지터(Jitter) 감소로 실시간 서비스의 안정성 확보.

### 2. 미래 전망
최근에는 빅리틀(big.LITTLE)이나 하이브리드 아키텍처가 등장하면서, 프로세스의 성격에 따라 고성능 코어(P-core) 또는 고효율 코어(E-core)에 친화성을 부여하는 **이기종 친화성** 기술이 핵심이 되고 있다. 운영체제는 이제 단순히 '같은 코어'를 넘어 '가장 적합한 유형의 코어'를 찾아주는 지능형 큐레이터로 진화하고 있다.

- **📢 섹션 요약 비유**: 익숙한 장소에서 최고의 실력을 발휘하게 돕는, 운영체제의 배려 깊은 공간 관리 기술입니다.
