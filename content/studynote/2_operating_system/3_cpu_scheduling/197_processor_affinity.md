# 197. 레이트 모노토닉 (Rate Monotonic)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 주기가 짧은 태스크에 높은 우선순위 부여
> 2. **가치**: 고정 우선순위 실시간 스케줄링의 최적 알고리즘
> 3. **융합**: 실시간 스케줄링, 정적 우선순위, 스케줄 가능성과 연관

---

## Ⅰ. 개요

### 개념 정의

레이트 모노토닉(Rate Monotonic, RM) 스케줄링은 **주기적인 실시간 태스크에서 주기가 짧을수록 높은 우선순위를 부여하는 정적 우선순위 알고리즘**이다.

### 💡 비유: 심박수 순서
레이트 모노토닉은 **심박수가 빠른 환자 먼저 진료**하는 것과 같다. 심박수가 빠를수록(주기가 짧을수록) 더 긴급하므로 먼저 처리한다.

### 레이트 모노토닉 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                레이트 모노토닉 구조                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【핵심 원칙】                                                           │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  주기(Period) 짧음 → 우선순위 높음                              │ │
│  │  주기(Period) 김 → 우선순위 낮음                                 │ │
│  │                                                             │ │
│  │  정적 우선순위: 실행 전에 결정, 실행 중 변경 없음                   │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【예시】                                                                │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  태스크    주기(ms)    실행시간(ms)    우선순위(RM)                │ │
│  │  ────            ────               ────                 ────      │ │
│  │  T1          4           1              높음 (1)                   │ │
│  │  T2          6           2              중간 (2)                   │ │
│  │  T3          8           2              낮음 (3)                   │ │
│  │                                                             │ │
│  │  T1이 가장 짧은 주기 → 가장 높은 우선순위                          │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【스케줄 가능성 테스트】                                                │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  CPU 이용률: U = Σ(Ci/Ti)                                         │ │
│  │                                                             │ │
│  │  충분 조건 (RM):                                                  │ │
│  │  U ≤ n(2^(1/n) - 1)                                              │ │
│  │                                                             │ │
│  │  n=1: 100%                                                       │ │
│  │  n=2: 82.8%                                                      │ │
│  │  n=3: 77.9%                                                      │ │
│  │  n=∞: 69.3% (ln 2)                                               │
│  │                                                             │ │
│  │  위 예시: U = 1/4 + 2/6 + 2/8 = 0.25 + 0.33 + 0.25 = 0.83        │ │
│  │  n=3 한계 0.779보다 높음 → 보장 안 됨 (실제로는 가능할 수 있음)       │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                레이트 모노토닉 상세                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【특징】                                                               │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  장점:                                                           │ │
│  │  • 구현 간단                                                      │ │
│  │  • 오버헤드 적음                                                   │ │
│  │  • 예측 가능                                                      │ │
│  │  • 고정 우선순위로 분석 쉬움                                        │ │
│  │                                                             │ │
│  │  단점:                                                           │ │
│  │  • CPU 이용률 한계 (~69%)                                         │ │
│  │  • 주기적 태스크에만 적용                                           │ │
│  │  • 비주기적 태스크 처리 어려움                                      │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【최적성】                                                              │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  RM은 고정 우선순위 스케줄링 중 최적:                                │ │
│  │  • RM으로 스케줄 불가능하면, 다른 고정 우선순위도 불가능             │ │
│  │  • 단, 동적 우선순위(EDF)는 더 나을 수 있음                          │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【RM vs EDF】                                                           │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  구분              RM                   EDF                     │ │
│  │  ────                    ────                     ────           │ │
│  │  우선순위          정적                  동적                     │ │
│  │  CPU 이용률        ≤ 69%                ≤ 100%                   │ │
│  │  구현 복잡도        낮음                  높음                     │ │
│  │  런타임 오버헤드     적음                  많음                     │ │
│  │  예측성            높음                  낮음                     │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【RM 스케줄링 시뮬레이션】                                              │
│  ──────────────────                                                  │
│  #include <stdio.h>                                                  │
│  #include <stdlib.h>                                                 │
│                                                                     │
│  typedef struct {                                                    │
│      int id;                                                         │
│      int period;      // 주기                                         │
│      int exec_time;   // 실행 시간                                    │
│      int priority;    // RM 우선순위 (낮은 수 = 높은 우선순위)         │
│  } Task;                                                             │
│                                                                     │
│  int compare_period(const void *a, const void *b) {                  │
│      return ((Task*)a)->period - ((Task*)b)->period;                 │
│  }                                                                   │
│                                                                     │
│  void assign_rm_priority(Task tasks[], int n) {                      │
│      // 주기순 정렬 (짧은 주기 = 높은 우선순위)                           │
│      qsort(tasks, n, sizeof(Task), compare_period);                  │
│      for (int i = 0; i < n; i++) {                                   │
│          tasks[i].priority = i + 1;                                  │
│      }                                                               │
│  }                                                                   │
│                                                                     │
│  double calculate_utilization(Task tasks[], int n) {                 │
│      double util = 0;                                                │
│      for (int i = 0; i < n; i++) {                                   │
│          util += (double)tasks[i].exec_time / tasks[i].period;       │
│      }                                                               │
│      return util;                                                    │
│  }                                                                   │
│                                                                     │
│  int is_rm_schedulable(double util, int n) {                         │
│      double bound = n * (pow(2, 1.0/n) - 1);                         │
│      return util <= bound;                                           │
│  }                                                                   │
│                                                                     │
│  【Linux에서 RM 구현】                                                  │
│  ──────────────────                                                  │
│  // SCHED_FIFO로 RM 구현                                              │
│  void setup_rm_task(int period_ms, int priority) {                   │
│      struct sched_param param;                                       │
│      param.sched_priority = priority;  // 낮은 수 = 높은 우선순위      │
│      sched_setscheduler(0, SCHED_FIFO, &param);                      │
│  }                                                                   │
│                                                                     │
│  void periodic_task(int period_ms, void (*work)()) {                 │
│      while (1) {                                                     │
│          struct timespec next;                                       │
│          clock_gettime(CLOCK_MONOTONIC, &next);                      │
│          next.tv_nsec += period_ms * 1000000;                        │
│                                                                     │
│          work();  // 실제 작업                                        │
│          clock_nanosleep(CLOCK_MONOTONIC, TIMER_ABSTIME, &next, NULL);│
│      }                                                               │
│  }                                                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 주기 짧은 태스크에 높은 우선순위

• 최적성: 고정 우선순위 중 최적

• 적용: 주기적 태스크 시스템
```
