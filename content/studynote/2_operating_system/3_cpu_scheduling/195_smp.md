# # [대칭 다중 처리 (SMP) 스케줄링]

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: SMP (Symmetric Multiprocessing)는 복수의 프로세서가 단일의 공유 메모리 주소 공간과 운영체제 커널을 대등한 권한으로 공유하는 아키텍처이며, 이를 통해 병렬 처리 성능을 극대화하는 것이 핵심 원리이다.
> 2. **가치**: 마스터(Master) 코어의 병목을 제거하여 수평적 확장성(Scalability)을 확보하며, 멀티코어 환경에서 처리량(Throughput)을 선형적으로 증가시킬 수 있는 구조적 기반을 제공한다.
> 3. **융합**: 하드웨어적 캐시 일관성(Cache Coherence) 프로토콜(MESI)과 운영체제의 세밀한 동기화 기법(Spinlock, RCU)이 결합되어야만 설계된 성능을 달성할 수 있는 고도화된 시스템 설계 영역이다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: SMP (Symmetric Multiprocessing)는 시스템 내의 모든 프로세서(CPU)가 동등한 지위를 가지고 메인 메모리(Main Memory), 입출력(I/O) 장치, 그리고 운영체제 커널(Operating System Kernel) 영역에 접근하고 실행할 수 있는 컴퓨터 아키텍처를 의미한다. 단순히 여러 개의 CPU가 장착된 것을 넘어, 모든 CPU가 인터럽트(Interrupt) 처리를 포함한 모든 작업을 수행할 수 있는 '대칭성'이 핵심 특징이다.
- **💡 비유**: SMP는 **'원탁의 기사 회의'**와 같다. 회의 테이블(메모리)에 둘러앉은 모든 기사(CPU)는 평등한 발언권(커널 실행 권한)을 가지며, 누구나 문서(프로세스)를 집어 실행할 수 있다. 반면 비대칭형(ASMP)은 왕(King) 혼자만 명령을 내리고 신하는 복종만 하는 구조에 불과하다.
- **등장 배경**: 
    ① **기존 한계**: 초기 다중 처리 시스템(ASMP)에서는 운영체제 커널 코드를 실행할 수 있는 CPU가 하나로 제한되어 있었다. 이로 인해 애플리케이션 처리 속도는 빨라졌으나, 시스템 콜(System Call)이나 인터럽트가 빈번하게 발생하는 환경에서는 단일 마스터 CPU가 병목(Bottleneck)이 되어 전체 성능이 급격히 저하되었다.
    ② **혁신적 패러다임**: 모든 CPU가 운영체제를 병렬로 실행할 수 있도록 허용하여, 커널 자체의 처리 성능을 코어 수에 비례해 높이자는 패러다임이 등장했다.
    ③ **현재의 비즈니스 요구**: 클라우드(Cloud) 시대의 고가용성(High Availability)과 대규모 트랜잭션 처리 요구를 충족시키기 위해 x86 서버부터 모바일 AP(Application Processor)까지 표준 아키텍처로 자리 잡았다.

```text
      [ ASMP (Asymmetric) ]              [ SMP (Symmetric) ]
      
   +-------------------+           +-------------------+
   |     Master CPU    |           |      CPU 0        |
   | (OS Kernel Only)  |           | (User + Kernel)   |
   +--------+----------+           +---------+---------+
            |                              |
            v                              v
   +--------+----------+           +---------+---------+
   |    Slave CPUs     |           |      CPU 1        |
   | (App Processing)  |           | (User + Kernel)   |
   +-------------------+           +---------+---------+
                                           |
                                           v
                                   +---------+---------+
                                   |      CPU 2        |
                                   | (User + Kernel)   |
                                   +-------------------+
```
*ASMP는 마스터 코어에 모든 OS 관리 부하가 집중되어 병목이 발생하지만, SMP는 모든 코어가 관리 작업을 분담하여 효율적이다.*

> **📢 섹션 요약 비유**: 마치 고속도로 톨게이트에 차선을 여러 개 뚫어두고, 모든 요금 정산기(CPU)가 평등하게 차량을 처리할 수 있도록 하여 특정 게이트가 막히는 현상을 방지하는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

SMP 시스템은 단순한 하드웨어 결합이 아니라, 메모리 접근, 캐시 일관성, 인터럽트 분산, 그리고 커널 동기화라는 복잡한 소프트웨어/하드웨어 계층이 얽혀 있는 정교한 시스템이다.

#### 1. SMP 시스템 구조도 (Architecture)

아래 다이어그램은 SMP 시스템에서 각 CPU가 공유 버스(또는 인터커넥트)를 통해 메모리와 I/O에 접근하며, 독립된 캐시(Cache)를 유지하는 전형적인 구조를 보여준다.

```text
      [CPU 0]                       [CPU 1]                       [CPU 2]
 ┌─────────────────────┐     ┌─────────────────────┐     ┌─────────────────────┐
 │  Core 0 Execution    │     │  Core 1 Execution    │     │  Core 2 Execution    │
 │  (User / Kernel)     │     │  (User / Kernel)     │     │  (User / Kernel)     │
 └──────────┬───────────┘     └──────────┬───────────┘     └──────────┬───────────┘
            │                            │                            │
 ┌──────────┴───────────┐     ┌──────────┴───────────┐     ┌──────────┴───────────┐
 │   L1 / L2 Cache      │     │   L1 / L2 Cache      │     │   L1 / L2 Cache      │
 │   (Private)          │     │   (Private)          │     │   (Private)          │
 └──────────┬───────────┘     └──────────┬───────────┘     └──────────┬───────────┘
            │ Snooping(Bus)            │ Snooping(Bus)            │ Snooping(Bus)
            └─────────────┬────────────┴────────────┬────────────┘
                          │                         │
          ╔═══════════════╧═════════════════════════╧════════════════════╗
          ║               System Bus / Interconnect                   ║
          ║       (FSB: Front Side Bus or Point-to-Point Link)        ║
          ╚═══════════════╤═════════════════════════╤════════════════════╝
                          │                         │
          ┌───────────────┴─────────────┐ ┌─────────┴───────────────┐
          │    Main Memory (DRAM)       │ │   I/O Devices (NIC/Disk)  │
          │    (Shared Address Space)   │ │   (Shared Access)         │
          └─────────────────────────────┘ └───────────────────────────┘
```

**[다이어그램 해설]** SMP의 가장 큰 특징은 메모리가 '단일 주소 공간(Single Address Space)'을 가진다는 점이다. CPU 0이 주소 `0x1000`에 데이터를 쓰면, CPU 1과 CPU 2도 동일한 주소 `0x1000`을 통해 해당 데이터에 접근해야 한다. 각 CPU는 자신만의 L1/L2 캐시(Private Cache)를 가지고 있어 성능을 높이지만, 이로 인해 동일한 데이터의 사본이 여러 개 생겨 데이터 일관성(Data Consistency) 문제가 발생한다. 이를 해결하기 위해 시스템 버스 단에서 하드웨어적인 '스누핑(Snooping)' 기능을 수행하며, I/O 장치 역시 모든 CPU가 공통으로 사용한다.

#### 2. 핵심 구성 요소 및 동작 매커니즘

SMP의 안정적인 작동을 위해 다음과 같은 주요 구성 요소들이 상호작용한다.

| 구성 요소 | 역할 및 내부 동작 | 주요 프로토콜/기술 | 관련 비유 |
|:---:|:---|:---|:---|
| **공유 메모리<br>(Shared Memory)** | 모든 CPU가 공통으로 접근하는 물리적 저장소. 단일 주소 공간을 제공하여 어떤 CPU가 쓴 데이터도 다른 CPU가 읽을 수 있게 함. | UMA (Uniform Memory Access) | 회의실 중앙의 투명한 칠판 |
| **캐시 일관성<br>(Cache Coherence)** | 분산된 캐시 간 데이터 불일치를 방지. 한 CPU가 데이터를 수정하면 다른 CPU의 캐시 데이터를 무효화(Invalidation)하거나 갱신(Update)함. | **MESI Protocol**<br>(Modified, Exclusive, Shared, Invalid) | 칠판 내용을 수정하면 모두에게 지우라고 알림 |
| **동기화 기법<br>(Synchronization)** | 여러 CPU가 동시에 커널 자료구조(예: 실행 큐)를 접근할 때 데이터 파손을 막기 위해 상호 배제(Mutual Exclusion)를 보장함. | **Spinlock**, Semaphore, RCU | 화자 지정권(마이크)을 한 명에게만 허용 |
| **인터럽트 분산<br>(IRQ Balancing)** | 특정 CPU에 인터럽트 부하가 몰리는 것을 방지하기 위해 I/O 장치의 인터럽트를 여러 CPU에 균등하게 분배하거나 특정 CPU로 리다이렉트함. | IO-APIC, MSI (Message Signaled Interrupts) | 전화 문의 상담원에게 호출을 골고루 분배 |
| **부하 균등화<br>(Load Balancing)** | 특정 CPU는 바쁘고 다른 CPU는 놀고 있는 상황을 방지하기 위해 OS 스케줄러가 실행 큐(Queue)의 프로세스를 다른 CPU로 이주(Migration)시킴. | Push/Pull Migration (Linux CFS) | 바쁜 사람의 업무를 한가한 사람에게 떠넘기기 |

#### 3. 핵심 알고리즘: MESI 프로토콜 상태 전이

SMP의 성능은 캐시 일관성 프로토콜의 효율성에 달려있다. 가장 널리 쓰이는 **MESI (Modified, Exclusive, Shared, Invalid)** 프로토콜의 상태 천이 과정은 다음과 같다.

```text
             (Read Miss)
        +--------------------> [S] Shared (공유 상태)
        |                     (여러 캐시가 읽기 권한 보유)
        |                         |
   [I]  Invalid                     | (다른 캐시에서 쓰기 발생 시)
   (무효 상태)                      v
        ^                     [I] Invalid (무효화)
        |                         ^
        |          (Bus Write - Invalidate Signal)
        |                         |
        |    [M] Modified (수정 상태)
        |    (단일 캐시만 독점적 소유/수정 권한 보유)
        |         ^
        |         | (Read Hit -> Read Miss)
        +---------+
```
*Cache Line State Transition Diagram*

**[다이어그램 해설]** MESI 프로토콜은 캐시 라인(Cache Line)의 상태를 4가지로 관리하여 데이터 일관성을 유지한다.
1.  **Modified (M)**: 해당 캐시만 유일한 데이터를 가지며, 메모리 데이터와 다름(Dirty). 쓰기 권한 독점.
2.  **Exclusive (E)**: 해당 캐시만 데이터를 가지며, 메모리와 데이터가 같음(Clean). 다른 캐시에 없음. 쓰기 즉시 M으로 전이.
3.  **Shared (S)**: 다른 캐시에도 동일한 데이터가 존재할 수 있음. 읽기만 가능.
4.  **Invalid (I)**: 데이터가 유효하지 않음.
CPU가 데이터를 쓰려고 할 때, 다른 CPU가 S 상태를 가지고 있다면 버스를 통해 무효화 신호(Invalidation)를 보내 I로 만든 후 M 상태로 변경하여 쓴다. 이 과정에서 버스 트래픽이 발생한다.

#### 4. 커널 데이터 동기화 (Pseudo Code)

SMP 환경의 커널에서는 두 개 이상의 CPU가 커널의 핵심 자료구조(예: 프로세스 연결 리스트)를 동시에 수정하는 것을 막기 위해 **스핀락(Spinlock)**을 사용한다.

```c
// 스핀락을 이용한 크리티컬 섹션 보호 (C 언어 스타일 의사코드)
void kernel_critical_section() {
    // 1. 락 획득 시도 (원자적 명령어 Test-and-Set 사용)
    // 다른 코어가 락을 잡고 있으면 False를 반환하며 대기(Busy Waiting)
    while (atomic_test_and_set(&lock->state) == LOCKED) {
        // CPU가 놀지 않고 계속 루프를 돌며 락을 확인함 (Context Switch 발생 안 함)
        pause(); // 하이퍼스레딩/파이프라인 최적화를 위한 pause 명령어
    }

    // === 크리티컬 섹션 (Critical Section) ===
    // 공유 자원(예: Run Queue) 수정
    global_run_queue->add(process); 
    // =====================================

    // 2. 락 해제
    atomic_release(&lock->state);
}
```

> **📢 섹션 요약 비유**: 수십 명의 요리사가 부엌에서 일할 때, 냉장고(공유 메모리)에 있는 재료를 가져가기 위해 서로 "지금 사용 중이다!"라고 선언(락킹)하고, 재료를 가져다 쓰면 다른 요리사들의 손에 들린 똑같은 재료 사본을 버리게(캐시 무효화) 하는 정교한 식당 운영 시스템과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

SMP는 단순한 HW 구조를 넘어 OS의 스케줄링 전략과 밀접하게 연관되어 있으며, 비대칭 처리(ASMP) 및 분산 메모리 구조(NUMA)와 명확히 구분된다.

#### 1. 심층 기술 비교표: SMP vs ASMP vs NUMA

| 구분 | **SMP (Symmetric)** | **ASMP (Asymmetric)** | **NUMA (Non-Uniform Memory Access)** |
|:---:|:---|:---|:---|
| **정의** | 모든 프로세서가 메모리와 I/O에 대해 **동등한 접근 권한**을 가짐 | 마스터 프로세서가 시스템 제어를 독점하고, 슬레이브는 연산만 담당 | 프로세서가 **로컬 메모리(Local)**와 **원격 메모리(Remote)**에 접근 속도가 다름 |
| **메모리 접근** | **UMA
