# 199. 데드라인 (Deadline)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 작업이 반드시 완료되어야 하는 시점
> 2. **가치**: 실시간 시스템의 핵심 제약 조건
> 3. **융합**: 실시간 스케줄링, EDF, RM과 연관

---

## Ⅰ. 개요

### 개념 정의

데드라인(Deadline)은 **실시간 태스크가 반드시 완료되어야 하는 절대 시간 또는 상대 시간**이다. 실시간 시스템에서 가장 중요한 제약 조건이다.

### 💡 비유: 기차 출발 시간
데드라인은 **기차 출발 시간**과 같다. 기차는 정해진 시간에 출발하므로, 그 전에 탑승해야 한다. 1초라도 늦으면 탑승 불가다.

### 데드라인 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                데드라인 구조                                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【데드라인 유형】                                                        │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  절대 데드라인 (Absolute Deadline):                              │ │
│  │  • 특정 시점 (예: 14:00:00)                                      │ │
│  │  • 시계 시간 기준                                                │ │
│  │                                                             │ │
│  │  상대 데드라인 (Relative Deadline):                              │ │
│  │  • 릴리스 시점으로부터의 시간 (예: 릴리스 후 100ms 내)              │ │
│  │  • 가장 일반적                                                   │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【데드라인 타임라인】                                                    │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  릴리스        실행 시작        실행 중        데드라인           │ │
│  │    r     ───────►    │    ──────────────►    │        d        │ │
│  │                       │                        │               │ │
│  │                       └────────────────────────┘               │ │
│  │                                실행 시간 (C)                     │ │
│  │                                                             │ │
│  │  ┌──────────────────────────────────────────────────────────┐ │ │
│  │  │                여유 시간 (Slack/Laxity)                   │ │ │
│  │  │  slack = (d - r) - C                                      │ │ │
│  │  │  slack > 0: 여유 있음                                      │ │ │
│  │  │  slack = 0: 정확히 맞춤                                     │ │ │
│  │  │  slack < 0: 데드라인 위반                                   │ │ │
│  │  └──────────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【주기적 태스크의 데드라인】                                              │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  태스크 T(period=10, exec=3, deadline=10):                       │ │
│  │                                                             │ │
│  │  시간  0   10   20   30   40   50                             │ │
│  │       │    │    │    │    │    │                              │ │
│  │       ├──┤ ├──┤ ├──┤ ├──┤ ├──┤                               │ │
│  │       d1   d2   d3   d4   d5                                  │ │
│  │                                                             │ │
│  │  각 주기마다 새로운 데드라인 발생                                  │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                데드라인 상세                                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【경성 vs 연성 데드라인】                                                │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  경성 데드라인 (Hard Deadline):                                   │ │
│  │  • 위반 시 치명적 결과                                            │ │
│  │  • 반드시 보장 필요                                               │ │
│  │  • 예: 자동차 브레이크, 원자로 제어                                │ │
│  │                                                             │ │
│  │  연성 데드라인 (Soft Deadline):                                   │ │
│  │  • 위반 시 성능 저하만 발생                                        │ │
│  │  • 최선 노력으로 충족                                              │ │
│  │  • 예: 비디오 프레임, 오디오 샘플                                  │ │
│  │                                                             │ │
│  │  Firm Deadline:                                                   │ │
│  │  • 위반 시 결과 무용지물 (폐기)                                    │ │
│  │  • 예: 센서 데이터 (오래되면 의미 없음)                             │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【데드라인과 스케줄링】                                                  │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  알고리즘              데드라인 활용                               │ │
│  │  ────                        ────                                  │ │
│  │  EDF                  데드라인 가까운 것 우선                       │ │
│  │  RM                   주기 사용 (데드라인=주기 가정)                 │ │
│  │  DM                   데드라인 짧은 것 우선                         │ │
│  │  LST                  여유 시간(slack) 적은 것 우선                 │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【데드라인 미스의 영향】                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • 시스템 안정성 저하                                              │ │
│  │  • 데이터 손실                                                    │ │
│  │  • 안전 문제                                                      │ │
│  │  • 사용자 경험 악화                                                │ │
│  │  • 시스템 오작동                                                   │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 모니터링 방법
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【데드라인 모니터링 코드】                                              │
│  ──────────────────                                                  │
│  #include <time.h>                                                   │
│  #include <stdio.h>                                                  │
│                                                                     │
│  typedef struct {                                                    │
│      struct timespec release;                                        │
│      struct timespec deadline;                                       │
│      int exec_time_ms;                                               │
│  } RealTimeTask;                                                     │
│                                                                     │
│  int check_deadline(RealTimeTask *task) {                            │
│      struct timespec now;                                            │
│      clock_gettime(CLOCK_MONOTONIC, &now);                           │
│                                                                     │
│      if (timespec_cmp(&now, &task->deadline) > 0) {                  │
│          printf("Deadline missed!\n");                               │
│          return -1;  // 데드라인 위반                                 │
│      }                                                               │
│      return 0;  // 정상                                              │
│  }                                                                   │
│                                                                     │
│  long get_slack(RealTimeTask *task) {                                │
│      struct timespec now;                                            │
│      clock_gettime(CLOCK_MONOTONIC, &now);                           │
│      return timespec_diff_ms(&now, &task->deadline) - task->exec_time_ms;│
│  }                                                                   │
│                                                                     │
│  【Linux 데드라인 스케줄링】                                             │
│  ──────────────────                                                  │
│  // SCHED_DEADLINE 설정                                               │
│  struct sched_attr attr;                                             │
│  attr.sched_policy = SCHED_DEADLINE;                                 │
│  attr.sched_runtime = 10 * 1000 * 1000;   // 10ms                    │
│  attr.sched_deadline = 30 * 1000 * 1000;   // 30ms                   │
│  attr.sched_period = 30 * 1000 * 1000;     // 30ms                   │
│  sched_setattr(0, &attr, 0);                                          │
│                                                                     │
│  // 데드라인 통계 확인                                                 │
│  $ cat /proc/<pid>/sched | grep dl                                   │
│                                                                     │
│  【데드라인 위반 감지】                                                  │
│  ──────────────────                                                  │
│  // cyclictest로 데드라인 위반 측정                                    │
│  $ cyclictest -t1 -p 80 -n -i 1000 -l 10000                          │
│  # Max 값이 목표 데드라인 초과하면 위반                                   │
│                                                                     │
│  // ftrace로 스케줄링 지연 분석                                        │
│  $ trace-cmd record -e sched:sched_switch \                         │
│      -e sched:sched_wakeup ./realtime_app                            │
│  $ trace-cmd report                                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```

• 위반: 경성=치명적, 연성=성능저하

```
