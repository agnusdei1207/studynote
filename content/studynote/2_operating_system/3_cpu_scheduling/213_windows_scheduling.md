# 윈도우 스케줄링 (Windows Scheduling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 우선순위 기반의 선점형 (Preemptive) 다단계 피드백 큐 (MLFQ) 스케줄링 아키텍처를 사용하며, 스레드를 실행 단위로 관리하는 윈도우 커널의 핵심 메커니즘이다.
> 2. **가치**: 32단계의 정교한 우선순위와 동적 우선순위 승급 (Priority Boost) 기능을 통해, GUI 기반 대화형 응용 프로그램의 사용자 반응성을 극대화하는 동시에 서버 작업의 효율성을 보장한다.
> 3. **융합**: 디스패처 객체 (Dispatcher Object), 퀀텀 (Quantum) 관리 기술, 그리고 최신 버전의 하이브리드 코어(P-core/E-core) 최적화 전략이 통합되어 있다.

+++

## Ⅰ. 개요 (Context & Background)

- **개념**: 윈도우 운영체제는 '프로세스'가 아닌 '스레드'를 스케줄링의 기본 단위로 삼는다. 디스패처 (Dispatcher)라고 불리는 커널 모듈이 32개의 실행 큐를 관리하며, 가장 높은 우선순위의 준비된 스레드를 찾아 CPU에 할당한다.

- **💡 비유**: 윈도우 스케줄링은 "VIP 등급이 수시로 변하는 백화점 고객 관리"와 같다. 기본 등급이 있지만, 지금 당장 점원과 대화 중이거나(I/O 완료) 화면 맨 앞에 서 있는 고객(전경 윈도우)에게는 임시로 VIP 카드를 발급(Priority Boost)하여 빠른 서비스를 제공하는 방식이다.

- **📢 섹션 요약 비유**: 철저한 계급 사회(우선순위)를 유지하면서도, 급한 사정이 생긴 사람에게는 일시적으로 높은 계급을 부여하여 사회 전체의 활력(반응성)을 유지하는 유연한 시스템입니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 32단계 우선순위 구조 (Priority Levels)

윈도우 스레드 우선순위는 0부터 31까지로 나뉘며, 크게 두 영역으로 구분된다.

```text
  [ Real-time Levels (16-31) ]  <-- 우선순위 변동 없음 (고정)
  [ Variable Levels (1-15) ]    <-- 상황에 따라 Boost 발생
  [ Zero Page Thread (0) ]      <-- 유휴 시 메모리 제로화 작업
  
  Ready Queue (Priority 31): [T1] -> [T2]
  Ready Queue (Priority 30): [ ]
  ...
  Ready Queue (Priority 8):  [T3] <-- 일반적인 앱들
```

**[다이어그램 해설]** 
1. **실시간 영역 (16-31)**: 커널이 우선순위를 절대 건드리지 않는다. 마감 시간이 중요한 작업에 사용된다.
2. **가변 영역 (1-15)**: 대부분의 사용자 프로그램이 속하며, 스케줄러가 반응성을 위해 우선순위를 위아래로 조정한다.
3. **디스패처 비트맵**: 어떤 순위에 스레드가 있는지 O(1)로 찾기 위해 비트맵을 사용한다 (리눅스 O(1)과 유사).

### 동적 우선순위 승급 (Priority Boost) 조건
- **I/O 완료**: 키보드나 마우스 입력을 기다리다 완료된 스레드는 일시적으로 우선순위가 크게 뛴다.
- **전경 (Foreground) 프로세스**: 현재 사용자가 보고 있는 창의 스레드는 퀀텀(시간 할당량)을 3배가량 늘려준다.
- **기아 방지 (Hunger Management)**: 낮은 순위에서 3~4초간 실행 못한 스레드는 강제로 15단계로 올려서 한 번 실행시켜준다.

- **📢 섹션 요약 비유**: 무대 주인공(전경 창)에게는 마이크 사용 시간(퀀텀)을 넉넉히 주고, 너무 오래 기다린 보조 출연자(기아 스레드)에게는 가끔 단독 대사 기회를 주는 감독의 연출과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 윈도우 스케줄링 vs 리눅스 CFS

| 비교 항목 | 윈도우 (Windows) | 리눅스 (CFS) |
|:---|:---|:---|
| **핵심 철학** | 우선순위 기반 응답성 강조 | 가상 시간(vruntime) 기반 공정성 강조 |
| **스케줄링 단위** | 스레드 (Thread) | 태스크 (Task - 프로세스/스레드 통합) |
| **시간 할당량** | 퀀텀 (Quantum - 고정/부스트) | 가변적 (태스크 수에 비례) |
| **우선순위 조정** | 명시적인 Boost/Decay | vruntime 증가 속도 차등 (Nice) |

- **📢 섹션 요약 비유**: 윈도우는 "중요한 놈을 확실히 밀어주자"는 주의이고, 리눅스는 "아무도 서운하지 않게 똑같이 나눠주자"는 주의에 가깝습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 서버 성능을 위한 퀀텀 최적화
- **상황**: 윈도우 서버(Windows Server)에서 돌아가는 데이터베이스 성능이 워크스테이션(Windows 10/11)보다 더 안정적으로 나옴.
- **판단**: 윈도우는 OS 에디션에 따라 스케줄링 기본값이 다르다. 데스크탑용은 짧고 변동이 심한 퀀텀을 써서 반응성을 높이지만, 서버용은 길고 고정된 퀀텀을 써서 문맥 교환을 줄이고 처리량을 높인다. 실무자는 `고급 시스템 설정 -> 성능 -> 고급` 탭에서 '프로그램' 대신 '백그라운드 서비스'를 선택하여 수동으로 서버 최적화 스케줄링을 활성화할 수 있다.

- **📢 섹션 요약 비유**: 짧은 수다(데스크탑용 퀀텀)는 즐겁지만, 큰일을 하려면 깊은 몰입 시간(서버용 퀀텀)이 필요하므로 용도에 맞게 설정을 바꿔야 합니다.

+++

## Ⅴ. 기대효과 및 결론

- **기대효과**: 수백 개의 백그라운드 프로세스가 돌아가는 복잡한 GUI 환경에서도 사용자가 지연을 느끼지 못하게 하는 매끄러운 반응성을 보장한다.
- **미래 전망**: 인텔 Alder Lake 이후의 하이브리드 아키텍처 대응을 위해, 윈도우 11 스케줄러는 **Thread Director** 하드웨어 피드백을 받아 P-코어(성능)와 E-코어(효율)에 스레드를 지능적으로 배치하는 방향으로 고도화되고 있다.

- **📢 섹션 요약 비유**: 하드웨어라는 악기와 스케줄러라는 지휘자가 만나, 사용자 경험이라는 아름다운 교향곡을 연주하는 과정입니다.
