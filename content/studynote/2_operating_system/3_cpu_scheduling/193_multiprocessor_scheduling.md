# 다중 처리기 스케줄링 (Multiprocessor Scheduling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다중 처리기 스케줄링 (Multiple-Processor Scheduling)은 여러 개의 CPU 또는 코어가 존재하는 시스템에서 **어느 프로세스를 어느 처리기에 배정할지** 결정하여 병렬 연산 성능을 최적화하는 기술이다.
> 2. **가치**: 단일 처리기 환경의 복잡도를 넘어 부하 분산(Load Balancing)과 프로세서 친화성(Processor Affinity) 사이의 균형을 맞춤으로써, 하드웨어 자원의 활용도를 극대화하고 시스템의 전체 처리량(Throughput)을 비약적으로 향상시킨다.
> 3. **융합**: 비대칭 다중 처리(ASMP)와 대칭 다중 처리(SMP)라는 두 가지 아키텍처로 진화해 왔으며, 현대 운영체제는 모든 코어가 대등하게 업무를 나누는 SMP 방식을 기반으로 지능형 스케줄링을 수행한다.

+++

## Ⅰ. 다중 처리기 스케줄링의 개요

### 1. 정의
- 두 개 이상의 독립적인 CPU(또는 코어)가 하나의 시스템 공유 메모리를 사용하며 협력하는 환경에서, 실행 가능한 프로세스들을 각 CPU에 효율적으로 할당하는 과정이다.

### 2. 주요 해결 과제: '병렬성의 딜레마'
- **부하 분산 (Load Balancing)**: 노는 코어 없이 모두가 바쁘게 일하게 해야 한다.
- **오버헤드 관리**: 여러 코어가 하나의 준비 큐를 공유할 때 발생하는 락(Lock) 경합을 줄여야 한다.
- **데이터 로컬리티**: 이전에 실행되었던 코어에서 다시 실행하여 캐시 적중률을 높여야 한다.

### 3. 💡 비유: '여러 개의 계산대가 있는 대형 마트'
- 다중 처리기 스케줄링은 **'손님이 몰리는 마트에서 여러 명의 계산원(CPU)에게 손님(프로세스)을 어떻게 나누어 보낼지 결정하는 관리자'**와 같다.
- 한 줄만 세워서 빈 계산대로 보낼지(Global Queue), 계산대마다 줄을 따로 세울지(Per-CPU Queue) 결정하는 것이 스케줄링의 핵심이다.

+++

## Ⅱ. 다중 처리기 아키텍처의 분류 (Deep Dive)

처리기에 역할을 어떻게 배분하느냐에 따라 크게 두 가지로 나뉜다.

### 1. 비대칭 다중 처리 (ASMP, Asymmetric Multiprocessing)
- **방식**: 하나의 '마스터' 처리기가 스케줄링과 I/O 처리를 전담하고, 나머지 '슬레이브' 처리기들은 연산만 수행한다.
- **특징**: 데이터 공유와 락 관리가 단순하지만, 마스터 처리기가 병목(Bottleneck)이 될 수 있다.

### 2. 대칭 다중 처리 (SMP, Symmetric Multiprocessing)
- **방식**: 모든 처리기가 대등한 권한을 가지며, 각자 스스로 스케줄링을 수행한다.
- **특징**: 현대 운영체제의 표준이다. 설계가 복잡하지만 확장성이 뛰어나고 병목 현상이 적다.

### 3. ASCII 구조 다이어그램
```text
    [ ASMP 모델 ]                       [ SMP 모델 ]
    ┌───────────┐ (Scheduling)          ┌───────────┬───────────┐
    │ Master CPU│ ───┐                  │ CPU 1     │ CPU 2     │
    └───────────┘    │                  │ (Self-Sch)│ (Self-Sch)│
    ┌───────────┐    ▼                  └─────┬─────┴─────┬─────┘
    │ Slave CPU │ ◀──┘                        │           │
    └───────────┘                             ▼           ▼
                                        ┌───────────────────────┐
                                        │    Common Ready Queue │
                                        └───────────────────────┘
```

- **📢 섹션 요약 비유**: 사장님이 모든 업무를 배분하는 수직적 조직(ASMP)과, 팀원들이 각자 할 일을 찾아 하는 수평적 조직(SMP)의 차이입니다.

## Ⅲ. 다중 처리기 스케줄링의 핵심 전략

| 전략 기법 | 설명 | 목적 |
|:---|:---|:---|
| **프로세서 친화성** | 프로세스가 이전에 실행된 코어에서 계속 실행되도록 유도 | 캐시 재사용 및 성능 향상 |
| **부하 분산** | 바쁜 코어의 작업을 노는 코어로 강제 이동 (Push/Pull) | 자원 활용률의 균등화 |
| **갱 스케줄링** | 관련된 스레드들을 여러 코어에 동시에 배치 | 스레드 간 통신 지연 최소화 |

- **📢 섹션 요약 비유**: 단골 손님을 원래 응대하던 점원이 계속 응대하게 하여(친화성), 업무 효율을 높이면서도 줄이 너무 길어지면 옆 줄로 옮겨주는(부하 분산) 유연한 매장 관리와 같습니다.

+++

## Ⅳ. 실무 적용 및 운영 인사이트 (Strategy)

### 1. 실무 시나리오: NUMA(Non-Uniform Memory Access) 환경 최적화
- **상황**: 수십 개의 CPU 코어와 거대한 메모리를 가진 엔터프라이즈 서버.
- **판단**: 멀리 있는 메모리에 접근하는 것은 오버헤드가 크다. 스케줄러는 프로세스가 현재 접근 중인 메모리 노드와 물리적으로 가까운 CPU 코어에 해당 프로세스를 고정(Binding)하는 **NUMA-aware 스케줄링**을 수행해야 한다. 기술사라면 이를 통해 메모리 대역폭 병목을 해결하는 전략을 제시해야 한다.

### 2. 기술사적 인사이트: 하이퍼스레딩 (SMT)과 스케줄링
- 물리 코어는 하나지만 논리 코어가 두 개로 보이는 하이퍼스레딩 환경에서, 스케줄러는 두 논리 코어가 실제로는 연산 유닛을 공유함을 인지해야 한다. 기술사라면 성능 향상을 위해 서로 다른 물리 코어에 작업을 우선 배정하고, 자리가 없을 때만 하이퍼스레딩 코어를 채우는 **'스마트 코어 배정'** 논리를 강조해야 한다.

- **📢 섹션 요약 비조**: 겉으로 보이는 의자 개수(논리 코어)만 보고 손님을 앉히는 것이 아니라, 주방의 실제 화력(물리 자원)을 고려하여 주문을 넣는 것이 진정한 고수의 스케줄링입니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **처리량(Throughput) 극대화**: 여러 코어를 동시에 가동하여 작업 완료 시간 단축.
- **고가용성 확보**: 특정 CPU 장애 시 다른 CPU가 업무를 즉시 승계 가능.

### 2. 미래 전망
최근에는 고성능 코어(P-core)와 고효율 코어(E-core)가 섞인 **이기종 다중 처리 (HMP)** 환경이 모바일을 넘어 데스크톱으로 확산되고 있다. 운영체제 스케줄러는 이제 단순히 '노는 코어'를 찾는 것을 넘어, 작업의 성격(연산 vs 배경 작업)에 가장 적합한 '유형의 코어'를 선택하는 지능형 매칭 엔진으로 진화하고 있다.

- **📢 섹션 요약 비유**: 각기 다른 재능을 가진 팀원들에게 가장 잘하는 일을 맡겨 최고의 성과를 내는, 미래형 리더십 스케줄링의 시대입니다.
