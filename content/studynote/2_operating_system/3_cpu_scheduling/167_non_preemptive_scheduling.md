# 비선점형 스케줄링 (Non-preemptive Scheduling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비선점형 스케줄링 (Non-preemptive Scheduling)은 일단 CPU를 할당받은 프로세스가 스스로 CPU를 반납(종료 또는 I/O 요청)할 때까지 **운영체제가 CPU를 강제로 뺏을 수 없는** 방식이다.
> 2. **가치**: 강제적인 문맥 교환이 일어나지 않아 오버헤드가 매우 적으며, 실행 순서가 단순하고 예측 가능하여 일괄 처리(Batch) 시스템이나 정해진 순서가 중요한 환경에 유리하다.
> 3. **융합**: FCFS, SJF, HRN 등의 알고리즘이 대표적이며, 대화형(Interactive) 시스템보다는 공정성과 오버헤드 최소화가 우선인 미션 크리티컬한 실시간 시스템의 일부 영역에서 여전히 활용된다.

+++

## Ⅰ. 비선점형 스케줄링의 개요

### 1. 정의
- CPU가 특정 프로세스에게 할당되면, 그 프로세스가 완료되거나 대기 상태로 전환될 때까지 다른 프로세스가 CPU를 점유할 수 없는 스케줄링 정책이다.

### 2. 특징: '신사와 양보'
- **자율 반납**: 프로세스가 자신의 업무를 다 마치거나, 데이터가 필요해 잠시 쉴 때만 CPU를 내놓는다.
- **낮은 오버헤드**: 스케줄러가 개입하는 횟수가 적어 문맥 교환 비용이 최소화된다.
- **기아 현상 (Starvation) 위험**: 아주 긴 작업이 CPU를 잡으면 짧은 작업들이 무한정 기다려야 한다.

### 3. 💡 비유: '화장실 줄 서기'
- 비선점형 스케줄링은 **'한 번 화장실에 들어간 사람이 볼일을 다 보고 나올 때까지 아무도 문을 열 수 없는 것'**과 같다.
- 밖에서 아무리 급하다고 문을 두드려도(인터럽트), 안의 사람이 스스로 문을 열고 나오기 전까지는 꼼짝없이 기다려야 한다.

+++

## Ⅱ. 주요 비선점형 알고리즘 (Deep Dive)

### 1. FCFS (First-Come, First-Served)
- 가장 먼저 도착한 프로세스를 먼저 처리한다. 단순하지만 앞의 긴 작업 때문에 뒤의 짧은 작업들이 늦어지는 **호위 효과 (Convoy Effect)**가 발생할 수 있다.

### 2. SJF (Shortest Job First) - 비선점 버전
- 실행 시간이 가장 짧은 작업을 먼저 선택한다. 평균 대기 시간을 줄이는 데 효과적이지만, 작업 시간을 미리 알기 어렵다는 단점이 있다.

### 3. HRN (Highest Response Ratio Next)
- SJF의 단점(긴 작업의 무한 대기)을 보완하기 위해 대기 시간과 처리 시간을 고려하여 우선순위를 매긴다.
- **우선순위 = (대기 시간 + 서비스 시간) / 서비스 시간**

### 4. ASCII 구조 다이어그램
```text
    [ Ready Queue ]             [ CPU ]                 [ Execution Flow ]
    ┌───────────┐               ┌───────────┐           P1 (Start)
    │ P2 (Short)│               │           │           ■■■■■■■■■■■■■■■ (P1 일함)
    ├───────────┤               │    P1     │           ■■■■■■■■■■■■■■■
    │ P3 (Med)  │               │ (Running) │           P1 (Finish/Yield)
    └─────┬─────┘               └─────┬─────┘           P2 (Next Start)
          │                           │                 ■■ (P2 일함)
          └────(P1 종료 후 선택 가능)────┘
```

- **📢 섹션 요약 비유**: 먼저 온 손님에게 요리를 다 대접한 후에야 다음 손님을 받는 1인 셰프 식당과 같습니다. 정성은 가득하지만(오버헤드 적음), 뒤의 손님은 배가 많이 고플 수 있습니다.

## Ⅲ. 비선점형 vs 선점형 비교

| 비교 항목 | 비선점형 (Non-preemptive) | 선점형 (Preemptive) |
|:---|:---|:---|
| **제어권** | 프로세스가 보유 (자율) | **운영체제 커널이 보유 (강제)** |
| **문맥 교환** | 적음 (효율적) | **많음 (오버헤드 발생)** |
| **응답성** | 낮음 (대화형 부적합) | **높음 (시분할 시스템 적합)** |
| **대표 방식** | FCFS, SJF, HRN | Round Robin, SRTF, MLFQ |

- **📢 섹션 요약 비유**: 마이크를 잡은 사람이 말을 다 끝내야 넘겨주는 토론회(비선점)와, 사회자가 1분마다 발언권을 강제로 뺏는 토론회(선점)의 차이입니다.

+++

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 일괄 처리(Batch) 서버 최적화
- **상황**: 매일 밤 전날의 매출 데이터를 정산하는 배치 서버.
- **판단**: 사용자와의 인터랙션이 전혀 없고, 정해진 순서대로 대량의 연산을 끝내는 것이 목적이다. 이때는 잦은 문맥 교환이 오히려 전체 작업 시간을 늦춘다. 따라서 **비선점형 FCFS**나 **SJF** 방식을 적용하여 CPU가 온전히 연산에만 집중하게 함으로써 총 처리량(Throughput)을 극대화한다.

### 2. 기술사적 인사이트: 협력적 멀티태스킹 (Cooperative Multitasking)
- 초기 윈도우(3.1)나 맥 OS는 비선점형 방식의 일종인 '협력적 멀티태스킹'을 사용했다. 기술사라면 이 방식의 가장 큰 위험이 특정 앱이 무한 루프에 빠졌을 때 **시스템 전체가 얼어버리는(Hang)** 현상임을 인지하고, 이를 해결하기 위해 현대 OS가 하드웨어 타이머와 결합한 선점형으로 진화했음을 논리적으로 설명해야 한다.

- **📢 섹션 요약 비유**: 팀원들이 알아서 잘 협조하겠거니 믿고 일을 맡기는 것(비선점)은 좋지만, 한 명이라도 딴짓을 하면 프로젝트 전체가 망가지는 리스크가 있습니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **시스템 처리량 향상**: 불필요한 커널 개입 최소화로 순수 연산 시간 증대.
- **예측 가능성**: 작업 순서가 명확하여 시스템 동작 시나리오 단순화.

### 2. 미래 전망
비선점형 스케줄링은 범용 데스크톱에서는 자취를 감췄으나, 초저전력을 지향하는 **임베디드 센서 노드**나 오버헤드를 극단적으로 줄여야 하는 **슈퍼컴퓨팅(HPC)의 특정 가속 노드**에서 다시금 조명받고 있다. 하드웨어가 극도로 단순화되거나, 반대로 극도로 고도화되는 양극단에서 비선점형의 '오버헤드 제로' 철학은 여전히 유효한 설계 옵션이다.

- **📢 섹션 요약 비유**: 화려한 규칙은 없지만, 묵묵하게 가장 많은 일을 해내는 운영체제의 '전통적인 일꾼'입니다.
