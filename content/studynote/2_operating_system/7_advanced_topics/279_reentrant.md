# 재진입성 (Reentrancy)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 재진입성 (Reentrancy)은 어떤 서브루틴(함수)이 한 스레드에 의해 실행되던 도중 인터럽트에 의해 멈추고, **다른 스레드가 해당 서브루틴을 동시에(다시 진입하여) 호출하더라도 각자의 실행 결과가 원래 의도대로 완벽하게 보장되는 성질**이다.
> 2. **가치**: 락(Lock)을 걸어 스레드를 통제하는 '스레드 세이프(Thread-Safe)'의 수동적 방어를 넘어, 함수 내부에 **전역 변수(Global State)를 완전히 배제하여 락 자체를 필요 없게 만드는 가장 능동적이고 궁극적인 동시성 최적화 설계**다.
> 3. **융합**: 운영체제의 하드웨어 인터럽트 핸들러(ISR)나 유닉스 시그널(Signal) 처리 같은 예측 불가능한 비동기 환경에서 시스템의 패닉을 막기 위한 절대적 규범이며, 현대의 **함수형 프로그래밍(Functional Programming)** 철학과 완벽히 맞닿아 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: "다시(Re) 진입(Enter)할 수 있다"는 뜻이다. 함수 A가 실행되다 중간에 끊겼을 때, 다른 놈이 와서 함수 A를 처음부터 다시 부르더라도 이전에 실행되던 A의 상태가 오염되지 않고, 둘 다 무사히 제 갈 길을 가는 상태 공간(State Space)의 고립을 말한다.
- **필요성**: 싱글 코어 시스템에서조차 하드웨어 인터럽트는 예고 없이 찾아온다. 내가 `printf()`로 화면에 글씨를 쓰다가 타이머 인터럽트로 쫓겨났는데, 새로 들어온 프로세스가 또 `printf()`를 호출한다면? 만약 `printf` 내부에서 전역 버퍼(Global Buffer)를 쓰고 있었다면, 첫 번째 프로세스의 글씨와 두 번째 프로세스의 글씨가 끔찍하게 섞여서 출력된다. 이를 막으려면 **인터럽트가 걸려도 상태가 꼬이지 않는 '순수한' 함수**가 필요했다.
- **💡 비유**: 길거리에 있는 **'공중 전화박스'**와 같다. 스레드 세이프(Thread-Safe)는 전화박스 문을 닫아버려(Lock) 남이 못 들어오게 막는 것이라면, 재진입성(Reentrancy)은 아예 **'스마트폰(Local Variable)'**을 개인별로 쥐여주는 것이다. 스마트폰은 100명이 동시에 통화해도 남의 통화 내용이 섞이지 않는다.
- **등장 배경**: 초기 유닉스(Unix) C 라이브러리 함수들(`strtok`, `rand`, `ctime`)은 편의성을 위해 내부적으로 `static` 변수를 남발했다. 그러나 시분할과 멀티스레딩이 일상화되면서 이 함수들이 시스템 전체를 붕괴시키는 주범으로 지목되었다. 이에 POSIX 표준은 기존 함수 뒤에 `_r`을 붙인 재진입 가능 버전(`strtok_r`, `rand_r`)을 강제로 새로 만들어 배포하며 뼈아픈 아키텍처 개혁을 단행했다.

```text
  [Thread-Safe (Lock 기반) vs Reentrancy (재진입성) 의 동작 원리 비교]

  [ 1. 스레드 세이프 (Thread-Safe 하지만 Reentrant 하지 않은 함수) ]
  int global_sum = 0;
  void add() {
      lock();            ◀─ (1. 스레드 A 진입 후 락 획득)
      global_sum++;      ◀─ (2. 🚨 이 순간 인터럽트 발생! 스레드 B가 난입함)
      unlock();
  }
  ▶ 스레드 B의 동작: add() 호출 ─▶ 락이 잠겨있어 Sleep! (데드락 위험 및 병목 발생)

  [ 2. 재진입성 (Reentrant 한 완벽한 함수) ]
  void add_reentrant(int local_sum) {
      local_sum++;       ◀─ (오직 스택에 있는 자기 지역 변수만 건드림)
      return local_sum;
  }
  ▶ 스레드 B의 동작: add_reentrant() 호출 ─▶ 자신의 스택에 1 추가. 
  ▶ 결과: A와 B가 락(Lock) 없이 100% 동시에 평화롭게 실행됨! 🚀
```
**[다이어그램 해설]** 재진입 가능한 코드는 "공유(Shared)의 개념 자체를 삭제"한 코드다. 글로벌 상태를 버리고 오직 파라미터(Parameter)로 값을 받아서 지역 변수(Local Variable)로만 계산한다. 지역 변수는 각 스레드의 독립된 메모리인 **스택(Stack)**에 할당되므로, 스레드가 1만 개 생성되어도 절대 서로 부딪히지 않는 물리적 완전 격리가 달성된다.

- **📢 섹션 요약 비유**: 주방 공용 도마(전역 변수)를 쓰면 앞사람이 쓰다 나갔을 때 도마를 씻을 때까지 기다려야(Lock) 합니다. 재진입성은 요리사가 아예 집에서 자기 개인 도마(스택 변수)를 들고 와서 허공에서 요리하는 것입니다. 주방장이 100명이어도 서로 절대 부딪히지 않습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 재진입성 코드를 짜기 위한 4가지 절대 법칙

코드에 아래 4가지 중 단 하나라도 포함되어 있다면, 그 코드는 절대 재진입할 수 없다.

1. **전역(Global) 또는 정적(Static) 변수를 수정하지 마라.**
   - 읽기 전용(Read-only, Constant)으로 쓰는 것은 괜찮다. 하지만 값을 1이라도 덮어쓰는 순간 그 함수는 오염된다.
2. **상태를 저장하지 마라 (Stateless).**
   - 함수 호출과 다음 함수 호출 사이에 기억해야 할 값(예: `strtok`의 이전 파싱 위치)이 있다면, 함수 내부에 숨기지 말고 호출자가 포인터로 명시적으로 넘겨주게(Pass-by-reference) 만들어야 한다.
3. **자물쇠(Lock)를 쓰지 마라.**
   - 싱글 코어에서 인터럽트 핸들러가 락이 걸린 함수를 재호출하면 자기 자신을 기다리게 되는 영원한 교착 상태(Deadlock)에 빠진다.
4. **다른 재진입 불가능한 함수를 호출하지 마라.**
   - 내 코드를 아무리 깨끗하게 짰어도, 내부에서 `malloc()`이나 `printf()`를 호출하면 끝이다. 그 함수들 내부에 글로벌 락이나 전역 변수가 숨어있기 때문이다. (전염성).

### 유명한 재진입 불가능 함수의 대명사: `strtok` vs `strtok_r`
C언어에서 문자열을 자르는 함수다.

```c
  // ❌ 낡은 C 표준 (재진입 불가, 스레드 지옥의 원흉)
  char *strtok(char *str, const char *delim);
  // 내부 동작: 다음에 자를 문자열의 위치를 기억하기 위해
  // 눈에 안 보이는 'static char *last_pos' 라는 전역 변수를 몰래 쓴다.

  // ✅ POSIX 스레드 표준 (재진입 가능)
  char *strtok_r(char *str, const char *delim, char **saveptr);
  // 내부 동작: 전역 변수를 없앴다. 대신 개발자가 직접 'saveptr'이라는 
  // 지역 변수를 던져주면, 함수는 이 스레드 개인 변수에 위치를 저장한다.
```
**[아키텍처적 교훈]** 상태(State)의 소유권을 '함수(Kernel/Library)'가 가지면 안 된다. 그 상태를 '호출자(User Thread)'의 스택으로 밀어내어 100% 호출자가 책임지게 만들어야만 진정한 병렬 처리의 자유를 얻을 수 있다.

- **📢 섹션 요약 비유**: `strtok`는 미용실 주인이 "고객님 저번에 앞머리 길게 잘랐죠? 제가 기억(static 변수)하고 있어요!" 하는 겁니다. 주인이 치매에 걸리거나 다른 손님과 헷갈리면 머리를 망칩니다. `strtok_r`은 손님이 직접 "저번 머리 사진(saveptr)"을 들고 와서 보여주는 겁니다. 1,000명의 손님이 와도 절대 헷갈리지 않습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 재진입성(Reentrancy)과 스레드 세이프(Thread-Safety)의 부분집합 벤다이어그램

면접관이 가장 좋아하는 운영체제 개념 비교다.

| 개념 | 의미 | 구현 방식 (동기화) | 성능 |
|:---|:---|:---|:---|
| **스레드 세이프 (Thread-Safe)** | 100개의 스레드가 동시에 찔러도 에러 안 남. | **Mutex, Semaphore 떡칠 (방어적)** | 느림 (락 병목 발생) |
| **재진입 가능 (Reentrant)** | 스레드뿐 아니라 **인터럽트 도중 재진입해도** 안전함. | **전역 변수 0%, 락 0% (구조적 완벽함)** | **극강의 속도 (Zero Lock)** |

**[핵심 명제]**
1. **재진입 가능하면 ─▶ 무조건 스레드 세이프하다 (True).**
   - 락도 없고 전역 변수도 없으니, 1억 개의 스레드가 들어와도 각자 자기 스택에서만 놀기 때문에 충돌 날 건더기조차 없다.
2. **스레드 세이프하다고 ─▶ 무조건 재진입 가능한 건 아니다 (False).**
   - 락(Mutex)을 걸어서 스레드 세이프를 달성한 함수를 생각해 보자. 인터럽트 핸들러가 이 함수를 호출하다가 중단되고, 다른 놈이 이 함수를 부르면? 락이 잠겨있어서 데드락에 빠진다. 따라서 락을 쓴 함수는 절대 재진입 가능(Reentrant)하다고 부르지 않는다.

### 함수형 프로그래밍(Functional Programming)의 본질
"왜 요즘 백엔드에서 함수형 프로그래밍(Scala, Clojure)이 대세인가?"
함수형 프로그래밍의 1원칙인 **'순수 함수(Pure Function)'**와 **'불변성(Immutability)'**이 바로 운영체제가 그토록 부르짖던 **재진입성(Reentrancy)의 현대적 구현체**이기 때문이다. 외부에 부수 효과(Side Effect)를 일으키지 않고, 입력값에 의해서만 출력값이 결정되는 순수 함수는 태생적으로 100% 재진입이 가능하여 멀티코어의 축복을 그대로 흡수한다.

- **📢 섹션 요약 비유**: 스레드 세이프는 폭탄(전역 변수) 주위에 두꺼운 방폭 유리(락)를 덮어둔 상태입니다. 터져도 남은 안 다치지만 폭탄을 처리하느라 속도가 느립니다. 재진입성은 아예 방 안에 폭탄이라는 물건 자체를 반입 금지시킨 무결점 구역입니다. 누구나 안전하게 100km/h로 뛸 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오
1. **리눅스 커널의 인터럽트 핸들러(ISR) 개발**: 임베디드나 커널 드라이버를 짜는 하드코어 시스템 개발자에게 내려지는 1원칙이다.
   - "인터럽트 핸들러 내부에서는 절대로 `malloc()`, `printf()`, `mutex_lock()`을 쓰지 마라."
   - **사망 선고**: 저 함수들은 재진입이 불가능한 함수들이다. ISR 도중에 저 함수를 불렀는데, 또 인터럽트가 터져서 ISR이 재진입되면 커널은 락이 꼬여 그 즉시 **'Kernel Panic'**을 뿜고 컴퓨터가 재부팅된다. 커널 모드에서는 무조건 100% 순수한 재진입 가능 함수만 사용해야 한다.
2. **Spring Boot (Java) 컨트롤러의 인스턴스 변수 재앙**:
   - 주니어 백엔드 개발자가 사용자 인증 정보를 저장한답시고 Controller 클래스에 `private User user;` 라는 전역 필드를 만들었다.
   - **아키텍트 분석**: Spring의 Controller는 기본적으로 **'싱글톤(Singleton)'**으로 떠서 수백 개의 스레드에 의해 동시에 호출된다. 여기에 멤버 변수를 박는 순간 이 컨트롤러는 '재진입 불가(Non-reentrant)' 상태가 된다.
   - **결과**: 유저 A가 로그인했는데 유저 B의 개인정보가 화면에 뜨는 미친 보안 사고(Cross-talk)가 터진다.
   - **실무 조치**: 백엔드 비즈니스 로직(Controller, Service)은 무조건 상태를 갖지 않는 무상태(Stateless)로 짜야 하며, 모든 데이터는 함수의 Parameter와 Return으로만 전달하는 '재진입성 코딩 컨벤션'을 준수해야만 클라우드 스케일 아웃이 가능해진다.

```text
  ┌────────────────────────────────────────────────────────┐
  │     고성능 서버의 상태(State) 관리 및 아키텍처 리팩토링 트리      │
  ├────────────────────────────────────────────────────────┤
  │                                                        │
  │   [요구사항: 초당 1만 건의 트래픽을 처리하는 글로벌 캐시 모듈 개발]│
  │                                                        │
  │   [ 1. 낡은 C/C++ 레거시 스타일 (Thread-Unsafe) ]       │
  │     - `static int counter = 0;` (전역 변수 사용)         │
  │     ▶ 판정: 쓰레기 데이터 발생. 운영 불가.                   │
  │                                                        │
  │   [ 2. 락 기반 땜질 (Thread-Safe, Non-Reentrant) ]      │
  │     - `mutex.lock(); counter++; mutex.unlock();`       │
  │     ▶ 판정: 에러는 안 나지만 TPS가 100으로 폭락(병목 발생).    │
  │                                                        │
  │   [ 3. 궁극의 아키텍처 (Reentrant / Stateless) ]        │
  │     - 상태를 함수 밖으로 밀어냄.                             │
  │     - `int add(int local_counter) { return local_counter + 1; }`│
  │     - 또는 불변 객체(Immutable)를 던지고 새로 복사해서 리턴함. │
  │     ▶ 판정: ✅ 락 0%. 동시성 버그 0%. 초당 100만 건 무정지 통과!│
  └────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** "어떻게 락을 잘 걸까?"를 고민하는 것은 하수다. 시니어 아키텍트는 "어떻게 하면 락을 안 걸 수 있도록(재진입 가능하게) 데이터의 흐름과 아키텍처 자체를 뜯어고칠까?"를 고민한다. 메모리를 조금 더 쓰더라도 상태(State)를 분리하는 것이 현대 클라우드 네이티브(Cloud Native) 개발의 절대 진리다.

- **📢 섹션 요약 비유**: 수영장에 공용 수건(전역 변수)을 1개 두고 락커를 잠그면(Mutex), 뒤에 줄 선 사람들이 감기에 걸립니다(병목). 가장 좋은 수영장(재진입성 아키텍처)은 입장할 때 각자 자기 개인 수건(지역 변수)을 1장씩 나눠주어, 남의 눈치를 볼 필요 없이 언제든 자유롭게 씻고 나가게 하는 곳입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
모든 핵심 비즈니스 로직과 라이브러리를 재진입 가능(Reentrant)하게 설계하면, 128코어 이상의 거대 SMP 환경에서 락(Lock) 경합으로 인한 CPU 캐시 파괴(False Sharing)나 컨텍스트 스위칭 오버헤드가 제로화되어, **물리적인 하드웨어 스펙에 100% 정비례하는 선형적 성능 확장성(Linear Scalability)**을 획득하게 된다.

### 결론 및 미래 전망
재진입성(Reentrancy)은 1970년대 열악한 메모리 환경에서 코드를 공유하기 위해 짠 눈물겨운 테크닉이었으나, 현대 멀티코어 시대에 이르러 **"동시성 프로그래밍의 궁극적 도달점"**으로 격상되었다.
미래의 패러다임은 개발자가 이 재진입성을 수동으로 지키는 것을 넘어, 프로그래밍 언어 자체가 이를 강제하는 방향으로 흐르고 있다. 공유 메모리 자체를 죄악시하는 Rust의 소유권(Ownership) 모델과, 액터 모델(Erlang, Akka)의 메시지 패싱 구조는 "모든 코드는 본질적으로 재진입 가능해야 한다"는 철학을 언어 스펙으로 녹여낸 것이다. 미래의 클라우드 서버리스(Serverless) 환경에서 상태(State)는 분산 DB에 가두고, 연산 유닛(Function)은 오직 순수한 재진입성 코드로만 작성되는 궁극의 무상태(Stateless) 아키텍처가 세상을 지배할 것이다.

- **📢 섹션 요약 비유**: 재진입성은 오염되지 않는 맑은 물방울(함수)과 같습니다. 물방울 1만 개가 한데 모여 웅덩이가 되어도, 다시 튀어 오를 땐 남의 색깔에 물들지 않고 투명한 본질을 유지합니다. 이것이 바로 동시성 지옥을 뚫고 나온 인류 최고의 소프트웨어 엔지니어링 미학입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **스레드 세이프 (Thread-Safe)** | 재진입성이 포함되는 더 넓은 개념. 재진입 코드는 당연히 스레드 세이프하지만, 그 반대는 성립하지 않는다. |
| **뮤텍스 (Mutex)** | 코드를 스레드 세이프하게 만들어주지만, 인터럽트 환경에서는 데드락을 낳아 재진입성을 파괴하는 원흉이다. |
| **문맥 교환 (Context Switch)** | 함수 실행 중간에 이 녀석이 치고 들어와서 딴 놈을 실행시킬 때, 변수가 꼬이지 않아야 재진입성이 성립한다. |
| **함수형 프로그래밍 (FP)** | 전역 변수(상태 변경)를 철저히 배제하는 패러다임으로, 이 언어로 짜인 코드는 99% 재진입이 가능하다. |
| **Thread-Local Storage (TLS)** | 재진입성을 완벽히 못 지킬 때(어쩔 수 없이 전역 변수를 쓸 때), 꼼수로 스레드마다 사물함을 따로 파주는 기법이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 내가 일기장을 쓰고 있는데 엄마가 심부름을 시켰어요. 심부름을 다녀왔더니 동생이 내 일기장(전역 변수)에 낙서를 해놨어요! (재진입 불가능)
2. 하지만 내가 일기를 쓸 때마다 학교에서 새로 나눠주는 빈 공책(지역 변수, 스택)에 쓴다면 어떨까요?
3. 엄마 심부름을 다녀와서 동생이 자기 공책에 일기를 써도, 내 빈 공책에는 아무 낙서가 없기 때문에 이어서 깨끗하게 일기를 완성할 수 있어요. 이걸 **재진입성**이라고 부른답니다!
