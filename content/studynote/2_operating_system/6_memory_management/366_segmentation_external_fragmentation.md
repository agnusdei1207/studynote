# 세그멘테이션과 외부 단편화

## 핵심 인사이트 (3줄 요약)
> 1. **인과관계**: 세그멘테이션은 프로세스를 논리적 단위인 '가변 크기' 세그먼트로 나누어 할당하므로, 메모리 할당과 해제가 반복되면 물리 메모리 사이에 작은 빈 공간들이 흩어지는 '외부 단편화'가 필연적으로 발생한다.
> 2. **문제점**: 전체 자유 공간의 합은 충분하더라도, 정작 필요한 세그먼트를 수용할 만큼의 '연속된' 공간이 없어서 새로운 프로세스나 세그먼트를 로드하지 못하는 자원 낭비 상태에 빠진다.
> 3. **해결 전략**: 이를 해결하기 위해 메모리 압축(Compaction)으로 빈 공간을 모으거나, 물리적으로는 고정 크기(Paging)로 관리하되 논리적으로만 세그먼트를 유지하는 하이브리드 방식을 채택한다.

+++

## Ⅰ. 외부 단편화 (External Fragmentation)의 발생 원인

### 1. 가변 크기 할당 (Variable-sized Allocation)
- 세그멘테이션은 코드 10KB, 데이터 50KB 등 각기 다른 크기의 메모리 덩어리를 요구한다.
- 운영체제는 물리 메모리의 빈 공간(Hole) 중에서 적절한 위치를 찾아 할당한다.

### 2. 동적 메모리 할당의 결과
- 시간이 흐르며 프로세스들이 종료되고 새로운 프로세스들이 들어오면, 메모리는 '할당된 영역'과 '빈 영역'이 잘게 쪼개진 체스판 같은 모양이 된다.

- **📢 섹션 요약 비유**: "크기가 제각각인 책들을 책꽂이에 꽂았다 뺐다 반복하다 보면, 책꽂이 여기저기에 책 한 권도 안 들어가는 좁은 틈새들만 남게 되는 것과 같습니다."

+++

## Ⅱ. 외부 단편화 현상의 시각화 및 ASCII 다이어그램

### 1. 단편화 진행 시나리오
1. 프로세스 A, B, C가 연속적으로 할당됨.
2. 프로세스 B가 종료되어 빈 공간(Hole)이 생김.
3. 새로운 프로세스 D가 들어오려는데, B가 나간 자리보다 커서 들어가지 못함.

### 2. ASCII: 세그멘테이션 메모리 단편화
```text
[ Physical Memory State ]
+-----------------------+ 0KB
| Segment A-1 (Code)    |
+-----------------------+ 20KB
| ****** Hole 1 ******  | (10KB - 이전 세그먼트 해제됨)
+-----------------------+ 30KB
| Segment C-2 (Data)    |
+-----------------------+ 80KB
| ****** Hole 2 ******  | (15KB - 자투리 공간)
+-----------------------+ 95KB
| Segment A-2 (Stack)   |
+-----------------------+ 120KB

[ New Request ]
- Need: Segment D (20KB)
- Status: Total Free = 10KB + 15KB = 25KB
- Result: FAILED (No contiguous 20KB space!)
```

- **📢 섹션 요약 비유**: "주차장에 경차들이 빠져나간 자리가 여러 개 있어도, 그 자리들이 떨어져 있으면 대형 버스 한 대를 세울 수 없는 상황입니다."

+++

## Ⅲ. 세그멘테이션 vs 페이징: 단편화 비교

### 1. 세그멘테이션 (Segmentation)
- **외부 단편화**: 심각함. 가변 크기 할당으로 인해 발생.
- **내부 단편화**: 발생하지 않음. 필요한 만큼만 딱 맞춰서 할당하기 때문.

### 2. 페이징 (Paging)
- **외부 단편화**: 발생하지 않음. 물리 메모리를 고정 크기(Frame)로 나누어 빈 프레임 어디든 들어갈 수 있기 때문.
- **내부 단편화**: 발생함. 페이지 크기의 배수로 할당하므로 마지막 페이지에 남는 공간이 생김.

- **📢 섹션 요약 비유**: "맞춤복(세그먼트)은 몸에 딱 맞지만 남는 천(외부 공간)을 버리게 되고, 기성복(페이지)은 천을 안 버리지만 옷이 몸보다 조금 클(내부 공간) 수 있는 것과 같습니다."

+++

## Ⅳ. 외부 단편화 해결을 위한 기술적 대응 (Strategy)

### 1. 메모리 압축 (Compaction)
- 실행 중인 세그먼트들을 메모리의 한쪽 끝으로 밀어붙여, 흩어진 빈 공간들을 하나의 거대한 연속된 공간으로 합치는 작업이다.
- **단점**: 압축하는 동안 시스템이 멈춰야 하며, 엄청난 메모리 복사 비용이 발생한다.

### 2. 배치 전략 (Placement Strategy)
- **First-fit, Best-fit, Worst-fit** 등의 알고리즘을 사용하여 단편화를 최소화하려 노력하지만, 근본적인 해결책은 되지 못한다.

### 3. 세그멘테이션 기반 페이징 (Paged Segmentation)
- 현대적인 해결책으로, 논리적으로는 세그먼트로 나누되 물리적으로는 다시 페이지로 쪼개어 할당함으로써 외부 단편화를 원천 차단한다.

- **📢 섹션 요약 비유**: "흩어진 짐들을 한곳으로 몰아넣어 공간을 만드는 '방 정리(Compaction)'를 하거나, 아예 모든 물건을 똑같은 규격 상자(Paging)에 담아 보관하는 전략입니다."

+++

## Ⅴ. 결론 및 요약

### 1. 핵심 요약
- 세그멘테이션의 외부 단편화는 **'논리적 가변성'과 '물리적 연속성'이 충돌**하여 발생하는 현상으로, 메모리 활용률을 떨어뜨리는 주요 원인이다.

### 2. 시스템적 의의
- 이 문제로 인해 순수 세그멘테이션은 도태되었으며, 오늘날의 메모리 관리는 페이징의 효율성을 기반으로 세그멘테이션의 논리를 수용하는 방향으로 진화했다.

- **📢 섹션 요약 비유**: "결국 '자유로운 크기'의 편리함을 누리기 위해서는 '빈 공간 관리'라는 혹독한 대가를 치러야 함을 보여주는 사례입니다."
