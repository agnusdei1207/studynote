# 가비지 컬렉션 기초 (Garbage Collection)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 가비지 컬렉션 (Garbage Collection, GC)은 프로그램이 더 이상 참조하지 않는 메모리(가비지)를 자동으로 탐지하고 회수하여, 개발자의 수동 `free()`/`delete` 없이 메모리 누수와 댕글링 포인터를 방지하는 자동 메모리 관리 시스템이다.
> 2. **가치**: Java·Python·Go·C# 같은 현대 언어에서 GC는 개발 생산성을 크게 높이고 메모리 관련 버그를 원천 차단하지만, STW (Stop-The-World) 일시 정지, 예측 불가한 GC 타이밍이 지연 민감 시스템(실시간 트레이딩, 게임)에서 도전이 된다.
> 3. **융합**: 참조 카운팅 (Reference Counting), Mark-and-Sweep, 세대별 GC (Generational GC), 증분 GC (Incremental GC), ZGC 등의 알고리즘이 STW 시간을 ms→μs 수준으로 줄이는 방향으로 진화하며, 현대 JVM과 Go 런타임이 서로 다른 전략으로 이 목표를 추구한다.

---

## Ⅰ. 개요 및 필요성

C/C++에서 `malloc`으로 할당한 메모리는 반드시 `free`로 해제해야 한다. 해제를 잊으면 **메모리 누수 (Memory Leak)**, 이미 해제된 메모리를 참조하면 **댕글링 포인터 (Dangling Pointer)**, 같은 메모리를 두 번 해제하면 **이중 해제 (Double Free)** — 이 모든 버그가 GC로 자동 해결된다.

**💡 비유**: GC는 식당 청소부 — 손님(프로세스)이 밥을 먹고 자리를 뜨면 자동으로 식기(메모리)를 치워주어 새 손님이 앉을 수 있게 한다. C/C++는 손님이 직접 식기를 설거지해야 하는 셀프 식당.

```text
┌──────────────────────────────────────────────────────────────┐
│         가비지 탐지 기준 — 도달 가능성 (Reachability)      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  GC 루트 (Root Set):                                        │
│  ● 스택 변수 (지역변수, 매개변수)                           │
│  ● 전역 변수                                                 │
│  ● 레지스터 내 참조                                         │
│                                                              │
│  GC 루트 ──참조──▶ 객체 A ──참조──▶ 객체 B  (도달 가능)   │
│  GC 루트 ✗──────▶ 객체 C                    (도달 불가능)  │
│                                                              │
│  도달 불가능 = 가비지 → 회수 대상!                          │
│                                                              │
│  ※ 참조 카운팅의 문제점:                                   │
│  A ──▶ B, B ──▶ A (순환 참조) → 카운트가 0이 안 됨!       │
│  → Mark-and-Sweep으로 해결 필요                             │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: GC는 아무도 길을 안내하지 않는 미로 속 고아 방 — 어느 입구에서도 접근할 수 없는 방은 더 이상 필요 없는 가비지입니다.

## Ⅱ. 아키텍처 및 핵심 원리

### GC 알고리즘 비교

| 알고리즘 | 동작 원리 | 장점 | 단점 |
|:---|:---|:---|:---|
| 참조 카운팅 | 참조 수를 세어 0이 되면 즉시 해제 | 즉각적 해제 | 순환 참조 처리 불가 |
| Mark-and-Sweep | 도달가능 객체 마킹 후 나머지 회수 | 순환 참조 해결 | STW 발생 |
| 복사 GC | 도달가능 객체를 새 영역으로 복사 | 단편화 없음 | 메모리 2배 필요 |
| 세대별 GC | 나이(세대)로 나눠 단기 객체 빠르게 회수 | GC 빈도·비용 균형 | 구현 복잡 |
| ZGC / Shenandoah | 동시 마킹·컴팩션 (STW ms↓) | 매우 낮은 레이턴시 | CPU 오버헤드 증가 |

### 세대별 GC (Generational GC) — JVM 기반

```text
┌──────────────────────────────────────────────────────────────┐
│         JVM 세대별 GC 구조                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [Young Generation]           [Old Generation]              │
│   Eden | S0 | S1                   Old                       │
│   ↑                                  ↑                      │
│   새 객체 할당               오래 살아남은 객체              │
│                                                              │
│  Minor GC (Young GC):                                       │
│  ● 빈도 높음 (수십 ms마다)                                  │
│  ● STW 수 ms 미만                                           │
│  ● Eden에서 살아남으면 S0/S1으로 이동                       │
│  ● S0/S1에서 N회 이상 살아남으면 Old로 승격               │
│                                                              │
│  Major GC (Full GC):                                         │
│  ● 빈도 낮음 (수 분에 한 번)                                │
│  ● STW 수백 ms~수 초 (이것이 문제!)                        │
│  ● Old Generation 회수                                       │
│                                                              │
│  가설: "대부분의 객체는 젊어서 죽는다" (더 빠른 Minor GC)  │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: 세대별 GC는 학교 청소 시스템 — 매일 교실(Young) 청소하고, 학기말(Major GC)에 창고(Old) 정리합니다. 매일 청소가 학기말 대청소보다 훨씬 빠르고 수월해요.

## Ⅲ. 융합 비교 및 다각도 분석

### STW (Stop-The-World) 최소화 전략

| 기법 | STW | 설명 |
|:---|:---|:---|
| Serial GC | 수백ms | 단일 스레드, 소형 앱용 |
| Parallel GC | 수십ms | 멀티스레드 마킹·회수 |
| G1 GC (Java 9+ 기본) | <200ms 목표 | 힙을 Region으로 나눠 증분 회수 |
| ZGC (Java 15+) | <10ms 목표 | 동시 마킹, 컬러 포인터 |
| Go GC (concurrent) | <1ms | 쓰기 배리어 기반 동시 GC |

**📢 섹션 요약 비유**: GC 진화는 식당 청소 방식 발전 — 영업 종료 후 대청소(Serial)에서, 영업 중 청소원이 틈틈이 치우는(ZGC) 방식으로 손님(요청) 대기시간을 0에 가깝게 만듭니다.

## Ⅳ. 실무 적용 및 기술사적 판단

**실무 시나리오**:
1. **금융 저지연 시스템**: JVM 기반 HFT에서 Full GC STW 수백ms → 주문 누락. ZGC 또는 GC-free (오프힙 메모리) 아키텍처 채택.
2. **Go 마이크로서비스**: Go의 동시 GC로 STW <1ms → 서비스 레이턴시 안정성 확보에 기여.

**안티패턴**:
- **JVM 힙 과대 설정**: 64GB 힙 설정 시 Full GC STW가 수 분에 달할 수 있음. 힙 크기와 GC 레이턴시 균형 필요.
- **순환 참조 방치** (Python refcount): Python에서 순환 참조 방치 → 주기적 cycle collector 실행. `weakref` 사용으로 방지.

**📢 섹션 요약 비유**: JVM 힙 64GB는 창고를 너무 크게 만드는 것 — 대청소(Full GC) 시간이 너무 길어서 영업(서비스)을 한참 쉬어야 합니다.

## Ⅴ. 기대효과 및 결론

| 구분 | 수동 메모리 관리 | 가비지 컬렉션 |
|:---|:---|:---|
| 메모리 누수 | 개발자 책임 | 자동 방지 |
| 댕글링 포인터 | 발생 가능 | 방지 |
| GC STW 레이턴시 | 없음 | 알고리즘에 따라 다름 |
| 개발 생산성 | 낮음 (메모리 관리 부담) | 높음 |

GC는 현대 언어의 필수 런타임 기능이다. 알고리즘의 진화(세대별 → ZGC)로 STW를 극소화하여 지연 민감 서비스에서도 사용 가능해졌으며, Rust의 소유권 시스템처럼 GC 없이도 메모리 안전성을 달성하는 방향도 진화의 축이다.

## 👶 어린이를 위한 3줄 비유 설명
1. GC는 자동 청소 로봇 — 아무도 안 쓰는 물건(메모리)을 찾아서 자동으로 치워줘요. 직접 치울 필요 없어요!
2. 가끔 청소 로봇이 일할 때 방(앱)에서 잠깐 나가야 해요 — 이게 STW, 청소하는 동안 방을 비우는 거예요.
3. 최신 ZGC는 내가 방에 있어도 청소할 수 있어요 — 쉬는 시간(STW)이 10ms도 안 돼서 거의 느끼지 못해요!
