# 한계 레지스터 (Limit Register)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 한계 레지스터(Limit Register)는 프로세스가 사용할 수 있는 **논리적 주소 공간의 크기**를 저장하여, 허용된 범위를 벗어난 메모리 접근을 차단하는 레지스터이다.
> 2. **가치**: 프로세스 간의 메모리 침범을 물리적으로 방지함으로써 운영체제의 안정성을 보장하고, 고의적인 해킹이나 프로그래밍 오류(Buffer Overflow 등)로부터 시스템을 보호한다.
> 3. **융합**: 베이스 레지스터와 쌍을 이루어 '베이스 ~ 베이스 + 한계'라는 안전한 메모리 울타리를 형성하며, 하드웨어 기반 메모리 보호의 초석이 된다.

+++

## Ⅰ. 한계 레지스터 (Limit Register)의 개념 및 배경

### 1. 정의
- **한계 레지스터 (Limit Register)**란 CPU 내부에 존재하며, 현재 실행 중인 프로세스가 접근할 수 있는 논리 주소의 최대 범위(크기)를 나타내는 값을 보관하는 레지스터이다.

### 2. 등장 배경: '무질서한 침범 방지'
- 여러 프로그램이 동시에 메모리에 올라가는 환경에서, 한 프로그램의 버그가 다른 프로그램의 데이터를 지워버리거나 OS 커널을 파괴하는 일이 빈번했다. 이를 소프트웨어적으로 검사하면 너무 느리기 때문에, 하드웨어적으로 '선'을 긋는 장치가 필요해졌다.

### 3. 💡 비유: '개인 정원의 울타리'
- 베이스 레지스터가 내 집의 '대문 위치'라면, 한계 레지스터는 '내 마당의 가로세로 길이'와 같다. 내가 내 마당 안에서 노는 것은 자유지만, 담장(한계)을 넘어가 옆집 마당을 건드리려 하면 경보음(트랩)이 울리게 된다.

- **📢 섹션 요약 비유**: "허락된 구역 안에서만 움직이게 만드는 '보이지 않는 벽'입니다."

+++

## Ⅱ. 한계 레지스터의 작동 원리 (Deep Dive)

### 1. 유효성 검사 로직
- CPU가 논리 주소 $L$을 생성하면, 하드웨어(MMU)는 즉시 다음 조건을 검사한다.
- **조건: $0 \le L < \text{Limit Register Value}$**
- 만약 이 조건을 만족하지 못하면, 주소 변환을 중단하고 CPU에 **Addressing Error (Trap)**를 발생시킨다.

### 2. ASCII 메모리 보호 검사 프로세스
```text
    [ CPU Logical Addr: 1500 ]
               │
               ▼
    +---------------------------------------+
    |        MMU (Memory Protection)        |
    |                                       |
    |  [ Is 1500 < Limit(1000)? ] ──▶ NO!   | ──▶ [ SEGMENTATION FAULT ]
    |                                       |
    +---------------------------------------+
    
    * 결과: 프로세스는 즉시 종료(Terminated)되어 시스템 보호.
```

### 3. 기술적 상세 해설: 베이스 레지스터와의 결합
- 실제로 물리 메모리 상에서의 보호 범위는 $\text{Base}, \text{Base} + \text{Limit})$가 된다. 주소를 더하기(Relocation) 전에 크기를 먼저 비교(Protection)함으로써, 단 1바이트의 오차도 허용하지 않는 엄격한 보호를 실현한다.

- **📢 섹션 요약 비유**: "계산기를 두드리기 전, 먼저 줄자로 길이를 재서 규격 미달이면 바로 탈락시키는 검문소와 같습니다."

+++

## Ⅲ. 메모리 보호(Memory Protection) 관점의 분석

### 1. 보안 위협 방어
- **버퍼 오버플로우 (Buffer Overflow)**: 할당된 배열 범위를 넘어서 코드를 실행하려는 공격을 하드웨어 수준에서 차단한다.
- **잘못된 포인터 참조**: 널 포인터(Null Pointer)나 초기화되지 않은 포인터가 엉뚱한 곳을 가리킬 때 시스템 전체가 멈추는 것을 방지한다.

### 2. 베이스 vs 한계 레지스터 비교

| 레지스터 | 저장 값 | 목적 |
|:---|:---|:---|
| **베이스 (Base)** | 시작 주소 (물리적) | 위치 지정 및 재배치 |
| **한계 (Limit)** | 주소 공간 크기 (논리적) | 범위 제한 및 보호 |

- **📢 섹션 요약 비유**: "베이스가 '어디서 시작할지'를 알려주는 상냥한 가이드라면, 한계는 '어디까지 갈 수 있는지'를 감시하는 엄격한 보안 요원입니다."

+++

## Ⅳ. 실무 적용 및 시스템 장애 분석 (Strategy)

### 1. 세그먼테이션 폴트 (Segmentation Fault)
- 개발자가 가장 자주 마주치는 오류 중 하나인 '세그먼테이션 폴트'는 대부분 CPU가 발생시킨 논리 주소가 MMU의 한계 레지스터 범위를 벗어났을 때 발생한다.

### 2. 문맥 교환 (Context Switch) 시 관리
- 운영체제는 프로세스를 전환할 때마다 해당 프로세스가 원래 할당받았던 메모리 크기(Limit)를 한계 레지스터에 로드한다. 이를 통해 각 프로세스는 서로 다른 크기의 '안전 구역'을 보장받는다.

- **📢 섹션 요약 비유**: "전학생이 올 때마다 그 학생의 행동 반경(한계)을 교칙(레지스터)에 새로 적어넣어 사고를 미연에 방지하는 전략입니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 기대효과
- **시스템 안정성 극대화**: 한 프로세스의 오류가 다른 프로세스나 운영체제에 전파되지 않는 '결함 격리(Fault Isolation)'를 실현한다.
- **하드웨어 강제력**: 소프트웨어의 실수와 상관없이 하드웨어가 절대적인 규칙을 집행하여 신뢰성을 높인다.

### 2. 결론 및 향후 전망
- 한계 레지스터 방식은 현대의 페이징 기법에서 '페이지 테이블 항목 수'나 '세그먼트 길이' 등으로 구체화되어 여전히 살아 숨 쉬고 있다. 주소의 경계를 짓는 이 단순한 원리는 안전한 컴퓨팅을 위한 영원한 철칙이다.

- **📢 섹션 요약 비유**: "울타리가 있기에 우리는 각자의 정원에서 안심하고 꽃을 피울 수 있는 것입니다."
