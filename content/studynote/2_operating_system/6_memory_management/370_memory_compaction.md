# 파편화 관리 및 조각 모음 - 리눅스 메모리 컴팩션 (Memory Compaction)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메모리 컴팩션(Memory Compaction)은 물리 메모리 곳곳에 흩어져 있는 '사용 중인 페이지'들을 한곳으로 몰아넣어, 반대편에 거대한 '연속된 빈 공간'을 만들어내는 기술이다.
> 2. **가치**: 외부 단편화로 인해 큰 메모리 블록(Huge Pages 등) 할당이 불가능해지는 상황을 해결하며, 시스템을 재부팅하지 않고도 메모리 효율을 동적으로 최적화한다.
> 3. **작동 원리**: 두 개의 스캐너(Free 스캐너와 Migration 스캐너)가 양끝에서 서로를 향해 이동하며, 이동 가능한 페이지들을 빈 공간으로 옮기고 주소 매핑을 갱신하는 방식으로 진행된다.

+++

## Ⅰ. 메모리 컴팩션 (Memory Compaction)의 정의 및 목적

### 1. 정의
- **메모리 컴팩션 (Memory Compaction)**은 물리 메모리의 파편화(Fragmentation)를 해결하기 위한 '조각 모음' 과정이다. 리눅스 커널 2.6.35부터 도입되어 메모리 관리의 유연성을 획기적으로 높였다.

### 2. 목적: 거대 페이지(Huge Pages) 지원
- 현대 시스템은 TLB 효율을 위해 2MB나 1GB 크기의 거대 페이지를 요구하는 경우가 많다.
- 하지만 시스템이 오래 가동되면 외부 단편화로 인해 전체 빈 공간은 많아도 '연속된 2MB'를 찾기 힘들어지는데, 컴팩션이 이 문제를 해결한다.

### 3. 이동 가능한 페이지 (Migratable Pages)
- 모든 페이지를 옮길 수 있는 것은 아니다. 사용자 프로세스의 익명 메모리(Anonymous Memory)나 페이지 캐시처럼 주소 변환 정보를 수정할 수 있는 페이지들이 주요 대상이다.

- **📢 섹션 요약 비유**: "주차장에 차들이 띄엄띄엄 서 있어서 대형 버스가 못 들어올 때, 서 있는 차들을 한쪽으로 촘촘히 다시 주차시켜 넓은 자리를 만드는 것과 같습니다."

+++

## Ⅱ. 리눅스 컴팩션 메커니즘 및 ASCII 다이어그램

### 1. 투-스캐너 알고리즘 (Two-scanner Algorithm)
- **Migration Scanner**: 메모리 하단(Low)에서 상단으로 이동하며 '옮길 수 있는 페이지'를 찾는다.
- **Free Scanner**: 메모리 상단(High)에서 하단으로 이동하며 '빈 페이지(Hole)'를 찾는다.

### 2. ASCII: 컴팩션 진행 과정
```text
[ Initial State: Fragmented Memory ]
[U][U][F][U][F][F][U][U][F][U] (U: Used, F: Free)
 ^                          ^
 Mig_Scanner              Free_Scanner

[ Mid Process: Moving Pages ]
[U][U][.][U][F][F][U][U][F][U]
       |                 ^
       +-----------------+ (Page Migration)

[ Final State: Compacted Memory ]
[U][U][U][U][U][U][F][F][F][F] (Continuous Free Space!)
                   |---------| -> Huge Page Allocation Possible
```

### 3. 종료 조건
- 두 스캐너가 메모리 중간 지점에서 서로 만날 때 컴팩션 과정이 완료된 것으로 간주한다.

- **📢 섹션 요약 비유**: "교실 뒷줄의 빈자리로 앞줄에 앉은 학생들을 한 명씩 옮겨서, 앞쪽의 대형 칠판 앞에 넓은 공간을 확보하는 청소 시간과 같습니다."

+++

## Ⅲ. 구현 상세: 페이지 가동성 (Page Mobility)

### 1. MIGRATE_TYPES
- 리눅스 커널은 페이지 블록을 성격에 따라 분류한다.
    - **MIGRATE_MOVABLE**: 사용자 데이터 등 언제든 옮길 수 있는 페이지. (컴팩션의 주 대상)
    - **MIGRATE_RECLAIMABLE**: 옮길 수는 없지만 삭제 후 다시 읽어올 수 있는 페이지 (예: 슬랩 캐시).
    - **MIGRATE_UNMOVABLE**: 커널 핵심 코드 등 절대 옮길 수 없는 페이지.

### 2. 비용적 측면
- 페이지를 옮기는 것은 메모리 복사와 페이지 테이블 갱신을 동반하므로 CPU 자원을 많이 소모한다. 따라서 커널은 평소에는 백그라운드(`kcompactd`)에서 조용히 수행하다가, 긴급한 할당 요청이 있을 때만 강력하게 수행한다.

- **📢 섹션 요약 비유**: "짐을 옮기는 건 힘든 일이기 때문에, 평소에 틈틈이 정리해두지만 손님이 갑자기 들이닥칠 때만 허겁지겁 짐을 옮기는 전략입니다."

+++

## Ⅳ. 컴팩션의 성능 영향 및 최적화 (Strategy)

### 1. 시스템 지연 (Latency)
- 대용량 메모리 환경에서 컴팩션이 발생하면 일시적으로 시스템 성능이 저하될 수 있다. 이를 **'Direct Compaction'**에 의한 지연이라고 한다.

### 2. 최적화: 물량 기반 정리 (Lumpy Reclaim)
- 단순히 빈 공간을 찾는 것을 넘어, 거대 페이지가 들어갈 특정 구역의 페이지만 집중적으로 비우거나 옮기는 공격적인 기법들이 병행된다.

- **📢 섹션 요약 비유**: "도서관 전체를 정리하는 게 너무 힘들면, 딱 백과사전 세트가 들어갈 칸들만 골라서 집중적으로 비우는 지혜를 발휘하는 것입니다."

+++

## Ⅴ. 결론 및 요약

### 1. 핵심 요약
- 메모리 컴팩션은 **'물리적 한계(단편화)'를 '소프트웨어적 노력(이동)'**으로 극복하여, 시스템의 장기 가동 안정성을 보장하는 핵심 기술이다.

### 2. 시스템적 의의
- 정적인 메모리 할당의 패러다임을 '유동적이고 유기적인 자원 관리'로 변화시켰으며, 특히 고성능 연산이 필요한 서버 환경에서 필수적인 기능을 수행한다.

- **📢 섹션 요약 비유**: "결국 '고여서 썩는 공간'이 없도록 끊임없이 자리를 재배치함으로써, 메모리라는 한정된 자원에 생명력을 불어넣는 작업입니다."

+++

### 📌 관련 개념 맵 (Knowledge Graph)
- **외부 단편화 (External Fragmentation)**: 컴팩션이 해결하고자 하는 근본적인 문제.
- **버디 시스템 (Buddy System)**: 컴팩션의 결과를 토대로 실제 할당을 수행하는 시스템.
- **Huge Pages**: 컴팩션의 가장 큰 수혜자이자 존재 이유.
- **페이지 교체 알고리즘 (Page Replacement)**: 컴팩션과 함께 메모리 공간을 확보하는 또 다른 축.
