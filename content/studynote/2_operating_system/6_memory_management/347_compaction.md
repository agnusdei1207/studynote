# 압축 (Compaction)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 압축(Compaction)은 메모리 곳곳에 흩어져 있는 프로세스들을 한쪽 끝으로 밀어 넣고, 파편화된 빈 공간들을 하나의 커다란 연속된 공간으로 합치는 기법이다.
> 2. **전제 조건**: 실행 중인 프로세스의 메모리 주소를 동적으로 바꿀 수 있는 '동적 재배치(Dynamic Relocation)' 기능이 하드웨어적으로 지원되어야만 수행 가능하다.
> 3. **비용**: 메모리의 데이터를 실제로 복사하고 이동시켜야 하므로 매우 높은 CPU 오버헤드가 발생하며, 작업 동안 시스템이 멈추는 현상이 동반될 수 있다.

+++

## Ⅰ. 압축 (Compaction)의 개념 및 목적

### 1. 정의
- **압축 (Compaction)**은 외부 단편화(External Fragmentation)를 해결하기 위해 사용 중인 메모리 블록들을 인접하게 배치하여, 흩어진 가용 공간(Hole)들을 하나의 큰 블록으로 통합하는 메모리 관리 기술이다.

### 2. 목적
- 새로운 대형 프로세스를 수용할 수 있는 '연속된' 빈 공간을 확보한다.
- 메모리 이용률을 극대화하여 'Out of Memory' 상황을 방지한다.

### 3. 💡 비유: '책상 위 잡동사니 정리'
- 책상 위에 연필, 지우개, 책이 어지럽게 널려 있어 노트북을 놓을 자리가 없을 때, 물건들을 한쪽 구석으로 싹 밀어버리고 넓은 빈 자리를 만드는 것과 같다.

- **📢 섹션 요약 비유**: "가방 속에 짐들이 띄엄띄엄 들어있어서 큰 물건이 안 들어갈 때, 짐들을 바닥으로 꾹꾹 눌러 담아 윗부분에 넓은 공간을 만드는 작업입니다."

+++

## Ⅱ. 압축의 메커니즘 및 ASCII 다이어그램

### 1. 수행 과정
1. 현재 메모리를 점유 중인 모든 프로세스의 상태를 중지시킨다.
2. 모든 프로세스의 메모리 데이터를 새로운 주소로 복사(Copy)한다.
3. 재배치 레지스터(Relocation Register) 값을 업데이트하여 프로세스가 새로운 위치를 인식하게 한다.

### 2. ASCII 전/후 비교 다이어그램
```text
[ Before Compaction ]              [ After Compaction ]
+-----------------------+          +-----------------------+
|  OS                   |          |  OS                   |
+-----------------------+          +-----------------------+
|  Process A            |          |  Process A            |
+-----------------------+          +-----------------------+
|  (Hole: 20KB)         |          |  Process B            |
+-----------------------+          +-----------------------+
|  Process B            |   --->   |  Process C            |
+-----------------------+   Comp.  +-----------------------+
|  (Hole: 30KB)         |          |                       |
+-----------------------+          |  (Combined Hole: 60KB)|
|  Process C            |          |                       |
+-----------------------+          |                       |
|  (Hole: 10KB)         |          |                       |
+-----------------------+          +-----------------------+
```

### 3. 기술적 제약: 주소 바인딩(Binding)
- 압축은 **실행 시간 바인딩(Execution-time Binding)** 방식에서만 가능하다. 컴파일 시점이나 로드 시점에 주소가 고정된 경우에는 프로세스를 옮길 수 없기 때문이다.

- **📢 섹션 요약 비유**: "퍼즐 조각들을 한데 모으는 것인데, 조각들이 바닥에 본드로 붙어 있지 않아야(동적 재배치) 가능한 일입니다."

+++

## Ⅲ. 압축의 전략 및 비용 분석

### 1. 이동 전략 (Movement Strategy)
- **최소 이동**: 이동해야 하는 데이터의 양이 가장 적은 프로세스들만 골라 옮긴다.
- **단방향 밀기**: 모든 프로세스를 저주소(0번지 방향)로 몰아넣는다.

### 2. 성능 오버헤드 (The Cost)
- **데이터 복사 부하**: 수 GB의 데이터를 메모리 내에서 이동시키는 것은 엄청난 메모리 대역폭을 소모한다.
- **시스템 중단**: 압축이 진행되는 동안 프로세스들은 CPU를 점유할 수 없어 사용자에게 '렉(Lag)'이나 '프리징(Freezing)'으로 느껴진다.

- **📢 섹션 요약 비유**: "이사 가는 것과 비슷합니다. 짐이 많을수록 힘들고, 이사하는 동안에는 집에서 편히 쉴 수 없는 것과 같은 고통이 따릅니다."

+++

## Ⅳ. 실무적 대안 및 관련 기술 (Strategy)

### 1. 가비지 컬렉션 (Garbage Collection)과의 관계
- Java나 Python 같은 언어의 런타임 환경에서는 힙(Heap) 메모리 관리를 위해 주기적으로 압축 작업을 수행한다. (예: Mark-and-Compact 알고리즘)

### 2. 현대적 대안: 페이징 (Paging)
- 현대 OS는 압축의 높은 비용을 감당하는 대신, **페이징** 기법을 선택했다. 페이징은 물리적으로 데이터를 옮기지 않고도 조각난 공간들을 논리적으로 하나처럼 쓸 수 있게 해주기 때문이다.

- **📢 섹션 요약 비유**: "물건을 억지로 밀어서 치우는 대신(압축), 물건을 작은 부품으로 분해해서 빈틈마다 쏙쏙 집어넣는(페이징) 기술로 진화했습니다."

+++

## Ⅴ. 결론 및 요약

### 1. 핵심 요약
- 압축은 외부 단편화를 물리적으로 해결하는 '정면 돌파' 방식이지만, 시스템 자원 소모가 막대한 '고비용 전략'이다.

### 2. 역사적 의의
- 과거 연속 할당 방식을 쓰던 시절에는 필수적인 기술이었으나, 현재는 하드웨어 레벨의 메모리 관리보다는 특정 언어의 가상 머신(VM)이나 특수 목적 시스템의 메모리 최적화 기법으로 그 명맥을 이어가고 있다.

- **📢 섹션 요약 비유**: "방이 너무 지저분할 때 가끔 하는 '대청소'와 같습니다. 깨끗해지긴 하지만, 청소하는 동안에는 아무것도 못 하니 평소에 잘 정리하는 것이 더 중요합니다."

+++

### 📌 관련 개념 맵 (Knowledge Graph)
- **외부 단편화 (External Fragmentation)**: 압축이 필요한 근본 원인.
- **실행 시간 바인딩 (Execution-time Binding)**: 압축을 가능하게 하는 기술적 전제.
- **페이징 (Paging)**: 압축 없이 단편화를 해결하는 현대적 대안.
- **버디 시스템 (Buddy System)**: 단편화 발생 자체를 억제하려는 또 다른 시도.
