# 해시 페이지 테이블 (Hashed Page Table)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해시 페이지 테이블(Hashed Page Table)은 가상 페이지 번호를 해시 함수(Hash Function)에 통과시켜 주소 변환 정보를 저장할 인덱스를 결정하는 방식이다.
> 2. **가치**: 주소 공간이 매우 넓은(64비트 등) 환경에서 페이지 테이블의 크기를 물리 메모리 크기에 비례하게 유지할 수 있어, 메모리 낭비를 줄이는 효율적인 대안이 된다.
> 3. **특징**: 해시 충돌(Collision)이 발생할 경우 연결 리스트(Linked List) 형태의 체이닝(Chaining)을 통해 여러 엔트리를 관리하며, 리스트가 길어질수록 탐색 성능이 저하될 수 있다.

+++

## Ⅰ. 해시 페이지 테이블 (Hashed Page Table)의 정의 및 필요성

### 1. 정의
- **해시 페이지 테이블 (Hashed Page Table)**은 해시 자료구조를 이용하여 가상 주소를 물리 주소로 매핑하는 기법이다. 가상 페이지 번호($p$)가 해시 키(Key)가 되고, 물리 프레임 번호($f$)가 값(Value)이 된다.

### 2. 등장 배경: 64비트 시스템의 한계
- 다단계 페이징은 주소 비트 수가 늘어날수록 계층이 깊어져 메모리 접근 횟수가 과도하게 늘어난다.
- 해시 방식은 주소 공간의 크기에 상관없이 실제 사용 중인 페이지의 수에 맞춰 테이블 크기를 최적화할 수 있다.

### 3. Sparse Address Space 최적화
- 가상 주소 공간의 대부분이 비어 있는 '희소 주소 공간'에서, 존재하는 페이지만을 해시 테이블에 담아 관리하므로 공간 효율성이 매우 높다.

- **📢 섹션 요약 비유**: "수만 명의 학생 이름을 가나다순으로 나열한 두꺼운 명부 대신, '생일'이라는 해시 함수를 이용해 특정 날짜의 칸(Bucket)만 확인하는 것과 같습니다."

+++

## Ⅱ. 해시 페이지 테이블의 작동 메커니즘 및 ASCII 다이어그램

### 1. 주소 변환 프로세스
1. CPU가 가상 주소에서 페이지 번호 $p$를 추출한다.
2. $p$를 해시 함수에 대입하여 해시 테이블의 인덱스를 계산한다.
3. 해당 인덱스의 연결 리스트를 따라가며 $p$와 일치하는 태그(Tag)를 찾는다.
4. 일치하는 엔트리를 찾으면 프레임 번호 $f$를 추출하고 Offset과 결합한다.

### 2. ASCII: 해시 기반 주소 변환
```text
[ Virtual Address ]
+---------------------+-----------+
| Page Number (p)     |  Offset   |
+---------------------+-----------+
           |
           v
    +---------------+
    | Hash Function |
    +---------------+
           |
           v (Hash Index)
    +-----------+      +---------------------------+
    | Hash Table|      | Linked List of Entries    |
    +-----------+      +---------------------------+
    |   ...     |      | [p_i | f_i | next]        |
    +-----------+      +-----------|---------------+
    |   Index   |----> | [p_a | f_a | *]-->(p_b | f_b | NULL)
    +-----------+      +---------------------------+
    |   ...     |       (p_a가 우리가 찾는 p인가? Yes/No)
    +-----------+
```

### 3. 해시 충돌(Collision) 처리
- 서로 다른 $p$가 동일한 해시 값을 가질 경우, 해당 버킷에 연결 리스트(Chaining)를 형성하여 순차적으로 탐색한다.

- **📢 섹션 요약 비유**: "사물함 번호를 '이름 끝자리'로 정했을 때, '김철수'와 '이영수'가 같은 0번 사물함에 짐을 나란히 넣어두고 자기 이름표를 찾는 과정입니다."

+++

## Ⅲ. 기술적 특징 및 장단점

### 1. 장점 (Pros)
- **공간 효율성**: 물리 메모리 크기에 비례하거나 실제 사용량에 맞춘 테이블 크기 설정이 가능하여 메모리 점유가 적다.
- **확장성**: 64비트 이상의 거대 주소 체계에서도 계층을 늘리지 않고 대응 가능하다.

### 2. 단점 (Cons)
- **탐색 성능의 불확실성**: 해시 충돌이 잦아져 연결 리스트가 길어지면 최악의 경우 선형 탐색($O(N)$)의 오버헤드가 발생한다.
- **해시 오버헤드**: 매번 해시 함수를 계산하는 연산 비용이 발생한다.

- **📢 섹션 요약 비유**: "서류를 찾을 때 분류함만 열면 바로 나올 수도 있지만, 한 칸에 서류가 너무 많이 쌓여 있으면 하나씩 다 들춰봐야 하는 것과 같습니다."

+++

## Ⅳ. 변형 및 현대적 응용 (Strategy)

### 1. 클러스터 페이지 테이블 (Clustered Page Table)
- 해시 테이블의 각 엔트리가 단일 페이지가 아닌, 연속된 여러 페이지(Cluster)의 정보를 담는 방식이다.
- 메모리 참조의 국부성(Locality)을 활용하여, 인접한 페이지들을 한꺼번에 가져옴으로써 탐색 횟수를 줄인다.

### 2. 64비트 아키텍처에서의 위치
- 울트라스파크(UltraSPARC)나 IA-64와 같은 아키텍처에서 주로 사용되었으며, 현대의 범용 CPU보다는 특수 목적 시스템이나 특정 가상화 환경에서 효율성을 발휘한다.

- **📢 섹션 요약 비유**: "사물함에 물건을 하나씩 넣는 대신, 연관된 물건들을 한 바구니(Cluster)에 담아 한 번의 열람으로 여러 정보를 얻는 최적화 전략입니다."

+++

## Ⅴ. 결론 및 요약

### 1. 핵심 요약
- 해시 페이지 테이블은 **'주소 공간의 크기'가 아닌 '사용 중인 주소의 양'**에 집중하여 메모리 매핑 성능을 최적화하는 기법이다.

### 2. 시스템적 의의
- 결정론적인 계층 구조(다단계 페이징)의 경직성을 탈피하여, 확률적 효율성(해싱)을 메모리 관리에 도입한 사례로 평가받는다.

- **📢 섹션 요약 비유**: "결국 '모든 가능성을 미리 준비'하는 대신 '실제 일어나는 일들에 대해서만 빠르게 대응'하는 실용적인 접근법입니다."
