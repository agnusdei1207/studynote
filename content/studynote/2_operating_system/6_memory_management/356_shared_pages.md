# 페이징에서의 공유 페이지 (Shared Pages)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 공유 페이지(Shared Pages)는 여러 프로세스의 서로 다른 페이지 테이블 엔트리가 물리 메모리의 동일한 프레임을 가리키도록 설정하여 메모리를 공동 사용하는 기술이다.
> 2. **가치**: 공통으로 사용되는 코드(표준 라이브러리, 컴파일러 등)를 중복 로드하지 않음으로써 메인 메모리 낭비를 획기적으로 줄인다.
> 3. **조건**: 공유되는 코드는 실행 중에 자기 자신을 수정하지 않는 **재진입 가능 코드(Reentrant Code)**여야 하며, 각 프로세스는 해당 페이지에 대해 읽기 전용(Read-only) 권한을 가져야 한다.

+++

## Ⅰ. 공유 페이지 (Shared Pages)의 개념 및 필요성

### 1. 정의
- **공유 페이지 (Shared Pages)**란 둘 이상의 프로세스가 물리적 메모리 상의 동일한 페이지 프레임을 자신의 주소 공간의 일부로 매핑하여 사용하는 방식이다.

### 2. 등장 배경
- 100명의 사용자가 동시에 `vi` 에디터를 사용한다면, 각자 1MB씩 총 100MB의 메모리가 필요하다. 하지만 에디터의 실행 코드는 모두 동일하므로, 이를 1MB만 메모리에 올리고 모두가 공유하면 99MB를 아낄 수 있다.

### 3. 기술적 핵심: 재진입 가능 코드 (Reentrant Code / Pure Code)
- 공유되는 코드는 실행 도중 값이 변하지 않아야 한다. 데이터는 각 프로세스마다 별도의 페이지에 보관하고, 실행 지침(Code)만 공유하는 것이 원칙이다.

- **📢 섹션 요약 비유**: "각자 다른 집(프로세스)에 살지만, 마을의 공용 도서관(공유 페이지)에 있는 백과사전은 한 권만 두고 다 같이 가서 읽는 것과 같습니다."

+++

## Ⅱ. 공유 페이지의 메커니즘 및 ASCII 다이어그램

### 1. 매핑 구조
- 프로세스 A의 페이지 테이블 3번 엔트리와 프로세스 B의 페이지 테이블 5번 엔트리가 모두 물리 프레임 100번을 가리키도록 설정한다.

### 2. ASCII: 프로세스 간 메모리 공유
```text
[ Process P1 Table ]      [ Physical Memory ]      [ Process P2 Table ]
+-----+-------+           +-----------------+           +-----+-------+
|Page | Frame |           |                 |           |Page | Frame |
+-----+-------+           +-----------------+           +-----+-------+
|  0  |  100  | --------> | Frame 100:      | <-------- |  0  |  100  |
+-----+-------+           | (Shared Code)   |           +-----+-------+
|  1  |  200  | --+       +-----------------+       +-- |  1  |  300  |
+-----+-------+   |       | Frame 200:      |       |   +-----+-------+
                  |       | (P1 Data)       |       |
                  +-----> +-----------------+ <-----+
                          | Frame 300:      |
                          | (P2 Data)       |
                          +-----------------+
```

### 3. 읽기 전용 권한 설정
- 페이지 테이블의 권한 비트(Protection Bits)를 사용하여 공유 페이지를 'Read-only'로 설정함으로써, 한 프로세스가 공유 코드를 실수로 수정하여 다른 프로세스에 영향을 주는 것을 방지한다.

- **📢 섹션 요약 비유**: "여러 명이 같은 지도를 보고 있지만, 누구도 지도 위에 낙서(Write)를 할 수는 없게 코팅해둔 상태입니다."

+++

## Ⅲ. 공유 페이지의 장점 및 제약 사항

### 1. 장점 (Pros)
- **메모리 절약**: 시스템 전체의 가용 메모리 공간이 늘어난다.
- **프로세스 생성 속도 향상**: `fork()` 시 자식 프로세스가 부모의 메모리를 통째로 복사하는 대신 페이지를 공유(Copy-on-Write와 연계)함으로써 빠르게 시작할 수 있다.
- **IPC(Inter-Process Communication) 활용**: 공유 메모리(Shared Memory) 기법을 통해 프로세스 간 데이터를 매우 빠르게 주고받을 수 있다.

### 2. 제약 사항 (Cons/Challenges)
- **코드의 불변성 유지**: 데이터와 코드가 섞여 있으면 공유가 불가능하다.
- **주소 일관성**: 공유 페이지가 각 프로세스의 논리 주소 공간에서 같은 위치에 있어야 관리가 쉬운 경우(예: 절대 주소 참조 시)가 있어 설계가 복잡해질 수 있다.

- **📢 섹션 요약 비유**: "주차장 공간을 아끼기 위해 카풀(공유)을 하는 것과 같습니다. 기름값은 아끼지만, 서로의 목적지와 시간이 맞아야 한다는 제약이 있죠."

+++

## Ⅳ. 현대 운영체제에서의 응용 사례 (Strategy)

### 1. 공유 라이브러리 (Shared Libraries / DLL)
- Windows의 `.dll`이나 Linux의 `.so` 파일들이 대표적인 예시다. `printf()` 같은 표준 함수는 수천 개의 프로세스가 단 하나의 물리 메모리 복사본을 공유한다.

### 2. Copy-on-Write (COW)
- 프로세스 복제 시 처음에는 모든 페이지를 공유하다가, 어느 한쪽이 내용을 수정(Write)하려 할 때만 해당 페이지를 별도로 복사해 나가는 지능적인 공유 전략이다.

### 3. 가상 머신(VM) 및 컨테이너
- 동일한 OS 커널을 사용하는 여러 컨테이너는 커널의 읽기 전용 코드 부분을 공유 페이지로 관리하여 오버헤드를 극소화한다.

- **📢 섹션 요약 비유**: "모든 요리책(라이브러리)을 집마다 살 필요 없이, 클라우드 서버(공유 메모리)에 한 권만 올려두고 필요할 때마다 접속해서 보는 스마트 요리법입니다."

+++

## Ⅴ. 결론 및 요약

### 1. 핵심 요약
- 공유 페이지는 페이징 기법의 유연성을 극대화하여, '독립된 주소 공간'이라는 추상화를 유지하면서도 '물리 자원의 공동 활용'이라는 실리적 효율을 달성한다.

### 2. 기술적 의의
- 현대 운영체제가 수많은 프로세스를 동시에, 그리고 빠르게 구동할 수 있는 것은 이 공유 페이지 기술을 통한 메모리 다이어트 덕분이다.

- **📢 섹션 요약 비유**: "나만의 공간은 확실히 보장받으면서, 공용 시설은 알뜰하게 나눠 쓰는 '최첨단 쉐어하우스' 시스템입니다."

- **Copy-on-Write (COW)**: 쓰기 발생 시 공유를 해제하는 고급 기법.
- **공유 메모리 (Shared Memory)**: 프로세스 간 통신(IPC)을 위한 공유 페이지 활용.
