# 페이지 크기 (Page Size)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이지 크기(Page Size)는 운영체제가 논리 메모리와 물리 메모리를 나누는 최소 단위의 용량으로, 시스템 성능과 메모리 이용 효율 사이의 핵심 트레이드오프(Trade-off) 요소다.
> 2. **결정 요인**: 페이지가 작으면 내부 단편화가 줄어드나 관리 부담(테이블 크기)이 늘고, 페이지가 크면 입출력 효율과 TLB 히트율은 좋아지나 내부 단편화가 심해진다.
> 3. **현대적 추세**: 메모리 용량이 비약적으로 증가함에 따라 기본 페이지(보통 4KB) 외에도 성능 최적화를 위한 대용량 페이지(Huge/Large Pages, 2MB~1GB) 사용이 늘고 있다.

---

## Ⅰ. 페이지 크기 (Page Size)의 정의 및 결정 방식

### 1. 정의
- **페이지 크기 (Page Size)**는 페이징 기법에서 논리적 페이지와 물리적 프레임의 고정된 크기를 의미한다. 하드웨어(CPU/MMU) 아키텍처에 의해 결정되며 보통 2의 거듭제곱(2^n) 단위를 사용한다.

### 2. 주소 구조와의 관계
- 페이지 크기가 $2^n$ 바이트일 때, 논리 주소의 하위 $n$ 비트는 페이지 내의 위치를 나타내는 **오프셋(Offset)**으로 사용되고, 상위 비트는 **페이지 번호(Page Number)**로 사용된다.

### 3. 하드웨어 의존성
- 프로세서 아키텍처(x86, ARM 등)에 따라 지원하는 기본 페이지 크기가 다르며, 현대 x86-64 시스템의 기본값은 보통 **4KB ($2^{12}$ Bytes)**이다.

- **📢 섹션 요약 비유**: "이사 갈 때 짐을 담는 '박스의 규격'을 정하는 것입니다. 박스가 너무 작으면 박스 개수가 너무 많아지고, 너무 크면 박스 안의 빈 공간이 아까워집니다."

---

## Ⅱ. 페이지 크기에 따른 성능 트레이드오프 및 ASCII 비교

### 1. 작은 페이지 vs 큰 페이지 비교
| 항목 | 작은 페이지 (Small Page) | 큰 페이지 (Large Page) |
| :--- | :--- | :--- |
| **내부 단편화** | 감소 (정밀한 할당 가능) | 증가 (마지막 페이지 낭비) |
| **페이지 테이블 크기** | 증가 (항목이 많아짐) | 감소 (항목이 적어짐) |
| **TLB 효율** | 낮음 (커버 범위 좁음) | 높음 (커버 범위 넓음) |
| **I/O 효율** | 낮음 (여러 번 읽어야 함) | 높음 (연속 데이터 전송 유리) |

### 2. ASCII: 주소 변환 범위 비교
```text
[ Small Page (4KB) ]                [ Large Page (2MB) ]
31      12 11        0              31      21 20        0
+---------+----------+              +---------+----------+
|  Page   |  Offset  |              |  Page   |  Offset  |
|  (20b)  |  (12b)   |              |  (11b)  |  (21b)   |
+---------+----------+              +---------+----------+
-> Table Entries: 2^20              -> Table Entries: 2^11
-> 1,048,576 entries                -> 2,048 entries
```

### 3. 메모리 해상도(Resolution)의 개념
- 페이지 크기는 메모리를 바라보는 '해상도'와 같다. 작은 페이지는 고해상도(정교함)를 제공하지만 연산량이 많고, 큰 페이지는 저해상도(투박함)지만 처리가 빠르다.

- **📢 섹션 요약 비유**: "모자이크 그림을 그릴 때 타일(페이지)이 작으면 정교한 그림이 나오지만 타일 붙이기가 힘들고, 타일이 크면 금방 붙이지만 세밀한 표현이 안 되는 것과 같습니다."

## Ⅲ. 페이지 크기 결정의 기술적 상세 분석

### 1. 페이지 테이블 오버헤드
- 페이지 크기가 작을수록 페이지 테이블의 엔트리(Entry) 수가 기하급수적으로 늘어나, 테이블 자체가 차지하는 메모리 공간이 프로세스 실제 데이터보다 커지는 역전 현상이 발생할 수 있다.

### 2. TLB Reach (TLB 커버 범위)
- TLB는 엔트리 수가 제한적이다. 페이지 크기가 4KB일 때 TLB 100개는 400KB를 커버하지만, 2MB일 때는 200MB를 커버한다. 따라서 큰 페이지는 CPU의 메모리 접근 속도를 비약적으로 높인다.

### 3. 디스크 입출력 (Disk I/O) 최적화
- 운영체제는 디스크에서 데이터를 가져올 때 '지역성(Locality)'을 활용한다. 페이지가 클수록 한 번의 디스크 접근으로 더 많은 관련 데이터를 가져올 수 있어 효율적이다.

- **📢 섹션 요약 비유**: "한 번에 한 숟가락(작은 페이지)씩 밥을 먹으면 흘리는 건 없지만 오래 걸리고, 국자(큰 페이지)로 먹으면 빨리 먹지만 입가에 묻는 게 많은 것과 같습니다."

---

## Ⅳ. 현대 운영체제의 전략: 다중 페이지 크기 (Huge Pages)

### 1. Huge Pages / Large Pages 기술
- 현대 OS(Linux, Windows 등)는 기본 4KB 외에 2MB나 1GB 크기의 페이지를 동시에 지원한다. DB나 가상화 솔루션처럼 방대한 메모리를 쓰는 프로그램에 할당하여 성능을 높인다.

### 2. 하드웨어 발전과 페이지 크기 확대
- 메모리 용량이 테라바이트(TB) 단위로 커지면서, 4KB 페이지 방식은 관리 비용이 너무 커져 점차 기본 페이지 크기 자체를 늘리려는(예: ARM의 16KB/64KB 지원) 움직임이 있다.

- **📢 섹션 요약 비유**: "택배를 보낼 때 일반 물건은 표준 상자(4KB)에 담지만, 엄청 큰 기계는 컨테이너(Huge Page)에 통째로 담아 한 번에 옮기는 복합 운송 전략입니다."

---

## Ⅴ. 결론 및 요약

### 1. 핵심 요약
- 페이지 크기는 시스템의 **공간 효율성(내부 단편화 최소화)**과 **시간 효율성(주소 변환 및 I/O 속도)** 사이에서 균형을 맞추는 지점이다.

### 2. 향후 전망
- 데이터 센터와 AI 연산 등 고성능 컴퓨팅 수요가 늘어남에 따라, 관리 오버헤드를 줄이기 위한 대형 페이지 기술의 중요성은 더욱 커질 것이다.

- **📢 섹션 요약 비유**: "시스템의 덩치가 커질수록 작은 자루보다는 큰 포대가 관리에 더 유리해지는 규모의 경제 원리가 메모리 세계에도 적용됩니다."
