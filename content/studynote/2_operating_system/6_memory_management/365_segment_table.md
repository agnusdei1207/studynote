# 세그먼트 테이블 (Segment Table)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 세그먼트 테이블(Segment Table)은 세그멘테이션 기법에서 가상 주소를 물리 주소로 변환하기 위해, 각 세그먼트의 시작 주소(Base)와 길이(Limit) 정보를 저장하는 매핑 테이블이다.
> 2. **가치**: 고정 크기인 페이징과 달리 각 세그먼트의 '실제 크기'를 Limit에 기록함으로써, 프로세스가 허용된 영역을 벗어나는 잘못된 접근을 시도할 때 하드웨어 레벨에서 즉각 차단한다.
> 3. **작동 원리**: CPU의 세그먼트 테이블 레지스터(STBR, STLR)와 연동되어 작동하며, 주소 변환 과정에서 베이스 주소와의 단순 덧셈 연산을 통해 최종 물리 주소를 산출한다.

+++

## Ⅰ. 세그먼트 테이블 (Segment Table)의 정의 및 구성

### 1. 정의
- **세그먼트 테이블 (Segment Table)**은 프로세스의 논리적 세그먼트들을 물리 메모리의 연속된 공간에 매핑하기 위한 정보를 관리하는 표다.

### 2. 엔트리(Entry)의 구성 요소
- **Base (기준 주소)**: 해당 세그먼트가 물리 메모리에서 시작되는 시작 지점의 주소.
- **Limit (한계/길이)**: 해당 세그먼트의 전체 크기(바이트 또는 워드 단위).

### 3. 하드웨어 지원 레지스터
- **STBR (Segment-Table Base Register)**: 현재 실행 중인 프로세스의 세그먼트 테이블이 메모리 어디에 있는지 가리키는 시작 주소.
- **STLR (Segment-Table Length Register)**: 현재 프로세스가 가진 세그먼트의 총 개수. (세그먼트 번호 $s$가 STLR보다 크면 오류)

- **📢 섹션 요약 비유**: "각 세그먼트라는 물건들에 대해, 보관 창고의 몇 번 칸(Base)부터 시작해서 몇 미터(Limit)만큼 차지하고 있는지 적어둔 보관 명부와 같습니다."

+++

## Ⅱ. 세그먼트 테이블의 작동 메커니즘 및 ASCII 다이어그램

### 1. 주소 변환 구조
- 논리 주소 $[s, d]$를 입력받아, 세그먼트 테이블의 $s$번째 행을 참조한다.

### 2. ASCII: 세그먼트 테이블 상세 구조
```text
[ Segment Table ]
+-------+-----------+-----------+
| Index | Limit (L) | Base (B)  |
+-------+-----------+-----------+
|   0   |   1000    |   1400    | -> Segment 0 (Code)
+-------+-----------+-----------+
|   1   |    400    |   6300    | -> Segment 1 (Data)
+-------+-----------+-----------+
|   2   |    400    |   4300    | -> Segment 2 (Stack)
+-------+-----------+-----------+
|  ...  |    ...    |    ...    |
+-------+-----------+-----------+

[ Address Calculation ]
Logical: <s=1, d=100>
Table(1): Base=6300, Limit=400
Check: (100 < 400)? -> YES
Physical: 6300 + 100 = 6400
```

### 3. 유효성 검사 (Validity Check)
- $s < STLR$: 요청한 세그먼트 번호가 실제로 존재하는지 확인.
- $d < Limit$: 요청한 오프셋이 세그먼트의 할당된 범위를 넘지 않는지 확인. (넘으면 Trap 발생)

- **📢 섹션 요약 비유**: "주소를 찾아갈 때, 먼저 해당 층이 실존하는지(STLR) 확인하고, 그 층의 방 번호가 유효한지(Limit) 이중으로 체크하는 보안 가이드와 같습니다."

+++

## Ⅲ. 세그먼트 테이블의 관리 및 오버헤드

### 1. 문맥 교환 (Context Switch)
- 프로세스가 교체될 때 운영체제는 CPU의 **STBR**과 **STLR** 값을 새 프로세스의 정보로 업데이트한다. 페이지 테이블과 마찬가지로 테이블 자체는 메인 메모리에 위치한다.

### 2. 메모리 접근 오버헤드
- 논리 주소 하나를 변환하기 위해 **'메모리에 있는 세그먼트 테이블 접근' + '실제 메모리 접근'**으로 총 2번의 메모리 참조가 발생한다.
- 이를 해결하기 위해 페이징과 마찬가지로 **TLB(Translation Look-aside Buffer)**를 사용하여 자주 참조되는 매핑 정보를 캐싱한다.

- **📢 섹션 요약 비유**: "길을 물어볼 때마다 관리 사무소에 들러서 명부(세그먼트 테이블)를 확인해야 하므로, 자주 가는 길은 머릿속(TLB)에 외워두는 전략이 필요합니다."

+++

## Ⅳ. 페이징 테이블과의 차이점 (Strategy)

### 1. 고정 크기 vs 가변 크기
- 페이지 테이블의 각 엔트리는 모두 동일한 크기의 페이지를 가리키므로 'Limit' 필드가 필요 없다.
- 세그먼트 테이블은 세그먼트마다 크기가 다르므로 반드시 **'Limit'** 정보가 있어야 하며, 이를 통해 정밀한 메모리 보호가 가능하다.

### 2. 단편화 양상
- 페이지 테이블은 내부 단편화(Internal)를 발생시키지만, 세그먼트 테이블은 물리 메모리에 가변 크기로 배치되므로 **외부 단편화(External)**를 유발한다.

- **📢 섹션 요약 비유**: "모두 같은 크기의 상자(페이지)를 쌓을 때는 높이만 알면 되지만, 크기가 제각각인 상자(세그먼트)를 쌓을 때는 각 상자의 가로세로 길이(Limit)를 일일이 확인해야 하는 차이입니다."

+++

## Ⅴ. 결론 및 요약

### 1. 핵심 요약
- 세그먼트 테이블은 **논리적 구획(Segmentation)**과 **물리적 배치(Base)** 사이를 연결하는 지도로서, 'Limit' 필드를 통한 강력한 영역 보호 기능을 제공한다.

### 2. 시스템적 의의
- 단순한 주소 변환을 넘어, 프로세스가 자신의 권한 밖 영역을 침범하지 못하도록 막는 하드웨어 보안의 제1방어선 역할을 수행한다.

- **📢 섹션 요약 비유**: "결국 '어디서 시작하는가'보다 '어디까지 허용되는가'를 명확히 함으로써, 메모리 내의 질서를 유지하는 법전과 같은 존재입니다."

### 📌 관련 개념 맵 (Knowledge Graph)
- **세그멘테이션 (Segmentation)**: 세그먼트 테이블을 사용하는 근본적인 메모리 관리 기법.
- **STBR/STLR (Registers)**: 테이블의 위치와 크기를 정의하는 하드웨어 레지스터.
- **외부 단편화 (External Fragmentation)**: 가변 크기 세그먼트 할당의 고질적 문제.
- **TLB (Translation Look-aside Buffer)**: 테이블 접근 성능을 가속화하는 캐시.
