# 공유 라이브러리 (Shared Library) 스터브 (Stub) 코드

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스터브(Stub)는 공유 라이브러리의 실제 루틴 대신 실행 파일에 포함되는 아주 작은 코드 조각으로, 실제 루틴을 찾고 호출하는 '대리인' 역할을 수행한다.
> 2. **가치**: 라이브러리가 메모리에 있는지 확인하고, 없을 경우 로드하며, 최종적으로는 자신의 위치를 실제 함수의 주소로 덮어씌워 연결 오버헤드를 최소화한다.
> 3. **융합**: 동적 연결(Dynamic Linking)의 핵심 기법으로, '지연 로딩(Lazy Loading)'을 가능케 하여 시스템 자원의 효율성을 극대화한다.

+++

## Ⅰ. 스터브 (Stub) 코드의 개념 및 배경

### 1. 정의
- **스터브 (Stub)**란 동적 연결 라이브러리에서 실제 함수 코드를 대신하여 실행 파일에 자리 잡고 있는 '껍데기' 혹은 '자리 표시자' 코드를 의미한다.

### 2. 등장 배경: '모두를 다 가질 수는 없다'
- 동적 연결 시스템에서 실행 파일은 라이브러리 함수가 어디에 위치할지 미리 알 수 없다. 그렇다고 호출 시마다 전체 시스템을 뒤져서 함수를 찾는 것은 너무 느리다. 따라서 "일단 이 작은 코드(Stub)만 넣어두고, 실행할 때 진짜 위치를 알아내자"는 전략이 필요하게 되었다.

### 3. 💡 비유: '부재중 자동 응답기'
- 내가 친구(Library)에게 직접 전화를 거는 대신, 친구의 번호를 모를 때 **'안내원(Stub)'**에게 전화를 거는 것과 같다. 안내원은 친구의 번호를 확인해서 연결해주고, 다음번에는 내가 직접 전화를 걸 수 있도록 번호를 알려준다.

- **📢 섹션 요약 비유**: "진짜 주인공이 오기 전까지 무대를 지키는 대역 배우와 같습니다."

+++

## Ⅱ. 스터브의 작동 원리 (Deep Dive)

### 1. 호출 및 주소 변환 프로세스
1.  **최초 호출**: 프로세스가 라이브러리 함수 `foo()`를 호출하면, 실행 파일 내의 `foo_stub`이 실행된다.
2.  **검사 및 적재**: 스터브는 운영체제에 "라이브러리 $L$이 메모리에 있는가?"를 묻는다. 없으면 운영체제가 $L$을 물리 메모리에 로드한다.
3.  **바인딩(Binding)**: 스터브는 실제 `foo()`의 메모리 주소를 찾아낸다.
4.  **자가 교체**: 스터브는 자신을 호출했던 위치의 주소 정보를 실제 함수의 주소로 업데이트(Patch)한다.
5.  **이후 호출**: 다음부터는 스터브를 거치지 않고 실제 함수가 직접 실행된다.

### 2. ASCII 다이어그램: 스터브의 자가 업데이트
```text
 [ Before First Call ]             [ After Binding ]
 
 Main Code:                         Main Code:
   CALL [Stub_Address]                CALL [Actual_Address]
           │                                   │
           ▼                                   ▼
 +-------------------+             +-------------------+
 |   Stub Code:      |             |   Actual Lib Code:|
 | 1. Load Lib       |             |   (Real Logic)    |
 | 2. Get Address    |             +-------------------+
 | 3. Replace ME!    |
 +-------------------+
```

### 3. 기술적 상세 해설
- **점프 테이블 (Jump Table)**: 현대의 시스템(ELF 등)에서는 스터브가 직접 코드를 수정하기보다 **PLT(Procedure Linkage Table)**와 **GOT(Global Offset Table)**라는 구조를 사용하여 주소를 우회적으로 관리한다. 스터브는 이 테이블의 값을 채우는 역할을 한다.

- **📢 섹션 요약 비유**: "처음에는 지도를 보고 길을 찾지만, 한 번 가본 뒤에는 길을 외워서 바로 가는 것과 같습니다."

## Ⅲ. 주요 특징 및 분석

### 1. 기능적 분석
| 특징 | 내용 |
|:---|:---|
| **크기** | 수 바이트에서 수십 바이트 정도로 매우 작음. |
| **위치** | 실행 파일(Binary)의 코드 세그먼트 내부에 위치. |
| **자동화** | 컴파일러와 링커가 자동으로 생성하여 프로그래머의 개입이 불필요. |

### 2. 기술적 특이점: 지연 바인딩 (Lazy Binding)
- 모든 스터브를 프로그램 시작 시점에 연결하지 않고, 실제 그 함수가 '처음 호출될 때'만 연결 프로세스를 진행함으로써 초기 구동 속도를 높인다.

- **📢 섹션 요약 비유**: "시험지를 다 풀지 않고, 풀어야 하는 문제의 답만 그때그때 확인하는 효율성입니다."

+++

## Ⅳ. 실무 적용 및 장애 분석 (Strategy)

### 1. 실제 사례
- **Windows Import Address Table (IAT)**: 윈도우 실행 파일(PE)에서 스터브와 연결된 주소들이 저장되는 테이블이다.
- **Linux PLT/GOT**: 리눅스에서 동적 연결을 처리하는 핵심 하드웨어/소프트웨어 구조다.

### 2. 문제 해결 전략
- **심볼 해석 오류 (Symbol Resolution Error)**: 스터브가 실제 함수를 찾지 못할 경우 'Undefined Symbol' 에러가 발생한다. 이는 라이브러리 경로(LD_LIBRARY_PATH 등)가 잘못되었을 때 빈번하다.

- **📢 섹션 요약 비유**: "안내원이 친구 번호를 잃어버려서 연결을 못 해주는 상황입니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 기대효과
- **메모리 절약**: 중복되는 라이브러리 코드를 제거하여 전체 시스템의 가용 메모리를 확보한다.
- **유연한 업데이트**: 라이브러리 수정 시 실행 파일의 스터브 코드는 건드릴 필요가 없어 배포가 자유롭다.

### 2. 결론 및 향후 전망
- 스터브 코드는 동적 연결의 복잡함을 사용자로부터 숨기는 '추상화의 벽'이다. 현대의 서버리스 환경이나 MSA(Microservices Architecture)에서도 이와 유사한 프록시(Proxy) 개념으로 확장되어 사용되고 있다.

- **📢 섹션 요약 비유**: "작은 다리(Stub) 하나가 거대한 성(Library)으로 가는 길을 열어줍니다."

### 👶 어린이를 위한 3줄 비유 설명
1. 스터브는 피자를 주문할 때 사용하는 **'피자집 전화번호가 적힌 쪽지'**와 같아요.
2. 집에 진짜 피자는 없지만, 쪽지에 적힌 번호로 전화를 걸면 맛있는 피자가 배달되어 오죠.
3. 한 번 전화를 걸어본 뒤에는 번호를 다 외워서, 다음에는 쪽지를 안 봐도 바로 주문할 수 있답니다!
