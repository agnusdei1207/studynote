# 역 페이지 테이블 (Inverted Page Table)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 역 페이지 테이블(Inverted Page Table)은 가상 주소 공간이 아닌 '물리 메모리의 프레임'을 기준으로 테이블을 구성하여, 시스템 전체에 단 하나의 페이지 테이블만 유지하는 기법이다.
> 2. **가치**: 프로세스마다 별도의 페이지 테이블을 가질 필요가 없어 메모리 낭비를 극적으로 줄여주며, 특히 64비트와 같은 거대 주소 체계에서 공간 효율성이 극대화된다.
> 3. **도전 과제**: 주소를 찾을 때 테이블 전체를 탐색해야 하는 성능 저하 문제가 발생하므로, 이를 해결하기 위해 해시 테이블(Hash Table)과 연동하여 탐색 속도를 보완한다.

+++

## Ⅰ. 역 페이지 테이블 (Inverted Page Table)의 정의 및 필요성

### 1. 정의
- **역 페이지 테이블 (Inverted Page Table, IPT)**은 물리 메모리의 각 프레임(Frame)마다 하나의 엔트리를 갖는 테이블이다. 엔트리에는 해당 프레임에 올라와 있는 페이지 번호와 소유 프로세스의 ID(PID)가 저장된다.

### 2. 등장 배경: 프로세스별 테이블의 부담
- 기존 페이징은 프로세스마다 페이지 테이블이 필요하며, 사용하지 않는 가상 주소 영역까지 테이블 엔트리를 유지해야 하는 경우가 많아 메모리 점유가 컸다.
- IPT는 물리 메모리 크기에 비례하여 테이블 크기가 고정되므로, 시스템 내 프로세스가 많아져도 테이블 메모리 사용량이 늘어나지 않는다.

### 3. 논리적 반전
- "가상 페이지 $p$는 어디(프레임)에 있는가?"라는 질문 대신, **"물리 프레임 $f$에는 어떤 페이지가 들어있는가?"**를 기록하는 역발상적 구조다.

- **📢 섹션 요약 비유**: "모든 시민(프로세스)에게 개인 지도를 나눠주는 대신, 도시의 모든 집(물리 프레임) 대문 앞에 '누가 살고 있는지' 적어둔 전체 명부 하나만 관리하는 것과 같습니다."

+++

## Ⅱ. 역 페이지 테이블의 작동 메커니즘 및 ASCII 다이어그램

### 1. 주소 구성
- CPU가 제시하는 주소는 **[PID | Page Number (p) | Offset (d)]**로 구성된다.
- PID가 필요한 이유는 여러 프로세스가 동일한 가상 주소 $p$를 가질 수 있기 때문에, IPT 내에서 주인을 식별하기 위함이다.

### 2. ASCII: IPT의 주소 변환 과정
```text
[ CPU Logical Address ]
+-------+-----------+---------+
|  PID  |  Page (p) | Offset  |
+-------+-----------+---------+
    |         |          |
    +----+----+          |
         v               |
[ Inverted Page Table ]  |
+-------+-------+-------+|
| Index |  PID  | Page# ||
+-------+-------+-------+|
|   0   |  102  |  0x40 ||
+-------+-------+-------+|
|   i   |  PID  |   p   || <--- (Match Found!)
+-------+-------+-------+|
|  ...  |  ...  |  ...  ||
+-------+-------+-------+|
         |               |
         v (Index i)     v
    +-------------------------+
    | Physical Address: (i, d)|
    +-------------------------+
```

### 3. 변환 단계
1. IPT에서 (PID, p) 쌍을 찾을 때까지 테이블을 검색한다.
2. 일치하는 항목을 찾으면, 그 항목의 **인덱스(i)**가 곧 물리 메모리의 프레임 번호가 된다.
3. 인덱스 $i$와 Offset $d$를 결합하여 물리 주소를 생성한다.

- **📢 섹션 요약 비유**: "출석부에서 내 이름을 찾기 위해 1번 학생부터 차례대로 대조해보고, 내가 15번째에 있다면 15번 자리에 앉는 것과 같습니다."

## Ⅲ. 기술적 특징 및 장단점

### 1. 장점 (Pros)
- **메모리 절약**: 시스템 전체에 단 하나의 테이블만 존재하므로 페이지 테이블을 위한 메모리 오버헤드가 매우 낮다.
- **예측 가능성**: 물리 메모리 크기에 따라 테이블 크기가 결정되므로 관리가 용이하다.

### 2. 단점 (Cons)
- **탐색 오버헤드**: 주소를 찾을 때 최악의 경우 테이블 전체를 뒤져야 하므로($O(N)$), 기존의 직접 인덱싱 방식($O(1)$)보다 훨씬 느리다.
- **공유 메모리 구현의 어려움**: 하나의 프레임 엔트리에 하나의 (PID, p)만 기록할 수 있어, 여러 프로세스가 동일한 프레임을 공유하는 '공유 메모리' 구현이 복잡해진다.

- **📢 섹션 요약 비유**: "책의 두께는 얇아졌지만, 특정 내용을 찾으려면 첫 페이지부터 끝까지 다 읽어봐야 하는 비효율이 생기는 것과 같습니다."

+++

## Ⅳ. 성능 최적화 전략: 해시와 IPT (Strategy)

### 1. 해시 앵커 테이블 (Hash Anchor Table) 연동
- IPT의 탐색 속도 문제를 해결하기 위해, 앞에 해시 테이블을 두어 (PID, p)를 해싱한 결과로 IPT의 후보 인덱스를 즉시 가리키게 한다.
- 이를 통해 평균 탐색 시간을 $O(1)$에 가깝게 유지한다.

### 2. TLB의 역할
- IPT 환경에서도 TLB는 필수적이다. 자주 사용하는 (PID, p) -> Frame 매핑을 캐싱하여 IPT 접근 자체를 최소화한다.

- **📢 섹션 요약 비유**: "출석부 전체를 뒤지는 대신, '이름 초성'별로 모아둔 인덱스 페이지(해시 테이블)를 먼저 보고 바로 해당 페이지로 넘어가는 최적화입니다."

+++

## Ⅴ. 결론 및 요약

### 1. 핵심 요약
- 역 페이지 테이블은 **'물리적 자원(Frame)' 중심**의 관리 체계로, 메모리 절약이라는 목표를 위해 탐색 성능과 구현의 복잡성을 타협한 모델이다.

### 2. 시스템적 의의
- 64비트 이상의 광활한 가상 주소 공간에서 페이지 테이블이 메모리를 집어삼키는 현상을 방지할 수 있는 가장 강력한 솔루션 중 하나다.

- **📢 섹션 요약 비유**: "결국 '소유자 위주'가 아닌 '장소 위주'로 장부를 정리함으로써, 방대한 명단을 슬림하게 다이어트하는 데 성공한 방식입니다."

### 📌 관련 개념 맵 (Knowledge Graph)
- **페이지 테이블 (Page Table)**: 전통적인 프로세스 중심의 매핑 방식.
- **해시 페이지 테이블 (Hashed Page Table)**: IPT의 탐색 성능을 보완하기 위해 자주 결합되는 기술.
- **공유 메모리 (Shared Memory)**: IPT 구조에서 구현 시 가장 까다로운 부분.
- **PID (Process Identifier)**: IPT에서 서로 다른 프로세스의 주소를 구분하는 핵심 키.
