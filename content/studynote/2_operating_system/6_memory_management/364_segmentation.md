# 세그멘테이션 (Segmentation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 세그멘테이션(Segmentation)은 메모리를 일정한 크기가 아닌, 코드, 데이터, 스택 등 사용자가 인식하는 '논리적 단위(Segment)'로 나누어 관리하는 기법이다.
> 2. **가치**: 프로그램의 논리적 구조를 그대로 반영하므로 메모리 보호(Protection)와 공유(Sharing)가 매우 직관적이고 용이하다는 장점이 있다.
> 3. **한계**: 세그먼트들의 크기가 서로 다르기 때문에, 메모리 할당과 해제가 반복되면 물리 메모리 사이에 작은 빈 공간이 생기는 '외부 단편화(External Fragmentation)' 문제가 발생한다.

+++

## Ⅰ. 세그멘테이션 (Segmentation)의 정의 및 필요성

### 1. 정의
- **세그멘테이션 (Segmentation)**은 프로세스를 물리적인 크기가 아닌 **논리적 의미**를 가진 세그먼트들로 나누어 메모리에 배치하는 방식이다. 각 세그먼트는 고유한 번호와 길이를 가진다.

### 2. 사용자 관점의 메모리 (User's View)
- 프로그래머는 프로그램을 "연속된 바이트의 덩어리"가 아니라, "메인 함수, 서브루틴, 전역 변수 영역, 스택 영역" 등의 집합으로 이해한다. 세그멘테이션은 이 관점을 하드웨어적으로 지원한다.

### 3. 세그먼트의 예시
- Code Segment (명령어)
- Data Segment (전역 변수)
- Stack Segment (로컬 변수, 복귀 주소)
- Symbol Table, Library 함수 등

- **📢 섹션 요약 비유**: "책을 무조건 100페이지씩 자르는 것(페이징)이 아니라, '1장 서론', '2장 본론'처럼 내용의 흐름(논리적 단위)에 따라 장을 나누는 것과 같습니다."

+++

## Ⅱ. 세그멘테이션의 작동 원리 및 ASCII 다이어그램

### 1. 논리 주소의 구성
- 세그멘테이션 환경에서 논리 주소는 **<Segment-number, Offset>**의 쌍으로 표현된다.
- 페이징과 달리 세그먼트마다 크기가 다르므로, 오프셋의 최대값이 세그먼트마다 다르다.

### 2. ASCII: 세그멘테이션 하드웨어 구조
```text
[ Logical Address ]
< Segment # (s) | Offset (d) >
          |           |
          v           |
   +---------------+  |
   | Segment Table |  |
   +---------------+  |      (d < Limit?)
   | s |Limit|Base |  |      /        \
   +---|-----|-----+  |    YES        NO (Trap: Addressing Error)
       |     |        |     |
       +--------------+-----+
             |
             v
      +--------------+
      | Physical Addr| (Base + d)
      +--------------+
```

### 3. 주소 변환 과정
1. CPU가 논리 주소 <s, d>를 제시한다.
2. 세그먼트 테이블에서 s번 엔트리를 찾아 해당 세그먼트의 **Base(시작 물리 주소)**와 **Limit(길이)** 정보를 가져온다.
3. 오프셋 **d**가 **Limit**보다 작은지 검사한다. (보호 기능)
4. 유효하다면 **Base + d**를 통해 물리 주소에 접근한다.

- **📢 섹션 요약 비유**: "백화점 안내도에서 '3층(s)의 20m 지점(d)'을 찾을 때, 먼저 3층이 어디서 시작하는지(Base) 확인하고 3층의 전체 길이(Limit)를 넘지 않는지 체크하는 것과 같습니다."

+++

## Ⅲ. 세그멘테이션의 기술적 특징: 보호와 공유

### 1. 보호 (Protection)
- 세그먼트 단위로 권한을 부여하기 쉽다.
- 예: Code 세그먼트는 'Read-only', Data 세그먼트는 'Read-Write'로 설정하여 잘못된 메모리 접근을 논리적 단계에서 차단한다.

### 2. 공유 (Sharing)
- 여러 프로세스가 공통으로 사용하는 라이브러리 함수 등을 하나의 세그먼트로 만들어 공유하기 편리하다. 페이징보다 공유 단위가 명확하기 때문이다.

- **📢 섹션 요약 비유**: "거실(공유 세그먼트)은 누구나 들어올 수 있지만, 침실(보호 세그먼트)은 주인만 들어갈 수 있게 구역별로 규칙을 정하는 것과 같습니다."

+++

## Ⅳ. 세그멘테이션의 문제점: 외부 단편화 (Strategy)

### 1. 가변 크기 할당의 폐해
- 세그먼트들은 크기가 제각각이므로, 메모리에 넣고 빼는 과정에서 물리 메모리 곳곳에 자투리 공간(Hole)이 남는다.
- 전체 빈 공간의 합은 충분하더라도, 정작 필요한 세그먼트를 넣을 만큼 연속된 공간이 없어 할당이 불가능한 '외부 단편화'가 발생한다.

### 2. 해결책: Compaction 또는 Paging 연동
- 메모리 압축(Compaction)을 통해 공간을 모으거나, 세그먼트를 다시 페이지로 쪼개는 'Paged Segmentation' 기법을 사용한다.

- **📢 섹션 요약 비유**: "주차장에 차 크기대로 칸을 그려두면, 작은 차들이 빠져나간 자리에 큰 차가 못 들어가는 낭비가 생기는 것과 같습니다."

+++

## Ⅴ. 결론 및 요약

### 1. 핵심 요약
- 세그멘테이션은 **사용자의 논리적 의도**를 메모리 관리에 반영한 기법으로, 보호와 공유에 강점이 있지만 외부 단편화라는 물리적 한계를 갖는다.

### 2. 시스템적 의의
- 순수 세그멘테이션만 사용하는 현대 시스템은 드물지만, 운영체제가 프로세스의 영역(Text, Data, Stack)을 구분하는 철학적 기반이 되었다.

- **📢 섹션 요약 비유**: "결국 '의미 있는 덩어리로 나눈다'는 개념은 프로그래밍의 효율성을 높여주지만, '빈틈없이 채워 넣기'에는 페이징보다 까다로운 방식입니다."

### 📌 관련 개념 맵 (Knowledge Graph)
- **세그먼트 테이블 (Segment Table)**: 세그멘테이션 주소 변환의 핵심 도구.
- **페이징 (Paging)**: 세그멘테이션과 대비되는 고정 크기 할당 방식.
- **외부 단편화 (External Fragmentation)**: 세그멘테이션의 가장 큰 고질병.
- **세그멘테이션 기반 페이징 (Paged Segmentation)**: 두 방식의 장점만 합친 하이브리드 기법.
