# 세그멘테이션 (Segmentation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 세그멘테이션 (Segmentation)은 프로그램을 사용자 관점의 가변 크기 논리 단위(코드·스택·힙·데이터 세그먼트)로 분할하여, 각 세그먼트를 독립적으로 관리·보호·공유하는 메모리 관리 기법이다.
> 2. **가치**: 페이징이 페이지 크기로 강제 분할하는 것과 달리, 세그멘테이션은 프로그램의 논리적 구조(함수, 모듈, 객체)에 맞게 분할하여 세그먼트 단위 접근 보호(읽기 전용 코드, 실행 불가 스택)와 공유(read-only 코드 세그먼트 공유)가 자연스럽다.
> 3. **융합**: x86 아키텍처는 역사적으로 세그멘테이션+페이징을 혼합한 페이지드 세그멘테이션(Paged Segmentation)을 사용하였으나, x86-64 (64비트)에서는 세그멘테이션이 사실상 무력화(모든 세그먼트 베이스 0)되고 페이징으로 수렴하였다.

---

## Ⅰ. 개요 및 필요성

프로그래머 관점에서 프로그램은 하나의 연속 주소 공간이 아니라 함수들의 집합, 배열, 스택 등 논리적 단위들의 모음이다. 세그멘테이션은 이런 논리적 분류를 하드웨어 수준에서 직접 지원한다.

**💡 비유**: 교과서를 장(Chapter)으로 나누는 것처럼 — 1장(코드 세그먼트), 2장(데이터), 3장(스택). 각 장은 크기가 다르지만 논리적으로 의미 있는 단위.

```text
┌──────────────────────────────────────────────────────────────┐
│         세그멘테이션 주소 변환 구조                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  논리 주소: [세그먼트 번호 s | 오프셋 d]                    │
│                                                              │
│  세그먼트 테이블[s]:                                        │
│  ┌─────────────┬──────────────┐                             │
│  │ Base (기준) │ Limit (한계) │                             │
│  └─────────────┴──────────────┘                             │
│                                                              │
│  변환 과정:                                                  │
│  1. d < Limit 검사 → 위반 시 Segmentation Fault!           │
│  2. 물리 주소 = Base + d                                    │
│                                                              │
│  예시:                                                       │
│  세그먼트 2(데이터), 오프셋 500                              │
│  세그먼트 테이블[2] = { Base: 0x4000, Limit: 1000 }        │
│  500 < 1000 ✅ → 물리 주소 = 0x4000 + 500 = 0x41F4         │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: 세그멘테이션은 도시 구역 지도 — 주거지역(코드), 상업지역(데이터), 주차장(스택)으로 각각 다른 크기로 구분하고, 각 구역마다 다른 규칙(접근 권한)을 적용합니다.

## Ⅱ. 아키텍처 및 핵심 원리

### 세그멘테이션 특성

| 특성 | 세그멘테이션 | 페이징 |
|:---|:---|:---|
| 분할 기준 | 논리적 의미 (가변 크기) | 고정 크기 (4KB) |
| 외부 단편화 | 발생 | 없음 |
| 내부 단편화 | 없음 | 존재 |
| 공유 | 세그먼트 단위 자연스러운 공유 | 페이지 단위 |
| 보호 | 세그먼트별 읽기/쓰기/실행 비트 | 페이지별 |

### 세그멘테이션 기반 메모리 보호

각 세그먼트 테이블 엔트리에 보호 비트를 설정:
- 코드 세그먼트: **읽기+실행**, 쓰기 불가 (Read-Only)
- 데이터 세그먼트: 읽기+쓰기, 실행 불가
- 스택 세그먼트: 읽기+쓰기, 실행 불가 (DEP/NX 비트)

이 메커니즘이 현대 보안 기능인 **NX (No-Execute) 비트**와 **W^X (Write XOR Execute)** 정책의 기반이다.

**📢 섹션 요약 비유**: 세그먼트 보호 비트는 도서관 구역별 규칙 — 열람실(코드)은 읽기만, 사무실(데이터)은 쓰기 가능, 어느 구역도 불 놓기(임의 실행)는 금지!

---

## Ⅲ. 융합 비교 및 다각도 분석

### x86의 세그멘테이션 역사

- **16비트 x86**: 세그먼트 레지스터(CS, DS, SS, ES)로 1MB 메모리 접근 확장
- **32비트 x86 (Protected Mode)**: 세그멘테이션 + 페이징 혼합. GDT/LDT 세그먼트 디스크립터
- **64비트 x86-64**: CS·SS 제외 모든 세그먼트 베이스=0, Limit=무한. **사실상 세그멘테이션 무력화**, 페이징만 사용

**📢 섹션 요약 비유**: x86-64에서 세그멘테이션은 장식품 — 예전엔 중요했지만 이제는 페이징이 모든 보호와 주소 변환을 담당합니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단

**실무 시나리오**:
1. **ELF 바이너리 세그먼트**: Linux의 실행 파일 형식 ELF는 `.text(코드)`, `.data(초기화 데이터)`, `.bss(미초기화 데이터)` 세그먼트로 구성. OS 로더가 각 세그먼트를 페이징 메모리에 매핑.
2. **보안 NX 비트**: x86-64의 페이지 테이블 NX 비트로 스택/힙 실행 불가 → 쉘코드 공격 방어.

**안티패턴**:
- **세그멘테이션 폴트 무시**: `SIGSEGV` (Segmentation Fault)는 NULL 포인터 역참조 등 메모리 보호 위반 신호. 억지로 핸들링 말고 근본 원인을 수정.

**📢 섹션 요약 비유**: Segfault는 '출입 금지 구역 침입 경보' — 경보를 꺼버리지 말고 왜 금지 구역에 들어갔는지(버그)를 찾으세요.

## Ⅴ. 기대효과 및 결론

| 구분 | 세그멘테이션 | 페이징 | 페이지드 세그멘테이션 |
|:---|:---|:---|:---|
| 외부 단편화 | 있음 | 없음 | 없음 |
| 내부 단편화 | 없음 | 있음 | 있음 (페이지 단위) |
| 논리적 보호 | 우수 | 보통 | 우수 |
| 현대 OS 채택 | 거의 없음 | 주류 | x86 32비트 |

현대 OS는 페이징 중심으로 수렴했지만, 세그멘테이션의 논리적 분할 개념은 ELF 세그먼트, NX 비트, W^X 정책으로 계속 살아있다.

## 📌 관련 개념 맵

| 개념 | 관계 |
|:---|:---|
| 세그멘테이션 폴트 (Segfault) | 세그먼트 한계(Limit) 초과 시 발생 예외 |
| NX (No-Execute) 비트 | 세그멘테이션 보호 개념의 현대적 구현 |
| ELF 세그먼트 | 프로그램 바이너리를 .text/.data/.bss로 분할 |
| x86 Flat Mode | x86-64에서 세그멘테이션 무력화 상태 |
| 페이지드 세그멘테이션 | 세그멘테이션+페이징 혼합 (Intel x86 32비트) |

## 👶 어린이를 위한 3줄 비유 설명
1. 세그멘테이션은 교과서를 단원별로 나누는 것 — 코드(1단원), 데이터(2단원), 스택(3단원)으로 각자 크기가 달라요.
2. 각 단원마다 규칙이 달라요 — 코드는 읽기만, 데이터는 읽고 쓰기, 모든 단원은 불 놓기(실행) 금지!
3. 요즘 컴퓨터는 이 단원 구분을 페이징이 대신하지만, 단원(세그먼트) 이름은 파일 속에 여전히 남아있어요!
