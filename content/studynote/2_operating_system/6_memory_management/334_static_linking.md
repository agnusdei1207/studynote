# 정적 연결 (Static Linking)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정적 연결(Static Linking)은 컴파일 후 링킹 단계에서 프로그램이 사용하는 모든 라이브러리 코드를 실행 파일에 직접 복사하여 포함하는 방식이다.
> 2. **가치**: 외부 라이브러리에 대한 의존성 없이 독립적으로 실행 가능한 바이너리를 생성하며, 실행 시 별도의 연결 과정이 없어 동적 연결보다 호출 속도가 빠르다.
> 3. **융합**: 고성능이 요구되는 시스템 루틴이나, 실행 환경이 제한되어 라이브러리 공유가 어려운 임베디드 시스템 등에서 여전히 핵심적인 기법으로 사용된다.

+++

## Ⅰ. 정적 연결 (Static Linking)의 개념 및 배경

### 1. 정의
- **정적 연결 (Static Linking)**이란 링커(Linker)가 프로그램의 목적 파일(Object File)들과 필요한 라이브러리 파일(`.lib`, `.a`)들을 하나로 묶어, 모든 코드가 포함된 단일 실행 파일을 생성하는 과정이다.

### 2. 등장 배경: '모듈화된 개발의 통합'
- 프로그래밍의 규모가 커지면서 모든 코드를 한 파일에 짤 수 없게 되었고, 기능별로 분리된 모듈을 합치는 과정이 필요해졌다. 초기 컴퓨터 환경에서는 실행 시점에 무언가를 연결할 여유가 없었기에, 빌드 시점에 모든 준비를 마치는 정적 연결이 자연스럽게 표준이 되었다.

### 3. 💡 비유: '일체형 밀키트'
- 정적 연결은 요리에 필요한 모든 재료(Library)와 도구를 상자 하나에 다 담아둔 **'밀키트'**와 같다. 요리할 때 따로 마트에 갈 필요가 없어 편리하지만, 상자가 매우 크고 무거워진다는 단점이 있다.

- **📢 섹션 요약 비유**: "필요한 모든 장비를 가방에 다 챙겨서 떠나는 완벽주의자의 여행 준비입니다."

+++

## Ⅱ. 정적 연결의 작동 원리 (Deep Dive)

### 1. 링킹(Linking) 단계의 처리
- 컴파일러가 소스 코드를 목적 파일로 변환하면, 링커가 호출된 함수의 이름(Symbol)을 확인한다.
- 링커는 라이브러리 파일에서 해당 함수를 찾아 기계어 코드를 추출하고, 실행 파일의 적절한 위치에 삽입한다.
- 호출부의 주소를 상대 주소나 절대 주소로 확정(Resolve)하여 실행 시점에 추가 연산이 필요 없게 만든다.

### 2. ASCII 다이어그램: 빌드 프로세스
```text
 [ Source A ]    [ Source B ]      [ Static Lib ]
      │               │                  │
      ▼               ▼                  │
 [ Compiler ]    [ Compiler ]            │
      │               │                  │
      ▼               ▼                  │
 [ Object A ]    [ Object B ]            │
      │               │                  │
      └───────────────┴────────┬─────────┘
                               ▼
                        [ Linker (Static) ]
                               │
                               ▼
                      [ Single Executable ]
                      (All codes included)
```

### 3. 기술적 상세 해설
- **심볼 해석 (Symbol Resolution)**: 중복된 함수 이름이 있는지 확인하고 각 식별자에 고유한 주소를 할당한다.
- **재배치 (Relocation)**: 개별 모듈 내에서의 상대 주소를 전체 실행 파일 기준의 주소로 재조정한다.

- **📢 섹션 요약 비유**: "여러 권의 낱권 책을 한 권의 두꺼운 백과사전으로 제본하는 과정과 같습니다."

## Ⅲ. 주요 특징 및 장단점 분석

### 1. 비교 분석
| 구분 | 내용 |
|:---|:---|
| **장점** | 실행 시 의존성 문제가 없음(Portable), 실행 속도가 가장 빠름, 배포가 단순함. |
| **단점** | 실행 파일 크기가 큼, 라이브러리 수정 시 전체를 재컴파일/재링크해야 함, 메모리 낭비. |

### 2. 기술적 특이점
- **메모리 중복**: 똑같은 정적 라이브러리를 쓰는 프로그램 10개가 실행되면, 메모리에는 해당 라이브러리 코드가 10번 복사되어 올라가게 된다.

- **📢 섹션 요약 비유**: "속도는 광속이지만, 짐이 너무 많아 집이 좁아지는 결과를 초래합니다."

+++

## Ⅳ. 실무 적용 및 시스템 장애 분석 (Strategy)

### 1. 실제 사례
- **임베디드 시스템**: OS의 지원이 미비하거나 메모리 구조가 단순한 환경에서 신뢰성을 위해 사용한다.
- **Go / Rust**: 현대 언어들 중에서도 의존성 복잡성을 피하기 위해 기본적으로 정적 연결(또는 그에 준하는 방식)을 선호하는 경우가 많다.

### 2. 문제 해결 전략
- **선택적 포함(Selective Inclusion)**: 링커의 최적화 기능을 사용하여 라이브러리 전체가 아닌, 실제로 사용되는 함수(Dead Code Elimination)만 골라 담아 파일 크기를 줄인다.

- **📢 섹션 요약 비유**: "가방이 너무 무거우면 꼭 필요한 물건만 골라 담는 '미니멀리즘 링킹'이 답입니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 기대효과
- **독립성 확보**: 외부 환경 변화(라이브러리 삭제/업데이트)와 무관하게 프로그램이 항상 동일하게 작동함을 보장한다.
- **보안성**: 외부 라이브러리 인젝션 등의 공격으로부터 상대적으로 안전하다.

### 2. 결론 및 향후 전망
- 동적 연결의 편리함에 밀려 한때 구식 기법으로 취급받기도 했으나, 최근 컨테이너 기술(Docker)과 단일 바이너리 배포 트렌드가 확산되면서 정적 연결의 가치가 다시 재조명받고 있다.

- **📢 섹션 요약 비유**: "가장 고전적이지만, 가장 믿음직한 연결 방식입니다."
