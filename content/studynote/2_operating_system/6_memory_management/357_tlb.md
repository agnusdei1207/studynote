# TLB (Translation Lookaside Buffer)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TLB (Translation Lookaside Buffer)는 페이지 테이블(Page Table)의 최근 변환 결과를 캐싱하는 하드웨어 연관 메모리 (Associative Memory)로, 모든 메모리 접근 시 필요한 논리→물리 주소 변환 속도를 드라마틱하게 가속한다.
> 2. **가치**: TLB 적중률이 99% 수준일 때 실질 메모리 접근 시간(EAT)은 페이지 테이블 직접 참조(200ns) 대비 절반 이하(~102ns)로 줄어들어, 현대 CPU가 DRAM 속도 병목 없이 동작하는 핵심 이유가 된다.
> 3. **융합**: ASID (Address Space Identifier)로 컨텍스트 스위치 시 TLB 전체 플러시를 회피하고, 멀티레벨 TLB (L1 iTLB/dTLB, L2 sTLB), 거대 페이지 (Huge Page) 기반 TLB 엔트리 효율화, 세큐어 부팅 ASID 격리가 TLB의 진화를 이끈다.

---

## Ⅰ. 개요 및 필요성

모든 메모리 접근(명령어 패치, 데이터 읽기/쓰기)에는 논리 주소 → 물리 주소 변환이 필요하다. 페이지 테이블은 DRAM에 저장되므로 매번 참조하면 메모리 접근이 2배 느려진다.

TLB는 CPU 내부(L1 캐시보다 더 가까운 위치)에 있는 고속 연관 메모리(CAM)로, 최근 페이지 변환 결과 수십~수백 개를 저장한다. 지역성 원리(Locality of Reference) 덕분에 대부분의 접근이 TLB 히트로 처리된다.

**💡 비유**: TLB는 자주 전화하는 사람 번호를 외우는 것 — 매번 전화번호부(페이지 테이블)를 뒤지지 않아도 머릿속(TLB)에서 바로 찾을 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│         TLB 포함 주소 변환 흐름                             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  CPU ──[논리 주소]──▶ TLB 탐색                              │
│                          │                                   │
│                ┌─────────┴──────────┐                        │
│            TLB Hit                TLB Miss                   │
│         (99%+ 확률)              (1% 미만)                  │
│                │                      │                      │
│          물리 주소 즉시               페이지 테이블 (DRAM)  │
│          (1~2 클럭)                  ──[프레임 번호]──▶     │
│                │                      TLB 갱신              │
│                │                          │                  │
│                └──────────┬───────────────┘                  │
│                           ▼                                  │
│                     물리 메모리 접근                         │
│                                                              │
│  EAT = α(t+m) + (1-α)(t+2m)                                │
│  α=0.99, t=1ns, m=100ns: EAT ≈ 102ns                       │
│  (TLB 없으면: t+2m = 201ns)                                 │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: TLB는 단골 배달 기사의 단골집 주소 암기 — 새 손님은 GPS(페이지 테이블)가 필요하지만, 단골은 바로 출발합니다.

## Ⅱ. 아키텍처 및 핵심 원리

### TLB 구조 — 완전 연관 vs 집합 연관

| 항목 | 완전 연관 TLB | 집합 연관 TLB |
|:---|:---|:---|
| 검색 방식 | 모든 엔트리 병렬 탐색 | 특정 집합만 탐색 |
| 속도 | 매우 빠름 | 빠름 |
| 크기 | 작음 (수십 엔트리) | 상대적으로 큼 |
| 구현 비용 | 높음 | 낮음 |
| 적용 | L1 TLB | L2 sTLB |

### ASID (Address Space Identifier)

컨텍스트 스위치(프로세스 전환) 시 페이지 테이블이 바뀌므로 TLB 내 이전 프로세스 엔트리는 무효화해야 한다.

- **ASID 없는 경우**: 컨텍스트 스위치마다 TLB 전체 플러시 → 대규모 TLB 미스 급증
- **ASID 있는 경우**: TLB 엔트리에 ASID 태그 → 프로세스 A와 B 엔트리가 TLB에 공존 → 컨텍스트 스위치 비용 최소화

```text
┌──────────────────────────────────────────────────────────────┐
│         ASID 기반 TLB 엔트리 구조                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  TLB 엔트리: [ASID | 가상 페이지 번호 | 물리 프레임 번호]  │
│                                                              │
│  프로세스 A (ASID=1) TLB 엔트리: [1 | 0x100 | 프레임 5]   │
│  프로세스 B (ASID=2) TLB 엔트리: [2 | 0x100 | 프레임 12]  │
│                                                              │
│  → 두 엔트리가 같은 가상 페이지(0x100)지만 ASID가 다름    │
│  → 컨텍스트 스위치 시 ASID 레지스터만 바꾸면 됨            │
│  → TLB 플러시 없이 two 프로세스 엔트리 공존               │
│                                                              │
│  ASID 포화 시: ASID 범위 초과 프로세스 등장 시 플러시 필요 │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: ASID는 사원증 번호 — TLB 엔트리마다 어느 직원(프로세스) 것인지 표시해서 퇴근(컨텍스트 스위치)해도 다른 직원 엔트리를 지우지 않아도 됩니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 거대 페이지 (Huge Page)와 TLB 효율

| 페이지 크기 | 1GB 데이터 커버 TLB 필요 수 | TLB 미스율 |
|:---|:---|:---|
| 4KB (standard) | 262,144 엔트리 | 높음 |
| 2MB (Huge Page) | 512 엔트리 | 크게 감소 |
| 1GB (Giga Page) | 1 엔트리 | 최소 |

대규모 메모리 접근(DB 버퍼 풀, JVM 힙 등)에서 Huge Page는 TLB 미스율을 극적으로 줄여준다.

**📢 섹션 요약 비유**: 거대 페이지는 큰 지도 한 장 — 작은 지도(4KB) 수백 장이 필요한 지역을 큰 지도(2MB) 한 장으로 커버해서 지도 탐색(TLB 참조) 횟수를 줄입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

**실무 시나리오**:
1. **대규모 Java 서비스**: JVM 힙 16GB에 4KB 페이지 사용 시 수백만 TLB 미스/초 발생. Transparent Huge Pages (THP) 활성화로 TLB 미스 50% 이상 감소.
2. **Database 버퍼 풀**: Oracle, MySQL의 버퍼 풀에 Huge Page 적용 → 쿼리 레이턴시 10~30% 단축.

**안티패턴**:
- **THP 자동 적용** (일부 워크로드): 스캔 위주 배치 처리에서 THP의 메모리 할당 지연(khugepaged 컴팩션)이 역효과. 워크로드별 검증 필요.

**📢 섹션 요약 비유**: Huge Page는 만능이 아니에요 — 복잡한 도시 지도(단편화 메모리)에선 큰 지도 한 장이 오히려 찾기 어려울 수 있어요.

## Ⅴ. 기대효과 및 결론

| 구분 | TLB 없음 | TLB (99% Hit) | TLB (Huge Page) |
|:---|:---|:---|:---|
| EAT (DRAM 100ns 기준) | ~200ns | ~102ns | ~101ns + 미스율↓↓ |
| 컨텍스트 스위치 비용 | - | 플러시 오버헤드 | ASID로 최소화 |

TLB는 현대 가상 메모리 시스템의 성능 핵심이다. ASID, 멀티레벨 구조, Huge Page와 결합하여 가상-물리 주소 변환 비용을 최소화하고, CPU 전체 메모리 접근 성능을 결정하는 하드웨어다.

## 👶 어린이를 위한 3줄 비유 설명
1. TLB는 자주 가는 집 주소를 외우는 메모리 — 전화번호부(페이지 테이블)를 매번 뒤지지 않아도 바로 찾아가요!
2. 99번은 외운 주소로 가고, 1번만 전화번호부를 찾아보면 돼요 — 그게 99% 적중률이에요.
3. ASID는 사원증 — 여러 직원(프로세스) 주소를 함께 외워도 헷갈리지 않게 이름표(ASID)가 붙어있어요!
