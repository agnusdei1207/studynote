# 적재 시간 바인딩 (Load Time)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 적재 시간 바인딩(Load Time Binding)은 프로그램이 메모리에 실제로 올라가는(Loading) 시점에 로더(Loader)에 의해 물리 주소가 최종 결정되는 방식이다.
> 2. **가치**: 컴파일 시점에 물리 주소를 알 필요가 없어 다양한 메모리 위치에 로드될 수 있는 유연성을 제공하며, **재배치 가능 코드(Relocatable Code)**를 기반으로 작동한다.
> 3. **융합**: 고정 주소 방식의 한계를 극복한 중기적 주소 결정 모델이며, 멀티프로그래밍 환경으로 넘어가는 과도기적 핵심 기술이다.

+++

## Ⅰ. 적재 시간 바인딩 (Load Time Binding)의 개념 및 배경

### 1. 정의
- **적재 시간 바인딩 (Load Time Binding)**이란 컴파일러가 '0'번지부터 시작하는 상대적인 주소를 생성하고, 프로그램이 메모리에 적재되는 순간 로더가 비어 있는 실제 물리 주소를 찾아 주소를 확정하는 방식이다.

### 2. 등장 배경: '위치 독립성'
- 프로그램이 항상 같은 위치에 로드되어야 한다는 제약(Compile Time Binding)을 없애기 위해 등장했다. 메모리의 빈 공간이 유동적인 멀티프로그래밍 초기 환경에서 필수적인 기술이었다.

### 3. 💡 비유: '현장 배정 좌석'
- 티켓에 자리가 적혀 있지 않고, 공연장 입구에서 안내원(로더)이 "지금 10번 자리가 비어있으니 그리로 가세요"라고 알려주는 것과 같다. 손님은 들어가는 순간 자신의 자리를 알게 된다.

- **📢 섹션 요약 비유**: "이사 갈 동네는 정했지만, 실제 몇 호에 살지는 이삿짐 트럭이 단지에 들어설 때 경비원(로더)에게 묻는 방식입니다."

+++

## Ⅱ. 적재 시간 바인딩의 메커니즘 (Deep Dive)

### 1. 재배치 가능 코드 (Relocatable Code)
- 컴파일러는 물리 주소 대신 **재배치 가능 코드**를 생성한다. 모든 주소 참조는 '시작점으로부터의 거리(Offset)'로 기록된다.

### 2. ASCII 적재 시간 바인딩 프로세스
```text
    [ Object File ]       [ Loader ]       [ Main Memory ]
    ---------------      ----------       ---------------
    ADDR 0: JMP 100  --+              +-> ADDR 5000: JMP 5100
    ADDR 100: ...      | (Base: 5000) |   ADDR 5100: ...
                       +--------------+
    
    * 계산식: 물리 주소 = 베이스 주소(5000) + 상대 주소(100)
    * 확정 시점: 메모리에 적재(Loading)되는 바로 그 순간.
```

### 3. 기술적 상세 해설: 로더의 역할
- 로더(Loader)는 메모리의 현재 사용 상황을 체크하고, 비어 있는 연속된 공간의 시작 주소(Base Address)를 파악한다. 그 후 코드 내의 모든 상대 주소에 이 베이스 주소를 더해주는 '주소 보정' 작업을 수행한 뒤 메모리에 올린다.

- **📢 섹션 요약 비유**: "건물의 층수(상대 주소)는 변하지 않지만, 실제 해발 고도(물리 주소)는 건물이 세워진 지반 높이(베이스)에 따라 결정되는 것과 같습니다."

## Ⅲ. 성능 및 운영 특성 분석

### 1. 특징 비교

| 구분 | 적재 시간 바인딩 (Load Time) |
|:---|:---|
| **생성 주체** | 로더 (Loader) |
| **코드 형태** | 재배치 가능 코드 (Relocatable Code) |
| **적재 속도** | 주소 계산 오버헤드로 인해 다소 느림 |
| **실행 중 이동** | 불가능 (한번 로드되면 고정됨) |

### 2. 한계점: 스와핑(Swapping)의 제약
- 메모리에 적재되는 시점에 주소가 고정되어 버리므로, 실행 도중에 프로그램이 메모리 밖으로 나갔다가 다시 들어올 때(Swapping) 반드시 **원래 있던 그 자리**로만 돌아와야 한다. 만약 원래 자리에 다른 프로그램이 들어왔다면 실행이 불가능해진다.

- **📢 섹션 요약 비유**: "입구에서 자리를 안내받아 앉은 후에는, 화장실에 다녀와도 반드시 내 자리에만 다시 앉아야 하는 규칙과 같습니다."

+++

## Ⅳ. 실무 적용 및 잔존 영역 (Strategy)

### 1. 초기 멀티프로그래밍 시스템
- 가상 메모리나 MMU 하드웨어가 부족했던 시절의 OS(예: 초기 UNIX 등)에서 여러 프로그램을 동시에 올리기 위해 널리 사용되었다.

### 2. 정적 링크된 라이브러리
- 현대 시스템에서도 실행 파일이 로드될 때 함께 로드되는 정적 라이브러리들의 배치 과정에서 이 원리가 부분적으로 활용된다.

- **📢 섹션 요약 비유**: "자유로운 자리 이동은 안 되지만, 적어도 매번 공연 때마다 다른 자리에 앉을 수는 있는 최소한의 자유입니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 기대효과
- **메모리 활용 유연성**: 컴파일을 다시 하지 않고도 메모리의 빈 곳 어디에나 프로그램을 배치할 수 있다.
- **다중 실행**: 동일한 프로그램을 여러 번 실행할 때 각각 서로 다른 메모리 위치에 올릴 수 있다.

### 2. 결론 및 향후 전망
- 적재 시간 바인딩은 '컴파일 타임'의 경직성을 깨뜨린 중요한 진보였지만, 실행 중 주소를 바꿀 수 없다는 한계로 인해 결국 '실행 시간 바인딩'에 자리를 내주게 되었다. 그러나 주소 재배치(Relocation)라는 핵심 개념은 현대 컴퓨팅의 근간으로 남아 있다.

- **📢 섹션 요약 비유**: "고정된 뿌리를 내리기 직전, 어디에 씨를 뿌릴지 결정할 수 있는 찰나의 순간입니다."
