# 스왑 아웃 (Swap out) / 스왑 인 (Swap in)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스왑 아웃(Swap-out)은 메모리에서 디스크로 프로세스를 내보내는 과정이며, 스왑 인(Swap-in)은 디스크에서 메모리로 프로세스를 다시 불러들이는 과정이다.
> 2. **가치**: 이 두 과정의 유기적인 결합을 통해 한정된 물리 메모리의 경계를 넘어 프로세스들이 번갈아 가며 CPU 자원을 점유할 수 있는 가상 환경을 구축한다.
> 3. **융합**: 중기 스케줄러의 결정에 따라 수행되며, 문맥 교환(Context Switch)의 일부로서 막대한 디스크 I/O 비용을 발생시키는 '비싼 연산'이다.

+++

## Ⅰ. 스왑 아웃/인의 개념 및 배경

### 1. 정의
- **스왑 아웃 (Swap-out)**: 현재 실행 중이지 않거나 우선순위가 낮은 프로세스를 주 기억장치(RAM)에서 보조 기억장치(Backing Store)의 스왑 영역으로 이동시키는 동작.
- **스왑 인 (Swap-in)**: 실행이 필요한 프로세스를 스왑 영역에서 주 기억장치로 다시 적재하는 동작.

### 2. 등장 배경: '활동의 일시 정지와 재개'
- 시스템에 프로세스가 너무 많아지면 메모리 부족으로 인해 새로운 프로세스를 시작할 수 없게 된다. 이때 잠시 쉬고 있는 프로세스를 '잠시 냉동(Swap-out)'시켰다가 필요할 때 '해동(Swap-in)'시키는 메커니즘이 고안되었다.

### 3. 💡 비유: '백화점 매장의 시즌 오프'
- 백화점 매장(메모리)은 한정되어 있다. 겨울이 지나면 겨울옷을 **'창고(Backing Store)'**로 보내고(Swap-out), 여름이 오면 창고에 있던 여름옷을 다시 매장에 진열한다(Swap-in).

- **📢 섹션 요약 비유**: "자리가 없으면 잠시 창고로 보내고, 순서가 되면 다시 매장으로 꺼내오는 물류 시스템입니다."

## Ⅱ. 스왑 아웃/인의 작동 원리 (Deep Dive)

### 1. 전이 프로세스
1.  **Swap-out 결정**: OS는 메모리가 부족하거나 특정 프로세스가 오래 대기 중일 때 이를 대상으로 선정한다.
2.  **데이터 복사**: 프로세스의 전체 메모리 이미지(코드, 데이터, 스택 등)를 스왑 파티션에 기록한다.
3.  **상태 전이**: 프로세스의 상태를 'Suspended Ready' 또는 'Suspended Blocked'로 변경한다.
4.  **Swap-in 복구**: 스케줄러가 해당 프로세스를 선택하면, 디스크에서 이미지를 읽어 메모리의 빈 주소에 적재하고 실행을 재개한다.

### 2. ASCII 다이어그램: 전송 오버헤드 구조
```text
  [ Memory ]                     [ Backing Store ]
 +----------+                   +------------------+
 | Proc A   | --- (Swap-out) ---▶ [ Slot A ]       |
 | 100 MB   |    (Disk Write)   |                  |
 +----------+                   |                  |
 |          |                   |                  |
 | (Empty)  | ◀--- (Swap-in) --- [ Slot B ]       |
 +----------+    (Disk Read)    | Proc B (100 MB)  |
                                +------------------+
 
 * 소요 시간 = (데이터 크기 / 디스크 전송 속도) + 탐색 시간
```

### 3. 기술적 상세 해설
- **I/O 대기 문제**: Swap-out 대상 프로세스가 현재 입출력(I/O) 작업을 수행 중이라면, 스와핑을 지연시켜야 한다. 디스크에서 메모리로 데이터를 쓰고 있는데 메모리 주인이 바뀌어버리면 엉뚱한 곳에 데이터가 써지는 오류가 발생하기 때문이다.

- **📢 섹션 요약 비유**: "짐을 싸서 나가는 동안에도 배달시킨 택배(I/O)가 오고 있다면, 택배를 받을 때까지는 이사를 갈 수 없는 것과 같습니다."

## Ⅲ. 주요 특징 및 분석

### 1. 성능 분석 (Transfer Time)
- 스와핑 성능의 대부분은 **전송 시간(Transfer Time)**이 차지한다. 예를 들어, 100MB 프로세스를 50MB/s 속도의 디스크로 스와핑하려면 아웃/인 각각 2초씩, 총 4초가 소요된다. 이는 CPU 입장에서 엄청난 시간 낭비다.

### 2. 기술적 특이점
- **우선순위 스와핑 (Roll-out / Roll-in)**: 우선순위가 낮은 프로세스를 내보내고(Roll-out), 높은 프로세스를 들여오는(Roll-in) 특수한 형태의 스와핑이다.

- **📢 섹션 요약 비유**: "VIP 손님이 오면 일반 손님을 잠시 밖으로 안내하고 자리를 마련해드리는 것과 같습니다."

+++

## Ⅳ. 실무 적용 및 장애 분석 (Strategy)

### 1. 실제 사례
- **Windows 가상 메모리 관리**: `pagefile.sys`를 통해 스왑 인/아웃을 처리한다.
- **Linux OOM Killer**: 스왑 영역마저 가득 차서 더 이상 스왑 아웃할 공간이 없으면, 운영체제는 가장 덜 중요한 프로세스를 강제 종료(Kill)하여 메모리를 확보한다.

### 2. 문제 해결 전략
- **스왑 영역 최적화**: 스왑 영역을 디스크의 가장 빠른 구간(바깥쪽 트랙)에 배치하거나, SSD를 사용하여 전송 시간을 획기적으로 단축시킨다.

- **📢 섹션 요약 비유**: "창고가 너무 멀면 물건을 나르다 하루가 다 가니, 창고를 매장 바로 옆에 짓는 전략입니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 기대효과
- **메모리 과할당(Overcommitment) 지원**: 물리적 한계를 초과하는 수의 프로세스를 관리할 수 있게 한다.
- **안정성 유지**: 급격한 메모리 사용량 증가 시에도 시스템이 즉시 멈추지 않고 스와핑을 통해 버틸 수 있는 완충 역할을 한다.

### 2. 결론 및 향후 전망
- 현대 시스템은 메모리가 매우 커졌지만, 여전히 스왑 인/아웃은 필수적이다. 특히 컨테이너나 가상머신(VM) 환경에서는 자원 할당의 유연성을 위해 이 메커니즘을 적극 활용하고 있다.

- **📢 섹션 요약 비유**: "버리고 채우는 순환이 있어야, 정체되지 않고 흐르는 메모리 생태계가 유지됩니다."
