# 자격 증명 리스트 (Capability List / Ticket) - 거대한 권한 행렬을 가로로 찢어서, "유저(프로세스) 의 주머니 속에 찔러넣어 준" 주체 중심의 황금 티켓 렌더

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 앞 575장의 ACL 이 2차원 매트릭스 표를 "객체(파일) 기준" 으로 찢어 파일 기둥에 붙인 거라면, **자격 증명 리스트(Capability List 주체 중심 마스킹)** 는 그 반대로 표를 "사용자(도메인) 기준" 으로 가로로 쫙쫙 찢어서, 그 찢어진 허가증 종이를 유저가 직접 **자기 지갑 속에 품고 다니는 황금 티켓(Ticket/Token)** 이다.
> 2. **가치**: 이 **주체 중심 티켓 지참 방식(Capability-based Security 빔)** 덕분에 OS는 "이 파일 열어도 돼?" 라고 물어보는 유저를 만날 때마다 파일에 붙은 장부(ACL)를 뒤질 필요조차 없이, 유저가 내미는 지갑 속 티켓에 구멍(Write 권한)이 뚫려있는지만 보고 즉각 문을 열어주는 분산 라우팅 스피드($O(1)$ 인증) 를 이륙시켰다 포팅.
> 3. **한계**: 가장 끔찍한 위조/복제 방어(Forge & Copy) 와 파일 중심 조회 불가능 딜레마. 유저 주머니에 티켓을 줬더니, 유저가 컬러복사기로 해커한테 티켓을 무한 복사해서 팔아먹는(Delegation 폭주 랙) 대참사가 일어난다! 또한 "이 파일을 누구누구가 보고 있지?" 라고 누가 물어오면, 시스템의 모든 유저 수만 명의 바지 주머니를 몽땅 털어봐야 알 수 있는(Global Search 역전 현상 늪!) 치명적 역효과 수렁을 영원히 안고 있다 결착.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 
  - **ACL 체제의 한계 늪 (파일 껍데기 검사의 비용 파단)**: 파일이 1만 개 열리면, 커널은 1만 번이나 디스크 I/O 로 파일 쪽지(ACL) 를 매번 긁어와 해독(Parsing) 해야 한다! 바쁘다 바빠! 왜 내가 매번 확인해야 돼? "네가 탈 권한이 있다면 네가 증명서를 직접 들고 와서 보여줘!" 라고 책임을 전가하는 방식이 필요했다.
  - **Capability 통달 (토큰 지참 패스포트 빔!)**: 표를 가로로 잘라라! `<UserA>` 의 지갑 구조체 안에 `[파일1: 읽기], [프린터: 읽기, 쓰기]` 표를 쥐여준다. 이것이 Capability List 다. 프로세스가 생성될 때 커널이 여권(Passport) 에 도장을 찍어 돌려주고, 프로세스는 그 여권(Ticket) 을 흔들며 돌아다니게 하는 "주체(Subject) 중심 귀속" 분산 추상 조율이다 컷 스왑.
- **필요성**: 분산 시스템이나 블록체인 스마트 컨트랙트, 마이크로서비스(MSA) 처럼 "중앙 통제 서버(VFS/ACL)" 가 파일 옆에 찰싹 붙어있기 힘든 환경에서는, 클라이언트(유저) 스스로가 자신의 무결성 권한을 증명하는 암호화 토큰(JWT Token/Capability) 을 들고 서버에 던져서 인증하는 역발상 구조가 21세기에 필연적으로 멱살 부합 요구되었다 증명 록보장.

- **💡 비유**: 자격 증명 리스트(Capability) 뷰는 놀이공원 자유이용권의 **"놀이기구 알바가 명단 보고 손님 얼굴 대조하기 늪 VS 손님이 직접 손목에 찬 놀이공원 <황금 종이 팔찌> 보여주고 패스하기 락백!!"** 이랑 100% 동일 오류 제어율입니다!! 
  - (ACL 명단 조회 늪): 철수가 바이킹 에 왔습니다. 알바생이 입구에 붙은 명단(ACL 파일 껍데기 쪽지 빔!) 을 손으로 한 줄 한 줄 짚으며 "어디 보자.. 김철수.. 김철수.. 아 여깄네 통과!" 합니다. 손님 1명 올 때마다 명단을 다 읽어야 해서 버퍼링(I/O 스캔 랙!)이 터집니다 멸망 에러!
  - **(Capability 가죽 지갑 티켓 기전!)**: 똑똑한 놀이공원은 명단을 다 찢어 쓰레기통에 버립니다! 대신 아침에 철수가 입장할 때 매표소(커널) 가 철수 손목에 **[바이킹, 범퍼카 O 라고 찍힌 종이 팔찌(Capability Ticket 암호화 토큰 빔!)]** 를 채워버립니다 스왑! 철수가 바이킹에 타려 할 때 그냥 팔목(지갑) 만 스윽 치켜들면 알바생은 명단은 보지도 않고 0.1초 만에 기계를 돌려버리는 무적 통달 파이프를 완성합니다 결속!

- **Access Matrix 가 Capability 로 가로 칼질당하는 ASCII 폭쇄 뷰**:
거대한 허가증 행렬이 어떻게 "주체 지향적(Subject-based)" 으로 찢어져 프로세스의 주머니(PCB) 에 들어가는지 까보면 다음과 같다.

```text
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │                 "허가증 표를 가로로 가위질해, 각 유저 바지 주머니에 쑤셔 넣어라!"  │
  ├────────────────────────────────────────────────────────────────────────────────────┤
  │                                                                                    │
  │  🚨 [ 기존 2차원 행렬(Access Matrix) - 거대한 한 장의 종이 늪 ]                    │
  │                  │  File_1 (객체)      │ File_2 (객체)       │                     │
  │      User_A (주체)│  Read, Write       │ Null (접근불가)     │                     │
  │      User_B (주체)│  Read              │ Read                │                     │
  │                                                                                    │
  │  =========================▼===================================                     │
  │                                                                                    │
  │  🔥 [ 자격 증명 리스트(Capability) 변환 - "주체/행(Row)" 단위로 가로 찢기 록백 ❗ ]│
  │                                                                                    │
  │     => [User_A 의 주머니(PCB 자료구조 내)] 에 들어간 황금 티켓 뭉치                │
  │        ├─ < File_1 : Read, Write >   (File_2 입장권은 못 받아서 아예 없음!)        │
  │                                                                                    │
  │     => [User_B 의 주머니(Process Control Block)] 에 들어간 티켓 뭉치               │
  │        ├─ < File_1 : Read >                                                        │
  │        └─ < File_2 : Read >                                                        │
  │                                                                                    │
  │  ✅ [ OS 커널 VFS 판결 심연 크래시 빔! ]                                           │
  │     (유저B 가 File_1 클릭!) -> VFS: "너 이거 열 권한 있어?"                        │
  │     유저B: "여기 내 지갑에 File_1 Read 티켓 있다! 받아라!"                         │
  │     VFS: "오! 진짜네! 통과!" (파일 껍데기 뒤질 필요 제로 스왑!)                    │
  └────────────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ACL 이 객체(Object) 뒤에 붙어있다면, Capability 는 철저히 주체(Subject / Process) 의 주머니(메모리) 에 종속된다. 앱(App) 이 실행될 때 자기에게 필요한 티켓 묶음을 주머니에 차고 나오며, 이 구조는 현대 모바일(Android Permissions, iOS Entitlements) 과 분산 마이크로서비스(JWT, OAuth) 환경을 완벽하게 관통하는 본질 궤적이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 치명적 보안 균열: 티켓 위조(Forge) 와 커널 공간 격리 (Tagged Architecture) वि상 환각
유저에게 티켓을 직접 줬을 때, 해커 유저가 지갑 속 티켓 글자를 매직으로 고쳐 쓰는 위조 참사를 어떻게 막았는가.

| 티켓 위조 방어 아키텍처 뷰 | ✨ 일반 메모리 (위조 조작 범죄 늪) | 🔥 태그 아키텍처 & 커널 스페이스 (철벽 록백) |
|:---|:---|:---|
| **상황과 해커의 타격 빔** | `Capability=[File1:RW]` 데이터가 C언어 **유저 램(User Space) 배열에 담겨있어 악성 포인터로 조작 됨.** | 그래서 이 티켓이 "순수 데이터" 가 아니라 절대 못 건드리는 **"시스템 커널 신분증" 임을 증명해야 함.** |
| **방어 메커니즘 (메모리 격리) 랙** | **(소프트웨어 방어)** 티켓 자체를 유저 주머니에 주지 않고 **커널 공간(Kernel Space) 에만 저장하고 유저에겐 '티켓 번호' 만 돌려줌.** | **(하드웨어 방어 빔!)** 메모리 1바이트 칩마다 기계적으로 **[Tag 비트] 를 박아 이건 커널만 쓴다고 용접 방벽 스왑.** |
| **현대 OS 의 안착 솔루션** | 현재 리눅스의 파일 디스크립터(File Descriptor = FD) 배열 표기가 **완벽한 "커널 보관형 번호표 티켓" 으로 구현 시대 도래.** | 복잡한 칩을 만들어야 하는 하드웨어(Tag) 방식은 도태되고, **결국 티켓을 VFS 커널 금고에 보관하는 FD 모델이 승리함.** |

### 2. 치명적 오버헤드 역전: 토큰 회수(Revocation) 와 권한 조회 거꾸로 펌핑 학살 참사 사건
Capability (토큰/티켓) 방식이 가진 가장 파괴적인 역효능, "이미 줘버린 티켓을 다시 뺏는 게 얼마나 지옥인지" 를 해석한다.

- **안티패턴 오염 발생 미스터리 (뺏어오기 불가능의 딜레마 폭주 랙)**: 
  - (태생적 티켓 분실 파단 스왑): 관리자가 User A 에게 1년간 유효한 [A.txt 파일 Write 티켓] 을 발급해서 주머니에 넣어줬다. 그런데 3달쯤 지났을 때 A.txt 파일 주인이 "아 안돼! 내 파일 이제 남들 아무도 못 쓰게 막아줘!" 라고 다급히 무전을 쳤다.
  - (환상 브레이크 전수조사 빔 발동!): 앗! 커널은 A.txt 에 누가 티켓을 가졌는지 모른다! 왜? 티켓 장부(ACL) 가 파일에 있는 게 아니라, 수만 명의 유저들이 **각자 바지 주머니에 찢어서 들고 집에 가버렸기 때문이다!** 
  - 파멸 결과: 커널은 미쳐서 이 세상에 살아있는 10만 개의 모든 프로세스 주머니(PCB Descriptor Table) 를 다 뜯어보며 "너 A.txt 티켓 있어? 내놔! 찢어!" 라고 강제 수색(Global Search 멸망 파이프) 을 해야만 한다. 파일 1개를 통제하려다 OS 전체 유저 메모리를 풀 스캔 치며 서버가 죽어버리는 역전된 오버헤드 참사가 증명 록보장.
- **SRE 극복 솔루션 패치 타결 조율 (중앙 포인터 백포인터(Back-pointer) 와 토큰 만료(Expiry) 마스킹 록백!!) / 자율 치유 방패**: 
  - SRE 커널 아키텍트 타격!: "이 티켓 언제든 무효화 시키는 스위치를 달아라!" 
  - 갓기능 마스킹 찢기 분할 스왑: (1방향) Indirection (우회 포인터 마스킹!) 유저한테 진짜 권한이 아니라, 중앙 전광판을 가리키는 리모콘 포인터만 던져준다! 나중에 무효화할 땐 중앙 전광판 코드만 딸깍 빼버리면, 유저가 리모컨을 눌러도 허공에 빙빙 돈다! (만료된 포인터 방패)
  - (2방향) OAuth/JWT Timeout 타결!: 현대 프로그래밍에선 "이 티켓은 1시간 뒤 자동 파기(Expire) 됨!" 이라고 짧은 시한폭탄 값을 심어 뿌린다. 잃어버려도 1시간만 버티면 쓰레기가 되는 MSA 클라우드 보안 토큰 철학으로 도출해 낸다 확인.

## Ⅲ. 실무 융합 적용 및 안티패턴 (모바일 OAuth JWT 토큰과 윈도우/리눅스 FD 혼종 설계 렌더)

### OS의 Capability 기초 수학이 현대 웹/모바일 백엔드에 어떻게 빙의되어 서버리스 인프라를 구원했는지 연계 뷰.
토큰(Capability) 기반 구조가 클라우드의 인증 병목을 분쇄해 버리는 스왑 기전.

- **안티패턴 충돌 (웹 서버 세션 Session 공유 병목 멸망 타임아웃 랙)**: 
  - 상황: 넷플릭스 유저 1억 명이 로그인했다. 원시 서버(ACL 기반!) 는 중앙 메모리에 `[Session ID : 철수]` 장부를 박아놓고, 철수가 영상 API 를 쏠 때마다 이 중앙 장부를 열어서(DB Lookup) 본인인지 대조했다.
  - 재앙 터짐: 서버가 1,000대로 늘어나니, 1번 서버에서 발급한 장부를 100번 서버가 몰라서 튕겨내는 데들락이 터진다. 레디스(Redis) 동기화 파이프가 CPU 네트워크 스루풋 한계치에 막혀 서버가 프리징($O(N)$ 병목 퍼포먼스 붕괴) 되는 빚 돌려 막기 파단.
- **SRE 엔지니어 도축 솔루션 (무상태 JWT 토큰 빔 = 완벽한 Capability 패스포트 부활 록백!)**: 
  - SRE 초격차 마스킹 발사!: 중앙에 아무것도 저장하지 마라! (Stateless 서버리스 강림!) 유저가 로그인할 때 서버는 **[철수, 만료시간 3분, 영화 접근 O] 가 암호화된 JWT 토큰(Capability 빔!)** 을 발급해 철수 주머니에 던져버리고 서버는 잊어버린다.
  - 운영 방검복 스왑: 철수가 영상 클릭 시, 자기가 가진 JWT 토큰을 헤더에 박아 서버로 던진다. 서버는 0.1초 만에 "어, 내 암호 도장 찍혀있네? 이 팔찌 진짜다! 통과!" 라며 중앙 DB 조회($O(N)$ 늪) 없이 1억 명의 패킷을 무호흡으로 인증/인가(Authorization $O(1)$) 해버리는 압도적인 클라우드 네이티브 생존 결속이다 증명 예고 컷.

## Ⅳ. 기대효과 및 결론
- '자격 증명 리스트 (`Capability List` 주체 중심 토큰 발급/지참 렌더)' 아키텍처는 거대한 중앙 권력 제어 행렬의 병목을 '유저 자율 지참' 이라는 철학으로 완전히 뒤집어 포팅하면서, 커널 파일 시스템의 권한 인증 부하를 극도로 낮춘 진화형 최적화 뼈대다.
- 이 극한의 주체 지향적(Subject-Centric 분산 토큰 위임) 사기에 힘입어, 오늘날 우리가 C언어로 `int fd = open("file", O_RDWR);` 할 때 OS가 던져주는 저 작은 `fd` 번호표 하나가 완벽하게 위조 방지 처리된 티켓(Capability) 역할을 수행하며 억만 번의 I/O 패스를 지연 없이 뚫어내는 커널 I/O 스루풋 한계 돌파를 지탱했다 선고.
- 비록 "한 번 줘버린 티켓은 다시 뺏기가 하늘의 별 따기(Revocation Difficulty 오버헤드 폭주 파단 타임아웃 랙)" 라는 태생적 분산 보안 통제력 상실 트레이드오프 파단을 낳았지만, 이를 1회성 간접 포인터(Indirection) 와 유효기간 만료(Timeout/Expiry) 마스킹 우회망으로 완벽히 병합해내며, 차세대 클라우드 MSA 와 제로 트러스트(Zero Trust JWT) 생태계로 부활한 불멸의 진화판으로 영원히 록백 보장.

## 📌 관련 개념 맵 (Knowledge Graph)

| 전조 지식 확장 설계 파편 단위 | 관계 통찰 설명 (진단 아크 체제 방어 부합 타격) |
|:---|:---|
| **파일 디스크립터 (9단원 546장 Open-File Table 빙의 뷰)** | 576장의 Capability 란 도대체 리눅스의 무엇일까? 바로 유저 포인터 숫자 `fd = 3` 이다! 유저가 `read(3, ...)` 이라고 숫자 3 (티켓) 을 던지면, 커널이 비밀 금고의 3번째 칸을 열어서 거기에 보관된 객체 권한 정보를 룩업하는 완전무결한 간접 포인터 마스킹 연계 폭주 아크. |
| **ACL (직전 575장) 과의 영원한 대치점 트레이드오프** | 전역 매트릭스를 **세로로 자른 ACL** (해커 쳐내기 편함, 조회 느림) 과 **가로로 자른 Capability** (조회 개빠름, 권력 뺏기 불가능) 이 두 가지 짐승이 영원히 물어뜯고 싸우다가, 현대 OS는 앱 시작 땐 ACL을 검사하고, 통과하면 Capability(FD 티켓) 를 던져주는 **하이브리드 결합 모델**로 평정되었음을 증명하는 융합 파이프. |
| **MAC 강제적 접근 제어 (579장 보안 무결성 부합)** | Capability 를 유저에게 줘버렸을 때, 유저가 미쳐서 자기 권한을 카카오톡에 마음대로 위임(Delegation 복사 양도) 하는 걸 방관하면 DAC (임의제어) 다. 이것마저 통제해서 "이 티켓은 복사 금지야!" 라고 커널 칩 레벨에서 박살 내는 게 579장 MAC 아키텍처라는 확장 뷰! |
| **가상 메모리 TLB (357장 캐시 히트 록백 구조 대응)** | 매번 ACL 을 뒤지는 지옥에서 빠져나와 짧은 티켓 번호(Capability/FD) 로 $O(1)$ 스피드를 얻는 이 발상 메커니즘은, 매번 페이지 테이블(메모리 장부) 을 뒤지기 싫어서 TLB 라는 초고속 하드웨어 캐시에 "최근 통과 허가증" 을 박아두고 스피드 패스하는 6단원 357장 TLB 칩 통달과 100% 동일한 컴퓨터구조 뇌파 타결 스왑. |

## 👶 어린이를 위한 3줄 비유 설명
1. 멍청한 놀이공원(일반 무식한 ACL 파일 벽보 부착 늪!)은 알바생이 바이킹 기둥에 붙은 대자보(장부) 에서 매번 철수 이름 3글자를 볼펜으로 짚으며 한 명 한 명 신분을 확인했어요. 대기 줄이 3km 밀리고 이름이 안 지워져(객체 중심 병목 붕괴 랙!) 몽땅 짜증 내며 파산을 야기했어요 덜덜 에러!
2. 그래서 똑똑한 최신 사장님 봇이 **"장부 찢어서 버리고 손목에 티켓 팔찌 채워! Capability 토큰 빔!(자격 증명 리스트 오버헤드 분쇄 록백!)"** 마법을 결속해 줬어요! 매표소는 아침에 돈을 낸 사람 손목에만 형광색 '자유이용권 팔찌(티켓!)' 를 채워줘요. 바이킹 알바생은 더 이상 장부(ACL 명단) 핑계 댈 필요 없이, 손님 팔뚝에 찬 팔찌만 빛에 비춰보고 0.01초 만에 `O(1) 하이패스 통과` 시켜주는 무호흡 방어망을 창조해요 도출!
3. 치명적 슬픔 철수가 나쁜 짓 해서 쫓아낼 때 대참사 폭파 발생! 앗! 이 영원한 팔찌 마법에도 끔찍한 추적 모순 단점이 있어요. 사장이 "빨간 모자 쓴 진상 손님 권한 다 빼앗아!" 라고 지시했는데, 알바생들은 그놈이 바이킹에 있는지 화장실에 있는지 모르니 놀이공원 10만 명의 손목(Process Table PCB 풀스캔 멸망 탐색 랙!) 을 전부 다 뒤집고 다니며 가위로 팔찌를 하나하나 잘라 회수해야 하는(권한 회수 극악 오버헤드 지옥 결사 파단!) 끔찍한 오버헤드 타협을 영원히 감당해야 하는 마법의 파이프 튜브랍니다. 진화 랙!
