# 접근 제어 행렬 (Access Matrix) - 주체(행)와 객체(열) 교차점마다 꽂혀 있는 OS 권한 심판의 절대 십자포화 명부

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 앞선 장에서 배운 "보호 도메인(권한의 묶음 집합)" 을 진짜로 커널 데이터베이스에 기록하려면, 수학적인 표가 하나 필요하다. 그것이 바로 **"가로줄/행(Row) 에 도메인(주체) 들을 쫙 깔고, 세로줄/열(Column) 에 파일/프린터(객체) 들을 쫙 깐 다음, 서로 교차하는 눈금 십자점 칸에 '읽기/쓰기/실행' 이라는 텍스트를 적어 넣는(Access Matrix 마스킹 렌더)"** 추상적 권한 엑셀 기판 구조다.
> 2. **가치**: 이 **2차원 행렬 매핑(Subject-Object Intersect 빔)** 덕분에 OS는 어떤 앱(D1) 이 파일(F2) 에 손을 뻗을 때 막연히 헷갈리지 않고, $(D1, F2)$ 좌표만 딱 찍어서 그 칸에 `W(쓰기)` 글자가 있는지 없는지를 $O(1)$ 레이턴시로 직관적 룩업(Lookup) 차단해 버리는 거시 통치 잣대를 이륙시켰다 포팅.
> 3. **한계**: 가장 끔찍한 성긴 행렬(Sparse Matrix) 딜레마. 현업 서버의 도메인이 10만 명이고 파일 개수가 1억 개라면, 이 행렬 칸은 10조 개(10 Tio칸 메모리) 가 된다! 그런데 대부분의 유저는 남의 파일에 아예 접근 못 하니 10조 칸 중 99.9%가 비어있는 빈칸(Null 늪)이다. 이 미친 "빈칸에 낭비되는 RAM 메모리 압살 폭주" 를 견딜 수가 없어, 결국 이 통짜 행렬을 가로로 찢거나 세로로 찢어버리는 트레이드오프 수렁(ACL vs Capability 분기)을 영원히 안고 있다 결착.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 
  - **뒤죽박죽 권한 늪 (Rule Spaghetti 멸망 파단)**: 파일 보호 원시 시절, "A유저는 B를 읽는다", "프린터는 C만 쓴다" 는 등 if-else 코드가 커널 여기저기 소스에 흩뿌려져(Hard-coded) 있었다. 권한 관리가 난장판이고 해커가 뚫고 들어와도 뭘 뚫렸는지 감사(Audit) 가 불가능한 혼돈의 늪.
  - **Access Matrix 통달 (2차원 엑셀 표의 직관적 빔!)**: 수학적 모델링의 압살 방패! 모든 운영체제 권한을 단 한 장의 시트 구도로 압축한다. **Rows = 도메인들**, **Columns = 자원 객체들**. `Matrix[i][j] = 도메인 i가 객체 j에 행사할 수 있는 권한 집합`. 유저 계정끼리 도메인을 바꿀 수 있는가(Switch) 통제 여부도 이 행렬 표 안에 `Switch` 라는 권한 글자로 그냥 쓱 끼워 넣어 완벽히 포섭해버리는 추상 조율 컷 스왑.
- **필요성**: 보안 관리자(Sysadmin) 가 시스템 전체의 권력 구조를 한눈에 모니터링하고 증명(Mathematical Validation) 해야 했다. 누가 누구 파일에 접근할 수 있는지 1장의 지도로 펴보기 위해, 복잡한 로직을 전부 걷어내고 순수한 "주체-객체 표" 라는 메타데이터 아키텍처가 21세기에 필연적으로 멱살 부합 요구되었다 증명 록보장.

- **💡 비유**: 접근 제어 행렬(Access Matrix 심판 엑셀) 뷰는 놀이공원 자유이용권의 **"각 놀이기구 표 받는 사람이 일일이 누구 들어올지 외우기 늪 VS 요금소 엑셀 표 딱 1장으로 <홍길동, 롤러코스터, [탑승가능]> 좌표 점찍기 락백!!"** 이랑 100% 동일 오류 제어율입니다!! 
  - (제각각 규칙 늪): 1번 놀이기구, 2번 놀이기구 알바생이 각자 메모장을 들고 "철수는 타게 해줘" 라고 따로 적어두면 일관성 붕괴(보안 정책 파편화 랙!) 가 일어납니다 멸망 에러!
  - **(Access Matrix 중앙 엑셀 마스킹 기전!)**: 똑똑한 놀이공원은 본사 컴퓨터에 아주 거대한 **[엑셀 시트 행렬 표(Matrix 빔!)]** 를 하나 띄웁니다 스왑! 가로줄은 손님 1번, 2번(도메인). 세로줄은 바이킹, 범퍼카(객체). 철수가 바이킹 타려 시도(Access Request!) 하면 커널 알바생은 `[철수]행, [바이킹]열` 의 크로스 칸을 봅니다. 칸 안에 텅 비었으면(Empty)? 쫓아냅니다! 칸 안에 `[탑승 O]` 라고 적혔으면 통과! 만물이 평등한 무적 통달 파이프를 완성합니다 결속!

- **접근 제어 행렬(Access Matrix) 의 논리 구조 ASCII 폭쇄 뷰**:
도메인끼리의 권력 변경(Switch)과 객체 접근 권한이 단 한 장의 VFS 행렬로 어떻게 우겨들어가는지 렌더 체계를 까보면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────────────────────────────┐
  │                 "인간이든 프린터든 모두 이 행렬 표의 좌표(i, j) 안의 노예일 뿐이다!"        │
  ├─────────────────────────────────────────────────────────────────────────────────────────────┤
  │                                                                                             │
  │  🚨 [ 자원 객체(열) : 파일1, 프린터1, 그리고 '다른 도메인' 마저도 객체다! ]                 │
  │                                                                                             │
  │  =========================▼===================================                              │
  │                                                                                             │
  │  🔥 [ 2차원 매트릭스 (Access Matrix) 데이터베이스 권력표 록백 ❗ ]                          │
  │                                                                                             │
  │    (주체 / Domain행) │ File_1         │ Printer_1  │ Domain_B (타인 구역)                   │
  │    ─────────────────┼────────────────┼────────────┼─────────────────────────────────────────┤
  │    Domain_A (알바)   │ Read           │            │ Switch (권한 강탈)                     │
  │    Domain_B (점장)   │ Read, Write    │ Print      │                                        │
  │    Domain_C (포스기) │ Execute        │ Print      │                                        │
  │                                                                                             │
  │  ✅ [ OS 커널 VFS 판결 심연 크래시 빔! ]                                                    │
  │     1) 알바(Domain_A) 가 프린터1을 뽑으려 함 -> 좌표(A, Printer_1) 룩업!                    │
  │     => VFS 확인: "칸이 완전 비었다(Null)! 프린터 권한 위반 에러 쾅!"                        │
  │                                                                                             │
  │     2) 도메인 환승 마법: 알바(Domain_A) 가 점장(Domain_B) 모드로 변신 시도!                 │
  │     => 좌표(A, Domain_B) 를 룩업! 그 안에 `Switch` 권한이 있다? 오! 점장 옷 입기 성공 스왑! │
  └─────────────────────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 표를 잘 보면 '다른 도메인(Domain_B)' 도 사실상 하나의 '자원 객체 열(열)' 로 취급받아 표에 우겨 들어가 있다. 즉 "내가 저 신분으로 변신할 수 있는가(Switch)" 라는 극강의 링 교환 보안 정책마저 이 단순한 가로세로 표 1장 위에서 완벽하게 증명되고 연산된다. 이것이 바로 커널 보안 수학의 우아함이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 트레이드오프 전선 종결: 수학적 완전성(Matrix) vs 물리적 메모리 초토화(Sparse) વિ상 환각
이 완벽한 매트릭스를 왜 이론서에만 두고 실제 RAM 에 통째로 못 올리는지에 대한 SRE 최전선 타결.

| 접근 제어 엑셀 할당 아키텍처 뷰 | ✨ 논리적 Access Matrix (개념 늪) | 🔥 희소 행렬(Sparse Matrix) 메모리 멸망 파단 |
|:---|:---|:---|
| **설계 완전성과 증명 빔** | 해커가 어디로 들어와서 어딜 갈 수 있는지 **한 장 표로 최단 경로 크로스 룩업 방어 무결성.** | 서버 파일이 1억 개면 표 시트의 길이는 **1억 개의 열(Column) 로 무한 팽창하는 펌핑 데들락.** |
| **빈칸(Null) 에 의한 RAM 오링 학살 랙** | "A유저는 A파일만 쓴다" 가 대부분의 규칙이므로 **행렬 칸의 99%는 아무 글씨 없는 텅 빈 빈칸(Null).** | 이 빈칸의 가로세로 좌표(포인터)를 유지하는 데만 **수십 GB 의 미친 커널 램(RAM) 이 증발하는 파단.** |
| **극복 대안 조율 스왑** | 결국 이 통짜 행렬을 메모리에 통으로 굽는 건 미친 짓으로 **판정되어 산산조각 찢기 결정 록백.** | 열(Column) 기준으로 찢은 게 **ACL**, 행(Row) 기준으로 찢은 게 **Capability 리스트** 시대 도래. |

### 2. 치명적 오버헤드 폭발: 행렬 권한 통치의 동적 갱신(Copy/Owner)과 무한 증식 학살 참사 사건
행렬 표 안에서 새로운 파일을 만들거나 타인에게 권한을 나눠줄 때 벌어지는 수학적 무결성 붕괴 현상을 해석한다.

- **안티패턴 오염 발생 미스터리 (문서를 복사해 줄 때의 권한 복제 트롤링 폭주 랙)**: 
  - (태생적 권한 핑퐁 파단 스왑): 관리자가 파일1을 만드니 행렬의 파일1 열(Column) 이 생태. "나만 Read, Write 해야지." 그런데 동생한테 파일1을 읽게 해주고 싶다. 그럼 행렬 칸에 동생(Domain i) - 파일1(Object j) 칸에 `Read` 라고 쳐준다. 
  - (환상 브레이크 권력 탈취 빔 발동!): 앗! 권한을 넘길 때 `Copy` 플래그(별표 *) 를 붙여서(권한 복사 가능 록백) 줘버렸다! 동생이 미쳐서 동네방네 5만 명한테 싹 다 파일 Read 권한을 복사해서 표를 파탄 내버린다.
  - 파멸 결과: 행렬 안에 `R*` 처럼 복제 전파 가능 마크가 찍히는 순간, 그 권한은 암세포처럼 모든 도메인 칸으로 전염 증식(Cascading Copy) 될 수 있다! 나중에 주인이 "앗! 다 회수해!" 하려고 해도, 10만 칸에 쓰인 이놈의 권력을 전부 찾아서 지워(Revocation 멸망 타임아웃 랙) 야 하는 끔찍한 추적 비용 오버헤드 참사가 실무 VFS 학살의 메인 트리거로 군림했다 증명.
- **SRE 극복 솔루션 패치 타결 조율 (Owner 권력의 하드코딩과 통제권(Control) 제한 록백!!) / 자율 치유 방패**: 
  - SRE 보안 엔지니어 타격!: 무법 `Copy` 를 방지하기 위해 파일은 무조건 `Owner(주권자)` 비트를 행렬 칸에 박아둔다! 
  - 갓기능 거시 스왑 로직: 한 열(하나의 파일) 의 권한 변경(새로운 놈한테 읽기 주거나 뺏거나) 은, 그 행렬 좌표 안에 `Owner` 라는 왕관 마크가 쓰인 놈만 행사하게 한다! 이로써 아무리 사본이 돌아다녀도 통치권은 절대 원작자 도메인을 이탈하지 못하는 다이하드 격리망으로 도출해 낸다 확인.

## Ⅲ. 실무 융합 적용 및 안티패턴 (전역 테이블 Global Table 사상과 연결 리스트 압착 렌더)

### 빈칸 99% 의 10조 칸 엑셀 표를 어떻게 1만 칸으로 접어서 RAM 에 구겨 넣었는지, 전역 테이블 뷰.
리눅스는 이론상의 Access Matrix 를 미련하게 통으로 만들지 않는다.

- **안티패턴 충돌 (빈칸까지 메모리에 그리는 2차원 배열 Array 멸망 타임아웃 랙)**: 
  - 상황: 대학교 컴공 주니어가 OS 권한 행렬을 `char matrix[100만][100만]` 이라는 C언어 2차원 배열로 메모리에 `malloc` 할당했다. 
  - 재앙 터짐: 이 배열 껍데기에만 메모리 1테라바이트(TB) 가 소비되며 서버가 즉시 OOM(Out of Memory) 즉사한다. 1테라 중 권한 텍스트가 쓰인 칸은 고작 4KB뿐이다. 
- **SRE 엔지니어 도축 솔루션 (희소성을 파괴하는 Global Table 튜플 세트 방어 빔!)**: 
  - 선배 SRE 멘토 격파 발사!: 빈칸은 기억하지 마라! 권한이 존재하는 '딱 그 알맹이 교차점 좌표' 만 쏙쏙 빼서 1차원 데이터베이스 목록으로 만든다. (이를 Global Table 튜플 록백이라 한다!)
  - 압착 방검복 스왑: `<도메인A, 파일2, 쓰기>`, `<도메인B, 프린터, 읽기>` ... 딱 이렇게 "권한이 있는 튜플(Tuple 3개조)" 명단만 중앙 링크드 리스트(Linked List) 나 해시 맵으로 이어 붙여 커널에 저장한다. 메모리 포화 1TB 늪에서 순식간에 4KB 영구 생존형 슬림핏으로 서버 메모리를 구원하는 실무 아키텍처 생존 결속이다 증명 예고 컷.

## Ⅳ. 기대효과 및 결론
- '접근 제어 행렬 (`Access Matrix` 거시 좌표계 권력망 렌더)' 아키텍처는 유닉스 50년 역사에서 "누가 누구를 어찌하는가" 에 대한 철학적 보안 질문을 단 하나의 직관적 2차원 십자 좌표계로 치환해, 모든 OS 보안 설계자들에게 불멸의 수학적 기준점 체계 뼈대를 잡아준 위대한 유산이다.
- 비록 이 통짜 매트릭스 자체가 실무 서버의 방대한 객체(1억 개 파일) 앞에서는 메모리 오링을 낳는 희소 배열(Sparse Array) 폭발 트레이드오프 파단을 낳아 실제품에는 거의 탑재되지 않았지만,
- 이 개념의 등장으로 인해 OS 진영은 "그럼 이 표를 파일 쪽으로 잘라 붙일까?(ACL)" 아니면 "유저 주머니에 찢어 넣어줄까?(Capability)" 라는 양대 산맥 구현체(2개의 튜닝 마스킹) 가 탄생하는 원자폭탄급 촉매제로 작용하며 현대 모든 권한 제어 생태계의 어머니 진화판으로 영원히 록백 보장.

| **보호 도메인 Protection Domain (직전 572장 권리증 박스 캡슐화 뷰)** | 573장 행렬 표의 가로줄(Row) 에 놓이는 주인공 이름칸표가 바로 직전 572장에서 배운 "보호 도메인" 이다! 도메인 개념을 수학 장부의 세로 뼈대로 끼워 넣는 완벽한 연계 통달 구조화 파이프. |
| **ACL & 자격 증명 리스트 (직후 575장, 576장 행렬 절단 마스킹 빔)** | 573장 거대 행렬은 뼈대지만 메모리가 터진다! 그래서 이 표를 **세로 1줄(객체 1개)** 씩 쭉쭉 칼로 잘라서 그 파일 껍데기(i-node) 에만 스카치테이프로 붙여놓는 게 ACL (575장) 이고, **가로 1줄(도메인 1명)** 씩 잘라서 유저 주민등록증 뒷면에 붙여주는 게 자격 증명(Capability 576장) 임을 꿰뚫는 극한 세계관 융합 뷰! |
| **RBAC 역할 기반 접근 제어 (577장 다수당 통합 파편 랙 회피)** | 100만 명의 직원이 있으면 행렬 가로줄(도메인) 이 100만 줄이다! 너무 길어서 엑셀이 터지니, 비슷한 1만 명을 하나로 묶어 "회계팀" 이라는 1개의 가로줄(Role 역할 도메인) 로 뭉뚱그려버리면 행렬이 확 줄어드는 마법 연계 단절 아크. |
| **페이징 테이블 페이지 맵핑 (6단원 333장 희소 행렬 다단계 회피 이론)** | 이 '희소 메모리 낭비(Sparse 빈칸 폭발)' 이슈는 어디서 봤다? 6단원의 미친 크기의 통짜 페이지 테이블 낭비 늪! 권한 행렬도 빈칸을 압축하려고 리스트로 찢거나 다단계로 꼬듯이 커널은 모든 희소 데이터 낭비를 똑같은 철학으로 타개해 나가는 쌍둥이 영혼 동기화. |

## 👶 어린이를 위한 3줄 비유 설명
1. 멍청한 놀이공원(일반 무식한 1차원 철판 룰셋 늪!)은 관리자가 100개의 놀이기구마다 뛰어다니며 "저기 파란 모자 꼬마는 키 커서 바이킹 태워! 빨간 모자는 회전목마만!" 하고 귀에 대고 속삭이며 통제했어요. 사람이 1만 명이 오자 머리가 뒤엉켜 파란 모자 꼬마가 절벽 롤러코스터에 타버려(권한 월권 혼돈 시스템 붕괴 랙!) 몽땅 털려 파산을 야기했어요 덜덜 에러!
2. 그래서 똑똑한 최신 아키텍트 건축가 로봇이 **"단 한 장의 요금소 엑셀 표 빔! (Access Matrix 권력 십자선 좌표 록백!)"** 마법을 결속해 줬어요! 엄마는 그냥 거대한 화이트보드를 꺼내서 세로줄엔 손님 이름, 가로줄엔 놀이기구 이름을 딱 긋고, 만나는 십자 크로스 칸마다 동그라미(O) X 를 딱딱 그어놔요! 알바생은 헷갈일 일 없이 그냥 `[손님이름] + [놀이기구]` 좌표 칸만 손으로 찍어보고 O 면 통과시키는 궁극의 1초 방어망을 창조해요 도출!
3. 치명적 슬픔 종이 낭비 쓰레기장 대참사 멍청한 설계 폭발 발생! 앗! 이 영원한 화이트보드 마법에도 끔찍한 모순 단점이 있어요. 대한민국 국민 5천만 명의 이름을 세로줄에 다 적고 전 세계 놀이기구 1억 개를 가로줄에 적었더니, 그 거대한 종이를 놓을 방이 없어서(커널 램 RAM OOM 메모리 터짐 늪!) 건물이 터져버렸어요. 99% 의 국민은 그 놀이공원 안 오니까 칸이 텅텅 빈칸(Null 희소 행렬 버그!) 인데 종이만 버린 거죠! 결국 이 종이를 가로세로로 가위로 잘게 잘라(ACL 분리 아키텍처 연성 스왑!) 나눠 가져야 영원히 살아남는다는 마법의 딜레마 튜브랍니다. 진화 랙!
