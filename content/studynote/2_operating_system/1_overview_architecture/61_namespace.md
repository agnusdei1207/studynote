# 네임스페이스 (Namespace) - 자원 격리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 네임스페이스 (Namespace)는 리눅스 커널이 제공하는 자원 격리 기술로, 프로세스 그룹별로 시스템 자원(PID, Network, Mount 등)에 대해 **독립된 가상 뷰 (Isolated View)**를 제공하는 기능이다.
> 2. **가치**: 동일한 호스트 내에서 서로 다른 프로세스들이 같은 포트 번호나 프로세스 ID를 사용하더라도 충돌 없이 독립적으로 동작하게 함으로써 컨테이너 기술의 '벽'을 형성한다.
> 3. **융합**: 컨트롤 그룹(Cgroups)과 함께 컨테이너 아키텍처를 지탱하는 양대 지주이며, 클라우드 멀티테넌시 환경에서 보안과 자원 독립성을 보장하는 핵심 메커니즘이다.

+++

## Ⅰ. 네임스페이스의 개요

### 1. 정의
- 특정 프로세스나 그 자식 프로세스들이 시스템의 다른 부분으로부터 격리된 자원들을 볼 수 있게 하는 커널 기능이다. 각 네임스페이스 안의 프로세스는 자신이 시스템의 유일한 자원을 점유하고 있다고 착각하게 된다.

### 2. 등장 배경: '모두가 1번 프로세스가 되고 싶어 한다'
- 기존 리눅스 시스템에서는 PID 1번은 시스템당 하나뿐이었다. 하지만 여러 사용자가 자신만의 독립된 환경(컨테이너)을 가지려면, 각자만의 1번 프로세스, 각자만의 네트워크 카드(eth0)가 필요했다. 이를 위해 커널 자원을 논리적으로 쪼개는 네임스페이스 기술이 도입되었다.

### 3. 💡 비유: '연극 무대의 독립된 파티션'
- 네임스페이스는 거대한 무대(호스트 OS) 위에 **'투명한 칸막이'**를 치는 것과 같다.
- A칸의 배우들은 B칸에서 무슨 일이 일어나는지 전혀 모르며, 각자 자기 칸에 있는 소품(자원)이 무대 전체의 소품인 줄 알고 연기에 집중한다.

+++

## Ⅱ. 네임스페이스의 주요 종류 (Deep Dive)

리눅스 커널은 현재 다음과 같은 주요 네임스페이스를 제공한다.

### 1. PID Namespace (Process ID)
- 프로세스 ID를 격리한다. 컨테이너 내부의 프로세스는 호스트와 별개의 PID 체계를 가지며, 컨테이너 안에서 스스로를 **PID 1 (init)**로 인식한다.

### 2. Network Namespace (NET)
- 네트워크 장치, IP 주소, 포트 번호, 라우팅 테이블 등을 격리한다. 이를 통해 동일 호스트 내의 여러 컨테이너가 모두 **80번 포트**를 동시에 열어 웹 서비스를 할 수 있다.

### 3. Mount Namespace (MNT)
- 파일 시스템 마운트 지점을 격리한다. 각 컨테이너는 자신만의 `/`, `/etc`, `/var` 구조를 가지며, 다른 컨테이너의 파일 시스템을 볼 수 없다.

### 4. UTS Namespace (Hostname)
- 호스트 이름과 도메인 이름을 격리한다. 컨테이너마다 서로 다른 `hostname`을 부여할 수 있다.

### 5. IPC Namespace (Inter-Process Communication)
- 세마포어, 메시지 큐 등 프로세스 간 통신 자원을 격리한다.

### 6. User Namespace (USER)
- 사용자 및 그룹 ID를 격리한다. 컨테이너 내부의 `root (UID 0)` 사용자가 호스트에서는 일반 사용자로 매핑되어 보안을 강화한다.

- **📢 섹션 요약 비유**: 주소(Net), 이름(UTS), 가족 관계(PID), 보관함(MNT)을 모두 따로 관리하여 서로 모르는 사람처럼 살게 만드는 개인정보 보호 시스템과 같습니다.

+++

## Ⅲ. 동작 메커니즘 및 시스템 콜

### 1. 핵심 시스템 콜 (API)
- **clone()**: 새로운 프로세스를 생성할 때 네임스페이스 플래그(`CLONE_NEWPID` 등)를 전달하여 격리된 채로 태어나게 한다.
- **unshare()**: 이미 실행 중인 프로세스가 기존 자원 공유를 끊고 새로운 네임스페이스로 독립한다.
- **setns()**: 프로세스가 기존에 존재하는 다른 네임스페이스로 진입한다. (예: `docker exec`)

### 2. ASCII 구조 다이어그램
```text
    [ Host OS Kernel Resources ]
    ┌──────────────────────────────────────────────────┐
    │  Global PID, Net, Mount, UTS, IPC, User          │
    └────────────────────────┬─────────────────────────┘
                             ▼
    ┌────────────────────────┴─────────────────────────┐
    │ [ Namespace 1 ]      [ Namespace 2 ]             │
    │  - PID 1 (web)        - PID 1 (db)               │ ◀── 독립된 뷰
    │  - eth0 (10.0.0.1)    - eth0 (10.0.0.2)          │
    │  - / (Debian FS)      - / (CentOS FS)            │
    └──────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 한 권의 원본 장부를 복사해서 각 부서에 나눠주고, 각자 자기 장부만 보고 일하게 하여 전체 흐름을 모르게 하는 정보 격리 전략과 같습니다.

## Ⅳ. 실무 적용 및 보안 인사이트 (Strategy)

### 1. 실무 시나리오: 포트 충돌 해결
- **상황**: 한 서버에서 Nginx와 Apache 웹 서버를 모두 80번 포트로 띄워야 함.
- **판단**: 두 웹 서버를 각각 별도의 **Network Namespace**를 가진 컨테이너에 담는다. 커널은 두 80번 포트를 별개의 네트워크 스택으로 관리하므로 충돌 없이 동시 서비스가 가능하며, 호스트의 포트 포워딩을 통해 외부와 연결한다.

### 2. 기술사적 인사이트: User Namespace와 보안 강화
- 컨테이너 안에서 `root` 권한으로 실행되는 앱이 해킹되어 컨테이너를 탈출하더라도, **User Namespace**가 적용되어 있다면 호스트 시스템에서는 권한이 없는 '일반 사용자'일 뿐이다. 이는 컨테이너 보안 설계의 핵심인 '최소 권한 원칙'을 물리적으로 구현하는 방법이다.

- **📢 섹션 요약 비유**: 가상 세계의 왕(컨테이너 root)이 현실 세계(호스트)로 넘어오면 평범한 시민(일반 유저)이 되도록 마법을 걸어두는 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **자원 충돌 방지**: 동일한 설정의 앱을 여러 개 동시에 안전하게 실행.
- **보안성 향상**: 프로세스의 시야를 제한하여 인접 컨테이너로의 횡적 이동(Lateral Movement) 차단.

### 2. 미래 전망
최근 네임스페이스는 **Time Namespace** (시스템 시간 격리) 등이 추가되며 더욱 정교해지고 있다. 또한, 네임스페이스 기술을 커널이 아닌 하드웨어 레벨(Enclave)에서 보장하려는 기밀 컴퓨팅(Confidential Computing) 기술과 결합하여, 운영체제조차 믿지 못하는 초고보안 환경으로 발전하고 있다.

- **📢 섹션 요약 비유**: 완벽한 개인 공간을 보장하여 다채로운 삶(다양한 앱)이 평화롭게 공존할 수 있게 돕는 현대 운영체제의 평화 유지군입니다.

2. 1번 안경을 쓰면 1번 장난감만 보이고, 2번 안경을 쓰면 2번 장난감만 보여서 서로 싸울 일이 없죠.
3. 덕분에 수많은 친구가 한 방에서 놀아도, 각자 자기 방에서 혼자 노는 것처럼 편안하게 지낼 수 있답니다!
