# 호스트형 하이퍼바이저 (Type 2)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 호스트형 (Hosted) 하이퍼바이저는 윈도우나 리눅스 같은 **기존 호스트 운영체제(Host OS) 위에 일반 애플리케이션처럼 설치**되어 동작하는 가상화 소프트웨어이다.
> 2. **가치**: 기존 운영체제를 그대로 유지하면서 다양한 이기종 게스트 OS를 손쉽게 실행할 수 있어, 소프트웨어 개발, 테스트, 교육 및 개인용 가상화 환경에 최적화되어 있다.
> 3. **융합**: 호스트 OS의 드라이버와 자원 관리 기능을 활용하므로 설치와 구성이 매우 간편하지만, 호스트 OS를 거치는 이중 구조로 인해 성능 오버헤드가 발생하는 특징이 있다.

+++

## Ⅰ. 호스트형 하이퍼바이저의 개요

### 1. 정의
- 하드웨어가 아닌 '운영체제' 위에서 실행되는 하이퍼바이저이다. 호스트 OS가 하드웨어 자원을 관리하고, 하이퍼바이저는 호스트 OS에게 자원을 요청하여 게스트 OS에게 전달한다.

### 2. 특징: '편의성과 범용성'
- **설치의 용이성**: 일반 프로그램(hwp, chrome 등)을 설치하듯 간단히 설치 가능하다.
- **하드웨어 호환성**: 호스트 OS가 인식하는 모든 하드웨어 장치를 가상 머신에서도 (에뮬레이션을 통해) 사용 가능하다.
- **오버헤드 발생**: 게스트 OS ↔ 하이퍼바이저 ↔ 호스트 OS ↔ 하드웨어라는 다단계 구조로 인해 데이터 전송 지연이 발생한다.

### 3. 💡 비유: '내 집 안에 들여놓은 텐트나 조립식 창고'
- 베어메탈이 아예 건물을 새로 짓는 것이라면, 호스트형은 **'이미 살고 있는 우리 집(호스트 OS) 방 한 칸에 캠핑용 텐트(가상 머신)를 치는 것'**과 같다.
- 거실 화장실이나 부엌(하드웨어 자원)을 쓰려면 집주인(호스트 OS)의 허락을 받아야 하므로 조금 불편할 수 있지만, 필요할 때 바로 펴고 접을 수 있어 매우 편리하다.

+++

## Ⅱ. 호스트형 하이퍼바이저 동작 메커니즘 (Deep Dive)

### 1. 자원 요청 흐름 (I/O Path)
- 게스트 OS가 디스크에 저장을 시도하면, 하이퍼바이저는 이를 가로채어 호스트 OS의 시스템 콜(System Call)로 변환한다. 호스트 OS는 이를 일반 파일 쓰기로 인식하여 실제 디스크에 저장한다. 이 과정에서 발생하는 **문맥 교환 (Context Switch)**이 성능 저하의 주범이다.

### 2. 에뮬레이션 vs 패스스루
- **에뮬레이션 (Emulation)**: 실제 존재하지 않는 하드웨어(예: 구형 사운드 카드)를 소프트웨어로 흉내 내어 게스트 OS에게 보여준다. 호환성은 좋지만 느리다.
- **패스스루 (Passthrough)**: 성능 향상을 위해 특정 하드웨어(예: USB 장치)를 호스트 OS를 거치지 않고 게스트 OS에게 직접 연결해주는 기술이다.

### 3. ASCII 구조 다이어그램
```text
    ┌──────────────────────────────────┐
    │  [ Guest OS 1 ]  [ Guest OS 2 ]  │ ◀── 게스트 운영체제
    ├──────────────────────────────────┤
    │      Hypervisor (Type 2)         │ ◀── 앱 형태로 동작 (VMM)
    ├──────────────────────────────────┤
    │      Host Operating System       │ ◀── 실제 자원 관리자 (Windows, macOS)
    ├──────────────────────────────────┤
    │        Physical Hardware         │ ◀── 실제 기계
    └──────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 중간에 통역사(하이퍼바이저)와 비서(호스트 OS)를 모두 거쳐야 사장님(하드웨어)에게 보고가 전달되는 다단계 보고 체계와 같습니다.

## Ⅲ. 주요 사례 및 비교 분석

### 1. 대표적인 제품군
| 제품명 | 특징 |
|:---|:---|
| **Oracle VirtualBox** | 오픈 소스, 강력한 하드웨어 에뮬레이션, 개인 사용자 필수템 |
| **VMware Workstation** | 높은 안정성, 스냅샷 기능 우수, 유료 기반의 강력한 기능 |
| **VMware Fusion** | macOS 사용자를 위한 윈도우 가상화 솔루션 |
| **Parallels Desktop** | macOS와 윈도우 간의 매끄러운 통합(Coherence 모드) 강점 |

### 2. Type 1 vs Type 2 비교
| 항목 | Type 1 (Bare-metal) | Type 2 (Hosted) |
|:---|:---|:---|
| **설치 위치** | 하드웨어 직접 설치 | OS 위에 앱으로 설치 |
| **주 용도** | 서버, 클라우드 인프라 | 개발, 테스트, 개인 PC |
| **성능** | 최상 (Native에 근접) | 보통 (오버헤드 있음) |
| **관리 난이도** | 높음 (전용 관리 도구 필요) | 낮음 (UI 기반 직관적 관리) |

- **📢 섹션 요약 비유**: 경주 전용 트랙(Type 1)과 일반 도로 위에서 레이싱 게임을 하는 것(Type 2)의 차이입니다.

## Ⅳ. 실무 적용 및 활용 전략 (Strategy)

### 1. 실무 시나리오: 멀티 플랫폼 개발 및 테스트
- **상황**: 윈도우 개발자가 리눅스 환경에서만 돌아가는 서버 코드를 테스트해야 함.
- **판단**: 포맷 후 리눅스를 깔 필요 없이, VirtualBox를 설치하고 그 안에 우분투(Ubuntu)를 올린다. 개발 도구는 윈도우에서 쓰고, 실행은 가상 머신에서 하면서 공유 폴더(Shared Folder) 기능을 통해 소스 코드를 실시간으로 공유한다.

### 2. 기술사적 인사이트: 가상화 탈출 (VM Escape) 공격
- 호스트형 하이퍼바이저는 호스트 OS의 보안 취약점에 그대로 노출된다. 만약 게스트 OS에서 하이퍼바이저의 버그를 이용해 호스트 OS의 권한을 획득하는 **'가상화 탈출'**이 발생하면 전체 시스템이 위험해진다. 따라서 호스트 OS의 최신 보안 패치 유지가 Type 1보다 훨씬 중요하다.

- **📢 섹션 요약 비유**: 텐트(게스트 OS) 안에서 불이 났는데, 그 불이 집 전체(호스트 OS)로 옮겨붙지 않도록 방화 시설(보안 패치)을 잘 갖춰야 하는 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **생산성 향상**: 리부팅 없이 여러 OS를 동시에 사용하며 작업 전환 속도 향상.
- **비용 절감**: 테스트용 물리 장비를 별도로 구매할 필요 없이 기존 PC 활용.

### 2. 미래 전망
최근 윈도우의 **WSL2 (Windows Subsystem for Linux 2)**는 호스트형의 편리함과 베어메탈(KVM 기술 응용)의 성능을 결합한 하이브리드 형태로 진화하고 있다. 결국 호스트형 하이퍼바이저는 성능의 한계를 극복하며 일반 사용자의 컴퓨팅 환경을 더욱 풍성하게 만드는 방향으로 발전할 것이다.

- **📢 섹션 요약 비유**: 한 화면에서 윈도우와 리눅스를 자유롭게 넘나드는 'OS 비빔밥'의 맛있는 비결입니다.
