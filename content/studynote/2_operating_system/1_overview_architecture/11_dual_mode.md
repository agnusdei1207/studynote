# 듀얼 모드 (Dual Mode: User vs Kernel)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 듀얼 모드(Dual Mode)는 하드웨어(CPU) 수준에서 실행 권한을 **사용자 모드(User Mode)**와 **커널 모드(Kernel Mode)**로 분리하여, 운영체제와 핵심 자원을 사용자 프로그램의 위협으로부터 보호하는 하드웨어 기반 보안 아키텍처다.
> 2. **가치**: 응용 프로그램이 하드웨어를 직접 제어하는 치명적인 실수나 악의적인 공격을 원천 차단하며, 오직 검증된 통로인 **시스템 호출(System Call)**을 통해서만 자원 접근을 허용하여 시스템 전체의 안정성을 보장한다.
> 3. **융합**: 가상화 기술에서는 모드 계층이 확장되어 하이퍼바이저 모드(Root Mode) 등이 추가되며, 이는 현대 클라우드 보안과 샌드박싱 기술의 물리적인 토대가 된다.

+++

## Ⅰ. 듀얼 모드 (Dual Mode)의 개요

### 1. 정의
- 운영체제를 보호하고 시스템의 올바른 동작을 보장하기 위해, CPU가 현재 실행 중인 코드의 권한 수준을 구분하여 동작하는 방식이다.

### 2. 배경: "응용 프로그램은 믿을 수 없다"
- 만약 모든 프로그램이 `halt`(시스템 종료)나 `I/O` 명령을 마음대로 내릴 수 있다면, 하나의 버그 섞인 앱이 컴퓨터 전체를 멈추게 할 수 있다. 이를 막기 위해 '관리자'와 '일반인'의 영역을 나눈 것이다.

### 3. 💡 비유: 은행 창구
- **사용자 모드(로비)**: 고객이 자유롭게 걸어 다니고 신청서를 쓸 수 있는 공간. 하지만 금고 안의 돈을 직접 만질 수는 없다.
- **커널 모드(금고 안)**: 은행원만이 들어갈 수 있는 공간. 돈을 꺼내거나 장부를 수정하는 중요한 업무를 처리한다.
- 고객이 돈을 인출하려면 반드시 창구 직원에게 **요청(System Call)**해야 하며, 직원은 권한을 확인한 뒤 대신 업무를 수행해준다.

+++

## Ⅱ. 모드별 특징 및 전환 매커니즘 (ASCII)

CPU 내부의 **모드 비트 (Mode Bit)**가 이 권한을 결정한다.

```text
    [ User Process ] ──────────┐
    (Mode Bit = 1)             │
                               ▼
    [ System Call / Interrupt / Trap ]  <─── 모드 전환 트리거 (Mode Switch)
                               │
    ┌──────────────────────────┴──────────┐
    │        [ Operating System ]        │
    │        (Mode Bit = 0)              │ <─── 특권 명령어(Privileged) 실행 가능
    └──────────────────────────┬──────────┘
                               │
    (Return from System Call)  ▼
    [ User Process Resume ]
    (Mode Bit = 1)
```

### 1. 사용자 모드 (User Mode, Mode Bit = 1)
- **제한된 권한**: 일반적인 응용 프로그램이 실행되는 모드다.
- **명령어**: 산술 연산, 논리 연산 등 시스템에 위해를 가하지 않는 명령어만 수행 가능하다.
- **자원 접근**: 하드웨어(디스크, 네트워크 등)에 직접 접근할 수 없으며, 시도 시 예외(Exception)가 발생한다.

### 2. 커널 모드 (Kernel Mode, Mode Bit = 0)
- **전권 보유**: 운영체제 커널이 실행되는 모드로, '특권 모드', '시스템 모드'라고도 한다.
- **특권 명령어 (Privileged Instruction)**: I/O 제어, 타이머 설정, 인터럽트 관리 등 하드웨어를 제어하는 모든 명령어를 실행할 수 있다.

+++

## Ⅲ. 모드 전환과 특권 명령어

| 구분 | 사용자 모드 (User) | 커널 모드 (Kernel) |
|:---|:---|:---|
| **Mode Bit** | 1 | **0** |
| **실행 주체** | 응용 프로그램 (Excel, Browser 등) | **OS 커널** |
| **명령어 제한** | 일반 명령어만 가능 | **특권 명령어 포함 모든 명령어 가능** |
| **메모리 접근** | 자신에게 할당된 영역만 | **시스템 전체 메모리 영역** |
| **목적** | 사용자 작업 수행 | **자원 관리 및 시스템 보호** |

- **📢 섹션 요약 비유**: 손님(사용자)은 주방(커널)에 들어갈 수 없고, 오직 웨이터(시스템 호출)에게 주문을 통해서만 요리(하드웨어 자원)를 받을 수 있는 안전한 레스토랑 시스템과 같습니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. 시스템 호출(System Call) 오버헤드
- **기술사적 인사이트**: 모드 전환은 CPU 레지스터 저장, 스택 교체 등 **콘텍스트 스위칭**에 준하는 비용이 발생한다. 
- **결단**: 실무에서 고성능 서버 개발 시, 자잘한 I/O를 자주 호출하기보다 데이터를 모아서 한 번에 처리하는 **버퍼링(Buffering)**이나 유저 레벨에서 네트워크를 직접 처리하는 **DPDK** 기술을 고려하는 것은 모드 전환 비용을 줄이기 위한 전략적 선택이다.

### 2. 보안 취약점: 권한 상승 (Privilege Escalation)
- **판단**: 해커의 궁극적 목표는 사용자 모드에서 실행 중인 자신의 코드를 커널 모드 권한으로 실행시키는 것이다. 
- **인사이트**: 커널의 시스템 호출 핸들러에서 입력값을 제대로 검증하지 않으면 커널 모드 권한이 탈취될 수 있으므로, 하드웨어적 보호(SMAP, SMEP)와 소프트웨어적 검증의 이중 방어막이 필수적이다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **시스템 가용성**: 특정 앱의 오류가 전체 시스템 정지로 번지는 현상 원천 차단.
- **무결성 유지**: 운영체제의 핵심 데이터 구조가 사용자로부터 물리적으로 격리됨.

### 2. 미래 전망
최근 가상화와 클라우드 컴퓨팅의 발전으로 듀얼 모드를 넘어 **멀티 레벨 권한(Ring 0 ~ Ring 3)** 구조가 복잡하게 활용되고 있다. 특히 하드웨어 보안 영역인 TEE(Trusted Execution Environment)는 커널조차 접근할 수 없는 '더 깊은 모드'를 제공하며, 이러한 권한 분리의 미학은 미래 보안 아키텍처의 영원한 핵심 원리가 될 것이다.

- **📢 섹션 요약 비유**: 울타리(듀얼 모드)가 튼튼해야 양(응용 프로그램)들이 안전하게 풀을 뜯고, 늑대(악성 코드)가 목장(시스템) 전체를 망치는 일을 막을 수 있습니다.

+++

### 📌 관련 개념 맵 (Knowledge Graph)
- **시스템 호출 (System Call)**: 모드 전환을 일으키는 명시적 통로.
- **특권 명령어 (Privileged Instruction)**: 커널 모드에서만 허용되는 검(Sword).
- **트랩 (Trap) 및 인터럽트**: 비자발적으로 모드 전환을 일으키는 이벤트.

+++

### 👶 어린이를 위한 3줄 비유 설명
1. 듀얼 모드는 컴퓨터 속에 **'안전한 울타리'**를 친 것과 같아요.
2. 우리는 울타리 밖에서 마음껏 놀 수 있지만, 컴퓨터 전체를 망가뜨릴 수 있는 무거운 중장비(하드웨어 제어)는 울타리 안의 관리자 아저씨만 만질 수 있죠.
3. 우리가 중장비가 필요할 때 아저씨에게 부탁하면, 아저씨가 안전한지 확인하고 대신 도와주기 때문에 컴퓨터가 고장 나지 않고 잘 돌아가는 거랍니다!
