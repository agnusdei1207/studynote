# 022. 마이크로커널 (Microkernel)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 최소한의 핵심 기능만 커널 공간에 유지, 서비스는 사용자 공간
> 2. **가치**: 높은 신뢰성, 확장성, 보안성
> 3. **융합**: Mach, Minix, QNX, L4와 연관

---

## Ⅰ. 개요

### 개념 정의

마이크로커널(Microkernel)은 **핵심 기능(스케줄링, IPC, 주소 공간 관리)만 커널 공간에 유지하고, 파일 시스템, 드라이버, 네트워크 스택 등은 사용자 공간 서버로 구현하는 커널 아키텍처**다.

### 💡 비유: 분산 캠퍼스
마이크로커널은 **각 부서가 별도 건물에 있는 대학 캠퍼스**와 같다. 본관(마이크로커널)은 행정만 담당한다. 공학관(파일 시스템), 의대(네트워크), 법대(드라이버)는 독립된 건물에서 운영된다. 한 건물에 불이 나도 다른 건물은 안전하다.

## Ⅱ. 마이크로커널 상세

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                마이크로커널 상세                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【마이크로커널 핵심 기능】                                            │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  기능              설명                                        │ │
│  │  ────              ────                                        │ │
│  │  IPC              메시지 패싱, 포트                             │ │
│  │  스케줄링          기본 CPU 할당만                              │ │
│  │  주소 공간         가상 메모리 관리                             │ │
│  │  스레드            경량 실행 단위                               │ │
│  │  인터럽트          최소 처리만                                  │ │
│  │                                                               │ │
│  │  커널에서 제외: 파일시스템, 드라이버, 네트워크 스택             │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【장점】                                                            │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  1. 신뢰성                                                      │ │
│  │     • 서버 장애가 커널에 영향 없음                              │ │
│  │     • 격리된 주소 공간                                          │ │
│  │     • 개별 서버 재시작 가능                                     │ │
│  │                                                               │ │
│  │  2. 보안                                                        │ │
│  │     • 최소 권한 원칙                                            │ │
│  │     • 서비스 간 격리                                            │ │
│  │     • 공격 표면 최소화                                          │ │
│  │                                                               │ │
│  │  3. 확장성                                                      │ │
│  │     • 새 서비스 추가 용이                                       │ │
│  │     • 기존 서비스 수정 영향 적음                                 │ │
│  │     • 이기종 지원                                               │ │
│  │                                                               │ │
│  │  4. 이식성                                                      │ │
│  │     • 작은 커널, 하드웨어 종속 최소                              │ │
│  │     • 다른 아키텍처로 이식 용이                                  │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【단점】                                                            │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  1. 성능                                                        │ │
│  │     • IPC 오버헤드                                              │ │
│  │     • 컨텍스트 스위치 빈번                                       │ │
│  │     • 사용자-커널 전환 많음                                     │ │
│  │                                                               │ │
│  │  2. 복잡성                                                      │ │
│  │     • 분산 시스템 설계 필요                                      │ │
│  │     • IPC 프로토콜 구현                                         │ │
│  │     • 동기화 어려움                                             │ │
│  │                                                               │ │
│  │  3. 개발 난이도                                                  │ │
│  │     • 서버 간 통신 설계                                         │ │
│  │     • 분산 디버깅 어려움                                        │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### 마이크로커널 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                마이크로커널 분석                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【선택 기준】                                                        │
│  ──────────────────                                                  │
│  마이크로커널 선택:                                                   │
│  • 실시간 보장 필요 (QNX)                                            │
│  • 높은 신뢰성 요구 (seL4)                                           │
│  • 보안이 중요 (격리 필요)                                            │
│  • 이식성이 중요                                                      │
│  • 교육/연구 목적 (Minix)                                            │
│                                                                     │
│  모놀리식 선택:                                                       │
│  • 최고 성능 필요                                                     │
│  • 광범위한 하드웨어 지원                                              │
│  • 풍부한 드라이버                                                    │
│  • 일반 목적                                                          │
│                                                                     │
│  【실제 활용 사례】                                                    │
│  ──────────────────                                                  │
│  • QNX: 자동차 인포테인먼트, 산업 제어                                 │
│  • seL4: 보안 크리티컬 시스템                                         │
│  • Fuchsia: Google Nest Hub                                         │
│  • XNU (macOS): Mach + BSD 하이브리드                                │
│                                                                     │
│  【마이크로커널 vs 모놀리식 성능】                                     │
│  ──────────────────                                                  │
│  IPC 오버헤드:                                                        │
│  • 모놀리식: 함수 호출 ~ns                                            │
│  • 마이크로커널: IPC ~μs (1000배 느림)                                │
│                                                                     │
│  최적화 기법:                                                         │
│  • L4: IPC 최적화로 μs 이하 달성                                      │
│  • 배칭: 여러 요청 묶기                                               │
│  • 캐싱: 자주 사용하는 서비스                                          │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• 마이크로커널: 최소 핵심만 커널 공간, 나머지는 사용자 서버
• 장점: 신뢰성, 보안, 확장성, 이식성
• 단점: IPC 오버헤드, 설계 복잡성, 개발 난이도
• 예시: Mach, Minix, QNX, L4/seL4, Fuchsia
• 적용: 실시간 시스템, 보안 크리티컬, 임베디드
• 선택: 신뢰성 우선 → 마이크로, 성능 우선 → 모놀리식
```
