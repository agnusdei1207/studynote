# 타이머 인터럽트 - 선점형 스케줄링의 기반

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 타이머 인터럽트 (Timer Interrupt)는 하드웨어 타이머가 일정한 주기마다 CPU에 보내는 신호로, 실행 중인 프로세스를 잠시 멈추고 커널의 **스케줄러 코드**를 강제로 실행시키는 하드웨어 기반 제어 메커니즘이다.
> 2. **가치**: 특정 프로세스가 CPU를 무한정 점유(Starvation)하는 것을 방지하여 모든 프로세스가 공평하게 실행 기회를 갖는 **선점형 (Preemptive)** 운영체제를 구현하는 기술적 토대가 된다.
> 3. **융합**: 하드웨어의 클럭 발생 로직과 커널의 인터럽트 핸들러 로직이 결합하여, 응용 프로그램의 협조 없이도 운영체제가 시스템 전체를 통제할 수 있는 권한을 보장한다.

+++

## Ⅰ. 타이머 인터럽트의 개요

### 1. 정의
- 하드웨어 타이머 장치가 미리 설정된 시간 간격이 지날 때마다 CPU의 Interrupt Line에 전기적 신호를 보내어 현재 수행 중인 작업을 중단시키고 커널 모드로의 전환을 유도하는 이벤트이다.

### 2. 왜 선점형의 기반인가?
- 비선점형 OS에서는 프로세스가 스스로 CPU를 반납할 때까지 기다려야 했다. 하지만 선점형 OS는 **타이머 인터럽트**라는 '강제 알람'을 통해 프로세스의 의사와 상관없이 CPU를 회수하여 다른 프로세스에게 넘겨줄 수 있다.

### 3. 💡 비유: '교실의 수업 종료 종소리'
- 학생(프로세스)이 공부(연산)에 너무 집중해서 시간 가는 줄 모르고 있을 때, 학교 종(타이머 인터럽트)이 울리면 선생님(운영체제)이 들어와 다음 시간(다른 프로세스)을 준비하게 하는 것과 같다.
- 종소리가 없다면 공부하고 싶은 학생이 하루 종일 교실을 독차지할 수도 있는 상황을 방지한다.

+++

## Ⅱ. 타이머 인터럽트와 스케줄링 흐름 (Deep Dive)

타이머 인터럽트가 발생했을 때 커널 내부에서 일어나는 과정이다.

### 1. 인터럽트 처리 과정
1. **하드웨어 신호**: 타이머가 CPU에 인터럽트 신호를 보낸다.
2. **Context Save**: CPU는 현재 실행 중이던 프로세스의 상태(PC, 레지스터 등)를 PCB에 저장한다.
3. **ISR 실행**: 인터럽트 벡터 테이블을 참조하여 타이머 핸들러(ISR)를 실행한다.
4. **Time Slice 차감**: 현재 프로세스에게 할당된 시간(Time Quantum)을 1 감소시킨다.
5. **재스케줄링 판단**: 남은 시간이 0이 되면 `need_resched` 플래그를 설정한다.
6. **Context Switch**: 스케줄러가 호출되어 가장 우선순위가 높은 다른 프로세스를 선택하고 CPU를 할당한다.

### 2. ASCII 구조 다이어그램
```text
    [ User Process ]        [ CPU / Hardware ]        [ OS Kernel ]
           │                        │                       │
           │ (Running...)           │                       │
           │                        │                       │
           │◀──────(Signal)──────── [ Timer Device ]        │
           │                        │                       │
    [ State Saved ]  ──────▶ [ Interrupt ] ────────▶ [ Timer ISR ]
           │                        │                (Check Quanta)
           │                        │                       │
           │                        │                [ Scheduler ]
           │                        │                (Pick Next)
           │                        │                       │
    [ New Process ]  ◀──────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 마라톤 선수(프로세스)가 달리고 있을 때, 심판(타이머)이 호각을 불어 강제로 멈춰 세우고 다음 주자에게 바톤을 넘기게 하는 정밀한 경기 운영 시스템과 같습니다.

## Ⅲ. 타이머 인터럽트의 주기와 성능 트레이드오프

타이머 인터럽트가 발생하는 빈도(Hz)는 시스템 성능에 큰 영향을 미친다.

| 주기 (Frequency) | 장점 | 단점 |
|:---|:---|:---|
| **고주파 (예: 1000Hz)** | 응답성이 좋고 스케줄링이 세밀함 | 잦은 문맥 교환으로 오버헤드 증가 |
| **저주파 (예: 100Hz)** | 오버헤드가 적고 처리량(Throughput) 향상 | 사용자 체감 응답 속도가 느려짐 |

- **📢 섹션 요약 비유**: 알람을 1분마다 맞추면 일은 꼼꼼히 챙기겠지만 알람 끄는 데 시간을 다 쓰는 것(오버헤드)과 같고, 1시간마다 맞추면 여유는 있겠지만 중요한 약속을 놓칠 수 있는 것과 같습니다.

## Ⅳ. 실무 적용 및 아키텍처적 가치 (Strategy)

### 1. 실무 시나리오: 무한 루프에 빠진 프로세스 제어
- **상황**: 어떤 개발자가 실수로 `while(1);` 코드를 실행하여 CPU 하나를 100% 점유함.
- **판단**: 타이머 인터럽트가 없다면 이 서버는 해당 CPU 제어권을 영영 잃게 된다. 하지만 타이머 인터럽트가 주기적으로 발생하므로, 운영체제는 무한 루프 중인 프로세스를 잠시 멈추고 관리자 명령을 처리하거나 다른 정상 프로세스들을 실행시켜 시스템 전체의 마비를 막는다.

### 2. 기술사적 인사이트: 실시간성 (Determinism) 보장
- RTOS (Real-Time OS)에서는 타이머 인터럽트 지연 시간(**Interrupt Latency**)을 최소화하는 것이 생명이다. 하드웨어 신호가 왔을 때 얼마나 빨리 핸들러가 실행되느냐가 자동차의 에어백 터지는 속도나 미사일 제어의 정확도를 결정하기 때문이다.

- **📢 섹션 요약 비유**: 불이 났을 때(긴급 작업), 소방차(핸들러)가 교통 체증(인터럽트 지연) 없이 얼마나 빨리 현장에 도착하느냐가 시스템의 생존을 결정합니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **멀티태스킹 구현**: 여러 프로그램이 동시에 돌아가는 것처럼 보이게 하는 마법의 원천.
- **시스템 가용성**: 특정 앱의 결함으로부터 시스템 전체를 보호하는 안정 장치.

### 2. 미래 전망
최근에는 가상화 환경에서 타이머 인터럽트를 효율적으로 전달하기 위한 **Virtual Timer** 기술이 중요해지고 있다. 또한, 전력 효율을 위해 인터럽트를 불필요하게 발생시키지 않는 **Tickless 커널**로의 진화는 타이머 인터럽트가 '주기적'인 것에서 '필요 기반(Event-driven)'인 것으로 변하고 있음을 보여준다.

- **📢 섹션 요약 비유**: 정해진 시간마다 종을 치는 시계탑에서, 할 일이 있을 때만 똑똑하게 깨워주는 스마트 알람으로 진화하고 있습니다.
