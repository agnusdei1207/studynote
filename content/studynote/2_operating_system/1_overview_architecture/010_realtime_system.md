# 010. 실시간 시스템 (Real-time System)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 엄격한 시간 제약(데드라인) 내에 작업 완료를 보장하는 시스템
> 2. **가치**: 안전 크리티컬, 산업 제어, 통신 시스템의 핵심 요구사항
> 3. **융합**: RTOS, 우선순위 스케줄링, 인터럽트 지연과 연관

---

## Ⅰ. 개요

### 개념 정의

실시간 시스템(Real-time System)은 **작업의 논리적 정확성뿐 아니라 지정된 시간 제약(Deadline) 내 완료를 보장해야 하는 시스템**이다.

### 💡 비유: 심장 수술 의사
실시간 시스템은 **심장 수술 의사**와 같다. 수술(작업)은 정확해야 하고, 정해진 시간 내에 완료되어야 한다. 1초 늦어도 환자에게 치명적이다. 비실시간 시스템은 일반 진료와 같다. 천천히 검사해도 큰 문제가 없다.

## Ⅱ. 실시간 시스템 상세

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                실시간 시스템 상세                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【실시간 스케줄링 알고리즘】                                         │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  알고리즘          설명                    특징                  │ │
│  │  ────              ────                    ────                  │ │
│  │  RM                주기 짧을수록 높은 우선순위   정적, 최적      │ │
│  │  (Rate Monotonic)                                               │ │
│  │  EDF               마감 빠를수록 높은 우선순위   동적, 최적      │ │
│  │  (Earliest Deadline)                                            │ │
│  │  DM                마감 짧을수록 높은 우선순위   정적             │ │
│  │  (Deadline Monotonic)                                           │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【RM 스케줄링 분석】                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  스케줄러빌리티 조건 (Liu & Layland):                           │ │
│  │                                                               │ │
│  │  U = Σ(Ci/Ti) ≤ n(2^(1/n) - 1)                               │ │
│  │                                                               │ │
│  │  U: CPU 이용률                                                 │ │
│  │  Ci: 작업 i의 실행 시간                                         │ │
│  │  Ti: 작업 i의 주기                                              │ │
│  │  n: 작업 수                                                    │ │
│  │                                                               │ │
│  │  n=1: U ≤ 100%                                                │ │
│  │  n=2: U ≤ 82.8%                                               │ │
│  │  n→∞: U ≤ 69.3% (ln 2)                                        │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【실시간 시스템 예시】                                               │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  분야              시스템               요구사항                 │ │
│  │  ────              ────               ────────                  │ │
│  │  항공우주          비행 제어            ms 단위 응답             │ │
│  │  자동차            ABS, 에어백          10ms 이내 반응           │ │
│  │  산업 자동화        로봇 제어            주기적 작업 보장         │ │
│  │  통신              스위칭, 라우팅        패킷 처리 데드라인       │ │
│  │  의료              심박 조율기           정확한 타이밍            │ │
│  │  멀티미디어         오디오/비디오        버퍼 언더런 방지         │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실시간 시스템 × 타 과목 융합

### 과목 융합 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                실시간 시스템 × 타 과목 융합                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【운영체제】                                                        │
│  • RTOS: VxWorks, QNX, FreeRTOS, RT-Linux                          │
│  • 선점형 커널: PREEMPT_RT 패치                                      │
│  • 우선순위 상속: 우선순위 역전 방지                                   │
│  • 고해상도 타이머: 나노초 단위                                       │
│                                                                     │
│  【컴퓨터 구조】                                                      │
│  • 인터럽트 지연: 최소화 필수                                         │
│  • 캐시: WCET 분석 고려                                              │
│  • 파이프라인: 분기 예측 영향                                         │
│  • DMA: 예측 가능한 전송                                             │
│                                                                     │
│  【네트워크】                                                        │
│  • TSN (Time-Sensitive Networking): 결정적 이더넷                    │
│  • CAN 버스: 자동차 실시간 통신                                       │
│  • 프로파일: EtherCAT, PROFINET                                      │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  우선순위 역전 (Priority Inversion)                           │ │
│  │  ───────────────────────────────────                          │ │
│  │                                                               │ │
│  │  상황: 저우선순위 작업이 락 보유 → 고우선순위 작업 대기         │ │
│  │                                                               │ │
│  │  해결책:                                                        │ │
│  │  1. Priority Inheritance Protocol (PIP)                       │ │
│  │     → 락 보유자에게 대기자의 우선순위 부여                       │ │
│  │                                                               │ │
│  │  2. Priority Ceiling Protocol (PCP)                           │ │
│  │     → 락 획득 시 천장 우선순위로 상승                           │ │
│  │                                                               │ │
│  │  ┌────────┐      ┌────────┐      ┌────────┐                  │ │
│  │  │ Low P  │──────▶│  Lock  │◀──────│ High P │                  │ │
│  │  │ (보유) │      │ (대기) │      │ (대기) │                  │ │
│  │  └────────┘      └────────┘      └────────┘                  │ │
│  │       ↑                                │                      │ │
│  │       └────────────────────────────────┘                      │ │
│  │              우선순위 상승 (Inheritance)                       │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

│  □ 인터럽트 지연 최소화                                              │

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• 실시간: 데드라인 내 작업 완료 보장
• Hard: 위반 시 치명적 (항공, 의료, 원자력)
• Soft: 위반 시 성능 저하 (멀티미디어)
• 스케줄링: RM (정적), EDF (동적), 최적
• 도전: 우선순위 역전, 인터럽트 지연
• RTOS: VxWorks, QNX, FreeRTOS, RT-Linux
```
