# 스토리지 영역 네트워크 (SAN) 연동

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SAN (Storage Area Network)은 서버와 대규모 저장 장치를 고속 전용 네트워크(Fibre Channel 등)로 연결하여, 운영체제가 원격 저장소를 로컬 디스크처럼 인식하게 하는 **블록 레벨 (Block-level)** 인터페이스 기술이다.
> 2. **가치**: 운영체제 관점에서 스토리지 자원을 중앙 집중화하고 가상화함으로써 고가용성(HA), 빠른 백업, 재해 복구(DR)를 가능하게 하며 서버의 I/O 부하를 전용 하드웨어로 분산시킨다.
> 3. **융합**: 현대의 클라우드 데이터센터에서 가상 머신(VM)이나 컨테이너의 영구 볼륨을 할당하는 핵심 기반 기술이며, OS는 이를 HBA (Host Bus Adapter)를 통해 직접 제어한다.

+++

## Ⅰ. SAN 연동의 개요

### 1. 정의
- 서버와 스토리지를 결합하기 위해 광케이블이나 기가비트 이더넷 등의 고속 네트워크를 사용하는 전용망 기술이다. 운영체제는 네트워크 너머의 스토리지 볼륨을 자신의 직접 연결된 드라이브(DAS)처럼 취급한다.

### 2. OS와의 연동 방식
- **블록 인터페이스 (Block Interface)**: 파일 시스템 레벨이 아닌, 섹터/블록 단위로 데이터를 주고받는다. 이로 인해 OS는 SAN 스토리지 위에 NTFS, ext4 등 자신이 원하는 파일 시스템을 직접 구축할 수 있다.

### 3. 💡 비유: 우리 집 마당의 '초대형 지하 창고'
- 일반 하드디스크가 내 책상 밑의 서랍(DAS)이라면, SAN은 **'우리 집 마당 밑에 지하 터널로 연결된 거대 창고'**와 같다.
- 창고는 저 멀리 공용 공간에 있지만, 전용 터널(FC)이 너무 빠르고 직접 연결되어 있어서 주인(OS)은 그냥 안방 서랍장을 열듯 자연스럽게 물건을 넣고 뺀다.

+++

## Ⅱ. SAN 연동 메커니즘 (Deep Dive)

### 1. 전송 프로토콜 (Protocol Stack)
- **Fibre Channel (FC)**: 초고속 광전송 프로토콜로, 하드웨어 레벨에서 프레임을 전송한다. 지연 시간이 극도로 낮다.
- **iSCSI (Internet SCSI)**: 기존 TCP/IP 네트워크를 통해 SCSI 명령어를 캡슐화하여 전달한다. 전용 장비 없이 기존 이더넷 인프라를 활용 가능하다.
- **FCoE (Fibre Channel over Ethernet)**: FC 프레임을 이더넷 프레임 안에 담아 전송하여 인프라를 통합한다.

### 2. OS 커널 내부 처리 과정
- **HBA (Host Bus Adapter) 드라이버**: OS 커널은 HBA를 통해 스토리지 장치를 스캔한다.
- **LUN (Logical Unit Number) 매핑**: SAN 관리자가 할당한 논리적 볼륨(LUN)이 OS의 `/dev/sdb`와 같은 장치 파일로 매핑된다.
- **멀티패싱 (Multipathing)**: 가용성을 위해 서버와 스토리지 사이에 여러 개의 경로를 둔다. OS 커널은 `dm-multipath` 등을 통해 경로 장애 시 자동으로 우회한다.

### 3. ASCII 구조 다이어그램
```text
    [ 서버 (OS: Linux/Windows) ]        [ SAN 스위치 (Fabric) ]       [ 스토리지 어레이 ]
    ┌───────────────────────┐            ┌──────────────┐            ┌──────────────┐
    │  App / File System    │            │              │            │  LUN 0 (HDD) │
    ├───────────────────────┤            │  Fibre       │            ├──────────────┤
    │  Block Device Driver  │◀──(FC)──▶ │  Channel     │◀──(FC)──▶ │  LUN 1 (SSD) │
    ├───────────────────────┤            │  Switch      │            ├──────────────┤
    │  HBA (Fibre Card)     │            │              │            │  Cache Memory│
    └───────────────────────┘            └──────────────┘            └──────────────┘
```

- **📢 섹션 요약 비유**: 복잡한 네트워크 스위치를 거치지만, OS에게는 단순한 1:1 케이블 연결처럼 투명하게(Transparent) 보이게 만드는 고속 전용 통로 기술입니다.

## Ⅲ. SAN vs NAS (운영체제 관점의 분석)

| 비교 항목 | SAN (Storage Area Network) | NAS (Network Attached Storage) |
|:---|:---|:---|
| **데이터 단위** | **블록 (Block)** | **파일 (File)** |
| **인식 형태** | 로컬 디스크 (C:/, /dev/sda) | 네트워크 공유 폴더 (\\Server\Share) |
| **파일 시스템** | OS가 직접 생성/관리 | NAS 기기가 관리 (NFS, SMB) |
| **속도** | 매우 빠름 (저지연) | 상대적 느림 (이더넷 오버헤드) |
| **주 용도** | 데이터베이스, 대규모 DB | 파일 공유, 일반 데이터 저장 |

- **📢 섹션 요약 비유**: SAN은 빈 땅(블록)을 빌려와서 내가 직접 집을 짓는 것이고, NAS는 이미 지어진 아파트의 방 한 칸(공유 폴더)을 빌려 쓰는 것과 같습니다.

## Ⅳ. 실무 적용 및 OS 최적화 (Strategy)

### 1. 실무 시나리오: 클러스터 파일 시스템 (Shared Storage)
- **상황**: 두 대의 서버(Active-Standby)가 동일한 SAN 스토리지에 동시 연결된 상황.
- **판단**: 일반 파일 시스템은 동시 쓰기 시 데이터가 깨지므로, OS 레벨에서 GFS2나 OCFS2 같은 **클러스터 파일 시스템**을 사용하거나, 분산 잠금 관리자(DLM)를 연동해야 한다.

### 2. 기술사적 인사이트: NVMe over Fabrics (NVMe-oF)
- 최근에는 전통적인 FC 대신 NVMe 프로토콜을 네트워크(RDMA, TCP) 상으로 확장한 NVMe-oF가 대두되고 있다. 이는 OS가 원격 스토리지를 로컬 NVMe SSD만큼의 속도로 접근하게 하며, 폰 노이만 병목을 인프라 레벨에서 해결하는 핵심 기술이다.

- **📢 섹션 요약 비유**: 과거에는 트럭(FC)으로 짐을 날랐다면, 이제는 컨베이어 벨트(NVMe-oF)를 서버와 창고 사이에 설치하여 지연 시간을 제로에 가깝게 줄이는 추세입니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **중앙 집중 관리**: 여러 서버의 데이터를 한 곳에서 백업 및 스냅샷 관리 가능.
- **성능 극대화**: 이더넷 트래픽과 분리된 전용망 사용으로 I/O 병목 해소.

### 2. 미래 전망
클라우드 네이티브 환경에서는 소프트웨어 정의 스토리지 (SDS) 기술과 결합하여, 하드웨어 SAN 장비 없이 일반 서버의 디스크를 묶어 가상 SAN을 만드는 방향(Ceph, vSAN)으로 진화하고 있다. 하지만 여전히 고성능 미션 크리티컬 시스템에서는 물리적 SAN 연동이 신뢰성의 표준이다.

- **📢 섹션 요약 비유**: 서버들이 함께 사용하는 '공동 금고'와 같으며, 보안과 속도가 생명인 시스템의 필수 요소입니다.

1. SAN은 컴퓨터 밖에 있는 커다란 보물 상자를 **'마치 내 컴퓨터 안에 있는 것처럼'** 아주 빠르게 연결해 주는 기술이에요.
2. 광케이블이라는 특별한 고속도로를 사용해서, 멀리 있는 데이터도 바로 옆에 있는 것처럼 빨리 가져올 수 있죠.
3. 여러 대의 컴퓨터가 이 커다란 보물 상자를 같이 나눠 쓸 수 있어서 아주 편리하답니다!
