# 보호 및 보안 (Protection & Security)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보호(Protection)는 시스템 자원에 대한 프로그램 간 무단 접근을 제어하는 메커니즘이고, 보안(Security)은 외부 공격과 내부 위협으로부터 시스템을 방어하는 포괄적 체계다.
> 2. **가치**: 보호는 **신뢰성 있는 멀티프로그래밍**을 가능하게 하고, 보안은 **데이터 기밀성·무결성·가용성(CIA)**을 보장하여 비즈니스 연속성을 지탱한다.
> 3. **융합**: 현대 OS는 하드웨어 보안(Intel SGX, ARM TrustZone), 가상화 격리, 컨테이너 보안(seccomp, AppArmor)까지 확장된 다계층 방어 체계를 구축한다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 정의
- **보호(Protection)**: 운영체제 내부에서 프로세스 간, 또는 프로세스와 시스템 자원 간의 무단 접근을 제어하는 메커니즘으로, 주로 하드웨어와 커널 수준에서 구현된다.
- **보안(Security)**: 시스템 전체를 대상으로 외부 공격자, 악의적 내부자, 악성코드로부터 데이터와 자원을 보호하는 포괄적인 정책, 기술, 절차의 집합이다.

### 2. 💡 비유: 건물 출입 통제 시스템
- 보호는 **"건물 내부의 잠금장치"**와 같다. 회의실 열쇠, 금고 비밀번호, 층별 출입카드 권한 등 내부 자원에 대한 접근을 제어한다.
- 보안은 **"건물 전체 경비 시스템"**과 같다. 경비원, CCTV, 방범창, 화재경보기 등 외부 침입과 내부 사고를 모두 방어한다.

### 3. 등장 배경
- **멀티프로그래밍 환경**: 여러 프로세스가 메모리와 자원을 공유하면서 상호 간섭 방지 필요
- **네트워크 연결**: 인터넷을 통한 원격 공격 가능성 증대
- **데이터 가치 상승**: 개인정보, 금융정보, 기업 기밀의 디지털화
- **규제 강화**: GDPR, CCPA, PCI-DSS 등 보안 의무 법제화

### 4. 보안의 3대 원칙 (CIA Triad)

| 원칙 | Full Name | 설명 | 예시 |
|:---|:---|:---|:---|
| **C** | Confidentiality (기밀성) | 인가된 사용자만 접근 가능 | 암호화, 접근 제어 |
| **I** | Integrity (무결성) | 데이터의 정확성과 완전성 보장 | 해시, 디지털 서명 |
| **A** | Availability (가용성) | 필요 시 서비스 제공 보장 | 이중화, DDoS 방어 |

- **📢 섹션 요약 비유**: 보호는 자동차의 안전벨트와 에어백처럼 내부 탑승자를 보호하고, 보안은 차량 도난 방지 시스템과 운전자 보험처럼 전체적인 위험을 관리합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 다계층 보안 아키텍처

```text
    ┌─────────────────────────────────────────────────────────────────────────┐
    │               Multi-Layer Protection & Security Architecture           │
    │                                                                         │
    │  [Layer 5: 사용자 공간 보안]                                            │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  Application Security  │  Input Validation  │  Secure Coding      │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    │                                                                         │
    │  [Layer 4: OS 서비스 보안]                                              │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  Authentication (PAM)  │  Authorization (sudo/capabilities)       │ │
    │  │  Audit Logging         │  Mandatory Access Control (SELinux)      │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    │                                                                         │
    │  [Layer 3: 커널 보호 메커니즘]                                          │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  Process Isolation     │  Memory Protection (Page Tables)         │ │
    │  │  Privilege Rings       │  System Call Filtering (seccomp)         │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    │                                                                         │
    │  [Layer 2: 하드웨어 보안]                                               │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  MMU (Memory Mgmt)     │  TPM (Trusted Platform Module)           │ │
    │  │  Ring 0-3 Support      │  Secure Enclave (SGX/TrustZone)          │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    │                                                                         │
    │  [Layer 1: 물리적 보안]                                                 │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  Physical Access Control  │  Secure Boot  │  Hardware Encryption  │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    └─────────────────────────────────────────────────────────────────────────┘
```

### 2. 구성 요소 상세

| 구성 요소 | Full Name | 역할 | 구현 예시 |
|:---|:---|:---|:---|
| **Authentication** | 인증 (Authentication) | 신원 확인 | PAM, Kerberos, OAuth |
| **Authorization** | 인가 (Authorization) | 권한 부여 | ACL, RBAC, Capabilities |
| **MMU** | 메모리 관리 장치 (Memory Management Unit) | 주소 변환 및 보호 | 페이지 테이블, TLB |
| **DAC/MAC** | 임의/강제 접근 제어 | 접근 정책 | Unix 퍼미션, SELinux |
| **Audit** | 감사 (Auditing) | 행위 기록 | auditd, Windows Event Log |

### 3. CPU 권한 링 (Privilege Rings)

```text
    ┌─────────────────────────────────────────────────────────────────────────┐
    │                      CPU Privilege Rings (x86)                         │
    │                                                                         │
    │        ┌─────────────────────────────────────────────────────────┐     │
    │        │                    Ring 3 (User Mode)                   │     │
    │        │  ┌───────────┐ ┌───────────┐ ┌───────────┐             │     │
    │        │  │ App A     │ │ App B     │ │ App C     │             │     │
    │        │  │ (Web)     │ │ (DB)      │ │ (Shell)   │             │     │
    │        │  └───────────┘ └───────────┘ └───────────┘             │     │
    │        │         제한된 명령어, I/O 접근 불가                    │     │
    │        └─────────────────────────────────────────────────────────┘     │
    │                                │                                       │
    │                     System Call │ (Mode Switch)                        │
    │                                ▼                                       │
    │        ┌─────────────────────────────────────────────────────────┐     │
    │        │                    Ring 0 (Kernel Mode)                 │     │
    │        │  ┌───────────────────────────────────────────────────┐  │     │
    │        │  │                   Kernel                          │  │     │
    │        │  │  - 모든 명령어 실행 가능                          │  │     │
    │        │  │  - 하드웨어 직접 접근                             │  │     │
    │        │  │  - 메모리 전 영역 접근                            │  │     │
    │        │  └───────────────────────────────────────────────────┘  │     │
    │        └─────────────────────────────────────────────────────────┘     │
    │                                                                         │
    │        Ring 1, 2: (거의 사용 안 함, 일부 가상화에서 활용)              │
    └─────────────────────────────────────────────────────────────────────────┘
```

**[권한 링 해설]**
1. **Ring 0 (커널 모드)**: 가장 높은 권한, 모든 하드웨어 자원에 직접 접근 가능
2. **Ring 3 (사용자 모드)**: 가장 낮은 권한, 제한된 명령어 집합, I/O 직접 접근 불가
3. **전환**: 시스템 호출, 인터럽트 발생 시 Ring 3 → Ring 0 전환 (Context Switch)
4. **보안 효과**: 사용자 프로그램이 커널 영역을 침범하거나 다른 프로세스 메모리에 접근하는 것을 하드웨어 수준에서 차단

### 4. 접근 제어 모델 비교

```text
    ┌─────────────────────────────────────────────────────────────────────────┐
    │                  Access Control Matrix Model                           │
    │                                                                         │
    │                    Resources (Objects)                                  │
    │              ┌─────────┬─────────┬─────────┬─────────┐                 │
    │              │ File F1 │ File F2 │ Device  │ Memory  │                 │
    │  ┌───────────┼─────────┼─────────┼─────────┼─────────┤                 │
    │  │ User U1   │  R--    │  RW-    │  ---    │  R--    │                 │
    │  ├───────────┼─────────┼─────────┼─────────┼─────────┤                 │
    │  │ User U2   │  RW-    │  R--    │  RW-    │  ---    │  Subjects      │
    │  ├───────────┼─────────┼─────────┼─────────┼─────────┤                 │
    │  │ Group G1  │  R-X    │  ---    │  R--    │  R-X    │                 │
    │  └───────────┴─────────┴─────────┴─────────┴─────────┘                 │
    │                                                                         │
    │  R: Read, W: Write, X: Execute, -: No permission                       │
    └─────────────────────────────────────────────────────────────────────────┘

    [구현 방식]
    ┌───────────────────────────────────────────────────────────────────────┐
    │  ACL (Access Control List)         │  Capability List                  │
    │  ┌─────────────────────────────┐   │  ┌─────────────────────────────┐ │
    │  │ File F1:                    │   │  │ User U1:                    │ │
    │  │   U1: Read                  │   │  │   F1: Read                  │ │
    │  │   U2: Read, Write           │   │  │   F2: Read, Write           │ │
    │  │   G1: Read, Execute         │   │  │   Memory: Read              │ │
    │  └─────────────────────────────┘   │  └─────────────────────────────┘ │
    │  객체 중심 (열 기준)               │  주체 중심 (행 기준)             │
    └───────────────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 권한 링은 건물의 층별 출입 권한과 같습니다. 1층(커널)은 전체 건물 출입이 가능하지만, 4층(사용자 앱)은 본인 책상 근처만 이동할 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 접근 제어 모델 비교

| 모델 | Full Name | 특징 | 장점 | 단점 | 적용 |
|:---|:---|:---|:---|:---|:---|
| **DAC** | 임의 접근 제어 (Discretionary Access Control) | 소유자가 권한 부여 | 유연성 | 권한 남용 위험 | Unix 퍼미션 |
| **MAC** | 강제 접근 제어 (Mandatory Access Control) | 정책 기반 강제 적용 | 강력한 보안 | 운영 복잡 | SELinux, AppArmor |
| **RBAC** | 역할 기반 접근 제어 (Role-Based Access Control) | 역할에 권한 연결 | 관리 용이 | 역할 폭발 | 기업 시스템 |
| **ABAC** | 속성 기반 접근 제어 (Attribute-Based Access Control) | 속성 기반 동적 결정 | 세밀한 제어 | 복잡성 | XACML, AWS IAM |

### 2. 인증 방식 비교

| 방식 | Full Name | 보안 강도 | 사용자 편의성 | 적용 환경 |
|:---|:---|:---|:---|:---|
| **Password** | 비밀번호 | 중간 | 높음 | 범용 |
| **MFA** | 다중 인증 (Multi-Factor Authentication) | 높음 | 중간 | 민감 시스템 |
| **Certificate** | 인증서 기반 | 높음 | 높음 (설정 후) | 서버, IoT |
| **Biometric** | 생체 인증 | 높음 | 높음 | 모바일, 출입 |
| **SSO** | 단일 로그온 (Single Sign-On) | 중간 | 매우 높음 | 기업 내부 |

### 3. 과목 융합 관점

| 융합 과목 | 연관 개념 | 융합 포인트 |
|:---|:---|:---|
| **컴퓨터 구조** | MMU, TLB, 권한 비트 | 하드웨어 기반 메모리 보호 |
| **네트워크** | 방화벽, TLS, VPN | 네트워크 계층 보안 |
| **데이터베이스** | Row-Level Security, 암호화 | 데이터 보호 |
| **알고리즘** | 암호화 알고리즘, 해시 | AES, RSA, SHA |
| **소프트웨어 공학** | Secure SDLC, 위협 모델링 | 보안 설계 |

- **📢 섹션 요약 비유**: 접근 제어 모델은 호텔의 객실 키 시스템과 같습니다. DAC는 투숙객이 키를 빌려주는 것, MAC은 호텔 정책으로 층별 출입 제어, RBAC는 직원별 담당 구역 권한입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오

#### 시나리오 A: SELinux 강제 접근 제어 구성
```text
    [요구사항] 웹 서버 프로세스가 웹 콘텐츠 외 파일 접근 차단

    ┌─────────────────────────────────────────────────────────────────────┐
    │                    SELinux Policy Enforcement                       │
    │                                                                     │
    │  [SELinux Context]                                                  │
    │  ┌───────────────────────────────────────────────────────────────┐ │
    │  │  User:Role:Type:Level                                         │ │
    │  │                                                               │ │
    │  │  /var/www/html/index.html                                     │ │
    │  │  → system_u:object_r:httpd_sys_content_t:s0                   │ │
    │  │                                                               │ │
    │  │  httpd process                                                │ │
    │  │  → system_u:system_r:httpd_t:s0                               │ │
    │  └───────────────────────────────────────────────────────────────┘ │
    │                                                                     │
    │  [Policy Rule]                                                      │
    │  allow httpd_t httpd_sys_content_t:file { read getattr open };     │
    │                                                                     │
    │  [효과]                                                             │
    │  - httpd 프로세스는 httpd_sys_content_t 타입 파일만 읽기 가능      │
    │  - /etc/shadow 등 다른 타입 파일 접근 시 AVC(Access Vector Cache)   │
    │    거부 및 로그 기록                                               │
    └─────────────────────────────────────────────────────────────────────┘
```

#### 시나리오 B: 컨테이너 보안 격리
```text
    [요구사항] 컨테이너 탈출 방지 및 최소 권한 실행

    ┌─────────────────────────────────────────────────────────────────────┐
    │                Container Security Hardening                         │
    │                                                                     │
    │  [1] Capability Drop                                                │
    │  docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE ...          │
    │                                                                     │
    │  [2] Seccomp Profile                                                │
    │  ┌───────────────────────────────────────────────────────────────┐ │
    │  │  Default: 300+ syscalls 중 60+ 차단                           │ │
    │  │  Custom: 필요한 syscall만 허용 (allowlist)                    │ │
    │  │  예: ptrace, mount, kexec_load 차단                           │ │
    │  └───────────────────────────────────────────────────────────────┘ │
    │                                                                     │
    │  [3] AppArmor/SELinux Profile                                       │
    │  docker run --security-opt apparmor=docker-default ...             │
    │                                                                     │
    │  [4] Read-Only Root Filesystem                                      │
    │  docker run --read-only ...                                         │
    │                                                                     │
    │  [5] No New Privileges                                              │
    │  docker run --security-opt no-new-privileges ...                   │
    └─────────────────────────────────────────────────────────────────────┘
```

### 2. 도입 체크리스트

| 항목 | 체크 포인트 | 권장 사항 |
|:---|:---|:---|
| 인증 | MFA 적용 | 관리자 계정 필수, 일반 계정 권장 |
| 인가 | 최소 권한 원칙 | 필요한 권한만 부여 |
| 감사 | 로그 기록 | 인증 실패, 권한 변경 로그 |
| 암호화 | 전송/저장 암호화 | TLS 1.3, AES-256 |
| 패치 | 정기 업데이트 | CVE 모니터링, 월간 패치 |

### 3. 안티패턴

| 안티패턴 | 문제점 | 올바른 접근 |
|:---|:---|:---|
| **과도한 권한** | root 실행, 모든 capability | 최소 권한, capability drop |
| **Security-Through-Obscurity** | 숨김으로 보안 착각 | 심층 방어, 공개 검증 |
| **SELinux Permissive** | 탐지만 하고 차단 안 함 | Enforcing 모드로 전환 |
| **하드코딩된 비밀번호** | 코드에 평문 저장 | Vault, Secrets Manager |

- **📢 섹션 요약 비유**: 보안은 양파와 같아요. 껍질을 하나 벗겨도 또 다른 보호막이 있어야 합니다. 단일 방어선은 단일 실패점이 됩니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과

| 구분 | 효과 | 지표 |
|:---|:---|:---|
| **정량** | 보안 사고 감소 | 연간 사고 건수 80% 감소 |
| **정량** | 규정 준수 | 감사 지적 사항 0건 |
| **정성** | 신뢰도 향상 | 고객 신뢰도 증가 |
| **정성** | 비즈니스 연속성 | RTO/RPO 달성 |

### 2. 미래 전망
- **Zero Trust Architecture**: "절대 신뢰하지 않고 항상 검증"
- **하드웨어 보안 확대**: CXL 메모리 암호화, TEE(Trusted Execution Environment)
- **AI 기반 위협 탐지**: 이상 행위 실시간 감지

### 3. 참고 표준
- **ISO/IEC 27001**: 정보보호 관리체계
- **NIST Cybersecurity Framework**: 보안 프레임워크
- **Common Criteria (ISO 15408)**: 보안 평가 기준

- **📢 섹션 요약 비유**: 보호와 보안은 건물의 기초와 지붕처럼, 하나는 내부 구조를 튼튼하게 하고 다른 하나는 외부 위협으로부터 보호합니다.
