# 시스템 호출 (System Call)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 호출 (System Call)은 사용자 응용 프로그램이 OS (Operating System) 커널이 제공하는 보호된 서비스 (파일 I/O, 프로세스 관리, 네트워크 등)를 요청하기 위해 사용하는 유일하고 공식적인 하드웨어 게이트웨이 인터페이스다.
> 2. **가치**: 모드 비트 (Mode Bit)를 1 (User)에서 0 (Kernel)으로 전환하는 트랩 (Trap) 메커니즘을 통해, 사용자 프로세스가 시스템 자원에 직접 접근하는 것을 차단하고 검증된 경로를 통해서만 자원을 할당받게 함으로써 시스템 보안성과 추상화를 제공한다.
> 3. **융합**: 고수준 언어 개발자는 직접적인 어셈블리 트랩 명령어를 사용하는 대신, POSIX (Portable Operating System Interface) 표준을 따르는 API (Application Programming Interface) 라이브러리 (예: glibc)를 통해 시스템 호출 (System Call)을 투명하게 호출한다.

---

## Ⅰ. 개요 (Context & Background)

- **概念**: 시스템 호출 (System Call)은 운영체제가 사용자 프로그램에게 제공하는 '서비스 요청 창구'다. 현대 OS (Operating System)는 보안을 위해 사용자 프로그램이 하드웨어를 직접 제어하는 것을 금지하므로, 프로그램이 파일 읽기, 화면 출력, 메모리 할당 등을 수행하려면 반드시 이 창구를 통해 커널에게 권한 대행을 요청해야 한다.
- **💡 비유**: 시스템 호출 (System Call)은 **은행의 입구와 창구**와 같다. 고객 (사용자 프로그램)은 은행 금고 (하드웨어/자원)에 직접 들어가 돈을 꺼낼 수 없다. 오직 정해진 창구 (시스템 호출 인터페이스)를 통해 신분증 (권한)을 제시하고 전표 (매개변수)를 작성해야만, 은행원 (커널)이 대신 금고에서 돈을 꺼내어 전달해준다.
- **등장 배경**: 초기 일괄 처리 시스템과 달리 다중 프로그래밍 환경이 도입되면서, 여러 프로그램이 동시에 하드웨어를 제어하려 할 때 발생하는 충돌과 보안 위협을 막아야 했다. 이에 듀얼 모드 (Dual Mode) 하드웨어 아키텍처가 정착되었고, 사용자 모드에서 커널 모드로 안전하게 진입할 수 있는 유일한 '통제된 통로'로서 시스템 호출 (System Call)이 탄생했다.

- **📢 섹션 요약 비유**: 마치 외부인이 국가 기밀 시설 (커널)에 함부로 들어갈 수 없도록, 정해진 면회실 (시스템 호출 인터페이스)에서만 신분 확인 후 허가된 작업만 수행할 수 있게 하는 보안 프로토콜과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 시스템 호출 (System Call) 실행 메커니즘

사용자 영역에서 시스템 호출 (System Call)이 발생하여 커널 영역의 실제 기능이 수행되고 복귀하기까지의 흐름은 하드웨어와 소프트웨어의 긴밀한 협력을 필요로 한다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               시스템 호출 (System Call) 처리 아키텍처             │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │  [ 사용자 공간 (User Space) ]                                  │
 │   1. API 호출 : read(fd, buf, size);                          │
 │          │                                                    │
 │   2. 라이브러리 (libc Wrapper):                                 │
 │      - 시스템 호출 번호 (예: sys_read=3)를 EAX 레지스터에 저장    │
 │      - 인자들을 나머지 레지스터 (EBX, ECX 등)에 저장              │
 │      - 트랩 발생 : INT 0x80 (Legacy) 또는 SYSCALL (Fast)       │
 │          │                                                    │
 │ ─ ─ ─ ─ ─▼─ ─ ─ ─ ─ ( 권한 전환: User → Kernel ) ─ ─ ─ ─ ─ ─  │
 │                                                               │
 │  [ 커널 공간 (Kernel Space) ]                                  │
 │   3. 시스템 호출 핸들러 (System Call Handler):                  │
 │      - 레지스터에서 번호(3)를 확인하고 유효성 검사                  │
 │          │                                                    │
 │   4. 시스템 호출 테이블 (System Call Table):                   │
 │      [0] sys_exit                                             │
 │      [1] sys_fork                                             │
 │      [2] sys_open                                             │
 │      [3] sys_read  ──────────▶ [ 실제 커널 함수 주소로 점프 ]     │
 │          │                                                    │
 │   5. 서비스 실행 : sys_read() 루틴 수행                         │
 │      - 하드웨어 I/O 제어, 데이터 복사 등                        │
 │          │                                                    │
 │ ─ ─ ─ ─ ─▲─ ─ ─ ─ ─ ( 권한 전환: Kernel → User ) ─ ─ ─ ─ ─ ─  │
 │          │                                                    │
 │   6. 사용자 프로그램 복귀 (Return Value 전달)                    │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 응용 프로그램이 `read()` 같은 표준 API (Application Programming Interface)를 호출하면, C 라이브러리는 해당 서비스의 고유 번호를 특정 레지스터에 적재하고 소프트웨어 인터럽트 (Software Interrupt)인 트랩 (Trap)을 발생시킨다. CPU (Central Processing Unit)는 이를 감지하여 즉시 모드 비트 (Mode Bit)를 0으로 바꾸고 커널 내의 시스템 호출 핸들러로 제어권을 넘긴다. 핸들러는 시스템 호출 테이블 (System Call Table)을 참조하여 요청 번호에 매핑된 진짜 커널 함수 주소로 분기하여 작업을 수행한다.

### 구성 요소 및 매개변수 전달 방식

| 구성 요소 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **API 래퍼** | 호출 편의성 제공 | 라이브러리 수준에서 시스템 호출 번호 세팅 | 대리 작성 서비스 |
| **시스템 호출 번호** | 서비스 고유 식별자 | EAX/RAX 레지스터에 저장되는 정수 값 | 메뉴 번호 |
| **시스템 호출 테이블** | 서비스 주소 매핑 | 시스템 호출 번호와 커널 함수 주소의 배열 | 식당 메뉴판 |
| **트랩 (Trap)** | 모드 전환 유발 | 하드웨어에 의한 강제 권한 상승 및 점프 | 비상 호출 버튼 |

**매개변수 전달 방식 (Parameter Passing)**:
1. **레지스터 방식**: 파라미터를 CPU 레지스터에 직접 저장. 속도가 가장 빠르나 레지스터 개수만큼만 전달 가능.
2. **블록/표본 방식**: 파라미터를 메모리 블록에 저장하고, 그 블록의 시작 주소만 레지스터로 전달. 대량의 데이터 전달에 유리.
3. **스택 방식**: 파라미터를 사용자 스택에 Push하고 커널이 이를 Pop하여 사용.

- **📢 섹션 요약 비유**: 마치 복잡한 세금 신고를 직접 국세청 서버에 접속해 처리하는 대신, 세무사 (API)에게 서류를 주면 세무사가 규격화된 양식 (레지스터/번호)으로 정부 시스템 (커널)에 안전하게 접수하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### API vs 시스템 호출 vs 커널 함수 비교

| 비교 항목 | API (Application Programming Interface) | 시스템 호출 (System Call) | 커널 함수 (Kernel Function) |
|:---|:---|:---|:---|
| **실행 주체** | 라이브러리 (libc, Win32) | **OS와 하드웨어의 경계선** | OS (Operating System) 커널 내부 |
| **권한 모드** | 사용자 모드 (Ring 3) | **모드 전환 발생 (Trap)** | 커널 모드 (Ring 0) |
| **의존성** | OS 표준에 따라 이식성 높음 | 특정 OS 커널 버전에 종속적 | 커널 소스 코드 내부에 종속 |
| **주요 예시** | `printf()`, `malloc()` | `sys_write()`, `sys_brk()` | `vfs_read()`, `kmalloc()` |

### 정량적 오버헤드 분석: System Call vs Local Call
시스템 호출 (System Call)은 일반 함수 호출 (Local Function Call)보다 수십~수백 배 더 많은 사이클을 소모한다. 이는 ① 레지스터 백업, ② 모드 비트 (Mode Bit) 전환, ③ 커널 스택 전환, ④ 시스템 호출 테이블 탐색, ⑤ 권한 및 인자 유효성 검사 등의 추가 과정이 포함되기 때문이다.

- **📢 섹션 요약 비유**: 내 방 책상에서 책을 꺼내는 것 (일반 함수)은 즉시 가능하지만, 도서관 폐쇄 서고의 책을 빌리려면 신청서 작성, 신분증 확인, 사서의 반출 작업 (시스템 호출)을 기다려야 하는 것과 같은 시간 차이가 발생합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 고성능 서버의 시스템 호출 (System Call) 최소화 (Batching)
- **상황**: 초당 수백만 건의 요청을 처리하는 고성능 DB (Database) 엔진 개발 중, 빈번한 `write()` 시스템 호출로 인해 문맥 교환 (Context Switch) 오버헤드가 CPU (Central Processing Unit) 점유율의 30%를 차지하는 성능 저하가 발견됨.
- **판단**: 시스템 호출 (System Call) 자체를 줄이는 설계가 필요함. 사용자 공간에 일정량의 데이터를 모았다가 한 번에 내보내는 **버퍼링 (Buffering)** 기법이나, 여러 개의 I/O (Input/Output) 요청을 단 한 번의 시스템 호출로 처리하는 리눅스의 `io_uring` 또는 `vmsplice` 기술을 적용해야 함.

### 도입 체크리스트 및 안티패턴
- **체크리스트**: 루프 내에서 불필요하게 `getpid()`나 `time()` 같은 시스템 호출을 반복하고 있지 않은지 확인하라. 결과값이 변하지 않는다면 변수에 캐싱 (Caching)하여 사용해야 한다.
- **안티패턴**: 보안 검증 없이 사용자로부터 넘어온 포인터 주소를 커널에서 바로 사용하는 행위. 이는 사용자 프로그램이 커널 메모리 영역을 조작하게 만드는 치명적인 보안 취약점 (Kernel Exploit)이 된다. 커널은 반드시 `copy_from_user()`와 같은 안전한 복사 함수를 사용해야 한다.

- **📢 섹션 요약 비유**: 물 한 컵을 마실 때마다 매번 주방 (커널)에 가는 대신, 정수기 물통 (버퍼)을 거실 (사용자 공간)에 가져다 놓고 마시는 것이 효율적인 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 시스템 호출 (System Call) 도입의 기대효과

| 구분 | 도입 전 (직접 접근) | 도입 후 (시스템 호출 방식) | 기대효과 |
|:---|:---|:---|:---|
| **보안** | 하드웨어 파괴 가능 | 커널 검증 후 접근 | 시스템 침해 사고 90% 예방 |
| **추상화** | 특정 HDD 하드웨어 코드 작성 | `read()` 함수로 모든 저장장치 제어 | 개발 생산성 300% 향상 |
| **안정성** | 프로그램 오류 시 전체 정지 | 해당 시스템 호출만 에러 반환 | 시스템 가용성 (Availability) 증대 |

- **결론**: 시스템 호출 (System Call)은 애플리케이션의 편의성과 하드웨어의 견고한 보호라는 상충하는 목표를 동시에 달성하기 위한 운영체제의 최상위 설계 철학이다. 개발자는 기저의 복잡한 하드웨어 동작을 몰라도 시스템 호출 (System Call)이라는 표준화된 창구를 통해 전 세계 어떤 컴퓨터에서든 동일하게 동작하는 소프트웨어를 개발할 수 있게 되었다.

- **📢 섹션 요약 비유**: 마치 복잡한 자동차 엔진 내부 (하드웨어)를 몰라도 핸들과 페달 (시스템 호출 인터페이스)만 조작할 줄 알면 누구나 안전하게 운전할 수 있는 것과 같은 현대 컴퓨팅의 핵심 기반 기술입니다.
