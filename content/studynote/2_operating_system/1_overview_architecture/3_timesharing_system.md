# 시분할 시스템 (Time-sharing System)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시분할 시스템(Time-sharing System)은 CPU 시간을 매우 짧은 시간 단위(Time Slice/Quantum)로 나누어 여러 사용자나 프로세스에게 번갈아 할당함으로써, 각 사용자가 **컴퓨터를 독점하고 있는 듯한 착각**을 주는 다중 사용자 인터랙티브 시스템이다.
> 2. **가치**: 다중 프로그래밍이 'CPU 활용도'에 집중했다면, 시분할 시스템은 사용자의 **'응답 시간(Response Time) 최소화'**를 최우선 목표로 하여 대화형 프로그래밍 환경을 가능케 했다.
> 3. **융합**: 현대 멀티태스킹 OS(Windows, Linux)의 근간이 되는 철학이며, 이를 실현하기 위해 라운드 로빈 스케줄링, 선점형 커널, 메모리 보호 및 스와핑 기술이 필수적으로 수반된다.

+++

## Ⅰ. 시분할 시스템 (Time-sharing System)의 개요

### 1. 정의
- 하나의 컴퓨터 시스템을 여러 명의 사용자가 동시에 접속하여 사용할 때, CPU (Central Processing Unit)가 아주 빠른 속도로 사용자들을 교대로 처리하여 각자가 독립된 컴퓨터를 쓰고 있다고 느끼게 하는 방식이다.

### 2. 등장 배경: '인간 소외'의 극복
- 과거 일괄 처리(Batch Processing)나 단순 다중 프로그래밍 환경에서는 프로그램 하나가 끝나기 전까지 사용자가 결과를 확인할 수 없었다.
- 엔지니어들은 "컴퓨터는 빨라졌는데 왜 사람은 기다려야 하는가?"라는 질문에 답하기 위해, 컴퓨터가 사람의 속도에 맞춰 '반응'하도록 설계했다.

### 3. 💡 비유: 뷔페의 회전 초밥
- **일괄 처리**: 손님 한 명이 초밥 100접시를 다 먹을 때까지 다음 손님은 줄 서서 기다려야 함.
- **시분할 시스템**: 회전 레일이 계속 돌면서, 모든 손님이 한 번에 한 접시씩 계속해서 먹을 수 있음. 레일이 충분히 빠르면 손님들은 기다린다는 느낌 없이 계속 식사하는 것과 같다.

- **📢 섹션 요약 비유**: 모든 손님이 조금씩, 하지만 끊임없이 음식을 받을 수 있도록 빠르게 움직이는 베테랑 서버의 서비스와 같습니다.

## Ⅱ. 시분할 시스템의 동작 메커니즘 (ASCII)

핵심은 **시간 할당량(Time Quantum)**과 **선점(Preemption)**이다.

```text
    [ User Terminals ]         [ Operating System (Scheduler) ]         [ CPU Time ]
    ┌───┐                      ┌──────────────────────────────┐        ┌─────────────┐
    │ U1│ ─── (Request) ──▶    │ Time Slice: 10ms             │        │ U1 (10ms)   │
    ├───┤                      │                              │        ├─────────────┤
    │ U2│ ─── (Request) ──▶    │ Round-Robin Queue:           │ ───▶   │ U2 (10ms)   │
    ├───┤                      │ [ U1 -> U2 -> U3 -> U1 ]     │        ├─────────────┤
    │ U3│ ─── (Request) ──▶    │                              │        │ U3 (10ms)   │
    └───┘                      └──────────────────────────────┘        ├─────────────┤
                                                                       │ U1 (Resume) │
                                                                       └─────────────┘
```

**[심층 동작 과정]**
1. **타이머 인터럽트**: 하드웨어 타이머가 설정된 시간(예: 10ms)이 지남을 OS (Operating System)에 알린다.
2. **상태 보존**: 현재 실행 중인 사용자의 레지스터 상태를 PCB (Process Control Block)에 저장한다.
3. **스케줄링**: 대기 큐에서 다음 순서의 사용자를 선택한다.
4. **콘텍스트 스위칭**: 새로운 사용자의 상태를 CPU에 복구하고 실행을 시작한다.

- **📢 섹션 요약 비유**: 정해진 발언 시간(Time Slice)이 지나면 마이크를 다음 사람에게 넘겨야 하는 아주 공정한 끝장 토론 시스템과 같습니다.

## Ⅲ. 다중 프로그래밍 vs 시분할 시스템 비교

| 비교 항목 | 다중 프로그래밍 (Multiprogramming) | 시분할 시스템 (Time-sharing) |
|:---|:---|:---|
| **최우선 목표** | **CPU 사용률 극대화** | **응답 시간 단축 (사용자 편의)** |
| **전환 시점** | 실행 중인 작업이 I/O를 만날 때 | **정해진 시간이 만료되었을 때** |
| **사용자 관계** | 비대화형 (결과를 나중에 확인) | **대화형 (Interactive, 실시간 상호작용)** |
| **스케줄링** | 비선점/선점 혼합 | **강력한 선점형 (Preemptive)** |

- **📢 섹션 요약 비유**: 다중 프로그래밍이 '가스레인지 화력을 낭비하지 않는 요리사'라면, 시분할 시스템은 '여러 손님의 질문에 실시간으로 답하며 서빙하는 홀 지배인'과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. 시간 할당량(Time Quantum)의 딜레마
- **기술사적 결단**: 
  - 시간 할당량이 **너무 크면**: 시분할의 의미가 퇴색되고 선입선출(FIFO, First-In First-Out)처럼 동작하여 응답성이 떨어진다.
  - 시간 할당량이 **너무 작으면**: 콘텍스트 스위칭 오버헤드(Context Switching Overhead)가 커져 실제 작업보다 주소록 교체에 CPU가 더 많은 시간을 쓴다.
- **인사이트**: 현대 OS는 시스템 부하와 작업 성격에 따라 시간 할당량을 동적으로 조절하는 **적응형 스케줄링**을 사용한다.

### 2. 가상화와 클라우드의 멀티테넌시(Multi-tenancy)
- 클라우드 서버(AWS EC2 등)에서 하나의 물리 장비를 여러 고객이 나누어 쓰는 원리는 시분할 시스템의 극단적인 확장판이다. 하이퍼바이저(Hypervisor) 수준에서 수행되는 정교한 시분할은 현대 IT 인프라의 경제적 근간이 된다.

- **📢 섹션 요약 비유**: 한정된 자원을 쪼개어 수많은 사람에게 나눠주면서도 모두가 만족할 수 있게 설계된 초정밀 배급 시스템과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **사용자 경험 혁신**: 여러 사용자가 동시에 코딩하고 결과를 즉시 확인하는 '대화형 개발' 가능.
- **자원 민주화**: 값비싼 고성능 컴퓨터를 여러 명이 동시에 저렴하게 공유.

### 2. 미래 전망
앞으로의 시분할 기술은 하드웨어의 병렬성 극대화와 맞물려 **'극미세 시분할'**로 진화할 것이다. 특히 자율주행이나 로봇 제어와 같은 실시간성(Real-time)이 중요한 영역에서는 시분할의 공정성보다 **'결정론적 응답(Deterministic Response)'**을 보장하는 특수 시분할 알고리즘이 핵심 경쟁력이 될 것이다.

- **📢 섹션 요약 비유**: 수만 번의 날갯짓으로 허공에 정지해 있는 벌새처럼, CPU는 쉴 새 없이 움직여 우리에게 평온한 디지털 정적(독점 사용 경험)을 선물합니다.

+++

### 📌 관련 개념 맵 (Knowledge Graph)
- **라운드 로빈 (Round Robin)**: 시분할 시스템의 대표적 알고리즘.
- **콘텍스트 스위칭 (Context Switching)**: 시분할을 가능케 하는 기술적 교체 작업.
- **가상 메모리 (Virtual Memory)**: 여러 사용자가 동시에 메모리를 쓰기 위한 필수 요건.
