## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다중 처리 시스템(Multiprocessing System)은 단일 컴퓨터 시스템 내에 두 개 이상의 물리적 프로세서(CPU) 코어가 존재하여, 컴퓨터 버스, 클럭, 메모리, 주변장치를 공유하는 아키텍처다.
> 2. **가치**: 병렬(Parallel) 처리를 통해 시스템의 처리량(Throughput)을 비약적으로 높이고, 프로세서 하나의 고장이 전체 시스템 다운으로 이어지지 않는 결함 허용(Fault Tolerance)과 신뢰성을 제공한다.
> 3. **융합**: 운영체제는 이러한 다중 코어를 투명하게 관리하기 위해 SMP(대칭형 다중 처리) 아키텍처, 캐시 일관성(Cache Coherence) 유지, 그리고 코어 간 부하 균등화(Load Balancing) 알고리즘을 필수로 구현해야 한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 다중 처리 시스템은 2개 이상의 프로세서(요즘은 하나의 칩 안에 여러 코어를 두는 Multicore 구조를 포함)가 묶여서 작업을 분담하는 시스템이다. 병행(Concurrency, 빠르게 번갈아 실행)을 넘어 진정한 물리적 병렬성(Parallelism, 동시에 실행)을 제공한다. 강결합(Tightly Coupled) 시스템이라고도 불린다.
- **💡 비유**: 다중 처리 시스템은 **여러 명의 요리사가 있는 대형 주방**과 같다. 단일 프로세서(시분할)가 한 요리사가 여러 요리를 번갈아 하는 것이라면, 다중 처리는 요리사 4명이 각자 프라이팬을 잡고 동시에 4개의 요리를 만들어 내는 것이다. 도마와 식재료 창고(공유 메모리)는 함께 사용한다.
- **도입 목적 (3대 장점)**:
  1. **처리량 증가 (Increased Throughput)**: N개의 프로세서가 N배 빠른 것은 아니지만(오버헤드 존재), 짧은 시간 안에 훨씬 더 많은 작업을 수행한다.
  2. **규모의 경제 (Economy of Scale)**: 여러 대의 컴퓨터를 구성하는 것보다, 하나의 보드에 여러 프로세서를 두고 메모리와 디스크 전원을 공유하는 것이 비용 효율적이다.
  3. **신뢰성 증가 (Increased Reliability)**: 프로세서 하나가 고장 나더라도 시스템은 멈추지 않고, 속도만 약간 느려진 채 정상 동작을 이어간다(우아한 성능 저하, Graceful Degradation).

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 다중 처리 아키텍처 모델

다중 처리 시스템은 운영체제와 하드웨어의 결합 방식에 따라 비대칭형(ASMP)과 대칭형(SMP)으로 나뉜다. 현대의 거의 모든 범용 컴퓨터(PC, 모바일, 서버)는 SMP 구조를 따른다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               대칭형 다중 처리 (SMP) 아키텍처                     │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │       ┌─────────┐         ┌─────────┐         ┌─────────┐     │
 │       │ CPU #0  │         │ CPU #1  │         │ CPU #2  │     │
 │       │(Registers)        │(Registers)        │(Registers)    │
 │       └───┬─────┘         └───┬─────┘         └───┬─────┘     │
 │       ┌───┴─────┐         ┌───┴─────┐         ┌───┴─────┐     │
 │       │ L1 Cache│         │ L1 Cache│         │ L1 Cache│     │
 │       └────┬────┘         └────┬────┘         └────┬────┘     │
 │            │                   │                   │          │
 │  ══════════╧═══════════════════╧═══════════════════╧════════  │
 │                        System Bus (공유 버스)                   │
 │  ═════════════════════════════════╤═════════════════════════  │
 │                                   │                           │
 │                           ┌───────┴───────┐                   │
 │                           │  Main Memory  │                   │
 │                           │ (공유 메모리)   │                   │
 │                           └───────────────┘                   │
 │                                                               │
 │  * SMP 특징: 모든 CPU가 대등한 권한을 가지며 동일한 물리 메모리에 직접 접근 │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SMP(Symmetric Multiprocessing) 구조에서는 각 CPU 코어가 자신만의 캐시(L1/L2)를 가지면서도 메인 메모리는 시스템 버스를 통해 공동으로 소유한다. 운영체제는 특정 CPU에 종속되지 않고 어떤 코어에서든 실행될 수 있다. 이 구조의 가장 큰 과제는 CPU #0이 캐시의 값을 변경했을 때, CPU #1이 그 변경 사실을 즉각 알아채야 하는 **캐시 일관성(Cache Coherence)** 문제다.

### 다중 처리 환경에서의 OS 핵심 역할

| OS 역할 | 해결해야 할 문제 | 해결 기법 (예시) |
|:---|:---|:---|
| **동기화 (Synchronization)** | 여러 코어가 동시에 OS 커널 자료구조(예: 준비 큐)를 수정하려 할 때 데이터 꼬임 방지 | 스핀락(Spinlock), 세마포어, 원자적 연산 |
| **부하 균등화 (Load Balancing)** | CPU 0은 일이 너무 많고 CPU 1은 노는 상황 방지 | Push Migration(바쁜 쪽이 넘김), Pull Migration(노는 쪽이 가져옴) |
| **프로세서 친화성 (CPU Affinity)** | 프로세스가 다른 코어로 이주하면 기존 코어에 쌓여있던 캐시 데이터가 무용지물이 되는 현상 최소화 | Soft/Hard Affinity, 동일 코어 재할당 정책 |

## Ⅲ. 융합 비교 및 다각도 분석

### 대칭형 다중 처리 (SMP) vs 비대칭형 다중 처리 (ASMP)

| 비교 항목 | 대칭형 다중 처리 (SMP, Symmetric) | 비대칭형 다중 처리 (ASMP, Asymmetric) |
|:---|:---|:---|
| **권한 관계** | 모든 프로세서가 동등 (Peer-to-Peer) | 주(Master) 프로세서와 종(Slave) 프로세서 구분 |
| **OS 실행** | 모든 코어가 OS 커널 코드를 실행할 수 있음 | 주 프로세서만 OS 관리/I/O 수행, 종은 연산만 |
| **장점** | 강력한 결함 허용(하나 죽어도 문제없음), 병목 감소 | 구조가 단순, 동기화(데이터 충돌) 문제 적음 |
| **단점** | 복잡한 동기화 및 캐시 일관성 설계 필요 | 주 프로세서 과부하 시 전체 시스템 병목 발생 |
| **적용 사례** | Windows, Linux, macOS 등 현대 범용 OS | 초창기 다중 처리기, 특수 목적 I/O 가속 시스템 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: NUMA (Non-Uniform Memory Access) 아키텍처의 도입
- **상황**: SMP 시스템에서 코어 개수가 16개, 32개, 64개로 급격히 증가하자, 모든 코어가 하나의 시스템 버스를 통해 단일 메인 메모리에 접근하는 과정에서 병목 현상(Bus Contention)이 발생하여 코어를 늘려도 성능이 향상되지 않았다.
- **판단 (운영체제의 대응)**: 하드웨어 아키텍처가 NUMA로 진화함에 따라 OS도 변화했다. NUMA에서는 여러 코어가 각자의 '로컬 메모리' 모음을 갖는다. 다른 코어의 메모리(원격 메모리)에 접근할 수는 있지만 속도가 느리다.
- **기술사적 OS 튜닝**: OS 메모리 할당기는 프로세스 스케줄링 시, 해당 프로세스가 실행되는 코어와 가장 가까운 로컬 메모리 노드에서 물리적 페이지를 우선적으로 할당하도록(NUMA-aware Memory Allocation) 정책을 수정하여 병목을 극복했다.

### 안티패턴: 과도한 스레드 이동 (Ping-ponging)
- 부하 균등화(Load Balancing)에 너무 집착하여 운영체제가 프로세스를 코어 A에서 코어 B로 끊임없이 이동시키면, 각 코어의 L1/L2 캐시에 쌓인 유효 데이터(Cache Warm-up)가 초기화되는 현상이 반복된다. 이를 캐시 콜드(Cache Cold) 상태라고 하며, 스케줄링 최적화가 오히려 시스템 처리량을 급감시키는 결과를 낳는다. CPU Affinity를 적절히 보장하는 것이 핵심이다.

## Ⅴ. 기대효과 및 결론

- 다중 처리 시스템은 칩 내 트랜지스터 집적도를 속도(클럭) 증가 대신 코어 수 증가로 전환한 멀티코어 시대에 맞추어, 물리적 병렬성을 통한 극적인 처리량 향상을 달성했다.
- 이 과정에서 운영체제는 단순히 프로세스를 관리하는 수준을 넘어, 캐시 일관성 관리, 락 경합 방지, NUMA 인지형 스케줄링 등 병렬 컴퓨팅 아키텍처의 한계를 소프트웨어적으로 극복하는 최고 난도의 시스템 공학 결정체로 진화하였다.
