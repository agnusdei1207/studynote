# 핫 스탠바이 / 콜드 스탠바이 (Hot Standby / Cold Standby)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스탠바이(Standby) 시스템은 주 시스템 장애 시 서비스 연속성을 보장하기 위해 미리 준비된 예비 시스템으로, 핫 스탠바이는 실시간 동기화, 콜드 스탠바이는 필요 시 구동 방식이다.
> 2. **가치**: 핫 스탠바이는 **RTO(복구 시간) 최소화**를, 콜드 스탠바이는 **비용 최소화**를 제공하여 비즈니스 요구사항에 따른 최적의 가용성 전략 선택이 가능하다.
> 3. **융합**: 클라우드 기반 DR(Disaster Recovery), 멀티 리전 배포, 그리고 글로벌 Active-Active 아키텍처는 스탠바이 개념의 현대적 확장이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 정의
- **핫 스탠바이(Hot Standby)**: 주 시스템(Primary)과 동일한 구성으로 항상 가동 중이며, 실시간 또는 준실시간 데이터 동기화를 유지하는 대기 시스템이다. 장애 발생 시 즉시 서비스 인계가 가능하다.
- **콜드 스탠바이(Cold Standby)**: 평소에는 전원이 꺼진 상태로 대기하다가, 주 시스템 장애 시 수동으로 구동하여 서비스를 인계하는 방식이다. 데이터 복원이 필요하다.

### 2. 💡 비유: 소방차와 예비 소방관
- **핫 스탠바이**는 **"대기 중인 소방차"**와 같다. 항상 시동을 걸어두고 있어 화재 신호가 오면 즉시 출동한다.
- **콜드 스탠바이**는 **"예비 소방관"**과 같다. 평소에는 일상생활을 하다가 비상 호출 시 소방서로 이동하여 장비를 착용하고 출동한다.

### 3. 등장 배경
- **비즈니스 연속성**: IT 서비스 중단이 비즈니스에 미치는 영향 증대
- **재해 대비**: 자연재해, 화재, 정전 등 물리적 재해 위험
- **규제 준수**: 금융권, 의료 등 높은 가용성 요구 규정
- **SLA 달성**: 99.9%, 99.99% 등 높은 가용성 SLA 충족 필요

### 4. 스탠바이 유형 스펙트럼

| 유형 | 상태 | 동기화 | 전환 시간 | 비용 | 적용 |
|:---|:---|:---|:---|:---|:---|
| **Hot Standby** | 항상 가동 | 실시간 | 초~분 | 높음 | 미션 크리티컬 |
| **Warm Standby** | 가동 (부하 없음) | 주기적 | 분~시간 | 중간 | 일반 업무 |
| **Cold Standby** | 전원 OFF | 백업 테이프 | 시간~일 | 낮음 | 개발/테스트 |

- **📢 섹션 요약 비유**: 핫 스탠바이는 대기 타자 선수처럼 항상 경기에 투입될 준비가 되어 있고, 콜드 스탠바이는 외부의 후보 선수처럼 호출되면 준비해서 들어와야 합니다.

### 4. 데이터 동기화 방식

```text
    ┌─────────────────────────────────────────────────────────────────────────┐
    │                 Data Synchronization Methods                            │
    │                                                                         │
    │  [Synchronous Replication]                                             │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  Primary                              Standby                     │ │
    │  │    │ Write                              │                         │ │
    │  │    │───────┐                            │                         │ │
    │  │    │       │                            │                         │ │
    │  │    │       ▼                            │                         │ │
    │  │    │   ┌───────────┐    Replicate      │                         │ │
    │  │    │   │ Local DB  │ ─────────────────▶│ Remote DB               │ │
    │  │    │   └───────────┘                    │                         │ │
    │  │    │       │                            │                         │ │
    │  │    │       │◀───────────────────────────│ ACK                     │ │
    │  │    │       │                            │                         │ │
    │  │    │   Commit OK                        │                         │ │
    │  │    ▼                                    │                         │ │
    │  │  [강한 일관성, 높은 지연, 데이터 손실 없음]                         │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    │                                                                         │
    │  [Asynchronous Replication]                                            │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  Primary                              Standby                     │ │
    │  │    │ Write                              │                         │ │
    │  │    │───────┐                            │                         │ │
    │  │    │       ▼                            │                         │ │
    │  │    │   ┌───────────┐    Replicate      │                         │ │
    │  │    │   │ Local DB  │ ─────────────────▶│ Remote DB               │ │
    │  │    │   └───────────┘    (Background)   │                         │ │
    │  │    │   Commit OK                        │                         │ │
    │  │    ▼                                    │                         │ │
    │  │  [약한 일관성, 낮은 지연, 약간의 데이터 손실 가능]                  │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    └─────────────────────────────────────────────────────────────────────────┘
```

**[동기화 방식 해설]**
1. **동기식(Synchronous)**: Primary가 쓰기 작업 후 Standby의 확인(ACK)을 받아야 커밋 완료. 데이터 무결성 보장, 지연 증가
2. **비동기식(Asynchronous)**: Primary가 로컬에 쓰고 즉시 커밋, Standby로는 백그라운드 전송. 빠르지만 장애 시 데이터 손실 가능
3. **선택 기준**: 금융거래는 동기식, 로그/캐시는 비동기식 권장

- **📢 섹션 요약 비유**: 핫 스탠바이는 언제든 출전할 준비를 하고 벤치에 앉은 1군 선수이고, 콜드 스탠바이는 2군 리그에서 훈련하다가 필요 시 1군으로 호출되는 선수입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 스탠바이 방식 비교

| 비교 항목 | Hot Standby | Warm Standby | Cold Standby |
|:---|:---|:---|:---|
| **Full Name** | 핫 스탠바이 | 웜 스탠바이 | 콜드 스탠바이 |
| **시스템 상태** | 항상 가동 | 가동 (유휴) | 전원 OFF |
| **데이터 동기화** | 실시간 | 주기적 (분~시간) | 백업 테이프 |
| **RTO** | 초 ~ 분 | 분 ~ 시간 | 시간 ~ 일 |
| **RPO** | 0 ~ 분 | 분 ~ 시간 | 시간 ~ 일 |
| **비용** | 높음 (100%+) | 중간 (50-80%) | 낮음 (20-40%) |
| **자동화** | 완전 자동 | 반자동 | 수동 |
| **복잡도** | 높음 | 중간 | 낮음 |

### 2. 비용 vs 복구 시간 트레이드오프

```text
    비용
      ▲
      │
   100%├─────● Hot Standby (실시간 복제, 즉시 전환)
      │      ╲
    80%│       ╲ Warm Standby (주기적 동기화, 빠른 전환)
      │        ╲
    50%│         ╲
      │          ╲ Cold Standby (백업 복원, 수동 전환)
    20%│           ●
      │
      └──────────────────────────────────────────▶ 복구 시간
         1분      1시간      4시간      24시간
```

### 3. 과목 융합 관점

| 융합 과목 | 연관 개념 | 융합 포인트 |
|:---|:---|:---|
| **데이터베이스** | 복제, 로그 전달 | DB 동기화 메커니즘 |
| **네트워크** | VIP, DNS 페일오버 | 트래픽 전환 메커니즘 |
| **스토리지** | SAN, 복제 | 공유 스토리지, 데이터 동기화 |
| **클라우드** | 리전, AZ | 멀티 AZ/리전 DR |
| **보안** | 접근 제어, 암호화 | DR 사이트 보안 |

- **📢 섹션 요약 비유**: 스탠바이 선택은 보험 선택과 같습니다. 종합 보험(핫 스탠바이)은 비싸지만 완벽한 보장을, 기본 보험(콜드 스탠바이)은 저렴하지만 제한적 보장을 제공합니다.
