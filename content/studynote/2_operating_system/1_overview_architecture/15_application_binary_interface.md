## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ABI(Application Binary Interface)는 운영체제와 응용 프로그램(또는 라이브러리 간)이 상호작용하는 규약으로, 소스 코드가 아닌 **컴파일된 바이너리(기계어) 수준**에서의 기계적 호출 규칙이다.
> 2. **가치**: 데이터 크기, 레지스터 사용 규약, 스택 프레임 구조 등을 명확히 정의하여, 서로 다른 컴파일러로 빌드된 라이브러리 파일이나 프로그램이 하나의 시스템에서 충돌 없이 연동될 수 있도록 보장한다.
> 3. **융합**: 소스 코드 호환성을 다루는 API와 달리, ABI는 하드웨어 아키텍처(x86, ARM) 및 운영체제에 절대적으로 종속되며, 리버스 엔지니어링이나 JIT 컴파일러 개발 시 반드시 숙지해야 하는 핵심 규약이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: ABI(응용 프로그램 이진 인터페이스)는 기계어 수준에서 프로그램 모듈들이 어떻게 소통할 것인지를 정한 저수준 약속이다. 함수를 호출할 때 인자를 어느 레지스터에 넣을지, 리턴값은 어떻게 받을지, 시스템 콜 번호는 무엇인지 등을 0과 1의 관점에서 규정한다.
- **💡 비유**: API가 서로 다른 국적의 사람들을 위한 **공용어 사전(문법)**이라면, ABI는 공장에 있는 톱니바퀴들이 서로 맞물려 돌아가기 위한 **규격화된 톱니의 모양과 굵기(물리적 치수)**다. 사전(API)이 같아도 톱니의 치수(ABI)가 다르면 기계는 헛돌거나 박살 난다.
- **등장 배경**: 과거에는 같은 C 언어 코드라도 A 회사 컴파일러로 만든 라이브러리(`.dll`, `.so`)를 B 회사 컴파일러로 만든 프로그램에서 가져다 쓰면 크래시가 났다. (함수 파라미터를 넘기는 순서가 달랐기 때문). 이를 해결하기 위해 운영체제와 하드웨어 벤더가 "우리 환경에서는 기계어 레벨에서 이런 규칙으로 데이터를 주고받자"라고 표준 ABI를 제정하게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### API와 ABI의 적용 시점 비교

API는 개발자가 코드를 '작성할 때' 중요하고, ABI는 그 코드가 '컴파일되고 실행될 때' 중요하다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               소스코드에서 기계어까지: API vs ABI                 │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │   [ 소스 코드 (Source Code) ]                                   │
 │       printf("Hello"); ◀──────────────── API 계층 (POSIX 등) │
 │                                          (인간이 읽는 규칙)    │
 │           │                                                   │
 │           ▼  (컴파일러 GCC, Clang)                              │
 │                                                               │
 │   [ 어셈블리 / 기계어 (Binary) ]                                 │
 │       mov rdi, 0x400010  (문자열 주소)                          │
 │       call printf@plt                                         │
 │       ...                ◀──────────────── ABI 계층 (x86-64)   │
 │                                          (기계가 읽는 규칙)    │
 │           │                                                   │
 │           ▼  (OS 로더 및 링커)                                  │
 │                                                               │
 │   [ 운영체제 커널 (OS Kernel) / 하드웨어 ]                       │
 └───────────────────────────────────────────────────────────────┘
```

### ABI가 규정하는 핵심 요소들

ABI 규약 매뉴얼(예: System V AMD64 ABI)에는 다음과 같은 매우 하드웨어 종속적인 규칙들이 빼곡히 정의되어 있다.

1. **레지스터 호출 규약 (Calling Convention)**: 
   - 함수 A가 함수 B를 부를 때 인자 1, 2, 3을 어떻게 전달할 것인가?
   - 예 (Linux x86-64): 1번째 인자는 `rdi`, 2번째는 `rsi`, 3번째는 `rdx`에 넣는다. 리턴값은 `rax`에 담는다. (윈도우 x64 ABI는 이 순서가 다름!)
2. **데이터 타입의 크기, 정렬 (Size & Alignment)**: 
   - `int`는 4바이트인가 8바이트인가? 메모리에 올릴 때 4바이트 경계에 맞춰 정렬(Alignment)할 것인가?
3. **시스템 콜 인터페이스 (System Call Interface)**:
   - 커널에 진입할 때 트랩 인터럽트 번호는 무엇인가? (`int 0x80` 인가 `syscall` 명령어인가?)
4. **객체 파일 포맷 (Object File Format)**: 
   - 컴파일된 실행 파일이 Linux의 ELF 형식인가, Windows의 PE 형식인가?

## Ⅲ. 융합 비교 및 다각도 분석

### API vs ABI 명확한 비교

| 비교 기준 | API (Application Programming Interface) | ABI (Application Binary Interface) |
|:---|:---|:---|
| **대상 수준** | 소스 코드 (Source Code) 레벨 | 바이너리 (기계어, Machine Code) 레벨 |
| **주 사용처** | 프로그래머가 코드 작성 및 컴파일 시 | 컴파일러, 링커, OS가 바이너리 실행 시 |
| **종속성** | OS 간 어느 정도 표준화 (POSIX 등) | **OS 및 하드웨어(CPU 아키텍처)에 완전 종속** |
| **호환성 의미** | 소스를 수정하지 않고 재컴파일 가능 (Source Compatible) | **재컴파일 없이** 기존 실행 파일을 그대로 실행 가능 (Binary Compatible) |
| **깨졌을 때** | 컴파일 에러 발생 (`error: undeclared identifier`) | 런타임 크래시 발생 (`Segmentation Fault`, 쓰레기값 출력) |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 애플 실리콘(M1/M2) 전환 시 로제타(Rosetta 2)의 마법
- **상황**: 애플이 Mac의 CPU를 인텔(x86_64)에서 자사 칩인 M1(ARM64)으로 완전히 교체했다. 인텔용으로 컴파일된 수백만 개의 기존 카카오톡, 포토샵 실행 파일들은 ARM 환경에서 ABI(명령어 구조, 레지스터가 완전 다름)가 맞지 않아 실행이 불가능한 사태가 발생했다.
- **판단 및 해결**: 기존 코드를 모두 다시 컴파일(API 호환성)하라고 개발자들에게 강요하면 생태계가 붕괴한다. 애플은 **Rosetta 2**라는 동적 바이너리 변환(Binary Translation) 기술을 OS 로더에 탑재했다. 
- **원리**: 인텔 x86_64 ABI 규격으로 만들어진 기계어를, 실행 직전에 ARM64 ABI 규격의 기계어로 실시간 번역하여 실행시켰다. 이로 인해 개발자는 코드를 고치지 않아도 기존 앱이 새 하드웨어에서 문제없이 돌아가는(Binary Compatibility 우회) 기적을 선보였다.

### 안티패턴 (C++ 맹글링 문제)
- 외부 라이브러리(DLL)를 제공할 때 C++ 클래스나 오버로딩된 함수를 그대로 노출(Export)하는 것. C++는 컴파일러마다 함수 이름을 기계어로 뭉개버리는(Name Mangling) 방식, 즉 C++ ABI가 제각각이다. MSVC로 빌드한 C++ DLL을 GCC에서 불러 쓰면 함수 이름을 찾지 못해 링크 에러가 발생한다. 
- **해법**: 외부 인터페이스는 반드시 C 언어 규격(`extern "C"`)으로 감싸서 표준 C ABI를 따르도록 노출해야 이기종 컴파일러 간 충돌이 없다.

## Ⅴ. 기대효과 및 결론

- ABI는 우리가 인터넷에서 다운받은 실행 파일(`.exe`나 리눅스 실행파일)이 우리 컴퓨터에서 바로 더블클릭만으로 실행될 수 있게 해주는 저수준의 숨은 공로자다.
- 프로그래머가 평소에 ABI를 의식할 일은 거의 없지만, 커널 모듈을 작성하거나, 크로스 플랫폼 라이브러리를 배포하거나, 악성코드를 분석(리버싱)할 때는 반드시 숙지해야 하는 가장 기저(Base)의 하드웨어-소프트웨어 약속이다.
