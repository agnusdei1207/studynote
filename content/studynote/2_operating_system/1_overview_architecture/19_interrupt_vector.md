# 인터럽트 벡터 (Interrupt Vector)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터럽트 벡터 (Interrupt Vector)는 특정 인터럽트 (Interrupt) 발생 시 이를 처리할 ISR (Interrupt Service Routine)의 시작 주소를 담고 있는 메모리 내의 고정된 매핑 테이블 또는 배열이다.
> 2. **가치**: 수백 가지의 다양한 인터럽트 (Interrupt) 종류에 대해 CPU (Central Processing Unit)가 하드웨어적으로 즉각적인 처리 루틴을 찾아가게 함으로써, 소프트웨어적인 조건문 분기 (if-else) 오버헤드 없이 나노초 단위의 응답 속도를 제공한다.
> 3. **융합**: x86 아키텍처의 IDT (Interrupt Descriptor Table)나 ARM의 Vector Table과 같은 하드웨어 구조체로 실체화되며, 이는 시스템 부팅 단계인 부트스트랩 (Bootstrap) 과정에서 OS (Operating System) 커널에 의해 가장 먼저 설정되어야 하는 시스템의 핵심 이정표다.

## Ⅰ. 개요 (Context & Background)

- **개념**: 인터럽트 벡터 (Interrupt Vector)는 인터럽트 번호와 해당 인터럽트의 처리 코드 (ISR) 주소를 연결해주는 '연락처 목록'이다. 인터럽트 (Interrupt)가 발생하면 CPU (Central Processing Unit)는 발생한 인터럽트 (Interrupt)의 번호 (Vector Number)를 인덱스로 사용하여 이 테이블에서 주소를 읽어온 뒤, 즉시 그 지점으로 점프한다.
- **💡 비유**: 인터럽트 벡터 (Interrupt Vector)는 건물의 **'층별 안내 표지판'**과 같다. 화재 발생 (인터럽트 1번), 도둑 침입 (인터럽트 2번) 등 사건이 발생했을 때, 안내판 (인터럽트 벡터)을 보고 1번은 119 소방대, 2번은 경찰서라는 대응 주소 (ISR)를 즉시 찾아가는 것과 같다. 표지판이 없다면 범인을 잡기 위해 모든 층을 일일이 돌아다녀야 할 것이다.
- **등장 배경**: 초기 컴퓨터는 인터럽트 (Interrupt)가 오면 단 하나의 공통 루틴으로 점프한 뒤, 소프트웨어가 "너 누구니?"라고 장치들에 일일이 물어봐야 했다. 장치가 많아질수록 이 확인 시간이 길어지는 문제를 해결하기 위해, 하드웨어 번호에 따라 직접 분기하는 벡터 방식이 도입되었다.

- **📢 섹션 요약 비유**: 복잡한 내선 번호마다 담당자가 정해져 있는 기업용 전화 교환 시스템 (벡터 방식)이 개인 가정의 단일 전화 (비벡터 방식)보다 훨씬 빠른 응대 속도를 갖는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 인터럽트 벡터 테이블 (IVT / IDT) 아키텍처

CPU (Central Processing Unit) 아키텍처에 따라 메모리의 특정 위치 (예: x86의 낮은 메모리 주소 또는 특수 레지스터가 가리키는 곳)에 이 테이블이 위치한다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               인터럽트 벡터 테이블 (Interrupt Vector Table) 구조    │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │  [ CPU 내부 ]                 [ 메모리 내 벡터 테이블 (IDT) ]     │
 │  Interrupt Number ─────┐       ┌───────────────────────────┐  │
 │  (예: 14번 Page Fault) │       │ Vector 0 : Divide Error   │  │
 │                        │       ├───────────────────────────┤  │
 │          ▼             │       │ Vector 1 : Debug          │  │
 │  ┌─────────────────┐   │       ├───────────────────────────┤  │
 │  │ IDTR Register  │───┼──────▶│          ...              │  │
 │  └─────────────────┘   │       ├───────────────────────────┤  │
 │    (테이블 시작 주소)     └──────▶│ Vector 14: Page Fault ISR │  │
 │                                │ (주소: 0x8048A00)         │  │
 │                                ├───────────────────────────┤  │
 │                                │          ...              │  │
 │                                ├───────────────────────────┤  │
 │                                │ Vector 255: User Defined  │  │
 │                                └─────────────┬─────────────┘  │
 │                                              │                │
 │  [ 실제 처리 루틴 실행 ]                        │                │
 │   - CPU는 0x8048A00 으로 점프하여 ISR 실행 ◀───┘                │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 인터럽트 (Interrupt)가 발생하면 CPU (Central Processing Unit)는 하드웨어적으로 인터럽트 번호 (Vector)를 인식한다. 그 후 IDTR (Interrupt Descriptor Table Register)과 같은 특수 레지스터에 저장된 테이블의 시작 주소를 기준으로 `시작 주소 + (번호 * 엔트리 크기)`를 계산하여 메모리에 접근한다. 여기서 읽어낸 4~16바이트의 데이터가 바로 다음에 실행할 ISR (Interrupt Service Routine)의 절대 주소다. 이 과정은 하드웨어 회로 수준에서 이루어지므로 소프트웨어 개입 없이 극도로 빠르게 처리된다.

### 인터럽트 벡터 (Interrupt Vector)의 주요 필드 구성 (x86 IDT 기준)

단순한 주소 외에도 보안과 상태 제어를 위한 추가 정보가 포함된다.

| 필드명 | 역할 | 상세 내용 | 비유 |
|:---|:---|:---|:---|
| **Offset** | ISR 주소 | 실제로 점프할 커널 코드의 메모리 주소 | 목적지 번지수 |
| **Selector** | 코드 세그먼트 | 실행할 코드가 속한 세그먼트 (권한 레벨 결정) | 소속 부서 |
| **DPL** | 권한 레벨 | 이 인터럽트를 호출할 수 있는 최소 권한 (Ring 0~3) | 출입 보안 등급 |
| **P (Present)** | 유효 플래그 | 해당 엔트리가 현재 메모리에 로드되어 있는지 여부 | 거주 여부 |
| **Type** | 게이트 종류 | 인터럽트 게이트 (IF 클리어) 또는 트랩 게이트 구분 | 문 종류 (자동문/수동문) |

- **📢 섹션 요약 비유**: 전화번호부 (벡터 테이블)에 이름과 번호 (주소)만 있는 것이 아니라, 그 사람의 직급 (권한)과 현재 통화 가능 여부 (P 비트)까지 한눈에 보이는 스마트 주소록과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### x86 (IDT) vs ARM (Vector Table) 비교 분석

| 비교 항목 | x86 (Interrupt Descriptor Table) | ARM (Vector Table) |
|:---|:---|:---|
| **위치 가변성** | IDTR 레지스터로 어디든 지정 가능 | 보통 0x00 또는 하이 벡터 주소 고정 |
| **엔트리 크기** | 8 Byte 또는 16 Byte (복잡한 구조) | 4 Byte (보통 점프 명령어 `LDR` 상주) |
| **번호 범위** | 0 ~ 255 (총 256개) | Reset, Abort 등 7~8개 핵심 벡터 중심 |
| **확장 방식** | 모든 인터럽트가 개별 벡터 가짐 | 외부 인터럽트는 IRQ 벡터 하나로 공유 (SW 분기) |

### 융합 관점: 부팅 과정과 인터럽트 벡터 (Interrupt Vector)
OS (Operating System) 커널이 부팅될 때 가장 먼저 하는 일 중 하나가 **'인터럽트 벡터 테이블 초기화'**다. 만약 테이블이 채워지기 전에 인터럽트 (Interrupt)가 발생하면 CPU (Central Processing Unit)는 어디로 점프할지 몰라 시스템을 즉시 중단 (Double Fault) 시킨다. 따라서 부트로더 (Bootloader)와 커널 초기화 코드는 이 테이블을 안전하게 구축하는 것이 최우선 과제다.

- **📢 섹션 요약 비유**: 신도시 (OS)가 건설될 때 도로가 뚫리기도 전에 이정표 (인터럽트 벡터)를 먼저 세워야 길을 잃는 사고 (시스템 크래시)를 막을 수 있는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 인터럽트 벡터 (Interrupt Vector) 부족 문제 (Vector Exhaustion)
- **상황**: 대규모 가상화 서버에서 수백 개의 가상 NIC (Network Interface Card)와 디스크 장치를 연결하려 할 때, x86이 제공하는 256개의 벡터 번호가 모자라는 상황이 발생함. (0~31번은 예외용으로 이미 예약됨)
- **판단**: 물리적 벡터 번호는 한정된 자원이므로 효율적 관리가 필요함.
- **해결책**: 하나의 벡터 번호를 여러 장치가 공유하게 하는 **Interrupt Sharing** 기술을 적용하거나, 최신 서버에서는 **MSI-X (Message Signaled Interrupts)** 방식을 사용하여 전통적인 핀 기반 인터럽트 (Interrupt) 대신 메시지 기반으로 벡터 공간을 확장하여 해결해야 한다.

### 도입 체크리스트 및 안티패턴
- **체크리스트**: 커널 드라이버 작성 시 할당받은 벡터 번호가 다른 장치와 충돌하지 않는지 `/proc/interrupts`를 통해 반드시 확인하라.
- **안티패턴**: 보안 강화를 위해 모든 인터럽트 벡터의 DPL (Privilege Level)을 0으로 설정하는 행위. `int 3` (디버깅)이나 시스템 콜 (System Call) 벡터의 DPL을 0으로 설정하면, 사용자 프로그램이 디버깅이나 OS (Operating System) 서비스를 아예 이용할 수 없게 되어 시스템이 무용지물이 된다.

- **📢 섹션 요약 비유**: 아파트의 모든 인터럽트 벨 (벡터)에 지문 인식 (권한 0)을 걸어버리면, 정작 벨을 눌러야 할 택배 기사 (사용자 프로그램)가 벨을 누를 수 없어 택배를 못 받는 상황과 같습니다.

## Ⅴ. 기대효과 및 결론

### 인터럽트 벡터 (Interrupt Vector) 도입의 기대효과

| 구분 | 도입 전 (비벡터 방식) | 도입 후 (벡터 방식) | 개선 효과 |
|:---|:---|:---|:---|
| **응답성** | 모든 장치를 전수 조사 | 주소로 즉시 점프 | 인터럽트 처리 시간 70% 단축 |
| **코드 구조** | 거대한 `switch` 문 필요 | 모듈화된 ISR 관리 가능 | 커널 코드 가독성 및 유지보수성 향상 |
| **보안성** | 비공식 진입로 다수 존재 | 통제된 벡터 테이블만 허용 | 시스템 진입 보안 강화 |

- **결론**: 인터럽트 벡터 (Interrupt Vector)는 하드웨어의 무질서한 신호를 소프트웨어의 질서 있는 실행으로 연결하는 **'디지털 신호등'**이다. 이 작고 정밀한 테이블이 존재하기에 현대 컴퓨터는 수만 개의 비동기적 사건 속에서도 길을 잃지 않고 각각의 사건에 알맞은 해결사를 신속하게 파견할 수 있다.

- **📢 섹션 요약 비유**: 119 상황실에서 신고 전화 (인터럽트)가 오자마자 지도의 위치 (벡터 번호)를 확인하고 가장 가까운 소방서 (ISR)를 즉시 매칭하여 출동시키는 최첨단 관제 시스템과 같습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **인터럽트 서비스 루틴 (ISR)**: 벡터 테이블이 가리키는 실제 처리 코드.
- **IDT (Interrupt Descriptor Table)**: x86 계열에서 인터럽트 벡터 (Interrupt Vector)를 실체화한 하드웨어 구조체.
- **시스템 호출 (System Call)**: 특정 벡터 번호 (예: 0x80)를 약속하여 사용하는 소프트웨어 인터럽트 행위.
- **부트스트랩 (Bootstrap)**: 인터럽트 벡터 (Interrupt Vector)를 최초로 구축하는 시스템 초기화 과정.
