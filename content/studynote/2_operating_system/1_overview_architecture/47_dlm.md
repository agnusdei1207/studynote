# 분산 잠금 관리자 (DLM - Distributed Lock Manager)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DLM(Distributed Lock Manager)은 클러스터 환경에서 여러 노드가 공유 자원에 동시 접근할 때 데이터 일관성을 보장하는 분산 동기화 메커니즘이다.
> 2. **가치**: DLM은 **분산 환경의 교착상태(Deadlock) 방지**, **데이터 무결성 보장**, 그리고 **효율적인 동시성 제어**를 통해 고가용성 클러스터의 핵심 기반을 제공한다.
> 3. **융합**: Oracle RAC, GFS2, OCFS2, 그리고 분산 데이터베이스의 2PC(Two-Phase Commit) 모두 DLM 개념을 기반으로 구현된다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 정의
- **분산 잠금 관리자(DLM, Distributed Lock Manager)**: 네트워크로 연결된 여러 노드가 공유 자원(파일, 데이터베이스 레코드, 메모리 영역 등)에 접근할 때, 상호 배제(Mutual Exclusion)와 동시성 제어를 제공하는 시스템 서비스다.

### 2. 💡 비유: 도서관 대출 시스템
- DLM은 **"도서관의 책 대출 시스템"**과 같다.
- 한 사람이 책을 대출하면(잠금 획득), 다른 사람은 그 책을 대출할 수 없다. 책을 반납하면(잠금 해제) 다음 대기자가 대출할 수 있다. 여러 지점(노드)이 있어도 중앙 시스템이 대출 상태를 관리한다.

### 3. 등장 배경
- **공유 스토리지 클러스터**: 여러 노드가 동일 디스크에 동시 접근
- **분산 데이터베이스**: 다중 노드에서 동일 레코드 갱신
- **병렬 파일 시스템**: GFS2, OCFS2 등의 메타데이터 동기화
- **HA 클러스터**: Pacemaker, Corosync의 리소스 관리

### 4. 잠금의 분류

| 잠금 모드 | Full Name | 설명 | 호환 가능 모드 |
|:---|:---|:---|:---|
| **NL** | Null Lock (널 락) | 잠금 없음 | 모두 |
| **CR** | Concurrent Read (동시 읽기) | 읽기 전용, 다른 쓰기 허용 | NL, CR, CW, PR |
| **CW** | Concurrent Write (동시 쓰기) | 읽기/쓰기, 다른 읽기/쓰기 허용 | NL, CR, CW |
| **PR** | Protected Read (보호 읽기) | 읽기 전용, 다른 쓰기 차단 | NL, CR, PR |
| **PW** | Protected Write (보호 쓰기) | 읽기/쓰기, 다른 쓰기 차단 | NL, CR |
| **EX** | Exclusive (배타적) | 독점 접근 | NL |

- **📢 섹션 요약 비유**: DLM은 여러 식구가 하나의 욕실을 사용할 때 누가 먼저 쓸지 정해주는 가족 규칙과 같습니다. 문을 잠그면(Exclusive Lock) 다른 사람은 기다려야 하죠.

### 2. 구성 요소 상세

| 구성 요소 | Full Name | 역할 | 특징 |
|:---|:---|:---|:---|
| **DLM Agent** | DLM 에이전트 | 로컬 노드의 잠금 요청 처리 | 커널 모듈 또는 데몬 |
| **Lock Table** | 잠금 테이블 | 잠금 상태 분산 저장 | 해시 기반 분산 |
| **Lock Master** | 잠금 마스터 | 특정 자원의 잠금 관리 책임 | 동적 마스터 선정 |
| **Lock Queue** | 잠금 큐 | 대기 중인 요청 관리 | FIFO 또는 우선순위 |
| **Recovery** | 복구 메커니즘 | 노드 장애 시 잠금 복구 | 타임아웃 + 재할당 |

### 3. 잠금 요청 처리 흐름

```text
    ┌─────────────────────────────────────────────────────────────────────────┐
    │                    Lock Request Processing Flow                         │
    │                                                                         │
    │  [1] 잠금 요청 (Node 1 → DLM)                                          │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  Node 1: Lock Request (Resource: File_A, Mode: EX)               │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    │                                                                         │
    │  [2] 잠금 마스터 조회                                                   │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  DLM: Hash(File_A) → Master Node = Node 2                         │ │
    │  │  (잠금 마스터는 자원별로 정적으로 또는 동적으로 결정)             │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    │                                                                         │
    │  [3] 호환성 검사                                                        │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  Current Lock State:                                               │ │
    │  │  ┌─────────────────────────────────────────────────────────────┐ │ │
    │  │  │  Resource: File_A                                            │ │ │
    │  │  │  Current Mode: PR (Protected Read)                          │ │ │
    │  │  │  Owners: Node 2, Node 3                                     │ │ │
    │  │  │  Requested: EX (Exclusive) by Node 1                        │ │ │
    │  │  │  Compatible? NO → Queue Request                             │ │ │
    │  │  └─────────────────────────────────────────────────────────────┘ │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    │                                                                         │
    │  [4] 잠금 대기 또는 즉시 획득                                           │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  Case A: 호환 → 즉시 획득                                         │ │
    │  │  Case B: 비호환 → 큐에서 대기                                     │ │
    │  │  Case C: 변환 요청 (Conversion) → 기존 잠금 모드 변경             │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    │                                                                         │
    │  [5] 잠금 해제 및 대기자 깨우기                                         │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  Node 2: Unlock(File_A)                                          │ │
    │  │  DLM: Check Queue → Wake Node 1                                  │ │
    │  │  Node 1: Lock Granted (EX mode)                                  │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    └─────────────────────────────────────────────────────────────────────────┘
```

### 4. 교착상태 탐지 및 해결

```text
    ┌─────────────────────────────────────────────────────────────────────────┐
    │                   Deadlock Detection in DLM                             │
    │                                                                         │
    │  [교착상태 시나리오]                                                    │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  Node 1: Holds Lock(A), Waits for Lock(B)                        │ │
    │  │  Node 2: Holds Lock(B), Waits for Lock(A)                        │ │
    │  │                                                                   │ │
    │  │       Node 1                Node 2                               │ │
    │  │       ┌──────┐              ┌──────┐                             │ │
    │  │       │ A    │◀─────────────│ B    │  Waits for A                │ │
    │  │       │(Held)│              │(Held)│                             │ │
    │  │       └──────┘              └──────┘                             │ │
    │  │          │                      ▲                                 │ │
    │  │          │ Waits for B          │                                 │ │
    │  │          └──────────────────────┘                                 │ │
    │  │                     CYCLE = DEADLOCK!                             │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    │                                                                         │
    │  [DLM 교착상태 해결]                                                    │
    │  ┌───────────────────────────────────────────────────────────────────┐ │
    │  │  1. 타임아웃 기반 탐지: 일정 시간 대기 시 주기적 검사             │ │
    │  │  2. 대기 그래프 분석: 주기적으로 사이클 존재 여부 확인            │ │
    │  │  3. 희생자 선정: 가장 최근 요청 또는 우선순위 낮은 요청 선택      │ │
    │  │  4. 희생자 롤백: 선택된 요청 취소, 에러 반환                      │ │
    │  │  5. 재시도: 애플리케이션이 적절히 재시도                          │ │
    │  └───────────────────────────────────────────────────────────────────┘ │
    └─────────────────────────────────────────────────────────────────────────┘
```

**[DLM 교착상태 해설]**
1. **분산 교착상태**: 여러 노드에 걸친 사이클로, 중앙 탐지가 어려움
2. **타임아웃 활용**: 각 잠금 요청에 타임아웃 설정, 초과 시 교착상태 의심
3. **분산 탐지 알고리즘**: Chandy-Misra-Haas 등의 간선 추적 알고리즘
4. **희생자 선택**: 트랜잭션 비용, 우선순위, 진행 시간 등 고려

- **📢 섹션 요약 비유**: DLM은 교통 신호등처럼, 여러 차량(노드)이 교차로(공유 자원)에서 충돌하지 않도록 순서를 정해줍니다. 신호가 빨간색이면 기다리고, 초록색이면 진행합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. DLM 구현 비교

| 구현 | Full Name | 플랫폼 | 특징 | 적용 |
|:---|:---|:---|:---|:---|
| **Oracle DLM** | Oracle Distributed Lock Manager | Oracle RAC | ROWID 기반 세밀한 잠금 | Oracle DB 클러스터 |
| **GFS2 DLM** | Global File System 2 DLM | Linux | 커널 모듈, POSIX 호환 | Red Hat Cluster |
| **OCFS2** | Oracle Cluster File System 2 | Linux | 경량, 메타데이터 잠금 | Oracle DB 스토리지 |
| **Red Hat DLM** | Red Hat Distributed Lock Manager | Linux | Pacemaker 통합 | HA 클러스터 |
| **Zookeeper** | Apache Zookeeper | Cross-platform | CP 잠금, 레시피 기반 | 분산 코디네이터 |
| **etcd** | CoreOS etcd | Cross-platform | Raft 기반, Lease 잠금 | Kubernetes |

### 2. 중앙형 vs 분산형 DLM

| 구분 | 중앙형 DLM | 분산형 DLM |
|:---|:---|:---|
| **구조** | 단일 마스터 노드 | 모든 노드가 마스터 가능 |
| **확장성** | 제한적 (마스터 병목) | 높음 (분산 부하) |
| **지연** | 낮음 (단일 홉) | 가변적 (멀티 홉 가능) |
| **SPOF** | 마스터 장애 시 전체 영향 | 부분적 영향 |
| **복구** | 마스터 선출 필요 | 자동 복구 |
| **복잡도** | 낮음 | 높음 |

### 3. 과목 융합 관점

| 융합 과목 | 연관 개념 | 융합 포인트 |
|:---|:---|:---|
| **데이터베이스** | 트랜잭션, 2PL | 분산 DB의 동시성 제어 |
| **알고리즘** | 그래프, 위상 정렬 | 교착상태 탐지 |
| **네트워크** | 분산 합의, Paxos | 분산 상태 동기화 |
| **운영체제** | 세마포어, 뮤텍스 | 분산 확장 |
| **파일 시스템** | 메타데이터 잠금 | 클러스터 파일 시스템 |

- **📢 섹션 요약 비유**: DLM은 여러 지점이 있는 은행의 중앙 계좌 시스템과 같습니다. 한 지점에서 계좌에 접근하면 다른 지점에서도 그 상태를 알 수 있죠.

    │     리더는 주기적으로 Lease 갱신                                    │
    │     갱신 실패 시 잠금 자동 해제                                      │
    │                                                                     │
    │  [Kubernetes 구현]                                                   │
    │  ┌───────────────────────────────────────────────────────────────┐ │
    │  │  pkg/leaderelection/                                           │ │
    │  │  - LeaseLock: etcd 기반 잠금                                   │ │
    │  │  - ConfigMapLock: ConfigMap 기반 (레거시)                      │ │
    │  │  - Leader Election: 리더 선출 로직                             │ │
    │  └───────────────────────────────────────────────────────────────┘ │
    └─────────────────────────────────────────────────────────────────────┘

### 2. 도입 체크리스트

| 항목 | 체크 포인트 | 권장 사항 |
|:---|:---|:---|
| 잠금 세분성 | Row vs Block vs File | 충돌 빈도에 따라 선택 |
| 타임아웃 | 잠금 획득 대기 시간 | 적절한 타임아웃 설정 |

| 모니터링 | 잠금 대기 시간, 교착상태 | DLM 메트릭 수집 |
| 장애 복구 | 노드 장애 시 잠금 정리 | 타임아웃 기반 자동 해제 |

### 3. 안티패턴

| 안티패턴 | 문제점 | 올바른 접근 |
|:---|:---|:---|
| **잠금 장시간 보유** | 다른 노드 대기 증가 | 최소 범위, 최단 시간 |

### 2. 미래 전망
- **Software Transactional Memory (STM)**: 잠금 없는 동시성
- **CRDT 기반**: 충돌 없는 복제 데이터 타입
- **AI 기반 잠금 최적화**: 접근 패턴 학습 후 자동 튜닝

### 3. 참고 표준
- **DLM API**: POSIX 1003.1d (Draft)
- **Oracle RAC**: Oracle Grid Infrastructure
- **Red Hat Cluster**: DLM Kernel Module

- **📢 섹션 요약 비유**: DLM은 오케스트라 지휘자처럼, 여러 악기(노드)가 하나의 악보(데이터)를 동시에 연주할 때 조화를 이루도록 돕습니다.
