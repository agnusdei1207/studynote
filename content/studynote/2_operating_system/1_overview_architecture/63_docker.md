# 도커 (Docker) 아키텍처

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도커 (Docker)는 리눅스 커널의 격리 기술(Namespace, Cgroups)을 활용하여 애플리케이션을 신속하게 구축, 테스트 및 배포할 수 있게 돕는 **오픈 소스 컨테이너 플랫폼**이다.
> 2. **가치**: 클라이언트-서버 아키텍처를 기반으로 'Build once, Run anywhere' 철학을 실현하며, 레이어형 파일 시스템을 통해 이미지 관리 효율성을 극대화한다.
> 3. **융합**: 운영체제의 프로세스 관리 기능을 서비스 단위로 추상화하였으며, 현대 클라우드 네이티브 환경에서 컨테이너 표준(OCI)의 사실상 기준(De-facto Standard) 역할을 한다.

+++

## Ⅰ. 도커 아키텍처의 개요

### 1. 정의
- 애플리케이션 실행에 필요한 모든 것(코드, 런타임, 라이브러리, 설정)을 하나로 묶어 컨테이너화하고, 이를 관리하기 위한 클라이언트-서버 구조의 소프트웨어 엔진이다.

### 2. 핵심 구성 요소
- **Docker Client**: 사용자가 명령(`docker build`, `run` 등)을 내리는 인터페이스.
- **Docker Host (Daemon)**: 실제 컨테이너를 관리하는 백그라운드 프로세스(`dockerd`). 클라이언트의 요청을 받아 이미지 빌드 및 컨테이너 실행을 수행한다.
- **Docker Registry**: 컨테이너 이미지를 저장하고 공유하는 저장소 (예: Docker Hub).

### 3. 💡 비유: '표준 규격의 화물 배송 시스템'
- 도커는 **'전 세계 어디서든 똑같이 사용할 수 있는 표준 컨테이너 박스 시스템'**과 같다.
- **Client**는 운송장을 끊는 '사무실', **Daemon**은 실제로 짐을 싣고 내리는 '부두의 기계', **Registry**는 박스들이 모여 있는 '거대 물류 창고'에 해당한다.

+++

## Ⅱ. 도커 내부 동작 메커니즘 (Deep Dive)

### 1. 런타임 구조 (containerd & runc)
- 초기의 도커는 하나의 거대한 덩어리였으나, 현재는 표준화를 위해 분리되었다.
- **containerd**: 컨테이너의 생명 주기(네트워크, 스토리지 관리 등)를 관리하는 상위 런타임.
- **runc**: 리눅스 커널과 직접 소통하여 네임스페이스와 Cgroups를 설정하고 컨테이너 프로세스를 실행하는 실제 실행기(Low-level Runtime).

### 2. 이미지 레이어와 스토리지 드라이버
- 도커 이미지는 여러 개의 **읽기 전용 (Read-only) 레이어**로 쌓여 있다.
- 컨테이너가 실행되면 그 위에 얇은 **쓰기 가능 (Writable) 레이어**가 추가된다. 이를 통해 동일한 이미지로 수백 개의 컨테이너를 띄워도 원본 이미지는 수정되지 않고 공유된다. (Union File System 활용)

### 3. ASCII 구조 다이어그램
```text
    [ Docker Client ]        [ Docker Host (Server) ]       [ Registry ]
    ┌──────────────┐         ┌──────────────────────┐      ┌────────────┐
    │ docker build │ ───┐     │  Docker Daemon       │      │            │
    │ docker pull  │ ───┼────▶│ (dockerd)            │◀────▶│ Docker Hub │
    │ docker run   │ ───┘     └──────────┬───────────┘      │            │
    └──────────────┘                     │                  └────────────┘
                                         ▼
                             ┌──────────────────────┐
                             │ containerd / runc    │
                             └──────────┬───────────┘
                                         ▼
                          ┌─────────────┬─────────────┐
                          │ Container A │ Container B │
                          └─────────────┴─────────────┘
```

- **📢 섹션 요약 비유**: 마스터키(데몬)를 가진 관리인이 입주자(클라이언트)의 요청에 따라 창고(레지스트리)에서 방 설계를 가져와 방(컨테이너)을 꾸며주는 호텔 관리 시스템과 같습니다.

## Ⅲ. 도커의 핵심 객체 (Objects)

### 1. Docker Images
- 컨테이너 실행에 필요한 모든 파일과 설정을 포함하는 **불변(Immutable)의 템플릿**이다. 소스 코드로 치면 '클래스'에 해당한다.

### 2. Docker Containers
- 이미지를 실행한 상태이다. 소스 코드로 치면 '인스턴스'에 해당한다. 격리된 환경에서 실제 프로세스가 돌아가는 단위다.

### 3. Docker Volumes
- 컨테이너는 삭제되면 내부 데이터도 사라진다(Ephemeral). 데이터를 영구적으로 보존하기 위해 호스트 OS의 특정 디렉토리를 컨테이너에 매핑하는 장치다.

- **📢 섹션 요약 비유**: 이미지는 붕어빵 틀이고, 컨테이너는 그 틀에서 구워져 나온 실제 붕어빵이며, 볼륨은 붕어빵 속의 팥(데이터)을 따로 저장해두는 통과 같습니다.

+++

## Ⅳ. 실무 적용 및 OS 인사이트 (Strategy)

### 1. 실무 시나리오: 마이크로서비스 배포
- **상황**: 하나의 서버에서 서로 다른 버전의 Python(2.7, 3.8, 3.11)을 사용하는 앱 3개를 돌려야 함.
- **판단**: 호스트 OS에 Python을 설치하지 않고, 각 앱을 해당 버전의 Python이 담긴 도커 컨테이너로 패키징한다. 도커 데몬은 네임스페이스를 통해 각 앱의 라이브러리 경로를 완벽히 분리하므로 충돌 없이 안정적으로 서비스된다.

### 2. 기술사적 인사이트: OCI (Open Container Initiative) 표준
- 도커가 시장을 독점하자 업계는 특정 회사에 종속되지 않는 표준을 요구했다. 이에 따라 이미지 규격과 런타임 규격인 **OCI 표준**이 제정되었다. 현대 OS 기술사라면 도커뿐만 아니라 Podman, CRI-O 등 표준을 준수하는 다양한 컨테이너 엔진의 특징을 비교 분석할 수 있어야 한다.

- **📢 섹션 요약 비유**: 특정 회사의 충전기만 쓰던 시대에서 USB-C 타입(표준)으로 통일되어, 어떤 기기든 자유롭게 연결할 수 있게 된 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **배포 속도 향상**: 기존 VM 방식 대비 10배 이상의 빠른 서비스 런칭.
- **인프라 비용 절감**: 자원 점유율이 낮아 하드웨어 집적도 향상.

### 2. 미래 전망
도커는 단순한 컨테이너 엔진을 넘어, 개발 환경 자체를 컨테이너화하는 **Dev Containers**나 AI 모델의 복잡한 의존성을 해결하는 핵심 도구로 자리 잡았다. 앞으로는 보안이 더욱 강화된 루트리스(Rootless) 모드와 WebAssembly와의 결합을 통해 더욱 안전하고 범용적인 플랫폼으로 진화할 것이다.

- **📢 섹션 요약 비유**: 현대 소프트웨어 공학의 '컨테이너화된 공장'을 완성하여, 전 세계 어디서나 똑같은 품질의 제품을 생산하게 돕는 일등 공신입니다.
