# 자원 할당 (Resource Allocation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 자원 할당은 한정된 시스템 자원(CPU, 메모리, I/O)을 다수의 프로세스에게 공정하고 효율적으로 배분하는 운영체제의 핵심 기능이다.
> 2. **가치**: 최적의 자원 할당은 **시스템 처리량(Throughput) 극대화**, **대기 시간 최소화**, 그리고 **교착상태(Deadlock) 방지**를 동시에 달성한다.
> 3. **융합**: 클라우드 컴퓨팅의 오토스케일링, 컨테이너 오케스트레이션(Kubernetes), 그리고 실시간 시스템의 우선순위 스케줄링 모두 자원 할당 이론의 확장이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 정의
- 자원 할당(Resource Allocation)은 운영체제가 프로세스의 요청에 따라 CPU 시간, 메모리 공간, I/O 장치, 파일 등의 시스템 자원을 배분하고 회수하는 일련의 과정이다.

### 2. 💡 비유: 도서관 좌석 배정 시스템
- 자원 할당은 **"도서관 좌석 배정"**과 같다.
- 한정된 좌석(자원)을 이용하려는 학생들(프로세스)에게 공정하게 배정하고, 사용이 끝나면 반환받아 다음 학생에게 배정한다. 특정 학생이 좌석을 독점하거나 서로가 서로의 좌석을 기다리며 대치하는 상황(교착상태)도 방지해야 한다.

### 3. 등장 배경
- **다중 프로그래밍**: 여러 프로세스가 동시에 자원을 요청하면서 자원 경쟁 발생
- **자원 희소성**: CPU, 메모리 등 핵심 자원은 항상 요구보다 부족
- **공정성 요구**: 모든 프로세스가 기아(Starvation) 상태에 빠지지 않아야 함

### 4. 자원의 분류

| 분류 기준 | 유형 | 특성 | 예시 |
|:---|:---|:---|:---|
| 선점 가능성 | 선점형 (Preemptible) | 실행 중인 프로세스에서 회수 가능 | CPU, 메모리 |
| 선점 가능성 | 비선점형 (Non-preemptible) | 할당 후 자발적 반환만 가능 | 프린터, 테이프 드라이브 |
| 재사용성 | 가용 자원 (Reusable) | 사용 후 회수하여 재사용 가능 | CPU, 메모리, 디스크 |
| 재사용성 | 소모 자원 (Consumable) | 사용 후 사라짐 | 시그널, 메시지, 인터럽트 |

- **📢 섹션 요약 비유**: 레스토랑에서 제한된 테이블을 손님들에게 배정하는 호스트(host)처럼, 운영체제는 시스템 자원을 공정하고 효율적으로 배분합니다.

### 2. 구성 요소 상세

| 구성 요소 | Full Name | 역할 | 핵심 알고리즘 |
|:---|:---|:---|:---|
| **Admission Control** | 어드미션 컨트롤 (승인 제어) | 새 프로세스 수용 여부 결정 | 자원 가용성 체크 |
| **CPU Scheduler** | CPU 스케줄러 | 프로세스에 CPU 시간 배분 | FCFS, RR, Priority, MLFQ |
| **Memory Allocator** | 메모리 할당자 | 가상/물리 메모리 공간 배분 | Paging, Buddy, Slab |
| **I/O Scheduler** | I/O 스케줄러 | 디스크/장치 접근 순서 결정 | CFQ, Deadline, NOOP |
| **Deadlock Handler** | 교착상태 처리기 | 교착상태 탐지 및 복구 | Banker's Algorithm |

### 3. 자원 할당 상태 다이어그램

```text
    ┌───────────────────────────────────────────────────────────────────────┐
    │                    Resource Allocation State Graph                     │
    │                                                                       │
    │   ┌─────────────┐                              ┌─────────────┐        │
    │   │  Process    │                              │  Process    │        │
    │   │    (P1)     │                              │    (P2)     │        │
    │   └──────┬──────┘                              └──────┬──────┘        │
    │          │                                            │               │
    │          │ Request                                    │ Request       │
    │          ▼                                            ▼               │
    │   ┌──────────────────────────────────────────────────────────┐        │
    │   │                    WAITING QUEUE                          │        │
    │   │    [P1 → R1]  [P2 → R2]  [P3 → R1]  [P4 → R3]            │        │
    │   │         요청 대기열 (자원별 큐 또는 통합 큐)               │        │
    │   └────────────────────────────┬─────────────────────────────┘        │
    │                                │                                       │
    │          ┌─────────────────────┼─────────────────────┐                │
    │          │                     │                     │                │
    │          ▼                     ▼                     ▼                │
    │   ┌────────────┐        ┌────────────┐        ┌────────────┐          │
    │   │ Resource   │        │ Resource   │        │ Resource   │          │
    │   │   (R1)     │        │   (R2)     │        │   (R3)     │          │
    │   │  CPU       │        │  Memory    │        │  Printer   │          │
    │   │ [4 cores]  │        │ [16 GB]    │        │ [1 unit]   │          │
    │   └────────────┘        └────────────┘        └────────────┘          │
    │                                                                       │
    │   [범례] P: Process, R: Resource                                      │
    │   P → R: 자원 요청 (Request Edge)                                     │
    │   R → P: 자원 할당 (Assignment Edge)                                  │
    └───────────────────────────────────────────────────────────────────────┘
```

**[자원 할당 그래프 해설]**
1. **프로세스 노드(P)**: 자원을 요청하는 주체로, 원형으로 표현한다.
2. **자원 노드(R)**: 할당 가능한 자원으로, 사각형 내 인스턴스 점으로 표현한다.
3. **요청 간선(Request Edge)**: P → R 방향, 프로세스가 자원을 요청 중임을 의미한다.
4. **할당 간선(Assignment Edge)**: R → P 방향, 자원이 프로세스에 할당됨을 의미한다.
5. **교착상태 판별**: 그래프에 사이클이 존재하고, 각 자원이 단일 인스턴스면 교착상태다.

### 4. 은행가 알고리즘 (Banker's Algorithm)

```text
    ┌─────────────────────────────────────────────────────────────────────┐
    │              Banker's Algorithm (Dijkstra, 1965)                    │
    │                                                                     │
    │  [입력 데이터 구조]                                                 │
    │  ┌─────────────────────────────────────────────────────────────┐   │
    │  │ Available  = [A, B, C]  // 현재 가용 자원 벡터              │   │
    │  │ Max[i]     = [A, B, C]  // 프로세스 i의 최대 요구량         │   │
    │  │ Allocation[i] = [A,B,C] // 프로세스 i에 이미 할당된 양      │   │
    │  │ Need[i] = Max[i] - Allocation[i] // 프로세스 i의 추가 요구  │   │
    │  └─────────────────────────────────────────────────────────────┘   │
    │                                                                     │
    │  [안전성 검사 알고리즘]                                             │
    │  Work = Available                                                  │
    │  Finish[i] = false for all i                                       │
    │                                                                     │
    │  WHILE exists i such that:                                         │
    │        (Finish[i] == false) AND (Need[i] <= Work)                  │
    │  DO                                                                 │
    │        Work = Work + Allocation[i]                                 │
    │        Finish[i] = true                                            │
    │  END WHILE                                                          │
    │                                                                     │
    │  IF Finish[i] == true for all i → SAFE STATE                       │
    │  ELSE → UNSAFE STATE (교착상태 가능성)                             │
    └─────────────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 은행이 모든 고객이 동시에 예금을 인출해도 지급할 수 있는지 미리 시뮬레이션하듯, 은행가 알고리즘은 모든 프로세스가 자원을 요청해도 안전하게 처리 가능한지 미리 검증합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 자원 할당 전략 비교

| 전략 | Full Name | 접근 방식 | 장점 | 단점 | 적용 분야 |
|:---|:---|:---|:---|:---|:---|
| **Static** | 정적 할당 | 실행 전 전체 자원 확정 | 단순, 교착상태 없음 | 자원 낭비 | 임베디드, RTOS |
| **Dynamic** | 동적 할당 | 실행 중 요청 시 배분 | 효율적 자원 활용 | 교착상태 위험 | 범용 OS |
| **Preemptive** | 선점형 | 강제 회수 가능 | 응답성 우수 | 오버헤드 | 시분할 시스템 |
| **Non-preemptive** | 비선점형 | 자발적 반환만 | 구현 단순 | 대기 시간 길 수 있음 | 일괄 처리 |

### 2. 교착상태 처리 전략 비교

| 전략 | 설명 | 장점 | 단점 |
|:---|:---|:---|:---|
| **예방 (Prevention)** | 필요조건 부정 | 교착상태 발생 불가 | 자원 활용도 저하 |
| **회피 (Avoidance)** | 안전 상태 유지 | 유연성 확보 | 실행 시간 오버헤드 |
| **탐지 (Detection)** | 주기적 검사 후 복구 | 자유로운 할당 | 복구 비용 |
| **무시 (Ignorance)** | 아무것도 안 함 | 성능 저하 없음 | 장애 시 재부팅 필요 |

### 3. 과목 융합 관점

| 융합 과목 | 연관 개념 | 융합 포인트 |
|:---|:---|:---|
| **자료구조** | 그래프, 큐 | 자원 할당 그래프, 대기 큐 |
| **알고리즘** | 그래프 탐색, 정렬 | 교착상태 탐지, 스케줄링 |
| **데이터베이스** | 트랜잭션, 락 | 2PL, 데드락 탐지 |
| **네트워크** | 대역폭 할당 | QoS, 흐름 제어 |
| **컴퓨터 구조** | 파이프라인 | 구조적 위험(Hazard) 해결 |

- **📢 섹션 요약 비유**: 교통경찰이 교차로에서 차량들을 조율하여 교통체증(교착상태)을 막듯, 자원 할당 알고리즘은 시스템 내 자원 흐름을 원활하게 관리합니다.


#### 시나리오 B: Kubernetes 자원 할당
```text
    [Pod Resource Specification]

    resources:
      requests:        # 최소 보장 (스케줄링 기준)
        cpu: "500m"    # 0.5 CPU core
        memory: "512Mi"
      limits:          # 최대 제한 (OOM Kill 기준)
        cpu: "1000m"   # 1 CPU core
        memory: "1Gi"

    [QoS Class]
    - Guaranteed: requests == limits (최고 우선순위)
    - Burstable: requests < limits (중간)
    - BestEffort: requests/limits 없음 (최하위, 첫 번째로 퇴거)
```

### 2. 도입 체크리스트

| 항목 | 체크 포인트 | 권장 사항 |
|:---|:---|:---|
| 자원 파악 | 시스템 총 자원량 확인 | CPU, Memory, I/O, Network 대역폭 |
| 워크로드 분석 | 피크/평균 요구량 측정 | 모니터링 도구로 2주 이상 데이터 수집 |
| 우선순위 | 업무 중요도 분류 | Gold/Silver/Bronze 등급화 |
| 한계 설정 | Limits/Quotas 구성 | OOM 방지, 공정한 분배 |
| 모니터링 | 자원 사용률 추적 | Prometheus + Grafana |

### 3. 안티패턴

| 안티패턴 | 문제점 | 올바른 접근 |
|:---|:---|:---|
| **Over-provisioning** | 모든 서비스에 최대 자원 할당 | 요청/제한 분리, 오토스케일링 |
| **자원 경합 무시** | 우선순위 없이 공유 | cgroups, QoS 클래스 적용 |
| **교착상태 방치** | 탐지 메커니즘 없음 | 타임아웃, 순서 강제, 탐지 주기 설정 |

- **📢 섹션 요약 비유**: 호텔이 객실을 프리미엄/스탠다드로 구분하고 예약 시스템으로 초과 예약을 방지하듯, 자원 할당도 등급화와 한계 설정이 필수적입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 자원 할당은 **'장난감 나눠주기'**와 같아요.
2. 하나의 장난감(자원)을 여러 친구(프로세스)가 함께 쓰려면 누가 언제 얼마나 쓸지 정해주는 규칙이 필요하죠.
3. 이 규칙 덕분에 모든 친구가 공평하게 장난감을 가지고 놀 수 있답니다!
