# 루트 파일 시스템 (Root Filesystem) / 오버레이 파일 시스템 (OverlayFS)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 오버레이 파일 시스템 (OverlayFS)은 여러 개의 디렉토리를 하나의 논리적 파일 시스템으로 통합하여 보여주는 **유니온 파일 시스템 (Union File System)**의 일종으로, 컨테이너 이미지의 레이어링 구조를 구현하는 핵심 기술이다.
> 2. **가치**: 읽기 전용(Lower) 레이어 위에 쓰기 가능(Upper) 레이어를 얹어 변경 사항만 효율적으로 저장하는 **복사 후 쓰기 (COW, Copy-on-Write)** 메커니즘을 통해 스토리지 사용량을 획기적으로 줄인다.
> 3. **융합**: 운영체제는 이를 통해 컨테이너마다 독립된 루트 파일 시스템(RootFS)을 제공하면서도, 공통된 베이스 이미지는 메모리와 디스크에서 공유하여 시스템 효율성을 극대화한다.

+++

## Ⅰ. RootFS와 OverlayFS의 개요

### 1. 루트 파일 시스템 (RootFS)
- 운영체제가 부팅될 때 커널이 가장 먼저 마운트하는 최상위 디렉토리(`/`) 환경이다. 컨테이너 환경에서는 해당 컨테이너가 실행되는 데 필요한 최소한의 바이너리, 라이브러리, 설정 파일들의 집합을 의미한다.

### 2. 오버레이 파일 시스템 (OverlayFS)
- 여러 개의 디렉토리를 겹쳐서(Overlay) 사용자에게는 하나의 통합된 디렉토리로 보이게 하는 기술이다. 리눅스 커널 3.18부터 정식 통합되어 도커의 기본 스토리지 드라이버로 사용된다.

### 3. 💡 비유: '투명한 유리판 위에 겹쳐 그린 그림'
- **Lower 레이어**: 배경이 그려진 유리판. 수정할 수 없다.
- **Upper 레이어**: 배경 위에 겹쳐 놓은 빈 유리판. 내가 새로 그리고 싶은 부분만 이 판에 그린다.
- **Merged 뷰**: 위에서 내려다봤을 때 배경과 내가 그린 그림이 하나로 합쳐져 보이는 최종 결과물이다.

+++

## Ⅱ. OverlayFS의 구조와 동작 (Deep Dive)

OverlayFS는 크게 4가지 논리적 계층으로 구성된다.

### 1. 4단계 계층 구조
- **LowerDir (Read-Only)**: 컨테이너 이미지 레이어들. 수정이 불가능하며 여러 컨테이너가 공유한다.
- **UpperDir (Read-Write)**: 컨테이너에서 발생하는 모든 변경 사항(생성, 수정, 삭제)이 기록되는 계층이다.
- **MergedDir (Unified View)**: 사용자나 프로세스가 실제로 보고 사용하는 통합 마운트 지점이다.
- **WorkDir (Intermediate)**: 원자적(Atomic) 연산을 위해 시스템이 내부적으로 사용하는 작업 공간이다.

### 2. 주요 동작 메커니즘
- **읽기 (Read)**: 파일이 Upper에 있으면 Upper에서 읽고, 없으면 Lower에서 찾아 읽는다.
- **수정 (Copy-on-Write)**: Lower에 있는 파일을 수정하려 하면, 커널이 즉시 해당 파일을 Upper로 복사한 뒤 수정을 진행한다. 원본은 보존된다.
- **삭제 (Whiteout)**: Lower의 파일을 삭제하면 실제로 지우는 대신 Upper에 **화이트아웃 (Whiteout)** 이라는 특수 파일을 생성하여 사용자에게 보이지 않게 가린다.

### 3. ASCII 구조 다이어그램
```text
    [ Merged View ]  <── 사용자가 보는 통합 화면 (/)
    ┌───────────────────────────┐
    │  File A (New)   File B'   │
    └─────────────┬─────────────┘
                  ▼
    ┌─────────────┴─────────────┐
    │  Upper Layer (R/W)        │  <── 변경된 내용 (File B', File A)
    ├───────────────────────────┤
    │  Lower Layer 2 (R/O)      │  <── 이미지 레이어 2 (File B)
    ├───────────────────────────┤
    │  Lower Layer 1 (R/O)      │  <── 베이스 이미지 (libc, bash)
    └───────────────────────────┘
```

- **📢 섹션 요약 비유**: 원본 문서(Lower) 위에 투명 필름(Upper)을 대고 글자를 고치거나 덧칠(COW)하여, 최종적으로는 고쳐진 문서(Merged)처럼 보이게 만드는 스마트 편집 기법과 같습니다.

## Ⅲ. 성능 및 효율성 분석

### 1. 장점
- **빠른 컨테이너 생성**: 데이터를 복사하지 않고 마운트만 수행하므로 생성 속도가 매우 빠름.
- **디스크 절약**: 100개의 컨테이너가 동일한 베이스 이미지를 써도 디스크에는 1개 분량만 존재.
- **이미지 배포 효율**: 변경된 레이어만 전송(Layer Caching)하면 되므로 네트워크 대역폭 절약.

### 2. 단점 및 고려사항
- **I/O 성능 저하**: COW 발생 시 파일 복사 오버헤드 존재. (데이터베이스 등 쓰기가 잦은 앱은 Volume 권장)
- **복잡한 디버깅**: 여러 레이어가 겹쳐 있어 실제 파일이 어느 레이어에 있는지 확인하기 번거로움.

- **📢 섹션 요약 비유**: 메모지를 낭비하지 않고 필요한 부분만 덧붙여 쓰는 효율성은 좋지만, 너무 많이 겹치면 원래 내용을 찾기 힘들어지는 것과 같습니다.

+++

## Ⅳ. 실무 적용 및 OS 전략 (Strategy)

### 1. 실무 시나리오: 골든 이미지 (Golden Image) 관리
- **상황**: 수천 대의 서버에 보안 패치를 적용해야 함.
- **판단**: 각 서버를 일일이 업데이트하는 대신, 베어메탈 하이퍼바이저나 컨테이너의 **Lower 레이어**만 보안 패치된 최신 버전으로 교체한다. OverlayFS 덕분에 상위 앱 설정은 유지되면서 하부의 취약점만 순식간에 해결된다.

### 2. 기술사적 인사이트: 스토리지 드라이버 선택
- 과거에는 `aufs`, `devicemapper` 등 다양한 드라이버가 있었으나, 성능과 안정성 면에서 `overlay2`가 표준이 되었다. OS 기술사라면 대규모 I/O가 발생하는 환경(예: 빅데이터 분석 노드)에서는 OverlayFS의 오버헤드를 피하기 위해 호스트 경로를 직접 매핑하는 **바인드 마운트 (Bind Mount)**를 적절히 혼용 설계해야 한다.

- **📢 섹션 요약 비유**: 그림을 겹쳐 그리는 방식(OverlayFS)은 보기엔 좋지만, 매번 덧칠하는 속도가 중요하다면 직접 종이(Bind Mount)에 그리는 게 가장 빠른 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **이미지 이식성 극대화**: 계층화된 구조 덕분에 작고 가벼운 배포 단위 형성.
- **시스템 안정성**: Lower 레이어의 불변성(Immutability) 보장으로 시스템 환경 오염 방지.

### 2. 미래 전망
최근에는 네트워크 지연 없이 원격지의 레이어를 즉시 마운트하여 사용하는 **지연 로딩 (Lazy Loading)** 기술이 발전하고 있다. 또한, OverlayFS의 보안 취약점을 막기 위해 커널 레벨에서 더욱 강력한 레이어 격리 및 서명 검증 기술이 도입되고 있다.

- **📢 섹션 요약 비유**: 필요한 조각들을 착착 쌓아 올리는 레고 블록처럼, 현대 운영체제의 소프트웨어 배포 방식을 완전히 바꾼 마법의 적층 기술입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **도커 (Docker)**: OverlayFS를 가장 활발히 사용하는 플랫폼.
- **Copy-on-Write (COW)**: OverlayFS의 핵심 데이터 저장 기법.
- **VFS (Virtual File System)**: OverlayFS가 상주하는 커널의 파일 추상화 계층.

### 👶 어린이를 위한 3줄 비유 설명
1. 오버레이 파일 시스템은 **'여러 장의 투명 스티커를 겹쳐서 하나의 그림을 만드는 것'**과 같아요.
2. 밑에 있는 스티커는 고칠 수 없지만, 그 위에 새 스티커를 붙여서 글자를 바꾸거나 그림을 새로 그릴 수 있죠.
3. 이렇게 하면 원래 스티커를 망가뜨리지 않고도 나만의 새로운 스티커 북을 아주 쉽게 만들 수 있답니다!
