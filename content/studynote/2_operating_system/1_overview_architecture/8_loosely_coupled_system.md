## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 약결합 시스템(Loosely Coupled System)은 독자적인 CPU와 메모리, 운영체제를 가진 여러 대의 독립된 컴퓨터(노드)가 네트워크를 통해 연결되어 하나의 시스템처럼 동작하는 분산 시스템 아키텍처다.
> 2. **가치**: 공유 메모리로 인한 병목이 없어 이론상 무한대에 가까운 확장성(Scale-out)을 제공하며, 특정 노드가 고장 나도 전체 시스템이 생존하는 극강의 결함 허용성(Fault Tolerance)을 자랑한다.
> 3. **융합**: 과거의 단순 클러스터를 넘어, 현재는 클라우드 컴퓨팅, 하둡(Hadoop), 마이크로서비스 아키텍처(MSA) 등 현대 대규모 데이터 분산 처리의 물리적·논리적 근간이 되는 아키텍처다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 약결합 시스템은 하드웨어 자원을 공유하지 않는 독립적인 시스템들의 집합이다. 서로 메모리를 공유하지 않기 때문에 분산 시스템(Distributed System) 또는 클러스터(Cluster)라고도 불린다. 이들은 고속 버스가 아닌 LAN, WAN, 인터넷과 같은 통신망(Network)을 통해 연결된다.
- **💡 비유**: 약결합 시스템은 **전 세계에 흩어져 있는 재택근무자 팀**과 같다. 각자 자신의 집(노드), 자신의 노트북(독립 메모리)으로 일하며, 결과물이나 도움을 요청할 때는 슬랙이나 이메일(네트워크 메시지)을 보낸다. 누군가 정전으로 일을 못 해도(노드 장애), 팀 전체의 프로젝트는 이메일로 인수인계되어 계속 진행된다.
- **등장 배경**: 강결합 시스템(SMP)은 성능은 뛰어나지만 한 대의 기계 안에 CPU를 무한정 집어넣을 수 없다는 물리적 한계와 극심한 비용 문제에 부딪혔다. 이에 저렴한 상용 컴퓨터(Commodity Hardware) 수천 대를 네트워크로 묶어 슈퍼컴퓨터를 능가하는 성능을 내고자 하는 분산 처리 패러다임이 대두되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 메시지 전달 기반의 통신 아키텍처

약결합 시스템의 노드들은 물리적 메모리를 공유하지 않으므로, 데이터 교환은 전적으로 네트워크 패킷 형태의 **메시지 전달(Message Passing)** 방식을 사용한다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               약결합 시스템 (Loosely Coupled System) 구조          │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │     [ Node 1 ]             [ Node 2 ]             [ Node 3 ]  │
 │  ┌──────────────┐       ┌──────────────┐       ┌──────────────┐ │
 │  │    CPU A     │       │    CPU B     │       │    CPU C     │ │
 │  ├──────────────┤       ├──────────────┤       ├──────────────┤ │
 │  │ Local Memory │       │ Local Memory │       │ Local Memory │ │
 │  │   (독립됨)    │       │   (독립됨)    │       │   (독립됨)    │ │
 │  ├──────────────┤       ├──────────────┤       ├──────────────┤ │
 │  │   Local OS   │       │   Local OS   │       │   Local OS   │ │
 │  └──────┬───────┘       └──────┬───────┘       └──────┬───────┘ │
 │         │ Message              │ Message              │         │
 │         ▼ (Send)               ▼ (Receive)            ▼         │
 │  ═══════════════════════════════════════════════════════════  │
 │                       네트워크 (LAN / WAN)                      │
 │  ═══════════════════════════════════════════════════════════  │
 │                                                               │
 │  통신 방식: Node 1이 소켓/RPC를 통해 Node 2로 데이터를 명시적 패킷 전송 │
 │  특징: 높은 확장성(Scale-out), 고장 격리 우수, 통신 지연시간 큼        │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 그림에서 보듯 각 노드는 완전히 독립된 컴퓨터다. Node 1의 CPU가 Node 2의 메모리에 있는 값을 직접 읽어올 수 있는 방법은 없다. 반드시 Node 1의 OS가 네트워크 스택을 통해 "데이터를 줘"라는 메시지를 보내고, Node 2의 OS가 이를 해석하여 메시지로 응답해야 한다. 이 때문에 강결합에 비해 통신 오버헤드(Overhead)가 크다.

### 분산 운영체제(Distributed OS)의 역할
약결합 시스템에서는 사용자가 여러 대의 컴퓨터를 다루고 있다는 사실을 모르게 하는 **투명성(Transparency)**이 핵심이다.
- **위치 투명성**: 데이터가 어느 서버에 있는지 몰라도 접근할 수 있음 (분산 파일 시스템 HDFS 등).
- **장애 투명성**: 특정 서버가 죽어도 다른 서버가 이어받아 사용자는 장애를 눈치채지 못함.

## Ⅲ. 융합 비교 및 다각도 분석

### 통신 패러다임: 공유 메모리(강결합) vs 메시지 전달(약결합)

| 비교 항목 | 강결합 시스템 (Shared Memory) | 약결합 시스템 (Message Passing) |
|:---|:---|:---|
| **통신 매체** | 시스템 버스 (System Bus) | 네트워크 (Network / 소켓 통신) |
| **속도 및 지연** | 나노초(ns) 단위 지연, 초고속 | 밀리초(ms) 단위 지연, 네트워크 오버헤드 발생 |
| **데이터 동기화** | 커널 락(Lock), 스핀락 등 | 메시지 큐(Message Queue), 분산 락(Zookeeper) |
| **프로그래밍 모델** | 스레드(Thread) 간 메모리 공유 | 프로세스 간 통신(IPC), RPC, REST API |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 빅데이터 분산 처리 환경 (Scale-out)
- **상황**: 페타바이트(PB) 단위의 웹 로그 데이터를 분석해야 한다. 한 대의 초고성능 메인프레임(강결합)으로는 메모리와 디스크 용량 한계로 처리가 불가능하다.
- **판단 및 설계**: 
  1. 1,000대의 일반 x86 서버(노드)를 네트워크로 묶은 약결합 시스템 클러스터를 구축한다.
  2. 하둡(Hadoop) 또는 스파크(Spark) 분산 프레임워크를 도입한다.
  3. 거대한 데이터를 1,000개로 쪼개 각 노드의 로컬 디스크에 분산 저장하고, 각 노드의 CPU가 자신의 로컬 데이터만 독립적으로 연산(MapReduce)하게 한다. 
  4. 통신 오버헤드를 줄이기 위해 노드 간 데이터 교환은 최종 결과 병합 단계에서만 수행한다.
- **결과**: 노드가 고장 나도 다른 노드가 데이터를 복제해 두었기 때문에 무중단 처리가 가능하며, 성능은 1,000배에 가깝게 향상된다.

### 안티패턴
- 네트워크 지연(Network Latency)에 극도로 민감한 작업(예: 초고빈도 주식 자동 매매, 실시간 캐시 동기화)을 약결합 분산 시스템 위에서 설계하는 것. 이 경우 네트워크를 타는 수 밀리초의 지연 때문에 시스템이 요건을 충족하지 못하게 되므로, 단일 강결합 서버 내에서 처리하도록 설계해야 한다.

## Ⅴ. 기대효과 및 결론

- 약결합 시스템은 인터넷과 네트워크 인프라의 발전과 함께 현대 IT 아키텍처의 패권을 쥐었다. 무한한 확장성(Scale-out) 덕분에 구글, 아마존, 넷플릭스 등 글로벌 IT 기업의 서비스는 모두 수만 대의 약결합 시스템 위에서 동작한다.
- 과거 하드웨어적 관점의 약결합 시스템은, 현재 클라우드 컴퓨팅과 쿠버네티스(Kubernetes) 기반의 컨테이너 오케스트레이션이라는 소프트웨어적 약결합 생태계로 진화하여 인프라의 표준이 되었다.
