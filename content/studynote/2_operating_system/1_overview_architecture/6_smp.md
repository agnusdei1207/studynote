# 대칭형 다중 처리 (Symmetric Multiprocessing, SMP)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 대칭형 다중 처리(SMP)는 두 개 이상의 대등한 프로세서가 하나의 공통된 주메모리를 공유하며, 단일 운영체제 하에서 독립적으로 작업을 수행하는 **피어 투 피어(Peer-to-Peer)** 구조의 다중 처리 방식이다.
> 2. **가치**: 모든 프로세서가 커널 기능을 수행할 수 있어 **부하 분산(Load Balancing)**이 매우 효율적이며, 특정 프로세서 장애 시에도 시스템이 중단되지 않는 높은 가용성과 성능 확장성을 제공한다.
> 3. **융합**: 현대 멀티코어 시스템의 표준 모델로 자리 잡았으나, 프로세서 수 증가에 따른 공유 버스 경합과 **캐시 일관성(Cache Coherency)** 유지 비용이 시스템 성능의 임계점으로 작용한다.

+++

## Ⅰ. 대칭형 다중 처리 (SMP)의 개요

### 1. 정의
- 모든 프로세서가 하드웨어적으로 동일한 권한을 가지며, 메모리와 입출력 장치 등 모든 자원을 공유하는 다중 처리 아키텍처다.

### 2. 등장 배경: ASMP의 한계 극복
- 주 프로세서(Master)만 커널을 실행하던 비대칭형(ASMP)의 병목 현상과 낮은 신뢰성을 해결하기 위해, 모든 프로세서가 주인이 되는 SMP가 설계되었다.

### 3. 💡 비유: 민주적인 공동 주방
- ASMP가 주방장 한 명이 지시하고 나머지는 심부름만 하는 구조라면, SMP는 **"모든 요리사가 실력자인 공동 주방"**과 같다.
- 누구나 재료(메모리)를 꺼내 요리를 시작할 수 있고, 주문(작업)이 들어오면 노는 요리사가 즉시 처리한다. 한 명이 쉬어도 주방은 멈추지 않는다.

+++

## Ⅱ. SMP의 아키텍처 및 통신 구조 (ASCII)

강결합(Tightly-coupled) 시스템의 전형적인 구조를 띤다.

```text
    ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
    │  Processor  │  │  Processor  │  │  Processor  │
    │     P1      │  │     P2      │  │     Pn      │
    └──────┬──────┘  └──────┬──────┘  └──────┬──────┘
           │ [Local Cache]  │ [Local Cache]  │ [Local Cache]
    ───────┴────────┬───────┴────────┬───────┴────────
                    │ (System Bus / Interconnect)
           ┌────────┴────────────────┴────────┐
           │        Shared Main Memory        │
           │   (Global Operating System)      │
           └──────────────────────────────────┘
```

**[핵심 동작 원리]**
1. **커널 공유**: 메모리에 단 하나의 OS 커널이 상주하며, 어떤 프로세서든 커널 코드를 실행하여 시스템 호출을 처리할 수 있다.
2. **동기화**: 여러 CPU가 동시에 공유 자원을 수정하지 못하도록 Spinlock, Semaphore 등의 **상호 배제(Mutual Exclusion)** 기술이 필수적이다.
3. **캐시 일관성**: P1이 메모리 값을 바꿔도 P2가 가진 복사본(Cache)이 구식이 되지 않도록 하드웨어적으로 감시(Snooping 등)한다.

+++

## Ⅲ. ASMP vs SMP 상세 비교

| 비교 항목 | 비대칭형 (ASMP) | 대칭형 (SMP) |
|:---|:---|:---|
| **프로세서 관계** | Master-Slave (수직적) | **Peer-to-Peer (수평적)** |
| **자원 효율성** | 보통 (Master 병목) | **매우 높음 (동적 배분)** |
| **가용성** | 낮음 (Master 고장 시 마비) | **높음 (대체 실행 가능)** |
| **운영체제 설계** | 쉬움 (Master만 관리) | **매우 어려움 (동기화 복잡)** |
| **현대적 용도** | 특수 가속기 구조 | **범용 서버, PC, 스마트폰** |

- **📢 섹션 요약 비유**: 팀장 한 명의 컨디션에 팀 전체 실적이 갈리는 조직(ASMP)과, 모든 팀원이 팀장급 역량을 갖춰 서로를 백업하는 '어벤져스 팀'(SMP)의 차이입니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. 확장성의 한계 (Amdahl's Law)
- **기술사적 인사이트**: CPU를 2배로 늘린다고 성능이 2배가 되지는 않는다. 프로세서 수가 많아질수록 공유 버스를 차지하려는 싸움과 캐시 일관성 유지를 위한 통신량(Traffic)이 기하급수적으로 늘어나기 때문이다.
- **결단**: 수십 개 이상의 코어를 사용하는 대규모 시스템에서는 SMP의 한계를 극복하기 위해 메모리를 분산 배치하는 **NUMA (Non-Uniform Memory Access)** 아키텍처를 혼합 설계해야 한다.

### 2. 가상화 환경에서의 SMP (vSMP)
- **판단**: 클라우드 가상 머신에 8개의 vCPU를 할당했다는 것은 하이퍼바이저가 해당 VM에게 SMP 환경을 제공하고 있음을 의미한다. 
- **인사이트**: 이때 실제 물리 코어(pCPU)와의 매핑 최적화가 이루어지지 않으면, CPU 간 스케줄링 간섭으로 인한 성능 급락(**L-스케줄링 문제**)이 발생할 수 있으므로 핀닝(Pinning) 전략이 중요하다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **성능 극대화**: 병렬 처리를 통한 처리량(Throughput) 및 응답 속도 동시 향상.
- **범용성**: 응용 프로그램 입장에서는 CPU 개수를 몰라도 OS가 알아서 배분해주므로 개발 생산성 증대.

### 2. 미래 전망
단일 다이 내에 수백 개의 코어가 들어가는 **Many-core** 시대로 접어들면서, 전통적인 '모든 자원 공유' 방식의 SMP는 통신 병목에 직면하고 있다. 앞으로의 SMP는 칩 내부 네트워크(NoC, Network-on-Chip) 기술과 결합하여, '논리적으로는 SMP이되 물리적으로는 분산 처리'되는 하이브리드 형태로 진화하여 컴퓨팅 성능의 한계를 돌파할 것이다.

- **📢 섹션 요약 비유**: 모두가 주인인 세상은 아름답지만, 서로의 의견(데이터)을 맞추는 비용도 커집니다. 그 조율의 기술이 바로 운영체제의 실력입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **캐시 일관성 (Cache Coherency)**: SMP 하드웨어의 최대 과제.
- **NUMA (Non-Uniform Memory Access)**: SMP의 확장성 한계를 넘는 대안.
- **상호 배제 (Mutual Exclusion)**: SMP 소프트웨어의 필수 안전 장치.

### 👶 어린이를 위한 3줄 비유 설명
1. SMP는 여러 명의 요리사가 **'커다란 식탁 하나'를 같이 쓰는 주방**과 같아요.
2. 누구든 요리 기구와 재료를 똑같이 쓸 수 있어서, 주문이 밀려도 여럿이서 힘을 합쳐 아주 빨리 요리를 끝내죠.
3. 한 명이 감기에 걸려 쉬어도 다른 요리사들이 대신할 수 있어서, 식당은 일 년 내내 문을 닫지 않고 맛있는 음식을 만들 수 있답니다!
