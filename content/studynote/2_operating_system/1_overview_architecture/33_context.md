## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 문맥(Context)은 특정 시점에 프로세스나 스레드가 실행되던 도중 중단되었을 때, 나중에 동일한 상태에서 다시 시작할 수 있도록 저장해야 하는 **모든 기술적 정보의 집합**이다.
> 2. **가치**: CPU 레지스터 상태, 프로그램 카운터(PC), 스택 포인터(SP), 메모리 관리 정보 등을 포함하여 멀티태스킹 환경에서 프로세스 간 전환의 논리적 연속성을 보장한다.
> 3. **융합**: 커널의 프로세스 제어 블록(PCB)에 저장되며, 운영체제의 스케줄링 및 인터럽트 처리 메커니즘과 직결되는 핵심 데이터 구조다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 운영체제에서 문맥(Context)이란 CPU가 해당 프로세스를 실행하기 위해 필요한 모든 정보의 '스냅샷'이다. CPU는 한 번에 하나의 일만 할 수 있으므로, 여러 프로그램을 번갈아 실행하려면 현재 어디까지 했는지 완벽하게 기억해둬야 한다.
- **💡 비유**: 문맥은 **'읽다 만 책의 책갈피와 메모'**와 같습니다. 책을 읽다가 잠시 멈춰야 할 때, 몇 페이지(PC)까지 읽었는지, 중요한 등장인물 이름(레지스터)은 무엇이었는지 책갈피를 끼워두고 메모를 남겨야 나중에 다시 펼쳤을 때 바로 흐름을 이어갈 수 있는 것과 같습니다.
- **등장 배경**: 시분할 시스템(Time-sharing System)이 도입되면서 CPU 자원을 여러 프로세스가 나누어 쓰게 되었습니다. 프로세스가 CPU 점유권을 뺏길 때 자신의 실행 상태를 안전하게 보관하고, 나중에 복구할 수 있는 표준화된 저장 체계가 필요해졌습니다.
- **📢 섹션 요약 비유**: "영화 제작 현장에서 다음 날 촬영을 위해 배우의 분장 상태와 소품 위치를 정확히 기록해두는 '스크립트 기록'과 같습니다."

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 문맥(Context)의 구성 요소 및 계층

문맥은 크게 하드웨어 문맥, 메모리 문맥, 시스템 문맥으로 분류됩니다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               프로세스 문맥 (Process Context) 구성                │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │  1. 하드웨어 문맥 (Hardware Context)                            │
 │     - PC (Program Counter): 다음 실행할 명령어 주소               │
 │     - SP (Stack Pointer): 현재 스택의 최상단 주소                 │
 │     - 범용 레지스터 (EAX, EBX 등): 연산 중간 결과값               │
 │     - 상태 레지스터 (EFLAGS): 연산 결과의 상태 (Zero, Carry 등)   │
 │                                                               │
 │  2. 메모리 문맥 (Memory Context)                                │
 │     - Code, Data, Stack, Heap 세그먼트의 시작과 끝                │
 │     - 페이지 테이블 (Page Table): 가상 메모리 매핑 정보            │
 │                                                               │
 │  3. 커널 문맥 (Kernel Context / System Context)                │
 │     - PCB (Process Control Block): 프로세스 ID, 상태, 우선순위     │
 │     - 열려 있는 파일 기술자 (File Descriptors)                    │
 │     - 커널 스택 (Kernel Stack): 시스템 콜 실행 기록               │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CPU 내부에 실시간으로 변하는 값들은 하드웨어 문맥에 해당하며, 문맥 교환 발생 시 가장 먼저 PCB로 복사되어야 합니다. 메모리 문맥은 주소 공간을 정의하며, 커널 문맥은 운영체제가 해당 프로세스를 관리하기 위해 필요한 행정적 정보를 담고 있습니다.

### 문맥 정보의 상세 내역 (Component Table)

| 구성 요소 | 역할 | 내부 동작 | 비유 |
|:---:|:---|:---|:---|:---|
| **PC** | 다음 명령어 지시 | 현재 실행 중인 명령어 다음 주소를 가리킴 | 내비게이션 목적지 |
| **Registers** | 데이터 고속 저장 | 연산에 필요한 데이터 및 중간 결과 임시 저장 | 작업대의 공구들 |
| **Stack Pointer** | 함수 실행 관리 | 로컬 변수 및 복귀 주소가 담긴 스택의 위치 | 식당의 접시 쌓기 |
| **Page Table** | 주소 변환 | 가상 주소를 물리 주소로 변환하는 맵 정보 | 지도 데이터 |
| **File Table** | 자원 점유 확인 | 프로세스가 사용 중인 I/O 장치 및 파일 목록 | 대여 물품 리스트 |

- **📢 섹션 요약 비유**: "게임을 하다가 '저장(Save)' 버튼을 눌렀을 때 저장되는 캐릭터의 위치, 체력, 아이템 목록이 바로 문맥입니다."

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 프로세스 문맥 vs 스레드 문맥 (Thread Context)

| 비교 항목 | 프로세스 문맥 (Process Context) | 스레드 문맥 (Thread Context) |
|:---|:---|:---|
| **저장 위치** | PCB (Process Control Block) | TCB (Thread Control Block) |
| **공유 범위** | 독립적 (공유 안 함) | **같은 프로세스 내 자원 공유** (Code, Data) |
| **전환 비용** | **매우 높음** (캐시/TLB 플러시 발생) | 낮음 (레지스터와 스택만 교체) |
| **메모리 문맥** | 매번 교체 (Page Table 변경) | **교체 안 함** (동일 주소 공간 사용) |

### 과목 융합: 인터럽트와 문맥
- **인터럽트(Interrupt)**: 외부 하드웨어 신호가 발생하면 CPU는 현재 문맥을 즉시 중단하고 **인터럽트 문맥(Interrupt Context)**으로 전환합니다. 이때 기존 프로세스의 문맥은 안전하게 보존되어야 합니다.
- **📢 섹션 요약 비유**: "전교생의 기록부를 다 바꾸는 것(프로세스 전환)보다, 한 반 안에서 번호표만 바꾸는 것(스레드 전환)이 훨씬 빠른 것과 같습니다."

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 임베디드 실시간성(Real-time) 확보
- **상황**: 0.001초의 오차도 허용하지 않는 엔진 제어 시스템에서 문맥 저장이 너무 오래 걸려 타이밍을 놓침.
- **판단**: 저장해야 할 레지스터 개수가 너무 많거나, 문맥 정보가 비대하여 발생하는 문제입니다.
- **해결**: 레지스터 뱅크(Register Bank) 기능이 있는 CPU를 사용하여 하드웨어적으로 문맥을 한 번에 스위칭하거나, **Lightweight Process** 설계를 통해 저장할 문맥의 양을 최소화하는 전략을 사용합니다.

### 안티패턴
- 문맥에 너무 많은 '상태값'을 정의하여 커널의 오버헤드를 키우는 것. 문맥은 CPU가 다시 시작하는 데 필요한 '최소한의 필수 정보'로 압축되어야 시스템 전체의 처리량(Throughput)이 향상됩니다.
- **📢 섹션 요약 비유**: "여행 가방(문맥)을 너무 무겁게 싸면 이동 속도가 느려지듯, 꼭 필요한 짐만 챙기는 미니멀리즘이 필요합니다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

- **기대효과**: 멀티프로그래밍을 가능하게 하여 CPU 이용률을 극대화하고, 사용자에게는 여러 프로그램이 동시에 실행되는 듯한 인터랙티브한 경험을 제공합니다.
- **미래 전망**: 하드웨어 수준에서 문맥 전환을 가속하는 기능(Hardware-assisted Switching)이 발전하고 있으며, 가상화 환경에서는 가상 머신 전체의 문맥을 저장하는 스냅샷 기술로 확장되고 있습니다.
- **참고 표준**: POSIX 스레드 표준, 시스템 V 프로세스 모델.

- **📢 섹션 요약 비유**: "흐름이 끊기지 않는 매끄러운 세상을 만들기 위해, 찰나의 순간을 영원히 기록하는 운영체제의 기억 장치입니다."

### 📌 관련 개념 맵 (Knowledge Graph)
- **PCB (Process Control Block)**: 문맥 정보가 저장되는 실제 데이터 구조.
- **문맥 교환 (Context Switch)**: 문맥을 바꾸는 작업 그 자체와 오버헤드.
- **스케줄링 (Scheduling)**: 어떤 문맥을 다음에 실행할지 결정하는 알고리즘.
