# 클러스터 시스템 (Clustered System)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클러스터 시스템은 여러 대의 독립적인 컴퓨터(노드)를 네트워크로 연결하여 단일 시스템처럼 동작하게 하는 분산 컴퓨팅 아키텍처이다.
> 2. **가치**: 단일 실패점(SPOF)을 제거하여 **고가용성(HA)**을 확보하고, 워크로드 분산으로 **수평적 확장성(Scalability)**과 **처리량 증대**를 실현한다.
> 3. **융합**: 클러스터는 HPC(고성능 컴퓨팅), 웹 서비스, 데이터베이스, 컨테이너 오케스트레이션(Kubernetes) 등 현대 IT 인프라의 핵심 패턴이다.

+++

## Ⅰ. 개요 (Context & Background)

### 1. 정의
- 클러스터 시스템(Clustered System)은 물리적으로 분리된 여러 서버(노드)가 고속 네트워크로 연결되어, 외부에서는 하나의 시스템처럼 보이고 동작하는 컴퓨팅 환경이다.

### 2. 💡 비유: 오케스트라
- 클러스터는 **"오케스트라"**와 같다.
- 각 악기 연주자(노드)는 독립적으로 연주할 수 있지만, 지휘자(클러스터 매니저)의 통제 아래 하나의 교향곡을 연주한다. 한 명의 연주자가 아파도 공연은 계속된다.

### 3. 등장 배경
- **SPOF 문제**: 단일 서버 고장 시 서비스 중단 위험
- **성능 한계**: 단일 서버의 CPU/메모리 확장 한계
- **비용 효율성**: 고가의 대형 서버보다 범용 서버 다수가 경제적
- **유연성**: 필요에 따라 노드 추가/제거 가능

### 4. 클러스터의 분류

| 분류 기준 | 유형 | 특징 | 적용 분야 |
|:---|:---|:---|:---|
| 목적 | HA (High Availability) | 장애 조치, 가용성 | 웹 서비스, DB |
| 목적 | HPC (High Performance) | 병렬 연산, 성능 | 과학 연산, 렌더링 |
| 목적 | Load Balancing | 부하 분산 | 웹 서버 팜 |
| 구조 | Shared-Nothing | 독립 스토리지 | 웹 서버, HPC |
| 구조 | Shared-Disk | 공유 스토리지 | DB 클러스터, Oracle RAC |

- **📢 섹션 요약 비유**: 클러스터는 여러 명의 레이서가 한 팀으로 달리는 릴레이 경주와 같습니다. 한 명이 넘어져도 다음 주자가 이어받아 결승선을 향해 달립니다.

### 2. 구성 요소 상세

| 구성 요소 | Full Name | 역할 | 기술 예시 |
|:---|:---|:---|:---|
| **Load Balancer** | 로드 밸런서 | 트래픽 분산, 헬스체크 | HAProxy, Nginx, F5 |
| **Cluster Manager** | 클러스터 매니저 | 노드 관리, 장애 조치 | Pacemaker, Kubernetes |
| **Heartbeat** | 하트비트 | 노드 생존 확인 | Corosync, keepalived |
| **Interconnect** | 인터커넥트 | 노드 간 통신 | InfiniBand, 10GbE |
| **Shared Storage** | 공유 스토리지 | 데이터 일관성 | SAN, Ceph, GlusterFS |

### 3. HA 클러스터 장애 조치 메커니즘

```text
    ┌─────────────────────────────────────────────────────────────────────────┐
    │              High Availability Failover Mechanism                       │
    │                                                                         │
    │  [정상 상태]                                                            │
    │  ┌─────────────────────────────────────────────────────────────────┐   │
    │  │  Client ──▶ [VIP: 192.168.1.100] ──▶ Node1 (Active)             │   │
    │  │                                        ▲                         │   │
    │  │                                        │ Heartbeat OK            │   │
    │  │                    Node2 (Standby) ◀──┘                         │   │
    │  └─────────────────────────────────────────────────────────────────┘   │
    │                                                                         │
    │  [장애 발생]                                                            │
    │  ┌─────────────────────────────────────────────────────────────────┐   │
    │  │  Node1 ──▶ X (Heartbeat Timeout!)                               │   │
    │  │             │                                                   │   │
    │  │             ▼                                                   │   │
    │  │  [Cluster Manager]                                              │   │
    │  │  1. Node1 장애 감지 (3회 연속 미응답)                           │   │
    │  │  2. Fencing (STONITH): Node1 강제 차단                          │   │
    │  │  3. VIP 이동: 192.168.1.100 → Node2                             │   │
    │  │  4. 서비스 시작: Node2에서 App 실행                             │   │
    │  │  5. 클라이언트에 알림 (ARP 갱신)                                │   │
    │  └─────────────────────────────────────────────────────────────────┘   │
    │                                                                         │
    │  [복구 후 상태]                                                         │
    │  ┌─────────────────────────────────────────────────────────────────┐   │
    │  │  Client ──▶ [VIP: 192.168.1.100] ──▶ Node2 (Active)             │   │
    │  │                                        ▲                         │   │
    │  │                                        │ Heartbeat OK            │   │
    │  │                    Node1 (Recovery) ◀─┘                         │   │
    │  │                    (Standby로 복귀)                             │   │
    │  └─────────────────────────────────────────────────────────────────┘   │
    └─────────────────────────────────────────────────────────────────────────┘
```

**[장애 조치 해설]**
1. **Heartbeat**: 노드 간 주기적 신호 교환 (1초 간격 등)
2. **장애 감지**: N회 연속 미응답 시 장애로 판정
3. **Fencing (STONITH)**: Split-Brain 방지를 위해 장애 노드 강제 차단
4. **Failover**: VIP와 서비스를 대기 노드로 이동
5. **Failback**: 원래 노드 복구 시 자동/수동 복귀

```text
    ┌─────────────────────────────────────────────────────────────────────────┐

    │  ┌───────────────────────────────────────────────────────────────────┐ │

    │  │  ┌─────────────────────────────────────────────────────────────┐ │ │

    │  │  │  - 작업 분배, 자원 할당                                      │ │ │
    │  │  └─────────────────────────────────────────────────────────────┘ │ │
    │  └───────────────────────────────┬───────────────────────────────────┘ │
    │                                  │                                      │
    │          ┌───────────────────────┼───────────────────────┐             │
    │          │                       │                       │             │
    │          ▼                       ▼                       ▼             │
    │  ┌───────────────┐      ┌───────────────┐      ┌───────────────┐      │
    │  │ Compute Node 1│      │ Compute Node 2│      │ Compute Node N│      │
    │  │ ┌───────────┐ │      │ ┌───────────┐ │      │ ┌───────────┐ │      │
    │  │ │ MPI Rank 0│◄├──────┼─┤ MPI Rank 1│◄├──────┼─┤ MPI Rank N│ │      │
    │  │ └───────────┘ │      │ └───────────┘ │      │ └───────────┘ │      │
    │  │  Local Data   │      │  Local Data   │      │  Local Data   │      │
    │  └───────────────┘      └───────────────┘      └───────────────┘      │
    │                                                                         │
    │  [메시지 전달]                                                          │
    │  MPI_Send() / MPI_Recv() - 노드 간 데이터 교환                         │
    │  MPI_Bcast() - 브로드캐스트                                            │
    │  MPI_Reduce() - 결과 집계                                              │
    └─────────────────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: HA 클러스터는 예비 타이어가 있는 자동차처럼, 한 바퀴가 펑크 나도 예비 타이어로 계속 달릴 수 있습니다. HPC 클러스터는 여러 명이 함께 짐을 나르는 이삿짐 센터처럼, 큰 짐을 분담해 빠르게 옮깁니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 클러스터 유형 비교

| 유형 | Full Name | 목적 | 노드 수 | 데이터 일관성 | 대표 기술 |
|:---|:---|:---|:---|:---|:---|
| **HA** | 고가용성 | 서비스 연속성 | 2-10 | 강한 일관성 | Pacemaker, Keepalived |
| **HPC** | 고성능 컴퓨팅 | 연산 성능 | 10-10000+ | 약한 일관성 | MPI, Slurm |
| **Web Farm** | 웹 서버 팜 | 부하 분산 | 3-100+ | 무상태 | Nginx, HAProxy |
| **DB Cluster** | DB 클러스터 | 데이터 일관성 | 3-9 | 강한 일관성 | Oracle RAC, Galera |
| **Container** | 컨테이너 클러스터 | 오케스트레이션 | 3-1000+ | 가변적 | Kubernetes, Docker Swarm |

### 2. Active-Active vs Active-Passive

| 구분 | Active-Active | Active-Passive |
|:---|:---|:---|
| **구조** | 모든 노드가 서비스 제공 | 일부만 서비스, 나머지 대기 |
| **자원 활용** | 높음 (100%) | 낮음 (50% 미만) |
| **복구 시간** | 없음 (자동 분산) | 있음 (Failover 시간) |
| **복잡도** | 높음 (동기화 필요) | 낮음 |
| **비용 효율** | 좋음 | 낮음 |
| **적용** | 웹 서버, 캐시 | DB, 상태ful 서비스 |

### 3. 과목 융합 관점

| 융합 과목 | 연관 개념 | 융합 포인트 |
|:---|:---|:---|
| **네트워크** | VIP, ARP, 로드 밸런싱 | L4/L7 스위치, DNS RR |
| **데이터베이스** | 복제, 샤딩, Consensus | Galera, Raft, Paxos |
| **분산 시스템** | CAP 이론, 합의 알고리즘 | 일관성 vs 가용성 트레이드오프 |
| **스토리지** | SAN, NAS, 분산 파일 시스템 | 공유 스토리지, Ceph |
| **컨테이너** | 오케스트레이션, 서비스 메시 | Kubernetes, Istio |

- **📢 섹션 요약 비유**: Active-Active는 모든 레지 직원이 동시에 손님을 응대하는 것이고, Active-Passive는 한 명이 일하고 다른 한 명은 대기하다가 필요 시 교대하는 것과 같습니다.

### 3. 안티패턴

| 안티패턴 | 문제점 | 올바른 접근 |
|:---|:---|:---|
| **Fencing 없음** | Split-Brain 발생 | STONITH/ILO 구성 필수 |
| **공유 스토리지 SPOF** | 스토리지 장애 시 전체 중단 | 스토리지 이중화 |
| **Heartbeat 단일 경로** | 네트워크 장애 시 오판 | 이중화된 Heartbeat |
| **과도한 노드 수** | 관리 복잡도 증가 | 적정 규모 유지 |

- **📢 섹션 요약 비유**: 클러스터는 마치 턴팀처럼, 팀원들이 서로를 지탱하며 목표를 향해 함께 나아갑니다.
