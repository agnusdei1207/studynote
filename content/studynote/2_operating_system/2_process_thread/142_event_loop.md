# 이벤트 루프 (Event Loop) 기반 비동기 처리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트 루프(Event Loop)는 **단일 스레드** 환경에서 실행 흐름을 멈추지 않고, 발생한 이벤트(I/O 완료, 타이머 등)를 순차적으로 꺼내어 처리하는 **무한 반복 스케줄링 메커니즘**이다.
> 2. **가치**: 무거운 작업을 외부(커널 또는 스레드 풀)에 맡기고 즉시 다음 코드를 실행하는 **넌블로킹 (Non-blocking)** 방식을 통해, 적은 자원으로도 수만 개의 동시 연결을 효율적으로 처리한다.
> 3. **융합**: Node.js, 웹 브라우저 자바스크립트 엔진의 핵심 구동 원리이며, 복잡한 멀티스레드 동기화 문제 없이 고성능 네트워크 서버를 구축할 수 있는 현대적인 동시성 모델이다.

+++

## Ⅰ. 이벤트 루프의 정의 및 동작 원리

### 1. 정의
- 프로그램의 메인 실행 흐름이 큐(Queue)에 쌓인 이벤트들을 감시하다가, 이벤트가 발생하면 그에 매핑된 콜백 함수를 실행하는 루프 구조이다.

### 2. 핵심 구성 요소
- **Call Stack**: 현재 실행 중인 함수들이 쌓이는 공간.
- **Web APIs / Worker Threads**: 네트워크 요청, 타이머 등 시간이 걸리는 작업을 대신 처리해 주는 외부 환경.
- **Task Queue (Callback Queue)**: 완료된 작업의 콜백 함수들이 실행을 기다리며 줄 서는 곳.
- **Event Loop**: Call Stack이 비어있는지 확인하고, 비어있다면 Task Queue의 첫 번째 작업을 Call Stack으로 옮겨주는 감시자.

### 3. 💡 비유: '놀이공원 매표소 직원'
- 이벤트 루프는 **'매표소 직원 한 명(단일 스레드)'**과 같다.
- 직원은 손님의 요청을 받고, 표가 나오길 기다리는 동안 손님을 옆으로 비켜서게 한 뒤(비동기) 다음 손님을 즉시 받는다. 표가 나오면(이벤트 발생), 벨이 울리고 직원은 틈이 날 때마다 그 손님에게 표를 건네준다(콜백 실행). 직원은 한 명이지만, 손님들은 줄 서서 기다리지 않고 각자 볼일을 보다가 표를 받을 수 있다.

+++

## II. 동작 메커니즘 (Deep Dive)

### 1. 비동기 작업의 흐름
1. 함수가 호출되면 **Call Stack**에 쌓인다.
2. 비동기 작업(예: `setTimeout`, `DB Query`)이라면 즉시 **Web APIs**로 작업을 넘기고 스택에서 빠진다. (넌블로킹)
3. 메인 스레드는 다음 코드를 계속 실행한다.
4. 작업이 완료되면 결과값이 **Task Queue**로 들어간다.
5. **Event Loop**가 스택이 빈 것을 확인하는 순간, 큐의 작업을 스택으로 올려 실행시킨다.

### 2. ASCII 구조 다이어그램
```text
    ┌────────────────┐       ┌────────────────────────┐
    │   Call Stack   │       │  Background / APIs     │
    │  (Executing)   │       │  (Network, File, ...)  │
    └───────▲────────┘       └───────────┬────────────┘
            │                            │
      [ Event Loop ] ◀───────────────────┘ (Done!)
            │
    ┌───────┴────────┐
    │   Task Queue   │ ◀── [ Callback 3 | Callback 2 | Callback 1 ]
    └────────────────┘
```

- **📢 섹션 요약 비유**: 주방장(Call Stack)은 요리만 하고, 재료 배달(I/O)은 배달원(APIs)이 합니다. 배달이 완료된 재료들이 창고(Task Queue)에 쌓이면, 주방장이 한가할 때마다 하나씩 꺼내 요리를 완성하는 분업 체계입니다.

## Ⅲ. 이벤트 루프 기반 모델의 장단점

| 구분 | 장점 (Pros) | 단점 (Cons) |
|:---|:---|:---|
| **자원 효율** | 적은 메모리와 스레드로 많은 연결 처리 | **CPU 집중 작업에 취약** (루프가 멈춤) |
| **복잡도** | 락(Lock), 데드락 걱정 없는 프로그래밍 | 콜백 지옥(Callback Hell) 발생 가능성 |
| **확장성** | I/O 중심의 네트워크 서비스에 최적 | 단일 코어만 사용하므로 멀티코어 활용에 한계 |

- **📢 섹션 요약 비조**: "멈추지 않는 소통(I/O)에는 최고의 선택이지만, 혼자서 복잡한 수학 문제를 풀 때는 오히려 병목이 되는 '경량급 육상 선수'와 같은 모델입니다."

+++

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: CPU Bound 작업 처리
- **상황**: Node.js 서버에서 이미지 압축(CPU 연산)을 수행하자 모든 사용자의 접속이 끊김.
- **판단**: 압축 작업이 **Call Stack**을 점유하여 **Event Loop**가 돌지 못했기 때문이다. 기술사라면 이 문제를 해결하기 위해 **Worker Threads**를 사용하여 무거운 연산을 다른 스레드로 분리하거나, 마이크로서비스로 쪼개어 별도 프로세스에 맡기는 설계를 제안해야 한다.

### 2. 기술사적 인사이트: 마이크로태스크 vs 매크로태스크
- 기술사라면 이벤트 루프 내의 우선순위인 **마이크로태스크 큐 (Microtask Queue)**를 언급해야 한다. `Promise`와 같은 마이크로태스크는 일반적인 `setTimeout`보다 우선적으로 처리된다. 이 미세한 차이를 이해해야만 고도의 실시간성이 요구되는 웹 애플리케이션의 렌더링 지연(Jank) 문제를 근본적으로 해결할 수 있다.

- **📢 섹션 요약 비조**: "순서만 지키는 것이 아니라, 급한 일(Microtask)과 덜 급한 일(Macrotask)을 하드웨어 레벨에서 지능적으로 선별하는 것이 현대 이벤트 드리븐 설계의 정수입니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **높은 처리량(Throughput)**: 대기 시간 없는 CPU 활용으로 동시 접속 성능 향상.
- **비용 절감**: 멀티스레드 대비 낮은 메모리 점유로 인프라 비용 최적화.

### 2. 미래 전망
앞으로는 하드웨어가 비동기 이벤트를 직접 큐잉하고 처리하는 **하드웨어 가속 이벤트 루프** 기술이 발전할 것이다. 또한, 코루틴(Coroutine)과 결합하여 비동기 코드를 동기식 코드처럼 읽기 쉽게 만드는 추상화 기술이 더욱 고도화될 것이다. 이벤트 루프는 '기다림'을 '기회'로 바꾼 운영체제와 런타임 설계의 가장 실용적인 혁신이다.

- **📢 섹션 요약 비유**: 쉴 새 없이 돌아가는 관람차처럼, 각 칸(이벤트)에 손님(데이터)을 태우고 내리며 멈추지 않고 시스템을 가동하는 디지털 세상의 영구 기관과 같습니다.
