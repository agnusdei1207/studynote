# 지명 파이프 (Named Pipe / FIFO)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 지명 파이프 (Named Pipe) 또는 **FIFO**는 파일 시스템에 고유한 이름을 가진 특수 파일을 생성하여, 부모-자식 관계가 없는 **서로 독립적인 프로세스 간의 통신**을 가능케 하는 IPC 방식이다.
> 2. **가치**: 익명 파이프의 '혈연 관계 한정'이라는 제약을 극복하며, 프로세스가 종료되어도 파이프 파일이 유지되어 언제든 새로운 프로세스가 통신에 참여할 수 있는 영속성을 제공한다.
> 3. **융합**: 유닉스/리눅스의 파일 인터페이스(`mkfifo`)를 그대로 사용하며, 윈도우 환경에서는 네트워크를 통한 원격 프로세스 간의 양방향 통신 수단으로까지 확장되어 활용된다.

+++

## Ⅰ. 지명 파이프 (FIFO)의 개요

### 1. 정의
- 파일 시스템 내에 실제로 존재하는 이름(Path)을 가진 파이프이다. 데이터를 선입선출(First-In-First-Out) 방식으로 전달하기 때문에 FIFO라고도 불린다.

### 2. 주요 특징
- **관계 독립성**: 부모-자식 관계가 아니더라도 파일 경로만 알면 어떤 프로세스든 접근 가능하다.
- **영속성 (Persistence)**: 통신 중인 프로세스가 없어도 파일 시스템에 객체로 남아있다. (데이터는 메모리 버퍼에만 있으므로 파일 크기는 항상 0으로 표시됨)
- **파일 인터페이스**: `open()`, `read()`, `write()` 등 표준 파일 함수를 그대로 사용하여 구현이 매우 쉽다.

### 3. 💡 비유: '동네 공용 우체통'
- 익명 파이프가 우리 집 식구들만 쓰는 '내부 인터폰'이라면, 지명 파이프는 **'길거리에 세워진 공용 우체통'**과 같다.
- 우체통의 위치(파일 경로)만 알면 모르는 사람(독립 프로세스)끼리도 편지를 넣고(Write) 가져갈(Read) 수 있다. 편지를 다 가져가도 우체통(파일 객체)은 그 자리에 계속 남아있다.

## Ⅱ. 지명 파이프 동작 메커니즘 (Deep Dive)

### 1. 생성 및 연결 흐름
1. **Creation**: `mkfifo("/tmp/my_pipe", 0666)` 명령으로 특수 파일을 생성한다.
2. **Open**: 프로세스 A는 쓰기 모드(`O_WRONLY`)로, 프로세스 B는 읽기 모드(`O_RDONLY`)로 해당 파일을 연다.
3. **Blocking**: 한쪽에서만 열면 상대방이 열 때까지 커널이 해당 프로세스를 블록(Block)시킨다. (동기화 보장)
4. **Communicate**: 데이터를 주고받는다.
5. **Close/Remove**: 사용 후 파일을 닫고, 필요 없으면 `rm` 명령으로 삭제한다.

### 2. 익명 파이프와의 차이점
| 항목 | 익명 파이프 (Anonymous Pipe) | 지명 파이프 (Named Pipe) |
|:---|:---|:---|
| **파일 이름** | 없음 (커널 내부 FD만 존재) | **있음 (디렉토리 내 파일)** |
| **통신 범위** | 부모-자식 등 혈연 관계 | **관계없는 모든 프로세스** |
| **지속 시간** | 프로세스 종료 시 소멸 | **명시적 삭제 전까지 유지** |
| **통신 방향** | 단방향 | 단방향 (단, 윈도우는 양방향 지원) |

### 3. ASCII 구조 다이어그램
```text
    [ Process A (Sender) ]      [ File System ]         [ Process B (Receiver) ]
    ┌──────────────────┐        ┌──────────────┐        ┌────────────────────┐
    │ open("my_fifo",W)│ ──▶    │  /tmp/my_fifo│ ◀──    │ open("my_fifo", R) │
    └────────┬─────────┘        └──────┬───────┘        └──────────┬─────────┘
             │                         │                           │
             │ (Write Data)            ▼ (Kernel Buffer)           │ (Read Data)
             └──────────────────▶ [ FIFO Queue ] ──────────────────┘
```

- **📢 섹션 요약 비유**: 특정 장소에 비밀 함(FIFO 파일)을 만들어두고, 약속된 시간에 만나지 않아도 물건을 두고 가면 나중에 찾아가는 비동기식 협업 공간과 같습니다.

## Ⅲ. 플랫폼별 특이점 (Linux vs Windows)

지명 파이프는 운영체제마다 구현 방식에 차이가 있다.

| 구분 | 리눅스 / 유닉스 (POSIX) | 윈도우 (Win32) |
|:---|:---|:---|
| **기본 철학** | "모든 것은 파일이다" | "통신을 위한 객체이다" |
| **통신 방향** | 주로 단방향 (FIFO) | **양방향 (Full-duplex) 지원** |
| **원격 통신** | 불가 (로컬 전용) | **네트워크 기반 원격 IPC 가능** |
| **보안** | 파일 권한(Permission) 기반 | ACL (Access Control List) 기반 |

- **📢 섹션 요약 비유**: 리눅스의 FIFO가 간단한 일방통행 골목길이라면, 윈도우의 Named Pipe는 다리 건너 마을(원격지)까지 연결된 2차선 고속도로와 같습니다.

## Ⅳ. 실무 적용 및 운영 전략 (Strategy)

### 1. 실무 시나리오: 로그 수집기 (Log Aggregator) 구현
- **상황**: 서로 다른 언어로 짜여진 10개의 독립적인 프로세스들이 남기는 로그를 하나의 중앙 프로세스가 수집해야 함.
- **판단**: 각 프로세스마다 복잡한 소켓 설정을 하는 대신, `/var/log/app_fifo`라는 **지명 파이프**를 만든다. 모든 프로세스는 이 파일에 로그를 쓰기만 하면 된다. 수집 프로세스는 이 파일을 읽기 모드로 열어두고 들어오는 데이터를 순서대로 처리한다. 구성이 매우 단순하고 강력한 로그 파이프라인이 완성된다.

### 2. 기술사적 인사이트: 데이터 정합성과 원자성 (Atomicity)
- 지명 파이프에 여러 프로세스가 동시에 쓸 때, 데이터가 섞이지 않으려면 **PIPE_BUF** 크기를 고려해야 한다. 기술사라면 POSIX 표준에서 보장하는 원자적 쓰기 크기(보통 4KB) 이하의 메시지는 섞이지 않음을 인지하고, 이를 초과하는 대용량 메시지는 별도의 뮤텍스(Mutex)나 애플리케이션 레벨의 프레임워크로 보호해야 함을 제시해야 한다.

- **📢 섹션 요약 비유**: 우체통에 편지를 넣을 때, 한 통씩 쏙 집어넣으면 안전하지만(원자성), 아주 긴 두루마리 편지를 여러 명이 동시에 밀어 넣으면 중간에 글자가 겹쳐 읽을 수 없게 되는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **확장성 제고**: 프로세스 간의 물리적 결합도를 낮추어 유연한 시스템 아키텍처 구축.
- **개발 생산성**: 복잡한 네트워크 스택 지식 없이도 안정적인 IPC 구현 가능.

### 2. 미래 전망
최근 컨테이너 환경에서는 지명 파이프를 통해 호스트의 도커 데몬(`docker.sock`)과 소통하는 방식이 일반화되어 있다. 비록 대규모 분산 시스템에서는 gRPC나 메시지 큐에 자리를 내주고 있지만, 로컬 머신 내에서 가장 가볍고 확실하게 '이름 기반의 소통'을 보장하는 지명 파이프의 가치는 여전히 유효하다.

- **📢 섹션 요약 비유**: 화려한 SNS(네트워크 IPC) 시대에도, 집 앞의 정겨운 우체통(지명 파이프)은 여전히 가장 확실한 소식통입니다.
