# Grand Central Dispatch (GCD) 블록 및 디스패치 큐 기반 동시성 구조

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 개발자가 직접 스레드를 관리(Thread Management)하는 대신, 실행할 작업(Block/Closure)을 큐(Dispatch Queue)에 던지면 **OS 수준에서 시스템 가용 자원에 맞춰 효율적으로 스케줄링**하는 저수준 동시성 프레임워크.
> 2. **가치**: 스레드 생성 오버헤드와 컨텍스트 스위칭 비용을 시스템이 최적화하여 최소화하고, 비동기(Asynchronous) 처리를 통해 UI 응답성(Responsiveness)을 보장함.
> 3. **융합**: C언어 확장인 **Blocks**, 커널의 **kqueue**, 그리고 하드웨어의 멀티코어 성능을 유기적으로 결합하여 현대 애플 시스템의 성능을 지탱하는 핵심 엔진.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: GCD(Grand Central Dispatch)는 애플이 멀티코어 프로세서 시스템에서 병렬 실행을 최적화하기 위해 도입한 기술이다. 개발자는 `Dispatch Queue`라는 추상화된 대기열에 작업을 넣기만 하면 된다.
- **💡 비유**: **중앙 통제실이 있는 거대한 물류 터미널**과 같다. 개별 기사(스레드)를 고용할 필요 없이, 택배상자(작업 블록)를 컨베이어 벨트(디스패치 큐)에 올려두면 통제실(OS 커널)이 현재 사용 가능한 트럭 대수에 맞춰 자동으로 배분하여 배송을 완료한다.
- **등장 배경**:
  - ① **기존 한계**: `POSIX Threads(pthreads)`와 같은 전통적인 스레딩 모델은 개발자가 스레드 개수, 생명주기, 락(Lock) 경합을 직접 관리해야 했으며, 이는 시스템 부하와 무관하게 설계되어 자원 낭비가 심했다.
  - ② **혁신적 패러다임**: "스레드 중심"에서 "작업(Task) 중심"으로 패러다임을 전환했다. 운영체제가 시스템 전체의 로드를 파악하여 스레드 풀의 크기를 동적으로 조절한다.
  - ③ **현재의 비즈니스 요구**: 모바일 기기의 전력 효율(Power Efficiency)과 부드러운 사용자 경험(UX)이 중요해지면서, 코어 자원을 낭비하지 않는 정교한 동시성 제어가 필수가 되었다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. Dispatch Queue의 유형 및 특징

| 유형 | 설명 | 동작 방식 | 주요 용도 |
|:---|:---|:---|:---|
| **Main Queue** | 앱의 메인 스레드에서 실행되는 큐 | 직렬(Serial), 오직 하나만 존재. | **UI 업데이트**, 사용자 입력 처리 |
| **Global Queues** | 시스템 전체에서 공유되는 병렬 큐 | 병렬(Concurrent), 우선순위(QoS)별로 구분. | 백그라운드 계산, 네트워크 요청 |
| **Custom Queues** | 개발자가 직접 생성하는 큐 | Serial 또는 Concurrent 선택 가능. | 특정 작업 그룹의 순서 보장/격리 |

#### 2. GCD 실행 메커니즘 (ASCII)

GCD는 사용자 공간의 라이브러리(`libdispatch`)와 커널 공간의 지원이 결합된 형태다.

```ascii
    [ User Application ]
    |  dispatch_async(queue, ^{ /* Work Block */ });
    v
    +------------------------------------------+
    | libdispatch (User Space Library)         |
    | - Queue Management / Task Enqueueing     |
    +------------------|-----------------------+
                       v
    +------------------------------------------+
    | Darwin Kernel (XNU)                      |
    | - Workloop / Thread Pool Management      |
    | - kqueue / Mach Ports (Events)           |
    +------------------|-----------------------+
                       v
    +------------------------------------------+
    | Hardware (Multi-core CPU)                |
    | [ Core 0 ] [ Core 1 ] [ Core 2 ] [ Core 3]
    +------------------------------------------+
```

**[다이어그램 해설]**
1. 개발자가 작업을 블록 형태로 큐에 비동기(`dispatch_async`)로 투입한다.
2. `libdispatch`는 해당 작업을 내부 데이터 구조에 저장하고 시스템 상태를 확인한다.
3. 커널은 현재 시스템의 유휴 CPU 자원과 작업의 우선순위(QoS)를 고려하여 스레드 풀에서 적절한 스레드를 깨우거나 생성한다.
4. 하드웨어 코어에 작업이 할당되어 실행되며, 작업이 끝나면 스레드는 다시 풀로 돌아가거나 대기한다.

#### 3. QoS (Quality of Service) 우선순위 정책
GCD는 작업의 성격에 따라 4가지 주요 QoS 레벨을 제공하여 시스템 자원 배분을 최적화한다.
- **User Interactive**: 즉각적인 UI 응답이 필요한 작업 (최고 우선순위).
- **User Initiated**: 사용자가 시작한 빠른 결과가 필요한 작업 (예: 문서 열기).
- **Utility**: 데이터 계산, 네트워킹 등 시간이 좀 걸리는 작업 (진행 바 표시).
- **Background**: 사용자에게 보이지 않는 유지보수 작업 (예: 인덱싱, 백업).

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. GCD vs 전통적인 멀티스레딩 (pthreads)

| 비교 항목 | GCD (Grand Central Dispatch) | Traditional Multi-threading |
|:---|:---|:---|
| **추상화 수준** | **높음 (Task 중심)** | 낮음 (Thread 중심) |
| **자원 관리** | OS가 시스템 로드에 따라 자동 조절 | 개발자가 수동으로 개수 및 수명 관리 |
| **코드 가독성** | 클로저(Block) 사용으로 간결함 | 스레드 함수 정의 및 파라미터 전달 복잡 |
| **오버헤드** | 최소화 (커널-유저 협업 최적화) | 높음 (잦은 스레드 생성/소멸 시) |
| **메인 용도** | 일반적인 앱 동시성 구현 | 하위 호환성, 크로스 플랫폼 저수준 제어 |

#### 2. 과목 융합 관점 (Synergy)
- **컴퓨터 구조 (CA)**: GCD는 하드웨어의 멀티코어 아키텍처를 최대한 활용하면서도, 과도한 스레드 생성으로 인한 **캐시 오염(Cache Thrashing)**을 방지하는 스마트한 스케줄러 역할을 한다.
- **알고리즘 (Algorithm)**: 병렬 처리가 가능한 알고리즘(예: 병합 정렬, 이미지 필터링)을 GCD의 `dispatch_apply` 등을 통해 간단하게 멀티코어 알고리즘으로 가속할 수 있다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실무 시나리오: UI 프리징(Freezing) 해결
- **문제**: 사용자가 버튼을 눌렀을 때 5초 동안 화면이 멈추고 조작이 되지 않음.
- **원인 분석**: 메인 스레드(Main Queue)에서 대용량 이미지 처리나 네트워크 동기 호출을 수행하여 UI 이벤트 루프가 차단됨.
- **기술사적 결단**:
  - 무거운 작업은 `dispatch_async(global_queue, ...)`를 통해 백그라운드 큐로 보낸다.
  - 작업이 완료된 후 결과를 UI에 반영할 때만 다시 `dispatch_async(dispatch_get_main_queue(), ...)`를 호출하여 안전하게 UI를 업데이트한다.

#### 2. 도입 체크리스트 및 주의사항
- **Deadlock 주의**: Serial 큐에서 현재 실행 중인 블록이 동일한 큐에 동기(`dispatch_sync`)로 작업을 던지면 데드락이 발생한다. (절대 금기)
- **스레드 폭발(Thread Explosion)**: Concurrent 큐에 너무 많은 작업을 동시에 던지면 시스템이 수백 개의 스레드를 만들려고 시도하여 오히려 성능이 급락할 수 있다. (Semaphore나 전용 Serial 큐로 흐름 제어 필요)

#### 3. 안티패턴
- **UI 업데이트를 백그라운드에서 수행**: iOS/macOS의 UI 프레임워크는 스레드 안전(Thread-safe)하지 않다. 백그라운드 큐에서 UI를 건드리면 앱이 예측 불가능하게 크래시되거나 그래픽 깨짐 현상이 발생한다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과

| 구분 | 도입 전 | GCD 도입 후 |
|:---|:---|:---|
| **응답 속도** | 작업 완료 전까지 UI 먹통 | 비동기 처리를 통한 상시 응답성 유지 |
| **배터리 수명** | 비효율적인 스레드 생성으로 CPU 낭비 | 필요 최소한의 자원만 사용하는 최적 스케줄링 |
| **유지보수성** | 복잡한 락(Lock)과 동기화 로직 | 큐 기반의 단순한 동시성 설계 가능 |

#### 2. 미래 전망
최신 Swift 언어에서는 GCD를 한 단계 더 추상화한 **Swift Concurrency (async/await, Actor)**가 도입되었다. 이는 컴파일 타임에 데이터 레이스(Data Race)를 체크하여 더 안전한 프로그래밍을 가능하게 한다. 하지만 그 밑단에서는 여전히 GCD와 커널의 협업 구조가 시스템을 지탱하고 있으며, GCD는 앞으로도 고성능 시스템 프로그래밍의 핵심 표준으로 남을 것이다.
