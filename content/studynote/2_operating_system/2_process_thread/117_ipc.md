# 프로세스 간 통신 (IPC, Inter-Process Communication)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IPC (Inter-Process Communication)는 운영체제에 의해 메모리가 격리된 독립적인 프로세스들이 서로 **데이터를 주고받고 실행을 동기화**할 수 있게 해주는 커널 제공 메커니즘의 총칭이다.
> 2. **가치**: 프로세스 간의 물리적 장벽을 허물어 복잡한 시스템을 모듈화된 다중 프로세스로 설계할 수 있게 하며, 병렬 컴퓨팅과 분산 시스템 구축의 필수 기술이다.
> 3. **융합**: 고성능 연산을 위한 **공유 메모리 (Shared Memory)** 방식과 안정적인 통신을 위한 **메시지 전달 (Message Passing)** 방식으로 대별되며, 현대 OS는 파이프, 소켓, 신호 등 다양한 파생 형태를 통해 최적의 소통 통로를 제공한다.

+++

## Ⅰ. IPC의 개요

### 1. 정의
- 독립된 주소 공간을 가진 프로세스들 사이에서 정보를 교환하는 기술이다.

### 2. 왜 필요한가? (The Barrier)
- 운영체제는 보안을 위해 프로세스 A가 프로세스 B의 메모리를 함부로 읽지 못하게 막는다 (**Memory Protection**). 하지만 협력을 위해서는 정보를 넘겨줘야 하므로, 운영체제가 공인한 '안전한 대화 창구'가 필요한데 그것이 바로 IPC이다.

### 3. 💡 비유: '격리된 섬들 사이의 소통'
- 각 프로세스는 바다(커널)로 둘러싸인 **'격리된 섬'**과 같다.
- IPC는 섬 사이에 **다리(공유 메모리)**를 놓아 물건을 직접 나르거나, 바다에 **메시지가 담긴 병(메시지 전달)**을 띄워 소식을 전하는 것과 같다.

+++

## Ⅱ. IPC의 두 가지 핵심 모델 (Deep Dive)

모든 IPC 기법은 본질적으로 다음 두 모델 중 하나에 속한다.

### 1. 공유 메모리 (Shared Memory)
- **개념**: 두 프로세스가 공통으로 접근할 수 있는 메모리 영역을 설정한다.
- **장점**: 커널 개입 없이 직접 접근하므로 **속도가 가장 빠르다**.
- **단점**: 동시에 접근할 때 데이터가 깨질 수 있어 별도의 **동기화(세마포어 등)**가 필수적이다.

### 2. 메시지 전달 (Message Passing)
- **개념**: 커널을 통해 고정된 크기나 가변 크기의 메시지를 송수신한다 (`send`, `receive`).
- **장점**: 커널이 동기화를 대신 해주므로 안전하며, 분산 시스템(네트워크)으로 확장이 쉽다.
- **단점**: 매번 커널 모드 진입(System Call)이 필요해 속도가 상대적으로 느리다.

### 3. ASCII 구조 다이어그램
```text
    [ Shared Memory Model ]             [ Message Passing Model ]
    ┌──────────┐   ┌──────────┐         ┌──────────┐   ┌──────────┐
    │ Process A │   │ Process B │         │ Process A │   │ Process B │
    └────┬─────┘   └─────┬────┘         └────┬─────┘   └─────┬────┘
         │   (Direct)    │                   │   (Indirect)  │
         └──────┬────────┘                   ▼               ▼
                ▼                   ┌──────────────────────────────────┐
    ┌──────────────────────────┐    │         OS Kernel (Queue)        │
    │      Shared Memory       │    └──────────────────────────────────┘
    └──────────────────────────┘
```

- **📢 섹션 요약 비유**: 공유 메모리는 공용 게시판에 글을 써두고 같이 보는 것이고, 메시지 전달은 편지(커널)를 써서 상대방 우체통에 넣어주는 방식입니다.

+++

## Ⅲ. IPC 모델별 특징 비교 분석

| 비교 항목 | 공유 메모리 (Shared Memory) | 메시지 전달 (Message Passing) |
|:---|:---|:---|
| **통신 속도** | **매우 빠름** (최상) | 보통 (커널 오버헤드 발생) |
| **동기화 주체** | **사용자 앱** (직접 관리) | **운영체제 커널** (자동 관리) |
| **데이터 크기** | 대용량 데이터 적합 | 작은 제어 정보 적합 |
| **구현 난이도** | 높음 (Race Condition 위험) | 낮음 (System Call 사용) |

- **📢 섹션 요약 비유**: 친한 친구끼리 일기장을 같이 쓰는 것(공유 메모리)은 빠르지만 싸우기 쉽고, 우체국을 통해 편지를 주고받는 것(메시지 전달)은 조금 느려도 질서가 잡히는 것과 같습니다.

+++

## Ⅳ. 실무 적용 및 아키텍처 가이드 (Strategy)

### 1. 실무 시나리오: 비디오 플레이어의 싱크 조절
- **상황**: 영상 재생 프로세스와 음성 재생 프로세스가 시간을 맞춰야 함.
- **판단**: 아주 잦은 신호 교환이 필요하므로 **공유 메모리**에 현재 재생 시간(Timestamp)을 적어두고 서로 확인하게 한다. 대신, 값이 꼬이지 않도록 가벼운 락(Mutex)을 사용한다.

### 2. 기술사적 인사이트: 마이크로 커널과 IPC
- 마이크로 커널(Microkernel) 시스템은 거의 모든 기능을 별도 프로세스로 분리하므로 **IPC 성능이 곧 OS 전체 성능**을 결정한다. 기술사라면 IPC 오버헤드를 줄이기 위한 **Zero-copy** 기술이나, L4 커널의 고속 메시지 전달 아키텍처를 제시하며 성능 최적화 방안을 논해야 한다.

- **📢 섹션 요약 비유**: 회사의 모든 부서가 칸막이로 나뉘어 있다면(마이크로 커널), 부서 간의 연락망(IPC)이 얼마나 빠르냐가 회사의 경쟁력이 되는 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **모듈성 향상**: 큰 프로그램을 작고 관리하기 쉬운 프로세스들로 쪼개어 개발 가능.
- **내결함성 확보**: 한 프로세스가 죽어도 IPC 통로만 살아있다면 다른 프로세스는 안전함.

### 2. 미래 전망
최근 클라우드 네이티브 환경에서는 IPC의 개념이 단일 노드를 넘어 **gRPC**나 **Message Broker(Kafka, RabbitMQ)** 등을 통한 분산 IPC로 확장되고 있다. 또한 하드웨어 레벨에서 프로세스 간 통신을 가속하는 **CXL (Compute Express Link)** 기술 등이 도입되면서, IPC는 소프트웨어의 한계를 넘어 하드웨어 아키텍처의 핵심 요소로 자리매김할 것이다.

- **📢 섹션 요약 비유**: 끊어진 섬들을 연결하여 거대한 대륙(강력한 시스템)으로 만드는 운영체제의 위대한 통신망입니다.
