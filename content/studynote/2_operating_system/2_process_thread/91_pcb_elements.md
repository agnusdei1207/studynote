# PCB 요소 - PID, 상태, PC, 레지스터, 스케줄링 정보 등

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PCB 요소는 운영체제가 프로세스를 개별적으로 식별하고 실행 흐름을 제어하기 위해 기록하는 **메타데이터의 집합**으로, 프로세스의 신분, 상태, 자원 소유 현황을 망라한다.
> 2. **가치**: CPU 레지스터 값과 프로그램 카운터(PC) 정보를 보존함으로써 시분할 환경의 핵심인 문맥 교환(Context Switch)을 가능케 하며, 우선순위 및 회계 정보를 통해 공정한 자원 배분을 지원한다.
> 3. **융합**: 하드웨어의 현재 연산 상태가 소프트웨어적 자료구조로 변환되어 저장되는 지점이며, 커널 스케줄러가 '다음에 누구를 실행할지' 결정하는 판단의 데이터 소스가 된다.

---

## Ⅰ. PCB의 주요 구성 요소 (Deep Dive)

PCB는 크게 4가지 카테고리의 정보를 담고 있다.

### 1. 프로세스 식별 및 상태 정보
- **PID (Process ID)**: 시스템 전체에서 유일한 프로세스 고유 번호.
- **PPID (Parent PID)**: 자신을 생성한 부모 프로세스의 번호.
- **Process State**: 현재 상태 (Ready, Running, Waiting 등).

### 2. 실행 컨텍스트 (Execution Context)
- **PC (Program Counter)**: 이 프로세스가 다음에 실행할 명령어의 주소.
- **CPU Registers**: 누산기(Accumulator), 인덱스 레지스터, 스택 포인터 등 현재 CPU의 모든 레지스터 값 보존. (문맥 교환의 핵심 데이터)

### 3. CPU 스케줄링 정보
- **Priority**: 프로세스의 우선순위.
- **Scheduling Queue Pointers**: 준비 큐나 대기 큐에서 다음/이전 PCB를 가리키는 포인터.
- **Time Slice / Quanta**: 남은 CPU 할당 시간.

### 4. 자원 관리 및 회계 정보
- **Memory Management Info**: 페이지 테이블 정보, 세그먼트 테이블 주소 등.
- **I/O Status Info**: 프로세스가 열어놓은 파일 목록(File Descriptor), 할당된 입출력 장치 목록.
- **Accounting Info**: CPU 사용 시간 기록, 프로세스 시작 이후 경과 시간 등.

### 5. 💡 비유: '여행자의 여권과 수하물 태그'
- **PID/상태**: 여권의 이름과 사진. 내가 누구인지, 지금 입국 대기 중인지 여행 중인지를 알려준다.
- **PC/레지스터**: 소설책 사이에 끼워둔 **'책갈피'**. 어디까지 읽었는지 정확히 표시해준다.
- **메모리/파일 정보**: 여행 가방(자원)의 목록. 내가 어떤 짐을 가지고 있는지 적혀 있다.

## Ⅱ. PCB 요소의 기술적 역할과 전이

### 1. 문맥 교환 시의 데이터 흐름
- **Save**: 실행 중인 프로세스가 CPU를 반납할 때, 현재 CPU의 PC와 모든 레지스터 값을 해당 프로세스의 **PCB**에 덤프(Dump)한다.
- **Restore**: 새로운 프로세스가 실행될 때, 그 프로세스의 **PCB**에 적힌 PC와 레지스터 값을 CPU로 로드(Load)한다.

### 2. ASCII 구조 다이어그램
```text
    ┌──────────────────────────────────┐
    │      Process ID (PID)            │ ◀── "Who am I?"
    ├──────────────────────────────────┤
    │      Process State               │ ◀── "Ready, Running, Waiting..."
    ├──────────────────────────────────┤
    │      Program Counter (PC)        │ ◀── "Where was I?"
    ├──────────────────────────────────┤
    │      Registers (EAX, EBX, SP...) │ ◀── "What was I doing?"
    ├──────────────────────────────────┤
    │      Memory Management Info      │ ◀── "Where is my data?"
    ├──────────────────────────────────┤
    │      List of Open Files          │ ◀── "What resources do I have?"
    ├──────────────────────────────────┤
    │      Scheduling / Priority       │ ◀── "How important am I?"
    └──────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 영화 촬영 중 감독(운영체제)이 "컷!"을 외치면, 소품 위치와 배우의 동작(레지스터)을 사진 찍어두고(저장), 나중에 다시 "레디 액션!"을 할 때 그 사진대로 세팅(복구)하는 것과 같습니다.

## Ⅲ. PCB 요소 간의 상관관계 분석

| 요소 | 영향받는 커널 모듈 | 가치 |
|:---|:---|:---|
| **Priority** | CPU 스케줄러 | 중요도에 따른 차등 서비스 구현 |
| **Memory Ptr** | 메모리 관리자 (MMU) | 프로세스 간 메모리 침범 방지 |
| **Accounting** | 회계/과금 모듈 | 자원 사용량에 따른 과금 및 분석 |
| **PC** | 인터럽트 핸들러 | 비정상 중단 후의 안전한 재개 보장 |

- **📢 섹션 요약 비유**: 이력서의 경력 사항(Accounting)과 보유 자격증(Resource List)이 서로 맞물려 그 사람의 연봉(Priority)을 결정하는 것과 같습니다.

## Ⅳ. 실무 적용 및 보안 인사이트 (Strategy)

### 1. 실무 시나리오: 파일 디스크립터 누수 (FD Leak)
- **상황**: 프로그램이 "Too many open files" 에러를 내며 멈춤.
- **판단**: PCB 내부의 **Open Files List**가 꽉 찬 것이다. 프로세스가 파일을 열고(`open`) 닫지(`close`) 않아 생긴 문제로, `lsof` 명령어를 통해 해당 프로세스의 PCB 자원 점유 상태를 확인하고 코드를 수정해야 한다.

### 2. 기술사적 인사이트: PCB 정보 변조 공격
- 커널 모드 권한을 얻은 공격자가 PCB의 **Priority** 필드를 조작하여 자신의 악성 코드를 최우선으로 실행시키거나, **Memory Pointer**를 조작하여 다른 프로세스의 데이터를 훔쳐볼 수 있다. 이를 막기 위해 현대 OS는 PCB 구조체를 읽기 전용 영역에 두거나 하드웨어적으로 보호하는 기술을 적용한다.

- **📢 섹션 요약 비유**: 신분증(PID) 사진을 바꿔치기하거나 성적표(Priority)를 고치는 부정행위를 막기 위해, 장부를 금고(커널 메모리)에 넣고 철저히 감시해야 합니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **무결성 보장**: 프로세스 전환 전후의 데이터 정합성을 완벽히 유지.
- **추적 가능성**: 프로세스의 모든 행적과 자원 소유를 체계적으로 기록.

### 2. 미래 전망
최근에는 멀티코어를 넘어 수만 개의 코어를 가진 가속기 환경에서 PCB의 정보를 더욱 가볍게 유지하려는 **Lightweight PCB** 기술이 연구되고 있다. 또한, 프로세스가 컨테이너나 가상 머신 간에 이동할 때 PCB 요소를 직렬화(Serialization)하여 전송하는 기술이 클라우드 마이그레이션의 핵심이 되고 있다. PCB 요소는 운영체제가 프로세스라는 생명체에게 부여한 '영혼의 설계도'와 같다.

- **📢 섹션 요약 비유**: 한 사람의 인생 기록(PCB 요소)이 꼼꼼할수록 그 사람을 더 잘 이해하고 도울 수 있는 것과 같습니다.
