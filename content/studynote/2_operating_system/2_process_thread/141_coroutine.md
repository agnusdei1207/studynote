# 코루틴 (Coroutine)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 코루틴(Coroutine)은 실행 중인 지점에서 잠시 멈췄다가(Suspend), 나중에 **멈춘 지점부터 다시 실행을 재개(Resume)**할 수 있는 비선점형(Non-preemptive) 서브루틴의 확장 개념이다.
> 2. **가치**: 운영체제 커널의 개입 없이 사용자 영역에서 실행 흐름을 제어하므로 문맥 교환 오버헤드가 거의 없으며, **협력적 멀티태스킹 (Cooperative Multitasking)**을 통해 고도의 동시성을 효율적으로 구현한다.
> 3. **융합**: 일반적인 함수(서브루틴)가 '호출-종료'의 단선적 구조인 것과 달리, 코루틴은 여러 번 진입하고 나갈 수 있는 구조를 가지며 비동기 프로그래밍(Async/Await)의 핵심 원리로 사용된다.

+++

## Ⅰ. 코루틴의 정의 및 핵심 철학

### 1. 정의
- "함께(Co)"와 "루틴(Routine)"의 합성어로, 서로를 호출하고 제어권을 주고받으며 협력하는 일련의 코드 블록들이다.

### 2. 핵심 철학: '협력과 양보'
- **비선점형**: 운영체제가 강제로 실행권을 뺏지 않는다. 코루틴 스스로가 "여기까지 하고 쉴게"라고 양보(`yield`)해야 다른 코루틴이 실행될 수 있다.
- **상태 보존**: 실행이 멈출 때 현재의 지역 변수와 스택 정보를 그대로 유지하여, 재개 시 이전 상태에서 곧바로 이어갈 수 있다.

### 3. 💡 비유: '여러 명이 같이 쓰는 조리대'
- 일반 함수(서브루틴)는 **'요리사가 주방에 들어가서 요리 하나를 완성하고 아예 퇴근하는 것'**과 같다.
- 코루틴은 **'요리사가 스테이크를 굽다가 고기가 익는 동안(대기), 다른 요리사에게 조리대를 양보하고 자기는 옆에서 재료를 손질하다가, 다시 돌아와 고기를 뒤집는 것'**과 같다.
- 주방(CPU)은 하나지만, 요리사들(코루틴)이 서로 양보하며 동시에 여러 요리를 진행하는 지혜로운 방식이다.

+++

## II. 서브루틴 vs 코루틴 (Deep Dive)

### 1. 서브루틴 (Subroutine)
- 진입점(Entry)은 하나이며, 한 번 종료(Return)되면 모든 정보가 사라진다. 부모-자식의 상하 관계가 뚜렷하다.

### 2. 코루틴 (Coroutine)
- 진입점이 여러 개일 수 있다. `yield`를 통해 중단하고 `resume`을 통해 복귀한다. 호출자와 대등한 관계에서 데이터를 주고받는다.

### 3. ASCII 구조 다이어그램
```text
    [ Subroutine Flow ]                 [ Coroutine Flow ]
    ┌──────────┐                        ┌──────────┐
    │  Call A  │ ──▶ Start              │ Resume A │ ──▶ Start
    └──────────┘      │                 └────┬─────┘      │
                      ▼                      ▲            ▼
    ┌──────────┐    Finish              ┌────┴─────┐    Suspend (yield)
    │ Return   │ ◀──┘                   │ Yield A  │ ◀──┘
    └──────────┘                        └──────────┘
                                             │
                                        ┌────▼─────┐
                                        │ Resume A │ ──▶ Resume from yield
                                        └──────────┘
```

- **📢 섹션 요약 비유**: 서브루틴이 '한 번에 쭉 읽고 끝내는 소설'이라면, 코루틴은 '중간에 책갈피를 끼워두고 나중에 다시 펼쳐보는 전집'과 같습니다.

+++

## Ⅲ. 코루틴의 유형 및 장단점

| 구분 | 스택 기반 (Stackful) | 스택리스 (Stackless) |
|:---|:---|:---|
| **상태 저장** | 별도의 스택을 가짐 | 현재 함수의 상태만 저장 |
| **유연성** | 깊은 함수 호출 중에도 중단 가능 | 오직 최상위 함수에서만 중단 가능 |
| **언어 사례** | Lua, Go (Goroutine) | Python (Generator), JavaScript, C# |

### 장단점 요약
- **장점**: 문맥 교환 비용이 프로세스나 스레드에 비해 압도적으로 낮음. 락(Lock) 없이도 동기화가 용이함.
- **단점**: 코드가 무한 루프에 빠지면 시스템 전체(또는 해당 스레드)가 멈춤. CPU 집중 작업에는 부적합.

+++

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 비동기 데이터 처리 (Async/Await)
- **상황**: 수천 개의 API 요청을 보내고 결과를 기다려야 함.
- **판단**: 스레드를 수천 개 만드는 것은 낭비이다. **코루틴**을 사용한다. 네트워크 응답을 기다리는 동안 코루틴은 `yield`를 통해 실행권을 반납하고, 데이터가 도착하면 `resume`되어 다음 로직을 처리한다. 이를 통해 단 한 개의 스레드만으로도 수만 개의 비동기 요청을 효율적으로 관리할 수 있다.

### 2. 기술사적 인사이트: 제어의 역전 (Inversion of Control)
- 기술사라면 코루틴이 프로그래밍 패러다임에 미친 영향을 논해야 한다. 과거에는 호출자가 주도권을 쥐었으나, 코루틴은 피호출자가 스스로 실행을 중단하고 상태를 유지함으로써 **제어의 역전**을 하드웨어 레벨이 아닌 코드 레벨에서 실현했다. 이는 이벤트 기반 아키텍처에서 '가독성'과 '성능'을 동시에 잡는 핵심 설계 도구임을 강조해야 한다.

- **📢 섹션 요약 비조**: "강제로 뺏는 것(선점형)보다 스스로 비켜주는 것(협력적)이, 신뢰할 수 있는 개발 환경에서는 훨씬 더 정교하고 강력한 성능을 만들어냅니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **메모리 절약**: 스레드 스택 오버헤드 제거 (수 MB -> 수 KB).
- **응답성 극대화**: 초고속 실행 제어권 전환을 통한 서비스 지연 최소화.

### 2. 미래 전망
앞으로는 코루틴이 언어 차원의 문법을 넘어, 운영체제 커널 내부의 I/O 서브시스템(io_uring 등)과 직접 결합하여 **하드웨어 가속형 코루틴** 시대로 진화할 것이다. 또한, 머신러닝의 텐서 연산 파이프라인 최적화에 코루틴의 '상태 유지 중단' 개념이 적극 도입될 것이다. 코루틴은 '멈출 줄 아는 지혜'가 최고의 성능을 만든다는 것을 증명한 현대 전산학의 정수이다.

- **📢 섹션 요약 비유**: 복잡한 사거리에서 신호등(선점형) 없이도 운전자들이 서로 수신호를 보내며 한 대씩 비켜가는(코루틴), 성숙한 시민 의식을 갖춘 하이테크 교통 시스템과 같습니다.
