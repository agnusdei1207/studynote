# 부모 프로세스 (Parent Process) / 자식 프로세스 (Child Process)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 부모-자식 프로세스 관계는 하나의 프로세스가 시스템 콜을 통해 새로운 프로세스를 생성함으로써 형성되는 **운영체제 내의 계층적 지배-종속 관계**이다.
> 2. **가치**: 부모는 자식의 생성과 종료를 관리(PPID 추적)하며, 자식은 부모의 자원을 물려받아 독립적인 작업을 수행함으로써 **작업의 병렬화와 구조화**를 가능케 한다.
> 3. **융합**: 시스템 전체적으로는 최초의 프로세스(init/systemd)를 뿌리로 하는 거대한 **프로세스 트리(Process Tree)**를 형성하여 일관된 자원 통제와 보안 격리를 실현한다.

+++

## Ⅰ. 부모 및 자식 프로세스의 정의

### 1. 부모 프로세스 (Parent Process)
- 새로운 프로세스를 생성한 주체이다. 자식의 실행 결과를 기다리거나(`wait`), 자식에게 특정 신호(`signal`)를 보낼 권한을 가진다.

### 2. 자식 프로세스 (Child Process)
- 부모에 의해 생성된 새로운 실행 단위이다. 부모의 메모리 문맥과 환경 변수를 복사받아 시작하며, 고유의 PID(Process ID)를 부여받는다.

### 3. 식별자 관계
- **PID (Process ID)**: 자신의 고유 번호.
- **PPID (Parent Process ID)**: 자신을 낳아준 부모의 번호. 모든 자식은 부모의 주소를 기억하고 있다.

### 4. 💡 비유: '창업주와 계열사'
- 부모 프로세스는 **'창업주(본사)'**와 같고, 자식 프로세스는 **'계열사'**와 같다.
- 본사는 계열사를 설립(fork)하고 초기 자본(자원)을 대주지만, 일단 설립된 계열사는 독립적으로 사업(연산)을 진행한다. 계열사가 파산(종료)하면 본사에 보고해야 하며, 본사가 망하면 계열사들의 운명(고아 프로세스화)도 위태로워진다.

+++

## II. 자원 상속과 실행 관계 (Deep Dive)

### 1. 자원 상속 (Inheritance)
- **복사되는 것**: 환경 변수, 열린 파일 디스크립터, 사용자 ID, 그룹 ID, 현재 작업 디렉터리 등.
- **공유되는 것**: 공유 메모리 세그먼트 (물리적 주소 공간은 COW 기법에 의해 논리적으로만 공유).

### 2. 실행 시나리오
- **병행 실행**: 부모와 자식이 서로 간섭하지 않고 각자 갈 길을 간다.
- **동기 실행**: 자식이 일을 마칠 때까지 부모가 `wait()` 상태로 기다린다.

### 3. ASCII 구조 다이어그램 (Process Tree)
```text
    [ systemd (PID: 1) ]  <── Root
             │
    ┌────────┴────────┬────────┐
    │                 │        │
 [ sshd ]          [ bash ]  [ getty ]
                      │
             ┌────────┴────────┐
             │                 │
          [ vim ]           [ gcc ]  <── Child of bash
                               │
                        [ cc1 / as / ld ] <── Grandchildren of bash
```

- **📢 섹션 요약 비유**: 가계도(Family Tree)처럼 선조 프로세스부터 자손 프로세스까지 핏줄(PPID)로 연결되어, 시스템의 질서와 계보를 유지하는 구조입니다.

+++

## Ⅲ. 상태 변화에 따른 특수 관계

부모와 자식 간의 소통이 원활하지 않을 때 발생하는 비정상 상태들이다.

| 유형 | 상황 | 결과 |
|:---|:---|:---|
| **좀비 프로세스** | 자식은 종료됐으나 부모가 확인(`wait`) 안 함 | 프로세스 테이블 점유 (자원 낭비) |
| **고아 프로세스** | 부모가 자식보다 먼저 종료됨 | `init`(PID 1)이 입양하여 처리 |
| **연쇄 종료** | 부모 종료 시 모든 자식을 강제 종료함 | 일부 OS 및 쉘 스크립트 정책 |

- **📢 섹션 요약 비조**: "부모는 자식의 끝(종료)을 책임져야 하며, 부모가 사라진 자식은 사회적 안전망(init)에 의해 보호받아야 시스템이 건강하게 유지됩니다."

+++

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 다중 프로세스 서버 (Prefork Model)
- **상황**: 동시 접속자가 많은 웹 서버를 안정적으로 운영해야 함.
- **판단**: 메인 프로세스(부모)가 미리 여러 개의 자식 프로세스를 생성(`fork`)해두는 **Prefork 방식**을 사용한다. 하나의 자식이 에러로 죽어도 다른 자식들이나 부모에게는 영향이 없으므로, 멀티스레드 방식보다 **격리성(Isolation)** 측면에서 우수한 안정성을 확보할 수 있다.

### 2. 기술사적 인사이트: 프로세스 복제의 보안 위험
- 기술사라면 부모-자식 관계에서의 **보안 상속** 문제를 지적해야 한다. 부모가 가진 민감한 권한이나 암호화 키가 `fork()` 시 자식에게 고스란히 전달될 수 있다. 이를 막기 위해 `FD_CLOEXEC` 플래그를 설정하여 `exec()` 호출 시 자원이 상속되지 않게 하거나, 자식 생성 직후 권한을 최소화(Privilege Dropping)하는 설계 역량이 중요함을 강조해야 한다.

- **📢 섹션 요약 비조**: "자식은 부모의 좋은 점뿐만 아니라 나쁜 습관(보안 취약점)까지 물려받으므로, 독립(exec) 시점에 철저한 자기 정화 과정이 필요합니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **모듈화된 제어**: 대규모 작업을 작은 프로세스 단위로 쪼개어 관리 용이성 증대.
- **장애 확산 방지**: 프로세스 단위의 메모리 장벽을 통한 시스템 전체 보호.

### 2. 미래 전망
앞으로는 마이크로서비스 아키텍처(MSA)가 하드웨어 레벨로 내려오면서, 부모-자식 관계가 단순히 한 OS 내부를 넘어 **네트워크로 연결된 컨테이너 간의 논리적 계층**으로 확장될 것이다. 하지만 그 뿌리가 되는 '생성자의 책임'과 '피생성자의 독립성'이라는 아키텍처 원칙은 변하지 않을 것이다. 부모와 자식 프로세스는 운영체제라는 거대한 질서를 지탱하는 가장 인간적인(?) 메커니즘이다.

- **📢 섹션 요약 비유**: 나무 기둥(부모)에서 뻗어 나온 가지(자식)들이 각자 햇빛(CPU)을 받아 잎을 틔우며, 거대한 숲(시스템)을 이루는 자연의 섭리와 같습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 부모 프로세스는 **'엄마 컴퓨터'**이고, 자식 프로세스는 **'아기 컴퓨터'**예요.
2. 엄마가 "심부름 다녀와!" 하고 아기를 낳으면, 아기는 엄마가 준 가방(자원)을 메고 일을 하러 가죠.
3. 아기가 일을 다 하고 돌아오면 엄마가 "수고했어"라고 말해줘야(wait) 아기가 편히 쉴 수 있답니다!
