# 스택 (Stack) 영역 - 지역 변수, 매개변수, 리턴 주소

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스택 (Stack) 영역은 프로세스 메모리 구조 중 **함수 호출과 관련된 실행 정보**를 임시로 저장하는 공간으로, **LIFO (Last-In-First-Out)** 방식으로 작동하여 함수의 호출과 복귀를 정교하게 관리한다.
> 2. **가치**: 지역 변수와 매개변수의 생명 주기를 함수 실행 기간으로 한정하여 메모리를 효율적으로 재사용하며, 리턴 주소를 저장함으로써 복잡한 중첩 호출이나 재귀 호출 후에도 정확한 복귀 지점을 찾아가게 한다.
> 3. **융합**: CPU의 스택 포인터(SP) 레지스터와 컴파일러의 코드 생성 로직이 결합하여 물리적 메모리를 동적인 함수 실행 흐름에 최적화된 논리적 구조(Stack Frame)로 변환한다.

+++

## Ⅰ. 스택 (Stack) 영역의 개요

### 1. 정의
- 프로그램이 자동으로 사용하는 임시 메모리 영역으로, 함수가 호출될 때 할당되고 함수가 종료될 때 자동으로 반환된다.

### 2. 특징: '자동과 고속'
- **관리 주체**: 운영체제와 컴파일러가 자동으로 관리한다. 개발자가 직접 해제할 필요가 없다.
- **성장 방향**: 일반적으로 높은 주소에서 낮은 주소 방향으로 커진다 (힙과 반대).
- **데이터 저장 단위**: **스택 프레임 (Stack Frame)**이라는 단위로 함수별 정보가 쌓인다.

### 3. 💡 비유: '식당의 접시 쌓기'
- 스택은 **'식당 주방에 쌓아놓은 접시 더미'**와 같다.
- 새로운 일을 시작할 때(함수 호출) 접시 하나를 맨 위에 올려놓고(Push), 그 위에 해야 할 메모(지역 변수)를 적는다. 일이 끝나면 맨 위의 접시를 치워서(Pop) 바로 아래에 있던 이전 상태로 돌아가는 구조이다.

+++

## Ⅱ. 스택 프레임 (Stack Frame)의 내부 구조 (Deep Dive)

함수가 호출될 때마다 스택 영역에 생성되는 논리적 블록이다.

### 1. 주요 구성 요소
- **매개변수 (Parameters)**: 호출된 함수로 전달되는 인자값.
- **리턴 주소 (Return Address)**: 함수 종료 후 돌아가야 할 코드의 위치.
- **이전 프레임 포인터 (Base Pointer)**: 이전 함수의 스택 시작 주소를 가리켜, 복귀 시 스택을 원상복구 하는 기준점이 됨.
- **지역 변수 (Local Variables)**: 함수 내부에서만 선언되고 사용되는 변수.

### 2. 동작 과정
1. **Call**: 함수 호출 시 리턴 주소와 매개변수를 스택에 넣고(Push), 새로운 스택 프레임을 생성한다.
2. **Execute**: 함수 내부 로직을 수행하며 지역 변수를 스택에 쌓는다.
3. **Return**: 함수 종료 시 리턴 주소를 읽어 복귀하고, 현재 스택 프레임을 통째로 날려(Pop) 메모리를 즉시 비운다.

### 3. ASCII 구조 다이어그램
```text
    [ High Address ]
    ┌──────────────────────────┐
    │  main() Stack Frame      │
    ├──────────────────────────┤
    │  func_A() Parameters     │ ◀── 매개변수 전달
    │  func_A() Return Addr    │ ◀── 돌아갈 길 저장
    │  func_A() Base Pointer   │ ◀── 기준점 저장
    │  func_A() Local Vars     │ ◀── 지역 변수 공간
    ├──────────────────────────┤ ◀── Stack Pointer (SP)
    │          ▼               │
    │    (Free Space)          │ ◀── 아래로 확장 (성장)
    └──────────────────────────┘
    [ Low Address ]
```

- **📢 섹션 요약 비유**: 헨젤과 그레텔이 숲속으로 들어갈 때 길을 잃지 않으려고 조약돌(리턴 주소)을 떨어뜨리며 가고, 나올 때는 그 돌을 하나씩 주우며(Pop) 집으로 돌아오는 것과 같습니다.

## Ⅲ. Stack vs Heap의 결정적 차이

| 비교 항목 | Stack 영역 | Heap 영역 |
|:---|:---|:---|
| **할당 속도** | **매우 빠름** (포인터 이동만 수행) | 상대적 느림 (공간 탐색 필요) |
| **관리 방식** | 자동 (함수 단위) | 수동 (개발자 직접 관리) |
| **메모리 크기** | 고정적/제한적 (보통 수 MB) | 유연함/광대함 (가상 메모리 전체) |
| **위험 요소** | **Stack Overflow** (너무 깊은 호출) | Memory Leak (해제 누락) |

- **📢 섹션 요약 비유**: 주머니에서 바로 꺼내 쓰는 손수건(Stack)과, 창고 대장을 확인하고 가져와야 하는 무거운 짐(Heap)의 속도 차이입니다.

## Ⅳ. 실무 적용 및 보안 인사이트 (Strategy)

### 1. 실무 시나리오: 재귀 함수(Recursion)의 위험성
- **상황**: 팩토리얼 계산이나 트리 탐색을 재귀 함수로 짰는데, 큰 데이터가 들어오면 프로그램이 뻗어버림.
- **판단**: 각 호출마다 스택 프레임이 계속 쌓이다가 운영체제가 허용한 **스택 크기(Stack Limit)**를 초과하여 **Stack Overflow**가 발생한 것이다. 이를 해결하기 위해 재귀 대신 반복문을 쓰거나, 스택 프레임 생성을 최소화하는 '꼬리 재귀 최적화(Tail Call Optimization)'를 적용해야 한다.

### 2. 기술사적 인사이트: 스택 버퍼 오버플로우 (Stack Smashing)
- 해커는 스택의 지역 변수 영역에 할당된 크기보다 더 큰 데이터를 밀어 넣어, 바로 위에 있는 **리턴 주소(Return Address)**를 자신의 악성 코드 주소로 덮어씌우는 공격을 시도한다. 이를 막기 위해 현대 OS는 스택 프레임 사이에 '카나리(Canary)'라는 특수 값을 심어 변조 여부를 감시하거나, 스택 실행 권한을 뺏는 보안 기법을 적용한다.

- **📢 섹션 요약 비유**: 우편함(지역 변수)에 쓰레기를 꽉 채워 넣어, 우편함 뚜껑 뒤에 적힌 집 주소(리턴 주소)를 다른 곳으로 고쳐버리는 도둑의 수법을 막기 위해 잠금 장치를 강화하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **실행 컨텍스트 보존**: 복잡한 호출 구조에서도 실행 흐름의 무결성 보장.
- **메모리 자정 능력**: 별도의 청소 과정 없이도 함수 종료 시 즉시 자원 회수.

### 2. 미래 전망
최근에는 멀티스레드 환경의 안정성을 위해 스택 간의 간격을 벌리는 **Stack Guard Page** 기술이 더욱 정교해지고 있다. 또한, Rust와 같이 소유권(Ownership) 개념을 가진 언어들은 스택 데이터의 유효 범위를 컴파일 타임에 더욱 엄격히 제한하여 메모리 안전성을 높이고 있다. 운영체제의 스택 영역은 프로그램의 '질서'를 유지하는 가장 정적인 동시에 역동적인 무대이다.

- **📢 섹션 요약 비유**: 복잡하게 얽힌 실타래(함수 호출)를 엉키지 않게 한 가닥씩 풀어내어 다시 감는, 운영체제의 가장 완벽한 정리 정돈 기술입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **프로세스 메모리 구조**: 스택이 포함된 논리 주소 공간.
- **문맥 교환 (Context Switch)**: 프로세스 전환 시 스택 정보를 포함한 컨텍스트 저장.
- **시스템 호출 (System Call)**: 시스템 콜 처리 시 사용되는 커널 스택(Kernel Stack).
