# 멀티프로세스 아키텍처 (크롬 브라우저 등)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 멀티프로세스 아키텍처는 하나의 거대한 애플리케이션을 기능별로 쪼개어 **각각 독립된 프로세스에서 실행**함으로써, 자원 공유의 위험을 줄이고 시스템 전체의 안정성을 높이는 설계 방식이다.
> 2. **가치**: 특정 컴포넌트(예: 브라우저 탭)에서 오류가 발생해도 다른 프로세스에 영향을 주지 않는 **결함 격리(Fault Isolation)**가 가능하며, 강력한 보안 샌드박스를 적용하기에 용이하다.
> 3. **융합**: 멀티스레드 아키텍처의 낮은 오버헤드 대신 '안정성'과 '보안'을 최우선으로 선택한 현대적 웹 브라우저 및 고신뢰 시스템의 표준 모델이다.

---

## Ⅰ. 멀티프로세스 아키텍처의 정의 및 배경

### 1. 정의
- 프로그램의 주요 기능들(UI, 렌더링, 네트워크 등)을 별개의 프로세스로 분리하여 운영하는 구조이다.

### 2. 등장 배경: '모놀리식의 위기'
- 과거의 웹 브라우저는 하나의 프로세스 안에서 수많은 탭을 스레드로 관리했다. 하지만 하나의 탭에서 무거운 스크립트가 실행되거나 크래시가 발생하면 브라우저 전체가 죽어버리는 문제가 빈번했다. 이를 해결하기 위해 **탭 단위 프로세스 분리**라는 혁신적인 아키텍처가 등장했다.

### 3. 💡 비유: '개별 칸막이가 있는 사무실'
- 멀티프로세스 아키텍처는 **'모든 직원이 칸막이가 있는 개별 사무실(프로세스)에서 일하는 것'**과 같다.
- 한 직원의 사무실에 불이 나거나 큰 사고(크래시)가 나도, 옆 사무실 직원은 전혀 영향을 받지 않고 계속 일을 할 수 있다.

---

## II. 구글 크롬(Chrome)의 멀티프로세스 사례 (Deep Dive)

크롬은 이 아키텍처를 가장 성공적으로 대중화시킨 사례이다.

### 1. 프로세스 구성
- **브라우저 프로세스 (Browser Process)**: UI, 주소창, 북마크 등을 관리하는 총괄 부모 프로세스.
- **렌더러 프로세스 (Renderer Process)**: 각 탭(Tab)마다 할당되어 웹 페이지의 HTML/CSS/JS를 렌더링함.
- **플러그인 프로세스 (Plugin Process)**: Flash나 PDF 뷰어 등 외부 플러그인 전담.
- **GPU 프로세스**: 화면 그래픽 처리를 전담.

### 2. 기술적 장점: 샌드박싱 (Sandboxing)
- 렌더러 프로세스는 네트워크나 파일 시스템에 직접 접근할 권한이 없다. 오직 브라우저 프로세스를 통해서만 자원을 요청할 수 있는데, 이를 통해 악성 스크립트가 로컬 파일을 훔쳐보는 등의 보안 위협을 원천 차단한다.

### 3. ASCII 구조 다이어그램
```text
    [ Browser Process (Main) ] ◀────── IPC ──────▶ [ GPU Process ]
             │
    ┌────────┴────────┬────────┐
    ▼ (IPC)           ▼ (IPC)  ▼ (IPC)
 [ Tab 1 Process ] [ Tab 2 Process ] [ Extension Process ]
 (Renderer)        (Renderer)        (Utility)
```

- **📢 섹션 요약 비유**: 대통령(브라우저 프로세스)이 각 부처 장관(탭 프로세스)들에게 업무를 맡기고, 장관들끼리는 서로의 집무실에 허락 없이 들어가지 못하게 하여 국가 전체의 보안과 질서를 유지하는 것과 같습니다.

## Ⅲ. 멀티프로세스 vs 멀티스레드 비교

| 비교 항목 | 멀티스레드 (Multi-threaded) | 멀티프로세스 (Multi-process) |
|:---|:---|:---|
| **자원 공유** | 매우 쉬움 (메모리 공유) | **어려움 (IPC 필요)** |
| **안정성** | 낮음 (하나 죽으면 전체 사망) | **매우 높음 (결함 격리)** |
| **메모리 소모** | 적음 | **많음 (중복 리소스 로딩)** |
| **보안** | 취약 (주소 공간 공유) | **우수 (프로세스 샌드박싱)** |

- **📢 섹션 요약 비조**: "효율성을 원한다면 스레드를, 목숨(안정성)이 중요하다면 프로세스를 선택하는 전략적 판단이 필요합니다."

---

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 대규모 동시 요청 처리 서버
- **상황**: 결제 엔진처럼 에러가 발생했을 때 파급력이 큰 미션 크리티컬 시스템 개발.
- **판단**: 멀티스레드 대신 **멀티프로세스(Pre-fork) 아키텍처**를 선택한다. 메모리를 조금 더 쓰더라도 특정 사용자의 요청이 프로세스 크래시를 유발했을 때 다른 사용자의 세션이 끊기지 않도록 보장하는 것이 신뢰성 측면에서 훨씬 가치 있기 때문이다.

### 2. 기술사적 인사이트: 사이트 격리 (Site Isolation)
- 기술사라면 현대 브라우저 보안의 정수인 **사이트 격리**를 논해야 한다. 단순히 탭 단위가 아니라 출처(Domain)가 다른 웹 사이트들을 물리적으로 다른 프로세스에 배치함으로써, 최근의 CPU 취약점(Spectre 등)을 이용한 프로세스 간 데이터 탈취를 방지하는 최신 운영체제 보안 트렌드를 이해해야 한다.

- **📢 섹션 요약 비조**: "성능을 대가로 안전을 사는 것은 아키텍처의 후퇴가 아니라, 복잡해지는 현대 보안 환경에 대응하는 필연적인 진화입니다."

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **가용성 극대화**: 특정 작업의 오류가 애플리케이션 전체의 '먹통'으로 이어지는 현상 제거.
- **보안성 강화**: 프로세스 경계를 넘지 못하는 물리적 장벽 구축.

### 2. 미래 전망
앞으로는 하드웨어 자원이 더욱 풍족해짐에 따라, **웹 어셈블리(WebAssembly)**나 **마이크로 VM** 기술과 결합하여 더욱 잘게 쪼개진 멀티프로세스 아키텍처가 주류가 될 것이다. 또한, 운영체제 레벨에서 프로세스 생성 비용을 획기적으로 낮추는 기술이 발전하면서 멀티프로세스의 단점인 메모리 소모 문제도 완화될 것이다. 멀티프로세스 아키텍처는 '분할과 정복'을 통해 시스템의 견고함을 증명하는 최고의 설계 철학이다.

- **📢 섹션 요약 비유**: 배 전체가 물바다가 되지 않도록 배 밑바닥을 여러 개의 격실로 나누어 설계한 타이타닉의 교훈(하지만 실제로 작동하는!)과 같은 안전 설계입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 멀티프로세스 아키텍처는 여러 친구가 한 방에서 노는 대신, **'각자 자기 방에서 따로 노는 것'**과 같아요.
2. 한 친구가 장난감을 부수거나 울어도(에러), 내 방에 있는 나는 영향을 받지 않고 계속 재미있게 놀 수 있죠.
3. 방을 여러 개 써서 집이 좀 좁아질 수는 있지만, 가장 안전하고 평화롭게 노는 방법이랍니다!
