# 프로세스 제어 블록 (PCB, Process Control Block) / 태스크 제어 블록 (TCB)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PCB (Process Control Block)는 운영체제가 각 프로세스를 관리하기 위해 필요한 모든 정보를 담고 있는 **커널 내의 데이터 구조(장부)**이며, 리눅스 커널에서는 `task_struct`로 구현된다.
> 2. **가치**: 프로세스의 현재 실행 상태(Context)를 저장함으로써 시분할 시스템에서 CPU를 뺏겼다가 다시 할당받을 때 **중단된 지점부터 정확히 재개**할 수 있게 하는 핵심 기반이 된다.
> 3. **융합**: 프로세스 생성 시 커널 스택 근처에 할당되어 프로세스 종료 시까지 유지되며, CPU 스케줄링, 메모리 관리, 입출력 상태 관리 등 모든 OS 관리 로직의 판단 근거가 된다.

---

## Ⅰ. PCB / TCB의 개요

### 1. 정의
- **PCB**: 프로세스 하나에 대해 하나씩 생성되는 고유한 관리 정보 블록이다.
- **TCB (Task Control Block)**: 리눅스 등 일부 운영체제에서는 프로세스와 스레드를 통일된 'Task'로 보기에 PCB 대신 TCB라는 용어를 사용하기도 한다.

### 2. 왜 필요한가? (The Context)
- CPU는 한 번에 하나의 일만 할 수 있다. 수많은 프로세스를 동시에 돌리는 것처럼 보이게 하려면 CPU가 프로세스를 빠르게 갈아타야 하는데, 이때 **"어디까지 일했는지"** 적어둘 장부가 없으면 작업 연속성이 깨지기 때문이다.

### 3. 💡 비유: '은행 창구의 고객 전표'
- PCB는 **'은행에서 업무를 보던 고객이 잠시 전화를 받으러 나갈 때 남겨두는 업무 전표'**와 같다.
- 전표에는 고객 번호(PID), 보던 업무(PC), 현재 잔액(Registers) 등이 적혀 있다. 전화를 마치고 돌아온 고객은 창구 직원에게 전표(PCB)를 보여주어 중단했던 업무를 즉시 이어갈 수 있다.

---

## Ⅱ. PCB의 생명 주기와 위치 (Deep Dive)

### 1. 생성과 소멸
- **생성**: 프로세스가 생성되는 시점(New 상태)에 커널에 의해 메모리 공간(커널 메모리)에 할당된다.
- **소멸**: 프로세스가 종료(Terminated)되어 운영체제가 자원을 완전히 회수할 때 삭제된다.

### 2. 저장 위치: 커널 스택 (Kernel Stack)
- 프로세스마다 고유의 커널 스택이 할당되며, 그 하단에 PCB(task_struct)를 가리키는 포인터나 정보가 위치한다. 이는 사용자 공간(User Space)이 아닌 보안이 유지되는 **커널 공간 (Kernel Space)**에 저장되어 응용 프로그램이 직접 수정할 수 없다.

### 3. ASCII 구조 다이어그램
```text
    [ Process Memory ]          [ Kernel Memory ]
    ┌────────────────┐          ┌──────────────────────────────────┐
    │  User Stack    │          │  Kernel Stack (Process 1)        │
    ├────────────────┤          ├──────────────────────────────────┤
    │  Heap          │          │  PCB (task_struct)               │
    ├────────────────┤          │  - PID: 1234                     │
    │  Data          │          │  - State: Running                │
    ├────────────────┤          │  - PC: 0x00401234                │
    │  Text (Code)   │          │  - Memory Ptr: 0x...             │
    └────────────────┘          └──────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 학생의 성적표(PCB)를 학생이 직접 고칠 수 없게 교무실(커널 공간) 금고에 보관하는 것과 같습니다.

## Ⅲ. PCB의 핵심 구성 요소 요약

상세한 요소는 다음 장(#91)에서 다루며, 여기서는 주요 범주를 파악한다.

| 범주 | 포함 정보 | 목적 |
|:---|:---|:---|
| **식별자** | PID, PPID | 프로세스 구분 및 부모-자식 관계 확인 |
| **상태 정보** | Ready, Running, Waiting | 스케줄링 대상 선별 |
| **실행 컨텍스트** | PC, 레지스터 상태 | 작업 재개(Context Restore)용 |
| **자원 할당 정보** | 메모리 맵, 열린 파일 목록 | 프로세스가 소유한 자원 관리 |

- **📢 섹션 요약 비유**: 이력서에 사진(PID), 경력(컨텍스트), 학력(메모리 정보)이 적혀 있어 인사권자(운영체제)가 인력 배치를 효율적으로 할 수 있게 돕는 것과 같습니다.

---

## Ⅳ. 실무 적용 및 OS 전략 (Strategy)

### 1. 실무 시나리오: 커널 메모리 부족 (Kmem Leak)
- **상황**: 시스템에 프로세스 개수가 많지 않은데 커널 메모리가 부족하다고 나옴.
- **판단**: 프로세스가 종료되었음에도 PCB가 적절히 해제되지 않고 남아있는 **좀비(Zombie)** 프로세스가 누적된 경우일 수 있다. PCB는 커널 메모리를 점유하므로, 기술사라면 `wait()` 시스템 콜 호출 여부나 커널 모듈의 자원 해제 로직을 점검해야 한다.

### 2. 기술사적 인사이트: task_struct의 비대화 방지
- 리눅스 커널이 발전함에 따라 `task_struct` 구조체는 수천 라인에 달할 정도로 거대해졌다. 이는 문맥 교환 시 오버헤드 증가로 이어진다. 따라서 현대 OS 튜닝에서는 PCB의 크기를 직접 줄이기보다, **캐시 친화적(Cache-friendly)**인 배치를 통해 PCB 정보 접근 시의 캐시 미스를 최소화하는 것이 고성능 서버 아키텍처의 핵심이다.

- **📢 섹션 요약 비유**: 장부가 너무 두꺼워지면 넘기기 힘들 듯(오버헤드), 자주 보는 핵심 내용(PID, PC)을 앞장에 배치하여 빨리 읽을 수 있게 하는 전략이 필요합니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **멀티태스킹의 실현**: 수많은 프로세스의 상태를 안전하게 격리 보존하여 동시 실행 구현.
- **자원 관리의 정밀도**: 각 프로세스에게 할당된 자원을 1비트의 오차 없이 추적 관리.

### 2. 미래 전망
앞으로는 프로세스가 노드 간에 자유롭게 이동하는 분산 환경에서, PCB의 내용을 네트워크를 통해 고속 전송하는 **Process Migration** 기술이 고도화될 것이다. 또한 보안 강화를 위해 PCB의 민감 정보를 하드웨어적으로 암호화하는 **Enclave 기반 PCB 관리** 기술이 도입되면서, 운영체제는 더욱 견고한 신뢰의 토대를 구축하게 될 것이다.

- **📢 섹션 요약 비유**: 운영체제가 쓰는 모든 소설(프로세스 실행)의 줄거리를 담고 있는 가장 소중한 일기장입니다.
