# 준비 큐 (Ready Queue)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 준비 큐 (Ready Queue)는 실행에 필요한 모든 자원을 확보하고 **CPU 할당(Dispatch)만을 기다리는** 준비(Ready) 상태의 프로세스들을 모아놓은 커널 내부의 대기열 자료구조이다.
> 2. **가치**: 운영체제 스케줄러가 다음에 실행할 프로세스를 빠르고 효율적으로 선택할 수 있는 물리적 기반을 제공하며, 시스템의 부하 상태(Load Average)를 가늠하는 지표가 된다.
> 3. **융합**: 각 프로세스의 PCB(Process Control Block)들이 **연결 리스트 (Linked List)** 형태로 큐에 매달리며, 우선순위 스케줄링 시에는 여러 개의 준비 큐(Multi-level Queue)로 확장되어 관리된다.

+++

## Ⅰ. 준비 큐의 개요

### 1. 정의
- 주 기억장치에 상주하면서 CPU를 할당받아 실행되기를 기다리는 모든 프로세스의 집합체이다.

### 2. 특징: '공정한 순번 대기'
- **자료 구조**: 주로 연결 리스트(Linked List)로 구현된다. 새로운 프로세스는 꼬리(Tail)에 추가되고, 스케줄러는 머리(Head)에서 프로세스를 꺼내 실행한다.
- **관리 대상**: 실제 프로세스 전체가 아닌, 프로세스의 모든 정보를 담고 있는 **PCB의 포인터**를 관리한다.

### 3. 💡 비유: '마트 계산대의 대기 줄'
- 준비 큐는 마트에서 장을 다 보고(자원 확보 완료) **'계산대 앞에서 내 차례를 기다리는 줄'**과 같다.
- 줄에 서 있는 사람(프로세스)은 계산원(CPU)이 부르기만 하면 즉시 계산(연산)을 시작할 수 있는 상태이다.

+++

## Ⅱ. 준비 큐의 동작 메커니즘 (Deep Dive)

### 1. 큐 진입 시점 (Enqueue)
- 프로세스가 처음 생성되어 승인(Admitted)되었을 때.
- 실행 중인 프로세스가 할당 시간 만료(Timeout)로 쫓겨났을 때.
- 대기(Waiting) 상태의 프로세스가 I/O 완료 신호를 받고 깨어났을 때.

### 2. 큐 탈출 시점 (Dequeue)
- CPU 스케줄러에 의해 선택되어 실행(Running) 상태로 전이될 때 (**Dispatch**).

### 3. ASCII 구조 다이어그램
```text
    [ Ready Queue (Linked List of PCBs) ]
    ┌───────────┐      ┌───────────┐      ┌───────────┐
    │ Head Ptr  │ ──▶  │ PCB (P1)  │ ──▶  │ PCB (P2)  │ ──▶ (Tail)
    └───────────┘      │ Next: P2  │      │ Next: NULL│
                       │ State: Rdy│      │ State: Rdy│
                       └─────┬─────┘      └─────┬─────┘
                             │                  │
                             ▼                  ▼
                      (실제 프로세스 P1)   (실제 프로세스 P2)
```

- **📢 섹션 요약 비유**: 각 PCB를 고리로 연결한 긴 기차와 같습니다. 스케줄러는 이 기차의 맨 앞 칸(Head)을 떼어내어 CPU라는 엔진에 연결하는 역할을 합니다.

+++

## Ⅲ. 준비 큐의 발전된 형태 (Advanced Structures)

현대 운영체제는 단순히 하나의 줄만 세우지 않는다.

| 구조 | 설명 | 목적 |
|:---|:---|:---|
| **다단계 큐** | 우선순위별로 별도의 큐를 운영 (High, Normal, Low) | 차등적 서비스 제공 |
| **다단계 피드백 큐** | 프로세스의 행동에 따라 큐 사이를 이동시킴 | I/O 바운드 프로세스 우대 |
| **코어별 준비 큐** | 멀티코어 환경에서 각 CPU 코어마다 전용 큐 배치 | 락 경합(Lock Contention) 방지 |

- **📢 섹션 요약 비유**: 일반 고객 줄과 VIP 고객 줄을 따로 운영(다단계 큐)하여 업무 효율과 고객 만족도를 동시에 높이는 전략과 같습니다.

+++

## Ⅳ. 실무 적용 및 성능 지표 (Strategy)

### 1. 실무 시나리오: Load Average의 해석
- **상황**: 리눅스 서버에서 `top` 명령어를 쳤는데 Load Average가 5.0으로 매우 높음. (CPU 코어는 1개)
- **판단**: 현재 **준비 큐**에 대기 중인 프로세스가 평균 4개(실행 중인 1개 제외)나 된다는 뜻이다. CPU가 처리할 수 있는 양보다 줄이 너무 길게 늘어선 상태이므로, CPU 사양을 높이거나(Scale-up) 부하를 분산(Scale-out)해야 한다.

### 2. 기술사적 인사이트: 큐 관리와 확장성 (Scalability)
- 대규모 멀티코어 시스템에서는 하나의 전역 준비 큐(Global Ready Queue)를 모든 코어가 공유하면 **뮤텍스(Mutex) 락**으로 인해 성능이 급격히 저하된다. 기술사라면 이를 해결하기 위해 각 코어마다 독립된 큐를 두는 **Per-CPU Runqueue** 구조와, 코어 간 부하를 맞추는 **로드 밸런싱(Load Balancing)** 기법을 함께 제시해야 한다.

- **📢 섹션 요약 비유**: 마트의 계산대(CPU 코어)가 10개인데 줄(준비 큐)이 하나뿐이라면 손님들이 서로 밀치며 싸우게 되므로, 계산대마다 줄을 따로 세우는 것이 훨씬 빠릅니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **스케줄링 응답성 향상**: 준비된 프로세스들을 체계적으로 관리하여 즉각적인 CPU 할당 가능.
- **시스템 상태 가시화**: 대기열의 길이를 통해 전체 시스템의 병목 현상 진단.

### 2. 미래 전망
최근 인공지능 워크로드가 증가함에 따라, 단순히 CPU만을 위한 준비 큐가 아닌 GPU, NPU 등 이기종 가속기 자원을 통합적으로 관리하는 **범용 자원 준비 큐** 기술이 중요해지고 있다. 운영체제는 이제 수많은 이기종 실행 유닛들에게 누가 먼저 들어갈지 결정하는 거대한 '교통 관제 센터'의 역할을 수행하게 될 것이다.

- **📢 섹션 요약 비유**: 모두가 질서 있게 줄을 서는 시스템(준비 큐)이 있기에, 컴퓨터라는 복잡한 사회가 사고 없이 돌아갈 수 있는 것입니다.
