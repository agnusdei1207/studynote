## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Bell-LaPadula 모델(1973)은 **"기밀성(Confidentiality) 보장을 위한 다중 levels 보안(Multi-Level Security, MLS)"**의 수학적 모델로서, **"No Read Up, No Write Down"** 두 가지 핵심 규칙으로 정보의 상향/하향 유출을 방지한다.
> 2. **가치**: 이 모델은 미 국방부의 **"기밀 문서가 일반 문서로 유출되는 것을 방지"**하기 위해 설계되었다. Top Secret(1등급) 등급 사용자는 Secret(2등급) 문서를 읽을 수 있지만, 반대는 불가능하다.
> 3. **융합**: SELinux, Trusted Solaris, SGX/TrustZone 등 현대 보안 시스템은 Bell-LaPadula의 **"레이블 기반 접근 제어"**를 구현하며, 클라우드의 멀티테넌트 격리, 컨테이너 보안에도 응용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: Bell-LaPadula 모델은 David Bell와 Leonard LaPadula이 1973년에 발표한 **"기밀성(Confidentiality) 보장을 위한 상태 머신(State Machine) 모델"**이다. 시스템의 모든 주체(Subject)와 객체(Object)에 **보안 등급(Security Level)**을 부여하고, **"No Read Up, No Write Down"** 두 가지 규칙으로 접근을 제어한다. 수학적으로는 부분 순서 집합(Partially Ordered Set)인 **격자(Lattice)** 구조를 사용하여, 보안 등급 간의 상하 관계를 정의한다.

- **💡 비유**: Bell-LaPadula 모델은 **"회사의 기밀 등급과 문서 분류 규정"**과 같다. 회사에는 **"1등급(경영진 전용), 2등급(임원), 3등급(직원), 4등급(공개)"** 보안 등급이 있고, 각 문서도 등급별로 분류된다. **"3등급 직원은 4등급, 3등급 문서를 읽을 수 있지만, 2등급, 1등급 문서는 읽을 수 없다(No Read Up)"**. 또한 **"1등급 경영진은 1등급 문서에만 쓸 수 있고, 3등급 문서에 쓸 수 없다(No Write Down)"**. 이렇게 하면 기밀이 상향 유출(직원이 경영진 문서를 탈취)되거나 하향 유출(경영진이 기밀을 일반 문서에 숨겨서 공개)되는 것을 방지할 수 있다.

- **등장 배경**:
  1. **미 국방부의 MLS 요구(1960s-70s)**: 냉전 시대, 미 국방부는 **"기밀 정보가 소련으로 유출되는 것을 방지"**하는 컴퓨터 시스템을 필요로 했다. 베트남전 기밀 문서 유출 사건 등이 발생하자, 이를 방지할 기술적 솔루션이 절실했다.
  2. **AES(Air Force Electronic System) 사업(1970s)**: 미 공군의 자금 지원으로 MITRE 연구소에서 다중 levels 보안 시스템을 개발했고, Bell-LaPadula는 이론적 기반이 되었다.
  3. **TCSEC(Orange Book) 표준화(1985)**: DoD의 Trusted Computer System Evaluation Criteria(TCSEC) **B2 등급 이상**은 Bell-LaPadula 모델을 구현해야 한다고 명시하여, Trusted UNIX, Trusted Solaris 등 상용 제품으로 이어졌다.

Bell-LaPadula 모델의 격자(Lattice) 구조를 시각화하면 다음과 같다. 보안 등급은 Top Secret(TS) > Secret(S) > Confidential(C) > Unclassified(U) 순서로 상하 관계가 있다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │           Bell-LaPadula 모델 — 보안 등급 격자 구조                     │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [격자(Lattice) — 부분 순서 집합]                                    │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │           Top Secret (TS)                                     │  │
 │  │              │  ──────────────────────────────────────────┐      │  │
 │  │              │  상위 등급(Higher Level)                        │  │
 │  │              ▼                                                 │  │
 │  │           Secret (S)                                          │  │
 │  │              │  ──────────────────────────────────────────┐      │  │
 │  │              │                                                 │  │
 │  │              ▼                                                 │  │
 │  │           Confidential (C)                                     │  │
 │  │              │  ──────────────────────────────────────────┐      │  │
 │  │              │  하위 등급(Lower Level)                         │  │
 │  │              ▼                                                 │  │
 │  │           Unclassified (U)                                    │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [핵심 규칙]                                                         │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ① No Read Up (Simple Security Property)                     │  │
 │  │     • 주체 S는 자신의 등급보다 높은 등급의 객체를 읽을 수 없음│  │
 │  │     • 수학적 표현: L(S) ≥ L(O)이어야 read 허용               │  │
 │  │                                                                │  │
 │  │     예: Secret 등급 사용자는 Top Secret 문서를 읽을 수 없음   │  │
 │  │         (정보 상향 유출 방지)                                  │  │
 │  │                                                                │  │
 │  │  ② No Write Down (*-Property)                                 │  │
 │  │     • 주체 S는 자신의 등급보다 낮은 등급의 객체에 쓸 수 없음 │  │
 │  │     • 수학적 표현: L(S) ≤ L(O)이어야 write 허용              │  │
 │  │                                                                │  │
 │  │     예: Top Secret 등급 사용자는 Unclassified 파일에 쓸 수 없음│  │
 │  │         (정보 하향 유출 방지)                                  │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [허용 가능한 정보 흐름]                                              │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │   TS ────read──▶ TS    (같은 등급 간 읽기 허용)               │  │
 │  │   │                                                              │  │
 │  │   └──write────▶ TS    (같은 등급 간 쓰기 허용)               │  │
 │  │                                                                │  │
 │  │   TS ────read──▶ S     (상위 → 하위 읽기 허용)                │  │
 │  │   │                                                              │  │
 │  │   S  ────read──▶ C     (상위 → 하위 읽기 허용)                │  │
 │  │   │                                                              │  │
 │  │   C  ────read──▶ U     (상위 → 하위 읽기 허용)                │  │
 │  │                                                                │  │
 │  │   U  ───write──▶ C     (하위 → 상위 쓰기 허용)                │  │
 │  │   │                                                              │  │
 │  │   C  ───write──▶ S     (하위 → 상위 쓰기 허용)                │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  ✅ Read: 상위 등급 → 하위 등급 흐름만 허용                     │
 │  ✅ Write: 하위 등급 → 상위 등급 흐름만 허용                     │
 │  ❌ Read: 하위 → 상위 (예: S가 TS를 읽기) = 금지                │
 │  ❌ Write: 상위 → 하위 (예: TS가 U에 쓰기) = 금지               │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Bell-LaPadula 모델의 **"No Read Up"**은 정보의 상향 유출(예: Secret 등급 병사가 Top Secret 문서를 읽고 적에게 공개)을 방지한다. **"No Write Down"**은 정보의 하향 유출(예: Top Secret 등급 장교가 기밀 정보를 Unclassified 파일에 숨겨서 적에게 공개)을 방지한다. 이 두 규칙을 조합하면, **"정보는 같은 등급 내에서만 흐르거나, 상위에서 하위로만 흐를 수 있다(Read 전용)"**는 특성이 생겨, 기밀성이 보장된다. 그러나 이 모델은 무결성을 보장하지 못한다(예: Secret 사용자가 Top Secret 문서를 수정하여 무결성 파괴).

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **주체(Subject)** | 정보를 접근하는 능동 개체 | 사용자, 프로세스 | UID, 프로세스 ID | 보유자 |
| **객체(Object)** | 접근의 대상인 수동 개체 | 파일, 메모리, 장치 | inode, 메모리 페이지 | 문서 |
| **보안 등급(Security Level)** | 주체/객체의 기밀 등급 | Classification + Category | TS, S, C, U | 보안 등급 마크 |
| **격자(Lattice)** | 등급 간 상하 관계 정의 | 부분 순서 집합(Partial Order) | Poset(L, ≤) | 조직도 |
| **상태(State)** | 시스템의 현재 보안 상태 | (S, O, L)의 집합 | 시스템 상태 벡터 | 현재 상황 |
| **전이(Transition)** | 상태 변화 규칙 | read/write 명령 | 시스템 콜 | 상태 변화 |

### 수학적 정의와 시스템 모델

Bell-LaPadula 모델은 **상태 머신(State Machine)**으로 정의된다. 시스템의 상태는 **(S, O, L)**의 세 순서쌍으로 표현되며, 여기서 S는 주체 집합, O는 객체 집합, L은 보안 등급 함수다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │           Bell-LaPadula 모델 — 수학적 정의와 시스템 모델               │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [시스템 구성 요소]                                                  │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  S = {s₁, s₂, ...}  ← 주체(Subject) 집합                   │  │
 │  │  O = {o₁, o₂, ...}  ← 객체(Object) 집합                     │  │
 │  │  L = {l₁, l₂, ...}  ← 보안 등급(Security Level) 집합         │  │
 │  │                                                                │  │
 │  │  부분 순서: ≤ 관계 (예: U ≤ C ≤ S ≤ TS)                      │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [상태(State) 정의]                                                 │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  상태 Q = (S, O, L, F)                                         │  │
 │  │    • S: 현재 활성 주체 집합                                   │  │
 │  │    • O: 현재 접근 가능 객체 집합                               │  │
 │  │    • L: 보안 등급 함수                                         │  │
 │  │      - f_S(s): S → L  (주체 s의 등급)                       │  │
 │  │      - f_O(o): O → L  (객체 o의 등급)                       │  │
 │  │    • F: 현재 접근 권한 행렬                                  │  │
 │  │      - F(s, o) = {read, write, ...} 허용된 연산               │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [보안 규칙(Security Rules)]                                        │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  ① Simple Security Property (ss-Property)                     │  │
 │  │     ∀s ∈ S, ∀o ∈ O:                                          │  │
 │  │       if read ∈ F(s, o) then f_S(s) ≥ f_O(o)                 │  │
 │  │     (읽기 권한이 있으면, 주체 등급 ≥ 객체 등급)              │  │
 │  │                                                                │  │
 │  │  ② *-Property (star Property)                                 │  │
 │  │     ∀s ∈ S, ∀o ∈ O:                                          │  │
 │  │       if write ∈ F(s, o) then f_S(s) ≤ f_O(o)                │  │
 │  │     (쓰기 권한이 있으면, 주체 등급 ≤ 객체 등급)              │  │
 │  │                                                                │  │
 │  │  → 두 규칙을 조합하면:                                         │  │
 │  │     같은 등급끼리만 읽기/쓰기 허용, 또는                     │  │
 │  │     상위 등급 주체가 하위 등급 객체를 읽기만 허용          │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [시스템 전이(Transition) 규칙]                                      │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  시스템은 다음 상태로 전이:                                   │  │
 │  │  Q = (S, O, L, F)  ───operation──▶  Q' = (S, O, L, F')        │  │
 │  │                                                                │  │
 │  │  요청: (s, o, read) 또는 (s, o, write)                        │  │
 │  │                                                                │  │
 │  │  ① read 요청:                                                │  │
 │  │     if f_S(s) ≥ f_O(o) then F' = F ∪ {(s, o, read)}         │  │
 │  │     else 거부(No Read Up 위반)                               │  │
 │  │                                                                │  │
 │  │  ② write 요청:                                               │  │
 │  │     if f_S(s) ≤ f_O(o) then F' = F ∪ {(s, o, write)}        │  │
 │  │     else 거부(No Write Down 위반)                            │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Bell-LaPadula 모델은 **"현재 상태에서 보안 규칙을 위반하지 않는 다음 상태로만 전이"**하는 시스템을 정의한다. 예를 들어 Secret 등급 사용자 `s_secret`가 Top Secret 등급 파일 `o_TS`를 읽기(read) 요청하면, `f_S(s_secret) = S < TS = f_O(o_TS)`이므로 **No Read Up 위반**으로 거부된다. 반대로 Top Secret 등급 사용자 `s_TS`가 Unclassified 파일 `o_U`에 쓰기(write) 요청하면, `f_S(s_TS) = TS > U = f_O(o_U)`이므로 **No Write Down 위반**으로 거부된다.

### Bell-LaPadula 모델의 한계와 해결책

Bell-LaPadula 모델은 기밀성을 보장하지만, **무결성(Integrity)을 보장하지 못하며**, **"코펑 마루쌓기 문제"**와 같은 실용적 문제가 있다.

```text
 ┌──────────────────────────────────────────────────────────────────────┐
 │            Bell-LaPadula 모델의 한계와 해결책                        │
 ├──────────────────────────────────────────────────────────────────────┤
 │                                                                      │
 │  [한계 1: 무결성 미보장]                                             │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  문제: Secret 등급 사용자는 Top Secret 문서를 읽을 수 없지만, │  │
 │  │        쓸 수는 없음 확인 안 됨 → Top Secret 문서를 수정 가능  │  │
 │  │                                                                │  │
 │  │  예: Secret 등급 해커가 Top Secret 문서의 내용을 모르더라도 │  │
 │  │      "rm /path/to/TS_doc"로 파일 삭제 가능(무결성 파괴)      │  │
 │  │                                                                │  │
 │  │  해결책: Biba 모델(무결성)와 결합                             │  │
 │  │         Bell-LaPadula(기밀성) + Biba(무결성)                   │  │
 │  │         → "같은 등급 내에서만 읽기/쓰기 허용"                    │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [한계 2: 코펑 마루쌓기(Covert Channel) 문제]                        │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  문제: Top Secret 등급 사용자가 기밀 정보를                   │  │
 │  │        공용 자원(프린터, 디스크)의 속성에 숨겨서           │  │
 │  │        Unclassified 사용자가 탈취 가능                         │  │
 │  │                                                                │  │
 │  │  예: TS 등급 사용자 A가 프린터 큐의 파일명에                 │  │
 │  │      "/tmp/SECRET_KEY_IS_12345.txt"라고 쓰면,                 │  │
 │  │      U 등급 사용자 B는 파일명(Read 가능)을 보고 키를 탈취    │  │
 │  │                                                                │  │
 │  │  해결책: Storage Covert Channel 예방                          │  │
 │  │         • 공유 자원의 메타데이터 접근 제어                     │  │
 │  │         • Polyinstantiation(다중 인스턴스)로 자원 분리       │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 │  [한계 3: Category(Compartment) 문제]                               │
 │  ┌────────────────────────────────────────────────────────────────┐  │
 │  │                                                                │  │
 │  │  문제: 같은 등급이라도 서로 다른 프로젝트 기밀이              │  │
 │  │        섞이는 것을 방지하지 못함                               │  │
 │  │                                                                │  │
 │  │  예: Project Alpha와 Project Beta는 모두 Secret 등급           │  │
 │  │      하지만 서로의 기밀을 공유하면 안 됨                       │  │
 │  │                                                                │  │
 │  │  해결책: Category(Compartment) 추가                              │  │
 │  │         보안 등급을 (Level, {Category}) 형태로 확장          │  │
 │  │         예: L = (Secret, {Alpha}) ≠ (Secret, {Beta})           │  │
 │  │                                                                │  │
 │  └────────────────────────────────────────────────────────────────┘  │
 │                                                                      │
 └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Bell-LaPadula 모델의 가장 큰 한계는 **"무결성을 보장하지 못한다"**는 것이다. Secret 등급 사용자는 Top Secret 문서의 내용을 읽을 수 없지만(No Read Up), 파일을 삭제하거나 수정하는 것을 막을 수는 없다(쓰기 규칙은 **"상위 등급은 하위에 쓸 수 없다"**고만 되어 있어서, 같은 등급이거나 상위 등급은 쓰기 가능). 이를 해결하기 위해 Biba 모델(무결성)을 동시에 적용하여, **"같은 등급 내에서만 읽기/쓰기 허용"**으로 강력하게 제한한다. 또한 **"코펑 마루쌓기(Covert Channel)"** 문제를 해결하기 위해, 공유 자원(프린터, 공유 메모리)의 메타데이터 접근을 제어하고, **Polyinstantiation**(다중 인스턴스)으로 등급별로 별도 자원을 할당한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: Bell-LaPadula vs Biba vs Clark-Wilson

| 비교 항목 | Bell-LaPadula | Biba | Clark-Wilson |
|:---|:---|:---|:---|
| **목적** | 기밀성(Confidentiality) | 무결성(Integrity) | 무결성(거래 중심) |
| **핵심 규칙** | No Read Up, No Write Down | No Read Down, No Write Up | Well-formed Transactions |
| **등급 체계** | 상-하 구조(TS>S>C>U) | 신뢰 구조(T>H>M>L) | 역할 분리(SoD) |
| **주 용도** | 군사 MLS 시스템 | 금융, 의료 시스템 | 전자 상거래 |
| **결합 시** | Biba와 결합 → 완전한 격리 | Bell-LaPadula와 결합 | SoD와 결합 |

Bell-LaPadula는 **"정보의 상향 유출(하위가 상위를 읽기)을 방지"**하는 데 초점을 맞추고, Biba는 **"정보의 하향 오염(상위가 하위에 쓰기)을 방지"**하는 데 초점을 맞춘다. 두 모델을 동시에 적용하면 **"같은 등급 내에서만 읽기/쓰기 허용"**이라는 강력한 격리가 실현된다. Clark-Wilson 모델은 거래 중심 시스템을 위해 **"역할 분리(Separation of Duty)"**와 **"검증된 절차(Well-formed Transactions)"**에 초점을 맞춘다.

### 비교 2: MLS(Multi-Level Security) 구현 방식

| 구현 방식 | 특징 | 장점 | 단점 |
|:---|:---|:---|:---|
| **격리형 MLS** | 등급별 별도 서버 | 완전한 격리 | 비용 비싸, 자원 낭비 |
| **매핑형 MLS** | 단일 서버 + 레이블링 | 비용 절감 | 코펑 채널 위험 |
| **암호형 MLS** | 암호화된 파티션 | 레이블링 오버헤드 작음 | 암호화 키 관리 복잡 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — SELinux로 Bell-LaPadula 구현**: SELinux는 **Type Enforcement(TE)**가 주력이지만, MLS 정책을 로드하면 Bell-LaPadula 모델을 구현할 수 있다.
   ① `semanage boolean -m mls -1`로 MLS 활성화
   ② 보안 등급별 카테고리 정의: `semanage user -a -R "staff_r -l s0-s0:c0.c1023" staff_u`
   ③ 파일에 레이블 부착: `chcon -l s2:c0,c10 /confidential/data.txt`
   ④ 사용자와 파일의 레벨을 비교하여 접근 제어

2. **시나리오 — 클라우드 멀티테넌트 격리**: AWS S3 같은 클라우드 스토리는 Bell-LaPadula 스타일의 **"버킷 정책(Bucket Policy)"**으로 테넌트 간 격리를 구현한다.
   ① Tenant A는 `tenant-a-bucket`에만 Full Access
   ② Tenant B는 `tenant-b-bucket`에만 Full Access
   ③ Bucket Policy의 `Principal` 필드에 각 테넌트의 AWS Account ID만 명시
   → **"Tenant A는 Tenant B의 버킷을 열람 불가(No Read Up)"** 효과 달성

3. **시나리오 — 컨테이너 격리**: Docker/Kubernetes는 **"네임스페이스 + SELinux/AppArmor"**로 컨테이너 간 격리를 구현한다.
   ① 각 컨테이너는 별도 PID, Network, Mount 네임스페이스
   ② SELinux MCS(Multi-Category Security)로 각 컨테이너에 고유 카테고리(`c0`, `c1`, ...) 할당
   → 컨테이너 A는 컨테이너 B의 파일을 접근 불가(No Read Up)

### 도입 체크리스트
- **기술적**: SELinux/AppArmor가 enforcing 모드인가?(`sestatus`, `aa-status`) AVC 거부 로그를 정기적으로 점검하는가?
- **운영·보안적**: 보안 등급(Classification)과 카테고리(Category)가 문서화되어 있는가? 정기적으로(연 1회) 재검토하는가?

### 안티패턴
- **과도한 permissive 모드**: SELinux permissive 모드는 MAC 차단을 하지 않으므로, 실제 공격에 대한 방어력이 없다. 초기 테스트 후에는 반드시 enforcing 모드로 전환해야 한다.
- **레이블링 누락**: 중요 파일에 보안 레이블을 부착하지 않으면, 기본적으로 최하위 등급으로 간주되어 유출 가능하다. 모든 중요 파일은 명시적으로 레이블링해야 한다.

## 👶 어린이를 위한 3줄 비유 설명
1. Bell-LaPadula 모델은 **"회사의 기밀 등급과 문서 분류"**예요. "1등급(사장), 2등급(임원), 3등급(직원)"으로 나뉘어 있어요.
2. "직원은 임원 문서를 읽을 수 있지만, 임원은 직원 문서를 못 읽어요(No Read Up). 그리고 "사장은 1등급 문서에만 쓸 수 있고, 3등급 파일에 못 써요(No Write Down)."
3. 그래서 기밀이 새어 나가지 않아요! 직원이 사장님 비밀 문서를 읽을 수도 없고, 사장님이 기밀을 일반 문서에 숨겨서 공개할 수도 없어요.
