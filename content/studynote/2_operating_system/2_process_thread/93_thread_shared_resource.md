# 스레드의 자원 공유 - Code, Data, Heap, 열린 파일

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스레드 자원 공유는 동일한 프로세스에 속한 여러 스레드가 **프로세스의 주소 공간과 시스템 자원**을 공동으로 사용하여 데이터 교환 효율을 극대화하는 메커니즘이다.
> 2. **가치**: 프로세스 간 통신(IPC)과 같은 복잡한 절차 없이 전역 변수나 동적 메모리를 통해 즉각적으로 정보를 주고받을 수 있어, 협업 작업의 속도와 경제성을 비약적으로 높인다.
> 3. **융합**: 공유의 편리함 이면에 데이터 정합성이 깨지는 임계 영역(Critical Section) 문제가 수반되며, 이는 운영체제 동기화 기법을 통해 반드시 보호되어야 할 양날의 검과 같다.

+++

## Ⅰ. 스레드 공유 자원의 범주 (Deep Dive)

프로세스 내의 모든 스레드는 다음과 같은 영역을 100% 공유한다.

### 1. Code (Text) 영역
- **내용**: 실행할 프로그램의 기계어 코드.
- **특징**: 모든 스레드는 동일한 함수와 로직을 실행할 권한을 가진다. 읽기 전용이므로 충돌 위험이 없다.

### 2. Data 영역
- **내용**: 초기화된/초기화되지 않은(BSS) 전역 변수와 정적(static) 변수.
- **특징**: 프로그램 시작부터 종료까지 유지되는 데이터로, 한 스레드가 수정하면 모든 스레드가 그 변화를 즉시 확인한다.

### 3. Heap 영역
- **내용**: 런타임에 동적으로 할당된 메모리 (`malloc`, `new`).
- **특징**: 공유된 포인터만 있다면 어떤 스레드라도 접근 가능하다. 대규모 데이터를 공동 작업할 때 핵심적인 역할을 한다.

### 4. OS 자원 (Open Files, Signals)
- **내용**: 열려 있는 파일 디스크립터, 소켓, 환경 변수 등.
- **특징**: 한 스레드가 파일을 열면 다른 스레드도 그 파일을 읽거나 쓸 수 있다.

### 5. 💡 비유: '한 지붕 아래 한 가족'
- **Code**: 집에 비치된 공용 가전제품 사용 설명서.
- **Data**: 거실 냉장고에 넣어둔 공용 반찬(전역 변수). 누구나 꺼내 먹을 수 있다.
- **Heap**: 마당에 새로 지은 창고(동적 할당). 가족이면 누구나 짐을 넣고 뺄 수 있다.
- **Files**: 집으로 배달되는 우편물이나 신문.

## Ⅱ. 자원 공유의 기술적 가치 (Deep Dive)

### 1. 통신 오버헤드 제거
- 프로세스는 서로 다른 주소 공간을 쓰기 때문에 데이터를 주고받으려면 커널을 거치는 IPC(메시지 큐, 소켓 등)를 써야 하지만, 스레드는 **단순히 메모리 주소에 값을 쓰는 것**만으로 통신이 완료된다.

### 2. 메모리 사용 효율화
- 동일한 라이브러리나 대형 데이터를 스레드마다 따로 복사할 필요가 없어 실제 물리 메모리 점유율을 획기적으로 낮춘다.

### 3. ASCII 구조 다이어그램
```text
    [ Process Memory Layout ]
    ┌──────────────────────────────────┐
    │  Shared: Code Segment            │ ◀── 동일 로직 실행
    ├──────────────────────────────────┤
    │  Shared: Data (Global Variables) │ ◀── 전역 데이터 공유
    ├──────────────────────────────────┤
    │  Shared: Heap (Dynamic Memory)   │ ◀── 대규모 데이터 협업
    ├──────────────────────────────────┤
    │  Shared: Open Files / Sockets    │ ◀── 시스템 자원 공유
    ├──────────────────┬───────────────┤
    │ [ Thread A ]     │ [ Thread B ]  │ ◀── (실행 흐름만 분리)
    └──────────────────┴───────────────┘
```

- **📢 섹션 요약 비유**: 각자 자기 방(Stack)은 따로 쓰지만, 주방과 거실(공유 영역)을 같이 쓰기 때문에 훨씬 넓고 편리한 생활이 가능한 것과 같습니다.

+++

## Ⅲ. 자원 공유의 위험성: 경쟁 상태 (Race Condition)

공유의 대가로 지불해야 하는 가장 큰 비용은 **정합성 관리**이다.

| 위험 요소 | 설명 | 결과 |
|:---|:---|:---|
| **Race Condition** | 여러 스레드가 동시에 공유 변수를 수정하려 함 | 최종 값이 예상과 다르게 저장됨 |
| **Deadlock** | 서로가 가진 공유 자원의 락을 기다리며 멈춤 | 시스템 마비 |
| **Inconsistency** | 데이터 수정 도중 다른 스레드가 읽어감 | 깨진 데이터를 기반으로 잘못된 연산 수행 |

- **📢 섹션 요약 비유**: 냉장고에 우유가 하나 있는데, 엄마와 아빠가 동시에 "우유 사야지"라고 생각하고 각각 사 오면 냉장고가 우유로 넘치는 것(경쟁 상태)과 같습니다. "내가 사 올게!"라는 메모(동기화)가 필요합니다.

+++

## Ⅳ. 실무 적용 및 설계 전략 (Strategy)

### 1. 실무 시나리오: 이미지 필터링 가속
- **상황**: 4K 해상도의 거대 이미지에 필터를 적용해야 함.
- **판단**: 이미지를 힙(Heap) 영역에 하나만 로드하고, 4개의 스레드에게 각각 이미지의 1/4씩 영역을 할당하여 동시에 연산한다. 스레드들은 원본 이미지 데이터(Heap)를 공유하므로 복사 시간이 들지 않아 싱글 스레드보다 4배 가까운 성능 향상을 얻는다.

### 2. 기술사적 인사이트: Thread-Safe 설계
- 자원 공유 환경에서는 **Thread-safe**한 라이브러리 사용이 필수다. 기술사라면 공유 자원 접근 시 발생하는 락(Lock) 경합으로 인한 성능 저하를 방지하기 위해, 공유를 최소화하는 **TLS (Thread Local Storage)** 기술이나, 락을 걸지 않고 연산하는 **Atomic Operation**의 활용 방안을 제시할 수 있어야 한다.

- **📢 섹션 요약 비유**: 모두가 거실(공유 공간)에서 부딪히지 않게 하려면, 거실 이용 시간을 나누거나 각자 자기 방(TLS)에서 할 수 있는 일은 최대한 방에서 하는 것이 효율적입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **애플리케이션 성능 향상**: 저비용 통신을 통한 연산 병렬화 극대화.
- **시스템 자원 절약**: 중복 데이터 제거를 통한 효율적인 메모리 운용.

### 2. 미래 전망
최근에는 메모리 공유의 위험성을 근본적으로 해결하기 위해, 공유 자원의 소유권을 언어 차원에서 관리하는 **Rust**나, 공유 대신 메시지를 주고받는 **Go(Goroutine)** 언어가 부상하고 있다. 하지만 운영체제 커널 수준에서의 자원 공유 메커니즘은 여전히 모든 고성능 컴퓨팅 아키텍처의 가장 밑바닥을 지탱하는 바닥 돌로 남을 것이다.

- **📢 섹션 요약 비유**: 함께 나누는 기쁨(성능)이 크지만, 질서(동기화)가 없으면 아수라장이 되는 현대 운영체제의 핵심 원리입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **스레드 독립 자원**: 공유되지 않는 스레드만의 영역.

- **임계 영역 (Critical Section)**: 공유 자원 접근을 보호해야 하는 구역.
