## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPU 친화성 (CPU Affinity)은 특정 프로세스나 스레드가 실행될 수 있는 CPU 코어의 집합을 제한하여, 프로세서와 프로세스 간의 실행 관계를 고정하거나 선호도를 부여하는 기법이다.
> 2. **가치**: 캐시 지역성(Locality)을 보존하여 성능을 최적화하고, 실시간 시스템에서 결정론적(Deterministic) 실행 시간을 보장하며, 특정 코어를 특정 작업에 전담시켜 간섭(Interference)을 최소화한다.
> 3. **융합**: 고성능 컴퓨팅(HPC), 가상화 하이퍼바이저(KVM/Xen), 그리고 컨테이너 오케스트레이션(Kubernetes CPU Manager)에서 리소스 격리 및 성능 예측 가능성을 확보하는 핵심 기술이다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: **CPU 친화성 (CPU Affinity)**은 운영체제의 스케줄러가 프로세스를 어느 CPU 코어에 할당할지 결정할 때 사용하는 정책이다. **소프트 친화성(Soft Affinity)**은 가급적 이전에 실행되었던 코어를 선호하는 방식이며, **하드 친화성(Hard Affinity)**은 사용자가 명시적으로 `sched_setaffinity` 등을 통해 실행 가능한 코어 범위를 강제하는 방식이다.

- **💡 비유**: CPU 친화성은 대형 병원에서 **"전담 주치의"**를 배정하는 것과 같다. 환자(프로세스)가 올 때마다 매번 다른 의사에게 진료받는 것보다, 환자의 상태를 잘 아는 전담 의사(이전 코어)에게 진료받는 것이 진료 차트(캐시 데이터)를 다시 읽는 시간을 줄여 훨씬 효율적인 것과 유사하다.

- **등장 배경**:
  1. **멀티코어 아키텍처의 일반화**: 코어 간 데이터 이동 비용이 증가하면서, 스케줄러의 무분별한 프로세스 이동(Migration)이 성능 병목의 원인이 되었다.
  2. **공유 캐시 경합**: 여러 프로세스가 한 코어를 공유할 때 발생하는 캐시 오염(Cache Pollution)을 막기 위해 물리적 격리가 필요해졌다.
  3. **실시간성 요구**: 금융 거래나 산업 제어 시스템에서 OS의 스케줄링 개입으로 인한 지터(Jitter)를 제거하기 위해 특정 코어를 전점(Exclusive)으로 할당해야 했다.

- **📢 섹션 요약 비유**: 뜨내기 손님이 아닌 단골 고객에게 전용 좌석을 제공하여 서비스 속도와 만족도를 높이는 예약제 운영 전략과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### CPU 친화성 구성 요소 및 제어 체계 (표)

| 요소명 | 역할 | 내부 동작 | 관련 도구 | 비유 |
|:---|:---|:---|:---|:---|
| **Affinity Mask** | 실행 가능한 CPU 비트맵 | TCB/PCB 내 비트마스크 필드로 관리 | `taskset` 명령 | 출입 허가증 |
| **Scheduler** | 실제 코어 할당 결정 | 마스크를 참조하여 `Runqueue` 배정 | `Completely Fair Scheduler (CFS)` | 배정 담당자 |
| **Migration** | 코어 간 태스크 이동 | 부하 분산(Load Balancing) 시 발생 | `sched_migrate_task` | 강제 전근 |
| **Isolcpus** | 특정 코어 격리 | 커널 수준에서 일반 스케줄링 제외 | 커널 파라미터 | VIP 전용 구역 |
| **CPU Set** | 프로세스 그룹별 코어 할당 | 컨트롤 그룹(cgroups)과 연계하여 관리 | `cpuset` (cgroup) | 부서별 전용 사무실 |

#### CPU 친화성과 캐시 계층 구조 아키텍처

프로세스가 특정 코어에 고정되면 해당 코어의 L1/L2 캐시 적중률이 극대화된다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │                 CPU 친화성 기반 리소스 격리 구조                  │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │   [Physical CPU 0]              [Physical CPU 1]             │
  │   ┌──────────────┐              ┌──────────────┐             │
  │   │  L1/L2 Cache │              │  L1/L2 Cache │             │
  │   │ (Warm State) │              │ (Cold State) │             │
  │   └──────▲───────┘              └──────┬───────┘             │
  │          │                             │                     │
  │     [Process A] <──(Affinity)──▶ [Process B]                 │
  │     (Core 0 Only)               (Core 1 Only)                │
  │                                                              │
  │   ┌──────────────────────────────────────────────────────┐   │
  │   │            L3 Cache (Shared / Last Level)            │   │
  │   └──────────────────────────────────────────────────────┘   │
  │                                                              │
  │   1. Hard Affinity: A는 반드시 0번, B는 1번 코어에서만 실행    │
  │   2. Benefit: A의 데이터는 0번 캐시에 유지되어 메모리 접근 최소화 │
  │   3. Isolation: B의 작업이 A의 L1/L2 캐시를 오염시키지 않음      │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 현대 CPU 아키텍처에서 L1/L2 캐시는 각 코어에 종속적이다. 프로세스 A가 코어 0에 친화성이 설정되면, A가 사용하는 데이터는 코어 0의 캐시에 "Warm" 상태로 머물게 된다. 만약 친화성이 없다면 스케줄러가 부하 분산을 위해 A를 코어 1로 옮길 수 있고, 이 경우 코어 1의 "Cold" 캐시로 인해 성능 저하가 발생한다. CPU 친화성을 통해 프로세스별 전담 코어를 지정하면 캐시 적중률(Cache Hit Rate)을 보존하고 프로세스 간 간섭을 물리적으로 차단할 수 있다.

#### 심층 동작 원리: `sched_setaffinity` 시스템 콜

리눅스 커널에서 CPU 친화성은 프로세스의 `task_struct` 내 `cpus_ptr` 필드에 의해 관리된다. 사용자가 `sched_setaffinity`를 호출하면 다음과 같은 단계로 동작한다.

1. **유효성 검사**: 요청된 CPU 마스크가 현재 시스템의 가용한 CPU 범위 내에 있는지 확인한다.
2. **마스크 업데이트**: 해당 프로세스의 `task_struct` 내 친화성 비트마스크를 갱신한다.
3. **강제 이동(Push/Pull)**: 만약 프로세스가 현재 실행 중인 코어가 새로운 마스크에 포함되지 않는다면, 즉시 유효한 코어로 마이그레이션을 수행한다.
4. **스케줄러 반영**: 이후 스케줄러는 `select_task_rq` 함수 호출 시 해당 마스크를 참조하여 다음 실행 코어를 결정한다.

- **📢 섹션 요약 비유**: 여행자가 갈 수 있는 목적지를 여권(Affinity Mask)에 미리 도장 찍어 제한함으로써, 정해진 경로 안에서만 이동하게 관리하는 시스템과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

#### 비교: 소프트 친화성 vs 하드 친화성

| 비교 항목 | 소프트 친화성 (Soft Affinity) | 하드 친화성 (Hard Affinity) |
|:---|:---|:---|
| **제어 주체** | OS 스케줄러 (자동) | 사용자/관리자 (명시적) |
| **강제성** | 권장 사항 (Best Effort) | 강제 사항 (Strict) |
| **유연성** | 높음 (코어 유휴 시 이동 가능) | 낮음 (지정 코어 바쁘면 대기) |
| **성능 예측성** | 보통 | 매우 높음 |
| **주 사용처** | 일반적인 데스크탑 응용 프로그램 | 데이터베이스 엔진, 실시간 제어, NFV |

#### 과목 융합 관점: 가상화와 컨테이너

- **가상화 (Cloud)**: 가상 CPU(vCPU)를 물리 CPU(pCPU)에 1:1로 고정하는 'vCPU Pinning'은 클라우드 환경에서 노이즈 네이버(Noisy Neighbor) 문제를 해결하는 핵심 기법이다.
- **보안 (Security)**: 사이드 채널 공격(Side-channel Attack)을 방어하기 위해 민감한 데이터를 처리하는 프로세스를 격리된 코어에 배치하여 캐시 공유를 통한 정보 유출을 차단할 수 있다.

- **📢 섹션 요약 비유**: 가급적 자기 자리에 앉는 것(소프트)과, 반드시 지정된 번호의 좌석에만 앉아야 하는 것(하드)의 차이로 성능과 통제력을 조절하는 전략입니다.

### Ⅳ. 실무 적용 및 기술사적 판단

#### 실무 시나리오: 고성능 DB 서버 최적화
대규모 트랜잭션을 처리하는 DB 서버에서 특정 워커 스레드들이 코어를 자주 옮겨 다닐 경우 L3 캐시 경합과 컨텍스트 스위칭 오버헤드로 인해 처리량이 급감한다. 이때 `taskset` 명령어를 사용하여 인터럽트 처리를 담당하는 코어와 DB 엔진 스레드가 실행되는 코어를 분리(Isolation)함으로써, I/O 부하가 쿼리 처리 성능에 주는 간섭을 최소화하고 응답 속도를 20% 이상 개선할 수 있다.

#### 도입 체크리스트
- **CPU 토폴로지 확인**: `lscpu`를 통해 물리 코어, 논리 코어(HT), NUMA 노드 구조를 정확히 파악했는가?
- **하이퍼스레딩(HT) 고려**: 논리 코어 두 개에 연산 집약적 프로세스를 친화시키면 자원 경합이 발생하므로 물리 코어 단위로 배분했는가?
- **인터럽트 친화성(IRQ Affinity)**: 프로세스뿐만 아니라 NIC 등의 하드웨어 인터럽트도 적절한 코어에 분산시켰는가?

#### 안티패턴
- **과도한 Pinning**: 시스템의 모든 프로세스를 수동으로 고정하면 스케줄러의 부하 분산 기능이 마비되어 특정 코어만 과열되고 나머지는 노는 '자원 불균형'이 발생한다.
- **커널 스레드 무시**: 사용자 프로세스를 특정 코어에 고정해도 커널의 백그라운드 작업(kworker 등)이 해당 코어를 침범할 수 있으므로 `isolcpus`를 통한 완전 격리가 병행되어야 한다.

- **📢 섹션 요약 비유**: 주방장(코어)에게 전담 요리(프로세스)를 맡기는 것은 좋지만, 한 명에게만 주문을 몰아주면 다른 주방장들이 노는 비효율이 발생하는 것과 같습니다.

### Ⅴ. 기대효과 및 결론

#### 정량/정성 기대효과

| 구분 | 도입 전 (Dynamic) | 도입 후 (Affinity) | 개선 효과 |
|:---|:---|:---|:---|
| **캐시 적중률** | 70% 내외 (불안정) | 90% 이상 (안정적) | 메모리 병목 현상 완화 |
| **지연 시간 (Latency)** | 50ms (최대 지연) | 5ms 이내 (일정) | 실시간 응답성 10배 향상 |
| **시스템 처리량** | 가변적 | 고부하 상황에서도 안정적 유지 | 예측 가능한 서비스 품질(QoS) 보장 |

#### 미래 전망
ARM 네오버스(Neoverse)와 같은 서버용 매니코어 아키텍처가 확산되면서 코어 수가 수백 개로 늘어나고 있다. 이에 따라 수동으로 친화성을 설정하는 단계를 넘어, AI 기반 스케줄러가 워크로드의 특성(Compute-bound vs I/O-bound)을 분석하여 실시간으로 최적의 CPU 친화성 맵을 생성하고 관리하는 자동 최적화 기술이 보편화될 것이다.

- **참고 표준**: POSIX `sched_setaffinity`, Linux `cpuset` (cgroups v2)

- **📢 섹션 요약 비유**: 선수마다 가장 잘 뛰는 포지션(코어)을 정해주어 팀 전체의 승률(시스템 성능)을 끌어올리는 감독의 지혜와 같습니다.

### 📌 관련 개념 맵
- **스레드 고정 (Thread Affinity)**: 스레드 단위의 세밀한 코어 바인딩.
- **NUMA (Non-Uniform Memory Access)**: 메모리 위치를 고려한 고도화된 친화성 전략.
- **컨텍스트 스위칭 (Context Switching)**: 친화성을 통해 줄이고자 하는 주요 비용.
- **cgroups (Control Groups)**: 리소스 제한과 친화성을 결합 관리하는 커널 메커니즘.

### 👶 어린이를 위한 3줄 비유 설명
1. CPU 친화성은 요리사 아저씨가 **"항상 자기 자리에서만 요리하게"** 약속하는 거예요.
2. 매번 자리를 옮기지 않으니까 자기 칼과 도구가 어디 있는지 금방 찾아서 요리를 훨씬 빨리 완성할 수 있죠.
3. 다른 사람이 내 도구를 건드리지 못하게 해서, 중요한 요리를 아주 맛있고 정확하게 만들어낼 수 있답니다!
