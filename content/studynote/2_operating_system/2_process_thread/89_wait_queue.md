# 대기 큐 (Wait Queue / Device Queue)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 대기 큐 (Wait Queue)는 입출력(I/O) 완료나 특정 신호(세마포어 등)를 기다리며 실행이 중단된 **대기(Waiting/Blocked)** 상태의 프로세스들을 모아놓은 커널 내부의 리스트이다.
> 2. **가치**: 자원을 점유할 수 없는 프로세스들을 별도의 장소에 격리함으로써 스케줄러가 준비 큐(Ready Queue)에만 집중하게 하여 CPU 할당 효율을 높이고, 이벤트 발생 시 대상을 정확히 깨워(Wakeup)준다.
> 3. **융합**: 각 I/O 장치(Disk, Network Card 등)마다 전용의 **장치 큐 (Device Queue)**가 존재하며, 장치 인터럽트가 발생하면 해당 큐에 머물던 프로세스가 다시 준비 큐로 이동하게 된다.

+++

## Ⅰ. 대기 큐의 개요

### 1. 정의
- 특정 자원을 사용하기 위해 기다리거나, 요청한 작업이 끝나기를 기다리는 프로세스들의 연결 리스트이다.

### 2. 특징: '용도별 격리 수용'
- **다수 존재**: 준비 큐는 보통 시스템에 하나(또는 코어당 하나)이지만, 대기 큐는 기다려야 하는 대상(키보드, 마우스, 하드디스크, 네트워크, 뮤텍스 등)마다 각각 별도로 존재한다.
- **블로킹(Blocking)의 결과**: 프로세스가 `read()` 등의 시스템 콜을 호출하여 자원을 기다리게 되면 커널에 의해 강제로 대기 큐에 매달리게 된다.

### 3. 💡 비유: '병원의 정밀 검사실 대기 의자'
- **준비 큐**가 진료실(CPU)에 들어가기 위해 복도 의자에 앉아 있는 환자들이라면, **대기 큐**는 **'피 검사 결과가 나오기를 기다리며 지하 검사실 의자에 앉아 있는 환자들'**과 같다.
- 결과가 나오기 전까지는 진료실 순번이 되어도 들어갈 수 없으므로, 아예 다른 장소(대기 큐)에 가 있는 것이다.

+++

## Ⅱ. 대기 큐의 동작 메커니즘 (Deep Dive)

### 1. 큐 진입 (Sleep / Block)
1. 실행 중인 프로세스가 I/O 장치에 데이터를 요청한다.
2. 커널은 해당 프로세스의 PCB 상태를 **Waiting**으로 바꾼다.
3. 해당 I/O 장치 전용의 **대기 큐 (Wait Queue)** 맨 끝에 프로세스를 매단다.
4. CPU 스케줄러를 호출하여 다른 프로세스를 실행시킨다.

### 2. 큐 탈출 (Wakeup / Unblock)
1. I/O 장치가 작업을 마치고 CPU에 **인터럽트**를 보낸다.
2. 인터럽트 핸들러가 해당 장치의 대기 큐에서 기다리던 프로세스를 찾아낸다.
3. 프로세스의 상태를 **Ready**로 바꾸고, **준비 큐 (Ready Queue)**로 이동시킨다.

### 3. ASCII 구조 다이어그램
```text
    ┌────────────────┐ [ I/O Request ] ┌────────────────────────┐
    │ Running Process│ ───────────────▶│ Device Wait Queue (Disk)│
    └────────────────┘                 ├────────────────────────┤
                                       │ PCB (P1) ──▶ PCB (P2)  │
                                       └───────────┬────────────┘
                                                   │
                                                   │ [ I/O Done (Interrupt) ]
                                                   ▼
    ┌────────────────┐ [ Dispatch ]    ┌────────────────────────┐
    │ Actual CPU     │ ◀────────────── │      Ready Queue       │
    └────────────────┘                 └────────────────────────┘
```

- **📢 섹션 요약 비유**: 주문한 음식이 나오기를 기다리는 진동 벨(인터럽트)을 들고 푸드코트 구석 자리(대기 큐)에서 쉬다가, 벨이 울리면 다시 음식 받는 줄(준비 큐)로 이동하는 것과 같습니다.

## Ⅲ. 대기 큐의 종류와 관리

| 큐 종류 | 대기 대상 | 특징 |
|:---|:---|:---|
| **장치 큐 (Device Queue)** | 디스크, 네트워크, 터미널 등 | 물리적 장치의 응답 대기 |
| **조건 대기 큐** | 세마포어, 뮤텍스, 조건 변수 | 소프트웨어적 신호 및 동기화 대기 |
| **전역 대기 큐** | 특정되지 않은 커널 이벤트 | 시스템 전체 범위의 이벤트 대기 |

- **📢 섹션 요약 비유**: 마트의 계산대 줄(준비 큐)은 하나지만, 화장실 대기 줄, 고객센터 대기 줄, 주차 정산 대기 줄(각종 대기 큐)은 목적에 따라 여러 곳에 흩어져 있는 것과 같습니다.

+++

## Ⅳ. 실무 적용 및 운영 가이드 (Strategy)

### 1. 실무 시나리오: 좀비 프로세스와 대기 큐
- **상황**: 부모 프로세스가 자식의 종료를 기다리며 `wait()` 함수를 호출함.
- **판단**: 부모 프로세스는 자식이 죽었다는 신호(SIGCHLD)가 올 때까지 **대기 큐**에 들어가 잠들게 된다. 자식이 종료되면 커널이 부모를 깨워 준비 큐로 보내고, 부모는 자식의 종료 코드를 읽어 뒷정리를 한다. 만약 부모가 깨어나지 않으면 자식은 **좀비(Zombie)** 상태로 남게 된다.

### 2. 기술사적 인사이트: I/O Bound 프로세스의 처리량 개선
- 대기 큐에 머무는 시간이 긴 프로세스들은 시스템의 전반적인 처리량(Throughput)을 떨어뜨린다. 기술사라면 **비동기 I/O (Async I/O)**나 **입출력 다중화 (Select/Poll/Epoll)** 기술을 통해, 프로세스가 대기 큐에서 잠들지 않고 다른 일을 계속 수행할 수 있도록 설계하는 방안을 제시해야 한다.

- **📢 섹션 요약 비유**: 은행 업무를 보면서 결과가 나올 때까지 마냥 앉아 있는 것보다, 스마트폰 앱(비동기)으로 결과를 확인하고 그동안 다른 일을 하는 것이 훨씬 효율적이라는 논리입니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **CPU 활용도 최적화**: 일할 수 없는 프로세스를 스케줄링 대상에서 제외하여 자원 낭비 방지.
- **시스템 질서 유지**: 수천 개의 프로세스가 엉키지 않고 자기 순서와 자원을 기다릴 수 있는 체계 제공.

### 2. 미래 전망
최근의 고성능 서버 환경에서는 대기 큐로 인한 컨텍스트 스위칭 비용을 줄이기 위해, 짧은 시간의 대기라면 잠들지 않고 CPU를 계속 돌리며 확인하는 **스핀락 (Spinlock)**이나, 사용자 공간에서 대기를 처리하는 **User-level Wait Queue** 기술이 각광받고 있다. 하지만 안정성을 최우선으로 하는 범용 운영체제에서 대기 큐는 영원한 '프로세스의 휴식처'이자 '질서의 근간'으로 남을 것이다.

- **📢 섹션 요약 비유**: 쉼(Waiting)이 있기에 더 큰 도약(Running)이 가능하다는, 운영체제 안의 작은 쉼표 같은 공간입니다.

- **인터럽트 (Interrupt)**: 대기 큐에서 프로세스를 깨우는 전기적 신호.
