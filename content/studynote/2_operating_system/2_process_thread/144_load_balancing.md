# 부하 분산 (Load Balancing) - 푸시 이주, 풀 이주

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 부하 분산(Load Balancing)은 멀티프로세서 환경에서 특정 CPU 코어에만 작업이 몰리지 않도록, **시스템 전체의 프로세스나 스레드 부하를 모든 코어에 균등하게 배분**하는 기술이다.
> 2. **가치**: 노는 코어(Idle Core)가 없게 하여 **시스템의 처리량(Throughput)을 극대화**하며, 특정 코어의 과부하로 인한 응답 지연을 방지하여 사용자 경험을 최적화한다.
> 3. **융합**: 과부하된 코어가 작업을 밀어내는 **푸시 이주(Push Migration)**와 한가한 코어가 작업을 가져오는 **풀 이주(Pull Migration)** 기법을 유기적으로 결합하여 동적인 부하 조절을 실현한다.

---

## Ⅰ. 부하 분산의 정의 및 필요성

### 1. 정의
- 다중 처리기 시스템에서 각 처리기의 작업 부하(Workload)를 주기적으로 감시하고, 불균형이 발견되면 프로세스를 한 처리기에서 다른 처리기로 이동(Migration)시키는 과정이다.

### 2. 왜 필요한가? (Efficiency vs Fairness)
- **자원 활용**: 어떤 코어는 100%인데 어떤 코어는 0%라면 하드웨어 낭비이다.
- **응답 시간**: 특정 코어에 긴 줄이 생기면 해당 프로세스들의 대기 시간이 길어진다.
- **시스템 안정성**: 코어 간 온도 불균형(Hotspot)을 해소하여 하드웨어 수명을 보호한다.

### 3. 💡 비유: '마트 계산대의 줄 서기'
- 부하 분산은 **'마트 점원이 손님이 없는 계산대로 손님들을 유도하는 과정'**과 같다.
- 내 줄은 너무 긴데 옆 계산대가 비어 있다면, 점원이 "이쪽으로 오세요"라고 밀어주거나(Push), 옆 계산대 직원이 "이분들 이쪽으로 오세요"라고 데려가는(Pull) 것과 같다.

---

## II. 부하 분산의 두 가지 핵심 기법 (Deep Dive)

### 1. 푸시 이주 (Push Migration)
- **주체**: 특정 감시 태스크(Monitor Task).
- **동작**: 시스템의 전반적인 부하를 체크하다가, 과부하된 코어를 발견하면 그곳의 프로세스를 한가한 코어로 **강제로 밀어낸다(Push)**.
- **장점**: 시스템 전체 관점에서의 선제적 조절이 가능하다.

### 2. 풀 이주 (Pull Migration)
- **주체**: 한가한 코어 (Idle Processor).
- **동작**: 코어가 할 일이 없어지면, 바쁜 다른 코어의 대기 큐를 살펴보고 기다리는 프로세스를 **자신에게로 가져온다(Pull)**.
- **장점**: 코어 스스로가 노는 시간을 최소화하려는 자발적 효율성이 높다.

### 3. ASCII 구조 다이어그램
```text
    [ Core 0 (Busy) ]                   [ Core 1 (Idle) ]
    ┌─────────────┐                     ┌─────────────┐
    │  Task A     │                     │             │
    │  Task B     │ ───(1. Push)──▶     │             │
    │  Task C     │                     │             │
    │  Task D     │ ◀──(2. Pull)───     │             │
    └─────────────┘                     └─────────────┘
```

- **📢 섹션 요약 비유**: "넘치는 쪽에서 덜어주는 배려(Push)"와 "부족한 쪽에서 채우려는 의지(Pull)"가 만나 전체의 조화(Balancing)를 이루는 시스템입니다.

## Ⅲ. 부하 분산과 CPU 친화성의 갈등 (Conflict)

부하 분산은 성능 최적화의 또 다른 축인 **CPU 친화성(CPU Affinity)**과 정면으로 충돌한다.

| 항목 | 부하 분산 (Load Balancing) | CPU 친화성 (Affinity) |
|:---|:---|:---|
| **목표** | **모든 코어를 고르게 사용** | **한 코어에 오래 머물기** |
| **장점** | 처리량(Throughput) 향상 | 캐시 적중률(Cache Hit) 향상 |
| **단점** | 캐시 무효화로 개별 성능 저하 우려 | 특정 코어 병목 유발 우려 |
| **해결책** | 불균형이 심각할 때만 이주 수행 | **NUMA 인식 스케줄링** 도입 |

- **📢 섹션 요약 비조**: "이사는 집안 정리(부하 분산)를 위해 필요하지만, 이사를 너무 자주 가면 짐 정리(캐시)가 안 되어 생활이 힘들어집니다. 적절한 이사 횟수를 정하는 것이 스케줄러의 지혜입니다."

---

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 가상화 서버 부하 조절
- **상황**: 클라우드 서버 내의 특정 VM이 연산을 폭발적으로 수행하여 특정 물리 코어 점유율이 100%에 도달.
- **판단**: 하이퍼바이저 수준의 스케줄러가 **푸시 이주**를 가동한다. 해당 VM의 가상 CPU 스레드를 즉시 다른 유휴 물리 코어로 이동시켜 서비스 중단을 막는다. 이때 메모리 접근 지연을 최소화하기 위해 가급적 같은 소켓(Socket) 내의 코어로 이동시키는 정교한 아키텍처 설계가 필요하다.

### 2. 기술사적 인사이트: 전력 관리와의 결합 (DPM)
- 기술사라면 부하 분산을 **전력 관리(Dynamic Power Management)** 관점에서도 바라봐야 한다. 부하가 적을 때는 모든 코어에 분산시키는 대신, 오히려 한두 코어로 작업을 몰아넣고(Consolidation) 나머지 코어를 절전 모드(Deep Sleep)로 전환하는 것이 에너지 효율 면에서 유리할 수 있다. "성능 중심의 분산과 전력 중심의 집중" 사이에서 동적으로 정책을 변경하는 능력이 중요함을 강조해야 한다.

- **📢 섹션 요약 비조**: "최고의 성능은 분산에서 나오지만, 최고의 효율은 지능적인 집중에서 나올 수 있습니다."

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **자원 회수율 극대화**: 하드웨어 가용 자원의 90% 이상 상시 활용 지원.
- **공정성 보장**: 모든 프로세스가 차별 없이 CPU 시간을 할당받는 환경 조성.

### 2. 미래 전망
앞으로는 수천 개의 미세 코어가 탑재되는 Many-core 아키텍처와 이기종(Heterogeneous) 환경이 대세가 됨에 따라, 신경망 기반의 **자가 학습 부하 분산기**가 도입될 것이다. 부하 분산은 단순히 일을 나누는 행위를 넘어, 시스템의 열 역학적 안정성과 에너지 가성비를 조율하는 '운영체제의 관제탑'으로 진화할 것이다.

- **📢 섹션 요약 비유**: 오케스트라 지휘자가 특정 악기 파트가 너무 힘들지 않게, 그리고 한 명도 놀지 않게 악보를 연주 순서대로 정교하게 배분하는 마법 같은 지휘력과 같습니다.
