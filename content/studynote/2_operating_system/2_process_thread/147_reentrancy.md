# 재진입성 (Reentrancy) - 스레드 안전의 부분집합

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 재진입성(Reentrancy)은 어떤 함수가 **실행 도중 중단되었다가 다시 호출(재진입)되어도 결과의 정확성이 보장**되는 성질로, 전역 변수나 정적 변수를 배제하고 오직 자신의 로컬 자원만 사용하는 구조이다.
> 2. **가치**: 스레드 안전(Thread-Safe)보다 더 엄격한 조건이며, 다중 스레드 환경뿐만 아니라 **재귀 함수 호출이나 인터럽트 핸들러** 내에서 안전하게 함수를 사용할 수 있게 한다.
> 3. **융합**: 모든 재진입 함수는 스레드 안전하지만, 모든 스레드 안전 함수가 재진입 가능한 것은 아니다(예: 뮤텍스를 쓰는 함수는 재진입 시 데드락 위험).

---

## Ⅰ. 재진입성(Reentrancy)의 정의 및 조건

### 1. 정의
- 여러 실행 흐름이 동시에 혹은 순차적으로 같은 함수 코드를 밟고 지나가더라도, 각 흐름의 연산 결과가 서로 간섭받지 않고 독립적으로 유지되는 코드의 특성이다.

### 2. 재진입이 가능하기 위한 3대 조건
- **전역/정적 변수 사용 금지**: 공유 자원을 참조하거나 수정하지 않는다.
- **자신의 코드 수정 금지**: 실행 중에 스스로의 명령어를 바꾸지 않는다. (Self-modifying code 금지)
- **비재진입 함수 호출 금지**: 내부에서 재진입이 안 되는 다른 함수를 불러서는 안 된다.

### 3. 💡 비유: '공중화장실 vs 개인용 나침반'
- **비재진입 (뮤텍스 사용)**: **'한 번에 한 명만 들어가는 공중화장실'**과 같다. 스레드 안전은 보장되지만, 안에 사람이 있는데 또 들어가려 하면(재진입) 문이 잠겨 있어 사고가 난다.
- **재진입**: **'누구나 자기 손에 들고 있는 개인용 나침반'**과 같다. 여러 명이 동시에 나침반을 봐도 되고, 보다가 잠시 멈추고 다른 일을 하다 돌아와도 내 나침반의 바늘은 여전히 나만 볼 수 있는 위치를 정확히 가리키고 있다.

---

## II. 재진입성 vs 스레드 안전 (Deep Dive)

두 개념은 혼동하기 쉬우나 명확한 벤 다이어그램 관계를 가진다.

### 1. 포함 관계
- **재진입성 $\subset$ 스레드 안전**
- 즉, 재진입이 가능하게 설계된 코드는 무조건 스레드 안전하지만, 거꾸로는 성립하지 않는다.

### 2. 결정적 차이: 락(Lock)의 사용
- **스레드 안전**: 락(Lock)을 사용하여 공유 자원 접근을 제어한다. 하지만 함수 실행 도중 인터럽트가 발생하여 동일한 함수가 다시 호출되면, 이미 내가 쥐고 있는 락을 다시 얻으려다 **자기 자신과 데드락(Deadlock)**에 빠질 수 있다.
- **재진입성**: 락 자체가 필요 없는 구조(Stateless)를 지향하므로 인터럽트 상황에서도 안전하다.

### 3. ASCII 구조 다이어그램
```text
    [ Execution Path 1 ] ──▶ [ Function Area ] ◀── [ Execution Path 2 ]
                                    │
           ┌────────────────────────┴────────────────────────┐
           ▼ (Non-Reentrant)                                 ▼ (Reentrant)
    ┌──────────────────────────┐                      ┌──────────────────────────┐
    │ Uses Static Variable X   │                      │ Uses Stack Variable Y    │
    │ (Path 1 overwrites X,    │                      │ (Each path has own Y on  │
    │  Path 2 gets wrong X)    │                      │  its own stack frame)    │
    └──────────────────────────┘                      └──────────────────────────┘
```

- **📢 섹션 요약 비유**: 비재진입 함수는 '함께 쓰는 칠판'에 답을 적는 것이고, 재진입 함수는 '각자의 연습장'에 답을 적는 것입니다. 칠판은 남이 지울 수 있지만, 내 연습장은 나만 씁니다.

---

## Ⅲ. 재진입성이 필수적인 환경

| 환경 | 이유 |
|:---|:---|
| **인터럽트 핸들러** | 메인 루틴 수행 중 갑자기 호출되므로, 공유 자원 접근 시 시스템 마비 위험 |
| **재귀 함수 (Recursive)** | 함수가 자기 자신을 끝내기 전에 다시 진입하므로 재진입성 필수 |
| **시그널 핸들러** | 비동기적으로 발생하여 기존 연산을 중단시키고 재진입함 |
| **고성능 라이브러리** | 수많은 스레드가 동시에 호출해도 성능 저하(Lock) 없이 서비스해야 함 |

- **📢 섹션 요약 비조**: "인터럽트가 일상인 운영체제 커널의 핵심 함수들은 반드시 재진입이 가능하도록 설계되어야만 시스템의 생존을 보장할 수 있습니다."

---

## Ⅳ. 실무 적용 및 코드 아키텍처 (Strategy)

### 1. 실무 시나리오: C 표준 라이브러리의 진화
- **상황**: `strtok()` 함수를 멀티스레드에서 썼더니 문자열이 엉망으로 분리됨.
- **판단**: `strtok`은 내부적으로 정적 변수(Static)에 이전 위치를 저장하는 비재진입 함수이다. 이를 해결하기 위해 재진입 가능한 버전인 **`strtok_r()`**을 사용한다. 이 함수는 상태 정보를 사용자가 관리하는 외부 포인터(인자)로 넘겨받아 '상태 없음(Stateless)'을 유지한다.

### 2. 기술사적 인사이트: 순수 함수(Pure Function)의 지향
- 기술사라면 재진입성이 현대의 **함수형 프로그래밍** 철학과 맞닿아 있음을 강조해야 한다. 입력값이 같으면 항상 같은 결과를 내놓고 부수 효과(Side Effect)가 없는 '순수 함수'는 완벽한 재진입성을 보장한다. "코드의 복잡도를 낮추고 검증 가능성을 높이는 최선의 방법은 락을 거는 것이 아니라, 함수를 재진입 가능하게 만드는 것"임을 역설해야 한다.

- **📢 섹션 요약 비조**: "과거의 상태(State)에 의존하지 않는 독립적인 코드 한 줄이, 미래의 수만 개 코어에서 자유롭게 춤출 수 있는 자격을 얻습니다."

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **시스템 가용성 극대화**: 인터럽트나 시그널 발생 시에도 교착 상태 없는 안정적 운영.
- **성능 병목 제거**: 락 경합 없는 병렬 실행으로 CPU 자원 100% 활용.

### 2. 미래 전망
앞으로는 수천 개의 가벼운 태스크들이 동적으로 스케줄링되는 환경에서, 함수 하나하나의 **재진입성**은 설계의 선택이 아닌 '기본값'이 될 것이다. 특히 하드웨어 자원을 직접 제어하는 엣지 컴퓨팅과 임베디드 AI 분야에서 재진입 가능한 라이브러리 설계 역량은 핵심 경쟁력이 될 것이다. 재진입성은 프로그램에 '불멸의 정직함'을 부여하는 아키텍처의 미덕이다.

- **📢 섹션 요약 비유**: 수천 번을 다시 태어나도(재진입) 이전의 기억에 휘둘리지 않고 오직 현재의 도구(인자)로만 승부하는, 가장 깨끗하고 강력한 디지털 전사와 같습니다.
