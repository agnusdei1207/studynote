# CPU 친화성 (CPU Affinity) - 특정 코어에 스레드 고정

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPU 친화성(CPU Affinity)은 특정 프로세스나 스레드가 **지정된 CPU 코어에서만 실행되도록 제한하거나 우선순위를 부여**하는 운영체제 스케줄링 기법이다.
> 2. **가치**: 스레드가 코어를 옮겨 다닐 때 발생하는 **캐시 무효화(Cache Invalidation)와 캐시 채우기(Refill) 비용을 최소화**하여, 데이터 지역성(Locality)을 극대화하고 성능을 최적화한다.
> 3. **융합**: 소프트웨어적으로 강제하는 하드 친화성(Hard Affinity)과 스케줄러가 가급적 유지하려고 노력하는 소프트 친화성(Soft Affinity)으로 나뉘며, 실시간 시스템과 고성능 서버 설계의 필수 요소이다.

---

## Ⅰ. CPU 친화성의 정의 및 필요성

### 1. 정의
- 프로세스가 마지막으로 실행되었던 프로세서(Core)를 기억하고, 다음 실행 시에도 가급적 동일한 프로세서에 할당하려는 성질 또는 강제 규칙이다.

### 2. 왜 필요한가? (The Cache Problem)
- **캐시 메모리**: 각 코어는 자신만의 고속 L1/L2 캐시를 가진다.
- **오버헤드**: 스레드가 A 코어에서 B 코어로 이동하면, A 코어 캐시에 정성껏 쌓아둔 데이터들이 무용지물이 되고 B 코어에서 다시 메모리를 읽어 캐시를 채워야 한다. 이 과정에서 막대한 성능 손실이 발생한다.

### 3. 💡 비유: '단골 식당의 지정석'
- CPU 친화성은 **'내가 어제 먹다 남긴 음료수와 소스가 그대로 놓여 있는 식당 테이블(코어 캐시)에 오늘도 앉는 것'**과 같다.
- 매번 다른 테이블(다른 코어)에 앉으면 어제 쓰던 도구들을 다시 가져와야 하지만, 지정석(Affinity)에 앉으면 바로 밥을 먹기 시작(고속 연산)할 수 있다.

---

## II. 하드 친화성 vs 소프트 친화성 (Deep Dive)

운영체제는 두 가지 방식으로 친화성을 관리한다.

### 1. 소프트 친화성 (Soft Affinity)
- **동작**: 스케줄러가 "웬만하면 이전에 썼던 코어에 배치해주겠다"라고 노력은 하지만, 부득이한 상황(다른 코어가 너무 놀고 있을 때 등)에는 다른 코어로 옮길 수 있다.
- **특징**: 부하 분산(Load Balancing)과 성능 사이의 균형을 중시한다.

### 2. 하드 친화성 (Hard Affinity)
- **동작**: 개발자가 시스템 콜(`sched_setaffinity` 등)을 통해 "이 스레드는 무조건 0번과 1번 코어에서만 돌아야 한다"라고 **강제**하는 방식이다.
- **특징**: 특정 스레드의 성능을 완벽히 보장해야 할 때 사용하며, 스케줄러는 이 규칙을 절대 어길 수 없다.

### 3. ASCII 구조 다이어그램
```text
    [ Thread Pool ]                     [ CPU Cores ]
    ┌──────────┐                        ┌──────────┐
    │ Thread A │ ───(Hard Affinity)───▶ │  Core 0  │ (Private Cache)
    └──────────┘                        └──────────┘
    ┌──────────┐                        ┌──────────┐
    │ Thread B │ ───(Soft Affinity)───▶ │  Core 1  │ (Prefer this!)
    └────┬─────┘                        └────▲─────┘
         └──────────(Migration)──────────────┘
```

- **📢 섹션 요약 비유**: 소프트 친화성이 "자주 가던 길로 가자"는 습관이라면, 하드 친화성은 "이 길 아니면 안 가!"라고 고집을 부리는 전용 도로 설정과 같습니다.

## Ⅲ. CPU 친화성의 장단점 분석

| 구분 | 장점 (Pros) | 단점 (Cons) |
|:---|:---|:---|
| **성능** | **캐시 적중률 극대화, 파이프라인 효율 향상** | - |
| **예측성** | 실행 시간의 편차(Jitter) 감소 | - |
| **부하 분산** | - | **부하 불균형** 초래 (특정 코어만 바쁠 수 있음) |
| **유연성** | - | 시스템 구성 변경(코어 추가/제거) 시 관리 복잡 |

- **📢 섹션 요약 비조**: "개별 선수의 컨디션(캐시) 관리에는 최고지만, 팀 전체의 업무 분담(부하 분산) 측면에서는 비효율이 생길 수 있는 양날의 검입니다."

---

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 고성능 패킷 처리 (DPDK)
- **상황**: 100Gbps 이상의 네트워크 트래픽을 처리해야 하는 보안 장비 개발.
- **판단**: 네트워크 인터페이스 카드(NIC)와 직접 통신하는 스레드들을 특정 코어에 **하드 친화성**으로 고정한다. 해당 코어에서는 운영체제의 다른 서비스가 끼어들지 못하게 격리(Isolate) 처리한다. 이를 통해 인터럽트 지연을 0에 가깝게 줄이고, 캐시 미스 없는 초고속 패킷 처리를 구현한다.

### 2. 기술사적 인사이트: NUMA 아키텍처와의 연계
- 기술사라면 CPU 친화성을 **NUMA(Non-Uniform Memory Access)** 구조와 연결해 생각해야 한다. 단순히 코어만 맞추는 것이 아니라, 해당 코어와 물리적으로 가까운 메모리 노드에 데이터를 배치하는 **NUMA Affinity**까지 고려해야 진정한 고성능 아키텍처를 완성할 수 있음을 강조해야 한다.

- **📢 섹션 요약 비조**: "가까운 곳에 앉는 것(CPU Affinity)만큼이나, 필요한 재료를 가까운 냉장고(Memory Affinity)에 두는 설계가 대규모 서버 성능의 핵심입니다."

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **IPC(Instructions Per Cycle) 향상**: 유효 데이터가 캐시에 상주함으로써 실제 연산 속도 증가.
- **전력 효율 개선**: 불필요한 메모리 버스 가동 및 데이터 이동 감소.

### 2. 미래 전망
앞으로는 이기종 코어 아키텍처(Intel Alder Lake 등)가 보편화됨에 따라, 단순한 '고정'을 넘어 작업의 성격(고성능 vs 저전력)에 맞춰 최적의 코어군을 자동으로 찾아가는 **지능형 친화성 스케줄링** 기술이 발전할 것이다. CPU 친화성은 하드웨어의 미세한 물리적 특성을 이해하고 이를 소프트웨어의 성능으로 승화시키는 운영체제 최적화의 정수이다.

- **📢 섹션 요약 비유**: 수만 조각의 퍼즐을 맞출 때, 내가 어제 맞추던 부분의 조각들이 모여 있는 책상을 떠나지 않고 계속 작업하여 완성 시간을 단축하는 영리한 전략입니다.

2. 내 자리에 앉으면 어제 쓰던 연필이랑 지우개가 책상 위에 그대로 있어서 바로 숙제를 시작할 수 있죠.
3. 자리를 자꾸 옮기면 물건을 다시 챙겨야 해서 늦어지니까, 한 자리에 콕 박혀서 열심히 일하는 거랍니다!
