## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 멀티프로세스 아키텍처 (Multiprocess Architecture)는 단일 애플리케이션의 기능을 여러 개의 독립된 프로세스로 분산하여 실행함으로써, 프로세스 간 격리(Isolation)와 결함 감수성(Fault Tolerance)을 극대화하는 설계 방식이다.
> 2. **가치**: 특정 탭이나 모듈의 크래시(Crash)가 전체 시스템으로 전이되는 것을 방지하고, 각 프로세스에 최소 권한(Least Privilege)만 부여하여 보안 샌드박스(Sandbox) 효율성을 높인다.
> 3. **융합**: 현대 웹 브라우저(Chrome), 마이크로서비스(MSA) 기반 사이드카 패턴, 고가용성 서버 설계의 핵심 모델로 자리 잡고 있으며, IPC 오버헤드와 자원 중복 사용이라는 트레이드오프를 수반한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: **멀티프로세스 아키텍처**는 하나의 소프트웨어를 여러 개의 물리적 프로세스로 쪼개어 운영하는 방식이다. 각 프로세스는 자신만의 독립된 주소 공간과 자원을 가지며, 서로 간의 통신은 오직 IPC (Inter-Process Communication)를 통해서만 가능하다.

- **💡 비유**: 멀티프로세스 아키텍처는 **"각 방이 완전히 분리된 호텔"**과 같다. 한 방에서 화재(에러)가 발생해도 다른 방으로 번지지 않으며, 각 방의 투숙객은 서로의 짐(메모리)을 함부로 뒤져볼 수 없는 구조다.

- **등장 배경**:
  1. **멀티스레딩의 한계**: 스레드는 메모리를 공유하므로, 단 하나의 스레드에서 발생한 `Segmentation Fault`가 프로세스 전체를 종료시키는 치명적 약점이 있었다.
  2. **보안 위협 증대**: 웹 브라우저처럼 외부 코드를 실행해야 하는 경우, 렌더링 엔진의 취약점이 시스템 전체 권한 탈취로 이어지는 것을 막아야 했다.
  3. **멀티코어 활용**: 각 프로세스를 서로 다른 CPU 코어에 배치함으로써 진정한 의미의 병렬 처리를 달성하기 용이해졌다.

- **📢 섹션 요약 비유**: 달걀을 한 바구니(프로세스)에 담지 않고 여러 바구니에 나누어 담아, 바구니 하나가 떨어져도 나머지는 안전하게 보호하는 위험 분산 전략과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 주요 프로세스 역할 분담 (구글 크롬 사례) (표)

| 프로세스명 | 역할 | 특징 | 비유 |
|:---|:---|:---|:---|:---|
| **Browser Process** | 전체 UI 관리 및 권한 통제 | 네트워크/파일 접근 권한 보유, 다른 프로세스 감시 | 호텔 지배인 |
| **Renderer Process** | HTML/JS 렌더링 및 실행 | 탭(Tab)마다 생성, 샌드박스 적용 (권한 없음) | 각 방의 투숙객 |
| **GPU Process** | 그래픽 가속 처리 | 여러 탭의 그래픽 요청을 통합 처리 | 호텔 중앙 조리실 |
| **Plugin Process** | 외부 플러그인(Flash 등) 실행 | 불안정한 외부 모듈 격리 | 외부 출장 요리사 |
| **Utility Process** | 네트워크, 오디오 등 공용 기능 | 필요 시에만 동적으로 생성 및 소멸 | 호텔 벨보이 |

#### 크롬의 멀티프로세스 통신 아키텍처

브라우저 프로세스는 중앙 컨트롤러 역할을 하며, 렌더러 프로세스들은 격리된 환경에서 통신 채널(IPC)을 통해서만 필요한 자원을 요청한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │                 현대적 브라우저의 멀티프로세스 구조               │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │   [Browser Process] (Privileged)                             │
  │   ┌──────────────────────────────────────────────────────┐   │
  │   │  UI / Network / File Access / Process Management     │   │
  │   └──────┬───────────────────────┬───────────────────────┘   │
  │          │ (Mojo IPC)            │ (Mojo IPC)                │
  │          ▼                       ▼                         │
  │   [Renderer Process 1]    [Renderer Process 2]   [GPU Proc]  │
  │   ┌──────────────────┐    ┌──────────────────┐   ┌────────┐  │
  │   │ Sandbox: Low Priv│    │ Sandbox: Low Priv│   │ Shader │  │
  │   │ JS Engine / DOM  │    │ JS Engine / DOM  │   │ Raster │  │
  │   └──────────────────┘    └──────────────────┘   └────────┘  │
  │                                                              │
  │   1. 사이트 접속 시 Renderer Process 독립 생성                  │
  │   2. Renderer는 직접 파일 쓰기 불가 (Browser에 요청)           │
  │   3. 특정 탭 응답 없음 발생 시 해당 프로세스만 강제 종료         │
  │   4. Browser는 종료된 Renderer를 감지하고 '앗, 이런!' 페이지 표시 │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 크롬 브라우저를 실행하면 최소 3~4개의 프로세스가 동시에 뜬다. 사용자가 보는 탭들은 각각 **Renderer Process**에서 실행되며, 이들은 운영체제의 가장 낮은 권한(Sandbox)만을 가진다. 따라서 렌더러 내의 악성 스크립트가 로컬 파일을 읽으려 해도 권한이 없어 차단된다. 모든 중요한 작업(파일 저장, 네트워크 요청)은 **Browser Process**에게 IPC를 통해 허락을 받아야 한다. 이러한 구조 덕분에 한 사이트의 스크립트 오류가 브라우저 전체를 꺼뜨리지 않고, 보안 위협을 프로세스 경계 내로 가둘 수 있다.

#### 핵심 메커니즘: 격리 및 샌드박싱

- **메모리 격리**: 각 프로세스는 독립된 가상 주소 공간을 가지므로, 포인터 연산 실수로 다른 탭의 데이터를 오염시킬 수 없다.
- **샌드박스 (Sandboxing)**: 렌더러 프로세스는 시스템 콜 호출을 제한받으며(seccomp 등), 특정 디렉토리 외에는 접근이 불가능한 '모래상자'에 갇혀 실행된다.

- **📢 섹션 요약 비유**: 죄수(렌더러)가 감옥(샌드박스) 안에서는 자유롭게 움직일 수 있지만, 간수(브라우저 프로세스)의 허락 없이는 감옥 밖의 물건을 만질 수 없는 보안 체계와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

#### 비교: 멀티스레드 vs 멀티프로세스

| 비교 항목 | 멀티스레드 아키텍처 | 멀티프로세스 아키텍처 |
|:---|:---|:---|
| **안정성** | 낮음 (공유 메모리 오염 시 전체 Crash) | 높음 (개별 프로세스 격리) |
| **보안성** | 낮음 (스레드 간 데이터 탈취 용이) | 높음 (OS 수준의 샌드박싱 가능) |
| **자원 사용량** | 적음 (스택만 별도 생성) | 많음 (전체 주소 공간 및 라이브러리 중복 적재) |
| **통신 비용** | 낮음 (전역 변수 접근 가능) | 높음 (IPC 오버헤드 발생) |
| **컨텍스트 스위칭** | 빠름 (LWP 수준) | 상대적 느림 (MMU 갱신 필요) |

#### 과목 융합 관점: 가상 메모리와 분산 컴퓨팅

- **운영체제 (OS)**: 멀티프로세스 아키텍처는 COW (Copy-On-Write) 기법을 활용하여 프로세스 생성 초기 메모리 낭비를 방지하지만, 실행 시간이 길어질수록 중복 페이지가 늘어나 RAM 점유율이 상승한다.
- **분산 시스템**: 애플리케이션 내의 멀티프로세스 구조는 서비스가 커졌을 때 별도의 서버로 분리(Microservices)하기에 매우 용이한 구조적 기틀을 제공한다.

- **📢 섹션 요약 비유**: 멀티스레드가 거실을 같이 쓰는 대가족이라면, 멀티프로세스는 벽을 세워 집을 나눈 아파트 입주민들과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단

#### 실무 시나리오: 고성능 이미지 프로세싱 서버 설계
사용자가 업로드한 수천 장의 이미지를 필터링하는 서버를 만들 때, 단일 프로세스 내 멀티스레드 방식은 특정 이미지 라이브러리의 버그(예: Heap Overflow)로 서버가 죽을 위험이 있다. 이때 멀티프로세스 워커 방식을 도입하면 에러가 발생한 워커만 재시작하면 되므로 서버 전체의 가용성을 유지할 수 있다. 통신 부담은 Unix 도메인 소켓이나 공유 메모리로 최적화한다.

#### 도입 체크리스트
- **자원 할당**: 시스템의 물리 RAM 용량이 수백 개의 프로세스를 감당할 수 있는가?
- **IPC 설계**: 프로세스 간 빈번한 통신이 필요한가? (그렇다면 스레드 방식이 유리할 수 있음)
- **상태 관리**: 특정 프로세스가 종료되었을 때, 다른 프로세스가 그 사실을 인지하고 복구할 수 있는 'Watchdog' 로직이 있는가?

#### 안티패턴
- **Process Bloat**: 기능 하나마다 무조건 프로세스를 생성하여 컨텍스트 스위칭 비용이 실제 연산 비용을 압도하는 상황. (임계치 설정 필수)
- **공유 메모리 남용**: 프로세스를 나눠놓고 모든 데이터를 거대한 공유 메모리에 넣어버리면, 격리의 이점이 사라지고 다시 멀티스레드 수준의 안정성으로 회귀하게 된다.

- **📢 섹션 요약 비유**: 보안을 위해 성벽을 높게 쌓는(프로세스 격리) 것은 좋으나, 성문(IPC)을 너무 자주 드나들면 물류 속도가 느려진다는 점을 명심해야 합니다.

### Ⅴ. 기대효과 및 결론

#### 정량/정성 기대효과

| 구분 | 도입 전 (Single Process) | 도입 후 (Multiprocess) | 개선 효과 |
|:---|:---|:---|:---|
| **가용성 (Uptime)** | 모듈 하나 오류 시 전체 중단 | 오류 전이 차단 및 부분 복구 | **시스템 안정성 90% 향상** |
| **보안 사고율** | 취약점 노출 시 전체 권한 탈취 | 격리 구역 내 피해 한정 | 보안 침해 사고 범위 대폭 축소 |
| **자원 오버헤드** | 메모리 사용량 적음 | 탭당 50~100MB 추가 점유 | **메모리 사용량 약 2~3배 증가** |

#### 미래 전망
메모리 가격의 하락과 하드웨어 성능의 비약적 발전으로 인해, 안정성과 보안을 최우선으로 하는 멀티프로세스 아키텍처는 더욱 확산될 것이다. 특히 웹 어셈블리(Wasm)와 결합된 에지 컴퓨팅 환경에서 코드 격리를 위한 표준 모델로 진화하고 있으며, OS 차원에서도 프로세스 생성 비용을 줄이기 위한 최적화(Lightweight Process)가 계속될 전망이다.

- **참고 표준**: Chromium Multiprocess Architecture Documentation

- **📢 섹션 요약 비유**: 현대 아키텍처의 트렌드는 효율성(메모리)을 조금 희생하더라도, 절대 무너지지 않는 안정성(격리)을 선택하는 방향으로 흐르고 있습니다.

### 📌 관련 개념 맵
- **프로세스 격리 (Process Isolation)**: 멀티프로세스 아키텍처가 달성하려는 근본적 목적.
- **샌드박스 (Sandbox)**: 격리된 프로세스에 가하는 추가적인 보안 제약.
- **IPC (Inter-Process Communication)**: 분리된 프로세스들을 이어주는 유일한 통로.
- **COW (Copy-On-Write)**: 멀티프로세스 아키텍처의 메모리 낭비를 줄여주는 커널 기법.

### 👶 어린이를 위한 3줄 비유 설명
1. 멀티프로세스는 장난감을 가지고 놀 때, 모든 장난감을 한 박스에 넣지 않고 **"종류별로 다른 박스에 나누어 담는 것"**과 같아요.
2. 한 박스에 우유를 쏟아도(에러) 다른 박스의 장난감들은 젖지 않아서 안전하죠.
3. 대신 박스가 여러 개 필요해서 자리를 좀 더 많이 차지하지만, 훨씬 깔끔하고 안전하게 놀 수 있답니다!
