# 프로세스 생성 (Process Creation) - fork(), exec()

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 생성은 실행 중인 프로세스(부모)가 운영체제에 시스템 콜을 요청하여 **자신과 동일하거나 새로운 작업을 수행하는 새로운 프로세스(자식)를 만드는 과정**이다.
> 2. **가치**: `fork()`를 통해 부모의 문맥을 그대로 복제하고, `exec()`를 통해 새로운 프로그램을 덮어씌움으로써 **유연한 다중 작업 구조**를 형성하며, 프로세스 간의 계층 구조(Tree)를 유지한다.
> 3. **융합**: 자원 할당, 주소 공간 설정, 초기화 등 커널의 복합적인 관리가 수반되며, 현대 OS에서는 효율성을 위해 COW(Copy-on-Write) 기법과 결합하여 동작한다.

+++

## Ⅰ. 프로세스 생성의 기본 원리

### 1. 계층 구조 (Process Tree)
- 프로세스는 실행 도중 다른 프로세스를 생성할 수 있다.
- **부모 프로세스 (Parent Process)**: 생성자.
- **자식 프로세스 (Child Process)**: 피생성자.
- 시스템 전체적으로는 root(리눅스의 init/systemd)를 정점으로 하는 거대한 트리 구조를 형성한다.

### 2. 자원 할당 방식
- **공유 방식**: 부모의 모든 자원을 자식과 공유한다.
- **부분 공유**: 자식에게 부모 자원의 일부만 할당한다.
- **독립 방식**: 부모와 자식이 자원을 공유하지 않고 독립적으로 운영된다.

### 3. 💡 비유: '세포 분열과 이직'
- `fork()`는 **'세포 분열'**과 같다. 나와 똑같은 유전 정보(데이터, 코드)를 가진 분신을 하나 더 만드는 것이다.
- `exec()`는 **'이직'**과 같다. 분신을 만든 뒤, 그 분신에게 "너는 이제부터 다른 일(새로운 프로그램)을 해!"라고 지시하며 완전히 새로운 사람으로 변신시키는 것과 같다.

+++

## II. 핵심 시스템 콜: fork()와 exec() (Deep Dive)

유닉스/리눅스 계열에서의 표준적인 프로세스 생성 2단계 모델이다.

### 1. fork() 시스템 콜
- **동작**: 부모 프로세스의 주소 공간과 문맥을 **그대로 복사**하여 새로운 자식 프로세스를 생성한다.
- **반환값 (Return Value)**: 
  - 부모에게는 자식의 **PID**를 반환.
  - 자식에게는 **0**을 반환. (이를 통해 부모와 자식을 코드상에서 구분 가능)

### 2. exec() 시스템 콜
- **동작**: 자식 프로세스의 메모리 공간을 **새로운 바이너리 프로그램으로 덮어씌운다.**
- **결과**: 이전의 데이터와 코드는 모두 사라지고, 새로운 프로그램의 `main()`부터 실행이 시작된다.

### 3. ASCII 구조 다이어그램
```text
    [ Parent Process (PID: 100) ]
             │
             ▼ fork() 요청
    ┌───────────────────────────┐
    │ [ Parent ]                │ ──▶ [ Child (PID: 101) ] (Copy of Parent)
    │ if (pid > 0) {            │     │ if (pid == 0) {
    │   wait(); // wait child   │     │   exec("new_prog"); ──▶ [ Totally New Program ]
    │ }                         │     │ }
    └───────────────────────────┘     └────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 부모가 자신과 똑같은 복제 인간(fork)을 만든 뒤, 그 복제 인간에게 새로운 교복을 입히고 전학(exec)을 보내는 과정과 같습니다.

## Ⅲ. 생성 시 자원 및 실행 관리

| 구분 | 내용 | 비고 |
|:---|:---|:---|
| **주소 공간** | 자식은 부모의 주소 공간 복사본을 가짐 | exec() 호출 전까지 동일 내용 유지 |
| **실행 방식** | 부모와 자식이 병행 실행되거나, 부모가 대기함 | wait() 시스템 콜로 동기화 |
| **자원 상속** | 열린 파일, 환경 변수 등 대부분 상속 | PID, 실행 시간 등은 고유함 |

- **📢 섹션 요약 비조**: "부모의 재산을 물려받아(상속) 독립적인 인생을 시작하되, 상황에 따라 부모가 자식의 성장을 지켜보며 기다려주는(Wait) 질서 있는 독립 과정입니다."

+++

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 쉘(Shell)의 동작 원리
- **상황**: 사용자가 터미널에서 `ls` 명령어를 입력함.
- **판단**: 쉘(부모)은 **fork()**를 호출하여 자신과 똑같은 자식을 만든다. 그 후 자식은 **exec("ls")**를 수행하여 자신의 몸을 `ls` 프로그램으로 바꾼다. 부모 쉘은 자식이 끝날 때까지 **wait()**를 호출하며 대기한다. 기술사라면 이 과정이 운영체제의 유연성과 안정성을 동시에 잡는 고전적인 설계임을 설명해야 한다.

### 2. 기술사적 인사이트: fork()의 오버헤드와 COW
- 기술사라면 `fork()` 시 발생하는 메모리 복사 오버헤드를 지적해야 한다. 수 GB의 부모 프로세스를 그대로 복사하는 것은 비효율적이다. 이를 해결하기 위해 현대 OS는 **COW (Copy-on-Write)** 기술을 사용하여, 실제로 데이터를 쓰기(Write) 전까지는 페이지를 공유하다가 변경이 일어날 때만 물리 복사를 수행하는 지능형 최적화 전략을 취함을 강조해야 한다.

- **📢 섹션 요약 비조**: "물리적인 복사(Heavy Copy)보다 논리적인 공유(Lazy Copy)가 현대 운영체제 성능의 핵심 열쇠입니다."

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **병렬성 확보**: 여러 프로세스를 생성하여 다중 작업 수행.
- **안정성 격리**: 자식 프로세스가 비정상 종료되어도 부모에게 미치는 영향 최소화.

### 2. 미래 전망
앞으로는 컨테이너 기술의 발전에 따라 프로세스 생성보다 더 가벼운 **격리된 프로세스 생성(Namespace/Cgroup 기반)** 기술이 더욱 고도화될 것이다. 또한, 서버리스 환경에서는 밀리초 단위로 프로세스를 생성하고 소멸시키는 **Fast-Boot 프로세스** 아키텍처가 주류가 될 것이다. 프로세스 생성은 운영체제가 생태계의 다양성과 생명력을 유지하는 가장 근본적인 창조의 과정이다.

- **📢 섹션 요약 비유**: 수많은 별(프로세스)이 탄생하고 사라지는 우주처럼, 운영체제는 생성과 소멸의 법칙(fork/exec)을 통해 시스템의 거대한 질서를 유지합니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 프로세스 생성은 컴퓨터가 **'동생을 만드는 일'**과 같아요.
2. `fork`라는 마법을 쓰면 나랑 똑같이 생긴 동생이 생기고, `exec`를 쓰면 동생에게 새로운 옷을 입혀서 다른 일을 시킬 수 있죠.
3. 형(부모)이랑 동생(자식)이 힘을 합쳐서 일을 하면, 복잡한 문제도 훨씬 빨리 풀 수 있답니다!
