# 시스템 V IPC - 공유 메모리, 세마포어, 메시지 큐

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 V IPC는 유닉스 시스템 V(AT&T)에서 시작된 고전적이고 강력한 **프로세스 간 통신 규격**으로, 공유 메모리, 세마포어, 메시지 큐라는 세 가지 핵심 매커니즘을 제공한다.
> 2. **가치**: 커널 수준에서 전역적으로 관리되는 **IPC 식별자(Key)**를 통해 서로 부모-자식 관계가 아닌 독립된 프로세스들 간에도 정교한 데이터 교환과 동기화를 가능케 한다.
> 3. **융합**: 고속 데이터 전송(공유 메모리), 실행 순서 제어(세마포어), 비동기 메시지 전달(메시지 큐)을 유기적으로 결합하여 복잡한 다중 프로세스 시스템의 중추적 역할을 수행한다.

---

## Ⅰ. 시스템 V IPC의 공통 특징

세 가지 매커니즘은 모두 다음의 공통적인 속성을 공유한다.

- **IPC 키 (Key)**: 프로세스들이 동일한 IPC 자원에 접근하기 위해 사용하는 외부 식별자이다. (주로 `ftok()` 함수로 생성)
- **식별자 (ID)**: 키를 통해 할당받은 커널 내부의 실제 자원 번호이다.
- **지속성 (Persistence)**: 프로세스가 종료되어도 명시적으로 삭제하거나 시스템을 재부팅하기 전까지 커널 내에 유지된다.

---

## II. 시스템 V IPC의 3대 요소 (Deep Dive)

### 1. 공유 메모리 (Shared Memory)
- **개념**: 커널이 관리하는 물리 메모리 영역을 여러 프로세스의 주소 공간에 매핑한다.
- **특징**: 데이터 복사 오버헤드가 없어 **가장 빠른 IPC** 방식이다.
- **주의**: 데이터 충돌을 막기 위해 반드시 '세마포어'와 같은 동기화 장치가 수반되어야 한다.

### 2. 세마포어 (Semaphores)
- **개념**: 여러 프로세스가 공유 자원에 접근할 때 순서를 제어하기 위한 카운터(Signaling) 장치이다.
- **특징**: 단순히 데이터를 보내는 용도가 아니라, **동기화(Synchronization)**를 위한 도구이다. 시스템 V 세마포어는 여러 개의 세마포어를 하나의 '집합'으로 관리할 수 있다.

### 3. 메시지 큐 (Message Queues)
- **개념**: 커널 내부에 메시지들을 연결 리스트 형태로 유지하는 큐이다.
- **특징**: 메시지 타입(Type)을 지정하여 특정 종류의 메시지만 골라 읽을 수 있는 **비동기적** 통신을 지원한다.

### 4. ASCII 구조 다이어그램
```text
    [ Process A ]       [ Process B ]       [ Process C ]
          │                   │                   │
    ==========================▼===================▼================
    [ Operating System Kernel ]
    ┌─────────────────────────────────────────────────────────────┐
    │  System V IPC Object Table                                  │
    │  ---------------------------------------------------------  │
    │  [ Msg Queue ID: 10 ] <── Message A, Message B...           │
    │  [ Shared Mem ID: 20] <── [ Data Block ] ◀──(Direct Map)    │
    │  [ Semaphore  ID: 30] <── [ Counter: 1 ]                    │
    └─────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 시스템 V IPC는 **'정부(커널)가 관리하는 공용 시설'**과 같다. 게시판(메시지 큐)에 글을 남기거나, 공용 작업실(공유 메모리)을 빌려 쓰거나, 작업실 출입증(세마포어)을 받아 질서를 유지하는 거대한 사회적 약속 체계이다.

---

## Ⅲ. IPC 요소별 비교 분석

| 비교 항목 | 공유 메모리 | 메시지 큐 | 세마포어 |
|:---|:---|:---|:---|
| **주요 목적** | 고속 데이터 공유 | 데이터 전달 (비동기) | 프로세스 동기화 |
| **복사 횟수** | 0회 (매핑 방식) | 2회 (유저↔커널) | N/A (상태값만 변경) |
| **단위** | 바이트/페이지 | 메시지 구조체 | 정수형 카운터 |
| **장점** | 압도적인 속도 | 쉬운 동기화, 타입 필터링 | 자원 관리의 정밀성 |

- **📢 섹션 요약 비조**: "속도가 생명인 큰 짐은 공유 마당(메모리)에 두고, 자잘한 소식은 우편함(큐)에 넣으며, 출입 통제는 열쇠(세마포어)로 하는 종합 물류 시스템입니다."

---

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: DB 엔진과 리스너 통신
- **상황**: 오라클(Oracle) 같은 거대 DB 엔진이 수백 개의 리스너 프로세스와 데이터를 주고받아야 함.
- **판단**: **시스템 V 공유 메모리**를 대규모로 할당(SGA 영역)하여 모든 프로세스가 접근하게 하고, 각 트랜잭션의 정합성은 **세마포어** 집합을 통해 제어한다. 이는 성능과 데이터 무결성을 동시에 잡는 엔터프라이즈급 소프트웨어의 표준 설계 패턴이다.

### 2. 기술사적 인사이트: POSIX IPC와의 선택 기준
- 기술사라면 시스템 V IPC와 현대적인 **POSIX IPC**의 차이를 논해야 한다. 시스템 V는 인터페이스가 복잡하고 오래되었으나, 세마포어 집합 처리 등 기능이 더 강력한 면이 있다. 반면 POSIX는 사용법이 직관적이고 스레드 안전성이 뛰어나다. 기술사라면 "레거시 시스템과의 호환성이나 복합 동기화가 필요할 때는 시스템 V를, 신규 모던 앱 설계 시에는 POSIX를 지향해야 함"을 제언해야 한다.

- **📢 섹션 요약 비조**: "오래된 도구(System V)는 다루기 까다롭지만 그만큼 깊은 내공을 발휘하며, 현대의 도구(POSIX)는 세련되고 민첩한 협업을 가능케 합니다."

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **다중 프로세스 협업 완성**: 독립된 앱들이 하나의 거대한 서비스처럼 동작하게 하는 결합력 제공.
- **시스템 자원 중앙 통제**: 커널이 IPC 자원을 관리함으로써 비정상 프로세스 종료 시에도 자원 회수 가능성 확보.

### 2. 미래 전망
앞으로는 컨테이너 격리 기술(Namespace)이 발전함에 따라, 호스트 전체가 아닌 특정 컨테이너 그룹 내에서만 유효한 **IPC 네임스페이스** 기반의 시스템 V 활용이 주류가 될 것이다. 또한, 고속 비휘발성 메모리(Pmem)와 결합하여 재부팅 후에도 유지되는 '영구 IPC' 기술이 시스템 아키텍처의 새로운 지평을 열 것이다. 시스템 V IPC는 유닉스의 역사를 지탱해온 가장 묵직하고 신뢰받는 통신 인프라이다.

- **📢 섹션 요약 비유**: 수백 명의 일꾼이 각자의 방에서 일하면서도, 중앙의 큰 도서관(IPC)을 통해 완벽하게 정보를 공유하며 하나의 거대 성을 쌓는 것과 같습니다.
