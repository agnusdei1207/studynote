# 간접 통신 (Indirect Communication) - 메일박스, 포트

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 간접 통신 (Indirect Communication)은 송신자와 수신자가 직접 이름을 부르지 않고, **메일박스 (Mailbox)** 또는 **포트 (Port)**라 불리는 중간 공유 객체를 통해 메시지를 주고받는 IPC 방식이다.
> 2. **가치**: 프로세스 간의 이름을 몰라도 통신이 가능하게 함으로써 **낮은 결합도 (Loose Coupling)**를 실현하며, 하나의 메일박스에 여러 송수신자가 참여하는 다대다(N:M) 통신 구조를 지원한다.
> 3. **융합**: 운영체제 커널은 메일박스의 생성, 삭제, 메시지 큐 관리를 담당하며, 이는 현대 메시지 브로커(Kafka, RabbitMQ 등) 아키텍처의 논리적 뿌리가 된다.

+++

## Ⅰ. 간접 통신의 개요

### 1. 정의
- 메시지를 보낼 장소(메일박스)와 메시지를 꺼낼 장소가 동일한 추상적 객체를 통해 통신하는 방식이다.
- 예: `send(A, message)` (메일박스 A로 전송), `receive(A, message)` (메일박스 A로부터 수신).

### 2. 주요 특징
- **링크 조건**: 두 프로세스가 **공통의 메일박스**를 소유하고 있어야 통신 링크가 형성된다.
- **다중 연결**: 하나의 링크가 여러 프로세스와 연결될 수 있으며, 한 쌍의 프로세스 사이에 여러 개의 메일박스(링크)가 존재할 수도 있다.
- **소유권**: 메일박스는 특정 프로세스가 소유하거나 운영체제 커널이 소유할 수 있다.

### 3. 💡 비유: '아파트 공용 무인 택배함'
- 간접 통신은 **'받는 사람의 얼굴을 몰라도 동호수(메일박스 ID)만 알면 물건을 넣어두는 택배함'**과 같다.
- 내가 택배함에 넣어두면(Send), 물건이 필요한 수신자가 나중에 와서 가져간다(Receive). 택배 기사(송신자)는 집주인(수신자)이 누구인지, 지금 집에 있는지 알 필요가 없다.

+++

## Ⅱ. 메일박스의 동작 및 통신 관계 (Deep Dive)

### 1. 통신 관계의 유형
- **1:1**: 한 송신자와 한 수신자만 연결 (단순 채널).
- **1:N**: 한 송신자가 보낸 메시지를 여러 수신자 중 하나가 가져감 (부하 분산).
- **N:1**: 여러 송신자가 하나의 수신자(서버)에게 메시지를 보냄 (서비스 요청).
- **N:M**: 여러 송신자와 여러 수신자가 하나의 메일박스를 공유 (클러스터 협업).

### 2. 메시지 수신 결정 (Selection)
- 메일박스에 메시지가 있을 때 누가 가져갈 것인가?
  - **선착순**: 가장 먼저 `receive`를 호출한 프로세스.
  - **라운드 로빈**: 운영체제가 순서대로 지정.
  - **명시적 지정**: 송신자가 특정 프로세스만 가져가도록 태그 부착.

### 3. ASCII 구조 다이어그램
```text
    [ Process A ]       [ Process B ]       [ Process C ]
          │                   │                   │
          └─────────┬─────────┴─────────┬─────────┘
                    ▼ (send)            ▼ (receive)
          ┌───────────────────────────────────────┐
          │         Mailbox / Port (ID: 1004)     │
          │  ┌─────────┐  ┌─────────┐  ┌─────────┐│ ◀── Kernel Managed
          │  │ Msg 1   │  │ Msg 2   │  │ Msg 3   ││     Queue
          │  └─────────┘  └─────────┘  └─────────┘│
          └───────────────────────────────────────┘
                    ▲ (receive)         ▲ (send)
          ┌─────────┴─────────┐         └─────────┐
          │                   │                   │
    [ Process D ]       [ Process E ]       [ Process F ]
```

- **📢 섹션 요약 비유**: 직접 대화하는 무전기(직접 통신)에서, 모두가 글을 남기고 읽을 수 있는 게시판(간접 통신)으로 소통 방식이 진화한 것과 같습니다.

## Ⅲ. 직접 통신 vs 간접 통신 비교 분석

| 비교 항목 | 직접 통신 (Direct) | 간접 통신 (Indirect) |
|:---|:---|:---|
| **이름 명시** | 상대 프로세스 이름 필수 | **메일박스 ID 필수** |
| **결합도** | 강함 (Tight) | **약함 (Loose)** |
| **통신 상대** | 고정됨 (1:1) | **유연함 (1:N, N:M 가능)** |
| **관리 오버헤드** | 적음 (별도 객체 없음) | 보통 (메일박스 생성/삭제 필요) |

- **📢 섹션 요약 비유**: 번호를 아는 사람에게만 전화하는 것(직접)과, 카카오톡 오픈채팅방(간접)에 들어가 대화하는 것의 자유도 차이입니다.

+++

## Ⅳ. 실무 적용 및 운영 전략 (Strategy)

### 1. 실무 시나리오: 작업 큐 (Job Queue) 시스템
- **상황**: 웹 서버로 들어오는 대량의 이미지 처리 요청을 여러 워커(Worker) 프로세스가 나누어 처리해야 함.
- **판단**: 웹 서버와 워커들을 직접 연결하지 않고, 중간에 **메일박스(작업 큐)**를 둔다. 웹 서버는 요청을 큐에 던지기만 하고, 노는 워커들이 알아서 하나씩 꺼내 간다. 워커를 추가하거나 제거해도 웹 서버 코드를 수정할 필요가 없는 완벽한 간접 통신 아키텍처다.

### 2. 기술사적 인사이트: 모듈성과 확장성 (Scalability)
- 현대 대규모 시스템 설계의 핵심은 **디커플링 (Decoupling)**이다. 기술사라면 간접 통신이 시스템의 구성 요소들을 독립적으로 진화하게 만든다는 점을 강조해야 한다. 송신자는 수신자의 존재 여부나 개수에 상관없이 메시지를 발행(Publish)하고, 수신자는 필요에 따라 구독(Subscribe)하는 이 구조가 현대 MSA의 근간임을 제시해야 한다.

- **📢 섹션 요약 비유**: 부서 간 업무 협조를 위해 담당자를 일일이 찾아다니는 대신, 그룹웨어 협업 게시판을 만드는 것이 회사의 확장성을 높이는 비결입니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **유연한 시스템 구성**: 프로세스 식별 정보 변경에 유연하게 대처 가능.
- **부하 분산 효율화**: 여러 프로세스가 하나의 통로를 공유하며 업무 처리량 조절 가능.

### 2. 미래 전망
앞으로는 클라우드 메시징 서비스가 운영체제의 간접 통신 기능을 완전히 추상화하여 제공할 것이다. 하지만 커널 수준에서의 고속 포트 관리와 메일박스 정합성 유지 기술은 지연 시간에 민감한 실시간 시스템이나 에지 컴퓨팅 환경에서 여전히 독보적인 성능 지표로 남을 것이다.

- **📢 섹션 요약 비유**: 서로 몰라도 통하는 마법, 운영체제가 보장하는 소통의 '공유 경제' 모델입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 간접 통신은 친구에게 선물을 줄 때 **'학교에 있는 공용 보관함에 넣어두는 것'**과 같아요.
2. 친구의 이름을 몰라도 보관함 번호만 알면 선물을 줄 수 있고, 친구도 아무 때나 와서 찾아갈 수 있죠.
3. 보관함 하나를 여러 친구가 같이 쓸 수 있어서, "누가 먼저 가져갈래?"라고 정해두면 싸우지 않고 사이좋게 나눠 가질 수 있답니다!
