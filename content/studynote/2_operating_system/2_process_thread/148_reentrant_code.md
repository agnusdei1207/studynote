## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 재진입 가능 코드 (Reentrant Code)는 실행 도중 중단되고 다시 호출(재진입)되어도 실행 결과의 정확성이 보장되는 코드로, 주로 전역/정적 변수를 사용하지 않고 로컬 스택 자원만 사용하는 구조를 가진다.
> 2. **가치**: 인터럽트 핸들러(ISR)나 신호 처리기(Signal Handler)에서 안전하게 호출될 수 있으며, 동기화 기구(Mutex 등) 없이도 멀티스레드 환경에서 안전성을 확보하여 데드락 위험을 원천 차단한다.
> 3. **융합**: 운영체제 커널의 핵심 루틴, 임베디드 시스템의 인터럽트 처리, 그리고 함수형 프로그래밍의 순수 함수(Pure Function) 개념과 맞닿아 있는 고신뢰성 소프트웨어의 기반 기술이다.

---

### Ⅰ. 개요 (Context & Background)

- **概念**: **재진입 가능성 (Reentrancy)**은 코드가 동시에 여러 번 호출되거나, 한 번의 실행이 끝나기 전에 다시 호출되어도 각 호출이 서로 독립적으로 올바르게 수행되는 성질이다. 이는 코드 내부에 '상태(State)'를 저장하지 않거나, 상태를 스택(Stack)과 같은 호출자 전용 공간에 격리함으로써 달성된다.

- **💡 비유**: 재진입 가능 코드는 **"종이 쪽지에 적힌 요리 레시피"**와 같다. 여러 요리사가 같은 레시피를 보고 동시에 요리를 해도, 각자 자기만의 냄비와 재료(로컬 변수)를 사용한다면 서로의 요리에 영향을 주지 않고 완성할 수 있는 것과 유사하다. 반면 재진입 불가능한 코드는 모든 요리사가 하나의 냄비(전역 변수)를 공유하여 요리하는 상황에 비유할 수 있다.

- **등장 배경**:
  1. **인터럽트 기반 시스템**: 실행 중인 메인 루틴이 인터럽트에 의해 중단되고, 인터럽트 핸들러가 다시 메인 루틴이 사용하던 함수를 호출할 때 데이터가 파손되는 문제가 발생했다.
  2. **신호(Signal) 처리**: 리눅스/유닉스 환경에서 비동기적으로 발생하는 신호를 처리할 때, 안전하게 호출할 수 있는 함수(Async-signal-safe)의 기준이 필요해졌다.
  3. **재귀 호출(Recursion)**: 함수가 자기 자신을 호출할 때 이전 호출의 데이터가 유지되어야 하는 알고리즘 구현을 위해 재진입성은 필수적이었다.

- **📢 섹션 요약 비유**: 중간에 멈추고 다른 일을 하다가 돌아와도, 혹은 다른 사람이 동시에 사용해도 항상 처음 의도한 결과를 내놓는 '독립적이고 견고한 설계'입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 재진입 가능 코드의 3대 조건 (표)

| 조건 | 세부 내용 | 위반 사례 | 해결 방법 | 비유 |
|:---|:---|:---|:---|:---|
| **상태 미유지** | 전역(Global)/정적(Static) 변수 사용 금지 | `static int count` | 매개변수나 스택 변수 활용 | 공용 장부 대신 개인 수첩 |
| **비동기 안전성** | 재진입 불가능한 함수 호출 금지 | `malloc()`, `printf()` 호출 | 재진입 버전(`_r`) 함수 사용 | 믿을 수 없는 조수 해고 |
| **자기 수정 금지** | 코드 본문을 실행 중에 수정하지 않음 | Self-modifying code | 코드를 Read-only 메모리에 배치 | 레시피 수정 금지 |

#### 재진입 가능 vs 불가능 코드 구조 비교

공유 자원(정적 변수)의 사용 여부가 재진입성을 결정짓는 핵심 차이다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │                 재진입 가능성과 실행 흐름의 격리                    │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │   [Non-Reentrant: static 사용]      [Reentrant: stack 사용]    │
  │   int swap(int *a, int *b) {      int swap_r(int *a, int *b) {│
  │       static int temp; <── ⚠          int temp; <── ✅        │
  │       temp = *a;                      temp = *a;              │
  │       /* Interrupt! */                /* Interrupt! */        │
  │       *a = *b;                        *a = *b;                │
  │       *b = temp;                      *b = temp;              │
  │   }                               }                           │
  │                                                              │
  │   1. Non-Reentrant: 인터럽트 시 temp 값이 덮어씌워져 데이터 오염    │
  │   2. Reentrant: 각 호출마다 스택에 temp가 개별 생성되어 안전함     │
  │   3. Key: "상태를 어디에 두는가?"가 재진입성의 핵심              │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 좌측의 `swap` 함수는 정적 변수 `temp`를 사용한다. 만약 이 함수 실행 도중 인터럽트가 발생하여 인터럽트 핸들러가 다시 `swap`을 호출하면, 기존 실행 중이던 `temp` 값이 새로운 호출에 의해 덮어씌워진다. 인터럽트 처리가 끝나고 원래 루틴으로 돌아오면 `temp` 값이 변해 있어 잘못된 결과가 도출된다. 반면 우측의 `swap_r` 함수는 지역 변수(스택)를 사용한다. 각 호출은 자신만의 스택 프레임을 가지므로, 인터럽트에 의해 수천 번 재진입되어도 서로의 `temp` 자원을 침범하지 않아 안전하다.

#### 심층 동작 원리: 스택 프레임(Stack Frame) 기반의 격리

재진입 가능 코드가 동작하는 근간은 CPU의 **스택 아키텍처**다.
1. **함수 호출**: 새로운 스택 프레임이 생성된다.
2. **로컬 데이터**: 모든 계산 결과와 중간 상태는 해당 스택 프레임 내의 로컬 변수에 저장된다.
3. **중단(Interrupt)**: 현재 CPU 레지스터(PC, SP 등)가 스택에 보존되고, 새로운 함수 실행을 위해 또 다른 스택 프레임이 쌓인다.
4. **복귀**: 새로운 함수가 끝나고 스택이 정리되면, 보존되었던 레지스터가 복구되어 이전 상태를 완벽히 재현한다.

- **📢 섹션 요약 비유**: 각 손님에게 개인별로 깨끗한 접시(스택 프레임)를 제공하여, 앞 손님이 먹던 음식과 섞이지 않게 하는 고급 뷔페 시스템과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

#### 비교: 재진입 가능(Reentrant) vs 스레드 안전(Thread-safe)

| 비교 항목 | 재진입 가능 (Reentrant) | 스레드 안전 (Thread-safe) |
|:---|:---|:---|
| **핵심 목표** | 중첩 호출/인터럽트 시 안전성 | 멀티스레드 동시 접근 시 안전성 |
| **동기화 기구** | 뮤텍스/락 사용 불가 (사용 시 데드락) | 뮤텍스/락 사용 권장 |
| **데이터 공유** | 공유 데이터 절대 금지 | 공유 데이터를 락으로 보호하여 사용 가능 |
| **포함 관계** | 모든 재진입 가능 코드는 스레드 안전함 | 모든 스레드 안전 코드가 재진입 가능한 것은 아님 |
| **비유** | 혼자서 쓰다 멈춰도 괜찮은 도구 | 여러 사람이 같이 써도 괜찮은 도구 |

#### 과목 융합 관점: 커널과 함수형 프로그래밍

- **운영체제 커널**: 커널 루틴은 실행 중 다른 인터럽트에 의해 언제든 중단될 수 있으므로, 대부분의 커널 함수는 재진입 가능하게 설계되어야 한다. 이를 통해 시스템의 안정성과 높은 응답성을 확보한다.
- **함수형 프로그래밍**: 함수형 언어의 '순수 함수'는 부수 효과(Side-effect)가 없고 외부 상태를 참조하지 않으므로, 정의상 완벽한 재진입 가능 코드다. 이는 병렬 처리가 중요한 현대 컴퓨팅에서 함수형 언어가 주목받는 이유이기도 하다.

- **📢 섹션 요약 비유**: 자물쇠(Mutex)를 채워 문을 잠그는 것이 '스레드 안전'이라면, 아예 누구나 자기 방에서 따로 일하게 하는 것이 '재진입 가능'입니다.

### Ⅳ. 실무 적용 및 기술사적 판단

#### 실무 시나리오: 비동기 신호 처리기(Signal Handler) 구현
리눅스 앱에서 `SIGINT` (Ctrl+C) 신호를 받았을 때 로그를 남기는 기능을 구현하려 한다. 이때 신호 처리기 내부에서 `printf()`를 호출하면 안 된다. `printf()`는 내부적으로 정적 버퍼를 사용하는 재진입 불가능한 함수이기 때문에, 메인 루틴이 `printf()`를 실행 중에 신호가 터지면 데드락이 발생하거나 출력이 깨질 수 있다. 대신 재진입이 보장되는 `write()` 시스템 콜을 사용하거나, 신호 발생 여부만 플래그로 기록하고 메인 루틴에서 처리하도록 설계해야 한다.

#### 도입 체크리스트
- **정적 변수 검사**: `static` 또는 전역 변수가 포함되어 있는가? (포함 시 재진입 불가능)
- **표준 라이브러리 확인**: 호출하는 라이브러리 함수가 `_r` 접미사가 붙은 재진입 버전인가? (예: `strtok_r`, `gethostbyname_r`)
- **락(Lock) 사용 여부**: 함수 내부에 뮤텍스 락이 있는가? (인터럽트 핸들러에서 호출 시 데드락 위험)

#### 안티패턴
- **재진입 불가능한 함수에서의 락 사용**: 인터럽트 핸들러가 락을 소유한 메인 루틴을 중단시키고 다시 그 락을 얻으려 하면 영원히 대기하는 데드락에 빠진다.
- **전역 버퍼 공유**: 여러 호출이 하나의 전역 버퍼를 공유하여 문자열을 조작하면, 결과값이 뒤섞여 쓰레기 값이 출력된다.

- **📢 섹션 요약 비유**: 전화 통화를 하다가 중간에 끊고 다른 전화를 받았는데, 두 전화의 대화 내용이 서로 섞여버리는 혼선 사고를 방지하는 설계와 같습니다.

### Ⅴ. 기대효과 및 결론

#### 정량/정성 기대효과

| 구분 | 도입 전 (Non-Reentrant) | 도입 후 (Reentrant) | 개선 효과 |
|:---|:---|:---|:---|
| **시스템 안정성** | 알 수 없는 이유로 시스템 프리징 | 인터럽트/신호 상황에서도 견고함 | 커널 및 임베디드 신뢰성 극대화 |
| **병렬 처리 성능** | 락(Lock) 경합으로 인한 성능 저하 | 락 없는 실행으로 최대 병렬성 확보 | 멀티코어 성능 선형적 향상 |
| **테스트 용이성** | 상태 의존적 버그로 재현 어려움 | 입력값에만 의존하여 단위 테스트 용이 | 개발 생산성 및 품질 향상 |

#### 미래 전망
실시간 제어와 초병렬 처리가 중요해지는 미래 컴퓨팅 환경에서 재진입성은 더욱 강조될 것이다. 특히 Rust와 같은 언어가 강제하는 '가변성 제어'와 '불변 데이터 구조'는 재진입 가능한 코드를 작성하는 것을 기본값으로 만들고 있으며, 이는 하드웨어 인터럽트와 동시성 처리가 얽힌 복잡한 시스템 개발의 난이도를 낮추는 결정적 역할을 할 것이다.

- **참고 표준**: POSIX.1 Async-Signal-Safe Functions

- **📢 섹션 요약 비유**: 어떤 상황에서도 자기 자리를 지키며 묵묵히 제 역할을 다하는 독립적인 전문가들이 모여, 흔들림 없는 시스템의 기초를 형성하는 것과 같습니다.

### 📌 관련 개념 맵
- **스레드 안전 (Thread-safe)**: 동시성 제어의 일반적인 개념.
- **임계 구역 (Critical Section)**: 재진입성 확보를 위해 제거해야 할 영역.
- **순수 함수 (Pure Function)**: 재진입성의 가장 이상적인 형태.
- **Async-Signal-Safe**: 리눅스 신호 처리에서의 재진입 표준.

### 👶 어린이를 위한 3줄 비유 설명
1. 재진입 가능 코드는 **"각자 자기 도시락을 먹는 친구들"**과 같아요.
2. 같이 밥을 먹다가 중간에 잠시 나가도, 혹은 친구가 옆에서 같이 먹어도 내 도시락 반찬이 줄어들거나 섞이지 않죠.
3. 내 물건(변수)만 챙겨서 사용하기 때문에, 언제 어디서나 안전하고 정확하게 일을 끝낼 수 있답니다!
