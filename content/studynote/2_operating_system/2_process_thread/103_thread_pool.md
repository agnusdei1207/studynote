# 스레드 풀 (Thread Pool)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스레드 풀(Thread Pool)은 일정한 수의 스레드를 **미리 생성하여 대기(Waiting) 상태로 유지**하고, 작업 요청이 들어올 때마다 대기 중인 스레드를 할당하여 처리하는 자원 관리 기법이다.
> 2. **가치**: 스레드의 빈번한 생성과 제거에 따른 시스템 오버헤드를 획기적으로 줄이며, 동시에 실행되는 **스레드 총수를 제한**함으로써 시스템 자원 고갈을 방지하고 안정성을 확보한다.
> 3. **융합**: 작업 큐(Work Queue)와 결합된 프로듀서-컨슈머 모델의 전형적인 구현체이며, 현대의 고성능 서버 프레임워크와 비동기 프로그래밍 모델의 핵심 인프라이다.

+++

## Ⅰ. 스레드 풀의 정의 및 필요성

### 1. 정의
- 프로세스 실행 초기에 일정 수의 스레드를 미리 만들어 풀(Pool)에 넣어두고, 작업(Task)이 발생하면 풀에서 스레드를 빌려와 작업을 수행한 뒤 다시 풀에 반납하는 방식이다.

### 2. 왜 필요한가? (The Problem of Direct Threading)
- **생성 비용**: 스레드를 새로 만드는 것은 커널 자원을 할당받는 무거운 작업이다.
- **자원 고갈**: 동시 접속자가 폭증할 때 접속마다 스레드를 만들면 메모리가 부족해져 전체 시스템이 다운될 수 있다.
- **스케줄링 부하**: 스레드가 너무 많아지면 CPU가 실제 일을 하는 시간보다 스레드 간 문맥 교환을 하는 시간이 더 길어지는 스래싱(Thrashing)이 발생한다.

### 3. 💡 비유: '대기 중인 택시 승강장'
- 스레드 풀은 **'승객(Task)이 올 때마다 새 차(Thread)를 공장에서 찍어내는 대신, 승강장에 미리 대기 중인 택시들을 순서대로 배차하는 시스템'**과 같다.
- 승객이 많아도 승강장의 택시 수만큼만 동시에 이동할 수 있어 도로(CPU/메모리) 정체를 막을 수 있고, 손님은 차가 만들어지길 기다릴 필요 없이 즉시 탈 수 있다.

## II. 동작 메커니즘과 구성 요소 (Deep Dive)

### 1. 핵심 구성 요소
- **작업 큐 (Work Queue)**: 처리해야 할 작업(함수, 로직)들이 줄을 서서 기다리는 공간.
- **스레드 풀 (Worker Threads)**: 작업을 집어 실행할 준비가 된 스레드들의 집합.
- **관리자 (Manager)**: 풀의 크기를 조절하거나 유휴 스레드를 관리하는 로직.

### 2. 동작 프로세스
1. 클라이언트가 작업을 요청하면 해당 작업을 **작업 큐**에 넣는다.
2. 풀에 있는 유휴(Idle) 스레드가 큐에서 작업을 하나 꺼내어 실행한다.
3. 작업이 완료되면 스레드는 종료되지 않고 다시 풀로 돌아가 다음 작업을 기다린다.
4. 만약 풀의 모든 스레드가 바쁘다면, 작업은 큐에서 대기하거나 거절(Reject)된다.

### 3. ASCII 구조 다이어그램
```text
    [ Task Requests ] ──▶ [ Work Queue ]
                               │
    ===========================│==============================
    [ Thread Pool ]            ▼ (Task Dispatching)
    ┌────────────────────────────────────────────────────────┐
    │  Worker 1 (Busy)  Worker 2 (Idle)  Worker 3 (Busy)     │
    │  ┌──────────┐     ┌──────────┐     ┌──────────┐        │
    │  │  Doing.. │     │ Waiting..│     │  Doing.. │        │
    │  └──────────┘     └──────────┘     └──────────┘        │
    └────────┬───────────────┬───────────────┬───────────────┘
             ▼               ▼               ▼
         [ Core 0 ]      [ Core 1 ]      [ Core 2 ]
```

- **📢 섹션 요약 비유**: 주문서(Task)가 쌓이면 미리 주방에서 대기하던 요리사(Worker Thread)들이 하나씩 가져가 요리하고, 다 하면 다시 제자리에 서서 다음 주문을 기다리는 체계적인 주방 시스템입니다.

## Ⅲ. 스레드 풀의 크기 결정 전략 (Tuning)

적절한 스레드 풀의 크기는 시스템 성능의 성패를 좌우한다.

| 작업 성격 | 권장 전략 | 이유 |
|:---|:---|:---|
| **CPU 집중 작업** | **코어 개수 + 1** | 문맥 교환 최소화로 계산 성능 집중 |
| **I/O 집중 작업** | **코어 개수 * N (큰 값)** | I/O 대기 시간 동안 다른 스레드가 실행되어야 함 |
| **혼합 작업** | 동적 크기 조절 (Dynamic) | 부하에 따라 풀 크기를 유동적으로 변경 |

- **📢 섹션 요약 비조**: "일꾼이 너무 적으면 줄이 길어지고(지연), 너무 많으면 서로 부딪혀 싸웁니다(오버헤드). 작업의 결에 맞는 황금비율을 찾는 것이 기술사의 통찰입니다."

+++

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: Java ExecutorService 활용
- **상황**: 대규모 트랜잭션을 처리하는 스프링(Spring) 기반 백엔드 서버 개발.
- **판단**: 직접 스레드를 생성하지 않고 `Executors.newFixedThreadPool(n)`과 같은 라이브러리를 사용한다. 이를 통해 하부의 복잡한 스레드 생성 로직을 감추고, 안정적인 큐 관리와 예외 처리를 시스템 레벨에서 보장받는다.

### 2. 기술사적 인사이트: 거절 정책 (Rejection Policy)
- 기술사라면 풀이 가득 찼을 때의 **거절 정책**을 반드시 고려해야 한다. 무한정 큐에 쌓으면 메모리 누수(OOM)가 발생하므로, "실패 반환(Abort)", "호출자가 직접 실행(Caller Runs)", "가장 오래된 작업 버리기(Discard Oldest)" 등의 정책 중 비즈니스 요구사항에 맞는 최적의 선택을 제시해야 한다.

- **📢 섹션 요약 비조**: "가득 찬 컵에 물을 계속 부으면 넘치듯이, 시스템의 한계 상황에서 어떤 '질서 있는 포기'를 할 것인가가 아키텍처의 완성도를 결정합니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **응답 시간 단축**: 스레드 생성 지연(Latency) 제거로 빠른 서비스 제공.
- **시스템 안정성 극대화**: 예측 가능한 자원 점유를 통한 연쇄 장애 방지.

### 2. 미래 전망
앞으로는 가상화 및 서버리스 환경에 최적화되어, 수만 개의 초경량 스레드를 풀링 없이도 효율적으로 관리하는 **가상 스레드(Virtual Threads)** 기술이 스레드 풀의 일부 영역을 대체할 것이다. 하지만 하드웨어 자원의 물리적 한계가 존재하는 한, '제한된 자원의 효율적 배분'이라는 스레드 풀의 핵심 철학은 모든 컴퓨팅 아키텍처의 근간으로 남을 것이다.

- **📢 섹션 요약 비유**: 물을 아껴 쓰기 위해 저수지(Pool)를 만들고 수로(Queue)를 정비하여 온 마을(시스템)에 골고루 물을 공급하는 지혜로운 치수 사업과 같습니다.
