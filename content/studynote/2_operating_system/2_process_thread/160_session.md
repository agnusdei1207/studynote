# 세션 (Session)과 제어 터미널

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 세션 (Session)은 하나 이상의 **프로세스 그룹들의 집합**으로, 사용자가 시스템에 로그인하여 로그아웃할 때까지 수행하는 모든 활동을 논리적으로 묶어 관리하는 최상위 작업 단위이다.
> 2. **가치**: **제어 터미널 (Controlling Terminal)**과의 연결을 통해 사용자의 키보드 입력(인터럽트 시그널 등)을 특정 프로세스 그룹에 전달하며, 세션 리더가 종료될 때 하위의 모든 프로세스에게 신호를 보내 안전한 일괄 종료를 보장한다.
> 3. **융합**: 프로세스 ID(PID), 프로세스 그룹 ID(PGID)를 넘어선 세션 ID(SID) 체계를 통해 운영체제는 다중 사용자 환경에서 각 사용자의 작업 영역을 완벽히 격리하고 제어한다.

+++

## Ⅰ. 세션(Session)의 개요

### 1. 정의
- 사용자가 시스템에 로그인했을 때 생성되는 프로세스들의 가장 큰 묶음이다. 하나의 세션은 여러 개의 프로세스 그룹(Foreground, Background)을 포함할 수 있다.

### 2. 세션 리더 (Session Leader)
- 세션을 생성한 프로세스를 의미하며, 보통 사용자가 로그인할 때 실행되는 **셸 (Shell)**이 세션 리더가 된다. 세션 리더의 PID가 곧 세션 ID(SID)가 된다.

### 3. 💡 비유: '놀이공원 단체 관람권'
- **프로세스**가 '개인 관람객'이라면, **프로세스 그룹**은 '가족 단위', **세션**은 '여행사 단체 팀'과 같다.
- 여행사 가이드(세션 리더)가 깃발을 들고 팀원들을 이끌며, 가이드가 "이제 집에 갑시다!"라고 하면 팀에 속한 모든 가족과 개인(프로세스들)이 한꺼번에 움직이는 것과 같다.

+++

## Ⅱ. 제어 터미널 (Controlling Terminal) (Deep Dive)

### 1. 정의 및 연결
- 하나의 세션은 최대 하나의 **제어 터미널**을 가질 수 있다. 보통 로그인 시 사용하는 터미널 장치(`/dev/ttyX`)가 이에 해당한다.

### 2. 역할: '소통의 창구'
- 사용자가 터미널에서 `Ctrl+C` (SIGINT)를 누르면, 커널은 제어 터미널과 연결된 세션 내의 **전면 프로세스 그룹 (Foreground Process Group)**에게 해당 시그널을 전달한다.

### 3. ASCII 구조 다이어그램
```text
    [ Controlling Terminal (/dev/pts/0) ]
                   │ (Signals: Ctrl+C, Ctrl+Z)
                   ▼
    ┌──────────────────────────────────────────────┐
    │               Session (SID: 100)             │
    │  ┌────────────────────────────────────────┐  │
    │  │ [ Session Leader: Bash (PID: 100) ]    │  │
    │  └──────────────────┬─────────────────────┘  │
    │                     ▼                        │
    │  ┌─────────────────────┐  ┌────────────────┐ │
    │  │ FG Process Group    │  │ BG Process Group│ │
    │  │ (ls | grep)         │  │ (find &)       │ │
    │  └─────────────────────┘  └────────────────┘ │
    └──────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 터미널은 세션이라는 거대 조직과 외부 세계(사용자)를 연결해 주는 유일한 '전화기'와 같습니다. 전화벨이 울리면(시그널), 조직 내 담당 부서(Foreground Group)가 즉시 응답합니다.

+++

## Ⅲ. 세션과 프로세스 계층 구조 비교

| 구분 | 프로세스 (Process) | 프로세스 그룹 (Group) | 세션 (Session) |
|:---|:---|:---|:---|
| **식별자** | PID | PGID | SID |
| **목적** | 개별 실행 단위 | 파이프라인/협업 관리 | **사용자 활동/터미널 관리** |
| **관계** | 1 (최하위) | N (프로세스의 집합) | **M (그룹의 집합)** |
| **특징** | 실행의 실체 | 시그널 일괄 전달 단위 | 로그아웃 시 일괄 종료 단위 |

- **📢 섹션 요약 비유**: 병사(프로세스)들이 모여 소대(그룹)를 이루고, 소대들이 모여 연대(세션)를 이루어 일사불란하게 움직이는 군대 조직과 같습니다.

+++

## Ⅳ. 실무 적용 및 운영 인사이트 (Strategy)

### 1. 실무 시나리오: 데몬(Daemon) 프로세스 생성
- **상황**: 터미널을 닫아도 배경에서 계속 돌아가는 서버 프로그램을 만들어야 함.
- **판단**: 프로세스가 현재 세션에 속해 있으면 터미널 종료 시 `SIGHUP` 시그널을 받고 죽게 된다. 이를 방지하기 위해 **`setsid()`** 시스템 콜을 호출한다. 이 명령을 통해 프로세스는 기존 세션과 터미널의 결합을 끊고, **자신이 리더가 되는 새로운 세션**을 창조하여 독자적으로 생존하게 된다. (데몬화의 핵심)

### 2. 기술사적 인사이트: 좀비 세션 방지
- SSH 연결이 끊겼는데도 서버에 프로세스가 남아 자원을 낭비하는 경우가 있다. 기술사라면 세션 리더인 셸이 종료될 때 자식들에게 시그널을 보내는 메커니즘을 이해하고, `nohup`이나 `tmux/screen`과 같은 도구가 어떻게 세션 구조를 조작하여 프로세스를 보호하는지 아키텍처적으로 설명할 수 있어야 한다.

- **📢 섹션 요약 비유**: 집주인(터미널)이 이사를 가면 입주민(프로세스)들도 다 나가야 하지만, 미리 독립(setsid)하여 자기 집을 차린 사람은 계속 살 수 있는 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **자원 격리 및 보안**: 사용자별 작업 영역을 명확히 구분하여 권한 침범 방지.
- **일괄 관리 효율성**: 로그인/로그아웃에 따른 복잡한 프로세스 정리를 자동화.

### 2. 미래 전망
최근에는 물리적 터미널 대신 웹 터미널이나 컨테이너 내부의 가상 터미널 환경이 보편화되고 있다. 하지만 프로세스들을 세션이라는 논리적 단위로 묶어 생명 주기를 관리하는 운영체제의 기본 철학은, 서버리스 환경의 인스턴스 관리나 클라우드 가상 세션 제어 기술로 그 생명력을 이어가고 있다.

- **📢 섹션 요약 비유**: 시작과 끝(세션)을 명확히 관리하는 것이야말로, 혼란스러운 다중 사용자 시스템을 지탱하는 질서의 힘입니다.

+++

### 📌 관련 개념 맵 (Knowledge Graph)
- **프로세스 그룹**: 세션을 구성하는 하위 단위.
- **데몬 프로세스**: 세션에서 탈출한 독립 프로세스.
- **시그널 (Signal)**: 세션과 터미널 사이의 소통 수단.

2. 내가 놀이 시간(세션)이 끝나서 집에 가면, 내가 가지고 놀던 모든 장난감(프로세스들)도 선생님이 깨끗이 정리해 주시죠.
3. 이 덕분에 다음 친구가 올 때 컴퓨터가 지저분하지 않고 항상 깨끗한 상태로 유지될 수 있답니다!
