## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OOM (Out Of Memory) Killer는 시스템 메모리가 극도로 부족하여 OS 운영이 불가능한 한계 상황에 도달했을 때, 특정 프로세스를 강제로 종료시켜 가용 메모리를 즉시 확보하는 커널의 최후방 방어 기제다.
> 2. **가치**: 전체 시스템이 응답 불능(Panic) 상태에 빠지는 것을 막아 핵심 커널 서비스와 여타 무고한 프로세스의 생존을 도모하며, 'Overcommit' 환경에서도 최소한의 시스템 가용성을 유지한다.
> 3. **융합**: 가상 메모리 관리의 가용 페이지 추적, 프로세스 우선순위 및 가치 판단 알고리즘(Badness Score), 그리고 리소스 격리 기술인 컨트롤 그룹(cgroups)과 연계되어 동작한다.

---

### Ⅰ. 개요 (Context & Background)

- **概念**: **OOM Killer**는 리눅스 커널의 메모리 관리 서브시스템에 내장된 알고리즘이다. 리눅스는 실제 물리 메모리보다 더 많은 메모리를 프로세스에게 할당하는 **Memory Overcommit**을 허용하는데, 프로세스들이 실제로 그 메모리를 한꺼번에 사용하려고 하면 물리 메모리가 고갈된다. 이때 커널은 '희생양' 프로세스를 골라 죽임으로써 시스템 전체의 붕괴를 막는다.

- **💡 비유**: OOM Killer는 침몰해가는 배에서 배가 완전히 가라앉지 않도록 **"무거운 짐을 바다로 던지는 선장"**과 같다. 모든 승객을 살릴 수 없는 절박한 순간에, 배의 평형을 위해 가장 덜 중요한 짐(부적절한 프로세스)을 골라 버리는 아픈 결단을 내리는 것과 유사하다.

- **등장 배경**:
  1. **Memory Overcommit의 탄생**: 많은 프로세스가 메모리를 할당만 받고 실제로는 다 쓰지 않는다는 점에 착안하여, 가용 자원을 최대한 활용하기 위해 물리적 한계를 넘는 할당을 허용하게 되었다.
  2. **시스템 안정성 확보**: 할당은 해줬는데 실제 줄 메모리가 없을 때, 커널이 멈춰버리는(Kernel Panic) 대신 스스로를 치유할 수 있는 매커니즘이 필요했다.
  3. **가용 자원의 한계**: 물리 메모리(RAM)와 스왑(Swap) 공간이 모두 가득 찬 상태에서 커널의 관리 작업조차 불가능해지는 '데드락' 상황을 타개해야 했다.

- **📢 섹션 요약 비유**: 전체를 살리기 위해 일부를 희생시켜야만 하는 시스템의 비정한 생존 본능이자, 최후의 보루입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### OOM Killer의 동작 매커니즘 및 지표 (표)

| 요소명 | 역할 | 내부 동작 | 관련 지표 | 비유 |
|:---|:---|:---|:---|:---|
| **Badness Score** | 희생양 선정을 위한 점수 | 많이 쓰고, 오래되고, 중요도 낮은 프로세스에 높은 점수 | `oom_score` | 처분 대상 리스트 순위 |
| **oom_score_adj** | 사용자가 부여하는 가중치 | 특정 프로세스를 절대 죽이지 않거나 먼저 죽게 설정 | `/proc/[pid]/oom_score_adj` | VIP 면제권 / 우선 처분표 |
| **Heuristics** | 점수 계산 알고리즘 | 자식 프로세스 포함 여부, 실행 시간 등을 고려 | - | 선장의 판단 기준 |
| **OOM Event** | 실행 트리거 | 페이지 할당 실패 및 메모리 회수 불가 시 발생 | `out_of_memory()` 함수 | 경보 발령 |
| **Victim Selection** | 실제 프로세스 선택 | 가장 높은 Badness 점수를 가진 태스크 선택 | `select_bad_process()` | 희생양 낙점 |

#### OOM Killer의 트리거 및 프로세스 종료 구조

메모리 부족 감지부터 프로세스 종료를 통한 메모리 회수까지의 과정이다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │                 OOM Killer의 희생양 선정 및 처리 흐름                │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │   1. Memory Request (malloc/fork)                            │
  │              │                                               │
  │   2. Low Memory Condition Detected                           │
  │              │                                               │
  │   3. Kernel tries to reclaim pages (Swap-out, Cache drop)    │
  │              │                                               │
  │   4. Still Out of Memory? ──▶ [ Invoke OOM Killer ]          │
  │                                      │                       │
  │   5. Calculate Badness Score for all processes               │
  │      - Memory Usage (+)                                      │
  │      - Process Age (-)                                       │
  │      - Root Privileges (-)                                   │
  │      - oom_score_adj (Custom)                                │
  │                                      │                       │
  │   6. Select Victim (Highest Score) ──┴──▶ [ Kill Process ]   │
  │                                               │              │
  │   7. Memory Reclaimed ◀───────────────────────┘              │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 프로세스가 메모리를 요청하면 커널은 먼저 사용하지 않는 페이지를 디스크로 보내거나(Swap) 캐시를 비워 메모리를 확보하려 노력한다. 하지만 이러한 노력에도 불구하고 메모리를 줄 수 없는 상황이 되면 **OOM Killer**가 소환된다. 커널은 시스템의 모든 프로세스를 훑으며 '누구를 죽이는 것이 시스템에 가장 이득인가'를 계산한다. 기본적으로 메모리를 많이 쓰면서 시스템의 핵심 작업이 아닌 프로세스가 높은 점수를 받는다. 희생양이 선정되면 커널은 해당 프로세스에게 `SIGKILL`을 보내 즉시 종료시키고, 그가 쥐고 있던 메모리를 회수하여 시스템에 숨통을 틔워준다.

#### 심층 동작 원리: Badness Score의 계산 로직

커널 2.6.36 이후의 현대적 리눅스 커널에서는 다음과 같이 점수를 계산한다.
1. **기본 점수**: 프로세스가 사용하는 물리 메모리(RSS)와 페이지 테이블 점유량에 비례한다. (메모리 0.1% 당 10점, 최대 1,000점)
2. **보정 (Adjustment)**: `/proc/[pid]/oom_score_adj` 값을 더한다. 이 값은 -1000부터 1000까지 설정 가능하다.
3. **특권 우대**: root 권한으로 실행되는 프로세스는 아주 미세한 감점을 받아 일반 프로세스보다 약간 더 보호받는다.
4. **결과**: 최종 합산 점수가 가장 높은 녀석이 죽는다. `-1000`으로 설정된 프로세스는 절대 OOM Killer의 대상이 되지 않는다(OOM_SCORE_ADJ_MIN).

- **📢 섹션 요약 비유**: 평소 밥(메모리)을 많이 먹으면서 집안일(시스템 기여)은 안 하는 구성원에게 가장 높은 '처분 점수'를 매기는 것과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

#### 비교: Swap(가상 메모리) vs OOM Killer

| 비교 항목 | Swap (Swapping) | OOM Killer |
|:---|:---|:---|
| **작동 시점** | 메모리가 부족하기 시작할 때 (완만함) | 메모리가 완전히 고갈되었을 때 (급박함) |
| **데이터 보존** | 디스크에 보관 (데이터 유지됨) | 프로세스 강제 종료 (데이터 손실됨) |
| **시스템 성능** | 디스크 I/O로 인해 급격히 느려짐 (Thrashing) | 희생 후 나머지 프로세스는 정상 속도 회복 |
| **목적** | 메모리 용량의 논리적 확장 | 시스템 전체의 생존 보장 |
| **비유** | 좁은 방에서 짐을 창고로 옮기기 | 방이 터질 것 같아 짐을 쓰레기통에 버리기 |

#### 과목 융합 관점: 가상화와 클라우드

- **컨테이너 (Docker/K8s)**: 컨테이너는 `cgroups`를 사용하여 메모리를 제한한다. 만약 컨테이너가 자신에게 할당된 제한치를 넘으면, 시스템 전체 메모리가 남아있더라도 해당 컨테이너 '내부'에서 OOM Killer가 작동하여 프로세스를 죽인다.
- **데이터베이스 (DB)**: Redis나 Memcached처럼 대량의 메모리를 쓰는 앱은 OOM Killer의 1순위 타겟이다. 이를 방지하기 위해 DB 관리자는 반드시 `oom_score_adj`를 낮게 설정하여 보호 장치를 마련해야 한다.

- **📢 섹션 요약 비유**: 전체 국가의 식량 부족(시스템 OOM)과 특정 마을의 식량 부족(cgroup OOM) 상황에서 각각 다른 기준으로 배급 중단(종료)이 일어나는 것과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단

#### 실무 시나리오: 자바(JVM) 힙 메모리 설정 오류 대응
자바 기반의 웹 서버가 불규칙하게 죽는 현상이 발생했다. 로그에는 아무런 에러가 없는데 프로세스만 사라진 상태였다. `dmesg`를 확인하니 "Out of memory: Kill process (java)" 메시지가 발견되었다. 원인은 물리 RAM은 8GB인데 JVM 힙을 12GB로 과도하게 설정하여 사용량이 늘어나는 순간 OOM Killer가 작동한 것이었다. 해결을 위해 힙 크기를 물리 메모리의 70% 수준으로 조정하고, 만약의 사태를 대비해 서버 프로세스의 `oom_score_adj`를 `-500`으로 설정하여 다른 비핵심 프로세스보다 먼저 죽지 않게 조치했다.

#### 도입 체크리스트
- **Overcommit 모드**: `/proc/sys/vm/overcommit_memory` 설정이 시스템 특성(안정성 중시 vs 효율성 중시)에 맞는가?
- **핵심 프로세스 보호**: DB, SSHD 등 시스템 운영에 필수적인 프로세스의 `oom_score_adj`가 적절히 낮게 설정되어 있는가?
- **모니터링**: `dmesg`나 `/var/log/messages`를 주기적으로 스캔하여 OOM 발생 이력을 감시하고 있는가?

#### 안티패턴
- **OOM 무력화 시도**: 시스템 전체의 OOM을 끄는 행위는 메모리 고갈 시 시스템을 커널 패닉 상태로 몰아넣어 더 큰 장애를 유발한다.
- **Swap 없는 시스템 운영**: Swap이 없으면 메모리 부족 시 완충 지대 없이 즉시 OOM Killer가 작동하므로, 서비스가 매우 불안정해질 수 있다.

- **📢 섹션 요약 비유**: 소화기(OOM Killer)가 보기 싫다고 치워버리면, 작은 불씨(메모리 부족)가 집 전체를 태워버리는 대형 화재(시스템 다운)로 번질 수 있습니다.

### Ⅴ. 기대효과 및 결론

#### 정량/정성 기대효과

| 구분 | 도입 전 (No OOM Killer) | 도입 후 (With OOM Killer) | 개선 효과 |
|:---|:---|:---|:---|
| **시스템 가동 시간** | 메모리 부족 시 무조건 리부팅 | 희생을 통해 지속 가동 유지 | 가용성(Uptime) 획기적 향상 |
| **자원 활용률** | 물리 RAM 이내로만 할당 제한 | Overcommit을 통한 자원 낭비 최소화 | 인프라 비용 효율성 극대화 |
| **관리 자동화** | 관리자가 직접 판단하여 종료 | 커널이 휴리스틱하게 자동 처리 | 운영 부담 절감 및 실시간 대응 |

#### 미래 전망
메모리 계층 구조가 복잡해지고(CXL, Persistent Memory 등) 마이크로서비스가 극도로 세분화됨에 따라, OOM Killer의 알고리즘도 더 정교해지고 있다. 향후에는 AI가 프로세스의 중요도와 사용자 패턴을 학습하여 단순히 '많이 쓰는 놈'이 아닌 '죽여도 서비스에 지장이 없는 놈'을 골라내는 지능형 자원 회수 기술로 진화할 전망이다.

- **참고 표준**: Linux Kernel `mm/oom_kill.c`

- **📢 섹션 요약 비유**: 고통스러운 선택이지만, 더 큰 가치를 지키기 위해 결단을 내리는 차가운 이성이 시스템의 마지막 생명줄이 됩니다.

### 📌 관련 개념 맵
- **가상 메모리 (Virtual Memory)**: OOM 상황이 발생하는 근본적인 배경.
- **oom_score_adj**: OOM Killer의 판단에 개입하는 직접적인 도구.
- **cgroups (Control Groups)**: 메모리 제한과 OOM 발생 범위를 제어하는 기술.
- **스왑 (Swap)**: OOM Killer 작동 전의 1차 완충 지대.

### 👶 어린이를 위한 3줄 비유 설명
1. OOM Killer는 컴퓨터 메모리라는 **"작은 간식 주머니"**가 텅 비었을 때 나타나는 해결사예요.
2. 모든 프로그램이 간식을 더 달라고 떼를 쓰면 주머니가 터질 수 있어서, 가장 많이 먹고 덜 중요한 프로그램 하나를 골라 집으로 보내버리죠.
3. 슬픈 일이지만, 이렇게 해야 나머지 프로그램들이 남은 간식을 나눠 먹으며 계속 공부를 할 수 있답니다!
