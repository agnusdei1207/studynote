# 액터 모델 (Actor Model) - Erlang, Akka 동시성 모델

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 액터 모델(Actor Model)은 가변적인 상태 공유를 지양하고, 독립된 개체인 **'액터(Actor)'들이 오직 메시지 전달(Message Passing)만을 통해 소통**하는 병렬 컴퓨팅 모델이다.
> 2. **가치**: 락(Lock) 기반의 복잡한 동기화 문제와 데드락(Deadlock) 위험을 원천적으로 제거하며, 수백만 개의 액터를 생성할 수 있는 초경량성과 고도의 장애 허용성(Fault Tolerance)을 제공한다.
> 3. **융합**: Erlang 언어의 핵심 철학이자 Java/Scala의 Akka 프레임워크의 기반이며, 현대 분산 시스템과 마이크로서비스 아키텍처(MSA)의 논리적 실행 모델로 널리 활용된다.

---

## Ⅰ. 액터 모델의 정의 및 3대 원칙

### 1. 정의
- 동시성 연산을 위한 수학적 모델로, 모든 것을 '액터'로 취급한다. 액터는 상태, 동작, 메일박스(Mailbox)를 가진 독립된 계산 단위이다.

### 2. 액터의 3대 행위 (메시지 수신 시)
- **다른 액터에게 메시지 전송**: 비동기적으로 다른 액터와 통신한다.
- **새로운 액터 생성**: 필요에 따라 자식 액터를 만들어 작업을 분산한다.
- **다음 메시지 처리 상태 결정**: 자신의 내부 상태를 갱신하여 다음 메시지에 대비한다.

### 3. 💡 비유: '우편함이 있는 개별 사무실'
- 액터 모델은 **'직원들이 서로 대화하지 않고, 각자 사무실 문 앞에 있는 우편함(메일박스)으로만 업무 지시를 주고받는 회사'**와 같다.
- 직원(액터)은 자기 방(내부 상태)에서 혼자 일하므로 남과 부딪힐 일이 없다. 옆 직원이 아파서 쉬어도(에러), 우편함에 편지만 쌓일 뿐 회사는 멈추지 않는다.

---

## II. 동작 메커니즘: '공유 없음 (Share Nothing)' (Deep Dive)

### 1. 메시지 큐와 비동기 통신
- 액터는 전달받은 메시지를 자신의 **메일박스**에 쌓아두고, 하나씩 순차적으로 꺼내 처리한다. 이 과정은 다른 액터의 작업과 완전히 독립적으로 진행된다.

### 2. 장애 허용 (Let it Crash)
- 부모 액터는 자식 액터의 생사 여부를 감시(Monitoring)한다. 자식이 에러로 죽으면 부모가 이를 감지하여 재시작하거나 복구 전략을 수행한다.

### 3. ASCII 구조 다이어그램
```text
    [ Actor A ]                         [ Actor B ]
    ┌─────────────┐                     ┌─────────────┐
    │ State: 10   │ ──(Send Msg)──┐     │ State: 20   │
    │ Behavior    │               │     │ Behavior    │
    └─────────────┘               │     └──────▲──────┘
                                  │            │ (Process)
                                  │     ┌──────┴──────┐
                                  └────▶│ Mailbox (B) │ ◀── [ Msg 3 | Msg 2 | Msg 1 ]
                                        └─────────────┘
```

- **📢 섹션 요약 비유**: 각자가 독립적인 방에서 자기 일만 하는 완벽한 분업 시스템입니다. 정보 공유는 오직 공식적인 서신(메시지)을 통해서만 이루어지므로, 불필요한 간섭과 갈등(Lock 경합)이 발생하지 않습니다.

---

## Ⅲ. 락 기반 모델 vs 액터 모델 비교

| 비교 항목 | 락(Lock) 기반 모델 | 액터(Actor) 모델 |
|:---|:---|:---|
| **데이터 공유** | 공유 메모리 접근 (Shared State) | **메시지 전달 (Shared Nothing)** |
| **동기화 기법** | Mutex, Semaphore 사용 | **단일 스레드 메시지 루프** |
| **주요 위험** | 데드락, 데이터 레이스 | 메시지 홍수 (Mailbox Overflow) |
| **확장성** | 수평 확장이 어려움 | **분산 환경으로의 확장 용이** |

- **📢 섹션 요약 비조**: "락 기반 모델이 '하나의 큰 가마솥에 숟가락을 같이 넣는 것'이라면, 액터 모델은 '각자 자기 그릇에 담긴 음식을 먹으며 대화하는 것'입니다."

---

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 채팅 서버 및 게임 엔진
- **상황**: 수백만 명의 사용자가 동시에 접속하는 실시간 채팅 서비스.
- **판단**: 각 사용자 세션을 하나의 **액터**로 만든다. 사용자가 메시지를 보내면 해당 액터의 메일박스에 넣기만 하면 된다. 수백만 개의 액터는 커널 스레드보다 훨씬 가벼우므로 메모리 효율이 극대화되며, 특정 사용자 액터가 죽어도 시스템 전체는 영향을 받지 않는다. (WhatsApp이 Erlang을 선택한 핵심 이유)

### 2. 기술사적 인사이트: 위치 투명성 (Location Transparency)
- 기술사라면 액터 모델의 **위치 투명성** 가치를 강조해야 한다. 메시지를 보내는 쪽은 상대 액터가 같은 메모리에 있는지, 아니면 지구 반대편 서버에 있는지 알 필요가 없다. 이는 로컬 병렬 처리를 분산 컴퓨팅으로 즉시 확장할 수 있게 하는 마이크로서비스 아키텍처의 이상적인 구현체임을 설명해야 한다.

- **📢 섹션 요약 비조**: "물리적인 거리를 논리적인 주소(Actor Ref)로 극복하는 것이, 거대 분산 시스템을 하나처럼 다스리는 통치 기술입니다."

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **시스템 가용성 혁신**: 장애 발생 시 자동 복구(Supervision)를 통한 99.999% 신뢰성 달성.
- **개발 단순화**: 복잡한 락 로직을 제거하여 비즈니스 로직에만 집중 가능.

### 2. 미래 전망
앞으로는 엣지 컴퓨팅과 서버리스 환경이 결합되면서, 거대한 상태를 관리하는 대신 작고 민첩한 액터들이 협업하는 **상태 저장형 서버리스(Stateful Serverless)** 모델이 부상할 것이다. 액터 모델은 수십 년 전 탄생했음에도 불구하고, 멀티코어와 분산 네트워크가 지배하는 현재와 미래에 가장 적합한 '동시성의 정답'으로 평가받고 있다.

- **📢 섹션 요약 비유**: 수만 마리의 개미(액터)가 각자의 역할에 충실하며 거대한 개미집(시스템)을 유지하는, 자연계의 완벽한 협동 알고리즘을 디지털로 재현한 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 액터 모델은 컴퓨터 친구들이 **'각자 자기 방에서 편지로만 대화하는 것'**과 같아요.
2. "이 인형 좀 빌려줘"라고 편지를 보내면, 친구가 자기 방에서 인형을 챙겨서 답장으로 보내주는 거죠.
3. 서로의 방에 함부로 들어가지 않으니까 싸울 일이 없고, 한 친구가 낮잠을 자도(에러) 다른 친구들은 계속 재미있게 놀 수 있답니다!
