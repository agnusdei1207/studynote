# 프로세스 종료 (Process Termination) - exit(), wait()

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 종료는 실행 중인 프로그램이 모든 명령을 수행하거나 오류가 발생하여 **운영체제에 할당된 모든 자원을 반납하고 시스템에서 사라지는 과정**이다.
> 2. **가치**: `exit()`를 통해 종료 상태(Status Code)를 알리고 `wait()`를 통해 부모가 이를 수거함으로써 프로세스 간 동기화를 실현하며, 자원 누수(Memory Leak)를 방지한다.
> 3. **융합**: 프로세스가 종료되어도 최소한의 정보가 남는 '좀비' 상태를 거치며, 부모의 확인이 있어야만 비로소 시스템에서 완전히 삭제되는 질서 있는 퇴장 메커니즘을 가진다.

+++

## Ⅰ. 프로세스 종료의 정의 및 유형

### 1. 자발적 종료 (Voluntary)
- 프로그램이 `main()`의 끝에 도달하거나, 코드 내부에서 명시적으로 `exit()` 시스템 콜을 호출하여 작업을 마치는 경우이다.

### 2. 비자발적 종료 (Involuntary / Abort)
- 0으로 나누기, 잘못된 메모리 접근(Segment Fault) 등 예외 발생 시 운영체제에 의해 강제 종료되는 경우이다.
- 부모 프로세스가 `terminate` 신호를 보내 자식을 강제 종료할 수도 있다.

### 3. 💡 비유: '직원의 퇴사 과정'
- 프로세스 종료는 **'직원이 업무를 마치고 사직서(exit)를 내는 과정'**과 같다.
- 사직서를 냈다고 바로 정보가 사라지는 게 아니라, 인사팀(운영체제)이 퇴직금 정산(자원 반납)을 하고 사장님(부모)이 사직서를 확인(wait)해야 비로소 회사 명단(프로세스 테이블)에서 이름이 지워진다.

+++

## II. 핵심 시스템 콜: exit()와 wait() (Deep Dive)

### 1. exit() 시스템 콜
- **역할**: 프로세스를 종료 상태로 전환한다.
- **동작**: 열린 파일을 닫고, 할당된 메모리를 해제한 뒤, 정수값의 **종료 상태 코드**를 커널에 남긴다.
- **결과**: 프로세스는 종료되었으나 아직 프로세스 테이블에는 남아있는 '좀비(Zombie)' 상태가 된다.

### 2. wait() 시스템 콜
- **역할**: 자식 프로세스가 종료될 때까지 부모 프로세스의 실행을 일시 중단한다.
- **동작**: 자식이 남긴 종료 상태 코드를 읽어오고, 자식의 PID를 프로세스 테이블에서 완전히 삭제(Reaping)한다.

### 3. ASCII 구조 다이어그램
```text
    [ Parent ]                          [ Child ]
    ┌────────────────┐                  ┌────────────────┐
    │  wait() 호출    │ ───(대기)────▶  │  실행 중..      │
    │                │                  ├────────────────┤
    │                │  ◀──(exit 알림)── │  exit(0) 호출   │
    ├────────────────┤                  └────────────────┘
    │ 자식 상태 확인  │                  (좀비 상태로 대기)
    │ 자식 정보 삭제  │ ──▶ [ Kernel: Remove PCB ]
    └────────────────┘
```

- **📢 섹션 요약 비유**: 바톤 터치를 하는 육상 경기와 같습니다. 앞 주자(자식)가 골인 지점(exit)에 들어와도, 뒤 주자(부모)가 바톤(상태 코드)을 넘겨받아야(wait) 비로소 경기가 깔끔하게 마무리되는 것과 같습니다.

## Ⅲ. 프로세스 종료 시 자원 회수 절차

| 단계 | 수행 내용 | 비고 |
|:---|:---|:---|
| **1. 상태 기록** | 종료 코드(0: 정상, 그 외: 에러)를 PCB에 기록 | 부모 전달용 |
| **2. 자원 해제** | 물리 메모리, 파일, 세마포어 등 반납 | 커널이 자동 수행 |
| **3. 좀비 전이** | 프로세스 테이블 내 항목만 유지 | 최소한의 존재 증명 |
| **4. 최종 삭제** | 부모의 `wait` 호출 시 PCB 완전 소멸 | 프로세스 테이블 비움 |

- **📢 섹션 요약 비조**: "몸체(메모리)는 즉시 사라지지만, 영혼(PCB 정보)은 부모의 마지막 확인을 기다리며 구천(프로세스 테이블)을 떠돕니다."

+++

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 좀비 프로세스 대량 발생 조치
- **상황**: 시스템에 좀비 프로세스가 수백 개 생겨서 프로세스 생성이 안 됨.
- **판단**: 부모 프로세스가 루프를 도느라 바빠서 자식의 종료를 처리해주는 **wait()**를 호출하지 않았기 때문이다. 기술사라면 부모 프로세스에 `SIGCHLD` 시그널 핸들러를 등록하여 자식 종료 시 즉각 비동기적으로 `wait`를 수행하게 하거나, 자식 생성 시 `double fork` 기법을 써서 자식을 고아로 만들어 `init`이 처리하게 하는 아키텍처적 해법을 제시해야 한다.

### 2. 기술사적 인사이트: 연쇄 종료 (Cascading Termination)
- 기술사라면 일부 운영체제나 쉘 환경에서 부모가 종료될 때 자식들을 모두 함께 죽이는 **연쇄 종료** 정책을 논해야 한다. 이는 고아 프로세스가 시스템 자원을 점유하는 것을 막기 위한 강제적 질서 유지 기법이다. 반면, 서버 데몬 프로세스처럼 부모가 죽어도 계속 돌아가야 하는 경우 `setsid()`를 통해 새로운 세션을 만들어 독립시키는 설계 능력이 중요함을 강조해야 한다.

- **📢 섹션 요약 비조**: "떠날 때를 아는 프로세스의 뒷모습은 아름답습니다. 그 뒷모습을 정갈하게 치워주는 것이 부모 프로세스와 운영체제의 공동 책임입니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **자원 선순환**: 사용하지 않는 메모리와 I/O 자원을 즉시 회수하여 시스템 효율성 유지.
- **실행 흐름 동기화**: 작업의 완료 여부를 부모에게 명확히 전달하여 후속 작업 연계 지원.

### 2. 미래 전망
앞으로는 서버리스나 마이크로 VM 환경에서 프로세스 종료 후 즉시 메모리를 제로화(Zeroing)하여 보안을 강화하는 **Secure-Termination** 기술이 표준이 될 것이다. 또한, 비정상 종료 시 원인을 AI가 분석하여 자동 복구하는 지능형 커널 가디언 기술이 도입될 것이다. 프로세스 종료는 끝이 아니라, 새로운 프로세스에게 자원을 넘겨주는 시스템의 건강한 대사 작용이다.

- **📢 섹션 요약 비유**: 낙엽이 져야 내년에 새싹이 돋듯, 종료 시스템이 건강해야 운영체제라는 거대한 나무가 끊임없이 성장할 수 있습니다.

2. 일을 다 하면 "다 했어요!"라고 쪽지(`exit`)를 남기고, 엄마(부모)가 그 쪽지를 읽어줘야(`wait`) 완전히 끝난 게 되죠.
3. 책상을 잘 치워야 다음 친구가 와서 또 즐겁게 공부할 수 있답니다!
