# 프로세스 제어 블록 (PCB) / 태스크 제어 블록 (TCB)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PCB는 운영체제가 각 프로세스를 관리하기 위해 필요한 모든 정보를 담고 있는 **커널 내의 데이터 구조(Process Descriptor)**로, 프로세스의 '주민등록증'이자 '상태 기록부' 역할을 한다.
> 2. **가치**: 문맥 교환(Context Switch) 발생 시 현재 프로세스의 실행 상태(PC, 레지스터 등)를 PCB에 보관하고, 다음 프로세스의 PCB 정보를 복구함으로써 **프로세스 실행의 연속성**을 보장한다.
> 3. **융합**: 스레드 단위의 관리가 필요한 현대 OS에서는 각 스레드별 정보를 담은 **TCB (Thread Control Block)**와 연계되어 다중 프로그래밍의 효율성과 안정성을 동시에 제공한다.

---

## Ⅰ. PCB의 정의 및 생성 주기

### 1. 정의
- 특정 프로세스를 제어하기 위해 운영체제가 해당 프로세스에 대해 유지하는 정보를 담는 커널 메모리의 블록이다.

### 2. 생성 및 소멸 주기
- **생성**: 프로세스가 생성(New)될 때 커널 메모리(주로 커널 스택 근처)에 할당된다.
- **활용**: 프로세스가 실행, 대기, 준비 상태를 오가는 동안 수시로 갱신된다.
- **소멸**: 프로세스가 종료(Terminated)되면 부모 프로세스가 상태를 확인한 후 시스템에 의해 반납된다.

### 3. 💡 비유: '병원의 환자 차트'
- PCB는 병원에서 의사(CPU)가 환자(프로세스)를 진료할 때 기록하는 **'진료 차트'**와 같다.
- 진료 도중 다른 환자를 봐야 할 때(문맥 교환), 지금까지의 처방과 상태(레지스터, PC)를 차트(PCB)에 적어두어야 나중에 다시 그 환자가 왔을 때 끊김 없이 치료를 이어갈 수 있다.

## II. PCB의 주요 구성 요소 (Deep Dive)

커널 소스(리눅스의 `task_struct` 등)에서는 더욱 복잡하지만, 핵심 요소는 다음과 같다.

### 1. 프로세스 관리 정보
- **PID (Process ID)**: 시스템 내 유일한 프로세스 식별자.
- **프로세스 상태 (State)**: 생성, 준비, 실행, 대기, 종료 상태 정보.
- **우선순위 (Priority)**: 스케줄링을 위한 순위 정보.

### 2. 실행 문맥 (Context) 정보
- **프로그램 카운터 (PC)**: 이 프로세스가 다음에 실행할 명령어의 주소.
- **CPU 레지스터**: 누산기, 인덱스 레지스터, 범용 레지스터 등의 현재 값.

### 3. 자원 관리 정보
- **메모리 정보**: 페이지 테이블 정보, 세그먼트 주소 등.
- **I/O 상태**: 할당된 입출력 장치, 열린 파일 목록(File Descriptor) 등.

### 4. ASCII 구조 다이어그램
```text
    ┌───────────────────────────┐
    │      Process ID (PID)     │
    ├───────────────────────────┤
    │      Process State        │
    ├───────────────────────────┤
    │   Program Counter (PC)    │ ◀── 문맥 교환의 핵심
    ├───────────────────────────┤
    │      Registers (Saved)    │
    ├───────────────────────────┤
    │    Memory Management      │ ◀── Page Table, etc.
    ├───────────────────────────┤
    │    Accounting / Scheduling│
    ├───────────────────────────┤
    │    I/O Status (Files)     │
    └───────────────────────────┘
```

- **📢 섹션 요약 비유**: 이력서에 내 이름(PID)뿐만 아니라, 어제 어디까지 일했는지(PC), 내 가방에 뭐가 들어있는지(자원 목록)를 꼼꼼히 적어둔 마법의 수첩과 같습니다.

## Ⅲ. TCB (Thread Control Block)와의 관계

현대 운영체제는 프로세스 내부에 여러 스레드를 운영하므로 PCB와 TCB가 협력한다.

| 구분 | PCB (Process) | TCB (Thread) |
|:---|:---|:---|
| **소유권** | 프로세스 전체의 공유 자원 관리 | 개별 스레드의 독립적 실행 정보 관리 |
| **포함 내용** | 페이지 테이블, 열린 파일, PID 등 | Thread ID, 스택 포인터, 레지스터, PC 등 |
| **관계** | **하나의 PCB는 여러 TCB를 가리킴** | TCB는 PCB의 공통 자원을 공유함 |

- **📢 섹션 요약 비유**: PCB가 '집 한 채(공용 거실, 주방)'에 대한 관리 대장이라면, TCB는 그 집에 사는 '각 가족 구성원(개인 방, 옷)'에 대한 활동 기록지입니다.

---

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 문맥 교환 오버헤드 최적화
- **상황**: 시스템 부하가 높을 때 문맥 교환 비용이 급증함.
- **판단**: PCB/TCB 정보가 너무 비대해지면, 이를 메모리에 쓰고 읽는 오버헤드가 커진다. 기술사라면 이를 해결하기 위해 **레지스터 윈도우(Register Window)** 기술이나 하드웨어가 지원하는 고속 문맥 저장 기능을 연계하여 설명해야 한다. 또한, PCB가 위치한 커널 메모리의 캐시 적중률(Locality)이 성능의 핵심임을 인지해야 한다.

### 2. 기술사적 인사이트: 리눅스의 task_struct
- 기술사라면 리눅스 커널에서 PCB/TCB의 경계가 모호해진 **'Task'** 개념을 논해야 한다. `task_struct`라는 단일 구조체가 프로세스와 스레드를 모두 표현하며, `clone()` 시스템 콜의 플래그에 따라 자원 공유 여부만 결정되는 현대적인 통일 모델(Task-centric)을 이해하는 것이 중요하다.

- **📢 섹션 요약 비조**: 규격화된 서류(PCB)를 하나씩 넘기며 일하는 것보다, 필요한 정보만 빠르게 복사해 쓰는 유연한 구조(LWP 모델)가 현대 고성능 OS의 핵심입니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **멀티태스킹 구현**: CPU 자원을 시분할하여 여러 프로세스가 동시에 도는 것처럼 착각을 일으키는 물리적 토대.
- **시스템 안정성**: 프로세스 간의 정보를 완벽히 격리하고 보호하여 장애 전파 방지.

### 2. 미래 전망
앞으로는 보안 가상화 기술(TEE, Enclave)과 결합하여, PCB의 일부 실행 문맥을 하드웨어적으로 암호화하여 보호하는 **Secure-PCB** 기술이 발전할 것이다. 또한, 수천 개의 코어가 탑재되는 Many-core 환경에서 PCB를 코어별로 분산 관리하여 경합을 줄이는 기술이 더욱 중요해질 것이다. PCB는 운영체제가 '혼돈의 프로세스 세계'를 다스리기 위해 만든 가장 정교한 장부이다.

- **📢 섹션 요약 비유**: 수백 명의 배우(프로세스)가 무대(CPU)를 오가는 연극에서, 각자의 대본과 소품 위치를 완벽히 기억하게 돕는 최고의 '무대 감독 수첩'입니다.
