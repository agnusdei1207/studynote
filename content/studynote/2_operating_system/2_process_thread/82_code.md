# NX Bit / DEP (Data Execution Prevention)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하드웨어 수준에서 데이터 영역(스택, 힙)의 코드 실행을 차단하는 메모리 보호 기술
> 2. **가치**: 버퍼 오버플로우를 통한 셸코드 실행을 원천적으로 방지하여 가장 효과적인 방어책
> 3. **융합**: CPU의 XD 비트(Intel), NX 비트(AMD), ARM의 XN 비트와 운영체제의 DEP가 협력

---

### Ⅰ. 개요 (Context & Background)

**개념**
NX Bit(No-Execute Bit) 또는 DEP(Data Execution Prevention)는 **메모리 페이지를 실행 가능(Executable)과 실행 불가능(Non-Executable)로 구분**하여, 데이터 영역(스택, 힙)에서의 코드 실행을 하드웨어 수준에서 차단하는 보안 기술이다. 셸코드 인젝션 공격을 방어하는 가장 효과적인 수단이다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         NX Bit 기본 작동 원리                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   NX Bit가 없는 시스템 (취약)                                              │
│   ─────────────────────────────────                                         │
│   ┌─────────────────────────────────────────────────────────────────┐      │
│  │                        메모리 맵                                │      │
│  │                                                                   │      │
│  │   .text 섹션     │  코드  │  ◀─── 실행 가능 ◀───▶               │      │
│  │   .data 섹션     │  데이터 │  ◀─── 실행 가능 ◀───▶               │      │
│  │   Stack (스택)   │  데이터 │  ◀─── 실행 가능 ◀───▶               │      │
│  │   Heap (힙)     │  데이터 │  ◀─── 실행 가능 ◀───▶               │      │
│  │                                                                   │      │
│  │   → 셸코드를 스택에 넣고 실행 가능!                              │      │
│  │                                                                   │      │
│  └───────────────────────────────────────────────────────────────────┘      │
│                                                                             │
│   NX Bit가 있는 시스템 (보안)                                              │
│   ─────────────────────────────────                                         │
│   ┌─────────────────────────────────────────────────────────────────┐      │
│  │                        메모리 맵                                │      │
│  │                                                                   │      │
│  │   .text 섹션     │  코드  │  ◀─── 실행 가능 ◀───▶               │      │
│  │   .data 섹션     │  데이터 │  ◀─── 실행 불가 ✗                  │      │
│  │   Stack (스택)   │  데이터 │  ◀─── 실행 불가 ✗                  │      │
│  │   Heap (힙)     │  데이터 │  ◀─── 실행 불가 ✗                  │      │
│  │                                                                   │      │
│  │   → 셸코드를 스택에 넣어도 실행 불가!                             │      │
│  │     → Segmentation Fault 발생                                      │      │
│  │                                                                   │      │
│  └───────────────────────────────────────────────────────────────────┘      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**💡 비유**
NX Bit는 **쓰기 전용 종이와 실행 전용 종이**를 구분하는 것과 같다.
- 실행 전용(코드 영역): 프로그램만 작성된 종이
- 쓰기 전용(데이터 영역): 메모장용 종이
- 보안: 메모장에 악성코드를 써도 실행할 수 없음

**등장 배경**
① **2000년대 초**: 셸코드 인젝션 공격 급증
② **2004년**: AMD AMD64(No-Execute), Intel XD(Execute Disable) 발표
③ **2004년**: Windows XP SP2에 DEP 도입
④ **현재**: 거의 모든现代 CPU, OS에서 표준 지원

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소**

| 요소 | 역할 | 내부 동작 | CPU 지원 | 비유 |
|:---|:---|:---|:---|:---|
| **NX/XD Bit** | 페이지 실행 권한 | PTE의 NX 비트 | AMD64, Intel EM64T | 금지 표시 |
| **硬件 페이징** | 페이지 단위 보호 | CR3, PDE, PTE | x86, ARM | 구역 분리 |
| **DEP (Windows)** | 정책 관리 | 영구, opt-out, opt-in | Windows 이상 | 관리자 |
| **Exec-shield** | Linux 구현 | SEGMENTATION 활용 | Linux 2.6+ | 리눅스용 |

**ASCII 구조 다이어그램**

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                       페이지 테이블 엔트리와 NX Bit                             │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  x86-64 페이지 테이블 구조                                                   │
│  ────────────────────────────────────────────────────────────────────────   │
│                                                                              │
│  PML4 (Page Map Level 4)                                                    │
│    │                                                                        │
│    └─ PDP (Page Directory Pointer)                                         │
│           │                                                                 │
│           └─ PD (Page Directory)                                           │
│                  │                                                          │
│                  └─ PT (Page Table)                                        │
│                         │                                                 │
│                         └─ Physical Page                                   │
│                                                                              │
│  각 레벨의 PTE(Page Table Entry) 구조                                       │
│  ────────────────────────────────────────────────────────────────────────   │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Bit 63-52 │ NX │ AVL │ G │ PAT │ D │ A │ PCD │ PWT │ U/S │ R/W │ P │   │
│  │     ↑       │    │     │   │     │   │   │     │     │     │     │ │   │
│  │  예약      │ No-Exec│ Available│ Global│ Dirty│ Accessed│         │ │   │
│  │                                                │ User/Supervisor│   │   │
│  │                                                │ Read/Write      │   │   │
│  │                                                │ Present         │   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│  NX Bit 설정 예시                                                            │
│  ────────────────────────────────────────────────────────────────────────   │
│  • .text 섹션: NX=0 (실행 가능)                                             │
│  • .data 섹션: NX=1 (실행 불가)                                             │
│  • Stack: NX=1 (실행 불가)                                                  │
│  • Heap: NX=1 (실행 불가)                                                   │
│                                                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                   NX Bit가 동작하는 시나리오                         │   │
│  ├─────────────────────────────────────────────────────────────────────┤   │
│  │                                                                     │   │
│  │  1. 공격자가 버퍼 오버플로우로 셸코드를 스택에 삽입                 │   │
│  │                                                                     │   │
│  │     스택: [NOP Sled][Shellcode][Return Address]                      │   │
│  │                                                                     │   │
│  │  2. 함수가 반환하면서 셸코드 주소로 점프                             │   │
│  │                                                                     │   │
│  │  3. CPU는 해당 페이지의 PTE를 확인                                   │   │
│  │     → NX Bit가 1로 설정됨 (실행 불가)                                │   │
│  │                                                                     │   │
│  │  4. 하드웨어는 예외(Exception) 발생                                    │   │
│  │     → Linux: SIGSEGV, Windows: STATUS_ACCESS_VIOLATION               │   │
│  │     → 프로그램 종료                                                 │   │
│  │                                                                     │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

**해설**
1. **하드웨어 지원**: CPU가 페이지별 실행 권한 관리
2. **페이지 단위 보호**: 4KB 단위로 실행 가능/불가 지정
3. **예외 처리**: 실행 불가 영역에서의 코드 실행 시 즉시 종료

**심층 동작 원리: DEP 모드별 동작**

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                         Windows DEP 정책 모드                                 │
├──────────────────────────────────────────────────────────────────────────────┤

│  1. AlwaysOff (DEP 비활성)                                                   │
│  ────────────────────────────────────────────────────────────────────────   │
│  • 모든 프로그램에서 DEP 비활성                                              │
│  • 호환성을 위해 사용 (예전 프로그램)                                        │
│                                                                              │
│  2. OptIn (기본값)                                                           │
│  ────────────────────────────────────────────────────────────────────────   │

│                                                                              │
│  3. OptOut                                                                  │
│  ────────────────────────────────────────────────────────────────────────   │
│  • 모든 프로그램에 DEP 적용                                                  │

│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │                    DEP 설정 확인 (Windows)                           │  │
│  ├──────────────────────────────────────────────────────────────────────┤  │

│  │                                                                      │  │
│  │  현재 상태 확인:                                                      │  │
│  │  > wmic OS Get DataExecutionPrevention_Available                      │  │
│  │  > wmic OS Get DataExecutionPrevention_SupportPolicy                  │  │
```

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**심층 기술 비교: 실행 제어 기법**

| 기법 | 보호 대상 | 우회 가능성 | 성능 영향 | 범용성 |
|:---|:---|:---|:---|:---|
| **NX/DEP** | 스택/힙 실행 방지 | ROP으로 우회 | 미미 | 매우 높음 |
| **Stack Canary** | 반환 주소 보호 | 정보 누출로 우회 | 미미 | 높음 |
| **ASLR** | 주소 예측 방지 | 정보 누출로 우회 | 미미 | 높음 |
| **SafeSEH** | 예외 처리 체인 | 메모리 누출로 우회 | 미미 | 중간 |
| **CET (Control-flow Enforcement)** | ROP 방지 | 아직 우회 없음 | 미미 | 낮음 (최신 HW) |

**과목 융합 관점**

1. **컴퓨터 구조**: MMU, 페이징, TLB와 직접적 연관

2. **운영체제**: 메모리 관리, 프로세스 가상 주소 공간

3. **보안**: Defense in Depth의 핵심 요소

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**실무 시나리오**

**시나시오 1: DEP 활성화 확인**
- Windows: 시스템 속성 → 성능 → DEP 확인
- Linux: `/proc/cpuinfo`에서 nx 플래그 확인

**시나리오 2: 예외 처리**
- 일부 프로그램(예전 게임, JIT)는 DEP와 충돌
- 개별 예외 추가로 해결

**도입 체크리스트**

**기술적 측면**
- [ ] NX Bit 지원 CPU 확인
- [ ] DEP/NX 활성화
- [ ] 실행 불가 페이지 설정 확인
- [ ] 컴파일러 NX 옵션 활성화
- [ ] JIT 컴파일러 재설계

**운영·보안적 측면**
- [ ] DEP 예외 최소화
- [ ] 정기적 NX 상태 확인
- [ ] 보안 모니터링

**안티패턴**
1. **"호환성 문제"**: DEP를 끄지 말고 예외 처리
2. **"성능 저하"**: 영향 거의 없음
3. **"JIT 때문에 불가"**: W^X 전략으로 가능

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**정량/정성 기대효과**

| 지표 | DEP 비활성 | DEP 활성 | 개선 효과 |
|:---|:---|:---|:---|
| **셸코드 실행 가능성** | 100% | 0% | 100% 차단 |
| **버퍼 오버플로우 위험도** | 높음 | 낮음 (ROP 필요) | 90% 감소 |
| **호환성 문제** | 없음 | 일부 프로그램 | 예외로 해결 |

**미래 전망**
1. **CET(Control-flow Enforcement Technology)**: Intel의 ROP 방지
2. **Shadow Stack**: 반환 주소 별도 저장
3. **ARM PAC/MSC**: 포인터 인증

**참고 표준**
- **AMD64 Architecture**: NX Bit 정의
- **Intel SDM**: XD Bit 설명
- **MSDN**: DEP 기술 문서
