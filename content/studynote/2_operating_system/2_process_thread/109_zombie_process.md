# 좀비 프로세스 (Zombie Process)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 좀비 프로세스는 실행이 종료되어 메모리와 자원은 반납했으나, **부모 프로세스가 종료 상태를 확인(`wait`)하지 않아 프로세스 테이블에 항목이 남아있는 상태**의 프로세스이다.
> 2. **가치**: 자식 프로세스가 남긴 마지막 정보(종료 코드 등)를 부모가 읽어갈 수 있도록 보존하는 유예 기간 역할을 하지만, 장시간 방치될 경우 프로세스 ID(PID) 고갈 문제를 야기할 수 있다.
> 3. **융합**: 부모 프로세스가 `wait()` 시스템 콜을 호출하여 종료 상태를 수거(Reaping)하는 순간 좀비는 완전히 소멸하며, 부모가 먼저 죽을 경우 `init` 프로세스에 의해 수거된다.

---

## Ⅰ. 좀비 프로세스의 정의 및 발생 원인

### 1. 정의
- 프로세스 상태 중 'Terminated'에 해당하지만, 시스템 리소스(Process Table Entry)를 여전히 점유하고 있는 유령 같은 프로세스이다. 리눅스의 `ps` 명령어에서는 `<defunct>` 상태로 표시된다.

### 2. 발생 원인: '부모의 무관심'
- 자식 프로세스가 `exit()`를 호출하면 커널은 프로세스가 사용하던 메모리, 파일 등을 모두 회수한다.
- 하지만 자식의 **종료 상태 정보**는 부모에게 전달하기 위해 PCB에 남겨둔다.
- 부모 프로세스가 `wait()` 또는 `waitpid()`를 호출하여 이 정보를 읽어가지 않으면, 자식 프로세스는 죽었으나 완전히 사라지지 못한 '좀비'가 된다.

### 3. 💡 비유: '수령해가지 않은 졸업장'
- 좀비 프로세스는 **'학교를 졸업(종료)해서 교실(메모리)은 비웠지만, 부모님이 졸업장(종료 코드)을 받으러 오지 않아 행정실 명단(프로세스 테이블)에 이름이 남아있는 졸업생'**과 같다.
- 부모님이 와서 졸업장을 챙겨야 비로소 학교 명단에서 이름이 지워진다.

## II. 좀비 프로세스의 관리 및 영향 (Deep Dive)

### 1. 시스템에 미치는 영향
- **메모리 소모**: 미미함 (최소한의 PCB 정보만 남음).
- **PID 고갈**: **치명적.** 프로세스 테이블의 크기는 유한하며 PID 개수도 한정되어 있다. 좀비가 가득 차면 더 이상 새로운 프로세스를 생성할 수 없게 된다.

### 2. 해결 방법 (Reaping)
- **부모의 책임**: 자식 생성 후 반드시 `wait()`를 호출한다.
- **비동기 처리**: `SIGCHLD` 시그널 핸들러를 등록하여 자식이 죽는 즉시 자동으로 `wait`를 수행하게 한다.
- **부모 강제 종료**: 부모가 죽으면 좀비 자식들은 '고아'가 되어 `init` 프로세스에 입양되며, `init`은 주기적으로 이들을 수거한다.

### 3. ASCII 구조 다이어그램
```text
    [ Parent Process ]                  [ Child Process ]
    ┌────────────────┐                  ┌────────────────┐
    │  Keep Running  │                  │  exit(0) 호출   │
    │  (No wait())   │                  └────────┬───────┘
    └────────┬───────┘                           │
             │                                   ▼
             │                          ┌────────────────┐
             │                          │ Zombie State   │
             │                          │ (Defunct)      │ ◀── PID 점유 중
             │                          └────────────────┘
             │                                   │
             └─────────── wait() 호출 ───────────┘
                               │
                               ▼
                    [ Child Process Removed ]
```

- **📢 섹션 요약 비유**: 시신(자원)은 거두었으나 사망 신고(wait)가 접수되지 않아 호적(프로세스 테이블) 상에 살아있는 사람으로 남아있는 행정적 오류 상태와 같습니다.

## Ⅲ. 좀비 vs 고아 프로세스 비교

| 비교 항목 | 좀비 프로세스 (Zombie) | 고아 프로세스 (Orphan) |
|:---|:---|:---|
| **상태** | **이미 종료됨** | **여전히 실행 중** |
| **원인** | 부모가 `wait()`를 안 함 | 부모가 먼저 죽음 |
| **위험성** | PID 고갈 위험 | 자원 낭비 (수동 종료 필요할 수도) |
| **수거 주체** | 부모 프로세스 | `init` 프로세스 (PID 1) |

- **📢 섹션 요약 비조**: "좀비는 '확인이 필요한 시체'이고, 고아는 '보호자가 사라진 아이'입니다. 둘 다 운영체제가 해결해야 할 관리 대상입니다."

---

## Ⅳ. 실무 적용 및 프로그래밍 인사이트 (Strategy)

### 1. 실무 시나리오: 잦은 fork를 수행하는 서버 프로그램
- **상황**: 대량의 접속마다 `fork`로 처리하는 구형 서버에서 갑자기 신규 접속이 거부됨.
- **판단**: `ps -ef | grep defunct` 명령어로 확인한 결과 좀비 프로세스가 수천 개 존재함. 부모 프로세스 코드에 `wait` 로직이 누락되었거나, 시그널 처리가 막혀 있는 상태이다. 기술사라면 즉시 부모 프로세스를 재시작하거나, 커널 시그널 설정을 점검하여 좀비를 일괄 소멸시켜야 한다.

### 2. 기술사적 인사이트: Double Fork 기법
- 기술사라면 좀비 생성을 원천 차단하기 위한 **Double Fork** 기법을 언급해야 한다. 부모가 자식을 낳고, 그 자식이 다시 손자를 낳은 뒤 자식이 즉시 죽어버리면, 손자는 자동으로 `init`에 입양된다. 부모는 자식 한 명만 관리하면 되고, 실제 작업을 수행하는 손자는 `init`이 책임지고 수거하므로 좀비 문제를 우아하게 회피할 수 있다.

- **📢 섹션 요약 비조**: "부모가 자식을 끝까지 챙길 수 없다면, 국가(init)에 맡기는 대리 입양 전략(Double Fork)이 대규모 시스템 설계의 지혜입니다."

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **시스템 가용성 유지**: 프로세스 테이블 공간 확보를 통한 지속적 서비스 가능.
- **상태 정보의 안전한 전달**: 자식의 실패 원인을 부모가 정확히 파악하여 복구 로직 가동.

### 2. 미래 전망
앞으로는 컨테이너 오케스트레이션(K8s) 환경에서 사이드카(Sidecar) 컨테이너가 메인 컨테이너의 프로세스들을 감시하고 좀비를 수거하는 **자동화된 Reaping 서비스**가 내재화될 것이다. 또한, 자원 격리가 강화된 마이크로 VM에서는 좀비 발생 시 하드웨어 트랩을 통해 즉시 소멸시키는 기술이 연구되고 있다. 좀비 프로세스는 운영체제의 '완벽한 정리 정돈'을 위해 꼭 거쳐야 하는 필연적인 과도기 상태이다.

- **📢 섹션 요약 비유**: 다 쓴 물건을 재활용 함(좀비 상태)에 넣어두면, 누군가(부모)가 확인하고 비워줘야(wait) 창고(시스템)가 깨끗해지는 원리와 같습니다.
