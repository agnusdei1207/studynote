# RPC (Remote Procedure Call) - 원격 프로시저 호출

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RPC (Remote Procedure Call)는 네트워크로 연결된 다른 컴퓨터에 있는 함수(프로시저)를 실행할 때, 개발자가 네트워크 통신의 상세 로직을 몰라도 **마치 로컬 함수를 호출하듯** 사용할 수 있게 해주는 원격 통신 기술이다.
> 2. **가치**: 소켓 통신의 복잡한 메시지 생성 및 전송 과정을 운영체제와 라이브러리 계층으로 추상화하여 분산 애플리케이션의 개발 생산성을 획기적으로 높인다.
> 3. **융합**: 송수신 양측의 인터페이스를 맞추는 **스터브 (Stub)** 메커니즘을 사용하며, 현대 마이크로서비스 아키텍처(MSA)의 핵심인 gRPC 등의 기술적 모태가 된다.

+++

## Ⅰ. RPC의 개요

### 1. 정의
- 별도의 원격 제어를 위한 코딩 없이 다른 주소 공간(보통 다른 물리적 장치)에서 함수나 프로시저를 실행할 수 있게 하는 프로세스 간 통신 기술이다.

### 2. 등장 배경: '소켓의 복잡성 해결'
- 소켓 통신은 데이터 포맷 정의, 연결 관리, 에러 처리 등을 개발자가 일일이 코딩해야 한다. "그냥 함수 호출하듯이 네트워크 너머의 일을 시킬 수는 없을까?"라는 고민에서 RPC가 탄생했다.

### 3. 💡 비유: '분점 요리사에게 시키는 전화 주문'
- RPC는 본점 셰프(클라이언트)가 분점 셰프(서버)에게 전화를 걸어 **"내 레시피대로 파스타 하나만 만들어줘"**라고 요청하는 것과 같다.
- 본점 셰프는 분점 주방이 어떻게 생겼는지, 재료가 어디 있는지 몰라도(네트워크 상세 은닉) 결과물(반환값)을 받을 수 있다.

+++

## Ⅱ. RPC 동작 메커니즘: 스터브 (Stub) (Deep Dive)

RPC의 핵심은 중간 대리인인 **스터브 (Stub)**의 존재이다.

### 1. 클라이언트 스터브 (Client Stub)
- 호출자가 함수를 부르면 실제로는 이 스터브가 호출된다.
- 매개변수를 네트워크로 보낼 수 있는 형태(마샬링)로 변환하고 서버로 전송한다.

### 2. 서버 스터브 (Server Stub / Skeleton)
- 네트워크 패킷을 받아 다시 원래의 매개변수 형태로 복원(언마샬링)한다.
- 실제 서버 내의 로컬 함수를 실행하고, 결과값을 다시 마샬링하여 클라이언트에게 보낸다.

### 3. ASCII 구조 다이어그램
```text
    [ Client Machine ]                      [ Server Machine ]
    ┌────────────────┐                      ┌────────────────┐
    │ Application    │                      │ Service Logic  │
    └──────┬─────────┘                      └────────▲───────┘
           │ (Call func)                             │ (Invoke)
    ┌──────▼─────────┐                      ┌────────┴───────┐
    │ Client Stub    │                      │ Server Stub    │
    │ (Marshalling)  │                      │ (Unmarshalling)│
    └──────┬─────────┘                      └────────▲───────┘
           │                                         │
    ┌──────▼─────────┐      (Network)       ┌────────┴───────┐
    │ OS / RPC Runtime│ ◀──────────────────▶ │ OS / RPC Runtime│
    └────────────────┘                      └────────────────┘
```

- **📢 섹션 요약 비유**: 클라이언트 스터브는 해외 배송 대행업체와 같습니다. 내가 물건을 주면(함수 호출), 업체가 해외용으로 포장(마샬링)하여 배를 태우고(전송), 현지 업체(서버 스터브)가 포장을 뜯어 배달(함수 실행)하는 과정입니다.

## Ⅲ. RPC의 기술적 당면 과제

| 과제 | 설명 | 해결책 |
|:---|:---|:---|
| **데이터 표현** | CPU마다 정수 저장 방식(Endian)이 다름 | **XDR (External Data Rep.)** 표준 사용 |
| **장애 처리** | 네트워크가 끊기거나 서버가 죽을 수 있음 | 재시도 정책 (At-most-once, At-least-once) |
| **위치 투명성** | 서버의 주소가 바뀔 수 있음 | **네임 서버 / 바인딩 (Binding)** 서비스 활용 |

- **📢 섹션 요약 비유**: 나라마다 쓰는 언어(데이터 표현)가 다르므로 공용어(XDR)를 사용하고, 편지가 안 가면 다시 보내는(재시도 정책) 약속이 필요합니다.

+++

## Ⅳ. 실무 적용 및 현대적 진화 (Strategy)

### 1. 실무 시나리오: 마이크로서비스 간 데이터 요청
- **상황**: 주문 서비스가 사용자 서비스에 '사용자 주소' 정보를 요청해야 함.
- **판단**: 직접 DB를 뒤지는 대신 **RPC(또는 gRPC)**를 사용한다. 주문 서비스는 `getUserAddress(userId)`라는 함수를 호출하기만 하면 된다. 내부적으로는 Protocol Buffers를 통한 고속 마샬링이 일어나며, 서버 간의 통신 부하를 최소화하면서도 객체 지향적인 코드 구조를 유지한다.

### 2. 기술사적 인사이트: 위치 투명성 (Location Transparency)
- RPC의 궁극적 목표는 **위치 투명성**이다. 기술사라면 호출자가 해당 프로시저가 로컬에 있는지 원격에 있는지 알 필요가 없어야 함을 강조해야 한다. 이를 위해 동적 바인딩(Dynamic Binding)을 지원하는 **포트매퍼 (Portmapper)**나 **서비스 메쉬 (Service Mesh)** 환경에서의 사이드카(Sidecar) 패턴과의 연동 방안을 제시할 수 있어야 한다.

- **📢 섹션 요약 비유**: 고객은 전화 한 통으로 피자를 시키지만, 그 피자가 옆집 가게에서 오는지 옆 동네 본사에서 오는지(위치 투명성)는 알 필요가 없는 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **개발 생산성 향상**: 저수준 네트워크 프로그래밍 작업의 90% 이상 제거.
- **시스템 유연성**: 서비스의 물리적 위치를 자유롭게 배치하며 분산 시스템 구축 용이.

### 2. 미래 전망
과거의 XML-RPC, SOAP을 거쳐 현재는 **gRPC**와 **JSON-RPC**가 클라우드 인프라를 지배하고 있다. 앞으로는 서버리스(Serverless) 환경에서 함수 단위의 실행이 더욱 미세화됨에 따라, 초저지연 RPC 기술은 운영체제와 클라우드 플랫폼이 제공해야 할 가장 핵심적인 통신 인프라가 될 것이다.

- **📢 섹션 요약 비유**: "모든 컴퓨터를 하나의 거대한 컴퓨터처럼" 쓰게 만드는, 운영체제의 확장된 팔과 다리입니다.
