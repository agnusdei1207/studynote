# 프로세스 상태 (Process State)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 상태 (Process State)는 프로세스가 생성되어 종료될 때까지의 생명 주기 동안 **CPU 점유 여부 및 가용 자원 상태**에 따라 변화하는 현재의 지위를 나타낸다.
> 2. **가치**: 운영체제는 프로세스의 상태를 추적함으로써 CPU 스케줄링의 대상을 선별하고, 입출력 완료나 인터럽트 발생 시 적절한 프로세스를 깨워 시스템 전체의 처리량(Throughput)을 최적화한다.
> 3. **융합**: 하드웨어 인터럽트와 시스템 호출에 반응하여 상태 전이(State Transition)가 일어나며, 이는 PCB(Process Control Block)에 기록되어 운영체제 관리 로직의 핵심 데이터로 활용된다.

+++

## Ⅰ. 프로세스 상태의 주요 단계 (5-State Model)

가장 기본적인 5가지 상태는 다음과 같다.

### 1. New (생성)
- 프로세스가 막 생성된 상태이다. OS가 PCB를 만들고 메모리 할당을 준비하지만, 아직 스케줄러에 의해 선택될 준비는 되지 않은 단계이다.

### 2. Ready (준비)
- 실행에 필요한 모든 자원을 갖추고 **CPU 할당만을 기다리는** 상태이다. 준비 큐(Ready Queue)에서 대기한다.

### 3. Running (실행)
- CPU를 점유하여 실제로 명령어를 실행 중인 상태이다.

### 4. Waiting / Blocked (대기 / 차단)
- 입출력(I/O) 완료나 특정 이벤트 발생을 기다리며 **스스로 실행을 중단한** 상태이다. CPU를 주어도 실행할 수 없는 상태이며, 대기 큐에서 대기한다.

### 5. Terminated (종료)
- 실행이 완료되어 자원을 반납하고 소멸하기 직전의 상태이다.

### 6. 💡 비유: '은행의 업무 처리 과정'
- **New**: 은행 문을 열고 들어와 번호표를 뽑는 단계.
- **Ready**: 내 번호가 불리기를 소파에 앉아 기다리는 단계.
- **Running**: 창구 직원 앞에 앉아 실제로 업무를 보는 단계.
- **Waiting**: 서류 서명이 필요해서 잠시 옆 책상으로 가서 서류를 작성하는 단계 (자리를 비켜줌).
- **Terminated**: 업무를 마치고 은행 문을 나가는 단계.

## Ⅱ. 프로세스 상태 전이 (State Transition) (Deep Dive)

프로세스는 특정 이벤트에 의해 상태가 변한다.

### 1. 주요 전이 과정
- **Admitted (New → Ready)**: 커널이 프로세스 생성을 승인하여 준비 큐에 등록함.
- **Dispatch (Ready → Running)**: 스케줄러가 준비 큐의 프로세스 중 하나를 선택하여 CPU를 할당함.
- **Interrupt / Timeout (Running → Ready)**: 할당된 시간(Time Slice)이 다 되었거나 높은 우선순위 프로세스가 나타나 CPU를 강제로 뺏김 (선점).
- **I/O or Event Wait (Running → Waiting)**: 프로세스가 입출력을 요청하거나 데이터를 기다려야 해서 CPU를 스스로 반납함.
- **I/O or Event Completion (Waiting → Ready)**: 기다리던 작업이 끝나 다시 실행될 자격을 갖춤. (주의: 바로 Running으로 가지 않음)

### 2. ASCII 상태 전이 다이어그램
```text
           [ Admitted ]             [ Dispatch ]
    (New) ──────▶ (Ready) ◀───────────────▶ (Running) ──▶ (Terminated)
                    ▲    └───────────────┐     │
                    │      [ Interrupt ] │     │ [ I/O or Event Wait ]
                    │                    │     │
                    └────── (Ready) ◀────┘     ▼
                                         (Waiting/Blocked)
                                               ▲
                                               │ [ I/O or Event Done ]
```

- **📢 섹션 요약 비유**: 달리기를 할 때 출발선에 서기(Ready), 실제로 뛰기(Running), 신발 끈이 풀려 잠시 멈추기(Waiting), 그리고 다시 출발선으로 돌아가기(Ready)의 반복과 같습니다.

## Ⅲ. 성능 최적화를 위한 확장 상태 (7-State Model)

메모리가 부족할 때 운영체제는 프로세스를 하드디스크로 잠시 쫓아내는데, 이를 **지연(Suspended)** 상태라 한다.

| 확장 상태 | 설명 |
|:---|:---|
| **Ready Suspended** | 준비 상태였으나 메모리가 부족해 디스크(Swap)로 쫓겨난 상태. |
| **Blocked Suspended** | 대기 상태였으나 디스크로 쫓겨난 상태. |

- **📢 섹션 요약 비유**: 은행 대기실이 너무 꽉 차서 "서류 다 쓰면 밖에서 기다리다가 부르면 들어오세요"라고 밖으로 내보내는(Swapping) 상황과 같습니다.

## Ⅳ. 실무 적용 및 스케줄링 인사이트 (Strategy)

### 1. 실무 시나리오: I/O 바운드 vs CPU 바운드
- **상황**: 어떤 서버 프로그램은 CPU 점유율은 낮은데 자꾸 멈추는 현상이 발생함.
- **판단**: 해당 프로세스는 **Waiting** 상태에 머무는 시간이 너무 긴 **I/O Bound** 프로세스일 가능성이 높다. 운영체제는 이러한 프로세스를 효율적으로 관리하기 위해, Waiting 상태에서 깨어나는 프로세스에게 높은 우선순위를 주어 짧은 CPU 사용 기회를 즉시 제공하는 스케줄링 전략을 사용한다.

### 2. 기술사적 인사이트: 문맥 교환 (Context Switch) 오버헤드
- 상태 전이가 너무 빈번하게 일어나면(Ready ↔ Running), 실제 연산 시간보다 상태를 저장하고 복구하는 **문맥 교환 오버헤드**가 커진다. 기술사라면 시스템 부하에 따라 타임 슬라이스(Time Slice) 크기를 조절하여, 상태 전이의 빈도와 응답성 사이의 균형을 맞추는 최적화 방안을 제시해야 한다.

- **📢 섹션 요약 비유**: 공부를 하다가 1분마다 전화를 받는다면(빈번한 상태 전이), 공부 내용은 머리에 남지 않고 전화를 껐다 켰다 하는 데 진을 다 빼게 되는 것과 같습니다.

+++

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **자원 활용 극대화**: 특정 프로세스가 자원을 기다리는 동안 다른 프로세스를 실행하여 CPU 유휴 시간 최소화.
- **응답성 향상**: 여러 상태를 유기적으로 관리하여 사용자에게 실시간성 환상 제공.

### 2. 미래 전망
최근의 서버리스(Serverless) 컴퓨팅 환경에서는 프로세스의 상태가 단순히 켜지고 꺼지는 것을 넘어, 상태를 완전히 동결(Freeze)했다가 다른 서버에서 즉시 재개(Resume)하는 **Check-point/Restore** 기술이 중요해지고 있다. 프로세스 상태 관리는 이제 단일 운영체제를 넘어 네트워크 기반의 분산 인프라 전체로 그 개념이 확장되고 있다.

- **📢 섹션 요약 비유**: 잠자는 숲속의 공주를 깨우듯, 언제 어디서든 프로세스의 시간을 멈췄다 다시 흐르게 하는 마법 같은 관리가 현대 운영체제의 미래입니다.
