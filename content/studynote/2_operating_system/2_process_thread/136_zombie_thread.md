# 좀비 스레드 (Zombie Thread)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 좀비 스레드는 실행이 종료되었으나, **다른 스레드에 의해 종료 상태가 확인(Join)되지 않아 커널 자원(스레드 ID, 종료 코드 등)이 회수되지 않고 남아있는 상태**의 스레드이다.
> 2. **가치**: 스레드 생성 시 'Joinable' 모드로 설정된 경우 발생하며, 종료된 스레드의 최종 결과값을 안전하게 수거할 수 있는 기회를 보장하지만, 방치될 경우 시스템 자원 누수의 원인이 된다.
> 3. **융합**: `pthread_join()`을 통해 명시적으로 자원을 회수하거나, `pthread_detach()`를 통해 종료 즉시 자동으로 자원을 반납하게 함으로써 시스템의 청결성을 유지한다.

---

## Ⅰ. 좀비 스레드의 정의 및 발생 조건

### 1. 정의
- 스레드 함수가 반환(Return)되거나 `pthread_exit()`를 호출하여 실행은 멈췄지만, 프로세스 내의 다른 스레드가 그 스레드의 종료를 기다려주지 않아(Wait/Join) 시스템 상에 유령처럼 남아있는 상태이다.

### 2. 발생 조건: 'Joinable Thread'
- POSIX 스레드(Pthreads)에서 스레드는 기본적으로 **Joinable** 상태로 생성된다.
- 이 상태의 스레드는 누군가가 `pthread_join()`을 호출해 줄 때까지 자신의 최소 정보를 커널에 유지해야 한다.

### 3. 💡 비유: '반납되지 않은 대여 도서'
- 좀비 스레드는 **'도서관에서 책을 다 읽고(실행 종료) 자리에 놔두고 갔지만, 사서가 반납 처리(Join)를 하지 않아 대출 목록에는 여전히 남아있는 상황'**과 같다.
- 책장은 비었지만 장부상에는 점유 중으로 표시되어, 나중에 다른 사람이 그 책(스레드 ID)을 빌릴 수 없게 된다.

---

## II. 좀비 스레드의 관리 및 해결 (Deep Dive)

### 1. 자원 수거 (Reaping)
- **pthread_join()**: 호출한 스레드는 대상 스레드가 종료될 때까지 대기하며, 종료 시 대상 스레드의 자원을 즉시 회수하고 결과값을 가져온다. (좀비 상태 방지)

### 2. 자동 반납 (Detaching)
- **pthread_detach()**: 스레드가 종료될 때 부모를 기다리지 않고 **즉시 자원을 스스로 반납**하게 설정한다. 결과값이 궁금하지 않은 단순 작업 스레드에 주로 사용한다.

### 3. ASCII 구조 다이어그램
```text
    [ Thread A ]                        [ Thread B (Target) ]
    ┌────────────────┐                  ┌────────────────┐
    │  Keep Working  │                  │  exit / return  │
    │  (No join call)│                  └────────┬───────┘
    └────────┬───────┘                           │
             │                                   ▼
             │                          ┌────────────────┐
             │                          │ Zombie Thread  │
             │                          │ (TCB remains)  │ ◀── TID 점유 중
             │                          └────────────────┘
             │                                   │
             └────────── pthread_join() ─────────┘
                               │
                               ▼
                    [ Thread B Fully Removed ]
```

- **📢 섹션 요약 비유**: 일을 마친 직원에게 퇴직금(종료 코드)을 주고 서류를 정리(Join)해야 비로소 인사 기록(TCB)이 지워지는 행정 절차와 같습니다.

## Ⅲ. 좀비 프로세스 vs 좀비 스레드 비교

| 비교 항목 | 좀비 프로세스 (Zombie) | 좀비 스레드 (Zombie Thread) |
|:---|:---|:---|
| **수거 주체** | 부모 프로세스 (`wait`) | 같은 프로세스 내 동료 스레드 (`join`) |
| **남는 자원** | PCB 항목, PID | TCB 항목, TID, 스택의 일부 정보 |
| **영향 범위** | 시스템 전체 (PID 고갈) | 프로세스 내부 (TID/메모리 부족) |
| **기본 설정** | 항상 수거 필요 | 생성 시 Detach 옵션으로 회피 가능 |

- **📢 섹션 요약 비조**: "좀비 프로세스는 국가적인 호적 문제라면, 좀비 스레드는 집안 내부의 가계부 정리 문제입니다. 어느 쪽이든 정리가 안 되면 파산(시스템 오류)에 이릅니다."

---

## Ⅳ. 실무 적용 및 프로그래밍 인사이트 (Strategy)

### 1. 실무 시나리오: 메모리 누수(Memory Leak) 진단
- **상황**: 오랫동안 켜둔 서버 프로그램이 시간이 갈수록 메모리 점유율이 올라가다가 결국 죽음.
- **판단**: 매 요청마다 스레드를 생성하여 작업하고 종료했는데, `pthread_join`이나 `pthread_detach` 중 어느 것도 호출하지 않았다면 **좀비 스레드**가 수만 개 쌓여 메모리를 갉아먹은 것이다. 기술사라면 코드 전수 조사를 통해 모든 스레드 생성 지점에 자원 회수 로직이 있는지 점검해야 한다.

### 2. 기술사적 인사이트: 화재 후 처리 (Signal Handling)
- 기술사라면 스레드가 예기치 않게 종료되었을 때의 좀비 처리를 논해야 한다. 메인 스레드가 비정상 종료된 스레드들을 감지하여 처리하지 못하면 시스템의 자원 가용성은 계속 떨어진다. 이를 방어하기 위해 **스레드 풀(Thread Pool)**을 사용하거나, 모든 작업 스레드를 생성 즉시 `detach` 상태로 만드는 설계가 대규모 동시성 처리의 정석임을 강조해야 한다.

- **📢 섹션 요약 비조**: "일꾼을 부리는 것보다, 일을 마친 일꾼을 안전하게 돌려보내는 뒷마무리가 시스템의 장기 생존을 결정합니다."

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **메모리 정합성 확보**: 불필요한 커널 메모리 점유 제거.
- **TID 가용성 유지**: 스레드 식별자 재사용을 통한 연속적인 스레드 생성 지원.

### 2. 미래 전망
앞으로는 언어 차원에서 스레드의 생명주기를 자동으로 관리하는 **구조적 동시성(Structured Concurrency)** 모델이 강화됨에 따라, 좀비 스레드와 같은 하위 레벨의 실수는 줄어들 것이다. Java의 가상 스레드나 Rust의 비동기 런타임은 이러한 좀비 문제를 설계적으로 원천 봉쇄하는 방향으로 발전하고 있다. 좀비 스레드는 운영체제가 우리에게 부여한 '뒷정리의 책임'을 상징하는 상태이다.

- **📢 섹션 요약 비유**: 다 쓴 촛불을 제때 꺼야(Join) 방 안이 연기로 가득 차지(자원 누수) 않고 쾌적하게 유지되는 것과 같습니다.
