# 스레드 (Thread) - 경량 프로세스 (LWP)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스레드(Thread)는 프로세스 내부에서 실행되는 **CPU 활용의 기본 단위**로, 프로세스가 할당받은 자원을 공유하면서 독립적인 실행 흐름(PC, 레지스터, 스택)을 갖는다.
> 2. **가치**: 프로세스 생성 및 문맥 교환보다 훨씬 적은 오버헤드로 다중 실행을 가능케 하여 **응답성 향상과 시스템 자원 절약**을 동시에 달성하는 '경량 프로세스(LWP)'의 역할을 수행한다.
> 3. **융합**: 현대 운영체제는 멀티코어 환경에서 성능을 극대화하기 위해 다중 스레드(Multithreading) 모델을 기본으로 채택하며, 이는 서버와 대규모 앱 설계의 핵심 기술이다.

---

## Ⅰ. 스레드의 정의 및 등장 배경

### 1. 정의
- 한 프로세스 내에서 제어의 흐름을 나타내는 단위이다. 하나의 프로세스는 하나 이상의 스레드를 가질 수 있으며, 이를 다중 스레드 프로세스라고 한다.

### 2. 등장 배경: '프로세스의 한계 극복'
- 과거에는 하나의 앱을 위해 여러 프로세스를 생성했으나, 프로세스 생성 비용이 크고 프로세스 간 통신(IPC)이 복잡하며 문맥 교환 오버헤드가 심각했다. 이를 해결하기 위해 **자원은 공유하되 제어 흐름만 분리**한 스레드 개념이 도입되었다.

### 3. 💡 비유: '한 집(프로세스) 안의 가족 구성원(스레드)'
- 프로세스는 **'집'**과 같고, 스레드는 그 집에 사는 **'가족 구성원'**과 같다.
- 거실, 냉장고, TV(주소 공간, 자원)는 가족 모두가 공유하지만, 각자 자기 방에서 공부를 하거나 잠을 자는 등(독립적인 실행 흐름)은 따로 한다. 새로 집을 짓는 것보다 가족이 늘어나는 것이 훨씬 효율적이다.

## II. 스레드의 내부 구조 (Deep Dive)

스레드는 공유하는 부분과 독립적인 부분을 명확히 구분한다.

### 1. 독립적 영역 (Private Context)
- **Thread ID**: 스레드 고유 식별자.
- **프로그램 카운터 (PC)**: 스레드가 다음에 실행할 명령어 주소.
- **레지스터 집합**: 현재 연산 상태 보존.
- **스택 (Stack)**: 함수 호출과 지역 변수를 저장하는 독립된 공간.

### 2. 공유 영역 (Shared Context)
- **코드 (Code)**: 실행할 프로그램 명령문 전체.
- **데이터 (Data)**: 전역 변수 및 정적 변수.
- **힙 (Heap)**: 동적으로 할당받은 메모리 공간.
- **기타 자원**: 열린 파일, 신호(Signal) 등.

### 3. ASCII 구조 다이어그램
```text
    [ Process Space ]
    ┌───────────────────────────────────────────────┐
    │ Code Segment | Data Segment | Heap Segment    │ ◀── Shared
    ├───────────────────┬───────────────┬───────────┤
    │  [ Thread 1 ]     │  [ Thread 2 ] │  ...      │
    │  - PC             │  - PC         │           │
    │  - Registers      │  - Registers  │           │ ◀── Independent
    │  - Stack          │  - Stack      │           │
    └───────────────────┴───────────────┴───────────┘
```

- **📢 섹션 요약 비유**: 각자 개인용 수첩(스택)과 볼펜(레지스터)은 따로 들고 다니지만, 도서관의 책(코드)과 책상(데이터/힙)은 같이 쓰는 시스템입니다.

## Ⅲ. 스레드 vs 프로세스 비교

| 비교 항목 | 프로세스 (Process) | 스레드 (Thread) |
|:---|:---|:---|
| **자원 할당** | 운영체제로부터 독립적 할당 | 프로세스 내 자원을 공유 |
| **생성 비용** | 매우 큼 (Heavyweight) | **매우 적음 (Lightweight)** |
| **통신 방식** | IPC 필요 (오버헤드 발생) | 공유 메모리를 통한 직접 통신 (빠름) |
| **문맥 교환** | 느림 (캐시 플러시 발생) | **빠름 (공유 자원 유지)** |
| **안정성** | 독립적이라 한쪽이 죽어도 안전 | **공유 자원 오염 시 전체 프로세스 영향** |

- **📢 섹션 요약 비유**: 대형 트럭(프로세스)을 여러 대 굴리는 것보다, 트럭 한 대에 여러 명의 배달원(스레드)이 타고 내리는 것이 물류 효율 면에서 압도적으로 유리합니다.

---

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 웹 서버 아키텍처 선택 (Process vs Thread)
- **상황**: 수만 명의 동시 접속자를 처리해야 하는 웹 서버 개발.
- **판단**: 멀티프로세스 방식(Apache 초기 모델)보다는 **멀티스레드 방식(Nginx, Java Servlet 등)** 혹은 비동기 이벤트 루프 방식을 선택한다. 스레드는 메모리 점유율이 낮고 사용자 간 전환이 빨라 대규모 동시성 처리에 적합하기 때문이다. 단, 스레드 간 동기화(Race Condition) 문제를 해결하는 것이 기술사의 핵심 역량이 된다.

### 2. 기술사적 인사이트: LWP (Lightweight Process)의 실체
- 기술사라면 솔라리스나 리눅스 커널에서의 **LWP** 개념을 짚어야 한다. 사용자 수준 스레드와 커널 수준 스레드를 연결하는 다리 역할을 하는 LWP는, 운영체제가 실제로 스케줄링하는 단위를 의미한다. 현대 리눅스는 NPTL(Native POSIX Thread Library)을 통해 일대일 매핑 모델을 사용하여 성능을 극대화하고 있음을 이해해야 한다.

- **📢 섹션 요약 비조**: 겉으로는 수많은 스레드가 떠다니는 것 같지만, 커널 속에서는 정교하게 설계된 일꾼들(LWP)이 이들을 등에 업고 CPU로 실어 나릅니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **경제성**: 프로세스 대비 자원 생성/관리 비용 10~100배 절감.
- **병렬성 극대화**: 멀티코어 하드웨어의 성능을 온전히 활용 가능한 구조 제공.

### 2. 미래 전망
앞으로는 수백만 개의 초경량 실행 단위를 관리하기 위한 **유저 레벨 스케줄링(Go의 Goroutine, Java의 Virtual Thread)** 기술이 더욱 보편화될 것이다. 운영체제 커널의 개입을 최소화하면서도 하드웨어의 병렬성을 뽑아내는 '스레드의 극단적 경량화'가 미래 컴퓨팅의 핵심 테마가 될 것이다. 스레드는 컴퓨터 과학이 복잡함을 이겨내고 효율을 쟁취한 가장 우아한 도구이다.

- **📢 섹션 요약 비유**: 한 명의 거인이 일하는 시대에서, 수만 명의 요정들이 한마음으로 움직이며 거대한 성을 쌓는 시대로의 진화입니다.
