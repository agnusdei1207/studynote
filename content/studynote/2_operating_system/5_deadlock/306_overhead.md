# 탐지 알고리즘의 오버헤드 (Overhead)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 교착 상태 탐지 알고리즘은 실행 시 상당한 CPU 시간과 메모리 자원을 소모하므로, 시스템 성능과 탐지의 정밀도 사이의 전략적 선택이 필수적이다.
> 2. **오버헤드**: 프로세스 수와 자원 수에 비례하여 계산량이 늘어나며, 특히 다중 인스턴스 환경에서는 복잡한 행렬 연산이 시스템의 실시간성을 저해할 수 있다.
> 3. **실행 시점**: 데드락 발생 빈도와 중요도를 고려하여 자원 요청 시, 주기적 시점, 또는 CPU 이용률 저하 시점 등 최적의 타이밍을 결정해야 한다.

+++

## Ⅰ. 탐지 알고리즘의 오버헤드 구성 요소

### 1. 연산 오버헤드
- 그래프 순회(DFS) 또는 행렬 연산(Safety Algorithm)을 수행하는 데 필요한 CPU 사이클. $O(N^2)$ 또는 $O(M \times N^2)$의 비용 발생.

### 2. 메모리 오버헤드
- 현재 자원 할당 상태를 저장하기 위한 Allocation, Request 행렬 등을 유지하는 비용.

- **📢 섹션 요약 비유**: "가게 주인이 장사가 잘되는지 확인하려고 손님 한 명 한 명을 다 조사하느라 정작 물건을 팔 시간이 줄어드는 것과 같습니다."

+++

## Ⅱ. 실행 빈도와 성능의 트레이드오프 (ASCII)

### 1. 잦은 실행 vs 드문 실행
```text
  [ High Frequency ]               [ Low Frequency ]
  (Check every request)            (Check every 1 hour)
  - Pros: 즉각적인 데드락 발견     - Pros: 평소 성능 저하 거의 없음
  - Cons: 엄청난 CPU 부하          - Cons: 데드락 피해가 커진 후 발견
  
  [ Optimal Balance ]
  * CPU 사용량 급감 시 (Wait 상태인 프로세스가 많음을 암시)
  * 특정 시간 간격 (Idle 타임 활용)
```

- **📢 섹션 요약 비유**: "건강검진을 매일 받으면 병은 빨리 찾겠지만 지갑이 거덜 날 것이고, 평생 안 받으면 병을 키우게 되는 것과 같습니다."

+++

## Ⅲ. 오버헤드를 유발하는 주요 요인

### 1. 프로세스와 자원의 규모
- 시스템 내의 활성 프로세스($N$)와 자원 종류($M$)가 늘어날수록 연산량은 제곱으로 증가한다.

### 2. 데드락의 영향 범위
- 데드락 상태인 프로세스가 많아질수록 탐지 후 복구 단계에서 드는 비용(프로세스 롤백 등)도 기하급수적으로 늘어난다.

- **📢 섹션 요약 비유**: "교실에 학생이 많을수록 출석 체크하는 데 시간이 오래 걸리고, 지각생이 많을수록 수업 시작이 더 늦어지는 것과 같습니다."

+++

## Ⅳ. 탐지 정책의 최적화 전략 (Strategy)

| 정책 | 실행 트리거 | 오버헤드 수준 | 추천 환경 |
|:---|:---|:---|:---|
| **즉시 탐지** | 자원 요청 시마다 | 매우 높음 | 데드락이 치명적인 시스템 |
| **주기 탐지** | 일정 시간 간격 | 중간 | 일반적인 범용 OS |
| **적응형 탐지** | CPU 이용률 < 40% | 낮음 | 가용 자원이 넉넉한 시스템 |

- **📢 섹션 요약 비유**: "CCTV를 24시간 돌릴지, 움직임이 감지될 때만 녹화할지 정하는 방범 효율화 전략과 같습니다."
