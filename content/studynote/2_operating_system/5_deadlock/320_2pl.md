# 320. 두 단계 락킹 (Two-Phase Locking)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 락 획득과 해제를 두 단계로 분리
> 2. **가치**: 직렬 가능성 보장
> 3. **융합**: 트랜잭션, 교착상태 예방과 연관

---

## Ⅰ. 개요

### 개념 정의

두 단계 락킹(Two-Phase Locking, 2PL)은 **락 획득 단계와 락 해제 단계를 명확히 분리하는 동시성 제어 프로토콜**이다. 확장 단계와 수축 단계로 구성된다.

### 💡 비유: 도서관 이용
두 단계 락킹은 **도서관 이용**과 같다. 먼저 필요한 책을 모두 찾아서 가져온다(확장). 다 읽은 후 모두 반납한다(수축). 중간에 빌리다 말다 하지 않는다.

### 두 단계 락킹 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                두 단계 락킹 구조                                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【기본 원리】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  시간 ─────────────────────────────────────────────►              │ │
│  │                                                             │ │
│  │  락 수      ▲                                                       │ │
│  │  ─────────┐ │ ─────────────────┐                               │ │
│  │           │ │                   │  수축 단계                     │ │
│  │  확장 단계 │ │  Lock Point       │  (Shrinking Phase)            │ │
│  │  (Growing) │ │                   │                               │ │
│  │           │ │                   ▼                               │ │
│  │           └─────────────────────┴───────────────────────────   │ │
│  │                                                             │ │
│  │  확장 단계: 락 획득만 가능, 해제 불가                                   │ │
│  │  수축 단계: 락 해제만 가능, 획득 불가                                   │ │
│  │  Lock Point: 확장에서 수축으로 전환 지점                               │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【2PL 규칙】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 확장 단계 (Growing Phase):                                       │ │
│  │     • 새로운 락 획득 가능                                             │ │
│  │     • 기존 락 해제 불가                                               │ │
│  │                                                             │ │
│  │  2. 수축 단계 (Shrinking Phase):                                     │ │
│  │     • 새로운 락 획득 불가                                             │ │
│  │     • 기존 락 해제 가능                                               │ │
│  │                                                             │ │
│  │  3. Lock Point:                                                     │ │
│  │     • 마지막 락 획득 시점                                              │ │
│  │     • 이후 수축 단계 시작                                              │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【직렬 가능성 보장】                                                    │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  2PL을 따르는 모든 트랜잭션은 직렬 가능 (Serializable)                  │ │
│  │                                                             │ │
│  │  직렬 가능: 동시 실행 결과가 순차 실행 결과와 동일                        │ │
│  │                                                             │ │
│  │  예: T1, T2가 2PL을 따르면, 결과는 T1→T2 또는 T2→T1과 동일              │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                두 단계 락킹 상세                                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【2PL 변형】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. Basic 2PL (기본):                                                │ │
│  │     • 확장 후 수축                                                   │ │
│  │     • 교착상태 가능                                                  │ │
│  │                                                             │ │
│  │  2. Strict 2PL (엄격):                                               │ │
│  │     • 쓰기 락을 트랜잭션 종료까지 유지                                 │ │
│  │     • 연쇄 복귀(Cascading Rollback) 방지                             │ │
│  │     • 가장 많이 사용                                                  │ │
│  │                                                             │ │
│  │  3. Rigorous 2PL (강력):                                             │ │
│  │     • 모든 락을 트랜잭션 종료까지 유지                                 │ │
│  │     • 구현 가장 간단                                                  │ │
│  │     • 동시성 가장 낮음                                                │ │
│  │                                                             │ │
│  │  4. Conservative 2PL (보수적):                                       │ │
│  │     • 모든 락을 시작할 때 획득                                         │ │
│  │     • 교착상태 예방                                                   │ │
│  │     • 동시성 매우 낮음                                                │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【2PL과 교착상태】                                                      │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  Basic 2PL은 교착상태 가능:                                           │ │
│  │                                                             │ │
│  │  T1: lock(A), lock(B), unlock(A), unlock(B)                        │ │
│  │  T2: lock(B), lock(A), unlock(B), unlock(A)                        │ │
│  │                                                             │ │
│  │  T1이 A 획득, T2가 B 획득 → 서로 상대방의 락 대기 → 교착상태            │ │
│  │                                                             │ │
│  │  해결:                                                                 │ │
│  │  • 타임아웃                                                            │ │
│  │  • 교착상태 탐지                                                       │ │
│  │  • Conservative 2PL 사용                                              │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【연쇄 복귀 문제】                                                      │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  Basic 2PL의 문제:                                                    │ │
│  │  T1이 A 쓰고 unlock → T2가 A 읽음                                     │ │
│  │  T1이 롤백 → T2도 롤백 필요 (연쇄 복귀)                                 │ │
│  │                                                             │ │
│  │  Strict 2PL로 해결:                                                   │ │
│  │  쓰기 락을 커밋까지 유지 → 연쇄 복귀 방지                               │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【기본 2PL 구현】                                                      │
│  ──────────────────                                                  │
│  class TwoPhaseLocking {                                            │
│      Map<String, Lock> locks = new ConcurrentHashMap<>();           │
│      Set<String> acquiredLocks = new HashSet<>();                   │
│      boolean inGrowingPhase = true;                                 │
│                                                                     │
│      void lock(String resource, LockMode mode) {                    │
│          if (!inGrowingPhase) {                                     │
│              throw new IllegalStateException("Cannot lock in shrinking phase");│
│          }                                                          │
│          Lock lock = locks.computeIfAbsent(resource, k -> new Lock());│
│          lock.acquire(mode);                                        │
│          acquiredLocks.add(resource);                               │
│      }                                                              │
│                                                                     │
│      void unlock(String resource) {                                 │
│          if (acquiredLocks.contains(resource)) {                    │
│              inGrowingPhase = false;  // 첫 unlock 시 수축 단계 시작  │ │
│              Lock lock = locks.get(resource);                       │
│              lock.release();                                        │
│              acquiredLocks.remove(resource);                        │
│          }                                                          │
│      }                                                              │
│  }                                                                  │
│                                                                     │
│  【Strict 2PL 구현】                                                   │
│  ──────────────────                                                  │
│  class Strict2PL {                                                  │
│      Map<String, Lock> locks = new ConcurrentHashMap<>();           │
│      ThreadLocal<Set<String>> heldLocks = ThreadLocal.withInitial(HashSet::new);│
│                                                                     │
│      void lock(String resource, LockMode mode) {                    │
│          Lock lock = locks.computeIfAbsent(resource, k -> new Lock());│
│          lock.acquire(mode);                                        │
│          heldLocks.get().add(resource);                             │
│      }                                                              │
│                                                                     │
│      // 쓰기 락은 commit까지 해제하지 않음                               │ │
│      void unlockRead(String resource) {                             │
│          Lock lock = locks.get(resource);                           │
│          if (lock.isReadLock()) {                                   │
│              lock.release();                                        │
│              heldLocks.get().remove(resource);                      │
│          }                                                          │
│          // 쓰기 락은 해제하지 않음                                      │ │
│      }                                                              │
│                                                                     │
│      void commit() {                                                │
│          // 모든 락 해제                                               │ │
│          for (String resource : heldLocks.get()) {                  │
│              locks.get(resource).release();                         │
│          }                                                          │
│          heldLocks.get().clear();                                   │
│      }                                                              │
│  }                                                                  │
│                                                                     │
│  【데이터베이스 2PL】                                                    │
│  ──────────────────                                                  │
│  // 대부분의 DBMS가 Strict 2PL 사용                                     │ │
│  // MySQL InnoDB:                                                    │
│  // - 공유 락(S): 읽기                                                 │ │
│  // - 배타 락(X): 쓰기                                                 │ │
│  // - 쓰기 락은 트랜잭션 종료까지 유지                                     │ │
│                                                                     │
│  BEGIN TRANSACTION;                                                  │
│  SELECT * FROM accounts WHERE id = 1 FOR UPDATE;  -- X락 획득        │
│  UPDATE accounts SET balance = balance - 100 WHERE id = 1;          │
│  -- 다른 작업                                                          │ │
│  COMMIT;  -- 모든 락 해제                                              │ │
│                                                                     │
│  【교착상태 처리】                                                        │
│  ──────────────────                                                  │
│  class DeadlockHandling2PL {                                        │
│      TwoPhaseLocking lockManager = new TwoPhaseLocking();           │
│      int maxRetries = 3;                                            │
│                                                                     │
│      void executeWithRetry(Runnable transaction) {                  │
│          for (int i = 0; i < maxRetries; i++) {                     │
│              try {                                                   │
│                  transaction.run();                                  │ │
│                  return;                                             │ │
│              } catch (DeadlockException e) {                        │
│                  // 롤백하고 재시도                                      │ │
│                  lockManager.releaseAll();                          │
│                  sleep(100 * (i + 1));  // 백오프                    │ │
│              }                                                      │
│          }                                                          │
│          throw new RuntimeException("Max retries exceeded");        │
│      }                                                              │
│  }                                                                  │
│                                                                     │
│  【성능 최적화】                                                          │
│  ──────────────────                                                  │
│  // 락 순서 지정으로 교착상태 예방                                        │ │
│  class Ordered2PL {                                                 │
│      void transfer(Account from, Account to, int amount) {          │
│          // ID 순으로 락 획득                                           │ │
│          Account first = from.getId() < to.getId() ? from : to;     │
│          Account second = from.getId() < to.getId() ? to : from;    │
│                                                                     │
│          lock(first.getId());                                       │
│          lock(second.getId());                                      │
│          try {                                                       │
│              from.withdraw(amount);                                 │
│              to.deposit(amount);                                    │
│          } finally {                                                 │
│              unlock(second.getId());                                │
│              unlock(first.getId());                                 │
│          }                                                          │
│      }                                                              │
│  }                                                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 락 획득과 해제를 두 단계로 분리
• 단계: 확장(Growing) → 수축(Shrinking)
• 보장: 직렬 가능성 (Serializability)
• 변형: Basic, Strict, Rigorous, Conservative
• 교착상태: Basic은 가능, Conservative는 예방
• 활용: 대부분의 DBMS
```
