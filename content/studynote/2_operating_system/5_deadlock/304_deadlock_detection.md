# 교착 상태 탐지 (Deadlock Detection)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 교착 상태 탐지는 예방이나 회피를 포기하고 자원을 자유롭게 할당한 뒤, 주기적으로 시스템이 교착 상태에 빠졌는지를 대기 그래프 (Wait-for Graph)나 탐지 알고리즘으로 확인하는 방법이다.
> 2. **가치**: 자원 이용률을 최대화할 수 있으며, 교착이 드물게 발생하는 환경에서 예방/회피 오버헤드를 피할 수 있다. 대부분의 RDBMS (Relational Database Management System)가 채택하는 현실적 전략이다.
> 3. **융합**: 탐지 알고리즘은 그래프 이론의 DFS (Depth-First Search)/사이클 탐지에 기반하며, 탐지 주기와 복구 비용의 트레이드오프가 시스템 설계의 핵심 판단이다.

---

## Ⅰ. 개요 및 필요성

교착 상태 탐지는 예방적 제약 없이 자원을 할당하되, 교착이 발생하면 이를 감지하고 복구하는 사후 처리 전략이다. 교착이 발생하지 않으면 오버헤드가 없고, 발생 시에만 탐지 알고리즘을 실행한다.

**핵심 트레이드오프**:
- 탐지를 자주하면: 교착을 빠르게 감지 & 복구, 그러나 탐지 오버헤드 증가
- 탐지를 드물게 하면: 교착이 오래 지속되어 영향 범위 증가, 탐지 때 복구할 사이클 여러 개

**💡 비유**: 주기적인 건강검진처럼 — 매월 검진(자주 탐지)은 비싸지만 조기 발견, 1년에 한 번(드물게)은 저렴하지만 병이 진행될 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│          교착 상태 탐지 전략 비교                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  탐지 주기      │ 오버헤드  │ 교착 지속 시간 │ 적합 환경  │
│  ──────────────┼───────────┼────────────────┼────────────│
│  항시 (매 요청) │ 매우 높음 │ 즉시           │ 고신뢰 DB  │
│  주기적 (N분)   │ 중간      │ 최대 N분       │ 일반 DB    │
│  사용자 발견 시 │ 거의 0    │ 사용자 인식까지│ OS (Ostrich)│
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: 교착 탐지는 암 조기 검진과 같습니다 — 주기가 짧을수록 빨리 발견하지만 검진(탐지 알고리즘 실행) 비용도 커집니다.

## Ⅱ. 아키텍처 및 핵심 원리

### 단일 인스턴스: 대기 그래프 (Wait-for Graph)

단일 인스턴스 자원 환경에서는 자원 할당 그래프에서 자원 정점을 제거하고 프로세스 간 대기 관계만 남기면 대기 그래프가 된다. **사이클이 존재하면 교착 상태**다.

```text
┌──────────────────────────────────────────────────────────────┐
│        자원 할당 그래프 → 대기 그래프 변환                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [자원 할당 그래프]         [대기 그래프]                   │
│                                                              │
│  P1 ──요청──▶ R1            P1 ──▶ P2                       │
│  R1 ──할당──▶ P2            P2 ──▶ P3                       │
│  P2 ──요청──▶ R2            P3 ──▶ P1  ← 사이클! 교착!     │
│  R2 ──할당──▶ P3                                             │
│  P3 ──요청──▶ R1                                             │
│                                                              │
│  (P1은 R1이 필요한데, R1은 P2가 보유)                       │
│  → "P1은 P2를 기다림" 간선으로 단순화                       │
│                                                              │
│  탐지: DFS로 사이클 탐색, O(n²)                             │
└──────────────────────────────────────────────────────────────┘
```

### 다중 인스턴스: 탐지 알고리즘

은행원 알고리즘과 유사하지만 Max를 사용하지 않는다. 현재 요청(Request)과 현재 가용(Available), 현재 할당(Allocation)만으로 탐지한다.

```
Detection Algorithm:
① Work = Available.copy()
② Finish[i]:
     Allocation[i] == 0 → true (자원 미보유 프로세스는 완료 가능)
     그 외 → false

③ 반복:
     i를 찾기: Finish[i] == false AND Request[i] ≤ Work
     찾으면:
       Work += Allocation[i]
       Finish[i] = true
     못 찾으면: 종료

④ Finish[i] == false인 프로세스 → 교착 상태에 있음
```

```text
┌──────────────────────────────────────────────────────────────┐
│   다중 인스턴스 탐지 알고리즘 예시                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│       Allocation    Request    Available                     │
│         A  B  C     A  B  C    A  B  C                       │
│  P0:    0  1  0     0  0  0                                  │
│  P1:    2  0  0     2  0  2    0  0  0                       │
│  P2:    3  0  3     0  0  0                                  │
│  P3:    2  1  1     1  0  0                                  │
│  P4:    0  0  2     0  0  2                                  │
│                                                              │
│  ① Work=[0,0,0]                                              │
│     P0, P2: Request=0 → 즉시 Finish=true                    │
│     Work = [0,1,0]+[3,0,3] = [3,1,3]                        │
│                                                              │
│  ② P3: Request[3]=[1,0,0] ≤ Work=[3,1,3] → Finish=true    │
│     Work = [3,1,3]+[2,1,1] = [5,2,4]                        │
│                                                              │
│  ③ P1: Request[1]=[2,0,2] ≤ Work=[5,2,4] → Finish=true    │
│     Work = [5,2,4]+[2,0,0] = [7,2,4]                        │
│                                                              │
│  ④ P4: Request[4]=[0,0,2] ≤ Work=[7,2,4] → Finish=true    │
│                                                              │
│  모두 Finish=true → 교착 없음 ✅                             │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 탐지 알고리즘은 은행원 알고리즘과 달리 Max(최대 요구량) 없이 **현재 요청(Request)**만 사용한다. 이는 현재 시점의 자원 요청들이 모두 충족될 수 있는지를 시뮬레이션하는 것이다. 탐지 알고리즘 실행 결과 Finish[i]=false인 프로세스들이 교착 상태다. 이 프로세스들의 집합이 교착 사이클을 형성한다.

**📢 섹션 요약 비유**: 탐지 알고리즘은 '지금 서있는 모든 손님'에게 순서대로 서비스해줄 수 있는지를 확인하는 것 — 안 되는 손님이 있으면 그게 교착입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 교착 탐지 후 복구 방법

```text
┌──────────────────────────────────────────────────────────────┐
│         교착 복구 방법 비교                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [프로세스 종료 방식]                                        │
│  ① 전체 종료: 교착된 모든 프로세스 강제 종료               │
│     → 간단하지만 많은 작업 손실                             │
│                                                              │
│  ② 순차 종료: 사이클이 제거될 때까지 하나씩 종료           │
│     → 종료마다 탐지 재실행 (O(n²) 반복 비용)               │
│     → 희생자 선택 기준: 우선순위, 실행 시간, 자원 수       │
│                                                              │
│  [자원 선점 방식]                                            │
│  ① 희생자 선택: 최소 손실로 교착 해소할 프로세스 선택      │
│  ② 후퇴 (Rollback): 프로세스를 이전 안전 상태로 롤백       │
│  ③ 기아 방지: 같은 프로세스가 반복 희생되지 않도록          │
│     (희생 카운터, 비용에 카운터 반영)                       │
│                                                              │
│  실무: PostgreSQL은 교착 감지 시 가장 최근 트랜잭션을 롤백  │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 실무 데이터베이스에서는 교착 탐지 후 가장 비용이 적은 트랜잭션을 희생자로 선택하여 롤백한다. PostgreSQL은 교착 탐지 알고리즘을 내부적으로 실행하여 사이클을 발견하면 그 중 가장 짧게 실행된 트랜잭션에 `ERROR: deadlock detected`를 반환하고 롤백한다. 애플리케이션은 이 오류를 캐치하고 재시도 로직을 구현해야 한다.

**📢 섹션 요약 비유**: 교착 복구는 교통 경찰이 막힌 교차로에서 한 차량에 "후진하세요"를 요청하는 것 — 최소 비용으로 순환 대기를 끊는 선택입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **PostgreSQL 교착 탐지**: 트랜잭션이 `lock_timeout` 초과 시 대기 그래프를 분석하여 사이클 발견. 가장 최근 트랜잭션에 `ERROR: deadlock detected` 발생 → 자동 롤백.
2. **MySQL InnoDB**: `innodb_lock_wait_timeout` (기본 50초) 후 교착 탐지. `SHOW ENGINE INNODB STATUS`로 교착 세부 정보 확인 가능.
3. **Java Thread Dump**: JVM `jstack <pid>`로 스레드 상태에서 `BLOCKED`인 스레드의 락 보유 정보를 확인하여 교착 사이클 수동 탐지.

### 도입 체크리스트
- **탐지 주기 설정**: 교착 발생 빈도와 허용 가능한 대기 시간을 기반으로 주기 결정.
- **재시도 로직**: 교착 후 롤백된 트랜잭션에 대한 자동 재시도 구현 여부 확인.
- **기아 방지**: 반복 희생자 차단을 위한 카운터 기반 비용 함수 적용.

### 안티패턴
- **탐지 없이 타임아웃만**: 단순 타임아웃은 교착 프로세스를 식별하지 못해 불필요한 정상 프로세스까지 종료할 수 있다.
- **재시도 없는 롤백 처리**: 교착 후 오류를 로그만 남기고 재시도하지 않으면 사용자 작업이 영구 손실된다.

**📢 섹션 요약 비유**: 교착 탐지 없이 타임아웃만 사용하는 것은 교통 정체가 생길 때 비정상적인 차량만 찾지 않고 모든 차를 무작위로 돌리는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

| 전략 | 자원 이용률 | 교착 방지 | 구현 복잡도 |
|:---|:---|:---|:---|
| 예방 (Prevention) | 낮음 | 100% | 낮음 |
| 회피 (Avoidance) | 중간 | 100% | 높음 |
| **탐지+복구** | **높음** | 사후 처리 | 중간 |
| 무시 (Ostrich) | 최고 | 없음 | 최저 |

교착 탐지+복구는 자원 이용률과 교착 방지의 현실적 균형점이다. RDBMS, 클라우드 자원 관리자 등 교착이 드물지만 발생 시 복구가 가능한 환경에 최적이다.
