# 2단계 잠금 프로토콜 (2PL, Two-Phase Locking)과 데드락

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 2PL (Two-Phase Locking)은 데이터베이스 트랜잭션의 직렬화 가능성 (Serializability)을 보장하기 위해 확장 단계 (Acquiring)와 수축 단계 (Releasing)로 나누어 락을 관리하는 프로토콜이며, 이 구조 자체가 교착 상태 발생 가능성을 내재한다.
> 2. **가치**: 2PL은 동시 트랜잭션이 직렬 실행과 동등한 결과를 보장하는 가장 강력한 락 기반 동시성 제어 기법이지만, 교착 상태를 완전히 예방하지 못하므로 DB 엔진이 탐지+복구를 병행해야 한다.
> 3. **융합**: MySQL InnoDB, PostgreSQL, Oracle의 행 수준 잠금과 교착 탐지 메커니즘은 모두 2PL과 대기 그래프 (Wait-for Graph) 기반 탐지의 조합으로 구현된다.

---

## Ⅰ. 개요 및 필요성

데이터베이스에서 여러 트랜잭션이 동시에 실행될 때, 올바른 결과를 보장하려면 동시 실행이 특정 직렬 순서와 동등해야 한다(직렬화 가능성). 2PL은 이를 보장하는 검증된 로킹 프로토콜이다.

**2단계 구조**:
1. **확장 단계 (Growing Phase)**: 락을 획득하기만 하고, 절대로 락을 해제하지 않는다.
2. **수축 단계 (Shrinking Phase)**: 락을 해제하기만 하고, 새 락을 획득하지 않는다.

락 획득과 해제 사이의 엄격한 분리가 직렬화를 보장한다. 그러나 여러 트랜잭션이 동시에 확장 단계에 있을 때 순환 대기가 형성되면 교착 상태가 발생한다.

**💡 비유**: 도서관 규칙 → "책을 빌리는 중(확장)에는 절대 반납 금지. 처음 반납을 시작하면(수축) 새 책 대출 금지." 이 규칙이 빌리는 순서 충돌을 없애지만, 두 사람이 서로 상대방 책을 기다리면 교착이 발생한다.

```text
┌──────────────────────────────────────────────────────────────┐
│          2PL 타임라인 및 교착 상태 발생 시나리오            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  시간 ─────────────────────────────────────────────────▶   │
│                                                              │
│  T1: [lock A] ─────── [lock B (대기!)] ─── (교착)          │
│        확장 단계                                             │
│                                                              │
│  T2:           [lock B] ─────── [lock A (대기!)] ─ (교착) │
│                  확장 단계                                   │
│                                                              │
│  ┌──────────────────────────────────────────────────┐       │
│  │  Wait-for Graph:                                 │       │
│  │  T1 ──▶ T2  (T1 is waiting for T2's lock on B)  │       │
│  │  T2 ──▶ T1  (T2 is waiting for T1's lock on A)  │       │
│  │  → 사이클 = 교착 상태!                           │       │
│  └──────────────────────────────────────────────────┘       │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: 2PL은 손님이 주문을 다 마치고 나서야 결제하는 규칙 — 주문 중에는 계산하지 않아 계산 오류(비직렬화)를 막지만, 두 손님이 서로가 먼저 주문 완료하기를 기다리면 교착이 됩니다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2PL 변형 유형

```text
┌──────────────────────────────────────────────────────────────┐
│          2PL 변형 비교                                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [기본 2PL (Basic 2PL)]                                     │
│  최초 락 해제 시점에서 확장→수축 전환                       │
│  교착 가능성: 높음 (가장 유연, 가장 위험)                  │
│                                                              │
│  [엄격 2PL (Strict 2PL)]                                    │
│  모든 EXCLUSIVE 락을 트랜잭션 완료(COMMIT/ROLLBACK) 시 해제  │
│  연쇄 롤백(Cascading Abort) 방지 → 실무 DB의 표준 채택      │
│                                                              │
│  [강화 2PL (Rigorous 2PL)]                                  │
│  모든 락 (SHARED + EXCLUSIVE)을 트랜잭션 완료 시까지 유지   │
│  직렬화 순서 = 커밋 순서 → 분산 DB에서 유용                │
└──────────────────────────────────────────────────────────────┘
```

### 교착 상태 예방: 타임스탬프 기반 방법

```text
┌──────────────────────────────────────────────────────────────┐
│     Wait-Die vs Wound-Wait 프로토콜                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  (나이 = 타임스탬프 작을수록 오래된/중요한 트랜잭션)        │
│                                                              │
│  [Wait-Die (비선점)]                                        │
│  오래된 트랜잭션(Ti)이 새 트랜잭션(Tj)의 락을 요청:        │
│    Ti < Tj → Ti는 대기 (오래된 것이 기다림)                 │
│    Ti > Tj → Ti는 롤백 (새 것이 기다리지 않게 Ti 제거)     │
│                                                              │
│  [Wound-Wait (선점)]                                        │
│  오래된 트랜잭션(Ti)이 새 트랜잭션(Tj)의 락을 요청:        │
│    Ti < Tj → Ti가 Tj를 강제 롤백 (오래된 것이 우선)       │
│    Ti > Tj → Ti는 대기 (새 것이 전행 중)                   │
│                                                              │
│  공통: 순환 대기 발생 불가 → 교착 예방                      │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Wait-Die와 Wound-Wait은 타임스탬프를 이용하여 랙 요청에 우선순위를 부여함으로써 순환 대기를 구조적으로 차단한다. Wait-Die는 오래된 트랜잭션이 대기하고 젊은 것이 롤백되는 방식으로, Wound-Wait은 그 반대로 오래된 것이 젊은 것을 선점하는 방식이다. 각각은 교착 상태의 4조건 중 '순환 대기'를 제거한다. 단, 불필요한 롤백이 발생할 수 있어 처리량에 영향을 준다.

### 실무 DB의 교착 처리 패턴

```text
┌──────────────────────────────────────────────────────────────┐
│       PostgreSQL 교착 탐지 및 복구 흐름                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  T1: BEGIN → UPDATE row A → UPDATE row B (대기)             │
│  T2: BEGIN → UPDATE row B → UPDATE row A (대기)             │
│                                                              │
│  PostgreSQL deadlock detector (주기적 실행):                 │
│  ① 대기 그래프 구성: T1→T2→T1 사이클 발견                  │
│  ② 희생자 선택: 가장 최근 트랜잭션 (T2) 롤백               │
│  ③ T2에 ERROR: deadlock detected 반환                       │
│                                                              │
│  T1: row B 락 획득 성공 → COMMIT                           │
│  T2: 애플리케이션에서 오류 캐치 → 재시도 로직 실행         │
│                                                              │
│  설정 파라미터:                                              │
│  deadlock_timeout = 1s (탐지 대기 시간, 기본값)             │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PostgreSQL의 교착 탐지는 `deadlock_timeout`(기본 1초) 대기 후 시작된다. 먼저 단순 락 타임아웃이 아니라 대기 그래프를 구석하여 실제 사이클이 있는지 확인한다. 탐지 시 가장 최근 트랜잭션을 롤백하여 교착을 해소한다. 애플리케이션은 이 오류를 처리하고 재시도해야 한다.

**📢 섹션 요약 비유**: 2PL과 교착 탐지는 은행 계좌 이체 시스템 — 동시에 A→B, B→A 이체가 발생하면 한 쪽을 잠깐 취소했다가 재시도하는 현실 시스템의 동작 원리입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 2PL vs MVCC 비교

```text
┌─────────────────────┬──────────────────┬──────────────────────┐
│ 항목                │ 2PL              │ MVCC                 │
├─────────────────────┼──────────────────┼──────────────────────┤
│ 교착 가능성         │ 있음             │ 읽기-쓰기 간 없음    │
│ 읽기-쓰기 차단      │ 있음             │ 없음 (스냅샷 읽기)   │
│ 구현 복잡도         │ 낮음             │ 높음                 │
│ 쓰기-쓰기 교착      │ 있음             │ 있음                 │
│ 적합 환경           │ 읽기-쓰기 균등   │ 읽기 집중형          │
└─────────────────────┴──────────────────┴──────────────────────┘
```

**📢 섹션 요약 비유**: 2PL은 교통신호로 교차로 통행 순서 조율, MVCC는 교차로에 입체 교차로(육교)를 만들어 충돌 자체를 없애는 방식입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **뱅킹 시스템 송금**: T1(A→B 송금)이 A 잔액을 잠근 후 B를 잠그려 할 때, T2(B→A 송금)가 B를 잠근 후 A를 잠그려 대기 → 교착. 해결: 모든 이체 트랜잭션이 계좌 번호 오름차순으로 잠금.
2. **배치 데이터 이전**: 대량 UPDATE 시 작은 단위로 분할하여 락 보유 시간 최소화, 교착 발생빈도 감소.

### 도입 체크리스트
- **락 순서 규칙**: 여러 행을 업데이트할 때 항상 같은 순서(예: 기본키 오름차순)로 잠금.
- **데드락 재시도**: 애플리케이션 레이어에서 `deadlock detected` 오류 시 자동 재시도를 구현.
- **lock timeout 설정**: 무한 대기 방지를 위한 `lock_timeout`(PG) / `innodb_lock_wait_timeout`(MySQL) 설정.

### 안티패턴
- **다른 순서로 여러 테이블 잠금**: 서로 다른 순서로 A→B, B→A 잠그는 두 트랜잭션 패턴이 존재하면 즉각 교착 후보.
- **재시도 로직 없는 교착 처리**: ERROR deadlock를 캐치하고 상위로 그냥 전파하면 사용자 요청이 영구 실패.

**📢 섹션 요약 비유**: 락 순서 규칙 없이 2PL을 쓰는 것은 차가 마음대로 진입하는 교차로와 같습니다 — 규칙(순서)만 정하면 교통 정체(교착)를 대부분 막을 수 있어요.

## Ⅴ. 기대효과 및 결론

2PL은 직렬화 가능성을 보장하는 강력한 동시성 제어 기법이지만, 교착 상태를 완전히 예방하지 못한다. 실무에서는 엄격 2PL + 탐지+복구 + 재시도 로직의 조합이 표준이며, 잠금 순서화로 교착 빈도를 최소화한다. MVCC는 읽기-쓰기 교착을 구조적으로 제거하는 현대적 대안이지만 쓰기-쓰기 교착은 여전히 존재한다.

## 📌 관련 개념 맵

| 개념 | 관계 |
|:---|:---|
| 직렬화 가능성 (Serializability) | 2PL이 보장하는 핵심 데이터 일관성 속성 |
| 엄격 2PL (Strict 2PL) | 연쇄 롤백 방지를 위한 실무 표준 변형 |
| Wait-for Graph | 2PL 환경의 교착 탐지 자료구조 |
| MVCC | 2PL의 읽기-쓰기 교착 제거 대안 |
| Wait-Die / Wound-Wait | 타임스탬프 기반 교착 예방 프로토콜 |

## 👶 어린이를 위한 3줄 비유 설명
1. 2PL은 숙제 규칙 — "자료 수집(확장)이 끝날 때까지 자료를 반납(수축)하면 안 돼요."
2. 두 친구 A, B가 서로가 반납하기를 기다리면 아무도 완성 못해요 — 이게 교착 상태!
3. 해결: 선생님이 "항상 수학 자료부터, 그 다음 과학 자료 순서로 받아요" 규칙을 정하면 서로 안 겹쳐요.
