# 294. 교착상태 회피 (Deadlock Avoidance)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 실행 중 안전하지 않은 자원 요청을 거부
> 2. **가치**: 교착상태 발생 전에 방지
> 3. **융합**: 은행원 알고리즘, 안전 상태와 연관

---

## Ⅰ. 개요

### 개념 정의

교착상태 회피(Deadlock Avoidance)는 **실행 시간에 시스템 상태를 검사하여, 교착상태로 이어질 수 있는 자원 요청을 거부하는 전략**이다. 예방보다 유연하다.

### 💡 비유: 교통 신호
교착상태 회피는 **교통 신호**와 같다. 차가 많으면 신호가 빨간불로 바뀐다. 교통 체증을 동적으로 예방한다.

### 회피 전략 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                교착상태 회피 구조                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【예방 vs 회피】                                                      │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  예방 (Prevention):                                                │ │
│  │  • 컴파일 타임/설계 타임에 결정                                        │ │
│  │  • 엄격한 규칙 적용                                                   │ │
│  │  • 정적                                                             │ │
│  │                                                             │ │
│  │  회피 (Avoidance):                                                  │ │
│  │  • 런타임에 동적 결정                                                  │ │
│  │  • 현재 상태 기반 판단                                                │ │
│  │  • 동적, 더 유연                                                     │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【안전 상태 vs 불안전 상태】                                             │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  안전 상태 (Safe State):                                            │ │
│  │  • 모든 프로세스가 완료될 수 있는 순서가 존재                               │ │
│  │  • 교착상태 발생 불가                                                  │ │
│  │                                                             │ │
│  │  불안전 상태 (Unsafe State):                                         │ │
│  │  • 교착상태 발생 가능                                                  │ │
│  │  • 안전 순서가 없음                                                   │ │
│  │                                                             │ │
│  │  회피: 불안전 상태로 가는 요청을 거부                                      │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【핵심 아이디어】                                                       │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  각 자원 요청 시:                                                    │ │
│  │  1. 요청을 승인한다고 가정                                             │ │
│  │  2. 결과 상태가 안전한지 확인                                          │ │
│  │  3. 안전하면 승인, 불안전하면 거부/대기                                   │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                교착상태 회피 상세                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【은행원 알고리즘 (Banker's Algorithm)】                                │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  입력:                                                               │ │
│  │  • Available: 각 자원의 가용 개수                                     │ │
│  │  • Max: 각 프로세스의 최대 자원 필요량                                  │ │
│  │  • Allocation: 현재 할당된 자원                                       │ │
│  │  • Need: Max - Allocation (추가 필요량)                             │ │
│  │                                                             │ │
│  │  안전성 알고리즘:                                                       │ │
│  │  1. Work = Available, Finish[i] = false                            │ │
│  │  2. Finish[i] = false && Need[i] <= Work인 i 찾기                   │ │
│  │  3. 있으면: Work += Allocation[i], Finish[i] = true, 2로             │ │
│  │  4. 모든 Finish[i] = true면 안전, 아니면 불안전                         │ │
│  │                                                             │ │
│  │  자원 요청 알고리즘:                                                    │ │
│  │  1. Request[i] <= Need[i] 확인                                       │ │
│  │  2. Request[i] <= Available 확인                                      │ │
│  │  3. 시험적 할당 후 안전성 검사                                           │ │
│  │  4. 안전하면 승인, 아니면 롤백하고 대기                                    │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【장단점】                                                             │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  장점:                                                               │ │
│  │  • 예방보다 유연                                                       │ │
│  │  • 더 높은 자원 활용도                                                  │ │
│  │  • 교착상태 확실히 방지                                                  │ │
│  │                                                             │ │
│  │  단점:                                                               │ │
│  │  • 최대 자원 필요량을 미리 알아야 함                                      │ │
│  │  • 프로세스 수가 고정되어야 함                                           │ │
│  │  • 오버헤드 (매 요청마다 안전성 검사)                                     │ │
│  │  • 보수적 (불필요한 대기 발생 가능)                                      │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【은행원 알고리즘 구현 (간단 버전)】                                     │
│  ──────────────────                                                  │
│  class BankersAlgorithm {                                           │
│      int[] available;     // 가용 자원                                 │
│      int[][] max;         // 최대 필요량 [프로세스][자원]                  │
│      int[][] allocation;  // 현재 할당 [프로세스][자원]                    │
│      int[][] need;        // 추가 필요량 [프로세스][자원]                   │
│                                                                     │
│      bool isSafe() {                                                │
│          int[] work = available.clone();                            │
│          boolean[] finish = new boolean[n];                         │
│                                                                     │
│          while (true) {                                              │
│              int i = findProcess(work, finish);                     │
│              if (i == -1) break;                                    │
│                                                                     │
│              for (int j = 0; j < m; j++)                            │
│                  work[j] += allocation[i][j];                       │
│              finish[i] = true;                                      │
│          }                                                          │
│                                                                     │
│          for (boolean f : finish)                                   │
│              if (!f) return false;                                  │
│          return true;                                               │
│      }                                                              │
│                                                                     │
│      int findProcess(int[] work, boolean[] finish) {                │
│          for (int i = 0; i < n; i++) {                              │
│              if (!finish[i] && need[i] <= work)                     │
│                  return i;                                          │
│          }                                                          │
│          return -1;                                                 │
│      }                                                              │
│                                                                     │
│      boolean requestResources(int pid, int[] request) {             │
│          // 1. 요청이 필요량 이하인지 확인                                 │
│          if (request > need[pid]) return false;                     │
│                                                                     │
│          // 2. 요청이 가용량 이하인지 확인                                 │
│          if (request > available) return false;  // 대기              │
│                                                                     │
│          // 3. 시험적 할당                                              │
│          for (int j = 0; j < m; j++) {                              │
│              available[j] -= request[j];                            │
│              allocation[pid][j] += request[j];                      │
│              need[pid][j] -= request[j];                            │
│          }                                                          │
│                                                                     │
│          // 4. 안전성 검사                                              │
│          if (isSafe()) {                                            │
│              return true;  // 승인                                    │
│          } else {                                                    │
│              // 롤백                                                   │
│              for (int j = 0; j < m; j++) {                          │
│                  available[j] += request[j];                        │
│                  allocation[pid][j] -= request[j];                  │
│                  need[pid][j] += request[j];                        │
│              }                                                      │
│              return false;  // 거부/대기                                │
│          }                                                          │
│      }                                                              │
│  }                                                                  │
│                                                                     │
│  【실제 사용 사례】                                                      │
│  ──────────────────                                                  │
│  // 데이터베이스 시스템                                                   │
│  // - 트랜잭션의 최대 락 필요량 예측                                        │
│  // - 락 요청 시 안전성 검사                                               │
│                                                                     │
│  // 실시간 시스템                                                       │
│  // - 자원 예약 시스템                                                   │
│  // - 최대 자원 사용량 보장                                                │
│                                                                     │
│  【성능 분석】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  연산              시간 복잡도          설명                         │ │
│  │  ────────          ──────────          ────                         │ │
│  │  안전성 검사        O(m × n²)          m=자원, n=프로세스             │ │
│  │  자원 요청         O(m × n²)          매 요청마다 검사                │ │
│  │  자원 해제         O(m)               단순 갱신                      │ │
│  │                                                             │ │
│  │  실제로는 오버헤드로 인해 잘 사용 안 됨                                  │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【현실적 대안】                                                        │
│  ──────────────────                                                  │
│  • 타임아웃 기반 (더 간단)                                                │
│  • 교착상태 탐지 + 복구 (발생 후 처리)                                     │
│  • 예방 전략 (락 순서 규칙)                                               │
│  • 트랜잭션 격리 수준 조정                                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 런타임에 안전하지 않은 요청 거부

• 알고리즘: 은행원 알고리즘
• 장점: 유연함, 높은 자원 활용
• 단점: 최대 필요량 미리 알아야, 오버헤드
• 현실: 복잡성으로 실제 사용 제한적
```
