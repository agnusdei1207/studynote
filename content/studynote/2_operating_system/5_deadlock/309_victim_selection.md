# 종료 대상 선택 (Victim Selection) 기준

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 종료 대상 선택(Victim Selection)은 교착 상태 해소를 위해 종료할 프로세스를 선정할 때, 시스템 전체의 비용과 작업 손실을 최소화하기 위한 전략적 의사결정 과정이다.
> 2. **기준**: 프로세스의 우선순위, 수행 시간, 필요한 자원량, 그리고 지금까지 소모한 자원량 등 다각적인 지표를 점수화하여 '가장 저렴한 희생자'를 찾아낸다.
> 3. **가치**: 감정에 치우치지 않는 수치 기반의 선택을 통해, 시스템 복구 시 발생하는 비효율을 줄이고 합리적인 자원 재분배를 실현한다.

+++

## Ⅰ. 희생자 선택의 기본 원칙

### 1. 최소 비용의 원칙
- 시스템에 가장 적은 피해를 주는 프로세스를 골라야 한다.

### 2. 고려 요소
- 프로세스의 우선순위 (고순위 프로세스는 최대한 살림).
- 지금까지 수행된 시간과 완료까지 필요한 시간.
- 점유 중인 자원의 종류와 개수.
- 프로세스가 대화형(Interactive)인지 일괄 처리(Batch)인지 여부.

- **📢 섹션 요약 비유**: "배가 가라앉으려 할 때, 어떤 짐을 먼저 바다로 던져야 배가 뜨고 피해가 적을지 목록을 보고 고르는 것과 같습니다."

+++

## Ⅱ. 기술적 선택 기준 상세 (ASCII Mapping)

### 1. 기준 지표 시각화
```text
 [ Victim Selection Scoring ]
 
   Factor            Score Weight   Description
  -----------------------------------------------
  Priority (P)       ★★★★★        High priority stays.
  Work Done (T1)     ★★★☆☆        Long running stays.
  Work Left (T2)     ★★☆☆☆        Almost done stays.
  Resource Use (R)   ★★★★☆        Many resources = Target!
  
  * Total Cost = f(P, T1, T2, R, ...)
  * Pick Min(Total Cost)
```

- **📢 섹션 요약 비유**: "누구를 집에 보낼지 정할 때 성적표, 출석일수, 남은 숙제 양을 다 합쳐서 점수를 내는 것과 같습니다."

+++

## Ⅲ. 세부 판단 시나리오 (Decision Tree)

### 1. 작업 진행도 vs 남은 시간
- 거의 다 끝난 프로세스를 죽이는 것은 매우 비효율적이다. 반면, 이제 막 시작한 프로세스는 죽여도 손실이 적다.

### 2. 자원 점유 상태
- 구하기 힘든 자원을 독점하고 있는 프로세스는 데드락 해결의 핵심 열쇠가 되므로 희생 순위가 높다.

- **📢 섹션 요약 비유**: "거의 다 쓴 일기장을 찢는 것보다, 이제 막 한 줄 쓴 연습장을 찢는 게 덜 아까운 것과 같습니다."

+++

## Ⅳ. 희생자 선택 정책의 한계와 주의점 (Starvation)

| 문제점 | 상세 설명 | 해결 방안 |
|:---|:---|:---|
| **기아 상태 (Starvation)** | 동일한 프로세스가 계속 희생자로 선정됨 | 비용 계산 시 '희생 횟수'를 가중치로 추가 |
| **복잡한 계산 오버헤드** | 기준이 너무 많으면 선택 자체에 시간 소모 | 주요 지표 2~3개로 단순화 |

- **📢 섹션 요약 비유**: "맨날 청소 당번만 하는 애가 생기지 않도록, 청소한 횟수를 장부에 적어서 골고루 시키는 것과 같습니다."

+++

## Ⅴ. 결론 및 실무적 교훈

### 1. 요약
- 희생자 선택은 '최소 비용으로 최대 효과'를 내기 위한 냉정한 최적화 기법이다.

### 2. 최종 결론
- 운영체제 설계자는 단순히 프로세스를 죽이는 것에 그치지 않고, 어떤 기준으로 죽여야 사용자가 가장 덜 불편해할지를 끊임없이 고민해야 한다. 이는 기술적인 문제를 넘어 정책적 판단의 영역임을 시사한다.

- **📢 섹션 요약 비유**: "가장 아끼는 물건을 지키기 위해 덜 중요한 것을 포기할 줄 아는 것이 진짜 관리자의 능력이라는 교훈을 줍니다."
