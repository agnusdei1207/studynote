# 비선점 (No Preemption)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비선점(No Preemption)은 프로세스가 자원을 점유하고 있는 동안, 운영체제가 해당 자원을 **강제로 회수할 수 없는** 자원 관리 정책이다.
> 2. **가치**: 프로세스의 수행 안정성을 보장하고 상태 복구의 복잡성을 낮추지만, 자원을 쥐고 대기하는 '알박기' 현상을 허용하여 교착 상태를 심화시킨다.
> 3. **융합**: CPU 스케줄링의 선점형 정책과는 대조적으로, 일반적 하드웨어 자원(메모리, I/O)에서는 데이터 무결성을 위해 비선점이 기본 원칙으로 적용된다.

+++

## Ⅰ. 비선점 (No Preemption)의 개념 및 배경

### 1. 정의
- 자원을 점유하고 있는 프로세스가 해당 자원을 스스로 해제(Release)하기 전까지는, 시스템이나 다른 프로세스가 그 자원을 빼앗을 수 없는 성질이다.
- 자원 사용의 **독점적 정당성**을 보장하는 원칙이다.

### 2. 왜 비선점이 필요한가?
- 만약 어떤 프로세스가 파일에 데이터를 쓰고 있는 도중에 자원을 선점(회수)당한다면, 데이터가 불완전한 상태로 남거나 시스템의 일관성이 파괴될 수 있다.
- 따라서 많은 하드웨어 자원과 소프트웨어 락(Lock)은 설계 단계에서부터 비선점 속성을 갖도록 만들어진다.

### 3. 💡 비유: '화장실 안의 사람'
- 화장실을 사용 중인 사람을 강제로 밖으로 끌어낼 수 없는 것과 같다. 그 사람이 스스로 나올 때까지 밖의 사람들은 무한정 기다려야만 한다.

- **📢 섹션 요약 비유**: "빌려준 물건을 상대방이 돌려줄 때까지 강제로 뺏어올 수 없는 '신사협정'과 같습니다."

## Ⅱ. 기술적 구조 및 동작 분석 (Deep Dive)

### 1. 비선점이 교착 상태에 미치는 영향
- 프로세스 $P_1$이 자원 $R_1$을 점유한 채로 다른 자원을 기다릴 때, 만약 $R_1$이 선점 가능했다면 OS가 이를 회수하여 다른 프로세스에게 줌으로써 대기 체인을 끊을 수 있다. 하지만 비선점 환경에서는 이 체인이 **영구적으로 유지**된다.

### 2. ASCII 비선점 대기 메커니즘
```text
    [ OS Kernel ] (강제 회수 불능)
         X
         │
    [ Process A ] <─────── (점유) ─────── [ Resource 1 ]
         │
    (추가 요청)
         │
         ▼
    [ Resource 2 ] <─────── (점유) ─────── [ Process B ]
                                              │
                                         (추가 요청)
                                              │
                                              ▼
                                         [ Resource 1 ]

    * OS는 A로부터 R1을 뺏어 B에게 줄 수 없음 (No Preemption).
    * 결과: A와 B는 서로의 자원을 영원히 뺏지 못한 채 굳어버림.
```

### 3. 기술적 상세 해설
- **상태 보존의 문제**: 선점을 허용하려면 프로세스가 자원을 쓰던 시점의 중간 상태(Register, Memory dump 등)를 완벽히 저장해야 한다. CPU는 이것이 쉽지만(Context Switch), 프린터나 네트워크 소켓 등은 중간 상태 보존이 매우 어렵거나 불가능하다.
- **임계 영역 보호**: 소프트웨어적인 임계 영역(Critical Section)에서 비선점은 데이터 경쟁(Race Condition)을 막는 핵심 방어선이다.

- **📢 섹션 요약 비유**: "책을 읽고 있는 사람에게서 책을 뺏으려면, 그 사람이 읽던 페이지를 표시해두어야 하는데 그 과정이 너무 복잡해 아예 뺏지 않기로 한 것과 같습니다."

## Ⅲ. 선점 허용 정책과의 비교 (Prevention Strategy)

### 1. 자원 선점 시나리오 비교

| 구분 | 비선점 정책 (No Preemption) | 선점형 예방 전략 (Preemption allowed) |
|:---|:---|:---|
| **데드락 위험** | 높음 (4조건 중 하나) | **낮음 (데드락 원천 차단 가능)** |
| **시스템 복잡도** | 낮음 (그냥 기다림) | **매우 높음 (상태 저장/복구 로직 필수)** |
| **자원 활용도** | 낮음 (유휴 자원 발생) | 높음 (필요한 곳에 즉시 배분) |
| **적용 가능 자원** | 모든 자원 | CPU, 메모리 등 상태 복구가 용이한 자원 |

### 2. 선점 기반 예방 로직 (Protocol)
- 프로세스 $P$가 자원을 요청했는데 거절당하면, $P$가 이미 점유한 모든 자원을 **강제로 해제**하고 대기 목록에 들어간다.
- 이후 $P$는 자신이 원래 가졌던 자원과 새로 요청한 자원을 모두 확보할 수 있을 때만 다시 시작한다.

- **📢 섹션 요약 비유**: "식당 자리가 없어 대기해야 한다면, 이미 주문한 음식을 취소하고 식당 밖으로 나가서 자리가 날 때까지 기다리는 것과 같습니다."

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. 실무 시나리오: 가상 메모리의 페이지 선점
- **상황**: 물리 메모리가 부족하여 프로세스 A가 페이지 할당을 기다림.
- **판단**: 메모리는 '선점 가능 자원'이다. OS는 프로세스 B가 점유한 페이지 중 사용 빈도가 낮은 것을 디스크로 쫓아내고(Page-out), 그 자원을 A에게 준다. 이를 통해 메모리 부족으로 인한 데드락을 방지한다.

### 2. 안티패턴: 인터럽트 핸들러 내의 락 대기
- 인터럽트 핸들러는 비선점적으로 실행되는 경우가 많다. 여기서 다른 프로세스가 쥔 락을 기다리게 되면 시스템 전체가 멈추는 데드락에 빠진다. 인터럽트 내에서는 절대 블로킹(Wait)이 발생하는 자원을 요청해서는 안 된다.

- **📢 섹션 요약 비유**: "긴급 출동 중인 소방차가 좁은 골목에서 마주 오는 차를 비켜주지 않고 계속 버티는 바람에 도시 전체가 마비되는 상황을 피해야 합니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 기대효과
- **데이터 무결성 보장**: 작업 도중 자원을 뺏기지 않으므로 연산의 연속성과 일관성이 유지된다.
- **설계 단순화**: 선점 시 발생하는 복잡한 롤백(Rollback) 로직을 생략할 수 있어 커널 설계가 단순해진다.

### 2. 결론
비선점은 시스템의 안정성을 위한 필수 선택이지만, 교착 상태의 온상이 되기도 한다. 따라서 현대 OS는 CPU나 메모리 같은 핵심 자원에는 '선점'을, 파일 락이나 뮤텍스 같은 논리 자원에는 '비선점'을 적용하는 하이브리드 전략을 구사한다.

- **📢 섹션 요약 비유**: "중요한 수술을 진행 중인 의사를 방해하지 않는 예의(비선점)와, 급한 불을 끄기 위해 길을 터주는 결단(선점)이 조화를 이루어야 합니다."

### 📌 관련 개념 맵 (Knowledge Graph)
- **교착 상태 발생 4조건**: 비선점이 포함된 전체 요건.
- **문맥 교환 (Context Switch)**: 자원 선점 시 발생하는 대표적 기술적 동작.
- **가상 메모리 (Virtual Memory)**: 선점형 자원 관리의 대표 사례.
- **트랜잭션 롤백 (Rollback)**: 소프트웨어 자원 선점 시의 복구 메커니즘.
