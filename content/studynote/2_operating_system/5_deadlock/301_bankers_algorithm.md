# 다중 인스턴스 환경의 회피 (Banker's Algorithm)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 은행원 알고리즘(Banker's Algorithm)은 자원 유형당 인스턴스가 여러 개일 때, 자원을 할당하기 전 "모든 프로세스가 완료될 수 있는 안전 순서(Safe Sequence)가 존재하는가"를 시뮬레이션하여 교착 상태를 회피하는 기법이다.
> 2. **작동**: 프로세스가 자원을 요청하면, 시스템은 그 요청을 들어주었을 때도 시스템이 안전 상태(Safe State)를 유지하는지 계산하며, 만약 불안전 상태(Unsafe State)로 빠질 가능성이 있다면 할당을 거부하고 대기시킨다.
> 3. **가치**: 자원 할당 그래프 알고리즘이 해결하지 못하는 다중 인스턴스 환경의 데드락 회피를 수학적으로 보장하지만, 모든 프로세스의 최대 자원 요구량을 미리 알아야 한다는 전제가 필요하다.

+++

## Ⅰ. 은행원 알고리즘의 개요 및 철학

### 1. 정의
- 에츠허르 데이크스트라(Edsger W. Dijkstra)가 제안한 알고리즘으로, 은행에서 고객의 대출 요청을 처리할 때 보유 현금 범위 내에서 모든 고객의 요구를 최종적으로 들어줄 수 있는지 확인하는 과정에서 유래했다.

### 2. 핵심 원리
- 자원을 할당하기 전에 **"만약 빌려준다면, 나중에 다른 애들도 다 무사히 끝낼 수 있을까?"**를 미리 계산해 보는 시뮬레이션 기반의 회피 전략이다.

- **📢 섹션 요약 비유**: "은행 직원이 손님들에게 돈을 빌려줄 때, 금고에 남은 돈으로 다른 손님들의 급한 불까지 다 꺼줄 수 있는지 미리 머릿속으로 계산기를 두드려보는 것과 같습니다."

+++

## Ⅱ. 알고리즘 작동 메커니즘 및 ASCII 구조

### 1. 기본 알고리즘 (Safety Algorithm)
- 시스템이 안전 상태인지 확인하기 위해 '가용 자원'으로 끝낼 수 있는 프로세스를 찾아 실행시키고, 끝난 프로세스의 자원을 회수하는 과정을 반복한다.

### 2. ASCII 시뮬레이션 구조
```text
 [ 시뮬레이션 과정 ]
 1. 현재 가용 자원(Available) 확인
 2. (남은 필요량 <= 가용 자원)인 프로세스 Pi 탐색
 3. Pi를 실행 완료로 가정 (Finish[i] = true)
 4. Pi가 가졌던 자원(Allocation)을 가용 자원에 합산
 5. 모든 프로세스가 완료되면 '안전 상태(Safe State)'!
 
  [ Safe Sequence 예시 ]
  P1 완료 -> 자원 반납 -> P3 완료 -> 자원 반납 -> P2 완료 (OK!)
```

- **📢 섹션 요약 비유**: "도미노를 하나씩 쓰러뜨릴 때, 중간에 멈추지 않고 끝까지 다 쓰러뜨릴 수 있는 경로를 미리 눈으로 따라가 보는 것과 같습니다."

+++

## Ⅲ. 자원 요청 알고리즘 (Resource-Request Algorithm)

### 1. 수행 단계
- **Step 1**: 요청량(Request)이 최대 필요량(Need)보다 크면 오류 (사기 대출 방지).
- **Step 2**: 요청량이 현재 가용량(Available)보다 크면 대기 (금고에 돈이 없음).
- **Step 3**: 가상 할당을 실시하고, 시스템 상태를 업데이트한 후 **Safety Algorithm**을 돌려 안전한지 확인한다.
- **Step 4**: 안전하면 진짜 빌려주고, 불안전하면 가상 할당을 취소하고 프로세스를 대기시킨다.

- **📢 섹션 요약 비유**: "돈을 빌려주기 전에 장부에 미리 적어보고, '아, 이렇게 적으면 나중에 파산하겠는데?' 싶으면 장부를 지우고 대출을 거절하는 철저한 기록 관리와 같습니다."

+++

## Ⅳ. 알고리즘의 복잡도 분석

| 항목 | 복잡도 | 설명 |
|:---|:---|:---|
| **시간 복잡도** | $O(M \times N^2)$ | $M$(자원 종류), $N$(프로세스 수). 모든 프로세스를 돌며 자원을 비교해야 함. |
| **공간 복잡도** | $O(M \times N)$ | Available, Max, Allocation, Need 등 행렬 저장 공간 필요. |

- **📢 섹션 요약 비유**: "손님도 많고 돈의 종류(달러, 유로, 원화)도 많을수록 은행원이 계산기를 두드리는 시간이 제곱으로 늘어나는 것과 같습니다."

+++

## Ⅴ. 결론 및 실무적 교훈

### 1. 요약
- 은행원 알고리즘은 다중 자원 환경에서 데드락을 원천 봉쇄하는 강력한 수학적 도구이다.

### 2. 최종 결론
- 이론적으로는 완벽하지만, 현실의 OS에서는 프로세스의 개수가 유동적이고 최대 자원 요구량을 알기 어렵기 때문에 직접 구현하기보다는 임계 구역(Critical Section) 설계의 논리적 기반으로 활용된다.

- **📢 섹션 요약 비유**: "모든 상황을 예측할 수만 있다면 사고를 100% 막을 수 있지만, 현실은 늘 예상치 못한 손님이 찾아오는 법이라는 교훈을 줍니다."
