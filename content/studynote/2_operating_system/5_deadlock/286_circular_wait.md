# 순환 대기 (Circular Wait)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 순환 대기(Circular Wait)는 대기 중인 프로세스 집합 $\{P_0, P_1, \dots, P_n\}$이 서로가 가진 자원을 꼬리에 꼬리를 물고 대기하여 **닫힌 루프(Cycle)**를 형성한 상태이다.
> 2. **가치**: 교착 상태 발생 4조건 중 가장 직접적인 징후이며, 자원에 고유 번호를 부여하는 **계층적 할당(Ordering)**을 통해 가장 효과적으로 예방할 수 있다.
> 3. **융합**: 그래프 이론의 사이클 탐지 알고리즘(DFS 등)을 통해 시스템 수준에서 교착 상태를 식별하고 해소하는 핵심 논리적 근거가 된다.

+++

## Ⅰ. 순환 대기 (Circular Wait)의 개념

### 1. 정의
- 프로세스들의 대기 관계가 원형을 이루는 것을 말한다.
- $P_0$는 $P_1$이 점유한 자원을 기다리고, $P_1$은 $P_2$가 점유한 자원을 기다리며, 최종적으로 $P_n$이 $P_0$이 점유한 자원을 기다리는 형태이다.

### 2. 특징
- **상호 의존의 역설**: 모든 프로세스가 무언가를 얻기 위해 기다리고 있지만, 그 '무언가'를 줄 수 있는 프로세스 역시 다른 것을 기다리느라 멈춰 있다.
- **필요충분성**: 단일 인스턴스 자원 환경에서는 순환 대기(사이클)의 존재가 곧 교착 상태의 **필요충분조건**이 된다.

### 3. 💡 비유: '둥글게 앉은 선물 교환'
- 사람들이 원형으로 둘러앉아 자기 오른쪽 사람이 가진 선물을 받으려고 기다리는 것과 같다. 누구도 먼저 선물을 내려놓지 않으면, 아무도 새로운 선물을 받을 수 없다.

- **📢 섹션 요약 비유**: "꼬리가 머리를 물고 있는 뱀처럼, 시작과 끝이 맞물려 전체가 굳어버린 형국입니다."

+++

## Ⅱ. 기술적 구조 및 사이클 분석 (Deep Dive)

### 1. 순환 대기의 형성 조건
- 앞선 세 가지 조건(상호 배제, 점유 대기, 비선점)이 이미 충족된 상태에서, 마지막으로 대기 화살표가 원을 그리며 연결될 때 순환 대기가 완성된다.

### 2. ASCII 순환 대기 그래프 (Resource-Allocation Graph)
```text
      [ Resource R1 ] <───── (Holds) ───── [ Process P1 ]
             │                                    ▲
             │                                    │
        (Requests)                           (Requests)
             │                                    │
             ▼                                    │
      [ Process P2 ] ───── (Holds) ─────> [ Resource R2 ]

    [설명]
    - P1은 R1을 갖고 R2를 원함.
    - P2는 R2를 갖고 R1을 원함.
    - 화살표를 따라가면 P1 -> R2 -> P2 -> R1 -> P1 의 폐곡선(Cycle)이 형성됨.
```

### 3. 기술적 상세 해설
- **사이클 탐지 (Cycle Detection)**: 운영체제는 주기적으로 자원 할당 그래프를 분석한다. $N$개의 노드가 있을 때 $O(N^2)$ 정도의 복잡도로 사이클 존재 여부를 확인할 수 있다.
- **다중 인스턴스의 변수**: 자원이 여러 개(Multi-instance)인 경우, 사이클이 있다고 해서 반드시 교착 상태인 것은 아니다. 다른 경로를 통해 자원이 해제될 가능성이 있기 때문이다. 하지만 순환 대기는 교착 상태의 **강력한 전조**이다.

- **📢 섹션 요약 비유**: "길이 사방으로 막힌 교차로에서, 모든 차가 앞 차가 빠지기만을 기다리며 꼬리에 꼬리를 물고 서 있는 교통 마비와 같습니다."

+++

## Ⅲ. 순환 대기 예방 전략 (Prevention Strategy)

### 1. 자원 순서화 (Resource Ordering)
- 모든 자원 유형에 일련번호 $F(R_i)$를 부여하고, 프로세스는 반드시 **번호가 증가하는 순서**로만 자원을 요청할 수 있게 강제한다.

| 요청 시나리오 | 규칙 위반 여부 | 결과 |
|:---|:---|:---|
| Lock(1) -> Lock(5) | 정상 (순차 증가) | 허용됨 |
| Lock(5) -> Lock(1) | **위반 (역순 요청)** | 거부됨 또는 이전 자원 반납 필수 |

### 2. 기술적 정당성 증명
- 모든 프로세스가 오름차순으로만 자원을 요청한다면, 번호가 가장 큰 자원을 가진 프로세스는 더 이상 기다릴 상위 자원이 없거나, 하위 자원을 기다릴 일이 없으므로 순환 구조가 절대 만들어질 수 없다.

- **📢 섹션 요약 비유**: "계단을 올라갈 때 반드시 아래 칸부터 밟아야 한다는 규칙을 세워, 내려오는 사람과 중간에서 부딪히는 일을 원천 차단하는 것과 같습니다."

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy)

### 1. 실무 시나리오: 리눅스 커널의 LWP 락 순서
- **상황**: 커널 개발자가 두 개의 프로세스 제어 블록(PCB)을 동시에 수정해야 함.
- **판단**: 각 PCB의 메모리 주소(Address)를 비교하여, 주소값이 **낮은 것부터 먼저 락을 거는** 관례를 준수한다. 주소값 자체가 고유 번호 역할을 하여 순환 대기를 방지한다.

### 2. 안티패턴: 비대칭적 자원 확보
- 스레드 A는 `A->B` 순서로, 스레드 B는 `B->A` 순서로 자원을 잡는 코드. 이는 순환 대기를 만드는 가장 전형적인 'Deadlock Recipe'이다.

- **📢 섹션 요약 비유**: "오른쪽 통행과 왼쪽 통행이 정해지지 않은 좁은 골목길에서 서로 마주쳐 꼼짝 못 하는 무질서한 상황을 경계해야 합니다."

+++

## Ⅴ. 기대효과 및 결론

### 1. 기대효과
- **확정적 예방**: 4조건 중 가장 제어하기 쉽고 확실한 예방책을 제공한다.
- **시스템 안정성**: 프로그래밍 규약(Convention)만으로도 복잡한 런타임 검사 없이 데드락을 방지할 수 있다.

### 2. 결론
순환 대기는 교착 상태의 기하학적 완성이다. 이를 깨뜨리기 위한 '자원 순서화' 전략은 오늘날 고성능 서버 프레임워크부터 디바이스 드라이버 설계에 이르기까지 가장 널리 쓰이는 표준적인 데드락 방지 기법이다.

- **📢 섹션 요약 비유**: "순환하는 소용돌이의 흐름을 끊어, 물줄기가 한 방향으로 시원하게 흐르도록 만드는 수로 정비 작업과 같습니다."

### 📌 관련 개념 맵 (Knowledge Graph)
- **자원 할당 그래프 (RAG)**: 순환 대기를 시각화하는 도구.
- **교착 상태 발생 4조건**: 순환 대기가 포함된 전체 요건.
- **DFS (Depth First Search)**: 사이클 탐지에 쓰이는 알고리즘.
- **계층적 락 (Hierarchical Locking)**: 순서화 전략의 DB 응용.

2. 0번 친구는 1번을, 1번은 2번을, 마지막 친구는 다시 0번 친구를 기다리니까 아무도 장난감을 못 바꾸죠.
3. 이럴 땐 '번호가 작은 장난감부터 먼저 빌리기'라는 규칙을 정하면, 줄이 꼬이지 않고 차례차례 장난감을 가질 수 있답니다!
