# 교착 상태 회피 (Deadlock Avoidance)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 교착 상태 회피(Deadlock Avoidance)는 자원 할당 요청이 들어왔을 때, 해당 할당이 시스템을 교착 상태에 빠뜨릴 가능성이 있는 '불안전 상태'로 만드는지 미리 검사하여 거절하거나 승인하는 동적 전략이다.
> 2. **가치**: 데드락 예방(Prevention)보다 자원 할당 조건을 완화하여 시스템 처리량(Throughput)을 높이면서도, 데드락은 발생하지 않도록 보장하는 지능형 리소스 관리 모델이다.
> 3. **융합**: 프로세스가 필요로 할 최대 자원량(Max Demand)을 미리 알고 있어야 한다는 강력한 전제 조건이 필요하며, 대표적으로 은행원 알고리즘(Banker's Algorithm)이 사용된다.

+++

## Ⅰ. 교착 상태 회피의 개념 및 특징

### 1. 정의
- 자원 할당 후에도 시스템이 항상 '안전 상태(Safe State)'를 유지할 수 있도록 동적으로 제어하는 방법이다.

### 2. 주요 전제 조건
- 프로세스 수 고정: 실행 중인 프로세스의 수가 일정해야 함.
- 자원 종류 및 수 고정: 가용 자원의 양이 변하지 않아야 함.
- **최대 요구량 고지**: 모든 프로세스는 실행 전 자기가 쓸 최대 자원량을 미리 신고해야 함.

- **📢 섹션 요약 비유**: "돈을 빌려줄 때마다 은행원이 이 사람이 나중에 갚을 수 있을지, 다른 사람들에게 줄 돈이 부족해지진 않을지 미리 계산해 보고 빌려주는 것과 같습니다."

+++

## Ⅱ. 기술적 작동 원리 및 ASCII 다이어그램

### 1. 안전성 검사 (Safety Check) 매커니즘
- 시스템은 '할당 가능한 자원'이 있더라도, 그 자원을 줬을 때 나중에 다른 프로세스가 자원을 요청하면 데드락이 생길 것 같으면 할당을 미룬다.

### 2. ASCII 작동 흐름도
```text
    [ Resource Request from Pi ]
               │
               ▼
    [ Current State + This Allocation ]
               │
       [ Run Safety Algorithm ]
               │
    ┌──────────┴──────────┐
    ▼                    ▼
 [ SAFE STATE? ]    [ UNSAFE STATE? ]
    │                    │
    ├─▶ YES: Allocate    └─▶ NO: Pi must WAIT
    └─▶ Status Update        (Keep Current Safe State)
```

### 3. 핵심 판단 기준
- **안전 상태 (Safe State)**: 모든 프로세스가 데드락 없이 완료될 수 있는 할당 순서(Safe Sequence)가 존재하는 상태.
- **불안전 상태 (Unsafe State)**: 데드락이 발생할 가능성이 있는 상태 (반드시 발생하는 것은 아님).

- **📢 섹션 요약 비유**: "바둑을 둘 때 한 수를 두기 전에, 이 수를 둠으로써 내 돌이 다 잡히는 '외통수'에 걸리지 않을지 미리 몇 수 앞을 내다보는 것과 같습니다."

+++

## Ⅲ. 회피 전략의 기술적 비교

| 비교 항목 | 예방 (Prevention) | 회피 (Avoidance) |
|:---|:---|:---|
| **제한 방식** | 정적 (규칙 기반) | **동적 (상태 기반)** |
| **자원 정보** | 필요 없음 | **최대 요구량(Max) 필요** |
| **자원 이용률** | 낮음 | **상대적으로 높음** |
| **시스템 오버헤드** | 거의 없음 | **높음 (할당 시마다 계산)** |

- **📢 섹션 요약 비유**: "비가 올까 봐 아예 외출을 안 하는 것이 '예방'이라면, 하늘을 보고 구름이 몰려올 때만 처마 밑으로 피하는 것이 '회피'입니다."

+++

## Ⅳ. 주요 알고리즘 및 실무적 한계 (Strategy)

### 1. 대표 알고리즘
- **자원 할당 그래프 (Single Instance)**: 자원 종류당 인스턴스가 하나일 때 순환(Cycle) 여부로 판단.
- **은행원 알고리즘 (Multiple Instances)**: 자원 종류당 인스턴스가 여러 개일 때 수치적 계산으로 판단.

### 2. 실무적 한계
- **미래 예측의 어려움**: 프로세스가 실행 중에 얼마나 많은 자원을 쓸지 미리 알기는 매우 힘들다.
- **연산 비용**: 자원 요청이 빈번할 때마다 복잡한 안전성 검사 알고리즘을 돌리는 것은 CPU 성능에 큰 부담을 준다.

- **📢 섹션 요약 비유**: "모든 손님이 식당에 들어올 때 '나는 오늘 소주 3병과 안주 5개를 먹을 거야'라고 미리 말해야만 장사를 하는 식당은 운영하기 힘든 것과 같습니다."

## Ⅴ. 결론 및 향후 전망

### 1. 요약
- 회피 전략은 안전성과 효율성 사이의 훌륭한 타협점이다. 비록 제약 사항이 많지만 데드락을 방지하는 가장 똑똑한 방법으로 평가받는다.

### 2. 현대적 의의
- 최근에는 클라우드 리소스 스케줄러나 컨테이너 오케스트레이션(K8s)에서 자원 한도(Limit)와 요청(Request)을 기반으로 워크로드를 배치할 때 이 철학이 응용된다.

- **📢 섹션 요약 비유**: "무모하게 돌진하지도, 겁먹어 멈춰 서지도 않고, 주변 상황을 살피며 조심스럽게 나아가는 것이 가장 오래 생존하는 길임을 보여줍니다."
