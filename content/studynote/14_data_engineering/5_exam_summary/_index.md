> **핵심 인사이트**
> 1. **본질**: 데이터 엔지니어링은 원시 데이터를 신뢰할 수 있는 인사이트로 변환하는 End-to-End 배관 시스템으로, 수집 (Ingestion), 저장 (Storage), 처리 (Processing), 조율 (Orchestration), 거버넌스 (Governance)의 5대 축으로 구성된다.
> 2. **가치**: 고품질 피처와 신뢰할 수 있는 파이프라인을 구축하는 데이터 엔지니어링이 전체 AI/ML 시스템의 성능을 70% 이상 결정하며, 데이터 레이크하우스 (Lakehouse) 아키텍처가 정형과 비정형 데이터를 통합 관리하는 modern 표준이 되었다.
> 3. **융합**: Kafka, Spark, Flink为代表的 실시간 스트리밍과 배치 처리 시스템이 융합되고, LLM을 활용한 데이터 품질 자동화와 피처 엔지니어링 자동화가加速하고 있으며, 이는 데이터 엔지니어링의 미래 방향을 제시한다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

데이터 엔지니어링 시험 빈출 요약은 데이터 엔지니어링 도메인의 핵심 개념, 아키텍처 패턴, 실무 기술을 기술사 시험 준비 관점에서 체계적으로 정리한 cheat sheet이다. 이 섹션은 각 기술의 깊이보다는"무엇을 아는가"와"어디에 적용하는가"에 집중하며, 상호 연관성을 통해 통합적 사고력을 기르는 것을 목표로 한다.

### 필요성: 기술사 시험에서 데이터 엔지니어링의 중요성

현대 소프트웨어 시스템에서 데이터는 단순한 부산물이 아니라 기업의 핵심 자산이다. 시스템 아키텍처, 운영 효율성, 인공지능 활용 등 모든 측면에서 데이터의 품질과 파이프라인의 신뢰성이直接或간접적 영향을 미친다. 따라서 기술사 시험에서 데이터 엔지니어링 관련 문제는 반드시 출제되며, 특히 다음 영역에서 빈출한다.

첫째, 분산 시스템 아키텍처 설계에서 HDFS, Kafka, Spark 등의 역할과 트레이드오프를 묻는 문제. 둘째, 데이터 모델링에서 데이터 레이크, 레이크하우스, 데이터 웨어하우스의 차이와 선택 기준을 묻는 문제. 셋째, 실시간 vs 배치 처리에서 Lambda, Kappa 아키텍처의 비교와 적용 시나리오를 묻는 문제. 넷째, 데이터 품질과 거버넌스에서 피처 스토어, 데이터 계보, 데이터 카탈로그의 중요성을 묻는 문제.

### 문제의식: 데이터 엔지니어링의 5대 축

데이터 엔지니어링은 다음 5대 축으로体系화할 수 있으며, 모든 기술적 판단은 이 축들의 트레이드오프에서 출발한다.

첫째, 수집 (Ingestion): 배치 vs 실시간, Push vs Pull, CDC (Change Data Capture) 등. 둘째, 저장 (Storage): 파일 형식 (Parquet, ORC, Avro), 스토리지 계층 (Hot/Warm/Cold), 복제 전략 등. 셋째, 처리 (Processing): 배치 vs 스트리밍, 내림차순 vs 오름차순 처리, 상태 관리 등. 넷째, 조율 (Orchestration): DAG 설계, 작업 의존성, 재시도 및 복구 정책 등. 다섯째, 거버넌스 (Governance): 데이터 품질, 보안, 카탈로그, 계보 등.

### 출제 포인트: 기술사적 사고방식

기술사 시험에서 데이터 엔지니어링 문제는 단순히 도구 사용법을 묻는 것이 아니라,"어떤 상황에 어떤 도구를 왜 선택하는가"를 묻는다. 따라서 이 요약은 개별 도구의 설명보다 도구 간의 비교와 선택 기준에 집중한다. 출제자는 후보자의""어떻게"보다"왜"를 묻는倾向이 있으므로, 각 결정의 Trade-off를 명확히 이해해야 한다.

### 비유

데이터 엔지니어링의 5대 축은，就好像 수도꼭지에서 집안 각 방의 수도 파이프에 이르기까지의 전체 배관 시스템과 같다. 수도꼭지(수집)에서 나온 물이 서로 다른 관(저장)을 지나, 정수 시설(처리)을 거치고, 배관 제어 시스템(조율)에 의해 각 방으로 전달되며, 수질 검사소(거버넌스)가全程의 물質을管理한다. 만약 한 구간이라도 문제가 생기면 最终적으로到達하는水の品質受到影响を受ける.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 출제 빈출 데이터 아키텍처 패턴

#### 데이터 레이크 vs 데이터 레이크하우스 vs 데이터 웨어하우스

이 세 아키텍처는 실무에서 빈번히 출제되며, 각각 다른 사용 시나리오와 Trade-off를 가진다.

데이터 레이크 (Data Lake)는 원시 데이터를 원본 그대로 객체 스토리지 (S3, GCS, Azure Blob)에 저장하는架构로, 정형, 반정형, 비정형 데이터를 모두 저장할 수 있어 유연성이 높다. 그러나 스키마 제약을 적용하지 않으므로"데이터 늪 (Data Swamp)"으로 전락할 위험이 높다.

데이터 웨어하우스 (Data Warehouse)는 정형 데이터만을 대상으로 엄격한 스키마를 적용하여 분석에 적합한 구조로 정리하는 architecture다. ACID 트랜잭션을 지원하며 BI (Business Intelligence) 대시보드와 애드혹 쿼리에 최적화되어 있지만, 비정형 데이터와 반정형 데이터의 저장이 어렵고 확장 비용이 높다.

데이터 레이크하우스 (Data Lakehouse)는 두 architecture의 장점을 결합한 것으로, 객체 스토리지 위에 Data Lake 포맷 (Apache Iceberg, Delta Lake, Apache Hudi)을 적용하여 스키마 enforcement와 ACID 트랜잭션을 제공한다. 이를 통해 Data Lake의 유연성과 Data Warehouse의 신뢰성을 동시에 확보한다. 표면상으로는 S3에 Parquet 파일을 저장하지만, 내부적으로는 트랜잭션 로그, 스키마 enforcement, time travel query等功能을 제공한다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │              데이터 레이크 vs 레이크하우스 vs 웨어하우스 비교               │
  ├─────────────────────────────────────────────────────────────────────┤
  │
  │  [Data Lake]              [Data Lakehouse]         [Data Warehouse] │
  │
  │  ┌─────────────┐         ┌─────────────┐         ┌─────────────┐  │
  │  │  Raw Data   │         │  Raw Data   │         │ 정형 데이터  │  │
  │  │  (JSON/CSV/ │         │  (Parquet/  │         │ (Schema-on- │  │
  │  │   XML/IMG)  │         │   ORC)      │         │   Write)    │  │

  │         │                       │                       │          │

  │    (유연하지만             (유연성 + 신뢰성)         (엄격한 무결성) │
  │     관리 필요)                                              │          │
  │         │                       │                       │          │
  │         ▼                       ▼                       ▼          │
  │  ┌─────────────┐         ┌─────────────┐         ┌─────────────┐  │
  │  │   분석가    │         │ 분석가 +     │         │   BI 사용자  │  │
  │  │ (데이터 과학)│         │ ML 엔지니어  │         │  (SQL/Dash) │  │

  │                                                                      │
  │  선택 기준:                                                          │
  │  - 비정형/반정형 중심 + 비용 절감 + 유연성 필요? → Data Lake         │
  │  - 정형 분석 + BI + ACID 필요? → Data Warehouse                     │
```

**[다이어그램 해설]** 데이터 아키텍처 선택은"데이터 종류"와"사용자 역할"에 따라 달라진다. Data Lake는 원시 데이터를 있는 그대로 저장하므로 나중에 다양한 분석이 가능하지만,Schema-on-Read 방식이므로 데이터를 읽을 때 스키마를 적용한다. 이 말은 즉, 저장은 유연하지만 데이터 품질 관리와 탐색이 어렵다는 뜻이다. Data Warehouse는Schema-on-Write 방식으로 데이터를 저장할 때 이미 스키마를 적용하므로 저장 시점에 데이터 무결성이 검증되지만, 데이터 종류가 정형으로 제한되고 사전 스키마 정의에 시간과 도메인 지식이 필요하다. Data Lakehouse는 이 문제를 해결하기 위해 Iceberg/Delta Lake/Hudi 같은 테이블 포맷을 사용하여Schema-on-Write의 일부 기능 (트랜잭션, 스키마 enforcement, time travel)을 제공하면서도 원시 파일 기반 스토리지를 활용한다. 시험에서는"비정형 데이터부터 시작해서 점진적으로 구조화한다"는 레이크하우스의 핵심 사상을 이해해야 한다.

### 핵심 기술 비교: Kafka vs Spark Streaming vs Flink

실시간 데이터 처리에서 가장 중요한 세 가지 도구인 Kafka, Spark Streaming, Flink를 비교한다.

Kafka는 메시징 시스템으로, 고처리량 (>1M messages/sec)의耐久性 메시지 스트림을 제공한다. Topic, Partition, Consumer Group 개념으로 데이터 흐름을 관리하며, 스트림 처리 자체보다는"전달 보장 (Delivery Guarantee)"에 집중한다.

Spark Structured Streaming은 micro-batch 처리 방식으로, 짧은 시간 간격 (예: 1초)마다 미니 배치를 처리한다. 기존 Spark Batch API와 unified programming 모델을 제공하므로 기존 Spark 사용자가 쉽게 전환할 수 있다. 다만 micro-batch 방식은 very low latency (<100ms)가 필요한 사용에는 부적합하다.

Apache Flink는 true streaming (one record at a time) 방식으로, very low latency (>1M records/sec) 처리가 가능하며, event time processing, watermarks, session window 등 고급 스트림 개념을 지원한다. 상태ful 연산과 exactly-once 시맨틱스를 native하게 지원하여 복잡한 스트림 분석에 적합하다.

### 출제 빈출 핵심 개념 정리

| 개념 | 핵심 설명 | 출제 포인트 |
|:---|:---|:---|
| **ETL vs ELT** | ETL은 적재 전 변환, ELT는 적재 후 목적지 내부에서 변환 | ELT가 cloud DW의 컴퓨팅 파워活用에 유리 |
| **Lambda vs Kappa** | Lambda는 배치+스트리밍 2-layer, Kappa는 단일 스트림 처리 | Kappa 선택 시 복잡한 배치 연산이 없는 경우 |
| **Schema-on-Write vs Schema-on-Read** | Write 시 검증 vs Read 시 스키마 적용 | 레이크는 전자, 레이크하우스는Hybrid |
| **ACID 트랜잭션** | 원자성, 일관성, 격리, 내구성 | DW는 완전 지원, DL은 제한적 |
| **피처 스토어** | 훈련/서빙 피처 중앙 관리 | 훈련-서빙 스큐 해결 |
| **CDC (Change Data Capture)** | DB 변경분을 캡처하여 스트림으로 | 데이터 동기화 도구 (Debezium 등) |
| **파티셔닝** | 대용량 데이터를 파티션으로 분할 | 쿼리 성능, 병렬 처리 최적화 |
| **셔플 (Shuffle)** | RDD 간 데이터 재분배 | 네트워크 병목의 주요 원인 |
| **Wal (Write-Ahead Log)** | 장애 복구를 위한 사전 기록 | 카프카, 플링크 모두 사용 |
| **Checkpoint** | 스트림 처리 상태 스냅샷 | exactly-once 보장 핵심 |

### 비유

Schema-on-Write와 Schema-on-Read의 차이는，好像是一次性筷子和 reusable餐具と的比喱一样. Schema-on-Read는，就像把各种形状的食材（数据）先全部放入冰箱（Data Lake），需要时（Read）再拿出需要的量和解冻方法（应用Schema）。 장점是可以把所有原材料都留下来，缺点是需要管理一个混乱的冰箱。 Schema-on-Write는，就像提前把食材按固定规格包装好再放入冰箱，存储时就确定好了用途。优点是冰箱整洁有序，缺点是如果需要新的规格就很难处理。

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 출제 빈출 비교표

#### 배치 처리 vs 실시간 처리

| 항목 | 배치 처리 (Batch) | 실시간 처리 (Real-time) |
|:---|:---|:---|
| **처리 지연** | 수분~수시간 | 수십 ms~수초 |
| **처리량** | 매우 높음 | 중간~높음 |
| **데이터 범위** | 전체/대규모 배치 | 최근 데이터 (윈도우) |
| **적용scenario** | 일일 보고서, 대규모 조인 | 모니터링, 추천, 사기 탐지 |
| **결합 방식** | Lambda/Kappa 아키텍처 | Kafka + Flink/Spark |
| **대표 도구** | Spark, Hadoop MapReduce | Flink, Kafka Streams |

#### SQL vs NoSQL vs NewSQL

| 항목 | SQL (RDBMS) | NoSQL | NewSQL |
|:---|:---|:---|:---|
| **데이터 모델** | 관계형 | 다양 (Key-Value, Document, Column, Graph) | 관계형 + 수평 확장성 |
| **트랜잭션** | ACID 완전 지원 | BASE (Eventually Consistent) | ACID + 분산 |
| **확장 방식** | Scale-up 중심 | Scale-out | Scale-out |
| **대표 제품** | PostgreSQL, MySQL | MongoDB, Cassandra, Redis | CockroachDB, TiDB, Spanner |
| **적용scenario** | 금융, 주문 관리 | 로그, 세션, 카탈로그 | 분산 트랜잭션 필요 OLTP |

#### 스트리밍 처리 엔진 비교

| 항목 | Apache Kafka Streams | Apache Flink | Spark Structured Streaming |
|:---|:---|:---|:---|
| **처리 모델** | One record at a time (내장) | True streaming | Micro-batch |
| **상태 관리** | 내장 (RocksDB) | 내장 (多样) |Checkpoint via RDD |
| **장애 복구** | Kafka Partition 활용 | Checkpoint 기반 | Checkpoint 기반 |
| ** exactement once** | Kafka transaction 활용 | 내장 | Delta Lake 연동 |
| **사용 난이도** | 낮음 (라이브러리) | 높음 (standalone cluster) | 중간 (Spark 통합) |
| **연속 처리 적합성** | 낮음 (stateless 위주) | 매우 높음 | 중간 |

### 과목 융합: 타 과목과의 연결점

데이터 엔지니어링은 컴퓨터구조, 운영체제, 네트워크, 데이터베이스 등 여러 과목과 深하게 연결되어 있다. 이러한 융합점을 이해하면 기술사 시험에서統合的な 관점의 문제가 출제될 때対応할 수 있다.

**컴퓨터구조 (CA) 연결점**: Spark의 병렬 처리는 CPU 코어 수에 영향을 받으며, 셔플 (Shuffle) 연산은 네트워크 Bandwidth가 병목이 된다. GPU 활용 (RAPIDS)은 데이터 처리 가속에 활용된다. HDFS의 Rack-aware 배치는 네트워크 토폴로지를 고려한 설계로, 장애耐성 Alongside 성능을 동시에 확보한다.

**운영체제 (OS) 연결점**: Kafka의 Journal (Write-Ahead Log)은 OS의 fsync 메커니즘을 활용하여 내구성을 확보한다. HDFS의 DataNode는 로컬 파일 시스템 위에서 블록을 관리하며, Linux I/O 스케줄러의 영향을 받는다. Kubernetes 위에서 Airflow, Spark, Flink가 실행될 때 container resource limits와 ephemeral storage 관리가 중요하다.

**네트워크 (Network) 연결점**: 분산 처리 환경에서 네트워크 병목은 성능 저하의 주요 원인이다. Spark의 셔플 시 데이터 크기와 네트워크 대역폭 사이의 관계, Kafka의 복제 시 네트워크 활용, Flink의 네트워크 버퍼管理等이 핵심이다. Rack-to-Rack 복제로 네트워크 병목을 줄이는 설계가 출제될 수 있다.

### 비유

NewSQL은，就好像多年积累的老牌技术（关系型数据库）的经验和新技术（分布式系统）の结合，就像一个经验丰富的工匠学会了使用电动工具，在保持传统手艺的同时，也能够大批量高效地生产。

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 출제 빈출 실무 시나리오 및 의사결정

#### 시나리오: 레거시 MySQL에서 Kafka를 통한 실시간 분석 시스템 전환

**배경**: 전자상거래平台的 주문 데이터가 매일 일일 배치로 데이터 웨어하우스에 적재되고 있으며, 관리자들은"오늘 오전 10시 이후 주문량"을 당일 오후에나 확인할 수 있는 상황이다. 이에 대한 개선 요구가 있었다.

**선택 과장**: 기존 nightly batch를 유지하면서, 동시에 Debezium CDC를 통해 MySQL의 변경 로그를 Kafka Topic으로 스트리밍하고, Apache Flink로 이를 소비하여 실시간 주문 대시보드용 Key-Value 스토어 (Redis)에写入한다.

**의사결정 포인트**:
1. 배치와 실시간을 별도 파이프라인으로 운영할 것인가 (Lambda 아키텍처) 아니면 통합할 것인가?
2. CDC 도구로 Debezium을 선택한 이유는 무엇인가?
3. 분석용으로만 사용된다면 Kafka Streams가 아니라 Flink를 선택한 이유는?
4. 대시보드 데이터 저장은 Redis가 적합한가, 다른 것이 더 나은가?

**답변 핵심**: 이 시나리오의 핵심 의사결정은"실시간 요약 데이터"와"전체 주문 데이터"가 다른 읽기 패턴을 가진다는 점이다. 대시보드는 수십 개의 Key (예:時間帯별 주문 수, 카테고리별 주문额 등)에 대한 빠른 조회가 필요하므로, Redis와 같은 Key-Value 스토어가 적합하다. 반면 전체 주문 데이터에 대한 분석은 배치로 DW에 적재하여 처리한다. CDC를 Debezium으로 선택한 이유는 MySQL의 binary log를解析하는 방식으로 애플리케이션에 영향을 주지 않기 때문이다.

#### 시나리오: 피처 드리프트로 인한 모델 재훈련 트리거

**배경**: 금융사기 탐지 모델이 프로덕션에 배포된 지 3개월이 지나면서, 피처"PROMOTION_YN" (프로모션 여부)의 분포가 크게 변화했다. 과거에는 5% 수준이었으나, 최근에는 30%로 급증했다.

**분석**: 이 피처의 급증은 마케팅 캠페인Launch에 따른 일시적 현상인지, 지속적인 트렌드인지 분석해야 한다. 마케팅 팀과의 상담 결과, 경쟁사 대응을 위한 평생 promotional campaign이 시작되었음이 밝혀졌다.

**의사결정**: 피처 분포 변화가 일시적이면 모델을 재훈련할 필요가 없지만, 지속적이라면 재훈련이 필수적이다. 그러나 마케팅 캠페인에 따른"PROMOTION_YN"의 의미 자체는 동일하므로 (프로모션 참여 여부), 이 Feature 자체의 예측력은 유지된다. 오히려"PROMOTION_YN" 비율 변화에 따라 다른 Features와의 상호작용이 변화할 수 있으므로, 데이터 사이언티스트의 판단이 필요하다.

**트레이드오프**: 즉각적 재훈련은 비용이 크고, 재훈련 없는 방치는 성능 저하를忍受하게 만든다. 실무에서는 PSI (Population Stability Index) 등의定量 지표와 도메인 전문가의 판단을 결합하여 재훈련 여부를 결정한다.

### 출제 체크리스트: 아키텍처 선택 기준

| 상황 | 권장 아키텍처 | 선택 이유 |
|:---|:---|:---|
| 정형 데이터 + BI 분석 + ACID 필요 | Data Warehouse |-schema-enforcement + 트랜잭션 |
| 비정형 + ML 훈련 + 비용 절감 | Data Lake | 객체 스토리지低廉 + 유연성 |
| 둘 다 + 실시간 필요 | Data Lakehouse | Iceberg/Delta Lake로 통합 |
| 배치 정확성 + 실시간 속도 | Lambda 아키텍처 | 배치+스트리밍 결합 |
| 단순 스트리밍 only | Kappa 아키텍처 | 단일 코드베이스 |
| 초저지연 스트리밍 | Apache Flink | True streaming |
| 마이크로 배치 + 개발 용이성 | Spark Structured Streaming | Spark 생태계 통합 |

### 출제 빈출 안티패턴 정리

| 안티패턴 | 설명 | 올바른 대안 |
|:---|:---|:---|
| **데이터 늪** | Data Lake에 스키마/카탈로그 없이 원시 데이터만 누적 | 데이터 카탈로그 + 거버넌스 정책 |
| **샤드 화제** | 파티션 키 설계 실수로 특정 파티션에 데이터 집중 | 파티션 키 再検討 + Salting |
| **Lambda 이중 관리** | 배치/스트리밍 코드 중복으로 인한 일관성 문제 | Kappa 또는 공통 라이브러리 추출 |
| **피처 엔지니어링 중복** | 훈련/서빙 피처 로직 불일치 | 피처 스토어 도입 |
| **빈번한 셔플** |过度한 셔플로 네트워크 병목 발생 | 파티션 수 조정 + co-partitioned 조인 |

### 비유

피처 드리프트监控는，就好像汽車のダイヤ改正と比喱一样的. 列車のダイヤ（데이터 분포）が大幅に変われば、それに合わせて运行计划（모델）を修正する必要がある。 しかし、ダイヤが多少変わっても路線の本质は変わらないので、过度な修正は不要。 关键在于"どの程度の变化"を"どの程度で修正するか"の判断である。

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 데이터 엔지니어링의 미래 기술 트렌드

#### 실시간 AI (Real-time AI)의 확산

배치 ML에서 실시간 ML로의 전환이加速하고 있다. Kafka + Flink 기반의 실시간 피처 계산과 모델 서빙의 결합으로, 수백 ms 이내의 실시간 추론이 가능해지고 있다. 향후에는 더 많은 애플리케이션이"지금 이 순간의 데이터"를 활용하는 방향으로 진화할 것이다.

#### 데이터 팩토리 (Data Fabric)와 데이터 메시 (Data Mesh)의 융합

Data Fabric은メタデータ 기반의 자동화된 데이터 통합 아키텍처이고, Data Mesh는 도메인별 데이터 소유와_product thinking을 도입한 조직적 아키텍처다. 향후에는 기술적 아키텍처 (Fabric)와 조직적 아키텍처 (Mesh)의融合으로, 데이터 거버넌스와 데이터 Producer-Consumer 관계가より効率的になる展望이다.

#### AI 협업 데이터 관리 (AI-augmented Data Management)

머신러닝이 데이터 관리 자체에 적용되는案例が的增加하고 있다. 예를 들어, 데이터 품질 이상을 자동으로 탐지하고修復하거나, 피처 선택을 자동화하거나, 데이터 파이프라인의障害를予測하고预防하거나, 자연어로 데이터 쿼리를 수행하는 등이 그것이다. 이趋势는 데이터 엔지니어의 역할을"데이터 파이프라인 구축자"에서"데이터 시스템策士"로 변화시킬 것이다.

### 최종 정리: 데이터 엔지니어링 핵심 Checklist

데이터 엔지니어링 시스템을 설계하거나 평가할 때 반드시 점검해야 할 핵심 포인트은 다음과 같다.

첫째, 데이터 수집에서 소스와sink의 연결은 안전한가? CDC, API, 스트리밍 등 수집 방식에 따라 데이터 중복과 순서 보장이 다르다. 둘째, 저장소에서 데이터 형식과 파티셔닝 전략은 적절한가? Parquet + Z-order 또는 분할 파티셔닝이 쿼리 성능에 결정적 영향을 미친다. 셋째, 처리 시스템에서 배치와 스트리밍의 조합은 요구사항에 맞는가? Lambda/Kappa 선택의 근거가 명확한가? 넷째, 조율 시스템에서 파이프라인의멱등성은 보장되는가?故障 시 재현이 가능한가? 다섯째, 거버넌스에서 데이터 계보와 품질 모니터링은 충분한가? 피처 드리프트와 모델 성능 저하를 감지할 수 있는가?

### 비유

데이터 엔지니어링의 최종 목표는，就好像すべての家庭에 깨끗하고 안전한 수도水を 공급하는 상수도 시스템과 같다. 水源（数据源）에서 每가정（分析/AI系统)에 이르기까지의全程에서 水質（데이터 품질）을 관리하고, 배관（파이프라인）이 고장나면 자동 복구되고, 수질 검사소（거버넌스）가持續적으로管理한다. 이것이 완벽な 데이터 엔지니어링 시스템が实现하면, 後段의 분석과 AI가"좋은 Decisions"를 내릴 수 있는 기초가 된다.

## 📌 관련 개념 맵 (Knowledge Graph)

- **[데이터 레이크 (Data Lake)]**: 원시 데이터를 원본 그대로 객체 스토리지에 저장하는架构로,Schema-on-Read 방식의 유연성을 가지지만 데이터 품질 관리 어려움.
- **[데이터 레이크하우스 (Data Lakehouse)]**: Data Lake와 Data Warehouse의 장점을 결합한架构로, Iceberg/Delta Lake/Hudi 포맷으로 스키마 enforcement와 ACID 트랜잭션 지원.
- **[Apache Kafka]**: 높은 처리량의 분산 메시징 시스템으로, 파티션 단위 병렬 처리와 오프셋 기반耐久性을 제공하여 스트림 처리의hub로 활용.
- **[Apache Flink]**: True streaming 처리 엔진으로, event time processing, watermarks, 상태ful 연산, exactly-once 시맨틱스를 native하게 지원.
- **[피처 스토어 (Feature Store)]**: 훈련과 서빙 간 피처 일관성을 보장하는 中央管理 시스템으로, 훈련-서빙 스큐를 원천적으로防止.
- **[MLOps]**: ML 모델의 개발, 훈련, 배포, 모니터링을 자동화하는 일련의 실천和方法論으로, DevOps의 원리를 ML에 적용한 것.
- **[CDC (Change Data Capture)]**: 데이터베이스의 변경 로그를 캡처하여 스트림으로伝送하는 기술로, Debezium이 대표적実装.
- **[Lambda/Kappa 아키텍처]**: 배치와 스트리밍 처리를 결합 (Lambda)하거나 단일 스트림으로 통합 (Kappa)하는 설계 패턴.

## 👶 어린이를 위한 3줄 비유 설명

1. 데이터 엔지니어링은，好像是一家大型自来水流处理工厂と一样的. 从水源（数据源）取水，通过各种处理过程（清洗、消毒、沉淀），最后向每家每户（用户）提供干净的饮用水（水等价物——高质量数据）。

2. 데이터 레이크は就好像一个 large 储水库に 모든 종류의 원시 데이터를 원본 그대로 저장하지만, 시간이 지나면 물에 이끼가 끼고雜草이 자라 관리가 어려워진다. 이것이"데이터 늪"이다.

3. 데이터 레이크하우스는，就好像安装了先进的水质管理系统的储水库一样. 원시 데이터를 저장하면서도 자동 수질 검사(ACID 트랜잭션)와 물 맛 관리(스키마 enforcement)를 함께하여, 깨끗한 물을 지속적으로 공급할 수 있게 되었다.

## 기술사 시험 참고: 데이터 엔지니어링 출제 예상 패턴

### 예상 논술형 문제

**문제 1**: 일 1000만 건의 사용자 이벤트를 수집하여 실시간 대시보드와 일일 ML 모델 재훈련에 활용하는 시스템을 설계하라. 데이터는 반정형 (JSON)이며,レイテン시 요구사항은 대시보드 5초, ML 재훈련은 매일 새벽이다.

**답변 포인트**: Kafka를 통한 실시간 수집과 S3への 배치 적재를 병행하고, Flink로 대시보드용 윈도우 집계를 수행하며, Airflow DAG로 새벽 ML 재훈련 파이프라인을トリガー한다. 피처 스토어를 활용하여 훈련-서빙 일관성을 보장한다.

**문제 2**: Apache Kafka와 Apache Flink의 차이를 설명하고, 각각 어떤 시나리오에 적합한지 기술하라.

**답변 포인트**: Kafka는 메시징 시스템 (数据传输/缓存)에 초첨, Flink는 스트림 처리 엔진 (상태 관리, 복잡한 윈도우, exactly-once)에 초점.低지연 복잡한 스트림 처리는 Flink, 단순한 이벤트 전달은 Kafka가 적합.

**문제 3**: 데이터 레이크가 데이터 늪으로 전락하는主な原因와 예방策을 설명하라.

**답변 포인트**: 원인: Schema 관리 부재, 카탈로그 부재, 품질监控 미흡. 예방策: 데이터 카탈로그 도입, 거버넌스 정책 적용, 피처 스토어 활용, 자동화된 품질 체크.
