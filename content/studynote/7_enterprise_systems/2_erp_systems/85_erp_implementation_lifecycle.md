# ERP 구축 생명주기

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ERP 구축 생명주기는 패키지 소프트웨어의 특성을 고려하여 계획, 분석, 설계, 구축, 구현, 운영의 단계를 거치며 기업 전반의 업무를 최적화된 시스템으로 전환하는 전체 공정이다.
> 2. **차별점**: 일반 개발(SI)과 달리 '패키지 선정'과 'Gap 분석'이 핵심이며, 시스템에 업무를 맞출 것인지(BPR) 시스템을 고칠 것인지(Customizing/CBO) 결정하는 과정이 프로젝트 성패를 좌우한다.
> 3. **가치**: 정형화된 방법론을 통해 구축 기간을 단축하고, 데이터 마이닝과 클렌징을 통한 고품질 데이터 이관을 실현하며, 빅뱅(Big Bang) 또는 단계적 오픈을 통해 안정적인 비즈니스 전환을 유도한다.

+++

## Ⅰ. 개요 (Context & Background)

1. **개념**: ERP 구축 생명주기는 단순히 프로그램을 코딩하는 과정이 아니라, 선진 업무 프로세스(Best Practice)를 우리 회사에 이식하는 '경영 혁신 프로젝트'의 라이프사이클이다. 패키지 기반이기 때문에 분석 단계에서 "이 프로그램이 우리 일을 할 수 있나?"를 따지는 **Gap 분석**이 프로젝트의 출발점이 된다.

2. **💡 비유**: ERP 구축은 **'명품 기성복을 내 몸에 맞게 수선해서 입는 과정'**과 같습니다. 먼저 내 체형(업무)과 어울리는 옷(패키지)을 고르고, 내 팔다리가 옷보다 길거나 짧은 부분(Gap)을 찾아내어 옷을 줄이거나 늘리는(커스터마이징) 단계를 거쳐 최종적으로 완벽한 핏(Go-live)을 완성하는 과정입니다.

3. **구축 방식의 변화**:
    - **과거 (Waterfall 중심)**: 철저한 사전 설계와 대규모 구축 기간 (1~2년 소요).
    - **현재 (Agile & Cloud 중심)**: 표준 기능을 먼저 쓰고 나중에 기능을 붙이는 'Fit-to-Standard' 방식의 빠른 도입 (3~6개월).

📢 섹션 요약 비유: ERP 구축 생명주기는 설계도(분석)를 보고 기성 부품(패키지)을 조립하며, 우리 집 구조에 안 맞는 부분만 깎아내어(CBO) 가장 튼튼한 집(통합 시스템)을 짓는 공정표입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

ERP 구축의 표준 5단계 프로세스와 각 단계별 핵심 활동을 분석한다.

### 1. ERP 구축 5단계 로드맵 (ASCII)

```text
    [ 1. 계획/선정 ] ──> [ 2. 분석 (Analyze) ] ──> [ 3. 설계 (Design) ]
           │                    │                      │
    - 전략 및 범위 수립  - AS-IS 업무 분석      - TO-BE 프로세스 확정
    - 패키지 선정 (RFP)  - **Gap 분석**         - **커스터마이징 범위**
                                │                      │
                                ▼                      ▼
    [ 5. 구현 (Realize) ] <── [ 4. 구축 (Build) ] <────┘
           │                    │
    - 데이터 이관 (Mig.) - **CBO 개발**
    - 통합 테스트        - 사용자 교육 / 매뉴얼
    - Cut-over (오픈)    - 시스템 환경 설정
```

**[다이어그램 해설]**  
- **계획/선정**: 우리 회사 규모와 업종에 맞는 솔루션(SAP, Oracle 등)을 고르는 '중요한 첫 단추'다.
- **분석/설계**: 현행 업무(As-Is)와 시스템 기능 간의 격차(Gap)를 찾아내어 TO-BE를 그린다.
- **구축**: 부족한 기능을 채우기 위해 코딩(CBO)을 하거나 시스템 옵션을 설정(Config)한다.
- **구현**: 과거 데이터를 새 시스템으로 옮기고(Migration), 사고 없이 서비스를 켠다(Cut-over).

### 2. 단계별 상세 활동 및 핵심 산출물

| 단계 | 핵심 활동 | 주요 산출물 | 기술사적 포인트 |
|:---|:---|:---|:---|
| **분석** | 인터뷰, 현행 프로세스 맵핑 | AS-IS 정의서, **Gap 목록** | 업무 누락 방지 |
| **설계** | **Fit-to-Standard** 분석 | TO-BE 설계서, CBO 정의서 | **BPR 선행** 여부 결정 |
| **구축** | 패키지 설정(Config), 개발 | 시스템 환경설정서, 프로그램 소스 | 모듈 간 데이터 연동(I/F) |
| **구현** | **데이터 클렌징**, 마이그레이션 | 전환 결과 리포트, 사용자 매뉴얼 | 데이터 정합성 확보 |

### 3. ERP 구축의 결정적 순간: Gap 분석

- **Fit**: 패키지 기능이 업무와 일치함. (그대로 사용)
- **Gap**: 패키지 기능과 업무가 불일치함.
  - **해결 A (Process change)**: 시스템에 맞춰 업무를 바꿈 (BPR 권장).
  - **해결 B (Customizing)**: 업무에 맞춰 시스템을 고침 (CBO/Add-on).

📢 섹션 요약 비유: 분석 단계가 '신체 검사'라면, 설계 단계는 '옷 수선 계획'이고, 구축 단계는 '실제 바느질'입니다. 수선(커스터마이징)을 너무 많이 하면 옷의 원래 맵시(안정성)가 망가집니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

SI 개발과 ERP 구축의 차이 및 최근의 데이터 전환 트렌드다.

### 1. 일반 SI 개발 vs ERP 구축 생명주기 비교

| 구분 | 일반 SI (In-house 개발) | ERP 구축 (Package 기반) |
|:---|:---|:---|
| **중심 활동** | **코딩 및 개발** (Build) | **선정 및 설정** (Buy & Config) |
| **요구사항** | 사용자가 원하는 대로 | **시스템의 표준** 프로세스 수용 |
| **설계 기준** | 업무 그 자체 (Logic) | 패키지와의 격차 (Gap) |
| **리스크** | 성능 및 품질 리스크 | 업무 적응 및 조직 저항 리스크 |
| **비유** | 맞춤 정장 제작 | 기성 브랜드복 수선 |

### 2. 과목 융합: 데이터 마이그레이션(Migration)의 중요성

ERP 구축의 마지막 성공 관문은 데이터다.
- **ETL 프로세스**: Extract(추출) -> Transform(변환) -> Load(적재).
- **데이터 클렌징**: 낡은 시스템의 엉터리 주소나 중복 고객 정보를 정제하는 작업.
- **정합성 검증**: 이관 후 재무 제표 숫자가 일원 단위까지 맞는지 검증.

📢 섹션 요약 비유: SI 개발이 '백지에 그림 그리기'라면, ERP 구축은 '완성된 색칠 공부'에서 우리 회사 색깔(데이터)만 입히는 과정입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 ERP 생명주기를 관리할 때의 핵심 의사결정 포인트다.

### 1. CBO (Custom Built Object) 최소화 전략 (Decision)

기술사적 관점에서 **CBO가 많을수록 프로젝트는 실패**에 가까워진다.
- **문제**: 소스 코드를 많이 고치면 나중에 패키지 업데이트를 못 하고, 시스템은 무거워지며 버그가 늘어난다.
- **전략**: **'Zero Customizing'** 지향.
- **Action**: Gap이 발생하면 우선적으로 **현업의 업무를 시스템 표준에 맞추도록(BPR)** 설득한다. 정 안되는 경우에만 승인 절차를 거쳐 CBO 개발을 허용한다.

### 2. 성공적인 Cut-over를 위한 체크리스트

| 점검 항목 | 확인 질문 | 기술사적 제언 |
|:---|:---|:---|
| **마스터 데이터** | 고객, 상품, 자산 코드가 전사 단일화되었는가? | MDM 체계 선행 확인 |
| **통합 테스트** | 전 모듈을 엮어 한 사이클(End-to-End)을 돌려봤는가? | 실제 업무 시나리오 기반의 스트레스 테스트 |
| **사용자 숙련도** | 현업이 도움 없이 스스로 전표를 칠 수 있는가? | 부서별 '수퍼 유저(Super User)' 밀착 교육 |
| **비상 복구** | 데이터 전환 실패 시 구 시스템으로 돌아갈 수 있는가? | 상세한 Roll-back 매뉴얼 및 백업 확보 |

### 3. 안티패턴
- **"Package Selection by Name"**: 기능 분석 없이 브랜드 이름만 보고 1등 솔루션 선택.
- **"Skipping Unit Test"**: 일정이 바쁘다고 개별 모듈 테스트를 건너뛰고 통합 테스트로 바로 가는 경우.

📢 섹션 요약 비유: CBO를 만드는 것은 기성복에 화려한 단추(특화 기능)를 다는 것과 같습니다. 적당하면 멋지지만 너무 많으면 옷이 무거워져 입기 힘들어집니다.

+++

## Ⅴ. 기대효과 및 결론 (Future & Standard)

ERP 구축 생명주기는 단순한 시스템 구축을 넘어 기업의 체질을 바꾸는 과정이다.

| 기대효과 | 설명 |
|:---|:---|
| **투명 경영 실현** | 표준화된 생애주기를 거치며 흩어진 데이터를 하나로 통합 |
| **운영 효율성 향상** | 글로벌 베스트 프랙티스 도입을 통한 업무 리드타임 단축 |
| **유지보수 용이성** | 표준 패키지 구조 유지를 통한 향후 신기술 업그레이드 경로 확보 |

미래의 ERP 구축 생명주기는 **"AI-driven Automated Lifecycle"**로 진화할 것이다. 사람이 일일이 Gap 분석을 하는 대신, AI가 현업의 업무 로그를 분석하여 최적의 패키지 설정값을 자동으로 추천하고, 데이터 마이닝을 통해 이관 데이터의 오류를 스스로 정정하는 '자율 주행형 구축' 시대가 올 것이다. 이제 ERP 구축은 '인고의 시간'이 아닌, 지능형 플랫폼이 제공하는 **'신속한 가치 경험'**으로 변모하고 있다.

📢 섹션 요약 비유: 미래의 ERP 구축은 내 몸의 치수를 스캔하면 딱 맞는 옷을 3D 프린터로 즉시 뽑아주는 '스마트 맞춤 서비스'처럼 빠르고 완벽해질 것입니다.

### 📌 관련 개념 맵
| 관련 개념 | 관계 및 시너지 설명 |
|:---|:---|
| BPR (Business Process Reengineering) | ERP 구축 전이나 도중에 수행하는 업무 근본 재설계 활동 |
| Gap Analysis | 현행 업무와 패키지 표준 기능 사이의 차이를 분석하는 핵심 활동 |
| CBO (Custom Built Object) | 패키지에 없는 기능을 보완하기 위해 자체 개발하는 프로그램 |
| Cut-over | 과거 시스템을 중단하고 새 ERP로 전환하는 결정적 순간 |
| Fit-to-Standard | 업무를 시스템에 맞추는 클라우드 시대의 새로운 구축 전략 |

### 👶 어린이를 위한 3줄 비유 설명
1. 우리 집의 모든 물건을 똑똑한 새 상자에 옮겨 담는 커다란 이사와 같아요.
2. 먼저 어떤 상자가 좋을지 고르고(선정), 낡은 물건은 버리고(클렌징), 새 상자에 맞게 물건을 잘 정리하는 과정이죠.
3. 이 이사가 끝나면 물건을 훨씬 빨리 찾을 수 있고 집이 아주 깨끗해진답니다!
