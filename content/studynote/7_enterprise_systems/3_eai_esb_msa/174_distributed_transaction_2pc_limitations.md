# 분산 트랜잭션 한계 및 2PC 배제 이유

> 분산 시스템에서 ACID 트랜잭션을 보장하기 위한 2PC (Two-Phase Commit) 프로토콜의 구조적 한계와 블로킹 오버헤드를 분석하고, 마이크로서비스 아키텍처(MSA)에서 왜 2PC를 배제하고 사가 패턴, 이벤트 소싱, CQRS 등의 대안을 선택하는지 기술적 근거를 제시

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 트랜잭션은 물리적으로 분리된 다수 데이터베이스에 걸친 단일 논리적 트랜잭션을 ACID(Atomicity, Consistency, Isolation, Durability) 원칙으로 원자적으로 실행하는 것으로, 2PC(2-Phase Commit)는 이를 보장하는 표준 프로토콜이지만 구조적 블로킹과 단일 장애점(SPOF) 문제를 가진다.
> 2. **가치**: 2PC는 준비(Prepare) 단계에서 모든 참여자의 잠금(Lock)과 동시 확보가 필요하므로, 네트워크 지연이 긴 클라우드 환경에서는 심각한 성능 저하를 유발하고, 하나의 참여자 장애로 전체 트랜잭션이 롤백되어 가용성이 급격히 떨어진다.
> 3. **융합**: CAP 정리(Consistency, Availability, Partition tolerance)와 PACELC 모델에 따라 분산 환경에서는 CP(일관성+분할 내성) 또는 AP(가용성+분할 내성)를 선택해야 하며, 2PC는 CP를 선택하지만 마이크로서비스의 높은 가용성 요구와 충돌한다. 따라서 결과적 일관성(Eventual Consistency)을 허용하는 사가 패턴, 이벤트 소싱, CQRS 등이 대안으로 부상한다.

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**분산 트랜잭션 (Distributed Transaction)**은 물리적으로 분리된 다수의 데이터베이스나 서버에 걸친 작업을 하나의 논리적 단위로 묶어, 모든 작업이 전부 성공하거나 전부 실패(All-or-Nothing)하는 ACID 원칙을 보장하는 기술이다. 단일 데이터베이스 내에서는 DBMS가 트랜잭션 관리를 담당하지만, 분산 환경에서는 여러 독립적인 리소스 매니저가 조정된 프로토콜을 통해 원자성을 보장해야 한다.

**2PC (Two-Phase Commit, 2단계 커밋)**는 분산 트랜잭션을 보장하는 가장 대표적인 프로토콜로, (1) 준비(Prepare) 단계에서 모든 참여자(Participant)가 커밋 가능한지 묻고 모두가 "Yes"하면 (2) 커밋(Commit) 단계에서 실제 커밋을 명령하는 방식이다. 하나라도 "No"하거나 타임아웃이 발생하면 전체를 롤백(Rollback)한다.

### 💡 비유

분산 트랜잭션과 2PC는 **"여러 나라에 사는 친구들이 동시에 여행을 가기로 약속"**하는 상황과 같다.

- **단일 DB 트랜잭션**: 혼자 여행 가면 결정만 하면 됨 (쉬움)
- **분산 트랜잭션**: A(한국), B(일본), C(미국)에 사는 친구들이 동시에 같은 날짜에 만나기로 약속
- **2PC Prepare 단계**: "너희들 그날 가능해?"라고 일일이 물어봄 (모두 OK 해야 진행)
- **2PC Commit 단계**: "그럼 다들 표사!" (비행기/호텔 예약)
- **문제점**: C가 전화를 안 받거나(네트워크 장애), 갑자기 일정이 생겨(장애) → 전체 약속 취소 (블로킹)

### 등장 배경

1. **단일 DB 시대의 ACID 보장**: 1970~80년대 OLTP(Online Transaction Processing) 시스템에서는 단일 RDBMS 내에서의 트랜잭션 ACID가 DBMS 엔진이 자동 보장했다. BEGIN~COMMIT 사이의 SQL 문장이 원자적으로 실행되며, 시스템 충돌 시에는 롤백 세그먼트(Rollback Segment)를 통해 복구가 가능했다.

2. **분산 시스템의 부상과 2PC 등장 (1980년대)**: IBM CICS, Tuxedo 같은 TP Monitor (Transaction Processing Monitor)가 등장하면서, 여러 데이터베이스를 연동하는 분산 트랜잭션의 필요성이 대두되었다. 2PC 프로토콜이 X/Open XA 표준으로 정의되어, 오라클(Oracle), DB2, Sybase 등 상용 DBMS가 표준 인터페이스를 제공했다.

3. **모놀리식 앱 시대의 2PC 사용 (1990~2000년대)**: 초기 엔터프라이즈 애플리케이션은 단일 서버(Application Server)가 EAI(Enterprise Application Integration) 허브(Hub) 또는 ESB(Enterprise Service Bus)를 통해 여러 DB에 접속하면서, 2PC 기반 분산 트랜잭션을 빈번하게 사용했다. JTA(Java Transaction API), JTS(Java Transaction Service), .NET System.Transactions 등이 2PC를 지원했다.

4. **MSA와 클라우드 시대의 2PC 한계 노출 (2010년대~)**: 마이크로서비스 아키텍처(MSA)가 도입되면서 데이터베이스 퍼 서비스(Database per Service) 설계가 표준이 되었고, AWS RDS, Google Cloud SQL 같은 클라우드 DB가 보편화되었다. 클라우드 환경에서는 네트워크 지연(RTT)이 온프레미스보다 길고(50~200ms), 장애 발생 빈도가 높아 2PC의 블로킹 특성이 치명적인 병목이 되었다.

5. **결과적 일관성(Eventual Consistency)의 수용 (2010년대 후반~)**: 구글, 아마존, 넷플릭스 등 초대형 플랫폼은 CAP 정리의 AP(가용성+분할 내성)를 선택하여, 데이터가 즉시 일관되지 않아도(Eventual Consistency) 최종적으로 일치하면 된다는 철학으로 전환했다. 이에 따라 2PC 대신 사가 패턴(Saga Pattern), 이벤트 소싱(Event Sourcing), CQRS(Command Query Responsibility Segregation) 등이 대안으로 부상했다.

```text
┌─────────────────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────────────────┤

│    │                          │                          │                  │
```

**[다이어그램 해설]** 분산 트랜잭션 처리 방식은 2PC 기반 강한 일관성에서 MSA/클라우드 시대의 결과적 일관성으로 진화했다. 1980년대에는 IBM CICS, Tuxedo 같은 TP Monitor가 2PC 프로토콜을 구현하여 여러 DB를 연동하는 분산 트랜잭션을 지원했다. 2000년대 모놀리식 앱 시대에는 Application Server가 EAI Hub나 ESB를 통해 여러 DB에 접속하며 2PC를 표준으로 사용했다. 하지만 2010년대 이후 MSA와 클라우드가 도입되면서 2PC의 한계이 명확해졌다. 첫째, 클라우드 환경에서는 네트워크 지연이 길어(50~200ms) 2PC의 2단계 커밋에 따른 지연이 누적되어 성능이 급격히 저하된다. 둘째, 데이터베이스 퍼 서비스 설계로 각 서비스가 독립적인 DB를 가지게 되면서, 하나의 DB 장애가 전체 트랜잭션을 실패로 만드는 단일 장애점(SPOF) 문제가 심각해졌다. 셋째, MSA의 확장성 요구와 2PC의 잠금(Lock) 기반 동시 실행 보장은 상충관계에 있었다. 따라서 구글, 아마존 등은 결과적 일관성(Eventual Consistency)을 수용하고, 사가 패턴, 이벤트 소싱, CQRS 등의 대안을 도입하여 높은 가용성과 확장성을 확보했다.

### 📢 섹션 요약 비유

마치 여러 친구가 동시에 만나기로 약속할 때, 한 명이 안 오면 아무도 못 가는 "전체 원칙(All-or-Nothing)" 대신, 각자 출발했다가 목적지에서 모이는 "분산 합의(Consensus)" 방식으로 바꾸는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 2PC 프로토콜 상세 구조

2PC는 코디네이터(Coordinator)와 참여자(Participant) 간의 2단계 합의(Commit Protocol)를 통해 원자성을 보장한다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                    2PC (Two-Phase Commit) 프로토콜                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  [코디네이터]                 [참여자 1]         [참여자 2]           │
│      (Coordinator)              (Participant)       (Participant)        │
│                                                                         │
│    트랜잭션 시작                  │                   │                  │
│         │                       │                   │                  │
│         ▼                       ▼                   ▼                  │
│  ┌─────────────────┐         ┌──────────┐      ┌──────────┐        │
│  │  1단계: Prepare  │         │ DB Lock   │      │ DB Lock   │        │
│  │   Can Commit?  │         │ 자원 잠금 │      │ 자원 잠금 │        │
│  └────────┬────────┘         └────┬─────┘      └────┬─────┘        │
│           │                      │                  │                 │
│           │                   Yes                Yes                 │
│           └──────────┬───────────┴──────────────────┘                 │
│                      │                                             │
│         모두 Yes?                                                │
│                      │                                             │
│           ┌──────────┴──────────┐                                 │
│           ▼                       ▼                                 │
│   ┌─────────────┐          ┌─────────────┐                       │
│   │ 2단계: Commit│          │ 2단계: Rollback│                    │
│   │   Do Commit! │          │  Do Rollback!  │                    │
│   └─────────────┘          └─────────────┘                       │
│           │                       │                                 │
│           ▼                       ▼                                 │
│    ┌─────────┐              ┌─────────┐                           │
│    │ 트랜잭션 │              │ 트랜잭션 │                           │
│    │  커밋   │              │  롤백   │                           │
│    │  완료   │              │  완료   │                           │
│    └─────────┘              └─────────┘                           │
│                                                                         │
│  ⚠️ 블로킹 구간: Prepare 시작 ~ Commit/Rollback 완료까지             │
│     (DB Lock 해지 전까지 다른 트랜잭션 대기 불가)                       │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 2PC의 블로킹 구조를 명확히 보여준다. (1) Prepare 단계에서 코디네이터는 모든 참여자에게 "Can Commit?" 메시지를 보내고, 각 참여자는 DB Lock을 획득하고 트랜잭션 로그를 기록한 후 "Yes"를 응답한다. 이 시점부터 Commit/Rollback 완료까지 DB가 잠겨 있어 다른 트랜잭션은 대기해야 한다. (2) 모든 참여자가 "Yes"하면 코디네이터는 "Commit" 명령을 보내고, 하나라도 "No" 또는 타임아웃이 발생하면 "Rollback" 명령을 보낸다. (3) 참여자는 명령을 수행하고 Lock을 해제한다. 핵심 문제는 Prepare 단계에서 모든 참여자가 동시에 Lock을 획득해야 하므로, 병목(Concurrency)이 심각하고, 하나의 참여자 장애로 전체 트랜잭션이 실패(SPOF)한다는 점이다. 클라우드 환경에서는 네트워크 왕복(RTT)이 길어 Prepare→Commit 간의 대기 시간이 길어, 다른 트랜잭션이 기다리는 큐(Queue)가 길어질 수밖에 없다.

### 2PC의 구조적 한계

| 한계 | 설명 | 영향 |
|:---|:---|:---|
| **블로킹 오버헤드** | Prepare 단계에서 모든 참여자가 Lock을 획득할 때까지 다른 트랜잭션 대기 | 처리량(Throughput) 급격히 저하 |
| **단일 장애점(SPOF)** | 코디네이터 장애 시 전체 트랜잭션 불가 | 가용성(Availability) 저하 |
| **잠금 경합(Lock Contention)** | 동일 자원에 대한 트랜잭션이 많으면 Lock 대기 시간 급증 | 성능 저하, 데드락(Deadlock) 위험 |
| **네트워크 지연 민감성** | Prepare→Commit 사이의 RTT가 2PC 총 지연에 직접 더해짐 | 클라우드 환경(50~200ms RTT)에서 치명적 |
| **롤백 비용** | Prepare 후 Rollback 시 이미 수행한 작업 복구 비용 | 분산 환경에서는 부분 성공 후 Rollback 복잡 |

### CAP 정리와 2PC

CAP 정리는 분산 시스템에서 일관성(Consistency, C), 가용성(Availability, A), 분할 내성(Partition Tolerance, P) 3가지를 동시에 만족할 수 없고, 최대 2가지만 선택 가능하다는 정리다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                    CAP 정리와 트랜잭션 전략                             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                          │
│  │   C     │    │   A     │    │   P     │                          │
│  │일관성  │    │가용성  │    │분할내성│                          │
│  └────┬────┘    └────┬────┘    └────┬────┘                          │
│       │              │              │                              │
│       └──────┬───────┘              │                              │
│              │                      │                              │
│              ▼                      │                              │
│      [분산 환경에서는 P 필수]      │                              │
│      (네트워크 분단은 불가피)     │                              │
│              │                      │                              │
│              └──────┬───────┬───────┘                              │
│                             │                                      │
│              ┌────────────────┴──────┐                             │
│              ▼                       ▼                             │
│         [CP 선택]                 [AP 선택]                         │
│   ┌───────────────┐           ┌───────────────┐                      │
│   │일관성+분할내성 │           │가용성+분할내성 │                    │
│   │               │           │               │                    │
│   │2PC, RDBMS      │           │NoSQL, Eventual│                   │
│   │강한 일관성    │           │Consistency    │                   │
│   │Write 불가능   │           │Write 가능     │                   │
│   │장애 시 중단   │           │장애 시 서비스 │                    │
│   └───────────────┘           └───────────────┘                      │
│                                                                         │
│  2PC의 위치: CP 영역 (일관성 포기 불가 → 가용성 희생)               │
│  MSA의 요구: AP 영역 (가용성 최우선 → 일관성 타협)                    │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CAP 정리 상 2PC는 명확히 CP(일관성+분할 내성) 전략을 선택한다. 2PC는 분산 환경에서도 데이터의 일관성을 ACID 수준으로 강하게 보장하려 하므로, 가용성(Availability)을 희생한다. 네트워크 분단(Partition) 발생 시, 2PC는 일관성을 지키기 위해 전체 시스템을 중단시키거나 트랜잭션을 거부한다. 반면 MSA(마이크로서비스 아키텍처)는 높은 가용성과 확장성이 핵심이므로 AP(가용성+분할 내성) 전략을 선택한다. AP 전략에서는 네트워크 분단 시에도 서비스를 계속 제공(Available)하고, 쓰기(Write)는 계속 가능하며 데이터는 결과적으로 일관성(Eventual Consistency)을 가진다. 이러한 철학적 차이로 인해 2PC는 MSA 환경과 본질적으로 상충돌며, 결과적 일관성을 허용하는 사가 패턴, 이벤트 소싱, CQRS 등이 대안으로 부상하게 된다.

### 📢 섹션 요약 비유

마치 "혼자 결정하면 빠르지만, 여럿이 결정하면 천천히 합의해야 하고 한 명이라도 반대하면 전체 무산"인 위원회와 같아서, 2PC는 분산 환경에서 합의 비용이 너무 높아 실용하기 어렵습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 강한 일관성 vs 결과적 일관성

| 비교 항목 | 강한 일관성 (Strong Consistency) | 결과적 일관성 (Eventual Consistency) |
|:---|:---|:---|
| **정의** | 트랜잭션 직후 즉시 데이터 일관 | 시간 경과 후 최종적으로 데이터 일관 |
| **보장 방식** | 2PC, 분산 락(Distributed Lock) | 사가 패턴, 이벤트 소싱, CQRS |
| **가용성** | 낮음 (장애 시 서비스 중단) | 높음 (장애 시에도 서비스 가능) |
| **성능** | 낮음 (Lock 경합, 대기 시간) | 높음 (비동기 처리, 큐잉) |
| **복잡도** | 낮음 (DBMS가 자동 처리) | 높음 (애플리케이션 로직 구현) |
| **사례** | 뱅킹 이체, 재고 (정확성 중요) | SNS 댓글, 장바구니 (가용성 중요) |
| **CAP 위치** | CP (일관성+분할 내성) | AP (가용성+분할 내성) |
| **지연 허용** | milisecond(ms) 수준 | second~minute 수준 |

강한 일관성은 데이터의 정확성이 중요하지만 가용성과 성능을 희생하고, 결과적 일관성은 가용성과 성능을 높이지만 데이터가 일시적으로 불일치할 수 있다. MSA에서는 비즈니스 요구에 따라 두 방식을 선택적으로 적용한다. 예를 들어 "재고"는 강한 일관성을, "좋아요 누르기"는 결과적 일관성을 적용할 수 있다.

### 비교 2: 2PC vs 사가 패턴 (Saga Pattern)

| 비교 항목 | 2PC (Two-Phase Commit) | 사가 패턴 (Saga Pattern) |
|:---|:---|:---|
| **동기 방식** | 동기(Synchronous): Blocking | 비동기(Asynchronous): Non-Blocking |
| **트랜잭션 단위** | 전체를 하나의 원자적 단위로 관리 | 각 서비스의 로컬 트랜잭션으로 분리 |
| **실패 복구** | Rollback (전체 취소) | 보상 트랜잭션(Compensating Transaction) |
| **데이터 일관성** | 강한 일관성(ACID) | 결과적 일관성(Eventual Consistency) |
| **성능** | 낮음 (Lock 대기) | 높음 (비동기 병렬 처리) |
| **복잡도** | 낮음 (DBMS가 지원) | 높음 (앱 로직으로 보상 트랜잭션 구현) |
| **가용성** | 낮음 (SPOF 문제) | 높음 (서비스별 독립 장애 격리) |
| **MSA 적합성** | 낮음 (확장성 저하) | 높음 (서비스 독립 배포 가능) |

사가 패턴은 긴 트랜잭션을 여러 로컬 트랜잭션으로 분리하고, 각 단계가 실패하면 역순으로 보상 트랜잭션을 실행하여 롤백 효과를 내는 방식이다. 예를 들어 "주문 → 배송 → 결제" 트랜잭션에서, (1) 주문 성공 → (2) 배송 실패 → (3) 결제 취소(보상) → (4) 주문 취소(보상)와 같이 역순으로 보상한다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                사가 패턴 (Saga Pattern) 구조                             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  [비즈니스 트랜잭션]: 주문 → 배송 → 결제                                  │
│                                                                         │
│  ┌─────────┐      ┌─────────┐      ┌─────────┐                      │
│  │ 주문    │─────▶│ 배송    │─────▶│ 결제    │                      │
│  │ Service │      │ Service │      │ Service │                      │
│  └────┬────┘      └────┬────┘      └────┬────┘                      │
│       │                │                │                            │
│       ▼                ▼                ▼                            │
│  ┌─────────┐      ┌─────────┐      ┌─────────┐                      │
│  │로컬 트랜 │      │로컬 트랜 │      │로컬 트랜 │                      │
│  │잭션 Commit│    │잭션 Commit│    │잭션 Commit│                      │
│  └────┬────┘      └────┬────┘      └────┬────┘                      │
│       │                │                │                            │
│       ▼                ▼                ▼                            │
│   [성공]            [실패]             [보상 필요]                       │
│       │                │                │                            │
│       │                └────────────────┘                            │
│       │                         │                                   │
│       │                ┌─────────────┐                               │
│       │                │ 보상 트랜잭션│                               │
│       │                │  (역순 실행)   │                               │
│       │                │ 결제 취소────▶│ 주문 취소                     │
│       │                └─────────────┘                               │
│       │                                                            │
│       ▼                                                            │
│  [최종 상태]: 롤백 완료 (주문→배송→결제 모두 취소)                      │
│                                                                         │
│  장점:                                                               │
│  • 비동기 병렬 처리로 성능 향상                                      │
│  • 서비스별 독립 배포로 가용성 향상                                  │
│  • 로컬 트랜잭션만 관리하므로 Lock 경합 감소                             │
│                                                                         │
│  단점:                                                               │
│  • 중간 상태에서 데이터 불일치 발생 (결제는 완료, 배송은 실패 등)        │
│  • 보상 트랜잭션 로직을 앱 개발자가 구현해야 함 (복잡도 증가)             │
│  • 역순 실행 시 재시도(Reentrancy) 문제 발생 가능                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 사가 패턴은 긴 트랜잭션을 여러 로컬 트랜잭션으로 분리하고, 각 단계가 실패하면 역순으로 보상 트랜잭션을 실행한다. 예를 들어 "주문 → 배송 → 결제" 트랜잭션에서, (1) 주문 서비스가 주문 로컬 트랜잭션을 커밋하고 → (2) 배송 서비스가 배송 로컬 트랜잭션을 시도하는데 실패하면 → (3) 결제 취소(보상) → (4) 주문 취소(보상)와 같이 역순으로 보상 트랜잭션을 실행한다. 핵심 장점은 각 서비스가 자신의 로컬 트랜잭션만 관리하므로 Lock 경합이 적고 비동기 병렬 처리가 가능하여 성능과 가용성이 높다는 점이다. 단점은 중간 상태에서 데이터 불일치가 발생할 수 있다. 예를 들어 배송이 실패했는데 결제는 이미 완료된 상태에서 보상 트랜잭션이 실행되기 전까지는 "결제는 완료, 배송은 실패"한 상태로 데이터가 불일치한다. 따라서 사가 패턴은 결과적 일관성(Eventual Consistency)을 허용하는 비즈니스 로직에만 적용 가능하다.

### 과목 융합 관점

- **네트워크 (NW)**: 2PC의 성능은 RTT(Round Trip Time)에 민감하다. 온프레미스에서 RTT는 1ms 이내이지만, 클라우드(AWS 리전 간)에서는 50~200ms로 100배 이상 차이 난다. 2PC는 Prepare→Commit 사이에 2회의 왕복이 필요하므로, 클라우드에서는 단순히 네트워크 지연만으로도 100~400ms가 소요되어 실시간성 요구에 부합하기 어렵다.

- **운영체제 (OS)**: 2PC는 DB Lock이 해제될 때까지 다른 트랜잭션이 대기해야 하므로, OS 커널의 스레드 풀(Thread Pool)이 고갈될 수 있다. 반면 사가 패턴은 비동기 메시지 큐(Kafka, RabbitMQ)를 활용하여 트랜잭션 간 의존성을 느슨하게 결합할 수 있어, OS 리소스를 효율적으로 사용할 수 있다.

- **데이터베이스 (DB)**: 2PC는 DBMS가 지원하는 표준 프로토콜(XA, JTA)이므로 구현이 쉽지만, 사가 패턴은 애플리케이션 레벨에서 보상 트랜잭션 로직을 구현해야 하므로 복잡도가 증가한다. 또한 2PC는 DB Lock 경합이 발생하지만, 사가 패턴은 로컬 트랜잭션만 잠그므로 경합이 적다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│              2PC vs 사가 패턴: 네트워크 지연 영향 비교                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  [온프레미스 환경: RTT 1ms]                                         │
│                                                                         │
│  2PC: Prepare(1ms 왕복) ──┬─▶ Commit(1ms 왕복) ──┬─▶ 완료            │
│      ────────────────┘    ────────────────┘    = 2ms                │
│                                                                         │
│  사가: 트랜잭션 1(1ms) ──▶ 트랜잭션 2(1ms) ──▶ 트랜잭션 3(1ms)         │
│      = 3ms (순차 실행)                                                │
│                                                                         │
│  ────────────────────────────────────────────────────────────────────   │
│                                                                         │
│  [클라우드 환경: RTT 100ms]                                           │
│                                                                         │
│  2PC: Prepare(100ms 왕복) ──┬─▶ Commit(100ms 왕복) ──┬─▶ 완료          │
│      ──────────────────┘    ──────────────────┘    = 200ms            │
│      ⚠️ 여기서 Lock이 걸려 다른 트랜잭션 대기                            │
│                                                                         │
│  사가: 트랜잭션 1(1ms) ──▶ 트랜잭션 2(1ms) ──▶ 트랜잭션 3(1ms)          │
│      + 메시지 큐 지연(10ms)                                         │
│      = 13ms (비동기 병렬 처리 가능)                                 │
│                                                                         │
│  ────────────────────────────────────────────────────────────────────   │
│                                                                         │
│  결론: 클라우드 환경에서는 사가 패턴의 비동기 처리가 압도적으로 유리   │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 온프레미스(RTT 1ms)와 클라우드(RTT 100ms) 환경에서 2PC와 사가 패턴의 성능 차이를 보여준다. 온프레미스에서는 2PC가 2ms, 사가가 3ms로 큰 차이가 없지만, 클라우드에서는 2PC가 200ms, 사가가 13ms로 15배 이상의 차이가 난다. 더 중요한 것은 2PC는 Prepare~Commit 사이에 DB Lock이 걸려 다른 트랜잭션이 대기해야 하므로, 시스템 전체의 처리량(Throughput)이 급격히 저하된다는 점이다. 반면 사가 패턴은 비동기 메시지 큐를 활용하여 트랜잭션을 병렬 처리할 수 있으므로, 클라우드 환경의 높은 지연에도 불구하고 높은 처리량을 유지할 수 있다. 이것이 MSA와 클라우드 시대에 2PC가 배제되고 사가 패턴이 대안으로 부상하는 결정적인 이유다.

### 📢 섹션 요약 비유

마치 여러 사람이 동시에 화장실을 쓰려면 대기 시간이 길어지는 것처럼, 2PC는 DB Lock을 통해 동시성을 보장하느라 대기 시간이 길어 병목이 발생합니다. 반면 사가 패턴은 각자 이용 시간을 다르게 하여(Membership scheduling), 대기 시간을 줄이는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 전자상거래 주문/결제 분산 트랜잭션**: 전자상거레 플랫폼에서 "주문 서비스"와 "결제 서비스"가 분리된 MSA 환경에서 2PC 도입 시도. (결과) Prepare 단계에서 결제 PG(Payment Gateway) Lock 획득 대기 시간이 평균 300ms로 증가하고, 배송 서비스 장애 시 전체 주문이 롤백되어 주문 취소율이 15%에서 8%로 급감함을 경험. 사가 패턴으로 전환 후 결제 실패 시 "주문 취소 → 재고" 보상 로직을 구현하여 가용성 99.9% 달성.

2. **시나리오 — 항공사 발권 서비스 분산 트랜잭션**: 항공사 발권 시 "항공편 예약"과 "결제"가 여러 항공사 API를 거쳐야 하는 복잡한 분산 환경. 2PC 도입 시 외부 항공사 API 장애 시 전체 발권이 불가능해져 고객 불만이 급증. 사가 패턴 + 메시지 큐(Kafka) + 재시도(Retry) 메커니즘으로 전환하여 외부 API 장애 시에도 큐에 메시지를 쌓아두고 장애 복구 후 자동 재실행으로 가용성 개선.

3. **시나리오 — 금융 기관 계좄 이체 분산 트랜잭션**: A은행에서 B은행으로 계좄 이체 시 2PC 기반 분산 트랜잭션 사용 중 네트워크 장애로 인해 트랜잭션이 중단된 상태에서 롤백되지 않고 "보류 중(Held)" 상태로 남아 잠금이 풀리지 않는 사례 발생. 이후 사가 패턴과 "트랜잭셔 아웃박스 패턴(Transaction Outbox)"을 도입하여, 로컬 DB의 Outbox 테이블에 이벤트를 기록하고 메시지 브로커(Kafka)를 통해 비동기로 전송하는 구조로 전환하여 원자성을 보장.

### 도입 체크리스트

- **기술적**:
  - 강한 일관성이 필요한지 결과적 일관성으로 충분한지 비즈니스 요구사항을 먼저 확인할 것
  - 2PC 사용 시 RTT 10ms 이내의 로컬 네트워크로 제한할 것
  - 사가 패턴 사용 시 보상 트랜잭션 로직의 재시도(Reentrancy) 문제를 방지할 것
  - 트랜잭셔 아웃박스(Outbox) 패턴으로 로컬 트랜잭션과 이벤트 발행의 원자성을 보장할 것

- **운영·보안적**:
  - 분산 트랜잭션의 중간 상태를 모니터링하고, 불일치 상태(Data Inconsistency)를 탐지할 수 있는 대시보드를 구축할 것
  - 실패한 트랜잭션의 재시(Retry) 정책과 최종 실패 시 알림(Alert) 메커니즘을 구현할 것
  - 보상 트랜잭션의 IDEMPOTENCE(멱등 방지) 처리를 보장할 것
  - 트랜잭션 로그를 감사(Audit) 추적에 활용할 것

### 안티패턴

- **MSA에서 2PC 사용**: MSA의 핵심 목표인 서비스 독립 배포와 확장성이 2PC의 Lock 기반 동시 실행 보장과 상충되므로, 가용성과 확장성이 급격히 저하된다.

- **보상 트랜잭션 미구현**: 사가 패턴 도입 시 보상 트랜잭션 로직을 누락하면 데이터 불일치가 발생하여 심각한 비즈니스 손실을 야기할 수 있다.

- **트랜잭션 아웃박스 미적용**: 로컬 트랜잭션 커밋과 메시지 브로커 이벤트 발행 사이의 원자성을 보장하지 않으면, 메시지 발행 실패 시 로컬 트랜잭션만 커밋되어 데이터 불일치가 발생한다.

- **무한 재시(Retry) 정책**: 실패한 트랜잭션을 무한정 재시하면 시스템 리소스가 고갈되어 죽음의 스파럴(Death Spiral)로 이어질 수 있다.

```text
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  [문제: 로컬 트랜잭션과 메시지 발행 사이의 원자성 부재]                  │
│                                                                         │
│  ┌─────────┐                                                         │
│  │Service  │                                                         │
│  └────┬────┘                                                         │
│       │                                                             │
│       ├──────────────────────────────────────────────────────┐        │
│       │                                                              │
│       ▼                                                              │
│  ┌─────────────┐     ┌───────────────┐                              │
│  │ 로컬 DB     │     │ 메시지 브로커 │                              │
│  │ TRX Commit │     │ (Kafka)      │                              │

│       ▼                                                             │
│  ┌─────────────┐     ┌───────────────┐                              │
│  │ 로컬 DB     │     │ 메시지 브로커 │                              │
│  │ ┌─────────┐│     └──────┬───────┘                              │
│  │ │Outbox  ││            │                                       │

│  │ └────┬───┘│     ┌──────────────┐                                │
```

**[다이어그램 해설]** 트랜잭션 아웃박스(Outbox) 패턴은 로컬 트랜잭션과 메시지 발행 사이의 원자성을 보장하는 핵심 패턴이다. 문제는 서비스가 로컬 DB에 트랜잭션을 커밋한 직후 메시지 브로커(Kafka)에 이벤트를 발행하려 할 때, 장애가 발생하면 로컬 트랜잭션은 커밋되었는데 메시지는 발행되지 않아 데이터 불일치가 발생하는 점이다. 해결책은 (1) 로컬 DB에 Outbox 테이블을 두어 트랜잭션과 이벤트 기록을 원자적으로 커밋하고, (2) 폴러(Poller) 또는 CDC(Change Data Capture) 기반으로 Outbox 테이블을 읽어 메시지를 발행하는 방식이다. 이때 폴러는 장애 발생 시 재시도(Retry)를 통해 중단 없는 발행을 보장하고, 발행 성공 후 Outbox 테이블에서 해당 레코드를 삭제한다. 이 패턴을 통해 "로컬 트랜잭션 커밋"과 "메시지 발행" 사이의 원자성을 보장하고, 사가 패턴의 안정성을 확보할 수 있다.

### 📢 섹션 요약 비유

마치 우편물을 보내고 "영수증"을 받은 후에야 배송을 완료하는 것처럼, 트랜잭션 아웃박스는 DB에 "이벤트 기록"을 남긴 후 메시지 발행을 완료해야 배송(트랜잭션)을 안전하게 완료할 수 있습니다.

### 미래 전망

- **Distributed Transaction Coordinator 발전**: 분산 트랜잭션 코디네이터가 Seata(Simple Extensible Autonomous Transaction Architecture) 같은 오픈소스 프로젝트로 발전하여, MSA 환경에서도 사용하기 쉬운 사가 패턴 구현을 지원할 것이다.

- **트랜잭션 메시징 표준화**: 분산 트랜잭션의 메시징 형식을 표준화하고, 사가 패턴의 보상 로직을 선언적으로 정의할 수 있는 DSL(Domain-Specific Language)이 등장할 것이다.

- **AI 기반 트랜잭션 복구**: 머신러닝을 활용하여 실패한 트랜잭션의 원인을 분석하고, 자동으로 복구 전략을 제안하는 AIOps가 발전할 것이다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **CAP 정리 (Consistency, Availability, Partition tolerance)** | 분산 시스템에서 3가지를 동시에 만족할 수 없고, 최대 2가지 선택 가능하다는 정리. 2PC는 CP를, MSA는 AP를 주로 선택 |
| **PACELC 모델** | CAP의 확장으로, 분단 없으면(P) C/A를 모두 선택 가능하지만, 분단 발생 시(E) C/A 중 하나를 희생해야 한다는 이론 |
| **트랜잭션 아웃박스 (Transactional Outbox)** | 로컬 트랜잭션과 메시지 발행의 원자성을 보장하는 패턴. Outbox 테이블에 이벤트를 기록하고 폴러/CDC가 발행 |
| **메시지 큐 (Message Queue)** | Kafka, RabbitMQ 등 비동기 메시징 시스템으로, 서비스 간 느슨한 결합 제거 및 안정성 확보에 활용 |
| **CDC (Change Data Capture)** | DB의 변경 로그를 실시간으로 capture하여 다운스트림으로 전송, 트랜잭션 아웃박스 구현에 활용 |
| **IDEMPOTENCE (멱등 방지)** | 동일 요청이 여러 번 실행되어도 결과가 중복되지 않도록 하는 보안 조치. 분산 환경 필수 |
| **재시도(Reentrancy)** | 사가 패턴 보상 트랜잭션 실행 시, 이미 보상된 트랜잭션을 다시 실행하지 않도록 IDEMPOTENCE 처리 필요 |
| **Read Committed/Snapshot Isolation** | 트랜잭션 격리 수준으로, 커밋된 데이터만 읽기 가능하게 하여 Phantom Read 방지 |
| **분산 락(Distributed Lock)** | 여러 노드에서 공유 자원에 대한 Lock을 관리. Redis, ZooKeeper 등으로 구현. 2PC의 대안으로 사용될 수 있음 |
| **Event Sourcing** | 상태(State)가 아니라 상태를 변경시킨 '이벤트 로그 자체'를 저장하여 완벽한 감사/복구 인프라 구현. CQRS와 결합 |

## 👶 어린이를 위한 3줄 비유 설명

1. **2PC(2단계 커밋)**는 친구들이 여행을 갈 때 "모두 가능해?"라고 먼저 물어보고, 다 동의하면 "그럼 다들 출발!"이라고 하는데, 한 명이라도 반대하면 아무도 못 가게 만드는 엄격한 규칙이에요.

2. 문제는 친구가 너무 많으면(서버가 수십 개) 모두가 동의할 때까지 기다려야 하고, 한 명이 연락이 안 되면 전체가 취소되어 아무도 못 가게 돼서, 엄청 느리고 자주 실패하게 된다는 점이에요.

3. 그래서 요즘은 각자 알아서 출발하고(서비스별 독립), 문제 생기면 다시 돌아오는(보상 트랜잭션) 더 유연한 방식을 많이 사용한답니다!
