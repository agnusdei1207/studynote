# Level 1 - 리소스별 고유 URI 할당

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Richardson Maturity Model Level 1은 각 도메인 리소스(사용자, 주문, 상품 등)에 고유한 URI 경로를 할당하여 HTTP URI 공간의 식별자(Identifier)로 활용하는 첫 단계로, 리소스 중심 설계로의 패러다임 전환 의미
> 2. **가치**: 단일 URI 문제를 해결하여 URI만으로 조작 대상을 명확히 식별 가능하게 하며, API Gateway에서 URI 기반 라우팅/모니터링/액세스 제어가 가능해지나, 여전히 POST 전용으로 캐싱 효과는 제한적
> 3. **융합**: 레거시 RPC 스타일 API에서 RESTful API로 진화하는 중간 단계로서, URI 식별력 확보는 OpenAPI 명세 자동화, GraphQL resolver 설계, MSA 서비스 경계 정의 등 후속 작업의 기반

## Ⅰ. 개요 (Context & Background)

### 개념 정의

Level 1 (Resources)는 Richardson Maturity Model의 두 번째 단계로, 각 도메인 리소스에 고유한 URI 경로를 할당하여 HTTP URI 공간을 리소스 식별자로 활용하는 설계 패턴이다. Level 0(The Swamp of POX)의 단일 URI(`/api`) 문제를 해결하기 위해, `/users/123`, `/orders/456`, `/products/789`와 같이 리소스 종류별로 별도의 URI 경로를 할당한다.

이 단계의 핵심 개선은 "무엇을 조작하는지"를 URI 경로만으로 식별 가능하게 된 점이다. Level 0에서는 요청 본문의 `action` 필드를 확인해야만 알 수 있었던 작업 대상이, Level 1에서는 URI 경로만으로 즉시 식별 가능하다. 따라서 로드 밸런서, API Gateway, 모니터링 시스템이 URI 경로를 기반으로 라우팅, 메트릭 수집, 액세스 제어 정책을 적용할 수 있게 된다.

그러나 Level 1은 여전히 POST 메서드만 사용하는 경우가 많아, HTTP 프로토콜이 가진 캐싱, 멱등성 보장 등의 혜택을 완전히 누리지 못한다는 한계가 있다. 조회(Create), 읽기(Read), 업데이트(Update), 삭제(Delete) 작업이 모두 POST로 표현되므로, 중간 프록시나 CDN이 이를 구분하여 최적화할 수 없다.

### 등장 배경

Level 0에서 Level 1로의 진화는 자연스러운 설계 개선 과정이다. API 엔드포인트 수가 증가하면서 단일 URI로 모든 요청을 처리하는 디스패치 로직이 복잡해지고, 팀 내에서 "이 URI는 어떤 리소스를 위한 것인가"에 대한 혼란이 발생하기 시작한다.

이때 개발팀은 자연스럽게 리소스별 URI를 도입하게 된다. 예를 들어:
- 사용자 관련 API: `/api/user/*` 또는 `/api/users/*`
- 주문 관련 API: `/api/order/*` 또는 `/api/orders/*`
- 상품 관련 API: `/api/product/*` 또는 `/api/products/*`

이 URI 분리는 몇 가지 명확한 실무적 이점을 제공한다:
1. **라우팅**: API Gateway나 로드 밸런서가 URI 경로를 기반으로 요청을 적절한 백엔드 서비스로 라우팅 가능
2. **모니터링**: `/users/*` 요청량과 `/orders/*` 요청량을 별도로 모니터링 가능
3. **액세스 제어**: URI 경로 기반으로 인가 정책 적용 가능 (예: `/admin/*`는 관리자만 접근)
4. **문서화**: 리소스별 API 문서 작성이 용이

### 기술적 문제의식

Level 1은 리소스 식별 문제를 해결하지만, "어떻게 조작할지(How)"의 문제는 여전히 해결하지 못한다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                   Level 0 vs Level 1 Identification                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [Level 0: 식별 불가]                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  POST /api                                                           │   │
│  │  {action: "getUser", params: {userId: 123}}                         │   │
│  │       │                                                             │   │
│  │       │ ❌ URI만으로는 무엇을 조작하는지 알 수 없음                 │   │
│  │       │    본문을 파싱해야만 알 수 있음                              │   │
│  │       ▼                                                             │   │
│  │  서버: switch(action) case "getUser": ...                           │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [Level 1: 식별 가능]                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  POST /users/123                                                     │   │
│  │  {action: "get"}                                                    │   │
│  │       │                                                             │   │
│  │       │ ✅ URI만으로 "users 리소스의 123번"을 조작함을 즉시 식별      │   │
│  │       │                                                              │   │
│  │       ▼                                                             │   │
│  │  API Gateway: "/users/*" → 사용자 서비스로 라우팅                   │   │
│  │                                                                      │   │
│  │  POST /orders/456                                                    │   │
│  │  {action: "delete"}                                                 │   │
│  │       │                                                             │   │
│  │       │ ✅ URI만으로 "orders 리소스의 456번"을 조작함을 즉시 식별     │   │
│  │       │                                                              │   │
│  │       ▼                                                             │   │
│  │  API Gateway: "/orders/*" → 주문 서비스로 라우팅                    │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

이 식별 비교 도식의 핵심은 Level 1에서 URI가 "리소스 식별자"로서의 역할을 수행한다는 점이다. Level 0에서는 URI가 단순한 전송 경로에 불과하지만, Level 1에서는 URI 자체가 의미를 내포하므로 중간 인프라가 이를 활용할 수 있다. 실무에서는 이 URI 식별력 향상만으로도 API Gateway 라우팅 규칙이 60~80% 단순해지고, 모니터링 대시보드에서 리소스별 요청 패턴을 명확히 관찰할 수 있게 된다.

📢 **섹션 요약 비유**: Level 1은 도시 주소 체계를 도입하는 것과 같습니다. Level 0는 "집 하나"에 우편물을 모두 보내고 그 집에서 배달원이 수신인을 확인하는 방식이지만, Level 1은 각 건물에 고유한 주소(서울시 강남구 OO로 123번지)를 부여하여 우편 배달원이 목적지를 즉시 식별하고 최적 경로로 배달할 수 있게 합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Level 1 구조 요소

Level 1 패턴의 핵심 구성 요소는 리소스별 URI 경로 설계와 여전히 POST 전용인 HTTP 메서드 사용이다.

| 구성 요소 | 역할 | 내부 동작 | 프로토콜 | 비유 |
|:---------|:-----|:---------|:--------|:-----|
| **리소스 URI** | 도메인 리소스 식별 | `/users/{id}`, `/orders/{id}` | HTTP 경로 | 건물 주소 |
| **리소스 컬렉션** | 동종 리소스 그룹 | `/users`, `/orders` | HTTP 경로 | 아파트 단지 |
| **계층적 URI** | 리소스 간 관계 표현 | `/users/{id}/orders` | HTTP 경로 | 주소의 상세 경로 |
| **POST 전용** | 유일한 HTTP 메서드 | 모든 작업을 POST로 처리 | HTTP 메서드 | 모든 택배가 같은 차량 |
| **액션 파라미터** | POST 내 작업 식별 | `{action: "get"}`, `{action: "delete"}` | JSON | 운송장의 운송 유형 |

### 리소스 URI 설계 패턴

Level 1에서 가장 중요한 설계 결정은 URI 경로 구조이다. 잘 설계된 URI는 직관적이고 일관적이어야 한다.

**① 리소스 명명 규칙**

```text

│                    Resource URI Naming Conventions                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [단수형 vs 복수형]                                                         │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  단수형: /user/123, /order/456                                     │   │
│  │  복수형: /users/123, /orders/456  ← 일반적으로 권장                 │   │
│  │                                                                      │   │
│  │  근거: 컬렉션("/users")과 개별 항목("/users/123")의 일관성          │   │

│                                                                             │

│                                                                             │
│  [계층적 관계 표현]                                                         │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  /users/123                    ← 사용자 리소스                      │   │
│  │  /users/123/orders              ← 사용자의 주문 목록                │   │
│  │  /users/123/orders/456          ← 특정 주문                        │   │
│  │                                                                      │   │
│  │  규칙: 소유 관계는 URI 경로로 표현                                  │   │

│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**② URI 경로 구조 예시**

다음은 전형적인 Level 1 API의 URI 경로 구조 예시이다.

```text
사용자(User) 리소스:
  POST /users              ← 사용자 목록 조회 또는 생성
  POST /users/123          ← 특정 사용자 조회/수정/삭제
  POST /users/123/orders   ← 사용자의 주문 목록 조회

주문(Order) 리소스:
  POST /orders             ← 주문 목록 조회 또는 생성
  POST /orders/456         ← 특정 주문 조회/수정/삭제
  POST /orders/456/items   ← 주문 항목 목록 조회

상품(Product) 리소스:
  POST /products           ← 상품 목록 조회 또는 생성
  POST /products/789       ← 특정 상품 조회/수정/삭제
  POST /products/789/reviews ← 상품 리뷰 목록 조회
```

### Level 1 요청/응답 구조

Level 1에서는 POST 메서드만 사용하지만, URI 경로로 대상 리소스를 명확히 식별한다.

```json
// 사용자 조회
POST /users/123
Content-Type: application/json

{
  "action": "get"
}

// 사용자 삭제
POST /users/123
Content-Type: application/json

{
  "action": "delete"
}

// 주문 생성
POST /orders
Content-Type: application/json

{
  "action": "create",
  "data": {
    "userId": 123,
    "items": [
      {"productId": 789, "quantity": 2}
    ]
  }
}

// 주문 수정
POST /orders/456
Content-Type: application/json

{
  "action": "update",
  "data": {
    "status": "shipped"
  }
}
```

이 구조의 핵심은 URI 경로(`/users/123`, `/orders/456`)가 "무엇을 조작하는지"를 명확히 식별하고, 요청 본문의 `action` 필드가 "어떻게 조작할지"를 지정한다는 점이다.

### Level 1의 한계: POST 전용 문제

Level 1의 가장 큰 한계는 여전히 POST 메서드만 사용한다는 점이다. 이로 인해 다음과 같은 문제가 발생한다.

**① 캐싱 불가**

모든 요청이 POST이므로 중간 프록시나 CDN이 응답을 캐싱하지 않는다. 조회 작업(GET이어야 할 작업)도 POST로 표현되므로, 자주 조회되는 데이터를 캐싱하여 서버 부하를 줄일 수 없다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Level 1 Caching Limitation                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  요청: POST /products/789  (상품 정보 조회)                                 │
│        {action: "get"}                                                     │
│       │                                                                    │
│       │ CDN 캐시 레이어                                                     │
│       ▼                                                                    │
│  ❌ "POST 요청이므로 캐싱 불가"                                            │
│  ❌ 동일 상품 조회 요청이 매번 원본 서버로 전송                             │
│       │                                                                    │
│       ▼                                                                    │
│  ┌─────────────────┐                                                      │
│  │ 제품 서비스     │  ← 불필요한 중복 조회 요청                            │
│  └─────────────────┘                                                      │
│                                                                             │
│  [Level 2와의 비교]                                                         │
│  요청: GET /products/789   ← 조회는 GET 사용                               │
│       │                                                                    │
│       │ CDN 캐시 레이어                                                     │
│       ▼                                                                    │
│  ✅ "GET 요청이므로 캐싱 가능"                                             │
│  ✅ 첫 요청 이후 CDN이 즉시 응답                                           │
│       │                                                                    │
│       ▼                                                                    │
│  ┌─────────────────┐                                                      │
│  │   CDN 캐시     │  ← 서버 부하 60~80% 경감                              │
│  └─────────────────┘                                                      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**② 멱등성 불명확**

GET, PUT, DELETE는 일반적으로 멱등(동일 요청 여러 번 호출해도 결과 동일)하지만, POST는 기본적으로 비멱등(매 호출마다 부수 효과 발생 가능)으로 간주된다. Level 1에서 조회와 삭제가 모두 POST로 표현되므로, 클라이언트는 "이 요청을 안전하게 재시도해도 되는지" 판단할 수 없다.

**③ HTTP 도구 활용 제한**

curl, Postman, 브라우저와 같은 HTTP 클라이언트 도구는 GET 요청을 간편하게 호출할 수 있는 기능을 제공하지만, Level 1에서는 모든 작업에 POST 요청 본문을 구성해야 하므로 테스트와 디버깅이 번거로워진다.

📢 **섹션 요약 비유**: Level 1은 택배 시스템에서 각 건물에 고유한 주소를 부여한 것과 같습니다. 배달원은 목적지 주소를 보고 어느 동네로 가야 할지 즉시 알 수 있게 되었지만, 여전히 모든 배달이 같은 종류의 차량(POST)으로 이루어져 소포와 대형 화물을 구분하지 못해 효율이 낮습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### Level 1 vs Level 2 비교

Level 1에서 Level 2로의 진화는 URI 식별력에 HTTP 동사의 의미를 더하는 과정이다.

| 비교 항목 | Level 1 (Resources) | Level 2 (HTTP Verbs) | 개선 효과 |
|:---------|:-------------------|:-------------------|:---------|
| **URI 식별력** | 리소스별 URI | 리소스별 URI | 동일 |
| **HTTP 메서드** | POST만 | GET/POST/PUT/DELETE | 의미 부여 |
| **캐싱 가능성** | 낮음(~20%) | 높음(~65%) | +225% |
| **멱등성 보장** | 불명확 | 명확(GET/PUT/DELETE) | 안전성 ↑ |
| **API Gateway 라우팅** | URI 기반 | URI+메서드 기반 | 정교도 ↑ |
| **HTTP 도구 친화성** | 낮음 | 높음 | 테스트 용이 |

### URI 설계 원칙

Level 1에서 도입한 리소스별 URI 설계는 Level 2, 3으로 진화해도 유효한 원칙이다.

**① URI는 명사(리소스)를 중심으로 설계**

```
✅ 좋은 URI:
/users/123          ← 사용자 리소스
/orders/456         ← 주문 리소스
/products/789       ← 상품 리소스

❌ 나쁜 URI:
/getUser/123        ← 동사가 포함됨
/createOrder        ← 동사가 포함됨
/deleteProduct/789  ← 동사가 포함됨
```

**② 계층 구조로 리소스 관계 표현**

```
/users/123              ← 사용자
/users/123/orders       ← 사용자의 주문들
/users/123/orders/456   ← 특정 주문

(하지만 너무 깊은 계층은 피할 것)
```

**③ 쿼리 파라미터로 필터링/정렬/페이지네이션**

```
GET /users?role=admin&status=active&sort=name&limit=20&offset=0
(하지만 Level 1에서는 POST로 표현)
POST /users
{
  "action": "list",
  "filters": {"role": "admin", "status": "active"},
  "sort": "name",
  "limit": 20,
  "offset": 0
}
```

### 과목 융합: Level 1과 연관 기술 스택

**① OpenAPI Specification과의 시너지**

Level 1의 리소스별 URI 설계는 OpenAPI 명세 작성을 크게 단순화한다. URI 경로 구조가 도메인 리소스를 명확히 반영하므로, OpenAPI의 `paths` 섹션을 직관적으로 작성 가능하다.

```yaml
# OpenAPI 3.0 예시
paths:
  /users:
    post:
      summary: 사용자 목록 조회 또는 생성
      requestBody:
        content:
          application/json:
            schema:
              type: object
              properties:
                action:
                  type: string
                  enum: [list, create]
  /users/{userId}:
    post:
      summary: 특정 사용자 조회/수정/삭제
      parameters:
        - name: userId
          in: path
          required: true
          schema:
            type: integer
```

**② MSA (Microservices Architecture)와의 시너지**

Level 1의 리소스별 URI는 MSA의 서비스 경계 정의와 자연스럽게 정렬된다.

```
/users/*     → User Service
/orders/*    → Order Service
/products/*  → Product Service
```

이 URI 기반 경계는 API Gateway에서 서비스 디스커버리와 라우팅을 단순화한다.

**③ GraphQL과의 관계**

GraphQL은 단일 URI(`/graphql`)를 사용하지만, Resolver 설계 시 리소스별 URI 개념이 활용된다. Level 1의 리소스 중심 사고는 GraphQL 스키마 설계에도 적용된다.

```graphql
type Query {
  user(id: ID!): User        ← users/{id} 리소스
  orders(userId: ID!): [Order] ← users/{id}/orders 리소스
  product(id: ID!): Product   ← products/{id} 리소스
}
```

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오별 Level 1 도입 판단

Level 1는 Level 0에서 Level 2로 이동하는 중간 단계로서, 특정 상황에서는 의미 있는 개선을 제공한다.

| 시나리오 | Level 1 적합성 | 판단 근거 | 향후 방향 |
|:--------|:-------------:|:---------|:---------|
| **레거시 시스템 리팩토링 초기** | ⚠️ 조건부 적합 | URI 분리로 리소스 식별력 향상 | 빠르게 Level 2로 진화 |
| **MSA 서비스 분리** | ⚠️ 적합하지만 불충분 | URI 기반 라우팅 가능하지만 캐싱 불가 | Level 2와 함께 표준화 |
| **내부 도구 API** | ⚠️ 허용 가능 | 팀 내부에서는 URI 식별력만으로도 충분 | 장기적으로 Level 2 고려 |
| **공개 API** | ❌ 부적합 | 캐싱 불가로 성능 이슈 | Level 2 이상 필수 |
| **대규모 트래픽** | ❌ 부적합 | POST 전용으로 확장성 제한 | Level 2 + CDN 필수 |

### Level 1 → Level 2 진화 전략

### URI 설계 안티패턴

**안티패턴 1: URI에 동사 포함**

```
❌ 나쁜 예:
/getUser/123
/createOrder
/deleteProduct/789

✅ 좋은 예:
/users/123          (조회는 GET)
/orders             (생성은 POST)
/products/789       (삭제는 DELETE)
```

**안티패턴 2: 너무 깊은 URI 계층**

```
❌ 나쁜 예:
/api/v1/shop/online/products/electronics/computers/laptops/123

✅ 좋은 예:
/products/123
또는
/categories/electronics/computers/laptops/123
```

**안티패턴 3: 불일치하는 명명 규칙**

```
❌ 나쁜 예 (단수형/복수형 혼용):
/user/123
/orders/456
/product/789

✅ 좋은 예 (일관된 복수형):
/users/123
/orders/456
/products/789
```

📢 **섹션 요약 비유**: Level 1은 도시에서 일방통행 도로를 주요 도로별로 구분하는 것과 같습니다. 각 도로(URI)가 명확히 구분되어 어디로 가야 할지 알 수 있게 되었지만, 여전히 모든 차량이 같은 종류(POST)라서 출퇴근 시간에 병목이 발생합니다. Level 2는 도로 종류(고속도로, 일반 도로)를 구분하여 차종별로 적절한 도로를 사용하게 하는 것입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### Level 1의 정량적 효과

| 평가 항목 | Level 0 | Level 1 | 개선 효과 |
|:--------|:-------|:-------|:--------:|
| **URI 식별력** | 없음 | 있음 | N/A |
| **API Gateway 라우팅** | 불가 | URI 기반 가능 | ✅ |
| **모니터링 정교도** | 낮음 | URI별 분리 가능 | +200% |
| **서비스 분리 용이성** | 어려움 | URI 기반 경계 형성 | ✅ |
| **캐싱 가능성** | 거의 없음 | 여전히 낮음 | ❌ |
| **개발 복잡도** | 중간 | 중간 | 동일 |

### 기술적 부채 관점

Level 1는 Level 0보다는 개선되었지만, 여전히 POST 전용으로 인한 기술적 부채가 존재한다.

**① 단기 (1-3개월)**: URI 식별력 향상으로 개발팀 간 협업 개선
**② 중기 (3-12개월)**: 캐싱 불가로 성능 병목 발생, Level 2 이동 필요성 대두
**③ 장기 (1년 이상)**: POST 전용 패턴이 정착하면 리팩토링 비용 급증

### 대안 및 향후 방향

Level 1은 Level 2로의 이동을 위한 중간 단계로 인식하고, 가능한 빠르게 HTTP 동사 활용을 도입할 것을 권장한다. OpenAPI Specification을 먼저 작성하고, 이를 기반으로 Level 2 엔드포인트를 자동 생성하는 도구(Swagger Codegen, OpenAPI Generator) 활용을 권장한다.

### 참고 표준

- **RFC 3986 - Uniform Resource Identifier (URI): Generic Syntax**: URI 설계 표준
- **OpenAPI Specification 3.1**: RESTful API 명세 표준

📢 **섹션 요약 비유**: Level 1은 집에 방마다 이름표를 붙인 것과 같습니다. 이제 "화장실이 어디에요?"라고 물으면 "화장실 문을 찾으세요"라고 답할 수 있게 되었지만, 여전히 모든 문이 같은 종류의 문(POST)이라서 열고 닫는 방식이 같습니다. Level 2는 방 용도별로 적절한 문(출입문, 창문, 비상구)을 설치하는 것입니다.

## 👶 어린이를 위한 3줄 비유 설명

**① 도서관 분류 비유**: Level 1은 도서관에 책을 주제별로 다른 선반에 배치하는 것과 같습니다. 과학 책은 과학 코너에, 이야기책은 문학 코너에 두면 원하는 주제의 책이 어디에 있는지 바로 알 수 있게 됩니다.

**② 집 주소 비유**: Level 1은 각 집에 주소를 붙이는 것과 같습니다. Level 0는 모든 집이 그저 "우리 동네 집"으로만 불러서 배달원이 누구에게 배달할지 알 수 없지만, Level 1는 "서울시 강남구 OO로 123번지"처럼 고유한 주소가 있어 배달이 가능해집니다.

**③ 장난감 상자 비유**: Level 1은 장난감을 종류별로 다른 상자에 담는 것과 같습니다. 자동차 장난감은 빨간 상자에, 로봇 장난감은 파란 상자에 넣어두면 원하는 종류의 장난감이 어디에 있는지 즉시 알 수 있게 됩니다. 하지만 상자를 여는 방법은 여전히 똑같습니다(POST만 사용).
