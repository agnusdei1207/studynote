# Level 3 - HATEOAS (Hypermedia as the Engine of Application State)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Richardson Maturity Model의 최상위 단계로, Roy Fielding이 정의한 REST 아키텍처의 핵심 제약 조건인 HATEOAS (Hypermedia as the Engine of Application State)를 구현하여, API 응답에 다음 상태 전이용 하이퍼미디어 링크를 동적으로 포함시키는 설계 패턴
> 2. **가치**: 클라이언트가 URI를 하드코딩하지 않고 서버 응답의 `_links` 필드를 통해 다음 가능한 액션을 동적으로 발견하므로, API 진화 시 URI 구조 변경에도 클라이언트 코드 수정 범위가 평균 65% 감소하고 서버-클라이언트 간 결합도가 최소화됨
> 3. **융합**: 공개 API, 장기 유지보수가 필요한 엔터프라이즈 API, 대규모 파트너 생태계를 갖는 플랫폼에서 가치가 크며, HAL(Hypertext Application Language), JSON:API, JSON Hyper-Schema 등 표준 포맷과 Spring HATEOAS, Spring Data REST 등 프레임워크가 도입 장벽 낮춤

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

Level 3 (HATEOAS)는 Richardson Maturity Model의 최상위 단계로, REST 아키텍처의 가장 본질적인 원칙인 "하이퍼미디어 기반 상태 전이(Hypermedia as the Engine of Application State)"를 구현한다. HATEOAS의 핵심은 클라이언트가 서버 URI를 사전에 알 필요 없이, 서버 응답에 포함된 하이퍼미디어 링크를 통해 다음 상태로 전이 가능한 경로를 동적으로 발견한다는 점이다.

Roy Fielding이 2000년 박사학위 논문에서 정의한 REST 아키텍처의 4가지 핵심 제약 조건 중, "하이퍼미디어 기반 상태 전이"는 가장 이해하기 어렵고 구현하기도 까다로운 원칙이다. 이는 REST의 발상지인 웹(World Wide Web)이 HTML 문서에 포함된 `<a>` 태그(하이퍼링크)를 통해 사용자가 다음 페이지로 이동하는 방식과 동일한 패러다임을 API에도 적용하는 것이다.

Level 2까지는 클라이언트가 URI를 하드코딩하여 서버 리소스에 접근한다. 예를 들어 `GET /users/123`으로 사용자를 조회한 후, 해당 사용자의 주문 목록을 가져오려면 클라이언트가 미리 `GET /users/123/orders` URI를 알고 있어야 한다. 하지만 Level 3(HATEOAS)에서는 `GET /users/123` 응답 본문에 `_links` 필드가 포함되어 있어, 클라이언트는 이 링크를 따라가기만 하면 된다.

```json
// Level 3 HATEOAS 응답 예시
{
  "id": 123,
  "name": "John Doe",
  "email": "john@example.com",
  "_links": {
    "self": {"href": "/users/123"},
    "orders": {"href": "/users/123/orders"},
    "update": {"href": "/users/123", "method": "PUT"},
    "delete": {"href": "/users/123", "method": "DELETE"}
  }
}
```

### 등장 배경

HATEOAS 개념은 Roy Fielding의 REST 논문(2000)에서 처음 정의되었으나, 실무 적용은 수년 뒤에야 시작되었다.

**① 웹의 성공 패턴 모방**

웹의 엄청난 성공은 HTML 문서에 포함된 하이퍼링크(`<a>`, `<img>`, `<form>`)를 통해 사용자가 URI를 외우지 않고도 다음 페이지로 이동할 수 있는 "발견 가능한(Discoverable)" 설계 때문이다. Fielding은 이 패턴을 API에도 적용하면, 클라이언트와 서버의 결합도를 낮추고 API 진화를 더 용이하게 할 수 있다고 주장했다.

**② SOA/Web Services의 URI 결합 문제**

SOAP/WS-* 기반 웹 서비스는 WSDL(Web Services Description Language)에 고정된 URI를 명시하므로, 서버 측에서 URI 구조를 변경하면 모든 클라이언트 코드를 수정해야 하는 강한 결합(Tight Coupling) 문제가 있었다. HATEOAS는 이 문제를 해결하기 위해 URI를 동적으로 제공하는 방안을 제시했다.

**③ 마이크로서비스 아키텍처(MSA)의 등장**

MSA 환경에서는 수십~수백 개의 서비스 간 통신이 발생하며, 각 서비스의 URI 구조가 독립적으로 진화할 수 있다. HATEOAS는 이러한 환경에서 서비스 간 결합도를 낮추고 독립적 배포/진화를 가능하게 하는 중요한 패턴으로 재조명되었다.

### 기술적 문제의식

Level 2까지의 RESTful API는 여전히 클라이언트-서버 간의 강한 결합 문제를 안고 있다.

**① URI 하드코딩 문제**

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Level 2 URI Hard-coupling Problem                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [클라이언트 코드]                                                           │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  // 사용자 조회                                                      │   │
│  │  const user = await fetch('/users/123');                           │   │
│  │                                                                      │   │
│  │  // 사용자 주문 조회 (URI 하드코딩)                                  │   │
│  │  const orders = await fetch('/users/123/orders');  ← 하드코딩된 URI  │   │
│  │                                                                      │   │
│  │  // 주문 취소 (URI 하드코딩)                                         │   │
│  │  await fetch(`/orders/${orderId}/cancel`, {method: 'POST'});        │   │
│  │                                                                      │   │
│  │  문제점:                                                             │   │
│  │  • 클라이언트가 URI 구조을 미리 알아야 함                            │   │
│  │  • 서버가 URI 구조를 변경하면 클라이언트 코드 수정 필요              │   │
│  │  • API 버전 업데이트 시 모든 클라이언트 배포 필요                  │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [서버 URI 구조 변경]                                                       │
│  /users/123/orders  →  /api/v2/users/123/orders                          │
│  /orders/{id}/cancel →  /api/v2/orders/{id}/cancel                        │
│       │                                                                     │
│       │ 강한 결합으로 인한 영향                                             │
│       ▼                                                                     │
│  모든 클라이언트 코드 수정 → 모든 클라이언트 재배포                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**② Level 3(HATEOAS)로의 해결**

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Level 3 HATEOAS Decoupling Solution                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [클라이언트 코드 - 링크 기반]                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  // 사용자 조회                                                      │   │
│  │  const userResponse = await fetch('/users/123');                   │   │
│  │  const user = await userResponse.json();                           │   │
│  │                                                                      │   │
│  │  // 응답에 포함된 링크 추출                                          │   │
│  │  const ordersLink = user._links.orders.href;                       │   │
│  │  const orders = await fetch(ordersLink);  ← 서버가 제공한 URI 사용   │   │
│  │                                                                      │   │
│  │  특징:                                                               │   │
│  │  • 클라이언트는 URI 구조를 알 필요 없음                               │   │
│  │  • 서버가 URI 구조을 변경해도 클라이언트 코드 영향 없음              │   │
│  │  • API 버전 업데이트 시 클라이언트 재배포 불필요                    │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [서버 URI 구조 변경]                                                       │
│  /users/123/orders  →  /api/v2/users/123/orders                          │
│       │                                                                     │
│       │ HATEOAS로 인한 영향                                                │
│       ▼                                                                     │
│  응답의 _links 필드만 변경 → 클라이언트는 링크를 그대로 사용                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

이 결합도 비교의 핵심은 Level 3(HATEOAS)가 클라이언트-서버 간의 URI 결합을 끊어(Decoupling), 서버가 독립적으로 진화할 수 있게 한다는 점이다. 실무 데이터에 따르면, HATEOAS를 적용한 공개 API에서 서버 측 URI 구조 변경 시 클라이언트 코드 수정 범위가 평균 65% 감소하고, API 버전 호환성 이슈가 약 40% 감소한다는 보고가 있다.

📢 **섹션 요약 비유**: Level 3(HATEOAS)은 쇼핑몰에서 안내 직원과 함께 쇼핑하는 것과 같습니다. Level 2는 쇼핑몰 지도를 외워서 혼자 찾아다녀야 하지만, 매장이 이사하면 지도가 쓸모없게 됩니다. 반면 Level 3는 안내 직원에게 "다음에 어디로 가면 되나요?"라고 물어보면, 매장이 이사하더라도 안내 직원이 새로운 위치를 알려주므로 계속해서 쇼핑을 계속할 수 있습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### HATEOAS 핵심 구성 요소

HATEOAS 구현을 위한 핵심 요소와 각 요소의 역할을 분석한다.

| 구성 요소 | 역할 | 포맷 예시 | 비유 |
|:---------|:-----|:---------|:-----|
| **링크 관계(rel)** | 링크의 의미를 정의 | `self`, `next`, `orders`, `update` | 도로 표지판의 목적지 |
| **링크 URI(href)** | 다음 상태의 실제 경로 | `/users/123/orders` | 도로 표지판의 방향 |
| **링크 유형(type)** | 대상 리소스의 미디어 타입 | `application/json`, `text/html` | 배송물의 종류 |
| **HTTP 메서드(hint)** | 링크 사용에 필요한 메서드 | `GET`, `POST`, `PUT`, `DELETE` | 운송 수단 |
| **템플릿(template)** | URI 템플릿 (확장 가능한 URI) | `/search{?page,limit}` | 동적 경로 |

### HATEOAS 응답 구조

**① HAL (Hypertext Application Language) 포맷**

HAL은 HATEOAS 구현을 위한 가장 널리 사용되는 포맷 중 하나이다.

```json
{
  "_links": {
    "self": {"href": "/users/123"},
    "next": {"href": "/users?page=2"},
    "orders": {"href": "/users/123/orders"}
  },
  "_embedded": {
    "orders": [
      {
        "_links": {
          "self": {"href": "/orders/456"}
        },
        "id": 456,
        "status": "pending"
      }
    ]
  },
  "id": 123,
  "name": "John Doe"
}
```

**② JSON:API 포맷**

JSON:API는 HATEOAS와 더불어 관계(Relationships) 표현에 특화된 포맷이다.

```json
{
  "data": {
    "type": "users",
    "id": "123",
    "attributes": {
      "name": "John Doe",
      "email": "john@example.com"
    },
    "relationships": {
      "orders": {
        "links": {
          "self": "/users/123/relationships/orders",
          "related": "/users/123/orders"
        }
      }
    }
  },
  "links": {
    "self": "/users/123"
  }
}
```

**③ Collection+JSON 포맷**

Collection+JSON은 컬렉션 기반 리소스 표현에 특화된 포맷이다.

```json
{
  "collection": {
    "version": "1.0",
    "href": "/users",
    "links": [
      {"rel": "next", "href": "/users?page=2"}
    ],
    "items": [
      {
        "href": "/users/123",
        "data": [
          {"name": "id", "value": "123"},
          {"name": "name", "value": "John Doe"}
        ]
      }
    ]
  }
}
```

### HATEOAS 상태 전이 다이어그램

다음 그림은 HATEOAS에서 클라이언트가 링크를 따라 상태를 전이하는 과정을 보여준다.

```text
│  │       ▼                                                             │   │
│  │  {                                                                  │   │
│  │    "_links": {                                                      │   │
│  │      "users": {"href": "/users"},                                   │   │
│  │      "orders": {"href": "/orders"},                                 │   │
│  │      "products": {"href": "/products"}                              │   │
│  │    }                                                                │   │
│  │  }                                                                  │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                    │                                        │
│                                    │ 클라이언트: users 링크 탐색            │
│                                    ▼                                        │
│  [상태 1: 사용자 목록]                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  GET /users                                                          │   │
│  │       │                                                             │   │
│  │       ▼                                                             │   │
│  │  {                                                                  │   │
│  │    "users": [...],                                                  │   │

│  │      "self": {"href": "/users"},                                    │   │
│  │      "next": {"href": "/users?page=2"},                            │   │

│  │    "_links": {                                                      │   │
│  │      "self": {"href": "/users/123"},                                │   │

│  │  }                                                                  │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [핵심 특성]                                                                │
│  • 클라이언트는 URI를 하드코딩하지 않음                                     │
│  • 서버 응답의 _links 필드가 다음 가능한 상태 전이를 안내                  │
│  • URI 구조 변경 시 _links만 업데이트하면 클라이언트 영향 없음              │
│  • REST의 "하이퍼미디어 기반 상태 전이" 원칙 실현                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

이 상태 전이 흐름도의 핵심은 클라이언트가 "다음에 무엇을 할 수 있는지"를 서버 응답에서 동적으로 발견한다는 점이다. 웹 브라우저가 HTML 페이지의 `<a>` 태그를 통해 다음 페이지를 발견하듯, API 클라이언트도 `_links` 필드를 통해 다음 상태로 전이 가능한 경로를 발견한다. 따라서 서버가 URI 구조를 변경하더라도 `_links` 필드만 업데이트하면, 클라이언트는 별도 코드 수정 없이 새로운 URI로 계속해서 상태 전이가 가능하다.

### HATEOAS 링크 관계(rel) 표준

IANA(Internet Assigned Numbers Authority)는 RFC 5988에 따라 링크 관계(rel) 표준을 관리한다.

| rel 값 | 의미 | 사용 예시 |
|:------|:-----|:---------|
| **self** | 현재 리소스 자신 | `{"rel": "self", "href": "/users/123"}` |
| **next** | 컬렉션의 다음 페이지 | `{"rel": "next", "href": "/users?page=2"}` |
| **prev** | 컬렉션의 이전 페이지 | `{"rel": "prev", "href": "/users?page=1"}` |
| **first** | 컬렉션의 첫 페이지 | `{"rel": "first", "href": "/users?page=1"}` |
| **last** | 컬렉션의 마지막 페이지 | `{"rel": "last", "href": "/users?page=10"}` |
| **edit** | 리소스 편집 | `{"rel": "edit", "href": "/users/123", "method": "PUT"}` |
| **create** | 새 리소스 생성 | `{"rel": "create", "href": "/users", "method": "POST"}` |
| **delete** | 리소스 삭제 | `{"rel": "delete", "href": "/users/123", "method": "DELETE"}` |
| **related** | 관련 리소스 | `{"rel": "related", "href": "/users/123/orders"}` |

📢 **섹션 요약 비유**: Level 3(HATEOAS)은 미로 게임에서 안내 지도를 받고 탐험하는 것과 같습니다. Level 2는 미로 지도를 미리 외워서 혼자 찾아다녀야 하지만, 미로 구조가 바뀌면 지도가 쓸모없게 됩니다. 반면 Level 3는 각 지점마다 "다음에 갈 수 있는 곳"을 알려주는 표지판이 있어서, 미로 구조가 바뀌어도 표지판만 업데이트하면 계속해서 탈출할 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### Level 2 vs Level 3 비교

Level 2와 Level 3의 핵심 차이는 클라이언트-서버 간 결합도와 진화 용이성에 있다.

| 비교 항목 | Level 2 | Level 3(HATEOAS) | 차이점 |
|:---------|:-------|:----------------|:------|
| **URI 결합도** | 높음 (하드코딩) | 낮음 (동적 링크) | 결합도 65% 저감 |
| **API 진화 용이성** | 낮음 | 높음 | URI 변경 시 클라이언트 영향 최소화 |
| **클라이언트 복잡도** | 낮음 | 중간 | 링크 추적 로직 필요 |
| **응답 크기** | 작음 | 큼 (링크 포함) | 약 20~40% 증가 |
| **캐싱 효율** | 높음 | 중간 | 링크 포함으로 캐시 키 복잡 |
| **표준화** | 널리 사용 | 제한적 사용 | 도구 생태계 차이 |

### HATEOAS 표준 포맷 비교

| 포맷 | 특징 | 장점 | 단점 | 사용 사례 |
|:-----|:-----|:-----|:-----|:---------|

| **JSON:API** | `data`, `relationships`, `included` | 관계 표현 강력, 중복 제거 | 복잡한 구조, 러닝 커브 | 복잡한 관계 데이터 |

| **JSON Hyper-Schema** | 스키마 + 링크 정의 | 강력한 검증, 문서화 | 복잡함 | OpenAPI 통합 |
| **Siren** | `entities`, `actions`, `links` | 행위(Actions) 표현 강력 | 사용 빈도 낮음 | 행위 중심 API |

### 과목 융합: HATEOAS와 연관 기술 스택

**① Spring HATEOAS / Spring Data REST**

Java 생태계에서 가장 널리 사용되는 HATEOAS 구현 프레임워크로, 개발자가 쉽게 HATEOAS 링크를 추가할 수 있는 추상화를 제공한다.

```java
// Spring HATEOAS 예시
@GetMapping("/users/{id}")
EntityModel<User> getUser(@PathVariable Long id) {
    User user = userRepository.findById(id);
    Link selfLink = linkTo(methodOn(UserController.class).getUser(id)).withSelfRel();
    Link ordersLink = linkTo(methodOn(UserController.class).getOrders(id)).withRel("orders");
    return EntityModel.of(user, selfLink, ordersLink);
}
```

**② GraphQL과의 비교**

GraphQL은 단일 URI(`/graphql`)와 POST만 사용하지만, 쿼리 언어를 통해 클라이언트가 필요한 데이터를 정확히 지정(Over-fetching/Under-fetching 해결)할 수 있다. HATEOAS는 HTTP 표준 메서드와 URI를 그대로 활용하면서 링크 기반 발견 가능성을 제공한다는 점에서 상호 보완적이다.

**③ API 버전 관리와의 시너지**

HATEOAS는 API 버전 관리에 강력한 도구를 제공한다. 서버가 `/api/v1/users`에서 `/api/v2/users`로 마이그레이션할 때, v1 응답에 `"migrate": {"href": "/api/v2/users/123", "method": "GET"}` 링크를 포함하면, 클라이언트는 자연스럽게 v2로 이동할 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오별 HATEOAS 적용

HATEOAS는 모든 API에 필수적인 것은 아니며, 특정 상황에서 가치가 크다.

| 시나리오 | HATEOAS 적합성 | 판단 근거 |
|:--------|:-------------:|:---------|
| **공개 API/플랫폼** | ✅ 매우 적합 | 다수 파트너 통합, API 진화 필요성 높음 |
| **장기 유지보수 API** | ✅ 매우 적합 | URI 구조 변경 가능성, 클라이언트 통제 어려움 |
| **대규모 파트너 생태계** | ✅ 매우 적합 | 파트너별 클라이언트 버전 관리 복잡 |
| **내부 MSA 통신** | ⚠️ 선택적 | 서비스 수가 적으면 Level 2로 충분 |
| **모바일 앱 백엔드** | ⚠️ 선택적 | 앱 배포 주기 고려 필요 |
| **간단한 CRUD API** | ❌ 과도함 | Level 2로 충분, HATEOAS 오버헤드 |

### HATEOAS 구현 체크리스트

**링크 포함 기준**
- [ ] 모든 리소스 응답에 `self` 링크를 포함하는가
- [ ] 컬렉션 응답에 `next`, `prev`, `first`, `last` 링크를 포함하는가
- [ ] 관계 리소스에 `related` 링크를 포함하는가
- [ ] 가능한 액션에 `edit`, `create`, `delete` 링크를 포함하는가

**링크 품질**
- [ ] 링크 관계(rel) 값이 명확한 의미를 갖는가
- [ ] 링크가 상대 경로인가 (절대 경로는 도메인 결합 유발)
- [ ] HTTP 메서드 힌트를 포함하는가 (POST/PUT/DELETE 필요 시)

**클라이언트 친화성**
- [ ] 응답 크기가 30% 이상 증가하지 않는가
- [ ] 링크 깊이가 3단계 이내인가
- [ ] 링크 순환 참조(Circular Reference)을 방지하는가

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과

HATEOAS 도입 시 기대되는 주요 효과를 정량적으로 분석한다.

| 평가 항목 | Level 2 | Level 3(HATEOAS) | 개선 효과 |
|:--------|:-------|:---------------|:--------:|
| **API 진화 시 클라이언트 수정 범위** | 100% | 35% | -65% |
| **URI 구조 변경 유연성** | 낮음 | 매우 높음 | ✅ |
| **응답 크기** | 기준 | +20~40% | 오버헤드 |
| **클라이언트 복잡도** | 낮음 | 중간 | +50% |
| **API 문서 유지보수 비용** | 중간 | 낮음 | -40% |
| **파트너 통합 용이성** | 중간 | 높음 | +80% |

### 기술적 효과

**① 결합도 저감**

URI 구조 변경 시 클라이언트 코드 수정 범위가 평균 65% 감소하며, 서버-클라이언트 간 독립적 진화가 가능해진다.

**② API 문서 간소**

HATEOAS API는 `_links` 필드가 스스로 문서화 역할을 하므로, 별도 API 문서 작성 비용이 약 40% 감소한다.

**③ 파트너 통합 용이**

공개 API 제공자는 URI 구조를 자유롭게 진화시킬 수 있으며, 파트너는 링크만 따르면 되므로 통합 기간이 평균 30% 단축된다.

### 향후 전망

HATEOAS는 2025-2030년에도 특정 도메인에서 중요성을 유지할 것이다. 특히:
- 대규모 플랫폼 API(Amazon, PayPal, Stripe)
- 장기 유지보수가 필요한 금융/정부 API
- 복잡한 파트너 생태계를 갖는 B2B API

에서 HATEOAS 채택이 지속적으로 증가할 것이다. 동시에 HAL, JSON:API 등 표준 포맷이 더욱 안정화되고, 도구 생태계(Spring HATEOAS, Hypermedia libraries)가 성숙될 것이다.

### 참고 표준

- **RFC 5988 - Web Linking**: HTTP Link 헤더와 링크 관계(rel) 표준
- **RFC 8288 - Web Linking**: RFC 5988의 갱신 버전
- **RFC 9110 - HTTP Semantics**: HTTP 메서드와 HATEOAS 관계
- **HAL (Hypertext Application Language)**: IETF RFC에서 제안된 HATEOAS 포맷
- **JSON:API specification**: HATEOAS와 관계 표현에 특화된 JSON 포맷

📢 **섹션 요약 비유**: Level 3(HATEOAS)은 GPS 내비게이션과 실시간 교통 정보가 결합된 최신 내비게이션 시스템과 같습니다. Level 2는 종이 지도로 운전하는 것과 같아서 도로가 바뀌면 지도를 교체해야 하지만, Level 3는 실시간으로 도로 상황을 업데이트받아 최적 경로를 안내받을 수 있습니다. 모든 운전자에게 필요한 것은 아니지만, 배달 기사나 택시 기사처럼 매일 도로를 달리는 사람들에게는 필수적인 도구입니다.

## 👶 어린이를 위한 3줄 비유 설명

**① 보물찾기 지도 비유**: Level 3(HATEOAS)은 보물찾기 게임에서 각 지점마다 다음 지점으로 가는 단서가 있는 지도와 같습니다. Level 2는 미리 전체 지도를 외워서 찾아야 하지만, 길이 바뀌면 지도가 쓸모없게 됩니다. 반면 Level 3는 각 지점에 "다음에 동쪽으로 가면 보물이 있어" 같은 단서가 있어서, 길이 바뀌어도 단서만 업데이트하면 계속해서 보물을 찾을 수 있습니다.

**② 쇼핑 안내 비유**: Level 3는 백화점에서 안내 직원과 함께 쇼핑하는 것과 같습니다. Level 2는 쇼핑몰 지도를 외워서 혼자 찾아다녀야 하지만, 매장이 이사하면 지도가 쓸모없게 됩니다. 반면 Level 3는 각 층마다 안내 직원이 "신발 코너는 3층입니다"라고 알려주므로, 매장이 이사하더라도 계속해서 쇼핑을 할 수 있습니다.

**③ 비행기 여정 비유**: Level 3는 공항에서 안내 표지판을 따라가는 것과 같습니다. Level 2는 공항 지도를 미리 외워서 혼자 찾아가야 하지만, 탑승구 위치가 바뀌면 지도가 쓸모없게 됩니다. 반면 Level 3는 각 표지판이 "A탑승구는 오른쪽"이라고 알려주므로, 탑승구가 바뀌어도 표지판만 업데이트하면 계속해서 비행기를 탈 수 있습니다.
