# 사가 패턴 (Saga Pattern) - 분산 트랜잭션 보장

> MSA(Microservices Architecture) 내에서 여러 서비스에 걸친 긴 트랜잭션을 안정적으로 처리하기 위해, 각 서비스의 로컬 트랜잭션을 연속적으로 실행하고 중간 실패 시 역순으로 보상 트랜잭션(Compensating Transaction)을 발행하여 논리적 롤백을 수행하는 결과적 일관성(Eventual Consistency) 기반 패턴

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사가 패턴(Saga Pattern)은 분산 환경에서 여러 서비스에 걸친 긴 트랜잭션을 각 서비스의 독립적인 로컬 트랜잭션으로 분리하여 실행하고, 실패 시 역순으로 보상 트랜잭션(Compensating Transaction)을 발행하여 전체를 롤백하는 결과적 일관성(Eventual Consistency) 기반 분산 트랜잭션 관리 패턴이다.
> 2. **가치**: 2PC(2-Phase Commit)의 블로킹/단일 장애점(SPOF) 문제를 해결하여 높은 가용성(99.9%+)과 확장성(수평 확장)을 확보하며, 각 서비스가 독립적으로 배포/확장 가능하여 MSA의 핵심 목표인 "서비스 독립성"을 달성한다.
> 3. **융합**: 이벤트 소싱(Event Sourcing), 트랜잭션 아웃박스(Transactional Outbox), CQRS(Command Query Responsibility Segregation)와 결합하여 MSA의 데이터 일관성을 보장하는 핵심 아키텍처로, 코레오그래피(Choreography) 사가와 오케스트레이션(Orchestration) 사가 두 가지 구현 방식이 있다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

사가 패턴(Saga Pattern)은 1987년 Hector Garcia-Molina와 Kenneth Salem이 제안한 분산 트랜잭션 관리 패턴으로, 여러 서비스에 걸친 긴 트랜잭션(Long-Lived Transaction)을 각 서비스의 로컬 트랜잭션(Local Transaction)으로 분리하여 순차적으로 실행하고, 중간 단계에서 실패가 발생하면 이미 완료된 단계를 역순으로 취소하는 보상 트랜잭션(Compensating Transaction)을 실행하여 전체 트랜잭션을 롤백하는 방식이다.

핵심은 **"하나의 큰 트랜잭션을 여러 개의 작은 트랜잭션으로 쪼개고, 실패 시 되돌리기(Cleanup)"**다는 것으로, 2PC가 모든 참여자가 동시에 Lock을 걸고 합의하는 방식인 반면, 사가는 각 서비스가 독립적으로 로컬 트랜잭션을 실행하고 실패 시만 보상하므로 블로킹이 적고 가용성이 높다.

### 💡 비유

사가 패턴은 **"여행사 일정 짜기"**와 같다.

- **2PC**: 모든 참여자가 동시에 "가능!" 확인하고 동시에 "예약!" (한 명이라도 불가능하면 전체 취소)
- **사가 패턴**: 항공권 예약 → 호텔 예약 → 렌터카 예약을 순차적으로 진행하고, 렌터카 예약 실패 시 → 호텔 취소 → 항공권 취소 (역순 보상)

장점은 각 단계를 독립적으로 진행하므로 병렬 처리가 가능하고, 중간에 실패해도 이미 완료된 단계만 취소하면 된다. 단점은 일시적으로 "항공권은 예약됐는데 렌터카는 예약 안 된" 상태처럼 데이터 불일치가 발생할 수 있다.

### 등장 배경

1. **분산 트랜잭션의 등장 (1980년대)**: IBM CICS, Tuxedo 같은 TP Monitor가 2PC 프로토콜을 구현하여 여러 데이터베이스를 연동하는 분산 트랜잭션을 지원했으나, 블로킹과 단일 장애점(SPOF) 문제가 있었음.

2. **사가 패턴 제안 (1987)**: Garcia-Molina & Salem이 "Sagas" 논문에서 장기 실행 트랜잭션(Long-Lived Transaction)을 관리하기 위한 패턴을 제안. 각 단계를 로컬 트랜잭션으로 분리하고, 실패 시 역순으로 보상하는 방식 제시.

3. **MSA와 클라우드 시대의 부황 (2010년대~)**: 마이크로서비스 아키텍처(MSA)가 도입되면서 데이터베이스 퍼 서비스(Database per Service) 설계가 표준이 되고, 2PC의 한계(블로킹, SPOF)이 명확해짐. 구글, 아마존, 넷플릭스 등이 사가 패턴을 MSA에 적용하여 결과적 일관성(Eventual Consistency)을 수용.

4. **오케스트레이션 vs 코레오그래피 (2010년대 후반~)**: 초기 사가는 중앙 코디네이터(Orchestration)가 전체 흐름을 제어했으나, 마이크로서비스가 수십 개로 늘어나면서 코디네이터가 병목이 되고 SPOF가 다시 문제가 됨. 이에 각 서비스가 이벤트를 발행/구독하며 자율적으로 조정하는 코레오그래피(Choreography) 사가가 부상하게 됨.

**[다이어그램 해설]** 사가 패턴은 1987년에 학술적으로 제안되었으나, 2010년대 MSA와 클라우드 시대에 부활하여 실무적으로 널리 사용되기 시작했다. 초기 MSA에서는 오케스트레이션(Orchestration) 사가가 주류였는데, 중앙 코디네이터가 전체 트랜잭션 흐름을 제어하다 보니 서비스 수가 늘어나면 코디네이터 자체가 병목(SPOF, 성능 병목)이 되는 문제가 발생했다. 이에 각 서비스가 이벤트를 발행/구독하며 자율적으로 조정하는 코레오그래피(Choreography) 사가가 부상하게 되었다. 코레오그래피 사가는 이벤트 기반 비동기 통신(예: Kafka)을 통해 서비스 간 느슨한 결합을 제거하고, 각 서비스가 독립적으로 확장 가능한 MSA의 핵심 철학에 부합한다.

### 📢 섹션 요약 비유

마치 여행 일정을 짤 때, 모두가 동시에 예약하는 2PC 방식 대신, 순서대로 예약하고 실패 시 되돌리는 사가 방식으로 전환하여 각자 독립적으로 일정을 조율하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **로컬 트랜잭션 (Local Transaction)** | 각 서비스 내에서의 원자적 단위 트랜잭션 | DB의 ACID 보장 | @Transactional, BEGIN~COMMIT | 개별 예약 |
| **보상 트랜잭션 (Compensating Transaction)** | 실패 시 이전 상태로 복원하는 트랜잭션 | 역순 실행 로직 | Undo/코드 | 예약 취소 |
| **사가 로그 (Saga Log)** | 트랜잭션의 진행 상태 기록 | 각 단계의 시작/완료 이벤트 발행 | 이벤트 스토어 | 여행 일정표 |
| **코디네이터 (Orchestrator)** | 전체 사가 흐름을 제어하는 중앙 컴포넌트 | 순서 실행, 상태 관리, 보상 트리거 | Camunda, zeebe | 여행사 가이드 |
| **이벤트 버스 (Event Bus)** | 서비스 간 비동기 메시지 전송 | Pub/Sub, 메시지 큐 | Kafka, RabbitMQ | 연락 수단 |
| **트랜잭션 아웃박스 (Transactional Outbox)** | 로컬 트랜잭션과 이벤트 발행의 원자성 보장 | Outbox 테이블 + 폴러/CDC | Outbox Pattern | 영수증 기록 |
| **IDEMPOTENCE 처리** | 중복 메시지 처리 방지 | 메시지 ID 체크, 중복 제거 | 메시지 키 | 중복 방지 |
| **상태 저장소 (Saga State Store)** | 사가 실행 상태 지속 저장 | DB, Redis, ZooKeeper | 상태 기반 제어 | 진행 상태 확인 |

**[다이어그램 해설]** 오케스트레이션 사가는 중앙 코디네이터가 전체 흐름을 제어하는 방식이다. (1) 코디네이터가 주문 서비스를 호출하고 완료 이벤트를 받으면 → (2) 배송 서비스를 호출하고 완료 이벤트를 받으면 → (3) 결제 서비스를 호출하여 사가를 완료한다. 장점은 전체 흐름이 중앙에서 보이므로 이해하기 쉽고, 복잡한 비즈니스 로직을 구현하기 용이하다. 단점은 코디네이터가 SPOF(단일 장애점)가 될 수 있고, 서비스 수가 늘어나면 코디네이터가 병목(SPOF)이 발생한다는 점이다.

반면 코레오그래피 사가는 각 서비스가 독립적으로 이벤트를 발행하고, 다른 서비스의 이벤트를 구독(Subscribe)하며 자신의 로컬 트랜잭션을 실행한다. (1) 주문 서비스가 "주문 완료" 이벤트를 발행하면 → (2) 배송 서비스가 이벤트를 구독하고 로컬 트랜잭션을 실행하여 "배송 완료" 이벤트를 발행한다 → (3) 결제 서비스가 이벤트를 구독하고 로컬 트랜잭션을 실행하여 "결제 완료" 이벤트를 발행한다. 장점은 코디네이터가 없으므로 SPOF 문제가 해결되고, 서비스 간 느슨한 결합(Loose Coupling)으로 확장성이 우수하다. 단점은 전체 흐름 파악이 어려워 디버깅이 어려울 수 있고, 보상 로직이 각 서비스에 분산되어 관리가 복잡하다는 점이다.

### 코레오그래피 사가 상세 다이어그램

코레오그래피 사가의 핵심은 각 서비스가 자신의 로컬 트랜잭션을 실행하고, 완료 이벤트를 발행하며, 다른 서비스의 이벤트를 구독(Subscribe)하여 다음 단계를 진행하는 것이다. 실패 시에는 보상 트랜잭션을 실행하여 롤백한다.

```text
│           코레오그래피 사가: 주문 → 배송 → 결제                            │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ① 주문 서비스                                                         │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │ BEGIN LOCAL TRANSACTION                                          │   │
│  │ INSERT INTO order (user_id, product_id, status='PENDING')     │   │
│  │ ────────────────────────────────────────────────────────┐  │   │
│  │ │ 커밋 완료                                                     │  │   │
│  │ └───────────────────────────────────────────────────────┘  │   │
│  └──────────────────────────────────────────────────────────────┘   │
│       │                                                                │
│       ▼                                                                │
│  [이벤트 발행] kafka.emit('OrderCreated', order_id)                      │
│                                                                         │
│  ────────────────────────────────────────────────────────────────────   │
│                                                                         │
│  ② 배송 서비스 (OrderCreated 이벤트 구독)                                    │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │ BEGIN LOCAL TRANSACTION                                          │   │
│  │ UPDATE order SET status='SHIPPING'                              │   │
│  │ INSERT INTO shipment (order_id, tracking_no)                     │   │
│  │ ────────────────────────────────────────────────────────┐  │   │
│  │ │ 커밋 완료                                                     │  │   │
│  │ └───────────────────────────────────────────────────────┘  │   │
│  └──────────────────────────────────────────────────────────────┘   │
│       │                                                                │
│       ▼                                                                │
│  [이벤트 발행] kafka.emit('ShipmentCreated', shipment_id)                  │

│                                                                         │
│  ③ 결제 서비스 (ShipmentCreated 이벤트 구독)                                    │

│  │ INSERT INTO payment (order_id, amount, method='CARD')             │   │
│  │ ────────────────────────────────────────────────────────┐  │   │
│  │ │ 커밋 완료                                                     │  │   │
│  │ └───────────────────────────────────────────────────────┘  │   │
│  └──────────────────────────────────────────────────────────────┘   │
│       │                                                                │
│       ▼                                                                │
│  [이벤트 발행] kafka.emit('PaymentCompleted', payment_id)                  │
│                                                                         │
│  ────────────────────────────────────────────────────────────────────   │
│
│  [⚠️ 실패 시나리오: 결제 서비스 장애로 카드 결제 실패]                     │
│                                                                         │
│  ③ 결제 서비스 (PaymentFailed 이벤�트 구독)                                    │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │ BEGIN LOCAL TRANSACTION                                          │   │
│  │ UPDATE order SET status='PAYMENT_FAILED'                         │   │
│  │ ────────────────────────────────────────────────────────┐  │   │
│  │ │ 커밋 완료                                                     │  │   │

│  ────────────────────────────────────────────────────────────────────   │

│  ① 주문 서비스 (ShipmentCancelled 이벤트 구독) → 보상 트랜잭션                    │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │ BEGIN LOCAL TRANSACTION                                          │   │

│  │ UPDATE inventory SET quantity = quantity + 1                         │   │
│  │ ────────────────────────────────────────────────────────┐  │   │
│  │ │ 보상 트랜잭션 커밋                                            │  │   │

│       │                                                                │
│       ▼                                                                │
│  [이벤트 발행] kafka.emit('OrderCancelled', order_id)                         │

└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 코레오그래피 사가의 성공/실패 시나리오를 보여준다. (1) 주문 서비스가 로컬 트랜잭션을 커밋하고 "OrderCreated" 이벤트를 발행한다. (2) 배송 서비스가 "OrderCreated" 이벤트를 구독하고 로컬 트랜잭션(배송 정보 저장)을 커밋하고 "ShipmentCreated" 이벤트를 발행한다. (3) 결제 서비스가 "ShipmentCreated" 이벤트를 구독하고 로컬 트랜잭션(결제 처리)을 시도하는데, 장애(카드 한도 초과, 잔액 부족 등)로 실패하면 "PaymentFailed" 이벤트를 발행한다. 그러면 (2) 배송 서비스가 "PaymentFailed" 이벤트를 구독하여 보상 트랜잭션(배송 취소, 주문 상태를 "PENDING_PAYMENT"로 변경, 재고 증가 등)을 실행하고 "ShipmentCancelled" 이벤트를 발행한다. 마지막으로 (1) 주문 서비스가 "ShipmentCancelled" 이벤트를 구독하여 보상 트랜잭션(주문 취소, 재고 증가)을 실행하고 "OrderCancelled" 이벤트를 발행하여 사가를 롤백한다. 이 과정에서 각 서비스는 독립적으로 로컬 트랜잭션을 관리하고, 이벤트를 발행/구독하며 자신의 보상 로직을 수행한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 오케스트레이션 vs 코레오그래피 사가

| 비교 항목 | 오케스트레이션 (Orchestration) | 코레오그래피 (Choreography) |
|:---|:---|:---|
| **제어 방식** | 중앙 코디네이터가 제어 | 각 서비스가 자율 조정 |
| **결합 강도** | 강한 결합 (Tight Coupling) | 느슨한 결합 (Loose Coupling) |
| **SPOF 위험** | 코디네이터가 SPOF 됨 | SPOF 없음 |
| **확장성** | 낮음 (코디네이터 병목) | 높음 (서비스별 독립 확장) |
| **복잡도** | 낮음 (중앙 집중) | 높음 (로직 분산) |
| **가시성** | 쉬움 (중앙에서 흐름 파악) | 어려움 (각 서비스 흐름 추적 필요) |
| **적합 규모** | 소규 MSA (서비스 수 < 10) | 대규 MSA (서비스 수 10+) |

### 비교 2: 사가 vs 2PC vs TCC

| 비교 항목 | 사가 (Saga) | 2PC (Two-Phase Commit) | TCC (Try-Confirm-Cancel) |
|:---|:---|:---|:---|
| **동기 방식** | 비동기 (Asynchronous) | 동기 (Synchronous) | 동기 (Synchronous) |
| **Lock 방식** | Lock 없음 (또는 짧은 Lock) | 긴 Lock (Prepare~Commit) | 이중 Lock (Try~Confirm) |
| **성능** | 높음 (병렬 처리 가능) | 낮음 (순차 처리) | 중간 (2PC보다는 빠름) |
| **가용성** | 높음 (서비스별 장애 격리) | 낮음 (전체 중단) | 중간 |
| **복잡도** | 높음 (보상 로직 필요) | 낮음 (DBMS 자체 지원) | 높음 (보상 로직 필요) |
| **데이터 일관성** | 결과적 일관성 (Eventual) | 강한 일관성 (Immediate) | 강한 일관성 (Immediate) |
| **MSA 적합성** | 매우 높음 | 낮음 | 중간 |

### 과목 융합 관점

- **데이터베이스 (DB)**: 사가 패턴은 각 서비스가 독립적인 DB를 가지므로 "Database per Service" 설계와 자연스러운 결합이다. 반면 2PC는 여러 DB를 하나의 트랜잭션으로 묶으려 하므로 DB 간 Join이 빈번해져 DB Lock 경합이 심각하다.

- **네트워크 (NW)**: 코레오그래피 사가는 Kafka 같은 메시지 큐를 통해 비동기 통신을 하므로, 네트워크 지연(RTT)에 민감하지 않고 높은 처리량(Throughput)을 달성할 수 있다. 반면 2PC는 동기 통신에 의존하므로 RTT가 성능에 직접적인 영향을 미친다.

- **운영체제 (OS)**: 사가 패턴의 메시지 큐(Kafka)는 운영체제 관점에서 분산 시스템의 이벤트 기반 병렬 처리를 지원하며, 장애 복구에 강한 내구성을 가진다. 또한 폴러(Poller)나 CDC(Change Data Capture)를 통해 트랜잭션 아웃박스(Outbox) 패턴을 구현하여 로컬 트랜잭션과 이벤트 발행의 원자성을 보장한다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│         트랜잭션 아웃박스(Transactional Outbox) 패턴 구조                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  [주문 서비스]                                                     │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │ BEGIN LOCAL TRANSACTION                                          │   │
│  │   INSERT INTO order ...                                           │   │
│  │   INSERT INTO outbox (event_type, payload)                     │   │
│       │                                                            │
│       │   ┌─────────────────────────────────────────────────────┐  │   │
│       │   │ 커밋 지연 (Outbox 테이블에 먼저 기록)            │  │   │
│       │   └─────────────────────────────────────────────┘  │   │
│       │                                                            │
│       ▼                                                            │
│  [COMMIT]                                                          │
│  └──────────────────────────────────────────────────────────────┘   │
│       │                                                            │
│       ▼                                                            │
│  ┌──────────────────────────────────────────────────────┐           │
│  │ 폴러(Poller) 또는 CDC가 Outbox 테이블을 주기적으로 스캔    │   │
││  └──────┬───────────────────────────────────────────────┘           │
│         │                                                           │
│         ▼                                                           │
│  [이벤트 발행] kafka.produce('order-created', payload)               │
│                                                                         │
│  ────────────────────────────────────────────────────────────────────   │
│                                                                         │
│  [장애 시나리오: 이벤트 발행 실패]                                      │
│                                                                         │
│  ┌──────────────────────────────────────────────────────┐           │
│  │ 폴러가 Outbox 테이블을 다시 스캔                                │   │
│  │   → 이벤트 재발행 (Retry)                                         │   │
│  │   → 재시도 횟 초과 후 로그에 실패 기록                             │   │
│  └───────────────────────────────────────────────────────┘           │
│                                                                         │
│  핵심: 로컬 트랜잭션 커밋과 Outbox 이벤트 기록이 원자적                 │
│       → 이벤트 발행 실패 시 폴러가 재시도하여 중단 없는 발행 보장        │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 트랜잭션 아웃박스(Outbox) 패턴은 로컬 트랜잭션과 이벤트 발행 사이의 원자성을 보장하는 핵심 패턴이다. (1) 서비스가 로컬 트랜잭션을 시작하여 비즈니스 로직을 수행(INSERT INTO order)하고, 동일 트랜잭션 내에서 Outbox 테이블에 이벤트를 기록한다. (2) 로컬 트랜잭션이 커밋되면 Outbox 테이블에도 이벤트가 원자적으로 기록되므로 "커밋 지연" 상태가 된다. (3) 폴러(Poller)나 CDC(Change Data Capture)가 주기적으로 Outbox 테이블을 스캔하여 아직 발행되지 않은 이벤트를 찾아 Kafka로 발행한다. (4) 발행 성공 후 Outbox 테이블에서 해당 레코드를 삭제한다. 장애 발생 시에도 폴러가 재시도(Retry)를 계속 수행하므로 "중단 없는 발행"이 보장된다. 이 패턴이 없으면 로컬 트랜잭션은 커밋되었는데 이벤트 발행이 실패하여 데이터 불일치가 발생한다.

### 📢 섹션 요약 비유

마치 "우편물을 보내고 영수증을 받은 후에야 배송을 완료"하는 우편 시스템처럼, 트랜잭션 아웃박스는 "영수증(Outbox 테이블 기록)"을 확보한 후에 배송(이벤트 발행)을 완료하여 안정성을 보장합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 전자상거래 주문/결제 분산 트랜잭션**: 전자상거래 플랫폼에서 "주문 서비스", "재고 서비스", "배송 서비스", "결제 PG"가 분리된 MSA 환경에서 코레오그래피 사가 도입. (결과) 결제 PG 장애 시 자동 재시도로 가용성 99.9% 달성, 보상 트랜잭션 레턴시 3초 이내 롤백 완료로 고객 불만 최소화. 이벤트 기반 트랜잭션 추적으로 가능하여 재무 정합성 확보.

2. **시나리오 — 항공권 예약/발권 분산 트랜잭션**: 항공사가 여러 항공사의 API를 연동하여 실시간 좌석 예약 시스템 구축. (결과) 오케스트레이션 사가에서 코레오그래피 사가로 전환하여 SPOF 해소, 각 항공사 장애 시 독립적으로 보상 트랜잭션 실행으로 전체 가용성 99.95% 달성. Kafka 기반 이벤트 버스로 실시간 좌석 현황 공유.

3. **시나리오 — 핀 fintech 송금 이체 분산 트랜잭션**: 핀테크 스�트업에서 계좌 이체 시 "출금 서비스", "입금 서비스", "외환 서비스", "거래 내역" 분산 처리. (결과) 사가 패턴 + 트랜잭션 아웃박스로 ACID 보장하며, 이벤트 소싱으로 모든 트랜잭션 내역 추적 가능하여 감사 준비 시간 80% 단축. 재무 정합성 100% 보장으로 금융 규제 준수.

### 도입 체크리스트

- **기술적**:
  - 코레오그래피 사가를 우선 고려하여 SPOF 문제 해결할 것
  - 보상 트랜잭션의 IDEMPOTENCE(멱등 방지) 처리를 보장할 것
  - 트랜잭션 아웃박스 패턴으로 로컬 트랜잭션과 이벤트 발행의 원자성을 보장할 것
  - 이벤트 소싱과 결합하여 완전한 트랜잭션 내역 추적 기능을 구현할 것

- **운영·보안적**:
  - 각 서비스의 보상 트랜잭션 로직을 문서화하여 운영자가 이해하기 쉽게 할 것
  - 트랜잭션 실패 시 알림(Alert) 메커니즘을 구축하여 신속 대응 가능하게 할 것
  - 사가 상태를 지속적으로 모니터링하고, 중단 상태(Held) 트랜잭션이 없도록 할 것
  - 감사(Audit) 요구사항에 맞춰 모든 트랜잭션 내역을 추적 가능하게 할 것

- **조직적**:
  - 각 서비스 팀이 독립적으로 사가 로직을 개발하고 운영할 수 있는 문화를 조성할 것
  - 사가 패턴 교육 및 실습을 통해 팀의 역량을 강화할 것
  - 장애 시 복구 절차(Recovery Playbook)을 마련하여 신속한 대응이 가능하게 할 것

### 안티패턴

- **오케스트레이션 사가의 과도 사용**: 코디네이터가 SPOF가 되어 시스템 전체의 병목(SPOF)을 초래하는 것은 사가 패턴의 핵심 장점인 서비스 독립성을 저해한다.

- **보상 트랜잭션 미구현**: "보상 트랜잭션은 필요 없다"고 생각하고 구현하지 않으면, 장애 발생 시 데이터 불일치가 영구화된다.

- **IDEMPOTENCE 처리 누�**: 동일한 이벤트가 중복 발행하여 보상 트랜잭션이 두 번 실행되어 데이터가 중복되는 문제가 발생할 수 있다.

- **이벤트 순서 의존**: 코레오그래피 사가에서 이벤트 처리 순서에 오류가 있으면 전체 트랜잭션이 실패할 수 있다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                오케스트레이션 사가의 SPOF 문제                             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  [서비스 A]    [서비스 B]    [서비스 C]    [코디네이터]                │
│      │            │             │        │                            │
│      ├────────┴────────────┴─────────┤                            │
│                           │                                        │
│                           ▼                                        │
│  ┌──────────────────────────────────────────────────────────┐       │
│  │                 코디네이터 (SPOF)                       │       │
│  │                 • 모든 사가 요청 경유                  │       │
│  │                 • 단일 장애점 (SPOF)                      │       │
│  │                 • 병목(Bottleneck) 발생               │       │
│  │                 • 확장성 저하                             │       │
│  └──────────────────────────────────────────────────────────┘       │
│                                                                         │
│  ⚠️ 문제점:                                                          │
│  • 코디네이터 장애 시 전체 서비스 중단                             │
│  • 사가 요청이 몰리면 코디네이터 리소스 부족                       │
│  • 각 서비스가 코디네이터를 호출해야 하므로 결합도 증가               │
│  • MSA의 핵심 목표인 "서비스 독립성" 저해                           │
│                                                                         │
│  ✅ 코레오그래피 사가로 전환:                                         │
│  • 각 서비스가 독립적으로 이벤트를 발행/구독                         │
│  • 코디네이터 없으므로 SPOF 문제 해결                                 │
│  • 서비스별 독립 배포/확장 가능                                       │
│  • 서비스 간 느승한 결합(Loose Coupling)으로 MSA 철학에 부합            │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 오케스트레이션 사가의 핵심 문제는 코디네이터가 SPOF(Single Point of Failure)가 된다는 점이다. 모든 사가 요청이 코디네이터를 경유하므로, 코디네이터 장애 시 전체 서비스가 중단되고, 사가 요청이 몰리면 코디네이터의 리소스(CPU, 메모리)가 부족하여 병목(Bottleneck)이 발생한다. 또한 각 서비스가 코디네이터를 호출해야 하므로 서비스 간 결합도(Tight Coupling)이 강해져 MSA의 핵심 목표인 "서비스 독립성"이 저해된다. 반면 코레오그래피 사가는 각 서비스가 독립적으로 이벤트를 발행/구독하므로 코디네이터가 필요 없으며, 서비스별 독립 배포/확장이 가능하여 MSA의 철학에 부합한다.

### 📢 섹션 요약 비유

마치 여행사 가이드가 전체 일정을 관리하면(오케스트레이션), 한 명이라도 늦어지면 전체가 취소되는 반면, 각자 여행자가 자율적으로 움직이고 문제 시 본인이 알아서 되돌아오는 방식(코레오그래피)으로 바꾸면 가이드가 없어도 여행을 계속할 수 있습니다.

### 미래 전망

- **오토마이즈드 사가(Automated Saga)**: AWS Step Functions, Temporal, Netflix Conductor 등이 오토마이즈드 사가 관리 프레임워크를 제공하여, 개발자가 비즈니스 로직에 집중할 수 있도록 지원할 것이다.

- **서비스 메시(Service Mesh)와의 결합: Istio, Linkerd 같은 서비스 메시가 사가 패턴의 트랜잭션 조정, 재시도(Retry), 회로 제어(Circuit Breaker)를 MESH 레벨에서 자동으로 제공하여, 개발자가 이를 직접 구현하지 않아도 되도록 할 것이다.

- **이벤트 기반 사가(Event-Driven Saga)의 표준화**: CloudEvents, Kafka Streams, Pulsar 등의 이벤트 표준화가 확대되어, 서비스 간 인터페이스가 더욱 간편해질 것이다.

- **관찰 가능성(Observability) 강화: 분산 트랜잭션의 실행 내역을 추적하고 시각화하는 도구가 발전하여, 장애 원인 분석과 성능 튜닝이 용이해질 것이다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **2PC (Two-Phase Commit)** | 사가 패턴과 대비되는 강한 일관성 기반 분산 트랜잭션 프로토콜. 블로킹/단일 장애점 문제로 MSA에 부적합 |
| **결과적 일관성 (Eventual Consistency)** | 데이터가 즉시 일관되지 않아도 시간이 지나면 최종적으로 일관되는 성질. 사가 패턴의 핵심 철학 |
| **트랜잭션 아웃박스 (Transactional Outbox)** | 로컬 트랜잭션 커밋과 이벤트 발행 사이의 원자성을 보장하는 패턴. 사가 패턴의 안정성 강화에 필수 |
| **이벤트 소싱 (Event Sourcing)** | 상태(State)가 아니라 상태를 변경시킨 '이벤트 로그'를 저장하여 완전한 감사/복구 인프라 구현. CQRS와 결합 |
| **CQRS (Command Query Responsibility Segregation)** | 상태 변경(Command) 서비스와 조회(Query) 서비스를 물리/논리적으로 분리하여 조회 성능 최적화 |
| **CAP 정리** | 일관성(C), 가용성(A), 분할 내성(P) 중 2가지만 선택 가능. 사가는 AP(가용성+분할 내성) 선택 |
| **IDEMPOTENCE** | 동일 요청이 여러 번 실행되어도 결과가 중복되지 않도록 하는 보안 조치. 분산 환경 필수 |
| **재시도(Retry)** | 장애 발생 시 실패한 작업을 다시 시도하는 메커니즘. 사가 패턴의 안정성 강화에 활용 |
| **회로 제어(Circuit Breaker)** | 연속 실패 시 호출을 조기 차단(SPOF 연� 방지)하여 시스템 전체의 연� 장애(Cascading Failure) 방지 |
| **메시지 큐 (Message Queue)** | 비동기 메시징 전송을 위한 인프라. Kafka, RabbitMQ, ActiveMQ 등. 사가 패턴의 기반 |
