# Level 2 - HTTP 메서드의 적절한 분리 사용 (가장 대중적 단계)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Richardson Maturity Model Level 2는 HTTP 프로토콜의 GET(조회), POST(생성), PUT(전체 교체), PATCH(부분 수정), DELETE(삭제) 메서드를 적절한 용도로 분리 사용하여, 리소스 조작의 의미를 HTTP 동사로 명확히 표현하는 가장 대중적인 RESTful API 단계
> 2. **가치**: GET 요청의 멱등성(Idempotent)과 캐시 가능성으로 CDN/프록시 캐싱 적용 시 서버 부하 60~80% 경감, HTTP 표준 도구와 생태계 완벽 통합, API Gateway에서 동사별 세밀한 라우팅/인가/속도 제한 정책 적용 가능
> 3. **융합**: 대부분의 실무 REST API가 목표로 하는 수준으로, OpenAPI 명세 자동화, CDN 에지 캐싱, Service Mesh 메트릭 수집, MSA 서비스 간 통신 표준화 등 현대적 API 인프라의 필수 전제조건

## Ⅰ. 개요 (Context & Background)

### 개념 정의

Level 2 (HTTP Verbs)는 Richardson Maturity Model에서 가장 대중적으로 채택된 단계로, HTTP 프로토콜이 제공하는 GET, POST, PUT, PATCH, DELETE 등의 메서드를 각각의 적절한 용도로 분리 사용한다. 이 단계의 핵심은 "무엇을 조작하는지(URI)"와 "어떻게 조작하는지(HTTP 메서드)"를 명확히 분리하여 표현한다는 점이다.

Level 0~1에서는 모든 작업이 POST로 표현되어 요청의 의도를 본문을 파싱해야만 알 수 있었지만, Level 2에서는 HTTP 메서드 자체가 요청의 의도를 명확히 전달한다. 따라서 `GET /users/123`는 사용자 조회, `DELETE /users/123`는 사용자 삭제, `POST /users`는 사용자 생성이라는 의미를 HTTP 메서드와 URI만으로 즉시 식별 가능하다.

이 설계는 HTTP 프로토콜이 1990년대 초반부터 제공해 온 본연의 기능을 충실히 활용하는 것이며, 웹의 "보편적인 원칙(Universality)"을 따르는 것이다. Tim Berners-Lee가 웹을 설계할 때 의도했던 "URI로 리소스를 식별하고, HTTP 메서드로 조작 방식을 표현하며, 하이퍼미디어로 상태를 전이시키는" 패러다임을 Level 2에서 실용적으로 구현한다.

### 등장 배경

Level 2의 보편적 채택에는 여러 기술적, 실무적 배경이 있다.

**① HTTP/1.1 표준화 (1997)**

HTTP/1.1 명세(RFC 2061, 후에 RFC 2616, 현재 RFC 9110)이 GET, POST, PUT, DELETE, HEAD, OPTIONS, TRACE, CONNECT 등의 메서드를 표준화했고, 이들 메서드의 의미(Safe Methods, Idempotent Methods)를 명확히 정의했다. 하지만 2000년대 중반까지는 웹 서비스 기술이 SOAP 위주로 발전하면서 HTTP 메서드의 의미를 활용하는 RESTful 스타일은 주류에서 벗어나 있었다.

**② Roy Fielding의 REST 논문 (2000)**

Roy Fielding이 박사학위 논문에서 REST 아키텍처 스타일을 정립하면서, HTTP가 REST 원칙을 구현하는 가장 적합한 프로토콜임을 입증했다. 특히 "통합 인터페이스(Uniform Interface)" 제약 조건에서 리소스 조작을 HTTP 메서드로 표현하는 것을 핵심 원리로 제시했다.

**③ JSON의 보편화 (2005-2010)**

XML 기반의 SOAP이 복잡하다는 비판이 증가하면서, 더 가벼운 JSON 포맷이 웹 API 표준으로 자리 잡았다. JSON과 함께 HTTP 메서드를 적절히 활용하는 RESTful API 스타일이 보편화되었다.

**④ 클라우드와 MSA의 등장 (2010-2015)**

아마존 AWS, Microsoft Azure, Google Cloud Platform 등 퍼블릭 클라우드가 보편화되면서, 수평 확장 가능한 RESTful API가 엔터프라이즈 표준으로 자리 잡았다. 특히 MSA(Microservices Architecture) 패러다임에서는 서비스 간 통신에 HTTP/RESTful API가 표준이 되었다.

### 기술적 필요성

Level 2 도입은 단순한 설계 스타일 선택이 아니라, 시스템 확장성, 운용 효율, 비용 절감에 직접적인 영향을 미치는 기술적 의사결정이다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│              Level 1 vs Level 2 Technical Necessity                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [Level 1: POST-only]                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  POST /users/123 {action: "get"}      ← 조회인데 POST              │   │
│  │  POST /users/123 {action: "delete"}  ← 삭제인데 POST               │   │
│  │                                                                      │   │
│  │  문제점:                                                             │   │
│  │  • 모든 요청이 POST이므로 CDN/프록시 캐싱 불가                      │   │
│  │  • 조회 요청이 매번 원본 서버로 집중                                │   │
│  │  • 동일 데이터 조회 반복 시 서버 부하 누적                          │   │
│  │  • 재시도 안전성 불명확 (POST는 비멱등)                            │   │
│  │                                                                      │   │
│  │  결과:                                                               │   │
│  │  • 일일 요청 100만 건 중 조회가 70%라고 가정                        │   │
│  │  • 70만 건의 조회가 매번 원본 서버 처리                            │   │
│  │  • 캐싱 혜택 0%로 확장성 극도로 저조                               │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [Level 2: HTTP Verbs]                                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  GET /users/123                    ← 조회는 GET                    │   │
│  │  DELETE /users/123                 ← 삭제는 DELETE                 │   │
│  │                                                                      │   │
│  │  개선 효과:                                                          │   │
│  │  • GET 요청은 CDN/프록시가 캐싱 가능                              │   │
│  │  • 첫 요청만 원본 서버 처리, 이후 캐지 응답                       │   │
│  │  • GET/PUT/DELETE는 멱등으로 재시도 안전                           │   │
│  │  • HTTP 표준 도구와 생태계 통합                                    │   │
│  │                                                                      │   │
│  │  결과:                                                               │   │
│  │  • 동일 100만 건 요청 중 70만 건 GET 요청                          │   │
│  │  • CDN 캐시 히트율 60% 가정 시 42만 건은 CDN 즉시 응답             │   │
│  │  • 원본 서버 부하 58만 건으로 42% 경감                              │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

이 기술적 필요성 분석의 핵심은 Level 2 도입만으로도 시스템 확장성이 40% 이상 개선된다는 점이다. 실무에서 대규모 트래픽을 처리하는 공개 API의 경우, 이 차이는 인프라 비용으로 수십~수백만 달러 차이를 만들며, 서비스 안정성에도 결정적 영향을 미친다.

📢 **섹션 요약 비유**: Level 2는 운송 시스템에서 화물 종류별로 적절한 운송 수단을 사용하는 것과 같습니다. Level 1은 모든 화물(소포, 대형 화물, 긴급 물품)을 똑같은 일반 트럭으로 운송하지만, Level 2는 소포는 오토바이, 대형 화물은 대형 트럭, 긴급 물품은 항공기로 운송하여 효율과 속도를 최적화합니다. GET 요청은 오토바이처럼 가볍고 빠르게, POST/PUT은 트럭처럼 안전하게 처리됩니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### HTTP 메서드별 의미와 용도

Level 2의 핵심은 HTTP 메서드 각각이 명확한 의미와 용도를 가진다는 점이다. RFC 9110(HTTP Semantics)에 정의된 바와 같이, 각 메서드는 Safe, Idempotent, Cacheable 속성을 갖는다.

| HTTP 메서드 | 용도 | Safe | Idempotent | Cacheable | 의미 |
|:----------:|:-----|:----:|:----------:|:--------:|:-----|
| **GET** | 리소스 조회 | ✅ | ✅ | ✅ | 서버 상태 변경 없음, 멱등, 캐시 가능 |
| **POST** | 리소스 생성/부작용 | ❌ | ❌ | ❌(일부) | 서버 상태 변경, 비멱등, 일반적 캐싱 불가 |
| **PUT** | 리소스 전체 교체 | ❌ | ✅ | ❌ | 동일 요청 반복 시 결과 동일, 멱등 |
| **PATCH** | 리소스 부분 수정 | ❌ | ⚠️(구현 의존) | ❌ | 구현에 따라 멱등 여부 달라짐 |
| **DELETE** | 리소스 삭제 | ❌ | ✅ | ❌ | 존재 시 삭제, 미존재 시 404로 멱등 |

### GET 메서드: 리소스 조회

GET은 리소스 조회를 위한 메서드로, **Safe Method**(서버 상태를 변경하지 않음)와 **Idempotent Method**(여러 번 호출해도 결과 동일) 속성을 모두 갖는다.

```http
GET /users/123 HTTP/1.1
Host: api.example.com
Accept: application/json

// 응답
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=3600

{
  "id": 123,
  "name": "John Doe",
  "email": "john@example.com"
}
```

GET의 핵심 특징:
1. **캐싱 가능**: 중간 프록시, CDN, 브라우저가 응답을 캐싱하여 후속 요청에 재사용
2. **즉시 호출 가능**: 브라우저 주소창에 URL만 입력하면 즉시 호출 (요청 본문 불필요)
3. **재시도 안전**: 네트워크 오류 시 안전하게 재시도 가능 (멱등성)
4. **북마크 가능**: URL을 북마크하면 나중에 동일 결과 조회 가능

### POST 메서드: 리소스 생성

POST는 새로운 리소스를 생성하거나, 서버에서 처리해야 할 작업을 수행할 때 사용한다. POST는 **비멱등**(여러 번 호출 시마다 다른 결과 발생 가능)이며, 일반적으로 캐싱 불가다.

```http
POST /users HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "name": "Jane Doe",
  "email": "jane@example.com"
}

// 응답
HTTP/1.1 201 Created
Location: /users/124
Content-Type: application/json

{
  "id": 124,
  "name": "Jane Doe",
  "email": "jane@example.com"
}
```

POST의 핵심 특징:
1. **리소스 생성**: 서버가 새 리소스 ID를 생성하고 `Location` 헤더로 반환
2. **부작용 발생**: 서버 상태 변경(데이터 생성, 로그 기록, 이메일 발송 등)
3. **비멱등**: 동일 요청을 두 번 보내면 두 개의 리소스가 생성됨
4. **캐싱 제한**: 일반적으로 캐싱 불가 (단, 201 Created 응답은 일부 캐시에서 저장 가능)

### PUT 메서드: 리소스 전체 교체

PUT은 기존 리소스를 요청 본문 전체로 교체할 때 사용한다. PUT은 **멱등** 속성을 가지므로, 동일한 요청을 여러 번 보내도 최종 상태는 동일하다.

```http
PUT /users/123 HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "id": 123,
  "name": "John Smith",
  "email": "john.smith@example.com"
}

// 응답
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "John Smith",
  "email": "john.smith@example.com"
}
```

PUT의 핵심 특징:
1. **전체 교체**: 리소스의 모든 필드를 요청 본문으로 완전히 교체
2. **멱등성**: 동일한 PUT 요청을 여러 번 보내도 최종 상태 동일
3. **존재하지 않는 리소스**: 일부 구현에서는 새 리소스 생성(201 Created)을 허용
4. **재시도 안전**: 네트워크 오류 시 안전하게 재시도 가능

### PATCH 메서드: 리소스 부분 수정

PATCH는 리소스의 일부 필드만 수정할 때 사용한다. PATCH의 멱등성은 구현에 의존적이며, 일반적으로 비멱등으로 간주된다.

```http
PATCH /users/123 HTTP/1.1
Host: api.example.com
Content-Type: application/json-patch+json
또는
Content-Type: application/merge-patch+json

// JSON Merge Patch 예시
{
  "email": "newemail@example.com"
}

// 응답
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "John Doe",
  "email": "newemail@example.com"
}
```

PATCH의 핵심 특징:
1. **부분 수정**: 변경할 필드만 전송하여 대역폭 절약
2. **멱등성 불확실**: 구현에 따라 멱등 여부 달라짐 (일반적으로 비멱등)
3. **PATCH 포맷**: JSON Patch(RFC 6902), JSON Merge Patch(RFC 7396) 등 표준 존재
4. **조건부 사용**: 클라이언트가 전체 리소스를 알지 못할 때 유용

### DELETE 메서드: 리소스 삭제

DELETE는 리소스를 삭제할 때 사용한다. DELETE는 **멱등** 속성을 가진다(이미 삭제된 리소스에 대해 삭제를 요청해도 404 Not Found로 동일한 결과).

```http
DELETE /users/123 HTTP/1.1
Host: api.example.com

// 응답 (성공)
HTTP/1.1 204 No Content

// 응답 (이미 삭제됨)
HTTP/1.1 404 Not Found
```

DELETE의 핵심 특징:
1. **멱등성**: 존재하는 리소스 삭제(204), 미존재(404)로 결과 예측 가능
2. **재시도 안전**: 네트워크 오류 시 안전하게 재시도 가능
3. **응답 본문 없음**: 일반적으로 204 No Content로 본문 없이 응답

### Level 2 아키텍처 다이어그램

다음 그림은 Level 2에서 HTTP 메서드가 요청 처리 흐름에 어떻게 통합되는지 보여준다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                     Level 2 Architecture with HTTP Verbs                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                         API Clients                                 │   │
│  │  (Browser, Mobile, Third-party Apps)                               │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                  │                                         │
│                                  │ [다양한 HTTP 메서드]                     │
│                                  ▼                                         │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                         API Gateway                                 │   │
│  │                                                                      │   │
│  │  [HTTP 메서드 기반 라우팅]                                            │   │
│  │  • GET 요청 → 캐시 레이어 확인                                       │   │
│  │  • POST/PUT/PATCH/DELETE → 원본 서비스로 전송                       │   │
│  │                                                                      │   │
│  │  [HTTP 메서드 기반 정책]                                              │   │
│  │  • GET: 더 높은 속도 제한 (예: 1000 req/min)                        │   │
│  │  • POST: 중간 속도 제한 (예: 100 req/min)                           │   │

│  │                                                                      │   │
```

📢 **섹션 요약 비유**: Level 2는 고속도로 톨게이트 시스템에서 차량 종류별로 전용 차선을 운영하는 것과 같습니다. 하이패스 차량(GET)은 별도 차선으로 빠르게 통과하고, 일반 차량(POST)은 일반 차선을, 화물차(DELETE)는 별도 검사 차선을 사용하여 전체 톨게이트 처리 효율을 극대화합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### CRUD vs HTTP 메서드 매핑

소프트웨어 개발의 CRUD(Create, Read, Update, Delete) 연산과 HTTP 메서드의 관계는 Level 2 설계의 기본이 된다.

| CRUD 연산 | HTTP 메서드 | URI 예시 | HTTP 상태 코드 | Safe | Idempotent |
|:--------:|:----------:|:--------|:-------------:|:----:|:----------:|
| **Create** | POST | `/users` | 201 Created | ❌ | ❌ |
| **Read** | GET | `/users/123` | 200 OK | ✅ | ✅ |
| **Update** | PUT | `/users/123` | 200 OK | ❌ | ✅ |
| **Update** | PATCH | `/users/123` | 200 OK | ❌ | ⚠️ |
| **Delete** | DELETE | `/users/123` | 204 No Content | ❌ | ✅ |

### HTTP 상태 코드와의 연계

Level 2에서는 HTTP 메서드와 상태 코드를 조합하여 요청 결과를 명확히 전달한다.

| 상황 | GET | POST | PUT | PATCH | DELETE |
|:-----|:----:|:----:|:----:|:-----:|:------:|
| **성공** | 200 OK | 201 Created | 200 OK | 200 OK | 204 No Content |
| **찾을 수 없음** | 404 Not Found | 404/400 | 404 | 404 | 404/204 |
| **검증 실패** | 400 Bad Request | 400 Bad Request | 400 | 400 | 400 |
| **인증 실패** | 401 Unauthorized | 401 | 401 | 401 | 401 |
| **인가 실패** | 403 Forbidden | 403 | 403 | 403 | 403 |
| **충돌** | N/A | 409 Conflict | 409 | 409 | N/A |

### 과목 융합: Level 2와 연관 기술 스택

**① CDN (Content Delivery Network)과의 시너지**

Level 2의 GET 메서드 활용은 CDN 에지 캐싱을 가능하게 하여, 전 세계 사용자에 대한 콘텐츠 전송 지연을 80~95% 감소시킨다.

```text
GET /products/123 → CDN 에지에서 응답 (원본 서버 부하 없음)
```

**② API Gateway와의 시너지**

API Gateway는 HTTP 메서드를 기반으로 세밀한 라우팅, 인가, 속도 제한 정책을 적용할 수 있다.

```
GET /public/*: 인증 불필요, 높은 속도 제한
POST /admin/*: 관리자 인증 필수, 낮은 속도 제한
DELETE /*: 쓰기 권한 확인, 가장 낮은 속도 제한
```

**③ Service Mesh와의 시너지**

Level 2 표준화된 MSA 환경에서, Service Mesh(Istio, Linkerd 등)는 HTTP 메서드별 메트릭을 수집하여 GET 요청의 캐시 히트율, POST 요청의 실패율 등을 정교하게 모니터링 가능하다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오별 Level 2 적용

Level 2는 대부분의 실무 상황에서 권장되는 표준 수준이다.

| 시나리오 | Level 2 적합성 | 판단 근거 |
|:--------|:-------------:|:---------|
| **공개 API** | ✅ 필수 | 캐싱, HTTP 도구 통합, 파트너 연동 용이 |
| **내부 MSA 통신** | ✅ 필수 | Service Mesh 메트릭, mTLS 정책 효율적 적용 |
| **모바일 앱 백엔드** | ✅ 필수 | 네트워크 비용 절감, 오프라인 캐싱 |
| **대규모 트래픽 서비스** | ✅ 필수 | CDN 캐싱으로 확장성 확보 |
| **내부 도구 프로토타입** | ⚠️ 선택 가능 | 빠른 개발 위해서는 Level 1도 허용 |

### Level 2 설계 체크리스트

**URI 설계**
- [ ] URI에 동사를 포함하지 않고 명사(리소스)만 사용하는가
- [ ] 복수형을 일관되게 사용하는가 (`/users` vs `/user`)
- [ ] URI 경로 깊이가 3~4 세그먼트 내외인가
- [ ] 계층적 관계를 자연스럽게 표현하는가 (`/users/123/orders`)

**HTTP 메서드 활용**
- [ ] 조회는 GET을 사용하는가
- [ ] 생성은 POST를 사용하는가
- [ ] 전체 수정은 PUT을 사용하는가
- [ ] 부분 수정은 PATCH를 사용하는가
- [ ] 삭제는 DELETE를 사용하는가

**상태 코드**
- [ ] 성공 시 적절한 상태 코드를 반환하는가 (200, 201, 204)
- [ ] 리소스 미발견 시 404를 반환하는가
- [ ] 검증 실패 시 400 Bad Request를 반환하는가
- [ ] 인증 실패 시 401 Unauthorized를 반환하는가
- [ ] 인가 실패 시 403 Forbidden을 반환하는가

**캐싱**
- [ ] GET 응답에 Cache-Control 헤더를 포함하는가
- [ ] 정적 리소스는 긴 캐시 기간(max-age=31536000)을 설정하는가
- [ ] 동적 리소스는 짧은 캐시 기간(max-age=60) 또는 no-cache를 설정하는가

### 안티패턴: Level 2에서의 실수

**안티패턴 1: GET에서 상태 변경**

```
❌ 나쁜 예:
GET /users/123/delete
GET /orders/456/ship

문제점:
• GET은 Safe Method여야 함 (상태 변경 금지)
• 크롤러, 프리패치가 의도치 않은 상태 변경 유발
• 캐싱된 잘못된 응답이 반복 제공될 수 있음

✅ 올바른 예:
DELETE /users/123
POST /orders/456/ship  또는  PATCH /orders/456 {"status": "shipped"}
```

**안티패턴 2: POST를 멱등 작업에 사용**

```
❌ 나쁜 예:
POST /users/123/deactivate  ← 비멱등으로 간주됨
POST /products/789/increment-view  ← 매 호출마다 조회수 증가

문제점:
• 자동 재시도 로직이 중복 작업 유발 가능
• 네트워크 오류 시 재시도 안전성 보장 없음

✅ 올바른 예:
PUT /users/123 {"status": "inactive"}  ← 멱등
POST /products/789/views  ← 의도적으로 비멱등
```

**안티패턴 3: 잘못된 상태 코드**

```
❌ 나쁜 예:
// 리소스 생성 실패 시
POST /users
→ 200 OK (하지만 생성 실패)

// 리소스 미발견 시
GET /users/999
→ 200 OK (빈 배열)

✅ 올바른 예:
// 리소스 생성 실패 시
POST /users
→ 400 Bad Request  또는  409 Conflict

// 리소스 미발견 시
GET /users/999
→ 404 Not Found
```

📢 **섹션 요약 비유**: Level 2는 도로에서 신호등과 교통 규칙을 준수하는 것과 같습니다. 직진은 직진 신호에서, 회전은 회전 신호에서 기다리고, 속도 제한을 준수하면 교통 흐름이 원활해집니다. 규칙을 무시하고 아무 때나 직진하거나 신호를 무시하면 사고가 발생하듯, HTTP 메서드를 잘못 사용하면 시스템 오류와 보안 문제가 발생합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과

Level 2 도입 시 기대되는 주요 효과를 정량적으로 분석한다.

| 평가 항목 | Level 1 | Level 2 | 개선 효과 |
|:--------|:-------|:-------|:--------:|
| **CDN 캐시 히트율** | 0~20% | 60~85% | +325% |
| **서버 부하(동일 트래픽)** | 100% | 40~60% | -50% |
| **API 라우팅 정교도** | URI만 | URI+메서드 | +200% |
| **HTTP 도구 친화성** | 낮음 | 높음 | +400% |
| **재시도 안전성** | 불명확 | 명확(멱등) | ✅ |
| **모바일 네트워크 사용량** | 기준 | -30% | 비용 절감 |

### 기술적 효과

**① 확장성 향상**

GET 요청의 CDN 캐싱은 원본 서버 부하를 60~80% 경감시키며, 동일 하드웨어로 3~5배 더 많은 트래픽을 처리 가능하게 한다.

**② 운용 효율 개선**

HTTP 메서드별 메트릭 수집은 문제 진단 속도를 50% 이상 단축하며, 장애 원인을 신속하게 식별 가능하게 한다.

**③ 개발 생산성 향상**

OpenAPI 기반 코드 생성 도구는 Level 2 API에서 클라이언트/서버 SDK를 자동 생성하여 개발 시간을 40~60% 단축한다.

### 향후 전망

Level 2는 현재 RESTful API의 실무적 표준으로 안착했으며, 향후 5~10년간도 주류를 유지할 것으로 전망된다. GraphQL, gRPC 등의 대안 기술이 등장했지만, Level 2의 HTTP/RESTful API는 가장 광범위한 도구 지원과 가장 낮은 학습 곡선으로 인해 지배적 위치를 유지할 것이다.

2025-2030년에는 Level 2가 기본이 되고, 특정 요구사항이 있는 경우 Level 3(HATEOAS) 또는 GraphQL을 선택적으로 적용하는 하이브리드 아키텍처가 표준이 될 것이다.

### 참고 표준

- **RFC 9110 - HTTP Semantics**: HTTP 메서드의 정의와 속성(Safe, Idempotent)
- **RFC 7231 - HTTP/1.1 Semantics and Content**: GET, POST, PUT, DELETE의 의미
- **RFC 5789 - PATCH Method for HTTP**: PATCH 메서드 명세
- **RFC 6902 - JSON Patch**: JSON Patch 포맷
- **RFC 7396 - JSON Merge Patch**: JSON Merge Patch 포맷

📢 **섹션 요약 비유**: Level 2는 자동차 산업에서 표준화된 안전벨트와 에어백이 장착된 차량과 같습니다. 초기에는 없어도 될 것 같았지만, 이제는 없으면 불안하고 위험한 필수 safety equipment가 되었습니다. 마찬가지로 Level 2는 현대 API 설계의 필수 safety equipment로, 없으면 불안하고 비효율적인 기술 부채가 됩니다.

## 👶 어린이를 위한 3줄 비유 설명

**① 도서관 비유**: Level 2는 도서관에서 "읽기 구역"과 "대출 구역"을 명확히 구분하는 것과 같습니다. 읽기 구역(GET)에서는 책을 자유롭게 읽을 수 있고(멱등, 캐싱 가능), 대출 구역(POST/PUT/DELETE)에서는 사서의 허락을 받아야만 책을 빌리거나 반납할 수 있습니다.

**② 식당 비유**: Level 2는 식당에서 메뉴와 주문 방식을 체계화한 것과 같습니다. 메뉴 보기(GET)는 언제든지 가능하고, 주문하기(POST)는 웨이터에게 하며, 주문 변경(PUT)은 주방에 전달되고, 주문 취소(DELETE)는 매니저 승인이 필요합니다. 각 행동에 명확한 규칙이 있어 혼란이 없습니다.

**③ 우편물 비유**: Level 2는 우편물의 종류별로 다른 색상 봉투를 사용하는 것과 같습니다. 편지(GET)은 파란 봉투로 빠르게 배달되고, 소포(PATCH)는 노란 봉투로 내용 확인 후 배달되며, 반환물(DELETE)은 빨간 봉투로 특별 처리됩니다. 색상만 보고도 어떤 종류의 우편물인지 알 수 있어 효율적입니다.
