# Level 0 - 단일 URI, 단일 POST 메서드 (RPC 스타일)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Richardson Maturity Model의 가장 미성숙한 단계로, HTTP를 단순한 전송 계층(TCP 소켓과 유사)으로만 활용하고 단일 URI(`/api`)와 단일 메서드(POST)로 모든 작업을 처리하는 RPC (Remote Procedure Call) 스타일 통신 패턴
> 2. **가치**: REST 원칙을 전혀 준수하지 않아 캐싱 불가, HTTP 표준 도구 활용 불가, 확장성 극히 저조하며, 대규모 트래픽 환경에서 치명적 병목 유발 → 반면 개발 초기 단계에서는 가장 빠르게 구현 가능하여 프로토타이핑에는 유리
> 3. **융합**: 레거시 시스템에서 SOAP/WS-*에서 REST로 마이그레이션하는 과도기적 단계로 발생하며, Level 1 이상으로 진화 시 CDN 캐싱, API Gateway 라우팅, Service Mesh 관측 가능성 등 인프라 혜택을 누릴 수 있음

## Ⅰ. 개요 (Context & Background)

### 개념 정의

Level 0(The Swamp of POX)는 Richardson Maturity Model에서 가장 미성숙한 단계로, HTTP 프로토콜을 완전히 활용하지 않고 단일 URI 엔드포인트와 POST 메서드만으로 모든 API 요청을 처리하는 패턴이다. 이 패턴은 기술적으로는 RESTful이라 부를 수 없으며, RPC 스타일의 원격 프로시저 호출을 HTTP 위에 얇게 감싼 것에 불과하다. 모든 요청이 동일한 URI(예: `http://api.example.com/service`, `http://api.example.com/api`)로 전송되며, 요청 본문(Request Body)에 포함된 액션 이름(`action`, `method`, `command`)이 수행할 작업을 지정한다.

이 패턴의 핵심 문제는 HTTP의 라우팅, 캐싱, 보안, 상태 코드 등의 풍부한 기능을 전혀 활용하지 못한다는 점이다. 모든 요청이 POST이므로 중간 프록시, CDN, 로드 밸런서가 요청의 의도를 이해하지 못해 최적화된 라우팅이나 캐싱 정책을 적용할 수 없으며, 단일 URI이므로 특정 리소스에 대한 액세스 제어나 모니터링도 불가능에 가깝다.

### 등장 배경

2000년대 중반 웹 서비스 기술은 SOAP (Simple Object Access Protocol)이 표준으로 자리 잡았으나, XML 기반의 무거운 페이로드와 WS-Security, WS-ReliableMessaging 등 복잡한 WS-* 프로토콜 스택으로 인해 실무에서는 과도한 복잡도 문제가 발생했다. 개발팀은 SOAP의 복잡성을 피하기 위해 더 가벼운 대안을 찾기 시작했으며, 이때 등장한 것이 HTTP POST 위에 JSON/XML을 실어 보내는 단순한 패턴이었다.

또 다른 배경은 기존 RPC 프레임워크(예: XML-RPC, JSON-RPC, SOAP)의 설계 패턴을 HTTP에 그대로 옮겨온 관행이다. RPC에서는 "원격지의 어떤 프로시저(함수)를 호출할지"가 핵심이므로, 이를 HTTP로 변환할 때도 "어떤 액션을 수행할지"를 요청 본문에 포함하는 자연스러운 스타일이 발생했다. 이 패턴은 개발자에게는 친숙한 함수 호출 모델과 유사하여 빠르게 구현할 수 있었고, 초기 프로토타이핑이나 내부 도구용 API에서 널리 사용되었다.

하지만 Roy Fielding이 2000년에 정립한 REST 아키텍처 스타일이 널리 알려지면서, 이러한 RPC 스타일의 HTTP 활용이 REST 원칙에 위배된다는 비판이 제기되었다. Leonard Richardson은 이러한 혼란을 정리하기 위해 2008년경 REST 성숙도를 4단계로 계량화하는 모델을 제안했으며, 그 최하위 단계가 바로 이 "The Swamp of POX"이다.

### 기술적 문제의식

Level 0 패턴의 근본적 문제는 HTTP를 "의미 있는 프로토콜"이 아니라 "단순한 전송 파이프"로만 취급한다는 점이다. 마치 TCP 소켓 위에 자체 프로토콜을 구축하듯, HTTP 위에 또 다른 액션 명명 규칙을 구축하게 되며, 이는 HTTP 표준 생태계와의 단절을 의미한다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                  HTTP Protocol Utilization Comparison                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [Level 0: RPC Style]                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │    클라이언트              HTTP              서버                     │   │
│  │  ┌─────────┐              │              ┌─────────┐                │   │
│  │  │ Action: │              │              │ Action: │                │   │
│  │  │ getUser │─────────>    │    ──────>   │ getUser │                │   │
│  │  │ Action: │    POST      │    POST      │ Action: │                │   │
│  │  │ deleteUser│─────────>   │    ──────>   │deleteUser│               │   │
│  │  └─────────┘    /api       │              └─────────┘                │   │
│  │       │                     │                  │                     │   │
│  │       └── HTTP는 투명한 파이프 역할만 ─────────┘                      │   │
│  │                                                                      │   │
│  │  문제:                                                               │   │
│  │  • 모든 요청이 동일한 URI(/api)와 메서드(POST)                      │   │
│  │  • 중간 프록시/CDN이 요청의 의도를 이해 불가                         │   │
│  │  • 캐싱, 라우팅, 보안 정책 적용 불가                                │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [Level 2: RESTful Style]                                                   │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │    클라이언트              HTTP              서버                     │   │
│  │  ┌─────────┐              │              ┌─────────┐                │   │
│  │  │   GET   │─────────>    │    ──────>   │   GET   │                │   │
│  │  │ /user/123│   /user/123 │              │ /user/123│                │   │
│  │  │ DELETE  │─────────>    │    ──────>   │ DELETE  │                │   │
│  │  │/user/123│  /user/123   │              │/user/123│                │   │
│  │  └─────────┘              │              └─────────┘                │   │
│  │       │                     │                  │                     │   │
│  │       └── HTTP 동사와 URI가 의미를 전달 ─────────────┘              │   │
│  │                                                                      │   │
│  │  장점:                                                               │   │
│  │  • URI와 HTTP 메서드가 요청의 의도를 명확히 전달                    │   │
│  │  • GET 요청은 CDN/프록시에서 캐싱 가능                              │   │
│  │  • 로드 밸런서/게이트웨이에서 세밀한 라우팅/제한 정책 적용           │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

이 비교 도식의 핵심은 Level 0에서 HTTP가 전혀 의미를 갖지 않는 "투명한 파이프"로 전락한다는 점이다. 반면 Level 2에서는 URI와 HTTP 동사가 요청의 의미를 명확히 전달하므로, 중간 인프라(CDN, 로드 밸런서, API Gateway)가 이 정보를 활용하여 지능적인 라우팅, 캐싱, 보안 정책을 적용할 수 있다. 실무에서는 이 차이가 대규모 트래픽 환경에서 수백만 달러의 인프라 비용 차이와 시스템 안정성 차이로 직결된다.

📢 **섹션 요약 비유**: Level 0은 마치 우편 시스템을 이용하면서 모든 편지를 "수신인: 우체국"으로 보내고, 편지 봉투 안에 "실제 수신: 김철수"라고 적는 것과 같습니다. 우체국 직원은 봉투를 뜯어보기 전까지는 편지를 어디로 배달해야 할지 알 수 없으므로, 모든 편지를 중앙 집중처리해야 하므로 효율이 극도로 낮습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Level 0 구조 요소

Level 0 패턴의 구조는 극도로 단순하며, 단일 진입점과 액션 기반 디스패치 로직으로 구성된다.

| 구성 요소 | 역할 | 내부 동작 | 프로토콜 | 비유 |
|:---------|:-----|:---------|:--------|:-----|
| **단일 URI** | 모든 API 요청의 진입점 | `/api`, `/service`, `/endpoint` 등 고정 경로 | HTTP | 모든 차량이 단일 진입로만 사용하는 주차장 |
| **POST 전용** | 유일한 HTTP 메서드 | GET, PUT, DELETE를 사용하지 않음 | HTTP | 모든 운송 수단이 화물선 하나만 사용 |
| **액션 파라미터** | 수행 작업 식별자 | `action`, `method`, `command`, `op` 필드 | JSON/XML | 배송 라벨에 "물건 종류" 적는 것 |
| **파라미터 객체** | 액션 실행 인자 전달 | `params`, `data`, `args` 중첩 객체 | JSON/XML | 상자 안에 물건 넣어 배송 |
| **응답 래퍼** | 결과 공통 포맷 | `{status, data, error}` 구조 | JSON/XML | 표준화된 배송 완료 통지서 |

### Level 0 요청/응답 구조 분석

다음은 Level 0 패턴의 전형적인 요청과 응답 구조 예시이다.

```json
// Level 0 요청 예시: 사용자 조회
POST http://api.example.com/service
Content-Type: application/json

{
  "action": "getUser",
  "params": {
    "userId": 123
  }
}

// Level 0 응답 예시: 성공
{
  "status": "success",
  "data": {
    "id": 123,
    "name": "John Doe",
    "email": "john@example.com"
  }
}

// Level 0 요청 예시: 사용자 삭제
POST http://api.example.com/service
Content-Type: application/json

{
  "action": "deleteUser",
  "params": {
    "userId": 123,
    "reason": "account_closure"
  }
}

// Level 0 응답 예시: 실패
{
  "status": "error",
  "error": {
    "code": "USER_NOT_FOUND",
    "message": "User with ID 123 not found"
  }
}
```

이 구조의 핵심은 모든 요청이 동일한 URL(`/service`)과 메서드(POST)를 공유하며, 요청 본문의 `action` 필드가 실제 수행할 작업을 결정한다는 점이다. 서버 측에서는 `action` 값을 기반으로 적절한 핸들러 함수로 디스패치하는 스위치 로직이 필요하다.

```javascript
// 서버 측 디스패치 로직 예시 (Node.js/Express)
function handleRequest(req, res) {
  const { action, params } = req.body;

  switch (action) {
    case 'getUser':
      return getUserHandler(params, res);
    case 'createUser':
      return createUserHandler(params, res);
    case 'updateUser':
      return updateUserHandler(params, res);
    case 'deleteUser':
      return deleteUserHandler(params, res);
    default:
      return res.status(200).json({
        status: 'error',
        error: { code: 'UNKNOWN_ACTION', message: `Action ${action} not found` }
      });
  }
}
```

### Level 0의 캐싱 불가 원리

Level 0 패턴의 가장 치명적 문제 중 하나는 캐싱이 거의 불가능하다는 점이다. 그 이유는 다음과 같다.

**① POST 메서드 특성**

HTTP 사양에서 POST는 "원서버의 특정 리소스에 하위 리소스로 처리되도록 데이터를 제출하는" 것으로 정의되며, 기본적으로 멱등성(Idempotent)을 보장하지 않는다. 따라서 중간 프록시나 CDN은 POST 응답을 캐싱하지 않으며, 명시적인 Cache-Control 헤더가 있어도 안전하게 캐싱을 거부한다.

**② 단일 URI 문제**

모든 요청이 동일한 URI(`/api`)를 공유하므로, 캐시 키(Cache Key)를 생성할 때 URI만으로는 충분하지 않고 요청 본문 전체를 고려해야 한다. 하지만 요청 본문은 캐시 레이어에서 일반적으로 접근하지 않거나, 접근하더라도 본문 기반 캐싱은 계산 비용이 매우 높다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Level 0 Caching Impossibility                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [요청 1]                                                                  │
│  POST /api                                                                 │
│  {action: "getUser", params: {userId: 123}}                                │
│       │                                                                    │
│       │ 캐시 레이어(CDN, 프록시)                                          │
│       ▼                                                                    │
│  ❌ "POST 요청이므로 캐싱 불가"                                            │
│  ❌ "단일 URI이므로 캐시 키 모호"                                         │
│       │                                                                    │
│       ▼                                                                    │
│  ┌─────────────────┐                                                      │
│  │ 애플리케이션    │  ← 매번 원본 서버로 직접                            │
│  │ 서버            │                                                            │
│  └─────────────────┘                                                      │
│                                                                             │
│  [요청 2 - 동일 사용자 조회]                                               │
│  POST /api                                                                 │
│  {action: "getUser", params: {userId: 123}}                                │
│       │                                                                    │
│       │ 캐시 레이어                                                        │
│       ▼                                                                    │
│  ❌ "이미 응답을 캐싱했더라도, POST이므로 재전송"                          │
│       │                                                                    │
│       ▼                                                                    │
│  ┌─────────────────┐                                                      │
│  │ 애플리케이션    │  ← 불필요한 중복 요청                               │
│  │ 서버            │                                                            │
│  └─────────────────┘                                                      │
│                                                                             │
│  [Level 2와의 비교]                                                         │
│  GET /users/123  → ✅ 캐시 레이어에서 즉시 응답 (CDN 히트)                │
│                   → 애플리케이션 서버 부하 60~80% 경감                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

이 캐싱 불가 도식의 핵심은 Level 0에서는 동일한 데이터를 조회하는 요청이 반복되더라도, POST 메서드이므로 매번 원본 서버까지 전달된다는 점이다. 반면 Level 2에서는 GET 요청과 리소스별 URI를 사용하므로, CDN이나 프록시 캐시에서 즉시 응답 가능하여 애플리케이션 서버 부하를 60~80% 경감할 수 있다. 실무에서 대규모 트래픽을 처리하는 공개 API의 경우, 이 차이만으로도 인프라 비용이 수백만 달러 단위로 차이가 날 수 있다.

### Level 0 디스패치 오버헤드

Level 0 패턴에서는 모든 요청이 단일 진입점을 통과하므로, 서버 측에서는 액션별 핸들러로 디스패치하는 스위치 로직이 필수적이다. 이 디스패치 오버헤드는 요청 수가 증가할수록 누적된다.

```javascript
// 디스패치 성능 비교 (의사코드)

// Level 0: 액션 기반 디스패치
function level0Handler(req, res) {
  const action = req.body.action;  // 본문 파싱 필요

  // 문자열 비교 기반 디스패치 (O(n))
  if (action === 'getUser') {
    return getUser(req.body.params, res);
  } else if (action === 'createUser') {
    return createUser(req.body.params, res);
  } else if (action === 'updateUser') {
    return updateUser(req.body.params, res);
  } else if (action === 'deleteUser') {
    return deleteUser(req.body.params, res);
  } // ... 수십~수백 개의 액션
}

// Level 2: URI + HTTP 동사 기반 라우팅
// 웹 프레임워크의 라우터가 해결 (O(log n) 해시 테이블 룩업)
app.get('/users/:id', getUser);      // 프레임워크가 자동 라우팅
app.post('/users', createUser);
app.put('/users/:id', updateUser);
app.delete('/users/:id', deleteUser);
```

Level 0의 디스패치 로직은 요청마다 문자열 비교(if-else 또는 switch)를 수행하므로 액션 수가 증가할수록 선형적으로 오버헤드가 증가한다. 반면 Level 2에서는 웹 프레임워크(Express, Spring MVC, Django 등)의 라우터가 해시 테이블 기반의 효율적인 라우팅을 수행하므로, 개발자는 비즈니스 로직에 집중할 수 있다.

📢 **섹션 요약 비유**: Level 0은 마치 거대한 공항에 출구가 하나만 있는 상황과 같습니다. 모든 승객이 단일 출구를 통해 나와야 하므로, 출구 직원은 각 승객의 목적지를 확인하고 적절한 게이트로 안내해야 합니다. 반면 Level 2는 목적지별로 출구가 따로 있어, 승객이 직접 해당 출구를 선택하여 나갈 수 있으므로 훨씬 효율적입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### Level 0 vs Level 1 vs Level 2 비교

Richardson Maturity Model의 세 단계를 기능, 성능, 운용 관점에서 비교 분석한다.

| 비교 항목 | Level 0 (RPC) | Level 1 (Resources) | Level 2 (HTTP Verbs) | 차이점 설명 |
|:---------|:--------------|:-------------------|:-------------------|:-----------|
| **URI 설계** | 단일 URI(`/api`) | 리소스별 URI(`/users/123`) | 리소스별 URI | 식별자 분리 |
| **HTTP 메서드** | POST만 | POST만 | GET/POST/PUT/DELETE | 통신 의미 부여 |
| **캐싱 가능성** | 거의 불가(~5%) | 낮음(~20%) | 높음(~65%) | CDN 활용도 |
| **멱등성 보장** | 불명확 | 불명확 | 명확(GET/PUT/DELETE) | 재시도 안전성 |
| **라우팅** | 액션 기반 디스패치 | URI 기반 라우팅 | URI+메서드 기반 | 인프라 지원 |
| **API Gateway 활용** | 제한적 | 부분 | 완전 | 세밀한 정책 |
| **CDN 에지 캐싱** | 불가 | 제한적 | 가능 | 전역 성능 |
| **개발 복잡도** | 낮음 | 중간 | 중간 | 구현 난이도 |

### 과목 융합: Level 0과 연관 기술 스택

**① SOAP/Web Services와의 관계**

Level 0 패턴은 SOAP 웹 서비스의 복잡성에 대한 반작용으로 등장했으나, 아이러니하게도 구조적 유사성을 공유한다. SOAP도 모든 요청이 단일 URL(`http://example.com/soap/service`)로 전송되며, 요청 본문의 XML 요소 이름이 수행할 작업을 결정한다. 차이점은 SOAP이 WSDL(Web Services Description Language) 기반의 강력한 타입 시스템과 WS-* 보안 프로토콜을 갖는 반면, Level 0는 JSON 기반의 단순한 구조를 갖는다.

```xml
<!-- SOAP 요청 예시 -->
<soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope">
  <soap:Body>
    <m:GetUser xmlns:m="http://example.com/user">
      <m:UserId>123</m:UserId>
    </m:GetUser>
  </soap:Body>
</soap:Envelope>
```

```json
// Level 0 요청 예시
{
  "action": "getUser",
  "params": {
    "userId": 123
  }
}
```

두 패턴 모두 "요청 본문의 액션 이름이 무엇을 할지 결정"한다는 공통점이 있으나, Level 0는 SOAP의 무거운 XML 스택을 피하기 위해 등장한 가벼운 대안이다.

**② RPC 프레임워크와의 관계**

Level 0는 기본적으로 HTTP 위에 구현된 RPC(Remote Procedure Call) 패턴이다. gRPC, Apache Thrift, JSON-RPC, XML-RPC 등 전통적인 RPC 프레임워크와 개념적으로 유사하나, 이들은 전용 프로토콜과 최적화된 직렬화(Protocol Buffers, Thrift Binary)를 사용하는 반면, Level 0는 HTTP/JSON 범용 스택을 사용한다는 차이가 있다.

```protobuf
// gRPC 예시 (IDL)
service UserService {
  rpc GetUser(GetUserRequest) returns (GetUserResponse);
  rpc DeleteUser(DeleteUserRequest) returns (DeleteUserResponse);
}
```

Level 0는 gRPC와 같은 전용 RPC 프레임워크의 성능과 기능을 희생하면서, HTTP/JSON의 범용성을 얻은 타협안이라고 할 수 있다. 실무에서는 전용 RPC 프레임워크의 학습 곡선을 피하면서 분산 통신이 필요한 경우에 Level 0 스타일이 선택되기도 한다.

**③ GraphQL과의 비교**

GraphQL은 Level 0와 유사하게 단일 URI(`/graphql`)와 POST 메서드를 사용하지만, 결정적 차이가 있다. GraphQL은 쿼리 언어와 스키마 정의가 있으며, 클라이언트가 필요한 데이터를 정확히 지정(Over-fetching/Under-fetching 해결)할 수 있는 반면, Level 0는 단순한 액션 호출이다.

```graphql
# GraphQL 요청 예시
POST /graphql
{
  "query": "query { user(id: 123) { name email } }"
}
```

Level 0와 GraphQL은 단일 URI/POST 사용이라는 외형적 유사성을 공유하지만, GraphQL은 강력한 쿼리 언어와 타입 시스템을 갖는 반면 Level 0는 단순한 RPC 호출이라는 내용적 차이가 있다.

📢 **섹션 요약 비유**: Level 0는 자전거와 자동차의 중간 단계인 "모터바이크에 페달 달린 것"과 같습니다. 자전거(SOAP 없는 간단한 통신)보다는 빠르지만, 자동차(Level 2 RESTful)의 안정성과 편의성에는 미치지 못합니다. 결국 양쪽 모두의 장점을 완전히 누리지 못하는 타협안입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오별 Level 0 사용 판단

Level 0는 대부분의 실무 상황에서 권장되지 않으나, 예외적으로 적합한 경우가 있다. 다음 체크리스트는 Level 0 도입을 고려할 때의 의사결정 기준을 제공한다.

| 시나리오 | Level 0 적합성 | 판단 근거 | 대안 |
|:--------|:-------------:|:---------|:-----|
| **내부 도구 프로토타이핑** | ⚠️ 조건부 적합 | 빠르게 구현 가능하나, 프로덕션 전에는 Level 1+로 진화 필요 | Level 1로 빠르게 이동 |
| **레거시 시스템 간 통신** | ⚠️ 기존 호환성 필요 | 기존 SOAP 연동 시 단순화 목적일 수 있으나, 장기적 이동 권장 | RESTful 래퍼 구현 |
| **공개 API** | ❌ 부적합 | HTTP 표준 도구 활용 불가, 파트너 통합 비용 증가 | Level 2 이상 필수 |
| **대규모 트래픽 서비스** | ❌ 치명적 | 캐싱 불가로 확장성 극도로 저조 | Level 2 + CDN 필수 |
| **마이크로서비스 간 통신** | ⚠️ 바람직하지 않음 | Service Mesh 메트릭 수집, mTLS 정책 적용 어려움 | Level 2 표준화 권장 |
| **모바일 앱 백엔드** | ❌ 부적합 | 모바일 네트워크 환경에서 캐싱 효과 중요 | Level 2로 전환 필수 |

### 실무 시나리오 상세 분석

**시나리오 1: 레거시 SOAP 시스템에서 REST로의 마이그레이션**

기존 엔터프라이즈 시스템이 SOAP 웹 서비스를 사용 중이며, 이를 REST API로 마이그레이션하는 과정에서 Level 0 스타일이 일시적으로 등장하는 경우가 있다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│            SOAP to REST Migration Path (Typical)                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [단계 1] 레거시 SOAP                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  POST /soap/service                                                 │   │
│  │  <soap:Envelope>                                                   │   │
│  │    <m:GetUser>...</m:GetUser>  ← XML 액션                          │   │
│  │  </soap:Envelope>                                                  │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                              │ 마이그레이션                                │
│                              ▼                                            │
│  [단계 2] Level 0로 변환 (XML → JSON)                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  POST /api                                                          │   │
│  │  {action: "getUser", params: {userId: 123}}  ← JSON 액션           │   │
│  │                                                                      │   │
│  │  • SOAP 복잡성 제거 (XML 스택 → JSON)                              │   │
│  │  • 하지만 여전히 POST-only, 단일 URI                                │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                              │ 진화 필요                                   │
│                              ▼                                            │
│  [단계 3] Level 1 리소스 URI 추가                                          │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  POST /users/123     ← 리소스별 URI                                 │   │
│  │  {action: "get"}                                                     │   │
│  │                                                                      │   │
│  │  • URI 식별력 획득                                                   │   │
│  │  • 여전히 POST-only                                                 │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                              │ 진화 필요                                   │
│                              ▼                                            │
│  [단계 4] Level 2 HTTP 동사 활용                                          │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  GET /users/123      ← 조회는 GET                                   │   │
│  │  DELETE /users/123   ← 삭제는 DELETE                                │   │
│  │                                                                      │   │
│  │  • 진정한 RESTful API                                                │   │
│  │  • 캐싱, 라우팅, 보안 혜택                                          │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  [핵심 교훈]                                                               │
│  • Level 0는 SOAP에서 REST로 이동하는 과도기적 단계로서 자연 발생         │
│  • 하지만 Level 0에 머무르면 SOAP의 단점(캐싱 불가)만 계승               │
│  • 빠른 시간 내에 Level 1 → Level 2로 진화하는 전략 필요                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

이 마이그레이션 경로의 핵심은 Level 0가 일시적으로 발생할 수 있으나, 빠른 시간 내에 상위 Level로 진화하는 전략이 필요하다는 점이다. 실무에서는 단계 2(Level 0)에 머무르는 기간을 1~2개월로 제한하고, 그 사이에 다음 단계로의 이동 계획을 수립하는 것이 좋다. Level 0에 장기간 머무르면 SOAP의 복잡성은 제거했지만 여전히 캐싱 불가라는 단점만 계승하는 최악의 상황이 발생한다.

**시나리오 2: Level 0에서 Level 2로의 진화 전략**

기존 Level 0 API를 Level 2 RESTful API로 진화시키는 실전 전략을 제시한다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│              Level 0 → Level 2 Evolution Strategy                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [Phase 1: 감사와 계획 (1-2주)]                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  ① 기존 액션 목록화                                                 │   │
│  │     - 예: getUser, createUser, updateUser, deleteUser, ...          │   │
│  │                                                                      │   │
│  │  ② 액션을 HTTP 동사로 분류                                           │   │
│  │     - 조회 → GET: getUser, listUsers, searchProducts               │   │
│  │     - 생성 → POST: createUser, createOrder                         │   │
│  │     - 수정 → PUT: updateUser, replaceProfile                       │   │
│  │     - 삭제 → DELETE: deleteUser, removeItem                       │   │
│  │                                                                      │   │
│  │  ③ URI 설계                                                         │   │
│  │     - /users/{id}, /orders/{id}, /products/{id}                   │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                    │                                        │
│                                    ▼                                        │
│  [Phase 2: 병렬 운용 기간 (2-4주)]                                          │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  ① 새로운 Level 2 엔드포인트 구축                                  │   │
│  │     - 기존 비즈니스 로직 재사용                                     │   │
│  │     - URI 라우팅 추가                                               │   │
│  │                                                                      │   │
│  │  ② 기존 Level 0 엔드포인트 유지                                     │   │
│  │     - "Deprecated" 헤더 추가                                         │   │
│  │     - 폐지 일자 공표                                                │   │
│  │                                                                      │   │
│  │  ③ 클라이언트 이동                                                  │   │
│  │     - 신규 클라이언트: Level 2만 사용                               │   │
│  │     - 기존 클라이언트: 점진적 이동                                   │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                    │                                        │
│                                    ▼                                        │
│  [Phase 3: 완전 전환 및 폐지 (1-2주)]                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  ① 모든 클라이언트가 Level 2로 이동 확인                            │   │
│  │                                                                      │   │
│  │  ② Level 0 엔드포인트 폐지                                          │   │
│  │     - HTTP 410 Gone 반환                                           │   │
│  │     - 모니터링으로 트래픽 0 확인                                     │   │
│  │                                                                      │   │
│  │  ③ CDN 캐싱 설정                                                   │   │
│  │     - GET 엔드포인트에 Cache-Control 추가                           │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

이 진화 전략의 핵심은 기존 클라이언트를 중단 없이 이동시키기 위해 병렬 운용 기간을 두는 점이다. API 버전 관리 전략(V1: Level 0, V2: Level 2)을 활용하거나, URI 경로 기반(`/api/v1/` vs `/api/v2/`) 분리로 클라이언트가 선택적으로 이동할 수 있게 한다. 실무에서는 이 전체 과정에 2~3개월이 소요되며, 완료 후 CDN 캐싱 적용으로 서버 부하가 평균 40~60% 감소하는 효과를 관찰할 수 있다.

### Level 0 도입 체크리스트 (부정적)

다음 조건 중 **하나라도 해당**하면 Level 0 사용을 재고해야 한다.

- [ ] 공개적으로 제공될 API인가
- [ ] 일일 요청 수가 10,000건 이상인가
- [ ] 3개 이상의 외부 팀/파트너가 연동하는가
- [ ] CDN 또는 에지 캐싱을 사용하는 환경인가
- [ ] API Gateway에서 세밀한 라우팅/인가/속도 제한이 필요한가
- [ ] Service Mesh(Istio, Linkerd 등)를 사용하는 MSA 환경인가
- [ ] 모바일 앱에서 호출되는 API인가 (네트워크 비용 중요)
- [ ] 장기적으로(1년 이상) 유지보수될 API인가

이 체크리스트에서 3개 이상 "예"라면 Level 0 사용은 권장되지 않으며, 적어도 Level 2로 설계할 것을 강력히 권장한다.

### 안티패턴: Level 0 오용 사례

**안티패턴 1: 공개 API에서의 Level 0 사용**

```json
// ❌ 공개 API에서의 Level 0 (사용자 경험 저하)
// 모든 개발자가 "POST /api" 하나만 알면 된다고 생각할 수 있으나...
POST https://api.public-service.com/api
{
  "action": "getWeather",
  "params": {"city": "Seoul"}
}

// 문제점:
// • API 문서가 복잡해짐 (모든 액션을 별도 문서화)
// • 표준 HTTP 클라이언트 라이브러리 활용 불가
// • curl, Postman에서 빠른 테스트 어려움
// • SDK 생성 도구가 자동화되지 않음
```

**안티패턴 2: 조회 작업에서의 POST 남용**

```json
// ❌ 조회 작업에 POST 사용 (캐싱 기회 상실)
POST /api
{
  "action": "getProductDetail",
  "params": {"productId": 12345}
}
// 상품 상세는 자주 조회되지만, POST이므로 CDN 캐싱 불가
// → 서버 부하 증가, 레이턴시 증가

// ✅ 올바른 접근 (Level 2)
GET /products/12345
// CDN이 직접 응답 가능 → 서버 부하 경감, 레이턴시 개선
```

**안티패턴 3: Level 0에 HATEOAS 흉내내기**

```json
// ❌ Level 0에서 응답에 링크를 포함하지만 하드코딩
{
  "status": "success",
  "data": {...},
  "next": "https://api.example.com/api"  // 정적 링크, 의미 없음
}

// 진정한 HATEOAS는 Level 3에서 리소스 URI와 결합
GET /users/123
{
  "id": 123,
  "_links": {
    "self": {"href": "/users/123"},
    "orders": {"href": "/users/123/orders"}
  }
}
```

📢 **섹션 요약 비유**: Level 0는 집짓기에 비유하면 "모든 방이 출입문 하나로만 연결된 구조"와 같습니다. 화장실에 가든, 주방에 가든, 침실에 가든 항상 같은 문을 통해 나와야 하므로, 여러 사람이 동시에 이용할 때 병목이 발생합니다. 반면 Level 2는 각 방에 독립적인 출입문이 있어 필요한 곳으로 직접 이동할 수 있어 효율적입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### Level 0의 한계 정량 분석

Level 0 패턴 사용 시 발생하는 주요 성능 및 운용상의 한계를 정량적으로 분석한다.

| 평가 항목 | Level 0 | Level 2 | 개선 효과 | 설명 |
|:--------|:-------|:-------|:--------:|:-----|
| **CDN 캐시 히트율** | 0~5% | 60~85% | +1,600% | GET 요청만으로도 60% 이상 캐싱 가능 |
| **API Gateway 라우팅 정교도** | 단일 경로 | URI+동사 기반 | N/A | 세밀한 정책 적용 가능 |
| **Service Mesh 메트릭 정확도** | 낮음(POST만) | 높음(동사별) | +300% | 동사별 분석으로 트러블슈팅 개선 |
| **클라이언트 SDK 자동화** | 불가 | 가능(OpenAPI) | N/A | OpenAPI 기반 코드 생성 |
| **서버 부하(동일 트래픽)** | 100% (기준) | 40~60% | -50% | 캐싱으로 서버 부하 경감 |
| **모바일 네트워크 사용량** | 높음 | 중간 | -30% | 캐싱으로 데이터 전송 감소 |

### 기술적 부채(Technical Debt) 관점

Level 0는 초기 개발 속도는 빠르지만, 시스템이 성장함에 따라 기술적 부채가 급격히 누적된다.

**① 초기 단계 (팀 1~3명, API 10개 미만)**

- 개발 속도: 빠름 (프레임워크 표준 라우터 불필요)
- 문서화: 간단 (액션 목록만 작성)
- 유지보수: 용이 (모든 로직이 단일 핸들러)

**② 성장 단계 (팀 5~10명, API 50~200개)**

- 개발 속도: 저하 (액션 수 증가로 디스패치 로직 복잡)
- 문서화: 어려움 (액션별 매개변수를 개별 문서화)
- 유지보수: 어려움 (중앙 디스패처가 병목)
- 충돌 가능성: 증가 (동일한 액션 이름 사용)

**③ 확장 단계 (팀 20+명, API 500+개)**

- 개발 속도: 극도로 저하 (디스패치 로직이 거대한 switch-case)
- 문서화: 거의 불가능 (액션 관계가 복잡하게 얽힘)
- 유지보수: 불가능 (한 줄 변경이 전체 시스템에 영향)
- 확장성: 극도로 저하 (모든 요청이 단일 진입점 경합)

따라서 프로젝트 초기에는 Level 0가 빠르게 시작할 수 있는 방법처럼 보이지만, 중장기적으로는 Level 2 이상으로 설계하는 것이 기술적 부채 관점에서 훨씬 유리하다.

### 대안 및 향후 방향

**① OpenAPI Specification 기반 설계**

OpenAPI(Swagger) 명세를 먼저 작성하고, 이를 기반으로 Level 2 RESTful API를 자동 생성하는 접근이 권장된다. OpenAPI 툴은 URI 설계, HTTP 동사 분배, 스키마 정의를 자동으로 검증하며, 클라이언트/서버 SDK 생성도 자동화할 수 있다.

**② API-First 설계 접근**

코드를 먼저 작성하는 API-Last 접근 대신, API 명세를 먼저 작성하는 API-First 접근을 사용한다. 이를 통해 Richardson Level 2 이상을 보장하고, 팀 간 인터페이스 명세를 초기에 확정할 수 있다.

**③ GraphQL 또는 gRPC로의 전환**

Level 0의 단순성은 유지하면서 성능과 기능을 개선하려면, GraphQL(단일 URI, POST 사용, 쿼리 언어) 또는 gRPC(전용 프로토콜, 고성능)로의 전환을 고려한다. 두 기술 모두 단일 엔드포인트 사용이라는 외형은 유지하면서, Level 0의 한계를 극복하는 강력한 기능을 제공한다.

### 참고 표준 및 규격

- **RFC 7231 - HTTP/1.1 Semantics and Content**: GET, POST, PUT, DELETE의 정의와 의미
- **RFC 9110 - HTTP Semantics**: 최신 HTTP 명세, 멱등성과 안전성(Safe Methods) 정의
- **OpenAPI Specification 3.1**: RESTful API 명세 표준

📢 **섹션 요약 비유**: Level 0는 도시 교통 계획에서 "모든 차량이 단일 도로를 사용하는 설계"와 같습니다. 초기에는 도로 하나만 건설하면 되므로 빠르고 저렴하지만, 교통량이 증가하면 극심한 정체가 발생합니다. 반면 Level 2는 목적지별 도로를 계획적으로 건설하는 것으로, 초기 비용은 더 들지만 장기적으로는 원활한 교통 흐름을 보장합니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---------|:------------------|
| **REST (Representational State Transfer)** | Richardson Maturity Model의 평가 기준. Level 0는 REST 원칙을 거의 준수하지 않음 |
| **SOAP (Simple Object Access Protocol)** | Level 0의 등장 배경인 복잡한 웹 서비스 스택. Level 0는 SOAP의 간소화된 대안이지만 동일한 단일 URI 패턴 공유 |
| **RPC (Remote Procedure Call)** | Level 0의 본질적 패러다임. HTTP 위에 RPC 스타일 호출을 얇게 감싼 것 |
| **Richardson Maturity Model Level 1** | Level 0의 다음 단계. 리소스별 URI 할당으로 식별력 확보 |
| **Richardson Maturity Model Level 2** | 궁극적 목표. HTTP 동사 적절 활용으로 캐싱, 멱등성, 라우팅 혜택 획득 |
| **GraphQL** | 단일 URI/POST 사용이라는 외형적 유사성 공유. 하지만 쿼리 언어와 스키마로 Level 0의 한계 극복 |
| **gRPC** | RPC 패러다임의 현대적 구현. HTTP/2와 Protocol Buffers 사용하여 Level 0보다 우수한 성능과 기능 제공 |
| **API Gateway** | Level 0에서는 라우팅 불가. Level 2에서는 URI+동사 기반 세밀한 라우팅, 속도 제한, 인가 정책 적용 가능 |
| **CDN (Content Delivery Network)** | Level 0의 POST-only 요청은 캐싱 불가. Level 2의 GET 요청은 CDN 에지 캐싱으로 전역 성능 향상 |
| **Service Mesh** | Level 0에서는 모든 요청이 동일한 POST로 보이므로 메트릭 수집이 어려움. Level 2에서는 HTTP 동사별 정교한 메트릭 가능 |

## 👶 어린이를 위한 3줄 비유 설명

**① 단일 출입구 비유**: Level 0는 학교에 교실이 수십 개 있는데 출입문이 딱 하나만 있는 것과 같습니다. 화장실에 가든, 급식실에 가든, 도서관에 가든 항상 같은 문으로 나가야 해서 여러 학생이 동시에 이용하면 문 앞에 줄이 길게 늘어집니다.

**② 우편 배달 비유**: Level 0는 모든 편지 봉투에 "수신인: 우체국"이라고 쓰고, 봉투 안에 진짜 받는 사람 이름을 적는 종이를 넣는 것과 같습니다. 우체부직원은 편지를 하나하나 뜯어보지 않으면 진짜 받는 사람을 알 수 없어서 배달이 아주 느려집니다.

**③ 장난감 정리 비유**: Level 0는 모든 장난감을 한 상자에 다 넣어두고, 필요할 때마다 상자를 뒤져서 찾는 것과 같습니다. 장난감이 몇 개 안 될 때는 괜찮지만, 수백 개가 되면 원하는 장난감을 찾는 데 시간이 너무 오래 걸립니다.
