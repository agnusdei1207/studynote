# 프로세스 마이닝 (Process Mining)

> **약어 (Abbreviation)**: Process Mining

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 마이닝 (Process Mining)은 전사적 자원 관리(ERP), 고객 관계 관리(CRM) 등 IT 시스템에 기록된 **이벤트 로그(Event Log)** 데이터를 추출 및 분석하여, 조직의 **실제(As-Is) 프로세스 모델을 자동으로 역공학(Reverse Engineering) 및 시각화**하는 기술이다.
> 2. **가치**: 직원의 기억이나 직관에 의존하던 전통적인 인터뷰 기반의 프로세스 분석 한계를 극복하고, 100% 팩트(데이터) 기반으로 숨겨진 **병목(Bottleneck)**, 규정 위반(Deviation), 비효율적 반복(Rework) 구간을 실시간으로 찾아낸다.
> 3. **융합**: 발견된 프로세스 병목을 해결하기 위해 **RPA(Robotic Process Automation)** 봇을 투입하거나, **BPM(Business Process Management)** 시스템의 모델을 최적화하는 초자동화(Hyperautomation) 파이프라인의 '진단(Diagnosis)' 핵심 엔진으로 사용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 프로세스 마이닝은 데이터 마이닝(Data Mining)과 비즈니스 프로세스 관리(BPM)의 교집합에 위치한 기술이다. 정보 시스템이 남긴 발자국(Log)을 모아, 어떤 일이 어떤 순서로 얼마나 걸려 일어났는지를 한눈에 볼 수 있는 '프로세스 지도'를 그려준다.
- **💡 비유**: 길거리에 사람들이 어떻게 걸어 다니는지 인터뷰(기존 방식)하는 것이 아니라, 사람들의 신발에 묻은 흙이나 휴대폰의 GPS 기록(로그)을 모아 지도 위에 뿌려보면 사람들이 실제로 가장 많이 다니는 지름길이나 막히는 골목을 정확히 알 수 있는 원리와 같습니다.
- **등장 배경**:
  - **기존 한계**: 전통적인 프로세스 컨설팅은 컨설턴트가 담당자를 일일이 인터뷰하고 워크숍을 열어 프로세스 맵(As-Is)을 그렸다. 이 과정은 수개월이 걸리고 막대한 비용이 들었으며, 담당자가 숨기거나 까먹은 '예외 상황(Exception)'은 도면에 반영되지 않아 분석 결과가 부정확했다.
  - **혁신적 패러다임**: 2000년대 후반 Wil van der Aalst 교수를 중심으로 "모든 IT 시스템은 로그를 남긴다. 이 로그의 Case ID, Activity, Timestamp 3가지만 연결하면 프로세스를 100% 자동으로 그려낼 수 있다"는 알고리즘(Alpha Algorithm, Heuristic Miner 등)이 발명되면서 패러다임이 전환되었다.
  - **현재의 비즈니스 요구**: 디지털 트랜스포메이션(DT) 시대를 맞아 수많은 시스템(SAP, Salesforce, ServiceNow)에서 데이터가 쏟아지고 있다. 기업은 "어디를 자동화해야 가장 비용 절감 효과가 큰가?"를 데이터로 증명해야 하므로, RPA나 ERP 고도화 프로젝트의 선행 필수 단계로 자리 잡았다.

> 다음은 기존 인터뷰 기반의 분석 방식과 프로세스 마이닝 기반 분석 방식의 구조적 차이를 보여주는 비교 도식이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [Traditional Consulting vs Process Mining]                  │
├──────────────────────────────┬──────────────────────────────┤
│ 1. Interview-based (기존)    │ 2. Data-based (Process Mining│
│                              │                              │
│ [Consultant]                 │ [Information System (ERP)]   │
│  | (Question)                │  | (Event Log Database)      │
│  v                           │  v                           │
│ [Employee] "보통 이렇게 해요"│ [Mining Engine (Celonis)]    │
│  | (Subjective Memory)       │  | (Algorithms)              │
│  v                           │  v                           │
│ [Process Map]                │ [Spaghetti Process Map]      │
│ (단순하고 예외가 없는 그림)  │ (복잡하지만 100% 팩트인 그림)│
└──────────────────────────────┴──────────────────────────────┘
```
- **해설**: 직원은 자신이 정해진 규정대로(Happy Path) 완벽하게 일한다고 믿거나 대답한다. 그러나 실제 IT 시스템 로그를 마이닝해 보면, 승인을 우회하거나 반려 후 재작업(Rework)하는 수많은 변칙 경로가 스파게티처럼 얽혀 있는 진짜 현실(Fact)이 드러난다.

> **📢 섹션 요약 비유**: 의사가 환자의 "배가 아파요"라는 말만 듣고 진단하는 것(인터뷰)에서 벗어나, 환자의 몸속을 MRI와 내시경(프로세스 마이닝)으로 직접 들여다보고 종양(병목)의 위치를 1mm 오차 없이 찾아내는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 핵심 구성 요소 (이벤트 로그의 3대 필수 속성)

프로세스 마이닝이 성립하기 위해서는 정보 시스템(ERP, CRM 등)에서 추출한 데이터에 반드시 **다음 3가지 속성**이 포함되어야 한다.

| 필수 요소 (Log Attributes) | 역할 | 내부 의미 및 예시 데이터 | 비유 |
|:---|:---|:---|:---|
| **Case ID (식별자)** | 프로세스 인스턴스 구분 | 특정 주문 건, 특정 환자, 특정 결재 번호 (예: `Order-001`) | 특정 사람의 이름표 |
| **Activity (작업명)** | 발생한 사건의 종류 | 해당 건에 대해 수행된 상태 변경 (예: `주문 접수`, `재고 확인`) | 그 사람이 한 행동 |
| **Timestamp (시간)** | 작업 발생 시간 | 사건이 일어난 정확한 시간 순서와 소요 시간 계산용 (예: `2026-03-04 10:00:00`) | 시계에 기록된 시간 |
| *(선택) Resource / User* | 작업 수행 주체 | 누가(사람 또는 시스템) 이 작업을 했는가 (예: `김대리`, `RPA_Bot`) | 일을 한 사람 |

### 2. 프로세스 마이닝 핵심 3대 기법 (Types of Process Mining)

> 아래 다이어그램은 프로세스 마이닝의 3가지 핵심 기법(Discovery, Conformance, Enhancement)이 각각 어떤 입출력을 가지는지 보여준다.

```text
[Process Mining 3 Core Techniques]

                    [ Event Logs ] (from System)
                          │
                          v
1. Discovery (도출)       ├──>  [ Actual Process Model (자동 시각화) ]
(로그 -> 모델)            │
                          │     [ Event Logs ] vs [ Ideal Model ]
                          v            \                 /
2. Conformance (적합성)   ├──────>  [ Deviation / 규정 위반 탐지 ]
(로그 vs 이상적 모델 비교)│
                          │     [ Mined Model ] + [ New Rules / Bottleneck Fix ]
                          v            \                 /
3. Enhancement (향상)     └──────>  [ Optimized Process Model ]
(기존 모델의 업그레이드)
```
- **해설**:
  1. **도출 (Discovery)**: 아무런 사전 지식 없이 오직 로그만으로 스파게티 맵을 그려낸다. (가장 많이 쓰임)
  2. **적합성 검사 (Conformance Checking)**: 회사가 정해둔 '표준 매뉴얼(Ideal Model)'과 직원이 실제로 일한 '로그'를 비교하여, "왜 500만 원 이상인데 본부장 결재를 패스했지?" 같은 규정 위반(Deviation)을 찾아낸다.
  3. **향상 (Enhancement)**: 발견된 병목 구간에 새로운 리소스나 IT 자동화를 투입하여 모델을 업그레이드한다.

### 3. 디스커버리 알고리즘 (Heuristic Miner) 원리
초기에는 Case 내에서 Activity A 다음에 B가 오면 단순히 화살표를 긋는 방식(Alpha Miner)을 썼다. 그러나 이는 노이즈(어쩌다 한 번 일어난 예외)에 취약했다. 최근에는 **Heuristic Miner**나 **Fuzzy Miner** 알고리즘을 사용하여, A->B로 가는 빈도(Frequency)를 확률적으로 계산하여 일정 임계치(Threshold)를 넘는 주요 흐름(Main Path)만 진하게 표시하고 노이즈를 걸러내는 통계적 기법을 사용한다.

> **📢 섹션 요약 비유**: 수백만 개의 점(로그 데이터)을 무작위로 뿌려놓고, 점들이 가장 많이 모여 있는 궤적을 따라 선을 이으면 거대한 은하수(실제 프로세스 맵)의 모양이 자동으로 드러나는 별자리 관측 알고리즘입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 데이터 분석 기술 비교 (BI / Data Mining vs Process Mining)

| 비교 항목 | Business Intelligence (BI) | Data Mining (데이터 마이닝) | Process Mining (프로세스 마이닝) |
|:---|:---|:---|:---|
| **분석 대상** | 정형화된 KPI 집계 데이터 (RDBMS) | 고객 행동, 매출 등 특정 결과 변수 | **이벤트 로그 (Case, Activity, Time)** |
| **결과물 (Output)**| 대시보드 (막대/파이 차트) | 예측 모델 (의사결정 나무, 클러스터링) | **프로세스 다이어그램 (End-to-End 흐름)** |
| **핵심 질문** | "이번 달 매출이 얼마인가?" | "어떤 고객이 이탈할 것인가?" | **"업무가 어떤 순서로 진행되며 어디서 막히는가?"** |
| **시간의 연속성** | 스냅샷 (특정 시점 점검) | 독립적 상관관계 분석 | **완벽한 시간적 전후 인과관계 (Sequence)** 중심 |

### 2. 기술 융합: 프로세스 마이닝과 RPA (Robotic Process Automation)
기업들이 RPA 봇을 도입할 때 가장 큰 실패 원인은 "RPA를 어디에 적용해야 할지" 잘못 짚는 것이다. 빈도도 적고 예외도 많은 프로세스에 RPA를 억지로 붙이면 유지보수 비용만 폭증한다.

> 다음은 프로세스 마이닝이 RPA 도입의 내비게이션 역할을 하는 시너지 아키텍처 다이어그램이다.

```text
[Hyperautomation Pipeline: Process Mining + RPA]

[ ERP / CRM / SCM Systems ]
             │ (1. 추출: Event Logs)
             v
[[ Process Mining Engine (Celonis / ProM) ]]
             │
             ├──> (2. 분석) "반복적이고 예외가 없는 '단순 복붙' 구간 발견!"
             ├──> (3. 평가) "이 구간을 자동화하면 연간 3만 시간 절감 예상"
             v
[ RPA Development Team ]
             │ (4. 개발) 해당 구간 전용 봇 개발 및 배포
             v
[ RPA Bot (UiPath / Automation Anywhere) ]
             │ (5. 실행)
             v
(System 자동 입력 및 처리 완료)
```
- **해설**: 프로세스 마이닝은 RPA 봇이 뛰어들 '가장 ROI가 높은 타겟 사거리'를 데이터로 정확히 찍어준다. 최근에는 데스크톱의 키보드/마우스 클릭 로그까지 수집하는 **작업 마이닝(Task Mining)** 기술과 융합되어, "화면 클릭 수준의 초정밀 프로세스 도출"까지 지원한다.

> **📢 섹션 요약 비유**: 프로세스 마이닝은 공장 전체를 내려다보며 "저 컨베이어 벨트가 제일 느리고 단순 반복이네!"라고 짚어주는 '레이더'이고, RPA는 그 지점에 정확히 투하되어 일을 대신해 주는 '특수 부대 로봇'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오와 의사결정
- **상황**: 대형 병원의 '응급실 환자 진료 프로세스'. 평균 대기 시간이 4시간으로 환자 클레임이 폭주하는데, 의사와 간호사, 원무과는 서로 상대방 때문에 늦어진다고 주장함.
- **판단**: 각 부서의 인터뷰는 감정적이고 부정확하다. 병원의 HIS(병원정보시스템)에서 접수시간, 채혈시간, 엑스레이 촬영시간, 의사 진단시간 등 **이벤트 로그를 추출하여 프로세스 마이닝 엔진에 투입**한다. 분석 결과, "야간에 엑스레이를 찍고 결과를 판독의가 시스템에 올리기까지"의 구간에서 평균 2시간의 병목(Bottleneck)이 발생함을 100% 데이터로 입증했다. 이를 근거로 야간 판독의 증원 및 AI 판독 시스템 도입이라는 최적화(Enhancement) 의사결정을 내린다.

### 2. 도입 시 핵심 체크리스트 (Technical & Operational)
1. **데이터 가용성 및 식별자(Case ID) 일관성**: 구매(ERP) 시스템의 ID와 입고(WMS) 시스템의 ID가 다르면 프로세스가 중간에 끊겨버린다. 여러 시스템을 관통하는 End-to-End 분석을 위해 유일한 식별자(Case ID)를 매핑하는 ETL 파이프라인이 확보되었는가?
2. **데이터 품질 (Timestamp 해상도)**: 이벤트가 초(Second) 단위로 일어나는데, 시스템 로그가 분(Minute)이나 일(Day) 단위로만 기록된다면 순서를 전혀 알 수 없다. Timestamp의 해상도가 프로세스 속도보다 정밀한가?
3. **스파게티 맵의 단순화 (Abstraction)**: 마이닝 엔진이 뱉어낸 생(Raw) 다이어그램은 수천 개의 선이 꼬여있어 인간이 해석할 수 없다. 빈도가 낮은 20%의 예외 경로를 숨기고(Filtering) 핵심 해피 패스(Happy Path) 80%만 추려내는 뷰(View) 컨트롤 역량이 있는가?

### 3. 치명적 안티패턴 (Anti-Pattern)
- **과도한 감시 도구로의 변질 (Big Brother Anti-Pattern)**: 프로세스 마이닝을 '조직의 비효율 개선'이 아니라 '특정 직원의 게으름 적발' 도구로 사용하는 것.

> 아래는 프로세스 마이닝이 빅 브라더로 변질되었을 때 조직의 거부반응을 시각화한 상태도이다.

```text
[Anti-Pattern: The Big Brother Syndrome]

[Mining Engine] "김대리가 결재 올리는 데 다른 사람보다 3시간 더 걸림!"
       |
       v
[경영진] "김대리, 업무 태만으로 인사고과 감점 처리해."
       |
       v
[현업 직원들] "시스템이 우리를 감시한다!" -> (반발 및 우회 행동)
       |
       v
(직원들이 시스템에 가짜로 시간 맞춰 버튼만 누르고 실제 일은 나중에 함)
==> 이벤트 로그 전체가 가짜 데이터(Garbage)로 오염되어 마이닝 프로젝트 완전 실패.
```
- **해설**: 프로세스 마이닝의 대상은 '사람'이 아니라 '프로세스 구조 자체'여야 한다. 분석 단계에서는 Resource(사용자 이름)를 익명화(Anonymization)하여 개인 평가에 쓰이지 않도록 철저히 차단해야 현업의 협조와 진짜 데이터를 얻을 수 있다.

> **📢 섹션 요약 비유**: 블랙박스(프로세스 마이닝)를 설치하는 이유는 도로의 위험한 커브길(병목)을 찾아 고치기 위함이지, 운전자 개개인이 코를 파는지 감시해서 벌금을 매기기 위함이 아닙니다. 목적이 왜곡되면 운전자들은 블랙박스를 꺼버립니다.

### 1. 정량/정성적 기대효과

| 기대효과 구분 | 도입 전 (인터뷰/직관 의존) | 도입 후 (데이터/로그 기반) | ROI 및 변화 지표 |
|:---|:---|:---|:---|
| **분석 소요 시간** | 컨설턴트 3명이 2개월간 인터뷰 진행 | 데이터 연동 후 10분 만에 맵 렌더링 | As-Is 현황 파악(Discovery) 소요 시간 95% 단축 |

### 2. 미래 전망 및 아키텍처 진화
- **작업 마이닝(Task Mining)과의 완전 결합**: 기존에는 서버에 남은 큰 덩어리의 ERP 로그만 분석했다면, 최근에는 직원의 PC 백그라운드에 에이전트를 설치하여 엑셀, 이메일, 메신저 사용 내역(Keystroke, Click)까지 수집하여 프로세스 마이닝과 결합하는 PTM(Process & Task Mining) 융합 분석이 대세가 되고 있다.
- **실시간 실행형 마이닝 (Actionable/Execution Management)**: 단순히 과거의 데이터를 보여주는 대시보드(Dashboard)를 넘어, 마이닝 엔진이 "지금 접수된 주문 건은 과거 패턴상 병목에 빠질 확률이 90%입니다. 지금 당장 재고 부서에 독촉 알람을 쏘겠습니다"라며 **자율적 액션(Execution Management System, EMS)**을 취하는 AI 에이전트로 진화(예: Celonis EMS)하고 있다.

> **📢 섹션 요약 비유**: 프로세스 마이닝은 과거를 비추는 '거울'에서 출발하여, 현재의 병목을 진단하는 '엑스레이'를 거쳐, 미래의 체증을 예방하고 스스로 길을 뚫어주는 '자율 주행 내비게이션'으로 진화하고 있습니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- 이벤트 로그 (Event Log) | 프로세스 마이닝의 필수 원재료가 되는 데이터로 Case ID, Activity, Timestamp의 3원칙을 반드시 포함해야 하는 데이터 세트
- 적합성 검사 (Conformance Checking) | 사전에 정의된 이상적인 BPMN 모델과 실제 마이닝된 로그 모델을 비교하여 보안 위반이나 우회 경로를 탐지하는 마이닝 기법
- 작업 마이닝 (Task Mining) | 엔터프라이즈 시스템 로그(서버)가 아닌 개인 PC 데스크톱의 화면 클릭, 키보드 입력 로그를 분석하는 마이크로 단위의 분석 기법
- RPA (Robotic Process Automation) | 프로세스 마이닝으로 발견된 단순 반복 병목 구간을 해결하기 위해 투입되는 인간 행동 모방 소프트웨어 로봇
- Celonis (셀로니스) | 독일에서 시작된 세계 1위의 프로세스 마이닝 및 실행 관리 시스템(EMS) 상용 소프트웨어 기업

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 개미들이 마당에서 집으로 가는 길이 너무 많아서 헷갈릴 때, 개미들 발에 아주 작은 물감을 묻혀두는 거예요.
2. **원리**: 하루가 지나고 마당을 보면, 물감 자국(로그)이 가장 진하게 겹친 곳이 '개미들의 진짜 지름길'이고, 빙글빙글 꼬인 곳이 '길을 잃고 헤맨 곳(병목)'이라는 걸 그림으로 딱 알 수 있어요.
3. **효과**: 개미들에게 일일이 물어보지 않아도, 물감 자국만 분석하면 어디에 맛있는 과자(해결책)를 놔주고 어디에 다리를 놔줘야 할지 100% 정확하게 알 수 있답니다!
