# 적합성 검사 (Conformance Checking)

> **약어 (Abbreviation)**: Conformance Checking

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 적합성 검사 (Conformance Checking)는 회사에서 정해둔 **'이상적인 프로세스 모델(Ideal Model)'**과 정보 시스템에 기록된 **'실제 이벤트 로그(Reality)'**를 겹쳐서 비교하여, 두 데이터 간의 불일치(Deviation)를 수학적/통계적으로 측정하는 프로세스 마이닝 기법이다.
> 2. **가치**: 직원의 규정 위반, 필수 결재 누락, 권한 우회(Bypass) 등 눈에 보이지 않던 컴플라이언스(Compliance) 리스크와 내부 통제 실패 지점을 100% 팩트(데이터) 기반으로 실시간 적발할 수 있다.
> 3. **융합**: 전통적인 회계 감사가 샘플링 기반의 수작업이었다면, 적합성 검사는 **데이터 마이닝** 기술 및 **AI 이상 탐지(Anomaly Detection)**와 결합하여 전수 조사(100% Coverage) 기반의 **연속적 감사(Continuous Auditing)** 인프라를 완성한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 적합성 검사는 모델과 로그의 "틀린 그림 찾기"다. "A 다음에는 반드시 B를 해야 한다"는 회사 규정(모델)이 있을 때, 실제 시스템 로그에서 "A 다음에 C를 하고 B를 건너뛴" 케이스를 찾아내어 얼마나 일치(Fitness)하는지 점수로 환산하고 그 원인을 추적하는 기술이다.
- **💡 비유**: 교장 선생님이 정해준 '학교에서 집으로 가는 가장 안전한 길(표준 모델)'과, 학생들의 휴대폰 GPS 기록(이벤트 로그)을 지도 위에 겹쳐 보며, 어떤 학생이 오락실로 우회(위반)했는지 찾아내는 것과 같습니다.
- **등장 배경**:
  - **기존 한계**: 과거의 IT 시스템 감리나 회계 감사는 10만 건의 결재 데이터 중 무작위로 100건만 뽑아서 사람이 일일이 대조하는 샘플링(Sampling) 방식이었다. 이로 인해 교묘한 횡령이나 규정 위반을 적발할 확률이 극히 낮았다.
  - **혁신적 패러다임**: 프로세스 마이닝 기술이 발전하면서, 단순히 로그를 그림으로 그리는 도출(Discovery)을 넘어, 미리 그려진 모델(BPMN 등) 위에 로그 데이터를 통과시켜보는(Replay) 알고리즘 방식이 개발되었다.
  - **현재의 비즈니스 요구**: SOX(사베인스-옥슬리법), 내부회계관리제도 등 기업 컴플라이언스가 강화되면서, 외부 감사인들은 기업에 "모든 프로세스가 규정대로 흘러가고 있음을 데이터로 증명하라"고 요구하게 되었다.

> 다음은 기존의 샘플링 감사 방식과 적합성 검사의 전수 조사 방식을 비교한 도식이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [Manual Audit vs Conformance Checking]                      │
├──────────────────────────────┬──────────────────────────────┤
│ 1. Manual Audit (Sampling)   │ 2. Conformance Checking      │
│                              │                              │
│ [ 100,000 Transactions ]     │ [ 100,000 Event Logs ]       │
│         │ (수작업 추출)      │         │ (100% 자동 투입)   │
│         v                    │         v                    │
│ [ 100 Samples ]              │ [ Validation Algorithm ]     │
│         │                    │         | (Replay on Model)  │
│         v                    │         v                    │
│ "1건의 규정 위반 발견"       │ "1,245건의 5가지 위반 패턴 발견!"│
│ (나머지 99,900건은 블랙박스) │ (완벽한 가시성과 통계 점수 제공)│
└──────────────────────────────┴──────────────────────────────┘
```
- **해설**: 기존 감사는 '운이 나쁜' 위반자만 적발되었다. 반면 적합성 검사는 엔진이 10만 건의 트랜잭션을 표준 BPMN 모델 위에 흘려보내면서(Replay), 정해진 선을 벗어나는 모든 데이터를 0.1초 만에 튕겨내어(Deviation) 대시보드에 뿌려준다.

> **📢 섹션 요약 비유**: 수만 개의 불량품이 섞여 있는 컨베이어 벨트에서 사람이 눈으로 10개만 골라내서 검사하던 방식에서, 모든 제품의 크기와 무게를 스캔하는 '정밀 레이저 센서(적합성 검사)'를 통과시키는 방식으로 바뀐 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 핵심 구성 요소

적합성 검사가 성립하기 위해서는 두 가지 입력값(Input)과 이를 비교하는 알고리즘 엔진이 필요하다.

| 요소명 | 역할 | 내부 형태 및 특징 | 비유 |
|:---|:---|:---|:---|
| **Reference Model** | 비교의 기준이 되는 '정답지' | BPMN, Petri Net 등 사전에 정의되거나, 과거 데이터로 학습된 '이상적 모델' | 법전 / 채점표 |
| **Event Log** | 비교의 대상이 되는 '현실' | IT 시스템(ERP, CRM)에서 추출된 Case ID, Activity, Timestamp 세트 | 학생의 실제 답안지 |
| **Replay Algorithm** | 두 요소 간의 차이를 계산 | 토큰(Token)을 모델 위에 흘려보내 모자라거나 남는 토큰을 계산함 | 자동 채점기 |
| **Diagnostic Result** | 검사 후 출력되는 지표/결과 | Fitness(적합도), Precision(정밀도), Deviation(위반 패턴) 리스트 | 성적표 및 오답 노트 |

### 2. 토큰 리플레이 (Token Replay) 동작 원리

적합성 검사의 가장 대표적인 알고리즘은 **토큰 리플레이(Token Replay)** 기법이다. 이벤트 로그의 순서대로 프로세스 모델에 '토큰'을 올려보면서, 모델과 로그가 얼마나 일치하는지 페널티 점수를 매긴다.

> 아래 다이어그램은 "A -> B -> C"가 정답인 모델에 "A -> C"라는 불량 로그가 들어왔을 때 알고리즘이 어떻게 위반을 탐지하는지 보여준다.

```text
[Token Replay Mechanism]

1. Reference Model (정답):  [Start] ──> (A) ──> (B) ──> (C) ──> [End]
2. Incoming Event Log:      <Case 1: A, C> (B가 누락된 상황)

[ Replay Step-by-Step ]
- Step 1: 로그 'A' 실행 
  -> 모델의 (A) 위치에 토큰 1개 생성 (정상)
  
- Step 2: 로그 'C' 실행 요청
  -> 모델은 (A) 다음 무조건 (B)를 거쳐야 하므로 (C)로 토큰을 보낼 수 없음!
  -> 엔진 조치: (B)를 강제로 실행한 척(Missing Token 생성) 하고 (C)로 진행시킴
  -> 페널티 부과: Missing Token = 1 (B를 안 거침)
                 Remaining Token = 1 (B에 남겨진 가짜 토큰)
```
- **해설**: 리플레이 알고리즘은 로그가 모델의 화살표를 따라가지 못할 때마다, 가짜 토큰(Missing Token)을 만들거나 갈 곳 없는 토큰(Remaining Token)을 남겨둔다. 10만 건을 모두 돌린 후 이 페널티 토큰의 개수를 수학적으로 계산하여 `적합도(Fitness) 95%`와 같은 정량적 점수를 산출해 낸다.

### 3. 평가의 4가지 차원 (Metrics)
적합성을 평가할 때는 단순히 "맞다/틀리다"를 넘어 4가지 차원의 통계 지표를 뽑아낸다.
1. **Fitness (적합도)**: 로그의 흐름을 모델이 얼마나 다 수용할 수 있는가? (결석률 평가)
2. **Precision (정밀도)**: 모델이 로그에 없는 쓸데없는 예외 상황을 너무 많이 허용하고 있지는 않은가? (과도한 융통성 평가)
3. **Generalization (일반화)**: 모델이 과거 로그만 100% 맞추고 미래 로그는 못 맞추는 오버피팅(Overfitting) 상태는 아닌가?
4. **Simplicity (단순성)**: 모델이 스파게티처럼 너무 복잡하지 않고 직관적인가?

> **📢 섹션 요약 비유**: 미로 찾기(모델)에 쥐(이벤트 로그)를 풀어놓았을 때, 쥐가 미로 벽을 뚫고 지나가거나 담을 넘어버리면 그 지점에 '빨간 경고등(Missing Token)'을 켜서 나중에 어디가 뚫렸는지 확인하는 감지 시스템입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 프로세스 마이닝 3대 기법 내에서의 포지셔닝 비교

| 특성 | Discovery (도출) | Conformance (적합성) | Enhancement (향상) |
|:---|:---|:---|:---|
| **입력 데이터** | Event Log만 필요 | **Event Log + Reference Model** | Event Log + Model + 신규 자원 |
| **주요 목적** | 현실(As-Is)의 투명한 시각화 | **규정 준수 여부 진단 및 감사** | 병목 제거 및 성과(To-Be) 최적화 |
| **출력 결과물** | 프로세스 맵 (Spaghetti) | **위반 건수, Fitness 통계 점수** | 시뮬레이션 기반의 개선된 모델 |
| **주 사용 부서** | 현업 프로세스 기획팀 | **내부 감사팀, 보안/컴플라이언스팀**| 혁신 TF, IT 개발팀, RPA 팀 |

### 2. 적합성 검사와 보안/감사 아키텍처의 융합
적합성 검사는 단순히 경영 컨설팅 도구가 아니라, 차세대 **DevSecOps** 및 **연속적 감사(Continuous Auditing)**의 인프라로 쓰인다. 최근에는 룰 엔진(Rule Engine)과 결합하여, 위반 이벤트가 발생하면 주 단위로 보고하는 것이 아니라 **"발생 즉시 알람(Alert)"**을 쏘는 실시간 통제 시스템으로 진화했다.

> 다음은 ERP 시스템에서 적합성 검사 엔진이 실시간으로 부정 결재를 적발해 내는 감사 파이프라인 아키텍처이다.

```text
[Continuous Auditing Architecture via Conformance Checking]

    [ ERP System (SAP) ]
            │ 
            │ (Real-time CDC / Event Stream)
            v
[[ Conformance Checking Engine ]] <==== [ Corporate Compliance Rules ]
  - Log: (Order -> Pay)                   - Rule 1: 무조건 (Order -> Approve -> Pay)
  - Engine Replay 중!                     - Rule 2: 100만원 이상은 2인 결재

            │ (Deviation 탐지: "승인(Approve) 누락 건 발생!")
            v
     [ Alert Manager ]
            │ (Webhook)
            v
[ Audit Dashboard ] & [ Security Team Slack ]
```
- **해설**: 구매 담당자가 악의적으로(또는 실수로) '승인(Approve)' 단계를 건너뛰고 '결제(Pay)' 버튼을 조작하여 실행했을 때, 적합성 검사 엔진은 이 로그를 받자마자 참조 모델(Compliance Rule)과 대조한다. 토큰 리플레이에서 에러가 발생하면 즉각 보안팀의 슬랙(Slack)으로 알람이 날아가 대금을 즉시 동결시킬 수 있다.

> **📢 섹션 요약 비유**: 과거 감사가 1년에 한 번 영수증을 뒤져보는 '연말 정산'이었다면, 실시간 적합성 검사는 내 신용카드가 평소 안 가던 해외에서 긁히자마자 은행에서 바로 정지시키고 전화를 거는 '실시간 이상 거래 탐지 시스템(FDS)'과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오와 의사결정
- **상황**: 글로벌 제조사의 '외상매출금(A/R) 회수 프로세스'. 원칙상 물건을 배송하고 한 달 안에 돈을 받아야 하는데, 영업사원들이 실적을 위해 신용등급이 낮은 고객에게 임의로 외상을 주고 있음. 경영진은 어떤 사원이 규정을 우회하고 있는지 증거를 원함.
- **판단**: 모든 영업사원을 심문하는 대신, **적합성 검사(Conformance Checking)**를 수행한다. 본사가 정의한 표준 영업 모델(Reference Model)을 설정하고, Salesforce(CRM)와 SAP(ERP)의 이벤트 로그를 부어 넣는다. 분석 결과, "신용조회(Credit Check) 액티비티를 누락한 채 배송(Shipment)으로 직행한" 위반 패턴이 전체의 15%에 달하며, 특정 지점의 영업사원에게 집중되어 있음을 데이터(Case ID)로 완벽하게 증명하여 감사 조치를 내린다.

### 2. 도입 시 핵심 체크리스트 (Technical & Operational)
1. **정답 모델(Reference Model)의 존재 유무**: "틀림"을 찾아내려면 완벽한 "정답지"가 있어야 한다. 회사 내부에 최신화된 업무 매뉴얼이나 표준 BPMN 모델이 없는 상태라면 적합성 검사 자체가 불가능하다. (이 경우 Discovery를 먼저 수행해 정답지를 합의해야 함).
2. **이상(Anomaly)과 허용(Flexibility)의 경계 설정**: 적합도 점수가 낮게 나왔다고 무조건 직원이 잘못한 것은 아니다. 때로는 현실의 비즈니스가 바뀌었는데 구식 규정(모델)이 업데이트되지 않은 것일 수 있다. 규정 위반(Violation)인지, 아니면 긍정적인 현실의 융통성(Agility)인지 판별할 프로세스 오너(Process Owner)의 개입이 필수적이다.
3. **루프(Loop)와 핑퐁(Ping-pong) 추적**: "결재 반려 -> 재상신 -> 반려" 같이 무한 루프를 도는 재작업(Rework) 패턴이 모델에 규정되어 있지 않다면 엔진은 이를 전부 에러로 간주한다. 프로세스 모델에 적절한 '허용 루프'가 정의되어 있는가?

### 3. 치명적 안티패턴 (Anti-Pattern)
- **비현실적인 100% 일치 추구 (The 100% Fitness Trap)**: 모든 직원이 기계처럼 표준 모델(Happy Path)과 100% 일치하게 행동해야 한다고 강제하고, 1%의 예외만 발생해도 시스템을 셧다운시키거나 처벌하는 현상.

> 아래는 비현실적 강제로 인해 오히려 업무 효율이 마비되는 안티패턴 구조도이다.

```text
[Anti-Pattern: The 100% Fitness Trap]

(Reference Model) : 완벽한 일자형 파이프라인 (예외 0개)

(현실 발생) : VVIP 고객이 급하게 배송을 먼저 해달라고 요청함.
       |
       v
(직원) "모델에 따르면 선배송은 절대 불가입니다." -> (결과: VVIP 고객 계약 취소)
       |
       v
(Audit Team) "훌륭해! 우리 회사의 적합도(Fitness)는 이번 달도 100%야!" 
             (==> 프로세스는 완벽하지만 회사는 망하는 모순 발생)
```
- **해설**: 적합성 검사의 목적은 100% 통제가 아니라 **"예측 불가능한 위반의 가시화"**다. 비즈니스 현실에서는 긴급 출고, 선결제, 전결 등 수많은 '합리적 우회(Workaround)'가 필요하다. 100% 일치를 강제하면 기업의 애질리티(Agility)가 완전히 파괴된다.

> **📢 섹션 요약 비유**: 도로의 속도 제한 규정(모델)을 어기는 차량(로그)을 찾는 것은 중요하지만, 앰뷸런스나 소방차(예외적 합리성)가 속도위반을 했다고 해서 딱지를 떼고 막아서면 환자가 목숨을 잃게 되는 것과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성적 기대효과

| 기대효과 구분 | 도입 전 (수동 샘플링 감사) | 도입 후 (적합성 검사 알고리즘) | ROI 및 변화 지표 |
|:---|:---|:---|:---|
| **감사 커버리지** | 전체 트랜잭션의 1~5% 샘플 확인 | 100% 전수 데이터 리플레이 검증 | 부정 행위 및 규정 누락 탐지율 99.9% 달성 |
| **원인 분석 시간** | 위반자 탐색 후 면담을 통한 원인 추정 | 엔진이 위반 노드(Node)와 로그 병렬 매핑 | 규정 위반의 근본 원인(Root Cause) 파악 90% 단축 |

### 2. 미래 전망 및 아키텍처 진화
- **AI 기반 이상 행동 예측 (Predictive Conformance)**: 지금까지는 일이 끝난 뒤(사후)에 위반을 찾아냈다면, 최근 딥러닝(RNN, LSTM) 기반 적합성 검사는 진행 중인 프로세스의 앞단만 보고도 "이 케이스는 3단계 뒤에 보안 규정을 위반할 확률이 95%입니다"라고 선제적 경고를 날려 차단하는 방향으로 진화하고 있다.
- **오브젝트 중심 프로세스 마이닝 (OCPM, Object-Centric Process Mining)**: 기존에는 1개의 Case ID(예: 주문번호)만 추적할 수 있어 복잡한 B2B 거래의 적합성을 평가하기 어려웠다. 최근에는 주문-배송-청구서 등 다중 객체(Object)가 얽힌 3차원 네트워크 데이터의 적합성 위반을 동시에 찾아내는 OCPM 기술이 차세대 표준으로 상용화(Celonis 등)되고 있다.

> **📢 섹션 요약 비유**: 적합성 검사는 기업의 건강 상태를 수치로 알려주는 '체성분 분석기(InBody)'입니다. 뼈(표준 모델)와 근육(로그 데이터)이 얼마나 균형 있게 맞물려 있는지 매일 점검함으로써, 기업은 부상(컴플라이언스 위반) 없이 가장 무거운 역기(비즈니스 가치)를 들어 올릴 수 있습니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- 프로세스 마이닝 (Process Mining) | 이벤트 로그를 활용하여 도출(Discovery), 적합성(Conformance), 향상(Enhancement)을 수행하는 상위 기술 범주
- 토큰 리플레이 (Token Replay) | 적합도(Fitness)를 계산하기 위해 이벤트 로그를 정답 모델 위에 통과시키면서 남는 토큰과 모자란 토큰을 세는 핵심 알고리즘
- BPMN (Business Process Model and Notation) | 적합성 검사 시 '정답지(Reference Model)' 역할을 수행하기 위해 시각적/규칙적으로 엄격히 정의된 프로세스 모델링 표준 언어
- OCPM (Object-Centric Process Mining) | 단일 식별자(Case ID)의 한계를 극복하고 주문, 품목, 배송 등 여러 객체 간의 복잡한 위반 관계를 3차원으로 찾아내는 최신 마이닝 기법
- 연속적 감사 (Continuous Auditing) | 1년에 한 번 하는 회계 감사가 아니라, IT 시스템 로그 기반으로 매일, 매시간 적합성 검사를 돌려 내부 통제를 자동화하는 현대 감사 기법
