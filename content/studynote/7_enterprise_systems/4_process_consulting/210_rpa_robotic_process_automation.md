# RPA (Robotic Process Automation) 봇 아키텍처

> **약어 (Abbreviation)**: RPA (Robotic Process Automation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RPA (Robotic Process Automation)는 기존 IT 시스템의 백엔드(API, DB)를 뜯어고치는 대신, 사람이 모니터 화면에서 마우스와 키보드로 수행하는 **프론트엔드 UI 조작(클릭, 복사/붙여넣기)을 소프트웨어 봇(Bot)이 흉내 내어 자동화**하는 기술이다.
> 2. **가치**: 기존 레거시 시스템을 전혀 건드리지 않는 **비침투적(Non-invasive)** 특성 덕분에 매우 빠르고 저렴하게 도입할 수 있으며, 직원을 단순 반복 노동(Swivel-chair work)에서 해방시켜 고부가가치 창조 업무에 집중하게 만든다.
> 3. **융합**: 초기에는 단순 매크로에 불과했으나, 현재는 **OCR(광학 문자 인식)**, **NLP(자연어 처리)**, **프로세스 마이닝**과 결합하여 스스로 문서를 읽고 판단하는 지능형 자동화인 **IPA (Intelligent Process Automation)** 및 초자동화(Hyperautomation)의 핵심 실행(Execution) 엔진으로 진화했다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: RPA는 물리적인 깡통 로봇이 아니라, 컴퓨터 내에서 작동하는 '소프트웨어 로봇'이다. 엑셀의 데이터를 복사해서 ERP 화면에 붙여넣고 확인 버튼을 누르는, 규칙이 명확한(Rule-based) 단순 업무를 사람보다 10배 빠르고 실수 없이 수행한다.
- **💡 비유**: 회사에 엑셀 복사/붙여넣기만 하루 종일 하는 아르바이트생을 고용했는데, 이 아르바이트생은 투명 인간(소프트웨어)이고 밥도 안 먹고 잠도 안 자고 1초에 100건씩 일하는 것과 같습니다.
- **등장 배경**:
  - **기존 한계**: A 시스템(예: 홈택스)의 데이터를 B 시스템(사내 ERP)으로 옮기려면 막대한 비용을 들여 API(EAI)를 개발해야 했다. 하지만 구형 레거시 시스템은 API 자체가 없었고, 결국 직원이 모니터 두 대를 띄워놓고 하루 종일 수작업으로 데이터를 옮겨 적는 '의자 돌리기(Swivel-chair) 업무'가 팽배했다.
  - **혁신적 패러다임**: 화면의 픽셀 단위 이미지나 HTML/Windows UI 요소를 정확히 인식하여 클릭하는 **컴퓨터 비전(Computer Vision) 및 UI 자동화 기술**이 상용화되면서, 백엔드를 고칠 필요 없이 프론트엔드에서 봇을 돌리는 RPA가 폭발적으로 성장했다(UiPath, Automation Anywhere 등).
  - **현재의 비즈니스 요구**: 주 52시간 근무제 도입, 인건비 상승, 저출산 고령화로 인해 '단순 반복 업무'를 할 인력을 구하기 어려워진 기업들이 디지털 노동력(Digital Workforce) 확보를 위해 전사적으로 RPA를 스케일아웃(Scale-out)하고 있다.

> 다음은 기존 API 기반의 시스템 통합(EAI)과 RPA 기반의 UI 통합 방식을 보여주는 비교 도식이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [API Integration (EAI) vs UI Integration (RPA)]             │
├──────────────────────────────┬──────────────────────────────┤
│ 1. API Integration (기존)    │ 2. UI Integration (RPA)      │
│                              │                              │
│ [ System A ]                 │ [ System A ]  [ System B ]   │
│      | (Backend)             │      |             ^         │
│     API                      │   (UI 화면)     (UI 화면)    │
│      |                       │      v             |         │
│ [ System B ]                 │  +-----------------------+   │
│ (개발 비용 높음, 시간 오래걸림)│  │ [ RPA Bot (로봇) ]    │   │
│                              │  │ "A 화면 복사 -> B 붙여넣기" │   │
│                              │  +-----------------------+   │
│                              │ (개발 비용 낮음, 즉시 적용)  │
└──────────────────────────────┴──────────────────────────────┘
```
- **해설**: API 방식은 두 시스템의 코드를 직접 연결하므로 완벽하지만 비싸고 느리다. 반면 RPA는 '사람인 척' 화면을 보고 조작하므로 코드를 건드리지 않고(Non-invasive) 매우 빠르고 싸게 시스템을 연결(통합)하는 효과를 낸다.

> **📢 섹션 요약 비유**: 두 건물(시스템)을 연결하기 위해 포크레인으로 땅을 파서 수백억짜리 지하 터널(API)을 뚫는 대신, 오토바이 배달부(RPA)를 고용해서 건물 정문(화면 UI)으로 문서를 실어 나르게 하는 가장 가성비 좋은 해결책입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 핵심 구성 요소 (RPA Architecture)

대규모 기업용 RPA(Enterprise RPA) 아키텍처는 봇(Bot) 단독으로 움직이는 것이 아니라 크게 3가지 컴포넌트로 구성된다.

| 요소명 | 역할 | 내부 형태 및 특징 | 비유 |
|:---|:---|:---|:---|
| **Studio (개발 도구)** | 봇의 작업 흐름 생성 | 드래그 앤 드롭 형태의 순서도(Flowchart) UI를 제공하여 로직(If, Loop)을 개발 (예: UiPath Studio) | 로봇 조립 공장 / 학원 |
| **Orchestrator (관제소)**| 수많은 봇의 중앙 통제 및 스케줄링 | 언제 어느 봇을 깨워 어떤 스크립트를 실행할지 지시하고, 에러와 로그를 관리하는 중앙 웹 서버 | 로봇 관제탑 / 매니저 |
| **Robot / Agent (실행기)**| 실제 업무 수행 | 사용자의 PC나 백그라운드 서버에 설치되어 Orchestrator의 명령에 따라 스크립트를 실행 | 파견된 로봇 일꾼 |

### 2. Attended Bot vs Unattended Bot의 동작 원리

RPA 봇은 사람이 곁에서 도와줘야 하는지에 따라 두 가지 아키텍처로 나뉜다.

> 아래 다이어그램은 Attended Bot과 Unattended Bot이 시스템과 어떻게 상호작용하는지 보여주는 아키텍처 비교도이다.

```text
[RPA Bot Execution Types]

[A] Attended Bot (협업형) - Front-office
 (직원 김대리 PC)
   1. 김대리가 콜센터 전화를 받음.
   2. 김대리가 단축키(Ctrl+Shift+R)를 눌러 봇 실행!
        │
        v
   [RPA Bot] -> 3. 고객 번호를 읽어 CRM/ERP 화면을 1초만에 다 띄워줌.
   4. 김대리가 화면을 보고 최종 판단 및 고객 응대.

[B] Unattended Bot (무인형) - Back-office
 (서버실의 가상 머신 VM)
   1. [Orchestrator]가 새벽 2시에 타이머 작동!
        │
        v (트리거)
   [RPA Bot] -> 2. 스스로 가상 윈도우에 로그인(Unlock)
             -> 3. 전날 매출 엑셀 다운로드 및 ERP 전표 1,000건 입력
             -> 4. 작업 완료 후 이메일 결과 보고 후 로그아웃
```
- **해설**: **Attended Bot**은 사람의 PC에 깔려 사람의 일을 '거들어주는' 조수 역할을 한다. 반면 **Unattended Bot**은 사람이 볼 필요가 없는 독립된 가상 머신(VM) 서버에서 중앙 관제탑의 스케줄에 따라 24시간 무인으로 돌아가는 일꾼이다. 엔터프라이즈 환경에서는 ROI가 압도적으로 높은 Unattended Bot을 주로 사용한다.

### 3. 화면 인식(UI Automation) 메커니즘
RPA는 어떻게 버튼을 찾을까? 단순 매크로(예: X=100, Y=200 픽셀 클릭)는 창이 조금만 이동해도 에러가 난다. 최신 RPA는 **Selector(선택자)** 기술을 사용하여, HTML의 DOM 구조나 윈도우 응용 프로그램의 API 구조(UI Automation API)를 긁어와 "아이디가 'btnSubmit'인 객체를 클릭해라"라고 코딩하므로 창의 위치나 해상도가 바뀌어도 정확히 버튼을 찾아낸다.

> **📢 섹션 요약 비유**: Attended 봇은 요리사(사람) 옆에서 파를 대신 썰어주는 '보조 셰프'이고, Unattended 봇은 모두가 퇴근한 새벽 공장에서 밤새 밀가루 반죽을 1만 개 만들어두는 '자동화 믹서기'입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 매크로(Macro) vs 엔터프라이즈 RPA 비교

| 비교 항목 | 엑셀 매크로 (VBA) / 단순 오토마우스 | 엔터프라이즈 RPA (UiPath, AA 등) | 시너지 및 진화 |
|:---|:---|:---|:---|
| **작동 범위** | 해당 프로그램(예: 엑셀 내부)에서만 동작 | 엑셀, SAP, 웹, 이메일 등 **모든 앱을 넘나들며 동작** | Cross-application 통합 능력 |
| **통제/보안** | 개인 PC에서 알 수 없이 돎 (보안 위협) | Orchestrator가 암호(Credential)를 은닉하고 **로그를 중앙 통제** | 강력한 감사(Audit) 및 보안성 |
| **에러 핸들링**| 예외 발생 시 스크립트 중단(Crash) | Try-Catch 로직, 스크린샷 캡처 후 다음 큐(Queue)로 이동 | 무중단(Fault-Tolerant) 설계 |
| **지능성** | 룰 기반 (정해진 좌표, 셀만 클릭) | AI, OCR 융합으로 영수증 글자 판독 후 자율 행동 | 단순 RPA에서 IPA(지능형)로 진화 |

### 2. 기술 융합: AI, OCR, NLP와 결합된 IPA (Intelligent Process Automation)
전통적인 RPA의 치명적인 한계는 "비정형 데이터"를 읽지 못한다는 것이다. "종이 영수증"이나 "고객의 환불 요청 이메일"을 RPA는 이해할 수 없다. 이를 극복하기 위해 최근 RPA는 AI와 결합된 **IPA (Int능형 프로세스 자동화)**로 진화했다.

> 다음은 전통적 RPA가 AI(OCR/NLP)와 결합하여 비정형 업무를 자동화하는 파이프라인 구조도이다.

```text
[Intelligent Process Automation (IPA) Pipeline]

(Unstructured Data)
[고객 이메일 도착] ──> [ RPA Bot ] "이메일 첨부파일(PDF 영수증)을 다운로드한다"
                          │
                          v
                   [ AI / OCR Engine ] (ABBYY, Google Vision)
                   "PDF 영수증 이미지에서 '금액: 50,000원'을 텍스트로 추출"
                          │
                          v
                   [ NLP Engine ] (LLM, GPT)
                   "이메일 본문 분석 결과, 고객의 의도는 '환불(Refund)'입니다"
                          │
                          v (Structured Data로 변환됨)
                   [ RPA Bot ]
                   "의도가 환불이고 금액이 5만 원이니, ERP 환불 화면에 
                    데이터를 입력하고 '승인' 버튼을 클릭한다."
```
- **해설**: RPA는 팔다리(Execution) 역할을 하고, AI는 눈과 뇌(Cognitive) 역할을 한다. 이 둘이 결합되면서 과거에는 사람이 반드시 읽고 판단해야 했던 '인지적 업무' 영역까지 봇이 처리할 수 있게 되었다(초자동화, Hyperautomation).

> **📢 섹션 요약 비유**: 기존 RPA가 눈을 감고 정해진 위치의 버튼만 누르는 '장님 로봇'이었다면, AI가 융합된 IPA는 눈(OCR)으로 영수증을 읽고 머리(NLP)로 내용을 이해해서 스스로 행동을 결정하는 '인조인간'으로 진화한 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오와 의사결정
- **상황**: 물류 회사에서 매일 아침 협력사 50곳에서 각기 다른 양식의 엑셀 주문서가 이메일로 500통 날아온다. 직원 3명이 오전 내내 엑셀을 열고, 데이터를 복사해 사내 발주 시스템(웹)에 손으로 옮겨 적고 있음. (휴먼 에러 빈발, 단순 노동으로 퇴사율 높음).
- **판단**: 완벽한 RPA 적용 대상이다(Rule-based, High volume, Repetitive). **Unattended RPA 봇**을 도입하여 새벽 5시에 스스로 이메일을 읽고 엑셀을 병합하여 사내 웹 시스템에 입력하도록 개발한다. 직원 출근 전인 8시에 모든 작업이 끝나 있어, 직원들은 단순 타이핑 대신 '재고 부족 고객에게 전화 걸기' 같은 고부가가치 응대 업무에만 집중하도록 직무를 재설계(Redesign)한다.

### 2. 도입 시 핵심 체크리스트 (Technical & Operational)
1. **Rule의 명확성 (Standardization)**: "담당자의 직감에 따라 할인율을 5~10% 유동적으로 준다"는 룰은 RPA로 절대 구현할 수 없다. RPA 도입 전, 업무 프로세스가 100% 명확한 논리(If-Then)로 문서화되어 있는가?
2. **시스템 변경 내성 (Maintenance Overhead)**: RPA는 화면(UI)의 버튼 위치나 HTML 태그에 의존한다. ERP나 웹사이트가 내일 갑자기 업데이트되어 버튼 이름이 바뀌면 봇은 100% 중단(Crash)된다. 타겟 시스템의 잦은 UI 변경 가능성을 파악하고, 에러 시 대응할 RPA 유지보수 인력(CoE 조직)이 있는가?
3. **권한과 보안 (Credential Management)**: Unattended 봇이 시스템에 로그인하려면 아이디와 비밀번호가 필요하다. 봇의 스크립트 안에 비밀번호를 평문(Plain text)으로 코딩하지 않고, CyberArk나 Orchestrator의 보안 볼트(Vault)에서 암호화하여 런타임에 동적으로 가져오게 설계했는가?

### 3. 치명적 안티패턴 (Anti-Pattern)
- **깨진 프로세스 위에 봇 덧씌우기 (Automating Broken Processes)**: 쓸데없는 결재, 불필요한 엑셀 다운로드 등 비효율의 극치인 기존 업무(As-Is)를 개선(BPR)하지 않고, 그 '비효율적인 행동 그대로' RPA 봇에게 코딩해버리는 현상.

> 아래는 프로세스 개선 없이 RPA를 도입했을 때 발생하는 자원 낭비 안티패턴 도식이다.

```text
[Anti-Pattern: Automating the Garbage]

(기존 비효율 업무) : 시스템에서 A 리포트 엑셀 다운 -> B 리포트 엑셀 다운 -> 수동 VLOOKUP -> 출력
        |
        v
(잘못된 RPA 도입) : RPA가 A 다운 -> B 다운 -> VLOOKUP 로직을 똑같이 모방함. 
                  (봇 개발에 3개월 소요, 엑셀 켜고 닫느라 봇 구동 시간도 오래 걸림)
        |
        v
(올바른 BPR+RPA 접근) : "애초에 왜 다운받아 합치지? DB에서 SQL 조인 쿼리 한 방이면 끝나는 거 아님?"
                  -> 시스템에 View 화면 1개 신규 개발 (하루 소요) -> RPA 도입 불필요!
```
- **해설**: 빌 게이츠는 "존재하지 말아야 할 일을 자동화하는 것만큼 어리석은 일은 없다"고 했다. RPA는 만병통치약이 아니다. RPA 도입 전 반드시 **프로세스 마이닝** 등을 통해 업무를 간소화(Lean)하고, 굳이 로봇이 UI를 클릭할 필요 없이 API나 DB로 해결할 수 있다면 그게 최우선 정답이다.

> **📢 섹션 요약 비유**: 물이 줄줄 새는 깨진 독(비효율 프로세스)에, 바가지(사람) 대신 최신식 모터 펌프(RPA 봇)를 달아 물을 붓는 격입니다. 펌프를 사기 전에 독부터 새 걸로 교체하는 것(프로세스 재설계)이 먼저입니다.

### 1. 정량/정성적 기대효과

| 기대효과 구분 | 도입 전 (인간에 의한 수작업) | 도입 후 (RPA 봇 자동화) | ROI 및 변화 지표 |
|:---|:---|:---|:---|
| **생산성 및 비용 (FTE)** | 하루 8시간 작업, 야근 수당 발생 | 24시간/365일 무휴 작업 수행 | 봇 1대가 평균 3~5인분(FTE) 노동력 대체 (비용 70% 절감) |
| **정확도 (Quality)** | 타이핑 오타 및 금액 누락 발생 (휴먼 에러) | 로직에 따른 100% 무결점 복사/입력 | 데이터 입력 불량률 0% 및 재작업 제거 |
| **직원 만족도 (EX)** | 단순 반복 노동(Ctrl+C, V)으로 인한 번아웃 | 기획, 분석, 소통 등 창의적 업무 집중 | 퇴사율 감소 및 직원 경험(Employee Experience) 향상 |

### 2. 미래 전망 및 아키텍처 진화
- **작업 마이닝 (Task Mining)과의 융합**: 과거에는 "어느 업무를 RPA로 만들까?"를 사람이 찾아다녔지만, 이제는 직원의 PC에 에이전트를 깔아 클릭 로그를 모으는 '태스크 마이닝' 도구가 "이 직원은 엑셀과 SAP 사이를 하루 500번 오갑니다. 여길 자동화하세요"라고 RPA 타겟을 자동으로 추천하고, 심지어 RPA 뼈대 코드를 자동 생성해 주는 수준에 이르렀다.
- **Agentic AI로의 진화**: 기존 RPA가 사람이 그려준 순서도(Flowchart) 안에서만 움직였다면, 미래의 로봇은 LLM(대형 언어 모델)이 장착된 **AI Agent**가 되어, "오늘 입금 안 된 거래처에 독촉 메일 다 보내놔"라는 자연어 명령 하나만으로 스스로 ERP를 조회하고 이메일을 작성하는 자율 주행 봇(Autonomous Agent)으로 진화하고 있다.

> **📢 섹션 요약 비유**: RPA는 기업이라는 거대한 공장에 투입된 1세대 '초정밀 산업용 로봇 팔'입니다. 이 로봇에 카메라(OCR)와 인공지능 두뇌(LLM)가 결합되면서, 미래에는 지시하지 않아도 알아서 일을 찾아 하는 '자율형 안드로이드 일꾼'으로 거듭날 것입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- 초자동화 (Hyperautomation) | RPA를 필두로 AI, 머신러닝, 프로세스 마이닝 등 모든 가용한 기술을 결합하여 기업의 모든 프로세스를 끝에서 끝까지(End-to-End) 자동화하려는 글로벌 IT 트렌드
- 프로세스 마이닝 (Process Mining) | RPA를 맹목적으로 도입하기 전, 이벤트 로그를 통해 병목을 파악하고 로봇이 투입될 가장 효과적인 타겟(Target)을 짚어주는 사전 분석 기술
- 광학 문자 인식 (OCR, Optical Character Recognition) | 종이 영수증이나 이미지화된 PDF의 글자를 텍스트 데이터로 변환하여 RPA 봇이 시스템에 입력할 수 있게 해주는 필수 인지(Cognitive) 기술
- 데스크톱 애널리틱스 (Desktop Analytics / Task Mining) | 직원의 PC 화면의 키스트로크와 클릭을 분석하여, RPA로 자동화하기 좋은 단순 반복 작업을 자동 발굴하는 기술
- 시민 개발자 (Citizen Developer) | 전문 IT 개발자가 아니더라도, UiPath Studio X 나 Power Automate 같은 로우코드/노코드 툴을 이용해 자신의 RPA 봇을 직접 만들어 쓰는 현업 실무자
