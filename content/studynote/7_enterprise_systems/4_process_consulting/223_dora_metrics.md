# 애자일 성과 관리 지표 DORA Metrics

> **약어 (Abbreviation)**: DORA (DevOps Research and Assessment) Metrics

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DORA Metrics는 구글(Google Cloud) 산하의 DORA 연구팀이 7년간 전 세계 3만 명 이상의 IT 전문가 데이터를 분석하여 도출해 낸, **소프트웨어 개발/운영 조직의 성과를 측정하는 글로벌 표준 4대 지표**다.
> 2. **가치**: "우리 개발팀이 얼마나 일을 잘하고 있는가?"라는 모호한 질문을 폐기하고, 개발 속도를 나타내는 **배포 빈도(DF), 변경 리드 타임(LTFC)**과 서비스 안정성을 나타내는 **변경 실패율(CFR), 복구 시간(MTTR)**이라는 구체적인 수치로 증명한다.
> 3. **융합**: 속도(Agile)와 안정성(Ops)은 서로 트레이드오프(Trade-off) 관계라는 기존의 편견을 깨고, **CI/CD 자동화**와 **데브옵스(DevOps)** 문화를 융합하여 두 마리 토끼를 동시에 잡는 하이퍼포머(Elite) 조직으로 도약하기 위한 북극성 지표(North Star Metric)로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: DORA Metrics는 개발팀의 실력을 측정하는 '체력장 점수표'다. 코드를 얼마나 자주(빠르게) 서버에 올리며, 올린 코드가 에러를 낼 확률은 얼마나 되고, 에러가 났을 때 얼마나 빨리 고치는지 딱 4가지 숫자로 팀을 평가한다.
- **💡 비유**: 피자 가게의 실력을 평가할 때 "피자를 몇 판 구웠나(코드 라인 수)"가 아니라, "주문부터 배달까지 얼마나 빠른가(속도)", "배달하다가 피자를 엎을 확률은 얼마인가(안정성)"를 측정하여 찐 맛집을 가려내는 것과 같습니다.
- **등장 배경**:
  - **기존 한계**: 과거(폭포수 시대) 개발팀의 성과는 '코드 라인 수(LoC)'나 '버그 발생 건수'로 평가되었다. 이는 개발자들이 쓸데없이 코드를 길게 짜거나, 버그를 숨기기 위해 배포를 1년에 한 번만 하게 만드는 끔찍한 부작용(모럴 해저드)을 낳았다.
  - **혁신적 패러다임**: 2014년 니콜 포스그렌(Nicole Forsgren), 즈네 킴(Gene Kim) 등이 설립한 DORA 팀이 『Accelerate(엑셀러레이트)』라는 책을 통해, "성공적인 데브옵스 조직은 속도와 안정성을 동시에 높인다"는 사실을 4가지 정량 지표로 수학적으로 입증해 냈다.
  - **현재의 비즈니스 요구**: 모든 기업이 애자일(Agile)을 도입했다고 주장하지만, 실제로는 이름만 애자일인 경우가 많다(Fake Agile). DORA Metrics는 이사회가 IT 부서의 애자일 성숙도를 팩트(데이터) 기반으로 감시하고 평가하는 가장 완벽한 대시보드로 자리 잡았다.

> 다음은 기존의 잘못된 측정 방식과 DORA Metrics의 철학적 차이를 보여주는 비교 도식이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [Traditional Metrics vs DORA Metrics]                       │
├──────────────────────────────┬──────────────────────────────┤
│ 1. Traditional (Vanity)      │ 2. DORA Metrics (Value)      │
│                              │                              │
│ [ 평가 지표 ]                │ [ 평가 지표 ]                │
│ - 작성한 코드 줄 수 (LoC)    │ - 배포 빈도 (속도)           │
│ - 근무/야근 시간             │ - 변경 리드 타임 (속도)      │
│ - 발견된 버그 개수           │ - 변경 실패율 (안정성)       │
│                              │ - 장애 복구 시간 (안정성)    │
│                              │                              │
│ * 결과: 배포를 두려워함      │ * 결과: 작은 단위로 빠르게   │
│   (장애 나면 혼나니까 뭉쳐서 │   배포하고, 장애가 나면 즉시 │
│    한 번에 배포 -> 대형장애) │   롤백하는 두려움 없는 문화  │
└──────────────────────────────┴──────────────────────────────┘
```
- **해설**: 전통적 지표(Vanity Metrics)는 개발자를 위축시킨다. DORA 지표는 "에러를 내지 마라!"가 아니라 "에러가 나도 빨리 고칠 수 있는 시스템(MTTR)을 만들어라!"에 초점을 맞추어 배포의 두려움을 없애준다.

> **📢 섹션 요약 비유**: 야구 선수를 평가할 때 "땀을 얼마나 많이 흘렸나(야근 시간)"를 보는 것은 의미가 없습니다. "타율이 몇 할인가(배포 빈도)", "수비 에러율이 몇인가(변경 실패율)"라는 명확한 데이터(DORA)만이 선수의 진짜 연봉을 결정합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. DORA 4대 지표의 구체적 의미 (The 4 Key Metrics)

DORA 매트릭스는 크게 **속도(Throughput/Velocity)**와 **안정성(Stability)**의 두 축으로 나뉜다.

| 분류 | 지표명 (Full Name) | 측정 내용 | Elite 등급 기준 | 비유 |
|:---|:---|:---|:---|:---|
| **속도** | **DF** (Deployment Frequency) | **배포 빈도**: 운영(Production) 서버에 코드를 얼마나 자주 배포하는가? | 하루에 여러 번 (On-demand) | 하루에 택배를 몇 번 출고하는가 |
| **속도** | **LTFC** (Lead Time for Changes) | **변경 리드 타임**: 코드가 커밋(Commit)된 순간부터 운영 환경에 배포되기까지 걸리는 시간 | 1시간 이내 | 주문 접수부터 문 앞 배송까지의 시간 |
| **안정성**| **CFR** (Change Failure Rate) | **변경 실패율**: 배포된 코드 중 서비스 장애(버그, 롤백)를 일으킨 비율 | 0% ~ 15% 사이 | 배송된 물건 중 반품/환불되는 비율 |
| **안정성**| **MTTR** (Mean Time to Recovery) | **평균 복구 시간**: 시스템에 장애가 났을 때 정상으로 복구(Restore)하는 데 걸리는 시간 | 1시간 이내 | 사고 났을 때 수리센터에서 고치는 시간 |

### 2. DORA 성숙도 4단계 (Four Profiles of Performance)

DORA 리포트는 전 세계 수만 개 팀의 위 4가지 데이터를 수집하여, 조직을 4가지 등급(Profile)으로 나눈다.

> 아래 다이어그램은 Low(하위) 조직과 Elite(최상위) 조직의 극단적인 격차를 보여주는 성숙도 모델이다.

```text
[DORA Performance Profiles (Low vs Elite)]

[ Low Performer ]
 - 배포 빈도 (DF): 1달 ~ 6개월에 한 번
 - 리드 타임 (LTFC): 1달 ~ 6개월 (무거운 결재 라인)
 - 실패율 (CFR): 46% ~ 60% (배포의 절반이 롤백됨)
 - 복구 시간 (MTTR): 1주일 이상 (장애 나면 전원 밤샘 패닉)
        │
        v (DevOps 및 CI/CD 인프라 고도화)
        │
[ Elite Performer (Amazon, Netflix 등) ]
 - 배포 빈도 (DF): 개발자가 원할 때 수시로 (하루 수천 번)
 - 리드 타임 (LTFC): 1시간 이내 (자동화 테스트 및 배포)
 - 실패율 (CFR): 0% ~ 15% 미만 (카나리 배포로 사전 차단)
 - 복구 시간 (MTTR): 1시간 이내 (버튼 클릭 한 번으로 이전 버전 롤백)
```
- **해설**: 많은 경영진이 "배포를 하루에 100번이나 하면, 서버가 맨날 터지지 않냐?"고 오해한다. 하지만 데이터는 정반대다. Elite 조직은 코드를 아주 작게 쪼개서(Small Batch) 배포하므로, 에러가 나도 어디서 났는지 1초 만에 파악하고 즉시 복구(MTTR 단축)한다. 즉, **속도가 곧 안정성을 만든다.**

### 3. LTFC (변경 리드 타임)의 시작점 논란
가장 헷갈리기 쉬운 지표가 LTFC다. "기획팀이 아이디어를 낸 시점"부터 잴 것인가? DORA는 철저히 IT 실행력을 측정하므로, 기획 단계의 지연을 배제하고 오직 **"개발자가 코드를 작성하여 Git에 Commit(Push)한 순간부터"** 시계를 누른다.

> **📢 섹션 요약 비유**: 아마추어 요리사(Low)는 재료를 한꺼번에 산더미처럼 냄비에 넣고 끓여서 맛이 이상해지면 원인을 몰라 다 버립니다. 미슐랭 셰프(Elite)는 소금을 0.1그램 단위로 자주(DF) 넣고 맛을 보며, 짜면 즉시 물을 부어(MTTR) 완벽한 요리를 유지합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 성과 측정 지표 비교 (DORA vs SPACE vs Flow Metrics)

DORA Metrics가 너무 코드 배포(기계적)에만 치중되어 있다는 비판이 나오면서, 최근 이를 보완하는 지표들이 등장했다.

| 지표/프레임워크 | 핵심 초점 (Focus) | 측정 항목 예시 | 시너지 포인트 |
|:---|:---|:---|:---|
| **DORA Metrics** | 시스템의 **배포 파이프라인 속도 및 안정성** | DF, LTFC, CFR, MTTR | CI/CD 시스템 레벨의 최적화 |
| **SPACE Framework** | 개발자의 **행복도 및 팀 협업 (인간 중심)** | 만족도(Satisfaction), 협업(Collaboration) | 번아웃 방지 (DORA의 기계적 압박 보완) |
| **Flow Metrics (VSM)**| 기획부터 고객 전달까지의 **비즈니스 가치 흐름** | Flow Time (기획~배포 전체), Flow Efficiency | DORA가 놓친 '기획 대기 시간' 등 비즈니스 병목 탐지 |

### 2. 기술 융합: DORA와 CI/CD 도구 체인 (Jenkins, GitLab, ArgoCD)
DORA Metrics는 사람이 엑셀로 기입하는 것이 아니다. 완벽한 데브옵스 인프라와 결합되어 모든 지표가 대시보드(예: Datadog, Grafana)에 자동 수집되어야 한다.

> 다음은 CI/CD 파이프라인에서 DORA 지표가 어떻게 자동 수집되는지 보여주는 아키텍처 다이어그램이다.

```text
[DORA Metrics Automated Data Pipeline]

[ Developer ]
  │ 1. Git Commit (Time A 기록 시작!)
  v
[[ Source Code Mgt (GitLab/GitHub) ]] ──────┐
  │                                         │
  v 2. Automated Testing & Build            │ (Webhooks)
[[ CI Server (Jenkins / GitHub Actions) ]] ─┼──> [[ DORA Metrics Engine ]]
  │                                         │    (Grafana / Datadog)
  v 3. Deploy to Prod (Time B 기록 종료!)   │    - LTFC = Time B - Time A
[[ CD Tool (ArgoCD / Spinnaker) ]] ─────────┘    - DF = 배포 카운트 + 1
  │
  v 4. Production Running
[[ APM / Monitoring (Datadog) ]] ──────────────> - 장애 발생 알람 (MTTR 시계 작동!)
                                                 - 롤백 발생 횟수 카운트 (CFR 증가)
```
- **해설**: 개발자가 커밋하는 순간 첫 번째 타이머가 돌고(LTFC 시작), 배포 성공 시 타이머가 멈춘다. 운영 중인 서버에서 에러 로그가 폭주하면 두 번째 타이머(MTTR)가 돌고 롤백 시 멈춘다. 이 모든 것이 100% 자동화되어 대시보드에 뿌려진다.

> **📢 섹션 요약 비유**: 마라톤 선수가 손목에 찬 스마트워치(CI/CD 파이프라인)입니다. 선수가 직접 "나 1km 뛰었어"라고 일기장에 적을 필요 없이, 맥박, 속도, 위치가 GPS를 통해 코치의 태블릿(DORA 대시보드)으로 실시간 전송되는 원리입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오와 의사결정
- **상황**: 대기업의 e커머스 사업부장. "왜 우리 회사는 쿠팡처럼 신기능이 매일 안 나오고 두 달에 한 번씩 나오냐!"고 IT 본부를 질타함. IT 본부는 "보안팀과 QA팀의 결재를 받아야 해서 어쩔 수 없다"고 핑계를 댐.
- **판단**: 누가 잘못했는지 감정 싸움을 멈추고 **DORA Metrics 측정 대시보드**를 도입한다.
- **결과**: 측정 결과 LTFC(변경 리드 타임)가 무려 45일이었다. 데이터를 까보니 개발자가 코딩하는 시간은 3일인데, QA팀의 통합 테스트를 기다리는 시간이 20일, 보안팀의 수동 소스코드 검사(결재) 대기가 22일로 밝혀졌다(병목 발견).
- **해결책 (Action)**: 수동 결재(결재 서류)를 완전히 폐지하고, 코드를 올리면 즉시 정적 분석(SonarQube)과 자동화 테스트(Selenium)가 도는 CI/CD 파이프라인을 구축했다. 결재를 자동화 툴로 위임(Shift-Left)함으로써 LTFC를 45일에서 2시간으로 극적으로 단축했다.

### 2. 도입 시 핵심 체크리스트 (Technical & Operational)
1. **무기화(Weaponization) 금지**: DORA 점수를 부서 평가(KPI)에 연동하여 "DF(배포 빈도)가 낮은 팀은 성과급을 깎겠다"고 선언하면 최악의 사태가 벌어진다. 개발자들은 의미 없는 공백(띄어쓰기)을 커밋해서 배포 빈도만 수백 번으로 부풀리는 게임화(Goodhart's Law)를 시작한다. 지표는 개인 평가가 아닌 '시스템 병목 개선' 목적으로만 써야 한다.
2. **트레이드오프의 균형 감시**: 배포 빈도(DF)만 올리라고 압박하면 개발자는 테스트를 생략하고 배포하여 결국 변경 실패율(CFR)이 폭발한다. 반드시 속도 지표 2개와 안정성 지표 2개를 '세트'로 묶어서 모니터링해야 한다.
3. **아키텍처의 한계 인정**: 거대한 1,000만 줄짜리 모놀리식(Monolithic) 아키텍처를 쓰는 팀에게 DORA Elite 등급(하루 10번 배포)을 요구하는 것은 불가능하다. 컴파일 시간만 1시간이 넘기 때문이다. DORA 수치를 올리려면 시스템을 마이크로서비스(MSA)로 쪼개는 아키텍처 개편이 반드시 선행되어야 한다.

### 3. 치명적 안티패턴 (Anti-Pattern)
- **변경 자문 위원회 (CAB, Change Advisory Board)의 비대화**: 장애를 막겠다는 명목으로 1주일에 한 번씩 수십 명의 임원과 부서장이 모여서 "이 코드를 배포해도 되는가?"를 구두로 검토하고 서명하는 회의체 운영.

> 아래는 수동 결재가 DORA 지표를 어떻게 파괴하는지 보여주는 안티패턴 도식이다.

```text
[Anti-Pattern: The CAB Bottleneck]

[ 개발자 ] "버그 수정 완료! 커밋했습니다." (코딩 1시간 소요)
     |
     v
[ 금요일 오후 3시 CAB 정기 회의 ]
 - 참석자: IT본부장, 보안팀장, 인프라팀장 (코드 내용 1도 모름)
 - 대기 안건: 150건
 - 진행: "이거 안전한가요? 네. 도장 쾅."
     | (결재 대기 6일 소요)
     v
[ 운영 배포 ] -> 에러 발생!! -> "롤백하려면 다시 다음 주 금요일 CAB 승인받아와!" (패닉)

=> 결과: LTFC = 7일, MTTR = 7일. (완벽한 Low Performer 조직 탄생)
```
- **해설**: DORA 연구팀은 수년간의 데이터를 통해 "무거운 CAB 회의(결재)는 안정성을 전혀 높이지 못하며, 오히려 LTFC만 지연시켜 한 번에 큰 배포를 하게 만들어 장애 확률(CFR)을 높인다"고 수학적으로 증명했다. 결재는 사람이 아니라 '자동화된 테스트 스크립트'가 해야 한다.

> **📢 섹션 요약 비유**: 교통사고를 줄이겠다고 모든 교차로마다 경찰을 배치해 차를 한 대씩 세우고 면허증을 검사하면(CAB 회의), 사고는 안 줄고 도시는 엄청난 교통지옥(LTFC 증가)에 빠집니다. 과속 방지 턱(자동화 테스트)을 설치하고 차들을 그냥 달리게 하는 것이 선진국입니다.

### 2. 미래 전망 및 아키텍처 진화
- **DORA 기반의 벤더 생태계 통합**: 최근 GitHub, GitLab, AWS, Azure 등 모든 글로벌 클라우드 및 형상 관리 플랫폼들이 기본 대시보드에 'DORA Metrics' 탭을 기본 탑재(Built-in)하여 제공하고 있다. 기업은 별도 도구 없이도 즉시 자사의 등급을 전 세계 벤치마크와 비교할 수 있게 되었다.
- **플랫폼 엔지니어링 (Platform Engineering)의 등장**: 개발자가 비즈니스 로직(코드)만 짜고 DORA 지표를 자동으로 끌어올릴 수 있도록, 사내 인프라를 '내부 개발자 포털(IDP, Internal Developer Portal)' 형태로 제공하여 인지 부하를 줄여주는 '플랫폼 엔지니어링' 트렌드가 DORA 지표 달성의 가장 강력한 해결책으로 부상하고 있다.

> **📢 섹션 요약 비유**: DORA Metrics는 기업의 IT 조직이 하늘을 날기 위해 꼭 봐야 하는 '비행기 계기판'입니다. 속도계(DF, LTFC)와 고도계(CFR, MTTR)만 정확히 보면, 우리는 구름 속에서도 절대 추락하지 않고 구글, 넷플릭스 같은 목적지(Elite Performer)를 향해 똑바로 날아갈 수 있습니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- 데브옵스 (DevOps) | 개발(Dev)과 운영(Ops)의 벽을 허물어 협업을 강조하는 문화로, DORA Metrics는 이 데브옵스가 얼마나 성공적으로 정착했는지를 보여주는 성적표
- CI/CD (Continuous Integration / Continuous Deployment) | 코드를 짤 때마다 자동으로 빌드, 테스트, 배포하는 파이프라인으로, DORA의 핵심 지표(DF, LTFC)를 물리적으로 달성하게 해주는 도구
- MTTR (Mean Time to Recovery) | 장애 발생 시 시스템을 정상 상태로 복구하는 데 걸리는 평균 시간으로, DORA의 핵심 안정성 지표 중 하나
- 카나리 배포 (Canary Deployment) | 전체 서버가 아닌 1%의 유저에게만 신규 코드를 배포해 보고 이상이 없으면 점진적으로 확대하는 방식으로, CFR(변경 실패율)을 극적으로 낮추는 기술
- 플랫폼 엔지니어링 (Platform Engineering) | 개발자가 인프라나 배포 툴(Jenkins 등)을 몰라도 쉽게 셀프서비스로 코드를 배포할 수 있게 사내 포털(IDP)을 만들어주는 최신 클라우드 조직 트렌드

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 우리 반 친구들이 레고 조립을 얼마나 잘하는지 진짜 실력을 4가지 점수로 평가하는 성적표예요.
2. **원리**: 1. 레고를 하루에 몇 번이나 완성하니?(배포 빈도) 2. 설명서를 보고 조립 끝낼 때까지 얼마나 걸리니?(리드 타임) 3. 완성한 레고가 부서질 확률은?(실패율) 4. 부서졌을 때 얼마나 빨리 다시 고치니?(복구 시간).
3. **효과**: "나 레고 1,000개나 썼어!"라며 엉터리로 자랑하는 걸 막고, 이 4가지 점수가 다 높은 친구(Elite)가 진짜 레고 조립의 천재라는 걸 전 세계가 인정해 준답니다!
