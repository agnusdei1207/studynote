# BPMN (Business Process Model and Notation)

> **약어 (Abbreviation)**: BPMN (Business Process Model and Notation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: BPMN (Business Process Model and Notation)은 기업의 업무 흐름(Workflow)과 규칙을 다이어그램으로 시각화하기 위해 OMG(Object Management Group)에서 제정한 **국제 표준 그래픽 모델링 언어**이다.
> 2. **가치**: 비즈니스 분석가(기획자), 기술 개발자(IT), 그리고 시스템 엔진이 **하나의 동일한 다이어그램**을 보고 완벽하게 같은 비즈니스 로직을 이해하고 실행할 수 있도록 완벽한 상호 운용성(Interoperability)을 제공한다.
> 3. **융합**: BPMN 다이어그램 자체는 내부적으로 **XML 형식**으로 저장되며, 이는 **BPM (Business Process Management)** 엔진에 직접 배포되어 사람과 시스템을 연결하는 실행 코드로 변환된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: BPMN은 플로우차트(Flowchart)나 UML(Unified Modeling Language)과 비슷해 보이지만, "비즈니스 프로세스"를 그리는 데 특화된 언어다. 누가 언제 어떤 조건에서 무슨 일을 해야 하는지를 아주 세밀한 규칙(Event, Gateway, Activity)으로 그려낸다.
- **💡 비유**: 건축에서 배관공, 전기공, 인테리어 업자가 모두 하나의 도면을 보고 각자의 작업을 오차 없이 수행할 수 있게 해주는 전 세계 공통의 '건축 표준 청사진'과 같습니다.
- **등장 배경**:
  - **기존 한계**: 과거에는 기획자가 파워포인트나 비지오(Visio)로 단순한 플로우차트를 그리면, IT 개발자는 이를 보고 UML이나 하드코딩으로 시스템을 다시 설계해야 했다. 이 과정에서 의사소통 오류가 발생하고, 기획 의도와 전혀 다른 시스템이 개발되는 참사가 빈번했다.
  - **혁신적 패러다임**: "사람이 보는 그림이 곧 기계가 읽는 코드여야 한다"는 사상 아래, 비즈니스 친화적이면서도 IT 엔진이 실행 가능한 엄격한 문법을 가진 BPMN 2.0 표준이 등장했다.
  - **현재의 비즈니스 요구**: 글로벌 협업과 시스템 연동이 폭증하면서, ERP, CRM, 그룹웨어가 서로 어떻게 데이터를 주고받을지를 표준화된 문서로 정의하고 자동화 엔진에 즉각 배포해야 하는 초자동화(Hyperautomation) 시대의 핵심 문법이 되었다.

> 다음은 기존 플로우차트와 BPMN의 차이점을 보여주는 비교 도식이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [Standard Flowchart vs BPMN 2.0]                            │
├──────────────────────────────┬──────────────────────────────┤
│ 1. Flowchart (Visio/PPT)     │ 2. BPMN 2.0 (Camunda/Bizagi) │
│                              │                              │
│ [Start]                      │ (Event) ◯ Start Order       │
│   │                          │   │                          │
│ [재고 확인] (네모 박스)      │ [Task] ▭ Check Inventory    │
│   │                          │   │                          │
│ <재고 있나?> (마름모)        │ <Gateway> ◇ Exclusive XOR   │
│  /      \                    │  / (Yes)       \ (No)        │
│(Yes)    (No)                 │[Task] ▭        (Event) ✉   │
│[배송]   [주문취소]           │ Ship Item      Send Email    │
│                              │                              │
│ * 한계: 기계가 실행 불가능   │ * 특징: 엔진이 읽고 즉시 실행│
└──────────────────────────────┴──────────────────────────────┘
```
- **해설**: 일반 플로우차트는 그리는 사람 마음대로 모양이 바뀔 수 있어 컴퓨터가 이해할 수 없다. 반면 BPMN은 동그라미(Event), 네모(Activity), 마름모(Gateway)의 의미가 엄격히 규정되어 있으며, 뒷단에는 XML 코드가 매핑되어 있어 엔진(Engine)이 이를 해석해 사람에게 업무를 할당하거나 외부 API를 호출할 수 있다.

> **📢 섹션 요약 비유**: 한국어와 영어로 따로 적혀있어 서로 못 알아듣던 기획서와 개발 코드를, 전 세계 누구나 읽을 수 있고 기계도 알아듣는 '음악의 악보(BPMN)'로 통일한 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 핵심 구성 요소 (BPMN 2.0 Core Elements)

BPMN 다이어그램은 크게 4가지 카테고리로 구성된다.

| 요소 분류 | 기호 형태 | 역할 및 특징 | 비유 |
|:---|:---|:---|:---|
| **Flow Objects** | ◯ (Event) <br> ▭ (Activity) <br> ◇ (Gateway) | 프로세스의 발생, 행동, 분기를 정의 (가장 핵심적인 요소) | 연극의 시작, 대사, 갈림길 |
| **Connecting Objects** | ➔ (Sequence) <br> ⇢ (Message) | 객체 간의 실행 순서나 메시지 흐름을 연결 | 대본의 순서와 주고받는 편지 |
| **Swimlanes** | ▩ (Pool) <br> ⊟ (Lane) | 프로세스 참여자(부서, 회사, 시스템)를 구분하는 영역 | 수영장의 레인(선수 구분) |
| **Artifacts** | [ ] (Data Object) <br> --- (Annotation) | 부가적인 데이터나 주석을 표현 (실행에는 영향 없음) | 무대 위의 소품 |

### 2. 세부 객체 동작 원리 (Flow Objects)

- **Events (이벤트 / ◯)**: 무언가 '발생'했음을 알린다.
  - *Start Event* (가는 선 ◯): "주문이 접수되었다."
  - *Intermediate Event* (겹선 ◯): "타이머가 3일 지났다.", "메시지가 도착했다."
  - *End Event* (굵은 선 ⬤): "프로세스가 종료되었다."
- **Activities (액티비티 / ▭)**: 무언가 '작업'을 수행한다.
  - *User Task*: 사람이 화면을 보고 직접 승인/입력하는 작업.
  - *Service Task*: 외부 시스템(REST API 등)을 자동 호출하는 작업.
- **Gateways (게이트웨이 / ◇)**: 흐름을 '분기'하거나 '병합'한다.
  - *Exclusive (XOR)*: 여러 경로 중 오직 하나만 선택 (예: 신용카드 OR 무통장).
  - *Parallel (AND)*: 여러 경로를 동시에 실행 (예: 포장 AND 영수증 발행 동시 진행).

> 아래 흐름도는 Pool과 Lane을 사용하여 구매팀과 재무팀 간의 결재 프로세스를 BPMN 요소로 표현한 도식이다.

```text
[BPMN Swimlane & Flow Diagram]

[[ Pool: Company A ]]
 ┌────────────────────────────────────────────────────────────┐
 │Lane 1: [ 구매팀 (Purchasing) ]                             │
 │  (Start)      [Task 1]               (Gateway) XOR         │
 │  ◯ 구매요청 ➔ ▭ 공급업체 선정 ➔ ◇ 1천만원 이상?        │
 │                                      /          \ (No)     │
 │                                (Yes)/            \         │
 ├────────────────────────────────────/──────────────\────────┤
 │Lane 2: [ 재무팀 (Finance) ]       v                v       │
 │                                [Task 2]         [Task 3]   │
 │                                ▭ 임원 승인      ▭ 즉시 결제│
 │                                   \                /       │
 │                                    v              v        │
 │                                    ◇ (병합) ➔ ⬤ (End)  │
 └────────────────────────────────────────────────────────────┘
```
- **해설**: Pool은 하나의 거대한 조직을 의미하고, Lane은 그 안의 부서를 의미한다. 동그라미(Start)에서 시작된 화설표(Sequence Flow)가 구매팀의 Task를 거쳐 마름모(Gateway)에서 1천만 원을 기준으로 두 갈래로 나뉜다. 이 다이어그램을 시스템에 올리면, 1천만 원 이상일 때 시스템이 알아서 재무팀 임원의 모니터에 '결재 요망' 알람을 띄운다(User Task).

> **📢 섹션 요약 비유**: 수영장(Pool) 안에 여러 레인(Lane)을 쳐놓고, 선수들(부서)이 언제 물에 뛰어들고(Event), 어떤 영법으로 수영하며(Activity), 언제 바통 터치를 할지(Gateway) 규칙을 완벽하게 그려놓은 작전판입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 모델링 표준 언어 비교 (BPMN vs UML vs EPC)

| 비교 항목 | BPMN (Biz Process) | UML (Unified Modeling) | EPC (Event-driven Process Chain) |
|:---|:---|:---|:---|
| **주 목적** | 비즈니스 **프로세스 흐름** 설계 | 소프트웨어 **객체/시스템 구조** 설계 | SAP ERP의 프로세스 모델링 |
| **주 사용층** | 비즈니스 분석가, 현업, IT | 소프트웨어 엔지니어, 아키텍트 | ERP 컨설턴트 |
| **기계 실행성**| **높음 (XML 기반 엔진 실행 가능)**| 낮음 (주로 문서화/설계 용도) | 중간 (SAP 환경 내 특화) |
| **복잡도 표현**| B2B 통신, 메시지 흐름 표현 탁월 | 클래스, 컴포넌트 간 관계 표현 탁월| 논리적 흐름에 집중 (다소 직관적) |

### 2. 마이크로서비스 아키텍처(MSA)와의 시너지
전통적인 BPMN은 모놀리식 환경에서 중앙 집중형 결재 엔진으로 주로 쓰였다. 하지만 MSA 시대가 열리면서, 여러 마이크로서비스에 걸쳐 있는 복잡한 트랜잭션을 조율하는 **오케스트레이션 사가 (Orchestration Saga)** 패턴의 핵심 엔진으로 부활했다.

> 다음은 MSA 환경에서 BPMN 엔진이 Saga 오케스트레이터로 동작하는 구조도이다.

```text
[BPMN as Saga Orchestrator in MSA]

      [ BPMN Engine (Camunda / Zeebe) ]
        (도식화된 BPMN XML 모델 탑재)
               |   |   |
      +--------+   |   +--------+
  (Cmd/Reply)  (Cmd/Reply)  (Cmd/Reply)
      v            v            v
  [ Svc A ]    [ Svc B ]    [ Svc C ]
   (Order)     (Payment)    (Delivery)
```
- **해설**: Camunda나 Zeebe 같은 최신 경량 BPMN 엔진은 마이크로서비스 중앙에 위치한다. 결제가 실패(Error Event)하면, BPMN 모델에 그려진 '보상 경계 이벤트(Compensation Boundary Event)'가 작동하여, 자동으로 서비스 A에게 주문 취소 명령을 내린다. 이 모든 장애 복구 흐름이 코드가 아닌 그림(BPMN)으로 관리된다.

> **📢 섹션 요약 비유**: UML이 자동차의 엔진과 톱니바퀴가 어떻게 맞물려 있는지 그리는 '기계 도면'이라면, BPMN은 운전자가 언제 시동을 걸고 언제 브레이크를 밟아야 목적지에 도착하는지 알려주는 '내비게이션 경로 도면'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오와 의사결정
- **상황**: 통신사에서 '신규 고객 가입 프로세스'를 개편하려 한다. 고객 정보 확인(시스템), 신용 조회(외부 API), 상담사 개통 확인(사람)이 뒤섞여 있어 시스템 개발팀과 현업 부서 간의 소통이 매번 실패하고 버그가 양산된다.
- **판단**: 워드나 PPT로 작성된 요구사항 정의서를 폐기하고, 전사 **BPMN 2.0 표준**을 도입한다. 현업 담당자가 직접 BPMN 툴로 흐름을 그리고, 개발자는 해당 모델의 `Service Task` 블록에 외부 API 연동 코드만 매핑한다. 그림이 곧 코드가 되므로 요구사항 누락(Missing Requirement)이 원천 차단된다.

### 2. 도입 시 핵심 체크리스트 (Technical & Operational)
1. **가독성 (Readability) 유지**: BPMN은 100개가 넘는 복잡한 기호를 제공한다. 한 화면에 너무 많은 객체를 욱여넣으면 스파게티가 된다. 서브 프로세스(Sub-Process) 기호를 활용해 큰 흐름과 상세 흐름을 캡슐화(계층화)했는가?
2. **실행 가능성 (Executability)**: 그냥 예쁘게 그린 그림(Descriptive Model)을 넘어, 변수명, 스크립트, API 엔드포인트가 XML 속성에 정확히 기입된 실행 가능한 모델(Executable Model)로 고도화할 수 있는 IT 역량이 있는가?
3. **토큰(Token)의 이해**: BPMN은 시작 지점부터 토큰(Token)이라는 가상의 제어권이 선을 따라 이동하며 동작한다. 병렬 게이트웨이(AND)에서 토큰이 2개로 갈라졌다가 다시 합쳐질 때 데드락(Deadlock)이 발생하지 않도록 동기화 설계를 완벽히 했는가?

### 3. 치명적 안티패턴 (Anti-Pattern)
- **IT 부서 전유물로 전락 (IT-Only Notation)**: BPMN 도입 후 기호가 너무 어렵다는 이유로 현업(비즈니스)이 작성을 포기하고, 결국 IT 개발자가 시스템 코딩을 마친 후 문서화 용도로 역(Reverse)으로 그림을 그리는 현상.

> 아래는 비즈니스와 IT의 단절을 시각화한 안티패턴이다.

```text
[Anti-Pattern: The Disconnected BPMN]

  (현업) "BPMN 너무 어려워, PPT로 다시 그려줄게"
     |
     v (Word/PPT 문서)
  (IT 개발자) "이 문서대로 하드코딩 해야지"
     |
     v (시스템 오픈 3달 후)
  (IT 개발자) "감사팀이 BPMN 문서 내놓으라니까, 
              지금 시스템 보고 역으로 BPMN 그려야겠다..." 
              (==> 아무도 안 보는 가짜 다이어그램 탄생)
```
- **해설**: BPMN의 존재 이유는 비즈니스와 IT의 '공통 언어'다. 현업이 읽을 수 없는 수준으로 복잡하게(개발자만 아는 IT 예외 처리 위주로) 모델링을 해버리면, BPMN은 쓰레기 문서로 전락한다. 비즈니스 레벨 모델과 기술 실행 레벨 모델을 분리해서 관리해야 한다.

> **📢 섹션 요약 비유**: 외국인(IT)과 대화하려고 세계 공용어(BPMN)를 배웠는데, 너무 어려운 전문 용어만 쓰다 보니 결국 다시 통역사(문서)를 고용하게 되어 공용어를 배운 의미가 사라져버리는 것을 경계해야 합니다.

|:---|:---|:---|:---|
| **커뮤니케이션 비용** | 기획-개발 간 끝없는 회의와 수정 반복 | 시각적 표준 언어로 오해 원천 차단 | 요구사항 분석 및 설계 기간 50% 단축 |
| **시스템 일치성** | 문서와 실제 동작하는 시스템 코드가 불일치 | 도면(BPMN)이 곧 시스템 실행 파일(XML) | 문서 최신화율 100% (버그 추적 용이) |
| **유지보수성** | 프로세스 변경 시 수만 줄의 코드 분석 필요 | 다이어그램 화살표 변경 후 즉시 재배포 | 비즈니스 변경 요구(CR) 대응 리드 타임 80% 감소 |

### 2. 미래 전망 및 아키텍처 진화
- **DMN (Decision Model and Notation) 및 CMMN 융합**: 프로세스의 '흐름'은 BPMN이 담당하지만, "할인율을 어떻게 계산할 것인가?" 같은 복잡한 '규칙'은 DMN 표준이, "의사의 진단처럼 정해진 순서가 없는 비정형 업무"는 CMMN(Case Management)이 담당하여 서로 융합되는 **Triple Crown** 표준화가 진행 중이다.
- **로우코드 (Low-Code) 플랫폼의 코어 엔진**: 사용자들은 더 이상 복잡한 코드를 치지 않는다. 화면에서 BPMN 기호를 드래그하여 배치하면, 생성형 AI(GenAI)가 이를 분석하고 뒷단의 API 연동 코드까지 모두 자동으로 짜주는 지능형 초자동화 플랫폼의 핵심 심장부로 진화하고 있다.

> **📢 섹션 요약 비유**: BPMN은 인간의 상상력(비즈니스 기획)을 기계의 물리적 동작(IT 시스템)으로 다이렉트 변환해 주는 완벽한 '마법의 통역기'입니다. 이 언어를 마스터한 기업만이 변화의 파도를 가장 빠르게 타고 넘을 수 있습니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- BPM (Business Process Management) | BPMN으로 그려진 모델을 실제로 시스템 상에서 실행하고 모니터링, 최적화하는 4단계 통합 관리 체계
- 오케스트레이션 사가 (Orchestration Saga) | MSA 환경에서 분산된 트랜잭션의 롤백/보상 흐름을 시각적으로 제어할 때 BPMN 엔진이 완벽하게 쓰이는 아키텍처 패턴
- DMN (Decision Model and Notation) | 프로세스 내의 복잡한 비즈니스 룰(If-Then 조건문)을 BPMN 다이어그램에서 분리하여 의사결정 테이블로 시각화하는 OMG 표준
- CMMN (Case Management Model and Notation) | 정해진 순서가 없는(Ad-hoc) 지식 노동자의 업무나 법적/의료적 케이스 처리를 모델링하는 OMG 표준
- 토큰 (Token) | BPMN 엔진이 프로세스를 실행할 때, 현재 어디까지 작업이 진행되었는지를 추적하기 위해 선을 따라 이동하는 가상의 제어점

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 레고 성을 만들 때 들어있는 '조립 설명서'와 같아요. 블록(작업)을 어떻게, 어느 순서로, 누가 끼워야 하는지 전 세계 약속된 기호로 그려둔 거예요.
2. **원리**: 기획자 아저씨가 동그라미(시작)와 네모(행동), 마름모(갈림길)를 사용해 설명서를 컴퓨터에 그리면, 컴퓨터 엔진이 그걸 똑같이 읽고 사람들에게 "이거 해!"라고 자동으로 시켜줘요.
3. **효과**: 예전에는 말로만 설명해서 서로 다르게 이해하고 엉뚱한 성을 만들었는데, 이제는 이 그림 하나만 보면 누구나 100% 똑같은 멋진 성을 실수 없이 지을 수 있답니다!
