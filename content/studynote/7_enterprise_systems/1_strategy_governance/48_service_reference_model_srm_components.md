# SRM (Service Reference Model, 서비스 참조 모델)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SRM은 범정부 EA(GEA)의 5대 참조 모델 중 하나로, 업무 목표 달성을 위해 필요한 서비스 구성 요소(Service Component)들을 기능별로 분류하고 표준화한 비즈니스 중심의 서비스 분류 체계다.
> 2. **구조**: 고객 서비스, 업무 관리, 디지털 자산, 분석, 백오피스, 지원 서비스 등 6대 서비스 도메인을 중심으로 계층화되어 있으며, 각 부처가 공통으로 사용할 수 있는 재사용 가능 컴포넌트를 식별한다.
> 3. **가치**: 중복된 시스템 개발을 방지하여 예산을 절감하고, 검증된 공통 컴포넌트 활용을 통해 정보화 사업의 구축 속도와 상호운용성을 비약적으로 향상시킨다.

+++

## Ⅰ. 개요 (Context & Background)

1. **개념**: 서비스 참조 모델(SRM, Service Reference Model)은 기업이나 정부가 제공하는 IT 기능들의 **'공용 부품 목록'**이다. 부처마다 제각각인 '본인 인증', '전자 결재', '우편번호 조회' 기능을 따로 만들지 말고, 잘 만들어진 공용 부품(Service Component)을 가져다 쓰자는 것이 SRM의 핵심 철학이다.

2. **💡 비유**: SRM은 프라모델이나 레고의 **'부품 조립 가이드'**와 같습니다. 성을 만들든 자동차를 만들든 바퀴, 엔진, 브릭(컴포넌트)은 표준화된 규격으로 제공되므로, 우리는 이 부품들을 조합하여 창의적인 결과물(비즈니스 서비스)을 빠르게 완성할 수 있는 것과 같습니다.

3. **등장 배경**:
    - **시스템 구축의 파편화**: 유사한 기능을 가진 시스템을 부처별로 중복 구축하여 막대한 국가 예산 낭비 발생.
    - **서비스 지향 아키텍처(SOA)의 부상**: 기능을 독립적인 서비스 단위로 쪼개고 이를 조립하여 전체 시스템을 구성하는 현대적 설계 방식 요구.
    - **민첩한 행정 서비스 요구**: 새로운 정책이나 법규 변화에 맞춰 며칠 내로 신규 서비스를 출시해야 하는 압박 증대.

📢 섹션 요약 비유: SRM은 기업이라는 공장의 '표준 금형 세트'입니다. 규격화된 금형(서비스)이 있으면 매번 새로 틀을 짤 필요 없이 원하는 제품을 찍어낼 수 있습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

범정부 SRM 2.0 표준을 중심으로 서비스 도메인 분류와 구성 원리를 분석한다.

### 1. SRM 6대 서비스 도메인 구조 (ASCII)

```text
    ┌─────────────────────────────────────────────────────────────┐
    │              [ SRM : 6 Service Domains ]                    │
    ├─────────────────────────────────────────────────────────────┤
    │                                                             │
    │  1. 고객 서비스 (Customer Services)                         │
    │     - 고객 관계 관리, 셀프 서비스, 고객 피드백              │
    │                                                             │
    │  2. 업무 관리 서비스 (Business Management Services)         │
    │     - 관리 프로세스, 디지털 자산 관리, 사업 관리            │
    │                                                             │
    │  3. 디지털 자산 서비스 (Digital Asset Services)             │
    │     - 콘텐츠 관리, 문서 관리, 지식 관리                     │
    │                                                             │
    │  4. 비즈니스 분석 서비스 (Business Analytical Services)     │
    │     - 분석 및 보고, 시각화, 데이터 마이닝                   │
    │                                                             │
    │  5. 백오피스 서비스 (Back Office Services)                  │
    │     - 인사, 재무, 자산, 물류 관리                           │
    │                                                             │
    │  6. 지원 서비스 (Support Services)                          │
    │     - 보안 관리, 통합 관리, 통신 및 협업                    │
    │                                                             │
    └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**  
SRM은 비즈니스 기능을 6개의 큰 덩어리(Domain)로 나눈다. 각 도메인 하부에는 수십 개의 서비스 유형(Type)과 세부 컴포넌트(Component)가 존재한다. 이를 통해 아키텍트는 "우리가 만들려는 대출 시스템은 '고객 서비스' 도메인의 '신용 조회' 컴포넌트를 재사용하면 되겠군"이라는 판단을 내릴 수 있다.

### 2. SRM 구성 요소의 계층화 (Leveling)

- **도메인 (Domain)**: 최상위 기능 그룹 (예: 고객 서비스).
- **서비스 유형 (Service Type)**: 도메인 내 세부 영역 (예: 고객 소통).
- **서비스 구성요소 (Component)**: 실제 재사용 가능한 기능 단위 (예: 설문 조사, 자동 응답).

### 3. SRM과 서비스 공유 메커니즘

1. **식별(Identification)**: 현행 시스템 분석을 통해 중복 사용되는 공통 기능을 추출.
2. **표준화(Standardization)**: 추출된 기능을 누구나 쓸 수 있게 인터페이스(API) 규격 확정.
3. **등록(Registry)**: 범정부 아키텍처 포털(GEAP)에 공용 컴포넌트로 등록.
4. **활용(Reuse)**: 신규 사업 기획 시 등록된 컴포넌트 활용 여부 우선 검토.

📢 섹션 요약 비유: 도메인은 백화점의 '층(남성복, 여성복)'이고, 유형은 '매장 종류', 컴포넌트는 매장에 진열된 '개별 옷(기능)'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

SRM과 타 참조 모델, 그리고 현대적 MSA 기술과의 관계다.

### 1. BRM(업무 참조 모델) vs SRM 비교

| 구분 | BRM (Business RM) | SRM (Service RM) |
|:---|:---|:---|
| **중점** | **조직의 목적** (Goal) | **실행 수단** (Enabler) |
| **질문** | "우리는 무엇을 위해 일하는가?" | "그 일을 하려면 무슨 기능이 필요한가?" |
| **분류 기준** | 행정 분야, 업무 영역 | 서비스 컴포넌트, 기능 단위 |
| **비유** | 요리사가 만들어야 할 '최종 요리' | 요리에 들어가는 '공용 소스 및 반조리 식품' |

### 2. 과목 융합: MSA(마이크로서비스 아키텍처)와 SRM

현대의 SRM은 문서상의 분류를 넘어 실제 작동하는 클라우드 서비스로 진화하고 있다.
- **Microservices as SRM**: SRM에서 정의된 각 컴포넌트가 독립적인 컨테이너(Docker) 기반 마이크로서비스로 구현됨.
- **Service Mesh**: 분산된 SRM 컴포넌트 간의 통신, 보안, 추적을 서비스 메시(Istio 등) 기술로 제어.
- **시너지**: 과거에는 라이브러리 형태의 공유(Copy & Paste)였다면, 이제는 API 호출 형태의 실시간 공유(Call & Response)로 발전.

📢 섹션 요약 비유: 과거의 SRM이 '공구함에 담긴 망치'였다면, 현대의 SRM은 언제 어디서든 스마트폰으로 호출하면 달려오는 '출장 수리 서비스'와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 SRM을 활용하여 개발 생산성을 높이는 전략이다.

### 1. 공통 컴포넌트 도입 판단 기준 (Decision)

기술사적 관점에서 모든 기능을 SRM 컴포넌트로 만드는 것은 비효율적이다.
- **표준 컴포넌트 채택**: 본인 인증, 결재 연동, 로그 관리 등 모든 부서가 쓰는 범용 기능.
- **자체 개발 채택**: 우리 부처만의 특수한 비즈니스 로직, 극도의 성능이 요구되는 핵심 연산.
- **효과**: 개발 리소스의 70%는 공통 컴포넌트(SRM)로 빠르게 처리하고, 나머지 30%의 핵심 가치 창출에 역량 집중.

### 2. SRM 관리 및 활용 체크리스트

| 점검 항목 | 확인 질문 | 기술사적 제언 |
|:---|:---|:---|
| **재사용성** | 이 컴포넌트가 최소 3개 이상의 시스템에서 쓰일 수 있는가? | 범용성 및 결합도(Coupling) 최소화 설계 |
| **인터페이스 표준** | 타 부서에서 호출하기 쉬운 REST API 규격을 따르는가? | Swagger 등을 통한 API 명세서 자동화 및 공유 |
| **성능 영향** | 공용 서비스를 호출할 때 발생하는 네트워크 지연이 허용 범위인가? | 캐싱 및 비동기 메시징(Kafka) 도입 검토 |
| **거버넌스** | 신규 구축 시 기존 SRM 자산을 먼저 조회해 보았는가? | 아키텍처 포털(GEAP) 검색 결과 증적 첨부 |

### 3. 안티패턴
- **"The Black Box Component"**: 내부 로직을 알 수 없어 장애 시 대응이 불가능한 컴포넌트. (오픈 소스 기반 가시성 확보 필요)
- **"Forced Reuse"**: 맞지도 않는 신발에 발을 맞추듯, 비즈니스 요건과 다른 공통 기능을 억지로 쓰는 경우. (적절한 커스터마이징 허용 필요)

📢 섹션 요약 비유: SRM 활용은 '밀키트(Meal Kit)' 요리와 같아서, 요리 시간을 획기적으로 줄여주지만 우리 가족의 입맛(특수한 요건)에 맞게 약간의 양념(커스터마이징)은 더할 수 있어야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

SRM은 기업과 정부의 IT 인프라를 '조립형 아키텍처'로 바꾸는 핵심 엔진이다.

| 기대효과 | 설명 |
|:---|:---|
| **개발 비용 절감** | 이미 만들어진 기능을 호출만 하여 사용함으로써 중복 투자 원천 차단 |
| **시장 대응 속도(Agility)** | 조립식 개발을 통한 신규 서비스 출시 리드타임 50% 이상 단축 |
| **품질 안정성** | 수많은 시스템에서 검증된 표준 컴포넌트 사용으로 결함 발생 감소 |

미래의 SRM은 **"AI-based Component Orchestration"**으로 진화할 것이다. 사람이 컴포넌트를 조립하는 것이 아니라, 비즈니스 요구사항을 자연어로 입력하면 AI가 전사 SRM 리포지토리에서 최적의 컴포넌트 조합을 찾아 자동으로 코드를 생성하고 배포하는 '자율 조립형 시스템'이 도래할 전망이다. SRM은 이제 단순한 분류 모델을 넘어, 디지털 비즈니스를 무한히 확장시키는 '서비스 세포(Service Cell)'의 라이브러리가 될 것이다.

📢 섹션 요약 비유: 미래의 SRM은 모든 부품이 스스로의 역할을 알고 있는 '트랜스포머'와 같아서, 상황에 맞춰 가장 강력한 로봇(최적의 시스템)으로 변신하여 비즈니스를 승리로 이끌 것입니다.

### 📌 관련 개념 맵
| 관련 개념 | 관계 및 시너지 설명 |
|:---|:---|
| Service Component | SRM의 가장 기본이 되는 재사용 가능한 기능 단위 |
| SOA (Service Oriented Architecture) | SRM이 추구하는 아키텍처적 설계 철학 |
| Shared Service Center (SSC) | 전사 공통 서비스를 전문적으로 운영하는 조직 체계 |
| API Economy | SRM 컴포넌트가 API를 통해 기업 안팎으로 가치를 유통하는 생태계 |
| BRM / TRM / DRM | SRM과 상호작용하는 GEA의 다른 참조 모델들 |

### 👶 어린이를 위한 3줄 비유 설명
1. 학교에 있는 모든 컴퓨터들이 똑같은 "숙제 도우미 프로그램"이나 "급식 확인 버튼"을 나누어 쓰는 거예요.
2. 교실마다 프로그램을 따로 만들지 않고 한 번만 잘 만들어서 나눠 쓰니까, 선생님들이 다른 일을 더 많이 도와주실 수 있어요.
3. 데이터들이 서로 잘 맞아서 싸우지 않고 사이좋게 일할 수 있게 만든 "공용 도구 상자"랍니다!
