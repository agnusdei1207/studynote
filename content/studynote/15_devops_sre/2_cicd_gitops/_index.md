> **핵심 인사이트**
> 1. **본질**: 지속적 통합(CI)은 개발자들의 코드를 수시로 메인 브랜치에 병합하여 자동 빌드와 테스트를 수행함으로써 통합 오류를 조기에 발견하는 개발 문화이며, 지속적 배포(CD)는 이 과정을 자동으로 프로덕션까지 이어지는 완전한 파이프라인으로 확장한다.
> 2. **가치**: 하루 수십 회에서 수백 회의 무중단 배포가 가능하며, 수동 배포에 따른 휴먼 에러를 0%로 수렴시키고, 변경 실패율을 5% 미만으로 제어한다.
> 3. **융합**: GitOps는 인프라와 애플리케이션의 목표 상태(Desired State)를 Git 리포지토리에 선언적으로 저장하고, 클러스터 내부 에이전트가 이를 지속적으로 동기화하는 현대적 CD 방식으로서, 풀 기반(Pull-based) 배포로 보안성을 극대화한다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 지속적 통합의 탄생 배경과 문제의식

지속적 통합(Continuous Integration, CI)이 등장한 이유는 소프트웨어 개발의 가장 큰 위험 요소 중 하나인 "통합 지옥(Integration Hell)"을 해결하기 위해서였다. 여러 개발자가 수 주에서 수 개월 동안 각자 별도 브랜치에서 개발한 코드를 마지막 순간에 합치면, 수십에서 수백 개의 충돌(conflict)과 버그가 동시에 폭발하여 프로젝트를 좌초시키는 것이 일반적이었다. Martin Fowler는 2006년 이러한 문제를 해결하기 위해 "코드를 하루에도 수십 번씩 메인 브랜치에 병합하라"고 제안했고, 이것이 CI의 기본 철학이 되었다.

CI의 핵심 전제는 "통합을 자주, 일찍, 자동화하라"이다. 개발자가 하루에 최소 한 번 이상 코드 변경을 공유 중앙 저장소에 병합하면, 각 병합에서의 충돌 범위가 작아져 해결이 용이하다. 또한, 자동 빌드와 테스트가 각 병합마다 실행되므로 버그가 발생하는 근본 원인을 코딩 컨텍스트와 가까운 시점에서 발견할 수 있다. 테스트가 코딩 후 5분 이내에 실행되면 디버깅 효율이 극대화되는 것은 명백하다.

### 지속적 통합에서 지속적 배포로의 진화

CI가 빌드와 테스트의 자동화에 그친다면, CD(Continuous Delivery)는 이 파이프라인을 확장하여 프로덕션 배포 준비 상태까지 자동화한다. CD에서는 CI를 통과한 코드가 스테이징 환경에 자동으로 배포되어 추가 인수 테스트를 거치지만, 실제 프로덕션 배포는 여전히 사람의 수동 승인을 필요로 한다. 반면, Continuous Deployment는 수동 승인마저 생략하고, 테스트를 통과한 모든 코드를 완전 자동으로 프로덕션 환경까지 릴리스한다. 이는 더 높은 자동화 성숙도를 요구하지만, 충분히 검증된 파이프라인에서만 가능하다.

CI/CD의 세 가지 수준을 정리하면 다음과 같다. 지속적 통합(CI)은 코드의 병합과 빌드, 단위 테스트까지만 자동화한다. 지속적 전달(Continuous Delivery)은 CI 단계에 더하여 스테이징 환경까지의 배포를 자동화하지만, 프로덕션 배포는 수동 승인制이다. 지속적 배포(Continuous Deployment)는 테스트를 통과한 모든 코드를 완전 자동으로 프로덕션까지 배포한다. 조직의 자동화 성숙도에 따라 이 중 적절한 수준을 선택해야 한다.

### GitOps 패러다임의 등장

GitOps는 Weaveworks가 2017년에 제안한 배포 패러다임으로, DevOps 원칙을 Git 리포지토리에 적용한 것이다. 핵심 아이디어는 매우 단순하다. 인프라 및 애플리케이션의 목표 상태(Desired State)를 YAML이나 Helm Chart 같은 선언적(Declarative) 형식으로 Git 리포지토리에 저장하고, Kubernetes 클러스터 내부의 에이전트(ArgoCD, FluxCD 등)가 이 Git 리포지토리를 지속적으로 폴링(감시)하여 클러스터 상태와 비교, 동기화한다. 이 모델의 가장 큰 혁신은 배포 권한의 구조적 변화에 있다.

다음 ASCII 다이어그램은 전통적인 푸시 기반 CI/CD와 GitOps 풀 기반 배포의 차이를 보여준다. 상단은 외부 CI 서버(Jenkins 등)가 Kubernetes 클러스터에 직접 kubectl 명령을 내려 배포하는 푸시 기반 방식이다. 하단은 GitOps 방식으로, 클러스터 내부의 ArgoCD가 Git 리포지토리에서 변경 사항을 감지하여 클러스터 상태를Desired State로 수렴시킨다.

**[다이어그램 해설]** 푸시 기반 배포의 핵심 문제점은 보안 자격 증명(credentials)의 관리에 있다. CI 서버가 클러스터에 접근하려면, 외부(公有망)에서 내부(프라이빗 네트워크) Kub API Server로의 통신을 허용해야 한다. 이 경로를 통해 CI 서버가 Compromised 되면, 공격자는 클러스터 전체를 장악할 수 있다. GitOps의 풀 기반 모델에서는 이 경로가 반대로 된다. 외부에서 내부로의 인바운드 방화벽 포트를 열 필요가 없으며, 오직 Git 리포지토리만 모니터링하는 에이전트가 클러스터 내부에서 실행된다. 에이전트 자체가 Compromised 되더라도 影响 범위는 해당 클러스터 내부로 한정된다. 실무에서는 둘을 혼용하되, 프로덕션 환경에서는GitOps(풀 기반)를 기본으로 채택하는 것이 업계 Best Practice다.

### 📢 섹션 요약 비유

CI/CD 파이프라인은 마치 피자 가게에서 반죽(Coding)을 반죽하고(CI 통합), 오븐에 넣고(CI 빌드&테스트), 배달원이 고객 집까지 배달하는 것(CD 배포)까지 모든 과정을 자동화 로봇으로 처리하는 것과 같다. GitOps는 그 로봇이 가게 안에 상주하면서 주방의 레시피(Git State)만 지켜보다가, 필요할 때마다 알아서 피자를 굽고 배달하는 것에 비유할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CI/CD 파이프라인의 7단계 구성

완전한 CI/CD 파이프라인은 크게 7단계로 구성된다. 각 단계는 순차적으로 실행되며, 이전 단계가 실패하면 이후 단계는 실행되지 않는다. 1단계는 소스 관리(Source Control)이다. 개발자가 코드를 Git 리포지토리에 푸시하면, 웹훅(Webhook)이 CI 서버를 트리거한다. 2단계는 빌드(Build)이다. 소스 코드를 컴파일하고, 의존성 패키지를 다운로드하며, 실행 가능한 아티팩트(Artifact)를 생성한다. 이 단계에서 빌드 실패가 발생하면 환경 설정이나 의존성 문제가 있는 것이다. 3단계는 단위 테스트(Unit Test)이다. 각 모듈이 독립적으로 제대로 작동하는지 검증한다. 4단계는 정적 코드 분석(Static Analysis)이다. SonarQube 등을 통해 코드 품질, 잠재적 버그, 보안 취약점을 스캔한다. 5단계는 통합 테스트(Integration Test)이다. 여러 모듈을 조합하여 전체 시스템이 제대로 작동하는지 검증한다. 6단계는 컨테이너 이미지 빌드 및 레지스트리推送이다. Docker 이미지를 빌드하고 태그를 부여한 후 프라이빗 레지스트리(ECR, GCR 등)에 푸시한다. 7단계는 배포(Deployment)이다. 스테이징 또는 프로덕션 환경에 배포한다.

### 빌드 도구 및 아티팩트 관리

프로그래밍 언어별로 빌드 도구가 다르다. Java 생태계에서는 Maven과 Gradle이主流이고, JavaScript/Node.js에서는 npm, yarn, pnpm이使用된다. Python에서는 pip와 Poetry가 있으며, Go에서는 go build가内置되어 있다. Ruby에서는 Bundler, .NET에서는 NuGet이 사용된다. 빌드 도구의 선택은 프로젝트 규모, 팀 역량,生态계成熟도에 따라 달라지지만, 중요한 것은CI 서버와의整合性이다.

아티팩트(Artifact)는 소스코드가 빌드되어 생성된 실행 가능한 결과물이다. JAR 파일, Docker 이미지, Lambda 함수 패키지, Helm Chart 등이 아티팩트에 해당한다. 이 아티팩트들은 아티팩트 리포지토리에 버전별로 저장된다. 대표적인 도구로는 Sonatype Nexus, JFrog Artifactory, AWS ECR(Elastic Container Registry), Google GCR(Google Container Registry)이 있다. 아티팩트 관리의 핵심은 版本管理(Provenance)과 不변性(Immutability)이다. 특정 배포版本的 아티팩트를 명확히 추적할 수 있어야 하고, 한 번 생성된 아티팩트는 변경하지 않아야(reproducibility) 한다.

### GitOps 아키텍처: ArgoCD의 내부 동작

ArgoCD는 Kubernetes 환경에서 가장 널리 사용되는 GitOps 도구이다. Its architecture consists of several key components. API Server는 웹 UI, CLI, Git 연동을 제공하는 중앙 인터페이스이다. Application Controller는 Kubernetes Custom Resource Definition(CRD)를 사용하여 애플리케이션 desired state를 관리하고, 현재 상태와의 차이(sync status)를 지속적으로 감시한다. Repository Server는 Git 리포지토리와 Helm Chart를 프록시하는 서비스이다. Redis는 메타데이터 캐싱에 사용된다.

ArgoCD의 동기화(Sync) 프로세스는 다음 단계로 구성된다. ① 개발자가 Git 리포지토리의 Kubernetes Manifest(YAML)를 업데이트하고 푸시한다. ② ArgoCD가 주기적으로(기본 3분) 또는 웹훅을 통해 Git 변경을 감지한다. ③ ArgoCD가 Git의 desired state와 클러스터의 현재 상태를 비교한다. ④ 차이가 발견되면,desired state를 클러스터에 적용한다. ⑤ 전체 과정을 시각화된 UI에서 실시간으로 확인할 수 있다.

### Git 브랜치 전략의 비교

Git 브랜치 전략은 팀 규모, 배포 빈도, 출시 주기에 따라 달라진다. Git Flow는 Master, Develop, Feature, Release, Hotfix의 5개 브랜치를 사용하는 전통적 전략이다._master 브랜치는 항상 프로덕션 배포 가능한 상태를 유지하고, develop 브랜치는 다음 출시를 위한 통합 브랜치이다. feature 브랜치는 개발자가 기능 개발을 위해 생성하며, 완료되면 develop에 병합한다. release 브랜치는 출시 준비를 위해 develop에서 분기하고,_master에 병합하여 태그를 붙인다. hotfix 브랜치는 프로덕션 긴급 수정을 위해_master에서 분기하여_master와 develop에 병합한다. 이 전략은 출시 주기가 긴 프로젝트에 적합하지만, 브랜치 관리가 복잡하고 배포 리드타임이 긴 단점이 있다.

GitHub Flow는_master와 feature 브랜치만 사용하는 단순한 전략이다. 모든 것이-master에서 분기하고, 완료되면_master로 풀 리퀘스트(PR)를 생성하여 코드 리뷰 후 병합한다. 병합하면 즉시 프로덕션에 배포한다. 이 전략은 지속적 배포(Continuous Deployment)를 전제로 하므로, 충분한 자동화 테스트가 갖춰져 있어야 한다. CD가 불가능한 조직에서는 적합하지 않다.

GitLab Flow는 GitHub Flow에 환경(Environment) 개념을 추가한 것이다. master에서 feature를 분기하고, 병합 후 staging, production 순서로 promotions하여 배포한다. 마이크로서비스 환경에서 각 서비스의 배포 시점을 조절할 수 있다.

### 📢 섹션 요약 비유

CI/CD 파이프라인은 마치 공장의 컨베이어 벨트와 같다. 반죽(코드)이 올라오면 물을 뺀다(빌드), 이물질을 솎아낸다(정적 분석), 조리한다(테스트), 포장한다(컨테이너 이미지), 그리고 출하한다(배포). 각 단계에서 불량품이 발견되면 컨베이어가 멈추고 해당 제품만 제거하여 나머지 제품이 계속 흐를 수 있게 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 무중단 배포 전략 심층 비교

무중단 배포(Zero Downtime Deployment)는 서비스 운영 중 새 버전으로 전환하는 기술이다. 대표적인 세 가지 전략을 비교 분석한다.

| 배포 전략 | 아키텍처 및 트래픽 라우팅 | 롤백 속도 | 인프라 비용 | 적합 시나리오 |
|:---|:---|:---|:---|:---|
| **롤링 업데이트** | 구버전 Pod를 하나씩 종료하고 신버전 Pod를 순차 생성 | 느림 (구버전을 하나씩 다시 띄워야 함) | **최저** (기존 자원 한도 내에서 교체) | 리소스가 제한적이고 API 호환성이 완벽히 보장될 때 |
| **블루/그린 배포** | 신버전(Green)을 100% 미리 배포하고, 로드밸런서 스위칭으로瞬時 전환 | **최고** (타겟만 Blue로 변경하면 즉시 원복) | **최고** (순간적으로 2배 서버 필요) | DB 스키마가 완전히 분리되어 빠른 롤오버가 필수일 때 |
| **카나리 배포** | 신버전을 1%에만 배포하여 트래픽 일부만 라우팅, 모니터링 후 점진적 확대 | 빠름 (1% 라우팅만 제거하면 됨) | 낮음 | UI/UX 변경에 대한 실제 사용자 반응 검증이 필요할 때 |

카나리 배포의 핵심은 점진적 노출(Progressive Exposure)이다. 새로운 버전의 코드를 모든 사용자에게 한 번에 노출하는 대신, 소수의 사용자에게만 제공하고 그들로부터 피드백을 수집한 후 전면 확대 결정한다. 이는 위험을 관리 가능한 수준으로 축소하면서, 실제 프로덕션 데이터로 신기능의 유효성을 검증할 수 있게 한다. 카나리 배포는 A/B 테스트와 결합하면 더욱 강력하다. 두 버전을 동시에 운영하여 전환율, 에러율, 응답 시간 등을 통계학적으로 비교할 수 있다.

### 선언적 IaC vs 절차적 스크립트: 인프라 코드의 두 갈래

코드형 인프라(IaC)는 인프라를 코드로 관리하는 접근법에 따라 선언적(Declarative) 방식과 절차적(Imperative) 방식으로 나뉜다. 선언적 IaC는 최종 desired state(결과 상태)만 정의하고, 도구가 현재 상태와 desired state 사이의 차이를 계산하여 변경을 적용한다. Terraform, CloudFormation, Pulumi가 이 방식에 해당한다. 절차적 IaC는 How(어떻게)를 순차적으로 기술한다. Ansible의 일부 모듈이 이 방식에 해당한다.

| 비교 항목 | 전통적 ClickOps | 절차적 스크립트 | 선언적 IaC (Terraform) |
|:---|:---|:---|:---|
| **구성 방식** | 클라우드 콘솔 수동 클릭 | "어떻게" 할지를 스크립트로 작성 | "무엇이" 필요한지 최종 상태만 선언 |
| **멱등성** | 보장 불가 (두 번 실행 시 에러 발생 가능) | 직접 조건문 작성 필요 | **완벽 보장** (desired state 비교로 변경분만 적용) |
| **드리프트 감지** | 어려움 (문서와 실제 상태 불일치) | 방어하기 어려움 | **Git 코드가 곧 진실(SSOT)** |
| **재해 복구** | 며칠~수 주 (기억과 수동 작업에 의존) | 수 시간 | **수 분 내에 완전 복제 가능** |
| **문서화** | 수동 (설계 문서와 실제 다를 수 있음) | 스크립트 자체가 문서 | **Git 코드가 문서** |

### GitOps 풀 모델 vs 푸시 모델 보안 비교

보안 관점에서 GitOps의 풀 기반(Pull-based) 모델이 푸시 기반 모델보다 우월하다. 푸시 모델에서는 외부 CI 서버가 내부 Kubernetes 클러스터에 직접 접근해야 하므로, CI 서버가 Compromised 되면 클러스터 전체가 위험에 처한다. 풀 모델에서는 클러스터 내부의 에이전트(ArgoCD, FluxCD)만 Git을 감시하고, 외부에서 내부로의 접근 경로가 존재하지 않는다. 이는 제로 트러스트(Zero Trust) 네트워크 원칙에 부합한다.

### 📢 섹션 요약 비유

세 가지 배포 전략은 각각 다른 배달 방식에 비유할 수 있다. 롤링은 기존 배달원을 하나씩 교체하는 것이고, 블루/그린은 새 배달원을 전원 미리 고용하여 준비가 되면 동시에 교체하는 것이고, 카나리는 새 배달원을 1명만 먼저 투입하여 반응을 본 후 점점 더 많이 투입하는 것이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 시나리오 1: 마이크로서비스 파이프라인의 복잡성 관리

**문제 상황**: 50개 이상의 마이크로서비스를 가진 조직에서, 각 서비스마다 별도의 CI/CD 파이프라인을 구축했다. 결과적으로 파이프라인 설정 파일(YAML)이 수십 개로 복제되고, 공통된 CI 로직을 변경하려면 모든 파이프라인을 일일이 수정해야 하는 관리 부담이 발생했다. 또한, 서비스 간 배포 순서 의존성(예: 사용자 서비스는 결제 서비스보다 먼저 배포되어야 함)을 관리할 방법이 없었다.

**기술사적 결단**: 이 문제는 파이프라인 템플릿화와 오케스트레이션 도구 도입으로 해결해야 한다. 첫째, 파이프라인 템플릿화(예: Jenkins Shared Library, GitHub Actions Reusable Workflows)를 통해 공통 빌드/테스트 단계를 하나의 템플릿으로 정의하고, 각 서비스는 이 템플릿을 상속받는다. 둘째, 서비스 간 배포 순서 의존성은 스핀네이커(Spinnnaker)의 파이프라인 스태이지(Stage) 기능을 사용하거나, ArgoCD의 ApplicationSet과 조합으로 관리한다. 셋째, Helm 또는 Kustomize를 사용하여 환경(dev, staging, production)별 설정 차이를 관리한다.

### 시나리오 2: 데이터베이스 스키마 변경과 무중단 배포

**문제 상황**: 새로운 버전의 애플리케이션이 데이터베이스 스키마 변경(예: 컬럼 추가, 인덱스 생성)을 요구한다. 하지만 이 변경을 프로덕션에 배포하는 동안 기존 버전의 애플리케이션이 작동해야 하며, 롤백 시에도 기존 스키마로 돌아가야 한다. 이 두 가지 조건을 동시에 만족하는 배포 전략이 필요하지만, 가장 흔한 실수는 스키마 변경과 앱 배포를 동시에 행하는 것이다.

**기술사적 결단**: 이 문제는 "확장하고 계약하기(Expand and Contract)" 패턴으로 해결한다. 첫 번째 배포(Expand 단계)에서는 새 스키마를 적용하되, 기존 애플리케이션 코드에서는 새 컬럼/인덱스를 사용하지 않는다. 이 상태에서新老アプリケーション이 모두 작동한다. 두 번째 배포(Contract 단계)에서는 새 스키마를 사용하는 새 애플리케이션 코드를 배포한다. 세 번째 배포(cleanup)에서는 더 이상 필요 없는 기존 컬럼/인덱스를 제거한다. 롤백이 필요한 경우, 두 번째 배포에서 첫 번째 상태로 돌아가면 된다. 데이터베이스 마이그레이션 도구로는 Flyway와 Liquibase가 널리 사용된다. 이들은 migration 스크립트의 버전을 관리하고, 스키마 변경 이력을 추적한다.

### 시나리오 3: 빌드 캐시 최적화로 CI 속도 혁신

**문제 상황**: CI 파이프라인의 빌드 단계가 점점 느려져, 전체 빌드에 30분 이상 소요된다. 이는 개발자의 생산성을 저하시키고, 피드백 루프를 지연시킨다. 원인을 분석해보니, 대부분의 빌드 시간이 의존성 패키지 다운로드에 소비되고 있었다.

**기술사적 결단**: 빌드 캐시(Build Cache) 활용이 핵심 해결책이다. Docker 빌드의 경우, Dockerfile에서 순서상 변경이 적은 레이어(예: FROM, RUN apt-get install)를 먼저 배치하고, 자주 변경되는 레이어(예: COPY 소스 코드)를 나중에 배치한다.こうすることで, 소스 코드만 변경되면 기존 레이어 캐시를 활용하여 빌드가 수 분에서 수 초로 단축된다. Maven/Gradle의 경우, 로컬/원격 메이븐 레포지토리에 의존성을 캐시한다. Jenkins에서는 Remote Cache Server(Nexus, Artifactory)를 구성하여 Jenkins Agent 간 공유 캐시를 운영한다. npm/yarn에서는 npm ci 명령어를 사용하여 package-lock.json 기반으로 정확한 버전을 설치하고, CI 환경에서는 npm-cache를 활용한다.

### 도입 체크리스트

- **파이프라인 보안**: 시크릿(Secret) 관리는 어떻게 하고 있는가? GitHub Actions Secrets, Jenkins Credentials, HashiCorp Vault 등 安全한 Secret Management 도구를 사용하고 있는가?
- **파이프라인 가시성**: 빌드/배포 성공률, 평균 빌드 시간, 병목 구간을 실시간으로 모니터링하고 있는가?
- **롤백 전략**: 배포 실패 시 이전 버전으로의 롤백이 얼마나 빠른가? 자동 롤백(메트릭 기반)과 수동 롤백 모두 준비되어 있는가?
- **컨테이너 이미지 보안**: 빌드된 이미지의 취약점 스캐닝(Trivy, Clair)을 파이프라인에 통합하고 있는가? 불필요한 apt 패키지는 제거했는가?

### 안티패턴

- **CI 파이프라인의 Spaghetti 의존성**: 파이프라인 단계 간 의존관계가 복잡하게 얽혀 있어, 한 단계 실패 시 전체 흐름을 파악하기 어려운 상황. 파이프라인은 명확한 단일 방향 그래프(DAG) 구조를 유지해야 한다.
- **빌드 시간의 눈덩이 효과**: 빌드 속도 최적화를 게을리 하면, 한 번의 빌드가 5분이면 적당하다는 핑계로放縦하면 1시간이 되는 것이 수 개월 후의 현실이다. 빌드 속도 Budget(예: 10분 초과 시 알람)을 설정하여 관리해야 한다.
- **데스티네이션 드리프트**: Git 리포지토리의 목표 상태와 실제 클러스터 상태가 불일치하는 상황. ArgoCD에서는 Sync Wave와 Diff 설정을 통해 관리하며, drift가 발생하면 즉시 알람이 발생해야 한다.

### 📢 섹션 요약 비유

CI/CD 파이프라인을 요리사로 비유하면, 레시피(설계도)가 있고, 재료 점검(CI 빌드), 조리 테스트(CI 테스트)를 거친 후, 손님에게 提供(CD 배포)까지 자동화된 시스템을 운영하는 것과 같다.GitOps는 조리사(Customer)가 직접厨房(Git Repo)을Watch하여 재료가 replenished되면 자동으로 식재료를补充하는 시스템이다.

### GitOps의 미래: Cluster API와 GitOps의 결합

GitOps의 미래 진화 방향 중 하나는 Cluster API(CAPI)와의 결합이다. CAPI는 Kubernetes 클러스터 자체를 선언적으로 관리할 수 있는 프로젝트로, Git 리포지토리에 클러스터 구성을 저장하면 CAPI가 이를 기반으로 클러스터를 프로비저닝, 업그레이드, 스케일링한다. GitOps와 결합하면, 인프라(클러스터 구성)와 애플리케이션(Deployment, Service)을 동일한 Git 리포지토리에서 관리할 수 있게 된다. 이는 인프라와 애플리케이션의 배포가 분리되지 않는 완전한 선언적 관리의 궁극적 형태다.

### 관련 표준 및 참고

- **GitOps Working Group (CNCF)**: GitOps의 표준 용어, 모범 사례, 검증 방법을 정의하는 산업 워킹 그룹
- **ArgoCD**: CNCF Graduated 프로젝트로서 Kubernetes용 GitOps 도구
- **FluxCD**: GitOps 도구로, ArgoCD와 경쟁 관계
- **Helm**: Kubernetes 패키지 매니저로, 복잡한 애플리케이션 설정을 템플릿화
- **Kustomize**: Kubernetes YAML 오버레이 방식으로, 템플릿 없이 환경별 설정을 관리

### 📢 섹션 요약 비유

CI/CD와 GitOps의 조합은 마치 모든 건축 도면(Git Repo)을 중앙 설계 사무실(Git)에 저장하고, 건설 현장(Kubernetes Cluster)의 로봇(ArgoCD)이 도면 변경을 감지하면 자동으로 시공하는 스마트 건설 시스템과 같다. 사람의介입이 줄어들수록 시공 속도와精度가 향상되고, 실수 가능성도 극적으로 줄어든다.

## 👶 어린이를 위한 3줄 비유 설명

1. CI/CD는 우체국에 비유할 수 있어. 편지를 쓰면(코드 작성) 우체부가 자동으로 분류하고(지속적 통합), 배달 루트를 최적화하여(자동 배포) 가장 빠르게 손님에게 배달하는 시스템이야.

2. GitOps는 학교 숙제의 원고(Desired State)를 항상 교실 선반(Git Repo)에 올려두고, 조교(ArgoCD)가 학생들의 책상(클러스터)이 선반과 다른지를 매일检查하여, 다르면 바로 올바른 상태로 맞추는 것과 같아.

3. 카나리 배포는 새 맛의 아이스크림을 모든 사람에게 파는 게 아니라, 먼저 친구 1명에게만试食시켜보고, 맛있다면 점점 더 많은 사람에게 판매하는 신중한 방법이야. 만약 친구가 맛없다고 하면 바로 기존 아이스크림만 계속 파는 것처럼即時 롤백도 가능해!
