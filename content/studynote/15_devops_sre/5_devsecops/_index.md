> **핵심 인사이트**
> 1. **본질**: DevSecOps는 DevOps의 신속한 CI/CD 파이프라인 내에 보안(Security) 검증 및 차단 프로세스를自动化 도구로 내재화하여, 배포 속도를 해치지 않으면서 코드 결함을 조기에 발견하고, 시프트 레프트(Shift-Left)를 통해 개발 단계에서 보안을확보하는 문화적 기술적 실천이다.
> 2. **가치**: 소규모 결함修正 비용은 개발 단계에서 100배 저렴하며, 컨테이너 이미지 스캐닝과 SAST/DAST 통합으로 보안 취약점을 사전 차단하여 데이터 유출과 공급망 공격 위험을 최소화한다.
> 3. **융합**: SBOM(Software Bill of Materials) 추출, 시크릿 관리(Secret Management), 컨테이너 이스케이프 방어, 정책 애즈 코드(OPA)와 결합하여 클라우드 네이티브 환경의 전生命周期 보안을実現한다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 보안의 역사적 격리에서 DevSecOps로

전통적 보안 모델에서는 보안팀이 개발 프로세스와 완전히 분리되어 있었다. 개발팀이 코드를 완성하면, 보안팀이 배포前夕에_security scan을 수행하여ulnerabilities를 보고했다. 이 방식의根本적 문제점은 늦은 탐지 시점이다. 개발 단계 초기에 발견했다면修正 비용이 1이라면, 프로덕션에서 발견한 취약점修正 비용은 100이라는 研究 결과가 있다(IBM System Science Institute). 또한, 전통적 보안 모델에서는 보안이 배포의阻碍要因으로 인식되어, 개발팀이 보안 검증을 회피하려는 incentive가 작동했다. 보안팀과 개발팀의 목표가 상충하는 "게임"이 된 것이다.

DevSecOps는 이Game을根本적으로 바꿈다. 보안 검증을 CI/CD 파이프라인의 자동화된 단계로組み込み,"보안 검사를 통과해야만 배포가 가능하다"는 구조적 강제를 도입한다. 보안은 더 이상 배포의阻碍要因이 아니라, 배포 파이프라인의 필수 구성 요소가 된다. 개발자는 파이프라인을 통해 즉각적인 보안 피드백을 받고, 보안팀은 정책과 도구를 제공하면서 역할이転換된다.

### 시프트 레프트: 보안을 왼쪽으로 이동

시프트 레프트(Shift-Left)는 소프트웨어 개발 생명주기(SDLC)에서 오른쪽 끝(테스트/운영)에 있던 활동을 왼쪽(기획/개발/빌드)으로 앞당겨 결함을 조기에 발견하고 비용을 절감하는 사상이다. 보안 관점에서 보면, 전통적 모델에서 보안 테스트는 SDLC의 맨 오른쪽에 위치했다. 개발 후 QC를 거치고, 스테이징에서 보안 테스트를 하고, 프로덕션 직전에 모의 침투 테스트를 하는 것이 일반적流程였다. 그러나 코드 베이스의 80%가 개발 단계에서 작성되므로, 개발 단계의 보안漏洞이Production에서 발견될 확률이 가장 높다.

시프트 레프트 보안의 핵심 원칙은 "보안은 개발자의 책임이다( Security is Developer's Job)"이다. 개발자가 코드를 작성하는 단계에서 보안漏洞을 발견하면,修正 비용이 최소화되고,Learning 효과도 극대화된다. 보안 검사 도구를 IDE(Integrated Development Environment)나Git pre-commit hook에統合하면, 개발자가 코드를 제출하는 순간 보안 피드백을 받을 수 있다. 이는 보안 교육의 효과도 높아진다. 개발자가 자신의 코드에서 보안 문제를 발견하고修正하는 경험이, 단순한Security 강의보다 훨씬 효과적인Learning이 된다.

### 공급망 공격의 일상화: SolarWinds의警示

2020년 발생한 솔라윈즈(SolarWinds) 공격은 소프트웨어 공급망 공격(Supply Chain Attack)의 대표적 사례이다. 솔라윈즈는 네트워크 관리 도구를 만드는 회사로, Their software build pipeline이 해킹되어, 정식 서명된 소프트웨어에 악성코드가 삽입되었다. 이 악성코드가 포함된 업데이트가 18,000개 이상의 고객 조직에 배포되었다. 이는"정식 서명된 소프트웨어는安全하다"는 기존 가정에根本적 의문을 제기했다. software supply chain의 어느 단계(개발, 빌드, 배포)에서도 공격이 가능하며, 그后果가 엄청나다는 것을 보여주었다.

이러한 공급망 공격에 대응하기 위해, SBOM(Software Bill of Materials) 추출이 필수적되고 있다. SBOM은 소프트웨어를 구성하는 모든 오픈소스 부품, 라이브러리, 버전 정보를 명세한 자재 명세서이다. SPDX(ISO/IEC 5962)와 CycloneDX가 대표적 SBOM 표준이다. SBOM이 있으면, 취약점 게시에 따라 영향도 분석이 빨라지고, 공급망 공격 탐지 시Impact 분석이 가능하다.

### 📢 섹션 요약 비유

DevSecOps는 요리教学中 비유할 수 있다. 전통적 모델은 요리사가 요리를 완성한 후 식품검疫소(보안팀)가 검검해서 문제가 있으면 폐기처분했다. DevSecOps는 조리하는全過程에 식품의 安全檢查儀(보안 도구)를 설치하여, 날 재료검수(의존성 스캐닝), 조리 과정监控(SAST), 완성된 음식抽查(DAST)를 자동화하여,食品安全 문제가 발생하는 것을根本적으로 예방한다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### DevSecOps 파이프라인: 보안 단계의 5層構造

DevSecOps 파이프라인에서 보안은 여러 단계로 구성된다. 1단계는 시크릿 검색(Secret Scanning)이다. Git 리포지토리에서 API 키, 패스워드, 개인 키 등 민감 정보가 커밋되었는지 검출한다. GitGuardian, Gitleaks 등의 도구가 사용된다. 2단계는 정적 애플리케이션 보안 테스트(Static Application Security Testing, SAST)이다. 소스코드를 실행하지 않고 구문, 데이터 흐름을 분석하여 SQL 인젝션, 버퍼 오버플로우, XSS 등 취약점을 검출한다. SonarQube, Checkmarx, Semgrep 등이 있다. 3단계는 소프트웨어 구성 분석(Software Composition Analysis, SCA)이다. 오픈소스 라이브러리의 알려진 취약점(CVE)과 라이선스 충돌 위험을 검사한다. Snyk, Dependabot, FOSSA 등이 있다. 4단계는 동적 애플리케이션 보안 테스트(Dynamic Application Security Testing, DAST)이다. 애플리케이션을 런타임으로 실행한 상태에서 외부 해커 관점으로 웹 취약점 스캐닝/퍼징 공격을 수행한다. OWASP ZAP, Burp Suite 등이 있다. 5단계는 컨테이너 이미지 스캐닝(Container Image Scanning)이다. 도커 이미지 빌드 시 OS 기본 패키지와 애플리케이션 취약점을 탐지한다. Trivy, Clair, Grype 등이 있다.

### SAST: 코드 취약점의 조기 탐지

SAST(Static Application Security Testing)는 소스 코드를 분석하여 보안漏洞을 탐지하는 기술이다. SAST의 동작 방식은 다음과 같다. 먼저, 소스 코드를 파싱하여 AST(Abstract Syntax Tree)를 생성한다.次に, 미리定義된 취약점 패턴(규칙, 규칙 Against Coding Standards)과 코드 구조를 비교한다. 데이터 흐름( Data Flow) 분석을 통해,tainted input(사용자 입력)이 security-sensitive function(예: SQL 쿼리 실행)에 도달하는 경로(Taint Path)를 추적한다. 최종적으로 패턴이나tainted 경로가 발견되면, 취약점으로 보고한다.

SAST의 장점은 다음과 같다. 코드 작성 단계(IDE에서) 또는 commit/PR 시점에 즉각 실행 가능하여フィードバックが速い。 source 코드 수준에서 취약점 위치(파일, 줄 번호)를 정확히 알려준다. 코드 커버리지가 높아 모든 코드 경로를 분석한다. 그러나SAST의 단점도 있다. 많은 False Positive(가양성)가 발생할 수 있어, 개발자가疲れる。 설계 수준의 보안 문제는 탐지 어렵다. third-party 라이브러리나 런타임 환경 문제는 파악 못한다.

### DAST vs SAST: 동적 vs 정적

DAST(Dynamic Application Security Testing)와 SAST는 상호 보완적이다. SAST가 소스 코드 수준에서 분석하는 반면, DAST는 실행 중인 애플리케이션에 실제 공격 페이로드를 보내어 취약점을 탐지한다. DAST의 동작 방식은 다음과 같다. 테스트 대상 애플리케이션에 HTTP 요청을 보내고, 응답을 분석한다. 사전에 정의된 공격 시나리오( SQL 인젝션, XSS, CSRF 등)를 적용한다. 응답에서 예상치 못한 동작(에러 메시지, 데이터 유출 등)이 있으면 취약점으로 판단한다.

| 비교 항목 | SAST | DAST |
|:---|:---|:---|
| **분석 시점** | 코드 작성/컴파일 시 (Shift-Left) | 런타임/배포 후 |
| **분석 대상** | 소스 코드 | 실행 중인 애플리케이션 |
| **세밀도** | 코드 위치까지 정확한 경로 제공 | 취약점 존재 여부만 확인 |
| **False Positive** | 높음 (많은 경고) | 낮음 (실제 공격 기반) |
| **설계 결함 탐지** | 어려움 | 가능 |
| **속도** | 빠름 (컴파일러 수준) | 느림 (실제 요청 전송) |
| **접근 필요** | 소스 코드 필요 | 실행 중인 앱만 필요 |

### 시크릿 관리: 하드코딩의 종말

시크릿(Secret)은 API 키, 데이터베이스 패스워드, 개인 키, OAuth 토큰 등 민감한 자격 증명 정보를 말한다. 전통적 모델에서는 이러한 시크릿을 소스코드에 하드코딩하거나, 설정 파일에 평문으로 저장했다. 이 방식의 문제점은 다음과 같다. Git 리포지토리에 시크릿이 커밋되면, 히스토리 자체에 영구히 남는다. 협업자나 퇴사자가 시크릿을 알게 된다. 시크릿이 변경되면 모든 관련 코드/설정을 수동 업데이트해야 한다. 시크릿이 유출되면 시스템 전체가 Compromised 될 수 있다.

시크릿 관리를 위한 올바른 아키텍처는 외부 시크릿 저장소(HashiCorp Vault, AWS Secrets Manager, Azure Key Vault)를 사용하고, 애플리케이션은 런타임에 시크릿을 가져오는 방식이다. Kubernetes 환경에서는External Secrets Operator(ESO)를 사용하여 Secrets Store와連携する。 컨테이너 이미지에는 시크릿을 포함하지 않고, 파드 시작 시 Secret Provider를 통해 마운트하거나 환경 변수로注入한다. GitOps 환경에서는 시크릿을 Git에 평문으로 저장하는 대신, Sealed Secrets나 SOPS(Age encryption)를 사용하여 암호화된 형태로만Git에 저장한다.

### 컨테이너 보안: 네 계층의 방어

컨테이너 환경의 보안은 네 개의 계층으로Defense in Depth(多层防御)를 실현한다. 1단계는 컨테이너 이미지 스캐닝(Image Scanning)이다. 빌드 시 Base 이미지와 애플리케이션 의존성의 알려진 취약점(CVE)을 검사한다. CVE 데이터베이스와 연동하여 심각도(Severity)별로 필터링한다. 예컨대严重(Critical), 高(High) 취약점이 발견되면 빌드를 실패 처리한다. 2단계는 최소 권한 원칙(Least Privilege Principle)이다. 컨테이너를root 대신 일반 사용자로 실행한다. 불필요한 Linux 기능(Capabilities)을 제거한다. 읽기 전용 파일시스템(ReadOnly Root Filesystem)을 적용한다. 3단계는 런타임 보안(Runtime Security)이다. 컨테이너의 비정상적举动(시스템콜, 네트워크 활동)를 모니터링한다. Falco, Sysdig와 같은 런타임 보안 도구가 사용된다. 4단계는 네트워크 격리(Network Isolation)이다. 쿠버네티스 NetworkPolicy를 적용하여 파드 간 통신을 제한한다. 서비스 메시(Istio)의 mTLS로서비스 간 통신을 암호화한다.

### 📢 섹션 요약 비유

DevSecOps 보안 체계를 성 Security Screening에 비유할 수 있다. 1단계는 신원 확인(시크릿 관리), 2단계는 짐 검사(SAST/SCA), 3단계는 신체 검사(DAST/ 런타임), 4단계는 출입 카드 발급(최소 권한 원칙)이다. 각 단계에서 의심征兆이 발견되면 입국이 거부(빌드 실패/배포 차단)된다.

## Ⅲ. 융합 비교 및 다각도 분석

### SAST vs DAST vs IAST vs SCA: 보안 테스팅 方法論 비교

보안 테스팅에는 다양한 方法論이 있고, 각각 StrengthsとWeaknesses가 있다. SAST는 정적 분석으로 빠르고 포괄적이지만 가양성이 많고, 설계 문제는 못 찾는다. DAST는 동적 분석으로 실제 공격을 시뮬레이션하여 가양성이 적지만,Slow하고 커버리지가 제한적이다. IAST(Interactive Application Security Testing)는 에이전트를 애플리케이션에 심어 실행 중의 데이터 흐름과 공격 페이로드를 동시에 분석한다. SAST의 정확도와 DAST의 현실성을 결합한 방식이다. RASP(Runtime Application Self-Protection)는 애플리케이션 자체에 라이브러리로 내장되어, 실제 공격을 실시간으로 탐지하고 차단한다. 방어 시점에서는 가장 빠르지만, 애플리케이션 성능에 영향을 줄 수 있다.

| 비교 항목 | SAST | DAST | IAST | RASP |
|:---|:---|:---|:---|:---|
| **분석 시점** | 컴파일/빌드 시 | 런타임 | 런타임 | 런타임 |
| **속도** | 빠름 | 느림 | 중간 | 매우 빠름 |
| **False Positive** | 높음 | 낮음 | 낮음 | 매우 낮음 |
| **코드 위치** | 정확 | 불명확 | 정확 | 정확 |
| **성능 영향** | 없음 | 일부 | 일부 | 있음 |
| **설계 결함** | 불가 | 가능 | 일부 | 불가 |

### 전통적 보안 vs DevSecOps: 문화적 Paradigm 차이

전통적 보안과 DevSecOps는 단순히 도구의 차원이 아니라, 문화와 프로세스의 Paradigm 차원이다. 전통적 보안 모델에서는 보안팀이"감시자(Gatekeeper)" 역할을 하며, 개발팀의 배포를 심사하고 차단하는 권한을 가졌다. 이 모델에서는 보안이 개발의 속도를 저해하는障碍物로 인식되었고, 개발자들은 보안을"피해야 할 것"으로 여기게 되었다. DevSecOps에서는 보안이"공급자(Enabler)" 역할로 전환된다. 보안팀이 도구와 정책을 개발팀에게 제공하고, 개발팀이 자체적으로 보안을 확보할 수 있게支援한다. 보안은 더 이상"检查서"가 아니라"가이드라인"이 된다.

| 비교 항목 | 전통적 보안 | DevSecOps |
|:---|:---|:---|
| **보안 주체** | 전용 보안팀 | 개발팀 + 보안팀 (공유) |
| **보안 시점** | SDLC 후반 (우측) | SDLC 전반 (좌측으로 시프트) |
| **주요 활동** | 침투 테스트, 모의 해킹, 감사 | 자동화된 보안 스캐닝, 정책-as-코드 |
| **보안의 위치** | 배포 후 (운영 환경) | 배포 전 (파이프라인 내) |
| **문화** | 기밀 유지, 정보 격리 | 공유, 협력, 학습 |
| **속도** | 느림 (긴 승인 프로세스) | 빠름 (자동화) |
| **적응성** | 낮음 (변화에 민감) | 높음 (애자일) |

### CSPM vs CWPP vs CNAPP: 클라우스 보안 플랫폼进化

클라우드 보안 플랫폼은 세대를 거쳐 진화해왔다. CSPM(Cloud Security Posture Management)은 클라우드 인프라의 잘못된 설정(Misconfiguration)을 지속적으로 감시하고 자동修正하는 도구이다. AWS Config, Prisma Cloud CSPM이 대표적이다. CWPP(Cloud Workload Protection Platform)는 VM, 컨테이너, 서버리스 등 런타임 환경 내부의 악성코드 탐지와 메모리 보호 기능을 제공한다. 보안의 초점이"설정( posture)"에서"워크로드(Workload)"로 확대된 것이다. CNAPP(Cloud-Native Application Protection Platform)는 CSPM과 CWPP, CI/CD 스캐닝을 단일 통합 대시보드로 묶은 최신 트렌드이다. 분리된 여러 도구를 사용하는 것보다, 통합 플랫폼이 더 эффектив하다. Prisma Cloud, Wiz가 대표적이다.

### 📢 섹션 요약 비유

여러 보안 도구를 사용하는 것은 항구에서 화물검사를 각각 다른 팀이 하는 것과 같다. 세관(CSPM)은 화물 신고内容을 확인하고,検疫(CWPP)은 화물 자체를 검사하고,保安(CNAPP)는 전체流程를 한눈에监控한다. CNAPP는 이러한 여러檢疫 절차를 한 번에 통합 처리하여, 화물의 안전과 통관 속도를 동시에保障한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 시나리오 1: 제로데이 취약점 대응의 속도 전투

**문제 상황**: 화요일 오후, 주요 오픈소스 라이브러리(Log4j)에서 критическая 취약점( Log4Shell, CVE-2021-44228)이 공개되었다. 이 취약점은 Remote Code Execution(RCE) 가능성이 있어 즉각적인_patch가 필요하다. 그러나 조직에는 50개 이상의 마이크로서비스가 있고, 각각이 다양한 의존성을 가진다. 이 취약점의 영향을받는 서비스 목록을即時に 파악하고, 패치를 적용해야 한다.

**기술사적 결단**: 이 상황에서의 핵심Challenge는速度이다. 전통적 방식("다운로드 받아서 확인")으로는追いつかない。 DevSecOps가 갖추고 있어야 하는 것은 다음과 같다. 첫째, SCA(Software Composition Analysis)도구가 CI/CD 파이프라인에 통합되어 있어, 모든 서비스의 의존성이 지속적으로監視되고 있다. 따라서 새로운 CVE 공개 시,影響 분석이 즉시 가능하다.둘째, SBOM(Software Bill of Materials)이 추출되어 있어, 전체 서비스의 부품 목록이 문서화되어 있다. Log4j 사용 여부를 SBOM에서 즉각 검색한다.셋째, 컨테이너 이미지 스캐닝이 빌드 시점에서 이미 수행되어 있어, 심각한 취약점이 있는 이미지는 레지스트리에 업로드 자체가 차단된다. 패치 적용은 새로운 Log4j 버전을 의존성에 지정하고, CI/CD 파이프라인을 통해 모든 서비스에 자동 배포한다. 롤링 업데이트로零 downtime으로 패치를 반영한다.

### 시나리오 2: Git 유출 시크릿 사고 대응

**문제 상황**: 개발자가 실수로 AWS API 키를 GitHub 퍼블릭 리포지토리에 커밋했다. 5분 후 GitGuardian이 알람을 보냈지만, 이미 그 사이 API 키가 악용되어 10만 달러 이상의AWS 비용이 청구되었다.事故 대응 프로세스가 명확하지 않아,対応遅延과 함께 트라우마가 확산되었다.

**기술사적 결단**: 이 상황의根本 문제는 事後 대응이 而不是事前防止이다.事前防止策으로는, Git pre-commit hook에 Gitleaks나Talisman을 설치하여 커밋前に시크릿을検出하면提交을 차단한다. GitHub의 Secret Scanning과 Push Protection을 활성화하여, 시크릿 커밋 시Push를 бл럭한다. IAM 정책에서 크레딧 카드 대신Specific 권한(Principle of Least Privilege)을 부여하여,万一 키가 유출되더라도 영향 범위를限定한다.事後 대응 프로세스로는, 키 Rotation 절차를 즉시 실행하고, CloudTrail 로그를 확인하여키 악용 여부를 확인하며, forensics를 수행하여침입 경로와 영향을 분석한다. 또한,보험 또는 FinOps Alert로異常 비용 발생을 감지하면自動 알람을 설정한다.

### 시나리오 3: PCI-DSS 준수 환경에서의 CI/CD

**문제 상황**: 금융 기관에서 PCI-DSS( Payment Card Industry Data Security Standard) 준수를要求받는 결제 시스템을 운영하고 있다. PCI-DSS는嚴格한 보안 요구사항(데이터 암호화, 접근 통제, 로깅 등)을要求하며, 모든 변경에 대한 감사 로그를 유지해야 한다. 그러나 보안 검증 단계가 많아 CI/CD 파이프라인이 4시간 이상 소요되어, 비즈니스 민첩성이 저하되고 있다.

**기술사적 결단**: PCI-DSS 준수를 유지하면서도 CI/CD 속도를 높이는 전략이 필요하다.첫째, 파이프라인 단계 분리이다. SAST/SCA(빠른 단계)는 CI 파이프라인 내에서 병렬로 실행하여 수 분 내에 결과를 얻는다. DAST(느린 단계)는 별도의 야간 파이프라인으로 분리하여, 낮 시간에는 빠른 보안 검사만 거치고 야간에 포괄적 보안을実行한다.둘째, 승인 게이트(Approval Gate)의 자동화이다. 수동 승인이 필요한 보안 검사를 OPA(Open Policy Agent) 기반의 자동화된 정책 검사로 대체한다. 정책이コード로管理되어, 변경 시 마다Human Approval이 아니라 자동 검증으로Compliance를 증명한다.셋째, 정책-as-코드(Policy as Code) 적용이다. Sentinel(HashiCorp), OPA(Rego language) 등을 사용하여, Terraform deployment 단계에서 보안 및 컴플라이언스 정책 위반을自动検出하여deployment를 차단한다.

### 도입 체크리스트

- **시크릿 관리**: HashiCorp Vault, AWS Secrets Manager 등 외부 시크릿 저장소를 사용하고 있는가? 시크릿이 코드/설정 파일에 하드코딩되지 않도록管理하고 있는가?
- **파이프라인 보안 도구 통합**: SAST, SCA, 컨테이너 스캐닝이 CI/CD 파이프라인에統合되어 있는가? 빌드 실패 시 배포가 차단되는 구조인가?
- **컨테이너 이미지 무결성**: 빌드된 이미지에 디지털 서명(Image Signing)을 적용하고 있는가? 컨테이너 레지스트리에서 서명되지 않은 이미지는pull을 차단하는가?
- **컴플라이언스 감사**: 규제 요구사항(PCI-DSS, GDPR, ISMS 등)에 대한 증거(evidence)가 자동 수집되어 감사 준비가 가능한 상태인가?

### 안티패턴

- **보안 도구 민갈**: 보안 도구를 CI/CD에追加했지만, 심각한 취약점이 발견되어도 무시하고 배포를 강행하는" alerte fatigue" 상황. 보안 알람의 잦은 False Positive가開發팀의 불신으로 이어져, 보안 도구의 효과가 없다.
- **컨테이너 이미지의 latest 태그**: 이미지에 latest 태그를 사용하면, 매번 다른 버전을pull하여 reproducibility가 없고, 취약점 스캐닝 결과가 일관되지 않다. 항상 구체적Digest(SHA256)로 이미지를 指定해야 한다.
- **시크릿을环境 변수에만 의존**: 컨테이너 환경 변수는 procfs를 통해 호스트에서可視화될 수 있으므로, 시크릿 관리에 환경 변수만 사용하는 것은不十分하다. Kubernetes Secret이나 Vault의动态 시크릿을活用해야 한다.

### 📢 섹션 요약 비유

DevSecOps 보안 체계를 요리节目中 비유하면, ISA(Ingredient Safety Assessment, SAST)는 재료의 산지를確認하여 농약 잔류 물질을檢출하는 것이고, DAST(Delivery Assessment)는 배달된 음식을 실험실에 보내어 위생 상태를 확인하는 것이다. 만약 재료검사에서 문제가 발견되면 당장 해당 재료로 만든 요리는 출하되지 않고, 배달 검사에서도 문제가 있으면 해당 식당은 즉시 운영 중단된다. 이러한 多层防御를 통해食品安全(보안)을担保한다.

### 정책-as-코드: 규제 준수의 자동화

규제 준수(Compliance)는 전통적으로 ManualAuditing과 Documentation을 통해 이루어졌다. 그러나 이것은 시간이 많이 소요되고, 인간의 판단 오류가 개입될 수 있다. 정책-as-코드(Policy as Code)는 보안 및 규제 정책을 Rego(Sentinel), HCL(Sentinel), YAML(OPA) 등의 코드 언어로定義하고, IaC deployment 단계나 Kubernetes API 서버 호출 시점에 강제 검증 필터링하는 방법이다. OPA(Open Policy Agent)와Gatekeeper는 Kubernetes 환경에서 Policy as Code를実現하는 대표적 도구이다. 예를 들어,"컨테이너는 Privileged 모드로 실행될 수 없다", "스토리지 클래스는 암호화되지 않은 것을 사용할 수 없다"와 같은 정책을 정책 코드로定義하면, 해당 정책을 위반하는 리소스는 deployment가 차단된다. 이렇게 하면 규제 준수가代码 배포의 일부로 자동화되어, 인간의 개입 없이Compliance가执行된다.

### 관련 표준 및 참고

- **OWASP Top 10**: 웹 애플리케이션 보안 취약점의 대표적 목록
- **CVE (Common Vulnerabilities and Exposures)**: 알려진 취약점에 대한 표준화된 식별자
- **SBOM (Software Bill of Materials)**: SPDX, CycloneDX 표준
- **PCI-DSS (Payment Card Industry Data Security Standard)**: 결제 카드 데이터 보안 표준
- **NIST SP 800-53**: Federal 정보 시스템 보안 통제 목록
- **OPA (Open Policy Agent)**: 정책-as-코드 실행을 위한 CNCF 도구

### 📢 섹션 요약 비유

DevSecOps의 보안 문화는 놀이공원 입구의 Security Screening과 같다. 입장 전 모든 방문객(코드)은Metal Detector(SAST), 짐 X-ray(SCA), 신원증检查(시크릿 관리)를 거친다. 문제가 없으면 입장(배포)이 허용되고, 만약 위험 물질(취약점)이 발견되면即時入场거부(빌드 실패)된다. 놀이공원 안에서도(런타임) Surveillance Camera(런타임 보안)가 지속적으로监控하여, 안에서 문제가 발생하면即時 대응한다.

## 👶 어린이를 위한 3줄 비유 설명

1. DevSecOps는 비행기 安全検査와 같아. 승객(코드)이 비행기에 타기(배포) 전에security scanner(SAST), 짐 검사(SCA), 신원 확인(시크릿 관리)을 거치는데, 만약 위험 물건(취약점/시크릿 유출)이 발견되면即時 입국이 거부(빌드 실패)돼.

2. 시프트 레프트(Shift-Left)는 급식실 위생 검사와 같아서, 음식(코드)을 만든 직후(개발 단계)에 맛평가(보안检查)를 하면 문제가 있어도 재료 비용만 loss이지만, 이미 배달(배포)된 후에 문제가 발견되면全部 폐기해야 하듯이, 문제発見 시점의 차이에 따라代价이 엄청나게 달라져.

3. 공급망 보안(SBOM)은 택배 물류의箱清单(箱子列表)과 같아서,万一 문제 제품(취약점 포함 부품)이 발견되면,箱清单을 보면 어떤 상품(서비스)에 들어갔는지即時に알 수 있어서, 리콜(패치 대상) 범위를 Quickly 결정할 수 있어!
