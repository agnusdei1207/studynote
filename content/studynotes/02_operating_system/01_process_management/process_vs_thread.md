# 프로세스 vs 스레드 (Process vs Thread)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 운영체제로부터 자원을 할당받는 작업의 단위인 **프로세스(Process)**와, 프로세스 내부에서 실행 흐름을 담당하는 최소 실행 단위인 **스레드(Thread)**는 현대 컴퓨팅의 병렬성과 동시성을 구현하는 핵심 메커니즘입니다.
> 2. **가치**: 프로세스는 독립된 메모리 공간을 통해 높은 안정성을 제공하며, 스레드는 자원 공유를 통한 빠른 통신과 낮은 생성/전환 비용으로 시스템의 응답성과 처리량(Throughput)을 극대화합니다.
> 3. **융합**: 멀티코어 프로세서 아키텍처와 결합하여 CPU 바운드 작업의 병렬 처리와 I/O 바운드 작업의 비동기 처리를 최적화하며, 최근에는 고루틴(Goroutine)이나 코루틴(Coroutine) 같은 초경량 실행 단위로 진화하고 있습니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 명확한 정의 (Definition)
**프로세스(Process)**는 실행 중인 프로그램(Program in execution)을 의미하며, 운영체제로부터 독립된 메모리 공간(Code, Data, Stack, Heap)과 시스템 자원(파일 핸들, I/O 상태 등)을 할당받는 **'자원 소유의 단위'**입니다. 반면, **스레드(Thread)**는 프로세스 내에서 실행되는 흐름의 단위로, 프로세스가 할당받은 자원의 일부(Stack)만 독립적으로 가지고 나머지(Code, Data, Heap)는 다른 스레드와 공유하는 **'CPU 스케줄링의 단위'**입니다.

### 💡 2. 구체적인 일상생활 비유
프로세스와 스레드는 **'회사(Company)'와 '직원(Employee)'**의 관계와 정확히 일치합니다. 
하나의 **회사(프로세스)**는 고유한 사무실(메모리 공간), 사업자 등록번호(PID), 예산(자원)을 가집니다. 다른 회사와는 벽으로 철저히 격리되어 있어 옆 회사가 망해도 우리 회사는 안전합니다. 하지만 회사 안의 **직원들(스레드)**은 사무실 공간과 탕비실, 비품(공유 자원)을 함께 사용하며 서로 빠르게 대화(통신)할 수 있습니다. 직원이 많아지면 일을 동시에 처리할 수 있어 효율적이지만, 두 직원이 하나의 펜을 동시에 잡으려고 싸우면(경쟁 상태) 문제가 생깁니다.

### 3. 등장 배경 및 발전 과정
1. **기존 기술의 치명적 한계점 (단일 작업의 비효율성)**: 
   초기 OS는 한 번에 하나의 프로세스만 실행했습니다. 이후 멀티태스킹이 도입되었으나, 새로운 작업을 수행하기 위해 매번 무거운 프로세스를 통째로 복사(fork)하고 생성하는 과정에서 막대한 시간과 메모리 낭비가 발생했습니다. 또한, 서로 격리된 프로세스 간에 데이터를 주고받으려면 복잡하고 느린 IPC(Inter-Process Communication) 과정을 거쳐야만 했습니다.
2. **혁신적 패러다임 변화의 시작**: 
   프로그램이 복잡해지면서(예: 웹 브라우저가 화면을 그리면서 동시에 파일을 다운로드해야 함) 프로세스보다 가벼운 실행 단위가 필요해졌습니다. 프로세스의 자원을 공유하면서 실행 흐름만 분리하는 '경량 프로세스(LWP, Light Weight Process)' 즉, 스레드 개념이 도입되었습니다. 이를 통해 컨텍스트 스위칭 오버헤드를 줄이고 시스템 성능을 획기적으로 끌어올렸습니다.
3. **현재 시장/산업의 비즈니스적 요구사항**: 
   현대의 서버는 수만 개의 동시 접속을 처리해야 합니다(C10K 문제). 전통적인 스레드 생성 방식만으로는 메모리 부족과 컨텍스트 스위칭 부하를 감당하기 어려워졌습니다. 이에 따라 커널 스레드와 사용자 스레드를 효율적으로 맵핑하는 모델(N:M)이나, 함수 호출 수준에서 비동기 처리를 수행하는 코루틴(Coroutine) 기반의 비동기 프로그래밍이 주류로 떠오르고 있습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

프로세스와 스레드의 가장 큰 차이는 메모리 구조와 이를 관리하는 제어 블록(PCB/TCB)의 구성에 있습니다.

### 1. 핵심 구성 요소 (표)

| 요소명 (Module) | 상세 역할 | 내부 동작 메커니즘 | 보관 위치 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **PCB (Process Control Block)** | 프로세스의 모든 상태 정보를 담은 운영체제의 장부 | PID, 프로세스 상태, PC, 레지스터 값, 메모리 관리 정보(페이지 테이블), 열린 파일 목록 저장 | 커널 메모리 (O/S 영역) | 회사의 '사업자 등록증 및 전체 자산 대장' |
| **TCB (Thread Control Block)** | 스레드별 실행 상태 정보를 담은 장부 | Thread ID, Stack Pointer, PC, 레지스터 셋 등 실행 흐름에 필요한 최소 정보 저장 | 커널 메모리 (PCB 내부 연동) | 직원의 '업무용 수첩 및 개인 소지품 목록' |
| **Stack (스레드 전용)** | 함수 호출 시 로컬 변수 및 복귀 주소 저장 | LIFO(Last-In-First-Out) 방식으로 동작하며, 스레드마다 독립적으로 할당되어 실행 흐름 보호 | 각 스레드의 개별 메모리 | 각 직원의 '개인 책상 서랍' |
| **Heap / Data / Code (공유)** | 동적 할당 메모리, 전역 변수, 실행 코드를 저장 | 모든 스레드가 동일한 주소 공간을 참조하여 자유롭게 데이터 공유 및 함수 호출 가능 | 프로세스 공유 메모리 영역 | 회사의 '공용 탕비실, 화이트보드, 매뉴얼' |
| **Context Switcher** | CPU 실행권을 한 단위에서 다른 단위로 교체 | 현재 CPU 상태를 PCB/TCB에 저장하고, 다음 대상의 상태를 CPU 레지스터에 복원(Restore) | 운영체제 스케줄러 | 일하던 사람이 교대할 때 '업무 인수인계' |

### 3. 심층 동작 원리 (컨텍스트 스위칭의 오버헤드 분석)
CPU가 실행 중인 프로세스나 스레드를 바꿀 때 발생하는 '컨텍스트 스위칭(Context Switching)'의 심층 메커니즘입니다.

1. **상태 저장 (Save Context)**: 하드웨어 인터럽트나 시스템 콜이 발생하면, 현재 실행 중인 단위의 PC(Program Counter), 레지스터(General Purpose, Floating Point), 스택 포인터 등을 해당 PCB나 TCB에 저장합니다.
2. **커널 모드 전환 (Mode Switch)**: 사용자 모드에서 커널 모드로 전환됩니다. 이는 하드웨어적인 권한 변경을 포함하며 오버헤드를 발생시킵니다.
3. **스케줄링 (Scheduling)**: OS 스케줄러가 우선순위, 타임 슬라이스 등을 고려하여 다음에 실행할 대상을 결정합니다.
4. **메모리 맵 교체 (MMU Flush - 프로세스 전용)**: 
   - **프로세스 전환 시**: 새로운 프로세스의 페이지 테이블 주소로 CR3 레지스터를 변경합니다. 이때 CPU의 주소 변환 캐시인 **TLB(Translation Lookaside Buffer)가 모두 지워져(Flush)** 이후 한동안 심각한 성능 저하가 발생합니다.
   - **스레드 전환 시**: 동일한 프로세스 내 스레드라면 메모리 맵이 같으므로 **TLB Flush가 필요 없습니다.** 이것이 스레드 전환이 프로세스 전환보다 압도적으로 빠른 핵심 이유입니다.
5. **상태 복원 (Restore Context)**: 새 대상의 TCB/PCB 정보를 CPU 레지스터에 로드하고 사용자 모드로 돌아가 실행을 시작합니다.

### 4. 핵심 알고리즘 및 실무 코드 예시

프로세스 생성(fork)과 스레드 생성(pthread)의 차이를 보여주는 C 언어 기반 실무 예시입니다.

```c
#include <stdio.h>
#include <unistd.h>
#include <pthread.h>
#include <sys/types.h>

int shared_value = 0; // 전역 변수 (스레드는 공유, 프로세스는 복사됨)

void* thread_function(void* arg) {
    shared_value += 10;
    printf("스레드 내부: shared_value = %d\n", shared_value);
    return NULL;
}

int main() {
    pid_t pid;
    pthread_t tid;

    // [1] 프로세스 생성 시도 (fork)
    pid = fork();
    if (pid == 0) { // 자식 프로세스
        shared_value += 5;
        printf("자식 프로세스: shared_value = %d (주소: %p)\n", shared_value, &shared_value);
        return 0;
    } else if (pid > 0) {
        wait(NULL); // 자식 종료 대기
        printf("부모 프로세스: shared_value = %d (주소: %p) - 자식의 영향 없음\n", shared_value, &shared_value);
    }

    // [2] 스레드 생성 시도 (pthread)
    printf("\n--- 스레드 테스트 시작 ---\n");
    pthread_create(&tid, NULL, thread_function, NULL);
    pthread_join(tid, NULL); // 스레드 종료 대기
    printf("메인 스레드: shared_value = %d - 스레드 작업이 반영됨\n", shared_value);

    return 0;
}

/*
[수학적 복잡도 분석 - 암달의 법칙 (Amdahl's Law)]
병렬화를 통한 성능 향상의 한계는 프로그램 중 '병렬화가 불가능한 순차 부분(S)'에 의해 결정됨.
Speedup = 1 / (S + (1-S)/N)  (N: 프로세서 개수)
결론: 아무리 스레드를 수만 개 늘려도, 순차적으로 실행되어야 하는 코드(락 경합 등)가 10%만 있어도 
전체 성능 향상은 최대 10배를 넘지 못함 (동기화 최소화의 중요성).
*/
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 심층 기술 비교표: 프로세스 (Process) vs 스레드 (Thread)

| 평가 지표 (Metrics) | 프로세스 (Process) | 스레드 (Thread) |
| :--- | :--- | :--- |
| **자원 할당 (Resource)** | 독립적 (Code, Data, Stack, Heap 별도) | 공유적 (Stack만 별도, 나머지 공유) |
| **생성/종료 비용** | **매우 높음** (메모리 할당, 복사 필요) | **낮음** (프로세스 자원 재사용) |
| **컨텍스트 스위칭 비용** | **높음** (TLB Flush 발생, 캐시 미스 심함) | **낮음** (메모리 맵 유지, 레지스터만 교체) |
| **통신 방식 (Comm)** | IPC 필요 (Pipe, Socket, Shared Memory) | 공유 변수(Data/Heap)를 통해 직접 통신 |
| **안정성 (Reliability)** | **매우 높음** (하나가 죽어도 다른 곳 영향 없음) | **낮음** (한 스레드 오류 시 프로세스 전체 종료) |
| **프로그래밍 난이도** | 상대적으로 쉬움 (동기화 문제 적음) | **매우 어려움** (데드락, 레이스 컨디션 관리) |

### 2. 스레딩 모델 비교 (Threading Models)

| 모델명 | 특징 | 장점 | 단점 |
| :--- | :--- | :--- | :--- |
| **User-level Thread (N:1)** | 커널 도움 없이 라이브러리가 스레드 관리 | 생성/전환 속도가 매우 빠름 | 한 스레드가 블로킹(I/O) 되면 전체가 멈춤 |
| **Kernel-level Thread (1:1)** | 커널이 직접 스레드 하나하나를 스케줄링 | 진정한 병렬 처리 가능, 멀티코어 최적화 | 시스템 콜 오버헤드로 생성 비용 큼 |
| **M:N Model (Hybrid)** | 사용자 스레드 M개를 커널 스레드 N개에 맵핑 | 유연하고 효율적임 (Go 언어 등 채택) | 구현 아키텍처가 매우 복잡함 |

### 3. 과목 융합 관점 분석 (프로세스/스레드 + 타 도메인 시너지)
- **프로세스/스레드 + 가상 메모리 (Address Space Isolation)**: 프로세스의 독립성은 가상 메모리의 **페이지 테이블 격리**를 통해 실현됩니다. 각 프로세스는 자신만의 페이지 테이블을 가지므로, 논리 주소가 같아도 물리 주소가 다르게 맵핑되어 서로 침범할 수 없습니다. 반면 스레드는 동일한 페이지 테이블을 공유하여 주소 공간을 통합함으로써 빠른 데이터 교환을 가능하게 합니다.
- **프로세스/스레드 + 네트워크 (Concurrent Server)**: 웹 서버 아키텍처에서 이들의 융합이 빛을 발합니다. **Apache**는 주로 멀티 프로세스 모델(Prefork)을 사용하여 보안성과 안정성을 중시하는 반면, **Nginx**는 이벤트 기반(Event-driven) 및 멀티 스레딩 아키텍처를 사용하여 적은 메모리로 수만 개의 동시 접속을 처리하는 고성능 비동기 처리를 구현합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

현업 시스템 설계 시 어떤 실행 단위를 선택할 것인가에 대한 기술사적 가이드라인입니다.

### 1. 실무 시나리오 기반 의사결정 전략
- **[상황 A] 고성능 이미지 렌더링 또는 암호 해독 작업 (CPU-bound)**
  - **문제점**: 수천 장의 고화질 이미지를 렌더링해야 함. 작업 간 데이터 공유는 적지만 연산량이 극도로 많음.
  - **기술사 판단 (전략)**: **멀티 프로세스(Multi-processing) 모델** 권장. 스레드는 전역 인터프리터 락(예: Python의 GIL) 등의 제약으로 멀티코어를 100% 활용하지 못하는 경우가 많음. 각각의 독립된 프로세스로 분산하여 여러 CPU 코어에 할당함으로써 진정한 병렬 연산을 수행하고, 한 작업의 크래시가 다른 작업에 영향을 주지 않도록 격리함.
- **[상황 B] 수만 명의 채팅 접속을 처리하는 메시징 서버 (I/O-bound)**
  - **문제점**: 대다수의 접속이 메시지를 기다리는 대기 상태(Waiting)임. 프로세스를 만 개 생성하면 메모리 부족으로 시스템이 뻗어버림.
  - **기술사 판단 (전략)**: **멀티 스레드 + 비동기 논블로킹 I/O (Event Loop)** 모델 권장. 스레드 풀(Thread Pool)을 생성하여 적정 수의 스레드만 유지하고, 실제 I/O가 발생할 때만 이벤트를 처리함. 이를 통해 컨텍스트 스위칭 횟수를 획기적으로 줄이고 메모리 사용량을 최소화함.

### 2. 도입 시 고려사항 (기술적/동기화 체크리스트)
- **공유 자원 보호 (Synchronization)**: 스레드를 사용할 때는 반드시 **뮤텍스(Mutex), 세마포어(Semaphore), 스핀락(Spinlock)** 등을 사용하여 임계 구역(Critical Section)을 보호해야 합니다. 하지만 과도한 락(Lock) 사용은 '데드락(Deadlock)'과 '우선순위 역전' 문제를 일으키므로, **Lock-free 알고리즘**이나 원자적 연산(Atomic Operation) 사용을 검토해야 합니다.
- **좀비 프로세스 및 고립 방지**: 프로세스 생성 시 부모 프로세스가 자식의 종료 상태를 회수(wait)하지 않으면 '좀비 프로세스'가 되어 시스템 리소스를 낭비합니다. 시그널 핸들러를 통해 자원 정리를 자동화해야 합니다.

### 3. 주의사항 및 안티패턴 (Anti-patterns)
- **무분별한 스레드 생성 (Thread Explosion)**: "스레드는 가벼우니까 많이 만들어도 되겠지?"라는 생각으로 요청당 스레드 하나를 무한히 생성하는 패턴. 스레드 개수가 CPU 코어 수의 수십 배를 넘어가면, 정작 작업하는 시간보다 **스레드 간 컨텍스트 스위칭을 위해 CPU가 낭비되는 시간**이 더 길어지는 성능 역전 현상이 발생합니다. 반드시 **Thread Pool**을 사용하여 개수를 제한해야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량적/정성적 기대효과 (도입 전후 ROI)

| 구분 | 단일 프로세스 환경 (AS-IS) | 멀티 프로세스/스레드 환경 (TO-BE) | 개선 지표 (Impact) |
| :--- | :--- | :--- | :--- |
| **자원 활용률 (CPU Utilization)** | 1개 코어만 사용, 나머지 코어 유휴 | 모든 코어에 작업 분산 배치 | **멀티코어 활용률 400~800% 향상** |
| **시스템 응답성 (Responsiveness)** | 긴 작업 시 UI가 멈춤 (Freezing) | 백그라운드 스레드 처리를 통해 UI 반응 유지 | **사용자 체감 지연 시간 90% 감소** |
| **자원 생성 오버헤드** | 요청마다 무거운 프로세스 생성 | 스레드 풀 및 경량 실행 단위 사용 | **서버 처리량(TPS) 5배 이상 증가** |

### 2. 미래 전망 및 진화 방향
- **초경량 스레드 (User-mode Scheduler)**: Go 언어의 고루틴(Goroutine)이나 Java의 프로젝트 룸(Loom)처럼, 수십 바이트의 스택만 사용하는 수백만 개의 초경량 스레드를 커널 개입 없이 관리하는 기술이 표준이 되고 있습니다.
- **서버리스(Serverless)와 프로세스 격리**: AWS Lambda와 같은 서버리스 환경은 극도로 짧은 시간 동안만 프로세스를 실행하고 즉시 파괴하는 'Ephemeral Process' 시대를 열었습니다. 이를 위해 Firecracker와 같은 마이크로 VM 기술을 통해 프로세스 급의 격리성과 스레드 급의 시작 속도를 동시에 잡으려 하고 있습니다.

### 3. 참고 표준/가이드
- **POSIX Threads (Pthreads)**: 유닉스 계열 운영체제에서 스레드를 생성하고 관리하기 위한 표준 API 규격.
- **OpenMP**: C, C++, Fortran에서 멀티플랫폼 공유 메모리 병렬 프로그래밍을 위한 API 표준.
- **Windows API (Win32)**: 윈도우 환경에서의 프로세스/스레드 생성 및 동기화 오브젝트 표준.

## 👶 어린이를 위한 3줄 비유 설명
1. **프로세스**는 여러분이 가지고 노는 각각의 **'게임기'**와 같아요. 닌텐도를 하다가 플리스테이션을 켠다고 해서 닌텐도 게임이 바뀌지는 않죠. 서로 완전히 따로 노니까요!
2. **스레드**는 하나의 게임기 안에서 움직이는 **'여러 명의 캐릭터'**와 같아요. 주인공이 점프할 때 옆의 강아지도 같이 뛰어갈 수 있는 건, 한 게임기 안에서 정보를 같이 나누어 쓰기 때문이에요.
3. 캐릭터가 너무 많아지면 게임기가 느려질 수 있지만, 캐릭터끼리 서로 도와가며 보스를 물리치는 게 훨씬 빠르고 재미있는 것처럼, 컴퓨터도 스레드 덕분에 여러 일을 동시에 잘할 수 있답니다!
