# DNS 전체 구조와 권한 관리의 이해

도메인을 운영하다 보면 "네임서버(NS) 변경"과 "DNS 레코드 수정" 사이에서 혼란을 겪곤 합니다. 특히 "권한 네임서버가 아니다"라는 에러를 마주했을 때의 당혹감을 해결하기 위해, DNS의 전체 흐름과 권한 구조를 실무 관점에서 정리합니다.

---

### 1. DNS 전체 구조 (큰 그림)

사용자가 브라우저에 도메인을 입력했을 때, IP 주소를 찾아가는 과정은 계층적인 탐색 과정입니다.

```text
[사용자]
   │
   ▼
[리졸버 (Resolver)] ---------------- (ISP 혹은 Google 8.8.8.8 등)
   │
   ▼
[루트 네임서버 (Root NS)] ----------- (.)
   │
   ▼
[TLD 네임서버] ---------------------- (.com, .kr, .org 등)
   │
   ▼
[권한 있는 네임서버 (Authoritative NS)] -- (Cloudflare, Gabia, AWS Route53 등)
   │
   ▼
[DNS 레코드 (A, MX, TXT)] ----------- (실제 IP 혹은 설정값)
   │
   ▼
IP 응답 (최종 목적지)
```

**핵심 인사이트:**
> "정답(레코드)"은 항상 **권한 있는 네임서버**가 가지고 있습니다. 리졸버는 단지 심부름꾼일 뿐입니다.

---

### 2. 관리 관점: 누가 무엇을 바꾸는가?

DNS 설정은 크게 두 곳에서 이루어지며, 역할이 완전히 분리되어 있습니다.

1.  **도메인 등록기관 (Registrar)**: 가비아, 후이즈, Namecheap 등
    *   **역할**: 도메인의 소유권을 관리하고, 해당 도메인이 사용할 **네임서버(NS)를 지정**합니다.
2.  **권한 있는 네임서버 (Authoritative NS)**: 클라우드플레어, AWS Route53 등
    *   **역할**: 지정된 네임서버 환경 내에서 실제 **DNS 레코드(A, CNAME, MX 등)를 관리**합니다.

---

### 3. 두 개념의 결합

실제 도메인 연결 과정은 다음과 같이 작동합니다.

1.  **Registrar (가비아 등)**
    *   설정: "example.com은 이제 `ns1.cloudflare.com`을 사용한다" (위임, Delegation)
2.  **Authoritative NS (Cloudflare)**
    *   설정: "A 레코드는 `1.2.3.4`다", "MX 레코드는 `mail.example.com`이다"

---

### 4. 실무에서 헷갈리는 포인트 (Troubleshooting)

#### (A) 레코드 수정 시도 시 발생하는 오류
현재 도메인의 NS가 `ns1.gabia.co.kr`로 설정되어 있다면:
*   **Cloudflare**에서 레코드를 수정하면? ❌ **(적용 안 됨 / 권한 없음 에러)**
*   **Gabia**에서 레코드를 수정하면? ✅ **(정상 적용)**

**이유:** 전 세계 인터넷 시스템이 가비아를 '정답지'로 보고 있기 때문입니다.

#### (B) 네임서버 변경
등록기관(Registrar) 관리 페이지에서 네임서버 주소를 `ns1.gabia.co.kr` → `ns1.cloudflare.com`으로 변경하는 작업은:
*   단순한 **포인터 변경**이므로 클릭 한 번(딸깍)으로 가능합니다.
*   레코드를 하나하나 옮기는 것이 아니라, **"이제부터 정답은 클라우드플레어에 물어봐라"**라고 권한 자체를 넘기는 행위입니다.

---

### 5. 왜 네임서버 변경은 쉬운데 적용은 시간이 걸리나?

네임서버 변경 자체는 단순한 위임(Delegation) 변경이지만, 전 세계에 반영되는 데는 시간이 필요합니다.

*   **캐시 (Cache)**: 각 단계의 리졸버들이 이전 정답(기존 NS)을 기억하고 있습니다.
*   **TTL (Time To Live)**: 기존 설정이 유효하다고 판단하는 유효 기간이 남아있기 때문입니다.
*   **전파 시간 (Propagation)**: 전 세계 수많은 DNS 서버가 새 포인터를 인지하는 데 보통 수 시간에서 최대 48시간이 소요됩니다.

---

### 6. 가장 중요한 실무 포인트 요약

1.  **네임서버(NS) 변경**은 **도메인 등록기관(Registrar)** 관리 페이지에서 한다.
2.  **실제 레코드(A, MX 등) 수정**은 지정된 **권한 있는 네임서버(Authoritative NS)** 관리 페이지에서 한다.
3.  **"권한 네임서버 아님"** 에러가 뜬다면, 내가 지금 수정하고 있는 곳이 실제 도메인에 설정된 NS 주소와 일치하는지 확인하라.

---

### 한 장 요약

| 구분 | 주체 | 작업 내용 |
| :--- | :--- | :--- |
| **위임 (Delegation)** | Registrar (도메인 등록처) | 사용할 네임서버(NS) 주소 입력 |
| **관리 (Management)** | Authoritative NS (네임서버 업체) | IP 주소 연결 및 메일 서버 설정 |

이 구조를 이해하면 DNS 설정 시 발생하는 문제의 90% 이상을 스스로 진단하고 해결할 수 있습니다.
