# LLM 에이전트 아키텍처 & 자율성 강화 전략

#### 핵심 인사이트 (3줄 요약)

> 1. **본질**: LLM 단독 호출은 "스마트한 자동완성"에 불과하다. 에이전트는 LLM에 **지각(Perception) → 추론(Reasoning) → 행동(Action) → 반성(Reflection)** 루프를 부여해 자율적 문제 해결 능력을 만드는 아키텍처다.
> 2. **가치**: 단일 LLM 대비 복잡한 작업 완료율 3~10배 상승, 코드 작성→테스트→디버그→배포까지 무인 파이프라인 구현 가능. Claude Code가 "무슨 일을 시켜도 잘하는" 이유가 바로 이 루프다.
> 3. **융합**: 멀티에이전트 + MCP(도구) + 구조화 메모리 + 오케스트레이션 레이어가 결합할 때 진정한 자율 시스템이 완성된다.

---

## Ⅰ. 개요 — 왜 "에이전트"인가?

### 단일 LLM의 한계

프롬프트를 넣고 응답을 받는 단순 호출 방식은 근본적인 제약이 있다. LLM은 **컨텍스트 윈도우 안에서만** 사고하고, 외부 세계와 상호작용하거나 실행 결과를 자신에게 피드백하는 구조가 없다. 한 번의 대화로 "파이썬 코드 짜줘"는 가능하지만, "이 레포에 PR 만들고, 테스트 통과시키고, 배포까지 해줘"는 단일 호출로 불가능하다.

**💡 비유**: LLM 단독 사용은 똑똑한 직원에게 메모 한 장 던져주고 혼자 알아서 하라는 것과 같다. 에이전트 시스템은 그 직원에게 컴퓨터, 인터넷, 동료, 작업 기록장, 그리고 "실패하면 다시 시도하라"는 권한까지 주는 것이다.

**등장 배경:**

```
① 기존 한계
   - LLM은 stateless: 매 호출이 독립적
   - 외부 도구 사용 불가 (검색, 코드실행, DB 조회)
   - 복잡한 멀티스텝 작업 불가

② 혁신적 패러다임 (2023~)
   - ReAct 논문: "생각하고 → 행동하고 → 관찰한다"
   - Function Calling: OpenAI가 LLM에 도구 호출 규격 추가
   - MCP: Anthropic이 도구 연결을 표준화

③ 현재의 요구
   - "ChatGPT로 코드 짜기" → "AI가 스스로 개발"
   - 소프트웨어 엔지니어 업무 50~80% 자동화 목표
```

## Ⅱ. 에이전트의 두뇌 — 핵심 루프와 패턴

### 2.1 ReAct 패턴 (Reasoning + Acting)

에이전트 동작의 근간이 되는 패턴이다. 단순히 한 번 추론하고 끝내는 것이 아니라, **생각(Thought) → 행동(Action) → 관찰(Observation)** 을 반복하며 목표에 수렴한다. 2022년 Google 논문에서 제안됐으며, 현재 모든 주요 에이전트 프레임워크의 기반 철학이다.

```
      사용자 목표 입력
           │
           ▼
    ┌──────────────┐
    │   THOUGHT    │  ← LLM이 "무엇을 해야 하는가?" 추론
    │  (추론 단계) │
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │    ACTION    │  ← 도구 호출, 코드 실행, 검색 등
    │  (행동 단계) │
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │ OBSERVATION  │  ← 행동 결과를 컨텍스트에 추가
    │  (관찰 단계) │
    └──────┬───────┘
           │
     목표 달성?
      ├─ NO  → THOUGHT로 돌아감 (루프)
      └─ YES → 최종 답변 생성
```

이 루프가 핵심이다. LLM은 단순히 "답을 한 번에 맞추는" 것이 아니라 **행동 → 피드백 → 재추론**을 반복하며 복잡한 문제를 풀어낸다. Claude Code가 "에러가 뜨면 스스로 고치는" 이유가 바로 Observation 단계에서 에러 메시지를 읽고 다시 THOUGHT 단계로 진입하기 때문이다.

---

### 2.2 Plan & Execute 패턴

ReAct가 "한 걸음씩 즉흥적으로 나아가는" 방식이라면, Plan & Execute는 **먼저 전체 계획을 세우고, 그 계획대로 실행**하는 방식이다. 복잡하고 예측 가능한 워크플로우에 적합하다.

```
  ┌─────────────────────────────────────────────┐
  │               PLANNER (LLM)                 │
  │  "전체 계획을 구조화된 단계로 분해"          │
  └──────────────────┬──────────────────────────┘
                     │
          ┌──────────▼──────────┐
          │   Task Queue        │
          │  [Step1, Step2, ... │
          │   Step3, StepN]     │
          └──────────┬──────────┘
                     │
       ┌─────────────▼──────────────┐
       │         EXECUTOR           │
       │  Step 1: 파일 분석          │──→ Tool 호출
       │  Step 2: 코드 생성          │──→ Tool 호출
       │  Step 3: 테스트 실행        │──→ Tool 호출
       │  Step N: 결과 정리          │──→ Tool 호출
       └─────────────┬──────────────┘
                     │
       ┌─────────────▼──────────────┐
       │         REPLANNER          │
       │  실행 결과를 보고           │
       │  계획 수정 필요시 재계획    │
       └────────────────────────────┘
```

**ReAct vs Plan & Execute 선택 기준:**

| 상황 | 추천 패턴 |
|:---|:---|
| 목표가 명확하고 단계가 예측 가능 | Plan & Execute |
| 탐색적이고 불확실한 작업 | ReAct |
| 긴 작업 (10단계 이상) | Plan & Execute |
| 짧고 즉흥적인 작업 | ReAct |
| 단계 실패 시 전략 수정 필요 | 두 패턴 조합 |

---

### 2.3 Reflection & Self-Critique 패턴

자신이 만든 결과물에 대해 **스스로 비평하고 개선**하는 루프다. 이 패턴만 추가해도 코드 품질이나 글의 완성도가 눈에 띄게 올라간다.

```
           결과물 생성
                │
                ▼
    ┌───────────────────────┐
    │   CRITIC (비평 LLM)   │
    │  - 논리적 허점 찾기   │
    │  - 엣지 케이스 지적   │
    │  - 개선점 목록화      │
    └───────────┬───────────┘
                │
         개선점 있음?
          ├─ YES → Generator에게 피드백 전달 → 재작성
          └─ NO  → 최종 결과물 확정

    [실제 구현 예시]
    Generator ──→ Critic ──→ Generator ──→ Critic
                                               │
                                           품질 기준 통과
                                               │
                                           최종 출력
```

## Ⅲ. 멀티에이전트 아키텍처 — 더 강한 자율성

### 3.1 왜 에이전트를 여러 개 쓰는가?

단일 에이전트가 만능처럼 보여도, 컨텍스트 윈도우 한계와 전문화 문제가 있다. 복잡한 작업(예: "새로운 웹서비스 기획 → 설계 → 개발 → 테스트 → 배포")은 한 에이전트가 처리하기에 너무 길고 넓다. 멀티에이전트는 **각 에이전트에게 전문 역할을 부여**하고, 병렬·협력 처리를 통해 복잡성을 정복한다.

```
┌─────────────────── 멀티에이전트 핵심 이점 ───────────────────┐
│                                                               │
│  단일 에이전트                  멀티에이전트                  │
│  ═══════════════                ══════════════                │
│  컨텍스트: 200K 토큰 한계       분산: 각 에이전트 독립 컨텍스트│
│  전문성: 제너럴리스트           전문화: 역할별 특화 시스템 프롬프트│
│  처리: 순차                     처리: 병렬 가능               │
│  장애: 단일 실패점              장애: 부분 실패 허용           │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

---

### 3.2 오케스트레이터-워커 패턴 (가장 많이 씀)

가장 널리 사용되는 멀티에이전트 아키텍처다. **오케스트레이터(지휘자)** 가 전체 목표를 파악하고 작업을 분해해 **워커(실무자) 에이전트**에게 위임한다. Claude Code의 내부 구조가 이와 유사하다.

```
                     사용자 요청
                          │
                          ▼
           ┌──────────────────────────┐
           │     ORCHESTRATOR         │
           │  (오케스트레이터 LLM)    │
           │  - 목표 분석             │
           │  - 작업 분해 (Task Dec.) │
           │  - 워커 선택 & 위임      │
           │  - 결과 통합             │
           └────┬──────┬──────┬───────┘
                │      │      │
         ┌──────▼─┐ ┌──▼───┐ ┌▼───────┐
         │Worker A│ │WorkerB│ │Worker C│
         │코드작성│ │테스트 │ │문서작성│
         │전문    │ │전문   │ │전문    │
         └──────┬─┘ └──┬───┘ └┬───────┘
                │      │      │
                └──────┴──────┘
                       │
                 결과 → Orchestrator 통합
                       │
                  최종 결과물
```

오케스트레이터는 워커에게 **좁고 명확한 작업만** 전달하고, 워커는 해당 작업에만 집중한다. 이렇게 하면 각 워커의 컨텍스트가 오염되지 않고, 실패한 워커만 재실행할 수 있다.

---

### 3.3 피어-투-피어 패턴 (P2P)

오케스트레이터 없이 에이전트들이 **동등하게 서로 소통**하는 구조다. 창의적 작업, 토론, 상호 검증에 적합하다.

```
    ┌──────────┐     메시지 전달      ┌──────────┐
    │ Agent A  │◄────────────────────►│ Agent B  │
    │ (제안자) │                      │ (비평가) │
    └────┬─────┘                      └────┬─────┘
         │                                 │
         │         ┌──────────┐            │
         └────────►│ Agent C  │◄───────────┘
                   │ (중재자) │
                   └──────────┘

    [예시 워크플로우]
    Agent A: "이 기능을 이렇게 구현하자" (제안)
    Agent B: "성능 문제가 있어, 이렇게 수정해야 해" (비평)
    Agent C: "둘 다 맞는 말인데, 절충안은 이거야" (중재)
    → 최종 합의된 구현 방안 도출
```

---

### 3.4 계층형 패턴 (Hierarchical)

현실의 조직 구조처럼 **여러 단계의 오케스트레이터**가 있는 구조다. 대규모 복잡 시스템에 적합하다.

```
                    CEO Agent (최상위 목표)
                          │
           ┌──────────────┼──────────────┐
           │              │              │
      PM Agent       DevLead Agent   QA Agent
      (기획)          (개발 총괄)    (품질 관리)
           │              │              │
      ┌────┴───┐    ┌─────┼────┐    ┌───┴───┐
   UX Agent  │  Backend  Frontend  Test   Report
   Market    │  Agent    Agent    Agent   Agent
   Agent     │
```

## Ⅳ. MCP — 도구를 표준화하는 프로토콜

### 4.1 MCP란 무엇인가?

MCP(Model Context Protocol)는 Anthropic이 2024년 공개한 **LLM과 외부 도구 사이의 표준 통신 프로토콜**이다. USB-C가 다양한 기기를 하나의 규격으로 연결하듯, MCP는 "어떤 AI든, 어떤 도구든" 같은 방식으로 연결한다. 기존에는 각 에이전트 프레임워크마다 도구 연결 방식이 달랐는데, MCP가 이를 통일했다.

```
   [MCP 이전]                      [MCP 이후]
   ═══════════                     ══════════

   LangChain ──→ 커스텀 코드        모든 Client
   AutoGen   ──→ 커스텀 코드   ──→  (Claude, GPT, 오픈소스)
   Claude    ──→ 커스텀 코드                │
   GPT       ──→ 커스텀 코드          MCP Protocol
   (각자 다름, 재사용 불가)                 │
                                    모든 Server
                                    (파일시스템, DB,
                                     GitHub, 검색, ...)
```

### 4.2 MCP 아키텍처 상세

```
┌─────────────────────────────────────────────────────────┐
│                    MCP 생태계                            │
│                                                         │
│  ┌─────────────────────┐                                │
│  │   MCP HOST (Claude) │                                │
│  │   ┌─────────────┐   │                                │
│  │   │ MCP Client  │   │                                │
│  │   └──────┬──────┘   │                                │
│  └──────────┼──────────┘                                │
│             │ JSON-RPC 2.0 (stdio / HTTP)               │
│    ┌────────┴───────────────────────────┐               │
│    │           MCP SERVERS              │               │
│    │                                    │               │
│    │  ┌──────────┐  ┌──────────┐       │               │
│    │  │ File     │  │ GitHub   │       │               │
│    │  │ System   │  │ Server   │       │               │
│    │  │ Server   │  │          │       │               │
│    │  └──────────┘  └──────────┘       │               │
│    │                                    │               │
│    │  ┌──────────┐  ┌──────────┐       │               │
│    │  │ Postgres │  │ Puppeteer│       │               │
│    │  │ Server   │  │ (Browser)│       │               │
│    │  └──────────┘  └──────────┘       │               │
│    └────────────────────────────────────┘               │
└─────────────────────────────────────────────────────────┘

MCP Server가 제공하는 3가지:
  ① Tools     : 실행 가능한 함수 (read_file, run_query...)
  ② Resources : 읽기 전용 데이터 (파일 내용, DB 레코드...)
  ③ Prompts   : 재사용 가능한 프롬프트 템플릿
```

### 4.3 실제로 도입할 수 있는 MCP 서버 목록

| MCP 서버 | 역할 | 사용 예 |
|:---|:---|:---|
| `filesystem` | 파일 읽기/쓰기 | 코드 파일 직접 수정 |
| `github` | GitHub API 연결 | PR 생성, 이슈 관리 |
| `postgres` / `sqlite` | DB 직접 쿼리 | 데이터 분석, 스키마 확인 |
| `puppeteer` / `playwright` | 브라우저 제어 | 웹 스크래핑, E2E 테스트 |
| `brave-search` | 실시간 웹 검색 | 최신 정보 조회 |
| `slack` | 슬랙 연동 | 자동 알림, 메시지 전송 |
| `memory` | 장기 기억 저장 | 대화 기억 유지 |
| `docker` | 컨테이너 실행 | 샌드박스 코드 실행 |

## Ⅴ. 메모리 시스템 — 에이전트가 "기억"하는 법

### 5.1 4가지 메모리 타입

단기 컨텍스트만 가지는 LLM에 다양한 메모리 레이어를 추가하면 사람처럼 "기억하고 학습"하는 에이전트를 만들 수 있다.

```
┌─────────────────────────────────────────────────────────────┐
│                    에이전트 메모리 계층                      │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  ① In-Context Memory (단기)                         │   │
│  │     현재 대화 내용, 최근 행동 결과                  │   │
│  │     용량: ~200K 토큰 / 지속: 세션 동안만            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  ② External Memory (장기 - 벡터 DB)                 │   │
│  │     과거 대화, 문서, 지식 → 임베딩 → 벡터 검색      │   │
│  │     도구: Chroma, Pinecone, pgvector                │   │
│  │     용량: 무제한 / 지속: 영구                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  ③ Episodic Memory (에피소드)                        │   │
│  │     "저번에 이 방식으로 했더니 실패했다"             │   │
│  │     성공/실패 경험 → 구조화 저장 → 유사 상황 참조   │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  ④ Procedural Memory (절차)                          │   │
│  │     시스템 프롬프트 & 파인튜닝에 내재화된 노하우    │   │
│  │     "이 에이전트는 항상 이렇게 생각한다"            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 RAG — 외부 지식을 주입하는 법

```
  사용자 질문
       │
       ▼
  [Query Encoder] → 질문을 벡터화
       │
       ▼
  [Vector DB 검색] → 유사 문서 K개 추출
       │
  관련 문서 조각
       │
       ▼
  ┌────────────┐     ┌──────────────────────────┐
  │  Context   │ +   │  원본 질문               │  →  LLM  →  답변
  │  (검색된   │     │                          │
  │   문서들)  │     └──────────────────────────┘
  └────────────┘

  핵심: LLM을 재학습시키지 않고 지식을 확장한다.
  비용: 파인튜닝 없이 최신 정보 주입 가능
```

## Ⅶ. 자율성 강화 전략 — 단계적 로드맵

### 7.1 자율성 스펙트럼

자율성은 한 번에 완성되는 것이 아니다. 아래 스펙트럼에서 자신의 현재 위치를 파악하고 단계적으로 올라가야 한다.

```
레벨 0: 단순 호출
  User ──입력──▶ LLM ──출력──▶ User
  (자율성 없음, 순수 Q&A)

레벨 1: 도구 사용 (Tool Use)
  User ──▶ LLM ──▶ [도구 호출] ──▶ LLM ──▶ User
  (검색, 계산, API 호출 가능)

레벨 2: 단일 에이전트 루프 (ReAct)
  User ──▶ [Think→Act→Observe 반복] ──▶ User
  (멀티스텝 작업 자동화)

레벨 3: 멀티에이전트 협업
  User ──▶ Orchestrator ──▶ [Worker 팀] ──▶ User
  (복잡한 프로젝트 자동화)

레벨 4: 자율 에이전트 시스템
  User ──▶ [자율적 판단, 계획, 실행, 학습] ──▶ User
  (인간 최소 개입)
  ← 현재 Claude Code가 여기 근접
```

---

### 7.2 실전 구현 로드맵

당신의 현재 수준(에이전트 + MCP 기초 인지)에서 출발해 자율 시스템을 구축하는 단계별 계획이다.

```
Phase 1 (1~2주): 기반 구축
│
├─ Ollama 설치 → 로컬 LLM 실행 (Qwen2.5 or Llama 3.3)
├─ 기본 ReAct 에이전트 직접 구현 (프레임워크 없이)
│    └─ 이유: 내부 동작 이해 없이 프레임워크 쓰면 블랙박스
└─ MCP 서버 1~2개 연결 (filesystem, brave-search)

Phase 2 (3~4주): 에이전트 심화
│
├─ CrewAI or LangGraph로 멀티에이전트 구현
│    └─ Orchestrator + Worker 2명 구조
├─ 메모리 추가: ChromaDB로 벡터 메모리 붙이기
└─ Reflection 패턴: Critic 에이전트 추가

Phase 3 (5~8주): 자율 시스템
│
├─ 자신의 워크플로우를 에이전트로 자동화
│    예: "레포 분석 → 문서 작성 → PR 생성" 파이프라인
├─ 오케스트레이터 + 전문 워커 팀 구성
└─ Human-in-the-loop: 중요 결정만 사람이 승인
```

---

### 7.3 Human-in-the-Loop — 완전 자율화의 함정을 피하는 법

완전 자율화는 매력적이지만, 현실에서는 **인간이 개입하는 체크포인트**가 필수다. 중요한 결정(파일 삭제, 배포, 비용 발생)에는 반드시 승인을 받아야 한다.

```
자율 실행 가능                    반드시 승인 필요
════════════════                  ════════════════
파일 읽기                         파일 삭제
웹 검색                           외부 API 호출 (비용)
분석 리포트 생성                  코드 배포
초안 작성                         DB 데이터 수정
단위 테스트 실행                  자금 관련 작업

[승인 게이트 패턴]
에이전트 실행
     │
     ├─ 안전한 작업 → 자동 실행
     │
     └─ 위험한 작업 ──▶ 인간에게 승인 요청
                              │
                     ├─ 승인 → 실행
                     └─ 거부 → 중단 or 대안 탐색
```

## 👶 어린이를 위한 비유

**에이전트란?** 레고처럼 똑똑한 로봇에게 손(MCP 도구), 메모장(메모리), 동료 로봇(멀티에이전트)을 달아주면 혼자서 레고를 설계하고 조립하고 검사까지 할 수 있어요.

**왜 여러 에이전트가 필요한가?** 혼자서 요리, 청소, 숙제, 그림 그리기를 동시에 하면 엉망이 되죠. 요리 로봇, 청소 로봇, 공부 로봇이 따로 있으면 훨씬 잘해요. 오케스트레이터는 엄마처럼 "너는 이거, 너는 저거" 시키는 역할이에요.

**MCP는?** 모든 로봇이 다 다른 플러그를 쓰면 불편하죠. MCP는 USB-C처럼 어떤 로봇도 어떤 도구에나 꽂을 수 있게 만들어주는 표준 플러그예요.
