# 빅데이터 3V / 5V (Big Data 3V/5V)

## 정의

**빅데이터(Big Data)**는 전통적인 데이터 처리 소프트웨어의 능력을 넘어서는 규모(Volume), 속도(Velocity), 다양성(Variety)을 갖춘 대용량 데이터 집합을 의미한다. Doug Laney가 2001년 META 그룹(현 Gartner)에서 처음 제시한 3V 모델은 빅데이터의 핵심 특성을 정의하는 산업 표준 프레임워크로 자리 잡았으며, 이후 IBM 등에서 진실성(Veracity)과 가치(Value)를 추가한 5V 모델로 확장되었다.

```
빅데이터 정의 (Gartner):
"대용량, 고속, 다양한 정보 자산으로, 비용 효율적이고 혁신적인
정보 처리 형태를 필요로 하며, 통찰력 향상과 의사결정 최적화,
프로세스 자동화를 가능하게 한다."

3V 모델 (Doug Laney, 2001):
- Volume (볼륨): 데이터 규모
- Velocity (속도): 데이터 생성 및 처리 속도
- Variety (다양성): 데이터 유형의 다양성

5V 모델 (확장):
+ Veracity (진실성): 데이터 품질 및 신뢰성
+ Value (가치): 비즈니스 가치 창출
```

데이터 엔지니어링 관점에서 빅데이터 3V/5V는 단순한 데이터 크기의 문제가 아니라, 데이터 수집(Ingestion), 저장(Storage), 처리(Processing), 분석(Analytics) 전 영역에 걸쳐 아키텍처 설계를 근본적으로 재고해야 하는 패러다임 전환을 의미한다.

---

## 핵심 개념

### 1. Volume (볼륨) - 데이터 규모

```
    [ 데이터 볼륨의 성장 추이 ]

    규모          연도         대표적 데이터 소스
    ─────────────────────────────────────────────────
    Megabyte      1990s        문서 파일, 간단한 DB
    Gigabyte      2000s        웹 로그, 이메일 보관
    Terabyte      2010s        소셜 미디어, IoT 센서
    Petabyte      2015s        동영상 스트리밍, 금융거래
    Exabyte       2020s        전사적 데이터 레이크, AI 학습
    Zettabyte     2025+        글로벌 데이터 헤드론

    [ 규모별 처리 아키텍처 변화 ]

    TB 급 (단일 서버)              PB 급 (분산 클러스터)
    +------------------+          +----------------------------------+
    |   RDBMS          |          |   Hadoop / Spark Cluster         |
    |   (Oracle, MySQL)|          |   (수십~수백 노드)                |
    +------------------+          +----------------------------------+
    |   SAN Storage    |          |   HDFS / S3 Object Storage       |
    +------------------+          +----------------------------------+
           |                                   |
      수직 확장(Scale-up)               수평 확장(Scale-out)
      (하드웨어 업그레이드)              (노드 추가)
```

**볼륨 특성**:
- **규모**: TB(테라바이트) ~ PB(페타바이트) ~ EB(엑사바이트) 단위
- **성장률**: 연평균 40~60% 성장 (IDC 기준)
- **처리 패러다임**: Scale-up (수직 확장) → Scale-out (수평 확장)
- **저장 매체**: SAN/NAS → 분산 파일 시스템(HDFS) → 오브젝트 스토리지(S3)

### 2. Velocity (속도) - 데이터 생성 및 처리 속도

```
    [ 데이터 속도의 스펙트럼 ]

    배치 처리 (Batch)              실시간 처리 (Real-time)
    ────────────────────────────────────────────────────────>
    시간/일/주 단위     분 단위      초 단위     밀리초 단위

    [ 속도별 처리 아키텍처 ]

    배치 (Batch)                   스트리밍 (Streaming)
    +-------------------+         +------------------------+
    |   T+1 (익일)      |         |   T+0 (실시간)         |
    |   Hadoop MapReduce|         |   Kafka + Flink/Spark  |
    |   일일 ETL        |         |   Streaming            |
    +-------------------+         +------------------------+
           |                              |
    대용량 일괄 처리              연속적 흐름 처리
    (지연 시간: 시간~일)          (지연 시간: 초~분)

    [ 실시간 처리 요구사항 예시 ]

    도메인              지연 시간 요구        처리 방식
    ─────────────────────────────────────────────────────
    금융 거래 감시      < 100ms              Complex Event Processing
    추천 시스템         < 500ms              Stream Processing
    IoT 센서 모니터링   < 1s                 Time-series DB
    로그 분석           < 1분                Micro-batch
    일일 리포트         < 24시간             Batch
```

**속도 특성**:
- **생성 속도**: 초당 수백만 건의 이벤트 생성 (예: IoT 센서, 클릭 로그)
- **처리 속도**: 실시간(Real-time) vs 준실시간(Near Real-time) vs 배치(Batch)
- **전송 속도**: 고속 메시지 큐(Kafka), CDC(Change Data Capture)
- **응답 속도**: 대화형 분석(Interactive) vs 백그라운드 처리(Background)

### 3. Variety (다양성) - 데이터 유형의 다양성

```
    [ 데이터 유형 스펙트럼 ]

    정형 (Structured)         반정형 (Semi-structured)    비정형 (Unstructured)
    ─────────────────────────────────────────────────────────────────────>
    엄격한 스키마            내장 메타데이터            스키마 없음

    +-------------------+    +----------------------+    +------------------+
    |   RDBMS 테이블    |    |   JSON / XML / YAML  |    |   이미지/비디오  |
    |   (행/열 구조)    |    |   로그 파일          |    |   오디오/음성    |
    |   CSV             |    |   NoSQL 도큐먼트     |    |   텍스트 문서    |
    +-------------------+    +----------------------+    +------------------+
           |                          |                         |
    스키마 온 라이트           스키마 온 리드              스키마 없음
    (Schema-on-Write)         (Schema-on-Read)           (No Schema)

    [ 데이터 유형별 비중 (추정) ]

    ┌────────────────────────────────────────────────────┐
    │                                                    │
    │   비정형 (80%)   ████████████████████████████      │
    │   반정형 (15%)   ██████                            │
    │   정형 (5%)      ██                                │
    │                                                    │
    └────────────────────────────────────────────────────┘

    출처: IBM, IDC 연구 결과 (기업 데이터의 대부분이 비정형)
```

**다양성 특성**:
- **정형 데이터(Structured)**: RDBMS 테이블, CSV, 고정 길이 레코드 (약 5~20%)
- **반정형 데이터(Semi-structured)**: JSON, XML, YAML, 로그, NoSQL 도큐먼트 (약 15~20%)
- **비정형 데이터(Unstructured)**: 텍스트, 이미지, 비디오, 오디오, PDF (약 60~80%)
- **데이터 소스**: 웹 로그, 소셜 미디어, IoT 센서, 이메일, 문서, 동영상

### 4. Veracity (진실성) - 데이터 품질 및 신뢰성

```
    [ 데이터 품질 문제 유형 ]

    문제 유형             원인                          영향
    ────────────────────────────────────────────────────────────────
    불완전성             결측치, 누락                  분석 왜곡
    부정확성             입력 오류, 센서 오차          잘못된 의사결정
    모순성               중복, 버전 불일치             신뢰성 저하
    시효성               오래된 데이터                  현실 반영 실패
    편향성               샘플링 편향                    통계적 오류

    [ 데이터 품질 관리 프레임워크 ]

    +----------------------------------------------------------+
    |                   Data Quality Pipeline                   |
    +----------------------------------------------------------+
    |                                                          |
    |   수집 단계       검증 단계       정제 단계       모니터링 |
    |   +--------+     +--------+     +--------+     +--------+ |
    |   | 수집   | --> | 품질   | --> | 정제   | --> | 품질   | |
    |   |        |     | 규칙   |     | 로직   |     | 지표   | |
    |   +--------+     +--------+     +--------+     +--------+ |
    |       |              |              |              |      |
    |   [소스]        [검증]         [변환]        [대시보드]  |
    |                                                          |
    +----------------------------------------------------------+

    [ 신뢰성 관리 기법 ]

    기법                      설명
    ────────────────────────────────────────────────────────
    데이터 프로파일링         통계적 특성 분석
    데이터 검증              스키마, 비즈니스 규칙 검사
    이상치 탐지              통계적, ML 기반 탐지
    데이터 계보(Lineage)     출처 및 변환 이력 추적
    마스터 데이터 관리       골든 레코드 구축
```

**진실성 특성**:
- **불확실성(Uncertainty)**: 데이터 출처, 수집 방법의 신뢰성 문제
- **품질 이슈**: 결측치, 중복, 오류, 불일치
- **관리 필요성**: 데이터 품질 관리(DQM), 데이터 거버넌스
- **규제 준수**: 개인정보보호, 데이터 보안, 컴플라이언스

### 5. Value (가치) - 비즈니스 가치 창출

```
    [ 빅데이터 가치 사슬 (Value Chain) ]

    +--------+   +--------+   +--------+   +--------+   +--------+
    |  수집  | -> |  저장  | -> |  처리  | -> |  분석  | -> |  활용  |
    +--------+   +--------+   +--------+   +--------+   +--------+
        |            |            |            |            |
    Data Source   Data Lake    Processing   Analytics    Business
                                                           Value

    [ 가치 창출 유형별 예시 ]

    가치 유형              활용 사례                        비즈니스 임팩트
    ───────────────────────────────────────────────────────────────────
    운영 최적화            공급망 관리, 재고 최적화         비용 절감 10~30%
    고객 경험 향상         개인화 추천, 챗봇               매출 증대 5~15%
    리스크 관리            사기 탐지, 신용 평가            손실 감소 20~50%
    신규 비즈니스          데이터 마켓플레이스             새로운 수익원
    의사결정 지원          대시보드, 예측 분석              의사결정 속도 향상

    [ ROI 계산 프레임워크 ]

    +----------------------------------------------------------+
    |                 빅데이터 ROI 계산                         |
    +----------------------------------------------------------+
    |                                                          |
    |   ROI = (가치 - 투자비용) / 투자비용 × 100               |
    |                                                          |
    |   투자비용:                                              |
    |   - 인프라 (Storage, Compute, Network)                   |
    |   - 소프트웨어 (License, SaaS)                           |
    |   - 인력 (Data Engineer, Data Scientist, Analyst)        |
    |   - 교육 및 변화 관리                                    |
    |                                                          |
    |   가치:                                                  |
    |   - 매출 증대                                            |
    |   - 비용 절감                                            |
    |   - 리스크 완화                                          |
    |   - 의사결정 품질 향상                                   |
    |                                                          |
    +----------------------------------------------------------+
```

**가치 특성**:
- **가치 밀도**: 대용량 데이터 중 가치 있는 정보의 비율은 낮음
- **잠재 가치**: 원시 데이터는 가공 후 가치 창출 (Data Refinery)
- **시간 가치**: 실시간 분석일수록 가치가 높음
- **네트워크 효과**: 데이터가 많을수록 분석 정확도 향상

## 상세 설명

### 1. 3V 모델의 상호 의존성

```
    [ 3V 상호작용 매트릭스 ]

                        Volume (규모)
                             |
                 +-----------+-----------+
                 |                       |
                 v                       v
    Velocity ----+----> [아키텍처] <----+---- Variety
    (속도)       |      결정 요소       |      (다양성)
                 |                       |
                 v                       v
          +------+------+         +------+------+
          | 실시간 처리 |         | 스키마 관리 |
          | 요구사항   |         | 복잡도     |
          +------------+         +------------+

    [ 3V 조합에 따른 처리 난이도 ]

                    높은 Velocity
                         |
         +---------------+---------------+
         |    Q3: 초고난이도            |
         |    (대용량 + 실시간 + 다양)   |
         |    예: 실시간 IoT + 비디오    |
         +---------------+---------------+
         |    Q1: 고난이도              |
         |    (대용량 + 실시간 + 정형)   |
         |    예: 금융 거래 실시간 분석  |
    높은 +---------------+---------------+ 낮은
    Volume      Q2: 중간 난이도          Volume
         |    (대용량 + 배치 + 다양)     |
         |    예: 데이터 레이크 구축     |
         +---------------+---------------+
         |    Q4: 기본 난이도            |
         |    (소용량 + 실시간 + 정형)   |
         |    예: 기존 RDBMS             |
         +---------------+---------------+
                         |
                    낮은 Velocity

    [ 3V 극복을 위한 기술 스택 ]

    Volume 극복:
    - 분산 스토리지: HDFS, S3, Azure Blob
    - 분산 처리: Hadoop MapReduce, Apache Spark
    - 컬럼형 저장: Parquet, ORC

    Velocity 극복:
    - 메시지 큐: Apache Kafka, RabbitMQ
    - 스트림 처리: Apache Flink, Spark Streaming
    - 인메모리 처리: Redis, Apache Ignite

    Variety 극복:
    - 데이터 레이크: 원시 데이터 저장
    - 스키마 온 리드: 동적 스키마 적용
    - 통합 메타데이터: Data Catalog
```

## 기술사 시험 포인트

### 1. 핵심 정의

| 항목 | 정의 | 핵심 포인트 |
|:---:|:---|:---|
| **Volume** | 데이터 규모 | TB~PB~EB, Scale-out 아키텍처 필요 |
| **Velocity** | 데이터 속도 | 실시간 vs 배치, 스트림 처리 |
| **Variety** | 데이터 다양성 | 정형/반정형/비정형, 스키마 관리 |
| **Veracity** | 데이터 품질 | 신뢰성, 정확성, 품질 관리 체계 |
| **Value** | 비즈니스 가치 | ROI, 의사결정, 혁신 |

### 2. 시험 빈출 키워드

1. **3V 모델**: Doug Laney (2001), Volume-Velocity-Variety
2. **5V 모델**: IBM 확장, +Veracity-Value
3. **Scale-out**: 수평 확장, Commodity Hardware, 선형 확장성
4. **Schema-on-Read**: 데이터 레이크의 핵심 특성, 유연성
5. **Lambda vs Kappa**: 배치+실시간 vs 단일 스트림 아키텍처

### 3. 단답형 예상 문제

- **Q**: 빅데이터 3V 모델을 정의하고 각 요소의 특징을 설명하시오.
- **A**: 3V 모델은 Doug Laney가 2001년 제시한 빅데이터 정의 프레임워크로, Volume(규모: TB~PB 단위의 대용량), Velocity(속도: 실시간 생성 및 처리), Variety(다양성: 정형/반정형/비정형 데이터)를 의미한다. 이후 IBM에서 Veracity(품질)와 Value(가치)를 추가한 5V 모델로 확장되었다.

### 4. 논술형 예상 문제

- **Q**: 빅데이터 5V 관점에서 기업의 데이터 플랫폼 구축 시 고려사항을 기술하시오.
- **A**:
  1. **Volume**: 3~5년 후 데이터 규모 예측, 스토리지 계층화(Hot/Warm/Cold), Scale-out 아키텍처
  2. **Velocity**: 실시간/배치 처리 요구 분석, Lambda/Kappa 아키텍처 선택, 지연 시간 SLA
  3. **Variety**: 정형/반정형/비정형 데이터 통합, 스키마 관리 전략, 메타데이터 관리
  4. **Veracity**: 데이터 품질 관리 파이프라인, 자동화 검증, 거버넌스 체계
  5. **Value**: 비즈니스 유즈케이스 정의, ROI 측정, 데이터 제품화 로드맵
