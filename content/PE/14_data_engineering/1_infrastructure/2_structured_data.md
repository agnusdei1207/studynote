# 정형 데이터 (Structured Data)

## 정의

**정형 데이터(Structured Data)**는 고정된 스키마(Schema)를 가지며, 행(Row)과 열(Column)로 구성된 테이블 형태로 조직화된 데이터를 의미한다. 데이터가 저장되기 전에 미리 정의된 데이터 모델(Data Model)과 구조(Structure)를 따라야 하며, 관계형 데이터베이스 관리 시스템(RDBMS)에서 주로 처리된다.

```
정형 데이터 정의:
"사전에 정의된 데이터 모델에 따라 구조화되어 저장되는 데이터로,
행과 열의 2차원 테이블 형태를 가지며, 각 컬럼은 명확한 데이터
타입과 제약조건을 가진다."

핵심 특징:
- 고정 스키마 (Fixed Schema)
- 행/열 구조 (Row/Column Structure)
- 명확한 데이터 타입 (Data Types)
- 관계형 모델 (Relational Model)
- SQL 질의 가능 (SQL Queryable)
```

데이터 엔지니어링 관점에서 정형 데이터는 가장 전통적이고 널리 사용되는 데이터 형태로, ACID 트랜잭션, 참조 무결성(Referential Integrity), 정규화(Normalization) 등의 핵심 데이터베이스 개념이 적용되는 대상이다.

## 핵심 개념

### 1. 정형 데이터의 구조

```
    [ 테이블 구조 예시: 고객(Customer) 테이블 ]

    +------------------------------------------------------------------+
    |                      Customer Table                               |
    +------------------------------------------------------------------+
    |  customer_id  |  name     |  email           |  join_date  | age |
    +------------------------------------------------------------------+
    |  INT (PK)     |  VARCHAR  |  VARCHAR         |  DATE       | INT |
    +------------------------------------------------------------------+
    |  1001         |  김철수   |  kim@mail.com    |  2024-01-15 | 32  |
    |  1002         |  이영희   |  lee@mail.com    |  2024-02-20 | 28  |
    |  1003         |  박민수   |  park@mail.com   |  2024-03-10 | 45  |
    +------------------------------------------------------------------+

    스키마 구성 요소:
    - 테이블명: Customer
    - 컬럼명: customer_id, name, email, join_date, age
    - 데이터 타입: INT, VARCHAR, DATE
    - 제약조건: PRIMARY KEY (customer_id)

    [ 관계형 모델 구조 ]

    +------------------------------------------------------------------+
    |                   Relational Model Components                     |
    +------------------------------------------------------------------+
    |                                                                  |
    |   +----------------+         +----------------+                  |
    |   |   Customers    |         |    Orders      |                  |
    |   +----------------+         +----------------+                  |
    |   | PK: customer_id|<--------| FK: customer_id|                  |
    |   |    name        |    1:N  | PK: order_id   |                  |
    |   |    email       |         |    order_date  |                  |
    |   |    join_date   |         |    total_amt   |                  |
    |   +----------------+         +----------------+                  |
    |                                      |                           |
    |                                      | N:1                       |
    |                                      v                           |
    |                              +----------------+                  |
    |                              |  Order_Items   |                  |
    |                              +----------------+                  |
    |                              | PK: item_id    |                  |
    |                              | FK: order_id   |                  |
    |                              | FK: product_id |                  |
    |                              |    quantity    |                  |
    |                              |    price       |                  |
    |                              +----------------+                  |
    |                                                                  |
    +------------------------------------------------------------------+

    관계 유형:
    - 1:1 (일대일): 고객-상세정보
    - 1:N (일대다): 고객-주문
    - N:M (다대다): 학생-과목 (중간 테이블 필요)
```

**구조적 특성**:
- **테이블(Table)**: 행과 열로 구성된 2차원 데이터 구조
- **행(Row/Tuple)**: 하나의 레코드(Record)를 나타내는 데이터 단위
- **열(Column/Attribute)**: 동일한 타입의 데이터를 가지는 속성
- **스키마(Schema)**: 테이블 구조, 데이터 타입, 제약조건의 정의

### 2. 데이터 타입과 제약조건

```
    [ 주요 데이터 타입 ]

    카테고리        데이터 타입           설명                    예시
    ──────────────────────────────────────────────────────────────────
    숫자형          INTEGER, INT         정수                    1, 100, -50
                    BIGINT               큰 정수                  9223372036854775807
                    DECIMAL(p,s)         고정 소수점              123.45
                    FLOAT, DOUBLE        부동 소수점              3.14159

    문자형          CHAR(n)              고정 길이 문자           'ABC    '
                    VARCHAR(n)           가변 길이 문자           'Hello'
                    TEXT, CLOB           대용량 텍스트            (긴 문서)

    날짜/시간       DATE                 날짜                     '2024-01-15'
                    TIME                 시간                     '14:30:00'
                    DATETIME             날짜+시간               '2024-01-15 14:30:00'
                    TIMESTAMP            타임존 포함             '2024-01-15 14:30:00+09'

    이진형          BINARY(n)            고정 길이 이진           (이미지, 파일)
                    VARBINARY(n)         가변 길이 이진
                    BLOB                 대용량 이진

    불리언          BOOLEAN              참/거짓                 TRUE, FALSE

    [ 제약조건(Constraints) ]

    +------------------------------------------------------------------+
    |                    Constraint Types                               |
    +------------------------------------------------------------------+
    |                                                                  |
    |   1. PRIMARY KEY (기본키)                                        |
    |      - 각 행을 고유하게 식별                                     |
    |      - NOT NULL + UNIQUE                                         |
    |      - 예: customer_id INT PRIMARY KEY                          |
    |                                                                  |
    |   2. FOREIGN KEY (외래키)                                        |
    |      - 다른 테이블의 기본키 참조                                 |
    |      - 참조 무결성 보장                                          |
    |      - 예: customer_id INT REFERENCES customers(id)             |
    |                                                                  |
    |   3. UNIQUE (고유)                                               |
    |      - 컬럼 값의 중복 방지                                       |
    |      - 예: email VARCHAR(100) UNIQUE                            |
    |                                                                  |
    |   4. NOT NULL (널 금지)                                          |
    |      - NULL 값 허용 안 함                                        |
    |      - 예: name VARCHAR(50) NOT NULL                            |
    |                                                                  |
    |   5. CHECK (검사)                                                |
    |      - 컬럼 값에 대한 조건 지정                                  |
    |      - 예: age INT CHECK (age >= 0 AND age <= 150)              |
    |                                                                  |
    |   6. DEFAULT (기본값)                                            |
    |      - 값 미지정 시 기본값 할당                                  |
    |      - 예: join_date DATE DEFAULT CURRENT_DATE                  |
    |                                                                  |
    +------------------------------------------------------------------+
```

### 3. 정규화(Normalization)

```
    [ 정규화 단계 ]

    단계          명칭                    목적                    이상현상 제거
    ─────────────────────────────────────────────────────────────────────────
    1NF          제1정규형               원자값(Atomic Value)     반복 그룹 제거
    2NF          제2정규형               부분 함수 종속 제거      부분 종속 제거
    3NF          제3정규형               이행 함수 종속 제거      이행 종속 제거
    BCNF         보이스-코드 정규형      결정자 함수 종속         모든 결정자 후보키
    4NF          제4정규형               다치 종속 제거           다치 종속 제거
    5NF          제5정규형               조인 종속 제거           조인 종속 제거

    [ 정규화 예시: 1NF -> 2NF -> 3NF ]

    [비정형 테이블] (1NF 위반 - 반복 그룹)
    +----------------------------------------------------------+
    | order_id | customer | customer_addr | products            |
    +----------------------------------------------------------+
    | 1001     | 김철수   | 서울시 강남구 | 상품A, 상품B, 상품C  |
    +----------------------------------------------------------+

    [1NF 적용] (원자값 분리)
    +------------------------------------------------------------------+
    | order_id | customer | customer_addr | product_name | quantity   |
    +------------------------------------------------------------------+
    | 1001     | 김철수   | 서울시 강남구 | 상품A        | 2          |
    | 1001     | 김철수   | 서울시 강남구 | 상품B        | 1          |
    | 1001     | 김철수   | 서울시 강남구 | 상품C        | 3          |
    +------------------------------------------------------------------+

    [2NF 적용] (부분 함수 종속 제거)
    Orders 테이블:
    +----------------------------------+
    | order_id (PK) | customer | addr  |
    +----------------------------------+
    | 1001          | 김철수   | 서울  |
    +----------------------------------+

    Order_Items 테이블:
    +--------------------------------------------+
    | order_id (PK) | product_name (PK) | qty   |
    +--------------------------------------------+
    | 1001          | 상품A             | 2     |
    | 1001          | 상품B             | 1     |
    | 1001          | 상품C             | 3     |
    +--------------------------------------------+

    [3NF 적용] (이행 함수 종속 제거)
    Customers 테이블:
    +---------------------------+
    | customer_id (PK) | name   |
    +---------------------------+
    | C001             | 김철수 |
    +---------------------------+

    Customer_Addresses 테이블:
    +----------------------------------+
    | customer_id (FK) | address      |
    +----------------------------------+
    | C001             | 서울시 강남구 |
    +----------------------------------+

    Orders 테이블:
    +------------------------------------+
    | order_id (PK) | customer_id (FK)  |
    +------------------------------------+
    | 1001          | C001              |
    +------------------------------------+

    [ 정규화 vs 반정규화 트레이드오프 ]

    +----------------------------------------------------------+
    |             Normalization vs Denormalization              |
    +----------------------------------------------------------+
    |                                                          |
    |   정규화 (Normalization)                                 |
    |   장점:                                                  |
    |   - 데이터 중복 최소화 -> 저장 공간 절약                 |
    |   - 이상현상(Anomaly) 방지                               |
    |   - 데이터 무결성 보장                                   |
    |                                                          |
    |   단점:                                                  |
    |   - 복잡한 조인(JOIN) 필요                               |
    |   - 읽기 성능 저하                                       |
    |   - 쿼리 복잡도 증가                                     |
    |                                                          |
    |   반정규화 (Denormalization)                             |
    |   장점:                                                  |
    |   - 조인 감소 -> 읽기 성능 향상                          |
    |   - 쿼리 단순화                                          |
    |   - 집계 성능 향상                                       |
    |                                                          |
    |   단점:                                                  |
    |   - 데이터 중복 증가                                     |
    |   - 갱신 비용 증가                                       |
    |   - 무결성 유지 어려움                                   |
    |                                                          |
    +----------------------------------------------------------+
```

### 4. Schema-on-Write 패턴

```
    [ Schema-on-Write 프로세스 ]

    +----------------------------------------------------------+
    |                Schema-on-Write Flow                       |
    +----------------------------------------------------------+
    |                                                          |
    |   Source Data                                            |
    |       |                                                  |
    |       v                                                  |
    |   +----------------+                                     |
    |   | Extract        |                                     |
    |   | (데이터 추출)  |                                     |
    |   +----------------+                                     |
    |       |                                                  |
    |       v                                                  |
    |   +----------------+                                     |
    |   | Transform      |  <-- 스키마 적용                    |
    |   | (데이터 변환)  |      타입 변환, 검증, 정제          |
    |   +----------------+                                     |
    |       |                                                  |
    |       v                                                  |
    |   +----------------+                                     |
    |   | Validate       |  <-- 제약조건 검증                 |
    |   | (데이터 검증)  |      PK, FK, NOT NULL, UNIQUE      |
    |   +----------------+                                     |
    |       |                                                  |
    |       v                                                  |
    |   +----------------+                                     |
    |   | Load           |  <-- 데이터 적재                   |
    |   | (데이터 적재)  |      (이미 구조화된 상태)          |
    |   +----------------+                                     |
    |       |                                                  |
    |       v                                                  |
    |   +----------------+                                     |
    |   | Structured     |                                     |
    |   | Storage (RDBMS)|                                     |
    |   +----------------+                                     |
    |                                                          |
    +----------------------------------------------------------+

    [ Schema-on-Write vs Schema-on-Read 비교 ]

    구분              Schema-on-Write           Schema-on-Read
    ────────────────────────────────────────────────────────────────
    적용 시점         저장 전 (ETL)             읽기 시 (쿼리)
    스키마 유연성     낮음 (고정)               높음 (동적)
    쓰기 성능         낮음 (변환 비용)          높음 (원시 저장)
    읽기 성능         높음 (최적화됨)           낮음 (변환 필요)
    데이터 품질       높음 (사전 검증)          낮음 (검증 미흡)
    적합 사례         OLTP, DW                  Data Lake
```

### 2. 인덱스(Index)와 쿼리 최적화

```
    [ 인덱스 구조 ]

    1. B-Tree 인덱스 (가장 일반적)

    +----------------------------------------------------------+
    |                      B-Tree Index                         |
    +----------------------------------------------------------+
    |                                                          |
    |                    [Root Node]                           |
    |                    +-------+                             |
    |                    | 50-100|                             |
    |                    +-------+                             |
    |                     /     \                              |
    |           [Internal]     [Internal]                      |
    |           +-------+       +-------+                      |
    |           | 25-50 |       | 75-100|                      |
    |           +-------+       +-------+                      |
    |            /    \           /    \                        |
    |      [Leaf]   [Leaf]   [Leaf]   [Leaf]                   |
    |      +-----+  +-----+  +-----+  +-----+                  |
    |      |1-25 |  |26-50|  |51-75|  |76-100|                 |
    |      +-----+  +-----+  +-----+  +-----+                  |
    |         |        |        |        |                     |
    |      [Data]   [Data]   [Data]   [Data]                   |
    |                                                          |
    |   특징: 범위 검색, 정렬, 등호/부등호 연산에 효율적       |
    |                                                          |
    +----------------------------------------------------------+

    2. 해시 인덱스

    +----------------------------------------------------------+
    |                      Hash Index                          |
    +----------------------------------------------------------+
    |                                                          |
    |   Key        Hash Function         Bucket                |
    |   +-----+         |                +------------------+  |
    |   |ABC  | ------> h(ABC) -------> | (ABC, ptr)       |  |
    |   +-----+                        +------------------+  |
    |                                                          |
    |   특징: 등호(=) 검색에만 사용, O(1) 시간 복잡도         |
    |                                                          |
    +----------------------------------------------------------+

    [ 인덱스 유형별 비교 ]

    인덱스 유형        검색 패턴          장점              단점
    ───────────────────────────────────────────────────────────────
    B-Tree            범위, 정렬, =     범용적            크기 큼
    Hash              = 만              빠름              범위 불가
    Full-text         텍스트 검색       LIKE 최적화       공간 많이 사용
    Bitmap            저카디널리티      압축 효율         갱신 비용 높음
    Composite         다중 컬럼         복합 조건         순서 중요

    [ 쿼리 실행 계획 (Execution Plan) ]

    +----------------------------------------------------------+
    |                 Query Execution Plan                      |
    +----------------------------------------------------------+
    |                                                          |
    |   Query: SELECT * FROM orders WHERE customer_id = 1001  |
    |                                                          |
    |   Execution Steps:                                       |
    |   1. Parse SQL -> AST (Abstract Syntax Tree)            |
    |   2. Validate (테이블, 컬럼 존재 확인)                  |
    |   3. Optimize (인덱스 사용 여부 결정)                   |
    |   4. Execute (물리적 데이터 접근)                       |
    |                                                          |
    |   +--------------------------------------------------+   |
    |   |  Seq Scan vs Index Scan                          |   |
    |   |                                                  |   |
    |   |  선택률(Selectivity)에 따른 인덱스 사용 결정     |   |
    |   |                                                  |   |
    |   |  - 선택률 < 10%: Index Scan 유리                |   |
    |   |  - 선택률 > 30%: Seq Scan 유리                  |   |
    |   |  - 중간: 통계 정보 기반 비용 판단               |   |
    |   +--------------------------------------------------+   |
    |                                                          |
    +----------------------------------------------------------+
```

### 3. ACID 트랜잭션

```
    [ ACID 특성 ]

    +----------------------------------------------------------+
    |                       ACID Properties                     |
    +----------------------------------------------------------+
    |                                                          |
    |   A - Atomicity (원자성)                                 |
    |   "트랜잭션은 모두 실행되거나 모두 실행되지 않는다"      |
    |                                                          |
    |   예: A 계좌 -> B 계좌 이체                              |
    |   [A 출금] + [B 입금] = 하나의 원자적 단위               |
    |                                                          |
    |   +-----------------------------------------------+       |
    |   | Transaction T1                               |       |
    |   | +-------------------------------------------+|       |
    |   | | 1. BEGIN TRANSACTION                      ||       |
    |   | | 2. UPDATE accounts SET bal = bal - 10000  ||       |
    |   | |    WHERE id = 'A'                         ||       |
    |   | | 3. UPDATE accounts SET bal = bal + 10000  ||       |
    |   | |    WHERE id = 'B'                         ||       |
    |   | | 4. COMMIT (or ROLLBACK on error)          ||       |
    |   | +-------------------------------------------+|       |
    |   +-----------------------------------------------+       |
    |                                                          |
    |   C - Consistency (일관성)                               |
    |   "트랜잭션 전후의 데이터베이스 상태가 일관되어야 한다"  |
    |   - 무결성 제약조건 유지                                 |
    |   - 비즈니스 규칙 준수                                   |
    |                                                          |
    |   I - Isolation (격리성)                                 |
    |   "동시 트랜잭션 간 간섭이 없어야 한다"                   |
    |                                                          |
    |   격리 수준:                                             |
    |   +---------------------------------------------+         |
    |   | Level          | Dirty Read | NRE | Phantom|         |
    |   +---------------------------------------------+         |
    |   | Read Uncommitted|   O       |  O  |   O    |         |
    |   | Read Committed  |   X       |  O  |   O    |         |
    |   | Repeatable Read |   X       |  X  |   O    |         |
    |   | Serializable    |   X       |  X  |   X    |         |
    |   +---------------------------------------------+         |
    |   NRE: Non-Repeatable Read                              |
    |                                                          |
    |   D - Durability (지속성)                                |
    |   "커밋된 트랜잭션은 영구히 보존된다"                    |
    |   - Write-Ahead Logging (WAL)                           |
    |   - Checkpoint                                          |
    |                                                          |
    +----------------------------------------------------------+

    [ 동시성 문제 ]

    문제                    설명                                    예시
    ────────────────────────────────────────────────────────────────────
    Dirty Read             미커밋 데이터 읽기                      T1이 롤백한 데이터를 T2가 읽음
    Non-Repeatable Read    같은 쿼리 결과가 다름                   T1이 읽은 후 T2가 수정
    Phantom Read           새로운 행이 나타남/사라짐               T1이 범위 쿼리 후 T2가 삽입
    Lost Update            동시 수정 시 한 쪽 손실                 T1, T2 동시 수정
```

### 4. OLTP vs OLAP

```
    [ OLTP (Online Transaction Processing) ]

    +----------------------------------------------------------+
    |                        OLTP                               |
    +----------------------------------------------------------+
    |                                                          |
    |   특성:                                                  |
    |   - 실시간 트랜잭션 처리                                 |
    |   - 빠른 응답 시간 (ms 단위)                             |
    |   - 많은 동시 사용자                                     |
    |   - 단순한 쿼리 (INSERT, UPDATE, SELECT)                |
    |   - 정규화된 스키마 (3NF)                               |
    |                                                          |
    |   예시:                                                  |
    |   - 은행 이체 시스템                                     |
    |   - 전자상거래 주문 시스템                               |
    |   - 예약 시스템                                          |
    |                                                          |
    |   데이터 특성:                                           |
    |   - 현재 데이터 (Current Data)                          |
    |   - 상세 데이터 (Detailed Data)                         |
    |   - 빈번한 갱신 (Frequent Updates)                      |
    |                                                          |
    +----------------------------------------------------------+

    [ OLAP (Online Analytical Processing) ]

    +----------------------------------------------------------+
    |                        OLAP                               |
    +----------------------------------------------------------+
    |                                                          |
    |   특성:                                                  |
    |   - 대용량 분석 쿼리                                     |
    |   - 복잡한 집계 쿼리                                     |
    |   - 읽기 위주 (Read-heavy)                              |
    |   - 긴 쿼리 실행 시간 (초~분)                           |
    |   - 반정규화된 스키마 (Star/Snowflake)                  |
    |                                                          |
    |   예시:                                                  |
    |   - 비즈니스 인텔리전스 (BI)                             |
    |   - 데이터 웨어하우스                                    |
    |   - 리포팅 및 대시보드                                   |
    |                                                          |
    |   데이터 특성:                                           |
    |   - 이력 데이터 (Historical Data)                       |
    |   - 집계 데이터 (Aggregated Data)                       |
    |   - 읽기 전용 (Read-only)                               |
    |                                                          |
    +----------------------------------------------------------+

    [ OLTP vs OLAP 비교 ]

    구분              OLTP                    OLAP
    ────────────────────────────────────────────────────────────────
    목적              트랜잭션 처리           분석 및 의사결정
    사용자            일반 사용자             분석가, 경영진
    데이터양          적음~중간               대용량
    쿼리 패턴         단순, 인덱스 활용       복잡, 전체 스캔
    응답 시간         ms                      초~분
    동시성            높음                    낮음
    갱신              빈번함                  드묾
    스키마            정규화                  반정규화
```

### 5. 정형 데이터의 한계와 진화

```
    [ 정형 데이터의 한계 ]

    1. 스키마 경직성 (Schema Rigidity)
    +----------------------------------------------------------+
    |                                                          |
    |   문제: 스키마 변경 어려움                               |
    |                                                          |
    |   기존:                                                  |
    |   +------------------+                                   |
    |   | id | name | email|                                   |
    |   +------------------+                                   |
    |                                                          |
    |   변경 필요: phone 컬럼 추가                             |
    |                                                          |
    |   1. ALTER TABLE ADD COLUMN                              |
    |   2. 기존 데이터 처리 (NULL 또는 기본값)                |
    |   3. 애플리케이션 코드 수정                              |
    |   4. 배포 및 테스트                                      |
    |                                                          |
    |   -> 비용이 크고 위험                                    |
    |                                                          |
    +----------------------------------------------------------+

    2. 비정형 데이터 처리 제약
    +----------------------------------------------------------+
    |                                                          |
    |   정형 데이터는 행/열로 표현 가능한 데이터만 처리        |
    |                                                          |
    |   비정형 데이터:                                         |
    |   - 이미지, 비디오                                       |
    |   - 음성, 자연어 텍스트                                  |
    |   - 센서 데이터                                          |
    |   - 소셜 미디어                                          |
    |                                                          |
    |   -> 별도 저장소 또는 BLOB으로 저장 (비효율적)          |
    |                                                          |
    +----------------------------------------------------------+

    3. 수평 확장성 한계
    +----------------------------------------------------------+
    |                                                          |
    |   RDBMS는 주로 수직 확장(Scale-up)에 적합               |
    |                                                          |
    |   Scale-out 제약:                                        |
    |   - 조인 연산의 분산 처리 복잡                           |
    |   - ACID 보장 어려움                                     |
    |   - 분산 트랜잭션 비용                                   |
    |                                                          |
    |   해결:                                                  |
    |   - 샤딩(Sharding)                                       |
    |   - 읽기 복제(Read Replica)                              |
    |   - NewSQL (Spanner, CockroachDB)                        |
    |                                                          |
    +----------------------------------------------------------+

    [ 정형 데이터의 진화: 하이브리드 접근 ]

    +----------------------------------------------------------+
    |              Modern Structured Data Stack                |
    +----------------------------------------------------------+
    |                                                          |
    |   1. Polyglot Persistence (다중 저장소)                 |
    |                                                          |
    |      +-----------+     +-----------+     +-----------+   |
    |      |   RDBMS   |     |  NoSQL    |     | Data Lake |   |
    |      | (정형)    |     | (반정형)  |     | (비정형)  |   |
    |      +-----------+     +-----------+     +-----------+   |
    |            |                |                 |          |
    |            +----------------+-----------------+          |
    |                             |                            |
    |                      통합 쿼리 엔진                       |
    |                      (Trino, Presto)                     |
    |                                                          |
    |   2. Data Lakehouse                                     |
    |                                                          |
    |      정형 데이터와 비정형 데이터를 단일 계층에서 처리    |
    |      - Delta Lake, Apache Iceberg                        |
    |      - ACID 트랜잭션 + 스키마 진화                       |
    |                                                          |
    |   3. Schema Registry                                    |
    |                                                          |
    |      동적 스키마 관리                                    |
    |      - Avro, Protobuf, JSON Schema                       |
    |      - 하위 호환성 유지                                  |
    |                                                          |
    +----------------------------------------------------------+
```

## 기술사 시험 포인트

### 1. 핵심 정의

| 항목 | 정의 | 핵심 포인트 |
|:---:|:---|:---|
| **정형 데이터** | 고정 스키마를 가진 행/열 구조 데이터 | RDBMS, SQL, ACID |
| **스키마** | 데이터 구조, 타입, 제약조건의 정의 | DDL로 정의 |
| **정규화** | 중복 최소화, 이상현상 방지 | 1NF~5NF, BCNF |
| **ACID** | 원자성, 일관성, 격리성, 지속성 | 트랜잭션 보장 |
| **Schema-on-Write** | 저장 시 스키마 적용 | ETL, 데이터 검증 |

### 2. 시험 빈출 키워드

1. **정규화 단계**: 1NF(원자값) → 2NF(부분종속제거) → 3NF(이행종속제거)
2. **ACID**: Atomicity, Consistency, Isolation, Durability
3. **격리 수준**: Read Uncommitted → Read Committed → Repeatable Read → Serializable
4. **인덱스**: B-Tree, Hash, Bitmap, Full-text
5. **OLTP vs OLAP**: 트랜잭션 처리 vs 분석, 정규화 vs 반정규화

### 3. 단답형 예상 문제

- **Q**: 정형 데이터의 특징과 장단점을 설명하시오.
- **A**: 정형 데이터는 고정된 스키마를 가지며 행과 열로 구성된 테이블 형태의 데이터이다. 장점으로는 SQL을 통한 효율적인 쿼리, ACID 트랜잭션 보장, 데이터 무결성 유지, 최적화된 인덱싱 등이 있다. 단점으로는 스키마 변경의 어려움, 비정형 데이터 처리 제약, 수평 확장성 한계 등이 있다.

### 4. 논술형 예상 문제

- **Q**: 정규화의 개념과 단계별 특징을 설명하고, 반정규화를 수행하는 상황을 기술하시오.
- **A**:
  1. **정규화 개념**: 데이터 중복을 최소화하고 이상현상(Anomaly)을 방지하기 위해 테이블을 분해하는 과정
  2. **단계별 특징**:
     - 1NF: 모든 컬럼이 원자값(Atomic Value)을 가짐
     - 2NF: 부분 함수 종속 제거 (복합키의 일부에만 종속되는 속성 제거)
     - 3NF: 이행 함수 종속 제거 (A→B→C인 경우 C를 별도 테이블로)
     - BCNF: 모든 결정자가 후보키여야 함
  3. **반정규화 수행 상황**:
     - 읽기 성능 향상이 필요한 OLAP 환경
     - 잦은 조인으로 인한 쿼리 복잡도 증가
     - 실시간 집계 성능 요구
     - 단, 데이터 갱신 비용 증가와 무결성 저하를 고려해야 함
