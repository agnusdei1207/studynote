# NUMA (Non-Uniform Memory Access)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NUMA는 **"메모리 **액세스 **시간**이 **노드**에 **따라 **다른 **다중 **프로세서 **아키텍처"**로, **SMP**(Uniform Memory Access)의 **확장성 **한계**를 **극복**하고 **로컬 **메모리**는 **빠르고 **원격 **메모리**는 **느린 **구조**이다.
> 2. **구조**: **Node**(CPU + **로컬 **메모리)로 **구성**되며 **Interconnect**(QPI, **UPI**, **HyperTransport**)로 **연결**되고 **OS**는 **Memory **Affinity**로 **스레드**를 **로컬 **메모리**에 **할당**한다.
> 3. **최적화**: **NUMA-aware **알고리즘**으로 **로컬 **메모리 **액세스**를 **최대화**하고 **Remote **액세스**를 **최소화**하며 **First-Touch **정책**과 **Thread **Binding**으로 **성능**을 **향상**시킨다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
NUMA는 **"비균일 메모리 액세스 다중 프로세서"**이다.

**SMP vs NUMA**:
| 구분 | SMP | NUMA |
|------|-----|------|
| **메모리** | 공유 | 분산 |
| **액세스** | 균일 | 비균일 |
| **확장성** | ≤16 코어 | 수백 코어 |
| **지연** | 일정 | 로컬 < 원격 |

### 💡 비유
NUMA는 ****여러 **집 **구조****와 같다.
- **각 집**: 노드 (CPU + 로컬 메모리)
- **이웃 집 방문**: 원격 메모리 액세스 (느림)

### 메모리 액세스 패턴

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         NUMA Memory Access                                                 │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Local Access (Fast):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Node 0                                                                               │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  CPU 0 ──→ [Local Memory Node 0] → ~50ns                                           │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Remote Access (Slow):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Node 0                              Node 1                                            │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  CPU 0 ──→ [Interconnect] ──→ [Remote Memory Node 1] → ~100ns                      │  │  │
    │  │          (QPI/UPI)                                                                  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Performance Impact:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Workload      │  Local  │  Remote  │  Penalty                                        │  │
    │  ──────────────────────────────────────────────────────────────────────────────────────│  │
    │  Sequential    │  100%   │   50%    │  2x slower                                     │  │
    │  Random        │  100%   │   30%    │  3x slower                                     │  │
    │  Stride (2MB)  │  100%   │   60%    │  1.7x slower                                   │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### NUMA vs SMP vs UMA

| 구분 | UMA (SMP) | NUMA |
|------|-----------|------|
| **메모리** | 단일 공유 | 분산 |
| **대역폭** | 공유 버스 | 노드당 독립 |
| **확장성** | 8-16 코어 | 100+ 코어 |
| **OS 지원** | 간단 | NUMA-aware 필요 |

### NUMA 스케줄링

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         NUMA-aware Scheduling                                             │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Without NUMA Awareness:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Thread on CPU 0 (Node 0)                                                               │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Malloc() → Memory allocated from Node 1 (free!)                                     │  │  │
    │  │  Access → Remote access (~100ns) × Slow!                                            │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    With NUMA Awareness (First-Touch):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Thread on CPU 0 (Node 0)                                                               │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Malloc() → Memory allocated from Node 0 (local!)                                    │  │  │
    │  │  Access → Local access (~50ns) ✓ Fast!                                              │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  → OS scheduler tries to keep thread on same node                                       │  │
    │  → Memory allocated from node of thread that first touches                              │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 인터커넥트 비교

| 구분 | Intel QPI | Intel UPI | AMD InfinityFabric |
|------|-----------|-----------|-------------------|
| **대역폭** | ~6.4 GT/s | ~10.4 GT/s | ~25 GB/s |
| **지연** | ~40ns | ~30ns | ~50ns |
| **구조** | Mesh | Mesh | Chiplet |

## Ⅳ. 실무 적용 및 기술사적 판단

### NUMA 최적화

```c
// NUMA-aware allocation (libnuma)
#include <numa.h>

// 1. 확인
int numa_num_nodes = numa_num_configured_nodes();

// 2. 로컬 메모리 할당
numa_set_preferred(0);  // Node 0 선호
void *local_mem = numa_alloc_local(1024);

// 3. 특정 노드에 할당
void *node0_mem = numa_alloc_onnode(1024, 0);

// 4. 스레드 바인딩
numa_run_on_node(0);  // Node 0에서만 실행

// 5. 정책 설정
numa_set_membind(&nodemask);  // 지정 노드에만 할당
```

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 데이터베이스 서버 구성
**상황**: 4소켓 NUMA 서버
**판단**: NUMA 최적화

```bash
# NUMA 토폴로지 확인
numactl --hardware

# 출력 예:
available: 4 nodes (0-3)
node 0 cpus: 0 1 2 3 4 5 6 7
node 0 size: 128 GB
node 1 cpus: 8 9 10 11 12 13 14 15
node 1 size: 128 GB
...

# PostgreSQL NUMA 최적화 실행

    -c config_file \
    -D /data/postgresql

# MySQL NUMA 설정
```

## Ⅴ. 기대효과 및 결론

### NUMA 기대 효과

| 효과 | SMP | NUMA |
|------|-----|------|
| **확장성** | 16 코어 | 100+ 코어 |
| **대역폭** | 공유 | 노드당 독립 |
| **지연** | 로컬 | 로컬 + 원격 |

### 모범 사례

1. **First-Touch**: 초기 접근 스레드 노드에 할당
2. **Binding**: 스레드를 노드에 고정
3. **Interleaving**: 대용량 메모리는 분산
4. **모니터링**: `numastat`, `lscpu`

### 미래 전망

1. **Chiplet**: AMD Rome/Milan 스타일
2. **CXL**: Cache Coherent Interconnect
3. **Composable**: 분산 리소스
