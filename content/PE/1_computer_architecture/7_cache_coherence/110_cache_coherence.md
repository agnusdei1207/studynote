# 캐시 일관성 (Cache Coherence)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 캐시 일관성은 **"다중 코어 환경에서 **각 Core의 **캐시가 **동일한 데이터에 대해 **일관된 **값을 **유지**하는 **메커니즘"**으로, **Cache Coherence Protocol**(MESI, MOESI)이 **상태 전이**(State Transition)을 **관리**하고 **Bus Snooping** 또는 **Directory-Based** 방식으로 **동기화**한다.
> 2. **가치**: **데이터 불일치**(Inconsistency)로 인한 **오동작**을 **방지**하고 **코어 간 데이터 **공유**를 **안전하게** 지원하며 **Multi-Core Processor**에서 **메모리 모델**(Memory Model)의 **기초**가 된다.
> 3. **융합**: **MESI**(Modified, Exclusive, Shared, Invalid) 프로토콜이 **대중적**이며 **Write-Through**, **Write-Back** 정책과 **연계**되고 **Intel**, **AMD**, **ARM**의 **Multi-Core** 시스템에서 **구현**된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
캐시 일관성은 **"여러 캐시가 동일한 데이터에 대해 일관된 값을 유지하는 성질"**이다.

**캐시 불일치 문제**:
- Core A: X = 5 (캐시 수정)
- Core B: X = 3 (구버전)
- → 데이터 불일치!

### 💡 비유
캐시 일관성은 **"공유 문서 동기화"**와 같다.
- **각자 수정**: 다른 버전
- **일관성**: 모두 같은 버전

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         캐시 일관성의 필요성                                              │
└─────────────────────────────────────────────────────────────────────────────────────────┘

Single Core:
    • 캐시 1개
    • 일관성 문제 없음
         ↓
Multi-Core (2000년대):
    • 각 Core의 L1 Cache
    • 데이터 공유
    • 불일치 발생
         ↓
Coherence Protocol:
    • Snooping (Bus 기반)
    • Directory (Point-to-Point)
```

### MESI 프로토콜

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         MESI State Machine                                               │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  State      │  Description        │  Read  │  Write   │  Bus 주도권                │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  Modified  │  캐시만 최신         │  Hit    │  Hit     │  O (Memory보다 최신)      │
    │            │  (다른 캐시는 Invalid)│         │          │                            │
    │  Exclusive │  캐시만 소유         │  Hit    │  → M    │  O (Memory와 동일)        │
    │            │  (다른 캐시에 없음)   │         │          │                            │
    │  Shared    │  여러 캐시에 존재     │  Hit    │  → M    │  X (Memory와 동일)        │
    │            │  (읽기만 가능)       │         │  (Bus 획득)│                          │
    │  Invalid   │  유효하지 않음       │  Miss   │  → M    │  X                        │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [State Transition]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │                                                                                         │
    │    Read Miss                              Read Hit                                      │
    │  ┌─────────┐      Read Miss          ┌─────────┐                                    │
    │  │ Invalid │ ───────────────────>   │  Shared  │                                    │
    │  └────┬────┘                        └────┬────┘                                    │
    │       │                                │                                           │
    │       │ Write                         │ Write (Bus 획득)                         │
    │       ▼                               ▼                                           │
    │    ┌─────────┐    Bus R/W           ┌─────────┐                                   │
    │    │Modified │ <─────────────────── │Modified │                                   │
    │    └─────────┘                        └─────────┘                                   │
    │         ▲                                   ▲                                       │
    │         │           Flush              │           Write (Shared)                     │
    │         └─────────────────────────────────┴───────────────────────────────────────   │
    │                                       Bus Invalidate                              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Bus Snooping

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Bus Snooping Protocol                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 모든 캐시가 Bus를 감청(Snoop)                                                          │
    │  │  • Broadcast 기반 통신                                                              │
    │  │                                                                                      │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  Core A 요청: Write X = 5                                                             │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  │  1. Core A가 Bus에 "Write X = 5" 브로드캐스트                                       │  │  │
    │  │  │                                                                                  │  │  │
    │  │  │  2. 모든 Core가 Bus 감청:                                                           │  │  │
    │  │  │     • Core B: "나도 X 가지고 있음" → Invalidate X                                   │  │  │
    │  │  │     • Core C: "나도 X 가지고 있음" → Invalidate X                                   │  │  │
    │  │  │                                                                                  │  │  │
    │  │  │  3. Core A만 X를 Modified 상태로 유지                                               │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Snoop Bus Message]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Message      │  Description                                                     │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  BusRd        │  Read 요청 (Shared 상태로)                                           │
    │  BusRdX       │  Read Exclusive 요청 (Modified 위해)                                   │
    │  Flush        │  Memory에 쓰기                                                         │
    │  FlushX       │  Memory에 쓰고 Exclusive 획득                                           │
    │  Invalidate   │  다른 캐시의 데이터 무효화                                               │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### Snooping vs Directory-Based

| 구분 | Bus Snooping | Directory-Based |
|------|-------------|-----------------|
| **통신** | Broadcast | Point-to-Point |
| **확장성** | 낮음 | 높음 |
| **복잡도** | 낮음 | 높음 |
| **사용** | 소규모 | 대규모/NUMA |

### Write Policy 영향

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Write Policy와 Cache Coherence                                   │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Write-Through]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 쓸 때마다 Memory에도 반영                                                           │
    │  │  • 일관성 유지 쉬움                                                                  │
    │  │  • Bus 트래픽 많음                                                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Write-Back]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 캐시만 수정, 나중에 Memory 반영                                                     │
    │  │  • 빠름, Bus 트래픽 적음                                                            │
    │  │  • 일관성 유지 복잡                                                                  │
    │  │  • MESI 프로토콜 필요                                                               │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 캐시 일관성 확인
**상황**: 멀티코어 성능 분석
**판단**:

```bash
# Linux perf로 캐시 코히어런스 확인
$ perf stat -e cache-references,cache-misses,instructions,cycles ./app

# Hardware Performance Counter
# Intel: MESI transition event
$ perf stat -e rNNr,HW_STROBE.COUNT ./multithread_app

# False Sharing 감지
# • 서로 다른 Core가 인접 데이터 쓰기
# • 캐시 라인 경계 문제
# → Padding으로 해결
```

## Ⅴ. 기대효과 및 결론

### 캐시 일관성 기대 효과

| 효과 | 일관성 없음 | 일관성 있음 |
|------|----------|----------|
| **데이터 정확성** | 오동작 | 보장 |
| **성능** | 빠름 (보장 없음) | 오버헤드 |
| **확장성** | - | 중요 |

### 미래 전망

1. **Non-Uniform Cache**: 계층적 캐시
2. **Coherent Interconnect**: CCIX, CXL
3. **Hybrid Protocol**: Snooping + Directory

### ※ 참고 표준/가이드
- **Intel**: Cache Coherence
- **AMD**: HyperTransport
- **ARM**: AMBA ACE
