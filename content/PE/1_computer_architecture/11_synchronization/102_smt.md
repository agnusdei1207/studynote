# SMT (Simultaneous Multithreading)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SMT는 단일 물리 코어의 수퍼스칼라 자원을 여러 하드웨어 스레드가 '동일 클럭 사이클' 내에서 동시에 공유하여 사용하게 함으로써 ILP와 TLP를 결합한 기술이다.
> 2. **가치**: 수퍼스칼라 아키텍처에서 발생하는 수직적/수평적 낭비 슬롯(Waste Slots)을 다른 스레드의 명령어로 채워, 프로세서 이용률과 전체 처리량을 획기적으로 높인다.
> 3. **융합**: Intel의 하이퍼스레딩(Hyper-Threading)으로 대중화되었으며, 비순차 실행(OoO) 기술과 결합하여 현대 고성능 CPU의 핵심 성능 동력이 된다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념
**SMT (Simultaneous Multithreading)**는 프로세서 내부의 연산 유닛(ALU, FPU 등)을 여러 스레드가 동시에 점유하여 실행할 수 있도록 허용하는 하드웨어 멀티스레딩 기법이다. 기존의 멀티스레딩이 클럭 단위로 스레드를 교체(Fine-grained)했다면, SMT는 한 클럭 내에 있는 여러 발급 슬롯(Issue Slots)에 서로 다른 스레드의 명령어를 섞어서 배치할 수 있다.

#### 💡 비유
SMT는 **"한 입에 여러 반찬을 골고루 먹는 것"**과 같다.
1. **싱글 스레드**: 한 입에 오직 밥만 먹는다. 반찬이 남으면 아깝다 (연산 유닛 낭비).
2. **기존 멀티스레딩**: 한 입은 밥만, 다음 입은 고기만 먹는다. 밥을 씹는 동안 고기는 못 먹는다 (시간 분할).
3. **SMT**: 한 입에 밥과 고기를 같이 넣어 씹는다. 입안 공간(연산 유닛)을 꽉 채워 효율적으로 영양을 섭취한다 (동시 처리).

#### 등장 배경 및 발전 과정
1. **수퍼스칼라의 낭비(Waste)**:
    - **수직적 낭비(Vertical Waste)**: 특정 클럭에 아무 명령어 실행 불가 (캐시 미스 등).
    - **수평적 낭비(Horizontal Waste)**: 일부 슬롯만 사용 (명령어 간 의존성).
2. **기술적 돌파구**: 1990년대 후반 연구되었으며, 2002년 Intel이 Pentium 4에 'Hyper-Threading'이라는 이름으로 도입하며 상용화되었다.
3. **현대적 의미**: 코어를 무작정 늘리는 것보다 면적 및 전력 효율이 좋아, 서버 및 데스크톱 CPU의 표준 사양이 되었다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 관련 기술 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **Replicated Resources** | 스레드별 독립 상태 유지 | PC, Register Set, RAT(Renaming Table) 등 복제 | Logical Processor | 개인별 식판 |
| **Shared Resources** | 실행 효율 극대화 위해 공유 | Cache, ALU, FPU, Branch Predictor 등 공유 | Pipeline Stages | 공용 식탁 |
| **Partitioned Resources** | 스레드 간 간섭 방지 위해 분할 | Instruction Queue, Reorder Buffer(ROB) 등 | Fairness Control | 줄 서기 라인 |
| **Thread Priority Logic** | 명령어 발급 우선순위 결정 | 부하가 적은 스레드에 더 많은 슬롯 할당 | Resource Allocation | 지배인의 안내 |

#### 정교한 구조 다이어그램 (ASCII)

```text
       [ Superscalar Issue Slots (e.g., 4-wide) ]
    ┌───────┬───────┬───────┬───────┐
    │ Slot0 │ Slot1 │ Slot2 │ Slot3 │
    └───────┴───────┴───────┴───────┘
                │
    [ 싱글 스레드 실행 ]      [ SMT 실행 (Thread A & B) ]
    ┌───┬───┬───┬───┐      ┌───┬───┬───┬───┐
 T1 │ A │ A │ - │ - │   T1 │ A │ A │ B │ B │ <── 가득 채움
    ├───┼───┼───┼───┤      ├───┼───┼───┼───┤
 T2 │ - │ - │ - │ - │   T2 │ B │ - │ A │ A │ <── 낭비 최소화
    └───┴───┴───┴───┘      └───┴───┴───┴───┘
      (수평/수직 낭비)        (높은 자원 이용률)

                │
    ┌───────────▼────────────────────────┐
    │       Execution Engine (Shared)    │
    │  ┌───────┐ ┌───────┐ ┌──────────┐  │
    │  │ ALU 0 │ │ ALU 1 │ │ Load/Str │  │
    │  └───────┘ └───────┘ └──────────┘  │
    └───────┬───────────────┬────────────┘
            ▼               ▼
    [ Context A ]       [ Context B ] <── 물리 코어는 하나지만
    (Regs, PC)          (Regs, PC)       논리 CPU는 두 개
```

#### 심층 동작 원리

1. **명령어 인출(Fetch)**: I-Cache에서 두 스레드의 명령어를 교대로 또는 동시에 가져온다.
2. **레지스터 리네이밍(Renaming)**: 각 스레드별로 독립된 RAT를 사용하여 논리 레지스터를 물리 레지스터로 매핑한다. 이때 물리 레지스터 파일은 공유된다.
3. **스케줄링 및 발급(Issue)**: 공유된 예약 스테이션(Reservation Station)에서 실행 준비가 된 명령어를 찾는다. 이때 스레드 구분 없이 '먼저 준비된' 명령어를 발급한다.
4. **실행(Execute)**: 공유된 연산 유닛에서 병렬로 실행된다.
5. **완료(Commit)**: 각 스레드별 ROB를 참조하여 최종적으로 프로그램 순서에 맞게 결과를 확정한다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교

| 항목 | Fine-grained MT | SMT (Simultaneous) | CMP (Chip Multi-core) |
| :--- | :--- | :--- | :--- |
| **동시 실행** | 불가능 (클럭 교대) | 가능 (동일 클럭 내) | 가능 (물리적 분리) |
| **슬롯 낭비** | 수평적 낭비 존재 | 낭비 거의 없음 | 낭비 없음 (별도 코어) |
| **구현 난이도** | 중간 | 매우 높음 (자원 경합 관리) | 중간 (코어 복제) |
| **전력/면적 효율** | 좋음 | 최상 | 낮음 (면적 2배) |

#### 과목 융합 관점 분석
- **보안 (Side Channel)**: SMT는 동일 코어의 실행 유닛과 캐시를 완벽히 공유하므로, 한 스레드가 실행되는 시간을 측정하여 다른 스레드의 데이터를 유추하는 **'L1 Terminal Fault(L1TF)'**나 **'PortSmash'** 같은 보안 취약점에 매우 취약합니다. 이를 방지하기 위해 기밀 데이터를 다루는 가상화 환경에서는 SMT를 끄거나 코어별로 격리하는 정책을 씁니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 기술사적 판단 (실무 시나리오)
**[시나리오]** 클라우드 가공 서버(VM)의 CPU 할당 전략
1. **분석**: 클라우드 사업자는 보통 vCPU(가상 CPU)를 SMT의 논리 스레드 단위로 판매합니다.
2. **위험**: 한 물리 코어에 입점한 두 고객(VM)이 모두 연산 집약적인 작업을 하면 성능이 절반으로 떨어지는 '노이지 네이버(Noisy Neighbor)' 문제가 발생합니다.
3. **판단**:
    - **일반 서비스**: 비용 절감을 위해 SMT 기반 vCPU 사용.
    - **고성능/금융 서비스**: 성능 보장 및 보안을 위해 물리 코어를 독점하는 'Dedicated Host' 또는 SMT를 비활성화한 인스턴스 권장.

#### 도입 시 고려사항 (체크리스트)
- [ ] **성능 프로파일링**: SMT 활성화 시 실제 IPC가 향상되는가, 아니면 캐시 경합으로 떨어지는가?
- [ ] **스케줄러 힌트**: OS가 물리 코어와 논리 스레드를 구분하여, 먼저 빈 물리 코어부터 채우도록 설정되었는가? (SMT-aware scheduling)

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량적/정성적 기대효과
- **처리량**: 약 5~10%의 다이 면적 증가만으로 전체 성능을 15~30% 향상시킬 수 있는 고효율 기술.
- **경제성**: 서버 집약도를 높여 데이터센터 상암 비용(TCO) 절감.

#### 미래 전망 및 진화 방향
- **가변 SMT**: 작업 부하가 적을 때는 스레드를 끄고 전압을 낮추며, 부하가 늘어날 때만 동적으로 스레드를 활성화하여 전력 효율을 극대화함.
- **보안 강화형 SMT**: 하드웨어 수준에서 스레드 간 자원 공유를 강제로 격리하거나 노이즈를 섞어 사이드 채널 공격을 무력화하는 연구가 활발함.
