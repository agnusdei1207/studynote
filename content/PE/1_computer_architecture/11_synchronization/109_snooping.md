# 스누핑 프로토콜 (Snooping Protocol)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스누핑 프로토콜은 모든 프로세서가 공유 버스(Shared Bus)를 통해 오가는 모든 메모리 트래픽을 지속적으로 '엿듣고(Snooping)' 감시하여, 자신의 캐시 데이터와 일치 여부를 판단하고 일관성을 유지하는 분산 제어 방식이다.
> 2. **가치**: 별도의 중앙 관리 장치 없이도 각 코어가 독립적으로 자신의 캐시 상태를 업데이트할 수 있어 구현이 간단하며, 소규모 멀티코어 시스템에서 매우 빠른 응답성을 제공한다.
> 3. **융합**: MESI, MOESI 등 상태 전이 프로토콜의 물리적 전달 메커니즘이며, 버스 기반 아키텍처의 대역폭 한계로 인해 시스템 확장성(Scalability)의 주요 결정 요인이 된다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념
**스누핑 프로토콜(Snooping Protocol)**은 멀티프로세서 시스템에서 캐시 일관성을 유지하기 위한 방법의 하나로, 모든 캐시 제어기가 시스템 버스에 흐르는 주소와 제어 신호를 모니터링하여 자신이 가진 캐시 블록의 상태를 결정하는 방식이다. 특정 코어가 메모리에 쓰기를 수행할 때 버스에 신호를 뿌리면(Broadcast), 이를 듣고 있던 다른 코어들이 자신의 복사본을 무효화하거나 업데이트한다.

#### 💡 비유
스누핑 프로토콜은 **"마을 방송을 항상 귀 기울여 듣는 주민들"**과 같다.
1. **상황**: 마을 사람들이 각자 자기 집 수첩(캐시)에 "내일 날씨"를 적어놨다.
2. **동작**: 누군가 기상청에 가서 날씨 정보를 바꾸면, 확성기로 마을 전체에 "날씨 바뀌었다!"라고 방송한다(Bus Broadcast).
3. **반응**: 집에서 쉬던 사람들은 방송을 듣고(Snooping), 자기 수첩의 내용을 지우거나(Invalidate) 새 내용으로 고친다(Update).
4. **특징**: 게시판(중앙 디렉터리)을 보러 갈 필요 없이 앉아서 소식을 듣지만, 방송이 너무 많아지면 시끄러워서(트래픽 과부하) 마을이 시끄러워진다.

#### 등장 배경 및 발전 과정
1. **공유 버스의 탄생**: 초창기 멀티프로세서는 모든 CPU가 하나의 버스에 매달려 있었다. 이 '모두가 연결된 통로'라는 특성을 활용해 정보를 공유하자는 아이디어가 스누핑의 시작이다.
2. **확장성의 한계**: 코어 수가 늘어나면서 버스 하나에 너무 많은 방송이 쏟아지자, 이를 해결하기 위해 디렉터리 기반 방식이 등장하며 보완 관계를 형성했다.
3. **현재**: 8~16코어 정도의 일반적인 데스크톱 및 모바일 CPU에서는 여전히 가장 효율적인 일관성 유지 수단으로 사용된다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 관련 기술 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **Snoop Controller** | 버스 신호 상시 감시 | 주소 비교기(Address Matcher) 작동 | Logic Gate, Tag RAM | 귀를 쫑긋 세운 주민 |
| **Shared Bus** | 정보의 브로드캐스트 통로 | 단일 주소 버스(Single Address Bus) | AMBA, AXI | 마을 확성기 |
| **Coherence States** | 데이터의 상태 관리 | MESI 등의 상태 머신 운용 | State Transition | 수첩의 메모 상태 |
| **Write Invalidate** | 타 캐시 무효화 유도 | 쓰기 시 타 코어 복사본 무효화 신호 발송 | Write-back Policy | "내용 지워!" 명령 |
| **Write Update** | 타 캐시 직접 업데이트 | 쓰기 시 새 데이터를 모든 캐시에 전송 | Write-through (일부) | "새 내용으로 고쳐!" |

#### 스누핑 아키텍처 구조 (ASCII)

```text
       [ Main Memory ]
              │
    ┌─────────┴─────────┐
    │    System Bus     │ <── 모든 스누핑 정보가 흐르는 통로 (공유 자원)
    └─┬────────┬────────┘
      │        │        │
      │ Snoop  │ Snoop  │ Snoop (버스 감시)
      ▼        ▼        ▼
    ┌───┐    ┌───┐    ┌───┐
    │ C │    │ C │    │ C │ (Cache Controller)
    └───┘    └───┘    └───┘
    ┌───┐    ┌───┐    ┌───┐
    │ P │    │ P │    │ P │ (Processor Core)
    └───┘    └───┘    └───┘

    [동작 예시]
    1. P1이 주소 A에 쓰기 시도
    2. C1이 버스에 "주소 A 무효화(Invalidate)" 신호 전송
    3. C2, C3가 버스 신호를 엿듣고(Snoop) 주소 A가 있는지 태그 확인
    4. 있으면 자신의 캐시 상태를 'Invalid'로 변경
```

#### 심층 동작 원리 (두 가지 접근법)

1. **Write-Invalidate (쓰기 무효화)**:
    - 한 코어가 데이터를 쓸 때, 다른 모든 코어의 해당 복사본을 **'못 쓰게(Invalid)'** 만드는 방식.
    - **특징**: 이후에 읽을 때 다시 가져와야 하므로 미스율은 높을 수 있지만, 버스 트래픽이 적어 가장 널리 쓰임 (MESI의 기본).
2. **Write-Update (쓰기 갱신)**:
    - 데이터를 쓸 때, 다른 모든 코어의 복사본에 **'새 값'**을 직접 써주는 방식.
    - **특징**: 데이터가 항상 최신이라 미스율은 낮지만, 쓰기 때마다 데이터 전체를 버스에 뿌려야 하므로 트래픽이 매우 큼.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교

| 항목 | 스누핑 (Snooping) | 디렉터리 (Directory) |
| :--- | :--- | :--- |
| **제어 방식** | 분산 제어 (각 코어가 감시) | 중앙/분산 관리 (장부 기록) |
| **통신 매체** | 공유 버스 (Broadcast) | 포인트-투-포인트 (Unicast) |
| **확장성** | **낮음 (코어 수 제한)** | **높음 (대규모 시스템 가능)** |
| **지연 시간** | **매우 낮음** | 중간 (디렉터리 조회 필요) |
| **데이터 일관성** | MESI, MOESI 등 | 프로토콜은 동일하나 전달법이 다름 |

#### 과목 융합 관점 분석
- **컴퓨터 구조 연계**: 버스의 대역폭이 스누핑 성능의 핵심입니다. 현대 CPU는 단일 버스 대신 링(Ring)이나 메쉬(Mesh) 구조를 사용하지만, 논리적으로는 모든 노드가 신호를 볼 수 있게 하여 스누핑 개념을 확장 적용합니다.
- **네트워크 연계**: 이더넷(Ethernet)의 초기 방식인 CSMA/CD와 철학적으로 유사합니다. 공유 매체를 누구나 들을 수 있고, 충돌을 방지하거나 감지하는 메커니즘이 필요하기 때문입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 기술사적 판단 (실무 시나리오)
**[시나리오]** 멀티코어 칩 설계 시 스누핑과 디렉터리 중 선택
1. **분석**:
    - **8코어 이하**: 버스 구조가 간단하고 지연 시간이 짧은 스누핑이 유리.
    - **32코어 이상**: 버스 트래픽이 기하급수적으로 늘어나 시스템이 멈출 수 있으므로 디렉터리 방식 필수.
2. **인사이트**: 최근에는 L2 캐시까지는 스누핑을 쓰고, 대규모 L3 캐시나 소켓 간 통신에는 디렉터리 방식을 섞어 쓰는 **'계층적 일관성 모델'**이 대세입니다.

#### 도입 시 고려사항 (체크리스트)
- [ ] **Bus Contention**: 코어 수 증가에 따른 버스 경합을 어떻게 제어할 것인가?
- [ ] **Snoop Filter**: 불필요한 스누핑 요청이 코어 내부 연산을 방해하지 않도록 필터링하고 있는가?
- [ ] **Power Consumption**: 모든 코어가 버스를 항상 듣고 있어야 하므로 발생하는 전력 소모 대책은 있는가?

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량적/정성적 기대효과
- **신속한 일관성 전파**: 데이터 수정 즉시 타 코어에 전파되어 대기 시간 최소화.
- **단순한 로직**: 중앙 관리자 없이 개별 코어의 로직만으로 전체 시스템 일관성 유지.

#### 미래 전망 및 진화 방향
- **Ring Snooping**: Intel의 링 버스 구조에서 패킷이 순환하며 스누핑을 수행하는 방식.
- **In-Network Snooping**: 인터커넥트 스위치 자체가 스누핑 정보를 처리하여 트래픽을 줄이는 기술.
- **CXL 2.0/3.0**: 가속기와 CPU 간의 저지연 캐시 공유를 위해 스누핑 메커니즘을 외부 버스(PCIe 기반)로 확장.
