# 디렉터리 기반 프로토콜 (Directory-based Protocol)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 디렉터리 기반 프로토콜은 중앙 또는 분산된 '디렉터리'라는 관리 대장에 캐시 블록의 복사 위치와 상태 정보를 저장하여, 노드 간 1:1(Unicast) 통신으로 캐시 일관성을 유지하는 기술이다.
> 2. **가치**: 공유 버스(Shared Bus)의 브로드캐스트 병목을 제거함으로써, 수백 개 이상의 프로세서 코어를 가진 대규모 MPP(Massively Parallel Processing) 및 NUMA 시스템의 확장성(Scalability)을 보장한다.
> 3. **융합**: 상호 연결망(Interconnection Network) 기술 및 분산 메모리 아키텍처와 결합되어, 현대 하이엔드 서버 CPU 및 슈퍼컴퓨터의 핵심 일관성 유지 체계를 형성한다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념
**디렉터리 기반 프로토콜(Directory-based Protocol)**은 멀티프로세서 환경에서 특정 데이터 블록이 어느 로컬 캐시에 복제되어 있는지를 중앙 저장소(Directory)에서 통합 관리하는 방식이다. 데이터를 수정할 때 모든 노드에 신호를 뿌리는 대신, 해당 데이터를 가진 노드들에게만 선택적으로 무효화 신호를 보냄으로써 통신 효율을 극대화한다.

#### 💡 비유
디렉터리 기반 프로토콜은 **"도서관의 대출 대장 관리"**와 같다.
1. **상황**: 마을 사람들이 도서관 책(메모리)을 복사해서 자기 집(캐시)에 가져갔다.
2. **동작**: 도서관 사서는 누가 어떤 책의 복사본을 가졌는지 **대출 대장(디렉터리)**에 꼼꼼히 적어둔다.
3. **수정**: 누군가 책 내용을 고치려 하면, 사서는 대장을 보고 그 책을 빌려간 사람들에게만 연락해서 "그거 옛날 거니까 버려!"라고 알려준다.
4. **특징**: 마을 방송(Snooping)을 할 필요가 없어 조용하지만, 사서가 대장을 일일이 확인해야 하는 수고(관리 오버헤드)가 든다.

#### 등장 배경 및 발전 과정
1. **스누핑의 한계**: 코어 수가 16~32개를 넘어가면 공유 버스의 대역폭이 포화되어 성능이 급격히 저하된다. 모든 노드가 모든 트래픽을 감시해야 하는 전력 소모 문제도 발생했다.
2. **분산 메모리의 등장**: 프로세서가 물리적으로 멀리 떨어져 있는 NUMA 아키텍처에서는 중앙 집중식 버스를 사용할 수 없게 되어, 메시지 기반의 디렉터리 방식이 필연적으로 도입되었다.
3. **현재**: 인텔의 UPI(Ultra Path Interconnect)나 AMD의 Infinity Fabric 등 대규모 인터커넥트를 사용하는 서버급 CPU의 표준 기술로 자리 잡았다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 관련 기술 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **Directory Controller** | 일관성 상태 관리의 주체 | 상태 비트(Shared, Modified) 유지 | FSM (State Machine) | 도서관 사서 |
| **Presence Vector** | 데이터 소유 노드 식별 | 비트맵(Bit-vector)으로 노드 위치 기록 | Node ID Tagging | 대출자 명단 |
| **Home Node** | 특정 주소를 담당하는 노드 | 해당 주소의 디렉터리 정보 상주 | Memory Interleaving | 해당 구역 담당자 |
| **Remote Node** | 데이터를 복제해간 타 노드 | 홈 노드에 데이터 요청 및 무효화 응답 | Local Cache | 원거리 대출자 |
| **Interconnect** | 노드 간 메시지 전송망 | 포인트-투-포인트 메시징 수행 | Mesh, Torus, Fat-tree | 전용 택배망 |

#### 심층 동작 원리 (The 3-Step Protocol)

1. **읽기 요청 (Read Miss)**:
    - 원격 노드가 홈 노드에 '읽기 요청' 메시지를 보낸다.
    - 홈 노드의 디렉터리는 상태를 **Shared**로 바꾸고, 요청 노드의 비트를 1로 셋팅한 후 데이터를 전송한다.
2. **쓰기 요청 (Write Miss)**:
    - 원격 노드가 홈 노드에 '쓰기 권한 및 데이터'를 요청한다.
    - 홈 노드는 디렉터리를 확인하여 이 데이터를 가진 **모든 노드에게 개별적으로 무효화(Invalidate) 메시지**를 보낸다.
3. **무효화 완료 (ACK)**:
    - 모든 노드로부터 무효화 완료 응답(ACK)을 받으면, 홈 노드는 요청 노드에게 "이제 네가 주인이야"라고 허락하며 상태를 **Modified**로 바꾼다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교

| 항목 | 스누핑 (Snooping) | 디렉터리 (Directory) |
| :--- | :--- | :--- |
| **통신 방식** | 브로드캐스트 (Broadcast) | **유니캐스트 (Unicast / Point-to-Point)** |
| **관리 위치** | 각 로컬 캐시 제어기 | **중앙 또는 분산 디렉터리** |
| **확장성** | 낮음 (버스 병목) | **매우 높음 (수천 노드 가능)** |
| **지연 시간** | 짧음 (즉시 감지) | 약간 김 (메시지 왕복 시간 필요) |
| **하드웨어 비용** | 낮음 | **높음 (디렉터리 저장 공간 추가)** |

#### 과목 융합 관점 분석
- **컴퓨터 구조 연계**: 디렉터리 정보를 저장하기 위한 'Directory Cache'나 'Snoop Filter' 기술과 결합됩니다. 디렉터리 자체가 메모리를 차지하므로, 이를 효율적으로 압축하거나 캐싱하는 기법이 연구됩니다.
- **분산 시스템 연계**: 분산 파일 시스템(HDFS, Ceph)의 메타데이터 관리 철학과 매우 흡사합니다. 데이터의 위치(Location)와 상태(Consistency)를 별도의 저장소에서 관리한다는 점이 일맥상통합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 기술사적 판단 (실무 시나리오)
**[시나리오]** 128코어 NUMA 서버 아키텍처 설계
1. **분석**: 노드 간 거리가 멀고 버스 병목이 예상됨.
2. **선택**: **분산 디렉터리 방식** 채택.
3. **최적화 전략**:
    - **First-touch Policy**: 데이터를 처음 쓰는 노드를 'Home Node'로 지정하여 디렉터리 접근 지연 최소화.
    - **Directory Inlining**: 메모리 컨트롤러 내부에 디렉터리 정보를 포함시켜 조회 속도 향상.

#### 도입 시 고려사항 (체크리스트)
- [ ] **디렉터리 오버헤드**: 전체 메모리의 몇 %를 디렉터리 정보 저장에 쓸 것인가?
- [ ] **Interconnect Topology**: 메시지 폭주 시 교착 상태(Deadlock)를 방지할 수 있는 라우팅 알고리즘이 있는가?
- [ ] **False Invalidation**: 디렉터리 정보가 부족하여 불필요한 무효화 신호가 발생하지 않는가?

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량적/정성적 기대효과
- **무한한 확장성**: 이론적으로 노드 수 증가에 따른 통신 오버헤드가 선형적으로만 증가.
- **전력 절감**: 불필요한 캐시 조회를 줄여 시스템 전체의 동적 전력 소모 감소.

#### 미래 전망 및 진화 방향
- **In-Network Coherence**: 네트워크 스위치 장비가 디렉터리 역할의 일부를 수행하여 메시지 홉(Hop) 수를 줄이는 기술.
- **CXL 3.0**: 외부 메모리 풀이나 가속기와 데이터를 공유할 때, 디렉터리 방식을 하드웨어 프로토콜 수준에서 지원하여 'Global Coherence' 달성.
