# 기아 현상 (Starvation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기아 현상(Starvation)은 **"특정 프로세스가 필요한 자원을 영원히 할당받지 못해 실행되지 못하는 상태"**로, **데드락(Deadlock)**과 달리 **다른 프로세스는 실행되지만** 특정 프로세스만 **영구히 대기**하는 현상이다.
> 2. **가치**: **우선순위 기반 스케줄링(Priority Scheduling)**에서 발생하기 쉬우며, **에이징(Aging)**, **우선순위 상향**, **Time Slice 조정** 등의 **공정성(Fairness)** 보장 기법으로 해결한다.
> 3. **융합**: **운영체제(Operating System)**의 **스케줄러**, **데이터베이스**의 **트랜잭션 병행 제어**, **분산 시스템(Distributed System)**의 **리더 선출(Leader Election)**에서 발생하며, **Priority Inversion**은 **Mars Pathfinder**의 실제 사례로 유명하다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
기아 현상(Starvation)은 **"특정 프로세스가 무한히 자원을 기다리는 상황"**이다.

**기아 현상의 정의**:
- 자원을 요청한 프로세스가 무한히 대기
- 다른 프로세스는 정상 실행
- 시스템 전체는 정상 동작

**데드락과의 차이**:
- **데드락**: 모든 프로세스가 대기 (시스템 교착)
- **기아**: 일부 프로세스만 대기 (시스템 정상)

### 💡 비유
기아 현상은 **"편애받는 손님"**과 같다.
- **식당**: VIP만 계속 들어오면 일반 손님은 영원히 대기
- **은행**: 직원이 아는 사람만 먼저 처리

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         기아 현상의 필요성 인식                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

초기 스케줄링 (1960년대):
    • FIFO, Round Robin
    • 기아 현상 드뭄
         ↓
우선순위 스케줄링 (1970년대):
    • Priority Scheduling 도입
    • 높은 우선순위 프로세스 선호
    • 낮은 우선순위 기아 발생
         ↓
에이징(Aging) 기법 (1980년대):
    • 오래 대기하면 우선순위 증가
    • 기아 현상 완화
         ↓
현대 시스템 (2000년대~):
    • CFS (Completely Fair Scheduler)
    • Fair Share Scheduling
    • Multi-level Feedback Queue
```

## Ⅱ. 아키텍처 및 핵심 원리

### 기아 현상 시나리오

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         우선순위 기반 기아 현상                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [상황: 우선순위 기반 스케줄링]

    Ready Queue (Priority 순)

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Priority 1 (Highest): P1 ──▶ P1' ──▶ P1'' ──▶ P1''' ──▶ ... (계속 도착)          │
    │                                                                                         │
    │  Priority 2:        P2 ──▶ P2' ──▶ P2''                                               │
    │                                                                                         │
    │  Priority 3 (Lowest): P3 (영원히 대기)                                                │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [문제 발생 과정]

    Time T0: P1, P2, P3 도착
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Ready Queue: [P1(1), P2(2), P3(3)]  ( ) 안은 우선순위                                   │
    │                                                                                         │
    │  P1 실행 (우선순위 가장 높음)                                                            │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Time T1: P1' 도착 (Priority 1)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Ready Queue: [P1'(1), P2(2), P3(3)]                                                  │
    │                                                                                         │
    │  P1' 실행 (P1보다 우선순위 높음은 아니지만, P1'도 Priority 1)                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Time T2: P1'' 도착 (Priority 1)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Ready Queue: [P1''(1), P2(2), P3(3)]                                                 │
    │                                                                                         │
    │  P1'' 실행                                                                              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    결과: Priority 1 프로세스가 계속 도착하면 P3는 영원히 실행되지 않음 (기아 현상)
```

### 에이징(Aging) 기법

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         에이징 (Aging)으로 기아 해결                                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [에이징 알고리즘]

    New Priority = Base Priority + (Wait Time / Aging Factor)

    예: Aging Factor = 10, 단위 = 1초

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Process   │ Base Priority │ Wait Time │ Aging Bonus │ New Priority │                 │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  P1        │       1       │    0s     │      0      │      1       │                 │
    │  P1'       │       1       │    0s     │      0      │      1       │                 │
    │  P2        │       2       │   10s     │      1      │      3       │                 │
    │  P3        │      10       │   90s     │      9      │     19       │                 │
    │                                                                                         │
    │  → P3가 충분히 오래 대기하면 우선순위가 상승하여 실행됨                                    │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [에이징 구현 예시]

    // 에이징이 포함된 우선순위 계산
    function calculatePriority(process, currentTime) {
        let waitTime = currentTime - process.arrivalTime;
        let agingBonus = Math.floor(waitTime / 10);  // 10초마다 1 증가
        return process.basePriority - agingBonus;  // 낮을수록 높은 우선순위
    }

    // 스케줄러
    function scheduler(readyQueue, currentTime) {
        readyQueue.forEach(p => {
            p.effectivePriority = calculatePriority(p, currentTime);
        });
        return readyQueue.sort((a, b) => a.effectivePriority - b.effectivePriority)[0];
    }
```

### Priority Inversion (우선순위 역전)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Priority Inversion (우선순위 역전)                               │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [정의]

    낮은 우선순위 프로세스가 높은 우선순위 프로세스가 필요한 자원을 점유하여
    높은 우선순위 프로세스가 낮은 우선순위 프로세스에 의해 간접적으로 블록되는 현상

    [시나리오]

    Priority: High(H) > Medium(M) > Low(L)

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Time 0:                                                                                 │
    │  L: Mutex Lock 획득                                                                     │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  L 상태: Running, Lock = True                                                       │  │
    │  │  M 상태: Ready, Lock = False                                                         │  │
    │  │  H 상태: Ready, Lock = False                                                         │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │
    │  Time 1:                                                                                 │
    │  H: Mutex Lock 요청 → Block (L가 점유 중)                                               │
    │  M: 실행 시작                                                                            │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  L 상태: Running, Lock = True                                                       │  │
    │  │  M 상태: Running, Lock = False                                                       │  │
    │  │  H 상태: Blocked, Lock = False (기다림)                                               │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │
    │  Time 2:                                                                                 │
    │  M: Mutex Lock 요청 → Block (L가 점유 중)                                               │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  L 상태: Running, Lock = True                                                       │  │
    │  │  M 상태: Blocked, Lock = False (기다림)                                              │  │
    │  │  H 상태: Blocked, Lock = False (기다림)                                               │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │
    │  → H는 L보다 우선순위 높지만, L가 Lock을 해제할 때까지 대기                                 │
    │  → M는 H보다 우선순위 낮지만, M가 Lock을 요청하여 H가 M보다 먼저 Lock을 얻지 못함                │
    │  → 우선순위 역전 발생: H < M < L 실제 실행 순서                                           │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [해결 방법: Priority Inheritance]

    Priority Inheritance: 낮은 우선순위 프로세스가 높은 우선순위 프로세스를 블록하면
                        일시적으로 우선순위 상속

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Time 2 (Priority Inheritance 적용):                                                   │
    │  L: 우선순위 상속 (L의 우선순위 → H의 우선순위)                                         │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  L 상태: Running, Lock = True, Priority = H (상속)                                 │  │
    │  │  M 상태: Blocked, Lock = False                                                      │  │
    │  │  H 상태: Blocked, Lock = False                                                      │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │
    │  Time 3:                                                                                 │
    │  L: Lock 해제 → 우선순위 복구                                                            │
    │  H: Lock 획득 → 실행 (우선순위 가장 높음)                                                │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 기아 현상 해결 기법 비교

| 기법 | 원리 | 장점 | 단점 | 적용 |
|------|------|------|------|------|
| **Aging** | 오래 대기 시 우선순위 증가 | 공정성 | 복잡함 | 일반적 |
| **Priority Ceiling** | 최대 우선순위 할당 | 예측 가능 | 유연성 ↓ | RTOS |
| **Fair Share** | 그룹별 CPU 할당 | 그룹 공정 | 복잡함 | 멀티테넌트 |
| **Round Robin** | Time Slice 순환 | 단순 | 응답시간 ↑ | 시분할 |

### 과목 융합 관점 분석

#### 1. 데이터베이스 ↔ 기아 현상
- **트랜잭션**: 오래된 트랜잭션 우선순위 상향
- **Lock Manager**: Wait-die vs Wound-wait
- **MVCC**: 기아 현상 감소

#### 2. 분산 시스템 ↔ 기아 현상
- **리더 선출**: Bully Algorithm, Ring Algorithm
- **분산 락**: Fairness 보장
- **Consensus**: Paxos, Raft

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: Mars Pathfinder Priority Inversion
**상황**: 1997년 Mars Pathfinder, Priority Inversion 발생
**문제**:
- High: 통신 작업
- Medium:气象 데이터 수집
- Low: 정보 시스템 작업
- Low가 Mutex 점유 → High 대기 → Medium이 계속 실행 → 시스템 리셋

**해결**:
- VxWorks Priority Inheritance 활성화
- 온라인 패치 전송 → 재부팅 후 문제 해결

```c
// VxWorks Priority Inheritance 설정
semMInit(lock, SEM_Q_PRIORITY, SEM_INVERSION_SAFE);
```
