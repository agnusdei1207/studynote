# 데드락 (Deadlock)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데드락은 **"두 개 이상의 프로세스가 서로가 가진 자원을 기다리며 영구히 대기 상태에 빠지는 현상"**으로, **상호 배제(Mutual Exclusion)**, **점유 대기(Hold and Wait)**, **비선점(No Preemption)**, **순환 대기(Circular Wait)**의 4가지 필요조건이 **동시에** 만족될 때 발생한다.
> 2. **가치**: **데드락 예방(Prevention)**, **회피(Avoidance)**, **검출 및 복구(Detection & Recovery)**의 3대 전략으로 관리하며, **은행원 알고리즘(Banker's Algorithm)**, **자원 할당 그래프(Resource Allocation Graph)**로 안전한 상태를 보장한다.
> 3. **융합**: **분산 시스템(Distributed System)**의 **분산 락(Distributed Lock)**, **분산 트랜잭션(Distributed Transaction)**의 **2PC(Two-Phase Commit)**, **마이크로서비스**의 **Saga 패턴**으로 확장되며, **데이터베이스 교착 상태**의 핵심 개념이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
데드락(Deadlock)은 **"여러 프로세스가 서로가 가진 자원을 기다리며 진행이 멈추는 상태"**이다.

**데드락의 정의 (Coffman et al., 1971)**:
- 일련의 프로세스가 각각 일부 자원을 점유한 상태에서
- 다른 프로세스가 점유한 자원을 기다리며
- 영구히 대기하는 상황

### 💡 비유
데드락은 **"교차로에서의 사면초가"**와 같다.
- **차량 A**: B를 기다림
- **차량 B**: A를 기다림
- **결과**: 둘 다 움직일 수 없음

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         데드락 연구의 역사                                               │
└─────────────────────────────────────────────────────────────────────────────────────────┘

초기 (1960년대):
    • 단일 프로그래밍: 데드락 문제 없음
    • 일괄처리 시스템
         ↓
다중 프로그래밍 (1970년대):
    • 여러 프로세스 동시 실행
    • 자원 공유 필요
    • Coffman: 데드락 4조건 정의
         ↓
은행원 알고리즘 (1971년, Dijkstra):
    • 안전 상태(safe state) 개념
    • 자원 할당 전 사전 검사
    • 데드락 회피 가능
         ↓
현대 시스템 (2000년대~):
    • 분산 락 (Distributed Lock)
    • 타임아웃 기반 회피
    • Saga 패턴 (마이크로서비스)
```

## Ⅱ. 아키텍처 및 핵심 원리

### 데드락 4가지 필요조건 (Coffman Conditions)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         데드락 4가지 필요조건                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [필요조건 1] 상호 배제 (Mutual Exclusion)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 자원은 한 번에 한 프로세스만 사용 가능                                              │
    │  • 예: 프린터, 파일, 락                                                                 │
    │  • 공유 가능 자원(읽기 전용)는 데드락 원인 아님                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [필요조건 2] 점유 대기 (Hold and Wait)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 최소한 하나의 자원을 점유한 상태에서                                                 │
    │  • 다른 프로세스가 가진 자원을 추가로 기다림                                              │
    │  • 예: 파일 A를 잡은 상태에서 파일 B 기다림                                             │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [필요조건 3] 비선점 (No Preemption)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 점유한 자원을 강제로 뺏을 수 없음                                                     │
    │  • 자원은 자발적으로 반환만 가능                                                         │
    │  • 예: CPU는 선점 가능하지만, 락은 비선점                                                 │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [필요조건 4] 순환 대기 (Circular Wait)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 대기 프로세스 집합이 순환 형태                                                        │
    │  • P0 → P1 → P2 → ... → Pn → P0                                                         │
    │  • 각 프로세스는 다음 프로세스가 가진 자원 기다림                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [데드락 발생: 4조건 모두 만족 시]

    Process P1           Process P2
    ┌──────────┐        ┌──────────┐
    │ Lock A   │        │ Lock B   │
    │ (점유)   │        │ (점유)   │
    └──────────┘        └──────────┘
        │                   │
        │ Wait B            │ Wait A
        │                   │
        ▼                   ▼
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  ◀────── 순환 대기 ──────▶                                                            │
    │  ┌──────────┐        ┌──────────┐                                                      │
    │  │ Lock A   │        │ Lock B   │                                                      │
    │  │ (P1 점유)│        │ (P2 점유)│                                                      │
    │  └──────────┘        └──────────┘                                                      │
    │                                                                                         │
    │  → P1은 B를 기다림, P2는 A를 기다림                                                      │
    │  → 영구히 대기 (데드락)                                                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 정교한 데드락 시나리오

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         식사하는 철학자 문제 (Dining Philosophers)                       │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [배치]
         Chopstick C
         │
         │
    ┌────┴────┐
    │         │
    P2        P1
    │         │
    │         │
Chopstick B Chopstick A
    │         │
    │         │
    P3        P0
    │         │
    │         │
    └────┬────┘
         │
         │
      Chopstick D
         │
         │
         P4
         │
         │
      Chopstick E

    [데드락 시나리오]

    시간 T0: 모든 철학자가 동시에 식사 시작
    • P0: Chopstick A 획득 → Chopstick E 대기
    • P1: Chopstick B 획득 → Chopstick A 대기
    • P2: Chopstick C 획득 → Chopstick B 대기
    • P3: Chopstick D 획득 → Chopstick C 대기
    • P4: Chopstick E 획득 → Chopstick D 대기

    시간 T1: 순환 대기 발생
    • P0 → P1 → P2 → P3 → P4 → P0 (순환)
    • 모두가 오른쪽 젓가락 기다림
    • 영구히 대기 (데드락)

    [해결 방법]

    1. 자원 계층 (Resource Hierarchy):
       • 젓가락에 번호 부여 (A < B < C < D < E)
       • 낮은 번호 먼저 획득
       • P0: A 먼저 획득 → E 획득 시도 (성공: A < E)

    2. 홀수/짝수 전략:
       • 홀수 철학자: 왼쪽 먼저
       • 짝수 철학자: 오른쪽 먼저

    3. 한 명만 기다림:
       • 마지막 젓가락이 없으면 기다리지 않고 생각
```

### 자원 할당 그래프 (Resource Allocation Graph)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         자원 할당 그래프 (RAG)                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [그래프 구성 요소]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • P₁, P₂, ... : 프로세스 (Process) - 원으로 표현                                      │
    │  • R₁, R₂, ... : 자원 (Resource) - 사각형으로 표현                                     │
    │  • Pᵢ → Rⱼ : 요청 간선 (Request Edge) - 점선                                          │
    │  • Rⱼ → Pᵢ : 할당 간선 (Assignment Edge) - 실선                                         │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [데드락 없는 상태]

    P₁ ──request──▶ R₁ ──assign──▶ P₂ ──request──▶ R₂
                              │
                         assign
                              │
                              ▼
                            P₃

    • 사이클 없음 → 데드락 없음

    [데드락 상태]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │                                                                                         │
    │    P₁ ──request──▶ R₁                                                                  │
    │                      │                                                                 │
    │                   assign                                                               │
    │                      │                                                                 │
    │                      ▼                                                                  │
    │    P₂ ◀─assign── R₂ ◀──request─── P₃                                                    │
    │      │                      │                                                           │
    │      │                   request                                                        │
    │      │                      │                                                           │
    │      └────── request ────▶ R₁                                                          │
    │                                                                                         │
    │    사이클: P₂ → R₂ → P₃ → R₁ → P₂ (순환)                                                │
    │    → 데드락 발생                                                                         │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [데드락 판정]
    • 사이클 존재 + 자원이 단일 인스턴스 → 데드락 확정
    • 사이클 존재 + 자원이 다중 인스턴스 → 추가 검사 필요
```

## Ⅲ. 융합 비교 및 다각도 분석

### 데드락 처리 전략 비교

| 전략 | 접근 방식 | 장점 | 단점 | 실무 적용 |
|------|----------|------|------|----------|
| **예방(Prevention)** | 4조건 중 하나 파괴 | 보장적 | 자원 이용률 저하 | 특수 시스템 |
| **회피(Avoidance)** | 안전 상태만 허용 | 유연함 | 계산 비용 | 은행원 알고리즘 |
| **검출(Detection)** | 주기적 검사 | 사후 대응 | 복구 복잡 | 범용 OS |
| **무시(Ignore)** | 타임아웃/재시작 | 간단 | 데이터 손실 가능 | Windows, UNIX |

### 과목 융합 관점 분석

#### 1. 데이터베이스 ↔ 데드락
- **교착 상태**: 트랜잭션 간 데드락
- **희생자 선택(Sacrifice)**: 가장 비용 적은 트랜잭션 롤백
- **Timeout 기반**: 잠금 대기 시간 제한

#### 2. 분산 시스템 ↔ 데드락
- **분산 데드락**: 여러 노드 간 순환 대기
- **2PC**: 분산 트랜잭션 원자성 보장
- **Saga**: 장기 실행 트랜잭션 보상

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 락 순서에 의한 데드락
**상황**: 계좌 이체 시 A→B, B→A 동시 발생
**판단**:
1. **문제**: 순서가 다른 락 획득
   - T1: Lock(A) → Lock(B)
   - T2: Lock(B) → Lock(A)
2. **해결**: 자원 계층 적용
   - 항상 작은 ID 먼저 락
   - Lock(min(A, B)) → Lock(max(A, B))

```java
// 안전한 락 순서
void transfer(Account from, Account to, int amount) {
    Account first = from.getId() < to.getId() ? from : to;
    Account second = from.getId() < to.getId() ? to : from;

    synchronized (first) {
        synchronized (second) {
            from.withdraw(amount);
            to.deposit(amount);
        }
    }
}
```

### 데드락 처리 기대 효과

| 전략 | 기대 효과 | 부작용 |
|------|----------|--------|
| **예방** | 데드락 발생 안 함 | 자원 이용률 저하 |

| **검출** | 발생 시 감지 | 복구 비용 |

3. **난관 불가능(FRESULT)**: 분산 트랜잭션
