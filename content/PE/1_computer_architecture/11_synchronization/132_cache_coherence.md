# 캐시 일관성 (Cache Coherence)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 캐시 일관성은 **"다중 **프로세서**가 **공유 **데이터**를 **캐싱**할 때 **값**의 **일관성**을 **보장**하는 **메커니즘\"**으로, **Write-Invalidate**(쓰기 **무효화)**와 **Write-Update**(쓰기 **갱신)** **프로토콜**이 **있고 **MESI**(Modified, **Exclusive, Shared, **Invalid)** **프로토콜**이 **표준**이다.
> 2. **Bus Snooping**: **버스**를 **감시**하여 **다른 **코어**의 **캐시 **액세스**를 **감지**하고 **무효화/갱신 **신호**를 **보내**며 **Directory-Based**(중앙 **디렉터리)** 방식은 **대규모 **시스템**에서 **확장**된다.
> 3. **상태**: **M**(Modified, **Dirty)**: **단독 **소유**, **E**(Exclusive): **단독 **소유**(Clean), **S**(Shared): **다중 **소유**, **I**(Invalid): **무효**이며 **Cache-to-Cache** 전송으로 **성능**을 **향상**시킨다.

## Ⅰ. 개요 (Context & Background)

### 개념
캐시 일관성은 **"공유 데이터 일관성 보장"**이다.

**일관성 정의**:
| 유형 | 설명 | 강도 |
|------|------|------|
| **Strict** | 모든 코어 동시에 가치함 | 강함 |
| **Sequential** | 프로그램 순서 | 중간 |
| **Weak** | 최종 일관성 | 약함 |

### 💡 비유
캐시 일관성은 ****도서관 **대출 **기록 ****과 같다.
- **공동**: 공유 자산
- **대출**: 캐시
- **갱신**: 반납

## Ⅱ. 아키텍처 및 핵심 원리

### MESI 프로토콜

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         MESI State Machine                                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    State Definitions:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  │  State  │  Valid  │  Dirty  │  Copies  │  Can Read  │  Can Write  │                   │  │
    │  │  ─────────────────────────────────────────────────────────────────────────────────────│  │
    │  │  M      │  Yes   │  Yes   │  1 only │  Yes       │  Yes         │                   │  │
    │  │  E      │  Yes   │  No    │  1 only │  Yes       │  Yes         │                   │  │
    │  │  S      │  Yes   │  No    │  ≥2     │  Yes       │  No          │                   │  │
    │  │  I      │  No    │  No    │  -       │  No        │  No          │                   │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    State Transitions:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  CPU 0 Cache Line: Address A (State: S)                                                │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. CPU 0 Read hit → S (no change)                                                │  │  │
    │  │  2. CPU 0 Write hit → S → M (upgrade to M)                                        │  │  │
    │  │     └─ Bus: Invalidate other copies                                                   │  │  │
    │  │  3. CPU 0 Read miss → I → S (Bus: Read, Shared response)                           │  │  │
    │  │  4. CPU 0 Write miss → I → M (Bus: Read For Ownership, Exclusive response)          │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Bus Snooping Protocol

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Write-Invalidate Bus Transaction                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Scenario: CPU 0 writes to shared cache line
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  CPU 0                 Bus                  CPU 1 (has copy)                               │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Cache Line A: State S (shared)                                                      │  │  │
    │  │  WRITE A ← new_value                                                                  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │            │                                                                             │  │
    │            ▼ Bus Transaction                                                             │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. BusRdX (Read For Ownership)                                                         │  │  │
    │  │     └─ "Who has copy of A? I need exclusive access"                                 │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │            │                                                                             │  │
    │            ▼ CPU 1 responds                                                              │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. Flush cache line A to memory (if dirty)                                          │  │  │
    │  │  2. Change state: S → I                                                                 │  │  │
    │  │  3. Acknowledge on bus                                                                  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │            │                                                                             │  │
    │            ▼ CPU 0 receives ACK                                                        │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. Change state: S → M                                                                │  │  │
    │  │  2. Update cache line A ← new_value                                                  │  │  │
    │  │  3. Future accesses: local (no bus traffic)                                          │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Write-Invalidate vs Write-Update

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Write Propagation Policies                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Write-Invalidate (MESI):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  CPU 0 writes A ← 1 (state: M)                                                           │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Bus: Invalidate A                                                                    │  │  │
    │  │  → CPU 1: A becomes I (next read: miss)                                                │  │  │
    │  │  → CPU 2: A becomes I (next read: miss)                                                │  │  │
    │  │  → Benefit: Single bus transaction                                                   │  │  │
    │  │  → Cost: Other caches miss on next access                                             │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Write-Update (Dragon/MOESI):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  CPU 0 writes A ← 1 (state: M)                                                           │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Bus: Update A ← 1                                                                   │  │  │
    │  │  → CPU 1: A updates to 1 (state: S)                                                   │  │  │
    │  │  → CPU 2: A updates to 1 (state: S)                                                   │  │  │
    │  │  → Benefit: Others have latest data                                                    │  │  │
    │  │  → Cost: Multiple bus transactions, bandwidth                                           │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 캐시 일관성 프로토콜 비교

| 프로토콜 | 상태 | 스토리지 | 버스 트래픽 | 복잡도 |
|----------|------|----------|-------------|--------|
| **MESI** | 4 | Write-back | Invalidate | 중간 |
| **MOESI** | 5 | Write-back | Update | 높음 |
| **Dragon** | 4 | Write-through | Update | 중간 |
| **MSI** | 3 | Write-back | Invalidate | 낮음 |

### Directory-Based Coherence

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Directory-Based Coherence (Scalable)                                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Problem: Bus snooping doesn't scale to 100+ cores
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Solution: Centralized directory tracks cache owners                                          │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Directory Structure:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Address: 0x1000 (Memory Block)                                                          │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Directory Entry:                                                                       │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │
    │  │  │  State: Shared                                                                      │  │  │  │
    │  │  │  Sharers: {CPU 0, CPU 3, CPU 7}                                               │  │  │  │
    │  │  │  Owner: None (no single owner for shared)                                         │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  CPU 0 Write: 0x1000 ← new_value                                                        │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. Send request to Directory                                                          │  │  │
    │  │  2. Directory sends Invalidate to CPU 3, CPU 7                                         │  │  │
    │  │  3. Directory updates state: Shared → Modified, Owner: CPU 0                            │  │  │
    │  │  4. CPU 0 writes locally                                                               │  │  │
    │  │  → Point-to-point messages (no broadcast)                                              │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Cache-to-Cache Transfer

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Cache-to-Cache Transfer (Optimization)                                 │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Scenario: CPU 0 read miss, CPU 1 has dirty copy
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Without C2C:                                                                            │  │
    │  CPU 0: BusRd → Memory ← Data → CPU 0 (slow, ~100 cycles)                                 │  │
    │  CPU 1: (no involvement)                                                                  │  │
    │                                                                                         │  │
    │  With C2C:                                                                               │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  CPU 0: BusRd → CPU 1 responds with data                                            │  │  │
    │  │  CPU 1: M → S, transfers data to CPU 0                                             │  │  │
    │  │  CPU 0: I → S, receives data (fast, ~30 cycles)                                      │  │  │
    │  │  Memory: (not accessed)                                                                 │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 멀티코어 서버 캐시 최적화
**상황**: 32코어 x86 서버
**판단**: NUMA-aware + Cache-friendly

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Multi-Core Cache Optimization                                        │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Considerations:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. False Sharing (보지-shared line)                                                       │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  struct Counter {                                                                     │  │  │
    │  │      int value;                      // 4 bytes                                   │  │  │
    │  │  };                                         // Cache line: 64 bytes                    │  │  │
    │  │                                                                                       │  │  │
    │  │  Counter counters[64];                // 256 bytes = 4 cache lines                  │  │  │
    │  │  → counters[0] and counters[1] on same cache line?                                    │  │  │
    │  │  → Core 0 writes to counters[0], invalidates Core 1's copy                           │  │  │
    │  │  → Cache thrashing!                                                                   │  │  │
    │  │                                                                                       │  │  │
    │  │  Fix: Pad to cache line size                                                           │  │  │
    │  │  struct Counter {                                                                     │  │  │
    │  │      int value;                      // 4 bytes                                   │  │  │
    │  │      char padding[60];               // 60 bytes                                  │  │  │
    │  │  };                                         // 64 bytes total                      │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  2. NUMA Locality                                                                       │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  // Allocate memory on local NUMA node                                              │  │  │
    │  │  void* ptr = numa_alloc_onnode(0, size, 0);  // Node 0                               │  │  │
    │  │                                                                                       │  │  │
    │  │  // First touch policy: thread that allocs also initializes                           │  │  │
    │  │  #pragma omp parallel for schedule(static)                                            │  │
    │  │  for (int i = 0; i < n; i++) {                                                        │  │
    │  │      array[i] = i;   // Each thread touches its own pages                              │  │  │
    │  │  }                                                                                    │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 캐시 일관성 기대 효과

| 프로토콜 | 쓰기 오버헤드 | 읽기 오버헤드 | 공유 데이터 |
|----------|------------|--------------|------------|
| **Write-Invalidate** | 낮음 | 높음 | Temporal locality |
| **Write-Update** | 높음 | 낮음 | Shared-writer |

### 모범 사례

1. **False Sharing**: 패딩
2. **NUMA**: 로컬 메모리
3. **Alignment**: 캐시 라인
4. **Affinity**: 코어 고정

### 미래 전망

1. **Non-Blocking**: 암호학적 프로토콜
2. **Hybrid**: Combine snooping + directory
3. **CXL**: Coherent interconnect
4. **3D Stacking**: HBM 통합

### ※ 참고 표준/가이드
- **Intel**: Cache Coherence
- **AMD: HyperTransport
- **ARM: AMBA ACE
