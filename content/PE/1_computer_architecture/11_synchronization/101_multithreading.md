# 멀티스레딩 (Multithreading)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 멀티스레딩은 프로세서 내부의 자원(연산 유닛 등)이 노는 시간(Stall)을 줄이기 위해, 여러 개의 스레드 컨텍스트를 하드웨어적으로 유지하며 교대로 실행하는 기술이다.
> 2. **가치**: 메모리 접근 지연(Memory Stall)이나 파이프라인 의존성으로 발생하는 대기 시간을 다른 스레드의 연산으로 채워, 프로세서 전체의 처리량(Throughput)을 극대화한다.
> 3. **융합**: TLP(Thread Level Parallelism)를 활용하여 ILP(Instruction Level Parallelism)의 한계를 극복하며, SMT, CMP 아키텍처와 결합하여 현대 CPU 성능의 근간을 이룬다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념
**멀티스레딩(Multithreading)**은 하드웨어적으로 여러 개의 스레드 상태(PC, Register Set 등)를 프로세서 내부에 복제하여, 하나의 물리적 코어가 마치 여러 개의 논리적 코어인 것처럼 동작하게 하는 기술이다. 이는 소프트웨어 레벨의 멀티스레딩과는 달리, 하드웨어 파이프라인 수준에서 스레드 간 전환이 일어나는 것을 의미한다.

#### 💡 비유
멀티스레딩은 **"한 명의 요리사가 여러 개의 가스레인지를 동시에 사용하는 것"**과 같다.
1. **싱글 스레드**: 국이 끓을 때까지(메모리 대기) 요리사가 가만히 기다린다.
2. **멀티 스레드**: 국이 끓는 동안 다른 가스레인지에서 채소를 썬다. 요리사의 손(연산 유닛)은 쉴 틈 없이 움직여 전체 요리 시간(처리량)을 단축한다.

#### 등장 배경 및 발전 과정
1. **메모리 월(Memory Wall) 문제**: CPU 속도는 비약적으로 발전했으나 메모리(DRAM) 속도는 정체되어, CPU가 데이터를 기다리느라 수백 클럭씩 노는 현상이 심화됨.
2. **ILP의 한계**: 수퍼스칼라 기술로 한 스레드 내에서 병렬성을 찾는 것이 복잡해지고 전력 소모가 극심해지자, 차라리 여러 스레드를 섞어서 실행하는 방향으로 선회.
3. **발전 과정**:
    - **1세대 (Coarse-grained)**: 긴 지연 시간 발생 시에만 스레드 전환.
    - **2세대 (Fine-grained)**: 매 클럭마다 스레드를 번갈아 실행.
    - **3세대 (Simultaneous)**: 동일 클럭 내에서 서로 다른 스레드의 명령어를 동시에 실행 (SMT).

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 관련 기술 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **Thread Context** | 스레드의 상태 정보 저장 | PC, SP, Register Set 등을 스레드별로 복제 | PRF (Physical Reg File) | 요리사별 레시피 카드 |
| **Instruction Buffer** | 실행 전 명령어 대기 | 각 스레드에서 인출된 명령어를 별도 보관 | Fetch Queue | 재료 준비대 |
| **Thread Scheduler** | 실행할 스레드 결정 | 우선순위나 라운드 로빈 방식으로 선택 | Dispatch Logic | 요리사의 판단 |
| **Execution Units** | 실제 연산 수행 | ALU, FPU 등을 모든 스레드가 공유 | Shared Resources | 가스레인지, 칼 |
| **ROB / LSQ** | 비순차 실행 및 완료 관리 | 스레드 식별자를 붙여 태깅(Tagging) | Reorder Buffer | 주문서 번호판 |

#### 심층 동작 원리 (3가지 주요 기법)

1. **거친 멀티스레딩 (Coarse-grained Multithreading)**:
    - 하나의 스레드가 실행되다가 캐시 미스 등 **'긴 지연'**이 발생하면 다른 스레드로 교체.
    - 스레드 전환 시 파이프라인을 비워야(Flush) 하므로 짧은 지연에는 비효율적.
2. **세밀한 멀티스레딩 (Fine-grained Multithreading)**:
    - 매 클럭 사이클마다 라운드 로빈 방식으로 스레드를 교체.
    - 한 스레드가 막혀도 다음 클럭에 다른 스레드가 실행되므로 처리량이 일정함.
    - 단일 스레드 입장에서 보면 실행 속도가 느려지는 단점.
3. **동시 멀티스레딩 (Simultaneous Multithreading, SMT)**:
    - 수퍼스칼라의 빈 슬롯(Issue Slot)을 서로 다른 스레드의 명령어로 동시에 채움.
    - 하드웨어 이용률을 극대화하는 가장 진보된 방식 (Intel의 하이퍼스레딩).

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교

| 항목 | 싱글 스레딩 | 멀티스레딩 (FGMT) | SMT (Simultaneous) | CMP (Multi-core) |
| :--- | :--- | :--- | :--- | :--- |
| **자원 공유** | 없음 | 모두 공유 | 모두 공유 | 코어별 전용 자원 |
| **병렬성 종류** | ILP | TLP (시간 분할) | ILP + TLP (공간 분할) | TLP (완전 분리) |
| **하드웨어 비용** | 낮음 | 중간 (레지스터 복제) | 높음 (복잡한 스케줄링) | 매우 높음 (코어 복제) |
| **단일 스레드 성능** | 최고 | 낮음 | 보통 | 최고 |
| **시스템 처리량** | 낮음 | 높음 | 매우 높음 | 최고 |

#### 과목 융합 관점 분석
- **OS 연계**: 운영체제는 하드웨어 스레드를 독립된 CPU(Logical Processor)로 인식합니다. 하지만 두 논리 CPU가 실제로는 캐시와 ALU를 공유하므로, 연산 집약적인 두 작업을 같은 물리 코어의 스레드에 배치하면 성능이 오히려 저하될 수 있는 **'공유 자원 경합'** 문제가 발생합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 기술사적 판단 (실무 시나리오)
**[시나리오]** 데이터베이스(DB) 서버와 과학 연산(HPC) 서버의 멀티스레딩 설정
1. **DB 서버**: I/O 대기 및 메모리 접근이 잦음. 멀티스레딩(SMT)을 활성화하여 대기 시간을 다른 쿼리 처리로 채우는 것이 유리함.
2. **HPC 서버**: 모든 연산 유닛을 풀가동함. SMT 활성 시 스레드 간 ALU 경합으로 인해 오히려 컨텍스트 스위칭 오버헤드만 늘어날 수 있음. 작업 특성에 따라 SMT를 끄는 것이 성능에 더 유리할 수 있음.

#### 도입 시 고려사항 (체크리스트)
- [ ] **거짓 공유(False Sharing)**: 여러 스레드가 같은 캐시 라인을 건드려 불필요한 캐시 일관성 트래픽을 유발하지 않는가?
- [ ] **보안(Side Channel)**: 한 스레드가 공유 캐시를 사용하는 패턴을 통해 다른 스레드의 비밀번호 등을 훔치는 공격(L1TF 등)에 대비가 되었는가?

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량적/정성적 기대효과
- **처리량 향상**: 서버 워크로드에서 약 20~30%의 전체 처리량 향상 기대.
- **에너지 효율**: 코어를 새로 추가하는 것보다 훨씬 적은 전력(약 5% 추가)으로 성능 향상 가능.

#### 미래 전망 및 진화 방향
- **가상화 최적화**: 클라우드 환경에서 가상 머신(VM) 간의 자원 격리를 보장하면서도 멀티스레딩 효율을 높이는 하드웨어 기술 개발 중.
- **가변적 멀티스레딩**: 작업 부하에 따라 하나의 코어를 싱글 스레드 모드(고속) 또는 멀티 스레드 모드(다중 처리)로 동적 전환하는 기술.
