# 순차적 일관성 (Sequential Consistency)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 순차적 일관성은 모든 프로세서의 메모리 연산이 어떤 '하나의 순차적인 순서'로 실행된 것과 같으며, 각 프로세서 내부의 연산들은 프로그램에 명시된 순서를 그대로 유지하는 가장 엄격한 메모리 모델이다.
> 2. **가치**: 프로그래머가 멀티스레드 코드를 작성할 때 가장 직관적으로 이해할 수 있는 동작 방식(프로그램 순서대로 실행됨)을 보장하여, 논리적 오류를 최소화하고 병렬 프로그래밍의 정석적인 기준을 제시한다.
> 3. **융합**: 램포트(Leslie Lamport)에 의해 정의되었으며, 현대의 고성능 CPU는 성능을 위해 이를 완화하지만, 원자적 연산이나 강력한 동기화가 필요한 지점에서는 반드시 이 모델을 모방하도록 설계된다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념
**순차적 일관성(Sequential Consistency, SC)**은 1979년 레슬리 램포트가 정의한 메모리 일관성 모델이다. "어떤 실행 결과가 모든 프로세서의 연산들을 어떤 순서에 따라 실행한 결과와 같고, 개별 프로세서의 연산들이 프로그램에 지정된 순서대로 나타난다면, 그 시스템은 순차적으로 일관되다"라고 정의한다. 즉, 하드웨어가 멋대로 명령어 순서를 바꾸지 않는 **'가장 정직한 모델'**이다.

#### 💡 비유
순차적 일관성은 **"단 한 명의 판사 앞에 줄을 서서 재판을 받는 상황"**과 같다.
1. **개별 순서**: 피고인 A가 "진술서 제출" 후 "증거 제출"을 했다면, 판사 앞에서도 반드시 이 순서대로 처리되어야 한다.
2. **전역 순서**: 여러 피고인(프로세서)이 제출한 서류들은 판사가 한 번에 하나씩만 처리하므로, 전 세계 누구(타 프로세서)가 보더라도 서류가 처리된 순서는 단 하나의 명확한 줄(Global Order)로 보인다.
3. **특징**: 판사가 너무 꼼꼼해서 일이 좀 느리지만(성능 저하), 판결 과정이 매우 투명하고 이해하기 쉽다(예측 가능성).

#### 등장 배경 및 발전 과정
1. **멀티프로세서의 혼란**: 여러 CPU가 동시에 메모리를 읽고 쓸 때, 누가 먼저 썼는지에 따라 결과가 판이하게 달라지는 문제가 발생했다.
2. **직관의 요구**: 개발자들은 "내가 코드를 짠 순서대로 CPU가 실행하겠지"라고 믿는다. 이 믿음을 저버리지 않는 하드웨어 규칙이 필요해졌다.
3. **현재**: 현대 CPU는 성능을 위해 이 모델을 거의 그대로 구현하지 않는다(재정렬 허용). 대신 `Atomic` 변수나 `Memory Fence`를 통해 개발자가 필요할 때만 이 SC 모델을 강제로 적용할 수 있게 한다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 순차적 일관성의 2대 조건 (표)

| 조건명 | 상세 내용 | 하드웨어적 의미 |
| :--- | :--- | :--- |
| **Program Order** | 각 프로세서는 자신의 프로그램을 **코드에 적힌 순서대로** 실행해야 함 | Read/Write 재정렬 금지 |
| **Global Memory Order** | 모든 프로세서의 메모리 접근은 **단일한 전역 순서**를 가져야 함 | 원자적(Atomic) 가시성 보장 |

#### SC 모델의 작동 메커니즘 (ASCII)

```text
    [ Processor P1 ]       [ Processor P2 ]       [ Processor P3 ]
    ┌──────────────┐       ┌──────────────┐       ┌──────────────┐
    │ Op A1 (Write)│       │ Op B1 (Write)│       │ Op C1 (Read) │
    │ Op A2 (Read) │       │ Op B2 (Read) │       │ Op C2 (Write)│
    └──────┬───────┘       └──────┬───────┘       └──────┬───────┘
           │                      │                      │
           └───────────┐          │          ┌───────────┘
                       ▼          ▼          ▼
                ┌──────────────────────────────────┐
                │      Memory Access Switch        │ <── 한 번에 하나씩만 통과
                └────────────────┬─────────────────┘
                                 │
                                 ▼
                        [ Main Memory / Bus ]

    [ 가능한 전역 순서 (Example) ]
    A1 -> B1 -> C1 -> A2 -> B2 -> C2  (O) : 각 프로세서의 순서가 지켜짐
    B1 -> A1 -> A2 -> C1 -> C2 -> B2  (O) : 가능
    A2 -> A1 -> ...                   (X) : P1 내부 순서가 뒤집힘 (불가)
```

#### 심층 동작 원리 (The Atomicity)

1. **Write Atomicity**: 코어 A가 값을 썼다면, 그 즉시 코어 B와 코어 C가 동시에 그 값을 볼 수 있어야 한다. 한 명에겐 보이고 한 명에겐 안 보이는 '중간 상태'는 SC에서 허용되지 않는다.
2. **비순차 실행 금지**: 현대 CPU의 핵심인 Out-of-Order 실행이 원칙적으로 불가능하다. 뒤에 있는 읽기 명령어가 앞의 쓰기 명령어보다 먼저 메모리에 도달할 수 없기 때문이다.
3. **가시성(Visibility)**: 모든 메모리 연산은 즉각적이며, 모든 코어에 동시에 전파되는 것처럼 동작해야 한다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교

| 비교 항목 | 순차적 일관성 (SC) | TSO (Total Store Order) | 약한 일관성 (Weak) |
| :--- | :--- | :--- | :--- |
| **엄격함** | **최상 (가장 엄격)** | 중간 (x86 방식) | 낮음 (ARM 방식) |
| **재정렬 허용** | **전혀 없음** | Store-Load만 허용 | 거의 모든 연산 허용 |
| **프로그래밍** | 매우 쉬움 (직관적) | 보통 | 매우 어려움 (Fence 필수) |
| **성능** | **매우 낮음 (CPU 병목)** | 높음 | **최고 (최적화 극대화)** |
| **가시성** | 즉시 전파 | 약간 지연 가능 | 동기화 지점에서만 전파 |

#### 과목 융합 관점 분석
- **컴퓨터 구조 연계**: SC를 하드웨어로 구현하려면 캐시 일관성 프로토콜이 매우 복잡해지고, 쓰기 버퍼(Write Buffer)를 사용할 수 없어 CPU가 메모리 쓰기가 끝날 때까지 멍하니 기다려야 합니다.
- **분산 시스템 연계**: 분산 데이터베이스의 일관성 모델 중 '강한 일관성(Strong Consistency)'이 바로 이 SC를 네트워크 환경으로 확장한 개념입니다. 모든 복제본이 동일한 순서로 업데이트를 처리해야 함을 의미합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 기술사적 판단 (실무 시나리오)
**[시나리오]** 멀티스레드 환경에서 Dekker의 알고리즘(상호배제) 구현
1. **상황**: 두 스레드가 서로의 플래그를 확인하여 임계 구역에 들어가는 로직.
2. **문제**: SC 모델이 보장되지 않는 현대 CPU(x86, ARM)에서는 내 플래그를 '쓰기' 전에 상대 플래그를 '읽기'가 먼저 일어날 수 있음 (재정렬).
3. **결과**: 두 스레드가 동시에 임계 구역에 진입하는 대참사 발생.
4. **판단**: 하드웨어가 SC가 아니므로, 프로그래머는 반드시 `MFENCE`(x86)나 `DMB`(ARM) 명령어를 사용하여 강제로 SC 환경을 만들어주어야 함.

#### 도입 시 고려사항 (체크리스트)
- [ ] **성능 저하 감수**: SC 모델을 소프트웨어적으로 강제할 때 발생하는 성능 하락(약 5~10배)을 서비스가 감당할 수 있는가?
- [ ] **Lock-free 알고리즘**: 뮤텍스 없이 코딩하려 할 때, 타겟 하드웨어의 메모리 모델이 SC를 어디까지 보장하는지 확인했는가?
- [ ] **컴파일러 최적화**: 하드웨어뿐만 아니라 컴파일러도 순서를 바꿀 수 있음을 인지하고 `volatile`이나 `atomic`을 썼는가?

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량적/정성적 기대효과
- **버그 프리 병렬 코딩**: 실행 순서 뒤바뀜으로 인한 '재현 불가능한 버그(Heisenbug)' 원천 차단.
- **논리적 명확성**: 수천 페이지의 하드웨어 매뉴얼을 읽지 않아도 코드가 의도대로 돌아간다는 확신 제공.

#### 미래 전망 및 진화 방향
- **DRF (Data-Race-Free) 모델**: 모든 상황에서 SC를 보장하는 대신, 데이터 경쟁이 없는 코드에 대해서만 SC를 보장하는 절충형 모델이 C++11, Java 등의 표준 메모리 모델로 채택되었습니다.
- **TSO의 승리**: 성능과 직관의 타협점인 TSO 모델이 x86의 성공과 함께 사실상의 표준이 되었으나, 저전력 ARM의 확산으로 다시금 약한 일관성 모델에 대한 이해가 중요해지고 있습니다.
