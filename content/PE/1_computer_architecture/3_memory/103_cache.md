# 캐시 (Cache)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 캐시는 **"CPU와 주기억장치(RAM) 사이의 **고속 버퍼 메모리"**로, **SRAM(Static RAM)**으로 구현되며 **Temporal Locality(시간적 지역성)**과 **Spatial Locality(공간적 지역성)**을 활용하여 **Cache Hit Ratio(적중률)**를 높이고 **메모리 액세스 지연**을 감소시킨다.
> 2. **가치**: **L1/L2/L3 계층**으로 **CPU 대기 시간**을 최소화하며 **Write-Through(즉시 기록)**와 **Write-Back(지연 기록)** 정책으로 **쓰기 성능**을 최적화하고 **Cache Coherence(캐시 일관성)** 프로토콜로 **다중 프로세서** 환경의 **데이터 정합성**을 유지한다.
> 3. **융합**: **MESI 프로토콜**(Modified, Exclusive, Shared, Invalid)로 **캐시 일관성**을 유지하며, **Virtual Cache**와 **Physical Cache**는 **주소 변환**과 연계되고 **TLB(Translation Lookaside Buffer)**와 협력하여 **효율적인 메모리 시스템**을 구현한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
캐시는 **"자주 사용하는 데이터를 고속 메모리에 저장하여 CPU 대기 시간을 줄이는 기술"**이다.

**캐시의 핵심 원리**:
- **Locality**: 최근/인접 데이터 재사용
- **Speed**: SRAM으로 빠름
- **Size**: 작지만 효율적

### 💡 비유
캐시는 **"책상 위의 책"**과 같다.
- **책상**: L1 캐시 (가장 빠름)
- **옆장**: L2 캐시
- **서가**: RAM

### 캐시 매핑 방식

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         캐시 매핑 방식                                                   │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Direct Mapped Cache (직접 매핑)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 메모리 블록이 캐시의 정해진 라인에만 매핑                                         │
    │                                                                                         │
    │  매핑: Cache Line Index = (Block Number) mod (Number of Cache Lines)                    │
    │                                                                                         │
    │  예: 16KB 캐시, 64B 라인 = 256 라인                                                     │
    │  메모리 블록 0 → 라인 0                                                                  │
    │  메모리 블록 1 → 라인 1                                                                  │
    │  메모리 블록 256 → 라인 0 (Conflict!)                                                  │
    │                                                                                         │
    │  Tag: [Block Number / Number of Cache Lines]                                           │
    │                                                                                         │
    │  장점: 구현 간단, Tag 비교 불필요                                                        │
    │  단점: Conflict Miss 빈번, 낮은 유연성                                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Fully Associative Cache (완전 연관 매핑)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 메모리 블록이 캐시의 모든 라인에 매핑 가능                                           │
    │                                                                                         │
    │  예: 4개 라인 캐시                                                                       │
    │  메모리 블록 0 → 라인 0, 1, 2, 3 (어디든지)                                              │
    │  메모리 블록 1 → 라인 0, 1, 2, 3                                                        │
    │  (가용한 라인 선택: LRU, LFU)                                                           │
    │                                                                                         │
    │  Tag: 전체 Block Number                                                                 │
    │                                                                                         │
    │  장점: Conflict Miss 최소, 유연성 높음                                                   │
    │  단점: 구현 복잡, 모든 라인 병렬 비교, 비용 큼                                            │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Set Associative Cache (세트 연관 매핑)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 캐시를 여러 Set으로 분리, 각 Set은 완전 연관                                         │
    │                                                                                         │
    │  예: 4-Way Set Associative, 256 Sets                                                    │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  Set 0: Line 0, 1, 2, 3 (Way 0, 1, 2, 3)                                           │  │
    │  │  Set 1: Line 4, 5, 6, 7                                                             │  │
    │  │  ...                                                                                 │  │
    │  │  Set 255: Line 1020, 1021, 1022, 1023                                              │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │
    │  매핑: Set Index = (Block Number) mod (Number of Sets)                                  │
    │                                                                                         │
    │  예: 블록 0 → Set 0, Way 0~3 중 선택                                                    │
    │      블록 256 → Set 0, Way 0~3 중 선택                                                   │
    │      블록 512 → Set 0, Way 0~3 중 선택                                                   │
    │                                                                                         │
    │  장점: Direct Mapped보다 낮은 Conflict, 완전 연관보다 저렴                               │
    │  단점: 구현 복잡도 중간                                                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 캐시 읽기/쓰기 정책

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         캐시 정책                                                        │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Read Policy (읽기 정책)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Cache Hit:                                                                             │
    │  • 캐시에서 데이터 반환                                                                 │
    │                                                                                         │
    │  Cache Miss:                                                                            │
    │  • 메모리에서 블록 로드                                                                  │
    │  │  • 필요한 데이터 반환                                                                 │
    │  │  • 캐시에 블록 저장 (Replace policy 적용)                                            │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Write-Through (즉시 기록)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 쓰기 시 캐시와 메모리 동시 업데이트                                                   │
    │                                                                                         │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  CPU Write Request                                                                 │  │
    │  │       │                                                                              │  │
    │  │       ├─────────────→ Cache Block Update                                            │  │
    │  │       │                                                                              │  │
    │  │       └─────────────→ Memory Update (동시)                                         │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │
    │  장점: 일관성 보장 (Cache = Memory)                                                      │
    │  단점: 느린 쓰기 (메모리 대기)                                                           │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Write-Back (지연 기록, Copy-Back)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 쓰기 시 캐시만 업데이트, Dirty Bit 설정                                             │
    │  • 블록 교체 시 메모리에 일괄 기록                                                       │
    │                                                                                         │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  CPU Write Request                                                                 │  │
    │  │       │                                                                              │  │
    │  │       └─────────────→ Cache Block Update + Dirty=1                                  │  │
    │  │                                                                                      │  │
    │  │  [Block Replace 시]                                                                  │  │
    │  │  if Dirty == 1:                                                                      │  │
    │  │      Cache → Memory Write-back                                                       │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │
    │  장점: 빠른 쓰기 (캐시만 업데이트), 메모리 트래픽 감소                                    │
    │  단점: 일관성 복잡, 구현 복잡                                                            │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Write Allocate vs No-Write Allocate]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Write Allocate (Miss 시 할당):                                                         │
    │  • Write Miss 발생 시 메모리 블록을 캐시에 로드 후 업데이트                                │
    │  • Write-Back과 주로 조합                                                                │
    │                                                                                         │
    │  No-Write Allocate (Miss 시 비할당):                                                     │
    │  • Write Miss 발생 시 캐시를 거치지 않고 메모리에 직접 업데이트                            │
    │  • Write-Through과 주로 조합                                                             │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Cache Miss 종류

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         3C Cache Miss                                                 │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. Compulsory Miss (Cold Miss, 강제 미스)                                              │
    │  • 처음 접근하는 데이터 (캐시에 없음)                                                      │
    │  │  • 필연적으로 발생                                                                    │
    │  • 해결: Prefetching, Larger Block Size                                                │
    │                                                                                         │
    │  2. Capacity Miss (용량 미스)                                                            │
    │  • 캐시 용량 초과로 필요한 데이터가 쫓겨남                                               │
    │  │  • Working Set > Cache Size                                                          │
    │  • 해결: Larger Cache                                                                   │
    │                                                                                         │
    │  3. Conflict Miss (충돌 미스)                                                            │
    │  • Direct Mapped에서 여러 블록이 같은 라인 경합                                         │
    │  │  • Set Associative로 감소                                                             │
    │  • 해결: Fully Associative, Victim Cache                                               │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  4. Coherence Miss (일관성 미스, Multi-processor)                                       │
    │  • 다른 코어가 데이터를 수정해서 캐시 무효화                                              │
    │  │  • Invalid Signal 수신                                                                │
    │  • 해결: Coherence Protocol 최적화                                                      │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### L1 vs L2 vs L3 Cache

| 특징 | L1 | L2 | L3 |
|------|-----|-----|-----|
| **크기** | 32-64KB | 256KB-1MB | 8-64MB |
| **지연시간** | 3-4 cycles | 10-12 cycles | 40+ cycles |
| **위치** | Core 내부 | Core 근접 | Core 외부 |
| **공유** | 개별 | 개별 | 모든 Core |

### MESI 프로토콜

| 상태 | 설명 | 다른 Core 읽기 | 다른 Core 쓰기 |
|------|------|--------------|--------------|
| **M** (Modified) | 수정됨, 메모리와 다름 | Flush 후 Shared | 무효화 |
| **E** (Exclusive) | 독점, 메모리와 같음 | Shared | 무효화 |
| **S** (Shared) | 공유, 읽기만 가능 | Shared | 무효화 |
| **I** (Invalid) | 무효, 사용 안 함 | - | - |

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 캐시 최적화
**상황**: Cache Miss 높음
**판단**:

```c
// 캐시 친화적 코드

// 좋은 예: Sequential Access
void sum_array(int* a, int n) {
    int sum = 0;
    for (int i = 0; i < n; i++) {
        sum += a[i];  // Spatial Locality 활용
    }
}

// 나쁜 예: Random Access
void sum_random(int* a, int* idx, int n) {
    int sum = 0;
    for (int i = 0; i < n; i++) {
        sum += a[idx[i]];  // Cache Miss 빈번
    }
}

// 구조체 정렬 (Padding)
struct __attribute__((aligned(64))) CacheLine {
    int data;
    char padding[60];  // False Sharing 방지
};
```

## Ⅴ. 기대효과 및 결론

### 캐시 기대 효과

| 효과 | 캐시 없을 시 | 캐시 있을 시 |
