# SMP (Symmetric Multiprocessing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SMP는 여러 개의 동등한(Equivalent) 프로세서가 단일 공유 메모리에 대칭적으로 접근하는 병렬 컴퓨팅 아키텍처로, 모든 CPU가 메모리와 I/O에 동일한 권한과 지연 시간으로 접근하는 것이 핵심 특징이다
> 2. **가치**: 단일 OS 이미지로 다중 CPU를 관리하여 프로그래밍 모델이 단순하고, 스레드 레벨 병렬성(TLP)을 활용해 서버 워크로드 처리량을 선형적으로 확장할 수 있다
> 3. **융합**: NUMA, 멀티코어, 클러스터와 비교되는 메모리 일관성 모델이며, OS 스케줄링, 캐시 일관성 프로토콜(MESI), 인터커넥트와 밀접하게 연관된다

---

## Ⅰ. 개요 (Context & Background)

### 개념
SMP(Symmetric Multiprocessing)는 다수의 프로세서가 하나의 공유 메모리 시스템에 연결되어, 모든 프로세서가 메모리의 모든 영역에 동일한 접근 권한과 균일한 지연 시간(Uniform Memory Access, UMA)을 가지는 병렬 컴퓨팅 아키텍처다. "대칭적(Symmetric)"이라는 용어는 모든 CPU가 동등한 지위를 가지며, 어떤 CPU도 특별한 권한을 가지지 않음을 의미한다. 단일 운영체제가 모든 CPU를 관리하며, 프로세스와 스레드를 여러 CPU에 분산하여 실행함으로써 시스템 전체 처리량(Throughput)을 향상시킨다.

### 💡 비유
SMP는 **"여러 점원이 동일한 창고를 공유하는 편의점"**에 비유할 수 있다.
- **단일 점원 (단일 CPU)**: 한 명이 모든 일을 처리, 대기열 길어짐
- **SMP (다수 점원 + 공유 창고)**:
  - 여러 점원이 동시에 다른 손님 응대
  - 모든 점원이 창고 어디든 동일하게 접근 가능
  - "창고 재고" = 공유 메모리, "점원 간 소통" = 캐시 일관성
- **비대칭 (ASMP)**: 점장님만 창고 키를 가지고 있음 (Master-Slave)

### 등장 배경 및 발전 과정

#### 1. 기존 기술의 치명적 한계점
- **단일 CPU의 성능 한계**: 클럭 속도 증가 둔화, ILP 추출 한계
- **멀티태스킹 병목**: 단일 CPU에서 컨텍스트 스위칭 오버헤드 증가
- **서버 워크로드 증가**: 다수의 동시 요청 처리를 위한 처리량 필요

#### 2. 패러다임 변화와 혁신
- **1960년대 Burroughs B5000**: 초기 SMP 개념 도입
- **1980년대 Sequent, Encore**: 상용 SMP 서버 출시
- **1990년대**: Intel Pentium Pro 듀얼 프로세서, Sun SPARCserver
- **2000년대**: 멀티코어로 진화 (SMP 개념이 칩 내부로)
- **현재**: NUMA 기반 대규모 SMP 서버, 클라우드 인스턴스

#### 3. 비즈니스적 요구사항
- **데이터베이스 서버**: 높은 TPS 달성을 위한 다중 CPU 활용
- **웹 서버**: 동시 연결 처리를 위한 스레드 풀
- **가상화**: VM마다 vCPU 할당, 물리 CPU 공유

```
             │
        Shared Bus
             │
        ┌────┴────┐
        │  DRAM   │  ← 모든 CPU가 동일한 지연 시간
        │ (Uniform│     메모리 접근

        └─────────┘


  [NUMA - Non-Uniform Memory Access]

   CPU0      CPU1      CPU2      CPU3
    │         │         │         │
   L3$      L3$       L3$       L3$
    │         │         │         │
   DRAM0    DRAM1     DRAM2     DRAM3   ← 각 CPU에 Local Memory
    │         │         │         │

   CPU0 → DRAM2: 느림 (Remote)

```

### 심층 동작 원리

#### 1. 캐시 일관성 (Cache Coherence) - MESI 프로토콜

**상태 정의**:
| 상태 | 의미 | 설명 |
|------|------|------|
| **M** (Modified) | 수정됨 | 이 캐시만이 유효한 데이터 가짐, 메모리와 다름 |
| **E** (Exclusive) | 배타적 | 이 캐시만이 데이터 가짐, 메모리와 동일함 |
| **S** (Shared) | 공유 | 여러 캐시가 데이터 가짐, 메모리와 동일함 |
| **I** (Invalid) | 무효 | 데이터가 유효하지 않음 |

**상태 전이 다이어그램**:
```
                        Local Read
                    ┌──────────────┐
                    │              │
                    ▼              │
         ┌─────────────────┐      │
         │                 │      │
         │   EXCLUSIVE     │◄─────┤
         │                 │      │
         └────────┬────────┘      │
                  │               │

         └────────┬────────┘      │
                  │               │

         │                 │

         │                 │
         └─────────────────┘
```

#### 2. OS 스케줄링과 SMP

**프로세스 스케줄링**:
1. **Run Queue**: CPU별 로컬 큐 또는 글로벌 큐
2. **Load Balancing**: CPU 간 부하 분산
3. **Processor Affinity**: 캐시 지역성 유지를 위한 CPU 고정
4. **Migration**: 필요시 프로세스를 다른 CPU로 이동

**스핀락(Spinlock)과 SMP**:
- 락 획득 시도 중 대기 (Busy Waiting)
- 짧은 임계 영역에 적합
- 캐시 라인 바운싱 문제 발생 가능

#### 3. 인터럽트 분배 (IRQ Balancing)

**인터럽트 처리**:
- I/O APIC가 인터럽트를 특정 CPU로 라우팅
- 인터럽트 스톰(Storm) 방지를 위한 분산
- CPU별 인터럽트 부하 모니터링

### 성능 특성

```
SMP 확장성 (Scalability):

이상적: 성능 = N × 단일_CPU_성능
실제:   성능 = N × 단일_CPU_성능 × (1 - 오버헤드(N))

오버헤드 요인:
1. 캐시 일관성 트래픽 (Coherence Traffic)
2. 버스/인터커넥트 경합 (Bus Contention)
3. 락 경합 (Lock Contention)
4. 메모리 대역폭 경합 (Memory Bandwidth Contention)

일반적 확장성:
- 2-8 CPU: 거의 선형 확장 (90-95%)
- 8-16 CPU: 완만한 감소 (80-90%)
- 16+ CPU: NUMA 권장
```

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 기술 비교표

| 특성 | SMP (UMA) | NUMA | AMP | 클러스터 |
|------|-----------|------|-----|----------|
| **메모리 접근** | 균일 | 비균일 | 비대칭 | 분산 |
| **OS** | 단일 | 단일 | 단일/다중 | 다중 |
| **CPU 권한** | 대등 | 대등 | 불평등 | 독립 |
| **확장성** | 2-16 | 16-1024+ | 제한적 | 무제한 |
| **복잡도** | 낮음 | 중간 | 낮음 | 높음 |
| **대표 예시** | 듀얼코어 PC | 서버급 | 임베디드 | HPC 클러스터 |

### SMP vs 멀티코어

| 측면 | 기존 SMP | 멀티코어 |
|------|----------|----------|
| **CPU 위치** | 별도 칩 | 동일 칩 |
| **인터커넥트** | 외부 버스 | 온칩 NoC |
| **지연 시간** | 상대적 높음 | 매우 낮음 |
| **L3 캐시** | 분리 또는 외부 | 공유 |
| **전력 효율** | 낮음 | 높음 |

### 과목 융합 관점 분석

#### 1. OS ↔ SMP: 스케줄링 최적화

- **SMP 스케줄러**: CPU 간 부하 분산
- **NUMA 인식 스케줄링**: Local Memory 우선 사용
- **Real-time SMP**: 우선순위 역전 방지

#### 2. 데이터베이스 ↔ SMP: 동시성 제어

- **락 매니저**: SMP 환경에서의 락 경합
- **Latch**: 메모리 구조 보호를 위한 스핀락
- **Parallel Query**: 다중 CPU 활용 쿼리 실행

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오 1: 데이터베이스 서버 CPU 선택
**상황**: Oracle/MySQL 서버용 하드웨어 선택
**판단**:
- OLTP: 높은 단일 스레드 성능 + 8-16코어 SMP
- OLAP: 높은 병렬성 + NUMA 대형 서버
- 메모리 대역폭이 병목이 될 수 있음

#### 시나리오 2: 가상화 호스트 설계
**상황**: VMware/KVM 호스트 구성
**판단**:
- vCPU 대 물리코어 비율: 2:1 ~ 4:1
- NUMA 노드 경계 고려한 VM 배치
- CPU Ready Time 모니터링

### 도입 시 고려사항 (체크리스트)

**기술적**:
- [ ] 워크로드 병렬화 가능성: 스레드 레벨 병렬성이 충분한가?
- [ ] 메모리 대역폭: CPU 수에 비례하여 대역폭 충분한가?
- [ ] 캐시 일관성 오버헤드: 스누핑 vs 디렉터리 기반

**운영적**:
- [ ] OS SMP 지원: 커널 SMP 최적화 여부
- [ ] 애플리케이션 스레딩: 멀티스레드 프로그래밍 적용 여부
- [ ] 모니터링: CPU별 부하, 캐시 미스, 락 대기 시간

### 주의사항 및 안티패턴

1. **False Sharing**: 다른 CPU가 같은 캐시 라인의 다른 변수 수정
2. **Lock Convoy**: 락 획득 실패로 인한 대기 행렬
3. **Cache Thrashing**: 과도한 캐시 일관성 트래픽

## Ⅴ. 기대효과 및 결론

### 정량적/정성적 기대효과

| 지표 | 단일 CPU | SMP (4-CPU) | SMP (8-CPU) |
|------|----------|-------------|-------------|
| 처리량 | 100% | 350-380% | 650-700% |
| 응답 시간 | 기준 | 유사/향상 | 유사/향상 |
| 동시 사용자 | 1x | 3-4x | 6-7x |
| 비용/성능비 | 기준 | 80-90% | 70-80% |

### 미래 전망 및 진화 방향

1. **칩레벨 SMP**: 멀티코어 = 칩 내부 SMP
2. **NUMA 일반화**: 대형 서버는 대부분 NUMA
3. **하이브리드**: SMP + 가속기 (GPU, NPU)

### ※ 참고 표준/가이드
- **Intel Multiprocessor Specification (MPS)**: x86 SMP 표준
- **ACPI**: SMP 구성 정보 제공
- **UEFI**: 멀티프로세서 부팅 지원

## 👶 어린이를 위한 3줄 비유 설명

SMP는 **"여러 점원이 같은 창고를 공유하는 편의점"**이에요.

1. **똑같은 권한**: 모든 점원이 창고 어디든 자유롭게 들어갈 수 있어요. 특별한 점원은 없어요.

2. **함께 일하기**: 한 점원이 손님을 도울 때, 다른 점원도 다른 손님을 도울 수 있어요. 손님이 많아도 빨리 처리돼요!

3. **창고 재고 확인**: 점원끼리 "이거 팔았어!" 하고 서로 알려줘야 해요. 안 그러면 다른 점원이 없는 물건을 찾을 수도 있거든요.
