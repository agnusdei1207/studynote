# 수퍼스칼라 (Superscalar)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 수퍼스칼라는 단일 클럭 사이클에 여러 명령어를 병렬로 인출(Fetch), 해독(Decode), 실행(Issue)할 수 있는 프로세서 아키텍처로, IPC(Instructions Per Cycle) > 1을 달성하는 것이 핵심 목표다
> 2. **가치**: 수퍼스칼라 설계는 클럭 속도 증가 없이도 성능을 2-8배 향상시킬 수 있으며, 비순차 실행(OoO), 레지스터 리네이밍, 분기 예측과 결합하여 현대 CPU의 성능 극대화를 가능하게 한다
> 3. **융합**: 파이프라이닝의 확장이며, 멀티코어, 하이퍼스레딩(SMT), VLIW와 비교되는 병렬 처리 기법으로, ILP(Instruction Level Parallelism)를 추출하는 하드웨어 방식이다

---

## Ⅰ. 개요 (Context & Background)

### 개념
수퍼스칼라(Superscalar) 프로세서는 하나의 클럭 사이클 동안 복수의 명령어를 처리할 수 있는 마이크로아키텍처 설계 방식이다. 스칼라(Scalar) 프로세서가 매 사이클마다 최대 1개의 명령어만 완료할 수 있는 반면, 수퍼스칼라는 Issue Width(발급 폭)라고 불리는 병렬 처리 능력을 통해 이론적으로 2개, 4개, 또는 그 이상의 명령어를 동시에 처리한다. 이를 위해 CPU 내부에 다중 파이프라인, 병렬 ALU, 다중 포트 레지스터 파일, 정교한 스케줄러 등의 하드웨어 자원을 갖추어야 하며, 명령어 간의 의존성을 분석하고 해결하는 로직이 필수적이다.

### 💡 비유
수퍼스칼라는 **"다중 계산대가 있는 슈퍼마켓"**에 비유할 수 있다.
- **스칼라 (1개 계산대)**: 한 번에 한 명의 손님만 계산 처리, 줄이 길어짐
- **수퍼스칼라 (다중 계산대)**: 여러 계산대에서 동시에 여러 손님 처리, 처리량 증가
- **Issue Width**: 계산대의 개수 (2-way = 2개 계산대, 4-way = 4개)
- **의존성 처리**: 앞 손님의 물건이 뒷 손님의 물건과 연관될 때 대기 발생

### 등장 배경 및 발전 과정

#### 1. 기존 기술의 치명적 한계점
- **스칼라 파이프라인의 IPC 한계**: 기본 파이프라인은 이상적으로 CPI=1이지만, 해저드(데이터, 제어, 구조)로 인해 실제 CPI > 1
- **클럭 속도의 물리적 한계**: 1990년대 후반 클럭 속도 증가가 전력 소모(P ∝ f³)로 인해 둔화되기 시작
- **ILP(Instruction Level Parallelism) 미활용**: 프로그램 내 존재하는 병렬성을 단일 파이프라인으로는 추출 불가

#### 2. 패러다임 변화와 혁신
- **1987년 Intel i960CA**: 최초의 상용 수퍼스칼라 프로세서 (2-way Issue)
- **1993년 Intel Pentium**: 2-way 수퍼스칼라, U/V 파이프라인 도입
- **1995년 Intel Pentium Pro**: 3-way 수퍼스칼라 + 비순차 실행(OoO) + 레지스터 리네이밍
- **2000년대**: 4-way, 6-way 수퍼스칼라 보편화 (Intel Core, AMD K8)
- **현재**: Apple M3 (8-wide decode), Intel Raptor Lake (6-wide decode)

#### 3. 비즈니스적 요구사항
- **서버 워크로드 처리**: 클라우드 데이터센터에서 단일 스레드 성능이 여전히 중요
- **모바일 전력 효율**: 높은 IPC로 낮은 클럭에서도 성능 확보, 배터리 수명 연장
- **실시간 처리**: 자율주행, 게임에서 프레임 드랍 방지를 위한 일관된 IPC 보장

### 심층 동작 원리

#### 1. Front-End: 명령어 인출 및 해독

**Fetch Stage (4-wide)**:
1. **PC 기반 인출**: Program Counter에서 I-Cache의 4개 명령어 위치 계산
2. **Branch Prediction**: BTB(Branch Target Buffer)에서 분기 예측
3. **Pre-decode**: 명령어 길이 파악 (x86의 가변 길이 명령어 처리)
4. **Cache Line Split**: 캐시 라인 경계를 넘는 명령어 처리

**Decode Stage**:
- **Simple Decoders**: 1:1 매핑 (1 instruction → 1 uop)
- **Complex Decoder**: 1:N 매핑 (1 instruction → multiple uops, 예: string ops)
- **Micro-op Cache (DSB)**: 자주 사용되는 uop 시퀀스를 캐싱하여 decode 스킵

#### 2. Register Renaming: WAW/WAR 해결

**동작 원리**:
```
Original Code (WAW Hazard):
  ADD R1, R2, R3   ; R1 = R2 + R3
  SUB R1, R4, R5   ; R1 = R4 - R5  (WAW: 이전 R1 쓰기 전에 다시 쓰기)
  MUL R6, R1, R7   ; R6 = R1 × R7

After Renaming:
  ADD P10, P2, P3  ; R1 → P10 매핑
  SUB P11, P4, P5  ; R1 → P11 매핑 (다른 물리 레지스터!)
  MUL P6, P11, P7  ; P11 사용 (최신 값)

결과: WAW/WAR 해제, 병렬 실행 가능
```

#### 3. Issue Queue & Scheduling: Wake-up & Select

**Tomasulo 알고리즘 기반 스케줄링**:
1. **Reservation Station 대기**: uop이 소스 오퍼랜드 준비될 때까지 대기
2. **Wake-up**: 결과 생성 시 대기 중인 uop들에게 브로드캐스트
3. **Select**: 준비된 uop 중 우선순위에 따라 실행 유닛에 발급
4. **Forwarding**: 결과를 바로 다음 uop에 전달

#### 4. Execution Units: 병렬 실행

**Integer Execution**:
- 4개 ALU: 단순 산술/논리 연산 (1 cycle latency)
- 1개 MUL/DIV: 복잡한 곱셈/나눗셈 (3-30 cycles)

**Floating-Point Execution**:
- FP Adder, FP Multiplier, FP Divider
- FMA (Fused Multiply-Add): `a × b + c`를 1 operation으로

**Load/Store Unit**:
- Address Generation Unit (AGU): 메모리 주소 계산
- Store-to-Load Forwarding: Store가 완료되기 전에 Load에 값 전달

#### 5. Reorder Buffer (ROB): In-Order Commit

**순차적 완료 보장**:
1. **Allocation**: uop 발급 시 ROB 엔트리 할당
2. **Execution**: 비순차로 실행되지만 ROB 순서 유지
3. **Completion**: 실행 완료 시 ROB에 결과 기록
4. **Commit**: ROB Head에서 순차적으로 아키텍처 상태 반영

### IPC 계산식

```
IPC = (# of Committed Instructions) / (# of Clock Cycles)

이상적 IPC = Issue Width (예: 4-way → IPC = 4)

실제 IPC = Issue Width × (1 - Stall_Rate)
         = Issue Width × (1 - Branch_Misprediction_Rate × Penalty)
         × (1 - Cache_Miss_Rate × Miss_Penalty / Bus_Width)

일반적 실제 IPC: 1.5 ~ 2.5 (4-way 수퍼스칼라)
```

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 기술 비교표

| 특성 | 스칼라 | 수퍼스칼라 | VLIW | EPIC |
|------|--------|-----------|------|------|
| **IPC** | ≤ 1 | 2-6 | 4-8 (이론적) | 6+ |
| **하드웨어 복잡도** | 낮음 | 높음 | 낮음 | 중간 |
| **컴파일러 의존도** | 낮음 | 낮음 | 매우 높음 | 높음 |
| **전력 효율** | 좋음 | 중간 | 좋음 | 중간 |
| **코드 밀도** | 좋음 | 좋음 | 나쁨 | 나쁨 |
| **대표 CPU** | 8086 | Core i9 | TI DSP | Itanium |

### 수퍼스칼라 vs VLIW 비교

| 비교 항목 | 수퍼스칼라 (Superscalar) | VLIW (Very Long Instruction Word) |
|-----------|--------------------------|-----------------------------------|
| **스케줄링 주체** | 하드웨어 (동적) | 컴파일러 (정적) |
| **명령어 형식** | 기존 ISA 유지 | 여러 연산을 묶은 긴 명령어 |
| **하드웨어 복잡도** | 높음 (스케줄러 필요) | 낮음 (단순 실행) |
| **전력 소모** | 높음 | 낮음 |
| **이진 호환성** | 보존됨 | 컴파일 시 고정 |
| **적합한 용도** | 범용 CPU | 임베디드, DSP, GPU |

### 과목 융합 관점 분석

#### 1. 컴퓨터구조 ↔ 운영체제: 스케줄링과의 관계

- **OS 스케줄러**: 프로세스 레벨 병렬성 활용 (TLP)
- **수퍼스칼라**: 명령어 레벨 병렬성 활용 (ILP)
- **SMT/Hyper-threading**: 수퍼스칼라 자원을 여러 스레드가 공유

#### 2. 컴파일러 ↔ 수퍼스칼라

- **Instruction Scheduling**: 컴파일러가 의존성을 줄이도록 명령어 재배치
- **Loop Unrolling**: 루프를 펼쳐 ILP 증가
- **Software Prefetching**: 캐시 미스 숨기기

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오 1: Issue Width 선택
**상황**: 새로운 CPU 설계에서 Issue Width 결정
**분석**:
- 2-way: 저전력 모바일, 간단한 설계
- 4-way: 범용 데스크톱/서버
- 6-8-way: 고성능 서버, HPC

**판단**:
- Issue Width 증가 시 Front-End 복잡도 기하급수적 증가
- 실제 ILP는 평균 2-3 수준 (Amdahl's Law)
- 4-6 wide가 비용 대비 효율 최적점

#### 시나리오 2: 비순차 실행 윈도우 크기
**상황**: ROB 크기 결정 (64 vs 128 vs 224)
**판단**:
- 큰 ROB = 더 많은 메모리 레이턴시 숨기기 가능
- 하지만 전력, 면적 비용 증가
- 메모리 지연이 큰 워크로드에서는 큰 ROB 유리

### 도입 시 고려사항 (체크리스트)

**기술적**:
- [ ] 워크로드 ILP 분석: 실제 병렬성이 얼마나 되는가?
- [ ] 전력 예산: 수퍼스칼라 하드웨어의 전력 소모
- [ ] 메모리 계층: 캐시 미스가 수퍼스칼라 효과를 상쇄하지 않는가?

**운영적**:
- [ ] 성능 모니터링: IPC, Stall Cycles, Branch Misprediction Rate
- [ ] 전력 관리: Issue Width 동적 조절 가능 여부

### 주의사항 및 안티패턴

1. **ILP Wall**: 일반 프로그램의 ILP는 2-3 정도로 제한적

3. **Memory Wall**: 수퍼스칼라가 메모리 병목을 해결하지 못함

## 👶 어린이를 위한 3줄 비유 설명

수퍼스칼라는 **"여러 명이 함께 일하는 작업반"**이에요.

1. **혼자 하는 일**: 보통은 한 명이 한 번에 하나씩만 일할 수 있어요. 마치 한 손으로만 일하는 것 같아요.

2. **여러 명이 함께**: 수퍼스칼라는 4명, 6명이 동시에 각자 다른 일을 해요. 한 명은 더하기, 한 명은 곱하기, 한 명은 가져오기...

3. **빨라지는 작업**: 서로 방해하지 않고 동시에 하니까, 일이 2배, 3배 빨리 끝나요! 하지만 너무 많으면 서로 부딪칠 수도 있어요.
