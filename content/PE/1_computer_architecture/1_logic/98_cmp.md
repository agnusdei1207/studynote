# CMP (Chip Multi-Processor)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CMP는 단일 실리콘 다이(Die) 내에 여러 개의 프로세서 코어를 집적하는 아키텍처로, 멀티코어와 동일한 개념이지만 설계 관점에서 "다중 프로세서를 단일 칩에 구현"한다는 점을 강조한다
> 2. **가치**: 칩 면적과 전력 대비 성능 효율을 극대화하며, 기존 멀티칩 SMP 대비 코어 간 통신 지연이 1/10 수준으로 낮아 병렬 프로그램 성능이 크게 향상된다
> 3. **융합**: SMP의 칩 내부 구현이며, SMT와 결합 가능하고, 칩렛(Chiplet), 3D-IC로 진화하여 HPC 및 클라우드 서버의 핵심 기술이 되었다

---

## Ⅰ. 개요 (Context & Background)

### 개념
CMP(Chip Multi-Processor)는 하나의 반도체 칩 위에 두 개 이상의 독립적인 프로세서 코어를 통합하는 마이크로프로세서 설계 방식이다. 기존의 멀티프로세서 시스템이 여러 개의 CPU 칩을 메인보드에 장착하는 방식이었다면, CMP는 이를 단일 칩 내부에 구현한다. 이는 "Multi-core"와 본질적으로 동일한 개념이지만, CMP라는 용어는 학술적/기술적으로 "칩 레벨 멀티프로세싱"을 강조할 때 주로 사용된다.

### 💡 비유
CMP는 **"하나의 공장 건물 안에 여러 생산라인을 설치한 것"**에 비유할 수 있다.
- **기존 SMP (여러 공장)**: 각 공장이 따로 떨어져 있어 자재 이동에 시간이 걸림
- **CMP (단일 공장)**:
  - 여러 생산라인(코어)이 한 건물 안에
  - 부품 창고(캐시)를 공유하여 이동 시간 단축
  - 관리자가 한 명(OS)이 모든 라인 관리

### 등장 배경 및 발전 과정

#### 1. 기존 기술의 치명적 한계점
- **단일 코어 성능 한계**: 클럭 3GHz 이상에서 전력/발열 폭증
- **멀티칩 SMP의 병목**: 칩 간 통신 지연이 수십 ns, 대역폭 제한
- **칩 간 I/O 오버헤드**: 신호 전송 전력, 패키징 비용 증가

#### 2. 패러다임 변화와 혁신
- **2000년대 초**: IBM POWER4, Sun MAJC - 초기 CMP
- **2005년**: Intel Core Duo, AMD Athlon 64 X2 - x86 CMP 대중화
- **2010년대**: 4-8코어 CMP 일반화
- **2020년대**: 64-128코어 CMP (AMD EPYC, Intel Xeon)

#### 3. 비즈니스적 요구사항
- **데이터센터 밀도**: 동일 전력으로 더 많은 처리량
- **모바일 전력**: CMP로 코어당 클럭 낮춰 전력 절약
- **실시간 처리**: 낮은 코어 간 지연으로 빠른 동기화

   장점: 코어 독립성 높음                    장점: 캐시 용량 효율적 사용

   예: Intel Core, AMD Ryzen                예: Intel Core 2 Duo

### 심층 동작 원리

#### 1. 코어 간 통신 (Inter-Core Communication)

**공유 메모리 방식**:
- 메모리를 통한 데이터 교환
- 캐시 일관성 프로토콜로 동기화
- 가장 일반적인 방식

**메시지 패싱 (온칩)**:
- Mailbox 레지스터
- 인터럽트 기반 알림
- 낮은 지연 시간

#### 2. 캐시 일관성 (Coherence)

**디렉터리 기반**:
- 각 캐시 라인의 소유자 추적
- 스누핑 대비 확장성 우수
- 대규모 CMP에 적합

**스누핑 기반**:
- 모든 트랜잭션 감시
- 구현 단순
- 소규모 CMP에 적합

#### 3. 메모리 일관성 (Consistency)

| 모델 | 설명 | 성능 |
|------|------|------|
| SC (Sequential) | 모든 코어가 동일 순서로 접근 관찰 | 낮음 |
| TSO (Total Store) | Store 순서만 유지 | 중간 |
| Weak | 동기화 명령어로만 순서 보장 | 높음 |

### CMP 성능 모델

```
CMP 성능 = N × 단일코어성능 × (1 - 오버헤드(N))

오버헤드 요인:
1. 캐시 경합 (Cache Contention)
   - 공유 캐시 용량 부족
   - 캐시 라인 경합

2. 메모리 대역폭 (Memory Bandwidth)
   - 코어 수 대비 대역폭 부족
   - 백엔드 병목

3. 동기화 (Synchronization)
   - 락 경합
   - 캐시 일관성 트래픽

일반적 확장성:
- 2-4코어: 90-95% 효율
- 8-16코어: 80-90% 효율
- 32+코어: NUMA 또는 칩렛 권장
```

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 기술 비교표

| 특성 | 단일코어 | CMP | 멀티칩 SMP |
|------|----------|-----|-----------|
| **코어 간 지연** | N/A | 5-10ns | 50-100ns |
| **캐시 공유** | N/A | L2/L3 | 불가능 |
| **면적 효율** | 낮음 | 높음 | 낮음 |
| **전력 효율** | 낮음 | 높음 | 낮음 |
| **확장성** | N/A | 32코어 정도 | 8-16개 |

### CMP + SMT 조합

| 구성 | 성능 | 전력 | 면적 |
|------|------|------|------|
| 4코어 | 4x | 4x | 4x |
| 2코어 + SMT | 2.4x | 2.1x | 2.1x |
| 4코어 + SMT | 4.8x | 4.2x | 4x |

### 과목 융합 관점

#### 1. OS ↔ CMP: 스케줄러 최적화

- **Load Balancing**: CMP 내 코어 간 부하 분산
- **Cache Affinity**: 동일 캐시 공유 코어 우선 할당
- **Power Gating**: 유휴 코어 전원 차단

#### 2. DB ↔ CMP: 병렬 쿼리

- **Parallel Execution**: 다중 코어 활용
- **Partitioning**: 코어별 데이터 파티션
- **Latch Management**: 캐시 라인 의식 설계

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단

#### 시나리오: CMP vs 칩렷 선택
**상황**: 고성능 서버 CPU 설계
**판단**:
- 32코어 이하: 단일 다이 CMP
- 64코어 이상: 칩렷(Chiplet) 방식 권장
- 수율과 비용 고려

### 도입 시 고려사항

- [ ] 워크로드 병렬성: TLP 활용도
- [ ] 캐시 구성: Private vs Shared
- [ ] 전력 예산: 코어 수와 클럭 균형

### 주의사항

1. **캐시 Thrashing**: 과도한 캐시 경합
2. **메모리 병목**: 코어 수 > 메모리 채널
3. **확장성 한계**: 단일 다이 CMP의 한계

## 👶 어린이를 위한 3줄 비유 설명

CMP는 **"한 건물에 여러 작업실이 있는 공장"**이에요.

1. **한 곳에 모아서**: 여러 작업실(코어)이 하나의 건물(칩) 안에 있어요. 서로 멀리 떨어져 있지 않아요.

2. **빠른 소통**: 작업실끼리 가까워서 "이거 좀 줘!" 하면 바로 줄 수 있어요. 멀리서 오라고 안 해도 돼요.

3. **창고 공유**: 큰 창고(캐시)를 같이 써서 자꾸 밖에서 가져오지 않아도 돼요. 더 빠르고 효율적이에요!
