# NUMA (Non-Uniform Memory Access)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NUMA는 **"각 프로세서가 **독립적인 **로컬 메모리**를 가지며 **다른 프로세서의 **메모리에 **접근할 때 **더 오래 걸리는 **아키텍처"**로, **SMP의 **확장성 문제**(Memory Bottleneck)를 **해결**하고 **Large-Scale Multi-Processor**에서 **확장성**을 **향상**한다.
> 2. **가치**: **메모리 병목**을 **완화**하고 **Scalability**를 **향상**하지만 **Remote Memory Access**가 **느려** **OS의 **NUMA-Aware Scheduling**이 **필수적**이며 **CPU affinity**, **Memory locality**를 **고려**한 **할당**이 **중요**하다.
> 3. **융합**: **AMD Opteron**, **Intel Xeon**(Multi-Chip Module), **ARM**(big.LITTLE + NUMA)에서 **구현**되며 **Linux**(NUMA Balancing), **Windows**(NUMA-aware Scheduler)가 **최적화**되어 있다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
NUMA는 **"메모리 접근 시간이 메모리 위치에 따라 다른 아키텍처"**이다.

**NUMA 특징**:
- **Local Memory**: 빠름 (자신 CPU의 메모리)
- **Remote Memory**: 느림 (다른 CPU의 메모리)
- **Node**: CPU + Local Memory

### 💡 비유
NUMA는 **"여러 사무실**"과 같다.
- **로컬**: 내 책상 (빠름)
- **원격**: 다른 사무실 책상 (느림)

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         SMP → NUMA                                                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘

SMP (Shared Bus):
    • 모든 CPU가 동일한 Bus로 메모리 접근
    • 확장성 한계 (Bus 병목)
         ↓
CC-NUMA (Cache-Coherent NUMA):
    • 각 Node에 독립 메모리
    • Interconnect 연결
         ↓
현대 x86:
    • Intel Xeon (Multi-Chip Module)
    • AMD EPYC (Chiplet 기반)
```

### NUMA 성능

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         NUMA 성능                                                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Access Time]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Local Memory: ~100ns                                                                 │
    │  │  • Remote Memory: ~200ns (2배 느림)                                                       │
    │  │  • NUMA Ratio: Remote/Local = 2                                                          │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────────┘

    [NUMA Hit/Miss]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • NUMA Hit: Local Memory 접근                                                            │  │
    │  │  • NUMA Miss: Remote Memory 접근                                                           │  │
    │  │                                                                                      │
    │  성능 최적화:                                                                             │
    │  • 데이터 Local Memory에 할당                                                           │  │
    │  • CPU affinity: Thread가 고정 Core에서 실행                                                   │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### UMA vs NUMA

| 구분 | UMA | NUMA |
|------|-----|------|
| **메모리** | 공유 | 분리 |
| **접근 시간** | 균등 | 불균등 |
| **확장성** | 제한 | 높음 |
| **복잡도** | 낮음 | 높음 |

### NUMA-Aware OS

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         NUMA-Aware Scheduling                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 메모리 할당 시 Local 우선                                                            │
    │  │  • Thread가 해당 메모리에 가까운 CPU에서 실행                                                │
    │  │                                                                                      │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  # NUMA 노드 확인                                                               │  │
    │  │  $ numactl --hardware                                                                │  │
    │  │  $ lscpu                                                                         │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: NUMA 최적화
**상황**: NUMA 서버
**판단**:

```bash
# NUMA topology 확인
$ numactl --hardware

# NUMA 스레드 설정
$ numactl --cpunodebind=0 --membind=0,1 ./app  # Node 0,1 사용

# 메모리 할당 최적화
$ numactl --interleave=all ./app

# 성능 모니터링
$ numastat
```

## Ⅴ. 기대효과 및 결론

### NUMA 기대 효과

| 효과 | UMA | NUMA |
|------|-----|------|
| **확장성** | 제한 | 높음 |
| **성능** | 일관 | 최적화 필요 |
| **복잡도** | 낮음 | 높음 |

### 미래 전망

1. **CC-NUMA**: Cache Coherent NUMA
2. **DIY NUMA**: ARM-based
3. **Heterogeneous**: 다양한 Core
