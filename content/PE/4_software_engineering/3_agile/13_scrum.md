# 스크럼 (Scrum)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스크럼은 **"제품 개발을 위한 **경량의 **프레임워크"**로, **Product Owner**, **Scrum Master**, **Development Team**의 **3가지 역할**과 **Sprint**(1~4주), **Daily Scrum**(일일 미팅), **Sprint Review**, **Sprint Retrospective**의 **5가지 이벤트**로 구성된다.
> 2. **가치**: **불확실한 환경**에서 **복잡한 문제**를 해결하기 위해 ** empiricism(경험적 프로세스)**, **iterative development(반복적 개발)**, **incremental delivery(점진적 전달)**을 통해 **가치(Working Software)**를 **빠르게** 전달한다.
> 3. **융합**: **Jira**, **Azure DevOps**, **GitHub Projects**와 통합되어 **스프린트 관리**, **백로그 추적**, **속도(Velocity) 측정**을 지원하며, **DevOps CI/CD**와 연계되어 **지속적 통합/배포** 파이프라인의 기반이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
스크럼은 **"농구(rugby)의 스크럼에서 유래한, 팀이 함께 목표를 향해 전진하는 애자일 프레임워크"**이다.

**스크럼의 핵심 가치**:
- **Courage**: 용기
- **Focus**: 집중
- **Commitment**: 헌신
- **Respect**: 존중
- **Openness**: 개방성

### 💡 비유
스크럼은 **"스프린트"**와 같다.
- **Sprint Goal**: 결승선
- **Daily Scrum**: 허들
- **Retrospective**: 팀 회피

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         스크럼의 발전                                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

Waterfall (1970년대):
    • 계획 → 설계 → 개발 → 테스트 → 배포
    • 변경 비용 큼
         ↓
Agile Manifesto (2001):
    • 개인과 상호작용 > 프로세스와 도구
    • 동작하는 소프트웨어 > 포괄적인 문서
         ↓
Scrum Guide (1995~):
    • Sutherland & Schwaber
    • 5가지 이벤트, 3가지 역할
    • 2017년: 경험적 프로세스 강조
         ↓
현대:
    • Large Scale Scrum (LeSS)
    • Scrum@Scale
    • SAFe (Scaled Agile Framework)
```

## Ⅱ. 아키텍처 및 핵심 원리

### 스크럼 역할 (3가지)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         스크럼 역할                                                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Product Owner (PO)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 제품 백로그 관리 (Product Backlog Management)                                        │
    │  │  • 백로그 우선순위 설정 (Prioritization)                                             │
    │  │  • User Story 정의 (Acceptance Criteria)                                             │
    │  │  • 이해관계자 대변 (Stakeholder Proxy)                                                │
    │                                                                                         │
    │  책임:                                                                                   │
    │  - "무엇(What)"을 만들지 결정                                                           │
    │  - ROI(Return on Investment) 최대화                                                      │
    │  - Sprint Backlog 승인                                                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Scrum Master (SM)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 팀 코치 (Team Coach)                                                                 │
    │  │  • 스크럼 이벤트 퍼실리테이팅                                                          │
    │  │  • 장애물 제거 (Impediment Remover)                                                   │
    │  │  • 프로세스 개선 coaching                                                             │
    │                                                                                         │
    │  책임:                                                                                   │
    │  - "어떻게(How)"를 잘 할 수 있도록 도움                                                   │
    │  - 스크럼 이벤트 진행                                                                   │
    │  - 팀 보호 (외부 간섭 차단)                                                              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Development Team (DT)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 자기 조직화 팀 (Self-organizing Team)                                                │
    │  │  • 크로스 기능팀 (Cross-functional)                                                   │
    │  │  • 3~9명 (이상적)                                                                      │
    │                                                                                         │
    │  책임:                                                                                   │
    │  - Sprint 목표 달성                                                                       │
    │  - Working Software 개발                                                                  │
    │  - 기술 결정 (How)                                                                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  역할          │  Product Owner    │  Scrum Master  │  Development Team               │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  책임 영역     │  What (무엇)      │  How (어떻게) │  How (어떻게) + Do (실행)     │
    │  권한          │  백로그 결정     │  없음 (서빙)  │  자기 조직화                   │
    │  주요 작업     │  User Story     │  코칭        │  개발, 테스트                   │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 스크럼 이벤트 (5가지)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         스크럼 이벤트 (Sprint Cycle)                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Sprint (1~4주)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │                    Sprint (1~4주, 고정 길이)                                       │  │
    │  │                                                                                      │  │
    │  │  Day 1                       Day N                                                │  │
    │  │    │                           │                                                   │  │
    │  │    ▼                           ▼                                                   │  │
    │  │  ┌─────────────┐      ┌───────────────┐      ┌───────────────┐                   │  │
    │  │  │  Planning   │ ...  │  Daily Scrum  │ ...  │  Review+ Retro │                   │  │
    │  │  │             │      │               │      │               │                   │  │
    │  │  │ Sprint Goal │      │ Sync (15min) │      │ Learning      │                   │  │
    │  │  │ Backlog     │      │ What done?   │      │ Improvement   │                   │  │
    │  │  │             │      │ Plan today   │      │               │                   │  │
    │  │  └─────────────┘      └───────────────┘      └───────────────┘                   │  │
    │  │         │                                           │                             │  │
    │  │         ▼                                           ▼                             │  │
    │  │    Sprint Backlog                            Increment (Working Software)        │  │
    │  │                                                                                      │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [1. Sprint Planning]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  목적: Sprint 목표와 Sprint Backlog 정의                                                 │
    │  시간: 1주 Sprint = 2시간, 4주 Sprint = 8시간                                               │
    │                                                                                         │
    │  일정:                                                                                   │
    │  1. Product Owner: What (Product Backlog에서 선택)                                        │
    │  2. Development Team: How (Task 분해, 추정)                                               │
    │  3. Sprint Goal 설정                                                                     │
    │  4. Sprint Backlog 완성                                                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [2. Daily Scrum (매일)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  목적: 동기화(Sync), 계획(Plan), 장애물 식별(Identify Impediments)                          │
    │  시간: 매일 같은 시간, 15분 이내 (timeboxed)                                              │
    │                                                                                         │
    │  3가지 질문:                                                                             │
    │  1. 어제 무엇을 했나요? (What did I do yesterday?)                                        │
    │  2. 오늘 무엇을 할 건가요? (What will I do today?)                                        │
    │  3. 장애물이 있나요? (Do I have any impediments?)                                         │
    │                                                                                         │
    │  규칙:                                                                                   │
    │  • Scrum Master만 발언                                                                   │
    │  │  • 해결책 논의 금지 (나중에 별도)                                                       │
    │  │  • 지각 벌칙(예: 스쿼트, 기부금) 선택 가능                                                │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [3. Sprint Review (검토 회의)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  목적: Increment 검토, 피드백 수집                                                        │
    │  시간: 1주 Sprint = 1시간, 4주 Sprint = 4시간                                             │
    │                                                                                         │
    │  참여자: Product Owner, Team, Stakeholders                                               │
    │                                                                                         │
    │  안건:                                                                                   │
    │  1. Sprint Goal 달성 여부 확인                                                           │
    │  2. Working Software 데모                                                               │
    │  │  • Done 정의 확인                                                                     │
    │  │  • Stakeholder 피드백                                                                 │
    │  3. Product Backlog 업데이트 (미완료, 새 아이템)                                         │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [4. Sprint Retrospective (회고 회의)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  목적: 팀 개선, 프로세스 최적화                                                            │
    │  시간: 1주 Sprint = 1시간, 4주 Sprint = 3시간                                             │
    │                                                                                         │
    │  참여자: Scrum Team (PO, SM, Development Team) - Stakeholders 제외                          │
    │                                                                                         │
    │  5단계 회고:                                                                             │
    │  1. Set the Stage (분위기 조성)                                                         │
    │  2. Gather Data (데이터 수집)                                                           │
    │  3. Generate Insights (통찰 도출)                                                       │
    │  4. Decide What to Do (실행 계획)                                                       │
    │  5. Close (종료)                                                                         │
    │                                                                                         │
    │  기법: Start-Stop-Continue, KPT, Mad-Sad-Glad                                            │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [5. Backlog Refinement (정기)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  목적: Product Backlog 정제                                                               │
    │  시간: Sprint 중간, Sprint당 10% 시간                                                      │
    │                                                                                         │
    │  활동:                                                                                   │
    │  • User Story 분해                                                                       │
    │  │  • Acceptance Criteria 정                                                             │
    │  │  • Story Point 추정                                                                   │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 스크럼 산출물 (3가지)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         스크럼 산출물                                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [1. Product Backlog]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 모든 작업의 목록 (Ordered List)                                                       │
    │  │  • 우선순위로 정렬                                                                   │
    │  │  • DEEP (Detailed appropriately, Estimated, Emergent, Prioritized)                     │
    │  │  • Product Owner가 소유                                                               │
    │                                                                                         │
    │  항목: User Story, Bug, Technical Debt, Spike, Non-functional Requirement                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [2. Sprint Backlog]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Sprint 동안 개발할 작업 목록                                                         │
    │  │  • Product Backlog에서 선정                                                           │
    │  │  • Task 단위로 분해 (4~16시간)                                                         │
    │  │  • Development Team이 소유                                                             │
    │                                                                                         │
    │  예:                                                                                     │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  Story: 로그인 기능 (5점)                                                             │  │
    │  │    • Task 1: UI 설계 (4시간)                                                          │  │
    │  │    • Task 2: API 개발 (8시간)                                                         │  │
    │  │    • Task 3: 테스트 (4시간)                                                           │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [3. Increment]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Sprint 결과물 (Working Software)                                                       │
    │  │  • "Done" 정의를 충족 (Potentially Releasable)                                        │
    │  │  • 이전 Increments와 누적 (Cumulative)                                                │
    │  │  • 풀질테스트, 문서화 완료                                                             │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### Scrum vs Kanban

| 구분 | Scrum | Kanban |
|------|-------|--------|
| **반복** | Sprint (고정) | Continuous Flow |
| **역할** | PO, SM, Team | 없음 (기존 역할) |
| **WIP 제한** | Sprint Scope | Column 당 |
| **시간boxing** | O | X |
| **변경** | Sprint 중 금지 | 언제나 |

### Definition of Done (DoD)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         DoD (완료의 정의)                                                │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [DoD 예시]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Code Review 완료                                                                     │
    │  │  • Unit Test 통과 (코드 커버리지 80% 이상)                                            │
    │  │  • Integration Test 통과                                                            │
    │  │  • Product Owner 승인                                                                │
    │  │  • Production Deploy 가능                                                             │
    │  │  • Documentation 완료 (README, API 문서)                                              │
    │  │  • Security Scan 통과                                                                │
    │  │  • Performance Test 통과                                                              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 스크럼 도입
**상황:** 웹 개발 팀
**판단**:

```yaml
# Sprint 1 설정

Team:
  - PO: 1명
  - SM: 1명
  - Dev: 5명 (FE 2, BE 2, QA 1)

Sprint:
  - 길이: 2주
  - 시간: 화요일 10시간 Daily Scrum

Events:
  - Planning: 월요일 2시간
  - Daily Scrum: 매일 15분 (10:00~10:15)
  - Review: 금요일 1시간
  - Retro: 금요일 1시간 (Review 후)

DoD:
  - CI/CD 통과
  - Code Review 1인 이상
  - Unit Test > 70%
  - PO 승인

Tools:
  - Jira (Backlog, Sprint Board)
  - Confluence (Documentation)
  - Slack (Daily Scrum Remote)
```
