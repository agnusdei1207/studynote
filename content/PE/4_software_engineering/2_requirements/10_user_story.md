# User Story (사용자 스토리)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: User Story는 **"사용자 관점에서의 간단한 요구사항 명세"**로, **"As a <역할>, I want <기능>, So that <가치>"** 형식으로 작성하며, **애자일(Agile)** 개발의 **Backlog**를 구성하는 기본 단위이다.
> 2. **가치**: **기능적 요구사항**을 사용자 시나리오로 표현하여 **개발자와 사용자 간 커뮤니케이션**을 개선하며, **Acceptance Criteria(검수 조건)**으로 **Definition of Done(완료 정의)**를 명확히 하고 **INVEST**(Independent, Negotiable, Valuable, Estimable, Small, Testable) 원칙을 따른다.
> 3. **융합**: **Scrum**의 **Product Backlog**, **Sprint Backlog** 항목으로 사용되며, **User Story Mapping**, **Story Splitting**, **Story Points**로 **스프린트 계획(Sprint Planning)**과 **속도(Velocity)** 추정의 기반이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
User Story는 **"사용자가 원하는 기능을 간단한 시나리오로 표현한 것"**이다.

**User Story의 정의**:
- 사용자의 **역할(Role)**
- 사용자가 원하는 **기능(Feature)**
- 그 기능으로 얻는 **가치(Value)**

### 💡 비유
User Story는 **"고객의 주문 내용"**과 같다.
- **"집에 빨리 배달받고 싶어"** → User Story
- **"30분 이내 배달"** → Acceptance Criteria

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         User Story의 등장                                             │
└─────────────────────────────────────────────────────────────────────────────────────────┘

전통적 요구사항 (1970~80년대):
    • 상세 SRS (Software Requirements Specification)
    • 수백 페이지의 문서
    • 사용자 이해 어려움
         ↓
XP (Extreme Programming) (1996년):
    • "User Story" 용어 등장 (Kent Beck)
    • 고객과 대화하며 요구사항 수집
    • 인덱스 카드에 작성
         ↓
Scrum (1990년대~2000년대):
    • Product Backlog에 User Story
    • Sprint Backlog에 선택
    • Sprint에서 개발
         ↓
현대 애자일:
    • Jira, Notion, GitHub Projects 관리
    • Story Points로 추정
    • Continuous Refinement
```

## Ⅱ. 아키텍처 및 핵심 원리

### User Story 템플릿

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         User Story 형식                                               │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [기본 형식]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  **As a** <role(역할)>                                                                │
    │  **I want** <function(기능)>                                                           │
    │  **So that** <value(가치)>                                                             │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [예시: 전자상거래]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  As a 고객 (Customer)                                                                 │
    │  I want to 상품을 검색할 수 있어야 한다 (search for products)                               │
    │  So that 내가 원하는 상품을 빨리 찾을 수 있다 (find what I want quickly)              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [예시: 핀테크 앱]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  As a 사용자 (User)                                                                    │
    │  I want to 계좌에서 다른 계좌로 송금하기 (transfer money between accounts)                   │
    │  So that 돈을 보낼 수 있다 (send money)                                                   │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [예시: 소셜 미디어]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  As a 작성자 (Creator)                                                                 │
    │  I want to 사진에 필터를 적용할 수 있어야 한다 (apply filters to photos)                   │
    │  So that 사진이 더 예뻐 보인다 (make photos look better)                               │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Acceptance Criteria (검수 조건)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Acceptance Criteria 작성법                                       │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Given-When-Then 형식 (BDD - Behavior Driven Development)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Story: 상품 검색                                                                        │
    │                                                                                         │
    │  As a 고객                                                                              │
    │  I want to 상품을 검색할 수 있어야 한다                                                   │
    │  So that 내가 원하는 상품을 찾을 수 있다                                                 │
    │                                                                                         │
    │  Acceptance Criteria:                                                                    │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  Given 사용자가 메인 화면에 있을 때                                                    │  │
    │  │  When 검색창에 "노트북"을 입력하고 검색 버튼을 클릭하면                                   │  │
    │  │  Then 노트북 관련 상품 목록이 1초 이내에 표시되어야 한다                                 │  │
    │  │    And 각 상품은 이미지, 이름, 가격이 표시되어야 한다                                   │  │
    │  │    And 최대 100개까지 표시되어야 한다                                                  │  │
    │  │    And 검색 결과가 없으면 "검색 결과가 없습니다" 메시지가 표시되어야 한다                  │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### INVEST 원칙

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         INVEST: 좋은 User Story의 6가지 원칙                               │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  I - Independent (독립적)                                                             │
    │  • 다른 Story와 의존관계가 없음                                                           │
    │  │  순서에 관계없이 개발 가능                                                             │
    │                                                                                         │
    │  ✗ 좋지 않은 예:                                                                        │
    │    "As a 사용자, I want to 로그인한 후 상품을 검색한다"                                     │
    │    → 로그인 Story와 검색 Story 분리 필요                                                    │
    │                                                                                         │
    │  ✓ 좋은 예:                                                                             │
    │    Story 1: "As a 사용자, I want to 로그인한다"                                            │
    │    Story 2: "As a 사용자, I want to 상품을 검색한다"                                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  N - Negotiable (협상 가능)                                                             │
    │  • 구현 방법은 유연함                                                                     │
    │  │  기능/가치는 명확하되어야 함                                                          │
    │                                                                                         │
    │  ✗ 좋지 않은 예:                                                                        │
    │    "React로 SPA를 구현하여 상품을 검색한다"                                                │
    │    → 구현 방법 지정 (협상 불가)                                                           │
    │                                                                                         │
    │  ✓ 좋은 예:                                                                             │
    │    "As a 사용자, I want to 상품을 검색한다" (어떻게는 PO와 협의)                           │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  V - Valuable (가치 있음)                                                               │
    │  • 사용자/비즈니스에 가치 제공                                                            │
    │  │  다음 Sprint에서 반드시 필요한 것                                                        │
    │                                                                                         │
    │  ✗ 좋지 않은 예:                                                                        │
    │    "As a 개발자, I want to 로깅 라이브러리를 최신 버전으로 업그레이드한다"                     │
    │    → 사용자 가치 미미                                                                   │
    │                                                                                         │
    │  ✓ 좋은 예:                                                                             │
    │    "As a 사용자, I want to 상품을 검색한다" (직접적인 사용자 가치)                         │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  E - Estimable (추정 가능)                                                             │
    │  • 크기가 명확하여 추정 가능                                                              │
    │  │  너무 크거나 모호하면 안 됨                                                              │
    │                                                                                         │
    │  ✗ 좋지 않은 예:                                                                        │
    │    "As a 사용자, I want to 모든 것을 할 수 있다"                                           │
    │    → 범위가 너무 넓음                                                                     │
    │                                                                                         │
    │  ✓ 좋은 예:                                                                             │
    │    "As a 사용자, I want to 키워드로 상품명을 검색한다"                                     │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  S - Small (작음)                                                                       │
    │  • 하나의 Sprint에서 완료 가능                                                             │
    │  │  일반적으로 3~10 Story Points                                                         │
    │                                                                                         │
    │  ✗ 좋지 않은 예:                                                                        │
    │    "As a 사용자, I want to 전자상거래 모든 기능을 이용한다"                                   │
    │    → 너무 큼므로 Splitting 필요                                                             │
    │                                                                                         │
    │  ✓ 좋은 예:                                                                             │
    │    Story: "As a 사용자, I want to 키워드로 검색한다"                                       │
    │    → 하나의 Sprint에서 완료 가능                                                           │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  T - Testable (테스트 가능)                                                             │
    │  • 완료 여부를 확인할 수 있음                                                              │
    │  │  자동화 테스트 가능하면 더 좋음                                                          │
    │                                                                                         │
    │  ✗ 좋지 않은 예:                                                                        │
    │    "As a 사용자, I want to 사용하기 쉬운 앱"                                                │
    │    → "사용하기 쉬운"이 주관적                                                               │
    │                                                                                         │
    │  ✓ 좋은 예:                                                                             │
    │    "As a 사용자, I want to 3클릭 내 구매할 수 있다"                                         │
    │    Acceptance Criteria: "메인 → 상품 → 장바구니 → 구매 클릭 (3번 클릭)"                      │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Story Splitting (Story 분해)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Story Splitting 기법                                             │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [SPIDR 기법]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  S - Spike (연구 작업)                                                                 │
    │  │  Story를 구현하기 위한 기술 조사가 필요한 경우                                         │
    │  │  "As a 개발자, I want to AI 추천 엔진을 조사한다" → Spike Story                      │
    │  │                                                                                      │
    │  P - Path (단순 경로)                                                                 │
    │  │  가장 단순한 Happy Path로 시작                                                          │
    │  │  "상품 검색 → 필터 적용 → 정렬"                                                        │
    │  │                                                                                      │
    │  I - Inversion (역순)                                                                  │
    │  │  에러 케이스를 먼저                                                                   │
    │  │  "검색 결과 없음", "네트워크 오류", "서버 다운"                                        │
    │  │                                                                                      │
    │  D - Data (데이터)                                                                     │
    │  │  데이터 입력/출력/변환                                                                 │
    │  │  "검색어 입력", "검색 결과 표시", "검색 기록 저장"                                        │
    │  │                                                                                      │
    │  R - Rules (비즈니스 규칙)                                                            │
    │  │  검증 로직, 제약 조건                                                                   │
    │  │  "최소 2글자", "최대 100개 결과", "인기도순 정렬"                                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### User Story vs 기능적 요구사항

| 구분 | 기능적 요구사항 | User Story |
|------|---------------|------------|
| **관점** | 시스템 중심 | 사용자 중심 |
| **형식** | 정형, 상세 | 비정형, 간단 |
| **문서** | SRS | Backlog |
| **변경** | 어려움 | 유연함 |

### 과목 융합 관점 분석

#### 1. BDD ↔ User Story
- **Gherkin Syntax**: Given-When-Then
- **Cucumber**: 자동화 테스트

#### 2. UX/UI ↔ User Story
- **User Journey**: Story 연결
- **Wireframe**: Story 시각화

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 핀테크 송금 기능 Story Splitting
**상황**: 거대한 Story "계좌 간 송금"
**판단**:
1. **Epic(대형 Story)**: 계좌 간 송금
2. **Split**: 다음과 같이 분리
   - Story 1: 내 계좌 목록 조회 (1pt)
   - Story 2: 송금할 계좌 선택 (2pt)
   - Story 3: 금액 입력 (1pt)
   - Story 4: 비밀번호 확인 (1pt)
   - Story 5: 잔액 확인 (2pt)
   - Story 6: 송금 실행 (3pt)
   - Story 7: 송금 완료 알림 (1pt)

## Ⅴ. 기대효과 및 결론

### User Story 기대 효과

| 효과 | User Story 미사용 | User Story 활용 |
|------|------------------|----------------|
| **사용자 이해** | 기술 중심 문서 | 사용자 시나리오 |
| **변경 유연성** | 문서 수정 어려움 | Backlog 재정렬 |
| **협업** | 일방적 전달 | 공동 작성 |
| **테스트** | 요구사항-테스트 불일치 | BDD 테스트 자동화 |
