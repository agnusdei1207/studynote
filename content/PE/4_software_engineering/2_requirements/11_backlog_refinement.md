# 백로그 정제 (Backlog Refinement)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 백로그 정제는 **"제품 백로그(Product Backlog)의 항목들을 **분석, 명확화, 우선순위 재조정**하여 **향후 스프린트**에서 **개발 가능한 상태**로 만드는 **지속적 활동"**으로, **User Story**를 **세분화**하고 **Acceptance Criteria(수용 기준)**을 정의한다.
> 2. **가치**: **스프린트 계획(Sprint Planning)**의 효율성을 높이고 **불확실성(Uncertainty)**을 감소시키며 **팀의 이해도**를 높여 **스토리 포인트 추정(Story Point Estimation)**의 정확도를 개선한다.
> 3. **융합**: **Scrum**의 **Sprint Review**와 **Sprint Retrospective**와 연계되며, **Jira**, **Azure DevOps**, **GitHub Projects**의 **백로그 관리** 기능과 통합되어 **애자일 개발 프로세스**의 핵심 활동이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
백로그 정제(Backlog Refinement, 전称 Backlog Grooming)는 **"제품 백로그 항목을 지속적으로 검토, 정제, 우선순위화하는 활동"**이다.

**백로그 정제의 목적**:
- **명확화**: User Story 이해도 향상
- **세분화**: 큰 스토리를 개발 가능한 단위로 분할
- **우선순위**: 비즈니스 가치 기준 재정렬
- **추정**: Story Point 또는 T-Shirt Sizing

### 💡 비유
백로그 정제는 **"요리 준비"**와 같다.
- **큰 스토리**: 통째로 재료
- **정제**: 깍둑썰기, 손질
- **스프린트**: 바로 요리 가능한 상태

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         백로그 정제의 필요성                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

초기 Agile (2000년대 초):
    • Sprint Planning만 존재
    • Planning에서 큰 스토리 다뤄서 비효율
         ↓
Backlog Grooming 등장:
    • Sprint 중간에 미리 정제
    • Planning 시간 단축
         ↓
Scrum Guide (2011~):
    • Backlog Refinement로 명칭 변경
    • 정식 활동으로 인식
         ↓
현대 Agile:
    • 지속적 활동 (이벤트 아님)
    • Sprint당 10% 시간 투자 권장
```

## Ⅱ. 아키텍처 및 핵심 원리

### 백로그 정제 절차

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         백로그 정제 5단계                                               │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [1. 백로그 항목 선정]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Product Owner이 상위 항목 선택                                                        │
    │  • 다음 2~3 Sprint에 포함될 스토리                                                       │
    │  • Team이 미리 검토할 목록 정의                                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [2. User Story 검토]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 스토리 내용 Team과 공유                                                               │
    │  • Business Value 확인                                                                  │
    │  • User의 Needs 이해                                                                    │
    │  • 질문과 답변                                                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [3. 세분화 (Breaking Down)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 큰 스토리(Epic) → 작은 스토리                                                         │
    │  • Sprint 기간(2주) 내 완료 가능한 크기                                                  │
    │  • INVEST 원칙 준수                                                                     │
    │                                                                                         │
    │  [Epic → Story 분해 예시]                                                               │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  Epic: 사용자 인증 시스템                                                           │  │
    │  │  → Story 1: 이메일 로그인                                                           │  │
    │  │  → Story 2: 소셜 로그인 (Google, Facebook)                                          │  │
    │  │  → Story 3: 비밀번호 찾기                                                           │  │
    │  │  → Story 4: 2단계 인증                                                              │  │
    │  │  → Story 5: 로그아웃                                                               │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [4. Acceptance Criteria 정의]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 완료 조건 명확화                                                                     │
    │  • Given-When-Then 형식                                                                 │
    │                                                                                         │
    │  예시: "이메일 로그인" 스토리                                                           │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  Given 사용자가 등록된 이메일과 비밀번호를 입력하고                                  │  │
    │  │  When "로그인" 버튼을 클릭하면                                                       │  │
    │  │  Then 대시보드로 이동하고                                                            │  │
    │  │       환영 메시지가 표시된다                                                        │  │
    │  │                                                                                      │  │
    │  │  Negative Scenario:                                                                 │  │
    │  │  - 잘못된 비밀번호 입력 시 "비밀번호가 틀렸습니다" 메시지                             │  │
    │  │  - 미등록 이메일 입력 시 "존재하지 않는 계정입니다" 메시지                            │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [5. 추정 (Estimation)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Story Point 또는 T-Shirt Sizing                                                      │
    │  • Planning Poker, Dot Voting                                                          │
    │                                                                                         │
    │  [Fibonacci Story Points]                                                               │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  1: 매우 간단 (30분~1시간)                                                          │  │
    │  │  2: 간단 (반나절)                                                                   │  │
    │  │  3: 보통 (1일)                                                                      │  │
    │  │  5: 중간 (2~3일)                                                                    │  │
    │  │  8: 큼 (1주)                                                                        │  │
    │  │  13: 매우 큼 (2주)                                                                  │  │
    │  │  21: Epic (분해 필요)                                                               │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### DEEP 백로그 특성

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         DEEP 모델 (Roman Pichler)                                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  [D]etailed Appropriately (적절히 상세함)                                              │
    │  • 상위 스토리: 상세함                                                                 │
    │  • 하위 스토리: 개괄적                                                                 │
    │  • 당장 개발 안 할 것은 굳이 상세X                                                      │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  [E]stimated (추정됨)                                                                   │
    │  • 상위 스토리: 추정 완료                                                               │
    │  • 하위 스토리: 추정 안 함                                                             │
    │  • 추정 = 이해도 반영                                                                   │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  [E]mergent (발전적)                                                                    │
    │  • 백로그은 정적이 아님                                                                 │
    │  • 새로운 스토리 추가, 기존 스토리 수정                                                 │
    │  • 시장 상황, 피드백 반영                                                                │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  [P]rioritized (우선순위화)                                                             │
    │  • 비즈니스 가치 기준 정렬                                                               │
    │  • MVP(Minimum Viable Product) 우선                                                    │
    │  • ROI(Return on Investment) 고려                                                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Planning Poker 절차                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Team 전원 참여                                                                       │
    │  • Fibonacci 카드 (1, 2, 3, 5, 8, 13, 21)                                             │
    │  • Product Owner이 스토리 설명                                                         │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [추정]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. 스토리 설명                                                                          │
    │  2. Team이 질문                                                                         │
    │  3. 동시에 카드 제시                                                                    │
    │  4. 최대/최소 제외자 재논의                                                             │
    │  5. 합의 도출할 때까지 반복                                                              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [합의 도출]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 만장일치 필요 X                                                                       │
    │  • 다수결로 결정                                                                        │
    │  • 추정치 = "복잡도" (시간 X)                                                            │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### Backlog Refinement vs Sprint Planning

| 구분 | Backlog Refinement | Sprint Planning |
|------|-------------------|----------------|
| **목적** | 스토리 정제 | Sprint 일정 계획 |
| **시점** | Sprint 중간 | Sprint 시작 |
| **참여자** | PO + Team | PO + Team + Scrum Master |
| **산출물** | 정제된 스토리, 추정치 | Sprint Backlog, Sprint Goal |

### INVEST 원칙

| 원칙 | 설명 | 예시 |
|------|------|------|
| **I**ndependent | 독립적 | 다른 스토리와 의존X |
| **N**egotiable | 협상 가능 | 구현 방법 유연 |
| **V**aluable | 가치 있는 | 사용자에게 가치 제공 |
| **E**stimable | 추정 가능 | 노력량 측정 가능 |
| **S**mall | 작은 크기 | Sprint 내 완료 가능 |
| **T**estable | 테스트 가능 | Acceptance Criteria 존재 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 백로그 정제 회의
**상황**: 신규 전자상거래 플랫폼
**판단**:

```yaml
# Sprint 3 백로그 정제

Stories for refinement:
  - 장바구니 기능 (Epic → 5개 Story로 분해)
  - 결제 모듈 (외부 API 연동 복잡)
  - 주문 조회 (Story 13점 → 분해 필요)

Decisions:
  - "장바구니" Epic 분해:
    • 상품 추가 (3점)
    • 수량 변경 (2점)
    • 상품 삭제 (1점)
    • 장바구니 저장 (5점)
    • 장바구니 복원 (3점)

  - 우선순위 재조정:
    • MVP: 추가, 삭제, 조회
    • Phase 2: 저장, 복원

Action Items:
  - PO: "결제 모듈" API 문서 공유
  - Tech Lead: 외부 PG사 POC 진행
  - Design: 장바구니 UI 와이어프레임
```

## Ⅴ. 기대효과 및 결론

### 백로그 정제 기대 효과

| 효과 | 미정제 시 | 정제 후 |
|------|----------|---------|
| **Planning 시간** | 4시간+ | 1~2시간 |
| **추정 정확도** | 편차 큼 | 개선 |
| **Team 이해도** | 낮음 | 높음 |
| **Sprint 안정성** | 중간 변경 적음 | 변경 최소화 |

### 미래 전망

1. **AI 기반 추정**: 자동 Story Point 예측
2. **Backlog Health**: 메트릭 기반 관리
3. **Continuous Refinement**: 정기 회의 없이 지속적 정제
