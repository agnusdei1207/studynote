# 수용 기준 (Acceptance Criteria)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 수용 기준은 **"User Story 또는 기능이 **완료(Done)**로 간주되기 위해 **충족해야 하는 **명시적 조건들의 집합"**으로, **Given-When-Then** 형식으로 정의하여 **개발자**, **테스터**, **Product Owner** 간 **명확한 기대치**를 설정한다.
> 2. **가치**: **Done(완료)의 정의**를 명확히 하고 **테스트 시나리오**를 제공하며 **범위 누락(Scope Creep)**을 방지하고 **QA(Quality Assurance)** 프로세스를 간소화하며 **스프린트 리뷰(Sprint Review)**에서 **완료 여부**를 객관적으로 판단한다.
> 3. **융합**: **BDD(Behavior-Driven Development)**의 **Gherkin 문법**과 연계되며, **Cucumber**, **SpecFlow**, **Behave** 같은 **BDD 도구**로 **자동화 테스트**를 생성하고 **CI/CD 파이프라인**에 통합된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
수용 기준(Acceptance Criteria, AC)는 **"기능이 완료되기 위해 충족해야 하는 조건"**이다.

**수용 기준의 목적**:
- **명확성**: 완료 기준 정의
- **테스트 가능성**: 검증 가능한 조건
- **범위 관리**: 기대치 설정

### 💡 비유
수용 기준은 **"시험 출제 범위"**와 같다.
- **User Story**: 문제
- **AC**: 채점 기준

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         수용 기준의 필요성                                              │
└─────────────────────────────────────────────────────────────────────────────────────────┘

초기 개발:
    • 요구사항 모호
    • 완료 기준 불분명
    • 재작업 빈번
         ↓
Agile 등장:
    • User Story 도입
    • 명확한 AC 필요
         ↓
BDD 발전:
    • Given-When-Then 형식
    • 자동화 테스트 연계
         ↓
현대:
    • DoD(Definition of Done)
    • DoR(Definition of Ready)
```

## Ⅱ. 아키텍처 및 핵심 원리

### Given-When-Then 형식

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Gherkin 문법 (BDD)                                             │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [기본 구조]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Feature: 사용자 로그인                                                                │
    │                                                                                         │
    │    Scenario: 유효한 자격증명으로 로그인                                                   │
    │      Given 사용자가 등록된 이메일 "user@example.com"과 비밀번호 "Pass123"을 가지고         │
    │      When "로그인" 페이지에서 이메일과 비밀번호를 입력하고 "로그인" 버튼을 클릭하면         │
    │      Then 대시보드 페이지로 이동하고                                                      │
    │           "환영합니다, user@example.com" 메시지가 표시된다                                 │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [문법 요소]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Given (전제 조건):                                                                      │
    │  • 시스템의 초기 상태                                                                     │
    │  • 테스트 실행 전 환경                                                                   │
    │  • 예: "사용자가 로그인 페이지에 있고"                                                    │
    │                                                                                         │
    │  When (동작):                                                                            │
    │  • 사용자가 수행하는 작업                                                                │
    │  • 트리거 이벤트                                                                         │
    │  • 예: "로그인 버튼을 클릭하면"                                                          │
    │                                                                                         │
    │  Then (결과):                                                                            │
    │  • 기대되는 결과                                                                         │
    │  • 검증 가능한 출력                                                                      │
    │  • 예: "로그인 성공 메시지가 표시된다"                                                   │
    │                                                                                         │
    │  And/But (추가 조건):                                                                    │
    │  • Given, When, Then 연결                                                               │
    │  • 예: "And 비밀번호는 8자 이상이어야 한다"                                               │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 수용 기준 작성 예시

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         User Story별 AC 예시                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Story 1: 이메일 로그인]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  User Story:                                                                           │
    │  As a 사용자                                                                            │
    │  I want to 이메일과 비밀번호로 로그인                                                    │
    │  So that 개인 계정에 접근할 수 있다                                                      │
    │                                                                                         │
    │  Acceptance Criteria:                                                                   │
    │  1. Given 사용자가 로그인 페이지에 있고                                                  │
    │     When 등록되지 않은 이메일을 입력하고 로그인 시도하면                                 │
    │     Then "존재하지 않는 계정입니다" 오류 메시지 표시                                      │
    │                                                                                         │
    │  2. Given 사용자가 로그인 페이지에 있고                                                  │
    │     When 잘못된 비밀번호를 입력하고 로그인 시도하면                                       │
    │     Then "비밀번호가 틀렸습니다" 오류 메시지 표시                                         │
    │                                                                                         │
    │  3. Given 사용자가 로그인 페이지에 있고                                                  │
    │     When 유효한 자격증명을 입력하고 로그인 버튼 클릭하면                                   │
    │     Then 대시보드로 이동하고 세션 쿠키 설정                                             │
    │                                                                                         │
    │  4. Given 사용자가 5회 연속 로그인 실패하고                                               │
    │     When 6번째 시도를 하면                                                              │
    │     Then 계정이 30분간 잠금                                                              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Story 2: 장바구니 담기]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  User Story:                                                                           │
    │  As a 쇼핑객                                                                            │
    │  I want to 상품을 장바구니에 담고                                                        │
    │  So that 나중에 한 번에 구매할 수 있다                                                   │
    │                                                                                         │
    │  Acceptance Criteria:                                                                   │
    │  1. Given 상품 상세 페이지에 있고                                                        │
    │     When "장바구니 담기" 버튼을 클릭하면                                                 │
    │     Then 장바구니에 상품 추가되고 장바구니 아이콘에 수량 표시                               │
    │                                                                                         │
    │  2. Given 상품이 품절이고                                                               │
    │     When "장바구니 담기" 버튼 클릭하면                                                   │
    │     Then 버튼 비활성화되고 "품절입니다" 메시지                                            │
    │                                                                                         │
    │  3. Given 장바구니에 동일 상품이 2개 있고                                                 │
    │     When 같은 상품을 추가하면                                                            │
    │     Then 수량이 3개로 증가                                                               │
    │                                                                                         │
    │  4. Given 장바구니에 10개 상품이 있고                                                    │
    │     When 새 상품을 추가하면                                                             │
    │     Then "장바구니는 최대 10개까지 담을 수 있습니다" 알림                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Story 3: 결제 처리]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  User Story:                                                                           │
    │  As a 구매자                                                                            │
    │  I want to 신용카드로 결제하고                                                           │
    │  So that 주문을 완료할 수 있다                                                           │
    │                                                                                         │
    │  Acceptance Criteria:                                                                   │
    │  1. Given 결제 페이지에 있고                                                            │
    │     When 유효하지 않은 카드 번호를 입력하고 결제 시도하면                                  │
    │     Then "유효하지 않은 카드입니다" 실시간 검증 메시지                                      │
    │                                                                                         │
    │  2. Given 결제 페이지에 있고                                                            │
    │     When CVV를 입력하지 않고 결제 시도하면                                                │
    │     Then CVV 필드에 "CVV를 입력해주세요" 오류 표시                                         │
    │                                                                                         │
    │  3. Given 결제이 성공하면                                                               │
    │     Then 주문 확인 페이지로 이동하고 영수증 이메일 전송                                    │
    │                                                                                         │
    │  4. Given 결제이 실패하면                                                               │
    │     Then "결제 실패: 은행에 문의해주세요" 메시지와 재시도 옵션 제공                         │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### INVEST + SMART AC

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         좋은 AC의 원칙                                                 │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  S - Specific (구체적):                                                                │
    │  • 모호한 표현 금지                                                                     │
    │  │  X "사용하기 편해야 한다"                                                            │
    │  │  O "3번의 클릭으로 주문 완료"                                                        │
    │                                                                                         │
    │  M - Measurable (측정 가능):                                                           │
    │  • 수치화 가능                                                                         │
    │  • X "빠르게 응답"                                                                     │
    │  • O "2초 이내에 응답"                                                                 │
    │                                                                                         │
    │  A - Achievable (달성 가능):                                                           │
    │  • 기술적으로 가능                                                                      │
    │  • X "실시간으로 사용자 마음을 읽음"                                                     │
    │  │  O "입력 완료 후 1초 내 추천 상품 표시"                                                │
    │                                                                                         │
    │  R - Relevant (관련성):                                                                │
    │  • User Story과 직접 연관                                                              │
    │  • X "관리자 페이지도 같이 개선" (범위 누락)                                            │
    │  │  O "사용자 프로필 수정 저장"                                                          │
    │                                                                                         │
    │  T - Testable (테스트 가능):                                                           │
    │  • 자동화 테스트 가능                                                                   │
    │  │  X "사용자가 만족한다"                                                                │
    │  │  O "별점 4점 이상을 유지한다"                                                         │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### DoD vs DoR vs AC

| 구분 | Definition of Done | Definition of Ready | Acceptance Criteria |
|------|-------------------|---------------------|---------------------|
| **목적** | 스프린트 완료 기준 | 시작 가능 기준 | Story 완료 기준 |
| **범위** | 전체 Sprint | 개별 Story | 개별 Story |
| **예시** | Code Review 완료 | AC 정의됨 | 모든 시나리오 통과 |

### BDD 도구 연계

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Cucumber 예시 (Ruby)                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    # features/login.feature

    Feature: 사용자 로그인
      As a 등록된 사용자
      I want to 이메일과 비밀번호로 로그인
      So that 내 계정에 접근할 수 있다

      Scenario: 유효한 자격증명으로 로그인
        Given 사용자가 로그인 페이지에 있고
        And 이메일 "user@example.com"과 비밀번호 "Pass123"으로 등록되어 있고
        When 이메일과 비밀번호를 입력하고 "로그인" 버튼을 클릭하면
        Then 대시보드 페이지로 이동하고
        And "환영합니다" 메시지가 표시된다

      Scenario: 잘못된 비밀번호로 로그인 시도
        Given 사용자가 로그인 페이지에 있고
        And 이메일 "user@example.com"으로 등록되어 있고
        When 이메일 "user@example.com"과 잘못된 비밀번호 "WrongPass"를 입력하고
        And "로그인" 버튼을 클릭하면
        Then "비밀번호가 틀렸습니다" 오류 메시지가 표시된다

    # features/step_definitions/login_steps.rb

    Given('사용자가 로그인 페이지에 있고') do
      visit login_path
    end

    When('이메일과 비밀번호를 입력하고 {string} 버튼을 클릭하면') do |button|
      fill_in 'email', with: 'user@example.com'
      fill_in 'password', with: 'Pass123'
      click_button button
    end

    Then('대시보드 페이지로 이동하고') do
      expect(page).to have_current_path(dashboard_path)
    end

    Then('{string} 메시지가 표시된다') do |message|
      expect(page).to have_content(message)
    end
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: AC 정의 워크샵
**상황**: 신규 이커머스 기능
**판단**:

```yaml
# AC 정의 체크리스트

Story: 주문 취소 기능

Checklist:
  Who:
    - PO: 비즈니스 요구사항 확인
    - Developer: 기술 가능성 확인
    - Tester: 테스트 가능성 확인

  Questions:
    1. 정상 시나리오는 무엇인가?
       → Given 주문 완료 상태 When 취소 버튼 클릭 Then 환불 처리

    2. 예외 시나리오는 무엇인가?
       → 배송 시작 후 취소 시도 → "취소 불가" 메시지

    3. 경계 조건은?
       → 배송 전까지만 취소 가능
       → 3회 구매 후 취소 시 계정 정지

    4. 성능 요구사항은?
       → 취소 처리 2초 이내 완료

Result AC:
  AC1: 배송 대기 상태에서 취소 가능
  AC2: 배송 시작 시 취소 불가 안내
  AC3: 취소 후 환불 처리 2영업일 이내
  AC4: 취소 내역 이메일 전송
```

## Ⅴ. 기대효과 및 결론

### 수용 기준 기대 효과

| 효과 | AC 없을 시 | AC 있을 시 |
|------|----------|----------|
| **완료 기준** | 주관적 | 객관적 |
| **재작업** | 빈번 | 감소 |
| **테스트** | 애매한 | 명확한 시나리오 |
| **의사소통** | 오해 가능 | 공유된 이해 |

### 미래 전망

1. **AI 기반 AC 생성**: NLP로 자동 생성
2. **Living Documentation**: AC = 문서 = 테스트
3. **Specification by Example**: 실제 예시로 AC 대체
