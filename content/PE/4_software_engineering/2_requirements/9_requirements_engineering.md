# 요구사항 공학 (Requirements Engineering)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구사항 공학은 **"이해관계자(Stakeholders)로부터 요구사항을 **수집(Elicitation)** → **분석(Analysis)** → **명세(Specification)** → **검증(Validation)**하는 체계적인 프로세스**"로, **SRS(Software Requirements Specification)**를 산출물로 한다.
> 2. **가치**: **애매한 요구사항(Ambiguous Requirements)**을 명확히 하여 **재작업(Rework)** 비용을 줄이며, **기능적 요구사항(Functional Requirements)**과 **비기능적 요구사항(Non-functional Requirements)**를 구분하여 **개발 범위(Scope)**를 명확히 한다.
> 3. **융합**: **애자일(Agile)**의 **User Story**, **Acceptance Criteria**, **Backlog Refinement**로 발전하며, **SysML**(Systems Modeling Language), **UML**(Unified Modeling Language)의 **Use Case Diagram**, **요구사항 추적성(Requirements Traceability)**의 기반이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
요구사항 공학(Requirements Engineering)은 **"소프트웨어가 무엇을 해야 하는지 정의하고 관리하는 프로세스"**이다.

**요구사항의 정의 (IEEE 830)**:
- **요구사항(Requirement)**: 시스템이 충족해야 하는 조건 또는 능력
- **명세(Specification)**: 요구사항을 문서화한 것

### 💡 비유
요구사항 공학은 **"주택 설계"**와 같다.
- **수집**: 가족 요구사항 (방 개수, 주방 크기)
- **분석**: 타당성 조사 (예산, 부지)
- **명세**: 설계도 작성
- **검증**: 설계도 검토

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         요구사항 공학의 필요성                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

소프트웨어 위기 (1960~70년대):
    • 요구사항 불충분
    • 개발 중 변경 잦음
    • 최종 제품 불만족
         ↓
요구사항 엔지니어링 (1980년대):
    • IEEE 830 표준
    • SRS 템플릿
    • 구조적 분석 (SA/SD)
         ↓
UML (1990년대):
    • Use Case 도입
    • 시각적 모델링
    • 객체지향 요구사항
         ↓
애자일 (2000년대):
    • User Story
    • Backlog
    • 지속적 리파인먼트
```

## Ⅱ. 아키텍처 및 핵심 원리

### 요구사항 공학 4단계 프로세스

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         요구사항 공학 프로세스                                           │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [1단계: 요구사항 수집 (Requirements Elicitation)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  목적: 이해관계자로부터 요구사항 발견                                                      │
    │                                                                                         │
    │  기법:                                                                                   │
    │  • 인터뷰 (Interview): 1:1 심층 면닡                                                    │
    │  │  워크숍 (Workshop): 다수 이해관자자 집단 토의                                          │
    │  │  설문 (Questionnaire): 넓은 범위 의견 수집                                             │
    │  │  관찰 (Observation): 실제 업무 현장 관찰                                               │
    │  │  프로토타이핑 (Prototyping): 시제품을 통한 피드백                                    │
    │  │  브레인스토밍 (Brainstorming): 창의적 아이디어 발산                                     │
    │                                                                                         │
    │  산출물:                                                                                 │
    │  • 비즈니스 요구사항 (Stakeholder Needs)                                                 │
    │  │  요구사항 목록 (Requirements List)                                                    │
    │  └───────────────────────────────────────────────────────────────────────────────────────┘
    │                              │                                                          │
    │                              ▼                                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [2단계: 요구사항 분석 (Requirements Analysis)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  목적: 수집된 요구사항 분석, 구조화, 충돌 해결                                            │
    │                                                                                         │
    │  기법:                                                                                   │
    │  • 요구사항 분류: 기능적 vs 비기능적                                                     │
    │  │  우선순위 부여: MoSCoW (Must, Should, Could, Won't)                                   │
    │  │  타당성 조사: 기술적, 경제적, 운영적                                                    │
    │  │  충돌 해결: 상충되는 요구사항 조정                                                     │
    │  │  모델링: Use Case, Data Flow Diagram                                                 │
    │                                                                                         │
    │  산출물:                                                                                 │
    │  • 정제된 요구사항 (Refined Requirements)                                                │
    │  │  Use Case Diagram                                                                   │
    │  │  데이터 모델                                                                          │
    │  └───────────────────────────────────────────────────────────────────────────────────────┘
    │                              │                                                          │
    │                              ▼                                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [3단계: 요구사항 명세 (Requirements Specification)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  목적: 요구사항을 정형적으로 문서화                                                         │
    │                                                                                         │
    │  문서: SRS (Software Requirements Specification)                                         │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  1. 서문 (Introduction)                                                              │  │
    │  │     • 목적, 범위, 정의, 용어                                                          │  │
    │  │                                                                                      │  │
    │  │  2. 전체 묘사 (General Description)                                                   │  │
    │  │     • 제품 perspective, 기능, 사용자 특성                                             │  │
    │  │                                                                                      │  │
    │  │  3. 기능적 요구사항 (Functional Requirements)                                          │  │
    │  │     • Use Case, 입력/출력, 기능 설명                                                  │  │
    │  │                                                                                      │  │
    │  │  4. 비기능적 요구사항 (Non-functional Requirements)                                      │  │
    │  │     │  성능, 보안, 신뢰성, 사용성                                                     │  │
    │  │                                                                                      │  │
    │  │  5. 제약 조건 (Constraints)                                                            │  │
    │  │     • 하드웨어, 소프트웨어, 표준                                                        │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                              │                                                          │
    │                              ▼                                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [4단계: 요구사항 검증 (Requirements Validation)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  목적: 요구사항이 올바르고 완전한지 확인                                                   │
    │                                                                                         │
    │  검증 기법:                                                                              │
    │  • 리뷰 (Review): 동료 검토                                                              │
    │  │  프로토타이핑 (Prototyping): 시연                                                     │
    │  │  테스트 케이스 작성 (Test Case Generation): 검증 가능한지                               │
    │  │  추적성 행렬 (Traceability Matrix): 요구사항-설계-코드 연결                               │
    │                                                                                         │
    │  검증 기준:                                                                              │
    │  ✓ 정확성 (Correct): 사용자 요구 반영                                                     │
    │  ✓ 무결성 (Consistent): 충돌 없음                                                        │
    │  ✓ 완전성 (Complete): 누락 없음                                                          │
    │  ✓ 명확성 (Unambiguous): 해석의 여지 없음                                                 │
    │  ✓ 검증 가능 (Verifiable): 테스트 가능                                                     │
    │  ✓ 수정 가능 (Modifiable): 변경 용이                                                      │
    │  ✓ 추적 가능 (Traceable): 추적성 확보                                                     │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 기능적 vs 비기능적 요구사항

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         요구사항 유형 구분                                                │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [기능적 요구사항 (Functional Requirements)]

    정의: 시스템이 수행해야 하는 기능과 동작

    예시 (전자상거래):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 사용자는 상품을 검색할 수 있어야 한다                                                  │
    │  │  장바구니에 상품을 추가할 수 있어야 한다                                               │
    │  │  주문을 완료할 수 있어야 한다                                                          │
    │  │  결제 수단(신용카드, 계좌이체)를 선택할 수 있어야 한다                                    │
    │  │  주문 내역을 조회할 수 있어야 한다                                                      │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    작성 예시 (SMART 원칙):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  [빈약] 검색 기능을 제공한다                                                             │
    │  │                                                                                       │  │
    │  [개선] 시스템은 사용자가 키워드를 입력하여 상품명으로 검색할 수 있어야 하며,                  │
    │        검색 결과는 1초 이내에 최대 100개까지 표시되어야 한다.                               │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [비기능적 요구사항 (Non-functional Requirements)]

    정의: 시스템의 성격, 특성, 제약 조건

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  카테고리        │  설명                        │  예시                                   │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  성능 (Performance)│  응답 시간, 처리량           │  검색 1초 이내, 동시 1000명 처리            │
    │  보안 (Security)    │  인증, 권한, 암호화         │  HTTPS, 2FA, PCI-DSS 준수                 │
    │  신뢰성 (Reliability)│  가용성, 장애 복구          │  99.9% 가용성, 5분 내 복구               │
    │  사용성 (Usability) │  학습 용이성, 효율성        │  3클릭 내 구매, 모바일 지원              │
    │  확장성 (Scalability)│  부하 증가 대응            │  수평 확장, 10배 트래픽 지원              │
    │  호환성 (Compatibility)│ 다른 시스템 연동         │  iOS, Android, Chrome 지원             │
    │  유지보수 (Maintainability)│  수정 용이성          │  모듈화, 문서화, 테스트 커버리지 80%     │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### MoSCoW 우선순위 기법

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         MoSCoW 우선순위 분류                                           │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Must have (반드시 있어야 함)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 릴리즈에 필수적임                                                                    │
    │  │  없으면 시스템 실패                                                                    │
    │  │  예: 로그인, 장바구니, 결제                                                           │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Should have (있어야 함)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 중요하지만 즉시 필요하지 않음                                                          │
    │  │  임시 방법이 있을 수 있음                                                              │
    │  │  예: 상품 검색, 주문 내역                                                              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Could have (있으면 좋음)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 가치가 있지만 필수는 아님                                                              │
    │  │  시간/예산 여유가 있으면 구현                                                            │
    │  │  예: 추천 시스템, 리뷰                                                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Won't have (이번에 없음)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  │  이번 릴리즈에서 제외                                                                  │
    │  │  미래로 연기                                                                           │
    │  │  예: AI 추천, 소셜 로그인                                                               │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 전통적 vs 애자일 요구사항 공학

| 구분 | 전통적 (Waterfall) | 애자일 (Agile) |
|------|-------------------|---------------|
| **문서** | 상세 SRS | User Story |
| **변경** | 어려움 | 환영 |
| **참여** | 초기에 한 번 | 지속적 |
| **형식** | 정형 | 유연 |

### 과목 융합 관점 분석

#### 1. UML ↔ 요구사항
- **Use Case Diagram**: 기능적 요구사항 시각화
- **Class Diagram**: 개념적 모델링

#### 2. DevOps ↔ 요구사항
- **Requirements as Code**: 요구사항 코드화
- **BDD (Behavior-Driven Development)**: Gherkin syntax

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 핀테크 앱 요구사항 수집
**상황**: 신규 서비스, 요구사항 불확실
**판단**:
1. **워크숍**: 3일간 핵심 이해관계자 참여
2. **프로토타이핑**: Figma로 UI 시연
3. **User Story 작성**:
   ```
   As a 고객
   I want to 송금하기
   So that 빠르게 돈을 보낼 수 있다

   Acceptance Criteria:
   - 계좌 선택 가능
   - 1천원~1백만원 송금
   - 즉시 이체 (24시간 제한)
   - 이체 내역 표시
   ```
