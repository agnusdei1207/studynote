# 클린룸 소프트웨어 공학 (Cleanroom Software Engineering)

## 정의

**클린룸 소프트웨어 공학(Cleanroom Software Engineering)**은 **결함 없는 소프트웨어** 개발을 목표로, 수학적 명세, 정적 검증, 통계적 품질 제어를 결합한 소프트웨어 개발 방법론이다. 1980년대 IBM의 Harlan Mills가 개발했다.

```
┌─────────────────────────────────────────────────────────────────┐
│           클린룸 소프트웨어 공학 개념                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   이름 유래: 반도체 클린룸에서 결함 없는 칩 생산처럼             │
│             소프트웨어도 결함 없이 생산하겠다는 철학              │
│                                                                 │
│   핵심 원칙:                                                    │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   ① 결함 예방 (Defect Prevention)                       │ │
│   │      • 디버깅 대신 검증 중심                             │ │
│   │      • 오류를 사전에 차단                                │ │
│   │                                                         │ │
│   │   ② 수학적 명세 (Mathematical Specification)            │ │
│   │      • 형식적 명세 언어 사용                             │ │
│   │      • 모호성 제거                                       │ │
│   │                                                         │ │
│   │   ③ 정적 검증 (Static Verification)                     │ │
│   │      • 실행 없이 코드 리뷰                               │ │
│   │      • 수학적 증명 기법                                  │ │
│   │                                                         │ │
│   │   ④ 통계적 품질 제어 (Statistical Quality Control)       │ │
│   │      • 신뢰성 모델링                                     │ │
│   │      • 통계적 테스팅                                     │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2. 박스 구조 명세 (Box Structure Specification)

```
┌─────────────────────────────────────────────────────────────────┐
│           박스 구조 명세                                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   3단계 추상화 계층:                                            │
│                                                                 │
│   ① 블랙 박스 (Black Box):                                     │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │             사용자 관점                        │    │ │
│   │   │                                               │    │ │
│   │   │   입력 ─────▶ [블랙 박스] ────▶ 출력          │    │ │
│   │   │                (내부 구조 숨김)                 │    │ │
│   │   │                                               │    │ │
│   │   │   • 외부 행위 명세                            │    │ │
│   │   │   • 입력/출력 관계 정의                       │    │ │
│   │   │   • 상태 변화 추상화                          │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   ② 스테이트 박스 (State Box):                                 │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │             상태 관점                          │    │ │
│   │   │                                               │    │ │
│   │   │   입력 + 현재상태 ─▶ [스테이트 박스]          │    │ │
│   │   │                       ────▶ 출력 + 다음상태    │    │ │
│   │   │                                               │    │ │
│   │   │   • 내부 상태 정의                            │    │ │
│   │   │   • 상태 전이 명세                            │    │ │
│   │   │   • 데이터 추상화                             │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   ③ 클리어 박스 (Clear Box):                                   │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │             구현 관점                          │    │ │
│   │   │                                               │    │ │
│   │   │   입력 + 상태 ─▶ [프로시저/제어구조]          │    │ │
│   │   │                   ────▶ 출력 + 새상태          │    │ │
│   │   │                                               │    │ │
│   │   │   • 구체적 구현 로직                          │    │ │
│   │   │   • 제어 구조 명세                            │    │ │
│   │   │   • 하위 모듈 호출                            │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 정확성 검증 (Correctness Verification)

```
┌─────────────────────────────────────────────────────────────────┐
│           정확성 검증                                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   팀 기반 검증 (Team-Based Verification):                       │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   ① 개발자: 코드 작성                                   │ │
│   │   ② 검증 팀: 정적 분석 수행                              │ │
│   │      • 코드를 실행하지 않고 논리적 검증                  │ │
│   │      • 수학적 증명 기법 적용                             │ │
│   │   ③ 워크샵: 집단 검토                                   │ │
│   │                                                         │ │
│   │   검증 질문:                                             │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │ Q1: 모든 조건이 올바르게 처리되는가?            │    │ │
│   │   │ Q2: 루프가 항상 종료되는가?                      │    │ │
│   │   │ Q3: 모든 경로가 명세와 일치하는가?              │    │ │
│   │   │ Q4: 부작용이 없는가?                            │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   정확성 정리 (Correctness Theorem):                             │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   프로그램이 올바르려면:                                 │ │
│   │                                                         │ │
│   │   ∀s ∈ States:                                          │ │
│   │     Precondition(s) → Postcondition(f(s))               │ │
│   │                                                         │ │
│   │   • Precondition: 실행 전 조건                          │ │
│   │   • Postcondition: 실행 후 조건                          │ │
│   │   • f(s): 상태 전이 함수                                 │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4. 통계적 테스팅

```
┌─────────────────────────────────────────────────────────────────┐
│           통계적 테스팅 (Statistical Testing)                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   목표: 소프트웨어 신뢰성을 통계적으로 인증                       │
│                                                                 │
│   프로세스:                                                     │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   ① 사용 프로파일 (Usage Profile) 정의                   │ │
│   │      • 실제 사용 패턴 분석                               │ │
│   │      • 기능별 사용 확률 할당                             │ │
│   │                                                         │ │
│   │   ② 테스트 케이스 생성                                   │ │
│   │      • 사용 프로파일 기반 무작위 생성                    │ │
│   │      • 운영 환경 시뮬레이션                              │ │
│   │                                                         │ │
│   │   ③ 신뢰성 측정                                          │ │
│   │      • MTTF (Mean Time To Failure)                      │ │
│   │      • 신뢰도 함수 R(t) = e^(-λt)                        │ │
│   │                                                         │ │
│   │   ④ 인증 (Certification)                                │ │
│   │      • 목표 신뢰도 달성 여부 판정                         │ │
│   │      • 실패 시 개발로 회귀                               │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   신뢰성 모델:                                                  │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   MTTF = 총 실행시간 / 결함 수                           │ │
│   │                                                         │ │
│   │   목표 예:                                               │ │
│   │   • MTTF ≥ 1000 실행시간                                │ │
│   │   • 신뢰도 99.9% (1000시간)                              │ │
│   │                                                         │ │
│   │   💡 결함 발견 시 디버깅 후 다시 테스트                   │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 기술사 시험 포인트

### 1. 핵심 정의

> **클린룸 공학**: 결함 없는 소프트웨어 개발 방법론
> **핵심 기법**: 수학적 명세, 정적 검증, 통계적 테스팅
> **목표**: 제로 디펙트 (Zero Defect)

### 2. 빈출 키워드

- **Box Structure**: Black/State/Clear Box 3단계 명세
- **정적 검증**: 실행 없는 코드 검증
- **통계적 테스팅**: 사용 프로파일 기반 신뢰성 측정
- **MTTF**: 평균 고장 시간

### 3. 예상 문제

**Q1. 클린룸 소프트웨어 공학의 개념과 핵심 원칙을 설명하시오.**

**Q2. 박스 구조 명세의 3단계를 설명하고 각각의 특징을 기술하시오.**

**Q3. 클린룸 공학에서 통계적 테스팅의 역할과 방법을 설명하시오.**
