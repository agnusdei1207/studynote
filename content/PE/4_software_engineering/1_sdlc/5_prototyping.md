# 프로토타이핑 모델 (Prototyping Model)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로토타이핑 모델은 **"실제 작동하는 시제품(Prototype)을 빠르게 개발하여 사용자 피드백을 받고, 이를 반영하여 최종 시스템을 개발하는 방법론"**으로, **요구사항 불확실성(Uncertainty)**을 해결하고 **사용자 참여(User Involvement)**를 극대화한다.
> 2. **가치**: **Throw-away Prototype(폐기형)**은 요구사항 파악만을 목적으로 하며, **Evolutionary Prototype(진화형)**은 최종 시스템으로 발전시켜 **반복적 개발(Iterative Development)**과 **위험 감소(Risk Reduction)**를 실현한다.
> 3. **융합**: **애자일(Agile)**의 **MVP(Minimum Viable Product)** 개념과 **스파인(Sprint)** 방식의 선구자이자, **UX/UI 디자인**의 **와이어프레임(Wireframe)**, **Mockup**, **PoC(Proof of Concept)** 개발의 기반이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
프로토타이핑(Prototyping)은 **"실제 시스템의 축소판 또는 시제품을 만들어 사용자에게 보여주고 피드백을 받아 개발하는 방식"**이다.

**프로토타입의 목적**:
- **요구사항 명확화**: 사용자가 원하는 기능 파악
- **기술적 타당성 검증**: 기술 구현 가능성 확인
- **사용자 인터페이스(UI) 확인**: 조작성, 직관성 평가

### 💡 비유
프로토타입은 **"주택 모형 하우스"**와 같다.
- **모형**: 실제 집의 축소판
- **체험**: 방의 크기, 배치 직접 확인
- **수정**: 설계 변경 비용 저렴

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         프로토타이핑의 필요성                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

폭포수 모델의 한계 (1970년대):
    • 요구사항 초기 완전 정의 어려움
    • 개발 후 사용자 불만족 발견
    • 수정 비용 매우 큼
         ↓
프로토타이핑 등장 (1980년대):
    • 빠른 시제품 개발
    • 사용자 피드백 조기 수집
    • "실행하며 배우기(Learn by Doing)"
         ↓
Rapid Application Development (1990년대):
    • CASE 도구로 프로토타이핑 가속화
    • 4GL(4th Generation Language)
    • 시각적 프로그래밍
         ↓
애자일 (2001년):
    • 프로토타이핑의 반복성 계승
    • 릴리스 주기 단축
    • 지속적 배포
```

## Ⅱ. 아키텍처 및 핵심 원리

### 프로토타이핑 개발 주기

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         프로토타이핑 개발 주기                                           │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [1단계: 요구사항 수집 (Requirements Gathering)]
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 사용자 인터뷰                                                                       │
    │  • 기본 요구사항 파악 (불완전해도 됨)                                                    │
    │  • 핵심 기능 식별                                                                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
    [2단계: 빠른 설계 (Quick Design)]
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 화면 설계 (Screen Design)                                                           │
    │  • 사용자 인터페이스(UI) 중심                                                            │
    │  • 핵심 기능만 선택                                                                     │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
    [3단계: 프로토타입 구축 (Prototype Build)]
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 빠른 개발 도구 활용                                                                  │
    │  • 기능보다 UI 중심                                                                     │
    │  • 일부 기능은 Mock (가짜 데이터)                                                        │
    │                                                                                         │
    │  [프로토타입 종류]                                                                      │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  • 와이어프레임 (Wireframe): 선도, 레이아웃                                         │  │
    │  │  • Mockup: 정적 UI 화면                                                              │  │
    │  │  • Clickable Prototype: 클릭 가능한 인터랙티브 모형                                  │  │
    │  │  • PoC (Proof of Concept): 기술적 타당성 검증                                        │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
    [4단계: 사용자 평가 (User Evaluation)]
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 사용자 데모                                                                          │
    │  • 직접 사용 및 피드백                                                                  │
    │  • 요구사항 수정 및 추가                                                                │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
    [5단계: 프로토타입 정제 (Refine Prototype)]
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 피드백 반영                                                                          │
    │  • 반복 개발 (Iterative)                                                                │
    │                                                                                         │
    │  [분기점]                                                                                │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  만족? ──No──▶ 3단계로 돌아가 수정 (Iterate)                                        │  │
    │  │     │                                                                               │  │
    │  │    Yes                                                                              │  │
    │  │     │                                                                               │  │
    │  │     ▼                                                                               │  │
    │  │  [Throw-away] 최종 시스템 재개발                                                     │  │
    │  │  [Evolutionary] 프로토타입을 발전시켜 최종 시스템                                    │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
    [6단계: 최종 시스템 구현 (Final System Implementation)]
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 요구사항 완전 정의                                                                   │
    │  • 품질, 성능 고려 개발                                                                 │
    │  • 테스트 및 배포                                                                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 프로토타입 유형 비교

| 구분 | Throw-away (폐기형) | Evolutionary (진화형) |
|------|---------------------|----------------------|
| **목적** | 요구사항 파악 | 최종 시스템 개발 |
| **구현** | 빠르고 저렴하게 | 품질 고려하여 |
| **코드** | 재사용 안 함 | 계속 발전 |
| **시간** | 짧음 (주-월) | 김 (월-년) |
| **적용** | 요구사항 불확실 | UI 중심 시스템 |

### Throw-away vs Evolutionary

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         프로토타입 유형별 접근                                           │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Throw-away Prototype (폐기형)]

    목적: 요구사항 파악용만 사용

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  프로토타입 ─────────────▶ 사용자 피드백 ─────────────▶ 폐기                            │
    │  (빠르게 개발)                                          (코드 버림)                     │
    │                              │                                                     │
    │                              ▼                                                     │
    │                    최종 시스템 (새로 개발)                                             │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    장점:                     단점:
    • 빠르고 저렴              • 재작업 필요
    • 요구사항 명확화          • 사용자 기대치 관리 어려움
    • 기술 실패 비용 낮음

    [Evolutionary Prototype (진화형)]

    목적: 최종 시스템으로 발전

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Version 1 ─────────────▶ Version 2 ─────────────▶ Version 3 ───────────▶ 최종 시스템   │
    │  (프로토타입)              (기능 추가)              (성능 최적화)                         │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    장점:                     단점:
    • 시간 단축                • 기술 부채 가능성
    • 지속적 피드백            • 아키텍처 변경 어려움
    • 사용자 만족도 높음       • 범위 확대(Risk)

    [예시]
    Throw-away:   • 핵심 로직의 알고리즘 검증
                  • 새로운 UI/UX 실험
                  • 성능 병목 지점 발견

    Evolutionary: • 웹 애플리케이션 (MVP)
                  • 모바일 앱 (베타 테스트)
                  • SaaS (지속적 배포)
```

## Ⅲ. 융합 비교 및 다각도 분석

### 프로토타이핑 도구 비교

| 도구 | 종류 | 장점 | 적용 |
|------|------|------|------|
| **Figma** | Mockup | 협업, 클라우드 | UI/UX |
| **Axure** | Prototype | 인터랙션 | 복잡한 UX |
| **Code** | PoC | 실제 기능 | 기술 검증 |
| **No-Code** | MVP | 빠른 개발 | 스타트업 |

### 과목 융합 관점 분석

#### 1. HCI ↔ 프로토타이핑
- **사용자 중심 설계(UCD)**: 프로토타입으로 사용자 테스트
- **반복적 설계**: 지속적 피드백 반영

#### 2. 애자일 ↔ 프로토타이핑
- **Sprint**: 짧은 프로토타입 개발
- **MVP**: 최소 기능 프로토타입
- **A/B 테스트**: 프로토타입 비교

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 핀테크 앱 개발
**상황**: 신규 금융 서비스, 요구사항 불확실
**판단**:
1. **1주차**: 와이어프레임 (Figma)
   - 핵심 화면 10개
   - 사용자 테스트
2. **2주차**: Clickable Prototype
   - 실제 데이터 연동 X
   - UX 흐름 확인
3. **3-4주차**: PoC (MVP)
   - 핵심 기능만 실제 구현
   - 기술적 타당성 검증
4. **5주차이후**: Evolutionary
   - 지속적 기능 추가
   - 실제 배포

## Ⅴ. 기대효과 및 결론

### 프로토타이핑 기대 효과

| 프로젝트 유형 | 적합성 | 이유 |
|-------------|--------|------|
| **UI 중심** | 높음 | 시각적 피드백 용이 |
| **신기술 도입** | 높음 | 기술 검증 필요 |
| **요구사항 불확실** | 높음 | 피드백으로 확정 |
| **안전 비행사** | 낮음 | 재작업 불가 |
| **성능 중요** | 낮음 | 프로토타입 부적합 |

### 미래 전망

1. **AI-Assisted Prototyping**: LLM이 코드 생성
2. **Low-Code/No-Code**: 빠른 프로토타이핑
3. **Digital Twin**: 시스템 복제
