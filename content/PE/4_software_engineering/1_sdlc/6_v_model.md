# V-Model (V-Model)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: V-Model은 **"개발 단계와 테스트 단계를 V자 형태로 매핑하여 각 개발 산출물에 대응하는 테스트를 정의하는 방법론"**으로, **검증(Verification)**과 **확인(Validation)**을 체계적으로 수행하며 **폭포수 모델의 확장**이다.
> 2. **가치**: **요구사항 → 인수 테스트**, **설계 → 시스템 테스트**, **상세 설계 → 통합 테스트**, **코딩 → 단위 테스트**의 **V자 매핑**을 통해 **조기 결함 발견(Early Defect Detection)**과 **품질 보증(Quality Assurance)**를 실현한다.
> 3. **융합**: **인증(Certification)**, **안전성(Safety)**이 중요한 **임베디드 시스템**, **의료기기**, **항공우주**, **방산** 분야의 표준(IEC 62304, DO-178C)으로 채택되며, **TMMi(Test Maturity Model integration)**의 기반이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
V-Model은 **"개발 각 단계마다 대응하는 테스트 단계를 정의하여 품질을 보장하는 방법론"**이다.

**V-Model의 핵심 개념**:
- **검증(Verification)**: "옳게 만들고 있는가?" (Do it right)
- **확인(Validation)**: "옳은 것을 만들고 있는가?" (Do the right thing)
- **V자 매핑**: 개발 단계와 테스트 단계의 1:1 대응

### 💡 비유
V-Model은 **"건축물 검사"**와 같다.
- **설계**: 건축 설계도
- **검사**: 설계도 검토
- **시공**: 실제 건축
- **확인**: 완성된 건물 검사

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         V-Model의 등장 배경                                             │
└─────────────────────────────────────────────────────────────────────────────────────────┘

폭포수 모델의 한계 (1970년대):
    • 테스트가 개발 후반에 집중
    • 결함 조기 발견 어려움
    • 수정 비용 큼
         ↓
V-Model 등장 (1980년대, 독일):
    • 개발과 테스트 동시 수행
    • "V"자 형태로 매핑
    • Bundeswehr 방산 프로젝트
         ↓
확장 V-Model (1990년대):
    • V-Model XT
    • agile V-Model
    • 테스트 중심 방법론
         ↓
현대 적용 (2000년대~):
    • 안전 임계 시스템 (Safety-critical)
    • IEC 62304 (의료기기)
    • DO-178C (항공우주)
```

### 검증(Verification) vs 확인(Validation)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         V&V (Verification & Validation)                                 │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Verification: 검증 - "Are we building the product right?"]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  목적: 산출물이 요구사항과 일치하는지 확인                                              │
    │                                                                                         │
    │  활동:                                                                                   │
    │  • 요구사항 리뷰                                                                        │
    │  • 설계 리뷰                                                                            │
    │  • 코드 리뷰                                                                            │
    │  • 워크스루스(Through)                                                                   │
    │  • 정적 분석(Static Analysis)                                                            │
    │                                                                                         │
    │  질문:                                                                                   │
    │  "설계가 요구사항을 충족하는가?"                                                        │
    │  "코드가 설계를 올바르게 구현했는가?"                                                    │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Validation: 확인 - "Are we building the right product?"]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  목적: 최종 제품이 사용자 요구를 충족하는지 확인                                         │
    │                                                                                         │
    │  활동:                                                                                   │
    │  • 단위 테스트(Unit Testing)                                                            │
    │  │  통합 테스트(Integration Testing)                                                     │
    │  │  시스템 테스트(System Testing)                                                        │
    │  │  인수 테스트(Acceptance Testing)                                                      │
    │  │  알파/베타 테스트                                                                     │
    │  │  사용자 테스트                                                                        │
    │  │                                                                                      │
    │  질문:                                                                                   │
    │  "제품이 사용자의 실제 요구를 충족하는가?"                                              │
    │  "제품이 비즈니스 목표를 달성하는가?"                                                   │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [V&V 매트릭스]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │                │  Verification    │   Validation                                     │
    │                │  (옳게 만들고    │   (옳은 것을                                   │
    │                │   있는가?)       │    만들고 있는가?)                             │
    │  ───────────────┼──────────────────┼──────────────────────────────────────────────────  │
    │  요구사항       │ 요구사항 리뷰   │ 사용자 요구 확인                                │
    │  설계           │ 설계 리뷰       │ 프로토타입 테스트                                │
    │  코딩           │ 코드 리뷰       │ 단위/통합 테스트                                 │
    │  시스템         │ 구성 검사       │ 시스템 테스트                                    │
    │  배포           │ 설치 검사       │ 인수 테스트                                      │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### SDLC 모델 비교

| 모델 | 테스트 접근 | 장점 | 단점 | 적용 |
|------|-----------|------|------|------|
| **폭포수** | 개발 후 테스트 | 문서화 | 결함 조기 발견 어려움 | 안정적 요구 |
| **V-Model** | 개발-테스트 병행 | 품질 보증 | 변경 비용 큼 | 안전 임계 |
| **나선형** | 위험 중심 테스트 | 위험 관리 | 복잡함 | 대형 프로젝트 |
| **애자일** | 지속적 테스트 | 유연성 | 문서화 부족 | 변화 많음 |

### 과목 융합 관점 분석

#### 1. 안전(Safety) ↔ V-Model
- **IEC 62304**: 의료기기 소프트웨어 라이프사이클
- **DO-178C**: 항공우주 시스템
- **ISO 26262**: 자동차 기능 안전

#### 2. 테스트 자동화 ↔ V-Model
- **TDD (Test-Driven Development)**: 단위 테스트 우선
- **CI/CD**: 자동화된 V&V
- **테스트 피라미드**: Unit → Integration → E2E

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 의료기기 소프트웨어 개발
**상황**: IEC 62304 Class C 소프트웨어
**판단**:
1. **요구사항 단계**:
   - URS(User Requirements Specification)
   - SRS(Software Requirements Specification)
   - 인수 테스트 계획
2. **설계 단계**:
   - SAD(Software Architecture Description)
   - DDS(Detailed Design Specification)
   - 통합 테스트 계획
3. **코딩 단계**:
   - 코딩 표준 (MISRA-C)
   - 정적 분석 (Coverity)
   - 단위 테스트 (코드 커버리지 100%)
4. **테스트 단계**:
   - 단위 → 통합 → 시스템 → 인수
   - 각 단계별 산출물 검증

## Ⅴ. 기대효과 및 결론

### V-Model 기대 효과

| 효과 | V-Model 적용 시 | 적용 안 할 시 |
|------|---------------|-------------|
| **결함 조기 발견** | 설계 단계에서 발견 | 테스트 단계에서 발견 |
| **수정 비용** | 낮음 | 높음 |
| **품질 보증** | 체계적 | 임시적 |
| **문서화** | 포괄적 | 부족 |
| **변경 유연성** | 낮음 | - |

### 미래 전망

1. **Hybrid V-Model**: V-Model + 애자일
2. **DevSecOps**: 보안 테스트 통합
3. **AI 테스트**: 자동화된 V&V

### ※ 참고 표준/가이드
- **IEC 62304**: Medical device software
- **DO-178C**: Avionics systems
- **ISO 26262**: Automotive safety
