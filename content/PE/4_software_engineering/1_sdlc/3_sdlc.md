# SDLC (소프트웨어 생명주기, Software Development Life Cycle)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SDLC는 소프트웨어가 요구사항 도출부터 개발, 테스트, 배포, 운영, 폐기에 이르기까지의 **전체 수명 주기(Life Cycle)**를 체계적으로 관리하는 프로세스 프레임워크이며, 각 단계별 산출물(Deliverable)과 검토(Review)를 통해 품질을 보장한다.
> 2. **가치**: 계획(Plan) → 분석(Analysis) → 설계(Design) → 구현(Implementation) → 테스트(Test) → 배포(Deployment) → 유지보수(Maintenance)의 단계별 **관리 포인트**를 명확히 하여 **비용 산정, 일정 관리, 리스크 통제**를 가능하게 한다.
> 3. **융합**: 폭포수(Waterfall)에서 애자일(Agile), DevOps로 진화하며 CI/CD 파이프라인과 자동화 테스트와 결합하여 현대의 **지속적 배포(Continuous Deployment)** 문화의 기반이 되었다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
소프트웨어 생명주기(SDLC, Software Development Life Cycle)는 **"소프트웨어 제품의 개념화부터 폐기까지의 전체 과정을 구조화한 프로세스"**이다.

**핵심 목표**:
1. **품질 보장**: 체계적인 개발로 결함 최소화
2. **일정 준수**: 마일스톤(Milestone) 기반 진척 관리
3. **비용 통제**: 단계별 비용 산정 및 모니터링
4. **고객 만족**: 요구사항 충족 및 피드백 반영

### 💡 비유
SDLC는 **"건축물 건설 프로세스"**와 같다.
- **요구사항 분석**: 주거 공간 필요성, 예산, 위치 선정
- **설계**: 설계도 작성, 구조, 인테리어 디자인
- **구현**: 시공, 자재 투입
- **테스트**: 안전 검사, 하자 보수
- **배포**: 입주, 인도
- **유지보수**: 수리, 리모델링

### 등장 배경 및 발전 과정

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         SDLC 모델의 진화                                                │
└─────────────────────────────────────────────────────────────────────────────────────────┘

1960년대: 코딩 & 수정 (Coding & Fixing)
         - 체계 없는 개발, 버그 수정 위주
         ↓
1970년대: 폭포수 모델 (Waterfall Model)
         - Winston Royce 제안
         - 단계별 순차적 진행
         ↓
1980년대: 프로토타입, 나선형 (Spiral) 모델
         - 위험 분석 강조 (Boehm)
         - 반복적 개발
         ↓
1990년대: RUP (Rational Unified Process)
         - 유즈 케이스 중심, 4단계 (Inception, Elaboration, Construction, Transition)
         ↓
2000년대: 애자일 (Agile) 선언문
         - 스크럼(Scrum), XP(eXtreme Programming)
         - 변화 대응, 반복적 개발
         ↓
2010년대: DevOps, CI/CD
         - 개발-운영 통합, 자동화
         ↓
2020년대: AI 기반 개발, DevSecOps
         - LLM 코파일럿, 자동화 테스트
         - 보안 좌측 이동 (Shift-Left Security)
```

## Ⅱ. 아키텍처 및 핵심 원리

### SDLC 7단계 표준 모델

| 단계 | 주요 활동 | 산출물 (Deliverables) | 관여자 |
|------|----------|----------------------|--------|
| **1. 요구사항 수집** | 사용자 인터뷰, 요구사항 도출 | SRS (요구사항 명세서) | 고객, BA |
| **2. 분석 (Analysis)** | 타당성 조사, 기술 분석 | 타당성 보고서 | 아키텍트, PM |
| **3. 설계 (Design)** | 아키텍처, 상세 설계 | SDD (설계 문서), 모델 | 설계자, DBA |
| **4. 구현 (Implementation)** | 코딩, 단위 테스트 | 소스 코드, 단위 테스트 결과 | 개발자 |
| **5. 테스트 (Testing)** | 통합, 시스템, 인수 테스트 | 테스트 계획서, 결함 보고서 | QA, 테스터 |
| **6. 배포 (Deployment)** | 설치, 사용자 교육 | 배포 가이드, 매뉴얼 | 운영자 |
| **7. 유지보수 (Maintenance)** | 버그 수정, 기능 개선 | 패치, 릴리즈 노트 | 운영자 |

    [단점]
    • 피드백 늦음 (막대형 문서)
    • 유연성 낮음 (변경 비용 큼)
    • 사용자 확인 늦음 (배포 후 확인)

## Ⅲ. 융합 비교 및 다각도 분석

### SDLC 모델 비교

| 비교 항목 | 폭포수(Waterfall) | 프로토타입 | 나선형(Spiral) | 애자일(Agile) |
|----------|------------------|-----------|---------------|--------------|
| **특징** | 순차적 단계 | 시제품 중심 | 위험 분석 중심 | 반복적, 증분 |
| **변경 대응** | 어려움 | 중간 | 중간 | 용이 |
| **사용자 참여** | 초기만 | 지속 | 주기적 | 지속 |
| **문서화** | 많음 | 중간 | 많음 | 최소 |
| **피드백** | 늦음 | 중간 | 중간 | 빠름 |
| **적용** | 명확한 요구사항 | UI/UX 중요 | 대형 프로젝트 | 변화 많은 프로젝트 |
| **비용** | 초기 낮음 | 중간 | 높음 | 중간 |
| **품질** | 높음 (완벽한 계획) | 중간 | 높음 | 높음 (지속 테스트) |

### 과목 융합 관점 분석

#### 1. 프로젝트 관리 ↔ SDLC
- **PMBOK**: 5개 프로세스 그룹(시착, 계획, 실행, 감시/통제, 종료)
- **WBS**: 작업 분할 구조로 SDLC 단계 세분화
- **EVM**: 획득 가치 관리로 진척/비용 모니터링

#### 2. 품질 관리 ↔ SDLC
- **V-Model**: 각 개발 단계에 대응하는 테스트 단계
- **FTR (정형 기술 검토)**: 단계별 산출물 검토

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 대규모 시스템 구축 (공공기관)
**상황**: 연간 1억 건 거래 처리하는 금융 시스템, 3년 개발, 50억 원 예산
**판단**:
1. **SDLC 모델**: 하이브리드 (폭포수 + 애자일)
   - 초기: 엄격한 요구사항 분석, 설계 (폭포수)
   - 개발: 스크럼 기반 반복 개발 (애자일)
2. **관리 포인트**:
   - **Gate Review**: 각 단계 종료 시 경영진 검토
   - **CM (Configuration Management)**: 형상 관리 철저
   - **테스트**: 자동화 테스트 80% 이상 목표
3. **위험 관리**:
   - 병렬 개발 (핵심 기능/기반 기능)
   - PoC (Proof of Concept) 선행 검증

### 도입 시 고려사항 (체크리스트)

**기술적**:
- [ ] SDLC 모델 선택 기준: 프로젝트 특성, 요구사항 명확도
- [ ] 단계별 산출물 템플릿: SRS, SDD, STD 등
- [ ] 형상 관리 도구: Git, SVN, Jira

**운영적**:
- [ ] 거버넌스: 조직별 역할/책임(R&R) 정의
- [ ] 커뮤니케이션: 일일/주간 진척 공유

## Ⅴ. 기대효과 및 결론

### SDLC 기반 프로젝트 관리 기대 효과

| 효과 | SDLC 도입 전 | SDLC 도입 후 | 개선 폭 |
|------|------------|--------------|----------|
| **요구사항 추적** | 불가 | RTM(추적 매트릭스) | 100% |
| **품질 결함** | 프로덕션 발견 | 테스트 단계 발견 | 70% 조기 발견 |
| **일정 준수** | 30% 지연 | 10% 지연 | 67% 개선 |
| **비용 초과** | 50% 초과 | 20% 초과 | 60% 개선 |

### 미래 전망

1. **AI 기반 SDLC**: LLM이 요구사항 → 코드 자동 생성
2. **DevSecOps**: 보안을 SDLC 초기 단계에 통합 (Shift-Left)
3. **NoOps**: AI가 운영까지 자동화하는 완전 자동화 SDLC

### ※ 참고 표준/가이드
- **ISO/IEC 12207**: 소프트웨어 생명주기 공정
- **CMMI**: 성숙도 모델 (초기~최적화 5단계)
