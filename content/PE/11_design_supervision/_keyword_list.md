# 정보시스템 감리 및 S/W 아키텍처 설계 키워드 목록 (심화 확장판)

정보시스템 감리사, 정보관리기술사, 컴퓨터응용시스템기술사 합격을 위한 IT 감사(Audit), 공공/민간 감리 프레임워크, 소프트웨어 아키텍처 평가, 객체지향 설계 원칙(SOLID), 그리고 GoF 디자인 패턴 전 영역을 완벽 해부한 800대 핵심 키워드입니다.

---

## 1. 정보시스템 감리 개요 및 프레임워크 (80개)
1. 정보시스템 감리 (Information System Audit) 정의 - 제3자적 관점에서 정보시스템의 효과성, 효율성, 안전성을 종합적으로 점검하고 개선을 권고하는 활동
2. 감리의 3대 목적 - 효과성(Effectiveness, 목적 달성 여부), 효율성(Efficiency, 자원 최적화), 안전성/보안성(Security/Safeguard, 자산 보호 및 무결성 유지)
3. 감리 발주자 (Client) / 피감리인 (Auditee, 사업자/주관기관) / 감리 법인 (Auditor)
4. 전자정부법 제57조 (감리 의무화 규정) - 행정/공공기관의 일정 규모 이상 정보화 사업 의무 감리 지정
5. 정보시스템 감리기준 (행정안전부 고시)
6. 감리 프레임워크 (Audit Framework) 3차원 구조 - 감리 영역, 감리 관점, 감리 단계
7. 감리 영역 (Audit Domain) - 사업 관리, 응용 시스템, 데이터베이스, 시스템 아키텍처/보안
8. 감리 관점 (Audit Perspective) - 절차(Procedure), 산출물(Deliverable), 성과(Performance) 관점 점검
9. 감리 단계 (Audit Phase) - 사업의 진행 단계 (요구정의, 설계, 종료/구현)
10. 예방 감리 (Preventive Audit) / 상주 감리 (Resident Audit) - 사업 진행 중 상주하며 상시 조언
11. 3단계 감리 - 요구정의 단계 감리, 설계 단계 감리, 종료 단계 감리
12. 2단계 감리 - 설계 단계 감리, 종료 단계 감리 (요구정의 감리 생략 조건 사업)
13. 추가 감리 / 시정조치 확인 (Follow-up Audit) - 감리 지적 사항(조치 권고) 이행 여부 최종 점검
14. 감리 계획 수립 (Audit Planning) - 예비조사, 감리 일정 및 인력 배치, 감리 계획서 작성
15. 예비 조사 (Preliminary Survey) - 피감리인 인터뷰, 과업내용서/제안서 분석을 통해 감리 주안점 도출
16. 착수 회의 (Kick-off Meeting) - 감리 목적, 일정, 범위, 협조 사항 공유
17. 감리 수행 (Audit Execution) - 실지 감사, 인터뷰, 문서 검토, 자동화 도구 진단
18. 감리 보고서 (Audit Report) 구조 - 총평, 분야별 감리 결과, 시정 조치 권고 사항
19. 종료 회의 (Exit Meeting) - 감리 결과 발표 및 이견 조율
20. 조치 결과 확인 (시정조치 확인 보고서 발행)
21. ISACA (Information Systems Audit and Control Association) - 정보시스템 감사 통제 협회
22. CISA (Certified Information Systems Auditor) - 국제 공인 정보시스템 감사사
23. ITA/EA (Enterprise Architecture) 프레임워크 기반 감리
24. 위험 기반 감리 (Risk-based Audit) - 리스크가 높은 영역에 감리 자원 집중
25. 과업 대비표 (Task Traceability Matrix) - RFP, 제안서, 요구사항 명세서 간의 과업 이행 여부 추적 맵
26. 베이스라인 (Baseline) 검증 - 요구사항, 설계 산출물의 형상 통제 및 승인 기준선 점검
27. 사업 관리 (PM) 영역 감리 - 통합 관리, 범위 관리, 일정 관리, 품질 관리, 인력 관리, 의사소통 관리, 위험 관리 점검
28. 응용 시스템 영역 감리 - 기능 요구사항 구현 여부, 소프트웨어 아키텍처, UI/UX, 테스트 적정성 점검
29. 데이터베이스 영역 감리 - 데이터 모델링(ERD 정규화/반정규화), 데이터 표준 관리, 이행(Migration) 무결성, 성능 튜닝 점검
30. 시스템 아키텍처/보안 영역 감리 - 인프라(HW/SW/네트워크) 용량 산정 적정성, 보안 지침(시큐어 코딩, 개인정보보호) 준수 여부, 장애 복구(DR) 체계 점검
31. 정보화 사업 유형별 감리 - SI(구축), 운영/유지보수, 데이터 마이그레이션, 클라우드 전환 감리 등
32. 감리 자동화 도구 - 소스코드 정적 분석 도구(SAST), 데이터 품질 진단 도구, DB 성능 진단 도구, 성능 부하 테스트 도구 활용 지침
33. 기능점수 (Function Point) 검증 - 발주 시 규모 산정과 구현 종료 시 최종 기능점수(FP) 일치 여부 및 정산 단가 검토
34. SW 개발보안 (시큐어 코딩) 진단 - KISA 47개 보안 약점 기준 위반 소스코드 점검 의무
35. 웹 접근성 (Web Accessibility, KWCAG) 및 웹 호환성 준수 여부 진단
36. 감리 품질 관리 (Quality Assurance of Audit) - 감리 법인 내부의 감리 품질 통제 프로세스
37. PMO (Project Management Office) - 발주자를 대행하여 프로젝트 관리 및 기술 지원을 수행하는 조직 (사전 예방 위주)
38. PMO와 감리의 차이 - PMO는 발주자 편에서 능동적 문제 해결 개입, 감리는 제3자 관점에서 객관적 평가 및 권고 (감리 독립성)
39. 전자정부 표준 프레임워크 (eGovFrame) 아키텍처 및 적용 기준 점검 (스프링 부트 기반 공통 컴포넌트 활용성)
40. 클라우드 기반 정보화 사업 감리 가이드 - IaaS, PaaS, SaaS SLA 점검 및 데이터 이관/종속성(Lock-in) 점검 지침
41. 데이터 품질 진단 (Data Quality Audit) - 완전성, 유효성, 일관성, 정확성, 적시성, 보안성 (6대 데이터 품질 지표 점검)
42. 객관적 증거 (Objective Evidence) 수집 원칙 - 면담, 관찰, 문서 검토, 직접 진단(테스트) 기법
43. 샘플링 감리 기법 - 전수 조사가 불가능할 때 통계적(확률적) 샘플링 추출을 통한 진단
44. 감리인의 독립성 (Independence) 및 객관성 원칙 유지 조항
45. 사업자 지체 상금 (Liquidated Damages) 분쟁 예방을 위한 진척도 및 지연 사유 증빙 점검
46. 테스트 계획, 시나리오, 결과서 (단위, 통합, 시스템, 인수 테스트) 완결성 대조 확인
47. 요구사항 추적 매트릭스 (RTM, Requirements Traceability Matrix)의 양방향 추적성 검증
48. 소프트웨어 인도물 (Deliverables) 명세 합치 여부 점검
49. 유지보수 이관 (Hand-over) 및 운영자 교육, 매뉴얼 적정성 진단
50. 모바일 앱 사업 감리 - 앱 스토어 배포 기준, 취약점(위변조 방지, 루팅 탐지), 난독화 적용 여부 점검
51. AI / 빅데이터 사업 감리 - 학습 데이터 편향성, 알고리즘 성능 지표(F1-score, MAE 등), 개인정보 가명 처리 적정성 평가
52. IoT 구축 사업 감리 - 디바이스 펌웨어 무결성, 경량 통신 프로토콜 안정성 점검
53. 블록체인 사업 감리 - 스마트 컨트랙트 취약점 점검 및 합의 알고리즘 노드 구성 점검
54. ITIL / ITSM 프레임워크 기반 운영 감리 프로세스 점검 (SLA 달성 여부)
55. ISMS-P 인증 연계 - 개인정보 처리 시스템 취약점 조치 여부 병행 점검
56. 하드웨어 Sizing (용량 산정) - tpmC, SPECint 지표 기반 CPU, 메모리, 디스크 계산식 검증
57. 데이터 이행 (Data Migration) 성공 기준 검증 - 추출, 정제, 적재 데이터 건수 및 체크섬 대조
58. 고가용성 (HA, High Availability) 및 이중화 클러스터 페일오버(Failover) 시나리오 실지 테스트 참관
59. 보안 장비 (방화벽, IPS, WAF) 정책 룰셋(Rule-set) 최적화 상태 점검
60. 공공데이터 개방 (Open Data) 표준 규격 (CSV, JSON, REST API) 포맷 준수 감리
61. 재해 복구 (DR) 모의 훈련 참관 및 RTO/RPO 지표 목표 달성 확인
62. 백업 및 아카이빙 정책 (Full, Incremental, Differential 백업) 스케줄 점검
63. 소프트웨어 라이선스 컴플라이언스 - 불법 SW 사용 여부 및 오픈소스 라이선스(GPL, MIT 등) 고지 점검
64. 접근 통제 및 권한 관리 (접근 통제 행렬, RBAC 적용) 권한 오남용 및 퇴사자 권한 회수 이력 감사
65. 로그 및 감사 추적 (Audit Trail) - 시스템 로그가 6하 원칙에 따라 위변조 불가능한(WORM) 스토리지에 저장되는지 점검
66. 비밀번호 암호화 저장 방식 (일방향 해시 알고리즘 SHA-256 이상 + 솔팅 적용 여부) 진단
67. SSL/TLS 인증서 적용 및 통신 구간 암호화 미적용 취약점 점검
68. 소프트웨어 접근성 (Software Accessibility) - 장애인 및 고령자 UI 사용성 점검
69. 프로젝트 스폰서 및 추진 위원회 (Steering Committee) 의사 결정 지연 여부 평가
70. 형상 관리 저장소 (Git, SVN) 브랜치 병합 관리 및 지속적 통합(CI) 빌드 환경 평가
71. 소스코드 난독화 적용 여부 점검 (금융/앱 사업의 경우)
72. 개인정보 파기 정책 및 로그 보존 기간 준수 평가 (법적 요건)
73. 서버/OS/DB 패치 및 취약점 스캐닝 내역 적용 확인
74. 모의 해킹 (Penetration Test) 수행 내역서 기반 미비점 재점검
75. IT 예산 및 계약 행정 처리, 선금/잔금 정산 요건(과업 완료) 부합 검토
76. 서비스 데스크 (Service Desk) 및 인시던트 관리 체계 구축 확인
77. 사용자 만족도 조사 결과 분석 및 개선 조치 
78. BPR/ISP 연계 - 구축된 시스템이 당초 전략적 목표(AS-IS 대비 TO-BE 효과)를 달성했는지 사후 평가
79. 개발자 클린룸 망분리(VDI) 환경 및 보안 이동 경로 점검
80. COBIT 프로세스 평가 (APO, BAI, DSS, MEA) 모델 연계 통제 진단

## 2. 소프트웨어 아키텍처 원칙 및 설계 (60개)
81. 소프트웨어 아키텍처 (Software Architecture) - 소프트웨어를 구성하는 요소들 간의 관계와 속성, 설계 및 진화를 통제하는 기본 구조 
82. IEEE 1471 (ISO/IEC 42010) - 소프트웨어 아키텍처 명세 국제 표준 프레임워크
83. 아키텍처 주요 요소 - 이해관계자(Stakeholder), 관심사(Concern), 관점(Viewpoint), 뷰(View), 아키텍처 명세서
84. 필립 크루첸 (Philippe Kruchten)의 4+1 View 모델
85. 논리 뷰 (Logical View) - 최종 사용자 관점, 시스템의 기능적 요구사항 설계 (클래스/객체 다이어그램)
86. 프로세스 뷰 (Process View) - 시스템 통합자 관점, 병행성/동기화/성능 설계 (액티비티/상태/시퀀스 다이어그램)
87. 구현 뷰 (Implementation View / Development View) - 프로그래머 관점, 모듈의 구조 및 패키징 (컴포넌트 다이어그램)
88. 물리/배포 뷰 (Physical/Deployment View) - 시스템 엔지니어 관점, HW 배포 매핑 (배포 다이어그램)
89. 유스케이스 뷰 (Use Case View / +1 View) - 모든 뷰의 중심, 아키텍처 검증 기준 (유스케이스 다이어그램)
90. 아키텍처 드라이버 (Architecture Drivers) - 아키텍처 결정에 가장 큰 영향을 미치는 핵심 요구사항 (기능, 품질 속성, 제약 사항)
91. 품질 속성 시나리오 (Quality Attribute Scenario) - 자극원, 자극, 환경, 대상, 응답, 응답 척도 6가지 구성요소로 품질 명세
92. 아키텍처 평가 방법론론 - ATAM (Architecture Trade-off Analysis Method)
93. ATAM의 4개 페이즈 - 초기화 -> 평가 -> 결과 도출 -> 종합
94. 민감도점 (Sensitivity Point) - 특정 아키텍처 결정이 한 가지 품질 속성에 큰 영향을 미치는 지점
95. 상충점 (Trade-off Point) - 특정 아키텍처 결정이 한 품질 속성에는 긍정적이나 다른 품질 속성에는 부정적 영향을 미치는 교차점 (성능 vs 보안)
96. 리스크 (Risk) / 비리스크 (Non-risk) 도출
97. CBAM (Cost Benefit Analysis Method) - ATAM 확장, 아키텍처 결정에 경제성(비용 대비 효익, ROI) 관점 추가 평가
98. SAAM (Software Architecture Analysis Method) - 최초의 아키텍처 평가법, 수정 용이성에 초점 (ATAM의 전신)
99. ADR (Architecture Decision Record) - 아키텍처 설계 결정의 배경, 맥락, 결과를 기록하는 마크다운 문서 포맷 관리
100. 아키텍처 전술 (Architecture Tactics) - 특정 품질 속성을 향상시키기 위한 설계 결정 (가용성 전술, 성능 전술 등)
101. 객체 지향 설계 원칙 (SOLID) - 로버트 C. 마틴 (Uncle Bob) 제안
102. SRP (Single Responsibility Principle, 단일 책임 원칙) - 클래스는 단 하나의 변경 이유(책임)만 가져야 한다. (응집도 극대화)
103. OCP (Open-Closed Principle, 개방-폐쇄 원칙) - 확장에는 열려(Open) 있어야 하고, 수정에는 닫혀(Closed) 있어야 한다. (다형성/인터페이스 활용)
104. LSP (Liskov Substitution Principle, 리스코프 치환 원칙) - 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다. (상속의 올바른 사용)
105. ISP (Interface Segregation Principle, 인터페이스 분리 원칙) - 클라이언트는 자신이 사용하지 않는 메서드에 의존하지 않아야 한다. (인터페이스를 작고 구체적으로 분해)
106. DIP (Dependency Inversion Principle, 의존성 역전 원칙) - 고수준 모듈은 저수준 모듈의 구현에 의존해선 안되며, 둘 다 추상화(인터페이스)에 의존해야 한다. (제어의 역전 IoC)
107. DRY 원칙 (Don't Repeat Yourself) - 코드 중복 방지 (데이터 일관성 보장)
108. KISS 원칙 (Keep It Simple, Stupid) - 아키텍처와 코드는 최대한 단순하고 직관적이어야 함
109. YAGNI 원칙 (You Aren't Gonna Need It) - 당장 필요하지 않은 확장 기능은 미리 설계/구현하지 말 것 (오버엔지니어링 방지)
110. 최소 지식의 원칙 (Principle of Least Knowledge / 디미터의 법칙, Law of Demeter) - 객체 간 협력 시 이웃 객체의 내부 구조에 대해 알지 못해야 함 (A.getB().getC().do() 금지, 느슨한 결합)
111. 할리우드 원칙 (Hollywood Principle) - "Don't call us, we'll call you", 상위 모듈이 하위 모듈을 호출할지 결정 (콜백, IoC의 근간)
112. 응집도 (Cohesion) - 하나의 모듈 내부 요소들이 하나의 목적을 위해 뭉쳐 있는 정도 (높을수록 좋음: 기능적 응집도 최고)
113. 결합도 (Coupling) - 서로 다른 모듈 간의 상호 의존성 정도 (낮을수록 좋음: 데이터 결합도 최적)
114. 아키텍처 스타일 (Architecture Style / Pattern) - 반복되는 아키텍처 설계 문제를 해결하기 위한 구조적 해법
115. 계층형 아키텍처 (Layered Architecture) - 프레젠테이션, 비즈니스, 데이터 접근 계층 등 수직적 분할 (관심사 분리, Layer Isolation)
116. 헥사고날 아키텍처 (Hexagonal Architecture) / 포트 앤 어댑터 (Ports and Adapters) - 도메인(비즈니스 로직)을 중심에 두고, 외부(DB, UI) 인프라는 어댑터를 통해 포트로 통신하도록 격리
117. 클린 아키텍처 (Clean Architecture) - 엔티티, 유스케이스, 컨트롤러, 프레젠테이션/DB 계층 원형 구조. 의존성은 항상 외부에서 내부(도메인)로 향해야 함
118. 어니언 아키텍처 (Onion Architecture) - 인프라 의존성을 외부로 밀어내고 핵심 도메인을 중앙에 보호
119. 모델-뷰-컨트롤러 (MVC, Model-View-Controller) 아키텍처 - 데이터(M), 사용자 인터페이스(V), 비즈니스 흐름 제어(C) 분리 (웹 아키텍처 기본)
120. 이벤트 주도 아키텍처 (EDA, Event-Driven Architecture) - 상태 변경을 이벤트로 발행(Publish)하고 비동기적으로 구독(Subscribe)하여 결합도를 낮추는 구조 (메시지 브로커 활용)
121. CQRS (Command Query Responsibility Segregation) - 성능 확장을 위해 상태를 변경하는 명령(Command) 로직과 상태를 읽는 쿼리(Query) 로직과 데이터베이스를 논리/물리적으로 완전히 분리
122. 이벤트 소싱 (Event Sourcing) - CRUD 상태 업데이트 덮어쓰기 방식 대신, 모든 상태 변화 이벤트를 영구 스트림(Append-Only)으로 저장하여 일관성 보장 및 롤백 재생(Replay) 지원
123. 마이크로서비스 아키텍처 (MSA, Microservices Architecture) - 크고 단일화된 모놀리식 시스템을 도메인별 독립된 소규모 서비스로 분해, 개별 DB 할당, 독립적 배포 지원
124. API 게이트웨이 (API Gateway) 아키텍처 패턴 - MSA 진입점 통제
125. 서비스 메시 (Service Mesh) 아키텍처 - 마이크로서비스 간 통신 제어(라우팅, 트래픽 제어, mTLS, 로깅) 코드를 비즈니스 로직에서 분리하여 인프라(사이드카 프록시) 단에서 처리 (Istio 등)
126. 스트랭글러 피그 패턴 (Strangler Fig Pattern) - 모놀리식 레거시 시스템을 완전히 갈아엎지 않고, 전면에 게이트웨이를 둔 뒤 신규 기능부터 MSA로 점진적으로 가로채어(교체하여) 레거시를 고사시키는 전략
127. 서버리스 아키텍처 (Serverless / FaaS) - IaaS 프로비저닝 없이 함수 코드만 배포하면 트리거 이벤트 발생 시 런타임에 동적 스케일링/실행되는 구조
128. 마이크로 프론트엔드 (Micro Frontends) - 백엔드 MSA 사상을 프론트엔드 UI에도 적용, 페이지를 독립적 개발/배포 가능한 뷰 조각으로 분리 (Module Federation)
129. 도메인 주도 설계 (DDD, Domain-Driven Design) - 에릭 에반스, 소프트웨어 복잡성을 해결하기 위해 비즈니스 도메인(업무) 전문가와 개발자가 동일한 보편 언어(Ubiquitous Language)로 소통하며 설계
130. 바운디드 컨텍스트 (Bounded Context) - 모델의 경계, 마이크로서비스 분할의 핵심 기준
131. 애그리게이트 (Aggregate) - 일관성(트랜잭션)을 유지해야 하는 객체들의 군집, 애그리게이트 루트(Root)를 통해서만 접근 허용
132. 엔티티 (Entity) - 고유 식별자를 가진 객체 / 값 객체 (Value Object) - 식별자 없이 속성(불변성)만으로 정의되는 객체
133. 안티 코럽션 레이어 (ACL, Anti-Corruption Layer) - 외부/레거시 시스템 연동 시 서로 다른 도메인 모델이 오염되지 않도록 중간에 변환 어댑터 배치
134. 파이프-필터 (Pipe-Filter) 패턴 - 컴포넌트(Filter)들이 데이터 스트림을 파이프(Pipe)로 전달하며 순차적/병렬적 변환 수행 (Unix 파이프, 빅데이터 스트리밍)
135. 블랙보드 패턴 (Blackboard Pattern) - 중앙 공용 데이터(Blackboard)를 바탕으로 여러 전문 지식 모듈(Knowledge Source)이 점진적으로 문제 해결책을 모색 (음성인식, AI 초기 구조)
136. 마이크로 커널 아키텍처 (Microkernel / 플러그인 아키텍처) - 최소한의 핵심(Core) 모듈만 두고 나머지 확장 기능을 플러그인 형태로 조립 (IDE, 브라우저 확장)
137. 공간 기반 아키텍처 (Space-Based Architecture) - 트랜잭션 병목(중앙 DB)을 제거하기 위해 인메모리 데이터 그리드(Tuple Space)로 데이터를 분산 동기화 (초고도 동시 접속 시스템)
138. 아키텍처 결정 검증 리팩토링 (Architectural Refactoring)
139. 설계의 개념적 무결성 (Conceptual Integrity) 유지 방안
140. 설계 부채 (Design Debt) 측정 및 청산

## 3. GoF 디자인 패턴 (Design Patterns) 생성 및 구조 (50개)
141. 디자인 패턴 (Design Pattern) - 에리히 감마 등 GoF (Gang of Four) 4인이 정리, 객체지향 소프트웨어 설계에서 자주 발생하는 문제에 대한 검증된 해법(템플릿)
142. 23가지 GoF 패턴 분류 - 생성 패턴(Creational, 5개), 구조 패턴(Structural, 7개), 행위 패턴(Behavioral, 11개)
143. 생성 패턴 (Creational Patterns)의 목적 - 객체 인스턴스 생성 프로세스를 캡슐화하여 클라이언트와 생성 로직의 결합도 분리
144. 싱글톤 (Singleton) 패턴 - 애플리케이션 전체에서 클래스의 인스턴스가 오직 1개만 생성되도록 보장하고, 어디서든 전역 접근점을 제공 (DB 커넥션 풀, 로거 등에 사용)
145. 싱글톤 구현 기법 - Double Checked Locking (더블 체크 락킹), Enum 싱글톤, Lazy Initialization (지연 초기화)의 동시성 이슈 대응
146. 팩토리 메서드 (Factory Method) 패턴 / 가상 생성자 패턴 - 객체를 생성하는 인터페이스는 상위 클래스가 정의하되, 실제 생성할 객체(인스턴스)의 클래스는 하위 클래스(서브클래스)가 결정하도록 팩토리 메서드 위임 (상속 활용)
147. 추상 팩토리 (Abstract Factory) 패턴 - 서로 연관되거나 의존적인 여러 객체군(Family)을, 구체적인 클래스를 지정하지 않고 하나의 팩토리 인터페이스를 통해 일괄 생성 (UI 테마 변환-Mac UI 팩토리, Win UI 팩토리)
148. 빌더 (Builder) 패턴 - 복잡한 인스턴스의 생성 과정(초기화)과 표현 방법을 분리, 메서드 체이닝(Method Chaining)을 이용해 다양한 구성의 객체를 단계별로 유연하게 생성 (파라미터가 많은 생성자 단점 극복)
149. 프로토타입 (Prototype) 패턴 - `new` 키워드로 인스턴스를 매번 생성하는 비용이 클 때, 미리 생성된 원본 객체(Prototype)를 복제(Clone / 깊은 복사)하여 새로운 객체를 셍성
150. 구조 패턴 (Structural Patterns)의 목적 - 클래스나 객체들을 조합하여 더 크고 복잡한 구조를 설계, 인터페이스 구성을 단순화 (상속/합성 융합)
151. 어댑터 (Adapter) 패턴 / 래퍼 (Wrapper) 패턴 - 호환되지 않는 인터페이스를 가진 두 클래스를 연결, 클라이언트가 요구하는 인터페이스로 래핑하여 함께 동작할 수 있도록 변환 (기존 코드 재사용 목적)
152. 객체 어댑터 (합성 위임 기반) vs 클래스 어댑터 (다중 상속 기반)
153. 브리지 (Bridge) 패턴 / 핸들/바디 패턴 - 기능의 클래스 계층(추상부)과 구현의 클래스 계층(구현부)을 분리하여 각각 독립적으로 확장할 수 있게 함 (상속의 조합 폭발 방지, 런타임에 구현체 교체)
154. 컴포지트 (Composite) 패턴 - 단일 객체(Leaf)와 복합 객체(Composite)를 동일한 인터페이스로 다루어, 객체들의 부분-전체(Part-Whole) 계층 트리 구조를 표현 (디렉터리-파일 구조 모델링)
155. 데코레이터 (Decorator) 패턴 - 상속을 통한 서브클래싱 대신, 객체를 장식자(Decorator) 객체로 동적으로 감싸서(래핑) 런타임에 새로운 기능과 책임을 유연하게 추가 (커피에 시럽, 우유 래핑)
156. 퍼사드 (Facade) 패턴 - 복잡하게 얽힌 서브시스템들의 내부 인터페이스들을 숨기고, 외부 클라이언트가 쉽게 사용할 수 있도록 단순화된 고수준의 단일 통합 인터페이스(Facade) 제공 (결합도 대폭 하락)
157. 플라이웨이트 (Flyweight) 패턴 - 인스턴스 생성이 너무 많아 메모리 부하가 심할 때, 고유 상태(Intrinsic State, 불변)는 메모리 팩토리에 공유 캐싱하고, 외부 상태(Extrinsic State, 가변)만 외부에서 주입받아 객체 재사용 극대화 (워드프로세서의 글자 폰트 객체, 게임의 다수 유닛)
158. 프록시 (Proxy) 패턴 - 실제 원본 객체(Real Subject)에 대한 접근을 제어하거나, 접근 전후에 부가 작업(로깅, 지연 로딩, 보안 통제)을 수행하기 위해 대리자(Proxy) 객체를 제공
159. 프록시 패턴 유형 - 가상 프록시 (Virtual Proxy, 리소스 무거운 객체 지연 로딩), 보호 프록시 (Protection Proxy, 접근 제어 권한 검사), 원격 프록시 (Remote Proxy, 원격 통신 캡슐화)
160. 디자인 패턴의 활용 원칙 - OCP (확장, 폐쇄)와 DIP (추상 인터페이스 의존) 구현 도구
161. 안티 패턴 (Anti-Pattern) - 잘못된 소프트웨어 구조나 설계 습관 (스파게티 코드, 갓 클래스 God Class, 황금 망치 Golden Hammer)
162. 갓 클래스 (God Class / Blob) - 단일 클래스에 너무 많은 기능과 데이터(전역 변수)가 집중되어 응집도가 낮고 수정이 불가능한 현상 (SRP 위반)
163. 싱글톤 패턴의 단점 (안티패턴 관점) - 전역 상태 오염, 단위 테스트 목업(Mock) 어려움, 객체 간 암묵적 강결합 유발. (해결책: 의존성 주입 DI 프레임워크 활용)
164. 어댑터와 퍼사드의 차이 - 어댑터는 인터페이스 호환을 위한 1:1 변환, 퍼사드는 복잡한 다수 인터페이스 캡슐화 통합 1:N 제공
165. 브리지 패턴과 전략(Strategy) 패턴 비교 - 브리지는 구조(추상-구현 구조 설계) 관점, 전략은 행위(알고리즘 교체 동작) 관점
166. 데코레이터와 프록시 비교 - 데코레이터는 객체에 기능(책임) 추가가 목적, 프록시는 객체에 대한 접근 제어가 목적
167. 추상 팩토리 패턴 팩토리 클래스 도출 (추상 모듈의 교체 비용 통제)
168. 템플릿 메서드와 팩토리 메서드의 결합
169. 객체 생성을 캡슐화하는 정적 팩토리 메서드(Static Factory Method) 구현 기법 (Effective Java 권장)
170. 모듈 (Module) 패턴 - 클로저(Closure)를 활용한 자바스크립트 등 비객체지향 언어 정보 은닉 구조

## 4. GoF 디자인 패턴 행위 (Behavioral) 및 구조 최적화 (60개)
191. 행위 패턴 (Behavioral Patterns)의 목적 - 객체나 클래스 간의 알고리즘 분배 및 책임, 협력, 메시지 통신 방식을 설계 (결합도 최소화)
192. 옵저버 (Observer) 패턴 / 발행-구독 (Pub/Sub) 구조 - 어떤 객체(Subject)의 상태가 변하면 그 객체에 의존성(구독)을 둔 다수의 옵저버들에게 자동으로 알림(Notify)이 가도록 일대다(1:N) 의존성 정의 (MVC 모델의 핵심)
193. 전략 (Strategy) 패턴 / Policy 패턴 - 동일한 계열의 알고리즘들을 캡슐화하고 인터페이스화하여, 런타임 시점에 클라이언트 코드 변경 없이 알고리즘(전략)을 쉽게 상호 교체할 수 있도록 설계 (정렬 알고리즘 선택, 결제 방식 선택)
194. 템플릿 메서드 (Template Method) 패턴 - 부모(추상) 클래스에 알고리즘의 전체 뼈대(템플릿)와 공통 실행 흐름을 정의하고, 세부적인 특정 스텝 구현은 자식(구체) 클래스로 지연시켜 오버라이딩 (제어의 역전, 훅 메서드 Hook Method 사용)
195. 팩토리 메서드와 템플릿 메서드의 관계 - 팩토리 메서드는 객체 생성에 특화된 템플릿 메서드의 일종
196. 커맨드 (Command) 패턴 - 클라이언트의 요청(Action) 자체를 객체(Command)로 캡슐화하여, 요청자(Invoker)와 수신자(Receiver)의 결합을 분리. 실행 취소(Undo), 재실행(Redo), 매크로 로깅, 큐(Queue) 작업에 필수 (스마트홈 리모컨 버튼)
197. 상태 (State) 패턴 - 객체의 내부 상태(State)가 바뀜에 따라 객체의 행위가 달라지도록, 상태 객체 자체를 캡슐화하여 위임. 클라이언트 관점에서는 객체의 클래스가 동적으로 바뀌는 것처럼 보임 (자판기, 미디어 플레이어 상태)
198. 상태 패턴 vs 전략 패턴 - 전략 패턴은 외부 클라이언트가 능동적으로 전략을 교체, 상태 패턴은 객체 내부 로직이나 이벤트에 의해 자기 스스로 상태를 전이
199. 책임 연쇄 (Chain of Responsibility) 패턴 - 요청을 처리할 수 있는 객체들이 사슬(Chain) 형태로 연결되어 있어, 처리할 수 있는 객체를 만날 때까지 요청을 순차적으로 전달 (결재 라인 승인, 이벤트 핸들러 버블링)
200. 책임 연쇄의 장단점 - 결합도 감소 및 유연한 룰 추가 가능, 하지만 체인 맨 끝까지 처리 못할 경우 요청 유실 가능성
201. 중재자 (Mediator) 패턴 - 객체 간의 M:N 복잡한 직접 통신 네트워크를 1:N 구조로 변환하여 중앙의 중재자 객체(Mediator)를 통해서만 통신하도록 강제. (항공 교통 관제탑, 채팅방 서버)
202. 중재자 패턴과 옵저버 패턴 혼합 구현 아키텍처
203. 방문자 (Visitor) 패턴 - 데이터 요소 객체 구조(컴포지트 구조 등)를 변경하지 않고도, 객체를 순회하며 새로운 연산이나 기능을 추가하기 위해 방문자 객체를 주입 (Double Dispatch 매커니즘 활용, 컴파일러 구문 트리 분석 시 사용)
204. 이터레이터 (Iterator) 패턴 / 반복자 - 컬렉션(List, Tree 등)의 내부 구조(배열인지 링크드리스트인지)를 노출하지 않고, 그 안의 원소들을 순차적으로 접근할 수 있는 통일된 인터페이스(hasNext, next) 제공 
205. 메멘토 (Memento) 패턴 / 토큰 패턴 - 객체의 캡슐화(정보 은닉)를 훼손하지 않으면서 객체의 내부 상태(특정 시점 스냅샷)를 외부에 저장하고, 필요 시 복원(Restore/Undo)할 수 있게 하는 패턴 (Originator, Memento, Caretaker 3요소)
206. 해석자 (Interpreter) 패턴 - 언어나 문법의 규칙을 클래스로 표현하여, 그 언어로 작성된 문장을 해석(AST 구문 트리 파싱)하고 실행하는 평가(Evaluate) 기능 구조 (정규 표현식 엔진, SQL 파서)
207. 널 객체 패턴 (Null Object Pattern) - 객체가 없을 때 NULL 참조 대신 예외/분기를 방지하기 위해 '아무 일도 하지 않는' 빈 껍데기 디폴트 객체를 반환 (NullPointerException 회피)
208. 동기화 패턴 - 가드 서스펜션 (Guarded Suspension)
209. 읽기-쓰기 락 (Read-Write Lock) 패턴 (행위 다중 제어)
210. 모니터 객체 (Monitor Object) 패턴 (자바 동기화 원리)
211. 액티브 오브젝트 (Active Object) 패턴 - 비동기 메서드 호출과 실행 스레드 분리

214. 하프-싱크/하프-어싱크 (Half-Sync/Half-Async) 패턴 - 큐를 사이로 비동기 수신과 동기 워커 스레드 풀 분리 성능 병목 분해
