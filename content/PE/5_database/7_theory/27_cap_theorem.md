# CAP 정리 (CAP Theorem)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CAP 정리는 **"분산 **시스템**은 **Consistency**(일관성), **Availability**(가용성), **Partition Tolerance**(분할 **내성) **3가지 **중 **최대 **2가지**만 **동시**에 **만족**할 수 **있다"는 **정리**이다.
> 2. **속성**: **C**(Consistency)는 **모든 **노드**가 **동시에 **같은 **데이터**를 **반환**, **A**(Availability)는 **모든 **요청**이 **응답**을 **보장**, **P**(Partition Tolerance)는 **네트워크 **분할**에도 **시스템 **동작**을 **의미**한다.
> 3. **선택**: **CA**(RDBMS, **단일 **데이터 **센터), **CP**(MongoDB, **Redis, **HBase), **AP**(Cassandra, **CouchDB, **DynamoDB)로 **구분**되며 **실제로는 **P**(분할)은 **불가피**하므로 **C vs A** **선택**이 **핵심**이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
CAP 정리는 **"분산 시스템 트레이드오프"**이다.

**CAP 정의**:
| 속성 | 설명 | 예시 |
|------|------|------|
| **C** | Consistency (일관성) | 모든 노드가 같은 데이터 반환 |
| **A** | Availability (가용성) | 모든 요청이 응답 수신 |
| **P** | Partition Tolerance (분할 내성) | 네트워크 단절 시에도 동작 |

### 💡 비유
CAP는 ****재택근무 **팀 ****협업 ****과 같다.
- **C**: 모두가 같은 정보
- **A**: 언제든 연락 가능
- **P**: 통신 두절 시에도 작업

## Ⅱ. 아키텍처 및 핵심 원리

### CAP 시나리오

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         CAP Scenarios                                                     │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Network Partition (P occurs):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Node A                   Network Partition               Node B                         │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Data: X = 10              X           X              Data: X = 5                      │  │  │
    │  │                                                               │                         │  │  │
    │  │  Client writes              │           │              Client reads                   │  │  │
    │  │  X = 20                    │           │              X = ?                          │  │  │
    │  │                           │           │                                              │  │  │
    │  │  Option 1: Return error    │           │  Option 2: Return old value (5)             │  │  │
    │  │  → Choose C (deny write)   │           │  → Choose A (serve read)                  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    CA System (No Partition Tolerance):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Single Node RDBMS                                                                      │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Client → [Single Database] → Response                                                │  │  │
    │  │                                                                                       │  │  │
    │  │  • Consistent: Always return latest data                                              │  │  │
    │  │  • Available: Always respond (unless system down)                                    │  │  │
    │  │  • Not Partition Tolerant: Single point of failure                                   │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    CP System (Choose Consistency):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  MongoDB (Primary-Secondary)                                                            │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Primary          Secondary        Client                                            │  │  │
    │  │  ┌───────────┐   ┌───────────┐     ┌───────────┐                                   │  │  │
    │  │  │ X = 20    │   │ X = 10    │     │ Read X    │                                   │  │  │
    │  │  └─────┬─────┘   └─────┬─────┘     └─────┬─────┘                                   │  │  │
    │  │        │ X              │ Partition    │                                          │  │  │
    │  │        └────────────────X              │                                          │  │  │
    │  │                                        │                                          │  │  │
    │  │  → Read from Primary: Returns 20 (Consistent)                                       │  │  │
    │  │  → If Primary unavailable: Error (Not Available)                                    │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    AP System (Choose Availability):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Cassandra (Eventual Consistency)                                                       │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Node A           Node B           Client                                            │  │  │
    │  │  ┌───────────┐   ┌───────────┐     ┌───────────┐                                   │  │  │
    │  │  │ X = 20    │   │ X = 10    │     │ Read X    │                                   │  │  │
    │  │  └─────┬─────┘   └─────┬─────┘     └─────┬─────┘                                   │  │  │
    │  │        │ Partition      │                │                                          │  │  │
    │  │        │                │                ▼                                          │  │  │
    │  │        │                │            Returns 10 (Available, Stale)                  │  │  │
    │  │        │                │                                                           │  │  │
    │  │  → Both nodes respond: Available                                                   │  │  │
    │  │  → Inconsistent until partition heals                                               │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### CAP 조합별 DBMS

| 조합 | DBMS | 특징 | 사용처 |
|------|------|------|--------|
| **CA** | MySQL (단일), PostgreSQL | 단일 데이터 센터 | 전통적 RDBMS |
| **CP** | MongoDB, Redis, HBase | 강한 일관성 | 금융, 재고 |
| **AP** | Cassandra, DynamoDB, CouchDB | 최종 일관성 | SNS, 로그 |

### 일관성 수준

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Consistency Levels                                                 │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Strong Consistency (CP):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Write X = 20 (Node A)                                                               │  │  │
    │  │  • Replicate to all nodes before ACK                                                │  │  │
    │  │  • Next read returns 20 (any node)                                                   │  │  │
    │  │  • Latency: High                                                                      │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Eventual Consistency (AP):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Write X = 20 (Node A)                                                               │  │  │
    │  │  • ACK immediately                                                                    │  │  │
    │  │  • Replicate async to other nodes                                                     │  │  │
    │  │  • Read might return old value temporarily                                           │  │  │
    │  │  • Latency: Low                                                                       │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Read Your Writes (RYW):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Client writes X = 20                                                                 │  │  │
    │  │  → Subsequent reads from same client return 20                                      │  │  │
    │  │  → Other clients might see old value temporarily                                    │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### PACELC 정리

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         PACELC Theorem (CAP Extension)                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    In case of Partition (P):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Trade-off between Availability (A) and Consistency (C)                                │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Else (E - no partition):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Trade-off between Latency (L) and Consistency (C)                                     │  │
    │                                                                                         │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Low Latency (LC)                     High Latency (CL)                              │  │  │
    │  │  • Async replication               • Sync replication                               │  │  │
    │  │  • Eventual consistency             • Strong consistency                            │  │  │
    │  │  • Example: Cassandra               • Example: MongoDB                             │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### CAP 기대 효과

| 선택 | 장점 | 단점 | 사용처 |
|------|------|------|--------|
| **CA** | 일관 + 가용 | 단일 장애점 | 단일 DC |
| **CP** | 강한 일관성 | 가용성 저하 | 금융, 재고 |
| **AP** | 높은 가용성 | 데이터 부정확 | SNS, 로그 |

### 모범 사례

1. **단일 DC**: CA (RDBMS)
2. **분산 CP**: MongoDB, HBase
3. **분산 AP**: Cassandra, DynamoDB
4. **하이브리드**: 다중 저장소 (Redis + MySQL)

### 미래 전망

1. **NewSQL**: 강한 일관성 + 분산 (CockroachDB)
2. **CRDT**: Conflict-free Replicated Data Types
3. **Multi-model**: 다중 CAP 모델

- **Brewer**: CAP Theorem Origin
- **Gilbert & Lynch**: CAP Proof
- **Abadi**: PACELC Theorem
