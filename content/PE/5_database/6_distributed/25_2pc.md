# 2단계 커밋 (Two-Phase Commit, 2PC)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 2PC는 **"분산 **데이터베이스 **환경**에서 **트랜잭션 **원자성**을 **보장**하는 **프로토콜"**으로, **Coordinator**(조정자)와 **Participants**(참여자)가 **2단계**(Prepare, Commit)로 **합의**한다.
> 2. **동작**: **Phase 1(Prepare)**에서 **Coordinator**가 **모든 **참여자**에게 **커밋 **가능 **여부**를 **묻고 **모두**가 **Yes**면 **Phase 2(Commit)**에서 **실제 **커밋**을 **전파**한다.
> 3. **문제**: **Blocking**(차단) **프로토콜**로 **Coordinator **장애** 시 **모든 **참여자**가 **대기**하며 **Single Point of Failure**와 **Blocking 문제**를 **3PC**(3-Phase Commit)나 **Paxos/Raft**로 **해결**한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
2PC는 **"분산 트랜잭션 합의 프로토콜"**이다.

**구성 요소**:
- **Coordinator**: 트랜잭션 조정자
- **Participants**: 분산 DB 노드
- **X/Open XA**: 표준 인터페이스

### 💡 비유
2PC는 **"결혼 **식 **사회 ****와 같다.
- **신부/신랑**: 참여자
- **사회**: Coordinator
- **물음**: "결혼하시겠습니까?"
- **결과**: 모두 Yes → 결혼 성립

## Ⅱ. 아키텍처 및 핵심 원리

### 2PC 프로토콜

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Two-Phase Commit Protocol                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Coordinator                         Participant 1    Participant 2    Participant 3    │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  PHASE 1: VOTING (Prepare)                                                            │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │
    │  │  │  ──────────────────────→  PREPARE          PREPARE          PREPARE              │  │  │  │
    │  │  │                          (Can commit?)     (Can commit?)     (Can commit?)        │  │  │  │
    │  │  │                                                                       │            │  │  │  │
    │  │  │  └────────────────────←  YES              YES              YES                   │  │  │  │
    │  │  │                          (Ready)          (Ready)          (Ready)               │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  │  → If all YES → Proceed to Phase 2                                                       │  │  │
    │  │  → If any NO → Abort all                                                                │  │  │
    │  │                                                                                          │  │  │
    │  │  PHASE 2: COMMIT / ABORT                                                                │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │
    │  │  │  ──────────────────────→  COMMIT           COMMIT           COMMIT               │  │  │  │
    │  │  │                          (Do commit)      (Do commit)      (Do commit)           │  │  │  │
    │  │  │                                                                       │            │  │  │  │
    │  │  │  └────────────────────←  ACK              ACK              ACK                   │  │  │  │
    │  │  │                          (Committed)      (Committed)      (Committed)           │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 2PC vs 3PC

| 구분 | 2PC | 3PC |
|------|-----|-----|
| **단계** | Prepare, Commit | CanCommit, PreCommit, DoCommit |
| **Blocking** | O | X |
| **복잡도** | 낮음 | 높음 |
| **실사용** | 널리 | 드묾 |

### 2PC 장단점

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         2PC Trade-offs                                                     │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Advantages:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Atomicity: All participants commit or all abort                                       │  │
    │  • Simplicity: Simple protocol, easy to implement                                       │  │
    │  • Standard: XA specification widely supported                                           │  │
    │  • Consistency: Strong consistency guaranteed                                             │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Disadvantages:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Blocking: Participants block while waiting for coordinator                            │  │
    │  • Single Point of Failure: Coordinator crash blocks all                                │  │
    │  • Performance: 2 round trips (prepare + commit)                                        │  │
    │  • Locking: Resources held during protocol                                              │  │
    │  • Partition: Network partition causes blocking                                         │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### Java XA 예시

```java
// Java XA Transaction (JTA)
import javax.transaction.*;
import javax.transaction.xa.*;

public class DistributedTransaction {
    public void transferMoney() throws Exception {
        XAConnection xaConn1 = ds1.getXAConnection();
        XAResource xaRes1 = xaConn1.getXAResource();

        XAConnection xaConn2 = ds2.getXAConnection();
        XAResource xaRes2 = xaConn2.getXAResource();

        // Start transaction
        Xid xid = createXid();
        xaRes1.start(xid, XAResource.TMNOFLAGS);
        xaRes2.start(xid, XAResource.TMNOFLAGS);

        // Execute operations
        updateAccount1(xaConn1);
        updateAccount2(xaConn2);

        // End transaction
        xaRes1.end(xid, XAResource.TMSUCCESS);
        xaRes2.end(xid, XAResource.TMSUCCESS);

        // Phase 1: Prepare
        int prep1 = xaRes1.prepare(xid);
        int prep2 = xaRes2.prepare(xid);

        // Phase 2: Commit (if all prepared)
        if (prep1 == XAResource.XA_OK && prep2 == XAResource.XA_OK) {
            xaRes1.commit(xid, false);
            xaRes2.commit(xid, false);
        } else {
            xaRes1.rollback(xid);
            xaRes2.rollback(xid);
        }
    }
}
```

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 분산 트랜잭션 선택
**상황**: MySQL + PostgreSQL 간 데이터 일관성
**판단**: 2PC 사용 (성능 저하 감수)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Decision Tree                                                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Is distributed transaction needed?
    │
    ├─ No → Use local transaction
    │
    └─ Yes → Can you tolerate eventual consistency?
              │
              ├─ Yes → Use Saga pattern (async compensation)
              │
              └─ No → Use 2PC (XA)
                        │
                        └─ Accept blocking & performance penalty
```

## Ⅴ. 기대효과 및 결론

### 2PC 기대 효과

| 구분 | Local Tx | 2PC |
|------|----------|-----|
| **원자성** | 단일 DB | 분산 DB |
| **성능** | 빠름 | 느림 (2 round trip) |
| **차단** | 짧음 | 김 |

### 모범 사례

1. **필요시만**: 진짜 분산 트랜잭션 필요할 때
2. **Timeout**: 적절한 타임아웃 설정
3. **모니터링**: in-doubt 트랜잭션 추적
4. **대안**: Saga, Eventual Consistency 고려

### 미래 전망

1. **Saga**: 비동기 보상 패턴
2. **Consensus**: Raft, Paxos 활용
3. **Microservices**: 서비스당 DB

### ※ 참고 표준/가이드
- **X/Open XA**: Distributed Transaction Processing
- **JTA**: Java Transaction API
- **ISO/IEC**: 2PC Standard
