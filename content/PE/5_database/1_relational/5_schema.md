# 스키마 (Schema)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스키마는 데이터베이스의 **논리적 구조(Structure), 제약조건(Constraints), 관계(Relationships)**를 정의한 **메타데이터(Metadata)**의 집합으로, 데이터 무결성을 보장하고 데이터 독립성을 실현하는 데이터베이스의 청사진도다.
> 2. **가치**: **3단계 스키마 아키텍처**를 통해 사용자 관점(외부 스키마), 조직 전체 관점(개념 스키마), 물리적 저장 관점(내부 스키마)의 추상화 계층을 제공하며, DDL(CREATE, ALTER, DROP)을 통해 정의/수정/삭제된다.
> 3. **융합**: 데이터베이스 설계의 핵심 산출물이자, ORM(Object-Relational Mapping), ETL(Extract-Transform-Load), 데이터 웨어하우스(차원 모델링)의 기반이 되며, NoSQL의 Schema-on-Read와 대비되는 **Schema-on-Write** 정합성을 보장한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
스키마(Schema)는 **"데이터베이스의 구조, 제약, 관계를 정의한 메타데이터의 집합"**이다.

**어원**:
- 그리스어어 "스케마"(σχῆμα) = 형태, 형식, 계획
- 데이터베이스에서는 **"데이터의 뼈대(Framework)"**

### 💡 비유
스키마는 **"건축 설계도"**와 같다.
- **구조**: 기둥집, 방의 배치 (테이블, 컬럼)
- **제약**: 내화력, 내장 하중 (데이터 무결성)
- **관계**: 기둥 간 연결 통로 (외래키)

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         스키마 개념의 진화                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

파일 시스템:
    • 스키마 없음
    • 프로그램이 파일 구조 정의
    • 데이터 종속성 문제
         ↓
관계형 모델 (1970년대):
    • 논리적 스키마 개념 등장
    • E.F. Codd의 관계형 모델
    • 테이블, 릴레이션, 키(Key)
         ↓
3단계 스키마 (ANSI/SPARC 1975):
    • 외부/개념/내부 스키마
    • 데이터 독립성 보장
         ↓
현대 데이터베이스:
    • 객체 관계형 스키마
    • XML, JSON, NoSQL 스키마
    • 동적 스키마 (Schema Registry)
```

## Ⅱ. 아키텍처 및 핵심 원리

### 스키마 구성 요소

| 요소 | 설명 | 예시 | DDL |
|------|------|------|-----|
| **테이블(Table)** | 데이터 저장 단위 | employees | CREATE TABLE |
| **컬럼(Column)** | 속성 (Attribute) | name, salary | VARCHAR(100) |
| **데이터 타입(DataType)** | 도메인 | INT, VARCHAR | INT, DECIMAL |
| **제약조건(Constraint)** | 무결성 규칙 | PK, FK, NOT NULL | PRIMARY KEY |
| **관계(Relationship)** | 테이블 간 연결 | 1:N, M:N | FOREIGN KEY |

## Ⅲ. 융합 비교 및 다각도 분석

### NoSQL Schema-on-Read vs RDBMS Schema-on-Write

| 비교 항목 | RDBMS (Schema-on-Write) | NoSQL (Schema-on-Read) |
|----------|------------------------|----------------------|
| **스키마 검증 시점** | 데이터 삽입/갱신 시 | 조회 시 |
| **유연성** | 엄격 (형식 맞춰어야 함) | 유연 (다양한 구조 허용) |
| **확장성** | 낮음 (ALTER TABLE 필요) | 높음 (즉시 저장 가능) |
| **데이터 타입** | 강력 (INT, VARCHAR 등) | 약한 (JSON, BLOB) |
| **일관성** | ACID 트랜잭션 보장 | BASE/Eventual Consistency |
| **적용** | 금융, 재무, ERP | SNS, 로그, IoT |

### 과목 융합 관점 분석

#### 1. 데이터 모델링 ↔ 스키마
- **ERD(다이어그램)**: 개념 스키마 설계
- **정규화**: 스키마 분해, 이상 현상 제거
- **반정규화**: 성능 최적화를 위한 스키마 병합

#### 2. 빅데이터 ↔ 스키마
- **Schema-less**: 필요한 경우 스키마 없이 저장
- **Dynamic Schema**: 시간에 따라 스키마 진화

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 대형 테이블의 컬럼 추가
**상황**: 1억 건의 orders 테이블에 고객 등급 컬럼 추가
**판단**:
1. **ALTER TABLE ADD COLUMN**:
   - Oracle: 빠른 추가 (메타데이터만 변경)
   - MySQL: 테이블 전체 재구축 (시간 소요)
2. **Default 값 설정**:
   - 기존 데이터: NULL 또는 기본값
   - 신규 데이터: 기본값 자동 적용

## Ⅴ. 기대효과 및 결론

### 스키마 관리의 기대 효과

| 효과 | 스키마 없을 시 | 스키마 있을 시 |
|------|--------------|--------------|
| **데이터 일관성** | 위배 | 보장 |
| **중복 최소화** | 어려움 | 정규화로 해결 |
| **질의 성능** | 전체 스캔 | 인덱스로 빠름 |
| **변경 영향** | 전체 앱영향 | 뷰로 격리 |

### 미래 전망

1. **자동 스키마 발견**: AI가 스키마 추론
2. **JSON 스키마**: PostgreSQL JSONB, MySQL JSON
3. **GraphQL 스키마**: 클라이언트 주도 쿼리

### ※ 참고 표준/가이드
- **ANSI/ISO SQL**: 스키마 정의 언어
- **Information Schema**: 데이터베이스 메타데이터 표준
