# 매핑 룰 (Mapping Rule)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 매핑 룰은 ER 모델의 **개체-관계(Entity-Relationship) 구조를 관계형 데이터베이스의 **테이블-릴레이션(Table-Relation) 구조로 변환하는 논리적 설계(Logical Design)의 표준 규칙**으로, 강(Strong) 개체와 약(Weak) 개체를 구분하고, 각 관계의 카디널러티(1:1, 1:N, M:N)별로 다른 매핑 전략을 적용한다.
> 2. **가치**: **개체-테이블 변환(1개체=1개 테이블)**, **관계-외래키 변환(관계=FK)**, **속성-컬럼 변환**의 3단계 매핑 룰을 통해 개념적 설계(ERD)에서 물리적 설계(DDL)로 자동화 가능하며, **M:N 관계의 연결 테이블(Association Table)** 생성을 통해 데이터 정규화를 보장한다.
> 3. **융합**: **데이터 모델링 도구(ERWin, PowerDesigner)**의 자동 DDL 생성 기반으로 설계 생산성을 높이며, ORM(JPA, Hibernate)의 엔티티-테이블 자동 매핑의 이론적 토대가 되어 개발자가 SQL을 직접 작성하지 않아도 객체지향 코드로 RDBMS를 운영할 수 있다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
매핑 룰(Mapping Rule)은 **"ER 다이어그램(ERD)으로 표현된 논리적 데이터 모델을 릴레이션 스키마(테이블 구조)으로 변환하는 규칙"**이다.

**매핑의 3단계**:
1. **개체(Entity) → 테이블(Table)** 변환
2. **속성(Attribute) → 컬럼(Column)** 변환
3. **관계(Relationship) → 외래키(Foreign Key)** 변환

### 💡 비유
매핑 룰은 **"설계도를 실제 건축물로 변환하는 시공 지침"**과 같다.
- **개체 → 건물**: 학생 개체 → 학생 테이블
- **속성 → 방**: 이름 속성 → name 컬럼
- **관계 → 통로**: 수업 관계 → course_id 외래키

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         매핑 룰의 필요성                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

1. 추상 모델 → 물리 모델 변환:
   ERD (개념적) → 릴레이션 스키마 (논리적) → DDL (물리적)

2. 자동화 도구 필요:
   • 수동 변환: 오류 발생 욀이 있음
   • 자동화: CASE 도구 (ERWin, PowerDesigner, Data Modeler)
   • ORM: JPA, Sequelize (코드 레벨 매핑)

3. 표준화된 변환 규칙:
   • 모든 설계자가 동일한 규칙 사용 필요
   • 문서화 및 형상 관리 용이
```

## Ⅱ. 아키텍처 및 핵심 원리

### 매핑 룰 7단계 프로세스

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         ERD → 릴레이션 스키마 매핑 7단계                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [ERD: ER 다이어그램]

    Step 1: 강(Strong) 개체 → 테이블 변환
    ┌─────────────┐      ┌─────────────┐      ┌─────────────┐
    │  EMPLOYEE   │      │ DEPARTMENT  │      │   PROJECT   │
    └─────┬───────┘      └─────┬───────┘      └─────┬───────┘
          │                     │                     │
          ▼                     ▼                     ▼
    ┌───────────────────────────────────────────────────────────────────────────────────┐
    │  CREATE TABLE employee ( ... )                                                    │
    │  CREATE TABLE department ( ... )                                                   │
    │  CREATE TABLE project ( ... )                                                     │
    └───────────────────────────────────────────────────────────────────────────────────┘

    Step 2: 약(Weak) 개체 → 식별자에 따라 테이블 또는 컬럼으로 변환
    ┌─────────────┐
    │ DEPENDENT   │ (식별자가 없으면)
    └─────────────┘
          │
          ▼
    ┌───────────────────────────────────────────────────────────────────────────────────────────┐
    │  CREATE TABLE employee_dependent (                                                     │
    │    employee_id PK,                                                             │
    │    dependent_name                                                              │
    │  );                                                                             │
    └───────────────────────────────────────────────────────────────────────────────────┘

    또는

    ┌───────────────────────────────────────────────────────────────────────────────────────────┐
    │  ALTER TABLE employee ADD COLUMN dependent_name;                                    │
    └───────────────────────────────────────────────────────────────────────────────────┘

    Step 3: 1:1 관계 → 기본 키(FK)로 변환
    ┌─────────────┐                   1:1                 ┌─────────────┐
    │  EMPLOYEE   │ ────────────────────────────────→│  PARKING_SPACE │
    └─────────────┘                                      └─────────────┘
    │                                                         │
    ▼                                                         │
    ┌───────────────────────────────────────────────────────────────────────────────────────────┐
    │  CREATE TABLE employee (                                                           │
    │    employee_id PK,                                                             │
    │    ...                                                                            │
    │    parking_space_id FK UNIQUE  -- 1:1 관계                                         │
    │  );                                                                             │
    └───────────────────────────────────────────────────────────────────────────────────┘

    Step 4: 1:N 관계 → 외래키(FK)로 변환
    ┌─────────────┐                   1:N                 ┌─────────────┐
    │ DEPARTMENT  │ ────────────────────────────────→│  EMPLOYEE   │
    │(1)         │                                      │(N)         │
    └─────────────┘                                      └─────────────┘
    │                                                         │
    ▼                                                         │
    ┌───────────────────────────────────────────────────────────────────────────────────────────┐
    │  CREATE TABLE department (                                                           │
    │    dept_id PK,                                                                 │
    │    ...                                                                            │
    │  );                                                                             │
    │                                                                                  │
    │  CREATE TABLE employee (                                                            │
    │    employee_id PK,                                                             │
    │    ...                                                                            │
    │    dept_id FK REFERENCES department(dept_id)                                     │
    │  );                                                                             │
    └───────────────────────────────────────────────────────────────────────────────────┘

    Step 5: M:N 관계 → 연결 테이블(Association Table) 생성
    ┌─────────────┐                   M:N                 ┌─────────────┐
    │   STUDENT   │ ────────────────────────────────→│   COURSE    │
    │  (N)        │                                      │   (M)       │
    └─────────────┘                                      └─────────────┘
    │                                                         │
    ▼                                                         │
    ┌───────────────────────────────────────────────────────────────────────────────────────────┐
    │  CREATE TABLE student (                                                             │
    │    student_id PK,                                                               │
    │    ...                                                                            │
    │  );                                                                             │
    │                                                                                  │
    │  CREATE TABLE course (                                                             │
    │    course_id PK,                                                               │
    │    ...                                                                            │
    │  );                                                                             │
    │                                                                                  │
    │  CREATE TABLE enrollment (  -- 연결 테이블                                            │
    │    student_id FK REFERENCES student(student_id),                                      │
    │    course_id FK REFERENCES course(course_id),                                        │
    │    enrollment_date,                                                             │
    │    PK (student_id, course_id)  -- 복합 기본키                                      │
    │  );                                                                             │
    └───────────────────────────────────────────────────────────────────────────────────┘

    Step 6: 다치 속성(Multi-valued Attribute) → 별도 테이블로 분리
    ┌─────────────┐                ┌─────────────┐
    │   EMPLOYEE   │                │   SKILL     │
    │   (1)        │                │   (N)        │
    └─────────────┘                └─────────────┘
    │    │                            │
    │    └────────────────────────────┘
    ▼
    ┌───────────────────────────────────────────────────────────────────────────────────────────┐
    │  CREATE TABLE employee (                                                            │
    │    employee_id PK,                                                             │
    │    ...                                                                            │
    │  );                                                                             │
    │                                                                                  │
    │  CREATE TABLE skill (                                                               │
    │    skill_id PK,                                                                │
    │    skill_name                                                                    │
    │  );                                                                             │
    │                                                                                  │
    │  CREATE TABLE employee_skill (  -- 연결 테이블                                           │
    │    employee_id FK REFERENCES employee(employee_id),                                     │
    │    skill_id FK REFERENCES skill(skill_id),                                           │
    │    PK (employee_id, skill_id)                                                      │
    │  );                                                                             │
    └───────────────────────────────────────────────────────────────────────────────────┘

    Step 7: 상속(Inheritance) → PK/FK 또는 단일 테이블 병합
    ┌─────────────────┐
    │      PERSON       │ (부모)    ┌─────────────┐
    │      ▲           │           │   EMPLOYEE   │ (자식)
    │      │                         │   STUDENT    │ (자식)
    │      └────────────────────────┘               │
    │                                     │
    │    [Single Table 상속]                         │
    │    • person_type 컬럼으로 구분 (직원, 학생)       │
    │    • 전문화되지 않은 컬럼만 공유                     │
    └──────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 매핑 전략별 DDL 결과 비교

| ERD 구조 | 매핑 전략 | DDL 결과 | 장점 | 단점 |
|---------|----------|---------|------|------|
| **강 개체 1개** | 1개체 → 1테이블 | `CREATE TABLE employee` | 간단 | 모든 하위 클래스 속성 포함 |
| **강 개체 N개** | 각각 → 테이블 | N개 테이블 + FK | 명확성 유지 | JOIN 필요 |
| **약 개체** | 식별자 없음 → 컬럼 | `ALTER TABLE ADD COLUMN` | 간단 | 테이블 컬럼 많아짐 |
| **약 개체** | 식별자 있음 → 테이블 | 별도 테이블 또는 컬럼 | 중복 제거 | 복잡도 증가 |

### 과목 융합 관점 분석

#### 1. 객체지향 ↔ 매핑 룰
- **JPA @Inheritance**: Single Table, Joined, Table per Class
- **상속 전략**: 전문화 vs 공유 컬럼

#### 2. 정규화 ↔ 매핑 룰
- **정규화 정도**: 1NF → 2NF → 3NF
- **이상 현상 제거**: 분해를 통해 M:N → 1:N + 새 테이블

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: E-Commerce ERD 매핑
**상황**: 고객, 주문, 주문 항목, 상품, 결제, 배송
**판단**:
1. **1:N 관계 매핑**: 고객-주문, 주문-주문항목, 주문-결제, 결제-배송
2. **M:N 관계 매핑**: 주문-상품 → 연결 테이블 (order_items)
3. **강 개체**: 상품(Product) → 테이블
4. **약 개체**: 주소(Address) → 고객 테이블에 컬럼 추가

## Ⅴ. 기대효과 및 결론

### 매핑 룰 기반 설계 자동화

| 도구 | 자동화 기능 | DDL 생성 | 프로그래밍 언어 |
|------|-----------|---------|-------------|
| **ERWin** | ERD → DDL | Oracle, SQL Server, MySQL | Visual Basic |
| **PowerDesigner** | ERD → DDL | 다양 DBMS | VBA, PowerScript |
| **Data Modeler** | ERD → DDL | Oracle, MySQL, PostgreSQL | Java |
| **ORM** | Class → Table | Hibernate, JPA | Java, C#, Python |

### 미래 전망

1. **AI 기반 매핑**: LLM이 ERD를 SQL DDL로 변환
2. **Reverse Engineering**: 기존 DB에서 ERD 자동 생성
3. **Model-First**: 코드 우선 설계, DB 스키마 자동 생성

### ※ 참고 표준/가이드
- **ANSI/SPARC**: 3단계 스키마 아키텍처
- **Crow's Foot**: ERD 표기법
