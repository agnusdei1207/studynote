# 데이터 독립성 (Data Independence)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 독립성은 응용 프로그램(사용자 관점)과 데이터의 물리적/논리적 저장 구조(저장 장치 관점)를 분리하여, 저장 구조의 변경이 응용 프로그램에 영향을 미치지 않도록 하는 추상화 계층이다.
> 2. **가치**: 3단계 스키마 아키텍처(ANSI/SPARC)를 통해 **논리적 독립성**(개념 스키마 변경→외부 스키마 무영향)과 **물리적 독립성**(내부 스키마 변경→개념 스키마 무영향)을 보장한다.
> 3. **융합**: 시스템 진화에 따른 DB 리팩토링, 클라우드 마이그레이션, NoSQL 다중 모델 지원의 이론적 토대이며, 현대의 마이크로서비스 API 설계와 데이터 추상화 계층(Mapper, Repository)의 근간이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
데이터 독립성(Data Independence)은 **"데이터베이스의 저장 구조나 접근 방법이 변경되더라도, 응용 프로그램을 수정할 필요가 없는 능력"**을 의미한다.

**핵심 원리**: 관심사 분리(Separation of Concerns)
- **What (무엇을)**: 사용자는 데이터의 논리적 구조만 관심
- **How (어떻게)**: DBMS가 물리적 저장 구조를 담당

### 💡 비유
데이터 독립성은 **"TV 리모컨"**과 같다.
- **논리적 독립성**: 채널 번호(1번=KBS, 2번=MBC)는 방송사가 바뀌어도 변하지 않음
- **물리적 독립성**: TV 화면의 기술(CRT → LCD → OLED)이 바뀌어도 채널 변경은 리모컨으로 동일
- **사용자**: 방송국의 송신 장비가 바뀌어도 알 필요 없음

### 등장 배경 및 발전 과정

#### 1. 파일 시스템의 데이터 종속성 문제

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         파일 시스템의 데이터 종속성 (문제점)                           │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [프로그램 A]           [파일 구조]
        │                    │
    고객.dat           ┌─────────────────────┐
    ┌──────────────────┤  순차 파일 구조     │
    │ struct Customer { │  [레코드1][레코드2] │
    │   int id;         │   [레코드3][레코드4] │
    │   char name[30];  │  ...                 │
    │   long phone;     │  [EOF]               │
    │ } customer;       └─────────────────────┘
    │                     ▲
    └─────────────────────┘
              │ (직접 의존)

    문제: 파일 구조를 인덱스 파일로 변경하면?
         → 프로그램 A의 코드를 모두 수정해야 함!
         → fread() → fseek() + read()로 변경
         → 데이터 종속성(Data Dependency)
```

#### 2. 3단계 스키마 아키텍처 도입 (ANSI/SPARC 1975)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         3단계 스키마로의 데이터 독립성                                  │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [응용 프로그램]           [외부 스키마]           [개념 스키마]           [내부 스키마]
        │                       │                       │                       │
    "급여 조회"            ┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
    SELECT salary        │ 급여 뷰         │    │ EMPLOYEE 테이블  │    │ B-Tree 인덱스  │
    FROM emp_salary      │ (emp_id, salary)│    │ (id, name, dept │    │ 페이지 크기 8KB │
    WHERE ...            └────────▲────────┘    │ salary, ...)     │    │ 저장 경로      │
                               │ 사상 매핑    └────────▲────────┘    └────────▲────────┘
                                       │ 사상 매핑               │ 사상 매핑

    ✅ 물리적 구조 변경 (내부 스키마):
       인덱스 추가, 파티셔닝, 스토리지 교체
           ↓
       사상 매핑만 변경, 응용 프로그램은 그대로!

    ✅ 논리적 구조 변경 (개념 스키마):
       컬럼 추가, 테이블 분할/합병
           ↓
       외부 스키마(뷰)만 재정의, 응용 프로그램은 그대로!
```

## Ⅱ. 아키텍처 및 핵심 원리

### 3단계 스키마 아키텍처

| 계층 | 명칭 | 관점 | 대상 | 예시 |
|------|------|------|------|------|
| **외부 스키마** | External Schema | 사용자 관점 | 뷰(View) | "급여팀이 보는 직원 정보" |
| **개념 스키마** | Conceptual Schema | 조직 전체 관점 | 논리적 구조 | 직원 테이블, 부서 테이블 |
| **내부 스키마** | Internal Schema | 물리적 저장 관점 | 저장 구조 | 인덱스, 파티션, 페이지 |

### 정교한 구조 다이어그램

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         데이터 독립성 계층 구조                                        │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Level 1: [외부 스키마 (External Schema)]
    ┌───────────────────────────────────────────────────────────────────────────────────┐
    │  사용자/응용별 다른 뷰(View) 정의                                                   │
    │                                                                                  │
    │  ┌─────────────────────┐  ┌─────────────────────┐  ┌─────────────────────┐       │
    │  │ 급여 담당자 뷰      │  │ 인사 담당자 뷰      │  │ 경영진 뷰           │       │
    │  │ V_EMP_SALARY        │  │ V_EMP_INFO          │  │ V_DEPT_SUMMARY      │       │
    │  │ (emp_id, salary,    │  │ (emp_id, name,      │  │ (dept_id, headcount│       │
    │  │  dept_name)         │  │  dept, position)    │  │  total_salary)      │       │
    │  └─────────────────────┘  └─────────────────────┘  └─────────────────────┘       │
    │            │                     │                       │                       │
    └────────────┼─────────────────────┼───────────────────────┼───────────────────────┘
                 │                     │                       │
                 │   외부/개념 사상(External/Conceptual Mapping)                      │
                 ▼                     ▼                       ▼

    Level 2: [개념 스키마 (Conceptual Schema)]
    ┌───────────────────────────────────────────────────────────────────────────────────┐
    │  조직 전체의 논리적 데이터 구조 (엔티티, 속성, 관계)                                  │
    │                                                                                  │
    │  ┌─────────────────────────────────────────────────────────────────────────────┐  │
    │  │  EMPLOYEE (직원)                                                          │  │
    │  │  PK: emp_id                                                               │  │
    │  │  Columns: emp_id, name, dept_id, position, salary, ...                    │  │
    │  │                              │                                             │  │
    │  │                              │ FK                                          │  │
    │  │                              ▼                                             │  │
    │  │  DEPARTMENT (부서)                                                        │  │
    │  │  PK: dept_id                                                              │  │
    │  │  Columns: dept_id, dept_name, location, budget, ...                      │  │
    │  └─────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                  │
    │  무결성 제약조건: PK, FK, Unique, Check, Not Null                                │
    │            │                                                                     │
    └────────────┼─────────────────────────────────────────────────────────────────────┘
                 │   개념/내부 사상(Conceptual/Internal Mapping)                        │
                 ▼

    Level 3: [내부 스키마 (Internal Schema)]
    ┌───────────────────────────────────────────────────────────────────────────────────┐
    │  물리적 저장 구조 (저장 장치, 인덱스, 파일 조직)                                   │
    │                                                                                  │
    │  ┌─────────────────────────────────────────────────────────────────────────────┐  │
    │  │  [저장소 구조]                                                             │  │
    │  │  • 테이블스페이스: TS_USERS, TS_INDEX                                       │  │
    │  │  • 파일 구조: 데이터 파일( .dbf), 인덱스 파일( .idx)                         │  │
    │  │  • 페이지 크기: 8KB (Oracle), 16KB (MySQL InnoDB)                          │  │
    │  │                                                                              │  │
    │  │  [인덱스]                                                                  │  │
    │  │  • PK_EMP_ID: B+Tree 인덱스 (Non-clustered)                                 │  │
    │  │  • IDX_DEPT_ID: B+Tree 인덱스                                               │  │
    │  │                                                                              │  │
    │  │  [파티셔닝]                                                                │  │
    │  │  • RANGE PARTITION: 입사일 기준                                            │  │
    │  └─────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                  │
    └───────────────────────────────────────────────────────────────────────────────────┘
```

### 심층 동작 원리

#### 1. 논리적 데이터 독립성 (Logical Data Independence)

**정의**: 개념 스키마(논리적 구조)가 변경되더라도 외부 스키마(뷰)를 변경할 필요가 없는 능력

**적용 가능한 변경**:
- 테이블에 새로운 컬럼 추가
- 테이블 분할 (Table Splitting)
- 테이블 합변 (Table Merging)
- 관계(Constraint) 추가/제거

**예시**:
```sql
-- [변경 전] 개념 스키마
CREATE TABLE employees (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    salary DECIMAL(10,2)
);

-- [외부 스키마] 급여팀 뷰
CREATE VIEW v_emp_salary AS
SELECT id, name, salary
FROM employees;

-- [변경 후] 개념 스키마: 부분 추가
ALTER TABLE employees ADD COLUMN dept_id INT;
ALTER TABLE employees ADD COLUMN position VARCHAR(50);

-- [외부 스키마] 변경 없이 그대로 사용 가능!
-- 급여팀 뷰는 dept_id, position을 보지 않아도 됨
```

**사상 매핑(Mapping)의 역할**:
```text
외부 스키마              개념 스키마
V_EMP_SALARY    ←───→   EMPLOYEE TABLE
(emp_id, salary)       (id, name, dept_id, salary, position)

사상 정의:
  V_EMP_SALARY.emp_id  = EMPLOYEE.id
  V_EMP_SALARY.salary  = EMPLOYEE.salary

개념 스키마에 컬럼이 추가되어도 사상 정의만 유지하면 뷰는 영향 없음
```

#### 2. 물리적 데이터 독립성 (Physical Data Independence)

**정의**: 내부 스키마(물리적 구조)가 변경되더라도 개념 스키마(논리적 구조)를 변경할 필요가 없는 능력

**적용 가능한 변경**:
- 새로운 인덱스 생성/삭제
- 저장 장치 교체 (HDD → SSD)
- 파티셔닝 적용
- 페이지 크기 변경
- 파일 구조 변경 (Heap → Clustered)

**예시**:
```sql
-- [변경 전] 내부 스키마: Heap 테이블
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    amount DECIMAL(10,2)
);

-- [변경 후] 내부 스키마: 인덱스 추가
CREATE INDEX idx_customer ON orders(customer_id);
CREATE INDEX idx_date ON orders(order_date);

-- [결과] 개념 스키마는 그대로!
-- SELECT * FROM orders WHERE customer_id = 100;
-- 응용 프로그램은 인덱스가 생성된 것을 모르고 동일하게 사용
```

**파티셔닝 예시**:
```sql
-- [변경 전] 단일 테이블
CREATE TABLE logs (
    log_id BIGINT PRIMARY KEY,
    log_date DATE,
    message TEXT
);

-- [변경 후] 파티셔닝 (내부 스키마만 변경)
ALTER TABLE logs PARTITION BY RANGE (log_date) (
    PARTITION p2023 VALUES LESS THAN ('2024-01-01'),
    PARTITION p2024 VALUES LESS THAN ('2025-01-01'),
    PARTITION p2025 VALUES LESS THAN ('2026-01-01')
);

-- [결과] 개념 스키마는 그대로!
-- SELECT * FROM logs WHERE log_date = '2024-06-15';
-- 응용 프로그램은 파티셔닝을 인식하지 않음
```

## Ⅲ. 융합 비교 및 다각도 분석

### 데이터 독립성의 계층별 비교

| 비교 항목 | 논리적 독립성 | 물리적 독립성 |
|----------|-------------|-------------|
| **대상 변경** | 개념 스키마 (논리적 구조) | 내부 스키마 (물리적 구조) |
| **보호 대상** | 외부 스키마 (뷰) | 개념 스키마 (테이블) |
| **영향 범위** | 일부 영향 (뷰 재정의 필요 시) | 영향 없음 |
| **구현 메커니즘** | 뷰(View), 사상 매핑 | 옵티마이저, 사상 매핑 |
| **변경 예시** | 컬럼 추가, 테이블 분할 | 인덱스 추가, 파티셔닝 |
| **난이도** | 높음 (제약 있음) | 낮음 (자동 처리) |

### 과목 융합 관점 분석

#### 1. 소프트웨어 공학 ↔ 데이터 독립성
- **Repository 패턴**: 데이터 접근 추상화 계층
- **DTO/VO 패턴**: 엔티티와 뷰 분리
- **Mapper 계층**: 논리적/물리적 매핑 (MyBatis, Hibernate)

#### 2. 클라우드 ↔ 데이터 독립성
- **DBaaS (Database as a Service)**: 스토리지 교체 투명성
- **Auto-scaling**: 자원 증감에 따른 응용 무영향

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: HDD → SSD로의 스토리지 마이그레이션
**상황**: 기존 HDD에서 SSD로 데이터베이스 이관
**판단**:
1. **물리적 독립성 활용**: 내부 스키마만 변경 (페이지 크기, 블록 크기 조정)
2. **응용 프로그램**: 수정 불필요
3. **옵티마이저 재통계**: 새로운 통계 정보 수집
4. **성능 테스트**: I/O 성능 향상 확인

#### 시나리오: 대규모 테이블 리팩토링 (수직 분할)
**상황**: 1억 건 이상의 CUSTOMER 테이블을 customer_base(공통), customer_detail(상세)로 분할
**판단**:
1. **논리적 독립성 활용**: 뷰(V_CUSTOMER_FULL)로 기존 인터페이스 유지
2. **응용 프로그램**: 최소 변경 (뷰를 통한 접근 유지)
3. **단계적 마이그레이션**: 뷰를 통해 점진적 전환

### 도입 시 고려사항 (체크리스트)

**기술적**:
- [ ] 뷰(View) 성능 저하 방지: 단순 뷰만 사용, 복잡한 뷰는 제한
- [ ] 사상 매핑 오버헤드 최소화
- [ ] 데이터 무결성 유지 (분할 후 참조 무결성)

**운영적**:
- [ ] 스키마 변경 시 다운타임 최소화 전략
- [ ] 뷰 의존성 추적 (영향도 분석)

## Ⅴ. 기대효과 및 결론

### 데이터 독립성의 기대 효과

| 효과 | 설명 | 기대 효과 |
|------|------|----------|
| **유지보수성 향상** | DB 구조 변경 시 응용 수정 최소화 | 유지보수 비용 50% 감소 |
| **확장성 확보** | 스토리지 교체, 파티셔닝 투명 적용 | 성능 확장 용이 |
| **이기종 통합** | 표준 인터페이스로 상이한 DBMS 연동 | 통합 시스템 구축 |
| **보안 강화** | 뷰를 통한 접근 제어 | 세밀한 권한 관리 |

### 미래 전망

1. **NoSQL에서의 데이터 독립성**: 다중 모델 API
2. **GraphQL**: 클라이언트 주도 쿼리 (필요한 필드만 요청)
3. **Polyglot Persistence**: 목적에 맞는 DB 선택, API 추상화

### ※ 참고 표준/가이드
- **ANSI/SPARC Architecture**: 3단계 스키마 표준 (1975)
- **ISO/IEC 9075**: SQL 표준
