# 키 (Key)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 키(Key)는 **"릴레이션에서 튜플을 유일하게 식별하는 속성(들)의 집합"**으로, **슈퍼키(Super Key)**, **후보키(Candidate Key)**, **기본키(Primary Key)**, **대체키(Alternate Key)**, **외래키(Foreign Key)**로 계층적으로 분류된다.
> 2. **가치**: **기본키(PK)**는 튜플의 **유일성(Uniqueness)**과 **최소성(Minimality)**을 보장하며, **외래키(FK)**는 릴레이션 간 **참조 무결성(Referential Integrity)**을 강제하여 데이터 일관성을 유지한다.
> 3. **융합**: **정규화(Normalization)**의 핵심 도구로 **함수적 종속(Functional Dependency)**를 결정하며, **조인(Join) 연산**의 기반이 되어 **인덱스(Index)** 설계와 **분산 데이터베이스(Distributed Database)**의 **샤딩(Sharding) 키** 선정의 기준이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
키(Key)는 **"릴레이션 내에서 튜플을 유일하게 식별할 수 있는 속성 또는 속성의 집합"**이다.

**키의 정의 (E.F. Codd, 1970)**:
- **유일성(Uniqueness)**: 키값으로 튜플 유일 식별
- **최소성(Minimality)**: 불필요한 속성 제거

### 💡 비유
키는 **"주민번호"**와 같다.
- **주민번호**: 사람을 유일하게 식별
- **유일성**: 동일한 주민번호 없음
- **최소성**: 주민번호 일부로는 식별 불가

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         키 개념의 진화                                                  │
└─────────────────────────────────────────────────────────────────────────────────────────┘

파일 시스템:
    • 레코드 순서번호로 식별
    • 중복 레코드 발생
    • 논리적 관계 없음
         ↓
관계형 모델 (1970년):
    • 기본키 개념 도입
    • 유일성, 최소성 정의
    • 참조 무결성 개념
         ↓
정규화 이론 (1970년대):
    • 함수적 종속 발견
    • 후보키와 정규형 연관
    • BCNF, 4NF, 5NF
         ↓
현대 데이터베이스:
    • 복합키, 서로게이트키
    • 분산 DB의 샤딩 키
    • 암호화 키 (보안)
```

## Ⅱ. 아키텍처 및 핵심 원리

### 키의 종류와 계층 구조

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         키의 계층 구조                                                 │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [슈퍼키 (Super Key)]
    ┌─────────────────────────────────────────────────────────────────────────────────────────┐
    │  정의: 튜플을 유일하게 식별하는 속성의 집합 (유일성 만족)                            │
    │                                                                                         │
    │  EMPLOYEE 테이블 예시:                                                                 │
    │  • {employee_id}                      ✓ 슈퍼키                                      │
    │  • {employee_id, name}                ✓ 슈퍼키 (name는 불필요하지만 슈퍼키임)         │
    │  • {employee_id, name, dept_id}       ✓ 슈퍼키 (불필요한 속성 포함)                   │
    │  • {email}                            ✓ 슈퍼키 (유일하면)                             │
    │  • {name}                             ✗ 슈퍼키 아님 (중복 가능)                      │
    └─────────────────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
    [후보키 (Candidate Key)]
    ┌─────────────────────────────────────────────────────────────────────────────────────────┐
    │  정의: 최소성을 만족하는 슈퍼키 (유일성 + 최소성)                                     │
    │                                                                                         │
    │  EMPLOYEE 테이블 예시:                                                                 │
    │  • {employee_id}               ✓ 후보키 (최소 슈퍼키)                                 │
    │  • {email}                     ✓ 후보키 (유일하면)                                   │
    │  • {employee_id, name}         ✗ 후보키 아님 (name 불필요)                           │
    │                                                                                         │
    │  [후보키 선정 기준]                                                                      │
    │  1. 유일성: 중복 값 없음                                                                │
    │  2. 최소성: 속성 제거 시 유일성 상실                                                    │
    │  3. NOT NULL: NULL 값 불가                                                             │
    │  4. 불변성: 변경되지 않는 속성                                                          │
    └─────────────────────────────────────────────────────────────────────────────────────────┘
                              │
                  ┌───────────┴───────────┐
                  ▼                       ▼
    [기본키 (Primary Key)]     [대체키 (Alternate Key)]
    ┌─────────────────────────┐ ┌─────────────────────────────────────────────────────────────┐
    │ 후보키 중 하나를 선택    │ │ 후보키 중 기본키가 되지 못한 나머지                        │
    │                         │ │                                                             │
    │ EMPLOYEE 기본키:        │ │ EMPLOYEE 대체키:                                           │
    │ • employee_id (PK)      │ │ • email (AK)                                               │
    │                         │ │                                                             │
    │ [기본키 선정 기준]       │ │ [대체키 용도]                                              │
    │ 1. 짧을수록 좋음        │ │ • 유일 인덱스로 활용                                        │
    │ 2. 변경되지 않을 것     │ │ • 대체 검색 경로                                           │
    │ 3. 의미가 있는 것이 좋음 │ │ • 백업 키로 사용                                          │
    │ 4. 단일 속성 선호       │ │                                                             │
    └─────────────────────────┘ └─────────────────────────────────────────────────────────────┘

    [외래키 (Foreign Key)]
    ┌─────────────────────────────────────────────────────────────────────────────────────────┐
    │  정의: 다른 릴레이션의 기본키를 참조하는 속성 (참조 무결성 보장)                       │
    │                                                                                         │
    │  EMPLOYEE.employee_id ──┐                                                             │
    │                         │                                                             │
    │                         ▼                                                             │
    │  PROJECT.manager_id (FK) → EMPLOYEE.employee_id (PK)                                   │
    │                                                                                         │
    │  [외래키 제약조건]                                                                      │
    │  • REFERENCES: 참조할 테이블과 컬럼 지정                                               │
    │  • ON DELETE: 참조된 행 삭제 시 동작                                                    │
    │    - CASCADE: 연쇄 삭제                                                                │
    │    - SET NULL: NULL로 설정                                                             │
    │    - SET DEFAULT: 기본값으로 설정                                                       │
    │    - RESTRICT/NO ACTION: 삭제 금지                                                      │
    │  • ON UPDATE: 참조된 행 갱신 시 동작                                                    │
    └─────────────────────────────────────────────────────────────────────────────────────────┘
```

### 키별 상세 비교

| 키 종류 | 정의 | 유일성 | 최소성 | 개수 | 예시 |
|---------|------|--------|--------|------|------|
| **슈퍼키(Super Key)** | 튜플 유일 식별 속성 집합 | ✓ | ✗ | 다수 | {id}, {id, name}, {id, name, dept} |
| **후보키(Candidate Key)** | 최소성 만족 슈퍼키 | ✓ | ✓ | 다수 | {id}, {email} |
| **기본키(Primary Key)** | 선택된 후보키 | ✓ | ✓ | 1개 | employee_id |
| **대체키(Alternate Key)** | 나머지 후보키 | ✓ | ✓ | N-1개 | email |
| **외래키(Foreign Key)** | 다른 테이블 PK 참조 | ✗ | - | 다수 | dept_id |

### 정교한 키 구조 다이어그램

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         릴레이션 간 키 관계                                               │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    DEPARTMENT 테이블                        EMPLOYEE 테이블
    ┌───────────────────────────────────────┐   ┌───────────────────────────────────────────┐
    │ dept_id (PK) │ dept_name  │ location │   │ emp_id (PK) │ name  │ dept_id (FK) │ email│
    ├───────────────────────────────────────┤   ├───────────────────────────────────────────┤
    │ 10          │ 인사팀     │ 서울     │   │ 101        │ 홍길동 │ 10           │ ...  │
    │ 20          │ 개발팀     │ 판교     │   │ 102        │ 김철수 │ 10           │ ...  │
    │ 30          │ 영업팀     │ 부산     │   │ 103        │ 이영희 │ 20           │ ...  │
    └───────────────────────────────────────┘   │ 104        │ 박민수 │ 20           │ ...  │
                     │                          │ 105        │ 최수진 │ 30           │ ...  │
                     │                          └───────────────────────────────────────────┘
                     │                                   │
                     │         ┌─────────────────────────┘
                     │         │
                     ▼         ▼
    ┌─────────────────────────────────────────────────────────────────────────────────────────┐
    │  참조 무결성 제약조건 (Referential Integrity Constraint)                                │
    │                                                                                         │
    │  1. EMPLOYEE.dept_id의 모든 값은 DEPARTMENT.dept_id에 존재해야 함                        │
    │  2. DEPARTMENT에서 dept_id=10 삭제 시:                                                  │
    │     • ON DELETE RESTRICT: 삭제 불가 (dept_id=10인 사원 존재)                            │
    │     • ON DELETE CASCADE: dept_id=10인 모든 사원도 삭제                                  │
    │     • ON DELETE SET NULL: 해당 사원들의 dept_id를 NULL로 설정                           │
    │                                                                                         │
    │  3. EMPLOYEE에서 존재하지 않는 dept_id=99 삽입 시:                                      │
    │     • FK 제약 위반 → 삽입 거부                                                          │
    └─────────────────────────────────────────────────────────────────────────────────────────┘

    [복합키 (Composite Key)]

    ENROLLMENT 테이블 (학생 수강 신청)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  student_id (FK) │ course_id (FK) │ enrollment_date │ grade                            │
    │  ────────────────┼───────────────┼─────────────────┼────────                          │
    │  (PK의 일부)     │ (PK의 일부)   │                 │                                  │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  101             │ CS101         │ 2026-03-01      │ A                                │
    │  101             │ MATH201       │ 2026-03-01      │ B+                               │
    │  102             │ CS101         │ 2026-03-02      │ A-                               │
    │  101             │ CS101         │ 2026-09-01      │ 재수강                           │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    PK: (student_id, course_id, enrollment_date)  ← 복합키
    • 한 학생이 같은 과목을 여러 번 수강 가능 (학기마다 다른 enrollment_date)
    • (student_id, course_id)만으로는 유일하지 않음
```

### 심층 동작 원리

#### 1. 서로게이트키(Surrogate Key)

**정의**: 자연스러운 비즈니스 속성이 아닌 시스템 생성 키

```sql
-- 자연키 (Natural Key)
CREATE TABLE employees (
    employee_id VARCHAR(20) PRIMARY KEY,  -- 사번 (업무 의미)
    ...
);

-- 서로게이트키 (Surrogate Key)
CREATE TABLE employees (
    id SERIAL PRIMARY KEY,              -- 자동 생성 (의미 없음)
    employee_id VARCHAR(20) UNIQUE,     -- 업무 식별자
    ...
);
```

**장단점 비교**:

| 구분 | 자연키 (Natural Key) | 서로게이트키 (Surrogate) |
|------|---------------------|-------------------------|
| **예시** | 주민번호, 사번 | AUTO_INCREMENT, UUID |
| **장점** | 업무 의미 명확 | 변경 없음, 조인 빠름 |
| **단점** | 변경 가능, 노출 위험 | 추가 JOIN 필요 |
| **추천** | 안정적이면 사용 | 대부분 사용 |

#### 2. 키와 인덱스 관계

```sql
-- 기본키 자동 인덱스 생성
CREATE TABLE employees (
    employee_id INT PRIMARY KEY,  -- Clustered Index 생성
    ...
);

-- 외래키 인덱스 권장 (성능)
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);

-- FK 컬럼에 인덱스 생성 (조인 성능)
CREATE INDEX idx_orders_customer_id ON orders(customer_id);
```

#### 3. 분산 DB에서의 키 설계

**샤딩 키(Sharding Key) 선정**:
```sql
-- 안 좋은 예: 낮은 카디널리티
SHARD KEY = gender  -- 'M', 'F' (2개 값) → 분산 효과 없음

-- 좋은 예: 높은 카디널리티, 균등 분포
SHARD KEY = customer_id  -- 각 고객이 다른 샤드로
SHARD KEY = HASH(user_id)  -- 해시 기반 분산
```

## Ⅲ. 융합 비교 및 다각도 분석

### 키 설계 전략 비교

| 전략 | 장점 | 단점 | 적용 |
|------|------|------|------|
| **단일 PK** | 간단, 조인 빠름 | 유일성 보장 어려움 | ID 기반 |
| **복합 PK** | 유일성 보장 | 조인 복잡, 긴 키 | 연결 테이블 |
| **Surrogate** | 변경 없음 | 추가 컬럼 | 대부분의 테이블 |
| **Natural** | 의미 명확 | 변경 가능 | 코드성 데이터 |

### 과목 융합 관점 분석

#### 1. 정규화 ↔ 키
- **2NF**: 후보키가 아닌 속성의 부분적 함수 종속 제거
- **3NF**: 이행적 함수 종속 제거
- **BCNF**: 모든 결정자가 후보키

#### 2. 분산 시스템 ↔ 키
- **샤딩 키**: 분산 기준
- **컴파운드 키**: 다차원 분산
- **UUID**: 글로벌 유일성

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 전자상거래 주문 테이블 키 설계
**상황**: 주문 번호, 고객, 상품, 배송지 관리
**판단**:
1. **기본키**: 주문 ID (서로게이트키)
   - `order_id BIGINT AUTO_INCREMENT`
2. **외래키**: 고객, 배송지 참조
   - `customer_id FK`, `shipping_address_id FK`
3. **대체키**: 주문 번호 (비즈니스 키)
   - `order_number VARCHAR(20) UNIQUE`
4. **복합키**: 주문 항목 테이블
   - `PK (order_id, item_seq)`

## Ⅴ. 기대효과 및 결론

### 키 설계 기대 효과

| 설계 측면 | 잘못된 키 설계 | 올바른 키 설계 |
|----------|--------------|--------------|
| **성능** | 긴 복합키 | 짧은 서로게이트키 |
| **유연성** | 자연키 변경 | 서로게이트키 사용 |
| **무결성** | FK 누락 | FK로 참조 무결성 |
| **분산** | 낮은 카디널리티 키 | 높은 카디널리티 키 |

### 미래 전망

1. **UUID v7**: 시간순 정렬 가능한 UUID
2. **ULID**: UUID 대안, URL 안전
3. **Snowflake ID**: 분산 환경 유일 ID

### ※ 참고 표준/가이드
- **Codd's Rule 1**: Information Rule (모든 데이터는 릴레이션)
- **ISO/IEC 9075**: SQL 키 제약조건
