# 도메인 (Domain)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도메인은 **"속성이 가질 수 있는 모든 원자값(Atomic Value)의 집합"**으로 정의되며, 데이터베이스에서 **데이터 타입(DataType)**, **길이(Length)**, **제약조건(Constraint)**을 포함하는 속성의 **값의 허용 범위(Value Set)**를 명시한다.
> 2. **가치**: **도메인 무결성(Domain Integrity)**의 핵심으로, 허용되지 않는 값의 삽입을 방지하고 **데이터 일관성(Data Consistency)**을 보장하며, **CREATE DOMAIN** 문으로 재사용 가능한 도메인을 정의하여 **데이터 표준화(Data Standardization)**를 실현한다.
> 3. **융합**: **정규화(Normalization)**의 기본 전제로 **원자값(Atomic Value)** 요구를 통해 제1정규형(1NF)을 보장하며, **형 변환(Type Casting)**, **데이터 검증(Data Validation)**, **ORM 매핑**의 기초가 되어 애플리케이션-데이터베이스 간 타입 안전성을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
도메인(Domain)은 **"속성(Attribute)이 가질 수 있는 원자값의 집합"**으로, 데이터의 값 범위와 타입을 정의한다.

**도메인의 정의 (Codd, 1970)**:
- 관계형 모델에서 모든 속성은 도메인에 속해야 함
- 동일 도메인에 속한 속성 간 비교/연산 가능
- 서로 다른 도메인의 속성은 비교 불가 (타입 불일치)

### 💡 비유
도메인은 **"변수에 할당할 수 있는 값의 범위"**와 같다.
- **INT 도메인**: {0, 1, 2, ..., 2147483647}
- **VARCHAR(10) 도메인**: 길이 10 이하의 모든 문자열
- **ENUM 도메인**: {'M', 'F'} (성별)
- **DATE 도메인**: 1000-01-01 ~ 9999-12-31

## Ⅱ. 아키텍처 및 핵심 원리

### 도메인 vs 데이터 타입

| 구분 | 데이터 타입 (DataType) | 도메인 (Domain) |
|------|----------------------|-----------------|
| **정의** | 물리적 저장 형식 | 논리적 값의 집합 |
| **예시** | INT, VARCHAR, DATE | 연령, 주소, 성별 |
| **범위** | 시스템 제공 | 사용자 정의 |
| **제약** | 길이, 타입 | 의미적 제약 포함 |
| **재사용** | 항상 가능 | CREATE DOMAIN으로 정의 |

### 심층 동작 원리

#### 1. 도메인과 원자성 (Atomic Domain)

**원자값(Atomic Value)**:
- 더 이상 분해 불가능한 값
- 제1정규형(1NF)의 핵심 요구사항

**원자적 도메인 예시**:
- **원자적**: {1, 2, 3}, {'Seoul', 'Busan'}, DATE
- **비원자적**: {{1, 2}, {3, 4}}, (Addr: {시, 구, 동})

#### 2. 도메인 무결성 제약

```sql
-- 기본 데이터 타입 제약
CREATE TABLE employees (
    employee_id INT NOT NULL,
    name VARCHAR(100) NOT NULL,
    salary DECIMAL(10,2) CHECK (salary >= 0),
    email VARCHAR(255) UNIQUE
);

-- 사용자 정의 도메인 (PostgreSQL)
CREATE DOMAIN us_postal_code AS TEXT
CHECK (
    VALUE ~ '^\d{5}$'
    OR VALUE ~ '^\d{5}-\d{4}$'
);

CREATE TABLE addresses (
    address_id SERIAL PRIMARY KEY,
    postal_code us_postal_code  -- 도메인 재사용
);

-- ENUM 도메인 (MySQL)
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    status ENUM('pending', 'processing', 'shipped', 'delivered')
);
```

#### 3. 도메인 간 비교 가능성

```sql
-- 동일 도메인: 비교 가능
SELECT *
FROM employees
WHERE salary > bonus;  -- 둘 다 DECIMAL 도메인

-- 다른 도메인: 비교 불가 (형변환 필요)
-- 잘못된 예: WHERE employee_id = department_id
-- 올바른 예: 형변환 후 비교 또는 의미적 연관성 확인

-- 도메인 호환성
SELECT *
FROM orders
WHERE order_date = ship_date;  -- 동일 DATE 도메인: ✓
```

## Ⅲ. 융합 비교 및 다각도 분석

### DBMS별 도메인 구현 비교

| DBMS | 사용자 정의 도메인 | ENUM 타입 | CHECK 제약 |
|------|------------------|-----------|------------|
| **PostgreSQL** | CREATE DOMAIN | CREATE TYPE | ✓ |
| **MySQL** | ✗ | ENUM | ✓ |
| **Oracle** | CREATE DOMAIN (제한적) | ✗ | ✓ |
| **SQL Server** | CREATE DOMAIN | ✗ | ✓ |

### 과목 융합 관점 분석

#### 1. 프로그래밍 언어 ↔ 도메인
- **정적 타이핑**: 컴파일 시 타입 검사
- **강 타이핑**: 암시적 변환 제한
- **타입 추론**: 도메인 기반 자동 추론

#### 2. 데이터 웨어하우스 ↔ 도메인
- **차원(Dimension) 속성**: 시간, 지리 도메인
- **팩트(Fact) 측정값**: 수치 도메인
- **느린 변화 차원(SCD)**: 도메인 값의 시계열 변화

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 글로벌 서비스 도메인 설계
**상황**: 다국가 주소 시스템 지원
**판단**:
1. **도메인 분리**: 국가별 주소 형식 차이 고려
   - 미국: {번호, 거리, 도시, 주, 우편번호}
   - 한국: {시, 구, 동, 번지}
   - 일본: {도도부현, 시, 구, 번지}
2. **공통 도메인**: 국가코드, 우편번호 형식 통일
3. **JSON 도메인**: 유연한 구조 지원

```sql
-- 유연한 주소 도메인 (JSONB)
CREATE TABLE addresses (
    address_id SERIAL PRIMARY KEY,
    country_code CHAR(2) NOT NULL,
    address_data JSONB NOT NULL,
    -- 예: {"street": "123 Main", "city": "NYC", "zip": "10001"}
    -- 예: {"시": "서울", "구": "강남구", "동": "역삼동"}
);
```

## Ⅴ. 기대효과 및 결론

### 도메인 설계 기대 효과

| 설계 측면 | 도메인 미사용 시 | 도메인 사용 시 |
|----------|----------------|---------------|
| **무결성** | 애플리케이션 검증 | DBMS가 보장 |
| **일관성** | 중복 정의 | 중앙 집중 관리 |
| **변경** | 전체 코드 수정 | 도메인만 수정 |
| **재사용** | 어려움 | CREATE DOMAIN |

### 미래 전망

1. **JSON Schema**: NoSQL의 유연성 + RDBMS의 무결성
2. **Smart Contracts**: 블록체인 도메인 검증
3. **Type Systems**: GraphQL, TypeScript와 통합

### ※ 참고 표준/가이드
- **SQL-92**: CREATE DOMAIN 표준
- **ISO/IEC 9075**: SQL 표준
- **Codd's 12 Rules**: Rule 1: Information Rule
