# 트랜잭션 (Transaction)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트랜잭션은 **"데이터베이스에서 **하나의 논리적 작업 단위**로 처리되는 **일련의 연산"**으로, **ACID 특성**(Atomicity, Consistency, Isolation, Durability)을 보장하며 **Commit(완료)** 또는 **Rollback(취소)**로 **원자적 실행**을 보장한다.
> 2. **가치**: **데이터 무결성(Data Integrity)**을 보장하고 **동시성 제어(Concurrency Control)**로 **Lost Update, Dirty Read, Non-repeatable Read, Phantom Read** 등의 **이상 현상(Anomaly)**을 방지하며 **장애 복구(Recovery)**를 지원한다.
> 3. **융합**: **분산 트랜잭션(Distributed Transaction)**은 **2PC(2-Phase Commit)** 프로토콜로 구현되며, **Spring @Transactional**, **JTA**, **XA Protocol**와 연계되고 **Microservice**의 **Saga Pattern**으로 대체되기도 한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
트랜잭션은 **"데이터베이스에서 All-or-Nothing 원칙으로 실행되는 작업 단위"**이다.

**트랜잭션의 예시**:
- **이체**: A 계좌 출금 → B 계좌 입금
- **주문**: 재고 감소 → 주문 생성 → 결제
- **예약**: 좌석 배정 → 결제 → 티켓 발급

### 💡 비유
트랜잭션은 **"계약 체결"**과 같다.
- **모든 조건 충족**: 계약 성립 (Commit)
- **하나라도 불충족**: 전체 취소 (Rollback)

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         트랜잭션의 필요성                                              │
└─────────────────────────────────────────────────────────────────────────────────────────┘

초기 데이터베이스:
    • 파일 시스템 기반
    • 데이터 중복, 불일치
         ↓
트랜잭션 개념 등장 (1970년대):
    • System R (IBM)
    • ACID 특성 정의
    • 회복 불가능한 오류 해결
         ↓
관계형 데이터베이스:
    • SQL 표준 (COMMIT, ROLLBACK)
    • 동시성 제어 프로토콜
         ↓
현대:
    • 분산 트랜잭션
    • Saga, CQRS
```

## Ⅱ. 아키텍처 및 핵심 원리

### ACID 특성

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         ACID 특성                                                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Atomicity (원자성)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 트랜잭션의 모든 연산은 **반드시 전부 실행**되거나 **전부 실행 안 되어야 함**                         │
    │  │  • All-or-Nothing                                                                   │
    │  │  • COMMIT(완료) 또는 ROLLBACK(취소)                                                  │
    │                                                                                         │
    │  예: 이체 트랜잭션                                                                       │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  BEGIN TRANSACTION                                                                   │  │
    │  │  UPDATE accounts SET balance = balance - 1000 WHERE id = 'A';  -- 출금               │  │
    │  │  UPDATE accounts SET balance = balance + 1000 WHERE id = 'B';  -- 입금               │  │
    │  │  -- 둘 다 성공 → COMMIT                                                               │  │
    │  │  -- 하나라도 실패 → ROLLBACK (원래 상태 복원)                                          │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Consistency (일관성)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 트랜잭션 실행 전후에 **데이터베이스는 일관된 상태**를 유지해야 함                                   │
    │  │  • 무결성 제약 조건(Primary Key, Foreign Key, Check) 위배 불가                             │
    │  │  • 예: 계좌 잔고는 음수 불가능                                                          │
    │                                                                                         │
    │  예: 제약 조건 위배 시 Rollback                                                          │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  BEGIN TRANSACTION                                                                   │  │
    │  │  INSERT INTO orders (qty) VALUES (-5);  -- CHECK 제약 위배                          │  │
    │  │  -- 자동 ROLLBACK (일관성 위배)                                                       │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Isolation (격리성)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • **동시 실행 트랜잭션 간 서로 간섭하지 않아야 함**                                               │
    │  │  • 각 트랜잭션은 독립적으로 실행되는 것처럼 동작                                        │
    │  │  • Isolation Level로 제어 (READ UNCOMMITTED ~ SERIALIZABLE)                              │
    │                                                                                         │
    │  예: 격리성 필요성                                                                        │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  T1: A 계좌에서 출금 (SELECT balance → UPDATE)                                       │  │
    │  │  T2: A 계좌에서 출금 (SELECT balance → UPDATE)                                       │  │
    │  │  → 격리성 없으면 T1과 T2가 같은 balance를 읽고 각각 출금 → Lost Update                  │  │
    │  │  → 격리성 있으면 T1 Lock → T1 완료 후 T2 실행                                         │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Durability (지속성)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • **Commit된 트랜잭션은 영구적으로 저장**되어야 함                                            │
    │  │  • 시스템 장애 발생해도 Commit된 내용은 보존                                             │
    │  │  • WAL(Write-Ahead Logging)로 구현                                                    │
    │                                                                                         │
    │  예: 장애 복구                                                                           │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  1. 트랜잭션 Commit → WAL에 기록                                                     │  │
    │  │  2. 디스크에 반영 중 시스템 다운                                                       │  │
    │  │  3. 재시작 시 WAL 확인 → 미반영 트랜잭션 재실행 (Redo)                                 │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 트랜잭션 상태

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         트랜잭션 생명주기                                                │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  ┌─────────┐      ┌──────────┐      ┌──────────┐      ┌──────────┐      ┌─────────┐│
    │  │ Active  │ ───→ │ Partial  │ ───→ │ Committed│ ───→ │ Terminated│      │         ││
    │  │ (실행중) │      │ Commited │      │ (완료)    │      │ (종료)    │      │         ││
    │  └─────────┘      └──────────┘      └──────────┘      └──────────┘      │         ││
    │       │                                   │                                  │         ││
    │       │ Failed                            │                                  │         ││
    │       ▼                                   ▼                                  │         ││
    │  ┌──────────┐      ┌──────────┐      ┌──────────┐                           │         ││
    │  │ Failed   │ ───→ │ Aborted  │ ───→ │ Terminated│                           │         ││
    │  │ (실패)   │      │ (취소)   │      │ (종료)    │                           │         ││
    │  └──────────┘      └──────────┘      └──────────┘                           │         ││
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [상태 설명]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Active: 트랜잭션 실행 중                                                                │
    │  Partially Committed: 마지막 연산 완료, Commit 진행 중                                     │
    │  Failed: 실행 중 오류 발생                                                               │
    │  Aborted: Rollback 완료                                                                  │
    │  Committed: Commit 완료                                                                  │
    │  Terminated: 최종 상태 (Committed 또는 Aborted)                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 동시성 제어 이상 현상

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         동시성 제어 이상 현상                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [1. Lost Update (갱신 분실)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 두 트랜잭션이 같은 데이터를 동시에 갱신하여 하나의 갱신이 분실                                       │
    │                                                                                         │
    │  T1: READ balance = 1000                                                               │
    │  T2: READ balance = 1000                                                               │
    │  T1: balance = 1000 + 500 → WRITE balance = 1500                                       │
    │  T2: balance = 1000 + 200 → WRITE balance = 1200 (T1의 갱신 분실!)                       │
    │                                                                                         │
    │  해결: Lock, Isolation Level (Repeatable Read)                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [2. Dirty Read (더티 읽기)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Commit 안 된 데이터를 읽는 현상                                                        │
    │                                                                                         │
    │  T1: UPDATE balance = 500 (미 Commit)                                                   │
    │  T2: READ balance = 500 (T1의 미 Commit 데이터)                                         │
    │  T1: ROLLBACK → balance = 1000                                                           │
    │  T2: 읽은 데이터(500)는 실제로 존재하지 않음 (Dirty Read)                                 │
    │                                                                                         │
    │  해결: READ COMMITTED 이상                                                              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [3. Non-repeatable Read (반복 불가능 읽기)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 한 트랜잭션 내에서 같은 데이터를 두 번 읽을 때 값이 다른 현상                                   │
    │                                                                                         │
    │  T1: READ balance = 1000                                                                │
    │  T2: UPDATE balance = 1500 COMMIT                                                       │
    │  T1: READ balance = 1500 (같은 트랜잭션 내에서 값이 변경!)                              │
    │                                                                                         │
    │  해결: REPEATABLE READ 이상                                                             │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [4. Phantom Read (팬텀 읽기)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 한 트랜잭션 내에서 같은 쿼리를 두 번 실행했을 때 결과 행 수가 다른 현상                              │
    │                                                                                         │
    │  T1: SELECT * FROM accounts WHERE balance > 1000  -- 5건 반환                           │
    │  T2: INSERT INTO accounts VALUES (..., 2000) COMMIT                                     │
    │  T1: SELECT * FROM accounts WHERE balance > 1000  -- 6건 반환 (Phantom!)               │
    │                                                                                         │
    │  해결: SERIALIZABLE                                                                     │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Isolation Level

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         격리 수준 (Isolation Level)                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Isolation Level   │  Dirty Read  │  Non-repeatable  │  Phantom Read  │  Lock        │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  READ UNCOMMITTED  │  발생         │  발생             │  발생          │  없음        │
    │  READ COMMITTED    │  방지         │  발생             │  발생          │  갱신 Lock   │
    │  REPEATABLE READ   │  방지         │  방지             │  발생          │  공유 Lock   │
    │  SERIALIZABLE      │  방지         │  방지             │  방지          │  범위 Lock   │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [MySQL InnoDB 기본: REPEATABLE READ]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  SET TRANSACTION ISOLATION LEVEL READ COMMITTED;                                      │
    │  START TRANSACTION;                                                                     │
    │  -- 트랜잭션 연산                                                                         │
    │  COMMIT;                                                                                 │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 트랜잭션 vs 일반 SQL

| 구분 | 일반 SQL | 트랜잭션 |
|------|---------|----------|
| **보장** | 없음 | ACID |
| **취소** | 불가능 | Rollback |
| **동시성** | 이상 발생 가능 | 제어 가능 |
| **오버헤드** | 낮음 | Lock, Logging |

### 과목 융합 관점 분석

#### 1. 운영체제 ↔ 트랜잭션
- **WAL (Write-Ahead Logging)**: ARIES 알고리즘
- **Checkpoint**: 복구 시간 단축

#### 2. 분산 시스템 ↔ 트랜잭션
- **2PC (2-Phase Commit)**: 분산 트랜잭션
- **Saga**: 장기 실행 트랜잭션

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 이체 시스템
**상황**: 계좌 이체
**판단**:

```sql
-- Stored Procedure (트랜잭션 포함)
CREATE PROCEDURE transfer_money(
    IN from_account VARCHAR(20),
    IN to_account VARCHAR(20),
    IN amount DECIMAL(10,2)
)
BEGIN
    DECLARE EXIT HANDLER FOR SQLEXCEPTION
    BEGIN
        ROLLBACK;
        SELECT 'Error: Transfer failed' AS message;
    END;

    START TRANSACTION;

    -- 잔고 확인
    SELECT balance INTO @balance FROM accounts WHERE id = from_account FOR UPDATE;
    IF @balance < amount THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'Insufficient balance';
    END IF;

    -- 출금
    UPDATE accounts SET balance = balance - amount WHERE id = from_account;

    -- 입금
    UPDATE accounts SET balance = balance + amount WHERE id = to_account;

    -- 이체 내역 기록
    INSERT INTO transfers (from_account, to_account, amount, timestamp)
    VALUES (from_account, to_account, amount, NOW());

    COMMIT;
    SELECT 'Transfer completed' AS message;
END;
```

## Ⅴ. 기대효과 및 결론

### 트랜잭션 기대 효과

| 효과 | 트랜잭션 없을 시 | 트랜잭션 있을 시 |
|------|---------------|---------------|
| **데이터 일관성** | 깨질 수 있음 | 보장 |
| **장애 복구** | 불가능 | 가능 |
| **동시성 제어** | 이상 발생 | 제어 가능 |
| **오버헤드** | 낮음 | Lock, Logging |
