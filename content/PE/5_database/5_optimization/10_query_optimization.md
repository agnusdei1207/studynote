# 쿼리 최적화 (Query Optimization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쿼리 최적화는 **"SQL 쿼리의 **실행 계획을 **분석하고 **개선하여 **성능을 향상시키는 과정"**으로, **Optimizer**(Query Optimizer)가 **비용 기반**(Cost-Based) 또는 **규칙 기반**(Rule-Based)으로 **최적의 실행 계획**을 수립한다.
> 2. **가치**: **Full Table Scan**을 **Index Scan**으로 변환하고 **Nested Loop Join**을 **Hash Join** 또는 **Merge Join**으로 변경하며 **불필요한 컬럼 제거**, **Predicate Pushdown**으로 **처리 범위 축소** 및 **I/O 감소**를 달성한다.
> 3. **융합**: **EXPLAIN**(MySQL), **EXPLAIN ANALYZE**(PostgreSQL), **Execution Plan**으로 **쿼리 동작**을 **분석**하고 **Covering Index**, **Partitioning**, **Materialized View**와 결합하여 **대량 데이터** 처리 성능을 **최적화**한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
쿼리 최적화는 **"SQL을 가장 효율적인 방식으로 실행하도록 계획하고 개선하는 과정"**이다.

**최적화의 3단계**:
1. **Parsing**: SQL 문법 분석
2. **Optimization**: 실행 계획 수립
3. **Execution**: 계획 실행

### 💡 비유
쿼리 최적화는 **"내비게이션 경로 찾기"**와 같다.
- **쿼리**: 출발지 → 목적지
- **Optimizer**: 최적 경로 탐색
- **Execution Plan**: 턴별 지침

### Join Algorithm 비교

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Join Algorithm                                                  │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Nested Loop Join]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 외부 테이블의 각 행에 대해 내부 테이블 스캔                                             │
    │  │  • 소량 데이터, 인덱스 있을 때 유리                                                   │
    │  │  • 복잡도: O(M×N)                                                                   │
    │  │                                                                                      │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  for each row in outer_table:                                                       │  │
    │  │      for each row in inner_table:                                                    │  │
    │  │          if match_condition(row_outer, row_inner):                                   │  │
    │  │              emit joined_row                                                         │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │
    │  예: SELECT * FROM orders o JOIN users u ON o.user_id = u.id WHERE o.id = 123;          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Hash Join]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Build 단계: 작은 테이블을 해시 테이블 생성                                              │
    │  │  • Probe 단계: 큰 테이블 스캔하며 해시 조회                                            │
    │  │  • 대량 데이터, 인덱스 없을 때 유리                                                    │
    │  │  • 복잡도: O(M+N)                                                                   │
    │  │                                                                                      │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  # Build Phase (작은 테이블)                                                          │  │
    │  │  hash_table = {}                                                                    │  │
    │  │  for row in small_table:                                                             │  │
    │  │      hash_key = hash(row.join_key)                                                   │  │
    │  │      hash_table[hash_key].append(row)                                                │  │
    │  │                                                                                      │  │
    │  │  # Probe Phase (큰 테이블)                                                           │  │
    │  │  for row in large_table:                                                             │  │
    │  │      hash_key = hash(row.join_key)                                                   │  │
    │  │      if hash_key in hash_table:                                                      │  │
    │  │          for match in hash_table[hash_key]:                                          │  │
    │  │              emit joined_row(match, row)                                              │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │
    │  예: SELECT * FROM orders o JOIN users u ON o.user_id = u.id;                            │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Merge Join]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 양쪽 테이블 정렬 후 병합                                                              │
    │  │  • 이미 정렬된 데이터, 대량 데이터에 유리                                               │
    │  │  • 복잡도: O(M log M + N log N)                                                      │
    │  │                                                                                      │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  # Sort both tables by join key (if not sorted)                                      │  │
    │  │  table1_sorted = sort(table1, key=join_key)                                          │  │
    │  │  table2_sorted = sort(table2, key=join_key)                                          │  │
    │  │                                                                                      │
    │  │  # Merge                                                                             │  │
    │  │  i, j = 0, 0                                                                         │  │
    │  │  while i < len(table1_sorted) and j < len(table2_sorted):                            │  │
    │  │      if table1_sorted[i].key == table2_sorted[j].key:                                │  │
    │  │          emit joined_row(table1_sorted[i], table2_sorted[j])                          │  │
    │  │          j += 1                                                                      │  │
    │  │      elif table1_sorted[i].key < table2_sorted[j].key:                                │  │
    │  │          i += 1                                                                      │  │
    │  │      else:                                                                           │  │
    │  │          j += 1                                                                      │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Access Method 비교

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Access Method                                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Method        │  Description           │  Use Case                     │  Cost         │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  Full Scan     │  전체 페이지 순차 읽기     │  소량 테이블, 대부분 행 조회      │  O(N)         │
    │  Index Scan    │  인덱스 순차 읽기         │  범위 조회 (BETWEEN, >, <)     │  O(log N + K) │
    │  Index Lookup  │  인덱스 직접 접근         │  동등 조건 (=, IN)              │  O(log N)     │
    │  Covering      │  인덱스만으로 조회 완료   │  인덱스에 모든 컬럼 포함         │  O(log N)     │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [예시: WHERE id = 123]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  -- Full Scan (인덱스 없음)                                                              │
    │  • 모든 데이터 페이지 읽기                                                               │
    │  • 느림                                                                                 │
    │                                                                                         │
    │  -- Index Lookup (PK 인덱스)                                                             │
    │  • B-Tree 탐색: O(log N)                                                                │
    │  • 데이터 페이지 직접 접근                                                               │
    │  • 빠름                                                                                 │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 쿼리 최적화 기법

| 기법 | 설명 | 예시 |
|------|------|------|
| **Predicate Pushdown** | 필터를 아래로 이동 | WHERE 먼저 적용 |
| **Projection Pushdown** | 불필요한 컬럼 제거 | SELECT 최소화 |
| **Join Reordering** | 순서 변경 | 작은 테이블 먼저 |
| **Materialization** | 중간 결과 저장 | CTE, Temp Table |

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 느린 쿼리 최적화
**상황**: 주문 조회 쿼리가 느림
**판단**:

```sql
-- 1. 문제 쿼리
SELECT o.id, u.name, p.name
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id
WHERE o.status = 'pending'
  AND o.created_at > '2026-01-01';

-- 2. EXPLAIN 확인 (PostgreSQL)
EXPLAIN ANALYZE
SELECT o.id, u.name, p.name
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id
WHERE o.status = 'pending'
  AND o.created_at > '2026-01-01';

-- 3. 인덱스 추가
CREATE INDEX idx_orders_status_created ON orders(status, created_at);
CREATE INDEX idx_orders_user ON orders(user_id);
CREATE INDEX idx_orders_product ON orders(product_id);

-- 4. 통계 정보 업데이트
ANALYZE orders;
ANALYZE users;
ANALYZE products;

-- 5. 다시 EXPLAIN 확인
-- → Index Scan 사용, Cost 감소 확인

-- 6. 커버링 인덱스로 최적화 (필요한 경우)
CREATE INDEX idx_orders_covering ON orders(status, created_at)
INCLUDE (user_id, product_id);
```
