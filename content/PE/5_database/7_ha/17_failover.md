# 장애 조치 (Failover)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 장애 조치는 **"Primary DB **장애 시 **Standby**(대기) 서버를 **새 Primary**로 **승격**(Promotion)시켜 **서비스 중단**(Downtime)을 **최소화**하는 **HA 기술"**으로, **자동 감지**(Detection), **자동 승격**(Automatic Promotion), **DNS/IP 변경**(Re routing)으로 **구현**된다.
> 2. **가치**: **SPOF**(Single Point of Failure)를 **제거**하고 **RTO**(Recovery Time Objective)를 **초~분 단위**로 **단축**하며 **Data Loss**(RPO)를 **최소화**하기 위해 **Synchronous Replication**(동기 복제) 또는 **Semi-Sync**(반동기 복제)를 **사용**한다.
> 3. **융합**: **MySQL InnoDB Cluster**(MySQL Shell), **PostgreSQL Patroni**(etcd, HAProxy), **MongoDB Replica Set**(Automatic Failover), **Oracle Data Guard**가 **대표적**이며 **Health Check**(TCP, HTTP, DB Query)로 **장애 감지**하고 **Quorum**(과반수)으로 **Split Brain**을 **방지**한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
장애 조치는 **"Primary 장애 시 Standby로 전환하는 HA 기술"**이다.

**Failover 특징**:
- **자동 감지**: Health Check
- **자동 승격**: Promotion
- **최소 중단**: Fast RTO
- **데이터 보존**: Replication

### 💡 비유
Failover는 **"비행기 조종사 교체****와 같다.
- **주 조종사**: Primary
- **부 조종사**: Standby
- **자동 교체**: Auto Failover

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         HA 발전                                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘

Single DB:
    • SPOF
    • 장애 시 다운타임
         ↓
Cold Standby:
    • 수동 복구
    • 긴 다운타임
         ↓
Warm Standby:
    • 복제 진행
    • 수동 승격
         ↓
Hot Standby + Auto Failover:
    • 실시간 복제
    • 자동 승격
```

## Ⅲ. 융합 비교 및 다각도 분석

### Standby 타입 비교

| 타입 | 복제 | 데이터 | RTO | 복잡도 |
|------|------|--------|-----|--------|
| **Cold** | 없음 | 없음 | 길음 | 낮음 |
| **Warm** | 비동기 | 일부 | 중간 | 중간 |
| **Hot** | 동기/반동기 | 최신 | 짧음 | 높음 |

### RTO/RPO 비교

| 구성 | RTO | RPO | 비용 |
|------|-----|-----|------|
| **Cold Standby** | 시간~일 | 긴 시간 | 낮음 |
| **Async Replication** | 분 | 초~분 | 중간 |
| **Semi-Sync** | 초~분 | 초 | 높음 |
| **Sync Replication** | 초 | 0 | 매우 높음 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: PostgreSQL Patroni Failover
**상황**: HA 구성
**판단**:

```yaml
# patroni.yml

scope: postgres-cluster
name: postgresql-0

restapi:
  listen: 0.0.0.0:8008
  connect_address: 10.0.0.1:8008

etcd:
  hosts:
    - 10.0.0.10:2379
    - 10.0.0.11:2379
    - 10.0.0.12:2379

bootstrap:
  dcs:
    ttl: 30
    loop_wait: 10
    retry_timeout: 10
    maximum_lag_on_failover: 1048576

postgresql:
  listen: 0.0.0.0:5432
  connect_address: 10.0.0.1:5432
  data_dir: /var/lib/postgresql/data
  replication:
    user: replicator
    network: 10.0.0.0/24

tags:
  nofailover: false
  noloadbalance: false
  clonefrom: false
  nosync: false
```

```sql
-- 복제 상태 확인
SELECT client_addr, state, sync_state, sync_priority
FROM pg_stat_replication;

-- Lag 확인
SELECT
    now() - pg_last_xact_replay_timestamp() AS replication_lag;
```

## Ⅴ. 기대효과 및 결론

### Failover 기대 효과

| 효과 | Failover 없음 | Failover |
|------|--------------|----------|
| **RTO** | 시간~일 | 초~분 |
| **RPO** | 긴 시간 | 0~초 |
| **자동화** | 수동 | 자동 |
| **비용** | 낮음 | 높음 |

### 모범 사례

1. **Health Check**: 여러 방식 사용
2. **Quorum**: 홀수 노드
3. **테스트**: 정기적 Failover 테스트
4. **모니터링**: 알림 설정

### 미래 전망

1. **Cloud Failover**: 자동화
2. **Multi-Active**: 다중 Active
3. **Serverless**: Cloud Native

- **MySQL**: InnoDB Cluster
