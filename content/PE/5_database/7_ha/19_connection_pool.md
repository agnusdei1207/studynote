# 커넥션 풀 (Connection Pool)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 커넥션 풀은 **"DB **연결**(Connection)을 **미리 생성**하여 **재사용**하는 **Pool** **기술"**으로, **TCP 3-Way Handshake**, **Authentication**과 같은 **비싼 **연결 **비용**을 **줄이고 **Connection Limit**을 **관리**한다.
> 2. **동작 원리**: **클라이언트**가 **DB**에 **접속**하면 **Pool**에서 **Idle Connection**을 **반환**하고 **사용 후**에는 **Connection**을 **닫지 않고 **Pool**로 **반환**(Return)하며 **Pool Size**(Min, Max)로 **리소스**를 **제어**한다.
> 3. **융합**: **HikariCP**(Java), **PgBouncer**(PostgreSQL), **ProxySQL**(MySQL), **cp**(Python), **sequelize**(Node.js)가 **대표적**이며 **Client-side Pool**(HikariCP)과 **Server-side Pool**(PgBouncer)로 **구분**된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
커넥션 풀은 **"DB 연결을 재사용하는 기술"**이다.

**커넥션 풀 이점**:
- **성능**: 연결 오버헤드 감소
- **제어**: 동시 연결 수 제한
- **관리**: 유휴 연결 재사용
- **안정성**: 리소스 고갈 방지

### 💡 비유
커넥션 풀은 **"렌터카****와 같다.
- **차량**: DB 연결
- **대여**: Connection 획득
- **반납**: Connection 해제
- **관리**: Pool Size

### 연결 생명주기

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Connection Lifecycle                                              │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. Pool Initialization                                                                │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  • minIdle만큼 연결 생성                                                              │  │  │
    │  │  │  • TCP 3-Way Handshake                                                            │  │  │
    │  │  │  • Authentication                                                                  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │  │
    │  2. getConnection()                                                                     │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  if (idle connection exists) {                                                       │  │  │
    │  │      return idle connection;    // 즉시 반환                                         │  │  │
    │  │  } else if (maxPoolSize not reached) {                                             │  │  │
    │  │      create new connection;     // 새 연결 생성                                      │  │  │
    │  │  } else {                                                                               │  │  │
    │  │      wait(connectionTimeout);   // 대기 (Connection이 반납될 때까지)                    │  │  │
    │  │  }                                                                                    │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │  │
    │  3. returnConnection()                                                                  │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  • 연결 닫지 않고 Pool로 반환                                                           │  │  │
    │  │  │  • 유휴 큐(IDLE Queue)에 추가                                                        │  │  │
    │  │  │  • idleTimeout 동안 사용되지 않으면 연결 닫기                                        │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### Pool 구현 비교

| 구현 | 위치 | 장점 | 단점 |
|------|------|------|------|
| **Client-side** | Application | 구현 간단 | 연결 낭비 |
| **Server-side** | Middleware | 효율적 | 추가 서버 |

### Pool 도구 비교

| 도구 | DB | 타입 | 특징 |
|------|-----|------|------|
| **HikariCP** | JDBC | Client | 빠름, 경량 |
| **PgBouncer** | PostgreSQL | Server | 효율적 |
| **ProxySQL** | MySQL | Server | Query 캐시 |
| **cp** | Python | Client | 간단함 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: HikariCP 설정
**상황**: Spring Boot
**판단**:

```yaml
# application.yml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/mydb
    username: user
    password: pass
    hikari:
      maximum-pool-size: 20          # 최대 연결 수
      minimum-idle: 5                # 최소 유휴 연결
      connection-timeout: 3000       # 3초 대기
      idle-timeout: 600000           # 10분 유휴 타임아웃
      max-lifetime: 1800000          # 30분 최대 수명
      connection-test-query: SELECT 1  # 유효성 검사
```

## Ⅴ. 기대효과 및 결론

### 커넥션 풀 기대 효과

| 효과 | Pool 없음 | Pool 있음 |
|------|----------|----------|
| **연결 시간** | 항상 생성 | 재사용 |
| **동시 연결** | 무제한 | 제어됨 |
| **CPU/메모리** | 낭비 | 절약 |

### 모범 사례

1. **Pool Size**: CPU 코어 * 2 + 디스크 수
2. **Timeout**: 적절한 대기 시간
3. **Validation**: 유효성 검사
4. **모니터링**: 사용량 추적

### 미래 전망

1. **Adaptive Pool**: 자동 조정
2. **Connection Mesh**: 분산 Pool
3. **Serverless**: 짧은 연결
