# 3NF (제3정규형: Third Normal Form)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 제3정규형(3NF)은 **"2NF를 만족하고 **이행적 함수적 종속(Transitive Functional Dependency)**을 제거한 정규형"**으로, **비키 속성(Non-key Attribute)**이 **다른 비키 속성**에 종속되는 관계를 **분리**하여 **독립 테이블로** 만든다.
> 2. **가치**: **데이터 중복(Data Redundancy)**을 최소화하고 **갱신 이상(Update Anomaly)**을 제거하며, **참조 무결성(Referential Integrity)**을 강화하고 **데이터 모델링**의 **안정성**을 확보한다.
> 3. **융합**: **BCNF(Boyce-Codd Normal Form)**로 확장 가능하며, **ER 모델링(Entity-Relationship)**에서 **Strong Entity**와 **Weak Entity**를 구분하고 **M:N 관계**를 **1:N + 연결 테이블**로 변환한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
제3정규형(Third Normal Form, 3NF)은 **"2NF를 만족하고 이행적 종속을 제거한 릴레이션"**이다.

**3NF의 정의**:
- **제2정규형(2NF)을 만족**
- **비키 속성이 기본키에 이행적으로 종속인 경우 분리**

**이행적 종속(Transitive Dependency)**:
- A → B, B → C이면 A → C (이행적)
- C는 A가 아닌 B에 종속

### 💡 비유
3NF는 **"조직도"**와 같다.
- **문제**: 사원 → 부서 → 건물 (건물 정보 중복)
- **해결**: 사원-부서, 부서-건물 분리

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         제3정규형의 발견                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

2NF 문제:
    • 부분 종속 제거했지만 이행 종속 존재
    • 비키 속성 간 종속으로 갱신 이상 발생
         ↓
Codd (1971년) 3NF 제안:
    • 이행적 종속 제거
    • 비키 속성의 간접 종속 해결
         ↓
BCNF 확장:
    • 모든 결정자가 후보키
    • 더 강력한 제약조건
```

## Ⅱ. 아키텍처 및 핵심 원리

### 3NF 위반 사례와 해결

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         3NF 위반: 이행적 종속                                        │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [비정규화 테이블: EMPLOYEE]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  emp_id │ emp_name │ dept_id │ dept_name │ building │ floor                         │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  1001   │ 김철수    │ D10     │ 개발팀    │ A동      │ 3층                          │
    │  1002   │ 이영희    │ D10     │ 개발팀    │ A동      │ 3층                          │
    │  1003   │ 박민수    │ D20     │ 인사팀    │ B동      │ 2층                          │
    │  1004   │ 최수진    │ D20     │ 인사팀    │ B동      │ 2층                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    PK: emp_id

    [문제: 이행적 종속]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  emp_id → dept_id                                                                      │
    │  dept_id → dept_name, building, floor                                                   │
    │                                                                                         │
    │  → dept_name, building, floor는 dept_id에 종속                                          │
    │  → emp_id에 이행적으로 종속 (3NF 위반)                                                   │
    │                                                                                         │
    │  문제점:                                                                                 │
    │  • D10 부서가 A동에서 C동으로 이동 시 모든 행 수정 필요                                   │
    │  • dept_name 중복 (개발팀이 2번)                                                        │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [3NF 해결: 분해]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  [EMPLOYEE]                                                                             │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  emp_id (PK)                                                                     │  │
    │  │  emp_name                                                                        │  │
    │  │  dept_id (FK)                                                                    │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                              │ FK                                                        │
    │                              ▼                                                        │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  [DEPARTMENT]                                                                           │  │
    │  │  ┌────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  │  dept_id (PK)                                                                    │  │  │
    │  │  │  dept_name                                                                      │  │  │
    │  │  │  building                                                                       │  │  │
    │  │  │  floor                                                                          │  │  │
    │  │  └────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    • dept_name, building, floor는 DEPARTMENT에만 저장
    • 중복 제거, 갱신 이상 해결
```

### 이행적 종속 식별

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         이행적 종속 식별 방법                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 1: 함수적 종속 확인]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  기본키(Primary Key) 확인                                                               │
    │  • A: 기본키                                                                            │
    │  • B, C, D: 비키 속성                                                                   │
    │                                                                                         │
    │  함수적 종속 관계 파악:                                                                   │
    │  1. A → B (직접 종속)                                                                   │
    │  2. A → C (직접 종속)                                                                   │
    │  3. B → D (비키 간 종속)                                                                │
    │  4. 따라서 A → D (이행적 종속)                                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 2: 이행적 종속 분리]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  원본 테이블 R(A, B, C, D)                                                               │
    │                                                                                         │
    │  분해:                                                                                   │
    │  • R1(A, B, C): 기본 테이블                                                             │
    │  • R2(B, D): 종속 테이블                                                                │
    │                                                                                         │
    │  외래키:                                                                                 │
    │  • R1.B → R2.B                                                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [예시: 주문 시스템]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  [이행적 종속 존재]                                                                      │
    │  ORDER(order_id, cust_id, cust_name, cust_addr, product_id, qty)                         │
    │                                                                                         │
    │  종속 관계:                                                                              │
    │  • order_id → cust_id                                                                   │
    │  • cust_id → cust_name, cust_addr                                                       │
    │  → cust_name, cust_addr는 cust_id에 종속 (이행적)                                         │
    │                                                                                         │
    │  [3NF 분해]                                                                              │
    │  ORDER(order_id, cust_id, product_id, qty)                                              │
    │  CUSTOMER(cust_id, cust_name, cust_addr)                                                │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 2NF → 3NF 변환

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         2NF → 3NF 변환 절차                                              │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 1: 비키 속성 간 종속 식별]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  기본키(A, B)에서                                                                       │
    │  • A → C (부분 종속: 2NF에서 제거)                                                       │
    │  • C → D (이행 종속: 3NF에서 제거)                                                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 2: 이행 종속 분리]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • C가 결정자인 경우 C와 D를 별도 테이블로 분리                                            │
    │  • 원본 테이블에서 D 제거                                                               │
    │  • 외래키(C)로 연결                                                                     │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 3NF vs BCNF

| 구분 | 3NF | BCNF |
|------|-----|------|
| **조건** | 비키 속성의 이행 종속 제거 | 모든 결정자가 후보키 |
| **제약 강도** | 약함 | 강함 |
| **다치 종속** | 허용 | 제거 |
| **복합 키** | 허용 | 제한적 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 이커머스 주문 정규화
**상황**: 주문(Order)에 배송지(Shipping) 정보 중복
**판단**:

1. **2NF**: 부분 종속 제거
2. **3NF**: 이행 종속 제거

```sql
-- 3NF 미준수 (문제)
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    customer_name VARCHAR(50),
    customer_address VARCHAR(200),
    shipping_region VARCHAR(50),
    shipping_fee INT,
    order_date DATE
);

-- 3NF 준수 (해결)
-- Customer 테이블
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    customer_name VARCHAR(50),
    customer_address VARCHAR(200)
);

-- Shipping Region 테이블
CREATE TABLE shipping_regions (
    region_id INT PRIMARY KEY,
    region_name VARCHAR(50),
    shipping_fee INT
);

-- Orders 테이블 (3NF)
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    region_id INT,
    order_date DATE,
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id),
    FOREIGN KEY (region_id) REFERENCES shipping_regions(region_id)
);
```

## Ⅴ. 기대효과 및 결론

### 3NF 기대 효과

| 효과 | 3NF 미달성 | 3NF 준수 |
|------|----------|----------|
| **중복** | 비키 속성 중복 | 제거 |
| **갱신** | 연쇄 수정 필요 | 단일 테이블만 수정 |
| **삽입** | 중복된 데이터 입력 필요 | 종속 테이블만 추가 |
| **삭제** | 연쇄 삭제 위험 | 참조 정의로 안전 삭제 |
