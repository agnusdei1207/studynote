# 정규화 개요 (Normalization Overview)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정규화는 **"이상 현상(Anomaly)을 제거하고 데이터 무결성을 보장하기 위해 릴레이션을 분해하는 과정"**으로, **함수적 종속(Functional Dependency)**를 기반으로 **제1정규형(1NF)**, **제2정규형(2NF)**, **제3정규형(3NF)**, **BCNF(Boyce-Codd Normal Form)**로 단계적 분해한다.
> 2. **가치**: **삽입 이상(Insertion Anomaly)**, **삭제 이상(Deletion Anomaly)**, **갱신 이상(Update Anomaly)**을 제거하여 **데이터 중복(Data Redundancy)**을 최소화하고 **저장 공간**을 절약하며, **데이터 일관성(Data Consistency)**을 보장한다.
> 3. **융합**: **ER 모델링**의 개체-관계 정의 후 **물리적 설계**의 전 단계로 수행되며, **NoSQL(Denormalization)**과의 Trade-off 관계에서 **OLTP(Transation Processing)**에는 정규화, **OLAP(Analytical Processing)**에는 반정규화를 적용하는 의사결정 기준이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
정규화(Normalization)는 **"데이터베이스 설계 시 데이터 중복과 이상 현상을 제거하는 체계적인 과정"**이다.

**정규화의 목적**:
- **데이터 중복 최소화**: 저장 공간 절약
- **이상 현상 제거**: 삽입, 삭제, 갱신 이상 방지
- **무결성 보장**: 데이터 일관성 유지
- **유연성 확보**: 스키마 변경 최소화

### 💡 비유
정규화는 **"서류 정리"**와 같다.
- **중복 제거**: 같은 내용 여러 곳에 ❌
- **분류**: 관련 내용별로 파일 분리
- **참조**: 필요 시 cross-reference

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         정규화의 필요성                                                │
└─────────────────────────────────────────────────────────────────────────────────────────┘

파일 시스템:
    • 데이터 중복 심각
    • 일관성 유지 어려움
    • 이상 현상 빈발
         ↓
관계형 모델 (1970년, E.F. Codd):
    • 정규화 이론 제시
    • 함수적 종속 개념 도입
    • 1NF, 2NF, 3NF 정의
         ↓
BCNF (1974년, Boyce, Codd):
    • 3NF의 확장
    • 더 엄격한 정의
    • 모든 결정자가 후보키
         ↓
고급 정규형 (1970년대~):
    • 4NF: 다치 종속 제거
    • 5NF: 조인 종속 제거
    • DKNF: Domain-Key Normal Form
```

## Ⅱ. 아키텍처 및 핵심 원리

### 함수적 종속(Functional Dependency)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         함수적 종속 (Functional Dependency)                             │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [정의]

    X → Y (X determines Y)

    속성 X의 값이 결정되면 Y의 값이 유일하게 결정됨

    [예시: EMPLOYEE 테이블]

    employee_id → name, dept_id, position, salary
    dept_id → dept_name, location, budget

    [용어]

    • 결정자(Determinant): X (함수적 종속의 왼쪽)
    • 종속자(Dependent): Y (함수적 종속의 오른쪽)
    • 후보키(Candidate Key): 모든 속성을 결정할 수 있는 최소 집합
    • 기본키(Primary Key): 선택된 후보키

    [함수적 종속 규칙]

    1. 반사성(Reflexivity): X ⊇ Y이면 X → Y
    2. 증가성(Augmentation): X → Y이면 XZ → YZ
    3. 이행성(Transitivity): X → Y, Y → Z이면 X → Z
    4. 합병성(Union): X → Y, X → Z이면 X → YZ
    5. 분해성(Decomposition): X → YZ이면 X → Y, X → Z
```

### 이상 현상(Anomaly)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         이상 현상 (Anomalies)                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [비정규화 테이블: ORDER_ITEM]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  customer_id │ customer_name │ product_id │ product_name │ supplier │ order_qty       │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  100         │ 홍길동       │ P001       │ 노트북      │ Samsung  │ 2               │
    │  100         │ 홍길동       │ P002       │ 마우스      │ Logitech │ 1               │
    │  101         │ 김철수       │ P001       │ 노트북      │ Samsung  │ 1               │
    │  100         │ 홍길동       │ P003       │ 키보드      │ Logitech │ 1               │
    │  102         │ 이영희       │ P001       │ 노트북      │ Samsung  │ 3               │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
    문제점:
    • customer_name 중복 (홍길동이 3번)
    • product_name 중복 (노트북이 3번)
    • supplier 중복 (Samsung이 3번)

    ┌─────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. 삽입 이상 (Insertion Anomaly)                                                        │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  문제: 주문이 없는 고객 정보를 삽입할 수 없음                                            │
    │                                                                                         │
    │  예: 신규 고객 '박민수' 등록                                                               │
    │  • 주문하지 않았기 때문에 product_id가 NULL                                              │
    │  • 하지만 product_id는 주문 항목의 식별자                                                  │
    │  • → 삽입 불가                                                                         │
    └─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────────────────────────┐
    │  2. 삭제 이상 (Deletion Anomaly)                                                         │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  문제: 주문 삭제 시 연관된 정보도 소실                                                     │
    │                                                                                         │
    │  예: customer_id=101인 주문 취소                                                          │
    │  • 김철수의 유일한 주문 삭제                                                              │
    │  • → 김철수의 정보도 함께 삭제됨                                                         │
    │  • → 김철수 정보 소실                                                                    │
    └─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────────────────────────┐
    │  3. 갱신 이상 (Update Anomaly)                                                           │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  문제: 정보 갱신 시 중복된 모든 행을 갱신해야 함                                           │
    │                                                                                         │
    │  예: 홍길동의 주소 변경                                                                  │
    │  • 홍길동은 3개의 주문에 중복                                                            │
    │  • 3개 행 모두 갱신해야 함                                                               │
    │  • 1개라도 누락 시 일관성 깨짐                                                           │
    └─────────────────────────────────────────────────────────────────────────────────────────┘
```

### 정규화 단계별 분해

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         정규화 단계별 분해                                               │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [비정규형 (Unnormalized Form)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  ORDER_ITEM                                                                            │
    │  (customer_id, customer_name, product_id, product_name, supplier, order_qty)            │
    │                                                                                         │
    │  함수적 종속:                                                                            │
    │  • customer_id → customer_name                                                          │
    │  • product_id → (product_name, supplier)                                                │
    │  • (customer_id, product_id) → order_qty                                                │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 제1정규형 (1NF)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1NF: 모든 속성이 원자값(Atomic Value)                                                 │
    │                                                                                         │
    │  • 반복 그룹 제거                                                                       │
    │  • 복합 속성 분해                                                                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 제2정규형 (2NF)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  2NF: 부분적 함수 종속 제거                                                             │
    │                                                                                         │
    │  [분해]                                                                                  │
    │  • CUSTOMER(customer_id, customer_name)                                                 │
    │  • PRODUCT(product_id, product_name, supplier)                                          │
    │  • ORDER(customer_id, product_id, order_qty)                                             │
    │                                                                                         │
    │  • 기본키에 속하지 않은 속성이 기본키의 일부에 종속하는 것 제거                            │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 제3정규형 (3NF)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  3NF: 이행적 함수 종속 제거                                                             │
    │                                                                                         │
    │  [분해]                                                                                  │
    │  • CUSTOMER(customer_id, customer_name)                                                 │
    │  • PRODUCT(product_id, product_name)                                                    │
    │  • SUPPLIER(supplier, supplier_name, address)                                            │
    │  • ORDER(customer_id, product_id, order_qty)                                             │
    │                                                                                         │
    │  • 기본키에 속하지 않은 속성 간 종속 제거                                                 │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ BCNF (Boyce-Codd Normal Form)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  BCNF: 모든 결정자가 후보키                                                              │
    │                                                                                         │
    │  • 3NF보다 엄격한 정의                                                                   │
    │  • 중복된 후보키 문제 해결                                                                │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 정규화 vs 반정규화

| 구분 | 정규화(Normalization) | 반정규화(Denormalization) |
|------|----------------------|------------------------|
| **목적** | 중복 제거, 무결성 | 성능 향상, 조회 단순화 |
| **적용** | OLTP(트랜잭션) | OLAP(분석), 데이터 웨어하우스 |
| **장점** | 일관성, 저장 공간 절약 | 조회 성능, JOIN 감소 |
| **단점** | JOIN 오버헤드 | 중복, 갱신 비용 |

### 과목 융합 관점 분석

#### 1. 데이터 웨어하우스 ↔ 정규화
- **Star Schema**: 차원 테이블 반정규화
- **Snowflake Schema**: 완전 정규화
- **Fact Table**: 측정값 중심

#### 2. NoSQL ↔ 정규화
- **Schema-less**: 정규화 불필요
- **Document Store**: 중보 데이터 허용
- **ACID vs BASE**: 일관성 Trade-off

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 전자상거래 주문 시스템 설계
**상황**: OLAP/OLAP 혼합
**판단**:
1. **OLTP(주문 처리)**: 3NF 정규화
   - 무결성 중요
   - 트랜잭션 빈번
2. **OLAP(매출 분석)**: 반정규화
   - 조회 성능 중요
   - 배치로 갱신

```sql
-- OLTP: 정규화 (3NF)
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);

CREATE TABLE order_items (
    item_id INT PRIMARY KEY,
    order_id INT,
    product_id INT,
    quantity INT,
    FOREIGN KEY (order_id) REFERENCES orders(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);

-- OLAP: 반정규화 (Star Schema)
CREATE TABLE fact_orders (
    order_id INT,
    order_date DATE,
    customer_id INT,
    customer_name VARCHAR(100),  -- 중복 허용
    product_id INT,
    product_name VARCHAR(100),   -- 중복 허용
    quantity INT,
    amount DECIMAL(10,2)
);
```

## Ⅴ. 기대효과 및 결론

### 정규화 기대 효과

| 효과 | 정규화 미적용 | 정규화 적용 |
|------|-------------|-----------|
| **데이터 중복** | 높음 | 낮음 |
| **저장 공간** | 큼 | 작음 |
| **갱신 비용** | 높음 | 낮음 |
| **조회 성능** | 높음 | 낮음(JOIN) |
| **무결성** | 취약 | 강함 |

### 미래 전망

1. **자동 정규화**: AI 기반 스키마 추천
2. **Hybrid Schema**: 정규화 + 반정규화 혼합
3. **Temporal Database**: 시계열 데이터 정규화
