# 제1정규형 (1NF: First Normal Form)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 제1정규형(1NF)은 **"모든 속성이 원자값(Atomic Value)을 가져야 하며 반복 그룹(Repeating Group)이 없어야 하는 정규형"**으로, **도메인의 원자성(Atomicity)**을 보장하고 **각 셀이 단일 값(Single Value)**만을 가진다.
> 2. **가치**: **반복 그룹 제거**를 통해 **데이터 중복(Data Redundancy)**을 최소화하고 **검색(Search)**, **정렬(Sort)**, **조인(Join)** 연산을 단순화하며, **관계형 모델(Relational Model)**의 **기본 전제**를 만족한다.
> 3. **융합**: **NoSQL(JSON)**의 **중첩 구조(Nested Structure)**와 대비되며, **Array Type**, **JSONB**(PostgreSQL), **XML Type**은 **Relational DBMS**에서 1NF를 완화하면서도 **계층적 데이터**를 표현하는 **대안**으로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
제1정규형(First Normal Form, 1NF)은 **"릴레이션의 모든 속성값이 원자적(Atomic)이어야 한다"**는 조건이다.

**1NF의 정의 (Codd, 1970)**:
- 도메인의 원자성(Atomic Domain)
- 반복 그룹(Repeating Group) 부재
- 각 셀은 단일 값(Single Value)

### 💡 비유
1NF은 **"엑셀 셀 하나에 값 하나"**와 같다.
- **O(정상)**: `A`, `100`, `홍길동`
- **X(비정상)**: `A,B,C`, `[1,2,3]`, `{name:"홍길동", age:30}`

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         제1정규형의 필요성                                              │
└─────────────────────────────────────────────────────────────────────────────────────────┘

파일 시스템 (Hierarchical):
    • 중첩된 구조 (Tree, Graph)
    • 반복 그룹
    • 복잡한 검색
         ↓
관계형 모델 (1970년, E.F. Codd):
    • 평탄한 구조 (Table)
    • 원자값(Atomic Value)
    • 단순한 SQL 쿼리
         ↓
1NF 정의 (1970년대):
    • 반복 그룹 제거
    • 모든 속성이 원자값
    • 관계형 모델의 기본
         ↓
현대 데이터베이스:
    • 1NF는 기본 전제
    • JSON/Array 타입으로 유연화
```

## Ⅱ. 아키텍처 및 핵심 원리

### 1NF 위반 사례와 해결

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         1NF 위반: 반복 그룹 (Repeating Group)                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [비정규화 테이블: STUDENT_COURSE]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  student_id │ name       │ courses (반복 그룹)                          │ grade_range      │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  1001       │ 홍길동     │ {"CS101", "MATH201", "ENG301"}              │ A ~ A+          │
    │  1002       │ 김철수     │ {"CS101"}                                   │ B                │
    │  1003       │ 이영희     │ {"MATH201", "ENG301"}                       │ B+ ~ A           │
    │  1004       │ 박민수     │ {"CS101", "CS202", "MATH201", "ENG301"}   │ A ~ B+           │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
                        │
                        ▼
    [문제점]
    • courses 컬럼이 복합 값 (JSON Array)
    • course_id별 검색 어려움
    • grade_range도 범위 값

    [1NF 해결: 반복 그룹 제거]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  [STUDENT]                                                                             │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  student_id │ name       │                                                       │  │

    │  │  1004       │ 박민수     │                                                       │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                              │                                                          │
    │                              │ FK                                                        │
    │                              ▼                                                          │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  [ENROLLMENT]                                                                        │  │
    │  │  ┌────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  │  student_id │ course_id │ grade                                              │  │  │
    │  │  │ 1001        │ CS101     │ A                                                 │  │  │
    │  │  │ 1001        │ MATH201   │ B+                                                │  │  │

    └──────────────────────────────────────────────────────────────────────────────────────────┘

    • 모든 속성이 원자값 (Atomic Value)
    • 반복 그룹 제거
```

### 원자값(Atomic Value)의 정의

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         원자값(Atomic Value) vs 복합 값(Composite Value)                       │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [원자값(Atomic Value) 예시]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 단일 값: "홍길동", 25, TRUE                                                             │
    │  │  분해 불가능한 단위                                                                    │
    │  │  SQL에서의 기본 타입: INT, VARCHAR, DATE, BOOLEAN                                     │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [비원자값(Non-atomic Value) 예시]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. 쉼표(Comma)로 구분된 값:                                                               │
    │     "CS101, MATH201, ENG301"                                                              │
    │     → 각 course_id를 별도 행으로 분해                                                        │
    │                                                                                         │
    │  2. 중첩 구조(Nested Structure):                                                          │
    │     {"name": "홍길동", "age": 25, "addr": {"city": "서울", "gu": "강남구"}}                    │
    │     → addr를 별도 테이블로 분해하거나 city, gu 컬럼으로 분리                                  │
    │                                                                                         │
    │  3. 리스트/배열:                                                                         │
    │     [1, 2, 3, 4]                                                                         │
    │     → 각 값을 별도 행으로 분해                                                              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [도메인 무결성 관점]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1NF은 "원자성"의 해석에 따라 완화 정도가 다름                                               │
    │                                                                                         │
    │  엄격한 1NF (Strict 1NF):                                                                  │
    │  • 도메인의 가장 작은 단위                                                               │
    │  │  예: "홍길동"은 원자값                                                                   │
    │  │  예: "서울 강남구"는 도시와 구로 분리 가능 → 비원자값                                    │
    │                                                                                         │
    │  현실적 1NF (Pragmatic 1NF):                                                              │
    │  • 비즈니스 로직에서 분해할 필요 없는 단위                                                   │
    │  │  예: 주소(도시+구+동)를 하나의 컬럼으로 유지                                             │
    │  │  예: 전화번호 (지역번호+번호)를 하나로 유지                                               │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 1NF 정규화 단계

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         1NF 변환 절차                                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 1: 반복 그룹 식별]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  ORDER 테이블                                                                              │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  order_id │ customer_id │ items (반복 그룹)                     │ total_amount  │  │
    │  │  ─────────────────────────────────────────────────────────────────────────────────  │  │
    │  │  1001      │ C001        │ {"P001", "P002"}                        │ 50000        │  │
    │  │  1002      │ C002        │ {"P001", "P003", "P004"}                │ 75000        │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                              │                                                          │
    │                              ▼                                                          │
    │  반복 그룹: items (여러 product_id)                                                      │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 2: 반복 그룹 제거 (테이블 분해)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  [ORDERS]                                                                                 │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  order_id │ customer_id │ total_amount                                              │  │
    │  │  1001      │ C001        │ 50000                                                     │  │
    │  │  1002      │ C002        │ 75000                                                     │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                              │                                                          │
    │                              │ FK                                                        │
    │                              ▼                                                          │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  [ORDER_ITEMS]                                                                          │  │
    │  │  ┌────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  │  order_id │ item_seq │ product_id │ quantity │ unit_price                         │  │  │
    │  │  │ 1001      │ 1         │ P001       │ 2        │ 20000                             │  │  │
    │  │  │ 1001      │ 2         │ P002       │ 1        │ 10000                             │  │  │
    │  │  │ 1002      │ 1         │ P001       │ 1        │ 20000                             │  │  │
    │  │  │ 1002      │ 2         │ P003       │ 3        │ 15000                             │  │  │
    │  │  │ 1002      │ 3         │ P004       │ 1        │ 40000                             │  │  │
    │  │  └────────────────────────────────────────────────────────────────────────────────┘  │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │
    │  PK: (order_id, item_seq)                                                               │
    │  FK: order_id → ORDERS(order_id)                                                         │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 3: 기본키 정의]

    • ORDERS: order_id (PK)
    • ORDER_ITEMS: (order_id, item_seq) (PK)
```

## Ⅲ. 융합 비교 및 다각도 분석

### 1NF 위패 유형

| 위반 유형 | 설명 | 해결 |
|----------|------|------|
| **반복 그룹** | 여러 값이 하나의 셀 | 별도 테이블 분해 |
| **중첩 속성** | 복합 객체 | 속성 분리 |
| **다치 데이터** | 중복 행 | 유일 식별자 추가 |

### 과목 융합 관점 분석

#### 1. NoSQL ↔ 1NF
- **MongoDB**: 중첩 문서 허용 (1NF 위반)
- **Cassandra**: Wide-column (1NF)
- **PostgreSQL**: JSONB (1NF 유연)

#### 2. 데이터 웨어하우스 ↔ 1NF
- **Star Schema**: 차원 테이블 1NF
- **Snowflake**: 더 깊은 정규화

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 로그 데이터 정규화
**상황**: JSON 형식 로그
**판단**:
1. **1NF 위반**: JSON 중첩 구조
2. **해결**:
   - SQL: JSONB 타입 사용 (PostgreSQL)
   - NoSQL: MongoDB 그대로 저장

```sql
-- PostgreSQL JSONB (1NF 유연화)
CREATE TABLE logs (
    id SERIAL PRIMARY KEY,
    timestamp TIMESTAMPTZ,
    level VARCHAR(10),
    message JSONB,
    metadata JSONB
);

-- JSON 내부 검색
SELECT * FROM logs
WHERE message->>'user_id' = '1001'
  AND level = 'ERROR';
```

## Ⅴ. 기대효과 및 결론

### 1NF 기대 효과

| 효과 | 비정규화 | 1NF 정규화 |
|------|---------|-----------|
| **검색** | 복잡 (LIKE, 파싱) | 단순 (WHERE) |
| **정렬** | 어려움 | 간단 (ORDER BY) |
| **조인** | 파싱 필요 | 직접 조인 |
| **무결성** | 위반 용이 | 제약조건 |

### 미래 전망

1. **Hybrid Storage**: Relational + Document
2. **Native JSON**: MySQL JSON, PostgreSQL JSONB
3. **Schema-on-Read**: NoSQL

- **Codd 1970**: Relational Model
- **Date 2003**: Database Design
