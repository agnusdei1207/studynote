# 2NF (제2정규형: Second Normal Form)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 제2정규형(2NF)은 **"1NF을 만족하고 **부분적 함수적 종속(Partial Functional Dependency)**을 제거한 정규형"**으로, **후보키(Primary Key)의 일부분에 종속되는 **비키 속성(Non-key Attribute)**를 **분리**하여 **독립 테이블로** 만든다.
> 2. **가치**: **데이터 중복(Data Redundancy)**을 줄이고 **삽입/삭제/갱신 이상(Anomaly)**을 제거하며, **개체-관계(Entity-Relationship)**을 **1:N 관계**로 명확히 정의하여 **데이터 무결성**을 강화한다.
> 3. **융합**: **ER 모델링(Entity-Relationship)**에서 **Strong Entity**와 **Weak Entity**를 구분하며, **复合 키(Composite Key)**를 **단일 테이블**로 분리하여 **M:N 관계**를 **1:N + 연결 테이블**로 변환한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
제2정규형(Second Normal Form, 2NF)은 **"1NF을 만족하고 부분적 함수적 종속을 제거한 릴레이션"**이다.

**2NF의 정의**:
- **제1정규형(1NF)을 만족**
- **후보키(Primary Key)가 아닌 속성이 후보키의 일부에 함수적 종속인 경우 분리

**부분적 종속(Partial Dependency)**:
- 기본키의 일부 속성에 종속
- A, B, C에서 (A, B) → C (C는 A와 B에 종속, A 단독으로 X)

### 💡 비유
2NF는 **"팀 구조"**와 같다.
- **문제**: 한 팀에 여러 역할
- **해결**: 각 역할을 별도 테이블로 분리

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         제2정규형의 발견                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

1NF 문제:
    • 반복 그룹 제거했지만 여전 중복 존재
    • 부분적 종속으로 인한 갱신 이상 발생
         ↓
Codd (1971년) 2NF 제안:
    • 부분적 종속 제거
    • 후보키에 완전 종속 보장
         ↓
ER 모델링 확장:
    • Strong Entity → 테이블
    • Weak Entity → 별도 테이블 + 식별자(FK)
    • M:N 관계 → 연결 테이블
```

## Ⅱ. 아키텍처 및 핵심 원리

### 2NF 위반 사례와 해결

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         2NF 위반: 부분적 종속                                        │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [비정규화 테이블: ORDER_ITEM]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  order_id │ product_id │ quantity │ unit_price │ total_price                           │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  1001     │ P001       │ 2         │ 20000      │ 40000                             │
    │  1001     │ P002       │ 1         │ 30000      │ 30000                             │
    │  1002     │ P001       │ 5         │ 20000      │ 100000                            │
    │  1002     │ P003       │ 1         │ 50000      │ 50000                             │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    PK: (order_id, product_id) → 복합 키

    [문제: 부분적 종속]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  unit_price → product_id (product_id에만 종속)                                            │
    │  • order_id 단독으로 unit_price 결정 불가                                                    │
    │  • (order_id, product_id)에서 product_id가 unit_price를 결정                                        │
    │                                                                                         │
    │  total_price = quantity × unit_price                                                          │
    │  • quantity × unit_price는 중복 계산 (저장 공간 낭비)                                        │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [2NF 해결: 분해]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  [PRODUCT]                                                                               │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  product_id (PK)                                                                     │  │
    │  │  unit_price                                                                       │  │
    │  │  ...                                                                                   │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                              │ FK                                                        │
    │                              ▼                                                        │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  [ORDER_ITEM]                                                                           │  │
    │  │  ┌────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  │  order_id (PK)                                                                    │  │
    │  │  │ product_id (FK)                                                                  │  │
    │  │  │ quantity                                                                         │  │  │
    │  │  └────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
    │                                                                                         │
    │  total_price는 VIEW 또는 애�리케이션 계산                                                    │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    • unit_price는 PRODUCT 테이블에서 중복 제거
    • quantity만 ORDER_ITEM에 저장
```

## Ⅲ. 융합 비교 및 다각도 분석

### 1NF → 2NF 변환

```
┌─────────────────────────────────────────────────────────────────────────────────────────────────┐
│                         1NF → 2NF 변환 절차                                              │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 1: 후보키 식별]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  기본키(Primary Key) 식별                                                                     │
    │  • 단일 키 또는 복합 키                                                                │
    │  │  • 각 속성의 함수적 종속 분석                                                           │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 2: 부분적 종속 식별]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  기본키의 진부분집합에 종속된 속성 식별                                               │
    │  • (A, B, C)에서 C가 A 또는 B에만 종속인 경우                                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 3: 테이블 분해]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 부분적 종속 속성을 별도 테이블로 분리                                                    │
    │  │  기본 테이블: 원래 테이블에서 해당 속성 제거                                               │
    │  │  종속 테이블: 분리된 속성 + 기본키 일부                                                      │
    │  • 외래키(Foreign Key)로 연결                                                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 2NF vs 3NF

| 구분 | 2NF | 3NF |
|------|-----|-----|
| **목적** | 부분 종속 제거 | 이행 종속 제거 |
| **종속 유형** | 기본키 부분 종속 | 비키 속성 간 종속 |
| **결과** | 완전 종속 보장 | 이행적 종속 제거 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 이커머스 쇼핑 정규화
**상황**: 장바구니(Item)에 카테고리(Category)와 공급(Supplier) 정보 중복
**판단**:

1. **1NF**: 단순 테이블 (모든 속성 포함)
2. **2NF**: 부분 종속 제거

```sql
-- 1NF (문제)
CREATE TABLE cart_items (
    cart_id INT,
    product_id INT,
    category_id INT,
    category_name VARCHAR(50),
    supplier_id INT,
    supplier_name VARCHAR(100),
    quantity INT,
    PRIMARY KEY (cart_id, product_id)
);

-- 2NF (해결)
-- Category 테이블
CREATE TABLE categories (
    category_id INT PRIMARY KEY,
    category_name VARCHAR(50)
);

-- Supplier 테이블
CREATE TABLE suppliers (
    supplier_id INT PRIMARY KEY,
    supplier_name VARCHAR(100)
);

-- Cart Items (2NF)
CREATE TABLE cart_items (
    cart_id INT,
    product_id INT,
    category_id INT,
    supplier_id INT,
    quantity INT,
    PRIMARY KEY (cart_id, product_id),
    FOREIGN KEY (category_id) REFERENCES categories(category_id),
    FOREIGN KEY (supplier_id) REFERENCES suppliers(supplier_id)
);
```

## Ⅴ. 기대효과 및 결론

### 2NF 기대 효과

| 효과 | 2NF 미달성 | 2NF 준수 |
|------|----------|----------|
| **중복** | category_name, supplier_name 중복 | 제거 |
| **갱신** | 단일 카테고리 변경 시 모든 행 수정 | Category 테이블만 수정 |
| **삽입** | 중복된 데이터 입력 필요 | Category만 추가 |
| **삭제** | 모든 cart_item에서 참조 무결성 위험 | 참조 정의로 안전 삭제 |
