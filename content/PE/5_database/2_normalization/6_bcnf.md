# BCNF (보이스-코드 정규형: Boyce-Codd Normal Form)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: BCNF는 **"3NF를 만족하고 **모든 결정자(Determinant)**가 **후보키(Candidate Key)**인 정규형"**으로, **3NF보다 더 강력한 제약조건**을 가지며 **다치 종속(Multivalued Dependency)** 문제를 해결하고 **데이터 중복**을 완전히 제거한다.
> 2. **가치**: **비키 속성이 결정자**인 경우 **3NF에서도 허용**되지만 BCNF는 **이를 제거**하여 **삽입/삭제/갱신 이상(Anomaly)**을 완전히 방지하고 **데이터 무결성**을 최적화한다.
> 3. **융합**: **ER 모델링(Entity-Relationship)**에서 **Strong Entity**와 **Weak Entity**를 명확히 구분하며, **M:N 관계**를 **연결 테이블**로 변환하여 **BCNF를 준수**하는 스키마를 설계한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
BCNF(Boyce-Codd Normal Form)는 **"3NF의 강력한 형태로, 모든 결정자가 후보키인 정규형"**이다.

**BCNF의 정의**:
- **제3정규형(3NF)을 만족**
- **모든 결정자(Determinant)가 후보키**
- **비키 속성이 결정자인 경우 불허**

**결정자(Determinant)**:
- A → B에서 A가 결정자
- BCNF에서는 A가 반드시 후보키여야 함

### 💡 비유
BCNF는 **"완벽한 조직도"**와 같다.
- **3NF**: 직장 상태까지 분리
- **BCNF**: 모든 결정 관계가 키로만

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         BCNF의 발견                                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

3NF 문제:
    • 이행 종속 제거했지만
    • 비키 속성이 결정자인 경우 존재
    • 삽입/갱신 이상 여전히 발생
         ↓
Boyce & Codd (1974년) BCNF 제안:
    • 모든 결정자가 후보키
    • 3NF보다 강력한 제약
         ↓
현대 데이터베이스:
    • BCNF가 표준 정규형
    • 대부분의 실무 스키마 목표
```

### 3NF vs BCNF

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         3NF vs BCNF 비교                                               │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [3NF 조건]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. 2NF를 만족                                                                            │
    │  2. 비키 속성이 기본키에 이행적으로 종속하지 않음                                         │
    │                                                                                         │
    │  허용:                                                                                   │
    │  • 비키 속성이 결정자여도 됨                                                             │
    │  • 예: A → B에서 B가 결정자여도 3NF 만족 가능                                            │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [BCNF 조건]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. 3NF를 만족                                                                            │
    │  2. 모든 결정자가 후보키                                                                  │
    │                                                                                         │
    │  불허:                                                                                   │
    │  • 비키 속성이 결정자인 경우                                                             │
    │  • 예: A → B에서 A가 후보키가 아니면 BCNF 위반                                            │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [비교 예시]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Relation: R(A, B, C)                                                                    │
    │  FD: A → B, B → C                                                                        │
    │                                                                                         │
    │  3NF:                                                                                    │
    │  • A가 후보키                                                                            │
    │  • B는 비키지만 결정자 (B → C)                                                           │
    │  • 3NF 만족 (C는 기본키 A에 이행적 종속)                                                   │
    │                                                                                         │
    │  BCNF:                                                                                   │
    │  • B가 결정자지만 후보키가 아님                                                           │
    │  • BCNF 위반                                                                            │
    │  • 분해 필요: R1(A, B), R2(B, C)                                                         │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### BCNF 분해 알고리즘

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         BCNF 분해 절차                                                  │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 1: 결정자 식별]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 모든 함수적 종속(FD) 식별                                                            │
    │  • 결정자(Determinant) 목록 작성                                                        │
    │  • 후보키(Candidate Key) 확인                                                           │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 2: BCNF 위반 확인]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 각 결정자가 후보키인지 확인                                                          │
    │  • 후보키가 아닌 결정자 있으면 BCNF 위반                                                 │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Step 3: 분해]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 위반하는 FD: X → A                                                                  │
    │  • X는 결정자지만 후보키가 아님                                                           │
    │  • 분해: R1(X, A), R2(R - A)                                                           │
    │  • 무손실 분해(Lossless Join) 보장                                                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [예시]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  원본: R(student_id, course_id, professor, grade)                                      │
    │  FD:                                                                                    │
    │  • (student_id, course_id) → grade                                                     │
    │  • course_id → professor                                                                │
    │                                                                                         │
    │  BCNF 위반: course_id는 결정자지만 후보키가 아님                                         │
    │                                                                                         │
    │  분해:                                                                                   │
    │  • R1(student_id, course_id, grade)                                                     │
    │  • R2(course_id, professor)                                                             │
    │                                                                                         │
    │  결과: 모든 결정자가 후보키                                                              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 정규형 계층 구조

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         정규형 계층                                                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    1NF (제1정규형)
         ↓  반복 그룹 제거
    2NF (제2정규형)
         ↓  부분 종속 제거
    3NF (제3정규형)
         ↓  이행 종속 제거
    BCNF (보이스-코드 정규형)
         ↓  모든 결정자가 후보키
    4NF (제4정규형)
         ↓  다치 종속 제거
    5NF (제5정규형)
         ↓  조인 종속 제거
```

### BCNF vs 4NF

| 구분 | BCNF | 4NF |
|------|------|-----|
| **목적** | 결정자 제약 | 다치 종속 제거 |
| **종속 유형** | 함수적 종속 | 다치 종속 |
| **실무 적용** | 널리 사용 | 드묾 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 강의 배정 시스템
**상황**: 학생, 과정, 교수, 교재
**판단**:

1. **3NF 분석**:
   - 이행 종속 확인
2. **BCNF 분해**:
   - 결정자가 후보키 확인

```sql
-- BCNF 미준수 (문제)
CREATE TABLE course_assignment (
    student_id INT,
    course_id INT,
    professor VARCHAR(50),
    textbook VARCHAR(100),
    grade CHAR(1),
    PRIMARY KEY (student_id, course_id)
);

-- BCNF 준수 (해결)
-- 학생-과정
CREATE TABLE student_course (
    student_id INT,
    course_id INT,
    grade CHAR(1),
    PRIMARY KEY (student_id, course_id)
);

-- 과정-교수-교재
CREATE TABLE course_info (
    course_id INT PRIMARY KEY,
    professor_id INT,
    textbook_id INT,
    FOREIGN KEY (professor_id) REFERENCES professors(professor_id),
    FOREIGN KEY (textbook_id) REFERENCES textbooks(textbook_id)
);
```
