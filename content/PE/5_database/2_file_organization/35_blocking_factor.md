# 블로킹 팩터 (Blocking Factor)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 블로킹 팩터는 **"하나의 **물리적 **블록**에 **저장**되는 **논리적 **레코드**의 **개수\\\"**로, **Blocking Factor = Block Size / Record Size** **공식**으로 **계산**하고 **I/O **성능**에 **직접적인 **영향**을 **미친다**.
> 2. **성능**: **블로킹 **팩터**가 **클수록 **한 번의 **I/O로 **더 **많은 **레코드**를 **읽을 **수 **있어 **순차 **접근 **성능**이 **향상**되지만 **레코드** 간 **검색**에는 **불리**할 **수 **있다**.
> 3. **최적화**: **레코드 **크기**, **블록 **크기**, **액세스 **패턴**을 **고려**하여 **블로킹 **팩터**를 **결정**하며 **현대 **DBMS**는 **페이지** **크기**(4KB, 8KB, 16KB)를 **기준**으로 **자동 **최적화**한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
블로킹 팩터는 **"블록 당 레코드 수"**이다.

**파일 구조 기본 용어**:
| 용어 | 설명 |典型 값 |
|------|------|--------|
| **필드 (Field)** | 최소 데이터 단위 | 가변 |
| **레코드 (Record)** | 필드의 집합 | 100-1000B |
| **블록 (Block)** | I/O 단위 | 4KB-16KB |
| **파일 (File)** | 레코드의 집합 | 가변 |
| **블로킹 팩터** | 블록 당 레코드 수 | 4-100 |

### 💡 비유
블로킹 팩터는 **책장에 꽂는 책 권수**와 같다.
- **블록**: 책장 한 칸
- **레코드**: 책 한 권
- **블로킹 팩터**: 한 칸에 꽂을 수 있는 책 권수

## Ⅱ. 아키텍처 및 핵심 원리

### 블로킹 팩터 계산

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Blocking Factor Calculation                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Basic Formula:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Blocking Factor (BF) = Block Size (B) / Record Size (R)                                │  │
    │                                                                                         │  │
    │  Example:                                                                               │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Block Size = 4096 bytes (4KB)                                                      │  │  │
    │  │  Record Size = 200 bytes                                                            │  │  │
    │  │  BF = 4096 / 200 = 20.48 → 20 records (integer division)                            │  │  │
    │  │                                                                                      │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐ │  │  │
    │  │  │ Block (4096 bytes)                                                             │ │  │  │
    │  │  │ ┌─────┐ ┌─────┐ ┌─────┐ ... ┌─────┐                                           │ │  │  │
    │  │  │ │ R1  │ │ R2  │ │ R3  │     │ R20  │  ← 20 records                           │ │  │  │
    │  │  │ │200B │ │200B │ │200B │     │200B  │                                           │ │  │  │
    │  │  │ └─────┘ └─────┘ └─────┘     └─────┘                                           │ │  │  │
    │  │  │ Remaining: 96 bytes (internal fragmentation)                                    │ │  │  │
    │  │  │ ┌──────────────────────────────┐ ← Unused space                                │ │  │  │
    │  │  │ └──────────────────────────────┘                                               │ │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘ │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 고려 사항

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Blocking Factor Design Considerations                             │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    1. Internal Fragmentation (내부 단편화):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Problem: Unused space at end of block                                                │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Block Size = 4096, Record Size = 300                                               │  │  │
    │  │  BF = 4096 / 300 = 13.65 → 13 records                                             │  │  │
    │  │  Used = 13 × 300 = 3900 bytes                                                      │  │  │
    │  │  Wasted = 4096 - 3900 = 196 bytes (4.8% waste)                                      │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Mitigation:                                                                            │  │
    │  • Choose block size divisible by common record sizes                                    │  │
    │  • Use variable-length records (with overhead)                                           │  │
    │  • Spanned records (record spans multiple blocks)                                       │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    2. Access Pattern (액세스 패턴):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Sequential Access (순차 접근):                                                          │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  High BF = Better (read many records per I/O)                                       │  │  │
    │  │  Example: Full table scan, batch processing                                          │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Random Access (임의 접근):                                                             │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Low BF may be acceptable (only need one record)                                    │  │  │
    │  │  Example: Primary key lookup, OLTP                                                  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Mixed Access (혼합):                                                                  │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Balance between sequential and random                                              │  │  │
    │  │  May use multiple blocking factors for different access patterns                   │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    3. Spanned vs Unspanned Records:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Unspanned (비스팬):                                                                     │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Record fits entirely in one block                                                  │  │  │
    │  │  Simple to implement                                                                  │  │  │
    │  │  Maximum record size ≤ block size                                                    │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Spanned (스팬):                                                                          │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Record can span multiple blocks                                                      │  │  │
    │  │  Allows larger records than block size                                               │  │  │
    │  │  Complex pointer management                                                           │  │  │
    │  │                                                                                      │  │  │
    │  │  Example:                                                                             │  │  │
    │  │  Block N:   ┌───────────────┐ ┌────────┐                                             │  │  │
    │  │            │ Record Part1 │ │Pointer  │ → Block N+1                                  │  │  │
    │  │            └───────────────┘ └────────┘                                             │  │  │
    │  │  Block N+1: ┌───────────────┐                                                       │  │  │
    │  │            │ Record Part2 │                                                       │  │  │
    │  │            └───────────────┘                                                       │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### I/O 성능 분석

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         I/O Performance vs Blocking Factor                                  │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Sequential Access Performance:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Scenario: Read 10,000 records                                                          │  │
    │  Record Size = 200 bytes                                                                │  │
    │  Block Size = 4096 bytes                                                                 │  │
    │                                                                                         │  │
    │  BF = 10:                                                                                │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Records per block = 10                                                              │  │  │
    │  │  Blocks needed = 10,000 / 10 = 1000 blocks                                          │  │  │
    │  │  I/O count = 1000                                                                     │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  BF = 20:                                                                                │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Records per block = 20                                                              │  │  │
    │  │  Blocks needed = 10,000 / 20 = 500 blocks                                           │  │  │
    │  │  I/O count = 500 (50% reduction)                                                     │  │  │
    │  │  Time saved = 500 × Disk Latency (~5ms) = 2.5 seconds                               │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Random Access Performance:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Scenario: Read 100 random records                                                       │  │
    │                                                                                         │  │
    │  Regardless of BF:                                                                       │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  I/O count = 100 (one block per record)                                             │  │  │
    │  │  Higher BF doesn't help (still read block)                                          │  │  │
    │  │  May even hurt (larger block = longer transfer time)                                │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Exception: Clustering (hot records in same block)                                      │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  If multiple target records in same block → Fewer I/Os                             │  │  │
    │  │  Requires spatial locality in access pattern                                        │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 블록 크기 vs 블로킹 팩터

| 블록 크기 | 레코드 크기 | BF | 장점 | 단점 |
|----------|-------------|----|----|----|
| **4KB** | 200B | 20 | 적은 낭비 | 많은 I/O |
| **8KB** | 200B | 40 | 균형 | 메모리 사용 |
| **16KB** | 200B | 80 | 적은 I/O | 낭비 증가 |
| **64KB** | 200B | 320 | 대량 처리 | 단편화 심각 |

### 파일 조직 방식별 BF

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Blocking Factor by File Organization                                │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    1. Sequential (Heap) Files:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Fixed-length records: Simple BF calculation                                         │  │
    │  • Variable-length: Spanned records, complex BF                                        │  │
    │  • Maximize BF for sequential scans                                                    │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    2. Indexed Files (ISAM, B-Tree):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Data blocks: Higher BF for range queries                                             │  │
    │  • Index blocks: Smaller BF (more index entries per block)                              │  │
    │  • Trade-off: Index fanout vs tree depth                                               │  │
    │                                                                                         │  │
    │  B-Tree Node Structure:                                                                 │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  ┌─────────┬─────────┬─────────┬─────┐                                             │  │  │
    │  │  │  Key1   │  Key2   │  Key3   │ ... │  ← More keys = Higher fanout                │  │  │
    │  │  ├─────────┼─────────┼─────────┼─────┤                                             │  │  │
    │  │  │ Ptr1    │ Ptr2    │ Ptr3    │ ... │                                             │  │  │
    │  │  └─────────┴─────────┴─────────┴─────┘                                             │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    3. Hash Files:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • BF based on bucket size                                                              │  │
    │  • Overflow chains may reduce effective BF                                               │  │
    │  • Aim for BF < 1 to avoid overflows                                                    │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: DW 스타(schema) 설계
**상황**: 데이터 웨하우스 팩트 테이블
**판단**: 블록 크기, BF 결정

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Data Warehouse Block Size Decision                                 │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Requirements:
    • Large table scans (full table scans common)
    • Star schema joins
    • Column-oriented queries

    Analysis:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  OLTP Block Size (4KB):                                                                  │  │
    │  • Optimized for single-row lookups                                                      │  │
    │  • Minimizes wasted space for random access                                              │  │
    │  • Poor for scans (many I/Os)                                                            │  │
    │                                                                                         │  │
    │  DW Block Size (32KB):                                                                   │  │
    │  • Optimized for sequential scans                                                        │  │
    │  • Fewer I/Os for full table scan                                                        │  │
    │  • Better compression (larger blocks compress better)                                    │  │
    │                                                                                         │  │
    │  Example: 1 billion row table, 1KB per row                                               │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  4KB blocks:  1B / 4 = 250M blocks → 250K I/Os (assuming 0.1% selectivity)          │  │  │
    │  │  32KB blocks: 1B / 32 = 31M blocks → 31K I/Os (8x reduction)                        │  │  │
    │  │  Time saved: 219K × 5ms = 1095 seconds (~18 minutes) per scan                       │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Decision: Use 32KB blocks for fact tables, 8KB for dimension tables                    │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 블로킹 팩터 기대 효과

| 워크로드 | 최적 BF | 이유 |
|----------|---------|------|
| **OLTP** | 낮은 BF | 랜덤 접근 |
| **DW** | 높은 BF | 순차 스캔 |
| **Batch** | 최대 BF | 대량 처리 |
| **Mixed** | 중간 BF | 균형 |

### 모범 사례

1. **압축**: 유효 BF 증가
2. **파티셔닝**: 로컬리티
3. **컬럼nar**: BF 무관
4. **모니터링**: I/O 분석

### 미래 전망

1. **NVMe**: 블록 크기 증가
2. **컬럼nar**: 페이지 형태 변경
3. **압축**: 자동 최적화
4. **SSD**: 순차/랜덤 차이 감소

### ※ 참고 표준/가이드
- **ANSI**: SQL 표준
- **ISO**: 데이터베이스 표준
- **Oracle**: 튜닝 가이드
- **MySQL**: InnoDB 아키텍처
