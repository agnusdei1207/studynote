# 데이터 독립성 (Data Independence)

## 정의

**데이터 독립성**은 데이터베이스 시스템에서 **하위 수준의 변화가 상위 수준에 영향을 주지 않는 능력**이다. ANSI/SPARC 3단계 아키텍처에서 정의된 핵심 개념으로, **논리적 독립성**과 **물리적 독립성**으로 구분된다.

```
┌─────────────────────────────────────────────────────────────────┐
│               ANSI/SPARC 3단계 아키텍처                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│    [외부 스키마] ── 외부/개념 매핑 ──▶ [개념 스키마]            │
│         ↑                                    ↑                  │
│         │                                    │                  │
│    ◀ 논리적 독립성 ▶                    ◀ 물리적 독립성 ▶       │
│         │                                    │                  │
│         ↓                                    ↓                  │
│    [응용 프로그램] ────────────────▶ [내부 스키마]             │
│                                                                 │
│    외부 스키마 변경 → 개념 스키마 영향 X  (논리적 독립성)       │
│    내부 스키마 변경 → 개념/외부 스키마 영향 X (물리적 독립성)   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 핵심 개념

### 1. 논리적 데이터 독립성

```
┌─────────────────────────────────────────────────────────────────┐
│                    논리적 데이터 독립성                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  정의: 개념 스키마(전체 논리 구조) 변경 시 외부 스키마(뷰)와     │
│        응용 프로그램이 영향받지 않는 특성                        │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │           [개념 스키마]                                  │   │
│  │    employees(id, name, dept, salary, hire_date)         │   │
│  │            ↓ 분리                                       │   │
│  │    employees(id, name, dept)                            │   │
│  │    salaries(emp_id, salary, hire_date)  ← 새 테이블     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                         │                                      │
│            ◀ 논리적 독립성 (매핑만 변경) ▶                      │
│                         │                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │           [외부 스키마 - 뷰]                             │   │
│  │    v_employees: CREATE VIEW AS                          │   │
│  │        SELECT id, name FROM employees                   │   │
│  │    → 변경 없이 동작!                                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  실현 방법: View 재정의, 매핑 정보 변경                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**논리적 독립성 예시**
- 컬럼 추가: 기존 뷰는 영향 없음
- 테이블 분할: 뷰로 JOIN 결과 제공
- 테이블 통합: 뷰로 분리된 형태 제공

### 2. 물리적 데이터 독립성

```
┌─────────────────────────────────────────────────────────────────┐
│                    물리적 데이터 독립성                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  정의: 내부 스키마(물리적 저장 구조) 변경 시 개념/외부 스키마와  │
│        응용 프로그램이 영향받지 않는 특성                        │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │           [내부 스키마 변경 예시]                        │   │
│  │                                                          │   │
│  │  ① 인덱스 추가/변경                                      │   │
│  │     CREATE INDEX idx_name ON employees(name);            │   │
│  │     → 응용 프로그램 코드 변경 없음                       │   │
│  │                                                          │   │
│  │  ② 저장 위치 변경                                        │   │
│  │     테이블스페이스 이동, 디스크 마이그레이션             │   │
│  │     → SQL 문은 그대로                                    │   │
│  │                                                          │   │
│  │  ③ 저장 방식 변경                                        │   │
│  │     Heap → B-Tree → Columnar                            │   │
│  │     → 논리적 구조는 동일                                 │   │
│  │                                                          │   │
│  │  ④ 압축/암호화 적용                                      │   │
│  │     Transparent Data Encryption (TDE)                    │   │
│  │     → 응용 프로그램 투명                                 │   │
│  │                                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                         │                                      │
│            ◀ 물리적 독립성 (매핑만 변경) ▶                      │
│                         │                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │           [개념 스키마]                                  │   │
│  │    SELECT * FROM employees WHERE name = 'Kim';          │   │
│  │    → 내부 변경과 무관하게 동일한 SQL 사용               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 3단계 스키마와 매핑

```
┌─────────────────────────────────────────────────────────────────┐
│              3단계 스키마 구조와 매핑                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  외부 스키마 (External Schema)                            │  │
│  │  ┌───────────┐ ┌───────────┐ ┌───────────┐              │  │
│  │  │ View A    │ │ View B    │ │ View C    │              │  │
│  │  │ (인사팀)  │ │ (영업팀)  │ │ (재무팀)  │              │  │
│  │  └─────┬─────┘ └─────┬─────┘ └─────┬─────┘              │  │
│  │        │             │             │                     │  │
│  │        └─────────────┼─────────────┘                     │  │
│  │                      │                                   │  │
│  │    외부/개념 매핑 (E/C Mapping)                         │  │
│  │    - 뷰 정의                                            │  │
│  │    - 논리적 독립성 보장                                  │  │
│  │                      │                                   │  │
│  └──────────────────────┼───────────────────────────────────┘  │
│                         ▼                                      │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  개념 스키마 (Conceptual Schema)                         │  │
│  │  ┌─────────────────────────────────────────────────────┐│  │
│  │  │  전체 논리 구조 (통합 뷰)                           ││  │
│  │  │  - 모든 엔티티, 속성, 관계                         ││  │
│  │  │  - 무결성 제약조건                                ││  │
│  │  │  - 보안 제약조건                                  ││  │
│  │  └─────────────────────────────────────────────────────┘│  │
│  │                      │                                   │  │
│  │    개념/내부 매핑 (C/I Mapping)                         │  │
│  │    - 저장 구조 정의                                     │  │
│  │    - 물리적 독립성 보장                                  │  │
│  │                      │                                   │  │
│  └──────────────────────┼───────────────────────────────────┘  │
│                         ▼                                      │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  내부 스키마 (Internal Schema)                           │  │
│  │  ┌─────────────────────────────────────────────────────┐│  │
│  │  │  물리적 저장 구조                                   ││  │
│  │  │  - 데이터 파일, 인덱스 파일                        ││  │
│  │  │  - 레코드 형식, 순서                               ││  │
│  │  │  - 저장 경로, 압축 방식                            ││  │
│  │  └─────────────────────────────────────────────────────┘│  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 상세 설명

### 데이터 독립성 비교

```
┌─────────────────────────────────────────────────────────────────┐
│             논리적 독립성 vs 물리적 독립성                      │
├───────────────┬─────────────────────┬───────────────────────────┤
│     구분      │    논리적 독립성     │     물리적 독립성        │
├───────────────┼─────────────────────┼───────────────────────────┤
│ 정의          │ 개념 스키마 변경이   │ 내부 스키마 변경이       │
│               │ 외부 스키마에 영향 X │ 상위 스키마에 영향 X     │
├───────────────┼─────────────────────┼───────────────────────────┤
│ 변경 대상     │ 논리적 구조 (테이블,│ 물리적 구조 (파일,       │
│               │ 컬럼, 관계)         │ 인덱스, 저장 위치)       │
├───────────────┼─────────────────────┼───────────────────────────┤
│ 구현 수단     │ 외부/개념 매핑      │ 개념/내부 매핑           │
├───────────────┼─────────────────────┼───────────────────────────┤
│ 영향 범위     │ 외부 스키마,        │ 개념 스키마,             │
│               │ 응용 프로그램       │ 외부 스키마              │
├───────────────┼─────────────────────┼───────────────────────────┤
│ 난이도        │ 상대적으로 어려움   │ 상대적으로 쉬움          │
├───────────────┼─────────────────────┼───────────────────────────┤
│ 예시          │ 테이블 정규화,      │ 인덱스 생성,             │
│               │ 컬럼 추가/삭제      │ 파티셔닝, 이관           │
└───────────────┴─────────────────────┴───────────────────────────┘
```

### 데이터 독립성의 이점

1. **유지보수성 향상**: 하부 구조 변경이 응용에 영향 없음
2. **생산성 증대**: 개발자는 논리 구조만 집중
3. **비용 절감**: 물리적 튜닝을 독립적으로 수행
4. **확장성 보장**: 시스템 성능 향상을 위한 구조 변경 용이

## 실무 적용

### 물리적 독립성 활용 사례

```
┌─────────────────────────────────────────────────────────────────┐
│              물리적 튜닝 (응용 프로그램 영향 없음)              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ① 인덱스 튜닝                                                 │
│     ┌───────────────────────────────────────────────────────┐  │
│     │ -- 성능 저하 감지                                     │  │
│     │ EXPLAIN SELECT * FROM orders WHERE customer_id = 100; │  │
│     │ -- 인덱스 추가                                        │  │
│     │ CREATE INDEX idx_customer ON orders(customer_id);     │  │
│     │ → 응용 코드 변경 없이 성능 향상                       │  │
│     └───────────────────────────────────────────────────────┘  │
│                                                                 │
│  ② 파티셔닝                                                    │
│     ┌───────────────────────────────────────────────────────┐  │
│     │ -- 대용량 테이블 파티셔닝                             │  │
│     │ ALTER TABLE logs PARTITION BY RANGE (YEAR(created));  │  │
│     │ → SELECT 문 변경 없이 관리 용이                       │  │
│     └───────────────────────────────────────────────────────┘  │
│                                                                 │
│  ③ 데이터 이관                                                 │
│     ┌───────────────────────────────────────────────────────┐  │
│     │ -- 스토리지 마이그레이션                              │  │
│     │ MOVE TABLESPACE users TO ssd_storage;                 │  │
│     │ → 논리적 접근 방식 동일                               │  │
│     └───────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 논리적 독립성 활용 사례

```
┌─────────────────────────────────────────────────────────────────┐
│              스키마 진화 (응용 프로그램 보호)                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Before: 단일 테이블                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ CREATE TABLE users (                                    │   │
│  │   id INT, name VARCHAR(100),                            │   │
│  │   email VARCHAR(100), phone VARCHAR(20)                 │   │
│  │ );                                                      │   │
│  │                                                         │   │
│  │ View: v_user_contact AS SELECT id, email FROM users;   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                         │                                      │
│                         ▼                                      │
│  After: 테이블 분리 (개인정보 분리)                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ CREATE TABLE user_basic (id INT, name VARCHAR(100));    │   │
│  │                                                         │   │
│  │ CREATE TABLE user_contact (                            │   │
│  │   user_id INT, email VARCHAR(100), phone VARCHAR(20)   │   │
│  │ );                                                      │   │
│  │                                                         │   │
│  │ View: v_user_contact AS                                │   │
│  │   SELECT b.id, c.email                                 │   │
│  │   FROM user_basic b JOIN user_contact c ON b.id=c.id;  │   │
│  │                                                         │   │
│  │ → 기존 응용은 View 통해 동일하게 접근                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 기술사 시험 포인트

### 1. 핵심 정의 (반드시 암기)

> **논리적 데이터 독립성**: 개념 스키마 변경이 외부 스키마에 영향을 주지 않음
> **물리적 데이터 독립성**: 내부 스키마 변경이 상위 스키마에 영향을 주지 않음

### 2. 빈출 키워드

- **ANSI/SPARC 3단계**: 외부, 개념, 내부 스키마
- **매핑**: 외부/개념(E/C), 개념/내부(C/I)
- **논리적 독립성**: 테이블 구조 변경 시 뷰로 보호
- **물리적 독립성**: 인덱스, 파티셔닝, 저장 구조 변경

### 3. 예상 문제

**Q1. 데이터 독립성의 개념과 종류에 대해 설명하시오.**

**Q3. 물리적 데이터 독립성의 실무 활용 예시 3가지를 들어 설명하시오.**
