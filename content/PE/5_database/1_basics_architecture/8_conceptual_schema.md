# 개념 스키마 (Conceptual Schema)

## 정의

**개념 스키마(Conceptual Schema)**는 데이터베이스의 **전체 논리적 구조**를 정의한 것으로, 조직 전체 관점에서 모든 사용자가 보는 데이터베이스의 구조를 기술한다. ANSI/SPARC 3단계 스키마 아키텍처의 중간 계층에 해당한다.

```
┌─────────────────────────────────────────────────────────────────┐
│           3단계 스키마 아키텍처에서의 위치                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   외부 스키마 (External Schema)                          │ │
│   │   • 사용자/응용프로그램 관점                             │ │
│   │   • 복수 개 존재 가능                                    │ │
│   │                        │                                │ │
│   │           외부/개념 매핑                                 │ │
│   │                        ▼                                │ │
│   │   ┌─────────────────────────────────────────────────┐  │ │
│   │   │      개념 스키마 (Conceptual Schema)            │  │ │
│   │   │      ★ 전체 논리적 구조                        │  │ │
│   │   │      ★ 단 하나만 존재                          │  │ │
│   │   │      ★ 물리적 구현 독립                        │  │ │
│   │   └─────────────────────────────────────────────────┘  │ │
│   │                        │                                │ │
│   │           개념/내부 매핑                                 │ │
│   │                        ▼                                │ │
│   │   내부 스키마 (Internal Schema)                          │ │
│   │   • 물리적 저장 구조                                     │ │
│   │   • 인덱스, 저장 방식 등                                 │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 핵심 개념

### 1. 개념 스키마의 특성

```
┌─────────────────────────────────────────────────────────────────┐
│           개념 스키마 핵심 특성                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   ① 단일성 (Singularity)                                │ │
│   │      • 데이터베이스당 오직 하나의 개념 스키마            │ │
│   │      • 전체 조직의 통합된 데이터 뷰                      │ │
│   │                                                         │ │
│   │   ② 논리적 독립성 (Logical Independence)                 │ │
│   │      • 물리적 저장 방식과 무관한 구조 정의               │ │
│   │      • 저장 구조 변경 시 개념 스키마 영향 없음           │ │
│   │                                                         │ │
│   │   ③ 완전성 (Completeness)                               │ │
│   │      • 조직의 모든 데이터 포함                          │ │
│   │      • 모든 엔티티와 관계 정의                          │ │
│   │                                                         │ │
│   │   ④ 중립성 (Neutrality)                                 │ │
│   │      • 특정 사용자나 응용에 치우치지 않음                │ │
│   │      • 범용적 관점에서 정의                              │ │
│   │                                                         │ │
│   │   ⑤ 무결성 제약 (Integrity Constraints)                 │ │
│   │      • 데이터 무결성 규칙 포함                          │ │
│   │      • 보안/접근 제어 명세                              │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2. 개념 스키마 구성 요소

```
┌─────────────────────────────────────────────────────────────────┐
│           개념 스키마 구성 요소                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   구조적 요소 (Structural Elements):                     │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │ • 엔티티 (Entity): 저장 대상 객체               │    │ │
│   │   │ • 속성 (Attribute): 엔티티의 특성              │    │ │
│   │   │ • 관계 (Relationship): 엔티티 간 연관          │    │ │
│   │   │ • 식별자 (Identifier): 엔티티 구분 키          │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                                                         │ │
│   │   제약적 요소 (Constraint Elements):                     │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │ • 무결성 제약조건                              │    │ │
│   │   │   - 개체 무결성 (PK NOT NULL, UNIQUE)          │    │ │
│   │   │   - 참조 무결성 (FK 제약)                      │    │ │
│   │   │   - 도메인 무결성 (데이터 타입, 범위)          │    │ │
│   │   │ • 보안 제약 (접근 권한)                        │    │ │
│   │   │ • 비즈니스 규칙 (CHECK 제약 등)                │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                                                         │ │
│   │   운영적 요소 (Operational Elements):                    │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │ • 연산 (Operations): CRUD 동작 정의            │    │ │
│   │   │ • 뷰 (Views): 가상 테이블 정의                 │    │ │
│   │   │ • 권한 (Privileges): 사용자별 접근 권한        │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 개념 스키마 표현 방법

```
┌─────────────────────────────────────────────────────────────────┐
│           개념 스키마 표현 방법                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ① ER 다이어그램 (Entity-Relationship Diagram):                │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   ┌─────────┐           ┌─────────┐                    │ │
│   │   │ 학  생  │───────────│ 수  강  │                    │ │
│   │   └─────────┘  수강한다  └─────────┘                    │ │
│   │       │                        │                        │ │
│   │       │ 1                      N │                      │ │
│   │       ▼                        ▼                        │ │
│   │   ┌─────────────────────────────────┐                  │ │
│   │   │           과  목                │                  │ │
│   │   └─────────────────────────────────┘                  │ │
│   │                                                         │ │
│   │   표기법: 피터 첸(Peter Chen), IE(까마귀발), IDEF1X     │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   ② DDL (Data Definition Language):                            │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   CREATE TABLE 학생 (                                   │ │
│   │       학번    CHAR(10) PRIMARY KEY,                     │ │
│   │       이름    VARCHAR(20) NOT NULL,                     │ │
│   │       학과코드 CHAR(5),                                 │ │
│   │       FOREIGN KEY (학과코드) REFERENCES 학과(학과코드)  │ │
│   │   );                                                    │ │
│   │                                                         │ │
│   │   CREATE TABLE 수강 (                                   │ │
│   │       학번    CHAR(10),                                 │ │
│   │       과목코드 CHAR(10),                                │ │
│   │       PRIMARY KEY (학번, 과목코드),                     │ │
│   │       FOREIGN KEY (학번) REFERENCES 학생(학번),         │ │
│   │       FOREIGN KEY (과목코드) REFERENCES 과목(과목코드)  │ │
│   │   );                                                    │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   ③ 관계형 모델 (Relational Model):                             │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   학생 (학번, 이름, 학과코드)                           │ │
│   │       PK: 학번                                          │ │
│   │       FK: 학과코드 → 학과.학과코드                      │ │
│   │                                                         │ │
│   │   수강 (학번, 과목코드, 성적)                           │ │
│   │       PK: (학번, 과목코드)                              │ │
│   │       FK: 학번 → 학생.학번                              │ │
│   │       FK: 과목코드 → 과목.과목코드                      │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4. 논리적 데이터 독립성

```
┌─────────────────────────────────────────────────────────────────┐
│           논리적 데이터 독립성                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   정의: 개념 스키마가 변경되어도 외부 스키마(응용프로그램)가      │
│         영향을 받지 않는 특성                                    │
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   개념 스키마 변경 예시:                                 │ │
│   │                                                         │ │
│   │   변경 전: 학생(학번, 이름, 학과)                        │ │
│   │   변경 후: 학생(학번, 이름, 학과코드)  ← 학과를 코드화   │ │
│   │            학과(학과코드, 학과명)                        │ │
│   │                                                         │ │
│   │   외부 스키마 영향:                                      │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │ 기존 뷰: CREATE VIEW 학생정보 AS              │    │ │
│   │   │          SELECT 학번, 이름, 학과 FROM 학생    │    │ │
│   │   │                                               │    │ │
│   │   │ → 외부/개념 매핑 변경으로 외부 스키마 유지    │    │ │
│   │   │                                               │    │ │
│   │   │ 수정된 뷰: CREATE VIEW 학생정보 AS            │    │ │
│   │   │   SELECT s.학번, s.이름, d.학과명             │    │ │
│   │   │   FROM 학생 s JOIN 학과 d                     │    │ │
│   │   │   ON s.학과코드 = d.학과코드                  │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                                                         │ │
│   │   장점:                                                  │ │
│   │   • 응용프로그램 수정 없이 DB 구조 변경 가능            │ │
│   │   • DB 확장/축소 용이                                   │ │
│   │   • 유지보수 비용 절감                                  │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 기술사 시험 포인트

### 1. 핵심 정의

> **개념 스키마**: 데이터베이스의 전체 논리적 구조, DB당 1개만 존재
> **논리적 독립성**: 개념 스키마 변경 시 외부 스키마 영향 없음
> **외부/개념 매핑**: 외부 스키마와 개념 스키마 간 대응 관계

### 2. 빈출 키워드

- **단일성**: DB당 하나의 개념 스키마
- **논리적 독립성**: 개념-외부 스키마 분리
- **ER 다이어그램**: 개념 스키마 시각화 도구
- **DDL**: 개념 스키마 정의 언어

### 3. 예상 문제

**Q1. 개념 스키마의 특성과 역할을 설명하시오.**

**Q2. 논리적 데이터 독립성의 개념과 개념 스키마와의 관계를 기술하시오.**

**Q3. ANSI/SPARC 3단계 스키마 아키텍처에서 개념 스키마의 위치와 역할을 설명하시오.**
