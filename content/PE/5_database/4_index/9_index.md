# 인덱스 (Index)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인덱스는 **"데이터 검색 속도**를 높이기 위해 **테이블의 특정 컬럼(들)**을 **정렬된 구조**로 저장하는 **데이터베이스 오브젝트"**로, **B-Tree**(RDBMS 표준), **Hash Index**(Equality 검색), **Full-Text Search**(문서 검색) 등의 **알고리즘**으로 구현되며 **쓰기 오버헤드**와 **저장 공간**을 **희생**하여 **읽기 성능**을 향상한다.
> 2. **가치**: **SELECT 쿼리의 성능**을 **수십 배~수백 배** 향상시키고 **Sorting**, **Grouping**, **Join 연산**의 효율을 높이며 **Unique Index**는 **중복 방지 제약**과 **검색 성능**을 동시에 제공한다.
> 3. **융합**: **Clustered Index**는 **데이터 자체를 정렬**하므로 **1개 테이블에 1개**만 존재하고 **Non-Clustered Index**는 **별도 저장**되며 **Covering Index**는 **테이블 접근 없이** 인덱스만으로 **쿼리를 완료**할 수 있게 한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
인덱스는 **"데이터의 물리적 위치를 빠르게 찾기 위한 자료 구조"**이다.

**인덱스의 기본 원리**:
- **Redundancy**: 데이터 복사 저장
- **Sorting**: 정렬된 순서 유지
- **Lookup**: 빠른 검색

### 💡 비유
인덱스는 **"책의 찾아보기(Index)"**와 같다.
- **책**: 테이블
- **찾아보기**: 인덱스
- **페이지 번호**: 데이터 위치

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         인덱스의 필요성                                                  │
└─────────────────────────────────────────────────────────────────────────────────────────┘

Full Table Scan:
    • 모든 페이지 스캔
    • O(n) 시간 복잡도
         ↓
Index 도입:
    • 정렬된 구조
    • O(log n) 또는 O(1)
         ↓
현대 DB:
    • B-Tree (MySQL InnoDB, PostgreSQL)
    • Hash Index (Memory Engine)
    • Full-Text (Elasticsearch)
```

## Ⅱ. 아키텍처 및 핵심 원리

### B-Tree Index 구조

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         B-Tree 구조                                                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  [B+Tree Example: Height = 3]                                                             │
    │                                                                                         │
    │                              Root Node (Internal)                                         │
    │                    ┌─────────────50─────────────┐                                         │
    │                    │                             │                                         │
    │          ┌─────────┴──────────┐        ┌──────────┴─────────┐                              │
    │          ▼                      ▼        ▼                      ▼                              │
    │    Internal Nodes              ...    Internal Nodes             ...                          │
    │     ┌─────30────┐              ...     ┌─────70─────┐          ...                          │
    │     │           │                     │             │                                        │
    │  ┌───┴───┐ ┌────┴───┐          ...  ┌───┴───┐ ┌────┴───┐       ...                             │
    │  ▼       ▼ ▼       ▼                 ▼       ▼ ▼       ▼                                        │
    │  Leaf Nodes (Linked List for Range Scan)                                             │
    │  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐                               │
    │  │  10 │  20 │  30 │  40 │  50 │  60 │  70 │  80 │  90 │                               │
    │  │(ptr)│(ptr)│(ptr)│(ptr)│(ptr)│(ptr)│(ptr)│(ptr)│(ptr)│                               │
    │  └───┬─┴───┬─┴───┬─┴───┬─┴───┬─┴───┬─┴───┬─┴───┬─┴───┬─┘                               │
    │      ▼      ▼      ▼      ▼      ▼      ▼      ▼      ▼      ▼                                │
    │    Data Pages (Sorted by Key)                                                         │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  PK=10  │  PK=20  │  PK=30  │  PK=40  │  ...                                    │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [검색 과정 예시: WHERE id = 50]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. Root Node: 50 >= 50, 오른쪽 서브트리로                                                 │
    │  2. Internal Node: 50 >= 50, 오른쪽                                                        │
    │  3. Leaf Node: Key=50 발견                                                              │
    │  4. Data Page의 Row直接 접근                                                            │
    │                                                                                         │
    │  → Height가 높으면 I/O 증가 (디스크 접근)                                             │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Clustered vs Non-Clustered Index

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Clustered vs Non-Clustered                                        │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Clustered Index (클러스터드 인덱스)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 데이터 자체를 **인덱스 순서대로 물리적 저장**                                                │
    │  │  • 1개 테이블에 **1개만** 존재 가능 (보통 Primary Key)                                      │
    │  │  • Insert/Update 시 **페이지 분할(PAGE Split)** 발생 가능                                    │
    │                                                                                         │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  Pages:                                                                                │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  │  Page 1:  ID 1~100   (정렬된 상태로 저장)                                        │  │  │
    │  │  │  Page 2:  ID 101~200                                                          │  │  │
    │  │  │  Page 3:  ID 201~300                                                          │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────┘  │  │
    │  │                                                                                      │  │
    │  │  Range Scan (WHERE id BETWEEN 100 AND 200):                                         │  │
    │  │  • 인접 Page가 순차적으로 저장되어 있어 Sequential I/O                                 │  │
    │  │  │  → 매우 빠름                                                                       │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Non-Clustered Index (논클러스터드 인덱스)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • **별도 저장 공간**에 **인덱스 구조** 유지                                                 │
    │  │  • **여러 개** 생성 가능                                                              │
    │  │  • **Logical Pointer**로 실제 데이터 위치 참조                                           │
    │                                                                                         │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  Index Pages (Sorted by Index Key)                                                 │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  │  Key: 50, 10, 90, 30, 70, ...  (정렬됨)                                         │  │  │
    │  │  │  Pointer: ┌───┬───┬───┬───┬───┬───┬───┐                                        │  │  │
    │  │  │           │   │   │   │   │   │   │   │                                        │  │  │
    │  │  │           ▼   ▼   ▼   ▼   ▼   ▼   ▼   ▼  (Random Order Data Pages)            │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │
    │  │                                                                                      │  │
    │  │  Range Scan:                                                                           │  │
    │  │  • Key는 정렬되어 있으나, Data Page는 Random                                           │  │
    │  │  │  → Random I/O 발생                                                                 │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 인덱스 종류

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         인덱스 종류                                                       │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [B-Tree Index]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 가장 일반적인 인덱스                                                                  │
    │  │  • =, >, <, >=, <=, BETWEEN, LIKE 'prefix%' 등 지원                                         │
    │  │  • MySQL InnoDB, PostgreSQL 기본                                                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Hash Index]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Hash 함수로 Key → Value 매핑                                                            │
    │  │  • =, IN 연산만 지원 (Range Query 불가)                                                    │
    │  │  • O(1) 평균 검색 속도                                                                 │
    │  │  • MySQL Memory Engine, PostgreSQL Index Type                                            │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Full-Text Index]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 텍스트 검색 최적화                                                                      │
    │  │  • MATCH ... AGAINST (MySQL), tsvector (PostgreSQL)                                     │
    │  │  • Inverted Index (단어 → 문서 위치)                                                   │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Bitmap Index]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Low Cardinality 컬럼에 사용 (성별, 진위형)                                               │
    │  │  • 비트맵으로 존재 여부 표현                                                             │
    │  │  • 데이터 웨어하우스에서 주로 사용                                                        │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [GiST / GIN (PostgreSQL)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • GiST (Generalized Search Tree): 복잡한 자료 구조                                           │
    │  │  • GIN (Generalized Inverted Index): 배열 값, JSONB                                   │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 인덱스 사용 판단

| 상황 | 인덱스 유용 | 인덱스 비추천 |
|------|----------|------------|
| **WHERE 절** | O | X |
| **조인 컬럼** | O | X |
| **Low Cardinality** | X (Bitmap) | B-Tree 비추천 |
| **자주 업데이트** | X (쓰기 비용) | X |
| **Small Table** | X (Full Scan 빠름) | X |

### Covering Index

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Covering Index                                                  │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 쿼리에 필요한 **모든 컬럼을 포함**하는 인덱스                                             │
    │  │  • 테이블 접근 없이 인덱스만으로 쿼리 완료                                                   │
    │  │  • Random I/O 최소화                                                                    │
    │                                                                                         │
    │  [예시]                                                                                  │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  -- 테이블                                                                             │  │
    │  │  CREATE TABLE users (                                                                 │  │
    │  │      id INT PRIMARY KEY,                                                             │  │
    │  │      name VARCHAR(50),                                                               │  │
    │  │      email VARCHAR(100),                                                             │  │
    │  │      age INT                                                                          │  │
    │  │  );                                                                                   │  │
    │  │                                                                                       │  │
    │  │  -- 인덱스 생성 (Covering)                                                             │  │
    │  │  CREATE INDEX idx_users_email_age ON users(email, age);                              │  │
    │  │                                                                                       │  │
    │  │  -- 쿼리 (Covering Index 사용)                                                          │  │
    │  │  SELECT email, age FROM users WHERE email = 'user@example.com';                       │  │
    │  │  -- → 인덱스만으로 결과 도출 (테이블 접근 불필요!)                                     │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 인덱스 설계
**상황**: 사용자 테이블
**판단**:

```sql
-- 1. Primary Key (Clustered)
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,  -- 자동 Clustered Index
    username VARCHAR(50),
    email VARCHAR(100),
    created_at TIMESTAMP
);

-- 2. Unique Index (중복 방지 + 검색)
CREATE UNIQUE INDEX idx_users_username ON users(username);
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- 3. Secondary Index (검색 최적화)
CREATE INDEX idx_users_created_at ON users(created_at);  -- Range Query

-- 4. Composite Index (다중 컬럼)
CREATE INDEX idx_users_name_created ON users(username, created_at);
-- SELECT * FROM users WHERE username = 'hong' AND created_at > '2023-01-01';
-- → Prefix 사용 가능

-- 5. Covering Index
CREATE INDEX idx_users_cover ON users(email, username, age);
-- SELECT email, username, age FROM users WHERE email = '...';
-- → Covering (테이블 접근 없음)

-- 6. Function-Based Index (PostgreSQL)
CREATE INDEX idx_users_email_lower ON users(LOWER(email));
-- WHERE LOWER(email) = 'USER@EXAMPLE.COM'
```

## Ⅴ. 기대효과 및 결론

### 인덱스 기대 효과

| 효과 | 인덱스 없을 시 | 인덱스 있을 시 |
|------|-------------|--------------|
| **SELECT 성능** | O(n) Full Table Scan | O(log n) Index Scan |
| **INSERT 성능** | 빠름 | 느림 (인덱스 업데이트) |
| **저장 공간** | 적음 | 큼음 (중복 저장) |
| **UPDAET 비용** | 낮음 | 높음 (인덱스 재구성) |

### 미래 전망

1. **Adaptive Index**: 자동 인덱스 생성
2. **Partial Index**: 조건부 인덱스
3. **HypothesisPG**: 자동 인덱스 추천
