# 정보보안 3요소 — CIA Triad (기밀성·무결성·가용성) + 확장 속성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정보보안의 근본 원칙인 CIA Triad는 데이터를 보호하기 위한 3대 핵심 목표(기밀성·무결성·가용성)를 정의하며, 모든 보안 통제는 이 3요소의 균형을 맞추는 것을 목적으로 한다.
> 2. **진화**: 현대 정보보안에서는 CIA에 인증성(Authenticity), 부인방지(Non-repudiation), 책임추적성(Accountability)을 추가하여 CIA+AAN 또는 Parkerian Hexad(6원칙)로 확장되었다.
> 3. **상충**: 기밀성과 가용성은 본질적으로 상충(Trade-off) 관계에 있으며, 보안 설계자는 비즈니스 요구사항에 따라 최적의 균형점을 찾아야 한다.

## Ⅰ. CIA Triad 개요 및 정의

### 1. 정보보안 3요소의 정의

```
                    ┌─────────────────────────────────────┐
                    │         Information Security        │
                    │         (정보보안의 목표)            │
                    └─────────────────────────────────────┘
                                      │
           ┌──────────────────────────┼──────────────────────────┐
           │                          │                          │
           ▼                          ▼                          ▼
    ┌─────────────┐           ┌─────────────┐           ┌─────────────┐
    │Confidentiality│         │  Integrity  │           │Availability │
    │   기밀성     │           │   무결성    │           │   가용성    │
    └─────────────┘           └─────────────┘           └─────────────┘
           │                          │                          │
           │                          │                          │
    "인가된 주체만"            "데이터의 정확성"          "필요할 때 접근"
    정보에 접근 가능           완전성 보장                서비스 지속성
```

| 요소 | 영문 | 정의 | 핵심 통제 기술 | 위반 시 위협 |
|:---:|:---:|:---|:---|:---|
| **기밀성** | Confidentiality | 인가된 사용자, 프로세스, 시스템만이 정보에 접근할 수 있음을 보장 | 암호화, 접근제어, 마스킹 | 정보 유출, 산업 스파이 |
| **무결성** | Integrity | 정보가 불법적/비인가적으로 생성, 변경, 삭제되지 않음을 보장 | 해시, 전자서명, MAC, 버전 관리 | 데이터 변조, 사기 |
| **가용성** | Availability | 인가된 사용자가 필요할 때 언제든 정보와 자원에 접근 가능함을 보장 | HA 설계, 이중화, DDoS 방어 | 서비스 중단, 랜섬웨어 |

### 2. CIA Triad의 상충 관계 (Trade-off Analysis)

```
                    [ 기밀성 (Confidentiality) ]
                              ▲
                             /│\
                            / │ \
                           /  │  \
               강한 암호화 /   │   \ 높은 가용성
               엄격한 접근제어   │     \ 빠른 응답시간
                          /    │      \
                         /     │       \
                        /      │        \
                       /       │         \
    [ 상충 ]◄─────────/────────┼──────────\─────────►[ 상충 ]
    (암호화 오버헤드)          │           (데이터 복제로 인한
                              │            무결성 위험)
                              │
                              ▼
                    [ 가용성 (Availability) ]
                              │
                              │
                     [ 무결성 (Integrity) ]
                    (강한 무결성 검증은
                     처리 지연 유발)
```

**상충 관계의 실제 사례**:
- **기밀성 vs 가용성**: 강력한 암호화는 처리 속도 저하, MFA는 사용자 편의성 저하
- **무결성 vs 가용성**: 실시간 무결성 검증(체크섬, MAC)은 지연 발생
- **기밀성 vs 무결성**: 암호화된 데이터에 대한 무결성 검증이 복잡해짐

```
┌─────────────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────────────┤
```

```
│  Level 4: 극비 (Top Secret / TS)                                   │

│  ├─ 정의: 조직의 생존에 심각한 피해를 줄 수 있는 정보                │
│  ├─ 예시: 영업비밀, 핵심 소스코드, M&A 정보                         │
│  └─ 통제: 강력한 암호화 + 접근 로그 + DLP                           │
│                                                                     │
│  Level 2: 내부용 (Confidential / Internal)                         │
│  ├─ 정의: 내부 직원만 접근 가능한 업무 정보                         │
│  ├─ 예시: 인사 정보, 재무 보고서, 내부 메일                         │
│  └─ 통제: 인증된 사용자만 + 디렉토리 접근 제어                      │
│                                                                     │
│  Level 1: 공개 (Public / Unclassified)                             │
│  ├─ 정의: 외부 공개가 허용된 정보                                   │
│  ├─ 예시: 홈페이지 콘텐츠, 보도 자료, 사용자 매뉴얼                  │
│  └─ 통제: 무결성 보호 중심 (변조 방지)                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 무결성 (Integrity) 심화

### 1. 무결성의 두 가지 측면

```
┌────────────────────────────────────────────────────────────────────┐
│                     무결성 (Integrity)의 이중 측면                 │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  ┌──────────────────────────┐   ┌──────────────────────────┐      │
│  │   데이터 무결성          │   │   시스템 무결성          │      │
│  │   (Data Integrity)       │   │   (System Integrity)     │      │
│  ├──────────────────────────┤   ├──────────────────────────┤      │
│  │                          │   │                          │      │
│  │  • 정확성 (Accuracy)     │   │  • 하드웨어 무결성       │      │
│  │    데이터가 참값과 일치  │   │    TPM, Secure Boot      │      │
│  │                          │   │                          │      │
│  │  • 완전성 (Completeness) │   │  • 소프트웨어 무결성     │      │
│  │    필요한 모든 데이터    │   │    코드 서명, SBOM       │      │
│  │    존재 보장             │   │                          │      │
│  │                          │   │  • 프로세스 무결성       │      │
│  │  • 일관성 (Consistency)  │   │    Change Control        │      │
│  │    충돌 없는 상태 유지   │   │    승인된 변경만 허용    │      │
│  │                          │   │                          │      │
│  └──────────────────────────┘   └──────────────────────────┘      │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 2. 무결성 검증 기술 비교

| 기술 | 원리 | 제공 속성 | 용도 | 보안 강도 |
|:---:|:---|:---|:---|:---:|
| **Checksum** | 단순 합계/패리티 | 오류 탐지 (우연적) | 통신 오류, 파일 무결성 | 낮음 |
| **CRC** | 다항식 기반 순환 코드 | 오류 탐지 (우연적) | 네트워크 프레임, 저장 장치 | 낮음 |
| **Hash (SHA-256)** | 암호학적 일방향 함수 | 위변조 탐지 | 파일 무결성, 블록체인 | 높음 |
| **MAC (HMAC)** | 키 기반 해시 | 무결성 + 인증 | 메시지 인증, API 보안 | 높음 |
| **Digital Signature** | 비대칭키 서명 | 무결성 + 인증 + 부인방지 | 문서 서명, 코드 서명 | 매우 높음 |

### 3. 무결성 보호 메커니즘 흐름도

```
┌─────────────────────────────────────────────────────────────────────┐
│              무결성 검증 흐름 (Hash-based Integrity Check)          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [ 송신자 (Sender) ]                                                │
│                                                                     │
│  ┌──────────────┐      ┌──────────────┐      ┌──────────────┐      │
│  │   원본 데이터  │──────►│  Hash 함수   │──────►│  해시값 (H1) │      │
│  │   (Message)   │      │  (SHA-256)   │      │  256-bit     │      │
│  └──────────────┘      └──────────────┘      └──────┬───────┘      │
│                                                       │              │
│                                                       ▼              │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │                 전송 (Data + Hash)                            │  │
│  │    [ 원본 데이터 ] + [ 해시값 H1 ] 또는 [ 서명된 해시 ]       │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                       │              │
│                                                       ▼              │
│  [ 수신자 (Receiver) ]                                              │
│                                                                     │
│  ┌──────────────┐      ┌──────────────┐      ┌──────────────┐      │
│  │   수신 데이터  │──────►│  Hash 함수   │──────►│  해시값 (H2) │      │
│  │   (Message)   │      │  (SHA-256)   │      │  256-bit     │      │
│  └──────────────┘      └──────────────┘      └──────┬───────┘      │
│                                                       │              │
│                                                       ▼              │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │                    비교 (H1 == H2 ?)                         │  │
│  ├──────────────────────────────────────────────────────────────┤  │
│  │  일치 (H1 == H2)  ──►  무결성 검증 성공 ✓                    │  │
│  │  불일치 (H1 != H2) ──►  무결성 검증 실패 ✗ (데이터 변조 의심) │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 가용성 (Availability) 심화

### 1. 가용성 보장 아키텍처 (High Availability Design)


### 2. 가용성 지표 및 SLA 체계

```
┌─────────────────────────────────────────────────────────────────────┐
│                   가용성 SLA 및 연간 가동 시간                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  SLA 등급    │ 가용성 (%) │ 연간 다운타임 │ 월간 다운타임 │ 적용    │
│  ────────────┼────────────┼───────────────┼───────────────┼─────────│
│  2 Nine      │   99%      │   3일 15시간  │    7시간 12분 │ 일반    │
│  3 Nine      │   99.9%    │   8시간 46분  │    43분 48초  │ 상용    │
│  4 Nine      │   99.99%   │   52분 36초   │    4분 23초   │ 금융    │
│  5 Nine      │   99.999%  │   5분 16초    │    26초       │ 통신    │
│  6 Nine      │   99.9999% │   32초        │    2.6초      │ 원전    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

가용성 계산 공식:
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│    Availability = MTBF / (MTBF + MTTR) × 100%                       │
│                                                                     │
│    MTBF: Mean Time Between Failures (평균 고장 간격)               │
│    MTTR: Mean Time To Repair (평균 복구 시간)                      │
│                                                                     │
│    예시: MTBF = 10,000시간, MTTR = 1시간                            │
│    Availability = 10,000 / (10,000 + 1) = 99.99%                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3. 가용성 위협 요인

```
┌─────────────────────────────────────────────────────────────────────┐
│                    가용성 위협 (Availability Threats)               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                    기술적 위협 (Technical)                     │ │
│  ├───────────────────────────────────────────────────────────────┤ │
│  │  • 하드웨어 고장 (HDD, SSD, NIC, Power Supply)                │ │
│  │  • 소프트웨어 버그 (Memory Leak, Deadlock)                    │ │
│  │  • 네트워크 장애 (라우터, 스위치, 광케이블 절단)               │ │
│  │  • 데이터베이스 락 (Lock Contention)                          │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                    인위적 위협 (Human-made)                    │ │
│  ├───────────────────────────────────────────────────────────────┤ │
│  │  • DDoS 공격 (Volumetric, Protocol, Application Layer)        │ │
│  │  • 랜섬웨어 (Ransomware) — 데이터 암호화/삭제                  │ │
│  │  • 사보타주 (내부자 악의적 행위)                               │ │
│  │  • 운영 실수 (설정 오류, 실수로 인한 삭제)                     │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                    자연재해 (Natural)                          │ │
│  ├───────────────────────────────────────────────────────────────┤ │
│  │  • 지진, 홍수, 화재, 태풍                                     │ │
│  │  • 정전 (Power Outage)                                        │ │
│  │  • 냉각 시스템 장애 (과열)                                     │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 확장 속성: CIA+AAN / Parkerian Hexad

### 1. CIA Triad 확장 개념도

```
                           ┌─────────────────────┐
                           │   Authenticity      │
                           │     인증성          │
                           │  (진위 확인)        │
                           └─────────────────────┘
                                      │
           ┌──────────────────────────┼──────────────────────────┐
           │                          │                          │
           ▼                          ▼                          ▼
    ┌─────────────┐           ┌─────────────┐           ┌─────────────┐
    │Confidentiality│         │  Integrity  │           │Availability │
    │   기밀성     │◄────────►│   무결성    │◄────────►│   가용성    │
    └─────────────┘           └─────────────┘           └─────────────┘
           ▲                          ▲                          ▲
           │                          │                          │
           │                          │                          │
    ┌─────────────┐           ┌─────────────┐           ┌─────────────┐
    │Accountability│          │Non-repudiation│         │  (Possession)│
    │  책임추적성  │           │   부인방지   │           │    소유성    │
    │ (감사 가능) │           │ (행위 증명)  │           │  (Parkerian) │
    └─────────────┘           └─────────────┘           └─────────────┘
```

### 2. 확장 속성 상세 분석

| 속성 | 영문 | 정의 | 핵심 기술 | 적용 분야 |
|:---:|:---:|:---|:---|:---|
| **인증성** | Authenticity | 데이터, 주체, 시스템이 주장하는 대로 진품임을 보장 | PKI, 전자서명, 생체인식 | 전자상거래, 전자문서 |
| **부인방지** | Non-repudiation | 특정 행위(전송, 수신, 생성)를 부인할 수 없게 만듦 | 전자서명, 타임스탬프, 감사로그 | 법적 증거, 계약 |
| **책임추적성** | Accountability | 모든 보안 관련 행위를 특정 주체로 추적 가능 | SIEM, 감사로그, IAM | 규정 준수, 포렌식 |
| **소유성** | Possession | 물리적/논리적 자산에 대한 소유권 보호 | 물리적 보안, DRM | 디지털 저작권 |

### 3. Parkerian Hexad (6원칙)

```
┌─────────────────────────────────────────────────────────────────────┐
│                Parkerian Hexad (Donn Parker의 6원칙)                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│         기존 CIA Triad에 3가지 속성을 추가하여 완전성 확보          │
│                                                                     │
│  1. Confidentiality (기밀성)                                        │
│     └─ 인가된 자만 정보에 접근 가능                                 │
│                                                                     │
│  2. Integrity (무결성)                                              │
│     └─ 정보의 정확성과 완전성 보장                                  │
│                                                                     │
│  3. Availability (가용성)                                           │
│     └─ 필요 시 정보에 접근 가능                                     │
│                                                                     │
│  4. Possession (소유성/점유)                                        │
│     └─ 정보의 물리적 소유권 통제 (도난 방지)                        │
│     └─ 예: 노트북 분실 시 데이터 소유권 상실                        │
│                                                                     │
│  5. Authenticity (진정성)                                           │
│     └─ 정보와 출처의 진위 확인                                      │
│     └─ 예: 피싱 사이트 vs 진짜 은행 사이트 구분                     │
│                                                                     │
│  6. Utility (유용성)                                                │
│     └─ 정보의 가치와 사용 가능성                                    │
│     └─ 예: 암호화된 백업은 기밀성은 있으나 유용성 없을 수 있음      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅵ. 보안 원칙과의 연관성

### 1. CIA와 보안 원칙 매핑

```
┌─────────────────────────────────────────────────────────────────────┐
│              CIA Triad ↔ 보안 원칙 매핑                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │           최소 권한 원칙 (Principle of Least Privilege)       │ │
│  ├───────────────────────────────────────────────────────────────┤ │
│  │  → 기밀성 강화: 불필요한 접근 차단                             │ │
│  │  → 무결성 강화: 불필요한 수정 권한 제거                        │ │
│  │  → 가용성 강화: 권한 남용으로 인한 서비스 영향 최소화          │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │           심층 방어 (Defense in Depth)                        │ │
│  ├───────────────────────────────────────────────────────────────┤ │
│  │  → 다층 보안으로 CIA 3요소를 다중으로 보호                     │ │
│  │  → 단일 통제 실패 시에도 다른 계층에서 보안 유지               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │           알 필요성 원칙 (Need-to-Know)                       │ │
│  ├───────────────────────────────────────────────────────────────┤ │
│  │  → 기밀성 핵심 원칙: 업무 수행에 필요한 정보만 접근            │ │
│  │  → 직무 분리(SoD)와 결합하여 내부자 위협 완화                  │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2. CIA 관점 위협-통제 매트릭스

| 위협 유형 | CIA 요소 영향 | 대응 통제 (Controls) | 관련 표준 |
|:---|:---:|:---|:---|
| 데이터 유출 | 기밀성 | 암호화, DLP, 접근제어 | ISO 27001 A.8.2 |
| 데이터 변조 | 무결성 | 해시, 전자서명, DB Audit | ISO 27001 A.8.3 |
| DDoS 공격 | 가용성 | WAF, CDN, Rate Limiting | ISO 27001 A.12.1 |
| 랜섬웨어 | 기밀성+가용성 | 백업, EDR, 네트워크 분리 | NIST CSF PR.IP |
| 피싱 | 기밀성+인증성 | MFA, 교육, 이메일 필터 | ISO 27001 A.7.2 |
| 내부자 위협 | 전 영역 | UEBA, PAM, 감사로그 | ISO 27001 A.7.1 |

## Ⅶ. 실무 적용 및 기술사적 판단

### 시나리오: 금융권 핀테크 서비스의 CIA 균형 설계

**요구사항**:
- 고객 계좌 정보의 기밀성: 최우선
- 실시간 이체 서비스의 가용성: 99.99% SLA
- 거래 데이터 무결성: 완벽 보장

**설계 결정**:

```
┌─────────────────────────────────────────────────────────────────────┐
│           금융 서비스 CIA 균형 아키텍처                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [ 기밀성: 최우선 ]                                                 │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • HSM 기반 키 관리 (FIPS 140-2 Level 3)                      │ │
│  │  • 필드 수준 암호화 (계좌번호, 주민번호)                       │ │
│  │  • TLS 1.3 전송 암호화 (PFS 필수)                             │ │
│  │  • MFA + 생체 인증 (FIDO2)                                    │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  [ 무결성: 완벽 보장 ]                                              │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • ACID 트랜잭션 (RDBMS)                                      │ │
│  │  • 전자서명 기반 거래 승인                                     │ │
│  │  • 감사 로그 (WORM Storage)                                   │ │
│  │  • 실시간 무결성 검증 (SHA-256)                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  [ 가용성: 99.99% SLA ]                                             │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • Active-Active 다중 AZ 배포                                 │ │
│  │  • 자동 페일오버 (RTO < 30초)                                  │ │
│  │  • DDoS 방어 (Scrubbng Center)                                │ │
│  │  • 정기적 재해 복구 훈련                                       │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```
