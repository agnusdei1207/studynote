# 모니터 (Monitor)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모니터는 **"공유 데이터와 **동기화 코드를 **캡슐화한 **고수준 동기화 추상화"**로, **Mutex Lock**과 **Condition Variable**로 구성되며 **한 번에 하나의 프로세스/스레드**만 **모니터 내부**에 **진입**할 수 있다.
> 2. **가치**: **Semaphore**, **Mutex Lock**의 **복잡성**을 **숨기**고 **캡슐화**로 **코드 안정성**을 높이며 **Producer-Consumer**, **Reader-Writer** 같은 **복잡한 동기화 문제**를 **우아하게** 해결한다.
> 3. **융합**: **Java synchronized**, **wait()/notify()**, **ReentrantLock**, **Condition**으로 **구현**되며 **Hoare Semantics**(즉시 재개)와 **Mesa Semantics**(신호 후 재검사)가 **잠금 반환 방식**에 따라 **다르다**.

---

## Ⅰ. 개요 (Context & Background)

### 개념
모니터는 **"공유 자원과 동기화를 캡슐화한 추상 데이터 타입"**이다.

**모니터의 구조**:
- **Lock**: 진입 시 잠금 획득
- **Shared Data**: 보호할 데이터
- **Condition Variables**: 대기/신호

### 💡 비유
모니터는 **"협상실"**과 같다.
- **문**: Lock (한 사람만 들어갈 수 있음)
- **협상**: Shared Data 접근
- **대기실**: Condition Variable (조건 대기)

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         모니터의 필요성                                                   │
└─────────────────────────────────────────────────────────────────────────────────────────┘

Semaphore/Mutex:
    • 저수준 원시어
    • 프로그래머 실수 유발
    • 복잡한 동기화 어려움
         ↓
Monitor (Hoare, 1974):
    • 고수준 추상화
    • 캡슐화
    • 자동 잠금
         ↓
현대:
    • Java synchronized
    • C# lock
```

### Condition Variable 동작

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Condition Variable (wait/signal)                                │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Wait 동작]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Thread A                              Monitor                                         │
    │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  monitor.lock.acquire();           # Lock 획득                                      │  │
    │  │  while (!condition) {               # 조건 확인                                      │  │
    │  │      condition.wait();              # 조건 대기                                      │  │
    │  │      # 1. Lock 해제                                                                 │  │
    │  │      # 2. 대기 큐에 진입                                                               │  │
    │  │      # 3. Block (sleep)                                                               │  │
    │  │  }                                                                                    │  │
    │  │  # signal() 받은 후 재개                                                            │  │
    │  │  # Lock 재획득                                                                       │  │
    │  │  # Critical Section                                                               │  │
    │  │  monitor.lock.release();           # Lock 해제                                      │  │
    │  └──────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Signal 동작]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Thread B                              Monitor                                         │
    │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  monitor.lock.acquire();           # Lock 획득                                      │  │
    │  │  condition = true;                  # 조건 충족                                     │  │
    │  │  condition.signal();                # 대기 중인 스레드 신호                           │  │
    │  │  # 대기 큐에서 Thread A 깨움                                                          │  │
    │  │  monitor.lock.release();           # Lock 해제                                      │  │
    │  └──────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Hoare vs Mesa Semantics

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Hoare vs Mesa Semantics                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [Hoare Semantics (1974)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • signal() 호출 즉시 대기 중인 스레드로 제어 이전                                         │
    │  │  • signal() 호출한 스레드는 대기 큐로 이동                                             │
    │  │  • 빠른 응답, 복잡한 구현                                                            │
    │  │                                                                                      │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  Thread A (waiter)                Thread B (signaler)                              │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  │ condition.wait()               condition.signal()                              │  │  │
    │  │  │   ↓ 즉시 재개                     ↓ 즉시 대기                                    │  │  │
    │  │  │  Lock 획득                       (B는 A가 완료될 때까지 대기)                       │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────┘  │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [Mesa Semantics (Xerox PARC, 1980)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • signal()은 알림만, 대기 중인 스레드은 나중에 재개                                        │
    │  │  • signal() 호출한 스레드 계속 실행                                                    │
    │  │  • wait() 후에는 항상 조건 재검사 (while 루프)                                         │
    │  │  • 실용적, 대부분 시스템 채택                                                         │
    │  │                                                                                      │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  while (!condition) {         # 항상 재검사 (Mesa)                                    │  │
    │  │      condition.wait();                                                            │  │
    │  │  }                                                                                  │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 동기화 도구 비교

| 구분 | Mutex | Semaphore | Monitor |
|------|-------|-----------|---------|
| **수준** | 저수준 | 저수준 | 고수준 |
| **캡슐화** | X | X | O |
| **Lock** | 명시적 | 명시적 | 자동 |
| **복잡성** | 높음 | 높음 | 낮음 |

### Bounded Buffer (Producer-Consumer) 예시

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         모니터로 구현한 Bounded Buffer                                   │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  monitor BoundedBuffer {                                                              │
    │  │  Condition notFull, notEmpty;                                                      │
    │  │  int[] buffer = new int[N];                                                         │
    │  │  int count = 0;                                                                     │
    │  │                                                                                      │
    │  │  void put(int item) {                                                              │  │
    │  │      while (count == N) {               # 버퍼 가득 참                              │  │
    │  │          notFull.wait();               # 여유 있을 때까지 대기                        │  │
    │  │      }                                                                                  │  │
    │  │      buffer[count++] = item;           # 데이터 삽입                                 │  │
    │  │      notEmpty.signal();                # 비어 있음 알림 해제                           │  │
    │  │  }                                                                                      │  │
    │  │                                                                                       │  │
    │  │  int get() {                                                                          │  │
    │  │      while (count == 0) {               # 버퍼 비어 있음                              │  │
    │  │          notEmpty.wait();              # 데이터 있을 때까지 대기                        │  │
    │  │      }                                                                                  │  │
    │  │      int item = buffer[--count];     # 데이터 제거                                   │  │
    │  │      notFull.signal();                # 여유 생겼음 알림                             │  │
    │  │      return item;                                                                      │  │
    │  │  }                                                                                      │  │
    │  }                                                                                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: Java Monitor
**상황**: 스레드 안전한 큐
**판단**:

```java
// Java synchronized/wait/notify 예시
public class BoundedBuffer<T> {
    private final Queue<T> queue = new LinkedList<>();
    private final int capacity;

    public BoundedBuffer(int capacity) {
        this.capacity = capacity;
    }

    public synchronized void put(T item) throws InterruptedException {
        // while: Mesa Semantics (조건 재검사)
        while (queue.size() == capacity) {
            wait();  // Lock 해제 후 대기
        }
        queue.add(item);
        notifyAll();  // 대기 중인 소비자 알림
    }

    public synchronized T get() throws InterruptedException {
        while (queue.isEmpty()) {
            wait();
        }
        T item = queue.poll();
        notifyAll();  // 대기 중인 생산자 알림
        return item;
    }
}

// ReentrantLock + Condition (Java 5+)
public class BoundedBuffer<T> {
    private final Queue<T> queue = new LinkedList<>();
    private final int capacity;
    private final Lock lock = new ReentrantLock();
    private final Condition notFull = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (queue.size() == capacity) {
                notFull.await();
            }
            queue.add(item);
            notEmpty.signal();
        } finally {
            lock.unlock();
        }
    }

    public T get() throws InterruptedException {
        lock.lock();
        try {
            while (queue.isEmpty()) {
                notEmpty.await();
            }
            T item = queue.poll();
            notFull.signal();
            return item;
        } finally {
            lock.unlock();
        }
    }
}
```

## Ⅴ. 기대효과 및 결론

### 모니터 기대 효과

| 효과 | Semaphore | Monitor |
|------|-----------|---------|
| **캡슐화** | X | O |
| **자동 Lock** | X | O |
| **실수 가능성** | 높음 | 낮음 |
| **복잡한 동기화** | 어려움 | 비교적 쉬움 |
