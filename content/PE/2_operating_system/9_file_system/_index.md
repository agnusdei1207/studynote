# 09. 파일 시스템 (File System)

## ## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보조기억장치의 물리적 블록을 사용자 친화적인 논리적 단위인 '파일(File)'과 '디렉토리(Directory)'로 조직화하는 기술.
> 2. **가치**: 데이터의 영구 저장, 이름 기반 접근, 권한 제어 및 메타데이터 관리를 통한 정보의 구조화.
> 3. **융합**: 아이노드(inode) 구조와 저널링(Journaling) 기법을 통한 시스템 장애 시 데이터 복구력 확보.

---

### Ⅰ. 개요 (Context & Background)
파일 시스템은 정보를 영구적으로 보존하고 관리하기 위한 추상화의 정수다. 사용자는 디스크의 섹터 번호를 알 필요 없이 '파일명'만으로 수 기가바이트의 데이터에 접근할 수 있다.

---

### Ⅱ. 아키텍처 및 핵심 원리

#### 1. 주요 구성 요소
- **inode**: 파일의 소유주, 권한, 위치 정보 등을 담은 메커니즘
- **VFS (Virtual File System)**: 서로 다른 파일 시스템을 통합 관리하는 계층
- **Journaling**: 변경 사항을 미리 로그에 기록하여 일관성 유지
- **Allocation**: Contiguous, Linked, Indexed 할당 기법

#### 2. inode 구조 (ASCII)
```text
    [ inode Structure ]
    
    +-----------------------+
    |  File Metadata        | (Owner, Permissions, Size, Time)
    +-----------------------+
    |  Direct Blocks (12)   | ----> [ Data Block 0-11 ]
    +-----------------------+
    |  Single Indirect      | ----> [ Block List ] ----> [ Data ]
    +-----------------------+
    |  Double Indirect      | ----> [ List ] -> [ List ] -> [ Data ]
    +-----------------------+
```

---

### Ⅲ. 융합 비교 및 다각도 분석

#### 파일 할당 기법 비교
| 항목 | 연속 할당 (Contiguous) | 연결 할당 (Linked) | 인덱스 할당 (Indexed) |
| :--- | :--- | :--- | :--- |
| **방식** | 연속된 블록 점유 | 포인터로 연결 | 인덱스 블록에 주소 모음 |
| **접근** | 직접 접근 가능 (빠름) | 순차 접근만 가능 (느림) | 직접 접근 가능 |
| **단편화** | 외부 단편화 심함 | 없음 | 인덱스 블록 공간 낭비 |
| **활용** | 초기 시스템, CD-ROM | 구형 FAT 계열 | 현대 UNIX/Linux (inode) |

---

### Ⅳ. 실무 적용 및 기술사적 판단
실무적으로 **File System Corruption**은 데이터 소실로 이어지는 재앙이다. 기술사는 저널링 파일 시스템(Ext4, XFS)을 기본으로 사용하고, 미러링이나 스냅샷 기능을 활용하여 최악의 상황에서도 데이터의 정합성을 보장해야 한다.

---

### Ⅴ. 기대효과 및 결론
파일 시스템은 분산 파일 시스템(HDFS)과 객체 스토리지(S3)로 확장되며 빅데이터 시대를 지탱하고 있다. 향후 비휘발성 메모리에 최적화된 초저지연 파일 시스템(DAX)이 대세가 될 전망이다.
