# 파일 시스템 (File System)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파일 시스템은 **\"디스크 **데이터**를 **파일**과 **디렉터리**로 **조직**하고 **저장**하고 **검색**하는 **시스템\"**으로, **Inode**(메타데이터), **Data Block**(실제 **내용), **Journal**(변경 **로그)**으로 **구성**된다.
> 2. **구조**: **VFS**(Virtual File System)가 **파일 **시스템 **추상화**를 **제공**하고 **Ext4**(Linux), **NTFS**(Windows), **APFS**(macOS), **ZFS**(Solaris)가 **대표적**이다.
> 3. **기능**: **Create**(생성), **Read**(읽기), **Write**(쓰기), **Delete**(삭제)**과 **Journaling**(안전성), **COW**(Copy-on-Write, **스냅샷)**, **Deduplication**(중복 **제거)**를 **지원**한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
파일 시스템은 **\"디스크 데이터 조직화\"**이다.

**파일 시스템 구성**:
| 구성요소 | 설명 | 예시 |
|----------|------|------|
| **Superblock** | 전체 메타데이터 | 전체 블록 수 |
| **Inode** | 파일 메타데이터 | 크기, 권한, 시간 |
| **Data Block** | 실 데이터 | 파일 내용 |
| **Journal** | 변경 로그 | 트랜잭션 |

### 💡 비유
파일 시스템은 ****서류 **정리 **시스템 ****과 같다.
- **파일**: 서류
- **디렉터리**: 폴더
- **Inode**: 목차

### Journaling

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Journaling (Ext3/4)                                               │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Problem: Crash during file write
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Scenario:                                                                             │  │
    │  1. Allocate data blocks                                                                │  │
    │  2. Write data to blocks     ← CRASH                                                   │  │
    │  3. Update inode                                                                       │  │
    │  4. Update bitmap                                                                      │  │
    │                                                                                         │  │
    │  Result: File corrupted (data written but metadata not updated)                          │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Solution: Journaling (Write-Ahead Log)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Transaction Steps:                                                                     │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. Journal Write: Write transaction to journal area first                           │  │  │
    │  │     ┌──────────────────────────────────────────────────────────────────────────────┐  │  │  │
    │  │     │  Transaction Header: {TXN_ID, "Allocate file X"}                             │  │  │  │
    │  │     │  Data Block 1: "Hello world..."                                              │  │  │  │
    │  │     │  Inode Update: size = 12, blocks = [...]                                     │  │  │  │
    │  │     │  Bitmap Update: mark blocks 1000, 1001 as used                               │  │  │  │
    │  │     │  Transaction Commit: TXN_ID committed                                        │  │  │  │
    │  │     └──────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  │                                                                                       │  │  │
    │  │  2. Checkpoint: Copy journal to actual filesystem locations                          │  │  │
    │  │     ┌──────────────────────────────────────────────────────────────────────────────┐  │  │  │
    │  │     │  Write data blocks → File blocks                                            │  │  │  │
    │  │     │  Write inode → Inode table                                                  │  │  │  │
    │  │     │  Write bitmap → Block bitmap                                                 │  │  │  │
    │  │     └──────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  │                                                                                       │  │  │
    │  │  3. Checkpoint: Mark transaction complete in journal                                 │  │  │
    │  │                                                                                       │  │  │
    │  │  Crash Recovery:                                                                     │  │  │
    │  │  • If crash before commit: Rollback (ignore incomplete transaction)                   │  │  │
    │  │  • If crash after commit but before checkpoint: Replay (redo transaction)              │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Journaling Modes:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  │  Mode         │  Data Journaling  │  Speed   │  Safety                               │  │
    │  │  ─────────────────────────────────────────────────────────────────────────────────────│  │
    │  │  data=journal │  Yes (metadata + data)│  Slowest │  Highest (all in journal)         │  │
    │  │  data=ordered │  No (data before metadata)│  Fast    │  Good (metadata in journal)      │  │
    │  │  data=writeback│ No (no ordering)     │  Fastest│  Lowest (old data after crash)     │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 데이터베이스 스토리지
**상황**: PostgreSQL 데이터 디렉터리
**판단**: XFS + Direct I/O

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Database Storage Configuration                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Requirements:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • High throughput: Random reads/writes                                                  │  │
```

## Ⅴ. 기대효과 및 결론

### 파일 시스템 기대 효과

| FS | 최대 크기 | 성능 | 안전성 |
|----|----------|------|--------|
| **Ext4** | 16TB | 좋음 | 좋음 |
| **XFS** | 8EB | 우수 | 우수 |
| **ZFS** | 256ZB | 중간 | 최고 |

### 모범 사례

1. **Journal**: 항상 활성화
2. **Flush**: sync, fsync
3. **Backup**: 스냅샷 활용

1. **COW**: Copy-on-Write 표준
2. **스냅샷**: 즉시 백업
3. **압축**: Zstd, LZ4
4. **암호화**: LUKS, BitLocker
