# 문맥 교환 (Context Switch)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 문맥 교환은 **"CPU **실행 **문맥**을 **저장**하고 **다른 **프로세스**나 **스레드**로 **전환**하는 **과정"**으로, **Process Switch**(커널 모드)와 **Thread Switch**(가능한 경우 사용자 모드)로 **구분**된다.
> 2. **비용**: **Register Save/Restore**, **PCB/TCB **업데이트**, **TLB Flush**(프로세스), **Cache **Pollution**으로 **오버헤드**가 **발생**하며 **일반적**으로 **1-20**마이크로초가 **소요**된다.
> 3. **최적화**: **Thread Switch**는 **같은 **프로세스** 내에서 **TLB Flush**가 **없어 **빠르며 **User-Level **Thread**(Fiber, **Coroutine**)는 **커널 **개입**없이 **더욱 **빠르다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
문맥 교환은 **"CPU 실행 상태 전환"**이다.

**교환 유형**:
- **Process Switch**: 다른 프로세스로 전환 (큰 비용)
- **Thread Switch**: 같은 프로세스 내 스레드 전환 (작은 비용)
- **Mode Switch**: 사용자/커널 모드 전환 (가장 작은 비용)

### 💡 비유
문맥 교환은 ****책상 **정리 ****후 **다른 **업무 **시작****과 같다.
- **책상**: CPU 레지스터
- **정리**: 상태 저장
- **다른 업무**: 다른 프로세스

## Ⅱ. 아키텍처 및 핵심 원리

### 문맥 교환 과정

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Context Switch Process                                             │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Time                                                                                   │  │
    │  ──────────────────────────────────────────────────────────────────────────────────────│  │
    │  Process A Running                                                                      │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  CPU executes Process A instructions                                                  │  │  │
    │  │  Registers: EAX=100, EBX=200, ESP=0x1000, EIP=0x4000                                 │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │            │                                                                                 │  │
    │            │ Interrupt (Timer, I/O, System Call)                                          │  │
    │            ▼                                                                                 │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. SAVE CONTEXT (Process A)                                                          │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │
    │  │  │  • Save all registers to PCB[A]                                                   │  │  │  │
    │  │  │  • Update PCB[A].state = READY                                                     │  │  │  │
    │  │  │  • Save current stack pointer                                                      │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │            │                                                                                 │  │
    │            ▼ Scheduler selects Process B                                                   │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  2. SWITCH ADDRESS SPACE (Process Switch only)                                        │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │
    │  │  │  • Load new page table (CR3 register on x86)                                      │  │  │  │
    │  │  │  • Flush TLB (all entries invalid)                                                │  │  │  │
    │  │  │  • → Cache misses after switch                                                    │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │            │                                                                                 │  │
    │            ▼                                                                                 │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  3. RESTORE CONTEXT (Process B)                                                        │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │
    │  │  │  • Load registers from PCB[B]                                                     │  │  │  │
    │  │  │  • Update PCB[B].state = RUNNING                                                   │  │  │  │
    │  │  │  • Jump to saved EIP                                                              │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │            │                                                                                 │  │
    │            ▼                                                                                 │  │
    │  Process B Running                                                                        │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  CPU executes Process B instructions                                                  │  │  │
    │  │  Registers: EAX=50, EBX=150, ESP=0x2000, EIP=0x8000                                   │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 프로세스 vs 스레드 문맥 교환

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Process vs Thread Switch                                           │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Process Switch (High Overhead):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. Save ~50 registers (GP, FP, SSE, Control)                                       │  │  │
    │  │  2. Update PCB                                                                       │  │  │
    │  │  3. Switch page table (CR3)                                                          │  │  │
    │  │  4. Flush TLB (all entries)                                                          │  │  │
    │  │  5. Update memory mapping                                                            │  │  │
    │  │  6. Restore registers                                                                │  │  │
    │  │  → Cost: ~5-20 microseconds                                                          │  │  │
    │  │  → TLB miss penalty after switch                                                     │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Thread Switch (Low Overhead):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. Save ~10 registers (SP, PC, GP)                                                  │  │  │
    │  │  2. Update TCB                                                                       │  │  │
    │  │  3. Update scheduler state                                                           │  │  │
    │  │  4. Restore registers                                                                │  │  │
    │  │  → Cost: ~1-2 microseconds                                                           │  │  │
    │  │  → NO TLB flush (same address space)                                                │  │  │
    │  │  → NO page table switch                                                              │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 오버헤드 비교

| 구분 | Mode Switch | Thread Switch | Process Switch |
|------|-------------|---------------|----------------|
| **Register** | 일부 | 전체 | 전체 |
| **TLB** | X | X | O (Flush) |
| **Page Table** | X | X | O (Switch) |
| **시간** | ~0.1μs | ~1-2μs | ~5-20μs |

### 문맥 교환 횟수

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Context Switch Frequency                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    High Frequency (Bad):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • 10,000 switches/sec → 10ms/sec CPU time lost (1% @ 1GHz)                            │  │
    │  • Cache pollution = lower overall performance                                          │  │
    │  • Thrashing: too much time switching, not enough executing                            │  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Measurement:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Linux: /proc/stat                                                                       │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  ctxt 1234567  ← Total context switches                                              │  │  │
    │  │                                                                                       │  │  │
    │  │  Rate = (ctxt2 - ctxt1) / (time2 - time1)                                            │  │  │
    │  │                                                                                       │  │  │
    │  │  vmstat 1                                                                             │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │
    │  │  │  procs -----------memory---------- ---swap-- -----io---- -system-- ----cpu----    │  │  │  │
    │  │  │   r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa    │  │  │  │
    │  │  │   1  0      0 6987340 208812 895452    0    0     0     0   42   78  1  0 99  0    │  │  │
    │  │  │                                                                         ↑         │  │  │  │
    │  │  │                                                                         cs        │  │  │  │
    │  │  │                                                                 (context switches)  │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 유저레벨 스레드

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         User-Level Threads                                                │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Kernel Threads (pthread):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Thread A → Kernel Scheduler → Thread B                                                 │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  → System call required for switch                                                   │  │  │
    │  │  → ~1-2 microseconds                                                                 │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    User-Level Threads (Fiber, Coroutine):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Fiber A → User Scheduler → Fiber B                                                      │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────~┐  │  │
    │  │  → NO system call (switch in user space)                                             │  │  │
    │  │  → ~10-100 nanoseconds                                                               │  │  │
    │  │  → Kernel sees only 1 thread                                                         │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Trade-offs:                                                                             │  │
    │  • Fast switching                                                                       │  │
    │  • No kernel preemption (cooperative)                                                    │  │
    │  • Blocking I/O blocks all fibers                                                       │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 측정 도구

```bash
# 문맥 교환 횟수 확인
vmstat 1 5
# cs 컬럼 확인

# 프로세스별 비자발적 문맥 교환
pidstat -w 1

# 출력:
#   PID   cswch/s nvcswch/s  Command
#   1234     100.5      10.2  myapp

# perf로 분석
perf sched record -- sleep 10
perf sched latency
```

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 높은 문맥 교환 최적화
**상황**: cs/sec > 50000
**판단**: 스레드 풀 사용

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Before vs After                                                   │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Before (Thread per Request):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Request 1 → Thread 1 ────┐                                                             │  │
    │  Request 2 → Thread 2 ────┼──→ 50,000 threads                                         │  │
    │  Request 3 → Thread 3 ────┤   → High context switches                                │  │
    │  ...                        │   → Memory overhead                                    │  │
    │  Request N → Thread N ────┘                                                             │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    After (Thread Pool):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Thread Pool (16 threads)                                                                │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Request Queue                                                                        │  │  │
    │  │    │                                                                                 │  │  │
    │  │    ▼                                                                                 │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │
    │  │  │  T1  T2  T3  ...  T16                                                            │  │  │  │
    │  │  │  ↓   ↓   ↓        ↓                                                              │  │  │  │
    │  │  │  R1  R2  R3  ...  R16 (assigned to idle threads)                                  │  │  │  │
    │  │  │  ↓   ↓   ↓        ↓                                                              │  │  │  │
    │  │  │  Done → back to pool                                                             │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  │                                                                                       │  │  │
    │  │  → 5,000 context switches/sec (90% reduction)                                        │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 문맥 교환 기대 효과

| 구분 | 높음 | 최적화 |
|------|------|--------|
| **cs/sec** | 50,000+ | < 10,000 |
| **CPU 사용** | 낭비 | 효율 |
| **성능** | 낮음 | 높음 |

### 모범 사례

1. **스레드 풀**: 과도한 생성 방지
2. **CPU 친화도**: 코어 고정
3. **코루틴**: 비동기 I/O
4. **모니터링**: vmstat, pidstat

### 미래 전망

1. **Fiber**: 경량 스레드
2. **Coroutine**: async/await
3. **User-Level Scheduling**: Go goroutine

### ※ 참고 표준/가이드
- **Linux**: sched(7)
- **Windows**: Thread Scheduling
- **Intel**: Context Switch Optimization
