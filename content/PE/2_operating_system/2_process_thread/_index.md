# 02. 프로세스 및 스레드 (Process & Thread)

## ## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 실행 중인 프로그램의 단위(Process)와 프로세스 내에서 실행되는 흐름의 최소 단위(Thread).
> 2. **가치**: 멀티태스킹을 통한 시스템 자원 활용 극대화 및 멀티스레딩을 통한 응답성 및 처리량 향상.
> 3. **융합**: PCB와 TCB를 통한 상태 관리 및 문맥 교환(Context Switch)을 통해 동시성(Concurrency) 구현.

---

### Ⅰ. 개요 (Context & Background)
프로세스는 운영체제가 자원을 할당하는 기본 단위이며, 스레드는 CPU가 연산을 수행하는 기본 단위다. 현대 운영체제는 수천 개의 프로세스와 스레드를 정교하게 교체하며 사용자가 마치 모든 작업이 동시에 일어나는 것처럼 느끼게 한다.

---

### Ⅱ. 아키텍처 및 핵심 원리

#### 1. 프로세스 제어 블록 (PCB) 구성
- **PID**: 프로세스 식별 번호
- **State**: New, Ready, Running, Waiting, Terminated
- **Program Counter**: 다음 실행할 명령어 주소
- **CPU Registers**: 문맥 교환 시 저장해야 할 레지스터 값
- **Memory Management Info**: 페이지 테이블 정보 등

#### 2. 프로세스 메모리 구조 및 스레드 공유 (ASCII)
```text
    [ Process Memory Layout ]          [ Multi-threaded Process ]
    
    +-----------------------+          +-----------------------+
    |   Stack (Local Vars)  |          | Thread 1 Stack | T2 S |
    +-----------v-----------+          +-----------------------+
    |           |           |          |         Heap          |
    +-----------^-----------+          +-----------------------+
    |   Heap (Dynamic)      |          |         Data          |
    +-----------------------+          +-----------------------+
    |   Data (Global)       |          |         Code          |
    +-----------------------+          +-----------------------+
    |   Code (Text)         |          | Shared by all threads |
    +-----------------------+          +-----------------------+
```

---

### Ⅲ. 융합 비교 및 다각도 분석

#### 프로세스 vs 스레드
| 비교 항목 | 프로세스 (Process) | 스레드 (Thread) |
| :--- | :--- | :--- |
| **자원 공유** | 독립적 (공유하려면 IPC 필요) | 프로세스 자원 공유 (Code/Data/Heap) |
| **생성 비용** | 높음 | 낮음 |
| **문맥 교환** | 무거움 (캐시 플러시 발생) | 가벼움 |
| **안정성** | 높음 (하나가 죽어도 영향 없음) | 낮음 (하나가 죽으면 프로세스 전체 영향) |

---

### Ⅳ. 실무 적용 및 기술사적 판단
실무적으로 **Context Switch Overhead**는 시스템 성능의 적이다. 기술사는 과도한 스레드 생성을 방지하기 위해 스레드 풀(Thread Pool)을 활용하고, CPU 친화성(CPU Affinity) 설정을 통해 캐시 적중률을 높이는 전략을 수립해야 한다.

---

### Ⅴ. 기대효과 및 결론
프로세스와 스레드는 현대 고성능 서버 아키텍처의 근간이다. 향후 서버리스 컴퓨팅 환경에서는 더욱 가벼운 실행 단위인 'Isolate'나 'Function' 단위의 관리가 핵심이 될 것이다.
