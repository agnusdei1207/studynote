# 프로세스 간 통신 (IPC, Inter-Process Communication)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IPC는 **"프로세스** 간 **데이터**를 **교환**하는 **메커니즘\\\"**으로, **공유 메모리**(Shared Memory)와 **메시지 전달**(Message Passing) **방식**이 **있고 **커널** 개입 **여부**에 **따라 **성능**이 **달라**진다.
> 2. **공유 메모리**: 빠르지만 **동기화**가 **필요**하고 **커널** 개입이 **없어 **User Mode**에서 **동작**하며 **메시지 전달**은 **안전**하지만 **시스템 콜** **오버헤드**가 **있다**.
> 3. **IPC 종류**: **Pipe**(단방향), **FIFO**(양방향), **Message Queue**(비동기), **Shared Memory**(가장 빠름), **Socket**(네트워크), **Signal**(이벤트)가 **있고 **용도**에 **따라 **선택**한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
IPC는 **"프로세스 간 대화 수단"**이다.

**IPC 필요성**:
| 상황 | 설명 | IPC 유형 |
|------|------|----------|
| **데이터 공유** | 부모-자식 간 데이터 전송 | Pipe, Shared Memory |
| **동기화** | 작업 완료 알림 | Signal, Semaphore |
| **제어** | 프로세스 제어 | Signal |
| **네트워크** | 원격 프로세스 통신 | Socket |

### 💡 비유
IPC는 **사람 간 커뮤니케이션**과 같다.
- **공유 메모리**: 화이트보드 공유
- **메시지 전달**: 쪽지/이메일
- **시그널**: 알림/신호

## Ⅱ. 아키텍처 및 핵심 원리

### IPC 분류

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         IPC Classification                                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Communication Methods:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Method              │  Kernel  │  Direction  │  Speed   │  Data Size   │  Use Case   │  │
    │  ─────────────────────────────────────────────────────────────────────────────────────│  │
    │  Pipe                │  Yes     │  One-way    │  Medium  │  Small       │  Parent-Child│  │
    │  FIFO (Named Pipe)   │  Yes     │  Bi-dir     │  Medium  │  Small       │  Unrelated  │  │
    │  Message Queue       │  Yes     │  Bi-dir     │  Slow    │  Medium      │  Async      │  │
    │  Shared Memory       │  Setup   │  Bi-dir     │  Fastest │  Large       │  High-perf  │  │
    │  Socket              │  Yes     │  Bi-dir     │  Slow    │  Any         │  Network    │  │
    │  Signal              │  Yes     │  One-way    │  Fast    │  None        │  Notification│  │
    │  UNIX Domain Socket  │  Yes     │  Bi-dir     │  Fast    │  Medium      │  Local      │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 파이프 (Pipe)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Anonymous Pipe (익명 파이프)                                           │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Structure: One-way communication
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Parent Process                             Child Process                               │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  fd[1] (write) ──┐                                                                 │  │  │
    │  │                  │    ┌─────────────┐                                               │  │  │
    │  │                  └───→│    Pipe     ├──→ fd[0] (read)                              │  │  │
    │  │                       │ (Kernel)    │                                               │  │  │
    │  │                       └─────────────┘                                               │  │  │
    │  │                                                                                       │  │  │
    │  │  Example:                                                                             │  │  │
    │  │  pipe(fd)  // fd[0]=read, fd[1]=write                                               │  │  │
    │  │  fork()     // Child inherits file descriptors                                       │  │  │
    │  │  parent: close(fd[0]), write(fd[1], "hello", 5)                                      │  │  │
    │  │  child:  close(fd[1]), read(fd[0], buf, 5)                                           │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Limitations:
    • One-way (Half-duplex) - need 2 pipes for bidirectional
    • Only related processes (parent-child)
    • Fixed size buffer (typically 64KB)
    • Data is stream-oriented (no message boundaries)
```

### FIFO (Named Pipe)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         FIFO (Named Pipe)                                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Structure: Filesystem entry for unrelated processes
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Process A                                Process B                                     │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  mkfifo("/tmp/myfifo", 0666)                                                       │  │  │
    │  │  fd = open("/tmp/myfifo", O_WRONLY)                                                │  │  │
    │  │  write(fd, "data", 4)                                                               │  │  │
    │  │                        │                                                           │  │  │
    │  │                        ▼                                                           │  │  │
    │  │                 /tmp/myfifo (Filesystem entry)                                      │  │  │
    │  │                        │                                                           │  │  │
    │  │                        ▼                                                           │  │  │
    │  │  fd = open("/tmp/myfifo", O_RDONLY)                                                │  │  │
    │  │  read(fd, buf, 4)                                                                  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Advantages over Pipe:
    • Unrelated processes can communicate
    • Persistent (exists until deleted)
    • Has filesystem entry (ls -l shows 'p' type)
```

### 메시지 큐 (Message Queue)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Message Queue (System V / POSIX)                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Structure: Kernel-managed queue with message boundaries
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Process A                                Process B                                     │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  msgid = msgget(key, IPC_CREAT)                                                    │  │  │
    │  │  msgsnd(msgid, &msg, sizeof(msg), 0)  ──┐                                          │  │  │
    │  │                                       │  Kernel Queue                              │  │  │
    │  │                                       ▼                                            │  │  │
    │  │                              ┌──────────────────┐                                   │  │  │
    │  │                              │ [msg1] [msg2]    │                                   │  │  │
    │  │                              │ [msg3]           │                                   │  │  │
    │  │                              └──────────────────┘                                   │  │  │
    │  │                                       │                                            │  │  │
    │  │  msgrcv(msgid, &msg, sizeof(msg), 0, 0) ←───┘                                    │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Features:
    • Message boundaries preserved (unlike pipes)
    • Priority-based receiving
    • Asynchronous (non-blocking)
    • Kernel persists until deleted or system reboot

    System V vs POSIX:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  System V: msgget, msgsnd, msgrcv, msgctl                                                 │  │
    │  POSIX:   mq_open, mq_send, mq_receive, mq_close (file descriptor based)                │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 공유 메모리 (Shared Memory)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Shared Memory (Fastest IPC)                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Structure: Direct memory access between processes
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Process A                                Process B                                     │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  shmid = shmget(key, size, IPC_CREAT)                                              │  │  │
    │  │  ptr = shmat(shmid, NULL, 0)                                                       │  │  │
    │  │                                                                                       │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐ │  │  │
    │  │  │  Shared Memory Segment                                                │ │  │  │
    │  │  │  ┌──────────────────────────────────────────────────────────────────────────────┐│ │  │  │
    │  │  │  │  ptr = 0x7f1234000                                                          ││ │  │  │
    │  │  │  │  memcpy(ptr, "data", 4)  ──┐                                                ││ │  │  │
    │  │  │  │                          │  Both can read/write                             ││ │  │  │
    │  │  │  │  memcpy(buf, ptr, 4) ←───┘                                                ││ │  │  │
    │  │  │  └──────────────────────────────────────────────────────────────────────────────┘│ │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘ │  │  │
    │  │  shmdt(ptr)  // Detach                                                                 │  │  │
    │  │  shmctl(shmid, IPC_RMID, NULL)  // Delete                                           │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Advantages:
    • Fastest IPC (no kernel data copying after setup)
    • Large data transfer possible
    • Random access

    Challenges:
    • Synchronization required (semaphore, mutex)
    • No automatic boundaries
    • Security (access control)
```

### 시그널 (Signal)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Signal (Notification Mechanism)                                     │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Standard Signals:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Signal │  Number │  Default Action  │  Purpose                                        │  │
    │  ─────────────────────────────────────────────────────────────────────────────────────│  │
    │  SIGINT │  2      │  Terminate        │  Ctrl+C (interrupt)                            │  │
    │  SIGKILL│  9      │  Terminate        │  Force kill (uncatchable)                      │  │
    │  SIGTERM│  15     │  Terminate        │  Termination request                           │  │
    │  SIGSTOP│  19     │  Stop             │  Suspend (uncatchable)                         │  │
    │  SIGCONT│  18     │  Continue         │  Resume if stopped                             │  │
    │  SIGALRM│  14     │  Terminate        │  Timer expiration                              │  │
    │  SIGCHLD│  17     │  Ignore           │  Child process status change                   │  │
    │  SIGUSR1│  10     │  Terminate        │  User-defined signal 1                         │  │
    │  SIGUSR2│  12     │  Terminate        │  User-defined signal 2                         │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Signal Handling:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. Default: OS-defined action (terminate, stop, ignore)                                 │  │
    │  2. Ignore: signal() - explicitly ignore (SIGKILL, SIGSTOP cannot be ignored)            │  │
    │  3. Handler: signal() or sigaction() - custom function                                   │  │
    │                                                                                         │  │
    │  Example:                                                                               │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  void handler(int sig) {                                                             │  │  │
    │  │      printf("Received signal %d\n", sig);                                            │  │  │
    │  │  }                                                                                    │  │  │
    │  │  signal(SIGUSR1, handler);  // Register handler                                      │  │  │
    │  │  kill(pid, SIGUSR1);        // Send signal                                          │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### IPC 방식 비교

| IPC 유형 | 속도 | 복잡도 | 데이터 크기 | 동기화 | 커널 개입 |
|----------|------|--------|-------------|---------|-----------|
| **Pipe** | 중간 | 낮음 | 작음 | 불필요 | 항상 |
| **FIFO** | 중간 | 낮음 | 작음 | 불필요 | 항상 |
| **Message Queue** | 느림 | 중간 | 중간 | 불필요 | 항상 |
| **Shared Memory** | 빠름 | 높음 | 큼 | 필수 | 초기만 |
| **Socket** | 느림 | 높음 | any | 불필요 | 항상 |
| **Signal** | 빠름 | 낮음 | 없음 | 불필요 | 항상 |

### IPC 선택 가이드

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         IPC Selection Decision Tree                                           │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Need large data transfer?
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  YES → Shared Memory (with synchronization)                                              │  │
    │  NO  → Continue                                                                          │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Related processes (parent-child)?
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  YES → Pipe (simple, one-way) or FIFO (bidirectional)                                    │  │
    │  NO  → Continue                                                                          │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Network communication?
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  YES → Socket (TCP/UDP)                                                                  │  │
    │  NO  → Continue                                                                          │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Event notification only?
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  YES → Signal (SIGUSR1, SIGUSR2)                                                         │  │
    │  NO  → Continue                                                                          │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Message boundaries + asynchronous?
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  YES → Message Queue (System V or POSIX)                                                 │  │
    │  NO  → FIFO (simpler)                                                                    │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 생산자-소비자 패턴 구현
**상황**: 멀티스레드 웹 서버 로그 처리
**판단**: Message Queue vs Shared Memory

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Log Processing IPC Selection                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Requirements:
    • Multiple producer threads (web workers)
    • Single consumer thread (log writer)
    • Asynchronous (non-blocking writes)
    • Message boundaries required

    Decision: Message Queue (POSIX)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Advantages:                                                                             │  │
    │  • Kernel manages queue (no lost messages)                                              │  │
    │  • Non-blocking writes (mq_send with O_NONBLOCK)                                        │  │
    │  • Priority support (urgent logs)                                                        │  │
    │  • Persistence across process crash (optional)                                          │  │
    │                                                                                         │  │
    │  Structure:                                                                             │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  typedef struct {                                                                    │  │  │
    │  │      int priority;                     // Log level                                  │  │  │
    │  │      time_t timestamp;                // When logged                                │  │  │
    │  │      char message[LOG_MAX_SIZE];      // Log content                                │  │  │
    │  │  } log_message_t;                                                                     │  │  │
    │  │                                                                                       │  │  │
    │  │  // Producer:                                                                         │  │  │
    │  │  mqd_t mq = mq_open("/log_queue", O_WRONLY);                                         │  │  │
    │  │  mq_send(mq, (const char*)&msg, sizeof(msg), msg.priority);                          │  │  │
    │  │                                                                                       │  │  │
    │  │  // Consumer:                                                                         │  │  │
    │  │  mqd_t mq = mq_open("/log_queue", O_RDONLY);                                         │  │  │
    │  │  mq_receive(mq, (char*)&msg, sizeof(msg), NULL);                                     │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Socket IPC (UNIX Domain)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         UNIX Domain Socket (Local IPC)                                       │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Advantages over Network Sockets:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • No network stack overhead (bypass TCP/IP)                                            │  │
    │  • File descriptor based (select/poll/epoll compatible)                                  │  │
    │  • Pass file descriptors (SCM_RIGHTS)                                                   │  │
    │  • Credential passing (PID, UID of sender)                                              │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Example: Client-Server
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Server:                                                                                 │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  int sockfd = socket(AF_UNIX, SOCK_STREAM, 0);                                       │  │  │
    │  │  struct sockaddr_un addr = {.sun_family = AF_UNIX, .sun_path = "/tmp/mysocket"};    │  │  │
    │  │  bind(sockfd, (struct sockaddr*)&addr, sizeof(addr));                                 │  │  │
    │  │  listen(sockfd, 5);                                                                   │  │  │
    │  │  int client = accept(sockfd, NULL, NULL);                                            │  │  │
    │  │  read(client, buf, sizeof(buf));                                                      │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Client:                                                                                 │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  int sockfd = socket(AF_UNIX, SOCK_STREAM, 0);                                       │  │  │
    │  │  struct sockaddr_un addr = {.sun_family = AF_UNIX, .sun_path = "/tmp/mysocket"};    │  │  │
    │  │  connect(sockfd, (struct sockaddr*)&addr, sizeof(addr));                              │  │  │
    │  │  write(sockfd, "hello", 5);                                                           │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### IPC 기대 효과

| 상황 | 최적 IPC | 이유 |
|------|----------|------|
| **단순 데이터 전송** | Pipe | 간단, 효율 |
| **대용량 데이터** | Shared Memory | 빠름 |
| **이벤트 알림** | Signal | 가벼움 |
| **비동기 큐** | Message Queue | 안전 |
| **네트워크** | Socket | 범용 |

### 모범 사례

1. **동기화**: 세마포어와 함께
2. **에러 처리**: 시그널 핸들러
3. **보안**: 권한 검사
4. **성능**: Zero-copy

### 미래 전망

1. **RDMA**: User-space 네트워킹
2. **io_uring**: 비동기 IPC
3. **Shared FD**: 메모리 공유
4. **eBPF**: 커널-less 통신

### ※ 참고 표준/가이드
- **POSIX**: IPC 표준
- **System V**: 전통 IPC
- **Linux**: ipc 파이프라인
- **UNIX**: 도메인 소켓
