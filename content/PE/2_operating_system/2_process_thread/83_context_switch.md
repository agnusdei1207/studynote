# 문맥 교환 (Context Switch)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 문맥 교환은 **"실행 중인 프로세스의 상태(Context)를 저장하고, 다른 프로세스의 상태를 복원하여 CPU를 이전하는 과정"**으로, **PCB(Process Control Block)**에 **레지스터, PC, SP** 등의 하드웨어 문맥을 저장/복원하는 OS의 핵심 메커니즘이다.
> 2. **가치**: **CPU 스케줄링(CPU Scheduling)**을 통한 **시분할(Time Sharing)** 시스템 구현의 기반으로, 한 개의 CPU로 여러 프로세스를 **동시에(concurrent)** 실행되는 것처럼 보이게 하며, **응답 시간(Response Time)**을 최적화한다.
> 3. **융합**: **하이퍼바이저(Hypervisor)**의 **vCPU 문맥 교환**, **컨테이너(Container)**의 **cgroup** 스케줄링, **코루틴(Coroutine)**의 **유저레벨 문맥 교환**으로 확장되며, **TLB Flush**, **캐시 Cold** 등의 **간접 비용(Indirect Overhead)**이 성능에 큰 영향을 미친다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
문맥 교환(Context Switch)은 **"CPU가 실행 중인 프로세스에서 다른 프로세스로 전환할 때, 현재 프로세스의 상태를 저장하고 대기 프로세스의 상태를 복원하는 과정"**이다.

**문맥(Context)의 구성**:
- **하드웨어 문맥**: PC(Program Counter), SP(Stack Pointer), 범용 레지스터
- **소프트웨어 문맥**: PCB의 프로세스 상태, 스케줄링 정보

### 💡 비유
문맥 교환은 **"책갈피"**와 같다.
- **책 읽기(프로세스 실행)**: 페이지 읽다가
- **책갈피 끼우기(Context 저장)**: 어까지 읽었는지 표시
- **다른 책 열기(Context 복원)**: 책갈피 위치부터 다시 읽기

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         문맥 교환의 필요성                                              │
└─────────────────────────────────────────────────────────────────────────────────────────┘

단일 프로그래밍 (Batch System):
    • 한 프로세스가 CPU를 독점
    • I/O 대기 시 CPU 유휴
    • 활용률 낮음
         ↓
다중 프로그래밍 (1960년대):
    • 여러 프로세스 메모리에 상주
    • I/O 대기 시 다른 프로세스 실행
    • 문맥 교환 필요
         ↓
시분할 시스템 (Time Sharing, 1970년대):
    • 각 프로세스에 Time Slice 할당
    • 빈번한 문맥 교환
    • 사용자에게 동시 실행 제공
         ↓
현대 OS:
    • 스레드 단위 문맥 교환
    • SMP(Symmetric Multi-Processing)
    • Hyper-Threading, SMT
```

## Ⅱ. 아키텍처 및 핵심 원리

### 문맥 교환 절차

```

│                         문맥 교환 5단계 절차                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [현재 실행: 프로세스 A]                              [대기 중: 프로세스 B]

    ┌─────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. 타이머 인터럽트 또는 시스템 콜 발생                                                │

    │     • 프로그램 카운터 (PC/EIP): 다음 실행 주소                                          │

    │     │  │  │  │  • SP = 0x7FFFFFDC                                           │  │  │  │   │

    │     │  └──────────────────────────────────────────────────────────────────────────┘  │   │
    │     └────────────────────────────────────────────────────────────────────────────────┘   │

                                          │
                                          ▼
    ┌─────────────────────────────────────────────────────────────────────────────────────────┐
    │  3. PCB 업데이트 및 스케줄러 호출                                                      │
    │     • PCB_A.state = Ready                                                             │
```

### 프로세스 vs 스레드 문맥 교환

| 구분 | 프로세스 문맥 교환 | 스레드 문맥 교환 |
|------|-------------------|-----------------|
| **대상** | 프로세스 간 | 동일 프로세스 내 스레드 간 |
| **주소 공간** | 새로운 페이지 테이블 | 동일 페이지 테이블 |
| **TLB** | Flush 필요 | Flush 불필요 |
| **캐시** | Cold (Cold Cache) | Warm (Warm Cache) |
| **오버헤드** | 큼 (~수십 µs) | 작음 (~수 µs) |

### 문맥 교환 비용 분석

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         문맥 교환 비용 구성                                             │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [직접 비용 (Direct Cost)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. 레지스터 저장/복원: ~100-200 사이클                                                │
    │  2. PCB 업데이트: ~50 사이클                                                           │
    │  3. 스케줄러 실행: ~200-500 사이클                                                      │
    │  4. 총 직접 비용: ~1-10 µs (CPU 클럭 의존)                                              │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [간접 비용 (Indirect Cost)]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. TLB Flush                                                                          │
    │     • 프로세스 간 교환 시: 페이지 테이블 변경 → TLB 무효화                               │
    │     • TLB Miss 발생: 메모리 접근 ~100-200 사이클 추가                                   │
    │                                                                                         │
    │  2. 캐시 Cold (Cache Cold Start)                                                       │
    │     • L1/L2/L3 캐시: 이전 프로세스 데이터                                               │
    │     • Cache Miss: 메모리에서 다시 로드 (~10-100 사이클)                                  │
    │                                                                                         │
    │  3. 파이프라인 플러시 (Pipeline Flush)                                                   │
    │     • 명령어 파이프라인: 진행 중인 명령어 취소                                           │
    │     • 분기 예측 실패: 파이프라인 재충전                                                   │
    │                                                                                         │
    │  4. 총 간접 비용: 수천~수만 사이클 (직접 비용의 수배)                                     │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [총 비용]

    프로세스 간 문맥 교환: ~10-100 µs
    스레드 간 문맥 교환: ~1-10 µs
```

## Ⅲ. 융합 비교 및 다각도 분석

### 커널레벨 vs 유저레벨 문맥 교환

| 구분 | 커널레벨 | 유저레벨 |
|------|----------|----------|
| **주체** | OS 스케줄러 | 라이브러리/런타임 |
| **대상** | 프로세스, 스레드 | 코루틴, 파이버 |
| **권한** | 커널 모드 필요 | 유저 모드 |
| **오버헤드** | 큼 | 작음 |
| **예시** | pthread, 스레드 | goroutine, async/await |

### 과목 융합 관점 분석

#### 1. 컴퓨터 구조 ↔ 문맥 교환
- **파이프라인**: 문맥 교환 시 플러시 필요
- **분기 예측**: State reset
- **캐시 일관성**: Cache coherency protocol

#### 2. 가상화 ↔ 문맥 교환
- **VM Exit**: Guest OS → Host OS 전환
- **vCPU 스케줄링**: Hypervisor가 관리
- **Nested Virtualization**: 2중 문맥 교환

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 고빈도 문맥 교환 최적화
**상황**: 1초에 10,000번 이상의 문맥 교환 발생
**판단**:
1. **원인 파악**:
   - Time Slice 너무 짧음 (예: 1ms)
   - 너무 많은 활성 프로세스
   - 불필요한 I/O 대기
2. **최적화**:
   - Time Slice 증가 (10ms → 50ms)
   - 스레드 풀(Thread Pool) 사용
   - 비동기 I/O로 대기 시간 감소
3. **모니터링**:
   - `vmstat`, `pidstat`로 context switch 수 확인

## Ⅴ. 기대효과 및 결론

### 문맥 교환 최적화 기대 효과

| 최적화 방법 | 기대 효과 | 부작용 |
|------------|----------|--------|
| **Time Slice 증가** | 문맥 교환 감소 | 응답 시간 증가 |
| **스레드 사용** | 오버헤드 감소 | 동기화 복잡 |
| **CPU Affinity** | 캐시 활용 | 로드 밸런싱 저하 |
| **코루틴** | 유저레벨 전환 | 스택 관리 |

### 미래 전망

1. **Hardware Support**: 가상화 전용 명령어
2. **eBPF**: 커널 내 유저레벨 로직
3. **User-Level Scheduling**: goroutine, green thread

- **POSIX 1003.1**: pthread scheduling
