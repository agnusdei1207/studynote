# 프로세스 제어 블록 (PCB: Process Control Block)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PCB는 **"운영체제가 각 프로세스를 관리하기 위한 데이터 구조"**로, **프로세스 식별자(PID)**, **프로세스 상태(State)**, **프로그램 카운터(Program Counter)**, **레지스터 저장 영역** 등을 포함하며, 문맥 교환 시 **PCB의 내용이 보관/복원**된다.
> 2. **가치**: **Context Switch**의 핵심 데이터로서 **CPU 레지스터**를 저장하고 **프로세스 상태**(New, Ready, Running, Waiting, Terminated)를 관리하며, **스케줄러**가 **Ready Queue**에서 **다음 실행 프로세스**를 선택할 때 **PCB 정보**를 참조한다.
> 3. **융합**: **Linux task_struct**, **Windows EPROCESS**로 구현되며, **PID(Process ID)**, **PPID(Parent PID)**, **UID(User ID)**, **GID(Group ID)**와 같은 **식별 정보**와 **파일 디스크립터**, **시그널 마스크** 같은 **자원 할당 정보**를 포함한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
PCB(Process Control Block)는 **"운영체제가 프로세스를 관리하기 위한 정보의 집합"**이다.

**PCB의 역할**:
- **식별**: 프로세스 고유 식별자
- **상태**: 실행 상태 추적
- **문맥 보관**: 레지스터, PC 등
- **스케줄링**: 우선순위, 큐 정보

### 💡 비유
PCB는 **"학생 종합 정보 시스템"**와 같다.
- **기본 정보**: 학번, 이름, 학과
- **성적**: 수업 성적
- **상태: 재학, 휴학, 졸업
- **기록**: 출결, 상벌

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         PCB의 발전                                                   │
└─────────────────────────────────────────────────────────────────────────────────────────┘

초기 시스템 (1960년대):
    • Process Control Area (PCA)
    • 단순한 정보만 저장
         ↓
Multics (1965년):
    • PCB 개념 등장
    • 다중 프로그래밍 지원
         ↓
UNIX System V (1980년대):
    • task_struct 구조체
    • 현대 PCB 기반
         ↓
현대 OS (2000년대~):
    • Linux: task_struct
    │  Windows: EPROCESS (Executive Process)
    │  macOS: proc 구조체
```

## Ⅱ. 아키텍처 및 핵심 원리

### PCB 구조

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         PCB 구조 (Linux task_struct)                                     │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    [프로세스 식별 정보]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  pid_t pid;                  │ Process ID                                                   │
    │  pid_t ppid;                 │ Parent Process ID                                            │
    │  uid_t uid;                  │ User ID                                                      │
    │  gid_t gid;                  │ Group ID                                                     │
    │  char comm[16];              │ 실행 파일명                                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [프로세스 상태]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  volatile long state;           │ 상태                                                        │
    │  • TASK_RUNNING (0): 실행 중                                                              │
    │  │  TASK_INTERRUPTIBLE (1): 준비 상태                                                   │
    │  │  TASK_UNINTERRUPTIBLE (2): uninterruptible sleep                                      │
    │  │  TASK_STOPPED (4): 중단됨                                                              │
    │  │  TASK_TRACED (8): 추적 중                                                             │
    │  │  TASK_DEAD (16): 소멸됨                                                              │
    │  │  TASK_ZOMBIE (32): 좀비 프로세스                                                          │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [스케줄링 정보]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  int prio;                    │ 정적 우선순위 (0~139)                                      │
    │  int static_prio;              │ 동적 우선순위                                                  │
    │  struct sched_entity se;        │ 스케줄러 엔티티                                               │
    │  │  struct list_head run_list;     │ Run Queue                                                      │
    │  │  struct list_head tasks;       │ Task List                                                     │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [메모리 관리]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  struct mm_struct *mm;          │ 메모리 관리                                                 │
    │  │  unsigned long start_code;     │ 시작 주소                                                     │
    │  │  unsigned long end_code;       │ 끝 주소                                                       │
    │  │  unsigned long brk;           │ 브레이크 포인터 (Heap)                                         │
    │  │  unsigned long arg_start;     │ 아규먼트 시작                                                │
    │  │  unsigned long arg_end;       │ 아규먼트 끝                                                  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [파일 시스템]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  struct files_struct *files;    │ 열린 파일 목록                                                 │
    │  │  struct fs_struct *fs;         │ 파일 시스템                                                   │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [자원 할당]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  struct signal_struct *pending;    │ 대기 중인 시그널                                              │
    │  struct signal_struct *signal;     │ 실행 중인 시그널                                              │
    │  struct rlimit rlim[RLIMIT_NLIMITS];  │ 자원 제한                                                     │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### Linux vs Windows PCB

| 항목 | Linux (task_struct) | Windows (EPROCESS) |
|------|-------------------|-----------------|
| **식별자** | PID, TGID | PID, UniqueProcessId |
| **상태** | 상태 플래그 | State |
| **부모** | PPID | ParentProcessId |
| **메모리** | mm_struct | VadRoot, Peb |
| **파일** | files_struct | FileObjectTable |

### 과목 융합 관점 분석

#### 1. 스케줄러 ↔ PCB
- **Ready Queue**: run_list
- **PID 할당**: fork() 시 새 PID 생성
- **부모-자식 관계**: PPID, Child List

#### 2 가상 메모리 ↔ PCB
- **PGD/Page Table**: mm_struct
- **Brk/Arg Stack**: 스택 포인터

## Ⅳ. 실무 적용 및 기술사항적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: PCB 정보 확인
**상황**: 프로세스 상태 모니터링
**판단**:

```bash
# /proc/[pid]/status 확인
cat /proc/1234/status

# ps 명령으로 PCB 정보 확인
ps -eo pid,ppid,state,cmd

# top으로 실시간 상태 확인
top -H -p 1234
```

## Ⅴ. 기대효과 및 결론

### PCB의 기대 효과

| 효과 | PCB 없을 시 | PCB 있을 시 |
|------|----------|-----------|
| **프로세스 식별** | 불가능 | PID/PPID |
| **상태 추적** | 어려움 | state 필드 |
| **문맥 교환** | 불가능 | 저장/복원 |
| **자원 관리** | 불가능 | files_struct |

### 미래 전망

1. **eBPF**: 동적으로 PCB 정보 추적
2. **Container**: Namespace 격리
3. **Microkernel**: 최소 PCB

### ※ 참고 표준/가이드
- **Linux Kernel**: include/linux/sched.h
- **Windows**: EPROCESS structure
- **POSIX**: Process Environment
