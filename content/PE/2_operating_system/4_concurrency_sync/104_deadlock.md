# 데드락 (Deadlock)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데드락은 **"두 개 **이상의 **프로세스**가 **서로가 **가진 **리소스**를 **기다리**며 **무한히 **대기**하는 **상태"**로, **4가지 **필요 **조건**(상호 배제, 점유 대기, 비선점, 순환 대기)이 **모두 **충족**될 때 **발생**한다.
> 2. **데드락 처리**: **Prevention**(예방), **Avoidance**(회피, Banker's Algorithm), **Detection & Recovery**(탐지 및 복구)로 **대응**하며 **리소스 **할당 **순서**(Ordered Locking)로 **순환 대기**를 **방지**할 **수 있다.
> 3. **융합**: **OS**의 **Mutex**, **Semaphore**, **Monitor**에서 **발생**하며 **Database**(Lock)의 **Deadlock**도 **동일**한 **원리**이고 **Wait-Die**, **Wound-Wait** **기법**으로 **분산 **시스템의 **데드락**을 **해결**한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
데드락은 **"프로세스가 서로를 기다리며 멈춘 상태"**이다.

**데드락 4가지 필요 조건**:
1. **상호 배제**: 리소스는 공유 불가
2. **점유 대기**: 일부 가진 상태로 대기
3. **비선점**: 리소스를 강제로 뺏을 수 없음
4. **순환 대기**: 대기 체인이 원형을 이룸

### 💡 비유
데드락은 **"교차로 정체****와 같다.
- **차량**: 프로세스
- **교차로**: 리소스
- **양보**: 리소스 해제
- **교착**: 데드락

### Banker's Algorithm

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Banker's Algorithm (Safe State Check)                              │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Data Structures:                                                                       │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Available[3] = {3, 3, 2}     // 가용 리소스                                        │  │  │
    │  │                                                                                       │  │  │
    │  │  Max[5][3]                  // 최대 요청                                            │  │  │
    │  │  Allocation[5][3]            // 현재 할당                                            │  │  │
    │  │  Need[5][3] = Max - Allocation // 필요 리소스                                        │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │  │
    │  Safety Check Algorithm:                                                                 │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Work = Available                                                                     │  │  │
    │  │  Finish[i] = false for all i                                                         │  │  │
    │  │                                                                                       │  │  │
    │  │  repeat:                                                                               │  │  │
    │  │    find i such that (Finish[i] == false && Need[i] ≤ Work)                            │  │  │
    │  │    if found:                                                                          │  │  │
    │  │       Work += Allocation[i]   // 프로세스 완료 후 리소스 반환                          │  │  │
    │  │       Finish[i] = true                                                                │  │  │
    │  │    else:                                                                               │  │  │
    │  │       break  // No safe sequence found → UNSAFE                                      │  │  │
    │  │                                                                                       │  │  │
    │  │  if all Finish[i] == true → SAFE                                                     │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 데드락 해결 전략

| 전략 | 방법 | 장점 | 단점 |
|------|------|------|------|
| **Prevention** | 조건 제거 | 근본적 | 제약 |
| **Avoidance** | 안전 상태 | 유연 | 오버헤드 |
| **Detection** | 탐지 후 회복 | 자유 | 손실 |
| **Ignorance** | Ostrich | 간단 | 재발 |

### 대처 기법

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Deadlock Prevention Techniques                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. Lock Ordering (순서 잠금)                                                            │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  • 모든 프로세스가 동일한 순서로 Lock 획득                                             │  │  │
    │  │  │  → 순환 대기 불가능                                                                │  │  │
    │  │                                                                                       │  │  │
    │  │  void transaction() {                                                                 │  │  │
    │  │      lock(&mutex1);  // 항상 먼저                                                     │  │  │
    │  │      lock(&mutex2);                                                                   │  │  │
    │  │      // critical section                                                             │  │  │
    │  │      unlock(&mutex2);                                                                 │  │  │
    │  │      unlock(&mutex1);  // 역순 해제                                                    │  │  │
    │  │  }                                                                                    │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                                                                         │  │
    │  2. Try Lock (Timeout)                                                                  │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  if (try_lock(&mutex2)) {                                                             │  │  │
    │  │      // got both locks                                                                │  │  │
    │  │  } else {                                                                             │  │  │
    │  │      unlock(&mutex1);  // rollback                                                    │  │  │
    │  │      retry();                                                                         │  │  │
    │  │  }                                                                                    │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: DB 데드락
**상황**: 트랜잭션 교착
**판단**:

```sql
-- Transaction 1
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- waits for account 2

-- Transaction 2
BEGIN;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
-- waits for account 1

-- Solution: Same order
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

### 데드락 기대 효과

| 효과 | 대응 없음 | 대응 있음 |

1. **Lock 순서**: 일관된 순서

1. **Lock-free**: CAS, Atomic
2. **Actor Model**: 메시지 전달
