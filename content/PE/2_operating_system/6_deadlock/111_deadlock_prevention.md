# 데드락 예방 (Deadlock Prevention)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데드락 예방은 **"데드락 **발생 **가능성**을 **사전**에 **차단**하는 **기법"**으로, **4가지 **필요 **조건**(상호 **배제, **점유 **대기, **비선점, **순환 **대기) 중 **하나를 **제거**한다.
> 2. **기법**: **Resource Ordering**(리소스 **순서화), **Hold and Wait**(모든 **자원 **선 **요청), **No Preemption**(선점 **허용), **Circular Wait**(순환 **대기 **방지)로 **예방**한다.
> 3. **회피**: **Banker's Algorithm**으로 **안전 **상태**(Safe State)만 **허용**하며 **Available**(가용 **자원), **Max**(최대 **요구), **Allocation**(현재 **할당), **Need**(필요 **자원) **행렬**로 **판단**한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
데드락 예방은 **"교착 상태 사전 차단"**이다.

**4가지 필요 조건**:
1. **Mutual Exclusion**: 자원은 한 번에 하나의 프로세스만 사용
2. **Hold and Wait**: 일부 자원을 보유하며 다른 자원 대기
3. **No Preemption**: 자원을 강제로 뺏을 수 없음
4. **Circular Wait**: 대기 순환 형성

### 💡 비유
데드락은 ****교차로 ****4량 **교통 ****체증 ****과 같다.
- **모두가 서로**: 순환 대기
- **양보 불가**: 비선점
- **해결**: 신호등 (자원 순서)

## Ⅱ. 아키텍처 및 핵심 원리

### 데드락 예방 기법

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Deadlock Prevention Strategies                                        │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    1. Eliminate Mutual Exclusion (Not Practical):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  → Most resources cannot be shared                                                     │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    2. Eliminate Hold and Wait:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Approach A: Request all resources at once                                               │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  • Process requests all resources before starting                                    │  │  │
    │  │  • Waits until all available                                                         │  │  │
    │  │  → Low resource utilization                                                          │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │  Approach B: Release resources before requesting new ones                                │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  • Temporarily release held resources                                                │  │  │
    │  │  • Request new resources                                                             │  │  │
    │  │  → May lose progress                                                                  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    3. Allow Preemption:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  If process holds resource R1 and requests R2:                                     │  │  │
    │  │    → If R2 not available, preempt R1 (release it)                                  │  │  │
    │  │    → Process waits with no resources                                                │  │  │
    │  │  → Works for resources where state can be saved/restored (e.g., CPU, memory)        │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    4. Eliminate Circular Wait:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Resource Ordering Approach:                                                            │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Assign global ordering to resources: R1 < R2 < R3                                    │  │  │
    │  │  Rule: Process can request Ri only after releasing all Rj where j > i              │  │  │
    │  │                                                                                       │  │  │
    │  │  Example:                                                                              │  │  │
    │  │  P1: Request R1 → OK → Request R3 → Must release R1 first!                           │  │  │
    │  │  P2: Request R2 → OK → Request R1 → Must release R2 first!                           │  │  │
    │  │  → No circular wait possible                                                          │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

    │  │  │    P4 completes, returns [0, 0, 2] → Available [5, 4, 5]                          │  │  │
    │  │  │  Try P0: Need [7, 4, 3] ≤ Available [5, 4, 5] ✗                                   │  │  │

    │  │  │    P1 completes → Available [7, 6, 7]                                              │  │  │
    │  │  │  Try P0: Need [7, 4, 3] ≤ Available [7, 6, 7] ✓                                   │  │  │
    │  │  │    P0 completes → Safe!                                                             │  │  │

    │  │  → Request granted (safe state)                                                        │  │  │

## Ⅳ. 실무 적용 및 기술사적 판단

### 리소스 순서화 (Circular Wait 제거)

```c
// Resource ordering with global lock order
#define LOCK_ORDER_MUTEX 1
#define LOCK_ORDER_FILE 2

void operation() {
    pthread_mutex_t *locks[2] = {&file_mutex, &mutex};
    int order[2] = {LOCK_ORDER_FILE, LOCK_ORDER_MUTEX};

    // Sort by lock order
    for (int i = 0; i < 2; i++) {
        for (int j = i + 1; j < 2; j++) {
            if (order[i] > order[j]) {
                int t = order[i]; order[i] = order[j]; order[j] = t;
                void *tp = locks[i]; locks[i] = locks[j]; locks[j] = tp;
            }
        }
    }

    // Acquire in order
    pthread_mutex_lock(locks[0]);
    pthread_mutex_lock(locks[1]);

    // Critical section

    pthread_mutex_unlock(locks[1]);
    pthread_mutex_unlock(locks[0]);
}
```

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 데이터베이스 트랜잭션
**상황**: 다중 테이블 업데이트
**판단**: 테이블 순서 lock

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Database Lock Ordering                                            │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Tables: users (id=1), orders (id=2), products (id=3)                                    │  │
    │                                                                                         │  │
    │  Transaction 1: Update users → Update orders                                             │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  BEGIN;                                                                                │  │  │
    │  │  SELECT * FROM users WHERE id = 1 FOR UPDATE;   -- Lock users (id=1)                │  │  │
    │  │  SELECT * FROM orders WHERE id = 100 FOR UPDATE;  -- Lock orders (id=2)            │  │  │
    │  │  UPDATE users SET ...;                                                              │  │  │
    │  │  UPDATE orders SET ...;                                                             │  │  │
    │  │  COMMIT;                                                                              │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Transaction 2: Update orders → Update products                                           │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  BEGIN;                                                                                │  │  │
    │  │  SELECT * FROM orders WHERE id = 200 FOR UPDATE;  -- Lock orders (id=2)            │  │  │
    │  │  -- Wait for Transaction 1 to release orders                                         │  │  │
    │  │  SELECT * FROM products WHERE id = 300 FOR UPDATE; -- Lock products (id=3)          │  │  │
    │  │  UPDATE orders SET ...;                                                             │  │  │
    │  │  UPDATE products SET ...;                                                           │  │  │
    │  │  COMMIT;                                                                              │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  → No deadlock: Always lock in table order (1 → 2 → 3)                                  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```
