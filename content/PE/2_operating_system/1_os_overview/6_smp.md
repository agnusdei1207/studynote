# 대칭 다중 처리 (SMP)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SMP는 모든 프로세서가 대등한 권한을 가지고 하나의 공유 메모리와 운영체제 커널을 공동으로 사용하는 멀티프로세서 아키텍처로, 현대 대부분의 컴퓨터와 서버가 채택하는 방식이다.
> 2. **가치**: 특정 CPU에 업무가 집중되는 병목 현상을 해결하여 뛰어난 확장성(Scalability)을 제공하며, 각 CPU가 독립적으로 스케줄링을 수행하여 시스템 효율을 극대화한다.
> 3. **융합**: 고도의 자원 공유로 인한 'Race Condition' 방지를 위해 스핀락(Spinlock), 세마포어와 같은 동기화 기술 및 캐시 일관성 프로토콜이 필수적으로 결합된다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념
**대칭 다중 처리(SMP, Symmetric Multiprocessing)**는 두 개 이상의 프로세서가 동등한 자격을 가지고 하나의 운영체제에 의해 제어되며, 공통의 메인 메모리와 입출력 장치를 공유하여 병렬로 작업을 처리하는 시스템 아키텍처이다.

#### 💡 비유
SMP는 **"모든 요리사가 주방장 권한을 가진 공동 주방"**과 같다.
1. **특징**: 모든 요리사가 메뉴를 정할 수 있고, 재료를 꺼낼 수 있으며, 빈 화구가 있으면 누구든 먼저 가서 요리를 시작한다.
2. **이점**: 주방장 한 명의 지시를 기다릴 필요가 없어 일이 매우 빠르다.
3. **주의사항**: 두 요리사가 동시에 같은 칼을 잡으려 하거나, 같은 냄비에 소금을 두 번 넣지 않도록 "지금 이 칼은 내가 쓰는 중이야!"라고 말하는 규칙(동기화)이 매우 중요하다.

#### 등장 배경 및 발전 과정
1. **ASMP의 한계 극복**: Master CPU에만 커널 처리가 집중되어 발생하는 성능 병목을 해결하고, 모든 CPU의 성능을 100% 활용하기 위해 등장했다.
2. **소프트웨어 기술의 발전**: 여러 CPU가 동시에 커널 코드를 실행해도 데이터가 꼬이지 않게 하는 'Reentrant Kernel'과 'Fine-grained Locking' 기술이 발전하면서 대중화되었다.
3. **현재**: 우리가 사용하는 인텔 Core 시리즈, AMD Ryzen, Apple 실리콘, 그리고 모든 클라우드 서버의 멀티코어 환경은 기본적으로 SMP 원칙에 따라 동작한다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 관련 기술 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **Identical CPUs** | 동등한 성능과 권한의 프로세서들 | 개별적으로 커널 코드 실행 가능 | Multi-core, Hyper-Threading | 동등한 실력의 요리사들 |
| **Shared Main Memory** | 단일 주소 공간 공유 | 모든 CPU가 동일한 메모리 맵 참조 | UMA (Uniform Memory Access) | 공용 식재료 창고 |
| **Symmetric OS** | 모든 CPU에서 동작하는 커널 | 재진입 가능(Reentrant) 구조 | Multi-threading Kernel | 공용 레시피 북 |
| **Locking Mechanism** | 공유 자원 독점권 제어 | 임계 구역(Critical Section) 보호 | Spinlock, Mutex, Semaphore | "사용 중" 표지판 |
| **Cache Coherency** | 프로세서별 캐시 데이터 일관성 | 데이터 수정 시 타 캐시 무효화 | MESI Protocol | 공지사항 실시간 동기화 |

#### 정교한 구조 다이어그램 (ASCII)

```text
       ┌─────────────────────────────────────────────────────────┐
       │                 Operating System (Kernel)               │
       │    (Shared by all CPUs, Reentrant, Multi-threaded)      │
       └────────────────────────────┬────────────────────────────┘
                                    │
           ┌────────────────────────┼────────────────────────┐
           ▼                        ▼                        ▼
    ┌─────────────┐          ┌─────────────┐          ┌─────────────┐
    │    CPU 0    │          │    CPU 1    │          │    CPU N    │
    │ (Self-Sched)│          │ (Self-Sched)│          │ (Self-Sched)│
    └──────┬──────┘          └──────┬──────┘          └──────┬──────┘
           │                        │                        │
    ───────┴──────────┬─────────────┴───────────┬────────────┴───────
                      │                         │
              ┌───────┴───────┐         ┌───────┴───────┐
              │ Shared Memory │         │ Shared I/O    │
              │ (Single Addr) │         │               │
              └───────────────┘         └───────────────┘

    [핵심 특징]
    - 대칭성: 모든 CPU가 I/O 장치와 메모리에 동등하게 접근
    - 독립성: 각 CPU가 스스로 준비 큐에서 작업을 선택하여 실행
```

#### 심층 동작 원리 (Self-Scheduling & Locking)

1. **자가 스케줄링(Self-Scheduling)**: 특정 Master가 일을 시키는 것이 아니라, 각 CPU가 준비 큐(Ready Queue)를 직접 뒤져서 할 일이 있으면 스스로 가져와서 실행한다.
2. **커널 동기화**: 여러 CPU가 동시에 `malloc()`과 같은 커널 함수를 호출할 수 있다. 이때 메모리 할당 표가 엉키지 않도록 아주 짧은 시간 동안 하드웨어 락(Spinlock)을 사용하여 순차 처리를 보장한다.
3. **인터럽트 분산**: 외부 장치에서 인터럽트가 발생하면 하드웨어(APIC)가 현재 가장 한가한 CPU에게 인터럽트를 전달하여 부하를 분산한다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교

| 항목 | 비대칭 다중 처리 (ASMP) | 대칭 다중 처리 (SMP) | NUMA (비균등 메모리 접근) |
| :--- | :--- | :--- | :--- |
| **권한** | Master/Slave 차등 | **완전 동등** | 동등 (물리적 거리는 차이) |
| **병목 지점** | Master CPU | **공유 버스 / 락(Lock)** | 로컬/원격 메모리 지연 |
| **확장성** | 낮음 | **중간 (코어 수 증가 시 버스 한계)** | 높음 (서버 급) |
| **OS 복잡도** | 낮음 | **높음** | 매우 높음 |

#### 과목 융합 관점 분석
- **데이터베이스 연계**: SMP 환경에서 DB 성능을 높이려면 여러 CPU가 동시에 같은 데이터 페이지를 수정할 때 발생하는 Latch 경합을 최소화하는 설계가 필수적입니다.
- **알고리즘 연계**: 공유 메모리 기반의 병렬 프로그래밍 모델(OpenMP, Pthreads)은 SMP 아키텍처의 성능을 100% 이끌어내기 위한 도구입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 기술사적 판단 (실무 시나리오)
**[시나리오]** 64코어 대형 서버를 도입했는데, CPU 개수에 비례한 성능 향상이 나타나지 않는 경우
1. **분석**:
    - **Global Lock 경합**: 커널이나 애플리케이션에 '거대한 하나의 락(Big Kernel Lock)'이 있어, CPU들이 일을 하기보다 락이 풀리기를 기다리는 데 시간을 다 씀.
    - **Bus Saturation**: 모든 CPU가 하나의 메모리 버스에 매달려 있어 데이터 전송 속도가 한계에 도달함.
2. **대응 전략**:
    - **Fine-grained Locking**: 락의 범위를 잘게 쪼개어 경합 확률을 낮춤.
    - **NUMA 아키텍처로의 전환**: 메모리를 물리적으로 분할하여 버스 부하를 분산.

#### 도입 시 고려사항 (체크리스트)
- [ ] **캐시 일관성 프로토콜**: MESI 등이 하드웨어적으로 잘 지원되는가?
- [ ] **OS 커널 최적화**: 멀티코어 효율을 극대화하도록 설계된 최신 커널인가?
- [ ] **소프트웨어 병렬성**: 애플리케이션이 멀티스레드 기반으로 작성되었는가?

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량적/정성적 기대효과
- **성능 비례 향상**: 워크로드에 따라 CPU 개수 증가에 따른 선형적인 성능 향상 기대.
- **가용성**: 특정 코어 오류 시에도 시스템 중단 없이 지속 운영 가능.

#### 미래 전망 및 진화 방향
- **NUMA와의 융합**: 코어 수가 수백 개로 늘어나면서, 논리적으로는 SMP(공유 메모리)지만 물리적으로는 메모리가 분산된 NUMA 구조가 기본이 되고 있습니다.
- **클라우드 인스턴스**: 가상화 기술을 통해 물리 코어들을 묶어 하나의 큰 SMP 시스템처럼 서비스하는 가상 SMP(vSMP) 기술이 발전하고 있습니다.
