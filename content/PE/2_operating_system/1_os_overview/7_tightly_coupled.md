# 강결합 시스템 (Tightly Coupled System)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 강결합 시스템은 여러 개의 프로세서가 하나의 공통된 메인 메모리와 운영체제를 공유하며, 고속의 시스템 버스를 통해 긴밀하게 연결되어 동작하는 멀티프로세서 아키텍처이다.
> 2. **가치**: 프로세서 간 데이터 공유가 메모리를 통해 직접 이루어지므로 통신 지연(Latency)이 매우 낮고, 시스템 전체가 하나의 자원 풀로 관리되어 작업 분배 효율이 극대화된다.
> 3. **융합**: SMP(대칭 다중 처리), 공유 메모리(Shared Memory) 모델, 캐시 일관성 기술이 집약된 형태이며, 현대의 멀티코어 서버 및 고성능 워크스테이션의 표준 구조이다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념
**강결합 시스템(Tightly Coupled System)**은 프로세서 간의 결합도가 매우 높은 시스템으로, 모든 프로세서가 동일한 주소 공간의 메인 메모리를 공유(Shared Memory)하고 하나의 운영체제에 의해 제어되는 병렬 컴퓨터 시스템이다. 프로세서들은 주로 전용 시스템 버스나 크로스바 스위치를 통해 연결되어 초고속으로 데이터를 주고받는다.

#### 💡 비유
강결합 시스템은 **"한 사무실 안에서 커다란 공용 화이트보드를 함께 쓰는 팀원들"**과 같다.
1. **공유 메모리**: 사무실 한가운데 있는 화이트보드. 누구나 적고 누구나 즉시 볼 수 있다.
2. **고속 통신**: 옆자리 팀원에게 어깨를 툭 치며 말하는 것만큼 소통이 빠르다.
3. **단일 OS**: 팀 전체를 관리하는 팀장님이 한 명이라 모든 업무 분배가 일사불란하다.
4. **단점**: 화이트보드 앞에 너무 많은 사람이 몰리면 서로 부딪히고(버스 병목), 팀장님이 너무 많은 팀원을 관리하기 힘들어질 수 있다(확장성 한계).

#### 등장 배경 및 발전 과정
1. **처리량 증대 요구**: 단일 CPU의 성능 한계를 극복하기 위해 여러 프로세서를 묶어야 했고, 가장 효율적인 방법은 메모리를 공유하는 것이었다.
2. **병렬 프로그래밍의 용이성**: 개발자 입장에서 메모리 주소가 하나인 것이 코딩하기 훨씬 편했기 때문에 강결합 방식이 주류가 되었다.
3. **발전 과정**: 과거에는 거대한 메인프레임 형태였으나, 현재는 하나의 칩 안에 여러 코어가 들어간 **멀티코어(Multi-core) SoC** 형태로 우리 곁에 존재한다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 관련 기술 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **Shared Memory** | 데이터 공유의 중심 매체 | 단일 주소 공간(Global Address Space) | UMA, NUMA | 공용 화이트보드 |
| **System Bus** | 프로세서-메모리 간 고속 통로 | 데이터 및 제어 신호 전달 | QPI, UPI, Mesh | 사무실 복도 |
| **Global OS** | 시스템 전체 자원 통합 관리 | 모든 프로세서 스케줄링 및 I/O 제어 | Symmetric OS | 카리스마 팀장 |
| **Cache Coherence** | 프로세서별 로컬 데이터 일치 | 수정된 데이터 실시간 동기화 | MESI Protocol | 정보 동기화 회의 |
| **Synchronization** | 공유 자원 접근 순서 제어 | 메모리 락 및 원자적 연산 | Spinlock, Mutex | 펜 사용 순서 |

#### 강결합 시스템 구조도 (ASCII)

```text
    ┌─────────────────────────────────────────────────────────┐
    │                 Single Operating System                 │
    └────────────────────────────┬────────────────────────────┘
                                 │
           ┌─────────────────────┼─────────────────────┐
           ▼                     ▼                     ▼
    ┌─────────────┐       ┌─────────────┐       ┌─────────────┐
    │ Processor 1 │       │ Processor 2 │       │ Processor N │
    │ ┌─────────┐ │       │ ┌─────────┐ │       │ ┌─────────┐ │
    │ │ L1 / L2 │ │       │ │ L1 / L2 │ │       │ │ L1 / L2 │ │
    │ └────┬────┘ │       │ └────┬────┘ │       │ └────┬────┘ │
    └──────┼──────┘       └──────┼──────┘       └──────┼──────┘
           │                     │                     │
    ───────┴──────────┬──────────┴──────────┬──────────┴───────
                      │                     │
              ┌───────┴───────┐     ┌───────┴───────┐
              │ Shared Memory │     │  Shared I/O   │
              │(Single Addr)  │     │   Devices     │
              └───────────────┘     └───────────────┘

    [특징]
    - 지연 시간: 매우 낮음 (Nano-seconds)
    - 데이터 공유: 메모리 직접 참조 (매우 빠름)
    - 확장성: 제한적 (수십~수백 코어)
```

#### 심층 동작 원리 (Communication & Management)

1. **메모리 기반 통신**: 한 프로세서가 메모리 주소 0x100에 값을 쓰면, 다른 프로세서가 즉시 그 값을 읽을 수 있다. 별도의 네트워크 프로토콜이 필요 없어 오버헤드가 거의 없다.
2. **단일 OS 제어**: 운영체제는 모든 프로세서를 자원 풀(Resource Pool)로 보고, 준비된 작업을 빈 프로세서에 즉각 할당한다. 이는 부하 균등화(Load Balancing)에 매우 유리하다.
3. **병목 현상(Bottleneck)**: 모든 프로세서가 하나의 메모리 버스나 메모리 모듈에 집중되므로, 프로세서 수가 늘어날수록 메모리 접근 경쟁이 심화되어 효율이 떨어진다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교

| 항목 | 강결합 (Tightly Coupled) | 약결합 (Loosely Coupled) |
| :--- | :--- | :--- |
| **메모리** | **공유 메모리 (Shared)** | 개별 메모리 (Private/Local) |
| **통신 매체** | 시스템 버스 / 백플레인 | 네트워크 (Ethernet, InfiniBand) |
| **통신 속도** | **매우 빠름 (GB/s)** | 상대적 느림 (Mb/s ~ Gb/s) |
| **OS** | **단일 운영체제** | 프로세서별 독립 운영체제 |
| **확장성** | 낮음 (물리적 한계) | **매우 높음 (무한 확장 가능)** |
| **주요 용도** | 서버, 슈퍼컴퓨터 노드, PC | 클러스터, 분산 시스템, 클라우드 |

#### 과목 융합 관점 분석
- **컴퓨터 구조 연계**: 멀티코어 프로세서 내부의 'L3 캐시 공유' 구조는 전형적인 강결합 시스템의 마이크로 버전입니다.
- **데이터베이스 연계**: Oracle RAC(Shared Everything) 구조는 논리적으로 강결합 시스템의 철학을 데이터베이스 계층에 적용한 사례입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 기술사적 판단 (실무 시나리오)
**[시나리오]** 실시간 데이터 분석을 위한 서버 아키텍처 선정
1. **분석**: 데이터 간 연관 관계가 높고 빈번한 공유가 필요함.
2. **선택**: **강결합 시스템(SMP 서버)**이 유리. 프로세서 간 통신 지연이 분석 성능의 핵심이기 때문.
3. **한계**: 하지만 분석 대상 데이터가 페타바이트급으로 커져서 서버 한 대의 메모리에 담을 수 없다면, 강결합 노드들을 약결합으로 묶은 **'하이브리드 분산 시스템'**으로 확장해야 함.

#### 도입 시 고려사항 (체크리스트)
- [ ] **메모리 대역폭**: 늘어난 프로세서들의 요구량을 메모리 버스가 감당할 수 있는가?
- [ ] **캐시 일관성 오버헤드**: 코어 수가 늘어남에 따라 발생하는 일관성 유지 트래픽이 성능을 갉아먹지 않는가?
- [ ] **Single Point of Failure**: 공유 메모리나 OS에 문제가 생기면 시스템 전체가 멈추는 위험(SPOF)에 대비했는가?

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량적/정성적 기대효과
- **최고의 응답 성능**: 프로세서 간 데이터 이동 시간이 거의 제로에 가까움.
- **자원 활용 극대화**: 전체 CPU와 메모리를 하나의 거대한 가상 자원으로 활용.

#### 미래 전망 및 진화 방향
- **NUMA 아키텍처의 보편화**: 강결합의 병목을 줄이기 위해 메모리를 물리적으로 분할하되 논리적으로는 공유하는 NUMA(Non-Uniform Memory Access)가 현대 강결합 시스템의 주류가 됨.
- **CXL(Compute Express Link)**: 서버 외부의 장치(GPU, 메모리 확장기)까지 강결합 수준의 속도로 묶어주는 차세대 인터커넥트 기술로 진화 중.
