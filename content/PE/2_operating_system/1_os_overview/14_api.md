# API (Application Programming Interface)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: API는 응용 프로그램이 운영체제나 라이브러리, 또는 타 서비스의 기능을 호출하여 사용할 수 있도록 정의된 함수, 프로토콜, 도구들의 규격이자 인터페이스다.
> 2. **가치**: 복잡한 내부 구현 로직을 은폐(Encapsulation)하고 표준화된 호출 방식만을 제공함으로써, 개발 생산성을 높이고 시스템 간의 결합도(Coupling)를 낮춘다.
> 3. **융합**: 운영체제의 시스템 호출(System Call)을 감싸는 래퍼(Wrapper) 역할부터, 현대 클라우드 기반의 REST/gRPC 통신 규격까지 소프트웨어 생태계의 연결 고리 역할을 한다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념
**API(Application Programming Interface)**는 소프트웨어 구성 요소들이 서로 소통하기 위한 규칙의 집합이다. 운영체제 관점에서는 프로그래머가 시스템 자원(파일, 메모리, 네트워크)을 직접 건드리지 않고도 안전하게 사용할 수 있도록 제공하는 '함수 주머니'와 같다.

#### 💡 비유
API는 **"식당의 메뉴판"**과 같다.
1. **손님 (응용 프로그램)**: 주방 안에서 어떤 요리 기구가 쓰이는지, 식재료가 어디 있는지 알 필요가 없다.
2. **요리사 (운영체제/커널)**: 손님이 주방에 들어오는 것을 원치 않는다. 위험하고 복잡하기 때문이다.
3. **메뉴판 (API)**: 손님에게 제공 가능한 요리 리스트와 주문 방법(인자값)을 보여준다. 손님은 메뉴판을 보고 주문만 하면 결과물(요리)을 받을 수 있다.
4. **특징**: 메뉴판만 그대로라면 주방장이 바뀌거나 주방 기구를 최신식으로 바꿔도(내부 구현 변경), 손님은 주문 방식을 바꿀 필요가 없다.

#### 등장 배경 및 발전 과정
1. **생산성 향상**: 모든 개발자가 하드웨어 제어 코드를 처음부터 짤 수는 없다. 잘 만들어진 기능을 재사용하려는 요구가 API를 탄생시켰다.
2. **이식성(Portability)**: 소스코드가 다른 OS에서도 돌아가게 하려면 동일한 API 규격(POSIX 등)이 필요했다.
3. **발전 과정**:
    - **Native API**: Windows API(Win32), POSIX API 등 OS 종속적 함수.
    - **Web API**: 인터넷을 통해 데이터를 주고받는 REST, SOAP, gRPC 등 서비스 간 API로 확장.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### API의 계층적 구조 (표)

| 계층 | 명칭 | 상세 역할 | 관련 기술 |
| :--- | :--- | :--- | :--- |
| **Upper** | **High-level API** | 복잡한 기능을 추상화하여 제공. 사용자 친화적 | Python Libraries, UI Frameworks |
| **Middle** | **Runtime API** | 언어 런타임이나 표준 라이브러리 제공 규격 | C Standard Lib (libc), Java API |
| **Lower** | **System API** | OS 커널과 직접 소통하는 최하위 인터페이스 | POSIX, Win32 API |
| **Internal** | **System Call** | API 내부에서 실제로 호출되는 하드웨어 트랩 | INT 0x80, SYSCALL |

#### API 호출과 시스템 호출의 관계도 (ASCII)

```text
    ┌──────────────────────────────────┐
    │      Application Program         │
    │  (printf("Hello") 호출)          │
    └──────────────┬───────────────────┘
                   │
    ┌──────────────▼───────────────────┐
    │      Standard Library API        │ (glibc 등)
    │  [ printf() { ... write(); } ]   │ <── API 영역 (User Mode)
    └──────────────┬───────────────────┘
                   │ Wrapper Call
    ┌──────────────▼───────────────────┐
    │      System Call Interface       │ <── 트랩 발생 (Mode Switch)
    └──────────────┬───────────────────┘
                   │
    ┌──────────────▼───────────────────┐
    │         OS Kernel                │
    │  [ sys_write() { HW Control } ]  │ <── 커널 영역 (Kernel Mode)
    └──────────────────────────────────┘
```

#### 심층 동작 원리 (Abstraction & Interaction)

1. **추상화(Abstraction)**: API는 '무엇(What)'을 할 수 있는지만 알려주고 '어떻게(How)' 하는지는 숨긴다. 예를 들어 `write()` API는 데이터가 하드디스크에 저장되는지 SSD에 저장되는지, 섹터 번호가 무엇인지 개발자가 신경 쓰지 않게 한다.
2. **매개변수 마샬링(Marshalling)**: API 호출 시 전달된 인자들을 운영체제가 이해할 수 있는 형식(레지스터 값 등)으로 변환하여 시스템 호출로 전달한다.
3. **동적 링크(Dynamic Linking)**: 대부분의 API는 실행 파일에 포함되지 않고 실행 시점에 DLL이나 SO 파일 형태로 연결되어, 메모리 낭비를 줄이고 라이브러리만 따로 업데이트할 수 있게 한다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교 (표준 규격)

| 항목 | POSIX API | Win32 API | Java API |
| :--- | :--- | :--- | :--- |
| **대상 OS** | Unix, Linux, macOS | Windows | Platform Independent (JVM) |
| **철학** | "모든 것은 파일이다" | 객체와 메시지 중심 | 객체지향 및 클래스 중심 |
| **이식성** | 매우 높음 (소스 레벨) | 낮음 (Windows 전용) | 매우 높음 (바이트코드 레벨) |
| **예시** | `fork()`, `open()` | `CreateProcess()`, `OpenFile()` | `Thread.start()`, `File.open()` |

#### 과목 융합 관점 분석
- **네트워크 연계**: API 개념이 네트워크로 확장된 것이 REST API입니다. HTTP 메서드(GET, POST)를 API 함수처럼 사용하여 원격지의 자원을 제어합니다.
- **보안 연계**: API는 보안의 '경계선'입니다. API 호출 시점에 사용자 권한을 검사(Authorization)하고, 잘못된 인자값이 들어오는지 확인(Sanitization)하는 것이 시스템 보안의 핵심입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 기술사적 판단 (실무 시나리오)
**[시나리오]** 멀티플랫폼(Windows, Linux) 지원 소프트웨어 개발
1. **문제**: 각 OS의 API가 서로 달라 코드를 두 번 짜야 함.
2. **진단**: Native API에 직접 의존하면 결합도가 너무 높아짐.
3. **해결 전략**:
    - **추상화 계층(HAL)** 도입: 공통 인터페이스를 만들고 OS별로 구현부를 나눔.
    - **표준 API(POSIX)** 준수 라이브러리 사용.
    - **C++ 표준 라이브러리**나 **Java/Go**처럼 가상화된 API 환경 제공 언어 채택.

#### 도입 시 고려사항 (체크리스트)
- [ ] **Backward Compatibility**: API 버전을 올렸을 때 기존 프로그램이 멈추지 않는가?
- [ ] **API Documentation**: 인자값, 반환값, 예외 상황이 명확히 문서화되었는가? (Swagger 등 활용)
- [ ] **Rate Limiting**: (Web API인 경우) 특정 사용자가 API를 무단으로 과다 호출하여 시스템을 마비시키지 않는가?

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량적/정성적 기대효과
- **개발 비용 절감**: 이미 검증된 API를 사용하여 바퀴를 다시 발명하는 수고 절감.
- **시스템 안정성**: 검증된 커널 서비스를 통해 자원을 관리하므로 시스템 가속화 및 오류 감소.

#### 미래 전망 및 진화 방향
- **API First Design**: 서비스를 개발할 때 API 규격을 먼저 정의하고 이를 바탕으로 프론트엔드와 백엔드를 동시 개발하는 패러다임이 정착되었습니다.
- **AI API**: 복잡한 딥러닝 모델을 직접 구축하지 않고 API 호출 한 번으로 이미지 인식, 자연어 처리를 수행하는 서비스가 인프라의 핵심이 되고 있습니다.
