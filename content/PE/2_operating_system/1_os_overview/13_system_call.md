# 시스템 호출 (System Call)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 호출은 응용 프로그램이 운영체제(커널)의 보호된 자원이나 서비스를 이용하기 위해 사용하는 공식적이고 유일한 인터페이스이자 프로그래밍 방식의 진입점이다.
> 2. **가치**: 일반 프로그램이 하드웨어에 직접 접근하는 것을 차단하고 반드시 커널을 거치게 함으로써, 자원 관리의 무결성과 시스템 전체의 안정성을 보장한다.
> 3. **융합**: 하드웨어 트랩(Trap), 듀얼 모드(Dual Mode), 인터럽트 벡터 테이블 기술이 집약된 메커니즘이며, 현대 컴퓨팅 아키텍처의 OS-사용자 간 통신 표준이다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념
**시스템 호출(System Call)**은 실행 중인 프로그램과 운영체제 사이의 인터페이스다. 응용 프로그램은 '사용자 모드'에서 실행되므로 하드웨어 조작이나 타 프로세스 간섭이 금지되어 있다. 따라서 파일 생성, 네트워크 통신, 메모리 할당 등 특권이 필요한 작업을 수행할 때는 운영체제에 도움을 요청해야 하는데, 이때 사용하는 수단이 바로 시스템 호출이다.

#### 💡 비유
시스템 호출은 **"구청의 민원 창구"**와 같다.
1. **상황**: 시민(응용 프로그램)이 집을 짓고 싶다(파일 생성). 하지만 시민이 직접 구청 장부를 고치거나 땅을 파헤칠 수는 없다.
2. **동작**: 시민은 정해진 양식(API)을 작성하여 민원 창구(시스템 호출)에 제출한다.
3. **처리**: 창구 직원(커널)은 서류가 적합한지 확인하고, 시민 대신 땅을 파거나 장부를 정리한 뒤 결과를 알려준다.
4. **특징**: 직원은 시민이 함부로 행정 시스템을 망가뜨리지 못하게 막는 수문장 역할을 한다.

#### 등장 배경 및 발전 과정
1. **자원 보호의 필요성**: 한 프로그램의 실수로 디스크의 모든 데이터가 날아가는 것을 방지하기 위해 '대리 수행' 방식이 도입되었다.
2. **하드웨어 추상화**: 하드웨어 제조사마다 조작법이 다른 문제를 운영체제가 시스템 호출이라는 표준화된 명령어로 감싸서 개발자에게 편의를 제공했다.
3. **현재**: 수백 개의 시스템 호출이 존재하며, 보안을 위해 호출 시마다 엄격한 권한 검사(Access Control)가 병행된다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 주요 구성 요소 및 절차 (표)

| 요소명 | 역할 및 메커니즘 | 관련 기술 |
| :--- | :--- | :--- |
| **System Call Interface** | 가용 시스템 호출 목록을 관리하는 테이블 | System Call Table |
| **System Call Number** | 각 서비스에 부여된 고유 번호 (예: read=3, write=4) | Indexing |
| **Trap (Software Interrupt)** | 모드를 커널 모드로 강제 전환하는 하드웨어 신호 | INT 0x80, SYSCALL |
| **Parameter Passing** | 호출에 필요한 인자(파일 이름 등)를 전달하는 방식 | Register, Stack, Block |
| **Return Value** | 작업 성공 여부 및 결과 데이터 반환 | errno |

#### 시스템 호출 실행 메커니즘 (ASCII)

```text
    [ User Mode ]                      [ Kernel Mode ]
          │                                   │
    1. Call API (e.g., read())                │
          ▼                                   │
    ┌───────────────┐                         │
    │ Standard Lib  │                         │
    │ (glibc / Win) │                         │
    └──────┬────────┘                         │
           │ 2. Set Syscall No. in Register   │
           │ 3. Execute TRAP (Software Int)   │
           └───────────────────┬─────────────►│
                               │              │ 4. Change Mode Bit to 0
                               │              │ 5. Consult Syscall Table
                               │              │ 6. Execute Kernel Code
                               │              │    (File system access)
                               │              │
           ┌───────────────────┴──────────────┤
           │ 8. Change Mode Bit back to 1     │ 7. Operation Finished
           ▼                                  │
    9. Resume User App                        │
```

#### 심층 동작 원리 (The Transition Process)

1. **매개변수 전달**: 커널은 사용자 영역의 메모리를 직접 믿지 않는다. 따라서 인자값(Argument)을 CPU 레지스터에 담아 넘기거나, 양이 많으면 별도의 메모리 블록 주소를 전달한다.
2. **모드 전환(Context Switch)**: '사용자 모드'에서 '커널 모드'로 바뀔 때, 현재 프로그램의 상태(PC, SP 등)를 안전한 곳에 저장한다.
3. **검증(Validation)**: 커널은 요청받은 주소값이 사용자의 영역을 벗어나는지, 해당 파일에 읽기 권한이 있는지 등 보안 검사를 최우선으로 수행한다.
4. **결과 반환**: 작업이 완료되면 결과값(보통 양수) 또는 에러 코드(음수)를 특정 레지스터에 담아 사용자 모드로 복귀한다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교

| 항목 | 시스템 호출 (System Call) | 라이브러리 호출 (Library Call) |
| :--- | :--- | :--- |
| **수행 주체** | **운영체제 커널** | 응용 프로그램 (라이브러리) |
| **모드 전환** | **필수적 (User ↔ Kernel)** | 없음 (User Mode 유지) |
| **처리 속도** | 상대적으로 느림 (오버헤드 발생) | 매우 빠름 |
| **주요 예시** | `open()`, `fork()`, `read()` | `printf()`, `sqrt()`, `strlen()` |
| **특징** | 하드웨어 자원 접근 시 필수 | 편의 기능 제공 (내부에서 Syscall 호출하기도 함) |

#### 과목 융합 관점 분석
- **컴퓨터 구조 연계**: 현대 CPU는 `SYSCALL`(Intel)이나 `SVC`(ARM) 같은 시스템 호출 전용 명령어를 하드웨어적으로 지원하여, 모드 전환 시 발생하는 클럭 낭비를 최소화합니다.
- **보안 연계**: '시스템 호출 필터링(Seccomp)' 기술을 통해 특정 앱이 사용할 수 있는 시스템 호출 종류를 제한하여, 컨테이너나 샌드박스 환경의 보안을 강화합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 기술사적 판단 (실무 시나리오)
**[시나리오]** 대규모 파일 쓰기 작업 시 프로그램 성능이 급격히 저하되는 경우
1. **진단**: 한 바이트를 쓸 때마다 `write()` 시스템 호출을 보냄.
2. **원인**: 시스템 호출은 모드 전환(Context Switch) 오버헤드가 큼. 100만 번의 쓰기를 위해 100만 번의 모드 전환이 일어나 성능이 마비됨.
3. **해결 전략**:
    - **Buffered I/O**: 라이브러리 레벨에서 데이터를 모았다가(Buffer), 일정 크기가 되면 한 번의 시스템 호출로 처리.
    - **Batching**: 여러 요청을 묶어서 한 번에 커널로 전달.

#### 도입 시 고려사항 (체크리스트)
- [ ] **보안 무결성**: 시스템 호출 테이블(System Call Table)이 변조되지 않도록 보호되고 있는가? (Rootkit 방지)
- [ ] **에러 처리**: 반환된 에러 코드(errno)를 응용 프로그램이 적절히 처리하고 있는가?
- [ ] **호환성**: 운영체제 버전에 따라 시스템 호출 번호나 인자가 바뀌지 않았는가? (POSIX 준수 확인)

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량적/정성적 기대효과
- **자원 독점 방지**: 운영체제가 자원을 중앙 관리하여 특정 앱의 폭주로부터 시스템 보호.
- **개발 생산성**: 복잡한 하드웨어 로직 대신 간단한 함수 호출만으로 고성능 기능 사용.

#### 미래 전망 및 진화 방향
- **io_uring (Linux)**: 시스템 호출의 잦은 전환 오버헤드를 극복하기 위해, 사용자 영역과 커널 영역이 공유하는 '링 버퍼'를 통해 비동기적으로 대량의 요청을 처리하는 차세대 I/O 기술이 대두되고 있습니다.
- **Serverless/WebAssembly**: 아예 시스템 호출을 차단하고 추상화된 런타임 위에서만 동작하게 하여, 극강의 보안과 이식성을 확보하려는 시도가 활발합니다.
