# 인터럽트 (Interrupt)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터럽트는 CPU가 프로그램을 실행하고 있는 도중에 외부나 내부에서 발생한 예기치 않은 사건(Event)을 감지하여, 현재 작업을 일시 중단하고 해당 사건을 즉시 처리하게 하는 하드웨어적 알림 메커니즘이다.
> 2. **가치**: CPU가 주변 장치의 작업 완료를 무작정 기다리는 '폴링(Polling)' 낭비를 제거하여 자원 효율성을 극대화하며, 시스템의 실시간 응답성(Responsiveness)을 보장하는 핵심 동력이다.
> 3. **융합**: 하드웨어 신호 라인, 인터럽트 벡터 테이블(IVT), 인터럽트 서비스 루틴(ISR)이 결합된 형태이며, 현대 운영체제의 선점형 스케줄링(Preemptive Scheduling)을 가능케 하는 근간 기술이다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념
**인터럽트(Interrupt)**는 '가로채기'라는 뜻으로, 프로세서가 현재 수행 중인 명령어 흐름을 잠시 멈추고 더 시급한 일을 처리하도록 하는 신호다. 하드웨어 장치가 데이터를 다 준비했음을 알리거나(HW 인터럽트), 프로그램이 0으로 나누는 오류를 범했을 때(SW 인터럽트/Exception) 발생한다.

#### 💡 비유
인터럽트는 **"요리 중 울리는 초인종 소리"**와 같다.
1. **폴링 (비효율)**: 요리사(CPU)가 택배가 왔는지 확인하려고 1분마다 현관문에 가서 밖을 내다본다. 요리 흐름이 계속 끊기고 다리만 아프다.
2. **인터럽트 (효율)**: 요리사는 요리에만 집중한다. 택배 기사(주변 장치)가 초인종(인터럽트 신호)을 누르면, 그때만 잠시 불을 줄이고 나가서 물건을 받는다.
3. **ISR (처리)**: 물건을 받고 문을 닫는 행동.
4. **복귀**: 다시 주방으로 돌아와 아까 하던 요리를 계속한다.

#### 등장 배경 및 발전 과정
1. **속도 차이의 극복**: CPU는 초당 수십억 번 연산하지만, 키보드나 디스크는 거북이처럼 느리다. 이 속도 차이를 메우기 위해 CPU가 노는 시간을 없애야 했다.
2. **멀티태스킹의 구현**: 타이머 인터럽트를 통해 특정 시간이 지나면 강제로 CPU 주인을 바꿀 수 있게 되면서, 여러 앱이 동시에 돌아가는 것처럼 보이게 되었다.
3. **현재**: 수천 개의 인터럽트를 관리하는 전용 컨트롤러(APIC)와 가상화 환경에서의 인터럽트 가속 기술로 발전했다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 인터럽트 종류 및 분류 (표)

| 분류 | 종류 | 발생 원인 | 예시 |
| :--- | :--- | :--- | :--- |
| **외부 (Hardware)** | **비동기적** | 하드웨어 장치의 신호 | 키보드 입력, 디스크 I/O 완료, 전원 이상 |
| **내부 (Software)** | **동기적 (Trap)** | 프로그램 실행 중 의도적 호출 | 시스템 호출(System Call) |
| **예외 (Exception)** | **동기적 (Fault)** | 프로그램의 잘못된 동작 | 0으로 나누기, 잘못된 메모리 접근(Page Fault) |

#### 인터럽트 처리 프로세스 (ASCII)

```text
    [ 메인 프로그램 실행 중 ]
           │
    (1) 인터럽트 발생 신호 (IRQ) ──► [ CPU ]
           │                          │
           ▼                          ▼
    (2) 현재 상태 보존 (Context Save) │ (PC, Registers -> Stack/PCB)
           │                          │
           ▼                          ▼
    (3) 인터럽트 벡터 테이블 조회     │ (사건 종류에 따른 처리 주소 찾기)
           │                          │
           ▼                          ▼
    (4) ISR 실행 (Interrupt Service Routine)
           │ (실제 작업 수행: 데이터 읽기 등)
           ▼
    (5) 상태 복원 (Context Restore)   │ (Stack/PCB -> CPU Registers)
           │                          │
           ▼                          ▼
    [ 중단되었던 지점부터 재실행 ]
```

#### 심층 동작 원리 (The Priority & Masking)

1. **인터럽트 벡터(Interrupt Vector)**: 인터럽트의 종류별로 번호를 매겨놓은 지도다. 예를 들어 0번은 전원 오류, 3번은 브레이크포인트 식이다. CPU는 이 번호를 보고 실행할 코드(ISR)의 주소를 즉시 찾아간다.
2. **우선순위(Priority)**: 여러 인터럽트가 동시에 터지면, 더 중요한 것(전원 이상 > 키보드)부터 처리한다.
3. **마스킹(Masking)**: 너무 바쁠 때는 "지금 전화 받지 마(Interrupt Disable)"라고 설정할 수 있다. 단, 전원 이상 같은 치명적인 신호는 무시할 수 없는 **NMI(Non-Maskable Interrupt)**로 처리한다.
4. **중첩 인터럽트**: ISR을 실행하는 도중 더 높은 우선순위의 인터럽트가 오면, 현재 ISR을 중단하고 또 다른 ISR로 들어갈 수 있다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교

| 항목 | 인터럽트 (Interrupt) | 폴링 (Polling) | DMA (Direct Memory Access) |
| :--- | :--- | :--- | :--- |
| **제어 주체** | **하드웨어가 CPU에 알림** | CPU가 하드웨어를 체크 | 하드웨어가 직접 메모리 전송 |
| **자원 효율** | **높음** | 매우 낮음 (무한 루프) | 최고 (CPU 개입 최소화) |
| **응답 속도** | 즉각적임 | 체크 주기에 따라 다름 | 전송 완료 시 인터럽트 발생 |
| **구현 난이도** | 중간 (하드웨어 지원 필요) | 매우 쉬움 | 높음 (전용 제어기 필요) |

#### 과목 융합 관점 분석
- **컴퓨터 구조 연계**: CPU 파이프라인에서 인터럽트는 명령어 실행 사이클이 끝나는 시점에 체크됩니다. 인터럽트가 걸리면 파이프라인에 채워진 명령어들을 비워야(Flush) 하므로 오버헤드가 발생합니다.
- **보안 연계**: '인터럽트 가로채기'는 루트킷(Rootkit) 같은 악성코드가 시스템의 제어권을 뺏는 전형적인 방법입니다. 인터럽트 벡터 테이블을 변조하여 백도어 코드를 실행하게 만듭니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 기술사적 판단 (실무 시나리오)
**[시나리오]** 실시간 데이터 수집 서버에서 인터럽트 폭주(Interrupt Storm)로 시스템 마비
1. **상황**: 초당 수십만 개의 네트워크 패킷이 들어와 매번 인터럽트가 발생함.
2. **문제**: CPU가 실제 패킷 처리는 못 하고, 인터럽트 문맥 교환(Context Switch)만 하다가 시간을 다 씀 (Livelock 현상).
3. **해결 전략**:
    - **NAPI (New API)**: 처음 패킷만 인터럽트로 받고, 이후에는 잠시 **폴링 모드**로 전환하여 한꺼번에 패킷을 긁어온 뒤 다시 인터럽트 모드로 복귀.
    - **Interrupt Coalescing**: 하드웨어(NIC) 레벨에서 패킷 여러 개가 모일 때까지 기다렸다가 인터럽트를 한 번만 발생시킴.

#### 도입 시 고려사항 (체크리스트)
- [ ] **ISR의 간결함**: ISR 내부에서 너무 긴 작업을 하고 있지는 않은가? (하부 처리/Bottom-half 권장)
- [ ] **재진입성(Reentrancy)**: ISR이 실행되는 도중 자기 자신이 다시 호출되어도 안전한가?
- [ ] **공유 자원 동기화**: ISR과 일반 프로그램이 같은 변수를 건드릴 때 레이스 컨디션이 발생하지 않는가?

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량적/정성적 기대효과
- **CPU 가동률 향상**: 입출력 대기 시간 중 다른 연산 수행 가능.
- **예측 가능한 반응성**: 이벤트 발생 즉시 처리를 시작하므로 시스템의 신뢰도 향상.

#### 미래 전망 및 진화 방향
- **MSI/MSI-X**: 전통적인 물리적 신호선 대신, PCI-e 메시지 형태로 인터럽트를 전달하여 수천 개의 독립적인 인터럽트 채널 확보.
- **Virtual Interrupt 가속**: 클라우드 가상화 환경에서 Guest OS에게 인터럽트를 전달할 때 발생하는 하이퍼바이저 개입 오버헤드를 하드웨어가 직접 처리(Intel VT-d 등)하여 지연 시간 단축.
