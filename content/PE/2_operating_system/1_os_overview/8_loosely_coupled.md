# 약결합 시스템 (Loosely Coupled System)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 약결합 시스템은 각각 독립적인 메인 메모리와 운영체제를 가진 여러 대의 컴퓨터(노드)가 네트워크를 통해 연결되어 협력하는 분산 컴퓨팅 아키텍처이다.
> 2. **가치**: 특정 노드의 장애가 전체 시스템으로 확산되지 않는 높은 결함 허용성(Fault Tolerance)과, 노드를 추가하는 것만으로 성능을 무한히 확장할 수 있는 유연한 확장성(Scalability)을 제공한다.
> 3. **융합**: 분산 파일 시스템, 메시지 패싱(Message Passing) 통신, 클러스터링 기술의 집약체이며 현대 클라우드 컴퓨팅과 빅데이터 처리 시스템의 근간이다.

---

### Ⅰ. 개요 (Context & Background)

#### 개념
**약결합 시스템(Loosely Coupled System)**은 프로세서 간의 결합도가 낮은 시스템으로, 각 프로세서가 자신만의 국부 메모리(Local Memory)를 가지고 독립적인 운영체제에 의해 제어되는 시스템들의 집합이다. 프로세서 간의 통신은 공유 메모리가 아닌 네트워크를 통한 **메시지 전달(Message Passing)** 방식을 사용한다.

#### 💡 비유
약결합 시스템은 **"각자 자기 사무실에서 일하며 이메일이나 메신저로 협업하는 팀원들"**과 같다.
1. **독립 메모리**: 각자 자기 책상(메모리)과 컴퓨터(OS)를 가지고 있다. 남의 책상을 직접 뒤질 수 없다.
2. **네트워크 통신**: 필요한 정보는 메신저(메시지 패싱)로 물어보고 답을 받아야 한다. 소통에 시간이 좀 걸리지만(지연 시간), 내 책상이 지저분해도 옆 사람 업무에는 지장이 없다.
3. **확장성**: 팀원이 100명이 되어도 각자 방을 주면 되므로 사무실을 무한히 늘릴 수 있다.
4. **단점**: 간단한 정보 하나 묻는 데도 메신저를 써야 하므로 소통 효율은 떨어진다.

#### 등장 배경 및 발전 과정
1. **강결합의 확장성 한계**: 단일 시스템(SMP)은 메모리 버스 병목 때문에 프로세서를 무한정 늘릴 수 없었다.
2. **저비용 고성능**: 비싼 슈퍼컴퓨터 한 대보다 저렴한 일반 PC 여러 대를 네트워크로 묶는 것이 가성비 면에서 압도적이었다 (Beowulf Cluster).
3. **인터넷과 데이터의 폭증**: 구글, 아마존 등 거대 IT 기업들이 전 세계에 흩어진 수만 대의 서버를 하나처럼 관리해야 하는 필요성이 생기며 전성기를 맞이했다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 관련 기술 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **Independent Node** | 독립적인 연산 및 저장 단위 | 개별 CPU + 개별 Memory + 개별 OS | Server, PC, Container | 각자의 사무실 |
| **Local Memory** | 각 노드 전용 데이터 저장 공간 | 타 노드가 직접 접근 불가 | Private RAM | 개인 책상 서랍 |
| **Communication Network** | 노드 간 데이터 전송 통로 | 패킷 스위칭, 프로토콜 처리 | Ethernet, InfiniBand | 사내 메신저 / 이메일 |
| **Message Passing** | 노드 간 정보 교환 방식 | Send / Receive API 호출 | MPI, RPC, REST API | 업무 협조 요청 |
| **Distributed OS / Middleware** | 전체 시스템 통합 관리 및 조정 | 분산 스케줄링, 장애 감지 | Kubernetes, Hadoop | 총무팀 / 프로젝트 관리툴 |

#### 약결합 시스템 구조도 (ASCII)

```text
    ┌─────────────┐       ┌─────────────┐       ┌─────────────┐
    │   Node 1    │       │   Node 2    │       │   Node N    │
    │ ┌─────────┐ │       │ ┌─────────┐ │       │ ┌─────────┐ │
    │ │ OS 1    │ │       │ │ OS 2    │ │       │ │ OS N    │ │
    │ ├─────────┤ │       │ ├─────────┤ │       │ ├─────────┤ │
    │ │ Local   │ │       │ │ Local   │ │       │ │ Local   │ │
    │ │ Memory  │ │       │ │ Memory  │ │       │ │ Memory  │ │
    │ └─────────┘ │       │ └─────────┘ │       │ └─────────┘ │
    └──────┬──────┘       └──────┬──────┘       └──────┬──────┘
           │                     │                     │
    ───────┴──────────┬──────────┴──────────┬──────────┴───────
                      │                     │
              [ Communication Network ]     │
              [ (e.g., High-speed LAN) ] ───┘

    [특징]
    - 지연 시간: 상대적 높음 (Micro-seconds ~ Milli-seconds)
    - 데이터 공유: 메시지 전달 (네트워크 오버헤드 존재)
    - 확장성: 매우 높음 (수만 대 이상의 노드 가능)
```

#### 심층 동작 원리 (Communication & Resilience)

1. **메시지 전달(Message Passing)**: 데이터 공유를 위해 송신 노드가 데이터를 패킷으로 싸서 보내고, 수신 노드가 이를 받아 자신의 메모리에 푼다. 이 과정에서 프로토콜 스택 오버헤드가 발생한다.
2. **분산 제어**: 각 노드가 독립적인 OS를 가지므로, 전체 시스템을 총괄하는 미들웨어가 작업을 분배하고 각 노드의 생존 여부(Heartbeat)를 체크한다.
3. **장애 격리(Fault Isolation)**: 특정 노드가 블루스크린이 뜨거나 타버려도, 네트워크 연결만 끊길 뿐 다른 노드의 메모리나 CPU에는 물리적 영향이 전혀 없다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 심층 기술 비교

| 항목 | 강결합 (Tightly Coupled) | 약결합 (Loosely Coupled) |
| :--- | :--- | :--- |
| **데이터 공유** | **공유 메모리 (Shared)** | 메시지 전달 (Message) |
| **결합도** | 높음 (공통 OS/Bus) | **낮음 (독립 OS/Network)** |
| **장애 영향** | 전체 시스템으로 파급 위험 | **해당 노드에 국한** |
| **프로그래밍 모델** | 스레드, 공유 변수 (쉬움) | **프로세스 간 통신 (어려움)** |
| **비용** | 고가의 전용 장비 | **저렴한 범용 장비 조합 가능** |
| **시스템 예시** | 멀티코어 PC, 메인프레임 | **Google 검색, AWS, 슈퍼컴 클러스터** |

#### 과목 융합 관점 분석
- **데이터베이스 연계**: NoSQL 데이터베이스(Cassandra, MongoDB)의 'Shared Nothing' 아키텍처는 약결합 시스템의 전형적인 소프트웨어 구현체입니다.
- **네트워크 연계**: 노드 간 지연 시간을 줄이기 위해 TCP/IP 대신 하드웨어 레벨에서 직접 메모리를 찌르는 RDMA(Remote Direct Memory Access) 기술이 약결합 시스템의 성능을 강결합 수준으로 끌어올리고 있습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 기술사적 판단 (실무 시나리오)
**[시나리오]** 전 세계 사용자를 대상으로 하는 소셜 미디어 서비스 구축
1. **분석**: 사용자 수가 수억 명이며, 트래픽을 예측하기 어렵고 24시간 중단 없는 서비스가 생명임.
2. **선택**: **약결합 시스템(Microservices on Cloud)**이 필수.
3. **판단**: 강결합 서버는 성능은 좋지만 확장성과 가용성 면에서 글로벌 서비스를 감당할 수 없음. 수천 대의 약결합 노드를 배치하여 부하에 따라 늘리고 줄이는(Auto-scaling) 전략이 최선임.

#### 도입 시 고려사항 (체크리스트)
- [ ] **네트워크 병목**: 노드 간 통신량이 너무 많아 네트워크가 마비되지 않는가? (Data Locality 고려)
- [ ] **데이터 일관성**: 여러 노드에 분산된 데이터의 일관성을 어떻게 유지할 것인가? (CAP 정리 고려)
- [ ] **오케스트레이션**: 수많은 노드의 상태를 관리하고 작업을 배분할 자동화 도구(Kubernetes 등)가 준비되었는가?

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량적/정성적 기대효과
- **무한 확장성**: 서비스 성장에 맞춰 서버를 추가하기만 하면 처리 능력 선형적 증가.
- **고가용성**: 일부 장비 고장 시에도 전체 서비스는 중단 없이 유지.

#### 미래 전망 및 진화 방향
- **서버리스(Serverless)**: 약결합의 극단적인 형태로, 개발자는 노드 자체를 신경 쓰지 않고 코드(Function)만 던지면 분산 시스템이 알아서 실행하는 구조.
- **엣지 컴퓨팅**: 약결합 노드들을 사용자 근처(엣지)에 배치하여 네트워크 지연 시간을 물리적으로 극복하는 방향으로 진화 중.
