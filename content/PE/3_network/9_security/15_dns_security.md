# DNS 보안 (DNS Security)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DNS는 **"도메인 이름을 **IP 주소로 **변환하는 **계층적 분산 데이터베이스"**로, **캐시 포이즈닝**(Cache Poisoning), **스푸핑**(Spoofing), **하이재킹**(Hijacking) 같은 **보안 취약점**이 존재하고 **DNSSEC**(Digital Signature)와 **암호화 전송**(DoH/DoT)으로 **보안**을 강화한다.
> 2. **가치**: **인터넷 인프라의 핵심**으로 **DNS 장애**는 **전체 서비스 중단**으로 이어지며 **DNSSEC**로 **데이터 무결성**을 **보장**하고 **DoH**(DNS over HTTPS), **DoT**(DNS over TLS)로 **프라이버시**를 **보호**한다.
> 3. **융합**: **DNSSEC**(RFC 4033-4035)는 **Digital Signature**로 **응답 위조**를 **방지**하고 **EDNS0**(Extension Mechanisms for DNS)는 **큰 응답**을 **지원**하며 **DNS over QUIC**(DoQ)로 **차세대 전송**이 **연구**된다.

## Ⅰ. 개요 (Context & Background)

### 개념
DNS는 **"계층적 분산 데이터베이스로 도메인 이름을 IP로 변환하는 시스템"**이다.

**DNS 보안 문제**:
- **Cache Poisoning**: 가짜 응답으로 캐시 오염
- **Spoofing**: 응답 위조
- **Amplification Attack**: DNS를 이용한 공격

### 💡 비유
DNS는 **"전화번호부"**와 같다.
- **보안 없음**: 누군가 번호 변경 가능
- **DNSSEC**: 번호부에 서명 (위조 방지)
- **DoH/DoT**: 암호화된 통신 (도청 방지)

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         DNS 보안의 발전                                                  │
└─────────────────────────────────────────────────────────────────────────────────────────┘

DNS (1983):
    • 보안 없음
    • 신뢰 기반
         ↓
Cache Poisoning (2000년대):
    • Dan Kaminsky 공격
    • DNS 취약점 노출
         ↓
DNSSEC (2005):
    • RFC 4033-4035
    • Digital Signature
         ↓
DoH/DoT (2010년대):
    • 프라이버시 강화
    • 암호화 전송
```

### DNSSEC (DNS Security Extensions)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         DNSSEC 서명 체인                                                 │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Root Zone                                                                              │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  • Root KSK (Key Signing Key): 신뢰의 시작점                                        │  │
    │  │  • Root ZSK (Zone Signing Key): Root Zone 서명                                      │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                      │                                                   │
    │                                      ▼                                                   │
    │  TLD Zone (.com)                                                                       │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  • .com KSK: Root로부터 신뢰                                                       │  │
    │  │  • .com ZSK: .com Zone 서명                                                         │  │
    │  │  • DS Record: Hash of .com KSK (Root에 저장)                                        │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    │                                      │                                                   │
    │                                      ▼                                                   │
    │  Example Zone (example.com)                                                            │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  • example.com KSK: .com로부터 신뢰                                                │  │
    │  │  • example.com ZSK: Zone 서명                                                      │  │
    │  │  • DNSKEY: Public Key                                                              │  │
    │  │  • RRSIG: Resource Record Signature                                                 │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [DNSSEC 레코드]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • DNSKEY: Public Key                                                                 │  │
    │  • RRSIG: Resource Record Signature (각 레코드별 서명)                                  │  │
    │  • DS: Delegation Signer (상위 Zone에 저장된 하위 Zone Key의 해시)                         │  │
    │  • NSEC/NSEC3: 존재하지 않는 이름 증명 (Non-existence)                                   │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

    │  │  • URI: /dns-query                                                                  │

    │  │                                                                                      │

    │  │  POST /dns-query HTTP/2                                                            │  │
    │  │  Host: dns.example.com                                                              │  │
    │  │  Content-Type: application/dns-message                                              │  │

    │  │                                                                                     │  │
    │  │  [Binary DNS Query Wire Format]                                                    │  │
    │  │                                                                                     │  │
    │  │  HTTP/2 200 OK                                                                      │  │
    │  │  Content-Type: application/dns-message                                              │  │
    │  │                                                                                     │  │
    │  │  [Binary DNS Response Wire Format]                                                 │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [DoT - DNS over TLS]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • TLS 1.3으로 DNS 쿼리 전송                                                            │
    │  │  • Port: 853                                                                         │
    │  │  • Wire Format 유지                                                                  │
    │  │                                                                                      │
    │  ┌────────────────────────────────────────────────────────────────────────────────────┐  │
    │  │  TCP Handshake → TLS Handshake → DNS Query/Response                               │  │
    │  └────────────────────────────────────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    [비교]

    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  구분      │ DoH                  │ DoT                  │ 기존 DNS                    │
    │  ─────────────────────────────────────────────────────────────────────────────────────  │
    │  Protocol  │ HTTPS/HTTP/2         │ TLS/TCP              │ UDP/TCP                    │
    │  Port      │ 443                  │ 853                  │ 53                         │
    │  Encryption│ O                    │ O                    │ X                           │
    │  Firewall │ 까다운 구분 어려움     │ 명확한 포트           │ 명확한 포트                │
    │  Overhead │ 높음 (HTTP)          │ 중간 (TLS)           │ 낮음                       │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

## Ⅲ. 융합 비교 및 다각도 분석

### DNS 보안 대책

| 공격 유형 | 대책 | 설명 |
|----------|------|------|
| **Cache Poisoning** | DNSSEC | 서명 검증 |
| **스푸핑** | TSIG, SIG(0) | 트랜잭션 서명 |
| **도청** | DoH/DoT | 암호화 |
| **Amplification** | Rate Limiting | 응답 제한 |

### DNSSEC 검증 과정

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         DNSSEC 검증                                                     │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    1. Resolver가 Root KSK로 Root Zone 서명 검증
       └── 신뢰의 닻 (Trust Anchor)

    2. Root의 DS 레코드로 .com KSK 검증
       └── Root가 서명한 .com 키의 해시

    3. .com ZSK로 example.com 서명 검증
       └── .com이 서명한 example.com 레코드

    4. example.com ZSK으로 A 레코드 서명 검증
       └── www.example.com = 93.184.216.34 (위조 여부 확인)
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: DNS 보안 설정
**상황**: 도메인 DNSSEC 배포
**판단**:

```bash
# 1. DNS Key 생성 (BIND)
$ dnssec-keygen -a RSASHA256 -b 2048 -n ZONE example.com
# Kexample.com.+008+12345.key  (Public Key)
# Kexample.com.+008+12345.private (Private Key)

# 2. Zone 파일 서명
$ dnssec-signzone -K /path/to/keys -o example.com example.com.db
# example.com.db.signed

# 3. DS 레코드 생성 (상위 Zone 등록용)
$ dnssec-dsfromkey Kexample.com.+008+12345.key
# example.com. IN DS 12345 8 2 <SHA256_DIGEST>

# 4. 상위 Zone(.com)에 DS 레코드 등록
# Registrar를 통해 등록

# 5. named.conf 설정
zone "example.com" {
    type master;
    file "example.com.db.signed";
    allow-update { none; };
};

# 6. DNSSEC 검증 확인
$ dig +dnssec www.example.com
$ delv www.example.com

# DoH 설정 (Cloudflare)
# /etc/systemd/resolved.conf
[Resolve]
DNS=1.1.1.1
DNS-over-HTTPS=yes
FallbackDNS=1.0.0.1
```
