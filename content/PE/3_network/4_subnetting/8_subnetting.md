# IP 서브넷팅 (IP Subnetting)

## 핵심 인사이트 (3줄 요담)
> 1. **본질**: IP 서브넷팅은 **"하나의 네트워크를 여러 개의 작은 논리적 네트워크(Subnet)로 분할하는 기술"**로, **CIDR(Classless Inter-Domain Routing)** 표기법으로 **유연하게 네트워크를 설계**하며 **VLSM(Variable Length Subnet Mask)**를 통해 **주소 낭비**를 최소화한다.
> 2. **가치**: **Broadcast Domain(브로드캐스트 도메인)**을 줄이 **네트워크 트래픽**을 감소시키고 **보안** 및 **관리**를 위해 **VLAN(Virtual LAN)**처럼 **논리적으로 분리**된 네트워크를 구현하며 **Classless Addressing**으로 고정된 Class(A, B, C)의 낭비를 해결한다.
> 3. **융합**: **AWS VPC**, **Azure VNet**, **GCP Network**의 **서브넷 설계**의 기반이 되며, **Cloudflare CDN**, **Route 53**, **Cloud Firewall**의 **DNS**와 **Routing** 설정의 기초가 되고 **SDN(Software-Defined Networking)**의 **Overlay Network** 구현에 사용된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
IP 서브넷팅은 **"하나의 네트워크를 여러 개의 작은 네트워크(Subnet)로 분할하는 기술"**이다.

**서브넷팅의 목적**:
- **주소 절약**: IP 주소 낭비 방지
- **네트워크 분리**: 보안, 관리, 성능 최적화
- **Broadcast 감소**: Broadcast Domain 축소

### 💡 비유
서브넷팅은 **"회사의 부서별 네트워크 구성"**과 같다.
- **본사 네트워크**: 203.0.113.0/24
- - **개발팀**: 203.0.113.128/25
- - **운영팀**: 203.0.113.64/26
- - **재무팀**: 203.0.113.192/26

### 등장 배경

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         IP 서브넷팅의 필요성                                              │
└─────────────────────────────────────────────────────────────────────────────────────────┘

Classful Addressing (1981년 이전):
    • A Class: 8비트 네트워크 (16,777,214개 주소)
    • B Class: 16비트 네트워크 (65,536개 주소)
    • C Class: 24비트 넌트워크 (254개 주소)
    • 문제: 낭비 심각 (큰 조직은 A Class를 전혀 사용 안 함)
         ↓
Subnetting (1985년, RFC 950):
    • 클래스 내 분할 허용
    • FLSM (Fixed Length Subnet Mask)
    • 8비트, 16비트, 24비트를 서브넷으로 분할
         ↓
VLSM (Variable Length Subnet Mask, 1993년, RFC 1519):
    • 가변 길이 서브넷 마스크
    • 주소 낭비 최소화
    • CIDR 표기법
         ↓
현대 네트워크:
    • VPC (Virtual Private Cloud)
    • SDN (Software-Defined Networking)
    • Overlay Network (VXLAN, NVGRE)
```

## Ⅲ. 융합 비교 및 다각도 분석

### FLSM vs VLSM

| 구분 | FLSM | VLSM |
|------|------|------|
| **마스크 길이** | 고정 (/8, /16, /24) | 가변 (/8~32) |
| **주소 낭비** | 심각 | 최소화 |
| **유연성** | 낮음 | 높음 |
| **관리** | 간단 | 복잡함 |

### 과목 융합 관점 분석

#### 1. 보안 ↔ 서브넷팅
- **DMZ**: Demilitarized Zone
- **VLAN**: L2 서브넷팅
- **Subnet**: L3 서브넷팅

#### 2. 클라우드 ↔ 서브넷팅
- **VPC Subnet**: 가상 서브넷
- **Subnet Route**: 트래픽 라우팅

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 3계층 VPC 설계
**상황**: 웹, 앱, DB, 관리 티어
**판단**:

```bash
# VPC: 10.0.0.0/16 (65,534개 주소)

# Web Tier: 10.0.1.0/24 (254개)
# App Tier: 10.0.2.0/24 (254개)
# DB Tier: 10.0.3.0/24 (254개)
# Mgmt: 10.0.0.0/24 (254개)

# Routing
# Web → App: 10.0.1.0/24 via 10.0.2.1
# App → DB: 10.0.3.0/24 via 10.0.3.1
```

## Ⅴ. 기대효과 및 결론

### 서브넷팅 기대 효과

| 효과 | 미서브넷팅 | 서브넷팅 |
|------|---------|----------|
| **주소 효율** | 낮음 | 높음 |
| **Broadcast** | 전체 네트워크 | 서브넷만 |
|| **보안** | 전체에 영향 | 서브넷만 영향 |
|| **유연성** | Class 기반 | 필요에 따라 |

### 미래 전망

1. **IPv6 Subnetting`: /64부터 /128
2. **SDN**: 동적 서브넷 할당
3. **Intent-Based Networking**: 정책 기반

### ※ 참고 표준/가이드
- **RFC 950**: Subnetting
- **RFC 1519**: CIDR
- **RFC 4632**: IPv6 SLAAC
